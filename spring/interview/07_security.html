Explain the Spring Security filter chain architecture from DelegatingFilterProxy to SecurityFilterChain.
FILTER_CHAIN
▼

The Full Request Path

  1. Servlet Container (Tomcat/Jetty) receives HTTP request
  2. DelegatingFilterProxy — a standard servlet Filter registered in the servlet context. It bridges the servlet container world and the Spring application context. It delegates to a Spring bean named springSecurityFilterChain.
  3. FilterChainProxy (the springSecurityFilterChain bean) — Spring-managed. Iterates registered SecurityFilterChain beans and delegates to the first one whose request matcher matches the incoming request URL.
  4. SecurityFilterChain — a list of ordered Spring Security filters specific to a URL pattern. Each filter handles one security concern.

Multiple SecurityFilterChain Beans — Different Rules per URL

@Configuration
@EnableWebSecurity
public class SecurityConfig {

    // API chain: JWT stateless, no CSRF, /api/** paths
    @Bean
    @Order(1)
    public SecurityFilterChain apiFilterChain(HttpSecurity http) throws Exception {
        http
            .securityMatcher("/api/**")
            .authorizeHttpRequests(auth -> auth
                .requestMatchers("/api/public/**").permitAll()
                .anyRequest().authenticated()
            )
            .sessionManagement(sm -> sm.sessionCreationPolicy(SessionCreationPolicy.STATELESS))
            .csrf(csrf -> csrf.disable())
            .oauth2ResourceServer(oauth2 -> oauth2.jwt(Customizer.withDefaults()));
        return http.build();
    }

    // Web chain: form login with sessions, /** paths
    @Bean
    @Order(2)
    public SecurityFilterChain webFilterChain(HttpSecurity http) throws Exception {
        http
            .authorizeHttpRequests(auth -> auth
                .requestMatchers("/public/**", "/login", "/register").permitAll()
                .anyRequest().authenticated()
            )
            .formLogin(form -> form
                .loginPage("/login")
                .defaultSuccessUrl("/dashboard")
            )
            .logout(logout -> logout.logoutSuccessUrl("/login"));
        return http.build();
    }
}

Why DelegatingFilterProxy Exists

The servlet container initializes filters before the Spring ApplicationContext is ready. DelegatingFilterProxy is a thin servlet filter registered at startup time that lazily looks up the actual Spring bean (springSecurityFilterChain) when the first request arrives. This allows Spring Security's filters to be full Spring beans with all dependency injection capabilities.

Best PracticeDefine separate SecurityFilterChain beans for different security domains (API vs web, admin vs user). Use @Order to control which chain matches first — more specific matchers should have lower order numbers. A request matches the FIRST chain whose securityMatcher() returns true — subsequent chains are skipped.
Google L6 AngleAt Google scale, the filter chain insight that matters is: every HTTP request walks through ALL registered filters before reaching your controller. Even 200-microsecond-per-filter overhead adds up at 100,000 RPS. Keep the chain lean: disable unnecessary filters for your security model (e.g., CsrfFilter for stateless JWT APIs). Profile filter chain overhead with Micrometer + distributed tracing when debugging latency spikes under load.

Step-by-Step Authentication Flow

  1. Client sends POST /login with username and password in the request body (form fields or JSON)
  2. UsernamePasswordAuthenticationFilter intercepts the request. Its requiresAuthentication() checks the path matches /login (or configured path). Calls attemptAuthentication().
  3. Creates an unauthenticated UsernamePasswordAuthenticationToken(username, password) — principal=username, credentials=password, authorities=null
  4. Calls AuthenticationManager.authenticate(token) — typically a ProviderManager instance
  5. ProviderManager iterates its list of AuthenticationProvider implementations, calling supports(authToken.getClass()) on each. Finds DaoAuthenticationProvider.
  6. DaoAuthenticationProvider calls UserDetailsService.loadUserByUsername(username) — loads your UserDetails object (username, encoded password, authorities, account flags)
  7. Calls PasswordEncoder.matches(rawPassword, userDetails.getPassword()) — BCrypt hash comparison
  8. On success: creates an authenticated UsernamePasswordAuthenticationToken(userDetails, null, authorities) — principal=UserDetails object, credentials=null (cleared for security), authorities=loaded roles
  9. SecurityContextHolder.getContext().setAuthentication(authenticatedToken)
  10. SecurityContextRepository.saveContext() — saves to HttpSession (stateful) or is a no-op (stateless JWT)
  11. AuthenticationSuccessHandler invoked — default redirects to originally requested URL or defaultSuccessUrl

Custom UserDetailsService

@Service
public class UserDetailsServiceImpl implements UserDetailsService {

    @Autowired UserRepository userRepo;

    @Override
    @Transactional(readOnly = true) // read-only — just loading user
    public UserDetails loadUserByUsername(String username) throws UsernameNotFoundException {
        User user = userRepo.findByEmail(username)
            .orElseThrow(() -> new UsernameNotFoundException("User not found: " + username));

        return org.springframework.security.core.userdetails.User
            .withUsername(user.getEmail())
            .password(user.getPasswordHash())
            .roles(user.getRoles().stream().map(Role::getName).toArray(String[]::new))
            .accountExpired(!user.isActive())
            .accountLocked(user.isLocked())
            .credentialsExpired(user.isPasswordExpired())
            .disabled(!user.isEnabled())
            .build();
    }
}

JWT Success Handler for REST APIs

@Component
public class JwtAuthenticationSuccessHandler implements AuthenticationSuccessHandler {

    @Autowired JwtTokenProvider jwtProvider;

    @Override
    public void onAuthenticationSuccess(HttpServletRequest request,
                                         HttpServletResponse response,
                                         Authentication auth) throws IOException {

            "refresh_token", refreshToken,
            "expires_in", 900 // 15 minutes
        )));
    }
}

Authentication Failure Flow

If DaoAuthenticationProvider throws BadCredentialsException or UsernameNotFoundException (Spring normalizes both to BadCredentialsException by default to prevent username enumeration), AuthenticationFailureHandler is invoked. Default: redirect to /login?error. Custom: return 401 JSON.

GotchaSpring Security hides whether a username exists by default — both "user not found" and "wrong password" throw BadCredentialsException to the client. If you throw UsernameNotFoundException directly in your UserDetailsService, some configurations will expose whether the username exists. Ensure hideUserNotFoundExceptions=true (DaoAuthenticationProvider default) is not disabled.
Google L6 AngleAt L6, the authentication flow discussion extends to: (1) brute-force protection via failed-attempt counting + account locking, (2) IP-based rate limiting on /auth/login (Spring Security doesn't do this — needs an API gateway or custom filter), (3) suspicious login detection (new country, new device), and (4) multi-factor authentication integration. Know that FIDO2/WebAuthn is the modern answer to credential phishing — Spring Security 6 has WebAuthn support.

Spring Security 6 JWT Resource Server Configuration

@Configuration
@EnableWebSecurity
public class JwtSecurityConfig {

    @Bean
    public SecurityFilterChain filterChain(HttpSecurity http) throws Exception {
        http
            .authorizeHttpRequests(auth -> auth
                .requestMatchers("/auth/**", "/actuator/health").permitAll()
                .requestMatchers("/admin/**").hasRole("ADMIN")
                .anyRequest().authenticated()
            )
            .sessionManagement(sm ->
                sm.sessionCreationPolicy(SessionCreationPolicy.STATELESS))
            .csrf(csrf -> csrf.disable()) // stateless = no CSRF risk
            .oauth2ResourceServer(oauth2 ->
                oauth2.jwt(jwt -> jwt
                    .decoder(jwtDecoder())
                    .jwtAuthenticationConverter(jwtAuthenticationConverter())
                ));
        return http.build();
    }

    // Option 1: Remote JWKS URI (production — Auth Server holds private key, we have public key via JWKS)
    @Bean
    public JwtDecoder jwtDecoder() {
        NimbusJwtDecoder decoder = NimbusJwtDecoder
            .withJwkSetUri("https://auth.example.com/.well-known/jwks.json")
            .jwsAlgorithm(SignatureAlgorithm.RS256)
            .build();

        // Add custom validators
        OAuth2TokenValidator<Jwt> withIssuer = JwtValidators.createDefaultWithIssuer("https://auth.example.com");
        OAuth2TokenValidator<Jwt> withAudience = new JwtClaimValidator<>(
            JwtClaimNames.AUD,
            aud -> aud != null && ((List<?>)aud).contains("order-service")
        );
        OAuth2TokenValidator<Jwt> combined = new DelegatingOAuth2TokenValidator<>(withIssuer, withAudience);
        decoder.setJwtValidator(combined);
        return decoder;
    }

    // Option 2: Local public key (symmetric or asymmetric self-issued JWTs)
    @Bean
    public JwtDecoder localJwtDecoder(@Value("${jwt.public-key}") RSAPublicKey publicKey) {
        return NimbusJwtDecoder.withPublicKey(publicKey).build();
    }

    // Map JWT claims to Spring Security GrantedAuthorities
    @Bean
    public JwtAuthenticationConverter jwtAuthenticationConverter() {
        JwtGrantedAuthoritiesConverter authoritiesConverter = new JwtGrantedAuthoritiesConverter();
        authoritiesConverter.setAuthoritiesClaimName("roles"); // custom claim name
        authoritiesConverter.setAuthorityPrefix("ROLE_");      // Spring Security convention

        JwtAuthenticationConverter jwtConverter = new JwtAuthenticationConverter();
        jwtConverter.setJwtGrantedAuthoritiesConverter(authoritiesConverter);
        jwtConverter.setPrincipalClaimName("sub"); // claim used as principal name
        return jwtConverter;
    }
}

JWT Validation Pipeline — Internal Flow

  1. BearerTokenAuthenticationFilter extracts the token: Authorization: Bearer <token>
  2. Creates BearerTokenAuthenticationToken(tokenValue)
  3. Calls AuthenticationManager.authenticate() → delegates to JwtAuthenticationProvider
  4. JwtDecoder.decode(tokenString) (NimbusJwtDecoder):
    • Parses JWT structure (header.payload.signature)
    • Fetches JWKS from Auth Server (cached with TTL)
    • Verifies signature using the correct public key (kid header matches)
    • Runs all registered OAuth2TokenValidator<Jwt>: exp, nbf, iss, aud, custom validators
  5. JwtAuthenticationConverter.convert(jwt) → creates JwtAuthenticationToken with authorities from claims
  6. SecurityContextHolder receives the authenticated token

Token Issuance — If You Own the Auth Server

// Using nimbus-jose-jwt directly (or use Spring Authorization Server)
@Component
public class JwtTokenProvider {

    private final RSAKey rsaKey;

    public JwtTokenProvider() throws JOSEException {
        this.rsaKey = new RSAKeyGenerator(2048)
            .keyID("key-2024-01")
            .algorithm(JWSAlgorithm.RS256)
            .generate();
    }

    public String generateAccessToken(UserDetails user) throws JOSEException {
        JWSHeader header = new JWSHeader.Builder(JWSAlgorithm.RS256)
            .keyID(rsaKey.getKeyID()).build();

        JWTClaimsSet claims = new JWTClaimsSet.Builder()
            .subject(user.getUsername())
            .issuer("https://auth.example.com")
            .audience("order-service")
            .issueTime(new Date())
            .expirationTime(Date.from(Instant.now().plus(15, ChronoUnit.MINUTES)))
            .claim("roles", user.getAuthorities().stream()
                .map(GrantedAuthority::getAuthority).toList())
            .build();

        SignedJWT jwt = new SignedJWT(header, claims);
        jwt.sign(new RSASSASigner(rsaKey));
        return jwt.serialize();
    }
}

Refresh Token Strategy

@RestController
@RequestMapping("/auth")
public class AuthController {

    @PostMapping("/refresh")
    public ResponseEntity<TokenResponse> refresh(@RequestBody RefreshRequest req) {
        String refreshToken = req.getRefreshToken();

        // 1. Look up in Redis/DB — not JWT-validated alone (can be revoked)
        StoredRefreshToken stored = refreshTokenStore.find(refreshToken)
            .orElseThrow(() -> new InvalidTokenException("Invalid or expired refresh token"));

        // 2. Check not expired and not revoked
        if (stored.isExpired() || stored.isRevoked()) {
            throw new InvalidTokenException("Refresh token expired or revoked");
        }

        // 3. Rotate: invalidate old, issue new
        refreshTokenStore.revoke(refreshToken);
        UserDetails user = userDetailsService.loadUserByUsername(stored.getUsername());
        String newAccessToken = jwtProvider.generateAccessToken(user);
        String newRefreshToken = jwtProvider.generateRefreshToken(user);
        refreshTokenStore.save(newRefreshToken, user.getUsername(), Duration.ofDays(7));

        return ResponseEntity.ok(new TokenResponse(newAccessToken, newRefreshToken));
    }
}
Best PracticeAccess tokens: short-lived (15 minutes), self-contained JWT, not stored on server. Refresh tokens: long-lived (7 days), opaque token stored in server-side store (Redis/DB), invalidatable. Deliver refresh tokens via HTTP-only Secure SameSite=Strict cookies — not localStorage (vulnerable to XSS). Rotate refresh tokens on each use (refresh token rotation) to detect token theft.
Google L6 AngleAt Google, token validation is a hot path — every authenticated API call validates a JWT. JWKS caching (Spring's JwtDecoder caches the key set) is critical — avoid fetching JWKS on every request. Know the JWKS cache TTL vs key rotation timing (staggered rotation to allow both old and new keys to be valid during transition). For high-security contexts, use short-lived access tokens (5 minutes) + sliding session refresh tokens.

@PreAuthorize — The Standard Choice

Evaluated BEFORE method execution using Spring Expression Language (SpEL). If the expression evaluates to false, AccessDeniedException is thrown and the method body never runs.

// Simple role check
@PreAuthorize("hasRole('ADMIN')")
public void deleteUser(Long userId) { ... }

// Multiple roles
@PreAuthorize("hasAnyRole('ADMIN', 'MODERATOR')")
public void banUser(Long userId) { ... }

// Check OAuth2 scope
@PreAuthorize("hasAuthority('SCOPE_orders:write')")
public Order createOrder(OrderRequest req) { ... }

// Access method parameter — note the #paramName syntax
@PreAuthorize("hasRole('ADMIN') or #targetUserId == authentication.principal.id")
public void updateUser(Long targetUserId, UserDTO dto) { ... }

// Complex expression: must be authenticated AND email must be verified
@PreAuthorize("isAuthenticated() and principal.emailVerified")
public void sendMessage(MessageRequest req) { ... }

// Access Spring beans in SpEL using @beanName
@PreAuthorize("@subscriptionService.isPremium(authentication.name)")
public List<Report> getPremiumReports() { ... }

@PostAuthorize — Filter Based on Return Value

// returnObject is available after method executes
// Throws AccessDeniedException if expression is false (after the method already ran)
@PostAuthorize("returnObject.ownerId == authentication.principal.id or hasRole('ADMIN')")
public Document getDocument(Long docId) {
    return documentRepo.findById(docId).orElseThrow();
    // Document is fetched regardless, then access is checked on the result
}

// Use case: check if result belongs to current user without pre-loading user context
@PostAuthorize("returnObject.username == authentication.name")
public UserProfile getProfile(Long userId) { ... }

@PreFilter / @PostFilter — Collection Filtering

// Remove elements from input collection that fail the expression (filterObject = each element)
@PreFilter("filterObject.ownerId == authentication.principal.id")
public void updateDocuments(List<Document> documents) { ... }

// Remove elements from returned collection
@PostFilter("filterObject.isPublic() or filterObject.ownerId == authentication.principal.id")
public List<Document> findAllDocuments() { ... }

@Secured and @RolesAllowed — Simpler but Less Powerful

// @Secured: Spring-specific, array of roles
@Secured({"ROLE_ADMIN", "ROLE_MANAGER"})
public void exportData() { ... }

// @RolesAllowed: JSR-250, portable across Java EE/Jakarta EE
@RolesAllowed({"ADMIN", "MANAGER"}) // Spring auto-adds ROLE_ prefix
public void importData() { ... }

// Neither supports SpEL, method arguments, or returnObject — use @PreAuthorize for those

GotchaMethod security uses AOP proxies — the same self-invocation limitation applies. If serviceA.methodA() calls this.methodB() @PreAuthorize, the annotation on methodB is IGNORED. Extract to a separate bean. Also: @PreAuthorize on private methods is silently ignored (CGLIB cannot proxy private methods). Always make secured methods public.
Best PracticeUse @PreAuthorize exclusively for all method-level security decisions — it is the most expressive and readable. Define security expressions as constants or delegate complex logic to a PermissionEvaluator bean via hasPermission(). Google L6 Angle@PreAuthorize is the entry point for fine-grained access control. For object-level authorization (can user X edit document Y?), the hasPermission() SpEL function delegates to a custom PermissionEvaluator — this is the pattern that scales to Zanzibar/ReBAC-style authorization. At Google scale, fine-grained authz is often a separate service (SpiceDB, Ory Keto) with local caching, not in-process Spring beans.

Explain OAuth2 Authorization Code flow with PKCE. How does Spring Security implement it as a client?
OAUTH2
▼

Authorization Code + PKCE Flow

  1. Client generates PKCE: code_verifier = cryptographically random 43-128 char string. code_challenge = Base64URL(SHA256(code_verifier)). code_challenge_method = S256.
  2. Redirect to Authorization Server: GET /authorize?response_type=code&client_id=app&redirect_uri=https://app.example.com/callback&scope=openid+email+profile&state=random_state&code_challenge=XYZ&code_challenge_method=S256
  3. User authenticates on the Authorization Server (login page). Grants consent if prompted.
  4. AS redirects back: https://app.example.com/callback?code=AUTH_CODE&state=random_state. Client validates state matches what was stored (prevents CSRF on the callback).
  5. Client exchanges code for tokens: POST /token with grant_type=authorization_code&code=AUTH_CODE&redirect_uri=...&code_verifier=VERIFIER. AS verifies SHA256(code_verifier) == code_challenge (stored in AS from step 2). Prevents code interception attacks.
  6. AS responds: access_token, id_token (OpenID Connect), refresh_token, expires_in.

Using Access Tokens to Call Downstream APIs

@Service
public class GoogleCalendarClient {

    @Autowired OAuth2AuthorizedClientManager authorizedClientManager;

    public List<CalendarEvent> getEvents(Authentication authentication) {
        OAuth2AuthorizedClient client = authorizedClientManager.authorize(
            OAuth2AuthorizeRequest
                .withClientRegistrationId("google")
                .principal(authentication)
                .build()
        );

        String accessToken = client.getAccessToken().getTokenValue();

        return webClient.get()
            .uri("https://www.googleapis.com/calendar/v3/calendars/primary/events")
            .header(HttpHeaders.AUTHORIZATION, "Bearer " + accessToken)
            .retrieve()
            .bodyToFlux(CalendarEvent.class)
            .collectList()
            .block();
    }
}
Best PracticePKCE is mandatory for public clients (SPAs, mobile apps) and recommended for confidential clients (server-side apps). Without PKCE, a stolen authorization code can be exchanged for tokens by the attacker. Spring Security 6 enables PKCE automatically for public clients. Never store OAuth2 access tokens in localStorage — use HTTP-only cookies for the session that contains the token reference.
Google L6 AngleAt Google scale, you will use Google Sign-In (OIDC) for consumer apps and Workload Identity Federation or service account credentials for service-to-service auth. The L6 insight: OAuth2 client credential flow (not authorization code) is used for machine-to-machine API calls — no user involved. Know the difference between OAuth2 (authorization) and OpenID Connect (authentication — adds id_token with user identity claims). OIDC is OAuth2 + identity layer.

Explain CSRF attacks and Spring Security's protection. When is it safe to disable?
CSRF
▼

How CSRF Attacks Work

Cross-Site Request Forgery exploits the browser's behavior of automatically sending cookies (including session cookies) with every request to a domain, regardless of which site initiated the request.

  1. User logs in to bank.com and has a valid session cookie
  2. User visits evil.com (in another tab)
  3. evil.com contains: <img src="https://bank.com/transfer?to=attacker&amount=10000"> or a hidden form auto-submitted with JS
  4. Browser sends the request to bank.com WITH the session cookie — legitimate from bank's perspective
  5. Transfer executes as the victim user

Spring Security CSRF Protection

CsrfFilter generates a random, session-unique token and stores it server-side. Every state-changing request (POST, PUT, DELETE, PATCH) must include this token in a header or form field. An attacker's site cannot read this token due to same-origin policy (cannot read cookies/responses from another origin).

// Default: HttpSession-based CSRF token
// Token stored in session, must be sent as form field _csrf or header X-CSRF-TOKEN

// Spring MVC template (Thymeleaf auto-includes _csrf)
<form th:action="@{/transfer}" method="post">
    <input type="hidden" th:name="${_csrf.parameterName}" th:value="${_csrf.token}"/>
    ...
</form>
// For SPA (Angular, React) — CookieCsrfTokenRepository
// Spring sends XSRF-TOKEN cookie; JS reads it and sends as X-XSRF-TOKEN header
@Bean
public SecurityFilterChain filterChain(HttpSecurity http) throws Exception {
    http
        .csrf(csrf -> csrf
            .csrfTokenRepository(CookieCsrfTokenRepository.withHttpOnlyFalse())
            // withHttpOnlyFalse() makes cookie readable by JavaScript
        );
    return http.build();
}

SameSite Cookies — Modern Defense Layer

# application.yml — set SameSite on session cookie
server:
  servlet:
    session:
      cookie:
        same-site: strict  # Strict: never sent cross-site (may break OAuth redirects)
        # Lax: sent for top-level navigation GET, not for POST cross-site (safer default)
        secure: true       # HTTPS only

GotchaSameSite=Strict is NOT a CSRF replacement — it is defense-in-depth. Use SameSite=Lax for session cookies + proper CSRF tokens for state-changing requests.
Google L6 AngleAt Google scale, most internal APIs use service accounts (not cookies), so CSRF is irrelevant. For consumer-facing web apps, CSRF protection is non-negotiable. The L6 insight: CSRF is a solved problem with cookie-based CSRF token (CookieCsrfTokenRepository) for SPAs. The harder security challenges are JWT key rotation, refresh token theft (use rotation + binding to device fingerprint), and supply chain attacks via dependencies.

Why doesn't @CrossOrigin work with Spring Security? How do you configure CORS correctly?
CORS_SECURITY
▼

The Root Problem

CORS preflight requests (OPTIONS) are sent by browsers BEFORE the actual request. Spring Security's filter chain runs before DispatcherServlet. If the request reaches Spring Security without CORS headers configured at the security level, the following happens:

  1. Browser sends OPTIONS /api/orders (preflight) — no Authorization header (that's the point of preflight)
  2. Spring Security's AuthorizationFilter sees an unauthenticated request to a protected endpoint
  3. Returns 401 Unauthorized
  4. Browser sees the preflight failed → blocks the actual request entirely
  5. @CrossOrigin and WebMvcConfigurer.addCorsMappings() run at DispatcherServlet level — they are NEVER reached because Security rejected the preflight

The Fix: Configure CORS at Security Level

@Configuration
@EnableWebSecurity
public class CorsSecurityConfig {

How is SecurityContext stored and propagated? What breaks with @Async and how do you fix it?

SecurityContextHolder — Thread-Local Storage

SecurityContextHolder stores the SecurityContext for the current thread. Default strategy: MODE_THREADLOCAL — uses ThreadLocal<SecurityContext>. Available anywhere in the call stack on the same thread via SecurityContextHolder.getContext().getAuthentication().

Alternative strategies: MODE_INHERITABLETHREADLOCAL — child threads (spawned from current thread via new Thread()) inherit the SecurityContext. MODE_GLOBAL — single shared instance (not thread-safe, not useful in production).

Session-Based Context Persistence

For stateful sessions: SecurityContextRepository (implementation: HttpSessionSecurityContextRepository) saves the SecurityContext to HttpSession after each request. On the next request, SecurityContextHolderFilter loads it back from the session and populates SecurityContextHolder.

Stateless JWT — No Persistence

With SessionCreationPolicy.STATELESS, Spring configures NullSecurityContextRepository. No session is created. Each request reconstructs the SecurityContext from the JWT. The SecurityContext is only alive for the duration of a single request thread.

GotchaMD5, SHA-1, SHA-256 without salt and iterations are cryptographically broken for password storage — they are too fast. A modern GPU can compute billions of unsalted SHA-256 hashes per second, making rainbow table and brute-force attacks trivial. Never use MessageDigestPasswordEncoder with these algorithms. Always use adaptive algorithms (BCrypt, Argon2, SCrypt) that are deliberately slow.
Best PracticeUse BCrypt with strength 12 (2024 recommendation) for typical web apps. Use Argon2id for high-security applications (financial, healthcare). Know the difference: BCrypt is CPU-hard (time), Argon2 is memory-hard (space), making them complementary defenses.

When to Use PermissionEvaluator

Role-based access control (@PreAuthorize("hasRole('ADMIN')")) is coarse-grained. Object-level authorization — "can user X edit document Y?" — requires loading the domain object and checking ownership/permission state. This is where PermissionEvaluator comes in.

PermissionEvaluator Interface

public interface PermissionEvaluator {
    // Called by hasPermission(targetDomainObject, permission) in SpEL
    boolean hasPermission(Authentication auth, Object targetDomainObject, Object permission);

    // Called by hasPermission(targetId, targetType, permission) in SpEL — lazy loading
    boolean hasPermission(Authentication auth, Serializable targetId,
                          String targetType, Object permission);
}

Full Implementation

@Component
public class DocumentPermissionEvaluator implements PermissionEvaluator {

    @Autowired DocumentRepository docRepo;

    @Override
    public boolean hasPermission(Authentication auth, Object target, Object permission) {
        if (auth == null || !auth.isAuthenticated()) return false;

        if (target instanceof Document doc) {
            return checkDocumentPermission(auth, doc, permission.toString());
        }
        // Add more domain types here
        return false;
    }

    @Override // Called when @PreAuthorize uses hasPermission(#docId, 'Document', 'WRITE')
    public boolean hasPermission(Authentication auth, Serializable targetId,
                                  String targetType, Object permission) {
        if (auth == null || !auth.isAuthenticated()) return false;

        if ("Document".equals(targetType)) {
            Document doc = docRepo.findById((Long) targetId)
                .orElseThrow(() -> new EntityNotFoundException("Document not found: " + targetId));
            return checkDocumentPermission(auth, doc, permission.toString());
        }
        return false;
    }

    private boolean checkDocumentPermission(Authentication auth, Document doc, String perm) {
        Long currentUserId = getUserId(auth);

        return switch (perm.toUpperCase()) {
            case "READ" ->
                doc.isPublic()
                || doc.getOwnerId().equals(currentUserId)
                || doc.getSharedWith().contains(currentUserId)
                || hasAnyRole(auth, "ADMIN", "VIEWER");

            case "WRITE" ->
                doc.getOwnerId().equals(currentUserId)
                || (doc.isCollaborative() && doc.getCollaborators().contains(currentUserId))
                || hasRole(auth, "ADMIN");

            case "DELETE" ->
                doc.getOwnerId().equals(currentUserId)
                || hasRole(auth, "ADMIN");

            case "SHARE" ->
                doc.getOwnerId().equals(currentUserId); // only owner can share

            default -> false;
        };
    }

    private Long getUserId(Authentication auth) {
        if (auth.getPrincipal() instanceof UserPrincipal up) {
            return up.getId();
        }
        throw new IllegalStateException("Unknown principal type: " + auth.getPrincipal().getClass());
    }

    private boolean hasRole(Authentication auth, String role) {
        return auth.getAuthorities().stream()
            .anyMatch(a -> a.getAuthority().equals("ROLE_" + role));
    }

    private boolean hasAnyRole(Authentication auth, String... roles) {
        Set<String> roleSet = Arrays.stream(roles)
            .map(r -> "ROLE_" + r).collect(Collectors.toSet());
        return auth.getAuthorities().stream()
            .anyMatch(a -> roleSet.contains(a.getAuthority()));
    }
}

Session Fixation Attack

An attacker tricks a victim into using a known session ID. If the server doesn't create a new session on login, the attacker's pre-known session ID becomes authenticated after the victim logs in.

  1. Attacker visits site, obtains session ID: JSESSIONID=KNOWN_VALUE
  2. Attacker tricks victim into clicking: https://bank.com/login?JSESSIONID=KNOWN_VALUE
  3. Victim logs in — if server reuses session ID, attacker's session becomes authenticated
  4. Attacker uses their cookies (same KNOWN_VALUE) to access the bank as the victim

Configuring Content Security Policy

CSP is the most powerful defense against XSS. Not set by default (no one-size-fits-all policy). Tells the browser which sources are trusted for each content type.

@Bean
public SecurityFilterChain filterChain(HttpSecurity http) throws Exception {
    http.headers(headers -> headers
        // Content Security Policy
        .contentSecurityPolicy(csp -> csp
            .policyDirectives(
                "default-src 'self'; " +                         // default: only same origin
                "script-src 'self' https://cdn.example.com; " +  // JS: self + trusted CDN
                "style-src 'self' 'unsafe-inline'; " +           // CSS: allow inline (for now)
                "img-src 'self' data: https://*.example.com; " +  // images: self + blob + CDN
                "font-src 'self' https://fonts.gstatic.com; " +   // fonts: Google Fonts
                "connect-src 'self' https://api.example.com; " +  // XHR/fetch: self + API
                "frame-ancestors 'none'; " +                      // replaces X-Frame-Options
                "form-action 'self'; " +                          // form submissions: self only
                "upgrade-insecure-requests; " +                   // auto-upgrade HTTP to HTTPS
                "report-uri /api/csp-report"                      // violation reporting endpoint
            )
        )
        // Customize X-Frame-Options (if not using CSP frame-ancestors)
        .frameOptions(fo -> fo.sameOrigin()) // allow iframes from same origin
        // Disable for APIs that don't serve browser pages
        // .frameOptions(fo -> fo.disable())

        // HSTS — already set by default; customize if needed
        .httpStrictTransportSecurity(hsts -> hsts
            .maxAgeInSeconds(63072000) // 2 years
            .includeSubDomains(true)
            .preload(true) // opt into HSTS preload list
        )

        // Permissions Policy (formerly Feature Policy)
        .permissionsPolicy(pp -> pp
            .policy("geolocation=(), camera=(), microphone=(), payment=(self)")
        )

        // Referrer Policy
        .referrerPolicy(rp -> rp
            .policy(ReferrerPolicyHeaderWriter.ReferrerPolicy.STRICT_ORIGIN_WHEN_CROSS_ORIGIN)
        )
    );
    return http.build();
}

Nonces for Inline Scripts (Better than unsafe-inline)

// Generate a nonce per request (random, stored in request attribute)
@Component
public class CspNonceFilter extends OncePerRequestFilter {

    @Override
    protected void doFilterInternal(HttpServletRequest request,
                                     HttpServletResponse response,
                                     FilterChain filterChain) throws Exception {
        String nonce = generateNonce();
        request.setAttribute("cspNonce", nonce);

        // Dynamically set CSP header with this request's nonce
        response.setHeader("Content-Security-Policy",
            "default-src 'self'; script-src 'self' 'nonce-" + nonce + "'");

        filterChain.doFilter(request, response);
    }

    private String generateNonce() {
        byte[] bytes = new byte[16];
        new SecureRandom().nextBytes(bytes);
        return Base64.getEncoder().encodeToString(bytes);
    }
}

// <script th:nonce="${cspNonce}">...</script>
Best PracticeStart with report-only CSP, collect violations for 1-2 weeks, then switch to enforcement mode. Use nonces for inline scripts instead of 'unsafe-inline'. Set CSP on API endpoints too (even though they don't serve HTML) — defense-in-depth. HSTS with preload: submit your domain to the browser preload list (hstspreload.org) for maximum protection.
Google L6 AngleCSP is Google's primary defense against XSS at scale. Google's CSP framework uses nonces + strict-dynamic to avoid 'unsafe-inline' while maintaining compatibility. The strict-dynamic directive allows a trusted script (with a valid nonce) to load additional scripts, enabling SPAs without whitelisting every CDN URL. L6 engineers audit the CSP header as part of security reviews — overly permissive CSP (unsafe-inline, wildcard origins) is a red flag.

    @Bean
    public Converter<Jwt, Mono<AbstractAuthenticationToken>> reactiveJwtConverter() {

        authoritiesConverter.setAuthoritiesClaimName("roles");
        authoritiesConverter.setAuthorityPrefix("ROLE_");

                new JwtAuthenticationConverter() {{

Comprehensive Audit Listener

@Component
@Slf4j
public class SecurityAuditListener {

    @Autowired AuditEventRepository auditRepo; // Spring Boot Actuator's audit repository
    @Autowired AuditLogRepository customAuditRepo; // custom for structured storage

    @EventListener
    public void onAuthenticationSuccess(AuthenticationSuccessEvent event) {
        Authentication auth = event.getAuthentication();
        AuditLogEntry entry = AuditLogEntry.builder()
            .eventType("LOGIN_SUCCESS")
            .username(auth.getName())
            .ipAddress(getClientIp(event))
            .userAgent(getUserAgent(event))
            .timestamp(Instant.now())
            .details("Authenticated via: " + auth.getClass().getSimpleName())
            .outcome("SUCCESS")
            .build();

        customAuditRepo.save(entry);
        log.info("LOGIN_SUCCESS user={} ip={}", auth.getName(), entry.getIpAddress());
    }

    @EventListener
    public void onAuthenticationFailure(AbstractAuthenticationFailureEvent event) {
        String username = event.getAuthentication().getName();
        String reason = event.getException().getClass().getSimpleName();

        customAuditRepo.save(AuditLogEntry.builder()
            .eventType("LOGIN_FAILURE")
            .username(username)
            .ipAddress(getClientIp(event))
            .timestamp(Instant.now())
            .details("Reason: " + reason + " - " + event.getException().getMessage())
            .outcome("FAILURE")
            .build());

        log.warn("LOGIN_FAILURE user={} reason={} ip={}", username, reason, getClientIp(event));

        // Alert on repeated failures
        failureCounter.increment(username);
        if (failureCounter.getCount(username) >= 5) {
            alertService.notify("Brute force detected for user: " + username);
        }
    }

    @EventListener
    public void onAuthorizationDenied(AuthorizationDeniedEvent<?> event) {
        Authentication auth = event.getAuthentication().get();
        String resource = event.getAuthorizationResult().toString();

        customAuditRepo.save(AuditLogEntry.builder()
            .eventType("ACCESS_DENIED")
            .username(auth != null ? auth.getName() : "anonymous")
            .timestamp(Instant.now())
            .details("Resource: " + resource)
            .outcome("DENIED")
            .build());

        log.warn("ACCESS_DENIED user={} resource={}", auth != null ? auth.getName() : "anon", resource);
    }

    @EventListener
    public void onLogout(LogoutSuccessEvent event) {
        log.info("LOGOUT user={}", event.getAuthentication().getName());
        customAuditRepo.save(AuditLogEntry.builder()
            .eventType("LOGOUT")
            .username(event.getAuthentication().getName())
            .timestamp(Instant.now())
            .outcome("SUCCESS")
            .build());
    }

    private String getClientIp(ApplicationEvent event) {
        // Access request context — Spring Security events carry the web request context
        HttpServletRequest request = ((ServletRequestAttributes)
            RequestContextHolder.getRequestAttributes()).getRequest();
        String xForwardedFor = request.getHeader("X-Forwarded-For");
        return xForwardedFor != null ? xForwardedFor.split(",")[0].trim()
                                     : request.getRemoteAddr();
    }

    private String getUserAgent(ApplicationEvent event) {
        HttpServletRequest request = ((ServletRequestAttributes)
            RequestContextHolder.getRequestAttributes()).getRequest();
        return request.getHeader("User-Agent");
    }
}
