1
Explain Spring's cache abstraction. What is CacheManager and what backends does it support?
CACHE_ABSTRACTION
▼

Core Concept

Spring's cache abstraction provides a unified API for caching via annotations, decoupling business code from the underlying cache implementation. The four main annotations are @Cacheable (read from cache or compute and store), @CacheEvict (remove entries), @CachePut (always execute method and update cache), and @Caching (compose multiple cache operations). @CacheConfig sets class-level defaults.

@EnableCaching on a @Configuration class activates the mechanism — it registers a CacheInterceptor AOP proxy that intercepts annotated methods.

CacheManager is the central interface that manages named Cache instances. Supported backends:

  • ConcurrentMapCacheManager — in-memory, unbounded, dev/test only
  • CaffeineCacheManager — Caffeine (W-TinyLFU eviction, high-performance local cache)
  • RedisCacheManager — distributed cache via Redis (spring-data-redis)
  • EhCacheCacheManager — EhCache 2.x (legacy)
  • JCacheCacheManager — JSR-107 compliant providers (EhCache 3, Hazelcast, Infinispan)
  • SimpleCacheManager — configure arbitrary Cache instances manually
  • CompositeCacheManager — delegate to multiple CacheManagers in order
@Configuration
@EnableCaching
public class CacheConfig {
    // Spring Boot auto-configures CacheManager — just annotate methods
}

// Service layer
@Service
public class UserService {

    @Cacheable(value = "users", key = "#id")
    public User findById(Long id) {
        return userRepo.findById(id).orElseThrow();
    }

    @CacheEvict(value = "users", key = "#user.id")
    public User updateUser(User user) {
        return userRepo.save(user);
    }

    @CachePut(value = "users", key = "#result.id")
    public User createUser(CreateUserDTO dto) {
        return userRepo.save(new User(dto));
    }

    // Multiple operations on one method
    @Caching(evict = {
        @CacheEvict(value = "users", key = "#id"),
        @CacheEvict(value = "userList", allEntries = true)
    })
    public void deleteUser(Long id) {
        userRepo.deleteById(id);
    }
}

// Class-level defaults
@CacheConfig(cacheNames = "users")
@Service
public class UserService2 {
    @Cacheable(key = "#id")  // inherits cacheNames="users"
    public User findById(Long id) { ... }
}
⚠ GotchaSame self-invocation limitation as @Transactional — calling a @Cacheable method from within the same class bypasses the AOP proxy entirely. The cache is never populated. Extract to a separate Spring bean or use ApplicationContext.getBean() (inelegant) to force proxy invocation.
✅ Best PracticeUse @CacheConfig at the class level to avoid repeating the cache name on every method. Use @CachePut on create/update methods to keep the cache warm rather than forcing the next read to be a cache miss.
🎯 Google L6 AngleAt scale, @CacheEvict(allEntries=true) is a hammer — it clears the entire cache namespace at once. For a user-service with millions of entries, a bulk evict on every list-modifying operation causes massive cache miss storms. Design fine-grained eviction with specific keys. If you need to invalidate a collection view, consider versioned cache keys (e.g., include a counter in the key) rather than allEntries=true.

How do you configure Redis as a Spring Cache backend? Show serialization, TTL, and namespace config.
REDIS_CACHE
▼

Dependency and Auto-Configuration

Add spring-boot-starter-data-redis. Spring Boot auto-configures a RedisCacheManager when it detects the dependency and @EnableCaching. For production control, define a RedisCacheManager bean explicitly.

@Bean
public RedisCacheManager cacheManager(RedisConnectionFactory factory) {
    RedisCacheConfiguration defaultConfig = RedisCacheConfiguration.defaultCacheConfig()
        .entryTtl(Duration.ofMinutes(30))
        .serializeValuesWith(RedisSerializationContext.SerializationPair
            .fromSerializer(new GenericJackson2JsonRedisSerializer()))
        .prefixCacheNameWith("myapp::")   // namespace — avoids key collisions between services
        .disableCachingNullValues();       // don't store nulls (saves memory, can re-query DB)

    Map<String, RedisCacheConfiguration> cacheConfigs = Map.of(
        "users",    defaultConfig.entryTtl(Duration.ofHours(1)),
        "products", defaultConfig.entryTtl(Duration.ofMinutes(5)),
        "sessions", defaultConfig.entryTtl(Duration.ofDays(1))
    );

    return RedisCacheManager.builder(factory)
        .cacheDefaults(defaultConfig)
        .withInitialCacheConfigurations(cacheConfigs)
        .transactionAware()   // only cache after TX commit
        .build();
}

Key Format and Namespace

With prefixCacheNameWith("myapp::"), a cache named users with key 42 produces Redis key: myapp::users::42. Namespacing prevents collision when multiple services share a Redis instance.

Redis Cluster Configuration

# application.yml
spring:
  data:
    redis:
      cluster:
        nodes: redis-node1:6379,redis-node2:6379,redis-node3:6379
      lettuce:

            period: 30s
⚠ GotchaGenericJackson2JsonRedisSerializer embeds a @class field in every JSON value. If you rename or move a class, all existing cache entries become undeserializable — reads will throw exceptions. Always handle deserialization errors gracefully with a fallback to the DB, especially during rolling deployments.
✅ Best PracticeUse disableCachingNullValues() in most cases. If you need negative caching (cache "not found" to avoid repeated DB lookups), create an explicit Optional wrapper or sentinel object rather than storing null — null handling across serialization formats is fragile.
🎯 Google L6 AnglePer-cache TTL configuration is critical at scale. Static reference data (countries, currencies) should have TTL of hours or days. User-session data: minutes. Price data: seconds. A single default TTL for all caches is a design smell. Also: transactionAware() ensures you don't cache data from a rolled-back transaction — always enable it when using @Transactional alongside @Cacheable.

How does Spring generate cache keys? When do you need a custom KeyGenerator?
KEY_GENERATION
▼

Default Key Generation — SimpleKeyGenerator

Spring's SimpleKeyGenerator generates keys as follows:

  • No parameters → SimpleKey.EMPTY
  • One parameter → the parameter itself (e.g., 42L)
  • Multiple parameters → new SimpleKey(param1, param2, ...)

SpEL Key Expressions

// Simple field access
@Cacheable(value = "users", key = "#id")
public User findById(Long id) { ... }

// Nested property
@Cacheable(value = "users", key = "#user.id")
public User updateUser(User user) { ... }

// Concatenation for composite keys
@Cacheable(value = "reports", key = "#tenantId + ':' + #reportType + ':' + #date")
public Report getReport(String tenantId, String reportType, LocalDate date) { ... }

// Method name for namespace disambiguation
@Cacheable(value = "search", key = "#root.method.name + ':' + #query")
public List<Product> search(String query) { ... }

// Static factory method for complex key logic
@Cacheable(value = "users", key = "T(com.example.CacheKeys).forUser(#userId, #tenantId)")
public User getUser(Long userId, String tenantId) { ... }

// Result-based key in @CachePut
@CachePut(value = "users", key = "#result.id")
public User createUser(CreateUserDTO dto) { ... }

Custom KeyGenerator

Implement KeyGenerator for cross-cutting key logic: tenant-awareness, version-stamping, or method-level namespacing.

@Bean
public KeyGenerator tenantAwareKeyGenerator() {
    return (target, method, params) -> {
        String tenantId = TenantContext.getCurrentTenant();  // ThreadLocal
        return tenantId + ":" + method.getName() + ":" + Arrays.toString(params);
    };
}

// Usage
@Cacheable(value = "users", keyGenerator = "tenantAwareKeyGenerator")
public User findById(Long id) { ... }

// Version-aware key: invalidates all cache on deploy
@Bean
public KeyGenerator versionedKeyGenerator(@Value("${app.version}") String version) {
    return (target, method, params) ->
        version + ":" + method.getName() + ":" + Arrays.toString(params);
}
⚠ GotchaThe default key for a no-parameter method is SimpleKey.EMPTY — a shared singleton. If two different no-param @Cacheable methods use the same cache name without an explicit key, they collide and corrupt each other's results. Always specify key="#root.method.name" or a unique literal for no-arg methods.
✅ Best PracticePrefer explicit SpEL key expressions over custom KeyGenerators when the key logic is method-specific. Use KeyGenerator only for truly cross-cutting concerns (tenant isolation, version stamping) where every method in a component needs the same key derivation strategy.
🎯 Google L6 AngleIn multi-tenant SaaS systems, tenant isolation in cache keys is a security boundary, not just a performance concern. A bug where tenant A reads tenant B's cached data is a data breach. Implement a KeyGenerator that enforces tenant prefixing centrally and test it with a multi-tenant integration test — don't rely on per-method key expressions that developers might forget.

Compare cache-aside, read-through, write-through, and write-behind caching strategies.
CACHE_STRATEGIES
▼

Cache-Aside (Lazy Loading)

The application checks the cache first. On a miss, it loads from the DB, populates the cache, and returns the result. On a write, it updates the DB and evicts or updates the cache entry. Spring @Cacheable implements this pattern. The application owns all caching logic.

Pros: cache only contains data actually requested. Resilient to cache failures (falls back to DB). Cons: first request always misses (cold start). Window of inconsistency between DB write and cache eviction.

Read-Through

The cache sits in front of the DB. On a miss, the cache itself fetches from the DB — the application only talks to the cache. Less common in Spring (requires a cache provider that supports it natively, e.g., Redis with a loader callback or Caffeine's CacheLoader).

// Caffeine read-through via CacheLoader
LoadingCache<Long, User> cache = Caffeine.newBuilder()
    .maximumSize(10_000)
    .expireAfterWrite(Duration.ofMinutes(10))
    .build(userId -> userRepo.findById(userId).orElseThrow());
// cache.get(id) triggers load automatically on miss

Write-Through

On every write, update BOTH the cache and the DB synchronously before returning. Always consistent — the cache is never stale. Higher write latency (two writes). In Spring, combine @CachePut + repository.save() in the same @Transactional method.

Write-Behind (Write-Back)

Write to the cache immediately, then flush to the DB asynchronously in the background. Lowest write latency. Risk: data loss if the cache process crashes before the DB flush. Complex to implement correctly. Rarely the right choice for business-critical data.

Refresh-Ahead

Proactively refresh cache entries before their TTL expires (Caffeine's refreshAfterWrite). The stale entry is served while a background thread refreshes it. Prevents cold miss latency spikes.

StrategyRead LatencyWrite LatencyConsistencySpring Support
Cache-AsideMiss penaltyNormalEventual@Cacheable / @CacheEvict
Read-ThroughMiss penaltyNormalEventualCaffeine CacheLoader
Write-ThroughFast (warm)2x writeStrong@CachePut + save()
Write-BehindFast (warm)FastestWeak (async)Custom / Caffeine writer

⚠ GotchaWrite-through with @CachePut and @Transactional requires careful ordering. If you use RedisCacheManager with transactionAware(true), the cache write happens only after the TX commits — which is correct. Without transactionAware, a rolled-back DB write can still pollute the cache with incorrect data.
✅ Best PracticeCache-aside is the right default for most Spring applications. It's explicit, easy to reason about, and degrades gracefully when the cache is unavailable. Only move to write-through if you have strong consistency requirements and can accept the write latency.
🎯 Google L6 AngleFor read-heavy workloads with infrequent updates (product catalogs, user profiles), cache-aside with Caffeine's refreshAfterWrite is the highest-performance pattern — reads never block, stale-while-revalidate handles freshness. For financial data (balances, prices) where stale reads cause real harm, skip the cache or use write-through with very short TTL as a safety net.

Explain condition and unless in @Cacheable. When are they evaluated?
CONDITION_UNLESS
▼

condition vs unless — The Key Difference

condition: SpEL evaluated before the method executes, with access to method parameters. If the expression is false, the method runs normally — no cache lookup, no cache store. The cache is completely bypassed.

unless: SpEL evaluated after the method executes, with access to #result. The cache lookup still happens (a cached value will be returned if present), but if the expression is true, the result is NOT stored in the cache.

// condition: skip cache for anonymous users (id = 0)
@Cacheable(value = "users", key = "#userId", condition = "#userId > 0")
public User getUser(Long userId) { ... }

// unless: don't store null results (avoid caching "not found")
@Cacheable(value = "users", key = "#email", unless = "#result == null")
public User findByEmail(String email) { ... }

// unless: don't cache error/fallback states
@Cacheable(value = "prices", key = "#sku", unless = "#result.isError()")
public PriceResult getPrice(String sku) { ... }

// condition: only cache large result sets (worth caching)
@Cacheable(value = "search", key = "#query", condition = "#query.length() > 3")
public List<Product> search(String query) { ... }

// Combining both
@Cacheable(
    value = "recommendations",
    key = "#userId",
    condition = "#userId != null",
    unless = "#result.isEmpty()"
)
public List<Product> getRecommendations(Long userId) { ... }

sync=true — Cache Stampede Prevention

// Only one thread computes the value on cache miss; others wait
@Cacheable(value = "expensive", key = "#id", sync = true)
public ExpensiveResult computeExpensive(Long id) {
    // expensive DB query or external API call
    return expensiveService.compute(id);
}

sync=true uses the cache provider's native locking to ensure only one thread fetches the value on a miss. Supported by Caffeine natively; with Redis, Spring uses a lightweight lock. Critical for preventing cache stampedes on expensive operations.

⚠ Gotchacondition cannot reference #result — it's evaluated before the method runs. Attempting to use #result in condition will throw a SpEL evaluation error at runtime. Use unless for post-execution filtering based on the return value.
✅ Best PracticeUse unless="#result == null" as a default on most @Cacheable methods to avoid caching null results. Null caching can mask bugs (a missing record looks like a cache hit forever until the TTL expires) and wastes Redis memory. If you need negative caching explicitly, use a typed Optional or a dedicated sentinel object.
🎯 Google L6 Anglesync=true is the simplest stampede prevention and should be the default for any @Cacheable method with significant computation behind it. The risk of NOT using sync=true on a popular endpoint with a 60-second TTL: at 10,000 req/s, the moment the TTL expires you get 10,000 simultaneous DB queries. That's a guaranteed incident. Always ask: "what happens the moment this cache entry expires?"

What is a cache stampede (thundering herd)? How do you prevent it in Spring?
CACHE_STAMPEDE
▼

The Problem

A cache stampede (also called a thundering herd) occurs when a popular cache entry expires and many concurrent requests simultaneously experience a cache miss. All threads then attempt to recompute or re-fetch the value from the database at the same moment, potentially overwhelming the DB and causing cascading failures.

Magnitude example: 1,000 req/s hit user record #42. TTL = 60 seconds. At the moment of expiry, all 1,000 in-flight requests miss simultaneously, producing 1,000 concurrent DB queries for a single row. The DB connection pool exhausts. Other queries timeout. The system cascades.

Prevention Strategy 1: sync=true (Built-in Spring)

@Cacheable(value = "users", key = "#id", sync = true)
public User findById(Long id) {
    return userRepo.findById(id).orElseThrow();
}
// On cache miss: only ONE thread fetches from DB.
// All other threads wait on the cache lock and receive the result.

Prevention Strategy 2: TTL Jitter

// Assign random TTL so entries don't all expire simultaneously
// e.g., TTL = base + random(0 to 20% of base)
@Bean
public RedisCacheManager cacheManager(RedisConnectionFactory factory) {
    Duration base = Duration.ofMinutes(60);
    Duration jitter = Duration.ofMinutes(ThreadLocalRandom.current().nextInt(0, 12));
    RedisCacheConfiguration config = RedisCacheConfiguration.defaultCacheConfig()
        .entryTtl(base.plus(jitter));
    return RedisCacheManager.builder(factory).cacheDefaults(config).build();
}

Prevention Strategy 3: Background Refresh (Caffeine refreshAfterWrite)

Caffeine.newBuilder()
    .maximumSize(10_000)
    .expireAfterWrite(Duration.ofMinutes(10))   // hard expiry
    .refreshAfterWrite(Duration.ofMinutes(8))   // async refresh before expiry
    .build(key -> expensiveLoad(key));           // CacheLoader required
// At 8 min: async background thread refreshes value.
// Requests between 8-10 min receive stale-but-valid data instantly.

Prevention Strategy 4: Probabilistic Early Expiration (XFetch)

Randomly trigger a refresh before the TTL expires, with probability increasing as the entry nears expiry. Prevents synchronized expiry without a central lock. More complex to implement; useful for very high-traffic entries.

Prevention Strategy 5: Stale-While-Revalidate

Serve the stale (expired) entry immediately while a single background thread refreshes it. The client never sees a miss. Requires cache implementation support or manual implementation with two TTLs (soft TTL for refresh trigger, hard TTL for absolute expiry).

⚠ Gotchasync=true with RedisCacheManager uses a pessimistic lock — all waiting threads block until the single loader thread completes. If the DB query is slow (e.g., 2 seconds), threads pile up waiting. Set an appropriate lock timeout and ensure the guarded method's execution time is bounded.
✅ Best PracticeLayer your strategies: sync=true as a baseline for all @Cacheable methods, plus TTL jitter for Redis to prevent synchronized expiry across the cluster, plus Caffeine refreshAfterWrite for the hottest local caches. Defense in depth against stampedes.
🎯 Google L6 AngleFor high-traffic services, the background refresh + stale-while-revalidate pattern is the production gold standard. Users never see latency from cache misses — the cache is always warm. The trade-off is bounded staleness (8–10 minutes in the Caffeine example), which is acceptable for most read-heavy data. For any data serving >1,000 req/s, this pattern should be the default, not an optimization.

How do you handle cache invalidation in a multi-node deployment? What are the consistency models?
DISTRIBUTED_CONSISTENCY
▼

The Problem

In a multi-node deployment (3+ app instances), a local in-memory cache (ConcurrentMap, Caffeine) is per-node. When node A updates a user and evicts its local cache entry, nodes B and C still have stale data. Node B may serve the old user for the duration of the TTL — or forever if there is no TTL.

Solution 1: Shared Remote Cache (Redis)

All nodes share a single Redis instance (or cluster). No local in-memory caching. Every cache read/write goes through Redis (~0.5–1ms network hop). Guarantees consistency — a @CacheEvict on any node removes the entry from the shared store. Simplest approach.

Drawback: every cache hit requires a network call. With 10,000 req/s and 1ms Redis latency, that's 10,000 network round-trips per second — acceptable but adds up.

Solution 2: Cache Invalidation Broadcast (Pub/Sub)

// Custom CacheManager that broadcasts invalidations via Redis Pub/Sub
@Component
public class BroadcastingCacheEvictListener {
    @Autowired private CacheManager localCacheManager;

    @EventListener
    public void onCacheEvict(CacheEvictedEvent event) {
        redisTemplate.convertAndSend("cache-invalidations",
            event.getCacheName() + ":" + event.getKey());
    }
}

@Component
public class CacheInvalidationSubscriber implements MessageListener {
    @Override
    public void onMessage(Message message, byte[] pattern) {
        String[] parts = message.toString().split(":", 2);
        Cache cache = localCacheManager.getCache(parts[0]);
        if (cache != null) cache.evict(parts[1]);
    }
}
// All nodes subscribe: when any node evicts, all nodes evict their local copy.

Solution 3: Short TTL Local Cache (Accept Bounded Staleness)

Use Caffeine with a very short TTL (10–30 seconds). Accept that any node may serve data up to TTL seconds stale. No invalidation broadcast needed. Simple. For data where bounded staleness is acceptable (product listings, public content, rate limits).

Solution 4: Two-Level Cache (L1 + L2)

Caffeine as L1 (hot reads, microsecond latency) + Redis as L2 (distributed, longer TTL). L1 hit → return immediately. L1 miss → check L2 → populate L1. On invalidation: evict from L2 (centralized) + broadcast to evict L1 on all nodes. Highest performance, highest complexity.

Redis Pub/Sub BroadcastMicroseconds (local hit)Near-strongMediumHigh-read, consistency-critical
Short TTL LocalMicrosecondsBounded staleVery lowPublic/read-heavy, tolerate staleness

⚠ GotchaSpring Cache does NOT handle multi-node local cache invalidation by default. If you deploy a service with ConcurrentMapCacheManager or CaffeineCacheManager behind a load balancer and never add an invalidation mechanism — you will have persistent inconsistency in production. This is a silent bug that's very hard to reproduce in single-node development environments.
✅ Best Practice"Cache invalidation is one of the two hard things in computer science." Keep it simple: use Redis (shared remote cache) as the default for any horizontally-scaled service. Only introduce local caching with an invalidation bus when you have measured Redis latency as a bottleneck.
🎯 Google L6 AngleAt Google scale, the two-level cache (Caffeine L1 + distributed L2) with a fanout invalidation bus is common for extremely hot read paths. But the complexity cost is real: you're now operating a distributed system within your caching layer. The invalidation bus itself becomes a point of failure. Always ask: "what is the blast radius if the invalidation message is lost?" Design for at-least-once invalidation and make your application tolerant of brief staleness rather than brittle in the face of invalidation failure.

Compare Caffeine (local) vs Redis (distributed) cache. When do you use each?
CAFFEINE
▼

Caffeine — Local In-Process Cache

  • Sub-microsecond latency (no network, no serialization)
  • W-TinyLFU eviction — near-optimal hit rate for skewed access patterns
  • Bounded by JVM heap — careful not to cause GC pressure
  • Per-node: each instance has its own independent cache
  • refreshAfterWrite: async background refresh (stale-while-revalidate)
  • expireAfterWrite vs expireAfterAccess: different staleness models
  • recordStats(): built-in hit rate, miss rate, load success metrics
@Bean
CaffeineCacheManager cacheManager() {
    CaffeineCacheManager mgr = new CaffeineCacheManager();
    mgr.setCaffeine(Caffeine.newBuilder()
        .maximumSize(10_000)
        .expireAfterWrite(Duration.ofMinutes(10))
        .refreshAfterWrite(Duration.ofMinutes(8))  // async refresh, requires CacheLoader
        .recordStats()                              // expose via Micrometer
        .removalListener((key, value, cause) ->
            log.debug("Evicted {}: {}", key, cause)));
    return mgr;
}

// Wire Micrometer metrics for Caffeine
@Bean
public CacheMetricsRegistrar cacheMetricsRegistrar(MeterRegistry registry,
                                                    CacheManager cacheManager) {
    return new CacheMetricsRegistrar(registry, List.of(cacheManager));
}

Redis — Distributed Shared Cache

  • ~0.5–1ms latency per operation (network + serialization)
  • Shared across ALL service nodes — inherently consistent
  • Not limited by JVM heap — can cache gigabytes
  • Supports TTL, key expiry notifications, LRU eviction policies
  • Cluster mode for horizontal scaling and high availability
  • Requires serialization — class evolution must be managed carefully

read:write Ratio Rule of Thumb

Cache ROI is proportional to the read:write ratio. A 100:1 ratio with low update frequency is ideal for Caffeine. A 10:1 ratio with high writes — reconsider whether caching helps at all, or switch to Redis for correctness.

⚠ GotchaCaffeine's refreshAfterWrite requires a CacheLoader (or AsyncCacheLoader) — it cannot be used with Spring's annotation-driven caching (@Cacheable) alone because Spring's abstraction does not expose the CacheLoader interface. You must either configure a native Caffeine LoadingCache or use a scheduler to pre-populate the cache. Don't configure refreshAfterWrite without a loader — it silently has no effect.
✅ Best PracticeAlways enable Caffeine's recordStats() and expose hit rates via Micrometer. A cache with a <70% hit rate is often not worth its complexity — the miss penalty (DB round-trip) + overhead may exceed the benefit. Monitor hit rate as a primary cache health metric and alert when it drops.
🎯 Google L6 AngleThe two-level cache (Caffeine L1 + Redis L2) is the right architecture for services handling >50,000 req/s on hot data. L1 absorbs 95%+ of reads at microsecond latency; L2 provides shared consistency and larger capacity. The key engineering challenge is making L1 invalidation reliable under failure scenarios — use a Redis Pub/Sub invalidation channel with at-least-once delivery and idempotent cache eviction. Budget significant engineering time for this: it is not a trivial add-on.
