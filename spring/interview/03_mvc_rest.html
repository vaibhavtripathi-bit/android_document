Walk through the complete Spring MVC request lifecycle from HTTP request to HTTP response.
DISPATCHER_SERVLET
▼

Overview

Every HTTP request processed by Spring MVC passes through a well-defined pipeline inside DispatcherServlet. Understanding this pipeline is fundamental to diagnosing issues and extending Spring MVC behavior.

Step-by-Step Pipeline

  1. Servlet container receives request — Routes to DispatcherServlet.service() which delegates to doDispatch().
  2. HandlerMapping lookup — Iterates registered HandlerMapping beans (ordered). RequestMappingHandlerMapping has highest priority and handles @RequestMapping/@GetMapping etc. Returns a HandlerExecutionChain containing the handler and zero or more HandlerInterceptors.
  3. HandlerAdapter lookup — Finds a HandlerAdapter that supports the resolved handler. RequestMappingHandlerAdapter handles @Controller methods.
  4. Pre-handle interceptors — Each HandlerInterceptor.preHandle() is called in order. If any returns false, the pipeline is short-circuited — no handler is invoked, no post-handle calls. afterCompletion IS still called for interceptors whose preHandle returned true.
  5. Handler invocation — HandlerAdapter.handle(): resolves method arguments (@PathVariable, @RequestBody, @RequestParam), invokes the controller method via reflection, processes return value into ModelAndView or writes to response directly (ResponseEntity, @ResponseBody).
  6. Post-handle interceptors — HandlerInterceptor.postHandle() called in reverse order. NOT called if handler threw exception.
  7. View resolution — If ModelAndView returned (non-null): ViewResolver.resolveViewName() → View.render(model, request, response). REST controllers writing directly skip this step.
  8. afterCompletion — Called in reverse order always (even on exception). Used for resource cleanup.
  9. Exception handling — Any exception from steps 2-7 triggers processHandlerException() which iterates HandlerExceptionResolver chain. ExceptionHandlerExceptionResolver handles @ExceptionHandler and @ControllerAdvice.
// Custom HandlerInterceptor — audit logging per endpoint
@Component
public class AuditInterceptor implements HandlerInterceptor {

    private static final String START_TIME_ATTR = "startTime";

    @Override
    public boolean preHandle(HttpServletRequest req,
                             HttpServletResponse res,
                             Object handler) {
        req.setAttribute(START_TIME_ATTR, System.currentTimeMillis());
        if (handler instanceof HandlerMethod hm) {
            log.info("Invoking {}.{}",
                hm.getBeanType().getSimpleName(),
                hm.getMethod().getName());
        }
        return true; // continue pipeline
    }

    @Override
    public void postHandle(HttpServletRequest req, HttpServletResponse res,
                           Object handler, ModelAndView mav) {
        // Called after handler, before view render
        // NOT called if handler threw an exception
    }

    @Override
    public void afterCompletion(HttpServletRequest req, HttpServletResponse res,
                                Object handler, Exception ex) {
        long elapsed = System.currentTimeMillis()
                       - (Long) req.getAttribute(START_TIME_ATTR);
        log.info("Request completed in {}ms, status={}", elapsed, res.getStatus());
        if (ex != null) log.warn("Handler threw exception", ex);
    }
}

// Register via WebMvcConfigurer
@Configuration
public class WebConfig implements WebMvcConfigurer {
    @Autowired AuditInterceptor auditInterceptor;

    @Override
    public void addInterceptors(InterceptorRegistry registry) {
        registry.addInterceptor(auditInterceptor)
                .addPathPatterns("/api/**")
                .excludePathPatterns("/api/health");
    }
}
⚠ GotchapostHandle is NOT called when an exception escapes the handler. If you need cleanup that always runs, use afterCompletion. Also: interceptors registered as Spring beans must be injected — do NOT call new MyInterceptor() in addInterceptors or you lose dependency injection.
✅ Best PracticeKeep interceptors stateless (no instance state). Use request attributes (req.setAttribute) to pass data between preHandle and afterCompletion.
🎯 Google L6 AngleAt L6 depth, trace which exact class calls what: DispatcherServlet.doDispatch() calls getHandler() → iterates handlerMappings list → returns first non-null result. The ordering of HandlerMapping beans (via Ordered/@Order) determines which mapping wins for ambiguous URLs. This is a source of subtle bugs when mixing @RequestMapping and functional routes.

RequestMappingHandlerMapping Internals

At application startup: scans all @Controller/@RequestMapping beans. For each annotated method, builds a RequestMappingInfo capturing: URL patterns, HTTP methods, params, headers, produces, consumes. Stores in a map RequestMappingInfo → HandlerMethod.

At request time: calls lookupHandlerMethod(). Finds all RequestMappingInfo entries that match the request. If multiple match, picks the most specific one using a comparator (concrete pattern beats wildcard, specific HTTP method beats none, etc.). Ambiguous matches at equal specificity → throws IllegalStateException at startup (not runtime).

Path Matching Evolution

Spring 5.3+ / Boot 3: uses PathPatternParser by default instead of AntPathMatcher. Patterns are parsed once at startup into a PathPattern tree structure and matched at O(n) per request instead of regex re-evaluation. 6-8x faster for complex patterns at high throughput.

What does HandlerAdapter do? Why is it needed?
HANDLER_ADAPTER
▼

Why HandlerAdapter Exists

DispatcherServlet doesn't know (and shouldn't know) how to invoke a handler. Handlers can be: annotated @Controller methods, HttpRequestHandler implementations, plain Servlet objects, or functional route handlers. HandlerAdapter is the Strategy pattern abstraction that decouples invocation from dispatch.

Contract: HandlerAdapter.supports(Object handler) — does this adapter handle this type? HandlerAdapter.handle(request, response, handler) — invoke and return ModelAndView.

RequestMappingHandlerAdapter (The Important One)

Handles HandlerMethod objects (the resolved @Controller method + bean). Does heavy lifting in three phases:

  1. Argument resolution — iterates HandlerMethodArgumentResolver chain for each parameter. First resolver whose supportsParameter() returns true wins.
  2. Method invocation — via InvocableHandlerMethod.invokeForRequest() using reflection.
  3. Return value handling — iterates HandlerMethodReturnValueHandler chain. Writes response or populates ModelAndView.

// Custom argument resolver — inject pagination settings from query params
public class PaginationArgumentResolver implements HandlerMethodArgumentResolver {

    @Override
    public boolean supportsParameter(MethodParameter parameter) {
        return parameter.getParameterType().equals(Pagination.class);
    }

    @Override
    public Object resolveArgument(MethodParameter parameter,
                                  ModelAndViewContainer mavContainer,
                                  NativeWebRequest webRequest,
                                  WebDataBinderFactory binderFactory) {
        String pageStr = webRequest.getParameter("page");
        String sizeStr = webRequest.getParameter("size");
        int page = pageStr != null ? Integer.parseInt(pageStr) : 0;
        int size = sizeStr != null ? Math.min(Integer.parseInt(sizeStr), 100) : 20;
        return new Pagination(page, size);
    }
}

// Register
@Configuration
public class WebConfig implements WebMvcConfigurer {
    @Override
    public void addArgumentResolvers(List<HandlerMethodArgumentResolver> resolvers) {
        resolvers.add(new PaginationArgumentResolver());
    }
}

// Usage in controller
@GetMapping("/users")
public Page<User> listUsers(Pagination pagination) { // No annotation needed!
    return userService.findAll(pagination.toPageable());
}
⚠ GotchaCustom argument resolvers added via addArgumentResolvers() are appended AFTER all built-in resolvers. If you need your resolver to run before built-ins (e.g., override @RequestParam behavior), you must replace the entire resolver list via RequestMappingHandlerAdapter.setArgumentResolvers() — which means you lose auto-configured resolvers unless you re-add them.
🎯 Google L6 AngleThe resolver chain is evaluated linearly for each parameter — N resolvers × M parameters per request. For hot paths with many parameters, consider the ordering of custom resolvers. If your resolver always returns false for most parameters, it's still O(N) overhead per request. Place unlikely resolvers at the end.

How does @RequestBody and @ResponseBody work? Explain HttpMessageConverter.
MESSAGE_CONVERTERS
▼

The Conversion Pipeline

@RequestBody and @ResponseBody both delegate to RequestResponseBodyMethodProcessor, which uses an ordered list of HttpMessageConverters to read/write the HTTP body.

@RequestBody — Reading

  1. Processor reads the Content-Type header (e.g., application/json).
  2. Iterates converters; finds first where canRead(targetClass, mediaType) returns true.
  3. Calls converter.read(targetClass, inputMessage) — deserializes body to Java object.
  4. If no converter matches: HttpMediaTypeNotSupportedException → 415 Unsupported Media Type.

@ResponseBody — Writing

  1. Processor performs content negotiation (see Q5) to determine acceptable media types.
  2. Iterates converters; finds first where canWrite(returnType, selectedMediaType) returns true.
  3. Calls converter.write(body, selectedMediaType, outputMessage).
  4. If no converter can produce any acceptable type: 406 Not Acceptable.

// Custom MessageConverter — handle text/csv input/output
@Component
public class CsvMessageConverter extends AbstractHttpMessageConverter<List<String[]>> {

    public CsvMessageConverter() {
        super(new MediaType("text", "csv"));
    }

    @Override
    protected boolean supports(Class<?> clazz) {
        return List.class.isAssignableFrom(clazz);
    }

    @Override
    protected List<String[]> readInternal(Class<? extends List<String[]>> clazz,
                                           HttpInputMessage inputMessage) throws IOException {
        try (CSVReader reader = new CSVReader(
                new InputStreamReader(inputMessage.getBody()))) {
            return reader.readAll();
        }
    }

    @Override
    protected void writeInternal(List<String[]> data,
                                  HttpOutputMessage outputMessage) throws IOException {
        try (CSVWriter writer = new CSVWriter(
                new OutputStreamWriter(outputMessage.getBody()))) {
            writer.writeAll(data);
        }
    }
}

// Register — use extendMessageConverters to KEEP existing converters
@Configuration
public class WebConfig implements WebMvcConfigurer {
    @Override
    public void extendMessageConverters(List<HttpMessageConverter<?>> converters) {
        converters.add(new CsvMessageConverter());
    }
    // configureMessageConverters() REPLACES all converters — usually wrong!
}
⚠ GotchaconfigureMessageConverters() replaces ALL default converters including Jackson. Use extendMessageConverters() to add converters while keeping defaults. Also: StringHttpMessageConverter has */* as a supported type and is registered BEFORE Jackson — for String return types, it will always win over Jackson even if Accept is application/json. This is usually correct behavior but can surprise you.
✅ Best PracticeWhen adding a custom converter that should take priority over defaults (e.g., for a custom vendor media type), insert it at position 0 of the converters list in extendMessageConverters().
🎯 Google L6 AngleIn high-throughput services, Jackson's ObjectMapper instantiation overhead matters. Spring Boot auto-configures a single shared ObjectMapper bean and passes it to MappingJackson2HttpMessageConverter. Customizing ObjectMapper via Jackson2ObjectMapperBuilderCustomizer is the right hook — not creating a new ObjectMapper bean which bypasses Boot's configuration.

Explain content negotiation in Spring MVC. How does it work with Accept headers, URL extensions, and parameters?
CONTENT_NEGOTIATION
▼

What is Content Negotiation?

Content negotiation is the process of selecting the best media type for the HTTP response, given: the client's acceptable types (from Accept header), the handler's declared producible types (produces), and the available HttpMessageConverters.

ContentNegotiationManager Strategies (evaluated in order)

  1. Path extension (deprecated, removed in 6.x): /api/users.json → application/json. Disabled by default since Spring 5.3. Security risk.
  2. Request parameter: /api/users?format=json. Must be explicitly enabled via ContentNegotiationConfigurer.favorParameter(true).
  3. Accept header (recommended): Accept: application/json. Always active. RFC-compliant. Supports quality factors (Accept: application/xml;q=0.9, application/json).

produces / consumes Constraints

produces = "application/json" on @RequestMapping: this handler can ONLY respond with JSON. If client sends Accept: text/xml and no other handler matches, Spring returns 406. The handler is excluded from matching entirely before invocation.

consumes = "application/json": handler only accepts JSON request body. If Content-Type doesn't match: 415 Unsupported Media Type.

// Content negotiation configuration
@Configuration
public class WebConfig implements WebMvcConfigurer {

    @Override
    public void configureContentNegotiation(ContentNegotiationConfigurer configurer) {
        configurer
            .favorPathExtension(false)      // disable suffix matching
            .favorParameter(true)           // enable ?format=json
            .parameterName("format")
            .ignoreAcceptHeader(false)      // keep Accept header support
            .defaultContentType(MediaType.APPLICATION_JSON)
            .mediaType("json", MediaType.APPLICATION_JSON)
            .mediaType("xml", MediaType.APPLICATION_XML);
    }
}

// Handler that serves both JSON and XML
@RestController
@RequestMapping("/api/users")
public class UserController {

    @GetMapping(produces = {MediaType.APPLICATION_JSON_VALUE,
                            MediaType.APPLICATION_XML_VALUE})
    public UserList getUsers() { ... }

    // Only JSON, explicit:
    @GetMapping(value = "/export",
                produces = "text/csv")
    public ResponseEntity<byte[]> exportCsv() { ... }
}
⚠ GotchaWhen using produces with Spring Security, the 406 response from content negotiation happens AFTER authentication. If you want unauthenticated clients to get a proper 401 instead of 406, ensure your security error handler also respects content negotiation.
✅ Best PracticeModern REST APIs should use Accept header negotiation exclusively. Avoid suffix and parameter-based strategies. Document supported media types in OpenAPI spec. Use produces annotations to make it explicit and self-documenting.
🎯 Google L6 AngleAccept header negotiation is the only RFC 7231-compliant strategy. URL suffix strategy has been removed in Spring 6 due to security concerns (path traversal vectors). L6 engineers understand that API versioning via vendor media types (Accept: application/vnd.company.v2+json) IS content negotiation — the cleanest versioning approach architecturally, even if URL versioning is more practical.

Explain all exception handling mechanisms in Spring MVC. What is @ControllerAdvice?
EXCEPTION_HANDLING
▼

Exception Resolution Chain (first match wins)

  1. Method-level @ExceptionHandler in the same @Controller class — handles exceptions from that controller only.
  2. @ControllerAdvice @ExceptionHandler — global, applies to all controllers (or scoped subset).
  3. ExceptionHandlerExceptionResolver — implements the above two.
  4. ResponseStatusExceptionResolver — handles @ResponseStatus-annotated exceptions.
  5. DefaultHandlerExceptionResolver — handles standard Spring MVC exceptions (MethodArgumentNotValidException → 400, HttpRequestMethodNotSupportedException → 405, etc.).

@ControllerAdvice

A specialization of @Component that applies cross-cutting concerns to all controllers. Can be scoped: basePackages, assignableTypes, annotations attributes. Contains: @ExceptionHandler, @InitBinder, @ModelAttribute.

@RestControllerAdvice = @ControllerAdvice + @ResponseBody — response body written as JSON by default.

ProblemDetail (Spring 6 / Boot 3)

RFC 7807 "Problem Details for HTTP APIs". Standardized error response format. Spring 6 introduces ProblemDetail and ErrorResponse interface. Enable via spring.mvc.problemdetails.enabled=true or extend ResponseEntityExceptionHandler.

// Global exception handler with ProblemDetail (Spring 6 / Boot 3)
@RestControllerAdvice
public class GlobalExceptionHandler extends ResponseEntityExceptionHandler {

    // Override standard Spring MVC exception handling
    @Override
    protected ResponseEntity<Object> handleMethodArgumentNotValid(
            MethodArgumentNotValidException ex,
            HttpHeaders headers, HttpStatusCode status, WebRequest request) {
        ProblemDetail pd = ProblemDetail.forStatusAndDetail(
            status, "Validation failed");
        pd.setTitle("Bad Request");
        pd.setProperty("violations", ex.getBindingResult()
            .getFieldErrors().stream()
            .map(fe -> Map.of("field", fe.getField(),
                              "message", fe.getDefaultMessage()))
            .toList());
        return ResponseEntity.of(pd).build();
    }

    // Custom business exception handler
    @ExceptionHandler(ResourceNotFoundException.class)
    public ProblemDetail handleNotFound(ResourceNotFoundException ex,
                                        HttpServletRequest request) {
        ProblemDetail pd = ProblemDetail.forStatusAndDetail(
            HttpStatus.NOT_FOUND, ex.getMessage());
        pd.setTitle("Resource Not Found");
        pd.setInstance(URI.create(request.getRequestURI()));
        return pd;
    }

    // Catch-all for unhandled exceptions
    @ExceptionHandler(Exception.class)
    public ResponseEntity<ProblemDetail> handleGeneric(Exception ex,
                                                        HttpServletRequest request) {
        log.error("Unhandled exception", ex);
        ProblemDetail pd = ProblemDetail.forStatus(HttpStatus.INTERNAL_SERVER_ERROR);
        pd.setDetail("An unexpected error occurred");
        // Don't expose internal details in production!
        return ResponseEntity.internalServerError().body(pd);
    }
}

// Custom exception with @ResponseStatus
@ResponseStatus(HttpStatus.CONFLICT)
public class DuplicateResourceException extends RuntimeException {
    public DuplicateResourceException(String message) { super(message); }
}
⚠ GotchaIf your @ExceptionHandler method itself throws an exception, that exception is NOT re-handled by another @ControllerAdvice — it propagates as an unhandled exception. Also: @ExceptionHandler in a @ControllerAdvice does NOT handle exceptions thrown by other @ControllerAdvice methods (e.g., from @ModelAttribute).
✅ Best PracticeAlways extend ResponseEntityExceptionHandler for global handlers — it already handles all standard Spring MVC exceptions (validation, method not supported, etc.) with proper status codes. Only add handlers for your custom exceptions on top.
🎯 Google L6 AngleL6 engineers design error responses with operational needs in mind: include a traceId (from MDC/distributed tracing context) in the error response so support engineers can correlate errors to traces in Stackdriver/Jaeger. ProblemDetail's extension properties (setProperty()) are ideal for this. Never expose stack traces or internal package names in production error responses.

What is the difference between a Servlet Filter, HandlerInterceptor, and @ControllerAdvice? When to use each?
FILTER_VS_INTERCEPTOR
▼

Execution Order

Request: Filter chain → DispatcherServlet → HandlerInterceptor.preHandle → Handler → HandlerInterceptor.postHandle → HandlerInterceptor.afterCompletion → Filter chain (reverse)

MechanismLevelAccess ToBest For
Servlet FilterServlet containerRaw request/response, no Spring contextAuth, compression, encoding, CORS, request wrapping
HandlerInterceptorSpring MVCHandler (knows which controller/method), ModelAndViewAudit logging, rate limiting per endpoint, locale
@ControllerAdviceSpring MVC (post-handler)Exceptions, model, data bindingGlobal exception handling, cross-cutting model attributes

Filter Details

Runs outside Spring's request pipeline. No access to Spring beans by default (though you can Autowire into Filter beans). OncePerRequestFilter: Spring's base class ensuring filter runs once per request (not on forward/include). Use FilterRegistrationBean to control order and URL patterns.

HandlerInterceptor Details

postHandle is NOT called when an exception escapes the handler. afterCompletion is always called (like finally). Interceptors know the handler type — can inspect annotations on the handler method for fine-grained decisions.

// Servlet Filter — JWT authentication (runs before Spring Security if ordered first)
@Component
@Order(1)
public class JwtAuthFilter extends OncePerRequestFilter {

    @Autowired JwtTokenService tokenService;

    @Override
    protected void doFilterInternal(HttpServletRequest request,
                                    HttpServletResponse response,
                                    FilterChain chain)
            throws ServletException, IOException {
        String authHeader = request.getHeader("Authorization");
        if (authHeader != null && authHeader.startsWith("Bearer ")) {
            try {
                String token = authHeader.substring(7);
                Authentication auth = tokenService.validate(token);
                SecurityContextHolder.getContext().setAuthentication(auth);
            } catch (JwtException e) {
                response.sendError(HttpStatus.UNAUTHORIZED.value(), "Invalid token");
                return; // short-circuit
            }
        }
        chain.doFilter(request, response); // continue chain
    }

    @Override
    protected boolean shouldNotFilter(HttpServletRequest request) {
        return request.getServletPath().startsWith("/api/public/");
    }
}

// HandlerInterceptor — Rate limiting per @RateLimited endpoints
@Component
public class RateLimitInterceptor implements HandlerInterceptor {

    @Autowired RateLimiterService rateLimiter;

    @Override
    public boolean preHandle(HttpServletRequest request,
                             HttpServletResponse response, Object handler) throws Exception {
        if (!(handler instanceof HandlerMethod hm)) return true;

        RateLimited annotation = hm.getMethodAnnotation(RateLimited.class);
        if (annotation == null) return true;

        String clientId = request.getHeader("X-Client-ID");
        if (!rateLimiter.tryAcquire(clientId, annotation.requestsPerSecond())) {
            response.setStatus(HttpStatus.TOO_MANY_REQUESTS.value());
            response.setHeader("Retry-After", "1");
            return false;
        }
        return true;
    }
}
⚠ GotchaFilters registered as Spring beans via @Component are automatically registered with /* pattern and no ordering. Use FilterRegistrationBean for explicit URL patterns and ordering. Without explicit ordering, filter execution order is non-deterministic across deployments.
✅ Best PracticeFor cross-cutting concerns that need Spring Security integration (auth, CSRF), use Filters. For business-level logging or endpoint-specific logic where you need to inspect the handler, use HandlerInterceptors. Never duplicate logic across both layers.
🎯 Google L6 AngleAt scale, filter and interceptor overhead compounds. A request hitting 5 filters × 10,000 RPS = 50,000 filter invocations per second. Keep filters lightweight. For expensive operations (DB lookups in auth), cache aggressively in the filter and profile filter overhead with Micrometer timers.

Explain validation in Spring MVC. What is the difference between @Valid and @Validated? How does BindingResult work?
VALIDATION
▼

@Valid vs @Validated

AnnotationSourceGroups SupportMethod Validation
@Validjakarta.validation (Bean Validation spec)NoNo (on its own)
@Validatedorg.springframework.validationYesYes (with AOP proxy)

@Valid: triggers Bean Validation on the annotated parameter. Hibernate Validator is the reference implementation. Validates constraints (@NotNull, @Size, @Email, @Pattern) on the object's fields. Also cascades to nested objects when @Valid is on the nested field.

@Validated on a controller parameter: same as @Valid but also supports specifying validation groups (e.g., @Validated(CreateGroup.class)). @Validated on a class (@Service, @Controller): enables method-level validation via MethodValidationPostProcessor AOP proxy — validates @NotNull/@Size etc. directly on method parameters and return values.

BindingResult

If a parameter immediately follows the @Valid/@Validated parameter with type BindingResult (or Errors), validation errors go into it instead of throwing MethodArgumentNotValidException. The controller then decides what to do.

// Bean with validation constraints
public class CreateUserRequest {
    @NotBlank(message = "Username is required")
    @Size(min = 3, max = 50, message = "Username must be 3-50 chars")
    private String username;

    @Email(message = "Valid email required")
    @NotBlank
    private String email;

    @NotNull
    @Min(value = 18, message = "Must be at least 18", groups = AgeVerification.class)
    private Integer age;

    @Valid // cascade validation to nested object
    @NotNull
    private Address address;
    // getters/setters...
}

// Controller with BindingResult
@RestController
@RequestMapping("/api/users")
public class UserController {

    // Option 1: Manual error handling via BindingResult
    @PostMapping
    public ResponseEntity<?> createUser(
            @RequestBody @Valid CreateUserRequest req,
            BindingResult result) {
        if (result.hasErrors()) {
            Map<String, String> errors = result.getFieldErrors().stream()
                .collect(Collectors.toMap(
                    FieldError::getField,
                    FieldError::getDefaultMessage,
                    (a, b) -> a)); // merge duplicate fields
            return ResponseEntity.badRequest().body(errors);
        }
        return ResponseEntity.status(201).body(userService.create(req));
    }

    // Option 2: Let MethodArgumentNotValidException propagate to @ControllerAdvice
    @PostMapping("/v2")
    public ResponseEntity<UserDto> createUserV2(
            @RequestBody @Valid CreateUserRequest req) {
        return ResponseEntity.status(201).body(userService.create(req));
    }
}

// Custom constraint annotation
@Documented
@Constraint(validatedBy = UniqueEmailValidator.class)
@Target({ElementType.FIELD})
@Retention(RetentionPolicy.RUNTIME)
public @interface UniqueEmail {
    String message() default "Email already registered";
    Class<?>[] groups() default {};
    Class<? extends Payload>[] payload() default {};
}

public class UniqueEmailValidator implements ConstraintValidator<UniqueEmail, String> {
    @Autowired UserRepository userRepository;

    @Override
    public boolean isValid(String email, ConstraintValidatorContext ctx) {
        if (email == null) return true; // let @NotNull handle null
        return !userRepository.existsByEmail(email);
    }
}

@Validated

⚠ Gotcha">BindingResult must be placed IMMEDIATELY after the validated parameter. If any other parameter is placed between them, Spring throws an exception at startup. Also: custom validators with @Autowired fields require the validator to be a Spring bean — don't instantiate it directly.
✅ Best PracticePrefer letting MethodArgumentNotValidException propagate and handle it in @ControllerAdvice (extending ResponseEntityExceptionHandler). Only use BindingResult when you need fine-grained control over the response format per endpoint.
🎯 Google L6 AngleCustom validators that hit the database (like UniqueEmailValidator) create implicit database reads during request validation — before your service layer runs. This can cause N+1 validation issues on bulk operations or double-reads when the service also checks uniqueness. Consider whether uniqueness validation belongs in the controller layer or purely in the service layer with proper error handling.

Explain async request processing in Spring MVC. What is DeferredResult vs Callable vs CompletableFuture?
ASYNC_MVC
▼

The Problem

Servlet containers have a fixed thread pool (Tomcat default: 200 threads). Long-running requests (DB, external API calls, message waiting) block threads, limiting concurrent request handling. Async processing releases the servlet thread while waiting, allowing it to serve other requests.

Callable<T>

Simplest form. Spring offloads execution to a configured thread pool (via AsyncTaskExecutor). The servlet thread is released. When Callable completes, Spring dispatches back to servlet container to complete the response.

Limitation: the callable starts immediately and runs on a Spring-managed thread. Cannot be triggered from an external event.

DeferredResult<T>

Result can be set from ANY thread, at ANY time. Controller returns DeferredResult immediately (servlet thread released). Later, any thread (Kafka listener, WebSocket handler, scheduled task) calls deferredResult.setResult(value). Spring then dispatches response writing.

Supports timeout: new DeferredResult<>(5000L, defaultResponse). Use for: long-polling, push notifications, comet patterns.

CompletableFuture<T> / ListenableFuture

Return CompletableFuture<T> directly from controller. Spring MVC detects it and registers a callback to write the response when the future completes. Integrates naturally with @Async methods.

Streaming Responses

ResponseBodyEmitter: send multiple objects to the response stream. SseEmitter: Server-Sent Events — push events to browser clients. StreamingResponseBody: stream large responses without loading into memory.

// Callable — simple async offload
@GetMapping("/reports/{id}")
public Callable<Report> generateReport(@PathVariable Long id) {
    return () -> reportService.generate(id); // runs on async thread pool
}

// DeferredResult — set from external event (e.g., Kafka)
private final Map<String, DeferredResult<OrderStatus>> pendingRequests = new ConcurrentHashMap<>();

@GetMapping("/orders/{id}/status/wait")
public DeferredResult<OrderStatus> waitForStatus(@PathVariable String id) {
    DeferredResult<OrderStatus> result = new DeferredResult<>(30_000L); // 30s timeout
    result.onTimeout(() -> result.setErrorResult(
        ResponseEntity.status(HttpStatus.REQUEST_TIMEOUT).body("Timeout")));
    result.onCompletion(() -> pendingRequests.remove(id));
    pendingRequests.put(id, result);
    return result;
}

// Called from Kafka listener or event handler
@EventListener
public void onOrderStatusUpdate(OrderStatusEvent event) {
    DeferredResult<OrderStatus> result = pendingRequests.get(event.getOrderId());
    if (result != null) result.setResult(event.getStatus());
}

// CompletableFuture — works with @Async services
@GetMapping("/data")
public CompletableFuture<Data> getData() {
    return asyncDataService.fetch(); // @Async method returns CompletableFuture
}

// SseEmitter — push events to browser
@GetMapping(value = "/events", produces = MediaType.TEXT_EVENT_STREAM_VALUE)
public SseEmitter streamEvents() {
    SseEmitter emitter = new SseEmitter(Long.MAX_VALUE);
    eventBroker.subscribe(emitter); // broker holds reference, sends events
    emitter.onCompletion(() -> eventBroker.unsubscribe(emitter));
    emitter.onTimeout(() -> eventBroker.unsubscribe(emitter));
    return emitter;
}

// StreamingResponseBody — large file download without loading into memory
@GetMapping("/files/{id}/download")
public ResponseEntity<StreamingResponseBody> download(@PathVariable Long id) {
    Resource resource = fileService.getResource(id);
    StreamingResponseBody body = outputStream -> {
        try (InputStream in = resource.getInputStream()) {
            in.transferTo(outputStream);
        }
    };
    return ResponseEntity.ok()
        .header(HttpHeaders.CONTENT_DISPOSITION, "attachment; filename=file.dat")
        .body(body);
}

@Configuration
public class AsyncConfig implements WebMvcConfigurer {
    @Override
    public void configureAsyncSupport(AsyncSupportConfigurer configurer) {
        configurer.setDefaultTimeout(60_000); // 60s default
        configurer.setTaskExecutor(asyncExecutor());

⚠ GotchaSpring Security's SecurityContext is thread-local. When using Callable or CompletableFuture, the security context is NOT automatically propagated to the async thread. Requires SecurityContextHolder.setStrategyName(MODE_INHERITABLETHREADLOCAL) or Spring Security's DelegatingSecurityContextAsyncTaskExecutor.
🎯 Google L6 AngleFor truly high-throughput non-blocking I/O (e.g., calling 10 microservices in parallel per request), async MVC is a band-aid. The servlet thread is released during waiting, but you still need threads in the async executor. Spring WebFlux on Netty with Project Reactor gives you truly non-blocking I/O with a fixed pool of event loop threads. Know when to reach for WebFlux vs async MVC.

How does CORS work in Spring MVC? Explain all configuration options.
CORS
▼

CORS Preflight Protocol

Browser sends a preflight OPTIONS request with Origin, Access-Control-Request-Method, Access-Control-Request-Headers. Server must respond with Access-Control-Allow-Origin, Access-Control-Allow-Methods, Access-Control-Allow-Headers, and Access-Control-Max-Age. Only then does the browser send the actual request.

Spring CORS Options (in order of specificity)

  1. @CrossOrigin on method or class — fine-grained, per-endpoint.
  2. WebMvcConfigurer.addCorsMappings() — global, path-pattern based.
  3. CorsFilter — servlet filter level, required when Spring Security is in play.

Spring Security Interaction (Critical!)

Spring Security filter chain runs BEFORE DispatcherServlet. If a CORS preflight OPTIONS request hits Spring Security before MVC processes it, Security may reject it with 401/403 before your CORS config has a chance to respond. Solution: configure CORS at the CorsConfigurationSource level consumed by CorsFilter or Spring Security's http.cors().

// Option 1: @CrossOrigin on controller
@RestController
@CrossOrigin(origins = "https://app.example.com",
             allowedHeaders = {"Authorization", "Content-Type"},
             methods = {RequestMethod.GET, RequestMethod.POST},
             maxAge = 3600)
public class UserController {
    @CrossOrigin(origins = "*") // override at method level
    @GetMapping("/public")
    public List<User> publicEndpoint() { ... }
}

// Option 2: Global MVC config (without Spring Security)
@Configuration
public class WebConfig implements WebMvcConfigurer {
    @Override
    public void addCorsMappings(CorsRegistry registry) {
        registry.addMapping("/api/**")
            .allowedOrigins("https://app.example.com", "https://admin.example.com")
            .allowedMethods("GET", "POST", "PUT", "DELETE", "PATCH")
            .allowedHeaders("Authorization", "Content-Type", "X-Request-ID")
            .exposedHeaders("X-Total-Count", "X-Page-Size") // expose custom response headers
            .allowCredentials(true) // allows cookies, auth headers
            .maxAge(3600);

        registry.addMapping("/public/**")
            .allowedOrigins("*")
            .allowedMethods("GET");
    }
}

// Option 3: CorsConfigurationSource + Spring Security (RECOMMENDED for secured apps)
@Configuration
@EnableWebSecurity
public class SecurityConfig {

    @Bean
    public SecurityFilterChain filterChain(HttpSecurity http) throws Exception {
        http
            .cors(cors -> cors.configurationSource(corsConfigurationSource()))
            .csrf(AbstractHttpConfigurer::disable)
            .authorizeHttpRequests(auth -> auth
                .requestMatchers("/api/public/**").permitAll()
                .anyRequest().authenticated());
        return http.build();
    }

    @Bean
    public CorsConfigurationSource corsConfigurationSource() {
        CorsConfiguration config = new CorsConfiguration();
        config.setAllowedOriginPatterns(List.of(
            "https://*.example.com",    // wildcard subdomain
            "https://app.example.com"
        ));
        config.setAllowedMethods(List.of("GET","POST","PUT","DELETE","PATCH","OPTIONS"));
        config.setAllowedHeaders(List.of("Authorization","Content-Type","X-Request-ID"));
        config.setExposedHeaders(List.of("X-Total-Count"));
        config.setAllowCredentials(true);
        config.setMaxAge(3600L);

        UrlBasedCorsConfigurationSource source = new UrlBasedCorsConfigurationSource();
        source.registerCorsConfiguration("/api/**", config);
        return source;
    }
}
⚠ GotchaallowCredentials(true) and allowedOrigins("*") cannot be combined — browser rejects it. When credentials are allowed, you must specify explicit origins. Use setAllowedOriginPatterns() with allowCredentials(true) for wildcard-like behavior.
✅ Best PracticeAlways configure CORS at the CorsConfigurationSource bean level when Spring Security is present. The WebMvcConfigurer.addCorsMappings() approach doesn't apply to OPTIONS preflight requests that are rejected by Spring Security before reaching MVC.
🎯 Google L6 AngleIn a microservices architecture, CORS should ideally be handled at the API gateway layer (Envoy, Kong, cloud load balancer), not in every individual service. Services behind a gateway typically don't need CORS config. Only edge-facing services or BFFs (Backend for Frontend) need it. This reduces operational complexity significantly.

Explain Spring MVC's path matching. What changed in Spring 5.3/6 with PathPatternParser?
PATH_PATTERNS
▼

AntPathMatcher (Legacy)

Based on Ant-style glob patterns. Evaluated at request time using regex-like string matching. Issues: doesn't handle URL-encoded characters consistently, slower for complex patterns, evaluated every request.

PathPatternParser (Default in Boot 3 / Spring 6)

Pre-parses URL patterns at application startup into a PathPattern object (AST). Matching at request time is structural — traverse the AST against request path segments. 6-8x faster for complex patterns. Handles URL encoding correctly. Fails fast on invalid/ambiguous patterns at startup.

Pattern Syntax

PatternMatchesNotes
/users/{id}/users/123Single path segment
/users/{id:.+}/users/abc.jsonGreedy — matches dots
/files/**/files/a/b/cMultiple segments
/files/{*path}/files/a/b/c → path=a/b/cPathPatternParser only — captures remaining path
/u?ers/users, /ubersSingle char wildcard

Ambiguous Pattern Detection

PathPatternParser detects ambiguous handler registrations at startup and throws IllegalStateException. AntPathMatcher would silently pick one at runtime (non-deterministic). This is a breaking change when upgrading — patterns that were ambiguously defined before will now fail fast.

// {*path} capture (PathPatternParser only — no equivalent in AntPathMatcher)
@GetMapping("/files/{*filePath}")
public Resource serveFile(@PathVariable String filePath) {
    // filePath = "/images/logo.png" (includes leading slash)
    return fileService.load(filePath);
}

// Configuring path matching strategy
@Configuration
public class WebConfig implements WebMvcConfigurer {
    @Override
    public void configurePathMatch(PathMatchConfigurer configurer) {
        // Spring Boot 3 default: PathPatternParser — no action needed
        // To use AntPathMatcher (not recommended):
        configurer.setPatternParser(null); // reverts to AntPathMatcher
    }
}

// Spring Boot property (application.properties)
// spring.mvc.pathmatch.matching-strategy=path-pattern-parser (default)
// spring.mvc.pathmatch.use-suffix-pattern=false (default, keep false)
// spring.mvc.pathmatch.use-registered-suffix-pattern=false (default)
⚠ Gotcha">When migrating from Spring Boot 2 to 3, if you have patterns like /api/users/{id:.+} or use of /** in places that might overlap with other patterns, test carefully. PathPatternParser is stricter about ambiguity. Also: Spring Security's mvcMatchers() uses the same path matching as MVC — if you switch strategies, security rules may behave differently.
🎯 Google L6 AngleFor a high-traffic API at 50k+ RPS, the path matching overhead with AntPathMatcher on complex routing tables (100+ endpoints) can consume 1-2ms per request — measurable at scale. PathPatternParser's startup-time compilation of patterns is the right trade-off: pay once at startup, save on every request. This is a systems design principle: move work out of the hot path.

Explain the different controller return types. When to use ResponseEntity vs @ResponseBody vs returning POJO directly?
RETURN_TYPES
▼

Return Type Comparison

Return TypeStatus CodeHeadersBest For
void200 (or what you set on response)ManualWriting directly to HttpServletResponse
String (in @RestController)200NonePlain text responses
POJOAlways 200NoneSimple always-200 responses

Mono<T> / Flux<T>Full controlVia Mono<ResponseEntity>Reactive streaming or WebFlux

When to Use Each

POJO return: clean and concise when the response is always 200 and needs no custom headers. @RestController + Jackson serializes automatically.

ResponseEntity<T>: whenever you need to control HTTP status (201 Created for POST, 204 No Content for DELETE, 202 Accepted for async), or add headers (Location, ETag, Cache-Control, X-Total-Count).

Optional + ResponseEntity: ResponseEntity.of(optionalValue) — returns 200 with body if present, 404 if empty. Clean idiom.

@RestController
@RequestMapping("/api/users")
public class UserController {

    // Simple GET — POJO return (always 200)
    @GetMapping("/{id}")
    public UserDto getUser(@PathVariable Long id) {
        return userService.findById(id) // throws ResourceNotFoundException if not found
                          .map(UserDto::from)
                          .orElseThrow(() -> new ResourceNotFoundException("User " + id));
    }

    // POST — 201 Created with Location header
    @PostMapping
    public ResponseEntity<UserDto> createUser(@RequestBody @Valid CreateUserRequest req) {
        UserDto created = userService.create(req);
        URI location = ServletUriComponentsBuilder
            .fromCurrentRequest()
            .path("/{id}")
            .buildAndExpand(created.getId())
            .toUri();
        return ResponseEntity.created(location).body(created);
    }

    // DELETE — 204 No Content
    @DeleteMapping("/{id}")
    public ResponseEntity<Void> deleteUser(@PathVariable Long id) {
        userService.delete(id);
        return ResponseEntity.noContent().build();
    }

    // PUT — conditional update with ETag
    @PutMapping("/{id}")
    public ResponseEntity<UserDto> updateUser(
            @PathVariable Long id,
            @RequestBody @Valid UpdateUserRequest req,
            @RequestHeader(value = "If-Match", required = false) String ifMatch) {
        UserDto updated = userService.update(id, req, ifMatch); // throws 412 if ETag mismatch
        return ResponseEntity.ok()
            .eTag(updated.getVersion().toString())
            .body(updated);
    }

    // Optional idiom — 200 or 404
    @GetMapping("/by-email")
    public ResponseEntity<UserDto> findByEmail(@RequestParam String email) {
        return ResponseEntity.of(userService.findByEmail(email).map(UserDto::from));
    }

    // Async — 202 Accepted
    @PostMapping("/bulk-import")
    public ResponseEntity<JobDto> bulkImport(@RequestBody ImportRequest req) {
        JobDto job = importService.startAsync(req);
        URI statusUri = URI.create("/api/jobs/" + job.getId());
        return ResponseEntity.accepted()
            .header("Location", statusUri.toString())
            .body(job);
    }
}
⚠ GotchaReturning null from a controller method with @ResponseBody sends a 200 with empty body. This is often a bug — either throw an exception for 404 or return ResponseEntity.noContent().build() explicitly. Jackson by default serializes null as JSON null in object fields but an empty body for top-level null.
🎯 Google L6 AngleProper HTTP semantics matter for caching infrastructure. GET responses with proper ETags and Cache-Control headers allow CDN/proxy caching. POST returning 201 with Location header allows clients to immediately fetch the created resource. 202 Accepted with a polling URL is the correct pattern for long-running operations. These aren't just conventions — they affect system architecture and observability.

How do you implement a custom HandlerMethodArgumentResolver? Give a practical use case.
CUSTOM_ARG_RESOLVER
▼

Use Case: @CurrentUser Annotation

Inject the currently authenticated user object directly into controller method parameters, eliminating repetitive SecurityContextHolder boilerplate across every endpoint.

// Step 1: Define the annotation
@Target(ElementType.PARAMETER)
@Retention(RetentionPolicy.RUNTIME)
@Documented
public @interface CurrentUser {
    boolean required() default true;
}

// Step 2: Implement the resolver
@Component
public class CurrentUserArgumentResolver implements HandlerMethodArgumentResolver {

    @Override
    public boolean supportsParameter(MethodParameter parameter) {
        return parameter.hasParameterAnnotation(CurrentUser.class)
            && UserPrincipal.class.isAssignableFrom(parameter.getParameterType());
    }

    @Override
    public Object resolveArgument(MethodParameter parameter,
                                  ModelAndViewContainer mavContainer,
                                  NativeWebRequest webRequest,
                                  WebDataBinderFactory binderFactory) {
        Authentication auth = SecurityContextHolder.getContext().getAuthentication();
        if (auth == null || !auth.isAuthenticated()
                || auth instanceof AnonymousAuthenticationToken) {
            CurrentUser annotation = parameter.getParameterAnnotation(CurrentUser.class);
            if (annotation != null && annotation.required()) {
                throw new AuthenticationCredentialsNotFoundException("No authenticated user");
            }
            return null;
        }
        return (UserPrincipal) auth.getPrincipal();
    }
}

// Step 3: Register
@Configuration
public class WebConfig implements WebMvcConfigurer {
    @Autowired CurrentUserArgumentResolver currentUserResolver;

    @Override
    public void addArgumentResolvers(List<HandlerMethodArgumentResolver> resolvers) {
        resolvers.add(currentUserResolver);
    }
}

// Step 4: Use in controllers — clean and testable
@RestController
@RequestMapping("/api")
public class ProfileController {

    @GetMapping("/profile")
    public UserProfileDto getProfile(@CurrentUser UserPrincipal user) {
        return profileService.getProfile(user.getId());
    }

    @PutMapping("/profile")
    public UserProfileDto updateProfile(@CurrentUser UserPrincipal user,
                                        @RequestBody @Valid UpdateProfileRequest req) {
        return profileService.update(user.getId(), req);
    }

    @GetMapping("/admin/resource")
    public AdminData adminResource(@CurrentUser(required = false) UserPrincipal user) {
        // user can be null for anonymous endpoints
        return adminService.getData(user);
    }
}

// Step 5: Test — easily mock the resolver behavior
@WebMvcTest(ProfileController.class)
class ProfileControllerTest {

    @Autowired MockMvc mockMvc;

    @MockBean ProfileService profileService;

    @Test
    void getProfile_authenticated() throws Exception {
        // Configure SecurityContext or use @WithMockUser
        mockMvc.perform(get("/api/profile")
                .with(user("testuser").roles("USER")))
               .andExpect(status().isOk());
    }
}
⚠ GotchaThe resolver is called once per matching parameter per request. If supportsParameter() is slow (e.g., involving reflection loops), it adds overhead on every request for every parameter. Keep supportsParameter() fast — annotation presence check is O(1).
✅ Best PracticeOther great use cases for custom argument resolvers: inject tenant context from a header (@TenantContext), inject pageable sorting preferences with default values, inject API version from URL or header, inject feature flags from a request-scoped bean.
🎯 Google L6 AngleCustom argument resolvers are the correct abstraction for cross-cutting request context injection. Compare this to the alternative: every controller method calling SecurityContextHolder.getContext().getAuthentication() — that's not only verbose but also untestable (requires security context setup). The resolver centralizes the logic, makes it mockable, and removes coupling between controllers and Spring Security internals.

Versioning Strategies Compared

StrategyExampleProsCons

@GetMapping("/v1/users")

// Interceptor to log usage of deprecated API versions
@Component
public class DeprecationWarningInterceptor implements HandlerInterceptor {
    @Override
    public void afterCompletion(HttpServletRequest req,
                                HttpServletResponse res, Object handler, Exception ex) {
        if (req.getRequestURI().contains("/v1/")) {
            log.warn("Deprecated API v1 called: {} by client {}",

✅ Best PracticeURL versioning is the pragmatic choice: easy to route at the API gateway, cacheable by CDN (version is in the URL), easy to document with OpenAPI, simple to test. Use deprecation headers (Deprecation, Sunset per RFC 8594) to give clients migration timelines. Never break a live version — only deprecate and sunset.
🎯 Google L6 AngleVersioning strategy has architectural implications: URL versioning allows CDN caching per version (v1 cache separate from v2), API gateway routing to different service versions for blue/green deployment, and independent scaling. Accept header versioning requires the CDN to vary on the Accept header (must set Vary: Accept response header) — otherwise cached v1 responses are served to v2 clients. This is a real production bug.

How does Spring MVC handle file uploads? What are the size limits and how do you configure them?
MULTIPART
▼

MultipartResolver

Spring MVC uses a MultipartResolver to parse multipart requests. Two implementations: CommonsMultipartResolver (Apache Commons FileUpload) and StandardServletMultipartResolver (Servlet 3.0 standard — default in Spring Boot).

Spring Boot auto-configures StandardServletMultipartResolver via MultipartAutoConfiguration.

Size Limits and Configuration

# application.properties
spring.servlet.multipart.enabled=true
spring.servlet.multipart.max-file-size=10MB        # per-file limit (default 1MB)
spring.servlet.multipart.max-request-size=50MB     # total request limit (default 10MB)
spring.servlet.multipart.file-size-threshold=2MB   # threshold before writing to disk
spring.servlet.multipart.location=/tmp/uploads     # temp storage location
// Basic file upload endpoint
@RestController
@RequestMapping("/api/files")
public class FileController {

    // Single file
    @PostMapping("/upload")
    public ResponseEntity<FileResponse> upload(
            @RequestParam("file") MultipartFile file,
            @RequestParam("description") String description) {
        if (file.isEmpty()) {
            return ResponseEntity.badRequest().build();
        }
        String fileId = fileService.store(file.getOriginalFilename(),
                                          file.getContentType(),
                                          file.getInputStream());
        return ResponseEntity.status(201)
            .body(new FileResponse(fileId, file.getSize()));
    }

    // Multiple files
    @PostMapping("/bulk-upload")
    public List<FileResponse> bulkUpload(
            @RequestParam("files") List<MultipartFile> files) {
        return files.stream()
            .map(f -> fileService.store(f))
            .toList();
    }

    // Multipart JSON + file (complex form)
    @PostMapping(value = "/document", consumes = MediaType.MULTIPART_FORM_DATA_VALUE)
    public DocumentResponse uploadDocument(
            @RequestPart("metadata") @Valid DocumentMetadata metadata,
            @RequestPart("file") MultipartFile file) {
        return documentService.save(metadata, file);
    }

    // Streaming upload to S3 — avoid loading entire file into memory
    @PostMapping("/stream-upload")
    public ResponseEntity<UploadResult> streamUpload(
            @RequestParam("file") MultipartFile file) throws IOException {
        // DON'T: file.getBytes() — loads entire file into JVM heap
        // DO: stream directly to destination
        try (InputStream is = file.getInputStream()) {
            String s3Key = s3Service.upload(
                is,
                file.getSize(),
                file.getContentType(),
                file.getOriginalFilename()
            );
            return ResponseEntity.ok(new UploadResult(s3Key));
        }
    }
}

// Handle MaxUploadSizeExceededException
@RestControllerAdvice
public class FileExceptionHandler {
    @ExceptionHandler(MaxUploadSizeExceededException.class)
    public ResponseEntity<ProblemDetail> handleFileTooLarge(
            MaxUploadSizeExceededException ex) {
        ProblemDetail pd = ProblemDetail.forStatus(HttpStatus.PAYLOAD_TOO_LARGE);
        pd.setDetail("File size exceeds the maximum allowed size");
        return ResponseEntity.status(HttpStatus.PAYLOAD_TOO_LARGE).body(pd);
    }
}
⚠ GotchaMaxUploadSizeExceededException is thrown BEFORE the request reaches your controller — it's thrown during multipart parsing in the filter chain. Ensure your @ControllerAdvice handler can still be invoked, or configure a @Bean of type TomcatEmbeddedWebappClassLoader. Also: file.getOriginalFilename() is provided by the client — NEVER use it as a filesystem path without sanitization (path traversal attack).
🎯 Google L6 AngleFor files >10MB in production, skip Spring MVC entirely. The correct pattern: client requests a pre-signed S3 URL from your API (1 small request), client uploads directly to S3 (no traffic through your servers), S3 triggers a Lambda/SQS notification when upload completes, your service processes the file. This removes the upload bottleneck from your application servers, eliminates memory pressure, and scales to arbitrarily large files. Your Spring service just generates signed URLs and processes metadata.

What is HATEOAS and how does Spring HATEOAS help implement it?
HATEOAS
▼

What is HATEOAS?

HATEOAS (Hypermedia As The Engine Of Application State) is Richardson Maturity Model Level 3. Responses include hypermedia links to related resources and state transitions, allowing clients to discover API capabilities dynamically rather than relying on out-of-band documentation.

// Without HATEOAS (Level 2 REST):
{"id": 1, "username": "alice", "status": "ACTIVE"}

// With HATEOAS (Level 3):
{
  "id": 1, "username": "alice", "status": "ACTIVE",
  "_links": {
    "self": {"href": "/api/users/1"},
    "edit": {"href": "/api/users/1", "type": "PUT"},
    "deactivate": {"href": "/api/users/1/deactivate", "type": "POST"},
    "orders": {"href": "/api/users/1/orders"}
  }
}
// Spring HATEOAS implementation
@RestController
@RequestMapping("/api/users")
public class UserController {

    @Autowired UserRepresentationModelAssembler assembler;

    @GetMapping("/{id}")
    public EntityModel<UserDto> getUser(@PathVariable Long id) {
        UserDto user = userService.findById(id);
        return assembler.toModel(user);
    }

    @GetMapping
    public CollectionModel<EntityModel<UserDto>> getUsers() {
        List<EntityModel<UserDto>> users = userService.findAll().stream()
            .map(assembler::toModel)
            .toList();
        return CollectionModel.of(users,
            linkTo(methodOn(UserController.class).getUsers()).withSelfRel());
    }

    @GetMapping(params = "page")
    public PagedModel<EntityModel<UserDto>> getUsersPaged(Pageable pageable,
            PagedResourcesAssembler<UserDto> pagedAssembler) {
        Page<UserDto> page = userService.findAll(pageable);
        return pagedAssembler.toModel(page, assembler);
        // Adds: first, prev, self, next, last links automatically
    }
}

// RepresentationModelAssembler — converts domain object to EntityModel with links
@Component
public class UserRepresentationModelAssembler
        implements RepresentationModelAssembler<UserDto, EntityModel<UserDto>> {

    @Override
    public EntityModel<UserDto> toModel(UserDto user) {
        EntityModel<UserDto> model = EntityModel.of(user);

        // Self link — always present
        model.add(linkTo(methodOn(UserController.class)
            .getUser(user.getId())).withSelfRel());

        // Conditional links based on state
        if (user.getStatus() == UserStatus.ACTIVE) {
            model.add(linkTo(methodOn(UserController.class)
                .deactivateUser(user.getId())).withRel("deactivate"));
        }
        if (user.getStatus() == UserStatus.INACTIVE) {
            model.add(linkTo(methodOn(UserController.class)
                .activateUser(user.getId())).withRel("activate"));
        }

        // Relation to sub-resources
        model.add(linkTo(methodOn(OrderController.class)
            .getOrdersByUser(user.getId())).withRel("orders"));

        return model;
    }
}
⚠ Gotcha">WebMvcLinkBuilder.linkTo(methodOn(...)) uses a Spring AOP proxy to record the method call. It does NOT actually invoke the method — but the class must be a Spring-managed bean for proxy to work. Calling methodOn() on a non-Spring class or a final class without proxy support will throw at runtime.
✅ Best PracticeHATEOAS is widely discussed but rarely fully implemented. Most production APIs stop at Level 2 (resources + HTTP methods) and document with OpenAPI/Swagger. The pragmatic middle ground: include _links.self (self-referential URL) in all responses — this is widely adopted and enables client caching without full HATEOAS. Full HATEOAS implementation is a significant investment in maintainability.
🎯 Google L6 AngleHATEOAS matters most for public APIs where you don't control all clients. Internal APIs between microservices are better served by strong contracts (OpenAPI spec + code generation). For public-facing APIs, HAL+JSON with pagination links is a reasonable pragmatic choice — it solves a real client problem (pagination traversal) without the full complexity of state machine hypermedia.

How does Spring Boot serve static resources? What is the resource chain and versioning?
STATIC_RESOURCES
▼

Default Static Resource Locations

Spring Boot's ResourceHttpRequestHandler (registered by WebMvcAutoConfiguration) serves files from four classpath locations (in order):

  1. classpath:/META-INF/resources/
  2. classpath:/resources/
  3. classpath:/static/
  4. classpath:/public/

Files in these locations are served at their path relative to the location root. E.g., classpath:/static/js/app.js → GET /js/app.js.

Resource Handler Features

  • HTTP caching: sets Last-Modified and ETag response headers. Supports If-Modified-Since and If-None-Match conditional requests → 304 Not Modified.
  • Range requests: Range: bytes=0-1023 for partial content (useful for video streaming).
  • Encoding: serves pre-compressed .gz or .br files if present and client accepts encoding.

Resource Chain and Versioning

The resource chain is a pipeline of ResourceResolvers and ResourceTransformers. VersionResourceResolver appends a content hash to the URL for long-term caching. CachingResourceResolver caches resolved resources to avoid repeated classpath lookups.

// Resource versioning configuration
@Configuration
public class StaticResourceConfig implements WebMvcConfigurer {

    @Override
    public void addResourceHandlers(ResourceHandlerRegistry registry) {
        registry.addResourceHandler("/static/**")
            .addResourceLocations("classpath:/static/")
            .setCacheControl(CacheControl.maxAge(365, TimeUnit.DAYS)
                                         .cachePublic()
                                         .immutable()) // 1-year cache for fingerprinted assets
            .resourceChain(true) // enable resource chain with caching
                .addResolver(new VersionResourceResolver()
                    .addContentVersionStrategy("/**")) // appends MD5 hash of file content
                .addResolver(new CachingResourceResolver(
                    new ConcurrentMapCache("staticResources")));

        // For non-fingerprinted assets (images, fonts) — shorter cache
        registry.addResourceHandler("/images/**")
            .addResourceLocations("classpath:/static/images/")
            .setCacheControl(CacheControl.maxAge(7, TimeUnit.DAYS).cachePublic());
    }
}

🎯 Google L6 AngleIn production at scale, serving static assets from your Spring Boot application is an anti-pattern. All static assets should be served from a CDN (CloudFront, Fastly) — the Spring app just serves the HTML shell. This reduces load on application servers, improves latency via CDN edge nodes, and makes releases simpler (CDN cache invalidation). Spring's resource versioning is still relevant for cache-busting strategy design.

What WebMvcConfigurer methods are most important? How does @EnableWebMvc affect auto-configuration?
GLOBAL_CONFIG
▼

@EnableWebMvc vs Spring Boot Auto-Configuration

@EnableWebMvc: imports DelegatingWebMvcConfiguration, which sets up the FULL Spring MVC infrastructure programmatically. In Spring Boot: adding @EnableWebMvc to ANY @Configuration class causes WebMvcAutoConfiguration to BACK OFF completely (@ConditionalOnMissingBean(WebMvcConfigurationSupport.class)).

Result: you lose all Boot defaults — Jackson ObjectMapper config, content negotiation, static resources, default ViewResolvers, etc. You must configure everything manually. This is almost never what you want.

Correct approach in Spring Boot: implement WebMvcConfigurer interface on your @Configuration class. This adds to Boot's existing configuration without replacing it.

// Complete WebMvcConfigurer example — adds to Boot's defaults
@Configuration
public class WebMvcConfig implements WebMvcConfigurer {

    @Autowired AuditInterceptor auditInterceptor;
    @Autowired RateLimitInterceptor rateLimitInterceptor;
    @Autowired CurrentUserArgumentResolver currentUserResolver;

    @Override
    public void addInterceptors(InterceptorRegistry registry) {
        registry.addInterceptor(auditInterceptor).addPathPatterns("/api/**");
        registry.addInterceptor(rateLimitInterceptor)
                .addPathPatterns("/api/**")
                .excludePathPatterns("/api/public/**");
    }

    @Override
    public void addCorsMappings(CorsRegistry registry) {
        registry.addMapping("/api/**")
                .allowedOrigins("https://app.example.com");
    }

    @Override
    public void addArgumentResolvers(List<HandlerMethodArgumentResolver> resolvers) {
        resolvers.add(currentUserResolver);
    }

    @Override
    public void extendMessageConverters(List<HttpMessageConverter<?>> converters) {
        converters.add(0, new ProtobufHttpMessageConverter()); // add Protobuf support
    }

    @Override
    public void addFormatters(FormatterRegistry registry) {
        registry.addConverter(String.class, UserId.class, UserId::from); // custom type conversion
        registry.addConverter(new StringToEnumIgnoreCaseConverter<>());
    }

    @Override
    public void configureAsyncSupport(AsyncSupportConfigurer configurer) {
        configurer.setDefaultTimeout(60_000);
        configurer.setTaskExecutor(asyncExecutor());
    }

    @Override
    public void addViewControllers(ViewControllerRegistry registry) {
        registry.addViewController("/").setViewName("forward:/index.html");
        registry.addRedirectViewController("/old-path", "/new-path");
    }
}

// WRONG: This disables ALL of Boot's MVC auto-configuration!
@Configuration
@EnableWebMvc // DON'T DO THIS in Spring Boot!
public class BadWebConfig implements WebMvcConfigurer { ... }
⚠ GotchaAdding @EnableWebMvc to "fix" a Spring MVC issue is a classic mistake — it hides the real problem by disabling auto-configuration and creates a new, larger problem of missing defaults. The symptom is usually: Jackson no longer serializes dates correctly, static resources stop working, or content negotiation breaks.
✅ Best PracticeAlways use WebMvcConfigurer (without @EnableWebMvc) to customize Spring Boot MVC. If you truly need to replace ALL MVC configuration (e.g., for a framework built on top of Spring Boot), use @EnableWebMvc — but document WHY explicitly.
🎯 Google L6 AngleUnderstanding the @ConditionalOnMissingBean(WebMvcConfigurationSupport.class) condition on WebMvcAutoConfiguration is key. @EnableWebMvc → imports DelegatingWebMvcConfiguration extends WebMvcConfigurationSupport → condition fails → auto-config backs off. This is the pattern Spring Boot uses everywhere: auto-config beans with @ConditionalOnMissingBean allow user override. Understanding this pattern lets you reason about any auto-configuration conflict.
