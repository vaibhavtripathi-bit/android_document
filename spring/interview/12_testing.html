Explain the testing pyramid for Spring Boot applications. What ratio should you target?
TEST_PYRAMID
▼

The Testing Pyramid

The testing pyramid describes the optimal distribution of test types: many fast, cheap unit tests at the base; fewer integration tests in the middle; and a small number of end-to-end tests at the top. An inverted pyramid — dominated by slow E2E tests — leads to slow CI pipelines, flaky test suites, and high maintenance cost.

Layer Breakdown for Spring Boot

LayerAnnotation / ToolRatioSpeedWhat It Covers
Unit TestsJUnit 5 + Mockito (no Spring)~70%MillisecondsPOJOs, service logic, domain objects, utilities
Slice Tests@WebMvcTest, @DataJpaTest, @DataMongoTest~20%SecondsSingle layer (controller, repository) in isolation
Integration Tests@SpringBootTest + TestContainers~9%10–30sFull context, real DB, real HTTP stack
E2E TestsFull deployment + REST / Selenium~1%MinutesCritical user journeys, smoke tests

Spring Context Caching

Spring caches the ApplicationContext between test classes that share an identical configuration (same annotations, properties, mock beans). This means multiple @SpringBootTest classes do NOT each pay the full startup cost — the context is reused. @DirtiesContext forces a context rebuild after the annotated test class or method, which is expensive and should be avoided unless the test genuinely mutates global state.

Key CI Metric

The total test suite should run in under 5 minutes in CI. If your suite takes 20+ minutes, the culprit is almost always too many @SpringBootTest tests that could be replaced with targeted slice tests or pure unit tests.

✅ Best PracticeWrite every new piece of business logic as a plain unit test first. Reach for @SpringBootTest only when you genuinely need to verify cross-cutting wiring — security filters, transaction boundaries, auto-configuration correctness.
🎯 Google L6 AngleAt Google scale, test speed is a force multiplier. A 10-minute suite run 200 times a day costs 33 engineer-hours per day of waiting. Invest in fast unit tests and targeted slices; treat @SpringBootTest as a premium resource to be justified, not the default.

What does @SpringBootTest do? Explain the webEnvironment modes and when to use each.
SPRING_BOOT_TEST
▼

What @SpringBootTest Does

@SpringBootTest loads the full ApplicationContext, including all auto-configurations, component scanning, and real bean wiring — essentially simulating a production startup. It is the most powerful but most expensive test annotation Spring Boot provides.

webEnvironment Modes

ModeDescriptionBest For
MOCK (default)Creates a fake MockServletContext. No real HTTP server started. Use MockMvc for HTTP simulation.Controller tests needing full context (but prefer @WebMvcTest)
RANDOM_PORTStarts a real embedded Tomcat on a random available port. Inject port with @LocalServerPort. Use TestRestTemplate or WebTestClient.True HTTP integration tests — auth filter behavior, CORS, real TCP stack
DEFINED_PORTStarts embedded server on the configured port (default 8080). Risk of port conflicts in parallel test runs.Port-specific behavior, legacy tests
NONELoads full ApplicationContext but no web infrastructure at all.Testing non-web service beans that need all auto-configuration

Configuration Attributes

// Explicit root class (usually auto-detected from @SpringBootApplication)
@SpringBootTest(classes = MyApplication.class,
                webEnvironment = SpringBootTest.WebEnvironment.RANDOM_PORT,
                properties = {
                    "spring.datasource.url=jdbc:h2:mem:testdb",
                    "feature.flag.new-checkout=true"
                },
                args = "--spring.profiles.active=test")
class CheckoutIntegrationTest {
    @LocalServerPort int port;
    @Autowired WebTestClient webClient; // auto-configured for RANDOM_PORT
}

Context Caching Implications

Two @SpringBootTest classes sharing the same configuration (annotations, properties, mock beans) reuse the same cached context. Any difference — a different @MockBean, a different property override — triggers a new context build. Minimize variation across test classes to maximize cache hits.

⚠ Gotcha@SpringBootTest with RANDOM_PORT and @Transactional on the test method will NOT roll back data — the server runs in a separate thread with its own transaction. Only use @Transactional for rollback in MOCK mode or @DataJpaTest.
✅ Best PracticePrefer @WebMvcTest for controller-layer tests. Reserve @SpringBootTest(RANDOM_PORT) for tests that specifically need to exercise the HTTP stack end-to-end, such as verifying Spring Security filter chain behavior or testing real response headers.

What are Spring Boot test slices? Explain @WebMvcTest, @DataJpaTest, @DataMongoTest.
TEST_SLICES
▼

What Test Slices Are

Test slices load only a partial ApplicationContext relevant to a specific application layer. They are faster than full @SpringBootTest because irrelevant beans (e.g., repository beans in a controller test) are excluded. Each slice annotation is backed by a curated list of auto-configurations defined in spring-boot-test-autoconfigure.

@WebMvcTest

Loads only the MVC layer: controllers, filters, @ControllerAdvice, HandlerInterceptor, Jackson configuration, Spring Security MVC config. Excludes service and repository beans — you must mock them with @MockBean.

@WebMvcTest(UserController.class)
class UserControllerTest {

    @Autowired MockMvc mockMvc;
    @MockBean UserService userService; // must mock — not loaded by slice

    @Test
    void shouldReturnUser() throws Exception {
        given(userService.findById(1L))
            .willReturn(new UserDTO(1L, "Alice", "alice@example.com"));

        mockMvc.perform(get("/api/users/1").accept(MediaType.APPLICATION_JSON))
            .andExpect(status().isOk())
            .andExpect(jsonPath("$.name").value("Alice"))
            .andExpect(jsonPath("$.id").value(1));
    }

    @Test
    void shouldReturn404WhenNotFound() throws Exception {
        given(userService.findById(99L))
            .willThrow(new UserNotFoundException(99L));

        mockMvc.perform(get("/api/users/99"))
            .andExpect(status().isNotFound())
            .andExpect(jsonPath("$.error").value("User not found: 99"));
    }
}

@DataJpaTest

Loads JPA auto-configuration, Spring Data repositories, and an in-memory H2 database. No web layer. Each test method is wrapped in a transaction and rolled back automatically after the test. Use TestEntityManager to set up test data.

@DataJpaTest
class UserRepositoryTest {

    @Autowired UserRepository userRepository;
    @Autowired TestEntityManager em;

    @Test
    void shouldFindActiveUsersByRegion() {
        em.persistAndFlush(new User("Alice", "EMEA", true));
        em.persistAndFlush(new User("Bob",   "APAC", true));
        em.persistAndFlush(new User("Carol", "EMEA", false)); // inactive

        List<User> result = userRepository.findActiveByRegion("EMEA");

        assertThat(result).hasSize(1)
            .extracting(User::getName)
            .containsExactly("Alice");
    }
}

Other Important Slices

AnnotationLoadsUse For
@DataMongoTestMongoDB auto-config + repositories (embedded Flapdoodle)MongoDB repository tests
@WebFluxTestWebFlux layer — no TomcatReactive controller tests with WebTestClient
@JsonTestJackson ObjectMapper onlySerialization/deserialization contract tests

✅ Best PracticeUse @WebMvcTest(SpecificController.class) rather than @WebMvcTest without a class argument. Scoping to one controller keeps the loaded context minimal and test output focused.
🎯 Google L6 AngleA well-structured Spring project should have the majority of its controller tests as @WebMvcTest slices and repository tests as @DataJpaTest slices. If every test loads the full context, that signals architectural coupling — services leaking into controllers, repositories referenced in web layer — which itself is a design smell worth addressing.

Explain @MockBean vs @SpyBean. When does the Spring context get rebuilt?
MOCK_BEAN
▼

@MockBean

@MockBean adds a Mockito mock to the Spring ApplicationContext, replacing any existing bean of that type. The mock is automatically reset between test methods (Mockito reset() called). Use when you need a collaborator injected into other Spring beans to be a full mock with no real behavior.

@SpyBean

@SpyBean wraps an existing Spring bean with a Mockito spy. Real methods are invoked unless explicitly stubbed with doReturn() or given().willReturn(). Use when you want to verify interactions on a real bean or stub only specific methods while leaving others real.

@WebMvcTest(OrderController.class)
class OrderControllerTest {

    @Autowired MockMvc mockMvc;

    // Full mock — no DB calls, no real OrderService logic
    @MockBean OrderService orderService;

    // Spy on real AuditService bean — actual methods run, interactions verifiable
    @SpyBean AuditService auditService;

    @Test
    void shouldAuditOnOrderCreation() throws Exception {
        given(orderService.createOrder(any())).willReturn(new Order(1L, "PENDING"));

        mockMvc.perform(post("/api/orders").contentType(APPLICATION_JSON)
                .content("""{"customerId":42,"items":[{"sku":"ABC","qty":1}]}"""))
            .andExpect(status().isCreated());

        // Verify the spy's real method was called
        verify(auditService, times(1)).recordOrderCreation(eq(1L));
    }
}

@MockBean vs Plain Mockito.mock()

Aspect@MockBeanMockito.mock()
Spring ContextRegistered as a bean — injected into other beansNot in Spring context — pure Java object
Use CaseSlice/integration tests where mock must be injectedUnit tests with no Spring context
ResetAuto-reset after each test methodManual or @ExtendWith(MockitoExtension)
SpeedRequires context (slower)No context (very fast)

Context Rebuild Triggers

Spring's context cache key includes the full set of @MockBean and @SpyBean types used. If two test classes declare different sets of @MockBean, they cannot share a context — each triggers a fresh context build. This is the most common cause of unexpectedly slow test suites.

⚠ GotchaAdding @MockBean in individual test methods (not class fields) is not supported — they must be class-level fields. Also, @MockBean on a @Configuration class forces context reload for every test class that imports it differently.
✅ Best PracticeCreate a shared abstract base test class that declares all @MockBean fields. All concrete test subclasses inherit the same set, maximizing context cache reuse. This is one of the most impactful optimizations for test suite speed.
🎯 Google L6 AngleContext rebuild cost in a large project can accumulate to minutes. Auditing @MockBean usage and consolidating into shared base classes is a high-ROI refactoring. Profile context creation counts with Spring Boot's own startup metrics or with -Dspring.test.context.cache.maxSize logging.

How do TestContainers work with Spring Boot? Show best practices for startup time.
TESTCONTAINERS
▼

How TestContainers Works

TestContainers starts real Docker containers during test execution, providing actual infrastructure (PostgreSQL, Kafka, Redis, MongoDB, etc.) instead of in-memory fakes. This catches bugs that H2 or embedded mocks hide — dialect differences, constraint enforcement, real wire protocols, actual query plans.

Basic Setup

@SpringBootTest
@Testcontainers
class UserRepositoryIntegrationTest {

    @Container
    static PostgreSQLContainer<?> postgres =
        new PostgreSQLContainer<>("postgres:16-alpine")
            .withDatabaseName("testdb")
            .withUsername("test")
            .withPassword("test");

    @DynamicPropertySource
    static void configureDataSource(DynamicPropertyRegistry registry) {
        registry.add("spring.datasource.url",      postgres::getJdbcUrl);
        registry.add("spring.datasource.username", postgres::getUsername);
        registry.add("spring.datasource.password", postgres::getPassword);
    }

    @Autowired UserRepository userRepository;

    @Test
    void shouldPersistAndRetrieveUser() {
        User saved = userRepository.save(new User("Alice", "alice@example.com"));
        assertThat(userRepository.findByEmail("alice@example.com"))
            .isPresent()
            .hasValueSatisfying(u -> assertThat(u.getId()).isEqualTo(saved.getId()));
    }
}

Singleton Container Pattern (Critical for Speed)

By default, a @Container static field is started once per test class. If ten test classes each declare their own container, Docker starts ten PostgreSQL instances sequentially — very slow. The singleton pattern starts ONE container for the entire test run:

// Shared base class — all integration tests extend this
public abstract class AbstractIntegrationTest {

    static final PostgreSQLContainer<?> POSTGRES;
    static final KafkaContainer KAFKA;

    static {
        POSTGRES = new PostgreSQLContainer<>("postgres:16-alpine")
            .withReuse(true); // Testcontainers Desktop: reuse across JVM restarts
        POSTGRES.start();

        KAFKA = new KafkaContainer(DockerImageName.parse("confluentinc/cp-kafka:7.4.0"));
        KAFKA.start();
    }

    @DynamicPropertySource
    static void overrideProperties(DynamicPropertyRegistry registry) {
        registry.add("spring.datasource.url",      POSTGRES::getJdbcUrl);
        registry.add("spring.datasource.username", POSTGRES::getUsername);
        registry.add("spring.datasource.password", POSTGRES::getPassword);
        registry.add("spring.kafka.bootstrap-servers", KAFKA::getBootstrapServers);
    }
}

Spring Boot 3.1+ @ServiceConnection (Simplest Approach)

@SpringBootTest
@Testcontainers
class ModernIntegrationTest {

    // @ServiceConnection auto-wires all datasource properties — no @DynamicPropertySource!
    @Container
    @ServiceConnection
    static PostgreSQLContainer<?> postgres =
        new PostgreSQLContainer<>("postgres:16-alpine");

    @Container
    @ServiceConnection
    static RedisContainer redis =
        new RedisContainer(DockerImageName.parse("redis:7-alpine"));
}
⚠ GotchaContainers declared as instance fields (not static) are restarted per test method — extremely slow. Always declare integration test containers as static fields for class-level lifecycle, or use the singleton pattern for suite-level reuse.
✅ Best PracticeUse .withReuse(true) during local development to skip container restart between JVM runs. In CI, disable reuse (TESTCONTAINERS_REUSE_ENABLE=false) to ensure clean state per pipeline run.
🎯 Google L6 AngleTestContainers with the singleton pattern typically adds 5-10 seconds to a test suite — a worthwhile price for testing against the real database engine. The bugs it catches (Postgres-specific index behavior, JSONB queries, real foreign key constraints) are exactly the bugs that slip through H2 and surface in production.

Explain MockMvc. What is the difference between standaloneSetup and webAppContextSetup?
MOCK_MVC
▼

What MockMvc Is

MockMvc simulates HTTP requests through the Spring MVC infrastructure without starting a real servlet container. Requests go through the full dispatcher servlet processing — handler mapping, argument resolution, message conversion, exception handling — but over an in-memory mock, not a real TCP socket. This makes it both fast and accurate for testing the MVC layer.

standaloneSetup vs webAppContextSetup

SetupWhat It LoadsProsCons
standaloneSetup(controller)Only the specified controller. Manually configure Jackson, validators, exception resolvers.Extremely fast, no Spring context neededMisses global config: ControllerAdvice, filters, real Jackson config
webAppContextSetup(ctx)Full WebApplicationContext. All filters, interceptors, ControllerAdvice, real Jackson ObjectMapper.Fully representative, tests global cross-cutting concernsRequires Spring context (slower). Usually paired with @WebMvcTest or @SpringBootTest.

Full-Context MockMvc with @AutoConfigureMockMvc

@SpringBootTest
@AutoConfigureMockMvc
class OrderControllerIntegrationTest {

    @Autowired MockMvc mockMvc;

    @Test
    void createOrder_shouldReturn201WithLocation() throws Exception {
        String requestBody = """
            {
              "customerId": 42,
              "items": [{"sku": "SKU-001", "qty": 2}]
            }
            """;

        mockMvc.perform(post("/api/orders")
                .contentType(MediaType.APPLICATION_JSON)
                .content(requestBody)
                .header("Authorization", "Bearer " + generateTestJwt("user-42")))
            .andExpect(status().isCreated())
            .andExpect(header().exists("Location"))
            .andExpect(header().string("Location", matchesPattern("/api/orders/\\d+")))
            .andExpect(jsonPath("$.status").value("PENDING"))
            .andExpect(jsonPath("$.customerId").value(42))
            .andDo(print()); // prints full request/response to stdout for debugging
    }

    @Test
    void createOrder_withInvalidBody_shouldReturn400() throws Exception {
        mockMvc.perform(post("/api/orders")
                .contentType(APPLICATION_JSON)
                .content("{}"))
            .andExpect(status().isBadRequest())
            .andExpect(jsonPath("$.violations[0].field").value("customerId"));
    }
}

MockMvc Result Matchers

  • status().isOk(), status().isCreated(), status().isBadRequest()
  • jsonPath("$.field").value("expected"), jsonPath("$.list", hasSize(3))
  • header().string("Content-Type", containsString("application/json"))
  • content().json(expectedJson) — strict JSON equality with Jayway
  • cookie().exists("SESSION"), redirectedUrl("/login")
✅ Best PracticeUse WebTestClient for WebFlux (reactive) applications — it works for both blocking MVC and reactive WebFlux, has a fluent API, and supports request body templating more naturally than MockMvc's byte-array content.
🎯 Google L6 AngleA critical use of MockMvc at L6 is testing error response contracts. Ensure your @ControllerAdvice produces the correct RFC 7807 Problem Details format, with the right status codes and field names. Consumer-driven contract tests break if your error format changes unexpectedly.

How do you use WireMock to stub external HTTP services in Spring Boot tests?
WIREMOCK
▼

Why WireMock

WireMock is an embeddable HTTP server that replaces external HTTP dependencies (payment gateways, third-party APIs, microservice partners) with controllable stubs. Tests run without network access, execute deterministically, and can inject failure scenarios (timeouts, 500s, slow responses) that real APIs never deliver on demand.

Spring Boot 3.x WireMock Integration

@SpringBootTest(webEnvironment = RANDOM_PORT)
@AutoConfigureWireMock(port = 0) // port = 0 means random available port
class PaymentGatewayIntegrationTest {

    @Value("${wiremock.server.port}")
    int wireMockPort;

    @Autowired PaymentService paymentService;

    @BeforeEach
    void setupStubs() {
        // Happy path stub
        stubFor(post(urlEqualTo("/api/v1/charge"))
            .withHeader("Authorization",  containing("Bearer"))
            .withHeader("Content-Type",   equalTo("application/json"))
            .withRequestBody(matchingJsonPath("$.amount", equalTo("100")))
            .willReturn(aResponse()
                .withStatus(200)
                .withHeader("Content-Type", "application/json")
                .withBody("""
                    {
                      "transactionId": "txn-abc-123",
                      "status": "APPROVED",
                      "timestamp": "2024-01-15T10:00:00Z"
                    }
                    """)));
    }

    @Test
    void shouldChargePaymentSuccessfully() {
        PaymentResult result = paymentService.charge(
            new ChargeRequest(100, "USD", "card-tok-xyz"));

        assertThat(result.getTransactionId()).isEqualTo("txn-abc-123");
        assertThat(result.getStatus()).isEqualTo(PaymentStatus.APPROVED);

        // Verify the stub was actually called
        verify(1, postRequestedFor(urlEqualTo("/api/v1/charge")));
    }

    @Test
    void shouldRetryOnGatewayTimeout() {
        // Override with failure then success (WireMock scenarios)
        stubFor(post(urlEqualTo("/api/v1/charge"))
            .inScenario("retry-test")
            .whenScenarioStateIs(STARTED)
            .willReturn(aResponse().withStatus(504).withFixedDelay(5000))
            .willSetStateTo("second-attempt"));

        stubFor(post(urlEqualTo("/api/v1/charge"))
            .inScenario("retry-test")
            .whenScenarioStateIs("second-attempt")
            .willReturn(okJson("""{"transactionId":"txn-retry","status":"APPROVED"}""")));

        PaymentResult result = paymentService.charge(new ChargeRequest(100, "USD", "tok"));
        assertThat(result.getStatus()).isEqualTo(PaymentStatus.APPROVED);
        verify(2, postRequestedFor(urlEqualTo("/api/v1/charge")));
    }
}

WireMock Features Summary

  • Request matching: URL patterns, HTTP method, headers, query params, JSON/XML body (JsonPath, XPath), multipart
  • Response templating: dynamic responses using Handlebars — echo request fields back in response
  • Scenarios: stateful stubs — model a sequence of interactions (first call fails, second succeeds)
  • Fault injection: withFault(Fault.CONNECTION_RESET_BY_PEER), withFault(Fault.EMPTY_RESPONSE)
  • Fixed delay / random delay: test timeout handling and circuit breakers
⚠ GotchaWireMock port must be wired into your service's HTTP client base URL via @DynamicPropertySource or the @AutoConfigureWireMock property override. If the service has a hardcoded URL in application.yml that is not overridden in the test profile, stubs will never be hit.
✅ Best PracticeStore WireMock stub definitions as JSON files in src/test/resources/__files and mappings/ for complex scenarios. This keeps test code clean and makes stubs reviewable as test fixtures rather than inline strings.

What is contract testing? How does Spring Cloud Contract work?
CONTRACT_TESTING
▼

The Problem Contract Testing Solves

In microservices, teams deploy independently. Service A's consumer may break when Service B's provider changes a response field — but this is only discovered in staging or production. Integration tests that hit a deployed service don't scale. Contract testing solves this: both sides are verified independently against a shared contract before any deployment.

How Spring Cloud Contract Works

Contracts are defined in Groovy DSL or YAML on the provider side. The Spring Cloud Contract plugin auto-generates two artifacts: (1) JVM tests that verify the provider honors the contract, and (2) WireMock stubs that consumers can run against. Consumer CI tests use the stubs; provider CI tests verify live behavior.

// src/test/resources/contracts/user/find_user_by_id.groovy
Contract.make {
    description "GET /users/1 returns user with id 1"
    request {
        method GET()
        url "/users/1"
        headers { accept(applicationJson()) }
    }
    response {
        status OK()
        body(
            id:    1,
            name:  "Alice",
            email: "alice@example.com",
            roles: ["USER"]
        )
        headers { contentType(applicationJson()) }
    }
}

Provider Side — Auto-Generated Tests

// Generated test (do not modify) — but you provide the base class
@SpringBootTest(webEnvironment = RANDOM_PORT)
public class UserContractTest extends UserContractTestBase {
    // Spring Cloud Contract generates the actual test method
    // You provide the base class with setup (mock beans, test data)
}

// Your base class
@SpringBootTest(webEnvironment = RANDOM_PORT)
public abstract class UserContractTestBase {
    @Autowired WebApplicationContext context;

    @BeforeEach
    void setup() {
        RestAssuredMockMvc.webAppContextSetup(context);
        // Seed test data: user with id=1, name="Alice" must exist
    }
}

Consumer Side — Running Against Stubs

@SpringBootTest
@AutoConfigureStubRunner(
    stubsMode = StubRunnerProperties.StubsMode.LOCAL,
    ids = "com.example:user-service:+:stubs:8090"  // groupId:artifactId:version:classifier:port
)
class UserClientTest {

    @Autowired UserServiceClient userClient; // your RestTemplate/WebClient wrapper

    @Test
    void shouldFetchUserById() {
        // Stub runner starts WireMock on port 8090 with the contract stubs
        UserDTO user = userClient.getUser(1L);
        assertThat(user.getName()).isEqualTo("Alice");
        assertThat(user.getRoles()).contains("USER");
    }
}

🎯 Google L6 AngleContract testing is the engineering practice that enables independent team deployments in microservices. @Transactional in tests — what does it do? What are the gotchas?
TX_TESTS
▼

What @Transactional Does in Tests

When @Transactional is placed on a test class or method, Spring wraps the entire test in a transaction that is automatically rolled back after the test completes. This gives you a clean database state for every test without needing explicit cleanup code. @DataJpaTest applies this by default.

@DataJpaTest
class UserRepositoryTest {

    @Autowired UserRepository userRepo;
    @Autowired TestEntityManager em;

    @Test
    // @Transactional applied by @DataJpaTest — rollback is automatic
    void shouldFindByEmail() {
        em.persistAndFlush(new User("Alice", "alice@example.com"));

        Optional<User> found = userRepo.findByEmail("alice@example.com");

        assertThat(found)
            .isPresent()
            .hasValueSatisfying(u -> {
                assertThat(u.getName()).isEqualTo("Alice");
                assertThat(u.getEmail()).isEqualTo("alice@example.com");
            });
    }
    // After test: transaction rolled back. Database is clean for the next test.
}

// Force commit within a transactional test (rare but sometimes needed)
@Test
@Commit // overrides rollback — actual commit to DB
void shouldCommitForVerification() { ... }

Gotcha 1: @Transactional + RANDOM_PORT = Rollback Does NOT Work

With @SpringBootTest(webEnvironment = RANDOM_PORT), HTTP requests are handled by a separate server thread with its own transaction context. The test-thread transaction is invisible to the server thread. Any data committed by the server is NOT rolled back by the test's @Transactional annotation.

// WRONG - @Transactional here does nothing useful
@SpringBootTest(webEnvironment = RANDOM_PORT)
@Transactional // ← has NO effect on data written by the HTTP server thread
class OrderApiTest {
    // Use @Sql or @AfterEach cleanup instead
}

Gotcha 2: LazyInitializationException in Tests

@DataJpaTest
class OrderTest {
    @Test
    void fetchOrderItems() {
        Order order = orderRepo.findById(1L).get();
        // Session still open inside @Transactional test — this works fine
        order.getItems().size(); // OK — same transaction/session

        // BUT if you call outside @Transactional scope → LazyInitializationException
    }
}

Gotcha 3: Transactions Not Propagated Across Threads in Tests

If your service method uses @Async or creates new threads, those threads have NO access to the test's transaction. They start their own, which commit independently and won't be rolled back. Use @DirtiesContext or explicit DB cleanup in @AfterEach for async tests.

Correct Pattern for Integration Tests with RANDOM_PORT

@SpringBootTest(webEnvironment = RANDOM_PORT)
@Testcontainers
class OrderIntegrationTest extends AbstractIntegrationTest {

    @Autowired OrderRepository orderRepo;

    @AfterEach
    void cleanup() {
        orderRepo.deleteAll(); // explicit cleanup — rollback not available
    }

    // OR use @Sql annotations
    @Test
    @Sql(scripts = "/test-data/orders.sql")
    @Sql(scripts = "/test-data/cleanup.sql", executionPhase = AFTER_TEST_METHOD)
    void shouldCreateOrder() { ... }
}
✅ Best PracticeUse @Transactional (rollback) for @DataJpaTest repository tests. Use explicit cleanup (@AfterEach, @Sql, or Flyway test migrations) for @SpringBootTest integration tests that exercise real HTTP. Never mix the two patterns.

Explain @TestConfiguration, @Import in tests, and how to isolate test beans.
TEST_CONFIG
▼

@TestConfiguration

@TestConfiguration is a specialization of @Configuration that is excluded from production component scanning. It is only activated when explicitly imported or when used as a nested class inside a test class. This prevents test beans from accidentally polluting the production context.

// Standalone test configuration — NOT picked up by production scan
@TestConfiguration
public class TestSecurityConfig {

    @Bean
    @Primary  // overrides the real UserDetailsService for this test context
    public UserDetailsService testUserDetailsService() {
        return username -> User.withUsername(username)
            .password("{noop}password")
            .roles("USER", "ADMIN")
            .build();
    }

    @Bean
    @Primary
    public PasswordEncoder testPasswordEncoder() {
        return NoOpPasswordEncoder.getInstance(); // fast, no bcrypt in tests
    }
}

// Import into specific test class
@WebMvcTest(UserController.class)
@Import(TestSecurityConfig.class) // explicitly activated here
class UserControllerSecurityTest {
    @Autowired MockMvc mockMvc;

    @Test

What is ArchUnit? How do you use it to enforce architecture rules in Spring apps?

▼

What ArchUnit Is

ArchUnit is a Java library that lets you write architecture constraints as unit tests. It analyzes compiled bytecode, allowing you to enforce layer boundaries, naming conventions, dependency rules, and cycle-freeness as part of your regular test suite — no separate analysis tool required.

Layered Architecture Rules

@AnalyzeClasses(packages = "com.example.myapp")
class ArchitectureTest {

    // Services must NOT depend on controllers
    @ArchTest
    static final ArchRule servicesMustNotDependOnControllers =
        noClasses()
            .that().resideInAPackage("..service..")
            .should().dependOnClassesThat().resideInAPackage("..controller..");

    // Repositories must only be accessed from services (and other repositories)
    @ArchTest
    static final ArchRule repositoriesAccessibleFromServicesOnly =
        classes()
            .that().resideInAPackage("..repository..")
            .should().onlyBeAccessed()
            .byAnyPackage("..service..", "..repository..");

    // All controllers must be annotated with @RestController
    @ArchTest
    static final ArchRule controllerAnnotationRule =
        classes()
            .that().resideInAPackage("..controller..")
            .and().areNotInterfaces()
            .should().beAnnotatedWith(RestController.class);

    // No circular dependencies between packages
    @ArchTest
    static final ArchRule noCyclicDependencies =
        slices().matching("com.example.myapp.(*)..").should().beFreeOfCycles();

    // Predefined layered architecture
    @ArchTest
    static final ArchRule layeredArchitecture = layeredArchitecture()
        .consideringAllDependencies()
        .layer("Controller").definedBy("..controller..")
        .layer("Service").definedBy("..service..")
        .layer("Repository").definedBy("..repository..")
        .whereLayer("Controller").mayNotBeAccessedByAnyLayer()
        .whereLayer("Service").mayOnlyBeAccessedByLayers("Controller")
        .whereLayer("Repository").mayOnlyBeAccessedByLayers("Service");
}

Naming Convention Rules

@ArchTest
static final ArchRule serviceNamingConvention =
    classes()
        .that().resideInAPackage("..service..")
        .and().areNotInterfaces()
        .and().areNotAnnotatedWith(Configuration.class)
        .should().haveSimpleNameEndingWith("Service");

@ArchTest
static final ArchRule noFieldInjection =
    noFields()
        .should().beAnnotatedWith(Autowired.class)
        .because("Use constructor injection for testability and immutability");

Freezing Violations for Gradual Adoption

// Allow existing violations to freeze; prevent NEW violations
@ArchTest
static final ArchRule frozenLayerRule = FreezingArchRule.freeze(
    noClasses()
        .that().resideInAPackage("..service..")
        .should().dependOnClassesThat().resideInAPackage("..controller..")
);
// Violations recorded in .arch-freezeStore/ directory (commit to version control)
// Any new violation beyond frozen set fails the test
✅ Best PracticeStart with cycle detection and the predefined layered architecture rule. These two alone catch the most common drift patterns. Add naming conventions and annotation rules incrementally as the team builds familiarity.
🎯 Google L6 AngleArchUnit prevents architectural drift — the silent erosion of a well-designed layered architecture as engineers add "one quick fix" that creates cross-layer dependencies. These violations compound over years: what starts as a service importing a DTO from a controller becomes a tangle of circular dependencies. L6 engineers design systems to enforce their own correctness, and ArchUnit is a lightweight way to do that for Java package structure.

JMH — Java Microbenchmark Harness

JMH is the authoritative tool for measuring individual method performance in the JVM. It handles JVM warmup, JIT compilation stabilization, dead-code elimination prevention, and statistical measurement — all pitfalls of naive benchmarking. Use it to measure serialization speed, algorithm performance, cache lookup times, or any method you suspect is a hot path.

@BenchmarkMode(Mode.AverageTime)
@OutputTimeUnit(TimeUnit.MICROSECONDS)
@State(Scope.Benchmark)
@Fork(2)
@Warmup(iterations = 5, time = 1)
@Measurement(iterations = 10, time = 1)
public class OrderSerializationBenchmark {

    @State(Scope.Benchmark)
    public static class BenchmarkState {
        ObjectMapper jacksonMapper = new ObjectMapper();
        Order order = Order.builder()
            .id(1L).customerId(42L).status("PENDING")
            .items(List.of(new OrderItem("SKU-001", 2, BigDecimal.TEN)))
            .build();
        byte[] serializedBytes = /* pre-serialize once */;
    }

    @Benchmark
    public String serializeWithJackson(BenchmarkState s) throws Exception {
        return s.jacksonMapper.writeValueAsString(s.order);
    }

    @Benchmark
    public Order deserializeWithJackson(BenchmarkState s) throws Exception {
        return s.jacksonMapper.readValue(s.serializedBytes, Order.class);
    }
}

Gatling CI Load Test Example

class OrderApiSimulation extends Simulation {

    val httpConf = http.baseUrl("http://localhost:8080")
        .acceptHeader("application/json")
        .contentTypeHeader("application/json")

    val createOrder = scenario("Create Order")
        .exec(http("POST /api/orders")
            .post("/api/orders")
            .body(StringBody("""{"customerId":42,"items":[{"sku":"SKU-1","qty":1}]}"""))
            .check(status.is(201))
            .check(responseTimeInMillis.lt(200))) // assert <200ms per request

    setUp(
        createOrder.inject(
            rampUsersPerSec(10).to(100).during(60.seconds), // ramp up
            constantUsersPerSec(100).during(120.seconds)    // sustain load
        )
    ).assertions(
        global.successfulRequests.percent.gt(99.0),  // 99% success rate
        global.responseTime.percentile(95).lt(500),   // p95 < 500ms
        global.responseTime.percentile(99).lt(1000)   // p99 < 1 second
    )
}
