How does @KafkaListener work internally? Explain the consumer container lifecycle.
KAFKA_LISTENER
▼

Internal Mechanism

When Spring detects a @KafkaListener-annotated method, a KafkaListenerEndpointRegistrar registers the method as a MethodKafkaListenerEndpoint. The KafkaListenerContainerFactory creates a message listener container — either a KafkaMessageListenerContainer (single consumer thread) or a ConcurrentMessageListenerContainer (N consumer threads) based on the concurrency setting.

Each container internally creates a KafkaConsumer (from the kafka-clients library), subscribes to the specified topics or topic patterns, and starts a dedicated polling thread. The polling thread calls consumer.poll(Duration) in a loop, dispatching received records to the annotated method.

Container Lifecycle

  1. start(): creates KafkaConsumer, subscribes to topics, joins consumer group, starts polling loop
  2. polling loop: poll → dispatch records → process → acknowledge/commit offsets → repeat
  3. stop(): signals polling thread to stop, commits current offsets, closes KafkaConsumer
  4. pause() / resume(): suspend/resume partition consumption without leaving the group
@KafkaListener(
    topics = "orders",
    groupId = "order-processor",
    concurrency = "3",                         // 3 concurrent consumer threads
    containerFactory = "kafkaListenerContainerFactory"
)
public void processOrder(
        ConsumerRecord<String, OrderEvent> record,
        @Header(KafkaHeaders.RECEIVED_PARTITION) int partition,
        @Header(KafkaHeaders.OFFSET) long offset,
        Acknowledgment ack) {

    try {
        orderService.process(record.value());
        ack.acknowledge();                     // manual offset commit
    } catch (RetryableException e) {
        ack.nack(Duration.ofSeconds(5));       // nack: pause, retry after 5s
    } catch (FatalException e) {
        ack.acknowledge();                     // commit to skip poison pill
        deadLetterService.send(record);        // send to DLT manually
    }
}

// ConcurrentKafkaListenerContainerFactory configuration
@Bean
public ConcurrentKafkaListenerContainerFactory<String, OrderEvent> kafkaListenerContainerFactory(
        ConsumerFactory<String, OrderEvent> cf) {
    ConcurrentKafkaListenerContainerFactory<String, OrderEvent> factory =
        new ConcurrentKafkaListenerContainerFactory<>();
    factory.setConsumerFactory(cf);
    factory.setConcurrency(3);
    factory.getContainerProperties().setAckMode(ContainerProperties.AckMode.MANUAL_IMMEDIATE);
    factory.getContainerProperties().setPollTimeout(3000);
    factory.setCommonErrorHandler(errorHandler());
    return factory;
}

⚠ Gotchaconcurrency is bounded by the number of partitions. If you set concurrency=6 on a topic with 3 partitions, 3 threads will be idle — Kafka assigns at most one partition per consumer in a group. Always set concurrency <= partition count. Plan partition count at topic creation time; it cannot be reduced without data loss.
✅ Best PracticeUse MANUAL or MANUAL_IMMEDIATE AckMode in production for all business-critical consumers. With BATCH mode, a crash mid-batch means all records in the batch are reprocessed — potentially hundreds. MANUAL gives you fine-grained control and makes at-least-once semantics explicit and auditable.
🎯 Google L6 AngleConsumer lag is the key operational metric: the difference between the latest offset (log end offset) and the committed offset. Monitor consumer lag per partition, not just per consumer group. A single partition with growing lag indicates an uneven partition key distribution (hot partition). Design partition keys for even distribution: hash(userId) % partitions works better than hash(countryCode) if traffic is geo-skewed.

Explain Kafka consumer groups. How does partition assignment work? What happens on rebalance?
CONSUMER_GROUPS
▼

Consumer Groups

A consumer group is a set of consumers that collectively consume a topic. Kafka guarantees that each partition is consumed by exactly one consumer within a group at a time. This provides horizontal scaling: more consumers = more partitions processed in parallel.

Multiple consumer groups on the same topic each receive all messages independently — one group gets every message (broadcast semantics). Within a single group, messages are load-balanced across consumers (queue semantics).

Partition Assignment Strategies

  • RangeAssignor (default): assigns contiguous ranges of partitions per topic. Can be uneven if partition count is not divisible by consumer count.
  • RoundRobinAssignor: distributes partitions evenly round-robin. Better balance across topics.
  • StickyAssignor: minimizes partition movement on rebalance. Retains previous assignments where possible, reducing cache warming overhead.
  • CooperativeStickyAssignor (Kafka 2.4+): incremental cooperative rebalance — only affected partitions stop. The rest continue processing during rebalance. Eliminates the stop-the-world pause.

Rebalance Triggers and Impact

Rebalance is triggered when: a consumer joins the group, a consumer leaves (crash or graceful shutdown), a heartbeat times out (session.timeout.ms), max poll interval exceeded (max.poll.interval.ms), topic partition count changes.

Classic (eager) rebalance: ALL consumers stop consuming, revoke all partitions, rebalance, then resume. The stop-the-world pause causes latency spikes and can trigger further heartbeat timeouts if the rebalance itself is slow — leading to rebalance storms.

# application.yml — tuning rebalance behavior
spring:
  kafka:
    consumer:
      group-id: order-processor
      max-poll-records: 500             # records per poll
      max-poll-interval-ms: 300000      # 5 min — processing budget
      session-timeout-ms: 45000         # 45s before broker declares consumer dead
      heartbeat-interval-ms: 15000      # 3x less than session timeout
      partition-assignment-strategy:
        - org.apache.kafka.clients.consumer.CooperativeStickyAssignor

# Spring Kafka listener — rebalance lifecycle hooks
@KafkaListener(topics = "orders", groupId = "order-processor")
public class OrderListener implements ConsumerSeekAware {

    @Override
    public void onPartitionsAssigned(Map<TopicPartition, Long> assignments,
                                     ConsumerSeekCallback callback) {
        // warm caches, initialize per-partition state
        log.info("Assigned partitions: {}", assignments.keySet());
    }

    @Override
    public void onPartitionsRevoked(Collection<TopicPartition> partitions) {
        // flush per-partition state, cleanup
        log.info("Revoked partitions: {}", partitions);
    }
}
⚠ Gotchamax.poll.interval.ms is the most common source of unexpected rebalances in production. If processing a batch of records takes longer than this value (default 5 minutes), the consumer is removed from the group and a rebalance triggers. This is especially dangerous with slow downstream calls (DB writes, HTTP calls). Solution: reduce max.poll.records, increase max.poll.interval.ms, or process records asynchronously with manual acknowledgment.
✅ Best PracticeMigrate to CooperativeStickyAssignor for any production consumer group. Incremental cooperative rebalance eliminates the stop-the-world pause — only the subset of partitions being moved stop briefly, while all other partitions continue processing. The operational improvement is dramatic on groups with many partitions.
🎯 Google L6 AngleRebalance storms are a real production hazard at scale. Pattern: high load → slow record processing → max.poll.interval.ms exceeded → consumer kicked from group → rebalance → other consumers take on extra partitions → they also slow down → cascade. Prevention: size your consumer groups conservatively (headroom for load spikes), use CooperativeStickyAssignor, keep processing per batch bounded and fast, and alert on rebalance frequency as a leading indicator of trouble.

Explain at-most-once, at-least-once, and exactly-once delivery in Kafka. How does Spring Kafka support each?
DELIVERY
▼

At-Most-Once Delivery

Commit the offset before processing the record. If the consumer crashes after committing but before completing processing, the record is lost — it will not be redelivered. Fastest; acceptable for telemetry, metrics, or analytics where occasional loss is tolerable. Rarely correct for business logic.

At-Least-Once Delivery (Default in Spring Kafka)

Process the record then commit the offset. If the consumer crashes after processing but before committing, the record is redelivered and processed again. This is the default behavior with AckMode.BATCH. Consumer processing must be idempotent — the same record processed twice must not cause double-effects (e.g., double charging a customer).

Exactly-Once Semantics (EOS)

Kafka transactions enable atomic read-process-write: consume from input topic, produce to output topic, and commit the input offset — all as a single atomic transaction. Consumers with isolation.level=read_committed skip messages from aborted transactions.

// 1. Configure idempotent producer (deduplicates retries at broker)
spring:
  kafka:
    producer:
      transaction-id-prefix: order-processor-   # enables transactions
      acks: all
      enable-idempotence: true

// 2. KafkaTransactionManager bean
@Bean
public KafkaTransactionManager<String, Object> kafkaTransactionManager(
        ProducerFactory<String, Object> pf) {
    return new KafkaTransactionManager<>(pf);
}

// 3. Transactional listener: consume + produce in one Kafka TX
@Transactional("kafkaTransactionManager")
@KafkaListener(topics = "raw-orders", groupId = "order-enricher")
public void enrichAndForward(ConsumerRecord<String, RawOrder> record) {
    EnrichedOrder enriched = enrichmentService.enrich(record.value());
    kafkaTemplate.send("enriched-orders", enriched);
    // Kafka TX: input offset committed + output published atomically
    // If either fails: both roll back. Consumer re-processes from last committed offset.
}

// Consumer with read_committed isolation (skips aborted TX messages)
spring:
  kafka:
    consumer:
      isolation-level: read_committed

Idempotent Consumer Pattern (At-Least-Once + Idempotency)

For most production use cases: at-least-once delivery + idempotent consumer processing is the practical choice. Exactly-once transactions add ~10–20ms latency overhead and significant operational complexity.

GuaranteeOffset CommitRiskSpring AckModeUse Case
At-most-onceBefore processingData lossauto.offset.reset + autocommitMetrics, telemetry
At-least-onceAfter processingDuplicatesMANUAL / BATCHMost business events (with idempotent consumer)
Exactly-onceAtomic with outputLatency, complexityKafkaTransactionManagerFinancial, billing, critical state machines
⚠ Gotchaenable.idempotence=true on the producer prevents duplicate messages from producer retries (e.g., network blip causes retry → broker deduplicates by sequence number). But it does NOT prevent duplicate consumer processing — that requires at-least-once + idempotent consumer logic. These are independent mechanisms solving different problems.
✅ Best PracticeDefault to at-least-once delivery with an idempotent consumer. Exactly-once Kafka transactions are semantically powerful but operationally expensive — they require careful producer configuration, read_committed consumers, and add latency. Reserve them for read-process-write pipelines where you cannot make the processing idempotent.
🎯 Google L6 AngleThe phrase "exactly-once" in Kafka is often misunderstood. Kafka EOS guarantees exactly-once for the Kafka-to-Kafka pipeline (consume → transform → produce). If your consumer writes to a DB or calls an external API, you're back to at-least-once — those side effects are not inside the Kafka transaction boundary. Design idempotent DB writes and API calls regardless of which delivery guarantee you choose at the Kafka layer.

Explain dead letter topics in Spring Kafka. How do you configure and consume from a DLT?
DLT
▼

What is a Dead Letter Topic?

When a message fails processing repeatedly (max retries exhausted), it is forwarded to a Dead Letter Topic (DLT) instead of blocking the consumer indefinitely. The DLT stores failed messages for manual inspection, alerting, replay after a fix, or selective discard. Without a DLT, a single "poison pill" message can block an entire partition forever.

DefaultErrorHandler + DeadLetterPublishingRecoverer (Spring Kafka 2.8+)

@Bean
public DefaultErrorHandler errorHandler(KafkaOperations<String, Object> template) {
    // Recoverer: send failed record to {topic}.DLT, same partition
    DeadLetterPublishingRecoverer recoverer = new DeadLetterPublishingRecoverer(template,
        (record, ex) -> new TopicPartition(
            record.topic() + ".DLT",
            record.partition()   // preserve partition for ordering
        ));

    // Retry 3 times with 1-second fixed delay before sending to DLT
    FixedBackOff backOff = new FixedBackOff(1000L, 3L);

    // Alternatively: exponential backoff
    // ExponentialBackOff backOff = new ExponentialBackOff(1000L, 2.0);
    // backOff.setMaxElapsedTime(30_000L);

    DefaultErrorHandler handler = new DefaultErrorHandler(recoverer, backOff);

    // These exceptions skip retries entirely — go straight to DLT
    handler.addNotRetryableExceptions(
        SerializationException.class,
        ValidationException.class,
        NullPointerException.class
    );

    return handler;
}

@Bean
public ConcurrentKafkaListenerContainerFactory<?, ?> kafkaListenerContainerFactory(
        ConsumerFactory<?, ?> cf, DefaultErrorHandler errorHandler) {
    var factory = new ConcurrentKafkaListenerContainerFactory<>();
    factory.setConsumerFactory(cf);
    factory.setCommonErrorHandler(errorHandler);
    return factory;
}

DLT Consumer

@KafkaListener(topics = "orders.DLT", groupId = "orders-dlt-processor")
public void processDLT(
        ConsumerRecord<String, OrderEvent> record,
        @Header(KafkaHeaders.EXCEPTION_MESSAGE) String exceptionMessage,
        @Header(KafkaHeaders.ORIGINAL_OFFSET) long originalOffset,
        @Header(KafkaHeaders.ORIGINAL_PARTITION) int originalPartition) {

    log.error("DLT message from orders partition={} offset={}: {}",
        originalPartition, originalOffset, exceptionMessage);

    // Options:
    // 1. Store in DB for manual review dashboard
    // 2. Alert on-call via PagerDuty
    // 3. Replay to original topic after fix
    dlqRepository.save(new DLQEntry(record, exceptionMessage));
    alertService.notifyOnCall(record, exceptionMessage);
}

@RetryableTopic — Non-Blocking Retry (Spring Kafka 2.7+)

// Creates multiple retry topics with increasing delays
// No blocking — consumer continues processing other records while retrying
@RetryableTopic(
    attempts = "4",                         // 1 original + 3 retries
    backoff = @Backoff(delay = 1000, multiplier = 2.0),  // 1s, 2s, 4s
    dltStrategy = DltStrategy.FAIL_ON_ERROR,
    topicSuffixingStrategy = TopicSuffixingStrategy.SUFFIX_WITH_DELAY_VALUE
)
@KafkaListener(topics = "orders")
public void processOrder(Order order) {
    orderService.process(order);  // thrown exceptions trigger retry
}
// Auto-creates topics: orders, orders-retry-1000, orders-retry-2000, orders-retry-4000, orders-DLT
// Non-blocking: retry messages are published to retry topics and consumed later
⚠ GotchaThe DLT must be pre-created with sufficient partitions if using partition-preserving routing. If the DLT has fewer partitions than the source topic, DeadLetterPublishingRecoverer will fail trying to route to a non-existent partition. Either pre-create the DLT with matching partition count, or use a destination resolver that falls back to partition 0.
✅ Best PracticeAlways separate retryable from non-retryable exceptions. SerializationException (bad payload format) will NEVER succeed on retry — retrying it 3 times with delays just wastes time and blocks the partition. Use addNotRetryableExceptions() for all exceptions that indicate bad data. Use retries only for transient failures (DB connection error, HTTP 503).
🎯 Google L6 AngleA DLT without monitoring is a graveyard. Messages silently accumulate, and nobody knows. At minimum: alert when DLT consumer lag grows (messages arriving faster than being reviewed), track DLT message count per original topic in a dashboard, and build a replay tool that can re-publish DLT messages back to the source topic after the underlying bug is fixed. The replay capability is what transforms a DLT from a dead end into a recovery mechanism.

How does KafkaTemplate work? Explain producer configuration for reliability.
KAFKA_PRODUCER
▼

KafkaTemplate Internals

KafkaTemplate is Spring's Kafka producer. It wraps a ProducerFactory that manages KafkaProducer instances. By default, the ProducerFactory is DefaultKafkaProducerFactory which maintains a single shared producer (thread-safe) per factory.

send() returns a CompletableFuture<SendResult<K, V>> (Spring Kafka 3.x; ListenableFuture in earlier versions). Use whenComplete for async handling or .get() for synchronous (blocking) confirmation.

@Service
public class OrderProducer {

    @Autowired
    private KafkaTemplate<String, OrderEvent> kafkaTemplate;

    // Async send with callback — preferred for throughput
    public void sendOrder(OrderEvent event) {
        kafkaTemplate.send("orders", event.getOrderId().toString(), event)
            .whenComplete((result, ex) -> {
                if (ex != null) {
                    log.error("Failed to send order {}: {}", event.getOrderId(), ex.getMessage());
                    // compensate: retry, fallback to outbox, alert
                } else {
                    RecordMetadata meta = result.getRecordMetadata();
                    log.info("Sent order {} to {}-{} at offset {}",
                        event.getOrderId(),
                        meta.topic(), meta.partition(), meta.offset());
                }
            });
    }

    // Synchronous send — use only when caller must confirm delivery
    public void sendOrderSync(OrderEvent event) throws ExecutionException, InterruptedException {
        SendResult<String, OrderEvent> result =
            kafkaTemplate.send("orders", event.getOrderId().toString(), event)
                         .get(10, TimeUnit.SECONDS);  // timeout prevents indefinite block
        log.info("Confirmed delivery to offset {}", result.getRecordMetadata().offset());
    }
}

Reliability Configuration

spring:
  kafka:
    producer:
      acks: all                   # leader + all ISR replicas must acknowledge
      retries: 5                  # auto-retry transient failures
      enable-idempotence: true    # sequence numbers; broker deduplicates retries
      compression-type: lz4       # ~4x size reduction, fast compression
      batch-size: 32768           # 32KB batch — increase for higher throughput
      linger-ms: 10               # wait 10ms to fill batches before sending
      buffer-memory: 33554432     # 32MB producer buffer
      max-block-ms: 60000         # max wait if buffer full (backpressure)

acks Configuration Impact

acksDurabilityLatency ImpactUse Case
0None (fire and forget)LowestMetrics, high-volume telemetry

⚠ Gotchaenable.idempotence=true requires acks=all and retries > 0 (which is now the default). If you set acks=1 alongside enable.idempotence=true, the producer will throw a ConfigException at startup. These settings are interdependent — idempotence requires the highest durability guarantee to work correctly.
✅ Best PracticeNever use fire-and-forget (no callback, no .get()) for business-critical events. The producer buffer can fill up and silently drop records if the broker is slow. Always attach a whenComplete callback that handles failures — at minimum, logging and alerting. For critical paths, consider a circuit breaker around Kafka sends to fail fast when the broker is unreachable rather than blocking threads on max-block-ms.
🎯 Google L6 AngleProducer batching (batch-size + linger-ms) is the primary throughput lever. A 10ms linger with 32KB batches can increase throughput 10–20x compared to per-record sends with minimal latency cost for most use cases. Profile your producer with and without batching under realistic load — the difference is almost always worth the extra 10ms latency. For latency-sensitive paths, linger-ms=0 is correct; for high-throughput ingestion, linger-ms=5-20 is the sweet spot.

Explain Spring AMQP and RabbitMQ. Compare to Kafka — when do you use each?
RABBITMQ
▼

Spring AMQP Core Concepts

Spring AMQP provides a higher-level abstraction over the AMQP protocol. RabbitTemplate sends messages; @RabbitListener consumes them. Core AMQP model:

  • Exchange: receives messages from producers; routes to queues based on bindings
  • Queue: stores messages until consumed; durable queues survive broker restarts
  • Binding: rule connecting an exchange to a queue (with optional routing key)
  • Exchange types: direct (exact routing key), topic (pattern matching, *.order.#), fanout (broadcast to all bound queues), headers (attribute-based routing)
@Configuration
public class RabbitConfig {
    @Bean Queue orderQueue() {
        return QueueBuilder.durable("order-queue")
            .withArgument("x-dead-letter-exchange", "dlx")   // DLX routing
            .withArgument("x-message-ttl", 3600000)          // message TTL 1hr
            .build();
    }
    @Bean TopicExchange orderExchange() { return new TopicExchange("order-exchange"); }
    @Bean Binding orderBinding(Queue q, TopicExchange ex) {
        return BindingBuilder.bind(q).to(ex).with("orders.#");
    }
}

// Consumer with manual acknowledgment
@RabbitListener(queues = "order-queue", concurrency = "3-10")  // min 3, max 10 consumers
public void processOrder(
        @Payload Order order,
        @Header(AmqpHeaders.DELIVERY_TAG) long tag,
        Channel channel) throws IOException {

    try {
        orderService.process(order);
        channel.basicAck(tag, false);                // ack: message processed
    } catch (RetryableException e) {
        channel.basicNack(tag, false, true);         // nack + requeue
    } catch (FatalException e) {
        channel.basicNack(tag, false, false);        // nack + dead-letter (no requeue)
    }
}

// Sending
rabbitTemplate.convertAndSend("order-exchange", "orders.created", order);

Decision Guide

  • Choose Kafka: event streaming, audit log, high-throughput ingestion, event sourcing, stream processing, replay required, multiple independent consumers
  • Choose RabbitMQ: task queues, work distribution, complex routing (multiple exchange types), request-reply RPC patterns, priority queues, when messages should be deleted after consumption
⚠ GotchaRabbitMQ's basicNack with requeue=true on a poison pill message creates an infinite retry loop — the message bounces between the queue and the consumer indefinitely, consuming broker and consumer resources. Always configure a Dead Letter Exchange (DLX) and set a max-retry count (via message headers + custom rejection logic) to break the loop.
✅ Best PracticeFor RabbitMQ, always declare queues as durable and messages as persistent (delivery-mode=2) for business-critical workloads. Non-durable queues and non-persistent messages are lost on broker restart. Use publisher confirms (RabbitTemplate with ConfirmCallback) to guarantee the broker received the message before returning to the caller.
🎯 Google L6 AngleThe replay capability is the defining architectural difference. Kafka's immutable log lets you re-process historical events with new business logic — invaluable for debugging production incidents, backfilling new data pipelines, or recovering from consumer bugs. If you might ever need to ask "what happened to all orders from last Tuesday?", Kafka is the right choice. RabbitMQ's transient model makes this impossible without a separate audit store.

JSON Serialization

Human-readable, easy to debug with console consumers, no schema enforcement. Spring's JsonSerializer/JsonDeserializer wraps Jackson. Trade-off: schema drift over time — a producer adding a required field silently breaks consumers that don't handle missing fields. No compile-time contract between producer and consumer teams.

Apache Avro + Schema Registry

Explain Kafka transactions in Spring. How do you achieve atomic DB write + Kafka publish?
KAFKA_TX
▼

The Core Problem

You need to save an Order to the DB AND publish an OrderPlaced event to Kafka. If DB succeeds but Kafka fails (or vice versa), your system is inconsistent: the order exists in the DB but downstream services never see the event — or the event fires for an order that was rolled back.

Naive Approach — Chained Transactions (Flawed)

// ANTI-PATTERN: ChainedTransactionManager is not atomic
@Transactional
public void placeOrder(Order order) {
    orderRepo.save(order);               // DB transaction
    kafkaTemplate.send("orders", event); // Kafka "transaction"
    // Problem: these are NOT atomic. DB can commit and Kafka can fail.
    // Or: Kafka commits first, then DB rolls back.
    // Result: inconsistency.
}

Correct Solution: Transactional Outbox Pattern

Write the event to an outbox table in the same DB transaction as the business record. A separate relay process reads the outbox and publishes to Kafka. This guarantees: if the DB transaction commits, the event will eventually be published. If the DB rolls back, the event is never written.

// Step 1: Outbox table entity
@Entity
@Table(name = "outbox_events")
public class OutboxEvent {
    @Id
    private UUID id = UUID.randomUUID();
    private String aggregateType;   // "Order"
    private String aggregateId;     // order ID
    private String eventType;       // "OrderPlaced"
    @Column(columnDefinition = "TEXT")
    private String payload;         // JSON event payload
    private boolean published = false;
    private LocalDateTime createdAt = LocalDateTime.now();
}

// Step 2: Service — atomic DB write + outbox write in ONE transaction
@Service
@Transactional
public class OrderService {

    @Autowired private OrderRepository orderRepo;
    @Autowired private OutboxEventRepository outboxRepo;
    @Autowired private ObjectMapper objectMapper;

    public Order placeOrder(CreateOrderDTO dto) throws JsonProcessingException {
        Order order = orderRepo.save(new Order(dto));          // save order

        OutboxEvent outbox = new OutboxEvent();
        outbox.setAggregateType("Order");
        outbox.setAggregateId(order.getId().toString());
        outbox.setEventType("OrderPlaced");
        outbox.setPayload(objectMapper.writeValueAsString(
            new OrderPlacedEvent(order.getId(), order.getCustomerId(), order.getAmount())));
        outboxRepo.save(outbox);    // save event — same TX as order

        return order;
        // TX commits: both order row + outbox row written atomically
        // TX rolls back: neither written
    }
}

// Step 3: Outbox relay — reads unpublished events, publishes to Kafka
@Component
public class OutboxRelay {

    @Autowired private OutboxEventRepository outboxRepo;
    @Autowired private KafkaTemplate<String, String> kafkaTemplate;

    @Scheduled(fixedDelay = 1000)   // poll every 1s
    @Transactional
    public void relay() {
        outboxRepo.findTop100ByPublishedFalseOrderByCreatedAtAsc()
            .forEach(event -> {
                kafkaTemplate.send(
                    event.getEventType(),
                    event.getAggregateId(),
                    event.getPayload()
                ).whenComplete((result, ex) -> {
                    if (ex == null) {
                        event.setPublished(true);  // mark as published in DB
                        outboxRepo.save(event);    // atomic update
                    }
                    // On failure: not marked published, will retry next poll
                });
            });
    }
}

Outbox with Debezium (CDC-based Relay)

Instead of a polling relay, use Debezium (Change Data Capture) to stream changes from the outbox table's DB transaction log (binlog/WAL) directly to Kafka. Lower latency (~ms), no polling overhead, and no risk of double-publish from concurrent relay instances.

⚠ GotchaThe polling relay can publish the same event twice if it crashes after publishing but before marking as published in the DB. This is at-least-once delivery at the relay level — your Kafka consumers must be idempotent. This is unavoidable without distributed transactions; it's the correct trade-off: occasional duplicates vs guaranteed at-least-once delivery.
✅ Best PracticeUse the outbox pattern for any business event that must be published transactionally with a DB write. It's the only pattern that gives you true atomicity without distributed transactions (2PC). Index the outbox table on (published, created_at) for efficient relay queries. Archive or delete published entries on a schedule to prevent unbounded growth.
🎯 Google L6 AngleThe transactional outbox + Debezium CDC is the production gold standard for reliable event publishing at scale. Debezium eliminates polling latency (sub-second delivery), handles relay failover automatically, and provides exactly-once delivery semantics at the CDC layer when combined with Kafka's idempotent producer. The trade-off: Debezium adds operational complexity (Kafka Connect cluster, connector monitoring). For services publishing <10,000 events/day, a simple polling relay is perfectly adequate and far simpler to operate.

Kafka Streams Overview

Kafka Streams is a client library for stream processing embedded directly in your Java application — no separate cluster required (unlike Flink or Spark). It reads from input topics, processes data using a defined topology, and writes results to output topics. State is managed locally using RocksDB-backed state stores.

Core abstractions:

  • KStream: unbounded stream of key-value pairs; each record is independent
  • KTable: changelog stream materialized as a table; records with same key replace previous (latest-wins)
  • GlobalKTable: full copy of a KTable on every application instance (for joins with streams)
  • StateStore: RocksDB-backed persistent state for aggregations (changelog topic for fault tolerance)
  • Windowing: tumbling (fixed non-overlapping), hopping (fixed overlapping), sliding, session windows

@EnableKafkaStreams
public class StreamsConfig { }

// Stream topology
@Bean
public KStream<String, OrderEvent> orderProcessingStream(StreamsBuilder builder) {

    // Source: read from "orders" topic
    KStream<String, OrderEvent> orders = builder.stream("orders",
        Consumed.with(Serdes.String(), new JsonSerde<>(OrderEvent.class)));

    // Filter confirmed orders
    KStream<String, OrderEvent> confirmed = orders
        .filter((key, order) -> order.getStatus() == OrderStatus.CONFIRMED);

    // Re-key by customerId for aggregation
    KGroupedStream<String, OrderEvent> byCustomer = confirmed
        .groupBy((key, order) -> order.getCustomerId());

    // Count per customer — windowed aggregation (last 1 hour)
    KTable<Windowed<String>, Long> orderCount = byCustomer
        .windowedBy(TimeWindows.ofSizeWithNoGrace(Duration.ofHours(1)))
        .count(Materialized.as("order-count-store"));

    // Route high-value orders to separate topic
    confirmed.filter((key, order) -> order.getAmount() > 1000)
             .to("high-value-orders",
                 Produced.with(Serdes.String(), new JsonSerde<>(OrderEvent.class)));

    return orders;
}

// Interactive Query — query state store from REST endpoint
@RestController
public class AnalyticsController {
    @Autowired KafkaStreamsInteractiveQueryService queryService;

    @GetMapping("/order-count/{customerId}")
    public long getOrderCount(@PathVariable String customerId) {
        ReadOnlyKeyValueStore<String, Long> store =
            queryService.retrieveQueryableStore("order-count-store",

How do you implement an idempotent Kafka consumer in Spring? What are the storage options for deduplication?
IDEMPOTENT_CONSUMER
▼

Why Idempotency is Required

At-least-once delivery means any message can be delivered more than once — on consumer restart, rebalance, or retry after processing failure. Without idempotency, a payment processed twice charges the customer double. An order created twice produces duplicate shipments. Idempotency is not optional for business-critical consumers.

Strategy 1: Inherently Idempotent Operations (Best)

Design the operation so that repeating it has no additional effect. No deduplication state needed.

// Idempotent by nature: UPSERT on primary key
@KafkaListener(topics = "user-updates")
@Transactional
public void processUserUpdate(UserUpdatedEvent event) {
    // INSERT ... ON CONFLICT (user_id) DO UPDATE — safe to call N times
    userRepo.upsert(event.getUserId(), event.getName(), event.getEmail());
}

// Idempotent status transition: guard transition logic
@KafkaListener(topics = "payments")
@Transactional
public void processPayment(PaymentEvent event) {
    Payment payment = paymentRepo.findById(event.getPaymentId()).orElseThrow();
    if (payment.getStatus() == PaymentStatus.PENDING) {  // only process if not yet processed
        payment.setStatus(PaymentStatus.COMPLETED);
        paymentRepo.save(payment);
        notificationService.sendConfirmation(payment);
    }
    // If already COMPLETED, silently ignore duplicate — correct behavior
}

Strategy 2: DB-Based Deduplication (Safest)

@Entity
@Table(name = "processed_messages",
       indexes = @Index(columnList = "message_id", unique = true))
public class ProcessedMessage {
    @Id String messageId;    // topic-partition-offset
    Instant processedAt;
}

@KafkaListener(topics = "payments")
@Transactional   // deduplication check + business logic + mark-processed: atomic
public void processPayment(PaymentEvent event,
                           ConsumerRecord<String, PaymentEvent> record) {

    String messageId = record.topic() + "-" + record.partition() + "-" + record.offset();

    if (processedMessageRepo.existsById(messageId)) {
        log.debug("Duplicate message {}, skipping", messageId);
        return;    // idempotent skip
    }

    paymentService.process(event);    // business logic

    // Mark as processed in same TX as business logic — atomic
    processedMessageRepo.save(new ProcessedMessage(messageId, Instant.now()));
}

// Cleanup old deduplication records (TTL via scheduled job)
@Scheduled(cron = "0 0 2 * * *")   // daily at 2am
@Transactional
public void cleanupOldRecords() {
    processedMessageRepo.deleteByProcessedAtBefore(Instant.now().minus(Duration.ofDays(7)));
}

Strategy 3: Redis-Based Deduplication (High Throughput)

@KafkaListener(topics = "payments")
public void processPayment(PaymentEvent event, ConsumerRecord<?,?> record) {
    String messageId = record.topic() + ":" + record.partition() + ":" + record.offset();

    // SETNX: set if not exists. Returns true if key was newly set (first time).
    Boolean isNew = redisTemplate.opsForValue()
        .setIfAbsent(messageId, "1", Duration.ofDays(1));

    if (Boolean.FALSE.equals(isNew)) {
        log.debug("Duplicate detected in Redis, skipping {}", messageId);
        return;
    }

    paymentService.process(event);
    // Note: Redis write and DB write are NOT atomic — race condition possible
    // If service crashes after Redis SETNX but before DB write: message skipped forever
    // Use only when Redis TTL-based deduplication risk is acceptable
}

Strategy 4: Bloom Filter Pre-Filter

A Bloom filter is a probabilistic data structure: no false negatives (never reports "not seen" for something it has seen) but small false positive rate (occasionally reports "seen" for something new — triggering a DB check). Use as a fast pre-filter in front of DB deduplication to reduce DB load by 99%+ for a very hot consumer.

Deduplication Strategy Comparison

StrategyAtomicityThroughputComplexityBest For
Inherently idempotentN/AHighestLowestAll cases where applicable

Redis deduplicationNone (two writes)HighMediumHigh-throughput with bounded risk

⚠ GotchaUsing Kafka offset as the deduplication key (topic+partition+offset) is robust but has a subtle issue: offsets are only unique within a partition. Two messages with the same offset on different partitions are different messages — always include the partition in the key. Also: if Kafka topic is deleted and recreated, offsets restart from 0 — old deduplication records could false-positive match new messages. Scope deduplication IDs to include a topic creation timestamp or use a business-level correlation ID instead.
✅ Best PracticeDesign your consumer to be inherently idempotent wherever possible — it's the simplest and most performant approach. DB-based deduplication is the safety net for cases where inherent idempotency is impossible. Always store the deduplication record in the same transaction as the business logic — if they're in separate transactions, you have a window where the business effect happened but the deduplication record wasn't saved, leading to double-processing on crash.
🎯 Google L6 AngleAt L6, the design question is: what is your idempotency key strategy? Kafka offsets work but are infrastructure-level identifiers. Business-level correlation IDs (paymentId, orderId) are more meaningful and survive Kafka topic recreation, offset resets, and consumer migrations. Require producers to include a correlation ID in every event payload — this also enables end-to-end tracing across the event pipeline. The extra field costs almost nothing and unlocks deduplication, tracing, and debugging capability at the same time.
