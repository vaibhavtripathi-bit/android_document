How do you tune HikariCP connection pool? What is the formula for pool size?
HIKARI
▼

HikariCP Overview

HikariCP is the default JDBC connection pool in Spring Boot. It is the fastest JVM connection pool — its benchmark lead comes from lock-free design (ConcurrentBag), minimal object allocation, and bytecode-optimized proxy generation. Understanding its configuration is critical because the pool sits on the critical path of every database-bound request.

The Pool Size Formula

The HikariCP authors publish a formula: pool_size = Tn × (Cm - 1) + 1 where Tn is thread count and Cm is the number of concurrent blocking resources per thread. A simplified practical formula: pool_size = (number_of_cores × 2) + effective_spindle_count. For most cloud database backends: start at 10-20, measure, then tune.

Key Configuration

spring:
  datasource:
    hikari:
      pool-name: OrderServicePool
      maximum-pool-size: 20      # max DB connections this pool will open
      minimum-idle: 5            # min idle connections kept alive
      connection-timeout: 30000  # max ms to wait for a connection (30s) — throw if exceeded
      idle-timeout: 600000       # close idle connections after 10 minutes
      max-lifetime: 1800000      # max connection lifetime: 30min (MUST be < DB server timeout)
      keepalive-time: 60000      # HikariCP 5+: ping idle connections every 60s
      leak-detection-threshold: 2000  # log warning if connection held >2s (dev/staging only)
      # Validation query (fallback if JDBC4 not available)
      # connection-test-query: "SELECT 1"

Critical Parameter: max-lifetime

MySQL wait_timeout defaults to 8 hours; PostgreSQL idle_in_transaction_session_timeout varies. If HikariCP holds a connection longer than the DB server's timeout, the server silently closes the connection. The next request receives a "connection is closed" error. Set max-lifetime to at least 30 seconds less than the DB server's connection timeout.

Monitoring HikariCP with Micrometer

# Auto-registered Micrometer metrics (once micrometer-registry-* is on classpath):
# hikaricp.connections.active      — currently in-use connections
# hikaricp.connections.idle        — idle connections in pool
# hikaricp.connections.pending     — threads waiting for a connection ← KEY METRIC
# hikaricp.connections.timeout.total — connections timed out waiting
# hikaricp.connections.creation    — time to create new connections
# hikaricp.connections.acquire     — time spent waiting to acquire a connection

# Alert: hikaricp.connections.pending > 0 for >5 seconds = pool undersized or DB slow
⚠ GotchaBigger pool is NOT always better. Each PostgreSQL connection consumes approximately 5-10 MB of server-side memory and a background process. 100 app instances × 20 connections = 2,000 DB connections — this can OOM the database server. Use a connection pooler (PgBouncer for Postgres, ProxySQL for MySQL) at the database tier for high-instance-count deployments.
✅ Best PracticeIn Kubernetes, if you scale to 50+ pods, switch to PgBouncer in transaction-pooling mode between your pods and Postgres. This multiplexes thousands of application connections onto a much smaller set of real server connections.
🎯 Google L6 AngleConnection pool sizing is a system design decision, not just a config knob. It touches: DB server capacity, application concurrency model, request timeout SLAs, and horizontal scaling strategy. L6 engineers reason about the pool as part of the end-to-end request flow and design for the failure mode (pool exhaustion) explicitly — circuit breakers, bulkheads, and timeouts all interact with pool behavior.

STARTUP_PERF

Why Startup Time Matters

In Kubernetes, slow startup means slow pod scaling — a traffic spike that requires 10 new pods × 8-second startup = 80 seconds before additional capacity is available. For serverless (AWS Lambda, Google Cloud Run), startup time is billed and adds to user-perceived latency on cold starts.

Measurement First

# Or start the app with:

     -jar myapp.jar \
     --spring.jmx.enabled=false
# Look for: "Started ApplicationName in X.XXX seconds"

# Detailed per-step breakdown (Spring Boot 3+):
management.endpoint.startup.enabled: true
spring.lifecycle.timeout-per-shutdown-phase: 30s

    DataSourceAutoConfiguration.class,    // if using NoSQL only

AppCDS (Application Class Data Sharing)

# 1. Generate class list on first run
java -XX:DumpLoadedClassList=classes.lst -jar myapp.jar --spring.main.web-environment=false

# 2. Create shared archive
java -Xshare:dump -XX:SharedClassListFile=classes.lst \
     -XX:SharedArchiveFile=myapp.jsa -jar myapp.jar

# 3. Run with shared archive (includes all app classes)
java -Xshare:on -XX:SharedArchiveFile=myapp.jsa -jar myapp.jar
# Typical result: 30-50% faster startup, lower memory for identical JVM instances
🎯 Google L6 AngleFor Kubernetes-based services, a target startup time under 3 seconds enables aggressive horizontal pod autoscaling. Combine Spring AOT + AppCDS for JVM deployments to get to the 1-2 second range without the complexity of GraalVM native. Reserve native compilation for truly serverless or memory-constrained workloads where the build-time cost is justified.

Explain Micrometer. How do you define custom metrics and configure Prometheus/Grafana?
MICROMETER
▼

What Micrometer Is

Micrometer is a vendor-neutral metrics instrumentation facade for JVM applications — the SLF4J of metrics. It provides a common API (Counter, Timer, Gauge, DistributionSummary) and pluggable backends: Prometheus, Datadog, CloudWatch, InfluxDB, New Relic, Dynatrace, Wavefront, and others. Spring Boot auto-configures Micrometer and registers a rich set of metrics out of the box.

Auto-Configured Metrics (Built-in)

  • JVM: jvm.memory.used, jvm.gc.pause, jvm.threads.live, jvm.classes.loaded
  • HTTP server: http.server.requests — count, total time, max, percentiles per URI+method+status
  • HikariCP: hikaricp.connections.active, hikaricp.connections.pending
  • Spring Cache: cache.gets (tag: result=hit/miss), cache.puts, cache.evictions
  • Kafka: kafka.consumer.records-lag, kafka.producer.record-send-total
  • Executor: executor.pool.size, executor.queue.remaining

Custom Metrics

@Service
public class OrderService {

    private final Counter orderCreatedCounter;
    private final Counter orderFailedCounter;
    private final Timer   orderProcessingTimer;
    private final Gauge   pendingOrdersGauge;

    public OrderService(MeterRegistry registry, PendingOrderTracker tracker) {
        orderCreatedCounter = Counter.builder("orders.created")
            .description("Total orders successfully created")
            .tag("region", "us-east-1")
            .register(registry);

        orderFailedCounter = Counter.builder("orders.failed")
            .description("Total orders that failed processing")
            .register(registry);

        orderProcessingTimer = Timer.builder("orders.processing.duration")
            .description("Time to process an order end-to-end")
            .publishPercentiles(0.5, 0.95, 0.99)       // client-side percentiles
            .publishPercentileHistogram(true)            // server-side histogram for Prometheus
            .serviceLevelObjectives(                     // SLO buckets
                Duration.ofMillis(100),
                Duration.ofMillis(500),
                Duration.ofSeconds(2))
            .register(registry);

        pendingOrdersGauge = Gauge.builder("orders.pending.count",
                tracker, PendingOrderTracker::getPendingCount)
            .description("Current number of orders awaiting processing")
            .register(registry);
    }

    public Order createOrder(OrderRequest req) {
        return orderProcessingTimer.record(() -> {
            try {
                Order order = processOrderInternal(req);
                orderCreatedCounter.increment();
                return order;
            } catch (Exception e) {
                orderFailedCounter.increment();
                throw e;
            }
        });
    }
}

Spring Boot 3 Tracing Stack

Spring Boot 3 replaced Spring Cloud Sleuth with Micrometer Tracing — an abstraction layer over Brave (Zipkin) and OpenTelemetry. The core is the Observation API, which produces both metrics and traces from a single instrumentation point. This is a fundamental architectural improvement over Sleuth.

Dependencies

<!-- Core tracing abstraction -->
<dependency>
    <groupId>io.micrometer</groupId>
    <artifactId>micrometer-tracing-bridge-brave</artifactId>
</dependency>
<!-- Zipkin reporter -->
<dependency>
    <groupId>io.zipkin.reporter2</groupId>
    <artifactId>zipkin-reporter-brave</artifactId>
</dependency>
<!-- Or use OpenTelemetry bridge: -->
<!-- micrometer-tracing-bridge-otel + opentelemetry-exporter-otlp -->

Configuration

management:
  tracing:
    sampling:
      probability: 0.1        # sample 10% of requests in production
  zipkin:
    tracing:
      endpoint: http://zipkin:9411/api/v2/spans

logging:
  pattern:
    console: "%d{HH:mm:ss} %-5level [%X{traceId},%X{spanId}] %logger{36} - %msg%n"
    #                              ^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^
    # traceId and spanId are automatically injected into MDC

What Gets Traced Automatically

  • HTTP requests: inbound via ObservationFilter — creates root span for each request
  • RestTemplate / WebClient: outbound HTTP calls — creates child spans with HTTP method, URI, status
  • JDBC: datasource-proxy or P6Spy — span per SQL query with query text tag
  • Kafka: producer/consumer operations — propagates context in Kafka headers
  • Redis: Lettuce/Jedis operations
  • @Async: context propagated to executor thread
  • @Scheduled: each execution is a root span

Trace Propagation Between Services

When Service A calls Service B via HTTP, the tracing bridge injects B3 propagation headers (or W3C traceparent) into the outgoing request. Service B's Spring Boot filter reads those headers and continues the same trace rather than starting a new one:

// Headers injected by Brave B3 propagation:
X-B3-TraceId:  abc123def456   // same across all services in the call chain
X-B3-SpanId:   789xyz          // unique to this service's span
X-B3-Sampled:  1               // propagate sampling decision — only sample once

// W3C traceparent (modern standard):
traceparent: 00-abc123def456-789xyz-01

Manual Instrumentation

@Service
public class OrderProcessingService {

    private final Tracer tracer;

    public OrderProcessingService(Tracer tracer) {
        this.tracer = tracer;
    }

    public void processOrder(Order order) {
        Span span = tracer.nextSpan()
            .name("order.process")
            .tag("order.id", String.valueOf(order.getId()))
            .tag("order.region", order.getRegion())
            .start();

        try (Tracer.SpanInScope ws = tracer.withSpan(span)) {
            // All child operations (DB calls, HTTP calls) auto-parented to this span
            validateOrder(order);
            deductInventory(order);
            chargePayment(order);
            sendConfirmationEmail(order);
        } catch (Exception e) {
            span.error(e); // marks span as error, attaches exception
            throw e;
        } finally {
            span.end(); // always end the span
        }
    }
}

// Using the higher-level Observation API (preferred in Spring Boot 3)
@Autowired ObservationRegistry observationRegistry;

Observation.createNotStarted("order.process", observationRegistry)
    .lowCardinalityKeyValue("region", order.getRegion())
    .highCardinalityKeyValue("orderId", order.getId().toString())
    .observe(() -> {
        validateOrder(order);
        chargePayment(order);
    });
// This produces BOTH a metric (orders.process.duration) AND a trace span automatically

Log-Trace Correlation

With the MDC pattern above, every log line includes the traceId. In your log aggregator (Kibana, Loki), filter by traceId=abc123 to see all log lines from a single distributed request — then click the traceId to jump to the Zipkin trace timeline. This correlation between logs and traces is the foundation of efficient incident investigation.

⚠ GotchaSampling at 100% in production creates significant overhead (network, memory, Zipkin ingest cost) for high-traffic services. Use 1-10% probabilistic sampling with "always sample on error" logic. Zipkin and Jaeger support tail-based sampling for post-hoc error filtering.
🎯 Google L6 AngleDistributed tracing is most valuable during incidents: "why is the p99 latency up?" Without traces, you grep logs across 20 service instances hoping to find the slow call. With traces, you sort by duration, click the slowest trace, and see in 10 seconds that the user-service call at hop 3 is taking 800ms due to a missing database index. The value compounds with service count — mandatory past 5 services.

What JVM flags are important for Spring Boot in production? Explain GC selection and heap sizing.
JVM_TUNING
▼

Heap Sizing in Containers

The #1 rule: never set -Xmx to the container memory limit. The JVM needs memory beyond the heap: metaspace, code cache, native libraries, thread stacks, direct buffers. Set heap to 60-75% of container memory and leave the rest for non-heap.

# Container-aware (preferred over absolute -Xmx):
-XX:MaxRAMPercentage=75.0    # 75% of container memory limit for heap
-XX:InitialRAMPercentage=50  # initial heap = 50% (avoids slow heap growth)

# OR explicit (for predictable sizing):
-Xms512m -Xmx1g              # set Xms = Xmx to avoid resize pauses

# Always set Xms = Xmx in production to eliminate heap resize overhead
# and make memory usage predictable for container OOM killer

GC Selection Guide

GCFlagPauseThroughputBest For
G1GC-XX:+UseG1GC (default Java 9+)10-200msHighMost web services, heap 4GB+, balanced
ZGC-XX:+UseZGC<1msSlightly lowerAPIs with p99 SLA <10ms, Java 15+ stable
Shenandoah-XX:+UseShenandoahGC<5msMediumSimilar to ZGC, OpenJDK builds
ParallelGC-XX:+UseParallelGCHighHighestBatch jobs, throughput over latency

Production JVM Flag Set

JAVA_OPTS="\
  -XX:+UseZGC \
  -XX:MaxRAMPercentage=75.0 \
  -XX:InitialRAMPercentage=50.0 \
  -Xss512k \
  -XX:+OptimizeStringConcat \
  -XX:+UseStringDeduplication \
  -Dfile.encoding=UTF-8 \
  -Duser.timezone=UTC \
  -XX:+HeapDumpOnOutOfMemoryError \
  -XX:HeapDumpPath=/tmp/heap-$(date +%Y%m%d-%H%M%S).hprof \
  -XX:+ExitOnOutOfMemoryError \
  -XX:ErrorFile=/tmp/hs_err.log \
  -Djava.security.egd=file:/dev/./urandom"

# -Xss512k: reduce thread stack from 1MB default (critical with many threads)
# -XX:+ExitOnOutOfMemoryError: exit cleanly on OOM — kubernetes restarts the pod
# -Djava.security.egd: faster random number generation (important for TLS startup)
# -XX:+UseStringDeduplication: G1GC only — deduplicate identical String objects in heap

Virtual Threads (Java 21 + Spring Boot 3.2)

spring.threads.virtual.enabled: true
# Result:
# - Tomcat request threads → virtual threads (millions possible)
# - @Async executor → virtual threads
# - @Scheduled tasks → virtual threads
# Benefits: -Xss impact reduced (virtual thread stacks heap-allocated)
#           blocking JDBC calls release carrier threads for other work
#           no need for -Xss tuning for blocking I/O workloads
⚠ Gotcha-XX:+ExitOnOutOfMemoryError is critical in production. An OOM'd JVM process has unpredictable state — partially executed transactions, corrupted caches, failed health checks. A clean restart via Kubernetes is far safer than limping along in an undefined state.
🎯 Google L6 AngleGC selection is a data-driven decision. Before switching from G1 to ZGC, measure your actual GC pause times in production via JFR or GC logs. ZGC's lower throughput is a real trade-off. If p99 latency is not bounded by GC pauses, G1 is the right choice. Profile first, tune second.

Async Profiler — Flame Graph Workflow

#         -d 30 seconds, -f output file

# 2. Or run with allocation profiling (find what's allocating most memory)

Reading Flame Graphs

  • Wide flat plateau: this method is consuming significant CPU time — investigate
  • Tall narrow tower: deep call stack, but if narrow at top = not the bottleneck
  • Broad tower: deep call stack AND consuming CPU — the full chain is expensive
  • Unexpected methods in hot path: Jackson serialization, String formatting, logger calls can appear surprisingly wide

How does caching impact performance? Walk through a caching strategy for a read-heavy service.
CACHING_PERF
▼

The Problem: Read-Heavy User Profile Service

Scenario: user profile service receiving 5,000 req/sec. Database can sustain 500 queries/sec before latency degrades. Without caching, the service is 10x over DB capacity. The caching strategy is a cascade of layers, each serving a different access pattern.

Layer 1: HTTP Caching (Zero Cost)

@GetMapping("/api/users/{id}/profile")
@ResponseBody
public ResponseEntity<UserProfileDTO> getProfile(@PathVariable Long id) {
    UserProfileDTO profile = userService.getProfile(id);
    return ResponseEntity.ok()
        .cacheControl(CacheControl.maxAge(60, TimeUnit.SECONDS)
            .cachePublic()
            .mustRevalidate())
        .eTag(String.valueOf(profile.getVersion()))  // ETag for conditional GET
        .body(profile);
}
// CDN/browser caches for 60 seconds → zero server hits for cached users in that window

Layer 2: In-Process Caffeine Cache (Sub-Millisecond)

# application.yml
spring:
  cache:
    type: caffeine
    caffeine:
      spec: maximumSize=10000,expireAfterWrite=5m,recordStats

@Service
public class UserService {
    @Cacheable(value = "user-profiles", key = "#userId",
               condition = "#userId > 0",
               unless = "#result == null")
    public UserProfileDTO getProfile(Long userId) {
        return userRepository.findProfileById(userId); // DB hit only on cache miss
    }

    @CacheEvict(value = "user-profiles", key = "#user.id")
    public void updateProfile(User user) { ... } // evict on write

    @CachePut(value = "user-profiles", key = "#result.id")
    public UserProfileDTO updateAndReturn(UpdateRequest req) { ... } // update cache with new value
}

// Monitor cache hit rate
@Bean
MeterBinder caffeineMetrics(CacheManager cacheManager) {
    return (registry) -> {
        CaffeineCache cache = (CaffeineCache) cacheManager.getCache("user-profiles");
        CacheMetricsRegistrar.bindCacheToRegistry(cache, registry, Tags.empty());
    };
}

Layer 3: Distributed Redis Cache (Cross-Node)

# application.yml — switch to Redis for distributed deployments
spring:
  cache:
    type: redis
  data:
    redis:
      host: redis-primary
      port: 6379
  cache:

# For large deployments: use two-level cache (Caffeine + Redis)
# Caffeine L1 (node-local, sub-ms) backed by Redis L2 (shared, ~0.5ms)
# Library: layering with @Cacheable + custom CacheManager

Cache-Aside Pattern (Manual Control)

@Service
public class UserService {
    @Autowired StringRedisTemplate redis;
    @Autowired ObjectMapper mapper;
    @Autowired UserRepository repo;

    public UserProfileDTO getProfile(Long userId) {
        String cacheKey = "user:profile:" + userId;
        String cached = redis.opsForValue().get(cacheKey);
        if (cached != null) {
            return mapper.readValue(cached, UserProfileDTO.class);
        }
        UserProfileDTO profile = repo.findProfileById(userId);
        redis.opsForValue().set(cacheKey,
            mapper.writeValueAsString(profile),
            Duration.ofMinutes(5));
        return profile;
    }
}

Cache Strategy Decision Matrix

Data CharacteristicRecommended StrategyTTL
Rarely changes (config, reference data)Caffeine L1, long TTL1-24h
Changes occasionally (user profiles)Redis + write-through eviction5-15min
Changes frequently (inventory counts)Very short TTL or no cache5-30s or none
Per-user session dataRedis, user-keyedSession duration
Computed aggregates (dashboard stats)Redis, background refresh1-5min
⚠ GotchaCache stampede: when a popular cache entry expires, thousands of requests simultaneously miss the cache and hammer the DB. Solutions: (1) probabilistic early expiration — refresh before TTL expires, (2) lock-based refresh — one thread fetches, others wait, (3) background refresh with stale-while-revalidate pattern.
🎯 Google L6 AngleCaching strategy is a consistency trade-off, not a free optimization. Increasing TTL from 1 minute to 10 minutes reduces DB load by 10x but means users see stale data for up to 10 minutes after writes. The correct TTL is a product decision (how stale is acceptable?) as much as a technical one. L6 engineers make this explicit in design docs and align with product stakeholders.

How does making endpoints async improve throughput? What is thread starvation?
ASYNC_PERF
▼

The Thread-Per-Request Model and Its Limit

Traditional Tomcat: one OS thread handles one request, blocked from arrival to response. With 200 Tomcat threads and 50ms average DB latency: theoretical max = 200 ÷ 0.050 = 4,000 req/sec. If DB latency spikes to 500ms: 200 ÷ 0.5 = 400 req/sec — a 10x throughput collapse with no CPU increase. CPU is idle (waiting on I/O), but threads are all occupied. This is thread starvation.

Thread Starvation Indicators

  • High request queue depth (Tomcat executor.queue.remaining approaching 0)
  • CPU utilization low but latency exploding
  • Thread dump shows almost all threads in WAITING or TIMED_WAITING state on DB/socket calls
  • hikaricp.connections.pending > 0 — threads waiting for DB connections that are all occupied by other blocked threads (cascading failure)

Solution 1: @Async for Background Processing

@Service
public class NotificationService {

    @Async("notificationExecutor") // named thread pool
    public CompletableFuture<Void> sendEmailAsync(String userId, String message) {
        // Runs on notificationExecutor thread pool — request thread freed immediately
        emailGateway.send(userId, message);
        return CompletableFuture.completedFuture(null);
    }
}

@Configuration
@EnableAsync
public class AsyncConfig {
    @Bean("notificationExecutor")
    public Executor notificationExecutor() {
        ThreadPoolTaskExecutor exec = new ThreadPoolTaskExecutor();
        exec.setCorePoolSize(10);
        exec.setMaxPoolSize(50);
        exec.setQueueCapacity(500);
        exec.setThreadNamePrefix("notification-");
        exec.initialize();
        return exec;
    }
}

Solution 2: Virtual Threads (Java 21 + Spring Boot 3.2)

# application.yml
spring:
  threads:
    virtual:
      enabled: true

# Effect: Tomcat uses virtual threads for request handling.
# When a virtual thread blocks on JDBC, network, or file I/O:
# → The virtual thread is PARKED (no OS thread consumed)
# → The carrier thread is FREE to run other virtual threads
# → You can have 100,000+ concurrent "blocking" requests
# → No special async code required — plain blocking code gets non-blocking behavior

# Example: this code now handles 10,000 concurrent requests on ~10 carrier threads:
@GetMapping("/users/{id}")
public UserDTO getUser(@PathVariable Long id) {
    return userService.findById(id); // JDBC call — blocks virtual thread, not carrier
}

Solution 3: WebFlux Reactive (Non-Blocking I/O)

@RestController
public class UserController {
    @GetMapping("/users/{id}")
    public Mono<UserDTO> getUser(@PathVariable Long id) {
        return userService.findById(id) // returns Mono, non-blocking
            .map(user -> new UserDTO(user))
            .doOnError(e -> log.error("Failed to get user {}", id, e));
    }
}

JDBC supportFull (blocking JDBC works)Requires R2DBC (reactive JDBC)

🎯 Google L6 AngleVirtual threads in Java 21 are the most significant change to Java backend architecture since non-blocking I/O. They obsolete most justifications for reactive programming in new services — you get the throughput of WebFlux with the simplicity of blocking code. An L6 engineer should understand this inflection point: "reactive programming was the right answer from 2016-2023; virtual threads are the right answer for new Java 21 services starting now."

What performance characteristics do GraalVM native images have vs JVM? When is native the wrong choice?
NATIVE_PERF

Native Image Advantages

  • Startup time: <100ms (vs 2-10s for JVM). Critical for Lambda cold starts and pod scaling.
  • Memory footprint: ~80-200MB RSS (vs 400-800MB for JVM). 3-4x better memory density on Kubernetes.
  • Peak memory predictability: no JIT compiler code cache, no warmup allocations.
  • Container image size: no JVM needed in image — 30-80MB binary vs 300MB+ JVM-based image.
  • Security surface: no JVM = smaller attack surface, no remote debugging port exposure.

Native Image Disadvantages

DisadvantageImpactMitigation
Build time 5-15 minutesSlow CI, developer iterationJVM build for dev/test; native for staging/prod
Lower peak throughput (10-30%)No JIT profile-guided optimizationAcceptable for most services; measure first
Reflection requires build-time configThird-party libs may breakSpring AOT handles Spring-managed beans; others need reflect-config.json
No remote debuggingHard to debug production issuesStructured logging, JFR (limited), profiling via perf
No runtime class loadingPlugins, dynamic proxies limitedRedesign dynamic features; use compile-time config

Spring AOT — Bridge Between JVM and Native

# Spring AOT (Ahead of Time) processes the application context at build time:
# 1. Computes bean definitions statically (no runtime reflection needed)
# 2. Generates proxy classes at build time (no dynamic proxies)

🎯 Google L6 Angle"Native is not always better — know your workload." An L6 engineer evaluates native compilation on three dimensions: startup time (does the deployment model benefit?), throughput (is the 10-30% JIT advantage material?), and operational complexity (can your team handle the build pipeline and debugging constraints?). The answer is rarely "always native" or "never native" — it is workload-specific and should be backed by measurement.

The Three Pillars

Observability rests on three complementary signals: Metrics (numerical aggregates — what is happening right now at scale), Logs (discrete events — what happened and in what order), and Traces (causal chains — how a request flowed across services). No single pillar suffices alone for production incident investigation.

Metrics Strategy

# RED method for services (Rate, Error rate, Duration):
# - orders.created (rate = orders/sec)
# - http.server.requests{status=5xx} (error rate)
# - http.server.requests{percentile=0.99} (duration p99)

# USE method for resources (Utilization, Saturation, Errors):
# - jvm.memory.used / jvm.memory.max (utilization)
# - hikaricp.connections.pending (saturation)
# - hikaricp.connections.timeout.total (errors)

# Business metrics (most actionable):
# - orders.created rate by region
# - payment.failures rate by error code
# - user.registrations by signup source

# application.yml — expose everything needed:
management:
  endpoints:
    web:
      exposure:
        include: health,info,metrics,prometheus,loggers,threaddump
  metrics:
    distribution:
      percentiles:
        http.server.requests: 0.5,0.95,0.99
      percentiles-histogram:
        http.server.requests: true
    tags:
      application: ${spring.application.name}
      environment: ${APP_ENV:production}
      region: ${APP_REGION:us-east-1}

Logging Rules

  • WARN and ERROR only in production by default — INFO is configurable at runtime
  • Runtime log level change: POST /actuator/loggers/com.example -d '{"configuredLevel":"DEBUG"}'
  • Never log PII (names, emails, SSN, payment data) — legal and security requirement
  • Never log secrets — tokens, passwords, API keys
  • Every log line must include enough context to be actionable without reading source code

Incident Response Workflow

  1. Alert fires: Prometheus alert — p99 latency > 500ms for 5 minutes
  2. Dashboard: Grafana — identify which endpoint is slow, which pod, when it started
  3. Correlation: note the timestamp → search Kibana/Loki logs at that time → filter by service + ERROR level
  4. Trace: find a traceId in the logs → open in Zipkin/Jaeger → see which span in the call chain is slow
  5. Root cause: slow span identifies the service and operation → check its metrics (DB connection pool? external API timeout?)
  6. Fix and verify: deploy fix → watch metrics normalize → confirm in traces → close alert
✅ Best PracticeDefine SLOs (Service Level Objectives) before instrumentation. "99.5% of requests complete in under 500ms" tells you exactly which metrics to alert on and what the success criteria for observability is. Observability without SLOs is monitoring for its own sake.
🎯 Google L6 AngleObservability is infrastructure — non-negotiable, not optional. An L6 engineer budgets time for observability like they budget for features and allocates it early in the project, not as an afterthought. The engineering principle: "you cannot improve what you cannot measure." An unobservable system is a liability that multiplies incident duration, increases MTTR, and erodes on-call quality of life. Design for observability from day one.
