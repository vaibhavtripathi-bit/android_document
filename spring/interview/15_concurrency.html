How does @Async work internally? What is the AOP proxy mechanism behind it?
ASYNC_INTERNALS
▼

Bootstrapping pipeline

@EnableAsync on a @Configuration class imports AsyncConfigurationSelector, which registers AsyncAnnotationBeanPostProcessor — a BeanPostProcessor that fires after every bean is instantiated. This BPP inspects each bean for @Async methods and, when found, replaces the bean reference in the context with a CGLIB (or JDK dynamic) proxy. The proxy intercepts method calls and, instead of executing the body on the calling thread, submits the work to a TaskExecutor and returns immediately.

Default executor — the production footgun

If no custom TaskExecutor bean is defined, Spring falls back to SimpleAsyncTaskExecutor, which creates a brand-new OS thread per invocation. It is NOT a pool and has no upper bound. Under load this causes unbounded thread creation → OOM or OS-level thread exhaustion. Always define a pooled executor in production.

Supported return types

  • void — fire-and-forget; exceptions silently swallowed unless handler configured
  • Future<T> — caller can block with .get()
  • CompletableFuture<T> — preferred; full async composition API
  • ListenableFuture<T> — deprecated since Spring 6
Gotcha — self-invocation Same AOP limitation as @Transactional: calling an @Async method from within the same bean bypasses the proxy entirely. The method runs synchronously on the calling thread. Inject the bean as a dependency or use ApplicationContext.getBean() to get the proxy reference.

Exception handling

For void methods, uncaught exceptions go nowhere by default. For Future/CompletableFuture, the exception is stored and re-thrown on .get(). Configure a custom handler to log/alert on void-method failures:

@Configuration
@EnableAsync
public class AsyncConfig implements AsyncConfigurer {

    @Override
    public Executor getAsyncExecutor() {
        ThreadPoolTaskExecutor executor = new ThreadPoolTaskExecutor();
        executor.setCorePoolSize(10);
        executor.setMaxPoolSize(50);
        executor.setQueueCapacity(500);
        executor.setThreadNamePrefix("Async-");
        executor.initialize();
        return executor;
    }

    @Override
    public AsyncUncaughtExceptionHandler getAsyncUncaughtExceptionHandler() {
        return (ex, method, params) ->
            log.error("Async error in {}: {}", method.getName(), ex.getMessage(), ex);
    }
}

Multiple executors

Name each @Bean and reference it: @Async("emailExecutor"), @Async("reportExecutor"). This isolates workloads — a slow report job cannot starve the email pool. The bean named taskExecutor becomes the default.

L6 Insight At Google scale you own many async pipelines simultaneously. Be prepared to explain the full proxy lifecycle: BeanPostProcessor order, how Spring decides CGLIB vs JDK proxy, and why @Async on a final method silently does nothing (CGLIB cannot subclass it). Also know that @Async strips the transaction context — combining @Async and @Transactional on the same method means the transaction starts on the async thread, not the calling thread.

How do you properly configure ThreadPoolTaskExecutor? What happens when the queue and max pool are exhausted?
THREAD_POOL
▼

Key parameters

ParameterMeaningTypical value
corePoolSizeThreads always kept alive, even when idle10–20 for I/O-bound
maxPoolSizeMaximum threads created when queue is full50–200
queueCapacityTask queue depth before spawning beyond core200–1000
keepAliveSecondsTime before idle threads above core are terminated60
threadNamePrefixPrefix for thread names — visible in thread dumps"Async-Email-"
allowCoreThreadTimeOutCore threads also time out when idlefalse

Execution order (Java ThreadPoolExecutor behaviour)

  1. If active threads < corePoolSize: spawn new thread immediately
  2. If active threads ≥ core AND queue not full: enqueue the task
  3. If queue full AND active threads < maxPoolSize: spawn new thread
  4. If queue full AND active == max: RejectedExecutionHandler fires
Gotcha — counter-intuitive queue vs max interaction Threads do NOT scale from core to max while the queue has space. The queue fills up completely before a single thread beyond core is created. This means with corePoolSize=5, maxPoolSize=50, queueCapacity=1000: you'll have 5 threads with 1000 tasks queued long before thread 6 is spawned. Set queueCapacity to 0 (SynchronousQueue) if you want immediate scaling to max.

Rejection policies

PolicyBehaviourUse when
AbortPolicy (default)Throws RejectedExecutionExceptionFast fail, caller handles
CallerRunsPolicyCaller thread runs the taskNatural backpressure on HTTP threads
DiscardPolicySilently drops the taskLossy telemetry, metrics aggregation
DiscardOldestPolicyDrops oldest queued task, retries submissionSliding-window style processing
@Bean("ioExecutor")
public ThreadPoolTaskExecutor ioExecutor() {
    ThreadPoolTaskExecutor executor = new ThreadPoolTaskExecutor();
    executor.setCorePoolSize(20);
    executor.setMaxPoolSize(100);
    executor.setQueueCapacity(500);
    executor.setKeepAliveSeconds(60);
    executor.setThreadNamePrefix("IO-");
    executor.setWaitForTasksToCompleteOnShutdown(true);
    executor.setAwaitTerminationSeconds(30);
    // CallerRunsPolicy = backpressure: HTTP thread runs task → slows Tomcat naturally
    executor.setRejectedExecutionHandler(new ThreadPoolExecutor.CallerRunsPolicy());
    executor.initialize();
    return executor;
}

CallerRunsPolicy as backpressure

When pool + queue are both exhausted, CallerRunsPolicy makes the caller thread execute the task synchronously. If the caller is a Tomcat thread handling an HTTP request, that thread is occupied → Tomcat accepts fewer new connections → the system naturally throttles inbound traffic. This is effective backpressure without dropping requests.

Pool sizing rules

  • I/O-bound (DB calls, HTTP, file I/O): core = target concurrent I/O operations. If you expect 50 simultaneous DB queries, set core=50.
  • CPU-bound: core = Runtime.getRuntime().availableProcessors() or N+1.
  • Mixed: profile. Little's Law: throughput = concurrency / latency. If avg latency = 200ms and target throughput = 500 req/s → need 100 threads minimum.
L6 Insight Wrong pool sizing is one of the top causes of production latency spikes. Too small: queue builds → p99 latency spikes while p50 looks fine. Too large: context-switching overhead + DB connection pool exhaustion (200 Tomcat threads, 10 HikariCP connections → 190 threads waiting). Always tune pool sizes in conjunction with HikariCP's maximumPoolSize. Micrometer exposes executor.pool.size, executor.queued, and executor.completed metrics — alert on queue depth, not just pool size.

Explain virtual threads in Java 21 and Spring Boot 3.2. What changes and what doesn't?
VIRTUAL_THREADS
▼

What are virtual threads?

Project Loom (GA in Java 21) introduces virtual threads: JVM-managed lightweight threads that are not 1:1 mapped to OS threads. The JVM multiplexes millions of virtual threads onto a small pool of platform "carrier" threads. When a virtual thread blocks (JDBC, HTTP, sleep, Object.wait()), the JVM automatically unmounts it from its carrier thread — freeing the carrier to run another virtual thread. No OS thread is parked during the wait.

Spring Boot 3.2 integration

A single property enables virtual threads throughout the stack:

spring:
  threads:
    virtual:
      enabled: true

This switches:

  • Tomcat: VirtualThreadExecutor replaces the platform thread pool for request handling
  • @Async: AsyncTaskExecutor uses virtual threads
  • @Scheduled: tasks run on virtual threads
  • Spring Data JPA: JDBC calls happen on virtual threads (carrier freed during blocking I/O)
  • Spring Security: SecurityContextHolder uses InheritableThreadLocal mode automatically

What changes vs. platform threads

AspectPlatform threadsVirtual threads
Count limit~10K OS threads max (typical)Millions in JVM
Blocking I/OOS thread parked, wastedCarrier freed, reused
Memory per thread~1 MB stack~few KB (grows on demand)
Thread pool needed?Yes, mandatoryNo — create per task
Context propagationThreadLocal manualThreadLocal inherited naturally

What does NOT change

  • Code is still imperative and synchronous — no reactive chains, no callbacks
  • JDBC still blocks — virtual threads just don't waste OS threads while blocked
  • CPU-bound code: same performance as platform threads (no carrier unmount during CPU work)
  • Reactive (WebFlux + R2DBC) still wins for: streaming, backpressure, complex async pipelines
Gotcha — pinning A virtual thread is "pinned" to its carrier (cannot unmount) when it enters a synchronized block while performing blocking I/O, or when calling native methods. Pinning defeats the throughput benefit. Fix: replace synchronized with ReentrantLock. Diagnose with: -Djdk.tracePinnedThreads=full JVM flag. Known Spring/Hibernate components have been updated to use locks, but third-party libraries may still pin. 
// Pinning risk — avoid synchronized + blocking I/O
synchronized (lock) {
    result = jdbcTemplate.query(...); // pins carrier thread!
}

// Fix: ReentrantLock
private final ReentrantLock lock = new ReentrantLock();
lock.lock();
try {
    result = jdbcTemplate.query(...); // virtual thread can unmount here
} finally {
    lock.unlock();
}
Tip — throughput vs. complexity tradeoff Virtual threads deliver WebFlux-class throughput for I/O-bound workloads with none of the reactive programming complexity. For most Spring Boot REST + JPA services, enabling virtual threads is a no-brainer upgrade. Only reach for WebFlux/R2DBC when you need streaming semantics, backpressure, or reactive operators.
L6 Insight At L6 you must articulate tradeoffs, not just features. Virtual threads make reactive programming optional for throughput, but not obsolete. Key argument: "Use the simplest model that meets your requirements." For greenfield: imperative + virtual threads. For existing reactive pipelines: keep them. For streaming (SSE, WebSocket): reactive still preferred. Also: virtual threads increase effective concurrency dramatically — your DB connection pool (HikariCP) must be sized accordingly or it becomes the new bottleneck.

How does @Scheduled work? Explain fixed-delay vs fixed-rate vs cron. What are the threading gotchas?
SCHEDULED
▼

Bootstrapping

@EnableScheduling registers ScheduledAnnotationBeanPostProcessor, which detects all @Scheduled methods across beans and registers them with a TaskScheduler. The default scheduler is a single-threaded ThreadPoolTaskScheduler — all scheduled tasks share ONE background thread.

Gotcha — default single thread If Task A runs for 8 seconds and is scheduled every 5 seconds, Task B (scheduled every 1 second) will not fire on time — it waits for the single shared thread. This is the most common @Scheduled production surprise. Always override the default scheduler in any non-trivial application.

Scheduling modes compared

ModeTiming basisOverlap riskUse when
fixedDelay=50005s after previous execution completesNone by defaultTasks that must not overlap (DB cleanup, file processing)
fixedRate=5000Every 5s from start, regardless of durationHigh if task > 5sMetrics collection, heartbeat pings where drift matters
cron="0 */5 * * * *"Calendar-based (cron expression)None by defaultBusiness-time jobs (daily at 2am, weekdays at 9am)

Spring cron syntax (6 fields)

// Format: seconds minutes hours day-of-month month day-of-week
@Scheduled(cron = "0 0 2 * * *")       // daily at 02:00:00
@Scheduled(cron = "0 */15 9-17 * * MON-FRI") // every 15 min, business hours weekdays
@Scheduled(cron = "0 0 8 * * *", zone = "America/Los_Angeles") // timezone-aware

initialDelay

Use initialDelay=30000 to wait before first execution. Critical when the task depends on data loaded during application startup (caches, config refresh). Combine with fixedDelay or fixedRate.

Overriding the default single-threaded scheduler

@Bean
public TaskScheduler taskScheduler() {
    ThreadPoolTaskScheduler scheduler = new ThreadPoolTaskScheduler();
    scheduler.setPoolSize(10);
    scheduler.setThreadNamePrefix("Scheduled-");
    scheduler.setWaitForTasksToCompleteOnShutdown(true);
    scheduler.setAwaitTerminationSeconds(30);
    scheduler.initialize();
    return scheduler;
}

Cluster problem: duplicate execution

In a 10-node deployment, @Scheduled runs on EVERY node simultaneously. For idempotent tasks (metrics push, cache warm) this is fine. For tasks with side effects (send daily email, generate invoice) it's a disaster. Solution: ShedLock for distributed scheduling:

// pom.xml: net.javacrumbs.shedlock:shedlock-spring + shedlock-provider-redis-spring

@Configuration
@EnableSchedulerLock(defaultLockAtMostFor = "PT10M")
public class SchedulerConfig { }

@Scheduled(cron = "0 0 8 * * *")
@SchedulerLock(name = "dailyReport", lockAtMostFor = "PT10M", lockAtLeastFor = "PT5M")
public void generateDailyReport() {
    // Only one node executes this at a time
    // lockAtMostFor: auto-release lock if app crashes during execution
    // lockAtLeastFor: prevent re-acquisition immediately after fast finish
    reportService.generate();
}
Tip — @Scheduled + @Async Adding @Async to a @Scheduled method allows overlapping executions (each invocation starts a new async task regardless of previous completion). Use only when overlap is explicitly desired and tasks are truly independent.
L6 Insight ShedLock with Redis is the standard production answer for distributed scheduling. The lockAtMostFor parameter is safety-critical: it ensures the lock is released even if the node holding it crashes, preventing permanent lock starvation. Set it to the maximum acceptable task duration plus a safety margin. At Google scale, you'd likely replace @Scheduled entirely with a managed job scheduler (Cloud Scheduler, Argo Workflows) for observability, retries, and audit trail.

How do you use CompletableFuture in Spring? Show parallel service calls and proper exception handling.
COMPLETABLE_FUTURE
▼

@Async + CompletableFuture pattern

Return CompletableFuture<T> from @Async methods to get both thread-pool offloading and async composition. Spring wraps the return value automatically.

@Service
public class UserService {

    @Async("ioExecutor")
    public CompletableFuture<UserDTO> findUserAsync(Long id) {
        return CompletableFuture.completedFuture(
            userRepo.findById(id).map(UserDTO::from).orElseThrow()
        );
    }
}

@Service
public class OrderService {

    @Async("ioExecutor")
    public CompletableFuture<List<Order>> findOrdersAsync(Long userId) {
        return CompletableFuture.completedFuture(
            orderRepo.findByUserId(userId)
        );
    }
}

@Service
public class NotificationService {

    @Async("ioExecutor")
    public CompletableFuture<List<Notification>> findNotificationsAsync(Long userId) {
        return CompletableFuture.completedFuture(
            notificationRepo.findByUserId(userId)
        );
    }
}

Parallel fan-out with allOf

@Service
public class DashboardService {

    public DashboardDTO getDashboard(Long userId) throws Exception {
        // Fire all 3 requests simultaneously
        CompletableFuture<UserDTO> userFuture = userService.findUserAsync(userId);
        CompletableFuture<List<Order>> ordersFuture = orderService.findOrdersAsync(userId);
        CompletableFuture<List<Notification>> notifFuture = notificationService.findNotificationsAsync(userId);

        return CompletableFuture
            .allOf(userFuture, ordersFuture, notifFuture)
            .thenApply(v -> DashboardDTO.builder()
                .user(userFuture.join())           // join() safe here — allOf guarantees completion
                .orders(ordersFuture.join())
                .notifications(notifFuture.join())
                .build()
            )
            .exceptionally(ex -> {
                log.error("Dashboard assembly failed for user {}", userId, ex);
                return DashboardDTO.degraded(userId);  // partial data fallback
            })
            .get(5, TimeUnit.SECONDS);             // ALWAYS add timeout
    }
}

Exception handling operators

OperatorBehaviourUse when
exceptionally(fn)Catches exception, transforms to fallback value; propagates if fn throwsProvide default on failure
handle(biFunction)Receives both result and exception (one will be null); must return valueUnified result/error processing
whenComplete(biConsumer)Side-effect only; exception still propagates downstreamLogging, metrics — don't swallow
thenApply(fn)Transform result; exception bypassesMap result type
thenCompose(fn)Chain another CompletableFuture; avoids nested futuresSequential async calls
// Sequential chain — thenCompose avoids CompletableFuture<CompletableFuture<T>>
userService.findUserAsync(userId)
    .thenCompose(user -> orderService.findOrdersAsync(user.getId()))
    .thenApply(orders -> orders.stream().filter(Order::isPending).toList())
    .exceptionally(ex -> Collections.emptyList())
    .thenAccept(pending -> notifyUser(pending));
Gotcha — SecurityContext and ForkJoinPool Chaining operators without @Async (e.g., supplyAsync() without explicit executor) uses the common ForkJoinPool. The SecurityContextHolder uses ThreadLocal — not inherited by ForkJoinPool threads. Authentication becomes null mid-chain. Fix: always supply an explicit executor that has DelegatingSecurityContextExecutor wrapping, or use @Async with a security-aware executor.
Gotcha — never call .get() without a timeout future.get() blocks the calling thread indefinitely if the async task hangs. This is a thread starvation vector. Always use .get(N, TimeUnit.SECONDS) and handle TimeoutException explicitly.
L6 Insight Know the difference between thenApply (synchronous transform, runs on completing thread) and thenApplyAsync (submitted to executor). In high-throughput services, thenApply for cheap transforms is fine; for anything blocking or heavy, use thenApplyAsync(fn, executor) to avoid tying up the async thread pool. Also: CompletableFuture.allOf fails fast on first exception only if you chain handlers — it doesn't short-circuit the other futures; they still run to completion (or failure). For true short-circuit, use anyOf or custom cancellation.

What is thread starvation in Spring Boot? Show a real scenario and the fix.
THREAD_STARVATION
▼

Definition

Thread starvation: all threads in a pool are blocked waiting for a resource (I/O, downstream service, lock, another task) while new work is queued. Symptoms: p99 latency explodes while CPU stays low, thread dump shows threads in WAITING/BLOCKED state, actuator/metrics shows executor queue depth climbing.

Scenario 1 — Tomcat pool exhaustion (slow downstream)

200 Tomcat threads. Each request calls a microservice averaging 2s. At 100 req/s: 100 × 2 = 200 threads busy. Queue builds. Latency escalates from 2s → 10s → timeouts cascade. Classic "slow caller" scenario.

// Fix 1: timeout on downstream call
WebClient.create("http://payment-service")
    .get().uri("/charge/{id}", chargeId)
    .retrieve()
    .bodyToMono(ChargeResult.class)
    .timeout(Duration.ofSeconds(2))  // don't hold thread indefinitely
    .block();

// Fix 2: circuit breaker (Resilience4j)
@CircuitBreaker(name = "payment", fallbackMethod = "chargeFallback")
public ChargeResult charge(Long chargeId) { ... }

Scenario 2 — @Async pool starvation

Async pool of 10 threads. Each task calls external API averaging 10s. 10+ simultaneous submissions → all threads occupied → queue fills → RejectedExecutionException or indefinite queuing.

// Fix: separate executors per workload — slow external API gets its own pool
@Bean("externalApiExecutor")
public Executor externalApiExecutor() {
    ThreadPoolTaskExecutor e = new ThreadPoolTaskExecutor();
    e.setCorePoolSize(50);          // sized for expected concurrency
    e.setQueueCapacity(100);
    e.setRejectedExecutionHandler(new ThreadPoolExecutor.CallerRunsPolicy());
    e.initialize();
    return e;
}

@Async("externalApiExecutor")
public CompletableFuture<ApiResult> callExternalApi(Request req) { ... }

Scenario 3 — Deadlock via nested @Async on shared pool

Pool has 10 threads. TaskA (10 instances) each submit TaskB to the SAME pool. All 10 threads running TaskA → submit 10 TaskB → queue full (TaskB can't start) → TaskA blocks waiting for TaskB result → deadlock. No exceptions, no CPU, just frozen threads.

// BROKEN: nested tasks on same pool → deadlock
@Async("sharedPool")
public CompletableFuture<ResultA> taskA() {
    CompletableFuture<ResultB> b = taskB(); // also @Async("sharedPool")
    return b.thenApply(this::buildResultA);  // blocks waiting for pool thread
}

// FIX: separate pools for submitting and submitted tasks
@Async("parentPool")
public CompletableFuture<ResultA> taskA() {
    CompletableFuture<ResultB> b = taskB(); // @Async("childPool")
    return b.thenApply(this::buildResultA);
}

Scenario 4 — HikariCP + Tomcat mismatch

Tomcat: 200 threads. HikariCP: 10 connections. All 200 Tomcat threads try to execute DB queries → 190 block waiting for a HikariCP connection → connection timeout exceptions or extreme latency. Pool sizing must be coordinated.

# Fix: align connection pool to actual concurrency needs
spring.datasource.hikari.maximum-pool-size=50
# Then Tomcat threads above 50 doing DB work will queue at HikariCP (not hang forever)
spring.datasource.hikari.connection-timeout=3000  # fail fast vs. wait forever

Detection checklist

  • GET /actuator/threaddump: all threads in WAITING or BLOCKED state
  • GET /actuator/metrics/executor.queued?tag=name:Async-: queue depth climbing
  • CPU near 0% while latency high: classic I/O starvation signature
  • HikariCP logs: "Connection is not available, request timed out after Xms"
L6 Insight Thread starvation is the #1 cause of production latency spikes in Spring Boot services. The key L6 answer demonstrates that you think about pools as a system — Tomcat → async executor → HikariCP → downstream pools must all be sized together. "I'd set up dashboards on executor queue depth and HikariCP pending connections, not just CPU and memory, because starvation is invisible to traditional metrics until it's already catastrophic." Also know that virtual threads (Java 21) largely eliminate Scenario 1 and 2 for I/O-bound code, making them a compelling upgrade path.

How do you propagate SecurityContext, MDC, and request scope to @Async threads?
ASYNC_SECURITY
▼

The root problem: ThreadLocal isolation

Spring Security's SecurityContextHolder, SLF4J's MDC, and request-scoped beans all use ThreadLocal. When @Async hands execution to a thread pool thread, a fresh ThreadLocal context is created — all caller-thread state is invisible. Authentication becomes null, MDC trace IDs vanish, request-scoped beans throw ScopeNotActiveException.

SecurityContext propagation

// Option 1: DelegatingSecurityContextAsyncTaskExecutor
@Bean
public Executor securityAwareExecutor() {
    ThreadPoolTaskExecutor executor = new ThreadPoolTaskExecutor();
    executor.setCorePoolSize(10);
    executor.setMaxPoolSize(50);
    executor.setQueueCapacity(200);
    executor.setThreadNamePrefix("SecAsync-");
    executor.initialize();
    // Wraps each submitted Runnable to copy SecurityContext from caller thread
    return new DelegatingSecurityContextAsyncTaskExecutor(executor);
}

// Option 2: Spring Boot 3.2 + virtual threads (automatic)
// spring.threads.virtual.enabled=true
// SecurityContextHolder switches to InheritableThreadLocal propagation mode
// No wrapper needed — context flows naturally to child virtual threads

MDC (trace ID) propagation via TaskDecorator

TaskDecorator is Spring's hook to wrap every task submitted to a ThreadPoolTaskExecutor. It captures context at submission time (on the caller thread) and restores it on the worker thread before execution.

public class MdcTaskDecorator implements TaskDecorator {
    @Override
    public Runnable decorate(Runnable runnable) {
        // Captured on caller thread BEFORE task is submitted
        Map<String, String> mdcContext = MDC.getCopyOfContextMap();
        Authentication auth = SecurityContextHolder.getContext().getAuthentication();

        return () -> {
            // Runs on worker thread
            SecurityContext ctx = SecurityContextHolder.createEmptyContext();
            ctx.setAuthentication(auth);
            SecurityContextHolder.setContext(ctx);
            MDC.setContextMap(mdcContext != null ? mdcContext : Map.of());
            try {
                runnable.run();
            } finally {
                SecurityContextHolder.clearContext();
                MDC.clear();
            }
        };
    }
}

// Register the decorator on the executor
@Bean
public Executor contextAwareExecutor() {
    ThreadPoolTaskExecutor executor = new ThreadPoolTaskExecutor();
    executor.setCorePoolSize(10);
    executor.setTaskDecorator(new MdcTaskDecorator());  // <-- key line
    executor.initialize();
    return executor;
}

Request scope: cannot propagate — extract instead

Request-scoped beans (@RequestScope) are tied to an active HttpServletRequest. There is no HttpServletRequest on an async worker thread — the request may have already returned a response by the time the async task runs.

// BROKEN: request-scoped bean injected into @Async service
@Async
public void processOrder(Long orderId) {
    String userId = requestContext.getUserId(); // ScopeNotActiveException at runtime!
}

// FIX: extract needed data from request scope BEFORE the async call
public void handleRequest(HttpServletRequest request) {
    String userId = request.getAttribute("userId").toString(); // on request thread
    orderService.processOrderAsync(orderId, userId);           // pass as param
}

@Async
public void processOrderAsync(Long orderId, String userId) {
    // userId is a plain String — no scope needed
}
Tip — combine everything in one decorator A single TaskDecorator can handle MDC propagation, SecurityContext copying, custom attributes, and even OpenTelemetry span context. Register one composite decorator rather than multiple executor wrappers — it's simpler and executed atomically per task.
L6 Insight Context propagation gaps are a leading cause of security audit failures and distributed tracing gaps in async services. At L6, you must proactively ask: "what context does this async task need, and have I proven it's available?" A missing trace ID means 30% of your spans in Jaeger have no parent — your distributed trace is broken. A missing SecurityContext means authorization checks silently pass (if null auth = PUBLIC) or silently fail (if null auth = DENIED), both dangerous. Treat context propagation as a first-class design concern, not an afterthought.

Compare event-driven (ApplicationEvents) vs polling vs @Async for background work. When to use each?
EVENT_DRIVEN
▼

Pattern 1: @Async (thread pool offload)

@Service
public class RegistrationService {

    public void registerUser(UserDTO dto) {
        User user = userRepo.save(User.from(dto));
        emailService.sendWelcomeEmail(user);  // returns immediately; email sent async
    }
}

@Service
public class EmailService {
    @Async("emailExecutor")
    public void sendWelcomeEmail(User user) {
        emailClient.send(WelcomeEmail.for(user));
    }
}

Use when: fire-and-forget side effects with no durability requirement, caller does not need result.

Downsides: tight coupling (RegistrationService knows about EmailService), no retry on failure, task lost if app restarts mid-execution.

Pattern 2: ApplicationEvent (in-process event bus)

// Publisher — decoupled from handler
eventPublisher.publishEvent(new UserRegisteredEvent(user));

// Handler — can be in any Spring bean
@Component
public class NotificationHandler {

    @EventListener
    @Async("notificationExecutor")
    public void onUserRegistered(UserRegisteredEvent event) {
        emailClient.sendWelcome(event.getUser());
        smsService.sendVerification(event.getUser());
    }

    @EventListener
    @Async("auditExecutor")
    public void auditUserRegistration(UserRegisteredEvent event) {
        auditLog.record("USER_REGISTERED", event.getUser().getId());
    }
}

Use when: decouple sender from multiple handlers, cross-cutting concerns (audit, notifications, cache invalidation), same JVM.

Downsides: still in-process — events lost on app restart, no persistence, no retry, no cross-service fan-out.

Pattern 3: @TransactionalEventListener (AFTER_COMMIT)

@Component
public class OrderEventHandler {

    // Fires ONLY after the publishing transaction successfully commits to DB
    // Prevents sending email for orders that get rolled back
    @TransactionalEventListener(phase = TransactionPhase.AFTER_COMMIT)
    @Async("emailExecutor")
    public void onOrderConfirmed(OrderConfirmedEvent event) {
        emailClient.sendOrderConfirmation(event.getOrder());
    }
}

Use when: handler must not fire if the business transaction rolls back. Best for: confirmation emails, cache eviction, downstream system notifications after a DB write is durable.

Critical gotcha: @TransactionalEventListener runs outside the original transaction (after commit). Any DB work in the handler requires a new transaction (@Transactional(propagation = REQUIRES_NEW)).

Pattern 4: @Scheduled polling (outbox relay)

@Scheduled(fixedDelay = 5000)
public void processOutboxEvents() {
    List<OutboxEvent> pending = outboxRepo.findPending(100);
    pending.forEach(event -> {
        kafkaProducer.publish(event.getTopic(), event.getPayload());
        outboxRepo.markPublished(event.getId());
    });
}

Use when: implementing the transactional outbox pattern, periodic batch processing, data sync jobs.

Downsides: latency = poll interval; wasted cycles when nothing to do; all-or-nothing retry semantics.

Pattern 5: External messaging (Kafka / RabbitMQ)

Durable, retryable, cross-service, at-least-once delivery. Events survive application restarts. Consumer groups enable fan-out to many services. Dead-letter queues handle poison messages.

Use when: cross-service events, durable delivery is required, high volume, consumer needs to replay events, multiple independent consumers.

Decision guide

RequirementPattern
Same JVM, no durability, simple offload@Async

Tip — transactional outbox combines patterns For critical events: write the event to an outbox table in the same transaction as the business operation (atomicity), then a @Scheduled poller or Debezium CDC relay publishes it to Kafka. This gives you: transaction-safe event emission + durable delivery + retry. Best of all worlds at the cost of extra infrastructure.
L6 Insight At L6 you're expected to match the tool to the requirement rather than defaulting to the most complex solution. Decision heuristic: "Is losing this event on a crash acceptable?" If yes, @Async or @EventListener is sufficient and far simpler. If no — the event must be durable — use Kafka + outbox. For critical business events (order confirmed → invoice generated → fulfillment triggered), Kafka + transactional outbox is non-negotiable. For non-critical UX notifications (welcome email, analytics ping), @TransactionalEventListener @Async is the pragmatic choice. Proposing Kafka for every event in a small service is over-engineering and signals poor judgment.
