Explain service discovery in microservices. Compare client-side (Eureka) vs server-side (Kubernetes) discovery.
SERVICE_DISCOVERY
▼

The Service Discovery Problem

In microservices, instances are ephemeral: they start, crash, scale up/down, and migrate. IP addresses and ports change constantly. Services can't use static URLs — they need a mechanism to find each other at runtime.

Client-Side Discovery: Netflix Eureka

Services register themselves with the Eureka registry on startup and send heartbeats every 30 seconds. When service A needs to call service B, A queries Eureka for available instances of B, selects one (via client-side load balancer), and calls directly.

// Eureka Server
@SpringBootApplication
@EnableEurekaServer
public class ServiceRegistryApplication { ... }

// Service registering with Eureka (client)
// application.yml:
spring:
  application.name: user-service
eureka:
  client:
    serviceUrl.defaultZone: http://eureka-server:8761/eureka/
  instance:
    preferIpAddress: true
    leaseRenewalIntervalInSeconds: 10  # heartbeat every 10s (default 30s)
    leaseExpirationDurationInSeconds: 30

// Calling another service via discovery
@Autowired DiscoveryClient discoveryClient;

List<ServiceInstance> instances = discoveryClient.getInstances("order-service");
String baseUrl = instances.get(0).getUri().toString(); // manual instance selection

// Or with @LoadBalanced RestTemplate (automatic lb)
@Bean
@LoadBalanced
public RestTemplate restTemplate() { return new RestTemplate(); }

// URL uses service name, not IP
restTemplate.getForObject("http://order-service/orders/{id}", OrderDTO.class, orderId);

Server-Side Discovery: Kubernetes

In K8s, every Service gets a stable DNS name. Pods register/deregister automatically via the control plane. Client calls http://user-service — kube-proxy + iptables/IPVS handles routing to healthy pods transparently. The client has zero discovery logic.

# K8s Service definition — creates stable DNS entry: user-service.default.svc.cluster.local
apiVersion: v1
kind: Service
metadata:
  name: user-service
spec:
  selector:
    app: user-service     # routes to pods with this label
  ports:
    - port: 80
      targetPort: 8080
  type: ClusterIP

# Service calls just use the service name:
# http://user-service/users/42
# http://user-service.default.svc.cluster.local/users/42

# Spring Cloud Kubernetes: discover K8s services as Spring beans
# spring-cloud-starter-kubernetes-client-all
spring:
  cloud:
    kubernetes:
      discovery:
        all-namespaces: false
        enabled: true

Comparison

CriterionEureka (Client-Side)Kubernetes (Server-Side)
Discovery logicIn application codePlatform-managed (transparent)
Load balancingClient-side (Spring LB)kube-proxy / IPVS
Single point of failureEureka cluster (mitigated with peer replication)K8s control plane (HA by design)
Registry stalenessUp to 3×heartbeat (90s worst case)Near-instant (endpoints updated by kubelet)
Extra infrastructureYes — run Eureka clusterNo — built into K8s
Polyglot supportRequires Eureka client libraryWorks for any language (DNS)
⚠ Gotcha Eureka has a known staleness window: it takes up to 3 heartbeat cycles (90 seconds default) to detect a dead instance and remove it from the registry. During this window, clients receive stale entries and calls fail. Tune leaseExpirationDurationInSeconds down in K8s environments or use readiness probes to remove bad instances from LB faster.
🎯 Google L6 Angle "For new Kubernetes deployments, use K8s native service discovery — it's platform-managed, works for all languages, and adds no extra component to operate. Eureka made sense in pre-container VM environments. Today, if your team is running K8s, the right answer is K8s Services + DNS, supplemented with Spring Cloud Kubernetes for config awareness. Don't add Eureka as an extra layer on top of K8s — you're solving a problem K8s already solved."

Spring Cloud LoadBalancer

Spring Cloud LoadBalancer replaces the deprecated Netflix Ribbon. It provides client-side load balancing that integrates with service discovery. Auto-configured when spring-cloud-starter-loadbalancer is on the classpath.

// @LoadBalanced makes WebClient/RestTemplate resolve service names via LB
@Configuration
public class LoadBalancedClients {
    @Bean
    @LoadBalanced
    public WebClient.Builder webClientBuilder() {
        return WebClient.builder();
    }

    @Bean
    @LoadBalanced
    public RestTemplate restTemplate() {
        return new RestTemplate();
    }
}

// Usage — "user-service" resolved via discovery + load balancer
@Autowired WebClient.Builder webClientBuilder;
WebClient webClient = webClientBuilder.baseUrl("http://user-service").build();
Mono<UserDTO> user = webClient.get().uri("/users/{id}", id).retrieve().bodyToMono(UserDTO.class);

Built-in Strategies

// Default: RoundRobinLoadBalancer — cycles through instances sequentially

// Configure RandomLoadBalancer per service
@Configuration
@LoadBalancerClient(name = "product-service", configuration = RandomLBConfig.class)
public class ProductServiceLBConfig {}

@Configuration
public class RandomLBConfig {
    @Bean
    public ReactorLoadBalancer<ServiceInstance> randomLoadBalancer(
            Environment env, LoadBalancerClientFactory factory) {
        String name = env.getProperty(LoadBalancerClientFactory.PROPERTY_NAME);
        return new RandomLoadBalancer(
            factory.getLazyProvider(name, ServiceInstanceListSupplier.class), name);
    }
}

Custom Strategy: Sticky Session (Consistent Hashing)

// Route same user to same instance (useful for stateful sessions)
public class StickySessionLoadBalancer implements ReactorServiceInstanceLoadBalancer {
    private final ObjectProvider<ServiceInstanceListSupplier> supplierProvider;

    @Override
    public Mono<Response<ServiceInstance>> choose(Request request) {
        HttpHeaders headers = ((RequestDataContext) request.getContext())
            .getClientRequest().getHeaders();
        String userId = headers.getFirst("X-User-Id");

        return supplierProvider.getIfAvailable()
            .get(request)
            .next()
            .map(instances -> {
                if (instances.isEmpty()) return new EmptyResponse();
                // Consistent hash: same userId always goes to same instance index
                int index = Math.abs(userId.hashCode()) % instances.size();
                return new DefaultResponse(instances.get(index));
            });
    }
}

Zone-Aware Load Balancing

// Prefer instances in same availability zone — reduces cross-AZ latency and cost
@Configuration
public class ZoneAwareLBConfig {
    @Bean
    public ServiceInstanceListSupplier zoneAwareSupplier(
            ConfigurableApplicationContext context) {
        return ServiceInstanceListSupplier.builder()
            .withDiscoveryClient()
            .withZonePreference()   // prefers same AZ, falls back to others
            .withHealthChecks()     // filters unhealthy instances
            .withCaching()          // caches list (35s TTL)
            .build(context);
    }
}

Instance List Caching

By default, LoadBalancer refreshes the instance list from discovery on every request (no cache). For high-throughput services, add caching:

spring:
  cloud:
    loadbalancer:
      cache:
        enabled: true
        ttl: 35s          # cache instance list for 35 seconds
        capacity: 256     # max number of service names to cache
⚠ Gotcha Without caching, every WebClient call triggers a discovery lookup. At 10k req/s, this hammers Eureka/K8s API server. Always enable caching in production. With K8s, prefer DNS-based discovery over API-server polling — DNS is cached by the OS resolver and far cheaper.
🎯 Google L6 Angle "Zone-aware load balancing is critical in multi-AZ deployments. Cross-AZ traffic costs money (AWS charges per GB) and adds latency. ZonePreferenceServiceInstanceListSupplier preferentially routes to instances in the same AZ, falling back cross-AZ only when the local AZ has no healthy instances. This is a non-obvious optimization that matters at scale."

Explain the circuit breaker pattern and Resilience4j's state machine. What are the configuration parameters?
CIRCUIT_BREAKER
▼

The Circuit Breaker Pattern

Named after electrical circuit breakers: when too many failures occur, "trip" the circuit to stop sending requests to the failing service, give it time to recover, then test with a few requests before fully reopening.

Resilience4j State Machine

StateBehaviorTransition
CLOSEDNormal operation. All calls pass through. Tracking failure/slow call rates.→ OPEN when failure rate ≥ threshold over sliding window
OPENFast-fail all calls with CallNotPermittedException. No calls reach the service.→ HALF_OPEN after waitDurationInOpenState
HALF_OPENAllow limited probe calls (permittedNumberOfCallsInHalfOpenState).→ CLOSED if probes succeed; → OPEN if probes fail
DISABLEDCircuit breaker disabled, all calls pass through.Manual only
FORCED_OPENAll calls rejected regardless of state.Manual only (for maintenance windows)
@Configuration
public class ResilienceConfig {

    @Bean
    public CircuitBreakerRegistry circuitBreakerRegistry() {
        CircuitBreakerConfig config = CircuitBreakerConfig.custom()
            // Sliding window: COUNT_BASED or TIME_BASED
            .slidingWindowType(COUNT_BASED)
            .slidingWindowSize(10)               // track last 10 calls
            .minimumNumberOfCalls(5)             // need at least 5 calls before calculating rates

            // Failure thresholds
            .failureRateThreshold(50f)           // >50% failures → OPEN
            .slowCallRateThreshold(30f)          // >30% slow calls → OPEN
            .slowCallDurationThreshold(Duration.ofSeconds(2))  // defines "slow"

            // Timing
            .waitDurationInOpenState(Duration.ofSeconds(30))
            .permittedNumberOfCallsInHalfOpenState(3)
            .automaticTransitionFromOpenToHalfOpenEnabled(true)

            // Exception filtering
            .recordExceptions(IOException.class, TimeoutException.class,
                              ServiceUnavailableException.class)
            .ignoreExceptions(BusinessException.class,  // business errors don't trip circuit
                             ValidationException.class)
            .build();

        return CircuitBreakerRegistry.of(config);
    }
}

Usage with @CircuitBreaker Annotation

@Service
public class PaymentService {

    @CircuitBreaker(name = "paymentGateway", fallbackMethod = "paymentFallback")
    public PaymentResult processPayment(PaymentRequest req) {
        return paymentGatewayClient.charge(req);
    }

    // Fallback must have same signature + Exception parameter
    private PaymentResult paymentFallback(PaymentRequest req, CallNotPermittedException e) {
        log.warn("Payment circuit OPEN — queuing payment for async retry");
        return paymentQueueService.enqueueForRetry(req);
    }

    private PaymentResult paymentFallback(PaymentRequest req, Exception e) {
        // Catch-all fallback
        log.error("Payment failed: {}", e.getMessage());
        return PaymentResult.failed("Payment temporarily unavailable");
    }
}

Metrics and Monitoring

// Automatically exposed via Micrometer:
// resilience4j.circuitbreaker.state{name="paymentGateway"} → 0=CLOSED, 1=OPEN, 2=HALF_OPEN
// resilience4j.circuitbreaker.calls{name,kind="successful/failed/not_permitted/slow"}
// resilience4j.circuitbreaker.failure.rate{name="paymentGateway"}

// Event publisher for custom monitoring
circuitBreaker.getEventPublisher()
    .onStateTransition(e -> metrics.recordStateChange(e.getCircuitBreakerName(),
        e.getStateTransition().getFromState(), e.getStateTransition().getToState()))
    .onCallNotPermitted(e -> metrics.incrementFastFail(e.getCircuitBreakerName()));
⚠ Gotcha The minimumNumberOfCalls parameter is critical and often overlooked. Without it, a single failed call in the first 5 calls sets failure rate to 100% and trips the circuit. In low-traffic services, set this to a meaningful minimum (e.g., 10–20) to avoid false tripping.
✅ Best Practice Configure a slow call threshold in addition to failure rate. A service that's responding slowly (e.g., 5-second responses under load) but not failing will exhaust thread pools and cause cascading failures — yet the circuit breaker won't trip without slowCallRateThreshold.
🎯 Google L6 Angle "A circuit breaker without a meaningful fallback is just a fast-fail mechanism — better than nothing, but not resilience. The L6 design question is: when the payment circuit opens, what does the user experience? 'Queued for async retry with confirmation email' is a product answer. 'HTTP 503' is a technical answer. The former shows systems thinking."

Explain retry and timeout patterns. How do you configure them with Resilience4j in Spring?

▼

Retry Pattern

Automatically re-invoke a failed operation. Useful for transient failures: network blips, temporary DB contention, brief service restarts. Critical caveat: only retry idempotent operations, or operations with idempotency keys.

// application.yml
resilience4j:
  retry:
    instances:
      orderService:
        maxAttempts: 3
        waitDuration: 500ms
        enableExponentialBackoff: true
        exponentialBackoffMultiplier: 2.0    # 500ms, 1000ms, 2000ms
        exponentialMaxWaitDuration: 10s
        randomizedWaitFactor: 0.5            # ±50% jitter on each wait
        retryOnResultPredicate: com.example.RetryOnHttpError  # retry on specific results
        retryExceptions:
          - java.io.IOException
          - com.example.TransientServiceException
        ignoreExceptions:
          - com.example.BusinessRuleException
          - com.example.ValidationException
// Annotation-based
@Service
public class OrderService {

    @Retry(name = "orderService", fallbackMethod = "orderFallback")
    @CircuitBreaker(name = "orderService")  // circuit breaker wraps retry
    public OrderDTO getOrder(Long orderId) {
        return orderClient.fetchOrder(orderId);
    }

    private OrderDTO orderFallback(Long orderId, Exception e) {
        log.warn("Order service unavailable, returning cached order {}", orderId);
        return orderCache.get(orderId).orElseThrow(() -> new ServiceUnavailableException(e));
    }
}

// Programmatic retry with custom logic
RetryConfig retryConfig = RetryConfig.custom()
    .maxAttempts(3)
    .waitDuration(Duration.ofMillis(500))
    .enableExponentialBackoff()
    .exponentialBackoffMultiplier(2.0)
    .retryOnException(ex -> ex instanceof IOException || isTransient(ex))
    .retryOnResult(response -> response.getStatus() == 429)  // retry on rate limit
    .build();

Timeout Pattern

Prevent calls from hanging indefinitely. Without timeouts, a slow dependency can cause thread pool exhaustion and cascade failures across your service.

resilience4j:
  timelimiter:
    instances:
      inventoryService:
        timeoutDuration: 2s
        cancelRunningFuture: true  # cancel the underlying Future on timeout

# Annotation (requires async - @Async or CompletableFuture return type)
@TimeLimiter(name = "inventoryService", fallbackMethod = "inventoryFallback")
@CircuitBreaker(name = "inventoryService")
public CompletableFuture<InventoryDTO> checkInventory(String sku) {
    return CompletableFuture.supplyAsync(() -> inventoryClient.check(sku));
}

Composition Order: The Onion Model

When combining Resilience4j patterns, order matters. The correct composition (inner → outer): TimeLimiter → Retry → CircuitBreaker → Bulkhead

// Annotation order (Spring executes outermost annotation first = applied last = innermost):
@Bulkhead(name = "inventory")         // outermost: applied last in AOP chain
@CircuitBreaker(name = "inventory")   // circuit tracks retried failures
@Retry(name = "inventory")            // retries before circuit counts it
@TimeLimiter(name = "inventory")      // innermost: timeout per individual attempt
public CompletableFuture<InventoryDTO> checkInventory(String sku) {
    return CompletableFuture.supplyAsync(() -> client.check(sku));
}

Interaction Between Retry and Circuit Breaker

ConfigurationBehaviorUse When
Retry inside Circuit BreakerEach retry attempt counts toward circuit breaker failure rateYou want retries to contribute to circuit tripping
Circuit Breaker inside RetryCircuit opens after enough attempts; retry stops when circuit is OPENYou want the circuit to absorb retried failures
⚠ Gotcha Retry on non-idempotent operations (POST /payments) without idempotency keys causes double-charges, duplicate orders, etc. Always check: is this operation safe to retry? For POST operations, add an idempotency key header (Idempotency-Key: UUID) that the server uses to deduplicate.
🎯 Google L6 Angle "Jitter is non-negotiable in distributed retry. Without jitter, when a service recovers, all retrying clients fire simultaneously — a 'thundering herd' that immediately overwhelms the recovering service and sends it back down. Randomizing the retry delay spreads load over time. exponentialBackoffMultiplier + randomizedWaitFactor is the correct default. Interviewers notice when candidates mention jitter unprompted."

Explain Spring Cloud Gateway. How does routing, filtering, and rate limiting work?
API_GATEWAY
▼

Spring Cloud Gateway Architecture

Spring Cloud Gateway is a reactive API gateway built on Spring WebFlux. It replaces Netflix Zuul 1 (blocking). Request processing: Route Predicate (match?) → Pre-Filters (auth, logging) → Proxied to Service → Post-Filters (add response headers, logging).

Route Configuration

spring:
  cloud:
    gateway:
      routes:
        - id: user-service
          uri: lb://USER-SERVICE     # "lb://" = load-balanced via discovery
          predicates:
            - Path=/api/users/**
            - Method=GET,POST,PUT
            - Header=X-API-Version, 2
            - Weight=user-service, 90  # canary: 90% here
          filters:
            - StripPrefix=1                     # remove /api prefix before forwarding
            - AddRequestHeader=X-Gateway-Version, 2.0
            - AddResponseHeader=X-Response-Time, 42ms
            - CircuitBreaker=name=userService,fallbackUri=forward:/fallback/users
            - Retry=retries:3,statuses:BAD_GATEWAY:SERVICE_UNAVAILABLE,methods:GET
            - RequestRateLimiter=redis-rate-limiter.replenishRate:100,redis-rate-limiter.burstCapacity:200

        - id: legacy-v1
          uri: lb://USER-SERVICE-V1
          predicates:
            - Path=/api/users/**
            - Weight=user-service, 10  # canary: 10% to new version

      # Global default filters applied to all routes
      default-filters:
        - AddRequestHeader=X-Gateway-Request-Id, \${random.uuid}
        - DedupeResponseHeader=Access-Control-Allow-Credentials Access-Control-Allow-Origin

Redis Rate Limiter (Token Bucket)

// application.yml
spring:
  cloud:
    gateway:
      routes:
        - id: api-route
          uri: lb://API-SERVICE
          predicates:
            - Path=/api/**
          filters:
            - name: RequestRateLimiter
              args:
                redis-rate-limiter.replenishRate: 10    # 10 tokens added/sec
                redis-rate-limiter.burstCapacity: 20    # max 20 tokens (burst)
                redis-rate-limiter.requestedTokens: 1  # cost per request
                key-resolver: "#{@userIdKeyResolver}"  # rate limit per user

// Custom key resolver — rate limit by user ID extracted from JWT
@Bean
public KeyResolver userIdKeyResolver() {
    return exchange -> {
        String token = exchange.getRequest().getHeaders().getFirst("Authorization");
        String userId = jwtParser.extractUserId(token);
        return Mono.just(userId != null ? userId : "anonymous");
    };
}

// IP-based rate limiter (simpler fallback)
@Bean
public KeyResolver ipKeyResolver() {
    return exchange -> Mono.just(
        exchange.getRequest().getRemoteAddress().getAddress().getHostAddress()
    );
}

Custom Global Filter

// Runs for every route: authentication, logging, tracing
@Component
@Order(Ordered.HIGHEST_PRECEDENCE)
public class AuthenticationFilter implements GlobalFilter {

    @Autowired JwtTokenValidator tokenValidator;

    @Override
    public Mono<Void> filter(ServerWebExchange exchange, GatewayFilterChain chain) {
        String path = exchange.getRequest().getPath().value();

        // Skip auth for public paths
        if (isPublicPath(path)) {
            return chain.filter(exchange);
        }

        String token = extractBearerToken(exchange.getRequest().getHeaders());
        if (token == null) {
            exchange.getResponse().setStatusCode(HttpStatus.UNAUTHORIZED);
            return exchange.getResponse().setComplete();
        }

        return tokenValidator.validate(token)
            .flatMap(claims -> {
                // Forward decoded user info to downstream services
                ServerWebExchange mutated = exchange.mutate()
                    .request(r -> r.header("X-User-Id", claims.getSubject())
                                   .header("X-User-Roles", String.join(",", claims.getRoles())))
                    .build();
                return chain.filter(mutated);
            })
            .onErrorResume(e -> {
                exchange.getResponse().setStatusCode(HttpStatus.UNAUTHORIZED);
                return exchange.getResponse().setComplete();
            });
    }
}

Spring Cloud Gateway makes this trivial with the Weight predicate — a capability that would otherwise require infrastructure-level (NGINX, Istio) configuration."

Spring Cloud Config Server

Centralizes externalized configuration for all microservices. Services fetch their config from the Config Server on startup instead of bundling it in their JARs. Backends: Git (default), filesystem, Vault, JDBC, AWS S3.

# Config Server setup
@SpringBootApplication
@EnableConfigServer
public class ConfigServerApplication { ... }

# Config Server application.yml
spring:
  cloud:
    config:
      server:
        git:
          uri: https://github.com/myorg/config-repo
          default-label: main              # branch
          search-paths: '{application}'    # per-service subdirectory
          clone-on-start: true             # fail fast if Git unreachable
          timeout: 10                      # Git clone timeout seconds
          force-pull: true                 # always pull latest
        encrypt:
          enabled: true                    # server-side decryption

# Encryption key
encrypt:
  key: my-secret-key  # symmetric; use RSA key pair for production

Client Configuration

# Spring Boot 3: import config directly (no bootstrap.yml needed)
spring:
  application:
    name: user-service
  profiles:
    active: prod
  config:
    import: "configserver:http://config-server:8888"
  cloud:
    config:
      fail-fast: true     # fail startup if config server unreachable
      retry:
        max-attempts: 6
        initial-interval: 1000
        multiplier: 1.1
        max-interval: 2000

Configuration File Resolution Order (Highest → Lowest Priority)

  1. {application}-{profile}.yml — e.g., user-service-prod.yml
  2. {application}.yml — e.g., user-service.yml
  3. application-{profile}.yml — shared profile-specific
  4. application.yml — shared defaults (all services)

⚠ Gotcha @RefreshScope is not free — the bean is recreated on every refresh, which tears down and re-creates all dependencies. Layer them together using Spring Boot's property source precedence."

Explain distributed tracing in Spring Boot 3. How does Micrometer Tracing work with Zipkin/Jaeger?
TRACING
▼

Why Distributed Tracing?

A single user request traverses 5–10 services. When it's slow or fails, you need to know: which service was slow? Which hop caused the latency? Without distributed tracing, you're grepping logs across 10 services hoping timestamps align. With tracing, you click one trace ID and see the entire request timeline.

Core Concepts

  • Trace: represents the entire journey of one request across all services. Identified by traceId.
  • Span: a single unit of work (one service call, one DB query). Has spanId, start time, duration, tags, and events.
  • Context Propagation: trace and span IDs are forwarded in HTTP headers between services.

Custom Spans

@Service
@RequiredArgsConstructor
public class RecommendationService {

    private final Tracer tracer;

    public List<Product> getRecommendations(Long userId) {
        Span span = tracer.nextSpan().name("ml-model-inference");
        try (Tracer.SpanInScope ws = tracer.withSpan(span.start())) {
            span.tag("userId", String.valueOf(userId));
            span.tag("model.version", "v2.3");
            span.event("inference-started");

            List<Product> recommendations = mlModelClient.infer(userId);

            span.tag("result.count", String.valueOf(recommendations.size()));
            span.event("inference-completed");
            return recommendations;
        } catch (Exception e) {
            span.error(e);
            throw e;
        } finally {
            span.end();
        }
    }
}

What is a service mesh? How does it compare to Spring Cloud patterns?

Service Mesh Architecture

A service mesh is an infrastructure layer that handles service-to-service communication. A sidecar proxy (typically Envoy) is injected into every pod automatically by the control plane. All network traffic goes through the sidecar — the application is completely unaware.

Control plane: manages config, certificates, traffic policies (Istio Pilot, Linkerd Control Plane). Data plane: the sidecar proxies that handle actual traffic (Envoy).

Key Advantage: mTLS Without Code Changes

With Istio, all service-to-service traffic is automatically encrypted with mutual TLS. Certificates are auto-rotated. No KeyStore management in Java code. In a polyglot environment (Java + Go + Python), this is transformative.

When to Use Each

  • Spring Cloud: single-language Java shop, more control over resilience logic, lighter infrastructure, team comfortable managing it in code
  • Service Mesh: polyglot microservices, platform team wants to enforce policies without app team changes, enterprise mTLS requirement, complex traffic management (canary, dark launches at infra level)
  • Both: defense in depth — mesh for infrastructure concerns (mTLS, infra-level circuit breaking), Spring Cloud for application-level resilience with custom fallbacks
⚠ Gotcha Istio adds latency per hop (Envoy sidecar = ~1ms per call). At 50 service calls per request, that's 50ms added. For latency-sensitive use cases, benchmark carefully. Also, Istio's operational complexity is significant — it's a full distributed system to operate alongside your application.
🎯 Google L6 Angle "In Google-scale Kubernetes deployments, Istio (or a managed equivalent like GKE Service Mesh) handles the infrastructure cross-cutting concerns. Application teams use Spring Cloud for application-level resilience where they need custom fallback behavior — the mesh can't know your business logic. It's not either/or; it's complementary layers. The mesh handles 'is the network encrypted and load-balanced?' — Spring Cloud handles 'what do we return when the inventory service is down?'"

Spring Cloud Stream: Programming Model

Spring Cloud Stream abstracts Kafka, RabbitMQ, Azure Service Bus, AWS Kinesis behind a unified functional API. Write business logic as plain Java functions — the framework wires them to message topics automatically.

// Functional model (Spring Cloud Stream 3+)
// Functions are auto-detected and bound to input/output topics

@Configuration
public class OrderProcessingFunctions {

    // Function: receives Order, returns Payment (auto-bound: orders → payments topic)
    @Bean
    public Function<Order, Payment> processOrder(PaymentService paymentService) {
        return order -> {
            log.info("Processing order {}", order.getId());
            return paymentService.process(order);
        };
    }

    // Consumer: receives Payment, no output (audit log)
    @Bean
    public Consumer<Payment> auditPayment(AuditService auditService) {
        return payment -> auditService.record(payment);
    }

    // Supplier: emits events periodically (no input)
    @Bean
    public Supplier<Heartbeat> heartbeat() {
        return () -> new Heartbeat(Instant.now());  // called by scheduler
    }

    // Reactive functions are also supported
    @Bean
    public Function<Flux<Order>, Flux<Payment>> processOrdersReactively(PaymentService svc) {
        return orders -> orders
            .filter(o -> o.getAmount().compareTo(BigDecimal.ZERO) > 0)
            .flatMap(o -> svc.processAsync(o), 16);
    }
}

Design a resilient service-to-service call. What patterns compose together?
RESILIENCE
▼

The Full Resilience Stack

A production-grade service call needs multiple patterns composing together as defense-in-depth layers:

@Service
@RequiredArgsConstructor
public class InventoryService {

    private final InventoryClient inventoryClient;
    private final InventoryCache inventoryCache;

    // Composition order (inner → outer):
    // 1. TimeLimiter: timeout per individual call attempt
    // 2. Retry: retry transient failures with backoff
    // 3. CircuitBreaker: track failures, fast-fail when service is down
    // 4. Bulkhead: limit concurrent calls to this service

    @Bulkhead(name = "inventoryService")
    @CircuitBreaker(name = "inventoryService", fallbackMethod = "getCachedInventory")
    @Retry(name = "inventoryService")
    @TimeLimiter(name = "inventoryService")
    public CompletableFuture<InventoryDTO> getInventory(String sku) {
        return CompletableFuture.supplyAsync(() -> inventoryClient.checkStock(sku));
    }

    // Fallback: circuit open OR all retries exhausted
    private CompletableFuture<InventoryDTO> getCachedInventory(String sku, Exception ex) {
        log.warn("Inventory service unavailable for sku={}, cause={}", sku, ex.getMessage());
        return CompletableFuture.completedFuture(
            inventoryCache.get(sku)
                .map(cached -> cached.withStale(true))  // mark as potentially stale
                .orElse(InventoryDTO.assumeInStock(sku)) // fail open: assume in stock
        );
    }
}

// Same patterns, reactive style
public Mono<InventoryDTO> getInventoryReactive(String sku) {
    return inventoryClient.checkStockReactive(sku)
        .timeout(Duration.ofSeconds(2))            // timeout
        .retryWhen(Retry.backoff(3, Duration.ofMillis(300))
            .maxBackoff(Duration.ofSeconds(5))
            .jitter(0.5)
            .filter(ex -> ex instanceof IOException))
        .transformDeferred(CircuitBreakerOperator.of(circuitBreaker))  // circuit breaker
        .transformDeferred(BulkheadOperator.of(bulkhead))              // bulkhead
        .onErrorResume(ex -> inventoryCache.get(sku)
            .map(Mono::just)
            .orElse(Mono.just(InventoryDTO.assumeInStock(sku))));
}

How do you implement liveness and readiness probes for Kubernetes with Spring Boot Actuator?
HEALTH
▼

Liveness vs Readiness: Critical Distinction

Liveness probe failure: K8s restarts the pod. Use this ONLY when the application is in an unrecoverable state and must be restarted (deadlock, OOM, infinite loop). Never fail liveness because a dependency (DB, Redis) is down — that causes a restart storm.

Readiness probe failure: K8s removes the pod from the Service load balancer (stops sending traffic). The pod keeps running. Use this when the app cannot handle requests: cold start, DB connection pool exhausted, dependency down, warming up ML model.

Spring Boot Actuator: Auto-Configuration

# application.yml — Spring Boot auto-detects K8s environment
management:

    health:
      show-details: always
      show-components: always
      probes:
        enabled: true           # explicitly enable /liveness and /readiness
  endpoints:
    web:
      exposure:
        include: health,info,metrics,prometheus

  health:
    livenessstate:
      enabled: true
    readinessstate:
      enabled: true
    db:
      enabled: true             # DataSource health check
    redis:
      enabled: true             # Redis health check
    kafka:
      enabled: true             # Kafka health check (reads metadata)

            successThreshold: 2         # require 2 successes before re-adding to LB

          startupProbe:                 # for slow-starting apps (Spring Boot cold start)
            httpGet:
              path: /actuator/health/liveness
              port: 8080
            failureThreshold: 30        # allow up to 5 minutes (30 × 10s) for startup
            periodSeconds: 10

// Custom health indicator: contributes to /actuator/health/readiness
@Component
public class CriticalDependencyHealthIndicator implements HealthIndicator {

    @Autowired DataSource dataSource;
    @Autowired RedisConnectionFactory redisConnectionFactory;

    @Override
    public Health health() {
        Health.Builder builder = Health.up();

        // Check DB connectivity
        try (Connection conn = dataSource.getConnection()) {
            conn.prepareStatement("SELECT 1").execute();
            builder.withDetail("db", "reachable");
        } catch (Exception e) {
            return Health.down()
                .withDetail("db", "unreachable: " + e.getMessage())
                .build();  // readiness probe fails → pod removed from LB
        }

        return builder.build();
    }
}

// Programmatic liveness/readiness state management
@Autowired ApplicationContext ctx;

// Mark app as not ready (e.g., during graceful shutdown warm-down)
AvailabilityChangeEvent.publish(ctx, ReadinessState.REFUSING_TRAFFIC);

// Mark app as ready again
AvailabilityChangeEvent.publish(ctx, ReadinessState.ACCEPTING_TRAFFIC);

// Mark app as broken (liveness will fail → K8s restarts pod)

Graceful Shutdown

  shutdown: graceful           # wait for in-flight requests to complete

spring:
  lifecycle:
    timeout-per-shutdown-phase: 30s   # max 30s for graceful shutdown

# On SIGTERM:
# 1. Spring publishes ReadinessState.REFUSING_TRAFFIC (readiness probe fails)
# 2. K8s removes pod from load balancer (propagates in ~5s with periodSeconds=5)
# 3. Spring waits for in-flight requests to complete (up to 30s)
# 4. Spring shuts down context cleanly
⚠ Gotcha Never add DB health checks to the liveness probe. If your database goes down, you don't want K8s restarting all pods in a restart loop — that makes recovery harder. DB failure belongs in readiness (stop sending traffic), not liveness (restart the pod).
🎯 Google L6 Angle "The startup probe is often overlooked but critical for Spring Boot. Cold start with AOT compilation and class loading can take 30–60 seconds. Without a startup probe, liveness probe failures during startup cause premature restarts — your pod restart-loops before it even finishes starting. startupProbe disables liveness/readiness probes until the startup probe succeeds, giving the app time to initialize."

Implement a Saga orchestrator using Spring State Machine. What are the alternatives?
SAGA_IMPL
▼

The Saga Pattern

Long-running business transactions spanning multiple microservices. Each step has a compensating transaction. Two styles: Choreography (services react to events) and Orchestration (central coordinator drives steps). Orchestration is easier to reason about for complex sagas.

Order Saga: States and Transitions

// States and events enums
public enum OrderState {
    ORDER_CREATED, INVENTORY_RESERVED, PAYMENT_PROCESSED,
    ORDER_CONFIRMED, COMPENSATION_TRIGGERED, ORDER_CANCELLED
}
public enum OrderEvent {
    RESERVE_INVENTORY, INVENTORY_RESERVED, INVENTORY_FAILED,
    PROCESS_PAYMENT, PAYMENT_PROCESSED, PAYMENT_FAILED,
    CONFIRM_ORDER, COMPENSATE
}

@Configuration
@EnableStateMachine
public class OrderSagaStateMachine extends StateMachineConfigurerAdapter<OrderState, OrderEvent> {

    @Override
    public void configure(StateMachineStateConfigurer<OrderState, OrderEvent> states) throws Exception {
        states.withStates()
            .initial(ORDER_CREATED)
            .states(EnumSet.allOf(OrderState.class))
            .end(ORDER_CONFIRMED)
            .end(ORDER_CANCELLED);
    }

    @Override
    public void configure(StateMachineTransitionConfigurer<OrderState, OrderEvent> transitions) throws Exception {
        transitions
            // Happy path
            .withExternal()
                .source(ORDER_CREATED).target(INVENTORY_RESERVED)
                .event(INVENTORY_RESERVED)
                .action(reserveInventorySuccessAction())
            .and().withExternal()
                .source(INVENTORY_RESERVED).target(ORDER_CONFIRMED)
                .event(PAYMENT_PROCESSED)
                .action(confirmOrderAction())

            // Compensation paths
            .and().withExternal()
                .source(ORDER_CREATED).target(COMPENSATION_TRIGGERED)
                .event(INVENTORY_FAILED)
                .action(cancelOrderAction())           // no inventory to release
            .and().withExternal()
                .source(INVENTORY_RESERVED).target(COMPENSATION_TRIGGERED)
                .event(PAYMENT_FAILED)
                .action(releaseInventoryAction())     // compensation: release held inventory
            .and().withExternal()
                .source(COMPENSATION_TRIGGERED).target(ORDER_CANCELLED)
                .event(COMPENSATE)
                .action(notifyCustomerCancelledAction());
    }

    @Bean
    Action<OrderState, OrderEvent> reserveInventorySuccessAction() {
        return ctx -> {
            Order order = (Order) ctx.getMessageHeader("order");
            log.info("Inventory reserved for order {}", order.getId());
            // Move to next step: initiate payment
            ctx.getStateMachine().sendEvent(
                MessageBuilder.withPayload(PROCESS_PAYMENT)
                    .setHeader("order", order)
                    .build()
            );
        };
    }
}

// Saga orchestrator service
@Service
@RequiredArgsConstructor
public class OrderSagaService {
    private final StateMachineFactory<OrderState, OrderEvent> stateMachineFactory;
    private final OrderRepository orderRepo;

    @Transactional
    public Order startOrderSaga(CreateOrderRequest req) {
        Order order = orderRepo.save(Order.from(req));

        StateMachine<OrderState, OrderEvent> sm = stateMachineFactory.getStateMachine(
            order.getId().toString()
        );
        sm.start();

        // Trigger first step
        sm.sendEvent(MessageBuilder.withPayload(RESERVE_INVENTORY)
            .setHeader("order", order)
            .build());

        return order;
    }
}

@WorkflowInterface
public interface OrderWorkflow {
    @WorkflowMethod
    OrderResult processOrder(CreateOrderRequest req);
}

@WorkflowImpl
public class OrderWorkflowImpl implements OrderWorkflow {

    InventoryActivities inventoryActivities = Workflow.newActivityStub(
        InventoryActivities.class,
        ActivityOptions.newBuilder()
            .setStartToCloseTimeout(Duration.ofSeconds(30))
            .setRetryOptions(RetryOptions.newBuilder()
                .setMaximumAttempts(3)
                .setInitialInterval(Duration.ofSeconds(1))
                .build())
            .build()
    );

    @Override
    public OrderResult processOrder(CreateOrderRequest req) {
        // Each step is durable: if worker crashes mid-workflow, it resumes from here
        try {
            inventoryActivities.reserveInventory(req.getProductId(), req.getQuantity());
            paymentActivities.processPayment(req.getPaymentMethod(), req.getAmount());
            orderActivities.confirmOrder(req.getOrderId());
            return OrderResult.success();
        } catch (Exception e) {
            // Compensation is explicit, readable, and resumable
            inventoryActivities.releaseInventory(req.getProductId(), req.getQuantity());
            return OrderResult.failed(e.getMessage());
        }
    }
}

The Three Pillars

Metrics: aggregated numbers over time. "What is the p99 latency of /api/users?" Logs: discrete timestamped events. "What exactly happened when this request failed?" Traces: request flows across services. "Which service caused this 500ms latency?"

The power of observability comes from connecting all three with a shared trace ID.

// Custom metrics
@Component
@RequiredArgsConstructor
public class OrderMetrics {
    private final MeterRegistry registry;

    // Counter: total orders by status
    public void recordOrder(String status) {
        registry.counter("orders.processed", "status", status).increment();
    }

    // Timer: payment processing duration
    public <T> T timePaymentProcessing(Supplier<T> processPayment) {
        return registry.timer("payment.processing.duration",
            "gateway", "stripe"
        ).record(processPayment);
    }

    // Gauge: live queue depth
    @Autowired Queue<Order> pendingOrders;
    @PostConstruct
    public void registerGauge() {
        Gauge.builder("orders.pending.queue.size", pendingOrders, Queue::size)
            .description("Number of orders awaiting processing")
            .register(registry);
    }

    // Distribution summary: order value distribution
    public void recordOrderValue(BigDecimal amount) {
        registry.summary("order.value.dollars",
            "currency", "USD"
        ).record(amount.doubleValue());
    }
}
