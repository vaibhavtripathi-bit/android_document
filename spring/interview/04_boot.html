How does Spring Boot auto-configuration work? Trace from @SpringBootApplication to a configured DataSource.
AUTO_CONFIGURATION
▼

@SpringBootApplication Decomposed

@SpringBootApplication is a composed annotation: @SpringBootConfiguration (marks this as the primary configuration class) + @EnableAutoConfiguration (triggers the auto-config machinery) + @ComponentScan (scans the package tree for @Component, @Service, @Repository, etc.).

The Auto-Configuration Machinery

  1. @EnableAutoConfiguration imports AutoConfigurationImportSelector — a DeferredImportSelector implementation.
  2. DeferredImportSelector: runs AFTER all user @Configuration classes are processed. This is critical — it ensures user-defined beans override auto-configured ones.
  3. selectImports(): reads META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports (Boot 3) or the spring.factories key EnableAutoConfiguration (Boot 2). Gets the fully-qualified names of ~150 auto-configuration classes.
  4. Filtering: applies exclusions (@EnableAutoConfiguration(exclude=...) or spring.autoconfigure.exclude property) and AutoConfigurationImportFilter (fast pre-filtering for classes/beans that can be excluded without full evaluation).
  5. Each auto-config class is a @Configuration with condition annotations. Spring evaluates conditions; matching classes contribute bean definitions.

DataSource Auto-Configuration Trace

  1. HikariCP is on classpath (from spring-boot-starter-data-jpa → spring-boot-starter-jdbc → HikariCP).
  2. DataSourceAutoConfiguration: @ConditionalOnClass({DataSource.class, EmbeddedDatabaseType.class}) → true. @ConditionalOnMissingBean(DataSource.class) → no user-defined DataSource → true.
  3. @EnableConfigurationProperties(DataSourceProperties.class) → binds spring.datasource.* to DataSourceProperties.
  4. Inner PooledDataSourceConfiguration: @ConditionalOnNotSingleCandidate(DataSource.class) → true. Imports HikariDataSourceConfiguration.
  5. HikariDataSourceConfiguration: @ConditionalOnClass(HikariDataSource.class) → true. Creates HikariDataSource bean using pooled properties.
// Custom auto-configuration class (for your own library/starter)
@AutoConfiguration // Boot 3: replaces @Configuration + @AutoConfigureBefore/After
@ConditionalOnClass(MyServiceClient.class)
@ConditionalOnMissingBean(MyServiceClient.class) // allow user override
@EnableConfigurationProperties(MyServiceProperties.class)
@AutoConfigureAfter(DataSourceAutoConfiguration.class) // ordering
public class MyServiceAutoConfiguration {

    @Bean
    @ConditionalOnMissingBean
    public MyServiceClient myServiceClient(MyServiceProperties props) {
        return MyServiceClient.builder()
            .baseUrl(props.getBaseUrl())
            .apiKey(props.getApiKey())
            .timeout(props.getTimeout())
            .build();
    }

    // Conditional bean — only if specific property set
    @Bean
    @ConditionalOnProperty(prefix = "my.service", name = "retry.enabled",
                           havingValue = "true", matchIfMissing = true)
    public RetryInterceptor retryInterceptor(MyServiceProperties props) {
        return new RetryInterceptor(props.getRetry().getMaxAttempts());
    }
}

// com.example.MyServiceAutoConfiguration
⚠ GotchaIn Boot 3, use @AutoConfiguration instead of @Configuration for auto-config classes. This ensures proper ordering behavior and prevents the class from being picked up by @ComponentScan (which would cause it to run twice — once via scanning and once via auto-config). Auto-config classes should NEVER be in the main application package where component scanning runs.
✅ Best PracticeDebug auto-configuration with the /actuator/conditions endpoint or with --debug flag (prints ConditionEvaluationReport). This shows exactly which auto-configurations matched, which didn't, and WHY — invaluable for diagnosing "why isn't my DataSource configured?"
🎯 Google L6 AngleThe DeferredImportSelector mechanism is the key insight: auto-configurations are processed AFTER user configurations precisely so that @ConditionalOnMissingBean can correctly detect user overrides. If auto-configs ran first, user beans would always be "missing" when the condition is evaluated — the framework would override the user's explicit beans. This is the foundational design decision that makes the entire auto-configuration model work.

Custom Condition Implementation

// Custom condition: only activate if running in a Docker container
public class DockerCondition implements Condition {

    @Override
    public boolean matches(ConditionContext context, AnnotatedTypeMetadata metadata) {
        // ConditionContext provides:
        // - BeanDefinitionRegistry registry
        // - ConfigurableListableBeanFactory beanFactory
        // - Environment environment
        // - ResourceLoader resourceLoader
        // - ClassLoader classLoader

        // Check for Docker-specific file
        ResourceLoader loader = context.getResourceLoader();
        try {
            return loader.getResource("file:/.dockerenv").exists();
        } catch (Exception e) {
            return false;
        }
    }
}

// Or extend SpringBootCondition for better condition report messages
public class DockerCondition extends SpringBootCondition {

    @Override
    public ConditionOutcome getMatchOutcome(ConditionContext context,
                                            AnnotatedTypeMetadata metadata) {
        boolean inDocker = new File("/.dockerenv").exists();
        ConditionMessage message = ConditionMessage
            .forCondition("DockerCondition")
            .found(inDocker ? "Docker environment" : "no Docker environment")
            .items("/.dockerenv");
        return new ConditionOutcome(inDocker, message);
        // ConditionMessage appears in /actuator/conditions report — very useful!
    }
}

@Target({ElementType.TYPE, ElementType.METHOD})
@Retention(RetentionPolicy.RUNTIME)
@Documented

public @interface ConditionalOnDocker {}

// Usage
@Bean
@ConditionalOnDocker
public MetricsCollector dockerMetricsCollector() {
    return new DockerMetricsCollector();
}

// Complex condition: multiple conditions (AND logic)
@ConditionalOnClass(RedisConnectionFactory.class)
@ConditionalOnProperty("app.cache.redis.enabled")
@ConditionalOnMissingBean(CacheManager.class)
@Bean
public CacheManager redisCacheManager(RedisConnectionFactory factory) { ... }
// All three conditions must be true (AND semantics by default)
⚠ Gotcha@ConditionalOnBean is dangerous: it depends on the order beans are registered. If bean A's condition checks for bean B, but B is defined in a later @Configuration class, the condition may evaluate to false even though B will eventually exist. Auto-configuration ordering (@AutoConfigureAfter, @AutoConfigureBefore) must be set correctly. Spring's documentation explicitly warns against using @ConditionalOnBean on user-defined beans (only auto-config classes).
🎯 Google L6 AngleThe condition system is evaluated using ASM bytecode inspection for @ConditionalOnClass — the class is NOT actually loaded. This prevents ClassNotFoundException during condition evaluation when the class isn't present. This distinction matters when writing custom conditions that check for class presence: use ClassUtils.isPresent() or implement via @ConditionalOnClass rather than calling Class.forName() in your condition.

Walk through Spring Boot's complete startup sequence. What happens between main() and the app being ready?
STARTUP_PROCESS
▼

Complete Startup Sequence

  1. main() calls SpringApplication.run(MainClass.class, args) — creates a SpringApplication instance, then calls instance's run().
  2. SpringApplication construction: deduces WebApplicationType (SERVLET if DispatcherServlet on classpath, REACTIVE if DispatcherHandler, NONE otherwise). Loads SpringApplicationRunListeners and ApplicationContextInitializers from spring.factories.
  3. Listeners: starting() — ApplicationStartingEvent published. Very early — no context yet.
  4. Environment prepared: creates StandardServletEnvironment. Loads property sources: system properties, OS env vars, application.properties/.yaml, profile-specific properties. Command-line args added. ApplicationEnvironmentPreparedEvent published.
  5. Banner printed to stdout (or configured logger).
  6. ApplicationContext created: AnnotationConfigServletWebServerApplicationContext for servlet, AnnotationConfigReactiveWebServerApplicationContext for reactive.
  7. Context prepared: ApplicationContextInitializers applied. ApplicationContextInitializedEvent published.
  8. Bean definitions loaded: main class registered, @ComponentScan runs, auto-configurations loaded via DeferredImportSelector. ApplicationPreparedEvent published (bean defs loaded, not yet instantiated).
  9. Context refreshed (AbstractApplicationContext.refresh()): BeanFactoryPostProcessors run (processes @Configuration, @PropertySource), BeanPostProcessors registered, all singleton beans instantiated and wired. This is the longest step.
  10. Embedded server started: TomcatServletWebServerFactory.getWebServer() creates and starts embedded Tomcat. DispatcherServlet registered.
  11. ApplicationStartedEvent published — context refreshed, server started.
  12. AvailabilityChangeEvent(LivenessState.CORRECT) — app is alive.
  13. ApplicationRunner and CommandLineRunner beans called — post-startup initialization (cache warming, validation, etc.).
  14. ApplicationReadyEvent published — app fully ready.
  15. AvailabilityChangeEvent(ReadinessState.ACCEPTING_TRAFFIC) — Actuator readiness probe returns UP.

⚠ GotchaWith spring.main.lazy-initialization=true, beans are created on first use. This means configuration errors (e.g., bad database URL, missing environment variables) are not caught at startup — they surface on the first request that needs the bean. For production services, either use lazy init only for non-critical beans, or add a smoke test endpoint that eagerly exercises all beans.
🎯 Google L6 AngleStartup time directly impacts deployment velocity and Kubernetes pod readiness time. A service that takes 30s to start creates 30s of reduced capacity during rolling deployments. At scale (100 pods × 2 deployments/day × 30s = 1.67 hours of reduced capacity daily), startup optimization has real operational value. AOT compilation reducing startup from 8s to 3s saves 625 pod-seconds per deployment — multiply by deployment frequency.

How does the embedded server (Tomcat/Jetty/Undertow) work in Spring Boot? How is it configured?
EMBEDDED_SERVER
▼

How Embedding Works

Spring Boot programmatically creates and starts a web server during ApplicationContext.refresh(). The key abstraction is WebServerFactory. TomcatServletWebServerFactory creates an embedded Tomcat instance: configures the HTTP connector, creates the servlet context, deploys the DispatcherServlet, then calls tomcat.start().

No WAR file, no external server — the application IS the server. The main class is both the entry point and the application container.

Switching Servers

All three servers (Tomcat, Jetty, Undertow) are interchangeable. Exclude the default and add your preferred:

<!-- Switch to Undertow (lowest overhead, best for high-concurrency) -->
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-web</artifactId>
    <exclusions>
        <exclusion>
            <groupId>org.springframework.boot</groupId>
            <artifactId>spring-boot-starter-tomcat</artifactId>
        </exclusion>
    </exclusions>
</dependency>
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-undertow</artifactId>
</dependency>

// Programmatic server customization
@Bean
public WebServerFactoryCustomizer<TomcatServletWebServerFactory> tomcatCustomizer() {
    return factory -> {
        // Add custom connector (e.g., HTTP redirect to HTTPS)
        factory.addAdditionalTomcatConnectors(createHttpConnector());

        // Customize connector settings
        factory.addConnectorCustomizers(connector -> {
            connector.setMaxPostSize(10 * 1024 * 1024); // 10MB max POST body
            connector.setAttribute("relaxedQueryChars", "|{}[]");
        });

        // Customize context
        factory.addContextCustomizers(context -> {
            context.setSessionTimeout(30); // 30 minutes session timeout
        });

        // Adjust access log
        AccessLogValve accessLogValve = new AccessLogValve();
        accessLogValve.setPattern("%h %l %u %t \"%r\" %s %b %D");
        factory.addEngineValves(accessLogValve);
    };
}

private Connector createHttpConnector() {
    Connector connector = new Connector("org.apache.coyote.http11.Http11NioProtocol");
    connector.setScheme("http");
    connector.setPort(8080);
    connector.setSecure(false);
    // Redirect to HTTPS
    connector.setRedirectPort(8443);
    return connector;
}

// Full factory replacement — maximum control
@Bean
public TomcatServletWebServerFactory servletContainer() {
    TomcatServletWebServerFactory factory = new TomcatServletWebServerFactory();
    factory.setPort(8080);
    factory.setSessionTimeout(10, TimeUnit.MINUTES);
    factory.addErrorPages(new ErrorPage(HttpStatus.NOT_FOUND, "/404"));
    return factory;
}
⚠ Gotchaserver.tomcat.threads.max=200 is per-connector. If you add a second connector (HTTP + HTTPS), each has its own thread pool. Under high load, you can have up to 400 threads. Monitor actual thread usage with /actuator/metrics/tomcat.threads.busy before tuning — don't just increase max-threads without evidence of thread exhaustion.
🎯 Google L6 AngleFor Spring Boot 3.2+ on Java 21, consider spring.threads.virtual.enabled=true — this switches Tomcat to use virtual threads (Project Loom). With virtual threads, thread pool size is effectively unlimited (JVM manages). The parameter server.tomcat.threads.max becomes less critical. This changes the concurrency model from "thread-per-request bounded by pool" to "virtual-thread-per-request" — better for I/O-bound workloads.

Relaxed Binding Rules

Property app.myProperty can be provided as any of: app.myProperty (camelCase), app.my-property (kebab-case — recommended), app.my_property (underscore), APP_MY_PROPERTY (env var format). All bind to the same field. This is essential for Kubernetes/Docker where environment variables cannot contain dots or hyphens.

// Complex @ConfigurationProperties with validation
@ConfigurationProperties(prefix = "app.database")
@Validated
public class DatabaseProperties {

    @NotBlank
    private String url;

    @NotBlank
    private String username;

    private String password; // can be blank (external secret injection)

    @Min(1) @Max(100)
    private int maxPoolSize = 10; // default value

    @DurationUnit(ChronoUnit.SECONDS) // parses "30s", "PT30S", "30"
    private Duration connectionTimeout = Duration.ofSeconds(30);

    @DataSizeUnit(DataUnit.MEGABYTES) // parses "10MB", "10485760"
    private DataSize maxPacketSize = DataSize.ofMegabytes(16);

    private Pool pool = new Pool(); // nested object

    public static class Pool {
        private int minIdle = 2;
        private int maxActive = 20;
        private boolean testOnBorrow = true;
        // getters/setters
    }

    // Map property: app.database.driver-options.cachePrepStmts=true
    private Map<String, String> driverOptions = new HashMap<>();

    // List property: app.database.read-replicas[0]=host1, [1]=host2
    private List<String> readReplicas = new ArrayList<>();

    // getters/setters required for mutable binding
}

⚠ GotchaRecord-based @ConfigurationProperties uses constructor binding — Spring Boot 3 uses the canonical constructor. If you have a record with a compact constructor that does validation, the validation runs during property binding — this is correct but can be confusing (validation error during context startup, not request handling). Also: @Validated on a record requires placing validation annotations on constructor parameters, not record components.
✅ Best PracticeCreate a dedicated properties class per feature/module (not one god-class for all app properties). This makes it easy to inject only what a service needs, and makes unit testing simple — just construct the properties POJO with test values, no Spring context required.
🎯 Google L6 AngleFor secret management (passwords, API keys), never store secrets in application.properties in source control. Use: environment variable injection (Kubernetes secrets → env vars → relaxed binding), or Spring Cloud Vault/AWS Secrets Manager integration. The @ConfigurationProperties class just needs the field — the VALUE comes from the secure external source. The field in the properties class acts as the binding contract, not the source of truth for secret values.

// Custom HealthIndicator
@Component
public class ExternalServiceHealthIndicator implements HealthIndicator {

    @Autowired ExternalServiceClient client;

    @Override
    public Health health() {
        try {
            boolean healthy = client.ping();
            return healthy
                ? Health.up()
                    .withDetail("url", client.getBaseUrl())
                    .withDetail("responseTimeMs", client.getLastPingMs())
                    .build()
                : Health.down()
                    .withDetail("reason", "Service not responding")
                    .build();
        } catch (Exception e) {
            return Health.down(e)
                .withDetail("url", client.getBaseUrl())
                .build();
        }
    }
}

// Securing Actuator with Spring Security
@Bean
public SecurityFilterChain actuatorSecurity(HttpSecurity http) throws Exception {
    http.securityMatcher("/actuator/**")
        .authorizeHttpRequests(auth -> auth
            .requestMatchers("/actuator/health/**").permitAll()  // K8s probes
            .requestMatchers("/actuator/info").permitAll()
            .requestMatchers("/actuator/prometheus").hasIpAddress("10.0.0.0/8") // internal only
            .anyRequest().hasRole("ACTUATOR_ADMIN"))
        .httpBasic(Customizer.withDefaults());
    return http.build();
}

⚠ GotchaThe /actuator/heapdump endpoint downloads the full JVM heap as a binary file. This can be hundreds of MB to GB and contains all in-memory data (including secrets, PII). NEVER expose this publicly. Also: /actuator/env shows all property values — despite sanitization, it can expose configuration that you'd rather keep private. Restrict to internal network or admin roles only.
✅ Best PracticeEnable Kubernetes health probes via management.endpoint.health.probes.enabled=true. This creates /actuator/health/liveness (LivenessState) and /actuator/health/readiness (ReadinessState). Configure K8s livenessProbe and readinessProbe to these endpoints. Spring Boot manages these states correctly during startup and graceful shutdown.
🎯 Google L6 AngleThe /actuator/conditions endpoint is invaluable during debugging: it shows a full report of which auto-configurations matched (and why), which didn't (and why not), and which were excluded. It's the first place to look when "DataSource auto-configuration didn't run" or "Security auto-config isn't applying". This endpoint alone can save hours of debugging. Know it exists and how to use it.

Explain Spring Profiles. How do profile-specific properties and conditional beans work?
PROFILES
▼

Profile Basics

@Profile("production") on a bean: the bean is only registered in the application context when the "production" profile is active. Inactive profile beans are completely absent from the context — they're not created, not wired, not visible to other beans.

Activate profiles: spring.profiles.active=production,feature-x (property), SPRING_PROFILES_ACTIVE=production (env var), --spring.profiles.active=production (command line). Multiple profiles can be active simultaneously.

Profile-Specific Properties

application-{profile}.properties: loaded when that profile is active. Properties in profile-specific file OVERRIDE those in application.properties. Same for YAML — use --- document separators with spring.config.activate.on-profile.

# application.properties (base config — all environments)
spring.datasource.pool-size=5
app.feature.analytics=false
logging.level.root=INFO

# application-production.properties (production overrides)
spring.datasource.pool-size=50
server.tomcat.threads.max=400
logging.level.root=WARN
app.feature.analytics=true

# application-local.properties (developer overrides)
spring.datasource.url=jdbc:h2:mem:devdb
spring.h2.console.enabled=true
logging.level.com.example=DEBUG

What application events does Spring Boot publish? How can you use them?
APPLICATION_EVENTS
▼

Spring Boot Application Events (in order)

  1. ApplicationStartingEvent — fired at the very start of a run, before any processing except registration of listeners. Context does not exist yet. Use: early logging setup.
  2. ApplicationEnvironmentPreparedEvent — Environment is ready, context not yet created. Use: environment validation, early configuration checks.
  3. ApplicationContextInitializedEvent — Context created, ApplicationContextInitializers applied, before bean definitions loaded.
  4. ApplicationPreparedEvent — Bean definitions loaded, context not yet refreshed. @Autowired won't work yet.
  5. ContextRefreshedEvent (from Spring core) — Context fully refreshed, all beans created. Standard Spring event.
  6. ApplicationStartedEvent — Context refreshed and started. ApplicationRunners not yet called. Use: post-startup that needs beans but before Runners.
  7. AvailabilityChangeEvent (LivenessState.CORRECT) — App has declared it is live (internal state healthy).
  8. ApplicationReadyEvent — All runners called, application is ready to serve requests. The "fully started" event.
  9. AvailabilityChangeEvent (ReadinessState.ACCEPTING_TRAFFIC) — App ready for load balancer traffic. Readiness probe returns UP.
  10. ApplicationFailedEvent — Startup failed. Use for alerting.
⚠ GotchaEvents fired before context creation (ApplicationStartingEvent, ApplicationEnvironmentPreparedEvent) cannot be received by beans registered in the context. You must register listeners programmatically: SpringApplication.addListeners(new MyEarlyListener()) or declare them in spring.factories/spring.listeners. Using @EventListener only works for events fired after context refresh.
// @EventListener — simplest approach for post-refresh events
@Component
public class ApplicationEventListeners {

    // Cache warmup after full startup
    @EventListener(ApplicationReadyEvent.class)
    public void onReady(ApplicationReadyEvent event) {
        log.info("Application ready in {}ms", event.getTimeTaken().toMillis());
        cacheService.warmUp();
    }

    // React to readiness state changes
    @EventListener
    public void onAvailabilityChange(AvailabilityChangeEvent<ReadinessState> event) {
        log.info("Readiness state changed to: {}", event.getState());
        if (event.getState() == ReadinessState.REFUSING_TRAFFIC) {
            // Take action — circuit breaker tripped, maintenance mode, etc.
        }
    }

    // Alert on startup failure
    @EventListener(ApplicationFailedEvent.class)
    public void onFailure(ApplicationFailedEvent event) {
        alertingService.sendAlert("Spring Boot startup failed: "
            + event.getException().getMessage());
    }
}

// ApplicationListener — for pre-context events or when @EventListener isn't available
public class EarlyConfigValidator implements ApplicationListener<ApplicationEnvironmentPreparedEvent> {
    @Override
    public void onApplicationEvent(ApplicationEnvironmentPreparedEvent event) {
        Environment env = event.getEnvironment();
        String requiredVar = env.getProperty("REQUIRED_SECRET");
        if (requiredVar == null || requiredVar.isBlank()) {
            throw new IllegalStateException("Required environment variable REQUIRED_SECRET is not set");
        }
    }
}
// Register in SpringApplication:
SpringApplication app = new SpringApplication(Application.class);
app.addListeners(new EarlyConfigValidator());
app.run(args);

// Programmatic readiness state management
@Autowired ApplicationContext context;

public void signalNotReady(String reason) {
    // Take pod out of load balancer rotation without killing it
    AvailabilityChangeEvent.publish(context, ReadinessState.REFUSING_TRAFFIC);
    log.warn("Service refusing traffic: {}", reason);
}

public void signalReady() {
    AvailabilityChangeEvent.publish(context, ReadinessState.ACCEPTING_TRAFFIC);
}
✅ Best PracticeUse ApplicationReadyEvent for post-startup initialization (cache warming, external service validation), not @PostConstruct or CommandLineRunner. ApplicationReadyEvent fires after the server is ready to accept traffic, making it the correct point for "app is ready" logic. @PostConstruct fires during bean initialization — before the HTTP server is even started.
🎯 Google L6 AngleThe separation between LivenessState and ReadinessState maps directly to Kubernetes probe semantics. Liveness = "should this pod be killed?" (restart if unhealthy). Readiness = "should this pod receive traffic?" (remove from service endpoints if not ready). A pod can be live but not ready (warming up cache, circuit breaker open). Programmatic readiness control via AvailabilityChangeEvent.publish() lets you implement maintenance mode without pod restarts.

What does Spring Boot DevTools provide? How does automatic restart work?
DEVTOOLS
▼

DevTools Features

  • Automatic restart: detects classpath changes and restarts the Spring context. Much faster than full restart.
  • LiveReload: embedded LiveReload server triggers browser refresh when static resources or templates change. Requires browser extension.
  • Property overrides: sets development-friendly defaults: caches disabled (Thymeleaf, Freemarker), debug logging for web layer, H2 console enabled.
  • Remote DevTools: restart and LiveReload over HTTP for remote development (never use in production).
  • Global settings: ~/.spring-boot-devtools.properties for developer machine defaults shared across projects.

How Automatic Restart Works (The Two ClassLoader Trick)

DevTools uses two ClassLoaders:

  1. Base ClassLoader: loads JAR dependencies (Spring JARs, third-party libs). These change rarely. Loaded once at startup.
  2. Restart ClassLoader: loads your application classes (everything in target/classes or build/classes). This is the ClassLoader that gets thrown away and recreated on change.

On file change: only the Restart ClassLoader is discarded and recreated. The Base ClassLoader stays. Since your app classes are relatively small compared to all JARs, restart is 5-10x faster than a full JVM restart.

⚠ GotchaThe two-ClassLoader architecture can cause ClassCastException when objects from the Base ClassLoader interact with objects from the Restart ClassLoader. This typically manifests as "cannot cast X to X" — the same class loaded by two different ClassLoaders is not the same type in Java. If you see this, check if the problematic class belongs to a library (base CL) that's being instantiated or cast to something in your code (restart CL).
🎯 Google L6 AngleThe base/restart ClassLoader split is a microcosm of a general systems principle: separate the stable from the volatile. Load the stable part once (dependencies), reload only the volatile part (your code). This principle appears at multiple scales: container layers (stable OS → stable JRE → changing app), Webpack code splitting (vendor bundle vs app bundle), database connection pooling (persistent connections vs per-request usage). Understanding the principle lets you apply it to novel systems design problems.

What is Spring AOT (Ahead-of-Time) processing? How does it differ from JIT?
AOT
▼

JIT vs AOT

JIT (standard JVM): Spring evaluates @ConditionalOn* conditions at runtime, creates CGLIB proxies on the fly, scans annotations via reflection on every startup. Fast developer iteration (no build step), but slow startup — every startup re-does the same work.

AOT (Spring Boot 3+): processes the application context at BUILD TIME. Generates Java source code representing bean definitions, proxy classes, and resource hints. This generated code is compiled and included in the final artifact. At runtime, Spring uses the pre-generated definitions instead of re-evaluating conditions and re-creating proxies.

What AOT Generates

  • BeanDefinitionRegistrar classes: pre-computed bean definitions — no condition evaluation at runtime.
  • Proxy classes: CGLIB proxy source code generated at build time, not runtime.
  • Reflection hints: metadata for GraalVM's native-image compiler about which classes/methods/fields need reflection access.
  • Resource hints: which classpath resources will be accessed at runtime.

AOT Build Pipeline

# Maven: Spring Boot 3+ AOT processing
mvn spring-boot:process-aot
# Output: target/spring-aot/main/sources/ — generated Java source files
#         target/spring-aot/main/resources/ — generated reflect-config.json, etc.

# Gradle:
./gradlew processAot

# The fat JAR built after processAot includes the AOT-generated sources
mvn spring-boot:repackage  # or just: mvn package
// Example of what AOT generates for a simple bean:
// Your code:
@Service
public class UserService {
    @Autowired UserRepository repo;
}

// AOT-generated (conceptually — actual output is more complex):
public class UserServiceBeanDefinitionRegistrar
        implements BeanDefinitionRegistrar {

    @Override
    public void register(BeanDefinitionRegistry registry) {
        BeanDefinitionBuilder builder = BeanDefinitionBuilder
            .rootBeanDefinition(UserService.class);
        // No condition evaluation needed — already resolved at build time
        // No annotation scanning — already done at build time
        builder.addPropertyReference("repo", "userRepository");
        registry.registerBeanDefinition("userService", builder.getBeanDefinition());
    }
}

// AOT Limitations and Constraints
// 1. Dynamic bean registration at runtime not supported:
//    beanFactory.registerBeanDefinition() at runtime is not compatible with AOT
// 2. @ConditionalOnProperty based on RUNTIME env vars: supported
//    (AOT detects these and defers evaluation to runtime)
// 3. @ConditionalOnClass based on CLASSPATH: evaluated at build time, baked in
// 4. ClassLoader manipulation, dynamic proxies (not via Spring): require hints

// Providing AOT hints for custom code
@Component
@ImportRuntimeHints(MyRuntimeHintsRegistrar.class)
public class MyComponent { ... }

public class MyRuntimeHintsRegistrar implements RuntimeHintsRegistrar {
    @Override
    public void registerHints(RuntimeHints hints, ClassLoader classLoader) {
        // Register reflection access
        hints.reflection().registerType(MyDynamicClass.class,
            MemberCategory.INVOKE_PUBLIC_METHODS);
        // Register resource access
        hints.resources().registerPattern("templates/*.html");
        // Register serialization
        hints.serialization().registerType(MySerializableClass.class);
    }
}
⚠ GotchaAOT processes the application context with the current build environment. If your beans use @ConditionalOnProperty for feature flags that vary per environment, AOT correctly detects these as runtime-conditional and defers them. But if you have beans that do runtime classpath scanning, dynamic class loading, or JVM-specific behavior, they may not work under AOT. Test with -Dspring.aot.enabled=true without going full native to validate.
🎯 Google L6 AngleAOT's primary value is as a prerequisite for GraalVM native compilation — GraalVM's closed-world analysis requires knowing all reflection, proxy, and resource access at compile time. Spring AOT generates the required metadata automatically for Spring-managed code. The secondary benefit (20-40% JVM startup reduction) is a useful side effect. When evaluating AOT adoption, the question is: "Are we targeting native compilation?" If yes, AOT is required. If not, evaluate the build complexity cost vs startup savings.

What is GraalVM native image? What are the trade-offs vs standard JVM for Spring Boot apps?
GRAALVM_NATIVE
▼

GraalVM Native Image

Compiles a Java application to a standalone native binary using AOT compilation. The resulting binary contains the application code, required JDK classes, statically linked native libraries, and a lightweight garbage collector. No JVM needed at runtime.

Closed-world assumption: everything that can be executed must be reachable at build time. Dynamic class loading, reflection, JNI, proxy generation, and serialization must be declared via configuration metadata (which Spring AOT generates automatically for Spring code).

Latency (tail)Consistent (no JIT compilation pauses)Warm-up latency spikes
Build time3-15 minutes10-60 seconds
DebuggabilityLimited (no dynamic class loading, restricted JDWP)Full
Dynamic featuresRestricted (reflection requires hints)Unrestricted
ProfilingLimited toolingJFR, async-profiler, heap dumps

🎯 Google L6 AngleThe transformative use case for native images is serverless at scale. AWS Lambda cold start latency with JVM: 3-10s. With GraalVM native: 50-200ms. Native images shift Lambda from "interesting experiment" to "production-viable architecture".

Explain Spring Boot's externalized configuration hierarchy. What is the priority order?
EXTERNALIZED_CONFIG
▼

Priority Order (Highest to Lowest)

  1. DevTools global settings (~/.spring-boot-devtools.properties) — developer machine only
  2. @TestPropertySource annotations in tests
  3. @SpringBootTest(properties=...) inline properties
  4. Command-line arguments (--server.port=9090)
  5. SPRING_APPLICATION_JSON (JSON as env var or system property)
  6. ServletConfig / ServletContext init params
  7. JNDI (java:comp/env)
  8. Java System properties (-Dserver.port=9090)
  9. OS environment variables (SERVER_PORT=9090) — 12-factor standard
  10. Profile-specific files OUTSIDE JAR (./application-prod.properties)
  11. Application files OUTSIDE JAR (./application.properties)
  12. Profile-specific files INSIDE JAR (classpath:application-prod.properties)
  13. Application files INSIDE JAR (classpath:application.properties)
  14. @PropertySource annotations on @Configuration classes
  15. Default properties (SpringApplication.setDefaultProperties())

Config File Locations

Spring Boot 2.4+ supports spring.config.import for importing additional config files (including optional imports): spring.config.import=optional:file:/etc/app/config.properties,optional:configserver:

Config data locations searched in order: classpath root, classpath /config, current directory, current directory /config, subdirectories of /config.

Walk through creating a custom Spring Boot starter with auto-configuration.
CUSTOM_AUTOCONFIGURATION
▼

Starter Project Structure

A proper Spring Boot starter consists of two Maven modules (by convention):

  • acme-spring-boot-autoconfigure: contains the @AutoConfiguration class, @ConfigurationProperties, and the AutoConfiguration.imports file. Has optional dependency on the library being configured.
  • acme-spring-boot-starter: thin POM that depends on autoconfigure + the library itself. Users depend on this.
// ===== acme-spring-boot-autoconfigure module =====

// 1. Properties class
@ConfigurationProperties(prefix = "acme.client")
public class AcmeClientProperties {

    /** Base URL of the Acme API */
    @NotBlank
    private String baseUrl;

    /** API key for authentication */
    @NotBlank
    private String apiKey;

    private int connectTimeoutMs = 5000;
    private int readTimeoutMs = 30000;

    @NestedConfigurationProperty
    private RetryProperties retry = new RetryProperties();

    public static class RetryProperties {
        private int maxAttempts = 3;
        private long backoffMs = 500;
        // getters/setters
    }
    // getters/setters
}

// 2. Auto-configuration class
@AutoConfiguration                                              // Spring Boot 3
@ConditionalOnClass(AcmeClient.class)                          // library on classpath
@EnableConfigurationProperties(AcmeClientProperties.class)     // bind properties
@ConditionalOnProperty(prefix = "acme.client",
                       name = "enabled",
                       havingValue = "true",
                       matchIfMissing = true)                   // on by default
public class AcmeClientAutoConfiguration {

    @Bean
    @ConditionalOnMissingBean(AcmeClient.class)                // allow user override
    public AcmeClient acmeClient(AcmeClientProperties props) {
        return AcmeClient.builder()
            .baseUrl(props.getBaseUrl())
            .apiKey(props.getApiKey())
            .connectTimeout(Duration.ofMillis(props.getConnectTimeoutMs()))
            .readTimeout(Duration.ofMillis(props.getReadTimeoutMs()))
            .build();
    }

    @Bean
    @ConditionalOnMissingBean
    @ConditionalOnBean(AcmeClient.class)
    public AcmeHealthIndicator acmeHealthIndicator(AcmeClient client) {
        return new AcmeHealthIndicator(client);
    }

    @Bean
    @ConditionalOnMissingBean
    @ConditionalOnProperty("acme.client.retry.max-attempts")
    public RetryingAcmeClient retryingAcmeClient(AcmeClient delegate,
                                                  AcmeClientProperties props) {
        return new RetryingAcmeClient(delegate,
            props.getRetry().getMaxAttempts(),
            props.getRetry().getBackoffMs());
    }
}

                                "acme.client.api-key=test-key")
            .withUserConfiguration(CustomAcmeClientConfig.class) // user provides their own bean
            .run(context -> {
                assertThat(context).hasSingleBean(AcmeClient.class);
                // verify it's the user's bean, not the auto-configured one
                assertThat(context.getBean(AcmeClient.class))

            });
    }

    void disabledViaProperty_noBeansCreated() {

    }

⚠ GotchaThe auto-configure module should have optional=true dependency on the library being wrapped. This way, if a user depends on the starter but not the library, they get a compile error (missing library), not a runtime condition-evaluation failure. The starter module adds the non-optional dependency on the library, pulling it in automatically.

How does Spring Boot graceful shutdown work? How do you ensure in-flight requests complete?
GRACEFUL_SHUTDOWN
▼

What Graceful Shutdown Does

Without graceful shutdown: SIGTERM → JVM exits immediately → in-flight requests get TCP connection reset → client sees connection error or 5xx. With graceful shutdown: SIGTERM → Tomcat stops accepting NEW requests (returns 503 for new connections) → waits for in-flight requests to complete → then JVM exits.

Configuration

# Enable graceful shutdown
server.shutdown=graceful

# How long to wait for in-flight requests to complete (default: 30s)
spring.lifecycle.timeout-per-shutdown-phase=30s

# Kubernetes deployment snippet
spec:
  containers:
  - name: myapp
    lifecycle:
      preStop:
        exec:
          command: ["/bin/sh", "-c", "sleep 15"]  # let LB drain before SIGTERM
    terminationGracePeriodSeconds: 60  # must be > preStop + shutdown timeout
