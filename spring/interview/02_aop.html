What is AOP and what problem does it solve? Define all AOP terminology.
FUNDAMENTALS
▼

The Problem: Cross-Cutting Concerns

Cross-cutting concerns are behaviors that span multiple application layers and classes — they "cut across" the regular modularity of object-oriented code. Examples: logging method entry/exit, security checks, transaction management, caching, performance metrics, retry logic, input sanitization. If implemented inline, they create:

  • Code duplication — same logging boilerplate in hundreds of methods
  • Entanglement — business logic buried inside security/transaction boilerplate
  • Change fragility — updating logging format requires touching every class

AOP extracts these concerns into separate, reusable modules (aspects) that are woven into target code at defined points.

// Minimal working aspect — logs execution of all service methods
@Aspect
@Component
public class ServiceLoggingAspect {

    private static final Logger log = LoggerFactory.getLogger(ServiceLoggingAspect.class);

    // Pointcut: any method in any class in com.example.service package
    @Pointcut("execution(* com.example.service.*.*(..))")
    public void serviceLayer() {}

    // Around advice: most powerful — full control
    @Around("serviceLayer()")
    public Object logExecution(ProceedingJoinPoint pjp) throws Throwable {
        String method = pjp.getSignature().toShortString();
        log.info("--> Entering: {}", method);
        long start = System.nanoTime();
        try {
            Object result = pjp.proceed(); // call the real method
            log.info("<-- Exited: {} in {}ms", method, (System.nanoTime() - start) / 1_000_000);
            return result;
        } catch (Throwable t) {
            log.error("<-- Exception in {}: {}", method, t.getMessage());
            throw t;
        }
    }
}
✅ Best PracticeEnable AspectJ auto proxy with @EnableAspectJAutoProxy (Spring Boot auto-configures this). Always declare @Aspect beans as @Component so they are detected by component scan — forgetting @Component is a very common gotcha where the aspect is parsed but never applied.
🎯 Google L6 AngleAOP is the mechanism that lets Google-scale systems enforce policies (security, rate limiting, auditing) uniformly without modifying every service method. A custom @RateLimited annotation backed by an @Around aspect applies rate limiting to any method — adding the annotation is the only change needed. This is the open/closed principle: open for extension (new rate limits), closed for modification (no changes to existing services).

What is the difference between Spring AOP and AspectJ? When must you use AspectJ?
SPRING_VS_ASPECTJ
▼

Spring AOP

  • Runtime proxy-based — creates CGLIB subclasses or JDK dynamic proxies at startup
  • Spring-managed beans only — objects created with new are invisible to Spring AOP
  • METHOD EXECUTION join points only — no field access, no constructors, no static methods
  • Uses AspectJ annotation syntax (@Aspect, @Around, etc.) but different execution engine
  • Zero configuration overhead for Spring apps — auto-configured by Spring Boot
  • Simpler debugging — proxy objects visible in stack traces

Full AspectJ

  • Bytecode weaving — compile-time (ajc compiler) or load-time (Java agent)
  • ANY Java object — including new Foo(), not just Spring beans
  • ALL join points: method call, method execution, constructor call/execution, field get/set, exception handler, static initializer, object initialization
  • Full AspectJ language: inter-type declarations, declare error/warning, pertarget instantiation models
  • Zero runtime overhead — woven directly into bytecode, no proxy dispatch
CapabilitySpring AOPFull AspectJ
Non-Spring objectsNoYes
Constructor interceptionNoYes
Field access interceptionNoYes
Static method interceptionNoYes
Self-invocationNo (proxy bypassed)Yes (woven into bytecode)
Runtime overhead~50-200ns per call~0ns (inlined)
Setup complexityLow (auto-configured)High (ajc or LTW agent)

When You MUST Use AspectJ

  1. Domain entity interception: JPA entities are not Spring beans — AspectJ LTW lets you apply @Auditable to entities created with new
  2. Field-access join points: auditing field changes on domain objects (before/after field set)
  3. Constructor join points: enforcing invariants at object construction time
  4. Fixing self-invocation: this.transactionalMethod() bypasses Spring proxy — AspectJ LTW fixes it without code changes
  5. Hot paths: when proxy dispatch overhead matters (sub-microsecond operations, HFT-style scenarios)

AspectJ LTW is what enables applying cross-cutting concerns (audit trails, domain events) to domain entities without polluting them with framework code. This is the architectural purity that separates senior from L6: knowing when the proxy model's limitations force you to step up to full AspectJ.

Explain JDK Dynamic Proxy vs CGLIB. How does Spring choose? What are the limitations?
PROXY_MECHANISM
▼

JDK Dynamic Proxy

Uses java.lang.reflect.Proxy. Creates a proxy object that implements the same interfaces as the target. The proxy implements InvocationHandler — every method call is routed through invoke(Object proxy, Method method, Object[] args). The handler applies advice, then calls the real target via reflection.

Requirement: target MUST implement at least one interface. Only interface methods are intercepted — any method not in an interface is not proxied. Slightly slower due to reflection-based dispatch.

CGLIB (now ByteBuddy in Spring 6)

Generates a subclass of the target class at runtime using bytecode generation. The subclass overrides non-final, non-private methods to intercept them. Proxy behaves like the target type (no interface needed).

Cannot proxy: final classes (can't subclass), final methods (can't override), private methods (not accessible in subclass), static methods.

Spring's Selection Logic

// Spring Boot default (since 2.0): proxyTargetClass=true → CGLIB for almost everything
@SpringBootApplication
// → @EnableAutoConfiguration → many @EnableXxx annotations
// → each sets proxyTargetClass=true
// → AbstractAutoProxyCreator uses CGLIB unless explicitly overridden

// Force JDK proxy (target must implement interface):
@EnableAspectJAutoProxy(proxyTargetClass = false)
@Configuration
public class AopConfig { }

// Check what type of proxy you got:
import org.springframework.aop.support.AopUtils;
AopUtils.isJdkDynamicProxy(userService);  // true = JDK proxy
AopUtils.isCglibProxy(userService);       // true = CGLIB proxy
AopUtils.isAopProxy(userService);         // true = either proxy type

// Unwrap to target:
Object target = AopTestUtils.getTargetObject(userService);

Common Proxy Limitations and Fixes

// Problem 1: final class — CGLIB cannot proxy
@Service
public final class OrderService { // BeanCreationException at startup!
    @Transactional
    public void placeOrder(Order o) { ... }
}
// Fix: remove final

// Problem 2: @Transactional on a class that only has interface injection
public interface PaymentService { void pay(Order o); }

@Service
public class StripePaymentService implements PaymentService {
    @Transactional
    public void pay(Order o) { ... }

    public void refund(Order o) { // NOT in interface — JDK proxy won't intercept!
        ...
    }
}
// If Spring uses JDK proxy: calling refund() bypasses @Transactional
// Fix: use CGLIB (proxyTargetClass=true, the Boot default) OR add refund() to interface

// Problem 3: casting proxy to concrete class fails with JDK proxy
PaymentService proxy = ctx.getBean(PaymentService.class);
StripePaymentService concrete = (StripePaymentService) proxy; // ClassCastException with JDK proxy
// Fix: inject by interface, not concrete class

// Problem 4: Kotlin data classes / Kotlin final-by-default
// Kotlin classes are final by default — Spring CGLIB fails
// Fix 1: open modifier on class and methods
// Fix 2: apply kotlin-allopen compiler plugin (recommended)
// Fix 3: @Configuration(proxyBeanMethods=false) for config classes
⚠ GotchaSpring Boot 2.x set proxyTargetClass=true as the default, which means CGLIB is used even when targets implement interfaces. This broke code that expected JDK proxies (e.g., code that relies on instanceof checks against interface types). The change was intentional to fix injection-by-concrete-class issues, but can surprise teams migrating from Spring Boot 1.x.
✅ Best PracticeAlways inject by interface type in production code — @Autowired PaymentService service, not @Autowired StripePaymentService service. This works with both JDK and CGLIB proxies and improves testability.
🎯 Google L6 AngleThe JDK vs CGLIB choice matters in three scenarios: (1) Kotlin where classes are final by default — requires kotlin-allopen plugin, (2) framework code that needs to support both proxy types (check with AopUtils), (3) performance profiling — CGLIB method dispatch is faster (direct subclass call vs reflection). Spring 6 replaced CGLIB with ByteBuddy for better Java 17+ module system compatibility.

Explain the self-invocation problem. Why doesn't @Transactional work when calling a method on 'this'?
SELF_INVOCATION
▼

How Spring Proxy Interception Works

When Spring injects a bean, it injects the PROXY, not the target object. The proxy wraps the target. External callers call the proxy → proxy applies advice (starts transaction, checks security, etc.) → proxy delegates to target. The AOP magic lives in the proxy layer.

The Self-Invocation Bug

// The proxy is "outside" the object. Inside the object, 'this' is the RAW target.
@Service
public class OrderService {

    public void placeOrder(Order o) {
        validateOrder(o);
        // THIS IS THE BUG: calling via 'this' — bypasses the proxy entirely
        processPayment(o);  // equivalent to: this.processPayment(o)
        // No transaction started! processPayment runs without @Transactional!
    }

    @Transactional
    public void processPayment(Order o) {
        // @Transactional has no effect when called from placeOrder above
        paymentRepo.save(new Payment(o));
        inventoryRepo.deduct(o.getItems());
        // If inventoryRepo.deduct() throws — payment is NOT rolled back (no tx!)
    }
}

// The injected proxy:
OrderService proxy = ctx.getBean(OrderService.class);
proxy.processPayment(order);  // works! goes through proxy, starts transaction
proxy.placeOrder(order);       // broken! placeOrder calls this.processPayment, bypasses proxy

Solution 1 — AopContext.currentProxy() (Ugly but simple)

@EnableAspectJAutoProxy(exposeProxy = true)  // must enable this!
@SpringBootApplication
public class Application { }

@Service
public class OrderService {
    public void placeOrder(Order o) {
        ((OrderService) AopContext.currentProxy()).processPayment(o); // goes through proxy
    }
    @Transactional
    public void processPayment(Order o) { ... }
}

Solution 2 — Self-Injection (Cleaner)

@Service
public class OrderService {
    @Lazy  // avoid circular dependency issue during context startup
    @Autowired
    private OrderService self;  // Spring injects the PROXY of this bean

    public void placeOrder(Order o) {
        self.processPayment(o);  // goes through proxy — transaction applied!
    }

    @Transactional
    public void processPayment(Order o) { ... }
}

Solution 3 — Refactor to Separate Bean (CORRECT DESIGN)

// Extract processPayment to its own service — clean, testable, no Spring coupling
@Service
public class PaymentService {
    @Transactional
    public void processPayment(Order o) {
        paymentRepo.save(new Payment(o));
        inventoryRepo.deduct(o.getItems());
    }
}

@Service
public class OrderService {
    private final PaymentService paymentService;

    public OrderService(PaymentService paymentService) {
        this.paymentService = paymentService;
    }

    public void placeOrder(Order o) {
        validateOrder(o);
        paymentService.processPayment(o); // calls through PaymentService's proxy — WORKS!
    }
}
// Bonus: OrderService is now easier to test (inject mock PaymentService)

Solution 4 — AspectJ LTW (Nuclear option — fixes all self-invocation)

# application.properties
spring.aop.proxy-target-class=true  # not enough for self-invocation

# LTW: -javaagent:aspectjweaver.jar
# Advice woven into bytecode — 'this.processPayment()' IS intercepted
# No proxy needed — works transparently
⚠ GotchaThis is one of the most common transaction bugs in production. Symptoms: method marked @Transactional but no transaction started (no rollback on exception, no commit), or multiple separate transactions instead of one. Always check if the method is called via a proxy or directly.
✅ Best PracticeThe correct answer is always Solution 3 — refactor to a separate bean. It improves cohesion (OrderService does orders, PaymentService does payments), makes each class independently testable, and doesn't require any Spring-specific hacks. Self-invocation problems are design problems in disguise.
🎯 Google L6 AngleAt L6, knowing the problem is table stakes. The L6 insight is that self-invocation failures reveal Single Responsibility Principle violations: if placeOrder() needs to call processPayment() transactionally, they probably belong in different domain objects. Drawing the class boundary correctly eliminates the problem entirely without framework workarounds.

Explain all five advice types in Spring AOP with production use cases and code.
ADVICE_TYPES
▼

@Before — Executes Before the Join Point

Cannot prevent method execution (only by throwing an exception). Has access to JoinPoint but not return value. Use for: pre-condition checks, security enforcement, logging method entry, input validation.

@Aspect @Component
public class SecurityAspect {
    @Before("@annotation(com.example.RequiresRole) && @annotation(role)")
    public void checkRole(JoinPoint jp, RequiresRole role) {
        String currentUserRole = SecurityContextHolder.getContext().getAuthentication().getAuthorities()
            .stream().findFirst().map(Object::toString).orElse("");
        if (!currentUserRole.equals(role.value())) {
            throw new AccessDeniedException("Required role: " + role.value());
        }
    }
}

@After — Executes After (Like Finally)

Runs after the join point regardless of outcome (success or exception). Like a try-finally block. Use for: releasing resources, cleanup, metrics recording.

@After("execution(* com.example.service.DataService.processFile(..))")
public void releaseFileLock(JoinPoint jp) {
    String filename = (String) jp.getArgs()[0];
    fileLockManager.release(filename); // always release, even on exception
}

@AfterReturning — Executes After Successful Return

Has access to the return value via returning attribute. Only fires on success (no exception). Use for: logging successful operations, post-processing return values, caching.

@AfterReturning(
    pointcut = "execution(* com.example.repository.UserRepository.findById(..))",
    returning = "result"  // must match parameter name
)
public void onUserFound(JoinPoint jp, Optional<User> result) {
    result.ifPresent(user -> {
        metricsService.increment("user.cache.miss"); // came from DB
        auditLog.record("USER_FETCHED", user.getId(), jp.getArgs()[0]);
    });
}

@AfterThrowing — Executes After Exception

Has access to the thrown exception via throwing attribute. Does NOT suppress the exception — it re-throws. Use for: error logging, alerting, exception enrichment.

@AfterThrowing(
    pointcut = "within(com.example.service..*)",
    throwing = "ex"  // must match parameter name
)
public void onServiceException(JoinPoint jp, Exception ex) {
    // Centralized error handling — no try-catch in every service method
    alertingService.notify(
        new Alert(jp.getSignature().toShortString(), ex.getMessage(), Instant.now())
    );
    // Exception continues propagating after this method returns
}

@Around — Full Control (Most Powerful)

Must call pjp.proceed() to invoke the real method (or skip it). Can modify args, return value, catch exceptions. Use for: timing, circuit breaking, retry logic, caching, rate limiting.

@Aspect @Component
public class RetryAspect {

    @Around("@annotation(retryable)")
    public Object retry(ProceedingJoinPoint pjp, Retryable retryable) throws Throwable {
        int maxAttempts = retryable.maxAttempts();
        long backoffMs = retryable.backoffMs();
        Throwable lastException = null;

        for (int attempt = 1; attempt <= maxAttempts; attempt++) {
            try {
                return pjp.proceed(); // invoke real method
            } catch (RetryableException e) {
                lastException = e;
                log.warn("Attempt {}/{} failed for {}: {}",
                    attempt, maxAttempts, pjp.getSignature().toShortString(), e.getMessage());
                if (attempt < maxAttempts) {
                    Thread.sleep(backoffMs * attempt); // exponential-ish backoff
                }
            }
        }
        throw new MaxRetriesExceededException("Failed after " + maxAttempts + " attempts", lastException);
    }
}

// Usage:
@Service
public class ExternalApiService {
    @Retryable(maxAttempts = 3, backoffMs = 500)
    public ApiResponse callExternalApi(Request req) {
        return httpClient.post(req);
    }
}
⚠ GotchaIf @Around advice does NOT call pjp.proceed(), the real method is NEVER executed. This is easy to accidentally do in error handling paths — e.g., catching an exception and returning null instead of rethrowing or calling proceed(). Always have exactly one proceed() call path (or explicitly document that you're swallowing the call).
✅ Best PracticePrefer @Around when you need any combination of before + after behavior — it's more explicit and avoids the complexity of coordinating multiple advice types on the same join point. Use specific advice types (@Before, @AfterReturning) only when you truly need just that behavior and want maximum clarity.
🎯 Google L6 AngleThe production-grade retry aspect shown above is actually inferior to Spring Retry (@Retryable) or Resilience4j for serious use — they handle thread interruption, provide circuit breakers, support backoff strategies, and are battle-tested. But implementing it from scratch in an interview demonstrates deep understanding of AOP mechanics. In production, always prefer libraries over hand-rolled infrastructure concerns.

Write and explain complex pointcut expressions. Show all designator types.
POINTCUT
▼

execution() Pattern Anatomy

// Full pattern:
// execution(modifier-pattern? return-type declaring-type? method-name(params) throws?)
//   ? = optional component
//   * = any single segment
//   .. = any number of segments (in type) or any number of args (in params)
//   + = type + all subtypes (in declaring-type)

execution(public * com.example.service..*.*(..))
//        ^^^^^^   ^ ^^^^^^^^^^^^^^^^^ ^ ^^  ^^
//        modifier return declaring-type name params
//                        (service + all subpackages, any class, any method, any args)

execution(* *(String, ..))  // any method with first arg String, rest anything
execution(* *Service.find*(..))  // methods starting with 'find' on classes ending in 'Service'
execution(* com.example..*.*(com.example.domain.+, ..))  // first arg is domain type or subtype

All Pointcut Designators

// 1. execution() — method execution (most common)
@Pointcut("execution(* com.example.service.*.*(..))")
public void serviceExecution() {}

// 2. within() — all methods within a type/package
@Pointcut("within(com.example.service..*)")  // all classes in service package
public void inServiceLayer() {}

@Pointcut("within(com.example.service.UserService)")  // specific class only
public void inUserService() {}

// 3. this() — proxy implements type (matters for JDK proxy)
@Pointcut("this(com.example.Auditable)")
public void auditableProxy() {}  // proxy bean is Auditable (via Introduction)

// 4. target() — target object IS-A type
@Pointcut("target(com.example.service.BaseService)")
public void baseServiceTarget() {}  // real object extends BaseService

// 5. args() — argument types match
@Pointcut("args(com.example.domain.Order, ..)")
public void orderOperations() {}  // first arg is Order (or subtype)

// 6. @annotation() — method has annotation (most useful for custom annotations)
@Pointcut("@annotation(com.example.Audited)")
public void auditedMethods() {}

// 7. @within() — class has annotation
@Pointcut("@within(org.springframework.stereotype.Service)")
public void serviceBeans() {}  // all methods on @Service classes

// 8. @args() — runtime type of args has annotation
@Pointcut("@args(com.example.Sensitive)")  // arg objects of @Sensitive-annotated types

// 9. @target() — target class has annotation
@Pointcut("@target(org.springframework.transaction.annotation.Transactional)")
public void transactionalBeans() {}

// 10. bean() — Spring-specific! matches by bean name (not in AspectJ spec)
@Pointcut("bean(*Service)")  // all beans with names ending in 'Service'
public void serviceBeans2() {}

@Pointcut("bean(userService) || bean(orderService)")
public void specificBeans() {}

Composing Pointcuts with Boolean Operators

@Aspect @Component
public class EnterpriseAspect {

    @Pointcut("within(com.example.service..*)")
    public void serviceLayer() {}

    @Pointcut("execution(public * *(..))")
    public void publicMethods() {}

    @Pointcut("@annotation(org.springframework.transaction.annotation.Transactional)")
    public void transactional() {}

    // AND: public service methods
    @Pointcut("serviceLayer() && publicMethods()")
    public void publicServiceMethods() {}

    // NOT: public service methods that are NOT transactional
    @Pointcut("publicServiceMethods() && !transactional()")
    public void nonTransactionalPublicServiceMethods() {}

    // Named pointcut with parameter binding
    @Before("@annotation(audited)")  // binds annotation instance
    public void audit(JoinPoint jp, Audited audited) {
        String action = audited.action(); // access annotation attribute
        auditService.log(jp.getSignature().toShortString(), action);
    }
}
✅ Best PracticeDefine named @Pointcut methods in a dedicated PointcutLibrary class and reference them across multiple aspects. This eliminates duplicate pointcut string literals, makes refactoring safe (change one place), and documents your interception architecture explicitly.
🎯 Google L6 AngleThe @annotation() designator is how you build custom behaviors behind custom annotations — the foundation of every framework extension. @annotation(RateLimited) + @Around = rate limiting on any annotated method. @annotation(Idempotent) + @Around = idempotency enforcement. This is the pattern Google uses for cross-cutting policy enforcement at scale.

Deep dive into @Around advice and ProceedingJoinPoint. Show argument manipulation and exception wrapping.
AROUND
▼

ProceedingJoinPoint API

// ProceedingJoinPoint extends JoinPoint — adds proceed()
@Around("execution(* com.example.service.*.*(..))")
public Object comprehensive(ProceedingJoinPoint pjp) throws Throwable {
    // JoinPoint methods:
    Object[] originalArgs = pjp.getArgs();            // method arguments
    Object target = pjp.getTarget();                  // real object (not proxy)
    Object thisProxy = pjp.getThis();                 // the proxy object
    Signature sig = pjp.getSignature();               // method signature
    String methodName = sig.getName();
    String className = sig.getDeclaringTypeName();

    // For method-specific info, cast to MethodSignature:
    MethodSignature ms = (MethodSignature) pjp.getSignature();
    Method method = ms.getMethod();                   // java.lang.reflect.Method
    Class<?> returnType = ms.getReturnType();
    String[] paramNames = ms.getParameterNames();     // requires debug info
    Annotation[] annotations = method.getAnnotations();

    // Call real method:
    Object result = pjp.proceed();              // with original args
    // Or with modified args:
    Object[] modifiedArgs = Arrays.copyOf(originalArgs, originalArgs.length);
    modifiedArgs[0] = sanitize(modifiedArgs[0]); // sanitize first arg
    Object result2 = pjp.proceed(modifiedArgs); // pass modified args

    return result;
}

Argument Manipulation — Input Sanitization

@Aspect @Component
public class InputSanitizationAspect {

    @Around("@annotation(com.example.SanitizeInput)")
    public Object sanitizeStringArgs(ProceedingJoinPoint pjp) throws Throwable {
        Object[] args = pjp.getArgs();
        Object[] sanitized = new Object[args.length];

        for (int i = 0; i < args.length; i++) {
            if (args[i] instanceof String) {
                sanitized[i] = HtmlUtils.htmlEscape((String) args[i]); // XSS prevention
            } else if (args[i] instanceof UserRequest req) {
                sanitized[i] = req.withSanitizedFields(); // immutable record pattern
            } else {
                sanitized[i] = args[i];
            }
        }
        return pjp.proceed(sanitized); // proceed with sanitized args
    }
}

Return Value Manipulation — Response Enrichment

@Aspect @Component
public class ResponseEnrichmentAspect {

    @Around("execution(* com.example.api.*.*(..)) && @annotation(com.example.Enriched)")
    public Object enrichResponse(ProceedingJoinPoint pjp) throws Throwable {
        Object result = pjp.proceed(); // get real result

        if (result instanceof ApiResponse<?> response) {
            // Enrich with metadata, not possible to do in the service itself
            return response
                .withRequestId(RequestContextHolder.currentRequestId())
                .withTimestamp(Instant.now())
                .withVersion("v2");
        }
        return result;
    }
}

Exception Wrapping — Uniform Error Translation

@Aspect @Component
public class ExceptionTranslationAspect {

    @Around("within(com.example.integration..*)")  // all integration layer classes
    public Object translateExceptions(ProceedingJoinPoint pjp) throws Throwable {
        try {
            return pjp.proceed();
        } catch (HttpClientErrorException e) {
            throw new ExternalServiceException("HTTP " + e.getStatusCode() + ": " + e.getMessage(), e);
        } catch (ResourceAccessException e) {
            throw new ExternalServiceUnavailableException("Service unreachable: " + e.getMessage(), e);
        } catch (JsonProcessingException e) {
            throw new ExternalServiceException("Invalid response from service", e);
        }
        // Other exceptions propagate unchanged
    }
}
// Result: service layer only catches ExternalServiceException — doesn't know HTTP details
⚠ GotchaIf your @Around advice catches an exception and returns a value (instead of rethrowing), the caller thinks the method succeeded. Spring's @Transactional proxy marks the transaction for rollback only on unchecked exceptions or explicitly marked checked ones — swallowing an exception in an outer @Around aspect prevents rollback. Be very deliberate about which exceptions you catch vs rethrow.
🎯 Google L6 AngleException translation via AOP (shown above) is a clean alternative to putting try-catch boilerplate in every integration adapter. However, it hides error handling from code readers — an engineer reading the method won't know exceptions are being wrapped. Document it in the aspect's Javadoc and the package README. Explicit is generally better at scale than transparent magic — use this pattern judiciously.

How do you order multiple aspects? What is the advisor chain?
ORDERING
▼

The Advisor Chain

When multiple aspects match the same join point, Spring builds an interceptor chain — identical to the Servlet Filter chain or a linked list of middleware. Each advisor wraps the next. The chain is traversed for @Before/@Around entry in order from outer to inner, and traversed in reverse for @After/@AfterReturning.

Ordering with @Order

// Lower value = OUTER = higher priority
// @Before in outer aspect runs FIRST
// @After/@AfterReturning in outer aspect runs LAST

@Aspect @Component
@Order(1)  // OUTERMOST — security checked first
public class SecurityAspect {
    @Before("execution(* com.example.service.*.*(..))")
    public void checkSecurity() { /* fail fast if unauthorized */ }
}

@Aspect @Component
@Order(2)  // SECOND — transaction wraps business logic
public class TransactionAspect { /* conceptual — Spring's real @Transactional */}

@Aspect @Component
@Order(3)  // THIRD — cache inside transaction (reads in same tx)
public class CacheAspect { /* conceptual — Spring's real @Cacheable */}

@Aspect @Component
@Order(100)  // INNERMOST — logging closest to real method
public class LoggingAspect {
    @Around("execution(* com.example.service.*.*(..))")
    public Object log(ProceedingJoinPoint pjp) throws Throwable {
        log.info("Entering {}", pjp.getSignature());
        Object result = pjp.proceed();
        log.info("Exiting {}", pjp.getSignature());
        return result;
    }
}

Visual Execution Order

// Call flow for a method covered by @Order(1), @Order(2), @Order(3) aspects:
//
// [Security @Before] (order 1)
//   [Transaction @Before] (order 2 — START TRANSACTION)
//     [Cache @Before] (order 3 — CHECK CACHE)
//       [Logging @Around - before proceed] (order 100)
//         [Real Method Executes]
//       [Logging @Around - after proceed] (order 100)
//     [Cache @AfterReturning] (order 3 — POPULATE CACHE)
//   [Transaction @After] (order 2 — COMMIT/ROLLBACK)
// [Security @After] (order 1 — cleanup)
//
// Note: INNER aspects' @After runs BEFORE outer aspects' @After
// This is why cache inside tx is correct: cache populated before tx commits

Critical Ordering: Transaction vs Cache

// Scenario: @Transactional @Cacheable method
// With Transaction OUTSIDE Cache (correct default — Spring's actual ordering):
// - Cache miss → open tx → read from DB within tx → commit tx → populate cache
// - Cache hit → return cached value immediately (no DB, no tx) — CORRECT

// If Cache were OUTSIDE Transaction (wrong order):
// - Cache miss → populate cache with value read BEFORE tx committed
// - Another thread could read stale cached data while original tx is still running
// - Consistency violation in multi-threaded environments
⚠ GotchaDefault aspect ordering is undefined (depends on JVM class loading order and Spring's internal registration order). NEVER rely on default ordering for aspects that interact. Always specify @Order on every @Aspect class. This is especially critical for Security + Transaction interactions: security MUST be outer to prevent unauthorized code from running inside a transaction.
✅ Best PracticeDefine ordering constants in a shared class: AopOrder.SECURITY=1, AopOrder.TRANSACTION=2, AopOrder.CACHE=3, AopOrder.LOGGING=100. This makes ordering intent explicit and avoids magic numbers scattered across aspects.
🎯 Google L6 AngleSpring's built-in aspects have specific internal orders: @EnableTransactionManagement's advice runs at Ordered.LOWEST_PRECEDENCE by default; @EnableCaching is similar. This means custom aspects with any @Order value less than Integer.MAX_VALUE will wrap them. Understanding this prevents subtle ordering bugs when adding custom aspects to apps that use both @Transactional and @Cacheable.

What is @DeclareParents (Introduction) in Spring AOP? Show a practical use case.
INTRODUCTION
▼

Introduction — Adding Interfaces to Existing Classes

Introduction (also called inter-type declaration in AspectJ) allows adding a new interface and a default implementation to matched beans without modifying them. The proxy implements both the original type AND the introduced interface. Callers can cast the bean to the introduced interface.

// Introduce auditing capability to ALL service beans:

// Step 1: Define the interface to introduce
public interface Auditable {
    String getCreatedBy();
    void setCreatedBy(String user);
    Instant getLastModifiedAt();
    void setLastModifiedAt(Instant time);
}

// Step 2: Default implementation (delegates from proxy to this)
public class DefaultAuditable implements Auditable {
    private String createdBy;
    private Instant lastModifiedAt;

    @Override public String getCreatedBy() { return createdBy; }
    @Override public void setCreatedBy(String user) { this.createdBy = user; }
    @Override public Instant getLastModifiedAt() { return lastModifiedAt; }
    @Override public void setLastModifiedAt(Instant t) { this.lastModifiedAt = t; }
}

// Step 3: Declare the introduction in an @Aspect
@Aspect @Component
public class AuditIntroductionAspect {

    // value = which beans get the interface (all classes in service package, + = subtypes too)
    // defaultImpl = the class providing the implementation
    @DeclareParents(
        value = "com.example.service.*+",
        defaultImpl = DefaultAuditable.class
    )
    public Auditable auditable; // field type = interface being introduced
}

// Step 4: Use the introduced interface
@Service
public class AuditInterceptor {
    @Before("execution(* com.example.service.*.*(..))")
    public void stampAuditInfo(JoinPoint jp) {
        Object target = jp.getThis(); // the proxy
        if (target instanceof Auditable auditable) {
            // Every service bean now has Auditable — cast works!
            auditable.setCreatedBy(SecurityContextHolder.getContext()
                .getAuthentication().getName());
            auditable.setLastModifiedAt(Instant.now());
        }
    }
}

// In application code:
UserService userService = ctx.getBean(UserService.class);
userService.createUser(req); // normal call

// Cast to introduced interface — works!
Auditable audited = (Auditable) userService;
System.out.println("Last op by: " + audited.getCreatedBy());

Other Practical Introduction Use Cases

Use CaseInterface IntroducedBenefit
Performance monitoringMeasurableAll service beans expose timing metrics without code changes
State machineStatefulEntityAdd state transitions to entities transparently
Dirty trackingDirtyTrackableTrack which fields changed (for partial updates)
Serialization controlSerializableAdd serialization to classes that don't implement it
⚠ GotchaThe DefaultAuditable instance is shared per proxy instance — it is NOT a new object per method call. State stored in it persists for the lifetime of the bean (singleton scope). This is appropriate for audit info attached to a bean, but would be wrong for per-request or per-call state.
🎯 Google L6 AngleIntroduction is a form of the Mixin pattern — composing behaviors into classes without inheritance. It's powerful for adding orthogonal capabilities (tracking, monitoring, state management) to existing class hierarchies without modifying them. This is the Open/Closed Principle: you extend behavior (open) without modifying source (closed). In large codebases, this is how infrastructure concerns can be added to hundreds of classes simultaneously.

What is Load-Time Weaving and how do you configure it in Spring?
LTW
▼

What is Load-Time Weaving?

Load-Time Weaving (LTW) uses the Java instrumentation API (java.lang.instrument) to modify class bytecode as classes are loaded by the JVM. The AspectJ weaver runs as a Java agent, intercepts class loading, and applies aspects at that moment. Unlike proxy-based Spring AOP (runtime proxy creation), LTW weaves aspects directly into the bytecode of every matching class.

Setup Steps

// Step 1: Add aspectjweaver dependency
<dependency>
    <groupId>org.aspectj</groupId>
    <artifactId>aspectjweaver</artifactId>
</dependency>

// Step 2: JVM startup argument
-javaagent:/path/to/aspectjweaver-1.9.x.jar
// Spring Boot: add to JAVA_TOOL_OPTIONS or JVM_OPTS

// Step 3: Enable LTW in Spring configuration
@Configuration
@EnableLoadTimeWeaving(aspectjWeaving = EnableLoadTimeWeaving.AspectJWeaving.ENABLED)
public class LtwConfig { }

// Step 4: META-INF/aop.xml — tell AspectJ which classes to instrument
<!-- META-INF/aop.xml -->
<aspectj>
  <weaver options="-verbose -showWeaveInfo">
    <!-- Only weave classes in our packages (performance: don't weave JDK/Spring classes) -->
    <include within="com.example..*"/>
    <exclude within="com.example.generated..*"/>
  </weaver>
  <aspects>
    <!-- Which aspect classes to apply -->
    <aspect name="com.example.aspect.DomainEntityAuditAspect"/>
    <aspect name="com.example.aspect.TransactionAspect"/>
  </aspects>
</aspectj>
// Step 5: Aspect that can now intercept non-Spring objects
@Aspect  // NO @Component needed — not a Spring bean, woven at load time
public class DomainEntityAuditAspect {

    // Field-set join point — IMPOSSIBLE in Spring AOP
    @Before("set(* com.example.domain.Order.*) && target(order)")
    public void beforeFieldChange(Order order) {
        AuditContextHolder.recordChange(order.getId(), Thread.currentThread().getStackTrace());
    }

    // Constructor join point — IMPOSSIBLE in Spring AOP
    @AfterReturning("execution(com.example.domain.Order.new(..))")
    public void onOrderCreated(JoinPoint jp) {
        Order order = (Order) jp.getThis();
        DomainEventBus.publish(new OrderCreatedEvent(order));
        // Domain event published even though Order is not a Spring bean!
    }
}

When to Use LTW

  • Intercepting non-Spring objects (domain entities created with new)
  • Field get/set join points for dirty tracking or auditing
  • Constructor join points for domain event publishing
  • Fixing self-invocation issues without code changes (most impactful reason)
  • Eliminating proxy overhead in performance-critical paths
⚠ GotchaLTW adds complexity: JVM agent must be configured in all environments (dev, CI, prod containers), startup is slower (weaving happens at class load time), debugging is harder (woven bytecode doesn't match source), and build pipelines need updating. The aop.xml include/exclude list must be carefully maintained or you'll weave third-party classes unexpectedly.
✅ Best PracticeFor Spring Boot applications: if the only reason you want LTW is fixing self-invocation, prefer refactoring (Solution 3 in Q4) instead. Use LTW only when you genuinely need non-Spring-bean interception or field join points — the operational overhead is significant. Compile-time weaving (CTW with ajc) is an alternative that avoids the runtime agent but requires build tool changes.
🎯 Google L6 AngleLTW is the enabler of "transparent persistence" patterns in DDD: Spring Data's @Configurable support uses LTW to inject Spring beans into JPA entities (created with new) so entities can use repositories or services. This allows rich domain models where entities contain behavior, not just data — a key DDD principle. The trade-off is operational complexity vs architectural purity.

How are @Transactional, @Cacheable, @Async, and @Secured implemented under the hood? Trace the BeanPostProcessor chain.
INTERNALS
▼

The Common Pattern

All four follow the same architectural pattern:

  1. @Enable* annotation → imports a @Configuration class via ImportSelector
  2. Configuration class registers a BeanPostProcessor (specifically an AbstractAutoProxyCreator subclass)
  3. BPP's postProcessAfterInitialization() checks each bean for matching annotations/pointcuts
  4. Matching beans are replaced with CGLIB/JDK proxies in the singleton cache
  5. Proxies delegate to MethodInterceptor implementations on method calls

@Transactional Full Trace

// @EnableTransactionManagement
//   → imports TransactionManagementConfigurationSelector (ImportSelector)
//   → mode=PROXY: imports ProxyTransactionManagementConfiguration (@Configuration)
//   → registers:
//     - InfrastructureAdvisorAutoProxyCreator (extends AbstractAutoProxyCreator, a BPP)
//     - TransactionAttributeSource (reads @Transactional metadata)
//     - TransactionInterceptor (the MethodInterceptor — starts/commits/rolls back tx)
//     - BeanFactoryTransactionAttributeSourceAdvisor (pointcut + interceptor combined)
//
// At context startup (postProcessAfterInitialization):
//   InfrastructureAdvisorAutoProxyCreator scans each bean
//   If bean has @Transactional methods → wrap in CGLIB proxy
//
// At runtime (proxy method call):
//   proxy.save(entity)
//   → TransactionInterceptor.invoke()
//   → TransactionAspectSupport.invokeWithinTransaction()
//   → determineTransactionManager() → get PlatformTransactionManager
//   → begin transaction
//   → call real method: target.save(entity)
//   → commit or rollback based on exception type
//   → return result

@Cacheable Full Trace

// @EnableCaching
//   → imports CachingConfigurationSelector
//   → mode=PROXY: imports ProxyCachingConfiguration
//   → registers:
//     - AnnotationAwareAspectJAutoProxyCreator (a BPP — same one as AOP)
//     - BeanFactoryCacheOperationSourceAdvisor
//     - CacheInterceptor (MethodInterceptor for cache get/put/evict)
//     - AnnotationCacheOperationSource (reads @Cacheable, @CachePut, @CacheEvict)
//
// At runtime: proxy.findUser(id)
//   → CacheInterceptor.invoke()
//   → CacheAspectSupport.execute()
//   → check cache: cacheManager.getCache("users").get(id)
//   → cache hit: return cached value (real method NOT called)
//   → cache miss: call real method, store result in cache, return result

@Async Full Trace

// @EnableAsync
//   → imports AsyncConfigurationSelector
//   → imports ProxyAsyncConfiguration
//   → registers AsyncAnnotationBeanPostProcessor (BPP)
//
// AsyncAnnotationBeanPostProcessor.postProcessAfterInitialization():
//   → wraps @Async beans in proxy
//
// At runtime: proxy.sendEmail(user)
//   → AsyncExecutionInterceptor.invoke()
//   → submit Callable to TaskExecutor (default: SimpleAsyncTaskExecutor, use ThreadPoolTaskExecutor in prod)
//   → return CompletableFuture immediately (non-blocking)
//   → real method runs in executor thread pool

Why Private/Final Methods Don't Work

@Service
public class UserService {
    @Transactional
    private void saveInternal(User u) { ... }  // CGLIB can't override private — NO TRANSACTION

    @Transactional
    public final void saveUser(User u) { ... } // CGLIB can't override final — NO TRANSACTION

    @Async
    private void sendAsync() { ... }  // Same — no proxy interception

    // Fix: make methods public and non-final
    @Transactional
    public void saveInternal(User u) { ... } // WORKS
}
✅ Best PracticeWhen "annotation magic" isn't working, ask: (1) Is this bean a Spring-managed proxy? (AopUtils.isAopProxy(bean)), (2) Is it self-invocation?, (3) Is the method public and non-final?, (4) Is the correct @Enable* annotation present?, (5) Is the BPP registered? Answering these systematically diagnoses 95% of AOP issues.
🎯 Google L6 AngleThe ability to trace "how does @Transactional actually work" from annotation → ImportSelector → Configuration → BPP → CGLIB proxy → MethodInterceptor → PlatformTransactionManager demonstrates the L6 depth of Spring internals understanding. This knowledge is also what enables you to build first-class Spring integrations using the same patterns.

What is the performance overhead of Spring AOP proxies? When should you worry and when not?
PERFORMANCE
▼

Measuring the Overhead

JMH (Java Microbenchmark Harness) measurements for a simple method call (no advice logic, just proxy dispatch):

Call TypeTypical LatencyNotes
Direct method call~2-5 nsJIT-compiled, inlined by JVM
CGLIB proxy (no advice)~10-30 nsSubclass method override, JIT-friendly
JDK dynamic proxy (no advice)~50-150 nsReflection-based InvocationHandler
CGLIB + non-trivial advice~50-200 nsDepends on advice complexity
AspectJ LTW (woven)~2-5 nsInlined by JIT — same as direct call

Context: Why It Almost Never Matters

// A typical REST endpoint:
// HTTP parsing:         ~1ms
// Auth token validation:~2ms
// Spring proxy chain:   ~0.0002ms (200ns)  ← 0.006% of total
// Database query:       ~5ms
// JSON serialization:   ~1ms
// Total:                ~9ms

// The proxy overhead is NOISE compared to I/O.
// Even with 50 proxied beans in the call path: 50 * 200ns = 10,000ns = 0.01ms
// Still negligible against database/network latency.

When Proxy Overhead IS a Concern

// Scenario: in-memory computational hot path, no I/O
@Service
public class PricingEngine {
    // Called 1,000,000 times/second in pricing loop
    @Transactional(readOnly=true)  // THIS is the problem — tx management per call
    public BigDecimal calculatePrice(Product p, Customer c) {
        // 200ns pure computation + 200ns proxy overhead = 100% overhead!
        return p.getBasePrice().multiply(c.getDiscount());
    }
}

// Fix: remove @Transactional from hot inner loop methods
// Manage transaction at outer call boundary instead:
@Transactional(readOnly=true)
public PricingResult batchCalculate(List<Product> products, Customer customer) {
    // One transaction wraps entire batch
    return products.stream()
        .map(p -> calculatePriceNoProxy(p, customer)) // private method, no proxy
        .collect(...);
}

private BigDecimal calculatePriceNoProxy(Product p, Customer c) {
    return p.getBasePrice().multiply(c.getDiscount());
}

Profiling and Optimization Strategies

// 1. Measure first with JMH — never optimize on assumption
@State(Scope.Thread)
@BenchmarkMode(Mode.AverageTime)
@OutputTimeUnit(TimeUnit.NANOSECONDS)
public class ProxyBenchmark {
    private UserService direct;
    private UserService proxied;

    @Benchmark
    public User directCall() { return direct.findById(1L); }

    @Benchmark
    public User proxiedCall() { return proxied.findById(1L); }
}

// 2. Narrow pointcuts to reduce unnecessary advice application
// BAD: matches ALL methods in all packages
@Pointcut("execution(* *.*(..))")
// GOOD: only matches relevant service methods
@Pointcut("execution(* com.example.service.*Service.*(..))")

// 3. Use @Transactional only at service boundaries, not on every DAO method
// 4. Use AspectJ LTW for genuinely hot paths that need cross-cutting behavior
// 5. Consider programmatic transaction management for tight loops
⚠ GotchaPremature removal of @Transactional to "avoid proxy overhead" is one of the most dangerous optimizations in Spring apps. Without transactions, partial failures leave data inconsistent. A 200ns overhead that prevents a data corruption bug is essentially free. Always benchmark first and verify the optimization is safe before removing transactional boundaries.
✅ Best PracticeProfile with async profilers (async-profiler, JFR) in production-like conditions before optimizing AOP overhead. In practice, AOP proxy overhead is almost never in the top-10 hot spots of a real application. Database query time, serialization, and garbage collection dominate.
🎯 Google L6 AngleAt Google scale (millions of RPS), even 200ns per request adds up: 200ns × 1,000,000 RPS = 200ms of cumulative CPU time per second per server. For services with no I/O (in-memory caches, pricing engines), AOP overhead CAN matter. The L6 answer acknowledges this, explains the measurement approach (JMH + async-profiler), and provides the correct fix (widen transaction scope, use AspectJ LTW, or redesign the hot path). "Always measure first" is the key principle.
