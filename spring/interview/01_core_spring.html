What is Inversion of Control and how does Spring implement it differently from the Service Locator pattern?
FUNDAMENTALS
▼

Core Concept

Inversion of Control (IoC) means the framework calls your code, not the other way around — the "Hollywood Principle": "Don't call us, we'll call you." Dependency Injection is the most common form of IoC: the container injects dependencies into objects rather than objects pulling them.

Service Locator vs DI

With the Service Locator pattern, objects still actively retrieve their dependencies: ServiceLocator.get(PaymentService.class). They still have a hard coupling — now to the locator registry itself. Unit testing requires a real or stubbed locator. The dependency relationship is hidden inside the method body.

With Dependency Injection, dependencies are declared as constructor parameters or fields. The container resolves the full dependency graph at startup (eager) or on first access (lazy). Business code has zero coupling to Spring — it's just plain Java.

// BAD — Service Locator (still coupled)
public class OrderService {
    private final PaymentService payment =
        ServiceLocator.getInstance().get(PaymentService.class); // coupled to locator
}

// GOOD — Constructor DI (pure POJO, testable)
public class OrderService {
    private final PaymentService payment;

    public OrderService(PaymentService payment) {  // container injects this
        this.payment = payment;
    }
}

// Spring wiring
@Configuration
public class AppConfig {
    @Bean
    public PaymentService paymentService() { return new StripePaymentService(); }

    @Bean
    public OrderService orderService(PaymentService payment) {
        return new OrderService(payment);
    }
}

Spring's IoC Container

Spring's BeanFactory / ApplicationContext is the IoC container. Beans are registered as BeanDefinition objects; the container resolves the full dependency graph. Singletons are eagerly instantiated at context refresh; prototypes are created on each getBean() call.

✅ Best PracticeKeep all Spring annotations (@Service, @Transactional) on your infrastructure/application layer classes. Domain model objects should remain pure POJOs — no Spring imports. This preserves framework-agnostic domain logic.
🎯 Google L6 AngleThe real distinction is testability and coupling direction. A system designed around DI allows injecting fakes/mocks via the constructor in unit tests — no Spring context needed. When an interviewer asks "how do you keep your code testable at scale?", DI and the dependency rule (clean architecture) are the answer.

Explain the difference between BeanFactory and ApplicationContext. When would you ever use BeanFactory directly?
CONTAINER
▼

BeanFactory

BeanFactory is the root interface providing basic DI capabilities: getBean(), lazy instantiation of singletons (beans not created until first requested), and fundamental dependency resolution. It is intentionally minimal.

ApplicationContext

ApplicationContext extends BeanFactory and adds enterprise features:

  • Eager singleton instantiation at context refresh (fail-fast at startup)
  • MessageSource: internationalization (i18n) support
  • ApplicationEvent publishing: publishEvent() / @EventListener
  • Environment / PropertySource: unified property resolution across profiles
  • AOP auto-proxy creation: AbstractAutoProxyCreator is a BeanPostProcessor that wraps beans — only runs in ApplicationContext
  • ResourceLoader: classpath / filesystem / URL resource loading
  • BeanFactoryPostProcessor auto-detection: BFPPs registered as beans are automatically applied

Key Implementations

ClassUse Case
AnnotationConfigApplicationContextStandalone apps, tests — @Configuration based
AnnotationConfigWebApplicationContextTraditional Spring MVC (web.xml bootstrap)
GenericWebApplicationContextSpring Boot / Servlet 3 programmatic setup
ClassPathXmlApplicationContextLegacy XML-based configuration
// BeanFactory — rarely used directly
DefaultListableBeanFactory factory = new DefaultListableBeanFactory();
XmlBeanDefinitionReader reader = new XmlBeanDefinitionReader(factory);
reader.loadBeanDefinitions(new ClassPathResource("beans.xml"));
MyBean bean = factory.getBean(MyBean.class); // lazy — created here

// ApplicationContext — standard
ApplicationContext ctx = new AnnotationConfigApplicationContext(AppConfig.class);
// All singletons already created at this point (fail-fast)
MyBean bean = ctx.getBean(MyBean.class);
⚠ GotchaWith BeanFactory, BeanPostProcessors and BeanFactoryPostProcessors are NOT automatically detected and applied — you must register them programmatically. This means @Autowired, @PostConstruct, AOP proxies, and ${} placeholder resolution all silently stop working.
🎯 Google L6 AngleThe only legitimate use case for raw BeanFactory is resource-constrained embedded environments (IoT devices, batch micro-jobs with extreme startup time budgets). Modern Spring always uses ApplicationContext. Knowing this distinction demonstrates deep container understanding beyond "I use Spring Boot."

Walk me through the COMPLETE Spring bean lifecycle from class scanning to destruction.
LIFECYCLE
▼

Phase 1 — Discovery & BeanDefinition Creation

@ComponentScan triggers ClassPathScanningCandidateComponentProvider to find @Component-annotated classes. @Configuration classes are processed by ConfigurationClassPostProcessor. All discovered beans are registered as BeanDefinition objects (metadata only — no instances yet).

Phase 2 — BeanFactoryPostProcessor

BeanFactoryPostProcessor.postProcessBeanFactory() runs. Key built-in: PropertySourcesPlaceholderConfigurer resolves ${} placeholders in BeanDefinitions. ConfigurationClassPostProcessor (itself a BFPP) processes @Import, @ImportResource, @Bean methods. BeanDefinition metadata is now finalized.

Phase 3 — Instantiation

Bean instantiated via constructor (or factory method). For constructor injection, dependencies resolved and injected here simultaneously.

Phase 4 — Dependency Injection

Setter injection and field injection (@Autowired) applied by AutowiredAnnotationBeanPostProcessor.

Phase 5 — Aware Callbacks

Spring calls Aware interfaces in this order: BeanNameAware, BeanClassLoaderAware, BeanFactoryAware, EnvironmentAware, EmbeddedValueResolverAware, ResourceLoaderAware, ApplicationEventPublisherAware, MessageSourceAware, ApplicationContextAware.

Phase 6 — BeanPostProcessor.postProcessBeforeInitialization()

CommonAnnotationBeanPostProcessor processes @PostConstruct here (before afterPropertiesSet()).

Phase 7 — Initialization Callbacks

In order: @PostConstruct (already called in phase 6), then InitializingBean.afterPropertiesSet(), then init-method attribute / @Bean(initMethod=...).

Phase 8 — BeanPostProcessor.postProcessAfterInitialization() ← AOP PROXIES CREATED HERE

AbstractAutoProxyCreator checks each bean against registered advisors. If a match is found (e.g., @Transactional method present), the bean is wrapped in a CGLIB or JDK dynamic proxy. The proxy replaces the original bean in the singleton cache.

Phase 9 — Bean Ready

Bean stored in singletonObjects map. Available via getBean() and injection.

Phase 10 — Destruction

On context.close() or JVM shutdown hook: @PreDestroy → DisposableBean.destroy() → destroy-method attribute.

@Component
public class MetricsBeanPostProcessor implements BeanPostProcessor {
    private final Map<String, long> startTimes = new ConcurrentHashMap<>();

    @Override
    public Object postProcessBeforeInitialization(Object bean, String beanName) {
        startTimes.put(beanName, System.nanoTime());
        return bean; // MUST return the bean (or a replacement)
    }

    @Override
    public Object postProcessAfterInitialization(Object bean, String beanName) {
        long elapsed = System.nanoTime() - startTimes.getOrDefault(beanName, 0L);
        log.info("Bean '{}' initialized in {}ms", beanName, elapsed / 1_000_000);
        return bean; // return the (possibly proxied) bean
    }
}

@Service
public class UserService {
    @PostConstruct
    public void init() {
        // Runs in phase 6 — dependencies already injected, safe to use them
        log.info("UserService initialized, validating data source connection...");
    }

    @PreDestroy
    public void cleanup() {
        // Runs on context shutdown — close resources, flush buffers
        log.info("UserService shutting down");
    }
}
⚠ GotchaAOP proxies are created in Phase 8 (postProcessAfterInitialization). This is why @Transactional doesn't work on private methods — CGLIB subclass proxies can't override private methods. It's also why calling a @Transactional method from @PostConstruct may not start a transaction — at that phase, the proxy may not yet be the registered bean.
🎯 Google L6 AngleThe phased lifecycle explains almost every Spring "magic not working" bug. AOP proxy in phase 8 explains self-invocation failures, private method limitations, and why @Autowired fields are null in constructors when using field injection with circular deps. Memorize the phase numbers — interviewers love to probe exactly "when does X happen."

Threading Implications of Singleton

Singleton beans are shared across ALL threads. Any mutable state in a singleton is a data race. Classic bugs:

// BUG — SimpleDateFormat is not thread-safe
@Service
public class ReportService {
    private final SimpleDateFormat sdf = new SimpleDateFormat("yyyy-MM-dd"); // SHARED MUTABLE

    public String formatDate(Date d) {
        return sdf.format(d); // Race condition — concurrent calls corrupt internal state
    }
}

// FIX 1 — Use thread-safe alternative
@Service
public class ReportService {
    private static final DateTimeFormatter DTF =
        DateTimeFormatter.ofPattern("yyyy-MM-dd"); // DateTimeFormatter is immutable

    public String formatDate(LocalDate d) { return d.format(DTF); }
}

// FIX 2 — ThreadLocal (use sparingly, remember to clean up)
@Service
public class ReportService {
    private static final ThreadLocal<SimpleDateFormat> SDF =
        ThreadLocal.withInitial(() -> new SimpleDateFormat("yyyy-MM-dd"));

    public String formatDate(Date d) { return SDF.get().format(d); }
}

Prototype-in-Singleton Problem

Injecting a prototype into a singleton via @Autowired injects once at startup — the prototype effectively becomes a singleton. See Q19 for all four solutions.

// Request scope — needs scoped proxy or ObjectProvider in singleton
@Component
@Scope(value = WebApplicationContext.SCOPE_REQUEST, proxyMode = ScopedProxyMode.TARGET_CLASS)
public class RequestContext {
    private String correlationId = UUID.randomUUID().toString();
    public String getCorrelationId() { return correlationId; }
}

@Service
public class AuditService {
    @Autowired
    private RequestContext requestContext; // proxy — fetches correct request-scoped bean per call
}
✅ Best PracticeDesign singleton beans to be stateless (hold no mutable instance state). All state goes in method-local variables. If you need per-request state in a singleton, use ThreadLocal or inject a scoped proxy.
🎯 Google L6 AngleThread safety of singletons is a common production bug source. Ask yourself: "if 1000 threads call this method simultaneously, does it corrupt shared state?" Spring services should be designed like functions — input in, output out, no side effects on fields. Stateless singletons scale horizontally without synchronization overhead.

How does Spring resolve circular dependencies? What changed in Spring 6?
CIRCULAR_DEPS
▼

The Three-Level Cache Mechanism

DefaultSingletonBeanRegistry maintains three maps to break cycles for setter/field-injected singletons:

  • singletonObjects: fully initialized beans (L1 cache)
  • earlySingletonObjects: early-exposed partially initialized beans (L2 cache)
  • singletonFactories: ObjectFactory lambdas that can produce an early reference (L3 cache)

Resolution Walkthrough: A depends on B, B depends on A

  1. Container starts creating A → A added to singletonFactories (not yet initialized)
  2. A needs B → container starts creating B
  3. B needs A → container checks L1 (not there), L2 (not there), L3 → finds A's ObjectFactory
  4. ObjectFactory produces A's early reference → moved to L2 (earlySingletonObjects)
  5. B injection of A completes with early reference → B finishes initialization → B moved to L1
  6. Back to A → A's B dependency injected → A finishes → A moved to L1

Constructor Injection Circular Deps — Impossible to Resolve

// This ALWAYS fails — Spring cannot create A without B, cannot create B without A
@Service
public class ServiceA {
    public ServiceA(ServiceB b) { ... } // needs B to instantiate
}
@Service
public class ServiceB {
    public ServiceB(ServiceA a) { ... } // needs A to instantiate
}
// Result: BeanCurrentlyInCreationException at startup

Spring 6 Breaking Change

In Spring 6 (and Spring Boot 3), setter/field circular dependencies now fail by default. The three-level cache still exists, but is disabled unless explicitly opted in. You must set:

# application.properties
spring.main.allow-circular-references=true

# Or programmatically
SpringApplication app = new SpringApplication(MyApp.class);
app.setAllowCircularReferences(true);
app.run(args);
⚠ GotchaThe three-level cache mechanism interacts badly with AOP proxies. If bean A gets proxied (e.g., has @Transactional), B receives an early reference to the RAW A, not the proxy. Spring detects this inconsistency and either throws an exception or applies a workaround — this is fragile and another reason circular deps are banned by default in Spring 6.
✅ Best PracticeTreat circular dependency detection as a design-time signal. A ↔ B cycle usually means: (1) A and B should be merged into one class, (2) a dependency should be inverted via interface, or (3) an event-based decoupling is needed (A publishes event, B listens — introduce ApplicationEventPublisher).
🎯 Google L6 AngleCircular dependencies violate the Dependency Inversion Principle and signal missing abstractions. The L6 answer goes beyond "configure Spring to allow it" to "why does this cycle exist and what design change eliminates it?" Common fixes: extract a shared dependency into a third service, use event publishing, or apply the mediator pattern.

What is a BeanPostProcessor? How does it differ from BeanFactoryPostProcessor? Give real examples of each.
BPP
▼

BeanPostProcessor — Operates on Bean INSTANCES

Runs after each bean is instantiated and dependencies injected. Two hooks:

  • postProcessBeforeInitialization() — before @PostConstruct / afterPropertiesSet()
  • postProcessAfterInitialization() — after init callbacks, this is where AOP proxies are created

Key built-in BPPs: AutowiredAnnotationBeanPostProcessor (processes @Autowired, @Value), CommonAnnotationBeanPostProcessor (processes @PostConstruct, @PreDestroy, @Resource), AbstractAutoProxyCreator (creates AOP proxies for @Transactional, @Cacheable, etc.).

BeanFactoryPostProcessor — Operates on BeanDefinition METADATA

Runs BEFORE any beans are instantiated. Modifies the blueprint, not the object. Key built-in BFPPs: ConfigurationClassPostProcessor (processes @Configuration, @ComponentScan, @Import, @Bean), PropertySourcesPlaceholderConfigurer (resolves ${} in XML/annotations).

// Custom BPP — wraps every @Service bean in a timing proxy
@Component
public class TimingBeanPostProcessor implements BeanPostProcessor {

    @Override
    public Object postProcessAfterInitialization(Object bean, String beanName) {
        if (bean.getClass().isAnnotationPresent(Service.class)) {
            return Proxy.newProxyInstance(
                bean.getClass().getClassLoader(),
                bean.getClass().getInterfaces(),
                (proxy, method, args) -> {
                    long start = System.nanoTime();
                    try {
                        return method.invoke(bean, args);
                    } finally {
                        log.info("{}.{} took {}ns", beanName, method.getName(),
                            System.nanoTime() - start);
                    }
                }
            );
        }
        return bean;
    }
}

// Custom BFPP — change all matching beans to prototype scope
@Component
public class PrototypeForcingBFPP implements BeanFactoryPostProcessor {

    @Override
    public void postProcessBeanFactory(ConfigurableListableBeanFactory factory) {
        for (String name : factory.getBeanDefinitionNames()) {
            BeanDefinition bd = factory.getBeanDefinition(name);
            if (bd.getBeanClassName() != null
                    && bd.getBeanClassName().startsWith("com.example.plugin.")) {
                bd.setScope(BeanDefinition.SCOPE_PROTOTYPE);
            }
        }
    }
}
⚠ GotchaBPPs themselves must be instantiated very early (before the beans they process). If your BPP declares @Autowired dependencies, those dependencies will be eagerly created too — potentially bypassing lazy initialization and causing other BPPs to not yet be registered. Keep BPPs lean and avoid injecting regular application beans into them.
🎯 Google L6 AngleThe entire Spring framework is built on these two extension points. When asked "how does @Transactional work?", the complete answer involves: @EnableTransactionManagement registers InfrastructureAdvisorAutoProxyCreator (a BPP subclass), which in postProcessAfterInitialization() wraps @Transactional beans in CGLIB proxies. This is the architecture behind every @Enable* annotation.

How does @Configuration work internally? What is CGLIB proxying of @Configuration classes and why does it matter?
CONFIGURATION
▼

The CGLIB Subclass Trick

When ConfigurationClassPostProcessor processes a @Configuration class, it uses CGLIB to create a subclass of it. This subclass is what gets registered as a bean. The subclass overrides all @Bean methods.

When one @Bean method calls another (inter-bean method reference), the CGLIB override intercepts the call and returns the cached singleton from the container instead of executing the method body again. This preserves singleton semantics.

// Without understanding this, you'd expect TWO DataSource instances:
@Configuration
public class DataConfig {

    @Bean
    public DataSource dataSource() {
        return new HikariDataSource(hikariConfig());  // calls hikariConfig() below
    }

    @Bean
    public HikariConfig hikariConfig() {
        HikariConfig config = new HikariConfig();
        config.setJdbcUrl("jdbc:postgresql://localhost/mydb");
        return config;
    }

    @Bean
    public JdbcTemplate jdbcTemplate() {
        return new JdbcTemplate(dataSource()); // calls dataSource() — but CGLIB returns singleton!
    }
}
// Result: dataSource() body executes ONCE. jdbcTemplate() gets the SAME DataSource instance.

// proxyBeanMethods=false — "lite" mode, each call creates new instance
@Configuration(proxyBeanMethods = false)
public class LiteConfig {
    @Bean
    public HikariConfig hikariConfig() { return new HikariConfig(); }

    @Bean
    public DataSource dataSource() {
        return new HikariDataSource(hikariConfig()); // creates SECOND HikariConfig — broken!
    }
}

// Safe use of proxyBeanMethods=false: when @Bean methods don't call each other
@Configuration(proxyBeanMethods = false) // Spring Boot uses this in auto-configs
public class MetricsAutoConfiguration {
    @Bean
    @ConditionalOnMissingBean
    public MeterRegistry meterRegistry() { return new SimpleMeterRegistry(); }
}

Why @Configuration Cannot Be Final

CGLIB generates a subclass at runtime. final classes cannot be subclassed → BeanDefinitionParsingException at startup. Same applies to @Bean methods — they can't be final in @Configuration classes.

✅ Best PracticeUse proxyBeanMethods=false in Spring Boot auto-configuration classes (where beans don't reference each other) for faster startup — no CGLIB subclassing needed. Use full proxy mode (default) in application @Configuration where @Bean methods call each other.
🎯 Google L6 AngleSpring Boot 2.x changed auto-configuration classes to proxyBeanMethods=false across the board, cutting startup time measurably on large apps. Knowing this explains why you can't call @Bean methods from each other inside @AutoConfiguration classes — a subtle gotcha when writing custom starters.

Explain the @Autowired resolution algorithm. What happens with multiple matching beans?
AUTOWIRED
▼

Resolution Algorithm — Step by Step

  1. Match by type — find all beans assignable to the injection point type
  2. Exactly one match → use it
  3. Multiple matches → check for @Qualifier on the injection point
  4. No @Qualifier → try matching by field/parameter name as bean name
  5. Multiple still unresolved → @Primary bean wins
  6. Still ambiguous → NoUniqueBeanDefinitionException
  7. Zero matches + required=true (default) → NoSuchBeanDefinitionException
// Scenario: two PaymentService implementations
@Service("stripePayment")
public class StripePaymentService implements PaymentService { ... }

@Service("paypalPayment")
@Primary  // wins when no qualifier specified
public class PaypalPaymentService implements PaymentService { ... }

@Service
public class OrderService {
    // Uses @Primary — gets PaypalPaymentService
    @Autowired
    private PaymentService defaultPayment;

    // Uses @Qualifier — gets StripePaymentService regardless of @Primary
    @Autowired
    @Qualifier("stripePayment")
    private PaymentService stripePayment;

    // Injects ALL PaymentService beans — ordered by @Order/@Priority
    @Autowired
    private List<PaymentService> allPayments;

    // Map: key=beanName, value=bean — useful for strategy pattern
    @Autowired
    private Map<String, PaymentService> paymentsByName;
    // paymentsByName.get("stripePayment") returns StripePaymentService

    // Optional dependency — no exception if missing
    @Autowired(required = false)
    private Optional<FraudDetectionService> fraudDetection;

    // Lazy, optional, supports getIfAvailable(), getIfUnique(), stream()
    @Autowired
    private ObjectProvider<NotificationService> notificationProvider;
}
⚠ GotchaName-based fallback (step 4) is fragile — it depends on the field variable name matching the bean name exactly. This is implicit and brittle. Always use explicit @Qualifier when multiple candidates exist in production code.
✅ Best PracticeObjectProvider<T> is the most flexible injection point: it handles optional deps, lazy initialization, and multiple candidates without throwing exceptions at startup. Prefer it over @Autowired(required=false) for optional dependencies and over direct injection for prototype beans.
🎯 Google L6 AngleThe strategy pattern with Map<String, T> injection is a powerful pattern for plugin architectures. A Map<String, PaymentProcessor> lets you select a payment processor by name at runtime without if-else chains. Each new processor just needs a @Service annotation and the map auto-populates.

Compare constructor vs setter vs field injection. Make the case for each and explain which to use and when.
INJECTION_STYLE
▼

Constructor Injection — RECOMMENDED

@Service
public class OrderService {
    private final PaymentService payment;  // final — immutable after construction
    private final InventoryService inventory;

    // Spring auto-detects single constructor — @Autowired optional since Spring 4.3
    public OrderService(PaymentService payment, InventoryService inventory) {
        this.payment = Objects.requireNonNull(payment, "PaymentService required");
        this.inventory = Objects.requireNonNull(inventory, "InventoryService required");
    }
}

// Unit test — NO Spring context needed
@Test
void testOrder() {
    var service = new OrderService(mockPayment, mockInventory); // plain Java
    // test...
}

Advantages: dependencies are final (immutable), explicit in API, fails fast at construction, fully testable without Spring, enforces no circular constructor deps, documents intent clearly.

Setter Injection — USE FOR OPTIONAL/RECONFIGURABLE

@Service
public class ReportService {
    private DataSource dataSource; // required
    private CacheManager cacheManager; // optional

    @Autowired
    public void setDataSource(DataSource ds) { this.dataSource = ds; }

    @Autowired(required = false)
    public void setCacheManager(CacheManager cm) { this.cacheManager = cm; }
}

Use when: the dependency is optional, circular deps exist (last resort), or the dependency needs to be replaceable after construction (rare).

Field Injection — AVOID in Production Code

@Service
public class UserService {
    @Autowired  // Anti-pattern — DO NOT USE in production
    private UserRepository repo;

    @Autowired
    private EmailService email;
}

// Problems:
// 1. Dependencies hidden — not visible in API/constructor
// 2. Cannot declare final — mutable after construction
// 3. Unit test requires @InjectMocks or Spring context (cannot just call new)
// 4. Allows creating object in invalid state (all fields null until Spring injects)
// 5. Encourages adding "just one more" dependency (big constructor smell is suppressed)
✅ Best PracticeConstructor injection for required dependencies. Setter injection for optional/reconfigurable. Never field injection in production code. In tests, field injection via @InjectMocks (Mockito) is acceptable since it's framework-test code, not production logic.
🎯 Google L6 AngleConstructor injection enforces design discipline at scale. In large teams, a growing constructor is a visible code smell that triggers refactoring discussions. With field injection, it's trivially easy to inject a 15th dependency nobody notices. Constructor length is a natural complexity budget — when it hurts to add more, the design is telling you to split the class.

Explain the Aware interfaces in Spring. When is it legitimate to use ApplicationContextAware?
AWARE
▼

The Aware Interface Family

Aware interfaces are callback mechanisms called during Phase 5 of the bean lifecycle. Implementing them couples your bean to Spring infrastructure — use sparingly.

InterfaceWhat You GetUse Instead
BeanNameAwareBean's name in the containerRarely needed
BeanFactoryAwareThe BeanFactory@Autowired BeanFactory
ApplicationContextAwareThe full ApplicationContext@Autowired ApplicationContext
EnvironmentAwareEnvironment / profiles / properties@Value, @Autowired Environment
ApplicationEventPublisherAwareEvent publisher@Autowired ApplicationEventPublisher
MessageSourceAwarei18n message source@Autowired MessageSource
ResourceLoaderAwareResourceLoader@Autowired ResourceLoader

Legitimate Uses of ApplicationContextAware

// Use case: dynamic bean lookup in a plugin/registry system
// (ObjectProvider is better for single-type lookups, but this handles dynamic types)
@Component
public class BeanRegistry implements ApplicationContextAware {
    private ApplicationContext ctx;

    @Override
    public void setApplicationContext(ApplicationContext ctx) {
        this.ctx = ctx;
    }

    // Retrieve a handler by name — type unknown at compile time
    public <T> T getHandler(String name, Class<T> type) {
        return ctx.getBean(name, type);
    }

    // Get all beans of a type — useful for plugin architectures
    public <T> Map<String, T> getAllHandlers(Class<T> type) {
        return ctx.getBeansOfType(type);
    }
}

// BETTER alternative using @Autowired directly:
@Component
public class BetterBeanRegistry {
    @Autowired
    private ApplicationContext ctx; // @Autowired is cleaner than Aware

    // Or inject the map directly:
    @Autowired
    private Map<String, PaymentHandler> handlers; // key=beanName
}
⚠ GotchaImplementing ApplicationContextAware couples your bean to the Spring ApplicationContext interface. This makes it impossible to use the class outside a Spring context (e.g., in a non-Spring microservice, or without an ApplicationContext in unit tests). The @Autowired ApplicationContext ctx approach has the same coupling but is at least consistent with Spring idioms.
🎯 Google L6 AngleFramework/library code (like Spring itself, third-party starters) legitimately uses Aware interfaces to avoid requiring users to add extra configuration. In application code, Aware interfaces are almost always a sign of missing design — a service that needs to dynamically look up beans probably needs a better-defined plugin interface, a Map<String, T> injection, or a ServiceLocator factory pattern via ObjectProvider.

How does @ComponentScan work internally? What are the performance implications for large projects?
COMPONENT_SCAN
▼

Internal Mechanics

ClassPathScanningCandidateComponentProvider scans the classpath. For each .class file found, it uses ASM bytecode reading (not full class loading — much faster) to inspect annotations without triggering static initializers or class loading overhead. It checks include/exclude filters, then detects the @Component meta-annotation (stereotype detection). A BeanDefinition is created for each candidate.

Performance Implications

Classpath scanning at startup is O(n) in the number of classes on the classpath. For a large monolith with hundreds of transitive dependencies:

  • @SpringBootApplication scans its base package and ALL sub-packages recursively
  • Placing @SpringBootApplication too high (e.g., at root) scans thousands of third-party classes
  • Multi-module projects with broad scans can add 500ms-2s to startup
// Performance-optimized: explicit base packages
@SpringBootApplication(scanBasePackages = {
    "com.example.service",
    "com.example.web",
    "com.example.data"
})
// Avoids scanning com.example.thirdparty, com.example.generated, etc.

// Even better: type-safe marker classes
@ComponentScan(basePackageClasses = {
    ServiceLayerMarker.class,  // just a marker interface in each module
    WebLayerMarker.class
})

// include/exclude filters
@ComponentScan(
    basePackages = "com.example",
    excludeFilters = @ComponentScan.Filter(
        type = FilterType.ANNOTATION,
        classes = Repository.class  // exclude repos from this context
    ),
    includeFilters = @ComponentScan.Filter(
        type = FilterType.CUSTOM,
        classes = MyCustomTypeFilter.class
    )
)

// Custom TypeFilter
public class PluginTypeFilter implements TypeFilter {
    @Override
    public boolean match(MetadataReader mr, MetadataReaderFactory factory) {
        return mr.getClassMetadata().getClassName().contains("Plugin");
    }
}

Spring Boot Auto-Configuration — NOT Component Scan

Spring Boot's auto-configuration mechanism does NOT use @ComponentScan. In Boot 3.x it reads META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports. In Boot 2.x it reads META-INF/spring.factories. This is significantly faster than scanning because it's an explicit list.

✅ Best PracticeSpring Boot 3.x AOT (Ahead-of-Time) processing moves component scanning to build time and generates a BeanFactory initializer class. At runtime, the pre-computed result is loaded instantly. Enable with spring-boot-maven-plugin's AOT processing for GraalVM native images or fast JVM startup.
🎯 Google L6 AngleAt Google-scale, startup time directly impacts autoscaling latency and deployment velocity. A service that takes 30s to start can't autoscale fast enough under load spikes. Spring Boot AOT + GraalVM native compilation reduces startup from seconds to milliseconds — a critical tool in the L6 toolkit for high-scale services.

Explain Spring's ApplicationEvent system. How does @TransactionalEventListener work and what are its gotchas?
EVENTS
▼

Publishing Events

// Define event (POJO since Spring 4.2 — doesn't need to extend ApplicationEvent)
public record UserRegisteredEvent(String userId, String email, Instant occurredAt) {}

// Publishing
@Service
public class UserService {
    private final ApplicationEventPublisher eventPublisher;

    public UserService(ApplicationEventPublisher eventPublisher) {
        this.eventPublisher = eventPublisher;
    }

    @Transactional
    public User registerUser(RegisterRequest req) {
        User user = userRepository.save(new User(req));
        // Event published synchronously within same thread/transaction
        eventPublisher.publishEvent(new UserRegisteredEvent(user.getId(), user.getEmail(), Instant.now()));
        return user;
    }
}

Listening to Events

@Component
public class WelcomeEmailListener {

    // Basic listener — runs synchronously in same thread as publisher
    @EventListener
    public void onUserRegistered(UserRegisteredEvent event) {
        emailService.sendWelcome(event.email());
    }

    // Conditional — SpEL expression
    @EventListener(condition = "#event.email.endsWith('.edu')")
    public void onStudentRegistered(UserRegisteredEvent event) { ... }

    // Async listener — runs in separate thread from TaskExecutor
    @Async
    @EventListener
    public void onUserRegisteredAsync(UserRegisteredEvent event) { ... }

    // Chained — return value is published as another event
    @EventListener
    public NotificationEvent handle(UserRegisteredEvent e) {
        return new NotificationEvent(e.userId(), "Welcome!");
    }
}

@TransactionalEventListener

@Component
public class AuditListener {

    // AFTER_COMMIT (default) — fires only if transaction commits successfully
    @TransactionalEventListener(phase = TransactionPhase.AFTER_COMMIT)
    public void onUserRegistered(UserRegisteredEvent event) {
        // Safe to call external services — user definitely persisted
        notificationService.send(event.email(), "Welcome!");
        auditLog.record(event.userId(), "USER_REGISTERED");
    }

    // BEFORE_COMMIT — still inside transaction, can roll back
    @TransactionalEventListener(phase = TransactionPhase.BEFORE_COMMIT)
    public void validate(UserRegisteredEvent event) {
        if (!complianceService.isAllowed(event.email())) {
            throw new ComplianceException("Blocked: " + event.email()); // rolls back tx
        }
    }

    // fallbackExecution=true — fires even with no active transaction
    @TransactionalEventListener(fallbackExecution = true)
    public void always(UserRegisteredEvent event) { ... }
}
⚠ Gotcha 1@TransactionalEventListener with AFTER_COMMIT (default): if there is NO active transaction when the event is published, the listener is NEVER called. This silently drops events. Set fallbackExecution=true to fire even without a transaction.
⚠ Gotcha 2If the AFTER_COMMIT listener throws an exception, the original transaction has already committed. You CANNOT roll it back. The exception is swallowed (logged) by default. For at-least-once delivery guarantees, pair with the Outbox Pattern: write the event to an outbox table within the same transaction, poll and deliver separately.
⚠ Gotcha 3@Async @TransactionalEventListener runs in a different thread — it has no transaction context. Do NOT call @Transactional methods from it expecting to inherit the original transaction (it's already committed). Start a new transaction if needed.
🎯 Google L6 AngleThe Outbox Pattern + AFTER_COMMIT listener is the correct architecture for reliable event delivery in distributed systems. Write event to outbox table (same TX as business data), separate relay process polls outbox and publishes to Kafka/SQS with at-least-once semantics. This avoids the dual-write problem entirely.

How do you implement a custom Spring bean scope? Walk through a tenant-scoped bean.
CUSTOM_SCOPE
▼

The Scope Interface

public class TenantScope implements Scope {
    // Per-thread map: tenantId -> (beanName -> bean instance)
    private final ThreadLocal<Map<String, Object>> beans =
        ThreadLocal.withInitial(HashMap::new);

    // Per-thread destruction callbacks: beanName -> callback
    private final ThreadLocal<Map<String, Runnable>> callbacks =
        ThreadLocal.withInitial(HashMap::new);

    @Override
    public Object get(String name, ObjectFactory<?> objectFactory) {
        Map<String, Object> tenantBeans = beans.get();
        return tenantBeans.computeIfAbsent(name, k -> objectFactory.getObject());
    }

    @Override
    public Object remove(String name) {
        callbacks.get().remove(name); // remove callback too
        return beans.get().remove(name);
    }

    @Override
    public void registerDestructionCallback(String name, Runnable callback) {
        callbacks.get().put(name, callback);
    }

    @Override
    public Object resolveContextualObject(String key) {
        // Expose current tenant id as contextual object
        if ("tenantId".equals(key)) return TenantContextHolder.currentTenantId();
        return null;
    }

    @Override
    public String getConversationId() {
        return TenantContextHolder.currentTenantId();
    }

    // Call this at end of tenant request/session to trigger destruction callbacks
    public void clearTenant() {
        callbacks.get().values().forEach(Runnable::run);
        beans.remove();
        callbacks.remove();
    }
}

// Registration
@Configuration
public class TenantScopeConfig {
    @Bean
    public static CustomScopeConfigurer customScopeConfigurer() {
        CustomScopeConfigurer configurer = new CustomScopeConfigurer();
        configurer.addScope("tenant", new TenantScope());
        return configurer;
    }
}

// Usage
@Component
@Scope("tenant")
public class TenantCache {
    private final Map<String, Object> store = new HashMap<>();

    public void put(String key, Object value) { store.put(key, value); }
    public Object get(String key) { return store.get(key); }
}

// In service — each tenant gets their own TenantCache instance
@Service
public class DataService {
    @Autowired
    private TenantCache tenantCache; // scoped proxy needed for singleton-to-tenant injection
}
⚠ GotchaDestruction callbacks must be called manually (e.g., in a servlet filter's finally block). Unlike request/session scopes which are managed by the servlet container lifecycle, custom scope destruction is entirely your responsibility. Forgetting this causes memory leaks.
✅ Best PracticeRegister the TenantScope bean as static @Bean — scope registrations must happen before any beans are instantiated, and static @Bean methods in @Configuration are processed earlier (before the configuration class itself is instantiated via CGLIB).
🎯 Google L6 AngleTenant-scoped beans are the foundation of multi-tenant SaaS architectures on Spring. Combined with TenantContextHolder populated in a HandlerInterceptor (from JWT claims or subdomain), you get tenant-specific configuration, caches, and connection pools all managed by the container — without polluting business logic with tenant ID parameters.

What is lazy initialization in Spring and what are the production trade-offs?
LAZY
▼

Lazy Initialization Options

// Per-bean lazy init
@Service
@Lazy
public class HeavyReportingService { ... }  // not created until first getBean()

// Lazy injection point — injects a proxy, real bean created on first method call
@Service
public class OrderService {
    @Lazy
    @Autowired
    private HeavyReportingService reporting; // proxy injected at startup, real bean deferred
}

// Global lazy init — all beans lazy (Spring Boot)
# application.properties
spring.main.lazy-initialization=true

// Or programmatically
SpringApplication app = new SpringApplication(MyApp.class);
app.setLazyInitialization(true);

Benefits

  • Faster startup time — measurable impact on large apps (10-40% reduction)
  • Lower initial memory footprint
  • Useful in dev/test loops where you don't exercise all code paths every run
  • Spring Boot DevTools enables lazy init by default in dev profile

Production Trade-offs

RiskImpactMitigation
Configuration errors surface at first requestRequest fails, not startup — breaks fail-fastAdd startup smoke tests or Actuator health checks that exercise all beans
First request latency spikeP99 latency spikes for first user per instanceActuator warm-up endpoint; K8s readiness probe that triggers all lazy beans
Memory spikes unpredictablyOOM under load when many beans initialize simultaneouslyLoad test thoroughly before enabling globally
Health checks may pass prematurelyKubernetes routes traffic before beans readyImplement proper readiness probe that initializes all lazy beans
✅ Best PracticeFor production JVM services: prefer eager initialization (default). For serverless/Lambda: evaluate lazy init carefully per code path. For GraalVM native: use AOT compilation instead — it eliminates startup overhead without lazy-init's trade-offs.
🎯 Google L6 AngleLazy init is a blunt instrument. The surgical approach at Google scale is: AOT compilation (Spring Boot 3 + GraalVM) for services that need sub-100ms startup, and structured lazy init (only specific heavyweight beans like JasperReports, complex cache warmers) with explicit readiness gate checks for everything else. Global spring.main.lazy-initialization=true in production is usually a red flag.

Explain BeanDefinition. How do you programmatically register beans at runtime?
BEAN_DEFINITION
▼

What is a BeanDefinition?

A BeanDefinition is the blueprint / metadata for a bean — the recipe the container uses to create instances. Key properties: beanClass, scope, lazyInit, propertyValues, constructorArguments, initMethodName, destroyMethodName, factoryBeanName, factoryMethodName, primary, dependsOn.

Programmatic Registration Methods

// Method 1 — Direct registration at runtime (after context refresh)
@Service
public class PluginLoader implements ApplicationContextAware {
    private DefaultListableBeanFactory beanFactory;

    @Override
    public void setApplicationContext(ApplicationContext ctx) {
        this.beanFactory = (DefaultListableBeanFactory) ctx.getAutowireCapableBeanFactory();
    }

    public void loadPlugin(String name, Class<?> pluginClass) {
        BeanDefinition bd = BeanDefinitionBuilder
            .genericBeanDefinition(pluginClass)
            .setScope(BeanDefinition.SCOPE_SINGLETON)
            .addPropertyValue("name", name)
            .getBeanDefinition();
        beanFactory.registerBeanDefinition(name + "Plugin", bd);
        // Bean is immediately instantiated and available
    }
}

// Method 2 — ImportBeanDefinitionRegistrar (at @Configuration processing time)
// Used by @MapperScan, @EnableFeignClients, @EnableJpaRepositories etc.
public class MetricsRegistrar implements ImportBeanDefinitionRegistrar {
    @Override
    public void registerBeanDefinitions(AnnotationMetadata importMeta,
                                         BeanDefinitionRegistry registry) {
        // Get annotation attributes
        Map<String, Object> attrs =
            importMeta.getAnnotationAttributes(EnableMetrics.class.getName());
        String prefix = (String) attrs.get("prefix");

        // Register a bean for each metric type
        for (MetricType type : MetricType.values()) {
            BeanDefinitionBuilder bdb = BeanDefinitionBuilder
                .genericBeanDefinition(MetricCollector.class)
                .addConstructorArgValue(prefix)
                .addConstructorArgValue(type);
            registry.registerBeanDefinition(prefix + "." + type.name(), bdb.getBeanDefinition());
        }
    }
}

// Method 3 — BeanDefinitionRegistryPostProcessor (most powerful BFPP)
@Component
public class DynamicBeanRegistrar implements BeanDefinitionRegistryPostProcessor {
    @Override
    public void postProcessBeanDefinitionRegistry(BeanDefinitionRegistry registry) {
        // Runs BEFORE all BFPPs — full control over BeanDefinition registration
        scanPluginJars().forEach(pluginClass -> {
            String beanName = pluginClass.getSimpleName();
            registry.registerBeanDefinition(beanName,
                BeanDefinitionBuilder.genericBeanDefinition(pluginClass).getBeanDefinition());
        });
    }

    @Override
    public void postProcessBeanFactory(ConfigurableListableBeanFactory factory) {}
}
✅ Best PracticeFor framework/library code that needs to register beans based on user annotations (like @EnableXxx), use ImportBeanDefinitionRegistrar — it runs at the right phase and has access to the registering annotation's attributes. For runtime dynamic plugin loading, the direct DefaultListableBeanFactory.registerBeanDefinition() approach works but is thread-safe only if called before context is published to other threads.
🎯 Google L6 AngleMyBatis Spring's @MapperScan is a textbook example: MapperScannerRegistrar implements ImportBeanDefinitionRegistrar, scans for mapper interfaces, and registers a MapperFactoryBean BeanDefinition for each. Understanding this pattern lets you build powerful Spring integrations and @Enable*-style starter libraries.

What is the difference between @Component, @Service, @Repository, @Controller and @RestController?
STEREOTYPES
▼

All Are Meta-Annotated with @Component

All stereotype annotations include @Component in their definition, so all are detected by component scanning. From a pure bean registration standpoint, they are equivalent.

// They are all equivalent for scanning:
@Target(ElementType.TYPE)
@Retention(RetentionPolicy.RUNTIME)
@Component  // <-- this is what makes @Service detectable
public @interface Service { String value() default ""; }

// The REAL difference is in additional behavior:

// @Repository — activates PersistenceExceptionTranslation
@Repository
public class UserRepositoryImpl implements UserRepository {
    // JPA PersistenceException thrown here
    // → PersistenceExceptionTranslationPostProcessor intercepts
    // → wraps in Spring's DataAccessException hierarchy
    // → calling code doesn't need to know if it's JPA, JDBC, or MongoDB
}

// Without @Repository: PersistenceException leaks out of DAO layer
// With @Repository: always DataAccessException (framework-independent)
AnnotationSemantic LayerExtra Behavior
@ComponentGeneric componentNone — just scanning detection
@ServiceBusiness/domain serviceNone — semantic only
@RepositoryData access layerException translation (JPA/JDBC → DataAccessException)
@ControllerMVC presentation layerDetected by RequestMappingHandlerMapping for @RequestMapping
@RestControllerREST API layer@Controller + @ResponseBody — all methods serialize to HTTP body
// Exception translation example:
@Repository
public class JpaUserRepository {
    @PersistenceContext EntityManager em;

    public User findById(long id) {
        // If no record: JPA throws javax.persistence.NoResultException
        // Spring translates to: org.springframework.dao.EmptyResultDataAccessException
        return em.createQuery("SELECT u FROM User u WHERE u.id = :id", User.class)
                 .setParameter("id", id).getSingleResult();
    }
}

// Caller only needs to catch Spring's exceptions:
try {
    User u = userRepo.findById(42);
} catch (EmptyResultDataAccessException e) {
    // Works regardless of whether JPA, JDBC, or MongoDB is underneath
}
✅ Best PracticeAlways use the most specific stereotype annotation for your layer. It documents architecture intent, enables IDE layer-specific navigation, and future Spring versions may add more behavior to specific stereotypes. Never use bare @Component for service or repository classes.
🎯 Google L6 Angle@Repository's exception translation is often overlooked. The goal is a DAO layer that's truly persistence-mechanism-independent: if you swap JPA for MongoDB, the exception types the service layer catches don't change. This is a concrete example of the Dependency Inversion Principle applied to exception types.

How does @Value work? Explain ${} vs #{} and the limitations.
VALUE
▼

${} — Property Placeholder

Resolved by PropertySourcesPlaceholderConfigurer (a BFPP) which looks up the key in the Environment's PropertySource chain: system properties → environment variables → application.properties → application.yml → custom sources.

@Service
public class PaymentService {
    @Value("${payment.timeout.ms:5000}")  // :5000 = default if not found
    private int timeoutMs;

    @Value("${payment.gateway.url}")  // required — throws at startup if missing
    private String gatewayUrl;

    @Value("${payment.retry.enabled:true}")
    private boolean retryEnabled;

    @Value("${payment.allowed.currencies:USD,EUR,GBP}")  // comma-separated list
    private List<String> allowedCurrencies;
}

#{} — Spring Expression Language (SpEL)

@Service
public class AnalyticsService {
    @Value("#{T(Math).PI}")  // static method call
    private double pi;

    @Value("#{systemProperties['user.home']}")  // system property via SpEL
    private String userHome;

    @Value("#{configBean.maxConnections * 2}")  // reference another bean's property
    private int doubledConnections;

    @Value("#{environment['spring.profiles.active'] == 'prod' ? 100 : 10}")  // conditional
    private int maxRetries;
}

Limitations of @Value

  • No static fields: Spring doesn't process @Value on static fields (container doesn't manage statics)
  • No runtime updates: value resolved once at bean creation — changing the property source doesn't re-inject
  • No type-safe binding: no IDE auto-complete on property keys, no validation
  • No hierarchical structure: can't bind a tree of properties to a POJO

@ConfigurationProperties — The Better Alternative for Complex Config

// application.yml
payment:
  timeout-ms: 5000
  gateway:
    url: https://api.stripe.com
    api-key: ${STRIPE_API_KEY}
  retry:
    max-attempts: 3
    backoff-multiplier: 2.0

@ConfigurationProperties(prefix = "payment")
@Validated  // enables Bean Validation on the properties class
public class PaymentProperties {
    @NotNull
    private Duration timeoutMs = Duration.ofMillis(5000);

    @Valid
    private Gateway gateway = new Gateway();

    private Retry retry = new Retry();

    // Relaxed binding: timeout-ms, TIMEOUT_MS, timeoutMs all work
    // Full type-safety, IDE auto-complete with spring-configuration-processor

    @Data
    public static class Gateway {
        @NotBlank private String url;
        @NotBlank private String apiKey;
    }
    // getters, setters...
}

@Service
public class PaymentService {
    private final PaymentProperties config;

    public PaymentService(PaymentProperties config) { this.config = config; }
    // use config.getGateway().getUrl() etc.
}
✅ Best PracticeUse @Value for simple single-property injection. Use @ConfigurationProperties for any config with more than 2-3 properties — it gives type safety, IDE auto-complete (add spring-boot-configuration-processor dependency), validation, and relaxed binding (MY_APP_TIMEOUT → myApp.timeout → my-app.timeout).
🎯 Google L6 AngleProduction config management at scale uses @ConfigurationProperties + @Validated to fail fast at startup if required config is missing or invalid. Pairing this with Spring Cloud Config Server (or Kubernetes ConfigMaps/Secrets) gives you externalized, environment-specific config with hot-reload capabilities via Spring Cloud Bus.

What is FactoryBean? How does the & prefix work? Is it still relevant?
FACTORY_BEAN
▼

The FactoryBean Contract

FactoryBean<T> is a special Spring interface. A bean implementing it acts as a factory for another bean. Three methods: T getObject() returns the product, Class<?> getObjectType() returns product's type, boolean isSingleton() whether product is cached.

The & Prefix

// ctx.getBean("connectionPool")   → returns the PRODUCT (ConnectionPool instance)
// ctx.getBean("&connectionPool")  → returns the FactoryBean ITSELF

public class ConnectionPoolFactoryBean implements FactoryBean<ConnectionPool> {
    private String jdbcUrl;
    private int maxConnections = 10;

    @Override
    public ConnectionPool getObject() {
        // Complex initialization logic — why FactoryBean exists
        ConnectionPool pool = new ConnectionPool(jdbcUrl);
        pool.warmUp(maxConnections);
        pool.validate();
        return pool;
    }

    @Override
    public Class<?> getObjectType() { return ConnectionPool.class; }

    @Override
    public boolean isSingleton() { return true; } // pool is a singleton
}

@Configuration
public class DbConfig {
    @Bean
    public ConnectionPoolFactoryBean connectionPool() {
        ConnectionPoolFactoryBean fb = new ConnectionPoolFactoryBean();
        fb.setJdbcUrl("jdbc:postgresql://localhost/db");
        fb.setMaxConnections(20);
        return fb;
    }
}

// Usage — gets ConnectionPool, not the FactoryBean
@Service
public class QueryService {
    @Autowired
    private ConnectionPool pool; // Spring calls getObject() automatically

    // To get the factory bean itself:
    @Autowired
    @Qualifier("&connectionPool")  // & prefix in qualifier
    private ConnectionPoolFactoryBean factory;
}

Historical Uses and Modern Status

Historical FactoryBean UsageModern Alternative
JndiObjectFactoryBean@Bean method with JNDI lookup
LocalSessionFactoryBean (Hibernate)Spring Boot Hibernate auto-configuration
ProxyFactoryBean (manual AOP)@Aspect + @EnableAspectJAutoProxy
MapperFactoryBean (MyBatis)Still used — @MapperScan registers it automatically
⚠ GotchaWhen Spring reports "no bean of type X found" but you know you registered one, check if it's actually a FactoryBean. The registered bean name maps to the PRODUCT type, not the FactoryBean type. Searching by FactoryBean.class type will find nothing — you need ProductClass.class or the bean name with & prefix.
🎯 Google L6 AngleFactoryBean is largely superseded by @Bean methods for application code. But knowing it explains a class of cryptic Spring error messages involving & prefixes and helps when debugging third-party integrations (MyBatis, some legacy JPA providers) that still use it internally. Understanding it also helps when reading Spring's own source code.

How do you inject a prototype-scoped bean into a singleton? Show ALL four solutions.
PROTOTYPE_IN_SINGLETON
▼

The Problem

@Component
@Scope("prototype")  // new instance per request
public class RequestHandler {
    private final String id = UUID.randomUUID().toString();
    public String getId() { return id; }
}

@Service  // singleton
public class HandlerService {
    @Autowired
    private RequestHandler handler; // PROBLEM: injected ONCE at startup
    // handler.getId() always returns same UUID — prototype semantics lost!
}

Solution 1 — ApplicationContext.getBean() (Avoid — couples to Spring)

@Service
public class HandlerService implements ApplicationContextAware {
    private ApplicationContext ctx;

    @Override
    public void setApplicationContext(ApplicationContext ctx) { this.ctx = ctx; }

    public void process() {
        RequestHandler handler = ctx.getBean(RequestHandler.class); // new each time
        handler.handle();
    }
}

Solution 2 — @Lookup Method Injection (Spring overrides via CGLIB)

@Service
public abstract class HandlerService {  // must be abstract (or non-final)
    public void process() {
        RequestHandler handler = createHandler(); // Spring intercepts
        handler.handle();
    }

    @Lookup  // Spring generates CGLIB override that calls ctx.getBean(RequestHandler.class)
    public abstract RequestHandler createHandler();
    // Or: @Lookup("requestHandler") for explicit bean name
}

Solution 3 — ObjectProvider (RECOMMENDED — cleanest, idiomatic)

@Service
public class HandlerService {
    private final ObjectProvider<RequestHandler> handlerProvider;

    public HandlerService(ObjectProvider<RequestHandler> handlerProvider) {
        this.handlerProvider = handlerProvider;
    }

    public void process() {
        RequestHandler handler = handlerProvider.getObject(); // new instance each call
        handler.handle();
    }

    // ObjectProvider bonus features:
    public void processIfAvailable() {
        handlerProvider.ifAvailable(RequestHandler::handle);
    }

    public List<String> allHandlerIds() {
        return handlerProvider.stream()
            .map(RequestHandler::getId)
            .collect(Collectors.toList());
    }
}

Solution 4 — Scoped Proxy (Transparent proxying)

@Component
@Scope(value = "prototype", proxyMode = ScopedProxyMode.TARGET_CLASS)
public class RequestHandler {
    // same as before
}

@Service
public class HandlerService {
    @Autowired
    private RequestHandler handler; // injects a PROXY
    // Each method call on handler goes through the proxy → fetches fresh prototype
    // Transparent to the caller — no special code needed!

    public void process() {
        handler.handle(); // proxy intercepts → gets new RequestHandler from container
    }
}
⚠ GotchaScoped proxy (Solution 4) has a subtle issue with prototype scope: calling multiple methods in sequence may get different RequestHandler instances for each call since each proxy invocation fetches a new prototype. If you need to work with the same prototype instance across multiple method calls in one operation, use Solution 3 (ObjectProvider) and store the result in a local variable.
✅ Best PracticeObjectProvider<T> is the recommended approach for prototype injection. It is also the best solution for optional dependencies (replaces @Autowired(required=false)), lazy initialization, and handling multiple candidates gracefully — a single injection type with multiple capabilities.
🎯 Google L6 AngleThis question tests whether you truly understand bean scopes vs bean references. The @Lookup solution is elegant but requires abstract class or non-final methods. The ObjectProvider solution is the most modern and testable — ObjectProvider can be easily mocked in tests. At L6 depth, the follow-up is: "why would you use prototype scope at all instead of just creating objects with new?" — the answer is when you want Spring to inject the prototype's OWN dependencies.

Explain @Import, ImportSelector, and ImportBeanDefinitionRegistrar. How do @Enable* annotations use this?

▼

@Import — Three Forms

// Form 1: Import a @Configuration class directly
@Import(SecurityConfig.class)
@Configuration
public class AppConfig { }

// Form 2: Import a regular @Component (not @Configuration)
@Import(MyComponent.class)
@Configuration
public class AppConfig { }

// Form 3: Import an ImportSelector or ImportBeanDefinitionRegistrar
@Import(MetricsImportSelector.class)
@Configuration

ImportSelector — Conditional Class Import

// Used by @EnableTransactionManagement, @EnableCaching, @EnableAsync, etc.
public class TransactionManagementConfigurationSelector
        implements ImportSelector {

    @Override
    public String[] selectImports(AnnotationMetadata importMeta) {
        // Read the @EnableTransactionManagement annotation attributes
        Map<String, Object> attrs = importMeta
            .getAnnotationAttributes(EnableTransactionManagement.class.getName());
        AdviceMode mode = (AdviceMode) attrs.get("mode");

        return switch (mode) {
            case PROXY -> new String[] {
                "org.springframework.transaction.annotation.ProxyTransactionManagementConfiguration"
            };
            case ASPECTJ -> new String[] {
                "org.springframework.transaction.aspectj.AspectJJtaTransactionManagementConfiguration"
            };
        };
    }
}

// Custom @Enable* using ImportSelector
@Target(ElementType.TYPE)
@Retention(RetentionPolicy.RUNTIME)
@Import(MetricsImportSelector.class)  // key line
public @interface EnableMetrics {
    String prefix() default "app";
    boolean detailed() default false;
}

public class MetricsImportSelector implements ImportSelector {
    @Override
    public String[] selectImports(AnnotationMetadata importMeta) {
        Map<String, Object> attrs =
            importMeta.getAnnotationAttributes(EnableMetrics.class.getName());
        boolean detailed = (boolean) attrs.get("detailed");
        if (detailed) {
            return new String[]{"com.example.metrics.DetailedMetricsConfig"};
        }
        return new String[]{"com.example.metrics.BasicMetricsConfig"};
    }
}

ImportBeanDefinitionRegistrar — Direct BeanDefinition Registration

// Used by @MapperScan (MyBatis), @EnableFeignClients, @EnableJpaRepositories
public class MapperScannerRegistrar implements ImportBeanDefinitionRegistrar {

    @Override
    public void registerBeanDefinitions(AnnotationMetadata importMeta,
                                         BeanDefinitionRegistry registry) {
        // 1. Read @MapperScan attributes for base packages
        AnnotationAttributes attrs = AnnotationAttributes.fromMap(
            importMeta.getAnnotationAttributes(MapperScan.class.getName()));
        String[] basePackages = attrs.getStringArray("basePackages");

        // 2. Scan for mapper interfaces in base packages
        // 3. For each interface, register a MapperFactoryBean BeanDefinition
        ClassPathMapperScanner scanner = new ClassPathMapperScanner(registry);
        scanner.doScan(basePackages);
        // Each mapper interface gets: MapperFactoryBean<MapperInterface> registered
        // Spring calls getObject() → returns MyBatis proxy implementing the interface
    }
}

DeferredImportSelector — Spring Boot's Secret Weapon

// Used by Spring Boot's auto-configuration
// Runs AFTER all regular @Configuration classes processed
// Ensures user-defined beans take precedence over auto-configured beans
public class AutoConfigurationImportSelector implements DeferredImportSelector {

    @Override
    public String[] selectImports(AnnotationMetadata metadata) {
        // Reads META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports
        // Returns list of auto-configuration class names
        // These run LAST — after all user @Configuration classes
        // @ConditionalOnMissingBean checks see user beans → skip auto-config
        return loadAutoConfigurations();
    }
}
✅ Best PracticeWhen writing a Spring starter library, always provide an @Enable* annotation that uses @Import with an ImportSelector or ImportBeanDefinitionRegistrar. This gives users explicit opt-in control and lets you conditionally register different beans based on annotation attributes — far more flexible than component scanning.
🎯 Google L6 AngleThe entire Spring ecosystem's extensibility is built on this pattern. Feign clients, JPA repositories, MyBatis mappers, Spring Security, Spring Data Redis — all use ImportBeanDefinitionRegistrar to dynamically register beans for interfaces that have no implementation class. Understanding this pattern enables you to build first-class Spring integrations, not just use them.
