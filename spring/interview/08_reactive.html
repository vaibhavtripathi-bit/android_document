What is the Reactive Streams specification? Explain the Publisher-Subscriber contract.
REACTIVE_STREAMS
▼

The Reactive Streams Specification

Reactive Streams is a standard for asynchronous stream processing with non-blocking backpressure. It defines four interfaces forming a demand-driven protocol:

  • Publisher<T> — produces items. Has one method: subscribe(Subscriber<? super T> s)
  • Subscriber<T> — consumes items. Callbacks: onSubscribe(Subscription s), onNext(T t), onError(Throwable t), onComplete()
  • Subscription — link between publisher and subscriber. Methods: request(long n) (signal demand), cancel()
  • Processor<T,R> — extends both Publisher and Subscriber; transforms a stream

The Demand-Driven Protocol

The key invariant: a Publisher may only emit items in response to demand signalled by the Subscriber. The flow is:

  1. Subscriber calls Publisher.subscribe(this)
  2. Publisher calls onSubscribe(subscription)
  3. Subscriber calls subscription.request(n) to signal it can handle n items
  4. Publisher emits at most n onNext signals
  5. Subscriber may call request(n) again for more
  6. Publisher terminates with onComplete() or onError()

Why This Matters: Preventing Overflow

Without backpressure, a fast producer sending to a slow consumer causes unbounded buffering (OOM), data loss, or producer throttling via ugly polling. Reactive Streams decouples production speed from consumption speed — the consumer controls the flow rate.

Project Reactor Implementation

Spring's reactive library implements Reactive Streams. Mono<T> = 0 or 1 item. Flux<T> = 0 to N items. Both implement Publisher<T>.

Critical: Mono and Flux are lazy. They are assembly-time descriptions of a computation pipeline. Nothing executes until subscribe() is called — analogous to Java Streams being lazy until a terminal operation.

// Custom Publisher from scratch (educational)
public class RangePublisher implements Publisher<Integer> {
    private final int start, count;
    public RangePublisher(int start, int count) { this.start = start; this.count = count; }

    @Override
    public void subscribe(Subscriber<? super Integer> subscriber) {
        subscriber.onSubscribe(new Subscription() {
            int current = start;
            int remaining = count;
            boolean cancelled = false;

            @Override
            public void request(long n) {
                if (cancelled) return;
                long toEmit = Math.min(n, remaining);
                for (long i = 0; i < toEmit; i++) {
                    subscriber.onNext(current++);
                    remaining--;
                }
                if (remaining == 0) subscriber.onComplete();
            }

            @Override
            public void cancel() { cancelled = true; }
        });
    }
}

// Testing with StepVerifier
StepVerifier.create(new RangePublisher(1, 5))
    .expectNext(1, 2, 3, 4, 5)
    .verifyComplete();

// Demand-driven: request only 2 at a time
new RangePublisher(1, 10).subscribe(new BaseSubscriber<Integer>() {
    @Override protected void hookOnSubscribe(Subscription s) { request(2); }
    @Override protected void hookOnNext(Integer value) {
        System.out.println("Got: " + value);
        if (value % 2 == 0) request(2); // request next batch after every 2nd item
    }
});
⚠ Gotcha The spec requires onNext calls to be serialized — never call onNext concurrently from multiple threads. Violating this breaks operator contracts silently. Reactor's serialize() operator enforces this if you're bridging async sources.
✅ Best Practice Never call subscribe() inside a reactive chain (nested subscription). Use flatMap instead. Nested subscribes cause untracked subscriptions that are impossible to cancel and leak resources.
🎯 Google L6 Angle At L6, interviewers expect you to articulate why the demand-driven protocol is fundamentally different from push-based models (like callbacks/futures). The key insight: Reactive Streams makes backpressure a first-class concern at the type system level. When you say "I use Reactor," you should be able to explain the full contract, not just the API surface.

Explain Mono and Flux. What are the most important operators to master?
MONO_FLUX
▼

Mono vs Flux

Mono<T>: 0 or 1 element. Think of it as a lazy, cancellable, composable CompletableFuture. Use for: single async values, empty results (Mono.empty()), or errors (Mono.error(ex)).

Flux<T>: 0 to N elements. For streams, collections, infinite sequences. Supports all reactive backpressure mechanics.

Creation Operators

OperatorUse Case
Mono.just(val)Eager — value captured at assembly time
Mono.defer(() -> ...)Lazy — supplier called per subscription. Use when value differs per subscriber
Mono.fromCallable(() -> ...)Wrap blocking call; propagates checked exceptions
Flux.fromIterable(list)Emit from collection
Flux.range(1, 10)Integer range
Flux.interval(Duration.ofSeconds(1))Periodic ticks (does not support backpressure)
Flux.generate()Stateful synchronous generation
Flux.create(sink -> ...)Bridge async/callback APIs with overflow strategy

The Critical Distinction: flatMap vs concatMap vs switchMap

// flatMap: async, concurrent, unordered — best for independent I/O
Flux.range(1, 5)
    .flatMap(i ->
        Mono.fromCallable(() -> fetchUser(i))
            .subscribeOn(Schedulers.boundedElastic()),
        4 // max concurrency: 4 parallel subscriptions
    ); // output order NOT guaranteed

// concatMap: sequential, ordered — for operations that must be serial
Flux.range(1, 5)
    .concatMap(i -> Mono.fromCallable(() -> processInOrder(i)));
// Waits for each inner publisher to complete before subscribing to next

// switchMap: cancels previous on new — for "latest wins" patterns
searchInput  // Flux<String> of keystrokes
    .switchMap(query -> searchService.search(query))
    .subscribe(this::displayResults);
// If user types "sp", "spr", "spri" fast, only "spri" search runs

Error Operators

userFlux
    .onErrorReturn(UserNotFoundException.class, UserDTO.ANONYMOUS)
    .onErrorResume(ServiceException.class, ex ->
        fallbackService.getDefaultUser()        // switch to fallback publisher
            .doOnSubscribe(s -> log.warn("Using fallback due to: {}", ex.getMessage())))
    .onErrorMap(IOException.class, ex -> new ServiceUnavailableException(ex))
    .retryWhen(Retry.backoff(3, Duration.ofMillis(100))
        .filter(ex -> ex instanceof TransientException)
        .jitter(0.3));

Key Transformation Operators

  • map — synchronous 1:1 transformation (never returns a Publisher)
  • flatMap — async 1:N, concurrent inner subscriptions
  • zipWith / Mono.zip() — combine multiple publishers, wait for all
  • merge — concurrent subscribe to multiple, interleave output
  • concat — sequential: complete first, then subscribe to second
  • combineLatest — emit when any source emits, using latest from all others
  • buffer(n) / window(n) — batch items
  • take(n) / skip(n) — limit/skip items

Side-Effect Operators (never modify the stream)

Flux.range(1, 3)
    .doOnSubscribe(s -> log.info("Subscribed"))
    .doOnNext(i -> log.info("Emitting: {}", i))
    .doOnError(e -> log.error("Error", e))
    .doOnComplete(() -> log.info("Done"))
    .log() // logs all Reactive Streams signals (debug)
    .subscribe();
⚠ Gotcha Mono.just(expensiveCall()) executes expensiveCall() eagerly at assembly time — even if nobody ever subscribes. Use Mono.defer(() -> Mono.just(expensiveCall())) or Mono.fromCallable(this::expensiveCall) for lazy execution.
✅ Best Practice Use flatMap with explicit concurrency limit (second argument) rather than default unbounded concurrency. Unbounded flatMap can overwhelm downstream services, DB connection pools, or file descriptors.
🎯 Google L6 Angle The flatMap/concatMap/switchMap distinction is a signal of real-world reactive experience. Interviewers use this to separate people who've read docs from people who've debugged out-of-order results at 2am. Know the semantics cold.

Explain backpressure in Reactor. What strategies handle a fast producer and slow consumer?
BACKPRESSURE
▼

The Problem Without Backpressure

Producer emits 1,000,000 items/sec. Consumer processes 100/sec. Without backpressure, the intermediate buffer grows by ~999,900 items/sec → OOM in seconds. This is not a hypothetical: any Kafka consumer reading faster than it can write to a DB hits this.

How Reactor Implements Backpressure

Reactor honors the Reactive Streams demand protocol. Most operators are backpressure-transparent: they pass request(n) signals upstream unchanged. The chain naturally slows when the subscriber slows its requests.

Operators That Break Backpressure

Some sources cannot be slowed: Flux.interval(), Flux.create() with certain strategies, WebSocket frames, external event streams. These require explicit overflow handling.

// Strategy 1: DROP — discard items that can't be consumed
Flux.interval(Duration.ofMillis(1))  // fires every 1ms
    .onBackpressureDrop(dropped -> log.warn("Dropped tick: {}", dropped))
    .flatMap(tick -> slowProcess(tick), 8)  // process 8 concurrently
    .subscribe();

// Strategy 2: BUFFER with bounded size — buffer up to 100, then apply LATEST
Flux.create(sink -> {
    // push items as fast as they arrive from external source
    externalSource.onData(item -> sink.next(item));
}, FluxSink.OverflowStrategy.BUFFER)
    .onBackpressureBuffer(100,
        overflow -> log.warn("Buffer overflow: {}", overflow),
        BufferOverflowStrategy.DROP_LATEST)
    .subscribe(this::process);

// Strategy 3: LATEST — keep only most recent item
Flux.interval(Duration.ofMillis(1))
    .onBackpressureLatest()
    .subscribe(i -> {
        Thread.sleep(100); // slow consumer
        process(i);
    });

// Strategy 4: ERROR — throw OverflowException immediately
Flux.range(1, 1000)
    .onBackpressureError()  // fails fast instead of silently dropping
    .subscribe(...);

Real Production Scenario: Kafka → DB

Kafka consumer (reactive) can read messages far faster than PostgreSQL can write them. Proper setup:

kafkaReceiver.receive()                    // Flux<ReceiverRecord<K,V>>
    .flatMap(record ->
        dbRepository.save(record.value())  // non-blocking R2DBC write
            .doOnSuccess(saved -> record.receiverOffset().acknowledge())
            .onErrorResume(e -> {
                log.error("Failed to save, sending to DLQ", e);
                return dlqProducer.send(record).then();
            }),
        16  // limit: max 16 concurrent DB writes (matches connection pool size)
    )
    .subscribe();

Overflow Strategies Summary

StrategyBehaviorRisk
BUFFER (unbounded)Buffer everythingOOM
BUFFER(n)Buffer up to nDrop/error when full
DROPDiscard new items when busyData loss
LATESTKeep only most recentData loss, but fresh
ERRORPropagate OverflowExceptionFails fast, no silent loss
⚠ Gotcha Flux.interval() is hot and time-driven — it does not support backpressure. If your consumer is slower than the interval, you must add an explicit backpressure strategy or items queue unboundedly. This is one of the most common reactive memory leaks.
🎯 Google L6 Angle Backpressure is THE key differentiator between reactive and plain async. In an L6 interview, articulate: "Reactive Streams makes backpressure a first-class protocol. In a Kafka→DB pipeline without backpressure, you get unbounded buffering or dropped messages. With flatMap(write, poolSize), the concurrency is bounded by your connection pool, and Kafka naturally slows to match." This demonstrates production-scale thinking.

When should you choose WebFlux over Spring MVC? Give a rigorous trade-off analysis.
MVC_VS_WEBFLUX
▼

Spring MVC Threading Model

One thread per request (Tomcat default: ~200 threads). When a request blocks — waiting for DB, HTTP call, file I/O — that thread parks and burns a slot. Under high concurrency with slow dependencies: thread pool exhausted → request queuing → latency spikes → timeouts → cascading failure.

WebFlux Threading Model

Fixed small event loop thread pool (Netty: 2 × CPU cores). Threads never block on I/O — they register callbacks and move on. One event loop thread can serve thousands of concurrent requests. Under high concurrency with slow I/O: threads stay active, request count scales to available memory (not thread count).

Choose WebFlux When

  • 10k+ concurrent connections — C10K problem. 200 Tomcat threads cannot serve 10k concurrent users. Netty can.
  • Fan-out patterns — call 5 microservices in parallel. Mono.zip() gives you 100ms instead of 500ms sequential.
  • Streaming responses — SSE, WebSocket, large CSV/JSON exports. WebFlux streams data as it arrives; MVC buffers the entire response.
  • Event-driven pipelines — Kafka → transform → SSE push. Reactive operators compose naturally.
  • Latency-sensitive at scale — tail latency degrades badly in MVC when thread pool is saturated.

Choose Spring MVC When

  • Team unfamiliar with reactive — the cognitive overhead is real and significant. Bugs are harder to find.
  • JDBC/JPA required — blocking drivers. WebFlux forces you onto boundedElastic scheduler, partially negating benefits.
  • Complex branching business logic — imperative code is easier to reason about with many conditional paths.
  • Third-party libraries don't support reactive — wrapping blocking calls adds overhead and complexity.
  • Current throughput is sufficient — premature optimization applies here. Measure first.

Hidden Costs of WebFlux

  • Stack traces are 50–100 frames deep — nearly unreadable without experience
  • Debugging requires understanding operator fusion and assembly vs subscription time
  • MDC/ThreadLocal for logging requires explicit Reactor Context wiring
  • Security context propagation needs ReactiveSecurityContextHolder
  • Transaction management is more complex (Reactor Context-based)

Performance Reality Check

ScenarioSpring MVCWebFlux
Single request latencyEqual or faster (no async overhead)Slightly higher per-request overhead
1000 concurrent users, fast DBFine (thread pool handles it)Overkill
10000 concurrent users, slow dependenciesThread pool exhausted, queuingHandles gracefully
Fan-out: 5 parallel HTTP calls500ms (sequential) or complex async100ms with Mono.zip()
Streaming 1GB fileBuffers entire response or needs special configStreams natively
⚠ Gotcha "We use WebFlux so we're reactive" is cargo cult. If your WebFlux app calls a blocking JDBC repository on the default parallel scheduler, you're blocking the Netty event loop — performance is worse than Spring MVC with the same throughput.
🎯 Google L6 Angle "Reactive is not faster per request, it's more efficient per thread under high concurrency. The decision gate is: do you have 10k+ concurrent connections and predominantly I/O-bound work? If not, MVC is simpler and your team will be more productive. Measure your p99 latency under load before migrating." This shows engineering judgment over trend-following.

WebClient Overview

WebClient is Spring's non-blocking, reactive HTTP client. It works in both Spring MVC (call .block() for synchronous result) and WebFlux (fully reactive chain). It replaces RestTemplate for new code.

Building a WebClient

// Production-grade WebClient with connection pool tuning
ConnectionProvider connectionProvider = ConnectionProvider.builder("myPool")
    .maxConnections(500)
    .maxIdleTime(Duration.ofSeconds(20))
    .maxLifeTime(Duration.ofSeconds(60))
    .pendingAcquireMaxCount(1000)
    .pendingAcquireTimeout(Duration.ofSeconds(5))
    .build();

ReactorClientHttpConnector connector = new ReactorClientHttpConnector(
    HttpClient.create(connectionProvider)
        .responseTimeout(Duration.ofSeconds(5))
        .option(ChannelOption.CONNECT_TIMEOUT_MILLIS, 3000)
);

WebClient webClient = WebClient.builder()
    .baseUrl("https://api.example.com")
    .defaultHeader(HttpHeaders.CONTENT_TYPE, MediaType.APPLICATION_JSON_VALUE)
    .defaultHeader(HttpHeaders.ACCEPT, MediaType.APPLICATION_JSON_VALUE)
    .clientConnector(connector)
    .codecs(c -> c.defaultCodecs().maxInMemorySize(10 * 1024 * 1024)) // 10MB buffer
    .filter(ExchangeFilterFunctions.basicAuthentication("user", "pass"))
    .build();

GET with Full Error Handling and Retry

Mono<UserDTO> user = webClient.get()
    .uri("/users/{id}", userId)
    .retrieve()
    .onStatus(HttpStatusCode::is4xxClientError, response ->
        response.bodyToMono(ErrorDTO.class)
            .flatMap(err -> Mono.error(new BusinessException(err.getMessage()))))
    .onStatus(HttpStatusCode::is5xxServerError, response ->
        response.bodyToMono(String.class)
            .flatMap(body -> Mono.error(new ServiceUnavailableException("Upstream error: " + body))))
    .bodyToMono(UserDTO.class)
    .timeout(Duration.ofSeconds(3))
    .retryWhen(Retry.backoff(3, Duration.ofMillis(500))
        .maxBackoff(Duration.ofSeconds(10))
        .jitter(0.5)
        .filter(ex -> ex instanceof ServiceUnavailableException)
        .doBeforeRetry(signal -> log.warn("Retrying after error: {}", signal.failure().getMessage())));

Parallel Fan-Out with Mono.zip()

// Sequential (bad): 3 × 100ms = 300ms total
UserDTO user = userClient.getUser(userId).block();
OrderDTO order = orderClient.getOrder(userId).block();
AddressDTO addr = addressClient.getAddress(userId).block();

// Parallel (correct): max(100ms, 100ms, 100ms) = ~100ms total
Mono<ProfileDTO> profile = Mono.zip(
    userClient.getUser(userId),
    orderClient.getLatestOrder(userId),
    addressClient.getPrimaryAddress(userId)
).map(tuple -> ProfileDTO.of(tuple.getT1(), tuple.getT2(), tuple.getT3()));

// With individual timeout and fallback per service
Mono<ProfileDTO> resilientProfile = Mono.zip(
    userClient.getUser(userId)
        .timeout(Duration.ofSeconds(2)),
    orderClient.getLatestOrder(userId)
        .timeout(Duration.ofSeconds(2))
        .onErrorReturn(OrderDTO.EMPTY),  // orders optional
    addressClient.getPrimaryAddress(userId)
        .timeout(Duration.ofSeconds(2))
        .onErrorReturn(AddressDTO.EMPTY)
).map(t -> ProfileDTO.of(t.getT1(), t.getT2(), t.getT3()));

Streaming Response

// Stream NDJSON — process each item as it arrives, never buffer all
Flux<OrderEvent> eventStream = webClient.get()
    .uri("/events/stream")
    .accept(MediaType.APPLICATION_NDJSON)
    .retrieve()
    .bodyToFlux(OrderEvent.class);  // deserialized per-line, not all-at-once

// Stream binary (file download without full buffering)
Flux<DataBuffer> fileStream = webClient.get()
    .uri("/files/{id}", fileId)
    .retrieve()
    .bodyToFlux(DataBuffer.class);
⚠ Gotcha exchange() (vs retrieve()) requires you to consume the body even on error — failing to do so leaks the connection. Always prefer retrieve() with onStatus() handlers. If you use exchange(), always call response.releaseBody() in error paths.
✅ Best Practice Create one WebClient instance per service (with baseUrl) and reuse it. Each builder creates a new connection pool. Creating WebClient per-request is a connection pool leak that will exhaust file descriptors under load.
🎯 Google L6 Angle The parallel fan-out pattern is the primary production argument for WebClient. At Google scale, aggregating from 5 services sequentially at 100ms each = 500ms p50. Parallel = 100ms p50. That's the difference between "acceptable" and "degraded" UX for millions of users. Know the concurrency math.

Explain Reactor Schedulers. How do you safely run blocking code in a reactive pipeline?
SCHEDULERS
▼

Schedulers Overview

SchedulerThreadsUse For
Schedulers.immediate()Current threadTesting, pass-through
Schedulers.single()1 dedicated threadSingle-threaded serial work
Schedulers.parallel()CPU-count threadsCPU-bound computation ONLY
Schedulers.boundedElastic()10×CPU max (default), expandableBlocking I/O, JDBC, legacy blocking APIs
Schedulers.fromExecutor(exec)CustomIntegration with existing thread pools

subscribeOn vs publishOn

subscribeOn(scheduler): affects where the subscription and source emission happen. Only the first subscribeOn in a chain takes effect — subsequent ones are ignored.

publishOn(scheduler): switches execution thread for all operators downstream from that point. Can appear multiple times to switch threads mid-pipeline.

// publishOn example: I/O on elastic, CPU transform on parallel
Flux.fromIterable(ids)
    .publishOn(Schedulers.boundedElastic())   // switch to elastic for blocking work
    .map(id -> blockingRepo.findById(id))      // runs on boundedElastic
    .publishOn(Schedulers.parallel())          // switch to parallel for CPU work
    .map(entity -> heavyTransform(entity))     // runs on parallel
    .publishOn(Schedulers.single())            // back to single for ordered output
    .subscribe(result -> log.info("{}", result));

Safe Blocking Code Pattern (The Correct Approach)

// CORRECT: wrap each blocking call, subscribe on boundedElastic
Flux.fromIterable(userIds)
    .flatMap(id ->
        Mono.fromCallable(() -> legacyBlockingRepo.findById(id))  // wrap blocking
            .subscribeOn(Schedulers.boundedElastic()),             // run on elastic thread
        16   // max 16 concurrent blocking calls (tune to pool size)
    )
    .subscribe(this::process);

// CORRECT: entire service call is blocking
@Service
public class LegacyIntegrationService {
    public Mono<Data> fetchFromLegacy(String id) {
        return Mono.fromCallable(() -> legacyClient.fetch(id))  // blocking call
            .subscribeOn(Schedulers.boundedElastic());
    }
}

// WRONG: blocking on parallel scheduler — starves event loop
Flux.range(1, 100)
    .flatMap(i ->
        Mono.fromCallable(() -> jdbcRepo.findById(i))
        // NO subscribeOn — runs on caller's thread (Netty event loop!)
    )
    .subscribe();

Virtual Threads Integration (Spring Boot 3.2+)

// Configure boundedElastic to use virtual threads
Schedulers.setFactory(new Schedulers.Factory() {
    @Override
    public ScheduledExecutorService newBoundedElastic(int threadCap, int queuedTaskCap,
                                                       String name, int ttlSeconds, boolean daemon) {
        // Virtual threads: near-unlimited, cheap, non-pinning with structured concurrency
        return Executors.newThreadPerTaskExecutor(
            Thread.ofVirtual().name(name + "-", 0).factory()
        );
    }
});
// Or via application.properties:
// spring.threads.virtual.enabled=true (Boot 3.2+ auto-config)
⚠ Gotcha Calling Thread.sleep(), any blocking JDBC call, or synchronized blocks on Schedulers.parallel() is catastrophic. It pins the Netty I/O thread, and since there are only 2×CPU threads, your entire application hangs. Always use boundedElastic for blocking work.
✅ Best Practice Add BlockHound in tests and CI to detect illegal blocking calls on reactive threads. It throws an exception the instant any reactive thread performs a blocking operation, catching bugs before production.
🎯 Google L6 Angle Virtual threads + boundedElastic is the modern answer for blocking integration. "With virtual threads in JDK 21 and Spring Boot 3.2, blocking on the elastic scheduler costs almost nothing — no OS thread parked, just a cheap continuation. This makes the Spring MVC vs WebFlux decision less about threading and more about the reactive composition model."

What is R2DBC? How does Spring Data R2DBC work and what are the limitations vs JPA?
R2DBC
▼

R2DBC Overview

R2DBC (Reactive Relational Database Connectivity) is a fully non-blocking alternative to JDBC. It uses async drivers that integrate with the event loop — a DB call never parks a thread. Drivers exist for: PostgreSQL (r2dbc-postgresql), MySQL (r2dbc-mysql), H2 (in-memory testing), MSSQL, Oracle, and MariaDB.

Spring Data R2DBC Repository

// Entity
@Table("users")
public class User {
    @Id private Long id;
    private String name;
    private String email;
    @Column("created_at") private LocalDateTime createdAt;
}

// Repository — same abstraction as JPA, returns reactive types
public interface UserRepository extends R2dbcRepository<User, Long> {
    Flux<User> findByEmail(String email);
    Flux<User> findByNameContainingIgnoreCase(String name);

    @Query("SELECT * FROM users WHERE created_at > :since ORDER BY created_at DESC")
    Flux<User> findRecentUsers(LocalDateTime since);

    @Modifying
    @Query("UPDATE users SET email = :email WHERE id = :id")
    Mono<Integer> updateEmail(Long id, String email);
}

// Service layer usage
@Service
@RequiredArgsConstructor
public class UserService {
    private final UserRepository userRepo;

    public Mono<User> createUser(CreateUserRequest req) {
        return userRepo.save(new User(req.getName(), req.getEmail()));
    }

    public Flux<User> findRecentUsers() {
        return userRepo.findRecentUsers(LocalDateTime.now().minusDays(7));
    }
}

Reactive Transactions

@Service
@Transactional  // works via Reactor Context propagation
public class OrderService {
    public Mono<Order> placeOrder(OrderRequest req) {
        return inventoryRepo.decrementStock(req.getProductId(), req.getQuantity())
            .then(orderRepo.save(Order.from(req)))
            .then(paymentRepo.createPendingPayment(req))
            .map(payment -> payment.getOrder());
        // If any step fails, transaction rolls back — handled by R2dbcTransactionManager
    }
}

⚠ Gotcha R2DBC has no concept of a persistence context. Every object you get back is a detached POJO. There is no dirty checking — you must explicitly call save() after modifications. Coming from JPA, forgetting this is the #1 source of "my changes aren't persisted" bugs.
✅ Best Practice For complex relationship queries (joining 3+ tables with aggregation), write explicit SQL with @Query in R2DBC. Don't fight the framework — R2DBC is SQL-first. If you need ORM-style relationship management heavily, JPA on boundedElastic might be the better choice.
🎯 Google L6 Angle "R2DBC is the right choice when you're fully committed to the reactive stack and want end-to-end non-blocking I/O. The lack of lazy loading is a feature in disguise — it forces you to think about what data you actually need (no N+1 surprises). The trade-off is more explicit code and losing the JPA relationship graph."

How do you handle errors in reactive pipelines? Compare to try-catch.
ERROR_HANDLING
▼

Why Try-Catch Doesn't Work Across Async Boundaries

In imperative code, try-catch works because the call stack is linear and synchronous. In reactive pipelines, errors propagate as onError signals through the operator chain — there's no call stack to unwind. You cannot wrap a Mono in try-catch and catch errors that occur later during subscription.

// WRONG: this catch block never fires for async errors
try {
    Mono<User> user = userService.findUser(id); // assembles pipeline, doesn't execute
    user.subscribe(this::process);              // error happens here asynchronously
} catch (UserNotFoundException e) {            // never reached
    handleError(e);
}

Reactive Error Operators

// onErrorReturn: emit a static fallback value
userService.findUser(id)
    .onErrorReturn(UserNotFoundException.class, UserDTO.ANONYMOUS)
    .onErrorReturn(UserDTO.ERROR_PLACEHOLDER);  // catch-all fallback

// onErrorResume: switch to a fallback Publisher (most powerful)
userService.findUser(id)
    .onErrorResume(UserNotFoundException.class, ex -> {
        log.warn("User {} not found, using guest profile", id);
        return Mono.just(UserDTO.ANONYMOUS);
    })
    .onErrorResume(ServiceException.class, ex -> {
        metricsService.recordError("user-service", ex);
        return cacheService.getCachedUser(id);  // fallback to cache
    })
    .onErrorResume(ex -> {
        log.error("Unexpected error fetching user {}", id, ex);
        return Mono.error(new InternalServiceException("User fetch failed", ex)); // re-wrap
    });

// onErrorMap: transform exception type (preserve error, change type)
webClient.get().uri("/users/{id}", id).retrieve()
    .bodyToMono(UserDTO.class)
    .onErrorMap(WebClientResponseException.NotFound.class,
        ex -> new UserNotFoundException("User " + id + " not found"))
    .onErrorMap(TimeoutException.class,
        ex -> new ServiceTimeoutException("User service timed out", ex));

// onErrorComplete: swallow error, complete the stream (use sparingly)
eventFlux
    .onErrorComplete(IllegalArgumentException.class); // ignore bad events, continue

// doOnError: side-effect logging (error still propagates)
userService.findUser(id)
    .doOnError(ex -> log.error("Error fetching user {}: {}", id, ex.getMessage()));

Error Handling Inside flatMap (Per-Item)

// Without per-item error handling: one failure terminates the entire stream
userIds.flatMap(id -> userService.findUser(id))  // first error kills all

// With per-item error handling: failures are isolated
userIds.flatMap(id ->
    userService.findUser(id)
        .onErrorResume(ex -> {
            log.warn("Skipping user {}: {}", id, ex.getMessage());
            return Mono.empty(); // skip this item, continue stream
        })
);

Retry Patterns

// Simple retry
mono.retry(3);

// Conditional retry with backoff
mono.retryWhen(
    Retry.backoff(3, Duration.ofMillis(200))
        .maxBackoff(Duration.ofSeconds(5))
        .jitter(0.5)
        .filter(ex -> ex instanceof TransientException)  // only retry transient errors
        .doBeforeRetry(s -> log.warn("Retry #{} after {}", s.totalRetries(), s.failure().getMessage()))
        .onRetryExhaustedThrow((spec, signal) ->
            new ServiceUnavailableException("Exhausted retries", signal.failure()))
);

Global Error Handling in WebFlux

// @ControllerAdvice works with WebFlux
@RestControllerAdvice
public class GlobalExceptionHandler {
    @ExceptionHandler(UserNotFoundException.class)
    public ResponseEntity<ErrorResponse> handleNotFound(UserNotFoundException ex) {

⚠ Gotcha An unhandled error in a subscribe() call with no error consumer defaults to throwing UnsupportedOperationException on the subscribing thread — in WebFlux this terminates the Netty worker thread. Always provide an error consumer in subscribe(consumer, errorConsumer) or let WebFlux handle the response pipeline error.
🎯 Google L6 Angle Production reactive services should have a layered error strategy: per-operator fallbacks for expected errors (not-found, timeout), per-item isolation in flatMap for stream processing, and global handlers for unexpected errors. The key question in an L6 interview: "What's your fallback when the user-service is down?" — not just "we throw an exception."

StepVerifier: Testing Publishers

StepVerifier subscribes to a publisher and asserts signals in strict order. It's the primary tool for unit-testing reactive pipelines — no need to block().

// Success scenario
StepVerifier.create(userService.findById(1L))
    .assertNext(user -> {
        assertThat(user.getId()).isEqualTo(1L);
        assertThat(user.getName()).isEqualTo("Alice");
        assertThat(user.getEmail()).endsWith("@example.com");
    })
    .verifyComplete();  // asserts onComplete and no error

// Error scenario
StepVerifier.create(userService.findById(999L))
    .expectError(UserNotFoundException.class)
    .verify();  // verifies error signal was received

// Error with message assertion
StepVerifier.create(userService.findById(999L))
    .expectErrorSatisfies(ex ->
        assertThat(ex)
            .isInstanceOf(UserNotFoundException.class)
            .hasMessageContaining("999"))
    .verify();

// Flux with multiple items
StepVerifier.create(Flux.range(1, 5).map(i -> i * 2))
    .expectNext(2, 4, 6, 8, 10)
    .verifyComplete();

// Flux: assert count without checking values
StepVerifier.create(userRepo.findAll())
    .expectNextCount(100)
    .verifyComplete();

// Flux: partial assertion
StepVerifier.create(orderService.getRecentOrders(userId))
    .assertNext(o -> assertThat(o.getStatus()).isEqualTo(PENDING))
    .assertNext(o -> assertThat(o.getStatus()).isEqualTo(CONFIRMED))
    .thenCancel()  // cancel after 2nd item instead of waiting for completion
    .verify();

Virtual Time: Testing Time-Based Operators

// Without virtual time: test would take 3 real minutes!
StepVerifier.withVirtualTime(
    () -> Flux.interval(Duration.ofMinutes(1)).take(3)
)
    .expectSubscription()
    .expectNoEvent(Duration.ofMinutes(1))  // advance time 1 min
    .expectNext(0L)
    .thenAwait(Duration.ofMinutes(1))      // advance time 1 more min
    .expectNext(1L)
    .thenAwait(Duration.ofMinutes(1))
    .expectNext(2L)
    .verifyComplete();                      // instant in real time

// Testing retry with backoff (would take seconds otherwise)
StepVerifier.withVirtualTime(() ->
    Mono.error(new TransientException("flaky"))
        .retryWhen(Retry.backoff(3, Duration.ofSeconds(1)))
)
    .expectSubscription()
    .thenAwait(Duration.ofSeconds(10))  // advance past all backoffs
    .expectError(TransientException.class)
    .verify();

How does Reactor Context work? How do you propagate security and tracing context across reactive chains?
CONTEXT
▼

The ThreadLocal Problem in Reactive Code

In Spring MVC, MDC (Mapped Diagnostic Context) and SecurityContextHolder use ThreadLocal. This works because one request = one thread throughout. In reactive pipelines, publishOn() and subscribeOn() switch threads mid-chain. The ThreadLocal on thread A is not visible on thread B — context is silently lost.

Reactor Context: The Solution

Reactor Context is an immutable key-value map that travels with the subscription upstream through the operator chain. It is per-subscription, not per-thread. Operators can read from it at subscription time.

// Writing to Context (downstream -> upstream, so write at end of chain or in subscriber)
Mono<String> mono = fetchData()
    .contextWrite(Context.of("traceId", "abc-123", "userId", 42L));

// Reading from Context (operators read ctx from downstream)
Mono<String> withContext = Mono.deferContextual(ctx -> {
    String traceId = ctx.getOrDefault("traceId", "unknown");
    log.info("[{}] Fetching data", traceId);
    return Mono.just("data for trace " + traceId);
});

// Transforming context mid-chain
Mono<User> userMono = fetchUser(id)
    .contextWrite(ctx -> ctx.put("operation", "fetch-user"));

Security Context Propagation in WebFlux

// Spring Security WebFlux stores SecurityContext in Reactor Context automatically
// Reading it in service layer:
public Mono<UserProfile> getCurrentUserProfile() {
    return ReactiveSecurityContextHolder.getContext()
        .map(SecurityContext::getAuthentication)
        .map(auth -> (UserDetails) auth.getPrincipal())
        .flatMap(userDetails -> userRepo.findByUsername(userDetails.getUsername()));
}

// @PreAuthorize works reactively with @EnableReactiveMethodSecurity
@PreAuthorize("hasRole('ADMIN')")
public Mono<List<User>> getAllUsers() {
    return userRepo.findAll().collectList();
}

Micrometer Context Propagation (Spring Boot 3+)

// Spring Boot 3 + Micrometer Context Propagation bridges ThreadLocal <-> Reactor Context
// Dependencies: micrometer-context-propagation, reactor-core (5.3.8+)

// Enable automatic propagation (usually auto-configured in Boot 3)
Hooks.enableAutomaticContextPropagation();

// Now MDC, SecurityContext, and TraceContext propagate automatically:
// - Request comes in with X-B3-TraceId header
// - Micrometer extracts trace context into Reactor Context
// - Across publishOn() switches, trace context is restored to ThreadLocal on each thread
// - MDC.get("traceId") works correctly on any thread in the pipeline
// - Logs automatically include trace ID without manual propagation

// Manual MDC propagation (pre-Boot 3 approach):
Mono<String> result = fetchData()
    .contextWrite(Context.of(
        "traceId", MDC.get("traceId"),
        "spanId", MDC.get("spanId")
    ))
    .doOnEach(signal -> {
        if (signal.isOnNext() || signal.isOnError()) {
            ContextView ctx = signal.getContextView();
            MDC.put("traceId", ctx.getOrDefault("traceId", "none"));
        }
    });
⚠ Gotcha contextWrite() applies upstream, not downstream. The context is visible to operators that appear before the contextWrite() call in the chain. This is counter-intuitive: write context at the bottom of the chain to make it available to everything above.
✅ Best Practice With Spring Boot 3 + Micrometer Context Propagation + Hooks.enableAutomaticContextPropagation(), ThreadLocal-based frameworks (MDC, SecurityContextHolder, OpenTelemetry) just work across reactive thread hops. Upgrade to Boot 3 rather than manually wiring context — it's already solved.
🎯 Google L6 Angle Context propagation is a subtle reactive pain point that trips up teams migrating from MVC. At L6, you should articulate: "Pre-Boot 3, we had to manually wire trace IDs into Reactor Context and restore them to MDC in doOnEach — fragile and error-prone. Boot 3's automatic context propagation bridge solves this at the framework level. It's one of the key reasons to upgrade."

Implement real-time Server-Sent Events and streaming with WebFlux.
SSE

Server-Sent Events (SSE) Overview

SSE is a one-directional server→client stream over a single persistent HTTP connection. The browser's native EventSource API handles reconnection automatically. Unlike WebSocket, SSE works over plain HTTP/1.1 and HTTP/2, is proxy-friendly, and doesn't require protocol upgrades.

// Controller: streaming order status updates to browser
@RestController
@RequestMapping("/api/events")
public class EventStreamController {

    @Autowired OrderEventService orderEventService;

    @GetMapping(value = "/orders/{userId}", produces = MediaType.TEXT_EVENT_STREAM_VALUE)
    public Flux<ServerSentEvent<OrderStatusDTO>> orderUpdates(
            @PathVariable Long userId,
            @AuthenticationPrincipal UserDetails user) {

        return orderEventService.getOrderUpdatesForUser(userId)
            .map(event -> ServerSentEvent.<OrderStatusDTO>builder()
                .id(String.valueOf(event.getEventId()))
                .event("order-update")
                .data(event.toDTO())
                .retry(Duration.ofSeconds(3))   // client reconnect delay
                .build())
            .mergeWith(
                // Keepalive ping every 30 seconds (prevents proxy timeouts)
                Flux.interval(Duration.ofSeconds(30))
                    .map(tick -> ServerSentEvent.<OrderStatusDTO>builder()
                        .comment("keepalive")
                        .build())
            )
            .doOnSubscribe(s -> log.info("SSE stream opened for user {}", userId))
            .doOnCancel(() -> log.info("SSE stream closed for user {}", userId));
    }
}
// Browser JavaScript client
const evtSource = new EventSource('/api/events/orders/42');
evtSource.addEventListener('order-update', (event) => {
    const order = JSON.parse(event.data);
    updateUI(order);
});
evtSource.onerror = () => console.log('SSE connection lost, auto-reconnecting...');

Streaming Large Datasets (NDJSON)

// Stream all users as NDJSON — R2DBC streams from DB, never buffers all rows in memory
@GetMapping(value = "/export/users", produces = MediaType.APPLICATION_NDJSON_VALUE)
public Flux<UserDTO> exportUsers(
        @RequestParam(required = false) String filter) {
    return userRepo.findAll()        // R2DBC: rows stream from DB cursor
        .filter(u -> filter == null || u.getName().contains(filter))
        .map(UserDTO::from)
        .doOnComplete(() -> log.info("Export stream completed"));
    // Each UserDTO is serialized and sent as a newline-delimited JSON line
    // Total memory: O(1), not O(n) regardless of result set size
}

WebSocket in WebFlux

@Component
public class ChatWebSocketHandler implements WebSocketHandler {

    @Autowired MessageService messageService;

    @Override
    public Mono<Void> handle(WebSocketSession session) {
        String userId = extractUserId(session);

        // Inbound: messages from this client
        Mono<Void> input = session.receive()
            .map(msg -> msg.getPayloadAsText())
            .flatMap(text -> messageService.broadcast(userId, text))
            .then();

        // Outbound: messages to this client
        Flux<WebSocketMessage> output = messageService.getMessagesForUser(userId)
            .map(msg -> session.textMessage(toJson(msg)));

        return session.send(output).and(input);
    }
}

// Registration
@Configuration
public class WebSocketConfig implements WebFluxConfigurer {
    @Bean
    public HandlerMapping wsHandlerMapping(ChatWebSocketHandler handler) {
        return new SimpleUrlHandlerMapping(Map.of("/ws/chat", handler), -1);
    }
}

Scatter-Gather with Timeout and Fallback

Fan out to multiple services in parallel, gather results, handle partial failures gracefully:

@Service
public class SearchAggregatorService {

    public Mono<AggregatedResult> search(SearchQuery query) {
        Mono<SearchResult> productSearch = productService.search(query)
            .timeout(Duration.ofSeconds(2))
            .onErrorReturn(SearchResult.empty("products"));

        Mono<SearchResult> reviewSearch = reviewService.search(query)
            .timeout(Duration.ofSeconds(2))
            .onErrorReturn(SearchResult.empty("reviews"));

        Mono<SearchResult> recommendationSearch = mlService.search(query)
            .timeout(Duration.ofSeconds(3))  // ML can be slower
            .onErrorReturn(SearchResult.empty("recommendations"));

        return Mono.zip(productSearch, reviewSearch, recommendationSearch)
            .map(tuple -> AggregatedResult.merge(
                tuple.getT1(), tuple.getT2(), tuple.getT3()
            ))
            .timeout(Duration.ofSeconds(4))   // overall timeout
            .doOnSuccess(r -> metrics.recordSearchLatency(r.getTotalTimeMs()));
    }
}

Circuit Breaker with Resilience4j

// Configuration
CircuitBreakerConfig cbConfig = CircuitBreakerConfig.custom()
    .slidingWindowType(COUNT_BASED)
    .slidingWindowSize(10)
    .failureRateThreshold(50f)
    .slowCallRateThreshold(30f)
    .slowCallDurationThreshold(Duration.ofSeconds(2))
    .waitDurationInOpenState(Duration.ofSeconds(30))
    .permittedNumberOfCallsInHalfOpenState(3)
    .automaticTransitionFromOpenToHalfOpenEnabled(true)
    .build();

CircuitBreaker cb = CircuitBreakerRegistry.of(cbConfig).circuitBreaker("paymentService");

// Usage in reactive pipeline
public Mono<PaymentResult> processPayment(PaymentRequest req) {
    return paymentClient.charge(req)
        .transformDeferred(CircuitBreakerOperator.of(cb))
        .onErrorResume(CallNotPermittedException.class, ex -> {
            // Circuit is OPEN — fast fail
            log.warn("Payment circuit OPEN, queueing for retry");
            return paymentQueueService.enqueue(req);  // graceful fallback
        })
        .onErrorResume(ex -> Mono.error(new PaymentServiceException(ex)));
}

// Monitoring state transitions
cb.getEventPublisher()
    .onStateTransition(event ->
        log.info("Circuit breaker '{}' transitioned from {} to {}",
            event.getCircuitBreakerName(),
            event.getStateTransition().getFromState(),
            event.getStateTransition().getToState()));

Rate Limiting

RateLimiterConfig rlConfig = RateLimiterConfig.custom()
    .limitForPeriod(100)                    // 100 calls per period
    .limitRefreshPeriod(Duration.ofSeconds(1)) // per second
    .timeoutDuration(Duration.ofMillis(100)) // wait up to 100ms for permission
    .build();

RateLimiter rateLimiter = RateLimiterRegistry.of(rlConfig).rateLimiter("externalApi");

// Apply to reactive pipeline
Flux.fromIterable(items)
    .flatMap(item ->
        processItem(item)
            .transformDeferred(RateLimiterOperator.of(rateLimiter)),
        1   // concurrency 1 ensures sequential rate-limited calls
    )
    .onErrorResume(RequestNotPermitted.class, ex -> {
        log.warn("Rate limit exceeded: {}", ex.getMessage());
        return Mono.empty();  // or Mono.error() to propagate
    });

Bulkhead: Concurrency Isolation

// Simple reactive bulkhead: limit concurrency with flatMap
public Flux<Result> processAll(Flux<Request> requests) {
    return requests
        .flatMap(req -> heavyIoOperation(req), 10); // max 10 concurrent, natural bulkhead
}

// Resilience4j Bulkhead (Semaphore-based)
BulkheadConfig bhConfig = BulkheadConfig.custom()
    .maxConcurrentCalls(20)         // max 20 simultaneous calls
    .maxWaitDuration(Duration.ofMillis(50))  // wait up to 50ms for slot
    .build();

Bulkhead bulkhead = BulkheadRegistry.of(bhConfig).bulkhead("inventoryService");

inventoryService.checkStock(sku)
    .transformDeferred(BulkheadOperator.of(bulkhead))
    .onErrorResume(BulkheadFullException.class,
        ex -> Mono.error(new ServiceBusyException("Inventory service at capacity")));

Full Resilience Composition

// Layered: TimeLimiter (innermost) → Retry → CircuitBreaker → Bulkhead (outermost)
inventoryClient.checkStock(sku)
    .transformDeferred(TimeLimiterOperator.of(timeLimiter))    // timeout per call
    .retryWhen(Retry.backoff(2, Duration.ofMillis(200))
        .filter(ex -> !(ex instanceof CallNotPermittedException)))  // don't retry if circuit open
    .transformDeferred(CircuitBreakerOperator.of(circuitBreaker))  // track failures
    .transformDeferred(BulkheadOperator.of(bulkhead));              // limit concurrency
✅ Best Practice Always pair a circuit breaker with a meaningful fallback: return cached data, queue the request, return a degraded response. A circuit breaker that just throws an exception is only half the pattern — the real value is graceful degradation when a dependency is down.
🎯 Google L6 Angle "In a reactive microservice, these patterns compose naturally as pipeline operators — no separate thread management, no aspect-oriented magic required. The key design question at L6 is not 'which pattern to use' but 'what is the right fallback when the circuit opens?' For payment, queue it. For search, return cached results. For auth, fail closed (deny). The fallback strategy is where the real engineering judgment lies."
