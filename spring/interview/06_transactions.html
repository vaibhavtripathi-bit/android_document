Explain Spring's transaction abstraction. What is PlatformTransactionManager and why does it matter?
ABSTRACTION
▼

The Core Abstraction

PlatformTransactionManager is the central Spring transaction SPI. It defines three methods:

public interface PlatformTransactionManager {
    TransactionStatus getTransaction(TransactionDefinition definition)
        throws TransactionException;
    void commit(TransactionStatus status) throws TransactionException;
    void rollback(TransactionStatus status) throws TransactionException;
}

This interface completely decouples your service code from the underlying persistence technology. Whether you are using plain JDBC, JPA/Hibernate, or a distributed JTA setup, the same @Transactional annotation works identically at the service layer.

Key Implementations

  • DataSourceTransactionManager — plain JDBC. Manages java.sql.Connection lifecycle (begin, commit, rollback). Binds connection to current thread via TransactionSynchronizationManager.
  • JpaTransactionManager — JPA/Hibernate. Manages EntityManager lifecycle. Also participates in JDBC transactions so JPA and plain JDBC code can share the same transaction.
  • JtaTransactionManager — XA distributed transactions across multiple resources. Delegates to javax.transaction.UserTransaction or TransactionManager from the JTA provider (Atomikos, Narayana, WebLogic JTA).
  • ReactiveTransactionManager — reactive pipelines (WebFlux + R2DBC). Returns Mono/Flux instead of blocking. R2dbcTransactionManager and ReactiveMongoTransactionManager implement this.

TransactionDefinition — What You Configure

  • propagation — how the tx interacts with any existing transaction (REQUIRED, REQUIRES_NEW, NESTED, etc.)
  • isolation — READ_COMMITTED, REPEATABLE_READ, SERIALIZABLE, DEFAULT
  • timeout — seconds before TransactionTimedOutException
  • readOnly — hint to driver/ORM for optimizations

TransactionStatus — Runtime State

  • isRollbackOnly() / setRollbackOnly() — flag tx for rollback without throwing
  • hasSavepoint() — true for NESTED propagation
  • isCompleted() — already committed or rolled back
  • isNewTransaction() — whether this call created a fresh tx vs joined an existing one

How @Transactional Hooks In

Spring's AOP proxy intercepts the annotated method call, creates a TransactionInterceptor, which calls getTransaction() on the configured PlatformTransactionManager, executes the method body, then calls commit() or rollback() based on whether an exception was thrown.

Programmatic Use via TransactionTemplate

@Service
public class OrderService {

    private final TransactionTemplate txTemplate;

    public OrderService(PlatformTransactionManager txManager) {
        this.txTemplate = new TransactionTemplate(txManager);
        this.txTemplate.setIsolationLevel(TransactionDefinition.ISOLATION_READ_COMMITTED);
        this.txTemplate.setTimeout(10); // 10 seconds
    }

    public Order createOrder(OrderRequest req) {
        return txTemplate.execute(status -> {
            try {
                Order order = orderRepo.save(new Order(req));
                inventoryService.reserve(req.getItems());
                return order;
            } catch (InsufficientInventoryException e) {
                status.setRollbackOnly();
                throw e;
            }
        });
    }
}
Best PracticeAlways define one PlatformTransactionManager bean per persistence layer. If you have JPA + a separate JDBC DataSource, configure a ChainedTransactionManager (or XA) explicitly. Spring Boot auto-configures JpaTransactionManager when it detects a JPA EntityManagerFactory.
Google L6 AngleThe abstraction matters most when migrating persistence tech (JDBC to JPA) or introducing distributed transactions. L6 candidates understand that the interface is a seam for testing — you can mock PlatformTransactionManager in unit tests to verify commit/rollback calls. Know that JpaTransactionManager exposes the EntityManagerFactory and can act as a DataSourceTransactionManager simultaneously — important when mixing native queries with JPA in the same tx.

Explain all 7 transaction propagation behaviors with practical examples and code.
PROPAGATION
▼

Overview Table

PropagationExisting Tx?No Existing Tx?Key Use Case
REQUIRED (default)Join itCreate newStandard service methods
REQUIRES_NEWSuspend it, create newCreate newAudit log, independent commit
SUPPORTSJoin itRun non-txRead-only queries that can tolerate both
NOT_SUPPORTEDSuspend it, run non-txRun non-txBatch ops that should bypass tx overhead
MANDATORYJoin itThrow IllegalTransactionStateExceptionInternal helpers that assume caller opened tx
NEVERThrow IllegalTransactionStateExceptionRun non-txCode that must not run inside a tx
NESTEDSavepoint inside existingCreate newTry risky sub-operation, rollback to savepoint on failure

REQUIRED — The Default

@Transactional(propagation = Propagation.REQUIRED) // same as @Transactional
public void processOrder(Order order) {
    orderRepo.save(order);
    paymentService.charge(order); // joins same tx — one commit for both
}

REQUIRES_NEW — Independent Commit

@Service
public class AuditService {

    // Even if the outer transaction rolls back, this audit record COMMITS
    @Transactional(propagation = Propagation.REQUIRES_NEW)
    public void logEvent(String event, String user) {
        auditRepo.save(new AuditEntry(event, user, Instant.now()));
    }
}

@Service
public class OrderService {

    @Transactional
    public void cancelOrder(Long orderId) {
        Order order = orderRepo.findById(orderId).orElseThrow();
        order.setStatus(OrderStatus.CANCELLED);
        orderRepo.save(order);
        // AuditService.logEvent runs in its OWN transaction — commits independently
        auditService.logEvent("ORDER_CANCELLED", order.getUserId());
        // Even if we throw here and outer tx rolls back, audit log is already committed
        throw new SomeException("Something went wrong");
    }
}
GotchaREQUIRES_NEW acquires a new DB connection from the pool while the outer tx is suspended. Under high concurrency, this can exhaust the connection pool if many threads each hold two connections simultaneously. With pool size 10 and 6 concurrent REQUIRES_NEW calls, you have 12 connections needed — deadlock waiting for pool slots.

NESTED — Savepoint Within Same TX

@Service
public class OrderService {

    @Transactional
    public Order createOrderWithOptionalDiscount(OrderRequest req) {
        Order order = orderRepo.save(new Order(req));

        try {
            discountService.applyDiscount(order); // NESTED propagation
        } catch (DiscountServiceException e) {
            // Rolls back ONLY the discount operation (to savepoint)
            // The order save above is still intact in the outer tx
            log.warn("Discount failed, continuing without discount: {}", e.getMessage());
        }

        // Outer tx commits: order is saved, discount may or may not be applied
        return order;
    }
}

@Service
public class DiscountService {
    @Transactional(propagation = Propagation.NESTED)
    public void applyDiscount(Order order) {
        // Creates a JDBC savepoint at start of this method
        discountRepo.save(new Discount(order.getId(), 0.10));
        promoRepo.markUsed(order.getPromoCode()); // if this throws -> rollback to savepoint
    }
}

MANDATORY — Enforce Caller Contract

@Transactional(propagation = Propagation.MANDATORY)
public void internalUpdate(Entity entity) {
    // Throws if called without an active transaction
    // Use for internal-only methods that MUST be called from a transactional context
    repo.save(entity);
}

NEVER — Prevent Accidental Tx Use

@Transactional(propagation = Propagation.NEVER)
public List<Report> generateHeavyReport() {
    // Long-running read that should NEVER hold a DB transaction
    // Throws IllegalTransactionStateException if called from inside a tx
    return reportRepo.findAll(bigSpec);
}
Best PracticeUse REQUIRED for 95% of cases. Use REQUIRES_NEW for audit logging, outbox event insertion, or any operation that must commit independently of the business transaction. Use NESTED (not REQUIRES_NEW) when you want inner changes to be part of the outer commit but rollback-able on inner failure. NESTED only works with DataSourceTransactionManager (plain JDBC) — JPA and JTA do not support JDBC savepoints.
Google L6 AngleAt scale, the difference between REQUIRES_NEW and NESTED is a connection pool resource question. REQUIRES_NEW adds another connection; NESTED uses a savepoint on the same connection. Know that NESTED is unsupported by most JTA providers and some databases. Be ready to discuss why overusing REQUIRES_NEW in a high-throughput service leads to pool exhaustion and connection starvation bugs that only appear under load testing.

Explain the 4 isolation levels. What concurrency problems does each solve?
ISOLATION
▼

The Three Concurrency Problems

  • Dirty Read — reading uncommitted data from another transaction (that may still roll back)
  • Non-repeatable Read — re-reading a row and getting different values because another committed transaction updated it between reads
  • Phantom Read — re-running a query and getting different rows because another committed transaction inserted/deleted rows between query executions

Isolation Levels vs Problems Prevented

Isolation LevelDirty ReadNon-Repeatable ReadPhantom ReadPerformance
READ_UNCOMMITTEDNot preventedNot preventedNot preventedFastest (no locks)
READ_COMMITTEDPreventedNot preventedNot preventedGood
REPEATABLE_READPreventedPreventedNot prevented (varies)Moderate
SERIALIZABLEPreventedPreventedPreventedSlowest

How MVCC Changes the Picture

PostgreSQL and MySQL InnoDB use MVCC (Multi-Version Concurrency Control). Readers never block writers; writers never block readers. Each transaction sees a consistent snapshot of the database as of when the transaction (or statement) started.

In PostgreSQL, READ_COMMITTED reads the latest committed version for each statement. REPEATABLE_READ and SERIALIZABLE use a snapshot taken at transaction start — no dirty or non-repeatable reads without any row-level read locks.

MySQL InnoDB's REPEATABLE_READ (its default) uses a snapshot but also adds gap locks to prevent phantom reads — going beyond the SQL standard definition.

SSI — Serializable Snapshot Isolation (PostgreSQL)

PostgreSQL's SERIALIZABLE uses SSI: tracks read/write dependencies between transactions and aborts any transaction involved in a cycle (a serialization anomaly). No predicate locks, dramatically less blocking than traditional locking-based SERIALIZABLE. Aborted transactions receive a SerializationFailure error and must be retried.

Configuring in Spring

// Read-heavy service — PostgreSQL default, good for most OLTP
@Transactional(isolation = Isolation.READ_COMMITTED)
public List<Product> listProducts() { ... }

// Wallet balance — prevent non-repeatable reads on same row
@Transactional(isolation = Isolation.REPEATABLE_READ)
public void transferFunds(Long fromId, Long toId, BigDecimal amount) {
    Account from = accountRepo.findById(fromId).orElseThrow();
    // Another transaction cannot change 'from' balance between here and the update below
    Account to = accountRepo.findById(toId).orElseThrow();
    from.debit(amount);
    to.credit(amount);
    accountRepo.saveAll(List.of(from, to));
}

// Isolation.DEFAULT = use whatever the DB is configured for (usually READ_COMMITTED)

SELECT FOR UPDATE — Pessimistic Locking

// In Spring Data JPA, combine with REPEATABLE_READ for inventory deduction
@Lock(LockModeType.PESSIMISTIC_WRITE)
@Query("SELECT p FROM Product p WHERE p.id = :id")
Optional<Product> findByIdForUpdate(@Param("id") Long id);
GotchaREAD_UNCOMMITTED is almost never the right choice. Even for analytics dashboards, use READ_COMMITTED — the marginal performance gain is not worth the risk of reading data that will be rolled back. Never use READ_UNCOMMITTED in financial or inventory contexts.
Best PracticeDefault to READ_COMMITTED (Postgres default). For inventory deduction, wallet operations, or any scenario involving "check then act" on the same data row: use REPEATABLE_READ + SELECT FOR UPDATE. For Google-scale write-heavy systems, prefer optimistic locking (@Version on entity) over pessimistic isolation levels — it avoids lock contention entirely.
Google L6 AngleAt Google scale, SERIALIZABLE is rarely used on the hot path — the retry overhead under high contention destroys throughput. Instead, design for optimistic locking at the application layer: @Version for OCC, compare-and-swap semantics, or conditional writes (DynamoDB ConditionExpression). Know the difference between isolation (ANSI SQL), consistency (CAP/PACELC), and serializability (schedule theory) — interviewers will probe all three.

When does Spring roll back? Explain rollback rules and common swallowed-exception bugs.
ROLLBACK
▼

Default Rollback Behavior

Spring follows the EJB convention: by default, a transaction rolls back on unchecked exceptions (RuntimeException and Error subclasses) and commits on checked exceptions. The rationale is that checked exceptions represent expected business outcomes (e.g., ItemNotFoundException), while unchecked exceptions represent unexpected failures.

Customizing Rollback Rules

// Roll back for ALL exceptions including checked
@Transactional(rollbackFor = Exception.class)
public void transferFunds(TransferRequest req) throws InsufficientFundsException { ... }

// Roll back for specific checked exception
@Transactional(rollbackFor = InsufficientFundsException.class)
public void reserveInventory(Long productId, int qty) throws InsufficientFundsException { ... }

// Do NOT roll back for a specific unchecked exception
@Transactional(noRollbackFor = OptimisticLockingFailureException.class)
public void updateWithRetry(Entity entity) {
    // Handle retry logic here without rolling back tx every time
}

// Multiple rules: roll back on IOException, but not on FileNotFoundException (subclass)
@Transactional(rollbackFor = IOException.class, noRollbackFor = FileNotFoundException.class)
public void processFile(String path) throws IOException { ... }

The Swallowed Exception Bug — Most Common Production Bug

// BUG: Exception is eaten, tx commits with partial state
@Transactional
public void processOrder(Order order) {
    try {
        orderRepo.save(order);
        inventoryService.deductStock(order);  // throws InsufficientStockException (RuntimeException)
        paymentService.charge(order);
    } catch (Exception e) {
        log.error("Order processing failed: {}", e.getMessage()); // exception swallowed!
        // No rethrow — Spring thinks method succeeded — TX COMMITS
        // Order is saved, stock is partially deducted, payment never charged
    }
}

Fix 1 — Rethrow the Exception

@Transactional
public void processOrder(Order order) {
    try {
        orderRepo.save(order);
        inventoryService.deductStock(order);
        paymentService.charge(order);
    } catch (Exception e) {
        log.error("Order processing failed: {}", e.getMessage());
        throw e; // Spring now sees the exception → ROLLBACK
    }
}

Fix 2 — Use TransactionStatus.setRollbackOnly()

@Service
public class OrderService {

    @Autowired
    private TransactionTemplate txTemplate;

    public OrderResult processOrder(Order order) {
        return txTemplate.execute(status -> {
            try {
                orderRepo.save(order);
                inventoryService.deductStock(order);
                paymentService.charge(order);
                return OrderResult.success(order);
            } catch (Exception e) {
                log.error("Order processing failed: {}", e.getMessage());
                status.setRollbackOnly(); // TX will roll back even without rethrow
                return OrderResult.failure(e.getMessage()); // caller gets a result object
            }
        });
    }
}

Fix 3 — Mark Rollback Without Exception via AOP Advice

// When you need to commit the outer tx but mark for rollback based on return value
@Transactional
public OrderResult processOrder(Order order) {
    orderRepo.save(order);
    InventoryResult inventoryResult = inventoryService.deductStock(order);
    if (inventoryResult.isFailed()) {
        TransactionAspectSupport.currentTransactionStatus().setRollbackOnly();
        return OrderResult.failure("Insufficient stock");
    }
    return OrderResult.success(order);
}
GotchaThe most subtle version of this bug: a service A calls service B which throws, but A's @Transactional wraps it in a try-catch and returns null or an empty Optional. The transaction commits. A code reviewer seeing the @Transactional annotation assumes the method is safe — it is not. Always grep for try-catch blocks inside @Transactional methods in production code reviews.
Best PracticeFor read-only operations, @Transactional(readOnly=true) is safe. For write operations, establish a team convention: never catch-and-swallow exceptions inside @Transactional methods. If you must return a result object instead of throwing, use setRollbackOnly() explicitly. Add a SpotBugs or SonarQube rule to flag swallowed exceptions in @Transactional methods.
Google L6 AngleThis is a correctness-at-scale issue. At Google, a swallowed exception in an order service could mean thousands of inventory deductions per minute with no corresponding payment. L6 engineers establish team coding standards, add integration tests that verify rollback behavior, and implement monitoring on partial order states. Know how to write a test that verifies rollback: save entity, call method that should rollback, verify entity count unchanged.

What does @Transactional(readOnly=true) actually do under the hood?
READONLY
▼

Hibernate Behavior — The Main Win

When readOnly=true is set, Spring passes the hint to the Hibernate Session (via Session.setDefaultReadOnly(true)) and sets FlushMode.MANUAL. This has two effects:

  1. No dirty checking snapshot — Hibernate normally takes a deep copy of every loaded entity at load time to compare against at flush time (dirty checking). With MANUAL flush mode, this snapshot is never taken. For a query loading 1000 entities with 20 fields each, this eliminates 1000 object snapshots from heap.
  2. No auto-flush before queries — Hibernate won't flush pending changes to the DB before executing a query (default behavior to ensure query sees latest state). Saves one potential flush round-trip.

JDBC/DataSource Behavior

Spring calls Connection.setReadOnly(true) on the underlying JDBC connection. The behavior depends on the driver and database:

  • PostgreSQL JDBC driver: passes a transaction mode hint. Minor optimization.
  • MySQL Connector/J: can route to read replica if using replication-aware driver.
  • Routing DataSource pattern: some DataSource wrappers (LazyConnectionDataSourceProxy + AbstractRoutingDataSource) inspect TransactionSynchronizationManager.isCurrentTransactionReadOnly() to route to read replicas. This is a significant production optimization.

Read Replica Routing Pattern

public class ReadWriteRoutingDataSource extends AbstractRoutingDataSource {

    @Override
    protected Object determineCurrentLookupKey() {
        // Routes to "read" DataSource for readOnly=true transactions
        return TransactionSynchronizationManager.isCurrentTransactionReadOnly()
                ? "read"
                : "write";
    }
}

// Configuration
@Bean
DataSource routingDataSource(DataSource primary, DataSource replica) {
    ReadWriteRoutingDataSource routing = new ReadWriteRoutingDataSource();
    routing.setTargetDataSources(Map.of("write", primary, "read", replica));
    routing.setDefaultTargetDataSource(primary);
    return routing;
}

// Service usage — automatically routes to replica
@Transactional(readOnly = true)
public List<Product> findAllProducts() {
    return productRepo.findAll();
}

What readOnly=true Does NOT Do

Critical misconception: readOnly=true is NOT a write-prevention contract. You can call save() inside a readOnly transaction. Hibernate will accept the call but silently ignore the dirty state (FlushMode.MANUAL). No exception is thrown. The data is never persisted. This is a silent data loss bug.

// SILENT DATA LOSS BUG
@Transactional(readOnly = true) // meant to be read-only, but typo on wrong method
public void updateUserProfile(Long userId, ProfileDTO dto) {
    User user = userRepo.findById(userId).orElseThrow();
    user.setName(dto.getName()); // entity state changed
    userRepo.save(user);         // Hibernate accepts but FlushMode.MANUAL = no flush
    // Method returns successfully. DB not updated. Caller has no idea.
}
Best PracticeAnnotate all repository/service query methods with @Transactional(readOnly=true). Document its purpose: performance hint, not write prevention. Use @Transactional (without readOnly) on all write methods. A typical read-heavy service achieves 10-15% throughput improvement purely from readOnly hint on query methods.
Google L6 AngleAt scale, readOnly=true on the service layer enables automatic read replica routing — distributing read traffic across multiple DB replicas without application code changes. This is a horizontal scaling lever for read-heavy services. L6 engineers also know that Hibernate's dirty checking overhead is O(n * fields) per entity — for aggregate queries loading large result sets, readOnly=true meaningfully reduces GC pressure.

How does TransactionTemplate work? Show programmatic vs declarative trade-offs.
PROGRAMMATIC
▼

TransactionTemplate Internals

TransactionTemplate wraps code in a transaction without needing an AOP proxy. It implements TransactionOperations and directly calls the PlatformTransactionManager. Key advantage: no self-invocation proxy problem — works even when calling from within the same class.

Basic Usage

@Service
public class OrderService {

    private final TransactionTemplate txTemplate;
    private final TransactionTemplate readTxTemplate;

    public OrderService(PlatformTransactionManager txManager) {
        this.txTemplate = new TransactionTemplate(txManager);
        this.txTemplate.setIsolationLevel(TransactionDefinition.ISOLATION_READ_COMMITTED);
        this.txTemplate.setTimeout(10);

        this.readTxTemplate = new TransactionTemplate(txManager);
        this.readTxTemplate.setReadOnly(true);
    }

    public Order createOrder(OrderRequest req) {
        return txTemplate.execute(status -> {
            try {
                Order order = orderRepo.save(new Order(req));
                inventoryService.deduct(order.getItems());
                return order;
            } catch (InsufficientInventoryException e) {
                status.setRollbackOnly();
                throw e;
            }
        });
    }

    // Multiple small transactions in a batch loop
    public void processBatch(List<OrderRequest> requests) {
        for (OrderRequest req : requests) {
            txTemplate.executeWithoutResult(status -> {
                // Each iteration is its own transaction — one failure doesn't fail the batch
                try {
                    processOneOrder(req);
                } catch (Exception e) {
                    log.error("Failed to process {}: {}", req.getId(), e.getMessage());
                    status.setRollbackOnly();
                }
            });
        }
    }
}

When to Use Programmatic over Declarative

ScenarioPreferReason
Standard service method@TransactionalLess boilerplate, clear intent
Fine-grained tx within a single methodTransactionTemplateMultiple short txs in one method
Batch processing loopTransactionTemplateOne tx per item, not one tx for all
Self-invocation neededTransactionTemplateNo proxy bypass problem
Dynamic tx settings at runtimeTransactionTemplateIsolation/timeout set from config
Reactive codeTransactionalOperatorReactive-native equivalent

TransactionalOperator — Reactive Equivalent

@Service
public class ReactiveOrderService {

    private final TransactionalOperator txOperator;

    public ReactiveOrderService(ReactiveTransactionManager txManager) {
        this.txOperator = TransactionalOperator.create(txManager);
    }

    public Mono<Order> createOrder(OrderRequest req) {
        return Mono.fromCallable(() -> new Order(req))
                .flatMap(orderRepo::save)
                .flatMap(order -> inventoryService.deduct(order.getItems())
                        .thenReturn(order))
                .as(txOperator::transactional); // wraps the entire Mono in a transaction
    }

    // Or use the execute() form for more control
    public Mono<Order> createOrderWithControl(OrderRequest req) {
        return txOperator.execute(status ->
            orderRepo.save(new Order(req))
                .flatMap(order -> inventoryService.deduct(order.getItems())
                        .onErrorResume(e -> {
                            status.setRollbackOnly();
                            return Mono.error(e);
                        })
                        .thenReturn(order))
        ).next();
    }
}
Best PracticeFor standard CRUD services, @Transactional is almost always the right choice. Reach for TransactionTemplate when you have genuinely complex transactional logic within a single method — particularly batch operations where you want one tx per item, or when writing to a repository directly from a non-bean context (command-line runner, test fixture).
Google L6 AngleL6 candidates know that TransactionTemplate is the escape hatch for the proxy limitation. A common interview scenario: "You need to run 10,000 records through a processing loop, each in its own transaction, but grouped in batches of 100 with a single commit per batch for performance — how?" Answer: TransactionTemplate with a counter and explicit flush. This is where declarative transactions cannot express the intent.

What is TransactionSynchronizationManager? How do you execute code exactly after a transaction commits?
TX_SYNC
▼

What TransactionSynchronizationManager Does

TransactionSynchronizationManager is a ThreadLocal-based registry that holds the current transaction's state and bound resources. It is Spring's internal infrastructure for:

  • Resource binding — ensures the same JDBC Connection / Hibernate Session is reused across multiple DAO calls within one transaction
  • Transaction synchronizations — callbacks invoked at transaction lifecycle events (beforeCommit, afterCommit, afterRollback, afterCompletion)
  • Transaction metadata — current transaction name, readOnly flag, isolation level, whether a tx is active

Registering a Synchronization Callback

@Service
public class OrderService {

    @Transactional
    public Order createOrder(OrderRequest req) {
        Order order = orderRepo.save(new Order(req));

        // Register code to run ONLY after this transaction successfully commits
        // This runs AFTER the DB commit — guaranteed delivery order
        TransactionSynchronizationManager.registerSynchronization(
            new TransactionSynchronizationAdapter() {
                @Override
                public void afterCommit() {
                    // DB row is committed — safe to send notification
                    emailService.sendOrderConfirmation(order);
                    // If emailService fails here, tx is already committed — handle gracefully
                }

                @Override
                public void afterRollback() {
                    // Optional: cleanup side effects that can't be rolled back (e.g., uploaded file)
                    fileStorage.deleteIfExists(order.getAttachmentPath());
                }

                @Override
                public void afterCompletion(int status) {
                    // STATUS_COMMITTED = 0, STATUS_ROLLED_BACK = 1, STATUS_UNKNOWN = 2
                    log.info("Transaction completed with status: {}", status);
                }
            }
        );
        return order;
    }
}

Checking Transaction State

// In a repository or utility — check if inside a real transaction
if (TransactionSynchronizationManager.isActualTransactionActive()) {
    // Bind a resource that requires the same tx
    TransactionSynchronizationManager.bindResource(dataSource, connectionHolder);
}

// Check readOnly flag for routing decisions
boolean isReadOnly = TransactionSynchronizationManager.isCurrentTransactionReadOnly();

// Get transaction name (useful in audit logging)
String txName = TransactionSynchronizationManager.getCurrentTransactionName();

How Resource Binding Works (JDBC Connection Reuse)

// Spring internals (simplified) — how one Connection serves multiple DAOs in one tx
// DataSourceTransactionManager.doBegin():
Connection conn = dataSource.getConnection();
TransactionSynchronizationManager.bindResource(dataSource, new ConnectionHolder(conn));

// DataSourceUtils.getConnection() (used by JdbcTemplate, JDBC):
ConnectionHolder holder = (ConnectionHolder) TransactionSynchronizationManager.getResource(dataSource);
if (holder != null) {
    return holder.getConnection(); // REUSES same connection for the whole tx
}
return dataSource.getConnection(); // new connection if no active tx

@TransactionalEventListener — Declarative Alternative

@Component
public class OrderEventHandler {

    // AFTER_COMMIT is the default phase
    @TransactionalEventListener(phase = TransactionPhase.AFTER_COMMIT)
    public void onOrderCreated(OrderCreatedEvent event) {
        emailService.sendConfirmation(event.getOrder());
    }

    @TransactionalEventListener(phase = TransactionPhase.AFTER_ROLLBACK)
    public void onOrderFailed(OrderCreatedEvent event) {
        alertService.notifyFailure(event.getOrder());
    }

    // fallbackExecution = true: also fire when there is no active transaction
    @TransactionalEventListener(phase = TransactionPhase.AFTER_COMMIT, fallbackExecution = true)
    public void onUserRegistered(UserRegisteredEvent event) {
        welcomeEmailService.send(event.getUser());
    }
}

// Publishing the event — inside @Transactional method
@Transactional
public Order createOrder(OrderRequest req) {
    Order order = orderRepo.save(new Order(req));
    eventPublisher.publishEvent(new OrderCreatedEvent(order)); // deferred until AFTER_COMMIT
    return order;
}
GotchaIf registerSynchronization() is called when no transaction is active, it throws IllegalStateException. Always check isActualTransactionActive() first, or use @TransactionalEventListener(fallbackExecution=true) for resilient handling. Also: exceptions in afterCommit() do NOT trigger rollback — the tx is already committed. Handle exceptions in the synchronization callback gracefully.
Best PracticeUse @TransactionalEventListener(phase=AFTER_COMMIT) as the declarative approach for post-commit side effects (emails, push notifications, cache eviction). Use registerSynchronization() programmatically only when you cannot use events (e.g., inside a repository, or when you need the specific object reference that isn't easily serialized to an event).
Google L6 AngleThe "send notification after commit" pattern is a classic correctness question. At L6, you explain not just the mechanism but the guarantees: AFTER_COMMIT is at-most-once delivery. If the JVM crashes after DB commit but before the email sends, the email is lost. For guaranteed delivery of critical notifications, combine @TransactionalEventListener for local events with the Transactional Outbox pattern for durable delivery.

Explain XA/2PC distributed transactions. Why are they problematic at scale and what are the alternatives?
DISTRIBUTED
▼

What XA/2PC Provides

XA is a protocol for coordinating a single ACID transaction across multiple independent resource managers (DB1 + DB2 + JMS broker + etc.). The Transaction Coordinator executes a Two-Phase Commit:

  1. Phase 1 — PREPARE: Coordinator sends PREPARE to all participants. Each writes an "I will commit" entry to its durable log (redo log), responds YES or NO.
  2. Phase 2 — COMMIT or ROLLBACK: If all participants voted YES → coordinator sends COMMIT to all. If any voted NO → sends ROLLBACK to all. Each participant executes the directive.

Why XA is Problematic at Scale

  1. Blocking protocol: Resources hold locks from PREPARE until COMMIT/ROLLBACK arrives. If the coordinator crashes between Phase 1 and Phase 2, all participants are stuck in PREPARED state holding locks indefinitely (in-doubt transactions). Recovery requires manual intervention or a recovery daemon.
  2. Performance overhead: Two network round-trips plus two disk flushes (coordinator log + participant redo logs). At 1000 tx/sec this adds meaningful latency compared to single-resource transactions.
  3. Cloud databases don't support it: DynamoDB, Cosmos DB, Redis, BigTable, and most cloud-native databases have no XA support. Your microservices architecture will inevitably include resources that cannot participate in XA.
  4. Tight coupling: All participating services must be reachable and healthy at commit time. Partial availability = entire operation fails or is stuck.
  5. Scalability cap: The coordinator is a single point of serialization. At Google-scale transaction rates, the coordinator becomes a bottleneck regardless of how many replicas you run.

The correct mental model: design for eventual consistency and compensating transactions from the start.

For most applications, design your services to own their data (database-per-service) and express cross-service consistency through events + compensations.

Explain the Saga pattern. Compare choreography vs orchestration with real examples.
SAGA
▼

What is a Saga?

A Saga is a sequence of local database transactions where each step publishes an event or message that triggers the next step. If any step fails, compensating transactions are executed in reverse order to undo completed steps. Sagas achieve eventual consistency across microservices without distributed locking.

Key insight: there is no global rollback — you must design explicit compensating actions for each forward step that might need to be undone.

Order Placement Saga Steps

StepActionCompensation
1Create Order (PENDING)Cancel Order
2Reserve InventoryRelease Inventory
3Charge PaymentRefund Payment
4Confirm Order (CONFIRMED)(final step, no compensation)

Choreography — Event-Driven, No Central Controller

Each service listens to events and emits its own events. Services are decoupled and don't know about each other.

// Order Service — publishes event after local tx commits (Outbox pattern)
@Transactional
public Order createOrder(OrderRequest req) {
    Order order = orderRepo.save(Order.pending(req));
    eventPublisher.publish(new OrderCreatedEvent(order.getId(), order.getItems()));
    return order;
}

// Inventory Service — consumes OrderCreated, reserves stock
@KafkaListener(topics = "order-created")
@Transactional
public void onOrderCreated(OrderCreatedEvent event) {
    try {
        inventoryService.reserve(event.getOrderId(), event.getItems());
        kafkaTemplate.send("inventory-reserved", new InventoryReservedEvent(event.getOrderId()));
    } catch (InsufficientStockException e) {
        kafkaTemplate.send("inventory-reservation-failed",
            new InventoryReservationFailedEvent(event.getOrderId(), e.getMessage()));
    }
}

// Payment Service — consumes InventoryReserved, charges customer
@KafkaListener(topics = "inventory-reserved")
@Transactional
public void onInventoryReserved(InventoryReservedEvent event) {
    try {
        paymentService.charge(event.getOrderId());
        kafkaTemplate.send("payment-processed", new PaymentProcessedEvent(event.getOrderId()));
    } catch (PaymentFailedException e) {
        // Publish failure — Inventory Service will compensate by releasing reservation
        kafkaTemplate.send("payment-failed", new PaymentFailedEvent(event.getOrderId()));
    }
}

// Inventory Service — compensates on payment failure
@KafkaListener(topics = "payment-failed")
@Transactional
public void onPaymentFailed(PaymentFailedEvent event) {
    inventoryService.releaseReservation(event.getOrderId()); // compensating transaction
    kafkaTemplate.send("inventory-released", new InventoryReleasedEvent(event.getOrderId()));
}

// Order Service — cancels order when inventory released after failure
@KafkaListener(topics = "inventory-released")
@Transactional
public void onInventoryReleased(InventoryReleasedEvent event) {
    orderRepo.updateStatus(event.getOrderId(), OrderStatus.CANCELLED);
}

Orchestration — Central Saga Orchestrator

A dedicated Saga Orchestrator calls each service in sequence and handles failures by calling compensations. Central visibility. Logic in one place.

@Component
public class OrderSagaOrchestrator {

    @Autowired InventoryServiceClient inventoryClient;
    @Autowired PaymentServiceClient paymentClient;
    @Autowired OrderRepository orderRepo;

    @Transactional
    public void execute(OrderSaga saga) {
        try {
            // Step 1: Reserve inventory
            inventoryClient.reserve(saga.getOrderId(), saga.getItems());
            saga.markInventoryReserved();

            // Step 2: Charge payment
            paymentClient.charge(saga.getOrderId(), saga.getAmount());
            saga.markPaymentProcessed();

            // Step 3: Confirm order
            orderRepo.updateStatus(saga.getOrderId(), OrderStatus.CONFIRMED);

        } catch (Exception e) {
            compensate(saga, e);
        }
    }

    private void compensate(OrderSaga saga, Exception reason) {
        if (saga.isPaymentProcessed()) {
            paymentClient.refund(saga.getOrderId());
        }
        if (saga.isInventoryReserved()) {
            inventoryClient.release(saga.getOrderId());
        }
        orderRepo.updateStatus(saga.getOrderId(), OrderStatus.FAILED);
    }
}

Explain the Transactional Outbox pattern. How does it guarantee at-least-once event delivery?
OUTBOX
▼

The Problem

In a microservice, you often need to: (1) save a state change to the DB, and (2) publish an event to a message broker (Kafka/RabbitMQ). These are two separate systems — you cannot do both atomically without XA. The naive approaches both fail:

  • Publish first, then save: app crashes after Kafka publish but before DB save → event published, data inconsistent.
  • Save first, then publish: app crashes after DB commit but before Kafka publish → data saved, event never delivered. Silent data loss.

The Outbox Solution

Write both the business entity AND the outbox event record in the same DB transaction. A separate relay process reads the outbox table and publishes to Kafka, then marks records as published.

Schema and Implementation

-- Outbox table schema
CREATE TABLE outbox_event (
    id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    aggregate_type VARCHAR(100) NOT NULL,    -- 'Order', 'Payment'
    aggregate_id   VARCHAR(100) NOT NULL,    -- entity ID
    event_type     VARCHAR(100) NOT NULL,    -- 'OrderCreated', 'PaymentProcessed'
    payload        JSONB         NOT NULL,   -- serialized event body
    status         VARCHAR(20)   NOT NULL DEFAULT 'PENDING',
    created_at     TIMESTAMPTZ   NOT NULL DEFAULT NOW(),
    published_at   TIMESTAMPTZ
);
CREATE INDEX idx_outbox_pending ON outbox_event (created_at) WHERE status = 'PENDING';
// Service — atomic: business entity + outbox record in same tx
@Service
public class OrderService {

    @Transactional
    public Order createOrder(OrderRequest req) {
        Order order = orderRepo.save(new Order(req));

        // Same DB transaction — both commit or both rollback
        OutboxEvent event = OutboxEvent.builder()
            .aggregateType("Order")
            .aggregateId(order.getId().toString())
            .eventType("OrderCreated")
            .payload(objectMapper.writeValueAsString(new OrderCreatedPayload(order)))
            .status("PENDING")
            .build();
        outboxRepo.save(event);

        return order;
    }
}
// Scheduled relay — polls outbox and publishes to Kafka
@Component
public class OutboxRelay {

    @Scheduled(fixedDelay = 500) // every 500ms
    @Transactional
    public void relay() {
        List<OutboxEvent> pending = outboxRepo.findByStatusOrderByCreatedAt("PENDING", Pageable.ofSize(100));

        for (OutboxEvent event : pending) {
            try {
                kafkaTemplate.send(
                    event.getEventType().toLowerCase().replace("_", "-"),
                    event.getAggregateId(),
                    event.getPayload()
                ).get(5, TimeUnit.SECONDS); // block to confirm delivery

                event.setStatus("PUBLISHED");
                event.setPublishedAt(Instant.now());
                outboxRepo.save(event);

            } catch (Exception e) {
                log.error("Failed to publish outbox event {}: {}", event.getId(), e.getMessage());
                // Leave as PENDING — will be retried next poll cycle
            }
        }
    }
}

At-Least-Once Delivery and Idempotency

The relay may publish a message to Kafka successfully but crash before marking it PUBLISHED. On restart, it will publish the same message again. Consumers MUST be idempotent. Common strategies:

  • Include event ID in message — consumer checks processed_event_ids table before processing
  • Use Kafka's idempotent producer + transaction log (exactly-once within Kafka ecosystem)
  • Make the consumer's action naturally idempotent (e.g., INSERT ON CONFLICT DO NOTHING)

Debezium CDC — Production-Grade Relay

Debezium reads the PostgreSQL WAL (Write-Ahead Log) — zero additional DB load, no polling, sub-second latency, and no missed events even under high write load. The polling approach is simpler to start with but suffers from polling latency and increased DB load at scale.
Google L6 AngleThe Transactional Outbox is the standard pattern for reliable event-driven microservices at scale. L6 candidates explain: (1) why polling is not ideal at scale (DB load, latency), (2) why Debezium CDC is superior (reads WAL, zero overhead), (3) that consumers must be idempotent regardless, and (4) that the outbox table needs periodic cleanup of PUBLISHED records. Also discuss: what happens if the relay lags — how does backpressure affect downstream consumers?

@TransactionalEventListener vs direct @EventListener — explain internals and failure scenarios.
TX_EVENT
▼

@EventListener — Synchronous, Immediate

@EventListener fires synchronously within the same call stack, regardless of any active transaction. The listener runs BEFORE the transaction commits. If the listener throws, it propagates up and may trigger a rollback of the outer transaction.

@EventListener
public void onOrderCreated(OrderCreatedEvent event) {
    // Fires immediately when publishEvent() is called — BEFORE tx commits
    // If emailService throws here, the outer @Transactional method rolls back
    emailService.send(event.getOrder()); // Dangerous: send email but tx may still rollback!
}

@TransactionalEventListener — Bound to Transaction Lifecycle

@Component
public class OrderEventHandler {

    // Fires AFTER the outer transaction commits — DB row guaranteed to be visible
    @TransactionalEventListener(phase = TransactionPhase.AFTER_COMMIT)
    public void onOrderCreated(OrderCreatedEvent event) {
        emailService.sendConfirmation(event.getOrder()); // DB is committed — safe
    }

    // Fires BEFORE commit — can read DB but cannot write (will be included in same commit)
    @TransactionalEventListener(phase = TransactionPhase.BEFORE_COMMIT)
    public void beforeOrderCommit(OrderCreatedEvent event) {
        validationService.finalValidate(event.getOrder());
    }

    // Fires after rollback — for cleanup
    @TransactionalEventListener(phase = TransactionPhase.AFTER_ROLLBACK)
    public void onOrderFailed(OrderCreatedEvent event) {
        cleanupService.rollbackSideEffects(event.getOrder());
    }

    // Fires after either commit or rollback — for metrics
    @TransactionalEventListener(phase = TransactionPhase.AFTER_COMPLETION)
    public void onOrderComplete(OrderCreatedEvent event) {
        metricsService.recordAttempt(event.getOrder());
    }

    // fallbackExecution = true: also fires when no active transaction
    @TransactionalEventListener(phase = TransactionPhase.AFTER_COMMIT, fallbackExecution = true)
    public void onUserRegistered(UserRegisteredEvent event) {
        welcomeEmailService.send(event.getUser());
    }
}

Internals — How It Works

When ApplicationEventPublisher.publishEvent() is called inside a @Transactional method, TransactionalEventListenerFactory detects the @TransactionalEventListener annotation and registers a TransactionSynchronization instead of immediately invoking the listener. The synchronization's afterCommit() / afterRollback() / afterCompletion() methods invoke the listener at the appropriate phase.

No Transaction Case

// This event is published from a non-transactional context
public void notifyUser(User user) { // no @Transactional
    eventPublisher.publishEvent(new UserRegisteredEvent(user));
    // Default behavior: @TransactionalEventListener is SILENTLY DROPPED
    // because there is no transaction to bind the synchronization to
    // Fix: use fallbackExecution = true on the listener
}

Failure in Listener After Commit

@TransactionalEventListener(phase = TransactionPhase.AFTER_COMMIT)
public void onOrderCreated(OrderCreatedEvent event) {
    try {
        emailService.sendConfirmation(event.getOrder());
    } catch (MailException e) {
        // Cannot rollback — tx is already committed
        // Exception propagates to the caller of the original @Transactional method
        // Log the error, maybe publish to a dead-letter queue
        log.error("Failed to send order confirmation for {}", event.getOrder().getId(), e);
        deadLetterService.enqueue(event); // for manual retry
        // Do NOT rethrow unless you want to fail the response to the client
    }
}

@Async + @TransactionalEventListener

// Async listener — fires in thread pool thread after commit
// No transaction context in the async thread
// Must start a new transaction explicitly if needed
@Async
@TransactionalEventListener(phase = TransactionPhase.AFTER_COMMIT)
@Transactional(propagation = Propagation.REQUIRES_NEW) // explicit new tx for DB reads
public void onOrderCreatedAsync(OrderCreatedEvent event) {
    // Runs in async thread — thread pool
    // @Transactional(REQUIRES_NEW) starts fresh tx here if needed
    emailService.sendConfirmation(event.getOrder());
}

Explain JDBC savepoints and NESTED propagation. When would you use NESTED over REQUIRES_NEW?
SAVEPOINTS
▼

JDBC Savepoints

A savepoint is a named point within a transaction to which you can roll back without rolling back the entire transaction. JDBC API:

Connection conn = dataSource.getConnection();
conn.setAutoCommit(false);

// Phase 1: Insert order
PreparedStatement orderStmt = conn.prepareStatement("INSERT INTO orders ...");
orderStmt.executeUpdate();

// Create savepoint before risky operation
Savepoint savepointBeforeDiscount = conn.setSavepoint("BEFORE_DISCOUNT");

try {
    // Phase 2: Try to apply discount (risky)
    PreparedStatement discountStmt = conn.prepareStatement("INSERT INTO discounts ...");
    discountStmt.executeUpdate();
} catch (SQLException e) {
    // Roll back ONLY to the savepoint — order insert is preserved
    conn.rollbackToSavepoint(savepointBeforeDiscount);
    log.warn("Discount failed, continuing without it");
} finally {
    conn.releaseSavepoint(savepointBeforeDiscount); // release when no longer needed
}

conn.commit(); // commits order (and discount if it succeeded)

NESTED Propagation in Spring

Spring's NESTED propagation uses JDBC savepoints internally. The inner transaction is demarcated by a savepoint within the outer transaction. There is only ONE DB transaction in total.

@Service
public class OrderProcessingService {

    @Transactional // OUTER transaction
    public ProcessResult processOrder(Order order) {
        // Step 1: Save order — always succeeds or outer tx fails
        orderRepo.save(order);

        // Step 2: Try premium discount — uses NESTED
        boolean discountApplied = false;
        try {
            discountService.applyPremiumDiscount(order); // NESTED
            discountApplied = true;
        } catch (DiscountServiceException e) {
            // Savepoint rolled back — order.save() still in outer tx
            log.warn("Premium discount unavailable: {}", e.getMessage());
        }

        // Step 3: Charge payment — always happens
        paymentService.charge(order);

        return new ProcessResult(order, discountApplied);
        // OUTER tx commits: order + (discount if succeeded) + payment
    }
}

@Service
public class DiscountService {
    @Transactional(propagation = Propagation.NESTED)
    public void applyPremiumDiscount(Order order) {
        // Spring creates JDBC savepoint here
        Discount discount = discountRepo.findBestFor(order.getCustomerId());
        if (discount == null) throw new DiscountServiceException("No discount available");
        order.applyDiscount(discount);
        discountUsageRepo.increment(discount.getId());
        // If any exception: rollback to savepoint — outer tx continues
    }
}

When to Choose NESTED Over REQUIRES_NEW

  • Inner changes should be part of the outer commit (same DB transaction visibility)
  • Connection pool pressure is a concern (high concurrency)
  • Using DataSourceTransactionManager (plain JDBC) — JPA is a blocker for NESTED
  • The "try a risky operation but keep outer tx" pattern
  • Use REQUIRES_NEW when inner operation must commit independently (audit, idempotency keys)
  • Use REQUIRES_NEW when you use JPA (NESTED not supported)
GotchaNESTED is only supported by DataSourceTransactionManager. If you use JpaTransactionManager (which is the Spring Boot default when JPA is on the classpath), @Transactional(propagation=NESTED) will throw NestedTransactionNotSupportedException. Check your transaction manager before reaching for NESTED. Most real-world Spring apps use JPA — meaning NESTED is often unavailable.
Google L6 AngleThe interviewer tests whether you know the subtle difference: NESTED changes aren't visible outside the outer transaction until it commits. REQUIRES_NEW changes are immediately visible. This matters for concurrent readers: with REQUIRES_NEW, another thread can see the inner commit before the outer commits, which can cause inconsistent reads. NESTED prevents this race condition. Know the trade-offs deeply, not just the surface behavior.

How does @Transactional(timeout=N) work? What happens when it fires?
TIMEOUT
▼

Timeout Mechanics

The timeout attribute specifies the maximum number of seconds the transaction is allowed to run. After N seconds, a TransactionTimedOutException is thrown — an unchecked exception that triggers a rollback by default.

// Transaction must complete within 10 seconds or it is rolled back
@Transactional(timeout = 10)
public Order processComplexOrder(OrderRequest req) {
    // Clock starts here — when the proxy intercepts the method call
    Order order = orderRepo.save(new Order(req));           // 1 second
    inventoryService.reserveFromMultipleWarehouses(req);    // 5 seconds
    shippingService.calculateRoutes(req.getAddress());      // 3 seconds
    // Total: 9 seconds. Just under the limit.
    paymentService.charge(req);                             // Would fail if >1s left
    return order;
}

How the Timeout is Applied

The implementation depends on the transaction manager:

  • DataSourceTransactionManager / JpaTransactionManager: calculates remaining time and calls Statement.setQueryTimeout(remainingSeconds) before each JDBC query execution. The DB driver cancels the query after the timeout period.
  • JtaTransactionManager: delegates to JTA provider which enforces tx timeout at the provider level (Atomikos, Narayana each have their own enforcement).

Sequence of Events on Timeout

// 1. Clock starts when TransactionInterceptor calls getTransaction()
// 2. Long-running query is executing...
// 3. Statement.getQueryTimeout() triggers at N seconds
// 4. DB driver sends query cancel to database (PostgreSQL: pg_cancel_backend())
// 5. JDBC driver throws SQLException: "canceling statement due to statement timeout"
// 6. Spring wraps this in TransactionTimedOutException (RuntimeException)
// 7. TransactionInterceptor catches, calls txManager.rollback()
// 8. Connection returned to pool
// 9. TransactionTimedOutException propagates to caller

Defensive Timeout Pattern

// Layer multiple timeouts for defense-in-depth

public Order placeOrder(OrderRequest req) {
    // Query-level timeout (5 seconds per individual query)
    TypedQuery<Inventory> query = em.createQuery("...", Inventory.class);
    query.setHint("javax.persistence.query.timeout", 5000);
    List<Inventory> inventory = query.getResultList();

    // Business logic...
    return orderRepo.save(new Order(req));
}
Best PracticeSet transaction timeouts on all write transactions in production. A runaway transaction holding locks can cascade into deadlocks across your service. Common values: 5-30 seconds depending on operation complexity. Combine with query-level timeouts for fine-grained control. Monitor TransactionTimedOutException occurrences as a leading indicator of performance degradation.
Google L6 AngleTransaction timeout is a circuit breaker against runaway transactions holding DB locks. At Google scale, one slow transaction holding row-level locks can create a queue of waiting transactions, causing latency spikes across the entire service. The root fix is always to optimize the slow operation. The timeout is the safety net. L6 engineers also implement distributed tracing (OpenTelemetry) to diagnose which operations are consuming transaction time.

How do transactions work in reactive Spring (WebFlux + R2DBC)?
REACTIVE_TX
▼

Why Imperative Transactions Break in Reactive

Imperative Spring transactions use ThreadLocal to bind the active transaction (and JDBC connection) to the current thread. Reactive pipelines use an event loop and hop between threads in a thread pool — by the time a subsequent operator runs, the thread has changed and the ThreadLocal is empty. The transaction is lost.

Reactive Transaction Infrastructure

Spring 5.2+ introduces a reactive transaction stack that uses Reactor Context (propagated through the reactive chain, not the thread) to carry transaction state:

  • ReactiveTransactionManager — reactive SPI (returns Mono<ReactiveTransaction>)
  • R2dbcTransactionManager — for R2DBC (reactive JDBC replacement)
  • ReactiveMongoTransactionManager — for reactive MongoDB
  • TransactionalOperator — programmatic reactive transaction demarcation

Declarative @Transactional on Reactive Methods

@Service
public class ReactiveOrderService {

    @Autowired R2dbcOrderRepository orderRepo;
    @Autowired R2dbcInventoryRepository inventoryRepo;

    // @Transactional on a method returning Mono/Flux — Spring detects reactive return type
    // Uses ReactiveTransactionManager via Reactor Context propagation
    @Transactional
    public Mono<Order> createOrder(OrderRequest req) {
        return orderRepo.save(new Order(req))
                .flatMap(order ->
                    inventoryRepo.deductStock(req.getItems())
                        .thenReturn(order)
                )
                // If deductStock fails with an exception: entire reactive chain rolls back
                .doOnError(e -> log.error("Order creation failed: {}", e.getMessage()));
    }

    // readOnly = true on reactive — R2DBC passes hint to driver
    @Transactional(readOnly = true)
    public Flux<Order> findOrdersByUser(Long userId) {
        return orderRepo.findByUserId(userId);
    }
}

Programmatic — TransactionalOperator

@Service
public class ReactivePaymentService {

    private final TransactionalOperator txOperator;

    public ReactivePaymentService(ReactiveTransactionManager txManager) {
        DefaultTransactionDefinition def = new DefaultTransactionDefinition();
        def.setIsolationLevel(TransactionDefinition.ISOLATION_READ_COMMITTED);
        def.setTimeout(10);
        this.txOperator = TransactionalOperator.create(txManager, def);
    }

    public Mono<Payment> processPayment(PaymentRequest req) {
        return txOperator.execute(tx ->
            accountRepo.findById(req.getAccountId())
                .flatMap(account -> {
                    if (account.getBalance().compareTo(req.getAmount()) < 0) {
                        tx.setRollbackOnly(); // mark for rollback without exception
                        return Mono.error(new InsufficientFundsException());
                    }
                    account.debit(req.getAmount());
                    return accountRepo.save(account);
                })
                .flatMap(account -> paymentRepo.save(new Payment(req, account)))
                .onErrorResume(e -> {
                    tx.setRollbackOnly();
                    return Mono.error(e);
                })
        ).next();
    }

    // Shorter form using .as(txOperator::transactional)
    public Mono<Order> createOrderSimple(OrderRequest req) {
        return orderRepo.save(new Order(req))
                .flatMap(order -> inventoryRepo.reserve(order).thenReturn(order))
                .as(txOperator::transactional);
    }
}

Critical Mixing Pitfall

// DO NOT MIX R2DBC reactive transactions with blocking JPA in same reactive chain!
@Transactional // R2dbcTransactionManager
public Mono<Order> createOrder(OrderRequest req) {
    return orderRepo.save(new Order(req))  // R2DBC reactive
        .flatMap(order -> Mono.fromCallable(() ->
            jpaInventoryService.deduct(order)  // BLOCKING JPA call — different tx manager!
            // JPA uses JpaTransactionManager with DIFFERENT ThreadLocal-based tx
            // Not part of the R2DBC transaction — atomicity is broken!
        ));
}
GotchaYou cannot mix R2DBC reactive transactions with JDBC/JPA imperative transactions in the same atomic unit. If your service uses reactive with R2DBC but calls a JDBC-based dependency, each has its own independent transaction. Keep your persistence stack pure: either fully reactive (R2DBC + WebFlux) or fully imperative (JPA + Spring MVC).
Best PracticeFor most business applications with complex relational queries and mature tooling requirements, Spring MVC + JPA remains the more battle-tested stack. Choose WebFlux + R2DBC only when you have specific high-concurrency I/O requirements (many concurrent slow external calls, streaming, SSE). The reactive transaction API is still maturing compared to the imperative stack.
Google L6 AngleAt Google scale, reactive is often used for gateway/fan-out services (calls many backends simultaneously) rather than data-heavy CRUD. The transaction semantics question becomes: if each downstream service has its own DB, are you actually doing distributed transactions? No — you are doing Sagas. The L6 insight: reactive transactions are a local concern; cross-service consistency requires the same Saga/Outbox patterns regardless of reactive or imperative.

Bug 1: Self-Invocation Proxy Bypass

// BUG: methodB is NOT transactional when called from methodA
@Service
public class OrderService {

    @Transactional
    public void methodA() {
        orderRepo.save(order1);
        this.methodB(); // calls on 'this', NOT on the proxy — @Transactional on methodB is IGNORED
    }

    @Transactional(propagation = Propagation.REQUIRES_NEW)
    public void methodB() {
        orderRepo.save(order2); // runs in methodA's transaction, NOT a new one!
    }
}

// FIX 1: Extract to a separate bean
@Service
public class OrderHelperService {
    @Transactional(propagation = Propagation.REQUIRES_NEW)
    public void methodB() { orderRepo.save(order2); }
}

// FIX 2: Self-inject the proxy (not recommended, but works)
@Service
public class OrderService {
    @Autowired @Lazy private OrderService self; // inject proxy
    public void methodA() {
        orderRepo.save(order1);
        self.methodB(); // calls through proxy — @Transactional honored
    }
}

// FIX 3: Use TransactionTemplate (no proxy needed)
public void methodA() {
    txTemplate.executeWithoutResult(status -> orderRepo.save(order1));
    requiresNewTemplate.executeWithoutResult(status -> orderRepo.save(order2));
}

Bug 2: Exception Swallowing

// BUG: exception caught and swallowed — tx commits with partial state
@Transactional
public void processPayment(Payment payment) {
    paymentRepo.save(payment);
    try {
        externalPaymentGateway.charge(payment); // throws PaymentGatewayException
    } catch (PaymentGatewayException e) {
        log.error("Gateway error: {}", e.getMessage()); // SWALLOWED
        // tx COMMITS — payment record saved, money NOT charged!
    }
}

// FIX: rethrow or use setRollbackOnly
@Transactional
public void processPayment(Payment payment) {
    paymentRepo.save(payment);
    try {
        externalPaymentGateway.charge(payment);
    } catch (PaymentGatewayException e) {
        log.error("Gateway error: {}", e.getMessage());
        TransactionAspectSupport.currentTransactionStatus().setRollbackOnly();
        throw new PaymentFailedException("Gateway failed", e); // rethrow
    }
}

Bug 4: Wrong Exception Type for Rollback

// BUG: checked exception thrown — Spring COMMITS by default!
@Transactional
public void importData(String filePath) throws DataImportException {
    // DataImportException is a checked exception
    List<Record> records = parseFile(filePath);      // saves 500 records
    validateRecords(records);                          // throws DataImportException (checked)
    recordRepo.saveAll(records);                       // never reached
    // Spring sees a checked exception → COMMITS the 500 partially-saved records!
}

// FIX: specify rollbackFor
@Transactional(rollbackFor = DataImportException.class)
public void importData(String filePath) throws DataImportException {
    List<Record> records = parseFile(filePath);
    validateRecords(records);  // throws → tx ROLLS BACK
    recordRepo.saveAll(records);
}

// OR: make the exception extend RuntimeException (no special config needed)
public class DataImportException extends RuntimeException { ... }

Bug 5: No Transaction on Service, Only on Repository

// BUG: no @Transactional on service — each repository call = separate micro-transaction
// If second save fails, first is already committed — PARTIAL STATE
@Service
public class RegistrationService {
    // No @Transactional here!

    }

// FIX: @Transactional on service wraps all saves in one atomic unit
@Service
public class RegistrationService {
    @Transactional
    public void registerUser(RegistrationRequest req) {
        userRepo.save(new User(req));
        profileRepo.save(new Profile(req));
        walletRepo.save(new Wallet(req));    // throws → ALL THREE rolled back
    }
}
