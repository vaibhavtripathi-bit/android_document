Explain the Spring Data repository abstraction. How does Spring implement repository interfaces at runtime?
REPOSITORY
▼

Core Concept

Spring Data generates proxy implementations of your repository interfaces at startup — you never write the implementation class. The key factory class is JpaRepositoryFactory, which creates a JpaRepositoryImplementation backed by SimpleJpaRepository<T, ID>. Spring scans the base packages declared in @EnableJpaRepositories, finds all sub-interfaces of Repository, and wires up the proxy — either via JDK dynamic proxy (if the interface extends nothing concrete) or CGLIB.

Repository Hierarchy

Repository → CrudRepository → PagingAndSortingRepository → JpaRepository. Each level adds more methods. JpaRepository adds flush(), saveAndFlush(), deleteAllInBatch(), and getReferenceById().

Best PracticeUse the narrowest interface that meets your needs. Exposing delete() and deleteAll() on a repository whose consumers should only read data is an accidental API surface. Extend Repository and declare only the methods you need.

SimpleJpaRepository Internals

SimpleJpaRepository wraps an injected EntityManager. findById calls em.find(domainClass, id). save(entity) checks entityInformation.isNew(entity): if new → em.persist(entity); if existing → em.merge(entity). findAll() builds a CriteriaQuery via CriteriaBuilder.

Custom Fragment Pattern

// 1. Declare the custom contract
public interface UserRepositoryCustom {
    List<User> searchByMultipleCriteria(UserSearchRequest request);
}

// 2. Implement it — class name MUST end in "Impl" by default
@Repository
public class UserRepositoryCustomImpl implements UserRepositoryCustom {

    @PersistenceContext
    private EntityManager em;

    @Override
    public List<User> searchByMultipleCriteria(UserSearchRequest req) {
        CriteriaBuilder cb = em.getCriteriaBuilder();
        CriteriaQuery<User> cq = cb.createQuery(User.class);
        Root<User> root = cq.from(User.class);
        List<Predicate> predicates = new ArrayList<>();
        if (req.getName() != null) predicates.add(cb.like(root.get("name"), "%" + req.getName() + "%"));
        if (req.getStatus() != null) predicates.add(cb.equal(root.get("status"), req.getStatus()));
        cq.where(predicates.toArray(new Predicate[0]));
        return em.createQuery(cq).getResultList();
    }
}

// 3. Compose into main repository — Spring merges both automatically
public interface UserRepository extends JpaRepository<User, Long>, UserRepositoryCustom {
    // derived queries here
}
Google L6 AngleAt L6, interviewers expect you to know the fragment pattern, discuss repositoryImplementationPostfix for custom naming conventions, and distinguish between customizing a single repository (fragment) versus all repositories (custom base class with repositoryBaseClass in @EnableJpaRepositories). Also: Spring Data uses RepositoryFactoryBeanSupport → RepositoryFactorySupport → JpaRepositoryFactory; knowing this stack shows deep framework understanding.

How do derived query methods work? What is the name-to-JPQL translation algorithm?
DERIVED_QUERIES
▼

Parsing Algorithm

Spring Data's PartTreeJpaQuery parses the method name at application startup using a PartTree. The parser first strips subject prefixes (find…By, read…By, get…By, query…By, count…By, exists…By, delete…By). Everything after By is the predicate, split on And / Or. Each segment is a Part that maps to a JPQL clause. Fail-fast: if a field doesn't exist on the entity, Spring throws at startup — not at query time.

Code Examples: Name → JPQL

// Simple equality
List<User> findByEmail(String email);
// → SELECT u FROM User u WHERE u.email = ?1

// Compound predicate
List<User> findByFirstNameAndLastName(String first, String last);
// → SELECT u FROM User u WHERE u.firstName = ?1 AND u.lastName = ?2

// Nested property traversal (joins automatically)
List<Order> findByCustomerCity(String city);
// → SELECT o FROM Order o WHERE o.customer.city = ?1

// Limit + sort in name
Optional<User> findFirstByEmailOrderByCreatedAtDesc(String email);
// → SELECT u FROM User u WHERE u.email = ?1 ORDER BY u.createdAt DESC LIMIT 1

// Pagination
Page<Product> findByCategoryAndPriceLessThan(String cat, BigDecimal price, Pageable p);
⚠ GotchaMethod names get unreadable fast. findByFirstNameAndLastNameAndStatusAndCreatedAtAfterOrderByLastNameAsc is technically valid but unmaintainable. Use @Query for anything beyond two predicates. Derived queries also generate the JPQL at startup, so any property typo is caught early — but this means derived queries cannot be fully dynamic (variable predicate count).
Best PracticeDerived queries are excellent for simple, stable lookups (findByEmail, findByUsername). Switch to @Query or Specifications when logic requires more than 2 predicates or optional filters.

How does @Query work? When do you use JPQL vs native SQL? What is @Modifying?
QUERY_ANNOTATION
▼

JPQL vs Native SQL

JPQL (@Query("SELECT u FROM User u WHERE u.email = :email")) uses entity and field names — portable across databases, benefits from Hibernate's type mapping. Native SQL (@Query(value="SELECT * FROM users WHERE email = :email", nativeQuery=true)) is database-specific but allows DB-specific features (window functions, CTEs, JSONB operators). Named parameters with :name or positional with ?1.

SpEL in @Query

Use #{#entityName} in generic base repository implementations where the entity type isn't known at compile time. This resolves to the actual entity name at runtime, enabling reusable base queries.

@NoRepositoryBean
public interface BaseRepository<T, ID> extends JpaRepository<T, ID> {

    @Query("SELECT e FROM #{#entityName} e WHERE e.active = true")
    List<T> findAllActive();
}

@Modifying for Bulk DML

@Modifying is required on any @Query that performs UPDATE or DELETE. Without it, Spring throws InvalidDataAccessApiUsageException. Two important attributes:

  • clearAutomatically (default true in newer versions): clears the persistence context after the DML executes, preventing stale entity reads. Set to false only if you know the affected rows aren't in the first-level cache.
  • flushAutomatically (default false): if true, flushes pending dirty-checked changes before the DML runs, ensuring the bulk update sees the latest application-side state.
// Bulk status update — far faster than loading entities one by one
@Modifying
@Transactional
@Query("UPDATE Order o SET o.status = :newStatus, o.updatedAt = :now " +
       "WHERE o.status = :oldStatus AND o.createdAt < :cutoff")
int bulkUpdateOrderStatus(
    @Param("oldStatus") OrderStatus oldStatus,
    @Param("newStatus") OrderStatus newStatus,
    @Param("cutoff") Instant cutoff,
    @Param("now") Instant now
);

// Native SQL for database-specific feature (PostgreSQL upsert)
@Modifying
@Transactional
@Query(value = """
    INSERT INTO user_scores (user_id, score, updated_at)
    VALUES (:userId, :score, NOW())
    ON CONFLICT (user_id) DO UPDATE SET score = EXCLUDED.score, updated_at = NOW()
    """, nativeQuery = true)
void upsertUserScore(@Param("userId") Long userId, @Param("score") int score);
⚠ GotchaBulk DML via @Modifying bypasses the persistence context entirely. If you load an Order entity, then call bulkUpdateOrderStatus() in the same transaction, the in-memory Order object still has the old status — it isn't automatically refreshed. This is why clearAutomatically=true is the safe default.
Google L6 AngleAt scale, bulk DML is orders of magnitude faster than "load N entities, modify, save all." A question about updating 100,000 order statuses should immediately lead to a @Modifying @Query answer, not findAll() + forEach(save()).

Explain the N+1 problem in detail. Show all solutions with trade-offs.
N+1_PROBLEM
▼

Root Cause

When you load a list of N entities and then access a LAZY-loaded association on each one, Hibernate fires a separate SELECT per entity. Loading 100 Orders where each has a LAZY Customer results in 1 query for orders + 100 queries for customers = 101 queries. LAZY is the JPA default for @OneToMany and @ManyToMany. The problem is invisible in tests (small data) and silent in production.

Solution Comparison

SolutionHowTrade-off
EAGER fetching@OneToMany(fetch=EAGER)Always fetches; Cartesian product for multiple collections. Never use on @OneToMany.
JOIN FETCH in JPQLSELECT o FROM Order o JOIN FETCH o.customerOne query. Cannot paginate correctly (memory pagination warning).
@EntityGraph@EntityGraph(attributePaths={"customer"})Declarative, composable. Same Cartesian product caveat with pagination.
@BatchSize@BatchSize(size=25) on collectionBatches N queries into ceil(N/25) queries. Works with pagination.
DTO ProjectionSELECT new DTO(o.id, c.name) FROM ...Fastest — no entity loading. Read-only.
@Fetch(SUBSELECT)Hibernate annotation on collectionFetches all collections in one subselect. Works but less control.

Correct Solutions in Practice

// Solution 1: JOIN FETCH JPQL (no pagination)
@Query("SELECT o FROM Order o JOIN FETCH o.customer JOIN FETCH o.items WHERE o.status = :s")
List<Order> findWithDetailsForStatus(@Param("s") OrderStatus status);

// Solution 2: @EntityGraph (clean, on repository method)
@EntityGraph(attributePaths = {"customer", "items"})
List<Order> findByStatus(OrderStatus status);

// Solution 3: @BatchSize (works alongside pagination)
@Entity
public class Order {
    @OneToMany(mappedBy = "order", fetch = FetchType.LAZY)
    @BatchSize(size = 50)
    private List<OrderItem> items;
}

// Solution 4: DTO projection (no entity at all)
@Query("SELECT new com.example.dto.OrderSummary(o.id, o.total, c.name) " +
       "FROM Order o JOIN o.customer c WHERE o.status = :s")
List<OrderSummary> findOrderSummaries(@Param("s") OrderStatus status);
⚠ GotchaJOIN FETCH with Pageable silently fetches ALL matching rows into memory, then paginates in Java. Hibernate logs warning HHH90003004. The fix is a two-pass query: first page the IDs, then fetch the full entities with JOIN FETCH for only those IDs.
Google L6 AngleN+1 is the single most common Spring/JPA production performance bug. An L6 answer enumerates all solutions, explains when each applies, and immediately proposes adding a query count assertion in integration tests using datasource-proxy to catch N+1 regressions in CI.

Explain JPA entity lifecycle states: transient, persistent, detached, removed.
ENTITY_LIFECYCLE
▼

The Four States

  • Transient: new Java object, no association with any persistence context. No database identity. Changes are completely invisible to JPA.
  • Persistent (Managed): associated with an active EntityManager. Has a database identity. Dirty checking is active — any field mutation is tracked and flushed as UPDATE at flush time.
  • Detached: previously persistent, but the persistence context was closed (transaction ended, em.detach(), em.clear(), or OSIV response committed). Retains its DB identity but changes are NOT tracked. Must re-attach via em.merge().
  • Removed: marked for deletion via em.remove(). Still in the context until flush; a DELETE SQL is issued at flush time.

State Transition Diagram

new User()           → [TRANSIENT]
  em.persist(u)      → [PERSISTENT]   // or repository.save(new entity)
  tx commits         → [DETACHED]     // persistence context closes
  em.merge(u)        → [PERSISTENT]   // new managed copy returned; original still detached
  em.remove(managed) → [REMOVED]
  flush()            → DELETE issued

  // Spring Data repository.save():
  //   isNew() ? em.persist() : em.merge()
  //   isNew() determined by: ID is null, or entity implements Persistable

OpenEntityManagerInView (OEMIV)

Spring Boot enables OEMIV by default (spring.jpa.open-in-view=true). This extends the persistence context from the start of the HTTP request all the way through the view rendering, enabling lazy loading in controllers and templates without LazyInitializationException.

⚠ GotchaOEMIV is an anti-pattern at scale. It holds a database connection for the entire HTTP request duration (including view rendering time), drastically reducing connection pool availability under load. It also hides N+1 problems — lazy collections silently fire queries inside JSP/Thymeleaf templates. Set spring.jpa.open-in-view=false in production-grade services and fix lazy loading explicitly.
Best PracticeModel your service layer to return DTOs (not entities) to controllers. This forces you to load everything you need within the transaction and eliminates all OEMIV dependency.
Google L6 AngleA common interview scenario: "A developer modifies a detached entity returned from a REST call, calls save(), but the change doesn't persist." The answer: the entity is detached, em.merge() creates a new managed copy — you must use the returned value from merge() / save(). L6 engineers can also explain the optimistic locking implication: merging a detached entity with a @Version field will trigger a version check.

How does Hibernate dirty checking work? What is the performance impact?
DIRTY_CHECKING
▼

Mechanism

When an entity becomes persistent, Hibernate stores a deep copy (hydrated state) of all its field values as a snapshot in the StatefulPersistenceContext. At flush time, Hibernate performs a field-by-field comparison of each managed entity against its snapshot. Any entity with at least one changed field gets an UPDATE SQL statement — even if you never called save().

Default Behavior vs Optimizations

// Default: UPDATE includes ALL columns, even unchanged ones
// UPDATE users SET name=?, email=?, age=?, address=?, ... WHERE id=?

// @DynamicUpdate: generates UPDATE only for changed columns
@Entity
@DynamicUpdate
public class User {
    // ...
}
// UPDATE users SET email=? WHERE id=?  (only email changed)

// @DynamicInsert: INSERT only non-null fields
@Entity
@DynamicInsert
public class Product {
    // Useful when many optional columns have DB defaults
}

// Flush modes
// FlushMode.AUTO (default): Hibernate flushes before executing a JPQL/criteria query
//   that might be affected by pending changes. Ensures consistency.
// FlushMode.COMMIT: only flush at transaction commit. Fewer flushes, risk of stale reads.
// FlushMode.MANUAL: never flush automatically — only em.flush() explicitly.
//   Used internally by @Transactional(readOnly=true).

Performance Considerations

  • Loading 1,000 entities + flushing = 1,000 snapshot comparisons. For wide entities (50+ fields), this CPU cost is measurable.
  • Large @OneToMany collections that are loaded and dirty-checked are especially expensive.
  • Use @Transactional(readOnly=true) for read paths — Hibernate sets FlushMode.MANUAL, skips snapshot creation entirely, and avoids dirty checking. Significant throughput improvement.
  • For write-heavy bulk operations, bypass entity loading entirely with @Modifying @Query.
⚠ GotchaDirty checking fires even if you never called any save/update method. Loading an entity, accidentally modifying it (even setting a field to the same value via a setter), and letting the transaction commit will issue an UPDATE. This can cause unexpected write amplification in ostensibly read-only operations.
Google L6 AngleIn write-heavy, high-throughput services, dirty checking overhead is measurable. L6 engineers instrument their persistence layer (Hibernate Statistics, Micrometer) to observe flush counts, dirty entity counts, and SQL counts per request. @DynamicUpdate reduces lock contention on wide tables by narrowing the columns locked in row-level locking databases.

Explain JPA first-level and second-level cache. How does Spring integrate with them?
CACHING_LEVELS
▼

First-Level Cache (L1)

Always enabled. Scope = EntityManager (one transaction). Within the same session, calling em.find(User.class, 1L) twice only issues one SQL — the second call returns the cached instance. em.clear() discards the L1 cache. This is why all JPA repository calls within a single @Transactional method share state — they use the same EntityManager.

Second-Level Cache (L2)

Opt-in. Scope = SessionFactory (shared across all sessions/transactions). Enabled per entity. Providers: Ehcache, Hazelcast, Caffeine (via JCache/JSR-107).

# application.properties — enabling L2 cache with Ehcache
spring.jpa.properties.hibernate.cache.use_second_level_cache=true
spring.jpa.properties.hibernate.cache.region.factory_class=\
    org.hibernate.cache.jcache.JCacheRegionFactory
spring.jpa.properties.javax.cache.provider=\
    org.ehcache.jsr107.EhcacheCachingProvider

// Entity annotation — must opt each entity in explicitly
@Entity
@Cache(usage = CacheConcurrencyStrategy.READ_WRITE)
public class Country {
    // Rarely-updated reference data — ideal for L2 cache
}

// Cache concurrency strategies:
// READ_ONLY        — immutable data (country codes). No locking overhead.
// NONSTRICT_READ_WRITE — eventual consistency. Short stale window tolerable.
// READ_WRITE       — soft-lock on update. Consistent but overhead.
// TRANSACTIONAL    — JTA environments. Full transactional cache.

// Query cache (cache JPQL result sets — list of IDs)
spring.jpa.properties.hibernate.cache.use_query_cache=true
// Repository:
@QueryHints({@QueryHint(name=HINT_CACHEABLE, value="true")})
List<Country> findAll();

AspectSpring @CacheableJPA L2 Cache

Google L6 AngleL2 cache adds operational complexity — especially in distributed deployments where stale cache entries across nodes can cause correctness issues. The L6 answer: for frequently-updated entities, prefer Redis-backed @Cacheable at the service layer with explicit @CacheEvict, giving you full control over invalidation semantics. Reserve L2 cache for truly immutable reference data (READ_ONLY strategy).

Explain optimistic vs pessimistic locking in JPA. When do you use each?
LOCKING
▼

Optimistic Locking

Uses a @Version field. No database lock is held. Before each UPDATE, Hibernate checks that the version in the DB matches the version on the entity. If another transaction modified the row first, the version won't match and Hibernate throws OptimisticLockException (wrapped in ObjectOptimisticLockingFailureException by Spring). The assumption: conflicts are rare — no need to hold locks speculatively.

@Entity
public class BankAccount {
    @Id @GeneratedValue
    private Long id;

    @Version
    private Long version;  // int, long, short, or Timestamp — auto-incremented by Hibernate

    private BigDecimal balance;
}

// Hibernate generates:
// UPDATE bank_accounts SET balance=?, version=2 WHERE id=1 AND version=1
// If 0 rows updated → OptimisticLockException

// Service layer with retry:
@Retryable(value = OptimisticLockingFailureException.class, maxAttempts = 3)
public void transfer(Long accountId, BigDecimal amount) {
    BankAccount account = accountRepo.findById(accountId).orElseThrow();
    account.setBalance(account.getBalance().subtract(amount));
    accountRepo.save(account);
}

Pessimistic Locking

Acquires a database-level lock at read time. No other transaction can modify the row until the lock is released (transaction commits or rolls back). Higher guarantee, lower throughput.

// PESSIMISTIC_WRITE → SELECT ... FOR UPDATE
@Lock(LockModeType.PESSIMISTIC_WRITE)
@Query("SELECT a FROM BankAccount a WHERE a.id = :id")
Optional<BankAccount> findByIdForUpdate(@Param("id") Long id);

// Or programmatically:
BankAccount account = em.find(BankAccount.class, id, LockModeType.PESSIMISTIC_WRITE);

// PESSIMISTIC_READ → SELECT ... FOR SHARE (shared lock, no writes by others)
// PESSIMISTIC_FORCE_INCREMENT → like PESSIMISTIC_WRITE but also increments @Version

Decision Framework

ScenarioRecommended LockReason
User profile updatesOptimistic (@Version)Conflicts rare; high concurrency needed
Inventory deduction at order timePessimistic (FOR UPDATE)Race condition causes oversell; conflicts frequent
Wallet/balance debitPessimistic (FOR UPDATE)Correctness critical; concurrent deductions
Document collaborative editOptimistic + merge UIUX: show conflict, let user resolve
Distributed job schedulerPessimistic or DB advisory lockLeader election — only one runner at a time
⚠ GotchaPessimistic locking can cause deadlocks when two transactions lock rows in different orders. Always lock resources in a consistent order, set lock timeouts (javax.persistence.lock.timeout), and handle PessimisticLockingFailureException (lock timeout or deadlock) with appropriate retry logic.

What are JPA projections? Explain interface, class (DTO), and dynamic projections.
PROJECTIONS
▼

Why Projections Matter

Loading full entities when you only need 2 fields wastes: SQL bandwidth, Hibernate mapping time, memory for unused fields, and triggers unnecessary dirty checking overhead. Projections narrow the SQL SELECT to exactly what's needed.

Interface Projection

Spring Data generates a proxy. Getter method names map to entity property names. Supports nested projections for associations.

public interface UserSummary {
    String getName();
    String getEmail();

    // Nested projection
    AddressSummary getAddress();

    interface AddressSummary {
        String getCity();
    }

    // SpEL expression — combine fields
    @Value("#{target.firstName + ' ' + target.lastName}")
    String getFullName();
}

// Repository usage:
List<UserSummary> findByActiveTrue();
Optional<UserSummary> findById(Long id, Class<UserSummary> type);

Class (DTO) Projection

Uses JPQL constructor expression. No proxy overhead. The DTO is a plain Java class with a matching constructor. Generates tight SQL.

// DTO
public record UserDTO(String name, String email) {}

// JPQL constructor expression
@Query("SELECT new com.example.dto.UserDTO(u.name, u.email) " +
       "FROM User u WHERE u.active = true ORDER BY u.name")
List<UserDTO> findActiveUserDTOs();

// Generated SQL:
// SELECT u.name, u.email FROM users u WHERE u.active = true ORDER BY u.name

Dynamic Projections

Pass the projection type at call time — same method returns different shapes.

// Generic repository method
<T> List<T> findByActive(boolean active, Class<T> type);

// Call with different projections based on context
List<UserSummary> summaries = repo.findByActive(true, UserSummary.class);
List<UserDTO> dtos = repo.findByActive(true, UserDTO.class);
List<User> full = repo.findByActive(true, User.class); // full entity if needed
⚠ GotchaInterface projections backed by a proxy sometimes still generate SELECT * behind the scenes, especially with more complex entity graphs. Always verify the generated SQL with logging. DTO projections (constructor expressions) reliably generate narrow selects. For truly performance-critical reads, DTO projections are the safer choice.
Best PracticeUse interface projections for simple read models in REST APIs — they're clean and refactoring-safe (rename the getter = compile error). Use DTO projections (records) for performance-sensitive queries and reporting. Never return JPA entities from REST controllers — you lose control of lazy loading, expose DB internals, and risk Jackson stack overflow on circular references.

What is the Specification pattern in Spring Data JPA? When is it appropriate?
SPECIFICATIONS
▼

Core Concept

The Specification<T> interface (from Spring Data) wraps a JPA Criteria API predicate into a reusable, composable unit. Your repository extends JpaSpecificationExecutor<T> to gain findAll(Specification), findOne(Specification), count(Specification), and exists(Specification).

Specification is a functional interface: Predicate toPredicate(Root<T> root, CriteriaQuery<?> query, CriteriaBuilder cb). Specs compose with .and(), .or(), .not().

Dynamic Search Implementation

public class UserSpecs {

    public static Specification<User> hasName(String name) {
        return (root, query, cb) ->
            name == null ? null : cb.like(cb.lower(root.get("name")), "%" + name.toLowerCase() + "%");
    }

    public static Specification<User> hasStatus(UserStatus status) {
        return (root, query, cb) ->
            status == null ? null : cb.equal(root.get("status"), status);
    }

    public static Specification<User> createdAfter(Instant date) {
        return (root, query, cb) ->
            date == null ? null : cb.greaterThan(root.get("createdAt"), date);
    }

    public static Specification<User> inDepartment(Long deptId) {
        return (root, query, cb) -> {
            if (deptId == null) return null;
            Join<User, Department> dept = root.join("department", JoinType.INNER);
            return cb.equal(dept.get("id"), deptId);
        };
    }
}

// Service — filter is purely data-driven, no if-else spaghetti
public Page<User> searchUsers(UserSearchRequest req, Pageable pageable) {
    Specification<User> spec = Specification.where(hasName(req.getName()))
        .and(hasStatus(req.getStatus()))
        .and(createdAfter(req.getCreatedAfter()))
        .and(inDepartment(req.getDepartmentId()));
    return userRepo.findAll(spec, pageable);
}

// Repository
public interface UserRepository extends JpaRepository<User, Long>,
        JpaSpecificationExecutor<User> {}

Specification vs Alternatives

ApproachBest ForLimitation
SpecificationDynamic multi-filter search, moderate complexityVerbose for complex joins; no compile-time safety
QueryDSL (Q-classes)Type-safe dynamic queries, IDE autocompleteAPT code generation setup; extra build dependency
JOOQComplex SQL, reporting, aggregatesRequires schema-to-code generation; more setup
@Query native SQLDB-specific features, performance-criticalNo dynamic predicates; string-based
Google L6 AngleSpecifications are the right tool for admin search panels and filtered list APIs with 3-8 optional parameters. For complex reporting queries with window functions, aggregates, or multi-table joins, Specifications become unwieldy. L6 engineers know when to switch to JOOQ or a native query RowMapper — and they justify the choice by the query complexity threshold.

How does pagination work in Spring Data JPA? What are the N+1 and count query issues?
PAGINATION
▼

Pageable and Page

Page<T> findAll(Pageable pageable) — Spring Data executes two queries: the data query with LIMIT/OFFSET and a COUNT(*) query for totalElements. Page exposes getContent(), getTotalElements(), getTotalPages(), hasNext(), hasPrevious().

// Controller — Spring MVC auto-resolves Pageable from request params
// GET /users?page=0&size=20&sort=name,asc
@GetMapping("/users")
public Page<UserSummary> getUsers(
        @PageableDefault(size = 20, sort = "createdAt", direction = DESC) Pageable pageable) {
    return userService.getUsers(pageable);
}

// Custom count query (avoid expensive COUNT on complex join)
@Query(value = "SELECT u FROM User u JOIN FETCH u.roles WHERE u.active = true",
       countQuery = "SELECT count(u) FROM User u WHERE u.active = true")
Page<User> findActiveUsersWithRoles(Pageable pageable);

Slice — Skip the Count Query

// Slice<T>: no COUNT query — just knows if there's a next page
// Perfect for infinite scroll, cursor-based UIs
Slice<Post> findByAuthorId(Long authorId, Pageable pageable);
// Only fires: SELECT ... LIMIT 21 (size+1 to detect hasNext)

Two-Phase Pagination for JOIN FETCH

// Problem: JOIN FETCH + Pageable = in-memory pagination (HHH90003004)
// Fix: paginate IDs first, then fetch full entities with JOIN FETCH

@Query(value = "SELECT u.id FROM User u WHERE u.active = true",
       countQuery = "SELECT count(u.id) FROM User u WHERE u.active = true")
Page<Long> findActiveUserIds(Pageable pageable);

public Page<User> findActiveUsersWithRoles(Pageable pageable) {
    Page<Long> idPage = userRepo.findActiveUserIds(pageable);
    List<User> users = userRepo.findAllWithRolesByIdIn(idPage.getContent()); // JOIN FETCH
    return new PageImpl<>(users, pageable, idPage.getTotalElements());
}

Keyset Pagination (Seek Method)

// Offset pagination degrades: OFFSET 10000 still scans 10000 rows
// Keyset: O(1) regardless of page number — requires indexed sort key

@Query("SELECT p FROM Post p WHERE p.createdAt < :lastSeen ORDER BY p.createdAt DESC")
List<Post> findPostsBefore(@Param("lastSeen") Instant lastSeen, Pageable pageable);

// Client sends the createdAt of the last seen item → server returns next N items
// No count query, no OFFSET — pure cursor-based navigation
Google L6 AngleAt scale, offset-based pagination is the hidden killer. Page 500 of a 20-item page means OFFSET 10,000 — the DB scans 10,000 rows just to discard them. Keyset pagination is O(1) and should be the default for any feed, timeline, or list that can grow unbounded. L6 engineers propose keyset pagination proactively in system design interviews.

What are @EntityGraph and @NamedEntityGraph? How do they solve the N+1 problem?
ENTITY_GRAPHS
▼

Core Concept

@EntityGraph is a declarative fetch plan that overrides the default fetch type for specific associations on a per-query basis. It adds JOIN FETCH semantics without embedding the join in the JPQL query string — keeping repository method names readable and fetch strategies separately declared.

// Named entity graph on the entity class
@NamedEntityGraph(
    name = "order.withCustomerAndItems",
    attributeNodes = {
        @NamedAttributeNode("customer"),
        @NamedAttributeNode(value = "items", subgraph = "items.withProduct")
    },
    subgraphs = @NamedSubgraph(
        name = "items.withProduct",
        attributeNodes = @NamedAttributeNode("product")
    )
)
@Entity
public class Order {
    @ManyToOne(fetch = LAZY) private Customer customer;
    @OneToMany(mappedBy = "order", fetch = LAZY) private List<OrderItem> items;
}

// Repository — reference by name
@EntityGraph("order.withCustomerAndItems")
Optional<Order> findById(Long id);

// Or inline ad-hoc (no @NamedEntityGraph needed)
@EntityGraph(attributePaths = {"customer", "items", "items.product"})
List<Order> findByStatus(OrderStatus status);

// Also works on derived queries:
@EntityGraph(attributePaths = {"roles", "permissions"})
Optional<User> findByEmail(String email);

Multiple Graphs for Different Use Cases

@NamedEntityGraphs({
    @NamedEntityGraph(name = "order.minimal",
        attributeNodes = @NamedAttributeNode("customer")),
    @NamedEntityGraph(name = "order.full",
        attributeNodes = {
            @NamedAttributeNode("customer"),
            @NamedAttributeNode("items"),
            @NamedAttributeNode("payments"),
            @NamedAttributeNode("shipments")
        })
})
@Entity
public class Order { ... }

EntityGraph Types

  • FETCH (default): specified attributes are EAGER; all others are LAZY. Overrides defaults.
  • LOAD: specified attributes are EAGER; all others use their declared fetch type. Additive.
⚠ Gotcha@EntityGraph with Pageable triggers the same Cartesian product in-memory pagination warning as JOIN FETCH when multiple collection-level attributes are fetched. Use the two-pass ID pagination pattern for paginated queries that need joined collections.

How does Spring Data auditing work? Explain @CreatedDate, @LastModifiedDate, @CreatedBy.
AUDITING
▼

Setup

// 1. Enable JPA auditing
@Configuration
@EnableJpaAuditing(auditorAwareRef = "auditorProvider")
public class JpaConfig {

    @Bean
    public AuditorAware<String> auditorProvider() {
        return () -> Optional.ofNullable(SecurityContextHolder.getContext().getAuthentication())
            .filter(Authentication::isAuthenticated)
            .map(Authentication::getName);
    }
}

// 2. Base entity with auditing fields (shared via @MappedSuperclass)
@MappedSuperclass
@EntityListeners(AuditingEntityListener.class)
public abstract class AuditableEntity {

    @CreatedDate
    @Column(updatable = false)
    private Instant createdAt;

    @LastModifiedDate
    private Instant updatedAt;

    @CreatedBy
    @Column(updatable = false)
    private String createdBy;

    @LastModifiedBy
    private String lastModifiedBy;
}

// 3. Domain entity extends base
@Entity
public class Product extends AuditableEntity {
    @Id @GeneratedValue private Long id;
    private String name;
    @Version private Long version;  // Combine with @LastModifiedDate for optimistic locking
}

How It Works Internally

AuditingEntityListener is a JPA entity listener that Spring registers. It intercepts @PrePersist (sets @CreatedDate, @CreatedBy) and @PreUpdate (sets @LastModifiedDate, @LastModifiedBy). The AuditorAware bean is called at write time to resolve the current principal — typically from SecurityContextHolder.

Best PracticeFor async operations (background jobs, event consumers) where SecurityContext is empty, provide a service-account identity in AuditorAware rather than returning Optional.empty(). Empty auditor leaves the createdBy field null, which breaks compliance audit trails.
Google L6 AngleAudit trails are a compliance requirement in regulated domains (GDPR, HIPAA, financial services). L6 engineers design auditing at the database level (not just application) — consider trigger-based audit tables or Hibernate Envers for immutable, tamper-proof audit logs. Spring Data auditing is convenient but not tamper-proof (application code can update createdAt).

How do you perform batch inserts and updates efficiently with Spring Data JPA?
BATCH_OPERATIONS
▼

The Problem

Saving 10,000 entities via repository.saveAll(list) with default settings issues 10,000 individual INSERT statements — 10,000 round-trips. For 10,000 rows, this can take 10-30 seconds instead of <1 second with batching.

Enabling Hibernate JDBC Batching

# application.properties
spring.jpa.properties.hibernate.jdbc.batch_size=50
spring.jpa.properties.hibernate.order_inserts=true   # group by entity type
spring.jpa.properties.hibernate.order_updates=true   # group by entity type
spring.jpa.properties.hibernate.generate_statistics=true  # verify batching is working

# CRITICAL: IDENTITY generation breaks batching — Hibernate can't batch because
# it needs the DB-generated ID immediately after each INSERT.
# Use SEQUENCE strategy instead:
@Entity
public class ImportRecord {
    @Id
    @GeneratedValue(strategy = GenerationType.SEQUENCE,
                    generator = "import_seq")
    @SequenceGenerator(name = "import_seq", sequenceName = "import_record_seq",
                       allocationSize = 50)  // pre-allocate 50 IDs — reduces round-trips
    private Long id;
}

Batch Save with Periodic Flush/Clear

@Service
@Transactional
public class BulkImportService {

    @PersistenceContext
    private EntityManager em;

    private static final int BATCH_SIZE = 50;

    public void bulkImport(List<ImportRecord> records) {
        for (int i = 0; i < records.size(); i++) {
            em.persist(records.get(i));
            if ((i + 1) % BATCH_SIZE == 0) {
                em.flush();   // execute SQL batch
                em.clear();   // release all managed entities from L1 cache (prevents OOM)
            }
        }
        em.flush(); // handle remaining records
    }
}

Performance Comparison

Approach100K rows (approx.)Notes
JPA saveAll(), no batching~60-120s100K round-trips
JPA + batch_size=50~5-10s2,000 batch calls
JdbcTemplate.batchUpdate()~0.5-2sRaw JDBC, zero ORM overhead
COPY (PostgreSQL)~0.1-0.5sBulk load; no row-by-row SQL
Google L6 AngleFor ETL pipelines, data migrations, and bulk import jobs, JdbcTemplate.batchUpdate() is 10-50x faster than JPA batch saving. The ORM's entity management, dirty checking, and L1 cache are pure overhead for write-only bulk operations. L6 engineers reach for the right tool: JPA for transactional business logic, JDBC for bulk I/O.

Explain all JPA relationship types. What are the fetch and cascade gotchas?

Relationship Types

Cascade Gotchas

@Entity
public class Order {
    // WRONG: CascadeType.ALL on a collection with orphanRemoval
    // Removing a single item from the list triggers a DELETE on that row
    // But: do you really want removing from a list to delete the DB record?
    @OneToMany(mappedBy = "order",
               cascade = CascadeType.ALL,
               orphanRemoval = true)  // ← correct for "order owns its items"
    private List<OrderItem> items = new ArrayList<>();

    // WRONG: CascadeType.REMOVE on @ManyToMany
    // Removing one user from a group → cascades and deletes all users!
    @ManyToMany(cascade = CascadeType.PERSIST)  // never REMOVE on @ManyToMany
    private Set<Role> roles;

    // Bidirectional helper method — essential for consistency
    public void addItem(OrderItem item) {
        items.add(item);
        item.setOrder(this);  // maintain both sides
    }

    public void removeItem(OrderItem item) {
        items.remove(item);
        item.setOrder(null);
    }
}

@ManyToMany vs Explicit Join Entity

// Prefer this (explicit join entity) over @ManyToMany when you need extra columns
@Entity
public class OrderItem {
    @Id @GeneratedValue private Long id;

    @ManyToOne(fetch = FetchType.LAZY)
    @JoinColumn(name = "order_id")
    private Order order;

    @ManyToOne(fetch = FetchType.LAZY)
    @JoinColumn(name = "product_id")
    private Product product;

    private int quantity;
    private BigDecimal unitPrice;  // extra columns — impossible with bare @ManyToMany
}
⚠ Gotcha@OneToMany without mappedBy tells JPA the current entity is the owning side — JPA creates a join table! This is almost never what you want. Always specify mappedBy on the @OneToMany side to point to the FK field on the child entity.

How do you add custom implementation to a Spring Data repository?
CUSTOM_REPOSITORY
▼

Fragment Pattern (Per-Repository Customization)

Create a separate interface for the custom methods, implement it in a class with the "Impl" suffix (or configure repositoryImplementationPostfix), then compose both into the main repository interface. Spring Data auto-detects the Impl class and merges it into the proxy.

// Step 1: declare custom contract
public interface UserRepositoryCustom {
    List<User> findByComplexCriteria(UserSearchRequest request);
    Map<UserStatus, Long> countByStatus();
}

// Step 2: implement — naming convention: {RepositoryName}Impl or {InterfaceName}Impl
@Repository
public class UserRepositoryCustomImpl implements UserRepositoryCustom {

    @PersistenceContext
    private EntityManager em;

    @Override
    public List<User> findByComplexCriteria(UserSearchRequest req) {
        // Use CriteriaBuilder, JPQL, or native SQL — full control
        TypedQuery<User> query = em.createQuery(
            "SELECT u FROM User u WHERE (:name IS NULL OR u.name LIKE :name) " +
            "AND (:dept IS NULL OR u.department.id = :dept)",
            User.class);
        query.setParameter("name", req.getName() != null ? "%" + req.getName() + "%" : null);
        query.setParameter("dept", req.getDepartmentId());
        return query.getResultList();
    }

    @Override
    public Map<UserStatus, Long> countByStatus() {
        return em.createQuery(
            "SELECT u.status, count(u) FROM User u GROUP BY u.status",
            Object[].class)
            .getResultStream()
            .collect(Collectors.toMap(r -> (UserStatus) r[0], r -> (Long) r[1]));
    }
}

// Step 3: compose into main repository
public interface UserRepository
        extends JpaRepository<User, Long>,
                JpaSpecificationExecutor<User>,
                UserRepositoryCustom {

    // Derived queries, @Query methods alongside custom methods
    Optional<User> findByEmail(String email);
}

Global Base Repository Customization

// Extend SimpleJpaRepository to add behavior to ALL repositories
@NoRepositoryBean
public class SoftDeleteRepositoryImpl<T extends SoftDeletable, ID>
        extends SimpleJpaRepository<T, ID> {

    public SoftDeleteRepositoryImpl(JpaEntityInformation<T, ?> ei, EntityManager em) {
        super(ei, em);
    }

    @Override
    public void delete(T entity) {
        entity.setDeleted(true);
        save(entity);  // soft delete instead of hard delete
    }
}

// Register as base class
@EnableJpaRepositories(repositoryBaseClass = SoftDeleteRepositoryImpl.class)
Google L6 AngleThe fragment pattern is how you escape Spring Data's generated query limitations without losing the repository abstraction. L6 engineers use it for: complex reporting queries that can't be expressed in JPQL, database-specific features (PostgreSQL full-text search, JSONB queries), or performance-optimized bulk operations using raw JDBC within the same repository interface contract.

How does @Transactional interact with JPA/Hibernate? What is the persistence context lifecycle?
TRANSACTION_JPA
▼

Persistence Context and Transaction Binding

When a @Transactional method is entered, Spring (via TransactionInterceptor) calls JpaTransactionManager, which opens a new EntityManager and binds it to the current thread via TransactionSynchronizationManager. All JPA repository calls within the method share this single EntityManager — same first-level cache, same dirty checking scope.

@Transactional
public void processOrder(Long orderId) {
    Order order = orderRepo.findById(orderId).orElseThrow();  // em opened, entity loaded
    order.setStatus(PROCESSING);                               // entity is managed — no save() needed!
    inventoryService.reserveStock(order);                      // same em, same tx
    // tx commits → dirty check → UPDATE orders SET status='PROCESSING' WHERE id=?
    // em closed
}

// WITHOUT @Transactional:
// Each repo call gets its OWN EntityManager (opened + closed immediately)
// Dirty checking does NOT work across calls
// order.setStatus() → no UPDATE issued (no managed context)
// Lazy loading after method returns → LazyInitializationException

Read-Only Transaction Optimizations

@Transactional(readOnly = true)
public UserDTO getUserProfile(Long userId) {
    // Hibernate: FlushMode.MANUAL → no dirty checking, no snapshot creation
    // Spring: hints DataSource → can route to read replica
    // Result: 20-40% faster for read-heavy paths
    return userRepo.findById(userId)
        .map(userMapper::toDTO)
        .orElseThrow();
}

// readOnly=true on class, override for writes:
@Service
@Transactional(readOnly = true)
public class UserService {
    public UserDTO getUser(Long id) { ... }         // read-only inherited
    public UserDTO getPage(Pageable p) { ... }      // read-only inherited

    @Transactional  // overrides class-level readOnly — defaults to readOnly=false
    public UserDTO createUser(CreateUserRequest req) { ... }
}
⚠ GotchaOSIV (spring.jpa.open-in-view=true, Spring Boot default) extends the EntityManager across the entire HTTP request. This enables lazy loading in controller/view layers but holds a DB connection open through the full request — including view rendering and outbound HTTP calls. Under load, this exhausts the connection pool. Always set spring.jpa.open-in-view=false in production services.

How does JPA entity inheritance work? Compare the three strategies.
INHERITANCE
▼

Three Strategies

// SINGLE_TABLE — all subclasses in one table
@Entity
@Inheritance(strategy = InheritanceType.SINGLE_TABLE)
@DiscriminatorColumn(name = "type", discriminatorType = DiscriminatorType.STRING)
public abstract class Payment { @Id @GeneratedValue Long id; BigDecimal amount; }

@Entity @DiscriminatorValue("CARD")
public class CardPayment extends Payment { String cardLast4; String network; }

@Entity @DiscriminatorValue("BANK")
public class BankTransfer extends Payment { String iban; String bankCode; }

// SQL: SELECT * FROM payment WHERE type='CARD'
// All columns in one table — subclass-specific columns are nullable for other types

// JOINED — superclass + subclass tables, FK join
@Entity
@Inheritance(strategy = InheritanceType.JOINED)
public abstract class Payment { ... }
// SQL: SELECT p.*, cp.card_last4 FROM payment p JOIN card_payment cp ON p.id = cp.id

// TABLE_PER_CLASS — each class has its own complete table, no shared table
@Entity
@Inheritance(strategy = InheritanceType.TABLE_PER_CLASS)
public abstract class Payment { ... }
// SQL for polymorphic query: UNION ALL across all subclass tables (slow)

Strategy Comparison

StrategyQuery PerformanceSchema QualityNullable ColumnsUse When
SINGLE_TABLEFastest (no JOIN)Poor (nullable cols)ManyDefault choice; shallow hierarchy
JOINEDJOIN per queryClean, normalizedNoneDeep hierarchy; schema clarity required
TABLE_PER_CLASSUNION (slow)Redundant columnsNone per tableRare; avoid for polymorphic queries
@MappedSuperclassN/A (not queryable)CleanNoneShared fields only (auditing base)

@MappedSuperclass — Most Common Use Case

@MappedSuperclass
public abstract class BaseEntity {
    @Id @GeneratedValue(strategy = SEQUENCE)
    private Long id;

    @Version private Long version;

    @CreatedDate @Column(updatable = false) private Instant createdAt;
    @LastModifiedDate private Instant updatedAt;
}

// All entities inherit id, version, timestamps — NO shared table
@Entity public class User extends BaseEntity { String email; }
@Entity public class Product extends BaseEntity { String name; }
Google L6 AngleSINGLE_TABLE is the correct default for JPA inheritance — it avoids JOIN overhead and is sufficient for most business hierarchies. JOINED when a DBA mandates referential integrity across subtype tables. TABLE_PER_CLASS is almost never the right choice — polymorphic queries require UNION ALL, killing query planner efficiency. @MappedSuperclass for shared infrastructure fields (audit, versioning) — used in virtually every production Spring app.

What are the trade-offs of enabling Hibernate L2 cache in a distributed system?
L2_CACHE_DISTRIBUTED
▼

Single Node: L2 Cache is a Win

On a single JVM, L2 cache is pure benefit for read-heavy, rarely-updated entities. A cache hit eliminates the DB round-trip entirely. Country codes, configuration entries, permission definitions — all ideal candidates.

Distributed: The Stale Cache Problem

With multiple application instances, each has its own L2 cache. Update on Node A invalidates A's cache — but Node B's cache still holds the stale entity. Subsequent reads from Node B return old data until the TTL expires.

# Correctness-critical data in a 3-node cluster:
# Node A: saves updated Price(id=1, value=9.99) → L2 cache on A evicted
# Node B: reads Price(id=1) → returns cached 8.99 (stale!)
# Node C: reads Price(id=1) → returns cached 8.99 (stale!)
# Problem: distributed cache without invalidation = stale reads

Solutions for Distributed L2 Cache

OptionHowTrade-off
Distributed L2 (Hazelcast/Infinispan)Cache provider broadcasts invalidations across clusterNetwork overhead; operational complexity
READ_ONLY strategyOnly cache truly immutable dataSafe but limited applicability
Short TTLAccept bounded staleness (e.g. 30s)Simple; only for non-critical data
Redis @CacheableExplicit cache at service layer with @CacheEvictFull control; easier distributed invalidation
Disable L2 cacheUse only DB + Redis for cachingSimpler ops; explicit invalidation
Google L6 AngleIn modern microservices architectures, each service owns its database — the distributed L2 cache problem doesn't arise (only one app touches one DB schema). For monoliths with multiple instances: the L6 answer is to prefer Redis-backed @Cacheable at the service layer over Hibernate L2 cache. Redis gives you a single invalidation point, TTL, distributed eviction, and monitoring — all independently of JPA. Hibernate L2 cache is a leaky abstraction in distributed settings.

What is Spring Data JDBC and when should you choose it over Spring Data JPA?
SPRING_DATA_JDBC
▼

What Spring Data JDBC Is

Spring Data JDBC is a simpler, opinionated persistence framework. No EntityManager, no lazy loading, no first-level cache, no dirty checking, no proxy generation for entities. It executes SQL explicitly when you call a repository method. Aggregate-oriented: each repository is responsible for one aggregate root, and loading the aggregate loads the entire object graph eagerly.

// Spring Data JDBC entity — plain Java, no JPA annotations
@Table("orders")
public class Order {
    @Id private Long id;
    private String status;

    // Collection: automatically loaded with the aggregate (no LAZY concept)
    private Set<OrderItem> items = new HashSet<>();
}

// Repository — same pattern as JPA
public interface OrderRepository extends CrudRepository<Order, Long> {
    List<Order> findByStatus(String status);

    @Query("SELECT * FROM orders o JOIN order_items i ON i.order_id = o.id WHERE o.customer_id = :cid")
    List<Order> findByCustomerId(@Param("cid") Long customerId);
}

DDD Aggregate Alignment

Spring Data JDBC enforces Domain-Driven Design aggregate boundaries. Each aggregate has one repository. Loading an aggregate always loads the entire graph — this is intentional. No partial loading, no lazy initialization exceptions, no OSIV.

JPA vs JDBC Decision Matrix

Choose JPA WhenChoose JDBC When
Complex domain with deep object graphsSimple CRUD-heavy domain
@Version optimistic locking neededDDD aggregate modeling
Hibernate L2 cache benefitMaximum performance / minimal overhead
Rich JPQL / Criteria queriesTeam struggles with JPA complexity
Legacy schema with complex mappingsGreenfield microservice with clear boundaries
Bidirectional relationshipsImmutable entities (records)

Performance Characteristics

// JPA: load 1000 entities
// - Open EntityManager
// - Execute SELECT *
// - For each row: instantiate entity, store hydrated state snapshot
// - Register all 1000 in persistence context
// - At flush: dirty check all 1000

// Spring Data JDBC: load 1000 entities
// - Execute SELECT *
// - For each row: instantiate via RowMapper
// - No persistence context, no snapshots, no dirty checking
// - Done. Zero managed state overhead.
Google L6 Angle"JPA is powerful but complex; Spring Data JDBC is simple but requires you to be explicit." The L6 answer frames this as a team and domain complexity decision. New microservices with clearly bounded aggregates and a team that keeps hitting JPA surprises (N+1, lazy init, detached entity) are better served by Spring Data JDBC. JPA is the right choice when your domain has rich associations, complex query needs, and the team has the expertise to manage JPA correctly.
