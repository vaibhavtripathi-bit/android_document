Core Concepts You Need Before Building a Backend

This lesson connects the pieces you have already seen: project files, built-in modules, external packages, and asynchronous thinking. These are the ideas that make Node.js feel less magical and more predictable.

Built-In Modules

Node.js comes with built-in modules. You do not need to install them. Common examples are fs for files, path for file paths, and http for basic web servers. 

const path = require("path");

console.log(path.join("users", "asha", "notes.txt"));

This example uses require, which is a CommonJS module style. You will see it in many Node.js codebases.

CommonJS and ESM

There are two common module systems in Node.js: CommonJS and ESM. Beginners should know both exist. Existing projects often use CommonJS. Newer projects may use ESM.

CommonJS

const fs = require("fs");

Very common in older and still many current Node projects.

ESM

import fs from "node:fs";

Modern import syntax. Often used in newer projects.

Beginner advice: you do not need to fight this topic on day one. Understand that both styles exist. Read the project you are in and follow that style consistently.

package.json Matters More Than Beginners Think

This file tells Node tooling what your project is and how it should behave. A simple example: 

{
  "name": "nodejs-learning",
  "version": "1.0.0",
  "main": "app.js",
  "scripts": {
    "start": "node app.js"
  }
}

The scripts section is especially useful. Instead of memorizing long terminal commands, you can run named project commands with npm run.

What npm Actually Does

  • Installs external packages
  • Tracks them in package.json
  • Creates a node_modules folder
  • Runs project scripts such as npm run start

Example of installing a package: 

npm install chalk

That package is external, unlike built-in modules such as path or fs.

Asynchronous Thinking

Backend code often waits for things: database results, API calls, file reads, cache lookups, and network responses. That is why async thinking is central in Node.js.

Simple Async Mental Model

Start Work
→
Wait for Result
→
Continue Later

Callbacks, Promises, and async/await

Older Node.js code often used callbacks. Modern code commonly uses promises and async/await because they are easier to read and combine.

Callback style

doSomething((error, result) => {
  if (error) {
    console.error(error);
    return;
  }

  console.log(result);
});

Promise style

doSomething()
  .then((result) => console.log(result))
  .catch((error) => console.error(error));

async/await style

async function run() {
  try {
    const result = await doSomething();
    console.log(result);
  } catch (error) {
    console.error(error);
  }
}

Why await Feels Easier

await lets asynchronous code look closer to step-by-step logic. That matters because backend programs often do several things in sequence, and you want the code to stay readable.

Important: await does not make code synchronous in a global sense. It only makes your code easier to write and read.

Process and Environment Variables

In backend applications, configuration often comes from environment variables. Node gives access through process.env. 

const port = process.env.PORT || 3000;
console.log("Running on port:", port);

This is common because production and local environments usually do not use exactly the same settings.
