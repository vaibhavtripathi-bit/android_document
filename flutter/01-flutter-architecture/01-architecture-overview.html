Architecture Overview

Flutter's architecture is built on three distinct layers: the Framework (Dart), the Engine (C++), and the Embedder (platform-specific). Understanding this layered approach is fundamental to mastering Flutter internals.

1. High-Level Architecture

Flutter is structured as a layered system where each layer builds upon the one below. Unlike React Native or other cross-platform frameworks, Flutter does not rely on platform UI components — it renders every pixel directly using its own graphics engine.

Architect Insight

Flutter's "own rendering" approach is its key differentiator. While React Native bridges to platform UI components and Xamarin uses platform renderers, Flutter draws every pixel with Skia/Impeller. This gives Flutter pixel-perfect consistency across platforms but means it cannot automatically inherit platform-specific UI behaviors — they must be explicitly implemented.

2. Framework Layer (Dart)

The Framework layer is written entirely in Dart and provides the high-level APIs that developers interact with. It's organized into sub-layers from high-level to low-level:

Sub-Layer Package Responsibility
Material / Cupertino material.dart, cupertino.dart Design system implementations (buttons, cards, navigation)
Widgets widgets.dart Compositional UI primitives (StatelessWidget, StatefulWidget)
Rendering rendering.dart Layout and painting (RenderObject, RenderBox)
Animation animation.dart Animation primitives (Tween, AnimationController)
Painting painting.dart Drawing primitives (Canvas wrappers, decorations)
Gestures gestures.dart Gesture recognition (tap, drag, scale)
Foundation foundation.dart Core utilities (ChangeNotifier, diagnostics) 
// Framework layer example: Widget → Element → RenderObject
class Container extends StatelessWidget {
  final Widget? child;
  final Color? color;
  final EdgeInsets? padding;

  @override
  Widget build(BuildContext context) {
    Widget current = child ?? const SizedBox();
    
    if (padding != null) {
      current = Padding(padding: padding!, child: current);
    }
    if (color != null) {
      current = ColoredBox(color: color!, child: current);
    }
    return current;
  }
}

// Container is a convenience widget that composes:
// Padding → ColoredBox → Align → ConstrainedBox → DecoratedBox
// Each is a simple single-purpose widget

3. Engine Layer (C++)

The Engine is written in C++ and provides the low-level implementation of Flutter's core capabilities. It's approximately 300,000 lines of C++ code.

3.1 Engine Components

Component Description
Skia 2D graphics library (also used by Chrome, Android). Handles all drawing operations.
Dart Runtime Executes Dart code. Includes JIT compiler (debug) and runs AOT code (release).
Text Rendering (libTxt) Text layout and shaping using HarfBuzz, ICU, and platform-specific font handling.
Platform Channels Binary message passing between Dart and native platform code.
Shell Coordinates between Dart and platform, manages surfaces and threads.

4. Embedder Layer

The Embedder is the platform-specific code that hosts the Flutter Engine. Each platform has its own embedder that handles:

  • Surface creation — Providing a rendering surface (Metal, OpenGL, Vulkan)
  • Input routing — Converting platform touch/mouse events to Flutter pointer events
  • Lifecycle management — Handling app foreground/background transitions
  • Accessibility — Bridging to platform accessibility services
  • Platform views — Embedding native platform widgets in Flutter
// iOS Embedder example (simplified FlutterViewController)
@interface FlutterViewController : UIViewController
- (void)viewDidLoad {
    [super viewDidLoad];
    
    // Create Flutter engine
    self.engine = [[FlutterEngine alloc] initWithName:@"main"];
    [self.engine run];
    
    // Create Metal rendering surface
    self.flutterView = [[FlutterView alloc] 
        initWithEngine:self.engine 
        renderingAPI:FlutterRenderingAPIMetal];
    
    // Set up input handling
    [self.view addGestureRecognizer:...];
}

// Forward touch events to Flutter
- (void)touchesBegan:(NSSet *)touches withEvent:(UIEvent *)event {
    [self.engine.viewController dispatchTouches:touches phase:UITouchPhaseBegan];
}
@end

// Android Embedder example (simplified FlutterActivity)
class FlutterActivity : Activity() {
    private lateinit var flutterEngine: FlutterEngine
    private lateinit var flutterView: FlutterView
    
    override fun onCreate(savedInstanceState: Bundle?) {
        super.onCreate(savedInstanceState)
        
        // Create and run Flutter engine
        flutterEngine = FlutterEngine(this)
        flutterEngine.dartExecutor.executeDartEntrypoint(
            DartExecutor.DartEntrypoint.createDefault()
        )
        
        // Create rendering surface (uses SurfaceView internally)
        flutterView = FlutterView(this)
        flutterView.attachToFlutterEngine(flutterEngine)
        setContentView(flutterView)
    }
}

5. Layer Communication

5.1 Dart to Engine Communication

Dart code communicates with the Engine through dart:ui, which exposes native methods that call into C++ Engine code:

// dart:ui provides the bridge between Dart and Engine
// This is a "native" call implemented in C++

// In Dart (dart:ui/window.dart)
void render(Scene scene) {
  _render(scene);  // Native call to Engine
}

@Native("Window_render")
external void _render(Scene scene);

// In C++ (engine/lib/ui/window/window.cc)
void Window_render(Dart_NativeArguments args) {
    Scene* scene = GetNativeScene(args);
    UIDartState::Current()
        ->platform_configuration()
        ->client()
        ->Render(scene->takeLayerTree());
}

6. Comparison with Other Frameworks

Aspect Flutter React Native Native (iOS/Android)
Rendering Custom (Skia/Impeller) Platform UI components Platform UI components
Language Dart (AOT compiled) JavaScript (interpreted) Swift/Kotlin (compiled)
Bridge Platform channels (async) JS Bridge (JSON, async) N/A (native)
UI Consistency Pixel-perfect cross-platform Platform-native look Platform-native look
Hot Reload Sub-second (JIT) Fast refresh (JS) Slow (recompile)
App Size ~5-15MB overhead ~7-15MB overhead Minimal
Key Difference

Flutter's custom rendering means it doesn't need a "bridge" for UI operations like React Native does. The only bridge-like communication is through Platform Channels for accessing native APIs (camera, GPS, etc.), which is typically much less frequent than UI updates.

7. Architect Considerations

7.1 When Flutter's Architecture Shines

  • Custom UI designs — Complex animations, custom widgets render identically everywhere
  • Consistent brand experience — Pixel-perfect UI across iOS, Android, Web, Desktop
  • High-performance graphics — Games, data visualizations, complex animations
  • Rapid iteration — Hot reload enables sub-second UI changes

7.2 When Flutter's Architecture Challenges

  • Platform-native feel — Material on iOS may feel foreign to users
  • Native API access — Requires platform channels (async, serialization overhead)
  • Platform views — Embedding native views (maps, webview) has performance cost
  • App size — Flutter runtime adds ~5MB minimum to app size
Staff-Level Mental Model

Think of Flutter as a game engine for apps. Like Unity or Unreal, Flutter takes full control of rendering rather than delegating to platform UI toolkits. This gives complete freedom and consistency but requires implementing platform behaviors explicitly. The trade-off is fundamental: you gain pixel-perfect control but lose automatic platform conventions. For L6 interviews, understand both when this trade-off benefits your architecture and when it creates friction.
