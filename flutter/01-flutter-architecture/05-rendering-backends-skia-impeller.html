Skia Architecture and the Shader Compilation Jank Problem

How Skia worked as Flutter's original renderer and why runtime shader compilation was its fundamental limitation.

1. Skia in Flutter

Skia is an open-source 2D graphics library developed by Google. It was Flutter's original and primary rendering backend from Flutter 1.0 through Flutter 3.x, and remains available as a fallback backend. Skia provides a high-level drawing API that abstracts over OpenGL ES, Vulkan, Metal, and software rasterization backends.

SkCanvas API and Display Lists

// Flutter's framework ultimately calls dart:ui's Canvas,
// which is a thin Dart wrapper around SkCanvas:

void paint(Canvas canvas, Size size) {
  // Each of these translates to a Skia draw call:
  canvas.drawRect(
    Rect.fromLTWH(0, 0, size.width, size.height),
    Paint()..color = Colors.blue,
  );

  canvas.drawPath(
    Path()..addOval(Rect.fromCircle(center: Offset(100, 100), radius: 50)),
    Paint()..color = Colors.red..style = PaintingStyle.stroke,
  );

  canvas.drawImage(image, Offset.zero, Paint());
}

// These calls do NOT immediately issue GPU commands.
// They are recorded into a DisplayList (sequence of DlOp structs).
// The DisplayList is passed from the UI runner to the GPU runner.
// The GPU runner "plays back" the DisplayList against an SkCanvas
// backed by the actual GPU surface.
//
// This separation enables:
// 1. UI thread builds the frame without touching GPU state
// 2. GPU thread renders without re-running Dart code
// 3. Caching: DisplayLists can be replayed multiple times (RepaintBoundary)

SkPicture Recording and Layer Caching

// RepaintBoundary in Flutter maps to OffscreenSurface caching in Skia:
//
// RepaintBoundary widget → creates a new Layer in the Layer tree
// When the layer's content hasn't changed, the Rasterizer reuses
// the previously rasterized result (a cached SkImage / GPU texture)
// instead of re-rasterizing.
//
// Under the hood:
// 1. First frame: Layer is rasterized to an offscreen texture
//    (GPU memory allocated, SkSurface created, DisplayList replayed)
// 2. Subsequent frames (if unchanged): cached texture composited directly
//    → Skips re-rasterization for that subtree
//    → O(1) GPU work for the cached layer regardless of complexity
//
// This is why RepaintBoundary is crucial for scroll performance:
ListView(
  children: children.map((item) =>
    RepaintBoundary(   // Cache each item; only re-rasterize when data changes
      child: ItemWidget(item: item),
    )
  ).toList(),
)
// Without RepaintBoundary: every scroll frame re-rasterizes ALL visible items
// With RepaintBoundary: scroll frame composites pre-cached item textures → fast

2. The Shader Compilation Jank Problem

Shader compilation jank was Skia's most significant and long-standing limitation in Flutter. Understanding it deeply — both the root cause and why it was so hard to fix within Skia — is essential at the L6 level.

What Are GPU Shaders?

A GPU shader is a program that runs on the GPU. To render any non-trivial visual effect (rounded corners, blurs, gradients, shadows, image filtering), the GPU needs a compiled shader program. Compiling a shader involves translating a high-level shader language (GLSL, SKSL, MSL, SPIRV) into the GPU-vendor-specific binary instruction set (a process analogous to CPU compilation but done at runtime by the GPU driver).

// Shader compilation cost is significant:
// Simple shape shader: ~0.5–2ms compilation
// Complex blur/filter shader: ~5–20ms compilation
// Complex path with multiple effects: can reach 50–100ms
//
// GPU driver shader compilation is:
// - Synchronous by default (blocks the calling thread)
// - Non-deterministic in duration (varies by GPU vendor/driver version)
// - Not parallelizable with the GPU's rendering work
// - NOT cacheable across app restarts on all Android GPU drivers
//   (Adreno drivers: limited persistent cache; Mali: no persistent cache on older)

// SKSL: Skia's intermediate shader language
// Skia writes shaders in SKSL, then translates to the target language:
// SKSL → GLSL (for OpenGL ES)
// SKSL → SPIRV → vendor binary (for Vulkan)
// SKSL → MSL → Metal library (for Metal on iOS)
//
// The translation + GPU driver compilation happens at DRAW TIME:
// On the first call to canvas.drawPath() with a complex paint,
// Skia discovers it needs a shader it hasn't compiled yet.
// → Shader compilation starts synchronously
// → GPU command submission blocks until compilation finishes
// → Frame is delayed → JANK

Why Skia Cannot Pre-Compile Shaders

// Skia generates shaders dynamically based on the exact combination
// of paint attributes used at draw time. The shader variant is
// determined by the runtime combination of:
// - BlendMode (kSrcOver, kMultiply, kColorBurn, ...)
// - ColorFilter presence and type
// - ImageFilter presence and type (blur, shadow, etc.)
// - MaskFilter presence (for text shadows)
// - Shader type (linear gradient, radial gradient, image shader, ...)
// - Antialiasing mode
// - Coverage (partial pixel coverage from path rasterization)
//
// The number of possible shader variants is combinatorially large.
// At build time, you cannot know WHICH variants your app will need.
// You would need to enumerate all possible Paint() combinations —
// which is effectively impossible for a general-purpose renderer.
//
// Flutter attempted to address this with the SkSL warmup approach:

// SkSL warmup (imperfect workaround):
// 1. Run the app on a device, capture which shaders get compiled
//    (using "--dump-skp-on-shader-compilation" engine flag)
// 2. Export the shader cache as a JSON blob
// 3. Bundle the shader blob with the app
// 4. At startup, "warm up" Skia by submitting the bundled shaders
//    for pre-compilation before the first frame
//
// Problems with this approach:
// - Device-specific: shaders compiled for one GPU (Adreno 650)
//   may not match another GPU (Mali G78) → warmup may not help
// - Fragile: Any paint attribute change invalidates the cached variant
// - Extra startup time: warmup itself takes ~100-500ms
// - Maintenance burden: must re-capture and update for each app version
Warning

Shader compilation jank disproportionately affects first-time interactions in Flutter apps: the first time a complex animation plays, the first scroll through a list with rich item designs, the first page transition. This is because the shader cache is cold on first run. Users often perceive a Flutter app as "janky" specifically because of this first-interaction stutter. This was the primary motivation for building Impeller from scratch.

Impeller: Architecture, Pipeline Objects, and GPU Backends

How Impeller eliminates shader jank by design, its internal architecture, and the tradeoffs of each GPU API backend.

3. Impeller: Flutter's New Rendering Engine

Impeller is Flutter's new rendering backend, built from scratch with a single overriding design constraint: eliminate shader compilation jank entirely. It was first shipped on iOS (Flutter 3.0, stable 3.10) and then on Android (stable 3.16).

The Core Design: Pre-Compiled Pipeline State Objects

Impeller's approach is fundamentally different from Skia's. Instead of generating shaders dynamically at runtime, Impeller pre-compiles a fixed, known set of shaders at app build time and bundles the compiled GPU programs directly into the app binary.

// Impeller's shader pre-compilation at build time:
//
// Impeller's shaders are written in GLSL (with Impeller extensions)
// and stored in flutter/impeller/entity/shaders/
//
// During flutter build:
// 1. impellerc (Impeller shader compiler) compiles each .glsl file
//    to the target format:
//    iOS/macOS:  .glsl → .metallib (pre-compiled Metal library)
//                      → embedded in the app bundle as a .metallib asset
//    Android:    .glsl → SPIRV → offline SPIRV optimization
//                      → stored as SPIRV blobs in the app's APK
//
// 2. At Flutter app startup:
//    iOS:    MTLLibrary loaded from the pre-compiled .metallib
//            MTLRenderPipelineState objects created once at startup
//    Android: SPIRV blobs loaded, GPU driver compiles to native code
//             (first-ever run on device → driver caches the result)
//             Subsequent runs: driver loads from its persistent cache
//
// Key insight: The number of shaders Impeller needs is FIXED and BOUNDED.
// Impeller's rendering model is designed so that all visual effects
// can be expressed using a predetermined set of ~100 shader programs.
// This is possible because Impeller uses a different rendering strategy
// than Skia (see: tessellation and entity model below).

// Checking Impeller status in your app:
// flutter run --enable-impeller    (force enable)
// flutter run --no-enable-impeller (force Skia)
// In release mode: Impeller is default on iOS (Flutter 3.10+)
//                  and Android (Flutter 3.16+)

Pipeline State Objects (PSOs)

// PSO = Pipeline State Object
// A PSO describes the COMPLETE GPU pipeline state:
// - Vertex shader (program)
// - Fragment shader (program)
// - Blend mode configuration
// - Depth/stencil state
// - Vertex buffer layout (attribute descriptions)
// - Color attachment pixel format
// - Sample count (MSAA level)
//
// Creating a PSO is expensive (10–100ms) because it triggers
// shader compilation and GPU pipeline linking.
// Impeller creates ALL its PSOs at startup — never during rendering.

// ContentContext: Impeller's PSO cache
// flutter/impeller/entity/content_context.h
class ContentContext {
  // All PSOs created at startup and reused:
  std::shared_ptr<Pipeline> solid_fill_pipeline_;       // drawRect with solid color
  std::shared_ptr<Pipeline> linear_gradient_fill_;      // gradient fills
  std::shared_ptr<Pipeline> texture_fill_pipeline_;     // drawImage
  std::shared_ptr<Pipeline> rrect_blur_pipeline_;       // rounded rect shadows
  std::shared_ptr<Pipeline> gaussian_blur_pipeline_;    // blur filter
  std::shared_ptr<Pipeline> border_mask_blur_pipeline_; // border blurs
  std::shared_ptr<Pipeline> glyph_atlas_pipeline_;      // text rendering
  // ... ~80 more PSOs
};
// At render time: ContentContext.GetPipeline(type) → O(1) lookup, no compilation

Tessellation: Impeller's Path Strategy vs Skia

// Skia's path rendering approach:
// Skia uses the GPU's stencil buffer for path filling (stencil-and-cover).
// Complex paths are rasterized using a stencil technique:
// 1. Fill stencil buffer: draw triangles that increment/decrement stencil
// 2. Cover pass: draw bounding box, discard pixels where stencil = 0
// This requires TWO render passes and dynamic shader generation based on path complexity.

// Impeller's path rendering approach:
// Impeller TESSELLATES paths on the CPU into triangles.
// Complex paths (Bezier curves, arcs) are converted to triangle fans
// before being sent to the GPU.
// The GPU then renders the triangles with a simple solid-color shader
// (no complex stencil logic, no multi-pass).
//
// Tradeoffs:
// PRO: Simple, fixed shaders → no dynamic shader compilation
//      Single render pass for most paths
//      Consistent performance across GPU vendors
// CON: CPU tessellation cost for very complex paths
//      Quality at low zoom levels (tessellation artifacts)
//      Anti-aliasing requires special handling (coverage via analytical AA)

// Impeller's analytical anti-aliasing for lines and rectangles:
// Instead of MSAA (expensive GPU samples), Impeller computes coverage
// analytically in the fragment shader for known simple shapes:
// - Rectangles: axis-aligned → exact 1-pixel feathering via math
// - Rounded rects: SDF (Signed Distance Field) evaluation in fragment shader
// - Lines: coverage computed from distance to line endpoints
// This gives crisp AA without the 2x-4x memory/bandwidth cost of MSAA
// for these common cases.

Impeller's Stencil-Then-Cover for Complex Clips

// When Flutter widgets clip their children (ClipRRect, ClipPath),
// Impeller needs to restrict rendering to the clip region.
//
// Impeller uses an EntityPass with a stencil buffer:
//
// EntityPass for a SaveLayer with clip:
// 1. Create offscreen texture (if needed for blending/filtering)
// 2. Stencil setup: write 1 to stencil within clip boundary
//    - For ClipRRect: tessellate the rounded rect, fill stencil
//    - For ClipPath: tessellate the path, fill stencil
// 3. Render all child entities with stencil test = equal(1)
//    → Children are automatically clipped to the stencil region
// 4. Stencil clear (or depth decrement for nested clips)
//
// This is a key architectural difference from Skia's general stencil approach:
// Impeller uses stencil for CLIPS ONLY, not for general path fills.
// Path fills use CPU tessellation → vertex buffers → triangle draw.
// This clean separation keeps the shader set fixed and bounded.

5. GPU API Backends

Metal (iOS and macOS)

// Metal is Apple's GPU API (introduced iOS 8 / macOS 10.11).
// Impeller uses Metal as the primary backend on iOS/macOS.

// Metal advantages for Impeller:
// 1. MTLRenderPipelineState == Impeller's PSO concept maps directly
//    Metal's pipeline state compilation is explicit and offline-capable
// 2. MTLLibrary can be pre-compiled to .metallib at build time
//    → impellerc compiles shaders → .metallib shipped in app bundle
//    → At startup: newLibraryWithData() loads pre-compiled metallib (fast!)
//    → ZERO shader compilation cost at runtime
// 3. Metal's command buffer model maps well to Impeller's render pass model
// 4. Apple Silicon GPUs are optimized for Metal → best performance on iOS

// Metal pipeline creation in Impeller (simplified):
// At startup (ContentContext initialization):
auto library = device->NewLibraryWithData(metallib_data);
auto vertex_fn = library->GetFunction("SolidFillVertexMain");
auto fragment_fn = library->GetFunction("SolidFillFragmentMain");

MTLRenderPipelineDescriptor* desc = [MTLRenderPipelineDescriptor new];
desc.vertexFunction = vertex_fn;
desc.fragmentFunction = fragment_fn;
desc.colorAttachments[0].pixelFormat = MTLPixelFormatBGRA8Unorm;
// ... blend state, depth/stencil, sample count ...

NSError* error;
id<MTLRenderPipelineState> pso =
    [device newRenderPipelineStateWithDescriptor:desc error:&error];
// This compilation is FAST because shaders are pre-compiled in .metallib.
// If using source strings: this would compile at runtime → slow.

Vulkan (Android)

// Vulkan is available on Android 7.0+ (API 24+).
// Impeller uses Vulkan as the primary backend on Android 7+.

// Vulkan and Impeller:
// 1. VkPipeline == PSO concept
//    VkPipeline creation requires SPIRV shader modules
//    Impeller ships pre-compiled SPIRV blobs in the APK
// 2. First-ever VkPipeline creation on a device:
//    GPU driver compiles SPIRV to native GPU ISA → takes 10-100ms
//    But: GPU drivers maintain a PERSISTENT PIPELINE CACHE across app runs
//    → Second launch: driver loads cached pipelines → fast startup
// 3. Pipeline cache management:
//    Impeller uses VkPipelineCache with serialization to disk
//    → vkGetPipelineCacheData() → save to app's data directory
//    → vkCreatePipelineCache(initialData) on next launch
//    → Eliminates recompilation cost on subsequent launches
//    → Even first launch jank is now moved to app startup, not frame rendering

// Vulkan explicit memory management:
// Unlike Metal (semi-automatic) or OpenGL (automatic), Vulkan requires
// explicit GPU memory allocation via VkDeviceMemory.
// Impeller uses a resource manager to pool and reuse GPU allocations:
// - Vertex buffers: pooled and recycled frame-to-frame
// - Texture memory: tracked with explicit lifetimes
// - Transient attachments: allocated and freed within a single render pass

// Android Vulkan availability check at runtime:
// Impeller checks: android.os.Build.VERSION.SDK_INT >= 24
// AND the Vulkan API is available (vkEnumerateInstanceVersion() success)
// Falls back to OpenGL ES if Vulkan is unavailable or has known issues

OpenGL ES (Fallback)

// OpenGL ES is Impeller's fallback backend for:
// - Android devices below API 24 (Android 7.0)
// - Devices with known Vulkan driver bugs
// - Some emulators that don't support Vulkan
//
// Impeller's OpenGL ES backend is less mature than Metal/Vulkan.
// Key limitations:
// 1. OpenGL ES does NOT support offline shader compilation to a cross-device format
//    GLSL must be compiled by the GPU driver at runtime for OGL ES
//    → Impeller's GLSL shaders are still compiled at startup on first run
//    → But: compiled once at startup (not during frames) → much better than Skia
// 2. No explicit pipeline state concept
//    OpenGL ES state machine requires careful state tracking
//    Impeller manages this via a state cache layer
// 3. Limited to OpenGL ES 3.0+ feature set
//
// Impeller + OpenGL ES is still better than Skia + OpenGL ES because:
// All shader compilations happen at startup (one time)
// Not scattered across random frames during user interaction

// Flutter will likely eventually drop OpenGL ES support entirely
// as Android 7+ adoption approaches 100%. The "Impeller on GLES"
// backend is primarily a transitional bridge.

Dawn / WebGPU (Future: Flutter Web)

// Dawn is Google's implementation of the WebGPU standard.
// WebGPU is the next-generation web graphics API.
//
// Flutter Web currently uses:
// - CanvasKit: Skia compiled to WebAssembly (WASM)
// - HTML renderer: maps Flutter painting to HTML/CSS (legacy)
//
// Future direction:
// Impeller + Dawn/WebGPU for Flutter Web would provide:
// - Same Impeller rendering model on web as mobile
// - Pre-compiled shaders (WGSL, WebGPU's shader language)
// - Consistent rendering behavior across platforms
// - Better performance than CanvasKit (Impeller is more efficient than Skia WASM)
//
// As of Flutter 3.24+, there is experimental work on an Impeller
// web backend using Dawn. Not production-ready as of 2024.

How Display Lists Enable RepaintBoundary Caching

// RepaintBoundary creates a new CompositedLayer in the Layer tree.
// Each CompositedLayer has its own DisplayList.
//
// Rendering optimization:
// 1. Layer is marked dirty → its DisplayList is re-recorded
//    (Dart's paint() methods are called again for that subtree)
// 2. Layer is clean → its CACHED TEXTURE is composited directly
//    → No DlOp recording, no backend rendering work
//
// This is why RepaintBoundary is especially valuable for:
// - ScrollView items (only items entering/leaving viewport are dirty)
// - Animated widgets (animation invalidates only the animated layer)
// - Static complex widgets (rendered once, composited forever)
//
// Cache invalidation is explicit in Flutter:
// - setState() → marks descendant layers dirty
// - GlobalKey reassignment → forces full re-layout + repaint
// - Opacity changes via Opacity widget → marks dirty (unless using
//   AnimatedOpacity which uses the compositor's opacity property)
//
// Staff-level optimization: AnimatedOpacity changes the LAYER's opacity
// property (handled by the compositor at zero Dart/CPU cost),
// whereas changing a Color that includes alpha requires re-recording
// the display list → much more expensive.

Staff-Level Insight

The Impeller transition represents a fundamental architectural shift in how Flutter thinks about rendering reliability. Skia's dynamic shader generation is powerful but unpredictable. Impeller trades some rendering flexibility for complete predictability — every frame takes the same code path, every GPU operation uses a pre-validated pipeline. This is the correct engineering choice for a UI framework targeting 120fps on modern mobile hardware. At the Staff level, when advising on Flutter adoption, the correct statement is: "Impeller eliminates the class of jank that was Skia's worst-case behavior, at the cost of minor limitations in advanced shader effects — a trade-off almost all production apps should embrace."
