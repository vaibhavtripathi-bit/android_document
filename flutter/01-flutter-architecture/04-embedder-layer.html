Part 1 — L4/L5 Foundations
What the Embedder Is and Why It Exists

The Flutter engine is pure C++ with no platform dependencies. The embedder is the thin adapter layer that connects it to each OS.

1. What Is the Embedder?

The Flutter engine (flutter/engine repository) is a platform-agnostic C++ library. It contains the Dart VM, the Skia/Impeller renderer, the text engine (HarfBuzz + ICU), platform channel dispatch, and the frame scheduling pipeline. However, it has no knowledge of Android, iOS, Windows, or any other OS.

The embedder is the layer that makes Flutter run on a specific platform. It provides the engine with exactly the things the engine needs but cannot supply itself:

  • A rendering surface (an OpenGL context, a Metal surface, a Vulkan swapchain)
  • A vsync signal (to drive the animation loop)
  • Input events (touch, mouse, keyboard) translated into Flutter's event model
  • A threading model (which threads to use for which task runners)
  • Asset loading (reading files from the app bundle)
  • Accessibility bridges (routing Flutter semantics to platform accessibility APIs)
  • Platform channel dispatch (routing Dart ↔ native method calls)

2. Shell Architecture

The Shell is the central coordinator class inside the Flutter engine (shell/common/shell.h). Every Flutter engine instance has exactly one Shell. It owns the major engine subsystems and mediates communication between them across task runner boundaries.

PlatformView: The Embedder's Interface to the Shell

The PlatformView abstract class in the engine defines the contract that every embedder must implement. It is the boundary between the engine's platform-agnostic C++ and the platform-specific embedder code.

// engine/shell/common/platform_view.h (simplified)
class PlatformView {
 public:
  // Called by engine when a frame is ready to present
  virtual void ScheduleFrame();

  // Returns the rendering surface for the current frame
  virtual std::unique_ptr<Surface> CreateRenderingSurface() = 0;

  // Embedder calls this to deliver input events
  void DispatchPointerDataPacket(std::unique_ptr<PointerDataPacket> packet);

  // Platform channel message delivery to Dart
  void DispatchPlatformMessage(std::unique_ptr<PlatformMessage> message);

  // Accessibility: engine calls this when semantics tree changes
  virtual void UpdateSemantics(SemanticsNodeUpdates updates,
                               CustomAccessibilityActionUpdates actions) = 0;

  // Called when Dart wants to send a platform message reply
  virtual void HandlePlatformMessage(
      std::unique_ptr<PlatformMessage> message) = 0;
};

// Android embedder implements PlatformView as PlatformViewAndroid
// iOS embedder implements PlatformView as PlatformViewIOS

3. Task Runner Architecture

Flutter's engine uses four dedicated task runners. Understanding which work happens on which runner — and the threading constraints — is essential for diagnosing Flutter performance problems.

Flutter Task Runner Model

  │ • Platform msgs  │  │ • Dart execution │  │ • Skia/Impeller  │

  └────────────────────────────────────────┘

  NOTE: Task Runners are NOT the same as threads.
  A task runner is an abstraction (a task queue + executor).
  On desktop platforms, the UI and GPU runners may share a thread.
  On mobile, they typically use separate threads for parallelism.

Platform Task Runner

Runs on the platform's main thread (Android's main looper thread, iOS's main run loop thread). All Android View operations, lifecycle callbacks, and plugin method channel calls happen here. The engine must not block this thread, as the platform's ANR watchdog monitors it.

// Android: Platform runner = main thread (Looper.getMainLooper())
// All Flutter plugin callbacks arrive on this thread:

public class MyPlugin implements MethodCallHandler {
  @Override
  public void onMethodCall(MethodCall call, Result result) {
    // THIS RUNS ON THE PLATFORM TASK RUNNER (main thread)
    // Do NOT do heavy work here — post to background thread instead
    if (call.method.equals("getLocation")) {
      executor.submit(() -> {
        Location loc = locationManager.getLastKnownLocation(...);
        // Must reply on the platform thread:
        mainHandler.post(() -> result.success(loc.toMap()));
      });
    }
  }
}

UI Task Runner

The UI task runner runs the Dart isolate. All Dart code — widget builds, animation callbacks, gesture handlers, async callbacks — executes on this runner. It is the only runner that is allowed to execute Dart code.

// The UI runner is single-threaded for a given isolate.
// All of the following happen serially on the UI runner:

// 1. SchedulerBinding.handleDrawFrame() — triggered by vsync
// 2. Widget tree rebuild (setState callbacks, FutureBuilder.builder, etc.)
// 3. RenderObject layout and paint
// 4. Layer tree composition (SceneBuilder calls)
// 5. Dart async microtasks and event loop processing
// 6. Platform channel message dispatch (via BinaryMessenger)

// This is why:
// - Blocking the UI runner with synchronous I/O causes jank
// - Heavy JSON parsing on the main isolate causes dropped frames
// - Use compute() or Isolate.run() to offload expensive work

// The UI runner sends the completed Layer tree (Scene) to:
// window.render(scene) → crosses to GPU runner via thread-safe queue

GPU / Raster Task Runner

The GPU (Raster) task runner receives the Layer tree from the UI runner and translates it into actual GPU commands via Skia or Impeller. It owns the GPU context and all GPU objects (textures, framebuffers, pipeline state objects).

// GPU Runner responsibilities:
// 1. Receives LayerTree from UI runner
// 2. Rasterizer::Draw():
//    a. Acquires a surface from the embedder (SurfaceFrame)
//    b. Creates a SkCanvas / Impeller Canvas backed by the surface
//    c. Traverses the LayerTree, calling each Layer's Paint() method
//    d. Layer.Paint() → issues Skia/Impeller drawing commands
//    e. Presents the frame (swapbuffers / commitCommandBuffer)
// 3. Manages the GPU resource cache (texture cache, path cache)

// KEY CONSTRAINT: GPU objects (textures, SkImage, etc.) can ONLY be
// created and accessed from the GPU runner thread.
// This is enforced by Skia's thread model.

// Dart code interacts with GPU resources indirectly:
// ui.Image lives in Dart heap but holds a reference to GPU texture
// Deleting a ui.Image in Dart schedules texture deletion on GPU runner
// This is why: image.dispose() should be called explicitly for large images
// (waiting for GC to collect ui.Image → texture cleanup is delayed)

IO Task Runner

The IO runner handles work that is expensive but doesn't need to happen on the UI or GPU runners. Its primary role is image decoding and uploading decoded image data to the GPU as textures.

// Image loading pipeline — IO runner's job:
//
// 1. dart:ui Image.fromAsset() or NetworkImage call
// 2. IO runner: read compressed image bytes (JPEG/PNG/WebP) from disk/network
// 3. IO runner: decode compressed bytes to raw RGBA pixels
//    (uses platform image codecs: libjpeg-turbo, libpng, libwebp)
// 4. IO runner: upload RGBA pixels to GPU as a texture
//    (uses a shared EGL/Metal context that shares resources with GPU runner)
// 5. Returns a ui.Image handle to Dart (UI runner)
//
// The IO runner has a SHARED GPU CONTEXT with the GPU runner,
// meaning it can upload textures without touching the GPU runner.
// This is a critical optimization: large image uploads don't block rendering.

// Why this matters for large images:
// Decoding a 4K image: ~20-50ms CPU time
// Without IO runner: this 50ms would block the UI runner → jank every scroll
// With IO runner: decoding happens asynchronously; UI runner gets handle
//                 only after texture is GPU-ready
Staff-Level Insight

The task runner model is the core reason Flutter can maintain smooth rendering while handling platform events. The platform runner (main thread) is kept free for Android/iOS system events. The UI runner handles only Dart execution. The GPU runner does only GPU work. The IO runner handles slow I/O. When Flutter jank occurs, the first diagnostic question is: which runner is overloaded? The DevTools Timeline view labels events by runner. A "frame build" spike on the UI runner means too much Dart work per frame. A "frame rasterize" spike on the GPU runner means complex render operations. Identifying the runner is step one of any serious Flutter performance investigation.

Entry Points: FlutterActivity and FlutterFragment

// Android embedder entry point hierarchy:
//
// FlutterActivity (android.app.Activity)
//   └── delegates to FlutterActivityAndFragmentDelegate
//         └── creates/uses FlutterEngine
//               └── wraps FlutterJNI (JNI bridge to C++ engine)
//                     └── calls into libflutter.so

// FlutterFragment (androidx.fragment.app.Fragment)
//   └── same delegation pattern
//         └── reuses FlutterEngine from FlutterEngineCache or creates new

// FlutterEngine is the Android-side handle to the C++ Shell:
public class FlutterEngine {
  private final FlutterJNI flutterJNI;         // JNI bridge
  private final DartExecutor dartExecutor;     // method channels
  private final FlutterRenderer renderer;      // surface management
  private final PlatformViewsController pvc;  // AndroidView support
  private final PluginRegistry pluginRegistry; // plugin management

  // Engine creation triggers:
  // 1. Shell creation in C++ (via FlutterJNI.init())
  // 2. Dart VM initialization
  // 3. Root isolate creation (but NOT yet running Dart code)
  // Dart starts running when attachToFlutterEngine() is called
  // and a rendering surface is attached
}

FlutterJNI: The JNI Bridge

FlutterJNI is the lowest-level Android Java class. It has JNI methods that call directly into the C++ engine. All cross-language calls between Java/Kotlin and the C++ engine go through this class.

// FlutterJNI key JNI methods (called from Java to C++):
public class FlutterJNI {
  // Initialize the C++ Shell
  private native void nativeInit(AssetManager assetManager,
                                  String[] commandLineArgs,
                                  String bundlePath,
                                  String appStoragePath,
                                  String engineCachesPath,
                                  long initTimeMillis);

  // Deliver a vsync signal to the engine
  private native void nativeOnVsync(long frameStartTimeNanos,
                                     long frameTargetTimeNanos,
                                     long cookie);

  // Notify engine that a rendering surface is available
  private native void nativeSurfaceCreated(Surface surface);
  private native void nativeSurfaceChanged(int width, int height);
  private native void nativeSurfaceDestroyed();

  // Deliver touch events to engine
  private native void nativeDispatchPointerDataPacket(
      ByteBuffer buffer, int position);

  // Platform channel: send message from Java to Dart
  private native void nativeDispatchPlatformMessage(
      String channel, ByteBuffer message, int position, int responseId);

  // Platform channel: receive message from Dart (called from C++ via JNI):
  // @CalledByNative
  private void handlePlatformMessage(String channel, byte[] message, int replyId);
}

// The @CalledByNative methods are invoked by the C++ engine via JNI.
// The C++ engine has a reference to the FlutterJNI Java object
// and calls these methods on the platform task runner thread.

FlutterEngineCache: Pre-Warming

Cold-starting a Flutter engine takes 500ms–2s on mid-range devices (Dart VM init + snapshot loading + first frame). FlutterEngineCache solves this by pre-warming the engine before the user navigates to the Flutter screen.

// Engine pre-warming pattern (Android):
class MyApplication : Application() {
  override fun onCreate() {
    super.onCreate()
    // Pre-warm the engine during Application.onCreate()
    // This runs on a background thread to not block startup
    val engine = FlutterEngine(this)
    engine.dartExecutor.executeDartEntrypoint(
      DartExecutor.DartEntrypoint.createDefault()
    )
    // Cache the pre-warmed engine
    FlutterEngineCache.getInstance().put("main_engine", engine)
  }
}

// When navigating to the Flutter screen:
class MainActivity : FlutterActivity() {
  override fun getCachedEngineId() = "main_engine"
  // FlutterActivity uses the pre-warmed engine — no cold start!
}

// Pre-warming tradeoffs:
// PRO: Flutter screen appears instantly (sub-100ms vs 1-2s)
// CON: Application.onCreate() is heavier → app cold start is slower
// CON: Engine consumes memory (~20-50MB) even if Flutter screen is never shown
// MITIGATION: Pre-warm lazily (after first screen renders) if Flutter is optional

CAEAGLLayer vs CAMetalLayer

// iOS surface types:
//
// CAEAGLLayer (OpenGL ES):
// - Legacy; iOS 12+; being deprecated
// - FlutterView backed by CAEAGLLayer when Metal unavailable
// - Higher CPU overhead (OpenGL ES driver overhead on Apple Silicon)
//
// CAMetalLayer (Metal):
// - Default for Flutter iOS on A7+ devices (iPhone 5S and newer)
// - Flutter Impeller REQUIRES Metal — falls back to Skia on OpenGL ES
// - CAMetalLayer directly interacts with Metal's command queue
// - No intermediate copies — Flutter draws to MTLDrawable provided by CAMetalLayer

// At the engine level (flutter/shell/platform/darwin/ios/):
// FlutterView.mm selects the layer type:
+ (Class)layerClass {
  return [FlutterMetalLayer class];  // Default (Metal)
  // Falls back to: [CAEAGLLayer class] if Metal unavailable
}

// The CAMetalLayer's nextDrawable is acquired each frame:
// 1. Engine asks embedder for a Surface (frame acquisition)
// 2. Embedder calls [metalLayer nextDrawable]
//    → blocks if no drawable available (back-pressure from GPU)
// 3. Impeller renders to MTLTexture of the drawable
// 4. Embedder presents: [commandBuffer presentDrawable:drawable]
// 5. CAMetalLayer displays the texture on screen at next vsync

RunLoop Integration

// iOS event delivery to Flutter runs through the main RunLoop.
// Touch events: UIKit delivers touches to FlutterViewController
// which translates them to Flutter's pointer event model.

// FlutterViewController overrides:
// - touchesBegan:withEvent: → DispatchPointerDataPacket
// - touchesMoved:withEvent: → DispatchPointerDataPacket
// - touchesEnded:withEvent: → DispatchPointerDataPacket

// Vsync: Flutter uses CADisplayLink (iOS display sync signal):
// CADisplayLink fires at screen refresh rate (60/90/120 Hz)
// → calls into engine's vsync callback
// → engine schedules a frame on the UI runner
// → Dart builds frame, sends layer tree to GPU runner
// → GPU runner renders, presents to CAMetalLayer

// The platform runner on iOS IS the main thread.
// All plugin method calls, view controller calls, UIKit operations
// happen on the main thread (platform runner).
// Flutter engine carefully synchronizes UI runner ↔ platform runner
// to avoid deadlocks when platform channels need synchronous responses.

Platform Views: Hybrid Composition vs Virtual Display

When a Flutter app embeds a native view (e.g., AndroidView, UiKitView, Google Maps, WebView), the embedder must composite native platform views with Flutter's own rendering output. This is one of the most complex parts of the embedder layer.

// Two composition modes on Android:

// 1. VIRTUAL DISPLAY (legacy, default pre-Flutter 3.x):
//    - Renders the native Android View into an off-screen virtual display
//    - The virtual display's output is captured as a texture
//    - The texture is composited into Flutter's render tree like any other texture
//    - PRO: Single Flutter surface, no Z-order issues
//    - CON: Accessibility broken (view not on real screen), some APIs fail
//           (e.g., Android video decoders, some WebViews)
//
// 2. HYBRID COMPOSITION (recommended for complex views):
//    - The native Android View is placed in the actual Android View hierarchy
//    - Flutter renders "around" it by splitting the Flutter surface
//    - Two Flutter surfaces: one below the native view, one above
//    - PRO: Full Android View functionality, accessibility works
//    - CON: Higher overhead (two surfaces), complex Z-ordering
//           Some GPU overdraw scenarios

// Flutter 3.x default: Hybrid Composition for AndroidView
// To force Virtual Display mode (legacy):
AndroidView(
  viewType: 'my_native_view',
  creationParams: {...},
  creationParamsCodec: const StandardMessageCodec(),
  // This uses hybrid composition by default since Flutter 3.0
)

// iOS UiKitView similarly uses hybrid composition:
// Flutter splits its CALayer tree, inserting the UIView between Flutter layers
// This is why maps and WebViews in Flutter sometimes have rendering artifacts
// at the layer boundary
Warning

Platform views are the most performance-sensitive part of the embedder. Each AndroidView or UiKitView instance adds significant GPU overhead — hybrid composition requires additional render passes to composite the native and Flutter layers. At the Staff level, you should advise teams to minimize platform view usage, consider alternatives (FFI plugins that render into a Flutter texture, or full-screen native screens with Flutter overlaid), and always profile with the DevTools Timeline when platform views are involved.

7. Custom Embedder: The flutter_embedder.h API

The public embedder API (flutter_embedder.h) allows implementing Flutter on any platform — embedded Linux, custom hardware, game engines, etc. It is a pure C API that the engine exposes.

// flutter_embedder.h key structures and functions:
// (Used to build the Flutter Desktop embedders and custom embedders)

// 1. Define task runners for your platform:
FlutterTaskRunnerDescription platform_task_runner = {};
platform_task_runner.struct_size = sizeof(FlutterTaskRunnerDescription);
platform_task_runner.runs_task_on_current_thread_callback =
    [](void* user_data) -> bool {
      return IsOnPlatformThread();
    };
platform_task_runner.post_task_callback =
    [](FlutterTask task, uint64_t target_time_nanos, void* user_data) {
      // Schedule 'task' to run at target_time_nanos on the platform thread
      ScheduleTask(task, target_time_nanos);
    };

// 2. Define vsync provider:
FlutterRendererConfig renderer_config = {};
renderer_config.type = kOpenGL;
renderer_config.open_gl.struct_size = sizeof(FlutterOpenGLRendererConfig);
renderer_config.open_gl.make_current = [](void* user_data) -> bool {
  return MakeOpenGLContextCurrent();
};
renderer_config.open_gl.present = [](void* user_data) -> bool {
  return SwapBuffers();
};

// 3. Configure and run the engine:
FlutterProjectArgs args = {};
args.struct_size = sizeof(FlutterProjectArgs);
args.assets_path = "/path/to/flutter_assets";
args.icu_data_path = "/path/to/icudtl.dat";
args.custom_task_runners = &task_runners;

FlutterEngine engine = nullptr;
FlutterEngineRun(FLUTTER_ENGINE_VERSION, &renderer_config, &args,
                 user_data, &engine);

// 4. Send vsync events:
FlutterEngineOnVsync(engine, baton, frame_start_ns, frame_target_ns);

// 5. Send pointer events:
FlutterPointerEvent events[1] = {};
events[0].struct_size = sizeof(FlutterPointerEvent);
events[0].phase = kDown;
events[0].x = touch_x; events[0].y = touch_y;
FlutterEngineSendPointerEvent(engine, events, 1);
Tip

The Flutter desktop embedders for Windows, macOS, and Linux are built using flutter_embedder.h. Studying the flutter/shell/platform/windows/ or flutter/shell/platform/linux/ source in the engine repository is the best way to understand the full embedder contract — they are the most readable reference implementations.
