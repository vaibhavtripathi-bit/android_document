Part 1 — The Paint Phase
From RenderObject.paint() to DisplayList
After layout assigns sizes and positions, the paint phase converts the render tree into a recorded sequence of drawing commands. These commands are not executed immediately — they are recorded into a DisplayList for deferred rasterization on the raster thread.

1. PaintingContext and Canvas

The paint phase begins when PipelineOwner.flushPaint() walks the render tree and calls paint(context, offset) on each dirty RenderObject. The PaintingContext passed to each render object is the central object of the paint phase — it provides the Canvas API and manages layer creation.

// RenderObject.paint() entry point
abstract class RenderObject {
  void paint(PaintingContext context, Offset offset);
}

// Example: a custom RenderBox that draws a filled circle
class CircleRenderBox extends RenderBox {
  final Color color;
  CircleRenderBox({required this.color});

  @override
  void paint(PaintingContext context, Offset offset) {
    final canvas = context.canvas; // the current Canvas
    final center = offset + Offset(size.width / 2, size.height / 2);
    final radius = size.shortestSide / 2;

    canvas.drawCircle(
      center,
      radius,
      Paint()..color = color,
    );
  }
}

Canvas Commands are Recorded, Not Executed

The Canvas in Flutter does not draw to a pixel buffer immediately. It records commands into a DisplayList (Skia's SkPicture in the Skia backend, or Impeller's own DisplayList format). This recording is a fast, serializable sequence of draw calls. The actual rasterization — converting draw commands to pixels — happens later on the raster thread.

// Canvas drawing commands (all recorded, not immediately rendered):
canvas.drawRect(rect, paint);
canvas.drawRRect(rrect, paint);
canvas.drawCircle(center, radius, paint);
canvas.drawPath(path, paint);
canvas.drawImage(image, offset, paint);
canvas.drawParagraph(paragraph, offset);
canvas.drawLine(p1, p2, paint);
canvas.save();           // save transform/clip state
canvas.restore();        // restore state
canvas.translate(dx, dy);
canvas.scale(sx, sy);
canvas.rotate(radians);
canvas.clipRect(rect);
canvas.clipRRect(rrect);
canvas.clipPath(path);
canvas.saveLayer(bounds, paint); // expensive — new render pass

2. Layer Types

Not all rendering in Flutter ends up in a single flat DisplayList. The output of the paint phase is a layer tree — a hierarchical structure of different layer types that the compositor assembles into the final frame. Each layer type has distinct GPU semantics.

Layer TypePurposeGPU cost
PictureLayerLeaf layer containing a recorded DisplayList (SkPicture / Impeller DL)Rasterization cost proportional to draw command complexity
TransformLayerApplies a 4x4 matrix transform to its childrenVery low — GPU matrix multiply
ClipRectLayerClips children to a rectangleLow — scissor test
ClipRRectLayerClips children to a rounded rectangleMedium — stencil buffer
ClipPathLayerClips children to an arbitrary pathHigh — complex stencil
OpacityLayerApplies alpha to entire subtreeHigh — requires offscreen buffer
PhysicalModelLayerElevation shadow + clip (Material cards)Medium
OffsetLayerTranslates children by a 2D offsetVery low
ContainerLayerBase class for layers with childrenNo direct GPU cost
AnnotatedRegionLayerCarries semantic / hit-test metadataNone (CPU only)

PictureLayer — The Leaf

PictureLayer is the primary leaf node in the layer tree. It holds the recorded Picture (the output of a PictureRecorder / Canvas session). During rasterization, the raster thread calls into Skia/Impeller to convert the Picture into GPU texture pixels.

// How PaintingContext creates a PictureLayer internally
class PaintingContext {
  PictureLayer _currentLayer;
  ui.PictureRecorder _recorder;
  Canvas _canvas;

  // When a new PictureLayer is needed:
  void _startRecording() {
    _currentLayer = PictureLayer(estimatedBounds);
    _recorder = ui.PictureRecorder();
    _canvas = Canvas(_recorder);
    _containerLayer.append(_currentLayer);
  }

  // When the layer is complete:
  void _stopRecordingIfNeeded() {
    _currentLayer.picture = _recorder.endRecording();
  }
}

3. RepaintBoundary: Isolated OffscreenLayer

When a RenderRepaintBoundary is encountered during the paint phase, PaintingContext stops the current PictureLayer recording and creates a new OffscreenLayer (a specialized ContainerLayer) for the boundary's subtree. The GPU caches the rasterized result of this layer as a texture.

// PaintingContext.paintChild detects RepaintBoundary
void paintChild(RenderObject child, Offset offset) {
  if (child.isRepaintBoundary) {
    // Stop current recording
    _stopRecordingIfNeeded();
    // Paint into a new isolated context for the boundary
    _compositeChild(child, offset);
  } else {
    // Continue painting into the current PictureLayer
    child.paint(this, offset);
  }
}

void _compositeChild(RenderObject child, Offset offset) {
  // Create an OffscreenLayer for this boundary
  final OffsetLayer childLayer = _acquireContext(child)._rootLayer;
  // Re-paint child only if it is dirty
  if (child.needsPaint) {
    _repaintCompositedChild(child);
  }
  // Append the child's layer to our current container
  childLayer.offset = offset;
  _containerLayer.append(childLayer);
}

The Repaint Isolation Guarantee

Once a subtree has been rasterized into an OffscreenLayer's GPU texture:

  • If the subtree repaints (its PictureLayer is dirty), only that layer is re-rasterized. The rest of the screen composites from cached GPU textures.
  • If the rest of the screen repaints but the subtree is clean, the subtree's cached GPU texture is composited in directly. No re-rasterization of the subtree.
Staff-Level Insight

The raster cache that backs RepaintBoundary is bounded by memory. Flutter's raster thread maintains a cache budget (default ~75 MB on mobile). When the budget is exceeded, older cached layers are evicted. Aggressively wrapping everything in RepaintBoundary can exhaust this budget, causing more cache misses and more re-rasterization than no boundaries at all. Use RepaintBoundary surgically — around widgets that animate or change frequently inside an otherwise-static scene, not as a blanket optimization.

4. needsCompositing: Deciding Whether to Use a Layer

Not every RenderObject needs its own compositing layer. The needsCompositing flag on RenderObject indicates whether the subtree requires a compositing layer. This flag is propagated bottom-up during the compositing bits update phase (between layout and paint).

// A RenderObject needs compositing if:
// 1. It is a RepaintBoundary
// 2. It uses certain effects that require layers (opacity, clip, etc.)
// 3. Any of its children needsCompositing

bool get needsCompositing {
  assert(!_needsCompositingBitsUpdate);
  return _needsCompositing;
}

// - RenderRepaintBoundary
// - RenderOpacity (when opacity < 1.0)

// - Platform views

5. Opacity: The Expensive Compositing Case

Using Opacity(opacity: 0.5, child: ...)` forces the subtree to be painted into an offscreen buffer, then composited with the given alpha. This requires an extra GPU render pass — the subtree is rendered to texture, then that texture is blended onto the screen at the given opacity. For animated opacity, this re-render happens every frame.

// EXPENSIVE: Opacity widget forces offscreen render pass every frame
Opacity(
  opacity: _fadeValue, // changes every animation frame
  child: ComplexWidget(), // entire subtree re-rendered to texture every frame
)

// EFFICIENT: FadeTransition uses a compositing layer with a color filter
// The layer transform (opacity) is applied by the GPU compositor
// without re-rasterizing the child's pixels
FadeTransition(
  opacity: _animation, // AnimationController value
  child: ComplexWidget(), // rasterized ONCE, opacity applied by GPU
)
Staff-Level Insight

FadeTransition is backed by RenderAnimatedOpacity, which uses an OpacityLayer. The opacity value is set directly on the layer (layer.alpha = (opacity * 255).round()) without invalidating the child's paint. The raster thread applies the alpha during compositing — no re-rasterization needed. This is why FadeTransition is O(1) GPU work per frame while Opacity with an animated value triggers a full re-rasterization of the subtree every frame. The same principle applies to any transform animation: use AnimatedTransform/Transform with a layer-level transform rather than rebuilding the widget.

6. saveLayer: Explicit Separate Render Pass

canvas.saveLayer(bounds, paint) is the most expensive Canvas operation. It creates a new offscreen surface, renders all subsequent commands into it, then composites the result back using the given paint (which may have a color filter, image filter, or blend mode applied).

// Operations that implicitly call saveLayer:
// 1. BackdropFilter — must read existing pixels to apply blur
BackdropFilter(
  filter: ImageFilter.blur(sigmaX: 10, sigmaY: 10),
  child: Container(color: Colors.white.withOpacity(0.5)),
)

// 2. ColorFiltered — applies a color matrix to entire subtree
ColorFiltered(
  colorFilter: const ColorFilter.matrix([...grayscale matrix...]),
  child: Image.asset('photo.jpg'),
)

// 3. ShaderMask — applies a shader mask
ShaderMask(
  shaderCallback: (bounds) => gradient.createShader(bounds),
  child: const Text('Gradient Text'),
)

// 4. Explicit saveLayer in custom paint:
canvas.saveLayer(
  Rect.fromLTWH(0, 0, size.width, size.height),
  Paint()..colorFilter = ColorFilter.mode(Colors.red, BlendMode.modulate),
);
// ... draw commands ...
canvas.restore(); // composites the offscreen surface back
Performance Warning

BackdropFilter is one of the most expensive widgets in Flutter. It requires: (1) rendering everything behind it to a texture, (2) applying a GPU image filter (blur), (3) rendering the BackdropFilter's child on top. Every frame the background changes, all three steps repeat. On complex screens with BackdropFilter, this can reduce frame rate significantly on low-end devices. Minimize BackdropFilter usage — prefer pre-blurred static assets or restrict to non-animated backgrounds.

7. Offstage: Invisible but In-Tree

The Offstage widget keeps its child in the widget and element trees (state preserved, layout computed) but does not paint it and removes it from hit-testing. This is different from Visibility(visible: false) which still calls paint but with a no-op.

// Offstage: child is laid out (size computed) but NOT painted
Offstage(
  offstage: true,  // when true: invisible, no paint, no hit test
  child: ExpensiveWidget(), // state preserved, size computed
)

// Use case: pre-warm a widget (measure its size) before showing it
// or keep a background tab's state without its repaint cost

Unlike Opacity(opacity: 0.0), Offstage does NOT create a compositing layer — it simply skips the paint call entirely. This makes it cheaper than 0-opacity for hiding content that should remain in the tree.

8. The Compositing Pass

After all render objects have painted into the layer tree, the UI thread sends the layer tree to the engine. The engine's compositor (running on the raster thread) walks the layer tree depth-first, executing each layer's GPU operations in order:

  1. PictureLayer: rasterize DisplayList → GPU texture (or use cached texture if clean)
  2. TransformLayer: push matrix onto GPU transform stack
  3. ClipRectLayer: enable scissor test
  4. OpacityLayer: set GPU blend alpha
  5. OffsetLayer: translate GPU transform
  6. At each container layer: recursively process children, then pop the layer's GPU state
// Layer tree example for a typical screen:
// ContainerLayer (root)
//   OffsetLayer (scroll offset)
//     PictureLayer (background + static content)
//     OffscreenLayer (RepaintBoundary wrapping animation)
//       TransformLayer (rotate transform)
//         PictureLayer (animated content)
//     PictureLayer (foreground / overlay content)
Interview Tip

When asked about the difference between the UI thread and raster thread: the UI thread runs Dart code and produces the layer tree. The raster thread consumes the layer tree and produces GPU commands. They run concurrently — the UI thread can be building the next frame's layer tree while the raster thread is still rasterizing the current frame. This pipeline parallelism is what allows Flutter to maintain smooth frame rates even with non-trivial per-frame Dart work.
