Part 1 — The Clock Signal
VSync: Flutter's Heartbeat
Flutter does not draw frames continuously. It draws only when the display signals it is ready — the vertical sync (vsync) signal. Understanding how Flutter requests and responds to vsync is foundational to understanding animation timing, frame budgets, and energy efficiency.

1. What VSync Is

A display refreshes its pixels at a fixed rate — historically 60 times per second (60 Hz), with modern devices supporting 90 Hz, 120 Hz (ProMotion), and variable-rate displays (LTPO). At the end of each refresh cycle, the display hardware emits a vertical synchronization signal (vsync) indicating it is ready to accept a new frame buffer to display.

If an app produces frames faster than the display can show them, the excess frames are wasted. If it produces frames slower, a refresh cycle is missed — the same frame is shown twice, causing visible stutter (jank). VSync-gated rendering aligns frame production with display consumption.

2. Flutter's Demand-Driven Frame Scheduling

Flutter does not request vsync callbacks continuously. It requests a frame only when it has work to do. This is critical for battery life — an idle Flutter app receives no vsync callbacks and uses minimal CPU/GPU resources.

// The full vsync request chain:

// 1. Something needs to change (e.g., setState, animation tick)
element.markNeedsBuild();
// OR
renderObject.markNeedsLayout();
// OR
animationController._tick(); // animation value changed

// 2. All paths eventually call:
SchedulerBinding.instance.scheduleFrame();

// 3. scheduleFrame() calls:
void scheduleFrame() {
  if (_hasScheduledFrame) return; // already requested, skip
  ensureFrameCallbacksRegistered();
  window.scheduleFrame(); // → platform channel → OS requests vsync
  _hasScheduledFrame = true;
}

// 4. OS delivers vsync callback → SchedulerBinding.handleBeginFrame()
// 5. Frame pipeline runs (transient → persistent → postFrame callbacks)
// 6. _hasScheduledFrame = false — next change triggers another request
Staff-Level Insight

The single _hasScheduledFrame flag is why calling setState() 100 times before the next vsync results in only one frame being drawn, not 100. All 100 setState() calls mark elements dirty and call scheduleFrame(), but only the first call actually schedules a vsync request. When the frame fires, all 100 dirty elements are rebuilt in a single buildScope() pass. This batch-and-coalesce behavior is fundamental to Flutter's efficiency.

3. SchedulerBinding Frame Phases

When a vsync arrives, SchedulerBinding processes callbacks in a strict sequence of phases:

// SchedulerBinding.handleBeginFrame() and handleDrawFrame()

// Phase 1: TRANSIENT CALLBACKS (handleBeginFrame)
// - Animation ticks happen here
// - Ticker callbacks fire with the current frame timestamp
// - AnimationController computes its current value
// - These callbacks are one-shot: registered for the current frame only
_transientCallbacks.forEach((id, callback) => callback(timeStamp));
_transientCallbacks.clear(); // cleared after each frame

// Phase 2: PERSISTENT CALLBACKS (handleDrawFrame)
// - The core render pipeline runs here
// - WidgetsBinding._handleBuildScheduled → BuildOwner.buildScope (widget rebuild)
// - PipelineOwner.flushLayout (layout pass)
// - PipelineOwner.flushCompositingBits (needsCompositing propagation)
// - PipelineOwner.flushPaint (paint pass)
// - RenderView.compositeFrame (layer tree → Scene → GPU)
// These run every frame unconditionally (but skip if nothing is dirty)

// Phase 3: POST-FRAME CALLBACKS (after frame is submitted)
// - One-shot callbacks registered via addPostFrameCallback()
// - Run after the frame is sent to GPU
// - Safe to call setState(), schedule animations, measure sizes
_postFrameCallbacks.forEach((callback) => callback(timeStamp));
_postFrameCallbacks.clear();

4. Ticker: The Animation Clock

A Ticker is a simple object that receives vsync callbacks and forwards them (with elapsed time) to a listener. It is the lowest-level animation primitive in Flutter.

// Ticker internals
class Ticker {
  final TickerCallback _onTick; // callback that receives elapsed Duration
  bool _muted = false;
  TickerFuture? _future;

  TickerFuture start() {
    // Register a transient frame callback
    _future = TickerFuture._();
    _scheduleTick();
    return _future!;
  }

  void _scheduleTick() {
    _animationId = SchedulerBinding.instance.scheduleFrameCallback(_tick);
  }

  void _tick(Duration timeStamp) {
    // Compute elapsed since start
    final elapsed = timeStamp - _startTime!;
    _onTick(elapsed); // fire the listener (e.g., AnimationController)

    // Re-register for next frame if still active
    if (shouldScheduleTick) _scheduleTick();
  }

  void stop({bool canceled = false}) {
    // Unregister from SchedulerBinding
    SchedulerBinding.instance.cancelFrameCallbackWithId(_animationId!);
  }
}

TickerProvider

A TickerProvider creates Ticker instances tied to a widget's lifecycle. The two common mixins are:

// For a single AnimationController
class _MyState extends State<MyWidget> with SingleTickerProviderStateMixin {
  late final AnimationController _ctrl;

  @override
  void initState() {
    super.initState();
    _ctrl = AnimationController(vsync: this, duration: const Duration(seconds: 1));
  }

  @override
  void dispose() {
    _ctrl.dispose(); // REQUIRED — stops Ticker, unregisters from SchedulerBinding
    super.dispose();
  }
}

// For multiple AnimationControllers
class _MultiState extends State<MultiWidget> with TickerProviderStateMixin {
  late final AnimationController _ctrl1;
  late final AnimationController _ctrl2;

  @override
  void initState() {
    super.initState();
    _ctrl1 = AnimationController(vsync: this, duration: const Duration(milliseconds: 300));
    _ctrl2 = AnimationController(vsync: this, duration: const Duration(milliseconds: 500));
  }

  @override
  void dispose() {
    _ctrl1.dispose();
    _ctrl2.dispose();
    super.dispose();
  }
}

5. AnimationController: Time → Value

AnimationController sits between the Ticker (which provides elapsed time) and the animation system (which consumes normalized values). It computes the current animation value as a fraction of the total duration, applies optional curves, and notifies listeners.

// AnimationController.forward() execution path:
// 1. _direction = _AnimationDirection.forward
// 2. _ticker.start() — registers transient callback with SchedulerBinding
// 3. On each vsync tick:
//    elapsed = timeStamp - _lastElapsedDuration
//    _value = (elapsed / duration).clamp(0.0, 1.0)
//    notifyListeners() — triggers AnimatedBuilder rebuilds, etc.
//    if (_value >= 1.0) _ticker.stop(); future.complete()
// 4. Continues until value reaches 1.0 or stop() is called

// The value is time-based, not frame-based:
// If a frame is dropped (takes 32ms instead of 16ms),
// the animation jumps forward proportionally.
// This ensures animations complete in the correct WALL CLOCK time
// even if some frames are dropped.
Staff-Level Insight

AnimationController is time-based, not frame-count-based. This means dropped frames cause animation values to jump (the animation skips ahead to catch up with wall clock time) rather than slowing down. This is intentional — a 300ms animation completes in 300ms of wall clock time regardless of how many frames were dropped. If you need frame-count-based animation (rare), you would use a manual Ticker and count ticks. Time-based animation is almost always correct for UI.

6. Ticker Leak: A Silent Memory and CPU Bug

Forgetting to call dispose() on an AnimationController creates a Ticker leak. The Ticker continues receiving vsync callbacks indefinitely, even after the widget is removed from the tree.

// LEAKED Ticker scenario:
class _LeakyState extends State<LeakyWidget> with SingleTickerProviderStateMixin {
  late final AnimationController _ctrl;

  @override
  void initState() {
    super.initState();
    _ctrl = AnimationController(vsync: this)..repeat(); // animating forever
  }

  // MISSING: dispose() with _ctrl.dispose()
  // Result:
  // - Ticker registered in SchedulerBinding._transientCallbacks
  // - SchedulerBinding holds reference to Ticker
  // - Ticker holds reference to AnimationController
  // - AnimationController holds reference to State (via vsync: this)
  // - State holds reference to Widget subtree
  // None of this can be garbage collected!
  // Flutter schedules a new frame every vsync — CPU/GPU never idle
  // Debug mode: throws FlutterError about undisposed AnimationController
}

// CORRECT: always dispose in dispose()
@override
void dispose() {
  _ctrl.dispose(); // stops Ticker, removes from SchedulerBinding
  super.dispose();
}

7. Adaptive Refresh Rate: 120Hz and LTPO

Flutter supports high-refresh-rate displays. On supported devices (iPhone 13 Pro+ with ProMotion, Pixel 6 Pro+, Samsung Galaxy S21+), Flutter can render at up to 120 Hz when animations are running, and drop to lower rates when idle.

// Enable 120Hz in Flutter (iOS):
// Info.plist
<key>CADisableMinimumFrameDurationOnPhone</key>
<true/>

// Enable on Android — Flutter handles this automatically
// via the Android Display.getSupportedModes() API

// Frame budget at 120Hz: 8.33ms
// This cuts the allowed UI thread + raster thread budget roughly in half
// Animations that ran fine at 60Hz may jank at 120Hz

LTPO (Low Temperature Polycrystalline Oxide)

LTPO displays can vary their refresh rate from 1 Hz to 120 Hz dynamically. The OS lowers the rate when content is static (saves battery) and raises it when animation is detected. Flutter integrates with this via the platform's frame callback timing — when Flutter requests vsync, the OS can choose to deliver it at the optimal rate based on current workload and power state.

Staff-Level Insight

On ProMotion / LTPO devices, if your app has a Ticker running (even with a static-looking animation value), the OS may lock the display to 120 Hz to accommodate any potential frame. This can drain battery unnecessarily. Best practice: stop Tickers when animations are not visually changing (animationController.stop() when value is stable), and use willChangeHint: false on picture layers that are static. This allows the OS to drop to 1 Hz for the relevant content.

8. scheduleFrame vs scheduleMicrotask

APIWhen it runsUse case
SchedulerBinding.instance.scheduleFrame()Next vsync callback from the OS displayRequesting a new render frame (widget rebuild, animation)
scheduleMicrotask(fn)Before the next event loop iteration (before any I/O, timers, or vsync)Deferred work that must run before any async events but within current event
Future.delayed(Duration.zero, fn)Next event loop iteration (after microtasks)Deferring until after current build completes
addPostFrameCallback(fn)After the current frame is submitted to GPUMeasuring widget sizes, triggering navigation after build
 
// scheduleFrame: requests OS vsync → frame pipeline runs
// Nothing happens until the next display refresh signal
SchedulerBinding.instance.scheduleFrame();

// scheduleMicrotask: runs immediately after current synchronous code
// Does NOT wait for vsync — can run multiple times before next frame
scheduleMicrotask(() {
  // Runs before any Future callbacks, before any vsync
  print('microtask');
});

// addPostFrameCallback: after frame submitted, safe to read layout
WidgetsBinding.instance.addPostFrameCallback((_) {
  final size = context.size; // layout complete — safe to read
  print('Widget size: $size');
});
