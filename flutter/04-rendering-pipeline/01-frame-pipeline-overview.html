Why This Matters at L6
Flutter's frame pipeline is the single most interrogated topic in Staff-level interviews. Being able to name every phase, explain what thread each runs on, quantify the budget, and connect it to real jank scenarios is table stakes. This document covers the full pipeline from vsync receipt to pixel display.

The Six Phases of a Flutter Frame

Every rendered frame travels through six sequential phases. The first four phases execute on the UI thread (Dart isolate), while rasterization executes on the dedicated raster thread (formerly GPU thread). Understanding this separation is critical for diagnosing performance issues.

Frame Budget
At 60 Hz: 16.67 ms total per frame. At 120 Hz (ProMotion, LTPO): 8.33 ms per frame. The UI and raster threads run concurrently via double-buffering, but each still has an independent 16.67 ms budget. Either thread exceeding budget causes a dropped frame.

Phase 1: Build — Widget to RenderObject Reconciliation

The build phase begins when BuildOwner.buildScope() is invoked. Flutter maintains three parallel trees: the widget tree (immutable, user-authored), the element tree (persistent, mutable identity), and the render object tree (layout/paint logic).

Dirty Element Marking and Rebuilding

When setState() is called on a State object, the corresponding StatefulElement is marked dirty via element.markNeedsBuild(). This adds the element to BuildOwner._dirtyElements — a sorted list ordered by depth (shallow elements first to prevent redundant deep rebuilds).

// Framework internals — simplified
void markNeedsBuild() {
  assert(_debugLifecycleState != _ElementLifecycle.defunct);
  if (dirty) return;
  _dirty = true;
  owner!.scheduleBuildFor(this);  // adds to _dirtyElements list
}

void buildScope(Element context, [VoidCallback? callback]) {
  // Processes _dirtyElements in depth order
  while (index < dirtyCount) {
    final Element element = _dirtyElements[index];
    element.rebuild();  // calls build() on StatefulElement → calls State.build()
    index++;
  }
}

Element Reconciliation — canUpdate()

The core of the build phase is determining whether to update an existing element or replace it entirely. This decision is made by Widget.canUpdate(Widget oldWidget, Widget newWidget):

static bool canUpdate(Widget oldWidget, Widget newWidget) {
  return oldWidget.runtimeType == newWidget.runtimeType
      && oldWidget.key == newWidget.key;
}

If canUpdate returns true, the existing element is updated with the new widget configuration (cheap: existing render object is retained and only its properties change). If false, the element subtree is deactivated and replaced (expensive: new render objects created, old ones disposed).

Staff-Level Insight
The depth-ordered rebuild in buildScope() is a critical optimization. If a parent and child are both dirty, building the parent first may cause the child to be removed from the tree, making its rebuild unnecessary. Flutter skips elements that are no longer in the tree during the build scope pass.

Widget Tree → Element Tree → Render Object Tree

  • StatelessWidget / StatefulWidget: proxied through StatelessElement / StatefulElement. No render object — pure composition nodes.
  • RenderObjectWidget (e.g., Padding, SizedBox): has a paired RenderObjectElement that creates and owns a RenderObject.
  • RenderObjectElement.update() calls widget.updateRenderObject(context, renderObject) to apply new configuration without creating a new render object.

Phase 2: Layout — Constraints Down, Sizes Up

After the build phase, dirty render objects (those marked via markNeedsLayout()) are laid out. Layout is a single-pass protocol with a strict contract:

  1. Parent calls child.layout(constraints), passing BoxConstraints downward.
  2. Child performs its layout computation and calls size = ... to set its dimensions.
  3. Parent reads child.size to position the child.

Layout Relayout Boundaries

Flutter optimizes layout via relayout boundaries. A render object is a relayout boundary if: (a) it is the root, (b) parentUsesSize is false, (c) the constraints are tight, or (d) it is explicitly marked as such. When a node inside a relayout boundary is marked dirty, only the subtree within the boundary is re-laid-out — the parent is not notified. This is why ListView children don't trigger full-tree relayout on scroll.

void markNeedsLayout() {
  if (_relayoutBoundary != this) {
    markParentNeedsLayout();  // bubble up to nearest boundary
  } else {
    _needsLayout = true;
    owner!.schedulePipelineUpdate(); // just mark this subtree
  }
}

performLayout() and the Render Object Protocol

Each RenderObject subclass overrides performLayout(). The base class layout() method handles caching and dirty checking before delegating to performLayout():

// RenderBox.performLayout() — simplified RenderPadding implementation
@override
void performLayout() {
  final BoxConstraints constraints = this.constraints;
  if (child != null) {
    child!.layout(constraints.deflate(padding), parentUsesSize: true);
    size = constraints.constrain(
      Size(child!.size.width + padding.horizontal,
           child!.size.height + padding.vertical));
  } else {
    size = constraints.constrain(
      Size(padding.horizontal, padding.vertical));
  }
}

Phase 3: Paint — Recording the Display List

After layout, dirty render objects (marked via markNeedsPaint()) are painted. Crucially, painting does not immediately issue GPU commands. Instead, canvas calls are recorded into a ui.PictureRecorder, producing a ui.Picture (wrapping Skia's SkPicture) that represents a serialized list of drawing operations.

PaintingContext and Layers

PipelineOwner.flushPaint() iterates _nodesNeedingPaint in reverse depth order (deepest first). For each node:

  1. A PaintingContext is created (or reused) with an associated Canvas backed by a PictureRecorder.
  2. renderObject.paint(context, offset) is called. The render object issues canvas commands: context.canvas.drawRect(...), drawImage(...), etc.
  3. The recorded picture is stored in the current PictureLayer.

Paint Repaint Boundaries

Similar to layout, paint has repaint boundaries. When isRepaintBoundary returns true (as in RepaintBoundary widget), a new OffsetLayer is introduced. Child paint calls are isolated — the child's picture can be re-used without repainting the parent.

Phase 4: Compositing — Assembling the Layer Tree

The compositing phase assembles the layer tree into a ui.Scene for the engine. RenderView.compositeFrame() calls layer.buildScene(SceneBuilder builder), which recursively traverses the layer tree, calling SceneBuilder methods:

// Layer tree → SceneBuilder API
builder.pushTransform(matrix4);
builder.pushClipRect(rect);
builder.addPicture(offset, picture);  // from PictureLayer
builder.pop();  // matches each push
final ui.Scene scene = builder.build();
window.render(scene);  // hand off to engine
Key Distinction
addPicture() does not rasterize — it hands the recorded ui.Picture to the engine. Rasterization happens on the raster thread, decoupled from the UI thread. This is the source of Flutter's double-buffered concurrency.

Phase 5: Rasterization — GPU Commands

On the raster thread, the engine's Rasterizer processes the Scene. It traverses the layer tree and, for each PictureLayer, replays the SkPicture via Skia (or Impeller), generating GPU draw calls (Vulkan command buffers on Android, Metal command encoders on iOS, OpenGL on older devices).

Raster Cache

Frequently rasterized pictures are cached by the raster cache. The cache key is a combination of the SkPicture content hash and the transformation matrix. If the picture and transform are unchanged, the cached GPU texture is composited directly, bypassing re-rasterization.

Shader Compilation

On first use, Skia must compile GLSL shaders to device-specific GPU programs. This compilation happens synchronously on the raster thread and can take 1–8 ms per shader, causing the notorious first-run jank. Impeller solves this by pre-compiling all shaders at build time into Metal/Vulkan PSOs.

Phase 6: Present — Buffer Swap

After rasterization, the raster thread signals the platform to swap the back buffer (where the frame was rendered) with the front buffer (what's displayed). On Android this goes through ANativeWindow_setBuffersGeometry and the SurfaceFlinger compositor. On iOS it's a Metal presentDrawable call synchronized with CADisplayLink.

SchedulerBinding: Orchestrating the Frame

SchedulerBinding is the mixin that connects Flutter's frame pipeline to the platform's vsync signal. It manages the lifecycle of each frame through three callback lists:

Frame Callback Phases

// SchedulerBinding — frame execution sequence
void _handleBeginFrame(Duration rawTimeStamp) {
  // 1. Transient callbacks (animations):
  //    AnimationController ticks happen here
  _invokeTransientCallbacks(timeStamp);
}

void _handleDrawFrame() {
  // 2. Persistent callbacks (the actual rendering pipeline):
  //    WidgetsBinding._persistentCallbacks → drawFrame()
  _persistentCallbacks.forEach((cb) => cb());

  // 3. Post-frame callbacks (one-shot, after frame is submitted):
  //    addPostFrameCallback(() { ... })
  _invokePostFrameCallbacks();
}

scheduleFrame() — The Vsync Request Loop

Flutter only requests vsync signals when there is work to do. This is an important optimization — an idle app pays zero platform cost.

void scheduleFrame() {
  if (_hasScheduledFrame) return;
  ensureFrameCallbacksRegistered();
  window.scheduleFrame();    // platform call → requests next vsync
  _hasScheduledFrame = true;
}

Triggers that call scheduleFrame():

  • State.setState() → marks element dirty → buildOwner.scheduleBuildFor() → scheduleFrame()
  • AnimationController.forward() → creates Ticker → scheduleFrame()
  • RenderObject.markNeedsLayout() / markNeedsPaint() → scheduleFrame()

WidgetsBinding.drawFrame() — The Full Call Stack

This is the persistent callback that drives the entire rendering pipeline each frame:

// WidgetsBinding.drawFrame() — the heart of every frame
@override
void drawFrame() {
  // 1. BUILD PHASE
  buildOwner!.buildScope(renderViewElement!);

  // 2–4. LAYOUT + PAINT + COMPOSITING (inside super.drawFrame())
  super.drawFrame();  // → RendererBinding.drawFrame()

  // 5. Post-build cleanup
  buildOwner!.finalizeTree();
}

// RendererBinding.drawFrame()
@override
void drawFrame() {
  pipelineOwner.flushLayout();       // Phase 2: Layout
  pipelineOwner.flushCompositingBits();  // Update needsCompositing flags
  pipelineOwner.flushPaint();        // Phase 3: Paint
  if (sendFramesToEngine) {
    renderView.compositeFrame();     // Phase 4: Composite + send Scene
    pipelineOwner.flushSemantics();  // Accessibility tree
    _firstFrameSent = true;
  }
}
Staff-Level Interview Answer
When asked "what happens when setState() is called?", the complete answer traces: setState() → markNeedsBuild() → element added to _dirtyElements → scheduleFrame() → platform vsync requested → _handleBeginFrame (animation ticks) → _handleDrawFrame → buildScope() (rebuild dirty elements) → flushLayout() → flushPaint() → compositeFrame() → window.render(scene) → raster thread rasterizes → pixels displayed.

UI Thread vs Raster Thread: Double Buffering

Flutter's two-thread rendering model provides a key performance advantage:

Frame N:   [UI Thread: Build/Layout/Paint/Composite] → Scene N sent
                                                           |
           [Raster Thread: --------------------------------+---> Rasterize N → Present N]

Frame N+1: [UI Thread: --------Build/Layout/Paint/Composite] → Scene N+1 sent
                                                                      |
           [Raster Thread: -----------------------------------------+---> Rasterize N+1 → Present N+1]

While the raster thread is rasterizing frame N, the UI thread is already building frame N+1. This pipeline parallelism is why Flutter can maintain smooth animation even with modest CPU performance.

Critical Misconception
Many engineers believe the UI thread is the only thread that matters. In practice, raster-thread jank is equally common and harder to diagnose because it doesn't show up in Dart profiling. Always check both the UI and Raster flame charts in Flutter DevTools.
Standard — L4/L5
What is the difference between the UI thread and raster thread in Flutter?
The UI thread runs all Dart code — widget builds, layout computations, and paint recording. It produces a ui.Scene (display list). The raster thread takes that Scene and issues actual GPU commands via Skia or Impeller. They run concurrently (double-buffered): while the raster thread rasterizes frame N, the UI thread builds frame N+1. Each thread has an independent 16.67ms budget at 60Hz.
Mention that jank can originate from either thread — most candidates only think of the UI thread.
Deep Dive — L6/Staff
Walk me through exactly what happens from the time setState() is called to the time pixels appear on screen.
1. setState(fn) calls fn, then calls element.markNeedsBuild(), which adds the element to BuildOwner._dirtyElements (sorted by depth) and calls scheduleFrame() if no frame is already scheduled.

2. scheduleFrame() calls window.scheduleFrame() — a platform channel call that registers for the next vsync interrupt.

3. On vsync, _handleBeginFrame() fires first, executing transient callbacks (AnimationController ticks).

4. _handleDrawFrame() fires next, calling persistent callbacks. WidgetsBinding.drawFrame() is the registered persistent callback.

5. buildOwner.buildScope() rebuilds all dirty elements in depth order by calling element.rebuild() → state.build() → updates child elements.

6. flushLayout() calls performLayout() on all dirty render objects (within relayout boundaries), propagating constraints down and receiving sizes up.

7. flushPaint() calls paint(context, offset) on dirty render objects. Commands are recorded into ui.Picture objects stored in PictureLayers.

8. renderView.compositeFrame() traverses the layer tree and calls SceneBuilder methods, then calls window.render(scene) to hand the Scene to the engine.

9. On the raster thread, RasterCache is checked; uncached pictures are replayed through Skia/Impeller generating GPU commands into command buffers.

10. GPU executes the command buffer, the back buffer is presented via the platform's swap mechanism (SurfaceFlinger on Android, CAMetalLayer on iOS).
