Part 1 — The Layer Tree
From Layer Tree to Scene
The layer tree produced by the paint phase is an intermediate representation — a structured description of compositing operations. The engine converts this into a Scene using SceneBuilder, which is then submitted to the GPU for final rendering.

1. ContainerLayer Hierarchy

The layer tree is built from two categories of nodes: container layers (which can have children) and leaf layers (which cannot). Every layer in Flutter ultimately derives from Layer, with container layers deriving from ContainerLayer.

Layer
├── ContainerLayer
│   ├── OffsetLayer          (root of each RenderView's layer subtree)
│   │   └── AnnotatedRegionLayer  (hit-testing / semantics metadata)
│   ├── TransformLayer        (4×4 matrix transform)
│   ├── OpacityLayer          (alpha compositing)
│   ├── ClipRectLayer         (rectangular clip)
│   ├── ClipRRectLayer        (rounded rect clip)
│   ├── ClipPathLayer         (arbitrary path clip)
│   ├── PhysicalModelLayer    (elevation + shadow + clip)
│   ├── ShaderMaskLayer       (shader-based masking)
│   └── BackdropFilterLayer   (reads existing pixels for filter)
└── PictureLayer              (leaf — holds recorded DisplayList)

Layer Tree Example for a Material Card

// A Material Card with elevation and a child that animates opacity:
OffsetLayer (root)
  PhysicalModelLayer (elevation: 4, clip: rounded rect)
    PictureLayer (card background, text, etc.)
    OpacityLayer (alpha: 0.8 for an animating child)
      PictureLayer (child content)

2. AnnotatedRegionLayer

AnnotatedRegionLayer is a transparent (non-rendering) layer that carries metadata through the layer tree. It is used by:

  • Hit testing: HitTestResult walks the layer tree to find which layers contain a pointer event position
  • SystemChrome: AnnotatedRegion<SystemUiOverlayStyle> communicates status bar / nav bar styling to the platform
  • Semantics: accessibility information propagated through the layer tree
// SystemChrome via AnnotatedRegion
AnnotatedRegion<SystemUiOverlayStyle>(
  value: const SystemUiOverlayStyle(
    statusBarColor: Colors.transparent,
    statusBarBrightness: Brightness.dark,
  ),
  child: Scaffold(...),
)

3. SceneBuilder: Low-Level Scene Assembly

SceneBuilder is the low-level Dart API that converts the layer tree into an engine Scene object. It uses a push/pop stack model mirroring the layer tree hierarchy. You rarely use it directly — the framework calls it during compositeFrame().

// SceneBuilder API (dart:ui — low level)
final builder = ui.SceneBuilder();

// Push a transform (e.g., device pixel ratio scaling)
builder.pushTransform(matrix4.storage);

// Push a clip
builder.pushClipRect(clipRect);

// Add a picture (leaf — the DisplayList from PictureLayer)
builder.addPicture(offset, picture);

// Pop the clip
builder.pop();

// Add a platform view (native Android/iOS view)
builder.addPlatformView(viewId, width: w, height: h);

// Pop the transform
builder.pop();

// Build the Scene
final scene = builder.build();

// Render the scene (sends to GPU)
window.render(scene);
scene.dispose(); // important: release GPU resources

How the Framework Uses SceneBuilder

// From RenderView (simplified)
void compositeFrame() {
  final builder = ui.SceneBuilder();
  layer.addToScene(builder);  // recursively walks layer tree
  final scene = builder.build();
  _window.render(scene);
  scene.dispose();
}

// Each Layer subclass implements addToScene():
class TransformLayer extends ContainerLayer {
  @override
  ui.EngineLayer addToScene(ui.SceneBuilder builder) {
    final engineLayer = builder.pushTransform(_transform.storage);
    addChildrenToScene(builder); // recurse into children
    builder.pop();
    return engineLayer;
  }
}

class PictureLayer extends Layer {
  @override
  ui.EngineLayer addToScene(ui.SceneBuilder builder) {
    builder.addPicture(
      _offset,
      _picture!,
      isComplexHint: isComplexHint,
      willChangeHint: willChangeHint,
    );
    return null; // leaf — no engine layer handle needed
  }
}

4. Retained Rendering and the Raster Cache

Flutter's raster thread maintains a raster cache that stores GPU-rasterized versions of PictureLayer contents. When a layer's picture has not changed since the last frame, the raster thread reuses the cached GPU texture instead of re-rasterizing. This is what makes RepaintBoundaries efficient — they ensure the same PictureLayer object persists across frames when clean.

Raster Cache Key

The cache key is a combination of:

  • The Picture object identity (same Skia SkPicture pointer)
  • The transform matrix (the accumulated transform at the point the layer is composited)

If either changes, the cache entry is invalid and the layer must be re-rasterized. This is why scroll position affects cache validity — the transform matrix changes as items scroll, causing cache misses even if the item's visual content is unchanged.

// RasterCache entry (conceptual)
struct RasterCacheEntry {
  SkPicture* picture;      // what to rasterize
  SkMatrix transform;       // at which transform
  SkBitmap cached_texture;  // the result
  int access_count;         // LRU tracking
};
Staff-Level Insight

The transform-matrix cache key has a subtle implication for scrolling lists: as the ListView scrolls, the transform matrix of each item changes (translation component changes by scroll delta). This means every item's raster cache entry is invalidated every scroll frame, even if the item's visual content is pixel-identical. For ListView items, Flutter avoids this by using RepaintBoundary + raster cache hints (isComplexHint: true, willChangeHint: false) — the framework signals to the raster cache that this picture is complex (worth caching) and won't change (so caching is beneficial). Scrolling then composites the cached textures with an updated transform rather than re-rasterizing.

5. Platform Views: Breaking the Single-Layer Model

Flutter renders entirely within its own layer stack, with all content drawn by the Flutter engine. Platform views (AndroidView, UiKitView) embed native platform UI components (maps, WebViews, camera previews) into the Flutter layer tree. This breaks the single-raster-thread model because native views are rendered by the platform, not by Flutter.

Virtual Display Mode (Android)

In Virtual Display mode, Flutter renders the native view into an off-screen surface (a VirtualDisplay in Android terms). The resulting pixels are captured as a texture, which Flutter then composites as a normal PictureLayer / TextureLayer.

// Virtual Display flow:
// 1. Native view renders into VirtualDisplay (off-screen buffer)
// 2. Flutter reads the VirtualDisplay pixels as a GPU texture
// 3. Flutter composites the texture as part of its layer tree
// 4. Single raster pass — Flutter controls the final composite

// Limitations:
// - Extra memory: off-screen buffer for native view
// - Extra GPU copy: reading VirtualDisplay pixels
// - Accessibility: native view accessibility tree not in Flutter's tree
// - Touch events: must be forwarded through Flutter's hit testing → native

Hybrid Composition (Android)

Hybrid Composition embeds the native view directly in the Android view hierarchy, at the correct z-position. Flutter renders in two layers: one below the native view, one above it. The Android compositor handles the final merge.

// Hybrid Composition flow:
// 1. Flutter renders bottom layer (content below native view) to texture
// 2. Native view renders in its normal Android view hierarchy position
// 3. Flutter renders top layer (content above native view, e.g. overlays)
// 4. Android SurfaceFlinger composites all three layers

// Trade-offs vs Virtual Display:
// Pros: Better accessibility, correct touch handling, no texture copy
// Cons: Flutter must synchronize with Android view system (frame sync issues),
//       potential for flickering during scroll, more complex z-ordering
Performance Warning

Both platform view modes impose significant overhead. Virtual Display adds a GPU texture copy per frame. Hybrid Composition requires Flutter to split its rendering into two separate GPU passes per frame (above and below the native view). On screens with many platform views (e.g., a map with 10 overlaid markers using native views), the overhead compounds. Prefer Flutter-native equivalents (flutter_map, webview_flutter's Surface rendering mode) where possible.

iOS Platform Views

On iOS, the equivalent of Hybrid Composition is the default mode. Native UIViews are placed in the platform view hierarchy, and Flutter renders its content into a CAMetalLayer / CAEAGLLayer that sits at the correct z-position. The iOS Core Animation compositor handles final composition.

// iOS platform view embedding
UiKitView(
  viewType: 'plugins.flutter.io/google_maps',
  onPlatformViewCreated: (int id) {
    // id is the platform view ID registered with SceneBuilder.addPlatformView
  },
  creationParams: mapOptions,
  creationParamsCodec: const StandardMessageCodec(),
)

6. Scene Submission and Frame Timing

The final step of each frame on the UI thread is calling window.render(scene). This passes the Scene object to the engine, which queues it for the raster thread. The sequence is:

  1. UI thread: builds widget tree, runs layout, runs paint, builds scene → sends scene to raster thread queue
  2. Raster thread: receives scene, walks layer tree, rasterizes PictureLayers (using raster cache), submits GPU draw calls
  3. GPU: executes draw calls, produces pixels in the framebuffer
  4. Display: at the next vsync, SurfaceFlinger (Android) / Core Animation (iOS) presents the framebuffer
// Frame timing from engine perspective
// UI thread budget: ~8ms (for 120Hz) or ~16ms (for 60Hz)
// Raster thread budget: ~8ms or ~16ms (runs concurrently with UI thread)
// Total: if both finish within one vsync period → smooth frame
//
// Jank: if UI thread OR raster thread exceeds budget → frame is late
// → missed vsync → dropped frame → visible stutter
Staff-Level Insight

The UI and raster threads are pipelined, not synchronized per-frame. The raster thread works on frame N's scene while the UI thread prepares frame N+1's scene. This pipeline means that a spike in UI thread time delays the raster thread's next frame, and vice versa. In Flutter DevTools' performance timeline, you can observe both threads' timelines side by side — a raster thread spike that exceeds 16ms causes a jank even if the UI thread finished in 2ms. This is a common confusion: jank is caused by whichever of the two threads misses the vsync deadline, not just UI thread slowness.

7. Engine Layer Handles and Retained Layers

When addToScene() is called on a ContainerLayer, it returns an ui.EngineLayer handle. This handle represents the GPU-side retained state for that layer. Flutter stores these handles and reuses them on subsequent frames where the layer hasn't changed.

// Retained layer usage — SceneBuilder can reuse engine layers
class TransformLayer extends ContainerLayer {
  ui.EngineLayer? _engineLayer;

  @override
  ui.EngineLayer addToScene(ui.SceneBuilder builder) {
    _engineLayer = builder.pushTransform(
      _transform.storage,
      oldLayer: _engineLayer, // reuse if available — GPU retained state
    );
    addChildrenToScene(builder);
    builder.pop();
    return _engineLayer!;
  }
}

Passing oldLayer to SceneBuilder methods allows the engine to skip re-uploading GPU state for unchanged layers, further reducing per-frame GPU overhead. This is "retained rendering" — unchanged portions of the layer tree are submitted to the GPU with handles to previously-uploaded state rather than re-uploading everything.
