Core Mental Model
Flutter's layout protocol is a single message exchange: the parent sends constraints down, the child returns a size up. The parent then positions the child. No negotiation, no second pass (unless intrinsics are used). Understanding when constraints are tight vs. loose explains almost every mysterious layout bug.

BoxConstraints in Detail

BoxConstraints is a rectangle of allowed sizes defined by four values: minWidth, maxWidth, minHeight, maxHeight. Any size where minWidth <= width <= maxWidth and minHeight <= height <= maxHeight is valid.

// The four constraint types every Flutter engineer must internalize

// 1. TIGHT: forced to exactly one size (min == max)
BoxConstraints.tight(Size(300, 200))
// → minWidth=300, maxWidth=300, minHeight=200, maxHeight=200
// Child MUST be 300×200. No choice.

// 2. LOOSE: child can be anything from zero up to a maximum
BoxConstraints.loose(Size(300, 200))
// → minWidth=0, maxWidth=300, minHeight=0, maxHeight=200
// Child can shrink-wrap or fill, as long as it doesn't exceed 300×200.

// 3. EXPAND: forces child to fill available space
BoxConstraints.expand()          // fill screen
BoxConstraints.expand(width: 200) // fixed width, expand height

// 4. UNBOUNDED: maxWidth or maxHeight is double.infinity
// Occurs inside ListView, Column with no constraints, etc.
// Child MUST NOT return infinity as its size.

Tight Constraints: The Root of Many Mysteries

The RenderView (the root of the render tree) passes tight constraints equal to the device's physical pixel dimensions divided by device pixel ratio. This propagates down as tight through most full-screen layouts, which is why a bare Center widget fills the screen — it receives tight constraints and sizes itself to them.

Screen (tight: 390×844)
  └── MaterialApp (tight: 390×844)
       └── Scaffold (tight: 390×844)
            └── body: Center (receives tight 390×844 — sizes itself to fill)
                 └── child: Text("Hi") (receives loose 0–390 × 0–844)
                      → Text reports its natural (intrinsic) size
                      → Center positions it at center

Loose Constraints: How Children Shrink-Wrap

When a widget converts tight constraints to loose ones (e.g., Align, Center, UnconstrainedBox), children are free to report their natural size. This is why wrapping a widget in Center causes it to shrink-wrap: Center passes loose constraints to its child and positions it at the center of its own (tight) bounds.

// Center's performLayout — simplified
@override
void performLayout() {
  // Center receives tight constraints from parent
  // It sizes itself to fill those constraints
  size = constraints.biggest;  // fill the tight constraints

  if (child != null) {
    // Pass LOOSE constraints to child → child shrink-wraps
    child!.layout(constraints.loosen(), parentUsesSize: true);
    // Position child at center of our size
    final double dx = (size.width - child!.size.width) * (widthFactor ?? 0.5);
    final double dy = (size.height - child!.size.height) * (heightFactor ?? 0.5);
    (child!.parentData! as BoxParentData).offset = Offset(dx, dy);
  }
}

Unbounded Constraints: The Source of RenderFlex Overflow

Certain widgets pass unbounded constraints (max = infinity) to their children. ListView passes maxHeight = infinity along the scroll axis so items can have any height. Row passes maxWidth = infinity to non-flex children. Problems arise when a child widget tries to fill an unbounded constraint — Expanded inside a Column inside another Column with unbounded height is the classic failure case.

Common Interview Trap
"Why does putting a Column inside a ListView throw an overflow error?" Because ListView passes unbounded vertical constraints to its children. A Column with Expanded children receives unbounded constraints and tries to compute the expanded children's sizes, which requires a finite available height. The fix: don't use Expanded inside a Column that's inside a ListView, or wrap the inner Column in a fixed-height container.

Common Layout Widget Internals

Container — A Delegation Chain

Container is a convenience widget that delegates to a chain of more primitive widgets based on which properties are set. This is important for performance intuition — a Container with all properties set creates many intermediate render objects:

Container (with all properties)
  └── LimitedBox (when unconstrained)
       └── ConstrainedBox (constraints: BoxConstraints)
            └── DecoratedBox (decoration: BoxDecoration)
                 └── Transform (transform: matrix)
                      └── Padding (padding: EdgeInsets)
                           └── Align (alignment: Alignment)
                                └── child

When only some properties are set, intermediate widgets are omitted. Container(child: child) with no other properties is equivalent to child wrapped in LimitedBox.

RenderFlex: Column and Row Layout Algorithm

The flex layout algorithm executes in two steps:

Step 1: Layout Inflexible Children (flex: 0)

Non-flexible children (no Expanded or Flexible wrapper) are laid out first with their cross-axis constrained and their main-axis unconstrained (unbounded). This determines how much space inflexible children consume.

Step 2: Distribute Remaining Space to Flexible Children

After step 1, the remaining main-axis space is computed: freeSpace = mainAxisExtent - inflexibleSpace. This is distributed proportionally by flex factor:

// RenderFlex.performLayout() — simplified core logic
double allocatedSize = 0.0;
double allocatedFlexSpace = 0.0;
int totalFlex = 0;

// Pass 1: inflexible children
RenderBox? child = firstChild;
while (child != null) {
  final int flex = _getFlex(child);
  if (flex == 0) {
    final BoxConstraints innerConstraints = _getInnerConstraints(constraints);
    child.layout(innerConstraints, parentUsesSize: true);
    allocatedSize += _getMainSize(child);
  } else {
    totalFlex += flex;
  }
  child = childAfter(child);
}

// Pass 2: flexible children
final double freeSpace = math.max(0.0, _getMainSize(constraints) - allocatedSize);
child = firstChild;
while (child != null) {
  final int flex = _getFlex(child);
  if (flex > 0) {
    final double spaceForChild = freeSpace * flex / totalFlex;
    // FlexFit.tight → child must fill exactly spaceForChild
    // FlexFit.loose → child can be up to spaceForChild
    child.layout(BoxConstraints.tightFor(main: spaceForChild, cross: crossConstraint),
                 parentUsesSize: true);
    allocatedFlexSpace += spaceForChild;
  }
  child = childAfter(child);
}

RenderStack: Positioned vs Non-Positioned Children

Stack layout is simpler than flex but has important nuances:

  • Non-positioned children: laid out with loose constraints (the stack's constraints loosened). The stack's size is the bounding box of all non-positioned children (or the stack's constraints if fit: StackFit.expand).
  • Positioned children (wrapped in Positioned): constrained by their position values (top, left, right, bottom, width, height). They do not affect the stack's size.

SizedBox vs ConstrainedBox vs FractionallySizedBox

Intrinsic Sizes: The O(n²) Trap

Normally, Flutter's layout is O(n) — each render object is visited once. Intrinsic size queries break this guarantee by forcing a second layout pass on the queried subtree.

The Four Intrinsic Methods

// On RenderBox — all four can be expensive
double computeMinIntrinsicWidth(double height);
// Minimum width at which content does not overflow or wrap.
// For Text: width of longest unbreakable word.

double computeMaxIntrinsicWidth(double height);
// Width the content would prefer if unconstrained.
// For Text: width of entire text on one line.

double computeMinIntrinsicHeight(double width);
// Minimum height at which content does not overflow.
// For single-line Text: line height.

double computeMaxIntrinsicHeight(double width);
// Natural height at given width.
// For multi-line Text: height of all wrapped lines.

Why Intrinsics Are Expensive

When IntrinsicWidth or IntrinsicHeight lays out its child, it first asks for the intrinsic size, then lays out the child with a tight constraint at that intrinsic size. This means every descendant must perform its intrinsic size computation, which may recursively query its children. The result: O(n²) layout complexity in the worst case for a tree with n nodes.

Performance Anti-Pattern
Using IntrinsicHeight inside a ListView is catastrophic. Each list item forces an O(n) intrinsic pass on its subtree. For a list of 100 items each with a subtree of depth 10, that's up to 1,000 extra layout computations per frame. The fix: give list items explicit heights, or restructure the layout to avoid needing intrinsics.

When Intrinsics Are Acceptable

  • One-time layout computations (not in a scroll list)
  • Small, shallow subtrees
  • When the layout truly cannot be expressed without knowing intrinsic sizes
  • Alternative: use LayoutBuilder to respond to constraints rather than querying intrinsics

Sliver Layout Protocol

The sliver protocol is a more complex, scroll-aware layout system used by CustomScrollView and its children. Unlike box layout, slivers understand scrolling and can paint lazily.

SliverConstraints

class SliverConstraints {
  final AxisDirection axisDirection;  // scroll direction
  final double scrollOffset;          // how far the sliver has been scrolled past
  final double overlap;               // how much preceding slivers overlap this one
  final double remainingPaintExtent;  // remaining space in viewport
  final double crossAxisExtent;       // width (for vertical scroll)
  final double viewportMainAxisExtent; // full viewport height
  final double remainingCacheExtent;  // remaining cache area
  final double cacheOrigin;           // where cache starts (negative for above-viewport)
}

SliverGeometry

class SliverGeometry {
  final double scrollExtent;     // total scrollable length of this sliver
  final double paintExtent;      // how much of this sliver is currently painted
  final double layoutExtent;     // how much layout space this sliver consumes this frame
  final double maxPaintExtent;   // maximum paint extent (for scrollbar calculations)
  final bool hasVisualOverflow;  // clips needed?
  final double hitTestExtent;    // hit test area (can differ from paintExtent)
  final bool visible;            // should be painted?
}

How RenderSliverList Achieves O(1) Rendering

RenderSliverList only lays out children that intersect the viewport plus a configurable cache extent (default: 250 logical pixels above and below the viewport). This is why ListView with 10,000 items has the same frame time as one with 10 items — only ~10-20 items are ever laid out.

Viewport (500px tall)
  Cache extent above: 250px
  ┌─────────────────────────────┐ ← cacheOrigin (−250px scrollOffset)
  │  [CACHED: item 3, item 4]   │   laid out, not painted
  ├─────────────────────────────┤ ← 0px (top of visible area)
  │  [VISIBLE: item 5]          │
  │  [VISIBLE: item 6]          │   laid out + painted
  │  [VISIBLE: item 7]          │
  ├─────────────────────────────┤ ← 500px (bottom of visible area)
  │  [CACHED: item 8, item 9]   │   laid out, not painted
  └─────────────────────────────┘ ← end of cache extent (+250px)
  [items 0-2, 10+: not laid out at all]

CustomScrollView Sliver Coordination

RenderViewport drives the sliver layout. It assigns each sliver a scrollOffset computed from the cumulative scrollExtent of preceding slivers. A sliver only receives the portion of scroll offset that falls within it — e.g., if a SliverAppBar has scrollExtent = 200 and the current scroll position is 150, the next sliver receives scrollOffset = 0 (hasn't scrolled into it yet).

LayoutBuilder: Responsive Layout Without Intrinsics

LayoutBuilder provides a responsive alternative to intrinsic size queries. It creates a RenderObject (RenderLayoutBuilder) that defers calling its builder function until layout time, passing the actual BoxConstraints from the parent:

LayoutBuilder(
  builder: (BuildContext context, BoxConstraints constraints) {
    // constraints.maxWidth is the ACTUAL available width at layout time
    if (constraints.maxWidth > 600) {
      return WideLayout();
    } else {
      return NarrowLayout();
    }
  },
)

Unlike MediaQuery, LayoutBuilder responds to the widget's position in the layout tree — the available width accounts for parent padding, sidebars, etc. This is the correct tool for widget-level responsive design.

Staff-Level Nuance
LayoutBuilder rebuilds its subtree during the layout phase, not the build phase. This means it runs later than normal widget builds. If the builder function calls setState() synchronously, it will schedule a rebuild but the current frame will proceed with the current state. The rebuild happens on the next frame. This is the correct behavior — never mutate state during layout.
Standard — L4/L5
What is the difference between tight and loose constraints in Flutter?
Tight constraints have minWidth == maxWidth and/or minHeight == maxHeight, forcing the child to be exactly that size. Loose constraints have min = 0, allowing the child to be anywhere from zero up to the max. The root RenderView passes tight constraints equal to screen size. Center and Align loosen constraints before passing to their children, which is why children of Center shrink-wrap.
Follow up with: "Why does a Container fill the screen when it's the direct child of Scaffold body?" Answer: Scaffold passes tight constraints from the screen size, so Container expands to fill them.
Deep Dive — L6/Staff
Why is IntrinsicHeight inside a ListView a performance problem, and what are the alternatives?
IntrinsicHeight forces each of its descendants to run computeMaxIntrinsicHeight(), which is a separate, uncached layout pass. In a list of N items each containing a widget tree of depth D, this becomes O(N×D) intrinsic computations per frame where N items are visible — instead of O(N×D) total per frame for the entire list normally, intrinsics can trigger multiple passes per node.

Concretely: for a ListView with cross-axis IntrinsicHeight children (e.g., trying to make a Row of widgets all the same height), each visible item triggers a full intrinsic pass on its subtree.

Alternatives:
1. Give items explicit heights — most performant.
2. Use CrossAxisAlignment.stretch on a Row — propagates constraints instead of querying intrinsics.
3. Restructure to avoid needing equal heights (e.g., use a fixed-height card design).
4. If heights are data-driven, cache them externally and use SliverFixedExtentList or pass the height as a tight constraint.
Demonstrate you know the O(n²) terminology and can name the specific cause: each node in the subtree runs computeMaxIntrinsicHeight(), which recursively queries children.
