Navigator Fundamentals

Standard — L4/L5
Explain the Navigator 1.0 back stack. What happens when you push and pop routes?
Navigator 1.0 maintains an ordered stack of Route objects. Navigator.push(context, route) adds a route to the top of the stack; the new screen is displayed and the previous screen is kept in memory (it's obscured, not destroyed). Navigator.pop(context) removes the top route, revealing the one below. pushReplacement replaces the top route without adding to stack depth. pushAndRemoveUntil pushes a new route and removes all existing routes matching a predicate — common after login to clear the auth stack. Each route has an associated AnimationController for the page transition.
Mention that routes below the top are not disposed — their state is preserved. This is why navigating back to a list screen restores scroll position automatically.
Standard — L4/L5
What are the three core components of Navigator 2.0 and what is each responsible for?
RouterDelegate — owns the current navigation state (a list of Pages) and rebuilds the Navigator widget when state changes. It listens to changes from the RouteInformationParser and translates them into pages. RouteInformationParser — converts a URL string (RouteInformation) into an app-specific route configuration object (e.g., a path segments model), and vice versa. This is what enables URL-to-state and state-to-URL synchronization. RouteInformationProvider — provides the initial URL and notifies the system when the app URL changes (used on web for browser history, on mobile for deep links). GoRouter implements all three internally, which is why you don't implement them manually.
Standard — L4/L5
What is the difference between context.go() and context.push() in GoRouter?
context.go('/path') navigates to the route and resets the navigator stack — there is no back button to the previous location. It's equivalent to opening a fresh page in the browser. Use it for top-level navigation: switching tabs, going home after logout, post-login redirect. context.push('/path') pushes the new route on top of the current stack, preserving the back button. Use it for drill-down navigation: home → product list → product detail. Pressing back from the detail returns to the list. A common bug is using go for detail screens — the user cannot go back.
Deep Dive — L5/L6
How do you implement an authentication flow with GoRouter so that deep links to protected pages still work after login?
Implement a global redirect in GoRouter. When an unauthenticated user navigates to a protected route, capture the intended destination in a query parameter: return '/login?from=${Uri.encodeComponent(state.uri.toString())}. The login screen reads this parameter. On successful login, the auth state changes, triggering GoRouterRefreshStream to notify GoRouter to re-evaluate redirects. At this point the user is authenticated, so the redirect function returns null — but the router is now on /login?from=/product/123. In the login screen's success callback, call context.go(Uri.decodeComponent(from)) to navigate to the original destination.

Key: use refreshListenable: GoRouterRefreshStream(authBloc.stream) so the redirect is re-evaluated on every auth state change — this handles both login (redirect to destination) and logout (redirect to login) automatically.
Without refreshListenable, the redirect is only evaluated on navigation events. Auth state changes (e.g., token expiry) would not trigger re-evaluation, leaving the user on a protected screen.
Standard — L4/L5
How do nested navigators work? When would you use them?
A nested Navigator is a Navigator widget inside another widget (itself inside a parent Navigator). Each Navigator has its own independent back stack. Use cases: (1) tab-based navigation — each tab has its own Navigator so navigating within Tab A doesn't affect Tab B's stack; (2) onboarding flows — a sub-navigator for multi-step onboarding that can be dismissed entirely; (3) in-app browser or WebView with its own navigation. In GoRouter, StatefulShellRoute.indexedStack creates a nested navigator per branch, enabling independent tab stacks. The OS back button/gesture pops from the innermost active navigator first.

GoRouter Deep Dives

Deep Dive — L5/L6
Explain ShellRoute vs StatefulShellRoute.indexedStack — what problem does each solve?
ShellRoute wraps a group of routes in a persistent shell widget (BottomNavigationBar, NavigationRail). The shell stays mounted; the child widget changes as routes change. However, navigating between tabs resets each tab's stack — navigating to Tab B and back to Tab A loses Tab A's back history. StatefulShellRoute.indexedStack solves this by creating a separate Navigator per branch (tab). Each branch's Navigator stack is preserved independently using an IndexedStack widget — only the active tab is visible, but all tabs remain mounted (their state is preserved). This is the correct pattern for apps like Gmail, Instagram, and YouTube where each tab maintains its own navigation history.
Tradeoff: StatefulShellRoute keeps all tab widgets mounted simultaneously, increasing memory usage. For memory-constrained scenarios, consider lazy mounting strategies or limiting the number of preserved tabs.
Deep Dive — L5/L6
Why can't you use state.extra for deep-linkable routes, and what should you use instead?
state.extra is an in-memory Dart object passed alongside navigation. It is only available when navigating programmatically within the running app. When the app is opened via a deep link (cold start), the OS provides only the URL string — the extra object does not exist in that URL. Similarly, on web, if the user refreshes the page or opens a bookmarked URL, extra is lost. For deep-linkable routes, pass identifiers (IDs) as path parameters or query parameters in the URL. The screen then fetches the full object from the repository using the ID. This also makes the URL self-contained and shareable.
Standard — L4/L5
How does GoRouter handle the initial deep link on cold start?
When the app is cold-started via a deep link (e.g., https://myapp.com/product/123), the OS delivers the URL to the Flutter engine via the platform channel. GoRouter's RouteInformationProvider reads this initial URL instead of initialLocation. GoRouter then matches the URL against its route tree and builds the appropriate page(s). If the URL matches a nested route (e.g., /home/settings), GoRouter builds the full stack including parent routes, so the back button works correctly. You don't need to handle the initial link manually when using GoRouter — the integration is automatic as long as your intent filters (Android) and AASA (iOS) are configured.

Deep Linking

Deep Dive — L5/L6
What is the difference between Android App Links and custom URI schemes? Why prefer App Links?
Custom URI schemes (myapp://path) are unverified — any app can register any scheme, making them spoofable (a malicious app can intercept your deep links). They also show an app chooser dialog if multiple apps claim the same scheme, and fail completely if the app is not installed. Android App Links use HTTPS URLs verified via Digital Asset Links JSON hosted at /.well-known/assetlinks.json. The OS verifies the link at install time using the app's signing certificate fingerprint. Verified links open directly in the app with no dialog. If the app is not installed, the link opens in the browser (where you can serve a web version or an install prompt). This also enables web-app parity and content indexing by Google.
Standard — L4/L5
How do you handle a deep link when the app is already running in the background (warm start)?
For warm start, the OS delivers the deep link via a platform message rather than as the initial URL. With the app_links package, listen to _appLinks.uriLinkStream. When a URI arrives, extract the path and navigate using _router.go(path). This must be set up during app initialization (in initState or a service initialized at startup) and the subscription must be cancelled on dispose. GoRouter does NOT automatically handle warm start links — only the initial cold-start URL is handled automatically. This is a common production bug where deep links work on cold start but not when the app is backgrounded.
Always test both cold start (kill the app fully, tap the link) and warm start (background the app, tap the link) — they exercise different code paths.
Deep Dive — L6
How would you design navigation for an app with both a bottom nav tab structure AND a root-level auth flow, with deep link support?
Route structure: two top-level route groups. (1) Auth group — /login, /signup, /forgot-password (no shell). (2) Main app group — StatefulShellRoute.indexedStack wrapping /home, /search, /profile and their sub-routes.

GoRouter global redirect: check auth state on every navigation. Unauthenticated users are redirected to /login?from=encodedDestination. refreshListenable re-evaluates on auth state change.

Deep link flow: user taps https://myapp.com/product/123. App cold-starts. GoRouter reads URL. Redirect function sees user is unauthenticated, redirects to /login?from=%2Fproduct%2F123. User logs in. Auth state changes. refreshListenable fires. Redirect re-evaluates — user is now authenticated, returns null. Login screen's success handler calls context.go('/product/123'). User sees product screen with shell (back navigates within the tab stack).

Product route is nested under /home branch of StatefulShellRoute so the shell (bottom nav) is present. The Home tab's back stack is: home → product/123.
Standard — L4/L5
What is GoRouter vs implementing Navigator 2.0 manually? When would you not use GoRouter?
GoRouter abstracts the RouterDelegate, RouteInformationParser, and RouteInformationProvider into a declarative route configuration. Benefits: URL-based routing out of the box, redirect guards, ShellRoute, deep link handling, named routes, and active development by the Flutter team. Manual Navigator 2.0 is warranted when: (1) your navigation logic is highly custom (e.g., a game with non-standard transitions), (2) you need to integrate with an existing navigation system, or (3) GoRouter's abstractions are insufficient for your route graph complexity. In practice, GoRouter handles 95%+ of production requirements and is the recommended choice for new apps.
Deep Dive — L5/L6
How do you pass data between routes in GoRouter, and what are the trade-offs of each approach?
Three approaches: (1) Path parameters (/product/:id) — encoded in the URL, survives deep links and refreshes, but limited to strings. Best for identifiers. (2) Query parameters (/search?q=flutter) — also URL-encoded, survives deep links, good for optional filters or pagination state. Still string-only. (3) extra — in-memory Dart object, type-safe, supports complex objects, but lost on deep link cold start, browser refresh, or state restoration. Use extra only for data that is not needed for the initial render from a URL (e.g., a pre-fetched object for performance, where the screen can also fetch by ID if extra is null). A robust pattern: pass the ID via path parameter, fetch the object in the screen, and optionally accept extra as a performance shortcut.
Standard — L4/L5
How do you handle the Android system back gesture with GoRouter?
GoRouter integrates with Flutter's Router widget which handles the Android back gesture via PopScope (formerly WillPopScope). When the user presses back or uses the predictive back gesture (Android 13+), GoRouter pops the top route from the current navigator. With StatefulShellRoute, pressing back within a tab pops within that tab's stack; when the tab stack is empty, GoRouter defers to the parent navigator or exits the app. To intercept back: wrap your screen in PopScope(canPop: false, onPopInvoked: (didPop) { ... }) — useful for "unsaved changes" confirmation dialogs. For Android 14+ predictive back, use the PopScope API which supports the predictive back animation.
Staff-Level Summary

Navigation interview questions at L6 probe: (1) Navigator 2.0 component roles and why GoRouter exists, (2) the exact semantics of go vs push vs replace and when each causes bugs, (3) the auth redirect pattern with refreshListenable, (4) StatefulShellRoute for per-tab stacks, (5) deep link cold vs warm start distinction, and (6) designing a complete navigation architecture for a complex app. The most common gap in L5 candidates: not understanding that warm start deep links require explicit stream listening — GoRouter only handles cold start automatically.
