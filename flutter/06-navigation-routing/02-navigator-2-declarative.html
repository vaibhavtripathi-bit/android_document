Staff / L6 Core Concept
Navigator 2.0 exists to solve a fundamental limitation of Navigator 1.0: the URL bar (on Flutter Web) and deep links cannot drive the navigation stack declaratively. At Staff level, you must articulate why Navigator 2.0 was designed this way, understand each of the four collaborating classes, and know why the ecosystem largely wraps it with GoRouter rather than using the raw API.

1. Why Navigator 2.0 Exists

Navigator 1.0 is imperative: you call push() / pop() and the stack is mutated. This works fine in mobile-only apps, but breaks down in two critical scenarios:

  1. Flutter Web — URL synchronization: When a user navigates to example.com/products/123, the app must reconstruct the correct navigation stack from the URL. Navigator 1.0 has no mechanism for this — you can set an initialRoute, but you cannot represent a full stack (e.g., Home → Products → Product 123) from a URL string without custom bootstrapping logic.
  2. Deep links from OS: The OS sends a URL to the app while it may already be running. The app must respond by pushing the correct sequence of routes. With Navigator 1.0, this requires imperative code scattered across your app lifecycle handlers.

Navigator 2.0 inverts the model: instead of the app calling the navigator, the navigator asks your app "given this URL / application state, what should the page stack look like?" — and your app returns a declarative list of Page objects.

2. The Four Collaborators

  Platform URL change / deep link
           │
           ▼
  RouteInformationProvider          ← wraps platform URL stream
           │ RouteInformation(uri, state)
           ▼
  RouteInformationParser            ← parse URI → app state object (T)
           │ T (your app state, e.g., AppRoute)
           ▼
  RouterDelegate<T>                 ← owns Navigator, builds page list
     ├── currentConfiguration: T   ← current state → URI (for browser bar)
     ├── build() → Navigator(pages: [...])
     ├── setNewRoutePath(T)         ← called when parser gives new state
     └── popRoute()                ← called by BackButtonDispatcher
           │
           ▼
  BackButtonDispatcher              ← intercepts Android back / browser back
           │
           ▼
  RouterDelegate.popRoute()

3. Router Widget: The Orchestrator

The Router widget (used internally by MaterialApp.router) wires the four collaborators together. You rarely instantiate it directly; instead you pass the pieces to MaterialApp.router:

MaterialApp.router(
  routerDelegate: MyRouterDelegate(),
  routeInformationParser: MyRouteInformationParser(),
  // routeInformationProvider defaults to PlatformRouteInformationProvider
  // backButtonDispatcher defaults to RootBackButtonDispatcher
);

When the platform reports a new URL (browser navigation, deep link, or initial route), Router:

  1. Gets RouteInformation from the RouteInformationProvider
  2. Calls parser.parseRouteInformation(routeInformation) → app state T
  3. Calls delegate.setNewRoutePath(T)
  4. Calls delegate.build(context) → Navigator(pages: [...])

When app state changes (user taps something), the delegate calls notifyListeners(), which triggers Router to call delegate.build() and optionally update the URL via delegate.currentConfiguration → parser.restoreRouteInformation(T).

4. Page vs Route

This distinction is critical and often confused.

Concept Class Role Lifecycle owner
Page Page<T> (abstract) Immutable configuration — describes what should be on screen Your app / RouterDelegate
Route Route<T> (abstract) Mutable runtime object — handles transitions, animations, overlay entries Navigator (creates from Page via page.createRoute())

When you provide a pages list to Navigator, the Navigator diffs the list and creates/destroys Route objects as needed. The Page's key is used for diffing — same key = same route (state preserved), different key = new route created.

// A typed Page subclass
class ProductPage extends Page<void> {
  final String productId;

  const ProductPage({required this.productId})
      : super(key: ValueKey(productId));  // key drives diffing

  @override
  Route<void> createRoute(BuildContext context) {
    return MaterialPageRoute<void>(
      settings: this,  // passes RouteSettings (including name/arguments)
      builder: (context) => ProductDetailScreen(productId: productId),
    );
  }
}

5. RouteInformationParser

Converts between RouteInformation (URI + state blob) and your typed app state T. Must be bidirectional.

// App state — represents what navigation should look like
class AppRoutePath {
  final bool isHome;
  final String? productId;
  final bool isUnknown;

  const AppRoutePath.home() : isHome = true, productId = null, isUnknown = false;
  const AppRoutePath.product(this.productId) : isHome = false, isUnknown = false;
  const AppRoutePath.unknown() : isHome = false, productId = null, isUnknown = true;
}

class AppRouteInformationParser
    extends RouteInformationParser<AppRoutePath> {

  // Platform URL → app state
  @override
  Future<AppRoutePath> parseRouteInformation(
      RouteInformation routeInformation) async {
    final uri = routeInformation.uri;
    if (uri.pathSegments.isEmpty) return const AppRoutePath.home();
    if (uri.pathSegments.length == 2 &&
        uri.pathSegments[0] == 'products') {
      return AppRoutePath.product(uri.pathSegments[1]);
    }
    return const AppRoutePath.unknown();
  }

  // App state → URL (for browser address bar / state restoration)
  @override
  RouteInformation? restoreRouteInformation(AppRoutePath configuration) {
    if (configuration.isHome) {
      return RouteInformation(uri: Uri.parse('/'));
    }
    if (configuration.productId != null) {
      return RouteInformation(
          uri: Uri.parse('/products/${configuration.productId}'));
    }
    return null;
  }
}

6. RouterDelegate

RouterDelegate is the core class. It owns your navigation state and returns the correct Navigator widget from build(). It must notify listeners when state changes so Router can rebuild.

class AppRouterDelegate extends RouterDelegate<AppRoutePath>
    with ChangeNotifier, PopNavigatorRouterDelegateMixin<AppRoutePath> {

  // PopNavigatorRouterDelegateMixin requires this key
  @override
  final GlobalKey<NavigatorState> navigatorKey = GlobalKey<NavigatorState>();

  // Internal navigation state
  String? _selectedProductId;
  bool _show404 = false;

  // === currentConfiguration: converts state → URL ===
  // Called by Router to update the browser URL bar
  @override
  AppRoutePath get currentConfiguration {
    if (_show404) return const AppRoutePath.unknown();
    if (_selectedProductId != null) {
      return AppRoutePath.product(_selectedProductId);
    }
    return const AppRoutePath.home();
  }

  // === setNewRoutePath: URL change → update internal state ===
  // Called when a new URL is pushed (browser back/forward, deep link)
  @override
  Future<void> setNewRoutePath(AppRoutePath configuration) async {
    if (configuration.isUnknown) {
      _show404 = true;
      _selectedProductId = null;
    } else if (configuration.productId != null) {
      _selectedProductId = configuration.productId;
      _show404 = false;
    } else {
      _selectedProductId = null;
      _show404 = false;
    }
    notifyListeners();
  }

  // === build: state → Navigator with pages list ===
  @override
  Widget build(BuildContext context) {
    return Navigator(
      key: navigatorKey,
      pages: [
        const MaterialPage(
          key: ValueKey('HomePage'),
          child: HomePage(),
        ),
        if (_show404)
          const MaterialPage(
            key: ValueKey('404'),
            child: UnknownPage(),
          )
        else if (_selectedProductId != null)
          ProductPage(productId: _selectedProductId!),
      ],
      onDidRemovePage: (page) {
        // Called when Navigator removes a page (e.g., user pops)
        // Update your state to match
        if (page is ProductPage) {
          _selectedProductId = null;
          notifyListeners();
        }
      },
    );
  }

  // App-code-triggered navigation — not from URL
  void selectProduct(String productId) {
    _selectedProductId = productId;
    notifyListeners(); // triggers build() + URL bar update
  }
}
onPopPage vs onDidRemovePage
In Flutter 3.16+, onPopPage was replaced with onDidRemovePage. The old onPopPage returned a bool to control whether the pop was allowed; onDidRemovePage is purely a notification — the page has already been removed. Use PopScope inside your page widgets for pop prevention instead.

7. RouteInformationProvider

Wraps the platform's URL mechanism. The default implementation — PlatformRouteInformationProvider — listens to:

  • SystemChannels.navigation for initial route and deep-link pushes
  • Browser history API on Flutter Web (via BrowserHistory or HashUrlStrategy)

You rarely subclass this in production. The main use case is testing (inject a mock provider) or URL strategy selection on web:

// main.dart — web URL strategy
import 'package:flutter_web_plugins/url_strategy.dart';

void main() {
  // Use path-based URLs (/products/123) instead of hash (#/products/123)
  usePathUrlStrategy();
  runApp(const MyApp());
}

8. BackButtonDispatcher

RootBackButtonDispatcher listens to the platform back button (Android hardware back, browser back). When triggered, it calls RouterDelegate.popRoute(). Your delegate should return true if it handled the pop (e.g., removed a page from the stack), or false to let the OS handle it (exit app on Android).

// In RouterDelegate — handle back button
@override
Future<bool> popRoute() async {
  if (_selectedProductId != null) {
    _selectedProductId = null;
    notifyListeners();
    return true;  // handled — don't exit app
  }
  return false;  // let OS handle (exit app)
}

For nested navigators, use ChildBackButtonDispatcher:

// Nested Router (e.g., tab child) uses a child dispatcher
class NestedApp extends StatefulWidget { /* ... */ }

class _NestedAppState extends State<NestedApp> {
  late ChildBackButtonDispatcher _dispatcher;

  @override
  void didChangeDependencies() {
    super.didChangeDependencies();
    final parentDispatcher = Router.of(context).backButtonDispatcher!;
    _dispatcher = parentDispatcher.createChildBackButtonDispatcher();
    _dispatcher.takePriority();  // claim back button handling
  }

  @override
  Widget build(BuildContext context) {
    return Router(
      routerDelegate: _nestedDelegate,
      backButtonDispatcher: _dispatcher,
    );
  }
}

9. Page Diffing Algorithm

When RouterDelegate.build() returns a new Navigator with an updated pages list, the Navigator reconciles the old and new lists using a diffing algorithm. Understanding this is important for correctness and performance.

Staff / L6 Detail — Page Diffing
The Navigator compares old and new pages lists from the bottom up. It matches pages by key. Pages present in both lists → their routes are kept (state preserved). Pages in the new list but not the old → new routes are created and pushed. Pages in the old list but not the new → routes are popped/removed. The order matters: a reorder of pages is treated as removal + insertion, not a swap — this can cause unexpected animations. 
// Example: going from [Home, Product] to [Home, Cart]
// Navigator will:
// 1. Keep Home route (same key)
// 2. Pop Product route (no longer in list)
// 3. Push Cart route (new in list)
// This triggers a pop animation on Product and push animation on Cart.

// To avoid the pop animation when replacing (not returning):
// Use a custom Page.createRoute() that returns a route with no animation.
class ReplacePage extends Page<void> {
  final Widget child;
  const ReplacePage({required this.child, required super.key});

  @override
  Route<void> createRoute(BuildContext context) {
    return PageRouteBuilder<void>(
      settings: this,
      pageBuilder: (_, __, ___) => child,
      transitionDuration: Duration.zero,
    );
  }
}

10. Why the Ecosystem Wraps Navigator 2.0

Navigator 2.0's raw API is deliberately low-level and verbose. The Flutter team designed it as a foundation for routing libraries, not as an end-user API. The complexity is real:

  • You must manually synchronize app state ↔ URL ↔ page list — three representations of the same truth
  • Pop handling requires coordination between PopScope, onDidRemovePage, BackButtonDispatcher, and popRoute()
  • No built-in support for path parameters, query parameters, nested routing, or authentication guards
  • Error-prone page key management

The ecosystem's response:

Package Approach Best For
go_router Declarative URL-based routing; official Flutter team recommendation Most apps; web + mobile
auto_route Code-generation; strongly typed routes Large teams; refactoring safety
beamer Beam-based navigation state Complex nested routing
Raw Nav 2.0 Manual implementation of all 4 pieces Custom routing packages, extreme control
Deep Dive — Staff / L6
What is the data flow when the user types a new URL in the browser address bar in a Flutter Web app using Navigator 2.0?
1. The browser fires a popstate event (or the initial load triggers the URL). The Flutter Web embedder picks this up via BrowserHistory.
2. PlatformRouteInformationProvider receives the new RouteInformation(uri) from the platform channel.
3. Router calls parser.parseRouteInformation(info) → your typed app state T.
4. Router calls delegate.setNewRoutePath(T) → your delegate updates its internal state and calls notifyListeners().
5. Router calls delegate.build(context) → returns Navigator(pages: [...]) with the correct page stack.
6. Navigator diffs old vs new pages list, creating/destroying routes as needed.
7. The URL bar stays as-is (it was the source of the change). Subsequent in-app navigations call notifyListeners() → currentConfiguration → restoreRouteInformation() → URL bar updated.
Walk through all 7 steps sequentially. Interviewers want to see you understand the full round-trip, not just one direction.
