Staff / L6 Core Concept
The distinction between deactivate and dispose — and how GlobalKey exploits the gap between them — is a canonical Staff-level question. Most engineers know the happy path; interviewers probe the edges: what happens during tree moves, what triggers didChangeDependencies vs didUpdateWidget, and why calling setState() during build() is a framework error.

1. Element Lifecycle Methods

Every Element goes through a well-defined lifecycle. Understanding each transition is essential for reasoning about widget behavior, especially around state initialization, dependency changes, and cleanup.

  Widget inflated
       │
       ▼
  createElement()
       │
       ▼
  ┌─────────────┐
  │   INACTIVE  │ ← initial state before mount
  └──────┬──────┘
         │ mount(parent, slot)
         ▼
  ┌─────────────┐
  │    ACTIVE   │ ←─────────────────────────────────┐
  └──────┬──────┘                                   │
         │                                          │ reactivate()
         │ update(newWidget) ← parent rebuilt       │ [GlobalKey move]
         │    [stays ACTIVE]                        │
         │                                          │
         │ deactivate() ← removed from tree         │
         ▼                                          │
  ┌─────────────┐                                   │
  │ DEACTIVATED │ ──── reinserted via GlobalKey ────┘
  └──────┬──────┘
         │  not reinserted in same frame
         ▼
  ┌─────────────┐
  │  UNMOUNTED  │  (dispose() called on State)
  └─────────────┘

mount(parent, newSlot)

Called exactly once — when the element is first inserted into the element tree. For a StatefulElement, this is where createState() and initState() are called.

// StatefulElement.mount() — simplified from framework source
@override
void mount(Element? parent, Object? newSlot) {
  super.mount(parent, newSlot);
  // Create the State object — calls StatefulWidget.createState()
  _state = (widget as StatefulWidget).createState();
  // Link element ↔ state
  _state._element = this;
  _state._widget = widget as StatefulWidget;
  // First lifecycle callback on State
  _state.initState();
  // Notify about any InheritedWidget dependencies encountered during initState
  _state.didChangeDependencies();
  _dirty = true; // mark for first build
}

update(newWidget)

Called when the parent rebuilds and provides a new widget where Widget.canUpdate returned true. The element persists; only its widget reference changes.

// StatefulElement.update() — the didUpdateWidget pathway
@override
void update(StatefulWidget newWidget) {
  super.update(newWidget);
  final StatefulWidget oldWidget = _state._widget!;
  _dirty = true;
  // Update State's widget reference to the new widget instance
  _state._widget = widget as StatefulWidget;
  // Notify State about the widget change
  _state.didUpdateWidget(oldWidget);
  rebuild(); // triggers performRebuild → state.build()
}

deactivate()

Temporary removal. Called when an element is removed from the active tree but not yet unmounted. The element is placed in the BuildOwner._inactiveElements set and given a chance to be reinserted (via GlobalKey) in the same frame before unmount() is called.

// Element.deactivate() — base implementation
@mustCallSuper
void deactivate() {
  // Unregister from all InheritedElement dependencies
  if (_dependencies != null && _dependencies!.isNotEmpty) {
    for (final InheritedElement dependency in _dependencies!) {
      dependency._dependents.remove(this);
    }
  }
  _inheritedWidgets = null;
  // State.deactivate() is called here for StatefulElement
}

// State.deactivate() — override point for subclasses
@protected
void deactivate() {
  // Called when element is removed from tree.
  // Rarely overridden — most cleanup goes in dispose().
  // But: listeners added to objects obtained from BuildContext should be
  // removed here (not dispose) if they might fire after unmount.
}

unmount()

Permanent removal. Called after deactivate() when the element was not reactivated. This is where State.dispose() is called. After unmount, the element and its State object should not be used.

// StatefulElement.unmount()
@override
void unmount() {
  super.unmount();
  _state.dispose();     // State cleanup — release resources, cancel subscriptions
  _state._element = null; // Break the element ↔ state link
  _state._widget = null;
}
Critical Distinction
deactivate() may be called WITHOUT dispose() ever following (if the element is reactivated via GlobalKey). dispose() is ONLY called from unmount(). This means: cleanup that must always happen (cancel timers, close streams) belongs in dispose(). Cleanup that must happen whenever the widget leaves the visible tree (e.g., pause a video) belongs in deactivate().

2. StatefulElement Internals

Understanding how StatefulElement bridges the widget and state is essential for reasoning about setState(), hot reload, and the relationship between widget configuration and mutable state.

class StatefulElement extends ComponentElement {
  StatefulElement(StatefulWidget widget)
      : _state = widget.createState(),
        super(widget) {
    // Link state ↔ element immediately (before mount)
    _state._element = this;
    _state._widget = widget;
  }

  State<StatefulWidget> get state => _state;
  late State<StatefulWidget> _state;

  @override
  Widget build() => state.build(this);  // 'this' IS the BuildContext

  // Called during hot reload — triggers reassemble() on State
  @override
  void reassemble() {
    state.reassemble();
    super.reassemble();
  }
}

The State–Element Link

The bidirectional link between State and StatefulElement is how setState() reaches the framework:

abstract class State<T extends StatefulWidget> {
  // The element that owns this state.
  // Set during mount, cleared during unmount.
  StatefulElement? _element;

  // The current widget (may change on didUpdateWidget calls)
  T get widget => _widget!;
  T? _widget;

  // The BuildContext for this State (same object as _element)
  BuildContext get context {
    assert(() {
      if (_element == null) {
        throw FlutterError(
          'This widget has been unmounted, so the State no longer has a context...'
        );
      }
      return true;
    }());
    return _element!;
  }

  bool get mounted => _element != null;
}

3. State Lifecycle — Deep Dive

  createState()
       │
       ▼
  initState()
  ┌────────────────────────────────────────────────────────────────┐
  │ • Called ONCE on first mount                                   │
  │ • Safe to call setState() here (but no-op — first build       │
  │   happens right after)                                         │
  │ • DO: set up controllers, subscriptions, read widget.*        │
  │ • DON'T: call context.dependOnInheritedWidgetOfExactType      │
  │   (tree not fully built yet — use didChangeDependencies)      │
  └────────────────────────────────────────────────────────────────┘
       │
       ▼
  didChangeDependencies()
  ┌────────────────────────────────────────────────────────────────┐
  │ • Called after initState() always                              │
  │ • Called whenever an InheritedWidget this State depends on    │
  │   changes (e.g., Theme, MediaQuery, Localizations)            │
  │ • Safe to call context.dependOnInheritedWidgetOfExactType     │
  │ • DO: re-fetch locale-dependent resources here                │
  └────────────────────────────────────────────────────────────────┘
       │
       ▼
  build(context)  ←─────────────────────────────────────────────┐
  ┌────────────────────────────────────────────────────────────── │
  │ • Called every time state is dirty                            │
  │ • Must be pure: same inputs → same output                     │
  │ • DON'T call setState() here → framework error               │
  │ • DON'T perform side effects                                  │
  └──────────────────────────────────────────────────────────────┘
       │                                                          │
       │  parent rebuilds with new widget config                  │
       ▼                                                          │
  didUpdateWidget(T oldWidget)                                    │
  ┌────────────────────────────────────────────────────────────── │
  │ • Called when parent provides new widget (canUpdate = true)   │
  │ • oldWidget: the previous widget instance                     │
  │ • widget: the new widget instance (already updated)          │
  │ • Common use: re-init controllers when key props change       │
  │ • setState() here queues a rebuild (already rebuilding)       │
  └─────────────────────────────────────────────────────────────► ┘
       │
       │  removed from tree
       ▼
  deactivate()
       │
       │  reinserted via GlobalKey → back to build()
       │  OR
       ▼
  dispose()
  ┌────────────────────────────────────────────────────────────────┐
  │ • Called ONCE on permanent removal                             │
  │ • DO: dispose controllers, cancel timers, close streams       │
  │ • DON'T call setState() — element is unmounted               │
  │ • Context is no longer valid after this                       │
  └────────────────────────────────────────────────────────────────┘

initState — Common Patterns and Pitfalls

class _MyWidgetState extends State<MyWidget> {
  late AnimationController _controller;
  late StreamSubscription<int> _subscription;

  @override
  void initState() {
    super.initState(); // ALWAYS call super first

    // GOOD: widget properties are available
    _controller = AnimationController(
      vsync: this,
      duration: widget.animationDuration, // widget is accessible
    );

    // GOOD: subscribing to streams from widget
    _subscription = widget.stream.listen((value) {
      setState(() { /* update state */ });
    });

    // BAD: DO NOT access InheritedWidgets here
    // This will fail because the element is not yet
    // attached to the InheritedWidget subtree.
    // final theme = Theme.of(context); // WRONG in initState
  }

  @override
  void didChangeDependencies() {
    super.didChangeDependencies();
    // GOOD: safe to access InheritedWidgets here
    final theme = Theme.of(context); // fine
    final locale = Localizations.localeOf(context); // fine
    // Re-fetch locale-dependent data, update theme-dependent controllers, etc.
  }

  @override
  void didUpdateWidget(MyWidget oldWidget) {
    super.didUpdateWidget(oldWidget); // ALWAYS call super first
    // React to widget property changes
    if (widget.animationDuration != oldWidget.animationDuration) {
      _controller.duration = widget.animationDuration;
    }
    // Re-subscribe if stream changed
    if (widget.stream != oldWidget.stream) {
      _subscription.cancel();
      _subscription = widget.stream.listen((value) {
        setState(() { /* update state */ });
      });
    }
  }

  @override
  void dispose() {
    _controller.dispose();    // release AnimationController
    _subscription.cancel();   // cancel stream subscription
    super.dispose();          // ALWAYS call super last
  }
}

4. setState() Internals — Full Execution Path

Understanding every step of setState() from user call to frame schedule is a definitive L6 question. Here is the complete path:

// Step 1: User calls setState()
setState(() {
  _count++;
});

// Step 2: State.setState() implementation
@protected
void setState(VoidCallback fn) {
  assert(() {
    if (_element == null) {
      throw FlutterError('setState() called after dispose()');
    }
    if (_debugLifecycleState == _StateLifecycle.defunct) {
      throw FlutterError('setState() called after dispose()');
    }
    if (_debugLifecycleState == _StateLifecycle.created && !mounted) {
      throw FlutterError('setState() called in constructor');
    }
    return true;
  }());
  // Execute the mutation callback
  fn();
  // Step 3: Mark the element dirty
  _element!.markNeedsBuild();
}

// Step 4: Element.markNeedsBuild()
void markNeedsBuild() {
  if (_lifecycleState != _ElementLifecycle.active) return;
  if (dirty) return; // already marked — no-op
  _dirty = true;
  // Step 5: Register with BuildOwner
  owner!.scheduleBuildFor(this);
}

// Step 6: BuildOwner.scheduleBuildFor()
void scheduleBuildFor(Element element) {
  assert(element.owner == this);
  if (!_scheduledFlushDirtyElements && onBuildScheduled != null) {
    _scheduledFlushDirtyElements = true;
    onBuildScheduled!(); // → WidgetsBinding._handleBuildScheduled()
  }
  _dirtyElements.add(element);
  element._inDirtyList = true;
}

// Step 7: WidgetsBinding._handleBuildScheduled()
void _handleBuildScheduled() {
  // Request a new frame if not already scheduled
  ensureVisualUpdate(); // → SchedulerBinding.scheduleFrame()
}

// Step 8: On next vsync:
// SchedulerBinding.handleDrawFrame()
//   → WidgetsBinding.drawFrame()
//   → BuildOwner.buildScope()
//   → element.rebuild() [our dirty element]
//   → StatefulElement.performRebuild()
//   → state.build(this) [our build() method called]
Interview Point
Why is calling setState() during build() an error? During buildScope(), the framework is actively iterating _dirtyElements. Calling setState() inside build() would add to _dirtyElements mid-iteration, causing the framework's depth-sorted iteration to become invalid and potentially looping forever. The framework detects this with an assertion: _debugBuilding flag is set to true during buildScope(), and markNeedsBuild() asserts it is false.

_dirty Flag and the Double-Build Guard

// Element.rebuild() — the dirty check guard
void rebuild() {
  if (_lifecycleState != _ElementLifecycle.active || !_dirty) {
    return; // Skip if not dirty — prevents redundant builds
  }
  performRebuild(); // calls build() which clears _dirty
}

// After performRebuild():
// _dirty is set to false, element removed from _dirtyElements list.
// If setState() is called AGAIN from a child's build() (which it shouldn't),
// this element could be re-added to _dirtyElements — caught by assertion.

5. GlobalKey — Cross-Tree Element Identity

GlobalKey is Flutter's mechanism for maintaining element identity regardless of position in the widget tree. It enables scenarios impossible without it: moving a widget subtree across tree locations while preserving all state.

// GlobalKey registration: WidgetsBinding maintains a global registry
// Map<GlobalKey, Element>  ← O(1) lookup

abstract class GlobalKey<T extends State<StatefulWidget>> extends Key {
  // Factory constructor — creates a LabeledGlobalKey (debug label) or
  // a _GlobalKey (production)
  factory GlobalKey({ String? debugLabel }) => LabeledGlobalKey<T>(debugLabel);

  // Access the current element holding this key
  BuildContext? get currentContext => _currentElement;

  // Access the current widget
  Widget? get currentWidget => _currentElement?.widget;

  // Access the current State (only if element is StatefulElement)
  T? get currentState {
    final Element? element = _currentElement;
    if (element is StatefulElement) {
      final StatefulElement statefulElement = element;
      State state = statefulElement.state;
      if (state is T) return state;
    }
    return null;
  }
}

GlobalKey Element Move Mechanics

// Scenario: Widget with GlobalKey moves from position A to position B
// in the same build frame.

// Frame N: MyWidget(key: globalKey) is at position A in the tree
// Frame N+1: Parent rebuilds, MyWidget(key: globalKey) is now at position B

// What happens:
// 1. At position A: canUpdate fails (new widget at A is different type or no widget)
//    → Element at A is deactivated (NOT unmounted yet)
//    → Element goes into BuildOwner._inactiveElements

// 2. At position B: inflateWidget is called with MyWidget(key: globalKey)
//    → Framework checks _globalKeyRegistry[globalKey]
//    → Finds the deactivated element!
//    → Instead of creating a new element, REACTIVATES the existing one
//    → element.activate() is called (opposite of deactivate)
//    → State is fully preserved — no initState, no dispose

// 3. BuildOwner.finalizeTree():
//    → Any elements remaining in _inactiveElements after the build are unmounted

// Practical example: FormKey allows validation from outside the Form widget
final _formKey = GlobalKey<FormState>();

class MyForm extends StatelessWidget {
  @override
  Widget build(BuildContext context) {
    return Form(
      key: _formKey,  // GlobalKey attaches to the FormState
      child: Column(children: [
        TextFormField(validator: (v) => v!.isEmpty ? 'Required' : null),
        ElevatedButton(
          onPressed: () {
            // Access FormState from ANYWHERE in the app
            if (_formKey.currentState!.validate()) {
              _formKey.currentState!.save();
            }
          },
          child: const Text('Submit'),
        ),
      ]),
    );
  }
}

GlobalKey Performance Cost

GlobalKey carries real overhead. Understanding the cost is what separates a Senior from a Staff answer:

Cost Explanation
Global registry write on every mount/unmount Every GlobalKey element registers/deregisters from the app-wide Map<GlobalKey, Element>
Subtree invalidation on move When a GlobalKey element moves, the framework cannot assume parent subtree caching is valid — it must re-traverse
BuildOwner._inactiveElements bookkeeping Every deactivated element is tracked until the end of the frame — adds overhead to finalizeTree()
Breaks subtree equality optimizations The framework's optimization that skips subtrees with identical widget references cannot be applied when GlobalKeys are present
Anti-Pattern
Creating a GlobalKey inside a build() method is a particularly harmful mistake. A new GlobalKey object is created on every build, which means the framework cannot match the old key to the new key, forcing a full deactivate-and-reinflate cycle on every rebuild. Always create GlobalKeys as instance variables or static fields.

6. Widget Rebuilds vs Element Reuse

The distinction between "widget rebuild" and "element reuse" is fundamental. When developers say "my widget rebuilt," they usually mean the build() method was called — but the element was reused (updated), not recreated.

// Widget rebuild scope: only dirty elements rebuild, not entire tree
class Parent extends StatefulWidget {
  @override
  State<Parent> createState() => _ParentState();
}

class _ParentState extends State<Parent> {
  int _parentValue = 0;

  @override
  Widget build(BuildContext context) {
    // This is called when parent is dirty.
    // BUT: the ExpensiveChild below will only rebuild if:
    // 1. Its widget config changed (new widget instance with different props)
    // 2. OR its own state is dirty
    // 3. OR an InheritedWidget it depends on changed
    return Column(children: [
      Text('Parent: $_parentValue'),
      const ExpensiveChild(), // const → same instance → element.update() is no-op
    ]);
  }
}

// const widgets: the key insight
// const Text('Hello') is a compile-time constant.
// Every time this expression is evaluated, it returns the SAME object.
// When Parent rebuilds:
//   updateChild(existingTextElement, const Text('Hello'), slot)
//   → Widget.canUpdate: same runtimeType ✓, same key (null) ✓ → update called
//   → element.update(newWidget)
//   → In StatelessElement.update():
//       if (identical(oldWidget, widget)) {
//         // Same object! No need to rebuild child.
//         return;
//       }
//   → Build is SKIPPED entirely.

// Non-const equivalent:
// Text('Hello') creates a NEW object each time (even with same string).
// identical(oldWidget, newWidget) → false
// → child rebuild happens
// → RenderParagraph.text is updated (even though value is same string)
// → Potential unnecessary layout/paint invalidation
Staff-Level Summary
The element lifecycle is a state machine with five states: initial → active → deactivated → (reactivated OR unmounted). mount() transitions to active (triggers initState + didChangeDependencies). deactivate() transitions to deactivated (temporary hold). reactivate() brings it back to active (GlobalKey move — no initState or dispose). unmount() triggers dispose() and is permanent. setState() → markNeedsBuild() → BuildOwner.scheduleBuildFor() → ensureVisualUpdate() → next vsync → buildScope() → performRebuild() → state.build(). Every step is synchronous within a single frame tick, with the vsync as the only asynchronous boundary.
