Purpose and Architecture

EventChannel is the Flutter mechanism for continuous, asynchronous data streaming from the native platform to Dart. While MethodChannel models a single request-response call, EventChannel models an ongoing subscription — the native side pushes events whenever they occur, and Dart receives them as a Stream.

Typical use cases:

  • Location updates (GPS coordinates as the device moves)
  • BLE (Bluetooth Low Energy) scan results and characteristic notifications
  • Accelerometer / gyroscope sensor data
  • Battery state changes
  • Network connectivity changes
  • Background audio position updates
  • NFC tag scan events

The StreamHandler Interface (Native Side)

On the native side, you implement EventChannel.StreamHandler. It has exactly two methods:

Method When Called Your Responsibility
onListen(arguments, eventSink) First Dart listener subscribes (stream.listen(...)) Start the native data source; retain the EventSink to push events
onCancel(arguments) Last Dart listener cancels (subscription.cancel()) or stream closes Stop the native data source; release the EventSink reference; unregister callbacks

Kotlin Implementation — Location Updates

// LocationPlugin.kt
import io.flutter.plugin.common.EventChannel
import android.location.Location
import android.location.LocationListener
import android.location.LocationManager

class LocationStreamHandler(
    private val context: Context
) : EventChannel.StreamHandler {

    private var locationManager: LocationManager? = null
    private var locationListener: LocationListener? = null
    private var eventSink: EventChannel.EventSink? = null

    override fun onListen(arguments: Any?, sink: EventChannel.EventSink) {
        // Store sink — we'll use it to push events from native callbacks
        eventSink = sink

        locationManager = context.getSystemService(Context.LOCATION_SERVICE)
            as LocationManager

        locationListener = LocationListener { location ->
            // Called on the main thread by LocationManager
            // Push a map to Dart — will be decoded as Map
            eventSink?.success(mapOf(
                "latitude" to location.latitude,
                "longitude" to location.longitude,
                "accuracy" to location.accuracy,
                "timestamp" to location.time,
            ))
        }

        try {
            locationManager?.requestLocationUpdates(
                LocationManager.GPS_PROVIDER,
                1000L,   // min time interval ms
                1.0f,    // min distance meters
                locationListener!!
            )
        } catch (e: SecurityException) {
            // Permissions not granted — report error to Dart
            sink.error(
                "PERMISSION_DENIED",
                "Location permission not granted",
                e.localizedMessage
            )
        }
    }

    override fun onCancel(arguments: Any?) {
        // CRITICAL: stop native callbacks and release sink reference
        locationListener?.let { locationManager?.removeUpdates(it) }
        locationManager = null
        locationListener = null
        eventSink = null  // prevent memory leak
    }
}

// Registration in FlutterPlugin:
class LocationPlugin : FlutterPlugin {
    private lateinit var eventChannel: EventChannel

    override fun onAttachedToEngine(binding: FlutterPlugin.FlutterPluginBinding) {
        eventChannel = EventChannel(
            binding.binaryMessenger,
            "com.myapp/location"
        )
        eventChannel.setStreamHandler(
            LocationStreamHandler(binding.applicationContext)
        )
    }

    override fun onDetachedFromEngine(binding: FlutterPlugin.FlutterPluginBinding) {
        eventChannel.setStreamHandler(null)
    }
}

EventSink Methods

Method Dart Stream Effect Use When
sink.success(data) Emits a data event to stream listeners Normal event (location, sensor reading, BLE advertisement)
sink.error(code, message, details) Emits an error event; stream remains open for more events Recoverable error (GPS signal lost temporarily, BLE characteristic read failed)
sink.endOfStream() Closes the stream — no more events possible Native source is permanently done (file scan complete, finite data sequence)
Warning
Calling sink.success() or sink.error() after onCancel() has been called will crash the app (null pointer on the native sink implementation). Always null-check or null-assign the sink in onCancel() and guard every eventSink?.success() call.

Dart Side: receiveBroadcastStream()

// dart/location_service.dart
import 'package:flutter/services.dart';

class LocationService {
  static const _channel = EventChannel('com.myapp/location');

  // receiveBroadcastStream() returns a broadcast Stream
  // It calls onListen when the first subscriber listens,
  // and onCancel when the last subscriber cancels.
  Stream<LocationData> get locationStream =>
      _channel
          .receiveBroadcastStream()
          .map((event) => LocationData.fromMap(Map<String, dynamic>.from(event)));

  // NOTE: receiveBroadcastStream() creates a new subscription each call.
  // Cache the stream if multiple widgets need to share it:
  Stream<LocationData>? _cachedStream;

  Stream<LocationData> get stream {
    _cachedStream ??= _channel
        .receiveBroadcastStream()
        .map((event) => LocationData.fromMap(Map<String, dynamic>.from(event)))
        .asBroadcastStream(); // allow multiple Dart listeners
    return _cachedStream!;
  }
}

class LocationData {
  final double latitude;
  final double longitude;
  final double accuracy;
  final DateTime timestamp;

  LocationData({
    required this.latitude,
    required this.longitude,
    required this.accuracy,
    required this.timestamp,
  });

  factory LocationData.fromMap(Map<String, dynamic> map) => LocationData(
    latitude: (map['latitude'] as num).toDouble(),
    longitude: (map['longitude'] as num).toDouble(),
    accuracy: (map['accuracy'] as num).toDouble(),
    timestamp: DateTime.fromMillisecondsSinceEpoch(map['timestamp'] as int),
  );
}

Subscription Lifecycle in a Widget

class _LocationMapState extends State<LocationMap> {
  final _locationService = LocationService();
  StreamSubscription<LocationData>? _subscription;
  LocationData? _currentLocation;
  String? _error;

  @override
  void initState() {
    super.initState();
    _subscription = _locationService.locationStream.listen(
      (location) {
        if (mounted) {
          setState(() {
            _currentLocation = location;
            _error = null;
          });
        }
      },
      onError: (error) {
        if (mounted) {
          setState(() => _error = error.toString());
        }
      },
      onDone: () {
        // Stream closed by native side (endOfStream called)
        if (mounted) setState(() => _error = 'Location stream ended');
      },
    );
    // Subscribing here triggers onListen() on the native StreamHandler
  }

  @override
  void dispose() {
    _subscription?.cancel(); // triggers onCancel() on the native StreamHandler
    super.dispose();
  }

  @override
  Widget build(BuildContext context) {
    if (_error != null) return ErrorView(message: _error!);
    if (_currentLocation == null) return const LoadingView();
    return MapView(location: _currentLocation!);
  }
}

Handling Backpressure

Backpressure occurs when the native side emits events faster than the Dart side can process them. The EventChannel has no built-in buffering — each sink.success() call posts a message to the binary messenger immediately.

Strategy 1: Drop Events (Sensor Data)

// Kotlin: only emit if Dart is "ready" (tracked via a flag)
class SensorStreamHandler : EventChannel.StreamHandler {
    private var eventSink: EventChannel.EventSink? = null
    private var isProcessing = false

    // Sensor listener fires at 100Hz
    private val sensorListener = SensorEventListener { event ->
        if (!isProcessing) {
            isProcessing = true
            eventSink?.success(event.values.toList())
            // Reset flag after a short delay to throttle to ~30Hz
            Handler(Looper.getMainLooper()).postDelayed({ isProcessing = false }, 33)
        }
        // else: drop this event — Dart is still processing the previous one
    }
}

Strategy 2: Buffer with Conflation (Latest-Value Wins)

// Dart side: use conflate() to only process the latest value
_channel
    .receiveBroadcastStream()
    .cast<Map>()
    .map((m) => SensorReading.fromMap(Map<String, dynamic>.from(m)))
    .asyncMap((reading) async {
      // If this async processing takes longer than the emission interval,
      // conflate() drops intermediate values
      await processReading(reading);
      return reading;
    })

EventChannel vs MethodChannel Polling

Aspect EventChannel MethodChannel Polling
Latency Event-driven: immediate on native event Up to poll interval delay
CPU cost (idle) Near zero — callback-based Timer fires repeatedly even with no data
Battery impact Low — native source sleeps between events Higher — periodic wakeups
Code complexity Higher (StreamHandler, lifecycle) Lower (simple method calls)
Best for Continuous high-frequency data (GPS, BLE, sensors) Infrequent status checks (battery level on tap)
Missed events None (push-based) Possible between polls
Staff Interview Signal
Know the full lifecycle: Dart listen() → native onListen() → sink.success() → Dart stream event → Dart cancel() → native onCancel(). Know what happens when: (1) the widget is disposed while the native source is still active — onCancel is called, the native source should stop. (2) the native side calls sink.success() after onCancel() — crash if you don't null-guard. (3) multiple Dart widgets subscribe — by default EventChannel only supports one active listener; use asBroadcastStream() on the Dart side and a shared service layer so only one native subscription exists.
