Section 08 — Platform Integration
Pigeon: Eliminating Stringly-Typed Platform APIs

Pigeon generates type-safe Dart/Kotlin/Swift bindings from a single Dart input file, making platform channel contracts compile-time verified instead of runtime discovered.

1. Why Pigeon Exists

Manual MethodChannel is the foundation of Flutter's platform integration, but it has a fundamental problem: the entire contract between Dart and native code is expressed as strings and dynamic values.

// Manual MethodChannel — everything is stringly typed
final channel = MethodChannel('com.example/camera');

// Dart call site — method name is a string literal
// args is a Map<String, dynamic> — no compile-time checking
final result = await channel.invokeMethod<String>(
  'capturePhoto',           // typo here? only fails at runtime
  {'quality': 95, 'flash': true},  // wrong key? runtime crash
);

Problems with this approach:

  • No compile-time type checking — a typo in the method name or wrong argument type only surfaces at runtime, possibly on a device test
  • No contract between platforms — the Dart caller and Kotlin/Swift handler can silently diverge; adding a new required argument on native doesn't break the Dart build
  • No null safety — return types are dynamic or T? with manual casting
  • Duplication — data class definitions must be maintained separately in Dart, Kotlin, and Swift with manual serialization code in each

Pigeon solves all of these by generating the entire boilerplate from a single Dart schema file, with type safety enforced at code generation time.

2. Pigeon Architecture Overview

Pigeon is a code generation tool. You write a Pigeon input file — a plain Dart file annotated with Pigeon-specific annotations — and run the generator. It produces:

  • Dart file — the API classes and codec used by Flutter code
  • Kotlin file — handler interfaces and message codec for Android
  • Swift file — handler protocols and message codec for iOS/macOS
  • (Optionally) Java / Objective-C files for older targets

All generated files share the same wire format — the generated codec handles serialization/deserialization identically on both sides, so you never write manual encoding code.

Tip
The Pigeon input file lives in your repo (e.g., pigeons/api.dart) but is not part of the Flutter build — it is only read by the Pigeon generator. The generated output files are what get compiled.

3. Pigeon Input File — Structure

3.1 @ConfigurePigeon

The file starts with a @ConfigurePigeon annotation on the main() function, specifying where to write each output file:

// pigeons/camera_api.dart
import 'package:pigeon/pigeon.dart';

@ConfigurePigeon(PigeonOptions(
  dartOut: 'lib/src/generated/camera_api.g.dart',
  dartOptions: DartOptions(),
  kotlinOut: 'android/app/src/main/kotlin/com/example/CameraApi.g.kt',
  kotlinOptions: KotlinOptions(package: 'com.example'),
  swiftOut: 'ios/Runner/CameraApi.g.swift',
  swiftOptions: SwiftOptions(),
  dartPackageName: 'my_camera_plugin',
))
void configurePigeon(PigeonOptions opts) {}

3.2 Data Classes

Plain Dart classes annotated with nothing (just regular Dart with Pigeon-compatible types) become data transfer objects. Pigeon generates codec support for them automatically:

class PhotoConfig {
  PhotoConfig({
    required this.quality,
    required this.flashEnabled,
    this.targetWidth,
    this.targetHeight,
  });

  final int quality;          // 0–100
  final bool flashEnabled;
  final int? targetWidth;     // nullable = optional
  final int? targetHeight;
}

class PhotoResult {
  PhotoResult({required this.filePath, required this.fileSizeBytes});

  final String filePath;
  final int fileSizeBytes;
}

3.3 @HostApi — Dart Calls Native

@HostApi marks an abstract class where Dart is the caller and native (Kotlin/Swift) is the implementer. This is the most common direction:

@HostApi()
abstract class CameraHostApi {
  /// Capture a photo with the given configuration.
  /// Returns the photo result or throws PlatformException on failure.
  @async
  PhotoResult capturePhoto(PhotoConfig config);

  /// Returns the list of available camera IDs.
  List<String> getAvailableCameras();

  /// Set zoom level. Void return = fire and forget.
  void setZoom(double zoomLevel);
}

Key points:

  • All methods returning non-void are asynchronous on the Dart side by default (Pigeon wraps them in futures)
  • @async is required for methods that make async calls on the native side
  • Return types and parameter types must be Pigeon-compatible (primitives, nullable primitives, data classes, enums, List, Map)

3.4 @FlutterApi — Native Calls Dart

@FlutterApi marks an abstract class where native is the caller and Dart is the implementer. Used for callbacks from native back to Flutter:

@FlutterApi()
abstract class CameraFlutterApi {
  /// Called by native when a new frame is available from the camera preview.
  void onFrameAvailable(int frameTimestampMs);

  /// Called when the camera encounters a hardware error.
  void onCameraError(String errorMessage);
}

4. Generated Output — What Pigeon Produces

4.1 Generated Dart Side

// lib/src/generated/camera_api.g.dart (generated — do not edit)

// Data class with codec support
class PhotoConfig {
  PhotoConfig({
    required this.quality,
    required this.flashEnabled,
    this.targetWidth,
    this.targetHeight,
  });

  final int quality;
  final bool flashEnabled;
  final int? targetWidth;
  final int? targetHeight;

  // Generated encode/decode for the custom codec
  Object encode() => <Object?>[quality, flashEnabled, targetWidth, targetHeight];

  static PhotoConfig decode(Object result) {
    result as List<Object?>;
    return PhotoConfig(
      quality: result[0]! as int,
      flashEnabled: result[1]! as bool,
      targetWidth: result[2] as int?,
      targetHeight: result[3] as int?,
    );
  }
}

// Generated API class — Dart calls these methods
class CameraHostApi {
  CameraHostApi({BinaryMessenger? binaryMessenger, String messageChannelSuffix = ''})
      : _binaryMessenger = binaryMessenger,
        _messageChannelSuffix = messageChannelSuffix.isNotEmpty ? '.$messageChannelSuffix' : '';

  final BinaryMessenger? _binaryMessenger;
  final String _messageChannelSuffix;

  Future<PhotoResult> capturePhoto(PhotoConfig config) async {
    const String __pigeon_channelName =
        'dev.flutter.pigeon.my_camera_plugin.CameraHostApi.capturePhoto';
    final BasicMessageChannel<Object?> __pigeon_channel =
        BasicMessageChannel<Object?>(__pigeon_channelName, _PigeonCodec(),
            binaryMessenger: _binaryMessenger);
    final List<Object?>? __pigeon_replyList =
        await __pigeon_channel.send(<Object?>[config.encode()]) as List<Object?>?;
    if (__pigeon_replyList == null) {
      throw _createConnectionError(__pigeon_channelName);
    } else if (__pigeon_replyList.length > 1) {
      throw PlatformException(
        code: __pigeon_replyList[0]! as String,
        message: __pigeon_replyList[1] as String?,
        details: __pigeon_replyList[2],
      );
    } else if (__pigeon_replyList[0] == null) {
      throw PlatformException(code: 'null-error', message: 'Host returned null');
    } else {
      return PhotoResult.decode(__pigeon_replyList[0]!);
    }
  }

  // ... other methods generated similarly
}

4.2 Generated Kotlin Side

// android/.../CameraApi.g.kt (generated — do not edit)

// Data class
data class PhotoConfig(
  val quality: Long,
  val flashEnabled: Boolean,
  val targetWidth: Long? = null,
  val targetHeight: Long? = null
) {
  companion object {
    fun fromList(pigeonVar_list: List<Any?>): PhotoConfig {
      val quality = pigeonVar_list[0] as Long
      val flashEnabled = pigeonVar_list[1] as Boolean
      val targetWidth = pigeonVar_list[2] as Long?
      val targetHeight = pigeonVar_list[3] as Long?
      return PhotoConfig(quality, flashEnabled, targetWidth, targetHeight)
    }
  }
  fun toList(): List<Any?> = listOf(quality, flashEnabled, targetWidth, targetHeight)
}

// Handler interface — YOUR code implements this
interface CameraHostApi {
  fun capturePhoto(config: PhotoConfig, callback: (Result<PhotoResult>) -> Unit)
  fun getAvailableCameras(): List<String>
  fun setZoom(zoomLevel: Double)

  companion object {
    // Registration — call this in your FlutterPlugin.onAttachedToEngine()
    fun setUp(binaryMessenger: BinaryMessenger, api: CameraHostApi?) {
      run {
        val channel = BasicMessageChannel<Any?>(
          binaryMessenger,
          "dev.flutter.pigeon.my_camera_plugin.CameraHostApi.capturePhoto",
          _PigeonCodec()
        )
        if (api != null) {
          channel.setMessageHandler { message, reply ->
            val args = message as List<Any?>
            val configArg = PhotoConfig.fromList(args[0] as List<Any?>)
            api.capturePhoto(configArg) { result ->
              reply.reply(result.fold(
                onSuccess = { listOf(it.toList()) },
                onFailure = { listOf(it.message) }
              ))
            }
          }
        } else {
          channel.setMessageHandler(null)
        }
      }
      // ... other methods registered similarly
    }
  }
}

4.3 Your Plugin Implementation (Kotlin)

// Your code — implements the generated interface
class CameraPlugin : FlutterPlugin, CameraHostApi {

  override fun onAttachedToEngine(binding: FlutterPlugin.FlutterPluginBinding) {
    // Register this implementation with Pigeon's generated setUp()
    CameraHostApi.setUp(binding.binaryMessenger, this)
  }

  override fun onDetachedFromEngine(binding: FlutterPlugin.FlutterPluginBinding) {
    CameraHostApi.setUp(binding.binaryMessenger, null) // unregister
  }

  // Implement the generated interface — fully type-safe, no string matching
  override fun capturePhoto(
    config: PhotoConfig,
    callback: (Result<PhotoResult>) -> Unit
  ) {
    // config.quality, config.flashEnabled — typed fields, no casting
    nativeCameraSDK.capture(
      quality = config.quality.toInt(),
      flash = config.flashEnabled
    ) { filePath, size ->
      callback(Result.success(PhotoResult(filePath = filePath, fileSizeBytes = size)))
    }
  }

  override fun getAvailableCameras(): List<String> {
    return nativeCameraSDK.listCameras()
  }

  override fun setZoom(zoomLevel: Double) {
    nativeCameraSDK.zoom(zoomLevel.toFloat())
  }
}

5. Wire Format — The Generated Codec

Pigeon generates a custom MessageCodec subclass (called _PigeonCodec in generated code) that extends StandardMessageCodec. The standard codec handles primitives (int, double, bool, String, Uint8List, List, Map). The Pigeon codec adds type tags for your custom data classes and enums, encoding them as lists of their fields in declaration order.

This means:

  • The field order in your data class is significant — adding a field in the middle is a breaking change
  • The codec on Dart and native must always be in sync — which they are, because they're both generated from the same input file at the same time
  • No JSON, no reflection, no dynamic dispatch — encoding is direct index-based list operations
Breaking Change Warning
Reordering fields in a Pigeon data class is a breaking wire-format change. If you deploy a new app version with reordered fields but the user still has the old native plugin installed (e.g., in a federated plugin scenario), decoding will fail. Always add new fields at the end and treat field order as part of your API contract.

6. Async Methods and Error Handling

6.1 Async Flow

When a Dart caller awaits a @HostApi method, the generated code sends the message and awaits the reply. On the native side, the generated handler passes a Result<T> callback. The native implementation calls callback(Result.success(value)) or callback(Result.failure(exception)) — even from a background thread. The generated code marshals the reply back on the platform channel's reply mechanism.

// On native — you can call the callback from any thread
override fun capturePhoto(config: PhotoConfig, callback: (Result<PhotoResult>) -> Unit) {
  backgroundExecutor.execute {
    try {
      val result = heavyNativeWork(config)
      callback(Result.success(result))  // safe to call from background thread
    } catch (e: Exception) {
      callback(Result.failure(e))
    }
  }
}

6.2 Error Handling — FlutterError

Pigeon 10+ introduces FlutterError for typed errors — richer than a generic PlatformException:

// In your pigeon input file, define error types:
class CameraError {
  CameraError({required this.code, required this.message, this.details});
  final String code;
  final String message;
  final String? details;
}

// On Dart side, a FlutterError from native becomes a PlatformException
// On native side, throw FlutterError to signal a typed error:
// (Swift) throw CameraError(code: "PERMISSION_DENIED", message: "Camera access denied")
// (Kotlin) throw FlutterError("PERMISSION_DENIED", "Camera access denied", null)

7. Enum Support

Pigeon supports Dart enums natively. They are serialized as their integer index on the wire:

// In pigeon input file:
enum FlashMode { off, auto, always, torch }

@HostApi()
abstract class CameraHostApi {
  void setFlashMode(FlashMode mode);
}

Generated Kotlin:

enum class FlashMode(val raw: Int) {
  OFF(0), AUTO(1), ALWAYS(2), TORCH(3);

  companion object {
    fun ofRaw(raw: Int): FlashMode? = values().firstOrNull { it.raw == raw }
  }
}
Warning
Adding a new enum case must be done at the end. Inserting a case in the middle shifts indices and breaks deserialization on any native binary that hasn't been updated to match.

Tip
Pigeon maps Dart's int to Long on Kotlin (not Int). This is because Dart's int is 64-bit. If your native API expects an Int, you must explicitly call .toInt() in your handler.

9. Running Pigeon

# Add pigeon as a dev dependency
# pubspec.yaml:
dev_dependencies:
  pigeon: ^22.0.0

# Run the generator
dart run pigeon --input pigeons/camera_api.dart

# Or with a build_runner integration (if using build_runner):
# flutter pub run build_runner build

It is best practice to run Pigeon as part of your CI pipeline and fail the build if the generated files are out of date (compare the generated output to the committed version).

# CI check — regenerate and compare
dart run pigeon --input pigeons/camera_api.dart
git diff --exit-code lib/src/generated/  # fails if generated files changed

10. @FlutterApi — Native to Dart Callbacks

When native needs to call Dart (e.g., delivering sensor events, SDK callbacks), implement @FlutterApi:

// Pigeon input:
@FlutterApi()
abstract class CameraFlutterApi {
  void onFrameAvailable(int frameTimestampMs);
  void onCameraError(String errorMessage);
}
// Dart — register your implementation
class _CameraFlutterApiImpl extends CameraFlutterApi {
  @override
  void onFrameAvailable(int frameTimestampMs) {
    // Handle incoming frame notification from native
    _frameController.add(frameTimestampMs);
  }

  @override
  void onCameraError(String errorMessage) {
    _errorController.add(errorMessage);
  }
}

// Register on initialization:
CameraFlutterApi.setUp(_CameraFlutterApiImpl());
// Kotlin — call Dart from native
class CameraPlugin : FlutterPlugin {
  private lateinit var flutterApi: CameraFlutterApi

  override fun onAttachedToEngine(binding: FlutterPlugin.FlutterPluginBinding) {
    flutterApi = CameraFlutterApi(binding.binaryMessenger)
    // Now you can call Dart methods from any native context:
    // flutterApi.onFrameAvailable(timestamp) { /* reply callback */ }
  }
}

// Inside your native SDK callback:
nativeSDK.setFrameListener { timestampMs ->
  // Must call on main thread for platform channels:
  mainHandler.post {
    flutterApi.onFrameAvailable(timestampMs) { result ->
      result.onFailure { Log.e("Camera", "Flutter call failed", it) }
    }
  }
}
Threading Warning
Platform channel calls (including Pigeon-generated ones) must be made on the platform's main thread. If your native SDK delivers callbacks on a background thread, you must dispatch to the main thread before calling a @FlutterApi method. On Android: Handler(Looper.getMainLooper()).post { ... }. On iOS: DispatchQueue.main.async { ... }.

Staff-Level Insight
Pigeon is the correct choice for any production plugin or multi-team codebase where the platform boundary is a real API contract — not just a convenience wrapper. The key insight is that Pigeon makes the contract explicit and versioned: changes to the pigeon input file are visible in code review, and the generated output in CI catches mismatches. At scale, the "stringly typed MethodChannel" approach becomes a maintenance liability. Pigeon also enables better testing — you can mock the generated Dart class without needing a real platform.

12. Testing Pigeon APIs

12.1 Mocking in Widget Tests

The generated Dart class accepts an optional BinaryMessenger — inject a mock messenger in tests:

// In widget test:
TestWidgetsFlutterBinding.ensureInitialized();

// Set up a mock handler for the Pigeon channel
TestDefaultBinaryMessengerBinding.instance.defaultBinaryMessenger
    .setMockMessageHandler(
  'dev.flutter.pigeon.my_camera_plugin.CameraHostApi.capturePhoto',
  (ByteData? message) async {
    // Return a mock response in Pigeon's wire format
    final codec = StandardMessageCodec();
    return codec.encodeMessage([
      // Encoded PhotoResult
      ['/tmp/photo.jpg', 102400]
    ]);
  },
);

12.2 Pigeon's Generated Mock Classes

Pigeon can generate mock implementations directly. Add mockDartClass: true in @HostApi:

@HostApi(dartHostTestHandler: 'TestCameraHostApi')
abstract class CameraHostApi { ... }

This generates a TestCameraHostApi class you can register in tests to intercept calls without real platform code.
