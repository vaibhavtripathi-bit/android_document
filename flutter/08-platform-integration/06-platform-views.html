Section 08 — Platform Integration
Embedding Native UI in Flutter

Platform views let you embed native OS UI components — maps, webviews, camera previews — directly inside a Flutter widget tree. They come with significant compositing trade-offs that every senior Flutter engineer must understand.

1. Why Platform Views Exist

Flutter's rendering engine draws every pixel itself — there is no native button, native text field, or native scroll view in Flutter's widget tree. This gives Flutter precise cross-platform control, but it means some native capabilities simply cannot be replicated:

  • Google Maps SDK — the Maps SDK renders map tiles using its own OpenGL/Metal surface; there is no Flutter equivalent
  • WebView with full JS engine — a complete browser engine (Chromium/WebKit) cannot be reimplemented in Flutter widgets
  • Native video player — platform media players (ExoPlayer, AVPlayer) provide DRM, hardware decoding, and codec support that would be impractical to replicate
  • Camera preview — live camera frames rendered directly by the platform's camera HAL; the alternative (Texture widget) works for many cases but not all camera SDK features
  • AR frameworks — ARCore/ARKit render to their own surface, not a Flutter canvas

Platform views solve this by inserting a real native view (android.view.View or UIView) into the same screen as Flutter's rendered content, with Flutter's layout system controlling its position and size.

Cost Warning
Platform views are the most expensive feature in Flutter's compositing model. Every platform view in the viewport breaks Flutter's single-raster-thread rendering model and introduces per-frame synchronization overhead. Use them only when no texture-based alternative exists.

2. Android — Two Compositing Modes

Android platform views have two fundamentally different implementation strategies. Understanding both is essential for performance-sensitive engineering decisions.

2.1 Virtual Display (Legacy — Avoid)

Mechanism: The native Android View is rendered into a VirtualDisplay — an off-screen display surface that exists entirely in software. Flutter captures the pixels from this virtual display as a GPU texture, then composites that texture into its layer tree as a TextureLayer.

How it works internally:

  1. Flutter creates a VirtualDisplay via DisplayManager.createVirtualDisplay()
  2. The native view's Window is attached to this virtual display
  3. The view renders into a Surface backed by a SurfaceTexture
  4. Flutter's raster thread samples the SurfaceTexture as an OpenGL external texture
  5. This texture is included in Flutter's layer tree as a TextureLayer

Problems:

  • Accessibility broken — the view is on a virtual display, not the real display hierarchy; Android's accessibility tree cannot find it
  • Keyboard/IME issues — the soft keyboard insets don't apply correctly to a view on a virtual display; text input within the native view often fails or shows the keyboard in the wrong position
  • Nested scrolling broken — Android's nested scroll protocol requires real view hierarchy participation; a virtual display view cannot participate
  • Motion events — touch events must be manually forwarded from Flutter to the virtual display, introducing latency and edge cases
  • One frame of latency — the texture sampling introduces an extra frame of delay between the native view updating and Flutter displaying it
Deprecated
AndroidView (which uses Virtual Display by default in older Flutter versions) is effectively deprecated for new development. Use SurfaceAndroidView or PlatformViewLink with Hybrid Composition instead.

2.2 Hybrid Composition (Modern — Recommended)

Mechanism: The native Android View is inserted directly into the real Android view hierarchy — the same FlutterView hierarchy that the OS displays. Flutter renders its content into separate SurfaceViews that are positioned above and below the native view. The Android OS compositor merges all layers at display time.

How it works internally:

  1. Flutter splits its render output into two FlutterImageView layers: one rendered below the native view, one above
  2. The native View is added to the FlutterView's view hierarchy at the correct z-position
  3. Each frame, Flutter must synchronize its two image layers with the native view's position — this requires a mutex between the raster thread and the platform thread
  4. The Android hardware compositor (SurfaceFlinger) merges the three layers: Flutter-below, native view, Flutter-above

Advantages over Virtual Display:

  • Accessibility works — the view is in the real hierarchy; Android's a11y tree sees it
  • Keyboard/IME works — standard window inset handling applies correctly
  • Nested scrolling works — the view participates in Android's real touch dispatch chain
  • No extra frame latency — native view renders at native frame rate, not sampled

Cost of Hybrid Composition:

  • Thread synchronization per frame — Flutter's raster thread must synchronize with the platform thread every frame that contains a platform view, even if the platform view hasn't changed
  • Extra memory — two additional FlutterImageView surfaces allocated
  • Potential jank on older Android — the synchronization can cause the raster thread to occasionally miss a vsync; observed more on Android 9/10 than on Android 12+

3. Android API — PlatformViewFactory and Registration

3.1 Native Side (Kotlin)

// Step 1: Implement PlatformView — wraps your native View
class NativeMapView(
  private val context: Context,
  private val viewId: Int,
  private val creationParams: Map<String?, Any?>?
) : PlatformView {

  private val mapView: MapView = MapView(context).apply {
    // Initialize the native map SDK view
    val apiKey = creationParams?.get("apiKey") as? String ?: ""
    initialize(apiKey)
  }

  override fun getView(): View = mapView

  override fun dispose() {
    mapView.onDestroy()  // clean up native resources
  }
}

// Step 2: Implement PlatformViewFactory — creates your PlatformView
class NativeMapFactory(
  private val messenger: BinaryMessenger
) : PlatformViewFactory(StandardMessageCodec.INSTANCE) {

  override fun create(context: Context, viewId: Int, args: Any?): PlatformView {
    @Suppress("UNCHECKED_CAST")
    val creationParams = args as? Map<String?, Any?>
    return NativeMapView(context, viewId, creationParams)
  }
}

// Step 3: Register in your FlutterPlugin
class MapPlugin : FlutterPlugin {
  override fun onAttachedToEngine(binding: FlutterPlugin.FlutterPluginBinding) {
    binding.platformViewRegistry.registerViewFactory(
      "com.example/native_map",   // view type identifier — must match Dart side
      NativeMapFactory(binding.binaryMessenger)
    )
  }

  override fun onDetachedFromEngine(binding: FlutterPlugin.FlutterPluginBinding) {
    // factories don't need explicit unregistration
  }
}

3.2 Dart Side — Using the Platform View

// Simple usage with AndroidView (uses Virtual Display — legacy)
Widget buildLegacy(BuildContext context) {
  return AndroidView(
    viewType: 'com.example/native_map',
    layoutDirection: TextDirection.ltr,
    creationParams: {'apiKey': 'YOUR_API_KEY'},
    creationParamsCodec: const StandardMessageCodec(),
    onPlatformViewCreated: (int viewId) {
      // Set up method channels to communicate with the specific view instance
    },
  );
}

// Modern usage with PlatformViewLink (Hybrid Composition)
Widget buildModern(BuildContext context) {
  const String viewType = 'com.example/native_map';
  final Map<String, dynamic> creationParams = {'apiKey': 'YOUR_API_KEY'};

  return PlatformViewLink(
    viewType: viewType,
    surfaceFactory: (context, controller) {
      // AndroidViewSurface handles the Hybrid Composition compositing
      return AndroidViewSurface(
        controller: controller as AndroidViewController,
        gestureRecognizers: const <Factory<OneSequenceGestureRecognizer>>{},
        hitTestBehavior: PlatformViewHitTestBehavior.opaque,
      );
    },
    onCreatePlatformView: (params) {
      // initSurface() creates the view using Hybrid Composition
      return PlatformViewsService.initSurfaceAndroidView(
        id: params.id,
        viewType: viewType,
        layoutDirection: TextDirection.ltr,
        creationParams: creationParams,
        creationParamsCodec: const StandardMessageCodec(),
        onFocus: () => params.onFocusChanged(true),
      )
        ..addOnPlatformViewCreatedListener(params.onPlatformViewCreated)
        ..create();
    },
  );
}

4. iOS — UiKitView

iOS platform views use a single mechanism: the native UIView is inserted directly into the FlutterView's UIKit hierarchy. There is no "virtual display" option on iOS — native views always participate in the real view hierarchy.

4.1 iOS Compositing Model

Flutter on iOS renders its content into a CAMetalLayer (or CAEAGLLayer with OpenGL). When a platform view is present, Flutter must render content that appears both below and above the native UIView. It does this by:

  1. Rendering the "below" Flutter content to one CAMetalLayer
  2. Inserting the native UIView above it in the UIKit hierarchy
  3. Rendering the "above" Flutter content to another overlay CALayer

This is conceptually similar to Android's Hybrid Composition, but on iOS the UIKit compositor handles merging naturally — there's less synchronization overhead because UIKit and Metal are designed to interoperate on iOS.

4.2 iOS Native Side (Swift)

// Step 1: Implement FlutterPlatformView
class NativeMapView: NSObject, FlutterPlatformView {
  private let mapView: MKMapView
  private let frame: CGRect

  init(frame: CGRect, viewIdentifier viewId: Int64, arguments args: Any?) {
    mapView = MKMapView(frame: frame)
    self.frame = frame
    super.init()

    if let params = args as? [String: Any] {
      // Configure the map with creation params
      let region = MKCoordinateRegion(
        center: CLLocationCoordinate2D(latitude: 37.7749, longitude: -122.4194),
        latitudinalMeters: 1000,
        longitudinalMeters: 1000
      )
      mapView.setRegion(region, animated: false)
    }
  }

  func view() -> UIView { return mapView }
}

// Step 2: Implement FlutterPlatformViewFactory
class NativeMapFactory: NSObject, FlutterPlatformViewFactory {
  private var messenger: FlutterBinaryMessenger

  init(messenger: FlutterBinaryMessenger) {
    self.messenger = messenger
    super.init()
  }

  func create(
    withFrame frame: CGRect,
    viewIdentifier viewId: Int64,
    arguments args: Any?
  ) -> FlutterPlatformView {
    return NativeMapView(frame: frame, viewIdentifier: viewId, arguments: args)
  }

  // Required if you pass creationParams
  func createArgsCodec() -> FlutterMessageCodec & NSObjectProtocol {
    return FlutterStandardMessageCodec.sharedInstance()
  }
}

// Step 3: Register in AppDelegate or plugin
@UIApplicationMain
@objc class AppDelegate: FlutterAppDelegate {
  override func application(
    _ application: UIApplication,
    didFinishLaunchingWithOptions launchOptions: [UIApplication.LaunchOptionsKey: Any]?
  ) -> Bool {
    let controller = window?.rootViewController as! FlutterViewController
    let registrar = controller.registrar(forPlugin: "MapPlugin")!

    registrar.register(
      NativeMapFactory(messenger: registrar.messenger()),
      withId: "com.example/native_map"
    )

    return super.application(application, didFinishLaunchingWithOptions: launchOptions)
  }
}

5. Gesture Recognizer Conflict Resolution

When a platform view is inside a Flutter widget tree that also handles gestures (e.g., a map inside a scrollable), there is a conflict: who handles the touch event — Flutter or the native view? Flutter provides a gesture recognizer factory set to resolve this:

UiKitView(
  viewType: 'com.example/native_map',
  gestureRecognizers: <Factory<OneSequenceGestureRecognizer>>{
    // Pan — allow native map to handle drag/scroll
    Factory<PanGestureRecognizer>(() => PanGestureRecognizer()),
    // Scale — allow native map to handle pinch-to-zoom
    Factory<ScaleGestureRecognizer>(() => ScaleGestureRecognizer()),
    // Tap — allow native view to handle taps
    Factory<TapGestureRecognizer>(() => TapGestureRecognizer()),
  },
  // ...
)

Any gesture type listed in gestureRecognizers is forwarded to the native view. Gestures not listed are handled by Flutter's gesture arena first. Get this wrong and either the map won't scroll (Flutter swallows the pan) or the parent ListView won't scroll (native map swallows everything).

6. Performance Cost — Detailed Analysis

6.1 What Platform Views Break

Flutter's rendering model assumes a single GPU surface owned entirely by the raster thread. Platform views violate this by inserting GPU surfaces owned by the OS (Android SurfaceView, iOS CAMetalLayer). This forces Flutter to:

  • Split its own rendering into "above" and "below" layers relative to the native view
  • Synchronize the raster thread with the platform (UI) thread each frame — a mutex acquisition
  • Allocate additional FlutterImageView / overlay CALayer surfaces

6.2 Measured Cost

ScenarioRaster thread overheadNotes
No platform viewsBaselineSingle surface, no sync needed
1 platform view (Hybrid Composition)+2–5ms per frameThread sync + extra surface composite
Multiple platform viewsAdditive per viewEach view adds sync cost; avoid >1
Platform view in ListViewVery highEach visible item is a platform view — extreme cost; use Texture widget instead
Staff-Level Insight
The worst pattern is platform views inside a ListView — each list item becomes a separate native surface, multiplying the synchronization cost. If you need to show native content per list item (e.g., a native ad unit), use a texture-based approach: have native render to a SurfaceTexture / CVPixelBuffer and expose it to Flutter as a Texture widget. The Texture widget is composited as a regular TextureLayer without breaking Flutter's rendering model.

7. The Texture Widget — A Better Alternative

For native content that produces a stream of frames (camera preview, video, AR), the Texture widget is far more efficient than a platform view.

How it works:

  1. Native code creates a SurfaceTexture (Android) or registers a CVPixelBuffer producer (iOS)
  2. Native renders frames to this surface (camera HAL, video decoder, etc.)
  3. Flutter registers the texture via TextureRegistry and gets a texture ID
  4. Dart uses Texture(textureId: id) widget — this becomes a TextureLayer
  5. The raster thread samples the registered external texture as a GPU texture — no native view, no synchronization overhead
// Kotlin — register a surface texture
class CameraPlugin : FlutterPlugin {
  private var textureId: Long = -1

  override fun onAttachedToEngine(binding: FlutterPlugin.FlutterPluginBinding) {
    // Create a surface texture entry from Flutter's texture registry
    val entry = binding.textureRegistry.createSurfaceTexture()
    textureId = entry.id()

    // Give the SurfaceTexture to your camera SDK
    val surfaceTexture = entry.surfaceTexture()
    surfaceTexture.setDefaultBufferSize(1920, 1080)
    cameraDevice.startPreview(Surface(surfaceTexture))

    // Tell Dart the texture ID via MethodChannel
    channel.invokeMethod("onTextureRegistered", textureId)
  }
}
// Dart — display the texture
class CameraPreview extends StatefulWidget { ... }
class _CameraPreviewState extends State<CameraPreview> {
  late int _textureId;

  @override
  void initState() {
    super.initState();
    channel.setMethodCallHandler((call) async {
      if (call.method == 'onTextureRegistered') {
        setState(() => _textureId = call.arguments as int);
      }
    });
  }

  @override
  Widget build(BuildContext context) => _textureId == 0
      ? const CircularProgressIndicator()
      : Texture(textureId: _textureId);  // zero compositing overhead
}

9. Keyboard and Input Handling

9.1 Android

With Hybrid Composition, the native view is in the real Android view hierarchy, so the soft keyboard (IME) works correctly. When the native view requests focus and raises the keyboard, Android's window inset system handles it — Flutter is notified via MediaQuery.of(context).viewInsets in the normal way.

With Virtual Display, the soft keyboard often doesn't appear at all, or appears at the wrong size — because the IME interacts with the focused view's window, and the virtual display's window is not the real application window.

9.2 iOS

On iOS, the native UIView is in the real UIKit hierarchy, so keyboard handling works via standard UIKit responder chain. When a text field inside the native view becomes first responder, the keyboard appears and UIScrollView adjustments happen normally.

10. Lifecycle Management

Platform views must be properly cleaned up when the Flutter widget is removed from the tree. Implement dispose() in your PlatformView:

// Kotlin
class NativeMapView(...) : PlatformView {
  private val mapView = MapView(context)

  override fun getView() = mapView

  override fun dispose() {
    // Called when the Flutter widget is removed from the tree
    mapView.onStop()
    mapView.onDestroy()
    // Cancel any active network requests, release SDK resources, etc.
  }
}

// Swift
class NativeMapView: NSObject, FlutterPlatformView {
  private let mapView: MKMapView

  deinit {
    // Swift ARC handles most cleanup, but explicitly release delegates
    mapView.delegate = nil
  }
}
Tip
Platform view lifecycle also interacts with app lifecycle. If your native SDK (e.g., a map) needs onResume/onPause callbacks, implement ActivityAware on Android or use UIApplicationDelegate notifications on iOS to forward these to the native view.
