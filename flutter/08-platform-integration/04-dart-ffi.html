FFI vs Platform Channels

Aspect Dart FFI Platform Channel (MethodChannel)
Call overhead ~nanoseconds — direct function call, no serialization ~0.5–1ms — thread hop + codec serialize/deserialize
Threading Synchronous on calling isolate's thread (by default) Dart → platform thread → Dart (async)
Language support C ABI only (C, C++, Rust with C exports, Go cgo) Kotlin/Swift/Java/ObjC (platform language)
Type safety Manual (Pointer arithmetic, type casts) Dynamic (Map<String, dynamic>) unless Pigeon used
Memory management Manual (malloc/free or Arena) Automatic (handled by codec)
Cross-platform Yes — same C library works on Android/iOS/desktop No — separate Kotlin + Swift implementations required
Best for CPU-intensive computation, existing C/C++ libraries, high-frequency calls Platform UI, system APIs (camera, GPS), one-off calls

Loading a Native Library

import 'dart:ffi';
import 'dart:io';

// Three ways to load a library:

// 1. Open a specific shared library by name
final dylib = DynamicLibrary.open('libcrypto.so'); // Android
// On iOS: DynamicLibrary.open('libcrypto.dylib') or use process()

// 2. Search the process's already-loaded libraries
// (works for symbols in libflutter.so or statically linked libs)
final dylib = DynamicLibrary.process();

// 3. Search the main executable itself
final dylib = DynamicLibrary.executable();

// Platform-aware loading:
DynamicLibrary _openLib() {
  if (Platform.isAndroid) return DynamicLibrary.open('libmycrypto.so');
  if (Platform.isIOS) return DynamicLibrary.process(); // statically linked on iOS
  if (Platform.isMacOS) return DynamicLibrary.open('libmycrypto.dylib');
  if (Platform.isWindows) return DynamicLibrary.open('mycrypto.dll');
  throw UnsupportedError('Platform not supported');
}
final _lib = _openLib();

NativeFunction Type Signatures

// C function signature:
// int32_t add(int32_t a, int32_t b);

// Dart FFI type:
// NativeFunction<Int32 Function(Int32, Int32)>
// - Int32, Int64, Float, Double, Void, Pointer, etc. are native types
// - The Function(...) inside NativeFunction is the C-side type
// - The Dart callable type is: int Function(int, int)

typedef AddNative = Int32 Function(Int32 a, Int32 b);
typedef AddDart = int Function(int a, int b);

// Look up the symbol and cast to the Dart callable type:
final AddDart add = _lib
    .lookup<NativeFunction<AddNative>>('add')
    .asFunction<AddDart>();

// Call it — direct native call, no thread hop:
final result = add(3, 4); // result == 7

// More complex: function returning a pointer
// C: uint8_t* encrypt(const uint8_t* data, int32_t len, int32_t* out_len);
typedef EncryptNative = Pointer<Uint8> Function(
    Pointer<Uint8> data, Int32 len, Pointer<Int32> outLen);
typedef EncryptDart = Pointer<Uint8> Function(
    Pointer<Uint8> data, int len, Pointer<Int32> outLen);

Memory Management: Arena

Native memory allocated via FFI is outside the Dart GC. You must free it manually or use an Arena (scoped allocator) that frees everything when the scope exits.

import 'package:ffi/ffi.dart'; // provides Arena, using(), toNativeUtf8(), etc.

// Arena pattern — all allocations freed at end of using() block:
void encryptData(Uint8List plaintext) {
  using((Arena arena) {
    // Allocate native buffer for input data
    final nativePlaintext = arena<Uint8>(plaintext.length);
    // Copy Dart bytes to native buffer
    nativePlaintext.asTypedList(plaintext.length).setAll(0, plaintext);

    // Allocate output length pointer
    final outLen = arena<Int32>();

    // Call native function
    final encrypted = encrypt(nativePlaintext, plaintext.length, outLen);

    // Read result before arena is freed
    final resultBytes = encrypted.asTypedList(outLen.value);
    final dartResult = Uint8List.fromList(resultBytes);

    // Free the encrypted buffer (returned by C — must free manually)
    calloc.free(encrypted);

    // arena.releaseAll() is called automatically here
    // All arena-allocated pointers are freed
    return dartResult;
  });
}

// Manual allocation with calloc (must free explicitly):
final buffer = calloc<Uint8>(1024);
try {
  // use buffer
} finally {
  calloc.free(buffer); // always free, even on exception
}

Passing Strings

import 'package:ffi/ffi.dart';

// C: char* greet(const char* name);
typedef GreetNative = Pointer<Utf8> Function(Pointer<Utf8> name);
typedef GreetDart = Pointer<Utf8> Function(Pointer<Utf8> name);

final greet = _lib.lookup<NativeFunction<GreetNative>>('greet')
    .asFunction<GreetDart>();

String callGreet(String name) {
  return using((arena) {
    // Dart String → null-terminated UTF-8 C string
    final namePtr = name.toNativeUtf8(allocator: arena);

    // Call native function
    final resultPtr = greet(namePtr);

    // C string → Dart String
    final result = resultPtr.toDartString();

    // Free the returned C string if C allocated it:
    // calloc.free(resultPtr);
    // (skip if the library owns the memory)

    return result;
  });
}

Structs and Unions

// C struct:
// typedef struct {
//   double x;
//   double y;
//   double z;
// } Vec3;

final class Vec3 extends Struct {
  @Double()
  external double x;

  @Double()
  external double y;

  @Double()
  external double z;
}

// C: Vec3 cross_product(Vec3 a, Vec3 b);
typedef CrossNative = Vec3 Function(Vec3 a, Vec3 b);
typedef CrossDart = Vec3 Function(Vec3 a, Vec3 b);

// Creating a struct on native heap:
final v = calloc<Vec3>();
v.ref.x = 1.0;
v.ref.y = 0.0;
v.ref.z = 0.0;
calloc.free(v);

// Nested structs:
final class Quaternion extends Struct {
  external Vec3 xyz;

  @Double()
  external double w;
}

Arrays

// C: void fill_array(int32_t* arr, int32_t count, int32_t value);
typedef FillArrayNative = Void Function(Pointer<Int32>, Int32, Int32);
typedef FillArrayDart = void Function(Pointer<Int32>, int, int);

final fillArray = _lib.lookup<NativeFunction<FillArrayNative>>('fill_array')
    .asFunction<FillArrayDart>();

void example() {
  using((arena) {
    const count = 100;
    final arr = arena<Int32>(count); // allocate 100 int32s

    fillArray(arr, count, 42);

    // View as Dart typed list — zero-copy view into native memory
    final dartList = arr.asTypedList(count); // Int32List
    print(dartList[0]); // 42
  });
}

Callbacks: NativeCallable

Sometimes native code needs to call back into Dart. Use NativeCallable (Dart 3.3+) to create a native function pointer that invokes Dart code.

// C: void set_progress_callback(void (*callback)(int progress, void* context));

typedef ProgressCallbackNative = Void Function(Int32 progress);

// isolateLocal: callback runs synchronously on the current isolate's thread
// Safe to use when you know native will call back on the same thread as Dart
final callback = NativeCallable<ProgressCallbackNative>.isolateLocal(
  (int progress) {
    print('Progress: $progress%');
    // Safe to access Dart objects here
  },
  exceptionalReturn: ffi.nullptr, // what to return if Dart throws
);

// Pass the native function pointer to C:
setProgressCallback(callback.nativeFunction);

// IMPORTANT: close the callback when done to free resources
// If callback is GC'd before close(), native may call a dangling pointer
callback.close();

// listener: callback can be called from ANY native thread
// Posts the call to the isolate's event loop — safe but async
final safeCallback = NativeCallable<ProgressCallbackNative>.listener(
  (int progress) {
    // This runs on the Dart event loop, not on the calling thread
    // Safe to use setState, etc.
    updateProgressUI(progress);
  },
);

ffigen: Automatic Binding Generation

ffigen reads C/C++ header files and generates Dart FFI bindings automatically, eliminating hand-writing type signatures.

# pubspec.yaml
dev_dependencies:
  ffigen: ^12.0.0

# ffigen.yaml (binding configuration)
name: CryptoBindings
description: Bindings for libcrypto
output: lib/src/crypto_bindings.dart
headers:
  entry-points:
    - include/crypto.h
  include-directives:
    - include/crypto.h
compiler-opts:
  - '-I/usr/include'
functions:
  include:
    - 'crypto_.*'    # only generate bindings for functions starting with crypto_
structs:
  include:
    - 'CryptoKey'
    - 'CryptoResult'
enums:
  include:
    - 'CryptoAlgorithm'
# Run ffigen:
dart run ffigen --config ffigen.yaml

# Generated file: lib/src/crypto_bindings.dart
# Contains a class with all function bindings pre-typed:

class CryptoBindings {
  final DynamicLibrary _dylib;
  CryptoBindings(this._dylib);

  late final _crypto_encrypt = _dylib
      .lookup<NativeFunction<_EncryptNative>>('crypto_encrypt')
      .asFunction<_EncryptDart>();

  Pointer<Uint8> crypto_encrypt(
      Pointer<Uint8> data, int len, Pointer<Int32> outLen) =>
      _crypto_encrypt(data, len, outLen);
}
Staff Interview Signal
The key trade-off interviewers probe: FFI is synchronous on the calling isolate's thread. If the native function is slow (>1ms), it blocks the UI isolate and drops frames. The solution is to run FFI calls on a separate isolate via Isolate.run() or compute(). However, you cannot pass Dart objects or closures to another isolate — only primitive types and SendPort. For FFI + isolates: pass the native library path and function addresses (as ints) across the isolate boundary, then reload the library and call it in the worker isolate. This is the advanced pattern for truly parallel native computation from Flutter.
