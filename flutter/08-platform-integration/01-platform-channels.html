1. Channel Types Overview

Flutter provides three channel types to bridge the Dart world and the host platform. Each is built on top of the same BinaryMessenger abstraction but differs in its message flow pattern and intended use case.

Channel Direction Pattern Use Case
MethodChannel Bidirectional (RPC-style) Request / Response Call a native API and receive a single result: battery level, camera, file I/O
EventChannel Native → Dart (one-way stream) Producer / Consumer Continuous native events: location updates, sensors, Bluetooth state
BasicMessageChannel Bidirectional (raw messages) Send / Reply (optional) Arbitrary message exchange with custom codecs; lower-level than MethodChannel
Channel Type Decision Tree ══════════════════════════ Need to call a native function and get a result? │ ┌─────────┴─────────┐ YES NO │ │ MethodChannel Continuous event stream? │ ┌─────────┴─────────┐ YES NO │ │ EventChannel BasicMessageChannel (custom protocol)
Architect Insight

All three channels share the same BinaryMessenger plumbing. They differ only in codec and flow conventions layered on top. Understanding BinaryMessenger means you can build your own channel abstraction when none of the three built-in types fits — or when profiling shows the codec overhead is a bottleneck.

2. BinaryMessenger: The Low-Level Foundation

BinaryMessenger is the interface that all three channel types delegate to. It speaks only in ByteData — there is no concept of method names, arguments, or streams at this layer. The engine exposes it to the Dart framework via dart:ui.

// dart:ui — the raw interface exposed by the engine
abstract class BinaryMessenger {
  // Send a ByteData message on [channel] and await a ByteData reply.
  Future<ByteData?> send(String channel, ByteData? message);

  // Register a handler that will receive incoming ByteData messages.
  void setMessageHandler(
    String channel,
    Future<ByteData?> Function(ByteData? message)? handler,
  );
}

// The concrete implementation used at runtime.
// Backed by the engine's Shell and the platform's message loop.
class _DefaultBinaryMessenger extends BinaryMessenger {
  // Internal map: channel name → handler closure
  final Map<String, MessageHandler> _handlers = {};

  @override
  Future<ByteData?> send(String channel, ByteData? message) {
    // Hands the ByteData to the engine via a native call.
    // Suspends the current microtask; the engine routes the bytes to
    // the platform thread and eventually resumes with the response.
    final Completer<ByteData?> completer = Completer();
    _sendPlatformMessage(channel, message, completer.complete);
    return completer.future;
  }

  @override
  void setMessageHandler(String channel, MessageHandler? handler) {
    if (handler == null) {
      _handlers.remove(channel);
      _clearMessageHandler(channel);   // unregisters in engine
    } else {
      _handlers[channel] = handler;
      _setMessageHandler(channel);     // registers in engine
    }
  }
}
Key Detail

The _handlers map lives on the Dart side. When the engine delivers an incoming message on a channel, it calls into Dart, which looks up the handler in this map. There is no separate registration on the platform side — the engine routes by channel name string.

2.1 How MethodChannel Wraps BinaryMessenger

// Simplified flutter/src/services/platform_channel.dart
class MethodChannel {
  const MethodChannel(this.name, [
    this.codec = const StandardMethodCodec(),
    BinaryMessenger? binaryMessenger,
  ]) : _binaryMessenger = binaryMessenger;

  final String name;
  final MethodCodec codec;
  final BinaryMessenger? _binaryMessenger;

  // Lazily resolves to ServicesBinding.instance.defaultBinaryMessenger
  BinaryMessenger get binaryMessenger =>
      _binaryMessenger ?? ServicesBinding.instance.defaultBinaryMessenger;

  Future<T?> invokeMethod<T>(String method, [dynamic arguments]) async {
    // 1. Encode method name + arguments → ByteData
    final ByteData encoded =
        codec.encodeMethodCall(MethodCall(method, arguments));

    // 2. Delegate to BinaryMessenger (crosses to platform thread)
    final ByteData? result =
        await binaryMessenger.send(name, encoded);

    // 3. null reply → platform returned notImplemented
    if (result == null) {
      throw MissingPluginException(...);
    }

    // 4. Decode reply → T or throw PlatformException
    return codec.decodeEnvelope(result) as T?;
  }
}

3. Message Codecs

Codecs are responsible for encoding/decoding the Dart values to/from the binary wire format. The codec choice affects what types can be transmitted and the serialization cost.

3.1 StandardMessageCodec

The default codec. Produces a compact binary format using a type-tag byte followed by the payload. Supports a fixed set of Dart types:

Tag Byte Dart Type Notes
0x00 null 0 bytes payload
0x01 true 0 bytes payload
0x02 false 0 bytes payload
0x03 int (32-bit) 4 bytes LE
0x04 int (64-bit) 8 bytes LE
0x06 double 8 bytes IEEE 754
0x07 Uint8List length-prefixed bytes (zero-copy candidate)
0x0c String length-prefixed UTF-8
0x0d List length-prefixed, each element recursively encoded
0x0e Map length-prefixed, key-value pairs recursively encoded
Performance Pitfall

Sending a large Map or nested List over StandardMessageCodec copies every element during encoding and decoding. For high-throughput data (e.g., audio samples, image pixels), prefer Uint8List or Float64List — these map directly to contiguous memory and the codec skips per-element type tags.

3.2 JSONMessageCodec

JSONMessageCodec encodes the message as a JSON string then re-encodes that string as UTF-8 bytes. It supports only JSON-compatible types and is strictly slower than StandardMessageCodec for all value types — avoid it for new code.

3.3 StandardMethodCodec

Wraps StandardMessageCodec with an envelope format for method calls and replies:

// Wire format for a MethodCall
// ┌─────────────────────┬─────────────────────────────┐
// │  method name (String)│         arguments           │
// │  (StandardCodec)    │       (StandardCodec)        │
// └─────────────────────┴─────────────────────────────┘

// Wire format for a successful Reply
// ┌──────────┬──────────────────────────────────────────┐
// │  0x00    │         result (StandardCodec)           │
// │ (success)│                                          │
// └──────────┴──────────────────────────────────────────┘

// Wire format for an Error Reply
// ┌──────────┬─────────────┬─────────────┬──────────────┐
// │  0x01    │  errorCode  │ errorMessage│  errorDetails│
// │  (error) │  (String)   │ (String?)   │  (any?)      │
// └──────────┴─────────────┴─────────────┴──────────────┘

// Wire format for notImplemented
// ┌──────────┐
// │  (empty) │   — zero bytes; BinaryMessenger returns null
// └──────────┘

4. Channel Registration Internals

On the Dart side, the registry is simply the _handlers map in _DefaultBinaryMessenger. On the platform side, each platform maintains its own equivalent:

4.1 Android (Kotlin)

// android/src/main/java/io/flutter/plugin/common/BinaryMessenger.java
// The engine implementation is DartMessenger.kt

class DartMessenger(private val flutterJNI: FlutterJNI) : BinaryMessenger {
    // Channel name → handler registered by plugin
    private val messageHandlers: MutableMap<String, BinaryMessageHandler> = mutableMapOf()

    override fun setMessageHandler(
        channel: String,
        handler: BinaryMessageHandler?
    ) {
        if (handler == null) {
            messageHandlers.remove(channel)
        } else {
            messageHandlers[channel] = handler
        }
    }

    // Called by FlutterJNI when the engine delivers an incoming message
    fun handleMessageFromDart(channel: String, message: ByteBuffer?, replyId: Int) {
        val handler = messageHandlers[channel]
        if (handler == null) {
            flutterJNI.invokePlatformMessageEmptyResponseCallback(replyId)
            return
        }
        handler.onMessage(message) { reply ->
            flutterJNI.invokePlatformMessageResponseCallback(replyId, reply, reply?.limit() ?: 0)
        }
    }
}

4.2 iOS (Swift)

// FlutterBinaryMessengerRelay.swift (simplified)
class FlutterEngine {
    // Registry: channel → handler block
    var messageHandlers: [String: FlutterBinaryMessageHandler] = [:]

    func setMessageHandlerOnChannel(
        _ channel: String,
        binaryMessageHandler handler: FlutterBinaryMessageHandler?
    ) {
        if let h = handler {
            messageHandlers[channel] = h
        } else {
            messageHandlers.removeValue(forKey: channel)
        }
    }

    // Called from C++ engine when a Dart message arrives
    func handlePlatformMessage(
        _ channel: String,
        message: Data?,
        replyId: Int
    ) {
        guard let handler = messageHandlers[channel] else {
            // Send empty reply = notImplemented
            sendEmptyReply(replyId)
            return
        }
        handler(message) { reply in
            self.sendReply(replyId, data: reply)
        }
    }
}

5. Threading Model

Platform Channel Threading ══════════════════════════ Dart (UI Thread) Platform Thread (Main) ──────────────── ────────────────────── invokeMethod("getBattery") │ encode ByteData (sync) │ BinaryMessenger.send() ─────────▶ DartMessenger.handleMessageFromDart() [suspends Future] │ setMethodCallHandler callback fires │ BatteryManager.getCapacity() (sync OK) │ result.success(level) ──────────────┐ │ Future resumes <──────────────────────────────────────────────────────┘ decodeEnvelope() (sync) │ T returned to caller
Critical Threading Rule

The method call handler on Android runs on the main (UI) thread by default. Never perform blocking I/O or long computations there — it will cause ANR. Use a Handler or coroutine to move work to a background thread, then call result.success() from any thread. On iOS, the handler also runs on the main thread; use DispatchQueue.global().async to offload work.

5.1 Correct Async Pattern — Android

class FilePlugin : FlutterPlugin, MethodCallHandler {
    private lateinit var channel: MethodChannel
    private val scope = CoroutineScope(SupervisorJob() + Dispatchers.IO)

    override fun onMethodCall(call: MethodCall, result: Result) {
        when (call.method) {
            "readLargeFile" -> {
                // DO: move work off the main thread
                scope.launch {
                    try {
                        val content = readFileFromDisk(call.argument("path")!!)
                        // result.success is thread-safe — it posts back to main thread
                        withContext(Dispatchers.Main) {
                            result.success(content)
                        }
                    } catch (e: Exception) {
                        withContext(Dispatchers.Main) {
                            result.error("IO_ERROR", e.message, null)
                        }
                    }
                }
            }
            else -> result.notImplemented()
        }
    }
}

5.2 Correct Async Pattern — iOS (Swift)

class FilePlugin: NSObject, FlutterPlugin {
    func handle(_ call: FlutterMethodCall, result: @escaping FlutterResult) {
        switch call.method {
        case "readLargeFile":
            guard let path = (call.arguments as? [String: Any])?["path"] as? String
            else { result(FlutterError(code: "INVALID_ARG", message: nil, details: nil)); return }

            DispatchQueue.global(qos: .userInitiated).async {
                do {
                    let content = try String(contentsOfFile: path, encoding: .utf8)
                    DispatchQueue.main.async { result(content) }
                } catch {
                    DispatchQueue.main.async {
                        result(FlutterError(code: "IO_ERROR", message: error.localizedDescription, details: nil))
                    }
                }
            }
        default:
            result(FlutterMethodNotImplemented)
        }
    }
}

6. BackgroundIsolateBinaryMessenger (Flutter 3.7+)

Before Flutter 3.7, platform channels could only be used from the root isolate (the UI thread isolate). Background isolates spawned with Isolate.spawn had no access to ServicesBinding and would crash with MissingPluginException.

Flutter 3.7 introduced BackgroundIsolateBinaryMessenger and RootIsolateToken to solve this:

// In the root isolate — capture a token before spawning
final RootIsolateToken token = RootIsolateToken.instance!;
await Isolate.spawn(backgroundWork, token);

// In the background isolate entry point
void backgroundWork(RootIsolateToken token) {
  // Register this isolate with the root messenger infrastructure
  BackgroundIsolateBinaryMessenger.ensureInitialized(token);

  // Now platform channels work normally from this isolate
  const channel = MethodChannel('com.example/file');
  final result = await channel.invokeMethod<String>('readLargeFile', {
    'path': '/data/some/path',
  });
  // Use result...
}

// ⚠️ Responses are delivered on the background isolate's event loop,
// not the UI thread. You must send results back via SendPort or
// IsolateNameServer if the UI isolate needs them.
Architect Insight

BackgroundIsolateBinaryMessenger works by routing all messages through the root isolate's engine connection. The background isolate does not have its own engine thread; its channel calls are serialized through the root isolate's shell. This means channel latency from background isolates is slightly higher — they must cross two isolate boundaries. For compute-heavy tasks that also need platform APIs (e.g., background PDF generation + file save), this is still far preferable to polling via the UI isolate.
