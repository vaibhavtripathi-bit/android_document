Why Offload to Background Isolates?

Flutter's UI thread (the Dart root isolate) has a strict 16.6ms budget per frame at 60fps. Any synchronous Dart work that exceeds this budget will cause a dropped frame — visible jank. Common offload candidates:

  • JSON decoding of large API responses (10K+ objects)
  • Image processing (resizing, filtering, compression)
  • Cryptographic operations (hashing, encryption)
  • Database queries with heavy in-Dart processing
  • File parsing (CSV, XML, large binary formats)
  • Machine learning inference (when not using FFI-based TFLite)

Dart's answer is isolates — true parallel execution with separate heaps. Three APIs exist for different use cases: compute(), Isolate.run(), and the full Isolate.spawn() for long-lived workers.

compute() — Flutter's Convenience API

What compute() Does

compute() is a Flutter function (not Dart SDK) defined in package:flutter/foundation.dart. It spawns a new isolate, runs a top-level or static function with a single argument, waits for the result, kills the isolate, and returns the result as a Future. One task, one isolate, then isolate is destroyed.

import 'package:flutter/foundation.dart';

// The function to run in the background isolate
// Must be a top-level function or static method — NOT an instance method or closure
List<Product> _parseProducts(String jsonString) {
  final List<dynamic> decoded = jsonDecode(jsonString);
  return decoded.map((j) => Product.fromJson(j)).toList();
}

// In your widget or BLoC:
Future<void> loadProducts() async {
  final jsonString = await http.read(Uri.parse('https://api.example.com/products'));

  // Runs _parseProducts in a background isolate
  // UI thread is free to render frames while this runs
  final products = await compute(_parseProducts, jsonString);

  setState(() => _products = products);
}

compute() Constraints

Important Constraint

The callback passed to compute() must be a top-level function or static method. It cannot be an instance method (because the instance would need to be transferred across isolate boundaries — impossible without serialization). It also cannot capture variables from its outer scope (it's not a closure that can capture this). All data it needs must come through its single argument.

// WRONG — captures 'this' via instance method
Future<void> wrong() async {
  await compute(_processWithConfig, data); // OK if _processWithConfig is top-level

  // But this won't work:
  await compute((data) => _instanceMethod(data), data); // Error — can't pass closure
}

// CORRECT — pass all config in a record/map/class
Future<void> correct() async {
  final config = ProcessConfig(threshold: _threshold, mode: _mode);
  final result = await compute(_processData, (data: rawData, config: config));
}

// Top-level function takes a record
List<Item> _processData((List<RawItem> data, ProcessConfig config) args) {
  final (data, config) = args;
  return data
      .where((item) => item.value > config.threshold)
      .map((item) => Item.fromRaw(item, config.mode))
      .toList();
}

compute() Under the Hood

Internally, compute() calls Isolate.run() (since Flutter 3.7) which in turn uses Isolate.spawn() with the isolate group sharing mechanism from Dart 2.15. The message passing uses SendPort/ReceivePort but is wrapped so you only see the Future result.

Isolate.run() — Dart 2.19+ Clean API

The Modern Alternative

Isolate.run() was introduced in Dart 2.19 as a cleaner, more Dart-idiomatic API over raw Isolate.spawn(). Unlike compute(), it accepts closures (because it's a Dart SDK API that handles the closure capture mechanism internally).

import 'dart:isolate';

Future<void> processImageWithRun(Uint8List imageBytes) async {
  // Isolate.run accepts a closure — captures variables safely
  final processed = await Isolate.run(() {
    // This runs in a background isolate
    // Variables from outer scope are COPIED into the isolate (via message passing)
    return _heavyImageProcessing(imageBytes);
  });

  // Use processed result on UI thread
  displayImage(processed);
}

Uint8List _heavyImageProcessing(Uint8List input) {
  // CPU-intensive image manipulation
  final result = Uint8List(input.length);
  for (var i = 0; i < input.length; i += 4) {
    // Grayscale conversion
    final gray = (input[i] * 0.299 + input[i+1] * 0.587 + input[i+2] * 0.114).round();
    result[i] = result[i+1] = result[i+2] = gray;
    result[i+3] = input[i+3]; // Preserve alpha
  }
  return result;
}

Tip

Prefer Isolate.run() in new code if you're targeting Flutter 3.7+ (Dart 2.19+). It's cleaner and more expressive. Keep compute() for backward compatibility or when you want to be explicit about the top-level function constraint (which self-documents that the function has no captured state).

Long-Running Worker Isolates

When compute() Is Not Enough

Both compute() and Isolate.run() create a new isolate per task and destroy it after the result. Spawning an isolate takes ~1–5ms. For tasks that run hundreds of times (processing camera frames, handling a stream of sensor data), spawning a new isolate per task is expensive. The solution: a persistent worker isolate with a message loop.

Long-Running Worker Pattern

import 'dart:isolate';

/// A persistent background worker that processes tasks without respawning
class BackgroundWorker {
  late final Isolate _isolate;
  late final SendPort _sendPort;
  final _receivePort = ReceivePort();
  final _completerMap = <int, Completer<dynamic>>{};
  int _nextId = 0;

  Future<void> init() async {
    // Spawn the isolate — it lives for the app's lifetime
    _isolate = await Isolate.spawn(
      _workerEntryPoint,
      _receivePort.sendPort,
      debugName: 'BackgroundWorker',
    );

    // First message from isolate is its SendPort
    _sendPort = await _receivePort.first;

    // Listen for results
    _receivePort.listen((message) {
      if (message is _WorkerResult) {
        final completer = _completerMap.remove(message.id);
        if (message.error != null) {
          completer?.completeError(message.error!);
        } else {
          completer?.complete(message.result);
        }
      }
    });
  }

  Future<T> process<T>(dynamic data) {
    final id = _nextId++;
    final completer = Completer<T>();
    _completerMap[id] = completer;
    _sendPort.send(_WorkerTask(id: id, data: data));
    return completer.future;
  }

  void dispose() {
    _receivePort.close();
    _isolate.kill();
  }

  /// Entry point runs in the background isolate
  static void _workerEntryPoint(SendPort mainSendPort) {
    final receivePort = ReceivePort();
    // Send this isolate's SendPort back to main isolate
    mainSendPort.send(receivePort.sendPort);

    // Message loop — runs until isolate is killed
    receivePort.listen((message) {
      if (message is _WorkerTask) {
        try {
          final result = _processTask(message.data);
          mainSendPort.send(_WorkerResult(id: message.id, result: result));
        } catch (e) {
          mainSendPort.send(_WorkerResult(id: message.id, error: e));
        }
      }
    });
  }

  static dynamic _processTask(dynamic data) {
    // Heavy processing happens here
    if (data is String) {
      return jsonDecode(data); // Example: JSON parsing
    }
    throw ArgumentError('Unknown task type: ${data.runtimeType}');
  }
}

// Message types (must be transferable — no closures, no platform objects)
class _WorkerTask {
  final int id;
  final dynamic data;
  const _WorkerTask({required this.id, required this.data});
}

class _WorkerResult {
  final int id;
  final dynamic result;
  final dynamic error;
  const _WorkerResult({required this.id, this.result, this.error});
}

Usage Pattern

// In your app state / service locator
final worker = BackgroundWorker();
await worker.init(); // Once at startup

// Process multiple tasks efficiently (no isolate spawn overhead per task)
final result1 = await worker.process<Map<String, dynamic>>(jsonString1);
final result2 = await worker.process<Map<String, dynamic>>(jsonString2);

// Clean up
worker.dispose();

IsolateNameServer: Cross-Isolate Discovery

What IsolateNameServer Is

IsolateNameServer is a Flutter API that provides a global registry mapping string names to SendPort objects. It allows isolates to discover each other by name without needing to pass SendPort objects directly (which requires a shared reference at spawn time).

import 'dart:ui'; // IsolateNameServer is in dart:ui

// In the background isolate — register its port with a name
void backgroundIsolateMain(SendPort _) {
  final port = ReceivePort();

  // Register so other isolates can find us
  final registered = IsolateNameServer.registerPortWithName(
    port.sendPort,
    'background_processor', // unique name
  );

  if (!registered) {
    // Name already taken — remove old and re-register
    IsolateNameServer.removePortNameMapping('background_processor');
    IsolateNameServer.registerPortWithName(
      port.sendPort,
      'background_processor',
    );
  }

  port.listen((message) {
    // Handle messages
  });
}

// In another isolate — look up the background isolate's port
void sendToBackground(String data) {
  final sendPort = IsolateNameServer.lookupPortByName('background_processor');
  if (sendPort != null) {
    sendPort.send(data);
  } else {
    // Background isolate not yet started
    print('Background isolate not registered yet');
  }
}

IsolateNameServer Use Cases

  • Background download + UI notification: A background isolate doing file downloads registers a port; the UI isolate queries it to send cancellation signals.
  • Workmanager callbacks: Flutter's workmanager package runs its callback in a separate isolate — IsolateNameServer lets the callback communicate with the main isolate if it's running.
  • Flutter Local Notifications: Background notification tap callbacks use IsolateNameServer to forward tap events to the main isolate's event stream.
Caution

IsolateNameServer is a global mutable registry — it's not type-safe and name collisions between plugins are possible. Always use a sufficiently unique name (e.g., include your package name as a prefix). Also, the registry does not persist across process restarts — you must re-register on each app launch.

Worker Pool Pattern

Parallelizing CPU Work Across Multiple Isolates

class IsolatePool {
  final int size;
  final List<BackgroundWorker> _workers = [];
  int _roundRobin = 0;

  IsolatePool({required this.size});

  Future<void> init() async {
    for (var i = 0; i < size; i++) {
      final worker = BackgroundWorker();
      await worker.init();
      _workers.add(worker);
    }
  }

  // Round-robin task distribution
  Future<T> submit<T>(dynamic task) {
    final worker = _workers[_roundRobin % size];
    _roundRobin++;
    return worker.process<T>(task);
  }

  void dispose() {
    for (final w in _workers) w.dispose();
  }
}

// Usage: process 100 images in parallel across 4 isolates
final pool = IsolatePool(size: 4);
await pool.init();

final futures = images.map((img) => pool.submit<Uint8List>(img)).toList();
final results = await Future.wait(futures);

A pool of N isolates gives true parallelism on N CPU cores. Dart's isolates map to OS threads, so this actually uses multiple CPU cores unlike async/await (which is cooperative concurrency on a single thread).

Real Flutter Examples

JSON Parsing in Background

// products_repository.dart
class ProductsRepository {
  final Dio _dio;

  Future<List<Product>> fetchProducts() async {
    final response = await _dio.get('/api/products');
    final jsonString = response.data as String;

    // Parse 5000-item JSON in background — won't block UI
    return Isolate.run(() {
      final List decoded = jsonDecode(jsonString);
      return decoded.map((j) => Product.fromJson(j as Map<String, dynamic>)).toList();
    });
  }
}

Image Processing

// image_service.dart
class ImageService {
  /// Resize image to thumbnail without blocking UI
  Future<Uint8List> createThumbnail(Uint8List originalBytes, int size) {
    return compute(_resizeImage, (bytes: originalBytes, size: size));
  }
}

// Top-level function (required by compute)
Uint8List _resizeImage(({Uint8List bytes, int size}) args) {
  final image = img.decodeImage(args.bytes)!;
  final thumbnail = img.copyResize(image, width: args.size, height: args.size);
  return Uint8List.fromList(img.encodeJpg(thumbnail, quality: 85));
}

// Usage in Widget
Future<void> _processSelectedImage(File file) async {
  final bytes = await file.readAsBytes();
  // This completes in background — UI stays smooth
  final thumbnail = await ImageService().createThumbnail(bytes, 256);
  setState(() => _thumbnailBytes = thumbnail);
}

BackgroundIsolateBinaryMessenger (Flutter 3.7+)

Platform Channels from Background Isolates

Before Flutter 3.7, platform channels could only be called from the root isolate. Background isolates attempting platform channel calls would crash or silently fail. Flutter 3.7 introduced BackgroundIsolateBinaryMessenger to solve this.

// In main isolate — capture the token before spawning
import 'package:flutter/services.dart';

void main() {
  WidgetsFlutterBinding.ensureInitialized();

  // Capture root isolate token for use in background isolates
  final rootIsolateToken = RootIsolateToken.instance!;

  runApp(MyApp());

  // Pass token to background isolate
  Isolate.spawn(_backgroundIsolateMain, rootIsolateToken);
}

// Background isolate entry point
Future<void> _backgroundIsolateMain(RootIsolateToken rootToken) async {
  // Initialize the binary messenger for this background isolate
  BackgroundIsolateBinaryMessenger.ensureInitialized(rootToken);

  // Now platform channels work from this isolate!
  const channel = MethodChannel('com.example/heavy_processing');

  // This was impossible before Flutter 3.7
  final result = await channel.invokeMethod<String>('processInNative', {'data': 'input'});

  print('Result from native: $result');
}

Practical Use Cases for BackgroundIsolateBinaryMessenger

  • A background isolate doing heavy processing that needs to read device info (battery, connectivity) without round-tripping to the main isolate
  • Background download isolates that use platform file APIs (secure storage, documents directory)
  • Workmanager background tasks that need to interact with platform services
  • Plugin authors who want their plugin to work from any isolate without requiring main isolate marshalling
Staff Insight

BackgroundIsolateBinaryMessenger is a significant architectural improvement. Previously, apps had to use IsolateNameServer + manual message forwarding to have background isolates interact with platform APIs — a complex pattern prone to bugs. The new API democratizes platform channel access across isolates while maintaining thread safety (the BinaryMessenger implementation still routes calls to the platform thread).

Choosing the Right API: Decision Tree

ScenarioUseWhy
Single CPU-intensive task, one-shotIsolate.run() or compute()Simple API, isolate cleaned up automatically
Repeated tasks (many per second)Long-running worker isolateAvoid spawn overhead per task (~1–5ms)
Parallel processing of N independent itemsIsolatePoolTrue parallelism across CPU cores
Background service that needs platform APIsIsolate.spawn + BackgroundIsolateBinaryMessengerLong-lived isolate with platform access
Cross-isolate communication (lookup by name)IsolateNameServerGlobal registry — no need to share SendPort at spawn time
Task is async (network, file IO)async/await on UI isolateAsync IO is non-blocking — no isolate needed
Key Rule

Isolates are for CPU-bound work (computation). For IO-bound work (network, disk), async/await is sufficient — IO operations release the event loop while waiting, so they don't block the UI thread even without isolates.
