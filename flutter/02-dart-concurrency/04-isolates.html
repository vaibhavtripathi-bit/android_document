1. Isolate Memory Model

A Dart isolate is the fundamental unit of concurrency and memory isolation. Each isolate has its own:

  • Heap — all Dart objects live here; not accessible from other isolates
  • Stack — the call stack for the currently executing function
  • Event loop — microtask queue + event queue, processed on a dedicated OS thread
  • Globals — static variables are per-isolate; isolates do not share static state
  ┌─────────────────────────────────┐   ┌─────────────────────────────────┐
  │         MAIN ISOLATE            │   │       WORKER ISOLATE            │
  │  (UI thread in Flutter)         │   │                                 │
  │                                 │   │                                 │
  │  ┌──────────────────────────┐   │   │  ┌──────────────────────────┐  │
  │  │   Dart Heap (GC-managed) │   │   │  │   Dart Heap (GC-managed) │  │
  │  │  Widget tree objects     │   │   │  │  Worker-local objects     │  │
  │  │  State, Models, etc.     │   │   │  │  Task results, buffers   │  │
  │  └──────────────────────────┘   │   │  └──────────────────────────┘  │
  │                                 │   │                                 │
  │  Event Loop                     │   │  Event Loop                     │
  │  ┌──────────┐ ┌───────────┐    │   │  ┌──────────┐ ┌───────────┐   │
  │  │Microtask │ │  Event Q  │    │   │  │Microtask │ │  Event Q  │   │
  │  └──────────┘ └───────────┘    │   │  └──────────┘ └───────────┘   │
  │                                 │   │                                 │
  │  ReceivePort ◄─────────────────────────────── SendPort               │
  │  SendPort ─────────────────────────────────► ReceivePort             │
  │         (messages are DEEP COPIED on send)  │                        │
  └─────────────────────────────────┘   └─────────────────────────────────┘
            OS Thread 1                           OS Thread 2

Why Isolates Prevent Data Races by Design

Data races occur when two threads read/write shared mutable memory concurrently without synchronization. Dart's isolate model makes this impossible at the language level: because each isolate has its own heap and objects cannot be directly referenced across isolate boundaries, there is literally no shared mutable memory to race on.

When you send a message from one isolate to another, the Dart VM performs a deep copy of the message object graph. The sender retains its own copy; the receiver gets a new copy. Neither can corrupt the other's state.

Staff vs Threads Comparison
Java/Kotlin threads share a heap. A List<Int> can be passed between threads by reference — both can read/write it simultaneously, requiring synchronized, volatile, AtomicInteger, etc. In Dart, the equivalent list would be deep-copied on send. The trade-off: Dart avoids entire classes of concurrency bugs but pays a copying cost. For large data (e.g., a 10MB decoded image), this copy is expensive — which is why TransferableTypedData exists for zero-copy byte transfer.

2. Creating Isolates

Isolate.spawn()

import 'dart:isolate';

// Entry point MUST be a top-level or static function.
// Closures cannot be entry points because they capture variables
// from the enclosing scope — which may not be serializable.
void workerEntryPoint(SendPort mainSendPort) {
  // This runs in the new isolate's context
  // It has its OWN event loop, its OWN heap
  mainSendPort.send('Hello from worker!');
}

Future<void> main() async {
  final receivePort = ReceivePort();

  // Spawn a new isolate.
  // Dart VM creates:
  //   1. A new OS thread
  //   2. A new Dart heap
  //   3. A new event loop on that thread
  //   4. Calls workerEntryPoint(receivePort.sendPort) in the new isolate
  final isolate = await Isolate.spawn(
    workerEntryPoint,
    receivePort.sendPort, // this SendPort is COPIED to the new isolate
  );

  final message = await receivePort.first; // wait for one message
  print(message); // 'Hello from worker!'

  receivePort.close();
  isolate.kill(); // send kill signal to the isolate
}

Why Entry Points Must Be Top-Level or Static

class MyClass {
  final String name;
  MyClass(this.name);

  void instanceMethod(SendPort port) {
    port.send(name); // captures 'this' — references heap of spawning isolate!
  }

  static void staticMethod(SendPort port) {
    port.send('static'); // no captured state from spawning isolate — safe
  }
}

// This would fail at runtime:
// Isolate.spawn(MyClass('test').instanceMethod, port); // BAD

// This works:
// Isolate.spawn(MyClass.staticMethod, port); // OK

// Top-level functions work:
void topLevel(SendPort port) => port.send('ok');
// Isolate.spawn(topLevel, port); // OK

Isolate.spawnUri()

// spawnUri: load Dart code from a URI (separate .dart file or compiled .dill)
// Useful for dynamic loading, plugin systems, large isolated modules.
// In Flutter release builds, spawnUri is NOT supported (AOT limitation).

final isolate = await Isolate.spawnUri(
  Uri.parse('package:myapp/workers/image_processor.dart'),
  ['arg1', 'arg2'],     // List<String> args (like main() args)
  receivePort.sendPort, // message passed to main()
);

3. SendPort and ReceivePort

Establishing Bidirectional Communication

import 'dart:isolate';

// ---- Worker isolate code ----
void workerMain(SendPort mainToWorker) {
  // Create our own ReceivePort to receive messages FROM main
  final workerReceivePort = ReceivePort();

  // Send our SendPort back to main so it can send us messages
  mainToWorker.send(workerReceivePort.sendPort);

  // Listen for work items
  workerReceivePort.listen((message) {
    if (message is Map<String, dynamic>) {
      final taskId = message['taskId'] as int;
      final data = message['data'] as String;
      final result = expensiveProcess(data);

      // Send result back using the SendPort main provided
      final replyPort = message['replyPort'] as SendPort;
      replyPort.send({'taskId': taskId, 'result': result});
    } else if (message == 'shutdown') {
      workerReceivePort.close();
    }
  });
}

String expensiveProcess(String data) => data.toUpperCase(); // simplified

// ---- Main isolate code ----
Future<void> main() async {
  final mainReceivePort = ReceivePort();

  // Spawn worker, giving it a way to send to us
  await Isolate.spawn(workerMain, mainReceivePort.sendPort);

  // First message from worker is its SendPort
  final workerSendPort = await mainReceivePort.first as SendPort;

  // Now we have bidirectional communication.
  // For each task, create a one-shot reply port:
  final replyPort = ReceivePort();
  workerSendPort.send({
    'taskId': 1,
    'data': 'hello world',
    'replyPort': replyPort.sendPort,
  });

  final response = await replyPort.first as Map<String, dynamic>;
  print('Task ${response['taskId']} result: ${response['result']}');
  // Output: Task 1 result: HELLO WORLD

  replyPort.close();
  workerSendPort.send('shutdown');
  mainReceivePort.close();
}

RawReceivePort: Lower Overhead

// RawReceivePort: no stream overhead, direct callback, slightly faster
// Use when message rate is very high and you want minimum overhead
final rawPort = RawReceivePort((message) {
  // message handler called directly — no Stream machinery
  processMessage(message);
});

final sendPort = rawPort.sendPort;
// ... pass sendPort to worker ...

// Close when done:
rawPort.close();

// ReceivePort wraps RawReceivePort with a Stream interface.
// For most use cases, ReceivePort is preferable (easier to use).
// For high-frequency message passing (e.g., audio/video frames), RawReceivePort
// reduces per-message overhead.

4. Message Serialization

The Dart VM uses a structured clone algorithm (similar to the browser's postMessage) to deep-copy messages between isolates. Understanding what is and is not serializable is critical.

TypeCan Send?Notes
null, bool, int, double, StringYesPrimitives — copied by value
List, Map, SetYesDeep copied recursively
SendPortYesPassed by reference — ports are handles, not objects
Uint8List and other typed dataYesDeep copied by default
TransferableTypedDataYesZero-copy transfer — original becomes unusable
Plain Dart objects (class Foo {...})Yes (Dart 2.15+)Deep copied via structured clone
Closures / functionsNoThrows ArgumentError at runtime
Native resources (file handles, sockets)NoNot transferable
CapabilityYesIsolate control capability
Dart 2.15 Object Sending
Before Dart 2.15, only a limited set of types could be sent between isolates. Dart 2.15 added support for sending arbitrary Dart objects via structured cloning — the VM traverses the object graph and deep-copies it into the target isolate's heap. This eliminated the need to manually serialize objects to JSON/maps before sending. However, closures and native handles still cannot be sent.

TransferableTypedData: Zero-Copy Transfer

import 'dart:typed_data';
import 'dart:isolate';

// Problem: sending a large Uint8List (e.g., 4K image = ~8MB) deep-copies it.
// For high-throughput scenarios (video frames, audio buffers), this is expensive.

// Solution: TransferableTypedData
// The bytes are transferred (ownership moved) rather than copied.
// The original TransferableTypedData becomes invalid after send.

void imageProcessor(SendPort resultPort) {
  final rawPort = RawReceivePort((message) {
    final transferable = message as TransferableTypedData;
    // Materialize into a Uint8List in THIS isolate's heap
    final bytes = transferable.materialize().asUint8List();
    // Process bytes...
    final processed = processImageBytes(bytes);
    // Transfer result back
    resultPort.send(TransferableTypedData.fromList([processed]));
  });
  // ... send rawPort.sendPort to main ...
}

// In main isolate:
final imageBytes = Uint8List(8 * 1024 * 1024); // 8MB image
// Fill imageBytes with pixel data...

// Transfer without copying:
final transferable = TransferableTypedData.fromList([imageBytes]);
// After this point, imageBytes CANNOT be used (it has been detached)
workerSendPort.send(transferable);

Uint8List processImageBytes(Uint8List bytes) => bytes; // placeholder

5. Isolate Groups (Dart 2.15+)

Isolate groups are a Dart 2.15 optimization that significantly reduces the cost of spawning new isolates within the same application.

  Before Dart 2.15 (no isolate groups):
  ┌─────────────┐        ┌─────────────┐
  │  Isolate A  │        │  Isolate B  │
  │  + full     │        │  + full     │
  │    code     │        │    code     │
  │    copy     │        │    copy     │
  └─────────────┘        └─────────────┘
  Each isolate gets its own copy of ALL compiled code in memory.
  Spawning: ~100ms + significant memory overhead.

  Dart 2.15+ (isolate groups):
  ┌──────────────────────────────────────────────┐
  │              ISOLATE GROUP                   │
  │  ┌──────────────────────────────────────┐    │
  │  │  Shared read-only memory pages       │    │
  │  │  (AOT compiled code, constants)      │    │
  │  │  OS-level copy-on-write sharing      │    │
  │  └──────────────────────────────────────┘    │
  │                                              │
  │  ┌──────────┐   ┌──────────┐  ┌──────────┐  │
  │  │Isolate A │   │Isolate B │  │Isolate C │  │
  │  │heap only │   │heap only │  │heap only │  │
  │  └──────────┘   └──────────┘  └──────────┘  │
  │  Separate heaps — memory isolation preserved  │
  └──────────────────────────────────────────────┘
  Spawning: ~1ms, much less memory per isolate.

Key properties of isolate groups:

  • All isolates spawned via Isolate.spawn() in Flutter are in the same group as the main isolate.
  • Code (compiled instructions) and read-only data are shared via OS-level memory page sharing.
  • Heaps remain completely separate — no shared mutable objects.
  • Isolate.spawnUri() creates a new isolate group (separate compiled code).
  • Spawn time dropped from ~100ms to ~1ms in typical Flutter apps.

6. Error Handling in Isolates

import 'dart:isolate';
import 'dart:async';

Future<void> spawnWithErrorHandling() async {
  final receivePort = ReceivePort();
  final errorPort = RawReceivePort((message) {
    // message is a List: [errorString, stackTraceString]
    final error = message[0] as String;
    final stack = message[1] as String?;
    print('Worker isolate error: $error\n$stack');
    // Could also complete a Completer with error here
  });
  final exitPort = RawReceivePort((_) {
    print('Worker isolate has exited');
    errorPort.close();
    exitPort.close();
  });

  await Isolate.spawn(
    workerFunction,
    receivePort.sendPort,
    onError: errorPort.sendPort,    // uncaught errors sent here
    onExit: exitPort.sendPort,      // isolate exit notification sent here
    errorsAreFatal: true,           // isolate dies on uncaught error (default: true)
    debugName: 'MyWorkerIsolate',   // name for debugging/profiling
  );
}

void workerFunction(SendPort port) {
  // If this throws an uncaught exception:
  // 1. The error is sent to errorPort as ['error message', 'stack trace']
  // 2. If errorsAreFatal: true, the isolate terminates
  // 3. exitPort receives a notification
  throw Exception('Worker crashed!');
}

Isolate.addErrorListener and addOnExitListener

// After spawning, you can attach listeners dynamically:
final isolate = await Isolate.spawn(workerFn, sendPort);

final errorPort = ReceivePort();
isolate.addErrorListener(errorPort.sendPort);

final exitPort = ReceivePort();
isolate.addOnExitListener(exitPort.sendPort);

// Listen:
errorPort.listen((message) {
  print('Error: ${message[0]}');
});

exitPort.listen((_) {
  errorPort.close();
  exitPort.close();
  print('Isolate exited');
});

// Pause/resume the isolate (useful for debugging):
final capability = isolate.pause(); // returns a Capability
// ... debug ...
isolate.resume(capability);

// Kill the isolate (immediate or after current task):
isolate.kill(priority: Isolate.immediate); // forceful
isolate.kill(priority: Isolate.beforeNextEvent); // graceful

7. Flutter's Main Isolate and Background Isolates

The UI Isolate Constraint

Flutter's widget tree, rendering pipeline, and all Flutter framework code run exclusively on the main (UI) isolate. Background isolates spawned via Isolate.spawn() or compute() cannot access:

  • Flutter's widget tree or BuildContext
  • Platform channels (before Flutter 3.7)
  • MethodChannel, EventChannel
  • RootIsolateToken-gated APIs
  • Any singleton initialized in the main isolate (e.g., SharedPreferences instance)

BackgroundIsolateBinaryMessenger (Flutter 3.7+)

import 'package:flutter/services.dart';

// Before Flutter 3.7: background isolates could NOT use platform channels.
// Flutter 3.7 introduced BackgroundIsolateBinaryMessenger.

// In main isolate, before spawning:
final token = RootIsolateToken.instance!;

// Pass token to background isolate:
await Isolate.spawn(backgroundWork, token);

// In background isolate:
void backgroundWork(RootIsolateToken token) async {
  // Register the binary messenger for this isolate
  BackgroundIsolateBinaryMessenger.ensureInitialized(token);

  // NOW platform channels work in this isolate!
  const channel = MethodChannel('com.example/native');
  final result = await channel.invokeMethod('heavyNativeWork');
  print('Native result from background: $result');

  // Use cases:
  // - SQLite operations via sqflite in background
  // - File system operations via path_provider in background
  // - Bluetooth in background (with plugin support)
}
Interview Tip
A common L5+ interview question is: "Why can't you access Flutter widgets from a background isolate?" The answer demonstrates understanding of the memory model: each isolate has its own heap. The widget tree objects live on the UI isolate's heap. A background isolate cannot hold a reference to them — it would be a cross-heap pointer, which the Dart VM does not allow. The only way to interact with the UI from a background isolate is via message passing (SendPort/ReceivePort), which copies data across the boundary.

Practical Isolate Communication Pattern: Worker with Bidirectional Protocol

// A production-quality worker isolate implementation
import 'dart:isolate';
import 'dart:async';

// Message types for type-safe protocol
sealed class WorkerMessage {}
class WorkRequest extends WorkerMessage {
  final int id;
  final dynamic payload;
  WorkRequest(this.id, this.payload);
}
class ShutdownRequest extends WorkerMessage {}

sealed class WorkerResponse {}
class WorkResult extends WorkerResponse {
  final int requestId;
  final dynamic result;
  WorkResult(this.requestId, this.result);
}
class WorkError extends WorkerResponse {
  final int requestId;
  final String error;
  WorkError(this.requestId, this.error);
}

// Worker entry point
void _workerEntry(SendPort mainPort) {
  final workerPort = ReceivePort();
  mainPort.send(workerPort.sendPort); // handshake

  workerPort.listen((message) {
    if (message is ShutdownRequest) {
      workerPort.close();
      return;
    }
    if (message is WorkRequest) {
      try {
        final result = _doWork(message.payload);
        mainPort.send(WorkResult(message.id, result));
      } catch (e) {
        mainPort.send(WorkError(message.id, e.toString()));
      }
    }
  });
}

dynamic _doWork(dynamic payload) {
  // CPU-intensive work here
  return payload.toString().toUpperCase();
}

// Main-side wrapper
class WorkerIsolate {
  late final SendPort _sendPort;
  late final ReceivePort _receivePort;
  late final Isolate _isolate;
  final Map<int, Completer<dynamic>> _pending = {};
  int _nextId = 0;

  Future<void> init() async {
    _receivePort = ReceivePort();
    _isolate = await Isolate.spawn(_workerEntry, _receivePort.sendPort);

    // First message is the worker's SendPort
    _sendPort = await _receivePort.first as SendPort;

    // Handle subsequent messages
    _receivePort.skip(1).listen((message) {
      if (message is WorkResult) {
        _pending.remove(message.requestId)?.complete(message.result);
      } else if (message is WorkError) {
        _pending.remove(message.requestId)
            ?.completeError(Exception(message.error));
      }
    });
  }

  Future<dynamic> sendWork(dynamic payload) {
    final id = _nextId++;
    final completer = Completer<dynamic>();
    _pending[id] = completer;
    _sendPort.send(WorkRequest(id, payload));
    return completer.future;
  }

  Future<void> dispose() async {
    _sendPort.send(ShutdownRequest());
    _receivePort.close();
    _isolate.kill(priority: Isolate.beforeNextEvent);
  }
}
