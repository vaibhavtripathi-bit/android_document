1. Future Internals

A Future<T> represents a value (or error) that may not yet be available. Internally, Dart's dart:async implements this through a _Future<T> class that manages state, listener chains, and completion propagation.

Future States

  Future<T> lifecycle:

  ┌──────────────┐
  │  INCOMPLETE  │  ← initial state, no value yet
  └──────┬───────┘
         │
         ├──── complete(value) ────────▶ ┌──────────────────────┐
         │                               │  COMPLETED WITH VALUE │
         │                               └──────────────────────┘
         │
         └──── completeError(e, st) ───▶ ┌──────────────────────┐
                                         │  COMPLETED WITH ERROR │
                                         └──────────────────────┘

  Once completed, a Future's state never changes.
  Listeners registered AFTER completion are scheduled immediately as microtasks.

_Future Listener Chain

When you call future.then(onValue, onError: onError), Dart does NOT execute the callback immediately. Instead:

  1. If the Future is incomplete: the callback is stored as a listener in a linked list on the _Future object.
  2. If the Future is already complete: the callback is scheduled as a microtask immediately.

When the Future completes, it iterates its listener list and schedules each callback as a microtask. This is why .then() chains execute in order but asynchronously.

// Internally, _Future maintains a linked list of _FutureListener objects.
// Each then/catchError/whenComplete call appends to this list.
// On completion, all listeners are scheduled as microtasks in order.

final f = Future<int>.delayed(Duration(seconds: 1), () => 42);

// These three listeners are all registered before the Future completes.
// When it completes with 42, all three are scheduled on microtask queue.
f.then((v) => print('listener 1: $v'));
f.then((v) => print('listener 2: $v'));
f.then((v) => print('listener 3: $v'));
// Output after 1s: listener 1: 42, listener 2: 42, listener 3: 42
// (in microtask order)

FutureOr<T>: Sync vs Async Flexibility

FutureOr<T> is a union type that can hold either a T (synchronous value) or a Future<T>. It exists to allow APIs to be both synchronous and asynchronous without forcing callers to always deal with Futures.

// FutureOr allows a function to be synchronous OR asynchronous
FutureOr<int> getValue(bool cached) {
  if (cached) return 42;           // Returns int synchronously
  return Future.value(42);         // Returns Future<int> asynchronously
}

// Consumers handle both cases:
final result = getValue(true);
if (result is Future<int>) {
  result.then((v) => print(v));
} else {
  print(result); // synchronous path
}

// More commonly used in then() callbacks:
// then<R>(FutureOr<R> Function(T) onValue)
// The callback may return a plain R or a Future<R>
future.then((data) {
  if (data.isInCache) return data.cachedValue;  // R directly
  return fetchFromNetwork(data.key);             // Future<R>
});
Staff-Level Detail
FutureOr<T> is resolved at runtime, not compile time. When Dart's runtime encounters a FutureOr<T> value: if it's a plain T, it wraps it in Future.value(t) before continuing the chain. This means even a synchronous then callback will produce an asynchronous result propagation — you cannot use FutureOr to make then chains synchronous.

2. async/await Compiler Transformation

The Dart compiler transforms async functions into state machines. This is not just a concept — understanding the transformation helps diagnose subtle bugs and performance issues.

// Original async function:
Future<String> processUser(int id) async {
  final user = await fetchUser(id);      // suspension point 1
  final profile = await fetchProfile(user.profileId);  // suspension point 2
  return '${user.name}: ${profile.bio}';
}

// Approximate compiler output (conceptual — actual codegen differs):
Future<String> processUser(int id) {
  final _completer = _AsyncAwaitCompleter<String>();
  var _state = 0;
  late int _id;
  late User _user;
  late Profile _profile;

  void _body([dynamic _result, dynamic _error]) {
    try {
      switch (_state) {
        case 0: // initial
          _state = 1;
          final f1 = fetchUser(id);
          if (f1 is Future) {
            f1.then(_body, onError: _body_error);
            return;
          }
          _result = f1; // synchronous path
          continue state1;

        state1:
        case 1: // after first await
          _user = _result as User;
          _state = 2;
          final f2 = fetchProfile(_user.profileId);
          if (f2 is Future) {
            f2.then(_body, onError: _body_error);
            return;
          }
          _result = f2;
          continue state2;

        state2:
        case 2: // after second await
          _profile = _result as Profile;
          _completer.complete('${_user.name}: ${_profile.bio}');
          return;
      }
    } catch (e, st) {
      _completer.completeError(e, st);
    }
  }

  _body();
  return _completer.future;
}

Why You Can't Cancel a Future Mid-Execution

Once a Future is in flight (the async function body has started executing), there is no mechanism to interrupt it. The state machine advances whenever its awaited Future completes. There is no CancellationToken like in C# or Job.cancel() like in Kotlin coroutines.

The function body will run to completion or throw. You can ignore the result (by not awaiting it or dropping the Future reference), but the computation continues consuming resources. This is a fundamental limitation of Dart's Future model.

3. Completer Pattern

A Completer<T> lets you create and control a Future manually — separating the creation of the Future from its completion. This is the primary bridge between callback-based APIs and Future-based APIs.

// Basic Completer usage
Future<String> fetchWithTimeout(String url) {
  final completer = Completer<String>();

  // Callback-based HTTP library
  httpClient.get(url, onSuccess: (response) {
    if (!completer.isCompleted) {
      completer.complete(response.body);
    }
  }, onError: (error) {
    if (!completer.isCompleted) {
      completer.completeError(error);
    }
  });

  // Timeout: complete with error after 5 seconds
  Timer(Duration(seconds: 5), () {
    if (!completer.isCompleted) {
      completer.completeError(TimeoutException('Request timed out', Duration(seconds: 5)));
    }
  });

  return completer.future;
}

Completer.sync() — Synchronous Completion

// Regular Completer: listeners are scheduled as MICROTASKS
final regular = Completer<int>();
regular.future.then((v) => print('regular listener: $v'));
regular.complete(1);
print('after regular.complete'); // prints BEFORE 'regular listener: 1'
// Output: "after regular.complete", then "regular listener: 1"

// Completer.sync(): listeners are called SYNCHRONOUSLY during complete()
final sync = Completer<int>.sync();
sync.future.then((v) => print('sync listener: $v'));
sync.complete(1);
print('after sync.complete'); // prints AFTER 'sync listener: 1'
// Output: "sync listener: 1", then "after sync.complete"
Warning: Completer.sync() Pitfall
Completer.sync() calls listeners synchronously within the complete() call. If a listener throws, the exception propagates into the code that called complete(), not into the Future chain. Use with extreme caution — it violates the usual async contract and can cause unexpected re-entrant code execution. It is primarily used in the Dart SDK internals for performance.

Request Deduplication with Completer

class ApiService {
  final Map<String, Completer<dynamic>> _pendingRequests = {};

  Future<T> deduplicated<T>(String key, Future<T> Function() request) async {
    // If a request for this key is already in flight, return the same Future
    if (_pendingRequests.containsKey(key)) {
      return _pendingRequests[key]!.future as Future<T>;
    }

    final completer = Completer<T>();
    _pendingRequests[key] = completer;

    try {
      final result = await request();
      completer.complete(result);
      return result;
    } catch (e, st) {
      completer.completeError(e, st);
      rethrow;
    } finally {
      _pendingRequests.remove(key);
    }
  }
}

// Usage: simultaneous calls for same key reuse the same Future
final api = ApiService();
final f1 = api.deduplicated('user/123', () => fetchUser(123));
final f2 = api.deduplicated('user/123', () => fetchUser(123)); // same Future!
// Only ONE network request is made; both futures complete with same result

4. Error Propagation in Futures

The Future Error Chain

// Errors propagate through .then() chains until caught
Future.error(Exception('boom'))
  .then((v) => print('never runs'))   // skipped — Future is in error state
  .then((v) => print('also skipped'))
  .catchError((e) {
    print('caught: $e');
    return 'recovered'; // returns a non-error Future — chain continues normally
  })
  .then((v) => print('after recovery: $v')); // runs with 'recovered'

// catchError with type filtering
someAsyncOperation()
  .catchError(
    (e) => handleNetworkError(e),
    test: (e) => e is NetworkException,  // only catches NetworkException
  )
  .catchError(
    (e) => handleOtherError(e),
    // catches everything else
  );

whenComplete: Finally for Futures

Future<void> withCleanup() async {
  try {
    return someRiskyOperation();
  } finally {
    cleanup(); // Dart try/finally works with async, but note:
               // if cleanup() is async and you don't await it... problems
  }
}

// The Future equivalent:
someRiskyOperation()
  .whenComplete(() {
    // Runs whether Future completed with value OR error
    // If this callback throws, the original value/error is REPLACED
    // by the new error from whenComplete's callback
    cleanup();
  });

// whenComplete with async callback: return value is ignored!
// If you need to await in whenComplete, use .then():
someRiskyOperation()
  .then(
    (value) async {
      await asyncCleanup();
      return value; // must return the original value
    },
    onError: (e, st) async {
      await asyncCleanup();
      return Future.error(e, st); // rethrow
    },
  );

Unhandled Future Errors

In Dart, if a Future completes with an error and no error handler is attached within the same event loop turn, the error becomes "unhandled." What happens then depends on the Zone:

// Unhandled Future error flow:
// 1. Future completes with error
// 2. No then/catchError catches it
// 3. Future is garbage collected (or next event loop turn passes)
// 4. Zone.current.handleUncaughtError() is called
// 5. In Flutter: crashes the app in debug, logs in release (by default)

// How Flutter's binding catches uncaught async errors:
// FlutterError.onError handles framework errors (widget build exceptions etc.)
// PlatformDispatcher.instance.onError handles isolate-level uncaught errors

// Best practice: always handle Future errors
Future<void> safeFireAndForget(Future<void> operation) {
  // Explicitly mark as intentional fire-and-forget
  // (unawaited() from package:meta is documentation only, no runtime effect)
  unawaited(operation.catchError((e, st) {
    // Log but don't crash
    log('Background operation failed: $e\n$st');
  }));
}
Staff-Level Detail
The timing of "unhandled future error" detection changed between Dart versions. In modern Dart, a Future error is considered "unhandled" if no error handler is attached before the Future's _Future._reportError() is called — which happens when the Future is completed with an error and the listeners are being notified. If you attach a catchError synchronously right after creating/awaiting a Future (in the same synchronous sequence before any await), the error is handled. But if you attach it asynchronously (after an await), the window for "unhandled" may have already passed.

5. Common Patterns and Anti-Patterns

Future.wait, Future.any, Future.forEach, Future.doWhile

// Future.wait: run multiple futures concurrently, collect all results
// Fails fast: if any future fails, Future.wait completes with error immediately
// (but other futures continue running — no cancellation)
final results = await Future.wait([
  fetchUser(1),
  fetchUser(2),
  fetchUser(3),
], eagerError: true); // default: true — fail on first error

// Future.wait with eagerError: false — wait for ALL futures, then throw
// if any failed (useful for "report all errors" scenarios)
final results2 = await Future.wait(
  [fetchUser(1), fetchUser(2), fetchUser(3)],
  eagerError: false,  // collect all results, throw if any failed
);

// Future.any: completes with whichever future completes first
// Other futures continue running (no cancellation)
final winner = await Future.any([
  fetchFromPrimaryServer(),
  fetchFromFallbackServer(),
]);

// Future.forEach: sequential, not parallel — one at a time
await Future.forEach([1, 2, 3], (id) async {
  await processItem(id); // processes 1, then 2, then 3 in sequence
});

// For parallel processing with concurrency limit:
// Use a semaphore pattern or package:pool

Chaining vs await

// Chaining: good for simple transformations without try/catch
final displayName = await fetchUser(id)
    .then((user) => user.displayName)
    .catchError((_) => 'Unknown User');

// await: better for complex logic, multiple operations, try/catch
Future<String> getDisplayName(int id) async {
  try {
    final user = await fetchUser(id);
    return user.displayName;
  } catch (_) {
    return 'Unknown User';
  }
}

// Anti-pattern: nested async in then callbacks
// This creates confusing error propagation
fetchUser(id).then((user) async {
  // The Future returned by this async callback IS used by then,
  // but errors inside may not propagate as expected
  final profile = await fetchProfile(user.id); // if this throws...
  return profile.bio;
}).catchError((e) => print('caught? maybe not what you expect: $e'));

// Prefer: flat async/await

Avoiding Unawaited Futures

import 'package:flutter/foundation.dart'; // for unawaited

// WRONG: implicit fire-and-forget, error is silently swallowed
void badExample() {
  fetchData(); // returns Future<void> — not awaited, error is lost!
}

// BETTER: explicit about intent, errors are handled
void goodExample() {
  unawaited(fetchData().catchError((e) => reportError(e)));
}

// BEST in most cases: just await it
Future<void> bestExample() async {
  await fetchData(); // error propagates to caller
}

6. Cancellation Patterns

Dart does not have built-in Future cancellation. This is a deliberate design decision: since Futures represent one-shot computations, and Dart lacks structured concurrency, adding cancellation would require significant complexity (propagating tokens, handling partial results). Several patterns fill this gap.

Pattern 1: Cancellation Flag

class SearchController {
  bool _cancelled = false;

  void cancel() {
    _cancelled = true;
  }

  Future<List<Result>> search(String query) async {
    _cancelled = false;

    final networkResults = await fetchFromNetwork(query);
    if (_cancelled) return []; // early exit

    final processed = await processResults(networkResults);
    if (_cancelled) return []; // check again after each await

    return processed;
  }
}

Pattern 2: CancelableOperation (package:async)

import 'package:async/async.dart';

class DebounceSearch {
  CancelableOperation<List<Result>>? _currentSearch;

  Future<void> onSearchChanged(String query) async {
    // Cancel the previous search operation if still running
    await _currentSearch?.cancel();

    _currentSearch = CancelableOperation.fromFuture(
      _performSearch(query),
      onCancel: () => print('Search for "$query" cancelled'),
    );

    try {
      final results = await _currentSearch!.valueOrCancellation();
      if (results != null) {
        // Not cancelled — update UI
        _updateResults(results);
      }
    } catch (e) {
      _handleError(e);
    }
  }
}
Interview Tip
When asked about Future cancellation in Dart, mention: (1) no built-in support by design, (2) the cancellation flag pattern for simple cases, (3) CancelableOperation from package:async for structured cancellation, and (4) Streams for scenarios where you'd normally want a "stream of cancellable operations" — subscribing/unsubscribing to a Stream is a form of cancellation.

Pattern 3: Superseding with Generation Counter

// Common Flutter pattern: cancel stale async operations by generation
class ProductViewModel {
  int _loadGeneration = 0;

  Future<void> loadProduct(int productId) async {
    final generation = ++_loadGeneration;

    try {
      final product = await _repository.fetchProduct(productId);

      // Only apply result if no newer load has started
      if (generation == _loadGeneration) {
        setState(() => _product = product);
      }
    } catch (e) {
      if (generation == _loadGeneration) {
        setState(() => _error = e.toString());
      }
    }
  }
}

7. Advanced: Future.sync and Error Zones

// Future.sync(() => expr): runs expr synchronously
// If expr throws synchronously, returns a completed-with-error Future
// instead of throwing. Useful for ensuring function always returns a Future.

Future<int> safeCompute(int x) {
  return Future.sync(() {
    if (x < 0) throw ArgumentError('x must be non-negative');
    return x * x; // Synchronous computation wrapped in Future
  });
}

// Without Future.sync, a synchronous throw would escape the Future chain:
Future<int> unsafe(int x) {
  if (x < 0) throw ArgumentError('x must be non-negative'); // throws, not returned!
  return Future.value(x * x);
}

// With Future.sync, errors are always captured in the Future:
await safeCompute(-1).catchError((e) => print('caught: $e')); // works
// vs
try {
  await unsafe(-1); // synchronous throw escapes -- requires try/catch HERE
} catch (e) {
  print('caught: $e');
}

Async Error Zones in Tests

// In Flutter tests, you often need to test async error handling:
testWidgets('shows error state on fetch failure', (tester) async {
  // Use expectAsync to verify async error callbacks are called
  final mockFetch = MockFetch();
  when(mockFetch.fetchUser(any)).thenAnswer(
    (_) => Future.error(NetworkException('offline')),
  );

  await tester.pumpWidget(UserWidget(fetch: mockFetch));
  await tester.pump(); // trigger async gap

  expect(find.text('Error: offline'), findsOneWidget);
});

// Zone-based test error capture:
test('unhandled future error is caught', () async {
  Object? caughtError;

  await runZonedGuarded(
    () async {
      unawaited(Future.error(Exception('test error')));
      // yield to let the error propagate
      await Future.delayed(Duration.zero);
    },
    (error, _) {
      caughtError = error;
    },
  );

  expect(caughtError, isA<Exception>());
});
API Queue Use Case
Future.value(x) Microtask Wrap a sync value as a Future
Future.microtask(fn) Microtask Defer work to after current sync code
Future(fn) Event queue Defer work to after current microtasks AND events
Future.delayed(d, fn) Event queue (via Timer) Time-delayed work
Future.sync(fn) Synchronous, then microtask Wrap sync-or-async computation safely
Completer.complete() Microtask (listeners) Manually complete a Future
Completer.sync().complete() Synchronous (listeners) Synchronous completion (advanced, dangerous)
