Standard Questions

Standard — Event Loop
What is Dart's event loop? How do microtasks differ from regular event queue tasks?
Dart is single-threaded with an event loop that processes two queues: (1) The microtask queue — high-priority, processed to exhaustion before any event queue item runs. Dart SDK internals (Future completion callbacks, scheduleMicrotask()) use this. (2) The event queue — lower priority, processes one item per event loop iteration. UI events, timers, I/O completions, and platform channel callbacks land here. The loop: run current Dart code synchronously → drain microtask queue completely → pick one item from event queue → repeat. This means microtasks can starve the event queue if you continuously schedule more microtasks — a pathological scenario but important to understand.
Draw the loop: [Sync code] → [Microtask queue: drain all] → [Event queue: one item] → repeat. Interviewers often ask this first to filter candidates who haven't actually studied concurrency.
Standard — Execution Order Puzzle
What does this code print, and in what order? 
void main() {
  print('1');
  Future(() => print('2'));
  Future.microtask(() => print('3'));
  Future.value(42).then((_) => print('4'));
  print('5');
}
Output: 1, 5, 3, 4, 2

Reasoning: (1) print('1') — synchronous, runs immediately. (2) Future(() => print('2')) — schedules a callback on the event queue. (3) Future.microtask(() => print('3')) — schedules on the microtask queue. (4) Future.value(42).then((_) => print('4')) — Future.value() is already-completed; .then() on a completed future schedules its callback as a microtask. (5) print('5') — synchronous, runs immediately. Then synchronous code finishes. Microtask queue drains: '3' then '4'. Then event queue: '2'.
This exact pattern (or a variation) appears in ~70% of Dart concurrency interviews. The key insight: .then() on an already-resolved Future schedules as a microtask, not an event queue item.
Standard — Future vs Stream
When would you use a Stream instead of a Future?
Use a Future when the operation produces exactly one result at some point in the future (an HTTP response, a database read, a computed value). Use a Stream when the source produces zero or more values over time: real-time data (WebSocket messages, sensor readings, file upload progress), UI events (button taps, scroll position), or state changes that many listeners need (BLoC state, Riverpod stream providers). Also use Stream when you want composable transformations (map, where, switchMap via RxDart) applied to a sequence of events. Streams can also represent a single async result that may fail (a "cold" stream that emits one item then closes) — but Future is simpler and preferred for that use case.
Standard — async/await
What does the async keyword do to a function? What does await do?
The async keyword transforms the function into one that always returns a Future (even if you return a non-Future value — it's wrapped). It enables the use of await inside the function. The await keyword pauses execution of the current async function until the awaited Future completes, then resumes with the resolved value. Crucially, await does not block the thread — it suspends the current async frame and yields control back to the event loop, allowing other callbacks to run. When the Future completes, the async function's continuation is scheduled as a microtask. This is cooperative concurrency — the function voluntarily yields at each await point.
Standard — Isolate vs Thread
How do Dart isolates differ from threads? Why doesn't Dart have shared-memory threads?
Threads share memory — two threads can read and write the same variable, requiring synchronization (locks, mutexes, atomic operations) to prevent data races. Dart isolates have separate heaps — no shared mutable state. Communication is only via message passing (SendPort/ReceivePort). This eliminates data races by design: if you can't share memory, you can't have a race on shared memory. Dart chose this model to make concurrent code safer and simpler to reason about — the concurrency bugs that plague multithreaded code (deadlocks, race conditions, priority inversion) are simply impossible in Dart. The cost: data must be serialized (copied) to cross isolate boundaries, making large data transfers expensive. Dart 2.15 mitigated this with isolate groups that allow zero-copy transfer of immutable objects.
Standard — compute() Usage
When should you use compute() in a Flutter app? What are its limitations?
Use compute() for CPU-bound work that would block the UI thread for more than ~4ms (to stay safely within the 16.6ms frame budget). Common cases: parsing large JSON responses, image processing, cryptographic operations, in-memory data crunching. Limitations: (1) The callback must be a top-level or static function — cannot be a closure or instance method that captures this. (2) Accepts only one argument — pass a record or class if multiple values are needed. (3) Creates a new isolate per call — ~1–5ms overhead. Not suitable for high-frequency calls (use a persistent worker isolate instead). (4) The result must be serializable (copy-transferable). (5) Cannot call platform channels from within compute() unless using BackgroundIsolateBinaryMessenger (Flutter 3.7+).
Standard — BuildContext After Async
Why is it unsafe to use BuildContext after an async gap? How do you fix it?
After an await, the widget may have been unmounted from the tree — the user navigated away, the parent was rebuilt and replaced the widget, or the widget was disposed. Using a context that refers to an unmounted element throws FlutterError: Looking up a deactivated widget's ancestor is unsafe. Fix: check if (!mounted) return; after every await before using context in a StatefulWidget. Alternatively, capture what you need from context synchronously before the first await (e.g., final navigator = Navigator.of(context);) — Navigator and other objects captured this way remain valid even after the widget unmounts. In StatelessWidget, there's no mounted — you must capture before the await. The lint rule use_build_context_synchronously flags context usages after await.
Show the two patterns: (1) mounted check, (2) capture-before-await. Then mention the lint rule. Three concrete actions shows depth.
Standard — Stream Subscription
What happens if you don't cancel a StreamSubscription? How do you manage multiple subscriptions?
An uncancelled StreamSubscription keeps its listener alive indefinitely — even after the widget/object that created it is disposed. This causes: (1) Memory leaks — the listener closure captures references to widget state, preventing GC. (2) Logic errors — the listener continues to call setState() or other callbacks on a disposed state object, causing "setState after dispose" errors. Fix: always store subscriptions and cancel in dispose(). For multiple subscriptions, use a list or CompositeSubscription and cancel all in dispose(). RxDart provides CompositeSubscription. The lint rule cancel_subscriptions flags subscriptions that are never cancelled.
Standard — SingleSubscription vs Broadcast
What is the difference between a single-subscription stream and a broadcast stream?
A single-subscription stream can have at most one listener at a time. It buffers events until a listener subscribes, then delivers all buffered events in order. Subscribing twice throws an error. Most streams from dart:io (file reads, HTTP bodies, stdin) are single-subscription. A broadcast stream can have multiple listeners simultaneously. Events are not buffered for late subscribers — a listener that subscribes after an event was emitted misses that event. StreamController.broadcast() creates one. StreamControllers from BLoC/Cubit use broadcast streams (since multiple UI widgets may listen). Convert a single-subscription to broadcast: stream.asBroadcastStream(). Use broadcast for: state streams, event buses, UI event streams. Use single-subscription for: one-time data pipelines (reading a file), HTTP response bodies.
Standard — Debounce vs Throttle
What is the difference between debounce and throttle? Give a real Flutter example for each.
Debounce: delays execution until a burst of calls has stopped for a specified duration. The action fires after the last call in a burst. Example: search-as-you-type — don't fire a network request on every keystroke; wait until the user stops typing for 400ms. Implementation: cancel and restart a Timer on each call; the action runs only when the timer isn't cancelled. Throttle: allows execution at most once per time period; ignores intermediate calls during the cooldown. The action fires immediately then is suppressed for the duration. Example: scroll analytics — log scroll position at most once every 200ms, even if the scroll event fires 60 times per second. Implementation: a flag set to true when action fires, reset to false after the cooldown via Future.delayed. RxDart provides debounceTime() and throttleTime() as stream operators for stream-based UI events.

Deep-Dive Questions

Deep Dive — Microtask Starvation
Can microtasks starve the event queue? Show a scenario and explain the consequence in Flutter.
Yes. If a microtask schedules another microtask, which schedules another, the event queue never gets a turn — the UI thread hangs. Example: 
void infiniteMicrotasks() {
  scheduleMicrotask(() {
    // This creates an infinite microtask chain
    infiniteMicrotasks(); // Schedules another microtask immediately
  });
}
In Flutter: the raster thread cannot receive new layer trees (because the UI thread is spinning microtasks), frame deadlines are missed, and the app appears frozen. The event queue processes vsync signals, timer callbacks, and platform channel responses — all starved. In practice, this is rare but can happen if a recursive Future chain doesn't yield to the event loop. The fix: use Future(() => nextStep()) (event queue) instead of Future.microtask(() => nextStep()) (microtask queue) in recursive async patterns.
This is a gotcha question. Very few candidates know the starvation implication. Nail it and you signal deep understanding of the event loop model.
Deep Dive — Future Ordering Puzzle
What prints, and why? 
Future<void> main() async {
  print('A');
  await Future(() => print('B'));
  print('C');
  await Future.microtask(() => print('D'));
  print('E');
}
Output: A, B, C, D, E — all in order. Here's why: (1) print('A') — synchronous. (2) await Future(() => print('B')) — schedules 'B' on the event queue; the await suspends main. Event loop runs 'B', then resumes main. (3) print('C') — synchronous (continuation after await). (4) await Future.microtask(() => print('D')) — schedules 'D' as a microtask; await suspends main. Microtask queue drains — 'D' runs; main resumes. (5) print('E') — synchronous. Key insight: await always suspends the current function, even if the Future resolves quickly — the continuation is scheduled as a microtask, not run synchronously. So the ordering here is fully sequential because each await waits for its Future before proceeding.
Deep Dive — Isolate Group Sharing
What changed in Dart 2.15 with isolate groups, and how does it affect performance of isolate communication?
Before Dart 2.15, all data sent between isolates was deep-copied via serialization — sending a 10MB Uint8List required copying all 10MB. This made isolate communication expensive for large data, limiting their use for scenarios like image processing pipelines where you'd need to pass large pixel buffers back and forth. Dart 2.15 introduced isolate groups: isolates spawned within the same group (using Isolate.spawn() with the default group) share the same heap metadata, allowing the VM to pass immutable objects by reference (zero-copy). Large immutable data (frozen Lists, compiled RegExps, large Strings) can now cross isolate boundaries without copying. Mutable objects still require copying. Isolate.run() (Dart 2.19) uses isolate groups internally. The practical impact: JSON parsing in a background isolate and returning the (immutable) parsed result is now much faster for large payloads. Flutter's compute() also benefits from this since Flutter 3.7.
Deep Dive — Stream Backpressure
What is stream backpressure in Dart? How do you handle a fast producer with a slow consumer?
Backpressure occurs when a producer emits events faster than the consumer can process them, causing unbounded buffer growth and eventual OOM or lost events. Dart's single-subscription streams buffer events by default — a fast producer filling the buffer before a slow consumer processes will grow memory unboundedly. Strategies: (1) bufferCount / bufferTime (RxDart): batch events and deliver as lists — consumer processes fewer, larger batches. (2) throttleTime / debounceTime (RxDart): discard intermediate events — appropriate when only the latest value matters (UI state updates, scroll position). (3) switchMap: for each new event, cancel the previous async operation and start fresh — appropriate for search queries where only the latest matters. (4) StreamTransformer with custom backpressure: implement a sliding window or token bucket. (5) Pause/resume: subscription.pause() signals the producer to stop emitting (works only for streams that respect pausing, like file reads). For isolate-to-isolate communication, use flow control by having the consumer send acknowledgment messages before requesting the next item.
Deep Dive — Zone Error Handling
How do Zones work in Dart? How do you catch all unhandled async errors in a Flutter app?
Zones are execution contexts that intercept async operations. Every Dart program runs in a root zone. You can create child zones with runZonedGuarded() that intercept uncaught errors, schedule microtasks, create timers, and handle print(). The onError handler in runZonedGuarded catches all unhandled async errors within that zone — including errors from unawaited Futures that would otherwise be lost. Flutter app error handling pattern: 
void main() {
  // Flutter framework errors (widget build exceptions, etc.)
  FlutterError.onError = (details) {
    FirebaseCrashlytics.instance.recordFlutterFatalError(details);
  };

  // Platform dispatcher errors (errors on platform channels, etc.)
  PlatformDispatcher.instance.onError = (error, stack) {
    FirebaseCrashlytics.instance.recordError(error, stack, fatal: true);
    return true;
  };

  // All other Dart async errors
  runZonedGuarded(() {
    runApp(const MyApp());
  }, (error, stack) {
    FirebaseCrashlytics.instance.recordError(error, stack);
  });
}
Zones also enable: dependency injection of timing (test zones that control Timer), logging of all async operations, and custom error reporting per subsystem.
At L6, you're expected to own error observability for your Flutter app. Showing the three-layer error catch (FlutterError.onError + PlatformDispatcher.onError + runZonedGuarded) demonstrates production engineering experience.
Deep Dive — Logical Race Conditions
Dart has no shared memory, so there are no data races. Does that mean async Dart code is race-free?
No — Dart is free of memory-level data races, but logical race conditions are common. A logical race occurs when the ordering of async events produces unintended results. Classic patterns: (1) Check-then-act: check a condition (is token valid?), then act (return cached token) — but between the check and the act, an await yields control, and another async frame may change the condition. (2) Stale closure: a search field starts a request for query "fl", then starts another for "flu". The "fl" response arrives after the "flu" response and overwrites the correct results. (3) Double-submit: user taps a button twice before the first tap's async operation completes; both taps proceed because the "isProcessing" flag wasn't set before the first await. Solutions: (1) Use locks (package:synchronized) for mutual exclusion. (2) Capture and compare identifiers at completion time. (3) Set state synchronously before the first await. (4) Use switchMap in RxDart to cancel previous operations automatically.
Deep Dive — Dart Isolate vs Kotlin Coroutine
Compare Dart isolates to Kotlin coroutines. When would each be appropriate?
Kotlin coroutines: cooperative concurrency on shared threads. Coroutines are lightweight (thousands can run concurrently on a few threads). They share heap memory — communication is via shared objects + synchronization (Mutex, Channel). Dispatchers (Main, IO, Default) route coroutines to thread pools. Structured concurrency via CoroutineScope ensures cancellation propagates. Coroutines are appropriate for IO-bound work (all of it, on Dispatchers.IO) and CPU-bound work (Dispatchers.Default, which uses a thread pool sized to CPU count).

Dart isolates: true parallelism with separate heaps. Communication via message passing (copy semantics, except in isolate groups with immutable data). Isolates are heavier to spawn (~1–5ms) than coroutines. They're appropriate for CPU-bound work that would block the UI thread — JSON parsing, image processing, crypto. Dart's async/await handles IO-bound work on the single thread without isolates (analogous to Kotlin coroutines on the Main dispatcher).

Key difference: Kotlin coroutines can run on multiple threads sharing memory; Dart isolates are forced into message-passing parallelism. For CPU-intensive work, both achieve parallelism — Kotlin via thread pool, Dart via isolates. For IO-bound work, async/await (Dart) and coroutines (Kotlin) are equivalent — both are cooperative and non-blocking.
Deep Dive — StreamController Lifecycle
What is the correct lifecycle for a StreamController in a Flutter BLoC or Cubit?
A StreamController must be: (1) Created (typically in the BLoC constructor or as a late final field). (2) Exposed via controller.stream (or a derived stream). (3) Fed events via controller.add() or controller.addError(). (4) Closed via controller.close() when the BLoC is disposed. Failure to close causes: the stream's subscribers never receive a done event, the StreamController holds a reference to its stream listeners (memory leak), and potentially a ResourceWarning from the Dart VM. For BLoC pattern: close() is called in the BLoC's close() method (overriding BlocBase.close()). The onListen and onCancel callbacks can be used to start/stop producers (e.g., location updates) only when there are active listeners — efficient resource management. Use controller.hasListener to check before adding events to avoid adding to a closed/un-listened controller.
Deep Dive — Isolate.run() Internals
How does Isolate.run() handle errors and cancellation? What happens if the spawned isolate throws?
Isolate.run() wraps Isolate.spawn() with error handling infrastructure. Internally, it: (1) Sets up onError and onExit ports on the spawned isolate. (2) Runs the provided function in the spawned isolate. (3) If the function throws, the error and stack trace are serialized and sent back to the main isolate via the error port. Isolate.run()'s returned Future completes with that error — so you handle it with try/catch on the awaited result. (4) If the function completes normally, the result is sent back via the exit port and the Future completes with the value. (5) The spawned isolate is automatically killed after the function returns (normal or error). Cancellation: Isolate.run() does not directly support cancellation. To cancel in-progress work, you must use a custom long-lived worker isolate pattern with a cancellation message sent via SendPort. The spawned isolate must cooperatively check for cancellation — there's no forceful interruption of in-progress Dart code.
Deep Dive — async* and yield
What are async* functions in Dart? How do they differ from regular async functions?
An async* function is a generator that returns a Stream (analogous to how async functions return Future). Inside an async* function, you use yield to emit values to the stream and yield* to delegate to another stream. The function body is lazy — it doesn't execute until a listener subscribes. Execution proceeds until the next yield, then pauses until the listener requests more data (respects backpressure). Key difference from async: async returns one value (Future); async* emits multiple values over time (Stream). Example: 
Stream<int> countdown(int from) async* {
  for (var i = from; i >= 0; i--) {
    yield i;
    await Future.delayed(Duration(seconds: 1));
  }
}

// Consume:
await for (final count in countdown(10)) {
  print(count); // Prints 10, 9, 8... with 1-second delays
}
sync* is the synchronous generator equivalent (returns an Iterable). Useful for: paginated data streams, infinite sequences, wrapping event sources as streams, implementing custom stream pipelines.
