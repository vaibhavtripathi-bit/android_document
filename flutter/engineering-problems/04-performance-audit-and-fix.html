04 — Performance Audit: Social Feed Jank

Problem: Complex feed dropping frames — systematic diagnosis and fix

Problem Statement

A social feed with cards containing text, network images, reaction counts, and an animated like button runs at approximately 45fps instead of 60fps on a mid-range Android device. Users report visible stuttering when scrolling quickly. Find and fix all performance issues systematically.

Staff Signal
The answer is not a list of tips — it is a diagnostic process. L6 engineers are expected to describe exactly which DevTools view they open, what they look for in the flame chart, and how they confirm the root cause before writing any code. Jumping to fixes without measurement is a red flag.

Diagnostic Workflow

Step 1: Profile Build (Never Debug)

# Profile mode on a physical device — NOT an emulator, NOT debug mode.
flutter run --profile

# Debug mode disables optimizations and adds observatory overhead.
# Release mode strips debug info needed for flame charts.
# Profile mode = release optimizations + profiling hooks.

Step 2: Open DevTools Performance Tab

  1. Run flutter pub global activate devtools, then flutter devtools.
  2. Open the Performance tab.
  3. Enable Enhance Tracing → check "Track widget builds", "Track layouts", "Track paints".
  4. Record 10 seconds of steady-state scrolling.
  5. Stop recording. Examine the frame chart.

Step 3: Read the Frame Chart

Issue 1 — Rebuild Storm

ROOT CAUSE Entire feed rebuilds on every scroll tick

The repaint rainbow (DevTools → Toggle Repaint Rainbow) shows the whole screen cycling colors during scroll. The flame chart shows ListView.build spanning 8ms+ each frame. Investigation reveals two problems stacked:

  • ListView(children: [...allItems]) — all items built eagerly, not lazily.
  • A scroll position ChangeNotifier listener at the feed root calls setState, rebuilding the entire widget tree.
  • ReactionBar is an inline widget inside the card — its like count changes trigger full card rebuilds.
FIX 
// BEFORE — eager, all items built upfront
ListView(
  children: posts.map((p) => FeedCard(post: p)).toList(),
)

// AFTER — lazy builder, only visible items built
ListView.builder(
  itemCount: posts.length,
  itemBuilder: (context, index) => FeedCard(
    key: ValueKey(posts[index].id), // stable key prevents element reuse bugs
    post: posts[index],
  ),
)
// BEFORE — ReactionBar inline, rebuilds with parent
class FeedCard extends StatelessWidget {
  @override
  Widget build(BuildContext context) {
    return Column(children: [
      PostContent(post: post),
      Row(children: [
        Text('${post.likeCount} likes'), // Rebuilds entire card on like
        LikeButton(post: post),
      ]),
    ]);
  }
}

// AFTER — extract ReactionBar as independent StatefulWidget
// with its own ValueListenableBuilder scope
class ReactionBar extends StatelessWidget {
  final ValueListenable<int> likeCount;
  final VoidCallback onLike;

  @override
  Widget build(BuildContext context) {
    return Row(children: [
      ValueListenableBuilder<int>(
        valueListenable: likeCount,
        builder: (context, count, _) => Text('$count likes'),
      ),
      LikeButton(onTap: onLike),
    ]);
  }
}

UI thread per frame (scroll)~9ms~2ms
Widget builds per frame~180~12

Issue 2 — Image Loading Jank

ROOT CAUSE Large images decoded on UI thread during scroll

The flame chart shows ImageDecoder spans on the UI thread lasting 4–6ms each. Network images from the API are 2000×2000px but displayed at 300×200px logical pixels. Flutter decodes the full resolution into memory and on the UI thread.

FIX 
// BEFORE — full resolution decoded
Image.network(post.imageUrl)

// AFTER — constrain decode resolution to display size
// DevicePixelRatio of 3.0 × 300 logical px = 900 physical px max needed
Image.network(
  post.imageUrl,
  cacheWidth: 900,   // Flutter resizes during decode, not after
  cacheHeight: 600,
  fit: BoxFit.cover,
)
// Pre-cache images for the next few items during idle time
class FeedController {
  void precacheNextBatch(BuildContext context, int currentIndex) {
    final end = (currentIndex + 5).clamp(0, posts.length);
    for (final post in posts.sublist(currentIndex + 1, end)) {
      if (post.imageUrl != null) {
        precacheImage(
          ResizeImage(
            NetworkImage(post.imageUrl!),
            width: 900,
          ),
          context,
        );
      }
    }
  }
}

Issue 3 — Animation Compositing

ROOT CAUSE Like button pulse animation repaints entire card

The repaint rainbow shows the entire FeedCard flashing during the like animation. The LikeButton uses a ScaleTransition but is not isolated in its own repaint layer.

FIX 
// BEFORE — animation causes entire card repaint
FeedCard(
  child: Column(children: [
    PostContent(),
    LikeButton(), // animation here propagates upward
  ]),
)

// AFTER — RepaintBoundary isolates the animation layer
FeedCard(
  child: Column(children: [
    PostContent(),
    RepaintBoundary(
      child: LikeButton(),
    ),
  ]),
)
When to use RepaintBoundary
Use it when a subtree animates frequently AND the rest of the parent is static. The tradeoff: each RepaintBoundary creates a new GPU texture layer, which costs memory. Do not wrap every widget — only high-frequency animation roots.

Issue 4 — Shader Compilation Jank

ROOT CAUSE SkSL shaders compiled on first scroll

The timeline shows large SkSL::Compiler spans (15–30ms each) during the first pass through the feed. These happen once per unique draw operation type encountered. On Skia backend, shaders are compiled lazily — the first time Flutter draws a rounded rectangle with a specific gradient, it compiles a shader on the raster thread, blocking the frame.

FIX A — Migrate to Impeller 
# Android (add to AndroidManifest.xml or use flag)
flutter run --enable-impeller

# iOS: Impeller is the default since Flutter 3.10

# Impeller pre-compiles all shaders at engine startup using
# Metal (iOS) or Vulkan/OpenGL (Android). No runtime compilation.
FIX B — SkSL Warm-up (Skia fallback) 
# Step 1: Capture shader warm-up file during a profile run
flutter run --profile --cache-sksl --purge-persistent-cache

# Step 2: Bundle the captured SkSL with the app
flutter build apk --bundle-sksl-path flutter_01.sksl.json

Issue 5 — SaveLayer from BackdropFilter

ROOT CAUSE Frosted-glass card header triggers expensive saveLayer

The DevTools raster flame chart shows saveLayer calls costing 5ms each frame. Investigation finds the card header uses BackdropFilter for a frosted-glass effect. Every BackdropFilter forces Flutter to:

  1. Paint all content behind it into an offscreen buffer (saveLayer).
  2. Apply the blur filter to that buffer.
  3. Composite the result.

This happens every frame, even when nothing behind the header changes.

FIX A — Static blurred asset 
// Replace BackdropFilter with a pre-blurred image asset.
// The asset is blurred once at design time, not at runtime.
// BEFORE:
Stack(children: [
  Image.network(post.imageUrl),
  Positioned(
    top: 0,
    child: ClipRect(
      child: BackdropFilter(
        filter: ImageFilter.blur(sigmaX: 10, sigmaY: 10),
        child: Container(color: Colors.white.withOpacity(0.3), height: 60),
      ),
    ),
  ),
])

// AFTER: use a gradient overlay instead — zero raster cost
Stack(children: [
  Image.network(post.imageUrl, cacheWidth: 900),
  Positioned(
    top: 0,
    child: Container(
      height: 60,
      decoration: const BoxDecoration(
        gradient: LinearGradient(
          begin: Alignment.topCenter,
          end: Alignment.bottomCenter,
          colors: [Colors.black54, Colors.transparent],
        ),
      ),
    ),
  ),
])
FIX B — Cache blur with ImageFiltered (static content only) 
// If the blur must be dynamic, use ImageFiltered on a static subtree
// wrapped in RepaintBoundary — Flutter can cache the blur result.
RepaintBoundary(
  child: ImageFiltered(
    imageFilter: ImageFilter.blur(sigmaX: 10, sigmaY: 10),
    child: StaticHeaderContent(),
  ),
)

Performance Checklist for Feed UIs

  • Always use ListView.builder, never ListView(children:) for dynamic content.
  • Scope setState / notifyListeners to the smallest possible widget subtree.
  • Constrain image decode size with cacheWidth/cacheHeight.
  • Wrap high-frequency animations in RepaintBoundary.
  • Prefer Impeller on supported platforms — it eliminates runtime shader compilation.
  • Audit every BackdropFilter and saveLayer — each is a per-frame raster cost.
  • Use const constructors wherever possible — const widgets are never rebuilt.
  • Profile on a real mid-range device (Pixel 6a, Galaxy A53) — not a flagship or simulator.
