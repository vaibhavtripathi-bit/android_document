02 — Custom Scroll Physics & Snap Behavior

Problem: Page-snapping list with custom momentum and nested scroll conflict resolution

Problem Statement

Build a horizontal card scroller that satisfies three hard requirements simultaneously:

  1. Snaps to the nearest card boundary when the user flings — even at very low or very high velocities.
  2. Uses spring-like (damped oscillation) momentum rather than a plain linear deceleration so the snap feels physical.
  3. Works correctly when embedded inside a vertical ListView or CustomScrollView without the inner and outer scrollers fighting for the gesture.
Staff Signal
Interviewers probe whether you know the difference between physics (how the scroller behaves after the finger lifts) and gesture recognition (who captures the pointer event in a nested scroll). Conflating the two is a common mid-level mistake.

Solution Architecture

1. Snap Point Calculation

Given the current scroll offset and item width, the target snap point is simply the nearest multiple of itemWidth: 

double _snapOffset(double offset, double itemWidth) {
  final index = (offset / itemWidth).round();
  return (index * itemWidth).clamp(minExtent, maxExtent);
}

The .round() gives nearest-card semantics. Use .ceil() if you want forward-only snapping (e.g., a strict pager).

2. SpringDescription Tuning

SpringDescription.withDampingRatio takes three parameters:

  • mass — heavier mass = slower response. Keep at 1.
  • stiffness — higher = snappier spring. 500 is a good starting point.
  • ratio — damping ratio. 1.0 = critically damped (no overshoot). 0.8 = slightly underdamped (subtle bounce). Below 0.6 = obvious oscillation.
final spring = SpringDescription.withDampingRatio(
  mass: 1,
  stiffness: 500,
  ratio: 0.8,
);

3. Nested Scroll Conflict

Flutter's gesture arena gives the pointer event to exactly one recognizer. When an inner horizontal scroller and an outer vertical scroller are both in the tree, the first one that crosses its drag threshold wins. The fix has two parts:

  • Wrap the horizontal scroller in a GestureDetector with behavior: HitTestBehavior.opaque and handle onHorizontalDragStart to call _controller.notifyListeners() — this claims the gesture.
  • Alternatively, use ScrollConfiguration with a custom ScrollBehavior that sets dragDevices to only PointerDeviceKind.touch on the inner scroller, preventing the outer one from responding to touch when the inner scroller is active.
Tip
The cleanest approach in practice is wrapping the horizontal scroller in a NotificationListener<ScrollNotification> that calls notification.context.findRenderObject()... and returns true (absorbing the notification) to prevent the outer CustomScrollView from reacting to the inner scroll events.

Full Implementation

CustomSnapScrollPhysics

import 'dart:math';
import 'package:flutter/physics.dart';
import 'package:flutter/widgets.dart';

class CustomSnapScrollPhysics extends ScrollPhysics {
  final double itemWidth;

  const CustomSnapScrollPhysics({
    required this.itemWidth,
    super.parent,
  });

  @override
  CustomSnapScrollPhysics applyTo(ScrollPhysics? ancestor) {
    return CustomSnapScrollPhysics(
      itemWidth: itemWidth,
      parent: buildParent(ancestor),
    );
  }

  // Called after the finger lifts. Return a Simulation that carries
  // the scroll position to the nearest snap point.
  @override
  Simulation? createBallisticSimulation(
    ScrollMetrics position,
    double velocity,
  ) {
    // If we're already at a snap point and velocity is negligible, do nothing.
    final snap = _nearestSnap(position.pixels, position);
    if ((position.pixels - snap).abs() < 0.5 && velocity.abs() < 0.1) {
      return null;
    }

    // If the user flung hard in one direction, snap to the NEXT card
    // in that direction rather than the nearest.
    final targetIndex = velocity.abs() > 600
        ? velocity > 0
            ? (position.pixels / itemWidth).ceil()
            : (position.pixels / itemWidth).floor()
        : (position.pixels / itemWidth).round();

    final target = (targetIndex * itemWidth)
        .clamp(position.minScrollExtent, position.maxScrollExtent);

    final spring = SpringDescription.withDampingRatio(
      mass: 1,
      stiffness: 500,
      ratio: 0.8,
    );

    return SpringSimulation(spring, position.pixels, target, velocity);
  }

  // Smoothly resist dragging past the boundary instead of hard-stopping.
  @override
  double applyBoundaryConditions(ScrollMetrics position, double value) {
    if (value < position.minScrollExtent &&
        position.pixels <= position.minScrollExtent) {
      return value - position.minScrollExtent;
    }
    if (value > position.maxScrollExtent &&
        position.pixels >= position.maxScrollExtent) {
      return value - position.maxScrollExtent;
    }
    return 0.0;
  }

  // Apply a light friction multiplier during drag so fast swipes feel
  // intentional rather than runaway.
  @override
  double applyPhysicsToUserOffset(ScrollMetrics position, double offset) {
    return offset * 0.92;
  }

  double _nearestSnap(double pixels, ScrollMetrics position) {
    final index = (pixels / itemWidth).round();
    return (index * itemWidth)
        .clamp(position.minScrollExtent, position.maxScrollExtent);
  }
}

SnapCardScroller Widget

class SnapCardScroller extends StatelessWidget {
  final List<Widget> cards;
  final double cardWidth;
  final double cardSpacing;

  const SnapCardScroller({
    super.key,
    required this.cards,
    this.cardWidth = 300,
    this.cardSpacing = 16,
  });

  @override
  Widget build(BuildContext context) {
    final itemWidth = cardWidth + cardSpacing;

    return NotificationListener<ScrollNotification>(
      // Absorb scroll notifications so the parent vertical scroller
      // doesn't react to horizontal scroll events.
      onNotification: (notification) => notification is ScrollUpdateNotification,
      child: ListView.separated(
        scrollDirection: Axis.horizontal,
        physics: CustomSnapScrollPhysics(itemWidth: itemWidth),
        padding: const EdgeInsets.symmetric(horizontal: 16),
        itemCount: cards.length,
        separatorBuilder: (_, __) => SizedBox(width: cardSpacing),
        itemBuilder: (context, index) => SizedBox(
          width: cardWidth,
          child: cards[index],
        ),
      ),
    );
  }
}

Gesture Conflict: Deep Dive

When both scrollers are present, Flutter's HorizontalDragGestureRecognizer and VerticalDragGestureRecognizer compete. The winner is determined during the gesture arena resolution phase:

  1. Both recognizers enter the arena when a pointer-down event is received.
  2. As pointer-move events arrive, each recognizer checks if the cumulative offset in its axis exceeds kTouchSlop (8 logical pixels by default).
  3. The first recognizer to exceed the slop declares victory — all other recognizers in the arena lose and receive rejectGesture.

The problem: a diagonal swipe might activate the outer vertical scroller first, even when the user intended to scroll the card list. Solutions in order of invasiveness:

SolutionWhen to useTrade-off
NotificationListener absorbing ScrollUpdateNotificationSimple case, no custom gesture logic neededOnly prevents notification propagation; doesn't fix arena conflict
GestureDetector wrapping the horizontal scroller with onHorizontalDragStartNeed to claim gesture aggressivelyRequires managing scroll controller manually
RawGestureDetector with a custom HorizontalDragGestureRecognizer that wins immediatelyFully custom gesture arbitrationComplex; can break accessibility
Use PageView with physics: NeverScrollableScrollPhysics for inner, manage paging externallyPager use case with explicit page controlLoses custom snap physics
Common Mistake
Setting physics: NeverScrollableScrollPhysics() on the inner scroller "solves" the conflict but destroys the ability to scroll it with touch. Only use this when you're driving the inner scroll programmatically via a ScrollController.

Testing the Physics

void main() {
  testWidgets('snaps to nearest card on low-velocity fling', (tester) async {
    final physics = CustomSnapScrollPhysics(itemWidth: 316);
    final position = _FakeScrollMetrics(pixels: 400, min: 0, max: 5000);

    // Low velocity: should snap to nearest (index 1 = 316px)
    final sim = physics.createBallisticSimulation(position, 50);
    expect(sim, isNotNull);

    // Run simulation to completion
    var t = 0.0;
    double finalX = position.pixels;
    while (!sim!.isDone(t)) {
      finalX = sim.x(t);
      t += 0.016;
    }
    // Should have snapped to 316 (nearest card boundary)
    expect(finalX, closeTo(316, 2.0));
  });

  testWidgets('snaps to next card on high-velocity fling', (tester) async {
    final physics = CustomSnapScrollPhysics(itemWidth: 316);
    final position = _FakeScrollMetrics(pixels: 300, min: 0, max: 5000);

    // High velocity forward: should snap to ceil (index 1 = 316)
    final sim = physics.createBallisticSimulation(position, 800);
    expect(sim, isNotNull);
  });
}

class _FakeScrollMetrics extends FixedScrollMetrics {
  _FakeScrollMetrics({
    required double pixels,
    required double min,
    required double max,
  }) : super(
          minScrollExtent: min,
          maxScrollExtent: max,
          pixels: pixels,
          viewportDimension: 400,
          axisDirection: AxisDirection.right,
          devicePixelRatio: 1.0,
        );
}

Staff Signal
Be ready to explain the physics loop: Flutter calls createBallisticSimulation once, then ticks the returned Simulation each frame until isDone(t) returns true. The simulation is stateless — it takes time t and returns position x(t) and velocity dx(t). This means simulations are trivially testable without a running app.
