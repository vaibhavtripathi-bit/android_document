01 — State Synchronization Across Isolates

Problem: Heavy computation in background isolate + UI state update

Index 01 Isolate Sync 02 Scroll Physics 03 Plugin Streaming 04 Perf Audit 05 Offline Queue 06 RenderObject 07 Animation Sequence 08 Deep Link Auth
Engineering Problem
Heavy JSON parsing and computation must run off the main isolate, with results updating BLoC/Riverpod state on the main isolate — without blocking UI or losing messages.

1. Problem Statement

Your app receives large JSON payloads (500KB–5MB) from a REST API. Parsing and transforming this data on the main isolate causes frame drops — the JSON decode() call alone can take 40–200ms for large payloads. You need to:

  • Move JSON parsing and data transformation to a background isolate
  • Return the result to the main isolate to update a Riverpod AsyncNotifier or BLoC
  • Handle errors thrown in the background isolate
  • Support cancellation of in-flight work when the screen is disposed
  • Maintain result ordering for sequential requests

2. Option A: Isolate.run() for One-Off Tasks

Isolate.run() (introduced in Dart 2.19) is the simplest API for one-off background computation. It spawns a new isolate, runs the provided closure, returns the result to the caller via a Future, and then terminates the isolate. No manual port management required.

import 'dart:isolate';
import 'dart:convert';

// The computation function — runs in a fresh isolate
// Must be a top-level function or static method (closure variables are copied)
List<ProductModel> _parseProducts(String jsonString) {
  final List<dynamic> raw = json.decode(jsonString) as List<dynamic>;
  return raw
      .map((e) => ProductModel.fromJson(e as Map<String, dynamic>))
      .toList();
}

// In your Riverpod notifier or repository:
class ProductRepository {
  Future<List<ProductModel>> fetchProducts() async {
    // 1. Fetch raw JSON on main isolate (network is async, doesn't block)
    final response = await http.get(Uri.parse('https://api.example.com/products'));
    final jsonString = response.body;

    // 2. Offload parsing to a background isolate
    // Isolate.run creates isolate, sends jsonString (copied), returns result
    final products = await Isolate.run(
      () => _parseProducts(jsonString),
    );

    return products;
  }
}
When to use Isolate.run()
Use for one-shot tasks where spawn overhead (2–5ms) is acceptable. Avoid for tasks called frequently (e.g., parsing every scroll position update) or tasks that need to be cancelled — Isolate.run() returns a Future with no cancellation handle.

3. Option B: Long-Lived Worker Isolate with SendPort/ReceivePort

For repeated heavy operations (e.g., a data processing pipeline that runs dozens of times per session), spawning a new isolate each time is wasteful. A long-lived worker isolate with a message loop avoids repeated spawn overhead and enables request/response patterns with ordering guarantees.

3.1 Worker Isolate Entry Point

// Message types for the message loop
sealed class _IsolateMessage {}

class _ParseRequest extends _IsolateMessage {
  final int sequenceId; // for ordering guarantee
  final String jsonString;
  final SendPort replyPort;
  _ParseRequest(this.sequenceId, this.jsonString, this.replyPort);
}

class _ShutdownMessage extends _IsolateMessage {}

// The isolate's entry function — runs in the background isolate
// mainSendPort: port to send results back to the main isolate
void _workerEntryPoint(SendPort mainSendPort) {
  // Create a port for this isolate to receive incoming messages
  final receivePort = ReceivePort();

  // Send our port to the main isolate so it can send us work
  mainSendPort.send(receivePort.sendPort);

  // Message loop
  receivePort.listen((message) {
    if (message is _ParseRequest) {
      try {
        final result = _parseProducts(message.jsonString);
        message.replyPort.send(_ParseSuccess(message.sequenceId, result));
      } catch (e, st) {
        message.replyPort.send(_ParseError(message.sequenceId, e.toString(), st));
      }
    } else if (message is _ShutdownMessage) {
      receivePort.close(); // end the message loop
    }
  });
}

List<ProductModel> _parseProducts(String jsonString) {
  final raw = json.decode(jsonString) as List<dynamic>;
  return raw.map((e) => ProductModel.fromJson(e as Map<String, dynamic>)).toList();
}

3.2 Main Isolate Worker Manager

class IsolateWorker {
  Isolate? _isolate;
  SendPort? _workerSendPort;
  late ReceivePort _receivePort;
  int _nextSequenceId = 0;

  // Pending completers keyed by sequence ID for ordering guarantee
  final Map<int, Completer<List<ProductModel>>> _pending = {};

  Future<void> initialize() async {
    _receivePort = ReceivePort();

    // Spawn the worker isolate, passing our receive port's send port
    _isolate = await Isolate.spawn(
      _workerEntryPoint,
      _receivePort.sendPort,
      onError: _receivePort.sendPort, // route uncaught errors here too
    );

    // First message from worker = its SendPort
    _workerSendPort = await _receivePort.first;

    // Listen for subsequent results
    _receivePort.skip(1).listen(_handleResult);
  }

  void _handleResult(dynamic message) {
    if (message is _ParseSuccess) {
      _pending[message.sequenceId]?.complete(message.products);
      _pending.remove(message.sequenceId);
    } else if (message is _ParseError) {
      _pending[message.sequenceId]?.completeError(
        Exception(message.error), message.stackTrace);
      _pending.remove(message.sequenceId);
    } else if (message is List) {
      // Uncaught isolate error (from onError port) — format: [errorStr, stackStr]
      // Cancel all pending work
      for (final completer in _pending.values) {
        completer.completeError(Exception(message[0]));
      }
      _pending.clear();
    }
  }

  Future<List<ProductModel>> parse(String jsonString) {
    assert(_workerSendPort != null, 'Call initialize() first');
    final id = _nextSequenceId++;
    final completer = Completer<List<ProductModel>>();
    _pending[id] = completer;

    _workerSendPort!.send(
      _ParseRequest(id, jsonString, _receivePort.sendPort),
    );

    return completer.future;
  }

  void dispose() {
    _workerSendPort?.send(_ShutdownMessage());
    _isolate?.kill(priority: Isolate.immediate);
    _receivePort.close();
    // Cancel pending completers
    for (final c in _pending.values) {
      c.completeError(Exception('Worker disposed'));
    }
    _pending.clear();
  }
}

3.3 Sequence ID Ordering Guarantee

Because the worker processes messages sequentially (it has a single message loop), results arrive in the same order as requests. The sequence ID allows the main isolate to correlate results with their callers via the _pending map — even if you later add parallelism to the worker (e.g., multiple isolates in a pool).

4. TransferableTypedData: Zero-Copy Large Buffers

Normally, sending data between isolates copies it — a 5MB JSON string is copied byte-for-byte. For binary data (image buffers, typed arrays), use TransferableTypedData to transfer ownership without copying:

// Sending a large Uint8List without copying
final bytes = imageData as Uint8List;
final transferable = TransferableTypedData.fromList([bytes]);

// Send the transferable — original bytes are now invalid in this isolate
sendPort.send(transferable);

// In the receiving isolate — materialize the bytes
final received = message as TransferableTypedData;
final restoredBytes = received.materialize().asUint8List();
// restoredBytes is a view into the transferred memory (no copy)
JSON strings are always copied
TransferableTypedData only applies to typed data (Uint8List, Float32List, etc.). Regular Dart strings and objects are always deep-copied when sent between isolates. To minimize copy cost for JSON, consider sending the raw bytes (Uint8List) and decoding in the worker isolate using utf8.decode() before json.decode().

5. BackgroundIsolateBinaryMessenger for Platform Channels

If your background isolate needs to call a platform channel (e.g., to read from a native database or use a native crypto library), use BackgroundIsolateBinaryMessenger. This requires passing the root isolate token to the background isolate.

// On the main isolate — capture the token before spawning
final rootToken = RootIsolateToken.instance!;

// Spawn the isolate, passing the token
await Isolate.spawn(_workerWithPlatformChannel, {
  'token': rootToken,
  'replyPort': replyPort.sendPort,
});

// In the background isolate entry function
void _workerWithPlatformChannel(Map<String, dynamic> args) {
  final token = args['token'] as RootIsolateToken;
  final replyPort = args['replyPort'] as SendPort;

  // Register this isolate with the binary messenger
  BackgroundIsolateBinaryMessenger.ensureInitialized(token);

  // Now platform channels work from this isolate
  const channel = MethodChannel('com.example/crypto');
  channel.invokeMethod('hash', {'data': 'my data'}).then((result) {
    replyPort.send(result);
  });
}

6. Error Handling Patterns

// Pattern 1: Error port on Isolate.spawn
// Uncaught errors in the isolate are sent to this port as [errorStr, stackStr]
final errorPort = ReceivePort();
final isolate = await Isolate.spawn(
  _entryPoint,
  args,
  onError: errorPort.sendPort,
  onExit: exitPort.sendPort,
);

errorPort.listen((error) {
  final list = error as List;
  final errorString = list[0] as String;
  final stackString = list[1] as String?;
  log('Isolate error: $errorString\n$stackString');
  // Notify state management of the failure
});

// Pattern 2: try/catch inside the isolate (preferred for expected errors)
// Catch and serialize exceptions, send as error result objects
void _safeWorker(SendPort replyPort) {
  try {
    final result = _doHeavyWork();
    replyPort.send({'ok': true, 'result': result});
  } on FormatException catch (e) {
    replyPort.send({'ok': false, 'error': 'Invalid JSON: ${e.message}'});
  } catch (e, st) {
    replyPort.send({'ok': false, 'error': e.toString(), 'stack': st.toString()});
  }
}

7. Cancellation of In-Flight Work

// Using a CancelToken pattern over Isolate
class CancellableIsolateTask<T> {
  final Completer<T> _completer = Completer<T>();
  Isolate? _isolate;
  bool _cancelled = false;

  Future<T> get future => _completer.future;

  Future<void> start(void Function(SendPort) entry, dynamic args) async {
    final port = ReceivePort();
    _isolate = await Isolate.spawn(entry, {
      'port': port.sendPort,
      'args': args,
    });

    port.listen((message) {
      if (_cancelled) return; // discard result if cancelled
      if (message is T) {
        _completer.complete(message);
      } else {
        _completer.completeError(Exception(message));
      }
      port.close();
    });
  }

  void cancel() {
    _cancelled = true;
    _isolate?.kill(priority: Isolate.immediate);
    if (!_completer.isCompleted) {
      _completer.completeError(Exception('Cancelled'));
    }
  }
}

// Usage in a Riverpod notifier with ref.onDispose
class ProductNotifier extends AsyncNotifier<List<ProductModel>> {
  CancellableIsolateTask? _currentTask;

  @override
  Future<List<ProductModel>> build() async {
    ref.onDispose(() => _currentTask?.cancel());
    return _loadProducts();
  }

  Future<List<ProductModel>> _loadProducts() async {
    _currentTask?.cancel(); // cancel previous in-flight task
    final task = CancellableIsolateTask<List<ProductModel>>();
    _currentTask = task;
    // ... start and await task
    return task.future;
  }
}

8. Integration with Riverpod AsyncNotifier

// Clean integration: worker is a singleton provider
final isolateWorkerProvider = Provider<IsolateWorker>((ref) {
  final worker = IsolateWorker();
  ref.onDispose(worker.dispose);
  // Initialize async — worker is ready before first use
  worker.initialize();
  return worker;
});

// Notifier uses the worker
final productsProvider =
    AsyncNotifierProvider<ProductsNotifier, List<ProductModel>>(
        ProductsNotifier.new);

class ProductsNotifier extends AsyncNotifier<List<ProductModel>> {
  @override
  Future<List<ProductModel>> build() async {
    final response =
        await ref.read(apiClientProvider).get('/products');
    // Offload parsing to background isolate via the worker
    return ref.read(isolateWorkerProvider).parse(response.body);
  }
}
L6 Design Consideration
At L6, discuss the tradeoff: Isolate.run() is simpler but spawns a fresh isolate each call (2–5ms overhead + memory). A long-lived worker amortizes that cost but adds complexity (lifecycle management, error recovery, ordering). For a product used at scale, a worker pool (N isolates, work queue distributed across them) is the right answer for high-throughput parsing — similar to a thread pool in Android's Executors.newFixedThreadPool(). Dart's Isolate maps conceptually to a Java thread but without shared memory — all data is copied or transferred.
