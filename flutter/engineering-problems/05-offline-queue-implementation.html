05 — Robust Offline Mutation Queue

Problem: Queue mutations when offline, drain in order when online, survive restarts

Problem Statement

Users can like posts, follow other users, and post comments while offline. These actions must:

  1. Be reliably delivered to the server once connectivity is restored.
  2. Be delivered in the order they were performed.
  3. Never be delivered more than once (idempotent delivery).
  4. Survive app restarts — a queued action taken yesterday must be sent after a restart today.
  5. Apply to the local UI immediately (optimistic updates) with graceful rollback on permanent failure.
Staff Signal
The naive version (just retry failed requests in memory) fails on restart. The correct version uses a durable local store as the queue. Be ready to discuss idempotency keys, the difference between transient and permanent errors, and why optimistic UI must be decoupled from the network result.

Architecture Overview

User Action
    │
    ▼
[Optimistic Update] ──► Local Drift DB (reactive watch)
    │                          │
    ▼                          ▼
Enqueue PendingOperation   UI updates immediately
    │
    ▼
QueueProcessor
    │   (on app start + on network reconnect)
    ▼
drainQueue()
    ├── Op success ──► mark completed, keep optimistic state
    ├── Transient error (5xx, timeout) ──► exponential backoff, retry
    └── Permanent error (400, 401, 404) ──► mark failed, revert optimistic, show error

Drift Schema

// lib/src/database/pending_operations_table.dart

import 'package:drift/drift.dart';

enum OpStatus { pending, processing, completed, failed }
enum OpType { like, unlike, follow, unfollow, comment }

class PendingOperations extends Table {
  IntColumn get id => integer().autoIncrement()();

  // Enum stored as integer index.
  IntColumn get type => intEnum<OpType>()();

  // JSON-encoded payload. E.g., {"postId":"abc123"} for a like.
  TextColumn get payload => text()();

  // UUID generated on the client. Sent to server as Idempotency-Key header.
  // UNIQUE constraint prevents double-enqueue of the same logical action.
  TextColumn get idempotencyKey => text().unique()();

  // Current queue status.
  IntColumn get status => intEnum<OpStatus>()();

  // How many times we've tried and failed transiently.
  IntColumn get retryCount => integer().withDefault(const Constant(0))();

  // Earliest time we're allowed to retry (used for backoff scheduling).
  DateTimeColumn get retryAfter => dateTime().nullable()();

  // Preserved for ordering — queue is drained in createdAt order.
  DateTimeColumn get createdAt => dateTime()();

  // Foreign key to the locally-created entity (for optimistic rollback).
  TextColumn get localEntityId => text().nullable()();
}

@DriftDatabase(tables: [PendingOperations])
class AppDatabase extends _$AppDatabase {
  AppDatabase(QueryExecutor e) : super(e);

  @override
  int get schemaVersion => 1;

  // Watch all pending operations for the debug overlay.
  Stream<List<PendingOperation>> watchPending() {
    return (select(pendingOperations)
          ..where((t) => t.status.equalsValue(OpStatus.pending))
          ..orderBy([(t) => OrderingTerm.asc(t.createdAt)]))
        .watch();
  }
}

Enqueue: Adding Operations

// lib/src/queue/queue_repository.dart

class QueueRepository {
  final AppDatabase _db;

  QueueRepository(this._db);

  Future<void> enqueue({
    required OpType type,
    required Map<String, dynamic> payload,
    String? localEntityId,
  }) async {
    final key = const Uuid().v4(); // unique idempotency key per action
    await _db.into(_db.pendingOperations).insertOnConflictUpdate(
      PendingOperationsCompanion.insert(
        type: type,
        payload: jsonEncode(payload),
        idempotencyKey: key,
        status: OpStatus.pending,
        createdAt: DateTime.now(),
        localEntityId: Value(localEntityId),
      ),
    );
  }

  Future<List<PendingOperation>> getPendingOps() {
    return (select(_db.pendingOperations)
          ..where((t) => t.status.equalsValue(OpStatus.pending)
              | t.status.equalsValue(OpStatus.processing))
          ..where((t) => t.retryAfter.isNull()
              | t.retryAfter.isSmallerOrEqualValue(DateTime.now()))
          ..orderBy([(t) => OrderingTerm.asc(t.createdAt)]))
        .get();
  }

  Future<void> markCompleted(int id) async {
    await (_db.update(_db.pendingOperations)..where((t) => t.id.equals(id)))
        .write(const PendingOperationsCompanion(status: Value(OpStatus.completed)));
  }

  Future<void> markFailed(int id) async {
    await (_db.update(_db.pendingOperations)..where((t) => t.id.equals(id)))
        .write(const PendingOperationsCompanion(status: Value(OpStatus.failed)));
  }

  Future<void> scheduleRetry(int id, int retryCount) async {
    final backoffSeconds = _backoff(retryCount);
    await (_db.update(_db.pendingOperations)..where((t) => t.id.equals(id)))
        .write(PendingOperationsCompanion(
          retryCount: Value(retryCount + 1),
          retryAfter: Value(DateTime.now().add(Duration(seconds: backoffSeconds))),
          status: const Value(OpStatus.pending),
        ));
  }

  // Exponential backoff: 1s, 2s, 4s, 8s... capped at 5 minutes.
  int _backoff(int retryCount) {
    return (1 << retryCount).clamp(1, 300);
  }
}

QueueProcessor: Draining the Queue

// lib/src/queue/queue_processor.dart

class QueueProcessor {
  final QueueRepository _queue;
  final ApiClient _api;
  final OptimisticStateRepository _optimistic;
  final ConnectivityStream _connectivity;

  StreamSubscription? _connectivitySub;
  bool _draining = false;

  QueueProcessor({
    required QueueRepository queue,
    required ApiClient api,
    required OptimisticStateRepository optimistic,
    required ConnectivityStream connectivity,
  })  : _queue = queue,
        _api = api,
        _optimistic = optimistic,
        _connectivity = connectivity;

  void start() {
    // Drain immediately on startup (handles restart recovery).
    drainQueue();

    // Drain whenever connectivity is restored.
    _connectivitySub = _connectivity.onConnected.listen((_) {
      drainQueue();
    });
  }

  void stop() {
    _connectivitySub?.cancel();
  }

  Future<void> drainQueue() async {
    // Prevent concurrent drain runs.
    if (_draining) return;
    _draining = true;

    try {
      final ops = await _queue.getPendingOps();
      for (final op in ops) {
        await _processOne(op);
        // Stop draining if we go offline mid-drain.
        if (!_connectivity.isConnected) break;
      }
    } finally {
      _draining = false;
    }
  }

  Future<void> _processOne(PendingOperation op) async {
    try {
      await _api.execute(
        type: op.type,
        payload: jsonDecode(op.payload) as Map<String, dynamic>,
        idempotencyKey: op.idempotencyKey,
      );
      await _queue.markCompleted(op.id);
      // Optimistic state was already correct — nothing to revert.

    } on TransientApiException catch (e) {
      // 5xx, timeout, network error — retry with backoff.
      await _queue.scheduleRetry(op.id, op.retryCount);

    } on PermanentApiException catch (e) {
      // 400, 401, 404, 409 — this operation will never succeed.
      await _queue.markFailed(op.id);
      // Revert the optimistic update in local DB.
      if (op.localEntityId != null) {
        await _optimistic.revert(op.type, op.localEntityId!);
      }
      // Bubble up to UI for error display.
      _errorStream.add(QueueError(op: op, cause: e));
    }
  }

  final _errorStream = StreamController<QueueError>.broadcast();
  Stream<QueueError> get errors => _errorStream.stream;
}

API Client: Idempotency Key Header

// lib/src/api/api_client.dart

class ApiClient {
  final Dio _dio;

  Future<void> execute({
    required OpType type,
    required Map<String, dynamic> payload,
    required String idempotencyKey,
  }) async {
    final response = await _dio.post(
      _endpointFor(type),
      data: payload,
      options: Options(headers: {
        // Server uses this to deduplicate — safe to retry.
        'Idempotency-Key': idempotencyKey,
      }),
    );

    if (response.statusCode! >= 500 || response.statusCode == null) {
      throw TransientApiException(response.statusCode);
    }
    if (response.statusCode! >= 400) {
      throw PermanentApiException(response.statusCode!, response.data);
    }
  }

  String _endpointFor(OpType type) => switch (type) {
    OpType.like => '/posts/like',
    OpType.unlike => '/posts/unlike',
    OpType.follow => '/users/follow',
    OpType.unfollow => '/users/unfollow',
    OpType.comment => '/posts/comment',
  };
}

Optimistic UI: Immediate Local Updates

// lib/src/features/feed/feed_notifier.dart

class FeedNotifier extends AsyncNotifier<List<Post>> {
  @override
  Future<List<Post>> build() async {
    // Watch Drift DB — UI updates reactively whenever DB changes.
    ref.watch(feedRepositoryProvider).watchPosts().listen((posts) {
      state = AsyncData(posts);
    });
    return ref.read(feedRepositoryProvider).getPosts();
  }

  Future<void> likePost(String postId) async {
    // 1. Apply optimistic update immediately to local DB.
    await ref.read(feedRepositoryProvider).setLiked(postId, true);

    // 2. Enqueue the network operation.
    await ref.read(queueRepositoryProvider).enqueue(
      type: OpType.like,
      payload: {'postId': postId},
      localEntityId: postId,
    );

    // 3. Trigger a drain attempt (no-op if offline, runs if online).
    ref.read(queueProcessorProvider).drainQueue();
  }
}

Optimistic Rollback

// lib/src/queue/optimistic_state_repository.dart

class OptimisticStateRepository {
  final AppDatabase _db;

  Future<void> revert(OpType type, String entityId) async {
    switch (type) {
      case OpType.like:
        // Server rejected the like — revert to unliked state.
        await _db.feedDao.setLiked(entityId, false);
      case OpType.follow:
        await _db.socialDao.setFollowing(entityId, false);
      case OpType.comment:
        // Delete the locally-created comment row.
        await _db.commentsDao.deleteLocal(entityId);
      default:
        break;
    }
  }
}

    // Enqueue 3 ops at slightly different times.
    await queue.enqueue(type: OpType.like, payload: {'postId': '1'});
    await Future.delayed(const Duration(milliseconds: 1));
    await queue.enqueue(type: OpType.follow, payload: {'userId': 'A'});
    await Future.delayed(const Duration(milliseconds: 1));
    await queue.enqueue(type: OpType.comment, payload: {'text': 'Hi'});

    await processor.drainQueue();

    verify(api.execute(type: OpType.like, any, any)).called(1);
    // Verify ordering via captor
    final callOrder = verify(api.execute(type: captureAny, any, any))
        .captured
        .map((t) => t as OpType)
        .toList();
    expect(callOrder, [OpType.like, OpType.follow, OpType.comment]);
  });

  test('retries transient failures with backoff', () async {
    when(api.execute(any, any, any)).thenThrow(TransientApiException(503));
    await queue.enqueue(type: OpType.like, payload: {'postId': '1'});

    await processor.drainQueue();

    final ops = await queue.getPendingOps(); // retryAfter is in the future
    expect(ops, isEmpty); // not eligible for retry yet
    final allOps = await db.select(db.pendingOperations).get();
    expect(allOps.first.retryCount, 1);

    when(api.execute(any, any, any)).thenThrow(PermanentApiException(400, {}));

    await processor.drainQueue();

    final allOps = await db.select(db.pendingOperations).get();
    expect(allOps.first.status, OpStatus.failed);
  });

Trade-offs & Design Decisions

DecisionChoiceAlternative & why rejected
Queue storageDrift (SQLite)SharedPreferences — no ordering guarantees, no atomic updates
IdempotencyClient-generated UUID + server deduplicationNo idempotency key — double-sends on retry cause duplicate likes
Error classificationTransient vs permanent based on HTTP statusRetry all errors — 400s will never succeed, loop forever
Optimistic stateImmediate local DB write, watched by UIIn-memory state only — lost on restart, UI inconsistent with DB
Drain triggerApp start + connectivity restoredPolling timer — wastes battery, misses reconnect events
ConcurrencySerial drain (one op at a time)Parallel drain — ordering not guaranteed, server-side conflicts
Edge Case: Conflicting Operations
If a user likes then unlikes a post while offline, the queue contains both ops. Naive serial execution sends like then unlike — correct. But if the user likes, the app crashes, restarts, and they unlike — the queue only contains unlike (the like was already drained). Server state may be inconsistent. Solution: when enqueueing, check for a pending inverse op and cancel both instead of sending either.
