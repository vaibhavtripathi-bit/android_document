ListView(children) vs ListView.builder

ListView(children: [...]) eagerly builds all child widgets regardless of whether they are visible. For any list longer than roughly 20–30 items this causes unnecessary build work, wasted memory, and potential frame drops on first render.

ListView.builder uses a viewport and a sliver internally to lazily build only the items that are currently visible plus a configurable buffer (cacheExtent). Items outside the viewport are not built, laid out, or painted.

// BAD: builds all 10,000 items immediately
ListView(
  children: items.map((item) => ItemCard(item: item)).toList(),
)

// GOOD: builds only ~10-15 items at a time (visible + cacheExtent buffer)
ListView.builder(
  itemCount: items.length,
  itemBuilder: (context, index) => ItemCard(item: items[index]),
)

// ALSO GOOD: when all items are known and can be const
// (small lists only)
ListView(
  children: const [
    HeaderTile(),
    FeatureTile(),
    FooterTile(),
  ],
)

How ListView.builder Works Internally

Internally, ListView.builder is a CustomScrollView with a single SliverList child. The SliverList uses a SliverChildBuilderDelegate which only calls itemBuilder for indices within the current viewport window. The Viewport widget manages which slivers are visible and triggers layout only for those.

SliverList vs SliverFixedExtentList

Widget Item Height Layout Cost Scroll-to-Index Best For
SliverList Variable (each item laid out) O(visible items) Must scroll through items Variable-height items, cards, chat messages
SliverFixedExtentList Fixed (itemExtent param) O(1) index calculation Direct O(1) Equal-height rows: contacts list, settings items
SliverPrototypeExtentList Measured from prototype widget O(1) after measurement Direct O(1) Items with same height as a prototype widget
SliverGrid Defined by SliverGridDelegate O(visible items) Direct O(1) for fixed delegates Photo grids, product grids 
// SliverFixedExtentList: fastest possible list scrolling
CustomScrollView(
  slivers: [
    SliverAppBar(title: Text('Contacts'), floating: true),
    SliverFixedExtentList(
      itemExtent: 72.0, // exact height of each item in logical pixels
      delegate: SliverChildBuilderDelegate(
        (context, index) => ContactTile(contact: contacts[index]),
        childCount: contacts.length,
      ),
    ),
  ],
)

// SliverPrototypeExtentList: good when items match a template
SliverPrototypeExtentList(
  prototypeItem: const ListTile(
    title: Text('Prototype'),
    subtitle: Text('Subtitle'),
    leading: CircleAvatar(),
  ),
  delegate: SliverChildBuilderDelegate(
    (context, index) => UserTile(user: users[index]),
    childCount: users.length,
  ),
)

cacheExtent: Pre-Building Items Off-Screen

cacheExtent controls how many logical pixels above and below the visible viewport are pre-built. The default is 250 logical pixels. Pre-built items are fully built and laid out (but not painted or uploaded to the GPU until they enter the visible area).

ListView.builder(
  cacheExtent: 500, // build 500px above and below viewport
  itemCount: items.length,
  itemBuilder: (context, index) => HeavyItemWidget(item: items[index]),
)

// For CustomScrollView:
CustomScrollView(
  cacheExtent: 1000, // aggressive pre-building for fast flings
  slivers: [/* ... */],
)

Tuning cacheExtent

Scenario Recommended cacheExtent Reasoning
Simple text list 250 (default) Items build fast; no need to pre-build more
Image-heavy feed 500–1000 Give time for image fetch + decode before item enters viewport
Complex cards with animations 400–800 Widget build takes longer; pre-build to avoid jank
Memory-constrained device 100–200 Reduce number of off-screen built widgets

addRepaintBoundaries

By default, ListView.builder wraps each item in a RepaintBoundary (controlled by the addRepaintBoundaries parameter, which defaults to true). A RepaintBoundary creates a separate compositing layer for each item. This means when one item changes (e.g., a like button tap), only that item's layer is repainted — not the entire list.

// Default: addRepaintBoundaries = true
// Each item gets its own RepaintBoundary automatically
ListView.builder(
  addRepaintBoundaries: true, // default — good for most cases
  itemBuilder: (context, index) => ItemCard(item: items[index]),
)

// When to set addRepaintBoundaries: false:
// 1. Items are very simple (e.g., just Text widgets) — RepaintBoundary overhead
//    exceeds benefit when items are cheap to repaint
// 2. Items never update individually after being built
// 3. Memory is critical — each RepaintBoundary allocates an offscreen buffer

ListView.builder(
  addRepaintBoundaries: false, // slightly less memory, fine for simple lists
  itemBuilder: (context, index) => SimpleTextRow(text: items[index]),
)
Key Insight
addRepaintBoundaries: true means O(1) repaint cost per item change. addRepaintBoundaries: false means the whole visible list repaints on any item change. For lists where items update (chat messages, live feeds), always keep it true.

addAutomaticKeepAlives

When addAutomaticKeepAlives is true (the default), list items that implement AutomaticKeepAliveClientMixin are kept alive even when scrolled off-screen. This is useful for items with expensive state (e.g., a video player position, form input). However it increases memory usage.

// Item that preserves state when scrolled off-screen:
class VideoListItem extends StatefulWidget { /* ... */ }

class _VideoListItemState extends State<VideoListItem>
    with AutomaticKeepAliveClientMixin {

  @override
  bool get wantKeepAlive => true; // this item requests to be kept alive

  @override
  Widget build(BuildContext context) {
    super.build(context); // MUST call super.build when using this mixin
    return VideoPlayer(/* ... */);
  }
}

// In the list, addAutomaticKeepAlives must be true (default):
ListView.builder(
  addAutomaticKeepAlives: true, // allow items to request keep-alive
  itemBuilder: (context, index) => VideoListItem(video: videos[index]),
)

// For lists where no items need keep-alive, set false to save overhead:
ListView.builder(
  addAutomaticKeepAlives: false,
  addRepaintBoundaries: false,
  itemBuilder: (context, index) => SimpleRow(data: data[index]),
)

CustomScrollView with Multiple Slivers

Real-world feeds often mix different content types. CustomScrollView lets you compose multiple slivers into a single scrollable with one unified scroll position.

CustomScrollView(
  controller: _scrollController,
  slivers: [
    // Collapsing app bar
    SliverAppBar(
      expandedHeight: 200,
      floating: false,
      pinned: true,
      flexibleSpace: FlexibleSpaceBar(
        title: const Text('Feed'),
        background: Image.network(coverUrl, fit: BoxFit.cover),
      ),
    ),

    // Horizontal pinned category chips
    SliverPersistentHeader(
      pinned: true,
      delegate: _CategoryChipsDelegate(categories),
    ),

    // Fixed-extent list of featured items (fast)
    SliverFixedExtentList(
      itemExtent: 80,
      delegate: SliverChildBuilderDelegate(
        (ctx, i) => FeaturedItemRow(item: featured[i]),
        childCount: featured.length,
      ),
    ),

    // Section header
    const SliverToBoxAdapter(
      child: Padding(
        padding: EdgeInsets.all(16),
        child: Text('For You', style: TextStyle(fontSize: 18, fontWeight: FontWeight.bold)),
      ),
    ),

    // Variable-height feed items
    SliverList(
      delegate: SliverChildBuilderDelegate(
        (ctx, i) => FeedCard(item: feed[i]),
        childCount: feed.length,
      ),
    ),

    // Loading indicator at bottom
    SliverToBoxAdapter(
      child: _isLoadingMore
          ? const Padding(
              padding: EdgeInsets.all(16),
              child: Center(child: CircularProgressIndicator()),
            )
          : const SizedBox(height: 80),
    ),
  ],
)

Infinite Scroll Pattern

class _FeedState extends State<Feed> {
  final _scrollController = ScrollController();
  final List<Post> _posts = [];
  bool _isLoading = false;
  bool _hasMore = true;

  @override
  void initState() {
    super.initState();
    _loadMore();
    _scrollController.addListener(_onScroll);
  }

  void _onScroll() {
    // Trigger load when 200px from the bottom
    final position = _scrollController.position;
    if (position.pixels >= position.maxScrollExtent - 200) {
      _loadMore();
    }
  }

  Future<void> _loadMore() async {
    if (_isLoading || !_hasMore) return;
    setState(() => _isLoading = true);
    try {
      final newPosts = await PostRepository.fetchPage(offset: _posts.length);
      setState(() {
        _posts.addAll(newPosts);
        _hasMore = newPosts.length == PostRepository.pageSize;
        _isLoading = false;
      });
    } catch (e) {
      setState(() => _isLoading = false);
    }
  }

  @override
  void dispose() {
    _scrollController.dispose();
    super.dispose();
  }

  @override
  Widget build(BuildContext context) => ListView.builder(
    controller: _scrollController,
    itemCount: _posts.length + (_isLoading ? 1 : 0),
    itemBuilder: (context, index) {
      if (index == _posts.length) {
        return const Center(child: CircularProgressIndicator());
      }
      return PostCard(post: _posts[index]);
    },
  );
}

const List Items

Using const constructors for list items that don't depend on index-specific data eliminates widget rebuilding when the parent rebuilds. Flutter's element reconciliation skips the subtree if the widget is identical (same runtimeType, same key, and for const widgets: same canonical instance).

// GOOD: header and footer are const — not rebuilt on scroll/refresh
ListView.builder(
  itemCount: items.length + 2, // +2 for header and footer
  itemBuilder: (context, index) {
    if (index == 0) return const FeedHeader(); // const = single canonical instance
    if (index == items.length + 1) return const FeedFooter();
    return ItemCard(item: items[index - 1]); // can't be const (depends on data)
  },
)

// GOOD: static separators
ListView.separated(
  itemCount: items.length,
  itemBuilder: (ctx, i) => ItemTile(item: items[i]),
  separatorBuilder: (ctx, i) => const Divider(height: 1), // const separator
)
Anti-Pattern
Never create widgets inside itemBuilder that capture mutable closures unnecessarily. Each call to itemBuilder should return a widget whose constructor arguments come solely from items[index] and stable references. Creating new function objects in itemBuilder prevents Flutter from detecting widget equality and forces full subtree rebuilds.

GridView Variants

// Fixed cross-axis count grid
GridView.builder(
  gridDelegate: const SliverGridDelegateWithFixedCrossAxisCount(
    crossAxisCount: 3,
    crossAxisSpacing: 4,
    mainAxisSpacing: 4,
    childAspectRatio: 1.0, // square cells
  ),
  itemCount: photos.length,
  itemBuilder: (context, index) => PhotoTile(photo: photos[index]),
)

// Max cross-axis extent grid (responsive: adapts column count to width)
GridView.builder(
  gridDelegate: const SliverGridDelegateWithMaxCrossAxisExtent(
    maxCrossAxisExtent: 200, // each cell max 200px wide
    childAspectRatio: 0.75, // portrait cards
    crossAxisSpacing: 8,
    mainAxisSpacing: 8,
  ),
  itemCount: products.length,
  itemBuilder: (context, index) => ProductCard(product: products[index]),
)
Staff Interview Signal
When asked about list performance, go beyond "use ListView.builder". Discuss: (1) fixed-extent vs variable-extent and the O(1) vs O(n) layout implications, (2) RepaintBoundary layer composition and GPU memory trade-offs, (3) cacheExtent tuning for image-heavy lists (pre-fetch window), (4) the relationship between list item widget complexity and the build budget per frame (16ms at 60fps), and (5) when to use flutter_list_view or super_sliver_list for jump-to-index in very long variable-height lists.
