Profiling & Jank

Standard — L5/L6
A user reports that scrolling a product list feels janky on mid-range Android devices. Walk me through your complete debugging approach.
First I reproduce in profile mode — never debug mode, since the JIT and debug overhead can create false jank. I connect DevTools and open the Performance tab, then scroll while recording. I look at the UI thread (Dart) and raster thread frame bars. If UI thread bars are tall, the problem is widget build or layout cost. If raster thread bars are tall, it's painting or GPU upload cost.

For UI thread jank: I check for expensive builds in itemBuilder — things like deep widget trees, synchronous I/O, or setState being called too broadly. I use the Widget Rebuild Stats overlay (debugProfileBuildsEnabled = true) to count rebuilds per widget. I ensure the list uses ListView.builder (lazy) and not ListView(children: [...]).

For raster thread jank: most commonly caused by large image textures being uploaded to the GPU. I check if images have cacheWidth/cacheHeight set, or if they're being decoded at full resolution. Shader compilation jank is another common cause — Impeller largely eliminates this on newer Flutter versions.

I also check for GC events in the Timeline — a major GC mid-scroll will drop multiple frames. The fix there is to reduce allocations in the scroll hot path.
Name the two threads, explain which symptoms point to each, and show you know multiple tools (DevTools flame chart, rebuild stats, memory timeline). That's L6 depth.
Deep Dive — Staff
How do you investigate shader compilation jank, and how does Impeller change this story?
Shader compilation jank occurs when Skia (the legacy renderer) encounters a new draw call that requires a GPU shader it hasn't compiled yet. Shader compilation is synchronous on the raster thread and can take 50–200ms, causing multi-frame drops on first encounter. The symptom: jank only on first scroll/animation, then smooth thereafter. DevTools shows raster thread spikes on first pass through a scroll position, not on subsequent passes.

The traditional fix was SkSL shader warmup: record a "warm-up run" with flutter drive, capture the shaders, bundle them, and replay on startup. This is fragile — shaders vary by GPU vendor and driver version, so a shader captured on a Snapdragon device may be different from one needed on a Mali GPU.

Impeller (Flutter's new renderer, default on iOS since 3.10, Android since 3.19) eliminates this problem entirely by pre-compiling all shaders at build time. Impeller uses a fixed, small set of shaders that cover all rendering operations, and these are compiled to Metal (iOS) or Vulkan SPIR-V (Android) during the app build. Zero runtime compilation = zero shader jank. This is arguably Impeller's most important user-facing improvement over Skia.
This shows awareness of the underlying GPU pipeline, not just Flutter API surface. Mention Impeller proactively — it signals you track the Flutter roadmap.

Memory Leaks

Standard — L5/L6
What are the most common memory leak patterns in Flutter, and how do you detect them?
The five most common patterns are: (1) StreamSubscription not cancelled in dispose() — the stream controller holds a closure capturing the State; (2) AnimationController not disposed — the Ticker stays registered with SchedulerBinding; (3) Timer.periodic not cancelled — the event loop holds the callback closure; (4) ChangeNotifier.addListener without removeListener — the notifier's listener list holds the State; (5) GlobalKey stored at class/module scope instead of State scope — the static registry keeps the Element alive.

Detection: in DevTools Memory tab, I take a heap snapshot before and after navigating away from a screen. If my State subclass still appears in the second snapshot, there's a leak. I click the class to view its retaining path — this shows exactly which object in the reference chain is preventing collection. For automated detection, I use the leak_tracker package which instruments dispose() calls and fails tests if objects aren't collected within a time window.
Always give the retaining path explanation — it shows you understand GC roots, not just the symptom.
Deep Dive — Staff
How would you prevent memory leaks at a team scale, not just fix them individually?
Individual fixes are reactive. Preventing them structurally requires: first, architectural choices that make leaks hard to write — Riverpod's autoDispose providers automatically cancel subscriptions when no widgets watch them, making the StreamSubscription leak pattern impossible by default. Second, integrate leak_tracker into integration tests that run in CI — if a test navigates to a screen and back, it asserts that all State objects were collected. This catches leaks before merge. Third, for teams using BLoC, enforce the pattern of wrapping BlocProvider in a widget that auto-disposes — flutter_bloc's BlocProvider does this correctly. Fourth, code review checklists: any initState that registers a subscription or listener must have a corresponding dispose. This can be partially enforced via custom lint rules using dart_code_metrics or package:custom_lint.

Image Cache

Standard — L5/L6
An image-heavy feed app is using too much memory. How do you investigate and fix the image caching?
First I measure: log imageCache.currentSizeBytes and currentSize periodically. If currentSizeBytes is near maximumSizeBytes (100 MB by default), the cache is at capacity and the LRU is evicting aggressively — or the limit is too high for the device. I check whether images are being decoded at full resolution by inspecting whether cacheWidth/cacheHeight are set on Image.network calls. A 4K image decodes to ~48 MB of RGBA regardless of the display size — this is the most common culprit.

Fixes in order of impact: (1) Add cacheWidth/cacheHeight to all Image.network calls, set to 2× the display size in logical pixels to account for DPR. (2) Reduce imageCache.maximumSizeBytes to something appropriate for the target device class, e.g., 50 MB. (3) Clear the cache on AppLifecycleState.paused to free memory when backgrounded. (4) Use CDN-side resizing (Cloudinary, Imgix) to send correctly-sized images from the server, so the full-resolution bytes never even hit the network.
Mentioning CDN-side resizing elevates the answer — it shows you think about the full system, not just the Flutter layer.

AOT & Tree Shaking

Deep Dive — Staff
Why does using dart:mirrors prevent tree shaking, and what's the correct alternative in Flutter?
Tree shaking works by static reachability analysis from main(). It can only eliminate code that is provably unreachable. dart:mirrors allows calling any method on any object by name at runtime — reflect(obj).invoke(Symbol('methodName'), []). Because the method name is a runtime string, the compiler cannot know at compile time which methods will be called. It must conservatively assume every method on every class might be called, forcing retention of all code. Tree shaking becomes ineffective.

Additionally, dart:mirrors is simply not available in Flutter release builds — the VM strips the mirror system from the AOT snapshot, so importing it causes a runtime error.

The correct alternative is code generation via build_runner. Packages like json_serializable, freezed, and injectable analyze your annotated Dart classes at build time and generate explicit fromJson/toJson/copyWith/factory code. No runtime reflection is needed — the generated code is regular Dart that tree shaking can analyze normally. The trade-off is a build step, but the result is smaller, faster, tree-shaking-friendly code.
Standard — L5/L6
What does --split-debug-info do and why is it required with --obfuscate?
--obfuscate renames all Dart symbols (class names, method names) to short meaningless identifiers (a, b, c...) in the AOT output. This makes the binary harder to reverse-engineer. However it also makes crash stack traces unreadable — instead of MyRepository._fetchUser() you see a.b().

--split-debug-info writes the mapping between obfuscated names and original source locations to a separate .symbols file. This file is kept in your CI artifact store (not shipped in the app). When a crash report comes in with obfuscated addresses, you run flutter symbolize --debug-info=app.symbols --input=crash.txt to convert it back to readable source locations.

--obfuscate requires --split-debug-info because without the symbol file, you lose the ability to debug crashes in production permanently. Flutter enforces this dependency. The .symbols files should be treated as sensitive — they reveal your full internal code structure.

Deferred Loading

Standard — L5/L6
How do you implement deferred library loading in Flutter, and when should you use it?
Dart's deferred imports let you split the AOT snapshot into multiple units. You mark an import as deferred: import 'package:myapp/analytics.dart' deferred as analytics. The code in analytics.dart and everything it transitively imports is excluded from the main snapshot. At runtime, you call await analytics.loadLibrary() before using any code from that library. On Android with app bundles, deferred libraries become separate dynamic feature modules downloaded on demand.

Use it for: features used by a minority of users (admin panels, advanced settings, pro features), large optional dependencies (chart libraries, video editors, PDF renderers), and features that are never used in the critical startup path.

Don't use it for: small libraries (the loadLibrary() overhead and UX delay isn't worth it for small code), code needed on the first screen, or dependencies of the main entry point. The loading delay (disk read + link) is typically 100–500ms, so you need a loading state in the UI and ideally background pre-loading when you can predict user intent.
Mention the UX requirement (loading state) — interviewers often probe whether you've thought about the user experience of async loading, not just the technical mechanism.

Startup Optimization

Deep Dive — Staff
Describe the full Android Flutter cold start optimization strategy for a production app targeting L6 quality bar.
I approach this in phases, from highest to lowest impact: Phase 1 — measure first. Use Android Vitals (Play Console) for p50/p95 startup times in the wild. Use macrobenchmark locally to get reproducible cold start numbers broken into TTID (Time to Initial Display) and TTFD (Time to Full Display). Phase 2 — engine pre-warming. If the main entry point is Flutter, pre-warm the FlutterEngine in Application.onCreate() using FlutterEngineCache. This moves the ~200ms engine init off the critical path. For add-to-app scenarios, lazy pre-warm when the user navigates toward the Flutter screen. Phase 3 — minimize main() blocking work. Identify all awaits in main() and move everything non-essential to a post-frame callback. Only Firebase init and critical session state restoration belong in main(). Phase 4 — baseline profiles. Generate with macrobenchmark and include in the release build. This pre-compiles ART's interpretation of the startup path, saving 30–40% on cold start latency after the first session. Phase 5 — native splash. Use FlutterNativeSplash to display a platform-native splash screen during engine init, covering the white flash. Report fully-drawn with reportFullyDrawn() for accurate Play Console TTFD metrics. Phase 6 — font pre-loading to eliminate FOUT. Measure after each phase with macrobenchmark to confirm actual improvement before moving to the next.
The structured phase approach (measure → biggest impact → verify) is the L6 answer. Listing techniques without measurement discipline is L4/L5.

GC Pauses

Standard — L5/L6
How do GC pauses cause frame drops in Flutter, and how do you mitigate them?
Dart's GC pauses the UI isolate (stop-the-world) during the new-space scavenge and during the initial/final mark phases of old-space collection. New-space pauses are typically under 1ms — usually within the 16ms frame budget. Old-space major collection with compaction can pause for 10–50ms, which drops multiple frames.

The root cause of major GC pressure is promoting too many objects from new space to old space. Objects that survive multiple scavenges get promoted. Reducing allocations in hot paths (build, paint, animation listeners) means fewer promotions to old space, fewer major GCs.

Concrete mitigations: reuse Paint/Rect/Offset instances in CustomPainter instead of allocating new ones each paint call; use const constructors for widgets and style objects; prefer StringBuffer over string concatenation in loops; use object pooling for frequently created/destroyed objects like particles or list item models; schedule heavy work (JSON parsing, image decoding) on separate isolates via compute() so their GC doesn't affect the UI isolate at all.
Mentioning compute() / separate isolates shows awareness that GC is per-isolate — isolating heavy work protects the UI isolate's GC budget.
Deep Dive — Staff
What is the bump pointer allocator and why does it make Dart allocation fast?
In Dart's new space, all allocations use a bump pointer allocator. The allocator maintains a single pointer (the "bump pointer") that points to the next free address in the current semi-space. Allocating an object simply advances this pointer by the object's size — it's a single pointer increment and a bounds check. This is O(1) with minimal overhead and no lock contention.

Compare to a malloc-style allocator: it must search a free list for a block of the right size, handle fragmentation, and update multiple data structures. That's O(log n) at best and involves more cache misses.

The cost of this simplicity is that when the semi-space fills up, the scavenger must copy all live objects to the other semi-space and reset the bump pointer. The copying cost is proportional to the number of surviving objects, not total allocations. So the strategy "allocate freely, collect rarely" works when most objects die young — which is why the generational hypothesis underpins this design. For Flutter, most widget and layout objects created during a build() call die within that frame — exactly the pattern bump pointer allocation is optimized for.
