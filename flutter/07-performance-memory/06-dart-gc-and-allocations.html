Dart's Generational GC Model

Dart uses a generational garbage collector based on the empirical observation that most objects die young. The heap is split into two spaces with different collection strategies:

Space Algorithm Trigger Typical Pause Purpose
New space (young gen) Semi-space scavenger (copying) New space full (~4 MB) <1 ms Short-lived objects: widget builds, closures, temporaries
Old space (old gen) Concurrent mark-sweep Old space pressure / threshold 1–100 ms (concurrent marking reduces pause) Long-lived objects: singletons, caches, large data

New Space: Semi-Space Scavenger

New space is divided into two equal halves: from-space and to-space. All allocations happen in from-space using a bump pointer — just incrementing a pointer, making allocation O(1) and extremely fast (no free-list lookup, no lock contention).

When from-space is full, the scavenger runs:

  1. Pause the Dart isolate (stop-the-world for new-space only)
  2. Scan from GC roots and references from old space (remembered set)
  3. Copy all reachable objects from from-space to to-space (compacting — no fragmentation)
  4. Promote objects that have survived multiple scavenges to old space
  5. Swap from-space and to-space roles; reset bump pointer
  6. Resume the isolate
Key Insight
Because copying GC moves objects, Dart pointers are always managed references — you cannot take the raw address of a Dart object from Dart code. Objects that survive long enough get promoted to old space, which is why long-lived objects should be allocated once and reused, not recreated on every frame.

Old Space: Concurrent Mark-Sweep

Old space uses a mark-sweep algorithm with concurrent marking to reduce pause times:

  1. Initial mark (stop-the-world, very short): Mark GC roots as grey
  2. Concurrent mark (runs alongside Dart code): Trace grey objects, mark reachable objects black. A write barrier tracks any new references created during this phase.
  3. Final mark / remark (stop-the-world, short): Process objects modified during concurrent marking (write barrier log)
  4. Sweep (may be concurrent): Reclaim unmarked (white) objects, build free lists
  5. Compact (optional, stop-the-world): Reduce fragmentation — not always triggered

Minimizing Allocations in Hot Paths

The best way to reduce GC pressure is to allocate less. In Flutter, hot paths include: build(), paint() in CustomPainter, layout callbacks, and animation value listeners.

1. Reuse Paint, Rect, and Offset in CustomPainter

// BAD: allocates new Paint and Rect objects on every paint call
class BadPainter extends CustomPainter {
  @override
  void paint(Canvas canvas, Size size) {
    canvas.drawRect(
      Rect.fromLTWH(0, 0, size.width, size.height), // new Rect each call
      Paint()..color = Colors.blue..strokeWidth = 2, // new Paint each call
    );
  }
}

// GOOD: reuse cached instances
class GoodPainter extends CustomPainter {
  // Allocated once, reused on every paint call
  final _paint = Paint()
    ..color = Colors.blue
    ..strokeWidth = 2
    ..style = PaintingStyle.stroke;

  Rect? _cachedRect;

  @override
  void paint(Canvas canvas, Size size) {
    // Reuse or recreate only when size changes
    _cachedRect = Rect.fromLTWH(0, 0, size.width, size.height);
    canvas.drawRect(_cachedRect!, _paint);
  }

  @override
  bool shouldRepaint(GoodPainter old) => false; // or compare relevant data
}

2. const Objects

// const objects are canonicalized — only one instance exists in the VM
// Every reference to the same const value shares the same object

// BAD: creates a new EdgeInsets object on every build
Padding(padding: EdgeInsets.symmetric(horizontal: 16, vertical: 8))

// GOOD: canonical const instance
Padding(padding: const EdgeInsets.symmetric(horizontal: 16, vertical: 8))

// BAD: new Color every time
Container(color: Color(0xFF2563EB))

// GOOD: const or static final
static const _primaryBlue = Color(0xFF2563EB);
// or: const Color(0xFF2563EB) inline

// BAD: new TextStyle in build()
Text('Hello', style: TextStyle(fontSize: 16, fontWeight: FontWeight.bold))

// GOOD: declare once
static const _headingStyle = TextStyle(fontSize: 16, fontWeight: FontWeight.bold);
Text('Hello', style: _headingStyle)

3. StringBuffer vs String Concatenation

// BAD: O(n^2) allocations — each += creates a new String object
String buildReport(List<String> lines) {
  String result = '';
  for (final line in lines) {
    result += line + '\n'; // allocates a new String each iteration
  }
  return result;
}

// GOOD: StringBuffer pre-allocates and appends efficiently
String buildReport(List<String> lines) {
  final buffer = StringBuffer();
  for (final line in lines) {
    buffer.writeln(line); // no intermediate String allocation
  }
  return buffer.toString(); // single final allocation
}

// For short concatenations (2-3 parts), string interpolation is fine:
final message = 'Hello, $name! You have $count messages.'; // optimized by compiler
// But in a loop: always use StringBuffer

4. List.filled vs List.generate vs List.of

// Use the right List constructor for the use case:

// Filled: fixed-length list with a single initial value — most efficient
final zeros = List<int>.filled(1000, 0); // one allocation, no growth

// Generate: fixed-length list with computed values — one allocation
final squares = List<int>.generate(100, (i) => i * i);

// of: copies an iterable — use when you need a mutable copy
final copy = List<String>.of(original);

// BAD in hot paths: growable list with unknown final size
// Each growth doubles capacity: O(log n) reallocations
final items = <int>[];
for (int i = 0; i < 10000; i++) {
  items.add(i); // may trigger multiple reallocs
}

// GOOD: pre-size if you know the count
final items = List<int>.filled(10000, 0, growable: false);
for (int i = 0; i < 10000; i++) {
  items[i] = i;
}

5. Avoiding Closure Capture in Hot Paths

// BAD: creates a new closure object on every animation tick
animationController.addListener(() {
  setState(() {}); // new closure created every time addListener is called!
});

// GOOD: named method reference — no closure allocation
animationController.addListener(_onAnimationTick);

void _onAnimationTick() {
  setState(() {});
}

// BAD: creating closures in itemBuilder (called for every visible item)
ListView.builder(
  itemBuilder: (context, index) {
    return GestureDetector(
      onTap: () => _handleTap(index), // new closure per item per rebuild
      child: ItemWidget(item: items[index]),
    );
  },
)

// GOOD: pass index as data, handle in the widget
ListView.builder(
  itemBuilder: (context, index) => ItemWidget(
    item: items[index],
    onTap: _handleTap, // stable reference
    index: index,
  ),
)

6. Object Pooling Pattern

// For objects that are frequently created and destroyed:
class ParticlePool {
  final List<Particle> _available = [];
  int _created = 0;
  final int maxSize;

  ParticlePool({this.maxSize = 100});

  Particle acquire() {
    if (_available.isNotEmpty) {
      return _available.removeLast()..reset(); // reuse existing
    }
    _created++;
    return Particle(); // create new only when pool is empty
  }

  void release(Particle particle) {
    if (_available.length < maxSize) {
      _available.add(particle); // return to pool
    }
    // else: let it be GC'd — pool is full
  }
}

// Usage in a particle system:
class _ParticleSystemState extends State<ParticleSystem> {
  final _pool = ParticlePool(maxSize: 200);
  final List<Particle> _active = [];

  void _spawnParticle() {
    _active.add(_pool.acquire());
  }

  void _update() {
    _active.removeWhere((p) {
      if (p.isDead) {
        _pool.release(p); // return to pool instead of GC
        return true;
      }
      p.update();
      return false;
    });
  }
}

DevTools Allocation Tracker

The Memory tab in Flutter DevTools includes an Allocation Tracker that records every object allocation with a stack trace, allowing you to pinpoint exactly which call site is creating too many objects.

  1. Open DevTools → Memory tab
  2. Click "Track Allocations" (or enable "Allocation Tracking" toggle)
  3. Perform the action that causes jank (e.g., scroll a list, run an animation)
  4. Stop tracking
  5. Sort by "Count" to find the most-allocated types
  6. Click a type to see the stack traces of allocation sites
// You can also use dart:developer for programmatic profiling:
import 'dart:developer';

// Enable allocation tracking for a specific type in tests:
// (This is lower-level and usually done via DevTools UI)

// Custom timeline events for correlating with allocations:
void performScroll() {

  // ... scroll logic ...
  Timeline.finishSync();
}

Staff Interview Angle
At L6, the question isn't just "how do you minimize allocations" — it's "how do you ensure this discipline is maintained across a 20-person team?" Key answers: (1) establish allocation budgets per frame via automated performance tests (macrobenchmark frame timing), (2) use DevTools allocation tracker during code review for performance-critical paths, (3) lint rules to flag obvious anti-patterns (e.g., dart_code_metrics for cognitive complexity, which correlates with allocation density), (4) profile-mode CI tests that assert on 99th-percentile frame times, which catch GC-induced jank regressions before they reach production.
