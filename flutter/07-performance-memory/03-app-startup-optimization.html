L6 Staff Focus
Startup performance is one of the highest-impact metrics for user acquisition and retention (Google Play's Core Vitals include startup time). At Staff level you must know every phase, which phase dominates for a given app, which techniques apply to which phase, and the trade-offs of each optimization (e.g., pre-warming saves time on second launch but uses RAM pre-emptively).

Flutter Cold Start Phases

A Flutter cold start (process not in memory) has six sequential phases. Each has different optimization levers:

Phase What Happens Who Controls Typical Duration
1. OS loads binary Dynamic linker loads libapp.so, libflutter.so; maps pages into memory OS + app size 50–200ms
2. Platform init Android: Application.onCreate(), content providers; iOS: +load methods, frameworks Native code 50–500ms
3. Flutter engine init FlutterEngine starts: loads app.so, initializes rendering, sets up channels FlutterEngineCache / warm-up 100–400ms
4. Dart VM startup Dart isolate created, AOT snapshot loaded, root library initialized Baseline profiles 50–200ms
5. main() + app init runApp() called, widget tree built, plugins initialized Your Dart code 50–300ms
6. First frame Layout + paint + rasterize + GPU upload → pixels on screen Widget complexity 16–100ms
Key Insight
Phase 3 (engine init) is often the single largest contributor on Android. Pre-warming the engine eliminates this cost from the user-visible critical path by doing it in the background before the user launches the screen.

Android: Pre-Warming the FlutterEngine

By default, Flutter creates a new FlutterEngine when a FlutterActivity or FlutterFragment is shown. Engine initialization takes 100–400ms. The FlutterEngineCache and FlutterEngineGroup APIs let you initialize the engine in the background before it's needed.

FlutterEngineCache Pattern

// In MyApplication.kt (Application subclass)
class MyApplication : Application() {
    override fun onCreate() {
        super.onCreate()
        // Initialize engine in background during application startup
        // This runs before any Activity, in the OS process init phase
        val engine = FlutterEngine(this)

        // Pre-run Dart entrypoint to warm up the Dart VM and run main()
        engine.dartExecutor.executeDartEntrypoint(
            DartExecutor.DartEntrypoint.createDefault()
        )

        // Cache with a string key for later retrieval
        FlutterEngineCache.getInstance().put("main_engine", engine)
    }
}

// In MainActivity.kt — retrieve instead of creating new
class MainActivity : FlutterActivity() {
    override fun getCachedEngineId() = "main_engine"
    // FlutterActivity checks this and uses the cached engine,
    // skipping the 100-400ms init cost
}

FlutterEngineGroup for Multiple Engines

// FlutterEngineGroup shares the Dart VM across all spawned engines
// Each spawned engine gets its own isolate but shares the snapshot
// Memory cost: ~180KB per additional engine (vs ~19MB for a new group)

class MyApplication : Application() {
    lateinit var engineGroup: FlutterEngineGroup

    override fun onCreate() {
        super.onCreate()
        engineGroup = FlutterEngineGroup(this)
    }
}

// When showing a Flutter screen:
val entrypoint = FlutterEngineGroup.Options(this)
    .setDartEntrypointFunctionName("mainFeature")
val engine = (application as MyApplication).engineGroup.createAndRunEngine(entrypoint)
Trade-off
Pre-warming the engine uses ~19MB of RAM immediately on app launch, even if the user never reaches a Flutter screen. Only pre-warm if your app's primary entry point is Flutter. For add-to-app scenarios, pre-warm lazily when the user begins navigating toward the Flutter screen.

Deferring Heavy Initialization from main()

Everything in main() runs synchronously before the first frame is rendered. Every millisecond spent in main() delays Time-to-First-Frame (TTFF).

Anti-Pattern: Blocking main()

void main() {
  WidgetsFlutterBinding.ensureInitialized();

  // BAD: These all block first frame
  await Firebase.initializeApp();        // ~200ms network/disk
  await SharedPreferences.getInstance(); // ~10ms disk
  await loadUserSettings();              // ~50ms disk
  await initializeCrashlytics();         // ~30ms
  await SomeAnalyticsSDK.initialize();   // ~100ms

  runApp(const MyApp());
  // First frame is delayed by 390ms+ of synchronous init
}

Optimized Pattern: Post-Frame Callbacks

void main() async {
  WidgetsFlutterBinding.ensureInitialized();

  // Only the MINIMUM required before first frame:
  await Firebase.initializeApp(
    options: DefaultFirebaseOptions.currentPlatform,
  );
  // Firebase is synchronously needed by other initializers,
  // so it stays. Everything else moves to post-frame.

  runApp(const MyApp());
}

class MyApp extends StatefulWidget {
  @override
  State<MyApp> createState() => _MyAppState();
}

class _MyAppState extends State<MyApp> {
  @override
  void initState() {
    super.initState();
    // Schedule heavy init AFTER first frame renders
    WidgetsBinding.instance.addPostFrameCallback((_) async {
      await _initializeAfterFirstFrame();
    });
  }

  Future<void> _initializeAfterFirstFrame() async {
    // These run after the user sees the first frame
    final prefs = await SharedPreferences.getInstance();
    await SomeAnalyticsSDK.initialize();
    await RemoteConfig.instance.fetchAndActivate();
    // Update app state to reflect loaded data
    if (mounted) setState(() { /* apply loaded settings */ });
  }
}

Splash Screen as Loading Cover

// Use the native splash to cover initialization time
// flutter_native_splash generates platform splash screens

// pubspec.yaml:
// flutter_native_splash:
//   color: "#FFFFFF"
//   image: assets/splash_logo.png
//   android_12: true
//   fullscreen: true

// In main():
void main() async {
  WidgetsFlutterBinding.ensureInitialized();
  FlutterNativeSplash.preserve(widgetsBinding: WidgetsBinding.instance);

  await _criticalInit(); // runs while splash is visible

  FlutterNativeSplash.remove(); // NOW show Flutter UI
  runApp(const MyApp());
}

Future<void> _criticalInit() async {
  await Firebase.initializeApp();
  // Load minimal data needed for the first screen
  await UserSession.restore(); // check if user is logged in
}

Deferred Component Loading (Dart Deferred Libraries)

Dart supports deferred imports, which split code into separate compilation units loaded on demand. This reduces the initial binary size that must be loaded and parsed at startup.

Deferred Import Syntax

// heavy_feature.dart — a large library with complex UI, charts, video, etc.
library heavy_feature;

class HeavyFeatureScreen extends StatelessWidget {
  // ... lots of code, imports chart libraries, video player, etc.
}

// main_screen.dart — the import is deferred
import 'package:myapp/heavy_feature.dart' deferred as heavyFeature;

class MainScreen extends StatelessWidget {
  Future<void> _openHeavyFeature(BuildContext context) async {
    // Load the library on demand — downloads/decrypts/links the deferred module
    await heavyFeature.loadLibrary();
    // Now the library is available
    Navigator.push(context, MaterialPageRoute(
      builder: (_) => heavyFeature.HeavyFeatureScreen(),
    ));
  }
}

Loading State UI

class DeferredRoute extends StatefulWidget {
  final Future<void> Function() loader;
  final Widget Function() builder;

  const DeferredRoute({required this.loader, required this.builder});

  @override
  State<DeferredRoute> createState() => _DeferredRouteState();
}

class _DeferredRouteState extends State<DeferredRoute> {
  late Future<void> _loadFuture;

  @override
  void initState() {
    super.initState();
    _loadFuture = widget.loader();
  }

  @override
  Widget build(BuildContext context) => FutureBuilder(
    future: _loadFuture,
    builder: (context, snapshot) {
      if (snapshot.connectionState != ConnectionState.done) {
        return const Scaffold(body: Center(child: CircularProgressIndicator()));
      }
      if (snapshot.hasError) {
        return Scaffold(body: Center(child: Text('Failed to load: ${snapshot.error}')));
      }
      return widget.builder();
    },
  );
}

// Usage:
DeferredRoute(
  loader: () => heavyFeature.loadLibrary(),
  builder: () => heavyFeature.HeavyFeatureScreen(),
)
Tip
On Android, deferred libraries are packaged as separate "split APKs" via the dynamic delivery mechanism. They are not downloaded at install time — they are downloaded when loadLibrary() is first called. This is most effective for features used by only a subset of users (e.g., a professional editing mode, an admin panel).

Android Baseline Profiles

Baseline Profiles are a key optimization introduced in Android 7.0 (ART) and refined in Android 9+. They tell ART which code paths to pre-compile (Ahead-of-Time) at app install time, eliminating JIT compilation overhead on first run.

Why They Matter for Flutter

Even though Flutter itself is AOT compiled (the Dart code in app.so is native machine code), the Flutter framework JARs and Android platform code are not. The first time your app's startup path touches these Java/Kotlin code paths, ART JIT-compiles them, causing latency spikes. Baseline Profiles pre-compile these hot paths.

Generating Baseline Profiles with Macrobenchmark

// In your android/ directory, add a macrobenchmark module

// macrobenchmark/src/main/java/com/example/StartupBenchmark.kt
@RunWith(AndroidJUnit4::class)
class StartupBenchmark {
    @get:Rule
    val benchmarkRule = MacrobenchmarkRule()

    @Test
    fun startup() = benchmarkRule.measureRepeated(
        packageName = "com.example.myapp",
        metrics = listOf(StartupTimingMetric()),
        iterations = 5,
        startupMode = StartupMode.COLD,
        setupBlock = {
            pressHome()
        }
    ) {
        startActivityAndWait()
        // Interact with the app to capture the hot paths
        device.findObject(By.text("Explore")).click()
        device.waitForIdle()
    }

    // Generate the baseline profile:
    @Test
    fun baselineProfile() = baselineProfileRule.collect(
        packageName = "com.example.myapp",
        profileBlock = {
            startActivityAndWait()
            // Navigate through the most critical user journeys
        }
    )
}

Including the Profile in Your Build

# Run to generate:
./gradlew :macrobenchmark:generateBaselineProfile

# Profile is written to:
# app/src/main/baseline-prof.txt

# In app/build.gradle:
android {
    defaultConfig {
        # The profile is automatically included in release builds
        # when the file exists at app/src/main/baseline-prof.txt
    }
}

# Verify it's included:
./gradlew :app:bundleRelease
# Check the AAB: it should contain a baseline.prof file

Impact

Metric Without Baseline Profile With Baseline Profile Improvement
Cold TTID (median) ~1800ms ~1200ms ~33%
Cold TTFD (median) ~2200ms ~1500ms ~32%
Jank on first scroll High (JIT compiling scroll path) Low (pre-compiled) Significant

Font Loading Optimization

Custom fonts not loaded before first frame cause a flash of unstyled text (FOUT). Flutter downloads and decodes font files asynchronously. The fix is to pre-load fonts in main().

void main() async {
  WidgetsFlutterBinding.ensureInitialized();

  // Pre-load fonts before first frame to avoid FOUT
  final fontLoader = FontLoader('Roboto')
    ..addFont(rootBundle.load('assets/fonts/Roboto-Regular.ttf'))
    ..addFont(rootBundle.load('assets/fonts/Roboto-Bold.ttf'))
    ..addFont(rootBundle.load('assets/fonts/Roboto-Italic.ttf'));

  await fontLoader.load();

  runApp(const MyApp());
}

// For Google Fonts (google_fonts package):
// Pre-load specific fonts to avoid network fetch on first frame:
await GoogleFonts.pendingFonts([
  GoogleFonts.inter(),
  GoogleFonts.jetBrainsMono(),
]);

Measuring Startup Time

// Using dart:developer for custom startup timeline:
import 'dart:developer' as developer;

void main() async {
  developer.Timeline.startSync('AppStartup');

  WidgetsFlutterBinding.ensureInitialized();
  developer.Timeline.startSync('CriticalInit');
  await _criticalInit();
  developer.Timeline.finishSync(); // CriticalInit

  runApp(const MyApp());
  developer.Timeline.finishSync(); // AppStartup
}

// View in DevTools: Performance tab → Timeline events
// Filter by 'AppStartup' to see your custom spans alongside framework events

// On Android, use reportFullyDrawn() for Play Store metrics:
// In MainActivity.kt:
override fun onResume() {
    super.onResume()
    // After all critical content is visible:
    reportFullyDrawn()
}
Staff Answer Framework
When asked "How would you improve startup time for this app?", structure your answer: (1) Measure first — use DevTools timeline + Android Vitals to identify which phase dominates. (2) Address the largest phase first — usually engine init (pre-warm) or main() blocking work (defer). (3) Verify with macrobenchmark — not just local testing. (4) Baseline profiles for Android release. (5) Monitor in production — set up Firebase Performance startup traces. This shows you think in systems, not just individual tricks.
