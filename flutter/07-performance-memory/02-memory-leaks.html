L6 Staff Focus
Memory leaks are a top interview signal at Staff level. You must be able to identify the root cause from code, explain the GC reference chain, and implement the fix — not just "add dispose()". Interviewers will probe deeper: "How would you detect this at scale in production? How do you prevent this class of bug in a team of 20?"

Why Memory Leaks Matter in Flutter

Dart uses a garbage collector, so objects are reclaimed when no live references to them exist. A memory leak occurs when an object you expect to be collected is still reachable through a reference chain you forgot to sever. In Flutter, the most common reference chains are:

  • Subscriptions holding closures that capture State objects
  • Tickers/animations holding references to TickerProvider (the State)
  • Listeners registered on long-lived objects (e.g., ChangeNotifier from a service locator)
  • Platform or native callbacks capturing BuildContext
  • Global caches (image cache, widget state via GlobalKey) that grow without bound

The consequence is an ever-growing heap, more frequent GC pauses, and ultimately an OOM crash on low-memory devices. Even if the app doesn't crash, GC pressure causes frame drops because Dart's concurrent mark-sweep pauses the isolate at safepoints.

Part 1
The Six Major Leak Patterns
Each pattern includes the broken code, why it leaks, and the correct fix.

1. StreamSubscription Not Cancelled

This is the single most common Flutter memory leak. When a StatefulWidget's State subscribes to a Stream, the resulting StreamSubscription holds a closure that captures this (the State object). If you forget to cancel in dispose(), the stream controller keeps the subscription alive, which keeps the State alive, which keeps the subtree, all associated controllers, and the BuildContext alive.

Leaked Version

class _DataScreenState extends State<DataScreen> {
  List<String> _items = [];

  @override
  void initState() {
    super.initState();
    // LEAK: subscription captured but never stored or cancelled
    DataRepository.instance.itemStream.listen((items) {
      if (mounted) setState(() => _items = items);
    });
  }

  @override
  Widget build(BuildContext context) => ListView(
    children: _items.map((i) => Text(i)).toList(),
  );
}
// When DataScreen is popped, the State object is disposed but NOT collected.
// DataRepository.instance._controller still holds the StreamSubscription,
// which holds a closure capturing _DataScreenState.
// GC root: DataRepository.instance → StreamController → _ControllerSubscription → closure → _DataScreenState

Fixed Version

class _DataScreenState extends State<DataScreen> {
  List<String> _items = [];
  StreamSubscription<List<String>>? _subscription; // store the subscription

  @override
  void initState() {
    super.initState();
    _subscription = DataRepository.instance.itemStream.listen((items) {
      if (mounted) setState(() => _items = items);
    });
  }

  @override
  void dispose() {
    _subscription?.cancel(); // sever the reference chain
    super.dispose();
  }

  @override
  Widget build(BuildContext context) => ListView(
    children: _items.map((i) => Text(i)).toList(),
  );
}

Multiple Subscriptions Pattern

class _DashboardState extends State<Dashboard> {
  final List<StreamSubscription> _subs = [];

  @override
  void initState() {
    super.initState();
    _subs.addAll([
      authService.userStream.listen(_onUserChanged),
      locationService.positionStream.listen(_onPosition),
      connectivityService.statusStream.listen(_onConnectivity),
    ]);
  }

  @override
  void dispose() {
    for (final sub in _subs) sub.cancel();
    super.dispose();
  }
}
Subtle Variant
Even if you call subscription.cancel(), if the stream's StreamController is never closed, the subscription object lives until the next GC cycle. Always also close() controllers you own. Check that third-party packages properly close their controllers.

2. AnimationController Not Disposed

An AnimationController requires a TickerProvider (typically via SingleTickerProviderStateMixin or TickerProviderStateMixin). Internally it creates a Ticker that is registered with the Flutter engine's scheduler. The scheduler holds a reference to the Ticker, which holds the AnimationController, which holds the closure callbacks you've added. If you never dispose the controller, the engine scheduler keeps the entire object graph alive forever.

Leaked Version

class _PulsingButtonState extends State<PulsingButton>
    with SingleTickerProviderStateMixin {
  late AnimationController _controller;

  @override
  void initState() {
    super.initState();
    _controller = AnimationController(
      vsync: this,
      duration: const Duration(milliseconds: 800),
    )..repeat(reverse: true);
    // LEAK: no dispose() override
  }

  // The Ticker inside _controller calls SchedulerBinding.instance.scheduleFrameCallback()
  // which keeps _controller and thus _PulsingButtonState in the GC root set.
}

Fixed Version

class _PulsingButtonState extends State<PulsingButton>
    with SingleTickerProviderStateMixin {
  late AnimationController _controller;
  late Animation<double> _animation;

  @override
  void initState() {
    super.initState();
    _controller = AnimationController(
      vsync: this,
      duration: const Duration(milliseconds: 800),
    )..repeat(reverse: true);

    _animation = Tween<double>(begin: 0.8, end: 1.0).animate(
      CurvedAnimation(parent: _controller, curve: Curves.easeInOut),
    );
  }

  @override
  void dispose() {
    _controller.dispose(); // stops the Ticker and unregisters from scheduler
    super.dispose();
  }

  @override
  Widget build(BuildContext context) => ScaleTransition(
    scale: _animation,
    child: widget.child,
  );
}

Multiple Controllers

class _ComplexAnimState extends State<ComplexAnim>
    with TickerProviderStateMixin { // note: plural for multiple controllers
  late AnimationController _fadeController;
  late AnimationController _slideController;
  late AnimationController _scaleController;

  @override
  void initState() {
    super.initState();
    _fadeController = AnimationController(vsync: this, duration: 300.ms);
    _slideController = AnimationController(vsync: this, duration: 500.ms);
    _scaleController = AnimationController(vsync: this, duration: 400.ms);
  }

  @override
  void dispose() {
    // Dispose ALL controllers — missing even one causes a leak
    _fadeController.dispose();
    _slideController.dispose();
    _scaleController.dispose();
    super.dispose();
  }
}
Tip
Flutter's debug mode prints a warning when a Ticker is active after dispose(): "A Ticker was active when dispose() was called." Always test widget removal (e.g., Navigator.pop) and check the console.

3. Timer.periodic Not Cancelled

Timer.periodic registers a callback with Dart's event loop. The event loop holds a reference to the timer's closure. If that closure captures this (the State), the event loop keeps the State alive even after the widget is removed from the tree.

Leaked Version

class _ClockState extends State<Clock> {
  DateTime _now = DateTime.now();

  @override
  void initState() {
    super.initState();
    // LEAK: timer runs forever, closure captures _ClockState
    Timer.periodic(const Duration(seconds: 1), (_) {
      setState(() => _now = DateTime.now());
    });
  }

  @override
  Widget build(BuildContext context) => Text(_now.toIso8601String());
}
// After Clock is disposed, the Timer still fires every second.
// setState() will be called on a disposed State, which Flutter will assert on in debug.
// In release mode: silent memory leak + potential NPEs.

Fixed Version

class _ClockState extends State<Clock> {
  DateTime _now = DateTime.now();
  Timer? _timer;

  @override
  void initState() {
    super.initState();
    _timer = Timer.periodic(const Duration(seconds: 1), (_) {
      if (mounted) setState(() => _now = DateTime.now());
    });
  }

  @override
  void dispose() {
    _timer?.cancel();
    super.dispose();
  }

  @override
  Widget build(BuildContext context) => Text(_now.toIso8601String());
}

One-Shot Timer Leak

// Even Timer() (non-periodic) can leak if the widget is disposed before it fires
class _DelayedNavState extends State<DelayedNav> {
  Timer? _timer;

  @override
  void initState() {
    super.initState();
    _timer = Timer(const Duration(seconds: 3), () {
      // if widget is popped before 3s, this fires on a dead State
      Navigator.of(context).pushNamed('/home'); // context may be invalid!
    });
  }

  @override
  void dispose() {
    _timer?.cancel(); // cancel before timer fires
    super.dispose();
  }
}

4. ChangeNotifier Listener Not Removed

When you call notifier.addListener(callback), the ChangeNotifier stores the callback in an internal list. If that callback is a closure capturing this (the State), and you never call removeListener, the notifier keeps the State alive as long as the notifier itself is alive (often the app's lifetime for app-level services).

Leaked Version

class _SettingsScreenState extends State<SettingsScreen> {
  late ThemeNotifier _themeNotifier;

  @override
  void initState() {
    super.initState();
    _themeNotifier = context.read<ThemeNotifier>();
    // LEAK: adds a listener but never removes it
    _themeNotifier.addListener(_onThemeChanged);
  }

  void _onThemeChanged() {
    if (mounted) setState(() {});
  }

  // No dispose() override!
  // ThemeNotifier (app-level singleton) holds a reference to _onThemeChanged,
  // which is a closure bound to _SettingsScreenState.
  // _SettingsScreenState will never be GC'd.
}

Fixed Version

class _SettingsScreenState extends State<SettingsScreen> {
  late ThemeNotifier _themeNotifier;

  @override
  void initState() {
    super.initState();
    _themeNotifier = context.read<ThemeNotifier>();
    _themeNotifier.addListener(_onThemeChanged);
  }

  void _onThemeChanged() {
    if (mounted) setState(() {});
  }

  @override
  void dispose() {
    _themeNotifier.removeListener(_onThemeChanged); // critical
    super.dispose();
  }
}
Note
This is why Consumer and context.watch() in the provider package are safer — the package manages addListener/removeListener for you through the InheritedWidget subscription mechanism. Manual addListener is the dangerous path.

ValueNotifier Pattern (same risk)

// ValueNotifier is a ChangeNotifier — same rules apply
class _CounterState extends State<CounterWidget> {
  // If counterNotifier is from a parent / service locator, you must remove listener
  final ValueNotifier<int> counterNotifier;

  _CounterState({required this.counterNotifier});

  void _rebuild() { if (mounted) setState(() {}); }

  @override
  void initState() {
    super.initState();
    counterNotifier.addListener(_rebuild);
  }

  @override
  void dispose() {
    counterNotifier.removeListener(_rebuild);
    super.dispose();
  }
}

5. GlobalKey Retaining Large Subtrees

A GlobalKey maintains a reference to its associated Element in a static registry (GlobalKey._registry). As long as the key object exists, the element (and the entire subtree beneath it, including all its State objects) is kept alive. This is intentional for GlobalKey's purpose (state preservation across tree positions), but becomes a leak when keys outlive their useful life.

The Problem

class AppState {
  // Anti-pattern: GlobalKey stored at app level, widget may have been removed
  static final GlobalKey<FormState> checkoutFormKey = GlobalKey<FormState>();
  static final GlobalKey<ScaffoldState> scaffoldKey = GlobalKey<ScaffoldState>();
}

// These keys hold Element references alive permanently.
// If CheckoutScreen is removed from the tree, its FormState,
// all TextFormField states, and all controllers inside them remain in memory.

The Fix

// Create GlobalKeys at the lowest scope possible — usually the StatefulWidget's State
class _CheckoutScreenState extends State<CheckoutScreen> {
  // Key is created and GC'd with this State — correct scope
  final _formKey = GlobalKey<FormState>();

  @override
  Widget build(BuildContext context) => Form(
    key: _formKey,
    child: /* ... */,
  );
}
// When _CheckoutScreenState is disposed, _formKey goes out of scope.
// GlobalKey._registry removes the entry.
// The subtree element is GC'd normally.

RepaintBoundary Key Anti-Pattern

// Also seen with RepaintBoundary for screenshot capture
// BAD: stored at module level
final _screenshotKey = GlobalKey(); // lives forever

// GOOD: created per usage
class _ExportableWidgetState extends State<ExportableWidget> {
  final _key = GlobalKey(); // scoped to this State
  // ...
}

6. Unbounded Image Cache

Flutter's PaintingBinding.instance.imageCache is a global LRU cache that stores decoded image data. By default it caches up to 1000 images or 100 MB. In apps that display many unique images (e.g., a feed), the cache can grow to its limit and stay there, holding megabytes of decoded RGBA pixel data (not compressed) in memory.

Examining and Configuring the Cache

// Understanding the defaults
final cache = PaintingBinding.instance.imageCache;
print(cache.maximumSize);        // 1000 (number of images)
print(cache.maximumSizeBytes);   // 104857600 (100 MB in bytes)
print(cache.currentSize);        // current number of cached images
print(cache.currentSizeBytes);   // current bytes used

// In main() or app initialization, configure to appropriate limits
void main() {
  WidgetsFlutterBinding.ensureInitialized();

  // Reduce cache for memory-constrained devices
  PaintingBinding.instance.imageCache.maximumSizeBytes = 50 << 20; // 50 MB
  PaintingBinding.instance.imageCache.maximumSize = 100;

  runApp(const MyApp());
}

Clearing Cache on Memory Pressure

class _AppState extends State<App> {
  @override
  void didChangeAppLifecycleState(AppLifecycleState state) {
    if (state == AppLifecycleState.paused) {
      // App backgrounded — free image memory
      PaintingBinding.instance.imageCache.clear();
      PaintingBinding.instance.imageCache.clearLiveImages();
    }
  }
}

// Also available via flutter_cache_manager for network images:
// DefaultCacheManager().emptyCache();

Per-Widget Cache Eviction

// When navigating away from a heavy image screen:
@override
void dispose() {
  // Evict specific images from cache when their screen leaves
  const AssetImage('assets/large_hero.jpg').evict();
  NetworkImage('https://example.com/banner.jpg').evict();
  super.dispose();
}
Warning
imageCache.currentSizeBytes counts decoded RGBA bytes, not file sizes. A 200KB JPEG at 2000x3000 pixels decodes to approximately 24 MB of RGBA data. Caching 10 such images uses 240 MB — well within the default limit but catastrophic on a 512 MB RAM device.

7. Platform Channel Callbacks Capturing BuildContext

Platform channel callbacks are invoked on the platform thread (or via the binary messenger on the UI thread), potentially after the widget has been disposed. Capturing BuildContext or this (the State) in these callbacks is dangerous: the callback keeps the objects alive and may operate on a stale/invalid context.

The Problem

class _PaymentScreenState extends State<PaymentScreen> {
  static const _channel = MethodChannel('com.app/payment');

  @override
  void initState() {
    super.initState();
    // LEAK + BUG: closure captures `context` and `this`
    _channel.setMethodCallHandler((call) async {
      if (call.method == 'paymentResult') {
        // context may be invalid if screen was popped
        Navigator.of(context).pushReplacementNamed('/success');
        // `this` prevents GC of _PaymentScreenState
      }
    });
  }
  // No cleanup! Handler is registered globally on the channel.
}

Fixed Version

class _PaymentScreenState extends State<PaymentScreen> {
  static const _channel = MethodChannel('com.app/payment');

  @override
  void initState() {
    super.initState();
    _channel.setMethodCallHandler(_handlePaymentResult);
  }

  Future<void> _handlePaymentResult(MethodCall call) async {
    if (call.method == 'paymentResult' && mounted) {
      final result = call.arguments as String;
      if (result == 'success') {
        Navigator.of(context).pushReplacementNamed('/success');
      }
    }
  }

  @override
  void dispose() {
    // Remove the handler — set to null to unregister
    _channel.setMethodCallHandler(null);
    super.dispose();
  }
}
Part 2
Detection & Tooling
How to find leaks before and after they ship to production.

Detection: DevTools Memory Tab

Heap Snapshot Workflow

  1. Open Flutter DevTools → Memory tab
  2. Navigate to the screen you suspect leaks
  3. Take a heap snapshot (click "Snapshot")
  4. Navigate away from the screen (pop it)
  5. Trigger GC manually (click "GC" in DevTools)
  6. Take a second snapshot
  7. Compare: look for instances of your State subclass still present
  8. Click on the class in the snapshot → "Retaining path" reveals the reference chain keeping it alive

Allocation Tracker

// Mark allocation tracking programmatically for precise measurement
import 'dart:developer' as developer;

void startTracking() {
  developer.startIsolatePauseCapability(); // not common
  // More useful: DevTools Memory > Track Allocations toggle
}

// In code, annotate sections for timeline correlation:
developer.Timeline.startSync('HeavyOperation');
doHeavyWork();
developer.Timeline.finishSync();

Detection: leak_tracker Package

The leak_tracker package (pub.dev, official Flutter team) instruments your dispose() calls and alerts when objects are not disposed within an expected time window after becoming unreachable.

Setup

# pubspec.yaml
dev_dependencies:
  leak_tracker: ^10.0.0
  leak_tracker_flutter_testing: ^3.0.0
// In test setup or debug main:
import 'package:leak_tracker_flutter_testing/leak_tracker_flutter_testing.dart';

void main() {
  LeakTesting.settings = LeakTesting.settings.withIgnoredAll(); // baseline
  // Then enable for specific types:
  LeakTesting.settings = LeakTesting.settings
      .withTracked(allNotDisposed: true, allNotGCed: true);

  runApp(const MyApp());
}

// In tests:
testWidgets('no leaks', (tester) async {
  await tester.pumpWidget(const MyScreen());
  await tester.pumpAndSettle();
  // Navigate away
  await tester.pumpWidget(const SizedBox());
  await tester.pumpAndSettle();
  // leak_tracker will fail the test if leaks detected
}, experimentalLeakTesting: LeakTesting.settings);

Detection: WeakReference and Expando

Dart provides WeakReference<T> (Dart 2.17+) and Expando<T> for writing your own leak detection utilities. A WeakReference does not prevent GC of its target.

// Manual leak check pattern in debug builds
class LeakDetector {
  static final List<WeakReference<Object>> _tracked = [];

  static void track(Object obj) {
    if (kDebugMode) _tracked.add(WeakReference(obj));
  }

  static void checkForLeaks() {
    // Force GC (not guaranteed, but often works in tests)
    for (var i = 0; i < 5; i++) {
      // Allocate temporary large objects to trigger GC
      List.generate(1000000, (i) => i);
    }

    final alive = _tracked.where((ref) => ref.target != null).toList();
    if (alive.isNotEmpty) {
      debugPrint('LEAK DETECTED: ${alive.length} objects still alive');
      for (final ref in alive) {
        debugPrint('  ${ref.target.runtimeType}');
      }
    }
  }
}

// Usage in State:
@override
void initState() {
  super.initState();
  LeakDetector.track(this); // track this State instance
}

// After navigation away:
// LeakDetector.checkForLeaks(); // should show 0 tracked objects

Expando for Metadata Without Strong References

// Expando attaches metadata to objects without creating strong references
final Expando<String> _debugNames = Expando();

class TrackedAnimationController extends AnimationController {
  TrackedAnimationController({required super.vsync, required super.duration, String? debugName}) {
    if (debugName != null) _debugNames[this] = debugName;
  }
}

// _debugNames does not prevent GC of the AnimationController
// It's automatically cleaned up when the controller is collected

GC Roots Analysis

Understanding GC roots is essential for Staff-level debugging. Dart's GC starts from roots and traces all reachable objects. Objects reachable from any root are not collected. Common GC roots in Flutter:

Root Type Examples in Flutter Leak Risk
Stack frames Local variables in running functions Low — cleared when function returns
Static fields GlobalKey._registry, PaintingBinding.imageCache High — permanent roots
Dart VM isolate globals Top-level variables, singletons High — app lifetime
Native references MethodChannel handlers, platform callbacks Medium — must be explicitly cleared
Stream subscriptions StreamController._subscriptions High — most common leak
Timer/Ticker callbacks Dart event loop, SchedulerBinding High — runs until cancelled
Staff Interview Angle
Be prepared to discuss team-scale prevention. Individual fixes are L4/L5. L6 answers cover: "How do you prevent this class from ever being introduced?" Options: lint rules (prefer_void_to_null doesn't help here, but custom lint via dart_code_metrics), mandatory code review checklists, automated leak testing in CI (using leak_tracker in integration tests), and architectural patterns that make leaks structurally impossible (e.g., Riverpod's auto-dispose providers, which automatically cancel the subscription when no widgets are watching).
