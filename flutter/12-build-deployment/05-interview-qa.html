Build modes · Flavors · Code signing · CI/CD · App size · build_runner

Part 1 — L4 / L5 Standard
Core Build & Release Concepts

Foundational questions on Flutter's build system, environments, signing, and tooling that every Flutter engineer at L4+ should answer without hesitation.

Standard — L4
What are the three Flutter build modes and what is different about each?
ModeCommandCompilationOptimizationsDev tools
Debugflutter runJIT (Just-In-Time)None — fast recompile, hot reloadDevTools, hot reload, hot restart, Observatory, asserts enabled
Profileflutter run --profileAOT (Ahead-Of-Time)Most release optimizations enabledDevTools performance overlay enabled; no hot reload
Releaseflutter build apk/ipaAOTFull: tree shaking, minification, obfuscation (opt-in)No DevTools, no Observatory, asserts disabled
Key distinction: Debug uses JIT so hot reload works (the VM can swap out code at runtime). Profile and Release use AOT — the Dart code is compiled to native machine code ahead of time, which is faster at runtime but cannot be swapped live. Always use Profile mode for performance measurement — Debug mode's JIT overhead makes it unrepresentative.
Interviewers often ask why you can't performance-test in debug mode. The answer is JIT overhead — debug is significantly slower than release because it prioritizes reload speed over runtime performance.
Standard — L4
What is --dart-define and how do you use it?
--dart-define injects compile-time constants into the Dart build. Values are embedded at compile time — they are not environment variables read at runtime. This makes them slightly more secure (not in the binary as a plain string, though not truly secret) and available without any runtime file loading. 
# Pass at build time
flutter run --dart-define=API_URL=https://api.staging.example.com
flutter build apk --dart-define=API_URL=https://api.prod.example.com --dart-define=ANALYTICS_KEY=abc123

# Read in Dart code
const apiUrl = String.fromEnvironment('API_URL', defaultValue: 'https://api.dev.example.com');
const analyticsKey = String.fromEnvironment('ANALYTICS_KEY');
const isProd = bool.fromEnvironment('IS_PROD', defaultValue: false);
From Flutter 3.7+, you can use --dart-define-from-file=config.json to pass a JSON file of defines instead of repeating --dart-define for every key. This is much cleaner for many config values. 
# config/staging.json
{ "API_URL": "https://api.staging.example.com", "IS_PROD": false }

flutter run --dart-define-from-file=config/staging.json
Mention --dart-define-from-file if you know about it — it shows awareness of recent Flutter tooling improvements.
Standard — L4
How do you set up dev/staging/prod environments in Flutter?
Two main approaches:

Option 1 — Flavors (recommended for full separation): Flutter flavors map to Android product flavors and iOS schemes. Each flavor has its own app ID, app name, icon, and Firebase config. You select a flavor at build time: flutter run --flavor dev -t lib/main_dev.dart. This gives you separate app installations on the same device — essential for QA testing staging and production simultaneously.

Option 2 — dart-define (simpler, single app ID): Pass environment-specific values via --dart-define or --dart-define-from-file. No separate app IDs — the same binary is built for different environments. Simpler to set up, but you cannot install dev and prod simultaneously on the same device.

Best practice: use flavors for any app that has QA/beta testers (they need to run staging alongside production). Use dart-define for simpler projects or when flavor setup overhead is not justified.

Common flavor structure: 
lib/
  main_dev.dart      # void main() => runApp(App(env: Env.dev));
  main_staging.dart
  main_prod.dart

flutter run --flavor dev -t lib/main_dev.dart
flutter build apk --flavor prod -t lib/main_prod.dart --release
Be ready to explain why flavors require separate entry points (main_dev.dart etc.) — it's because Dart's compile-time constants and the flavor name are both needed at the app level.
Standard — L5
What does build_runner do and how does it decide which files to regenerate?
build_runner is a build system for Dart that runs code generators (builders). It reads your source files, applies builder transformations (e.g., json_serializable, freezed, drift, injectable, pigeon), and writes generated .g.dart and .freezed.dart files.

Incremental build / change detection: build_runner tracks a build cache (stored in .dart_tool/build/). For each source file, it stores a hash of the file content and the hash of its generated output. On the next run, it compares the current hash to the cached hash. If unchanged, the builder is skipped. If changed, the builder runs and the output is regenerated.

Dependency tracking: builders declare which files they read (inputs) and write (outputs). If an input changes, all outputs depending on it are invalidated and regenerated.

Common commands: 
# One-time build
dart run build_runner build

# Delete cache and rebuild everything (fixes stale cache issues)
dart run build_runner build --delete-conflicting-outputs

# Watch mode — rebuild on file save
dart run build_runner watch --delete-conflicting-outputs
Stale cache issues: if generated files seem wrong, --delete-conflicting-outputs forces regeneration of any file where the generated content conflicts with what's on disk.
Knowing that build_runner uses content hashing (not file timestamps) is the internal detail that distinguishes a thorough answer.
Standard — L5
What is the difference between the upload key and the release key in Play App Signing?
Play App Signing (mandatory for new apps since 2021) introduces a two-key model:
  • Upload key: The key you control. You sign your AAB with this key before uploading to the Play Console. Google verifies the upload key to authenticate that the upload came from you. This is the key stored in your CI keystore.
  • Release key: The key Google controls. After you upload a signed AAB, Google strips your upload signature, re-signs the final APK/AAB with its release key, and delivers that to users. Users' devices verify the release key, not the upload key.
Practical consequences:
  • Losing the upload key: contact Google support to reset it. Your app continues to work for existing users since their devices trust the release key (which Google still holds).
  • The release key fingerprint (SHA-256) is what you register with Firebase, Google Sign-In, Maps SDK, and other services. Using the upload key fingerprint in these services is a common misconfiguration that causes auth failures.
  • The release key fingerprint is found in Play Console under Setup > App integrity > App signing.
The SHA-256 fingerprint misconfiguration (using upload key instead of release key in Firebase) is a production bug that stumps many teams. Knowing this detail shows real-world Android signing experience.
Part 2 — L6 / Staff Deep Dive
Internals, Architecture & Diagnosis

Questions requiring deep knowledge of the Flutter build pipeline, AOT compilation, signing internals, and the ability to diagnose real production failures.

Deep — L6
Walk through exactly what happens during flutter build apk --release — from Dart source to final APK.
  1. Pub resolution: The build system reads pubspec.lock and ensures all dependencies are resolved and available in the pub cache. No network calls if the cache is warm.
  2. build_runner (if needed): If any source files have changed that affect codegen (e.g., files annotated with @JsonSerializable), build_runner regenerates the .g.dart files. In a clean CI environment, all builders run.
  3. Dart AOT compilation: The Dart compiler (dart compile / gen_snapshot) compiles all Dart code — your app + all packages — to native ARM/x86 machine code using AOT. This produces a .so shared library (e.g., libapp.so for ARM64). Tree shaking eliminates unreachable Dart code at this stage.
  4. Flutter engine: The prebuilt Flutter engine (libflutter.so) is pulled from the Flutter SDK's artifact cache. This is the C++ runtime that handles rendering, platform channels, and Dart VM hosting.
  5. Asset bundling: Fonts, images, and other assets declared in pubspec.yaml are copied into the flutter_assets/ directory. Icon fonts are subset to include only used glyphs (if font tree-shaking is enabled).
  6. Gradle build: Gradle compiles the Android Kotlin/Java wrapper code, merges the AndroidManifest.xml from all plugins, processes resources (aapt2), and packages everything into an unsigned APK. The Dart .so and Flutter engine .so are included as native libraries.
  7. R8/ProGuard (if enabled): R8 runs on the Kotlin/Java bytecode (not Dart — Dart is already native). It shrinks, obfuscates, and optimizes the Java bytecode for the plugin wrapper code and the Flutter embedding.
  8. APK signing: Gradle's signingConfigs (or jarsigner) signs the APK with your upload keystore. The signature is stored in the APK's META-INF/.
  9. zipalign: The APK is zip-aligned so that uncompressed data is aligned on 4-byte boundaries, enabling efficient memory mapping on device.
Final output: a signed, zipaligned app-release.apk (or app-release.aab for App Bundle).
The key steps interviewers listen for: AOT compilation via gen_snapshot, tree shaking at the Dart level, R8 at the Java/Kotlin level, and the two separate native .so files (libapp + libflutter). Most engineers only know "Gradle builds it."
Deep — L6
How does AOT tree shaking work and what can break it?
Dart's AOT compiler performs whole-program tree shaking: it starts from entry points (main()) and traces all reachable code paths. Any class, method, or field that is not reachable from any entry point is eliminated from the final binary.

What tree shaking eliminates:
  • Unused classes and their methods.
  • Unused fields (for constant folding).
  • Unused icon glyphs from icon fonts (font tree shaking, Flutter 1.22+).
What breaks tree shaking (forces code to be retained):
  • Reflection / dart:mirrors: Not available in Flutter (by design). If it were, the compiler could not know at compile time which classes/methods would be accessed.
  • Dynamic dispatch via String-based method names: Calling a method by a string name at runtime (e.g., serialization frameworks that use reflection) prevents the compiler from tracing the call graph. This is why json_serializable generates explicit fromJson methods rather than using reflection.
  • @pragma('vm:entry-point'): This annotation marks a class or function as a tree-shaking root — it is retained even if not called from Dart. Used for code called from native (via FFI callbacks or MethodChannel handlers). Overuse bloats the binary.
  • Plugin registration: The generated plugin registrant code references all plugin classes by name, keeping them in the binary even if they are unused at runtime.
Auditing tree shaking: use flutter build apk --analyze-size to get a breakdown of binary size by package and class. Large unexpected entries often indicate tree shaking is not eliminating what you expect.
Mentioning @pragma('vm:entry-point') and its tree-shaking implications is a genuine L6 signal — it comes up in FFI and plugin authoring contexts.
Deep — L6
How would you reduce the size of a Flutter release APK?
App size reduction is multi-layered — Dart, assets, and native code each have separate levers.

1. Use App Bundle (AAB) instead of APK. Play Store delivers device-specific APKs from the AAB, so each user only downloads the native libraries and resources for their ABI and screen density. This alone typically reduces download size by 30–50%.

2. Split per ABI (--split-per-abi). If you must distribute APKs, generate separate APKs per ABI (arm64-v8a, armeabi-v7a, x86_64). Each is 30–40% smaller than a fat APK.

3. Enable R8 / minification and shrinking. In build.gradle: minifyEnabled true, shrinkResources true. Shrinks the Java/Kotlin plugin code.

4. Compress and resize assets. Images are often the largest asset contributor. Use WebP instead of PNG/JPEG (30–50% smaller). Use the correct resolution for the target density. Avoid bundling assets that are not used at startup — consider loading them from network on demand.

5. Audit icon font usage. Flutter's font tree shaking only eliminates unused glyphs if you declare the font in pubspec.yaml. If you are using a font with thousands of glyphs but only 20, tree shaking saves significant size. Verify it is working with --analyze-size.

6. Remove unused packages. Every package adds code. Audit pubspec.yaml — remove packages that are no longer needed. Some packages (e.g., Firebase Auth, Maps) add significant size.

7. Use --split-debug-info. Moves debug symbols out of the APK into a separate file. Reduces APK size by 10–20MB and is required for meaningful crash stack traces in production.

8. Deferred components (advanced). Flutter supports deferred loading of Dart code — rarely-used features can be loaded on demand from the Play Store's dynamic delivery. Significant complexity; only worthwhile for very large apps. 
# Analyze size breakdown
flutter build apk --release --analyze-size --target-platform android-arm64
Lead with AAB + split-per-abi as the highest-impact changes. Then assets (WebP). Then R8. Interviewers want a prioritized list, not an exhaustive one.
Deep — L6
What does --split-debug-info do and why is it useful in production?
In AOT mode, the compiled Dart native library (libapp.so) includes debugging symbols by default — function names, line numbers, and source file paths. These symbols are large (often 15–25 MB) and are what enable human-readable stack traces. But they bloat the binary delivered to users.

--split-debug-info=<dir> strips the debug symbols from libapp.so and writes them to a separate .symbols file in the specified directory. The APK is smaller, but crash stack traces from the store (Firebase Crashlytics, Play Console) will show obfuscated names (_kDartIsolateSnapshotInstructions+0x12345) unless you symbolicate them.

Symbolication: use the flutter symbolize command with the original .symbols file to convert obfuscated stack traces back to human-readable form: 
flutter symbolize \
  --debug-info=build/symbols/app.android-arm64.symbols \
  --input=obfuscated_stacktrace.txt
Production workflow:
  1. Build with --split-debug-info=build/symbols/ --obfuscate (obfuscate is optional but recommended for security).
  2. Archive the .symbols files with the build artifacts (tied to the build number).
  3. Upload the symbols to Firebase Crashlytics: firebase crashlytics:symbols:upload --app=APP_ID build/symbols/. Crashlytics then symbolicates crash reports automatically.
Without retaining the .symbols files, you cannot symbolicate crash reports from that build. This is a one-way door — you cannot regenerate the symbols after the fact.
The "you cannot regenerate symbols" point is critical. Teams that lose their symbols files lose the ability to debug crashes from that release. Archive them with the build.
Deep — L6
How does Fastlane match work? What does it store and where?
Fastlane match implements the "codesigning.guide" philosophy: one set of signing identities shared across the whole team, stored in a central location, encrypted at rest.

What it stores:
  • Apple Distribution / Development certificates (.cer files) — the public certificate from Apple.
  • The certificate's private keys (.p12 files, encrypted with the match password) — the private half of the signing keypair.
  • Provisioning profiles (.mobileprovision files, encrypted) — the profile tying your app ID + certificate + capabilities.
Where it stores them: by default, a private git repository. The files are OpenSSL-encrypted before being committed, using a team-shared password (MATCH_PASSWORD). Alternative storage backends: Google Cloud Storage, AWS S3, or a custom storage_mode.

How it works at sync time:
  1. match clones (or pulls) the certs repo.
  2. Decrypts the .p12 and .mobileprovision files using the match password.
  3. Imports the certificate + private key into the local Keychain.
  4. Copies the provisioning profiles to ~/Library/MobileDevice/Provisioning Profiles/.
  5. Xcode picks them up automatically by bundle ID and certificate type.
In CI (--readonly flag): match does not modify the certs repo — it only reads and installs. The setup_ci Fastlane action creates a temporary keychain for the CI session, preventing interference with any existing keychain.
The "encrypted .p12 in a private git repo, decrypted and imported into the local Keychain at sync time" description is the correct mechanical answer. Many engineers say "stores certs in git" without the encryption and Keychain import details.
Deep — L6
Your iOS build fails in CI with a provisioning profile error. What's your diagnosis process?
iOS provisioning profile errors are among the most frustrating CI failures because the error messages are often generic. Systematic diagnosis:

Step 1 — Read the exact error message. Common variants:
  • "No profiles for 'com.example.app' were found" — Xcode cannot find a matching profile.
  • "Provisioning profile doesn't include the entitlement" — profile is outdated; missing a capability added after the profile was created.
  • "Certificate is not trusted" — the certificate's root CA is not in the CI keychain.
  • "Code signing is required" — no signing configuration found at all.
Step 2 — Verify match sync succeeded. Check the Fastlane match output in CI logs. Did it successfully clone the certs repo? Did it decrypt and import the profile? Does the output show the correct profile name?

Step 3 — Check profile validity. Provisioning profiles expire (typically after 1 year for App Store profiles). Run fastlane match appstore locally (without --readonly) to regenerate expired profiles. Then push to the certs repo.

Step 4 — Check bundle ID match. The profile is tied to a specific bundle ID. If your PRODUCT_BUNDLE_IDENTIFIER in Xcode does not exactly match the profile's app ID, the profile won't be used.

Step 5 — Check entitlements. If you added a new capability (push notifications, Sign in with Apple, etc.) after the profile was last regenerated, the profile is missing the entitlement. Regenerate: fastlane match appstore --force.

Step 6 — Verify the CI keychain. setup_ci creates a temporary keychain. Verify the certificate was imported into it (not just the system keychain). Run security find-identity -v -p codesigning in the CI step to list available signing identities.

Step 7 — Check Xcode automatic signing vs manual. If Xcode is set to automatic signing in CI, it tries to download profiles directly from Apple (requires Apple ID credentials). For CI, always use manual signing configured via match.
Structured diagnosis (read error → verify match → check expiry → check bundle ID → check entitlements → check keychain) is the L6 answer. An L4 answer says "re-run match" without understanding why the failure occurred.
Deep — L6
How do you handle different google-services.json files per flavor?
Each Firebase project has its own google-services.json (Android) and GoogleService-Info.plist (iOS). For multi-flavor apps (dev/staging/prod), each flavor connects to a different Firebase project — so each needs its own config file.

Android — flavor-specific source sets: Android's Gradle build system supports per-flavor source directories. Place each config in the matching source set: 
android/app/
  src/
    dev/
      google-services.json      # Firebase dev project
    staging/
      google-services.json      # Firebase staging project
    main/                       # (prod, or use a 'prod' flavor dir)
      google-services.json      # Firebase prod project
Gradle automatically picks the correct file based on the active flavor. No extra configuration needed.

iOS — custom build phase script: iOS does not have a built-in flavor source set mechanism. The standard approach is a Run Script build phase in Xcode that copies the correct plist based on the scheme: 
# Xcode Run Script (Build Phases)
if [ "${CONFIGURATION}" == "Debug-dev" ]; then
  cp "${SRCROOT}/firebase/dev/GoogleService-Info.plist" \
     "${BUILT_PRODUCTS_DIR}/${PRODUCT_NAME}.app/GoogleService-Info.plist"
elif [ "${CONFIGURATION}" == "Release-prod" ]; then
  cp "${SRCROOT}/firebase/prod/GoogleService-Info.plist" \
     "${BUILT_PRODUCTS_DIR}/${PRODUCT_NAME}.app/GoogleService-Info.plist"
fi
In CI: store each google-services.json as a separate GitHub secret (base64-encoded). Decode the appropriate one based on the flavor being built: 
- name: Write Firebase config
  run: |
    echo "${{ secrets.GOOGLE_SERVICES_PROD }}" | base64 --decode \
      > android/app/src/prod/google-services.json
    echo "${{ secrets.GOOGLE_SERVICES_DEV }}" | base64 --decode \
      > android/app/src/dev/google-services.json
The Android flavor source set approach (placing the file in src/<flavorName>/) is the clean native solution — no scripts needed. Many teams use a script when the source set approach is simpler.

What is --split-per-abi and when would you use it?
--split-per-abi generates separate APKs for each CPU architecture (ABI) instead of a single fat APK containing native libraries for all ABIs. Flutter supports: armeabi-v7a (32-bit ARM), arm64-v8a (64-bit ARM, most modern devices), x86_64 (emulators, some Chromebooks).

A fat APK includes libflutter.so and libapp.so for all three ABIs. The Flutter engine alone is ~6–8 MB per ABI, so a fat APK is 18–24 MB larger than necessary for any individual device.

With --split-per-abi, each generated APK contains only one ABI's native libraries — the device downloads and installs only the one matching its architecture. 
flutter build apk --release --split-per-abi
# Generates:
# app-armeabi-v7a-release.apk  (~15 MB)
# app-arm64-v8a-release.apk    (~18 MB)
# app-x86_64-release.apk       (~20 MB)
When to use it:
  • When you distribute APKs directly (sideloading, Firebase App Distribution, direct download). Each tester/user gets the APK matching their device.
  • For Play Store, prefer AAB instead — Play Store handles ABI splitting automatically and more efficiently. --split-per-abi is not needed when using AAB.
When not to use it:
  • When distributing via Play Store with AAB — redundant and adds CI complexity.
  • When you need a single APK (e.g., enterprise MDM distribution that requires one file).
The key distinction: split-per-abi is for APK distribution. For Play Store, use AAB — Google handles the splitting for you. Confusing the two is a common answer in interviews.
