Why Flavors and Environments?

Production Flutter apps need multiple configurations: a dev environment pointing to a staging API with verbose logging, a staging environment for QA testing with a separate Firebase project, and a prod environment with the production API, crash reporting, and no debug features. Each environment may also need a different app ID (so they can coexist on the same device), different app icons, and different Firebase configuration files.

The solution is a combination of two mechanisms: platform flavors (Android productFlavors / iOS schemes) for native-side configuration, and --dart-define for Dart-side configuration. These work together but serve different purposes.

Android: productFlavors in build.gradle

// android/app/build.gradle
android {
    flavorDimensions "environment"

    productFlavors {
        dev {
            dimension "environment"
            applicationId "com.example.myapp.dev"
            versionNameSuffix "-dev"
            resValue "string", "app_name", "MyApp DEV"
            buildConfigField "String", "API_BASE_URL", '"https://api-dev.example.com"'
        }

        staging {
            dimension "environment"
            applicationId "com.example.myapp.staging"
            versionNameSuffix "-staging"
            resValue "string", "app_name", "MyApp Staging"
            buildConfigField "String", "API_BASE_URL", '"https://api-staging.example.com"'
        }

        prod {
            dimension "environment"
            applicationId "com.example.myapp"
            resValue "string", "app_name", "MyApp"
            buildConfigField "String", "API_BASE_URL", '"https://api.example.com"'
        }
    }

    buildTypes {
        debug {
            debuggable true
            signingConfig signingConfigs.debug
        }
        release {
            minifyEnabled true
            proguardFiles getDefaultProguardFile('proguard-android-optimize.txt'), 'proguard-rules.pro'
            signingConfig signingConfigs.release
        }
    }
}

Per-Flavor Firebase Configuration

Each flavor needs its own google-services.json pointing to the correct Firebase project:

# Directory structure for per-flavor Firebase config:
android/app/
├── src/
│   ├── dev/
│   │   └── google-services.json        # Dev Firebase project
│   ├── staging/
│   │   └── google-services.json        # Staging Firebase project
│   └── prod/
│       └── google-services.json        # Production Firebase project
├── main/
│   └── (shared resources)
└── build.gradle

Gradle automatically picks the correct google-services.json from the flavor's source set when building. The google-services Gradle plugin reads it from src/{flavorName}/google-services.json.

iOS: Schemes and xcconfig Files

Setting Up iOS Schemes

In Xcode, each environment needs its own scheme and optionally its own configuration (Debug-Dev, Release-Dev, etc.). The typical setup:

  1. Open Xcode → Product → Scheme → Manage Schemes
  2. Duplicate the existing "Runner" scheme, rename to "Runner-Dev", "Runner-Staging", "Runner-Prod"
  3. Each scheme can specify a different build configuration
  4. Add new configurations: Product → Scheme → Edit Scheme → Build → + (Debug-Dev, Debug-Staging, Release-Dev, Release-Staging, Release-Prod)

xcconfig Files for Per-Environment iOS Config

# ios/Flutter/Dev.xcconfig
#include "Generated.xcconfig"
FLUTTER_TARGET=lib/main_dev.dart
DART_DEFINES=FLAVOR=dev,API_URL=https://api-dev.example.com
BUNDLE_ID_SUFFIX=.dev
APP_DISPLAY_NAME=MyApp DEV

# ios/Flutter/Prod.xcconfig
#include "Generated.xcconfig"
FLUTTER_TARGET=lib/main_prod.dart
DART_DEFINES=FLAVOR=prod,API_URL=https://api.example.com
APP_DISPLAY_NAME=MyApp

# ios/Runner/Info.plist — use xcconfig variables
<key>CFBundleDisplayName</key>
<string>$(APP_DISPLAY_NAME)</string>

<key>CFBundleIdentifier</key>
<string>$(PRODUCT_BUNDLE_IDENTIFIER)$(BUNDLE_ID_SUFFIX)</string>

--dart-define: Compile-Time Dart Configuration

Basic Usage

--dart-define passes key-value pairs to the Dart compiler, making them available at compile time via String.fromEnvironment(), bool.fromEnvironment(), and int.fromEnvironment(). These values are baked into the compiled binary — they're constants resolved at compile time, not runtime environment variables.

# Running with --dart-define
flutter run --flavor dev \
  --dart-define=FLAVOR=dev \
  --dart-define=API_BASE_URL=https://api-dev.example.com \
  --dart-define=ENABLE_LOGS=true \
  -t lib/main_dev.dart

# Building release with --dart-define
flutter build apk --release --flavor prod \
  --dart-define=FLAVOR=prod \
  --dart-define=API_BASE_URL=https://api.example.com \
  --dart-define=ENABLE_LOGS=false

Accessing dart-define Values in Dart

// lib/config/env.dart

class Env {
  // String.fromEnvironment reads the --dart-define value at compile time
  // The second argument is the default value if not defined
  static const String flavor = String.fromEnvironment(
    'FLAVOR',
    defaultValue: 'dev',
  );

  static const String apiBaseUrl = String.fromEnvironment(
    'API_BASE_URL',
    defaultValue: 'https://api-dev.example.com',
  );

  static const bool enableLogs = bool.fromEnvironment(
    'ENABLE_LOGS',
    defaultValue: true,
  );

  static const int requestTimeoutSeconds = int.fromEnvironment(
    'REQUEST_TIMEOUT_SECONDS',
    defaultValue: 30,
  );

  // Convenience getters
  static bool get isDev => flavor == 'dev';
  static bool get isProduction => flavor == 'prod';
}

// Usage anywhere in your code:
if (Env.enableLogs) {
  logger.d('Request to ${Env.apiBaseUrl}');
}
Performance Tip

Because String.fromEnvironment() values are compile-time constants, the Dart AOT compiler can dead-code-eliminate branches based on them. if (Env.enableLogs) { ... } in a release build with ENABLE_LOGS=false will have the entire branch eliminated — the logging code is not even compiled into the binary. This makes --dart-define ideal for feature flags that should genuinely disappear from production builds.

--dart-define-from-file: JSON Config Files

For many define values, typing them all on the command line is cumbersome. Flutter 3.7+ supports --dart-define-from-file which reads a JSON file and automatically expands all key-value pairs as dart-defines:

# config/dev.json
{
  "FLAVOR": "dev",
  "API_BASE_URL": "https://api-dev.example.com",
  "ENABLE_LOGS": "true",
  "FIREBASE_PROJECT_ID": "myapp-dev",
  "REQUEST_TIMEOUT_SECONDS": "30",
  "ENABLE_CRASHLYTICS": "false",
  "FEATURE_DARK_MODE": "true",
  "FEATURE_BETA_CHECKOUT": "true"
}

# config/prod.json
{
  "FLAVOR": "prod",
  "API_BASE_URL": "https://api.example.com",
  "ENABLE_LOGS": "false",
  "FIREBASE_PROJECT_ID": "myapp-prod",
  "REQUEST_TIMEOUT_SECONDS": "15",
  "ENABLE_CRASHLYTICS": "true",
  "FEATURE_DARK_MODE": "true",
  "FEATURE_BETA_CHECKOUT": "false"
}

# Use --dart-define-from-file instead of many --dart-define flags
flutter run --flavor dev \
  --dart-define-from-file=config/dev.json \
  -t lib/main_dev.dart

flutter build appbundle --release --flavor prod \
  --dart-define-from-file=config/prod.json
Security Warning

Do NOT put secrets (API keys, private keys, OAuth secrets) in --dart-define or config JSON files committed to source control. dart-define values are baked into the binary and can be extracted by decompiling the app. For secrets, use platform-specific secure storage (Android Keystore, iOS Keychain) or server-side secret management. Config JSON files should only contain non-sensitive configuration that you'd be comfortable having extracted from the APK.

AppConfig Environment Class Pattern

// lib/config/app_config.dart

/// Centralized, type-safe app configuration
/// Reads from compile-time constants (dart-define) with sensible defaults
class AppConfig {
  static final AppConfig _instance = AppConfig._internal();
  factory AppConfig() => _instance;
  AppConfig._internal();

  // Compile-time constants from --dart-define
  final String flavor = const String.fromEnvironment('FLAVOR', defaultValue: 'dev');
  final String apiBaseUrl = const String.fromEnvironment(
    'API_BASE_URL',
    defaultValue: 'https://api-dev.example.com',
  );
  final bool enableCrashlytics = const bool.fromEnvironment('ENABLE_CRASHLYTICS');
  final bool enableLogs = const bool.fromEnvironment('ENABLE_LOGS', defaultValue: true);
  final int requestTimeoutSeconds = const int.fromEnvironment(
    'REQUEST_TIMEOUT_SECONDS',
    defaultValue: 30,
  );

  // Feature flags
  final bool featureBetaCheckout = const bool.fromEnvironment('FEATURE_BETA_CHECKOUT');
  final bool featureDarkMode = const bool.fromEnvironment('FEATURE_DARK_MODE', defaultValue: true);

  // Derived properties
  bool get isProduction => flavor == 'prod';
  bool get isDevelopment => flavor == 'dev';
  bool get isStaging => flavor == 'staging';

  Duration get requestTimeout => Duration(seconds: requestTimeoutSeconds);

  @override
  String toString() => 'AppConfig(flavor: $flavor, apiBaseUrl: $apiBaseUrl)';
}

// lib/main_dev.dart
void main() {
  // Validate config at startup in debug/staging
  if (!AppConfig().isProduction) {
    debugPrint('Starting with config: ${AppConfig()}');
  }
  runApp(const MyApp());
}

Separate Entry Points per Flavor

// lib/main_dev.dart
import 'package:flutter/material.dart';
import 'app.dart';

void main() async {
  WidgetsFlutterBinding.ensureInitialized();
  // Dev-specific initialization
  await Firebase.initializeApp(options: DevFirebaseOptions.currentPlatform);
  runApp(const MyApp());
}

// lib/main_staging.dart
void main() async {
  WidgetsFlutterBinding.ensureInitialized();
  await Firebase.initializeApp(options: StagingFirebaseOptions.currentPlatform);
  runApp(const MyApp());
}

// lib/main_prod.dart
void main() async {
  WidgetsFlutterBinding.ensureInitialized();
  await Firebase.initializeApp(options: ProdFirebaseOptions.currentPlatform);
  FlutterError.onError = FirebaseCrashlytics.instance.recordFlutterFatalError;
  runApp(const MyApp());
}

CI Matrix Builds

# .github/workflows/build.yml
name: Flutter CI Matrix Build

on:
  push:
    branches: [main, develop]

jobs:
  build:
    strategy:
      matrix:
        include:
          # Android builds
          - platform: android
            flavor: dev
            target: lib/main_dev.dart
            config: config/dev.json
            build_command: "flutter build apk --flavor dev"
          - platform: android
            flavor: prod
            target: lib/main_prod.dart
            config: config/prod.json
            build_command: "flutter build appbundle --flavor prod --release"
          # iOS builds
          - platform: ios
            flavor: Dev
            target: lib/main_dev.dart
            config: config/dev.json
            build_command: "flutter build ios --flavor Dev --no-codesign"
          - platform: ios
            flavor: Prod
            target: lib/main_prod.dart
            config: config/prod.json
            build_command: "flutter build ios --flavor Prod --release"
            needs_signing: true

    runs-on: ${{ matrix.platform == 'ios' && 'macos-latest' || 'ubuntu-latest' }}

    steps:
      - uses: actions/checkout@v4

      - uses: subosito/flutter-action@v2
        with:
          flutter-version: '3.22.0'
          channel: 'stable'
          cache: true

      - run: flutter pub get

      - run: flutter test   # Always run tests before building

      - run: |
          ${{ matrix.build_command }} \
            -t ${{ matrix.target }} \
            --dart-define-from-file=${{ matrix.config }}
Staff Design Pattern

The cleanest flavor architecture separates concerns: (1) Platform flavors own what the OS sees — app ID, app name, icons, native Firebase config. (2) --dart-define owns what Dart code sees — API URLs, feature flags, log level. (3) Separate entry points (main_dev.dart, main_prod.dart) own initialization order differences. This separation means changing a feature flag doesn't require modifying Gradle or Xcode configs, and changing the app ID doesn't require touching Dart code. Everything is in its correct layer.
