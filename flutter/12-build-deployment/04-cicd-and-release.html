04 — CI/CD, Code Signing & Release Management

GitHub Actions · Fastlane · Android signing · iOS signing · App distribution · Versioning

4.1 CI/CD Pipeline Structure

A production Flutter CI/CD pipeline has five conceptual stages that run in order. Each stage gates the next — a lint failure prevents a broken build from being signed and distributed.

StageWhat runsFails fast?
Lintflutter analyze, custom lint rules, dart format --set-exit-if-changedYes — formatting or analysis errors block everything
Testflutter test --coverage, coverage threshold checkYes — failing tests block the build
Buildflutter build apk/aab/ipa per platformYes — build errors block signing
SignAndroid: jarsigner / Gradle signingConfigs. iOS: Fastlane match + Xcode codesignYes — unsigned artifacts cannot be distributed
DistributeFirebase App Distribution, TestFlight, Play Store internal trackNo — partial distribution failure is acceptable

GitHub Actions workflow skeleton

name: Flutter CI

on:
  push:
    branches: [main, develop]
  pull_request:
    branches: [main]

jobs:
  lint-and-test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: subosito/flutter-action@v2
        with:
          flutter-version: '3.19.0'
          cache: true
      - run: flutter pub get
      - run: dart format --set-exit-if-changed .
      - run: flutter analyze --fatal-infos
      - run: flutter test --coverage
      - name: Check coverage threshold
        run: |
          COVERAGE=$(lcov --summary coverage/lcov.info 2>&1 | grep "lines" | awk '{print $2}' | tr -d '%')
          if (( $(echo "$COVERAGE < 80" | bc -l) )); then
            echo "Coverage $COVERAGE% is below 80% threshold"
            exit 1
          fi

  build-android:
    needs: lint-and-test
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-java@v4
        with:
          distribution: 'temurin'
          java-version: '17'
      - uses: subosito/flutter-action@v2
        with:
          flutter-version: '3.19.0'
          cache: true
      - name: Cache pub dependencies
        uses: actions/cache@v4
        with:
          path: ~/.pub-cache
          key: pub-${{ hashFiles('**/pubspec.lock') }}
      - run: flutter pub get
      - name: Decode keystore
        run: echo "${{ secrets.KEYSTORE_BASE64 }}" | base64 -d > android/upload.jks
      - name: Write key.properties
        run: |
          echo "storePassword=${{ secrets.STORE_PASSWORD }}" > android/key.properties
          echo "keyPassword=${{ secrets.KEY_PASSWORD }}" >> android/key.properties
          echo "keyAlias=${{ secrets.KEY_ALIAS }}" >> android/key.properties
          echo "storeFile=upload.jks" >> android/key.properties
      - run: flutter build appbundle --release
      - uses: actions/upload-artifact@v4
        with:
          name: android-release-aab
          path: build/app/outputs/bundle/release/app-release.aab

  build-ios:
    needs: lint-and-test
    runs-on: macos-14
    steps:
      - uses: actions/checkout@v4
      - uses: subosito/flutter-action@v2
        with:
          flutter-version: '3.19.0'
          cache: true
      - run: flutter pub get
      - name: Install Fastlane
        run: gem install fastlane
      - name: Sync certs via Fastlane match
        run: fastlane match appstore --readonly
        env:
          MATCH_PASSWORD: ${{ secrets.MATCH_PASSWORD }}
          MATCH_GIT_URL: ${{ secrets.MATCH_GIT_URL }}
      - run: flutter build ipa --release --export-options-plist=ios/ExportOptions.plist
      - uses: actions/upload-artifact@v4
        with:
          name: ios-release-ipa
          path: build/ios/ipa/*.ipa

4.2 GitHub Actions for Flutter

Flutter action and caching

The subosito/flutter-action@v2 action installs a specific Flutter SDK version and optionally caches it. Setting cache: true caches the Flutter SDK between runs, saving 2–3 minutes per run. The pub cache is separate and should be cached with actions/cache keyed on pubspec.lock.

- uses: subosito/flutter-action@v2
  with:
    flutter-version: '3.19.0'
    channel: 'stable'
    cache: true

- run: flutter pub get

- run: flutter analyze

- run: flutter test --coverage

# Split per ABI for smaller APK downloads (not needed for AAB)
- run: flutter build apk --release --split-per-abi

# AAB for Play Store (preferred over APK for production)
- run: flutter build appbundle --release

Key GitHub Actions patterns

  • Matrix builds: use strategy.matrix to run Android and iOS builds in parallel, sharing the lint/test job as a prerequisite.
  • Secrets: store keystores, passwords, and API keys as GitHub Actions secrets. Access via ${{ secrets.MY_SECRET }}. Never print secrets — GitHub masks them, but be careful with base64-decoded output.
  • Artifacts: actions/upload-artifact stores build outputs so the distribute job can download them without rebuilding.
  • Concurrency: use concurrency groups to cancel in-progress runs when a new commit is pushed to the same branch — prevents queue buildup.
  • Environment protection rules: require manual approval for production releases via GitHub Environments.
# Concurrency — cancel in-progress runs on the same branch
concurrency:
  group: ${{ github.workflow }}-${{ github.ref }}
  cancel-in-progress: true

4.3 Fastlane for Flutter

Fastlane automates repetitive release tasks: building, signing, testing, and uploading. For Flutter teams, it is most valuable for iOS signing (match) and store uploads (pilot, supply).

Fastfile structure

# ios/fastlane/Fastfile

default_platform(:ios)

platform :ios do
  before_all do
    setup_ci if ENV['CI']  # Configures keychain for CI environments
  end

  desc "Sync certificates and profiles (readonly in CI)"
  lane :sync_certs do
    match(
      type: "appstore",
      readonly: is_ci,
      app_identifier: "com.example.myapp"
    )
  end

  desc "Build and upload to TestFlight"
  lane :beta do
    sync_certs
    increment_build_number(
      build_number: ENV['BUILD_NUMBER'] || latest_testflight_build_number + 1
    )
    build_app(
      workspace: "Runner.xcworkspace",
      scheme: "Runner",
      configuration: "Release",
      export_method: "app-store"
    )
    upload_to_testflight(
      skip_waiting_for_build_processing: true,
      changelog: "Beta build #{lane_context[SharedValues::BUILD_NUMBER]}"
    )
  end

  desc "Submit to App Store for review"
  lane :release do
    sync_certs
    build_app(
      workspace: "Runner.xcworkspace",
      scheme: "Runner",
      configuration: "Release",
      export_method: "app-store"
    )
    upload_to_app_store(
      submit_for_review: false,  # Manual review submission
      automatic_release: false
    )
  end
end

Android Fastfile

# android/fastlane/Fastfile

default_platform(:android)

platform :android do
  desc "Build and upload to Play Store internal testing"
  lane :internal do
    gradle(
      task: "bundle",
      build_type: "Release",
      project_dir: "../android"
    )
    upload_to_play_store(
      track: "internal",
      aab: "../build/app/outputs/bundle/release/app-release.aab",
      skip_upload_apk: true
    )
  end

  desc "Promote internal to production"
  lane :promote_to_production do
    upload_to_play_store(
      track: "internal",
      track_promote_to: "production",
      rollout: "0.1"  # 10% staged rollout
    )
  end
end

4.4 Android Code Signing

Generating an upload keystore

keytool -genkey -v \
  -keystore upload.jks \
  -alias upload \
  -keyalg RSA \
  -keysize 2048 \
  -validity 10000

Store this keystore securely — if you lose it and are not using Play App Signing, you cannot publish updates. Store the password in a password manager and in CI secrets.

key.properties (gitignored)

storePassword=your_store_password
keyPassword=your_key_password
keyAlias=upload
storeFile=/path/to/upload.jks

build.gradle signingConfigs

// android/app/build.gradle
def keystoreProperties = new Properties()
def keystorePropertiesFile = rootProject.file('key.properties')
if (keystorePropertiesFile.exists()) {
    keystoreProperties.load(new FileInputStream(keystorePropertiesFile))
}

android {
    signingConfigs {
        release {
            keyAlias keystoreProperties['keyAlias']
            keyPassword keystoreProperties['keyPassword']
            storeFile keystoreProperties['storeFile'] ?
                file(keystoreProperties['storeFile']) : null
            storePassword keystoreProperties['storePassword']
        }
    }
    buildTypes {
        release {
            signingConfig signingConfigs.release
            minifyEnabled true
            shrinkResources true
            proguardFiles getDefaultProguardFile('proguard-android-optimize.txt'),
                'proguard-rules.pro'
        }
    }
}

Play App Signing — the critical distinction

Play App Signing (now mandatory for new apps) separates two keys:

KeyWho holds itWhat it signsConsequence if lost
Upload keyYou (the developer)The AAB you upload to Play ConsoleRequest a key reset from Google — no user impact
Release keyGoogleThe final APK delivered to usersGoogle holds it — cannot be lost by you

The upload key is what your CI keystore signs. Google re-signs the final APK with its release key before delivering to users. Users never see the upload key. This means:

  • Losing the upload key is recoverable — request a reset from the Play Console.
  • Rotating the upload key requires contacting Google support (takes ~2 weeks).
  • The certificate fingerprint shown in Firebase, OAuth, and other integrations is the release key fingerprint, not the upload key fingerprint. This is a common misconfiguration.
Common Mistake
When configuring Firebase Auth, Google Sign-In, or Maps SDK, you must use the Play App Signing release certificate fingerprint from the Play Console (Setup > App Signing), not the fingerprint of your local upload keystore. This trips up almost every team the first time.

4.5 iOS Code Signing

iOS code signing has two components that must both be valid for a build to run on a device or be accepted by the App Store:

ComponentWhat it isIssued by
CertificateIdentifies you or your team. Private key lives in your Keychain.Apple (via Certificates, Identifiers & Profiles portal)
Provisioning ProfileTies your App Bundle ID to a certificate + allowed devices + entitlements (capabilities).Apple (via portal or Xcode)

Certificate types

  • Development: For running on devices during development. Registered device UDIDs must be in the profile.
  • Ad Hoc: For distributing to specific registered devices (up to 100/year). Used for beta testing without TestFlight.
  • App Store: For TestFlight and App Store submission. No device restriction.
  • Enterprise: For in-house distribution (requires Apple Developer Enterprise Program).

Fastlane match — team cert management

match is Fastlane's solution to the team cert management problem. Instead of each developer manually generating certs and profiles (which creates duplicates and confusion), match stores a single set of encrypted certificates and profiles in a git repository. Every team member (and CI) runs match to sync.

# First-time setup (creates certs and stores in git repo)
fastlane match appstore --generate_apple_certs true

# Developer machine: sync certs to local keychain
fastlane match development
fastlane match appstore

# CI: readonly sync (never modifies the git repo)
fastlane match appstore --readonly

# Matchfile configuration
git_url("https://github.com/yourorg/certs-repo")
storage_mode("git")
type("appstore")
app_identifier(["com.example.app"])
username("dev@example.com")

The certs repo is encrypted with a password (MATCH_PASSWORD). Anyone with the password and git access can sync certs. The private key (the sensitive part) is encrypted inside the repo — never exposed in plaintext.

Alternative Storage
match also supports Google Cloud Storage and S3 as storage backends instead of git, which avoids having cert data in a git history. Prefer GCS/S3 for large teams or stricter security requirements.

4.6 Versioning

Flutter's version string in pubspec.yaml:

version: 1.2.3+45
#        ^^^^^  ^^
#        semver build number
  • 1.2.3 — the semantic version displayed to users in the App Store / Play Store. Follow semver: major.minor.patch.
  • 45 — the build number. Must be a monotonically increasing integer. The App Store and Play Store use this to determine which build is newer. You cannot upload a build with a lower or equal build number than an existing one.

Automating the build number in CI

# Option 1: Use GitHub Actions run number (simple, works well)
- run: |
    flutter build appbundle --release \
      --build-number=${{ github.run_number }}

# Option 2: Derive from git tag (semantic, requires tagged releases)
- run: |
    VERSION=$(git describe --tags --abbrev=0 | tr -d 'v')
    BUILD=$(git rev-list --count HEAD)
    flutter build appbundle --release \
      --build-name=$VERSION \
      --build-number=$BUILD

Keeping pubspec.yaml version in sync

# Update pubspec.yaml version before building (useful for release branches)
- name: Set version
  run: |
    VERSION="1.2.3"
    BUILD="${{ github.run_number }}"
    sed -i "s/^version:.*/version: $VERSION+$BUILD/" pubspec.yaml

4.7 Distribution

Firebase App Distribution (internal/beta)

# Install Firebase CLI
npm install -g firebase-tools

# Distribute Android APK
firebase appdistribution:distribute build/app/outputs/apk/release/app-release.apk \
  --app "${{ secrets.FIREBASE_APP_ID_ANDROID }}" \
  --groups "internal-testers,qa-team" \
  --release-notes "Build ${{ github.run_number }}: ${{ github.event.head_commit.message }}"

# Distribute iOS IPA
firebase appdistribution:distribute build/ios/ipa/MyApp.ipa \
  --app "${{ secrets.FIREBASE_APP_ID_IOS }}" \
  --groups "internal-testers"

TestFlight (iOS beta)

# Via Fastlane pilot
fastlane pilot upload \
  --ipa "build/ios/ipa/MyApp.ipa" \
  --skip_waiting_for_build_processing true \
  --distribute_external false

# External beta distribution requires additional review by Apple

Play Store tracks

# Via Fastlane supply
fastlane supply \
  --track internal \
  --aab build/app/outputs/bundle/release/app-release.aab

# Promote from internal to production with staged rollout
fastlane supply \
  --track internal \
  --track_promote_to production \
  --rollout 0.05  # 5% staged rollout

4.8 Secrets Management

Secrets that must never be committed to source control:

  • *.jks — Android keystores
  • key.properties — keystore passwords and alias
  • GoogleService-Info.plist — iOS Firebase config (contains API keys)
  • google-services.json — Android Firebase config
  • *.p12, *.mobileprovision — iOS certificates and profiles (managed by match instead)
  • .env — any environment variable files with secrets

.gitignore entries

# Android signing
*.jks
*.keystore
android/key.properties

# Firebase
google-services.json
GoogleService-Info.plist

# Environment
.env
.env.*
!.env.example

# iOS certs (managed by Fastlane match)
*.p12
*.mobileprovision

Keystore in CI — base64 encode/decode pattern

# Encode your keystore for storage as a GitHub secret (run locally once):
base64 -i upload.jks | pbcopy  # macOS — copies to clipboard
# Paste the output as the KEYSTORE_BASE64 secret in GitHub

# In CI: decode back to a file before building
- name: Decode keystore
  run: echo "${{ secrets.KEYSTORE_BASE64 }}" | base64 --decode > android/app/upload.jks

# Clean up after build (good practice)
- name: Clean up keystore
  if: always()
  run: rm -f android/app/upload.jks

Per-flavor Firebase configs

# Store separate configs as secrets
# GOOGLE_SERVICES_DEV, GOOGLE_SERVICES_STAGING, GOOGLE_SERVICES_PROD

- name: Write Firebase config
  run: |
    echo "${{ secrets.GOOGLE_SERVICES_PROD }}" > android/app/src/prod/google-services.json
    echo "${{ secrets.GOOGLE_SERVICES_DEV }}" > android/app/src/dev/google-services.json
Staff-Level Insight
The most common CI/CD failure I've seen is codesigning — specifically expired provisioning profiles or a rotated upload key. Fastlane match solves the team cert problem: one source of truth for all certs, readonly sync in CI, no more "it works on my machine" cert issues. Play App Signing solves the Android key loss problem: your upload key is recoverable, and Google holds the release key. Both should be non-negotiable for any production Flutter app. If a team is not using match and Play App Signing, they are one expired cert or lost keystore away from being unable to ship.
