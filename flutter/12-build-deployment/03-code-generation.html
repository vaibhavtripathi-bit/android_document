build_runner Architecture

build_runner is Dart's build system for code generation. It provides a framework where Builders — plugins that transform input assets into output assets — are discovered, ordered, and executed. Unlike macro systems or annotation processors in other languages, build_runner operates on files and produces new files as outputs.

Core Concepts

  • Asset: Any file in your project — a .dart source file, a .json config, an image, or a generated .g.dart file. Assets are identified by a package-relative path.
  • Builder: A plugin that declares input extensions and output extensions. It reads input assets, processes them (via code analysis using package:analyzer), and writes output assets. Builders are declared in their package's build.yaml.
  • BuildStep: A single invocation of a Builder for a particular input. The BuildStep API provides methods to read inputs, resolve Dart libraries, and write outputs.
  • Build graph: build_runner constructs a dependency graph of all builders and their inputs/outputs. Builders that produce outputs consumed by other builders are ordered automatically.

How build_runner Decides What to Rebuild

build_runner uses a content-hash-based incremental build system. For each Builder invocation, it records:

  1. A hash of all input assets read during the build step
  2. The hashes of all output assets written
  3. The builder's configuration (from build.yaml)

On subsequent runs, it compares current input hashes to cached hashes. If nothing changed, the builder is skipped — its cached outputs are reused. Only builders whose inputs changed are re-invoked. This makes incremental --watch builds very fast: only changed files trigger regeneration.

# Running build_runner
dart run build_runner build          # One-time build
dart run build_runner watch          # Watch mode — rebuilds on file changes
dart run build_runner build --delete-conflicting-outputs  # Fix conflicting output errors

# In pubspec.yaml — build_runner is a dev dependency
dev_dependencies:
  build_runner: ^2.4.0
  json_serializable: ^6.7.0
  freezed: ^2.4.0
  riverpod_generator: ^2.3.0

build.yaml: Configuring Builders

# build.yaml (in your project root)
targets:
  $default:
    builders:
      # Configure json_serializable
      json_serializable:
        options:
          # Generate toJson / fromJson
          explicit_to_json: true
          # Include null fields in JSON output
          include_if_null: false
          # Field naming convention
          field_rename: snake

      # Configure Freezed
      freezed:
        options:
          # Generate copyWith
          copy_with: true
          # Generate when/map methods
          when: true

      # Restrict which files a builder runs on (improves performance)
      source_gen|combining_builder:
        generate_for:
          include:
            - lib/**/*.dart
          exclude:
            - lib/**/*.g.dart   # Don't run generator on generated files

Common Code Generators

json_serializable: JSON Serialization

// lib/models/product.dart
import 'package:json_annotation/json_annotation.dart';

part 'product.g.dart';   // Generated file

@JsonSerializable(explicitToJson: true)
class Product {
  final String id;
  final String name;
  final double price;

  @JsonKey(name: 'image_url')    // Custom JSON field name
  final String imageUrl;

  @JsonKey(includeIfNull: false) // Omit null from JSON
  final String? description;

  @JsonKey(fromJson: _dateFromJson, toJson: _dateToJson)
  final DateTime createdAt;

  const Product({
    required this.id,
    required this.name,
    required this.price,
    required this.imageUrl,
    this.description,
    required this.createdAt,
  });

  factory Product.fromJson(Map<String, dynamic> json) => _$ProductFromJson(json);
  Map<String, dynamic> toJson() => _$ProductToJson(this);

  static DateTime _dateFromJson(String date) => DateTime.parse(date);
  static String _dateToJson(DateTime date) => date.toIso8601String();
}

Freezed: Immutable Data Classes

Freezed generates immutable data classes with copyWith, value equality, toString, pattern matching (when/map), and sealed class support:

// lib/models/cart_item.dart
import 'package:freezed_annotation/freezed_annotation.dart';
import 'package:json_annotation/json_annotation.dart';

part 'cart_item.freezed.dart';
part 'cart_item.g.dart';

@freezed
class CartItem with _$CartItem {
  const factory CartItem({
    required String productId,
    required String name,
    required double price,
    @Default(1) int quantity,
    String? note,
  }) = _CartItem;

  factory CartItem.fromJson(Map<String, dynamic> json) => _$CartItemFromJson(json);
}

// Freezed also supports sealed classes (union types):
@freezed
sealed class CartState with _$CartState {
  const factory CartState.initial() = CartInitial;
  const factory CartState.loading() = CartLoading;
  const factory CartState.loaded({required List<CartItem> items}) = CartLoaded;
  const factory CartState.error({required String message}) = CartError;
}

// Usage with pattern matching:
Widget build(BuildContext context) {
  return state.when(
    initial: () => const SizedBox.shrink(),
    loading: () => const CircularProgressIndicator(),
    loaded: (items) => CartListView(items: items),
    error: (message) => ErrorWidget(message),
  );
}

riverpod_generator: Auto-Generating Providers

// lib/providers/products_provider.dart
import 'package:riverpod_annotation/riverpod_annotation.dart';

part 'products_provider.g.dart';

// Simple FutureProvider
@riverpod
Future<List<Product>> products(ProductsRef ref) async {
  return ref.watch(productsRepositoryProvider).fetchProducts();
}

// AsyncNotifier provider (equivalent of StateNotifier)
@riverpod
class CartNotifier extends _$CartNotifier {
  @override
  List<CartItem> build() => [];

  void addItem(CartItem item) {
    state = [...state, item];
  }

  void removeItem(String productId) {
    state = state.where((i) => i.productId != productId).toList();
  }
}

build_runner Performance

Incremental Watch Mode

In development, always use dart run build_runner watch rather than repeatedly running build. Watch mode keeps build_runner running in the background, detecting file changes via filesystem events and triggering only the affected builders:

# Watch mode — recommended for development
dart run build_runner watch --delete-conflicting-outputs

# Build time comparison (medium-sized app, ~50 annotated files):
# First full build:     45–120 seconds
# Incremental rebuild:  1–5 seconds (only changed file + dependents)
# Watch mode idle:      0ms (no work until files change)

Full Build Cost and Mitigation

Full builds (clean + rebuild) are expensive because:

  1. build_runner resolves the full dependency graph
  2. Each Builder runs the Dart analyzer on its input files to read annotations
  3. The combining builder merges all generated parts

For large projects with many annotated files, full builds can take 2–5 minutes. Mitigation strategies:

  • Commit generated files: CI doesn't need to run build_runner if .g.dart files are committed and up-to-date. Verify with dart run build_runner build --delete-conflicting-outputs && git diff --exit-code
  • Restrict builder targets: In build.yaml, use generate_for.include to limit builders to specific directories instead of running on all Dart files.
  • Split into packages: Large monorepos can split features into separate packages; build_runner runs per-package, parallelizing across packages in CI.
  • Use build_runner cache: The .dart_tool/build directory is build_runner's cache. Cache it in CI keyed on relevant source file hashes to enable incremental CI builds.
# CI: cache build_runner outputs
- name: Cache build_runner
  uses: actions/cache@v3
  with:
    path: .dart_tool/build
    key: ${{ runner.os }}-build-runner-${{ hashFiles('lib/**/*.dart', 'pubspec.lock') }}

# Verify generated files are up-to-date (no uncommitted changes)
- name: Verify generated code
  run: |
    dart run build_runner build --delete-conflicting-outputs
    git diff --exit-code || (echo "Generated files are out of date. Run build_runner." && exit 1)

Generated File Management: Commit vs Gitignore

The Debate

ApproachProsCons
Commit generated files CI doesn't run build_runner (faster builds). Code review can see generated output. Works in environments without dart toolchain. Simpler onboarding (clone + run, no build step). PRs include generated file diffs (noisy). Generated files can go stale if developer forgets to regenerate. Merge conflicts in generated files.
Gitignore generated files Clean PRs — only source changes visible. No stale generated files — always regenerated. No merge conflicts in generated files. Every CI build must run build_runner (adds 1–5 min). Every developer must run build_runner after checkout. IDE may show errors until build_runner runs.
Staff Recommendation

For most teams: commit generated files and add a CI check that verifies they're up-to-date. This gives fast CI builds (no build_runner on every CI run) and catches developers who forget to regenerate. Use a pre-commit hook or CI lint step that runs dart run build_runner build and fails if there are uncommitted changes. The noisy diffs are a small cost compared to slow CI and onboarding friction.

Common Errors and Fixes

Conflicting Outputs

# Error: "Conflicting outputs were detected"
# Occurs when a generated file already exists and build_runner refuses to overwrite

# Fix: pass --delete-conflicting-outputs
dart run build_runner build --delete-conflicting-outputs

# Or add to build.yaml:
# targets:
#   $default:
#     sources:
#       - lib/**
#       - test/**

MissingPluginException in Generated Code

# Error: MissingPluginException(No implementation found for method X on channel Y)
# Causes: generated plugin registration code not included,
#         or flutter clean needed after plugin changes

flutter clean
flutter pub get
# Then rebuild
flutter run

Part File Not Found

# Error: Target of URI doesn't exist: 'package:myapp/models/product.g.dart'
# Cause: build_runner hasn't been run since the `part` directive was added

dart run build_runner build --delete-conflicting-outputs

Builder Version Conflicts

# Error: version solving failed — build_runner version conflict with generators
# Cause: json_serializable, freezed, riverpod_generator have mismatched build_runner requirements

# Fix: check each package's changelog for compatible versions
# Use pub upgrade --major-versions carefully
# Pin versions in pubspec.yaml if needed:
dev_dependencies:
  build_runner: ^2.4.6
  json_serializable: ^6.7.1
  freezed: ^2.4.5
  riverpod_generator: ^2.3.9
Tip: Speed Up IDE with Part Files

When you add a new part 'foo.g.dart' line and the file doesn't exist yet, the IDE shows red errors everywhere. Instead of running build_runner and waiting, you can create an empty placeholder file touch lib/models/foo.g.dart with just a // placeholder comment. The IDE stops complaining, and build_runner will overwrite it with the real content when you run it. This is especially useful when setting up a new model class.
