The Network Stack in Dart
dart:io HttpClient, the http package, and Dio — when to use each and what they cost

dart:io HttpClient — The Raw Layer

dart:io's HttpClient is the lowest-level HTTP API available on the Dart VM. All higher-level packages ultimately delegate to it (or to the platform's native network stack via dart:html on the web). Understanding it illuminates every abstraction above it.

import 'dart:io';
import 'dart:convert';

Future<Map<String, dynamic>> fetchRaw(String url) async {
  final client = HttpClient();
  client.connectionTimeout = const Duration(seconds: 10);

  // Returns an HttpClientRequest — headers can be modified here
  final request = await client.getUrl(Uri.parse(url));
  request.headers.set(HttpHeaders.acceptHeader, 'application/json');
  request.headers.set('X-Client-Version', '2.1.0');

  // Finalises headers and sends the request
  final response = await request.close();

  if (response.statusCode != 200) {
    throw HttpException('Status ${response.statusCode}', uri: Uri.parse(url));
  }

  // response is a Stream<List<int>>
  final body = await utf8.decodeStream(response);
  client.close(); // releases the socket pool
  return jsonDecode(body) as Map<String, dynamic>;
}
Pitfall
Always call client.close() (or client.close(force: true) on error paths). Each HttpClient instance maintains a connection pool; leaking instances leaks OS file descriptors and eventually causes SocketException: Failed host lookup failures.

HttpClient Connection Pool

The HttpClient connection pool keeps up to maxConnectionsPerHost (default 6) persistent HTTP/1.1 keep-alive connections per host. HTTP/2 multiplexes over a single connection; Dart's HttpClient supports HTTP/2 when the server advertises it via ALPN. Connection pooling means calling new HttpClient() per request is expensive — create a singleton.

The http Package

The http package wraps dart:io with a simpler API and a testable Client abstraction. Its MockClient makes unit testing trivial without network stubs.

import 'package:http/http.dart' as http;

// Use a shared client — never create http.get() with a one-shot client in production
final _client = http.Client();

Future<void> fetchWithHttp() async {
  final uri = Uri.https('api.example.com', '/v1/users', {'page': '1'});
  final response = await _client.get(uri, headers: {
    'Authorization': 'Bearer $token',
  });

  if (response.statusCode == 200) {
    final data = jsonDecode(response.body);
    // ...
  }
}

// Testing: inject MockClient
class UserRepository {
  final http.Client _client;
  UserRepository({http.Client? client}) : _client = client ?? http.Client();

  Future<User> getUser(int id) async {
    final response = await _client.get(Uri.parse('/users/$id'));
    return User.fromJson(jsonDecode(response.body));
  }
}

// In test:
final mockClient = MockClient((request) async =>
  http.Response('{"id":1,"name":"Alice"}', 200));
final repo = UserRepository(client: mockClient);

Dio — Production-Grade HTTP

Dio adds interceptors, cancellation tokens, FormData, upload/download progress callbacks, and a rich error hierarchy on top of dart:io. It is the de-facto standard for Flutter production apps.

BaseOptions and Configuration

import 'package:dio/dio.dart';

// Typically created once in a DI container (e.g., get_it / Riverpod)
final dio = Dio(
  BaseOptions(
    baseUrl: 'https://api.example.com/v1',
    connectTimeout: const Duration(seconds: 10),
    receiveTimeout: const Duration(seconds: 30),
    sendTimeout: const Duration(seconds: 30),
    headers: {
      'Accept': 'application/json',
      'X-App-Version': '2.1.0',
    },
    // ResponseType.json parses body automatically via jsonDecode
    responseType: ResponseType.json,
    // validateStatus lets you treat 4xx as non-exceptions if needed
    validateStatus: (status) => status != null && status < 500,
  ),
);
Part 2
Interceptors & the Request Pipeline
How Dio chains interceptors, implementing auth with token refresh, and retry logic

Dio Interceptor Chain

Dio's interceptor system is modelled after OkHttp's. Each Interceptor sits in a doubly-linked chain. The chain processes a request through all onRequest handlers sequentially, dispatches to the network, then routes the response back through all onResponse handlers (or onError handlers on failure).

Request ──►  Interceptor 1 (onRequest)
              │
              ▼
             Interceptor 2 (onRequest)  e.g. AuthInterceptor adds token
              │
              ▼
             Interceptor 3 (onRequest)  e.g. LoggingInterceptor
              │
              ▼
           [ Network ]  ◄─── actual HTTP round trip
              │
              ▼
             Interceptor 3 (onResponse)
              │
              ▼
             Interceptor 2 (onResponse)
              │
              ▼
             Interceptor 1 (onResponse)
              │
              ▼
          Response ──► caller

On any error:
              ▼
             Interceptor N (onError)  ← can resolve/retry/transform

Interceptors call handler.next() to forward, handler.resolve() to short-circuit with a synthetic response, or handler.reject() to inject an error. This makes them powerful for caching (short-circuit before the network) or mock injection in tests.

Auth Interceptor with Token Refresh

A production auth interceptor must handle the race condition where multiple in-flight requests all receive 401 simultaneously. A naive implementation spawns N refresh calls; the correct solution serializes the refresh with a Completer.

class AuthInterceptor extends Interceptor {
  final TokenStorage _tokenStorage;
  final Dio _dio; // the same Dio instance (or a separate one to avoid loops)

  // Serialize concurrent token refreshes
  bool _isRefreshing = false;
  final _refreshCompleters = <Completer<String>>[];

  AuthInterceptor(this._tokenStorage, this._dio);

  @override
  void onRequest(
    RequestOptions options,
    RequestInterceptorHandler handler,
  ) async {
    final token = await _tokenStorage.getAccessToken();
    if (token != null) {
      options.headers['Authorization'] = 'Bearer $token';
    }
    handler.next(options);
  }

  @override
  void onError(DioException err, ErrorInterceptorHandler handler) async {
    if (err.response?.statusCode != 401) {
      handler.next(err);
      return;
    }

    final options = err.requestOptions;

    // Prevent refresh loops on the refresh endpoint itself
    if (options.path.contains('/auth/refresh')) {
      handler.next(err);
      return;
    }

    if (_isRefreshing) {
      // Queue this request; it will be retried after refresh completes
      final completer = Completer<String>();
      _refreshCompleters.add(completer);
      try {
        final newToken = await completer.future;
        options.headers['Authorization'] = 'Bearer $newToken';
        final response = await _dio.fetch(options);
        handler.resolve(response);
      } catch (e) {
        handler.next(err);
      }
      return;
    }

    _isRefreshing = true;
    try {
      final newToken = await _refreshAccessToken();
      await _tokenStorage.saveAccessToken(newToken);

      // Resolve all queued requests
      for (final c in _refreshCompleters) {
        c.complete(newToken);
      }
      _refreshCompleters.clear();

      // Retry the original request
      options.headers['Authorization'] = 'Bearer $newToken';
      final response = await _dio.fetch(options);
      handler.resolve(response);
    } catch (e) {
      for (final c in _refreshCompleters) {
        c.completeError(e);
      }
      _refreshCompleters.clear();
      // Force logout
      await _tokenStorage.clearTokens();
      handler.next(err);
    } finally {
      _isRefreshing = false;
    }
  }

  Future<String> _refreshAccessToken() async {
    final refreshToken = await _tokenStorage.getRefreshToken();
    // Use a separate Dio instance to skip the auth interceptor
    final response = await Dio().post(
      'https://api.example.com/v1/auth/refresh',
      data: {'refresh_token': refreshToken},
    );
    return response.data['access_token'] as String;
  }
}
L6 Depth
The race condition above is the most common auth interceptor bug in production Flutter apps. When asked "how would you implement auth refresh in Dio?", describing the Completer-based serialization with the queue pattern signals deep practical experience.

Logging Interceptor

class NetworkLogInterceptor extends Interceptor {
  final Logger _logger;

  NetworkLogInterceptor(this._logger);

  @override
  void onRequest(RequestOptions options, RequestInterceptorHandler handler) {
    _logger.d('[REQ] ${options.method} ${options.uri}');
    if (options.data != null) {
      _logger.d('[REQ body] ${options.data}');
    }
    handler.next(options);
  }

  @override
  void onResponse(Response response, ResponseInterceptorHandler handler) {
    _logger.d('[RES] ${response.statusCode} ${response.requestOptions.uri}');
    handler.next(response);
  }

  @override
  void onError(DioException err, ErrorInterceptorHandler handler) {
    _logger.e('[ERR] ${err.type} ${err.requestOptions.uri}: ${err.message}');
    handler.next(err);
  }
}

Retry Interceptor with Exponential Backoff

class RetryInterceptor extends Interceptor {
  final Dio _dio;
  final int maxRetries;
  final Duration initialDelay;

  RetryInterceptor({
    required Dio dio,
    this.maxRetries = 3,
    this.initialDelay = const Duration(milliseconds: 500),
  }) : _dio = dio;

  @override
  void onError(DioException err, ErrorInterceptorHandler handler) async {
    final options = err.requestOptions;
    final retryCount = (options.extra['retryCount'] as int?) ?? 0;

    final shouldRetry = _isRetryable(err) && retryCount < maxRetries;

    if (!shouldRetry) {
      handler.next(err);
      return;
    }

    final delay = initialDelay * (1 << retryCount); // 500ms, 1s, 2s
    await Future.delayed(delay);

    options.extra['retryCount'] = retryCount + 1;
    try {
      final response = await _dio.fetch(options);
      handler.resolve(response);
    } on DioException catch (e) {
      handler.next(e);
    }
  }

  bool _isRetryable(DioException err) {
    return err.type == DioExceptionType.connectionTimeout ||
        err.type == DioExceptionType.receiveTimeout ||
        err.type == DioExceptionType.connectionError ||
        (err.response?.statusCode == 503) ||
        (err.response?.statusCode == 429); // rate-limited
  }
}

Certificate Pinning

Certificate pinning validates that the server's certificate or its public key matches a locally stored value, defeating MITM attacks even from CAs trusted by the OS. There are two approaches: certificate pinning (pin the leaf cert — breaks on renewal) and public key pinning (pin the SPKI — survives certificate rotation).

import 'dart:io';
import 'package:dio/dio.dart';
import 'package:flutter/services.dart'; // for rootBundle

Future<Dio> createPinnedDio() async {
  // Load certificate from Flutter assets
  final certBytes = await rootBundle.load('assets/certs/api_cert.pem');
  final cert = certBytes.buffer.asUint8List();

  final securityContext = SecurityContext(withTrustedRoots: false);
  // Only trust our specific certificate — reject all others
  securityContext.setTrustedCertificatesBytes(cert);

  final httpClient = HttpClient(context: securityContext);
  // Optionally validate hostname manually (defense in depth)
  httpClient.badCertificateCallback = (cert, host, port) {
    // Return true to ALLOW a bad certificate — only do this for debugging
    return false; // reject by default
  };

  final adapter = IOHttpClientAdapter()
    ..createHttpClient = () => httpClient;

  return Dio()..httpClientAdapter = adapter;
}

// For public-key pinning (preferred):
Dio createSpkiPinnedDio(Set<String> allowedSha256Pins) {
  final adapter = IOHttpClientAdapter()
    ..createHttpClient = () {
      final client = HttpClient();
      client.badCertificateCallback = (X509Certificate cert, String host, int port) {
        // X509Certificate does not expose SPKI directly in dart:io;
        // use pointycastle or a native plugin (ssl_pinning_plugin) for production.
        return false;
      };
      return client;
    };
  return Dio()..httpClientAdapter = adapter;
}
Production Warning
Leaf-certificate pinning breaks every time you rotate the TLS certificate. For production, pin the intermediate CA's public key or use a package like ssl_pinning_plugin that supports SPKI SHA-256 fingerprints. Always include a backup pin and an expiry date.

CancelToken — Request Cancellation

// Widget-level cancellation pattern:
class UserProfileScreen extends StatefulWidget { ... }

class _UserProfileScreenState extends State<UserProfileScreen> {
  final _cancelToken = CancelToken();
  late final Future<User> _userFuture;

  @override
  void initState() {
    super.initState();
    _userFuture = _fetchUser();
  }

  Future<User> _fetchUser() async {
    try {
      final response = await dio.get(
        '/users/${widget.userId}',
        cancelToken: _cancelToken,
      );
      return User.fromJson(response.data);
    } on DioException catch (e) {
      if (CancelToken.isCancel(e)) {
        // Normal — widget was disposed before response arrived
        rethrow;
      }
      // Handle actual network error
      rethrow;
    }
  }

  @override
  void dispose() {
    // Cancel any in-flight request when the widget leaves the tree
    _cancelToken.cancel('Widget disposed');
    super.dispose();
  }
}

File Upload with Progress

Future<String> uploadProfileImage(File imageFile) async {
  final formData = FormData.fromMap({
    'image': await MultipartFile.fromFile(
      imageFile.path,
      filename: 'profile_${DateTime.now().millisecondsSinceEpoch}.jpg',
      contentType: DioMediaType('image', 'jpeg'),
    ),
    'userId': '123',
  });

  final response = await dio.post(
    '/users/avatar',
    data: formData,
    onSendProgress: (sent, total) {
      final progress = sent / total;
      // Update UI: progressNotifier.value = progress;
    },
    options: Options(
      headers: {'Content-Type': 'multipart/form-data'},
      receiveTimeout: const Duration(seconds: 60), // larger for uploads
    ),
  );

  return response.data['imageUrl'] as String;
}

DioException Hierarchy

Dio maps all network failures to DioException with a DioExceptionType discriminant. Handling these granularly in a production app provides meaningful UX feedback instead of generic "network error" messages.

DioExceptionTypeCauseUser Message
connectionTimeoutTCP handshake exceeded connectTimeout"Server is slow to respond"
sendTimeoutUpload stalled beyond sendTimeout"Upload failed — try again"
receiveTimeoutServer started but stalled mid-response"Download timed out"
badCertificateTLS validation failure (including pinning)"Secure connection failed"
connectionErrorNo network, DNS failure, refused connection"No internet connection"
badResponsevalidateStatus rejected the status codeDerive from statusCode
cancelCancelToken was triggeredSilent (not a user error)
unknownOther (e.g., redirect loop)"Unexpected error"
 
// Centralised error handling:
AppError mapDioError(DioException e) {
  return switch (e.type) {
    DioExceptionType.connectionError =>
        const AppError.noNetwork(),
    DioExceptionType.connectionTimeout ||
    DioExceptionType.receiveTimeout ||
    DioExceptionType.sendTimeout =>
        const AppError.timeout(),
    DioExceptionType.badResponse =>
        _mapStatusCode(e.response!.statusCode!),
    DioExceptionType.cancel =>
        const AppError.cancelled(),
    DioExceptionType.badCertificate =>
        const AppError.securityError(),
    _ =>
        AppError.unknown(e.message ?? 'Unknown network error'),
  };
}

AppError _mapStatusCode(int code) => switch (code) {
  400 => const AppError.badRequest(),
  401 => const AppError.unauthorized(),
  403 => const AppError.forbidden(),
  404 => const AppError.notFound(),
  409 => const AppError.conflict(),
  422 => const AppError.validationError(),
  429 => const AppError.rateLimited(),
  503 => const AppError.serviceUnavailable(),
  _ when code >= 500 => const AppError.serverError(),
  _ => AppError.unknown('HTTP $code'),
};

If asked "why use Dio over the http package?", frame it as: "For a production app I need auth token refresh with queued retry, granular timeouts, upload progress, and request cancellation tied to widget lifecycle. Offline Queue Pattern

When a mutation request fails due to no connectivity, it should be stored and replayed when connectivity is restored — without the user having to retry manually.

@immutable
class PendingRequest {
  final String id;
  final String method;
  final String path;
  final Map<String, dynamic>? body;
  final DateTime createdAt;
  final int retryCount;

  const PendingRequest({ ... });
}

class OfflineQueueInterceptor extends Interceptor {
  final PendingRequestStore _store;
  final ConnectivityService _connectivity;

  @override
  void onRequest(RequestOptions options, RequestInterceptorHandler handler) async {
    // Only queue mutation requests
    if (!['POST', 'PUT', 'PATCH', 'DELETE'].contains(options.method)) {
      handler.next(options);
      return;
    }

    final hasConnectivity = await _connectivity.hasConnection;
    if (hasConnectivity) {
      handler.next(options);
      return;
    }

    // Offline — persist to queue
    await _store.enqueue(PendingRequest(
      id: uuid.v4(),
      method: options.method,
      path: options.path,
      body: options.data,
      createdAt: DateTime.now(),
      retryCount: 0,
    ));

    // Short-circuit with synthetic 202 to keep UI happy
    handler.resolve(Response(
      requestOptions: options,
      statusCode: 202,
      data: {'queued': true},
    ));
  }
}

// Drain queue on connectivity restore:
class QueueDrainService {
  final Dio _dio;
  final PendingRequestStore _store;

  Future<void> drainQueue() async {
    final pending = await _store.getAll();
    for (final req in pending) {
      try {
        await _dio.request(
          req.path,
          data: req.body,
          options: Options(method: req.method),
        );
        await _store.remove(req.id);
      } catch (e) {
        if (req.retryCount >= 5) {
          // Give up — notify user or move to dead-letter queue
          await _store.remove(req.id);
        } else {
          await _store.incrementRetry(req.id);
        }
      }
    }
  }
}
