Core Networking & Data Concepts

Foundational questions on HTTP clients, serialization, local storage, and data modeling that every Flutter engineer should answer confidently.

Standard — L4
How does Dio's interceptor chain work?
Dio processes interceptors in a chain — each interceptor can observe, modify, or short-circuit the request or response. The chain runs in three phases:
  1. Request interceptors — run in order before the request is sent. Each can modify headers, add auth tokens, log the outgoing request, or cancel entirely with handler.reject().
  2. Response interceptors — run in order after a successful response is received. Can transform the response body, cache it, or convert it into a domain object.
  3. Error interceptors — run when any step throws a DioException. Can recover (e.g., refresh the token and retry) or rethrow.
Each interceptor calls handler.next() to pass control to the next interceptor, handler.resolve(response) to short-circuit with a successful response, or handler.reject(error) to short-circuit with an error. Interceptors are added via dio.interceptors.add() and run in the order they are added.
Be ready to sketch a token-refresh interceptor — it's the canonical example of error interceptor usage and appears in almost every Dio interview question.

What is the difference between Hive and Drift? When would you choose each?
DimensionHiveDrift (formerly Moor)
Storage modelKey-value / document (boxes)Relational (SQLite)
SchemaSchema-less (or typed adapters)Strongly typed, schema migrations
QueriesNo SQL — iterate or filter in DartFull SQL + type-safe query builder
Reactive queriesVia listenable boxes, coarsewatch() — table-level reactive streams
RelationsManual (store IDs, resolve in Dart)Native foreign keys, joins
MigrationVersion-based adapter changes (fragile)First-class migration API with MigrationStrategy
Setup complexityLow — no codegen neededMedium — requires build_runner
Choose Hive for: simple key-value caching, user settings, small unstructured data, or when you need zero codegen setup. Choose Drift for: relational data with joins and foreign keys, complex queries, offline-first sync (reactive watch() is essential), or any data layer that needs migrations.
The reactive watch() capability is the key Drift advantage for offline-first. Mention it explicitly.
Standard — L4
How do you implement a retry on 401 with token refresh in Dio?
Use an error interceptor that intercepts DioException with a 401 status. The interceptor:
  1. Checks whether the request has already been retried (to prevent infinite loops — add a flag to the request options).
  2. Calls the token refresh endpoint to get a new access token.
  3. Updates the stored token and the request's Authorization header.
  4. Retries the original request via dio.fetch(requestOptions).
  5. On refresh failure, logs out the user and calls handler.reject().
class TokenRefreshInterceptor extends Interceptor {
  final AuthService _authService;
  final Dio _dio;

  @override
  Future<void> onError(
    DioException err,
    ErrorInterceptorHandler handler,
  ) async {
    if (err.response?.statusCode == 401 &&
        err.requestOptions.extra['retried'] != true) {
      try {
        final newToken = await _authService.refreshToken();
        err.requestOptions.headers['Authorization'] = 'Bearer $newToken';
        err.requestOptions.extra['retried'] = true;
        final response = await _dio.fetch(err.requestOptions);
        return handler.resolve(response);
      } catch (_) {
        await _authService.logout();
      }
    }
    handler.next(err);
  }
}
The extra['retried'] guard is the detail that prevents infinite retry loops — mention it explicitly.
Standard — L4
What does json_serializable generate and how does it work with build_runner?
json_serializable is a code generation package. You annotate a Dart class with @JsonSerializable() and add factory and toJson stubs. When you run dart run build_runner build, build_runner scans for @JsonSerializable annotations and generates a .g.dart file alongside your source file. The generated file contains:
  • _$MyClassFromJson(Map<String, dynamic> json) — constructs the object from a JSON map, handling type coercion and null safety.
  • _$MyClassToJson(MyClass instance) — converts the object back to a JSON map.
build_runner watch watches for file changes and regenerates incrementally. The generated .g.dart files should be committed to source control (they are not build artifacts — your app cannot compile without them). Common options: fieldRename for snake_case keys, explicitToJson: true for nested objects, converters for custom types like DateTime.
Mentioning that .g.dart files should be committed (unlike Android's generated code) is a subtle correctness point interviewers notice.
Standard — L5
What is flutter_secure_storage backed by on Android and iOS?
  • iOS: Backed by the iOS Keychain. Data is encrypted by the OS using the device's hardware security module. Keychain items can be scoped to accessibility options (e.g., only accessible when device is unlocked, or accessible after first unlock for background tasks).
  • Android: Backed by EncryptedSharedPreferences (using the Jetpack Security library / androidx.security.crypto). The encryption key is stored in the Android Keystore (hardware-backed on devices with a secure element). The actual data is stored in an encrypted SharedPreferences file.
Important nuances: on Android, data is not automatically deleted when the app is uninstalled (unless you configure resetOnError: true or handle it explicitly). On iOS, Keychain data persists across reinstalls by default — this can be a privacy/security concern if you store sensitive auth tokens. You may need to clear on first launch after install.
The "persists across iOS reinstalls" behavior is a real production bug many teams discover late. Mentioning it demonstrates real-world experience.

Deep — L6
Design an offline-first architecture for a social media app. What are the key components?
Key components and their responsibilities:
  1. Local Database (Drift/SQLite): Source of truth. Tables for posts, comments, likes, users, pending operations. Reactive queries power the UI stream.
  2. Remote Data Source: Thin wrapper over Dio. Only responsible for HTTP calls — no business logic, no caching. Returns domain models or throws typed exceptions.
  3. Repository: Combines local and remote. Exposes watch*() streams (local DB) and sync*() methods. Handles cache-then-network, optimistic updates, and queuing.
  4. Sync Queue: Durable queue of pending mutations stored in Drift. Operations include type, JSON payload, idempotency key, retry count, and next retry timestamp.
  5. Queue Processor: Drains the queue when online. Executes ops in FIFO order. Exponential backoff on transient errors. Discards on 409 conflict (logs for analytics). Nulls out on 4xx non-retryable errors.
  6. Connectivity Service: Two-layer check: connectivity_plus for network type + periodic HTTP ping for reachability. Emits a stream that the queue processor listens to.
  7. Conflict Resolution Policy: Last-write-wins using server-assigned updated_at timestamps. For likes/follows (counters), use server-side idempotent operations.
  8. Delta Sync: Store a last_synced_at cursor per entity type. On reconnect, fetch only records changed since the cursor. Server uses soft deletes.
Data flow: UI observes repository stream (local DB) → background sync writes to DB → stream re-emits → UI updates. User mutations → local DB write → queue enqueue → UI updates immediately (optimistic) → queue processor drains on connectivity.
Draw this as a layered diagram if on a whiteboard: UI → Repository → [Local DB | Remote DS] → Sync Queue → Queue Processor. The interviewer wants to see clean layer separation and reactive data flow.
Deep — L6
How does Drift's reactive query (watch()) work internally? What triggers a re-emission?
Drift's watch() returns a Stream backed by a stream query store. Internally:
  1. When you call select(table).watch(), Drift registers the stream along with the set of table names it depends on.
  2. Every write operation that goes through Drift (insert, update, delete, upsert) — whether via the type-safe API or raw SQL — notifies the stream query store of which tables were modified.
  3. The store checks which registered streams depend on the modified tables and triggers a re-query for each affected stream.
  4. The re-query runs the original SQL query against the current DB state and emits the new result list.
This is implemented using SQLite's update hook (a C-level callback that fires on every write). Drift installs this hook and uses it to track table-level changes. Re-emissions are debounced slightly to coalesce multiple rapid writes into a single emission.

Important implication: if you write to the DB via raw SQL outside of Drift's managed API (e.g., via customStatement), you must call notifyUpdates({table}) manually, or the reactive streams will not know about the change.
Mentioning the SQLite update hook is the internal detail that shows L6-level library knowledge. Most Flutter engineers only know "watch() re-emits on change" — not why.
Deep — L6
How would you handle conflict resolution when two users edit the same record on different devices simultaneously?
The right strategy depends on the data type and business requirements:

For simple fields (user bio, profile photo, display name): Last-write-wins with server-assigned timestamps. The server assigns an updated_at on every write. When a stale write arrives (client's based_on_version is older than the current server version), the server returns 409 with the current version. The client can show a conflict UI or silently accept the server version.

For counters (like counts, view counts): Use idempotent server-side operations. "User A liked post P" is an idempotent fact, not a counter increment. The server maintains the set of users who liked a post; the count is derived. This is conflict-free.

For collaborative text (post body, shared document): Operational Transformation (OT) or CRDTs. Most Flutter apps don't need this. If you do, use a backend that supports it (e.g., Firebase Realtime Database with merge semantics, or a purpose-built CRDT library).

Implementation with optimistic locking: 
// Client sends the version it based its edit on
PATCH /posts/{id}
{
  "content": "new content",
  "based_on_version": "2024-01-15T10:30:00Z"
}

// Server response on conflict:
// 409 Conflict
// { "current_version": "2024-01-15T11:00:00Z", "current_content": "..." }
The client receives the 409, shows the user both versions, and lets them resolve manually or auto-merges based on rules.
Showing awareness of multiple strategies (LWW, idempotent ops, OT/CRDT) and knowing when to apply each is the L6 answer. An L4 answer says "use timestamps" without understanding the failure modes.
Deep — L6
Explain the optimistic update pattern and how you'd implement rollback.
An optimistic update applies the expected result of a mutation to the local state immediately, without waiting for server confirmation. The assumption is that the server will accept the change — hence "optimistic." If the server rejects it, the local state is rolled back.

Implementation pattern:
  1. Capture the current local state (for rollback).
  2. Apply the expected new state to the local DB.
  3. Emit the new state to the UI (instant update).
  4. Fire the network request.
  5. On success: nothing to do — local state is already correct.
  6. On failure: restore the captured state, emit it, show an error to the user.
Future<void> toggleLike(Post post) async {
  // Capture current state for rollback
  final previousLiked = post.isLikedByMe;
  final previousCount = post.likeCount;

  // Optimistic update
  await _local.updatePost(post.copyWith(
    isLikedByMe: !previousLiked,
    likeCount: previousLiked ? previousCount - 1 : previousCount + 1,
  ));

  try {
    if (previousLiked) {
      await _remote.unlikePost(post.id);
    } else {
      await _remote.likePost(post.id);
    }
  } on DioException catch (e) {
    // Rollback
    await _local.updatePost(post.copyWith(
      isLikedByMe: previousLiked,
      likeCount: previousCount,
    ));
    rethrow;
  }
}
Rollback edge cases: If the user makes additional changes between the optimistic write and the rollback, a naive rollback restores a stale state. Prefer rollback operations that are relative ("undo the toggle") rather than absolute ("set liked to false") when possible.
The rollback edge case (user makes additional changes before rollback completes) is the detail that separates L5 from L6 answers.
Deep — L6
How would you implement delta sync with a cursor-based API?
Delta sync fetches only records that have changed since the last successful sync, using a cursor (typically a timestamp or sequence number) to mark the sync boundary.

Server requirements: The server must support a since query parameter and must use soft deletes (so deletions appear in the delta). The response must include: upserted records, deleted record IDs, and the server's current timestamp (to use as the next cursor — using server time avoids client clock skew).

Client implementation: 
class DeltaSyncService {
  static const _cursorKey = 'posts_last_synced_at';

  Future<void> sync() async {
    final cursor = await _prefs.getString(_cursorKey);

    final delta = await _remote.fetchPostDelta(since: cursor);

    await _db.transaction(() async {
      for (final post in delta.upserted) {
        await _db.postDao.upsert(post);
      }
      for (final id in delta.deletedIds) {
        await _db.postDao.hardDelete(id);
      }
    });

    // Store the server's reported sync time as the next cursor
    await _prefs.setString(_cursorKey, delta.syncedAt.toIso8601String());
  }
}
Edge cases:
  • First sync (null cursor): Fetch all records. This is the only full sync.
  • Cursor after a long offline period: Server may have pruned its change log. Detect this (server returns a 410 Gone or a specific error) and fall back to a full sync.
  • Transaction atomicity: Wrap the upsert + delete + cursor update in a single DB transaction. If the app crashes mid-sync, the cursor is not advanced, so the next sync re-fetches the same delta. This is safe because upserts are idempotent.
The transaction atomicity point (cursor only advances after successful write) is the production correctness detail that distinguishes a well-thought-out answer.
Deep — L6
Freezed sealed classes — how would you use them to model API response states (loading/success/error)?
Freezed generates sealed (exhaustive) union types in Dart — similar to Kotlin's sealed classes or Rust's enums. For API response states: 
@freezed
sealed class PostsState with _$PostsState {
  const factory PostsState.initial() = PostsInitial;
  const factory PostsState.loading() = PostsLoading;
  const factory PostsState.success({required List<Post> posts}) = PostsSuccess;
  const factory PostsState.error({required String message, Object? exception}) = PostsError;
}
In your BLoC/ViewModel: 
Future<void> loadPosts() async {
  emit(const PostsState.loading());
  try {
    final posts = await _repository.getPosts();
    emit(PostsState.success(posts: posts));
  } catch (e) {
    emit(PostsState.error(message: 'Failed to load posts', exception: e));
  }
}
In the UI, use switch (Dart 3 pattern matching) for exhaustive handling: 
return switch (state) {
  PostsInitial() => const SizedBox.shrink(),
  PostsLoading() => const CircularProgressIndicator(),
  PostsSuccess(:final posts) => PostsList(posts: posts),
  PostsError(:final message) => ErrorView(message: message),
};
The compiler enforces that all states are handled — if you add a new state variant, every switch expression fails to compile until you handle it. This is the key advantage over using enums with nullable fields or exception-based state.
Showing Dart 3 pattern matching (switch expressions with destructuring) alongside Freezed is a strong signal of current Dart knowledge.
Deep — L6
How does certificate pinning work in Flutter and what are the risks?
Certificate pinning rejects TLS connections to servers whose certificate (or public key) does not match a hardcoded expected value. It prevents MITM attacks even from a trusted CA.

Implementation in Flutter with Dio: 
final dio = Dio();
(dio.httpClientAdapter as IOHttpClientAdapter).createHttpClient = () {
  final client = HttpClient();
  client.badCertificateCallback = (cert, host, port) => false;

  // For full pinning — check the certificate's SHA-256 fingerprint
  client.connectionFactory = (uri, proxyHost, proxyPort) async {
    final socket = await SecureSocket.connect(uri.host, uri.port,
      onBadCertificate: (cert) {
        final pin = sha256.convert(cert.der).toString();
        return pin == 'expected-pin-hash';
      });
    return socket;
  };
  return client;
};
In practice, most teams use a package like http_certificate_pinning or configure pinning at the native layer (Network Security Config on Android, App Transport Security on iOS + custom validation).

Risks:
  • Certificate rotation. If the server rotates its certificate (e.g., annual renewal) without a coordinated app update, all users on old app versions break. Mitigation: pin to the intermediate CA or public key (which rotates less often) rather than the leaf certificate. Or pin multiple certificates (primary + backup).
  • App update lag. Users on old versions with a broken pin cannot use the app until they update. This can strand a significant user population.
  • Debugging difficulty. Charles Proxy, Proxyman, and other debugging proxies break pinned apps. Your dev/staging builds should have pinning disabled or use a dev certificate pin.
The certificate rotation risk is the most important practical concern. Mentioning "pin to the intermediate CA" as mitigation shows production experience.

How would you paginate a large API in an offline-first architecture?
Pagination and offline-first combine awkwardly because pagination is inherently network-centric (pages are defined by server state), while offline-first wants the local DB to be the source of truth.

Strategy — paginate at the network layer, store all fetched data locally:
  1. The local DB stores all previously fetched records (not just one page).
  2. The UI displays a Drift watch() stream with an OFFSET/LIMIT query that grows as the user scrolls.
  3. When the user reaches the end of what's locally available, fire a network request for the next page.
  4. Append the fetched page to the local DB. The Drift stream re-emits with the expanded list.
  5. Store the next-page cursor (offset or token) in local state.
// Repository
Future<void> loadNextPage() async {
  final cursor = await _local.getPostsCursor();
  if (cursor?.hasMore == false) return; // No more pages

  final page = await _remote.fetchPosts(after: cursor?.nextToken);
  await _local.insertAll(page.posts);
  await _local.updateCursor(PostsCursor(
    nextToken: page.nextPageToken,
    hasMore: page.hasMore,
  ));
}

// UI observes the whole local DB (all fetched posts)
Stream<List<Post>> watchPosts() => _local.watchAllPosts();
When offline: the user can still scroll through all previously fetched posts. They simply cannot load new pages until connectivity returns. This is the correct offline-first behavior — show what you have, gracefully indicate more is available when online.

Cache size management: for very large datasets, consider evicting old pages (keep the N most recent) and re-fetching them on demand when the user scrolls back.
The key insight is "paginate at the network layer, store all fetched data locally." Many candidates try to paginate the local DB, which misses the point entirely.
