04 — Offline-First Architecture

Repository pattern · Cache-then-network · Optimistic updates · Sync queue · Conflict resolution

4.1 The Offline-First Principle

Offline-first is an architectural stance: the local database is the source of truth. The network is a sync mechanism — a way to share state with a server and other clients — not a dependency required for the app to function. This is the opposite of the naive "load from network, show result" pattern.

The core mental shift: instead of thinking "the server has the data, I fetch it," think "the local DB has the data, I sync it." Every read comes from the local DB. Every write goes to the local DB first, then syncs to the server in the background.

Benefits

  • Instant UI. The UI renders from the local DB immediately on app open — no loading spinner for data the user has already seen.
  • Works without connectivity. The user can browse, create, and edit data with no network. Changes queue up and sync when connectivity returns.
  • Resilient to flaky networks. A slow or dropped connection does not degrade the user experience — local data is always available.
  • Reduced server load. Background syncs can be batched and throttled. You don't refetch data on every screen open.

When offline-first is the right choice

Not every app needs full offline-first. It adds complexity — particularly in conflict resolution and sync queue management. It's the right choice when: the app is used in variable-connectivity environments (field workers, travel, emerging markets), the user creates data the app must not lose, or instant UI response is a core UX requirement. A simple content-consumption app (news reader, stock quotes) can get away with cache-then-network without a full sync queue.

Architecture Principle
The UI should never know whether data came from cache or network. The repository abstracts this completely. The UI layer observes a stream from the local DB and does not care how the DB got populated.

4.2 Connectivity Monitoring

Knowing whether the device is online is more nuanced than it sounds. The connectivity_plus package reports the network type (WiFi, mobile, none) — but a device can be connected to a WiFi network with no actual internet access (captive portals, no gateway route). You need to distinguish "network interface present" from "internet reachable."

Two-layer connectivity check

  1. Layer 1 — network type: connectivity_plus stream reports changes immediately (WiFi connected, WiFi disconnected, mobile connected).
  2. Layer 2 — reachability ping: When layer 1 reports a network is present, fire an HTTP HEAD request to a reliable endpoint (e.g., https://connectivitycheck.gstatic.com/generate_204 or your own server). A 204 response confirms actual internet access.
import 'package:connectivity_plus/connectivity_plus.dart';

class ConnectivityService {
  final _connectivity = Connectivity();
  final _controller = StreamController<bool>.broadcast();
  Timer? _pingTimer;

  Stream<bool> get isOnlineStream => _controller.stream;
  bool _isOnline = false;
  bool get isOnline => _isOnline;

  ConnectivityService() {
    _connectivity.onConnectivityChanged.listen(_onConnectivityChanged);
    // Periodic ping every 30s to detect silent connectivity loss
    _pingTimer = Timer.periodic(
      const Duration(seconds: 30),
      (_) => _checkReachability(),
    );
    _checkReachability(); // Initial check
  }

  void _onConnectivityChanged(List<ConnectivityResult> results) {
    if (results.contains(ConnectivityResult.none)) {
      _updateStatus(false);
    } else {
      // Network type changed — verify reachability
      _checkReachability();
    }
  }

  Future<void> _checkReachability() async {
    try {
      final response = await http
          .head(Uri.parse('https://connectivitycheck.gstatic.com/generate_204'))
          .timeout(const Duration(seconds: 5));
      _updateStatus(response.statusCode == 204);
    } catch (_) {
      _updateStatus(false);
    }
  }

  void _updateStatus(bool online) {
    if (_isOnline != online) {
      _isOnline = online;
      _controller.add(online);
    }
  }

  void dispose() {
    _pingTimer?.cancel();
    _controller.close();
  }
}

Inject ConnectivityService as a singleton. The sync queue processor listens to isOnlineStream to know when to attempt pending operation drain.

4.3 Repository Pattern

The repository is the single point of contact between the domain/presentation layer and all data sources. It hides whether data comes from the local DB or the network. The UI never touches a data source directly.

// Domain layer — pure Dart, no Flutter imports
abstract interface class PostRepository {
  /// Returns a stream that emits on every local DB change.
  Stream<List<Post>> watchPosts();

  /// Fetches latest from remote and writes to local DB.
  /// The watchPosts() stream will emit automatically after this completes.
  Future<void> syncFromRemote();

  /// Creates a post. Writes locally immediately; syncs in background.
  Future<void> createPost(CreatePostRequest req);

  /// Deletes a post. Writes locally immediately; syncs in background.
  Future<void> deletePost(String postId);
}

// Data layer
class PostRepositoryImpl implements PostRepository {
  PostRepositoryImpl(this._local, this._remote, this._queue);

  final LocalPostDataSource _local;    // Drift DAO
  final RemotePostDataSource _remote;  // Dio API client
  final SyncQueue _queue;              // Pending operations queue

  @override
  Stream<List<Post>> watchPosts() {
    // Start a background sync and return the live DB stream.
    // The stream will automatically emit when syncFromRemote writes to the DB.
    syncFromRemote().ignore();
    return _local.watchPosts();
  }

  @override
  Future<void> syncFromRemote() async {
    final remotePosts = await _remote.fetchPosts();
    await _local.upsertAll(remotePosts);
  }

  @override
  Future<void> createPost(CreatePostRequest req) async {
    // 1. Write to local DB immediately (user sees result instantly)
    final localId = await _local.insertPost(req);
    // 2. Enqueue for sync
    await _queue.enqueue(SyncOperation(
      type: SyncOpType.createPost,
      payload: req.toJson(),
      localId: localId,
      idempotencyKey: const Uuid().v4(),
    ));
    // 3. Attempt immediate sync if online
    _queue.drain();
  }

  @override
  Future<void> deletePost(String postId) async {
    await _local.markDeleted(postId);
    await _queue.enqueue(SyncOperation(
      type: SyncOpType.deletePost,
      payload: {'postId': postId},
      idempotencyKey: 'delete-$postId',
    ));
    _queue.drain();
  }
}
Design Note
The repository interface lives in the domain layer (no framework dependencies). PostRepositoryImpl lives in the data layer. This means you can swap the implementation (e.g., swap Drift for Isar, or swap Dio for another HTTP client) without touching the domain or presentation layers.

4.4 Cache-Then-Network Pattern

The cache-then-network pattern delivers two emissions for every request: first the cached data (instant), then the fresh remote data (after network). With Drift's reactive queries, you don't need to manually emit twice — you emit the DB stream, and it automatically re-emits whenever the DB changes (including when the background sync writes to it).

@override
Stream<List<Post>> watchPosts() async* {
  // Step 1: Yield the Drift watch stream.
  // This emits immediately with cached data, then re-emits on every DB change.
  yield* _local.watchPosts();

  // Step 2: Trigger a background sync.
  // When sync completes and writes to the DB, the stream above re-emits
  // automatically — no additional yield needed.
  syncFromRemote().ignore();
}

This works because Drift's watch() returns a Stream that stays open and emits a new list every time the underlying SQLite table changes. The yield* forwards all of those emissions. The background syncFromRemote() is a side effect that writes to the DB, which triggers a new emission through the already-open stream.

Drift reactive query setup

// In your Drift DAO
Stream<List<PostEntity>> watchPosts() {
  return (select(postsTable)
        ..orderBy([(t) => OrderingTerm.desc(t.createdAt)]))
      .watch(); // Returns a Stream<List<PostEntity>> that emits on every change
}

// Data source wraps the DAO and maps to domain models
class LocalPostDataSource {
  LocalPostDataSource(this._db);
  final AppDatabase _db;

  Stream<List<Post>> watchPosts() {
    return _db.postDao.watchPosts().map(
      (entities) => entities.map(Post.fromEntity).toList(),
    );
  }
}
Why Drift for offline-first
Drift (formerly Moor) is the best-suited Flutter local DB for offline-first because its reactive queries (watch()) are powered by SQLite triggers — they emit precisely when the relevant tables change. Hive and SharedPreferences do not have this reactive capability out of the box.

4.5 Optimistic Updates

An optimistic update assumes the server will accept the change and applies it locally before hearing back from the server. If the server rejects it, you roll back. This makes the UI feel instant — the user sees their action take effect immediately, with no waiting spinner.

Future<void> likePost(String postId) async {
  // Optimistic: apply locally immediately
  await _local.setLiked(postId, true);

  try {
    await _remote.likePost(postId);
    // Success: local state is already correct, nothing to do
  } on DioException catch (e) {
    // Failure: roll back the local change
    await _local.setLiked(postId, false);
    // Rethrow so the UI can show an error message
    rethrow;
  }
}

Rollback considerations

  • What if the rollback also fails? Rare, but possible (DB error). Log it and schedule a consistency check on next app open.
  • What if the user navigates away before rollback completes? The rollback still runs — it is not tied to the UI. The DB write will complete and the reactive stream will emit the rolled-back state to whoever is listening (if anyone).
  • Idempotent rollbacks. Design rollbacks as idempotent operations. "Set liked to false" is idempotent; "decrement like count" is not.
  • User feedback. Show a snackbar after rollback: "Could not like post. Check your connection." Don't leave the user confused about why the UI reverted.

Optimistic update with loading state

// Using a sealed class for per-item state
sealed class LikeState {}
class LikeIdle extends LikeState {}
class LikePending extends LikeState {}  // Waiting for server confirmation
class LikeError extends LikeState { final String message; LikeError(this.message); }

// In your ViewModel/BLoC
Future<void> onLikeTapped(String postId) async {
  emit(state.copyWith(likeState: LikePending()));
  try {
    await _repository.likePost(postId);
    emit(state.copyWith(likeState: LikeIdle()));
  } catch (e) {
    emit(state.copyWith(likeState: LikeError('Failed to like post')));
  }
}

4.6 Offline Mutation Queue

When the device is offline and the user creates, edits, or deletes data, those changes must be persisted locally and synced when connectivity returns. The sync queue is a durable, ordered list of pending operations stored in the local DB (so it survives app restarts).

Queue schema (Drift)

@DataClassName('PendingOp')
class PendingOps extends Table {
  IntColumn get id => integer().autoIncrement()();
  // Operation type: 'create_post', 'delete_post', 'like_post', etc.
  TextColumn get type => text()();
  // JSON-serialized operation payload
  TextColumn get payload => text()();
  // Idempotency key — prevents duplicate execution if the op is retried
  // after a partial success (server received but response was lost)
  TextColumn get idempotencyKey => text().unique()();
  // Retry tracking
  IntColumn get retryCount => integer().withDefault(const Constant(0))();
  DateTimeColumn get nextRetryAt => dateTime().nullable()();
  DateTimeColumn get createdAt => dateTime()();
}

Queue processor

class SyncQueueProcessor {
  SyncQueueProcessor(this._db, this._remote, this._connectivity);

  final AppDatabase _db;
  final RemoteDataSource _remote;
  final ConnectivityService _connectivity;

  bool _draining = false;

  void start() {
    // Drain on connectivity restore
    _connectivity.isOnlineStream
        .where((online) => online)
        .listen((_) => drain());
  }

  Future<void> drain() async {
    if (_draining || !_connectivity.isOnline) return;
    _draining = true;

    try {
      // Process ops in insertion order (FIFO)
      final ops = await _db.pendingOpDao.getPendingOps();
      for (final op in ops) {
        await _processOp(op);
      }
    } finally {
      _draining = false;
    }
  }

  Future<void> _processOp(PendingOp op) async {
    try {
      await _executeOp(op);
      // Success: remove from queue
      await _db.pendingOpDao.delete(op.id);
    } on DioException catch (e) {
      if (e.response?.statusCode == 409) {
        // Conflict — server rejected this op. Log and discard.
        await _db.pendingOpDao.delete(op.id);
        _logConflict(op, e);
      } else {
        // Transient error — schedule exponential backoff retry
        final delay = _backoffDelay(op.retryCount);
        await _db.pendingOpDao.updateRetry(
          op.id,
          retryCount: op.retryCount + 1,
          nextRetryAt: DateTime.now().add(delay),
        );
      }
    }
  }

  Duration _backoffDelay(int retryCount) {
    // Exponential backoff: 1s, 2s, 4s, 8s … max 5 minutes
    final seconds = min(pow(2, retryCount).toInt(), 300);
    // Add jitter to prevent thundering herd
    final jitter = Random().nextInt(1000);
    return Duration(seconds: seconds, milliseconds: jitter);
  }

  Future<void> _executeOp(PendingOp op) async {
    final payload = jsonDecode(op.payload) as Map<String, dynamic>;
    switch (op.type) {
      case 'create_post':
        await _remote.createPost(
          CreatePostRequest.fromJson(payload),
          idempotencyKey: op.idempotencyKey,
        );
      case 'delete_post':
        await _remote.deletePost(payload['postId'] as String);
      case 'like_post':
        await _remote.likePost(payload['postId'] as String);
      default:
        throw UnimplementedError('Unknown op type: ${op.type}');
    }
  }
}
Critical Detail
The idempotency key is non-negotiable. Without it, a retry after a partial success (server processed the request but the client never received the response) will create duplicate data. Send the idempotency key as an HTTP header (Idempotency-Key: <uuid>) and ensure your server handles it correctly.

4.7 Conflict Resolution Strategies

A conflict occurs when two clients (or a client and the server) modify the same record independently. The right resolution strategy depends on the domain.

Strategy How it works Best for Worst for
Last-Write-Wins (LWW) The write with the latest timestamp wins. Earlier writes are discarded. Settings, preferences, profile data. Cases where the user's latest intent is always correct. Collaborative editing where concurrent writes should both be preserved.
Server-Wins Client changes are rejected if the server version is newer. Client must refetch and reapply. Financial data, inventory, anything where server authority is non-negotiable. Apps with a strong offline-first UX requirement — users lose work.
Client-Wins Client change always accepted, overwrites server state. Rarely correct. Only when the client is the sole authoritative source (local notes, draft). Anything multi-device or multi-user.
User Resolution Present both versions to the user and let them choose (or merge). Document editing (Google Docs conflict UI), anything where data loss is unacceptable. High-frequency data or background syncs — user can't resolve 1000 conflicts manually.
CRDTs Conflict-free Replicated Data Types — mathematical data structures designed so all concurrent writes can be merged deterministically without conflicts. Collaborative text editing, counters, sets. Distributed systems where eventual consistency is required. Simple apps — CRDTs add significant implementation complexity.

Implementing Last-Write-Wins with version vectors

// Server sends a version or updatedAt timestamp with every record
class Post {
  final String id;
  final String content;
  final DateTime updatedAt;  // Used for LWW conflict detection
  // ...
}

// In your upsert logic:
Future<void> upsertPost(Post remote) async {
  final local = await _db.postDao.findById(remote.id);
  if (local == null || remote.updatedAt.isAfter(local.updatedAt)) {
    // Remote is newer (or we have no local copy) — accept remote
    await _db.postDao.upsert(remote.toEntity());
  }
  // else: local is newer (pending local edit) — keep local, it will sync later
}
Clock Skew Warning
Last-write-wins with wall-clock timestamps is vulnerable to clock skew between devices. A device with a clock 5 minutes ahead will always "win" conflicts. Use server-assigned timestamps when possible, or use logical clocks (Lamport timestamps, vector clocks) for more correctness guarantees.

4.8 Delta Sync vs Full Sync

When reconnecting after time offline, you need to bring the local DB up to date with the server. Two strategies:

Full Sync

Fetch all records from the server, diff against local state, apply changes. Simple to implement; correct by construction. Expensive at scale — fetching 10,000 posts to find 3 have changed is wasteful.

Future<void> fullSync() async {
  final allRemote = await _remote.fetchAllPosts(); // Expensive for large datasets
  for (final post in allRemote) {
    await _local.upsert(post);
  }
  // Mark locally-deleted records that don't appear in remote
  final remoteIds = allRemote.map((p) => p.id).toSet();
  final localIds = await _local.getAllIds();
  for (final id in localIds.difference(remoteIds)) {
    await _local.delete(id);
  }
}

Delta Sync (Cursor-based)

The server supports a since parameter — it returns only records that have changed since a given cursor (typically a timestamp or sequence number). The client stores the cursor locally and sends it with each sync request.

Future<void> deltaSync() async {
  // Retrieve the last sync cursor (stored in SharedPreferences or Drift)
  final lastSyncedAt = await _prefs.getLastSyncedAt();

  // Fetch only records changed since last sync
  final delta = await _remote.fetchPostsSince(since: lastSyncedAt);

  // Apply changes
  for (final post in delta.upserted) {
    await _local.upsert(post);
  }
  for (final id in delta.deletedIds) {
    await _local.delete(id);
  }

  // Update cursor to the server's reported timestamp
  await _prefs.setLastSyncedAt(delta.syncedAt);
}

// API call shape
// GET /posts?since=2024-01-15T10:30:00Z
// Response: { upserted: [...], deletedIds: [...], syncedAt: "2024-01-15T12:00:00Z" }

Handling soft deletes for delta sync

Delta sync requires the server to tell the client about deletions (not just upserts). This means the server must use soft deletes — marking records as deleted rather than removing them from the DB — so they appear in the since query results. Hard-deleting a record means the client can never learn about the deletion via delta sync.

// Server schema (conceptual)
CREATE TABLE posts (
  id UUID PRIMARY KEY,
  content TEXT,
  updated_at TIMESTAMPTZ NOT NULL,
  deleted_at TIMESTAMPTZ,  -- NULL = not deleted; non-null = soft deleted
  -- Include deleted records in /posts?since= queries
  -- Client checks deleted_at to know whether to upsert or delete locally
);
Staff-Level Insight
At scale, the sync queue and conflict resolution strategy are the hardest parts of offline-first. Most apps can get away with last-write-wins + idempotency keys, but you need a clear answer ready for the interview on what your conflict strategy is and why. "We use last-write-wins with server-assigned timestamps because our users edit their own data and rarely share records — the probability of a meaningful conflict is low, and the simplicity is worth it" is an excellent answer. "We use CRDTs" without being able to explain how they work is not.

Choosing between full sync and delta sync

Factor Full Sync Delta Sync
Implementation complexity Low Medium (requires server cursor support)
Data volume Expensive — all records every sync Efficient — only changes since cursor
Server requirement None beyond basic list endpoint Requires since param + soft deletes
Correctness Simple — hard to miss changes Requires correct cursor management; clock skew can cause missed updates
Recommendation Small datasets (<500 records) Large datasets or frequent syncs
