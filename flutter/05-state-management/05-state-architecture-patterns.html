Part 1 — Principles
Unidirectional Data Flow
The single most important architectural principle for scalable Flutter apps. Data flows in one direction; UI never mutates state directly.

1. Unidirectional Data Flow (UDF)

UDF is the principle that state flows in a single direction: from a single source of truth, through the presentation layer, to the UI. User interactions flow back as events or commands — they never mutate state directly.

User Action
     │
     ▼
  Event / Intent
     │
     ▼
 State Holder (BLoC / Notifier / ViewModel)
     │  reads/writes
     ▼
  Repository (abstracts data source)
     │
     ▼
  Data Source (API / DB / Cache)
     │
     ▼  (new state emitted)
  State Holder
     │
     ▼
  UI Widget (rebuilds reactively)

The key properties of UDF:

  • Single source of truth — state lives in one place; widgets read from it, never store it locally
  • Immutable state — state objects are never mutated in place; new state objects are emitted
  • Predictable — given the same sequence of events, the system always produces the same state sequence
  • Debuggable — you can log every event and state transition (BlocObserver, ProviderObserver)
  • Testable — feed events in, assert states out — no widget tree needed
Staff-Level Insight
UDF is not just a pattern — it's the foundation that makes time-travel debugging, state replay, and crash reporting meaningful. When state is scattered across multiple setState calls in widget trees, reproducing a bug requires reconstructing the entire sequence of user interactions. With UDF, you can log the event stream and replay it exactly.

2. Clean Architecture Layers

Clean Architecture (Robert Martin) applied to Flutter divides code into three layers with strict dependency rules: outer layers depend on inner layers, never the reverse.

┌─────────────────────────────────────────┐
│         Presentation Layer              │
│   Widgets · BLoC/Cubit · Notifier       │
│   ViewModels · UI state models          │
├─────────────────────────────────────────┤
│           Domain Layer                  │
│   Entities · Use Cases · Repositories  │
│   (interfaces only — pure Dart)         │
├─────────────────────────────────────────┤
│            Data Layer                   │
│   Repository Implementations           │
│   Remote DataSources (Dio/GraphQL)      │
│   Local DataSources (Drift/Hive)        │
│   DTOs · Mappers                        │
└─────────────────────────────────────────┘

Domain Layer — No Flutter Dependencies

// Entity — pure business object
class User {
  final String id;
  final String name;
  final String email;
  final UserRole role;

  const User({
    required this.id,
    required this.name,
    required this.email,
    required this.role,
  });

  bool get isAdmin => role == UserRole.admin;
}

// Repository interface — defined in domain, implemented in data
abstract interface class UserRepository {
  Future<User> getUser(String id);
  Future<List<User>> searchUsers(String query);
  Future<void> updateProfile(String id, UserProfile profile);
  Stream<User> watchUser(String id);
}

// Use Case — single responsibility, orchestrates domain logic
class GetUserProfileUseCase {
  final UserRepository _userRepository;
  final PostRepository _postRepository;

  const GetUserProfileUseCase(this._userRepository, this._postRepository);

  Future<UserProfileData> call(String userId) async {
    final (user, posts) = await (
      _userRepository.getUser(userId),
      _postRepository.getPostsByUser(userId),
    ).wait;

    return UserProfileData(
      user: user,
      recentPosts: posts.take(5).toList(),
      postCount: posts.length,
    );
  }
}

Data Layer — Repository Implementation

class UserRepositoryImpl implements UserRepository {
  final UserRemoteDataSource _remote;
  final UserLocalDataSource _local;

  const UserRepositoryImpl(this._remote, this._local);

  @override
  Future<User> getUser(String id) async {
    // Cache-first strategy
    try {
      final cached = await _local.getUser(id);
      if (cached != null && !cached.isStale) {
        return cached.toDomain();
      }
    } catch (_) {} // cache miss is acceptable

    final dto = await _remote.getUser(id);
    await _local.cacheUser(dto);
    return dto.toDomain(); // DTO → Domain entity mapping
  }

  @override
  Stream<User> watchUser(String id) {
    // Local DB stream (reactive — emits on every DB update)
    return _local.watchUser(id).map((dto) => dto.toDomain());
  }
}

// DTO (Data Transfer Object) — mirrors API/DB structure
class UserDto {
  final String id;
  final String displayName;
  final String emailAddress;
  final String roleString;
  final DateTime cachedAt;

  const UserDto({...});

  factory UserDto.fromJson(Map<String, dynamic> json) => UserDto(
    id: json['id'] as String,
    displayName: json['display_name'] as String,
    emailAddress: json['email'] as String,
    roleString: json['role'] as String,
    cachedAt: DateTime.now(),
  );

  // DTO → Domain mapping
  User toDomain() => User(
    id: id,
    name: displayName,
    email: emailAddress,
    role: UserRole.fromString(roleString),
  );

  bool get isStale =>
    DateTime.now().difference(cachedAt) > const Duration(minutes: 5);
}

Presentation Layer — BLoC/Notifier Integration

class UserProfileBloc extends Bloc<UserProfileEvent, UserProfileState> {
  final GetUserProfileUseCase _getUserProfile;

  UserProfileBloc(this._getUserProfile) : super(UserProfileInitial()) {
    on<UserProfileFetched>(_onFetched);
  }

  Future<void> _onFetched(
    UserProfileFetched event,
    Emitter<UserProfileState> emit,
  ) async {
    emit(UserProfileLoading());
    try {
      // Calls use case, not repository directly
      final data = await _getUserProfile(event.userId);
      emit(UserProfileLoaded(data));
    } catch (e, st) {
      addError(e, st);
      emit(UserProfileError(e.toString()));
    }
  }
}

3. AsyncValue Pattern

Riverpod's AsyncValue<T> is a sealed class that models three states of async data: loading, data, and error. It eliminates the need for manual boolean flags and nullable fields.

sealed class AsyncValue<T> {
  const AsyncValue();
}

class AsyncData<T> extends AsyncValue<T> {
  final T value;
  const AsyncData(this.value);
}

class AsyncLoading<T> extends AsyncValue<T> {
  const AsyncLoading();
}

class AsyncError<T> extends AsyncValue<T> {
  final Object error;
  final StackTrace stackTrace;
  const AsyncError(this.error, this.stackTrace);
}

Using AsyncValue in Widgets

// .when() — handle all three cases
ref.watch(productsProvider).when(
  data:    (products) => ProductList(products: products),
  loading: ()         => const SkeletonLoader(),
  error:   (e, st)    => ErrorRetryWidget(error: e, onRetry: () {
    ref.invalidate(productsProvider);
  }),
);

// .whenOrNull() — handle only specific cases, others return null
ref.watch(productsProvider).whenOrNull(
  data: (products) => Text('${products.length} products'),
);

// .maybeWhen() — with orElse fallback
ref.watch(productsProvider).maybeWhen(
  data: (products) => ProductGrid(products: products),
  orElse: () => const LoadingPlaceholder(),
);

// .valueOrNull — get value or null without throwing
final products = ref.watch(productsProvider).valueOrNull ?? [];

// Guard — show previous data while refreshing:
ref.watch(productsProvider).when(
  skipLoadingOnRefresh: true, // shows stale data during refresh
  data:    (p) => ProductList(products: p),
  loading: ()  => const FullPageLoader(),
  error:   (e, _) => ErrorWidget(error: e),
);

Building Your Own AsyncValue Pattern (without Riverpod)

// Usable with BLoC or any state management
sealed class AsyncState<T> extends Equatable {
  const AsyncState();
}

final class AsyncLoading<T> extends AsyncState<T> {
  const AsyncLoading();
  @override List<Object?> get props => [];
}

final class AsyncData<T> extends AsyncState<T> {
  final T data;
  const AsyncData(this.data);
  @override List<Object?> get props => [data];
}

final class AsyncFailure<T> extends AsyncState<T> {
  final String message;
  final Object? error;
  const AsyncFailure(this.message, {this.error});
  @override List<Object?> get props => [message];
}

// In BLoC:
class ProductsBloc extends Bloc<ProductsEvent, AsyncState<List<Product>>> {
  ProductsBloc() : super(const AsyncLoading()) {
    on<ProductsFetched>(_onFetched);
  }

  Future<void> _onFetched(ProductsFetched e, Emitter emit) async {
    emit(const AsyncLoading());
    try {
      final products = await _repo.getAll();
      emit(AsyncData(products));
    } catch (err) {
      emit(AsyncFailure(err.toString(), error: err));
    }
  }
}

4. State Persistence

HydratedBloc

// Automatically persists and restores BLoC state
class CartBloc extends HydratedBloc<CartEvent, CartState> {
  CartBloc() : super(CartState.empty()) {
    on<AddItemToCart>(_onAddItem);
    on<RemoveItemFromCart>(_onRemoveItem);
  }

  @override
  CartState? fromJson(Map<String, dynamic> json) {
    try {
      return CartState.fromJson(json);
    } catch (_) {
      return null; // return null to use initial state on corruption
    }
  }

  @override
  Map<String, dynamic>? toJson(CartState state) => state.toJson();
}

// Setup:
void main() async {
  WidgetsFlutterBinding.ensureInitialized();
  HydratedBloc.storage = await HydratedStorage.build(
    storageDirectory: kIsWeb
      ? HydratedStorage.webStorageDirectory
      : await getApplicationDocumentsDirectory(),
  );
  runApp(const App());
}

Hive for Lightweight Persistence

@HiveType(typeId: 0)
class CachedUser extends HiveObject {
  @HiveField(0) String id;
  @HiveField(1) String name;
  @HiveField(2) String email;
  @HiveField(3) DateTime cachedAt;

  CachedUser({
    required this.id,
    required this.name,
    required this.email,
    required this.cachedAt,
  });
}

// Opening and using a Hive box
class UserLocalDataSource {
  static const _boxName = 'users';

  Future<Box<CachedUser>> get _box async =>
    Hive.isBoxOpen(_boxName)
      ? Hive.box(_boxName)
      : await Hive.openBox(_boxName);

  Future<void> cacheUser(CachedUser user) async {
    final box = await _box;
    await box.put(user.id, user);
  }

  Future<CachedUser?> getUser(String id) async {
    final box = await _box;
    return box.get(id);
  }
}

5. State Restoration

Flutter's state restoration API allows the OS to save and restore widget state across process restarts (Android process death, iOS memory pressure).

// Enable in MaterialApp
MaterialApp(
  restorationScopeId: 'root',
  home: const HomeScreen(),
)

// In a StatefulWidget
class CounterScreen extends StatefulWidget {
  const CounterScreen({super.key});

  @override
  State<CounterScreen> createState() => _CounterScreenState();
}

class _CounterScreenState extends State<CounterScreen>
    with RestorationMixin {

  // RestorableInt automatically saves/restores via RestorationBucket
  final RestorableInt _counter = RestorableInt(0);

  @override
  String get restorationId => 'counter_screen';

  @override
  void restoreState(RestorationBucket? oldBucket, bool initialRestore) {
    // Register all restorable properties
    registerForRestoration(_counter, 'counter');
  }

  @override
  void dispose() {
    _counter.dispose();
    super.dispose();
  }

  @override
  Widget build(BuildContext context) {
    return Scaffold(
      body: Text('Count: ${_counter.value}'),
      floatingActionButton: FloatingActionButton(
        onPressed: () => setState(() => _counter.value++),
        child: const Icon(Icons.add),
      ),
    );
  }
}

Staff-Level Insight — Team Scaling
At L6/Staff level, the key architectural decision is defining the contracts between layers. Repository interfaces in the domain layer serve as the contract between feature teams and platform teams. Feature teams mock the repository interface in tests; platform teams own the implementation. This means teams can work in parallel without blocking each other, and implementations can be swapped (e.g., REST → GraphQL) without touching feature code.

Interview Preparation

When asked "which state management should I use?", demonstrate L6 thinking: it depends on team size, app complexity, and existing patterns. For a new large-scale app with a team, BLoC or Riverpod with Clean Architecture is appropriate. For a prototype or small feature, setState or Cubit suffices. Mention the cost of over-engineering (BLoC for a counter) and under-engineering (setState for auth state shared across 20 screens).
