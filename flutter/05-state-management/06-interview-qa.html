Fundamentals

Standard — L4/L5
When would you use setState vs a state management library?
Use setState when state is truly local to a single widget and does not need to be shared, persisted, or tested in isolation — e.g., a toggle button, an expansion panel, or a text field focus state. Once state needs to be shared across widgets, accessed from business logic, tested independently, or persisted, reach for a library. The rule: if removing the widget also removes the need for the state, it's local state. If not, it belongs outside the widget tree.
Avoid the trap of refactoring prematurely. Starting with setState and lifting state is fine; upgrade when you feel the pain of prop drilling or shared state drift.
Standard — L4/L5
How does InheritedWidget enable dependency tracking?
When a widget calls context.dependOnInheritedWidgetOfExactType<T>(), Flutter registers that widget's Element as a dependent of the nearest ancestor InheritedElement of type T. When the InheritedWidget rebuilds and updateShouldNotify returns true, Flutter calls didChangeDependencies() on every registered dependent element, triggering a rebuild. This is O(n) in the number of dependents but Flutter batches it efficiently within a single frame.
Mention that context.getInheritedWidgetOfExactType<T>() (without "dependOn") reads the value without registering a dependency — useful for one-time reads in callbacks.
Standard — L4/L5
What is the difference between BlocBuilder and BlocListener? Can you use both at once?
BlocBuilder rebuilds its subtree when state changes — use it for rendering UI. BlocListener fires a callback for side effects (navigation, SnackBars, dialogs) without rebuilding. BlocConsumer combines both: it has a builder (for UI) and a listener (for side effects), each with an optional buildWhen/listenWhen predicate. Never put navigation or dialogs inside builder — builder can be called multiple times during a frame, causing duplicate dialogs or navigation pushes.
Standard — L4/L5
What is the difference between ref.watch, ref.read, and ref.listen in Riverpod?
ref.watch subscribes to a provider and causes the calling widget or provider to rebuild when the watched value changes — use it in build() or in a provider's body. ref.read reads the current value once with no subscription — use it inside callbacks, event handlers, or Notifier methods where rebuilds are not needed and would cause errors. ref.listen subscribes but instead of rebuilding, it calls a callback — use it for side effects like navigation or showing a SnackBar in response to state changes.
A common mistake: calling ref.watch inside an onPressed handler. This only works during the build phase; at callback time it reads the stale captured value or throws. Always ref.read in callbacks.
Standard — L4/L5
How does Provider handle memory leaks — and when can they occur?
Provider disposes values when the widget that created them is removed from the tree, provided the provider implements ChangeNotifierProvider or uses dispose. A memory leak can occur when: (1) you use Provider.value() to pass an externally created object — Provider will NOT dispose it; (2) you forget to cancel stream subscriptions inside a ChangeNotifier's dispose(); (3) you hold a reference to a BuildContext that has been unmounted inside an async callback. Always cancel subscriptions and timers in ChangeNotifier.dispose().

BLoC Deep Dives

Deep Dive — L5/L6
Explain EventTransformer and when you'd use restartable() vs droppable().
EventTransformer controls concurrency for a specific event type in a BLoC. It is a function that takes the event stream and an event mapper (the handler function) and returns a transformed stream. Under the hood: restartable() maps to RxDart's switchMap — when a new event arrives, the current handler is cancelled (the Emitter is closed) and a fresh handler starts. droppable() maps to exhaustMap — new events are ignored while one is already processing. Use restartable() for search-as-you-type where stale results from slow requests must be discarded. Use droppable() for button taps that trigger a network mutation — you want to ignore rapid double-taps while the first request is in flight.
Mention the race condition scenario: without restartable(), a slow "dart" query returns after a fast "flutter" query, replacing correct results with stale ones. restartable() cancels the slow request when the faster one starts.
Deep Dive — L5/L6
How do you safely communicate between two BLoCs without tight coupling?
Two recommended patterns: (1) StreamSubscription in constructor — BLoC B subscribes to BLoC A's stream in its constructor and adds internal events in response. Always cancel the subscription in close(). (2) emit.forEach — inside an event handler in BLoC B, call await emit.forEach(blocA.stream, onData: ...). This is preferred because the subscription is automatically cancelled when the Emitter closes (due to BLoC disposal or a restartable() cancellation). Avoid direct method calls between BLoCs — this creates tight coupling and makes testing difficult.
Deep Dive — L5/L6
Walk me through what happens internally when you call bloc.add(MyEvent()).
The event is added to the internal _eventController StreamController. The stream is then piped through the registered EventTransformer for that event type (default: sequential, i.e., asyncExpand). The transformer emits the event to the handler function, which receives an Emitter<State>. When the handler calls emit(newState), BlocBase.emit() checks if the state is equal to the current state (using ==); if not, it calls onChange(), updates _state, and adds the new state to the broadcast _stateController. All BlocBuilder widgets subscribed to bloc.stream receive the new state and check buildWhen before calling setState on themselves.
Standard — L4/L5
How do you test a BLoC that makes async network calls?
Use the bloc_test package with mocked dependencies (mocktail or mockito). Create a mock repository, configure it with when(), build the BLoC with the mock, and use blocTest(): provide the event in act, assert the emitted states in expect (ordered list), and verify repository calls in verify. For error cases, configure the mock to throw and assert the error state is emitted. Use errors: () => [isA<Exception>()] to assert that onError was called. Always call bloc.close() in tearDown.

Riverpod Deep Dives

Deep Dive — L5/L6
How does Riverpod's ProviderContainer differ from InheritedWidget-based Provider?
InheritedWidget-based Provider stores values in the widget tree — you need a BuildContext to access any provider, which means business logic cannot read providers without a widget reference. ProviderContainer is a standalone Dart object outside the widget tree. It maintains a graph of ProviderElements (one per active provider instance), tracks inter-provider dependencies, and manages invalidation cascades. Widgets access it through ProviderScope (which creates the container) and ConsumerWidget (which gets a WidgetRef backed by the container). Business logic (Notifiers) gets a Ref — also backed by the container — with no widget dependency.
This is the key architectural advantage of Riverpod: you can unit test a Notifier by creating a ProviderContainer directly, overriding dependencies, and reading/watching providers — zero widget tree needed.
Deep Dive — L5/L6
When should you use autoDispose and when should you avoid it?
Use autoDispose (or rely on Riverpod 2 codegen's default) for screen-specific data, search results per query, form state, and any data that becomes irrelevant when the user leaves a screen. The provider's state is destroyed when the last listener unsubscribes (typically when a widget is removed from tree), freeing memory and cancelling in-flight requests. Avoid autoDispose for global state: auth session, user preferences, feature flags, shopping cart. These should persist across navigation. If you need controlled lifetime (e.g., cache for 30 seconds), use ref.keepAlive() with a Timer that calls link.close().
Deep Dive — L5/L6
What are the requirements for a family provider parameter? What goes wrong if you ignore them?
The family parameter must implement == and hashCode correctly, because Riverpod uses it as a key in a map to identify provider instances. Primitive types (String, int, bool, enums) satisfy this automatically. If you pass a plain Dart class without overriding equality (e.g., SearchParams(query: 'dart', page: 1)), each call creates a new object with a unique identity, so provider(params1) != provider(params2) even if the fields are identical. This creates a new provider instance (and new network request) every time the widget rebuilds — a major memory leak and performance issue. Fix by using Dart 3 records, Equatable, or Freezed for the parameter type.
Standard — L4/L5
How do you handle loading and error states elegantly with AsyncValue?
AsyncValue<T> is a sealed class with three subtypes: AsyncData, AsyncLoading, and AsyncError. Use .when(data:, loading:, error:) to handle all cases exhaustively in the widget. For less verbose handling: .maybeWhen(data:, orElse:), .whenOrNull(data:), or .valueOrNull. For refresh scenarios where you want to show stale data while refreshing, pass skipLoadingOnRefresh: true to .when(). For mutations inside Notifiers, set state = AsyncLoading(), then AsyncData(result) or AsyncError(e, st) on completion.

Architecture & Design

Design — L6
Design the state management architecture for a large e-commerce app with cart, auth, product catalog, and orders.
Use Clean Architecture with BLoC or Riverpod. Layer separation: Domain — entities (User, Product, CartItem, Order), repository interfaces, use cases (AddToCart, Checkout, GetProductById). Data — repository impls with local cache (Drift) + remote (Dio). Presentation — one BLoC/Notifier per feature area.

State scope: Auth is global (keepAlive, persisted with HydratedBloc). Cart is global (persisted — survives app restart). Product catalog is screen-scoped (autoDispose, paginated with family). Orders list is user-scoped (autoDispose, fetched on demand).

Cross-cutting concerns: Auth state changes should invalidate all user-specific providers (cart, orders, profile). Implement this by watching auth state in each notifier and calling ref.invalidateSelf() when user changes, or by using emit.forEach on the auth BLoC stream.

Concurrency: Cart mutations use sequential() transformer to prevent race conditions. Product search uses restartable(). Checkout uses droppable() to prevent double-submission.
Draw a dependency diagram if on a whiteboard. Show Auth → Cart → Checkout as a dependency chain. Emphasize that invalidating auth cascades down.
Design — L6
How would you migrate a large app from setState/Provider to BLoC without breaking the app?
Use the strangler fig pattern: (1) Identify the most problematic shared state (auth, cart) — start there. (2) Create the BLoC alongside existing code without removing Provider yet. (3) Write tests for the new BLoC first — this forces you to define the event/state contract. (4) Replace one screen at a time, wrapping it in BlocProvider and swapping context.watch for BlocBuilder. (5) Once all consumers of a Provider are migrated, remove the Provider. (6) Move to lower-priority shared state, then local state last.

Key: never do a big-bang rewrite. Keep both systems running side-by-side per feature. Use the BlocObserver and ProviderObserver to compare behavior. Run integration tests after each feature migration.
Design — L6
Compare reactive state management (BLoC/Riverpod) vs imperative (setState/callbacks). When is each appropriate?
Reactive: state is a stream; UI subscribes and rebuilds automatically when state changes. Business logic is decoupled from UI — it emits state, doesn't call UI methods. Advantages: easy to observe, log, test, and replay. Disadvantages: higher abstraction cost, harder to trace call stack for simple operations.

Imperative: state changes are triggered by explicit method calls; UI is told what to update. Advantages: easy to understand, low overhead for simple cases. Disadvantages: state scattered across widgets, hard to test, prone to race conditions in async code.

Rule of thumb: use reactive for domain state (auth, cart, feeds) shared across the app; use imperative (setState) for ephemeral UI state (animation progress, focus, hover) that is purely local. The worst of both worlds is putting domain state in setState and using reactive streams for UI micro-state.
Standard — L4/L5
How do you test state management code that depends on streams (e.g., Firestore real-time listeners)?
Create a mock data source that returns a controlled StreamController. In your test, push events to the controller and assert the resulting state. With BLoC: use blocTest with a mock repository that returns a stream; use act to trigger the event and let the handler subscribe. With Riverpod: create a ProviderContainer, override the stream provider with a StreamController, push values, then read the provider's AsyncValue. Always addTearDown(controller.close) to prevent dangling subscriptions. For time-based behavior, use fake_async to control the event loop.
Standard — L4/L5
What is the role of Equatable in BLoC and why is it important?
BLoC's emit() checks newState == currentState before emitting. Without proper equality, all objects are distinct (default Dart identity equality), so BLoC would emit even when the logical state hasn't changed, causing unnecessary rebuilds. Equatable overrides == and hashCode based on the list returned by props, enabling value-based comparison. This means CartLoaded(items: [a, b]) == CartLoaded(items: [a, b]) even if they are different object instances. Alternatively, use Freezed which generates equality automatically. Important: if a state class contains a List in props, equality compares by list content, not reference — but nested mutable objects still need proper equality.
Deep Dive — L5/L6
How do you handle optimistic updates in BLoC or Riverpod?
Optimistic update pattern: (1) capture current state, (2) immediately emit the "success" state as if the operation succeeded, (3) perform the async operation, (4) on error, emit the captured previous state (rollback) and show an error.

In BLoC: store the current state in a local variable before emitting the optimistic state. On network error, emit(previousState). In Riverpod Notifier: final previous = state; state = optimisticState; then state = previous; in the catch block. Add a SnackBar via ref.listen or BlocListener to notify the user of rollback. For list mutations (add/remove), use copyWith or spread operators to create new state without mutating the existing list.
Emphasize the importance of not mutating the existing state object. Always create a new state instance. Mutations break equality checks and can cause BLoC to miss state changes.
Standard — L4/L5
What is Riverpod's autoDispose and how do you use ref.keepAlive() to control it?
With autoDispose, Riverpod destroys a provider's state when it has no active listeners (typically when the last widget using it leaves the tree). ref.keepAlive() returns a KeepAliveLink that prevents disposal even when no listeners exist. Call link.close() to allow disposal again. A common pattern: cache a provider's value for N seconds after its last listener unsubscribes — Timer(duration, link.close). This prevents thrashing when a user rapidly navigates away and back to the same screen, while still eventually freeing memory for screens that are truly abandoned.
Deep Dive — L6
How would you implement a global error handling strategy across all BLoCs in a large app?
Three complementary layers: (1) BlocObserver — override onError() to catch all unhandled BLoC errors globally. Report to Crashlytics, log to analytics. This is your safety net. (2) Per-BLoC error handling — wrap async operations in try/catch within event handlers. Emit an error state for user-facing errors. Call addError() for unexpected errors so the observer is notified. (3) UI error boundaries — use BlocListener with listenWhen: (_, curr) => curr is ErrorState to show SnackBars or dialogs. For critical errors (session expired, force update), use a root-level BlocListener that can navigate to an error screen from anywhere in the app.

Classify errors: recoverable (show retry), non-recoverable (log out, show error screen), silent (log only). Pass an ErrorType enum in your error state to drive different UI responses.
Standard — L4/L5
How does BlocProvider.value differ from BlocProvider, and when should you use it?
BlocProvider(create: ...) creates a new BLoC instance and owns its lifecycle — it calls close() when the widget is removed from the tree. BlocProvider.value(value: existingBloc) provides an already-created BLoC to a subtree without owning it — it does NOT call close() on disposal. Use BlocProvider.value when: passing a BLoC to a route pushed with Navigator.push (the new route is a separate subtree), sharing a BLoC across multiple tabs without recreating it, or providing a BLoC created at a higher level (e.g., in a DI container) into a subtree. Never use BlocProvider.value to create a new BLoC — the BLoC will never be closed.
