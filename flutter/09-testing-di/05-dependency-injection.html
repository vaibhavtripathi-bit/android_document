DI Approaches Overview

Dependency injection in Flutter has no single canonical solution — the ecosystem offers three distinct philosophies, each with different tradeoffs:

ApproachPackagePatternBest for
Service locator get_it Global registry, pull dependencies Large apps, cross-cutting services
Code generation injectable + get_it Annotations, auto-wired graph Teams preferring compile-time safety
Provider/Riverpod riverpod Reactive DI nodes in the widget tree Flutter-first, tight UI integration
Manual constructor injection None Pass dependencies explicitly Libraries, small features, maximum testability

get_it — Service Locator Internals

get_it is a singleton service locator. Its core is a Map<String, _ServiceFactory> keyed by type.toString() + instanceName. When you call GetIt.I<MyService>(), it looks up the map and returns (or creates) the instance.

Registration Methods

import 'package:get_it/get_it.dart';

final getIt = GetIt.instance;

void setupDependencies() {
  // Eager singleton — created immediately at registration time
  getIt.registerSingleton<AppRouter>(AppRouter());

  // Lazy singleton — created on first call to getIt<AuthService>()
  getIt.registerLazySingleton<AuthService>(
    () => AuthServiceImpl(httpClient: getIt()),
  );

  // Factory — new instance on every call to getIt<LoginViewModel>()
  getIt.registerFactory<LoginViewModel>(
    () => LoginViewModel(authService: getIt()),
  );

  // Factory with parameters
  getIt.registerFactoryParam<ItemViewModel, String, void>(
    (itemId, _) => ItemViewModel(itemId: itemId, repo: getIt()),
  );

  // Named registration — two implementations of the same interface
  getIt.registerLazySingleton<Logger>(
    () => ConsoleLogger(),
    instanceName: 'console',
  );
  getIt.registerLazySingleton<Logger>(
    () => FileLogger(),
    instanceName: 'file',
  );
}

// Usage
final auth = getIt<AuthService>();
final fileLogger = getIt<Logger>(instanceName: 'file');
final vm = getIt<ItemViewModel>(param1: 'item-123');

Async Initialization

// Register a singleton that needs async setup
getIt.registerSingletonAsync<Database>(() async {
  final db = Database();
  await db.initialize();
  return db;
});

// Wait for all async singletons before using them
await getIt.allReady();

get_it Internals: Registration Flow

When you call registerLazySingleton<T>(factory), get_it stores a _ServiceFactory object that holds:

  • The factory closure
  • A flag indicating it's a lazy singleton
  • A reference to the created instance (initially null)

On the first getIt<T>() call, it checks if the instance is null, calls the factory, stores the result, and returns it. On subsequent calls it returns the stored instance directly — O(1) hash map lookup.

injectable — Code Generation for get_it

injectable generates the setupDependencies() function for you from annotations, eliminating manual wiring boilerplate.

# pubspec.yaml
dependencies:
  get_it: ^7.6.0
  injectable: ^2.3.0

dev_dependencies:
  injectable_generator: ^2.4.0
  build_runner: ^2.4.0

Core Annotations

import 'package:injectable/injectable.dart';

// Singleton — one instance for the app lifetime
@singleton
class AppRouter {}

// Lazy singleton — created on first use
@lazySingleton
class AuthService {
  AuthService(this._client);
  final HttpClient _client;
}

// Factory — new instance per injection
@injectable
class LoginViewModel {
  LoginViewModel(this._authService);
  final AuthService _authService;
}

// Abstract type binding — inject MyInterface, get MyServiceImpl
@LazySingleton(as: MyInterface)
class MyServiceImpl implements MyInterface {}

Modules for Third-Party Types

Third-party classes cannot be annotated directly. Use @module to provide them:

@module
abstract class NetworkModule {
  @singleton
  Dio get dio => Dio(BaseOptions(baseUrl: 'https://api.example.com'));

  @singleton
  SharedPreferences get prefs => throw UnimplementedError(); // async — see below
}

// For async third-party dependencies:
@module
abstract class StorageModule {
  @preResolve  // tells injectable to await this before proceeding
  @singleton
  Future<SharedPreferences> get prefs => SharedPreferences.getInstance();
}

Environment-Specific Registrations

// Register different implementations per environment
@dev
@LazySingleton(as: AuthService)
class FakeAuthService implements AuthService { ... }

@prod
@LazySingleton(as: AuthService)
class RealAuthService implements AuthService { ... }

@test
@LazySingleton(as: AuthService)
class MockAuthService implements AuthService { ... }

// main_dev.dart
void main() async {
  await configureDependencies(environment: Environment.dev);
  runApp(const MyApp());
}

// main_prod.dart
void main() async {
  await configureDependencies(environment: Environment.prod);
  runApp(const MyApp());
}

Named Registrations with @Named

@module
abstract class LoggingModule {
  @Named('console')
  @singleton
  Logger get consoleLogger => ConsoleLogger();

  @Named('remote')
  @singleton
  Logger get remoteLogger => RemoteLogger();
}

// Inject by name
@injectable
class AnalyticsService {
  AnalyticsService(@Named('remote') this._logger);
  final Logger _logger;
}

Scoped Dependencies with get_it

For feature-level DI — dependencies that should exist only during a feature's lifecycle — use pushNewScope and popScope:

// When entering the checkout feature
void onEnterCheckout() {
  getIt.pushNewScope(
    scopeName: 'checkout',
    init: (scope) {
      scope.registerLazySingleton<CheckoutRepository>(
        () => CheckoutRepositoryImpl(getIt()),
      );
      scope.registerFactory<CheckoutViewModel>(
        () => CheckoutViewModel(getIt()),
      );
    },
  );
}

// When leaving the checkout feature — disposes all scoped instances
void onExitCheckout() {
  getIt.popScope(); // removes 'checkout' scope and its registrations
}

This is useful for leaking prevention: checkout-scoped services (cart state, payment session) are automatically cleaned up when the user leaves checkout.

Riverpod as a DI Framework

Riverpod providers are dependency injection nodes. Unlike get_it's global mutable registry, Riverpod's ProviderContainer holds all state and can be overridden per test or per feature scope without global mutation.

Providers as DI Nodes

import 'package:flutter_riverpod/flutter_riverpod.dart';

// Infrastructure layer
final httpClientProvider = Provider<Dio>((ref) {
  return Dio(BaseOptions(baseUrl: 'https://api.example.com'));
});

// Repository layer — depends on httpClient
final userRepositoryProvider = Provider<UserRepository>((ref) {
  return UserRepositoryImpl(ref.watch(httpClientProvider));
});

// ViewModel layer — depends on repository
final userViewModelProvider = NotifierProvider<UserViewModel, UserState>(() {
  return UserViewModel();
});

class UserViewModel extends Notifier<UserState> {
  @override
  UserState build() {
    // ref.watch injects the repository at build time
    final repo = ref.watch(userRepositoryProvider);
    return UserState.initial();
  }
}

ref.read for Imperative Access

class UserViewModel extends Notifier<UserState> {
  @override
  UserState build() => UserState.initial();

  Future<void> loadUser(String id) async {
    // ref.read for one-shot reads in methods (not in build)
    final repo = ref.read(userRepositoryProvider);
    state = UserState.loading();
    try {
      final user = await repo.getUser(id);
      state = UserState.loaded(user);
    } catch (e) {
      state = UserState.error(e.toString());
    }
  }
}

Riverpod for Testing — ProviderContainer with Overrides

import 'package:flutter_riverpod/flutter_riverpod.dart';
import 'package:flutter_test/flutter_test.dart';
import 'package:mockito/mockito.dart';

class MockUserRepository extends Mock implements UserRepository {}

void main() {
  late MockUserRepository mockRepo;
  late ProviderContainer container;

  setUp(() {
    mockRepo = MockUserRepository();
    container = ProviderContainer(
      overrides: [
        // Replace the real repository with the mock
        userRepositoryProvider.overrideWithValue(mockRepo),
      ],
    );
  });

  tearDown(() {
    container.dispose();
  });

  test('loads user successfully', () async {
    when(mockRepo.getUser('1')).thenAnswer(
      (_) async => const User(id: '1', name: 'Alice'),
    );

    final vm = container.read(userViewModelProvider.notifier);
    await vm.loadUser('1');

    expect(
      container.read(userViewModelProvider),
      equals(UserState.loaded(const User(id: '1', name: 'Alice'))),
    );
  });
}

Widget Testing with Riverpod Overrides

testWidgets('displays user name after load', (tester) async {
  final mockRepo = MockUserRepository();
  when(mockRepo.getUser(any)).thenAnswer(
    (_) async => const User(id: '1', name: 'Alice'),
  );

  await tester.pumpWidget(
    ProviderScope(
      overrides: [
        userRepositoryProvider.overrideWithValue(mockRepo),
      ],
      child: const MaterialApp(home: UserProfilePage(userId: '1')),
    ),
  );

  await tester.pumpAndSettle();
  expect(find.text('Alice'), findsOneWidget);
});

Manual Constructor Injection

The simplest DI approach: pass every dependency through constructors. No framework, no registration, no code generation — just Dart.

// Pure constructor injection — maximum testability, zero framework dependency
class OrderService {
  const OrderService({
    required this.orderRepository,
    required this.paymentGateway,
    required this.emailService,
  });

  final OrderRepository orderRepository;
  final PaymentGateway paymentGateway;
  final EmailService emailService;

  Future<Order> placeOrder(Cart cart) async {
    final order = await orderRepository.create(cart);
    await paymentGateway.charge(order.total);
    await emailService.sendConfirmation(order);
    return order;
  }
}

// Wiring at the app level — explicit, readable, no magic
void main() {
  final httpClient = Dio();
  final orderRepo = OrderRepositoryImpl(httpClient);
  final payment = StripeGateway(apiKey: const String.fromEnvironment('STRIPE_KEY'));
  final email = SendGridEmailService(apiKey: const String.fromEnvironment('SG_KEY'));

  final orderService = OrderService(
    orderRepository: orderRepo,
    paymentGateway: payment,
    emailService: email,
  );

  runApp(MyApp(orderService: orderService));
}

Constructor injection is best for:

  • Dart packages and libraries (no framework dependency)
  • Small features with few dependencies
  • Domain layer classes that should have zero knowledge of Flutter or DI frameworks

Testing get_it-Based Code

void main() {
  setUp(() {
    // Register test doubles
    getIt.registerSingleton<AuthService>(FakeAuthService());
    getIt.registerSingleton<UserRepository>(FakeUserRepository());
  });

  tearDown(() {
    // CRITICAL: reset the registry between tests
    // Without this, registrations leak between test cases
    getIt.reset();
  });

  test('login succeeds with valid credentials', () async {
    final vm = getIt<LoginViewModel>(); // gets the wired test instance
    await vm.login('user@test.com', 'password');
    expect(vm.state, equals(LoginState.success));
  });
}
Warning

Always call getIt.reset() in tearDown, not just tearDownAll. Each test should start with a clean registry. A registration leak from one test can cause confusing failures in a later, unrelated test.

Anti-Patterns to Avoid

Calling get_it inside build()

// BAD — service locator in the widget tree
class MyWidget extends StatelessWidget {
  @override
  Widget build(BuildContext context) {
    final service = getIt<MyService>(); // anti-pattern: hidden dependency
    return Text(service.value);
  }
}

// GOOD — inject via constructor
class MyWidget extends StatelessWidget {
  const MyWidget({required this.service, super.key});
  final MyService service;

  @override
  Widget build(BuildContext context) => Text(service.value);
}

Hard-Coding Dependencies in Constructors

// BAD — untestable, concrete dependency
class OrderViewModel {
  final repo = OrderRepositoryImpl(Dio()); // cannot be replaced in tests
}

// GOOD — injected dependency
class OrderViewModel {
  OrderViewModel(this._repo);
  final OrderRepository _repo; // interface, replaceable in tests
}

Staff-level perspective

The best DI approach for a large Flutter app is often a hybrid: use Riverpod providers for UI-layer state and reactive dependencies, use get_it + injectable for infrastructure-layer singletons (analytics, crash reporting, HTTP client), and use manual injection for the domain layer. Keeping the domain layer free of all DI framework dependencies makes it portable and maximally testable.
