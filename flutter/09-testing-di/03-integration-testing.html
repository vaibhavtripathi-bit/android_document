What Is an Integration Test?

Integration tests (also called end-to-end tests) run your actual Flutter app on a real device or emulator, against real platform APIs and real network calls (or a staging environment). Unlike widget tests, there is no fake clock, no mock bindings, and no simulated rendering — the full platform stack is exercised.

This makes integration tests the most faithful representation of what your users actually experience, but also the slowest and most fragile. They are best reserved for critical user flows — login, checkout, onboarding — where end-to-end correctness matters most.

The integration_test Package

The integration_test package is the official Flutter package for writing integration tests. It replaces the older flutter_driver package and uses the same testWidgets / WidgetTester API as widget tests, which means your team only needs to learn one testing API.

Setup

Add to pubspec.yaml:

dev_dependencies:
  integration_test:
    sdk: flutter
  flutter_test:
    sdk: flutter

Create the test file at integration_test/app_test.dart:

import 'package:flutter_test/flutter_test.dart';
import 'package:integration_test/integration_test.dart';
import 'package:my_app/main.dart' as app;

void main() {
  // MUST be the first line in main()
  IntegrationTestWidgetsFlutterBinding.ensureInitialized();

  testWidgets('full login flow', (tester) async {
    app.main(); // launch the real app
    await tester.pumpAndSettle();

    // Now interact with the real UI on a real device
    await tester.enterText(find.byKey(const Key('email_field')), 'user@example.com');
    await tester.enterText(find.byKey(const Key('password_field')), 'secret');
    await tester.tap(find.text('Sign In'));
    await tester.pumpAndSettle();

    expect(find.text('Welcome back'), findsOneWidget);
  });
}

Running on Device or Emulator

# Run on a connected device
flutter test integration_test/app_test.dart

# Run on a specific device
flutter test integration_test/app_test.dart -d emulator-5554

# Run on iOS simulator
flutter test integration_test/app_test.dart -d "iPhone 15"

# Run all integration tests in the directory
flutter test integration_test/

IntegrationTestWidgetsFlutterBinding Internals

IntegrationTestWidgetsFlutterBinding.ensureInitialized() registers a binding that:

  • Uses the real scheduler and clock (not fakeAsync).
  • Collects test results and reports them back to the host machine (via the flutter test runner protocol).
  • Supports screenshot capture via binding.takeScreenshot('name').
  • Handles the communication bridge between the test process running on the device and the test runner on the host.
Note

Because integration tests use a real clock, pumpAndSettle works by polling for actual frame stability, not a fake clock. This means it can take several seconds on a real device for animations to complete. Add pumpAndSettle(const Duration(seconds: 10)) as a timeout if needed.

Screenshot Capture

Integration tests support taking screenshots at key points in the flow. These screenshots can be reviewed in CI artifacts or uploaded to services like Firebase Test Lab for visual inspection.

import 'package:integration_test/integration_test.dart';

void main() {
  final binding = IntegrationTestWidgetsFlutterBinding.ensureInitialized();

  testWidgets('captures login page screenshot', (tester) async {
    app.main();
    await tester.pumpAndSettle();

    // Take a screenshot at the login page
    await binding.takeScreenshot('login_page');

    await tester.enterText(find.byKey(const Key('email_field')), 'test@example.com');
    await binding.takeScreenshot('email_filled');

    await tester.tap(find.text('Sign In'));
    await tester.pumpAndSettle();
    await binding.takeScreenshot('home_page_after_login');
  });
}

Screenshots are saved to a directory on the device and retrieved by the test runner. On Firebase Test Lab, they appear in the test results UI.

Patrol — Handling Native UI

The biggest limitation of integration_test is that it can only interact with Flutter widgets. It cannot handle:

  • System permission dialogs (camera, location, notifications)
  • Native alert dialogs from the OS
  • Notifications in the notification shade
  • Deep links from other apps

The Patrol library (patrol package) solves this by providing a native automation layer on top of integration_test.

Patrol Setup

# pubspec.yaml
dev_dependencies:
  patrol: ^3.0.0

# Install the Patrol CLI
dart pub global activate patrol_cli

# Integrate into Android (adds UIAutomator2)
patrol doctor

PatrolTester API

PatrolTester wraps WidgetTester and adds native interaction capabilities. It uses the $ shorthand for finding widgets.

import 'package:patrol/patrol.dart';

void main() {
  patrolTest(
    'grants camera permission and takes photo',
    nativeAutomation: true,
    ($) async {
      await $.pumpWidgetAndSettle(const MyApp());

      // Interact with Flutter UI
      await $(#camera_button).tap();

      // Handle the native permission dialog
      await $.native.grantPermissionWhenInUse();

      // Continue with Flutter UI
      await $(#capture_button).tap();
      await $.pumpAndSettle();

      expect($(#photo_preview), findsOneWidget);
    },
  );
}

Firebase Test Lab

Firebase Test Lab lets you run your integration tests on a matrix of real physical devices in Google's data center — hundreds of device models, screen sizes, and OS versions.

Setup

# Build the app APK and test APK (Android)
flutter build apk --debug
flutter build apk --debug --target=integration_test/app_test.dart

# Or use the Gradle test APK target
pushd android
./gradlew app:assembleDebug app:assembleAndroidTest -PtestBuildType=debug
popd

# Upload and run on Firebase Test Lab
gcloud firebase test android run \
  --type instrumentation \
  --app build/app/outputs/flutter-apk/app-debug.apk \
  --test build/app/outputs/flutter-apk/app-debug-androidTest.apk \
  --device model=Pixel3,version=30,locale=en,orientation=portrait \
  --device model=oriole,version=32,locale=en,orientation=portrait

iOS on Firebase Test Lab

# Build the iOS test runner (XCTest bundle)
xcodebuild build-for-testing \
  -workspace ios/Runner.xcworkspace \
  -scheme Runner \
  -destination generic/platform=iOS \
  -derivedDataPath ios/DerivedData

# Zip the runner for upload
cd ios/DerivedData/Build/Products/Debug-iphoneos
zip -r Runner.zip Runner.app RunnerUITests-Runner.app

# Upload to Firebase
gcloud firebase test ios run \
  --test Runner.zip \
  --device model=iphone14pro,version=16.6

Interpreting Results

Firebase Test Lab provides:

  • Pass/fail per device per test case
  • Video recording of each test run
  • Logcat (Android) / device log (iOS) output
  • Screenshots captured via binding.takeScreenshot
  • Performance metrics (CPU, memory) optionally

Widget Tests vs Integration Tests — Decision Guide

DimensionWidget TestIntegration Test
Runs onHost machine (no device)Real device / emulator
SpeedSeconds per testMinutes per test
Platform channelsMust mockReal, works natively
AnimationsFake clock, full controlReal time, less control
NetworkMust mockReal or staging
FlakinessVery low (deterministic)Higher (timing, network)
CoverageComponent behaviorFull user flow
Best forUI components, state logicCritical paths, platform integration
Staff-level perspective

At Google scale, the testing pyramid matters: many unit tests, a moderate number of widget tests, a small number of carefully selected integration tests. Integration tests that cover platform integration (permissions, notifications, deep links, payment flows) are worth the cost. Integration tests that duplicate what widget tests already verify are not — they add CI time without adding signal.

Test Coverage from Integration Tests

You can collect Dart code coverage from integration tests, though it requires additional setup because the code runs on a device.

# Run with coverage collection
flutter test integration_test/app_test.dart --coverage

# Generate HTML report
genhtml coverage/lcov.info -o coverage/html
open coverage/html/index.html
Tip

Coverage from integration tests measures which code paths were exercised during the E2E flow, not whether individual edge cases were tested. Combine with unit test coverage for a complete picture. A high integration test coverage number with no unit tests is a red flag — edge cases are untested.

Structuring Integration Tests in CI

# GitHub Actions example: run integration tests on Android emulator
name: Integration Tests

on: [push]

jobs:
  integration-test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: subosito/flutter-action@v2
        with:
          flutter-version: '3.22.0'

      - name: Start Android emulator
        uses: reactivecircus/android-emulator-runner@v2
        with:
          api-level: 33
          script: flutter test integration_test/

      # OR: upload to Firebase Test Lab
      - name: Run on Firebase Test Lab
        run: |
          flutter build apk --debug
          gcloud firebase test android run \
            --app build/app/outputs/flutter-apk/app-debug.apk \
            --test build/app/outputs/flutter-apk/app-debug-androidTest.apk \
            --device model=Pixel6,version=33
