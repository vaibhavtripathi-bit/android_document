Widget Testing

Standard
Why does pumpAndSettle sometimes hang indefinitely?
pumpAndSettle repeatedly calls pump(100ms) until there are no pending frames, animations, or timers. If the widget tree contains an infinite animation — a looping AnimationController, a Timer.periodic, or a Stream.periodic — there will always be a pending frame, and pumpAndSettle will never terminate, eventually hitting its timeout and throwing a FlutterError. The fix is either to stop the animation before calling pumpAndSettle, or use a bounded pump(duration) sequence instead — e.g. await tester.pump(const Duration(seconds: 2)).
Mention the timeout parameter: pumpAndSettle(const Duration(seconds: 30)) — and note that hitting the timeout is itself a test failure, not a hang.
Deep dive
How do you test a widget that depends on a MethodChannel?
Widget tests run on the host machine with no native layer, so any MethodChannel call will throw a MissingPluginException unless you install a mock handler. Use TestDefaultBinaryMessengerBinding.instance.defaultBinaryMessenger.setMockMethodCallHandler(channel, handler) in setUp and clear it in tearDown by passing null as the handler. The handler receives a MethodCall and returns an Object? matching what the native side would return. For higher-level plugins, it is usually better to mock the Dart-level API (repository or service class) rather than the raw channel, making tests more resilient to plugin internals changing.
Show both approaches: raw channel mock and Dart-level mock. Interviewers want to see you know the escape hatch but prefer the clean solution.
Standard
What is WidgetTester.runAsync and when is it needed?
testWidgets wraps the test body in fakeAsync, which replaces the Dart event loop's timer/microtask handling with a synthetic clock. Real I/O — HTTP calls, file reads, isolate messages — does not work inside fakeAsync because those futures never resolve. tester.runAsync(fn) temporarily exits the fakeAsync zone, runs the provided async function in the real async context (real clock, real I/O), then re-enters fakeAsync. It should be avoided when possible — prefer to mock async dependencies so they return Future.value(data) and remain compatible with the fake clock.
Distinguish clearly between fakeAsync (synthetic clock, no real I/O) and runAsync (real clock, real I/O). This distinction trips up many candidates.
Standard
How do you test navigation in a widget test?
Inject a MockNavigatorObserver into the MaterialApp's navigatorObservers list. The observer's didPush, didPop, didReplace methods are called by the Navigator whenever routes change. After tapping the button that triggers navigation and calling pumpAndSettle, use Mockito's verify(observer.didPush(any, any)) to assert the push occurred. For named routes, you can also check route.settings.name in the captured argument to confirm the correct route was navigated to.
Also mention GoRouter testing: GoRouter provides a GoRouterNavigatorObserver and you can use GoRouter.of(context).location to assert the current route string.

Integration vs Widget Tests

Standard
What is the difference between widget tests and integration tests? When do you choose one over the other?
Widget tests run entirely on the host machine in a fake environment — no real device, no platform channels, fake clock. They are fast (seconds), deterministic, and can mock everything. Integration tests run on a real device or emulator with the full Flutter engine, real platform channels, and real timing. They take minutes and can be flaky due to timing and network variance. Choose widget tests for UI component behavior, state transitions, and anything you can mock. Choose integration tests for critical user flows that involve real platform features — permissions, payment SDKs, deep links, push notifications — where you need to verify the native layer works correctly end-to-end.
Frame it as the testing pyramid: many unit tests, a good number of widget tests, a small set of integration tests covering critical paths only. Staff engineers think about CI cost vs confidence tradeoff.

Golden Tests

Deep dive
When would you use golden tests and how do you manage them in CI?
Golden tests are best for design system components — buttons, cards, typography — where pixel accuracy is the contract. They catch visual regressions that no behavioral test would catch: a wrong border radius, a missing shadow, an off-by-one padding. For CI, the critical rules are: (1) pin a specific Linux runner and Flutter version — never run goldens on ubuntu-latest or they will spuriously fail on runner upgrades; (2) commit golden PNGs to git so diffs are reviewable in PRs; (3) treat golden failures as blocking, not warnings; (4) upload diff images as CI artifacts on failure so reviewers can see exactly what changed. Use the alchemist package for multi-scenario goldens and platform-agnostic rendering, and configure a tolerance comparator (0.3–0.5% pixel diff) to absorb sub-pixel antialiasing differences between generations.
Mention the tagging strategy: @Tags(['golden']) so goldens can be run separately from the main test suite and excluded from the fast feedback loop during development.
Standard
Why do golden tests fail when run on a different OS or Flutter version?
Golden tests compare rendered pixels. Font rendering differs between operating systems: macOS uses CoreText with its own hinting and antialiasing algorithm; Linux uses FreeType, which produces different sub-pixel values for the same glyph. Even the same OS can produce different results across Flutter versions because the Skia/Impeller rendering engine can change how curves are rasterized. The solutions are: load fonts explicitly via FontLoader so both environments use identical font bytes; run all CI goldens on a pinned Linux runner with a pinned Flutter version; use a tolerance comparator that allows a small percentage of pixel variance; or use alchemist with its platform-agnostic rendering mode which uses a fixed font to eliminate the OS difference entirely.
Concrete numbers help: "a 0.5% pixel tolerance absorbs antialiasing differences without masking real regressions."

Dependency Injection

Deep dive
Compare get_it and Riverpod as dependency injection solutions. When would you choose each?
get_it is a service locator — a global mutable registry keyed by type. It is framework-agnostic (works in plain Dart), has low setup cost, and handles eager/lazy singletons and factories. Its weaknesses: it is runtime-resolved (type errors only appear at runtime if a type is unregistered), it holds global mutable state that must be carefully cleaned up in tests via getIt.reset(), and it has no reactivity — when a dependency changes, nothing updates automatically. Riverpod is a reactive DI framework designed for Flutter. Providers compose and observe each other, the ProviderContainer is scoped (not global), and overrides in tests are clean and isolated. Its weakness is Flutter coupling — it is harder to use in pure Dart code outside the widget tree. The pragmatic choice: Riverpod for UI-layer state and reactive dependencies; get_it for infrastructure singletons (analytics, crash reporting, database) that are configured once at startup and do not need reactivity.
The "hybrid" answer — Riverpod for UI, get_it for infrastructure — shows architectural maturity and scores well at the L6 level.
Standard
How do you mock a repository in a widget test using Riverpod?
Wrap the widget under test in a ProviderScope and pass an overrides list that replaces the real provider with a mock value. For example: ProviderScope(overrides: [userRepositoryProvider.overrideWithValue(mockRepo)], child: MaterialApp(home: MyPage())). The widget tree will receive the mock repository through ref.watch(userRepositoryProvider) just as it would in production, but the mock controls what data is returned. This approach requires no global state mutation and is fully isolated per test — unlike get_it which requires reset() between tests.
Also mention overrideWith for providers that need their full build lifecycle, vs overrideWithValue for simple synchronous mocks.
Standard
What is the purpose of injectable and what does it generate?
injectable is a code generation package that reads annotations (@singleton, @lazySingleton, @injectable, @module) on your classes and generates a configureDependencies() function that registers all annotated types with get_it. Without injectable, you write and maintain a manual setupDependencies() function where you register each class one by one. With injectable, you annotate the class itself and the generator builds the wiring. It also supports environment-specific registrations via @dev / @prod / @test and named registrations via @Named. The tradeoff is the build_runner step — every time you add or change an injectable class you must run dart run build_runner build.
Mention that injectable requires build_runner and adds a codegen step to the workflow. At Google scale, codegen is common and accepted — but explain the tradeoff clearly.

BLoC and State Management Testing

Deep dive
How do you test a BLoC or Cubit?
The bloc_test package provides blocTest, a declarative API for testing BLoCs and Cubits without a widget tree. You specify an act (what events to add or methods to call), a build (how to create the BLoC with mocked dependencies), and an expect (the sequence of states expected). For example: 
blocTest<LoginCubit, LoginState>(
  'emits [loading, success] on valid login',
  build: () => LoginCubit(authService: mockAuthService),
  setUp: () => when(mockAuthService.login(any, any))
      .thenAnswer((_) async => const User(id: '1')),
  act: (cubit) => cubit.login('a@b.com', 'pass'),
  expect: () => [LoginState.loading(), LoginState.success(user)],
);
For raw BLoCs (event-based), use act: (bloc) => bloc.add(const LoginSubmitted()). You can also test error paths with errors: () => [isA<AuthException>()].
The key insight: BLoC/Cubit logic is pure Dart — no widget tree needed, no pump() calls. This is by design and is the main testability advantage of BLoC over approaches that put logic in widgets.
Standard
How do you test a widget that uses a BLoC?
Use BlocProvider.value to inject a pre-built BLoC (or mock Cubit) into the widget tree, bypassing the normal provider creation. Create a real or mock Cubit in setUp, seed it with the state you want to test, then wrap the widget: 
await tester.pumpWidget(
  BlocProvider<LoginCubit>.value(
    value: mockLoginCubit,
    child: const MaterialApp(home: LoginPage()),
  ),
);
when(mockLoginCubit.state).thenReturn(LoginState.loading());
await tester.pump();
expect(find.byType(CircularProgressIndicator), findsOneWidget);
For Cubits, you can also use the real Cubit with a mocked repository dependency, which avoids the complexity of mocking the Cubit itself — testing the full BLoC-to-widget integration with only the repository mocked.
Prefer injecting a real Cubit with mocked dependencies over mocking the Cubit itself. The former tests the actual state machine; the latter only tests the widget's reaction to states.

Advanced

Deep dive
How do you structure tests for a large Flutter app with many features and teams?
At scale, testing structure mirrors the architecture. Each feature module owns its own unit, widget, and integration tests co-located with the feature code. A shared test_helpers package provides: fake implementations of interfaces, test fixtures (canned data objects), custom matchers, pump helpers (e.g. pumpApp that wraps in MaterialApp + providers), and golden test infrastructure (font loading, comparator config). CI is split into fast and slow pipelines: unit + widget tests run on every PR (fast, <5 min); golden tests run on a pinned runner on every PR as a blocking check; integration tests run nightly on Firebase Test Lab. Feature flags and environment config are injected via --dart-define or injectable environments, so CI can run against a staging backend. Coverage gates (e.g. 80% line coverage) are enforced via flutter test --coverage + lcov thresholds in CI.
Structure your answer around three axes: (1) co-location of tests with features, (2) shared test infrastructure, (3) CI pipeline split by speed and confidence level. This demonstrates systems thinking, not just individual test knowledge.
Deep dive
What are scoped dependencies in get_it and when would you use them?
get_it supports multiple named scopes via pushNewScope and popScope. When you push a new scope, you can register additional types that are only available within that scope. When you pop the scope, those registrations and their instances are disposed. This is useful for feature-level dependencies that should not outlive their feature — for example, a checkout feature might push a scope that registers CheckoutRepository, PaymentSession, and CheckoutViewModel when the user enters checkout, and pop the scope when they leave. This prevents the payment session (potentially holding sensitive state) from lingering in memory after checkout is complete. Types registered in parent scopes remain accessible from child scopes, so you can still resolve the global AuthService from within the checkout scope.
This is an L6-level get_it topic — most candidates only know the basic singleton registration. Knowing scopes signals deep framework familiarity.
Standard
What is flutter_test_config.dart and what can you put in it?
flutter_test_config.dart is a special file that Flutter's test runner automatically picks up from the test/ directory (and any subdirectory). It must export a testExecutable function with the signature Future<void> testExecutable(FutureOr<void> Function() testMain) async. Flutter calls this function around every test file in the directory — you call testMain() inside it. This is the right place for global test setup that applies to all tests in a directory: loading fonts for golden tests, configuring the goldenFileComparator, registering mock packages, and setting up EasyLocalization or other global initialization. Subdirectories can have their own flutter_test_config.dart that overrides or extends the parent.
Many candidates know setUp/setUpAll but not flutter_test_config.dart. Knowing this file shows you've worked on real production test infrastructure.
Deep dive
How would you test a widget that uses SharedPreferences or other platform plugins?
Platform plugins that use MethodChannel need mock channel handlers in widget tests. For SharedPreferences specifically, the package provides a test helper: SharedPreferences.setMockInitialValues({'key': 'value'}) which installs a mock implementation that stores values in memory. For custom or third-party plugins, use TestDefaultBinaryMessengerBinding.instance.defaultBinaryMessenger.setMockMethodCallHandler. However, the cleanest approach is to abstract the plugin behind an interface in your app — e.g. abstract class KeyValueStore { Future<String?> get(String key); ... } — and mock that interface in tests. This way, widget tests have no knowledge of SharedPreferences at all, and you only need one integration test (or unit test) that verifies SharedPreferencesKeyValueStore actually calls the right methods.
The abstraction answer — hide plugins behind interfaces — is the L6 answer. It shows you design for testability at the architecture level, not just at the test level.
