1. The test Package

Dart's test package is the standard testing library for all Dart code, including Flutter. It provides a rich set of primitives for structuring test suites, managing setup/teardown, and composing assertions.

Core Primitives

// pubspec.yaml
// dev_dependencies:
//   test: ^1.24.0
//   flutter_test:  (re-exports test + flutter extras)

import 'package:test/test.dart';

void main() {
  // Top-level group — maps to a test suite
  group('ShoppingCart', () {
    late ShoppingCart cart;

    // Runs before EACH test in this group
    setUp(() {
      cart = ShoppingCart();
    });

    // Runs after EACH test
    tearDown(() {
      cart.dispose();
    });

    // Runs ONCE before the entire group
    setUpAll(() async {
      await Database.initialize();
    });

    // Runs ONCE after the entire group
    tearDownAll(() async {
      await Database.teardown();
    });

    test('adds item correctly', () {
      cart.addItem(Item('apple', price: 1.5));
      expect(cart.totalItems, equals(1));
      expect(cart.subtotal, closeTo(1.5, 0.001));
    });

    test('throws on negative quantity', () {
      expect(
        () => cart.addItem(Item('apple', quantity: -1)),
        throwsA(isA<ArgumentError>()),
      );
    });

    // Nested groups for sub-behavior
    group('when cart is empty', () {
      test('subtotal is zero', () {
        expect(cart.subtotal, isZero);
      });

      test('isEmpty returns true', () {
        expect(cart.isEmpty, isTrue);
      });
    });

    // skip a test
    test('discount logic', () { /* ... */ }, skip: 'discount not implemented yet');

    // solo test — run only this during development
    test('urgent fix', () { /* ... */ }, solo: true);

    // timeout override
    test('slow network call', () async {
      // ...
    }, timeout: Timeout(Duration(seconds: 30)));
  });
}
setUp vs setUpAll

setUp creates fresh state for every test — always prefer it over setUpAll to avoid test-order dependencies. Use setUpAll only for expensive side-effect-free resources (e.g., spawning a test database server once). Sharing mutable state via setUpAll is a common source of flaky tests.

2. Matchers and Assertions

The expect(actual, matcher) API is powered by Matcher objects from the matcher package. Matchers compose, enabling precise, readable assertions.

// Equality
expect(result, equals(42));
expect(list, orderedEquals([1, 2, 3]));
expect(set, unorderedEquals([3, 1, 2]));

// Type
expect(obj, isA<String>());
expect(obj, TypeMatcher<Repository>());

// Numeric
expect(value, greaterThan(10));
expect(value, inClosedOpenRange(0, 100));
expect(value, closeTo(3.14, 0.01));  // within delta

// Collections
expect(list, contains('item'));
expect(list, hasLength(3));
expect(list, everyElement(greaterThan(0)));
expect(map, containsPair('key', 'value'));

// String
expect(str, startsWith('flutter'));
expect(str, contains('test'));
expect(str, matches(RegExp(r'\d{4}-\d{2}-\d{2}')));

// Exceptions
expect(() => riskyFn(), throwsA(isA<StateError>()));
expect(() => riskyFn(), throwsA(
  predicate<StateError>((e) => e.message.contains('empty'))
));
expect(() => riskyFn(), throwsArgumentError);  // convenience
expect(() => riskyFn(), throwsException);

// Futures
expect(fetchUser(), completion(equals(User(id: '123'))));
expect(failingFuture(), throwsA(isA<NetworkException>()));

// Logical composition
expect(value, allOf(greaterThan(0), lessThan(100)));
expect(value, anyOf(equals(0), greaterThan(10)));
expect(value, isNot(equals(-1)));

3. Testing Async Code

Dart tests naturally support async functions. The test runner awaits the returned Future. Understanding pumpEventQueue is critical when testing microtask-queue behavior without Flutter's frame pump.

// Simple async test — test runner awaits the future
test('fetchUser returns a user', () async {
  final user = await repository.fetchUser('123');
  expect(user.name, equals('Alice'));
});

// Testing future-based error paths
test('fetchUser throws on 404', () async {
  expect(
    () async => await repository.fetchUser('unknown'),
    throwsA(isA<NotFoundException>()),
  );
  // Alternative with completion matcher for the future itself:
  await expectLater(
    repository.fetchUser('unknown'),
    throwsA(isA<NotFoundException>()),
  );
});

// pumpEventQueue: drain the microtask queue N times
// Essential when testing code that uses Future.microtask / scheduleMicrotask
test('processes microtask callbacks', () async {
  bool callbackInvoked = false;
  someService.scheduleCallback(() => callbackInvoked = true);

  // The microtask hasn't run yet — it's queued
  expect(callbackInvoked, isFalse);

  // Drain the event queue: runs all pending microtasks + timer callbacks
  await pumpEventQueue(times: 1);

  expect(callbackInvoked, isTrue);
});

// FakeAsync for deterministic time control (from 'package:fake_async')
import 'package:fake_async/fake_async.dart';

test('debounce fires after 500ms', () {
  fakeAsync((async) {
    bool fired = false;
    final debouncer = Debouncer(duration: Duration(milliseconds: 500));
    debouncer.run(() => fired = true);

    async.elapse(Duration(milliseconds: 499));
    expect(fired, isFalse);

    async.elapse(Duration(milliseconds: 1));
    expect(fired, isTrue);
  });
});
Architect Insight

pumpEventQueue drains the Dart event loop by repeatedly yielding via Future.delayed(Duration.zero). The times parameter controls how many scheduling rounds occur. For deep async chains, you may need times: 100. In Flutter widget tests, prefer pump() and pumpAndSettle() which integrate with the frame scheduler.

4. Testing Streams

The StreamMatcher API in the test package provides declarative assertions over stream emissions. expectLater is mandatory (not expect) since the stream assertion is asynchronous.

import 'package:test/test.dart';

test('counter stream emits 0, 1, 2', () async {
  final stream = counterBloc.stream;

  await expectLater(
    stream,
    emitsInOrder([0, 1, 2]),
  );
});

// emits: single emission matcher
test('emits error then completes', () async {
  await expectLater(
    errorStream,
    emitsInOrder([
      emitsError(isA<NetworkException>()),
      emitsDone,
    ]),
  );
});

// emitsThrough: ignore events UNTIL matcher matches, then check rest
test('eventually emits Loading state', () async {
  await expectLater(
    stateStream,
    emitsThrough(isA<LoadingState>()),
  );
});

// emitsAnyOf: matches if any sub-matcher matches the next emission
test('emits success or cached result', () async {
  await expectLater(
    resultStream,
    emitsAnyOf([
      isA<SuccessState>(),
      isA<CachedState>(),
    ]),
  );
});

// Broadcast streams — listen before triggering
test('repository emits on save', () async {
  final future = expectLater(
    repo.changeStream,         // broadcast stream
    emits(isA<UserUpdated>()),
  );
  await repo.saveUser(alice);  // trigger emission AFTER setting up matcher
  await future;
});

5. Mockito (Code Generation)

Mockito's Dart port uses build_runner code generation to produce null-safe mock classes. The @GenerateMocks annotation drives the generator.

Setup

# pubspec.yaml
# dev_dependencies:
#   mockito: ^5.4.0
#   build_runner: ^2.4.0

// test/mocks.dart
import 'package:mockito/annotations.dart';
import 'package:my_app/repositories/user_repository.dart';
import 'package:my_app/services/analytics_service.dart';

@GenerateMocks([UserRepository, AnalyticsService])
void main() {}

// Run: flutter pub run build_runner build --delete-conflicting-outputs
// Generates: test/mocks.mocks.dart

Stubbing and Verification

import 'package:mockito/mockito.dart';
import 'mocks.mocks.dart';

void main() {
  group('UserBloc', () {
    late MockUserRepository mockRepo;
    late UserBloc bloc;

    setUp(() {
      mockRepo = MockUserRepository();
      bloc = UserBloc(repository: mockRepo);
    });

    test('emits user on load', () async {
      final user = User(id: '1', name: 'Alice');

      // thenReturn: synchronous stubbing
      when(mockRepo.getUser('1')).thenReturn(user);

      // thenAnswer: async stubbing (use for Futures/Streams)
      when(mockRepo.fetchUser('1'))
          .thenAnswer((_) async => user);

      // thenThrow: stub exceptions
      when(mockRepo.fetchUser('bad'))
          .thenThrow(NotFoundException('bad'));

      bloc.add(LoadUserEvent('1'));
      await expectLater(bloc.stream, emits(UserLoadedState(user)));

      // Verify interactions
      verify(mockRepo.fetchUser('1')).called(1);
      verifyNever(mockRepo.deleteUser(any));
    });

    test('verify call order', () async {
      when(mockRepo.fetchUser(any)).thenAnswer((_) async => alice);
      when(mockRepo.saveUser(any)).thenAnswer((_) async {});

      await service.refreshAndSave('1');

      // verifyInOrder ensures calls happened in sequence
      verifyInOrder([
        mockRepo.fetchUser('1'),
        mockRepo.saveUser(any),
      ]);
    });

    test('no unexpected interactions', () {
      bloc.doNothing();

      // verifyNoMoreInteractions after verifying expected ones
      verifyZeroInteractions(mockRepo);
    });
  });
}
Common Pitfall: thenReturn with Futures

Never use thenReturn(Future.value(x)) — it evaluates the Future immediately and can cause "Future already completed" errors. Always use thenAnswer((_) async => x) or thenAnswer((_) => Future.value(x)) for Future-returning methods. This is a Mockito gotcha that trips even senior engineers.

6. Mocktail (No Code Generation)

Mocktail provides Mockito-compatible APIs without requiring code generation. It uses Dart's noSuchMethod to intercept calls at runtime. The trade-off: no compile-time verification of stubbing, but zero boilerplate for test setup.

# pubspec.yaml
# dev_dependencies:
#   mocktail: ^1.0.0

import 'package:mocktail/mocktail.dart';

// Define mocks by extending Mock — NO annotation or generation needed
class MockUserRepository extends Mock implements UserRepository {}
class MockAnalyticsService extends Mock implements AnalyticsService {}

// For classes with non-default constructors: use Fake
class FakeUser extends Fake implements User {}

void main() {
  // registerFallbackValue is REQUIRED for custom types used with any()
  // Must be called once, typically in setUpAll
  setUpAll(() {
    registerFallbackValue(FakeUser());
    registerFallbackValue(UserQuery.empty());
  });

  group('UserBloc', () {
    late MockUserRepository mockRepo;

    setUp(() {
      mockRepo = MockUserRepository();
    });

    test('fetches user by id', () async {
      final user = User(id: '1', name: 'Alice');

      // when/thenAnswer — identical API to Mockito
      when(() => mockRepo.fetchUser('1'))
          .thenAnswer((_) async => user);

      // Key difference from Mockito: lambdas wrap the call
      when(() => mockRepo.saveUser(any()))
          .thenAnswer((_) async {});

      final result = await mockRepo.fetchUser('1');
      expect(result, equals(user));

      // verify with lambda syntax
      verify(() => mockRepo.fetchUser('1')).called(1);
      verifyNever(() => mockRepo.deleteUser(any()));
    });

    test('any() with custom type requires fallback', () async {
      // Without registerFallbackValue(FakeUser()), any() throws:
      // "Unsupported argument matcher used for a non-nullable type"
      when(() => mockRepo.saveUser(any()))
          .thenAnswer((_) async {});

      await mockRepo.saveUser(alice);
      verify(() => mockRepo.saveUser(any())).called(1);
    });
  });
}
Feature Mockito Mocktail
Code generation Required (build_runner) Not required
Null safety Full (compile-time) Full (runtime)
Stub syntax when(mock.method()) when(() => mock.method())
Fallback values Not needed (generated) Required for custom types
Build time impact Slower (codegen) Faster
Verification errors Compile-time (method signatures) Runtime

7. Fakes vs Mocks

A mock records all interactions and allows you to assert which methods were called. A fake is a lightweight real implementation — it has actual behavior but simpler than production (e.g., an in-memory database instead of a real one). Choosing between them affects test complexity.

// MOCK: track interactions, no real behavior needed
class MockAnalyticsService extends Mock implements AnalyticsService {}

// Use mock when: you need to verify a side-effect was triggered
test('tracks event on purchase', () async {
  final analytics = MockAnalyticsService();
  when(() => analytics.trackEvent(any())).thenAnswer((_) async {});

  await purchaseService.completePurchase(cart, analytics);
  verify(() => analytics.trackEvent('purchase_completed')).called(1);
});

// FAKE: real behavior with in-memory backing
class FakeUserRepository implements UserRepository {
  final _store = <String, User>{};

  @override
  Future<User> fetchUser(String id) async {
    final user = _store[id];
    if (user == null) throw NotFoundException(id);
    return user;
  }

  @override
  Future<void> saveUser(User user) async => _store[user.id] = user;

  @override
  Future<void> deleteUser(String id) async => _store.remove(id);
}

// Use fake when: testing complex state transitions without verifying calls
test('round-trip save and retrieve', () async {
  final repo = FakeUserRepository();
  await repo.saveUser(alice);
  final retrieved = await repo.fetchUser(alice.id);
  expect(retrieved, equals(alice));
});
When to Use Each
  • Mock: When you need to assert that something was called (interaction testing). Prefer for services with side effects (analytics, logging, network).
  • Fake: When you need the dependency to behave correctly but want to avoid infrastructure (database, network). Better for multi-step state tests.
  • Stub: A minimal implementation returning hardcoded values — use when behavior doesn't matter, only return values do.
  • Spy: Wraps a real object, recording calls while delegating to the real implementation. Use sparingly — often indicates design smell.

8. Testing BLoC/Cubit with bloc_test

The bloc_test package provides a DSL specifically designed for BLoC and Cubit, handling stream subscription and comparison automatically.

# dev_dependencies:
#   bloc_test: ^9.1.0

import 'package:bloc_test/bloc_test.dart';

// CounterCubit under test:
class CounterCubit extends Cubit<int> {
  CounterCubit() : super(0);
  void increment() => emit(state + 1);
}

// blocTest signature:
// blocTest<BlocType, StateType>(description,
//   build: () => BlocType,
//   act: (bloc) async { ... },  // interactions
//   expect: () => [states],     // expected emission sequence
//   verify: (bloc) { ... },     // post-condition checks
//   errors: () => [errors],     // expected thrown errors
//   setUp: () async { ... },
//   tearDown: () async { ... },
// )

blocTest<CounterCubit, int>(
  'emits [1] when increment is called once',
  build: () => CounterCubit(),
  act: (cubit) => cubit.increment(),
  expect: () => [1],
);

blocTest<CounterCubit, int>(
  'emits [1, 2, 3] when increment called three times',
  build: () => CounterCubit(),
  act: (cubit) {
    cubit.increment();
    cubit.increment();
    cubit.increment();
  },
  expect: () => [1, 2, 3],
);

// Testing async BLoC with mocked dependencies
blocTest<UserBloc, UserState>(
  'emits [Loading, Loaded] when LoadUser event succeeds',
  build: () {
    when(() => mockRepo.fetchUser('1'))
        .thenAnswer((_) async => alice);
    return UserBloc(repository: mockRepo);
  },
  act: (bloc) => bloc.add(LoadUserEvent('1')),
  expect: () => [LoadingState(), LoadedState(alice)],
  verify: (_) {
    verify(() => mockRepo.fetchUser('1')).called(1);
  },
);

// Testing error states
blocTest<UserBloc, UserState>(
  'emits [Loading, Error] when LoadUser event fails',
  build: () {
    when(() => mockRepo.fetchUser(any()))
        .thenThrow(NetworkException('timeout'));
    return UserBloc(repository: mockRepo);
  },
  act: (bloc) => bloc.add(LoadUserEvent('1')),
  expect: () => [
    LoadingState(),
    ErrorState('timeout'),
  ],
);

// seed: set initial state before act
blocTest<UserBloc, UserState>(
  'refresh from loaded state',
  build: () => UserBloc(repository: mockRepo),
  seed: () => LoadedState(staleUser),  // pre-seeded state
  act: (bloc) => bloc.add(RefreshEvent()),
  expect: () => [LoadingState(), LoadedState(freshUser)],
);

9. Testing Riverpod Providers

Riverpod's ProviderContainer is the core testing primitive. It allows creating a fully isolated provider graph, overriding any provider, and reading state synchronously in tests.

import 'package:flutter_riverpod/flutter_riverpod.dart';
import 'package:test/test.dart';

// Production providers
final userRepositoryProvider = Provider<UserRepository>((ref) {
  return HttpUserRepository(ref.read(httpClientProvider));
});

final userProvider = FutureProvider.family<User, String>((ref, id) {
  return ref.read(userRepositoryProvider).fetchUser(id);
});

final counterProvider = StateNotifierProvider<CounterNotifier, int>(
  (ref) => CounterNotifier(),
);

// Tests
test('counter increments', () {
  // ProviderContainer is isolated — no shared state between tests
  final container = ProviderContainer();
  addTearDown(container.dispose);  // critical: prevents memory leaks

  expect(container.read(counterProvider), 0);
  container.read(counterProvider.notifier).increment();
  expect(container.read(counterProvider), 1);
});

test('userProvider fetches user with mocked repository', () async {
  final mockRepo = MockUserRepository();
  when(() => mockRepo.fetchUser('1'))
      .thenAnswer((_) async => alice);

  final container = ProviderContainer(
    overrides: [
      // Override the repository with the mock
      userRepositoryProvider.overrideWithValue(mockRepo),
    ],
  );
  addTearDown(container.dispose);

  final result = await container.read(userProvider('1').future);
  expect(result, equals(alice));
});

// Testing AsyncNotifier
test('UserListNotifier loads users', () async {
  final container = ProviderContainer(
    overrides: [
      userRepositoryProvider.overrideWithValue(FakeUserRepository()),
    ],
  );
  addTearDown(container.dispose);

  // Initially loading
  expect(
    container.read(userListProvider),
    isA<AsyncLoading>(),
  );

  // Await the future
  await container.read(userListProvider.future);

  // Now loaded
  expect(
    container.read(userListProvider),
    isA<AsyncData<List<User>>>(),
  );
});

// Listen to provider changes in test
test('provider emits changes', () {
  final container = ProviderContainer();
  addTearDown(container.dispose);

  final states = <int>[];
  container.listen(
    counterProvider,
    (prev, next) => states.add(next),
    fireImmediately: true,
  );

  container.read(counterProvider.notifier).increment();
  container.read(counterProvider.notifier).increment();

  expect(states, [0, 1, 2]);  // includes fireImmediately initial value
});

10. Architect Considerations

Test Strategy at Google L6

At L6, you're expected to define testing standards for a team, not just write tests. Key considerations:

  • Test Pyramid: Heavy unit test base, moderate widget tests, minimal integration tests. Unit tests provide the fastest feedback loop.
  • Isolation: Every test should be hermetic. Use setUp to recreate state. Shared state in setUpAll causes flaky tests — the root cause of most CI reliability issues.
  • Fakes over Mocks: At scale, maintaining mock stubs across API changes is expensive. Maintain a shared fake library that implements real behavior — it's more resilient to interface changes and tests true contracts.
  • Coverage vs confidence: 100% line coverage with poor assertions is worthless. Assert on observable behavior and invariants, not implementation details.
  • FakeAsync for timers: Never use Future.delayed in tests for actual waiting. FakeAsync makes time-dependent tests deterministic and instantaneous.
Test Double Purpose Use When
Dummy Passed but never used Satisfying required parameters
Stub Returns hardcoded values Controlling indirect inputs
Spy Records calls, delegates to real Rarely — often indicates design smell
Mock Records calls, verifies interactions Verifying side effects were triggered
Fake Simplified real implementation Complex state-based testing
