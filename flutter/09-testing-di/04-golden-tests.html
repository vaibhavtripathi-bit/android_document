What Are Golden Tests?

A golden test (also called a snapshot test or pixel test) renders a widget into an in-memory image and compares every pixel against a stored reference image — the "golden file". If the rendered output differs from the golden, the test fails.

This catches visual regressions that behavioral tests miss: a wrong color, a text overflow, a misaligned component, a missing padding. Golden tests are the only automated way to verify that a design system component still looks correct after a dependency upgrade or a layout refactor.

Basic Golden Test

import 'package:flutter/material.dart';
import 'package:flutter_test/flutter_test.dart';

void main() {
  testWidgets('PrimaryButton matches golden', (tester) async {
    await tester.pumpWidget(
      MaterialApp(
        home: Scaffold(
          body: Center(
            child: PrimaryButton(
              label: 'Submit',
              onPressed: () {},
            ),
          ),
        ),
      ),
    );

    // Compare rendered output against the stored golden file
    await expectLater(
      find.byType(PrimaryButton),
      matchesGoldenFile('goldens/primary_button.png'),
    );
  });
}

Generating and Updating Goldens

# First run — golden file does not exist yet. Generate it:
flutter test --update-goldens

# Subsequent runs — compare against the stored file:
flutter test

# Update a single test file's goldens:
flutter test test/widgets/primary_button_test.dart --update-goldens

The golden PNG files are committed to version control alongside the test code. They serve as the visual specification — a human can review them in code review to catch unintended visual changes.

Why Goldens Fail Across Platforms

This is the most common pain point with golden tests. A golden generated on a macOS developer machine will almost certainly fail when run on a Linux CI runner, even with identical code. The reasons:

  • Font rendering: macOS uses CoreText, Linux uses FreeType. Sub-pixel hinting and antialiasing differ, producing different pixel values for the same glyph.
  • Font availability: The system Roboto or SF Pro on macOS differs from the version bundled in the Flutter engine on Linux.
  • Antialiasing: Curve smoothing and gamma correction differ per platform.
  • Flutter version: Even a minor Flutter version upgrade can change rendering output by a few pixels.
Warning

Never generate goldens on macOS and run comparisons on Linux (or vice versa) without a tolerance threshold or a platform-agnostic strategy. The tests will fail even when the widget is pixel-perfect.

Solution 1: Load Fonts Explicitly

Load fonts from the assets bundle before running golden tests. This ensures both the generation machine and the CI machine use exactly the same font files.

import 'dart:io';
import 'package:flutter/services.dart';
import 'package:flutter_test/flutter_test.dart';

Future<void> loadAppFonts() async {
  // Load fonts from the assets declared in pubspec.yaml
  final fontLoader = FontLoader('Roboto')
    ..addFont(rootBundle.load('assets/fonts/Roboto-Regular.ttf'))
    ..addFont(rootBundle.load('assets/fonts/Roboto-Bold.ttf'));
  await fontLoader.load();
}

void main() {
  setUpAll(() async {
    await loadAppFonts();
  });

  testWidgets('Button with custom font matches golden', (tester) async {
    await tester.pumpWidget(
      const MaterialApp(home: Scaffold(body: MyButton())),
    );
    await expectLater(
      find.byType(MyButton),
      matchesGoldenFile('goldens/my_button.png'),
    );
  });
}

Solution 2: Custom Comparator with Tolerance

Replace the default pixel-exact comparator with one that allows a small percentage of pixels to differ. This absorbs sub-pixel rendering differences across platforms.

import 'package:flutter_test/flutter_test.dart';

class ToleranceGoldenFileComparator extends LocalFileComparator {
  final double tolerance;

  ToleranceGoldenFileComparator(super.testFile, {this.tolerance = 0.005});

  @override
  Future<bool> compare(Uint8List imageBytes, Uri golden) async {
    final result = await GoldenFileComparator.compareLists(
      imageBytes,
      await getGoldenBytes(golden),
    );

    if (!result.passed && result.diffPercent <= tolerance) {
      // Within tolerance — treat as passing
      print('Golden diff ${result.diffPercent * 100:.2f}% is within tolerance');
      return true;
    }
    return result.passed;
  }
}

// In flutter_test_config.dart (applies to all tests in the directory):
Future<void> testExecutable(FutureOr<void> Function() testMain) async {
  final testUrl = goldenFileComparator is LocalFileComparator
      ? (goldenFileComparator as LocalFileComparator).basedir
      : Uri.base;

  goldenFileComparator = ToleranceGoldenFileComparator(
    testUrl.resolve('test/'),
    tolerance: 0.005, // 0.5% pixel difference allowed
  );

  await testMain();
}

The alchemist Package

alchemist is a golden testing library built for design systems. Its key features:

  • Multi-scenario goldens: render multiple variants (light/dark, sizes, states) into a single golden image.
  • Platform-agnostic mode: by default, goldens are generated with a platform-neutral font so they match across macOS and Linux.
  • CI vs local modes: alchemist can be configured to use different comparators for CI vs local development.
# pubspec.yaml
dev_dependencies:
  alchemist: ^0.7.0

import 'package:alchemist/alchemist.dart';
import 'package:flutter/material.dart';
import 'package:flutter_test/flutter_test.dart';

void main() {
  goldenTest(
    'PrimaryButton renders correctly in all variants',
    fileName: 'primary_button',
    builder: () => GoldenTestGroup(
      children: [
        GoldenTestScenario(
          name: 'default',
          child: PrimaryButton(label: 'Submit', onPressed: () {}),
        ),
        GoldenTestScenario(
          name: 'disabled',
          child: const PrimaryButton(label: 'Submit', onPressed: null),
        ),
        GoldenTestScenario(
          name: 'loading',
          child: const PrimaryButton(label: 'Submit', isLoading: true, onPressed: null),
        ),
        GoldenTestScenario(
          name: 'dark mode',
          child: Theme(
            data: ThemeData.dark(),
            child: PrimaryButton(label: 'Submit', onPressed: () {}),
          ),
        ),
      ],
    ),
  );
}

This produces a single PNG with all four scenarios laid out side by side — making it easy to review all variants in one code review comment.

alchemist CI Configuration

// alchemist_config.dart — place in test/ directory
import 'package:alchemist/alchemist.dart';

AlchemistConfig get testConfig => const AlchemistConfig(
  // Use platform-agnostic rendering for CI
  platformGoldensConfig: PlatformGoldensConfig(enabled: false),
  // Use the tolerance comparator for CI
  ciGoldensConfig: CiGoldensConfig(
    enabled: true,
    comparator: ToleranceGoldenFileComparator(tolerance: 0.003),
  ),
);

The golden_toolkit Package

golden_toolkit provides utilities for loading fonts and rendering widgets in device frames for golden tests.

# pubspec.yaml
dev_dependencies:
  golden_toolkit: ^0.15.0

import 'package:flutter_test/flutter_test.dart';
import 'package:golden_toolkit/golden_toolkit.dart';

void main() {
  // Load fonts once for all golden tests in this file
  setUpAll(() async {
    await loadAppFonts(); // golden_toolkit helper
  });

  testGoldens('Card widget in multiple device frames', (tester) async {
    final builder = DeviceBuilder()
      ..overrideDevicesForAllScenarios(devices: [
        Device.phone,
        Device.iphone11,
        Device.tabletLandscape,
      ])
      ..addScenario(
        widget: const MyCard(title: 'Hello', subtitle: 'World'),
        name: 'default state',
      )
      ..addScenario(
        widget: const MyCard(title: 'Hello', subtitle: 'World', isHighlighted: true),
        name: 'highlighted state',
      );

    await tester.pumpDeviceBuilder(builder);

    await screenMatchesGolden(tester, 'my_card_device_frames');
  });
}

CI Strategy for Golden Tests

The most reliable CI strategy is:

  1. Pin a specific Linux runner and Flutter version for golden generation and comparison. Never run goldens on macOS in CI.
  2. Commit goldens to git (or to an artifact store with versioned keys). Store them in a test/goldens/ subdirectory.
  3. Treat golden failures as blocking — a failing golden means a visual regression, which requires intentional approval.
  4. Update goldens intentionally: when a design changes, run --update-goldens on the pinned Linux runner, review the diff, then commit.
  5. Upload failure diffs to CI artifacts: when a golden fails, save the diff image as a CI artifact so reviewers can see exactly what changed.
# GitHub Actions: pinned golden test job
golden-tests:
  runs-on: ubuntu-22.04  # PINNED — never use ubuntu-latest for goldens
  steps:
    - uses: actions/checkout@v4
    - uses: subosito/flutter-action@v2
      with:
        flutter-version: '3.22.2'  # PINNED version

    - name: Run golden tests
      run: flutter test test/ --tags golden

    - name: Upload failure diffs
      if: failure()
      uses: actions/upload-artifact@v4
      with:
        name: golden-failures
        path: test/failures/  # golden_toolkit writes diffs here

Staff-level insight

At scale, golden tests pay dividends for design systems with 50+ components shared across multiple teams. A single Figma spec change can ripple through dozens of components; goldens catch the regressions that behavioral tests miss. The investment in a solid CI golden infrastructure (pinned runner, font loading, tolerance comparator, diff artifacts) is justified when the design system is the team's shared contract.

Tagging Golden Tests

Because golden tests are slower and more sensitive than regular widget tests, it is good practice to tag them and run them separately.

// In the test file
@Tags(['golden'])
library;

void main() {
  testWidgets('Button golden', (tester) async { ... });
}

# Run only golden tests
flutter test --tags golden

# Run all tests except goldens (fast feedback loop)
flutter test --exclude-tags golden
