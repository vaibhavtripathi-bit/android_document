1. Background Execution Model

iOS is fundamentally different from desktop operating systems in how it handles background processes. The core philosophy: apps don't run in the background unless they have a specific, system-sanctioned reason. This is driven by battery life, thermal constraints, and memory pressure on mobile devices.

┌────────────────────────────────────────────────────────────┐ │ iOS BACKGROUND EXECUTION TIMELINE │ ├────────────────────────────────────────────────────────────┤ │ │ │ User leaves app │ │ ───────────────────────────────────────────────────────── │ │ │ │ │ │ sceneWillResignActive │ │ │ sceneDidEnterBackground │ │ │ │ │ │ ┌──────────────────────────┐ │ │ │ │ ~5 seconds of execution │ Default background time │ │ │ └──────────────────────────┘ │ │ │ │ │ │ ┌──────────────────────────────────────────┐ │ │ │ │ beginBackgroundTask: up to ~30 seconds │ Extended │ │ │ │ (was 180s pre-iOS 13, now ~30s) │ │ │ │ └──────────────────────────────────────────┘ │ │ │ │ │ │ ┌──────────────────────────────────────────┐ │ │ │ │ Background modes: varies by mode │ Ongoing │ │ │ │ Audio: unlimited while playing │ │ │ │ │ Location: unlimited with active session │ │ │ │ │ VoIP: push-based wakeup │ │ │ │ └──────────────────────────────────────────┘ │ │ │ │ │ ▼ SUSPENDED (in memory, not executing) │ │ │ │ │ ▼ JETSAM (killed, no callback) │ │ │ └────────────────────────────────────────────────────────────┘

1.1 The Background Execution Budget

When your app moves to the background, the system gives you a limited window to finish work:

Mechanism Time Budget iOS Version Notes
Default (no API call) ~5 seconds All After sceneDidEnterBackground
beginBackgroundTask ~30 seconds iOS 13+ Was ~180s pre-iOS 13
Background URL session System-managed iOS 7+ Transfers continue after suspension
BGAppRefreshTask ~30 seconds iOS 13+ System-scheduled, not guaranteed
BGProcessingTask Minutes iOS 13+ Requires charging + Wi-Fi (optional)
Common Pitfall: Background Time Reduction

In iOS 13, Apple silently reduced beginBackgroundTask time from ~180 seconds to ~30 seconds. Many apps that relied on the longer window broke silently. The system doesn't guarantee any specific duration — UIApplication.shared.backgroundTimeRemaining is an estimate, not a contract. Design your background work to complete as quickly as possible.

2. beginBackgroundTask — The Critical API

2.1 Correct Usage

class DataSyncManager {
    private var backgroundTaskID: UIBackgroundTaskIdentifier = .invalid
    
    func syncBeforeBackground() {
        // Start background task BEFORE the work begins
        backgroundTaskID = UIApplication.shared.beginBackgroundTask(
            withName: "DataSync"
        ) { [weak self] in
            // Expiration handler: MUST call endBackgroundTask
            // System is about to suspend — clean up NOW
            self?.cancelSync()
            self?.endTask()
        }
        
        // Guard against failure to begin
        guard backgroundTaskID != .invalid else { return }
        
        // Perform the actual work
        performSync { [weak self] in
            self?.endTask()
        }
    }
    
    private func endTask() {
        guard backgroundTaskID != .invalid else { return }
        UIApplication.shared.endBackgroundTask(backgroundTaskID)
        backgroundTaskID = .invalid
    }
}

2.2 The Critical Mistake

Critical Bug: Not Calling endBackgroundTask

If you call beginBackgroundTask but never call endBackgroundTask, the system will:

  1. Call your expiration handler when time runs out
  2. If you still don't call endBackgroundTask in the expiration handler, the system kills your app immediately
  3. This appears as a crash in your crash logs (SIGKILL with 0x8badf00d — "ate bad food")

This is one of the most common causes of mysterious crashes in production iOS apps. Always pair begin with end, and always call end in the expiration handler.

// THE BUG: forgetting to end the task
func buggySync() {
    let taskID = UIApplication.shared.beginBackgroundTask {
        // ❌ Expiration handler does nothing
        // System will KILL the app
    }
    
    performSync {
        // ❌ If this closure never fires (error, timeout),
        // endBackgroundTask is never called
        UIApplication.shared.endBackgroundTask(taskID)
    }
}

// THE FIX: defensive pattern
func safeSync() {
    var taskID: UIBackgroundTaskIdentifier = .invalid
    
    taskID = UIApplication.shared.beginBackgroundTask {
        // ✅ Always end in expiration handler
        UIApplication.shared.endBackgroundTask(taskID)
        taskID = .invalid
    }
    
    guard taskID != .invalid else { return }
    
    performSync { result in
        // ✅ End on completion (success or failure)
        defer {
            UIApplication.shared.endBackgroundTask(taskID)
            taskID = .invalid
        }
        // Process result...
    }
}

2.3 Multiple Background Tasks

You can have multiple active background tasks simultaneously. Each gets its own identifier and must be ended independently. The system keeps the app running until all tasks are ended or expired:

class BackgroundTaskManager {
    private var activeTasks: [String: UIBackgroundTaskIdentifier] = [:]
    private let lock = NSLock()
    
    func begin(name: String, work: @escaping (@escaping () -> Void) -> Void) {
        var taskID: UIBackgroundTaskIdentifier = .invalid
        
        taskID = UIApplication.shared.beginBackgroundTask(withName: name) { [weak self] in
            self?.end(name: name)
        }
        
        guard taskID != .invalid else { return }
        
        lock.lock()
        activeTasks[name] = taskID
        lock.unlock()
        
        work { [weak self] in
            self?.end(name: name)
        }
    }
    
    private func end(name: String) {
        lock.lock()
        guard let taskID = activeTasks.removeValue(forKey: name) else {
            lock.unlock()
            return
        }
        lock.unlock()
        UIApplication.shared.endBackgroundTask(taskID)
    }
}

3. Background Modes

Background modes are declared in Info.plist under UIBackgroundModes and grant specific background execution privileges:

Mode Key Execution Model Review Scrutiny
Audio audio Continuous while playing/recording Must actually play audio
Location location Continuous with active CLLocationManager Must justify always-on location
VoIP voip Push-based wakeup via PushKit Must use CallKit; heavily scrutinized
Background Fetch fetch System-scheduled, ~30s per wake Deprecated in iOS 13 (use BGTaskScheduler)
Remote Notification remote-notification Wakes app on silent push, ~30s Must process notification content
Background Processing processing BGProcessingTask, minutes of execution iOS 13+; requires BGTaskScheduler
Bluetooth bluetooth-central / bluetooth-peripheral BLE scanning/advertising continues Must actively use BLE
External Accessory external-accessory Communication with MFi accessories Must have MFi accessory
Deep Dive: Silent Push Notifications

Silent pushes (content-available: 1) wake your app in the background for ~30 seconds. However, the system applies rate limiting — if your app uses too much energy or CPU during these wakeups, the system will throttle or stop delivering them. The system also won't wake your app if it was force-quit by the user. This is a common source of "my background sync stopped working" bugs.

4. BGTaskScheduler

Introduced in iOS 13, BGTaskScheduler replaced the old performFetchWithCompletionHandler and setMinimumBackgroundFetchInterval APIs with a more powerful and predictable system.

4.1 Task Types

Type Duration Constraints Use Case
BGAppRefreshTask ~30 seconds Network optional Fetch new content, sync small data
BGProcessingTask Several minutes Can require charging, network ML training, database maintenance, large sync

4.2 Registration and Scheduling

// Step 1: Register task identifiers in Info.plist
// BGTaskSchedulerPermittedIdentifiers: ["com.app.refresh", "com.app.dbCleanup"]

// Step 2: Register handlers at launch (before end of didFinishLaunching)
func application(_ app: UIApplication,
    didFinishLaunchingWithOptions opts: [UIApplication.LaunchOptionsKey: Any]?) -> Bool {
    
    BGTaskScheduler.shared.register(
        forTaskWithIdentifier: "com.app.refresh",
        using: nil  // nil = main queue
    ) { task in
        self.handleAppRefresh(task: task as! BGAppRefreshTask)
    }
    
    BGTaskScheduler.shared.register(
        forTaskWithIdentifier: "com.app.dbCleanup",
        using: nil
    ) { task in
        self.handleDatabaseCleanup(task: task as! BGProcessingTask)
    }
    
    return true
}

// Step 3: Schedule tasks (typically when entering background)
func scheduleAppRefresh() {
    let request = BGAppRefreshTaskRequest(identifier: "com.app.refresh")
    request.earliestBeginDate = Date(timeIntervalSinceNow: 15 * 60) // 15 min minimum
    
    do {
        try BGTaskScheduler.shared.submit(request)
    } catch {
        print("Failed to schedule: \(error)")
    }
}

func scheduleDatabaseCleanup() {
    let request = BGProcessingTaskRequest(identifier: "com.app.dbCleanup")
    request.requiresNetworkConnectivity = false
    request.requiresExternalPower = true  // Only when charging
    request.earliestBeginDate = Date(timeIntervalSinceNow: 60 * 60) // 1 hour
    
    do {
        try BGTaskScheduler.shared.submit(request)
    } catch {
        print("Failed to schedule: \(error)")
    }
}

// Step 4: Handle the task
func handleAppRefresh(task: BGAppRefreshTask) {
    // Schedule the next refresh
    scheduleAppRefresh()
    
    let operation = RefreshOperation()
    
    // Handle expiration
    task.expirationHandler = {
        operation.cancel()
    }
    
    operation.completionBlock = {
        task.setTaskCompleted(success: !operation.isCancelled)
    }
    
    OperationQueue().addOperation(operation)
}

// Simulate task launch from Xcode debugger:

// Force early execution (debug only):
// (lldb) e -l objc -- (void)[[BGTaskScheduler sharedScheduler] _simulateExpirationForTaskWithIdentifier:@"com.app.refresh"]
Best Practice: BGTaskScheduler Patterns
  • Always re-schedule the next task inside the handler (tasks are one-shot)
  • Always set an expirationHandler that cancels in-flight work
  • Always call setTaskCompleted(success:) — failure to do so is treated like the endBackgroundTask bug
  • Use BGProcessingTask with requiresExternalPower = true for heavy work — the system is much more generous with time when the device is charging
  • Register handlers in didFinishLaunching, not later — the system may launch your app directly into a background task

5. Jetsam — The iOS OOM Killer

Jetsam is iOS's memory management daemon, analogous to Linux's OOM killer but more aggressive and with different heuristics. Understanding Jetsam is critical for Staff-level iOS engineers.

5.1 How Jetsam Works

┌──────────────────────────────────────────────────────────┐ │ JETSAM PRIORITY BANDS │ ├──────────────────────────────────────────────────────────┤ │ │ │ Band 0-10: Idle / Background (KILLED FIRST) │ │ ├── Suspended apps (no execution) │ │ ├── Background apps (limited execution) │ │ └── Killed in LRU order (least recently used first) │ │ │ │ Band 10-20: Low Priority │ │ ├── Apps performing background tasks │ │ └── Extensions │ │ │ │ Band 20-30: Default │ │ ├── Foreground inactive apps │ │ └── Apps transitioning states │ │ │ │ Band 30-40: High Priority │ │ ├── Foreground active app │ │ └── System UI (SpringBoard) │ │ │ │ Band 40+: Critical (KILLED LAST) │ │ ├── Kernel tasks │ │ └── Core system daemons │ │ │ │ Memory Limit (foreground): ~1-2GB (device dependent) │ │ Memory Limit (background): ~50-80MB │ │ Memory Limit (extensions): ~40-120MB │ │ │ └──────────────────────────────────────────────────────────┘

5.2 Jetsam Kills vs Crashes

Aspect Jetsam Kill App Crash
Signal SIGKILL (unblockable) Various (SIGSEGV, SIGABRT, etc.)
Callback None Signal handler possible
didReceiveMemoryWarning May or may not fire first N/A
applicationWillTerminate Not called Not called
Crash report Jetsam event report (different format) Standard crash report
Prevention Reduce memory footprint Fix the bug

5.3 Memory Monitoring

import os

func checkMemoryPressure() {
    // Available memory (iOS 13+)
    let available = os_proc_available_memory()
    print("Available memory: \(available / 1024 / 1024) MB")
    
    // Current footprint
    var info = mach_task_basic_info()
    var count = mach_msg_type_number_t(
        MemoryLayout<mach_task_basic_info>.size / MemoryLayout<natural_t>.size
    )
    let result = withUnsafeMutablePointer(to: &info) {
        $0.withMemoryRebound(to: integer_t.self, capacity: Int(count)) {
            task_info(mach_task_self_, task_flavor_t(MACH_TASK_BASIC_INFO), $0, &count)
        }
    }
    if result == KERN_SUCCESS {
        print("Resident memory: \(info.resident_size / 1024 / 1024) MB")
    }
}

// Subscribe to memory pressure notifications
let source = DispatchSource.makeMemoryPressureSource(
    eventMask: [.warning, .critical],
    queue: .main
)
source.setEventHandler {
    let event = source.data
    if event.contains(.warning) {
        // Purge caches, release non-essential resources
        ImageCache.shared.purge()
    }
    if event.contains(.critical) {
        // Aggressive cleanup — Jetsam is imminent
        ImageCache.shared.purgeAll()
        DataCache.shared.purgeAll()
    }
}
source.resume()
Common Pitfall: didReceiveMemoryWarning is Unreliable

didReceiveMemoryWarning is not guaranteed to fire before Jetsam kills your app. On devices under extreme memory pressure, Jetsam may kill your app without any warning. Additionally, this callback runs on the main thread — if your main thread is blocked (e.g., during a large layout pass), the warning can't be delivered. Never rely on this as your only defense against OOM kills.

6.1 applicationWillTerminate

func applicationWillTerminate(_ application: UIApplication) {
    // Called ONLY when:
    // 1. App is in foreground and system needs to terminate
    // 2. App has opted out of scene lifecycle (legacy apps)
    //
    // NOT called when:
    // 1. Jetsam kills a suspended app (most common case)
    // 2. User force-quits from app switcher
    // 3. Device reboots
    //
    // You have ~5 seconds to save state.
    // For scene-based apps, save state in sceneDidEnterBackground instead.
}

7. Architect Considerations

Architect Must-Know: Designing Reliable Background Work Systems

At Staff level, you're expected to design systems that are resilient to process death. The fundamental principle: assume your app can be killed at any time with no callback. Every operation must be either idempotent and resumable, or transactional with rollback.

7.1 Reliable Background Sync Architecture

// Design principle: every sync operation is a state machine
// persisted to disk, resumable after process death

enum SyncState: Codable {
    case pending
    case uploading(progress: Double)
    case confirming(serverID: String)
    case completed
    case failed(retryCount: Int)
}

struct SyncOperation: Codable {
    let id: UUID
    let entityID: String
    let entityType: String
    var state: SyncState
    let createdAt: Date
    var lastAttempt: Date?
}

final class SyncEngine {
    private let store: SyncOperationStore  // Persisted to disk
    
    // On app launch: resume any incomplete operations
    func resumePendingOperations() {
        let pending = store.operations(in: [.pending, .uploading, .confirming])
        for op in pending {
            execute(op)
        }
    }
    
    // Use background URL session for large transfers
    // These survive process death
    private lazy var backgroundSession: URLSession = {
        let config = URLSessionConfiguration.background(
            withIdentifier: "com.app.sync"
        )
        config.isDiscretionary = false
        config.sessionSendsLaunchEvents = true
        return URLSession(configuration: config, delegate: self, delegateQueue: nil)
    }()
}

7.2 Handling Jetsam Gracefully

// Strategy: save state continuously, not just in callbacks

final class AutoSaveManager {
    private var saveTimer: Timer?
    private let interval: TimeInterval = 5.0
    
    func startAutoSaving() {
        saveTimer = Timer.scheduledTimer(
            withTimeInterval: interval,
            repeats: true
        ) { [weak self] _ in
            self?.saveState()
        }
    }
    
    private func saveState() {
        // Write to disk atomically
        let state = captureCurrentState()
        let data = try? JSONEncoder().encode(state)
        let url = stateFileURL()
        try? data?.write(to: url, options: .atomic)
    }
}
Best Practice: Background Work Decision Tree

Choose the right mechanism based on your needs:

  • Finish in-flight request → beginBackgroundTask (~30s)
  • Large file upload/download → Background URLSession (survives process death)
  • Periodic content refresh → BGAppRefreshTask (~30s, system-scheduled)
  • Heavy processing (ML, DB) → BGProcessingTask (minutes, while charging)
  • Real-time updates → Silent push notifications (rate-limited)
  • Continuous audio/location → Background modes (must justify to App Review)
