1. Scene Architecture Overview

iOS 13 introduced the scene architecture to support multiple windows of the same app running simultaneously on iPad. This was a fundamental shift from the single-window model that had existed since iPhone OS 2.0.

┌──────────────────────────────────────────────────────┐ │ UIApplication │ │ (process-level) │ │ │ │ ┌─────────────────┐ ┌─────────────────┐ │ │ │ UISceneSession │ │ UISceneSession │ │ │ │ (persistent) │ │ (persistent) │ │ │ │ │ │ │ │ │ │ ┌────────────┐ │ │ ┌────────────┐ │ │ │ │ │ UIScene │ │ │ │ UIScene │ │ │ │ │ │ (runtime) │ │ │ │ (runtime) │ │ │ │ │ │ │ │ │ │ │ │ │ │ │ │ UIWindow │ │ │ │ UIWindow │ │ │ │ │ │ UIWindow │ │ │ │ UIWindow │ │ │ │ │ └────────────┘ │ │ └────────────┘ │ │ │ └─────────────────┘ └─────────────────┘ │ │ │ │ AppDelegate: process-level events │ │ SceneDelegate: per-scene lifecycle │ └──────────────────────────────────────────────────────┘

1.1 Key Concepts

Concept Lifetime Purpose
UIApplication Process lifetime Singleton; manages the app process, delivers events
UISceneSession Persists across launches Represents a "slot" for a scene; survives process death
UIScene Runtime only Active instance of a scene; created/destroyed by system
UIWindowScene Runtime only Concrete subclass of UIScene that hosts UIWindows
UIWindow Runtime, owned by scene Root of a view hierarchy; belongs to a UIWindowScene
Deep Dive: Session Persistence

UISceneSession is the critical concept most developers miss. A session persists across process termination. When the user swipes up in the app switcher, the scene is disconnected but the session survives. When the user taps the app again, the system reconnects a new UIScene instance to the existing session. This is how state restoration works — the session carries userInfo and stateRestorationActivity across launches.

2. UIScene, UISceneSession, UIWindowScene

2.1 UIScene

UIScene is an abstract base class. You never create instances directly. The system creates them when connecting a scene session. Its key properties:

class UIScene: UIResponder {
    var session: UISceneSession { get }
    var delegate: UISceneDelegate? { get set }
    var activationState: UIScene.ActivationState { get }
    var title: String? { get set }
    var activationConditions: UISceneActivationConditions { get set }
    
    // Request the system to open a new session
    func open(
        externalURLs: [URL],
        options: UIScene.OpenExternalURLOptions?,
        completionHandler: ((Bool) -> Void)?
    )
}

2.2 UISceneSession

The session is the persistent identity of a scene. It survives process death and is how the system tracks which "windows" the user has open:

class UISceneSession: NSObject {
    var persistentIdentifier: String { get }  // Stable across launches
    var role: UISceneSession.Role { get }       // .windowApplication, .windowExternalDisplay, etc.
    var configuration: UISceneConfiguration { get }
    var scene: UIScene? { get }                // nil if disconnected
    var stateRestorationActivity: NSUserActivity? { get set }
    var userInfo: [String: Any]? { get set }  // Persisted across launches
}

2.3 UIWindowScene

UIWindowScene is the concrete subclass you work with for standard app windows:

class UIWindowScene: UIScene {
    var windows: [UIWindow] { get }
    var screen: UIScreen { get }
    var interfaceOrientation: UIInterfaceOrientation { get }
    var coordinateSpace: UICoordinateSpace { get }
    var traitCollection: UITraitCollection { get }
    var sizeRestrictions: UISceneSizeRestrictions? { get }
    var statusBarManager: UIStatusBarManager? { get }
    
    // iOS 15+: request geometry changes
    func requestGeometryUpdate(
        _ geometryPreferences: UIWindowScene.GeometryPreferences,
        errorHandler: ((Error) -> Void)?
    )
}
Common Pitfall: UIScreen.main Deprecation

UIScreen.main is deprecated in iOS 16. With multi-window, there's no single "main" screen. Use windowScene.screen instead. Similarly, UIApplication.shared.keyWindow is deprecated — use windowScene.keyWindow or iterate UIApplication.shared.connectedScenes.

3. Scene Activation States

┌─────────────┐ │ Unattached │ │ (no scene) │ └──────┬──────┘ │ system connects scene ▼ ┌─────────────────────┐ │ Foreground Inactive │ ◀──── app switcher visible │ │ ◀──── system alert shown └──────┬──────────────┘ ◀──── incoming call │ user interacts │ │ interruption ▼ │ ┌─────────────────────┐ │ Foreground Active │ ◀──── user is interacting └──────┬──────────────┘ │ user switches away ▼ ┌─────────────────────┐ │ Background │ ◀──── ~5s to complete work └──────┬──────────────┘ (or beginBackgroundTask) │ ▼ ┌─────────────────────┐ │ Suspended │ ◀──── in memory, not executing └──────┬──────────────┘ │ ┌────────────┼────────────┐ ▼ ▼ ▼ Jetsam User returns System disconnects kills it (foreground) (scene disconnected, session persists)

3.1 State Transitions and Delegate Callbacks

class SceneDelegate: UIResponder, UIWindowSceneDelegate {
    
    // Scene is being created or reconnected
    func scene(_ scene: UIScene,
        willConnectTo session: UISceneSession,
        options connectionOptions: UIScene.ConnectionOptions) {
        // Create UI, restore state from session
    }
    
    // About to enter foreground
    func sceneWillEnterForeground(_ scene: UIScene) {
        // Undo background changes, refresh data
    }
    
    // Scene is now interactive
    func sceneDidBecomeActive(_ scene: UIScene) {
        // Resume paused tasks, start timers
    }
    
    // About to lose focus (alert, app switcher, etc.)
    func sceneWillResignActive(_ scene: UIScene) {
        // Pause game, save draft, obscure sensitive data
    }
    
    // Scene moved to background
    func sceneDidEnterBackground(_ scene: UIScene) {
        // Save state, release shared resources
        // Snapshot will be taken for app switcher
    }
    
    // Scene is being released (NOT the session)
    func sceneDidDisconnect(_ scene: UIScene) {
        // Release scene-specific resources
        // Session still exists — scene may reconnect later
    }
}

3.2 State Transition Table

From To Callback Trigger
Unattached Foreground Inactive scene(_:willConnectTo:options:) App launch, new window request
Foreground Inactive Foreground Active sceneDidBecomeActive User interaction begins
Foreground Active Foreground Inactive sceneWillResignActive Alert, app switcher, notification center
Foreground Inactive Background sceneDidEnterBackground User switches to another app
Background Foreground Inactive sceneWillEnterForeground User returns to app
Background Suspended No callback System suspends (no code runs)
Suspended Unattached sceneDidDisconnect System reclaims resources
Common Pitfall: No Callback for Suspension

There is no delegate callback when a scene moves from Background to Suspended. The system simply stops executing your code. You must save all critical state in sceneDidEnterBackground. Similarly, Jetsam can kill a suspended app with no callback at all — not even applicationWillTerminate.

4. UISceneConfiguration

4.1 Info.plist Configuration (Static)

Scene configurations can be declared in Info.plist for static setup:

<!-- Info.plist scene manifest -->
<key>UIApplicationSceneManifest</key>
<dict>
    <key>UIApplicationSupportsMultipleScenes</key>
    <true/>
    <key>UISceneConfigurations</key>
    <dict>
        <key>UIWindowSceneSessionRoleApplication</key>
        <array>
            <dict>
                <key>UISceneConfigurationName</key>
                <string>Default Configuration</string>
                <key>UISceneDelegateClassName</key>
                <string>$(PRODUCT_MODULE_NAME).SceneDelegate</string>
                <key>UISceneStoryboardFile</key>
                <string>Main</string>
            </dict>
        </array>
    </dict>
</dict>

4.2 Runtime Configuration (Dynamic)

For more control, provide configuration at runtime via the AppDelegate:

func application(_ app: UIApplication,
    configurationForConnecting session: UISceneSession,
    options: UIScene.ConnectionOptions) -> UISceneConfiguration {
    
    // Choose configuration based on how the scene was requested
    if options.userActivities.contains(where: {
        $0.activityType == "com.app.compose"
    }) {
        let config = UISceneConfiguration(
            name: "Compose",
            sessionRole: session.role
        )
        config.delegateClass = ComposeSceneDelegate.self
        return config
    }
    
    // Default configuration
    let config = UISceneConfiguration(
        name: "Default",
        sessionRole: session.role
    )
    config.delegateClass = MainSceneDelegate.self
    return config
}

5. Connection vs Disconnection vs Destruction

This is one of the most misunderstood aspects of the scene lifecycle. There are three distinct events with very different semantics:

┌──────────────────────────────────────────────────────────────┐ │ SCENE LIFECYCLE EVENTS │ ├──────────────────────────────────────────────────────────────┤ │ │ │ CONNECTION (scene(_:willConnectTo:options:)) │ │ ├── Scene object created and attached to session │ │ ├── Happens on: app launch, new window, reconnection │ │ ├── Session may have prior state (stateRestorationActivity) │ │ └── You create UIWindow and set up UI here │ │ │ │ DISCONNECTION (sceneDidDisconnect(_:)) │ │ ├── Scene object destroyed, session SURVIVES │ │ ├── Happens when: system reclaims memory, background scene │ │ ├── NOT the same as user closing the window │ │ ├── Scene may reconnect later with same session │ │ └── Release scene-specific resources, keep session data │ │ │ │ DESTRUCTION (application(_:didDiscardSceneSessions:)) │ │ ├── Session itself is destroyed │ │ ├── Happens when: user swipes away in app switcher │ │ ├── Called on AppDelegate, NOT SceneDelegate │ │ ├── May be called on NEXT launch if app wasn't running │ │ └── Delete all data associated with this scene │ │ │ └──────────────────────────────────────────────────────────────┘ 
class SceneDelegate: UIResponder, UIWindowSceneDelegate {
    
    func sceneDidDisconnect(_ scene: UIScene) {
        // Scene object is being released, but session persists.
        // DO: release heavy resources (image caches, temp files)
        // DON'T: delete user data for this scene
        // The scene may reconnect to the same session later.
        
        let sessionID = scene.session.persistentIdentifier
        ImageCache.shared.purge(for: sessionID)
    }
}

// In AppDelegate:
func application(_ app: UIApplication,
    didDiscardSceneSessions sceneSessions: Set<UISceneSession>) {
    // Sessions are being permanently destroyed.
    // User swiped away in app switcher.
    // NOW delete all data for these scenes.
    
    for session in sceneSessions {
        let id = session.persistentIdentifier
        SceneDataStore.shared.deleteData(for: id)
        DraftManager.shared.deleteDraft(for: id)
    }
}
Architect Must-Know: Deferred Destruction

If the user swipes away a scene in the app switcher while your app is not running, the system queues the destruction. didDiscardSceneSessions is called on the next launch of your app. This means your app must handle cleanup for scenes it may not even remember creating. Always key scene data by persistentIdentifier and be prepared to clean up stale data.

6. Multi-Window on iPad

6.1 Enabling Multi-Window

Set UIApplicationSupportsMultipleScenes to true in Info.plist. Without this, the system will only ever create one scene session.

6.2 Creating New Windows Programmatically

// Request a new scene from code
func openNewWindow(with activity: NSUserActivity) {
    let options = UIScene.ActivationRequestOptions()
    options.requestingScene = self.view.window?.windowScene
    
    UIApplication.shared.requestSceneSessionActivation(
        nil,           // nil = create new session
        userActivity: activity,
        options: options,
        errorHandler: { error in
            print("Failed to open scene: \(error)")
        }
    )
}

// Close a scene programmatically
func closeWindow(scene: UIWindowScene) {
    UIApplication.shared.requestSceneSessionDestruction(
        scene.session,
        options: nil,
        errorHandler: { error in
            print("Failed to close scene: \(error)")
        }
    )
}

6.3 Scene Activation Conditions

You can control which scene handles incoming activities using activation conditions:

func scene(_ scene: UIScene,
    willConnectTo session: UISceneSession,
    options: UIScene.ConnectionOptions) {
    
    // This scene prefers to handle document activities
    scene.activationConditions.prefersToActivateForTargetContentIdentifierPredicate =
        NSPredicate(format: "self == 'document-editor'")
    
    // This scene CAN handle any activity (fallback)
    scene.activationConditions.canActivateForTargetContentIdentifierPredicate =
        NSPredicate(value: true)
}

6.4 Drag and Drop to Create Windows

On iPad, users can drag content to the edge of the screen to create a new window. Your app receives this as a new scene connection with the dragged NSUserActivity:

// Register an activity for drag-to-new-window
let activity = NSUserActivity(activityType: "com.app.viewDocument")
activity.title = "View Document"
activity.userInfo = ["documentID": document.id]
activity.targetContentIdentifier = "document-\(document.id)"

// Attach to a UIDragItem
let itemProvider = NSItemProvider(object: activity)
let dragItem = UIDragItem(itemProvider: itemProvider)

7. State Restoration with NSUserActivity

7.1 Saving State

func stateRestorationActivity(for scene: UIScene) -> NSUserActivity? {
    let activity = NSUserActivity(activityType: "com.app.mainScene")
    activity.userInfo = [
        "selectedTab": currentTab.rawValue,
        "scrollPosition": scrollView.contentOffset.y,
        "searchQuery": searchBar.text ?? ""
    ]
    return activity
}

7.2 Restoring State

func scene(_ scene: UIScene,
    willConnectTo session: UISceneSession,
    options: UIScene.ConnectionOptions) {
    
    // Check for restoration activity
    if let activity = session.stateRestorationActivity {
        restore(from: activity)
    }
    
    // Also check connectionOptions for incoming activities
    if let userActivity = options.userActivities.first {
        handle(userActivity: userActivity)
    }
}

private func restore(from activity: NSUserActivity) {
    guard activity.activityType == "com.app.mainScene" else { return }
    
    if let tabRaw = activity.userInfo?["selectedTab"] as? String,
       let tab = Tab(rawValue: tabRaw) {
        tabController.selectedTab = tab
    }
    
    if let offset = activity.userInfo?["scrollPosition"] as? CGFloat {
        scrollView.contentOffset = CGPoint(x: 0, y: offset)
    }
}
Best Practice: NSUserActivity for Restoration

Use NSUserActivity for scene state restoration because it serves double duty: the same activity can power Handoff, Spotlight indexing, and Siri Shortcuts. Design your activities to be meaningful beyond just restoration — this is a multiplier on engineering effort.

8. Architect Considerations

Architect Must-Know: Designing Data Layer for Multi-Scene Apps

Multi-scene apps require a fundamentally different approach to state management. The core challenge: multiple scenes share the same process and data, but have independent UI states.

8.1 Shared State Management Pattern

// Layer 1: Process-scoped shared state (singleton OK)
@MainActor
final class AppState: ObservableObject {
    static let shared = AppState()
    
    @Published var currentUser: User?
    @Published var unreadCount: Int = 0
    
    // Changes here propagate to ALL scenes
}

// Layer 2: Scene-scoped state (one per scene)
@MainActor
final class SceneContext: ObservableObject {
    let sessionID: String
    @Published var navigationPath: [Route] = []
    @Published var selectedDocument: Document?
    
    init(session: UISceneSession) {
        self.sessionID = session.persistentIdentifier
    }
}

// Layer 3: Coordination (handle cross-scene events)
@MainActor
final class SceneCoordinator {
    static let shared = SceneCoordinator()
    private var contexts: [String: SceneContext] = [:]
    
    func context(for session: UISceneSession) -> SceneContext {
        let id = session.persistentIdentifier
        if let existing = contexts[id] { return existing }
        let ctx = SceneContext(session: session)
        contexts[id] = ctx
        return ctx
    }
    
    func discard(sessions: Set<UISceneSession>) {
        for session in sessions {
            contexts.removeValue(forKey: session.persistentIdentifier)
        }
    }
    
    // Broadcast event to all scenes
    func broadcast(_ event: AppEvent) {
        for (_, context) in contexts {
            context.handle(event)
        }
    }
}

8.2 Core Data with Multiple Scenes

// Each scene should use its own viewContext or child context
// to avoid UI conflicts when editing the same data

final class PersistenceController {
    static let shared = PersistenceController()
    let container: NSPersistentContainer
    
    // Shared viewContext for reading
    var viewContext: NSManagedObjectContext {
        container.viewContext
    }
    
    // Per-scene editing context
    func newEditingContext() -> NSManagedObjectContext {
        let context = container.newBackgroundContext()
        context.mergePolicy = NSMergeByPropertyObjectTrumpMergePolicy
        context.automaticallyMergesChangesFromParent = true
        return context
    }
}

8.3 Notification Handling in Multi-Scene

// Problem: which scene should handle a push notification?
// Solution: use targetContentIdentifier to route to the right scene

func scene(_ scene: UIScene,
    willConnectTo session: UISceneSession,
    options: UIScene.ConnectionOptions) {
    
    // Set activation conditions so the system routes
    // notifications to the correct scene
    if let chatID = session.userInfo?["chatID"] as? String {
        scene.activationConditions
            .prefersToActivateForTargetContentIdentifierPredicate =
            NSPredicate(format: "self == %@", "chat-\(chatID)")
    }
}

// content.targetContentIdentifier = "chat-\(chatID)"
Deep Dive: Scene Lifecycle Notifications

In addition to delegate callbacks, the system posts notifications for scene state changes. These are useful when non-delegate objects need to react to lifecycle events:

  • UIScene.willConnectNotification
  • UIScene.didDisconnectNotification
  • UIScene.didActivateNotification
  • UIScene.willDeactivateNotification
  • UIScene.willEnterForegroundNotification
  • UIScene.didEnterBackgroundNotification

Use these with NotificationCenter and filter by the scene's session.persistentIdentifier to handle events for specific scenes.
