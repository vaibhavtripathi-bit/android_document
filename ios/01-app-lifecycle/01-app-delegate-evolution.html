1. The Pre-main Boot Sequence

Before your Swift code ever executes, the operating system performs a complex series of steps to prepare your process. Understanding this is critical for optimizing app launch time — a key metric at scale.

1.1 dyld4 — The Dynamic Linker

When the kernel creates a process for your app, it maps the Mach-O binary into memory and hands control to dyld (the dynamic linker). Since iOS 15 / macOS 12, Apple uses dyld4, which introduced significant performance improvements over dyld3.

Kernel: fork + exec │ ▼ ┌─────────────────────────────────────────────────┐ │ dyld4 │ │ │ │ 1. Load shared cache (pre-linked system libs) │ │ 2. Parse Mach-O load commands │ │ 3. Map dependent dylibs into memory │ │ 4. Perform rebasing (ASLR slide) │ │ 5. Perform binding (resolve external symbols) │ │ 6. Run initializers: │ │ a. C++ static constructors │ │ b. ObjC +load methods │ │ c. __attribute__((constructor)) functions │ │ 7. Jump to main() │ └─────────────────────────────────────────────────┘

dyld Shared Cache

Apple pre-links all system frameworks (UIKit, Foundation, CoreGraphics, etc.) into a single shared cache file. This is a massive optimization — your app doesn't need to individually load and link hundreds of system dylibs. The shared cache is mapped into every process at the same address, enabling efficient sharing of read-only pages across all running apps.

ASLR and Rebasing

Address Space Layout Randomization (ASLR) loads your binary at a random base address for security. Since the Mach-O was compiled with a preferred base address, all internal pointers must be adjusted by the ASLR slide. This is rebasing. dyld4 uses "closures" — pre-computed data about how to load your app — cached to disk after the first launch, making subsequent launches faster.

1.2 +load vs +initialize

Two Objective-C class methods run during startup, but at very different times and with very different implications:

Aspect +load +initialize
When called During dyld initialization, before main() Lazily, before first message to the class
Thread safety Called with the runtime lock held (serial) Thread-safe via dispatch_once-like mechanism
Inheritance Not inherited — each class's own +load runs Inherited — subclass may trigger super's
Category behavior Category +load runs separately after class +load Category +initialize replaces class implementation
Impact on launch Directly adds to pre-main time Deferred — no launch impact
Use case Method swizzling (only valid place) Lazy one-time setup
Common Pitfall

Every +load method adds directly to your pre-main time. At scale, third-party SDKs often sneak in +load methods. Audit with: otool -oV YourApp | grep -A1 "load". A Staff engineer should know that eliminating unnecessary +load calls is one of the highest-ROI launch optimizations.

1.3 Static Initializers

C++ static constructors and __attribute__((constructor)) functions also run before main(). In Swift, global variables with complex initializers can trigger these. The key insight: Swift global lets are lazily initialized (via dispatch_once), but ObjC/C++ globals are not.

// Swift: This is lazy — initialized on first access (safe)
let sharedManager = NetworkManager()

// C++: This runs BEFORE main() — adds to pre-main time
// static std::string kDefaultEndpoint = "https://api.example.com";

// ObjC: +load runs before main()
// + (void)load { /* swizzle something */ }
Deep Dive: Measuring Pre-main Time

Set the environment variable DYLD_PRINT_STATISTICS=1 in your Xcode scheme to see a breakdown of pre-main time:

Total pre-main time: 347.42 ms
  dylib loading time: 182.34 ms
  rebase/binding time:  23.18 ms
  ObjC setup time:      41.67 ms
  initializer time:    100.23 ms

For iOS 15+, use DYLD_PRINT_STATISTICS_DETAILS=1 for even more granularity. In production, use MetricKit's MXAppLaunchMetric to track launch time across your user base.

2. UIApplicationMain & the @main Attribute

2.1 The Traditional Entry Point

In Objective-C, every iOS app had an explicit main.m file:

// main.m — the traditional entry point
int main(int argc, char *argv[]) {
    @autoreleasepool {
        return UIApplicationMain(argc, argv, nil, 
            NSStringFromClass([AppDelegate class]));
    }
}

UIApplicationMain does the following, in order:

  1. Creates the UIApplication singleton (or a custom subclass if the 3rd parameter is non-nil)
  2. Creates an instance of the delegate class (4th parameter)
  3. Sets the delegate on the UIApplication singleton
  4. Starts the main run loop (CFRunLoopRun)
  5. Never returns — the function signature returns Int but it runs the run loop forever

2.2 Swift's @UIApplicationMain

Swift eliminated the explicit main.swift file with the @UIApplicationMain attribute (later renamed to @main):

// Pre-iOS 14: @UIApplicationMain
@UIApplicationMain
class AppDelegate: UIResponder, UIApplicationDelegate {
    var window: UIWindow?
    
    func application(
        _ application: UIApplication,
        didFinishLaunchingWithOptions launchOptions: [UIApplication.LaunchOptionsKey: Any]?
    ) -> Bool {
        return true
    }
}

2.3 The @main Attribute (Swift 5.3+)

Swift 5.3 introduced the generalized @main attribute. Any type with a static main() method can be the entry point. UIKit's UIApplicationDelegate protocol extension provides this automatically:

// iOS 14+: @main replaces @UIApplicationMain
@main
class AppDelegate: UIResponder, UIApplicationDelegate {
    // The compiler synthesizes:
    // static func main() {
    //     UIApplicationMain(
    //         CommandLine.argc,
    //         CommandLine.unsafeArgv,
    //         nil,
    //         NSStringFromClass(Self.self)
    //     )
    // }
}
Deep Dive: Why @main is More Than Syntactic Sugar

The @main attribute works with any type that has a static main(). This is how SwiftUI's App protocol works — it provides a default main() implementation that bootstraps the SwiftUI runtime instead of calling UIApplicationMain. This decouples the entry point from UIKit entirely.

3. UIApplicationDelegate as UIResponder Subclass

A frequently overlooked detail: AppDelegate inherits from UIResponder, not NSObject. This is intentional and architecturally significant.

3.1 The Responder Chain Integration

Touch/Event Responder Chain: UIView (first responder) │ ▼ UIViewController │ ▼ UIWindow │ ▼ UIWindowScene (iOS 13+) │ ▼ UIApplication ◀── singleton, also a UIResponder │ ▼ AppDelegate ◀── LAST in the responder chain

Because AppDelegate is a UIResponder, it sits at the end of the responder chain. Any event that isn't handled by views, view controllers, windows, or the application itself will bubble up to the app delegate. This enables:

  • Remote notification handling — UIResponder methods like remoteControlReceived(with:)
  • Motion events — shake gestures that no view handles
  • Key commands — hardware keyboard shortcuts via keyCommands
  • Undo/redo — application-wide undo manager
  • Menu actions — unhandled menu bar actions (Catalyst/iPad)
class AppDelegate: UIResponder, UIApplicationDelegate {
    
    // AppDelegate can handle events that bubble up the entire chain
    override var canBecomeFirstResponder: Bool { true }
    
    // Handle hardware keyboard shortcuts at the app level
    override var keyCommands: [UIKeyCommand]? {
        [
            UIKeyCommand(
                title: "New Document",
                action: #selector(newDocument),
                input: "n",
                modifierFlags: .command
            )
        ]
    }
    
    @objc func newDocument() {
        // Handle Cmd+N at the application level
    }
    
    // Handle remote control events (audio playback controls)
    override func remoteControlReceived(with event: UIEvent?) {
        guard let event = event else { return }
        switch event.subtype {
        case .remoteControlPlay:
            AudioPlayer.shared.play()
        case .remoteControlPause:
            AudioPlayer.shared.pause()
        default: break
        }
    }
}
Architect Must-Know

In scene-based apps (iOS 13+), the responder chain was extended: UIView → UIViewController → UIWindow → UIWindowScene → UIApplication → AppDelegate. The UIWindowScene is now in the chain, which means scene delegates can also participate in event handling. This is critical for multi-window apps where different scenes might handle the same key command differently.

4. willFinishLaunching vs didFinishLaunching

4.1 Ordering and Purpose

func application(
    _ application: UIApplication,
    willFinishLaunchingWithOptions launchOptions: [UIApplication.LaunchOptionsKey: Any]?
) -> Bool {
    // FIRST callback. App state: .inactive
    // State restoration happens BETWEEN will and did
    // Use for: essential setup that state restoration depends on
    return true
}

func application(
    _ application: UIApplication,
    didFinishLaunchingWithOptions launchOptions: [UIApplication.LaunchOptionsKey: Any]?
) -> Bool {
    // SECOND callback. State restoration is complete.
    // Use for: final setup, analytics, feature flags, UI customization
    return true
}
UIApplicationMain called │ ▼ willFinishLaunchingWithOptions ◀── App state: .inactive │ ▼ State Restoration ◀── NSCoder-based restoration happens here │ ▼ didFinishLaunchingWithOptions ◀── App state: still .inactive │ ▼ Scene Connection (iOS 13+) ◀── scene(_:willConnectTo:options:) │ ▼ applicationDidBecomeActive ◀── App state: .active (pre-iOS 13) ─── OR ─── sceneDidBecomeActive ◀── Scene state: .foregroundActive (iOS 13+)

4.2 What to Do in Each

Task willFinish didFinish Why
Core Data stack setup Yes State restoration may need the persistent store
Register for remote notifications Yes Needs app to be fully initialized
Analytics SDK init Yes Non-essential; keep willFinish fast
Feature flag fetch Yes Network-dependent, non-blocking
Appearance customization Yes After restoration so it doesn't conflict
Dependency injection container Yes Other setup code depends on it
URL/notification handling check Yes Check launchOptions for pending URLs/notifications
Common Pitfall: Return Value

Returning false from willFinishLaunchingWithOptions tells the system your app cannot handle the launch URL. The system will not call application(_:open:options:). However, returning false does not prevent the app from launching — it only affects URL handling. Many developers confuse this with "abort launch."

5. Evolution: iOS 2 → iOS 13 → iOS 14+

5.1 Era 1: iOS 2–12 — AppDelegate Owns Everything

// The "classic" AppDelegate — one window, one lifecycle
@UIApplicationMain
class AppDelegate: UIResponder, UIApplicationDelegate {
    var window: UIWindow?
    
    func application(_ app: UIApplication,
        didFinishLaunchingWithOptions opts: [UIApplication.LaunchOptionsKey: Any]?) -> Bool {
        window = UIWindow(frame: UIScreen.main.bounds)
        window?.rootViewController = MainViewController()
        window?.makeKeyAndVisible()
        return true
    }
    
    // Lifecycle was app-level, not scene-level
    func applicationDidBecomeActive(_ app: UIApplication) { }
    func applicationWillResignActive(_ app: UIApplication) { }
    func applicationDidEnterBackground(_ app: UIApplication) { }
    func applicationWillEnterForeground(_ app: UIApplication) { }
}

In this era, the AppDelegate was responsible for:

  • Creating and owning the single UIWindow
  • All lifecycle transitions (active, inactive, background, foreground)
  • State restoration
  • URL handling
  • Remote notification registration and handling
  • Background fetch

5.2 Era 2: iOS 13 — Scene-Based Lifecycle

iOS 13 introduced UIScene to support multi-window on iPad. This was the biggest architectural shift in UIKit's history:

// iOS 13+: AppDelegate loses lifecycle duties to SceneDelegate
@main
class AppDelegate: UIResponder, UIApplicationDelegate {
    // NO more window property
    // NO more lifecycle methods (they're ignored if scenes are adopted)
    
    func application(_ app: UIApplication,
        didFinishLaunchingWithOptions opts: [UIApplication.LaunchOptionsKey: Any]?) -> Bool {
        // Process-level setup only
        return true
    }
    
    // NEW: Scene configuration
    func application(_ app: UIApplication,
        configurationForConnecting session: UISceneSession,
        options: UIScene.ConnectionOptions) -> UISceneConfiguration {
        return UISceneConfiguration(
            name: "Default",
            sessionRole: session.role
        )
    }
    
    // NEW: Scene disconnection
    func application(_ app: UIApplication,
        didDiscardSceneSessions sessions: Set<UISceneSession>) {
        // Clean up resources for discarded scenes
    }
}

class SceneDelegate: UIResponder, UIWindowSceneDelegate {
    var window: UIWindow?
    
    func scene(_ scene: UIScene,
        willConnectTo session: UISceneSession,
        options: UIScene.ConnectionOptions) {
        guard let windowScene = scene as? UIWindowScene else { return }
        window = UIWindow(windowScene: windowScene)
        window?.rootViewController = MainViewController()
        window?.makeKeyAndVisible()
    }
    
    // Lifecycle is now per-scene
    func sceneDidBecomeActive(_ scene: UIScene) { }
    func sceneWillResignActive(_ scene: UIScene) { }
    func sceneDidEnterBackground(_ scene: UIScene) { }
    func sceneWillEnterForeground(_ scene: UIScene) { }
}

5.3 Responsibility Split

Responsibility Pre-iOS 13 (AppDelegate) iOS 13+ (Split)
Process launch AppDelegate AppDelegate
Window creation AppDelegate SceneDelegate
Lifecycle (active/background) AppDelegate SceneDelegate
State restoration AppDelegate SceneDelegate
URL handling AppDelegate SceneDelegate (via ConnectionOptions)
Remote notification registration AppDelegate AppDelegate (unchanged)
Scene configuration N/A AppDelegate
Scene cleanup N/A AppDelegate

5.4 Era 3: iOS 14+ — SwiftUI App Protocol

SwiftUI introduced the App protocol, which can entirely replace both AppDelegate and SceneDelegate:

@main
struct MyApp: App {
    // @UIApplicationDelegateAdaptor bridges to UIKit when needed
    @UIApplicationDelegateAdaptor(AppDelegate.self) var appDelegate
    
    var body: some Scene {
        WindowGroup {
            ContentView()
        }
    }
}

// Still need AppDelegate for push notifications, etc.
class AppDelegate: NSObject, UIApplicationDelegate {
    func application(_ app: UIApplication,
        didRegisterForRemoteNotificationsWithDeviceToken token: Data) {
        // Handle push token
    }
}

6. SwiftUI App Protocol & @main

6.1 How SwiftUI Bootstraps

When you use @main on a SwiftUI App struct, the compiler generates a static func main() that does not call UIApplicationMain. Instead, it:

  1. Creates a UIApplication instance (still needed for system integration)
  2. Creates the SwiftUI app struct
  3. Evaluates the body to get Scene declarations
  4. Maps SwiftUI Scene types to UIKit UIScene configurations
  5. Starts the main run loop

6.2 Scene Types in SwiftUI

@main
struct MyApp: App {
    var body: some Scene {
        // WindowGroup: multi-instance windows (maps to UIWindowScene)
        WindowGroup {
            ContentView()
        }
        
        // DocumentGroup: document-based apps
        DocumentGroup(newDocument: MyDocument()) { file in
            DocumentEditor(document: file.$document)
        }
        
        // Settings: macOS/Catalyst preferences window
        #if os(macOS)
        Settings {
            PreferencesView()
        }
        #endif
    }
}

6.3 @UIApplicationDelegateAdaptor Limitations

The adaptor bridges UIKit delegate methods to SwiftUI, but not all methods are forwarded. Notably:

  • application(_:configurationForConnecting:options:) — NOT forwarded; SwiftUI manages scene configuration
  • application(_:didDiscardSceneSessions:) — NOT forwarded
  • Push notification methods — forwarded
  • application(_:didFinishLaunchingWithOptions:) — forwarded
  • Background URL session methods — forwarded

7. Full Boot Sequence: Icon Tap to First Frame

┌─────────────────────────────────────────────────────────────────┐ │ FULL iOS APP BOOT SEQUENCE │ ├─────────────────────────────────────────────────────────────────┤ │ │ │ ① KERNEL (XNU) │ │ fork() → exec() → load Mach-O → map pages → start dyld │ │ │ │ ② DYLD4 (Pre-main) ~100ms │ │ Load shared cache → map dylibs → rebase → bind │ │ → ObjC class registration → +load methods │ │ → C++ static constructors → jump to main() │ │ │ │ ③ UIApplicationMain ~10ms │ │ Create UIApplication singleton │ │ Create AppDelegate instance │ │ Set delegate on UIApplication │ │ Start main RunLoop │ │ │ │ ④ WILL FINISH LAUNCHING ~5ms │ │ application(_:willFinishLaunchingWithOptions:) │ │ → Core Data stack, DI container, essential setup │ │ │ │ ⑤ STATE RESTORATION (if applicable) ~20ms │ │ Decode saved state → recreate view controller hierarchy │ │ │ │ ⑥ DID FINISH LAUNCHING ~50ms │ │ application(_:didFinishLaunchingWithOptions:) │ │ → Analytics, feature flags, push registration │ │ │ │ ⑦ SCENE CONNECTION (iOS 13+) ~10ms │ │ scene(_:willConnectTo:options:) │ │ → Create UIWindow, set rootViewController │ │ → makeKeyAndVisible() │ │ │ │ ⑧ LAYOUT PASS ~20ms │ │ layoutSubviews() cascade │ │ Auto Layout constraint solving │ │ │ │ ⑨ FIRST RENDER ~16ms │ │ CATransaction commit → render server → GPU → display │ │ Launch storyboard dismissed → first app frame visible │ │ │ │ ⑩ ACTIVE │ │ sceneDidBecomeActive / applicationDidBecomeActive │ │ │ │ Total: ~200-400ms (target: <400ms warm, <1s cold) │ └─────────────────────────────────────────────────────────────────┘
Best Practice: Launch Time Budget

Apple's guidelines: cold launch should complete in under 400ms. The system watchdog will kill your app if launch takes more than 20 seconds. At Google scale, target p95 cold launch under 1 second. Key optimizations:

  • Eliminate +load methods (move to +initialize or lazy init)
  • Reduce dylib count (merge frameworks, use static linking)
  • Defer non-essential work past first frame
  • Use os_signpost to instrument each phase
  • Profile with Instruments App Launch template

8. Architect Considerations

Architect Must-Know: Scene-Based Lifecycle Broke Singleton Patterns

The transition to scene-based lifecycle in iOS 13 was the most significant architectural disruption in UIKit's history. Here's why:

The Problem: Pre-iOS 13, apps had one window, one lifecycle. Singletons like AppState.shared mapped naturally to this model. With multi-window, you now have multiple scenes, each with its own lifecycle, potentially showing different content.

What Broke:

  • UIScreen.main — which screen? Multiple scenes can be on different screens
  • UIApplication.shared.keyWindow — deprecated; there's no single "key" window
  • Singleton state managers that assumed one active view hierarchy
  • Deep link handlers that push onto "the" navigation stack
  • Alert presenters that use topViewController heuristics

The Architectural Solution:

  • Separate process-scoped state (user session, auth tokens, caches) from scene-scoped state (navigation, UI state, selected tab)
  • Use dependency injection to provide scene-specific context rather than singletons
  • Design navigation routers that accept a scene/window parameter
  • Use UIScene.session.persistentIdentifier to key scene-specific data

8.1 Process-Scoped vs Scene-Scoped State

// Process-scoped: shared across all scenes (singleton OK)
final class AuthManager {
    static let shared = AuthManager()
    var currentUser: User?
    var accessToken: String?
}

// Scene-scoped: unique per scene instance
final class SceneState: ObservableObject {
    let sceneIdentifier: String
    @Published var selectedTab: Tab = .home
    @Published var navigationPath: [Route] = []
    
    init(session: UISceneSession) {
        self.sceneIdentifier = session.persistentIdentifier
    }
}

// Scene state registry: maps scene IDs to their state
final class SceneStateRegistry {
    static let shared = SceneStateRegistry()
    private var states: [String: SceneState] = [:]
    
    func state(for session: UISceneSession) -> SceneState {
        if let existing = states[session.persistentIdentifier] {
            return existing
        }
        let newState = SceneState(session: session)
        states[session.persistentIdentifier] = newState
        return newState
    }
    
    func removeState(for session: UISceneSession) {
        states.removeValue(forKey: session.persistentIdentifier)
    }
}

8.2 Launch Time Architecture at Scale

// Phased launch: defer work past first frame
final class LaunchCoordinator {
    enum Phase {
        case critical    // Before first frame: DI, Core Data, auth check
        case important   // First idle: analytics, feature flags, prefetch
        case deferred    // After 2s: non-essential SDKs, prewarming
    }
    
    private var tasks: [Phase: [() -> Void]] = [:]
    
    func register(phase: Phase, task: @escaping () -> Void) {
        tasks[phase, default: []].append(task)
    }
    
    func executeCritical() {
        tasks[.critical]?.forEach { $0() }
    }
    
    func schedulePostLaunch() {
        DispatchQueue.main.async { [self] in
            tasks[.important]?.forEach { $0() }
        }
        DispatchQueue.main.asyncAfter(deadline: .now() + 2.0) { [self] in
            tasks[.deferred]?.forEach { $0() }
        }
    }
}
Best Practice: Startup Instrumentation

Instrument every phase of your launch with os_signpost:

import os

let launchLog = OSLog(subsystem: "com.app.launch", category: .pointsOfInterest)

func application(_ app: UIApplication,
    didFinishLaunchingWithOptions opts: [UIApplication.LaunchOptionsKey: Any]?) -> Bool {
    os_signpost(.begin, log: launchLog, name: "didFinishLaunching")
    // ... setup ...
    os_signpost(.end, log: launchLog, name: "didFinishLaunching")
    return true
}
