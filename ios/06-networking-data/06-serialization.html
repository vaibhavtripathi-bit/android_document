1. Codable Protocol Internals

Codable is a typealias combining two protocols:

public typealias Codable = Encodable & Decodable

public protocol Encodable {
    func encode(to encoder: any Encoder) throws
}

public protocol Decodable {
    init(from decoder: any Decoder) throws
}

Compiler-Synthesized Conformance

When all stored properties are themselves Codable, the compiler generates three things automatically: a CodingKeys enum, an init(from:) initializer, and an encode(to:) method. Here is what the compiler generates for a simple model:

// What you write:
struct User: Codable {
    let id: Int
    let name: String
    let email: String?
    let createdAt: Date
}

// What the compiler synthesizes:
struct User: Codable {
    let id: Int
    let name: String
    let email: String?
    let createdAt: Date

    private enum CodingKeys: String, CodingKey {
        case id
        case name
        case email
        case createdAt
    }

    init(from decoder: any Decoder) throws {
        let container = try decoder.container(keyedBy: CodingKeys.self)
        id = try container.decode(Int.self, forKey: .id)
        name = try container.decode(String.self, forKey: .name)
        email = try container.decodeIfPresent(String.self, forKey: .email)
        createdAt = try container.decode(Date.self, forKey: .createdAt)
    }

    func encode(to encoder: any Encoder) throws {
        var container = encoder.container(keyedBy: CodingKeys.self)
        try container.encode(id, forKey: .id)
        try container.encode(name, forKey: .name)
        try container.encodeIfPresent(email, forKey: .email)
        try container.encode(createdAt, forKey: .createdAt)
    }
}

Encoder / Decoder Protocol Hierarchy

Deep Dive: Container Types

KeyedContainer maps to JSON objects / dictionaries — each value accessed by a CodingKey. UnkeyedContainer maps to JSON arrays — sequential access. SingleValueContainer wraps a single primitive — used for types like Date, URL, or custom RawRepresentable enums. Understanding these three containers is the key to writing any custom Codable conformance.

2. JSONEncoder / JSONDecoder

Configuration Options

let decoder = JSONDecoder()

// Date strategies
decoder.dateDecodingStrategy = .iso8601
decoder.dateDecodingStrategy = .secondsSince1970
decoder.dateDecodingStrategy = .millisecondsSince1970
decoder.dateDecodingStrategy = .formatted(myDateFormatter)
decoder.dateDecodingStrategy = .custom { decoder in
    let container = try decoder.singleValueContainer()
    let string = try container.decode(String.self)
    guard let date = ISO8601DateFormatter().date(from: string) else {
        throw DecodingError.dataCorruptedError(
            in: container, debugDescription: "Invalid date: \(string)"
        )
    }
    return date
}

// Key strategies
decoder.keyDecodingStrategy = .convertFromSnakeCase
decoder.keyDecodingStrategy = .custom { keys in
    // Custom key mapping logic
    let lastKey = keys.last!.stringValue
    return AnyKey(stringValue: lastKey.lowercased())!
}

// Data strategies
decoder.dataDecodingStrategy = .base64
decoder.dataDecodingStrategy = .custom { decoder in
    let container = try decoder.singleValueContainer()
    let hex = try container.decode(String.self)
    return Data(hexString: hex)
}

// Non-conforming float handling
decoder.nonConformingFloatDecodingStrategy = .convertFromString(
    positiveInfinity: "Infinity",
    negativeInfinity: "-Infinity",
    nan: "NaN"
)

Encoder Configuration

let encoder = JSONEncoder()
encoder.outputFormatting = [.prettyPrinted, .sortedKeys, .withoutEscapingSlashes]
encoder.dateEncodingStrategy = .iso8601
encoder.keyEncodingStrategy = .convertToSnakeCase

Best Practice

Create a single shared JSONDecoder instance per configuration rather than allocating one per decode call. JSONDecoder is thread-safe for reads and reusing it avoids repeated internal setup. For apps decoding hundreds of API responses, this alone can save measurable allocation overhead.

3. PropertyListEncoder / PropertyListDecoder

Property lists are Apple's native serialization format, used extensively by the system for preferences, entitlements, and Info.plist. Two on-disk formats exist: XML and binary.

When to Use Plist Over JSON

  • UserDefaults storage — UserDefaults is backed by plist natively
  • App configuration — Info.plist, entitlements, launch storyboard settings
  • Local caching — Binary plist is faster to decode than JSON for simple structures
  • Apple ecosystem interop — Spotlight metadata, MDM profiles, provisioning profiles
struct AppSettings: Codable {
    var theme: String
    var fontSize: Int
    var notificationsEnabled: Bool
    var recentSearches: [String]
}

// Encode to binary plist
let encoder = PropertyListEncoder()
encoder.outputFormat = .binary  // .xml for human-readable

let settings = AppSettings(
    theme: "dark",
    fontSize: 14,
    notificationsEnabled: true,
    recentSearches: ["swift", "codable"]
)
let data = try encoder.encode(settings)
try data.write(to: settingsURL)

// Decode from plist
let decoder = PropertyListDecoder()
let loaded = try decoder.decode(AppSettings.self, from: data)

Binary Plist vs JSON Performance

Deep Dive: Binary Plist Internals

Binary plists use an offset table at the end of the file, allowing random access to any object without parsing the entire file. The format stores object types in a single byte header (0x0N for primitives, 0x1N for ints, 0x4N for data, 0x5N for strings, 0xAN for arrays, 0xDN for dicts). This makes binary plist particularly fast for partial reads — NSDictionary(contentsOf:) can memory-map and lazily fault objects.

4. Custom Encoding / Decoding

Custom CodingKeys

struct Product: Codable {
    let productId: Int
    let displayName: String
    let priceUSD: Double
    let isAvailable: Bool

    private enum CodingKeys: String, CodingKey {
        case productId = "product_id"
        case displayName = "display_name"
        case priceUSD = "price_usd"
        case isAvailable = "is_available"
    }
}

Flattening Nested JSON

APIs often return nested structures that you want to flatten in your model:

// JSON: { "id": 1, "attributes": { "name": "Alice", "role": "admin" } }

struct Employee: Decodable {
    let id: Int
    let name: String
    let role: String

    private enum CodingKeys: String, CodingKey {
        case id
        case attributes
    }

    private enum AttributeKeys: String, CodingKey {
        case name
        case role
    }

    init(from decoder: any Decoder) throws {
        let container = try decoder.container(keyedBy: CodingKeys.self)
        id = try container.decode(Int.self, forKey: .id)

        let attrs = try container.nestedContainer(
            keyedBy: AttributeKeys.self, forKey: .attributes
        )
        name = try attrs.decode(String.self, forKey: .name)
        role = try attrs.decode(String.self, forKey: .role)
    }
}

Handling Polymorphic Types

When an API returns an array of mixed types distinguished by a discriminator field:

// JSON: [
//   { "type": "text", "content": "Hello" },
//   { "type": "image", "url": "https://...", "width": 640, "height": 480 },
//   { "type": "video", "url": "https://...", "duration": 120.5 }
// ]

enum MediaItem: Decodable {
    case text(TextContent)
    case image(ImageContent)
    case video(VideoContent)
    case unknown

    struct TextContent: Decodable {
        let content: String
    }

    struct ImageContent: Decodable {
        let url: URL
        let width: Int
        let height: Int
    }

    struct VideoContent: Decodable {
        let url: URL
        let duration: Double
    }

    private enum CodingKeys: String, CodingKey {
        case type
    }

    init(from decoder: any Decoder) throws {
        let container = try decoder.container(keyedBy: CodingKeys.self)
        let type = try container.decode(String.self, forKey: .type)

        switch type {
        case "text":
            self = .text(try TextContent(from: decoder))
        case "image":
            self = .image(try ImageContent(from: decoder))
        case "video":
            self = .video(try VideoContent(from: decoder))
        default:
            self = .unknown
        }
    }
}

Handling API Response Wrappers

// Generic API response: { "data": T, "meta": { "page": 1, "total": 100 } }

struct APIResponse<T: Decodable>: Decodable {
    let data: T
    let meta: Meta

    struct Meta: Decodable {
        let page: Int
        let total: Int
    }
}

// Usage
let response = try decoder.decode(
    APIResponse<[User]>.self, from: jsonData
)
let users = response.data
let totalPages = response.meta.total

Handling Unknown Enum Cases

enum Status: String, Codable {
    case active
    case inactive
    case suspended
    case unknown

    init(from decoder: any Decoder) throws {
        let container = try decoder.singleValueContainer()
        let rawValue = try container.decode(String.self)
        self = Status(rawValue: rawValue) ?? .unknown
    }
}
Pitfall: Enum Without Unknown Case

If your RawRepresentable enum doesn't handle unknown raw values, decoding will throw a DecodingError.dataCorrupted when the server adds a new case. This is the #1 cause of Codable crashes in production. Always add an unknown case or use a custom init(from:) with a fallback.

5. @propertyWrapper for Decoding Strategies

Default Value on Missing Key

protocol DefaultValueProvider {
    associatedtype Value: Decodable
    static var defaultValue: Value { get }
}

@propertyWrapper
struct Default<Provider: DefaultValueProvider>: Decodable {
    var wrappedValue: Provider.Value

    init() {
        wrappedValue = Provider.defaultValue
    }

    init(wrappedValue: Provider.Value) {
        self.wrappedValue = wrappedValue
    }

    init(from decoder: any Decoder) throws {
        let container = try decoder.singleValueContainer()
        wrappedValue = (try? container.decode(Provider.Value.self))
            ?? Provider.defaultValue
    }
}

extension KeyedDecodingContainer {
    func decode<P: DefaultValueProvider>(
        _ type: Default<P>.Type,
        forKey key: Key
    ) throws -> Default<P> {
        (try? decodeIfPresent(type, forKey: key)) ?? Default()
    }
}

// Concrete providers
enum False: DefaultValueProvider {
    static let defaultValue = false
}
enum EmptyString: DefaultValueProvider {
    static let defaultValue = ""
}
enum Zero: DefaultValueProvider {
    static let defaultValue = 0
}

// Usage
struct Profile: Decodable {
    let name: String
    @Default<False> var isPremium: Bool
    @Default<EmptyString> var bio: String
    @Default<Zero> var followerCount: Int
}

Lossy Array (Skip Invalid Elements)

@propertyWrapper
struct LossyArray<Element: Decodable>: Decodable {
    var wrappedValue: [Element]

    private struct AnyDecodableValue: Decodable {
        let value: Element?

        init(from decoder: any Decoder) throws {
            value = try? Element(from: decoder)
        }
    }

    init(from decoder: any Decoder) throws {
        var container = try decoder.unkeyedContainer()
        var elements: [Element] = []

        while !container.isAtEnd {
            if let wrapper = try? container.decode(AnyDecodableValue.self),
               let value = wrapper.value {
                elements.append(value)
            }
        }
        wrappedValue = elements
    }

    init(wrappedValue: [Element]) {
        self.wrappedValue = wrappedValue
    }
}

// Usage: gracefully skips malformed items
struct Feed: Decodable {
    @LossyArray var posts: [Post]
}

String-to-Number Coercion

@propertyWrapper
struct StringToInt: Decodable {
    var wrappedValue: Int

    init(wrappedValue: Int) {
        self.wrappedValue = wrappedValue
    }

    init(from decoder: any Decoder) throws {
        let container = try decoder.singleValueContainer()
        if let intValue = try? container.decode(Int.self) {
            wrappedValue = intValue
        } else {
            let stringValue = try container.decode(String.self)
            guard let intValue = Int(stringValue) else {
                throw DecodingError.dataCorruptedError(
                    in: container,
                    debugDescription: "Cannot convert '\(stringValue)' to Int"
                )
            }
            wrappedValue = intValue
        }
    }
}

// Handles both { "count": 42 } and { "count": "42" }
struct Analytics: Decodable {
    @StringToInt var viewCount: Int
    @StringToInt var likeCount: Int
}

Custom Date Format Wrapper

@propertyWrapper
struct DateFormatted: Decodable {
    var wrappedValue: Date

    static let formatter: DateFormatter = {
        let f = DateFormatter()
        f.dateFormat = "yyyy-MM-dd'T'HH:mm:ss.SSSZ"
        f.locale = Locale(identifier: "en_US_POSIX")
        return f
    }()

    init(wrappedValue: Date) {
        self.wrappedValue = wrappedValue
    }

    init(from decoder: any Decoder) throws {
        let container = try decoder.singleValueContainer()
        let string = try container.decode(String.self)
        guard let date = DateFormatted.formatter.date(from: string) else {
            throw DecodingError.dataCorruptedError(
                in: container,
                debugDescription: "Date '\(string)' does not match expected format"
            )
        }
        wrappedValue = date
    }
}
Best Practice

Property wrappers compose well with Codable but have a caveat: the wrapper must handle the KeyedDecodingContainer extension for missing keys to work. Without the extension, a missing key for a wrapped property will throw, even if the wrapper provides a default. Always pair your wrapper with the container extension.

6. Codable Edge Cases & Pitfalls

null vs Missing Keys

struct Example: Decodable {
    let required: String       // Missing key → throws
    let optional: String?     // Missing key → nil (OK)
                                // null value  → nil (OK)
    let hasDefault: String    // Missing key → throws (no default for let)
}

// For structs, only Optional properties tolerate missing keys.
// For classes with default values, missing keys use the default.
Pitfall: null vs Absent in Encoding

When encoding, encodeIfPresent omits the key entirely when the value is nil. If your API requires explicit null values (e.g., PATCH endpoints), you must use encode(value, forKey:) with a custom strategy or wrap in a Nullable<T> type that encodes null explicitly. This is a frequent source of bugs in REST API integrations.

Inheritance and Codable

class Animal: Codable {
    let name: String
    let legs: Int
}

class Dog: Animal {
    let breed: String

    private enum CodingKeys: String, CodingKey {
        case breed
    }

    required init(from decoder: any Decoder) throws {
        let container = try decoder.container(keyedBy: CodingKeys.self)
        breed = try container.decode(String.self, forKey: .breed)
        try super.init(from: decoder)  // Must call super
    }

    override func encode(to encoder: any Encoder) throws {
        var container = encoder.container(keyedBy: CodingKeys.self)
        try container.encode(breed, forKey: .breed)
        try super.encode(to: encoder)  // Must call super
    }
}
Pitfall: Inheritance Breaks Synthesized Codable

The compiler does not synthesize init(from:) or encode(to:) for subclasses. If you add a stored property to a subclass and rely on synthesized conformance, the subclass property will be silently ignored — it won't encode or decode. You must manually implement both methods and call super. Prefer composition (structs) over inheritance for Codable types.

Enum with Associated Values

// Swift 5.5+ auto-synthesizes Codable for enums with associated values
enum PaymentMethod: Codable {
    case creditCard(number: String, expiry: String)
    case applePay(merchantId: String)
    case cash
}

// Synthesized JSON format:
// { "creditCard": { "number": { "_0": "4111..." }, "expiry": { "_0": "12/26" } } }
// This format is verbose — custom encoding is usually better for APIs:

enum Payment: Codable {
    case creditCard(number: String, expiry: String)
    case applePay(merchantId: String)
    case cash

    private enum CodingKeys: String, CodingKey {
        case type, number, expiry, merchantId
    }

    func encode(to encoder: any Encoder) throws {
        var container = encoder.container(keyedBy: CodingKeys.self)
        switch self {
        case .creditCard(let number, let expiry):
            try container.encode("credit_card", forKey: .type)
            try container.encode(number, forKey: .number)
            try container.encode(expiry, forKey: .expiry)
        case .applePay(let merchantId):
            try container.encode("apple_pay", forKey: .type)
            try container.encode(merchantId, forKey: .merchantId)
        case .cash:
            try container.encode("cash", forKey: .type)
        }
    }

    init(from decoder: any Decoder) throws {
        let container = try decoder.container(keyedBy: CodingKeys.self)
        let type = try container.decode(String.self, forKey: .type)
        switch type {
        case "credit_card":
            let number = try container.decode(String.self, forKey: .number)
            let expiry = try container.decode(String.self, forKey: .expiry)
            self = .creditCard(number: number, expiry: expiry)
        case "apple_pay":
            let merchantId = try container.decode(String.self, forKey: .merchantId)
            self = .applePay(merchantId: merchantId)
        case "cash":
            self = .cash
        default:
            throw DecodingError.dataCorruptedError(
                forKey: .type, in: container,
                debugDescription: "Unknown payment type: \(type)"
            )
        }
    }
}

7. Protocol Buffers on iOS (SwiftProtobuf)

Protocol Buffers (protobuf) is Google's language-neutral binary serialization format. It uses a schema (.proto file) to generate type-safe code, producing compact binary output that is 3–10x smaller than JSON.

.proto File Definition

// user.proto
syntax = "proto3";

package myapp;

message User {
    int64 id = 1;
    string name = 2;
    string email = 3;
    UserRole role = 4;
    repeated string tags = 5;
    google.protobuf.Timestamp created_at = 6;
}

enum UserRole {
    USER_ROLE_UNSPECIFIED = 0;
    USER_ROLE_ADMIN = 1;
    USER_ROLE_MEMBER = 2;
    USER_ROLE_GUEST = 3;
}

message UserList {
    repeated User users = 1;
    int32 total_count = 2;
}

Code Generation & Usage

// Install: brew install swift-protobuf
// Generate: protoc --swift_out=. user.proto
// Add SPM: .package(url: "https://github.com/apple/swift-protobuf.git", from: "1.25.0")

// Generated code gives you type-safe Swift structs:
import SwiftProtobuf

// Create a message
var user = Myapp_User()
user.id = 42
user.name = "Alice"
user.email = "alice@example.com"
user.role = .admin
user.tags = ["ios", "swift"]
user.createdAt = .init(date: Date())

// Serialize to binary (compact)
let binaryData = try user.serializedData()
print(binaryData.count)  // ~45 bytes vs ~180 bytes JSON

// Deserialize from binary
let decoded = try Myapp_User(serializedData: binaryData)

// JSON interop (for debugging or API compatibility)
let jsonString = try user.jsonString()
let fromJSON = try Myapp_User(jsonString: jsonString)
┌─────────────┐ protoc ┌──────────────────┐ │ .proto file │ ──────────────▶ │ Generated .swift │ │ (schema) │ --swift_out │ (value types) │ └─────────────┘ └────────┬─────────┘ │ ┌───────────────────────┤ ▼ ▼ ┌─────────────────┐ ┌─────────────────┐ │ serializedData()│ │ jsonString() │ │ Binary format │ │ JSON format │ │ ~45 bytes │ │ ~180 bytes │ └─────────────────┘ └─────────────────┘

Protobuf Binary Format Advantages

  • Size: 3–10x smaller than JSON (field tags are varint-encoded integers, not strings)
  • Speed: No string parsing; fields identified by numeric tags
  • Schema evolution: Add/remove fields without breaking existing clients (field numbers are stable)
  • Type safety: Generated code catches type mismatches at compile time
  • Cross-platform: Same .proto generates Swift, Kotlin, TypeScript, Go, etc.
Deep Dive: Wire Format

Protobuf encodes each field as a (field_number, wire_type) varint tag followed by the value. Wire types: 0 = varint (int, bool, enum), 1 = 64-bit fixed, 2 = length-delimited (string, bytes, embedded messages), 5 = 32-bit fixed. Unknown fields are preserved on decode, enabling forward compatibility. A decoder seeing an unknown field number simply skips it based on the wire type.

8. FlatBuffers

FlatBuffers is Google's zero-copy serialization library. Unlike protobuf, the serialized buffer is the in-memory representation — no deserialization step is needed. You access fields directly from the buffer via offset lookups.

Schema Definition

// game_state.fbs
namespace GameApp;

table Player {
    id: long;
    name: string;
    x: float;
    y: float;
    health: int;
    inventory: [Item];
}

table Item {
    id: int;
    name: string;
    quantity: int;
}

table GameState {
    players: [Player];
    tick: ulong;
    map_id: string;
}

root_type GameState;

Memory Layout

FlatBuffer Memory Layout (zero-copy access): ┌──────────────────────────────────────────────────────────┐ │ Buffer │ │ │ │ ┌──────────┐ ┌──────────┐ ┌──────────────────────┐ │ │ │ Root │ │ VTable │ │ Data (inline fields) │ │ │ │ Offset │→ │ offsets │→ │ + string offsets │ │ │ │ (4 bytes) │ │ to fields│ │ + vector offsets │ │ │ └──────────┘ └──────────┘ └──────────────────────┘ │ │ │ │ Access: buffer[rootOffset + vtable[fieldIndex]] │ │ No allocation, no parsing, no copying │ └──────────────────────────────────────────────────────────┘ Traditional Deserialization (Codable / Protobuf): Wire bytes → Parse → Allocate objects → Copy values → Use FlatBuffers: Wire bytes → Use (direct offset reads from buffer)

Swift Usage

// Generate: flatc --swift game_state.fbs
// Add SPM: .package(url: "https://github.com/nicklama/flatbuffers-swift", from: "0.6.0")

import FlatBuffers

// Building a FlatBuffer
var builder = FlatBufferBuilder(initialSize: 1024)

// Strings and vectors must be created before the table
let nameOffset = builder.create(string: "Alice")
let mapOffset = builder.create(string: "desert_01")

// Build player
let player = GameApp_Player.createPlayer(
    &builder,
    id: 1,
    nameOffset: nameOffset,
    x: 10.5,
    y: 20.3,
    health: 100
)

let playersVector = builder.createVector(ofOffsets: [player])

let gameState = GameApp_GameState.createGameState(
    &builder,
    playersVectorOffset: playersVector,
    tick: 42000,
    mapIdOffset: mapOffset
)
builder.finish(offset: gameState)

let data = builder.sizedBuffer

// Reading — zero-copy, no deserialization
let buf = ByteBuffer(data: data)
let state = GameApp_GameState.getRootAsGameState(bb: buf)
print(state.tick)                     // 42000 — direct read
print(state.players(at: 0)?.name)     // "Alice" — offset lookup
print(state.players(at: 0)?.health)   // 100

When to Use FlatBuffers Over Protobuf

  • Real-time data: Game state updates, sensor streams — zero-copy means zero allocation per frame
  • Memory-mapped files: Read directly from disk without loading into memory
  • Large datasets: Access individual records without parsing the entire buffer
  • Latency-critical paths: No deserialization step means predictable, minimal latency
Deep Dive: Zero-Copy Trade-offs

FlatBuffers achieves zero-copy by storing data in a format that matches the access pattern. The trade-off: building a FlatBuffer is slower than protobuf encoding (you must create strings/vectors before tables, and the builder works backwards). The read path is faster because it's just pointer arithmetic. This makes FlatBuffers ideal for "write once, read many" scenarios like cached data or configuration files.

9. MessagePack

MessagePack is a binary serialization format that is structurally equivalent to JSON but more compact. It encodes JSON-compatible types (maps, arrays, strings, integers, floats, booleans, nil) in a binary format that is typically 20–50% smaller than JSON.

Usage with Swift

// Add SPM: .package(url: "https://github.com/hirotosuzuki/msgpack-swift.git", from: "4.0.0")
// Alternative: .package(url: "https://github.com/Flight-School/MessagePack.git", from: "1.2.0")

import MessagePack

struct Event: Codable {
    let name: String
    let timestamp: Double
    let properties: [String: String]
}

let event = Event(
    name: "screen_view",
    timestamp: 1710936000.0,
    properties: ["screen": "home", "source": "push"]
)

// Encode to MessagePack
let encoder = MessagePackEncoder()
let packed = try encoder.encode(event)
print(packed.count)  // ~65 bytes vs ~95 bytes JSON

// Decode from MessagePack
let decoder = MessagePackDecoder()
let decoded = try decoder.decode(Event.self, from: packed)

Deep Dive: Why Foundation JSONDecoder is Slow

Foundation's JSONDecoder first calls JSONSerialization (Objective-C) to parse JSON into NSArray/NSDictionary, then bridges those to Swift types, then walks the Codable container hierarchy. This double-conversion (bytes → ObjC objects → Swift values) is why pure-Swift decoders like ZippyJSON (which uses simdjson under the hood) can be 3–4x faster — they parse directly from bytes to Swift values in a single pass.

Multi-Format Architecture

protocol SerializationFormat {
    func encode<T: Encodable>(_ value: T) throws -> Data
    func decode<T: Decodable>(_ type: T.Type, from data: Data) throws -> T
}

struct JSONFormat: SerializationFormat {
    private let encoder: JSONEncoder
    private let decoder: JSONDecoder

    init(dateStrategy: JSONDecoder.DateDecodingStrategy = .iso8601) {
        encoder = JSONEncoder()
        encoder.dateEncodingStrategy = .iso8601
        decoder = JSONDecoder()
        decoder.dateDecodingStrategy = dateStrategy
        decoder.keyDecodingStrategy = .convertFromSnakeCase
    }

    func encode<T: Encodable>(_ value: T) throws -> Data {
        try encoder.encode(value)
    }

    func decode<T: Decodable>(_ type: T.Type, from data: Data) throws -> T {
        try decoder.decode(type, from: data)
    }
}

struct BinaryPlistFormat: SerializationFormat {
    private let encoder: PropertyListEncoder
    private let decoder: PropertyListDecoder

    init() {
        encoder = PropertyListEncoder()
        encoder.outputFormat = .binary
        decoder = PropertyListDecoder()
    }

    func encode<T: Encodable>(_ value: T) throws -> Data {
        try encoder.encode(value)
    }

    func decode<T: Decodable>(_ type: T.Type, from data: Data) throws -> T {
        try decoder.decode(type, from: data)
    }
}

// Unified data layer that picks the right format per context
final class DataLayer {
    private let apiFormat: any SerializationFormat
    private let cacheFormat: any SerializationFormat

    init(
        apiFormat: any SerializationFormat = JSONFormat(),
        cacheFormat: any SerializationFormat = BinaryPlistFormat()
    ) {
        self.apiFormat = apiFormat
        self.cacheFormat = cacheFormat
    }

    func fetchAndCache<T: Codable>(
        _ type: T.Type,
        from networkData: Data,
        cachePath: URL
    ) throws -> T {
        // Decode from API (JSON)
        let value = try apiFormat.decode(type, from: networkData)

        // Re-encode to cache format (binary plist — faster reads)
        let cacheData = try cacheFormat.encode(value)
        try cacheData.write(to: cachePath)

        return value
    }

    func loadFromCache<T: Decodable>(
        _ type: T.Type,
        at path: URL
    ) throws -> T {
        let data = try Data(contentsOf: path)
        return try cacheFormat.decode(type, from: data)
    }
}

    private let encoder = MessagePackEncoder()
    private let maxBatchSize = 50
    private let queue = DispatchQueue(label: "analytics.batcher")

    func track<E: Encodable>(_ event: E) {
        queue.async { [weak self] in
            guard let self,
                  let data = try? self.encoder.encode(event) else { return }
            self.buffer.append(data)

            if self.buffer.count >= self.maxBatchSize {
                self.flush()
            }
        }

        uploadToServer(payload)

Pitfall: Mixing Formats Without Abstraction

Scattering JSONDecoder() calls throughout your codebase makes it impossible to switch formats later. Always abstract serialization behind a protocol. When you need to migrate from JSON to protobuf for a hot path, you change one conformance instead of hunting through 200 call sites. The SerializationFormat protocol above costs nothing at runtime (the compiler specializes generics) but saves weeks of refactoring.

Architect Must-Know: Serialization Strategy at Scale

At L6/Staff level, serialization isn't a per-model decision — it's an architectural layer. The right approach uses multiple formats, each optimized for its context:

  • API boundary: JSON with .convertFromSnakeCase — universal compatibility, easy debugging, works with every backend
  • Disk cache: Binary plist or Protobuf — 2–4x faster reads than JSON, smaller on disk, no human readability needed
  • Real-time pipelines: FlatBuffers — zero-copy means zero GC pressure, critical for 60fps rendering with server-driven data
  • Analytics / telemetry: MessagePack — schema-less like JSON but 30–50% smaller, reducing bandwidth for high-volume event streams
  • Cross-platform shared models: Protobuf — single .proto file generates Swift, Kotlin, TypeScript; schema evolution is built-in

The key architectural pattern is a SerializationFormat protocol that your data layer depends on. This lets you:

  • Swap formats per use case without touching business logic
  • A/B test serialization performance in production
  • Gradually migrate from JSON to binary formats path-by-path
  • Unit test with in-memory formats while production uses disk-optimized ones

In interviews, demonstrate that you think about serialization as a system-wide concern with measurable performance impact, not just "add Codable conformance." Know the numbers: protobuf is 3–10x smaller than JSON, FlatBuffers has zero decode cost, and Foundation's JSONDecoder has a hidden ObjC bridge penalty that pure-Swift alternatives eliminate.
