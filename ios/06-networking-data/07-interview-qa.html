URLSession & Networking (Questions 1–12)

Q1: Explain the difference between URLSessionConfiguration.default, .ephemeral, and .background. When would you choose each? Senior

.default creates a configuration that uses the shared cookie, cache, and credential stores persisted to disk. It is the standard choice for most API calls where you want standard HTTP caching behavior and cookies to persist across app launches.

.ephemeral stores all session data in RAM only. Cookies, cache, and credentials are never written to disk and are destroyed when the session is invalidated. Use this for privacy-sensitive requests (e.g., incognito browsing), or when you need a clean slate without affecting the shared stores.

.background enables out-of-process transfers managed by the system daemon nsurlsessiond. The system can schedule, pause, and resume transfers even after the app is terminated. You must provide a unique identifier string so the system can reconnect the session on relaunch.

Key differences in practice:

  • .background only supports upload/download tasks — data tasks are not allowed
  • .ephemeral has a separate in-memory cache, so it won't benefit from previously cached responses
  • .default shares the on-disk cache with other default sessions, which can cause subtle invalidation issues if multiple sessions write to the same cache
let defaultConfig = URLSessionConfiguration.default
defaultConfig.timeoutIntervalForRequest = 30
defaultConfig.waitsForConnectivity = true

let ephemeralConfig = URLSessionConfiguration.ephemeral

let bgConfig = URLSessionConfiguration.background(
    withIdentifier: "com.app.backgroundSync"
)
bgConfig.isDiscretionary = true
bgConfig.sessionSendsLaunchEvents = true
Q2: How does HTTP/2 multiplexing work in URLSession? What are the performance implications? L6

HTTP/2 multiplexing allows multiple request-response pairs to share a single TCP connection by interleaving binary frames identified by stream IDs. URLSession enables this automatically when the server supports HTTP/2 via ALPN negotiation during the TLS handshake.

How it works internally: URLSession maintains a connection pool keyed by (host, port, scheme). When a second request targets the same origin, URLSession reuses the existing HTTP/2 connection instead of opening a new TCP+TLS handshake. Each request gets its own stream ID, and frames from different streams are interleaved on the wire.

Performance implications:

  • Eliminates head-of-line blocking at the HTTP layer — one slow response doesn't block others (though TCP-level HOL blocking still exists, which HTTP/3 with QUIC solves)
  • Reduces connection setup cost — a single TLS handshake (1-2 RTT) serves all requests to that host
  • Server push — the server can proactively send resources the client hasn't requested yet
  • Header compression via HPACK reduces redundant header transmission

You can control the multiplexing behavior via httpMaximumConnectionsPerHost on the configuration. Setting it to 1 forces all requests through a single connection. The default is 6 for HTTP/1.1 but effectively 1 for HTTP/2 since multiplexing makes additional connections unnecessary.

Caveat: If you set httpMaximumConnectionsPerHost higher than 1 with HTTP/2, URLSession may open multiple connections, which wastes resources. The system is generally smart about this, but it's worth verifying with Instruments' Network profiler.

Q3: How would you implement certificate pinning? What's the difference between certificate pinning and public key pinning? L6

Certificate pinning validates the server's exact certificate (or one in the chain) against a known copy bundled in the app. Public key pinning extracts and compares only the Subject Public Key Info (SPKI), ignoring the certificate's expiry, issuer, and other metadata.

Public key pinning is preferred because it survives certificate rotation — when the server renews its certificate with the same key pair, the pin still matches. Certificate pinning breaks on every renewal, requiring an app update.

Implementation via URLSessionDelegate:

func urlSession(
    _ session: URLSession,
    didReceive challenge: URLAuthenticationChallenge,
    completionHandler: @escaping (URLSession.AuthChallengeDisposition, URLCredential?) -> Void
) {
    guard let serverTrust = challenge.protectionSpace.serverTrust,
          let serverCert = SecTrustCopyCertificateChain(serverTrust)?
              .first as? SecCertificate else {
        completionHandler(.cancelAuthenticationChallenge, nil)
        return
    }

    // Extract server's public key
    guard let serverKey = SecCertificateCopyKey(serverCert),
          let serverKeyData = SecKeyCopyExternalRepresentation(
              serverKey, nil
          ) as? Data else {
        completionHandler(.cancelAuthenticationChallenge, nil)
        return
    }

    let pinnedKeyHash = "sha256/AAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAA="
    let serverKeyHash = serverKeyData.sha256Base64()

    if serverKeyHash == pinnedKeyHash {
        let credential = URLCredential(trust: serverTrust)
        completionHandler(.useCredential, credential)
    } else {
        completionHandler(.cancelAuthenticationChallenge, nil)
    }
}
Pinning Risks

Always pin at least two keys (primary + backup) and have a remote kill switch. If your only pinned key is compromised or rotated, users are locked out until they update the app. Consider using a remote configuration to update pins without an app release.

Q4: How do background URLSession transfers survive app termination? Walk through the full lifecycle. L6

Background transfers are managed by the system daemon nsurlsessiond, which runs independently of your app process. Here is the full lifecycle:

  1. Task creation: Your app creates a background session with a unique identifier and enqueues download/upload tasks. The tasks are serialized and handed to the daemon.
  2. App suspension/termination: The system may suspend or terminate your app. The daemon continues the transfers using its own network stack.
  3. Transfer completion: When a transfer finishes, the daemon stores the result (downloaded file in a temp location, or error) and signals the system to relaunch your app.
  4. App relaunch: The system calls application(_:handleEventsForBackgroundURLSession:completionHandler:) on your AppDelegate. You must store the completion handler.
  5. Session reconnection: You recreate the URLSession with the same identifier. The system delivers all pending delegate callbacks (didFinishDownloading, didCompleteWithError, etc.).
  6. Finalization: After all events are delivered, urlSessionDidFinishEvents(forBackgroundURLSession:) is called. You invoke the stored completion handler to tell the system you're done, allowing it to snapshot the UI.
// AppDelegate
var backgroundCompletionHandlers: [String: () -> Void] = [:]

func application(
    _ application: UIApplication,
    handleEventsForBackgroundURLSession identifier: String,
    completionHandler: @escaping () -> Void
) {
    backgroundCompletionHandlers[identifier] = completionHandler
    // Recreate the session — delegate callbacks will fire
    _ = BackgroundSessionManager.shared.session(for: identifier)
}

// URLSessionDelegate
func urlSessionDidFinishEvents(
    forBackgroundURLSession session: URLSession
) {
    guard let id = session.configuration.identifier,
          let handler = backgroundCompletionHandlers.removeValue(forKey: id)
    else { return }
    DispatchQueue.main.async { handler() }
}

Key constraints: Only upload and download tasks are supported (not data tasks). The system may defer discretionary transfers to optimize battery. The isDiscretionary flag lets the system choose optimal timing (Wi-Fi, charging).

Q5: How does URLProtocol work and how would you use it for testing? Senior

URLProtocol is an abstract class that lets you intercept and handle URL loading at the protocol level. When registered, every URLSession request passes through your subclass's canInit(with:) method, giving you the chance to intercept it.

For testing, you create a mock protocol that returns canned responses:

final class MockURLProtocol: URLProtocol {
    static var requestHandler: ((URLRequest) throws -> (HTTPURLResponse, Data))?

    override class func canInit(with request: URLRequest) -> Bool {
        return true
    }

    override class func canonicalRequest(for request: URLRequest) -> URLRequest {
        return request
    }

    override func startLoading() {
        guard let handler = Self.requestHandler else {
            client?.urlProtocol(self, didFailWithError: URLError(.unknown))
            return
        }
        do {
            let (response, data) = try handler(request)
            client?.urlProtocol(self, didReceive: response, cacheStoragePolicy: .notAllowed)
            client?.urlProtocol(self, didLoad: data)
            client?.urlProtocolDidFinishLoading(self)
        } catch {
            client?.urlProtocol(self, didFailWithError: error)
        }
    }

    override func stopLoading() {}
}

// Usage in tests
let config = URLSessionConfiguration.ephemeral
config.protocolClasses = [MockURLProtocol.self]
let session = URLSession(configuration: config)

Advantages over mocking URLSession directly: URLProtocol intercepts at a lower level, so your production networking code runs unchanged. You don't need to inject a session protocol or abstract away URLSession. It also works with third-party libraries that use URLSession internally.

Limitation: URLProtocol.registerClass only works with the shared session and default configurations. For custom sessions, you must set protocolClasses on the configuration before creating the session.

Q6: Explain URLCache policies and how Cache-Control headers interact with them. Senior

URLSession's caching is controlled by two layers: the request's cache policy and the server's Cache-Control headers.

Request cache policies (URLRequest.CachePolicy):

  • .useProtocolCachePolicy (default) — follows HTTP semantics: checks Cache-Control, Expires, ETag, Last-Modified
  • .reloadIgnoringLocalCacheData — always fetches from the server, ignoring cache
  • .returnCacheDataElseLoad — returns cached response regardless of age; only fetches if no cache exists
  • .returnCacheDataDontLoad — returns cached response or fails; never hits the network (useful for offline mode)

Cache-Control header directives:

  • max-age=3600 — response is fresh for 3600 seconds
  • no-cache — must revalidate with server before using cached copy (sends If-None-Match / If-Modified-Since)
  • no-store — response must not be cached at all
  • private — only the end client can cache (not shared proxies)
  • must-revalidate — once stale, must revalidate before use

Interaction: When using .useProtocolCachePolicy, URLSession checks if the cached response is still fresh per max-age or Expires. If fresh, it returns the cached response without a network request. If stale, it sends a conditional request with If-None-Match (ETag) or If-Modified-Since. A 304 response means the cache is still valid.

The URLCache instance can be configured per-session via urlCache on the configuration. You can set memory and disk capacity limits. The default shared cache is 4MB memory / 20MB disk.

Q7: What is App Transport Security and when would you add exceptions? Senior

App Transport Security (ATS) is a security feature that enforces best-practice HTTPS connections. By default, it requires:

  • TLS 1.2 or later
  • Forward secrecy cipher suites (ECDHE)
  • Certificates signed with SHA-256 or better, with RSA key >= 2048 bits or ECC key >= 256 bits

Legitimate exceptions:

  • Local network: Communicating with hardware devices on the local network (printers, IoT) that don't support modern TLS
  • Legacy servers: Third-party APIs you don't control that haven't upgraded (add domain-specific exceptions, not blanket disabling)
  • Media streaming: Some CDNs serve media over HTTP for performance; use NSExceptionAllowsInsecureHTTPLoads for specific domains only
  • Web content: WKWebView loading arbitrary user-provided URLs may need NSAllowsArbitraryLoadsInWebContent

App Review will reject apps that disable ATS entirely (NSAllowsArbitraryLoads = YES) without justification. Always use the most specific exception possible — per-domain rather than global.

Q8: How would you implement retry with exponential backoff in a network layer? Senior

Exponential backoff increases the delay between retries exponentially (1s, 2s, 4s, 8s…) to avoid overwhelming a struggling server. Adding jitter (randomization) prevents thundering herd problems when many clients retry simultaneously.

func performWithRetry<T>(
    maxRetries: Int = 3,
    baseDelay: TimeInterval = 1.0,
    maxDelay: TimeInterval = 30.0,
    operation: () async throws -> T
) async throws -> T {
    var lastError: Error?

    for attempt in 0...maxRetries {
        do {
            return try await operation()
        } catch let error as URLError where error.isRetryable {
            lastError = error
            if attempt < maxRetries {
                let exponentialDelay = baseDelay * pow(2.0, Double(attempt))
                let jitter = Double.random(in: 0...0.5) * exponentialDelay
                let delay = min(exponentialDelay + jitter, maxDelay)
                try await Task.sleep(for: .seconds(delay))
            }
        }
    }
    throw lastError!
}

extension URLError {
    var isRetryable: Bool {
        switch code {
        case .timedOut, .networkConnectionLost,
             .notConnectedToInternet, .cannotConnectToHost:
            return true
        default:
            return false
        }
    }
}

Important considerations:

  • Only retry on transient errors (timeouts, connection lost) — not on 4xx client errors
  • Respect Retry-After headers from the server
  • For HTTP 429 (Too Many Requests), use the server-specified delay
  • Make retries cancellable via Task.checkCancellation()
  • Only retry idempotent requests (GET, PUT, DELETE) — retrying POST can cause duplicates
Q9: How does request deduplication work in a network layer? L6

Request deduplication ensures that multiple callers requesting the same resource simultaneously share a single in-flight network request rather than firing duplicate requests. This is critical for scenarios like multiple UI components loading the same image or API response.

The pattern uses a dictionary of in-flight continuations keyed by a request identifier:

actor RequestDeduplicator {
    private var inFlight: [String: Task<Data, Error>] = [:]

    func deduplicated(
        key: String,
        operation: @Sendable @escaping () async throws -> Data
    ) async throws -> Data {
        if let existing = inFlight[key] {
            return try await existing.value
        }

        let task = Task {
            defer { Task { await removeInFlight(key: key) } }
            return try await operation()
        }

        inFlight[key] = task
        return try await task.value
    }

    private func removeInFlight(key: String) {
        inFlight[key] = nil
    }
}

Key design decisions:

  • The deduplication key should include the URL, significant headers, and any parameters that affect the response
  • Use an actor to ensure thread-safe access to the in-flight dictionary
  • Clean up the entry after the task completes so subsequent requests trigger a fresh fetch
  • Consider whether cancellation of one caller should cancel the shared task (usually not — only cancel when all callers cancel)
Q10: Explain waitsForConnectivity and how it differs from reachability checks. Senior

waitsForConnectivity is a property on URLSessionConfiguration (iOS 11+) that tells the session to wait for network connectivity rather than immediately failing with a .notConnectedToInternet error. When set to true, the session queues the request and fires it when connectivity is restored.

The delegate method urlSession(_:taskIsWaitingForConnectivity:) is called when a task enters the waiting state, letting you update the UI (e.g., show an offline banner).

How it differs from reachability (NWPathMonitor):

  • Reachability is advisory — it tells you the current network state but doesn't guarantee a request will succeed. A device can be "reachable" but behind a captive portal.
  • waitsForConnectivity is actionable — it directly affects request behavior. The system handles the retry timing internally.
  • No polling — unlike reachability-based patterns where you check status and retry manually, waitsForConnectivity is event-driven at the system level.

Best practice: Use waitsForConnectivity for the network layer's behavior and NWPathMonitor for UI updates (showing offline indicators). Don't use reachability to gate network requests — just make the request and handle errors.

let config = URLSessionConfiguration.default
config.waitsForConnectivity = true
config.timeoutIntervalForResource = 300 // max wait time

let session = URLSession(configuration: config, delegate: self, delegateQueue: nil)

// Delegate callback
func urlSession(
    _ session: URLSession,
    taskIsWaitingForConnectivity task: URLSessionTask
) {
    DispatchQueue.main.async {
        self.showOfflineBanner()
    }
}
Q11: How does URLSessionWebSocketTask compare to third-party WebSocket libraries? Senior

URLSessionWebSocketTask (iOS 13+) provides native WebSocket support built on top of URLSession's infrastructure. It handles the HTTP upgrade handshake, frame parsing, ping/pong, and close handshake internally.

Advantages over third-party libraries:

  • No additional dependency — ships with the OS
  • Inherits URLSession features: proxy support, ATS, authentication challenges, metrics
  • Automatic TLS via the system's TLS stack
  • Integrates with Network.framework for path monitoring

Limitations compared to libraries like Starscream:

  • No automatic reconnection — you must implement reconnection logic yourself
  • No built-in heartbeat management beyond manual ping/pong
  • The receive API is pull-based (you must call receive repeatedly), not push-based (no delegate callback for incoming messages)
  • Limited control over compression (permessage-deflate)
  • No support for custom subprotocols negotiation in early versions

            }
            self?.receiveLoop() // must re-register
        case .failure(let error):
            self?.handleDisconnect(error)
        }
    }
}
Q12: Design a network layer that handles auth token refresh transparently. Architect

The core challenge is handling concurrent requests that all fail with 401 simultaneously — you need to refresh the token once and retry all queued requests with the new token, without triggering multiple refresh calls.

actor AuthenticatedNetworkClient {
    private let session: URLSession
    private let tokenStore: TokenStore
    private var refreshTask: Task<Token, Error>?

    func perform(_ request: URLRequest) async throws -> Data {
        var authedRequest = request
        let token = try await tokenStore.currentToken()
        authedRequest.setValue(
            "Bearer \(token.accessToken)",
            forHTTPHeaderField: "Authorization"
        )

        let (data, response) = try await session.data(for: authedRequest)

        if let httpResponse = response as? HTTPURLResponse,
           httpResponse.statusCode == 401 {
            let newToken = try await refreshTokenOnce()
            authedRequest.setValue(
                "Bearer \(newToken.accessToken)",
                forHTTPHeaderField: "Authorization"
            )
            let (retryData, _) = try await session.data(for: authedRequest)
            return retryData
        }

        return data
    }

    private func refreshTokenOnce() async throws -> Token {
        if let existingTask = refreshTask {
            return try await existingTask.value
        }

        let task = Task { [tokenStore] in
            let newToken = try await tokenStore.refresh()
            return newToken
        }

        refreshTask = task
        defer { refreshTask = nil }
        return try await task.value
    }
}
Architecture Note

Key design decisions: (1) Using an actor ensures thread-safe access to refreshTask. (2) The refreshTokenOnce method coalesces concurrent refresh attempts — if a refresh is already in flight, subsequent callers await the same task. (3) The defer clears the task after completion so future 401s trigger a fresh refresh. (4) Consider adding a retry limit to prevent infinite refresh loops when the refresh token itself is expired.

Core Data (Questions 13–22)

Q13: Explain the Core Data stack components and their responsibilities. Senior

The Core Data stack consists of four main components:

  • NSManagedObjectModel: The compiled .xcdatamodeld schema. Defines entities, attributes, relationships, and fetch request templates. Loaded once and shared across the stack.
  • NSPersistentStoreCoordinator: The bridge between the object graph and the persistent store. It serializes access to the underlying SQLite database (or other store types), handles mapping between the object model and the store format, and manages multiple stores simultaneously.
  • NSManagedObjectContext: A scratchpad for working with managed objects. It tracks changes (inserts, updates, deletes), manages the object lifecycle, and provides undo support. Contexts can be nested in parent-child relationships.
  • NSPersistentContainer: A convenience wrapper (iOS 10+) that sets up the entire stack. It creates the model, coordinator, and a viewContext (main queue context), and provides newBackgroundContext() and performBackgroundTask for background work.
let container = NSPersistentContainer(name: "Model")
container.loadPersistentStores { description, error in
    if let error { fatalError("Store failed: \(error)") }
}
container.viewContext.automaticallyMergesChangesFromParent = true
container.viewContext.mergePolicy = NSMergeByPropertyObjectTrumpMergePolicy

The coordinator serializes store access, so even though multiple contexts can exist, writes to the SQLite file are serialized. This is why batch operations that bypass the context (like NSBatchDeleteRequest) are significantly faster — they go directly to the store.

Q14: What happens when you call save() on a child context vs a root context? L6

Core Data contexts can be arranged in a parent-child hierarchy. The behavior of save() depends on where the context sits in this hierarchy:

Child context save(): Pushes changes up to the parent context's in-memory state. No disk I/O occurs. The parent context receives the changes as if they were made directly on it. The changes are not persisted until the root context saves.

Root context save(): Sends changes through the NSPersistentStoreCoordinator to the underlying SQLite store. This involves SQL generation, WAL writes, and actual disk I/O.

The typical pattern for background imports:

// Root context: private queue, connected to coordinator
let rootContext = NSManagedObjectContext(concurrencyType: .privateQueueConcurrencyType)
rootContext.persistentStoreCoordinator = coordinator

// Main context: main queue, child of root
let mainContext = NSManagedObjectContext(concurrencyType: .mainQueueConcurrencyType)
mainContext.parent = rootContext

// Worker context: private queue, child of main
let workerContext = NSManagedObjectContext(concurrencyType: .privateQueueConcurrencyType)
workerContext.parent = mainContext

// Save chain: worker → main → root → disk
workerContext.perform {
    // Make changes...
    try workerContext.save()  // pushes to mainContext
    mainContext.perform {
        try mainContext.save()   // pushes to rootContext
        rootContext.perform {
            try rootContext.save() // writes to disk
        }
    }
}

Gotcha: If you use NSPersistentContainer's newBackgroundContext(), the returned context is a sibling of viewContext, not a child. Both connect directly to the coordinator. Changes are merged via NSManagedObjectContextDidSave notifications when automaticallyMergesChangesFromParent is enabled.

Q15: How does Core Data's concurrency model work? What are perform and performAndWait? L6

Every NSManagedObjectContext is bound to a specific queue (main or private). You must access the context and its managed objects only from that queue. The perform and performAndWait methods ensure this:

  • perform(_:): Asynchronously enqueues the block on the context's queue. Returns immediately.
  • performAndWait(_:): Synchronously executes the block on the context's queue. Blocks the calling thread until complete. It is reentrant — if called from the context's own queue, the block executes immediately without deadlocking.

Why reentrancy matters: performAndWait uses a special "perform" queue marker. If the current thread already owns the context's queue, the block runs inline. This prevents the deadlock that would occur with a naive sync dispatch.

Common violations:

  • Passing managed objects between threads — always pass NSManagedObjectID and re-fetch in the target context
  • Accessing a background context's objects on the main thread
  • Forgetting that relationship traversal triggers faulting, which requires the correct queue

Enable -com.apple.CoreData.ConcurrencyDebug 1 in scheme arguments to catch violations at runtime — it triggers an assertion on incorrect queue access.

Q16: What is faulting and how does it affect performance? Senior

A fault is a placeholder object that represents a managed object whose attribute data hasn't been loaded from the store yet. When you execute a fetch request, Core Data returns fault objects by default — lightweight shells that know their objectID and entity but haven't materialized their properties.

Fault firing: When you access any attribute on a fault, Core Data transparently fetches the full row from SQLite. This is called "firing" or "fulfilling" the fault. The context caches the materialized data, so subsequent accesses don't hit the store again.

Performance implications:

  • N+1 problem: Iterating over 1000 faulted objects and accessing a property on each triggers 1000 individual SQL queries. Use relationshipKeyPathsForPrefetching or batch faulting to mitigate.
  • Memory efficiency: Faults use minimal memory. You can turn materialized objects back into faults with refresh(_:mergeChanges:) to reclaim memory.
  • Relationship faults: Relationships are also faulted. Accessing order.items fires a fault that fetches all related items. Use fetchBatchSize to load relationships in batches.
let request = Item.fetchRequest()
request.fetchBatchSize = 20
request.relationshipKeyPathsForPrefetching = ["category"]
request.returnsObjectsAsFaults = false // fully materialize

// propertiesToFetch for partial materialization
request.propertiesToFetch = ["name", "price"]
request.resultType = .dictionaryResultType
Q17: Explain NSMergePolicy types and when to use each. L6

Merge policies resolve conflicts when the same object has been modified in both the context and the persistent store (or parent context) since the last fetch. The five built-in policies are:

  • NSErrorMergePolicy (default): Fails the save with an error. Forces you to handle conflicts explicitly. Best for data integrity when you can't afford silent data loss.
  • NSMergeByPropertyStoreTrumpMergePolicy: Store wins on a per-property basis. If the store changed "name" and the context changed "email", both changes are kept. If both changed "name", the store's value wins.
  • NSMergeByPropertyObjectTrumpMergePolicy: Context (in-memory) wins on a per-property basis. Most commonly used — the user's latest edits take priority.
  • NSOverwriteMergePolicy: Context wins entirely — all properties from the context overwrite the store, even unchanged ones.
  • NSRollbackMergePolicy: Store wins entirely — discards all in-memory changes for conflicting objects.

When to use each:

  • CloudKit sync: NSMergeByPropertyObjectTrumpMergePolicy — local edits should win since the user just made them
  • Background import: NSMergeByPropertyStoreTrumpMergePolicy — the import should defer to existing user edits
  • Financial data: NSErrorMergePolicy — detect and resolve conflicts explicitly to prevent data corruption
Q18: How does NSFetchedResultsController work internally? Senior

NSFetchedResultsController (FRC) bridges Core Data and UITableView/UICollectionView by monitoring context changes and translating them into precise UI updates (insert, delete, move, update).

Internal mechanism:

  1. Initial fetch: performFetch() executes the fetch request and caches the results. If sectionNameKeyPath is set, it groups results into sections.
  2. Change observation: FRC registers for NSManagedObjectContextObjectsDidChange notifications on its context.
  3. Change processing: When the context changes, FRC re-evaluates which objects match the fetch predicate, checks sort order, and computes the diff between old and new result sets.
  4. Delegate callbacks: It calls controllerWillChangeContent, then individual didChange calls for each insert/delete/move/update, then controllerDidChangeContent.

Performance considerations:

  • FRC uses fetchBatchSize internally to avoid loading all objects into memory
  • The cacheName parameter persists section info to disk, avoiding recomputation on relaunch
  • For large datasets, ensure your sort descriptors and section key paths are backed by indexed attributes
  • In iOS 13+, use NSDiffableDataSourceSnapshot with FRC for smoother animations
Q19: What's the difference between lightweight and heavyweight migration? Senior

Lightweight migration is automatic schema migration that Core Data can infer without any custom code. It handles:

  • Adding/removing attributes (with default values for new non-optional attributes)
  • Making attributes optional/non-optional
  • Renaming entities or attributes (using renamingIdentifier in the model)
  • Adding/removing relationships
  • Changing relationship cardinality (to-one ↔ to-many)

Heavyweight (custom) migration requires an NSMappingModel that explicitly defines how to transform data from the source to the destination model. You need this when:

  • Splitting one entity into two (e.g., separating Address from User)
  • Merging entities
  • Transforming attribute values (e.g., combining firstName + lastName into fullName)
  • Complex relationship restructuring
// Enable lightweight migration
let description = NSPersistentStoreDescription()
description.shouldMigrateStoreAutomatically = true
description.shouldInferMappingModelAutomatically = true
container.persistentStoreDescriptions = [description]

Staged migration (iOS 17+): NSStagedMigrationManager lets you define a sequence of migration stages, each with its own lightweight or custom step. This is more maintainable than a single monolithic mapping model for apps with many model versions.

Q20: How do NSBatchInsertRequest/NSBatchDeleteRequest bypass the context? L6

Batch operations execute SQL directly against the persistent store, bypassing the NSManagedObjectContext entirely. This means:

  • No validation — Core Data validation rules are not checked
  • No change tracking — the context doesn't know about the changes
  • No notifications — NSManagedObjectContextDidSave is not posted
  • No cascade — delete rules (cascade, nullify) are not applied
  • No faulting overhead — objects are never materialized in memory

This makes them orders of magnitude faster for bulk operations. Inserting 100K records via batch insert takes seconds vs minutes through the context.

// Batch insert with dictionary provider
let batchInsert = NSBatchInsertRequest(
    entity: Item.entity(),
    dictionaryHandler: { dict in
        guard index < items.count else { return true }
        dict["name"] = items[index].name
        dict["price"] = items[index].price
        index += 1
        return false
    }
)
batchInsert.resultType = .objectIDs

Q21: How does NSPersistentCloudKitContainer sync with CloudKit? L6

NSPersistentCloudKitContainer automatically mirrors your Core Data store to CloudKit. It manages the full sync lifecycle:

  1. Schema mirroring: Core Data entities map to CloudKit record types. Attributes become record fields. Relationships become CKReference fields.
  2. Change tracking: A hidden "mirror" table in the SQLite store tracks which objects have been modified since the last sync. Each row stores the object ID, change type, and a transaction counter.
  3. Export: On save, modified objects are queued for export. The container batches changes into CKModifyRecordsOperation calls, handling the 400-record-per-operation CloudKit limit.
  4. Import: The container uses CKFetchDatabaseChangesOperation and CKFetchRecordZoneChangesOperation to pull remote changes. It uses server change tokens to fetch only new changes.
  5. Conflict resolution: Uses the context's merge policy. NSMergeByPropertyObjectTrumpMergePolicy is recommended — local edits win.

Limitations:

  • Unique constraints are not supported (CloudKit has no equivalent)
  • Ordered relationships are not synced
  • Binary data with "allows external storage" uses CKAsset, which has a 50MB limit
  • Initial sync of large datasets can be very slow
  • No real-time sync — relies on push notifications and periodic polling
Q22: Design a Core Data stack for an app with 1M+ records. Architect

At this scale, naive Core Data usage will cause memory pressure, slow fetches, and UI jank. Here's the architecture:

  • Use fetchLimit and fetchOffset for pagination
  • Index frequently queried attributes in the model editor
  • Use propertiesToFetch with dictionaryResultType for list views that only need a few fields
  • Use NSFetchedResultsController with section caching for table views

  • Use NSBatchDeleteRequest for bulk deletes

For read-heavy workloads with complex queries, direct SQLite (via GRDB or SQLite.swift) may outperform Core Data due to the overhead of the object graph, faulting, and change tracking. Core Data excels at object graph management; raw SQLite excels at large-scale data processing.

Q23: What does the @Model macro generate? L6

The @Model macro transforms a plain Swift class into a SwiftData-managed persistent model. At compile time, it generates:

  • PersistentModel conformance: The class conforms to PersistentModel (SwiftData's equivalent of NSManagedObject), which includes a persistentBackingData property for the underlying storage.
  • Property accessors: Each stored property is rewritten with custom getters and setters that go through the backing store. This is similar to Core Data's @NSManaged but implemented via macros instead of the Objective-C runtime.
  • Schema metadata: A static schemaMetadata array describing each property's name, type, and attributes (optional, unique, relationship info).
  • Observable conformance: The class conforms to Observable (Observation framework), so SwiftUI views automatically track which properties they access and re-render only when those specific properties change.
  • Hashable/Identifiable: Based on the persistent model ID.
// What you write
@Model
final class Item {
    var name: String
    var timestamp: Date
    @Relationship(deleteRule: .cascade)
    var tags: [Tag]
}

// What the macro generates (simplified)
final class Item: PersistentModel, Observable {
    var name: String {
        get { _$observationRegistrar.access(self, keyPath: \.name)
              return getValue(forKey: \.name) }
        set { _$observationRegistrar.withMutation(of: self, keyPath: \.name) {
              setValue(forKey: \.name, to: newValue) } }
    }
    // ... similar for other properties
}
Q24: How does @Query work under the hood? L6

@Query is a property wrapper that automatically fetches and observes SwiftData models within a SwiftUI view. Under the hood:

  1. ModelContext discovery: @Query reads the ModelContext from the SwiftUI environment (injected via .modelContainer()). It uses the @Environment(\.modelContext) path internally.
  2. Fetch execution: On first access, it creates a FetchDescriptor from the provided predicate, sort descriptors, and animation, then executes it against the context.
  3. Change observation: It registers for model context change notifications. When the context saves or merges changes, @Query re-executes the fetch and triggers a SwiftUI view update if the results changed.
  4. Diffing: Results are compared to avoid unnecessary view updates. Only actual changes to matching objects trigger re-renders.
struct ItemListView: View {
    @Query(
        filter: #Predicate<Item> { $0.isActive },
        sort: \.timestamp,
        order: .reverse,
        animation: .default
    )
    private var items: [Item]

    var body: some View {
        List(items) { item in
            Text(item.name)
        }
    }
}

Key difference from NSFetchedResultsController: @Query is declarative and tightly integrated with SwiftUI's view lifecycle. You don't manage delegate callbacks or manually apply diffs. However, it lacks FRC's section support and fine-grained change types (insert/delete/move).

Q26: How does SwiftData handle migrations? Senior

SwiftData uses a SchemaMigrationPlan to define ordered migration stages between schema versions:

enum ItemMigrationPlan: SchemaMigrationPlan {
    static var schemas: [any VersionedSchema.Type] = [
        SchemaV1.self,
        SchemaV2.self,
        SchemaV3.self
    ]

    static var stages: [MigrationStage] = [
        .lightweight(fromVersion: SchemaV1.self, toVersion: SchemaV2.self),
        .custom(
            fromVersion: SchemaV2.self,
            toVersion: SchemaV3.self,
            willMigrate: { context in
                // Transform data before schema change
                let items = try context.fetch(FetchDescriptor<SchemaV2.Item>())
                for item in items {
                    item.fullName = "\(item.firstName) \(item.lastName)"
                }
                try context.save()
            },
            didMigrate: nil
        )
    ]
}

enum SchemaV1: VersionedSchema {
    static var versionIdentifier = Schema.Version(1, 0, 0)
    static var models: [any PersistentModel.Type] = [Item.self]

    @Model final class Item {
        var name: String
    }
}

Key points: Each VersionedSchema contains a snapshot of the model classes at that version. SwiftData walks the stages array in order, applying lightweight or custom migrations sequentially. This is conceptually similar to Core Data's staged migration but with a more Swift-native API.

Q27: What are SwiftData's current limitations? Senior

As of iOS 17/18, SwiftData has several notable limitations compared to Core Data:

  • No parent-child contexts — you can't create a child context for discard-able edits (e.g., an edit sheet where the user can cancel)
  • Limited batch operations — no equivalent to NSBatchInsertRequest for bulk inserts that bypass the context
  • No NSFetchedResultsController equivalent with sections — @Query doesn't support section grouping
  • CloudKit sync is less mature — fewer configuration options than NSPersistentCloudKitContainer
  • No abstract entities — entity inheritance is limited
  • Background context ergonomics — creating and using background ModelContext instances requires manual ModelActor setup
  • Predicate limitations — #Predicate doesn't support all the operations that NSPredicate does (e.g., subqueries, SUBQUERY, certain aggregate functions)
  • Performance — for very large datasets, SwiftData's overhead is higher than Core Data's optimized C/Obj-C internals
  • No cross-platform — SwiftData is Apple-only, while Core Data at least has some cross-compilation potential
Q28: When would you choose SwiftData over Core Data? Architect

Choose SwiftData when:

  • Building a new SwiftUI-first app targeting iOS 17+
  • Your data model is relatively simple (no complex inheritance, no need for abstract entities)
  • You want tight SwiftUI integration via @Query and @Model
  • You prefer Swift-native APIs over Objective-C patterns
  • Your dataset is moderate (thousands to tens of thousands of records)
  • You want automatic observation without manual NSFetchedResultsController setup

Choose Core Data when:

  • Supporting iOS versions before 17
  • You need batch operations for large-scale data processing
  • Complex merge policies and conflict resolution are required
  • You need parent-child contexts for transactional editing
  • CloudKit sync with fine-grained control is needed
  • Performance at scale (1M+ records) is critical
  • You have an existing Core Data codebase — migration risk may not be worth it
Pragmatic Advice

SwiftData is built on top of Core Data. In an interview, demonstrating knowledge of both — and articulating when each is appropriate — shows more maturity than advocating for one exclusively.

Q29: How does the #Predicate macro work? L6

#Predicate is a Swift macro that transforms a closure-like expression into a Predicate<T> value at compile time. Unlike NSPredicate (which parses format strings at runtime), #Predicate is fully type-checked by the compiler.

How it works:

  1. The macro receives the closure's AST (abstract syntax tree) at compile time
  2. It transforms property accesses into key path expressions
  3. Operators (==, >, &&, ||) are converted into PredicateExpression nodes
  4. The result is a Predicate<T> that can be converted to an NSPredicate for Core Data's SQLite query generation
let minPrice: Double = 10.0
let predicate = #Predicate<Item> { item in
    item.price > minPrice &&
    item.name.contains("Pro") &&
    item.isActive == true
}

let descriptor = FetchDescriptor<Item>(
    predicate: predicate,
    sortBy: [SortDescriptor(\.timestamp, order: .reverse)]
)

Advantages: Compile-time type safety (no runtime format string crashes), refactoring support (rename a property and the predicate updates), and autocompletion. Limitations: Cannot express everything NSPredicate can — no SUBQUERY, no custom functions, no regex matching (as of iOS 17).

Q30: Can SwiftData and Core Data coexist in the same app? Senior

Yes, SwiftData and Core Data can coexist because SwiftData is built on top of Core Data. They can even share the same underlying SQLite store. There are two approaches:

1. Shared store: Point both stacks at the same SQLite file. SwiftData's ModelContainer and Core Data's NSPersistentContainer can read/write the same database. You must ensure the schemas are compatible — the entity names and attributes must match.

2. Separate stores: Use different SQLite files for each stack. This is simpler but means data isn't automatically shared. You'd need to manually sync between them.

// SwiftData container pointing to existing Core Data store
let schema = Schema([Item.self])
let config = ModelConfiguration(
    schema: schema,
    url: coreDataStoreURL // same .sqlite file
)
let container = try ModelContainer(
    for: schema,
    configurations: [config]
)

Migration strategy: For existing apps, a gradual migration is recommended — keep Core Data for existing features and use SwiftData for new features. Over time, migrate screens one by one. Ensure you don't have both stacks writing to the same entities simultaneously without proper merge handling.

Keychain & Security (Questions 31–38)

Q31: How does the iOS Keychain work under the hood? L6

The iOS Keychain is a system-level encrypted database managed by the securityd daemon. Here's how it works internally:

  • Storage: Keychain items are stored in an SQLite database encrypted with a device-specific hardware key derived from the Secure Enclave. The database file is at /var/Keychains/keychain-2.db.
  • Encryption layers: Each item is encrypted with a per-item key, which is itself wrapped by a class key. The class key is protected by the device passcode and/or the Secure Enclave's UID key, depending on the accessibility setting.
  • Access control: The securityd daemon enforces access based on the app's keychain-access-groups entitlement. Each item has an access group, and only apps in that group can read it.
  • Persistence: Keychain items survive app deletion and reinstallation (unless the item's accessibility requires the device passcode and the user has reset it). This is different from the app sandbox, which is deleted on uninstall.

API flow: Your app calls SecItemAdd/SecItemCopyMatching → Security framework → XPC to securityd → SQLite query → decrypt with class key → return plaintext to your process.

Common Misconception

Keychain items persist across app reinstalls by default. If you store a "first launch" token in the Keychain, it will still be there after the user deletes and reinstalls the app. Use UserDefaults for first-launch detection instead.

Q32: Explain all kSecAttrAccessible options and when to use each. L6

The kSecAttrAccessible attribute controls when a Keychain item's class key is available for decryption:

  • kSecAttrAccessibleWhenUnlocked: Available only when the device is unlocked. The class key is discarded from memory when the device locks. Best for: most credentials, API tokens.
  • kSecAttrAccessibleWhenUnlockedThisDeviceOnly: Same as above, but the item is not included in encrypted backups or transferred to new devices. Best for: device-specific secrets, session tokens.
  • kSecAttrAccessibleAfterFirstUnlock: Available after the first unlock after boot. The class key remains in memory until the device restarts. Best for: background app refresh tokens, push notification credentials.
  • kSecAttrAccessibleAfterFirstUnlockThisDeviceOnly: Same as above, non-migratable. Best for: background sync tokens that shouldn't transfer to new devices.
  • kSecAttrAccessibleWhenPasscodeSetThisDeviceOnly: Only available when a passcode is set AND the device is unlocked. If the user removes their passcode, the item is deleted. Best for: biometric-protected items, highest security credentials.

Deprecated: kSecAttrAccessibleAlways and kSecAttrAccessibleAlwaysThisDeviceOnly are deprecated — they provided no protection since the key was always in memory.

Rule of thumb: Use the most restrictive option that still allows your app to function. If you need background access, use AfterFirstUnlock. If the item is only needed while the user is actively using the app, use WhenUnlocked.

Q33: How would you implement biometric-protected Keychain access? Senior

Biometric protection uses SecAccessControl with the .biometryCurrentSet or .biometryAny flag to require Face ID/Touch ID before the Keychain item can be accessed:

import LocalAuthentication
import Security

func storeBiometricProtectedItem(
    data: Data,
    account: String
) throws {
    var error: Unmanaged<CFError>?
    guard let access = SecAccessControlCreateWithFlags(
        kCFAllocatorDefault,
        kSecAttrAccessibleWhenPasscodeSetThisDeviceOnly,
        [.biometryCurrentSet, .privateKeyUsage],
        &error
    ) else {
        throw error!.takeRetainedValue() as Error
    }

    let query: [String: Any] = [
        kSecClass as String: kSecClassGenericPassword,
        kSecAttrAccount as String: account,
        kSecValueData as String: data,
        kSecAttrAccessControl as String: access,
        kSecUseAuthenticationUI as String: kSecUseAuthenticationUIAllow
    ]

    let status = SecItemAdd(query as CFDictionary, nil)
    guard status == errSecSuccess else {
        throw KeychainError.unhandled(status)
    }
}

func readBiometricProtectedItem(account: String) throws -> Data {
    let context = LAContext()
    context.localizedReason = "Access your credentials"

    let query: [String: Any] = [
        kSecClass as String: kSecClassGenericPassword,
        kSecAttrAccount as String: account,
        kSecReturnData as String: true,
        kSecUseAuthenticationContext as String: context
    ]

    var result: AnyObject?
    let status = SecItemCopyMatching(query as CFDictionary, &result)
    guard status == errSecSuccess,
          let data = result as? Data else {
        throw KeychainError.unhandled(status)
    }
    return data
}

Q37: What is App Attest and how does it work? L6

App Attest (DeviceCheck framework, iOS 14+) lets your server verify that requests come from a legitimate, unmodified instance of your app running on genuine Apple hardware. It combats fraud, bot traffic, and jailbreak-based tampering.

How it works:

  1. Key generation: The app calls DCAppAttestService.shared.generateKey, which creates a P256 key pair in the Secure Enclave. The private key never leaves the device.
  2. Attestation: The app sends the key ID and a server-generated challenge to attestKey(_:clientDataHash:). The Secure Enclave signs an attestation object containing the public key, app identity (team ID + bundle ID), and a certificate chain rooted at Apple's CA.
  3. Server verification: Your server validates the attestation certificate chain against Apple's root CA, verifies the app identity, and stores the public key.
  4. Assertion: For subsequent requests, the app calls generateAssertion(_:clientDataHash:) to sign a request-specific payload. The server verifies the signature against the stored public key and checks a monotonic counter to prevent replay attacks.

Limitations: Not available on simulators. Has a rate limit on attestation calls. The attestation is per-device-per-app-install — reinstalling the app requires re-attestation. Not foolproof against sophisticated attacks but raises the bar significantly.

Q38: Design a secure credential storage system for a banking app. Architect

A banking app requires defense-in-depth with multiple security layers:

1. Authentication tokens:

  • Store access tokens in Keychain with kSecAttrAccessibleWhenPasscodeSetThisDeviceOnly
  • Protect with SecAccessControl requiring .biometryCurrentSet
  • Use short-lived access tokens (5-15 min) with longer-lived refresh tokens
  • Refresh tokens should require biometric re-authentication after a configurable timeout

2. Encryption keys:

  • Generate a per-user encryption key in the Secure Enclave (SecureEnclave.P256)
  • Use this key to derive a symmetric key for encrypting sensitive local data
  • The Secure Enclave key never leaves the hardware

3. Local data protection:

  • All local files use .completeFileProtection
  • Core Data store uses NSPersistentStoreFileProtectionKey: NSFileProtectionComplete
  • Disable backups for sensitive data with excludedFromBackup

4. Network security:

  • Certificate pinning with backup pins and remote kill switch
  • App Attest for server-side request validation
  • All API calls over TLS 1.3 with ATS fully enabled

5. Runtime protection:

  • Jailbreak detection (check for Cydia, writable system paths, fork() behavior)
  • Debugger detection (sysctl P_TRACED flag)
  • Clear sensitive data from memory when the app backgrounds
  • Implement a secure screen (blur/hide content in app switcher)

No client-side security is unbreakable. The goal is to raise the cost of attack above the value of the data. Server-side validation (rate limiting, anomaly detection, device fingerprinting) is equally important. Design for the assumption that the client is compromised.

  • Error recovery: Handles save/load failures with user-presentable errors

  • Apps that need iCloud document sync
  • When you want automatic save/load lifecycle management
  • When multiple processes might access the same file (app + extension)

Q45: What does the compiler synthesize for Codable conformance? Senior

When a type conforms to Codable (which is a typealias for Encodable & Decodable) and all its stored properties are themselves Codable, the compiler synthesizes three things:

  1. CodingKeys enum: A nested enum conforming to CodingKey with a case for each stored property. The raw values match the property names by default.
  2. init(from decoder: Decoder): Creates a keyed container from the decoder and decodes each property by its coding key.
  3. encode(to encoder: Encoder): Creates a keyed container and encodes each property.
// What you write
struct User: Codable {
    let name: String
    let age: Int
}

// What the compiler generates
struct User: Codable {
    let name: String
    let age: Int

    private enum CodingKeys: String, CodingKey {
        case name
        case age
    }

    init(from decoder: Decoder) throws {
        let container = try decoder.container(keyedBy: CodingKeys.self)
        name = try container.decode(String.self, forKey: .name)
        age = try container.decode(Int.self, forKey: .age)
    }

    func encode(to encoder: Encoder) throws {
        var container = encoder.container(keyedBy: CodingKeys.self)
        try container.encode(name, forKey: .name)
        try container.encode(age, forKey: .age)
    }
}

Customization points: You can provide your own CodingKeys to rename keys, exclude properties, or change the coding structure. You can also write custom init(from:) and encode(to:) for complex transformations while keeping the synthesized CodingKeys.

Q46: How would you handle a JSON response where a key might be null vs missing? Senior

In JSON, there are three distinct states: a key with a value ("name": "Alice"), a key with null ("name": null), and a missing key. Swift's Codable treats null and missing the same way for Optional properties — both decode to nil.

To distinguish between null and missing, use a double-optional or a custom wrapper:

// Approach 1: Double optional
struct PatchRequest: Encodable {
    var name: String??

    func encode(to encoder: Encoder) throws {
        var container = encoder.container(keyedBy: CodingKeys.self)
        switch name {
        case .none:
            break // key omitted entirely
        case .some(.none):
            try container.encodeNil(forKey: .name) // "name": null
        case .some(.some(let value)):
            try container.encode(value, forKey: .name) // "name": "Alice"
        }
    }
}

// Approach 2: Explicit enum
enum NullableField<T: Codable>: Codable {
    case present(T)
    case null
    case absent

    init(from decoder: Decoder) throws {
        let container = try decoder.singleValueContainer()
        if container.decodeNil() {
            self = .null
        } else {
            self = .present(try container.decode(T.self))
        }
    }
}

This distinction matters for PATCH APIs where omitting a key means "don't change this field" and sending null means "clear this field."

Q47: How would you decode a polymorphic JSON array? L6

A polymorphic array contains objects of different types, typically distinguished by a "type" discriminator field. For example: [{"type": "text", "content": "..."}, {"type": "image", "url": "..."}].

protocol ContentBlock: Decodable {}

struct TextBlock: ContentBlock {
    let content: String
}

struct ImageBlock: ContentBlock {
    let url: URL
    let caption: String?
}

enum AnyContentBlock: Decodable {
    case text(TextBlock)
    case image(ImageBlock)
    case unknown

    private enum Discriminator: String, Decodable {
        case text, image
    }

    private enum CodingKeys: String, CodingKey {
        case type
    }

    init(from decoder: Decoder) throws {
        let container = try decoder.container(keyedBy: CodingKeys.self)
        let type = try container.decode(Discriminator.self, forKey: .type)

        switch type {
        case .text:
            self = .text(try TextBlock(from: decoder))
        case .image:
            self = .image(try ImageBlock(from: decoder))
        }
    }
}

// Decode the array
let blocks = try JSONDecoder().decode([AnyContentBlock].self, from: data)

Key pattern: Decode the discriminator field first, then use it to decide which concrete type to decode from the same decoder. The .unknown case provides forward compatibility — if the server adds a new block type, existing app versions won't crash.

Alternative: For simpler cases, you can try decoding each type in sequence and use the first one that succeeds, but this is less efficient and less explicit about the expected structure.

Q49: How would you build a property wrapper for default Codable values? Senior

A common pain point with Codable is that missing keys for non-optional properties cause decoding failures. A property wrapper can provide defaults:

protocol DefaultValue {
    associatedtype Value: Codable
    static var defaultValue: Value { get }
}

@propertyWrapper
struct Default<Provider: DefaultValue>: Codable {
    var wrappedValue: Provider.Value

    init(wrappedValue: Provider.Value) {
        self.wrappedValue = wrappedValue
    }

    init(from decoder: Decoder) throws {
        let container = try decoder.singleValueContainer()
        wrappedValue = (try? container.decode(Provider.Value.self))
            ?? Provider.defaultValue
    }
}

extension KeyedDecodingContainer {
    func decode<P: DefaultValue>(
        _ type: Default<P>.Type,
        forKey key: Key
    ) throws -> Default<P> {
        (try? decodeIfPresent(type, forKey: key)) ?? Default(wrappedValue: P.defaultValue)
    }
}

// Define common defaults
enum Defaults {
    enum False: DefaultValue {
        static let defaultValue = false
    }
    enum EmptyString: DefaultValue {
        static let defaultValue = ""
    }
    enum EmptyArray<T: Codable>: DefaultValue {
        static var defaultValue: [T] { [] }
    }
}

// Usage
struct Settings: Codable {
    @Default<Defaults.False> var darkMode: Bool
    @Default<Defaults.EmptyString> var nickname: String
    @Default<Defaults.EmptyArray<String>> var tags: [String]
}

The KeyedDecodingContainer extension is critical — it handles the case where the key is entirely missing from the JSON (not just null). Without it, the synthesized init(from:) would call decode (not decodeIfPresent) and fail on missing keys.

Q50: When would you choose binary plist over JSON? Senior

Binary property lists (.plist) are Apple's native serialization format. Compared to JSON:

Advantages of binary plist:

  • Faster parsing: Binary plist uses an offset table for O(1) random access to top-level keys. JSON requires full sequential parsing.
  • Native type support: Plist natively supports Date, Data (binary blobs), and Int/Float without string conversion. JSON encodes dates as strings and binary data as Base64.
  • Smaller size: Binary encoding is more compact than JSON's text representation, especially for numeric data.
  • System integration: UserDefaults, Info.plist, and many system APIs use plist natively.

When to choose binary plist:

  • Local configuration files and caches
  • When you need to store Date and Data types without encoding overhead
  • Inter-process communication on Apple platforms (XPC uses plist)
  • When the data stays within the Apple ecosystem

When to stick with JSON:

  • API communication (universal format)
  • Cross-platform data exchange
  • When human readability matters for debugging
  • When the data will be consumed by web services

Q51: How does FlatBuffers achieve zero-copy deserialization? L6

FlatBuffers stores data in a flat binary buffer where objects are accessed directly via pointer arithmetic — no parsing, no object allocation, no copying.

How it works:

  1. Buffer layout: The serialized buffer contains the data inline. Strings and nested objects are stored at offsets within the same buffer.
  2. VTable: Each object has a virtual table (vtable) that maps field IDs to offsets within the object. Multiple objects with the same schema share a single vtable, saving space.
  3. Access pattern: To read a field, the accessor reads the vtable to find the field's offset, then reads the value directly from the buffer at that offset. No intermediate objects are created.
  4. Strings: Stored as length-prefixed UTF-8 bytes in the buffer. The accessor returns a pointer into the buffer — no String allocation until you explicitly convert.

Performance implications:

  • Deserialization time: Effectively zero — you just verify the buffer and start reading. Compare this to JSON (full parse) or Protobuf (decode all fields).
  • Memory: Only the original buffer is in memory. No duplicated object graph.
  • Selective access: You can read one field from a 10MB buffer without touching the rest. Ideal for large messages where you only need a few fields.
  • Trade-off: Buffers are larger than Protobuf due to alignment padding. Mutation is limited (you can mutate scalar fields in-place but not resize strings or add fields).

FlatBuffers is used by Facebook (FlatBuffers in the iOS app), Google (game engines), and Apple (in some internal frameworks) for performance-critical serialization.

Q52: Design a serialization strategy for a performance-critical app. Architect

Consider a real-time collaborative editing app that must serialize document state, sync changes, and persist locally with minimal latency.

Layered serialization strategy:

1. Network layer — Protocol Buffers:

  • Define .proto schemas for all API messages
  • Use gRPC for bidirectional streaming of real-time edits
  • Protobuf's compact binary format minimizes bandwidth on mobile networks
  • Schema evolution via field numbers ensures backward compatibility

2. Local persistence — Binary plist + SQLite:

  • Document metadata in SQLite (indexed, queryable)
  • Document content as binary plist for fast native serialization
  • Large binary assets (images, attachments) as separate files referenced by hash

3. In-memory representation — Native Swift structs:

  • Value types for the document model (copy-on-write for undo)
  • Convert to/from Protobuf only at network boundaries
  • Convert to/from plist only at persistence boundaries

4. Operational transforms — Custom binary format:

struct Operation {
    let type: UInt8      // insert=1, delete=2, retain=3
    let position: UInt32  // cursor position
    let length: UInt16   // affected range
    let content: Data?   // inserted content (if insert)

    func serialize() -> Data {
        var data = Data(capacity: 7 + (content?.count ?? 0))
        data.append(type)
        withUnsafeBytes(of: position.bigEndian) { data.append(contentsOf: $0) }
        withUnsafeBytes(of: length.bigEndian) { data.append(contentsOf: $0) }
        if let content { data.append(content) }
        return data
    }
}
Architecture Decision

Key principle: Use the right format at each boundary. JSON for public APIs and debugging, Protobuf for internal services and mobile networks, binary plist for local persistence, and custom binary for hot paths. Avoid the temptation to use one format everywhere — each has strengths that matter at different layers.
