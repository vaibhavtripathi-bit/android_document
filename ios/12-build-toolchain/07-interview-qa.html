Xcode Build System (Questions 1–8)

1. Walk through the complete xcodebuild pipeline from pressing Cmd+B to a running app. L6

The pipeline proceeds through these phases:

  1. Resolve Package Dependencies: SPM resolves and checks out package dependencies using the PubGrub algorithm.
  2. Create Build Description: The build system evaluates all build settings (resolving $(inherited), $(SRCROOT), xcconfig layering), computes file lists, and generates a DAG (directed acyclic graph) of all build tasks.
  3. Compile Sources: Swift files are compiled by swiftc (incremental or whole-module mode), ObjC files by clang, storyboards by ibtool. Each produces .o object files.
  4. Link Binary: The linker (ld64 or ld-prime) combines all .o files with framework/library references into a Mach-O executable.
  5. Copy Bundle Resources: Asset catalogs are compiled by actool into Assets.car, storyboards are compiled, and other resources are copied into the app bundle.
  6. Run Script Phases: Custom scripts execute (SwiftLint, code generation, etc.).
  7. Embed Frameworks: Dynamic frameworks are copied into the Frameworks/ directory of the app bundle.
  8. Code Sign: codesign signs the app with the appropriate identity and entitlements.

The new build system (llbuild) executes these as a task graph, running independent tasks in parallel. Target dependencies are built first, and within a target, tasks execute as soon as their inputs are ready.

2. Explain build settings inheritance. If the same setting is defined at multiple levels, which wins? Senior

Build settings resolve through a strict precedence chain (highest to lowest):

  1. Command-line overrides (xcodebuild FOO=bar) — always wins
  2. Target-level settings (Build Settings tab in Xcode)
  3. Target-level xcconfig file
  4. Project-level settings (Build Settings tab)
  5. Project-level xcconfig file
  6. Xcode default values

The $(inherited) macro walks UP this chain, collecting values from each level. For list-type settings (like OTHER_LDFLAGS), forgetting $(inherited) silently replaces parent values instead of appending — this is the #1 xcconfig debugging issue. Use the "Levels" view in Xcode's Build Settings tab to see exactly where each value comes from.

3. What is the difference between SWIFT_COMPILATION_MODE incremental and wholemodule? When would you use each? Senior

Incremental mode: The Swift driver launches separate compilation jobs for each file. Each job receives one "primary" file plus all other files for type information. The driver tracks cross-file dependencies and only recompiles files affected by changes. This is fast for iterative development.

Whole module mode: A single compiler invocation processes all files together. This enables cross-file optimizations (inlining, devirtualization, generic specialization) but means any change recompiles everything.

When to use each:

  • Debug builds: Incremental — fast iteration, only changed files recompile
  • Release builds: Whole module — enables 2-5x runtime performance improvements from cross-file optimization
  • CI builds: Whole module for release/archive, incremental for PR validation (speed)
4. Why do script phases without input/output files slow down builds? How do you fix this? Senior

Without input/output file lists, the build system has no way to determine if a script phase's output is up-to-date. It must run the script on every single build, even when nothing has changed. With input/output files, the build system compares timestamps: if all outputs exist and are newer than all inputs, the phase is skipped.

Fix: Add input files (the files the script reads) and output files (the files the script produces) to the script phase. For phases with many files, use .xcfilelist files. In Xcode 14+, enable "Based on dependency analysis" to allow the build system to reorder the script phase for better parallelism. For scripts that don't produce files (like linters), write a dummy output file that serves as a timestamp marker.

5. What is derived data? Describe its structure and how you'd configure it for CI caching. L6

Derived data is where Xcode stores all build artifacts. Located at ~/Library/Developer/Xcode/DerivedData/, each project gets a subdirectory with a hash suffix. Key subdirectories:

  • Build/Intermediates.noindex/ — Object files, Swift modules, dependency info
  • Build/Products/ — Final .app, .framework, .xctest bundles
  • Index.noindex/ — Source index for code completion (machine-specific, don't cache)
  • Logs/Build/ — Build logs in .xcactivitylog format

CI caching strategy: Use -derivedDataPath ./build/DerivedData to place derived data in a known location. Cache Build/Intermediates.noindex/ keyed by a hash of source files + build settings. Also cache SPM packages separately with -clonedSourcePackagesDirPath keyed by Package.resolved hash. Disable index-while-building on CI with COMPILER_INDEX_STORE_ENABLE=NO to save time.

6. How does the new build system (llbuild) differ from the legacy build system? L6

The new build system (default since Xcode 10, only option since Xcode 14) is built on llbuild, an open-source build engine. Key differences:

  • Dependency-based scheduling: Tasks execute as soon as dependencies are satisfied, not in fixed order. This enables higher parallelism.
  • Deterministic builds: Same inputs always produce the same task ordering.
  • Cycle detection: Detects dependency cycles at build description time, not at runtime.
  • Build description caching: Reuses build descriptions across builds when project structure hasn't changed.
  • Separate process: Runs as XCBBuildService (XPC service), isolated from Xcode IDE.

The legacy system used a simpler make-like approach with less parallelism and no cycle detection. The new system's DAG-based scheduling typically results in 10-30% faster builds for multi-target projects.

7. Design the build configuration architecture for a multi-target app with debug, staging, and release environments. Architect

Use a layered xcconfig architecture:

  1. Base.xcconfig: Shared settings (deployment target, Swift version, warning flags)
  2. Warnings.xcconfig: All warning flags isolated for easy management
  3. Debug/Staging/Release.xcconfig: Each includes Base + Warnings, overrides optimization level, compilation mode, and defines environment-specific flags via SWIFT_ACTIVE_COMPILATION_CONDITIONS
  4. Per-target xcconfigs: App.xcconfig, Tests.xcconfig, Framework-X.xcconfig — target-specific bundle IDs, entitlements, signing
  5. Local.xcconfig: .gitignored, developer-specific overrides

For feature flags, use SWIFT_ACTIVE_COMPILATION_CONDITIONS for compile-time flags and a runtime feature flag system (backed by remote config) for dynamic flags. Each environment gets a separate bundle ID and provisioning profile. CI builds use command-line overrides to select the configuration. The key principle: all build settings in diffable xcconfig files, nothing in the opaque .pbxproj.

8. What are the levels of build parallelism in Xcode, and how do you maximize them? L6

Three levels of parallelism:

  1. Target parallelism: Independent targets build simultaneously. Maximize by keeping the dependency graph wide and shallow — extract shared code into a Core module, keep feature modules independent.
  2. File parallelism: In incremental mode, each Swift file compiles in a separate process. Maximize by using incremental mode for debug and splitting large targets into smaller modules.
  3. Compiler-internal parallelism: Single swiftc uses multiple threads for type checking, SIL optimization, and LLVM codegen.

Maximizing parallelism: Flatten dependency graphs, split monolithic targets, use incremental mode for debug, add input/output files to script phases (so they don't block), eliminate bridging headers (they serialize Swift/ObjC compilation), and enable "Parallelize Build" in the scheme editor.

Swift Compiler (Questions 9–16)

9. Describe the complete Swift compilation pipeline from source code to machine code. L6

The Swift compiler processes code through these stages:

  1. Lexing: Source text → tokens (keywords, identifiers, literals, operators)
  2. Parsing: Tokens → Abstract Syntax Tree (AST). Recursive-descent parser handles Swift's complex grammar including trailing closures and result builders.
  3. Semantic Analysis (Sema): Type checking, name resolution, constraint-based type inference, overload resolution, protocol conformance checking. This is the most time-consuming phase.
  4. SILGen: AST → Raw SIL (Swift Intermediate Language). SIL preserves Swift semantics like ARC, protocol dispatch, and value types.
  5. Mandatory SIL Passes: Definite initialization, ownership verification, exclusivity enforcement. Run at all optimization levels.
  6. Optimization SIL Passes (with -O/-Osize): ARC optimization, devirtualization, generic specialization, inlining, dead code elimination.
  7. IRGen: SIL → LLVM IR. Lowers Swift concepts to LLVM's representation.
  8. LLVM Passes: Standard optimizations — instruction combining, loop optimization, vectorization, register allocation.
  9. Backend: LLVM IR → machine code (.o object file).
10. What is SIL and why does Swift need it? Why not go directly from AST to LLVM IR? L6

SIL (Swift Intermediate Language) is Swift's unique IR that sits between the AST and LLVM IR. It exists because LLVM IR is too low-level to represent Swift semantics:

  • ARC operations: In LLVM IR, retain/release become opaque function calls that LLVM can't optimize. In SIL, they're first-class instructions (strong_retain, strong_release) that the SIL optimizer can analyze and eliminate redundant pairs.
  • Protocol dispatch: LLVM sees indirect calls it can't devirtualize. SIL has witness_method and class_method instructions with type information, enabling devirtualization.
  • Value semantics: SIL preserves copy-on-write semantics, allowing the optimizer to eliminate unnecessary copies.
  • Generics: SIL represents generic functions with explicit type metadata parameters, enabling specialization for concrete types.
  • Ownership: SIL has a formal ownership model (@owned, @guaranteed, @unowned) that enables proving ARC operations are redundant.

Without SIL, Swift would lose 30-50% of its optimization potential, particularly for ARC-heavy code.

11. Explain the key SIL optimization passes and what each accomplishes. L6

Key SIL optimization passes (run with -O/-Osize):

  • ARC Optimization: Eliminates redundant retain/release pairs by proving objects stay alive. Can remove 50-80% of ARC traffic in optimized builds.
  • Devirtualization: Replaces indirect calls (witness_method, class_method) with direct function calls when the concrete type is known. Eliminates vtable/witness table lookup overhead.
  • Generic Specialization: Creates type-specific versions of generic functions (e.g., swap<Int> instead of swap<T>). Eliminates type metadata passing and enables further optimization.
  • Function Inlining: Substitutes function bodies at call sites based on cost heuristics. Eliminates call overhead and enables further optimization of the combined code.
  • Stack Promotion: Moves heap allocations to the stack when escape analysis proves the object doesn't outlive the function.
  • Closure Specialization: Specializes functions that take closures for specific closure bodies, eliminating indirect call overhead.
  • Dead Code Elimination: Removes unreachable code and unused computations.
12. What does @inlinable do? What are the ABI implications? L6

@inlinable exposes a function's body in the module's .swiftinterface file, allowing the compiler to inline it across module boundaries. Without it, cross-module calls are always indirect.

ABI implications: Once marked @inlinable, the function body becomes part of the module's ABI contract. Client modules may have inlined the old body, so changing the implementation without recompiling clients can lead to inconsistent behavior — some call sites use the old body, others use the new one.

@usableFromInline is the companion attribute that makes internal declarations accessible from @inlinable functions without making them public.

Best practices: Only use for small, stable, performance-critical functions (like stdlib's arithmetic operators). Never use for complex or evolving APIs. The binary size tradeoff is that inlined bodies are duplicated in every client module.

13. Why does the Swift type checker sometimes take exponentially long? How do you diagnose and fix it? L6

Swift uses a constraint solver for type inference. For each expression, it generates type constraints and solves them simultaneously. The worst case is exponential when many overloaded operators interact in a single expression.

Common culprits:

  • Long chains of + on numeric literals (each + has multiple overloads)
  • Complex dictionary literals with [String: Any]
  • Deeply nested closures without type annotations
  • Ternary expressions mixing different literal types

Diagnosis: Add -Xfrontend -warn-long-expression-type-checking=100 to OTHER_SWIFT_FLAGS. This warns on any expression taking more than 100ms to type-check.

Fixes: Break long expressions into named sub-expressions, add explicit type annotations to closures and complex expressions, use typed structs instead of [String: Any], and add return type annotations to functions.

15. How would you establish coding guidelines that keep Swift compile times fast across a large team? Architect

Key guidelines:

  1. Explicit return types on public functions — eliminates type inference for the most-imported declarations
  2. Type annotations on complex closures — especially in map/filter/reduce chains
  3. Break long expressions — no more than 3-4 operators in a single expression
  4. Use final on classes — enables devirtualization without WMO
  5. Avoid [String: Any] — use typed structs (Codable) instead
  6. Module architecture — wide and shallow dependency graph for parallel builds

Enforcement: Add -warn-long-function-bodies=200 -warn-long-expression-type-checking=200 to CI builds. Track build times per module over time. Use xclogparser to identify the slowest-compiling files. Set up a build time dashboard that alerts on regressions.

16. What LLVM optimizations apply after SIL is lowered to LLVM IR? L6

After IRGen lowers SIL to LLVM IR, standard LLVM optimization passes run:

  • Instruction combining: Simplifies instruction sequences (e.g., multiply by power of 2 → shift)
  • Loop optimizations: Unrolling, vectorization (NEON on ARM), loop-invariant code motion, strength reduction
  • Global value numbering: Eliminates redundant computations
  • Register allocation: Maps virtual registers to physical registers, minimizing spills
  • Instruction scheduling: Reorders instructions for pipeline efficiency
  • Auto-vectorization: Uses SIMD instructions for data-parallel operations
  • Tail call optimization: Converts tail calls to jumps, eliminating stack frame overhead
  • Constant folding/propagation: Evaluates constant expressions at compile time

These are architecture-independent optimizations followed by target-specific ones for ARM64 (iOS devices) or x86_64 (simulator).

Linker & dyld (Questions 17–26)

17. Compare static libraries, static frameworks, and dynamic frameworks. When would you choose each? Senior

Static library (.a): Archive of .o files. Code is copied into the executable at link time. No bundle, so resources need separate handling. Dead code stripping applies. Best for: internal modules without resources.

Static framework: A .framework bundle containing a static library. Has a bundle for resources. Code is still copied into the executable. Best for: internal modules that need bundled resources.

Dynamic framework: A .framework bundle containing a dynamic library (Mach-O dylib). Loaded by dyld at runtime. Can be shared between app and extensions. No dead code stripping (entire framework loaded). Best for: code shared with app extensions, or when using mergeable libraries.

Decision rule: Default to static. Use dynamic only when you need code sharing with extensions or when the shared code is large enough that duplication would significantly increase total binary size.

18. How does dyld4 load an app at launch? What determines pre-main time? L6

dyld4's launch sequence:

  1. Load closure from cache: dyld4 pre-computes a "launch closure" containing the dependency graph and symbol bindings. This is cached on disk or in the shared cache.
  2. Map dylibs: Each dynamic library is mmap'd into the process address space. System frameworks are already in the shared cache (near-zero cost).
  3. Apply fixups: Rebase pointers for ASLR slide, bind external symbol references. dyld4 uses chained fixups (iOS 15+) which are more compact and faster.
  4. Run initializers: C++ static constructors, ObjC +load methods, Swift type metadata initialization.
  5. Jump to main().

Pre-main time factors: Number of embedded dynamic frameworks (~2-5ms each), number of ObjC classes registered, number of +load methods, C++ static initializers, and total fixup count. Measure with DYLD_PRINT_STATISTICS=1. Target: under 400ms total pre-main time.

19. Explain @rpath, @executable_path, and @loader_path. When is each used? Senior

These are path variables in Mach-O load commands for locating dynamic libraries:

  • @executable_path: Resolves to the directory containing the main executable. Used for frameworks embedded directly in the app bundle: @executable_path/Frameworks/MyLib.framework/MyLib
  • @loader_path: Resolves to the directory containing the Mach-O that references the library. Used when a framework depends on another framework: @loader_path/Frameworks/Helper.framework/Helper
  • @rpath: Searched against the runpath search path list (LD_RUNPATH_SEARCH_PATHS). Most flexible — the same framework binary works in different contexts because the rpath is set by the consumer, not the framework.

@rpath is the standard for modern iOS development. Frameworks set their install name to @rpath/MyFramework.framework/MyFramework, and the app sets LD_RUNPATH_SEARCH_PATHS = @executable_path/Frameworks.

20. What is dead code stripping? Why doesn't it work with dynamic frameworks? Senior

Dead code stripping is a linker optimization that removes unreachable code and data. The linker starts from entry points (main(), ObjC class references, symbols marked no-dead-strip) and traces all reachable symbols. Everything not reachable is removed.

Why it doesn't work with dynamic frameworks: Dynamic frameworks are self-contained Mach-O binaries. The linker that builds the app doesn't modify the framework's binary — it just records a reference. dyld loads the entire framework at runtime. The framework's own linker already stripped dead code within the framework, but it can't know which symbols the app will use, so it keeps everything that's exported.

This is a major argument for static linking: when you statically link a library, the linker can see exactly which symbols the app uses and strip everything else. A framework with 10,000 functions where you use 100 will include all 10,000 when dynamically linked, but only ~100 when statically linked.

21. What are mergeable libraries in Xcode 15? How do they solve the static/dynamic tradeoff? L6

Mergeable libraries are a new linking mode where frameworks are built as dynamic frameworks but can be merged into the executable at link time, like static libraries. The build system automatically switches modes based on configuration:

  • Debug: Frameworks stay dynamic (fast incremental linking)
  • Release: Frameworks are merged into the executable (fast launch, dead code stripping)

This solves the classic tradeoff: static linking gives fast launch but slow debug builds (re-linking everything), while dynamic gives fast debug builds but slow launch. Mergeable libraries give you both.

Enable: Set MERGEABLE_LIBRARY = YES on frameworks and MERGED_BINARY_TYPE = automatic on the app. Requires Xcode 15+ and benefits most with iOS 17+ deployment target.

22. What are two-level namespaces and why are they important? L6

Two-level namespaces are a Mach-O feature where each undefined symbol records which library it expects to find the symbol in. This is the default on macOS/iOS.

Without two-level namespaces (flat namespace): Symbol "parse" is searched in ALL loaded libraries in load order. If both JSONKit and XMLKit export "parse", the first one loaded wins — potentially the wrong one.

With two-level namespaces: The binding says "parse from JSONKit" and "parse from XMLKit" — no ambiguity. This provides correctness (no accidental symbol interposition), faster lookup (direct library lookup vs linear search), and deterministic behavior regardless of load order.

Flat namespace is rarely used but can be forced with -flat_namespace linker flag or DYLD_FORCE_FLAT_NAMESPACE=1 environment variable. Some legacy Unix porting scenarios require it.

23. Explain weak linking. How does Swift's #available relate to it? Senior

Weak linking allows referencing symbols that may not exist at runtime. The linker marks the reference as "weak import" in the Mach-O. At runtime, dyld sets weak symbols to NULL if the providing library doesn't have them, instead of crashing.

Swift's #available check generates weak imports under the hood. When you write if #available(iOS 17, *) { ... }, the compiler emits weak references to iOS 17 APIs. On older devices, those symbols resolve to NULL, and the #available check prevents calling them.

Framework-level weak linking (marking a framework as "Optional" in Link Binary With Libraries) makes ALL symbols from that framework weak. This is used when an entire framework might not be present (e.g., using SwiftUI on iOS 12 deployment target).

24. What is symbol visibility and how does it affect binary size and launch time? L6

Symbol visibility controls which symbols are exported from a dynamic library. Exported symbols are listed in the symbol table and available for dynamic binding by dyld.

Impact on binary size: Each exported symbol has an entry in the symbol table (name string + metadata). A framework with 10,000 exported symbols has a significantly larger __LINKEDIT segment than one with 100.

Impact on launch time: dyld must process all exported symbols during binding. Fewer exports = faster binding = faster launch.

Best practice: Use an exported symbols file (EXPORTED_SYMBOLS_FILE) to explicitly list only the symbols clients need. In Swift, public maps to exported, internal is exported from dynamic frameworks (for testability), and private/fileprivate are hidden. For C/ObjC, use __attribute__((visibility("hidden"))).

25. What is the dyld shared cache and why do system frameworks load so fast? L6

The dyld shared cache is a single pre-linked file containing ALL system frameworks. It's generated by Apple and stored on device at /System/Library/dyld/. Key properties:

  • Pre-linked: All inter-framework references are already resolved. No binding work needed at launch.
  • Shared memory: The cache is mapped into every process's address space at the same virtual address. Pages are shared across processes via copy-on-write.
  • Optimized layout: Frequently-used code is placed on the same memory pages for better cache locality.
  • No per-framework overhead: Unlike embedded frameworks that each need mmap + fixups, system frameworks in the cache are "free" to load.

This is why linking against UIKit, Foundation, etc. has near-zero launch time cost, while each embedded dynamic framework adds 2-5ms.

26. Design the linking strategy for an app with 200+ internal modules, 3 app extensions, and 15 third-party SDKs. Architect

Strategy:

  1. Core modules (50+): Static libraries. These are used everywhere, so static linking enables dead code stripping and eliminates dyld overhead.
  2. Feature modules (150+): Static libraries or mergeable libraries (if targeting iOS 17+). Features are independent, so static linking with dead code stripping removes unused code.
  3. Shared extension code: Dynamic framework. Code shared between the main app and 3 extensions must be dynamic to avoid duplication. Limit this to one or two "shared" frameworks containing only truly shared code.
  4. Third-party SDKs: Static where possible (enables dead code stripping of unused SDK features). Some SDKs only ship as dynamic — embed those.

Target: Maximum 4-6 embedded dynamic frameworks (shared extension code + unavoidable dynamic SDKs). Everything else static. Measure with DYLD_PRINT_STATISTICS — target under 200ms dylib loading time.

Build time mitigation: Use mergeable libraries or maintain separate debug (dynamic) and release (static) configurations. Cache derived data on CI.

SPM (Questions 27–32)

27. How does SPM's PubGrub dependency resolution algorithm work? L6

PubGrub uses unit propagation and conflict-driven clause learning (borrowed from SAT solvers):

  1. Start with root constraints: The app's direct dependency version requirements.
  2. Unit propagation: Select a package version, add its transitive dependency constraints to the system.
  3. Constraint intersection: Compute the intersection of all version ranges for each package.
  4. Conflict detection: If any intersection is empty (no version satisfies all constraints), a conflict is detected.
  5. Conflict analysis: Analyze the conflict to find the root cause — which decision led to the impossible state.
  6. Learn a "nogood" clause: Record the incompatibility so it's never explored again.
  7. Backjump: Jump back to the decision that caused the conflict (not just one step back like simple backtracking).

This is much more efficient than naive backtracking because it learns from conflicts and avoids re-exploring dead-end solution spaces. The result is stored in Package.resolved as a lockfile.

28. What is BUILD_LIBRARY_FOR_DISTRIBUTION and when must you enable it? Senior

BUILD_LIBRARY_FOR_DISTRIBUTION enables module stability by generating .swiftinterface files instead of .swiftmodule files. Swift interface files are textual and forward-compatible, meaning binaries built with Swift 5.8 can be used by projects using Swift 5.9.

Must enable when:

  • Creating XCFramework binary targets for distribution
  • Distributing pre-built frameworks to other teams
  • Publishing binary SPM packages

Don't enable for: Internal source-based modules (adds compile time overhead and restricts some language features like @_specialize).

When enabled, the compiler also enforces stricter rules: all public API must use types that are themselves available in the module interface, and some compiler-internal attributes are forbidden.

29. Explain SPM build tool plugins vs command plugins. Give a use case for each. Senior

Build tool plugins run automatically during the build process:

  • .prebuildCommand: Runs before compilation, every build. Use for code generation that needs to run always (e.g., generating build info).
  • .buildCommand: Runs during build when input files change. Use for incremental code generation (e.g., SwiftGen from asset catalogs).

Build tool plugins are sandboxed and have read-only access to source files. They produce output files that become part of the build.

Command plugins run on-demand when the user invokes them:

  • Invoked via swift package plugin <name> or Xcode context menu.
  • Can request write access to modify source files.
  • Use for: code formatting, linting with auto-fix, documentation generation.

Example: SwiftGen as a build tool plugin (generates Swift code from assets on every build), swift-format as a command plugin (formats code on demand).

30. How do resources work in SPM? What's the difference between .process and .copy? Senior

SPM bundles resources with targets and generates a Bundle.module accessor.

.process: Applies platform-specific processing. Asset catalogs are compiled by actool, storyboards by ibtool, strings files are compiled. The directory structure may be flattened.

.copy: Copies files as-is, preserving the directory structure. Use for files that shouldn't be processed (HTML templates, JSON fixtures, custom file formats).

Access: Always use Bundle.module in library code, never Bundle.main. In tests, Bundle.module points to the test bundle's copy of resources. SPM generates a resource_bundle_accessor.swift file that provides the Bundle.module extension.

31. What are SPM package registries and how do they differ from Git-based resolution? L6

Package registries (SE-0292, Swift 5.7) allow resolving packages from an HTTP registry server instead of Git repositories.

Key differences:

  • Speed: Registry downloads only metadata + source archives for the needed version. Git clones the entire repository history.
  • Disk usage: Registry stores only needed versions. Git stores full history.
  • Security: Registry supports signed manifests and checksums. Git relies on commit hashes.
  • Identity: Registry uses scoped identifiers (company.package-name) instead of URLs.

Configure in .swiftpm/configuration/registries.json. In Package.swift, use .package(id: "scope.name", from: "1.0.0") instead of URL-based dependencies. Registries can coexist with Git-based dependencies.

32. Design the SPM package architecture for a mono-repo with 100+ modules and multiple apps. Architect

Structure: Each module is its own Package.swift with local path dependencies. Modules are organized in layers:

  • Shared layer: DesignSystem, TestHelpers — no dependencies on other layers
  • Core layer: Networking, Storage, Analytics — depend only on Shared
  • Feature layer: Home, Search, Profile — depend on Core, never on other Features
  • App layer: MainApp, InternalApp — depend on Features

Key patterns:

  • Interface/Implementation split: Each feature exposes a FeatureInterface target (protocols + models only) and a Feature target (implementation). Other features depend on interfaces, not implementations. This breaks dependency cycles.
  • Dependency validation: Custom CI tool that parses all Package.swift files and validates no cross-feature dependencies exist.
  • CI caching: Cache SourcePackages keyed by Package.resolved hash.
  • Build tool plugins: Shared code generation plugins instead of per-module script phases.

Binary Size (Questions 33–38)

33. What is app thinning? Describe its three components. Senior

App thinning reduces the size of the app delivered to each device:

  1. App Slicing: The App Store creates device-specific variants. Each variant contains only the architecture (arm64 vs arm64e), asset scale (@2x vs @3x), and GPU resources for that device. Savings: 30-50% for most apps.
  2. On-Demand Resources (ODR): Assets tagged for on-demand download. Not included in the initial install — downloaded when the app requests them. Hosted on Apple's CDN for free. Use for: game levels, tutorials, rarely-used features. Limit: 20GB total, 512MB initial install.
  3. Bitcode: (Deprecated in Xcode 14) Was an LLVM IR representation that allowed Apple to recompile for each architecture server-side. No longer used.

The size users see on the App Store is the thinned size for their specific device, not your archive size. Use -exportOptionsPlist with thinning key to generate local size reports.

34. What is the __TEXT segment and why is it usually the largest part of an iOS binary? L6

The __TEXT segment contains all read-only, executable content:

  • __text: Machine code instructions (typically 60-70% of __TEXT)
  • __stubs / __stub_helper: Lazy binding stubs for dynamic symbol resolution
  • __cstring: C string literals
  • __swift5_typeref: Swift type reference strings for reflection
  • __swift5_reflstr: Swift field name strings for Mirror
  • __swift5_proto: Swift protocol conformance records
  • __objc_methname: ObjC method name strings
  • __const: Constant data

It's the largest because it contains ALL executable code plus all string constants and metadata. Analyze with size -m MyApp and target the largest sections for optimization. The __text section (machine code) is usually the biggest target — reduce it with -Osize, dead code stripping, and reducing generic specializations.

35. How does Swift metadata contribute to binary size? How can you reduce it? L6

Every Swift type generates metadata: type descriptors (~100-200 bytes per type), type reference strings (~50-100 bytes), field metadata (property names and types), and protocol conformance records (~12 bytes each). Generic specializations also generate separate function bodies and metadata.

Reduction strategies:

  • Fewer types: Consolidate small wrapper types into larger structs
  • Fewer generic specializations: Use existentials (any Protocol) for cold paths instead of generics
  • Disable reflection names: -Xfrontend -disable-reflection-names saves 5-15% of Swift code size but breaks Mirror
  • Disable full reflection metadata: -Xfrontend -disable-reflection-metadata (more aggressive, test thoroughly)
  • Use @_specialize selectively: Control which specializations are generated instead of letting the compiler decide
36. Compare -O and -Osize. When would you use each? Senior

-O optimizes for speed: aggressive inlining, loop unrolling, and generic specialization. Code is duplicated where it helps performance.

-Osize optimizes for size: conservative inlining, minimal unrolling, selective specialization. Typically 10-30% smaller than -O with 0-5% speed reduction.

Strategy: Use -O for performance-critical modules (image processing, data parsing, animation) and -Osize for everything else (settings screens, onboarding, rarely-used features). In SPM, you can set per-target optimization via swiftSettings: [.unsafeFlags(["-Osize"])].

For most apps, using -Osize globally is the right default — the size savings are significant and the performance difference is negligible for typical app code. Only switch specific hot-path modules to -O if profiling shows a measurable difference.

37. What is Link-Time Optimization (LTO)? Compare monolithic and thin LTO. L6

LTO allows the compiler to optimize across translation unit boundaries during the link phase. Instead of .o files containing machine code, they contain LLVM bitcode. The linker merges this bitcode and runs optimization passes on the combined module.

Monolithic LTO: Merges ALL bitcode into one giant module and optimizes as a single unit. Best optimization but very slow (single-threaded, high memory usage).

Thin LTO: Each module includes a summary of its contents. The linker reads all summaries to build a cross-module call graph, then optimizes each module in parallel with cross-module information. 80-90% of monolithic LTO's benefit with much faster build times.

Typical savings: 5-15% of __TEXT segment size. Build time cost: 20-50% longer for Thin LTO, 2-5x for Monolithic. Recommendation: Use Thin LTO (LLVM_LTO = YES_THIN) for release builds.

38. You need to reduce a 200MB app to under 50MB. Walk through your systematic approach. Architect

Phase 1 — Measure (Week 1): Generate link maps, analyze segment sizes with size -m, inspect Assets.car with assetutil, list all bundle files by size, check for duplicate resources.

Phase 2 — Quick Wins (Week 2-3):

  • Enable -Osize (10-20% code reduction)
  • Enable dead code stripping + Thin LTO (5-15%)
  • Switch to static linking (10-30% from dead code stripping)
  • Compress/optimize images (30-60% of asset size)
  • Remove unused assets (10-20%)
  • Strip symbols (5-10%)

Phase 3 — Structural Changes (Week 4-8): Move large assets to on-demand resources, download ML models on first use instead of bundling, replace bundled configs with server-fetched ones, use SF Symbols instead of custom icons.

Phase 4 — Advanced (Week 8+): Audit generic specializations, disable reflection metadata, use existentials for cold paths, set up CI size tracking to prevent regressions.

Debugging Tools (Questions 39–48)

39. Explain the difference between po, v, and p in LLDB. When should you use each? Senior

v (frame variable): Reads variable values directly from DWARF debug info and process memory. No code compilation or execution. Instant, no side effects. Use as your default for inspecting local variables.

p (expression): Compiles your input as a Swift expression, injects the compiled code into the debuggee, executes it, and reads back the result. Slow (compiler warm-up on first use), can have side effects. Use when you need to evaluate expressions (call methods, compute values).

po (expression -O): Like p but additionally calls debugDescription (or description) on the result. Slowest. Use when you want custom string output.

Rule of thumb: Start with v. If you need to call a method or compute something, use p. If you want the custom description, use po.

40. What are symbolic breakpoints? Name five essential ones every iOS developer should have. Senior

Symbolic breakpoints break on function names rather than file/line locations. They work even without source code and catch issues in system frameworks.

Five essential symbolic breakpoints:

  1. UIViewAlertForUnsatisfiableConstraints — Catches Auto Layout conflicts. Add action: po [[UIWindow keyWindow] _autolayoutTrace]
  2. objc_exception_throw — Catches ObjC exceptions at the throw site, before the stack unwinds
  3. swift_willThrow — Catches Swift errors at the throw site
  4. -[NSObject doesNotRecognizeSelector:] — Catches unrecognized selector crashes
  5. malloc_error_break — Catches malloc heap corruption

Add these as user breakpoints in Xcode (shared across all projects). They cost nothing when not hit and save hours of debugging when they are.

41. How does Address Sanitizer (ASan) work? What's the performance overhead? L6

ASan instruments every memory access with bounds checking using a shadow memory scheme:

  1. Each 8 bytes of application memory maps to 1 byte of shadow memory
  2. "Redzones" (poisoned memory) are placed around every allocation
  3. Before every memory access, ASan checks the shadow byte: shadow = (addr >> 3) + offset
  4. If the shadow byte indicates poisoned memory → report error with full stack trace
  5. Freed memory is quarantined (not immediately reused) to catch use-after-free

Overhead: ~2x memory usage (shadow memory), 2-3x CPU slowdown. Not suitable for production. Can be combined with UBSan but NOT with TSan.

Detects: Heap/stack buffer overflow, use-after-free, use-after-return, double free, memory leaks (with LeakSanitizer).

42. How does Thread Sanitizer (TSan) detect data races? What are its limitations? L6

TSan records metadata for every memory access: thread ID, address, access type (read/write), and a logical timestamp. For each access, it checks if there's a concurrent access from another thread without a happens-before relationship (synchronization).

TSan understands synchronization primitives: locks, dispatch queues, semaphores, atomic operations, and Swift actors. If two accesses to the same location have no synchronization path between them, it's a data race.

Limitations:

  • Simulator only — not supported on physical devices
  • Cannot combine with ASan — mutually exclusive
  • 5-15x CPU slowdown, 5-10x memory — tests run much slower
  • False negatives possible — TSan only detects races that actually occur during the test run. A race that doesn't manifest won't be caught.
  • Some false positives — rare, usually with custom synchronization primitives TSan doesn't understand
43. What is Chisel? Name five useful Chisel commands for UI debugging. Senior

Chisel is Facebook's collection of LLDB Python scripts that add powerful debugging commands, especially for UI work.

  1. pviews — Prints the entire view hierarchy with frames, hidden state, and alpha. Much more readable than po view.recursiveDescription().
  2. pvc — Prints the view controller hierarchy showing presented and child controllers.
  3. fv UILabel — Finds all instances of a view class in the hierarchy. Returns addresses you can inspect.
  4. border 0x7f8b2c --color red — Adds a colored border to a view for visual identification. Use with caflush to update the screen.
  5. presponder 0x7f8b2c — Prints the responder chain from a given view, showing the path events travel.

Install with brew install chisel and add command script import /opt/homebrew/opt/chisel/libexec/fbchisellldb.py to ~/.lldbinit.

44. What is malloc scribble and guard malloc? When would you use each? L6

Malloc Scribble: Fills freed memory with 0x55 and newly allocated memory with 0xAA. This makes use-after-free bugs crash immediately with recognizable patterns instead of silently corrupting data. Low overhead — suitable for regular debug builds. Enable via Xcode diagnostics or MallocScribble=1 environment variable.

Guard Malloc: Places each allocation on its own virtual memory page with guard pages (inaccessible) before and after. Any buffer overflow or underflow immediately triggers a hardware exception (EXC_BAD_ACCESS) with a precise stack trace. Extremely high overhead (each allocation uses at least one 16KB page) — only for targeted debugging of specific memory corruption issues.

Use malloc scribble as a lightweight always-on diagnostic in debug builds. Use guard malloc when you have a specific memory corruption bug you're trying to isolate and can tolerate the extreme memory overhead.

45. How does the Memory Graph Debugger detect retain cycles? Senior

The Memory Graph Debugger captures a snapshot of all live objects and their references by:

  1. Pausing the process and scanning all heap allocations
  2. Tracing references: For each object, scanning its memory for pointers to other objects. It understands Swift/ObjC object layouts, including stored properties and captured variables in closures.
  3. Building a reference graph: Nodes are objects, edges are strong references.
  4. Detecting cycles: Finding strongly-connected components in the graph that have no external references (unreachable from root objects like UIApplication).

With malloc stack logging enabled, it also shows the allocation backtrace for each object, making it easy to identify where the leaking objects were created. The graph can be exported for offline analysis with leaks --outputGraph.

46. What is UBSan and why is it less relevant for pure Swift code? Senior

UBSan (Undefined Behavior Sanitizer) detects undefined behavior at runtime: signed integer overflow, null pointer dereference, misaligned memory access, invalid enum values, and more.

Less relevant for pure Swift because: Swift prevents most UB by design — integer overflow traps in debug mode, optionals prevent null dereference, array bounds are checked, and enum exhaustiveness is enforced at compile time.

Still valuable for: Projects with C/ObjC interop, code using UnsafePointer/UnsafeRawPointer, bridged C libraries, and ObjC code that does manual memory management or pointer arithmetic. UBSan has minimal overhead (5-20% CPU, negligible memory) and can be combined with ASan, making it a good addition to CI.

47. How would you use LLDB to debug a crash that only reproduces in release builds? L6

Release build crashes are harder because optimizations change code layout and debug info may be stripped. Approach:

  1. Symbolicate the crash log: Use the dSYM file with atos or Xcode's crash log viewer to get function names and line numbers.
  2. Reproduce with debug info: Build with -O optimization but keep DEBUG_INFORMATION_FORMAT = dwarf-with-dsym. This gives you optimized code with debug symbols.
  3. Use LLDB with dSYM: lldb MyApp, then target symbols add MyApp.dSYM. Set breakpoints and step through optimized code.
  4. Check for optimization-related issues: Race conditions (more likely with optimized code), uninitialized memory (optimizer may reorder), and assumptions about evaluation order.
  5. Use ASan with optimization: -O -fsanitize=address gives you optimized code with memory checking.
  6. Register inspection: In optimized code, variables may be in registers. Use register read and image lookup -a to correlate addresses with symbols.
48. Design a CI pipeline that uses sanitizers for continuous safety checking. What are the tradeoffs? Architect

Pipeline design:

  • Every PR: Regular tests (fast feedback) + ASan tests (memory safety) + TSan tests (thread safety) — run in parallel CI jobs
  • Nightly: UBSan tests + extended test suite with ASan + performance regression tests
  • Weekly: Full sanitizer suite with comprehensive UI tests

Tradeoffs:

  • CI cost: ASan adds 2-3x test time, TSan adds 5-15x. Budget for 3 parallel jobs per PR.
  • ASan + TSan mutual exclusion: Must be separate jobs — can't run both simultaneously.
  • TSan simulator-only: Can't test device-specific code paths.
  • False positives: Rare but possible, especially with third-party code. Use suppression files with tracking tickets.
  • Developer friction: Longer CI times. Mitigate by running sanitizer jobs in parallel with regular tests and making them required but not blocking initial feedback.

Key principle: Make sanitizer failures blocking for merge. A data race or memory error that ships to production costs 100x more than the CI time to catch it.
