1. Static vs Dynamic Linking

The choice between static and dynamic linking is one of the most impactful architectural decisions for large iOS apps. It affects launch time, binary size, build time, and code sharing.

1.1 Static Linking

Static Linking: Code copied into the executable at build time Build Time: ┌─────────┐ ┌─────────┐ ┌─────────┐ │ Core.a │ │ UI.a │ │Network.a│ ← Static libraries (.a) │ (code) │ │ (code) │ │ (code) │ └────┬────┘ └────┬────┘ └────┬────┘ │ │ │ └────────────┼────────────┘ │ ┌─────┴─────┐ │ ld64 │ ← Linker copies all code into executable └─────┬─────┘ │ ┌─────┴─────┐ │ MyApp │ ← Single executable with all code embedded │ (large) │ └───────────┘ Runtime: ┌───────────────────────┐ │ MyApp │ ← Everything in one binary │ ┌──────┐ ┌────────┐ │ │ │ Core │ │ UI │ │ No dynamic loading needed │ └──────┘ └────────┘ │ Fast launch (no dyld work) │ ┌─────────┐ │ │ │ Network │ │ │ └─────────┘ │ └───────────────────────┘

1.2 Dynamic Linking

Dynamic Linking: Code loaded at runtime by dyld Build Time: ┌──────────────┐ ┌──────────────┐ ┌──────────────┐ │Core.framework│ │ UI.framework │ │Net.framework │ ← Dynamic frameworks │ (Mach-O) │ │ (Mach-O) │ │ (Mach-O) │ └──────┬───────┘ └──────┬───────┘ └──────┬───────┘ │ │ │ └─────────────────┼──────────────────┘ │ ┌─────┴─────┐ │ ld64 │ ← Linker records references only └─────┬─────┘ │ ┌─────┴─────┐ │ MyApp │ ← Small executable with load commands │ (small) │ └───────────┘ Runtime: ┌───────────────────────┐ │ MyApp │ │ (load commands) │ dyld loads frameworks at launch └───────────┬───────────┘ │ ┌─────────┼──────────┬──────────────┐ ▼ ▼ ▼ ▼ ┌──────┐ ┌──────┐ ┌─────────┐ ┌──────────┐ │ Core │ │ UI │ │ Network │ │ System.fw│ └──────┘ └──────┘ └─────────┘ └──────────┘

1.3 Comparison Table

Aspect Static Library (.a) Static Framework Dynamic Framework
Launch time Fastest (no dyld work) Fastest (no dyld work) Slower (+2-5ms per framework)
Binary size Larger (dead code stripping helps) Larger (dead code stripping helps) Smaller executable, but frameworks add overhead
Build time Slower link phase Slower link phase Faster link phase
Code sharing No (copied per target) No (copied per target) Yes (app + extensions share)
Resources No bundle (resources need separate handling) Has bundle (resources included) Has bundle (resources included)
Dead code stripping Yes (linker removes unused symbols) Yes No (entire framework loaded)
App extensions Code duplicated in each extension Code duplicated in each extension Shared between app and extensions
When to Use Each

Static: Default choice for internal modules. Faster launch, dead code stripping, simpler deployment. Use when you don't need code sharing with extensions.

Dynamic: When you have app extensions that share significant code with the main app. Also required for SwiftUI previews in some configurations.

2. The Linker (ld64 / ld-prime)

The linker combines compiled object files (.o) and libraries into a final Mach-O executable. Apple ships two linkers: the classic ld64 and the new ld-prime (Xcode 15+, written in C++ with better performance).

2.1 Linker Pipeline

Linker Pipeline: Input: *.o files (compiled sources) *.a files (static libraries) *.dylib / *.tbd files (dynamic library references) │ ▼ ┌──────────────────────────────────────────────────────┐ │ 1. Parse Input Files │ │ Read Mach-O headers, symbol tables, relocations │ ├──────────────────────────────────────────────────────┤ │ 2. Symbol Resolution │ │ Match undefined symbols to definitions │ │ Resolve weak vs strong symbols │ │ Report undefined symbol errors │ ├──────────────────────────────────────────────────────┤ │ 3. Dead Code Stripping │ │ Mark reachable symbols from entry points │ │ Remove unreachable code and data │ ├──────────────────────────────────────────────────────┤ │ 4. Layout │ │ Assign addresses to segments and sections │ │ __TEXT, __DATA, __LINKEDIT layout │ ├──────────────────────────────────────────────────────┤ │ 5. Fixup Generation │ │ Generate rebase info (ASLR) │ │ Generate bind info (dyld symbol binding) │ │ Generate chained fixups (iOS 15+) │ ├──────────────────────────────────────────────────────┤ │ 6. Output │ │ Write final Mach-O executable │ │ Generate dSYM (debug symbols) if requested │ └──────────────────────────────────────────────────────┘

2.2 Key Linker Flags

# Common linker flags (set via OTHER_LDFLAGS)

# Link against a framework
-framework UIKit
-framework CoreData

# Link against a library
-lsqlite3          # links libsqlite3.dylib or .tbd
-lz                # links libz.dylib

# Force load all symbols from a static library
-force_load /path/to/libFoo.a

# Load all ObjC symbols (needed for categories in static libs)
-ObjC

# Dead code stripping
-dead_strip

# Export/hide symbols
-exported_symbols_list exports.txt
-unexported_symbols_list hidden.txt

# Map file (shows what's in the binary)
-map output.map

# Linker warning control
-warn_unused_dylibs
-no_warn_duplicate_libraries

2.3 install_name_tool

This tool modifies the install names and rpaths in Mach-O binaries. Useful for fixing up framework paths after building.

# View current install name
otool -D MyFramework.framework/MyFramework

# Change install name
install_name_tool -id @rpath/MyFramework.framework/MyFramework \
    MyFramework.framework/MyFramework

# Change a dependency path
install_name_tool -change /old/path/libFoo.dylib @rpath/libFoo.dylib \
    MyApp

# Add an rpath
install_name_tool -add_rpath @executable_path/Frameworks MyApp

# Delete an rpath
install_name_tool -delete_rpath /usr/local/lib MyApp

3. dyld4 and the Shared Cache

dyld (dynamic linker/loader) is responsible for loading dynamic libraries at runtime. dyld4 (iOS 16+, macOS 13+) is a complete rewrite with significant performance improvements.

3.1 dyld4 Launch Sequence

App Launch with dyld4: Kernel loads executable into memory │ ▼ ┌──────────────────────────────────────────────────────────┐ │ dyld4 Launch Closure (pre-computed, cached) │ ├──────────────────────────────────────────────────────────┤ │ │ │ 1. Load Closure from Cache │ │ └─ Pre-computed dependency graph, symbol bindings │ │ └─ Stored in dyld shared cache or on-disk closure │ │ │ │ 2. Map Dylibs │ │ └─ mmap() each dynamic library into address space │ │ └─ System frameworks already in shared cache (fast) │ │ │ │ 3. Apply Fixups │ │ ├─ Rebase: adjust pointers for ASLR slide │ │ └─ Bind: resolve external symbol references │ │ └─ Chained fixups (iOS 15+): walk fixup chains │ │ │ │ 4. Run Initializers │ │ ├─ C++ static constructors │ │ ├─ ObjC +load methods │ │ └─ Swift type metadata initialization │ │ │ │ 5. Jump to main() │ │ │ └──────────────────────────────────────────────────────────┘

3.2 Shared Cache

The dyld shared cache is a single file containing all system frameworks pre-linked together. It's generated by Apple and stored on device. This is why system frameworks have near-zero loading cost.

# View shared cache contents
dyld_shared_cache_util -list /System/Library/dyld/dyld_shared_cache_arm64e

# Extract a framework from the cache
dyld_shared_cache_util -extract /tmp/extracted \
    /System/Library/dyld/dyld_shared_cache_arm64e

# On macOS, view loaded dylibs
DYLD_PRINT_LIBRARIES=1 ./MyApp

# Measure dyld time
DYLD_PRINT_STATISTICS=1 ./MyApp
# Output:
# Total pre-main time: 312.00 milliseconds (100.0%)
#   dylib loading time: 198.00 milliseconds (63.4%)
#   rebase/binding time:  45.00 milliseconds (14.4%)
#   ObjC setup time:      32.00 milliseconds (10.2%)
#   initializer time:     37.00 milliseconds (11.8%)
dyld Launch Time Impact

Each embedded dynamic framework adds approximately 2-5ms to launch time. An app with 100 dynamic frameworks can add 200-500ms to pre-main time. This is why Apple recommends limiting embedded dynamic frameworks to ~6 for optimal launch performance. Use DYLD_PRINT_STATISTICS to measure.

3.3 dyld4 vs dyld3 Improvements

Feature dyld3 dyld4
Closure caching On-disk closures In-process + on-disk closures
Just-in-time binding No (all upfront) Yes (lazy binding for some symbols)
Chained fixups Limited Full support (smaller __LINKEDIT)
Page-in linking No Yes (apply fixups on page fault)
Compact info Separate rebase/bind opcodes Chained fixups (embedded in __DATA)

4. @rpath, @executable_path, @loader_path

These are special path variables used in Mach-O load commands to locate dynamic libraries at runtime. Understanding them is essential for debugging "library not found" crashes.

4.1 Path Variables

Variable Resolves To Use Case
@executable_path Directory containing the main executable Frameworks embedded in app bundle
@loader_path Directory containing the Mach-O that references the library Framework depending on another framework
@rpath Searched in order from the runpath search path list Flexible — works for apps and frameworks

4.2 How @rpath Resolution Works

Framework install name: @rpath/MyFramework.framework/MyFramework dyld resolves @rpath by searching LD_RUNPATH_SEARCH_PATHS in order: App's LD_RUNPATH_SEARCH_PATHS: 1. @executable_path/Frameworks 2. @loader_path/Frameworks Search order: 1. /path/to/MyApp.app/Frameworks/MyFramework.framework/MyFramework ← Found! 2. (would try @loader_path/Frameworks next if not found) For nested frameworks (framework depending on framework): Framework A's LD_RUNPATH_SEARCH_PATHS: 1. @executable_path/Frameworks ← Resolves to app's Frameworks/ 2. @loader_path/Frameworks ← Resolves to FrameworkA's Frameworks/ 
# View rpaths in a binary
otool -l MyApp | grep -A 2 LC_RPATH

# View library dependencies
otool -L MyApp.app/MyApp

# Typical output:
# @rpath/Core.framework/Core (compatibility version 1.0.0)
# @rpath/UI.framework/UI (compatibility version 1.0.0)
# /usr/lib/libobjc.A.dylib
# /usr/lib/libSystem.B.dylib
# /System/Library/Frameworks/UIKit.framework/UIKit

5. Framework Search Paths

The compiler and linker use different search paths to find frameworks. Misconfigured search paths are a common source of build failures.

5.1 Build Settings

// Framework search paths (for the compiler and linker to find .framework bundles)
FRAMEWORK_SEARCH_PATHS = $(inherited) $(SRCROOT)/Carthage/Build/iOS

// Library search paths (for .a and .dylib files)
LIBRARY_SEARCH_PATHS = $(inherited) $(SRCROOT)/Libraries

// Header search paths (for C/ObjC headers)
HEADER_SEARCH_PATHS = $(inherited) $(SRCROOT)/include

// Swift import paths (for .swiftmodule files)
SWIFT_INCLUDE_PATHS = $(inherited) $(SRCROOT)/Modules

// System framework search paths (searched after FRAMEWORK_SEARCH_PATHS)
SYSTEM_FRAMEWORK_SEARCH_PATHS = $(inherited)

// Runpath search paths (for dyld at runtime)
LD_RUNPATH_SEARCH_PATHS = $(inherited) @executable_path/Frameworks

5.2 Search Order

Framework Search Order (compile time): 1. FRAMEWORK_SEARCH_PATHS (project-specific) 2. SYSTEM_FRAMEWORK_SEARCH_PATHS 3. SDK framework directories └─ $(SDKROOT)/System/Library/Frameworks 4. Toolchain frameworks └─ $(TOOLCHAIN_DIR)/usr/lib/swift Library Search Order (link time): 1. LIBRARY_SEARCH_PATHS 2. SDK library directories └─ $(SDKROOT)/usr/lib 3. Toolchain libraries

6. Symbol Visibility

Symbol visibility controls which symbols are exported from a dynamic library. Reducing exported symbols improves link time, launch time, and binary size.

6.1 Visibility Levels

Visibility Exported? Description
default (public) Yes Visible to other binaries, can be dynamically bound
hidden No Only visible within the binary, not exported
internal (Swift) Depends Exported from dynamic frameworks, hidden from static 
// Swift access control maps to symbol visibility:
public func publicAPI()     // → exported symbol (visible to clients)
internal func internalAPI() // → exported from dylib (for testing), hidden from static lib
private func privateImpl()  // → hidden symbol (name-mangled, not exported)

// In C/ObjC:
__attribute__((visibility("default"))) void publicFunc(void);
__attribute__((visibility("hidden"))) void hiddenFunc(void);
# View exported symbols
nm -gU MyFramework.framework/MyFramework | head -20

# Count exported symbols
nm -gU MyFramework.framework/MyFramework | wc -l

# Use exported symbols list to limit exports
# In build settings:
EXPORTED_SYMBOLS_FILE = exports.exp

# exports.exp:
_main
_MyPublicFunction
_OBJC_CLASS_$_MyPublicClass
Best Practice: Minimize Exported Symbols

For dynamic frameworks, use an exported symbols file to explicitly list only the symbols clients need. Fewer exported symbols means faster dyld binding, smaller __LINKEDIT, and better encapsulation. A framework with 10,000 symbols that only needs to export 100 can see measurable launch time improvements.

7. Dead Code Stripping

Dead code stripping removes unreachable code and data from the final binary. It's one of the most effective ways to reduce binary size when using static libraries.

7.1 How It Works

Dead Code Stripping Algorithm: 1. Start from entry points (roots): • main() • ObjC class references • Symbols marked as "no dead strip" • C/C++ constructors 2. Trace all reachable symbols: main() → calls funcA() → calls funcB() → references ClassX 3. Remove everything not reachable: Before: After: ┌──────────┐ ┌──────────┐ │ main() │ ✓ reachable │ main() │ │ funcA() │ ✓ reachable │ funcA() │ │ funcB() │ ✓ reachable │ funcB() │ │ funcC() │ ✗ dead │ ClassX │ │ ClassX │ ✓ reachable └──────────┘ │ ClassY │ ✗ dead Removed: funcC(), ClassY └──────────┘

7.2 Limitations

// ObjC classes are always considered "live" due to runtime registration
// The linker can't know if a class is used via NSClassFromString()

// Swift classes with @objc are also kept alive
@objc class MyClass: NSObject { }  // Won't be stripped even if unused

// Pure Swift classes CAN be stripped if unreferenced
class PureSwiftClass { }  // Will be stripped if no references

// Dynamic frameworks: NO dead code stripping
// The entire framework is loaded, even if only one function is used
// This is a major argument for static linking
# Enable dead code stripping
DEAD_CODE_STRIPPING = YES

# View what was stripped (link map)
OTHER_LDFLAGS = -Wl,-map,output.map -Wl,-dead_strip

# The map file shows all symbols and their sizes
# Compare map files before/after to see stripping impact

8. Mergeable Libraries (Xcode 15)

Mergeable libraries are a new linking mode in Xcode 15 that gives you the best of both worlds: dynamic frameworks for debug builds (fast linking) and static linking for release builds (fast launch, dead code stripping).

8.1 How Mergeable Libraries Work

Mergeable Libraries: Automatic static/dynamic switching Debug Build (dynamic — fast link time): ┌──────────────┐ ┌──────────────┐ │Core.framework│ │ UI.framework │ ← Dynamic frameworks └──────┬───────┘ └──────┬───────┘ │ │ dyld loads at runtime (fast build, slower launch) Release Build (merged — fast launch): ┌──────────────┐ ┌──────────────┐ │Core.framework│ │ UI.framework │ ← Same frameworks └──────┬───────┘ └──────┬───────┘ │ │ Linker merges into executable (slower build, fast launch) │ ┌──────┴───────┐ │ MyApp │ ← Single binary with all code merged │ + dead strip │ Dead code stripped, no dyld overhead └──────────────┘

8.2 Enabling Mergeable Libraries

// In the framework's build settings:
MERGEABLE_LIBRARY = YES

// In the app's build settings:
MERGED_BINARY_TYPE = automatic  // or "manual"

// "automatic": Xcode merges in Release, keeps dynamic in Debug
// "manual": You control which frameworks merge via build settings

8.3 Requirements and Limitations

  • Requires Xcode 15+ and iOS 17+ deployment target (for full benefits)
  • Frameworks must be built as dynamic frameworks with MERGEABLE_LIBRARY = YES
  • ObjC categories in mergeable libraries need -ObjC linker flag
  • Can't merge frameworks that are also used by app extensions (they need to stay dynamic for sharing)
Mergeable Libraries vs Static Libraries

Mergeable libraries solve the "debug build time vs release launch time" tradeoff. With static libraries, you get fast launch but slow debug linking. With dynamic frameworks, you get fast debug linking but slow launch. Mergeable libraries automatically switch between the two modes based on build configuration.

9. Two-Level Namespaces

Two-level namespaces are a Mach-O feature where each undefined symbol records which library it expects to find the symbol in. This is the default on macOS/iOS and provides both correctness and performance benefits.

9.1 Two-Level vs Flat Namespace

Flat Namespace (legacy, rarely used): Symbol "foo" → search ALL loaded libraries in order Problem: Name collisions between libraries Two-Level Namespace (default): Symbol "foo" from "libBar" → look ONLY in libBar Benefit: No collisions, faster lookup, deterministic Example: App imports "parse" from JSONKit.framework App imports "parse" from XMLKit.framework Flat: Ambiguous — which "parse" gets called? Two-Level: Each binding specifies the library → no ambiguity 
# Check if binary uses two-level namespaces
otool -hv MyApp | grep TWOLEVEL

# Force flat namespace (rarely needed)
OTHER_LDFLAGS = -flat_namespace

# At runtime, force flat namespace
DYLD_FORCE_FLAT_NAMESPACE=1 ./MyApp

10. Weak Linking

Weak linking allows your app to reference symbols that may not exist at runtime. This is essential for supporting multiple OS versions where newer APIs may not be available on older devices.

10.1 Framework-Level Weak Linking

// In Build Phases → Link Binary With Libraries:
// Set framework status to "Optional" instead of "Required"

// Or via linker flags:
OTHER_LDFLAGS = -weak_framework SwiftUI

// At runtime, check if framework is available:
if NSClassFromString("SwiftUI.UIHostingController") != nil {
    // SwiftUI is available
}

10.2 Symbol-Level Weak Linking

// Swift handles this with @available and #available
if #available(iOS 17, *) {
    // Use iOS 17 API — compiler generates weak reference
    let observer = TipKit.TipObserver()
} else {
    // Fallback for older iOS
}

// Under the hood, the compiler emits a weak_import for the symbol.
// dyld sets weak symbols to NULL if the library doesn't provide them.
// The #available check prevents calling NULL function pointers.

11. Architect: Linking Strategy for 200+ Modules

Architect Must-Know

At Staff level, you need to design the linking strategy for a large modular app. The wrong choice can add 2+ seconds to launch time or 50MB+ to binary size. Here's a systematic approach.

11.1 Decision Framework

Linking Strategy Decision Tree: Does the module share code with app extensions? ├── YES → Dynamic framework (shared between app + extensions) │ └── Consider: Is the shared code > 1MB? │ ├── YES → Dynamic framework (saves total size) │ └── NO → Static library (duplication is acceptable) │ └── NO → Does the app target iOS 17+? ├── YES → Mergeable library (best of both worlds) └── NO → Static library (default choice) └── Consider: Is build time a problem? ├── YES → Dynamic for debug, static for release └── NO → Static everywhere

11.2 Recommended Architecture

// Typical large app linking architecture

// Layer 1: Core modules — STATIC (always linked, dead-code stripped)
// - Networking, Storage, Analytics, Logging, Common utilities
// - These are used by everything, static avoids dyld overhead

// Layer 2: Feature modules — STATIC or MERGEABLE
// - Home, Search, Profile, Settings, etc.
// - Static for best launch time, mergeable for build speed

// Layer 3: Shared extension modules — DYNAMIC
// - Code shared between app and extensions (Share, Widget, etc.)
// - Must be dynamic for code sharing

// Layer 4: Third-party SDKs — STATIC preferred
// - Firebase, analytics SDKs, etc.
// - Static to enable dead code stripping of unused features

Linking Strategy Checklist for Large Apps
  • Default to static linking for all internal modules
  • Use dynamic only when code sharing with extensions is required
  • Consider mergeable libraries for iOS 17+ targets
  • Enable dead code stripping (DEAD_CODE_STRIPPING = YES)
  • Minimize exported symbols from dynamic frameworks
  • Measure pre-main time with DYLD_PRINT_STATISTICS
  • Keep embedded dynamic frameworks under 6
  • Use -ObjC flag when static linking ObjC libraries with categories
  • Generate and review link maps to understand binary composition
