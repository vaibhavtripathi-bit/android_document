1. Package.swift Manifest DSL

The Package.swift file is an executable Swift script that defines a package's structure. It's compiled and executed by the Swift toolchain to produce a package description. Understanding its full capabilities is essential for complex package configurations.

1.1 Complete Manifest Example

// swift-tools-version: 5.9
import PackageDescription

let package = Package(
    name: "MyLibrary",
    defaultLocalization: "en",
    platforms: [
        .iOS(.v16),
        .macOS(.v13),
        .watchOS(.v9),
        .tvOS(.v16)
    ],
    products: [
        // Public API — what consumers can import
        .library(
            name: "MyLibrary",
            type: .static,  // .static, .dynamic, or omit for automatic
            targets: ["MyLibrary"]
        ),
        .library(
            name: "MyLibraryUI",
            targets: ["MyLibraryUI"]
        ),
        .executable(
            name: "my-tool",
            targets: ["MyTool"]
        ),
        .plugin(
            name: "MyLinter",
            targets: ["MyLinterPlugin"]
        )
    ],
    dependencies: [
        // Version-based (semver)
        .package(url: "https://github.com/apple/swift-algorithms", from: "1.0.0"),
        
        // Exact version
        .package(url: "https://github.com/apple/swift-collections", exact: "1.0.4"),
        
        // Version range
        .package(url: "https://github.com/apple/swift-log", "1.4.0"..<"2.0.0"),
        
        // Branch-based (for development)
        .package(url: "https://github.com/company/internal-sdk", branch: "main"),
        
        // Revision-based (pinned to commit)
        .package(url: "https://github.com/company/legacy-lib",
                 revision: "abc123def456"),
        
        // Local path (for development)
        .package(path: "../SharedKit")
    ],
    targets: [
        .target(
            name: "MyLibrary",
            dependencies: [
                .product(name: "Algorithms", package: "swift-algorithms"),
                .product(name: "Collections", package: "swift-collections"),
                "MyLibraryObjC",  // Internal target dependency
            ],
            path: "Sources/MyLibrary",
            exclude: ["Internal/Deprecated"],
            resources: [
                .process("Resources"),
                .copy("Templates")
            ],
            swiftSettings: [
                .define("FEATURE_FLAG_X"),
                .unsafeFlags(["-warn-long-function-bodies=100"],
                             .when(configuration: .debug))
            ],
            linkerSettings: [
                .linkedFramework("CoreData"),
                .linkedLibrary("sqlite3")
            ]
        ),
        .target(
            name: "MyLibraryObjC",
            dependencies: [],
            path: "Sources/MyLibraryObjC",
            publicHeadersPath: "include",
            cSettings: [
                .headerSearchPath("internal"),
                .define("NS_BLOCK_ASSERTIONS", .when(configuration: .release))
            ]
        ),
        .target(
            name: "MyLibraryUI",
            dependencies: ["MyLibrary"]
        ),
        .executableTarget(
            name: "MyTool",
            dependencies: ["MyLibrary"]
        ),
        .plugin(
            name: "MyLinterPlugin",
            capability: .buildTool()
        ),
        .testTarget(
            name: "MyLibraryTests",
            dependencies: ["MyLibrary"],
            resources: [.copy("Fixtures")]
        )
    ],
    swiftLanguageVersions: [.v5],
    cLanguageStandard: .c11,
    cxxLanguageStandard: .cxx17
)

2. Dependency Resolution (PubGrub)

SPM uses the PubGrub algorithm for dependency resolution — the same algorithm used by Dart's pub and Rust's cargo. It's a significant improvement over the older backtracking algorithm.

2.1 The Problem

Dependency Resolution Problem: App depends on: ├── LibA >= 1.0.0 └── LibB >= 2.0.0 LibA 1.2.0 depends on: └── LibC >= 1.0.0, < 2.0.0 LibB 2.1.0 depends on: └── LibC >= 1.5.0, < 3.0.0 Question: What version of LibC satisfies both constraints? Answer: LibC >= 1.5.0, < 2.0.0 (intersection of both ranges) Pick: LibC 1.9.0 (highest compatible version)

2.2 PubGrub Algorithm

PubGrub uses unit propagation and conflict-driven clause learning (borrowed from SAT solvers) to efficiently resolve dependencies. Unlike simple backtracking, it learns from conflicts to avoid exploring impossible solution spaces.

PubGrub Resolution Steps: 1. Start with root package constraints { App: 1.0.0, LibA: >=1.0.0, LibB: >=2.0.0 } 2. Unit Propagation — derive new constraints Select LibA 1.2.0 → adds { LibC: >=1.0.0, <2.0.0 } Select LibB 2.1.0 → adds { LibC: >=1.5.0, <3.0.0 } 3. Constraint Intersection LibC: >=1.5.0, <2.0.0 (intersection) 4. Select highest compatible version LibC: 1.9.0 5. If conflict detected: • Analyze conflict to find root cause • Learn a "nogood" clause (incompatibility) • Backjump to the decision that caused the conflict • Never explore that dead-end again Result: { App: 1.0.0, LibA: 1.2.0, LibB: 2.1.0, LibC: 1.9.0 }

2.3 Resolution Files

// Package.resolved — lockfile (commit this to source control)
{
  "pins": [
    {
      "identity": "swift-algorithms",
      "kind": "remoteSourceControl",
      "location": "https://github.com/apple/swift-algorithms",
      "state": {
        "revision": "abc123...",
        "version": "1.2.0"
      }
    }
  ],
  "version": 3
}
# Force re-resolution
swift package resolve

# Update all dependencies to latest compatible versions
swift package update

# Update a specific package
swift package update swift-algorithms

# Show dependency graph
swift package show-dependencies --format json

# Reset package caches
swift package reset
Resolution Pitfall: Version Conflicts

The most common resolution failure is when two dependencies require incompatible versions of a shared dependency. PubGrub provides clear error messages explaining the conflict chain. The fix is usually to update one of the conflicting packages to a version with relaxed constraints, or to fork and patch the dependency.

3. Binary Targets & XCFramework

Binary targets allow distributing pre-compiled code through SPM. XCFramework is the container format that supports multiple platforms and architectures.

3.1 XCFramework Structure

MySDK.xcframework/ ├── Info.plist ← Describes all variants ├── ios-arm64/ │ └── MySDK.framework/ │ ├── MySDK ← ARM64 binary │ ├── Headers/ │ ├── Modules/ │ │ ├── module.modulemap │ │ └── MySDK.swiftmodule/ │ │ └── arm64-apple-ios.swiftinterface │ └── Info.plist ├── ios-arm64_x86_64-simulator/ │ └── MySDK.framework/ │ ├── MySDK ← Fat binary (arm64 + x86_64) │ └── ... └── macos-arm64_x86_64/ └── MySDK.framework/ ├── MySDK ← Fat binary (arm64 + x86_64) └── ...

3.2 Creating XCFrameworks

# Build for each platform
xcodebuild archive \
    -scheme MySDK \
    -destination "generic/platform=iOS" \
    -archivePath ./build/ios.xcarchive \
    SKIP_INSTALL=NO BUILD_LIBRARY_FOR_DISTRIBUTION=YES

xcodebuild archive \
    -scheme MySDK \
    -destination "generic/platform=iOS Simulator" \
    -archivePath ./build/ios-sim.xcarchive \
    SKIP_INSTALL=NO BUILD_LIBRARY_FOR_DISTRIBUTION=YES

# Create XCFramework
xcodebuild -create-xcframework \
    -framework ./build/ios.xcarchive/Products/Library/Frameworks/MySDK.framework \
    -framework ./build/ios-sim.xcarchive/Products/Library/Frameworks/MySDK.framework \
    -output ./build/MySDK.xcframework

3.3 Binary Target in Package.swift

// Local binary target
.binaryTarget(
    name: "MySDK",
    path: "Frameworks/MySDK.xcframework"
)

// Remote binary target (with checksum for integrity)
.binaryTarget(
    name: "MySDK",
    url: "https://releases.company.com/MySDK-1.2.0.xcframework.zip",
    checksum: "a1b2c3d4e5f6..."
)

// Generate checksum
// swift package compute-checksum MySDK.xcframework.zip
BUILD_LIBRARY_FOR_DISTRIBUTION

This build setting enables module stability by generating .swiftinterface files instead of .swiftmodule files. Swift interface files are textual and forward-compatible, meaning binaries built with one Swift version can be used with a different Swift version. Always enable this for binary distribution.

4. Plugins (Build Tool & Command)

SPM plugins extend the build system with custom functionality. There are two types: build tool plugins (run automatically during builds) and command plugins (run on-demand).

4.1 Build Tool Plugin

// Plugins/SwiftGenPlugin/plugin.swift
import PackagePlugin

@main
struct SwiftGenPlugin: BuildToolPlugin {
    func createBuildCommands(
        context: PluginContext,
        target: Target
    ) async throws -> [Command] {
        guard let target = target as? SourceModuleTarget else {
            return []
        }

        let swiftgenTool = try context.tool(named: "swiftgen")
        let configFile = target.directory.appending("swiftgen.yml")
        let outputDir = context.pluginWorkDirectory

        return [
            .buildCommand(
                displayName: "Running SwiftGen for \(target.name)",
                executable: swiftgenTool.path,
                arguments: [
                    "config", "run",
                    "--config", configFile.string
                ],
                inputFiles: [configFile],
                outputFiles: [outputDir.appending("Generated.swift")]
            )
        ]
    }
}

// In Package.swift:
.target(
    name: "MyApp",
    dependencies: [],
    plugins: [.plugin(name: "SwiftGenPlugin")]
)

4.2 Command Plugin

// Plugins/FormatPlugin/plugin.swift
import PackagePlugin

@main
struct FormatPlugin: CommandPlugin {
    func performCommand(
        context: PluginContext,
        arguments: [String]
    ) async throws {
        let swiftformat = try context.tool(named: "swift-format")

        for target in context.package.targets {
            guard let target = target as? SourceModuleTarget else { continue }

            let process = Process()
            process.executableURL = URL(fileURLWithPath: swiftformat.path.string)
            process.arguments = [
                "format",
                "--in-place",
                "--recursive",
                target.directory.string
            ]
            try process.run()
            process.waitUntilExit()
        }
    }
}

// Run from command line:
// swift package plugin format

// Run from Xcode:
// Right-click package → "Format" (appears in context menu)

5. Resources in SPM

SPM supports bundling resources (images, JSON, localization files) with packages. Resources are accessed through a generated Bundle.module accessor.

5.1 Resource Rules

// In Package.swift target definition
.target(
    name: "MyLibrary",
    resources: [
        // .process — applies platform-specific processing
        // (compiles XIBs, optimizes images, compiles asset catalogs)
        .process("Resources"),

        // .copy — copies as-is, preserving directory structure
        // (for files that shouldn't be processed)
        .copy("Fixtures"),
        .copy("Templates/email.html")
    ]
)

5.2 Accessing Resources

// SPM generates a Bundle extension for each target with resources
// Access via Bundle.module

// Load an image
let image = UIImage(named: "logo", in: .module, with: nil)

// Load a JSON file
guard let url = Bundle.module.url(forResource: "config", withExtension: "json") else {
    fatalError("Missing config.json")
}
let data = try Data(contentsOf: url)

// Load localized strings
let greeting = NSLocalizedString("hello", bundle: .module, comment: "")

// SwiftUI
Image("logo", bundle: .module)
Text("hello", bundle: .module)
Bundle.module Pitfall

Bundle.module is only available in targets that declare resources. If you try to use it in a target without resources, you'll get a compile error. Also, Bundle.module resolves differently in tests — the test bundle contains a copy of the resources, not the original target's bundle. Always use Bundle.module (not Bundle.main) in library code.

6. Conditional Target Dependencies

SPM supports platform-conditional dependencies, allowing packages to have different dependencies on different platforms.

let package = Package(
    name: "CrossPlatformLib",
    products: [
        .library(name: "CrossPlatformLib", targets: ["CrossPlatformLib"])
    ],
    dependencies: [
        .package(url: "https://github.com/example/ios-analytics", from: "1.0.0"),
        .package(url: "https://github.com/example/server-logging", from: "2.0.0")
    ],
    targets: [
        .target(
            name: "CrossPlatformLib",
            dependencies: [
                // Only depend on ios-analytics on Apple platforms
                .product(name: "iOSAnalytics",
                         package: "ios-analytics",
                         condition: .when(platforms: [.iOS, .macOS, .watchOS, .tvOS])),

                // Only depend on server-logging on Linux
                .product(name: "ServerLogging",
                         package: "server-logging",
                         condition: .when(platforms: [.linux]))
            ],
            swiftSettings: [
                // Conditional compilation flags
                .define("USE_UIKIT", .when(platforms: [.iOS, .tvOS])),
                .define("USE_APPKIT", .when(platforms: [.macOS])),

                // Enable upcoming Swift features
                .enableUpcomingFeature("StrictConcurrency"),
                .enableExperimentalFeature("VariadicGenerics")
            ]
        )
    ]
)

7. Package Collections

Package collections are curated lists of Swift packages, signed and published as JSON. They help teams discover and manage approved packages.

# Add a collection
swift package-collection add https://packages.company.com/collection.json

# List collections
swift package-collection list

# Search within collections
swift package-collection search --keyword networking

# Get info about a specific package
swift package-collection describe https://github.com/apple/swift-algorithms

7.1 Creating a Collection

// collection.json structure
{
    "name": "Company Approved Packages",
    "overview": "Vetted packages for internal use",
    "packages": [
        {
            "url": "https://github.com/apple/swift-algorithms",
            "summary": "Sequence and collection algorithms",
            "versions": [
                {
                    "version": "1.2.0",
                    "manifests": { /* ... */ }
                }
            ]
        }
    ],
    "formatVersion": "1.0",
    "generatedAt": "2024-01-15T00:00:00Z"
}

// Sign the collection
// swift package-collection sign collection.json signed.json cert.pem key.pem

8. Registry Support

Swift 5.7 introduced package registry support (SE-0292), allowing packages to be resolved from a registry server instead of (or in addition to) Git repositories.

8.1 Registry vs Git-Based Resolution

Aspect Git-Based Registry-Based
Resolution speed Slow (clones repos) Fast (downloads metadata + source archives)
Disk usage High (full git history) Low (only needed version)
Offline support Cached clones Cached archives
Security Git commit hashes Signed manifests + checksums
Private packages SSH/HTTPS auth Token-based auth 
// Configure registry in .swiftpm/configuration/registries.json
{
    "registries": {
        "[default]": {
            "url": "https://packages.company.com"
        },
        "company": {
            "url": "https://internal-registry.company.com"
        }
    },
    "version": 1
}

// In Package.swift, use scope-based dependency
.package(id: "company.internal-sdk", from: "1.0.0")

9. SPM in Xcode Workspace

When SPM packages are used in Xcode, the integration has specific behaviors that differ from command-line SPM.

9.1 Xcode SPM Integration

Xcode SPM Integration: MyApp.xcodeproj ├── Package Dependencies (resolved by Xcode) │ ├── swift-algorithms (remote) │ └── swift-collections (remote) │ ├── Local Packages (drag-and-drop into navigator) │ └── SharedKit (local path reference) │ └── Derived Data └── SourcePackages/ ├── checkouts/ ← Cloned repositories │ ├── swift-algorithms/ │ └── swift-collections/ ├── artifacts/ ← Downloaded binary artifacts └── repositories/ ← Bare git repos (for resolution) 
# Xcode clones packages to a shared location
# Override for CI:
xcodebuild -clonedSourcePackagesDirPath ./build/SourcePackages build

# Resolve without building
xcodebuild -resolvePackageDependencies \
    -workspace MyApp.xcworkspace \
    -scheme MyApp

10. Architect: SPM for a Large Mono-Repo

Architect Must-Know

At Staff level, you need to design the SPM architecture for a large mono-repo with 100+ modules, multiple apps, and shared libraries. Here's a battle-tested approach.

10.1 Mono-Repo Package Structure

company-ios/ ├── Package.swift ← Root package (umbrella) ├── Apps/ │ ├── MainApp/ │ │ └── Package.swift ← App-specific package │ ├── InternalApp/ │ │ └── Package.swift │ └── WidgetExtension/ │ └── Package.swift ├── Modules/ │ ├── Core/ │ │ ├── Networking/ │ │ │ ├── Package.swift ← Each module is a package │ │ │ ├── Sources/ │ │ │ └── Tests/ │ │ ├── Storage/ │ │ │ └── Package.swift │ │ └── Analytics/ │ │ └── Package.swift │ ├── Features/ │ │ ├── Home/ │ │ │ └── Package.swift │ │ ├── Search/ │ │ │ └── Package.swift │ │ └── Profile/ │ │ └── Package.swift │ └── Shared/ │ ├── DesignSystem/ │ │ └── Package.swift │ └── TestHelpers/ │ └── Package.swift ├── Plugins/ │ ├── LintPlugin/ │ │ └── Package.swift │ └── CodeGenPlugin/ │ └── Package.swift └── Tools/ └── dependency-graph/ ← Custom tooling └── Package.swift

10.2 Dependency Rules

// Enforce layered architecture via dependency rules

// Rule 1: Features depend on Core, never on other Features
// Rule 2: Core modules can depend on other Core modules
// Rule 3: Shared modules have no dependencies on Features or Core
// Rule 4: Apps depend on Features, never directly on Core

// Modules/Features/Home/Package.swift
let package = Package(
    name: "Home",
    platforms: [.iOS(.v16)],
    products: [
        .library(name: "Home", targets: ["Home"]),
        .library(name: "HomeInterface", targets: ["HomeInterface"])
    ],
    dependencies: [
        .package(path: "../../Core/Networking"),
        .package(path: "../../Core/Storage"),
        .package(path: "../../Shared/DesignSystem"),
    ],
    targets: [
        // Interface target — only protocols and models
        .target(
            name: "HomeInterface",
            dependencies: []
        ),
        // Implementation target
        .target(
            name: "Home",
            dependencies: [
                "HomeInterface",
                .product(name: "Networking", package: "Networking"),
                .product(name: "Storage", package: "Storage"),
                .product(name: "DesignSystem", package: "DesignSystem"),
            ]
        ),
        .testTarget(
            name: "HomeTests",
            dependencies: [
                "Home",
                .package(path: "../../Shared/TestHelpers")
            ]
        )
    ]
)

10.3 CI Optimization for SPM

#!/bin/bash
# ci-spm-cache.sh — Cache SPM dependencies across CI runs

SPM_CACHE_DIR="./build/SourcePackages"
CACHE_KEY="spm-$(shasum Package.resolved | cut -d' ' -f1)"

# Restore cache
if cache-tool restore "$CACHE_KEY" "$SPM_CACHE_DIR"; then
    echo "SPM cache restored"
fi

# Resolve dependencies (uses cache if available)
xcodebuild -resolvePackageDependencies \
    -clonedSourcePackagesDirPath "$SPM_CACHE_DIR"

# Save cache
cache-tool save "$CACHE_KEY" "$SPM_CACHE_DIR"
SPM Mono-Repo Best Practices
  • Use interface/implementation split for feature modules to break dependency cycles
  • Keep the dependency graph wide and shallow for maximum build parallelism
  • Pin all external dependencies to exact versions in Package.resolved
  • Use local path dependencies for internal modules (not versioned)
  • Create a custom tool to validate dependency rules (no cross-feature deps)
  • Cache SourcePackages directory on CI keyed by Package.resolved hash
  • Use build tool plugins for code generation instead of script phases
  • Separate test helpers into a dedicated package for reuse
