1. LLDB Deep Dive

LLDB is Apple's debugger, built on the LLVM project. It's far more powerful than most developers realize — it can evaluate arbitrary expressions, script with Python, inspect memory layout, and even modify running code.

1.1 LLDB Architecture

LLDB Architecture: ┌─────────────────────────────────────────────────────────────┐ │ Xcode IDE │ │ (Debug navigator, variables view, console) │ └──────────────────────────┬──────────────────────────────────┘ │ Debug Adapter Protocol ▼ ┌─────────────────────────────────────────────────────────────┐ │ LLDB │ │ │ │ ┌──────────────┐ ┌──────────────┐ ┌──────────────────┐ │ │ │ Command │ │ Expression │ │ Python Scripting │ │ │ │ Interpreter │ │ Evaluator │ │ Bridge │ │ │ └──────┬───────┘ └──────┬───────┘ └──────────────────┘ │ │ │ │ │ │ ┌──────┴─────────────────┴──────────────────────────────┐ │ │ │ Target / Process / Thread │ │ │ │ Frame / Variable / Type │ │ │ └────────────────────────┬──────────────────────────────┘ │ │ │ │ └───────────────────────────┼──────────────────────────────────┘ │ ptrace / Mach exceptions ▼ ┌─────────────────────────────────────────────────────────────┐ │ Debugged Process │ │ (MyApp running on device/simulator) │ └─────────────────────────────────────────────────────────────┘

1.2 Essential LLDB Commands

# Navigation
(lldb) thread list                    # List all threads
(lldb) thread select 3                # Switch to thread 3
(lldb) bt                             # Backtrace current thread
(lldb) bt all                         # Backtrace all threads
(lldb) frame select 2                 # Switch to frame 2
(lldb) frame variable                 # Show all local variables

# Execution control
(lldb) continue                       # Resume execution (c)
(lldb) step                           # Step into (s)
(lldb) next                           # Step over (n)
(lldb) finish                         # Step out of current function
(lldb) thread return                  # Return from current function immediately

# Memory inspection
(lldb) memory read 0x7fff5fbff8a0     # Read memory at address
(lldb) memory read -s4 -fx -c8 ptr    # Read 8 x 4-byte hex values
(lldb) x/16xb ptr                     # Read 16 bytes in hex (GDB syntax)

# Type inspection
(lldb) type lookup MyClass            # Show type layout
(lldb) image lookup -t MyProtocol     # Find type definition
(lldb) image lookup -n myFunction     # Find function by name
(lldb) image lookup -a 0x100004000    # Find symbol at address

# Register inspection
(lldb) register read                  # All registers
(lldb) register read x0               # Specific register (ARM64)
(lldb) register read rax              # Specific register (x86_64)

2. po vs v vs p

These three LLDB commands look similar but work completely differently under the hood. Understanding the differences is critical for efficient debugging.

2.1 Comparison

Command Full Name How It Works Speed Side Effects
v frame variable Reads debug info directly from memory — no code execution Fastest None
p expression Compiles and executes a Swift expression in the debuggee Slow Can have side effects
po expression -O Like p but calls debugDescription on the result Slowest Can have side effects

2.2 When to Use Each

// Given this code paused at a breakpoint:
let user = User(name: "Alice", age: 30)
let items = [1, 2, 3, 4, 5]

// v — Use for simple variable inspection (FASTEST)
(lldb) v user
// (User) user = (name = "Alice", age = 30)
// Reads directly from debug info — no compilation, no execution

(lldb) v user.name
// (String) user.name = "Alice"

(lldb) v items
// ([Int]) items = 5 values { [0] = 1, [1] = 2, ... }

// p — Use when you need to evaluate an expression
(lldb) p user.name.count
// (Int) $R0 = 5
// Compiles "user.name.count" as Swift, executes it

(lldb) p items.filter { $0 > 2 }
// ([Int]) $R1 = [3, 4, 5]
// Compiles and runs the filter closure

// po — Use when you want custom description output
(lldb) po user
// User(name: "Alice", age: 30)
// Calls user.debugDescription (or description)

(lldb) po items
// [1, 2, 3, 4, 5]
// Calls Array's description
Why v Is So Much Faster

v reads variable values directly from DWARF debug info and the process's memory. It doesn't compile any code. p and po both spin up a Swift compiler instance, compile your expression, inject the compiled code into the debuggee process, execute it, and read back the result. This is why p can take seconds on the first invocation (compiler warm-up) while v is instant. Use v as your default and only switch to p/po when you need expression evaluation.

2.3 Advanced Expression Evaluation

# Evaluate multi-line expressions
(lldb) expression
Enter expressions, then terminate with an empty line:
  let filtered = items.filter { $0 > 2 }
  print("Count: \(filtered.count)")

# Import a module for expression evaluation
(lldb) expression import Foundation
(lldb) expression import UIKit

# Cast and inspect
(lldb) expression let vc = unsafeBitCast(0x7f8b2c, to: UIViewController.self)
(lldb) po vc.view.subviews

# Modify variables at runtime
(lldb) expression user.name = "Bob"
(lldb) expression items.append(6)

# Call methods
(lldb) expression UIApplication.shared.windows.first?.rootViewController

3. Breakpoint Actions & Conditions

Breakpoints in LLDB are far more powerful than simple "stop here" markers. They can have conditions, actions, and auto-continue behavior that turns them into powerful debugging instruments.

3.1 Conditional Breakpoints

# Break only when condition is true
(lldb) breakpoint set -f ViewController.swift -l 42 -c "index > 100"

# Break on Nth hit
(lldb) breakpoint set -f ViewController.swift -l 42 -i 100
# Skips first 99 hits, breaks on 100th

# Break on function with condition
(lldb) breakpoint set -n "processItem" -c "item.priority == .high"

3.2 Breakpoint Actions

# Log and continue (non-intrusive logging)
(lldb) breakpoint set -f NetworkManager.swift -l 55
(lldb) breakpoint command add 1
  expression print("Request URL: \(request.url!)")
  continue
  DONE

# Play a sound on breakpoint hit
(lldb) breakpoint set -n "didReceiveMemoryWarning"
(lldb) breakpoint command add 2
  shell say "Memory warning!"
  DONE

# Capture backtrace to file
(lldb) breakpoint set -n "crash_handler"
(lldb) breakpoint command add 3
  bt all
  script import datetime; open('/tmp/crash_' + str(datetime.datetime.now()) + '.txt', 'w').write(lldb.debugger.HandleCommand('bt all'))
  DONE

3.3 Watchpoints

# Break when a variable's value changes
(lldb) watchpoint set variable user.name
# Breaks whenever user.name is written to

# Watch a memory address
(lldb) watchpoint set expression -w write -- &myGlobal

# Watch with condition
(lldb) watchpoint modify -c "user.name == \"error\""

4. Symbolic Breakpoints

Symbolic breakpoints break on function names rather than file/line locations. They're invaluable for debugging system framework behavior and catching common issues.

4.1 Essential Symbolic Breakpoints

Symbol Purpose When to Use
UIViewAlertForUnsatisfiableConstraints Auto Layout constraint conflicts Debugging layout issues
-[UIApplication main] App launch entry point Debugging launch issues
objc_exception_throw ObjC exception thrown Catching NSExceptions before crash
swift_willThrow Swift error about to be thrown Catching Swift errors at throw site
-[NSObject doesNotRecognizeSelector:] Unrecognized selector sent Debugging selector crashes
malloc_error_break Malloc detected corruption Debugging memory corruption
_UIApplicationMainPreparations Before UIApplication setup Measuring pre-main time 
# Set symbolic breakpoints from LLDB
(lldb) breakpoint set -n "UIViewAlertForUnsatisfiableConstraints"
(lldb) breakpoint set -n "objc_exception_throw"
(lldb) breakpoint set -n "swift_willThrow"

# Break on all methods of a class
(lldb) breakpoint set -r "\[MyViewController .*\]"

# Break on Swift method (mangled name)
(lldb) breakpoint set -n "$s7MyApp14ViewControllerC12viewDidLoadyyF"

# Break on any method containing "network" (case insensitive)
(lldb) breakpoint set -r -i "network"

# In Xcode: Debug → Breakpoints → Create Symbolic Breakpoint
# Symbol: UIViewAlertForUnsatisfiableConstraints
# Action: po [[UIWindow keyWindow] _autolayoutTrace]
Must-Have Symbolic Breakpoints

Add these to your Xcode user breakpoints (shared across all projects): UIViewAlertForUnsatisfiableConstraints with action po [[UIWindow keyWindow] _autolayoutTrace], objc_exception_throw, and swift_willThrow. These catch the most common issues before they become hard-to-debug crashes.

6. Address Sanitizer (ASan)

ASan detects memory errors at runtime: use-after-free, buffer overflows, stack overflows, and more. It instruments every memory access with bounds checking.

6.1 How ASan Works

ASan Memory Layout: Normal memory: ┌────────────────────────────────────┐ │ Application Memory │ │ [allocated] [free] [allocated] │ └────────────────────────────────────┘ With ASan: ┌────────────────────────────────────┐ │ Application Memory │ │ [alloc][REDZONE][free][REDZONE] │ ← Redzones around allocations └──────────────────┬─────────────────┘ │ maps to ▼ ┌────────────────────────────────────┐ │ Shadow Memory (1/8 of app memory)│ │ Each byte tracks 8 bytes of app │ │ memory: accessible or poisoned │ └────────────────────────────────────┘ Every memory access: 1. Compute shadow address: shadow = (addr >> 3) + offset 2. Check shadow byte: if poisoned → REPORT ERROR 3. If clean → proceed with access

6.2 What ASan Detects

Error Type Description Example
Heap buffer overflow Reading/writing past allocation end array[array.count] in C/ObjC
Stack buffer overflow Overflowing stack-allocated buffers C array out of bounds
Use after free Accessing freed memory Dangling pointer dereference
Use after return Accessing stack memory after function returns Returning pointer to local variable
Use after scope Accessing variable outside its scope Capturing stack variable in escaping closure
Double free Freeing the same memory twice Manual memory management errors 
// Enable ASan in Xcode:
// Product → Scheme → Edit Scheme → Run → Diagnostics → Address Sanitizer

// Or via build settings:
OTHER_CFLAGS = -fsanitize=address
OTHER_SWIFT_FLAGS = -sanitize=address
OTHER_LDFLAGS = -fsanitize=address

// ASan overhead:
// - 2x memory usage (shadow memory)
// - 2-3x CPU slowdown
// - Not suitable for production
ASan + Swift

Pure Swift code has fewer memory safety issues than C/ObjC because of ARC and bounds checking. However, ASan is still valuable for Swift apps that use C/ObjC interop, unsafe pointers, or third-party C libraries. It also catches issues in system frameworks that your app triggers.

7. Thread Sanitizer (TSan)

TSan detects data races — when two threads access the same memory location concurrently and at least one access is a write, without synchronization. Data races are undefined behavior and can cause crashes, data corruption, and heisenbugs.

7.1 How TSan Works

TSan Detection: For every memory access, TSan records: • Thread ID • Memory address • Access type (read/write) • Timestamp (logical clock) Thread 1: Thread 2: write(x) read(x) timestamp: 5 timestamp: 7 TSan checks: Is there a happens-before relationship? • If Thread 1's write at T=5 is synchronized with Thread 2's read at T=7 → OK • If no synchronization → DATA RACE DETECTED TSan reports: WARNING: ThreadSanitizer: data race Write of size 8 at 0x7f8b2c by thread T1: #0 MyClass.updateState() MyClass.swift:42 Previous read of size 8 at 0x7f8b2c by thread T2: #0 MyClass.readState() MyClass.swift:58

7.2 Common Data Races in Swift

// Race 1: Shared mutable state without synchronization
class Counter {
    var count = 0  // ← TSan will flag concurrent access

    func increment() {
        count += 1  // Not atomic — read + write
    }
}

// Fix: Use actor or synchronization
actor Counter {
    var count = 0
    func increment() { count += 1 }
}

// Race 2: Dictionary access from multiple threads
var cache = [String: Data]()  // ← Not thread-safe

// Fix: Use a serial queue or actor
let cacheQueue = DispatchQueue(label: "cache")
cacheQueue.sync { cache[key] = value }

// Race 3: Delegate callback on wrong thread
class NetworkManager {
    weak var delegate: NetworkDelegate?  // ← Set on main, called on background
}
// Enable TSan:
// Product → Scheme → Edit Scheme → Run → Diagnostics → Thread Sanitizer

// TSan overhead:
// - 5-15x CPU slowdown
// - 5-10x memory usage
// - Cannot combine with ASan (mutually exclusive)
// - Simulator only (not supported on device)
TSan + Swift Concurrency

TSan understands Swift concurrency primitives (actors, async/await, Task). It knows that actor-isolated state is protected and won't report false positives for actor properties. However, it will correctly flag nonisolated access to mutable state and Sendable violations that the compiler doesn't catch (e.g., with @unchecked Sendable).

8. Undefined Behavior Sanitizer (UBSan)

UBSan detects undefined behavior in C, C++, and Objective-C code at runtime. While Swift prevents most UB by design, UBSan is valuable for mixed-language projects and C interop.

8.1 What UBSan Detects

UB Type Description Swift Relevance
Integer overflow Signed integer overflow C interop, UnsafePointer arithmetic
Null pointer dereference Dereferencing NULL C interop, UnsafePointer
Misaligned pointer Accessing misaligned memory UnsafeRawPointer.load(as:)
Invalid shift Shift amount >= bit width Bitwise operations in C code
Invalid enum value Enum with out-of-range value C enums bridged to Swift
Implicit conversion Value doesn't fit in target type C numeric conversions 
// Enable UBSan:
// Product → Scheme → Edit Scheme → Run → Diagnostics → Undefined Behavior Sanitizer

// Build settings:
OTHER_CFLAGS = -fsanitize=undefined
OTHER_SWIFT_FLAGS = -sanitize=undefined

// UBSan overhead: minimal (5-20% CPU, negligible memory)
// Can be combined with ASan

9. Malloc Debugging

macOS/iOS provide several malloc debugging features that help detect memory corruption, leaks, and use-after-free bugs.

9.1 Guard Malloc

// Guard Malloc places each allocation on its own virtual memory page
// with guard pages before and after. Any overflow immediately crashes
// with a clear stack trace.

// Enable in Xcode:
// Product → Scheme → Edit Scheme → Run → Diagnostics → Malloc Guard Edges

// Or via environment variable:
// DYLD_INSERT_LIBRARIES=/usr/lib/libgmalloc.dylib

// Guard Malloc overhead: EXTREME
// Each allocation uses at least one 16KB page
// Only usable for targeted debugging, not general testing

9.2 Malloc Scribble

// Malloc Scribble fills freed memory with 0x55 and allocated memory with 0xAA
// This makes use-after-free bugs crash immediately instead of silently corrupting

// Enable in Xcode:
// Product → Scheme → Edit Scheme → Run → Diagnostics → Malloc Scribble

// Environment variables:
MallocScribble=1          // Fill freed memory with 0x55
MallocPreScribble=1       // Fill allocated memory with 0xAA

// Other useful malloc environment variables:
MallocStackLogging=1      // Record allocation stack traces
MallocStackLoggingNoCompact=1  // Keep all stack traces (for leaks)
MallocCheckHeapStart=100  // Check heap integrity every N allocations
MallocCheckHeapEach=100

9.3 Malloc Stack Logging

// Enable malloc stack logging for memory leak detection
// Xcode: Product → Scheme → Edit Scheme → Run → Diagnostics → Malloc Stack Logging

// Then use the Memory Graph Debugger or leaks tool:
// Debug → Debug Memory Graph (in Xcode)

# Command-line leak detection

11. Architect: Sanitizers in CI

Architect Must-Know

At Staff level, you should set up sanitizers in CI to catch memory and threading bugs before they reach production. Here's a production-grade CI sanitizer configuration.

11.1 CI Sanitizer Strategy

CI Pipeline with Sanitizers: PR Opened / Push to main │ ├─── Job 1: Regular Tests (fast, every PR) │ └─ xcodebuild test (no sanitizers) │ ├─── Job 2: ASan Tests (memory safety) │ └─ xcodebuild test with -enableAddressSanitizer YES │ └─ Runs on: every PR to main │ ├─── Job 3: TSan Tests (thread safety) │ └─ xcodebuild test with -enableThreadSanitizer YES │ └─ Runs on: every PR to main │ └─── Job 4: UBSan Tests (undefined behavior) └─ xcodebuild test with -enableUndefinedBehaviorSanitizer YES └─ Runs on: nightly (low overhead, but extra coverage)

11.2 CI Build Scripts

#!/bin/bash
# ci-sanitizer-tests.sh

set -euo pipefail

WORKSPACE="MyApp.xcworkspace"
SCHEME="MyApp"
DESTINATION="platform=iOS Simulator,name=iPhone 15,OS=17.0"

# ASan test run
echo "=== Running Address Sanitizer Tests ==="
xcodebuild test \
    -workspace "$WORKSPACE" \
    -scheme "$SCHEME" \
    -destination "$DESTINATION" \
    -enableAddressSanitizer YES \
    -resultBundlePath ./results/asan.xcresult \
    | xcbeautify

# TSan test run (separate — can't combine with ASan)
echo "=== Running Thread Sanitizer Tests ==="
xcodebuild test \
    -workspace "$WORKSPACE" \
    -scheme "$SCHEME" \
    -destination "$DESTINATION" \
    -enableThreadSanitizer YES \
    -resultBundlePath ./results/tsan.xcresult \
    | xcbeautify

# UBSan test run
echo "=== Running UB Sanitizer Tests ==="
xcodebuild test \
    -workspace "$WORKSPACE" \
    -scheme "$SCHEME" \
    -destination "$DESTINATION" \
    -enableUndefinedBehaviorSanitizer YES \
    -resultBundlePath ./results/ubsan.xcresult \
    | xcbeautify

11.3 Handling Sanitizer Failures

// Suppress known false positives (use sparingly)

// TSan suppression file (tsan_suppressions.txt)
// race:ThirdPartyLib*
// race:legacy_module::*

// Set via environment variable:
// TSAN_OPTIONS=suppressions=tsan_suppressions.txt

// ASan suppression for known third-party issues:
// ASAN_OPTIONS=suppressions=asan_suppressions.txt

// Better approach: fix the issue or isolate the code
// Suppressions should be temporary with tracking tickets
Sanitizer CI Best Practices
  • Run ASan and TSan on every PR — they catch the most impactful bugs
  • Run UBSan nightly — lower priority but still valuable
  • ASan and TSan cannot run simultaneously — use separate CI jobs
  • TSan only works on simulator — plan test matrix accordingly
  • Set sanitizer tests as required checks for PR merge
  • Track and fix all sanitizer findings — don't suppress without a ticket
  • Expect 2-3x longer test runs with ASan, 5-15x with TSan
  • Use xcresult bundles to archive sanitizer reports
