1. The Three-Phase Layout: Propose → Measure → Place

SwiftUI layout is a recursive negotiation between parents and children. It is not UIKit Auto Layout’s constraint solver expressed in Swift; there is no global multi-pass “satisfy all constraints” phase. At L6/staff depth, narrate three phases that repeat down the tree:

  1. Proposal — The parent proposes a ProposedViewSize to each child. Width and height are each an optional CGFloat: concrete, nil (unspecified), 0, or .infinity when the axis is unbounded for measurement.
  2. Measurement — Each child answers “how big would I like to be?” by combining its intrinsic (ideal) behavior with the proposal. For custom containers this maps to Layout.sizeThatFits and child LayoutSubview.sizeThatFits. Built-in stacks implement the same idea internally.
  3. Placement — The parent assigns each child an origin (and often a final size) in its own coordinate space. Custom layouts do this in placeSubviews via place(at:proposal:).

The critical staff-level point: the parent owns policy. Children report sizes and alignment contributions; they do not “pull” arbitrary frames from ancestors except through the negotiated proposal/placement pipeline (and modifiers that reshape proposals).

Sound bite for the loop

“Parent proposes, child measures, parent places — recursively.” If asked what breaks the model, cite GeometryReader greediness, fixedSize overriding compression, or ViewThatFits branching measurement order.

2. Negotiation Walkthrough & ASCII Models

2.1 Single parent, two children (conceptual)

┌────────────────────────────────────────────────────────────────────────────┐ │ PARENT (e.g. HStack) │ │ │ │ proposal₁ ──► CHILD A ──► sizeA (CGSize) │ │ proposal₂ ──► CHILD B ──► sizeB (CGSize) │ │ │ │ Parent aggregates (sum widths, max height, + spacing) → own sizeThatFits │ │ Parent assigns origins: A at x0, B at x0 + sizeA.width + gap │ └────────────────────────────────────────────────────────────────────────────┘

2.2 Depth-first recursion

ROOT proposes to Column │ ▼ ┌─────────┐ │ VStack │──── proposes to Row ────► Leaf measures └─────────┘ │ ▲ ▼ │ ┌─────────┐ └────────────│ HStack │──── proposes to each chip └─────────┘ Order: measure leaves first (as driven by parent algorithms), bubble sizes up, then place top-down. Lazy containers defer work until near visibility.

2.3 Where “ideal size” enters

Every view has a notion of how big it wants to be when unconstrained on an axis. Text uses font + string + line limits; Image uses asset dimensions (unless resizable); flexible shapes expand to the proposal. Modifiers wrap the child and transform proposals, so the same leaf can report different sizes depending on its modifier stack.

3. ProposedViewSize: zero, infinity, unspecified

ProposedViewSize is the currency of layout. Along each axis the proposal is optional: a CGFloat, nil, 0, or CGFloat.infinity (conceptually “unbounded for this measurement pass”).

Proposal Typical meaning Child behavior (rule of thumb)
Concrete width/height “You are being offered this much space on this axis.” Text wraps; images scale per resizable rules; flexible views often fill.
nil (unspecified) “No opinion — report your natural/intrinsic size.” Ideal sizes dominate; used when parent sizes from children.
0 “How small can you become?” (tight probe) Minimum layouts; useful in custom Layout for bin-packing tests.
.infinity “Unbounded along this axis for measurement.” Single-line text width; scroll content; avoid confusing with screen size.
Infinity vs nil — interview trap

Both can yield large measurements, but they encode different parental intent. nil asks for natural sizing; infinity often appears when a scroll view or lazy container needs “maximum line width” style measurement. Mixing them up reads as shallow API familiarity.

// Illustrative: asking a subview for ideal vs bounded width
let natural = subview.sizeThatFits(.unspecified)
let wrapped = subview.sizeThatFits(ProposedViewSize(width: 200, height: nil))

4. Built-in Views: Text, Image, Color, Spacer

View Measurement behavior Staff-level note
Text String + font + environment; wraps when width is finite. Dynamic Type and locale change intrinsic size; test edge strings.
Image Intrinsic pixel size unless resizable + scaling. SF Symbols scale with font; template rendering affects hit targets.
Color / Rectangle Expands to fill proposal — “flexible cheap fills.” Great for backgrounds; pair with frame to avoid surprise expansion.
Spacer Minimum 0; absorbs remaining space per stack policy. Not magic — the stack’s algorithm distributes flexible gaps. 
// Text respects finite width; Color expands unless constrained
VStack(alignment: .leading, spacing: 8) {
    Text("This line wraps when the parent proposes a bounded width.")
        .border(.red)
    Color.blue
        .frame(height: 40)
    Spacer(minLength: 0)
}

5. Layout Protocol (iOS 16+)

The Layout protocol is how you become the parent: you receive a Subviews collection of LayoutSubview proxies, measure them under proposals, return your container’s size, then assign child positions.

5.1 Core requirements

  • sizeThatFits(proposal:subviews:cache:) — Return the size of your container given the incoming proposal and measured children.
  • placeSubviews(in:proposal:subviews:cache:) — Position each subview inside bounds using place(at:proposal:).

5.2 Cache: makeCache, updateCache, expensive work

Use the cache when your algorithm would otherwise repeat O(n²) work across sizeThatFits and placeSubviews, or when you need stable scratch storage for wrapping indices, prefix sums, or radix passes. The cache type is yours; SwiftUI passes it inout.

What “good” looks like

Factor shared packing logic (e.g. row breaking) into one function invoked from both sizeThatFits and placeSubviews, backed by a cache struct. Interviewers listen for that symmetry — measurement and placement must agree.

5.3 Complete example: horizontal flow layout with wrapping

import SwiftUI

struct FlowLayout: Layout {
    var spacing: CGFloat = 8
    var lineSpacing: CGFloat = 8

    struct Cache {
        var sizes: [CGSize] = []
        var rows: [[Int]] = []
        var rowHeights: [CGFloat] = []
        var totalSize: CGSize = .zero
    }

    func makeCache(subviews: Subviews) -> Cache { Cache() }

    func updateCache(_ cache: inout Cache, subviews: Subviews) {
        cache.sizes = subviews.map { $0.sizeThatFits(.unspecified) }
    }

    func sizeThatFits(
        proposal: ProposedViewSize,
        subviews: Subviews,
        cache: inout Cache
    ) -> CGSize {
        pack(proposal: proposal, subviews: subviews, cache: &cache)
        return cache.totalSize
    }

    func placeSubviews(
        in bounds: CGRect,
        proposal: ProposedViewSize,
        subviews: Subviews,
        cache: inout Cache
    ) {
        pack(proposal: proposal, subviews: subviews, cache: &cache)
        var y = bounds.minY
        for (rowIndex, row) in cache.rows.enumerated() {
            let rowHeight = cache.rowHeights[rowIndex]
            var x = bounds.minX
            for i in row {
                let size = cache.sizes[i]
                subviews[i].place(
                    at: CGPoint(x: x, y: y + (rowHeight - size.height) / 2),
                    proposal: ProposedViewSize(width: size.width, height: size.height)
                )
                x += size.width + spacing
            }
            y += rowHeight + lineSpacing
        }
    }

    private func pack(proposal: ProposedViewSize, subviews: Subviews, cache: inout Cache) {
        guard !subviews.isEmpty else {
            cache.rows = []; cache.rowHeights = []; cache.totalSize = .zero; return
        }
        updateCache(&cache, subviews: subviews)
        let maxWidth = proposal.width ?? .infinity
        cache.rows = []
        cache.rowHeights = []
        var row: [Int] = []
        var rowWidth: CGFloat = 0
        var rowHeight: CGFloat = 0

        for idx in subviews.indices {
            let size = cache.sizes[idx]
            let extra = row.isEmpty ? 0 : spacing
            if rowWidth + extra + size.width > maxWidth, !row.isEmpty {
                cache.rows.append(row)
                cache.rowHeights.append(rowHeight)
                row = []
                rowWidth = 0
                rowHeight = 0
            }
            if !row.isEmpty { rowWidth += spacing }
            row.append(idx)
            rowWidth += size.width
            rowHeight = max(rowHeight, size.height)
        }
        cache.rows.append(row)
        cache.rowHeights.append(rowHeight)

        let totalWidth = cache.rows.map { r in
            r.map { cache.sizes[$0].width }.reduce(0, +)
                + CGFloat(max(r.count - 1, 0)) * spacing
        }.max() ?? 0
        let totalHeight = cache.rowHeights.reduce(0, +)
            + CGFloat(max(cache.rowHeights.count - 1, 0)) * lineSpacing
        cache.totalSize = CGSize(width: totalWidth, height: totalHeight)
    }
}

5.4 Complete example: radial layout

struct RadialLayout: Layout {
    var radius: CGFloat

    func sizeThatFits(
        proposal: ProposedViewSize,
        subviews: Subviews,
        cache: inout ()
    ) -> CGSize {
        let maxChild = subviews
            .map { $0.sizeThatFits(proposal) }
            .max(by: { $0.width * $0.height < $1.width * $1.height }) ?? .zero
        let side = radius * 2 + max(maxChild.width, maxChild.height)
        return CGSize(width: side, height: side)
    }

    func placeSubviews(
        in bounds: CGRect,
        proposal: ProposedViewSize,
        subviews: Subviews,
        cache: inout ()
    ) {
        let center = CGPoint(x: bounds.midX, y: bounds.midY)
        let count = subviews.count
        guard count > 0 else { return }
        for (index, sub) in subviews.enumerated() {
            let angle = (CGFloat(index) / CGFloat(count)) * 2 * .pi - .pi / 2
            let childSize = sub.sizeThatFits(proposal)
            let point = CGPoint(
                x: center.x + cos(angle) * radius - childSize.width / 2,
                y: center.y + sin(angle) * radius - childSize.height / 2
            )
            sub.place(
                at: point,
                proposal: ProposedViewSize(width: childSize.width, height: childSize.height)
            )
        }
    }
}

5.5 LayoutSubview: dimensions, priorities, and place

LayoutSubview is the stand-in for a child inside a Layout. Typical operations:

  • sizeThatFits(_ proposal: ProposedViewSize) -> CGSize — Measure under a proposal without committing placement.
  • dimensions(in proposal: ProposedViewSize) -> ViewDimensions — Richer measurement including alignment guide contributions.
  • place(at:proposal:anchor:) — Set position; the proposal often matches the size you intend the child to occupy after layout.
  • Subscript with LayoutValueKey — Read per-child metadata (see section 10).

Staff-level detail: when you call place, you are both positioning and re-proposing a size for the child’s own subtree. Mismatch between what you measured and what you place is a common bug source (clipping, mis-centered labels).

6. GeometryReader (and Why It’s Often Misused)

GeometryReader provides a GeometryProxy with size, safe area, and coordinate-space transforms. Mechanically, it answers “what did my parent propose?” and forwards layout to its content.

6.1 Why it is “greedy”

In stacks, a reader often expands to occupy all flexible space along axes where the parent’s proposal allows growth. That means the reader’s reported size can be much larger than the intrinsic size of the content inside the closure — siblings see a large view, not a tight label.

Parent proposes: width = 300, vertical axis flexible │ ▼ ┌──────────────────────┐ │ GeometryReader │ occupies full proposal (greedy) └──────────┬───────────┘ │ closure uses proxy.size

6.2 Misuse: simple relative sizing

// MISUSE — reader becomes layout backbone for one Text
GeometryReader { geo in
    Text("Hello")
        .font(.system(size: geo.size.width * 0.1))
}

6.3 Prefer: frames, typography, and Layout

// BETTER — explicit width policy + semantic text style
Text("Hello")
    .font(.title)
    .frame(maxWidth: .infinity, alignment: .leading)

6.4 Scoped reader pattern (when you need pixels)

// Isolate greed: fixed height band, reader only for chart math
ChartContainer()
    .frame(height: 220)
    .overlay {
        GeometryReader { proxy in
            Canvas { context, size in
                // draw using proxy.size — bounded by overlay parent
            }
        }
    }

6.5 When GeometryReader is appropriate

  • Custom drawing whose strokes must track the container bounds.
  • Draggable overlays, parallax, hit testing mapped to parent coordinates.
  • Bridging precise rects to UIKit interop.
  • One-off effects where a custom Layout is heavier than the problem warrants — but be ready to justify the tradeoff.
Red flag phrase

“I wrap everything in GeometryReader.” Prefer policy in Layout, ViewThatFits, and modifiers; use readers at intentional boundaries.

7. alignmentGuide & Custom Alignments

Stacks align children along shared alignment lines (HorizontalAlignment, VerticalAlignment). alignmentGuide offsets where a particular child contributes its guide value relative to defaults.

7.1 Custom alignment IDs

Define an AlignmentID with defaultValue(in:), then expose a static on the alignment enum via extension.

private enum BubbleAlignment: AlignmentID {
    static func defaultValue(in context: ViewDimensions) -> CGFloat {
        context[VerticalAlignment.center]
    }
}

extension VerticalAlignment {
    static let bubbleTip = VerticalAlignment(BubbleAlignment.self)
}

HStack(alignment: .bubbleTip) {
    Image(systemName: "person.circle.fill")
        .alignmentGuide(.bubbleTip) { d in d[VerticalAlignment.center] }
    VStack(alignment: .leading) {
        Text("Title").font(.headline)
        Text("Subtitle").foregroundStyle(.secondary)
    }
    .alignmentGuide(.bubbleTip) { d in d[VerticalAlignment.top] + 6 }
}

7.2 Complex scenario: timeline markers

For timelines, you often want icons and text to share a vertical line while baselines differ. Custom vertical alignment + guides beat hard-coded padding when Dynamic Type doubles line counts.

private enum TimelineRail: AlignmentID {
    static func defaultValue(in context: ViewDimensions) -> CGFloat {
        context[HorizontalAlignment.leading]
    }
}

extension HorizontalAlignment {
    static let timeline = HorizontalAlignment(TimelineRail.self)
}

HStack(alignment: .top, spacing: 12) {
    Circle()
        .fill(.tint)
        .frame(width: 10, height: 10)
        .alignmentGuide(.timeline) { d in d[HorizontalAlignment.center] }
    VStack(alignment: .leading, spacing: 4) {
        Text("Event").font(.headline)
        Text("Detail text can wrap across multiple lines.")
    }
    .alignmentGuide(.timeline) { d in d[HorizontalAlignment.leading] - 18 }
}
Cross-branch alignment

alignmentGuide adjusts a child’s contribution to a stack line. Aligning arbitrary nodes in different subtrees often moves to PreferenceKey, overlays, or explicit Layout — say that aloud before diving into hacks.

8. frame vs fixedSize

Both reshape proposals; they answer different questions. frame expresses desired dimensions or flexibility bands; fixedSize pushes back on compression so ideal sizes win.

8.1 Fixed frame

Color.orange
    .frame(width: 100, height: 44)

8.2 Flexible frame (min / ideal / max)

Text("Flexible")
    .frame(minWidth: 120, idealWidth: 200, maxWidth: .infinity)

8.3 fixedSize — full and partial

Label("New", systemImage: "sparkles")
    .padding(.horizontal, 8)
    .fixedSize()

Text("Do not truncate this title vertically")
    .fixedSize(horizontal: false, vertical: true)
Modifier Effect on negotiation Typical use
frame(width:height:) Targets explicit dimensions; still interacts with parent clamps. Thumbnails, fixed buttons, skeleton placeholders.
frame(min:ideal:max:) Declares a flexibility band on an axis. Responsive rows, split views, “take remaining width.”
fixedSize() Prioritizes ideal size; overflow bubbles to ancestors. Chips, pills, protecting titles from compression.

9. ViewThatFits

ViewThatFits evaluates its child views in source order and selects the first whose measured size fits the proposal on the axes you specify (Axis.Set). It is declarative adaptive branching without imperative width checks.

9.1 Horizontal adaptation

ViewThatFits(in: .horizontal) {
    HStack {
        Text("Short")
        Text("Detail")
        Text("Extra")
    }
    VStack(alignment: .leading) {
        Text("Short")
        Text("Detail")
        Text("Extra")
    }
}

9.2 Vertical-only checks

ViewThatFits(in: .vertical) {
    Text("Compact summary").font(.title3)
    Text("Compact").font(.footnote)
}

Use cases: toolbar overflow, sidebar vs stack navigation, multi-column grids collapsing to one column, “rich vs compact” cell variants.

Ordering matters

List the most spacious variant first. The first fitting child wins; siblings are not composed together. Combine with Layout or breakpoints when partial visibility is required.

9.3 Mental model diagram

Incoming proposal to ViewThatFits │ ▼ Try child #1 (measure) │ ├─ fits? ──► select #1 (done) │ └─ else try child #2 …

10. LayoutValueKey — Passing Values from Child to Layout

LayoutValueKey lets children publish typed metadata that only their containing Layout reads — no global preferences, no mirror view hierarchies. Conform with a defaultValue; children set values via layoutValue(key:value:).

struct FlexibilityKey: LayoutValueKey {
    static var defaultValue: Double { 1 }
}

extension View {
    func layoutFlexibility(_ value: Double) -> some View {
        self.layoutValue(key: FlexibilityKey.self, value: value)
    }
}

struct WeightedRowLayout: Layout {
    func sizeThatFits(
        proposal: ProposedViewSize,
        subviews: Subviews,
        cache: inout ()
    ) -> CGSize {
        let widths = subviews.map { $0.sizeThatFits(proposal).width }
        let totalWidth = widths.reduce(0, +)
        let height = subviews.map { $0.sizeThatFits(proposal).height }.max() ?? 0
        return CGSize(width: totalWidth, height: height)
    }

    func placeSubviews(
        in bounds: CGRect,
        proposal: ProposedViewSize,
        subviews: Subviews,
        cache: inout ()
    ) {
        let weights = subviews.map { $0[FlexibilityKey.self] }
        let sum = weights.reduce(0, +)
        var x = bounds.minX
        for (idx, sub) in subviews.enumerated() {
            let fraction = sum > 0 ? weights[idx] / sum : 1 / CGFloat(subviews.count)
            let width = bounds.width * CGFloat(fraction)
            let h = sub.sizeThatFits(ProposedViewSize(width: width, height: proposal.height)).height
            sub.place(
                at: CGPoint(x: x, y: bounds.midY - h / 2),
                proposal: ProposedViewSize(width: width, height: h)
            )
            x += width
        }
    }
}
WeightedRowLayout {
    Color.red.layoutFlexibility(2)
    Color.green.layoutFlexibility(1)
    Color.blue.layoutFlexibility(1)
}
.frame(height: 44)

11. layoutPriority & Content Flexing

layoutPriority biases flexible space allocation in stacks: higher priority views resist shrinking and claim extra room when the parent has to distribute ambiguity. It does not override explicit frames or fixedSize in all cases — it adjusts relative claims among flexible siblings.

HStack {
    Text("Secondary")
        .foregroundStyle(.secondary)
        .layoutPriority(0)
    Text("Primary title should win space first")
        .lineLimit(1)
        .layoutPriority(1)
}

Pair this mentally with fixedSize: priority decides who gets flex; fixedSize changes whether compression is even on the table.

12. Architect Callout: Complex Adaptive Layouts Without GeometryReader

Whiteboard-grade system design

Breakpoints via measurement, not magic numbers: Prefer ViewThatFits to choose between a two-column dashboard and a stacked layout. The proposal encodes available space; you avoid brittle UIScreen.main-style checks in body.

Dashboard grids: Implement a custom Layout that reads tile spans from LayoutValueKey (column/row span, importance). Cache row heights and occupied cells — same conceptual job as compositional collection layouts, expressed as proposal-driven code.

Compose policies: Outer ViewThatFits selects column vs stack; inner FlowLayout wraps tags; leaves use fixedSize to protect chip intrinsic widths. Reserve GeometryReader for canvases (charts, rubber-band gestures) with explicit bounded frames.

Testing narrative: Layout driven by proposals snapshots cleanly across size classes; scattered geometry math in views does not.

┌──────────────────────────────────────────────────────────────────┐ │ Adaptive dashboard composition │ │ │ │ ViewThatFits (horizontal) │ │ │ │ │ ├── Wide: GridLayout + LayoutValueKey (spans) │ │ │ │ │ └── Narrow: FlowLayout (chips) + VStack (sections) │ │ │ │ GeometryReader only inside bounded chart / gesture overlays │ └──────────────────────────────────────────────────────────────────┘
Staff interview closing move

End with: “We pushed policy into Layout + ViewThatFits, used LayoutValueKey for per-tile metadata, and isolated GeometryReader to drawing layers.” That sentence signals architectural maturity.
