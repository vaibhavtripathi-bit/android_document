1. Darwin & XNU Kernel

Q1: Describe XNU’s hybrid kernel architecture. Why does Apple combine Mach with the BSD layer instead of a monolithic Unix kernel? L6

XNU (X is Not Unix) is a hybrid: the Mach microkernel-style layer provides low-level primitives—tasks (containers for virtual memory), threads, scheduling, IPC via ports, and VM—while the BSD personality layer implements POSIX semantics: processes, file descriptors, signals, networking stack integration, and the syscall table your app actually invokes.

Mach was chosen historically for portability, extensibility, and fine-grained IPC; BSD provides familiar Unix APIs and a mature networking and security model. The split lets the kernel enforce isolation and resource accounting at the Mach boundary while still exposing a coherent Unix surface to user space.

User space (app, frameworks) │ ▼ syscalls / libsystem ┌─────────────────────┐ │ BSD layer │ ← POSIX, VFS, sockets, policies ├─────────────────────┤ │ Mach │ ← tasks, ports, VM, IPC └─────────────────────┘
Staff angle

Interviewers want you to connect this to sandboxing and jetsam: policy often lives in user-space daemons and the kernel together; understanding Mach tasks vs BSD “processes” explains why debugging memory and IPC looks different on Apple platforms than on Linux.

Q2: What are Mach ports, and how does IPC differ from POSIX pipes or sockets? Architect

A Mach port is a kernel-protected right to send messages to an endpoint. IPC is message-based: you allocate port rights, send structured messages (often with out-of-line descriptors and memory ranges), and the kernel validates capabilities. Unlike byte-stream pipes, Mach messages are typed and capability-centric: holding a send right is analogous to holding a handle, not an open FD to a generic stream.

On Apple systems, higher-level IPC (XPC) builds on these primitives. POSIX sockets remain for networking and some local APIs, but intra-machine privileged communication often goes through Mach/XPC so the kernel and launchd can enforce entitlements and sandbox rules.

Practical note

You rarely call Mach APIs from app code; instead you reason about deadlocks and priority inversions when main thread waits on XPC replies, or about port exhaustion in daemons under load.

Q3: What is launchd’s role in the iOS/macOS boot and service model? Senior

launchd is PID 1 and the central service supervisor. It reads plist manifests (LaunchAgents, LaunchDaemons, XPC service bundles), manages socket activation, restarts on crash according to policy, and coordinates dependencies. On iOS, the surface area differs from macOS—fewer user-visible agents—but the same pattern holds: system daemons and app extensions are parented and scheduled by infrastructure launchd understands.

For apps, the important leap is that background work is often mediated by separate processes (e.g., extension processes) launched on demand; launchd-style lifecycle explains why “the app” is not one monolithic address space for features like widgets, Siri intents, or notification service extensions.

Q4: How does Jetsam interact with memory pressure and the virtual memory system on iOS? L6

Jetsam is the mechanism that terminates processes under memory pressure to protect the system and foreground experience. It uses heuristics: footprint, priority band (foreground vs background), recent CPU usage, and internal “memorystatus” thresholds. Unlike a swap-heavy desktop, iOS has limited reclaim options, so killing is common.

Your process shares the device with WebKit, SpringBoard, and daemons; large resident sets, memory spikes during image decode, or unbounded caches trigger jetsam events that look like crashes with EXC_RESOURCE or jetsam-specific reports in crash logs.

// Mitigation patterns (conceptual)
func didReceiveMemoryWarning() {
    imageCache.removeAllObjects()
    purgeDerivedTransientState()
}

// Use Instruments Allocations + VM Tracker to validate peaks
Staff angle

Relate Jetsam to extension memory limits (notification service extensions are tighter than the host app) and to photography / ML pipelines where peak RSS dominates steady-state.

Q5: Explain the entitlements system: how are they enforced, and why can’t an app simply declare arbitrary entitlements? L6

Entitlements are signed assertions embedded in the app’s code signature (and provision profile for development). The kernel, securityd, and various daemons consult them when granting access to protected resources: keychain access groups, push environments, iCloud containers, HealthKit, associated domains, etc.

Apple’s signing infrastructure only allows certain entitlement keys for a given App ID and capability toggles in the developer portal. At runtime, mismatches fail closed: for example, accessing a keychain group not in your entitlements returns errSecMissingEntitlement.

This model shifts security from “hope the app behaves” to cryptographically attestable capability lists, which is why enterprise builds and TestFlight behave differently when profiles drift.

Q6: What is a sandbox profile, and where does enforcement happen relative to the app? Architect

The sandbox is a Seatbelt / Apple profile applied per process, constraining filesystem paths, Mach lookup, network, and other operations via policy rules compiled from .sb profiles. Enforcement spans the kernel and trusted user-space: syscalls and IPC are vetted against the profile.

Third-party apps run under tight templates; system apps and daemons use broader profiles. Extensions often have more restrictive subsets than their host. Understanding sandbox denials (Console, sandboxd logs) separates “code bug” from “capability design” issues.

Interview tip

Tie this to App Groups and container URLs: the sandbox allows specific shared locations only when entitlements and profiles align.

Q7: Contrast the iOS process model with macOS for third-party software. What platform constraints drive the difference? Senior

iOS emphasizes single foreground app UX, aggressive power management, and a curated distribution model. Third-party code generally cannot spawn arbitrary helper processes; multitasking is mediated through extension points, background tasks, and system schedulers. macOS allows richer multi-process desktop patterns, user-installed daemons, and less uniform suspension—though hardened runtime and notarization narrow the gap.

Implications: on iOS, “move work to a child process” is rarely available; you choose BGTaskScheduler, URLSession background sessions, push-triggered wakes, or extension architectures instead.

Q8: What roles do SpringBoard and backboardd play in the system UI stack? L6

SpringBoard is the home screen / app switcher shell process: app launching, icon management, scene coordination with UIKit, and high-level gesture routing for the primary user interface. backboardd handles lower-level event delivery—touch digitizer data, display timing, keyboard events—bridging hardware to higher-level frameworks.

When debugging input latency or rotation issues, the split matters: hit testing and view hierarchy live in the app; raw touch synthesis and timing constraints involve system daemons. Security-sensitive paths (e.g., certain system gestures) are enforced outside app space.

Touch hardware │ ▼ backboardd ──► SpringBoard / app processes │ ▼ UIKit / SwiftUI scenes

2. Objective-C Runtime

Q9: Walk through what happens when objc_msgSend dispatches an instance method. L6

objc_msgSend is a hand-written assembly fast path. It loads the receiver’s isa, follows cached method lookup structures, and jumps to the implementation pointer. If the method is not in the class’s method list, it escalates to the forward path (class_lookupMethodAndLoadCache3 historically), which can invoke resolveInstanceMethod:, forwarding, and ultimately doesNotRecognizeSelector:.

The critical staff-level point is the interaction between inline caching, class hierarchy mutation (categories loaded late), and thread safety: the runtime holds locks during certain mutations; swizzling + concurrency can expose subtle ordering bugs.

Q10: Explain the isa pointer, classes, metaclasses, and where class methods live. Architect

Each object points via isa to its class. The class object is itself an object whose isa points to the metaclass. Instance methods are stored on the class; class methods are instance methods of the metaclass. The metaclass’s superclass chain parallels the class hierarchy, ultimately joining NSObject’s metaclass, which is why +initialize inherits oddly and why class method lookup mirrors instance lookup “one meta level up.”

instance ──isa──► class ──isa──► metaclass methods class methods superclass superclass (meta chain)

Modern runtimes use non-pointer isa packing extra bits; debugging raw memory requires masking.

Q11: When is method swizzling safe, and what are the failure modes? L6

Swizzling exchanges IMP pointers at runtime, typically in +load (runs before main, in linker order) or +initialize (lazy per class—riskier due to recursion). Safer patterns: swizzle in +load, dispatch once, prefer composition or method forwarding for new code.

Failure modes: double-swizzle, swizzling Apple frameworks across OS versions, breaking KVO’s expectations, deadlocks if swizzled code re-enters runtime locks, and App Store scrutiny for private API–like behavior.

extension UIViewController {
    static let swizzle: Void = {
        let original = #selector(UIViewController.viewDidLoad)
        let swizzled = #selector(UIViewController.swizzled_viewDidLoad)
        // objc_exchangeImplementations / method_setImplementation
    }()
}
Modern guidance

Prefer method replacement with original pointer capture and explicit naming; in Swift-heavy codebases, swizzling is a last resort—use composition, decorators, or framework hooks.

Q12: How does memory management work for associated objects (objc_setAssociatedObject)? Senior

Associated objects attach arbitrary key/value pairs to an object via a global hash table. Policies (OBJC_ASSOCIATION_RETAIN_NONATOMIC, etc.) mirror ARC semantics: retained associations bump retain counts; when the host object deallocates, associated values are released according to policy.

They are not a substitute for subclassing: abuse can create retain cycles (associated object retains host indirectly), obscure object graphs, and complicate debugging. Weak associations exist but have nuanced performance characteristics.

import ObjectiveC

private enum Keys {
    static var payload: UInt8 = 0
}

extension URLSessionTask {
    var requestID: String? {
        get {
            objc_getAssociatedObject(self, &Keys.payload) as? String
        }
        set {
            objc_setAssociatedObject(
                self, &Keys.payload, newValue,
                .OBJC_ASSOCIATION_RETAIN_NONATOMIC
            )
        }
    }
}
Q13: How is KVO implemented at runtime (isa-swizzling)? L6

When you register for KVO on a property, the runtime creates a dynamic subclass of your object’s class, overrides setters to call willChange/didChange, and swaps the instance’s isa to that subclass—hence “isa-swizzling.” object_getClass vs class diverge: the latter may still report the original class for API stability.

This implies interactions with manual KVO, Key Paths, and Swift properties: pure Swift structs and non-@objc types aren’t KVO-compatible without dynamic dispatch hooks.

Q14: Describe the KVC search path for value(forKey:) on an object. Senior

Key-Value Coding searches, in order: a getter <key>, is<Key> for booleans, get<Key>, _get<Key>; then instance variable access if allowed (+accessInstanceVariablesDirectly); then countOf<Key> and indexed accessor patterns for to-many relationships; finally valueForUndefinedKey: which raises by default.

For setters, the symmetric set<Key> / _set<Key> / ivar paths apply. Understanding this explains mysterious crashes when a typo silently hits undefined keys or when Swift private names don’t expose Objective-C selectors.

Q15: What is the method resolution chain before doesNotRecognizeSelector:? L6

After fast path cache miss, the runtime searches the class’s method list, then follows the superclass chain. If unresolved, it calls resolveInstanceMethod: / resolveClassMethod: (dynamic method addition). Then forwardingTargetForSelector: can substitute another object. If still unresolved, methodSignatureForSelector: and forwardInvocation: enable slow forwarding. Only then doesNotRecognizeSelector:.

Staff insight: message forwarding is rare in app code but powers proxies and some framework magic; each hop has a performance cost.

Q16: Compare NSProxy with NSObject for forwarding and root behaviors. Architect

NSProxy is an abstract root that does not implement NSObject’s default methods; it is designed for lightweight forwarding stubs. You implement forwardInvocation: and related methods to delegate messages. NSObject provides default implementations for retain/release (under ARC, synthesized), description, equality, and KVO integration assumptions.

Use NSProxy when building transparent stand-ins (classic lazy loading, distributed objects patterns); use composition with NSObject subclasses for most app-level indirection today.

Q17: Contrast @dynamic and @synthesize in Objective-C with modern defaults. Senior

By default, the compiler synthesizes ivars and accessors unless you provide manual implementations. @synthesize explicitly names backing ivars (less common now). @dynamic promises accessors exist at runtime—typical for Core Data NSManagedObject properties resolved via the model.

Mismatching @dynamic without runtime resolution yields unrecognized selector crashes at first access; this is a common interview probe for Core Data fluency.

Q18: What happens when two categories on the same class define a method with the same selector? L6

Category methods are flatly injected into the class’s method list; load order wins—the last category loaded overrides earlier implementations silently. There is no compile-time error for collisions across categories.

Mitigation: prefix category methods (my_prefix_), use extensions in Swift with module scoping, or consolidate into a single category. This is why method swizzling + third-party SDK categories can produce heisenbugs across dependency versions.

Linker detail

Objective-C +load order follows image link order; static libraries vs dynamic frameworks change collision outcomes between builds.

3. Swift Runtime & ABI

Q19: Describe the layout of an existential container for a protocol with associated type requirements vs one without. Architect

Small existentials (like any Equatable in modern Swift) pack value buffers inline (typically 3 word inline buffer + metadata pointers) when the dynamic type fits; otherwise values are heap-boxed. Protocols with associated types or Self constraints aren’t straightforward existentials—Swift models many as any P only when existential support exists; otherwise you use generics or type erasers.

Staff depth: witness tables (protocol witness tables, or PWTs) live alongside type metadata; the existential’s value witness table (VWT) handles copy/destroy/move for the erased value.

Q20: What is a protocol witness table (PWT), and how does it relate to dynamic dispatch in generics? L6

A PWT is a vtable-like structure mapping protocol requirements to concrete function pointers for a specific conforming type at a specific conformance. For func f<T: P>(_ x: T), the compiler passes (or reifies) metadata + PWT so calls to P requirements resolve to T’s implementations—this is reified generics.

Specialization can devirtualize these calls at compile time when T is known, removing indirection.

Q21: What is generic specialization, and what are the binary size trade-offs? Senior

The compiler emits dedicated machine code for concrete types used at call sites (e.g., Array<Int> vs Array<String>), removing runtime metadata lookups and enabling inlining. This improves performance but duplicates code, increasing binary size.

Large apps mitigate with @_specialize (internal), careful access control to limit visibility, and profiling hot generics. Library authors expose stable ABIs via resilience attributes to avoid over-specializing public APIs.

Q22: What does a value witness table (VWT) describe? Architect

VWTs encapsulate type-specific operations: size, stride, alignment, flags, and function pointers for destroy, copy with init, assign with copy, etc. Existentials and generics use VWT indirection to manipulate values of unknown layout safely.

This is why moves of large structs, COW containers, and existential boxing have different costs—the VWT path may heap-allocate or retain references.

Q23: Where does @objc bridging carry performance cost in hot paths? Senior

Crossing the Swift/ObjC boundary may involve selector dispatch, boxing of value types into NSNumber/NSString, and runtime bridging of collections. In tight loops calling Foundation APIs, these conversions dominate.

Mitigations: stay in Swift types, use CF APIs with careful lifetime rules, batch work, or wrap hot paths in objc_msgSend-friendly caches. Instruments’ Swift/ObjC churn counters help validate.

// Costly: repeated NSString bridging in a loop
let arr: [NSString] = swiftStrings.map { $0 as NSString }

// Often better: keep NSArray toll-free bridging once
Q24: What does Swift ABI stability mean for the OS vs third-party binaries? L6

ABI stability means the Swift runtime and standard library in the OS expose a frozen calling convention and type layout so binaries compiled with different compiler versions interoperate. Apple ships the runtime in iOS/macOS; apps link against it dynamically.

Practically, you can ship closed-source Swift frameworks on Apple platforms without forcing consumers to match your exact compiler—subject to library evolution constraints on public types.

Q25: Explain @frozen, @inlinable, @usableFromInline, and resilient enums/structs for library evolution. Architect

Resilient libraries must preserve client binaries when adding fields or cases. Non-frozen enums require unknown default handling in clients; resilient structs hide size/layout behind indirection for public fields added later. @frozen opts into fixed layout—binary compatibility risk if you change shape.

@inlinable exposes implementation across module boundaries for performance; @usableFromInline elevates internal symbols to be referenced from inlinable code. The tension is performance vs evolution flexibility.

Q26: How does dynamic dispatch work in Swift compared to Objective-C? Senior

Swift prefers static dispatch via generics, final, and whole-module optimization. Dynamic dispatch uses vtables for class methods and witness tables for protocol requirements. dynamic and @objc expose ObjC runtime dispatch where needed.

Staff angle: devirtualization failures show up in sil/LLVM; designing hot paths with protocols without exposing them as type-erased existentials preserves static dispatch opportunities.

Q27: Distinguish module stability from ABI stability. L6

ABI stability is about runtime calling conventions and standard library layout across compiler versions. Module stability (.swiftmodule format) allows compile-time interoperation: clients can import a binary Swift module’s interface without source.

Shipping a binary framework requires both concerns addressed; missing module stability forces clients to compile against textual interfaces or source.

Q28: Compare performance characteristics of existential types vs generics for algorithmic code. Architect

Generics specialized to concrete types avoid per-call witness indirection and enable inlining and scalar optimizations. Existentials pay for runtime type metadata, witness dispatch, and sometimes heap allocation for large or opaque values.

Rule of thumb: use some P (opaque result types) or generics in hot paths; use any P for heterogeneous collections where erasure is inherent. Measure with benchmarks—LLVM can optimize some existential patterns, but not all.

// Generic: often static dispatch
func sum<T: Numeric>(_ xs: [T]) -> T { xs.reduce(0, +) }

// Existential: dynamic witness calls per operation
func sumAny(_ xs: [any Numeric]) -> any Numeric { /* more overhead */ }

4. Notifications & Push

Q29: Describe the APNs architecture from app server to device. L6

Your provider server opens an HTTP/2 connection to Apple’s APNs endpoint (development vs production), authenticates with a JWT key or legacy certificate, and POSTs a JSON payload targeting a device token. Apple’s gateways route to the device over persistent connections maintained by iOS; the OS wakes or schedules the app to deliver the notification.

App server ──HTTP/2──► APNs ──► iOS (apsd) JWT / cert auth device token

Staff depth: APNs is best-effort; it is not a guaranteed message queue. Idempotency and server-side reconciliation matter.

Q30: What is the device token lifecycle, and when can it change? Senior

The token maps your app + device + environment to an APNs routing key. It can change after restores, OS reinstalls, app reinstalls, or Apple-side rotations. Always pass fresh tokens to your server on didRegisterForRemoteNotifications and stop assuming permanence.

Pair tokens with user accounts carefully: multi-device users need multiple tokens; logout should unregister server-side mappings.

Q31: What can a Notification Service Extension do, and what are its resource limits? L6

The NSE runs before presentation for remote notifications marked mutable-content. It can download media, decrypt payloads, and mutate the content (title, body, attachments). It runs under strict memory and time budgets—jetsam kills extensions that exceed limits; network must complete quickly.

Design idempotent downloads, cache intelligently, and never assume the extension runs for every push (delivery still best-effort).

Q32: Why are silent pushes unreliable for critical sync? Architect

Silent pushes (content-available: 1) are subject to power budgets, user settings (Background App Refresh), Low Power Mode, and throttling by the OS. Apple explicitly discourages treating them as guaranteed transport.

Reliable pattern

Combine push as a hint with periodic BGAppRefreshTask, foreground refresh, and server-driven reconciliation when the app opens. For VoIP-class immediacy, historical APIs were restricted; use appropriate PushKit flows only where entitled.

Q33: Compare JWT token-based APNs auth with legacy SSL certificates. Senior

JWT keys (.p8) are easier to rotate, work across multiple apps under a team key, and don’t expire yearly like certificates. Certificates tied push auth to a specific cert lifecycle and were operationally heavier at scale.

Modern providers almost universally use JWT; protect private keys with HSM/KMS patterns on the server.

Q34: How should you reason about push delivery reliability and failure feedback? L6

APNs returns immediate HTTP status for malformed requests or bad device tokens; it does not confirm the user saw the notification. Track 410 responses for invalid tokens and prune your datastore. For visible notifications, analytics are app-side.

Staff systems design: treat push as at-most-once delivery; use server timestamps and sync APIs for eventual consistency.

Q35: What are notification categories and actions, and how do they interact with extensions? Senior

Categories define actionable buttons (e.g., Reply, Snooze) bound to notification types. Actions can be foreground or background; background actions may launch the app briefly or run extension code depending on configuration.

Register categories early (UNUserNotificationCenter) and keep identifiers stable across versions to avoid orphaned actions on delivered notifications.

let reply = UNTextInputNotificationAction(
    identifier: "REPLY",
    title: "Reply",
    options: []
)
let category = UNNotificationCategory(
    identifier: "MESSAGE",
    actions: [reply],
    intentIdentifiers: [],
    options: []
)
UNUserNotificationCenter.current().setNotificationCategories([category])
Q36: What is the background push budget, and how does it affect product design? Architect

iOS throttles background wakeups driven by silent notifications to preserve battery. Excessive pushes for one app reduce delivery frequency—symptoms look like “sometimes sync works.” The exact algorithm is opaque and changes between OS versions.

Product/engineering response: batch server-side, prioritize user-visible notifications, use explicit user-initiated sync, and surface UX that doesn’t promise instant background consistency unless using privileged real-time channels.

5. Inter-Process Communication

Q37: Compare XPC with raw Mach ports for app and extension communication. L6

XPC is a higher-level, type-safe IPC layer with service lifecycle integration, automatic invalidation, and security attributes (sandbox, codesign requirements). Mach ports are lower-level capabilities; XPC builds on them.

App developers usually interact with XPC indirectly (NSXPCConnection, system frameworks). Direct Mach is reserved for low-level or internal use.

Q38: How do App Groups enable data sharing, and what must match across targets? Senior

App Groups declare a shared container UUID in entitlements for each participating target (app, extensions, widgets). FileManager.default.containerURL(forSecurityApplicationGroupIdentifier:) returns the shared sandbox subtree for coordinated file storage, Core Data SQLite, or caches.

All members need the exact same group identifier string in signing entitlements and provisioning profiles. Mismatches manifest as nil container URLs or sandbox denials.

Q39: Contrast custom URL schemes with Universal Links for app interoperability. L6

URL schemes are string namespaces resolved by the OS; any app can claim popular schemes, creating hijack risk. Universal Links verify domain ownership via apple-app-site-association files and route https URLs directly into your app when installed, falling back to the website.

Staff guidance: prefer Universal Links for production deep linking; reserve schemes for development or tightly controlled ecosystems.

Q40: What are Darwin notifications (CFNotificationCenterGetDarwinNotifyCenter), and when are they appropriate? Architect

Darwin notifications broadcast a simple name across processes with no payload—they are coalescing signals, not a data channel. Common pattern: extension writes to App Group container, posts CFNotificationCenterPostNotification, host observes and reads files.

They are lightweight but easy to misuse; they don’t guarantee ordering or delivery semantics beyond best effort. For richer messaging, XPC or file coordination patterns are clearer.

Q41: How do you secure an NSXPCConnection? L6

Set interface and export classes explicitly, validate auditToken / codesign requirements where available, use Objective-C protocols with NSXPCInterface, and avoid exposing arbitrary selectors. Minimize surface: each method should be purpose-built.

Combine with sandbox entitlements so only expected XPC services are reachable; treat IPC as a trust boundary even inside your own bundle (XPC services run separate processes).

Q42: Explain Keychain sharing via access groups and why it fails in CI sometimes. Senior

Keychain items carry access group attributes matching entitlements. Sharing requires the same team prefix and explicit group strings in app + extensions. Simulator vs device, provisioning profile drift, or missing keychain-access-groups cause errSecMissingEntitlement or -34018 errors.

Staff ops: align CI signing identities with local dev, use explicit access groups in queries, and avoid assuming Keychain behaves identically on macOS Catalyst vs iOS.

Q43: How would you choose an IPC mechanism for a widget, app clip, and main app architecture? Architect

Widgets and app clips are short-lived or constrained: prefer App Group files + atomic writes, UserDefaults(suiteName:) for small flags, and Darwin notifications or timeline reload APIs as hints. Avoid long-lived bidirectional IPC—there’s no always-on channel like a daemon.

Main app performs heavy lifting on next launch; use WidgetCenter.shared.reloadTimelines after data changes. For sensitive data, Keychain with shared access groups; for large structured state, versioned files with checksums.

Q44: What is NSFileCoordinator for, and when is it required? Senior

File coordinators serialize access to files presented to other processes (iCloud Drive, document providers, extensions) to avoid partial reads during writes. They integrate with NSFilePresenter for change notifications.

Use coordinators when reading/writing security-scoped URLs, packages, or shared containers concurrently with other processes. Pure in-app private container access often doesn’t need the overhead—but extensions sharing App Group storage can race without coordination discipline.

6. Accessibility

Q45: How does VoiceOver navigate the accessibility tree differently from visual layout? L6

VoiceOver walks the accessibility hierarchy: elements, containers, and custom actions—not necessarily visual z-order. Focus order follows accessibility frame, accessibilityElements arrays, and container protocols. Off-screen or visually hidden elements can be exposed or suppressed via traits and hidden flags.

Staff testing: validate rotor navigation, headings, links, and form controls independently of pixel-perfect UI.

Q46: How does Dynamic Type propagate through UIKit and SwiftUI? Senior

Dynamic Type maps user text size preferences to semantic styles (UIFont.TextStyle, SwiftUI .font(.body)). UIKit adjusts fonts via metrics and triggers traitCollectionDidChange; SwiftUI’s environment drives re-layout automatically when content size category changes.

Custom fonts must use UIFontMetrics scaling; fixed point sizes break accessibility. Test extremes (AX5) for clipping and multi-line truncation.

let font = UIFont.preferredFont(forTextStyle: .body)
let scaled = UIFontMetrics.forTextStyle(.body).scaledFont(for: customFont)
Q47: When do you implement UIAccessibilityContainer protocols vs exposing subviews? L6

Custom drawings or composite views may expose a simplified logical structure via UIAccessibilityElement children and hit testing overrides. Use container APIs when subviews don’t map 1:1 to meaningful accessible elements (charts, game boards, PDF overlays).

Ensure frames convert correctly to screen coordinates and that children update when data changes.

Q48: How do custom accessibility actions improve power-user workflows? Senior

accessibilityCustomActions expose context menu–like operations to assistive tech without visible buttons—critical when screen space is tight or gestures are undiscoverable. Each action has a name and handler.

Pair with hints and labels so users discover available operations via rotor or actions menu.

row.accessibilityCustomActions = [
    UIAccessibilityCustomAction(name: "Archive", target: self, selector: #selector(archive(_:)))
]
Q49: Outline an accessibility testing strategy beyond manual VoiceOver sweeps. Architect

Automate smoke checks with XCTest’s accessibility APIs, snapshot UI with large content sizes, run contrast analyzers on critical screens, and include accessibility acceptance criteria in design reviews (focus order, headings, live regions).

Team process

Tag bugs with severity for keyboard/VoiceOver blockers, run periodic audits with disabled engineers or consultants, and track regression in CI via screenshot diff + AX label presence tests for golden flows.

Q50: Compare SwiftUI and UIKit accessibility ergonomics at scale. L6

SwiftUI defaults propagate many traits automatically but obscure customizations behind modifiers; debugging can be harder than explicit accessibilityLabel on UIKit views. UIKit offers finer-grained control and long-standing patterns but requires manual maintenance.

Hybrid apps should standardize wrappers bridging UIKit components into SwiftUI with consistent AX modifiers. Establish lint rules for missing labels on interactive elements.

7. Internationalization

Q51: Describe the locale fallback chain when loading localized resources. Senior

Bundle localization searches preferred languages from user settings, region variants (e.g., pt-BR), script variants, and then language-only codes before defaulting to development language. Missing keys fall back to Base.lproj or the source language.

Understanding this prevents “works in simulator English, fails in German” when only partial .lproj folders exist.

Q52: Why are plural rules not expressible as simple string concatenation? L6

Languages partition plurals differently (zero, one, two, few, many, other in CLDR). ICU plural rules select grammatical forms by numeric magnitude and locale. Use String(localized:) with String.LocalizationValue pluralization or stringsdict / String Catalog variants.

// String Catalog: define plural variants for key "files" with variable "count"
let s = String(localized: "files.count")

// Legacy: Localizable.stringsdict maps (one/other) via ICU plural rules
Q53: How does RTL support work architecturally in UIKit and Auto Layout? Architect

UIKit uses leading/trailing constraints mapped to left/right per effectiveUserInterfaceLayoutDirection. Images may need imageFlippedForRightToLeftLayoutDirection; navigation controllers swap bar button semantic positions.

SwiftUI’s layout mirrors horizontally when environment layout direction is RTL. Test Arabic and Hebrew with pseudolanguages to catch hard-coded .left anchors.

Q54: Compare String Catalogs with legacy Localizable.strings workflows. Senior

String Catalogs (.xcstrings) centralize keys, variants (plural, device), state (new/stale/translated), and export for translators in Xcode. Legacy .strings files scatter across targets and complicate plural dictionaries via separate .stringsdict.

Migration strategy: adopt catalogs per module, wire CI to detect missing translations pre-release, and integrate with TMS via XLIFF export.

Q55: When do you prefer FormatStyle over DateFormatter? L6

FormatStyle (and ParseStrategy) are value-typed, thread-safe, and compose with locale-aware Swift idioms. DateFormatter is heavier, mutable, and easy to misconfigure (timezone bugs, calendar mismatches).

let style = Date.now.formatted(
    .dateTime.day().month(.wide).year()
        .locale(Locale(identifier: "ja_JP"))
)
Q56: How do you operationalize i18n at scale (40+ languages)? Architect

Establish pseudo-localization in debug builds, enforce no hard-coded user strings via lint, modularize translations per feature team, and contract SLA with LSPs using XLIFF round-tripping. Track coverage metrics (percent keys translated per locale) and block release on P0 markets.

Engineering leads should budget for layout expansion (German, Finnish), line breaking (CJK), and legal locale variants separately from language (e.g., en-GB vs en-US).
