L6 UIAccessibility Protocol

UIAccessibility is an informal protocol declared as a category on NSObject. Practically every UIView and UIResponder participates in the accessibility system because the category supplies defaults and because the runtime bridges responder objects into an accessibility element representation consumed by assistive technologies (VoiceOver, Switch Control, Voice Control, etc.).

At interview depth, distinguish three layers: (1) the view hierarchy you build in UIKit, (2) the accessibility tree the system derives or you override, and (3) the assistive technology UX (announcements, gestures, focus order) that emerges from that tree.

Core Properties

  • isAccessibilityElement — When true, the object is exposed as its own focusable node. When false, the view may still contribute layout or host children, but it is typically skipped as a direct VoiceOver target (common for purely decorative chrome or container backgrounds).
  • accessibilityLabel — Primary spoken name. Should be concise and describe purpose or content, not control type (VoiceOver already announces traits).
  • accessibilityHint — Spoken after a short delay when focus lingers; use for non-obvious outcomes (“Opens share sheet”)—not for duplicating the label.
  • accessibilityValue — Current value for controls: slider position, toggle state text, text field contents when appropriate, download percent, etc.
  • accessibilityTraits — Bitmask of UIAccessibilityTraits that shape announcements, rotor behavior, and default actions (button vs header vs adjustable).
  • accessibilityFrame — The rect in screen coordinates used for hit testing during direct touch exploration. Often you rely on the view’s conversion to window/screen; custom drawing may require manual updates in layoutSubviews or when scroll offsets change.
Deep Dive: How VoiceOver Reads an Element

VoiceOver composes an utterance from label, value, traits, and hint according to heuristics that depend on context and language. Traits influence both wording (“button”, “heading”) and available gestures (e.g., adjustable invokes swipe-up/down to change value). Custom actions appear as rotor or action menu affordances. The key Staff-level point: you are designing a spoken UI; redundant or overly long strings increase cognitive load and slow navigation for power users.

accessibilityIdentifier

accessibilityIdentifier is a string intended for UI testing (XCUIElement queries) and automation. It is not reliably exposed to VoiceOver users for comprehension—do not treat it as a second label. Keep identifiers stable across runs, unique where your test strategy requires disambiguation, and avoid embedding PII.

Pitfall: Labels That Encode Type or State Poorly

Avoid “Button: Submit” (trait already says button) and avoid stuffing dynamic data into hints that should live in accessibilityValue. For images that convey information, the label must carry the informational equivalent; empty labels on informative images are a common audit failure.

Best Practice: Label = What, Hint = Consequence, Value = Now

Think in roles: label identifies the object semantically (“Airplane mode”), value reflects current state (“On”), hint explains rare or destructive actions (“Turns off all wireless connections”). When in doubt, listen with VoiceOver at speed—Staff engineers optimize for clarity at high speech rates.

// UIKit: explicit semantic control
final class PlaybackToggle: UIButton {
    func applyAccessibility(isPlaying: Bool) {
        isAccessibilityElement = true
        accessibilityLabel = "Playback"
        accessibilityValue = isPlaying ? "Playing" : "Paused"
        accessibilityHint = "Double-tap to pause or resume the current track"
        accessibilityTraits = [.button]
    }
}

Senior+ Accessibility Traits

UIAccessibilityTraits is an OptionSet describing semantics and behavior. Traits affect default actions, rotor entries, and how adjustable controls respond to increment/decrement gestures.

Commonly Discussed Traits

  • .button — Announced as a button; double-tap activates.
  • .link — Announced as a link; commonly used for tappable text that navigates.
  • .header — Structural heading; enables heading navigation via rotor.
  • .image — Identifies graphical content; pair with meaningful labels for informative images.
  • .selected — Selection state for tabs, segmented controls, list selection.
  • .adjustable — VoiceOver uses vertical swipes to adjust; implement accessibilityIncrement/accessibilityDecrement (and value coherence).
  • .staticText — Read as text; often default for labels.
  • .searchField — Search field semantics.
  • .notEnabled — Communicates disabled state alongside visual styling.
  • .updatesFrequently — Hints that contents change often (timers, streaming counters) so assistive tech can throttle announcements.
  • .startsMediaSession — Indicates activation may start audio; helps assistive tech manage audio ducking/interruptions.
  • .causesPageTurn — Used for book/page interfaces to align gestures with page-turn affordances.
  • .tabBar — Identifies tab bar elements for navigation context.

Combining Traits

Union traits with array literal syntax: [.button, .selected]. Conflicting semantics (e.g., treating static text as a button without clear activation affordance) confuse users and fail audits. When you need a tappable heading, combine .header with .button only if the entire region behaves as one activation target.

Trait Typical VoiceOver impact Implementation note
.adjustable Announces “adjustable”; swipe up/down changes value Must update accessibilityValue; consider UIAccessibility.post(notification: .layoutChanged, argument:) after large jumps
.header “Heading”; rotor navigation between headings Use for real outline structure, not visual bolding alone
.updatesFrequently Reduces intrusive re-announce spam Still update value; don’t misuse to hide stale state
.startsMediaSession Audio session expectations for AT Pair with correct AVAudioSession category in media features
.notEnabled “Dimmed” / disabled phrasing Keep gesture parity—disabled controls should not activate 
var traits: UIAccessibilityTraits = [.button]
traits.insert(.selected)
traits.remove(.notEnabled)
customView.accessibilityTraits = traits

L6 Custom Actions & Rotor Items

UIAccessibilityCustomAction models actions beyond the default “activate.” Each action has a name shown in the rotor’s Actions menu (or related AT surfaces) and a handler invoked when chosen. Return true when your handler consumes the action.

UIAccessibilityCustomRotor extends the VoiceOver rotor with domain-specific navigation: move between comments, search results, chart data points, or timeline events. You supply a predicate callback that searches forward/backward from the current element and returns a UIAccessibilityCustomRotorItemResult targeting the next semantic location.

Deep Dive: When Custom Actions Beat Hidden Gestures

Discoverability is the differentiator. Swipe menus and long-press affordances are invisible to many AT users unless you mirror them with explicit actions (“Delete”, “Pin”, “Mark unread”). Staff-level solutions centralize actions in view models so the same operations are reachable via UI, shortcuts, and accessibility.

let delete = UIAccessibilityCustomAction(name: "Delete") { [weak self] _ in
    self?.deleteItem()
    return true
}
row.accessibilityCustomActions = [delete]

let rotor = UIAccessibilityCustomRotor(name: "Comments") { predicate in
    // Search comments forward/backward; return UIAccessibilityCustomRotorItemResult
    return nil
}
view.accessibilityCustomRotors = [rotor]
Pitfall: Action Handlers Capturing Views Strongly

Use [weak self] (or equivalent) in custom action closures to avoid retain cycles with views. Favor idempotent handlers that post UIAccessibility.Notification after mutations (.screenChanged, .layoutChanged) so focus lands sensibly.

Senior+ accessibilityElements & Ordering

Default accessibility ordering roughly follows view hierarchy and screen position, but complex layouts—custom collections, overlapping layers, decorative overlays—often produce nonsense traversal. Override accessibilityElements to publish an explicit array of elements in the intended focus order.

  • accessibilityElements — Ordered array of objects conforming to accessibility (often child views or UIAccessibilityElement proxies).
  • shouldGroupAccessibilityChildren — When true, children may be aggregated; useful for composite controls where inner labels should not be separately focusable.
  • accessibilityElementsHidden — Hides subtree from assistive tech (use sparingly; ensure no critical info exists only there).

Best Practice: Group What Speaks Together

If a single logical row is composed of four labels, expose one element with a composed label or use AXCustomContent for structured extras—avoid four separate swipes for one datum unless each is independently actionable.

L6 VoiceOver Navigation Patterns

VoiceOver combines multiple input modalities: linear swipe navigation, direct touch exploration, rotor settings, containers, and container escape gestures. Understanding this model is essential when debugging “focus jumps” and modal traps.

Linear navigation flow (simplified) User swipes Right ──► AT asks UIKit for "next" element in traversal order │ │ │ ├─ Respect accessibilityElements order │ ├─ Skip hidden / non-accessible views │ └─ Enter/exit containers per rules ▼ Focus lands on element ──► Announce label + value + traits (+ hint after delay) │ ├─ Double-tap ──► default action (activate button, toggle, etc.) ├─ Two-finger Z ──► pass-through gesture to app when configured └─ Rotor ──► change navigation mode (headings, links, form controls)

Containers & Modality

  • accessibilityViewIsModal — Treats presenting view as modal: sibling elements behind are ignored, focusing AT on the modal surface—critical for overlays and sheets.
  • accessibilityPerformEscape — Implement to dismiss transient UI (popovers, partial sheets) when users invoke the escape gesture equivalent.
  • accessibilityPerformMagicTap — Two-finger double-tap; common pattern for play/pause in media apps when appropriate.
extension MySheetViewController {
    override func accessibilityPerformEscape() -> Bool {
        dismiss(animated: true)
        return true
    }
}
Pitfall: Modal Flags Without Focus Management

Setting accessibilityViewIsModal without moving VoiceOver focus into the modal causes users to land “behind” the sheet. Pair modal presentation with UIAccessibility.post(notification: .screenChanged, argument: firstFocusView) (or layout changed) to announce context and relocate focus.

Senior+ Dynamic Type Implementation

Dynamic Type maps user text size preferences to concrete font metrics. Use UIFont.preferredFont(forTextStyle:) instead of hard-coded sizes so typography tracks the content size category.

  • UIFontMetrics — Scales custom fonts and layout constants relative to a text style baseline.
  • adjustsFontForContentSizeCategory — On UILabel and compatible controls, enables automatic font updates when categories change.
  • traitCollectionDidChange — Reload layout when preferredContentSizeCategory changes; combine with notifications if needed globally.
let title = UIFont.preferredFont(forTextStyle: .title1)
label.font = title
label.adjustsFontForContentSizeCategory = true

let metrics = UIFontMetrics(forTextStyle: .body)
let scaled = metrics.scaledFont(for: customFont, maximumPointSize: 34)

Images and glyphs should scale or reflow: use symbol configurations, asset catalogs with vector PDFs, or explicit size adjustments tied to metrics. Test at AX5 categories and verify truncation does not remove essential actions—prefer wrapping stacks and readable compression over single-line ellipsis for primary CTAs.

Deep Dive: Minimum / Maximum Scaling

Apple provides APIs to cap scaling for brand-critical headlines or dense dashboards, but caps are a product decision with accessibility tradeoffs. Document why limits exist and ensure critical controls remain tappable and readable; pair capped typography with layout that still accommodates large content size for body text.

    forName: UIContentSizeCategory.didChangeNotification,
    object: nil,

L6 UIAccessibilityContainer & Virtual Elements

When visual content is not a tree of subviews—Core Animation layers, drawRect charts, Metal-backed canvases—you still owe users a navigable semantics layer. Implement container callbacks or use UIAccessibilityElement instances attached to a container via accessibilityContainer.

  • accessibilityElementCount — Number of logical children.
  • accessibilityElement(at:) — Return child at index.
  • index(ofAccessibilityElement:) — Inverse mapping for focus algorithms.
final class ChartView: UIView {
    private var pointElements: [UIAccessibilityElement] = []

    func reloadAccessibility() {
        isAccessibilityElement = false
        pointElements = bars.enumerated().map { idx, bar in
            let el = UIAccessibilityElement(accessibilityContainer: self)
            el.accessibilityLabel = "Series \(idx)"
            el.accessibilityValue = "\(bar.value)"
            el.accessibilityFrameInContainerSpace = bar.rect
            el.accessibilityTraits = [.adjustable, .summaryElement]
            return el
        }
        accessibilityElements = pointElements
    }
}
Best Practice: Virtual Elements Mirror Data, Not Pixels

Each proxy should map to a meaningful data point users can understand without seeing color alone. Pair with custom rotors when datasets are large.

Senior+ Accessibility Inspector

Xcode’s Accessibility Inspector connects to Simulator or devices to audit runtime trees. Capabilities include element inspection (frame, label, traits), Audit for common issues (insufficient contrast, missing labels, small touch targets), and quick toggles for content size categories and reduced motion.

  • Use Navigation mode to simulate focus moves and verify ordering.
  • Use Contrast Calculator for text/background pairs—especially gradients and image overlays.
  • Iterate with different text sizes and Differentiate Without Color checks where applicable.
Pitfall: Simulator-Only Validation

Always spot-check on device with real VoiceOver; timing, audio ducking, and haptics differ. Automated audits catch shallow issues; they miss conversational UX quality.

L6 AXCustomContent

Introduced broadly at the UIKit level in iOS 14, AXCustomContent lets you attach structured supplemental descriptions with importance levels. This is ideal for cells where the primary label should stay short, but additional metrics (APR, mileage, status badges) still need programmatic access.

  • Importance — .default vs .high influences whether VoiceOver reads content automatically or only on demand (via the “Read Details” custom content action that the system can surface).
  • AXCustomContentProvider — Conformance for richer authoring flows; UIKit views adopt relevant hooks as the platform evolves—verify availability per OS target.
import Accessibility

let apr = AXCustomContent(
    label: "APR",
    value: "3.9 percent"
)!
apr.importance = .high

cell.accessibilityCustomContent = [apr]
Deep Dive: Structured Data vs Wall of Speech

AXCustomContent is how you avoid ten-second row announcements while preserving inspectability. Staff interviews reward knowing when to split primary vs secondary facts and how that maps to rotor/read-details behavior.

Senior+ SwiftUI Accessibility Modifiers

SwiftUI builds an accessibility representation parallel to the view graph. Modifiers attach semantics to nodes or reshape the child tree exposed to the platform.

  • .accessibilityLabel / .accessibilityHint / .accessibilityValue — UIKit analogs.
  • .accessibilityAddTraits / .accessibilityRemoveTraits — Trait tuning without fully replacing defaults.
  • .accessibilityAction — Named actions mirroring custom actions.
  • .accessibilityRotor — Declarative custom rotors.
  • .accessibilityElement(children:) — Combine or ignore children (.combine / .ignore) for grouping behavior.
  • .accessibilityRepresentation — Represent a complex view using a simpler accessibility subtree.
  • .accessibilityChildren — Explicitly provide child accessibility elements when hiding the default synthesized children.
Text(title)
    .accessibilityAddTraits(.isHeader)

Button("Save") { save() }
    .accessibilityHint("Writes your changes to the server")

HStack {
    Text(name)
    Text(subtitle)
}
.accessibilityElement(children: .combine)

Chart(data: series)
    .accessibilityRepresentation {
        VoiceOverChartSummary(series: series)
    }
Concern UIKit SwiftUI
Grouping children shouldGroupAccessibilityChildren, manual accessibilityElements .accessibilityElement(children: .combine)
Custom actions accessibilityCustomActions .accessibilityAction(named:)
Rotors UIAccessibilityCustomRotor .accessibilityRotor
Hide subtree accessibilityElementsHidden .accessibilityHidden(true)

Pitfall: SwiftUI Graph vs Platform-Synthesized Tree

SwiftUI may merge or split elements across platform versions. Validate with Inspector after OS upgrades; prefer explicit combine/ignore when designs are dense.

Architect Building Accessibility into Architecture

Staff engineers treat accessibility as a non-functional requirement with the same weight as performance and security—not as a release tail QA theme. That implies API design for feature modules that carry semantic metadata, design systems where components ship with default labels/traits/hints, and analytics that do not depend on inaccessible gestures alone.

Architect Must Know

Shift-left: define acceptance criteria per feature for VoiceOver traversal, Dynamic Type levels, and contrast. Wire CI with XCTest that queries accessibilityIdentifier stable contracts, plus periodic manual audits. Map organizational risk: ADA/WCAG-aligned expectations for customer contracts; in EU contexts, reference EN 301 549 alignment discussions. Performance cost of accessibility metadata is negligible; the cost of retrofitting tree semantics across a mature codebase is not.

Testing & Automation Strategy

  • Unit-level — View models expose strings for labels/values to keep localization and a11y testable.
  • UI tests — Use identifiers for selection; assert existence of actionable elements in critical flows.
  • Manual rotor passes — Quarterly audits with real users and experts where possible.

Business & Legal Framing

Accessible apps expand addressable market, reduce support burden, and improve SEO-like discoverability inside assistive tech catalogs. Legal exposure varies by jurisdiction and industry; enterprise buyers increasingly require VPAT-style evidence. Technical leadership should articulate that accessible design is compatible with velocity when component libraries and previews bake semantics by default.

UIAccessibility Notifications (Focus & Context)

Posting the right UIAccessibility.Notification after structural UI changes is the difference between a disorienting experience and a predictable one. Use them when the semantic screen changes, not on every layout pass.

Notification Typical use Argument
.screenChanged New screen or major context (push, full-screen modal) Element or container to focus first
.layoutChanged Significant restructure within the same screen (insert/delete rows, swap primary panel) Element that best represents the change
.pageScrolled Pagination and large scroll position changes where context should be re-announced Often nil or relevant view
.announcement Short status messages (“Saved”, “Failed to send”) NSString (keep brief) 
UIAccessibility.post(notification: .screenChanged, argument: headerLabel)
UIAccessibility.post(notification: .announcement, argument: "Profile updated")
Deep Dive: Coalescing Announcements

Rapid-fire .announcement posts queue awkwardly. Debounce non-critical toasts and prefer contextual focus moves (.layoutChanged) when the user’s next action is tied to a specific element. For async flows, align announcements with transaction completion, not intermediate spinner states.

SwiftUI: AccessibilityFocusState & programmatic focus

SwiftUI exposes focus binding for VoiceOver via @AccessibilityFocusState. Use it when opening sheets or toggling edit modes so assistive tech lands on the primary field or confirmation control—mirroring what you already do visually with default focus.

struct EditorView: View {
    @AccessibilityFocusState private var focusTitle: Bool
    @State private var isEditing = false

    var body: some View {
        TextField("Title", text: $title)
            .accessibilityFocused($focusTitle)
            .onChange(of: isEditing) { editing in
                if editing { focusTitle = true }
            }
    }
}

Assistive technologies beyond VoiceOver

Staff conversations should mention Voice Control (spoken commands rely on accessibility names), Switch Control (sequential scanning benefits from logical ordering and large hit regions), and Reduce Motion / Reduce Transparency (respect UIAccessibility.isReduceMotionEnabled and material choices). Traits like .keyboardKey or form fields affect how these modalities classify interactors—test critical flows under multiple settings.

Pitfall: Touch Target vs accessibilityFrame drift

Custom transforms, scroll views, and animated shifts can desynchronize the visible control from the reported frame. After animation, call UIAccessibility.post(notification: .layoutChanged, …) or override accessibilityFrame to return a converted rect so direct touch exploration remains accurate.
