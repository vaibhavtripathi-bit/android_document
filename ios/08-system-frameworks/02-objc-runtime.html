1. objc_msgSend: Full Flow

Every [receiver message] compiles to a call to objc_msgSend (or a variant). Understanding this path is the single highest-leverage Objective-C runtime topic in senior interviews: it connects compiler lowering, CPU calling conventions, cache behavior, and the class hierarchy walk.

Deep Dive — Not a “function call” in the C sense

objc_msgSend is implemented in hand-tuned assembly because the ABI varies by architecture and return-type class (integer/pointer vs. floating-point vs. struct). The runtime must preserve registers exactly as the callee expects while still performing lookup work. That is why you will not find a portable C implementation of the fast path in shipping Apple runtimes.

arm64 calling convention (conceptual)

On arm64, the first integer/pointer arguments go in x0, x1, x2, … For objc_msgSend, x0 is self, x1 is _cmd (the selector), and user arguments start at x2. The implementation must branch to the resolved IMP without disturbing argument registers the callee will read—hence careful tail-calling and specialized entry points.

┌─────────────────────────────────────────────────────────────────┐ │ objc_msgSend(self, _cmd, arg2, …) — arm64 fast path (concept) │ └─────────────────────────────────────────────────────────────────┘ x0 = self x1 = _cmd (SEL) │ ▼ Load class from self->isa (may need masking for non-pointer isa) │ ▼ class->cache lookup (bucket chain / open addressing — implementation detail) │ hit? ├──yes──► IMP in register ──tail branch (b / br)──► callee │ no │ ▼ __objc_msgSend_uncached / slow path: search method list on class, then superclass chain, attach to cache, then tail-call IMP

Method cache (why it feels “instant”)

Each Class owns a method cache: a hash table mapping (SEL → IMP) (plus associated metadata the runtime uses for correctness and invalidation). On a cache hit, dispatch is a small constant amount of work: a few loads, a compare, and an indirect branch. Interviewers often probe why this is fast:

  • Locality — cache data sits on the class object; repeated messages to the same receiver/class touch hot cache lines.
  • No virtual C++-style vtable indirection for the common case — once warm, you avoid scanning method lists.
  • Tail-call / branch-to-IMP — after resolution, control transfers directly to the implementation without extra frames on the fast path.
Best Practice — How to talk about “hash table in class”

Say: per-class cache, keyed by selector, storing the resolved IMP; probing uses a fixed strategy; on miss the runtime searches the class’s method list then superclasses, may need dynamic resolution, then fills the cache. Avoid claiming a specific open-addressing prime size unless you have read the exact runtime version—behavior is stable in meaning, not always in layout.

Cache miss → method lookup

When the selector is not cached, the runtime walks method lists attached to the class, then repeats up the superclass chain. Method lists can be lazily realized (categories, loaded images, runtime edits), so the first miss can be more expensive than steady-state. If no implementation is found, you enter the dynamic resolution chain (resolveInstanceMethod:, forwarding, etc.—covered later).

Tail-call optimization and the “no extra frame” story

Conceptually, objc_msgSend wants to end as: “jump to IMP with the same register state as if the caller had invoked IMP directly.” That is why the fast path is implemented as a branch (tail call) to the target function once IMP is known. Stack depth stays flat for the dispatch stub itself—important for microbenchmark narratives and for understanding backtraces (you may see objc_msgSend frames only when still inside the stub or uncached paths).

objc_msgSend variants

Symbol (family) When used Staff-level note
objc_msgSend Most returns that fit in integer/pointer registers per ABI Default lowering for id returns and many ObjC methods
objc_msgSend_fpret Legacy x86; floating-point return ABI quirks Rarely relevant on modern arm64 interviews; mention for cross-platform literacy
objc_msgSend_stret Large struct returns (ABI passes hidden sret pointer) On arm64 many structs are handled differently—know it as “ABI special case,” verify platform
objc_msgSendSuper / objc_msgSendSuper2 super calls: start lookup at superclass super2 uses receiver to get class but still begins search at super—compiler/runtime pairing detail 
// ObjC — [super doWork] lowers to objc_msgSendSuper2(&objc_super, _cmd)
// objc_super holds { receiver, super_class } so lookup starts at the superclass.
@implementation Child
- (void)doWork {
    [super doWork];
}
@end
// Swift — explicit super to ObjC requires @objc inheritance context

class Child: Parent {
    override func doWork() {
        super.doWork()
    }
}

2. Method Dispatch: “Vtable” vs Message Dispatch

Objective-C methods are dynamically bound by name (SEL) at send time (modulo compiler devirtualization optimizations). This differs materially from typical C++ virtual dispatch through a fixed vtable slot chosen at compile time for a given override layout.

Architect Must-Know

ObjC can use vtable-style fast paths for certain objc_msgSend optimizations in recent runtimes (e.g., some alloc/init fastpaths)—treat those as transparent optimizations. The semantic model interviewers expect is still message passing with swizzling, categories, and dynamic resolution.

Mechanism Binding time Mutation Interop story
ObjC message send At send (selector → IMP resolution) Categories, swizzling, +load order, runtime APIs Central to Apple frameworks; dynamic
C++ virtual (vtable) Compile-time slot + runtime vptr UB to mutate vtables; layout is ABI Deterministic; fragile across libraries
Swift protocol witness tables Static + runtime for existentials/generics Resilience via metadata; not “swizzle friendly” Optimization-friendly; different debugging story

Method lists and resolution order

Each class stores lists of struct method_t entries (name, types, IMP). Categories append methods; if a category defines a method already on the class, the category’s implementation typically wins (last loaded / runtime ordering rules)—this is a classic interview pitfall when debugging “wrong method ran.”

Pitfall — Category collisions

Two categories implementing the same selector on the same class: which IMP wins is undefined unless you control +load order and linkage. In production, avoid duplicate selectors across categories; use prefixes or composition.

“Does ObjC use a vtable?”

Precise answer for Staff: not in the C++ sense for arbitrary methods. The runtime uses class-relative method lists + cache. Some internal paths may resemble vtables for speed, but you should not rely on that as an API. Compare with Swift’s devirtualization and witness tables—Swift prefers static dispatch when proven safe.

3. isa Pointer and Class Hierarchy

An Objective-C object is a C struct whose first field is isa (historically a direct class pointer; on modern runtimes often a non-pointer isa bitfield that must be masked to reach the Class).

Non-pointer isa (arm64)

The low bits of isa may encode:

  • Pointer authentication or tagging schemes (platform/version dependent)
  • Extra refcount / side-table flags in some configurations—interviewers may ask conceptually “why not always store a raw Class*?”

Always use object_getClass / runtime APIs instead of manually poking isa in production code.

Class layout (conceptual)

  • superclass pointer — linearization of inheritance
  • cache — selector fast path
  • method list — instance methods for instances; class methods live on metaclass
  • protocol list — adopted protocols
  • ivar layout — offsets, for ivar access and KVC/KVO
Instance: myObject ┌──────────────────┐ │ isa ─────────────┼──► ┌─────────────────────┐ │ ivar storage… │ │ class object │ └──────────────────┘ │ name: MyWidget │ │ superclass ────────┼──► … ──► NSObject │ instance methods │ │ ivar layout │ │ isa ───────────────┼──► metaclass(MyWidget) └─────────────────────┘ Metaclass: metaclass(MyWidget) ┌─────────────────────┐ │ isa ────────────────┼──► root metaclass (NSObject’s metaclass chain) │ superclass ─────────┼──► metaclass(superclass of MyWidget) │ class methods here │ └─────────────────────┘ Rule of thumb: “Instance messages” → class object’s method list “Class messages” → metaclass’s method list “[NSObject class]” → oddity: NSObject class methods terminate via root metaclass ↔ NSObject pairing
Deep Dive — Why metaclasses exist

Classes are objects too (+[SomeClass load] is a message to the class object). The metaclass’s isa chain ensures class objects respond to +alloc, +new, etc., while instance methods live on the class object for instances. The NSObject root pairing is a common whiteboard checkpoint.

// Runtime introspection (ObjC)
#import <objc/runtime.h>

Class cls = objc_getClass("NSString");
NSLog("%p", cls);
Class meta = object_getClass((id)cls);
NSLog("metaclass %p", meta);
// Swift — type and object introspection bridge to ObjC runtime for @objc types
import Foundation

let obj: AnyObject = NSObject()
let c: AnyClass? = object_getClass(obj)
print(String(describing: c))

4. Method Swizzling

method_exchangeImplementations swaps two IMP pointers in the runtime’s method list. After exchange, selectors point to the opposite implementations—so each IMP often calls the “other” selector to retain original behavior (classic pattern).

Pitfall — Ordering and +load

Swizzling in +load runs early, but order across classes/categories is not a stable long-term contract for app logic. Multiple frameworks swizzling the same method create Heisenbugs. Prefer instrumentation hooks provided by the framework (e.g., URLSession delegate wrappers) when available.

class_addMethod + exchange pattern

If the target class does not yet expose a method entry (e.g., you add a stub), use class_addMethod to install an IMP, then exchange. This avoids crashing when class_getInstanceMethod would return NULL for missing selectors.

#import <objc/runtime.h>

static void orig_doThing(id self, SEL _cmd) {
    NSLog("before");
    // call “original” by messaging the swapped name:
    ((void(*)(id,SEL))objc_msgSend)(self, sel_registerName("orig_doThing:"));
}

void SwizzleDoThing(Class cls) {
    SEL orig = @selector(doThing);
    SEL alias = sel_registerName("orig_doThing:");
    Method m = class_getInstanceMethod(cls, orig);
    const char *types = method_getTypeEncoding(m);
    class_addMethod(cls, alias, method_getImplementation(m), types);
    Method m1 = class_getInstanceMethod(cls, orig);
    Method m2 = class_getInstanceMethod(cls, alias);
    method_exchangeImplementations(m1, m2);
}
Best Practice — Thread safety

Perform swizzles once under a lock, before concurrent use. Swapping IMPs while messages are in flight on other threads can theoretically race with first-time cache fills—treat swizzling as process-wide mutation done at startup.

Swizzling in Swift

Swift methods are not naturally dynamic unless marked @objc dynamic (or exposed in ways that force dispatch through the ObjC runtime). Swizzling Swift-defined methods without dynamic is fragile or ineffective because dispatch may be direct. For Staff interviews: prefer composition, protocols, and wrappers; treat runtime swizzling in Swift as a red-flag maintenance cost.

5. Associated Objects

objc_setAssociatedObject / objc_getAssociatedObject attach arbitrary key/value pairs to an object via a global side table (conceptually AssociationsManager: a hash map from object address to property map). This enables “stored properties” on categories—something the language otherwise forbids.

Policy Meaning Memory semantics
OBJC_ASSOCIATION_ASSIGN Raw pointer; no retention Unsafe with ObjC objects—dangling unless weak-safe patterns used
OBJC_ASSOCIATION_RETAIN_NONATOMIC Strong, nonatomic Common default for object storage
OBJC_ASSOCIATION_RETAIN Strong, atomic Synchronizes setter-like behavior; not a substitute for your own model locks
OBJC_ASSOCIATION_COPY_NONATOMIC Copy, nonatomic Matches “copy” property semantics for immutable snapshots
OBJC_ASSOCIATION_COPY Copy, atomic Rare; know it exists for completeness
Pitfall — Lifetimes and zombies

Associated objects are released when the host object is deallocated (after dealloc / Swift deinit for ObjC objects). Using ASSIGN for ObjC object pointers captures unsafe-unretained references—treat like a weak reference without automatic nil-ing.

import ObjectiveC

private var kToken: UInt8 = 0

extension UIView {
    var auditTag: String? {
        get {
            objc_getAssociatedObject(self, &kToken) as? String
        }
        set {
            objc_setAssociatedObject(
                self, &kToken, newValue,
                .retainNonatomic
            )
        }
    }
}

6. NSProxy

NSProxy is a root class (not a subclass of NSObject) designed for pure forwarding. It implements minimal instance behavior and raises on most selectors unless you override forwarding methods.

Topic NSObject NSProxy
Typical initialization Rich initializer graph init is not the usual NSObject pattern—often custom
Forwarding path Works, but NSObject carries many default methods Minimal surface → encourages forwarder design
Use cases General base Lazy stand-ins, RPC stubs, test doubles, indirection
Deep Dive — Why NSProxy still appears

Certain designs want an object that is obviously not an NSObject for method existence tests and forwarding. In modern Swift, many patterns migrate to protocols + structs, but Apple frameworks and legacy code still reference NSProxy concepts in interviews.

7. NSInvocation

NSInvocation packages a target, selector, arguments, and return value into an object. It is the backbone of forwardInvocation: once you have decided (via methodSignatureForSelector:) that a selector can be forwarded, you can mutate an NSInvocation and dispatch it to another object.

Pitfall — Performance

NSInvocation is flexible but expensive: boxing arguments, retaining targets, and indirect calls defeat static optimization. Use for compatibility layers, not inner loops.

@implementation ProxyThing
- (void)forwardInvocation:(NSInvocation *)inv {
    [inv invokeWithTarget:self.realObject];
}
- (NSMethodSignature *)methodSignatureForSelector:(SEL)sel {
    return [self.realObject methodSignatureForSelector:sel];
}
@end

8. KVO Internals (isa-swizzling)

When you register an observer with addObserver:forKeyPath:…, the runtime may create a dynamic subclass of your object’s class, e.g. NSKVONotifying_MyWidget, and point the instance’s isa to that subclass (isa-swizzling). The subclass overrides accessors to emit willChangeValueForKey: / didChangeValueForKey: around the original implementation.

Before KVO: object.isa ──► MyWidget (real class) After observing "foo": object.isa ──► NSKVONotifying_MyWidget │ ├─ override -setFoo: ──► calls super + KVO hooks │ └─ class method +class may report MyWidget (hiding swizzle) Detection: object_getClass(obj) vs [obj class] often diverge under KVO — use runtime introspection in debugging
Best Practice — KVO compliance

Ensure accessor symmetry: paired getters/setters or explicit KVC support; document dependent keys; avoid modifying collections without KVC mutation methods if you want array/order observers to behave.

Manual KVO

Wrap changes with willChange/didChange for computed or non-ivar-backed properties when automatic swizzling cannot see writes.

KVO in Swift

Requires @objc exposure and dynamic for Swift-defined properties so the runtime can swizzle accessors. For value types or non-ObjC dispatch, use other observation patterns (Observable, Combine, etc.).

class Model: NSObject {
    @objc dynamic var title: String = ""
}

9. KVC (Key-Value Coding)

KVC routes string keys to accessors or ivars through a documented search order—critical for Core Data, bindings-era patterns, and debugging mysterious valueForKey: crashes.

valueForKey: search path (instance)

  1. Accessors in order: getKey, key, isKey, _key (if enabled)
  2. Instance variable access if +accessInstanceVariablesDirectly returns YES (default YES for NSObject subclasses—surprises many)
  3. valueForUndefinedKey: (raises by default)

setValue:forKey:

  1. Setters: setKey:, _setKey:
  2. Ivar direct assignment under the same accessInstanceVariablesDirectly gate
  3. setValue:forUndefinedKey:
Architect Must-Know — Security & encapsulation

KVC can bypass private Swift access if an ObjC header exposes selectors and the object is id-messaged through KVC. Treat KVC as capability, not “reflection is safe.”

Collection operators

Keys like @avg.amount apply array/set operators (@count, @sum, @avg, @min, @max, @unionOfObjects, etc.) across collections—handy in bindings-era code, rare in modern SwiftUI but still fair game for legacy interviews.

KVC and Swift

Swift property names must be visible to ObjC (@objc) for KVC paths unless using NSObject KVC overrides. Swift-only structs are outside this mechanism.

10. @dynamic vs @synthesize

  • @synthesize — instructs the compiler to synthesize ivar + accessor implementations (historically explicit; modern ObjC defaults often synthesize properties automatically).
  • @dynamic — promises accessors exist at runtime without compile-time synthesis—used by Core Data NSManagedObject (properties resolved via dynamic method resolution or generated subclasses).
Deep Dive — Version drift

Default property synthesis behavior changed across Clang/ObjC eras. In interviews, emphasize: @dynamic shifts responsibility to the runtime or framework; mismatches manifest as unrecognized selectors until Core Data machinery loads.

11. Method Resolution Chain

When messaging finds no IMP, the runtime walks a structured recovery pipeline—each step trades flexibility for cost.

objc_msgSend ──► cache / method list ──► IMP? │ miss / nil ▼ +resolveInstanceMethod: / +resolveClassMethod: │ dynamic method install? ▼ -forwardingTargetForSelector: (fast forward) │ returns another object? ▼ -methodSignatureForSelector: + -forwardInvocation: │ full NSInvocation boxing ▼ -doesNotRecognizeSelector: (abort by default)
Stage Typical use Cost
resolveInstanceMethod: Add methods with class_addMethod (Core Data, message proxies) Moderate; runs before forwarding
forwardingTargetForSelector: Lightweight delegate to another object Low; no NSInvocation yet
methodSignatureForSelector: + forwardInvocation: Full proxying, RPC marshaling High; builds invocation object
doesNotRecognizeSelector: Last-chance logging / crash customization Terminal unless handled
Best Practice — Choose the cheapest hop

If you only need to redirect to a helper object, implement forwardingTargetForSelector: and avoid forwardInvocation: unless you must inspect arguments.

@implementation Router
- (id)forwardingTargetForSelector:(SEL)aSelector {
    if ([self.backend respondsToSelector:aSelector]) {
        return self.backend;
    }
    return [super forwardingTargetForSelector:aSelector];
}
@end

12. When ObjC Runtime Knowledge Is Critical for Swift Developers

Architect Perspective — Swift is not a clean room

Even “pure Swift” apps link Foundation and UIKit, which are ObjC objects under the hood. Staff engineers use runtime knowledge to:

  • Bridge — @objc, dynamic, selectors in UIKit/SwiftUI glue, and Clang module boundaries.
  • Debug — distinguish KVO isa-swizzling, swizzled methods, and unrecognized selector stacks.
  • Crash analysis — read objc_msgSend traps, corrupted isa, zombie messaging patterns.
  • Interop — expose Swift APIs to notifications, KVC, performSelector, and third-party ObjC SDKs.
  • Performance — reason about dynamic dispatch vs. devirtualization, and why “just make it NSObject” changes ABI costs.
Deep Dive — Interview narrative

Close with tradeoffs: dynamism enables Apple’s evolution of frameworks but complicates static reasoning. Strong Staff answers propose measurement (Instruments, objc runtime logging flags in debug), containment (no duplicate swizzles), and alternatives (composition, protocol witnesses in Swift) rather than clever runtime hacks.

Summary comparison — Messaging vs forwarding

Path Latency Flexibility
Cached objc_msgSend Lowest stable cost Standard method dispatch
Uncached first lookup Higher one-time Same semantics, fills cache
forwardingTargetForSelector: Extra message overhead Redirect without boxing
forwardInvocation: Highest generic cost Full introspection of arguments

You now have the skeleton to defend message passing end-to-end: from arm64 registers through caches and method lists, into dynamic resolution, KVO/KVC machinery, and the Swift boundary. In interviews, anchor claims to observable behavior and stable runtime contracts, and label version-specific layout as implementation detail unless a question explicitly targets it.
