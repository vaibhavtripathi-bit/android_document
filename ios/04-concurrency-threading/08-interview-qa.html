GCD Internals (Questions 1–10)

1. What happens internally when you call dispatch_async? Walk through the full path from your code to thread execution. L6

When you call dispatch_async(queue, block):

  1. Block copy: The block is copied from stack to heap via _Block_copy. All captured variables are retained (for objects) or copied (for value types).
  2. Continuation wrapping: The block is wrapped in a dispatch_continuation_t struct that holds the block, the queue, the QoS, and voucher information.
  3. Enqueue: The continuation is appended to the queue's internal FIFO list using a lock-free MPSC (Multiple Producer, Single Consumer) atomic linked list. This uses os_atomic_xchg for wait-free enqueue.
  4. Drain signal: libdispatch signals the kernel's pthread_workqueue facility that work is available, specifying the QoS class.
  5. Thread wakeup: The kernel either wakes an idle worker thread from the pool or creates a new one (up to the per-workqueue limit of 64).
  6. Dequeue & execute: The worker thread calls _dispatch_queue_drain, which dequeues the continuation and invokes the block.
  7. Cleanup: After execution, the block is released via _Block_release, releasing captured objects.

For serial queues, the drain function holds a "drain lock" ensuring only one block executes at a time. For concurrent queues, multiple threads can drain simultaneously.

2. Explain the 64-thread limit in GCD. What is thread explosion, and how do you prevent it? Senior

GCD manages a per-process thread pool through the kernel's pthread_workqueue. Each workqueue (mapped to a QoS class) can have up to 64 concurrent threads. Thread explosion occurs when you submit many blocks that each block their thread (e.g., synchronous I/O), causing GCD to create new threads up to and beyond this limit (serial queues use "overcommit" workqueues that can exceed 64).

Consequences: Each thread costs ~512KB stack space. 200 threads = 100MB just for stacks. Plus massive context switching overhead, cache thrashing, and potential Jetsam kills.

Prevention strategies:

  • Use OperationQueue with maxConcurrentOperationCount
  • Use DispatchSemaphore to limit concurrent work
  • Use async/await (cooperative thread pool limited to CPU core count)
  • Use TaskGroup with a sliding window pattern for bounded concurrency
  • Avoid blocking threads — use async APIs instead of synchronous ones
3. What is priority inversion in the context of GCD? How does GCD resolve it? L6

Priority inversion occurs when a high-priority task is blocked waiting for a low-priority task that can't run because medium-priority tasks are consuming CPU time. The high-priority task effectively runs at the low-priority task's priority — inverted.

GCD resolves this through priority boosting (priority inheritance):

  • dispatch_sync: When a high-QoS thread calls sync on a queue, GCD temporarily boosts the priority of the currently executing block on that queue to match the waiter's QoS.
  • Serial queues: The queue's effective QoS is raised to the highest QoS of any pending block.
  • DispatchSemaphore: Limited priority inheritance — the system can boost the signaling thread, but it's not as reliable as os_unfair_lock which has full kernel-level priority inheritance.

Important caveat: DispatchSemaphore does NOT fully support priority inheritance because the kernel doesn't know which thread holds the semaphore. For critical sections, prefer os_unfair_lock or serial dispatch queues.

4. Why does using a barrier flag on a global concurrent queue have no effect? Senior

Global concurrent queues are shared by the entire process — all frameworks, libraries, and your code submit work to the same global queues. If barriers worked on global queues, any code could block the entire system by submitting a barrier block. This would be a denial-of-service vulnerability within your own process.

The barrier flag is silently ignored on global queues. Barriers only work on custom concurrent queues that you create, because you control all the work submitted to them. This is why the reader-writer pattern requires a custom concurrent queue.

5. Explain the target queue hierarchy. How can you use it for subsystem-level synchronization? L6

Every dispatch queue has a target queue — the queue where its blocks are ultimately executed. By default, custom queues target the default-priority global concurrent queue. You can set a custom target to create hierarchies.

Key use case: If multiple serial queues share the same serial target queue, they become mutually exclusive. Blocks from any of these queues will never execute concurrently with each other, because they all funnel through the shared serial target.

This is powerful for subsystem-level synchronization: each module gets its own queue (for debugging labels and independent scheduling), but they all serialize through a shared target for thread safety. It's more granular than a single global lock but still prevents data races across the subsystem.

QoS interaction: The target queue's QoS acts as a floor. If a child queue has a higher QoS than its target, the effective QoS is the higher of the two (to prevent priority inversion within the hierarchy).

6. What are the five QoS classes in GCD? When would you use each one? Senior
  • .userInteractive (priority 33): Main thread work, animations, event handling. Uses performance cores, highest CPU priority.
  • .userInitiated (priority 25): User tapped something and is waiting for a result. High priority, performance cores.
  • .default (priority 21): Default when no QoS is specified. Balanced scheduling.
  • .utility (priority 17): Long-running tasks the user is aware of but not actively waiting for — downloads, imports, indexing. Prefers efficiency cores, I/O throttled, timer coalescing enabled.
  • .background (priority 9): Work the user doesn't know about — prefetching, maintenance, backups. Efficiency cores only, heavily throttled, may be deferred entirely during low battery.

On Apple Silicon, .userInteractive and .userInitiated preferentially schedule on performance (P) cores, while .utility and .background schedule on efficiency (E) cores for energy savings.

7. What is the difference between DispatchGroup's enter/leave pattern and the group-async pattern? Senior

The group-async pattern (queue.async(group: group) { ... }) automatically calls enter() before the block starts and leave() when the block completes. It works for synchronous blocks that complete when they return.

The enter/leave pattern is for asynchronous work where the block returns before the work is done (e.g., network callbacks). You manually call group.enter() before starting the async work and group.leave() in the completion handler.

Critical rule: Every enter() must have exactly one matching leave(). Extra leave() calls crash (EXC_BAD_INSTRUCTION). Missing leave() calls cause the group to never complete. Always use defer { group.leave() } to ensure balance.

8. Why should you avoid calling DispatchSemaphore.wait() on the main thread? Senior

semaphore.wait() blocks the calling thread until signal() is called. On the main thread, this freezes the UI — no touch events, no animations, no screen updates. If the work that will call signal() also needs the main thread (e.g., a UIKit callback, a Combine sink on DispatchQueue.main), you get a deadlock: the main thread is blocked waiting for a signal that can only be delivered on the main thread.

Even without deadlock, blocking the main thread for more than a few seconds triggers the watchdog timer, and iOS will kill your app (0x8badf00d — "ate bad food" — the watchdog termination code).

9. Compare DispatchSource.makeTimerSource with Timer (NSTimer). When would you choose each? L6

Timer (NSTimer): Requires a RunLoop. Added to a specific RunLoop mode. Pauses during scrolling if in .default mode. Fires on the thread whose RunLoop it's attached to. Tolerance property for energy optimization.

DispatchSource timer: Does not require a RunLoop. Fires on any dispatch queue. Not affected by RunLoop modes — always fires. Leeway parameter for energy optimization (allows coalescing). More precise. Must be manually resumed (starts suspended).

Choose Timer when: You need RunLoop mode awareness (e.g., intentionally pausing during scroll), or you're in a RunLoop-centric context.

Choose DispatchSource timer when: You need reliable firing regardless of RunLoop mode, you need to fire on a background queue, or you need precise timing.

10. What causes a deadlock with dispatch_sync, and how does it apply transitively through target queues? L6

Calling sync on the current serial queue deadlocks because: the current block holds the queue's execution token, and sync waits for that token to become available — classic self-deadlock.

Transitive deadlock: If queue A targets queue B (serial), and you're executing on queue A and call sync on queue B, it deadlocks. The block on A is holding B's execution token (through the target relationship), and sync on B waits for that token.

This also applies to the main queue: calling DispatchQueue.main.sync from the main thread deadlocks, because the main queue is serial and you're already on it.

Detection: libdispatch has some runtime deadlock detection that logs warnings, but it doesn't catch all transitive cases. The best prevention is to never call sync on a queue you might already be on, and to document target queue relationships.

Operation Queues (Questions 11–16)

11. Why must custom async Operations manually fire KVO notifications for isExecuting and isFinished? L6

The OperationQueue uses KVO (Key-Value Observing) to monitor operation state transitions. For synchronous operations (override main()), the base class automatically manages KVO: it sets isExecuting=true before calling main(), and isFinished=true after main() returns.

For asynchronous operations (override start()), the base class can't know when your async work completes — it returns from start() immediately. You must manually fire willChangeValue(forKey:) and didChangeValue(forKey:) for both "isExecuting" and "isFinished" when the state changes.

If you forget: The queue never removes the operation from its operations array. It stays in the queue forever. Any operations that depend on it will never start. The queue's operationCount never reaches zero. This is one of the most common bugs with custom async operations.

12. When an operation is cancelled, are its dependent operations also cancelled? Senior

No. Cancelling an operation does NOT automatically cancel its dependents. A cancelled operation still transitions to isFinished = true (after its start() is called), which satisfies the dependency. The dependent operation will then start with its own isCancelled still false.

For cascading cancellation, you must either: (1) check the dependency's isCancelled in the dependent's main(), (2) cancel the entire queue with cancelAllOperations(), or (3) implement custom cancellation propagation logic.

13. What happens if you create a circular dependency between two operations? Senior

If operation A depends on B and B depends on A, neither will ever execute. Both operations remain in the "pending" state forever because each is waiting for the other to finish. The queue silently deadlocks — there's no crash, no warning, no error. The operations just sit in the queue's operations array indefinitely.

For complex pipelines, you should validate that your dependency graph is a DAG (Directed Acyclic Graph) before adding operations to the queue. You can do this with a topological sort or cycle detection algorithm.

14. When should you choose OperationQueue over GCD or async/await? L6

Choose OperationQueue when:

  • You need complex dependency graphs — "B depends on A, C depends on A and B" — that would be awkward to express with nested closures or async/await chains
  • You need KVO observation of task state (isReady, isExecuting, isFinished) for progress UI or monitoring
  • You need reusable operation classes that encapsulate complex work with inputs, outputs, and cancellation
  • You're building a data processing pipeline where operations represent discrete stages

Choose GCD when: Simple async dispatch, barriers for reader-writer patterns, DispatchSource for system events.

Choose async/await when: Writing new code — it's safer (no thread explosion), more readable, and has better cancellation support. It's the default choice for most new code.

15. How does BlockOperation handle multiple execution blocks? Are they serial or concurrent? Senior

When you add multiple execution blocks to a BlockOperation via addExecutionBlock(_:), they run concurrently, not sequentially. The operation is only considered finished when ALL blocks have completed. The completionBlock runs after all execution blocks finish.

This means if your blocks share mutable state, you'll have a data race. For sequential execution, use separate operations with dependencies instead.

16. How would you pass data between dependent operations? L6

Operations don't have a built-in data passing mechanism. Common patterns:

  • Output properties: The producing operation stores results in a property. The consuming operation accesses the dependency's property in its main() via dependencies.compactMap { $0 as? ProducerOperation }.first?.output.
  • Adapter operations: Create a generic adapter that transforms one operation's output into another's input.
  • Shared context object: Pass a mutable context object that operations read from and write to (requires synchronization).
  • Closure-based wiring: Set up closures between operations before adding them to the queue.

The output property pattern is most common. It's safe because the dependency is guaranteed to be finished (isFinished = true) before the dependent starts, so reading the output property doesn't race.

Swift Concurrency (Questions 17–28)

17. How does async/await work under the hood? What are async frames and how do they differ from stack frames? L6

The Swift compiler transforms async functions into state machines. Each await point becomes a state transition. The function's local variables are stored in a heap-allocated async frame (not on the thread's stack), so they survive suspension.

Stack frames: Allocated on the thread's stack, automatically freed when the function returns. Tied to a specific thread.

Async frames: Heap-allocated structures that hold the function's local state across suspension points. Not tied to any thread — the function can resume on a different thread than it suspended on.

This is why async functions can suspend without holding a thread: their state lives on the heap. The tradeoff is heap allocation overhead, but this is amortized by the massive reduction in thread count (cooperative thread pool is limited to CPU core count, typically 6-8 on modern iPhones).

18. What is the critical difference between Task { } and Task.detached { }? L6

Task { } inherits the creating context: actor isolation, priority, and task-local values. If called from a @MainActor context, the task runs on the MainActor.

Task.detached { } does not inherit anything: no actor isolation (runs on the default concurrent executor), no priority inheritance (defaults to .medium), no task-local values.

Both are unstructured — neither is a child of the creating task. They can outlive their creating scope. You must store the Task handle and cancel it manually to avoid leaks.

Common mistake: Using Task.detached from a @MainActor context and then trying to update UI without explicitly hopping back to MainActor. Use Task { } when you want to stay on the same actor; use Task.detached { } only when you explicitly want to escape the current actor context.

19. Explain structured concurrency. Why is it important, and how does TaskGroup implement it? L6

Structured concurrency means concurrent tasks form a tree — child tasks are scoped to their parent, and the parent doesn't complete until all children complete (or are cancelled). This eliminates resource leaks and dangling tasks.

TaskGroup implements structured concurrency: child tasks created via group.addTask are children of the group. The withTaskGroup block doesn't return until all children complete. If the parent is cancelled, all children are cancelled. In ThrowingTaskGroup, if one child throws, the remaining children are automatically cancelled.

Why it matters: Without structured concurrency (e.g., fire-and-forget Task { }), you can create tasks that outlive their creating scope, leak resources, or continue doing work after the user has navigated away. Structured concurrency makes the task lifecycle match the code scope — when the scope exits, all concurrent work is guaranteed to be done.

20. How does task cancellation work in Swift concurrency? Is it preemptive or cooperative? Senior

Task cancellation is cooperative. Calling task.cancel() sets a flag but does not stop the task. The task must check for cancellation and respond:

  • Task.checkCancellation(): Throws CancellationError if cancelled
  • Task.isCancelled: Returns a Bool to check without throwing
  • Task.sleep: Automatically throws CancellationError if cancelled during sleep

In ThrowingTaskGroup, when one child throws, the group cancels remaining children — but those children still need to check for cancellation to actually stop.

Well-behaved async APIs (like URLSession.data(from:)) check for cancellation internally and throw when cancelled.

21. What is priority escalation in Swift concurrency? Give a concrete example. L6

Priority escalation occurs when a high-priority task awaits a lower-priority task. The runtime detects the dependency and boosts the lower-priority task's priority to match the waiter, preventing priority inversion.

Example: A .userInitiated task calls await lowPriorityTask.value where lowPriorityTask was created with .background priority. The runtime walks the dependency chain and escalates the background task to .userInitiated priority. For actors, this means the actor's executor will schedule the boosted job ahead of other same-priority jobs.

This is similar to GCD's priority inheritance but works at the task level rather than the thread level.

22. What are TaskLocal values? How do they differ from thread-local storage? L6

@TaskLocal values are scoped to a task and its children. They propagate automatically through the task hierarchy — child tasks (including TaskGroup children) inherit their parent's task-local values.

vs Thread-local storage: Thread-local values are tied to a specific thread. Since async code can resume on any thread, thread-local values are useless across await points — the task might resume on a different thread and see a different thread-local value. TaskLocal values follow the task, not the thread, so they work correctly across suspensions.

Key behaviors: Task { } inherits task-local values. Task.detached { } does NOT inherit them. Values are set via $taskLocal.withValue(value) { ... } and are scoped to that closure.

Use cases: Request IDs, trace IDs, user context, feature flags — anything that should propagate through a call chain without passing parameters explicitly.

23. What is the Sendable protocol? Why does Swift need it? Senior

Sendable is a marker protocol indicating a type is safe to share across concurrency domains (actors, tasks). Swift needs it because when data crosses an actor boundary (or is captured by a @Sendable closure), both the sender and receiver could access it concurrently — a data race.

Implicitly Sendable: Value types where all stored properties are Sendable; actor types; final classes with only immutable Sendable properties; tuples of Sendable types.

@unchecked Sendable: An escape hatch for types that handle their own synchronization (e.g., a class with a lock protecting mutable state). The compiler trusts you — if you're wrong, you get data races.

In Swift 6, all Sendable violations are compile errors by default, making data race safety a compile-time guarantee.

24. What is the difference between withCheckedContinuation and withUnsafeContinuation? Senior

Both bridge callback-based APIs to async/await. The difference is safety checking:

withCheckedContinuation: Runtime checks that the continuation is resumed exactly once. Double-resume causes a crash. Never-resume generates a runtime warning in debug. Slight performance overhead for the checking.

withUnsafeContinuation: No runtime checks. Double-resume is undefined behavior (memory corruption). Never-resume is a silent hang. Zero overhead.

Rule: Use withCheckedContinuation for most code. Only use withUnsafeContinuation in hot paths where the checking overhead matters and you've thoroughly verified correctness.

25. How would you limit concurrency in a TaskGroup to process 1000 items with max 10 concurrent tasks? L6

Use the sliding window pattern: seed the group with an initial batch of N tasks, then as each completes, add the next one. This maintains a constant concurrency level.

await withTaskGroup(of: Result.self) { group in
    var iterator = items.makeIterator()
    // Seed with initial batch
    for _ in 0..<min(10, items.count) {
        if let item = iterator.next() {
            group.addTask { await process(item) }
        }
    }
    // As each completes, add the next
    for await result in group {
        results.append(result)
        if let item = iterator.next() {
            group.addTask { await process(item) }
        }
    }
}

This pattern ensures exactly 10 tasks are in-flight at any time, preventing resource exhaustion while maximizing throughput.

26. Describe the cooperative thread pool used by Swift concurrency. How does it prevent thread explosion? L6

Swift concurrency uses a cooperative thread pool with a width equal to the number of CPU cores (typically 6-8 on modern iPhones). Unlike GCD's thread pool (which can grow to 64+ threads), the cooperative pool has a hard limit.

How it prevents thread explosion: Tasks don't block threads — they suspend at await points, freeing the thread to run other tasks. You can have thousands of in-flight tasks without creating thousands of threads. Each suspended task only consumes a small heap-allocated async frame (~a few hundred bytes), not a 512KB thread stack.

Under the hood: The pool is backed by dispatch_workloop_t (introduced in iOS 12), which provides priority-ordered execution. The Swift runtime's default concurrent executor manages job scheduling across these worker threads.

Important rule: You must never block a cooperative thread (e.g., with semaphore.wait() or synchronous I/O). Blocking reduces the effective pool width, potentially causing a thread starvation deadlock where all threads are blocked and no task can make progress.

27. How would you migrate a GCD-based codebase to async/await? What's the recommended strategy? Architect

Bottom-up migration:

  1. Phase 1 — Leaf functions: Wrap callback-based APIs with withCheckedContinuation. Convert network calls, file I/O, and database operations to async.
  2. Phase 2 — Service layer: Convert repositories and use cases. Replace DispatchGroup with TaskGroup. Replace serial queues protecting state with actors.
  3. Phase 3 — Presentation layer: Convert ViewModels from Combine to @MainActor + async. Replace @Published with @Observable where possible.

Bridge patterns: Provide both async and callback versions during transition. Use withCheckedContinuation to wrap legacy callbacks. Use Task { } to call async code from synchronous contexts.

Key decisions: Enable -strict-concurrency=complete early to see the scope. Use @unchecked Sendable as a temporary bridge. Migrate one module at a time. Keep both APIs until all callers migrate.

28. What happens if you resume a withCheckedContinuation zero times? What about more than once? Senior

Zero times: The task hangs forever — it's suspended waiting for a resume that never comes. With withCheckedContinuation, a runtime warning is generated in debug builds. With withUnsafeContinuation, it's a silent hang with no warning.

More than once: With withCheckedContinuation, the second resume triggers a runtime crash (trap). With withUnsafeContinuation, it's undefined behavior — typically memory corruption because the continuation's memory has already been freed.

Always ensure every code path through your callback resumes the continuation exactly once. Use defer or careful control flow to guarantee this.

Actors & Isolation (Questions 29–38)

29. Explain actor reentrancy. Why is it the #1 gotcha in Swift concurrency? L6

Actor reentrancy means that when an actor method suspends at an await point, other messages can be processed by the actor before the original method resumes. This breaks the assumption that actor state is unchanged across an await.

Why it's dangerous: Developers assume actors work like locks — exclusive access for the entire method. But actors release their "lock" at every await point. If you check a condition, await something, then act on the condition, the condition may have changed. This leads to duplicate work, violated invariants, and logic bugs.

Classic example: An image cache that checks if a URL is cached, awaits a download, then stores the result. Multiple concurrent calls for the same URL will all see a cache miss and all start downloads — the check-then-act pattern is broken by reentrancy.

Fix: Track in-flight work. Before starting an async operation, check if one is already in progress for the same key. If so, await the existing operation instead of starting a new one.

30. How does @MainActor differ from DispatchQueue.main.async? What are the advantages? L6

@MainActor advantages:

  • Compile-time enforcement: The compiler verifies that @MainActor code is only called from the main actor or with await. DispatchQueue.main has no compile-time checking.
  • Granularity: @MainActor can be applied to types, methods, properties, and closures. DispatchQueue.main only works at the block level.
  • Sendable checking: The compiler verifies that data crossing the MainActor boundary is Sendable. No such checking with DispatchQueue.main.
  • Optimization: @MainActor may skip the hop if already on the main actor. DispatchQueue.main.async always dispatches.
  • Async integration: After an await in a @MainActor method, execution automatically resumes on the main actor. With GCD, you must manually dispatch back to main.
31. What is a global actor? When would you create a custom one instead of using @MainActor? L6

A global actor is a singleton actor that can be used as an attribute to isolate code to a specific serial execution context. @MainActor is the built-in global actor for the main thread.

Create a custom global actor when: You have a subsystem that needs its own serial execution context — e.g., a database layer where all operations must be serialized but shouldn't block the main thread. A @DatabaseActor ensures all database code runs on the same serial executor without manual queue management.

Definition: Apply @globalActor to an actor type with a static let shared property. Then use it as an attribute on types, methods, and properties.

Tradeoff: Every cross-actor call requires an await (context switch). Don't create global actors for fine-grained operations — batch work into fewer, larger calls.

32. What does the nonisolated keyword do? When is it required? Senior

nonisolated opts a method or property out of actor isolation. It can be called synchronously from any context but cannot access the actor's mutable (var) state — only immutable (let) properties.

Required for: Protocol conformances that expect synchronous access. For example, Hashable, Equatable, and CustomStringConvertible require synchronous methods. Since actor methods are async from outside, you must mark these conformances as nonisolated.

nonisolated(unsafe): An escape hatch that disables isolation checking entirely. The compiler won't verify safety — you must ensure thread safety yourself. Use only for bridging to legacy code.

33. What changes in Swift 6 strict concurrency mode? How do you prepare for it? Architect

Swift 6 makes data race safety a compile-time guarantee. All Sendable violations that were warnings become errors:

  • Non-Sendable types crossing actor boundaries → error
  • Mutable captures in @Sendable closures → error
  • Global mutable variables → error (must be actor-isolated or Sendable)
  • Protocol conformance isolation mismatches → error

Preparation strategy:

  1. Enable -strict-concurrency=complete in Swift 5.10 to see all warnings
  2. Fix warnings module-by-module, starting with leaf modules
  3. Use @unchecked Sendable as a temporary bridge for types you can't modify yet
  4. Use nonisolated(unsafe) for global variables that are genuinely safe
  5. Enable Swift 6 language mode per-target as each is ready
  6. Remove @unchecked annotations as you properly fix each type
34. Can actors have inheritance? Why or why not? Senior

No, actors cannot inherit from other actors (or classes). This is by design: inheritance would complicate the isolation model. If a subclass could override an isolated method, it could potentially break the parent's isolation guarantees. The actor's serial executor is tied to the specific actor instance, and inheritance hierarchies would make it unclear which executor should be used.

Actors can conform to protocols and can be extended. For shared behavior, use protocols with default implementations or composition (an actor containing another actor).

35. How do actors handle their serial executor under the hood? L6

Each actor has a serial executor that processes jobs one at a time. By default, this is backed by a DispatchSerialQueue (or more precisely, a dispatch_workloop_t which provides priority-ordered execution).

When you call an actor method from outside, the call is wrapped in a job and submitted to the actor's executor queue. The executor processes jobs serially — only one job runs at a time, ensuring mutual exclusion of the actor's state.

At await points within an actor method, the current job yields the executor. Other pending jobs can then execute (this is reentrancy). When the awaited operation completes, a new job is created to resume the original method, and it's submitted back to the executor queue.

You can provide a custom executor by conforming to SerialExecutor protocol, which is useful for integrating actors with existing threading models (e.g., a Core Data managed object context that must run on a specific thread).

36. When do actors hurt performance? How do you design actor granularity? Architect

Actors hurt performance when:

  • Too fine-grained: An actor per pixel, per character, or per small data item creates millions of actor hops. Each hop involves a context switch through the executor.
  • Hot path contention: A single actor accessed by many concurrent tasks becomes a bottleneck — all callers serialize through its executor.
  • Excessive cross-actor calls: Each cross-actor call requires an await (potential suspension + resume). In tight loops, this overhead dominates.

Granularity guidelines:

  • Per-subsystem: One actor for a network manager, one for a cache — moderate contention, clear boundaries
  • Per-entity: One actor per chat room, per document — independent entities with their own state
  • Per-resource: One actor for a database connection pool — protecting a specific shared resource
  • Batch operations: Instead of calling an actor method 1000 times, provide a batch method that processes all items in one actor hop
37. What is the difference between MainActor.run { } and Task { @MainActor in }? L6

MainActor.run { } is an inline hop — the calling task suspends and resumes on the MainActor. The calling code waits for the block to complete. It's like a synchronous call that happens to execute on a different actor.

Task { @MainActor in } creates a new unstructured task that runs on the MainActor. The calling code continues immediately — it's fire-and-forget.

Use MainActor.run when: You need the result of the main-actor work, or you need it to complete before continuing.

Use Task { @MainActor in } when: You want fire-and-forget UI updates, or you're in a synchronous context that can't use await.

38. How would you fix the actor reentrancy problem in an image cache? L6

Use an in-flight tracking dictionary that maps URLs to their pending download tasks. Before starting a download, check if one is already in progress for that URL. If so, await the existing task instead of starting a new one.

actor ImageCache {
    private var cache: [URL: UIImage] = [:]
    private var inFlight: [URL: Task<UIImage, Error>] = [:]

    func fetchImage(url: URL) async throws -> UIImage {
        if let cached = cache[url] { return cached }
        if let existing = inFlight[url] {
            return try await existing.value
        }
        let task = Task {
            let (data, _) = try await URLSession.shared.data(from: url)
            return UIImage(data: data)!
        }
        inFlight[url] = task
        do {
            let image = try await task.value
            cache[url] = image
            inFlight[url] = nil
            return image
        } catch {
            inFlight[url] = nil
            throw error
        }
    }
}

This ensures only one download per URL, with all concurrent callers piggy-backing on the same task.

Async Sequences & Combine (Questions 39–48)

39. What is the default buffering policy of AsyncStream? Why is it dangerous for high-frequency producers? L6

The default buffering policy is .unbounded. If the producer yields values faster than the consumer processes them, the internal buffer grows without limit, eventually causing an out-of-memory crash.

For high-frequency producers (sensor data at 100Hz, network events), always specify a bounded policy: .bufferingNewest(N) keeps the N most recent values (drops oldest), or .bufferingOldest(N) keeps the N oldest values (drops newest).

40. How does backpressure work in Combine vs AsyncSequence? L6

Combine: Explicit demand-based backpressure. The subscriber tells the publisher how many values it can handle via Subscribers.Demand. The publisher should not send more values than demanded. In practice, most operators (and sink) request .unlimited, effectively disabling backpressure.

AsyncSequence: Natural pull-based backpressure. The consumer calls next() (via for await) and the producer is implicitly paused until the consumer is ready for the next value. However, AsyncStream with a continuation can buffer values if the producer yields faster than the consumer iterates, subject to the buffering policy.

Key difference: Combine's backpressure is opt-in and often ignored. AsyncSequence's backpressure is structural — the consumer drives the pace by calling next().

41. Explain the difference between debounce and throttle in Combine. Senior

debounce: Waits for a pause in values. Emits the last value after the specified interval of silence. If new values keep arriving, the timer resets. Use case: search-as-you-type (wait until user stops typing).

throttle: Emits at most one value per time interval. Has a latest parameter: if true, emits the most recent value in the window; if false, emits the first value. Use case: scroll position updates (limit to 10 updates/second).

Example: Input: A(0ms), B(50ms), C(100ms), [silence 400ms], D(500ms)

  • debounce(300ms): emits C at 400ms, D at 800ms
  • throttle(300ms, latest:true): emits A at 0ms, C at 300ms, D at 500ms
42. What is the most common memory leak with Combine, and how do you prevent it? Senior

The #1 leak: not storing the AnyCancellable. If you don't store the cancellable returned by sink, it's immediately deallocated, cancelling the subscription before receiving any values.

The #2 leak: retain cycles in sink closures. If the sink closure captures self strongly, and self holds the cancellable, you get a cycle: self → cancellable → sink closure → self.

Prevention:

  • Always use .store(in: &cancellables) to store cancellables
  • Use [weak self] in sink closures
  • Use .assign(to: &$property) instead of sink when assigning to @Published (no retain cycle)
43. How do you bridge a Combine publisher to an AsyncSequence? Senior

Use the built-in .values property on any Combine publisher:

let publisher = NotificationCenter.default.publisher(for: .someNotification)
for await notification in publisher.values {
    process(notification)
}

For @Published properties: for await value in viewModel.$property.values { ... }

You can also apply Combine operators first, then bridge: publisher.debounce(for: .milliseconds(300), scheduler: RunLoop.main).values

For custom bridging (e.g., adding buffering), wrap the publisher in an AsyncStream using sink to yield values to the continuation.

44. What is the @Published property wrapper gotcha regarding timing of value delivery? L6

@Published sends the new value to subscribers in willSet — before the property is actually updated. If you access the property directly in the sink closure (instead of using the received value), you'll see the old value.

model.$count.sink { newValue in
    print(newValue)      // ✅ New value (from publisher)
    print(model.count)   // ⚠️ OLD value (property not yet updated)
}

Always use the value passed to the sink, not the property itself. This is a subtle but common source of bugs.

45. Compare Combine, AsyncSequence, and the Observation framework. When would you use each in a modern app? Architect

Observation (@Observable): Best for UI state management in SwiftUI. Automatic per-property tracking (only re-renders views that read changed properties). Lowest learning curve. Use for ViewModels and UI state.

async/await + AsyncSequence: Best for streaming data and async operations. Natural backpressure. Integrates with structured concurrency. Use for network streams, file processing, and bridging delegate APIs.

Combine: Best for complex event processing with rich operators (debounce, combineLatest, switchToLatest). Multi-subscriber support. Use for complex reactive chains that have no AsyncSequence equivalent.

Recommended architecture (2024+):

  • UI Layer: @Observable + SwiftUI
  • ViewModel: @Observable + async/await
  • Service Layer: async/await + AsyncSequence
  • Complex reactive logic: Combine (keep existing, don't add new)
  • New streaming APIs: AsyncStream
46. What is the difference between receive(on:) and subscribe(on:) in Combine? Senior

receive(on:) affects downstream: values and completion are delivered on the specified scheduler. Use it to ensure UI updates happen on the main thread.

subscribe(on:) affects upstream: the subscription itself and the publisher's work happen on the specified scheduler. Use it to move heavy publisher work off the main thread.

In practice, receive(on: DispatchQueue.main) is used ~95% of the time. subscribe(on:) is less common because most publishers already do their work on appropriate threads.

47. How would you build a custom AsyncSequence for paginated API fetching? L6

Create a struct conforming to AsyncSequence with a nested AsyncIterator. The iterator's next() method fetches the next page, returns the items, and tracks pagination state. When the API returns fewer items than the page size, set an isExhausted flag and return nil on subsequent calls.

Key considerations: check Task.checkCancellation() before each network call; handle errors by throwing from next(); track the current page/cursor in the iterator's mutable state.

Usage: for try await page in PaginatedFetcher(url: baseURL, pageSize: 50) { ... }

48. What happens to an AsyncStream's continuation if you call finish() and then yield()? Senior

Yielding after finish() is silently ignored. The value is dropped. No crash, no error. However, it indicates a logic error in your code — you're producing values after declaring the stream is done.

Similarly, calling finish() multiple times is safe — subsequent calls are ignored. But storing a continuation that never has finish() called will keep the consuming task suspended forever.

RunLoop & Main Thread (Questions 49–57)

49. Why does Timer stop firing when the user scrolls a UIScrollView? How do you fix it? Senior

When a UIScrollView begins tracking a touch, the main RunLoop switches from .default mode to .tracking mode. Timer.scheduledTimer adds the timer to .default mode only. Since the RunLoop is now in .tracking mode, timers in .default mode are not processed.

Fixes:

  • Add the timer to .common mode: RunLoop.main.add(timer, forMode: .common) — .common includes both .default and .tracking
  • Use DispatchSource.makeTimerSource — not RunLoop-dependent, always fires
  • Use Combine's Timer.publish(every:on:in:) with .common mode
50. What is the difference between RunLoop.main and DispatchQueue.main? L6

RunLoop.main: The event processing loop on the main thread. Mode-aware — respects RunLoop modes. Work scheduled via RunLoop.main.perform may be delayed during scrolling (if in .default mode).

DispatchQueue.main: A serial dispatch queue bound to the main thread. NOT mode-aware — blocks always execute regardless of RunLoop mode. Implemented as a Source0 on the main RunLoop, which is processed in all modes.

Key implication: DispatchQueue.main.async is more reliable for general main-thread dispatch because it's not affected by RunLoop mode changes. Use RunLoop.main only when you specifically need mode-aware behavior.

In Combine, .receive(on: RunLoop.main) may delay delivery during scroll, while .receive(on: DispatchQueue.main) always delivers.

51. How does @MainActor relate to the RunLoop and DispatchQueue.main? L6

@MainActor is backed by the main thread's executor, which uses DispatchQueue.main (not RunLoop.main). This means @MainActor code behaves like DispatchQueue.main.async — it executes regardless of RunLoop mode, including during scrolling.

The MainActor's executor is a SerialExecutor that dispatches jobs to the main dispatch queue. When you await a @MainActor method, the continuation is submitted to the main queue and executed during the RunLoop's Source0 processing phase.

52. What are Source0 and Source1 in CFRunLoop? Give examples of each. L6

Source0 (non-port-based): Application-triggered sources. Must be manually signaled via CFRunLoopSourceSignal + CFRunLoopWakeUp. Examples: touch events (delivered by UIKit), performSelector:onThread:, GCD main queue blocks.

Source1 (port-based): Kernel-triggered sources via Mach ports. Automatically wake the RunLoop when a message arrives. Examples: inter-process communication from SpringBoard (touch hardware events before UIKit processing), CFMachPort callbacks, system events.

The key difference: Source0 requires explicit signaling (the app must tell the RunLoop "there's work"), while Source1 is automatically triggered by kernel events (the kernel wakes the RunLoop).

53. How would you detect UI jank (dropped frames) using RunLoop observers? Architect

Install a CFRunLoopObserver that monitors .afterWaiting and .beforeWaiting activities. Measure the time between these two events — this is the duration of one RunLoop iteration (one frame). If it exceeds 16.67ms (60fps) or 8.33ms (120fps ProMotion), a frame was dropped.

For production monitoring, log jank events with stack traces (using Thread.callStackSymbols) to identify the code causing the delay. Aggregate these reports to find the most common jank sources.

Alternative: Use CADisplayLink to measure frame timing. Compare targetTimestamp - timestamp with the actual time between callbacks. If the gap is larger than expected, frames were dropped.

Apple's MetricKit (MXMetricManager) also provides hang rate metrics in production without custom instrumentation.

54. Why must all UIKit operations happen on the main thread? Senior

Three reasons:

  1. UIKit is not thread-safe: UIView and its subclasses use non-atomic properties. Concurrent access from multiple threads causes data races — corrupted state, crashes, visual glitches.
  2. Rendering pipeline: Core Animation commits the layer tree to the render server at the end of each main RunLoop iteration. Modifying layers from a background thread races with this commit, causing rendering artifacts or crashes.
  3. Event handling: Touch events are delivered on the main thread through the responder chain. Modifying the view hierarchy from another thread while touches are being processed causes undefined behavior.

In Swift 6, @MainActor provides compile-time enforcement. In earlier versions, Xcode's Main Thread Checker (runtime diagnostic) catches violations.

55. What is the frame budget for 60fps and 120fps (ProMotion)? How do you stay within it? L6

60fps: 16.67ms per frame. 120fps (ProMotion): 8.33ms per frame.

Every piece of work on the main thread eats into this budget: touch event processing, layout passes, display updates, Core Animation commits, timer callbacks, and your application logic.

Techniques to stay within budget:

  • Move all non-UI work off the main thread (async/await, GCD)
  • Chunk expensive main-thread work across multiple RunLoop iterations
  • Use CADisplayLink for frame-synchronized work
  • Defer non-critical work to idle time via CFRunLoopPerformBlock
  • Profile with Instruments (Time Profiler, Core Animation template)
  • Avoid off-screen rendering triggers (cornerRadius + masksToBounds, shadows without shadowPath)
  • Use prefetching for table/collection views
56. What is the watchdog timer, and what does the 0x8badf00d termination code mean? Senior

iOS has a watchdog timer that monitors the main thread. If the main thread is blocked (unresponsive) for too long — typically around 20 seconds for foreground apps, less for background — the watchdog kills the app.

The termination code 0x8badf00d (reads "ate bad food") indicates a watchdog kill. You'll see this in crash logs as the exception code. Common causes: synchronous network calls on the main thread, semaphore.wait() on main, heavy computation blocking the main RunLoop.

During app launch, the watchdog is stricter — you have about 20 seconds to present the first frame, or the app is killed.

57. Design a concurrency architecture for a large iOS app. What tools would you use at each layer? Architect

Recommended architecture:

  • UI Layer: @MainActor for all UIKit/SwiftUI code. @Observable for SwiftUI state. No manual threading.
  • ViewModel Layer: @MainActor + async/await. Use Task { } for data loading. Store Task handles for cancellation on view disappear.
  • Service Layer: Actors for stateful services (caches, session managers). Async functions for stateless operations (API calls, parsing).
  • Data Layer: @DatabaseActor (custom global actor) for Core Data/SQLite. AsyncStream for real-time data (WebSocket, location).
  • Legacy Bridge: withCheckedContinuation for callback APIs. Combine for complex reactive chains that predate async/await.

Concurrency contract: All new code uses async/await + actors. GCD is only in the networking layer. Combine is only in existing ViewModel bindings (being migrated to @Observable). No raw threads. No semaphores on main thread. All types crossing actor boundaries must be Sendable.
