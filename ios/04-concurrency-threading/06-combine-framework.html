1. Publisher / Subscriber / Subscription

Combine is built on three core protocols that form a demand-driven reactive pipeline. Understanding this triad is essential — it's what makes Combine different from simple callback chains.

Combine Protocol Flow: ┌───────────┐ ┌──────────────┐ ┌────────────┐ │ Publisher │────1───▶│ Subscription │◀───2────│ Subscriber │ │ │ │ │ │ │ │ Produces │ │ Manages the │ │ Consumes │ │ values │ │ connection │ │ values │ └───────────┘ └──────────────┘ └────────────┘ Flow: 1. Subscriber calls subscribe() on Publisher 2. Publisher creates Subscription and sends it to Subscriber 3. Subscriber requests demand via Subscription.request(.max(N)) 4. Publisher sends up to N values via Subscriber.receive(_:) 5. Each receive() returns new Demand (additive) 6. Publisher sends completion via Subscriber.receive(completion:) ┌──────────┐ subscribe() ┌──────────┐ │Publisher │ ──────────────▶ │Subscriber│ │ │ receive(sub) │ │ │ │ ◀────────────── │ │ │ │ │ │ │ │ request(.max(3)) │ │ │ ◀────────────── │ │ │ │ │ │ │ │ receive(value1)│ │ │ │ ──────────────▶ │ returns │ │ │ receive(value2)│ .none │ │ │ ──────────────▶ │ or │ │ │ receive(value3)│ .max(N) │ │ │ ──────────────▶ │ │ │ │ │ │ │ │ receive(.finished) │ │ │ ──────────────▶ │ │ └──────────┘ └──────────┘ 
// The three core protocols
protocol Publisher {
    associatedtype Output
    associatedtype Failure: Error
    func receive<S: Subscriber>(subscriber: S)
        where S.Input == Output, S.Failure == Failure
}

protocol Subscriber {
    associatedtype Input
    associatedtype Failure: Error
    func receive(subscription: Subscription)
    func receive(_ input: Input) -> Subscribers.Demand
    func receive(completion: Subscribers.Completion<Failure>)
}

protocol Subscription: Cancellable {
    func request(_ demand: Subscribers.Demand)
}

2. Backpressure: The Demand Model

Combine's backpressure model is demand-based: the subscriber tells the publisher how many values it can handle. This prevents the producer from overwhelming the consumer.

// Demand is additive and never decreases
let demand1 = Subscribers.Demand.max(3)  // Request 3 values
let demand2 = Subscribers.Demand.none     // Request 0 more values
let demand3 = Subscribers.Demand.unlimited // No backpressure

// Custom subscriber with backpressure
class BatchSubscriber<Input, Failure: Error>: Subscriber {
    let batchSize: Int
    var subscription: Subscription?
    var received = 0

    init(batchSize: Int) {
        self.batchSize = batchSize
    }

    func receive(subscription: Subscription) {
        self.subscription = subscription
        subscription.request(.max(batchSize))  // Request first batch
    }

    func receive(_ input: Input) -> Subscribers.Demand {
        received += 1
        process(input)

        if received % batchSize == 0 {
            return .max(batchSize)  // Request next batch
        }
        return .none  // Don't request more yet
    }

    func receive(completion: Subscribers.Completion<Failure>) {
        subscription = nil
    }
}
Common Pitfall: .unlimited Demand Disables Backpressure

Most Combine operators (and sink) request .unlimited demand, effectively disabling backpressure. This means if your publisher produces values faster than the subscriber processes them, values queue up in memory. For high-frequency publishers (sensor data, network events), use buffer, throttle, or debounce operators to control the flow.

3. CurrentValueSubject & PassthroughSubject

Subjects are publishers that you can imperatively send values to. They bridge imperative code into the reactive world.

// PassthroughSubject: no initial value, no replay
let events = PassthroughSubject<UserEvent, Never>()

events.send(.login)
events.send(.viewProfile)
events.send(completion: .finished)

// Late subscribers miss all previously sent values
events.sink { event in
    print(event)  // Never called — events already sent
}

// CurrentValueSubject: has initial value, replays latest
let temperature = CurrentValueSubject<Double, Never>(72.0)

temperature.sink { value in
    print("Current: \(value)")  // Immediately prints 72.0
}

temperature.send(75.0)  // Prints 75.0
temperature.value         // 75.0 — synchronous access to current value

// Real-world usage: ViewModel state
class SearchViewModel {
    let searchResults = CurrentValueSubject<[Result], Never>([])
    let isLoading = CurrentValueSubject<Bool, Never>(false)
    let error = PassthroughSubject<Error, Never>()

    func search(_ query: String) {
        isLoading.send(true)
        api.search(query) { [weak self] result in
            self?.isLoading.send(false)
            switch result {
            case .success(let items):
                self?.searchResults.send(items)
            case .failure(let err):
                self?.error.send(err)
            }
        }
    }
}
Feature PassthroughSubject CurrentValueSubject
Initial value None Required
Replay on subscribe No Yes (latest value)
Synchronous access No Yes (.value)
Use case Events, actions, one-shot signals State, current values

4. Key Operators Deep Dive

4.1 Transformation Operators

// map: transform each value
[1, 2, 3].publisher
    .map { $0 * 2 }  // 2, 4, 6

// flatMap: transform to publisher, flatten results
// CRITICAL: flatMap in Combine has maxPublishers parameter
userIDs.publisher
    .flatMap(maxPublishers: .max(3)) { id in
        api.fetchUser(id: id)  // Returns a publisher
    }
    // Max 3 concurrent inner publishers

// compactMap: transform + filter nil
strings.publisher
    .compactMap { Int($0) }  // Only valid integers

// scan: accumulate values (like reduce, but emits each step)
[1, 2, 3, 4].publisher
    .scan(0, +)  // 1, 3, 6, 10

4.2 Combining Operators

// combineLatest: emit when ANY publisher emits, using latest from each
let username = PassthroughSubject<String, Never>()
let password = PassthroughSubject<String, Never>()

username.combineLatest(password)
    .map { user, pass in
        !user.isEmpty && pass.count >= 8
    }
    .sink { isValid in
        submitButton.isEnabled = isValid
    }

// merge: interleave values from multiple publishers (same type)
let local = localDB.fetchUsers()
let remote = api.fetchUsers()
local.merge(with: remote)
    .sink { users in display(users) }

// zip: pair values 1:1 (waits for both to have a value)
let names = ["Alice", "Bob"].publisher
let ages = [30, 25].publisher
names.zip(ages)
    .sink { name, age in
        print("\(name): \(age)")  // "Alice: 30", "Bob: 25"
    }

4.3 Timing Operators

// debounce: wait for pause in values
searchTextField.publisher
    .debounce(for: .milliseconds(300), scheduler: RunLoop.main)
    .removeDuplicates()
    .sink { query in
        performSearch(query)
    }

// throttle: emit at most once per interval
scrollView.publisher
    .throttle(for: .milliseconds(100), scheduler: RunLoop.main, latest: true)
    .sink { offset in
        updateParallax(offset)
    }

// delay: shift all values forward in time
publisher
    .delay(for: .seconds(2), scheduler: DispatchQueue.main)

// timeout: fail if no value within duration
publisher
    .timeout(.seconds(5), scheduler: DispatchQueue.main)
    .sink(receiveCompletion: { completion in
        if case .failure = completion {
            print("Timed out!")
        }
    }, receiveValue: { _ in })
debounce vs throttle: Input: ─A──B─C───────D──E─F─G────────H──────▶ time debounce(300ms): ─────────C──────────────G──────H──────▶ (emits after 300ms silence) throttle(300ms, latest: true): ─A────────C────D────────G──────H──────▶ (emits at most once per 300ms window) throttle(300ms, latest: false): ─A────────B────D────────E──────H──────▶ (emits FIRST value in each 300ms window)

4.4 Error Handling Operators

// catch: replace error with fallback publisher
api.fetchData()
    .catch { error in
        Just(fallbackData)  // Fallback publisher
    }

// retry: retry the upstream publisher N times
api.fetchData()
    .retry(3)  // Try up to 4 times total
    .catch { _ in Just(fallbackData) }

// mapError: transform error type
api.fetchData()
    .mapError { AppError.network($0) }

// replaceError: replace any error with a default value
api.fetchData()
    .replaceError(with: defaultData)

5. @Published Property Wrapper

class SettingsViewModel: ObservableObject {
    @Published var username: String = ""
    @Published var notifications: Bool = true
    @Published private(set) var isValid: Bool = false

    private var cancellables = Set<AnyCancellable>()

    init() {
        // $username is a Publisher<String, Never>
        $username
            .map { !$0.isEmpty && $0.count >= 3 }
            .assign(to: &$isValid)  // Assign to another @Published
    }
}
Common Pitfall: @Published Fires BEFORE the Value Changes

In the willSet observer, @Published sends the new value to subscribers. But if you access the property directly in the sink closure (not the received value), you'll see the old value because the property hasn't been updated yet. Always use the value passed to the sink, not the property itself.

class Model: ObservableObject {
    @Published var count = 0
}

let model = Model()
model.$count.sink { newValue in
    print("Received: \(newValue)")   // ✅ New value
    print("Property: \(model.count)") // ⚠️ May be OLD value!
}

6. Memory Management: AnyCancellable & store(in:)

Combine subscriptions are managed through AnyCancellable tokens. When the token is deallocated, the subscription is automatically cancelled. This is the primary memory management mechanism.

class ProfileViewController: UIViewController {
    private var cancellables = Set<AnyCancellable>()

    func bindViewModel() {
        // store(in:) adds the AnyCancellable to the set
        viewModel.$name
            .sink { [weak self] name in
                self?.nameLabel.text = name
            }
            .store(in: &cancellables)

        viewModel.$avatar
            .sink { [weak self] image in
                self?.avatarView.image = image
            }
            .store(in: &cancellables)
    }

    // When the VC is deallocated, cancellables set is deallocated,
    // all AnyCancellables are deallocated, all subscriptions cancelled.
}
Common Pitfall: Forgetting to Store the Cancellable

If you don't store the AnyCancellable returned by sink, it's immediately deallocated, and the subscription is cancelled before receiving any values. This is the #1 Combine bug.

// ❌ BUG: Cancellable not stored — subscription immediately cancelled
publisher.sink { value in
    print(value)  // Never called!
}

// ✅ CORRECT: Store the cancellable
let cancellable = publisher.sink { value in
    print(value)
}
// Or use store(in:)

6.1 Retain Cycle Risks

// ❌ Retain cycle: sink captures self strongly
class ViewModel {
    var cancellable: AnyCancellable?

    func setup() {
        cancellable = publisher.sink { value in
            self.process(value)  // Strong capture of self
            // self → cancellable → sink closure → self (CYCLE)
        }
    }
}

// ✅ Fix: Use [weak self]
cancellable = publisher.sink { [weak self] value in
    self?.process(value)
}

// ✅ Alternative: Use assign(to:) for @Published (no retain cycle)
publisher
    .assign(to: &$publishedProperty)  // Uses inout, no retain cycle

7. Scheduling & Threading

// receive(on:): switch downstream to a specific scheduler
api.fetchData()
    .receive(on: DispatchQueue.main)  // UI updates on main thread
    .sink { data in
        self.label.text = data.title  // ✅ Main thread
    }

// subscribe(on:): switch upstream subscription to a specific scheduler
heavyPublisher
    .subscribe(on: DispatchQueue.global(qos: .background))
    .receive(on: DispatchQueue.main)
    .sink { result in
        updateUI(result)
    }

// Schedulers available:
// - DispatchQueue (most common)
// - RunLoop (for RunLoop-sensitive operations)
// - ImmediateScheduler (synchronous, for testing)
// - OperationQueue
Deep Dive: receive(on:) vs subscribe(on:)

receive(on:) affects downstream — values and completion are delivered on the specified scheduler. subscribe(on:) affects upstream — the subscription itself (and the publisher's work) happens on the specified scheduler. In practice, receive(on: DispatchQueue.main) is used 95% of the time to ensure UI updates happen on the main thread.

9. Architect: When to Use Combine in 2024+

Architect Must-Know

Combine is in maintenance mode — Apple hasn't added significant new features since iOS 15. The Observation framework and async/await are the future. But Combine isn't dead: it's deeply embedded in existing codebases, and its operator library is still unmatched. The Staff-level question is: when do you keep Combine, and when do you migrate?

9.1 Keep Combine For

  • Complex event processing: debounce + combineLatest + switchToLatest chains that have no AsyncSequence equivalent
  • Multi-subscriber scenarios: share(), multicast(), where multiple consumers need the same stream
  • Existing codebases: If your ViewModel layer is Combine-based, don't rewrite just for the sake of it
  • Timer/scheduling: Timer.publish, RunLoop integration

9.2 Migrate Away From Combine For

  • Simple async operations: Replace Combine chains that just fetch-then-display with async/await
  • SwiftUI state: Replace @Published + ObservableObject with @Observable
  • Streaming data: Replace custom Publishers with AsyncStream
  • New code: Default to async/await + @Observable for all new features
Best Practice: Hybrid Architecture
  • New features: @Observable + async/await
  • Existing Combine code: Keep working, bridge to async via .values where needed
  • Complex reactive logic: Keep Combine for debounce/throttle/combineLatest chains
  • Gradual migration: Replace @Published with @Observable one ViewModel at a time
  • Never mix in the same pipeline: Don't convert Combine→AsyncSequence→Combine — pick one per data flow
