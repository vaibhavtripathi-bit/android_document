1. The Actor Model: How Actors Provide Data Race Safety

An actor is a reference type (like a class) that isolates its mutable state. All access to an actor's properties and methods from outside the actor must go through a serialized "mailbox" — ensuring only one task accesses the actor's state at a time.

1.1 Actor Internals

Actor Internal Architecture: ┌──────────────────────────────────────────────┐ │ actor BankAccount │ │ │ │ ┌─────────────────────────────────────────┐ │ │ │ Serial Executor │ │ │ │ (backed by DispatchSerialQueue or │ │ │ │ custom SerialExecutor) │ │ │ │ │ │ │ │ ┌──────┐ ┌──────┐ ┌──────┐ ┌──────┐ │ │ │ │ │Job 1 │→│Job 2 │→│Job 3 │→│Job 4 │ │ │ │ │ └──────┘ └──────┘ └──────┘ └──────┘ │ │ │ │ ▲ executing queued ──────────▶ │ │ │ └─────────────────────────────────────────┘ │ │ │ │ ┌─────────────────────────────────────────┐ │ │ │ Isolated State │ │ │ │ var balance: Double │ │ │ │ var transactions: [Transaction] │ │ │ └─────────────────────────────────────────┘ │ └──────────────────────────────────────────────┘ External access: must await (goes through executor queue) Internal access: synchronous (already on the executor) 
actor BankAccount {
    let id: String           // let properties: accessible without await
    private(set) var balance: Double  // var properties: isolated
    private var transactions: [Transaction] = []

    init(id: String, initialBalance: Double) {
        self.id = id
        self.balance = initialBalance
    }

    // Methods are isolated: only one caller at a time
    func deposit(_ amount: Double) {
        balance += amount
        transactions.append(Transaction(type: .deposit, amount: amount))
    }

    func withdraw(_ amount: Double) throws {
        guard balance >= amount else {
            throw BankError.insufficientFunds
        }
        balance -= amount
        transactions.append(Transaction(type: .withdrawal, amount: amount))
    }
}

// External access requires await
let account = BankAccount(id: "123", initialBalance: 1000)
let balance = await account.balance       // await required for var
let id = account.id                       // No await needed for let
try await account.withdraw(500)           // await required for methods

2. Actor Reentrancy: The #1 Gotcha

This is the most important concept in this entire document. Actor reentrancy means that when an actor method suspends at an await point, other messages can be processed by the actor before the original method resumes. This can break invariants that you assumed held across the await.

Critical: Actor Reentrancy Breaks Assumptions

Actors are NOT like locks. A lock holds exclusive access for the entire critical section. An actor releases its "lock" at every await point. If you check a condition, await something, then act on the condition — the condition may have changed.

actor ImageCache {
    private var cache: [URL: UIImage] = [:]

    // ❌ BUG: Reentrancy problem
    func fetchImage(url: URL) async throws -> UIImage {
        if let cached = cache[url] {
            return cached
        }

        // ⚠️ SUSPENSION POINT: actor is "unlocked" here
        // Another call to fetchImage(url:) for the SAME URL
        // can start executing while we're awaiting
        let (data, _) = try await URLSession.shared.data(from: url)

        // We resume, but another call may have already cached this URL
        // We're doing a DUPLICATE download!
        let image = UIImage(data: data)!
        cache[url] = image
        return image
    }
}
Actor Reentrancy Timeline: Task A calls fetchImage(url: X) │ ├─ Check cache: miss ├─ Start URLSession download ──── await (actor "unlocked") ────┐ │ │ │ Task B calls fetchImage(url: X) │ │ ├─ Check cache: still miss (A hasn't finished) │ │ ├─ Start ANOTHER download ── await ──┐ │ │ │ │ │ │ │ Task C calls fetchImage(url: Y) │ │ │ │ ├─ Check cache: miss │ │ │ │ ├─ Start download ── await ──┐ │ │ │ │ │ │ │ │ │ │ │ ◀── A's download completes ────┘ │ │ │ │ A resumes: cache[X] = image │ │ │ │ │ │ │ ◀── B's download completes (DUPLICATE!) │ │ │ B resumes: cache[X] = image (overwrites, wasted work) │ │ │ │ │ ◀── C's download completes │ │ C resumes: cache[Y] = image │

2.1 Fixing Reentrancy: In-Flight Tracking

actor ImageCache {
    private var cache: [URL: UIImage] = [:]
    private var inFlight: [URL: Task<UIImage, Error>] = [:]

    // ✅ CORRECT: Handles reentrancy
    func fetchImage(url: URL) async throws -> UIImage {
        // Check cache
        if let cached = cache[url] {
            return cached
        }

        // Check if already downloading
        if let existing = inFlight[url] {
            return try await existing.value  // Piggyback on existing download
        }

        // Start new download and track it
        let task = Task {
            let (data, _) = try await URLSession.shared.data(from: url)
            guard let image = UIImage(data: data) else {
                throw ImageError.decodingFailed
            }
            return image
        }

        inFlight[url] = task

        do {
            let image = try await task.value
            cache[url] = image
            inFlight[url] = nil
            return image
        } catch {
            inFlight[url] = nil
            throw error
        }
    }
}

2.2 General Reentrancy Rules

Best Practice: Reentrancy Safety
  • Never assume state is unchanged after an await — always re-check conditions after suspension
  • Use in-flight tracking for deduplication of async work
  • Perform state mutations synchronously — do all state changes before or after await, never split across one
  • Use the "check-await-check" pattern: validate preconditions, await, re-validate before mutating
  • Consider whether you need an actor at all — if all operations are synchronous, a struct or class with a lock may be simpler

3. @MainActor: What It Means, How It Works

@MainActor is a global actor that ensures code runs on the main thread. It's the compile-time replacement for DispatchQueue.main.async.

3.1 How @MainActor Works

// @MainActor on a class: ALL properties and methods are main-thread-isolated
@MainActor
class ViewModel: ObservableObject {
    @Published var items: [Item] = []  // Guaranteed main thread
    @Published var isLoading = false

    func loadItems() async {
        isLoading = true   // ✅ On main thread
        let items = try? await api.fetchItems()  // Suspends, may run elsewhere
        self.items = items ?? []  // ✅ Back on main thread after await
        isLoading = false  // ✅ On main thread
    }
}

// @MainActor on individual methods
class DataProcessor {
    func process() async -> [Item] {
        // Runs on any thread
        return await heavyWork()
    }

    @MainActor
    func updateUI(items: [Item]) {
        // Guaranteed main thread
        tableView.reloadData()
    }
}

// @MainActor on closures
func fetchAndDisplay() {
    Task { @MainActor in
        let data = try await fetchData()
        label.text = String(data: data, encoding: .utf8)
    }
}

3.2 @MainActor vs DispatchQueue.main

Aspect @MainActor DispatchQueue.main
Enforcement Compile-time Runtime only (crashes if wrong)
Granularity Type, method, property, closure Block-level only
Async integration Native (await resumes on main) Manual (dispatch back to main)
Sendable checking Compiler verifies data crossing boundaries No checking
Performance Optimized (may skip hop if already on main) Always dispatches (even if on main)
Deep Dive: MainActor.run vs Task { @MainActor in }

MainActor.run { } is a synchronous hop to the main actor — it suspends the current task and resumes on the main actor. Task { @MainActor in } creates a new unstructured task that runs on the main actor. The key difference: MainActor.run is inline (the calling task waits), while Task { @MainActor in } is fire-and-forget (the calling code continues immediately). Use MainActor.run when you need the result; use Task { @MainActor in } for fire-and-forget UI updates.

4. Global Actors: Creating Custom Ones

A global actor is a singleton actor that can be used as an attribute to isolate types, methods, and properties to a specific serial execution context. @MainActor is the most common global actor, but you can create your own.

// Defining a custom global actor
@globalActor
actor DatabaseActor {
    static let shared = DatabaseActor()
}

// Using the global actor
@DatabaseActor
class DatabaseManager {
    private var connection: SQLiteConnection?

    func query(_ sql: String) -> [Row] {
        // Guaranteed to run on DatabaseActor's serial executor
        connection?.execute(sql) ?? []
    }

    func transaction(_ block: () throws -> Void) rethrows {
        connection?.beginTransaction()
        try block()
        connection?.commitTransaction()
    }
}

// Isolated methods on non-actor types
class Repository {
    @DatabaseActor
    func saveUser(_ user: User) {
        // Runs on DatabaseActor
    }

    @MainActor
    func displayUser(_ user: User) {
        // Runs on MainActor
    }
}

// Calling across actor boundaries
func loadAndDisplay() async {
    let repo = Repository()
    let user = await repo.fetchUser()  // Hops to DatabaseActor
    await repo.displayUser(user)       // Hops to MainActor
}
Common Pitfall: Global Actor Overhead

Every call to a global-actor-isolated method requires a context switch (hop) if the caller is on a different executor. In hot paths, this can add significant latency. Don't use global actors for fine-grained operations called thousands of times — batch the work into fewer, larger calls to the actor.

5. nonisolated Keyword

nonisolated opts a method or property out of actor isolation. It can be called synchronously from any context, but cannot access the actor's isolated state.

actor UserSession {
    let userID: String
    var token: String?
    var lastActivity: Date = Date()

    init(userID: String) {
        self.userID = userID
    }

    // nonisolated: can access let properties only
    nonisolated var description: String {
        "Session for user \(userID)"  // ✅ userID is let
        // token  // ❌ Error: cannot access var from nonisolated
    }

    // nonisolated for protocol conformance
    nonisolated func hash(into hasher: inout Hasher) {
        hasher.combine(userID)  // ✅ let property
    }
}

// Common use: Hashable, Equatable, CustomStringConvertible
extension UserSession: Hashable {
    nonisolated static func == (lhs: UserSession, rhs: UserSession) -> Bool {
        lhs.userID == rhs.userID
    }
}

// nonisolated(unsafe) — escape hatch for bridging
actor LegacyBridge {
    nonisolated(unsafe) var cachedValue: String?
    // Compiler won't check isolation — YOU must ensure safety
    // Use only when bridging to legacy code that handles its own sync
}

6. Actor Isolation Checking at Compile Time

The Swift compiler performs static analysis to verify that actor-isolated state is never accessed from the wrong context. This is the core of Swift's data race safety guarantee.

6.1 Isolation Rules

actor Counter {
    var count = 0

    func increment() {
        count += 1  // ✅ Inside actor: synchronous access
    }
}

func useCounter() async {
    let counter = Counter()

    await counter.increment()    // ✅ Cross-actor: await required
    let c = await counter.count  // ✅ Cross-actor property access

    // counter.count  // ❌ Error: actor-isolated property 'count'
    //                //    can only be referenced on 'Counter'

    // counter.count += 1  // ❌ Error: cannot mutate from outside
}

// Closures and isolation
actor DataStore {
    var items: [String] = []

    func process() {
        // Synchronous closure: inherits actor isolation
        items.forEach { item in
            print(item)  // ✅ Still on actor
        }

        // Escaping async closure: loses actor isolation
        Task.detached {
            // await self.items  // Must await from detached task
            let items = await self.items
            print(items)
        }
    }
}

7. Swift 6 Strict Concurrency Mode

Swift 6 makes data race safety a compile-time guarantee. All Sendable violations that were warnings become errors. This is the biggest breaking change in Swift's history.

7.1 What Changes in Swift 6

Scenario Swift 5.x Swift 6
Non-Sendable type crossing actor boundary Warning (with strict flag) Error
Mutable capture in @Sendable closure Warning Error
Global mutable variable Allowed Error (must be actor-isolated or Sendable)
Protocol conformance isolation mismatch Warning Error
Implicit @MainActor inference Limited Expanded (UIView subclasses, etc.) 
// Swift 6 errors you'll encounter:

// 1. Global mutable state
var globalCache: [String: Any] = [:]  // ❌ Error in Swift 6
// Fix: Make it actor-isolated or use a lock

// 2. Non-Sendable type crossing boundary
class MutableConfig {
    var timeout = 30
}

func configure() async {
    let config = MutableConfig()
    await server.apply(config)  // ❌ MutableConfig is not Sendable
}

// 3. Closure capturing mutable state
var results: [Int] = []
Task {
    results.append(1)  // ❌ Mutation of captured var in @Sendable closure
}

// Fixes for common Swift 6 issues:

// Fix 1: Actor for global state
actor GlobalCache {
    static let shared = GlobalCache()
    private var storage: [String: Any] = [:]
    func get(_ key: String) -> Any? { storage[key] }
}

// Fix 2: Make types Sendable
struct ImmutableConfig: Sendable {
    let timeout: Int
}

// Fix 3: Use actor for collected results
await withTaskGroup(of: Int.self) { group in
    group.addTask { 1 }
    var results: [Int] = []
    for await result in group {
        results.append(result)  // ✅ Mutation inside structured scope
    }
}

7.2 Migration Strategy for Swift 6

Best Practice: Swift 6 Migration
  • Step 1: Enable -strict-concurrency=complete in Swift 5.10 to see all warnings
  • Step 2: Fix warnings module-by-module, starting with leaf modules
  • Step 3: Use @unchecked Sendable as a temporary escape hatch for types you can't modify yet
  • Step 4: Use nonisolated(unsafe) for global variables that are genuinely safe
  • Step 5: Enable Swift 6 language mode per-target as each is ready
  • Step 6: Remove @unchecked annotations as you properly fix each type

8. Architect: Actor Architecture for Complex Apps

Architect Must-Know

Actors are powerful but not free. Over-using actors creates "actor hop storms" where every method call requires a context switch. Under-using them leaves data races. The art is finding the right granularity.

8.1 When Actors Hurt Performance

// ❌ Anti-pattern: Too many fine-grained actors
actor PixelProcessor {
    var value: UInt8 = 0
    func adjust(by delta: Int8) { /* ... */ }
}

// Processing 1 million pixels with individual actors:
let pixels = (0..<1_000_000).map { _ in PixelProcessor() }
for pixel in pixels {
    await pixel.adjust(by: 10)  // 1M actor hops! Terrible performance.
}

// ✅ Better: One actor managing a collection
actor ImageProcessor {
    var pixels: [UInt8]

    func adjustAll(by delta: Int8) {
        for i in pixels.indices {
            pixels[i] = UInt8(clamping: Int(pixels[i]) + Int(delta))
        }
    }
}

8.2 Actor Granularity Guidelines

Granularity Example When to Use
Per-subsystem actor NetworkManager Shared services with moderate contention
Per-entity actor ChatRoom Independent entities with their own state
Per-resource actor DatabasePool Protecting a specific shared resource
Global actor @MainActor Cross-cutting concerns (UI, database)

8.3 Recommended Architecture

// Layer 1: UI (MainActor)
@MainActor
final class FeedViewController: UIViewController {
    private let viewModel: FeedViewModel
}

// Layer 2: ViewModel (MainActor for UI binding)
@MainActor
final class FeedViewModel: ObservableObject {
    @Published var posts: [Post] = []
    private let feedService: FeedService

    func refresh() async {
        posts = try? await feedService.fetchPosts()
    }
}

// Layer 3: Service (actor for thread safety)
actor FeedService {
    private let api: APIClient
    private let cache: FeedCache

    func fetchPosts() async throws -> [Post] {
        if let cached = await cache.getPosts() {
            return cached
        }
        let posts = try await api.fetchPosts()
        await cache.store(posts)
        return posts
    }
}

// Layer 4: Cache (actor for state protection)
actor FeedCache {
    private var posts: [Post]?
    private var lastFetch: Date?

    func getPosts() -> [Post]? {
        guard let lastFetch, Date().timeIntervalSince(lastFetch) < 300 else {
            return nil
        }
        return posts
    }
}

// Layer 5: API Client (nonisolated — stateless)
struct APIClient: Sendable {
    func fetchPosts() async throws -> [Post] {
        let (data, _) = try await URLSession.shared.data(from: postsURL)
        return try JSONDecoder().decode([Post].self, from: data)
    }
}
