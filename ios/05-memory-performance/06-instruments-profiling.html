Time Profiler

The Time Profiler samples the call stack at regular intervals (typically 1ms) to show where CPU time is spent. It's the first tool to reach for when investigating performance issues.

How It Works

Time Profiler Sampling: ┌───┬───┬───┬───┬───┬───┬───┬───┬───┬───┐ │ S │ S │ S │ S │ S │ S │ S │ S │ S │ S │ S = Sample (1ms intervals) └─┬─┴─┬─┴─┬─┴─┬─┴─┬─┴─┬─┴─┬─┴─┬─┴─┬─┴─┘ │ │ │ │ │ │ │ │ │ ▼ ▼ ▼ ▼ ▼ ▼ ▼ ▼ ▼ main main main main main main main main main │ │ │ │ │ │ │ │ ▼ ▼ ▼ ▼ ▼ ▼ ▼ ▼ viewDidLoad layoutSubviews drawRect │ │ │ │ ▼ ▼ ▼ ▼ loadData setupUI solve render │ ▼ JSONDecoder.decode ← 40% of samples here = bottleneck

Key Techniques

Technique How When
Call Tree (Top Down) Expand from main() down Understanding overall flow
Call Tree (Bottom Up) Start from heaviest leaf functions Finding specific bottlenecks
Invert Call Tree Checkbox in Instruments See which functions consume most time
Hide System Libraries Checkbox — shows only your code Focus on actionable items
Thread filter Select specific thread Main thread analysis for UI jank
Time range selection Drag on timeline Focus on specific interaction 
// Interpreting Time Profiler results:
//
// "Self" time: time spent IN the function (not callees)
// "Total" time: time in function + all callees
//
// If a function has high "Total" but low "Self":
//   → The bottleneck is in something it calls
//
// If a function has high "Self" time:
//   → The function itself is the bottleneck
//   → Optimize the algorithm or reduce call count
Best Practice: Profile on Device, Not Simulator

Always profile on a physical device. The Simulator runs x86 code on your Mac's CPU, which has completely different performance characteristics than the ARM chip in an iPhone. CPU-bound code may be 2-10x faster on the Simulator. Memory behavior also differs — the Simulator has access to your Mac's full RAM.

Allocations Instrument

The Allocations instrument tracks every heap allocation and deallocation, showing memory growth over time and identifying which code is allocating the most memory.

Key Views

View Shows Use For
Statistics Live/transient allocation counts by type Finding types with most live instances
Call Trees Which code paths allocate the most Finding allocation hotspots
Allocations List Every individual allocation Tracking specific object lifecycles
Generations Allocations between "Mark Generation" clicks Finding leaks during specific actions

Generation Analysis (Heap Shot)

// Generation analysis workflow:
// 1. Start Allocations instrument
// 2. Navigate to a screen → click "Mark Generation" (Generation A)
// 3. Navigate away from the screen
// 4. Click "Mark Generation" again (Generation B)
// 5. Navigate back to the screen → "Mark Generation" (Generation C)
// 6. Navigate away → "Mark Generation" (Generation D)
//
// If Generation B and D show growing "Still Living" counts
// for the same types → you have a leak!
//
// Common leaked types to watch for:
// - UIViewController subclasses
// - UIView subclasses
// - Closure contexts (__NSMallocBlock__)
// - NSNotificationCenter observers

Leaks Instrument

The Leaks instrument periodically scans the heap for objects that are allocated but unreachable from any root (stack, global, or register). It specifically detects retain cycles.

Leaks Detection: ┌──────────────────────────────────────────────────────┐ │ Periodic Scan (every ~10 seconds): │ │ │ │ 1. Pause all threads │ │ 2. Scan all roots (stack, globals, registers) │ │ 3. Follow all pointers transitively │ │ 4. Any heap allocation NOT reachable = LEAK │ │ 5. Resume threads │ │ │ │ Detects: │ │ ✅ Retain cycles (A → B → A) │ │ ✅ Orphaned allocations (no references at all) │ │ ❌ Logical leaks (reachable but unnecessary) │ │ ❌ Growing caches (reachable, just too big) │ └──────────────────────────────────────────────────────┘
Pitfall: Leaks Instrument Limitations

The Leaks instrument only finds unreachable memory. It cannot detect logical leaks — objects that are still reachable but should have been released (e.g., a growing cache, a view controller that's still in a navigation stack reference). For logical leaks, use the Allocations instrument's Generation analysis or the Memory Graph Debugger.

VM Tracker

VM Tracker shows virtual memory regions and their types (dirty, clean, swapped/compressed). It gives the most accurate picture of your app's memory footprint.

Region Type What It Contains Optimization Target?
MALLOC_* Heap allocations (your objects) Yes — reduce object count/size
CG raster data Decoded image bitmaps Yes — downsample, cache wisely
IOKit GPU buffers, Metal textures Yes — manage texture lifecycle
__TEXT Executable code (clean) Reduce binary size
__DATA Global/static variables (dirty) Reduce globals
Stack Thread stacks Reduce thread count
mapped file Memory-mapped files (clean) Usually fine (clean memory)

System Trace

System Trace shows low-level system events: thread scheduling, system calls, virtual memory faults, and inter-process communication. It's the tool for diagnosing why the main thread is blocked.

What System Trace Reveals

  • Thread states — running, blocked, preempted, interrupted
  • System calls — file I/O, network, mach messages
  • Virtual memory faults — page faults that cause disk reads
  • Priority inversions — low-priority thread holding a lock needed by high-priority thread
  • Context switches — how often your threads are preempted
// Common System Trace findings:
//
// 1. Main thread blocked on lock:
//    Main thread shows "Blocked" state waiting for a mutex
//    held by a background thread → priority inversion
//
// 2. Main thread doing file I/O:
//    System calls like read(), pread() on main thread
//    → Move to background thread
//
// 3. Excessive context switches:
//    Thread is running but frequently preempted
//    → Too many threads competing for CPU
//
// 4. Page faults during scroll:
//    VM faults causing disk reads during scrolling
//    → Prefetch data, use memory-mapped files

Metal System Trace

Metal System Trace profiles GPU work alongside CPU work, showing the full rendering pipeline from CPU submission to GPU completion.

What It Shows Why It Matters
GPU command buffer execution Is the GPU the bottleneck?
CPU encode time Is command encoding too slow?
GPU idle time Is the CPU not feeding the GPU fast enough?
Shader execution time Which shaders are expensive?
Texture uploads Are texture transfers blocking?

Xcode Memory Graph Debugger

The Memory Graph Debugger (Debug Navigator → Memory Graph button) captures a snapshot of all live objects and their reference relationships. It's the best tool for finding retain cycles interactively.

How to Use It

  1. Run the app in Xcode
  2. Navigate to the state you want to inspect
  3. Click the Memory Graph button in the debug bar
  4. Look for objects that shouldn't be alive (dismissed VCs, old views)
  5. Click an object to see its reference graph
  6. Purple exclamation marks indicate potential leaks

MXMetricManager for Production

import MetricKit

class PerformanceMetricsCollector: NSObject, MXMetricManagerSubscriber {

    func didReceive(_ payloads: [MXMetricPayload]) {
        for payload in payloads {
            // CPU metrics
            if let cpu = payload.cpuMetrics {
                report("cpu_time", value: cpu.cumulativeCPUTime)
                report("cpu_instructions", value: cpu.cumulativeCPUInstructions)
            }

            // Memory metrics
            if let mem = payload.memoryMetrics {
                report("peak_memory", value: mem.peakMemoryUsage)
                report("avg_suspended_memory", value: mem.averageSuspendedMemory)
            }

            // Launch metrics
            if let launch = payload.applicationLaunchMetrics {
                reportHistogram("time_to_first_draw",
                    histogram: launch.histogrammedTimeToFirstDraw)
                reportHistogram("resume_time",
                    histogram: launch.histogrammedApplicationResumeTime)
            }

            // Display metrics (scroll hitches)
            if let display = payload.displayMetrics {
                if let scrollMetrics = display.scrollHitchTimeRatio {
                    report("scroll_hitch_ratio", value: scrollMetrics)
                }
            }

            // Disk I/O
            if let disk = payload.diskIOMetrics {
                report("disk_writes", value: disk.cumulativeLogicalWrites)
            }
        }
    }

    func didReceive(_ payloads: [MXDiagnosticPayload]) {
        for payload in payloads {
            // Hang diagnostics (main thread blocked > 250ms)
            if let hangs = payload.hangDiagnostics {
                for hang in hangs {
                    reportHang(hang.callStackTree)
                }
            }

            // CPU exception diagnostics
            if let cpuExceptions = payload.cpuExceptionDiagnostics {
                for exception in cpuExceptions {
                    reportCPUException(exception)
                }
            }
        }
    }
}

Custom os_signpost Intervals

import os

extension OSSignpostID {
    static let networkRequest = OSSignpostID(log: .networking)
    static let imageProcessing = OSSignpostID(log: .imageProcessing)
}

extension OSLog {
    static let networking = OSLog(subsystem: "com.app", category: "Networking")
    static let imageProcessing = OSLog(subsystem: "com.app", category: "ImageProcessing")
}

class NetworkClient {
    func fetch(url: URL) async throws -> Data {
        let signpostID = OSSignpostID(log: .networking)

        os_signpost(.begin, log: .networking, name: "NetworkRequest",
            signpostID: signpostID, "URL: %{public}@", url.absoluteString)

        let (data, response) = try await URLSession.shared.data(from: url)

        let statusCode = (response as? HTTPURLResponse)?.statusCode ?? 0
        os_signpost(.end, log: .networking, name: "NetworkRequest",
            signpostID: signpostID, "Status: %d, Size: %d", statusCode, data.count)

        return data
    }
}

// These signposts appear in Instruments as named intervals
// on the Points of Interest track, with begin/end markers
// and metadata visible on hover.

OSLog Structured Logging

import os

struct AppLogger {
    static let general = Logger(subsystem: "com.app", category: "General")
    static let network = Logger(subsystem: "com.app", category: "Network")
    static let memory = Logger(subsystem: "com.app", category: "Memory")
    static let performance = Logger(subsystem: "com.app", category: "Performance")
}

// Usage with privacy controls:
AppLogger.network.info("Request to \(url, privacy: .public) completed in \(duration)ms")
AppLogger.memory.warning("Memory pressure: \(availableMB, privacy: .public)MB available")
AppLogger.performance.error("Frame drop: \(droppedFrames) frames missed")

// Log levels:
// .debug    — verbose, not persisted by default
// .info     — informational, persisted briefly
// .notice   — default level (previously "default")
// .warning  — warning conditions (previously "error")
// .error    — error conditions
// .fault    — system-level fault (always persisted)
//
// View logs: Console.app or `log stream --predicate 'subsystem == "com.app"'`

CI Performance Test Example

import XCTest

class PerformanceBudgetTests: XCTestCase {

    func testScrollPerformance() {
        let app = XCUIApplication()
        app.launch()

        let metrics: [XCTMetric] = [
            XCTClockMetric(),
            XCTMemoryMetric(),
            XCTCPUMetric()
        ]

        let options = XCTMeasureOptions()
        options.iterationCount = 5

        measure(metrics: metrics, options: options) {
            let list = app.tables.firstMatch
            list.swipeUp(velocity: .fast)
            list.swipeUp(velocity: .fast)
            list.swipeDown(velocity: .fast)
            list.swipeDown(velocity: .fast)
        }
    }

    func testMemoryBudget() {
        let app = XCUIApplication()
        measure(metrics: [XCTMemoryMetric(application: app)]) {
            app.launch()
            // Navigate through key flows
            app.tabBars.buttons["Search"].tap()
            app.tabBars.buttons["Profile"].tap()
            app.tabBars.buttons["Home"].tap()
        }
    }
}
