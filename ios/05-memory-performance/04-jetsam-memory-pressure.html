What Is Jetsam?

Jetsam is the iOS kernel's out-of-memory (OOM) killer. Unlike desktop operating systems that use swap space (virtual memory backed by disk), iOS has no swap. When physical memory runs low, Jetsam terminates processes to free memory. There is no negotiation — the process is killed with SIGKILL, which cannot be caught or handled.

Jetsam Decision Flow: ┌─────────────────────────────────────┐ │ System Memory Pressure Increases │ └──────────────┬──────────────────────┘ │ ▼ ┌─────────────────────────────────────┐ │ Jetsam evaluates all processes │ │ by priority band + memory footprint │ └──────────────┬──────────────────────┘ │ ▼ ┌─────────────────────────────────────┐ │ Kill lowest-priority processes │ │ with highest memory footprint first │ └──────────────┬──────────────────────┘ │ ▼ ┌─────────────────────────────────────┐ │ If still under pressure: │ │ Move up priority bands │ │ Eventually kill foreground app │ └─────────────────────────────────────┘ Key: Jetsam uses DIRTY memory, not total allocation

Jetsam events are logged in the device's crash logs with the jetsam reason. They look like crashes but have no stack trace — the process was killed externally by the kernel.

Pitfall: Jetsam Kills Look Like Crashes

Jetsam terminations appear in crash reporting tools as EXC_RESOURCE or simply as process terminations with no exception. They have no stack trace because the kill signal comes from the kernel, not from within the app. Many teams misclassify Jetsam kills as "random crashes" because they don't check the termination_reason field. Always check for jetsam in crash logs and use MetricKit's MXDiskWriteExceptionDiagnostic and memory metrics to identify OOM patterns.

Memory Footprint Categories

Understanding what Jetsam actually measures is critical. It doesn't care about your total allocation — it cares about your memory footprint, which is the amount of physical memory your process is using.

Memory Categories: ┌─────────────────────────────────────────────────────────┐ │ Virtual Memory │ │ ┌───────────────────────────────────────────────────┐ │ │ │ Resident Memory │ │ │ │ ┌─────────────────────────────────────────────┐ │ │ │ │ │ Memory Footprint │ │ │ │ │ │ ┌───────────────┐ ┌───────────────────┐ │ │ │ │ │ │ │ Dirty Memory │ │ Compressed Memory │ │ │ │ │ │ │ │ (modified │ │ (dirty pages │ │ │ │ │ │ │ │ pages) │ │ compressed by │ │ │ │ │ │ │ │ │ │ the kernel) │ │ │ │ │ │ │ └───────────────┘ └───────────────────┘ │ │ │ │ │ └─────────────────────────────────────────────┘ │ │ │ │ ┌─────────────────────────────────────────────┐ │ │ │ │ │ Clean Memory │ │ │ │ │ │ (memory-mapped files, purgeable, etc.) │ │ │ │ │ │ Can be evicted and reloaded from disk │ │ │ │ │ └─────────────────────────────────────────────┘ │ │ │ └───────────────────────────────────────────────────┘ │ │ ┌───────────────────────────────────────────────────┐ │ │ │ Non-Resident (paged out / not loaded) │ │ │ └───────────────────────────────────────────────────┘ │ └─────────────────────────────────────────────────────────┘ Jetsam cares about: Footprint = Dirty + Compressed

Dirty, Compressed, and Clean Memory

Category What It Is Examples Jetsam Impact
Dirty Memory written to by the app that can't be evicted Heap objects, stack, decoded images, malloc'd buffers Counts toward footprint
Compressed Dirty pages compressed by the kernel's compressor Inactive dirty pages the kernel compressed to save space Counts toward footprint (at compressed size)
Clean Memory that can be evicted and reloaded from disk Memory-mapped files, framework __TEXT segments, read-only data Does NOT count toward footprint
Purgeable Memory the app marks as discardable under pressure NSPurgeableData, NSCache contents Can be reclaimed without killing the app

What Creates Dirty Memory

// Each of these creates dirty memory:

// 1. Heap allocations (class instances)
let object = MyClass()  // Dirty: object + metadata

// 2. Decoded image bitmaps
let image = UIImage(named: "photo")
// The compressed JPEG is clean (mapped from disk)
// The decoded RGBA bitmap is DIRTY
// A 4000x3000 photo = 4000 * 3000 * 4 bytes = 48MB dirty!

// 3. Data buffers
var buffer = Data(count: 10_000_000)  // 10MB dirty

// 4. Array/Dictionary storage
var cache: [String: Data] = [:]  // Dirty: hash table + values

// 5. String buffers (for non-small strings)
let longString = String(repeating: "x", count: 100_000)  // Dirty
Deep Dive: Image Memory is the #1 Dirty Memory Source

The single biggest source of dirty memory in most iOS apps is decoded image bitmaps. A single 12MP photo (4032×3024) consumes 48MB of dirty memory when decoded to RGBA. Common mistakes: (1) loading full-resolution images for thumbnails, (2) keeping decoded images in memory after they scroll off screen, (3) not using ImageIO for downsampled loading. Use CGImageSourceCreateThumbnailAtIndex to load images at the exact size needed.

os_proc_available_memory()

Introduced in iOS 13, os_proc_available_memory() returns the number of bytes the app can allocate before Jetsam kills it. This is the most reliable way to know how close you are to the limit.

import os

func checkMemoryBudget() {
    let available = os_proc_available_memory()
    let availableMB = available / (1024 * 1024)

    print("Available memory: \(availableMB) MB")

    if availableMB < 50 {
        // Critical: purge caches immediately
        purgeCaches()
    } else if availableMB < 100 {
        // Warning: reduce memory usage
        reduceImageCache()
    }
}

// Typical values on modern devices:
// iPhone 15 Pro (8GB RAM): ~2-3GB available for foreground app
// iPhone SE 3 (4GB RAM):   ~1-1.5GB available
// iPad Pro (16GB RAM):     ~5-6GB available
// These vary based on system load, other apps, extensions
Best Practice: Periodic Memory Budget Checks

Call os_proc_available_memory() periodically (e.g., every 5 seconds via a timer, or before large allocations) to proactively manage memory. Don't wait for didReceiveMemoryWarning — by the time it fires, you may have milliseconds before Jetsam kills you. Build a tiered response: at 50% budget, stop prefetching; at 25%, purge caches; at 10%, drop everything non-essential.

didReceiveMemoryWarning Unreliability

UIViewController.didReceiveMemoryWarning() and UIApplication.didReceiveMemoryWarningNotification are not reliable for preventing Jetsam kills. Here's why:

Problem Details
Timing May fire too late — Jetsam can kill before the warning is delivered
Not guaranteed If memory spikes rapidly (e.g., large image decode), Jetsam may kill without warning
Main thread only Delivered on the main thread — if main thread is blocked, warning is delayed
Insufficient info Doesn't tell you how much memory to free or how urgent the situation is
Background apps Background apps may be killed without any warning at all 
class ViewController: UIViewController {
    override func didReceiveMemoryWarning() {
        super.didReceiveMemoryWarning()
        // This may never be called before Jetsam kills you!
        // Use os_proc_available_memory() proactively instead
        purgeImageCache()
        releaseReusableBuffers()
    }
}

Jetsam Priority Bands

Jetsam assigns each process a priority band. When memory pressure occurs, it kills processes starting from the lowest priority band, choosing the highest-memory process within each band.

Jetsam Priority Bands (lowest → highest): ┌────────────────────────────────────────────────────────┐ │ Band 0: IDLE │ │ Suspended apps with no active sessions │ ├────────────────────────────────────────────────────────┤ │ Band 1: BACKGROUND │ │ Apps in background (recently used) │ ├────────────────────────────────────────────────────────┤ │ Band 3: BACKGROUND (elevated) │ │ Apps doing background work (audio, location) │ ├────────────────────────────────────────────────────────┤ │ Band 5: MAIL / DAEMON │ │ System daemons, mail fetch │ ├────────────────────────────────────────────────────────┤ │ Band 10: FOREGROUND │ │ The currently visible app │ ├────────────────────────────────────────────────────────┤ │ Band 17: FOREGROUND (critical) │ │ Phone, FaceTime during active call │ ├────────────────────────────────────────────────────────┤ │ Band 18: JETSAM (kernel) │ │ Kernel and critical system processes │ │ (never killed — system panics instead) │ └────────────────────────────────────────────────────────┘ Kill order: Band 0 first → Band 1 → ... → Band 10 (last resort) Within each band: highest memory footprint killed first

What Determines Your Band

  • Foreground — app is visible and receiving events
  • Background (elevated) — app has active background mode (audio, location, VoIP)
  • Background — app was recently backgrounded
  • Suspended — app has been suspended by the system (no code running)
  • Idle — app has no active scenes or sessions

Background Memory Limits

Background apps have significantly lower memory limits than foreground apps. Exceeding these limits results in immediate Jetsam termination.

State Approximate Limit Notes
Foreground ~50-70% of device RAM Varies by device and system load
Background (active) ~50-80MB Apps doing background work
App Extensions ~40-120MB Depends on extension type
Widget Extensions ~30MB Very constrained
Notification Service ~24MB Must process quickly 
// Handling background transition — reduce memory immediately
func applicationDidEnterBackground(_ application: UIApplication) {
    // You have seconds to get under ~50MB or risk Jetsam
    ImageCache.shared.removeAll()
    URLCache.shared.removeAllCachedResponses()
    releaseDecodedImages()
    compactDataStructures()

    let available = os_proc_available_memory()
    print("Background available: \(available / 1024 / 1024) MB")
}
Pitfall: Background Memory Spikes

A common pattern that causes Jetsam kills: the app is backgrounded, a background fetch or push notification triggers processing, and the processing temporarily spikes memory above the background limit. The app is killed even though the spike is temporary. Solution: check os_proc_available_memory() before starting background work, and process data in small batches.

MetricKit Memory Reports

MetricKit provides production memory metrics that are aggregated and delivered daily. This is the only way to understand real-world memory behavior across your user base.

import MetricKit

class MemoryMetricsSubscriber: NSObject, MXMetricManagerSubscriber {

    func didReceive(_ payloads: [MXMetricPayload]) {
        for payload in payloads {
            if let memoryMetrics = payload.memoryMetrics {
                let peakMemory = memoryMetrics.peakMemoryUsage
                print("Peak memory: \(peakMemory)")

                let averageSuspended = memoryMetrics.averageSuspendedMemory
                print("Avg suspended: \(averageSuspended)")
            }
        }
    }

    func didReceive(_ payloads: [MXDiagnosticPayload]) {
        for payload in payloads {
            if let crashDiagnostics = payload.crashDiagnostics {
                for diagnostic in crashDiagnostics {
                    if diagnostic.terminationReason?.contains("jetsam") == true {
                        print("Jetsam kill detected!")
                        reportJetsamEvent(diagnostic)
                    }
                }
            }
        }
    }
}

// Register in AppDelegate or App init
let subscriber = MemoryMetricsSubscriber()
MXMetricManager.shared.add(subscriber)

Memory Pressure Dispatch Source

For more granular memory pressure notifications than didReceiveMemoryWarning, use a DispatchSource for memory pressure events:

class MemoryPressureMonitor {
    private var source: DispatchSourceMemoryPressure?

    func startMonitoring() {
        source = DispatchSource.makeMemoryPressureSource(
            eventMask: [.warning, .critical],
            queue: .main
        )

        source?.setEventHandler { [weak self] in
            guard let source = self?.source else { return }
            let event = source.data

            if event.contains(.critical) {
                print("🔴 CRITICAL memory pressure")
                self?.handleCriticalPressure()
            } else if event.contains(.warning) {
                print("🟡 WARNING memory pressure")
                self?.handleWarningPressure()
            }
        }

        source?.resume()
    }

    private func handleWarningPressure() {
        URLCache.shared.memoryCapacity = 0
        ImageCache.shared.trimToSize(50_000_000)
    }

    private func handleCriticalPressure() {
        ImageCache.shared.removeAll()
        URLCache.shared.removeAllCachedResponses()
        NSCache<AnyObject, AnyObject>().removeAllObjects()
    }

    deinit {
        source?.cancel()
    }
}

Building a Memory Budget System

Architect Must-Know: Memory Budget System for Large Apps

At the Staff level, you should be able to design a memory budget system that prevents Jetsam kills proactively. Here's a complete architecture:

Architecture Overview

Memory Budget System Architecture: ┌─────────────────────────────────────────────────────────┐ │ MemoryBudgetManager │ │ │ │ ┌──────────────┐ ┌──────────────┐ ┌──────────────┐ │ │ │ Budget Tiers │ │ Polling Loop │ │ Subscribers │ │ │ │ │ │ │ │ │ │ │ │ Green: >50% │ │ Timer-based │ │ ImageCache │ │ │ │ Yellow: 25% │ │ os_proc_ │ │ URLCache │ │ │ │ Orange: 10% │ │ available_ │ │ DataManager │ │ │ │ Red: <5% │ │ memory() │ │ FeatureFlags │ │ │ └──────────────┘ └──────────────┘ └──────────────┘ │ │ │ │ ┌──────────────────────────────────────────────────┐ │ │ │ Per-Subsystem Budgets │ │ │ │ Images: 200MB | Network: 50MB | Data: 100MB │ │ │ │ Each subsystem reports usage, receives limits │ │ │ └──────────────────────────────────────────────────┘ │ └─────────────────────────────────────────────────────────┘ 
import os

enum MemoryPressureLevel {
    case normal      // > 50% available
    case warning     // 25-50% available
    case critical    // 10-25% available
    case terminal    // < 10% available
}

protocol MemoryBudgetSubscriber: AnyObject {
    var currentMemoryUsage: Int { get }
    func reduceMemory(to level: MemoryPressureLevel)
}

class MemoryBudgetManager {
    static let shared = MemoryBudgetManager()
    private var subscribers: [WeakSubscriber] = []
    private var timer: Timer?
    private let logger = Logger(subsystem: "com.app", category: "memory")

    var currentLevel: MemoryPressureLevel = .normal {
        didSet {
            if currentLevel != oldValue {
                notifySubscribers()
            }
        }
    }

    func startMonitoring(interval: TimeInterval = 5.0) {
        timer = Timer.scheduledTimer(withTimeInterval: interval, repeats: true) {
            [weak self] _ in
            self?.checkMemory()
        }
    }

    func register(_ subscriber: MemoryBudgetSubscriber) {
        subscribers.append(WeakSubscriber(subscriber))
    }

    private func checkMemory() {
        let available = os_proc_available_memory()
        let totalEstimate = available + currentFootprint()
        let ratio = Double(available) / Double(totalEstimate)

        let newLevel: MemoryPressureLevel
        switch ratio {
        case 0.5...:  newLevel = .normal
        case 0.25...: newLevel = .warning
        case 0.10...: newLevel = .critical
        default:      newLevel = .terminal
        }

        logger.info("Memory: \(available/1024/1024)MB available, level: \(String(describing: newLevel))")
        currentLevel = newLevel
    }

    private func currentFootprint() -> Int {
        var info = task_vm_info_data_t()
        var count = mach_msg_type_number_t(
            MemoryLayout<task_vm_info_data_t>.size / MemoryLayout<natural_t>.size
        )
        let result = withUnsafeMutablePointer(to: &info) {
            $0.withMemoryRebound(to: integer_t.self, capacity: Int(count)) {
                task_info(mach_task_self_, task_flavor_t(TASK_VM_INFO), $0, &count)
            }
        }
        guard result == KERN_SUCCESS else { return 0 }
        return Int(info.phys_footprint)
    }

    private func notifySubscribers() {
        subscribers.removeAll { $0.subscriber == nil }
        for weak in subscribers {
            weak.subscriber?.reduceMemory(to: currentLevel)
        }
    }
}

private class WeakSubscriber {
    weak var subscriber: MemoryBudgetSubscriber?
    init(_ subscriber: MemoryBudgetSubscriber) {
        self.subscriber = subscriber
    }
}
