Stack vs Heap Allocation

Process Memory Layout (simplified): ┌──────────────────────────────┐ High addresses │ Stack │ ← grows downward │ (function frames, locals) │ │ ↓ │ ├──────────────────────────────┤ │ │ │ (unmapped gap) │ │ │ ├──────────────────────────────┤ │ ↑ │ │ Heap │ ← grows upward │ (malloc'd objects, classes) │ ├──────────────────────────────┤ │ Uninitialized Data (BSS) │ ├──────────────────────────────┤ │ Initialized Data │ │ (global vars, statics) │ ├──────────────────────────────┤ │ Text (Code) │ │ (read-only, executable) │ └──────────────────────────────┘ Low addresses
Aspect Stack Heap
Allocation speed ~1 instruction (move stack pointer) ~100+ instructions (malloc search)
Deallocation Automatic (function return) Manual (free/ARC release)
Thread safety Per-thread (no contention) Shared (lock contention in malloc)
Size limit ~1MB main thread, ~512KB secondary Limited by available memory
Fragmentation None (LIFO) Yes (requires compaction/coalescing)
Cache locality Excellent (contiguous) Poor (scattered)
What lives here Local value types, function params, return addresses Class instances, closures, escaped values 
func stackVsHeap() {
    // Stack allocated (value type, doesn't escape)
    var point = CGPoint(x: 10, y: 20)  // 16 bytes on stack
    var numbers = (1, 2, 3)               // Tuple on stack

    // Heap allocated (reference type)
    let view = UIView()                    // Object on heap
    let array = [1, 2, 3]                  // Buffer on heap

    // Stack frame layout for this function:
    // [return address] [saved registers] [point] [numbers] [view ptr] [array ptr]
    //                                     ↑ stack  ↑ stack    ↑ heap     ↑ heap
}

Value Types vs Reference Types

Characteristic Value Types (struct, enum, tuple) Reference Types (class, closure)
Default storage Stack (inline) Heap
Assignment Copy (bitwise for simple types) Reference copy (pointer)
Identity No identity (value equality) Has identity (===)
Reference counting None (unless contains refs) Yes (ARC)
Thread safety Inherently safe (copies) Requires synchronization
Polymorphism Protocol-based (witness tables) Inheritance-based (vtable)

The Hidden Cost of Value Types with References

// This struct is NOT free to copy — it contains reference types
struct Message {
    let id: UUID              // 16 bytes, value type ✅
    let text: String           // 16 bytes, but may heap-allocate buffer
    let sender: User          // Class → retain on copy ⚠️
    let attachments: [Data]   // Array buffer → retain on copy ⚠️
    let metadata: [String: Any] // Dictionary → retain on copy ⚠️
}

// Copying this struct triggers 3 retain operations
// (sender, attachments buffer, metadata buffer)
let copy = message  // Not just a memcpy!
Pitfall: Structs with Many References

A struct containing N reference-type properties requires N retain operations on copy and N release operations on destruction. For structs with many reference fields (e.g., a model with 10+ String properties), the copy cost can exceed that of a class. Profile before assuming "structs are always faster."

When Value Types Go to the Heap

Value types are not always stack-allocated. The compiler promotes them to the heap in several situations:

1. Existential Containers (Protocol Types)

protocol Shape {
    func area() -> Double
}

struct Circle: Shape {
    var radius: Double
    func area() -> Double { .pi * radius * radius }
}

struct LargeShape: Shape {
    var a: Double, b: Double, c: Double, d: Double, e: Double
    func area() -> Double { a * b }
}

// Existential container layout (5 words = 40 bytes):
// [value buffer: 3 words (24 bytes)] [value witness table ptr] [protocol witness table ptr]

let shape1: Shape = Circle(radius: 5)
// Circle fits in 24-byte inline buffer (1 Double = 8 bytes) → stack

let shape2: Shape = LargeShape(a: 1, b: 2, c: 3, d: 4, e: 5)
// LargeShape is 40 bytes → doesn't fit in 24-byte buffer → HEAP ALLOCATED
Existential Container (40 bytes): ┌────────────────────────────────────────────────────────┐ │ Value Buffer (24 bytes) │ │ ┌──────────────────────────────────────────────────┐ │ │ │ Small value: stored inline │ │ │ │ Large value: pointer to heap allocation ──────▶ HEAP│ │ └──────────────────────────────────────────────────┘ │ ├────────────────────────────────────────────────────────┤ │ Value Witness Table pointer (8 bytes) │ │ → size, alignment, copy, destroy functions │ ├────────────────────────────────────────────────────────┤ │ Protocol Witness Table pointer (8 bytes) │ │ → protocol method implementations │ └────────────────────────────────────────────────────────┘

2. Closures Capturing Value Types

func makeCounter() -> () -> Int {
    var count = 0  // Promoted to heap — captured by escaping closure
    return {
        count += 1
        return count
    }
}

let counter = makeCounter()
counter()  // 1
counter()  // 2 — count lives on the heap

3. Inout Parameters That Escape

struct Config {
    var timeout: Int = 30

    mutating func adjustTimeout() {
        // If the compiler can't prove this doesn't escape,
        // the struct may be heap-allocated
        DispatchQueue.global().async {
            // Can't capture inout — compiler error
            // But similar patterns with closures cause heap promotion
        }
    }
}

4. Generic Contexts Without Specialization

// Without specialization, the compiler uses value witness tables
// and may heap-allocate to handle arbitrary-sized T
func process<T>(_ value: T) {
    // If T is large or the function isn't specialized,
    // value may be passed via heap-allocated buffer
}

// With @inlinable, the compiler can specialize and stack-allocate:
@inlinable func processInlinable<T>(_ value: T) {
    // Compiler generates specialized version for each T
    // → stack allocation possible
}

ContiguousArray vs Array

Array in Swift can be backed by either a Swift native array buffer or an NSArray (when bridged from ObjC). ContiguousArray is always backed by a contiguous Swift buffer.

Aspect Array ContiguousArray
ObjC bridging Can wrap NSArray Always native Swift buffer
Element access May indirect through NSArray Always direct pointer arithmetic
Performance (class elements) Slightly slower (bridge check) Faster (no bridge check)
Performance (struct elements) Same (can't bridge structs) Same
Use when General purpose, ObjC interop Performance-critical, class elements 
// For arrays of class types in performance-critical code:
var views: ContiguousArray<UIView> = []
views.reserveCapacity(1000)
for _ in 0..<1000 {
    views.append(UIView())
}

// For arrays of value types, Array and ContiguousArray are identical:
var points: [CGPoint] = []  // Same performance as ContiguousArray<CGPoint>

ManagedBuffer

ManagedBuffer provides a way to allocate a single heap block containing both a header and a contiguous array of elements. This is useful for custom data structures that need to minimize allocations.

class RingBuffer<Element> {
    private var storage: ManagedBuffer<RingHeader, Element>

    struct RingHeader {
        var head: Int
        var tail: Int
        var count: Int
        let capacity: Int
    }

    init(capacity: Int) {
        storage = ManagedBuffer<RingHeader, Element>.create(
            minimumCapacity: capacity
        ) { buffer in
            RingHeader(head: 0, tail: 0, count: 0, capacity: capacity)
        }
    }
}
ManagedBuffer Memory Layout: ┌──────────────────────────────────────────┐ │ Swift Object Header (16 bytes) │ │ (metadata pointer + refcount) │ ├──────────────────────────────────────────┤ │ Header (RingHeader) │ │ head: Int, tail: Int, count: Int, │ │ capacity: Int │ ├──────────────────────────────────────────┤ │ Element[0] │ │ Element[1] │ │ Element[2] │ │ ... │ │ Element[capacity-1] │ └──────────────────────────────────────────┘ Single allocation for header + elements → better cache locality

// Safe access to array's contiguous storage
let numbers = [1.0, 2.0, 3.0, 4.0]
numbers.withUnsafeBufferPointer { buffer in
    // buffer.baseAddress is UnsafePointer<Double>
    // buffer.count is 4
    // Can pass to C APIs like vDSP
    var sum: Double = 0
    vDSP_sveD(buffer.baseAddress!, 1, &sum, vDSP_Length(buffer.count))
}

// Manual memory allocation
let buffer = UnsafeMutableBufferPointer<Int>.allocate(capacity: 100)
defer { buffer.deallocate() }
buffer.initialize(repeating: 0)
buffer[0] = 42
Pitfall: Pointer Lifetime

Pointers obtained from withUnsafeBufferPointer are only valid within the closure. Storing them for later use is undefined behavior — the array may reallocate its buffer, leaving the pointer dangling. Similarly, UnsafeMutablePointer.allocate memory must be manually deallocated to avoid leaks.

Autorelease Pools

Autorelease pools are an Objective-C concept that still matters in Swift when bridging to ObjC APIs. An autoreleased object is not released immediately — it's added to the current autorelease pool and released when the pool drains.

When Autorelease Pools Matter in Swift

// ❌ Without autoreleasepool: memory spikes
func processImages() {
    for i in 0..<10_000 {
        // UIImage.init may return autoreleased ObjC objects
        let image = UIImage(named: "photo_\(i)")
        processImage(image)
        // Autoreleased objects accumulate until RunLoop drains pool
    }
}

// ✅ With autoreleasepool: bounded memory
func processImagesFixed() {
    for i in 0..<10_000 {
        autoreleasepool {
            let image = UIImage(named: "photo_\(i)")
            processImage(image)
        } // Pool drains here — autoreleased objects freed
    }
}

When You Need @autoreleasepool in Swift

  • Tight loops calling ObjC APIs — especially image/data processing
  • Background threads — no RunLoop to drain the default pool
  • Command-line tools — no RunLoop at all
  • Unit tests — to ensure deterministic deallocation
Deep Dive: How Autorelease Pools Work

Each thread has a stack of autorelease pool pages. When autoreleasepool { } is entered, a sentinel is pushed. When the block exits, every object pushed after the sentinel is sent a release message. The main thread's RunLoop drains the autorelease pool at the end of each event loop iteration. On background threads without a RunLoop, objects accumulate until the thread exits — which is why explicit autoreleasepool blocks are critical in background processing loops.

ObjC Bridging Costs

Bridging between Swift and Objective-C types has real performance costs that are often invisible in source code.

Swift Type ObjC Type Bridging Cost
String NSString O(1) if ASCII/small, O(n) if UTF-8 → UTF-16 conversion needed
Array NSArray O(1) lazy bridge (elements bridged on access)
Dictionary NSDictionary O(1) lazy bridge
Set NSSet O(1) lazy bridge
Data NSData O(1) (shares buffer)
Int, Double NSNumber Heap allocation (unless tagged pointer) 
// Hidden bridging cost: String → NSString
let swiftString = "Hello, World!"
let nsString = swiftString as NSString  // May allocate NSString
let length = nsString.length              // UTF-16 length

// Avoid repeated bridging in loops:
func processStrings(_ strings: [String]) {
    // ❌ Each iteration bridges String → NSString
    for s in strings {
        let ns = s as NSString
        // ... use NSString API ...
    }

    // ✅ Bridge once, iterate the NSArray
    let nsArray = strings as NSArray
    for ns in nsArray {
        // ... use NSString API ...
    }
}
Best Practice: Minimize Bridging in Hot Paths

Profile bridging costs with Instruments (Allocations instrument, filter by NSString or __NSArrayI). In performance-critical code: (1) avoid as NSString in loops, (2) use Swift-native APIs instead of Foundation when possible, (3) cache bridged values if reused, (4) use ContiguousArray for class-element arrays to avoid NSArray bridge checks.

Tagged Pointers for Small Objects

Tagged pointers store small ObjC objects directly in the pointer value, avoiding heap allocation entirely. This is transparent to code but has significant performance implications.

// These are likely tagged pointers (no heap allocation):
let n1: NSNumber = 42           // Integer fits in 60 bits
let n2: NSNumber = 3.14         // Float fits in 60 bits
let s1: NSString = "Hi"          // 2 ASCII chars fit
let d1: NSDate = NSDate()        // Timestamp fits

// These are NOT tagged pointers (heap allocated):
let s2: NSString = "This is a longer string that won't fit"
let big: NSNumber = NSNumber(value: UInt64.max)

Tagged pointers have zero retain/release cost — the runtime recognizes them and skips reference counting entirely. This makes them extremely efficient for small, frequently-created objects.

Memory Layout Optimization

Architect Must-Know: Memory Layout Optimization for Data-Heavy Apps

For apps processing large datasets (maps, media, ML), memory layout directly impacts performance through cache utilization. Key strategies:

  1. Struct-of-Arrays (SoA) vs Array-of-Structs (AoS) — if you process one field at a time, SoA gives better cache locality
  2. Minimize padding — use MemoryLayout<T>.size vs .stride to understand padding. Order fields by decreasing alignment to minimize gaps.
  3. Avoid existential containers — use generics with concrete types instead of protocol types for data collections
  4. Use ContiguousArray for class-element collections in hot paths
  5. Pool allocations — reuse buffers instead of allocating/deallocating per frame
// Understanding memory layout
print(MemoryLayout<Int>.size)       // 8 bytes
print(MemoryLayout<Int>.stride)     // 8 bytes
print(MemoryLayout<Int>.alignment)  // 8 bytes

struct Padded {
    var a: Bool    // 1 byte + 7 padding
    var b: Double  // 8 bytes
    var c: Bool    // 1 byte + 7 padding
}
// MemoryLayout<Padded>.size = 24 (with padding)

struct Packed {
    var b: Double  // 8 bytes
    var a: Bool    // 1 byte
    var c: Bool    // 1 byte + 6 padding
}
// MemoryLayout<Packed>.size = 16 (less padding)

// Note: Swift compiler may reorder fields for optimal layout
// (unlike C, field order in source ≠ field order in memory)
