Cold vs Warm vs Hot Launch

Type Definition What's Cached Typical Time
Cold App not in memory, no cached data Nothing — full dyld load 1-5 seconds
Warm App not running but dylibs cached Shared cache, some file caches 0.5-2 seconds
Hot (Resume) App suspended in background Everything — just unsuspend < 100ms

Pre-main: dyld4 and Beyond

Pre-main time is everything that happens before your main() function (or @main entry point) is called. On a complex app, this can be 50-80% of total launch time.

Pre-main Breakdown: ┌──────────────────────────────────────────────────────┐ │ dyld4 Load (typically 50-200ms) │ │ ├── Map executable into memory │ │ ├── Load dependent dylibs (recursive) │ │ ├── Apply fixups: │ │ │ ├── Rebase: adjust pointers for ASLR │ │ │ └── Bind: resolve external symbol references │ │ ├── Run initializers: │ │ │ ├── ObjC +load methods │ │ │ ├── C++ static constructors │ │ │ └── Swift type metadata registration │ │ └── Call main() │ └──────────────────────────────────────────────────────┘ Time breakdown (typical large app): dylib loading: 30-50ms Rebasing/binding: 20-40ms ObjC setup: 10-30ms Initializers: 10-100ms (highly variable)

dyld4 Closures and Shared Cache

dyld4 (introduced in iOS 16 / macOS 13) is a major rewrite of the dynamic linker. It introduces pre-built closures that cache the results of linking, making subsequent launches faster.

Shared Cache

Apple pre-links all system frameworks into a single dyld shared cache file. This means loading UIKit, Foundation, CoreGraphics, etc. is essentially free — they're already mapped into every process's address space.

dyld Shared Cache: ┌──────────────────────────────────────────────────────┐ │ /System/Library/Caches/com.apple.dyld/ │ │ │ │ Single file containing ALL system frameworks: │ │ ┌──────────┬──────────┬──────────┬──────────┐ │ │ │ UIKit │Foundation│CoreGraphi│ Metal │ ... │ │ │ │ │cs │ │ │ │ └──────────┴──────────┴──────────┴──────────┘ │ │ │ │ Pre-linked: all inter-framework references resolved │ │ Pre-optimized: ObjC selectors uniqued │ │ Shared: mapped into every process (clean memory) │ └──────────────────────────────────────────────────────┘ Your app's dylibs are NOT in the shared cache. Each custom framework adds load time.

dyld4 Closures

// dyld4 creates a "closure" on first launch that caches:
// - Which dylibs to load and in what order
// - All fixup locations (rebase + bind)
// - Initializer order
//
// On subsequent launches, dyld4 replays the closure
// instead of re-computing everything.
//
// The closure is invalidated when:
// - App is updated
// - OS is updated
// - A dylib changes
// - Device is rebooted (sometimes)
Deep Dive: ASLR and Rebasing

Address Space Layout Randomization (ASLR) loads the executable at a random base address for security. But the Mach-O binary contains pointers that assume a fixed base address. Rebasing adjusts all these internal pointers by adding the ASLR slide. Binding resolves references to external symbols (functions in other dylibs). Both are O(n) in the number of pointers — more classes and methods means more fixups. dyld4 closures cache these fixup results.

+load vs +initialize

Aspect +load +initialize
When called During pre-main (dyld initializers) Lazily, before first message to class
Guaranteed to run? Yes, always Only if class is used
Thread Main thread (blocks launch) Whatever thread sends first message
Subclass behavior Each class's +load runs independently Subclass inherits if not overridden
Category behavior Category +load runs separately Category replaces class implementation
Impact on launch Direct — adds to pre-main time None at launch (lazy)
Use case Method swizzling (legacy) One-time class setup 
// ❌ +load runs during pre-main — slows launch
@objc class AnalyticsTracker: NSObject {
    override class func load() {
        // This runs before main()!
        // Every +load adds to pre-main time
        swizzleViewDidAppear()
    }
}

// ✅ +initialize runs lazily — no launch impact
@objc class ThemeManager: NSObject {
    override class func initialize() {
        // Runs on first use, not at launch
        setupDefaultTheme()
    }
}

// ✅ Best: Use Swift static properties (lazy by default)
class Configuration {
    static let shared = Configuration()  // Lazy, thread-safe
}
Pitfall: Hidden +load Methods

Third-party SDKs often include +load methods for automatic initialization. Each one adds to your pre-main time. Audit your dependencies: run otool -oV YourApp | grep -A 1 "load" to find all +load implementations. Consider replacing SDKs that use +load with ones that support explicit initialization.

Static Initializers

C++ static initializers and Swift global variables with complex initializers run during pre-main. They're executed in the order dyld processes the initializer list.

// C++ static initializer (runs at load time)
// static MyClass instance; // Constructor runs pre-main

// Swift: global variables with simple literals are fine
let maxRetries = 3  // No initializer cost

// Swift: global variables with complex init run lazily (dispatch_once)
let dateFormatter: DateFormatter = {
    let f = DateFormatter()
    f.dateFormat = "yyyy-MM-dd"
    return f
}()
// ✅ This is lazy in Swift — runs on first access, not at launch

// ⚠️ But __attribute__((constructor)) in C/ObjC runs at load time:
// __attribute__((constructor))
// void earlyInit(void) {
//     // Runs pre-main — avoid!
// }

ObjC Class Registration

During pre-main, the ObjC runtime registers every class and category in the binary. More classes = more registration time.

// Each ObjC class adds to registration time:
// - Class metadata is read from the binary
// - Class is inserted into the runtime's class table
// - Categories are attached to their classes
// - Selectors are uniqued
//
// A large app might have 20,000+ ObjC classes
// Registration cost: ~1-2ms per 1000 classes
//
// Swift classes that don't inherit from NSObject
// don't go through ObjC registration → faster

Post-main: First Frame

Post-main time is everything from main() to the first rendered frame. This is where most optimization opportunities lie because it's your code.

// Post-main timeline:
@main
class AppDelegate: UIResponder, UIApplicationDelegate {

    func application(
        _ application: UIApplication,
        willFinishLaunchingWithOptions launchOptions: [UIApplication.LaunchOptionsKey: Any]?
    ) -> Bool {
        // ⏱ Phase 1: Minimal setup only
        // - Crash reporting (must be first)
        // - Core Data stack (if needed for first screen)
        return true
    }

    func application(
        _ application: UIApplication,
        didFinishLaunchingWithOptions launchOptions: [UIApplication.LaunchOptionsKey: Any]?
    ) -> Bool {
        // ⏱ Phase 2: Essential initialization
        // - Analytics (but defer heavy init)
        // - Push notification registration
        // - Deep link handling

        // ❌ DON'T: Initialize all SDKs here
        // ❌ DON'T: Sync network calls
        // ❌ DON'T: Heavy data processing

        return true
    }
}

os_signpost for Custom Intervals

import os

let launchLog = OSLog(subsystem: "com.myapp", category: .pointsOfInterest)

class LaunchProfiler {
    static func beginPhase(_ name: StaticString) {
        os_signpost(.begin, log: launchLog, name: name)
    }

    static func endPhase(_ name: StaticString) {
        os_signpost(.end, log: launchLog, name: name)
    }
}

// Usage in AppDelegate:
func application(_ app: UIApplication,
    didFinishLaunchingWithOptions opts: [UIApplication.LaunchOptionsKey: Any]?) -> Bool {

    LaunchProfiler.beginPhase("CrashReporting")
    CrashReporter.initialize()
    LaunchProfiler.endPhase("CrashReporting")

    LaunchProfiler.beginPhase("CoreDataStack")
    CoreDataStack.shared.loadPersistentStores()
    LaunchProfiler.endPhase("CoreDataStack")

    LaunchProfiler.beginPhase("Analytics")
    Analytics.configure()
    LaunchProfiler.endPhase("Analytics")

    return true
}

MetricKit Launch Diagnostics

import MetricKit

class LaunchMetricsSubscriber: NSObject, MXMetricManagerSubscriber {
    func didReceive(_ payloads: [MXMetricPayload]) {
        for payload in payloads {
            if let launchMetrics = payload.applicationLaunchMetrics {
                let histogram = launchMetrics.histogrammedTimeToFirstDraw
                print("Time to first draw histogram:")
                print("  p50: \(histogram.bucketEnumerator)")

                let resumeHistogram = launchMetrics.histogrammedApplicationResumeTime
                print("Resume time histogram: \(resumeHistogram)")
            }
        }
    }
}

// - histogrammedApplicationResumeTime: warm resume time

Reducing Launch Time: Real Strategies

Architect Must-Know: From 3s to Under 400ms

Here's a battle-tested playbook for reducing launch time in a large production app. Each strategy is ordered by typical impact.

Pre-main Optimizations

Strategy Typical Savings Effort
Reduce dylib count (merge into fewer frameworks) 50-200ms High
Use mergeable libraries (Xcode 15+) 30-100ms Low
Remove +load methods 10-50ms per method Medium
Reduce ObjC class count (use Swift structs) 10-30ms High
Remove unused code (dead stripping) 5-20ms Low
Avoid static initializers in C/C++ code 5-50ms Medium

Post-main Optimizations

Strategy Typical Savings Details
Defer SDK initialization 100-500ms Init analytics, ads, etc. after first frame
Lazy-load non-critical features 50-200ms Don't create what's not visible

Deferred Initialization Pattern

class AppLauncher {
    enum Priority {
        case critical    // Before first frame
        case high        // First idle moment after launch
        case medium      // Within 2 seconds of launch
        case low         // When user first interacts
    }

    private var tasks: [(priority: Priority, work: () -> Void)] = []

    func register(priority: Priority, _ work: @escaping () -> Void) {
        tasks.append((priority, work))
    }

    func executeCritical() {
        tasks.filter { $0.priority == .critical }
             .forEach { $0.work() }
    }

    func executeDeferred() {
        // Run after first frame is rendered
        DispatchQueue.main.async { [weak self] in
            self?.tasks.filter { $0.priority == .high }
                       .forEach { $0.work() }
        }

        DispatchQueue.main.asyncAfter(deadline: .now() + 2.0) { [weak self] in
            self?.tasks.filter { $0.priority == .medium }
                       .forEach { $0.work() }
        }
    }
}

// Usage:
let launcher = AppLauncher()
launcher.register(priority: .critical) { CrashReporter.start() }
launcher.register(priority: .high)     { Analytics.configure() }
launcher.register(priority: .medium)   { AdSDK.initialize() }
launcher.register(priority: .low)      { FeatureFlags.refresh() }
Best Practice: Launch Time Regression Testing

Add launch time to your CI pipeline: (1) Use XCTMetric with XCTApplicationLaunchMetric in performance tests, (2) Set a budget (e.g., 400ms cold launch), (3) Fail the build if launch time regresses beyond the budget, (4) Track trends over time with MetricKit data from production.

import XCTest

class LaunchPerformanceTests: XCTestCase {
    func testLaunchPerformance() {
        measure(metrics: [XCTApplicationLaunchMetric()]) {
            XCUIApplication().launch()
        }
    }
}
