1. UIView / CALayer Duality

Every UIView in iOS has a backing CALayer. This is not optional — it's a fundamental architectural decision baked into UIKit since iPhone OS 2.0. The view owns the layer, and the layer is responsible for all visual content: drawing, compositing, and animations.

The Relationship

┌─────────────────────────────────────────────┐ │ UIView │ │ ┌─────────────────────────────────────┐ │ │ │ Event Handling │ │ │ │ • touchesBegan/Moved/Ended │ │ │ │ • hitTest(_:with:) │ │ │ │ • gestureRecognizers │ │ │ ├─────────────────────────────────────┤ │ │ │ Layout │ │ │ │ • Auto Layout constraints │ │ │ │ • layoutSubviews() │ │ │ │ • intrinsicContentSize │ │ │ ├─────────────────────────────────────┤ │ │ │ Backing Layer (CALayer) │ │ │ │ ┌─────────────────────────────┐ │ │ │ │ │ Visual Content │ │ │ │ │ │ • contents (CGImage) │ │ │ │ │ │ • backgroundColor │ │ │ │ │ │ • borderWidth/Color │ │ │ │ │ │ • cornerRadius │ │ │ │ │ │ • shadow* │ │ │ │ │ │ • transform (CATransform3D) │ │ │ │ │ │ • opacity │ │ │ │ │ │ • Animations │ │ │ │ │ └─────────────────────────────┘ │ │ │ └─────────────────────────────────────┘ │ └─────────────────────────────────────────────┘

The UIView is the layer's delegate. When you access view.layer, you get the backing CALayer. The view sets itself as layer.delegate and responds to delegate methods like display(_:) and draw(_:in:).

// UIView owns its layer — you can access it directly
let view = UIView(frame: .init(x: 0, y: 0, width: 200, height: 200))

// The layer is always non-nil and created automatically
let layer = view.layer  // CALayer

// The view IS the layer's delegate
print(layer.delegate === view)  // true

// Many UIView properties are just pass-throughs to the layer
view.backgroundColor = .red
// Equivalent to: view.layer.backgroundColor = UIColor.red.cgColor

view.alpha = 0.5
// Equivalent to: view.layer.opacity = 0.5

view.frame = CGRect(x: 10, y: 10, width: 100, height: 100)
// Sets layer.frame, which is derived from layer.position, layer.bounds, layer.anchorPoint, layer.transform

Properties That Live on CALayer Only

Several visual properties exist only on CALayer with no UIView wrapper:

// Shadow — no UIView equivalent
view.layer.shadowColor = UIColor.black.cgColor
view.layer.shadowOffset = CGSize(width: 0, height: 2)
view.layer.shadowRadius = 4
view.layer.shadowOpacity = 0.3

// Border — no UIView equivalent
view.layer.borderWidth = 1
view.layer.borderColor = UIColor.gray.cgColor

// Corner radius — no UIView equivalent (until iOS 13 layer.cornerCurve)
view.layer.cornerRadius = 12
view.layer.maskedCorners = [.layerMinXMinYCorner, .layerMaxXMinYCorner]

// 3D transforms — UIView only exposes CGAffineTransform (2D)
view.layer.transform = CATransform3DMakeRotation(.pi / 4, 1, 0, 0)

// Contents — direct bitmap assignment
view.layer.contents = someImage.cgImage
view.layer.contentsGravity = .resizeAspectFill
view.layer.contentsScale = UIScreen.main.scale

The Layer Class Override

You can customize which CALayer subclass a view uses by overriding the class property:

class GradientView: UIView {
    // Override layerClass to use a specialized layer
    override class var layerClass: AnyClass {
        return CAGradientLayer.self
    }

    var gradientLayer: CAGradientLayer {
        return layer as! CAGradientLayer
    }

    func configure() {
        gradientLayer.colors = [
            UIColor.systemBlue.cgColor,
            UIColor.systemPurple.cgColor
        ]
        gradientLayer.startPoint = CGPoint(x: 0, y: 0)
        gradientLayer.endPoint = CGPoint(x: 1, y: 1)
    }
}

// Common specialized layer classes:
// CAShapeLayer     — vector path rendering
// CAGradientLayer  — gradient fills
// CATextLayer      — text rendering
// CAScrollLayer    — scrollable content
// CAReplicatorLayer — duplicated sublayers
// CAEmitterLayer   — particle effects
// AVPlayerLayer    — video playback
// CAMetalLayer     — Metal rendering
Deep Dive

Why override layerClass instead of adding a sublayer? When you override layerClass, the specialized layer is the view's backing layer. This means it participates directly in Auto Layout sizing, the view's draw(_:) cycle, and implicit animations triggered by UIView animation blocks. A manually-added sublayer requires you to manage its frame manually in layoutSubviews() and won't get UIView animation block support.

2. The AppKit History

To truly understand why every UIView has a CALayer, you need to understand the macOS (AppKit) history that preceded iOS.

The NSView Problem

In the original AppKit (macOS), NSView was responsible for everything: event handling, layout, and drawing. Drawing happened via drawRect: which was called by the display system whenever a region was invalidated. This had several problems:

  • No hardware acceleration — All drawing was CPU-based via Quartz 2D (Core Graphics)
  • No implicit animations — Animating required manual timer-based redrawing
  • Expensive redraws — Any property change required a full drawRect: pass
  • No compositing tree — The window server composited flat bitmaps, not a tree of layers

Core Animation's Introduction (macOS 10.5 Leopard, 2007)

Apple introduced Core Animation as a separate framework that could optionally back NSView. On macOS, you had to explicitly opt in:

// macOS — layer-backing is OPT-IN
let nsView = NSView()
nsView.wantsLayer = true  // Now has a CALayer

// Or in Interface Builder: check "Core Animation Layer" checkbox

// Without wantsLayer, NSView draws directly via drawRect:
// With wantsLayer, NSView draws into a CALayer bitmap

UIKit's Clean-Sheet Design (iPhone OS 2.0, 2008)

When Apple built UIKit for the iPhone, they had a clean sheet. They made a critical architectural decision: every UIView always has a CALayer. There's no opt-in. This was driven by the iPhone's constraints:

Constraint Why CALayer Helps
Limited CPU (412 MHz ARM) GPU compositing offloads work from CPU
Touch-driven UI 60fps animations are essential for touch feedback
Battery constraints GPU compositing is more power-efficient than CPU drawing
No mouse cursor Direct manipulation requires smooth, low-latency rendering
macOS (AppKit) Evolution: ═══════════════════════════════════════════════════════ Pre-10.5: NSView ──────────────────────▶ Window Server (CPU drawRect:) (flat compositing) 10.5+: NSView ──opt-in──▶ CALayer ──▶ Window Server (wantsLayer=true) (layer tree compositing) iOS (UIKit) — Clean Sheet: ═══════════════════════════════════════════════════════ Always: UIView ──always──▶ CALayer ──▶ Render Server (events+layout) (drawing) (backboardd → GPU)
Architect Must-Know

Why this history matters for performance at scale: The UIView/CALayer split means you can manipulate visual properties (opacity, transform, position) without triggering CPU-side draw(_:) calls. These are "compositing-only" changes handled entirely by the GPU. At scale (complex UIs with hundreds of views), understanding which property changes are compositing-only vs. which require CPU drawing is the difference between 120fps and jank.

Compositing-only properties (GPU, no CPU draw): opacity, transform, position, bounds, backgroundColor (solid colors), hidden, zPosition

Properties requiring CPU draw: contents (new bitmap), custom draw(_:), cornerRadius + masksToBounds (off-screen render), shadow without shadowPath

3. UIWindow → UIView Tree Structure

The view hierarchy is a strict tree (not a graph). Every view has exactly one superview (or none, if it's a root window). The tree starts with UIWindow.

The Full Hierarchy

UIScreen └── UIWindowScene └── UIWindow (key window) ├── UITransitionView (system-managed) │ └── UIDropShadowView │ └── UIViewController.view (root VC) │ ├── UINavigationBar │ ├── Content View │ │ ├── UILabel │ │ ├── UIButton │ │ └── Custom UIView │ │ ├── CALayer (backing) │ │ ├── Sublayer 1 │ │ └── Sublayer 2 │ └── UITabBar └── UITextEffectsWindow (keyboard, system)

UIWindow Internals

UIWindow is a subclass of UIView. It serves as the root of the view hierarchy and has special responsibilities:

class UIWindow: UIView {
    // The window level determines z-ordering among windows
    var windowLevel: UIWindow.Level  // .normal, .alert, .statusBar

    // The root view controller — setting this adds its view as a subview
    var rootViewController: UIViewController?

    // The window scene (iOS 13+)
    var windowScene: UIWindowScene?

    // Key window receives keyboard events and is the default target
    var isKeyWindow: Bool { get }
    func makeKeyAndVisible()

    // Touch dispatch entry point
    func sendEvent(_ event: UIEvent)
}

View Tree Traversal

// Walking up the tree
var current: UIView? = someDeepView
while let view = current {
    print(type(of: view))  // Walk from leaf to root
    current = view.superview
}

// Walking down the tree (depth-first)
func traverseSubviews(_ view: UIView, depth: Int = 0) {
    let indent = String(repeating: "  ", count: depth)
    print("\(indent)\(type(of: view)) frame=\(view.frame)")
    for subview in view.subviews {
        traverseSubviews(subview, depth: depth + 1)
    }
}

// Finding the window
let window = view.window  // nil if view is not in a window hierarchy

// View controller relationship
extension UIView {
    var parentViewController: UIViewController? {
        var responder: UIResponder? = self
        while let next = responder?.next {
            if let vc = next as? UIViewController {
                return vc
            }
            responder = next
        }
        return nil
    }
}
Common Pitfall

Don't confuse the view hierarchy with the layer hierarchy. They are parallel trees but not identical. A UIView's backing layer is a sublayer of its superview's backing layer. But you can add CALayer sublayers directly to a view's layer without corresponding UIViews. These "orphan" layers won't receive touch events or participate in Auto Layout.

4. Hit Testing: hitTest(_:with:) & point(inside:with:)

Hit testing is the process of determining which view should receive a touch event. It uses a reverse pre-order depth-first traversal of the view tree — starting from the last subview (topmost visually) and working backwards.

The Algorithm

Here is the default implementation of hitTest(_:with:), reconstructed from UIKit's behavior:

// Reconstructed default implementation of UIView.hitTest(_:with:)
extension UIView {
    open func hitTest(_ point: CGPoint, with event: UIEvent?) -> UIView? {
        // 1. Check if this view can receive touches at all
        guard isUserInteractionEnabled,
              !isHidden,
              alpha > 0.01 else {
            return nil
        }

        // 2. Check if the point is inside this view's bounds
        guard point(inside: point, with: event) else {
            return nil
        }

        // 3. Traverse subviews in REVERSE order (back to front → last to first)
        // This ensures the topmost (visually) view is tested first
        for subview in subviews.reversed() {
            // Convert the point to the subview's coordinate space
            let convertedPoint = subview.convert(point, from: self)
            // Recursively hit test the subview
            if let hitView = subview.hitTest(convertedPoint, with: event) {
                return hitView
            }
        }

        // 4. No subview claimed the touch — this view is the hit
        return self
    }
}

Traversal Visualization

View Hierarchy: Hit Test Order: (numbers = testing order) ┌──────────A──────────┐ │ │ A is tested LAST (step 7) │ ┌───B───┐ ┌───C───┐│ because it checks children first │ │ │ │ ││ │ │ ┌─D─┐ │ │ ┌─E─┐ ││ Traversal (reverse pre-order): │ │ └───┘ │ │ │ │ ││ 1. C (last subview of A) │ │ │ │ │┌F┐│ ││ 2. E (last subview of C) │ └───────┘ │ │└─┘│ ││ 3. F (last subview of E) │ │ └───┘ ││ 4. F fails → E checks itself │ └───────┘│ 5. E fails → C checks itself └─────────────────────┘ 6. B (first subview of A) 7. D (last subview of B) Touch at point P 8. D fails → B checks itself inside D's bounds 9. B fails → A checks itself → D is returned!

Why Reverse Order?

The subviews array is ordered by addition time — the last-added subview is at the highest index and is drawn on top. By iterating in reverse, hit testing checks the visually topmost view first. This matches user expectation: tapping on an overlapping view should hit the one that appears on top.

Common hitTest Overrides

// 1. Expanding touch area beyond bounds
class ExpandedTouchView: UIView {
    var touchInsets = UIEdgeInsets(top: -10, left: -10, bottom: -10, right: -10)

    override func point(inside point: CGPoint, with event: UIEvent?) -> Bool {
        return bounds.inset(by: touchInsets).contains(point)
    }
}

// 2. Pass-through view (transparent to touches)
class PassthroughView: UIView {
    override func hitTest(_ point: CGPoint, with event: UIEvent?) -> UIView? {
        let hitView = super.hitTest(point, with: event)
        // If the hit view is self, return nil to pass through
        return hitView == self ? nil : hitView
    }
}

// 3. Forwarding touches to a sibling view
class ForwardingView: UIView {
    weak var forwardTarget: UIView?

    override func hitTest(_ point: CGPoint, with event: UIEvent?) -> UIView? {
        guard let target = forwardTarget else {
            return super.hitTest(point, with: event)
        }
        let convertedPoint = target.convert(point, from: self)
        return target.hitTest(convertedPoint, with: event)
    }
}

// 4. Allowing touches on subviews outside bounds (clipsToBounds = false)
class OverflowTouchView: UIView {
    override func hitTest(_ point: CGPoint, with event: UIEvent?) -> UIView? {
        // Don't check point(inside:) for self — check all subviews regardless
        guard isUserInteractionEnabled, !isHidden, alpha > 0.01 else {
            return nil
        }
        for subview in subviews.reversed() {
            let convertedPoint = subview.convert(point, from: self)
            if let hitView = subview.hitTest(convertedPoint, with: event) {
                return hitView
            }
        }
        return point(inside: point, with: event) ? self : nil
    }
}
Common Pitfall

Subviews outside bounds don't receive touches by default. If clipsToBounds is false, a subview can be visible outside its parent's bounds but still won't receive touches. The parent's point(inside:with:) returns false for points outside its bounds, so the subview's hitTest is never called. You must override hitTest on the parent (as shown in example 4 above) to fix this.

hitTest and Transforms

// When a view has a transform, hitTest automatically accounts for it.
// The point is converted through the inverse transform.

let button = UIButton()
button.transform = CGAffineTransform(rotationAngle: .pi / 4)  // 45° rotation

// hitTest converts the touch point through the inverse rotation,
// so the touch area matches the visual rotation.

// HOWEVER: If the transform is not invertible (e.g., scale to 0),
// hitTest returns nil — the view can't receive touches.
button.transform = CGAffineTransform(scaleX: 0, y: 0)
// hitTest always returns nil for this view

5. UIView vs CALayer Coordinate Systems

UIView and CALayer use subtly different coordinate concepts, which is a common source of confusion.

UIView: frame, bounds, center

// frame: position and size in SUPERVIEW's coordinate space
// bounds: position and size in OWN coordinate space
// center: center point in SUPERVIEW's coordinate space

let view = UIView()
view.frame = CGRect(x: 50, y: 100, width: 200, height: 150)

// These are now set automatically:
view.bounds   // CGRect(x: 0, y: 0, width: 200, height: 150)
view.center   // CGPoint(x: 150, y: 175)  — center of frame

// Changing bounds.origin scrolls the content (like UIScrollView)
view.bounds.origin = CGPoint(x: 0, y: 50)
// Subviews shift up by 50 points — this is how UIScrollView works!

CALayer: position, anchorPoint, bounds

// CALayer uses position + anchorPoint instead of center
let layer = CALayer()

// anchorPoint is in UNIT coordinates (0,0 to 1,1)
// Default is (0.5, 0.5) — the center
layer.anchorPoint = CGPoint(x: 0.5, y: 0.5)

// position is the anchorPoint's location in SUPERLAYER coordinates
layer.position = CGPoint(x: 150, y: 175)

// frame is DERIVED from position, bounds, anchorPoint, and transform
// frame = f(position, bounds.size, anchorPoint, transform)

// The relationship:
// frame.origin.x = position.x - bounds.size.width * anchorPoint.x
// frame.origin.y = position.y - bounds.size.height * anchorPoint.y

The anchorPoint Trap

// Changing anchorPoint moves the view visually because position stays fixed
let view = UIView(frame: CGRect(x: 100, y: 100, width: 200, height: 200))

// Before: anchorPoint = (0.5, 0.5), position = (200, 200)
view.layer.anchorPoint = CGPoint(x: 0, y: 0)
// After: position is still (200, 200), but now that's the top-left
// The view visually jumps! frame is now (200, 200, 200, 200)

// To change anchorPoint without visual movement:
extension UIView {
    func setAnchorPoint(_ anchorPoint: CGPoint) {
        let oldOrigin = frame.origin
        layer.anchorPoint = anchorPoint
        let newOrigin = frame.origin
        let translation = CGPoint(
            x: newOrigin.x - oldOrigin.x,
            y: newOrigin.y - oldOrigin.y
        )
        center = CGPoint(
            x: center.x - translation.x,
            y: center.y - translation.y
        )
    }
}

Coordinate Conversion

// UIView coordinate conversion
let pointInA = CGPoint(x: 50, y: 50)
let pointInB = viewA.convert(pointInA, to: viewB)
let pointInA2 = viewB.convert(pointInB, to: viewA)

// Rect conversion
let rectInWindow = view.convert(view.bounds, to: nil)  // nil = window

// CALayer coordinate conversion
let pointInLayer = layerA.convert(point, to: layerB)

// IMPORTANT: These conversions walk up the view/layer tree
// They account for transforms at each level
// They do NOT work across different windows
Concept UIView CALayer
Position reference center (always center) position (at anchorPoint)
Anchor Implicit (0.5, 0.5) anchorPoint (configurable)
Transform CGAffineTransform (2D) CATransform3D (3D)

6. Subview Ordering & Z-Index

UIView Subview Order

The subviews array determines draw order: index 0 is drawn first (back), last index is drawn last (front). This is the default z-ordering.

// Adding subviews — each new subview goes on top
parentView.addSubview(viewA)  // index 0 — back
parentView.addSubview(viewB)  // index 1 — front

// Reordering
parentView.bringSubviewToFront(viewA)   // viewA moves to front
parentView.sendSubviewToBack(viewB)      // viewB moves to back
parentView.insertSubview(viewC, at: 1)   // insert at specific index
parentView.insertSubview(viewC, aboveSubview: viewA)
parentView.insertSubview(viewC, belowSubview: viewA)
parentView.exchangeSubview(at: 0, withSubviewAt: 2)

// Removing
viewA.removeFromSuperview()  // removes from parent's subviews array

CALayer.zPosition

zPosition overrides the default subview-order-based z-ordering. It's a CGFloat that determines the layer's position on the z-axis.

// zPosition overrides subview order for DRAWING only
viewA.layer.zPosition = 1   // draws on top
viewB.layer.zPosition = 0   // draws behind viewA

// CRITICAL: zPosition affects DRAWING ORDER, not HIT TESTING
// Hit testing still uses the subviews array order!
// If viewB is later in subviews but has lower zPosition,
// viewB draws behind viewA but receives touches OVER viewA
Common Pitfall

zPosition and hit testing are disconnected. This is a frequent source of bugs. If you use zPosition to visually reorder views, the touch target order doesn't change. A view that appears behind another can still "steal" touches because it's later in the subviews array. Always reorder the subviews array (via bringSubviewToFront etc.) if you need both visual and touch ordering to match.

Layer Tree vs View Tree

// The view tree and layer tree are parallel but not identical

// Every UIView has a backing layer
// view.subviews[i].layer == view.layer.sublayers[j] (for some j)

// But you can add layers without views
let shapeLayer = CAShapeLayer()
shapeLayer.path = UIBezierPath(ovalIn: bounds).cgPath
view.layer.addSublayer(shapeLayer)
// shapeLayer has no UIView — no touch events, no Auto Layout

// Layer sublayers include both view-backing layers AND standalone layers
// view.layer.sublayers?.count >= view.subviews.count

8. Performance Implications

View Count Impact

View Count Impact Mitigation
< 100 Negligible None needed
100–500 Layout passes slow down Flatten hierarchy, use layers
500–1000 Noticeable frame drops Cell reuse, lazy loading
> 1000 Severe performance issues Virtualization (UICollectionView), custom drawing

Flattening the Hierarchy

// Instead of deeply nested views for a complex cell:
// BAD: 20+ views per cell × 50 visible cells = 1000+ views

// GOOD: Use CALayer sublayers for non-interactive elements
class OptimizedCell: UICollectionViewCell {
    private let titleLayer = CATextLayer()
    private let avatarLayer = CALayer()
    private let separatorLayer = CALayer()

    override init(frame: CGRect) {
        super.init(frame: frame)
        // Layers are cheaper than views — no event handling overhead
        contentView.layer.addSublayer(titleLayer)
        contentView.layer.addSublayer(avatarLayer)
        contentView.layer.addSublayer(separatorLayer)
    }

    override func layoutSubviews() {
        super.layoutSubviews()
        // Manual frame calculation — faster than Auto Layout for cells
        CATransaction.begin()
        CATransaction.setDisableActions(true)
        avatarLayer.frame = CGRect(x: 16, y: 12, width: 40, height: 40)
        titleLayer.frame = CGRect(x: 68, y: 16,
            width: bounds.width - 84, height: 20)
        separatorLayer.frame = CGRect(x: 68, y: bounds.height - 0.5,
            width: bounds.width - 68, height: 0.5)
        CATransaction.commit()
    }

    required init?(coder: NSCoder) { fatalError() }
}

// BEST: For truly performance-critical cells, draw everything in draw(_:)
// Single bitmap, zero subviews/sublayers
class DrawnCell: UICollectionViewCell {
    override func draw(_ rect: CGRect) {
        // Draw avatar, title, subtitle, separator all in one pass
        // Most efficient but hardest to maintain
    }
}
Architect Must-Know

Understanding the UIView/CALayer split is the key to performance at scale. In a large app with complex UIs:

  • Use views for interactive elements that need touch handling or Auto Layout
  • Use layers for decorative elements (separators, badges, shadows) that don't need events
  • Use custom drawing (draw(_:)) for the most performance-critical cells where even layers are too many
  • Profile with Instruments — the "View Debugging" tool shows the full view/layer hierarchy and highlights excessive nesting

The trade-off is always: maintainability vs performance. Auto Layout with views is the most maintainable. Manual layer layout is faster. Custom drawing is fastest but hardest to maintain. Choose based on your actual performance requirements.

Best Practice

Use Xcode's View Hierarchy Debugger (Debug → View Debugging → Capture View Hierarchy) regularly. It shows the 3D exploded view of your hierarchy, highlights clipped views, ambiguous layouts, and views that are off-screen but still in the hierarchy (wasting memory and layout time).
