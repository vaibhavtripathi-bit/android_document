1. The Full Touch Pipeline

When a user touches the screen, the event travels through multiple system layers before reaching your app's view. Understanding this full pipeline is a classic Staff/L6 interview question.

Step-by-Step Code Flow

// Step 4: UIApplication receives the event
// This is the entry point into your app's event handling
class UIApplication {
    open func sendEvent(_ event: UIEvent) {
        // Dispatches to the appropriate window
        for window in windows {
            if window.isEventRelevant(event) {
                window.sendEvent(event)
            }
        }
    }
}

// Step 5-6: UIWindow dispatches and performs hit testing
class UIWindow {
    open func sendEvent(_ event: UIEvent) {
        for touch in event.allTouches ?? [] {
            switch touch.phase {
            case .began:
                // Hit test to find the target view
                let hitView = hitTest(touch.location(in: self), with: event)
                touch._setView(hitView)
                // Deliver to gesture recognizers first, then view
                deliverToGestureRecognizers(touch, event)
                hitView?.touchesBegan([touch], with: event)

            case .moved:
                // Deliver to the SAME view that received .began
                touch.view?.touchesMoved([touch], with: event)

            case .ended:
                touch.view?.touchesEnded([touch], with: event)

            case .cancelled:
                touch.view?.touchesCancelled([touch], with: event)

            default: break
            }
        }
    }
}
Deep Dive

Why backboardd, not SpringBoard? Before iOS 6, SpringBoard handled all HID events. This was problematic because SpringBoard is a full app with its own UI — if it was busy (e.g., animating the app switcher), touch delivery to apps would be delayed. Apple moved HID handling to backboardd, a lightweight daemon that does nothing but route events. This reduced touch latency significantly.

2. UITouch Lifecycle

A UITouch object represents a single finger on the screen. It is reused across phases — the same object transitions through began → moved → ended.

UITouch Phase Lifecycle: ┌──────────┐ ┌──────────┐ ┌──────────┐ │ .began │────▶│ .moved │────▶│ .ended │ └──────────┘ └────┬─────┘ └──────────┘ │ │ ▲ │ └──────────────────┘ │ (multiple .moved events) │ │ ┌──────────────┐ └──────────▶│ .cancelled │ └──────────────┘ (system interruption: phone call, gesture recognizer claim, etc.) .stationary — finger is touching but hasn't moved .regionEntered / .regionMoved / .regionExited — indirect touches (Apple TV) 
class TouchTrackingView: UIView {

    override func touchesBegan(_ touches: Set<UITouch>, with event: UIEvent?) {
        for touch in touches {
            // UITouch properties available at each phase:
            let location = touch.location(in: self)
            let previousLocation = touch.previousLocation(in: self)
            let timestamp = touch.timestamp          // TimeInterval since boot
            let phase = touch.phase                   // .began
            let tapCount = touch.tapCount              // 1 for single, 2 for double
            let type = touch.type                     // .direct, .indirect, .pencil
            let force = touch.force                   // 3D Touch / Force Touch
            let maxForce = touch.maximumPossibleForce  // device capability
            let radius = touch.majorRadius             // touch size estimate
            let tolerance = touch.majorRadiusTolerance

            // The view that was hit-tested (doesn't change across phases)
            let hitView = touch.view

            // The window that received the touch
            let window = touch.window

            // Estimated properties (Apple Pencil)
            let azimuth = touch.azimuthAngle(in: self)
            let altitude = touch.altitudeAngle
        }
    }

    override func touchesMoved(_ touches: Set<UITouch>, with event: UIEvent?) {
        for touch in touches {
            let location = touch.location(in: self)
            let previous = touch.previousLocation(in: self)
            let delta = CGPoint(
                x: location.x - previous.x,
                y: location.y - previous.y
            )
            // Use delta for dragging, drawing, etc.
        }
    }

    override func touchesEnded(_ touches: Set<UITouch>, with event: UIEvent?) {
        // Clean up tracking state
    }

    override func touchesCancelled(_ touches: Set<UITouch>, with event: UIEvent?) {
        // MUST handle this — system can cancel at any time
        // Treat as touchesEnded but revert any in-progress changes
    }
}
Common Pitfall

Always implement touchesCancelled. The system cancels touches when: a gesture recognizer claims the touch, a phone call comes in, the app is backgrounded, a system alert appears, or Control Center/Notification Center is pulled. If you don't handle cancellation, your view can be stuck in a "pressed" state.

Coalesced and Predicted Touches

// On ProMotion displays (120Hz) and with Apple Pencil (240Hz),
// touches arrive faster than the 60Hz display refresh.
// UIKit coalesces them but provides access to the full set.

override func touchesMoved(_ touches: Set<UITouch>, with event: UIEvent?) {
    guard let touch = touches.first else { return }

    // Coalesced touches: all intermediate positions since last callback
    if let coalesced = event?.coalescedTouches(for: touch) {
        for coalescedTouch in coalesced {
            let point = coalescedTouch.location(in: self)
            // Use for smooth drawing — don't skip intermediate points
            drawingPath.addLine(to: point)
        }
    }

    // Predicted touches: where the system thinks the finger is heading
    if let predicted = event?.predictedTouches(for: touch) {
        for predictedTouch in predicted {
            let point = predictedTouch.location(in: self)
            // Use for low-latency drawing — render ahead of the finger
            // Must be replaced with actual touches on next callback
            predictedPath.addLine(to: point)
        }
    }
}

4. The Responder Chain

The responder chain is the path an event follows if the initial receiver doesn't handle it. Every UIResponder has a next property pointing to the next responder in the chain.

Responder Chain (typical): ┌─────────────────────┐ │ Hit-Tested UIView │ ← First responder for touch events └──────────┬──────────┘ │ .next ┌──────────▼──────────┐ │ Parent UIView │ ← superview └──────────┬──────────┘ │ .next ┌──────────▼──────────┐ │ UIViewController │ ← view.next is the view controller └──────────┬──────────┘ │ .next ┌──────────▼──────────┐ │ Parent UIView │ ← VC's view's superview │ (container VC view) │ └──────────┬──────────┘ │ .next ┌──────────▼──────────┐ │ Parent VC │ ← container view controller └──────────┬──────────┘ │ .next ┌──────────▼──────────┐ │ UIWindow │ └──────────┬──────────┘ │ .next ┌──────────▼──────────┐ │ UIWindowScene │ ← iOS 13+ └──────────┬──────────┘ │ .next ┌──────────▼──────────┐ │ UIApplication │ └──────────┬──────────┘ │ .next ┌──────────▼──────────┐ │ UIApplicationDelegate│ ← if it's a UIResponder (it is) └──────────┬──────────┘ │ .next ▼ nil ← event is dropped 
// The responder chain is built from the `next` property
class UIResponder {
    open var next: UIResponder? { get }

    // Touch events
    open func touchesBegan(_ touches: Set<UITouch>, with event: UIEvent?)
    open func touchesMoved(_ touches: Set<UITouch>, with event: UIEvent?)
    open func touchesEnded(_ touches: Set<UITouch>, with event: UIEvent?)
    open func touchesCancelled(_ touches: Set<UITouch>, with event: UIEvent?)

    // Motion events
    open func motionBegan(_ motion: UIEvent.EventSubtype, with event: UIEvent?)
    open func motionEnded(_ motion: UIEvent.EventSubtype, with event: UIEvent?)

    // Press events (physical buttons)
    open func pressesBegan(_ presses: Set<UIPress>, with event: UIPressesEvent?)
    open func pressesEnded(_ presses: Set<UIPress>, with event: UIPressesEvent?)

    // First responder management
    open var canBecomeFirstResponder: Bool { get }
    open func becomeFirstResponder() -> Bool
    open func resignFirstResponder() -> Bool
}

// The `next` property chain for UIView:
// UIView.next → superview (if no VC manages this view)
// UIView.next → UIViewController (if VC.view == this view)
// UIViewController.next → parent VC's view (or window)

Event Forwarding

// Default implementation of touchesBegan forwards up the chain
class UIResponder {
    open func touchesBegan(_ touches: Set<UITouch>, with event: UIEvent?) {
        // Default: forward to next responder
        next?.touchesBegan(touches, with: event)
    }
}

// To HANDLE an event (stop propagation), override WITHOUT calling super
class MyView: UIView {
    override func touchesBegan(_ touches: Set<UITouch>, with event: UIEvent?) {
        // Handle the touch — don't call super to stop propagation
        handleTouch(touches)
    }
}

// To OBSERVE but not consume, override AND call super
class ObservingView: UIView {
    override func touchesBegan(_ touches: Set<UITouch>, with event: UIEvent?) {
        logTouch(touches)
        super.touchesBegan(touches, with: event)  // Continue propagation
    }
}

5. Gesture Recognizer State Machine

Gesture recognizers are state machines that run in parallel with the responder chain. They intercept touches before the responder chain and can claim exclusive ownership.

UIGestureRecognizer State Machine: DISCRETE gestures (tap, swipe): .possible ──────▶ .recognized ──────▶ (reset to .possible) │ └──────────▶ .failed ───────────▶ (reset to .possible) CONTINUOUS gestures (pan, pinch, rotation): .possible ──▶ .began ──▶ .changed ──▶ .changed ──▶ .ended │ │ │ │ │ │ └──▶ .cancelled │ │ └──────────▶ .cancelled │ │ │ └──▶ .failed │ ▼ (reset to .possible) State transitions trigger action method calls: • .began → action called • .changed → action called (repeatedly) • .ended → action called • .cancelled → action called • .failed → action NOT called • .possible → action NOT called 
// Gesture recognizer action handling
class GestureViewController: UIViewController {

    override func viewDidLoad() {
        super.viewDidLoad()

        let pan = UIPanGestureRecognizer(target: self, action: #selector(handlePan))
        view.addGestureRecognizer(pan)
    }

    @objc func handlePan(_ gesture: UIPanGestureRecognizer) {
        switch gesture.state {
        case .began:
            // Gesture just recognized — start tracking
            let location = gesture.location(in: view)
            startDragging(at: location)

        case .changed:
            // Finger moved — update position
            let translation = gesture.translation(in: view)
            updateDrag(by: translation)
            gesture.setTranslation(.zero, in: view)  // Reset for delta

        case .ended:
            // Finger lifted — apply velocity for momentum
            let velocity = gesture.velocity(in: view)
            finishDragging(withVelocity: velocity)

        case .cancelled:
            // System cancelled — revert to original position
            cancelDragging()

        case .failed, .possible:
            break

        @unknown default:
            break
        }
    }
}

6. UIGestureRecognizer Internals

Touch Delivery with Gesture Recognizers

When gesture recognizers are present, the touch delivery becomes more complex. By default, both the gesture recognizer AND the view receive touches simultaneously until the GR makes a decision.

// The default behavior:
// 1. Touch begins → both GR and view receive touchesBegan
// 2. Touch moves → both GR and view receive touchesMoved
// 3. GR recognizes → view receives touchesCancelled
//    OR GR fails → view continues receiving touches normally

// Key properties that control this behavior:

// cancelsTouchesInView (default: true)
// When GR recognizes, the view gets touchesCancelled
gesture.cancelsTouchesInView = true

// delaysTouchesBegan (default: false)
// If true, view doesn't get touchesBegan until GR fails
gesture.delaysTouchesBegan = false

// delaysTouchesEnded (default: true)
// If true, view doesn't get touchesEnded until GR fails
gesture.delaysTouchesEnded = true

require(toFail:) — Dependency Between Recognizers

// Single tap should only fire if double tap fails
let singleTap = UITapGestureRecognizer(target: self, action: #selector(handleSingleTap))
singleTap.numberOfTapsRequired = 1

let doubleTap = UITapGestureRecognizer(target: self, action: #selector(handleDoubleTap))
doubleTap.numberOfTapsRequired = 2

// singleTap waits for doubleTap to fail before recognizing
singleTap.require(toFail: doubleTap)

// This introduces a ~300ms delay on single taps (waiting for potential second tap)
// Trade-off: correctness vs responsiveness

Simultaneous Recognition

// By default, only one gesture recognizer can be active at a time.
// Use the delegate to allow simultaneous recognition.

class MapViewController: UIViewController, UIGestureRecognizerDelegate {

    override func viewDidLoad() {
        super.viewDidLoad()

        let pan = UIPanGestureRecognizer(target: self, action: #selector(handlePan))
        let pinch = UIPinchGestureRecognizer(target: self, action: #selector(handlePinch))
        let rotation = UIRotationGestureRecognizer(target: self, action: #selector(handleRotation))

        pan.delegate = self
        pinch.delegate = self
        rotation.delegate = self

        view.addGestureRecognizer(pan)
        view.addGestureRecognizer(pinch)
        view.addGestureRecognizer(rotation)
    }

    // Allow all three gestures simultaneously (like Maps app)
    func gestureRecognizer(
        _ gestureRecognizer: UIGestureRecognizer,
        shouldRecognizeSimultaneouslyWith other: UIGestureRecognizer
    ) -> Bool {
        return true
    }

    // Other delegate methods for fine-grained control:

    func gestureRecognizerShouldBegin(
        _ gestureRecognizer: UIGestureRecognizer
    ) -> Bool {
        // Return false to prevent recognition (e.g., based on direction)
        if let pan = gestureRecognizer as? UIPanGestureRecognizer {
            let velocity = pan.velocity(in: view)
            // Only allow horizontal pans
            return abs(velocity.x) > abs(velocity.y)
        }
        return true
    }

    func gestureRecognizer(
        _ gestureRecognizer: UIGestureRecognizer,
        shouldReceive touch: UITouch
    ) -> Bool {
        // Prevent GR from receiving touches on certain views
        return !(touch.view is UIButton)
    }

    func gestureRecognizer(
        _ gestureRecognizer: UIGestureRecognizer,
        shouldBeRequiredToFailBy other: UIGestureRecognizer
    ) -> Bool {
        // Dynamic version of require(toFail:)
        return false
    }
}

Custom Gesture Recognizer Subclass

import UIKit.UIGestureRecognizerSubclass

class DirectionLockPanGesture: UIPanGestureRecognizer {

    enum Direction {
        case horizontal, vertical, undetermined
    }

    private(set) var direction: Direction = .undetermined
    private var initialTouchPoint: CGPoint = .zero
    private let lockThreshold: CGFloat = 10

    override func touchesBegan(_ touches: Set<UITouch>, with event: UIEvent) {
        super.touchesBegan(touches, with: event)
        direction = .undetermined
        initialTouchPoint = touches.first!.location(in: view)
    }

    override func touchesMoved(_ touches: Set<UITouch>, with event: UIEvent) {
        super.touchesMoved(touches, with: event)

        if direction == .undetermined {
            let currentPoint = touches.first!.location(in: view)
            let dx = abs(currentPoint.x - initialTouchPoint.x)
            let dy = abs(currentPoint.y - initialTouchPoint.y)

            if dx > lockThreshold || dy > lockThreshold {
                direction = dx > dy ? .horizontal : .vertical
            }
        }
    }

    override func reset() {
        super.reset()
        direction = .undetermined
    }
}

7. UIScrollView Delayed Touch Delivery

UIScrollView introduces a 150ms delay before delivering touches to its subviews. This is necessary to determine whether the user intends to scroll or tap a subview.

UIScrollView Touch Delivery Timeline: Touch begins ─────┬────────────────────────────────────────────────▶ time │ │ 0ms: UIScrollView intercepts touch │ (delaysContentTouches = true by default) │ │ 0-150ms: Waiting period │ • If finger moves > threshold → SCROLL │ • If finger stays still → deliver to subview │ │ 150ms: Timer fires │ • Touch delivered to content subview │ • touchesBegan sent to the hit-tested view │ │ If user starts scrolling AFTER touch delivered: │ • touchesCancelled sent to subview │ • canCancelContentTouches controls this behavior 
// UIScrollView touch delivery properties
let scrollView = UIScrollView()

// delaysContentTouches (default: true)
// If true, scroll view waits ~150ms before delivering touches to subviews
// Set to false for immediate touch response (e.g., drawing apps)
scrollView.delaysContentTouches = true

// canCancelContentTouches (default: true)
// If true, scroll view can cancel an already-delivered touch when scrolling starts
// Set to false if subviews need uninterrupted touch tracking
scrollView.canCancelContentTouches = true

// Override to selectively delay touches
class SelectiveScrollView: UIScrollView {
    override func touchesShouldCancel(in view: UIView) -> Bool {
        // Don't cancel touches on buttons — let them complete
        if view is UIButton { return false }
        return true
    }

    override func touchesShouldBegin(
        _ touches: Set<UITouch>,
        with event: UIEvent?,
        in view: UIView
    ) -> Bool {
        // Return false to prevent touch delivery to specific subviews
        return true
    }
}
Common Pitfall

The 150ms delay causes "sluggish" buttons inside scroll views. Users notice the delay when tapping buttons in a UITableView or UICollectionView. The cell highlights with a delay. Solutions: (1) Set delaysContentTouches = false on the scroll view (but this can break scrolling from button areas), or (2) Override touchesShouldCancel(in:) to return true for UIControl subclasses, which allows the scroll view to cancel the touch if the user starts scrolling.

8. Multi-Touch Handling

// Multi-touch is disabled by default on UIView
view.isMultipleTouchEnabled = true  // Must enable explicitly

// With multi-touch enabled, touchesBegan can receive multiple touches
class MultiTouchView: UIView {
    private var activeTouches: [UITouch: CGPoint] = [:]

    override func touchesBegan(_ touches: Set<UITouch>, with event: UIEvent?) {
        for touch in touches {
            activeTouches[touch] = touch.location(in: self)
        }
        // activeTouches.count = number of fingers currently touching
    }

    override func touchesMoved(_ touches: Set<UITouch>, with event: UIEvent?) {
        for touch in touches {
            activeTouches[touch] = touch.location(in: self)
        }
    }

    override func touchesEnded(_ touches: Set<UITouch>, with event: UIEvent?) {
        for touch in touches {
            activeTouches.removeValue(forKey: touch)
        }
    }

    override func touchesCancelled(_ touches: Set<UITouch>, with event: UIEvent?) {
        for touch in touches {
            activeTouches.removeValue(forKey: touch)
        }
    }
}

9. Architect: Custom Gesture Systems

Architect Must-Know

Building custom gesture systems for complex multi-touch scenarios:

At Staff level, you may need to design gesture systems that go beyond standard recognizers. Examples include:

  • Drawing apps — Palm rejection, pressure-sensitive strokes, simultaneous drawing + zoom
  • Map-like interfaces — Simultaneous pan + pinch + rotation with momentum
  • Custom transitions — Gesture-driven interactive transitions that can be interrupted and reversed
  • Collaborative canvases — Multiple independent touch streams for different tools

Key architectural decisions:

  1. Gesture recognizer vs raw touches — Use GRs when possible (they handle edge cases). Use raw touches only when GRs can't express your gesture.
  2. Failure requirements graph — Complex apps can create dependency chains. Keep the graph shallow (max 2 levels) to avoid latency.
  3. Simultaneous recognition policy — Centralize this in a single delegate object rather than scattering across view controllers.
  4. Touch routing — For complex canvases, consider a custom hitTest that routes touches to different subsystems based on tool selection.
// Example: Centralized gesture coordination for a complex app
class GestureCoordinator: NSObject, UIGestureRecognizerDelegate {

    enum GesturePolicy {
        case exclusive
        case simultaneousWithSameType
        case simultaneousWithAll
    }

    private var policies: [String: GesturePolicy] = [:]

    func register(
        _ recognizer: UIGestureRecognizer,
        policy: GesturePolicy,
        identifier: String
    ) {
        recognizer.delegate = self
        policies[identifier] = policy
        recognizer.name = identifier
    }

    func gestureRecognizer(
        _ gestureRecognizer: UIGestureRecognizer,
        shouldRecognizeSimultaneouslyWith other: UIGestureRecognizer
    ) -> Bool {
        guard let name = gestureRecognizer.name,
              let policy = policies[name] else {
            return false
        }

        switch policy {
        case .exclusive:
            return false
        case .simultaneousWithSameType:
            return type(of: gestureRecognizer) == type(of: other)
        case .simultaneousWithAll:
            return true
        }
    }
}
Best Practice

Interview tip: When asked "Describe the full touch event pipeline," start from the hardware (digitizer → IOKit → backboardd) and work your way through the app process (RunLoop → UIApplication → UIWindow → hit test → gesture recognizers → responder chain). Mention the key decision points: hit testing determines WHERE, gesture recognizers determine WHAT, and the responder chain determines WHO handles unrecognized events. This demonstrates systems-level thinking that L6 interviewers look for.
