1. UITableView / UICollectionView Internals

Both UITableView and UICollectionView are subclasses of UIScrollView. They implement a virtualization pattern: only cells visible on screen (plus a small buffer) exist in memory. As the user scrolls, cells leaving the screen are recycled and reused for new content.

Cell Virtualization: Visible Area (screen) ┌─────────────────────────┐ │ Cell A (index 5) │ ← Visible, in view hierarchy │ Cell B (index 6) │ │ Cell C (index 7) │ │ Cell D (index 8) │ │ Cell E (index 9) │ └─────────────────────────┘ ▲ Buffer Zone │ (prefetch area) ┌─────────────────────────┐ │ Cell F (index 10) │ ← Prepared but not yet visible │ Cell G (index 11) │ └─────────────────────────┘ Reuse Queue (off-screen pool) ┌─────────────────────────┐ │ Cell X (was index 2) │ ← Removed from hierarchy │ Cell Y (was index 3) │ Available for dequeue │ Cell Z (was index 4) │ └─────────────────────────┘ Total data: 10,000 items Cells in memory: ~7-10 (visible + buffer) Memory savings: ~99.9%

Architecture Comparison

Feature UITableView UICollectionView
Layout Single column, rows only Arbitrary layout via UICollectionViewLayout
Sections Headers/footers per section Supplementary views (any kind)
Decoration None Decoration views (backgrounds, etc.)
Cell sizing rowHeight or automaticDimension Layout-driven (estimatedItemSize)
Editing Built-in swipe actions, reorder Manual (or use UICollectionViewListCell)
Modern recommendation Use for simple lists Use for everything (with CompositionalLayout)

2. Cell Reuse Queue Mechanism

// The reuse queue is a dictionary of [String: [UICollectionViewCell]]
// Key = reuse identifier, Value = array of available cells

// Registration (tells the collection view how to create cells)
collectionView.register(
    MyCell.self,
    forCellWithReuseIdentifier: "myCell"
)

// Dequeue (gets a cell from the reuse queue or creates a new one)
let cell = collectionView.dequeueReusableCell(
    withReuseIdentifier: "myCell",
    for: indexPath
) as! MyCell

// What happens internally during dequeue:
// 1. Check reuse queue for a cell with matching identifier
// 2. If found: remove from queue, call prepareForReuse()
// 3. If not found: instantiate a new cell (init or nib)
// 4. Return the cell

// What happens when a cell scrolls off screen:
// 1. Cell is removed from the collection view's visible cells
// 2. Cell is added to the reuse queue
// 3. Cell remains in memory but is NOT in the view hierarchy

class MyCell: UICollectionViewCell {
    override func prepareForReuse() {
        super.prepareForReuse()
        // Reset cell to default state
        // Cancel any pending image loads
        imageView.image = nil
        imageLoadTask?.cancel()
        titleLabel.text = nil
        // DON'T reset constraints here — they should be set once in init
    }
}
Common Pitfall

Forgetting to reset state in prepareForReuse(). Since cells are recycled, any state from the previous use persists. If you set an image asynchronously and don't clear it in prepareForReuse(), a recycled cell will briefly show the wrong image before the new one loads. Always cancel pending async work and reset visual state.

3. UICollectionViewCompositionalLayout

Introduced in iOS 13, UICollectionViewCompositionalLayout replaces UICollectionViewFlowLayout for most use cases. It uses a hierarchical structure: Item → Group → Section → Layout.

Compositional Layout Hierarchy: Layout └── Section 0 │ └── Group (horizontal) │ ├── Item (fractionalWidth: 0.5) │ └── Item (fractionalWidth: 0.5) └── Section 1 │ └── Group (horizontal) │ ├── Item (fractionalWidth: 0.3) │ ├── Item (fractionalWidth: 0.3) │ └── Item (fractionalWidth: 0.3) └── Section 2 └── Group (horizontal, orthogonal scrolling) ├── Item (absolute: 200pt) ├── Item (absolute: 200pt) └── Item (absolute: 200pt) 
// Basic grid layout
func createGridLayout() -> UICollectionViewCompositionalLayout {
    // Item: each cell takes half the group width
    let itemSize = NSCollectionLayoutSize(
        widthDimension: .fractionalWidth(0.5),
        heightDimension: .fractionalHeight(1.0)
    )
    let item = NSCollectionLayoutItem(layoutSize: itemSize)
    item.contentInsets = NSDirectionalEdgeInsets(
        top: 4, leading: 4, bottom: 4, trailing: 4
    )

    // Group: horizontal, 2 items per row, estimated height
    let groupSize = NSCollectionLayoutSize(
        widthDimension: .fractionalWidth(1.0),
        heightDimension: .estimated(200)
    )
    let group = NSCollectionLayoutGroup.horizontal(
        layoutSize: groupSize,
        subitems: [item]
    )

    // Section
    let section = NSCollectionLayoutSection(group: group)
    section.contentInsets = NSDirectionalEdgeInsets(
        top: 16, leading: 16, bottom: 16, trailing: 16
    )

    // Section header
    let headerSize = NSCollectionLayoutSize(
        widthDimension: .fractionalWidth(1.0),
        heightDimension: .estimated(44)
    )
    let header = NSCollectionLayoutBoundarySupplementaryItem(
        layoutSize: headerSize,
        elementKind: UICollectionView.elementKindSectionHeader,
        alignment: .top
    )
    header.pinToVisibleBounds = true  // Sticky header
    section.boundarySupplementaryItems = [header]

    return UICollectionViewCompositionalLayout(section: section)
}

4. DiffableDataSource & Snapshots

// DiffableDataSource replaces UICollectionViewDataSource
// It uses snapshots for declarative data management

enum Section: Hashable {
    case featured, recent, categories
}

struct Item: Hashable {
    let id: UUID
    let title: String
    let imageURL: URL?

    func hash(into hasher: inout Hasher) {
        hasher.combine(id)  // Hash only by identity, not content
    }

    static func == (lhs: Item, rhs: Item) -> Bool {
        lhs.id == rhs.id  // Equality by identity
    }
}

// Create data source
var dataSource: UICollectionViewDiffableDataSource<Section, Item>!

dataSource = UICollectionViewDiffableDataSource(
    collectionView: collectionView
) { collectionView, indexPath, item in
    let cell = collectionView.dequeueReusableCell(
        withReuseIdentifier: "cell",
        for: indexPath
    ) as! ItemCell
    cell.configure(with: item)
    return cell
}

// Apply a snapshot
var snapshot = NSDiffableDataSourceSnapshot<Section, Item>()
snapshot.appendSections([.featured, .recent, .categories])
snapshot.appendItems(featuredItems, toSection: .featured)
snapshot.appendItems(recentItems, toSection: .recent)
snapshot.appendItems(categoryItems, toSection: .categories)

// Apply with animation
dataSource.apply(snapshot, animatingDifferences: true)

// Update existing items (iOS 15+)
var snapshot = dataSource.snapshot()
snapshot.reconfigureItems([updatedItem])  // Reconfigure without reload
dataSource.apply(snapshot)

// Difference between reloadItems and reconfigureItems:
// reloadItems: dequeues a NEW cell and configures it (expensive)
// reconfigureItems: updates the EXISTING cell in place (cheap, iOS 15+)
Deep Dive

How diffing works internally: When you call apply(snapshot), the data source computes the difference between the current and new snapshots using a diffing algorithm (based on the Hashable conformance of your section and item types). It then generates the minimal set of insert, delete, move, and reload operations needed to transform the current state into the new state. This is why your items must be Hashable — the diff algorithm uses hash values to identify items across snapshots.

5. CellRegistration API

// CellRegistration (iOS 14+) — type-safe cell configuration
// Replaces register() + dequeueReusableCell() pattern

let cellRegistration = UICollectionView.CellRegistration<UICollectionViewListCell, Item> {
    cell, indexPath, item in

    var content = cell.defaultContentConfiguration()
    content.text = item.title
    content.secondaryText = item.subtitle
    content.image = UIImage(systemName: item.iconName)
    cell.contentConfiguration = content

    // Accessories
    cell.accessories = [
        .disclosureIndicator(),
        .label(text: item.badge)
    ]
}

// Use in data source
dataSource = UICollectionViewDiffableDataSource(
    collectionView: collectionView
) { collectionView, indexPath, item in
    collectionView.dequeueConfiguredReusableCell(
        using: cellRegistration,
        for: indexPath,
        item: item
    )
}

// Benefits over the old pattern:
// 1. Type-safe — no force casting
// 2. No string identifiers — no runtime crashes from typos
// 3. Registration and configuration in one place
// 4. Works with UIContentConfiguration for modern cell setup

6. Prefetching

// UICollectionViewDataSourcePrefetching
// Tells you which cells are about to become visible
// Use to start async data loading ahead of time

class FeedViewController: UICollectionViewController,
    UICollectionViewDataSourcePrefetching {

    private var prefetchTasks: [IndexPath: Task<Void, Never>] = [:]

    override func viewDidLoad() {
        super.viewDidLoad()
        collectionView.prefetchDataSource = self
    }

    func collectionView(
        _ collectionView: UICollectionView,
        prefetchItemsAt indexPaths: [IndexPath]
    ) {
        for indexPath in indexPaths {
            let item = items[indexPath.item]
            guard imageCache[item.imageURL] == nil else { continue }

            prefetchTasks[indexPath] = Task {
                let image = try? await imageLoader.load(item.imageURL)
                imageCache[item.imageURL] = image
            }
        }
    }

    func collectionView(
        _ collectionView: UICollectionView,
        cancelPrefetchingForItemsAt indexPaths: [IndexPath]
    ) {
        for indexPath in indexPaths {
            prefetchTasks[indexPath]?.cancel()
            prefetchTasks.removeValue(forKey: indexPath)
        }
    }
}

7. Self-Sizing Cells

// Self-sizing with Auto Layout (most common)
let layout = createLayout()
// Use .estimated() for dimensions that should self-size
let itemSize = NSCollectionLayoutSize(
    widthDimension: .fractionalWidth(1.0),
    heightDimension: .estimated(100)  // Self-sizing height
)

// The cell's contentView must have constraints that define its height
class SelfSizingCell: UICollectionViewCell {
    private let titleLabel = UILabel()
    private let bodyLabel = UILabel()

    override init(frame: CGRect) {
        super.init(frame: frame)

        titleLabel.numberOfLines = 0
        bodyLabel.numberOfLines = 0

        let stack = UIStackView(arrangedSubviews: [titleLabel, bodyLabel])
        stack.axis = .vertical
        stack.spacing = 8
        stack.translatesAutoresizingMaskIntoConstraints = false
        contentView.addSubview(stack)

        NSLayoutConstraint.activate([
            stack.topAnchor.constraint(equalTo: contentView.topAnchor, constant: 16),
            stack.leadingAnchor.constraint(equalTo: contentView.leadingAnchor, constant: 16),
            stack.trailingAnchor.constraint(equalTo: contentView.trailingAnchor, constant: -16),
            stack.bottomAnchor.constraint(equalTo: contentView.bottomAnchor, constant: -16)
        ])
    }

    required init?(coder: NSCoder) { fatalError() }
}

// Self-sizing with preferredLayoutAttributesFitting (manual)
class ManualSizingCell: UICollectionViewCell {
    override func preferredLayoutAttributesFitting(
        _ layoutAttributes: UICollectionViewLayoutAttributes
    ) -> UICollectionViewLayoutAttributes {
        let attributes = super.preferredLayoutAttributesFitting(layoutAttributes)

        let targetSize = CGSize(
            width: layoutAttributes.frame.width,
            height: 0
        )
        let size = contentView.systemLayoutSizeFitting(
            targetSize,
            withHorizontalFittingPriority: .required,
            verticalFittingPriority: .fittingSizeLevel
        )
        attributes.frame.size.height = ceil(size.height)
        return attributes
    }
}

8. Advanced Compositional Layout

// Section provider — different layout per section
let layout = UICollectionViewCompositionalLayout { sectionIndex, environment in
    switch sectionIndex {
    case 0:
        return createFeaturedSection()    // Full-width cards
    case 1:
        return createHorizontalSection()  // Horizontal scroll
    default:
        return createGridSection(environment: environment)
    }
}

// Orthogonal scrolling section (horizontal scroll within vertical list)
func createHorizontalSection() -> NSCollectionLayoutSection {
    let itemSize = NSCollectionLayoutSize(
        widthDimension: .fractionalWidth(1.0),
        heightDimension: .fractionalHeight(1.0)
    )
    let item = NSCollectionLayoutItem(layoutSize: itemSize)

    let groupSize = NSCollectionLayoutSize(
        widthDimension: .absolute(200),
        heightDimension: .absolute(250)
    )
    let group = NSCollectionLayoutGroup.horizontal(
        layoutSize: groupSize,
        subitems: [item]
    )

    let section = NSCollectionLayoutSection(group: group)
    section.interGroupSpacing = 12
    section.contentInsets = NSDirectionalEdgeInsets(
        top: 0, leading: 16, bottom: 0, trailing: 16
    )

    // Enable horizontal scrolling for this section
    section.orthogonalScrollingBehavior = .continuousGroupLeadingBoundary

    // Orthogonal scrolling options:
    // .none                           — no orthogonal scroll
    // .continuous                     — free scrolling
    // .continuousGroupLeadingBoundary — snaps to group leading edge
    // .paging                         — page-based scrolling
    // .groupPaging                    — snaps to group boundaries
    // .groupPagingCentered            — centers each group

    return section
}

// Decoration views (section backgrounds)
func createSectionWithBackground() -> NSCollectionLayoutSection {
    // ... item and group setup ...
    let section = NSCollectionLayoutSection(group: group)

    let background = NSCollectionLayoutDecorationItem.background(
        elementKind: "sectionBackground"
    )
    background.contentInsets = NSDirectionalEdgeInsets(
        top: 8, leading: 8, bottom: 8, trailing: 8
    )
    section.decorationItems = [background]

    return section
}

// Register decoration view
layout.register(
    SectionBackgroundView.self,
    forDecorationViewOfKind: "sectionBackground"
)

10. Architect: Generic List Infrastructure

Architect Must-Know

Building a generic, performant list infrastructure for 100+ screen types:

At scale, you need a reusable collection view infrastructure that handles heterogeneous content, multiple layout types, and consistent performance. Key architectural decisions:

  • Protocol-based cell configuration — Define a protocol that all cell types conform to. Use UIContentConfiguration (iOS 14+) as the standard.
  • Section-based layout composition — Each section defines its own layout via NSCollectionLayoutSection. A section provider maps section types to layouts.
  • Diffable data source with type erasure — Use AnyHashable for item types to support heterogeneous content in a single data source.
  • Prefetching pipeline — Centralize image loading and data prefetching. Cancel on reuse.
  • Height caching — For self-sizing cells, cache computed heights keyed by content hash to avoid redundant layout passes.
// Generic section-based collection view controller
protocol SectionType: Hashable {
    var layoutSection: NSCollectionLayoutSection { get }
}

protocol CellConfigurable: Hashable {
    associatedtype CellType: UICollectionViewCell
    func configure(_ cell: CellType)
}

class GenericListViewController<S: SectionType>: UIViewController {
    private(set) lazy var collectionView: UICollectionView = {
        let layout = UICollectionViewCompositionalLayout {
            [weak self] sectionIndex, environment in
            guard let self,
                  let section = self.dataSource.sectionIdentifier(
                      for: sectionIndex
                  ) else { return nil }
            return section.layoutSection
        }
        return UICollectionView(frame: .zero, collectionViewLayout: layout)
    }()

    private(set) var dataSource: UICollectionViewDiffableDataSource<S, AnyHashable>!

    func applySnapshot(_ builder: (inout NSDiffableDataSourceSnapshot<S, AnyHashable>) -> Void) {
        var snapshot = NSDiffableDataSourceSnapshot<S, AnyHashable>()
        builder(&snapshot)
        dataSource.apply(snapshot, animatingDifferences: true)
    }
}
Best Practice

Modern UIKit collection view stack (iOS 16+): Use UICollectionViewCompositionalLayout for layout, UICollectionViewDiffableDataSource for data, CellRegistration for type-safe cell dequeue, UIContentConfiguration for cell content, and UICollectionViewDataSourcePrefetching for async data loading. This combination gives you the best performance, type safety, and maintainability.
