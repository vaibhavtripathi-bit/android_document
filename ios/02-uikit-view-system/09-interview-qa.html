View Hierarchy Internals (Questions 1–10)

1. Explain the UIView / CALayer duality. Why does every UIView have a backing CALayer, and what responsibilities does each own? L6

UIView is the responder-chain participant — it handles touch events, accessibility, Auto Layout constraints, and coordinate-space conversions. CALayer is the visual node — it owns the bitmap backing store, geometry (bounds, position, anchorPoint, transform), and compositing properties (opacity, cornerRadius, shadow).

Every UIView creates a CALayer in init and sets itself as the layer's delegate. The view forwards visual property setters (frame, backgroundColor, alpha) to the layer. This separation exists because:

  1. macOS heritage: On macOS, NSView can be layer-backed or non-layer-backed. The layer is a separate rendering concept. iOS always uses layers.
  2. Render-server isolation: CALayer's property tree is serialized and sent to the out-of-process render server (backboardd). UIView state (responder chain, constraints) stays in-process. This separation keeps the render server lean.
  3. Reuse: You can use CALayer, CAShapeLayer, CAGradientLayer, etc. without UIView overhead when you need pure visual elements with no event handling.
Architect Insight

You can replace a view's default layer class by overriding class var layerClass: AnyClass. This is how AVPlayerLayer, CAMetalLayer, and CAEAGLLayer integrate — the view returns the specialized layer class, avoiding an extra layer in the hierarchy.

2. Walk through the hitTest(_:with:) algorithm. Why does it use reverse pre-order traversal? L6

hitTest(_:with:) finds the deepest (front-most) view that contains the touch point. The algorithm:

  1. Check self.point(inside:with:). If false, return nil — prune the entire subtree.
  2. Check guards: isHidden, !isUserInteractionEnabled, alpha < 0.01 → return nil.
  3. Iterate subviews in reverse order (last subview first = front-most visually). For each subview, convert the point to the subview's coordinate space and recursively call hitTest.
  4. Return the first non-nil result. If no subview claims the hit, return self.

Reverse pre-order is used because subviews added later are drawn on top (higher z-order). The user taps what they see, so the front-most view must win. Iterating in reverse ensures the visually topmost subview is tested first, and the algorithm short-circuits on the first match.

override func hitTest(_ point: CGPoint, with event: UIEvent?) -> UIView? {
    guard isUserInteractionEnabled, !isHidden, alpha >= 0.01 else { return nil }
    guard self.point(inside: point, with: event) else { return nil }
    for subview in subviews.reversed() {
        let converted = subview.convert(point, from: self)
        if let hit = subview.hitTest(converted, with: event) {
            return hit
        }
    }
    return self
}
3. How do you make a UIView respond to touches outside its bounds? Senior

Override point(inside:with:) in the parent view (not the child) to return true for the expanded region. Alternatively, override hitTest(_:with:) in the parent to forward the point to the child even when it falls outside the parent's bounds.

A common pattern for expanding tap targets:

override func point(inside point: CGPoint, with event: UIEvent?) -> Bool {
    let expandedBounds = bounds.insetBy(dx: -20, dy: -20)
    return expandedBounds.contains(point)
}

The key insight: hitTest on the parent calls point(inside:) on itself first. If the parent's bounds don't include the child's overflow area, the parent returns nil before ever reaching the child. So the fix must be at the parent level.

4. What is the difference between frame, bounds, and center? How does the layer's anchorPoint affect them? Senior

bounds: The view's rectangle in its own coordinate space. Origin is typically (0,0) but changes when you scroll a UIScrollView (scrolling = changing bounds.origin).

center: The view's center point in its superview's coordinate space. Maps to the layer's position.

frame: The smallest axis-aligned rectangle enclosing the view in its superview's coordinate space. It is derived from bounds, center, and transform. When a transform is applied (e.g., rotation), the frame grows to enclose the rotated rectangle — it becomes unreliable for layout.

anchorPoint: Defines which point of the layer corresponds to position/center. Default is (0.5, 0.5) — the geometric center. Changing anchorPoint to (0, 0) makes the top-left corner the anchor. This shifts the layer visually without changing position, so the frame changes. Transforms rotate/scale around the anchorPoint.

Common Pitfall

Changing anchorPoint without adjusting position causes the view to jump. To change anchorPoint without visual movement: save the old frame, set the new anchorPoint, then restore the frame.

5. What is the difference between clipsToBounds (UIView) and masksToBounds (CALayer)? Senior

They are the same property. UIView.clipsToBounds is a wrapper that reads/writes layer.masksToBounds. Setting one updates the other. The naming difference reflects the UIKit vs Core Animation naming conventions.

When true, sublayers/subviews that extend beyond the parent's bounds are clipped. This has a performance implication: combined with cornerRadius, it triggers off-screen rendering because the GPU must render the subtree to an off-screen buffer, apply the mask, then composite the result. For scrolling lists, this can cause frame drops.

6. How does UIView manage its layer's delegate? What happens if you set a layer's delegate to a different object? L6

UIView sets itself as its layer's delegate during initialization. The view implements CALayerDelegate methods including display(_:), draw(_:in:), layoutSublayers(of:), and action(for:forKey:). This is how UIView intercepts layout and drawing requests from the layer.

Apple explicitly states: never reassign a UIView-owned layer's delegate. Doing so breaks the view's ability to respond to layout changes, drawing requests, and implicit animation queries. The view would stop updating its visual appearance and layout.

For standalone layers (not backing a UIView), you can freely set delegates. But a UIView's backing layer's delegate must always be the view itself.

7. Explain the layer tree, presentation tree, and render tree. When would you access each? L6

Model layer tree: The "source of truth" — the property values you set in code. Accessed via layer.position, layer.opacity, etc. Reflects the final state immediately, even during animation.

Presentation tree: Reflects the in-flight animated values. Accessed via layer.presentation(). During a 0.3s opacity animation from 1.0 to 0.0, the model layer reads 0.0 immediately, but presentation()?.opacity returns the interpolated value at the current moment (e.g., 0.6).

Render tree: A private copy held by the render server (out-of-process). You cannot access it directly. It's the data the GPU actually uses to composite each frame.

When to use presentation(): Hit-testing during animation (to test against the visual position, not the model position), reading current animated values for physics-based interruptions, or synchronizing non-animated properties with animated ones.

8. What is the purpose of UIView.layerClass? Give three real-world examples. L6

Overriding class var layerClass: AnyClass tells UIKit to create a specific CALayer subclass as the view's backing layer instead of the default CALayer. This avoids adding an extra sublayer and gives the view direct access to the specialized layer's properties.

Examples:

  1. Video playback: Return AVPlayerLayer.self so the view directly displays video content. AVPlayerView can then expose player as a property that forwards to the layer.
  2. Metal rendering: Return CAMetalLayer.self for GPU-rendered content. The view's layer is the drawable surface — no extra sublayer needed.
  3. Shape drawing: Return CAShapeLayer.self for a view that draws vector paths. Avoids the overhead of draw(_:) and bitmap backing stores — the shape is rendered by the GPU directly.
  4. Gradient backgrounds: Return CAGradientLayer.self so the view automatically renders a gradient without custom drawing code.
9. How does the view hierarchy map to the layer hierarchy? Can they diverge? L6

Every addSubview(_:) call also adds the child view's layer as a sublayer of the parent view's layer. The view hierarchy and layer hierarchy are kept in sync by UIKit. However, they can diverge:

  • You can add standalone CALayer sublayers to a view's layer directly (view.layer.addSublayer(shapeLayer)). These layers have no corresponding UIView — they don't participate in the responder chain, Auto Layout, or accessibility.
  • CATransformLayer can create 3D layer hierarchies that have no UIView equivalent.
  • UIKit internally adds private sublayers (e.g., for scroll indicators, selection highlights).

The divergence means: layer sublayers ⊇ subview layers. The view hierarchy is always a subset of the layer hierarchy.

10. What is the cost of a deep view hierarchy? How would you flatten it for performance? Architect

Costs of deep hierarchies:

  • Hit testing: O(n) traversal of all views for every touch. Deep trees with many siblings are expensive.
  • Layout: Auto Layout constraint solving scales super-linearly. More views = more constraints = longer layout passes.
  • Rendering: Each layer is a texture in VRAM. More layers = more GPU compositing passes and more memory.
  • Memory: Each UIView is ~700 bytes, each CALayer ~300 bytes. 1000 views ≈ 1MB just for the objects, plus backing stores.

Flattening strategies:

  • Override draw(_:) to render multiple elements into a single backing store (trading flexibility for performance).
  • Use CAShapeLayer / CATextLayer instead of UIView subclasses for non-interactive elements.
  • Use shouldRasterize = true on complex subtrees that don't change frequently.
  • In table/collection views, use flat cell layouts — avoid nesting stack views inside stack views.

Touch Event Flow (Questions 11–20)

11. Trace the full path of a touch from the user's finger to your UIButton action. Include every system component. L6
  1. Hardware: The digitizer detects capacitance changes and generates raw touch data at 60–120 Hz.
  2. IOKit driver: The kernel's IOHIDEvent system packages the raw data into a Mach message.
  3. SpringBoard: Receives the IOHIDEvent via a Mach port. Determines which app owns the touched screen region and forwards the event to that app's process via another Mach port.
  4. App's main RunLoop: The event arrives as a Source1 (port-based) callback. UIKit's internal event queue receives it.
  5. UIApplication.sendEvent(_:): The application object packages the touch into a UIEvent containing UITouch objects.
  6. UIWindow.sendEvent(_:): The key window receives the event.
  7. Hit testing: hitTest(_:with:) traverses the view hierarchy (reverse pre-order) to find the deepest view under the touch point.
  8. Gesture recognizer pipeline: The window sends touches to all gesture recognizers in the hit-test view's chain before the view itself. Recognizers get first priority.
  9. Responder chain: If no gesture recognizer claims the touch, touchesBegan/Moved/Ended is called on the hit-test view, then up the responder chain.
  10. UIControl target-action: For UIButton, the control's internal gesture recognizer fires the target-action pair on .touchUpInside.
12. What is the responder chain? How is it constructed, and how does it differ from the view hierarchy? Senior

The responder chain is a linked list of UIResponder objects that can handle events. It follows this path:

Hit-test view → superview → … → root view → UIViewController → parent VC → … → UIWindow → UIApplication → UIApplicationDelegate

It differs from the view hierarchy in two ways:

  • View controllers are inserted: Between a VC's root view and its superview, the VC itself is in the chain. This is how VCs can handle events that their views don't.
  • UIWindow, UIApplication, and the app delegate are at the end — they're not views but can handle events (e.g., keyboard shortcuts, menu commands).

The chain is used for more than touches: it routes canPerformAction queries, menu commands, keyboard input, undo/redo, and becomeFirstResponder/resignFirstResponder.

13. How do UIGestureRecognizers interact with the responder chain? What does delaysTouchesBegan do? L6

Gesture recognizers (GRs) operate in a parallel pipeline alongside the responder chain. When a touch begins:

  1. All GRs attached to the hit-test view and its ancestors receive the touch simultaneously.
  2. GRs are in .possible state. They analyze incoming touches to decide whether to recognize.
  3. If a GR transitions to .recognized (discrete) or .began (continuous), it "claims" the touch. By default, the responder chain receives touchesCancelled — the view's touch handling is cancelled.

delaysTouchesBegan: When true, the system withholds touchesBegan from the view until the GR fails. This prevents the view from starting visual feedback (e.g., highlight) that would need to be reversed. The downside: perceived latency — the view doesn't respond until the GR decides.

cancelsTouchesInView: When false, both the GR and the view receive touches simultaneously. Useful for analytics GRs that observe without interfering.

14. How does UIScrollView's gesture recognition interact with its subviews' tap gestures? L6

UIScrollView has a UIPanGestureRecognizer with delaysTouchesBegan = false and a private UIScrollViewDelayedTouchesBeganGestureRecognizer that introduces a ~150ms delay. This creates the "scroll vs tap" disambiguation:

  1. Touch begins → scroll view's delayed-touch GR starts a timer (~150ms).
  2. If the finger moves beyond a threshold before the timer fires → pan gesture recognizes → scroll begins, subview never gets touchesBegan.
  3. If the finger stays still → timer fires → touchesBegan is delivered to the subview → tap can be recognized.

delaysContentTouches = false disables this delay — subviews get touches immediately, but scrolling feels less responsive because the scroll view must wait for the subview's GR to fail.

canCancelContentTouches = true (default) allows the scroll view to cancel the subview's touches mid-stream if a pan is detected after the initial delay.

15. Explain require(toFail:) and how it creates gesture dependency chains. Senior

gestureA.require(toFail: gestureB) means gesture A cannot transition to .recognized until gesture B has transitioned to .failed. This creates a dependency: A waits for B to fail before it can succeed.

Classic use case: Single-tap vs double-tap. The single-tap GR requires the double-tap GR to fail. If the user double-taps, the double-tap recognizes and the single-tap never fires. If the user single-taps, the double-tap fails (timeout), then the single-tap fires — with a delay equal to the double-tap's timeout.

Performance implication: Every require(toFail:) adds latency equal to the dependent GR's failure timeout. Chaining multiple dependencies compounds this delay. Use sparingly.

16. What is the first responder? How does the keyboard relate to the responder chain? Senior

The first responder is the responder object that currently receives keyboard input and action messages. A UITextField becomes first responder when tapped (calling becomeFirstResponder()), which triggers the keyboard to appear.

Keyboard events are sent to the first responder. If the first responder doesn't handle a key command, the event travels up the responder chain. This is how a parent view controller can handle keyboard shortcuts even when a text field is focused.

resignFirstResponder() dismisses the keyboard. UIView.endEditing(true) forces the entire subtree to resign, which is the common "tap outside to dismiss keyboard" pattern.

17. How does hit testing work with transforms (rotation, scale)? What about 3D transforms? L6

When hitTest converts the touch point to a subview's coordinate space via convert(_:from:), it applies the inverse of the subview's transform. For 2D affine transforms (rotation, scale, translation), this works correctly — the touch point is mapped into the untransformed coordinate space, and point(inside:with:) tests against the original bounds.

For 3D transforms (CATransform3D), hit testing uses the flattened 2D projection. The 3D transform is projected onto the screen plane, and the inverse of that 2D projection is used. This means hit testing works for perspective-rotated views, but the hit area corresponds to the visual projection, not the original rectangle.

Edge case: If a 3D transform makes the layer edge-on (perpendicular to the screen), the projection degenerates and hit testing fails — the view becomes untappable.

18. How would you implement a passthrough view that ignores touches on itself but lets subviews receive them? Senior

Override hitTest(_:with:) to return nil when the hit lands on self (not on a subview):

override func hitTest(_ point: CGPoint, with event: UIEvent?) -> UIView? {
    let hit = super.hitTest(point, with: event)
    return hit === self ? nil : hit
}

This lets subviews receive touches normally, but touches on the empty areas of the view fall through to views behind it. This is useful for overlay containers, floating toolbars, or HUD views where the background should be transparent to touches.

19. What is UIEvent.EventType? What types of events flow through sendEvent(_:)? Senior

UIEvent carries several event types through the same sendEvent(_:) pipeline:

  • .touches: Direct and indirect touch events (finger, Apple Pencil).
  • .motion: Shake gesture events (motionBegan/Ended).
  • .remoteControl: Media remote events (play, pause, next track).
  • .presses: Physical button presses (keyboard keys, game controller buttons).
  • .hover (iPadOS/Catalyst): Pointer hover events.
  • .scroll (Catalyst): Trackpad/mouse scroll events.

All flow through UIApplication.sendEvent(_:) → UIWindow.sendEvent(_:), but the routing after that differs: touches go through hit testing, motion events go to the first responder, remote control events go up the responder chain.

20. How does Apple Pencil's touch handling differ from finger touches? What about predictive touches? L6

Apple Pencil touches have additional properties: force (pressure), altitudeAngle, azimuthAngle(in:), and azimuthUnitVector(in:). The type property is .pencil (vs .direct for fingers).

Higher sample rate: Pencil reports at 240 Hz (vs 60-120 Hz for fingers). Use event.coalescedTouches(for:) to get all intermediate samples between deliveries — critical for smooth drawing.

Predictive touches: event.predictedTouches(for:) returns estimated future touch positions (typically 2-4 frames ahead). Drawing apps render these predicted points to reduce perceived latency, then replace them with actual data as it arrives. This is how apps like Procreate achieve near-zero latency.

Pencil touches also support hover (Pencil 2nd gen on iPad Pro): the pencil reports position before touching the screen, enabling preview effects.

Layout System (Questions 21–30)

21. How does Auto Layout's Cassowary constraint solver work? What is its time complexity? L6

Auto Layout uses the Cassowary incremental linear constraint solver. Each constraint is a linear equation or inequality: view.left = superview.left + 16 becomes x₁ = x₂ + 16. The solver maintains a simplex tableau that represents all constraints simultaneously.

Incremental means when you add or remove a single constraint, the solver doesn't re-solve from scratch — it performs a pivot operation on the existing tableau. This is O(n) for a single change, where n is the number of constraints.

Priorities are handled by the simplex method's objective function. Required constraints (priority 1000) are hard constraints — the system is infeasible if they conflict. Optional constraints (priority < 1000) are soft constraints — the solver minimizes their violation, weighted by priority.

Practical complexity: For typical layouts, solving is roughly O(n) to O(n·log n). But pathological cases (deeply nested views with cross-hierarchy constraints) can approach O(n²) or worse. Apple's WWDC 2018 session showed that constraint count should stay under ~1000 for smooth 60fps layout.

Architect Insight

The solver's performance depends on constraint locality. Constraints between siblings are cheap (local). Constraints between distant views in the hierarchy are expensive because they create dependencies across the tableau. This is why flat hierarchies with local constraints perform best.

22. Describe the three layout passes: constraint update, layout, and display. When does each run? L6

UIKit's layout engine runs three passes per RunLoop iteration, in order:

  1. Constraints update pass (outside → in, leaf → root): Calls updateConstraints() on views that have been marked via setNeedsUpdateConstraints(). Views add/remove/modify constraints. The Cassowary solver is updated incrementally. Direction: from leaves toward the root.
  2. Layout pass (outside → in, root → leaf): Calls layoutSubviews() on views marked via setNeedsLayout(). The solver's solution is read and applied to each view's frame. Direction: from root toward leaves (parent positions children).
  3. Display pass (outside → in): Calls draw(_:) on views marked via setNeedsDisplay(). Views that override draw(_:) render into their backing store.

All three passes are deferred — calling setNeedsLayout() doesn't trigger layout immediately. It marks the view as "dirty" and the pass runs at the end of the current RunLoop iteration. Use layoutIfNeeded() to force immediate layout (useful for animations).

23. What are constraint priorities? Explain the difference between required, defaultHigh, defaultLow, and fittingSize. Senior

Constraint priority is a Float from 1 to 1000:

  • .required (1000): Must be satisfied. Conflicts between required constraints crash in debug (unsatisfiable constraint log) and silently break a constraint in release.
  • .defaultHigh (750): Used by content hugging. "I prefer not to grow beyond my intrinsic size."
  • .defaultLow (250): Used by compression resistance. "I prefer not to shrink below my intrinsic size."
  • .fittingSizeLevel (50): Used by systemLayoutSizeFitting — the weakest priority, used for size calculation queries.

Key rule: Never set a priority to 1000 and then try to change it at runtime — required constraints cannot be demoted. Set it to 999 if you might need to change it later. The solver treats 999 and 1000 very differently: 999 is optional (can be violated), 1000 is required (violation = error).

24. Explain content hugging and compression resistance. Give a concrete example where getting them wrong causes bugs. Senior

Content hugging: "Don't stretch me." A high hugging priority means the view resists growing beyond its intrinsicContentSize. The solver adds: width ≤ intrinsicWidth at the hugging priority.

Compression resistance: "Don't squish me." A high resistance priority means the view resists shrinking below its intrinsic size. The solver adds: width ≥ intrinsicWidth at the resistance priority.

Bug example: Two labels side by side in a horizontal stack, total text wider than the screen. Both have compression resistance = 750 (default). Auto Layout must shrink one — but which? It's ambiguous. The solver picks arbitrarily, causing inconsistent truncation. Fix: set the less important label's compression resistance to 749.

Similarly: a label next to a text field. Both have hugging = 250 (default). Extra space must go somewhere — ambiguous. Fix: set the text field's hugging to 249 so it stretches, keeping the label tight.

25. What is translatesAutoresizingMaskIntoConstraints? Why does forgetting it cause so many bugs? Senior

When true (the default for programmatically created views), UIKit automatically generates constraints from the view's autoresizingMask to replicate springs-and-struts behavior. These auto-generated constraints almost always conflict with your manually added constraints, producing unsatisfiable constraint errors.

You must set translatesAutoresizingMaskIntoConstraints = false before adding your own constraints. Views created in Interface Builder have this set to false automatically. Views created in code default to true.

The auto-generated constraints are required (priority 1000) and pin the view to its current frame. They conflict with almost any other constraint you add, which is why the "Unable to simultaneously satisfy constraints" log is the #1 Auto Layout debugging issue.

26. How does UIStackView work internally? Does it use constraints? L6

Yes, UIStackView generates and manages Auto Layout constraints internally. It creates constraints between arranged subviews for spacing, alignment, and distribution. When you add/remove views or change properties (axis, spacing, distribution), it tears down and recreates the relevant constraints.

Key behaviors:

  • Non-rendering: UIStackView does not have a backing store — it overrides class var layerClass to return CATransformLayer, which cannot render content. It's purely a layout container.
  • isHidden optimization: Setting arrangedSubview.isHidden = true collapses the space — the stack view removes the spacing constraint for that view. This is animatable.
  • Distribution modes: .fill stretches one view (based on hugging priorities), .fillEqually makes all views equal size, .fillProportionally distributes based on intrinsic sizes, .equalSpacing and .equalCentering adjust spacing.
Performance Warning

Deeply nested stack views generate exponentially more constraints. A 3-level nesting of stacks with 5 items each can produce 100+ constraints. For scrolling cells, prefer flat constraint-based layouts over nested stacks.

27. What is intrinsicContentSize? Which views have it, and what happens when it changes? Senior

intrinsicContentSize is the natural size a view wants to be based on its content. UILabel returns the size needed to display its text. UIImageView returns the image's point size. UIButton returns the size for its title + image + padding. Plain UIView returns (.noIntrinsicMetric, .noIntrinsicMetric) — no natural size.

When content changes (e.g., label text changes), the view calls invalidateIntrinsicContentSize(). This tells Auto Layout to re-query intrinsicContentSize and re-solve constraints. The solver treats intrinsic size as two optional constraints (hugging and compression resistance) per axis.

For custom views, override intrinsicContentSize and call invalidateIntrinsicContentSize() whenever the content changes. Return .noIntrinsicMetric for axes where the view has no natural size.

28. What is the layout feedback loop, and how do you avoid it? L6

A layout feedback loop occurs when layoutSubviews() modifies something that triggers another layout pass, which triggers another, ad infinitum. Common causes:

  • Adding/removing constraints inside layoutSubviews()
  • Calling setNeedsLayout() inside layoutSubviews()
  • Changing a property that invalidates intrinsic content size during layout
  • Modifying a view's frame in a way that triggers its superview's layout

UIKit detects this and breaks the loop after a few iterations, logging a warning. But the damage is done — the layout is incorrect and CPU time is wasted.

Prevention: In layoutSubviews(), only read the solver's output (frames) and apply it. Never modify constraints, add/remove views, or change properties that trigger layout. Move constraint changes to updateConstraints().

29. How do Safe Area insets propagate through the view hierarchy? What about keyboard avoidance? L6

safeAreaInsets propagate from the window down through the view hierarchy. The window's safe area accounts for the status bar, home indicator, and notch. Each view's safe area is the intersection of its parent's safe area with its own bounds.

View controllers can add to the safe area via additionalSafeAreaInsets. Container VCs (navigation, tab bar) add their bar heights. This is how content automatically avoids bars without hardcoded offsets.

Keyboard avoidance: Starting in iOS 15, UIView has keyboardLayoutGuide — a layout guide that tracks the keyboard's frame. Constrain views to this guide for automatic keyboard avoidance. Before iOS 15, you observe UIResponder.keyboardWillShowNotification and manually adjust constraints or content insets.

safeAreaInsetsDidChange() is called when insets change (e.g., rotation, keyboard appearance via the keyboard layout guide). Override it to respond to changes.

30. How would you debug an "Unable to simultaneously satisfy constraints" error? Senior

The console log lists all conflicting constraints. Debugging steps:

  1. Identify the views: Each constraint references views by memory address. Use po [view address] in the debugger, or add accessibilityIdentifier to views — it appears in the constraint log.
  2. Check for autoresizing mask constraints: Look for constraints with NSAutoresizingMaskLayoutConstraint — you forgot translatesAutoresizingMaskIntoConstraints = false.
  3. Find the odd one out: The system breaks one constraint to resolve the conflict. The log says "Will attempt to recover by breaking constraint [X]." That's often the one you need to lower in priority or remove.
  4. Use symbolic breakpoint: Set a breakpoint on UIViewAlertForUnsatisfiableConstraints to catch the exact moment of conflict.
  5. Xcode's view debugger: The runtime view hierarchy shows constraint conflicts visually with warning icons.

Rendering Pipeline (Questions 31–40)

31. Describe the full Core Animation rendering pipeline from CATransaction commit to pixels on screen. L6

The pipeline spans two processes:

  1. App process — Commit phase:
    • At the end of the RunLoop iteration, CATransaction.commit() is called implicitly.
    • The transaction collects all layer tree changes (property mutations, added/removed layers).
    • Layout is resolved (layoutSubviews), display is resolved (draw(_:) for dirty layers).
    • The layer tree is encoded (serialized) and sent to the render server via IPC (Mach messages).
  2. Render server (backboardd) — Render phase:
    • Decodes the layer tree.
    • Builds a render tree optimized for GPU execution.
    • Submits GPU commands: texture uploads, compositing operations, shader programs.
    • The GPU composites all layers into the final framebuffer.
    • The display controller scans out the framebuffer to the physical display at the next vsync.

The render server runs at a higher priority than your app, ensuring smooth animation even when the app is busy. This is why Core Animation animations (which run in the render server) are smoother than manual frame-by-frame animation in the app process.

32. What is a CATransaction? What is the difference between implicit and explicit transactions? L6

A CATransaction batches layer tree changes into a single atomic commit to the render server.

Implicit transaction: Created automatically by Core Animation at the beginning of each RunLoop iteration. All layer property changes during that iteration are batched. Committed at the end of the iteration. You never see it — it's automatic.

Explicit transaction: Created with CATransaction.begin() / CATransaction.commit(). Useful for:

  • Disabling implicit animations: CATransaction.setDisableActions(true)
  • Changing animation duration: CATransaction.setAnimationDuration(0.5)
  • Adding a completion block: CATransaction.setCompletionBlock { ... }
  • Grouping changes that should commit together

Transactions nest. An explicit transaction inside an implicit one creates a nested scope. The outermost commit triggers the actual IPC to the render server.

33. What is off-screen rendering? What triggers it, and why is it expensive? L6

Off-screen rendering occurs when the GPU cannot composite a layer in a single pass and must render it to a temporary off-screen buffer first, then composite that buffer into the final framebuffer.

Triggers:

  • cornerRadius + masksToBounds (clipping rounded corners)
  • shadow without shadowPath (GPU must compute the shadow shape from the layer's alpha channel)
  • mask property (layer masking)
  • allowsGroupOpacity with non-1.0 opacity on a layer with sublayers
  • Custom shouldRasterize = true (intentional off-screen render + cache)

Why expensive: The GPU must (1) allocate an off-screen buffer in VRAM, (2) switch rendering context to the off-screen buffer, (3) render the layer subtree, (4) switch back to the framebuffer, (5) composite the off-screen buffer. The context switches are the main cost — they flush the GPU pipeline.

Mitigation: Set shadowPath for shadows. Use layer.cornerRadius without masksToBounds (iOS 13+ rounds corners without off-screen rendering for simple cases). Use pre-rendered images with rounded corners.

34. Explain shouldRasterize. When does it help, and when does it hurt? L6

shouldRasterize = true tells Core Animation to render the layer subtree into a bitmap cache (off-screen buffer) and reuse that cache for subsequent frames. The cache is stored in GPU memory.

Helps when: The layer subtree is complex (many sublayers, shadows, masks) but doesn't change frequently. The off-screen render happens once, and subsequent frames use the cached bitmap — much cheaper than re-compositing the subtree every frame.

Hurts when:

  • The layer changes every frame (animation, scrolling content) — the cache is invalidated and re-rendered every frame, making it more expensive than no caching.
  • The layer is larger than the screen — the cache consumes significant VRAM.
  • The rasterizationScale doesn't match the screen scale — the cached bitmap is scaled, causing blurriness. Always set rasterizationScale = UIScreen.main.scale.

Core Animation evicts rasterized caches after ~100ms of non-use. Use Instruments' Core Animation template — "Color Hits Green and Misses Red" — to verify caching effectiveness.

35. What is the render server? Why does it run out-of-process? Architect

The render server is backboardd (previously SpringBoard handled this on older iOS). It's a system daemon that composites all app windows and system UI into the final display framebuffer.

Why out-of-process:

  • Isolation: If your app hangs or crashes, animations already committed to the render server continue smoothly. The user sees a frozen UI but animations don't stutter.
  • Priority: The render server runs at real-time priority, higher than any app. It never misses a vsync deadline due to app-level work.
  • Security: Apps can't directly access GPU resources or other apps' layer trees. The render server mediates all GPU access.
  • Efficiency: The render server composites all apps' layers in a single GPU pass, sharing the GPU efficiently.

The tradeoff: IPC overhead. Every frame, the layer tree diff must be serialized and sent via Mach messages. This is why the "commit" phase in Instruments shows IPC time.

36. What is the difference between a layer's backing store and its contents property? L6

The backing store is the bitmap buffer (IOSurface) that holds the layer's rendered pixels. It's allocated when the layer needs to display custom-drawn content (via draw(_:) or display()). Its size = bounds.size × contentsScale × screen scale. A 100×100pt view at 3x scale = 300×300px = 360KB (at 4 bytes/pixel).

The contents property (layer.contents) is a CGImage (or IOSurface) that the layer displays. You can set it directly: layer.contents = image.cgImage. When you do this, no backing store is allocated — the layer displays the provided image directly.

UIImageView works this way: it sets layer.contents to the image's CGImage, avoiding the cost of a backing store and draw(_:). This is why UIImageView is more efficient than a custom view that draws an image in draw(_:).

37. What is contentsScale? What happens if you set it incorrectly? Senior

contentsScale defines the mapping between the layer's logical coordinate space (points) and its backing store's pixel resolution. On a 3x Retina display, contentsScale should be 3.0.

If set too low (e.g., 1.0 on a 3x display): the backing store is 1/3 the needed resolution. Core Animation scales it up, resulting in blurry rendering.

If set too high: the backing store is unnecessarily large, wasting memory and GPU bandwidth.

UIView automatically sets its layer's contentsScale to UIScreen.main.scale. But standalone CALayers (not backing a UIView) default to 1.0 — you must set contentsScale manually. This is a common source of blurry CATextLayer and CAShapeLayer content.

38. How do implicit animations work in Core Animation? Why don't they fire on UIView-backed layers? L6

When you change an animatable property on a standalone CALayer (e.g., layer.opacity = 0), Core Animation automatically creates a CABasicAnimation with a default 0.25s duration. This is an implicit animation.

The mechanism: Core Animation calls action(forKey:) on the layer's delegate (and the layer itself). If an action object (conforming to CAAction) is returned, it's executed as an animation. Standalone layers return a CABasicAnimation by default.

UIView-backed layers: UIView implements action(for:forKey:) (the delegate method) and returns NSNull() when outside a UIView.animate block — this suppresses implicit animations. Inside an animation block, UIView returns a CABasicAnimation configured with the block's parameters. This is how UIView.animate works under the hood — it temporarily changes the action returned by the layer delegate.

39. What is CADisplayLink? How does it relate to ProMotion's variable refresh rate? L6

CADisplayLink is a timer synchronized to the display's refresh rate. Its callback fires once per frame, giving you the timestamp (when the frame will be displayed) and targetTimestamp (the next frame's display time). The delta between them is the frame duration.

ProMotion (120Hz, variable): The display dynamically adjusts its refresh rate (24, 30, 48, 60, 80, 120 Hz) based on content. CADisplayLink's preferredFrameRateRange (iOS 15+) lets you specify minimum, maximum, and preferred frame rates. The system picks the best rate.

let link = CADisplayLink(target: self, selector: #selector(step))
link.preferredFrameRateRange = CAFrameRateRange(
    minimum: 30, maximum: 120, preferred: 120
)
link.add(to: .main, forMode: .common)

For animations that don't need 120fps (e.g., a clock hand), set a lower preferred rate to save battery. The system may also lower the rate if the GPU is under pressure.

40. How would you diagnose and fix a scrolling performance problem caused by rendering? Architect

Diagnosis workflow:

  1. Instruments → Core Animation: Check FPS counter. Look for drops below 60 (or 120 on ProMotion).
  2. Color Blended Layers: Red = alpha blending (expensive compositing). Minimize transparent views over other content.
  3. Color Off-screen Rendered: Yellow = off-screen rendering. Identify which layers trigger it.
  4. Color Hits Green / Misses Red: For rasterized layers — green = cache hit (good), red = cache miss (re-rendering every frame).
  5. Time Profiler: Check if the bottleneck is CPU (layout, drawing) or GPU (compositing).

Common fixes:

  • Set shadowPath for all shadows
  • Set layer.opaque = true and backgroundColor on cells to avoid blending
  • Pre-decode images on a background queue (CGImage decoding is lazy and expensive)
  • Use shouldRasterize for complex static cell content
  • Flatten view hierarchies — fewer layers = fewer compositing passes
  • Avoid cornerRadius + masksToBounds — use pre-rounded images or CAShapeLayer masks

Drawing & Core Graphics (Questions 41–48)

41. When should you override draw(_:) vs using CAShapeLayer or UIImageView? L6

Override draw(_:) when: You need complex, dynamic, pixel-level rendering that changes based on state — custom charts, waveforms, handwriting. The content is too complex for layer composition.

Use CAShapeLayer when: You need vector paths (lines, curves, shapes). CAShapeLayer is GPU-accelerated, animatable (path morphing), and doesn't require a backing store bitmap. Better for simple to moderate vector graphics.

Use UIImageView when: Displaying static or pre-rendered images. UIImageView sets layer.contents directly — no backing store allocation, no draw(_:) overhead. Most memory-efficient option.

Key tradeoff: draw(_:) allocates a backing store (width × height × 4 bytes × scale²). A full-screen view on iPhone 15 Pro Max at 3x = 1290×2796×4 = ~14MB. This is why you should avoid draw(_:) for large views unless absolutely necessary.

Memory Impact

Every view that overrides draw(_:) gets a backing store bitmap. Ten 200×200pt views at 3x = 10 × 600×600×4 = 14.4MB. Prefer layer-based approaches (CAShapeLayer, CAGradientLayer) that don't need backing stores.

42. Explain the Core Graphics coordinate system. How does it differ from UIKit's? Senior

Core Graphics (Quartz): Origin at bottom-left, Y increases upward (mathematical convention). This is the native coordinate system of CGContext.

UIKit: Origin at top-left, Y increases downward (screen convention). UIKit applies a transform (flip + translate) to the CGContext before calling draw(_:), so you can draw in UIKit coordinates.

When it matters: If you create a CGContext manually (e.g., UIGraphicsBeginImageContext), UIKit applies the flip. But if you use CGContext directly (e.g., in a CALayerDelegate.draw(_:in:) method), the context is not flipped — you're in CG coordinates. Images drawn with CGContext.draw(image:in:) will appear upside down unless you flip the context manually.

43. What is the difference between UIGraphicsImageRenderer and UIGraphicsBeginImageContext? Senior

UIGraphicsImageRenderer (iOS 10+) is the modern replacement:

  • Automatic format selection: Chooses the optimal pixel format. For content without alpha, it uses an opaque format (no alpha channel = 25% less memory). UIGraphicsBeginImageContext always uses RGBA.
  • Wide color support: Automatically uses extended sRGB or Display P3 when appropriate. The legacy API is limited to sRGB.
  • Closure-based API: No risk of forgetting UIGraphicsEndImageContext(). The context is automatically cleaned up.
  • PDF and image data: Can generate PDF data, PNG data, and JPEG data directly.
let renderer = UIGraphicsImageRenderer(size: CGSize(width: 100, height: 100))
let image = renderer.image { ctx in
    UIColor.red.setFill()
    ctx.fill(CGRect(x: 0, y: 0, width: 100, height: 100))
}
44. How does image decoding work in iOS? Why is it a performance concern? L6

When you create a UIImage from data or a file, the image is not decoded immediately. The compressed data (JPEG/PNG) is stored in memory. Decoding (decompression to raw pixels) happens lazily — the first time the image is rendered (displayed in a UIImageView or drawn in a context).

Why it's a problem: Decoding happens on the main thread during rendering. A 12MP photo (4032×3024) decodes to ~48MB of raw pixels. This takes 20-50ms — enough to drop multiple frames during scrolling.

Solutions:

  • Background pre-decoding: Draw the image into a CGContext on a background queue. The resulting image is already decoded.
  • Downsampling: Use CGImageSourceCreateThumbnailAtIndex with kCGImageSourceThumbnailMaxPixelSize to decode at the display size, not the full resolution. A 100×100pt cell doesn't need a 4032×3024 image.
  • ImageIO framework: CGImageSource supports incremental decoding and efficient thumbnailing without loading the full image into memory.
45. What is drawsAsynchronously? How does it work? L6

CALayer.drawsAsynchronously = true tells Core Animation to record the drawing commands (from draw(_:)) into a command buffer and execute them on a background thread, rather than executing them synchronously on the main thread.

The drawing commands are captured into a CGContext command buffer during the display pass. The actual rasterization (executing those commands to produce pixels) happens asynchronously. The layer displays a placeholder until the rasterization completes.

When it helps: Views with expensive draw(_:) implementations (complex paths, text rendering, gradients). It moves the CPU-intensive rasterization off the main thread.

Caveats: The drawing must be thread-safe — no accessing shared mutable state. There may be a 1-frame delay before the content appears. Not effective if the drawing is already fast.

46. How do you draw text efficiently in Core Graphics? Compare CTFramesetter, NSAttributedString drawing, and CATextLayer. L6

CTFramesetter (Core Text): Lowest-level, most control. Create a framesetter from an attributed string, create a frame for a path, draw the frame into a CGContext. Gives you line-by-line control, custom truncation, and inline attachments. Used by UILabel internally.

NSAttributedString.draw(in:): Convenience wrapper around Core Text. Draws attributed text into the current UIKit graphics context. Simpler API but less control. Suitable for most custom drawing needs.

CATextLayer: A layer that renders text without a backing store (GPU-rendered). Set string to an NSAttributedString. No draw(_:) needed. But: limited layout control, no line-break customization, and you must set contentsScale manually.

Performance ranking: CATextLayer (no backing store) > NSAttributedString.draw (one backing store) > CTFramesetter (most flexible but same backing store cost). For scrolling cells with text, UILabel (which uses Core Text internally with optimizations) is usually the best choice.

47. What is the difference between setNeedsDisplay() and setNeedsLayout()? Can you call them from a background thread? Senior

setNeedsDisplay() marks the view's content as dirty — its draw(_:) will be called during the next display pass. Used when the view's visual content needs to change (e.g., data changed, colors changed).

setNeedsLayout() marks the view's layout as dirty — its layoutSubviews() will be called during the next layout pass. Used when subview positions/sizes need to change.

Threading: Both must be called on the main thread only. They modify UIView state, which is not thread-safe. Calling from a background thread is undefined behavior — it may work sometimes and crash other times. In Swift 6 with strict concurrency, the compiler enforces this via @MainActor.

Both are coalesced: calling setNeedsDisplay() 100 times in one RunLoop iteration results in a single draw(_:) call.

48. How would you implement a high-performance custom drawing view that updates at 120fps? Architect

Don't use draw(_:) — it allocates a backing store and rasterizes on the CPU every frame. Instead:

  1. Use CAMetalLayer: Override layerClass to return CAMetalLayer.self. Render directly to the GPU via Metal. This is how game engines and drawing apps achieve 120fps.
  2. Use CADisplayLink with preferredFrameRateRange set to 120fps to drive the render loop.
  3. Double/triple buffer: Metal's drawable pool provides this automatically. While the GPU renders frame N, the display shows frame N-1, and you're preparing frame N+1.
  4. Minimize CPU work per frame: Pre-compute geometry, use GPU compute shaders for data processing, batch draw calls.

For simpler cases (animated vector graphics), use CAShapeLayer with CADisplayLink-driven path updates. The shape is GPU-rendered without a backing store.

Custom Views & Accessibility (Questions 49–55)

49. What methods should a well-architected custom UIView subclass override? Walk through the lifecycle. L6

A complete custom view lifecycle:

  1. init(frame:) and init?(coder:): Call a shared setup method. Create subviews, add constraints, set default properties. Do NOT reference bounds or superview here — they're not set yet.
  2. intrinsicContentSize: Return the natural size based on content. Call invalidateIntrinsicContentSize() when content changes.
  3. updateConstraints(): Modify constraints based on current state. Call super.updateConstraints() last. Only called when setNeedsUpdateConstraints() was called.
  4. layoutSubviews(): Adjust frames for subviews that can't be expressed with constraints (e.g., circular layout). Call super.layoutSubviews() first to let Auto Layout run.
  5. draw(_:): Custom Core Graphics drawing. Only override if you need pixel-level rendering. The rect parameter is the dirty region.
  6. didMoveToWindow(): Start/stop animations or observation. window != nil means the view is on screen.
  7. traitCollectionDidChange(_:): Respond to dark mode, size class, or accessibility changes.
50. How do you make a custom view work correctly with Auto Layout? What is the role of intrinsicContentSize? Senior

For Auto Layout compatibility:

  • Override intrinsicContentSize to return the natural size. This lets the view participate in stack views and constraint-based layouts without explicit width/height constraints.
  • Call invalidateIntrinsicContentSize() whenever the content changes (text, image, data).
  • Set appropriate content hugging and compression resistance priorities.
  • If the view has a fixed aspect ratio, add an internal aspect ratio constraint.

intrinsicContentSize generates two constraints per axis: width ≤ intrinsicWidth (hugging) and width ≥ intrinsicWidth (compression resistance), both at their respective priorities. This is how Auto Layout knows the view's preferred size without explicit size constraints.

51. What accessibility properties must a custom view implement to be VoiceOver-compatible? Senior

At minimum:

  • isAccessibilityElement = true: Marks the view as an accessible element. Container views should be false; leaf elements should be true.
  • accessibilityLabel: A concise description ("Play button", "Profile photo"). Read by VoiceOver.
  • accessibilityTraits: Describes the element type (.button, .header, .image, .adjustable). Affects how VoiceOver interacts with it.
  • accessibilityValue: Current value for dynamic elements ("50%", "3 of 10").
  • accessibilityHint: Describes the result of the action ("Double-tap to play the song").

For custom controls: implement accessibilityActivate() for tap actions, accessibilityIncrement()/accessibilityDecrement() for adjustable elements (sliders).

For container views with custom-drawn children: override accessibilityElements to provide an ordered array of UIAccessibilityElement objects with correct frames.

52. How does Dynamic Type work? What must a custom view do to support it? Senior

Dynamic Type lets users choose their preferred text size in Settings. The system provides scaled fonts via UIFont.preferredFont(forTextStyle:).

Custom view requirements:

  • Use UIFont.preferredFont(forTextStyle:) or UIFontMetrics for custom fonts.
  • Set adjustsFontForContentSizeCategory = true on labels and text views for automatic updates.
  • Override traitCollectionDidChange(_:) and check traitCollection.preferredContentSizeCategory for layout changes needed at different sizes.
  • Call invalidateIntrinsicContentSize() when the font size changes.
  • Ensure layouts accommodate larger text — use scroll views, flexible heights, and avoid fixed-height containers.

UIFontMetrics scales custom fonts proportionally: UIFontMetrics(forTextStyle: .body).scaledFont(for: customFont).

53. How do you implement a custom UIControl subclass? What is the target-action mechanism? L6

Subclass UIControl (not UIView) to get the target-action mechanism. Override touch tracking methods:

override func beginTracking(_ touch: UITouch, with event: UIEvent?) -> Bool {
    sendActions(for: .touchDown)
    return true
}
override func endTracking(_ touch: UITouch?, with event: UIEvent?) {
    if let touch = touch, bounds.contains(touch.location(in: self)) {
        sendActions(for: .touchUpInside)
    } else {
        sendActions(for: .touchUpOutside)
    }
}
override func cancelTracking(with event: UIEvent?) {
    sendActions(for: .touchCancel)
}

The target-action mechanism: addTarget(_:action:for:) stores a (target, action, events) tuple. When sendActions(for:) is called, UIControl iterates stored tuples matching the event and calls UIApplication.sendAction(_:to:from:for:), which dispatches the action through the responder chain if the target is nil.

Update isHighlighted, isSelected, and isEnabled states and adjust visual appearance in their didSet observers.

54. What is sizeThatFits(_:) vs systemLayoutSizeFitting(_:)? When do you use each? L6

sizeThatFits(_:): A manual sizing method from the pre-Auto Layout era. The parameter is a "suggested" size (typically the available width). The view calculates and returns its ideal size. Does NOT change the view's frame. Used by sizeToFit() internally.

systemLayoutSizeFitting(_:): An Auto Layout method. Asks the constraint solver for the view's size given a target size. Uses .fittingSizeLevel priority (50) for the target, so constraints take precedence. Returns the size that satisfies all constraints while being as close to the target as possible.

When to use:

  • sizeThatFits: Legacy views, manual layout, or views that don't use Auto Layout.
  • systemLayoutSizeFitting: Auto Layout views. Common for self-sizing table view cells: call it on the cell's contentView with UIView.layoutFittingCompressedSize to get the minimum height.
55. How do you handle trait collection changes for dark mode, size classes, and accessibility? Senior

Override traitCollectionDidChange(_:) and check which traits changed:

override func traitCollectionDidChange(_ previous: UITraitCollection?) {
    super.traitCollectionDidChange(previous)
    if traitCollection.hasDifferentColorAppearance(comparedTo: previous) {
        updateColors()
    }
    if traitCollection.preferredContentSizeCategory
        != previous?.preferredContentSizeCategory {
        invalidateIntrinsicContentSize()
    }
}

iOS 17+ (trait registration): Use registerForTraitChanges([UITraitUserInterfaceStyle.self]) { (self: MyView, _) in ... } for targeted trait observation, replacing the broad traitCollectionDidChange.

Best practices: Use semantic colors (UIColor.label, UIColor.systemBackground) and asset catalog colors with dark mode variants. Use UIImage symbol configurations for SF Symbols. Avoid caching resolved colors — they change when traits change.

Animations (Questions 56–65)

56. Compare UIView.animate, UIViewPropertyAnimator, and Core Animation (CABasicAnimation). When would you choose each? L6

UIView.animate: Simple, fire-and-forget animations. Set properties in the closure, specify duration and curve. No ability to pause, reverse, or scrub. Best for simple transitions.

UIViewPropertyAnimator: Interactive, interruptible animations. Supports pausing (pauseAnimation()), reversing (isReversed = true), scrubbing (fractionComplete), and custom timing curves (including spring). Can be started, paused, and resumed. Best for gesture-driven animations (swipe-to-dismiss, interactive transitions).

CABasicAnimation / CAKeyframeAnimation: Low-level Core Animation. Runs entirely in the render server — survives app suspension. Supports keypath animations on any animatable layer property, including properties UIView doesn't expose (e.g., borderWidth, shadowOffset, individual transform components). Best for complex layer animations.

Decision Guide

Simple transition → UIView.animate. Gesture-driven or interruptible → UIViewPropertyAnimator. Layer-specific properties or render-server animations → CABasicAnimation.

57. Explain the model layer vs presentation layer during animation. Why does hit testing break during animation? L6

When you animate a view from position A to position B:

  • Model layer: Immediately set to position B (the final value). view.frame returns position B even during the animation.
  • Presentation layer: Interpolates from A to B over the animation duration. view.layer.presentation()?.position returns the current visual position.

Hit testing breaks because hitTest uses the model layer's frame, not the presentation layer's. The view is visually at position A (presentation layer) but hit-testable at position B (model layer). The user taps the visible view and misses.

Fix: Override hitTest to use the presentation layer's frame, or use UIViewPropertyAnimator which updates the model layer continuously during interactive animations.

58. How does UIViewPropertyAnimator handle animation interruption? What happens when you reverse a running animation? L6

When you pause a running UIViewPropertyAnimator:

  1. The animation is paused at its current fractionComplete.
  2. The model layer is updated to match the presentation layer's current values (the "in-flight" values are captured).
  3. The Core Animation animation is removed from the render server.

When you set isReversed = true and call continueAnimation:

  • A new animation is created from the current position back to the original values.
  • The timing curve is applied in reverse — the animation decelerates into the original position naturally.

This is how iOS's interactive dismiss gesture works: the pan gesture drives fractionComplete. On release, the system decides whether to complete or reverse based on velocity and position, then calls continueAnimation with the appropriate isReversed value.

59. What is a CAAnimationGroup? How do you synchronize multiple animations? Senior

CAAnimationGroup groups multiple CAAnimation objects to run simultaneously with a shared duration and timing. All animations in the group start at the same time and share the group's beginTime, duration, and timingFunction.

let group = CAAnimationGroup()
group.duration = 0.5
group.animations = [scaleAnim, opacityAnim, rotationAnim]
layer.add(group, forKey: "combinedAnimation")

Individual animations within the group can have their own beginTime (relative to the group's start) for staggered effects. Their durations are clamped to the group's duration.

For UIView.animate, you achieve synchronization by setting all properties in the same animation block — UIKit creates a single implicit CATransaction that groups them.

60. How do spring animations work? What are damping ratio, initial velocity, and stiffness? L6

Spring animations simulate a damped harmonic oscillator. The physics equation: F = -kx - cv where k = stiffness, c = damping, x = displacement, v = velocity.

UIView.animate(withDuration:delay:usingSpringWithDamping:initialSpringVelocity:):

  • dampingRatio (0-1): 1.0 = critically damped (no bounce). 0.5 = bouncy. 0.0 = oscillates forever.
  • initialSpringVelocity: Starting velocity in "total animation distance per second." A value of 1.0 means the animation starts at a speed that would cover the full distance in 1 second.

UISpringTimingParameters (iOS 10+): Takes mass, stiffness, damping, and initialVelocity (as CGVector for 2D). More physically accurate. Used with UIViewPropertyAnimator.

iOS 17+ spring(duration:bounce:): Simplified API. bounce = 0 is critically damped, positive values add bounce. Duration is approximate — the spring settles when energy drops below a threshold.

61. What is CATransaction.setDisableActions(true)? When and why would you use it? Senior

It disables implicit animations for all layer property changes within the transaction scope. Without it, changing a standalone layer's properties (position, opacity, backgroundColor) triggers a default 0.25s animation.

CATransaction.begin()
CATransaction.setDisableActions(true)
shapeLayer.strokeEnd = 1.0
shapeLayer.fillColor = UIColor.red.cgColor
CATransaction.commit()

When to use:

  • Updating layer properties in response to scroll position (you want instant updates, not animated).
  • Setting initial layer state before the view appears.
  • Updating CAShapeLayer paths during gesture tracking.
  • Any time you change layer properties and don't want the default animation.

For UIView-backed layers, implicit animations are already suppressed outside UIView.animate blocks, so this is mainly needed for standalone CALayers.

62. How do you create a custom view controller transition animation? L6

Implement the UIViewControllerAnimatedTransitioning protocol:

  1. transitionDuration(using:): Return the animation duration.
  2. animateTransition(using:): Receive a UIViewControllerContextTransitioning context. Get the from/to views and the container view. Set up initial frames, animate, and call context.completeTransition(!context.transitionWasCancelled).

For interactive transitions, also implement UIViewControllerInteractiveTransitioning (or use UIPercentDrivenInteractiveTransition). A gesture recognizer drives update(_:), cancel(), or finish().

Wire it up via UIViewControllerTransitioningDelegate: set the presenting VC's transitioningDelegate and return your animator from animationController(forPresented:presenting:source:).

Key pitfall: Always call completeTransition(_:). Forgetting it leaves the view hierarchy in an inconsistent state — the from-view stays in the container and the app appears frozen.

63. What is CAKeyframeAnimation? How does it differ from CABasicAnimation? Senior

CABasicAnimation interpolates between two values (fromValue → toValue) with a single timing function. CAKeyframeAnimation interpolates through an array of values (keyframes) with per-segment timing.

CAKeyframeAnimation properties:

  • values: Array of keyframe values. The animation interpolates through them in order.
  • keyTimes: Array of Floats (0.0-1.0) specifying when each keyframe is reached. Must match values count.
  • timingFunctions: Array of timing functions for each segment between keyframes. Count = values.count - 1.
  • path: A CGPath for position animations — the layer follows the path. Overrides values.
  • calculationMode: .linear, .discrete, .paced, .cubic, .cubicPaced — controls interpolation between keyframes.

Use keyframe animations for multi-step sequences, path-following, and complex easing that can't be expressed with a single timing function.

64. How do you animate Auto Layout constraint changes? Senior

Modify the constraint's constant (or activate/deactivate constraints) outside the animation block, then call layoutIfNeeded() inside the animation block:

heightConstraint.constant = 200
UIView.animate(withDuration: 0.3) {
    self.view.layoutIfNeeded()
}

Why this works: Changing the constant marks the view as needing layout. layoutIfNeeded() forces an immediate layout pass. Inside the animation block, UIKit intercepts the resulting frame changes and creates Core Animation animations for them.

Common mistake: Changing the constant inside the animation block. This works but is conceptually wrong — the constraint change is immediate; it's the layout pass that should be animated.

For UIViewPropertyAnimator: call addAnimations { self.view.layoutIfNeeded() } after modifying constraints.

65. What is CAReplicatorLayer? Give a practical use case. Senior

CAReplicatorLayer automatically creates copies of its sublayers with incremental transforms, colors, and timing offsets. It's GPU-accelerated — the copies are rendered by the GPU without additional backing stores.

Properties: instanceCount (number of copies), instanceTransform (incremental transform per copy), instanceDelay (animation delay per copy), instanceColor / instanceRedOffset / etc. (color modification per copy).

Use cases:

  • Loading indicator: A single dot sublayer replicated 8 times with rotational transform and animation delay creates a spinning activity indicator.
  • Reflection effect: One copy with a vertical flip transform and reduced opacity creates a mirror reflection.
  • Audio visualizer: Bars replicated horizontally with scale animations and staggered delays.

It's far more efficient than creating N separate layers because the GPU handles replication.

Table & Collection Views (Questions 66–75)

66. How does UITableView's cell reuse mechanism work internally? What happens during dequeueReusableCell? L6

UITableView maintains a reuse pool — a dictionary keyed by reuse identifier, with each value being an array of off-screen cell instances.

  1. Cell scrolls off-screen: The table view removes it from the view hierarchy and places it in the reuse pool. prepareForReuse() is called on the cell.
  2. dequeueReusableCell(withIdentifier:for:): Checks the reuse pool for an available cell with that identifier. If found, removes it from the pool and returns it. If not found, creates a new instance (from the registered class/nib/storyboard prototype).
  3. Configuration: Your code configures the dequeued cell with new data in cellForRowAt.
  4. Display: The table view adds the cell to its view hierarchy at the correct position.

prepareForReuse() should reset visual state (clear images, reset labels, cancel pending async work) but NOT reset structural properties (constraints, subview hierarchy). The cell's structure should be identical for all reuses — only content changes.

Common Bug

Forgetting to cancel async image loads in prepareForReuse(). A slow image load completes after the cell is reused for a different row, showing the wrong image. Always cancel pending work and verify the cell's index path in the completion handler.

67. What is the difference between UITableViewDiffableDataSource and the traditional UITableViewDataSource? L6

Traditional UITableViewDataSource: You implement numberOfSections, numberOfRowsInSection, and cellForRowAt. When data changes, you must manually calculate which rows to insert/delete/move and call performBatchUpdates. Getting this wrong causes "inconsistent number of rows" crashes.

DiffableDataSource (iOS 13+): You create snapshots (NSDiffableDataSourceSnapshot) representing the complete data state. Apply the snapshot, and the framework automatically computes the diff and generates animated insertions, deletions, moves, and reloads.

var snapshot = NSDiffableDataSourceSnapshot<Section, Item>()
snapshot.appendSections([.main])
snapshot.appendItems(items, toSection: .main)
dataSource.apply(snapshot, animatingDifferences: true)

Key requirements: Section and Item types must be Hashable. The diff algorithm uses hash values to identify items. Items with the same hash are considered the same item (possibly moved). Use stable, unique identifiers — not array indices.

Performance: The diff algorithm is O(n) using a Myers-like algorithm. For very large datasets (10K+ items), use apply(snapshot, animatingDifferences: false) to skip animation computation.

68. How does self-sizing work for table view cells? What are the performance implications? L6

Set tableView.rowHeight = UITableView.automaticDimension and estimatedRowHeight = 44. The table view uses Auto Layout to compute each cell's height:

  1. The cell is configured with data.
  2. systemLayoutSizeFitting(UIView.layoutFittingCompressedSize) is called on the cell's contentView.
  3. The constraint solver computes the minimum height that satisfies all constraints.
  4. This height is used for that row.

Performance implications:

  • Auto Layout solving happens for every cell as it's about to appear. Complex constraints = slow sizing.
  • estimatedRowHeight is critical: the table view uses it for scroll indicator size and content offset calculations without measuring every cell. A bad estimate causes scroll bar jumping.
  • For large datasets, consider caching computed heights keyed by content hash.

Requirement: The cell must have an unbroken chain of constraints from top to bottom of the contentView. Missing a constraint = zero height or unsatisfiable constraints.

69. What is UICollectionViewCompositionalLayout? How does it compare to UICollectionViewFlowLayout? L6

FlowLayout: A single-section, line-based layout. Items flow left-to-right, top-to-bottom (or along the scroll axis). All items in a section share the same layout behavior. Limited customization — complex layouts require subclassing and overriding layoutAttributesForElements(in:).

CompositionalLayout (iOS 13+): A declarative, composable layout system with three levels:

  • Item: Size relative to group (.fractionalWidth, .fractionalHeight, .absolute, .estimated).
  • Group: Horizontal, vertical, or custom arrangement of items. Groups define the repeating unit.
  • Section: Contains groups. Each section can have a different layout. Supports orthogonal scrolling (horizontal scroll within a vertical layout).

Key advantages: Per-section layouts (grid in section 0, carousel in section 1), orthogonal scrolling sections, section headers/footers as supplementary views, decoration views, and NSCollectionLayoutBoundarySupplementaryItem for sticky headers.

CompositionalLayout replaces nearly all use cases that previously required custom UICollectionViewLayout subclasses.

70. How does prefetching work in UITableView and UICollectionView? Senior

Implement UITableViewDataSourcePrefetching (or UICollectionViewDataSourcePrefetching):

  • prefetchRowsAt indexPaths:: Called for rows that are approaching the visible area. Start async data loading (image downloads, database queries) here.
  • cancelPrefetchingForRowsAt indexPaths:: Called when rows are no longer likely to be displayed (user changed scroll direction). Cancel pending work here.

The system predicts which cells will be needed based on scroll velocity and direction. It calls prefetch ~2-3 screens ahead of the visible area.

Implementation pattern: Start an async operation in prefetchRowsAt, store the operation/task keyed by index path, cancel it in cancelPrefetchingForRowsAt, and check for a cached result in cellForRowAt before starting a new load.

71. What is UICollectionView.CellRegistration? How does it improve over register/dequeue? Senior

CellRegistration (iOS 14+) combines registration and configuration into a single, type-safe object:

let registration = UICollectionView.CellRegistration<MyCell, Item> {
    cell, indexPath, item in
    cell.configure(with: item)
}

// In diffable data source:
dataSource = UICollectionViewDiffableDataSource(collectionView: cv) {
    cv, indexPath, item in
    cv.dequeueConfiguredReusableCell(using: registration,
                                      for: indexPath, item: item)
}

Advantages over register/dequeue:

  • Type safety: No string-based reuse identifiers. No force-casting from dequeueReusableCell.
  • Unified configuration: The configuration closure is called for both new and reused cells — no separate cellForRowAt logic.
  • No separate registration step: Registration happens implicitly on first use.
72. How do you implement a sticky header in UICollectionView? Compare FlowLayout and CompositionalLayout approaches. L6

FlowLayout: Set sectionHeadersPinToVisibleBounds = true. The flow layout automatically adjusts the header's layout attributes to keep it at the top of the visible area as the section scrolls.

CompositionalLayout: Create a NSCollectionLayoutBoundarySupplementaryItem with pinToVisibleBounds = true:

let headerSize = NSCollectionLayoutSize(
    widthDimension: .fractionalWidth(1.0),
    heightDimension: .estimated(44)
)
let header = NSCollectionLayoutBoundarySupplementaryItem(
    layoutSize: headerSize,
    elementKind: UICollectionView.elementKindSectionHeader,
    alignment: .top
)
header.pinToVisibleBounds = true
section.boundarySupplementaryItems = [header]

Custom FlowLayout subclass (for advanced behavior): Override layoutAttributesForElements(in:) and shouldInvalidateLayout(forBoundsChange:). Manually adjust the header's frame to clamp it within the visible bounds while staying within its section's range.

73. What is the reconfigureItems API in DiffableDataSource? How does it differ from reloadItems? L6

reconfigureItems (iOS 15+) updates existing cells in place without dequeuing new cells. The existing cell's configuration closure is called again with the updated data. The cell is not removed from the collection view — its identity is preserved.

reloadItems dequeues a new cell, configures it, and replaces the old cell with a cross-fade animation. This is more expensive and causes a visual flash.

When to use reconfigure: The item's identity hasn't changed, but its content has (e.g., a "like" count updated, an image finished loading). The cell stays in place and smoothly updates.

When to use reload: The cell type or structure might change based on the new data (e.g., switching from a text cell to an image cell).

var snapshot = dataSource.snapshot()
snapshot.reconfigureItems([updatedItem])
dataSource.apply(snapshot)
74. How would you build an infinite-scrolling feed with UICollectionView? What are the performance considerations? Architect

Architecture:

  1. Pagination trigger: In willDisplay cell or prefetching delegate, detect when the user is within N items of the end. Trigger a page fetch.
  2. Data source update: Append new items to the diffable data source snapshot. Use apply(snapshot, animatingDifferences: true) — the diff algorithm efficiently handles appends.
  3. Loading indicator: Add a footer supplementary view that shows a spinner while loading.

Performance considerations:

  • Memory: DiffableDataSource holds all item identifiers in memory. For 100K+ items, use SectionSnapshot with lazy loading or implement a custom data source that only holds visible + buffer items.
  • Diff performance: Appending to a snapshot is O(n) where n = new items. But applying a snapshot with 50K existing items still requires hashing all items. Consider animatingDifferences: false for large datasets.
  • Image management: Use prefetching to start image loads 2-3 screens ahead. Cancel loads for cells that scroll past. Use thumbnail sizes appropriate for cell dimensions.
  • Cell reuse: Ensure prepareForReuse() cancels all async work. Verify the cell's identity in async completion handlers.
75. Compare UIContentConfiguration (iOS 14+) with the traditional cell subclassing approach. What problem does it solve? L6

Traditional approach: Subclass UITableViewCell/UICollectionViewCell. Add custom subviews. Configure in cellForRowAt. The cell class owns both the view hierarchy and the configuration logic — tightly coupled.

UIContentConfiguration: Separates configuration (data) from presentation (views). A configuration is a lightweight value type describing what to display. A content view renders it.

var config = UIListContentConfiguration.subtitleCell()
config.text = "Title"
config.secondaryText = "Subtitle"
config.image = UIImage(systemName: "star")
cell.contentConfiguration = config

Problems solved:

  • Reusability: The same configuration can be applied to any cell, header, or custom view. No subclass needed.
  • State management: UIConfigurationState tracks highlighted, selected, disabled states. The configuration automatically updates appearance for each state via configurationUpdateHandler.
  • Composability: Background configurations (UIBackgroundConfiguration) and content configurations are independent — mix and match.
  • System integration: UIListContentConfiguration provides system-standard cell layouts that automatically adapt to Dynamic Type, accessibility, and platform conventions.
