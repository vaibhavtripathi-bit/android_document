Section 1: App Sandbox & Data Protection (Questions 1–10)

Architect 1. How does the iOS sandbox differ from Android's isolation model?

iOS and Android both isolate apps, but the mechanisms and philosophies differ fundamentally:

iOS Sandbox (Seatbelt / sandbox_init): Each app runs in a unique container directory with a kernel-enforced Mandatory Access Control (MAC) profile via the TrustedBSD framework. The sandbox profile is a whitelist — everything is denied by default, and only explicitly permitted operations succeed. The kernel evaluates every system call against the app's sandbox profile. Apps cannot discover or access other apps' containers. There is no concept of "permissions to access another app's files."

Android Isolation: Each app gets a unique Linux UID. Isolation is enforced via DAC (Discretionary Access Control) and SELinux (MAC). Apps can declare <provider> elements to expose data via ContentProviders, and other apps can request access through URI permissions. Android's model is more permissive by design — inter-app data sharing is a first-class concept via Intents and ContentProviders.

Key architectural differences:

  • Default posture: iOS denies everything not whitelisted; Android allows intra-UID access and explicit sharing
  • IPC: iOS restricts IPC to XPC services, App Groups, and URL schemes; Android has Binder IPC, Intents, ContentProviders, and BroadcastReceivers
  • File system visibility: iOS apps cannot enumerate other apps' containers; Android apps with storage permission historically could read shared storage
  • Dynamic permissions: iOS sandbox profiles are static (compiled into the binary); Android SELinux policies can be updated via system updates

At L6, the key insight is that iOS trades flexibility for security — the restrictive model makes entire classes of attacks impossible (e.g., one app reading another's database), while Android's openness enables richer inter-app workflows but increases the attack surface.

L6 2. Explain the four file protection classes and when you'd use each.

iOS Data Protection encrypts files with keys derived from the device passcode and hardware UID. The four protection classes control when the decryption key is available:

  • NSFileProtectionComplete (kSecAttrAccessibleWhenUnlocked): File is only accessible when the device is unlocked. The class key is evicted from memory 10 seconds after lock. Use for: sensitive user data — health records, financial data, private messages.
  • NSFileProtectionCompleteUnlessOpen: File can be created/opened while unlocked, then continues to be writable even after lock (using a derived per-file key). The file cannot be opened while locked, but an already-open file handle remains valid. Use for: large downloads or uploads that must continue in background.
  • NSFileProtectionCompleteUntilFirstUserAuthentication (default): File is accessible from first unlock until device restart. The class key is loaded at first unlock and stays in memory. Use for: most app data — databases, caches, preferences. This is the default for all files.
  • NSFileProtectionNone: File is always accessible, even before first unlock. The key is protected only by the hardware UID (no passcode component). Use for: data needed during boot before the user unlocks — device management profiles, certain system data. Avoid for any user data.

Key hierarchy: Each class has a class key, wrapped by the device's UID key and (for the first three classes) the user's passcode key. Per-file keys are generated randomly and wrapped with the appropriate class key. This means changing the passcode only requires re-wrapping the class keys, not re-encrypting every file.

L6 3. What are entitlements and how does the kernel validate them?

Entitlements are key-value pairs embedded in an app's code signature that declare capabilities the app requires. They are the bridge between the app and the kernel's security policy.

How they work:

  1. The developer declares entitlements in the .entitlements plist file
  2. During code signing, codesign embeds the entitlements into the code signature's CMS blob
  3. The provisioning profile (signed by Apple) contains a whitelist of allowed entitlements
  4. At launch, AMFI (Apple Mobile File Integrity) validates that the app's entitlements are a subset of those allowed by the provisioning profile
  5. The kernel's sandbox module reads the entitlements to configure the app's sandbox profile

Kernel validation path: When an app calls a protected API (e.g., accessing HealthKit), the kernel checks task_get_special_port → reads the entitlements blob from the process's cs_blob → looks up the specific entitlement key (e.g., com.apple.developer.healthkit) → allows or denies the operation.

Categories of entitlements:

  • App-level: Push notifications, App Groups, Keychain sharing, HealthKit, HomeKit
  • System-level: Only available to Apple's own apps — private entitlements for SpringBoard, kernel extensions
  • Restricted: Require Apple approval — CarPlay, NFC, network extensions

Entitlements cannot be forged because they're validated against the provisioning profile, which is signed by Apple's certificate authority. Modifying entitlements invalidates the code signature.

Architect 4. Why is jailbreak detection fundamentally unreliable?

Jailbreak detection is an arms race that the defender (your app) structurally cannot win. The fundamental problem: your code runs inside the environment it's trying to inspect. A jailbreak that controls the kernel can intercept and falsify every check your app makes.

Common detection techniques and their bypasses:

  • Check for Cydia/Sileo (canOpenURL, file existence): Trivially bypassed by hooking NSFileManager or UIApplication to return false for specific paths. Tools like Liberty Lite and Shadow do this automatically.
  • Check for writable system paths (/private/var/stash): Rootless jailbreaks (e.g., Dopamine) don't modify system paths at all.
  • Fork detection (sandbox prevents fork on stock iOS): Bypassed by hooking fork() to return -1.
  • Dylib injection detection (_dyld_image_count): Bypassed by hooking dyld functions to hide injected libraries.
  • Integrity checks (check own binary hash): Bypassed by returning the expected hash from hooked file read functions.

The architectural lesson: Never use jailbreak detection as a security boundary. Instead, design your security model assuming the device is compromised:

  • Use server-side validation for all security-critical decisions
  • Use certificate pinning to prevent MITM even on jailbroken devices
  • Use the Secure Enclave for key storage (hardware-backed, survives jailbreak)
  • Use Device Attestation (DeviceCheck / App Attest) for server-side device integrity verification
  • Treat jailbreak detection as a signal for risk scoring, not a gate for access control
Pitfall

Blocking jailbroken users entirely is often a poor UX decision and may violate regulations in some jurisdictions. Financial apps should increase monitoring and limit transaction amounts on suspected-jailbroken devices rather than refusing to launch.

L6 5. How does the Data Protection key hierarchy work with the Secure Enclave?

The Secure Enclave Processor (SEP) is an isolated coprocessor with its own encrypted memory, boot ROM, and hardware RNG. It plays a central role in the Data Protection key hierarchy:

Key generation and storage:

  1. The UID key is fused into the Secure Enclave's silicon during manufacturing. It never leaves the SEP — not even Apple can extract it. All key derivation happens inside the SEP.
  2. When the user sets a passcode, the SEP derives a passcode key using a hardware-accelerated key derivation function (based on Tangling, which mixes the passcode with the UID key through multiple iterations).
  3. Class keys are generated by the SEP and wrapped (encrypted) using a combination of the UID key and passcode key. The wrapped class keys are stored in the Effaceable Storage region of flash.
  4. When the device is unlocked, the SEP unwraps the class keys and holds them in its protected memory. The Application Processor can request cryptographic operations (encrypt/decrypt) but never sees the raw keys.

Anti-brute-force: The SEP enforces escalating delays between passcode attempts (up to 1 hour after 9 failed attempts). This is hardware-enforced — the AP cannot bypass it. After 10 failed attempts (if configured), the SEP destroys the Effaceable Storage, making all class keys unrecoverable.

Why this matters architecturally: Even if an attacker has physical access to the flash storage and can read every byte, they cannot decrypt files without the UID key (hardware-bound) and the passcode (user-known). The SEP ensures these are never exposed to software.

Architect 6. Design a defense-in-depth strategy assuming sandbox compromise.

A defense-in-depth strategy layers multiple independent security mechanisms so that compromising one layer doesn't compromise the system. Assuming the sandbox is breached (e.g., via a kernel exploit):

Layer 1 — Data at rest:

  • Use NSFileProtectionComplete for all sensitive files — even if the sandbox is breached, files are encrypted when locked
  • Store secrets in the Keychain with kSecAttrAccessibleWhenPasscodeSetThisDeviceOnly — hardware-backed, non-exportable
  • Use application-level encryption (AES-GCM via CryptoKit) on top of Data Protection for the most sensitive data

Layer 2 — Data in transit:

  • Certificate pinning prevents MITM even if the attacker installs a root CA
  • Mutual TLS (mTLS) for high-value API endpoints
  • End-to-end encryption for user-to-user communication (keys never touch your server)

Layer 3 — Server-side validation:

  • App Attest (DCAppAttestService) to verify the app binary hasn't been tampered with
  • Server-side rate limiting and anomaly detection
  • Never trust client-side authorization decisions — re-validate on the server

Layer 4 — Runtime integrity:

  • Use Swift (memory-safe) instead of ObjC/C where possible
  • Obfuscate sensitive logic (not as a security boundary, but to raise the cost of reverse engineering)
  • Detect debugger attachment and environment anomalies as risk signals

Layer 5 — Incident response:

  • Remote kill switch for compromised API keys
  • Token rotation on suspicion of compromise
  • Audit logging for all sensitive operations
Senior 7. What happens to protected files when the device is locked?

When the device locks, the Secure Enclave begins evicting class keys based on the protection class:

  • NSFileProtectionComplete: The class key is evicted from SEP memory ~10 seconds after lock. Any open file handles become invalid — reads and writes fail with an error. New files cannot be created or opened.
  • NSFileProtectionCompleteUnlessOpen: Files that were already open remain accessible (the per-file key was derived while unlocked and is held in memory). New files cannot be opened. When the last open handle is closed, the per-file key is discarded.
  • NSFileProtectionCompleteUntilFirstUserAuthentication: The class key remains in memory after lock. Files remain fully accessible. The key is only evicted on device restart (before first unlock).
  • NSFileProtectionNone: Always accessible. The key is derived solely from the hardware UID, which is always available.

Practical impact: Background tasks (BGTaskScheduler, push notification handlers) that need file access must use CompleteUntilFirstUserAuthentication or lower. If you use Complete protection, background operations will fail when the device is locked. Core Data databases should use CompleteUntilFirstUserAuthentication unless they contain highly sensitive data that justifies the background access limitation.

Senior 8. How do App Groups enable data sharing between apps?

App Groups create a shared container directory accessible to all apps and extensions that belong to the same group. The mechanism:

  1. Enable the App Groups capability and specify a group identifier (e.g., group.com.company.appname)
  2. The group ID is added to the app's entitlements and provisioning profile
  3. The kernel creates a shared container at a path accessible to all group members
  4. Access the shared container via FileManager.default.containerURL(forSecurityApplicationGroupIdentifier:)

What can be shared:

  • Files: Any files written to the shared container directory
  • UserDefaults: UserDefaults(suiteName: "group.com.company.appname")
  • Keychain items: Items with a matching kSecAttrAccessGroup
  • Core Data: A shared SQLite store in the group container

Security considerations: The shared container has CompleteUntilFirstUserAuthentication protection by default. All apps in the group must be signed by the same team. There is no fine-grained access control within the group — all members have full read/write access. For extensions (Today Widget, Share Extension), App Groups are the primary mechanism for sharing data with the host app.

L6 9. Explain the container structure of an iOS app.

Each iOS app has an isolated container with a well-defined directory structure. The container path is randomized (UUID-based) and changes on reinstall:

L6 10. What is the TrustedBSD MAC framework's role in sandboxing?

TrustedBSD is a Mandatory Access Control (MAC) framework integrated into the XNU kernel. It provides the enforcement mechanism for the iOS sandbox (also known as "Seatbelt").

How it works:

  1. The MAC framework installs policy hooks at every security-relevant kernel operation — file open, network connect, mach port allocation, process fork, etc.
  2. When an app performs a system call, the kernel invokes the MAC hook before executing the operation
  3. The sandbox policy module evaluates the operation against the app's sandbox profile (a compiled set of allow/deny rules)
  4. If the profile denies the operation, the system call returns an error (typically EPERM) without executing

Sandbox profiles: Each app type has a profile. Third-party apps use the container profile, which restricts file access to the app's container, limits network operations, prevents access to most hardware, and blocks inter-process communication except through approved channels (XPC, URL schemes).

Why MAC, not DAC? Discretionary Access Control (DAC, Unix permissions) lets the file owner change permissions. MAC is enforced by the kernel regardless of the process's UID or file ownership. Even if an app runs as root (impossible on stock iOS, but relevant for jailbreak analysis), the MAC policy still restricts it. This is why the sandbox survives privilege escalation — the MAC layer is independent of Unix permissions.

Section 2: Transport Security (Questions 11–18)

Senior 11. Explain ATS requirements and when exceptions are justified.

App Transport Security (ATS) enforces secure network connections by default. All URLSession and WKWebView connections must meet these requirements:

  • TLS 1.2 or later
  • Forward secrecy (ECDHE key exchange)
  • Certificate signed by a trusted CA with at least SHA-256 hash
  • RSA key ≥ 2048 bits or ECC key ≥ 256 bits
  • AES-128 or AES-256 encryption

Justified exceptions (configured in Info.plist under NSAppTransportSecurity):

  • NSAllowsArbitraryLoads: Disables ATS entirely. Only justified for apps that are primarily web browsers or must connect to arbitrary user-specified servers. App Review requires justification.
  • NSExceptionDomains: Per-domain exceptions for legacy servers you don't control. Must document why the server can't be upgraded.
  • NSAllowsLocalNetworking: Allows unencrypted connections to local network (Bonjour, IoT devices). Justified for home automation and local device communication.
  • NSAllowsArbitraryLoadsInWebContent: Allows WKWebView to load arbitrary content while keeping URLSession strict. Justified for apps with embedded browsers.
Pitfall

Apple reviews ATS exceptions during App Review. Blanket NSAllowsArbitraryLoads = YES without justification may cause rejection. Always use the most specific exception possible — per-domain exceptions are preferred over global ones.

L6 12. Compare TLS 1.2 and 1.3 handshake differences.

TLS 1.3 (supported since iOS 12.2) significantly improves the handshake over TLS 1.2:

TLS 1.2 handshake (2-RTT):

  1. Client → Server: ClientHello (supported cipher suites, random)
  2. Server → Client: ServerHello, Certificate, ServerKeyExchange, ServerHelloDone
  3. Client → Server: ClientKeyExchange, ChangeCipherSpec, Finished
  4. Server → Client: ChangeCipherSpec, Finished

TLS 1.3 handshake (1-RTT, 0-RTT resumption):

  1. Client → Server: ClientHello + key share (client guesses the key exchange algorithm)
  2. Server → Client: ServerHello + key share, encrypted extensions, certificate, Finished
  3. Client → Server: Finished (application data can be sent immediately)

Key improvements in TLS 1.3:

  • 1-RTT handshake: Reduces latency by ~100ms on typical mobile connections
  • 0-RTT resumption: Returning clients can send data in the first message (at the cost of replay vulnerability for that data)
  • Removed insecure algorithms: No RSA key exchange (no forward secrecy), no RC4, no SHA-1, no static DH. Only AEAD ciphers (AES-GCM, ChaCha20-Poly1305)
  • Encrypted handshake: Server certificate is encrypted (prevents passive fingerprinting of which sites a user visits)
  • Simplified cipher suites: Only 5 cipher suites vs dozens in TLS 1.2
L6 13. Implement certificate pinning using URLSessionDelegate.

Certificate pinning validates the server's certificate against a known-good copy embedded in the app, preventing MITM attacks even if a rogue CA issues a fraudulent certificate:

class PinnedSessionDelegate: NSObject, URLSessionDelegate {
    private let pinnedHashes: Set<String>

    init(pinnedHashes: Set<String>) {
        self.pinnedHashes = pinnedHashes
    }

    func urlSession(
        _ session: URLSession,
        didReceive challenge: URLAuthenticationChallenge,
        completionHandler: @escaping (URLSession.AuthChallengeDisposition, URLCredential?) -> Void
    ) {
        guard challenge.protectionSpace.authenticationMethod
                == NSURLAuthenticationMethodServerTrust,
              let serverTrust = challenge.protectionSpace.serverTrust
        else {
            completionHandler(.cancelAuthenticationChallenge, nil)
            return
        }

        // Evaluate the trust chain against system root CAs
        var error: CFError?
        guard SecTrustEvaluateWithError(serverTrust, &error) else {
            completionHandler(.cancelAuthenticationChallenge, nil)
            return
        }

        // Extract the server's public key and compute its SHA-256 hash
        guard let serverCert = SecTrustGetCertificateAtIndex(serverTrust, 0),
              let serverKey = SecCertificateCopyKey(serverCert),
              let serverKeyData = SecKeyCopyExternalRepresentation(serverKey, nil)
        else {
            completionHandler(.cancelAuthenticationChallenge, nil)
            return
        }

        let keyHash = SHA256.hash(data: serverKeyData as Data)
            .compactMap { String(format: "%02x", $0) }
            .joined()

        if pinnedHashes.contains(keyHash) {
            let credential = URLCredential(trust: serverTrust)
            completionHandler(.useCredential, credential)
        } else {
            completionHandler(.cancelAuthenticationChallenge, nil)
        }
    }
}

Important: Pin the public key hash, not the full certificate. Certificates rotate (typically annually), but the public key can remain the same across renewals. This reduces the risk of bricking your app when the certificate is renewed.

Architect 14. Public key pinning vs certificate pinning — tradeoffs?

Certificate pinning: Pin the exact leaf certificate (DER-encoded bytes or its hash).

  • Pro: Strictest validation — any certificate change is detected
  • Con: Certificates expire and rotate (typically every 1-2 years). Each rotation requires an app update with the new certificate. If you miss the rotation window, the app is bricked for users who haven't updated.

Public key pinning: Pin the Subject Public Key Info (SPKI) hash of the certificate's public key.

  • Pro: The public key can remain the same across certificate renewals (if you use the same CSR). Survives routine certificate rotation.
  • Con: If the private key is compromised and you must generate a new key pair, you still need an app update.

Recommended strategy for production:

  1. Pin the public key hash (not the certificate)
  2. Pin multiple keys: the current key, a backup key (generated but not yet deployed), and optionally the intermediate CA's key
  3. Include a remote pin update mechanism: fetch a signed pin list from a trusted endpoint (pinned with a long-lived backup key)
  4. Implement a fail-open timeout: if pinning fails for all users for more than N hours, temporarily disable pinning (with server-side monitoring) to prevent mass outage
Architect Must-Know

The biggest risk with pinning isn't an attacker — it's bricking your own app by forgetting to update pins before certificate rotation. Always have at least one backup pin and a remote update mechanism.

L6 15. How does certificate transparency work?

Certificate Transparency (CT) is a public audit system that makes it impossible for a CA to issue a certificate for your domain without it being publicly visible. Apple requires CT for all certificates trusted by iOS (since iOS 12.1.1 / macOS 10.14.2).

How it works:

  1. When a CA issues a certificate, it must submit the certificate to multiple CT logs (append-only, cryptographically verifiable Merkle trees)
  2. Each log returns a Signed Certificate Timestamp (SCT) — a promise that the certificate will be included in the log within a maximum merge delay (typically 24 hours)
  3. The SCTs are embedded in the certificate (via X.509 extension), delivered via TLS extension, or stapled via OCSP
  4. The client (iOS) verifies that the certificate has valid SCTs from the required number of independent logs

Why it matters: Before CT, a compromised or malicious CA could issue a fraudulent certificate for google.com and no one would know. With CT, any certificate is publicly logged, and domain owners can monitor logs for unauthorized certificates. Google, Apple, and others run log monitors that alert on suspicious issuance.

Apple's CT policy: Requires SCTs from at least two independent CT logs (one operated by Apple, one by a third party). Certificates without valid SCTs are rejected by the trust evaluation.

Architect 16. Design a certificate pinning strategy that handles rotation.

A production-grade pinning strategy must survive certificate rotation, key compromise, and CA changes without bricking the app:

Architecture:

┌──────────────────────────────────────────────────────────┐ │ Pin Management Architecture │ ├──────────────────────────────────────────────────────────┤ │ │ │ App Binary (hardcoded pins) │ │ ├── Current production SPKI hash │ │ ├── Backup SPKI hash (pre-generated key pair) │ │ └── Intermediate CA SPKI hash (fallback) │ │ │ │ Remote Pin Store (signed JSON, fetched periodically) │ │ ├── Primary pins (current + backup) │ │ ├── Emergency pins (CA-level fallback) │ │ ├── Pin expiry timestamps │ │ └── Signed by a long-lived offline key │ │ │ │ Validation Logic │ │ ├── Check server cert against local + remote pins │ │ ├── If all fail → report to monitoring + fail closed │ │ ├── If remote fetch fails → use cached + hardcoded │ │ └── Emergency bypass via server-signed kill switch │ └──────────────────────────────────────────────────────────┘

Rotation procedure:

  1. Generate a new key pair and CSR (keep the private key in HSM)
  2. Push the new public key hash to the remote pin store
  3. Wait for all active app versions to fetch the updated pins (monitor adoption)
  4. Deploy the new certificate to servers
  5. Remove the old pin from the remote store after a grace period
  6. Update the hardcoded pins in the next app release

Emergency key compromise: If the private key is compromised, activate the backup key immediately. The backup key's hash is already in the app binary, so no app update is needed. Generate a new backup key and push it to the remote store.

L6 17. What is the SecTrust evaluation process?

SecTrust is the Security framework's API for evaluating X.509 certificate chains. It's the foundation of all TLS trust decisions on iOS:

Evaluation steps:

  1. Chain building: Starting from the leaf certificate, SecTrust builds a chain to a trusted root by matching issuer/subject names and Authority Key Identifiers
  2. Signature verification: Each certificate's signature is verified using the issuer's public key
  3. Validity checking: Certificate dates (notBefore/notAfter) are checked against the current date
  4. Revocation checking: OCSP (Online Certificate Status Protocol) is checked to verify the certificate hasn't been revoked. iOS uses OCSP stapling by default.
  5. Policy evaluation: The certificate is checked against the specified policy (e.g., SSL policy verifies the hostname matches the certificate's SAN/CN)
  6. Trust anchor verification: The chain must terminate at a certificate in the system's trusted root store (or a custom anchor you've set)
  7. CT verification: Signed Certificate Timestamps are validated
func evaluateTrust(_ trust: SecTrust) -> Bool {
    // Set the SSL policy with hostname verification
    let policy = SecPolicyCreateSSL(true, "api.example.com" as CFString)
    SecTrustSetPolicies(trust, policy)

    var error: CFError?
    let result = SecTrustEvaluateWithError(trust, &error)

    if !result {
        print("Trust evaluation failed: \(error?.localizedDescription ?? "unknown")")
    }
    return result
}

Important: Always use SecTrustEvaluateWithError (iOS 12+) instead of the deprecated SecTrustEvaluate. The older API has confusing return semantics where "recoverable trust failure" was often misinterpreted as success.

Senior 18. How does NSPinnedDomains differ from programmatic pinning?

NSPinnedDomains (iOS 14+) is a declarative, Info.plist-based approach to certificate pinning that doesn't require any code:

<!-- Info.plist -->
<key>NSAppTransportSecurity</key>
<dict>
    <key>NSPinnedDomains</key>
    <dict>
        <key>api.example.com</key>
        <dict>
            <key>NSPinnedCAIdentities</key>
            <array>
                <dict>
                    <key>SPKI-SHA256-BASE64</key>
                    <string>abc123...=</string>
                </dict>
            </array>
        </dict>
    </dict>
</dict>

Differences from programmatic pinning:

  • No code required: Pinning is enforced by the system before your delegate is called. No risk of implementation bugs in your pinning logic.
  • Pins the CA identity: You pin the intermediate or root CA's SPKI hash, not the leaf certificate. This means any certificate issued by that CA for your domain is accepted — more resilient to leaf certificate rotation.
  • Cannot be bypassed: Unlike delegate-based pinning, which can be swizzled on jailbroken devices, NSPinnedDomains is enforced at the system level (harder to bypass).
  • No dynamic updates: Pins are baked into Info.plist. Changing them requires an app update. Programmatic pinning can fetch updated pins from a server.
  • Failure mode: Connection fails with a TLS error. No custom error handling or fallback logic possible.

Recommendation: Use NSPinnedDomains as a baseline (simple, hard to get wrong) and add programmatic pinning on top for dynamic pin updates and custom failure handling.

L6 19. Walk through the Keychain Services architecture.

The Keychain is a system-level encrypted database for storing small secrets (passwords, tokens, keys, certificates). It's managed by the securityd daemon, not your app process:

Architecture:

  • Persistence: Keychain items survive app deletion and reinstallation (unless kSecAttrAccessibleWhenPasscodeSetThisDeviceOnly is used with a passcode change)
  • Encryption: Items are encrypted using Data Protection class keys, same hierarchy as file protection
  • Access control: Items are scoped to the app's keychain access group (derived from the team ID and bundle ID)
  • Synchronization: Items with kSecAttrSynchronizable: true sync via iCloud Keychain (encrypted end-to-end)
Senior 20. Compare kSecAttrAccessible options.

The kSecAttrAccessible attribute controls when a Keychain item's decryption key is available:

  • kSecAttrAccessibleWhenUnlocked: Available only when device is unlocked. Default for most items. Use for: API tokens, session credentials.
  • kSecAttrAccessibleWhenUnlockedThisDeviceOnly: Same as above, but item is not included in backups or iCloud Keychain sync. Use for: device-specific tokens, biometric-protected secrets.
  • kSecAttrAccessibleAfterFirstUnlock: Available from first unlock until restart. Use for: background refresh tokens, push notification credentials.
  • kSecAttrAccessibleAfterFirstUnlockThisDeviceOnly: Same, non-migratable. Use for: device-bound background credentials.
  • kSecAttrAccessibleWhenPasscodeSetThisDeviceOnly: Only available when a passcode is set AND device is unlocked. If the user removes their passcode, the item is deleted. Use for: highest-security items that require device passcode as a precondition.
  • kSecAttrAccessibleAlways (deprecated): Always available. Don't use — provides minimal protection.
Best Practice

Use kSecAttrAccessibleWhenUnlockedThisDeviceOnly for most secrets. Use kSecAttrAccessibleAfterFirstUnlockThisDeviceOnly only for items needed by background processes. Never use kSecAttrAccessibleAlways.

L6 21. How does the Secure Enclave protect cryptographic keys?

The Secure Enclave Processor (SEP) provides hardware-backed key protection that is fundamentally stronger than software-only solutions:

Key properties of SE-backed keys:

  • Keys never leave the SEP: The private key is generated inside the Secure Enclave and cannot be exported. All cryptographic operations (sign, decrypt, key agreement) happen inside the SEP. The Application Processor only sees the results.
  • Hardware RNG: Key generation uses the SEP's dedicated hardware random number generator, not the AP's.
  • Tamper resistance: The SEP has its own encrypted memory bus. Physical probing attacks cannot extract keys.
  • Biometric gating: Keys can require Face ID / Touch ID authentication before use, enforced by the SEP (not bypassable by software).

    nil

    kSecAttrKeyType as String: kSecAttrKeyTypeECSECPrimeRandom,

Senior 22. Implement AES-GCM encryption using CryptoKit.

CryptoKit (iOS 13+) provides a modern, Swift-native API for cryptographic operations. AES-GCM provides both confidentiality and integrity (authenticated encryption):

import CryptoKit

struct SecureVault {
    private let key: SymmetricKey

    init() {
        self.key = SymmetricKey(size: .bits256)
    }

    func encrypt(_ data: Data) throws -> Data {
        let sealedBox = try AES.GCM.seal(data, using: key)
        // combined = nonce (12B) + ciphertext + tag (16B)
        guard let combined = sealedBox.combined else {
            throw CryptoError.sealFailure
        }
        return combined
    }

    func decrypt(_ data: Data) throws -> Data {
        let sealedBox = try AES.GCM.SealedBox(combined: data)
        return try AES.GCM.open(sealedBox, using: key)
    }

    func encryptWithAAD(_ data: Data, authenticating aad: Data) throws -> Data {
        let sealedBox = try AES.GCM.seal(
            data, using: key,
            authenticating: aad  // AAD is authenticated but not encrypted
        )
        guard let combined = sealedBox.combined else {
            throw CryptoError.sealFailure
        }
        return combined
    }
}

GCM internals: AES-GCM uses a 12-byte nonce (generated randomly by CryptoKit), AES-CTR mode for encryption, and GHASH for authentication. The 16-byte authentication tag covers both the ciphertext and any Additional Authenticated Data (AAD). Tampering with any byte causes decryption to fail with CryptoKitError.authenticationFailure.

L6 23. When would you use SecKey vs CryptoKit?

Use CryptoKit when:

  • You need symmetric encryption (AES-GCM, ChaCha20-Poly1305)
  • You need hashing (SHA-256/384/512) or HMAC
  • You need key agreement (Curve25519, P-256 ECDH)
  • You need digital signatures (Ed25519, P-256 ECDSA)
  • You're writing new code and targeting iOS 13+
  • You want a clean, type-safe Swift API

Use SecKey when:

  • You need Secure Enclave-backed keys — CryptoKit's SecureEnclave.P256 wraps SecKey but SecKey gives more control over access control flags
  • You need Keychain persistence for asymmetric keys — SecKey integrates directly with Keychain Services
  • You need RSA operations — CryptoKit doesn't support RSA
  • You need to work with X.509 certificates — SecCertificate/SecIdentity are SecKey-based
  • You're interoperating with legacy code or server-side systems that use RSA

In practice: Use CryptoKit for all new cryptographic logic. Use SecKey for Keychain storage, Secure Enclave operations, and RSA interop. They can work together — extract a SecKey's public key data and import it into CryptoKit for verification.

Architect 24. Design E2E encryption for a messaging app.

End-to-end encryption ensures that only the sender and recipient can read messages — not even the server. Here's a Signal Protocol-inspired design using CryptoKit primitives:

Key management:

  • Identity key pair: Long-lived Curve25519 key pair, generated on device registration, stored in Secure Enclave. Public key registered with the server.
  • Signed pre-key: Medium-lived Curve25519 key pair, rotated weekly. Signed by the identity key. Uploaded to server.
  • One-time pre-keys: Ephemeral Curve25519 key pairs, uploaded in batches. Each is used once for initial key agreement, then deleted.

Session establishment (X3DH):

  1. Alice fetches Bob's identity key, signed pre-key, and one one-time pre-key from the server
  2. Alice performs four ECDH operations to derive a shared secret
  3. The shared secret is fed into HKDF to derive the initial root key and chain key
func performX3DH(
    myIdentity: Curve25519.KeyAgreement.PrivateKey,
    myEphemeral: Curve25519.KeyAgreement.PrivateKey,
    theirIdentity: Curve25519.KeyAgreement.PublicKey,
    theirSignedPreKey: Curve25519.KeyAgreement.PublicKey,
    theirOneTimePreKey: Curve25519.KeyAgreement.PublicKey?
) throws -> SymmetricKey {
    var dhResults = Data()
    dhResults += try myIdentity.sharedSecretFromKeyAgreement(
        with: theirSignedPreKey
    ).withUnsafeBytes { Data($0) }
    dhResults += try myEphemeral.sharedSecretFromKeyAgreement(
        with: theirIdentity
    ).withUnsafeBytes { Data($0) }
    dhResults += try myEphemeral.sharedSecretFromKeyAgreement(
        with: theirSignedPreKey
    ).withUnsafeBytes { Data($0) }
    if let otpk = theirOneTimePreKey {
        dhResults += try myEphemeral.sharedSecretFromKeyAgreement(
            with: otpk
        ).withUnsafeBytes { Data($0) }
    }
    return HKDF<SHA256>.deriveKey(
        inputKeyMaterial: SymmetricKey(data: dhResults),
        info: Data("X3DH".utf8),
        outputByteCount: 32
    )
}

Message encryption (Double Ratchet): Each message uses a unique key derived from a ratcheting chain. The chain advances with every message, providing forward secrecy — compromising one message key doesn't reveal past or future messages. Messages are encrypted with AES-GCM using the derived message key.

L6 27. What are the limitations of Secure Enclave keys?

The Secure Enclave provides the strongest key protection on iOS, but has significant constraints:

  • Algorithm support: Only P-256 (secp256r1) elliptic curve. No RSA, no P-384/P-521, no Ed25519, no symmetric key storage. If you need AES, derive it from an ECDH shared secret.
  • No key export: Private keys cannot be extracted from the SEP under any circumstances. You cannot back up SE keys or migrate them between devices. If the device is lost, the keys are gone.
  • No iCloud sync: SE keys are bound to the specific hardware. They cannot be synced via iCloud Keychain. Each device must have its own SE key pair.
  • Performance: SE operations are slower than software crypto (~10ms per operation vs microseconds). Not suitable for bulk encryption — use SE for key agreement/signing, then use the derived key for bulk AES operations in software.
  • Storage limits: The SE has limited secure storage. Don't create thousands of keys — use key derivation to generate multiple purpose-specific keys from a single SE master key.
  • Biometric dependency: If you use .biometryCurrentSet and the user re-enrolls their biometrics, the key becomes inaccessible. Your app must handle this gracefully (re-create the key, re-authenticate the user).
Deep Dive

CryptoKit's SecureEnclave.P256.Signing.PrivateKey and SecureEnclave.P256.KeyAgreement.PrivateKey are thin wrappers around SecKey with kSecAttrTokenIDSecureEnclave. They provide the same hardware protection with a cleaner API.

L6 28. How does CryptoKit's Curve25519 key agreement work?

Curve25519 is a Montgomery curve designed by Daniel Bernstein, optimized for fast, constant-time key agreement. CryptoKit provides it via Curve25519.KeyAgreement:

import CryptoKit

// Each party generates a key pair
let alicePrivate = Curve25519.KeyAgreement.PrivateKey()
let alicePublic = alicePrivate.publicKey

let bobPrivate = Curve25519.KeyAgreement.PrivateKey()
let bobPublic = bobPrivate.publicKey

// Both derive the same shared secret
let aliceShared = try alicePrivate.sharedSecretFromKeyAgreement(
    with: bobPublic
)
let bobShared = try bobPrivate.sharedSecretFromKeyAgreement(
    with: alicePublic
)
// aliceShared == bobShared (Diffie-Hellman property)

// Derive a symmetric key using HKDF
let symmetricKey = aliceShared.hkdfDerivedSymmetricKey(
    using: SHA256.self,
    salt: protocolSalt,
    sharedInfo: Data("MyApp-v1".utf8),
    outputByteCount: 32
)

// Use the derived key for AES-GCM encryption
let sealed = try AES.GCM.seal(plaintext, using: symmetricKey)

Why Curve25519 over P-256?

  • Constant-time by design: The Montgomery ladder algorithm is inherently constant-time, reducing side-channel attack risk. P-256 implementations must be carefully written to avoid timing leaks.
  • Simpler implementation: Fewer parameters to get wrong, reducing the chance of implementation bugs.
  • Performance: Slightly faster than P-256 on most platforms.
  • No Secure Enclave support: The tradeoff — Curve25519 keys are software-only. For hardware-backed keys, you must use P-256.

Section 4: Privacy Frameworks (Questions 29–36)

L6 29. What are privacy manifests and required reason APIs?

Privacy manifests (introduced iOS 17, required Spring 2024) are structured declarations of an app's data collection and API usage. They address the gap between what apps claim in App Store privacy labels and what they actually do.

Privacy manifest file (PrivacyInfo.xcprivacy):

  • NSPrivacyTracking: Boolean — does the app or SDK use data for tracking?
  • NSPrivacyTrackingDomains: List of domains contacted for tracking purposes
  • NSPrivacyCollectedDataTypes: Structured list of data types collected, their purposes, and whether they're linked to the user's identity
  • NSPrivacyAccessedAPITypes: List of "required reason" APIs used, with the specific reason code for each

Required reason APIs: Apple identified APIs that can be used for fingerprinting and now requires a declared reason for using them:

  • File timestamp APIs (NSFileCreationDate, NSFileModificationDate): Can fingerprint the device by file creation patterns
  • System boot time (systemUptime, mach_absolute_time): Unique per-boot, usable as a session identifier
  • Disk space APIs (volumeAvailableCapacityKey): Disk space is somewhat unique per device
  • User defaults (UserDefaults): Can be used to store tracking identifiers
  • Active keyboard APIs: Keyboard list can fingerprint locale/language

Each SDK must include its own privacy manifest. Xcode aggregates all manifests into a single privacy report during archive.

Senior 30. Explain the ATTrackingManager flow and its impact.

App Tracking Transparency (ATT), introduced in iOS 14.5, requires explicit user consent before an app can track the user across other companies' apps and websites:

import AppTrackingTransparency

func requestTrackingPermission() {
    ATTrackingManager.requestTrackingAuthorization { status in
        switch status {
        case .authorized:
            // IDFA is available, can track
            let idfa = ASIdentifierManager.shared()
                .advertisingIdentifier
        case .denied, .restricted:
            // IDFA returns all zeros, must not track
            disableThirdPartyTracking()
        case .notDetermined:
            // Prompt hasn't been shown yet
            break
        @unknown default:
            break
        }
    }
}

Key rules:

  • The prompt can only be shown once. If denied, the user must go to Settings to change it.
  • You must request permission before accessing the IDFA or engaging in any tracking
  • "Tracking" is defined broadly: linking user/device data with third-party data for advertising, sharing data with data brokers, using third-party SDKs that combine your data with other apps' data
  • If ATTrackingManager.trackingAuthorizationStatus == .notDetermined, the IDFA returns all zeros

Impact: Opt-in rates are approximately 20-30% globally. This effectively killed deterministic cross-app attribution for most users, driving the industry toward SKAdNetwork, probabilistic attribution, and privacy-preserving measurement APIs.

L6 31. How does SKAdNetwork provide privacy-preserving attribution?

SKAdNetwork (SKAN) is Apple's privacy-preserving ad attribution framework. It tells advertisers which ad campaign led to an install without revealing the user's identity:

How it works:

  1. Ad impression: The ad network registers an impression with SKAdNetwork.startImpression, including the ad network ID, campaign ID, and source app ID
  2. Install: When the user installs and launches the advertised app, the system (not the app) detects the attribution
  3. Conversion value: The installed app can set a conversionValue (0-63 in SKAN 3, fine/coarse in SKAN 4) to signal post-install engagement
  4. Postback: After a privacy-preserving timer (24-48 hours), Apple sends a signed postback to the ad network containing: campaign ID, conversion value, and a cryptographic signature — but NO user identifier, NO device ID, NO timestamp precise enough to correlate

SKAN 4.0 improvements:

  • Multiple postbacks: Up to 3 postbacks at different time windows (0-2 days, 3-7 days, 8-35 days)
  • Hierarchical source identifiers: 2-digit, 3-digit, or 4-digit source ID depending on crowd anonymity thresholds
  • Coarse conversion values: Low/medium/high for lower-volume campaigns that don't meet the crowd anonymity threshold for fine-grained values

Privacy mechanism: The key insight is crowd anonymity — Apple only reveals granular data when enough installs from the same campaign exist to prevent singling out individual users. Low-volume campaigns get less data.

Architect 32. Design a privacy-first analytics system.

A privacy-first analytics system collects actionable product insights without tracking individual users:

Architecture principles:

  • No persistent user identifiers: Use session-scoped random IDs that rotate on each app launch. Never link sessions across launches.
  • On-device aggregation: Compute metrics locally before sending. Instead of "User X viewed screen Y 5 times," send "Screen Y was viewed 5 times in this session."
  • Differential privacy: Add calibrated noise to aggregated metrics before sending to the server. Apple uses this in their own analytics.
  • Data minimization: Only collect what you need. Don't collect "just in case."

Implementation:

struct PrivacyAnalytics {
    private var sessionEvents: [String: Int] = [:]
    private let sessionID = UUID()  // Ephemeral, not stored
    private let epsilon: Double = 1.0  // Differential privacy budget

    mutating func track(_ event: String) {
        sessionEvents[event, default: 0] += 1
    }

    func flush() -> AnalyticsPayload {
        let noisyEvents = sessionEvents.mapValues { count in
            max(0, count + laplacianNoise(epsilon: epsilon))
        }
        return AnalyticsPayload(
            appVersion: Bundle.main.shortVersion,
            osVersion: ProcessInfo.processInfo.operatingSystemVersionString,
            events: noisyEvents
            // No device ID, no IDFA, no user ID
        )
    }
}

Server-side: Aggregate across all sessions. Use statistical methods (confidence intervals, trend analysis) on the noisy data. The noise averages out across thousands of sessions, giving accurate aggregate metrics while protecting individual users.

Senior 34. Explain approximate vs precise location authorization.

iOS 14 introduced approximate location as a privacy option. Users can grant location access at two precision levels:

  • Precise (accuracyAuthorization == .fullAccuracy): GPS-level accuracy (~5-10 meters). Required for turn-by-turn navigation, geofencing, AR.
  • Approximate (accuracyAuthorization == .reducedAccuracy): ~5km radius. Sufficient for weather, regional content, city-level features.

How it works: When approximate location is granted, CLLocationManager returns a location with a large horizontalAccuracy value (~5000m). The returned coordinate is the center of a region containing the user, not their exact position. The region shifts periodically to prevent triangulation.

Temporary precise access: Apps can request one-time precise access via CLLocationManager.requestTemporaryFullAccuracyAuthorization(withPurposeKey:). The purpose key maps to a string in Info.plist explaining why precise location is needed. The precise access expires when the app moves to background.

Best Practice

Design your app to work with approximate location by default. Only request precise location when the user initiates a feature that genuinely requires it (e.g., tapping "Navigate here"). This respects user privacy and reduces permission fatigue.

L6 35. What changed with pasteboard access in iOS 16?

Apple progressively tightened pasteboard access across iOS versions to prevent apps from silently reading clipboard contents (which often contain passwords, 2FA codes, or sensitive text):

iOS 14: Added a transparency banner — a toast notification appears whenever an app reads the pasteboard ("App pasted from Messages"). This shamed apps that were silently reading the clipboard on every launch (TikTok, LinkedIn, and others were caught).

iOS 16: Added a permission prompt. The first time an app reads the pasteboard (via UIPasteboard.general.string or similar), the system shows a modal alert: "[App] would like to paste from [Source App]." The user must tap "Allow Paste" to proceed.

Exceptions (no prompt required):

  • The app that wrote to the pasteboard can read it back without a prompt
  • UIPasteboard.general.detectPatterns(for:) checks if the pasteboard contains certain data types (URLs, phone numbers) without reading the actual content — no prompt
  • User-initiated paste via the system paste menu (long-press → Paste) doesn't trigger the prompt

Architectural impact: Apps that relied on reading the clipboard for deep linking (e.g., "deferred deep links" via clipboard) must migrate to Universal Links, App Clips, or the system paste button (UIPasteControl, iOS 16+) which provides a system-rendered paste button that doesn't trigger the permission prompt.

Section 5: Code Signing (Questions 37–44)

L6 37. Trace the full code signing trust chain.

Code signing on iOS establishes a trust chain from Apple's root CA down to every page of your app's executable:

┌──────────────────────────────────────────────────────────┐ │ Code Signing Trust Chain │ ├──────────────────────────────────────────────────────────┤ │ │ │ Apple Root CA │ │ │ │ │ ├── Apple Worldwide Developer Relations CA │ │ │ │ │ │ │ └── Your Developer Certificate │ │ │ │ │ │ │ ├── Signs: App Binary │ │ │ ├── Signs: Frameworks │ │ │ └── Signs: Extensions │ │ │ │ │ └── Apple iPhone Certification Authority │ │ │ │ │ └── Signs: Provisioning Profile │ │ │ │ │ ├── Allowed Entitlements │ │ ├── Allowed Device UDIDs │ │ ├── App ID │ │ └── Expiration Date │ │ │ │ At Runtime: │ │ AMFI verifies: Binary signature → Certificate → │ │ Provisioning Profile → Entitlements match │ └──────────────────────────────────────────────────────────┘

Signing process:

  1. codesign computes a SHA-256 hash of every 4KB page of the executable
  2. These page hashes are stored in a CodeDirectory structure
  3. The CodeDirectory itself is hashed and signed with your developer certificate's private key, creating a CMS (Cryptographic Message Syntax) signature
  4. The signature, CodeDirectory, entitlements, and other metadata are embedded in the binary's __LINKEDIT segment
  5. The provisioning profile (embedded.mobileprovision) is included in the app bundle

Runtime verification: When the kernel loads a page into memory, it computes the page's hash and compares it to the hash in the CodeDirectory. If they don't match, the process is killed with SIGKILL (code 9). This means every page of code is verified on demand — modified code is detected immediately.

Senior 38. Compare the four provisioning profile types.

Provisioning profiles authorize an app to run on specific devices with specific capabilities:

  • Development: For local development and debugging. Contains specific device UDIDs (up to 100 per device type). Allows debugging entitlements (get-task-allow). Signed by your development certificate.
  • Ad Hoc: For testing on specific devices without Xcode. Contains specific device UDIDs (up to 100). No debugging entitlements. Signed by your distribution certificate. Used for TestFlight external testing before App Store Connect integration.
  • App Store: For App Store distribution. No device UDIDs — runs on any device. No debugging entitlements. Apple re-signs the app with an App Store certificate during submission. DRM (FairPlay) encryption is applied.
  • Enterprise (In-House): For internal distribution within an organization. No device UDIDs — runs on any device. Requires an Apple Developer Enterprise Program membership ($299/year). The certificate is valid for 3 years but must be renewed. If revoked, all apps signed with it stop launching.
Pitfall

Enterprise certificates are frequently abused for sideloading pirated apps and malware. Apple actively revokes certificates used for unauthorized distribution. If your enterprise certificate is revoked, ALL your internal apps stop working immediately — a significant business continuity risk.

L6 39. How does AMFI validate entitlements at runtime?

AMFI (Apple Mobile File Integrity) is a kernel extension that enforces code signing and entitlement policies at runtime:

Validation flow:

  1. Process launch: When posix_spawn or execve is called, AMFI hooks the exec path
  2. Signature verification: AMFI verifies the binary's CMS signature against the embedded certificate chain, tracing back to Apple's root CA
  3. Profile matching: AMFI reads the embedded.mobileprovision and verifies it's signed by Apple, not expired, and matches the app's bundle ID
  4. Entitlement validation: AMFI extracts the entitlements from the code signature and verifies they're a subset of the entitlements allowed by the provisioning profile
  5. Process tagging: The validated entitlements are attached to the process's cs_blob structure in the kernel. Subsequent entitlement checks read from this cached blob.

Runtime enforcement: When a process calls a protected API (e.g., accessing the camera), the kernel or framework checks the process's entitlements via csops() system call or SecTaskCopyValueForEntitlement(). If the required entitlement is missing, the call fails.

AMFI also enforces:

  • Library validation: Loaded dylibs must be signed by the same team ID or by Apple
  • get-task-allow: Only development-signed apps can be debugged (attached to by debugserver)
  • Dynamic code signing prohibition: Apps cannot generate and execute new code at runtime (JIT is blocked except for specific entitlements like Safari's)
Senior 40. What happens during App Store submission and review?

The App Store submission pipeline involves multiple automated and manual verification steps:

  1. Upload (Xcode/Transporter): The IPA is uploaded to App Store Connect. Apple validates the binary format, minimum OS version, required architectures, and provisioning profile.
  2. Automated processing:
    • Apple re-signs the binary with an App Store distribution certificate
    • FairPlay DRM encryption is applied to the executable
    • App thinning: Variant IPAs are generated for each device class (removing unused assets and architectures)
    • Static analysis: Automated scanners check for private API usage, deprecated APIs, and known malware patterns
    • Privacy manifest aggregation: All privacy manifests from the app and its SDKs are compiled into a privacy report
  3. App Review: Human reviewers test the app against the App Store Review Guidelines. They check functionality, content policies, privacy practices, and business model compliance.
  4. Release: Once approved, the app is available for download. The App Store serves the device-specific thinned variant.

Re-signing implications: Because Apple re-signs the binary, the code signature your build system created is replaced. This means any integrity checks that verify your own signature will fail on App Store builds. Use Apple's receipt validation instead.

L6 41. Why is enterprise distribution frequently abused?

Enterprise distribution (Apple Developer Enterprise Program) was designed for organizations to distribute internal apps to employees without going through the App Store. It's abused because:

The vulnerability: Enterprise-signed apps can run on any device — there's no UDID restriction. The only requirement is that the user manually trusts the enterprise certificate in Settings. This creates a distribution channel that bypasses App Store review entirely.

Common abuses:

  • Pirated app stores: Services like AltStore (legitimate) and various piracy platforms use enterprise certificates to distribute modified or pirated apps
  • Surveillance/spyware: Nation-state actors have used enterprise certificates to distribute spyware (e.g., the Hacking Team's RCS)
  • Gambling apps: Unlicensed gambling apps that would be rejected by App Review
  • Research abuse: Facebook and Google were caught using enterprise certificates to distribute data-collection VPN apps to consumers (violating the "internal use only" requirement)

Apple's countermeasures:

  • Active monitoring and revocation of abused certificates
  • Requiring explicit user trust in Settings (not just a tap on a link)
  • Online certificate validation — revoked certificates are blocked within hours
  • Managed Apple IDs and MDM for legitimate enterprise distribution
  • Custom App distribution via App Store Connect (apps distributed privately through the App Store)
L6 42. How does codesign create and verify signatures?

The codesign tool is the command-line interface for Apple's code signing infrastructure:

Signing process:

  1. Page hashing: The executable is divided into 4KB pages. Each page is hashed with SHA-256. These hashes form the code slots in the CodeDirectory.
  2. Special slots: Additional hashes are computed for the Info.plist (slot -1), requirements blob (slot -2), resource directory (slot -3), entitlements (slot -5), and other metadata.
  3. CodeDirectory creation: All slot hashes are assembled into a CodeDirectory structure, which also includes the team ID, signing flags, and platform identifier.
  4. CMS signature: The CodeDirectory hash is signed using the developer's private key, creating a CMS (PKCS#7) signature that includes the certificate chain.
  5. Embedding: The signature, CodeDirectory, entitlements, and requirements are embedded in the binary's __LINKEDIT segment (Mach-O) or as an extended attribute (other file types).

Verification (codesign --verify):

  1. Extract the CMS signature and verify it against the certificate chain
  2. Verify the certificate chain traces to Apple's root CA
  3. Recompute the CodeDirectory hash and compare with the signed hash
  4. Optionally verify individual page hashes (full verification)
  5. Check that the signing certificate is not revoked

At runtime, the kernel performs page-level verification on demand — each page is verified when it's paged into memory, not all at once at launch. This amortizes the verification cost.

Architect 43. Design a code signing strategy for a large team.

For a large team (50+ developers, multiple apps, CI/CD pipelines), code signing must be automated, secure, and resilient:

Certificate management:

  • Use a single distribution certificate for all production builds, stored in a hardware security module (HSM) or secure CI/CD secret store
  • Use individual development certificates for each developer (auto-managed by Xcode)
  • Rotate certificates before expiration with a 30-day overlap period
  • Never share certificate private keys via email or Slack — use a secure vault

Provisioning profile management:

  • Use Xcode Automatic Signing for development builds
  • Use manual signing for CI/CD and distribution builds with profiles managed via Fastlane Match or Apple's API
  • Store profiles in a private Git repository (Fastlane Match pattern) encrypted with a shared passphrase

CI/CD pipeline:

Senior 44. What is notarization and how does it differ from code signing?

Notarization is a macOS concept (not iOS), but understanding it is valuable for cross-platform teams and Mac Catalyst apps:

Code signing: Cryptographically binds the binary to a developer identity. Proves the code hasn't been modified since signing. Happens locally using your certificate.

Notarization: An additional step where you upload the signed binary to Apple's notary service. Apple scans it for malware, validates the signature, and issues a notarization ticket (a signed statement that Apple has inspected the software). The ticket is stapled to the binary.

Key differences:

  • Who validates: Code signing is self-asserted (your certificate). Notarization is Apple-asserted (Apple's servers scan and approve).
  • What it checks: Code signing verifies integrity. Notarization additionally checks for known malware, validates hardened runtime, and ensures no private entitlements are used.
  • Revocability: Apple can revoke a notarization ticket if malware is discovered post-approval, immediately blocking the software on all Macs.
  • Requirement: macOS Catalina+ requires notarization for all software distributed outside the Mac App Store. iOS apps distributed through the App Store are effectively "notarized" during App Review.

For Mac Catalyst apps, you need both: code signing for identity and integrity, plus notarization for Apple's malware scan and Gatekeeper approval.

Section 6: Runtime Security (Questions 45–52)

Senior 45. How does ASLR protect against exploitation?

Address Space Layout Randomization (ASLR) randomizes the memory addresses where code and data are loaded, making exploitation significantly harder:

What's randomized:

  • Executable base address: The main binary's load address is randomized (requires PIE — Position Independent Executable, mandatory on iOS)
  • Shared libraries: dyld loads frameworks at random addresses via the shared cache slide
  • Stack: The stack base address is randomized per-thread
  • Heap: Heap allocations have randomized base addresses
  • dyld shared cache: The entire shared cache (all system frameworks) is slid by a random offset at boot

How it prevents exploitation: Most exploits need to know the address of specific functions (e.g., system()) or gadgets (for ROP chains). With ASLR, these addresses change every launch. An attacker who finds a vulnerability (e.g., buffer overflow) can corrupt memory but can't reliably redirect execution to a known address.

Limitations: ASLR is defeated by information disclosure vulnerabilities that leak memory addresses. A single leaked pointer reveals the ASLR slide for that region, allowing the attacker to compute all other addresses. This is why modern exploits typically chain an info leak with a memory corruption bug.

iOS specifics: iOS uses a 64-bit address space with high entropy ASLR (~16 bits of entropy for the shared cache slide, more for the executable). Combined with PAC (A12+), the effective entropy is much higher.

L6 46. Explain PAC (Pointer Authentication Codes) on A12+.

Pointer Authentication Codes (PAC), introduced with the A12 chip, add cryptographic signatures to pointers, making it extremely difficult to forge or corrupt code and data pointers:

How PAC works:

  1. ARM64e extends the instruction set with pointer signing and authentication instructions
  2. When a pointer is stored (e.g., a return address pushed to the stack), the CPU computes a PAC using: the pointer value, a context (e.g., the stack pointer), and a key (stored in a special register, per-process)
  3. The PAC (a few bits) is stored in the unused upper bits of the 64-bit pointer (bits that would otherwise be zero in the virtual address space)
  4. Before the pointer is used (e.g., returning from a function), the CPU authenticates it: recomputes the PAC and compares. If they don't match, the pointer is corrupted to an invalid address, causing a crash on use.

What PAC protects:

  • Return addresses: Signed with PACIASP on function entry, authenticated with RETAB on return. Prevents ROP (Return-Oriented Programming) attacks.
  • Function pointers: vtable entries, block invoke pointers, and other indirect call targets are signed
  • Data pointers: Objective-C isa pointers and other critical data pointers are signed

Why it's powerful: Even if an attacker can overwrite a pointer in memory, they can't forge a valid PAC without knowing the key (hardware-protected, per-process). The context binding means a PAC for one call site can't be reused at another — each pointer is authenticated in its specific context.

Limitations: PAC uses a limited number of bits (~7-16 depending on virtual address space usage), so brute-force is theoretically possible but requires many attempts, each of which crashes the process. Combined with ASLR, the effective security is very high.

Senior 47. How do stack canaries detect buffer overflows?

Stack canaries (also called stack cookies or stack protectors) are random values placed between local variables and the saved return address on the stack:

How they work:

  1. At function entry, the compiler inserts code to place a random canary value on the stack between the local variables and the saved frame pointer / return address
  2. The canary value is generated at process startup and stored in a thread-local variable
  3. At function exit (before the return instruction), the compiler inserts code to compare the stack canary with the stored reference value
  4. If they don't match, a buffer overflow has corrupted the stack. The process calls __stack_chk_fail and aborts immediately (before the corrupted return address is used)

Compiler flags:

  • -fstack-protector: Protects functions with character arrays (basic)
  • -fstack-protector-strong: Protects functions with any local arrays or address-taken variables (default on iOS)
  • -fstack-protector-all: Protects all functions (higher overhead)

Limitations: Canaries only detect sequential buffer overflows that overwrite the canary on the way to the return address. They don't protect against: non-sequential overwrites (e.g., index-based out-of-bounds), heap overflows, or overwrites of other local variables that don't cross the canary. PAC provides stronger return address protection.

L6 48. Compare Swift vs ObjC runtime security.

Swift is fundamentally more secure than Objective-C at the language and runtime level:

Memory safety:

  • Swift: Bounds checking on all array accesses. Optional types prevent null pointer dereferences. ARC is deterministic with no manual memory management. Value types (structs, enums) are stack-allocated, reducing heap corruption risk.
  • ObjC: No bounds checking on C arrays. Nil messaging silently returns zero (masks bugs). Manual retain/release is error-prone (use-after-free, double-free). Heavy use of heap-allocated objects.

Type safety:

  • Swift: Strong static typing. No implicit type conversions. Exhaustive switch statements. The compiler catches type errors at compile time.
  • ObjC: Dynamic typing via id. Message dispatch can send any message to any object. Unrecognized selectors crash at runtime, not compile time.

Runtime attack surface:

  • Swift: Static dispatch for structs and final classes (no vtable to corrupt). Protocol witness tables are in read-only memory. Limited runtime metadata exposure.
  • ObjC: Dynamic dispatch via objc_msgSend. Method swizzling allows runtime method replacement. The isa pointer, method lists, and class metadata are all in writable memory (partially mitigated by __DATA_CONST). An attacker who can write to memory can redirect any method call.

Reverse engineering resistance: Swift binaries have less metadata exposure than ObjC (no readable method names for non-@objc methods). However, Swift's type metadata and protocol conformance tables still reveal significant structural information.

L6 49. What is __DATA_CONST and why does it matter?

__DATA_CONST is a Mach-O segment that contains data which is writable during process initialization but becomes read-only after dyld finishes loading:

What lives in __DATA_CONST:

  • Objective-C class metadata (objc_class structures)
  • Protocol conformance tables
  • Method lists (after fixup)
  • Swift type metadata
  • C++ vtables
  • GOT (Global Offset Table) entries after binding

Why it matters for security: Before __DATA_CONST, all this metadata lived in the writable __DATA segment. An attacker with a write primitive could modify an ObjC class's method list to redirect method calls (a technique called "isa swizzling" or "class patching"). With __DATA_CONST, the memory is marked read-only after initialization using mprotect(), so write attempts cause a crash.

The initialization window: During dyld loading, __DATA_CONST must be writable for pointer fixups (rebasing and binding). After all libraries are loaded and initialized, dyld calls mprotect(PROT_READ) to make it read-only. This creates a brief window during launch where the data is writable — a potential attack vector for very sophisticated exploits.

Relation to other protections: __DATA_CONST complements W^X (code pages are never writable) by also protecting critical data pages. Together, they significantly reduce the attack surface for memory corruption exploits.

L6 50. How does W^X enforcement prevent code injection?

W^X (Write XOR Execute) is a memory protection policy that ensures no memory page is simultaneously writable and executable. On iOS, this is enforced by the kernel:

The principle: A memory page can be either writable (W) or executable (X), but never both at the same time. This prevents an attacker from writing shellcode to memory and then executing it.

iOS enforcement:

  • Code pages (__TEXT segment) are mapped as r-x (read + execute, no write)
  • Data pages (__DATA segment) are mapped as rw- (read + write, no execute)
  • The kernel rejects mmap() and mprotect() calls that request both PROT_WRITE and PROT_EXEC
  • Even with a kernel exploit, changing page permissions requires bypassing AMFI's code signing enforcement

Exceptions:

  • JavaScriptCore JIT: Safari's JavaScript engine needs to generate and execute code at runtime. It has a special entitlement (dynamic-codesigning) that allows it to create RWX pages. This is one of the most attacked surfaces on iOS.
  • No third-party JIT: Third-party apps cannot request this entitlement. This is why alternative browser engines (Chromium's V8) cannot use JIT on iOS and must use the system's JavaScriptCore.

Attack implications: Without W^X bypass, attackers must use code reuse attacks (ROP/JOP) — chaining existing code snippets (gadgets) to achieve their goal. PAC makes even this significantly harder.

L6 51. What is MTE and why hasn't Apple adopted it?

Memory Tagging Extension (MTE) is an ARM hardware feature (ARMv8.5-A) that detects memory safety violations by tagging memory allocations:

How MTE works:

  1. Each 16-byte granule of memory is assigned a 4-bit tag (stored in dedicated tag memory)
  2. Pointers also carry a 4-bit tag in their upper bits
  3. On every memory access, the hardware compares the pointer's tag with the memory's tag
  4. If they don't match, a fault is raised — detecting use-after-free (freed memory gets a new tag), buffer overflow (adjacent allocation has a different tag), and other spatial/temporal memory errors

Why Apple hasn't adopted MTE:

  • PAC is Apple's chosen approach: Apple invested heavily in PAC (Pointer Authentication) starting with A12. PAC protects control flow (return addresses, function pointers) with cryptographic signatures, which is arguably stronger than MTE's probabilistic tag matching for the attack vectors Apple prioritizes.
  • Different threat model: MTE is excellent for detecting bugs during development (like ASan but in hardware). Apple's focus is on preventing exploitation of bugs that ship — PAC directly hardens the exploitation primitives (pointer corruption).
  • Memory overhead: MTE requires dedicated tag storage (3-6% memory overhead). On memory-constrained mobile devices, this is significant.
  • Probabilistic: With 4-bit tags, there's a 1/16 chance of a tag collision (the wrong tag happens to match). PAC provides stronger guarantees.
  • Swift: Apple's long-term strategy is memory-safe languages (Swift), which eliminate the bug classes MTE detects. MTE is most valuable for C/C++ code.

Google has adopted MTE in Android 14 (on Pixel 8 with Tensor G3), using it for both bug detection and exploit mitigation. The different approaches reflect different ecosystem priorities.

Architect 52. Design a security architecture for a banking app.

A banking app requires the highest security posture on iOS. Here's a comprehensive architecture:

Authentication layer:

  • Biometric authentication (Face ID / Touch ID) via Secure Enclave-backed Keychain items with .biometryCurrentSet
  • Device-bound asymmetric key pair in the Secure Enclave for mutual TLS and transaction signing
  • Step-up authentication for high-value transactions (re-authenticate with biometric + PIN)
  • Session tokens with short TTL (15 minutes), stored in Keychain with kSecAttrAccessibleWhenUnlockedThisDeviceOnly

Transport security:

  • Certificate pinning with public key pins (current + backup) and remote pin update
  • Mutual TLS (mTLS) — the client presents its SE-backed certificate to the server
  • TLS 1.3 only, no fallback to 1.2
  • All API responses signed by the server; client verifies signatures

Data protection:

// Sensitive data encryption with application-level AES-GCM
actor SecureStorage {
    private let encryptionKey: SymmetricKey

    init() throws {
        // Derive key from SE-backed ECDH + server key
        let seKey = try SecureEnclave.P256.KeyAgreement.PrivateKey()
        let shared = try seKey.sharedSecretFromKeyAgreement(
            with: serverPublicKey
        )
        self.encryptionKey = shared.hkdfDerivedSymmetricKey(
            using: SHA256.self,
            salt: deviceSalt,
            sharedInfo: Data("BankApp-Storage-v1".utf8),
            outputByteCount: 32
        )
    }

    func store(_ data: Data, forKey key: String) throws {
        let sealed = try AES.GCM.seal(data, using: encryptionKey)
        try sealed.combined!.write(to: fileURL(forKey: key),
            options: .completeFileProtection)
    }
}

Runtime integrity:

  • App Attest (DCAppAttestService) for server-side binary integrity verification
  • Environment anomaly detection (debugger, proxy, unusual frameworks) as risk signals — not hard blocks
  • Server-side risk engine that combines device signals, behavioral analytics, and transaction patterns
