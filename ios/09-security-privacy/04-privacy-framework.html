1. Privacy Manifests (iOS 17)

Introduced at WWDC 2023, Privacy Manifests are Apple's mechanism for making an app's data collection practices machine-readable and auditable. Every framework — first-party or third-party — declares exactly which privacy-sensitive APIs it uses and why, via a PrivacyInfo.xcprivacy file. Xcode aggregates these declarations across all dependencies into a single privacy report at build time.

1.1 What Is a Privacy Manifest?

A privacy manifest is a property list file named PrivacyInfo.xcprivacy placed at the root of a framework bundle or an app target. It contains four top-level keys:

Key Type Purpose
NSPrivacyTracking Boolean Whether the SDK uses data for tracking as defined by ATT
NSPrivacyTrackingDomains [String] Domains the SDK contacts for tracking — blocked when user denies ATT
NSPrivacyCollectedDataTypes [Dictionary] Data types collected, mapped to nutrition label categories
NSPrivacyAccessedAPITypes [Dictionary] Required-reason APIs used, with approved reason codes

1.2 Required Reason APIs

Apple identified specific APIs that can be used for fingerprinting. If your code (or any dependency) calls these APIs, you must declare an approved reason in the manifest. Using them without a declared reason triggers App Store rejection.

API Category Examples Fingerprinting Risk
File timestamp APIs NSFileCreationDate, NSFileModificationDate, stat(), getattrlist() Unique file timestamps can identify a device
System boot time systemUptime, mach_absolute_time(), sysctl(KERN_BOOTTIME) Boot time is a stable device identifier
Disk space APIs volumeAvailableCapacityKey, FileManager.attributesOfFileSystem Exact disk space is highly unique
Active keyboards UITextInputMode.activeInputModes Keyboard language set is a fingerprint signal
User defaults UserDefaults.standard Cross-SDK data sharing via standard suite
Pitfall — UserDefaults Is a Required Reason API

Almost every app uses UserDefaults, which means almost every app needs a privacy manifest entry for it. The approved reason CA92.1 covers "accessing user defaults to read/write data the app itself wrote." If you access defaults written by another app or SDK via a shared app group, you need additional reason codes. Many developers missed this during the initial rollout.

2. ATTrackingManager & App Tracking Transparency

App Tracking Transparency (ATT), introduced in iOS 14.5, is Apple's most impactful privacy change. It requires apps to obtain explicit user permission before tracking them across apps and websites owned by other companies. The framework is AppTrackingTransparency and the central class is ATTrackingManager.

2.1 What Constitutes "Tracking"?

Apple defines tracking as linking user or device data collected from your app with user or device data collected from other companies' apps, websites, or offline properties for advertising or data broker purposes. Specifically:

  • Displaying targeted ads based on data collected from apps and websites owned by other companies
  • Sharing device location data or email lists with a data broker
  • Sharing a list of emails, advertising IDs, or other identifiers with a third-party ad network that uses them to retarget users in other apps
  • Placing a third-party SDK in your app that combines user data from your app with data from other developers' apps to target advertising or measure advertising efficiency (even if you don't use the SDK for these purposes yourself)

Not tracking (does not require ATT):

  • Linking data to a user's account solely for your own first-party purposes (e.g., analytics for your own app)
  • Linking data on the device that is not sent off-device in a way that can identify the user
  • Sharing data with a data broker solely for fraud detection, prevention, or security

2.2 Authorization Statuses

Status Value Meaning IDFA Available?
.notDetermined 0 User has not been prompted yet No (all zeros)
.restricted 1 Device-level restriction (MDM, parental controls, or child account) No
.denied 2 User explicitly denied permission No
.authorized 3 User explicitly granted permission Yes

2.3 Full ATT Implementation

import AppTrackingTransparency
import AdSupport

final class TrackingPermissionManager {

    static let shared = TrackingPermissionManager()

    var currentStatus: ATTrackingManager.AuthorizationStatus {
        ATTrackingManager.trackingAuthorizationStatus
    }

    var idfa: String? {
        guard currentStatus == .authorized else { return nil }
        let idfa = ASIdentifierManager.shared().advertisingIdentifier.uuidString
        guard idfa != "00000000-0000-0000-0000-000000000000" else {
            return nil
        }
        return idfa
    }

    /// Request tracking authorization after a brief delay post-launch.
    /// Must be called when app is in .active state (not during launch).
    func requestPermission() async -> ATTrackingManager.AuthorizationStatus {
        guard currentStatus == .notDetermined else {
            return currentStatus
        }

        let status = await ATTrackingManager.requestTrackingAuthorization()

        switch status {
        case .authorized:
            enableThirdPartyTracking()
            configureAdNetworks(withIDFA: true)
        case .denied, .restricted:
            configureAdNetworks(withIDFA: false)
            enableSKAdNetworkAttribution()
        case .notDetermined:
            break // Should not happen after request
        @unknown default:
            configureAdNetworks(withIDFA: false)
        }

        return status
    }

    private func enableThirdPartyTracking() { /* ... */ }
    private func configureAdNetworks(withIDFA: Bool) { /* ... */ }
    private func enableSKAdNetworkAttribution() { /* ... */ }
}

2.4 Timing & Pre-Prompt Strategy

Pitfall — Calling requestTrackingAuthorization Too Early

If you call requestTrackingAuthorization() before the app reaches the .active scene phase, the system silently returns .notDetermined without showing the prompt. The prompt also won't appear if the app is in the background. Always wait for UIApplication.didBecomeActiveNotification or use scenePhase == .active in SwiftUI.

The pre-prompt strategy is essential for maximizing opt-in rates. Since Apple's system dialog is terse and non-customizable, show a custom screen first that explains the value exchange:

  1. Wait for a meaningful moment — after the user has experienced value from the app (not on first launch)
  2. Show a custom pre-prompt — explain why you're asking and what the user gets (e.g., "relevant recommendations" or "free access supported by ads")
  3. If user taps "Continue" — show the system ATT dialog
  4. If user taps "Not Now" — respect the choice and ask again later (you can re-show your pre-prompt, but the system dialog only appears once)

2.5 Impact on the Advertising Ecosystem

ATT fundamentally reshaped mobile advertising. Industry data shows opt-in rates of roughly 25–35% globally (varying by app category and region). This means ~70% of iOS users are invisible to cross-app tracking. The consequences:

  • IDFA deprecation in practice — most users return all-zeros IDFA, making deterministic cross-app matching impossible
  • SKAdNetwork adoption — Apple's privacy-preserving attribution became the primary iOS install attribution mechanism
  • Server-side attribution rise — platforms like Meta's Conversions API and Google's server-side tagging moved attribution logic server-side
  • Contextual advertising resurgence — targeting based on content context rather than user identity
  • First-party data premium — apps with direct user relationships (login, email) gained a significant competitive advantage

3. SKAdNetwork for Attribution

SKAdNetwork is Apple's privacy-preserving framework for measuring ad campaign effectiveness without revealing user-level data. It allows ad networks to attribute app installs to specific campaigns while keeping the user anonymous. The framework sits in StoreKit and operates entirely through Apple's servers as an intermediary.

3.2 Conversion Values

The advertised app can update a conversion value — a 6-bit integer (0–63) — to signal post-install events to the ad network. The ad network never learns which user performed the action, only that some user attributed to a campaign reached that conversion value.

import StoreKit

func updateConversionValue(event: PostInstallEvent) {
    let fineValue: Int
    let coarseValue: SKAdNetwork.CoarseConversionValue

    switch event {
    case .appLaunch:
        fineValue = 0
        coarseValue = .low
    case .registration:
        fineValue = 10
        coarseValue = .medium
    case .purchase(let revenue):
        fineValue = mapRevenueToFineValue(revenue) // 0–63
        coarseValue = revenue > 50 ? .high : .medium
    case .subscription:
        fineValue = 63
        coarseValue = .high
    }

    SKAdNetwork.updatePostbackConversionValue(
        fineValue,
        coarseValue: coarseValue,
        lockWindow: false
    ) { error in
        if let error {
            print("SKAdNetwork update failed: \(error)")
        }
    }
}

3.3 SKAdNetwork 4.0 Changes

Feature SKAN 3.0 SKAN 4.0
Conversion values 6-bit fine value only 6-bit fine + coarse (low/medium/high)
Postbacks Single postback Up to 3 postbacks (0–2 days, 3–7 days, 8–35 days)
Crowd anonymity Not applicable 4 tiers — more installs = more data in postback
Source identifier 2-digit campaign ID (0–99) 4-digit hierarchical (2/3/4 digits based on crowd anonymity)
Web-to-app App-to-app only Safari Private Click Measurement integration
Deep Dive — Crowd Anonymity Tiers

SKAN 4.0 introduces crowd anonymity: the more installs a campaign generates, the more granular data Apple includes in the postback. At the lowest tier (few installs), you only get the coarse conversion value and a 2-digit source ID. At the highest tier (many installs), you get the fine value, all 4 source ID digits, and the source app ID. This prevents re-identification of users in low-volume campaigns.

4.3 Linked vs Not Linked to Identity

For each data type, you must declare whether it is:

  • Used to Track You — linked with third-party data for advertising/data broker purposes (requires ATT)
  • Linked to You — associated with your identity (user account, name, email, etc.)
  • Not Linked to You — collected but not associated with your identity (e.g., anonymized analytics)
Best Practice — Accurate Nutrition Labels
  • Audit every third-party SDK for data collection — their data practices become your data practices
  • Use the Xcode privacy report (from Privacy Manifests) to cross-reference your labels
  • Update labels with every app update — Apple can reject if labels don't match actual behavior
  • When in doubt, over-declare rather than under-declare — Apple has rejected apps for missing disclosures
  • Document your data inventory internally — map every data flow from collection to deletion

5. PHPhotoLibrary Limited Access Mode

iOS 14 introduced Limited Photo Library Access, giving users granular control over which photos an app can see. Instead of the binary "all or nothing" permission of earlier iOS versions, users can now select specific photos and videos to share with an app while keeping the rest of their library private.

5.1 Access Levels

PHAccessLevel Authorization Status Options Use Case
.addOnly .notDetermined, .restricted, .denied, .authorized Camera apps that save photos but don't browse the library
.readWrite .notDetermined, .restricted, .denied, .authorized, .limited Photo editors, social media apps that need to read photos

5.2 PHPickerViewController — The Privacy-First Picker

PHPickerViewController (iOS 14+) runs in a separate process outside your app's sandbox. It can access the full photo library regardless of your app's authorization status. The user selects photos in the picker, and only the selected items are delivered to your app. This is the recommended approach for most apps.

import PhotosUI

final class PhotoPickerManager: NSObject, PHPickerViewControllerDelegate {

    func presentPicker(from viewController: UIViewController) {
        var config = PHPickerConfiguration(photoLibrary: .shared())
        config.selectionLimit = 10
        config.filter = .any(of: [.images, .livePhotos])
        config.preferredAssetRepresentationMode = .current

        let picker = PHPickerViewController(configuration: config)
        picker.delegate = self
        viewController.present(picker, animated: true)
    }

    func picker(
        _ picker: PHPickerViewController,
        didFinishPicking results: [PHPickerResult]
    ) {
        picker.dismiss(animated: true)

        for result in results {
            let provider = result.itemProvider
            if provider.canLoadObject(ofClass: UIImage.self) {
                provider.loadObject(ofClass: UIImage.self) { image, error in
                    guard let image = image as? UIImage else { return }
                    DispatchQueue.main.async {
                        self.handleSelectedImage(image)
                    }
                }
            }
        }
    }

    private func handleSelectedImage(_ image: UIImage) { /* ... */ }
}

Pitfall — Suppressing the "Select More Photos" Banner

When the user has granted .limited access, iOS shows a system banner prompting them to select more photos every time your app accesses the library. To suppress this, add PHPhotoLibraryPreventAutomaticLimitedAccessAlert = YES to your Info.plist. If you suppress it, you must provide your own UI to let the user modify their selection via presentLimitedLibraryPicker(from:).

6. Location Approximate vs Precise Mode

iOS 14 introduced approximate location, allowing users to share only a rough location (~10 km region) instead of their precise GPS coordinates. This is controlled independently from the "when-in-use" vs "always" authorization, giving users a 2×2 matrix of location permissions.

6.1 Authorization Matrix

Precise (Full Accuracy) Approximate (Reduced Accuracy)
When In Use GPS-level accuracy while app is in foreground ~10 km region center while app is in foreground
Always GPS-level accuracy at all times ~10 km region center at all times

6.2 CLAccuracyAuthorization

import CoreLocation

final class LocationPrivacyManager: NSObject, CLLocationManagerDelegate {

    private let manager = CLLocationManager()

    override init() {
        super.init()
        manager.delegate = self
    }

    func checkAccuracyAuthorization() {
        switch manager.accuracyAuthorization {
        case .fullAccuracy:
            // User granted precise location
            manager.desiredAccuracy = kCLLocationAccuracyBest

        case .reducedAccuracy:
            // User chose approximate location.
            // CLLocation.coordinate returns the center of a ~10 km region.
            // CLLocation.horizontalAccuracy is very large (thousands of meters).
            adaptUIForApproximateLocation()

        @unknown default:
            break
        }
    }

    /// Request temporary full accuracy for a specific feature.
    /// Requires a matching key in Info.plist under NSLocationTemporaryUsageDescriptionDictionary.
    func requestTemporaryFullAccuracy(purposeKey: String) {
        manager.requestTemporaryFullAccuracyAuthorization(
            withPurposeKey: purposeKey
        ) { [weak self] error in
            if let error {
                print("Temporary accuracy request failed: \(error)")
                return
            }
            if self?.manager.accuracyAuthorization == .fullAccuracy {
                self?.performPreciseLocationOperation()
            }
        }
    }

    func locationManagerDidChangeAuthorization(_ manager: CLLocationManager) {
        checkAccuracyAuthorization()
    }

    private func adaptUIForApproximateLocation() { /* ... */ }
    private func performPreciseLocationOperation() { /* ... */ }
}

6.3 Info.plist Configuration for Temporary Full Accuracy

<key>NSLocationTemporaryUsageDescriptionDictionary</key>
<dict>
    <key>NavigationPurpose</key>
    <string>We need your precise location for turn-by-turn navigation.</string>
    <key>NearbySearchPurpose</key>
    <string>Precise location helps find restaurants near you.</string>

Deep Dive — How Approximate Location Works Internally

When the user grants approximate location, Core Location doesn't simply degrade GPS accuracy. It computes the user's precise location, then maps it to the center of a pre-defined ~10 km² region. The regions are fixed geographic cells — they don't move with the user. This means the returned coordinate "snaps" to a grid center, and the user's actual position could be anywhere within that cell. The horizontalAccuracy value reflects the cell radius (typically 3000–5000 meters). The region boundaries are the same for all apps, preventing cross-app correlation of slightly different approximate locations.

Best Practice — Designing for Approximate Location
  • Weather apps — approximate location is sufficient; don't request precise
  • Maps/navigation — request temporary full accuracy only when the user initiates navigation
  • Social apps — show city-level location by default; request precise only for "share exact location" features
  • Always check accuracyAuthorization before assuming precise coordinates — even if the user granted "Always" authorization, they may have chosen approximate
  • Adapt your UI — show a larger map region, use city names instead of street addresses, hide "distance to" calculations when approximate

7.2 Handling Permission Denial Gracefully

import AVFoundation

final class CameraPermissionHandler {

    enum PermissionResult {
        case granted
        case denied
        case restricted
    }

    func requestCameraAccess() async -> PermissionResult {
        switch AVCaptureDevice.authorizationStatus(for: .video) {
        case .authorized:
            return .granted

        case .notDetermined:
            let granted = await AVCaptureDevice.requestAccess(for: .video)
            return granted ? .granted : .denied

        case .denied:
            return .denied

        case .restricted:
            return .restricted

        @unknown default:
            return .denied
        }
    }

    func openAppSettings() {
        guard let url = URL(string: UIApplication.openSettingsURLString),
              UIApplication.shared.canOpenURL(url) else { return }
        UIApplication.shared.open(url)
    }
}
Pitfall — Missing Usage Description Causes Crash, Not Rejection

If you call a privacy-sensitive API without the corresponding Info.plist key, your app crashes at runtime — it doesn't fail gracefully. During development, this crash may be caught, but if a code path is rarely exercised, it can slip into production. Always audit your Info.plist against every framework you link. Note: App Review may also reject your app if the usage description string is too vague (e.g., "We need camera access" without explaining why).

8. Pasteboard Access Changes

Starting with iOS 14, the system shows a banner notification whenever an app reads the pasteboard, alerting users to potential clipboard snooping. iOS 16 went further by requiring explicit user consent for paste operations via UIPasteControl.

8.1 Evolution of Pasteboard Privacy

iOS Version Behavior
iOS 13 and earlier Apps can silently read the pasteboard at any time — no user notification

8.2 UIPasteControl

UIPasteControl is a system-rendered button that, when tapped by the user, grants your app one-time access to the pasteboard without showing the confirmation dialog. It's the recommended way to implement paste functionality in iOS 16+.

import UIKit

final class PasteViewController: UIViewController, UIPasteConfigurationSupporting {

    var pasteConfiguration: UIPasteConfiguration? = UIPasteConfiguration(
        acceptableTypeIdentifiers: ["public.utf8-plain-text"]
    )

    private lazy var pasteControl: UIPasteControl = {
        var config = UIPasteControl.Configuration()
        config.displayMode = .iconAndLabel
        config.cornerStyle = .capsule
        config.baseForegroundColor = .systemBlue
        config.baseBackgroundColor = .secondarySystemBackground

        let control = UIPasteControl(configuration: config)
        control.target = self
        return control
    }()

    func paste(itemProviders: [NSItemProvider]) {
        for provider in itemProviders {
            provider.loadObject(ofClass: String.self) { [weak self] text, error in
                guard let text else { return }
                DispatchQueue.main.async {
                    self?.handlePastedText(text)
                }
            }
        }
    }

    private func handlePastedText(_ text: String) { /* ... */ }
}

8.3 Detecting Pasteboard Content Without Triggering the Banner

iOS 14+ provides type detection methods that check whether the pasteboard contains specific content types without actually reading the content — and therefore without triggering the paste banner:

import UIKit

func checkPasteboardContentTypes() {
    UIPasteboard.general.detectPatterns(
        for: [.probableWebURL, .number, .probableWebSearch]
    ) { result in
        switch result {
        case .success(let detectedPatterns):
            if detectedPatterns.contains(.probableWebURL) {
                // Show a "Paste link" button — reading the actual URL
                // requires user action (UIPasteControl or paste: action)
                self.showPasteLinkButton()
            }
        case .failure(let error):
            print("Pattern detection failed: \(error)")
        }
    }
}

private func showPasteLinkButton() { /* ... */ }
Best Practice — Pasteboard Access
  • Use UIPasteControl for all paste operations in iOS 16+ — it provides the best UX without the confirmation dialog
  • Use detectPatterns(for:) to check content types without reading data — show contextual paste buttons only when relevant content is detected
  • Never read the pasteboard on app launch — this was a common pattern for deep linking that now triggers the banner and erodes user trust
  • Migrate from UIPasteboard.general.string to the NSItemProvider-based API for forward compatibility

9. Architect: Privacy-First Analytics System

Architect Must-Know — System Design

At L6/Staff level, you're expected to design analytics and attribution systems that work within Apple's privacy constraints. This section covers a complete privacy-first analytics architecture — the kind of system design question you'll face in a Staff-level interview.

9.1 The Problem

With ATT opt-in rates around 25–35%, IDFA is effectively dead for most users. You need an analytics and attribution system that:

  • Measures user engagement, retention, and conversion without cross-app tracking
  • Supports A/B testing without linking user identity across sessions (for non-logged-in users)
  • Complies with GDPR, CCPA, and Apple's privacy requirements simultaneously
  • Provides actionable data to product and marketing teams
  • Degrades gracefully as privacy restrictions tighten

9.3 First-Party Identifiers

Without IDFA, you need a first-party identity strategy. The key principle: identifiers should be scoped to your app and never shared with third parties.

Identifier Scope Persistence Privacy Impact

9.4 Differential Privacy on Device

Differential privacy adds calibrated noise to data before it leaves the device, providing mathematical guarantees that individual user behavior cannot be inferred from the aggregate data. Apple uses this technique in their own telemetry (keyboard predictions, Health trends, Safari suggestions).

import Foundation

struct DifferentialPrivacyEncoder {

    let epsilon: Double  // Privacy budget — lower = more private

    /// Randomized response for boolean values.
    /// With probability p = e^ε / (e^ε + 1), report the true value.
    /// With probability 1-p, report a random value.
    func randomizedResponse(trueValue: Bool) -> Bool {
        let p = exp(epsilon) / (exp(epsilon) + 1.0)
        let coin = Double.random(in: 0...1)

        if coin < p {
            return trueValue
        } else {
            return Bool.random()
        }
    }

    /// Add Laplace noise to a numeric value.
    func addLaplaceNoise(value: Double, sensitivity: Double) -> Double {
        let scale = sensitivity / epsilon
        let u = Double.random(in: 0...1) - 0.5
        let noise = -scale * copysign(1.0, u) * log(1.0 - 2.0 * abs(u))
        return value + noise
    }
}

// Usage: encode whether user completed onboarding
let encoder = DifferentialPrivacyEncoder(epsilon: 2.0)
let noisyValue = encoder.randomizedResponse(trueValue: true)
// Server aggregates many noisy values to recover approximate true distribution

9.5 On-Device Aggregation

Instead of sending raw events, aggregate data on-device before transmission. This reduces both bandwidth and privacy exposure:

final class OnDeviceAggregator {

    struct SessionSummary: Codable {
        let sessionDurationBucket: String  // "0-30s", "30s-2m", "2m-10m", "10m+"
        let screenViewCount: Int
        let featureUsageBitmap: UInt32    // Bit flags for feature usage
        let errorOccurred: Bool
        let appVersion: String
        let osVersion: String
        let deviceCategory: String       // "iPhone", "iPad" — not model
    }

    private var events: [RawEvent] = []

    func record(_ event: RawEvent) {
        events.append(event)
    }

    func summarize() -> SessionSummary {
        let duration = events.last?.timestamp
            .timeIntervalSince(events.first?.timestamp ?? .now) ?? 0

        return SessionSummary(
            sessionDurationBucket: bucketize(duration),
            screenViewCount: events.filter { $0.type == .screenView }.count,
            featureUsageBitmap: computeFeatureBitmap(),
            errorOccurred: events.contains { $0.type == .error },
            appVersion: Bundle.main.shortVersionString,
            osVersion: ProcessInfo.processInfo.operatingSystemVersionString,
            deviceCategory: UIDevice.current.userInterfaceIdiom == .pad ? "iPad" : "iPhone"
        )
    }

    private func bucketize(_ seconds: TimeInterval) -> String {
        switch seconds {
        case 0..<30:    return "0-30s"
        case 30..<120:  return "30s-2m"
        case 120..<600: return "2m-10m"
        default:        return "10m+"
        }
    }

    private func computeFeatureBitmap() -> UInt32 { /* ... */ return 0 }
}

9.6 Privacy-Preserving A/B Testing

Traditional A/B testing assigns users to cohorts using a stable identifier and tracks conversion events. In a privacy-first world, the approach changes:

Aspect Traditional A/B Testing Privacy-Preserving A/B Testing
Assignment Server-side, based on user ID or IDFA On-device, based on deterministic hash of app-scoped ID + experiment name
Event tracking Individual user events sent to server Aggregated experiment metrics sent (e.g., "variant A: 47 conversions out of 200 exposures")
Analysis Per-user funnel analysis Aggregate statistical tests (chi-squared, t-test on bucketed data)
Segmentation Rich user-level segments Device-category segments only (no PII-based segments) 
struct PrivacyPreservingExperiment {

    let experimentName: String
    let variants: [String]

    /// Deterministic, on-device assignment using app-scoped ID.
    /// No server call needed — assignment is reproducible.
    func assignVariant(appScopedID: String) -> String {
        let input = "\(experimentName):\(appScopedID)"
        let hash = input.utf8.reduce(0) { (31 &* $0) &+ Int($1) }
        let index = abs(hash) % variants.count
        return variants[index]
    }
}

// On-device aggregation of experiment results
struct ExperimentResult: Codable {
    let experimentName: String
    let variant: String
    let exposureCount: Int
    let conversionCount: Int
    // No user ID transmitted — only aggregate counts
}

9.7 Server-Side Attribution

With IDFA unavailable, attribution moves server-side. The pattern involves matching install events with ad impressions using privacy-safe signals:

  • SKAdNetwork postbacks — Apple-mediated, privacy-preserving (see Section 3)
  • Conversion APIs — server-to-server event sharing (Meta CAPI, Google Ads API) using hashed first-party data (email, phone) that the user provided directly to you
  • Modeled conversions — ML models that estimate total conversions from the partial data available (Google's consent mode, Meta's Aggregated Event Measurement)
  • Media mix modeling — statistical analysis of spend vs. outcomes at the campaign level, without any user-level data

9.8 Consent Management Architecture

final class ConsentManager {

    enum ConsentPurpose: String, CaseIterable {
        case analytics
        case advertising
        case personalization
        case functionalCookies
    }

    enum ConsentStatus {
        case notDetermined
        case granted
        case denied
    }

    private let store: UserDefaults
    private let consentVersion = 2  // Bump to re-prompt on policy change

    init(store: UserDefaults = .standard) {
        self.store = store
    }

    var needsConsentPrompt: Bool {
        let lastVersion = store.integer(forKey: "consent_version")
        return lastVersion < consentVersion
    }

    func status(for purpose: ConsentPurpose) -> ConsentStatus {
        guard let raw = store.string(forKey: "consent_\(purpose.rawValue)")
        else { return .notDetermined }
        return raw == "granted" ? .granted : .denied
    }

    func setConsent(_ status: ConsentStatus, for purpose: ConsentPurpose) {
        let value = status == .granted ? "granted" : "denied"
        store.set(value, forKey: "consent_\(purpose.rawValue)")
        store.set(consentVersion, forKey: "consent_version")
        applyConsentDecisions()
    }

    func grantAll() {
        ConsentPurpose.allCases.forEach { setConsent(.granted, for: $0) }
    }

    func denyAll() {
        ConsentPurpose.allCases.forEach { setConsent(.denied, for: $0) }
    }

    private func applyConsentDecisions() {
        if status(for: .analytics) == .denied {
            AnalyticsService.shared.disable()
        }
        if status(for: .advertising) == .denied {
            AdService.shared.disablePersonalizedAds()
        }
    }
}
Architect Must-Know — Key Design Principles
  • Data minimization — collect only what you need, aggregate on-device, transmit summaries not raw events
  • Purpose limitation — each data point has a declared purpose; enforce this in code with the consent manager
  • Retention limits — auto-delete analytics data after its useful life (90 days for operational, 2 years for trends)
  • k-anonymity on the server — never surface analytics for cohorts smaller than k users (typically k=50) to prevent re-identification
  • Differential privacy budget — track cumulative epsilon across all queries; once the budget is exhausted, stop querying that dataset
  • Consent as a first-class entity — consent state should gate data collection at the SDK level, not just at the UI level. If consent is revoked, previously collected data must be deletable
  • Privacy by design audit — before any new feature ships, run a privacy impact assessment: what data is collected, where does it go, who can access it, when is it deleted?
