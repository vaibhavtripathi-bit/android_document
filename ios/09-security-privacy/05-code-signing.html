1. Code Signing Trust Chain

Every piece of executable code on iOS must be signed by a certificate that chains back to Apple's Root Certificate Authority. This is not optional — the XNU kernel refuses to map unsigned (or invalidly signed) executable pages into memory. This is the foundation of iOS's security model and the reason sideloading is impossible without jailbreaking or enterprise certificates.

1.1 The Certificate Hierarchy

Apple operates a three-level PKI (Public Key Infrastructure) for code signing:

┌─────────────────────────────────────────────────────────────────────────┐ │ APPLE CODE SIGNING TRUST CHAIN │ ├─────────────────────────────────────────────────────────────────────────┤ │ │ │ ┌───────────────────────────────────────────────────────────────────┐ │ │ │ Apple Root CA (self-signed) │ │ │ │ Subject: Apple Root CA │ │ │ │ Validity: ~2035 | Key: 2048-bit RSA → 384-bit ECC (newer) │ │ │ │ Embedded in: iOS Secure Enclave trust store, macOS keychain │ │ │ └────────────────────────────┬──────────────────────────────────────┘ │ │ │ signs │ │ ▼ │ │ ┌───────────────────────────────────────────────────────────────────┐ │ │ │ Apple Worldwide Developer Relations Certification Authority │ │ │ │ (Intermediate CA — "WWDR CA") │ │ │ │ Subject: Apple Worldwide Developer Relations CA - G3 │ │ │ │ Validity: ~2030 | Renewed periodically (G1 → G2 → G3 → G4) │ │ │ │ Purpose: Issues all developer certificates │ │ │ └──────────┬────────────────────────────────┬───────────────────────┘ │ │ │ signs │ signs │ │ ▼ ▼ │ │ ┌──────────────────────────┐ ┌──────────────────────────────────┐ │ │ │ Development Certificate │ │ Distribution Certificate │ │ │ │ CN: "iPhone Developer: │ │ CN: "iPhone Distribution: │ │ │ │ John Doe (XXXXXX)" │ │ ACME Inc (YYYYYY)" │ │ │ │ Usage: Debug builds, │ │ Usage: App Store, Ad Hoc, │ │ │ │ device testing │ │ Enterprise distribution │ │ │ │ Validity: ~1 year │ │ Validity: ~3 years │ │ │ └──────────────────────────┘ └──────────────────────────────────┘ │ │ │ │ ┌───────────────────────────────────────────────────────────────────┐ │ │ │ Apple Distribution Certificate (post-Xcode 11) │ │ │ │ CN: "Apple Distribution: ACME Inc (TEAMID)" │ │ │ │ Unified cert for App Store + Ad Hoc + Enterprise │ │ │ └───────────────────────────────────────────────────────────────────┘ │ │ │ └─────────────────────────────────────────────────────────────────────────┘

Deep Dive — CSR Process

When you create a certificate in Xcode or the Developer Portal, a Certificate Signing Request (CSR) is generated locally. The CSR contains your public key but not your private key. Apple's WWDR CA signs the CSR, producing your developer certificate. The private key never leaves your machine — this is why losing your Keychain means you need to revoke and recreate certificates. The CSR is a PKCS#10 structure encoded in PEM format.

2. CMS Signatures & What Gets Signed

iOS code signing uses CMS (Cryptographic Message Syntax), defined in RFC 5652. CMS is the successor to PKCS#7 and provides a standard way to digitally sign, authenticate, and encrypt data. In the context of code signing, CMS wraps the signature, the signing certificate, and the certificate chain into a single ASN.1 structure.

2.1 What Gets Signed

Code signing on iOS is not a simple hash-of-the-binary. It is a multi-layered process that covers every component of the app bundle:

┌─────────────────────────────────────────────────────────────────────┐ │ APP BUNDLE SIGNING ANATOMY │ ├─────────────────────────────────────────────────────────────────────┤ │ │ │ MyApp.app/ │ │ ├── MyApp (Mach-O executable) │ │ │ ├── __TEXT segment ─────── hashed per 4KB page ──────┐ │ │ │ ├── __DATA segment ─────── hashed per 4KB page ──────┤ │ │ │ ├── __LINKEDIT segment │ │ │ │ │ ├── Code Directory ◄── contains all page hashes ─┘ │ │ │ │ ├── Entitlements (DER-encoded) │ │ │ │ ├── Requirements blob │ │ │ │ └── CMS Signature ◄── signs the Code Directory hash │ │ │ └── LC_CODE_SIGNATURE load command → offset to __LINKEDIT │ │ │ │ │ ├── Info.plist ──────────────── hash in _CodeSignature/ │ │ ├── Assets.car ──────────────── hash in _CodeSignature/ │ │ ├── Base.lproj/ ─────────────── hashes in _CodeSignature/ │ │ ├── Frameworks/ │ │ │ └── MyFramework.framework ── independently signed │ │ ├── PlugIns/ │ │ │ └── Widget.appex ────────── independently signed │ │ ├── embedded.mobileprovision ── NOT signed by app signature │ │ └── _CodeSignature/ │ │ └── CodeResources ───────── plist of all resource hashes │ │ │ └─────────────────────────────────────────────────────────────────────┘

2.2 Page-Level Hashing

The Mach-O executable is divided into 4KB pages (matching the ARM page size). Each page is independently hashed (SHA-256 on modern systems, SHA-1 for legacy compatibility). These hashes are stored in the Code Directory — a data structure embedded in the __LINKEDIT segment.

This page-level granularity is critical: the kernel validates code pages lazily, on first access (page fault). When a page is faulted in, the kernel computes its hash and compares it against the Code Directory. If the hash doesn't match, the process is killed with SIGKILL (code 9). This means:

  • The entire binary doesn't need to be verified at launch time — only pages that are actually executed
  • Any tampering with a single byte of executable code is detected when that page is accessed
  • The Code Directory itself is signed by the CMS signature, creating a chain: CMS → Code Directory → individual page hashes

2.3 CMS Signature Structure

// Simplified CMS SignedData structure (ASN.1)
// RFC 5652 Section 5

SignedData ::= SEQUENCE {
    version          CMSVersion,           // 1 for basic
    digestAlgorithms SET OF DigestAlgorithmIdentifier,  // SHA-256
    encapContentInfo EncapsulatedContentInfo,
    certificates     [0] IMPLICIT CertificateSet,  // signing cert + chain
    signerInfos      SET OF SignerInfo
}

SignerInfo ::= SEQUENCE {
    version            CMSVersion,
    sid                SignerIdentifier,       // issuer + serial number
    digestAlgorithm    DigestAlgorithmIdentifier,  // SHA-256
    signedAttrs        [0] IMPLICIT SignedAttributes,
    signatureAlgorithm SignatureAlgorithmIdentifier,  // RSA/ECDSA
    signature          OCTET STRING           // the actual signature bytes
}

2.4 CodeResources — Resource Sealing

Non-executable resources (images, nibs, plists, localization files) are hashed and recorded in _CodeSignature/CodeResources, a plist file. This file has two hash dictionaries:

Dictionary Hash Algorithm Purpose
files SHA-1 Legacy compatibility (pre-iOS 11)
files2 SHA-256 Modern verification (iOS 11+)

Resources can be marked with rules: required (must be present and match), optional (may be absent), or excluded (ignored during verification). The _CodeSignature directory itself and embedded.mobileprovision are excluded from resource sealing.

Pitfall — Resource Modification After Signing

Any modification to a resource file after signing invalidates the signature. This is a common CI/CD issue: build scripts that modify Info.plist or inject configuration files after the signing step will produce an invalid bundle. Always perform all resource modifications before the code signing phase. In Xcode, the "Copy Bundle Resources" phase must come before "Code Signing."

3. Provisioning Profiles

A provisioning profile is the glue that binds together a certificate, an app ID, a set of entitlements, and (optionally) a list of device UDIDs. It answers the question: "Is this specific app, signed by this specific developer, allowed to run on this specific device with these specific capabilities?"

3.1 What's Inside a .mobileprovision File

A .mobileprovision file is a CMS-signed property list. The outer layer is a CMS envelope signed by Apple's provisioning profile signing certificate. Inside is an XML plist with these keys:

// Inspect a provisioning profile:
// $ security cms -D -i /path/to/profile.mobileprovision

// Key contents of the decoded plist:
{
    "AppIDName":              "My App",
    "ApplicationIdentifierPrefix": ["A1B2C3D4E5"],  // Team ID
    "CreationDate":           "2025-01-15T...",
    "ExpirationDate":         "2026-01-15T...",
    "Entitlements": {
        "application-identifier": "A1B2C3D4E5.com.acme.myapp",
        "keychain-access-groups": ["A1B2C3D4E5.*"],
        "get-task-allow":         true,  // Development only
        "aps-environment":        "development",
        // ... other entitlements
    },
    "DeveloperCertificates":  [<cert-data>, ...],  // DER-encoded X.509
    "Name":                   "My App Dev Profile",
    "ProvisionedDevices":     ["abc123...", "def456..."],  // UDIDs
    "TeamIdentifier":         ["A1B2C3D4E5"],
    "TeamName":               "ACME Inc",
    "TimeToLive":             365,  // days
    "UUID":                   "12345678-ABCD-...",
    "Version":                1,
    "ProvisionsAllDevices":   true  // Enterprise only
}

3.2 The Four Profile Types

Profile Type Certificate Devices get-task-allow Use Case
Development Development cert Specific UDIDs (up to 100) true Xcode debugging, on-device testing
Ad Hoc Distribution cert Specific UDIDs (up to 100) false Beta testing without TestFlight
App Store Distribution cert None (Apple re-signs) false App Store / TestFlight submission
Enterprise Enterprise dist cert None (ProvisionsAllDevices) false In-house distribution (no App Store)

3.3 Development Profile

A development profile ties together:

  • Developer certificate(s) — one or more development certs (each developer on the team can have their own)
  • Device UDIDs — the specific physical devices registered in the Developer Portal (100 per device type per year)
  • App ID — explicit (com.acme.myapp) or wildcard (com.acme.*)
  • Entitlements — includes get-task-allow: true, which allows the debugger (lldb) to attach to the process
Deep Dive — get-task-allow

The get-task-allow entitlement grants the task_for_pid() Mach trap, which gives another process (the debugger) complete control over the target process's memory and execution. This is why it's only present in development profiles — a distribution build with get-task-allow: true would be a massive security hole, allowing any process to read the app's memory.

3.4 Ad Hoc Profile

Ad Hoc profiles use a distribution certificate but still require specific device UDIDs. The 100-device limit is per device type (100 iPhones + 100 iPads + 100 Apple TVs, etc.) and resets annually — but removing a device doesn't free up a slot until the next membership year.

3.5 App Store Profile

App Store profiles contain no device UDIDs. The app is signed with the developer's distribution certificate for upload, but Apple re-signs the binary with Apple's own distribution certificate before delivering it to users. The provisioning profile embedded in the final App Store binary is Apple's, not the developer's.

3.6 Enterprise Profile

Enterprise profiles set ProvisionsAllDevices: true instead of listing specific UDIDs. This allows the app to run on any iOS device — which is both the feature and the security risk. Enterprise certificates require an Apple Developer Enterprise Program membership ($299/year) and are intended exclusively for internal distribution within an organization.

Pitfall — Profile Expiration

Provisioning profiles expire (typically after 1 year). When a profile expires, the app will not launch — iOS checks the profile's ExpirationDate at launch time. For enterprise apps, this means you must have a renewal and redistribution strategy. For App Store apps, Apple manages profile renewal during re-signing, so this is transparent to users.

3.7 Inspecting Profiles

# Decode a provisioning profile (strips CMS envelope)
$ security cms -D -i embedded.mobileprovision

# List all installed profiles
$ ls ~/Library/MobileDevice/Provisioning\ Profiles/

# Find profile by app ID
$ for f in ~/Library/MobileDevice/Provisioning\ Profiles/*.mobileprovision; do
    echo "=== $f ==="
    security cms -D -i "$f" | grep -A1 application-identifier
  done

# Extract certificates from profile
$ security cms -D -i profile.mobileprovision | \
    plutil -extract DeveloperCertificates.0 raw -o cert.der -
$ openssl x509 -inform DER -in cert.der -text -noout

4. Entitlements Embedding & AMFI Validation

Entitlements are key-value pairs that grant an app specific capabilities — push notifications, Keychain sharing, HealthKit access, etc. They exist in three places, and all three must be consistent for the app to function:

4.1 AMFI — Apple Mobile File Integrity

AMFI is a kernel extension (AppleMobileFileIntegrity.kext) that acts as the enforcement layer for code signing on iOS. It hooks into the process creation path and performs validation before any app code executes:

  1. Signature validation — verifies the CMS signature on the Code Directory, checks certificate chain to Apple Root CA
  2. Entitlement validation — ensures every entitlement claimed by the binary is also present in the provisioning profile (binary entitlements must be a subset of profile entitlements)
  3. Profile validation — checks the profile's CMS signature (signed by Apple), expiration date, and (for dev/ad-hoc) device UDID
  4. Runtime enforcement — on every page fault, verifies the page hash against the Code Directory
Architect Must Know — Entitlement Mismatch Debugging

The most common code signing failure in large teams is an entitlement mismatch: the binary requests an entitlement that the provisioning profile doesn't include. This manifests as a crash on launch with no useful error message. To debug: (1) extract binary entitlements with codesign -d --entitlements :- MyApp.app, (2) decode the profile with security cms -D -i embedded.mobileprovision, (3) diff the two entitlement sets. Automate this check in CI.

4.2 Entitlement Encoding Formats

Entitlements are embedded in the code signature in two formats:

Format Magic Bytes Used By Purpose
XML plist 0xFADE7171 Legacy systems, codesign -d --entitlements Human-readable extraction
DER-encoded 0xFADE7172 AMFI kernel validation (iOS 15+) Efficient kernel parsing

Starting with iOS 15, AMFI uses the DER-encoded entitlements for validation because parsing XML in the kernel is slow and error-prone. The DER format is a strict binary encoding that can be validated without a full XML parser.

4.3 Extracting Entitlements

# Extract entitlements from a signed binary (XML format)
$ codesign -d --entitlements :- MyApp.app/MyApp

# Extract entitlements in DER format
$ codesign -d --entitlements :entitlements.der MyApp.app/MyApp

# Compare binary entitlements vs profile entitlements
$ diff \
    <(codesign -d --entitlements :- MyApp.app/MyApp | plutil -convert xml1 -o - -) \
    <(security cms -D -i MyApp.app/embedded.mobileprovision | \
      plutil -extract Entitlements xml1 -o - -)

5.2 FairPlay DRM Encryption

Apple encrypts App Store binaries using FairPlay, the same DRM technology used for iTunes media. The encryption covers the __TEXT segment of the Mach-O binary (where executable code lives). Key details:

  • The LC_ENCRYPTION_INFO_64 load command in the Mach-O header indicates the encrypted range (offset and size) and the encryption system ID
  • The decryption key is unique per Apple ID and device, delivered via the FairPlay protocol during download
  • Decryption happens in-memory at load time — the on-disk binary remains encrypted
  • This is why tools like class-dump don't work directly on App Store binaries — you need to dump the decrypted memory image first

5.3 Private API Scanning

Apple's automated review includes static analysis to detect usage of private (undocumented) APIs. The scanner:

  • Runs nm and otool equivalents to extract all symbol references from the binary
  • Compares against a database of private selectors and C function symbols
  • Scans for dynamic invocations via NSClassFromString, NSSelectorFromString, dlsym
  • Checks for obfuscated strings that decode to private API names
Pitfall — Private API False Positives

The scanner can produce false positives if your code happens to use method names that collide with private APIs (e.g., a method named _setTitle:). If you receive a rejection for private API usage, check whether the flagged symbol is actually yours. You can use nm -u MyApp | grep _privateSymbol to verify. Renaming the method is usually the fastest fix.

5.4 App Store Re-signing

A critical detail many developers miss: the binary users download from the App Store is not signed with your certificate. Apple strips your signature and re-signs with Apple iPhone Distribution certificate. This means:

  • Your distribution certificate could expire or be revoked after submission — already-distributed apps are unaffected
  • The Team ID in the app's entitlements is preserved (for Keychain access groups, App Groups, etc.)
  • You cannot verify App Store builds against your own certificate

6. TestFlight Distribution

TestFlight is Apple's official beta testing platform, integrated into App Store Connect. It provides a distribution mechanism that sits between development builds and App Store release.

6.1 Internal vs External Testing

Aspect Internal Testing External Testing
Testers Up to 100 App Store Connect users (team members) Up to 10,000 external testers via email or public link
Review Required No — available immediately after processing Yes — first build requires Beta App Review
Build Expiry 90 days from upload 90 days from upload
Groups Automatic (all internal testers) Custom groups with different builds
Feedback Screenshots + text via TestFlight app Screenshots + text via TestFlight app
Crash Reports Available in App Store Connect Available in App Store Connect

6.2 Build Processing

When you upload a build for TestFlight, Apple performs a subset of the App Store review pipeline:

  1. Binary validation — same automated checks as App Store (architectures, entitlements, Info.plist)
  2. App thinning — device-specific variants are generated
  3. FairPlay encryption — TestFlight builds are encrypted just like App Store builds
  4. Re-signing — Apple re-signs with a beta distribution certificate
  5. Beta entitlements — the beta-reports-active entitlement is injected, enabling the TestFlight feedback UI

6.3 TestFlight-Specific Behavior

// Detect if running in TestFlight
import StoreKit

func isTestFlight() -> Bool {
    // TestFlight builds have a sandbox receipt
    guard let receiptURL = Bundle.main.appStoreReceiptURL else {
        return false
    }
    return receiptURL.lastPathComponent == "sandboxReceipt"
}

// Modern approach (iOS 16+)
func currentEnvironment() async -> AppTransaction.Environment? {
    guard case .verified(let transaction) = try? await AppTransaction.shared else {
        return nil
    }
    return transaction.environment  // .sandbox, .production, .xcode
}
Best Practice — TestFlight Strategy
  • Use internal testing for daily builds — no review delay, immediate availability
  • Use external testing groups to segment beta users (e.g., "QA Team", "Stakeholders", "Power Users")
  • Set up automatic distribution in App Store Connect to push new builds to testers without manual intervention
  • Monitor the 90-day expiry — set calendar reminders to upload refreshed builds before testers lose access
  • Use the beta-reports-active entitlement check to show in-app feedback prompts only in TestFlight builds

7.2 The Enterprise Certificate Abuse Problem

Enterprise certificates have been systematically abused to create third-party app stores that bypass the App Store entirely. Major incidents include:

Year Incident Impact Apple's Response
2018 Facebook Research VPN (Onavo) Collected user data via enterprise-distributed VPN app targeting teens Revoked Facebook's enterprise cert, breaking all internal FB iOS apps
2019 Google Screenwise Meter Similar data collection app distributed via enterprise cert Revoked Google's enterprise cert temporarily
2015–present TutuApp, AppValley, TweakBox Third-party app stores distributing pirated and modified apps Ongoing certificate revocations; apps rotate through new certs
2019 Illegal gambling apps Thousands of gambling apps distributed via enterprise certs in China Mass revocations, stricter enterprise program enrollment

7.3 Why Enterprise Certs Get Revoked

  • Distribution to non-employees — the license agreement explicitly prohibits distributing to anyone outside the organization
  • Public download links — making the IPA available on a public website is an immediate violation
  • Reselling certificates — some entities sell access to enterprise certificates for sideloading
  • Malware distribution — enterprise certs used to distribute spyware or surveillance tools
Pitfall — Enterprise Certificate Revocation Blast Radius

When Apple revokes an enterprise certificate, every app signed with that certificate stops working immediately. This includes all legitimate internal apps. Facebook's 2019 incident broke their internal iOS apps (employee shuttle bus, cafeteria menu, beta versions of Facebook/Instagram) for over a day. Mitigation: use separate enterprise certificates for internal tools vs. any "gray area" distribution, and have a rapid re-signing capability.

7.4 Apple's Countermeasures

  • OCSP (Online Certificate Status Protocol) — iOS checks certificate revocation status online before launching enterprise apps
  • Stricter enrollment — Enterprise Program now requires D-U-N-S number verification, longer approval times
  • Device trust prompt — users must explicitly trust each enterprise developer in Settings (added in iOS 9)
  • Managed distribution — Apple pushes organizations toward MDM-based distribution instead of direct install links

8. Notarization for macOS

While iOS relies entirely on App Store distribution (or enterprise certs), macOS allows direct distribution of apps outside the Mac App Store. Notarization is Apple's mechanism to add a layer of trust to these directly-distributed apps without requiring App Store review.

8.1 Notarization vs Code Signing vs App Store Review

Aspect Code Signing Notarization App Store Review
What it verifies Identity of signer, integrity of binary No known malware, valid Developer ID Guidelines compliance, functionality, content
Who performs it Developer (locally) Apple (automated scan) Apple (automated + human)
When it happens Build time Post-build, pre-distribution After upload to App Store Connect
Required for All code execution on iOS/macOS macOS apps outside Mac App Store App Store / TestFlight distribution
Result Signed binary Notarization ticket (stapled to binary) Approved for distribution

8.2 The Notarization Process

  1. Developer signs the app with a Developer ID certificate (not a distribution cert)
  2. Upload to Apple via notarytool (or legacy altool)
  3. Apple scans the binary for known malware signatures, suspicious behaviors, and private API usage
  4. Apple issues a ticket — a signed assertion that the binary passed the scan
  5. Developer staples the ticket to the app bundle (embeds it so Gatekeeper can check offline)
  6. Gatekeeper checks the stapled ticket (or queries Apple's servers) before allowing first launch
# Sign with Developer ID
$ codesign --deep --force --options runtime \
    --sign "Developer ID Application: ACME Inc (TEAMID)" \
    MyApp.app

# Submit for notarization
$ xcrun notarytool submit MyApp.zip \
    --apple-id "dev@acme.com" \
    --team-id "A1B2C3D4E5" \
    --password "@keychain:AC_PASSWORD" \
    --wait

# Staple the ticket
$ xcrun stapler staple MyApp.app

# Verify notarization
$ spctl --assess --type execute --verbose MyApp.app

8.3 Hardened Runtime

Notarization requires the Hardened Runtime capability (--options runtime flag). This enables a set of security protections similar to iOS's restrictions:

  • Library Validation — only Apple-signed or same-team-signed libraries can be loaded
  • No unsigned executable memory — MAP_JIT required for JIT compilation
  • DYLD environment variable restrictions — DYLD_INSERT_LIBRARIES blocked
  • Debugging restrictions — task_for_pid() blocked unless entitled

Exceptions can be granted via entitlements (e.g., com.apple.security.cs.allow-unsigned-executable-memory), but each exception must be justified and may trigger additional scrutiny during notarization.

Deep Dive — iOS Relevance of Notarization

While notarization is macOS-specific today, Apple has signaled interest in "notarization-like" processes for alternative app marketplaces under the EU Digital Markets Act (DMA). The iOS 17.4 sideloading framework in the EU requires apps distributed outside the App Store to be notarized by Apple — a scan for malware and policy violations, similar to macOS notarization but with additional requirements around marketplace operator responsibilities.

9.2 Code Directory Structure

The Code Directory is the central data structure of code signing. It contains all the hashes that define the binary's identity:

// Code Directory structure (from Apple's open-source Security framework)
struct CodeDirectory {
    uint32_t magic;          // 0xFADE0C02
    uint32_t length;
    uint32_t version;         // 0x20400 for current
    uint32_t flags;           // CS_ADHOC, CS_LINKER_SIGNED, etc.
    uint32_t hashOffset;      // offset to code slot hashes
    uint32_t identOffset;     // offset to signing identifier string
    uint32_t nSpecialSlots;   // number of special slots (Info.plist, etc.)
    uint32_t nCodeSlots;      // number of code page hashes
    uint32_t codeLimit;       // size of code to hash
    uint8_t  hashSize;        // 32 for SHA-256
    uint8_t  hashType;        // 2 = SHA-256, 1 = SHA-1
    uint8_t  platform;
    uint8_t  pageSize;        // log2(page_size) = 12 → 4096 bytes
    uint32_t spare2;
    // v0x20100+
    uint32_t scatterOffset;
    // v0x20200+
    uint32_t teamOffset;      // offset to team identifier string
    // v0x20300+
    uint32_t spare3;
    uint64_t codeLimit64;
    // v0x20400+
    uint64_t execSegBase;     // offset of executable segment
    uint64_t execSegLimit;
    uint64_t execSegFlags;    // CS_EXECSEG_MAIN_BINARY, etc.
    // Followed by: hash slots (special + code)
};

9.3 Essential codesign Commands

# Sign an app bundle
$ codesign --force --sign "iPhone Distribution: ACME Inc (TEAMID)" \
    --entitlements MyApp.entitlements \
    MyApp.app

# Sign with timestamp (recommended for distribution)
$ codesign --force --sign "Developer ID Application: ACME Inc" \
    --timestamp \
    --options runtime \
    MyApp.app

# Verify signature
$ codesign --verify --deep --strict --verbose=4 MyApp.app

# Display signing information
$ codesign -d --verbose=4 MyApp.app

# Display entitlements
$ codesign -d --entitlements :- MyApp.app

# Display code directory hashes
$ codesign -d --verbose=4 MyApp.app 2>&1 | grep CDHash

# Verify against a specific requirement
$ codesign --verify --requirement='anchor apple generic and
    certificate leaf[subject.OU] = "A1B2C3D4E5"' MyApp.app

# Remove signature
$ codesign --remove-signature MyApp.app

# Re-sign with different identity (common in CI/CD)
$ codesign --force --sign "Apple Distribution: ACME Inc (TEAMID)" \

    MyApp.app

Slot Index Content Description
-1 Info.plist Hash of the app's Info.plist
-2 Requirements Hash of the designated requirement
-3 Resource Directory Hash of _CodeSignature/CodeResources
-4 Application Specific Reserved (unused on iOS)
-5 Entitlements (XML) Hash of XML-encoded entitlements

0..N Code pages Hash of each 4KB page of executable code
Deep Dive — CDHash and Trust Caches

The CDHash (Code Directory Hash) is the SHA-256 hash of the entire Code Directory. It serves as the unique fingerprint of a specific build. Apple maintains trust caches — lists of CDHashes for system binaries baked into the kernel cache. These binaries don't need CMS signature verification at runtime because their CDHash is pre-trusted. This is how Apple's own system daemons bypass the normal certificate chain validation — they're identified by CDHash, not by certificate.

10. ARCHITECT L6 Managing Code Signing for Large Teams

Code signing at scale — across dozens of developers, multiple app targets, CI/CD pipelines, and framework dependencies — is one of the most operationally complex aspects of iOS development. A Staff engineer must design systems that are secure, reliable, and don't become a bottleneck.

10.1 Automatic vs Manual Signing

Certificate management Xcode creates/manages automatically Developer/CI explicitly selects certificate
Profile management Xcode creates/renews/downloads automatically Profiles downloaded from portal or managed by tool
Entitlements Xcode syncs with portal capabilities Developer maintains .entitlements file manually

Architect Must Know — Why Large Teams Use Manual Signing

Automatic signing's biggest problem at scale is non-determinism. When Developer A opens Xcode, it may regenerate the provisioning profile, invalidating the one Developer B is using. On CI, automatic signing requires storing Apple ID credentials on the build machine and dealing with 2FA. Manual signing with a tool like fastlane match ensures every developer and CI machine uses the exact same certificate and profile, stored in a shared encrypted repository.

10.2 fastlane match — The Industry Standard

fastlane match implements the "codesigning-as-a-service" pattern: certificates and profiles are stored in a shared, encrypted Git repository (or Google Cloud Storage / S3), and every developer and CI machine syncs from this single source of truth.

10.3 CI/CD Keychain Management

On CI servers (GitHub Actions, Jenkins, Bitrise, etc.), code signing requires careful Keychain management because the build agent may not have a login Keychain or may be running as a different user:

#!/bin/bash
# CI/CD code signing setup script

# Create a temporary keychain
KEYCHAIN_NAME="build.keychain-db"
KEYCHAIN_PASSWORD=$(openssl rand -base64 32)

security create-keychain -p "$KEYCHAIN_PASSWORD" "$KEYCHAIN_NAME"

# Configure keychain: no auto-lock, add to search list
security set-keychain-settings -lut 21600 "$KEYCHAIN_NAME"
security list-keychains -d user -s "$KEYCHAIN_NAME" $(security list-keychains -d user | tr -d '"')
security unlock-keychain -p "$KEYCHAIN_PASSWORD" "$KEYCHAIN_NAME"

# Import certificate and private key
security import certificate.p12 \
    -k "$KEYCHAIN_NAME" \
    -P "$P12_PASSWORD" \
    -T /usr/bin/codesign \
    -T /usr/bin/security

# Allow codesign to access the key without UI prompt
security set-key-partition-list -S apple-tool:,apple: \
    -s -k "$KEYCHAIN_PASSWORD" "$KEYCHAIN_NAME"

# Install provisioning profiles
mkdir -p ~/Library/MobileDevice/Provisioning\ Profiles/
cp *.mobileprovision ~/Library/MobileDevice/Provisioning\ Profiles/

# Build and sign
xcodebuild archive \
    -workspace MyApp.xcworkspace \
    -scheme MyApp \
    -archivePath build/MyApp.xcarchive \
    CODE_SIGN_IDENTITY="Apple Distribution: ACME Inc (TEAMID)" \
    PROVISIONING_PROFILE_SPECIFIER="match AppStore com.acme.myapp"

# Cleanup: delete temporary keychain
security delete-keychain "$KEYCHAIN_NAME"
Pitfall — set-key-partition-list

The security set-key-partition-list command is the most commonly missed step in CI code signing setup. Without it, codesign will fail with "User interaction is not allowed" because macOS requires explicit permission for a process to access Keychain items. This command grants codesign access without a UI prompt. The -S apple-tool:,apple: flags specify the partition IDs that are allowed access.

10.5 Framework and Extension Signing

In a multi-target app (main app + extensions + embedded frameworks), every executable component must be independently signed. The signing must be consistent:

Best Practice — Code Signing Architecture for Large Teams
  • Use manual signing for all targets — automatic signing creates non-deterministic behavior across team members and CI
  • Centralize certificates with fastlane match — single encrypted repo, one source of truth, readonly mode on CI
  • Create a dedicated CI/CD Apple ID — don't use a personal account; create a service account with App Manager role
  • Separate keychains on CI — create a temporary keychain per build, delete after; never pollute the system keychain
  • Automate profile regeneration — when adding new capabilities, run match with force_for_new_devices to regenerate profiles
  • Pin WWDR intermediate certificate — ensure CI machines have the correct WWDR CA installed (G3/G4); expired intermediates cause mysterious failures
  • Monitor certificate expiry — set up alerts 90, 60, and 30 days before expiry; automate rotation with match
  • Use Xcode Cloud for simpler setups — Apple manages signing automatically in Xcode Cloud, eliminating most of these concerns

10.7 Interview Questions

Senior Q: What happens if you sign an app with a certificate that doesn't match the provisioning profile?

A: AMFI will reject the app at launch. The provisioning profile contains the DER-encoded certificates that are authorized to sign the app. If the signing certificate's public key doesn't match any certificate in the profile's DeveloperCertificates array, the signature validation fails. On a device, this manifests as the app icon appearing briefly then disappearing, or a "Unable to Verify App" dialog. In the Console, you'll see amfid logging a verification failure.

Architect Q: How would you design a code signing system for a company with 50 iOS developers, 10 app targets, and 3 CI/CD environments?

A: The design should address five concerns:

  1. Certificate management — Use fastlane match with an encrypted Git repository. Create one distribution certificate shared across all targets (they share the same Team ID). Development certificates can be per-developer (automatic) or shared (match).
  2. Profile management — One provisioning profile per target per distribution type. Use match to generate and store all profiles. CI runs match in readonly mode; only the release manager can run match in write mode to regenerate profiles.
  3. CI/CD isolation — Each CI job creates a temporary keychain, imports certs, builds, and deletes the keychain. No persistent signing material on CI machines. Secrets (match passphrase, p12 password) stored in CI secret management (GitHub Secrets, Vault, etc.).
  4. Access control — Separate Apple Developer Portal roles: Admin (cert management), App Manager (profile management), Developer (read-only). Match repo access controlled via Git permissions.
  5. Monitoring — Automated alerts for certificate expiry (90/60/30 days), profile expiry, WWDR intermediate certificate updates. A Slack bot or dashboard showing signing health across all targets.

L6 Q: Explain the security implications of the get-task-allow entitlement and why it must never appear in production builds.

A: The get-task-allow entitlement grants the task_for_pid() Mach trap to other processes. This trap returns a task port — a Mach port that gives complete control over the target process: read/write memory (mach_vm_read/mach_vm_write), suspend/resume threads, modify registers, and inject code. This is how lldb implements breakpoints and memory inspection. In a production build, this would allow any process (including malware) to read encryption keys, authentication tokens, and user data from the app's memory. AMFI enforces that get-task-allow is only present in development profiles (which are device-restricted), never in App Store or enterprise profiles. If you somehow managed to embed it in a distribution build, the App Store automated checks would reject it.

Architect Q: Your enterprise-distributed app suddenly stops launching on all employee devices. What happened and how do you recover?

A: The most likely cause is certificate revocation — either Apple revoked the enterprise certificate (policy violation) or someone on the team accidentally revoked it in the Developer Portal. iOS checks certificate status via OCSP at launch time. Recovery steps: (1) Check the Developer Portal for certificate status. (2) If revoked by Apple, contact Apple Developer Support immediately — if it was accidental or a misunderstanding, they may reinstate it. (3) If the cert is permanently revoked, create a new enterprise certificate, regenerate all provisioning profiles, re-sign all apps, and redistribute via MDM. (4) For faster recovery, maintain a secondary enterprise certificate (Apple allows multiple) with pre-built profiles, so you can re-sign and push within hours. (5) Long-term: implement MDM-based distribution so re-deployment is automated, and maintain a "break glass" procedure documented in the team's runbook.
