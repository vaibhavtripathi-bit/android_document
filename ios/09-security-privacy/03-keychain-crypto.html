1. Keychain Services Deep Dive — SecItem API

The Keychain is Apple's secure credential store — an encrypted database managed by the securityd daemon. All interaction happens through four C functions that accept and return CFDictionary objects. Understanding the query dictionary structure is the key to mastering keychain operations.

1.1 The Four SecItem Functions

Function Purpose Key Parameters
SecItemAdd Create a new keychain item Attributes + value data
SecItemCopyMatching Read / search for items Query + return type keys
SecItemUpdate Modify existing items Query to match + new attributes
SecItemDelete Remove items Query to match

1.2 kSecClass Types

Every keychain item belongs to exactly one class. The class determines which attributes are valid in the query dictionary:

kSecClass Value Swift Constant Use Case Uniqueness Keys
kSecClassGenericPassword kSecClassGenericPassword App passwords, tokens, arbitrary secrets kSecAttrAccount + kSecAttrService
kSecClassInternetPassword kSecClassInternetPassword Website credentials (Safari autofill) kSecAttrAccount + kSecAttrServer + kSecAttrProtocol + kSecAttrPort
kSecClassCertificate kSecClassCertificate X.509 certificates kSecAttrCertificateType + kSecAttrIssuer + kSecAttrSerialNumber
kSecClassKey kSecClassKey Cryptographic keys (RSA, EC, AES) kSecAttrApplicationLabel + kSecAttrApplicationTag + kSecAttrKeyType
kSecClassIdentity kSecClassIdentity Certificate + private key pair Combination of cert + key attributes

1.3 kSecAttrAccessible Options

Controls when a keychain item is readable. This maps directly to the Data Protection class applied to the item's encryption key:

Accessibility Available When Survives Backup Use Case
kSecAttrAccessibleWhenUnlocked Device unlocked Yes Default — most credentials
kSecAttrAccessibleAfterFirstUnlock After first unlock (even if locked now) Yes Background fetch tokens
kSecAttrAccessibleWhenPasscodeSetThisDeviceOnly Device unlocked + passcode set No High-security items; deleted if passcode removed
kSecAttrAccessibleAlways Always (deprecated iOS 12+) Yes Avoid — no protection at rest
*ThisDeviceOnly variants Same as base + device-bound No Items that must never migrate
Common Pitfall: Accessibility Defaults

If you omit kSecAttrAccessible, the system defaults to kSecAttrAccessibleWhenUnlocked. This means your keychain item is not available during background fetch, silent push handling, or BGTask execution if the device is locked. Use kSecAttrAccessibleAfterFirstUnlock for items needed in background processing.

1.4 Full CRUD Operations

import Security

enum KeychainError: Error {
    case duplicateItem
    case itemNotFound
    case unexpectedStatus(OSStatus)
    case invalidData
}

struct KeychainManager {

    private let service: String

    init(service: String = Bundle.main.bundleIdentifier ?? "com.app") {
        self.service = service
    }

    // ── CREATE ──────────────────────────────────────────────
    func save(_ data: Data, forAccount account: String) throws {
        let query: [String: Any] = [
            kSecClass as String:            kSecClassGenericPassword,
            kSecAttrService as String:      service,
            kSecAttrAccount as String:      account,
            kSecValueData as String:        data,
            kSecAttrAccessible as String:   kSecAttrAccessibleAfterFirstUnlock
        ]

        let status = SecItemAdd(query as CFDictionary, nil)

        switch status {
        case errSecSuccess:
            return
        case errSecDuplicateItem:
            throw KeychainError.duplicateItem
        default:
            throw KeychainError.unexpectedStatus(status)
        }
    }

    // ── READ ────────────────────────────────────────────────
    func read(account: String) throws -> Data {
        let query: [String: Any] = [
            kSecClass as String:            kSecClassGenericPassword,
            kSecAttrService as String:      service,
            kSecAttrAccount as String:      account,
            kSecReturnData as String:       true,
            kSecMatchLimit as String:       kSecMatchLimitOne
        ]

        var result: AnyObject?
        let status = SecItemCopyMatching(query as CFDictionary, &result)

        switch status {
        case errSecSuccess:
            guard let data = result as? Data else {
                throw KeychainError.invalidData
            }
            return data
        case errSecItemNotFound:
            throw KeychainError.itemNotFound
        default:
            throw KeychainError.unexpectedStatus(status)
        }
    }

    // ── UPDATE ──────────────────────────────────────────────
    func update(_ data: Data, forAccount account: String) throws {
        let query: [String: Any] = [
            kSecClass as String:       kSecClassGenericPassword,
            kSecAttrService as String: service,
            kSecAttrAccount as String: account
        ]

        let attributes: [String: Any] = [
            kSecValueData as String: data
        ]

        let status = SecItemUpdate(
            query as CFDictionary,
            attributes as CFDictionary
        )

        switch status {
        case errSecSuccess:
            return
        case errSecItemNotFound:
            throw KeychainError.itemNotFound
        default:
            throw KeychainError.unexpectedStatus(status)
        }
    }

    // ── DELETE ──────────────────────────────────────────────
    func delete(account: String) throws {
        let query: [String: Any] = [
            kSecClass as String:       kSecClassGenericPassword,
            kSecAttrService as String: service,
            kSecAttrAccount as String: account
        ]

        let status = SecItemDelete(query as CFDictionary)

        guard status == errSecSuccess || status == errSecItemNotFound else {
            throw KeychainError.unexpectedStatus(status)
        }
    }

    // ── UPSERT (save-or-update) ─────────────────────────────
    func upsert(_ data: Data, forAccount account: String) throws {
        do {
            try save(data, forAccount: account)
        } catch KeychainError.duplicateItem {
            try update(data, forAccount: account)
        }
    }
}

Deep Dive: errSecInteractionNotAllowed (-25308)

This error surfaces when a keychain item requires user authentication (biometric or passcode) but the operation runs in a context where UI cannot be presented — typically background execution. Items with kSecAttrAccessibleWhenUnlocked also trigger this when the device is locked. In background modes, always use kSecAttrAccessibleAfterFirstUnlock and avoid items gated behind SecAccessControl with biometric requirements.

2. Keychain Architecture

The keychain is not a simple file your app reads directly. It's a system service with a multi-layered architecture designed to isolate secrets from app processes, even if the app is compromised.

2.1 XPC Communication

When your app calls SecItemAdd, the Security framework serializes the query dictionary into an XPC message and sends it to securityd via a Mach port. The daemon performs all cryptographic operations and database access in its own address space. Your app never sees the raw database or the encryption keys. This means even if an attacker achieves code execution in your app, they cannot dump the keychain database directly — they must go through the securityd API which enforces access control.

2.2 Item Uniqueness Rules

The keychain enforces uniqueness based on a primary key set that varies by class. Attempting to add an item with the same primary keys as an existing item yields errSecDuplicateItem.

Common Pitfall: Unexpected Duplicate Items

For kSecClassGenericPassword, the primary key is (kSecAttrAccount, kSecAttrService, kSecAttrAccessGroup). If you change only kSecAttrAccessible and call SecItemAdd, you get errSecDuplicateItem because accessibility is not part of the primary key. You must use SecItemUpdate to change attributes on an existing item. Conversely, if you accidentally omit kSecAttrService, different items may collide because they share the same default service value.

2.3 Data Protection Integration

Each keychain item is encrypted with a class key derived from the device's hardware UID key and the user's passcode. The kSecAttrAccessible value determines which class key is used, and therefore when the item can be decrypted:

3. SecKey for Asymmetric Cryptography

The SecKey API provides access to RSA and elliptic curve cryptographic operations. Keys can live in the keychain, in the Secure Enclave, or transiently in memory. SecKey is the lower-level API — for new code, prefer CryptoKit (Section 4) when possible, but SecKey remains essential for interoperability with server certificates, legacy systems, and Secure Enclave operations.

3.1 Generating Key Pairs

import Security

// ── RSA 2048-bit Key Pair ────────────────────────────────
func generateRSAKeyPair() throws -> (SecKey, SecKey) {
    let attributes: [String: Any] = [
        kSecAttrKeyType as String:          kSecAttrKeyTypeRSA,
        kSecAttrKeySizeInBits as String:    2048,
        kSecPrivateKeyAttrs as String: [
            kSecAttrIsPermanent as String:    true,
            kSecAttrApplicationTag as String: "com.app.rsa.private".data(using: .utf8)!
        ]
    ]

    var error: Unmanaged<CFError>?
    guard let privateKey = SecKeyCreateRandomKey(attributes as CFDictionary, &error) else {
        throw error!.takeRetainedValue() as Error
    }

    guard let publicKey = SecKeyCopyPublicKey(privateKey) else {
        throw NSError(domain: "KeyGen", code: -1)
    }

    return (privateKey, publicKey)
}

// ── EC P-256 Key Pair ────────────────────────────────────
func generateECKeyPair() throws -> (SecKey, SecKey) {
    let attributes: [String: Any] = [
        kSecAttrKeyType as String:          kSecAttrKeyTypeECSECPrimeRandom,
        kSecAttrKeySizeInBits as String:    256,
        kSecPrivateKeyAttrs as String: [
            kSecAttrIsPermanent as String:    false,
            kSecAttrApplicationTag as String: "com.app.ec.private".data(using: .utf8)!
        ]
    ]

    var error: Unmanaged<CFError>?
    guard let privateKey = SecKeyCreateRandomKey(attributes as CFDictionary, &error) else {
        throw error!.takeRetainedValue() as Error
    }

    let publicKey = SecKeyCopyPublicKey(privateKey)!
    return (privateKey, publicKey)
}

3.2 Signing & Verification

// ── ECDSA Signing ───────────────────────────────────────
func sign(data: Data, with privateKey: SecKey) throws -> Data {
    let algorithm: SecKeyAlgorithm = .ecdsaSignatureMessageX962SHA256

    guard SecKeyIsAlgorithmSupported(privateKey, .sign, algorithm) else {
        throw NSError(domain: "Crypto", code: -1)
    }

    var error: Unmanaged<CFError>?
    guard let signature = SecKeyCreateSignature(
        privateKey, algorithm, data as CFData, &error
    ) as Data? else {
        throw error!.takeRetainedValue() as Error
    }

    return signature
}

// ── ECDSA Verification ──────────────────────────────────
func verify(data: Data, signature: Data, with publicKey: SecKey) -> Bool {
    let algorithm: SecKeyAlgorithm = .ecdsaSignatureMessageX962SHA256

    var error: Unmanaged<CFError>?
    let result = SecKeyVerifySignature(
        publicKey, algorithm,
        data as CFData, signature as CFData,
        &error
    )

    return result
}

3.3 RSA Encryption & Decryption

// ── RSA-OAEP Encryption ─────────────────────────────────
func encrypt(data: Data, with publicKey: SecKey) throws -> Data {
    let algorithm: SecKeyAlgorithm = .rsaEncryptionOAEPSHA256

    guard SecKeyIsAlgorithmSupported(publicKey, .encrypt, algorithm) else {
        throw NSError(domain: "Crypto", code: -1)
    }

    let maxSize = SecKeyGetBlockSize(publicKey) - 66
    guard data.count <= maxSize else {
        throw NSError(domain: "Crypto", code: -2,
                      userInfo: [NSLocalizedDescriptionKey: "Data too large for RSA"])
    }

    var error: Unmanaged<CFError>?
    guard let ciphertext = SecKeyCreateEncryptedData(
        publicKey, algorithm, data as CFData, &error
    ) as Data? else {
        throw error!.takeRetainedValue() as Error
    }

    return ciphertext
}

// ── RSA-OAEP Decryption ─────────────────────────────────
func decrypt(ciphertext: Data, with privateKey: SecKey) throws -> Data {
    let algorithm: SecKeyAlgorithm = .rsaEncryptionOAEPSHA256

    var error: Unmanaged<CFError>?
    guard let plaintext = SecKeyCreateDecryptedData(
        privateKey, algorithm, ciphertext as CFData, &error
    ) as Data? else {
        throw error!.takeRetainedValue() as Error
    }

    return plaintext
}
Common Pitfall: RSA Plaintext Size Limit

RSA-OAEP with SHA-256 can only encrypt data up to keySize - 66 bytes (for 2048-bit: 256 - 66 = 190 bytes). For larger payloads, use hybrid encryption: generate a random AES key, encrypt the payload with AES-GCM, then encrypt the AES key with RSA. This is the standard pattern used by TLS, CMS/PKCS#7, and most real-world protocols.

4. CryptoKit Framework

Introduced in iOS 13, CryptoKit provides a modern, Swift-native API for common cryptographic operations. It uses constant-time implementations to resist timing attacks and integrates with the Secure Enclave. CryptoKit should be your default choice for new cryptographic code.

4.1 AES-GCM Encryption & Decryption

AES-GCM (Galois/Counter Mode) provides both confidentiality and authenticity in a single operation — it's an authenticated encryption with associated data (AEAD) cipher.

import CryptoKit

// ── AES-GCM Encryption ──────────────────────────────────
func encryptAESGCM(plaintext: Data, key: SymmetricKey) throws -> Data {
    let sealedBox = try AES.GCM.seal(plaintext, using: key)

    guard let combined = sealedBox.combined else {
        throw CryptoKitError.underlyingCoreCryptoError(error: 0)
    }

    return combined
}

// ── AES-GCM Decryption ──────────────────────────────────
func decryptAESGCM(ciphertext: Data, key: SymmetricKey) throws -> Data {
    let sealedBox = try AES.GCM.SealedBox(combined: ciphertext)
    let decrypted = try AES.GCM.open(sealedBox, using: key)
    return decrypted
}

// ── Usage ────────────────────────────────────────────────
let key = SymmetricKey(size: .bits256)
let message = "Secret message".data(using: .utf8)!

let encrypted = try encryptAESGCM(plaintext: message, key: key)
let decrypted = try decryptAESGCM(ciphertext: encrypted, key: key)

// With Additional Authenticated Data (AAD)
let aad = "user-id:12345".data(using: .utf8)!
let sealedBox = try AES.GCM.seal(
    message, using: key,
    authenticating: aad
)
let opened = try AES.GCM.open(sealedBox, using: key, authenticating: aad)
Deep Dive: SealedBox.combined Format

The combined property returns nonce (12 bytes) || ciphertext || tag (16 bytes). When transmitting encrypted data, you can send this single blob and reconstruct the SealedBox on the other end. Alternatively, access .nonce, .ciphertext, and .tag separately if your protocol requires a different wire format.

4.5 Curve25519 — Key Agreement & Signing

import CryptoKit

// ── X25519 Key Agreement (Diffie-Hellman) ───────────────
let alicePrivate = Curve25519.KeyAgreement.PrivateKey()
let bobPrivate = Curve25519.KeyAgreement.PrivateKey()

let alicePublic = alicePrivate.publicKey
let bobPublic = bobPrivate.publicKey

// Both sides derive the same shared secret
let aliceShared = try alicePrivate.sharedSecretFromKeyAgreement(with: bobPublic)
let bobShared = try bobPrivate.sharedSecretFromKeyAgreement(with: alicePublic)
// aliceShared == bobShared

// Derive a symmetric key from the shared secret using HKDF
let symmetricKey = aliceShared.hkdfDerivedSymmetricKey(
    using: SHA256.self,
    salt: "chat-encryption".data(using: .utf8)!,
    sharedInfo: Data(),
    outputByteCount: 32
)

// Now use symmetricKey with AES.GCM for message encryption

// ── Ed25519 Signing ─────────────────────────────────────
let edKey = Curve25519.Signing.PrivateKey()
let edSignature = try edKey.signature(for: data)
let edValid = edKey.publicKey.isValidSignature(edSignature, for: data)

Best Practice: Choosing Algorithms
  • Symmetric encryption: AES-GCM-256 (hardware-accelerated on Apple Silicon). Use ChaChaPoly only if you need software-only performance parity on older devices.
  • Hashing: SHA-256 for most purposes. SHA-512 only if your protocol requires it.
  • Signing: P256 for interoperability with existing infrastructure. Ed25519 for new protocols (faster, deterministic, no nonce pitfalls).
  • Key agreement: X25519 for new protocols. P256 ECDH for NIST compliance or Secure Enclave integration.

5. Secure Enclave Key Generation

The Secure Enclave (SE) is a hardware coprocessor embedded in Apple's SoC (A7 and later). It has its own encrypted memory, its own boot ROM, and a hardware random number generator. The critical property: private keys generated inside the SE never leave it. All cryptographic operations (signing, key agreement) happen inside the enclave — the application processor only sees the results.

┌──────────────────────────────────────────────────────────────────┐ │ Application Processor │ │ │ │ ┌──────────────────────────────────────────────────────────┐ │ │ │ Your App │ │ │ │ │ │ │ │ SecKeyCreateRandomKey(...) → SecKey (reference handle) │ │ │ │ SecKeyCreateSignature(...) → Signature bytes │ │ │ │ │ │ │ │ ⚠️ Private key bytes are NEVER available here │ │ │ │ SecKeyCopyExternalRepresentation → returns nil │ │ │ └──────────────────────────┬───────────────────────────────┘ │ │ │ XPC / Mailbox │ └──────────────────────────────┼───────────────────────────────────┘ │ ▼ ┌──────────────────────────────────────────────────────────────────┐ │ Secure Enclave Processor (SEP) │ │ │ │ ┌──────────────┐ ┌──────────────────────────────────────────┐ │ │ │ sepOS │ │ Key Store │ │ │ │ (dedicated │ │ │ │ │ │ microkernel) │ │ ┌─────────────────────────────────────┐ │ │ │ │ │ │ │ P-256 Private Key (generated here) │ │ │ │ │ │ │ │ Wrapped with per-device SE key │ │ │ │ │ │ │ │ Never exported, never in AP memory │ │ │ │ │ │ │ └─────────────────────────────────────┘ │ │ │ └──────────────┘ │ │ │ │ │ Operations: │ │ │ │ • ECDSA Sign (data in → signature out) │ │ │ │ • ECDH Key Agreement (pub in → secret) │ │ │ │ • Key generation (entropy from HW RNG) │ │ │ └──────────────────────────────────────────┘ │ │ │ │ ┌──────────────────────────────────────────────────────────┐ │ │ │ Biometric Engine (Face ID / Touch ID) │ │ │ │ Template matching happens entirely inside SEP │ │ │ │ Match result gates access to SE-backed keys │ │ │ └──────────────────────────────────────────────────────────┘ │ └──────────────────────────────────────────────────────────────────┘

5.2 Generating & Using SE-Backed Keys

import Security
import CryptoKit
import LocalAuthentication

func generateSecureEnclaveKey() throws -> SecKey {
    let access = SecAccessControlCreateWithFlags(
        kCFAllocatorDefault,
        kSecAttrAccessibleWhenUnlockedThisDeviceOnly,
        [.privateKeyUsage, .biometryCurrentSet],
        nil
    )!

    let attributes: [String: Any] = [
        kSecAttrKeyType as String:          kSecAttrKeyTypeECSECPrimeRandom,
        kSecAttrKeySizeInBits as String:    256,
        kSecAttrTokenID as String:           kSecAttrTokenIDSecureEnclave,
        kSecPrivateKeyAttrs as String: [
            kSecAttrIsPermanent as String:    true,
            kSecAttrApplicationTag as String: "com.app.se.signing".data(using: .utf8)!,
            kSecAttrAccessControl as String:  access
        ]
    ]

    var error: Unmanaged<CFError>?
    guard let privateKey = SecKeyCreateRandomKey(
        attributes as CFDictionary, &error
    ) else {
        throw error!.takeRetainedValue() as Error
    }

// ── CryptoKit Secure Enclave API (simpler) ──────────────
func generateSEKeyWithCryptoKit() throws -> SecureEnclave.P256.Signing.PrivateKey {
    guard SecureEnclave.isAvailable else {
        throw NSError(domain: "SE", code: -1,
                      userInfo: [NSLocalizedDescriptionKey: "Secure Enclave not available"])
    }

    let authContext = LAContext()
    authContext.localizedReason = "Sign transaction"

    let accessControl = SecAccessControlCreateWithFlags(
        nil,
        kSecAttrAccessibleWhenUnlockedThisDeviceOnly,
        [.privateKeyUsage, .biometryCurrentSet],
        nil
    )!

    let privateKey = try SecureEnclave.P256.Signing.PrivateKey(
        accessControl: accessControl,
        authenticationContext: authContext
    )

    return privateKey
}

// ── Signing with SE key ─────────────────────────────────
func signWithSEKey(_ data: Data,
                    key: SecureEnclave.P256.Signing.PrivateKey) throws -> P256.Signing.ECDSASignature {
    return try key.signature(for: data)
}
Common Pitfall: .biometryCurrentSet vs .biometryAny

.biometryCurrentSet invalidates the key if the user adds or removes a fingerprint/face. This is the secure choice — it prevents an attacker who adds their own biometric from accessing existing keys. .biometryAny allows any enrolled biometric, including newly added ones. For high-security keys (payment signing, authentication), always use .biometryCurrentSet.

6. Biometric Authentication via LAContext

The LocalAuthentication framework provides LAContext for evaluating biometric and passcode policies. It integrates with the Secure Enclave for biometric matching and with the keychain for access-controlled items.

6.1 LAContext Configuration

import LocalAuthentication

let context = LAContext()

// Configuration

    }

Deep Dive: LAContext Reuse Duration

Setting touchIDAuthenticationAllowableReuseDuration (e.g., 30 seconds) allows the system to reuse a recent biometric evaluation without re-prompting. This improves UX when multiple keychain operations happen in quick succession. The maximum value is LATouchIDAuthenticationMaximumAllowableReuseDuration (5 minutes). Setting it to 0 forces a fresh evaluation every time. The timer starts from the last successful biometric evaluation on the LAContext instance.

Common Pitfall: Info.plist Face ID Usage Description

If your app uses Face ID, you must include NSFaceIDUsageDescription in Info.plist. Without it, the biometric prompt silently fails with LAError.biometryNotAvailable on Face ID devices. Touch ID does not require a usage description. This is one of the most common causes of "biometrics suddenly stopped working" bugs when testing on Face ID devices.

Best Practice: Keychain Sharing Strategy
  • App + Extensions: Use App Groups for both file sharing and keychain sharing. The group.* keychain access group is automatically available.
  • Multiple apps, same team: Use explicit keychain-access-groups with a shared group identifier.
  • Sensitive items: Always use ThisDeviceOnly accessibility for shared secrets to prevent them from appearing in backups accessible to other devices.
  • Migration: When adding sharing to an existing app, existing items are in the default group. You must read them from the old group and re-save to the shared group.

8.2 Key Generation & Registration

import CryptoKit
import Security

struct DeviceKeyBundle {
    let identityKey: P256.Signing.PrivateKey
    let signedPreKey: P256.KeyAgreement.PrivateKey
    let signedPreKeySignature: P256.Signing.ECDSASignature
    var oneTimePreKeys: [P256.KeyAgreement.PrivateKey]
}

struct PublicKeyBundle: Codable {
    let identityKey: Data
    let signedPreKey: Data
    let signedPreKeySignature: Data
    let oneTimePreKeys: [Data]
}

final class E2EKeyManager {

    func generateDeviceKeys() throws -> DeviceKeyBundle {
        let identityKey = P256.Signing.PrivateKey()

        let signedPreKey = P256.KeyAgreement.PrivateKey()
        let preKeyData = signedPreKey.publicKey.rawRepresentation
        let preKeySignature = try identityKey.signature(for: preKeyData)

        var oneTimeKeys: [P256.KeyAgreement.PrivateKey] = []
        for _ in 0..<100 {
            oneTimeKeys.append(P256.KeyAgreement.PrivateKey())
        }

        return DeviceKeyBundle(
            identityKey: identityKey,
            signedPreKey: signedPreKey,
            signedPreKeySignature: preKeySignature,
            oneTimePreKeys: oneTimeKeys
        )
    }

    func exportPublicBundle(_ bundle: DeviceKeyBundle) -> PublicKeyBundle {
        return PublicKeyBundle(
            identityKey: bundle.identityKey.publicKey.rawRepresentation,
            signedPreKey: bundle.signedPreKey.publicKey.rawRepresentation,
            signedPreKeySignature: bundle.signedPreKeySignature.rawRepresentation,
            oneTimePreKeys: bundle.oneTimePreKeys.map { $0.publicKey.rawRepresentation }
        )
    }
}

8.3 Key Exchange (Simplified X3DH-like Protocol)

struct SessionKeys {
    let sendingKey: SymmetricKey
    let receivingKey: SymmetricKey
}

final class KeyExchange {

    func initiateSession(
        myIdentityKey: P256.KeyAgreement.PrivateKey,
        theirBundle: PublicKeyBundle
    ) throws -> (SessionKeys, ephemeralPublicKey: Data) {

        let theirSignedPreKey = try P256.KeyAgreement.PublicKey(
            rawRepresentation: theirBundle.signedPreKey
        )

        let ephemeralKey = P256.KeyAgreement.PrivateKey()

        let dh1 = try myIdentityKey.sharedSecretFromKeyAgreement(
            with: theirSignedPreKey
        )
        let dh2 = try ephemeralKey.sharedSecretFromKeyAgreement(
            with: theirSignedPreKey
        )

        var ikm = Data()
        ikm.append(contentsOf: dh1.withUnsafeBytes { Data($0) })
        ikm.append(contentsOf: dh2.withUnsafeBytes { Data($0) })

        if let otpkData = theirBundle.oneTimePreKeys.first {
            let theirOTPK = try P256.KeyAgreement.PublicKey(rawRepresentation: otpkData)
            let dh3 = try ephemeralKey.sharedSecretFromKeyAgreement(with: theirOTPK)
            ikm.append(contentsOf: dh3.withUnsafeBytes { Data($0) })
        }

        let salt = "e2e-messaging-v1".data(using: .utf8)!
        let masterKey = HKDF<SHA256>.deriveKey(
            inputKeyMaterial: SymmetricKey(data: ikm),
            salt: salt,
            info: "session-keys".data(using: .utf8)!,
            outputByteCount: 64
        )

        let keyData = masterKey.withUnsafeBytes { Data($0) }
        let sendKey = SymmetricKey(data: keyData[0..<32])
        let recvKey = SymmetricKey(data: keyData[32..<64])

        return (
            SessionKeys(sendingKey: sendKey, receivingKey: recvKey),
            ephemeralKey.publicKey.rawRepresentation
        )
    }
}

8.4 Message Encryption & Decryption

struct EncryptedMessage: Codable {
    let senderDeviceID: String
    let messageIndex: UInt64
    let ciphertext: Data
    let ephemeralPublicKey: Data?
}

final class MessageCrypto {

    func encrypt(
        plaintext: String,
        sessionKey: SymmetricKey,
        messageIndex: UInt64,
        senderDeviceID: String
    ) throws -> EncryptedMessage {

        let messageData = plaintext.data(using: .utf8)!

        var nonceData = Data(count: 12)
        withUnsafeMutableBytes(of: messageIndex.bigEndian) { indexBytes in
            nonceData.replaceSubrange(4..<12, with: indexBytes)
        }
        let nonce = try AES.GCM.Nonce(data: nonceData)

        let aad = "\(senderDeviceID):\(messageIndex)".data(using: .utf8)!

        let sealedBox = try AES.GCM.seal(
            messageData, using: sessionKey,
            nonce: nonce, authenticating: aad
        )

        return EncryptedMessage(
            senderDeviceID: senderDeviceID,
            messageIndex: messageIndex,
            ciphertext: sealedBox.combined!,
            ephemeralPublicKey: nil
        )
    }

    func decrypt(
        message: EncryptedMessage,
        sessionKey: SymmetricKey
    ) throws -> String {

        let sealedBox = try AES.GCM.SealedBox(combined: message.ciphertext)

        let aad = "\(message.senderDeviceID):\(message.messageIndex)"
            .data(using: .utf8)!

        let plaintext = try AES.GCM.open(
            sealedBox, using: sessionKey,
            authenticating: aad
        )

        guard let text = String(data: plaintext, encoding: .utf8) else {
            throw NSError(domain: "E2E", code: -1)
        }

        return text
    }
}

8.7 Design Decisions & Trade-offs

Decision Choice Rationale
Identity key algorithm P-256 (Secure Enclave) Hardware-backed, key never leaves device; SE only supports P-256
Message encryption AES-256-GCM AEAD, hardware-accelerated, standard choice for E2E
Key exchange Multi-DH (X3DH-like) Forward secrecy, deniability, works asynchronously
Key derivation HKDF-SHA256 Standard KDF, extracts and expands keying material
Nonce strategy Counter-based (message index) Deterministic, no nonce reuse if index is monotonic
Multi-device Per-device encryption Each device has independent keys; message sent N times for N devices
Server trust model Untrusted Server sees only ciphertext and public keys; compromise doesn't reveal messages
Key rotation Signed pre-key weekly, OTPKs on demand Limits window of compromise; OTPKs provide per-session forward secrecy
Architect Must-Know: Production Considerations
  • Double Ratchet: A production system should implement the Double Ratchet algorithm (as in Signal Protocol) for continuous forward secrecy and break-in recovery. Each message ratchets the key forward, so compromising a single message key doesn't reveal past or future messages.
  • Key Verification: Provide a mechanism for users to verify each other's identity keys out-of-band (e.g., QR code scanning, safety numbers) to prevent MITM attacks by a compromised server.
  • Sealed Sender: To hide metadata (who is messaging whom), encrypt the sender's identity inside the message envelope so the server cannot correlate sender and recipient.
  • Group Messaging: For groups, consider Sender Keys (each member distributes a symmetric key to all others) for O(1) encryption instead of O(N) per-device encryption. Trade-off: weaker forward secrecy within the group.
  • Key Storage: Store all private keys in the Keychain with kSecAttrAccessibleAfterFirstUnlockThisDeviceOnly. Identity keys should be Secure Enclave-backed. Never store keys in UserDefaults, files, or Core Data.
  • Memory Safety: Use Data with resetBytes(in:) or withUnsafeMutableBytes to zero out sensitive key material after use. Swift's ARC may delay deallocation, leaving keys in memory longer than expected.
Best Practice: Security Checklist for iOS Cryptography
  • Never roll your own crypto — use CryptoKit or Security.framework exclusively
  • Use authenticated encryption — AES-GCM, not AES-CBC (which requires separate HMAC)
  • Never reuse nonces — counter-based or random (12 bytes for AES-GCM gives ~2^96 space)
  • Derive keys with HKDF — never use raw DH output directly as an encryption key
  • Use Secure Enclave for identity/signing keys — hardware protection against extraction
  • Store secrets in Keychain — never in UserDefaults, plist, or unprotected files
  • Set correct accessibility — AfterFirstUnlock for background, WhenUnlocked for foreground-only
  • Use ThisDeviceOnly for keys that must not migrate to new devices via backup
  • Validate all inputs — check key sizes, data lengths, and algorithm support before operations
  • Handle errors explicitly — never ignore OSStatus or swallow CryptoKit errors
