1. Swift 1.0 (2014) — The Foundation

Swift was announced at WWDC 2014 as a replacement for Objective-C. Its core design principles — safety, speed, and expressiveness — shaped every subsequent version.

Key Features

Feature What It Replaced Why It Matters
Optionals ObjC nil messaging Compile-time null safety — no more silent nil failures
Type inference Verbose ObjC type declarations Reduced boilerplate while maintaining type safety
Generics ObjC id type Type-safe collections and algorithms
Value types (structs) ObjC objects (all heap-allocated) Stack allocation, copy semantics, no reference cycles
Closures ObjC blocks First-class functions with capture lists
Enums with associated values ObjC enums (integers only) Algebraic data types for modeling state
Pattern matching ObjC switch (integers only) Exhaustive matching, destructuring
ARC (no GC) ObjC manual retain/release (or GC on Mac) Deterministic memory management, no GC pauses 
// Swift 1.0 fundamentals that still define the language
let name: String? = fetchName()
if let name = name {
    print("Hello, \(name)")
}

enum Result<T> {
    case success(T)
    case failure(Error)
}

struct Point {
    var x: Double
    var y: Double
}
Deep Dive: Why ARC Instead of GC?

Apple chose ARC over garbage collection for Swift because: (1) deterministic deallocation — objects are freed immediately when their last reference is released, critical for resource management (file handles, GPU buffers); (2) no GC pauses — garbage collectors cause unpredictable latency spikes, unacceptable for 60fps UI rendering; (3) lower memory overhead — GC requires 2-3x memory headroom for efficient collection. The tradeoff is retain cycles, which require weak/unowned references.

2. Swift 2.0 (2015) — Error Handling & Protocol Extensions

Key Features

  • Error handling — try/catch/throw replaced NSError pointer patterns
  • Protocol extensions — Default implementations, enabling protocol-oriented programming
  • guard statement — Early return for precondition checking
  • defer statement — Cleanup code that runs when scope exits
  • Availability checking — #available(iOS 9, *)
  • Open-sourced — Swift became open source (December 2015)
// Protocol extensions — the foundation of protocol-oriented programming
protocol Identifiable {
    var id: String { get }
}

extension Identifiable {
    var debugIdentifier: String {
        "\(type(of: self))(\(id))"
    }
}

// Error handling replaced ObjC's NSError** pattern
enum NetworkError: Error {
    case noConnection
    case timeout(seconds: Int)
    case serverError(code: Int)
}

func fetchData() throws -> Data {
    guard isConnected() else { throw NetworkError.noConnection }
    defer { closeConnection() }
    return try download()
}

Why it matters for app developers: Protocol extensions enabled the "protocol-oriented programming" paradigm that became Swift's defining architectural pattern. Instead of class inheritance hierarchies, Swift developers compose behavior through protocol conformances with default implementations.

3. Swift 3.0 (2016) — The Grand Rename

Swift 3 was the most disruptive version for existing codebases. SE-0005 (Better Translation of Objective-C APIs into Swift) renamed virtually every API in the SDK.

The API Naming Revolution

// Before Swift 3 (Swift 2 style)
// array.indexOf(element)
// string.uppercaseString
// UIColor.redColor()
// NSBundle.mainBundle()
// dispatch_async(dispatch_get_main_queue()) { }

// After Swift 3
array.firstIndex(of: element)
string.uppercased()
UIColor.red
Bundle.main
DispatchQueue.main.async { }

Key Changes

Change Impact
SE-0005: ObjC API translation Every Cocoa API renamed — massive migration
SE-0006: Apply API guidelines to stdlib Standard library methods renamed
SE-0023: API Design Guidelines Established naming conventions still used today
GCD gets Swift wrapper DispatchQueue replaces C-style dispatch_*
NS prefix removal NSBundle → Bundle, NSDate → Date

Why it matters: Swift 3 was the last major source-breaking change. Apple committed to source stability from Swift 4 onward, meaning code written in Swift 3+ would compile in future versions (with compatibility modes).

4. Swift 4.0–4.2 (2017–2018) — Codable & Stability

Swift 4.0 — Codable

// Codable: compiler-synthesized serialization
struct User: Codable {
    let id: Int
    let name: String
    let email: String
    let joinDate: Date
}

// Encode to JSON — no manual serialization code
let encoder = JSONEncoder()
encoder.dateEncodingStrategy = .iso8601
encoder.keyEncodingStrategy = .convertToSnakeCase
let data = try encoder.encode(user)

// Key paths — type-safe property references
let namePath: KeyPath<User, String> = \User.name
let name = user[keyPath: namePath]

// String overhaul — Character is now a grapheme cluster
let emoji = "👨‍👩‍👧‍👦"
emoji.count  // 1 (was 7 in Swift 3)

Swift 4.2 — Quality of Life

// CaseIterable — auto-generated allCases
enum Direction: CaseIterable {
    case north, south, east, west
}
Direction.allCases  // [.north, .south, .east, .west]

// Unified random API
let random = Int.random(in: 1...100)
let shuffled = array.shuffled()

// Conditional conformance (Swift 4.1)
extension Array: Equatable where Element: Equatable { }
extension Optional: Equatable where Wrapped: Equatable { }

// Synthesized Equatable/Hashable for structs and enums
struct Coordinate: Hashable {
    let lat: Double
    let lon: Double
    // Hashable conformance auto-synthesized
}

Why Codable matters: Before Codable, JSON parsing required either manual dictionary extraction (error-prone) or third-party libraries (SwiftyJSON, ObjectMapper). Codable made type-safe serialization a first-class language feature, eliminating an entire category of runtime errors.

5. Swift 5.0 (2019) — ABI Stability

ABI stability was the most important infrastructure milestone in Swift's history. It meant the Swift runtime could ship with the OS instead of being bundled with every app.

What ABI Stability Means

Before ABI Stability (Swift 1–4): ┌─────────────────────────────────┐ │ Your App │ │ ┌───────────────────────────┐ │ │ │ Your Code (Swift 4.2) │ │ │ ├───────────────────────────┤ │ │ │ Swift Runtime (4.2) │ │ ← Bundled in EVERY app │ │ Swift Stdlib (4.2) │ │ ← ~5MB per app │ └───────────────────────────┘ │ └─────────────────────────────────┘ After ABI Stability (Swift 5+): ┌─────────────────────────────────┐ │ Your App │ │ ┌───────────────────────────┐ │ │ │ Your Code (Swift 5+) │ │ │ └───────────────────────────┘ │ └─────────────────────────────────┘ │ links to ┌───────────────▼─────────────────┐ │ iOS / macOS │ │ ┌───────────────────────────┐ │ │ │ Swift Runtime (shared) │ │ ← Ships with OS │ │ Swift Stdlib (shared) │ │ ← Shared by ALL apps │ └───────────────────────────┘ │ └─────────────────────────────────┘ Benefits: • App size reduced by ~5MB (no bundled runtime) • Faster app launch (runtime already loaded in memory) • OS frameworks can be written in Swift • Third-party binary frameworks become possible

Other Swift 5.0 Features

// Result type in standard library
enum Result<Success, Failure: Error> {
    case success(Success)
    case failure(Failure)
}

// Raw strings
let regex = #"\\d+\\.\\d+"#  // No double-escaping needed

// isMultiple(of:) — replaces % 2 == 0
if number.isMultiple(of: 2) { /* even */ }

// Flatten nested optionals with try?
let value = try? someThrowingFunc()  // T? not T?? in Swift 5
Deep Dive: ABI vs API vs Module Stability

ABI stability (Swift 5.0): Binary compatibility at runtime. An app compiled with Swift 5.0 can run against the Swift 5.3 runtime. The calling conventions, type metadata layout, and runtime functions are frozen.

Module stability (Swift 5.1): Binary compatibility at compile time. A framework compiled with Swift 5.1 can be imported by code compiled with Swift 5.3. Enabled by .swiftinterface files (textual module interfaces).

Library evolution (Swift 5.1): Allows a library to add new members, change implementations, and evolve without breaking binary compatibility. Requires @frozen for types that must not change layout.

6. Swift 5.1 (2019) — Module Stability & Property Wrappers

Property Wrappers (SE-0258)

// Property wrappers — reusable property behavior
@propertyWrapper
struct Clamped<Value: Comparable> {
    var wrappedValue: Value {
        didSet { wrappedValue = min(max(wrappedValue, range.lowerBound), range.upperBound) }
    }
    let range: ClosedRange<Value>

    init(wrappedValue: Value, _ range: ClosedRange<Value>) {
        self.range = range
        self.wrappedValue = min(max(wrappedValue, range.lowerBound), range.upperBound)
    }
}

struct Settings {
    @Clamped(0...100) var volume: Int = 50
    @Clamped(0.0...1.0) var brightness: Double = 0.5
}

// Property wrappers power SwiftUI's entire state system:
// @State, @Binding, @StateObject, @ObservedObject,
// @EnvironmentObject, @Environment, @Published, @AppStorage

Opaque Return Types (SE-0244)

// some — opaque return type (enables SwiftUI's body)
func makeView() -> some View {
    VStack {
        Text("Hello")
        Image(systemName: "star")
    }
}

// Without 'some', you'd need to spell out the exact type:
// VStack<TupleView<(Text, Image)>> — which changes with every body change

Why property wrappers matter: They enabled SwiftUI's entire state management system. Without property wrappers, @State, @Binding, @Published, and @Environment would not exist in their current form. They also enabled patterns like @UserDefault, @Injected, and @Clamped in app code.

8. Swift 5.5 (2021) — Concurrency Revolution

Swift 5.5 introduced the most significant language feature since Swift 1.0: structured concurrency with async/await, actors, and the Sendable protocol.

async/await (SE-0296)

// Before: Callback hell
func fetchUserProfile(completion: @escaping (Result<Profile, Error>) -> Void) {
    fetchUser { userResult in
        switch userResult {
        case .success(let user):
            fetchAvatar(for: user) { avatarResult in
                switch avatarResult {
                case .success(let avatar):
                    fetchPosts(for: user) { postsResult in
                        // ... more nesting
                    }
                case .failure(let error):
                    completion(.failure(error))
                }
            }
        case .failure(let error):
            completion(.failure(error))
        }
    }
}

// After: Linear async/await
func fetchUserProfile() async throws -> Profile {
    let user = try await fetchUser()
    async let avatar = fetchAvatar(for: user)
    async let posts = fetchPosts(for: user)
    return Profile(user: user, avatar: try await avatar, posts: try await posts)
}

Actors (SE-0306)

// Actors provide data race safety through isolation
actor BankAccount {
    private var balance: Decimal

    init(initialBalance: Decimal) {
        self.balance = initialBalance
    }

    func deposit(_ amount: Decimal) {
        balance += amount
    }

    func withdraw(_ amount: Decimal) throws {
        guard balance >= amount else {
            throw BankError.insufficientFunds
        }
        balance -= amount
    }
}

// Cross-actor calls require await
let account = BankAccount(initialBalance: 1000)
await account.deposit(500)  // Serialized access — no data races

Structured Concurrency (SE-0304)

// TaskGroup — structured parallel execution
func fetchAllImages(urls: [URL]) async throws -> [UIImage] {
    try await withThrowingTaskGroup(of: UIImage.self) { group in
        for url in urls {
            group.addTask {
                let (data, _) = try await URLSession.shared.data(from: url)
                return UIImage(data: data)!
            }
        }
        return try await group.reduce(into: []) { $0.append($1) }
    }
}

Sendable (SE-0302)

// Sendable marks types safe to share across concurrency domains
struct UserData: Sendable {
    let id: Int
    let name: String
}

// Classes need extra care
final class ImmutableConfig: Sendable {
    let apiKey: String       // OK: immutable
    let timeout: TimeInterval // OK: immutable
}

// @unchecked Sendable — "trust me, I handle synchronization"
final class ThreadSafeCache: @unchecked Sendable {
    private let lock = NSLock()
    private var storage: [String: Any] = [:]
}

Why concurrency matters: Before Swift 5.5, iOS developers used GCD (error-prone, thread explosion risk), OperationQueue (verbose), or Combine (complex). async/await provides compile-time safety against data races (with actors), eliminates callback hell, and prevents thread explosion (cooperative thread pool limited to CPU core count).

10. Swift 5.9 (2023) — Macros & Noncopyable Types

Macros (SE-0382, SE-0389, SE-0394, SE-0397)

// Macros generate code at compile time
// @Observable is a macro that generates observation infrastructure
@Observable
class UserViewModel {
    var name: String = ""
    var email: String = ""
    var isLoading: Bool = false
}

// Expands to (approximately):
// class UserViewModel {
//     @ObservationTracked var name: String = ""
//     @ObservationTracked var email: String = ""
//     @ObservationTracked var isLoading: Bool = false
//     internal let _$observationRegistrar = ObservationRegistrar()
// }

// Freestanding macros
let url = #URL("https://api.example.com/users")  // Validated at compile time

// #Predicate macro for SwiftData
let predicate = #Predicate<User> { user in
    user.age >= 18 && user.isActive
}

Noncopyable Types (SE-0390)

// ~Copyable: types that cannot be implicitly copied
struct FileHandle: ~Copyable {
    private let fd: Int32

    init(path: String) {
        self.fd = open(path, O_RDONLY)
    }

    deinit {
        close(fd)
    }

    consuming func close() {
        // Explicitly close — consumes the handle
        close(fd)
    }
}

// Cannot be copied — prevents double-close bugs
let handle = FileHandle(path: "/tmp/data")
// let copy = handle  // ERROR: cannot copy a noncopyable type

Parameter Packs (SE-0393, SE-0398)

// Variadic generics — type-safe variadic parameters
func all<each T: Equatable>(_ value: repeat each T, equals other: repeat each T) -> Bool {
    for pair in repeat (each value, each other) {
        guard pair.0 == pair.1 else { return false }
    }
    return true
}

Why macros matter: Macros replaced many uses of code generation tools (Sourcery, SwiftGen) and enabled frameworks like Observation and SwiftData to provide clean APIs with complex underlying implementations. They're the foundation of @Observable, @Model, and #Predicate.

11. Swift 6.0 (2024) — Complete Concurrency Checking

Swift 6 makes data race safety a compile-time guarantee. All Sendable violations that were warnings in Swift 5.10 become errors.

What Changes

// Swift 5: Warning — Swift 6: ERROR
var globalCounter = 0  // ❌ Mutable global is not safe

// Fix 1: Make it actor-isolated
@MainActor var globalCounter = 0

// Fix 2: Make it Sendable
let globalCounter = AtomicInt(0)  // Thread-safe type

// Fix 3: Use nonisolated(unsafe) for genuinely safe globals
nonisolated(unsafe) var globalCounter = 0  // "I guarantee safety"

// Non-Sendable type crossing actor boundary — ERROR in Swift 6
class MutableData {
    var value = 0
}

actor DataProcessor {
    func process(_ data: MutableData) {  // ❌ MutableData is not Sendable
        data.value += 1
    }
}

Typed Throws (SE-0413)

// Before: throws any Error — caller doesn't know the error type
func fetchUser() throws -> User { ... }

// Swift 6: Typed throws — caller knows the exact error type
func fetchUser() throws(NetworkError) -> User {
    guard isConnected() else {
        throw .noConnection  // Type inferred as NetworkError
    }
    return try await api.getUser()
}

// Caller can exhaustively match errors
do {
    let user = try fetchUser()
} catch .noConnection {
    showOfflineUI()
} catch .timeout(let seconds) {
    showRetry(after: seconds)
} catch .serverError(let code) {
    showServerError(code)
}
// No default case needed — compiler knows all cases

Violation Swift 5.10 Swift 6.0

Mutable capture in @Sendable closure Warning Error
Global mutable variable Warning Error

13. Architect: Swift 6 Strict Concurrency Migration Strategy

Architect Must-Know: Migrating a Large Codebase to Swift 6

Swift 6 strict concurrency checking is the most impactful migration since Swift 3. For a large codebase (500K+ lines), expect 3–6 months of focused effort. Here's the proven migration strategy.

Phase 1: Assessment (Week 1–2)

// Enable strict concurrency warnings in Swift 5.10
// In Package.swift:
.target(
    name: "MyModule",
    swiftSettings: [
        .enableExperimentalFeature("StrictConcurrency")
    ]
)

// Or in Xcode build settings:
// SWIFT_STRICT_CONCURRENCY = complete

// Count warnings per module to prioritize
// xcodebuild 2>&1 | grep "warning:" | grep "concurrency" | wc -l

Phase 2: Leaf Modules First (Week 3–8)

// Start with modules that have no internal dependencies
// Common fixes:

// 1. Make value types explicitly Sendable
struct APIResponse: Sendable {
    let data: Data
    let statusCode: Int
}

// 2. Use @unchecked Sendable for types with internal synchronization
final class ThreadSafeCache<Key: Hashable, Value>: @unchecked Sendable {
    private let lock = NSLock()
    private var storage: [Key: Value] = [:]
}

// 3. Convert global mutable state
// Before:
var sharedConfig = AppConfig()  // ❌ Global mutable

// After:
@MainActor var sharedConfig = AppConfig()  // ✅ Actor-isolated

// 4. Fix @Sendable closure captures
// Before:
var results: [String] = []
Task {
    results.append("done")  // ❌ Mutable capture in @Sendable
}

// After:
let result = await Task {
    return "done"
}.value

Phase 3: Core Modules (Week 9–16)

// Convert manager/service classes to actors
// Before:
class NetworkManager {
    private let queue = DispatchQueue(label: "network")
    private var cache: [URL: Data] = [:]

    func fetch(_ url: URL, completion: @escaping (Data?) -> Void) {
        queue.async { [weak self] in
            if let cached = self?.cache[url] {
                completion(cached)
                return
            }
            // ... network call
        }
    }
}

// After:
actor NetworkManager {
    private var cache: [URL: Data] = [:]

    func fetch(_ url: URL) async throws -> Data {
        if let cached = cache[url] {
            return cached
        }
        let (data, _) = try await URLSession.shared.data(from: url)
        cache[url] = data
        return data
    }
}

Phase 4: Enable Swift 6 Per-Target

// Enable Swift 6 mode per target as each is ready
// Package.swift:
.target(
    name: "ReadyModule",
    swiftSettings: [
        .swiftLanguageMode(.v6)
    ]
)

// Modules still being migrated stay on Swift 5:
.target(
    name: "LegacyModule",
    swiftSettings: [
        .swiftLanguageMode(.v5)
    ]
)
Best Practice: Swift 6 Migration Checklist
  1. Enable -strict-concurrency=complete on Swift 5.10 first
  2. Fix warnings module-by-module, starting with leaf modules (no dependencies)
  3. Use @unchecked Sendable as a temporary bridge — track with // TODO: Remove @unchecked
  4. Convert DispatchQueue-protected state to actors
  5. Use nonisolated(unsafe) sparingly for genuinely safe globals
  6. Enable Swift 6 language mode per-target as each is ready
  7. Remove @unchecked annotations as you properly fix each type
  8. Run Thread Sanitizer in CI to catch any remaining runtime races
