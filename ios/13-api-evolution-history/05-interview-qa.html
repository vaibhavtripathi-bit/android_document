API Evolution & Platform History — Interview Q&A

UIKit Evolution (Questions 1–10)

1. Why does every UIView have a backing CALayer? What's the historical reason for this split? L6

The UIView/CALayer split is inherited from macOS (AppKit). On macOS, NSView originally did its own drawing without layers. When Core Animation was introduced (Mac OS X 10.5), Apple added an optional "layer-backed" mode. On iOS, Apple made every view layer-backed from the start, but kept the split because:

  • Separation of concerns: UIView handles event handling (touches, hit testing, responder chain) while CALayer handles rendering (compositing, animation, GPU interaction)
  • Cross-platform sharing: CALayer is shared between iOS and macOS. UIView (iOS) and NSView (macOS) are platform-specific wrappers
  • Render server architecture: CALayer properties are committed to the render server (backboardd) for GPU compositing. The view layer provides the bridge between app-process logic and out-of-process rendering

This is why you can use CALayer directly for lightweight rendering without the overhead of a UIView (no event handling, no Auto Layout).

2. What was the significance of iOS 13's UIScene introduction? What architectural patterns did it break? Architect

UIScene (iOS 13) introduced multi-window support, fundamentally changing the app lifecycle model. Before iOS 13, UIApplication was a singleton that owned a single window. After iOS 13, each window is owned by a UIWindowScene, and multiple scenes can exist simultaneously.

Patterns it broke:

  • Singleton window access: UIApplication.shared.keyWindow became ambiguous with multiple scenes. Code that assumed a single window crashed or behaved incorrectly.
  • AppDelegate as lifecycle owner: Lifecycle events (foreground/background) moved to SceneDelegate. Apps that put all lifecycle logic in AppDelegate needed refactoring.
  • Global state tied to "the app": State that should be per-window (e.g., navigation state, selected tab) was often stored globally. Multi-window exposed these as bugs.
  • State restoration: The old NSCoder-based restoration didn't work with scenes. New NSUserActivity-based restoration was needed.

The migration strategy was to move window management and lifecycle handling to SceneDelegate while keeping process-level events (push notifications, URL handling) in AppDelegate.

3. Explain UICollectionViewCompositionalLayout's architecture. Why was it a significant improvement over FlowLayout? L6

CompositionalLayout uses a three-level hierarchy: Item → Group → Section. Each section can have a completely different layout, and sections are composed together into the full layout.

Key improvements over FlowLayout:

  • Per-section layouts: FlowLayout applied one layout to the entire collection. CompositionalLayout lets each section have its own item size, group arrangement, and scrolling behavior.
  • Orthogonal scrolling: Sections can scroll horizontally within a vertically scrolling collection — the "App Store" layout pattern. FlowLayout couldn't do this without nested collection views.
  • Fractional sizing: Items can be sized as fractions of their container (.fractionalWidth(0.5)), making responsive layouts trivial.
  • Declarative: Layout is defined as a composition of value types, not by implementing delegate methods with index path switching.
  • Performance: Apple optimized the layout engine for CompositionalLayout — it pre-computes layout attributes more efficiently than FlowLayout for complex layouts.
4. How does UIButton.Configuration differ from the legacy button customization API? Why was the change made? Senior

Legacy UIButton customization used scattered properties: setTitle(_:for:), setImage(_:for:), backgroundColor, contentEdgeInsets, layer.cornerRadius. Each property was set independently, and state-based appearance required setting each property for each state (.normal, .highlighted, .disabled).

UIButton.Configuration (iOS 15) provides a single struct that encapsulates all button appearance. Benefits:

  • Unified API: All appearance in one struct instead of scattered across properties
  • Dynamic updates: configurationUpdateHandler is called when button state changes, allowing dynamic appearance updates in one place
  • Built-in styles: .filled(), .tinted(), .gray(), .plain(), .bordered() match system button styles
  • Activity indicator: Built-in showsActivityIndicator replaces manual spinner management
  • Subtitle support: Native subtitle text without custom views

The two APIs are mutually exclusive — setting a configuration clears legacy properties and vice versa.

5. What changed in the iOS 17 trait system? Why was traitCollectionDidChange deprecated? L6

traitCollectionDidChange(_:) was called for any trait change — size class, dark mode, Dynamic Type, accessibility contrast, and more. Every view/view controller had to check which trait actually changed, leading to unnecessary work and bugs where developers forgot to check.

iOS 17 introduced:

  • registerForTraitChanges(_:handler:): Subscribe to specific traits. Only called when those specific traits change.
  • UITraitDefinition: Define custom traits that propagate through the view hierarchy.
  • traitOverrides: Override traits at any level of the hierarchy (view, view controller, window, scene).
  • UIMutableTraits: Mutable trait access for custom trait definitions.

The new system is more efficient (no unnecessary callbacks), more expressive (custom traits), and more flexible (overrides at any level).

6. How does UIContentConfiguration work? What problem does it solve compared to directly configuring cell subviews? Senior

UIContentConfiguration is a protocol that separates content definition from content rendering. Instead of directly accessing cell.textLabel or custom subviews, you create a configuration struct and assign it to cell.contentConfiguration.

Problems it solves:

  • Reusability: The same configuration can be used across different cell types (UITableViewCell, UICollectionViewCell, UICollectionViewListCell)
  • State-aware updates: updated(for:) method receives the cell's state (highlighted, selected, disabled) and returns an adjusted configuration
  • Composability: You can create custom configurations that compose with system configurations
  • SwiftUI bridge: UIHostingConfiguration (iOS 16) implements UIContentConfiguration, allowing SwiftUI views as cell content
7. What is UIAppearance's mechanism? Why does it only work with @objc dynamic properties? L6

UIAppearance uses Objective-C's message forwarding. When you call UINavigationBar.appearance().tintColor = .blue, you're not setting the property on a real navigation bar — you're sending a message to a proxy object that records the invocation. Later, when a real UINavigationBar is added to the view hierarchy, the recorded invocations are replayed on it.

This requires @objc dynamic because:

  • @objc: The proxy uses NSInvocation to record method calls, which is an Objective-C runtime feature
  • dynamic: The property must use Objective-C message dispatch (not Swift's static/vtable dispatch) so the proxy can intercept the call

Pure Swift properties use static dispatch and are invisible to the ObjC runtime, so UIAppearance cannot intercept them.

8. What is the difference between size classes and trait collections? How do they relate? Senior

Size classes are one component of trait collections. A size class is either Compact or Regular, and there are two: horizontal and vertical. They abstract device dimensions into categories for adaptive layout.

Trait collections (UITraitCollection) are containers that hold multiple traits, including but not limited to size classes. Other traits include: userInterfaceStyle (light/dark), userInterfaceIdiom (phone/pad/mac), displayScale (1x/2x/3x), preferredContentSizeCategory (Dynamic Type), accessibilityContrast, layoutDirection (LTR/RTL), and custom traits (iOS 17+).

Trait collections propagate through the view hierarchy: UIScreen → UIWindowScene → UIWindow → UIViewController → UIView. Each level can override traits for its children.

9. How would you design a migration plan for moving from UICollectionViewFlowLayout to CompositionalLayout in a large app? Architect

Phase 1 — Audit: Identify all collection views using FlowLayout. Categorize by complexity: simple grids (easy), custom delegate sizing (medium), nested collection views for horizontal scrolling (hard — these benefit most from CompositionalLayout's orthogonal scrolling).

Phase 2 — Abstract: Create a layout factory protocol. Existing screens use FlowLayout through the factory. New screens use CompositionalLayout through the same factory.

Phase 3 — Migrate bottom-up: Start with the simplest collection views (uniform grids). Move to complex layouts. Save nested collection views (horizontal-in-vertical) for last — these see the biggest improvement but require the most refactoring.

Phase 4 — Migrate data sources: Move from traditional data sources to DiffableDataSource simultaneously, since CompositionalLayout works best with DiffableDataSource.

Key risk: CompositionalLayout requires iOS 13+. If your minimum target is lower, you need #available checks and dual implementations until you drop the old target.

10. What was the impact of iOS 7's flat design on app architecture? Why was it more than just a visual change? L6

iOS 7's flat design was architecturally significant because:

  • UIKit Dynamics: The new physics-based animation system required rethinking how interactive elements responded to user input
  • View controller transitions API: Custom modal and navigation transitions became first-class, enabling new navigation patterns
  • Translucent bars: Navigation and tab bars became translucent by default, requiring content to extend behind them. This broke layouts that assumed opaque bars.
  • Edge-to-edge content: edgesForExtendedLayout and topLayoutGuide/bottomLayoutGuide were introduced to manage content insets
  • TextKit: The new text rendering system replaced most Core Text usage, changing how text-heavy apps were built
  • Tint color propagation: tintColor propagated through the view hierarchy, changing how theming worked

Every custom UI component needed redesigning — not just visual updates, but structural changes to handle translucency, edge-to-edge layout, and new interaction patterns.

Swift Evolution (Questions 11–20)

11. What is ABI stability and why was it the most important milestone in Swift's history? L6

ABI (Application Binary Interface) stability means the Swift runtime's binary interface is frozen — the calling conventions, type metadata layout, and runtime functions won't change between Swift versions. This has three major consequences:

  • Runtime ships with OS: Before ABI stability, every app bundled its own Swift runtime (~5MB). After, the runtime ships with iOS/macOS and is shared by all apps. This reduces app size and improves launch time (runtime is already in memory).
  • OS frameworks in Swift: Apple can now write OS frameworks in Swift (SwiftUI, Observation, SwiftData) because the runtime is guaranteed to be present and compatible.
  • Binary frameworks: Third-party libraries can distribute pre-compiled binary frameworks (XCFrameworks) that work across Swift versions.

Without ABI stability, Swift could never have become a first-class platform language — every OS update would have required recompiling every app.

12. Explain the difference between ABI stability, module stability, and library evolution. When does each matter? Architect

ABI stability (Swift 5.0): Runtime binary compatibility. An app compiled with Swift 5.0 can run against the Swift 5.x runtime. Matters for: app size, launch time, OS framework adoption.

Module stability (Swift 5.1): Compile-time binary compatibility. A framework compiled with Swift 5.1 can be imported by code compiled with Swift 5.3. Enabled by .swiftinterface files. Matters for: distributing binary frameworks (XCFrameworks).

Library evolution (Swift 5.1): Allows a library to change its implementation without breaking binary compatibility. New members can be added, implementations can change. Requires @frozen for types whose layout must not change. Matters for: OS frameworks that evolve across iOS versions.

For app developers, ABI stability is automatic. Module stability matters if you distribute binary frameworks. Library evolution matters if you maintain a framework used by others as a binary dependency.

13. Why did Apple choose property wrappers for SwiftUI's state management? What alternatives were considered? L6

Property wrappers (SE-0258) were designed specifically to enable SwiftUI's state system. The alternatives and why they were rejected:

  • Subclassing: Would require all state to be in a base class, limiting composition
  • Protocol requirements: Would require boilerplate conformance in every view
  • Compiler magic: Would make the behavior opaque and non-extensible
  • Macros: Didn't exist yet (came in Swift 5.9)

Property wrappers were chosen because they: (1) encapsulate storage and access patterns in a reusable type, (2) compose naturally (you can create custom wrappers), (3) have a clear mental model (wrappedValue + projectedValue), and (4) are a general-purpose language feature, not SwiftUI-specific.

The projected value ($ prefix) was specifically designed for SwiftUI's binding system: $state gives you a Binding.

14. What are Swift macros and why were they introduced? How do they differ from C preprocessor macros? L6

Swift macros (Swift 5.9) are compile-time code generation tools that operate on the AST (Abstract Syntax Tree), not on text. Key differences from C macros:

  • Type-safe: Swift macros receive typed AST nodes, not raw text. They can't produce syntactically invalid code.
  • Sandboxed: Macros run in a separate process (sandbox) and can only generate code — they can't read files, make network calls, or have side effects.
  • Expandable: You can "expand macro" in Xcode to see exactly what code is generated.
  • Testable: Macro implementations can be unit tested with SwiftSyntax.

They were introduced to enable: @Observable (generates observation infrastructure), @Model (generates SwiftData schema), #Predicate (generates type-safe predicates), and to replace code generation tools like Sourcery.

15. What are noncopyable types (~Copyable) and what problem do they solve? L6

Noncopyable types (~Copyable, Swift 5.9) are types that cannot be implicitly copied. They enforce unique ownership — only one variable can hold the value at a time.

Problems they solve:

  • Resource management: File handles, database connections, GPU buffers — resources that must be closed exactly once. Copying would create double-close bugs.
  • Ownership tracking: Ensures a value is consumed (moved) rather than copied, preventing use-after-move bugs.
  • Performance: Eliminates unnecessary copies for large value types.

Noncopyable types use consuming and borrowing parameter modifiers to express ownership transfer. A consuming function takes ownership (caller can't use the value after), while borrowing temporarily accesses without taking ownership.

16. What is typed throws in Swift 6? How does it improve error handling? Senior

Before Swift 6, throws meant "throws any Error" — the caller didn't know the specific error type at compile time. Typed throws (throws(MyError)) specifies the exact error type.

Benefits:

  • Exhaustive catching: The compiler knows all possible error cases, so catch can be exhaustive without a default case
  • No type erasure: The error type is preserved through the call chain, avoiding as? downcasting
  • Better API contracts: Callers know exactly what can go wrong
  • Performance: No existential boxing of the error value

Untyped throws still works and is equivalent to throws(any Error). Typed throws is opt-in for APIs that want to expose their error types.

17. How would you migrate a 500K-line codebase to Swift 6 strict concurrency? What's your phased approach? Architect

Phase 1 — Assessment (2 weeks): Enable -strict-concurrency=complete on Swift 5.10. Count warnings per module. Identify the dependency graph — leaf modules (no internal dependencies) are migrated first.

Phase 2 — Leaf modules (4-6 weeks): Fix Sendable conformances for value types. Add @unchecked Sendable as temporary bridges for classes with internal synchronization. Fix global mutable state (actor-isolate or use nonisolated(unsafe)).

Phase 3 — Core modules (6-8 weeks): Convert DispatchQueue-protected managers to actors. Migrate callback-based APIs to async/await using withCheckedContinuation. Fix @Sendable closure capture issues.

Phase 4 — Enable per-target (2-4 weeks): Enable Swift 6 language mode per-target as each is ready. Keep legacy modules on Swift 5 mode. Remove @unchecked Sendable annotations as types are properly fixed.

Key principle: Never block feature development. Migration happens alongside feature work, not instead of it.

18. Why was the Swift 3 "grand rename" necessary? What design principles drove it? Senior

Swift 3's grand rename (SE-0005, SE-0006, SE-0023) was necessary because Swift 1-2 APIs were mechanically translated from Objective-C, preserving ObjC naming conventions that didn't fit Swift's syntax.

Key principles (from the API Design Guidelines):

  • Clarity at the point of use: Names should read as English phrases at call sites
  • Omit needless words: removeObject(object) → remove(object)
  • First argument label: Include when it clarifies the role: move(from: a, to: b)
  • Fluent usage: x.insert(y, at: z) reads as "x, insert y at z"

Apple committed to source stability from Swift 4 onward, meaning Swift 3 was the last major source-breaking change. This was a deliberate "break it once, break it right" decision.

19. What is the Swift evolution process? How does a feature go from idea to release? Senior

The Swift evolution process is an open, community-driven process:

  1. Pitch: Author posts an idea on the Swift Forums for informal discussion
  2. Proposal: If the pitch has support, a formal proposal (SE-XXXX) is written with motivation, detailed design, alternatives considered, and source compatibility impact
  3. Review: The community reviews the proposal on the forums for 1-2 weeks
  4. Decision: The Swift Core Team accepts, rejects, or returns for revision
  5. Implementation: The feature is implemented in the Swift compiler (often as an experimental feature first)
  6. Release: Ships in a Swift version bundled with Xcode

Notable: Apple engineers participate but don't have veto power. Community members have successfully proposed and implemented major features. The process is transparent — all proposals, reviews, and decisions are public.

20. Compare Swift's approach to concurrency safety (actors + Sendable) with Rust's ownership model. What are the tradeoffs? Architect

Swift's approach: Runtime isolation through actors. Actors serialize access to their state. Sendable marks types safe to share. The compiler checks Sendable conformance at actor boundaries. Data race safety is a compile-time property (in Swift 6).

Rust's approach: Compile-time ownership tracking. Each value has exactly one owner. References are either shared-immutable (&T) or exclusive-mutable (&mut T), never both. The borrow checker enforces this at compile time.

Tradeoffs:

  • Learning curve: Swift's actor model is more familiar to OOP developers. Rust's ownership model requires a fundamental shift in thinking.
  • Granularity: Rust prevents races at the variable level. Swift prevents races at the actor boundary level — within an actor, you can still have logic bugs.
  • Performance: Rust has zero runtime overhead (all checks at compile time). Swift actors have runtime overhead (executor scheduling, message passing).
  • Escape hatches: Swift has @unchecked Sendable and nonisolated(unsafe). Rust has unsafe blocks. Both allow opting out of safety.

SwiftUI Adoption (Questions 21–30)

21. When would you recommend adopting SwiftUI for a new feature in a large UIKit app? What's your decision framework? Architect

My decision framework evaluates five factors:

  1. Minimum deployment target: iOS 16+ makes SwiftUI viable (NavigationStack). iOS 15 is borderline. Below iOS 15, UIKit is safer.
  2. Screen complexity: Forms, settings, lists, dashboards → SwiftUI. Custom transitions, camera, complex gestures → UIKit.
  3. Team readiness: Does the team have SwiftUI experience? SwiftUI has a steep debugging curve (opaque view types, identity issues).
  4. Integration surface: How much does the new feature interact with existing UIKit code? Heavy interaction → more bridging code → more risk.
  5. Timeline pressure: SwiftUI is faster for simple screens but slower when you hit gaps that require UIViewRepresentable workarounds.

My recommendation for 2025: Use SwiftUI for new leaf screens (settings, profile, onboarding). Keep UIKit for the navigation shell and complex interactive views. Use UIHostingConfiguration for SwiftUI cells in UIKit collection views.

22. What was wrong with NavigationView? Why did Apple replace it with NavigationStack? L6

NavigationView had fundamental design problems:

  • Eager destination creation: NavigationLink(destination: DetailView()) created the destination view immediately, even if the user never navigated to it. For lists with 1000 items, this created 1000 detail views.
  • No programmatic navigation: You couldn't push a view from code without the isActive binding hack (hidden NavigationLink with EmptyView label).
  • No pop-to-root: There was no clean way to pop to the root of the navigation stack.
  • No deep linking: You couldn't set the navigation stack from a URL or notification.
  • Inconsistent column behavior: NavigationView showed a sidebar on iPad but a stack on iPhone, with no way to control this behavior.

NavigationStack (iOS 16) fixed all of these: lazy destination creation via navigationDestination(for:), programmatic navigation via NavigationPath, pop-to-root via path = NavigationPath(), deep linking via path manipulation, and NavigationSplitView for explicit column control.

23. How does @Observable differ from ObservableObject? Why is it better for SwiftUI performance? L6

ObservableObject uses Combine's objectWillChange publisher. When any @Published property changes, it sends a single notification. SwiftUI treats this as "something changed" and re-evaluates the body of every view that observes the object — even views that only read unchanged properties.

@Observable uses the Observation framework's per-property tracking. During body evaluation, the framework records which properties each view actually reads. When a property changes, only views that read that specific property are invalidated.

Performance impact: For a view model with 10 properties observed by 10 different views, ObservableObject triggers 10 view updates for any single property change. @Observable triggers only 1 view update. This is a 10x reduction in unnecessary work.

API simplification: @Observable eliminates @Published, @ObservedObject, @StateObject, and @EnvironmentObject. You just use plain properties and pass the object directly.

24. What are the main gaps in SwiftUI that still require UIKit fallbacks? Senior

As of iOS 17/SwiftUI 5.0, the main gaps requiring UIKit are:

  • Rich text editing: TextEditor only supports plain text. For attributed text, you need UITextView via UIViewRepresentable.
  • Camera/video capture: No native SwiftUI camera view. Requires AVCaptureSession + UIViewRepresentable.
  • Web views: No native SwiftUI web view. Requires WKWebView + UIViewRepresentable.
  • Custom view controller transitions: SwiftUI has no equivalent to UIViewControllerTransitioningDelegate.
  • Complex collection layouts: LazyVGrid/LazyHGrid are less flexible than UICollectionViewCompositionalLayout for complex layouts with orthogonal scrolling, decorations, and supplementary views.
  • MapKit advanced features: SwiftUI's Map view covers basics but lacks full MapKit feature parity.
25. How does the .task modifier work? Why is it better than creating a Task in onAppear? Senior

The .task modifier creates a structured Task tied to the view's lifecycle. When the view appears, the task starts. When the view disappears, the task is automatically cancelled.

vs. Task in onAppear:

  • .task automatically cancels on disappear. Task { } in onAppear creates an unstructured task that continues running after the view disappears — you must manually cancel it in onDisappear.
  • .task(id:) restarts when the id changes — the old task is cancelled and a new one starts. This is perfect for search-as-you-type: .task(id: searchText) { await search(searchText) }.
  • .task is structured concurrency — it's a child of the view's task context, so cancellation propagates correctly.

Always prefer .task over onAppear { Task { } } for async work in SwiftUI views.

26. What is the Layout protocol in SwiftUI? When would you create a custom Layout? L6

The Layout protocol (iOS 16) lets you create custom container layouts. You implement two methods: sizeThatFits(proposal:subviews:cache:) to calculate the container's size, and placeSubviews(in:proposal:subviews:cache:) to position each child.

Use cases for custom Layout:

  • Flow layout: Tags/chips that wrap to the next line when they exceed the container width
  • Radial layout: Items arranged in a circle
  • Masonry/waterfall layout: Pinterest-style columns with variable heights
  • Animated layout transitions: Using AnyLayout to animate between different layouts

Before the Layout protocol, these required GeometryReader hacks or UIKit fallbacks. The Layout protocol provides a clean, performant, and animatable solution.

27. How would you implement deep linking in a SwiftUI app using NavigationStack? L6

NavigationStack's NavigationPath is the key. It's a type-erased stack of Hashable values that can be manipulated programmatically.

Strategy:

  1. Define route types that conform to Hashable and Codable
  2. Parse the deep link URL into a sequence of route values
  3. Set the NavigationPath to the parsed route sequence
  4. Register navigationDestination(for:) for each route type

For state restoration, NavigationPath conforms to Codable, so you can serialize it to UserDefaults or SceneStorage and restore it on next launch. This was impossible with NavigationView.

28. What is UIHostingConfiguration and why is it significant for hybrid UIKit/SwiftUI apps? Senior

UIHostingConfiguration (iOS 16) implements UIContentConfiguration, allowing you to use SwiftUI views as the content of UICollectionView and UITableView cells. This is significant because:

  • Incremental adoption: You can use SwiftUI for cell content without rewriting the entire collection view
  • Best of both worlds: UICollectionView's powerful layout system (CompositionalLayout) + SwiftUI's declarative cell content
  • Self-sizing: SwiftUI views automatically determine their size, working with self-sizing cells
  • State management: SwiftUI's @Observable works inside hosting configurations

This is the recommended bridge pattern for large UIKit apps adopting SwiftUI incrementally.

29. What were the biggest SwiftUI 1.0 mistakes that developers made? What lessons were learned? L6

Mistake 1: Full SwiftUI rewrite: Teams rewrote entire apps in SwiftUI 1.0, then discovered missing features (search, pull-to-refresh, programmatic navigation) too late. Lesson: Adopt incrementally, starting with leaf screens.

Mistake 2: Fighting the framework: Developers tried to force UIKit patterns into SwiftUI (imperative state updates, manual view lifecycle). Lesson: Embrace declarative patterns — state drives UI, not the other way around.

Mistake 3: Overusing @ObservedObject: Before @StateObject (iOS 14), using @ObservedObject for owned objects caused the object to be recreated on every parent re-render. Lesson: Understand ownership semantics.

Mistake 4: GeometryReader everywhere: Developers used GeometryReader for sizing that should have been handled by the layout system. Lesson: GeometryReader is a last resort, not a first choice.

Mistake 5: Ignoring identity: Not understanding structural vs explicit identity caused state loss and performance issues. Lesson: Learn how SwiftUI identifies views.

30. How do you measure SwiftUI performance? What tools are available? L6

Built-in tools:

  • Self._printChanges(): Prints which properties caused a view's body to be re-evaluated. Add to body: let _ = Self._printChanges()
  • Instruments SwiftUI template: Shows view body evaluations, update counts, and time spent in body
  • Core Animation template: Shows frame drops, off-screen rendering, blended layers
  • Time Profiler: Shows CPU time in SwiftUI layout and rendering

Common performance issues:

  • Excessive body re-evaluations (fix: @Observable, equatable views, stable identity)
  • Non-lazy containers loading all items (fix: LazyVStack/LazyHStack)
  • Heavy computation in body (fix: move to .task or computed properties)
  • AnyView type erasure (fix: use conditional views or generics instead)

Deprecation & Migration (Questions 31–40)

31. Why was UIWebView deprecated? What was the technical reason it couldn't just be updated? L6

UIWebView ran JavaScript in the app's process. iOS security policy prohibits JIT (Just-In-Time) compilation in the app process because JIT requires writable+executable memory pages (W^X violation), which is a security risk for code injection attacks.

Without JIT, UIWebView's JavaScript engine ran in interpreter mode — approximately 3-4x slower than Safari. WKWebView solved this by running in a separate com.apple.WebKit.WebContent process with its own sandbox that is allowed JIT. This multi-process architecture couldn't be retrofitted into UIWebView because it was fundamentally designed as an in-process view.

Additionally, UIWebView shared the app's memory budget. A complex web page could consume 100MB+ of memory, pushing the app toward Jetsam limits. WKWebView's separate process has its own memory budget.

32. What is Apple's typical deprecation timeline? How long do you have to migrate? Senior

Apple follows a predictable four-stage deprecation pattern:

  1. Soft deprecation (Year 0): New API introduced alongside old. No warnings. Example: WKWebView introduced in iOS 8 (2014).
  2. Formal deprecation (Year 2-4): Compiler warning added. Example: UIWebView deprecated in iOS 12 (2018).
  3. App Store rejection (Year 4-6): New submissions rejected. Example: UIWebView rejected from April 2020.
  4. Removal (Year 6+): API removed from SDK. Rare but happens (AddressBook framework).

The gap between soft deprecation and App Store rejection is typically 4-6 years. However, the gap between formal deprecation and rejection can be as short as 1-2 years. Start migration at formal deprecation, not at rejection.

33. How would you manage a large-scale migration from Objective-C to Swift? What's the strategy? Architect

Strategy: Incremental, not big-bang.

  1. New code in Swift: All new files are Swift. No exceptions.
  2. Modified files: When making significant changes to an ObjC file, rewrite it in Swift.
  3. Stable files: Leave working ObjC code alone. Migration for migration's sake wastes time and introduces bugs.
  4. Test files first: Migrate test files to Swift — lower risk, better assertion APIs.
  5. Bottom-up: Migrate leaf modules (models, utilities) before UI code.

Technical considerations:

  • Bridging header for ObjC → Swift. @objc annotation for Swift → ObjC.
  • Swift classes used from ObjC must inherit from NSObject.
  • Swift enums with associated values are invisible to ObjC.
  • Swift generics are invisible to ObjC.
  • Measure: Track Swift/ObjC line ratio over time. Set a target (e.g., 80% Swift by Q4).
34. Why did storyboards fall out of favor? Are they formally deprecated? Senior

Storyboards are not formally deprecated — they still work and Apple still includes them in Xcode templates. However, the industry has moved away from them for practical reasons:

  • Merge conflicts: Storyboard XML causes frequent, unresolvable git conflicts in teams
  • String-based identifiers: Segue IDs, cell reuse IDs — runtime crashes from typos
  • Untestable navigation: Navigation logic embedded in segues can't be unit tested
  • Unreadable diffs: XML diffs are meaningless in code review
  • SwiftUI: SwiftUI provides declarative UI without storyboard drawbacks

The migration path is: Storyboards → Programmatic UIKit (with coordinators) → SwiftUI. Many large apps skipped programmatic UIKit and went directly from storyboards to SwiftUI for new features.

35. What patterns should you use to manage deprecation warnings in a large codebase? Architect

1. Track warning count in CI: Add a build step that counts deprecation warnings and fails if the count increases. This prevents new deprecation debt.

2. Wrapper/abstraction layers: Create protocol wrappers around deprecated APIs. All call sites use the wrapper. The wrapper contains the #available check and both old/new implementations.

3. Suppress strategically: Use @available(iOS, deprecated: 15) on wrapper methods to suppress warnings at call sites while keeping them visible in the wrapper itself.

4. Migration tickets: Create tickets for each deprecated API usage. Tag with priority based on deprecation timeline. Include in sprint planning.

5. Feature flags: Gate new implementations behind feature flags for gradual rollout. Monitor crash rates per code path.

6. Deadline: Set a hard deadline for migration completion. Without a deadline, deprecation migrations never finish.

36. Compare NSURLConnection to URLSession. What architectural improvements did URLSession bring? Senior

Configuration: NSURLConnection used shared, global configuration. URLSession has per-session configuration (URLSessionConfiguration) — default, ephemeral (no persistence), and background (survives app suspension).

Background transfers: NSURLConnection couldn't continue when the app was suspended. URLSession's background configuration hands transfers to a system daemon that continues downloading/uploading even after the app is killed.

Task hierarchy: URLSession has typed tasks — URLSessionDataTask, URLSessionDownloadTask, URLSessionUploadTask, URLSessionStreamTask, URLSessionWebSocketTask — each optimized for its use case.

HTTP/2: URLSession supports HTTP/2 multiplexing (multiple requests over a single TCP connection). NSURLConnection was HTTP/1.1 only.

Modern API: URLSession has completion handlers, Combine publishers, and async/await APIs. NSURLConnection only had delegates.

37. Why is the move from GCD to async/await significant? What problems does it solve that GCD couldn't? L6

Thread explosion: GCD can create 64+ threads when work blocks. async/await uses a cooperative thread pool limited to CPU core count (~6-8). Tasks suspend at await points instead of blocking threads.

Data race safety: GCD provides no compile-time safety. You can access shared state from any queue without the compiler warning you. Actors + Sendable provide compile-time data race prevention.

Structured cancellation: GCD's DispatchWorkItem cancellation is manual and error-prone. Structured concurrency automatically cancels child tasks when the parent is cancelled.

Priority inversion: GCD requires manual priority boosting. Swift concurrency has automatic priority escalation when a high-priority task awaits a low-priority one.

Readability: Nested GCD callbacks create "pyramid of doom." async/await reads linearly like synchronous code.

GCD is not deprecated and remains important for low-level system programming, but async/await is the recommended approach for app-level concurrency.

38. What is the migration path from Combine to the Observation framework? Should you migrate existing Combine code? Architect

Migration path:

  • UI state (ViewModels): Migrate from ObservableObject + @Published to @Observable. This gives per-property tracking and eliminates unnecessary view updates.
  • Reactive chains: Keep Combine for complex operator chains (debounce, combineLatest, switchToLatest) that have no AsyncSequence equivalent.
  • Streaming data: Migrate from Combine publishers to AsyncStream/AsyncSequence for new streaming APIs.

Should you migrate existing Combine code? Only if: (1) you're already modifying the code for other reasons, (2) you're experiencing performance issues from whole-object invalidation, or (3) you're targeting iOS 17+ exclusively. Don't rewrite working Combine code just to use @Observable.

Combine is not deprecated and likely won't be for years. It still has unique capabilities (backpressure, rich operators) that Observation doesn't replace.

39. What deprecation patterns can you identify across iOS history? How do you predict future deprecations? Architect

Recurring patterns:

  • C API → Swift-native: AddressBook → Contacts, AudioToolbox → AVFoundation, Security → CryptoKit
  • Delegate → Closure/Async: UIAlertViewDelegate → UIAlertAction, NSURLConnectionDelegate → URLSession async
  • String-based → Type-safe: Segue IDs → programmatic, cell reuse IDs → CellRegistration
  • Imperative → Declarative: reloadData → DiffableDataSource, GCD → async/await, KVO → Observation
  • In-process → Out-of-process: UIWebView → WKWebView

Predicted future deprecations: Combine for UI state (→ Observation), Core Data (→ SwiftData), NSAttributedString (→ AttributedString), manual GCD for app code (→ async/await). UIKit itself will likely never be deprecated but will become the "advanced" option while SwiftUI becomes the default.

40. How do you handle the N-2 support constraint when adopting new APIs? Give a concrete example. Architect

The N-2 rule means supporting the current iOS version and two prior. As of 2025, this means iOS 16+.

Concrete example: Adopting @Observable (iOS 17)

Since @Observable requires iOS 17 and our minimum is iOS 16, we can't use it unconditionally. Strategy:

  1. Create a protocol that abstracts the view model interface
  2. Implement with ObservableObject for iOS 16
  3. Implement with @Observable for iOS 17+
  4. Use #available to select the implementation
  5. When minimum target reaches iOS 17 (likely 2026), remove the ObservableObject implementation

For simpler cases, use if #available(iOS 17, *) directly. For complex cases, use the abstraction layer approach. The key is planning the cleanup date when the old code path can be removed.

Platform History & Architecture (Questions 41–45)

41. What are the most significant architectural shifts in iOS history? Rank the top 5 and explain why. Architect

1. Swift introduction (2014): Changed the language of the entire platform. Enabled type safety, value types, protocol-oriented programming, and eventually concurrency safety. Every subsequent framework (SwiftUI, SwiftData, Observation) is Swift-first.

2. async/await + actors (2021): The most significant concurrency change since GCD. Compile-time data race safety, cooperative threading, structured concurrency. Changes how every async operation is written.

3. SwiftUI (2019): Declarative UI paradigm shift. Even if you don't use SwiftUI, it influences how UIKit evolves (UIContentConfiguration, UIHostingConfiguration). It's the future of Apple platform UI.

4. UIScene lifecycle (2019): Multi-window support broke the singleton app model. Required rethinking state management, lifecycle handling, and navigation architecture.

5. ABI stability (2019): Enabled Swift to become a platform language. Without it, no SwiftUI, no SwiftData, no Observation framework. The infrastructure that made everything else possible.

42. If you were starting a new large-scale iOS app today, what technology choices would you make and why? Architect

Language: Swift 6 with strict concurrency from day one. No Objective-C.

UI: SwiftUI for most screens. UIKit for navigation shell (UITabBarController, UINavigationController) and complex custom views (camera, maps, rich text). UIHostingConfiguration for SwiftUI cells in UIKit collection views.

State management: @Observable for view models. @State for view-local state. No Combine for UI state (use Observation instead). Combine only for complex reactive chains if needed.

Concurrency: async/await + actors exclusively. No GCD for app-level code. Actors for stateful services.

Navigation: NavigationStack with NavigationPath for programmatic navigation and deep linking.

Data: SwiftData for simple persistence. Core Data for complex requirements (CloudKit sync, migration). URLSession with async/await for networking.

Architecture: MVVM with @Observable. Coordinator pattern for navigation. SPM for modularization.

Minimum target: iOS 17 (to use @Observable, modern traits, and all SwiftUI 5.0 features).

43. How has Apple's approach to backward compatibility evolved? Compare early iOS to modern iOS. L6

Early iOS (2008-2013): Aggressive deprecation. iOS 7's flat design required every app to be redesigned. APIs were removed quickly. Developers expected to update apps for each iOS release.

Middle era (2014-2018): More gradual. Swift's source compatibility mode let old code compile. #available checks enabled progressive enhancement. Deprecated APIs stayed functional for years.

Modern iOS (2019+): Very careful backward compatibility. Swift source stability guarantee. ABI stability means old binaries keep working. New APIs are additive (CompositionalLayout doesn't remove FlowLayout). The N-2 support window is well-established.

Key shift: Apple moved from "update or break" to "new APIs alongside old, gradual migration." This reflects the maturity of the platform and the size of the installed base — breaking changes now affect billions of devices.

44. What is the relationship between Swift language features and iOS SDK features? Why do some features require a minimum iOS version? L6

Swift features fall into two categories:

Language-only features (no iOS version requirement): These are implemented entirely in the compiler. Examples: if-let shorthand, RegexBuilder, typed throws, macros (the macro system itself). These work on any iOS version your project targets.

Runtime-dependent features (require minimum iOS): These need runtime support that ships with the OS. Examples:

  • async/await: Requires the cooperative thread pool runtime (back-deployed to iOS 13)
  • @Observable: Requires the Observation framework (iOS 17+)
  • Actors: Require the actor executor runtime (back-deployed to iOS 13)

Apple sometimes back-deploys runtime features to older iOS versions. async/await was back-deployed to iOS 13 even though it shipped with iOS 15's SDK. This is possible because the runtime support is bundled with the app binary for older OS versions.

45. How do you maintain a technology radar for a large iOS team? What should be on it? Architect

A technology radar tracks APIs and tools across four rings: Adopt (use in production), Trial (use in non-critical features), Assess (evaluate, prototype), Hold (don't start new usage).

Example radar for 2025:

  • Adopt: async/await, NavigationStack, CompositionalLayout, DiffableDataSource, Swift 6 strict concurrency, UIHostingConfiguration
  • Trial: @Observable (if targeting iOS 17+), SwiftData (for simple persistence), TipKit
  • Assess: visionOS APIs, Swift 6 typed throws, Swift Testing framework
  • Hold: New Combine adoption (use Observation instead), new Storyboard usage, ObservableObject for new code, UIWebView (obviously)

Process: Review quarterly after WWDC and after each iOS release. Update based on: (1) minimum deployment target changes, (2) team experience with trial technologies, (3) stability of assessed technologies. Share with the team and use in architecture decision records (ADRs).
