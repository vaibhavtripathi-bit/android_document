1. XCTest Fundamentals

Senior Q1: Explain the complete XCTestCase lifecycle. In what order do setUp, setUpWithError, tearDown, tearDownWithError, and addTeardownBlock execute?

For each test method, XCTest creates a new instance of the test class. The execution order is:

  1. class func setUp() — once per class, before any tests
  2. init() — new instance created
  3. setUp() — instance method (legacy)
  4. setUpWithError() throws — instance method (preferred)
  5. setUp() async throws — async setup (iOS 15.4+)
  6. The test method itself
  7. tearDown() async throws
  8. tearDownWithError() throws
  9. tearDown()
  10. addTeardownBlock closures in LIFO order
  11. deinit

After all tests: class func tearDown() runs once.

Key insight: A new instance per test means stored properties are re-initialized. Class-level (static) properties persist across tests — a common source of flaky tests when execution order differs between local and CI runs.

Senior Q2: What is the difference between XCTestExpectation's isInverted and expectedFulfillmentCount? When would you use each?

isInverted = true: The expectation succeeds if fulfill() is NOT called within the timeout. Use this to assert that something does NOT happen — e.g., verifying that a debounced function doesn't fire prematurely, or that a notification is not posted for a certain action.

expectedFulfillmentCount: The expectation requires exactly N calls to fulfill(). Default is 1. Use this to verify that a callback fires exactly N times — e.g., a progress handler that should be called 3 times for 3 chunks of data.

Combined with assertForOverFulfill (default true), calling fulfill() more than expectedFulfillmentCount times causes a test failure. Set to false when you only care about minimum count.

Senior Q3: Why should you prefer XCTAssertEqual over XCTAssertTrue(a == b)? What about XCTUnwrap?

XCTAssertEqual provides a detailed failure message: "XCTAssertEqual failed: ("3") is not equal to ("5")". XCTAssertTrue(a == b) only says "XCTAssertTrue failed" — you don't know what the actual values were.

XCTUnwrap is the preferred way to unwrap optionals in tests. It returns the unwrapped value or fails the test with a clear message. This replaces the pattern of guard let x = optional else { XCTFail(); return } and allows subsequent assertions to use the unwrapped value directly.

The general principle: use the most specific assertion available. It produces better diagnostics in CI logs where you can't attach a debugger.

Architect Q4: What are test plans (.xctestplan)? How would you use them to run different configurations in CI?

Test plans are JSON files that define multiple configurations for the same test suite. Each configuration can specify different settings: sanitizers (ASAN, TSAN), language/region, environment variables, code coverage, and which tests to include/exclude.

CI strategy with test plans:

  • Default config: Runs on every PR — standard settings, code coverage enabled
  • ASAN config: Runs nightly — Address Sanitizer enabled (2-3x slower)
  • TSAN config: Runs nightly — Thread Sanitizer (5-10x slower, incompatible with ASAN)
  • Localization config: Runs weekly — Arabic (RTL), Japanese, double-length pseudolanguage

ASAN and TSAN cannot run simultaneously (they instrument the binary differently), so they need separate configurations. Running them nightly catches memory and threading bugs without slowing PR checks.

Architect Q5: Explain how parallel testing works in Xcode. What's the difference between parallel by class and parallel by individual test?

Xcode creates cloned simulators for parallel testing. Each clone has its own process, file system, UserDefaults, and Keychain.

Parallel by class (default): Tests within a single class run serially on one clone. Different classes run on different clones simultaneously. This is safer because tests within a class may share state via class-level setUp.

Parallel by individual test: Each test method can run on any clone. Maximum parallelism but requires complete test isolation — no shared state even within a class.

Key pitfall: Clones share the host machine. If tests use fixed ports (e.g., a local mock server on port 8080), they'll conflict. Each test or clone needs dynamic port allocation.

At scale, parallel testing provides 2-4x speedup but requires investment in test isolation. The ROI is highest for suites with 1000+ tests.

Senior Q6: What is XCTSkip and when should you use it instead of disabling a test?

XCTSkip conditionally skips a test at runtime. The test appears as "skipped" in results (not pass, not fail). Use XCTSkipIf(condition) or XCTSkipUnless(condition).

Use cases:

  • Hardware-dependent tests (AR, camera, NFC) — skip on simulator
  • OS-version-dependent tests — skip on older iOS versions
  • Environment-dependent tests — skip when CI env var isn't set

Why not just disable? Commented-out or #if-guarded tests become invisible. Nobody remembers to re-enable them. XCTSkip keeps the test in the report, making it visible how many tests are being skipped and why. This creates accountability.

Senior Q7: What metrics does XCTMetric support? How do performance baselines work?

XCTest supports these metrics: XCTClockMetric (wall time), XCTCPUMetric (CPU time/instructions), XCTMemoryMetric (peak memory), XCTStorageMetric (disk I/O), XCTOSSignpostMetric (OS signpost intervals like scrolling hitches), and XCTApplicationLaunchMetric (app launch time).

Baselines: First run records a baseline. Subsequent runs compare against it. If performance degrades beyond the allowed deviation (default 10%), the test fails. Baselines are stored per-device configuration in the xcresult bundle.

CI consideration: Performance baselines are machine-specific. CI machines may have different hardware than developer machines. Establish baselines on CI machines specifically, and use separate baselines for different hardware tiers.

Architect Q8: What is the difference between a unit test bundle with a host app and one without? When would you choose each?

With host app: The test bundle is loaded into the app's process. AppDelegate runs, UIKit is available, you can use @testable import on the app module. Slower due to app launch overhead.

Without host app (logic tests): Tests run in the xctest runner process. No UIKit, no app lifecycle. Only public API of frameworks is accessible (unless using @testable import on a framework). Much faster.

Architecture recommendation: Extract business logic into frameworks and test them without a host app (fastest). Only use a host app for tests that genuinely need UIKit (view controllers, Core Data with app's model). This is a key optimization for large test suites — logic tests can be 3-5x faster per test.

Test AppDelegate pattern: When using a host app, replace the real AppDelegate with a minimal test AppDelegate that skips analytics, network setup, and push registration. This prevents side effects during testing.

L6 Q9: You're designing test infrastructure for a codebase with 15,000 tests. CI takes 90 minutes. How do you get it under 15 minutes?

This requires a multi-pronged approach:

1. Selective testing (biggest impact): Build a module dependency graph. On PR, only run tests for changed modules and their transitive dependents. If someone changes a leaf module, run ~5% of tests instead of 100%. This alone can reduce PR check time by 10-50x.

2. Test sharding: Use build-for-testing once, then test-without-building across 4-8 parallel machines. Linear speedup with shard count.

3. Logic tests: Move tests off the host app. Logic tests skip app launch (saving 2-5 seconds per test class).

4. Build caching: Use Bazel or Tuist Cloud to cache module builds. If a module hasn't changed, don't recompile it.

5. Tiered CI: Fast checks (lint + affected tests) on every push. Full suite pre-merge. Nightly for sanitizers and full regression.

6. Flaky test quarantine: Auto-detect tests with >5% failure rate and quarantine them. Flaky tests waste CI time on retries.

Target architecture: Tier 1 (< 5 min) → Tier 2 (< 15 min) → Merge queue (< 10 min) → Nightly (unlimited).

L6 Q10: How do you handle the "test AppDelegate" problem? Why does it matter at scale?

When unit tests use a host app, the real AppDelegate runs before tests. This causes: analytics SDK initialization, push notification registration, network calls, database migrations, and feature flag fetches — all before your first test.

Solution: Create a TestAppDelegate with minimal setup. In main.swift (replacing @main), check if NSClassFromString("XCTestCase") != nil and use the test delegate if so.

At scale, this matters because:

  • Analytics calls during tests pollute production data
  • Network calls make tests non-deterministic
  • Database migrations add seconds to test startup
  • Push registration triggers system dialogs that block UI tests
  • Feature flag fetches can change test behavior between runs

A clean test AppDelegate is the foundation of reliable test infrastructure.

2. Mocking & Test Doubles

Senior Q11: What is the difference between a Spy, Mock, Stub, Fake, and Dummy? Give a Swift example of each.

Dummy: Fills parameter lists. No logic, never actually called. Example: an empty protocol conformance passed to satisfy a type requirement.

Stub: Returns canned/predetermined data. No behavior verification. Example: StubUserRepo that always returns a hardcoded user.

Spy: Records all interactions for later verification. Example: SpyAnalytics that appends every track() call to an array, which you assert on after the test.

Mock: Has pre-programmed expectations. Fails if expected interactions don't happen. Example: MockPaymentService that expects charge() to be called exactly once with a specific amount.

Fake: A working but simplified implementation. Example: FakeDatabase that uses an in-memory dictionary instead of SQLite.

In Swift, prefer Spies over strict Mocks. Spies are less brittle — they record interactions and let you assert after the fact, rather than failing on unexpected calls.

Senior Q12: Why can't you mock classes in Swift the way you can in Objective-C? How does this affect test architecture?

In Objective-C, you could mock any class using runtime manipulation (method_exchangeImplementations, NSProxy). The ObjC runtime uses message dispatch, so any method call can be intercepted.

Swift uses static/vtable dispatch for most methods. Only methods marked @objc or dynamic use message dispatch. You can't intercept a Swift struct method or a final class method at runtime.

Architectural impact: Swift requires explicit protocol boundaries for substitution. Every dependency that needs to be mocked must be behind a protocol. This is actually a better architecture — it makes dependencies explicit and enforces the Dependency Inversion Principle. But it requires upfront design effort that wasn't needed in ObjC.

This is why protocol-oriented design is so central to testable Swift code.

Senior Q13: What does @testable import do? What are its limitations?

@testable import promotes internal access to public for the importing test target. It does NOT expose fileprivate or private symbols.

Requirements: The module must be compiled with ENABLE_TESTABILITY = YES (default for Debug, off for Release).

Limitations:

  • Disables certain compiler optimizations (no cross-module inlining)
  • Doesn't work with Release builds — tests against Release configs will fail
  • Creates coupling between tests and internal implementation details
  • Can't expose fileprivate or private symbols

Best practice: Use @testable import for infrastructure modules (where internal algorithms need testing). Use regular import for feature modules (test the public API/behavior).

Architect Q14: Compare Sourcery and Mockolo for mock generation. When would you choose each?

Sourcery: General-purpose code generation using Stencil templates. Parses via SourceKit (full AST). Highly customizable — you can generate any code pattern. Speed: 30-60 seconds for 500 protocols.

Mockolo (by Uber): Purpose-built for mock generation. Uses a custom Swift parser (faster than SourceKit). Fixed output format (less customizable). Speed: 2-5 seconds for 500 protocols.

Decision matrix:

  • Small-medium project with custom needs → Sourcery
  • Large codebase where generation speed matters → Mockolo
  • Need to generate more than just mocks → Sourcery
  • 50+ modules, CI time is critical → Mockolo

At Uber's scale (1000+ protocols), Mockolo's speed advantage is significant — it runs in seconds vs minutes for Sourcery.

Senior Q15: Explain the closure-based mock pattern. Why is it preferred over class-based mocks in Swift?

A closure-based mock stores closures for each method, allowing each test to configure behavior inline:

struct MockNetworkClient: NetworkClient {
    var fetchHandler: ((URLRequest) async throws -> Any)?
    func fetch<T: Decodable>(_ req: URLRequest) async throws -> T {
        return try await fetchHandler!(req) as! T
    }
}

Advantages:

  • Can be a struct (value semantics, no accidental sharing)
  • Each test configures only the methods it cares about
  • Unconfigured methods crash with a clear message (fail-fast)
  • No inheritance hierarchy needed
  • Works naturally with Swift's protocol system

This pattern is essentially what TCA's @Dependency system uses internally — protocol witnesses with closure-based implementations.

Architect Q16: How do you design a protocol for testability? What is the Interface Segregation Principle in the context of mocking?

A "god protocol" with 20+ methods forces every mock to implement all methods, even when testing code that uses only one. This violates the Interface Segregation Principle.

Solution: Split into focused protocols:

protocol UserFetching { func fetchUsers() async throws -> [User] }
protocol UserSaving { func saveUser(_ user: User) async throws }
typealias UserRepository = UserFetching & UserSaving

Now a test that only needs fetching can mock just UserFetching — a single method. The production type conforms to the composed UserRepository.

At scale: This reduces mock boilerplate by 60-80% and makes tests more focused. Each test mocks only what it actually exercises.

Senior Q17: Should you test internal implementation or public API? What are the trade-offs?

Testing internal implementation (with @testable import): Higher coverage, easier to pinpoint failures, but tests are coupled to implementation. Refactoring breaks tests even when behavior doesn't change.

Testing public API only: Tests are behavior-focused and refactoring-safe. They serve as API documentation. But harder to test edge cases in internal algorithms, and lower coverage of internal code.

Recommended hybrid: Use public API testing for feature modules (behavior contracts). Use @testable import for infrastructure modules where internal algorithms need thorough testing (e.g., a custom layout engine, a caching layer, a parser).

The key question: "If I refactor the internals without changing behavior, should this test break?" If no → test the public API.

L6 Q18: Design a mock generation and sharing strategy for a 500-module codebase.

Architecture: Use interface modules (protocols only) with companion mock modules (generated mocks).

Module structure:

  • UserInterface — protocols + data types only
  • UserMocks — generated mocks for UserInterface protocols
  • UserFeature — implementation, depends on UserInterface
  • UserTests — depends on UserMocks

Generation strategy: Use Mockolo with per-module generation. Only regenerate mocks for modules whose protocols changed (incremental). Cache generated mocks in CI.

Sharing: Other modules' test targets can depend on UserMocks to get test doubles for user-related protocols. This avoids duplicate mock implementations across 500 modules.

Build time impact: Interface modules are tiny (protocols only) and compile in milliseconds. Mock modules are generated and cached. Feature modules only recompile when their own code or their interface dependencies change.

Senior Q19: How do you test async/await code? When do you still need XCTestExpectation?

With Swift concurrency, test methods can be async throws. You simply await the async operation and assert on the result — no expectations needed:

func testFetch() async throws {
    let user = try await service.fetchUser(id: "1")
    XCTAssertEqual(user.name, "Alice")
}

You still need XCTestExpectation for:

  • Callback-based APIs that haven't been migrated to async
  • Combine publisher testing (waiting for specific values)
  • NotificationCenter observations
  • KVO observations
  • Delegate callbacks

Over time, as codebases migrate to async/await, the need for expectations decreases. But legacy code and system APIs still use callbacks.

Architect Q20: What is the "Fake" pattern and when is it better than a stub or mock?

A Fake is a working but simplified implementation. Example: an in-memory database that implements the full UserRepository protocol with actual CRUD operations, but uses a dictionary instead of SQLite.

Fakes are better when:

  • You need to test complex interactions (save then fetch, update then verify)
  • The behavior depends on state that accumulates across multiple calls
  • Stubs would require complex setup to simulate realistic behavior
  • Integration tests need a fast but realistic backing store

Trade-off: Fakes require more maintenance (they're real code that can have bugs). But they provide more realistic test behavior than stubs, catching integration issues that stubs miss.

At Google, fakes are heavily used for infrastructure services (storage, networking). The fake is maintained alongside the real implementation and tested against the same contract tests.

3. UI Testing

Senior Q21: Explain the two-process architecture of XCUITest. Why can't you use @testable import in UI tests?

UI tests run in a separate process from the app. The test runner process communicates with the app process via XPC (inter-process communication). The test runner queries the app's accessibility tree to find and interact with elements.

Because they're separate processes, the test runner has no access to the app's memory, objects, or internal types. @testable import only works within the same process. This is by design — UI tests simulate real user interaction from outside the app.

Implications:

  • App state must be configured via launch arguments/environment
  • Assertions are based on what's visible in the UI (accessibility tree)
  • Network stubbing requires either launch-argument-based mock layers or a local server
  • Test setup is slower (full app launch)
Senior Q22: Why are accessibility identifiers critical for UI test stability? What naming convention do you recommend?

Without accessibility identifiers, UI tests rely on element labels (localized text), position (index), or type. All of these break easily: labels change with localization, positions change with UI redesigns, and types are ambiguous.

Accessibility identifiers provide a stable contract between the app and test suite that survives UI changes.

Naming convention: Use dot-notation: screen.element. Examples: login.emailField, home.feedTable, profile.editButton.

Best practices:

  • Share identifier constants between app and test targets (via a shared SPM package)
  • Never use localized strings as identifiers
  • Add identifiers to all interactive elements and key display elements
  • For dynamic lists, use item.\(id) pattern
Senior Q23: What is the Page Object pattern? How does it improve UI test maintainability?

The Page Object pattern encapsulates each screen's UI structure behind a clean API. Each "page" is a struct with properties for elements and methods for actions. Methods return the next page object, enabling fluent chaining.

Benefits:

  • When UI changes, update one page object — not every test
  • Tests read like user stories: LoginScreen().enterEmail("...").tapLogin()
  • Element queries are centralized (single source of truth for identifiers)
  • Navigation flow is encoded in return types (type-safe screen transitions)

Key design decision: Methods that navigate to a new screen return the new screen's page object. Methods that stay on the same screen return Self. This makes the test's navigation flow explicit and type-checked.

Architect Q24: How do you stub network responses in UI tests? Compare the launch-argument approach vs local server approach.

Launch-argument approach: Pass a mock scenario name via launchEnvironment. The app checks for this and loads bundled JSON responses. Pros: simple, no external dependencies. Cons: mock data is bundled in the app, limited flexibility.

Local server approach: Run a lightweight HTTP server (e.g., Swifter, Embassy) in the test runner process. Configure the app to use localhost as the API base URL. Pros: full control over responses, can simulate delays/errors dynamically. Cons: more setup, port conflicts in parallel testing.

URLProtocol approach: Register a custom URLProtocol subclass that intercepts requests. Pros: no server needed, works at the URL loading level. Cons: only works if the app uses URLSession with a configurable session (not .shared).

Recommendation: Launch-argument approach for simple scenarios. Local server for complex integration tests. URLProtocol for unit-level network testing.

Senior Q25: What is snapshot testing? How does it compare to XCUITest screenshot testing?

Snapshot testing (e.g., pointfree/swift-snapshot-testing) captures a view's rendered output and compares it against a reference image stored in the repo. It runs at the unit test level — no app launch, no simulator interaction. Very fast.

XCUITest screenshots capture the full device screen during a UI test. They're attachments in the test result bundle, useful for visual verification and debugging.

Key differences:

  • Snapshot tests are unit-level (fast, deterministic) vs UI-level (slow, full app)
  • Snapshot tests compare against reference images (regression detection) vs screenshots are just captured (no comparison)
  • Snapshot tests can test individual views/components vs screenshots capture the full screen

Pitfall: Snapshot tests are fragile across OS versions and simulator versions (rendering differences). Pin exact Xcode/simulator versions in CI and use precision: 0.99 for tolerance.

Senior Q26: Why should you never use sleep() in UI tests? What should you use instead?

sleep() is the #1 cause of flaky and slow UI tests. If you sleep for 2 seconds but the operation takes 0.5 seconds, you waste 1.5 seconds per test. If the operation takes 3 seconds (slow CI), the test fails.

Use instead:

  • element.waitForExistence(timeout:) — polls until element appears
  • XCTNSPredicateExpectation — waits for any predicate condition
  • Custom wait helpers: waitForEnabled(), waitForDisappearance()

These methods poll efficiently and return as soon as the condition is met, making tests both faster and more reliable.

Architect Q27: In iOS 16+, SwiftUI List is backed by UICollectionView, not UITableView. How does this affect UI tests?

This is a common source of test breakage during iOS upgrades. If your UI tests use app.tables to find SwiftUI Lists, they'll stop working on iOS 16+ because the backing view changed to UICollectionView.

Solution: Use app.collectionViews for iOS 16+ or, better yet, use accessibility identifiers on the List itself: app.otherElements["myList"]. This is implementation-agnostic.

Broader lesson: Never rely on the internal UIKit backing of SwiftUI views. Apple changes these between OS versions. Always use accessibility identifiers as the stable contract.

L6 Q28: Design a UI test strategy that balances coverage, speed, and reliability for a team of 50 developers.

Tiered approach:

  • Smoke tests (5-10): Run on every PR. App launches, login works, main tab loads. Under 3 minutes.
  • Critical journey tests (20-50): Run pre-merge. Login, purchase, core CRUD, settings. Under 10 minutes.
  • Full regression (100-300): Run nightly. All features, edge cases, error states. Under 60 minutes.
  • Snapshot tests (200-500): Run on every PR. Individual component rendering. Under 2 minutes (unit-level).

Reliability measures:

  • Disable animations in UI tests (UIView.setAnimationsEnabled(false))
  • Use predicate-based waiting (never sleep())
  • Stub all network calls (no real API dependency)
  • Auto-retry failed tests once before marking as failure
  • Quarantine tests with >2% flakiness rate

Key metric: Confidence-per-minute-of-CI-time. Maximize the bugs caught per minute of CI execution.

4. Swift Testing Framework

Senior Q29: How does Swift Testing discover tests without the ObjC runtime? What role do macros play?

XCTest discovers tests via the ObjC runtime: it scans for subclasses of XCTestCase and methods prefixed with test. This requires the ObjC runtime, which is why XCTest classes must be @objc-compatible.

Swift Testing uses Swift macros. The @Test attribute is a macro that generates metadata at compile time — specifically, it creates static registrations that the test runner can discover without runtime reflection. This is pure Swift, no ObjC needed.

Implications:

  • Tests can be on structs, actors, or free functions (not just classes)
  • No test prefix required (the macro marks them)
  • Better compile-time diagnostics (macros can emit warnings)
  • Works on non-Apple platforms (Linux, Windows) without ObjC
Senior Q30: What is the difference between #expect and #require? When do you use each?

#expect is a soft assertion — it records a failure but continues test execution. Use for independent checks where you want to see all failures at once.

#require is a hard assertion — it stops test execution on failure (throws). Use for preconditions and optional unwrapping where subsequent code depends on the result.

Key advantage over XCTest: Both macros show the actual values in the expression on failure. #expect(user.age == 18) produces: "Expectation failed: (user.age → 25) == 18". XCTest's XCTAssertEqual only shows the two values, not the full expression context.

#require replaces XCTUnwrap: let user = try #require(response.user) unwraps the optional or fails the test.

Senior Q31: How do parameterized tests work in Swift Testing? What types can be used as arguments?

Parameterized tests use the arguments: parameter of @Test. The test function takes parameters matching the argument types:

@Test(arguments: ["valid@email.com", "user+tag@domain.co.uk"])
func validEmails(email: String) { /* ... */ }

Arguments can be any Sendable collection. For custom types, conform to CustomTestStringConvertible for readable test names.

With two argument collections, Swift Testing generates the Cartesian product:

@Test(arguments: Currency.allCases, Currency.allCases)
func convert(from: Currency, to: Currency) { /* 4×4 = 16 tests */ }

Each parameter combination runs as a separate test case in the test navigator, making it easy to identify which specific input failed.

Senior Q32: How do tags work in Swift Testing? How are they different from XCTest's approach?

Tags are first-class in Swift Testing. Define them as extensions on Tag, then apply with .tags():

extension Tag { @Tag static var critical: Self }
@Test(.tags(.critical)) func loginFlow() { }

Tags can be applied to individual tests or entire suites. You can filter by tags in the CLI: swift test --filter .tags(.critical).

XCTest has no equivalent. The closest workaround is naming conventions (e.g., testCritical_LoginFlow) or test plans that include/exclude specific test classes. Tags are far more flexible — a single test can have multiple tags, and you can combine tag filters.

Use cases: .critical (run on every PR), .slow (exclude from fast CI), .network (needs network access), .flaky (quarantined).

Senior Q33: Swift Testing runs tests in parallel by default. How does this differ from XCTest? What problems can it cause?

XCTest: Tests within a class run serially by default. Parallel testing is opt-in and runs different classes on different simulator clones.

Swift Testing: All tests run in parallel by default, using Swift's structured concurrency (task groups). Opt out with @Suite(.serialized).

Problems:

  • Shared mutable state causes data races (use struct suites to avoid)
  • Tests that depend on ordering will fail intermittently
  • Singleton access from multiple tests simultaneously
  • File system operations on the same paths

Solution: Design tests to be fully isolated. Use struct suites (value semantics). Inject all dependencies. Use .serialized only when tests genuinely need ordering (rare).

Architect Q34: Can Swift Testing and XCTest coexist? What are the rules for mixing them?

Yes, they coexist in the same test target. Both run when you press Cmd+U. But there are strict rules:

  • Never put @Test methods inside an XCTestCase subclass
  • Never use XCTAssert inside a @Test function
  • Never use #expect inside an XCTest method
  • Each test file should use one framework consistently

The two frameworks have different runners and lifecycle management. Mixing them in a single class/function causes undefined behavior.

Migration strategy: Write all new tests with Swift Testing. Migrate existing tests file-by-file. Keep XCTest for UI tests and performance tests (not yet supported in Swift Testing).

L6 Q35: Design a migration strategy from XCTest to Swift Testing for a 10,000-test codebase.

Phase 1 (Month 1-2): Foundation

  • Enable Swift Testing in all test targets
  • Write all new tests with Swift Testing
  • Establish team conventions (tags, traits, naming)
  • Create a migration guide and training materials

Phase 2 (Month 2-4): Low-hanging fruit

  • Migrate simple tests (no setUp/tearDown, no expectations)
  • Mechanical conversion: XCTAssertEqual(a, b) → #expect(a == b)
  • Identify tests that benefit from parameterization

Phase 3 (Month 4-8): Complex tests

  • Migrate tests with setUp/tearDown → init/deinit
  • Convert async expectation tests → async/await
  • Keep performance tests in XCTest (no Swift Testing support yet)

Phase 4 (Month 8-12): Cleanup

  • Remove XCTest imports from migrated files
  • Keep XCTest only for UI tests and performance tests

Track metrics: % tests on Swift Testing, CI time comparison, developer satisfaction.

Senior Q36: What features does XCTest have that Swift Testing does not (yet)?

As of Xcode 16, Swift Testing does not support:

  • UI testing — XCUIApplication and related APIs are XCTest-only
  • Performance testing — measure {} blocks, XCTMetric, baselines
  • KVO expectations — XCTKVOExpectation
  • Notification expectations — XCTNSNotificationExpectation
  • Objective-C test support

These features will likely be added in future Xcode releases. For now, keep these test types in XCTest and migrate everything else to Swift Testing.

5. Dependency Injection

Senior Q37: What is constructor injection and why is it the "gold standard" of DI?

Constructor injection passes all dependencies through the initializer:

class UserService {
    init(repo: UserRepository, analytics: Analytics) { ... }
}

Why it's the gold standard:

  • Explicit: All dependencies are visible in the initializer signature
  • Immutable: Dependencies are set once and can be let properties
  • Compile-time safe: Can't create an instance without providing all dependencies
  • Testable: Inject test doubles directly in the initializer
  • No framework needed: Pure Swift, no DI container required

Downside: Initializer parameter lists can get long for complex objects. Mitigate with factories or builder patterns.

Senior Q38: How does SwiftUI's @Environment work as a DI mechanism? What are its limitations?

@Environment reads values from the SwiftUI environment, which propagates down the view hierarchy. You define custom keys with EnvironmentKey and inject values with .environment().

As DI: It's the idiomatic SwiftUI approach. Dependencies are injected at the root and available to all descendant views without explicit passing.

Limitations:

  • Only works in SwiftUI views (not in ViewModels or services)
  • @EnvironmentObject crashes if not provided (no compile-time safety)
  • Can't access environment values in init() — only in body
  • Default values in EnvironmentKey may hide missing injections
  • Testing requires wrapping views in a host with environment set

iOS 17+ improvement: @Observable objects can be injected directly via @Environment(MyType.self) without defining a custom key.

Architect Q39: What is the Service Locator pattern and why is it considered an anti-pattern?

A Service Locator is a global registry where components look up their dependencies at runtime: let repo: UserRepository = ServiceLocator.shared.resolve().

Why it's an anti-pattern:

  • Hidden dependencies: Can't see what a class needs from its interface
  • Global mutable state: Any code can register/override services
  • Runtime failures: Missing registration crashes at runtime, not compile time
  • Test isolation: Must carefully manage the global locator state between tests
  • Invisible dependency graph: Can't trace or validate dependencies statically

Note: Swinject and similar DI containers can become service locators if misused (calling container.resolve() deep inside business logic). The container should only be used at the composition root (app startup).

Architect Q40: Explain protocol witnesses. How do they differ from protocols and when are they better?

Protocol witnesses replace protocols with structs containing closures:

struct Logger {
    var log: (String) -> Void
}
extension Logger {
    static let console = Logger { print($0) }
    static let noop = Logger { _ in }
}

Advantages over protocols:

  • Composition: Combine witnesses naturally (e.g., logger.combined(with: fileLogger))
  • Partial conformance: Only implement the closures you need
  • No existential overhead: Structs are stack-allocated, no box allocation
  • Retroactive conformance: Create a witness for any type without extensions

When protocols are better: When you need Xcode autocomplete/discoverability, when the interface is large and stable, when you need protocol inheritance, or when working with SwiftUI's type system (which relies heavily on protocols).

TCA's @Dependency system is built on protocol witnesses.

Architect Q41: How does TCA's @Dependency system work under the hood?

TCA's dependency system uses Swift's task-local values (similar to thread-local storage but for structured concurrency). Each dependency is registered as a DependencyKey with a liveValue and testValue.

How it works:

  1. Dependencies are stored in a DependencyValues struct (like SwiftUI's EnvironmentValues)
  2. @Dependency(\.userClient) reads from the current task-local dependency values
  3. In production, liveValue is used automatically
  4. In tests, testValue is used (defaults to unimplemented() which crashes if called without explicit override)
  5. withDependencies { $0.userClient = ... } overrides for a specific scope

Key insight: The testValue defaulting to unimplemented() is a powerful design choice. It means tests fail loudly if they access a dependency that wasn't explicitly configured, preventing accidental use of live services in tests.

Senior Q42: What is the problem with default parameter values for DI? How do factories solve it?

Default values like init(repo: UserRepository = APIUserRepository()) create hidden compile-time coupling. The module containing the initializer must import the module containing APIUserRepository. In a modular codebase, this defeats the purpose of DI — the feature module now depends on the implementation module.

Factory solution: A factory protocol/struct centralizes object creation in the composition root (app module). Feature modules depend only on the factory interface, not on concrete implementations:

protocol ViewModelFactory {
    func makeUserVM(userId: String) -> UserProfileViewModel
}

The app module provides the concrete factory that knows about all implementations. Feature modules only know about the factory protocol.

Senior Q43: What are the scoping options in Swinject? When would you use each?

Transient (default): New instance every time resolve() is called. Use for stateless services, view models, and anything that shouldn't be shared.

Container (singleton): One instance per container, shared across all resolutions. Use for expensive resources: database connections, network sessions, analytics services.

Weak: Shared while at least one strong reference exists. When all references are released, the next resolution creates a new instance. Use for caches and managers that should be cleaned up when not in use.

Graph: One instance per resolution graph. If A depends on B and C, and both B and C depend on D, D is created once and shared between B and C within that single resolution. Use for request-scoped dependencies.

L6 Q44: Design a DI architecture for a modular app with 50+ feature modules.

Four-layer architecture:

  1. Interface modules (protocols + data types only): UserAPI, OrderAPI, PaymentAPI. Tiny, fast to compile.
  2. Feature modules (views + view models): Depend only on interface modules. Never import implementation modules.
  3. Implementation modules (concrete implementations): UserImpl, OrderImpl. Depend on interface modules and infrastructure.
  4. App module (composition root): Knows all implementations. Wires the dependency graph. This is the only module that imports everything.

Build time impact: Changing an implementation module only recompiles that module and the app module (link step). Feature modules are unaffected because they depend on interfaces, not implementations.

Testing: Each interface module has a companion mock module with generated mocks. Test targets depend on mock modules, not implementation modules.

Rule: Feature modules never import other feature modules' implementations. Cross-feature communication goes through interface protocols.

Senior Q45: What is the Reader Monad pattern? Is it practical in iOS development?

The Reader Monad wraps a function from an environment to a result: Reader<Env, Value>. It threads dependencies through computations without explicit parameter passing. You compose readers with map and flatMap, then "run" them by providing the environment.

In practice: The Reader Monad is rarely used directly in iOS development. It's more common in Haskell/Scala. However, TCA's @Dependency system is conceptually a Reader Monad implemented with task-local values.

When it's useful: If you're building a functional-style architecture where you want to defer dependency resolution and compose operations before executing them. For most iOS apps, constructor injection or TCA's @Dependency is more practical.

Architect Q46: Compare constructor injection, Swinject, TCA @Dependency, and protocol witnesses. When would you choose each?

Constructor injection: Best for most cases. Compile-time safe, explicit, no framework needed. Choose when you want simplicity and clarity.

Swinject: Best for large UIKit apps with complex object graphs. Runtime resolution with scoping. Choose when you need container-managed lifecycle and have many interdependent services.

TCA @Dependency: Best for TCA apps. Deeply integrated with the reducer pattern. Choose when using TCA — it's the idiomatic approach.

Protocol witnesses: Best for functional-style code and fine-grained composition. Choose when you need to compose behaviors (e.g., combine loggers, chain middleware).

Decision factors: Team familiarity, architecture style (UIKit vs SwiftUI vs TCA), module count, compile-time safety requirements, and whether you need runtime flexibility.

6. CI/CD & Infrastructure

Senior Q47: What is the difference between xcodebuild build-for-testing and test-without-building? Why split them?

build-for-testing compiles the app and test bundles, producing an .xctestrun file that describes how to run the tests. test-without-building takes that file and runs the tests without recompiling.

Why split:

  • Caching: Build once, test many times (different simulators, shards)
  • Sharding: Distribute the xctestrun to multiple machines for parallel execution
  • Retry: Re-run failed tests without rebuilding (saves 5+ minutes)
  • Separation of concerns: Build failures vs test failures are clearly distinguished

This is a fundamental optimization for CI pipelines with 1000+ tests.

Senior Q48: How does Fastlane's match solve code signing problems for teams?

Code signing is the #1 source of iOS CI pain. Each developer and CI machine needs the same certificates and provisioning profiles. Without match, teams manually share .p12 files and profiles — error-prone and insecure.

match stores certificates and profiles in an encrypted Git repo (or Google Cloud Storage / S3). When a developer or CI machine runs match, it decrypts and installs the correct signing assets into the Keychain.

Benefits:

  • Single source of truth for signing assets
  • New team members run one command to set up signing
  • CI machines automatically get the right certificates
  • Encrypted at rest (passphrase-protected)
  • Automatic renewal when certificates expire
Architect Q49: What are merge queues? Why are they essential for large teams?

Without a merge queue, two PRs can pass CI independently but break when merged together (semantic conflicts, not git conflicts). This is the "broken main" problem.

A merge queue serializes merges: Each PR is rebased on the latest main (including previously queued PRs) and CI runs again on the combined state. Only if this passes does the PR merge.

At scale (50+ PRs/day): Without a merge queue, main breaks multiple times per day. Each break costs 30-60 minutes of investigation. With a merge queue, main stays green.

Optimization: Merge queues can batch PRs — test A+B+C together. If the batch passes, merge all three. If it fails, bisect to find the culprit. This reduces CI load while maintaining safety.

GitHub, GitLab, and Bors all provide merge queue implementations.

Architect Q50: Compare Bazel and Tuist for iOS build caching. When would you choose each?

Bazel: Google's build system. Hermetic builds, remote caching, remote execution. Action-level granularity (caches individual compile/link steps). Supports distributed builds across machines.

Tuist: Xcode project generator with module-level caching (Tuist Cloud). Generates standard Xcode projects. Much simpler to adopt than Bazel.

Choose Bazel when: 50+ developers, build time is a critical bottleneck, you have a dedicated build infrastructure team (3-5 engineers), you need hermetic/reproducible builds.

Choose Tuist when: 10-50 developers, want caching without Bazel's complexity, team is comfortable with Xcode projects, need faster adoption (weeks vs months).

Choose neither when: Under 10 developers — native Xcode incremental builds are usually sufficient.

Senior Q51: How do you extract and analyze code coverage from xcresult bundles?

Use xcrun xccov to extract coverage from .xcresult bundles:

  • xcrun xccov view --report Results.xcresult — human-readable summary
  • xcrun xccov view --report --json Results.xcresult — JSON for CI processing
  • xcrun xccov view --file path/to/File.swift Results.xcresult — per-file detail

CI integration: Parse the JSON output to extract coverage percentages. Enforce minimum coverage for new code (e.g., 80% line coverage). Compare against the base branch to compute coverage delta — new code must not decrease overall coverage.

For third-party tools (Codecov, Coveralls): Export to LCOV format using xcrun llvm-cov export with the profdata and binary paths.

L6 Q52: Design a CI/CD pipeline for a team of 100+ iOS developers. What are the tiers and SLAs?

Tier 1 — Fast Checks (< 5 min, every push):

  • SwiftLint / SwiftFormat
  • Incremental build (cached)
  • Affected module unit tests only (selective testing)

Tier 2 — Standard Checks (< 15 min, every PR):

  • Full unit test suite (sharded across 4 machines)
  • Snapshot tests
  • Code coverage delta check

Merge Queue (< 10 min):

  • Rebase on latest main
  • UI smoke tests (5-10 critical flows)
  • Integration tests

Post-Merge (async, non-blocking):

  • Full UI regression suite
  • Performance benchmarks
  • Build + upload to TestFlight

Nightly:

  • ASAN / TSAN runs
  • Full localization test suite
  • Binary size tracking

SLAs: P50 PR check < 10 min, P95 < 20 min, main green rate > 98%, flaky rate < 1%.

Architect Q53: What is selective testing? How do you implement it?

Selective testing runs only the tests affected by the code changes in a PR, rather than the entire test suite.

Implementation:

  1. Build a module dependency graph (from Package.swift, Tuist manifests, or Xcode project)
  2. Determine which files changed (git diff)
  3. Map changed files to their containing modules
  4. Compute transitive dependents (modules that depend on changed modules)
  5. Run tests only for changed modules and their dependents

Example: Module graph: A → B → C. If B changes, run tests for B and C (dependents). Skip A's tests (A is a dependency of B, not a dependent).

Impact: For a 50-module app, a change to a leaf module might run 5% of tests instead of 100%. This can reduce PR check time from 15 minutes to 2 minutes.

Safety net: Run the full suite nightly or pre-release to catch any issues that selective testing might miss.

Senior Q54: What build time optimizations can you make without changing architecture?

Quick wins (low effort):

  • Add explicit type annotations to complex expressions (reduces type-checker work)
  • Use SWIFT_COMPILATION_MODE = wholemodule for Release builds
  • Enable ONLY_ACTIVE_ARCH = YES for Debug builds
  • Reduce bridging header size (remove unused ObjC imports)

Medium effort:

  • Use -Xfrontend -debug-time-function-bodies to find slow-to-compile functions
  • Break up large files (Swift compiles files in parallel within a module)
  • Pre-compile stable dependencies as XCFrameworks
  • Use SWIFT_STRICT_CONCURRENCY warnings to find and fix issues early

Measurement: Use xcodebuild with timing flags and sort by compilation time. Focus on the top 20 slowest files — they often account for 80% of build time.
