1. Framework Overview

Swift Testing is Apple's next-generation testing framework, introduced at WWDC 2024 with Xcode 16. It's built from the ground up for Swift, leveraging macros, concurrency, and modern language features to provide a more expressive and powerful testing experience.

1.1 Key Design Principles

┌─────────────────────────────────────────────────────────────┐ │ Swift Testing Design Philosophy │ │ │ │ ┌─────────────────┐ ┌─────────────────┐ │ │ │ Swift-Native │ │ Macro-Powered │ │ │ │ │ │ │ │ │ │ No ObjC runtime │ │ @Test, #expect │ │ │ │ No class needed │ │ compile-time │ │ │ │ Structs work │ │ diagnostics │ │ │ └─────────────────┘ └─────────────────┘ │ │ │ │ ┌─────────────────┐ ┌─────────────────┐ │ │ │ Parallel First │ │ Expressive │ │ │ │ │ │ │ │ │ │ Tests run in │ │ Parameterized │ │ │ │ parallel by │ │ tests, tags, │ │ │ │ default │ │ traits │ │ │ └─────────────────┘ └─────────────────┘ │ └─────────────────────────────────────────────────────────────┘

1.2 Minimal Example

import Testing

@Test func additionWorks() {
    #expect(2 + 2 == 4)
}

@Test("Division by zero throws")
func divisionByZero() throws {
    #expect(throws: MathError.divisionByZero) {
        try divide(10, by: 0)
    }
}
Deep Dive: No Class Required

Unlike XCTest, Swift Testing doesn't require subclassing XCTestCase. Test functions can be free functions, methods on structs, methods on classes, or methods on actors. This eliminates the ObjC runtime dependency and enables testing of actor-isolated code naturally.

2. @Test Attribute

The @Test attribute marks a function as a test. It's a Swift macro that generates the necessary metadata for the test runner to discover and execute the test.

2.1 Basic Usage

import Testing

// Simplest form
@Test func basicTest() {
    #expect(true)
}

// With display name
@Test("User creation with valid email succeeds")
func userCreation() throws {
    let user = try User(email: "alice@example.com", name: "Alice")
    #expect(user.email == "alice@example.com")
}

// Async test
@Test("Fetch user profile from API")
func fetchProfile() async throws {
    let service = UserService(client: MockClient())
    let profile = try await service.fetchProfile(id: "123")
    #expect(profile.name == "Alice")
}

// With traits
@Test("Performance-sensitive operation", .timeLimit(.minutes(1)))
func heavyComputation() async {
    let result = await DataProcessor.processLargeDataset()
    #expect(result.count > 0)
}

2.2 Test Function Requirements

Requirement XCTest Swift Testing

Can be async Yes (Xcode 13.3+) Yes (native)

3. @Suite for Grouping

@Suite groups related tests together. Unlike XCTest classes, suites can be structs, classes, or actors, and they can be nested.

3.1 Suite Types

// Struct suite (most common — value semantics, no shared state)
@Suite("User Validation")
struct UserValidationTests {
    let validator = UserValidator()

    @Test("Valid email passes")
    func validEmail() {
        #expect(validator.isValidEmail("user@example.com"))
    }

    @Test("Empty email fails")
    func emptyEmail() {
        #expect(!validator.isValidEmail(""))
    }
}

// Class suite (when you need setUp/tearDown-like behavior)
@Suite("Database Operations")
class DatabaseTests {
    var db: TestDatabase

    init() throws {
        db = try TestDatabase.create()
    }

    deinit {
        db.destroy()
    }

    @Test func insertUser() throws {
        try db.insert(User(name: "Alice"))
        #expect(db.userCount == 1)
    }
}

// Actor suite (for testing actor-isolated code)
@Suite("Cache Operations")
actor CacheTests {
    let cache = InMemoryCache()

    @Test func storeAndRetrieve() async {
        await cache.store("value", forKey: "key")
        let result = await cache.retrieve(forKey: "key")
        #expect(result == "value")
    }
}

// Nested suites
@Suite("Authentication")
struct AuthTests {

    @Suite("Login")
    struct LoginTests {
        @Test func validCredentials() { /* ... */ }
        @Test func invalidPassword() { /* ... */ }
    }

    @Suite("Registration")
    struct RegistrationTests {
        @Test func newUser() { /* ... */ }
        @Test func duplicateEmail() { /* ... */ }
    }
}

3.2 Suite Initialization as setUp

Deep Dive: init() Replaces setUp()

In Swift Testing, the suite's init() method replaces XCTest's setUp(). For struct suites, a new instance is created for each test (just like XCTest creates a new instance per test). For class suites, deinit replaces tearDown(). This is more natural Swift — no special lifecycle methods to remember.

4. #expect and #require Macros

4.1 #expect — Soft Assertion

#expect records a failure but continues test execution (like XCTAssert with continueAfterFailure = true):

@Test func userProperties() {
    let user = User(name: "Alice", age: 30)

    // Boolean expression — shows full expression on failure
    #expect(user.name == "Alice")
    #expect(user.age >= 18)
    #expect(user.isActive)

    // Failure message:
    // Expectation failed: (user.name → "Bob") == "Alice"
    // Note: Shows ACTUAL values in the expression!
}

// Custom failure message
@Test func customMessage() {
    let count = items.count
    #expect(count > 0, "Expected items but got empty list")
}

// Throwing expressions
@Test func expectThrows() {
    #expect(throws: ValidationError.invalidEmail) {
        try User(email: "")
    }

    // Expect any error of a type
    #expect(throws: ValidationError.self) {
        try User(email: "invalid")
    }

    // Expect no error
    #expect(throws: Never.self) {
        try User(email: "valid@example.com")
    }
}

4.2 #require — Hard Assertion

#require stops test execution on failure (like XCTest's XCTUnwrap or throwing in setUpWithError):

@Test func processUserData() throws {
    let response = try fetchResponse()

    // Unwrap or fail — replaces XCTUnwrap
    let user = try #require(response.user)

    // If user is nil, test stops here with:
    // "Expectation failed: response.user → nil"

    #expect(user.name == "Alice")
    #expect(user.email.contains("@"))
}

// Boolean require
@Test func requireCondition() throws {
    try #require(isFeatureEnabled, "Feature must be enabled for this test")
    // Continues only if condition is true
}

4.3 #expect vs #require Decision Matrix

Scenario Use Reason
Checking computed values #expect Want to see all failures
Unwrapping optionals #require Subsequent code needs the value
Preconditions for test #require Test is meaningless without it
Multiple independent checks #expect See all failures at once
Setup validation #require Fail fast if setup is wrong

5. Parameterized Tests

Parameterized tests are a first-class feature in Swift Testing. Instead of writing multiple test functions with similar logic, you define one test that runs with different inputs.

5.1 Basic Parameterization

@Test("Email validation", arguments: [
    "user@example.com",
    "admin@company.org",
    "test.user+tag@domain.co.uk",
])
func validEmails(email: String) {
    #expect(EmailValidator.isValid(email))
}

@Test("Invalid emails rejected", arguments: [
    "",
    "not-an-email",
    "@missing-local.com",
    "missing-domain@",
    "spaces in@email.com",
])
func invalidEmails(email: String) {
    #expect(!EmailValidator.isValid(email))
}

5.2 Multiple Parameters (Cartesian Product)

enum Currency: String, CaseIterable {
    case usd, eur, gbp, jpy
}

@Test("Currency conversion", arguments: Currency.allCases, Currency.allCases)
func conversion(from: Currency, to: Currency) async throws {
    let result = try await CurrencyConverter.convert(100, from: from, to: to)
    #expect(result > 0)
    if from == to {
        #expect(result == 100)
    }
}
// Runs 4 × 4 = 16 test cases

5.3 Custom Types as Arguments

struct ParseTestCase: CustomTestStringConvertible {
    let input: String
    let expected: Int

    var testDescription: String {
        "\"\(input)\" → \(expected)"
    }
}

@Test("Integer parsing", arguments: [
    ParseTestCase(input: "42", expected: 42),
    ParseTestCase(input: "-1", expected: -1),
    ParseTestCase(input: "0", expected: 0),
    ParseTestCase(input: "999999", expected: 999999),
])
func integerParsing(testCase: ParseTestCase) throws {
    let result = try IntParser.parse(testCase.input)
    #expect(result == testCase.expected)
}

6. Tags for Filtering

Tags allow you to categorize tests across suites and run subsets of tests based on categories.

// Define tags
extension Tag {
    @Tag static var critical: Self
    @Tag static var slow: Self
    @Tag static var network: Self
    @Tag static var database: Self
    @Tag static var ui: Self
}

// Apply tags to individual tests
@Test("Login flow", .tags(.critical, .network))
func loginFlow() async throws { /* ... */ }

@Test("Data migration", .tags(.slow, .database))
func dataMigration() throws { /* ... */ }

// Apply tags to entire suites
@Suite("API Integration", .tags(.network))
struct APITests {
    @Test func fetchUsers() async throws { /* ... */ }
    @Test func createOrder() async throws { /* ... */ }
}

// Run from CLI:
// swift test --filter .tags(.critical)     ← only critical tests
// swift test --filter .tags(.slow).not     ← exclude slow tests

7. Traits for Configuration

7.1 Built-in Traits

// Time limit
@Test(.timeLimit(.minutes(2)))
func longRunningTest() async { /* ... */ }

// Disabled (with reason)
@Test(.disabled("Waiting for backend fix — JIRA-1234"))
func brokenTest() { /* ... */ }

// Enabled conditionally
@Test(.enabled(if: ProcessInfo.processInfo.environment["CI"] != nil))
func ciOnlyTest() { /* ... */ }

// Bug reference
@Test(.bug("https://github.com/org/repo/issues/42", "Flaky on iOS 17.0"))
func flakyTest() { /* ... */ }

// Serial execution (disable parallelism for this suite)
@Suite(.serialized)
struct OrderedTests {
    @Test func step1() { /* ... */ }
    @Test func step2() { /* ... */ }
}

7.2 Combining Traits

@Test(
    "Complex integration test",
    .tags(.critical, .slow),
    .timeLimit(.minutes(5)),
    .bug("https://jira.company.com/PROJ-789")
)
func complexIntegration() async throws {
    // ...
}

8. Comparison with XCTest

8.1 Feature Comparison

Feature XCTest Swift Testing
Test discovery ObjC runtime (method prefix) Swift macros (@Test)
Test container XCTestCase subclass only Struct, class, actor, or free function
Assertions XCTAssert family (30+ functions) #expect / #require (2 macros)
Failure messages Generic ("XCTAssertEqual failed") Expression-level ("(x → 3) == 5")
Parameterized tests Not supported (manual loops) First-class (arguments:)
Tags Not supported First-class (.tags())
Parallel execution Opt-in, by class Default, by test
setUp/tearDown Special methods init/deinit
Skip tests XCTSkip (throws) .disabled() / .enabled(if:) traits
Performance testing measure {} blocks Not yet supported
UI testing XCUIApplication Not supported (use XCTest)
Xcode integration Full Full (Xcode 16+)

8.2 Assertion Comparison

// XCTest
XCTAssertEqual(user.name, "Alice")
XCTAssertNil(error)
XCTAssertTrue(user.isActive)
XCTAssertGreaterThan(count, 0)
XCTAssertThrowsError(try parse(""))
let value = try XCTUnwrap(optional)

// Swift Testing equivalents
#expect(user.name == "Alice")
#expect(error == nil)
#expect(user.isActive)
#expect(count > 0)
#expect(throws: ParseError.self) { try parse("") }
let value = try #require(optional)

9. Parallel Execution Model

┌─────────────────────────────────────────────────────────────┐ │ Swift Testing Parallel Execution │ │ │ │ XCTest (default): │ │ ┌──────┐ ┌──────┐ ┌──────┐ │ │ │test1 │→│test2 │→│test3 │ SERIAL within class │ │ └──────┘ └──────┘ └──────┘ │ │ │ │ Swift Testing (default): │ │ ┌──────┐ ┌──────┐ ┌──────┐ │ │ │test1 │ │test2 │ │test3 │ PARALLEL by default │ │ └──────┘ └──────┘ └──────┘ │ │ ↕ ↕ ↕ │ │ (concurrent task group) │ │ │ │ Opt-out with .serialized trait: │ │ @Suite(.serialized) │ │ struct OrderedTests { ... } │ └─────────────────────────────────────────────────────────────┘
Pitfall: Shared Mutable State

Since Swift Testing runs tests in parallel by default, any shared mutable state will cause data races. Use @Suite(.serialized) only when tests genuinely depend on ordering. The better solution is to eliminate shared state — use struct suites with value-type dependencies.

10. Coexistence with XCTest

Swift Testing and XCTest can coexist in the same test target. This is essential for incremental migration.

// Same test target can contain both:

// XCTest (existing)
import XCTest

class LegacyTests: XCTestCase {
    func testOldStyle() {
        XCTAssertEqual(1 + 1, 2)
    }
}

// Swift Testing (new)
import Testing

@Test func newStyle() {
    #expect(1 + 1 == 2)
}

// Both run when you press Cmd+U
Pitfall: Don't Mix in One Class

Never put @Test methods inside an XCTestCase subclass, and never use XCTAssert inside a @Test function. Each test must use one framework consistently. The two frameworks have different runners and mixing them causes undefined behavior.

11. Architect: Migration Strategy

Architect Must-Know

Migrating a 10,000+ test suite from XCTest to Swift Testing is a multi-quarter effort. The key is coexistence — both frameworks must work side by side for months or years during the transition.

11.1 Migration Phases

Phase 1: Foundation (Month 1-2) ├── Enable Swift Testing in test targets ├── Write all NEW tests with Swift Testing ├── Establish team conventions (tags, traits) └── Create migration guide for team Phase 2: Low-Hanging Fruit (Month 2-4) ├── Migrate simple unit tests (no setUp/tearDown) ├── Convert XCTAssert → #expect mechanically ├── Convert XCTUnwrap → #require └── Identify tests that benefit from parameterization Phase 3: Complex Tests (Month 4-8) ├── Migrate tests with setUp/tearDown → init/deinit ├── Convert async expectation tests → async/await ├── Migrate performance tests (keep in XCTest for now) └── Update CI to report both frameworks Phase 4: Cleanup (Month 8-12) ├── Remove XCTest imports from migrated files ├── Deprecate XCTest helper utilities ├── Update documentation └── Keep XCTest only for UI tests and perf tests

11.2 What Cannot Be Migrated

Feature Migration Path
UI tests (XCUIApplication) Keep in XCTest — no Swift Testing support
Performance tests (measure {}) Keep in XCTest — no Swift Testing support yet
XCTestExpectation (callback-based) Refactor to async/await first, then migrate
KVO expectations Keep in XCTest or refactor to Combine/async
Notification expectations Refactor to async notification sequences

11.3 Automated Migration Script

// Conceptual migration tool (simplified)
struct XCTestMigrator {
    func migrate(file: SourceFile) -> SourceFile {
        var result = file

        // Replace imports
        result = result.replacing("import XCTest", with: "import Testing")

        // Replace class declaration
        result = result.replacing(
            pattern: "class (\\w+): XCTestCase",
            with: "@Suite struct $1"
        )

        // Replace test methods
        result = result.replacing(
            pattern: "func (test\\w+)\\(\\)",
            with: "@Test func $1()"
        )

        // Replace assertions
        result = result.replacing(
            pattern: "XCTAssertEqual\\((.*), (.*)\\)",
            with: "#expect($1 == $2)"
        )

        return result
    }
}
Best Practice: Migration Metrics

Track migration progress with metrics: percentage of tests on Swift Testing, number of remaining XCTest-only features, CI time comparison between frameworks. Set quarterly targets and make migration part of the team's OKRs.
