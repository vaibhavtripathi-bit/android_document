1. Test Double Taxonomy

The terms "mock," "stub," "spy," and "fake" are often used interchangeably, but they have precise meanings that matter for test design. Understanding the distinctions helps you choose the right tool and communicate clearly in code reviews.

1.1 The Five Types of Test Doubles

Type Purpose Verifies Behavior? Has Real Logic? Example
Dummy Fill parameter lists No No Empty conformance passed but never called
Stub Provide canned answers No No Returns hardcoded data
Spy Record interactions Yes (after the fact) No Tracks method calls and arguments
Mock Verify expectations Yes (pre-programmed) No Fails if expected calls don't happen
Fake Working implementation No Yes In-memory database, local file server
Test Double Spectrum Simple ◄──────────────────────────────────────────► Complex Dummy Stub Spy Mock Fake │ │ │ │ │ │ │ │ │ │ No logic Returns Records Verifies Real logic No verify canned calls expected (simplified) data for later interactions

1.2 Concrete Examples

// The protocol we're testing against
protocol AnalyticsService {
    func track(event: String, properties: [String: Any])
    func identify(userId: String)
    func flush() async
}

// DUMMY — does nothing, just satisfies the type system
struct DummyAnalytics: AnalyticsService {
    func track(event: String, properties: [String: Any]) {}
    func identify(userId: String) {}
    func flush() async {}
}

// STUB — returns predetermined data
struct StubUserRepository: UserRepository {
    var usersToReturn: [User] = []
    var errorToThrow: Error?

    func fetchUsers() async throws -> [User] {
        if let error = errorToThrow { throw error }
        return usersToReturn
    }
}

// SPY — records all interactions for later verification
class SpyAnalytics: AnalyticsService {
    private(set) var trackedEvents: [(event: String, properties: [String: Any])] = []
    private(set) var identifiedUserIds: [String] = []
    private(set) var flushCount = 0

    func track(event: String, properties: [String: Any]) {
        trackedEvents.append((event, properties))
    }

    func identify(userId: String) {
        identifiedUserIds.append(userId)
    }

    func flush() async {
        flushCount += 1
    }
}

// MOCK — pre-programmed expectations (fails if not met)
class MockAnalytics: AnalyticsService {
    var expectedEvents: [String] = []
    private var receivedEvents: [String] = []

    func track(event: String, properties: [String: Any]) {
        receivedEvents.append(event)
    }

    func identify(userId: String) {}
    func flush() async {}

    func verify(file: StaticString = #file, line: UInt = #line) {
        XCTAssertEqual(receivedEvents, expectedEvents,
            "Expected events \(expectedEvents) but got \(receivedEvents)",
            file: file, line: line)
    }
}

// FAKE — real working implementation (simplified)
class FakeUserRepository: UserRepository {
    private var storage: [String: User] = [:]

    func save(_ user: User) async throws {
        storage[user.id] = user
    }

    func fetch(id: String) async throws -> User? {
        return storage[id]
    }

    func fetchUsers() async throws -> [User] {
        return Array(storage.values)
    }

    func delete(id: String) async throws {
        storage[id] = nil
    }
}
Best Practice: Prefer Spies Over Mocks

In Swift, spies are generally preferred over strict mocks. Spies record interactions and let you assert after the fact, making tests less brittle. Strict mocks that fail on unexpected calls create fragile tests that break when implementation details change.

2. Protocol-Oriented Testing in Swift

Swift's protocol system is the foundation of testable architecture. Unlike Objective-C (where you could mock any class via runtime manipulation), Swift requires explicit protocol boundaries for substitution.

2.1 The Protocol Boundary Pattern

// Define the boundary
protocol NetworkClient {
    func fetch<T: Decodable>(_ request: URLRequest) async throws -> T
}

// Production implementation
struct URLSessionNetworkClient: NetworkClient {
    let session: URLSession

    func fetch<T: Decodable>(_ request: URLRequest) async throws -> T {
        let (data, response) = try await session.data(for: request)
        guard let httpResponse = response as? HTTPURLResponse,
              (200..<300).contains(httpResponse.statusCode) else {
            throw NetworkError.badResponse
        }
        return try JSONDecoder().decode(T.self, from: data)
    }
}

// Test stub
struct StubNetworkClient: NetworkClient {
    var result: Result<Any, Error>

    func fetch<T: Decodable>(_ request: URLRequest) async throws -> T {
        switch result {
        case .success(let value):
            return value as! T
        case .failure(let error):
            throw error
        }
    }
}

// The class under test depends on the protocol, not the implementation
class UserService {
    private let client: NetworkClient

    init(client: NetworkClient) {
        self.client = client
    }

    func getUser(id: String) async throws -> User {
        var request = URLRequest(url: URL(string: "https://api.example.com/users/\(id)")!)
        request.httpMethod = "GET"
        return try await client.fetch(request)
    }
}

2.2 Protocol Design for Testability

Deep Dive: Granular vs Coarse Protocols

A common mistake is creating "god protocols" with 20+ methods. This forces every mock to implement all methods, even when testing code that uses only one. Follow the Interface Segregation Principle:

// BAD: God protocol — every mock must implement everything
protocol DataManager {
    func fetchUsers() async throws -> [User]
    func saveUser(_ user: User) async throws
    func deleteUser(id: String) async throws
    func fetchOrders() async throws -> [Order]
    func saveOrder(_ order: Order) async throws
    func fetchProducts() async throws -> [Product]
    // ... 15 more methods
}

// GOOD: Segregated protocols — mock only what you need
protocol UserFetching {
    func fetchUsers() async throws -> [User]
}

protocol UserSaving {
    func saveUser(_ user: User) async throws
}

protocol UserDeleting {
    func deleteUser(id: String) async throws
}

// Compose when needed
typealias UserRepository = UserFetching & UserSaving & UserDeleting

3. Manual Mock Creation Patterns

3.1 The Closure-Based Mock

The most flexible pattern for manual mocks uses closures, allowing each test to configure behavior inline:

struct MockNetworkClient: NetworkClient {
    var fetchHandler: ((URLRequest) async throws -> Any)?

    func fetch<T: Decodable>(_ request: URLRequest) async throws -> T {
        guard let handler = fetchHandler else {
            fatalError("fetchHandler not configured for \(request.url?.absoluteString ?? "nil")")
        }
        return try await handler(request) as! T
    }
}

// Usage in tests
func testFetchUserSuccess() async throws {
    var client = MockNetworkClient()
    client.fetchHandler = { request in
        XCTAssertTrue(request.url!.path.contains("/users/"))
        return User(id: "1", name: "Alice")
    }

    let service = UserService(client: client)
    let user = try await service.getUser(id: "1")
    XCTAssertEqual(user.name, "Alice")
}

func testFetchUserNetworkError() async {
    var client = MockNetworkClient()
    client.fetchHandler = { _ in
        throw NetworkError.noConnection
    }

    let service = UserService(client: client)
    do {
        _ = try await service.getUser(id: "1")
        XCTFail("Expected error")
    } catch {
        XCTAssertEqual(error as? NetworkError, .noConnection)
    }
}

3.2 The Recording Spy Pattern

class SpyNetworkClient: NetworkClient {
    enum Call: Equatable {
        case fetch(url: String)
    }

    private(set) var calls: [Call] = []
    var resultQueue: [Result<Any, Error>] = []

    func fetch<T: Decodable>(_ request: URLRequest) async throws -> T {
        calls.append(.fetch(url: request.url!.absoluteString))

        guard !resultQueue.isEmpty else {
            fatalError("No results queued")
        }

        switch resultQueue.removeFirst() {
        case .success(let value): return value as! T
        case .failure(let error): throw error
        }
    }
}

// Usage: verify call sequence
func testRetryLogic() async throws {
    let spy = SpyNetworkClient()
    spy.resultQueue = [
        .failure(NetworkError.timeout),
        .failure(NetworkError.timeout),
        .success(User(id: "1", name: "Alice")),
    ]

    let service = RetryingUserService(client: spy, maxRetries: 3)
    let user = try await service.getUser(id: "1")

    XCTAssertEqual(spy.calls.count, 3)
    XCTAssertEqual(user.name, "Alice")
}

4. Generated Mocks (Sourcery, Mockolo)

4.1 Sourcery

Sourcery is a code generation tool that uses Stencil templates to generate Swift code from your source. For mocking, it reads protocols annotated with a marker and generates full mock implementations.

// sourcery: AutoMockable
protocol PaymentService {
    func charge(amount: Decimal, currency: String) async throws -> PaymentResult
    func refund(transactionId: String) async throws
    var isAvailable: Bool { get }
}

// Generated by Sourcery (AutoMockable.stencil):
class PaymentServiceMock: PaymentService {
    // charge
    var chargeCallCount = 0
    var chargeArguments: [(amount: Decimal, currency: String)] = []
    var chargeReturnValue: PaymentResult!
    var chargeThrowableError: Error?

    func charge(amount: Decimal, currency: String) async throws -> PaymentResult {
        chargeCallCount += 1
        chargeArguments.append((amount, currency))
        if let error = chargeThrowableError { throw error }
        return chargeReturnValue
    }

    // refund
    var refundCallCount = 0
    var refundArguments: [String] = []
    var refundThrowableError: Error?

    func refund(transactionId: String) async throws {
        refundCallCount += 1
        refundArguments.append(transactionId)
        if let error = refundThrowableError { throw error }
    }

    // isAvailable
    var isAvailableReturnValue: Bool = false
    var isAvailable: Bool { isAvailableReturnValue }
}

4.2 Mockolo

Mockolo (by Uber) is purpose-built for mock generation and is significantly faster than Sourcery for large codebases. It parses Swift directly rather than using SourceKit.

// Build phase script
mockolo \
    --sourcedirs Sources/ \
    --destination Tests/GeneratedMocks.swift \
    --mock-filelist mockfilelist.txt \
    --annotation "@mockable"

// Annotate protocols
/// @mockable
protocol AuthService {
    func login(email: String, password: String) async throws -> AuthToken
    func logout() async
    var isAuthenticated: Bool { get }
}

4.3 Sourcery vs Mockolo Comparison

Aspect Sourcery Mockolo
Speed (500 protocols) 30-60 seconds 2-5 seconds
Parsing SourceKit (full AST) Custom Swift parser
Templates Customizable (Stencil) Fixed output format
Generics support Good Good
Associated types Via templates Built-in
Maintained by Community Uber
Best for Small-medium projects, custom needs Large codebases, CI speed

5. @testable import — What It Exposes

@testable import is a compiler directive that elevates internal access to public for the importing test target. Understanding exactly what it does (and doesn't do) is essential.

5.1 Access Level Changes

Access Level Normal import @testable import
open Visible, subclassable Visible, subclassable
public Visible Visible
internal Not visible Visible (promoted to public)
fileprivate Not visible Not visible
private Not visible Not visible

5.2 Requirements and Limitations

// The module must be compiled with testing enabled
// Build Settings: ENABLE_TESTABILITY = YES
// This is ON by default for Debug, OFF for Release

@testable import MyAppModule

// Now you can access internal types:
let parser = InternalJSONParser()  // Would be invisible without @testable

// But NOT fileprivate or private:
// let secret = PrivateHelper()  // ❌ Still invisible
Pitfall: @testable and Release Builds

ENABLE_TESTABILITY = YES disables certain compiler optimizations (like function inlining across modules). Never enable it for Release builds. If your CI runs tests against Release builds for performance testing, @testable import will fail. Design your test architecture to handle both cases.

6. Internal vs Public Testing Strategies

6.1 The Two Schools of Thought

Strategy 1: Test Internal Implementation ┌──────────────────────────────────────────────┐ │ @testable import MyModule │ │ │ │ ✅ Test internal helpers directly │ │ ✅ Higher coverage numbers │ │ ✅ Easier to pinpoint failures │ │ ❌ Tests coupled to implementation │ │ ❌ Refactoring breaks tests │ │ ❌ Requires ENABLE_TESTABILITY │ └──────────────────────────────────────────────┘ Strategy 2: Test Public API Only ┌──────────────────────────────────────────────┐ │ import MyModule (no @testable) │ │ │ │ ✅ Tests are behavior-focused │ │ ✅ Refactoring-safe │ │ ✅ Works with Release builds │ │ ✅ Tests serve as API documentation │ │ ❌ Harder to test edge cases │ │ ❌ Lower coverage of internal code │ └──────────────────────────────────────────────┘
Best Practice: Hybrid Approach

Use public API testing for feature modules (behavior contracts). Use @testable import for infrastructure modules where internal algorithms need thorough testing (e.g., a custom layout engine, a caching layer). The key question: "If I refactor the internals without changing behavior, should this test break?" If no, test the public API.

7. DI Patterns for Testability

7.1 Constructor Injection

// The gold standard for testability
class OrderProcessor {
    private let paymentService: PaymentService
    private let inventoryService: InventoryService
    private let notificationService: NotificationService

    init(
        paymentService: PaymentService,
        inventoryService: InventoryService,
        notificationService: NotificationService
    ) {
        self.paymentService = paymentService
        self.inventoryService = inventoryService
        self.notificationService = notificationService
    }
}

// Test: inject test doubles
func testOrderProcessing() async throws {
    let processor = OrderProcessor(
        paymentService: StubPaymentService(shouldSucceed: true),
        inventoryService: StubInventoryService(available: true),
        notificationService: SpyNotificationService()
    )
    // ...
}

7.2 Property Injection

// Useful when constructor injection is impractical (e.g., UIViewControllers from storyboards)
class ProfileViewController: UIViewController {
    var userService: UserService = DefaultUserService()
    var imageLoader: ImageLoading = KingfisherImageLoader()

    override func viewDidLoad() {
        super.viewDidLoad()
        Task {
            let user = try await userService.currentUser()
            // ...
        }
    }
}

// Test: override properties before viewDidLoad
func testProfileLoading() {
    let vc = ProfileViewController()
    vc.userService = StubUserService(user: .mock)
    vc.imageLoader = StubImageLoader()
    vc.loadViewIfNeeded()
    // ...
}

7.3 Environment-Based Injection

// SwiftUI's @Environment provides built-in DI
struct UserListView: View {
    @Environment(\.userRepository) private var repository

    var body: some View {
        List(users) { user in
            Text(user.name)
        }
        .task {
            users = try? await repository.fetchUsers()
        }
    }
}

// Test: inject via environment
func testUserList() {
    let view = UserListView()
        .environment(\.userRepository, StubUserRepository(users: [.mock]))
}

8. Advanced Mocking Patterns

8.1 Type-Erased Mock Container

class MockContainer {
    private var handlers: [String: Any] = [:]

    func register<T>(_ type: T.Type, handler: @escaping () -> T) {
        handlers[String(describing: type)] = handler
    }

    func resolve<T>(_ type: T.Type) -> T {
        let key = String(describing: type)
        guard let handler = handlers[key] as? () -> T else {
            fatalError("No handler registered for \(key)")
        }
        return handler()
    }
}

// Test helper
extension XCTestCase {
    func makeMockContainer() -> MockContainer {
        let container = MockContainer()
        container.register(NetworkClient.self) { StubNetworkClient() }
        container.register(AnalyticsService.self) { DummyAnalytics() }
        return container
    }
}

8.2 Async Spy with Continuation

class AsyncSpy<Input, Output> {
    private(set) var calls: [Input] = []
    private var continuations: [CheckedContinuation<Output, Error>] = []

    func call(_ input: Input) async throws -> Output {
        calls.append(input)
        return try await withCheckedThrowingContinuation { continuation in
            continuations.append(continuation)
        }
    }

    func complete(at index: Int, with result: Result<Output, Error>) {
        switch result {
        case .success(let value):
            continuations[index].resume(returning: value)
        case .failure(let error):
            continuations[index].resume(throwing: error)
        }
    }
}

9. Architect: Mock Strategy for 500-Module Codebase

Architect Must-Know

In a 500-module codebase with 2,000+ protocols, manual mock maintenance becomes a full-time job. The mock generation strategy you choose affects every developer's daily workflow.

9.1 The Decision Matrix

Module Count vs Mock Strategy Modules │ Recommended Strategy ─────────┼────────────────────────────────────────── 1-10 │ Manual mocks (closure-based) 10-50 │ Sourcery with custom templates 50-200 │ Mockolo (speed matters at this scale) 200-500 │ Mockolo + per-module generation 500+ │ Mockolo + incremental generation + caching

9.2 Per-Module Mock Generation

9.3 Mock Sharing Strategy

Approach Pros Cons
Mocks in each test target Isolated, no cross-module deps Duplicate mocks for shared protocols
Shared TestSupport module Single source of truth Becomes a bottleneck, slow compilation
Per-protocol mock modules Granular, cacheable Many small modules to manage
Interface + Mock companion modules Clean separation Module count doubles
Recommended Architecture at Scale

Use interface modules that contain only protocols and data types. Each interface module has a companion mock module with generated mocks. Feature modules depend on interface modules; test targets depend on mock modules. This gives you clean dependency graphs and cacheable mock generation.
