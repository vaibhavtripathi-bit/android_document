2. TextKit 1 Architecture

TextKit 1 (introduced iOS 7) is built on the MVC pattern with three core objects that collaborate to store, lay out, and display text.

TextKit 1 Object Relationships: ═══════════════════════════════ ┌──────────────────┐ ┌──────────────────┐ ┌──────────────────┐ │ NSTextStorage │────▶│ NSLayoutManager │────▶│ NSTextContainer │ │ │ │ │ │ │ │ • Stores text │ │ • Lays out │ │ • Defines │ │ • Attributed │ │ glyphs │ │ geometry │ │ string │ │ • Glyph ↔ char │ │ • Exclusion │ │ • Notifies on │ │ mapping │ │ paths │ │ changes │ │ • Line fragment │ │ • Size │ │ │ │ rects │ │ │ └──────────────────┘ └──────────────────┘ └──────────────────┘ │ │ │ │ 1:N relationship │ 1:N relationship │ │ (one storage, many │ (one layout mgr, │ │ layout managers) │ many containers) │ ▼ ▼ ▼ Model layer Controller layer View geometry

NSTextStorage

class HighlightingTextStorage: NSTextStorage {
    private let backingStore = NSMutableAttributedString()

    override var string: String {
        backingStore.string
    }

    override func attributes(
        at location: Int,
        effectiveRange range: NSRangePointer?
    ) -> [NSAttributedString.Key: Any] {
        backingStore.attributes(at: location, effectiveRange: range)
    }

    override func replaceCharacters(in range: NSRange, with str: String) {
        beginEditing()
        backingStore.replaceCharacters(in: range, with: str)
        edited(.editedCharacters, range: range,
               changeInLength: str.count - range.length)
        endEditing()
    }

    override func setAttributes(
        _ attrs: [NSAttributedString.Key: Any]?,
        range: NSRange
    ) {
        beginEditing()
        backingStore.setAttributes(attrs, range: range)
        edited(.editedAttributes, range: range, changeInLength: 0)
        endEditing()
    }

    override func processEditing() {
        // Apply syntax highlighting after each edit
        applySyntaxHighlighting()
        super.processEditing()
    }
}

Deep Dive: Glyph vs Character

A critical distinction in text layout: characters are Unicode code points (what the user typed), while glyphs are visual representations (what gets drawn). The mapping is not 1:1:

  • Ligatures: "fi" (2 characters) → 1 glyph (fi ligature)
  • Composed emoji: "👨‍👩‍👧‍👦" (7 code points) → 1 glyph
  • Decomposed characters: "é" can be 1 or 2 code points → 1 glyph
  • Arabic shaping: same character → different glyphs based on position in word

3. TextKit 2 Architecture

TextKit 2 (iOS 15+) redesigns the layout system to be non-contiguous and viewport-based, enabling better performance for large documents.

TextKit 2 Object Relationships: ═══════════════════════════════ ┌────────────────────────┐ ┌────────────────────────┐ │ NSTextContentStorage │────▶│ NSTextLayoutManager │ │ │ │ │ │ • Backs text content │ │ • Viewport-based │ │ • NSTextElement │ │ layout │ │ enumeration │ │ • Non-contiguous │ │ • Paragraph-level │ │ • NSTextLayoutFragment│ │ granularity │ │ based │ └────────────────────────┘ └───────────┬────────────┘ │ ┌───────────┼────────────┐ │ NSTextContainer │ │ (same as TextKit 1) │ └────────────────────────┘ Key difference: TextKit 2 only lays out text that's VISIBLE in the viewport, not the entire document.

TextKit 2 Setup

import UIKit

class TextKit2ViewController: UIViewController {
    var textView: UITextView!

    override func viewDidLoad() {
        super.viewDidLoad()

        // TextKit 2 setup (iOS 16+)
        let textContainer = NSTextContainer(size: CGSize(
            width: view.bounds.width - 32,
            height: .greatestFiniteMagnitude
        ))

        let textLayoutManager = NSTextLayoutManager()
        textLayoutManager.textContainer = textContainer

        let contentStorage = NSTextContentStorage()
        contentStorage.addTextLayoutManager(textLayoutManager)

        textView = UITextView(frame: view.bounds,
                               textContainer: textContainer)
        view.addSubview(textView)
    }
}

// Enumerate visible text layout fragments
func enumerateVisibleFragments(in textLayoutManager: NSTextLayoutManager) {
    guard let viewportRange = textLayoutManager.textViewportLayoutController
        .viewportRange else { return }

    textLayoutManager.enumerateTextLayoutFragments(
        from: viewportRange.location,
        options: [.ensuresLayout]
    ) { fragment in
        let frame = fragment.layoutFragmentFrame
        print("Fragment at: \(frame)")
        return true
    }
}

TextKit 1 vs TextKit 2

UITextView default iOS 7–15 iOS 16+ (opt-out available)
TextKit 2 Migration Pitfalls
  • No glyph-level access: TextKit 2 abstracts away glyphs. Code that uses NSLayoutManager.glyphRange(for:) or custom glyph generation won't port directly.
  • Behavioral differences: Line breaking, hyphenation, and text attachment layout may differ subtly between TextKit 1 and 2.
  • Fallback to TextKit 1: UITextView in iOS 16+ uses TextKit 2 by default but falls back to TextKit 1 if you access layoutManager property directly. This triggers a one-way conversion.

4. Core Text Deep Dive

Core Text is the lowest-level text layout API available to app developers. It's a C API (with Swift bridging) that provides direct control over typesetting.

Core Text Object Hierarchy: ═══════════════════════════ CTFramesetter │ ├──▶ CTTypesetter (line-breaking engine) │ └──▶ CTFrame (laid-out text in a path) │ └──▶ [CTLine] (one per visual line) │ └──▶ [CTRun] (contiguous run of same attributes) │ └──▶ Glyphs + Positions 
import CoreText

func renderWithCoreText(in context: CGContext, rect: CGRect) {
    let attributedString = NSAttributedString(
        string: "Hello, Core Text!",
        attributes: [
            .font: UIFont.systemFont(ofSize: 24, weight: .bold),
            .foregroundColor: UIColor.label
        ]
    )

    // Create framesetter
    let framesetter = CTFramesetterCreateWithAttributedString(
        attributedString as CFAttributedString
    )

    // Create path for text layout
    let path = CGPath(rect: rect, transform: nil)

    // Create frame
    let frame = CTFramesetterCreateFrame(
        framesetter,
        CFRange(location: 0, length: 0),
        path,
        nil
    )

    // Core Text uses a flipped coordinate system
    context.translateBy(x: 0, y: rect.height)
    context.scaleBy(x: 1.0, y: -1.0)

    // Draw the frame
    CTFrameDraw(frame, context)
}

// Measure text size with CTFramesetter
func measureText(_ attributedString: NSAttributedString,
                  maxWidth: CGFloat) -> CGSize {
    let framesetter = CTFramesetterCreateWithAttributedString(
        attributedString as CFAttributedString
    )
    let constraints = CGSize(width: maxWidth, height: .greatestFiniteMagnitude)
    let size = CTFramesetterSuggestFrameSizeWithConstraints(
        framesetter,
        CFRange(location: 0, length: 0),
        nil,
        constraints,
        nil
    )
    return size
}

Line-Level Control with CTLine

// Create a single line for inline text rendering
func drawSingleLine(_ text: NSAttributedString,
                     at point: CGPoint,
                     in context: CGContext) {
    let line = CTLineCreateWithAttributedString(text as CFAttributedString)

    // Get line metrics
    var ascent: CGFloat = 0
    var descent: CGFloat = 0
    var leading: CGFloat = 0
    let width = CGFloat(CTLineGetTypographicBounds(
        line, &ascent, &descent, &leading
    ))

    // Draw at baseline
    context.textPosition = point
    CTLineDraw(line, context)

    // Enumerate runs for custom rendering
    let runs = CTLineGetGlyphRuns(line) as! [CTRun]
    for run in runs {
        let glyphCount = CTRunGetGlyphCount(run)
        let attributes = CTRunGetAttributes(run) as! [String: Any]
        print("Run: \(glyphCount) glyphs, attrs: \(attributes.keys)")
    }
}

Multi-Column Layout

// Core Text: multi-column layout
func createMultiColumnLayout(
    text: NSAttributedString,
    columns: Int,
    in rect: CGRect
) -> [CTFrame] {
    let framesetter = CTFramesetterCreateWithAttributedString(
        text as CFAttributedString
    )

    let columnWidth = rect.width / CGFloat(columns)
    let columnSpacing: CGFloat = 16
    var frames: [CTFrame] = []
    var textPosition = 0

    for col in 0..<columns {
        let columnRect = CGRect(
            x: rect.origin.x + CGFloat(col) * columnWidth,
            y: rect.origin.y,
            width: columnWidth - columnSpacing,
            height: rect.height
        )
        let path = CGPath(rect: columnRect, transform: nil)
        let frame = CTFramesetterCreateFrame(
            framesetter,
            CFRange(location: textPosition, length: 0),
            path,
            nil
        )
        frames.append(frame)

        let visibleRange = CTFrameGetVisibleStringRange(frame)
        textPosition += visibleRange.length
    }

    return frames
}

6. Emoji Rendering Complexity

Emoji rendering is one of the most complex parts of the text stack. A single visible emoji can involve multiple Unicode code points, variation selectors, zero-width joiners, and skin tone modifiers.

EmojiCode PointsCountComplexity
😀 U+1F600 1 Simple
👍🏽 U+1F44D U+1F3FD 2 Skin tone modifier
🏳️‍🌈 U+1F3F3 U+FE0F U+200D U+1F308 4 ZWJ sequence
👨‍👩‍👧‍👦 U+1F468 U+200D U+1F469 U+200D U+1F467 U+200D U+1F466 7 Family ZWJ sequence
🏴󠁧󠁢󠁳󠁣󠁴󠁿 U+1F3F4 + 7 tag characters 8 Flag tag sequence
Emoji Performance Pitfalls
  • String length ≠ visual length: "👨‍👩‍👧‍👦".count is 1 in Swift (grapheme cluster), but the UTF-16 length is 11. Core Text works with UTF-16 offsets.
  • Apple Color Emoji font: Emoji are rendered from a bitmap font (Apple Color Emoji) at specific sizes. Scaling emoji text is more expensive than scaling regular text.
  • Emoji-heavy text: Chat messages with many emoji are significantly more expensive to lay out than plain text because of the complex glyph mapping and bitmap rendering.
// Safe character enumeration respecting grapheme clusters
let text = "Hello 👨‍👩‍👧‍👦 World"

// ✅ Correct: enumerate grapheme clusters
for char in text {
    print(char)  // "👨‍👩‍👧‍👦" is one Character
}

// ❌ Dangerous: UTF-16 indexing can split emoji
let nsString = text as NSString
// nsString.substring(with: NSRange(location: 6, length: 2))
// May split the family emoji, producing invalid Unicode

// ✅ Safe NSRange conversion
if let range = text.range(of: "👨‍👩‍👧‍👦") {
    let nsRange = NSRange(range, in: text)
    print("NSRange: \(nsRange)")  // location: 6, length: 11
}

8. Text Performance for Chat Apps

Chat apps face unique text rendering challenges: thousands of messages, mixed content (text + emoji + links), variable-height cells, and the need for instant scrolling.

Architect Must-Know: Chat Text Performance Strategy
  1. Cache text measurements: Computing NSAttributedString size is expensive. Cache the result keyed by (text, width, font configuration).
  2. Pre-compute on background thread: Use CTFramesetterSuggestFrameSizeWithConstraints on a background queue for height calculation.
  3. Avoid NSAttributedString for simple text: If a message is plain text with no formatting, use UILabel with a plain string. NSAttributedString creation is expensive.
  4. Batch layout: When loading a page of messages, compute all heights in a single background pass before updating the collection view.
  5. Consider Core Text for custom rendering: For maximum control and performance, render text directly with Core Text in a CALayer subclass.
class TextSizeCache {
    private var cache = NSCache<NSString, NSValue>()

    func size(for text: NSAttributedString,
              maxWidth: CGFloat) -> CGSize {
        let key = "\(text.string.hashValue)_\(maxWidth)" as NSString

        if let cached = cache.object(forKey: key) {
            return cached.cgSizeValue
        }

        let framesetter = CTFramesetterCreateWithAttributedString(
            text as CFAttributedString
        )
        let size = CTFramesetterSuggestFrameSizeWithConstraints(
            framesetter,
            CFRange(location: 0, length: 0),
            nil,
            CGSize(width: maxWidth, height: .greatestFiniteMagnitude),
            nil
        )

        cache.setObject(NSValue(cgSize: size), forKey: key)
        return size
    }
}

// Background text height computation for chat messages
func precomputeHeights(
    for messages: [Message],
    width: CGFloat,
    completion: @escaping ([CGFloat]) -> Void
) {
    DispatchQueue.global(qos: .userInitiated).async {
        let heights = messages.map { message -> CGFloat in
            let attributed = message.attributedText()
            let framesetter = CTFramesetterCreateWithAttributedString(
                attributed as CFAttributedString
            )
            let size = CTFramesetterSuggestFrameSizeWithConstraints(
                framesetter,
                CFRange(location: 0, length: 0),
                nil,
                CGSize(width: width, height: .greatestFiniteMagnitude),
                nil
            )
            return ceil(size.height) + 24  // padding
        }
        DispatchQueue.main.async {
            completion(heights)
        }
    }
}

Best Practice Summary
  • Use UILabel/UITextView for standard text — they handle TextKit selection automatically
  • Use TextKit 2 for new projects with large document editing
  • Use Core Text when you need maximum performance or custom layout (chat, PDF)
  • Always cache text measurements — CTFramesetterSuggestFrameSizeWithConstraints is expensive
  • Handle emoji correctly: use Swift Character (grapheme clusters), not UTF-16 offsets
  • Support Dynamic Type with UIFontMetrics and adjustsFontForContentSizeCategory
  • Pre-compute text heights on background threads for scroll performance
