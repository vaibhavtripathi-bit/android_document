1. Combine-Based Reactive Patterns

Combine (iOS 13+) is Apple's first-party reactive framework. It provides a declarative API for processing values over time through publishers, operators, and subscribers. Unlike RxSwift, Combine has built-in backpressure via the Subscribers.Demand protocol.

1.1 Reactive ViewModel with Combine

final class SearchViewModel: ObservableObject {
    // Input
    @Published var searchText = ""
    @Published var selectedCategory: Category?

    // Output
    @Published private(set) var results: [Product] = []
    @Published private(set) var isSearching = false
    @Published private(set) var error: SearchError?

    private let searchService: SearchServiceProtocol
    private var cancellables = Set<AnyCancellable>()

    init(searchService: SearchServiceProtocol) {
        self.searchService = searchService
        setupBindings()
    }

    private func setupBindings() {
        let debouncedSearch = $searchText
            .debounce(for: .milliseconds(300), scheduler: DispatchQueue.main)
            .removeDuplicates()

        let categoryChange = $selectedCategory
            .removeDuplicates()

        Publishers.CombineLatest(debouncedSearch, categoryChange)
            .filter { text, _ in text.count >= 2 }
            .handleEvents(receiveOutput: { [weak self] _ in
                self?.isSearching = true
                self?.error = nil
            })
            .flatMap { [searchService] text, category in
                searchService.search(query: text, category: category)
                    .catch { error -> Just<[Product]> in
                        return Just([])
                    }
            }
            .receive(on: DispatchQueue.main)
            .sink { [weak self] results in
                self?.results = results
                self?.isSearching = false
            }
            .store(in: &cancellables)
    }
}

1.2 Reactive Data Pipeline

┌─────────────────────────────────────────────────────────────┐ │ Combine Reactive Pipeline │ │ │ │ $searchText ─── debounce(300ms) ─── removeDuplicates ──┐ │ │ │ │ │ $category ─── removeDuplicates ─────────────────────────┤ │ │ │ │ │ combineLatest │ │ │ │ │ filter(≥2 chars)│ │ │ │ │ flatMap(search) │ │ │ │ │ catch(→ empty) │ │ │ │ │ receive(main) │ │ │ │ │ sink → results │ │ │ │ Key: Each operator transforms the stream │ │ Backpressure flows upstream via Demand │ └─────────────────────────────────────────────────────────────┘

1.3 Combine Subjects as Event Bus

final class EventBus {
    static let shared = EventBus()

    private let _events = PassthroughSubject<AppEvent, Never>()
    var events: AnyPublisher<AppEvent, Never> { _events.eraseToAnyPublisher() }

    func send(_ event: AppEvent) {
        _events.send(event)
    }

    func on<T: AppEvent>(_ type: T.Type) -> AnyPublisher<T, Never> {
        _events
            .compactMap { $0 as? T }
            .eraseToAnyPublisher()
    }
}

protocol AppEvent {}
struct UserLoggedIn: AppEvent { let userId: String }
struct CartUpdated: AppEvent { let itemCount: Int }
struct ThemeChanged: AppEvent { let theme: Theme }

// Usage
EventBus.shared.on(CartUpdated.self)
    .sink { event in
        badgeView.count = event.itemCount
    }
    .store(in: &cancellables)

2. Observation Framework (@Observable)

The Observation framework (iOS 17+) represents Apple's third-generation reactive system. Unlike Combine's publisher-subscriber model, Observation uses property-level tracking via compiler-generated accessors.

2.1 How @Observable Works Internally

┌─────────────────────────────────────────────────────────────┐ │ @Observable Macro Expansion │ │ │ │ What you write: │ │ ┌───────────────────────────────────────────┐ │ │ │ @Observable class UserModel { │ │ │ │ var name: String = "" │ │ │ │ var age: Int = 0 │ │ │ │ } │ │ │ └───────────────────────────────────────────┘ │ │ │ │ What the compiler generates: │ │ ┌───────────────────────────────────────────┐ │ │ │ class UserModel: Observable { │ │ │ │ @ObservationTracked var name: String │ │ │ │ @ObservationTracked var age: Int │ │ │ │ │ │ │ │ // Generated: │ │ │ │ private let _$observationRegistrar │ │ │ │ = ObservationRegistrar() │ │ │ │ │ │ │ │ var name: String { │ │ │ │ get { │ │ │ │ access(keyPath: \.name) │ │ │ │ return _name │ │ │ │ } │ │ │ │ set { │ │ │ │ withMutation(keyPath: \.name) {│ │ │ │ _name = newValue │ │ │ │ } │ │ │ │ } │ │ │ │ } │ │ │ │ } │ │ │ └───────────────────────────────────────────┘ │ │ │ │ access() → registers which property was read │ │ withMutation() → notifies observers of that property │ └─────────────────────────────────────────────────────────────┘

2.2 Per-Property Tracking vs objectWillChange

// ObservableObject: ANY property change triggers ALL subscribers
class OldModel: ObservableObject {
    @Published var name = ""    // Change this...
    @Published var age = 0      // ...and views reading only `age` ALSO re-render
}

// @Observable: Only views reading the CHANGED property re-render
@Observable class NewModel {
    var name = ""    // Change this...
    var age = 0      // ...views reading only `age` do NOT re-render
}

// SwiftUI integration
struct NameView: View {
    let model: NewModel
    var body: some View {
        // Only re-evaluates when model.name changes
        Text(model.name)
    }
}

struct AgeView: View {
    let model: NewModel
    var body: some View {
        // Only re-evaluates when model.age changes
        Text("\(model.age)")
    }
}

2.3 withObservationTracking

// Manual observation outside SwiftUI
func observeModel(_ model: NewModel) {
    withObservationTracking {
        // Access properties you care about
        print("Name: \(model.name)")
    } onChange: {
        // Called ONCE when any accessed property changes
        // Must re-register to observe again
        DispatchQueue.main.async {
            observeModel(model)
        }
    }
}
Observation Limitation

withObservationTracking fires its onChange closure exactly once, then stops. You must re-register to continue observing. This is by design — it prevents accidental infinite observation loops — but it means UIKit integration requires a re-registration loop.

3. Unidirectional Data Flow

Unidirectional data flow (UDF) architectures ensure state changes follow a single direction: View → Action → Reducer → State → View. This eliminates the "where did this state change come from?" debugging nightmare.

3.1 Redux-Like Pattern for iOS

┌─────────────────────────────────────────────────────────────┐ │ Unidirectional Data Flow │ │ │ │ ┌──────────┐ │ │ │ View │ │ │ │ │ │ │ ┌────│ Renders │◀────────────────────────┐ │ │ │ │ state │ │ │ │ │ └──────────┘ │ │ │ │ │ │ │ │ dispatch(action) new state │ │ │ │ │ │ ▼ │ │ │ ┌──────────┐ (state, action) ┌──────────┐ │ │ │ Store │──────────────────────▶│ Reducer │ │ │ │ │ │ │ │ │ │ Holds │◀──────────────────────│ Pure fn │ │ │ │ state │ new state + │ (no side │ │ │ └──────────┘ effects │ effects)│ │ │ │ └──────────┘ │ │ │ │ │ │ side effects │ │ ▼ │ │ ┌──────────┐ │ │ │ Effects │ → API calls, DB writes, analytics │ │ │ │ → Produce new actions when complete │ │ └──────────┘ │ └─────────────────────────────────────────────────────────────┘

3.2 Minimal Redux Store in Swift

@MainActor
final class Store<State, Action>: ObservableObject {
    @Published private(set) var state: State

    private let reducer: (inout State, Action) -> Effect<Action>
    private var effectTasks: [UUID: Task<Void, Never>] = [:]

    init(initialState: State,
         reducer: @escaping (inout State, Action) -> Effect<Action>) {
        self.state = initialState
        self.reducer = reducer
    }

    func send(_ action: Action) {
        let effect = reducer(&state, action)
        switch effect {
        case .none:
            break
        case .run(let operation):
            let id = UUID()
            effectTasks[id] = Task { [weak self] in
                let action = await operation()
                self?.send(action)
                self?.effectTasks[id] = nil
            }
        }
    }
}

enum Effect<Action> {
    case none
    case run(() async -> Action)
}

// Feature using the store
struct CounterState {
    var count = 0
    var isLoading = false
}

enum CounterAction {
    case increment
    case decrement
    case fetchCount
    case countFetched(Int)
}

func counterReducer(state: inout CounterState, action: CounterAction) -> Effect<CounterAction> {
    switch action {
    case .increment:
        state.count += 1
        return .none
    case .decrement:
        state.count -= 1
        return .none
    case .fetchCount:
        state.isLoading = true
        return .run {
            let count = try await API.fetchCount()
            return .countFetched(count)
        }
    case .countFetched(let count):
        state.count = count
        state.isLoading = false
        return .none
    }
}

4. State Machines

State machines formalize the valid states and transitions of a system. They prevent invalid state combinations (e.g., "loading" AND "error" simultaneously) and make the state space explicit.

4.1 Enum-Based State Machine

enum LoadingState<T> {
    case idle
    case loading
    case loaded(T)
    case error(Error)
    case refreshing(T)
}

enum LoadingEvent {
    case startLoading
    case dataReceived
    case errorOccurred(Error)
    case refresh
    case retry
}

struct StateMachine<T> {
    private(set) var state: LoadingState<T> = .idle

    mutating func transition(event: LoadingEvent, data: T? = nil) -> Bool {
        switch (state, event) {
        case (.idle, .startLoading):
            state = .loading
            return true

        case (.loading, .dataReceived):
            guard let data else { return false }
            state = .loaded(data)
            return true

        case (.loading, .errorOccurred(let error)):
            state = .error(error)
            return true

        case (.loaded(let existing), .refresh):
            state = .refreshing(existing)
            return true

        case (.refreshing, .dataReceived):
            guard let data else { return false }
            state = .loaded(data)
            return true

        case (.error, .retry):
            state = .loading
            return true

        default:
            return false
        }
    }
}

┌─────────────────────────────────────────────────────────────┐ │ Loading State Machine │ │ │ │ ┌──────┐ │ │ │ Idle │ │ │ └──┬───┘ │ │ │ startLoading │ │ ▼ │ │ ┌─────────┐ │ │ ┌─────│ Loading │─────┐ │ │ │ └─────────┘ │ │ │ │ dataReceived │ errorOccurred │ │ ▼ ▼ │ │ ┌─────────┐ ┌─────────┐ │ │ │ Loaded │ │ Error │ │ │ └────┬────┘ └────┬────┘ │ │ │ refresh │ retry │ │ ▼ │ │ │ ┌──────────┐ │ │ │ │Refreshing│ │ │ │ └────┬─────┘ │ │ │ │ dataReceived │ │ │ └──────▶ Loaded ◀───┘ │ │ │ │ Invalid transitions are rejected (return false) │ │ e.g., Idle → dataReceived, Error → refresh │ └─────────────────────────────────────────────────────────────┘

4.3 GKStateMachine (GameplayKit)

import GameplayKit

class IdleState: GKState {
    override func isValidNextState(_ stateClass: AnyClass) -> Bool {
        stateClass == LoadingState.self
    }
}

class LoadingState: GKState {
    override func isValidNextState(_ stateClass: AnyClass) -> Bool {
        stateClass == LoadedState.self || stateClass == ErrorState.self
    }

    override func didEnter(from previousState: GKState?) {
        // Start loading data
    }
}

let machine = GKStateMachine(states: [
    IdleState(), LoadingState(), LoadedState(), ErrorState()
])
machine.enter(IdleState.self)
machine.enter(LoadingState.self) // Valid → enters
machine.enter(IdleState.self)    // Invalid → returns false

5. Event-Driven Architecture

Event-driven architecture decouples producers from consumers. Components emit events without knowing who handles them. This is essential for cross-cutting concerns like analytics, logging, and feature flags.

5.1 Typed Event System

protocol DomainEvent: Sendable {
    var timestamp: Date { get }
    var source: String { get }
}

struct ProductViewed: DomainEvent {
    let timestamp = Date()
    let source: String
    let productId: String
    let referrer: String?
}

struct OrderPlaced: DomainEvent {
    let timestamp = Date()
    let source: String
    let orderId: String
    let total: Decimal
    let items: [OrderItem]
}

actor EventDispatcher {
    private var handlers: [ObjectIdentifier: [any EventHandler]] = [:]

    func register<E: DomainEvent>(_ handler: any EventHandler, for eventType: E.Type) {
        let key = ObjectIdentifier(eventType)
        handlers[key, default: []].append(handler)
    }

    func dispatch<E: DomainEvent>(_ event: E) async {
        let key = ObjectIdentifier(E.self)
        guard let eventHandlers = handlers[key] else { return }
        for handler in eventHandlers {
            await handler.handle(event)
        }
    }
}

protocol EventHandler: Sendable {
    func handle(_ event: any DomainEvent) async
}

6. MVVM+C with Reactive Bindings

MVVM+C (Model-View-ViewModel + Coordinator) combines MVVM's testable view models with the Coordinator pattern's navigation separation. This is the most common architecture in production iOS apps.

┌─────────────────────────────────────────────────────────────┐ │ MVVM+C Architecture │ │ │ │ ┌──────────────┐ binds ┌──────────────┐ │ │ │ View │────────────▶│ ViewModel │ │ │ │ (UIView/VC) │◀────────────│ │ │ │ └──────┬───────┘ publishes │ • State │ │ │ │ │ • Actions │ │ │ │ delegate │ • Transform │ │ │ │ (navigation) └──────┬───────┘ │ │ ▼ │ │ │ ┌──────────────┐ │ uses │ │ │ Coordinator │ ▼ │ │ │ │ ┌──────────────┐ │ │ │ • Creates │ │ Services │ │ │ │ VCs + VMs │ │ │ │ │ │ • Navigates │ │ • API │ │ │ │ • Deep links│ │ • Storage │ │ │ └──────────────┘ │ • Analytics │ │ │ └──────────────┘ │ └─────────────────────────────────────────────────────────────┘

6.2 Complete Example

// ViewModel — reactive, testable, no UIKit
@MainActor
final class ProductListViewModel: ObservableObject {
    @Published private(set) var products: [ProductCellModel] = []
    @Published private(set) var state: ViewState = .idle

    var onSelectProduct: ((Product) -> Void)?
    var onTapCart: (() -> Void)?

    private let productService: ProductServiceProtocol

    init(productService: ProductServiceProtocol) {
        self.productService = productService
    }

    func loadProducts() async {
        state = .loading
        do {
            let products = try await productService.fetchProducts()
            self.products = products.map(ProductCellModel.init)
            state = .loaded
        } catch {
            state = .error(error.localizedDescription)
        }
    }

    func selectProduct(at index: Int) {
        guard index < products.count else { return }
        onSelectProduct?(products[index].product)
    }
}

// Coordinator — owns navigation, creates VMs
final class ProductCoordinator: Coordinator {
    func start() {
        let vm = ProductListViewModel(productService: deps.productService)
        vm.onSelectProduct = { [weak self] product in
            self?.showDetail(product)
        }
        vm.onTapCart = { [weak self] in
            self?.showCart()
        }
        let vc = ProductListVC(viewModel: vm)
        nav.pushViewController(vc, animated: false)
    }
}

8. Architect: Reactive Patterns at Scale

Architect Must-Know: Choosing Reactive Patterns That Scale

The biggest risk with reactive programming at scale is creating debugging nightmares. Long Combine chains, implicit subscriptions, and hidden state dependencies can make bugs nearly impossible to trace. The architect's job is to set guardrails.

8.1 Reactive Architecture Guidelines

  • Limit chain length — No Combine chain should exceed 5-6 operators. Break long chains into named intermediate publishers.
  • Explicit subscription management — Every AnyCancellable must be stored in a named set. No orphaned subscriptions.
  • Prefer @Observable for UI state — On iOS 17+, use Observation for ViewModel → View binding. Reserve Combine for event streams.
  • Avoid nested subscriptions — Use flatMap or switchToLatest instead of subscribing inside a sink.
  • Test the pipeline — Use Combine's TestScheduler or TCA's TestStore to verify timing-dependent behavior.

8.2 Common Pitfalls at Scale

PitfallSymptomSolution
Subscription leaksMemory grows, stale data appearsStrict cancellable management, Memory Graph Debugger
Infinite loopsCPU spikes, UI freezesremoveDuplicates(), break cycles with weak refs
Thread confusionUI updates on background threadAlways .receive(on: DispatchQueue.main) before sink
Over-abstractionCan't trace data flowLimit reactive to boundaries, use imperative inside
Testing difficultyFlaky tests, timing issuesInject schedulers, use TestScheduler
Best Practice: The Reactive Boundary Pattern

Use reactive programming at system boundaries (network → ViewModel, user input → search) but keep business logic imperative. A ViewModel that uses async/await internally but exposes @Published properties externally gives you the best of both worlds: testable imperative logic with reactive UI binding.
