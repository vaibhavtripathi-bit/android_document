The App Sandbox

Linux UID/GID isolation, Zygote forking, SELinux mandatory access control, seccomp-bpf syscall filtering, verified boot chain, dm-verity, app data isolation

1. Sandbox Layers Overview

Android's app sandbox is a stack of independent security layers. Each layer defends against different attack vectors. Even if one layer is bypassed (e.g., a kernel bug), others remain. This is defense-in-depth at the platform level.

2. Linux Process Sandbox — UID/GID Isolation

At install time, PackageManagerService assigns each app a unique Linux UID. UIDs are allocated from a reserved range (typically 10000–99999 for apps, with u0_a<id> as the human-readable form). The app's private data directory is created with that UID as owner:

// /data/data/com.example.app/ structure:
// drwx------ u0_a123 u0_a123  cache/
// drwx------ u0_a123 u0_a123  code_cache/
// drwx------ u0_a123 u0_a123  databases/
// drwx------ u0_a123 u0_a123  files/
// drwx------ u0_a123 u0_a123  shared_prefs/
// Permissions: 0700 (owner rwx only)

// In Kotlin, your app accesses its sandbox via:
val appDir = context.filesDir        // /data/data/com.example.app/files
val cacheDir = context.cacheDir      // /data/data/com.example.app/cache
val dbPath = context.getDatabasePath("mydb").absolutePath

// These paths are ONLY accessible to your app's UID.
// Other apps get EACCES (Permission denied) from the kernel.

DAC (Discretionary Access Control) enforcement happens in the kernel. When App B (UID 10124) tries to open("/data/data/com.example.app/files/secret.txt"), the kernel checks: is the caller the owner or in the file's group? No → EACCES. No framework involvement — the VFS layer enforces this.

Staff-Level Insight — UID Assignment and SharedUserId

UID assignment is deterministic: it's derived from the package name hash and stored in /data/system/packages.xml. The same package always gets the same UID on a given device. Historically, android:sharedUserId allowed two packages to share a UID (and thus each other's data). This is deprecated and blocked for new apps — it bypasses the sandbox and creates security risks. System apps that share UIDs (e.g., android.uid.system) are special-cased and must be signed with the platform key. For L6: understand that UID is the fundamental isolation primitive; everything else (SELinux, seccomp) layers on top.

3. Zygote Process — Forking App Processes

Android doesn't start each app in a fresh JVM. Instead, Zygote pre-loads the Android framework (classes, resources, native libraries) and fork()s to create new app processes. This is a performance optimization (avoid re-parsing DEX, re-loading libart) and a security boundary.

Fork Flow

// Simplified flow when you launch an app:
// 1. ActivityManagerService receives startActivity()
// 2. AMS asks Zygote to fork via ZygoteProcess (Binder/socket)
// 3. Zygote calls fork() — child process is a copy of Zygote
// 4. In child: setuid(appUid), setgid(appGid) — CRITICAL: before any app code
// 5. In child: setresuid(), setresgid() for supplemental GIDs
// 6. In child: setseinfo() for SELinux domain transition
// 7. In child: specialize() — clear Zygote state, load app's APK
// 8. In child: RuntimeInit.applicationInit() → ActivityThread.main()
// 9. App code runs with correct UID and SELinux context

Copy-on-Write (COW): fork() doesn't physically copy memory. Pages are shared until written. Zygote's pre-loaded heap is mostly read-only (DEX, ART metadata), so app processes share it efficiently. When an app modifies memory, that page is copied. This is why Zygote can pre-load ~50MB of framework without paying the cost per app.

Zygote64 and Zygote32

64-bit devices run two Zygote processes: Zygote64 (primary, for 64-bit apps) and Zygote32 (for 32-bit-only apps). The ro.zygote property selects the zygote. Apps declare android:extractNativeLibs and CPU ABI; the system chooses the matching Zygote. This prevents 32-bit apps from running in a 64-bit process (and vice versa) and maintains ABI consistency.

Staff-Level Insight — Why UID Must Be Set Before App Code

The critical security invariant: no app code runs before the process has the correct UID. If Zygote set the UID after loading the app's DEX, a malicious app could inject code that runs in the Zygote child before the UID change — and that code would still have Zygote's privileges (often running as root or system during early init). The order is: fork → setuid/setgid → setseinfo → specialize (load APK) → run app. The app's Application.onCreate() is the first app code that runs, and by then the process is fully sandboxed.

4. SELinux — Mandatory Access Control

Since Android 5 (Lollipop), SELinux runs in enforcing mode. Unlike DAC (where the file owner decides permissions), MAC policies are defined by the system and cannot be overridden — not even by root. This prevents privilege escalation: a compromised app (or root exploit) cannot access resources outside its policy.

App Domains

DomainUsed ByRestrictions
untrusted_appThird-party appsMost restrictive. Cannot access system services, other app data, raw devices.
untrusted_app_27Target SDK 27+ appsAdditional restrictions (e.g., no execmem for JIT in some configs).
platform_appPrivileged apps in /systemCan access some platform internals.
system_appCore system apps (Settings, SystemUI)Broader access for system functionality.

Type enforcement: SELinux rules specify which source type (e.g., untrusted_app) can perform which action (read, write, execute) on which target type (e.g., app_data_file). Example:

// Simplified SELinux rule (actual syntax differs):
// allow untrusted_app app_data_file:dir { read getattr };
// → untrusted_app can read directories of type app_data_file
//   ONLY if that directory is its own (enforced by file context)

// neverallow rules — compile-time checks that BLOCK dangerous policies:
// neverallow untrusted_app system_data_file:file { read write };
// → No untrusted app can ever read/write system data files

When an app tries to access a file, the kernel checks: (1) DAC — does the UID have permission? (2) SELinux — does the source domain have allow rule for this target type and action? Both must pass.

5. seccomp-bpf — Syscall Filtering

Android restricts which Linux syscalls an app can invoke. A BPF (Berkeley Packet Filter) program is loaded into the kernel and evaluated on every syscall. If the syscall is not in the allowlist, it returns ENOSYS or kills the process. The filter is applied when the Zygote child is created, before app code runs.

Blocked Syscalls (Examples)

SyscallWhy Blocked
ptraceCould attach to other processes, inject code, read memory
mount / umountCould remount filesystems read-write, escape sandbox
rebootDenial of service
init_module / finit_moduleLoad kernel modules — massive privilege escalation
keyctlKernel keyring access
personalityChange process execution domain (could disable ASLR)
// From app's perspective — these just fail:
// System.loadLibrary("native") might use restricted syscalls internally.
// If your native code calls ptrace(), it gets ENOSYS.

// Checking seccomp status (requires root or debug build):
// /proc/self/status → Seccomp: 2 (SECCOMP_MODE_FILTER)
// Filter is inherited from Zygote; all app processes have it.

The BPF filter is defined in AOSP (seccomp_policy/) and compiled into the system image. Different process types (app, isolated_app, etc.) can have different filters. The filter is additive — it only allows explicitly listed syscalls; everything else is denied.

6. Verified Boot & dm-verity

Verified Boot ensures the device boots only trusted software. The chain is:

  1. Bootloader — verified by hardware (e.g., ARM TrustZone, AVB key in efuses)
  2. Kernel — verified by bootloader (hash/signature check)
  3. System partition — verified by kernel via dm-verity

dm-verity

dm-verity is a Linux device-mapper target that provides block-level integrity verification. The system partition is divided into 4KB blocks. A Merkle tree is built: each block has a hash; parent nodes hash their children; root hash is stored in the verified metadata. At read time, the block is hashed and the chain is verified up to the root. If any block is tampered with, verification fails and the read returns I/O error.

The root hash is stored in the partition's metadata (vbmeta), which is verified by the bootloader. So: bootloader verifies vbmeta → vbmeta contains root hash → kernel uses root hash for dm-verity → system partition is read-only and integrity-protected.

AVB (Android Verified Boot)

AVB 2.0 defines the metadata format (descriptors, hashes, hashtree) and rollback protection. Rollback protection: each partition has a rollback index stored in tamper-resistant storage. If you flash an older (vulnerable) system image, the bootloader checks the stored index and refuses to boot — preventing downgrade attacks.

Verification Failure

If verification fails at boot: device shows a warning ("Your device is corrupted" or "Cannot verify system"), and may boot into a limited recovery mode or refuse to boot. User data is not wiped by default, but the device is not trusted until re-flashing a valid image.

Staff-Level Insight — Why Verified Boot Matters for App Sandbox

The app sandbox relies on trusted system binaries. Zygote, system_server, and the kernel enforce the sandbox. If an attacker could modify the system partition (e.g., patch Zygote to not set UID, or patch SELinux to permissive), the entire sandbox collapses. Verified Boot + dm-verity ensure the system partition cannot be modified without the device detecting it and refusing to run. For L6: understand that app security is only as strong as the platform it runs on. Verified Boot is the foundation — without it, a rooted device could run a malicious "Android" that looks normal but has no real isolation.

7. App Data Isolation (Android 12+)

Historically, a root-capable app could su and read /data/data/com.other.app/ because DAC allows root to bypass ownership checks. Android 12 introduced stronger isolation so that even with root, an app cannot access other apps' private data.

Namespace Isolation

Each app process gets its own mount namespace. The /data/data/ directory is mounted with a view that only includes the current app's directory. Other apps' directories are not visible — they're not even present in the namespace. So even if you have root and run ls /data/data/, you only see your own app's folder (and a few shared system dirs). The kernel's mount namespace implementation ensures this.

// From your app's perspective (Android 12+):
val dataDir = context.dataDir  // /data/data/com.example.app
// You can only see and access this. Other apps' dirs don't exist in your namespace.

// Process-level isolation means:
// - No /data/data/com.other.app in your view
// - File APIs, ContentResolver, direct path access — all restricted
// - Even with root inside your process, you can't escape the namespace

This is implemented in the Zygote specialization path: when creating the app process, the system sets up a new mount namespace and performs a private bind mount of only the app's data directory into the expected path.

8. WebView Process Isolation

WebView renders web content inside your app. That content is untrusted — it can come from any website, including malicious ones. To limit blast radius, WebView runs in a separate renderer process, sandboxed with reduced privileges.

Architecture

// Your app process (main)          WebView renderer process (isolated)
// ┌─────────────────────┐          ┌─────────────────────┐
// │ Your Activity       │  Binder  │ Chromium renderer    │
// │ WebView (stub)      │◄────────►│ Runs web JS, layout │
// │ Has your app's UID  │   IPC    │ Has isolated UID    │
// └─────────────────────┘          └─────────────────────┘
//                                   SELinux: webview_zygote
//                                   seccomp: stricter filter
//                                   No access to your app's files

The renderer runs in a webview_zygote or similar SELinux domain. It has no access to your app's /data/data/, no ability to read your SharedPreferences, and a tighter seccomp filter. If a malicious webpage exploits a renderer bug, the attacker is confined to the renderer process — they cannot directly access your app's memory or storage.

// Kotlin — enabling WebView (process isolation is automatic):
// AndroidManifest.xml:
// <application android:process=":main" ...>
//   <activity android:name=".WebActivity" />
// </application>

// In your Activity:
val webView = findViewById<WebView>(R.id.webview)
webView.settings.javaScriptEnabled = true
webView.loadUrl("https://example.com")

// The actual rendering happens in a separate process.
// You communicate via Binder; the WebView in your process is a proxy.

Why it matters: Browser engines (Chromium in WebView) are huge attack surfaces. Memory corruption, type confusion, and logic bugs are regularly found. Process isolation ensures that even a full renderer compromise doesn't automatically give the attacker your app's data — they'd need an additional exploit to break out of the renderer and into the main app process.

9. Summary — Sandbox Checklist for L6

  • UID isolation — Each app gets a unique UID; /data/data/<package>/ is 0700, DAC enforced by kernel.
  • Zygote — Forks app processes; UID/SELinux set before any app code runs; COW for performance.
  • SELinux — MAC in enforcing mode; untrusted_app domain; even root cannot bypass.
  • seccomp-bpf — Syscall filter blocks ptrace, mount, reboot, etc.; applied at fork.
  • Verified Boot + dm-verity — Boot chain and system partition integrity; rollback protection.
  • Android 12+ — Mount namespaces; other apps' /data/data/ invisible even with root.
  • WebView — Renderer in separate process with stricter SELinux and seccomp.
