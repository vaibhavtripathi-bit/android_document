Security — Interview Q&A

60 L6 Staff Engineer questions covering permissions, scoped storage, Keystore, networking, app sandbox, R8, runtime security, deep internals, and attack scenarios

Permissions

Q1. Trace the permission check from checkSelfPermission() to the kernel. What layers are involved?

ContextImpl.checkPermission() → Binder IPC to ActivityManagerService.checkPermission(pid, uid) → delegates to PermissionManagerService.checkUidPermission() → looks up grant state in memory (loaded from /data/system/packages.xml at boot). For dangerous permissions, also checks AppOpsManager for runtime overrides (e.g., "while using the app" location). At the kernel level, file access is enforced by Linux DAC (UID/GID) and SELinux MAC policies. Even if the framework check is bypassed, the kernel prevents unauthorized access.

Q2. What is AppOpsManager and how does it differ from the standard permission model?

AppOpsManager is a secondary permission system that operates on op codes (e.g., OP_CAMERA, OP_FINE_LOCATION). Even if checkSelfPermission() returns GRANTED, AppOps can silently deny the operation. This powers: "While using the app" location (permission granted, AppOps restricts to foreground), privacy indicators (green dot for camera/mic — tracked by AppOps), approximate vs precise location toggle, and background permission restrictions. AppOps also provides usage tracking — the system knows exactly when each app last used each permission.

Q3. How does Android 14's selected photos access change app architecture?

The user can grant access to specific photos instead of all. MediaStore queries return only the user-selected subset. This means: (1) gallery apps can't assume a complete view of the media library, (2) caching strategies must handle a changing set of accessible photos, (3) the app must handle re-selection flows. Internally, the system uses URI grants — the photo picker grants persistent read URIs for selected photos, and MediaStore queries are filtered transparently. The same ContentResolver.query() works but returns fewer results.

Q4. Explain the shouldShowRequestPermissionRationale() state machine. How do you distinguish "never asked" from "permanently denied"?

Three states: (1) Never asked → returns false. (2) Denied once (without "Don't ask again") → returns true. (3) Permanently denied ("Don't ask again" or denied twice on Android 11+) → returns false. States 1 and 3 return the same value. Solution: track whether you've asked before in SharedPreferences. If you've asked AND rationale returns false → user permanently denied → direct them to Settings. This is a common interview gotcha — many developers don't handle the ambiguity.

Q5. What is the custom permission race condition and how was it fixed?

If two apps declare the same custom permission with different protection levels, the first-installed app's definition wins. An attacker could install their app first with protectionLevel="normal", then your app's signature-level permission is downgraded to normal — any app can access your protected component. Mitigation: declare custom permissions in the same app that enforces them, use package name as prefix. Android 12+ prevents permission redefinition across packages unless they share a signing certificate.

Scoped Storage

Q6. How does the FUSE daemon enforce scoped storage? What are the performance implications?

Android 11+ routes all shared storage file operations through a FUSE daemon (MediaProvider process). When an app opens a file path, the kernel routes the open() syscall to the FUSE daemon, which checks: is this the app's own file? Does the app have the relevant permission? Was this file granted via MediaStore URI? If none, returns EACCES. This means even native code (JNI/NDK) respects scoped storage. Performance cost: ~10-30% overhead for sequential reads due to the extra context switch (app → kernel → FUSE userspace → kernel → filesystem). Mitigation: use ContentResolver.openFileDescriptor() for a direct fd after the initial check.

Q7. Compare content:// URIs, file:// URIs, and direct file paths. When is each appropriate?

file:// URIs are banned since API 24 (FileUriExposedException) — they expose internal paths and don't work with scoped storage. Direct file paths work only for your own files (FUSE-enforced on Android 11+). content:// URIs are the correct approach: opaque paths (receiver doesn't know real location), temporary permissions (revoked when Activity finishes), explicit read/write control. For cross-app sharing: FileProvider converts file paths to content URIs. For media: MediaStore returns content URIs. For arbitrary documents: SAF returns content URIs.

Q8. How does IS_PENDING work in MediaStore and why is it important?

When creating a media file, set IS_PENDING = 1 before writing. The file is invisible to other apps and MediaStore queries until you set IS_PENDING = 0. This prevents other apps from seeing partially-written files (corrupted images, incomplete downloads). Without IS_PENDING, another app could query MediaStore and get a half-written file. This is especially important for large files (videos, downloads) where the write takes significant time.

Keystore & Cryptography

Q9. Explain the difference between TEE and StrongBox. When would you require StrongBox?

TEE (Trusted Execution Environment) uses ARM TrustZone — a secure world running on the same processor. Fast, available on all modern devices. StrongBox is a separate secure element chip (e.g., Pixel's Titan M) — physically isolated, tamper-resistant, has its own CPU/memory/storage. StrongBox is slower but resistant to physical attacks (side-channel, power analysis, chip decapping). Require StrongBox for: banking/payment apps, enterprise key management, DRM. TEE is sufficient for most apps. Check availability: packageManager.hasSystemFeature(FEATURE_STRONGBOX_KEYSTORE).

Q10. How does key attestation prove device integrity to a remote server?

Generate a key with setAttestationChallenge(serverNonce). Retrieve the certificate chain — it roots to Google's attestation CA. The leaf certificate contains an extension (OID 1.3.6.1.4.1.11129.2.1.17) with: security level (TEE/StrongBox), verified boot state, OS version, patch level, bootloader lock state, and the challenge nonce. The server verifies the chain and extension to confirm: the key is in genuine hardware, the device isn't rooted, the OS is up-to-date. This is the foundation of Play Integrity API.

Q11. Why does setInvalidatedByBiometricEnrollment matter for banking apps?

When set to true, the key becomes permanently unusable if a new biometric is enrolled. Without it, an attacker who gains physical access could add their own fingerprint and use existing biometric-bound keys. Banking apps must use this — the user re-enrolls (re-enters password) after adding a new biometric. The trade-off is UX friction vs security. The key is invalidated at the Keystore level — any attempt to use it throws KeyPermanentlyInvalidatedException.

Q12. How does EncryptedSharedPreferences work under the hood?

Uses Google's Tink library. A MasterKey (AES-256-GCM) is stored in Android Keystore. A keyset (collection of data encryption keys) is encrypted with the MasterKey and stored alongside the prefs file. Preference keys are encrypted with AES-256-SIV (deterministic — same key always produces same ciphertext, needed for HashMap lookup). Preference values are encrypted with AES-256-GCM (randomized). The keyset approach means rotating the MasterKey only requires re-encrypting the keyset wrapper, not all data.

Secure Networking

Q13. Why did Android 7 stop trusting user-installed CA certificates by default?

Before Android 7, user-installed CAs were trusted for all apps. Corporate MDM or attackers with physical access could install a CA and MITM all HTTPS traffic. Android 7 changed the default: apps targeting API 24+ only trust system CAs. User CAs require explicit opt-in via Network Security Config. This broke debugging proxies (Charles, Fiddler) — intentionally. For debug builds, opt in with <debug-overrides><trust-anchors><certificates src="user"/>.

Q14. Compare certificate pinning strategies. What are the risks of each?

Leaf pinning: pin the server cert's SPKI hash. Most secure but breaks when cert rotates — requires app update. Intermediate CA pinning: survives leaf rotation but breaks on CA change. Root CA pinning: survives most rotations but provides less protection (any cert from that CA is trusted). All strategies risk bricking the app if pins become invalid. Mitigations: always include a backup pin, set expiration dates, prefer Network Security Config (has expiration) over OkHttp pinning (no expiration). Google recommends against pinning for most apps — system CA trust is sufficient.

Q15. What is Certificate Transparency and how does Android enforce it?

CT requires every certificate to be logged in public CT logs. Android 10+ enforces CT: certificates issued after April 2018 must have SCTs (Signed Certificate Timestamps) from at least 2 CT logs. If SCTs are missing, the TLS connection fails. CT prevents a compromised CA from issuing rogue certificates without detection — the rogue cert would appear in public logs, and monitoring services would alert the domain owner. As a developer, CT is transparent — no code changes needed.

App Sandbox

Q16. Explain the Android app sandbox. What are the layers of isolation?

Six layers: (1) Linux DAC — each app gets a unique UID; files are owned by that UID with 0700 permissions. (2) SELinux MAC — mandatory access control policies restrict what each app domain can access, even with root. (3) seccomp-bpf — syscall filtering blocks dangerous syscalls (ptrace, mount, reboot). (4) Zygote isolation — new UID assigned after fork, before app code runs. (5) Namespace isolation (Android 12+) — per-app mount namespaces prevent accessing other apps' /data/data. (6) Verified boot — dm-verity ensures system partition integrity.

Q17. How does Zygote work and why is it a security boundary?

Zygote is a special process that pre-loads framework classes and resources. When a new app starts, Zygote fork()s — the child gets copy-on-write memory with all framework classes already loaded (fast startup). Critically, the new UID is assigned after fork but before app code runs. This means the app never runs with Zygote's elevated privileges. The sequence: fork() → setuid(appUid) → setgid(appGid) → apply SELinux context → apply seccomp filter → load app code. If any step fails, the process is killed.

Q18. How does SELinux prevent privilege escalation on Android?

SELinux uses type enforcement: every process has a domain (e.g., untrusted_app), every file has a type (e.g., app_data_file). Access is only allowed if an explicit allow rule exists. Even root can't bypass MAC — there's no rule allowing untrusted_app to access system_data_file. Android also uses neverallow rules that are verified at build time — they can't be overridden by device manufacturers. Key domains: untrusted_app (third-party apps), platform_app (system apps), system_server (AMS, PMS, etc.).

R8 / ProGuard

Q19. What are R8's three phases and what does each accomplish?

(1) Shrinking (tree shaking): traces from entry points, removes unreachable code. Typically removes 30-70% of code. (2) Obfuscation: renames classes/methods/fields to short names (a, b, c). Reduces DEX size 5-15% and makes reverse engineering harder. (3) Optimization: method inlining, class merging, enum unboxing, constant propagation, devirtualization. R8 does all three in a single pass (unlike ProGuard which required a separate dexing step).

Q20. What is R8 full mode and when should you use it?

Full mode (android.enableR8.fullMode=true) enables aggressive optimizations that break ProGuard assumptions: -keepclassmembers doesn't prevent class removal if unused, default constructors aren't kept unless explicitly used, attributes are stripped more aggressively. Full mode typically removes 10-20% more code. Use for mature apps with well-tested keep rules. Most modern libraries ship R8-compatible rules. Full mode is recommended for production — smaller APKs and faster startup.

Q21. A release build crashes with ClassNotFoundException. How do you debug it?

(1) Get the obfuscated stack trace from crash reporting. (2) Deobfuscate with retrace mapping.txt stacktrace.txt. (3) Identify the missing class. (4) Check usage.txt — if the class is listed, R8 removed it. (5) Determine why: is it accessed via reflection? JNI? XML layout? (6) Add appropriate keep rule. (7) For reflection: -keep class com.example.ReflectedClass { *; }. For JSON models: ensure @SerializedName or keep rules. (8) Rebuild and verify. Always archive mapping.txt for every release.

Q22. How does R8 handle Kotlin metadata and why does it matter?

Kotlin stores metadata in @Metadata annotations — data class structure, nullability, default parameters. Libraries like Moshi-kotlin and kotlinx.serialization read this via reflection. If R8 strips @Metadata, these libraries break silently (wrong field mapping) or crash. In full mode, metadata is stripped more aggressively. Fix: keep metadata for serialized classes, or use the library's shipped R8 rules. kotlinx.serialization's compiler plugin generates serializers at compile time, avoiding this issue entirely — prefer it over reflection-based serializers.

Runtime Security

Q23. What changed with android:exported in Android 12 and why?

Android 12 requires android:exported to be explicitly declared for all components with intent filters. Before, it defaulted to true if an intent filter was present — many developers didn't realize their Activities/Services were accessible to other apps. This caused security vulnerabilities: malicious apps could launch unexported Activities with crafted intents. Now the build fails if exported isn't declared, forcing developers to make a conscious decision.

Q24. Explain the PendingIntent hijacking attack. How does FLAG_IMMUTABLE prevent it?

A PendingIntent wraps an Intent that will be sent later by another app (e.g., AlarmManager, NotificationManager). If the PendingIntent is mutable, the sending app can modify the wrapped Intent — changing the target component, adding extras, or changing the action. An attacker could intercept a mutable PendingIntent and redirect it to their own component. FLAG_IMMUTABLE prevents modification — the wrapped Intent can't be changed after creation. Android 12+ requires FLAG_IMMUTABLE or FLAG_MUTABLE to be explicitly set. Use FLAG_MUTABLE only when needed (inline replies, bubbles).

Q25. How does SQL injection work through a ContentProvider? How do you prevent it?

If a ContentProvider builds SQL queries by concatenating user input into the selection string: "name = '" + userInput + "'", an attacker can inject: '; DROP TABLE users; --. Prevention: always use parameterized queries: selection = "name = ?", selectionArgs = arrayOf(userInput). The selection args are escaped by SQLite. Also validate the projection (column names) — an attacker could request columns from other tables via subqueries. Use a whitelist of allowed columns.

Q26. What is a tapjacking attack and how do you defend against it?

A malicious app draws an overlay (using SYSTEM_ALERT_WINDOW) on top of your app's UI, tricking the user into tapping a button they can't see (e.g., "Grant permission" hidden under a "Play game" button). Defenses: (1) android:filterTouchesWhenObscured="true" on sensitive views — ignores touches when an overlay is present. (2) Programmatically check MotionEvent.FLAG_WINDOW_IS_OBSCURED for fine-grained control. (3) Android 12+ restricts overlay touch passthrough — overlays can't pass touches to the app below unless they're fully transparent in that area. Note: FLAG_SECURE prevents screenshots/screen recording but does NOT prevent tapjacking — it is a separate defense for a different threat.

Q27. Why is addJavascriptInterface dangerous and how was it fixed?

Before API 17, addJavascriptInterface() exposed all public methods of the bound object to JavaScript — including inherited methods from Object (getClass(), etc.). An attacker could use reflection via JavaScript to execute arbitrary code: obj.getClass().forName("java.lang.Runtime").getMethod("exec").invoke(). API 17+ requires @JavascriptInterface annotation — only annotated methods are exposed. Always target API 17+ and only expose the minimum necessary methods. Never load untrusted URLs in a WebView with JavaScript interfaces.

System Design & Advanced

Q28. Design a secure authentication flow for a banking app on Android.

(1) Initial login: username/password over TLS with certificate pinning. (2) On success, generate an EC key pair in Keystore with setUserAuthenticationRequired(true), setInvalidatedByBiometricEnrollment(true), StrongBox-backed. (3) Send the public key to the server, signed with key attestation. Server verifies attestation chain. (4) For subsequent logins: BiometricPrompt with CryptoObject → sign a server challenge with the private key → server verifies signature. (5) Store refresh tokens in EncryptedSharedPreferences. (6) Use Play Integrity API to verify device integrity. (7) Implement session timeout and re-authentication for sensitive operations.

Q29. How would you implement secure data-at-rest for a health app that stores sensitive medical records?

(1) Use EncryptedFile (Jetpack Security) for all medical record files — AES-256-GCM with Keystore-backed MasterKey. (2) Use EncryptedSharedPreferences for metadata and settings. (3) For the database: SQLCipher with a key derived from Keystore. (4) Require biometric authentication to access records (setUserAuthenticationParameters(0, BIOMETRIC_STRONG)). (5) Implement FLAG_SECURE on all screens showing medical data. (6) Clear sensitive data from memory after use (zero-fill byte arrays). (7) Handle process death: re-authenticate on cold start. (8) Comply with HIPAA: audit logging of all data access.

Q30. A security audit finds that your app's API keys are visible in the decompiled APK. How do you fix this?

R8 obfuscation doesn't hide string constants — they're visible in the DEX constant pool. Solutions: (1) Don't embed secrets in the APK — fetch them from your server after authentication. (2) For API keys that must be in the APK (e.g., Maps API key): restrict the key server-side (by package name + signing cert fingerprint). (3) Use NDK to store keys in native code (harder to decompile but not impossible). (4) Use Google's Secrets Gradle Plugin to keep keys out of version control. (5) For truly sensitive operations, proxy through your server — the client never sees the third-party API key.

Q31. How does Android's verified boot chain protect the app sandbox?

Verified boot ensures the system partition hasn't been tampered with: bootloader verifies the kernel, kernel verifies the system partition via dm-verity (Merkle tree hash verification). If verification fails, the device either refuses to boot or shows a warning. This protects the sandbox because: SELinux policies live on the system partition — a tampered system could have weakened policies. The Zygote binary is on the system partition — a tampered Zygote could skip UID assignment. Framework JARs are on the system partition — tampered frameworks could bypass permission checks. Rollback protection prevents downgrading to a vulnerable version.

Q32. Compare SafetyNet Attestation, Play Integrity API, and Key Attestation. When would you use each?

SafetyNet is deprecated — replaced by Play Integrity API. Play Integrity API: high-level device/app/account integrity check. Easy to integrate, provides verdicts (MEETS_DEVICE_INTEGRITY, PLAY_RECOGNIZED, LICENSED). Use for: anti-fraud, anti-abuse, license verification. Key Attestation: low-level cryptographic proof that a key exists in genuine hardware. Provides detailed device info (boot state, OS version, patch level). Use for: strong authentication flows, device binding, enterprise security. Play Integrity uses key attestation internally but abstracts the complexity.

Q33. How would you implement secure inter-app communication between your suite of apps?

(1) Use signature-level custom permissions — only apps signed with your certificate can communicate. (2) Validate the caller's signing certificate in the receiving component: packageManager.getPackageInfo(callerPackage, GET_SIGNING_CERTIFICATES). (3) Use ContentProvider with android:protectionLevel="signature" for data sharing. (4) For real-time communication: bound service with AIDL, protected by signature permission. (5) Never trust intent extras without validation — even from your own apps (defense in depth). (6) Use PendingIntent with FLAG_IMMUTABLE for callbacks.

Q34. What are the security implications of WebView in your app?

WebView loads untrusted web content in your app's process (or a sandboxed renderer process on Android 8+). Risks: (1) JavaScript injection via addJavascriptInterface. (2) Intent scheme URLs launching arbitrary intents. (3) File access — setAllowFileAccess(true) lets JS read local files. (4) Mixed content — HTTP resources in HTTPS pages. (5) Cookie theft if WebView shares cookies with the browser. Mitigations: use @JavascriptInterface, override shouldOverrideUrlLoading(), disable file access, enable Safe Browsing, use process isolation (Android 8+), set setMixedContentMode(MIXED_CONTENT_NEVER_ALLOW).

Q35. Why is root detection unreliable and what's the alternative?

Root detection checks (su binary, Magisk files, build tags) are easily bypassed by Magisk Hide, Zygisk, and similar tools. The cat-and-mouse game is unwinnable — root hiding tools are always one step ahead. Better alternative: Play Integrity API — it uses hardware-backed key attestation to verify device integrity. The attestation happens in the TEE, which Magisk can't tamper with. Even a rooted device fails the MEETS_DEVICE_INTEGRITY check. For defense in depth: combine Play Integrity with server-side validation, and design your app so that sensitive operations happen server-side.

Q36. How does Android 12+ app data isolation work at the filesystem level?

Android 12+ uses per-app mount namespaces. Each app sees a different view of /data/data/ — it can only see its own directory. Even with root access, an app's mount namespace doesn't include other apps' directories. This is enforced at the kernel level via unshare(CLONE_NEWNS) during Zygote fork. Combined with SELinux policies that prevent remounting, this makes cross-app data access virtually impossible without a kernel exploit.

Q37. How would you securely store and rotate API tokens in a production app?

(1) Store tokens in EncryptedSharedPreferences (Keystore-backed). (2) Use short-lived access tokens (15-30 min) with long-lived refresh tokens. (3) Refresh tokens should be bound to the device — server validates device fingerprint. (4) On token refresh, invalidate the old refresh token (rotation). (5) If a refresh token is used twice (replay attack), invalidate all tokens for that user. (6) Use certificate pinning for the auth endpoint. (7) Clear tokens on logout (overwrite with zeros, then delete). (8) Handle token expiry gracefully — queue requests and retry after refresh.

Q38. What security considerations apply when using Jetpack Compose?

Compose-specific: (1) Modifier.semantics — ensure sensitive fields (passwords) have password = true to prevent screen readers from reading them aloud. (2) TextField with visualTransformation = PasswordVisualTransformation() — masks input. (3) FLAG_SECURE still applies to the window (set on the Activity). (4) Compose's AndroidView for WebView — same security considerations as View-based WebView. (5) State management — don't store sensitive data in rememberSaveable (persisted to Bundle, which can be read by other apps on rooted devices). Use EncryptedSharedPreferences or Keystore instead.

Q39. How does R8 interact with the security of your app beyond obfuscation?

R8 is a security tool in three ways: (1) Attack surface reduction — shrinking removes unused code, including unused library features that could have vulnerabilities. (2) Obfuscation — makes reverse engineering harder (but not impossible). (3) Log stripping — R8 rules can remove all Log.d()/Log.v() calls from release builds, preventing information leakage. Limitations: string constants are still visible in DEX, control flow can be reconstructed with tools like jadx, and native code isn't affected. For sensitive logic, server-side enforcement is always better.

Q40. Design a comprehensive security architecture for a large-scale Android app at Google.

(1) Network: TLS 1.3, certificate pinning for critical endpoints, Network Security Config. (2) Storage: EncryptedSharedPreferences for tokens, EncryptedFile for sensitive data, SQLCipher for databases. (3) Authentication: Keystore-backed biometric keys, key attestation for device binding, Play Integrity for device integrity. (4) IPC: signature permissions for inter-app communication, explicit exported declarations, PendingIntent FLAG_IMMUTABLE. (5) Code: R8 full mode, Baseline Profiles, log stripping. (6) Runtime: filterTouchesWhenObscured, FLAG_SECURE for sensitive screens, WebView hardening. (7) Monitoring: server-side anomaly detection, token rotation, Play Integrity checks on sensitive operations. (8) Process: security review in CI (lint checks for exported components, cleartext traffic, etc.).

Deep Internals & Edge Cases

Q41. Explain the AppOps enforcement layer. How does "While using the app" location work internally?

AppOps operates on op codes (e.g., OP_FINE_LOCATION = 1, OP_CAMERA = 26) that map to permissions. Three key methods: checkOp() (check without side effects), noteOp() (check and record usage — triggers privacy indicator), startOp()/finishOp() (for long-running ops — shows green dot). "While using the app" works by setting the AppOps mode to MODE_FOREGROUND: the permission is GRANTED, but AppOps only allows the op when the app has a visible Activity or foreground service. When the app goes to background, AppOps silently denies the op. The permission grant state doesn't change — only the AppOps enforcement mode.

Q42. What is the confused deputy attack in Android? How do you prevent it?

A malicious app asks your app (the "deputy") to perform an action your app has permission for but the caller doesn't. Example: App A asks your ContentProvider to read contacts — your provider has READ_CONTACTS but App A doesn't. If your provider doesn't check the caller's permissions, it proxies unauthorized access. Defense: use enforceCallingPermission() (checks only the caller's UID) not enforceCallingOrSelfPermission() (also allows the service itself). Use Binder.getCallingUid() to verify the caller. The clearCallingIdentity()/restoreCallingIdentity() pattern must always use try/finally — forgetting to restore is a privilege escalation vulnerability (multiple Android CVEs).

Q43. Trace the path from KeyStore API to hardware. What is KeyMint?

App (KeyStore API) → Binder IPC → keystored daemon (Rust-based on Android 12+, Keystore 2.0) → KeyMint HAL (AIDL-based, replaces Keymaster) → TEE/StrongBox hardware. KeyMint defines operations: generateKey, importKey, begin, update, finish, abort. Each is a Binder call to the HAL service. Keys are stored as encrypted "key blobs" in /data/misc/keystore2/ — the blob is encrypted with a hardware-bound key and is opaque to everything except the TEE. Keystore 2.0 uses SQLite for metadata and supports per-user key scoping.

Q44. What happens when you call setIsStrongBoxBacked(true) on a device without StrongBox?

It throws StrongBoxUnavailableException — there is no fallback to TEE. This is intentional: requesting StrongBox is a security decision that TEE isn't sufficient. You must handle this explicitly: check packageManager.hasSystemFeature(FEATURE_STRONGBOX_KEYSTORE) first, or catch the exception and fall back to TEE manually. Also note: StrongBox is 10-100x slower than TEE for RSA operations because the secure element has a much slower processor. Use StrongBox only for high-value keys (banking, payment) where physical attack resistance matters.

Q45. What is Conscrypt and why does it matter for Android security?

Conscrypt is Android's TLS provider — a Java Security Provider backed by BoringSSL (Google's OpenSSL fork). It handles TLS handshake, certificate verification, session management, and implements the X509TrustManager that reads Network Security Config. Since Android 10, Conscrypt is a Mainline module — updatable via Play Store without a full OS update. This means TLS vulnerabilities can be patched independently of device manufacturers. Conscrypt creates a RootTrustManager that delegates to per-domain NetworkSecurityTrustManager instances, each wrapping the configured trust anchors and pin set from NSC.

Q46. Explain TLS 1.3 0-RTT. Why is it dangerous and how does Android handle it?

TLS 1.3 0-RTT (Early Data) allows the client to send data in the first packet — zero round trips. The server can process it immediately, reducing latency. The danger: 0-RTT data is replayable. An attacker can capture the first packet and resend it — the server processes it again. Never use 0-RTT for non-idempotent operations (purchases, transfers, state changes). OkHttp and Conscrypt do NOT enable 0-RTT by default for this reason. TLS 1.3 also mandates forward secrecy (ECDHE) — even if the server's private key is later compromised, past sessions can't be decrypted.

Q47. How does seccomp-bpf protect Android apps? What syscalls are blocked?

seccomp-bpf (Secure Computing with Berkeley Packet Filter) restricts which syscalls an app process can make. Applied at Zygote fork time — the BPF filter program is loaded before app code runs. Blocked syscalls include: ptrace (debugging other processes), mount/umount (filesystem manipulation), reboot, init_module/delete_module (kernel modules), swapon/swapoff. Different policies for different domains: untrusted_app has the most restrictive filter, isolated_app (WebView renderer) is even more restricted. If a blocked syscall is attempted, the process receives SIGKILL.

Q48. How do broadcast injection attacks work? How do you secure BroadcastReceivers?

Any app can send broadcasts to exported receivers. Attack: malicious app sends Intent("com.example.PAYMENT_COMPLETE") to trigger your payment receiver with fake data. Defenses: (1) Add android:permission to the receiver — only apps with that permission can trigger it. (2) Use explicit intents for app-internal broadcasts. (3) For ordered broadcasts, a high-priority malicious receiver can abortBroadcast() — intercepting data meant for your receiver. (4) Never use sticky broadcasts for sensitive data (deprecated, any app can read). (5) Use LocalBroadcastManager or Kotlin Flows for in-process communication.

Q49. How does the Storage Access Framework (SAF) work internally? What are persistable URI permissions?

SAF uses DocumentsProvider (a ContentProvider subclass) to expose files. When the user picks a file via ACTION_OPEN_DOCUMENT, the system grants a temporary URI permission to the requesting app. takePersistableUriPermission() converts this to a persistent grant that survives reboots — stored in /data/system/urigrants.xml. Android 11+ restricts ACTION_OPEN_DOCUMENT_TREE: can't access root, Downloads, Android/data, or Android/obb. SAF is the only way to access arbitrary documents across apps without broad storage permissions.

Q50. What is the RecoverableSecurityException flow for modifying other apps' media?

On Android 10+, modifying or deleting another app's media throws RecoverableSecurityException. The exception contains a userAction with an IntentSender that launches a system dialog asking the user for consent. On Android 11+, batch operations are available: MediaStore.createDeleteRequest(), createWriteRequest(), createTrashRequest(), createFavoriteRequest() — each returns a PendingIntent for a system consent dialog covering multiple URIs at once. This replaces the per-file exception pattern with a more user-friendly batch flow.

Q51. How does Android's auto-revoke (permission hibernation) work? What are the implications for background apps?

Android 11+: if an app isn't launched for several months (typically 3, varies by OEM), all runtime permissions are revoked, notification channels are cleared, and on Android 12+ cached data may be cleared too. The app is "hibernated." Implications: (1) Background-only apps (health trackers, backup apps) lose permissions silently. (2) The app must re-request permissions on next launch. (3) Apps can request exemption via Settings.ACTION_APPLICATION_DETAILS_SETTINGS — user must manually toggle "Remove permissions if app isn't used." (4) Check status: packageManager.isAutoRevokeWhitelisted. Device management, family safety, and enterprise apps should request exemption.

Q52. Walk through the Android 13 granular media permissions. How do they differ from READ_EXTERNAL_STORAGE?

Android 13 split READ_EXTERNAL_STORAGE into three permissions: READ_MEDIA_IMAGES, READ_MEDIA_VIDEO, READ_MEDIA_AUDIO. Each grants access only to that media type. Apps targeting API 33+ must request the specific type they need — requesting READ_EXTERNAL_STORAGE on API 33+ grants nothing. Android 14 added READ_MEDIA_VISUAL_USER_SELECTED for partial photo access. The migration path: declare all needed media permissions with maxSdkVersion on READ_EXTERNAL_STORAGE for backward compatibility. On Android 13+, the system maps the old permission to the new ones automatically for apps targeting older APIs.

Q53. How does the Android clipboard expose sensitive data, and what defenses exist?

The system clipboard (ClipboardManager) is globally readable on Android < 10. Any app in the foreground could read clipboard contents. Android 10+ restricts clipboard read to the foreground app and the default IME. Android 12+ shows a toast when an app reads the clipboard. Android 13+ auto-clears clipboard after ~1 hour. Defenses: (1) Never copy passwords/tokens to clipboard — use Autofill API instead. (2) Mark sensitive data with ClipDescription.EXTRA_IS_SENSITIVE (Android 13+) so the keyboard preview is hidden. (3) For OTP fields, use the SMS Retriever API or Credential Manager. (4) If you must use clipboard, clear it immediately after paste with clipboardManager.setPrimaryClip(ClipData.newPlainText("", "")).

Q54. Explain the intent redirection attack. How does it differ from the confused deputy?

Intent redirection: an attacker sends an Intent to your exported Activity containing a nested Intent in extras. Your Activity reads the nested Intent and calls startActivity() on it, effectively laundering the attacker's intent through your app's identity and permissions. This differs from the confused deputy (where the attacker asks your Service/Provider to perform a privileged action) because the victim is an Activity that blindly forwards intents. Defense: (1) Never call startActivity() on an Intent received from extras without validation. (2) Verify the target component's package matches your own. (3) Use explicit intents with hardcoded component names instead of forwarding arbitrary intents. (4) If forwarding is required, strip all extras and rebuild the Intent with only validated data.

Q55. How do you prevent path traversal in a ContentProvider? Walk through the attack and defense.

Attack: a malicious app calls contentResolver.openFileDescriptor(Uri.parse("content://com.example/files/../../databases/secret.db"), "r"). If the provider naively concatenates the path segment with a base directory, the ../ escapes the intended directory and reads private files. Defense: (1) Resolve the file path using File(baseDir, pathSegment).canonicalFile. (2) Verify the canonical path starts with the base directory's canonical path. (3) Reject any path containing .. at the URI level as an early check. (4) Prefer FileProvider over custom providers — FileProvider validates paths against declared <paths> in XML. (5) Use android:grantUriPermissions with specific <grant-uri-permission> path patterns rather than granting access to the entire provider.

Q56. Why can't an app forge Binder.getCallingUid()? Trace the enforcement path from app to kernel.

When an app makes a Binder IPC call, the data passes through the Binder kernel driver (drivers/android/binder.c). In binder_transaction(), the driver copies the caller's UID and PID from current->cred — the kernel's process credential structure. This is set during fork() from Zygote when setuid() is called with the app's assigned UID. Since the credential is in kernel memory, no userspace code can modify it. On the receiving side, Binder.getCallingUid() reads this kernel-provided value from the transaction data. The entire chain is: App process → ioctl(BINDER_WRITE_READ) → kernel copies current->cred->uid into transaction → receiver reads UID from transaction. This is why Binder-based permission checks are trustworthy and form the foundation of Android's IPC security model.

Q57. Compare APK signing schemes v1, v2, v3, and v4. When would you use key rotation?

v1 (JAR signing): Signs individual files in META-INF/. Slow verification, vulnerable to adding unsigned files. v2 (APK Signature Scheme, Android 7+): Signs the entire APK as a binary blob. Faster verification, detects any modification. v3 (Android 9+): Adds key rotation support via proof-of-rotation struct — a linked list of old→new certificate pairs. Allows changing signing key while maintaining trust chain. v4 (Android 11+): Adds a separate .idsig file for incremental installation (adb install --incremental). Doesn't replace v2/v3 — it's supplementary. Key rotation use case: your signing key is compromised, or you're migrating from a legacy key. Use apksigner rotate to create the lineage, then sign with v3. Caveat: some OEMs and enterprise MDMs don't fully support v3 rotation — test thoroughly. Google Play App Signing holds your upload key separately from the signing key, making rotation simpler.

Q58. How does App Links verification work internally? What happens if verification fails?

When an app with android:autoVerify="true" is installed, the system's IntentFilterVerificationAgent sends HTTPS requests to each declared host at /.well-known/assetlinks.json. The JSON must contain the app's package name and SHA-256 certificate fingerprint. Verification happens at install time (and periodically on Android 12+). If verification succeeds, the app is set as the default handler for that domain — no disambiguation dialog. If verification fails: (1) The app is NOT the default handler. (2) The user sees a chooser dialog. (3) On Android 12+, unverified deep links to web domains open in the browser by default. Common failure causes: wrong certificate fingerprint (use keytool -printcert), server returns non-200, JSON syntax error, CDN caching stale file, redirect (must be direct 200). Debug with adb shell pm get-app-links com.example.app.

Q59. What is the WebView onReceivedSslError trap? Why does Google Play reject apps that call proceed()?

When WebView encounters a TLS error (expired cert, hostname mismatch, untrusted CA), it calls onReceivedSslError(). Calling handler.proceed() tells WebView to ignore the error and load the page anyway — this is a complete MITM vulnerability. Google Play's automated scanner flags any app that calls proceed() unconditionally and rejects the upload. The correct approach: always call handler.cancel() and show an error page. For development/testing, use Network Security Config's debug-overrides to trust user-installed CAs instead. Critical nuance: WebView uses its own TLS stack separate from OkHttp — Network Security Config does NOT apply to WebView.loadUrl(). WebView's trust decisions go through its own X509TrustManager implementation in the WebView APK (updatable via Play Store).

Q60. Design a secure OTA update system for a critical Android app (e.g., banking). What are the trust boundaries?

Architecture: (1) Server-side: Build server signs the update payload with an Ed25519 key (private key in HSM). Generates a manifest with version, SHA-256 hash of each file, and signature. (2) Delivery: Use certificate-pinned HTTPS. Include a nonce in the download URL to prevent replay. Use Play Integrity API to verify the requesting device is genuine. (3) Client-side verification: Before applying, verify the manifest signature using a hardcoded public key (stored in native code, not Java — harder to patch). Verify SHA-256 of downloaded payload matches manifest. Check version is strictly newer (prevent rollback). (4) Application: Use Android's PackageInstaller API for self-update (not ACTION_INSTALL_PACKAGE). The system verifies APK signature matches the installed app. (5) Trust boundaries: Server build pipeline (HSM protects signing key), network (TLS + pinning), client verification (signature + hash), system installer (APK signature scheme v2/v3). (6) Edge cases: Interrupted download (resume with range headers + re-verify hash), disk full (check before download), downgrade attack (enforce monotonic version), compromised CDN (signature verification catches tampered payload).
