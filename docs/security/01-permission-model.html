Permission Model Evolution

From install-time blanket grants to granular runtime permissions — how Android's permission system works internally, enforcement layers, and every major evolution through Android 15

1. Permission Architecture Overview

Permissions are enforced at multiple layers. Even if an app bypasses the framework check, the kernel's SELinux policies and UID isolation prevent unauthorized access. This defense-in-depth is critical for L6 understanding.

2. Permission Types

Protection LevelGrant TimeExamplesInternals
normal Install time (automatic) INTERNET, VIBRATE, BLUETOOTH, WAKE_LOCK PackageManagerService grants automatically. No user prompt. Low risk.
dangerous Runtime (user prompt) CAMERA, LOCATION, CONTACTS, MICROPHONE PermissionManagerService tracks grant state. Grouped by permission group. Revocable.
signature Install time (if signing cert matches) BIND_NOTIFICATION_LISTENER, MANAGE_DOCUMENTS Only granted if requesting app is signed with the same certificate as the declaring app (or platform).
signatureOrSystem Install time (deprecated) Legacy system permissions Replaced by signature|privileged. Granted to apps in /system/priv-app/.

4. Enforcement Internals — How Permissions Are Checked

// When your app calls context.checkSelfPermission(CAMERA):

// 1. ContextImpl.checkPermission() is called
//    → Binder IPC to ActivityManagerService

// 2. ActivityManagerService.checkPermission(permission, pid, uid)
//    → Delegates to PermissionManagerService

// 3. PermissionManagerService.checkUidPermission(permission, uid)
//    → Looks up the permission grant state in memory
//    → State loaded from /data/system/packages.xml at boot

// 4. If permission is "dangerous" and runtime:
//    → Also checks AppOpsManager for runtime overrides
//    → AppOps can silently deny even if permission is "granted"
//      (e.g., user toggled off in Settings after granting)

// 5. Returns PERMISSION_GRANTED or PERMISSION_DENIED

// For content providers, the check happens at the Binder level:
// ContentProvider.enforceReadPermission() checks the caller's UID
// against the provider's android:readPermission attribute
Staff-Level Insight — AppOpsManager: The Hidden Permission Layer

AppOpsManager is a secondary permission system that sits on top of the standard permission model. Even if checkSelfPermission() returns GRANTED, AppOps can silently deny the operation. This is how "While using the app" location works — the permission is granted, but AppOps restricts it to foreground-only. AppOps also tracks permission usage (the green dot indicator for camera/mic). For L6: understand that AppOps is the mechanism behind permission usage indicators, background permission restrictions, and the "approximate vs precise location" toggle. It operates on op codes (e.g., OP_CAMERA, OP_FINE_LOCATION) that map to but are distinct from Android permissions.

5. Runtime Permission Request Flow

// Modern permission request using Activity Result API:
class CameraActivity : AppCompatActivity() {

    private val cameraPermissionLauncher = registerForActivityResult(
        ActivityResultContracts.RequestPermission()
    ) { isGranted ->
        if (isGranted) {
            openCamera()
        } else {
            // Check if we should show rationale
            if (shouldShowRequestPermissionRationale(Manifest.permission.CAMERA)) {
                showRationaleDialog()
            } else {
                // User selected "Don't ask again" — direct to Settings
                showSettingsRedirect()
            }
        }
    }

    fun requestCamera() {
        when {
            checkSelfPermission(Manifest.permission.CAMERA) == PERMISSION_GRANTED -> {
                openCamera()
            }
            shouldShowRequestPermissionRationale(Manifest.permission.CAMERA) -> {
                showRationaleDialog()
            }
            else -> {
                cameraPermissionLauncher.launch(Manifest.permission.CAMERA)
            }
        }
    }
}

shouldShowRequestPermissionRationale() State Machine

// The return value depends on the permission's history:
//
// State 1: Never asked before
//   shouldShowRationale() = false
//   → Show the system dialog directly
//
// State 2: User denied once (without "Don't ask again")
//   shouldShowRationale() = true
//   → Show your rationale UI, then re-request
//
// State 3: User denied with "Don't ask again" (or denied twice on Android 11+)
//   shouldShowRationale() = false
//   → System dialog won't appear. Direct user to Settings.
//
// Problem: States 1 and 3 return the same value (false).
// Solution: Track whether you've asked before in SharedPreferences.
// If you've asked before AND rationale is false → user permanently denied.

6. Android 14+ Selected Photos Access

// Android 14 introduces partial media access:
// User can grant access to SPECIFIC photos instead of all.

// Manifest:
// <uses-permission android:name="android.permission.READ_MEDIA_IMAGES" />
// <uses-permission android:name="android.permission.READ_MEDIA_VISUAL_USER_SELECTED" />

// Request flow:
val mediaPermissionLauncher = registerForActivityResult(
    ActivityResultContracts.RequestMultiplePermissions()
) { permissions ->
    val fullAccess = permissions[Manifest.permission.READ_MEDIA_IMAGES] == true
    val partialAccess = permissions[
        Manifest.permission.READ_MEDIA_VISUAL_USER_SELECTED
    ] == true

    when {
        fullAccess -> loadAllPhotos()
        partialAccess -> {
            // User selected specific photos — query MediaStore
            // Only the selected photos will be returned
            loadSelectedPhotos()
        }
        else -> showPermissionDenied()
    }
}

// Key behavior: when partial access is granted, MediaStore queries
// only return the user-selected photos. Your app sees a filtered view.
// The user can re-select photos at any time via the system photo picker.
Staff-Level Insight — Partial Access Changes Your Architecture

With selected photos access, your app can no longer assume it has a complete view of the user's media library. This affects: (1) Photo gallery apps that show "all photos" — must handle partial views gracefully. (2) Backup apps — can't guarantee full backup without full access. (3) Caching strategies — the set of accessible photos can change at any time. For L6: the system implements this via a URI grant mechanism. The photo picker grants persistent read URIs for selected photos. MediaStore queries are filtered by the system to only return granted URIs. This is transparent to the app — the same ContentResolver.query() call works, but returns fewer results.

7. Auto-Revoke (Permission Hibernation)

// Android 11+: permissions for unused apps are automatically revoked.
// "Unused" = app not launched for several months (varies by OEM, typically 3 months).

// What gets revoked:
// - All runtime (dangerous) permissions
// - The app's notification channels are also cleared
// - The app may be "hibernated" (Android 12+): cached data cleared too

// Checking if auto-revoke is enabled:
val packageManager = context.packageManager
val status = packageManager.isAutoRevokeWhitelisted // true = exempt

// Requesting exemption (user must manually toggle):
val intent = Intent(
    Settings.ACTION_APPLICATION_DETAILS_SETTINGS,
    Uri.fromParts("scheme", packageName, null)
)
// User navigates to: Settings → App → Permissions → "Remove permissions if app isn't used"

// Apps that should request exemption:
// - Device management apps
// - Family safety apps
// - Backup apps that run on schedule
// - Apps that use permissions for background work (e.g., health tracking)

8. Custom Permissions & Signature-Level Protection

// Declaring a custom permission (in the provider app):
// AndroidManifest.xml:
// <permission
//     android:name="com.example.provider.ACCESS_DATA"
//     android:protectionLevel="signature"
//     android:label="Access provider data"
//     android:description="@string/perm_desc" />
//
// <provider
//     android:name=".DataProvider"
//     android:authorities="com.example.provider.data"
//     android:readPermission="com.example.provider.ACCESS_DATA"
//     android:exported="true" />

// In the client app (must be signed with same certificate):
// <uses-permission android:name="com.example.provider.ACCESS_DATA" />

// Signature permissions are enforced by PackageManagerService:
// At install time, PMS checks if the requesting app's signing certificate
// matches the certificate of the app that declared the permission.
// If not, the permission is silently not granted.
Custom Permission Race Condition (CVE-2017-13156)

If two apps declare the same custom permission with different protection levels, the first-installed app's definition wins. An attacker could install their app first with protectionLevel="normal", then your app's signature-level permission is downgraded. Mitigation: always declare custom permissions in the same app that enforces them, and use your package name as a prefix. Android 12+ mitigates this by preventing permission redefinition across packages unless they share a signing certificate.

9. Permission Best Practices for L6

  • Minimize permissions — use Photo Picker instead of READ_MEDIA_IMAGES, use MediaStore for media creation (no permission needed to create), use FileProvider for sharing.
  • Request in context — ask for camera permission when user taps the camera button, not at launch.
  • Handle all states — granted, denied, permanently denied, revoked, partial access.
  • Use Activity Result API — not the deprecated onRequestPermissionsResult().
  • Test permission edge cases — revoke permissions while app is in background, test auto-revoke, test "Don't ask again".
  • Declare maxSdkVersion — for permissions that are no longer needed on newer APIs (e.g., WRITE_EXTERNAL_STORAGE with maxSdkVersion="28").

10. AppOps Internals — The Hidden Enforcement Layer

// AppOps operates on "op codes" that map to (but are distinct from) permissions:
// OP_CAMERA (26)           ↔ CAMERA permission
// OP_FINE_LOCATION (1)     ↔ ACCESS_FINE_LOCATION
// OP_COARSE_LOCATION (0)   ↔ ACCESS_COARSE_LOCATION
// OP_RECORD_AUDIO (27)     ↔ RECORD_AUDIO
// OP_READ_CONTACTS (4)     ↔ READ_CONTACTS

// Three key AppOps methods:
// 1. checkOp() — check if the op is allowed (no side effects)
// 2. noteOp() — check AND record that the op was used (triggers privacy indicator)
// 3. startOp() / finishOp() — for long-running ops (camera, mic) — shows green dot

// How "While using the app" location works internally:
// Permission: ACCESS_FINE_LOCATION = GRANTED
// AppOps mode for OP_FINE_LOCATION:
//   MODE_FOREGROUND — allowed only when app has a visible Activity or FGS
//   When app goes to background: AppOps silently denies the op
//   The permission is still "granted" — only AppOps enforcement changes

// AppOps modes:
// MODE_ALLOWED    — always allowed
// MODE_IGNORED    — silently denied (returns empty/default data)
// MODE_ERRORED    — denied with SecurityException
// MODE_FOREGROUND — allowed only in foreground (Android 10+)

// Privacy indicators (green dot) are powered by AppOps:
// When camera/mic startOp() is called, AppOps notifies the system UI
// SystemUI shows the green dot indicator
// When finishOp() is called, the indicator is removed

11. Binder-Level Permission Enforcement

// Permissions are enforced at Binder IPC boundaries.
// When app A calls a system service (e.g., LocationManager):

// 1. The call crosses a Binder transaction
// 2. The system service can check the caller's identity:
val callingUid = Binder.getCallingUid()   // UID of the calling app
val callingPid = Binder.getCallingPid()   // PID of the calling process

// 3. Enforce permission:
context.enforceCallingPermission(
    Manifest.permission.ACCESS_FINE_LOCATION,
    "Location access requires permission"
)
// Throws SecurityException if caller doesn't have the permission

// 4. enforceCallingOrSelfPermission — also allows the service itself:
context.enforceCallingOrSelfPermission(
    Manifest.permission.ACCESS_FINE_LOCATION,
    "Location access requires permission"
)

// CRITICAL: clearCallingIdentity / restoreCallingIdentity
// When a system service needs to call another service on behalf of the caller:
val token = Binder.clearCallingIdentity()
try {
    // Now getCallingUid() returns the SERVICE's UID, not the caller's
    // This allows the service to access resources with its own permissions
    performPrivilegedOperation()
} finally {
    Binder.restoreCallingIdentity(token)
    // Caller identity is restored for subsequent checks
}
Staff-Level Insight — clearCallingIdentity Is a Common Vulnerability

Forgetting to call restoreCallingIdentity() after clearCallingIdentity() is a privilege escalation vulnerability. If the identity isn't restored, subsequent permission checks use the system service's UID (which has elevated permissions) instead of the caller's. This has been the root cause of multiple Android CVEs. Always use try/finally to ensure restoration. In interviews, this demonstrates understanding of Binder's security model at the framework level.
