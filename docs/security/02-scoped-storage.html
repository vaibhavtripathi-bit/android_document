Scoped Storage

How Android 10+ restricts file access — FUSE daemon internals, MediaStore API, Storage Access Framework, FileProvider, content URIs, and migration strategies

1. The Problem Scoped Storage Solves

Before Android 10, any app with READ_EXTERNAL_STORAGE could read every file on shared storage — photos, downloads, documents from other apps. This was a massive privacy and security hole. Scoped storage restricts each app to its own sandboxed directory, with controlled access to shared media via MediaStore.

2. FUSE Daemon — How Scoped Storage Is Enforced

// Android 11+ uses a FUSE (Filesystem in Userspace) daemon to enforce
// scoped storage at the filesystem level:

// Before FUSE (Android 10):
// - Scoped storage was enforced only at the MediaStore/ContentResolver level
// - Direct file paths (File, FileInputStream) could bypass restrictions
// - Apps with requestLegacyExternalStorage=true bypassed everything

// With FUSE (Android 11+):
// 1. App opens a file path: File("/sdcard/DCIM/photo.jpg")
// 2. The kernel routes the open() syscall to the FUSE daemon
//    (MediaProvider process, running as system_server)
// 3. FUSE daemon checks:
//    a. Is this the app's own file? → Allow
//    b. Does the app have the relevant media permission? → Allow
//    c. Was this file granted via MediaStore URI? → Allow
//    d. None of the above → EACCES (Permission Denied)
// 4. If allowed, FUSE daemon opens the real file and returns an fd

// This means even native code (JNI, NDK) respects scoped storage.
// No more bypassing via direct file paths.
Staff-Level Insight — FUSE Performance Implications

The FUSE daemon adds a context switch for every file operation on shared storage: app → kernel → FUSE daemon (userspace) → kernel → actual filesystem. This adds ~10-30% overhead for sequential reads and significantly more for random I/O. For media-heavy apps (photo editors, video players), this is measurable. Mitigations: (1) Use ContentResolver.openFileDescriptor() which returns a direct fd after the initial check — subsequent reads bypass FUSE. (2) Copy files to app-specific storage for intensive processing. (3) Use MediaStore.setRequireOriginal() for apps that need unmodified originals. Google has been optimizing the FUSE daemon in each release — Android 12+ added read-ahead caching and reduced context switches.

3. MediaStore API

// Creating a media file (no permission needed for your own files):
val contentValues = ContentValues().apply {
    put(MediaStore.Images.Media.DISPLAY_NAME, "photo_${System.currentTimeMillis()}.jpg")
    put(MediaStore.Images.Media.MIME_TYPE, "image/jpeg")
    put(MediaStore.Images.Media.RELATIVE_PATH, "DCIM/MyApp")
    put(MediaStore.Images.Media.IS_PENDING, 1) // Hide until write is complete
}

val uri = contentResolver.insert(
    MediaStore.Images.Media.EXTERNAL_CONTENT_URI,
    contentValues
) ?: return

contentResolver.openOutputStream(uri)?.use { outputStream ->
    bitmap.compress(Bitmap.CompressFormat.JPEG, 90, outputStream)
}

// Mark as complete — now visible to other apps and MediaStore queries
contentValues.clear()
contentValues.put(MediaStore.Images.Media.IS_PENDING, 0)
contentResolver.update(uri, contentValues, null, null)

// Querying other apps' media (requires READ_MEDIA_IMAGES on Android 13+):
val projection = arrayOf(
    MediaStore.Images.Media._ID,
    MediaStore.Images.Media.DISPLAY_NAME,
    MediaStore.Images.Media.SIZE,
    MediaStore.Images.Media.DATE_ADDED
)

val cursor = contentResolver.query(
    MediaStore.Images.Media.EXTERNAL_CONTENT_URI,
    projection,
    "${MediaStore.Images.Media.SIZE} > ?",
    arrayOf("1000000"), // > 1MB
    "${MediaStore.Images.Media.DATE_ADDED} DESC"
)

Modifying/Deleting Other Apps' Media

// Android 10+: modifying or deleting another app's media requires user consent.
// The system throws RecoverableSecurityException:

try {
    contentResolver.delete(otherAppMediaUri, null, null)
} catch (e: RecoverableSecurityException) {
    // Android 10: launch the recovery intent
    val intentSender = e.userAction.actionIntent.intentSender
    startIntentSenderForResult(intentSender, REQUEST_CODE, null, 0, 0, 0)
}

// Android 11+: batch operations with createDeleteRequest/createWriteRequest:
val urisToDelete = listOf(uri1, uri2, uri3)
val pendingIntent = MediaStore.createDeleteRequest(contentResolver, urisToDelete)
startIntentSenderForResult(pendingIntent.intentSender, REQUEST_CODE, null, 0, 0, 0)

4. Storage Access Framework (SAF)

// SAF lets users pick files/directories via a system UI.
// The app receives a content URI with persistent access.

// Pick a single document:
val openDocumentLauncher = registerForActivityResult(
    ActivityResultContracts.OpenDocument()
) { uri: Uri? ->
    uri?.let { processDocument(it) }
}
openDocumentLauncher.launch(arrayOf("application/pdf", "text/*"))

// Pick a directory (Android 5+):
val openTreeLauncher = registerForActivityResult(
    ActivityResultContracts.OpenDocumentTree()
) { uri: Uri? ->
    uri?.let { treeUri ->
        // Persist access across reboots:
        contentResolver.takePersistableUriPermission(
            treeUri,
            Intent.FLAG_GRANT_READ_URI_PERMISSION or
                Intent.FLAG_GRANT_WRITE_URI_PERMISSION
        )
        // Enumerate files in the directory:
        val childrenUri = DocumentsContract.buildChildDocumentsUriUsingTree(
            treeUri,
            DocumentsContract.getTreeDocumentId(treeUri)
        )
        contentResolver.query(childrenUri, null, null, null, null)?.use { cursor ->
            while (cursor.moveToNext()) {
                val name = cursor.getString(
                    cursor.getColumnIndex(DocumentsContract.Document.COLUMN_DISPLAY_NAME)
                )
            }
        }
    }
}
SAF Restrictions (Android 11+)

Android 11 restricts SAF: apps can no longer use ACTION_OPEN_DOCUMENT_TREE to access the root of internal storage, the Downloads directory, or the Android/data and Android/obb directories. These restrictions prevent apps from gaining broad access to other apps' data via the directory picker. File manager apps that need broad access must use MANAGE_EXTERNAL_STORAGE (requires Play Store review).

5. FileProvider — Secure Cross-App File Sharing

// FileProvider converts a file:// URI to a content:// URI
// with temporary read/write permissions.

// 1. Declare in AndroidManifest.xml:
// <provider
//     android:name="androidx.core.content.FileProvider"
//     android:authorities="${applicationId}.fileprovider"
//     android:exported="false"
//     android:grantUriPermissions="true">
//     <meta-data
//         android:name="android.support.FILE_PROVIDER_PATHS"
//         android:resource="@xml/file_paths" />
// </provider>

// 2. Define accessible paths in res/xml/file_paths.xml:
// <paths>
//     <files-path name="internal" path="shared/" />
//     <cache-path name="cache" path="images/" />
//     <external-files-path name="external" path="exports/" />
// </paths>

// 3. Share a file:
val file = File(context.filesDir, "shared/report.pdf")
val uri = FileProvider.getUriForFile(context, "${packageName}.fileprovider", file)

val shareIntent = Intent(Intent.ACTION_SEND).apply {
    type = "application/pdf"
    putExtra(Intent.EXTRA_STREAM, uri)
    addFlags(Intent.FLAG_GRANT_READ_URI_PERMISSION)
}
startActivity(Intent.createChooser(shareIntent, "Share report"))
Staff-Level Insight — Why file:// URIs Are Banned

Since Android 7 (API 24), passing a file:// URI to another app via an Intent throws FileUriExposedException. The reason: file:// URIs expose your app's internal file path, and the receiving app may not have filesystem access to that path (especially with scoped storage). content:// URIs via FileProvider are secure because: (1) the path is opaque — the receiver doesn't know the real file location, (2) access is temporary — revoked when the receiving Activity finishes, (3) permissions are explicit — you control read vs write. For L6: understand that FileProvider uses a ContentProvider internally. When the receiving app reads the content URI, it makes a Binder IPC call to your app's FileProvider, which opens the file and returns a ParcelFileDescriptor.

8. sdcardfs vs FUSE — Storage Enforcement History

// Android's storage enforcement has evolved through three eras:

// Era 1: sdcardfs (Android 7-10)
// - Kernel module that emulates FAT filesystem permissions on ext4
// - Mapped UIDs to provide per-app views of /sdcard
// - Fast (kernel-level), but inflexible and hard to maintain
// - Could not enforce scoped storage granularly

// Era 2: FUSE (Android 11+)
// - Userspace daemon (MediaProvider process) intercepts all file ops
// - Full control: can check ownership, permissions, MediaStore grants
// - Slower (~10-30% overhead) due to kernel ↔ userspace context switches
// - Enables scoped storage enforcement even for native code (JNI/NDK)

// Era 3: FUSE + optimizations (Android 12+)
// - Read-ahead caching reduces FUSE overhead
// - Passthrough mode for app's own files (bypasses FUSE after initial check)
// - ContentResolver.openFileDescriptor() returns direct fd after check

// requestLegacyExternalStorage behavior:
// Android 10 (API 29): Setting to true opts out of scoped storage entirely
//   → App gets full legacy access to /sdcard
// Android 11 (API 30): IGNORED — all apps use scoped storage regardless
//   → Apps that relied on this flag break on upgrade to Android 11
// Migration: replace File paths with MediaStore/SAF before targeting API 30
Staff-Level Insight — Version-Specific Permission Branching

A production app must handle three different permission models for media access depending on the Android version. This is a common L6 interview question — "how do you handle storage permissions across API levels?"

// Version-specific media permission handling:
fun requestMediaPermissions() {
    val permissions = when {
        Build.VERSION.SDK_INT >= 34 -> {
            // Android 14+: prefer Photo Picker, or use selected photos
            arrayOf(
                Manifest.permission.READ_MEDIA_IMAGES,
                Manifest.permission.READ_MEDIA_VISUAL_USER_SELECTED
            )
        }
        Build.VERSION.SDK_INT >= 33 -> {
            // Android 13: granular media permissions
            arrayOf(Manifest.permission.READ_MEDIA_IMAGES)
        }
        Build.VERSION.SDK_INT >= 29 -> {
            // Android 10-12: legacy permission (scoped storage enforced by FUSE)
            arrayOf(Manifest.permission.READ_EXTERNAL_STORAGE)
        }
        else -> {
            // Android 9 and below: legacy
            arrayOf(
                Manifest.permission.READ_EXTERNAL_STORAGE,
                Manifest.permission.WRITE_EXTERNAL_STORAGE
            )
        }
    }
    permissionLauncher.launch(permissions)
}
