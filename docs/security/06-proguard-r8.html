ProGuard / R8 — Code Optimization

Code shrinking, obfuscation, optimization — R8 vs ProGuard, keep rules, full mode vs compat mode, Kotlin metadata, desugaring, debugging obfuscated crashes

1. R8 vs ProGuard

AspectProGuardR8 (Default since AGP 3.4)
MaintainerGuardSquareGoogle (part of D8/R8 toolchain)
InputJava bytecode (.class)Java bytecode (.class) → DEX directly
PipelineProGuard → D8 (dexing) — two stepsSingle step: shrink + optimize + obfuscate + dex
SpeedSlower (two-pass)Faster (single-pass, whole-program analysis)

// build.gradle.kts — enabling R8:
android {
    buildTypes {
        release {
            isMinifyEnabled = true   // Enable shrinking + obfuscation
            isShrinkResources = true // Remove unused resources
            proguardFiles(
                getDefaultProguardFile("proguard-android-optimize.txt"),
                "proguard-rules.pro"
            )
        }
    }
}

2. What R8 Does — Three Phases

Phase 1: Shrinking (Tree Shaking)

// R8 traces from entry points (Activities, Services, etc.) and removes
// all code that is not reachable. This can remove 30-70% of code.

// Entry points are determined by:
// 1. AndroidManifest.xml components (Activities, Services, Receivers, Providers)
// 2. Keep rules in proguard-rules.pro
// 3. AAPT-generated keep rules (layouts referencing views by class name)
// 4. Reflection-based access (must be manually kept)

// Resource shrinking removes unused drawables, layouts, strings, etc.
// It works by analyzing code references to R.* constants.

Phase 2: Obfuscation (Name Minification)

// Renames classes, methods, and fields to short names:
// com.example.app.UserRepository → a.b.c
// getUserProfile() → a()
// firstName → a

// Benefits:
// 1. Smaller DEX (shorter strings in constant pool)
// 2. Harder to reverse-engineer
// 3. Typically saves 5-15% APK size

// The mapping is stored in mapping.txt:
// com.example.app.UserRepository -> a.b.c:
//     java.lang.String getUserProfile() -> a
//     java.lang.String firstName -> a

Phase 3: Optimization

// R8 performs whole-program optimizations:

// 1. Method inlining — small methods are inlined at call sites
//    Reduces method count and call overhead

// 2. Class merging — classes with single implementation merged
//    interface Foo { fun bar() }
//    class FooImpl : Foo { override fun bar() { ... } }
//    → FooImpl is merged, Foo interface removed

// 3. Enum unboxing — enums converted to ints where possible
//    enum class State { IDLE, LOADING, ERROR }
//    → Replaced with int constants (0, 1, 2)

// 4. Constant propagation — compile-time constants folded
//    val DEBUG = false
//    if (DEBUG) { log("debug") } // Entire block removed

// 5. Dead code elimination — unreachable branches removed

// 6. Devirtualization — virtual calls converted to direct calls
//    when the concrete type is known
Staff-Level Insight — R8 Full Mode vs Compat Mode

R8 has two modes: compat mode (default, ProGuard-compatible) and full mode. Full mode enables aggressive optimizations that break ProGuard assumptions: (1) -keepclassmembers doesn't prevent class from being removed if unused. (2) Default constructors aren't kept unless explicitly used. (3) -keepattributes is more aggressive. Enable full mode in gradle.properties: android.enableR8.fullMode=true. Full mode typically removes 10-20% more code but requires more careful keep rules. Most modern libraries (Retrofit, Moshi, Room) ship R8-compatible rules. For L6: full mode is the recommended production configuration for mature apps — it produces smaller APKs and faster startup.

3. Keep Rules — The Critical Part

// Keep rules tell R8 what NOT to remove or rename.

// Keep a class and all its members:
-keep class com.example.app.model.User { *; }

// Keep class name but allow member shrinking:
-keepnames class com.example.app.model.User

// Keep members but allow class renaming:
-keepclassmembers class com.example.app.model.User { *; }

// Keep classes implementing an interface:
-keep class * implements com.example.app.Plugin { *; }

// Keep annotated classes (e.g., Gson/Moshi models):
-keep @com.google.gson.annotations.SerializedName class * { *; }

// Keep methods with specific annotations:
-keepclassmembers class * {
    @com.google.gson.annotations.SerializedName <fields>;
}

// Keep enums (often needed for serialization):
-keepclassmembers enum * {
    public static **[] values();
    public static ** valueOf(java.lang.String);
}

// Keep native methods:
-keepclasseswithmembernames class * {
    native <methods>;
}

4. Common Pitfalls & Fixes

ProblemCauseFix
Crash: ClassNotFoundException R8 removed a class accessed via reflection -keep class com.example.ReflectedClass { *; }
Crash: NoSuchMethodException R8 removed/renamed a method used by reflection or JNI Keep the specific method
JSON parsing fails Field names obfuscated — Gson/Moshi can't match JSON keys Use @SerializedName or keep model classes
Retrofit 404 errors Interface methods renamed Retrofit ships its own R8 rules — ensure they're applied
Enum crash Enum values() or valueOf() removed Keep enum members (see rule above)
Kotlin data class copy() fails R8 removes unused copy() / componentN() Keep if used via reflection; otherwise safe to remove
Kotlin Metadata & R8

Kotlin stores metadata in @Metadata annotations on classes. Libraries like Moshi-kotlin and kotlinx.serialization read this metadata via reflection to understand data class structure. If R8 strips @Metadata, these libraries break silently (wrong field mapping) or crash. Keep Kotlin metadata for serialized classes: -keep class kotlin.Metadata { *; } or use the library's shipped R8 rules. In R8 full mode, metadata is stripped more aggressively — test serialization thoroughly.

5. Debugging Obfuscated Crashes

// R8 generates mapping.txt during build:
// app/build/outputs/mapping/release/mapping.txt

// Obfuscated stack trace:
// java.lang.NullPointerException
//     at a.b.c.a(Unknown Source:12)
//     at a.b.d.b(Unknown Source:45)
//     at a.a.a.onCreate(Unknown Source:8)

// Deobfuscate with ReTrace (bundled with R8):
// retrace mapping.txt stacktrace.txt

// Or use the Gradle task:
// ./gradlew retraceRelease

// Upload mapping.txt to:
// - Google Play Console (automatic deobfuscation in crash reports)
// - Firebase Crashlytics (automatic deobfuscation)
// - Sentry, Bugsnag, etc.

// CRITICAL: archive mapping.txt for every release build.
// Without it, crash reports are unreadable.
// CI/CD should automatically upload mapping.txt to crash reporting service.
// Preserving line numbers (recommended for production):
// In proguard-rules.pro:
-keepattributes SourceFile,LineNumberTable

// This keeps file names and line numbers in stack traces
// while still obfuscating class/method names.
// Minimal size impact (~2-5% larger DEX) but dramatically
// improves crash debugging.

// To also rename the source file attribute (extra obfuscation):
-renamesourcefileattribute SourceFile
// Stack traces will show "SourceFile" instead of the real filename
// but line numbers are preserved — mapping.txt resolves the rest.

6. Desugaring — Java 8+ on Older Devices

// R8 includes desugaring: converting Java 8+ features to bytecode
// that runs on older Android versions.

// Desugared features:
// - Lambda expressions → anonymous classes
// - Method references → synthetic methods
// - Default interface methods → companion static methods
// - Try-with-resources → explicit try/finally
// - Stream API (via core library desugaring)

// Core library desugaring (Java 8+ APIs on API 21+):
// build.gradle.kts:
android {
    compileOptions {
        isCoreLibraryDesugaringEnabled = true
        sourceCompatibility = JavaVersion.VERSION_11
        targetCompatibility = JavaVersion.VERSION_11
    }
}

dependencies {
    coreLibraryDesugaring("com.android.tools:desugar_jdk_libs:2.0.4")
}

// This enables: java.time.*, java.util.stream.*, java.util.Optional,
// CompletableFuture, and more on API 21+ devices.
Staff-Level Insight — R8 as a Security Tool

R8 is not just an optimization tool — it's a security layer. Obfuscation makes reverse engineering significantly harder: class/method names are meaningless, control flow is harder to follow, and string encryption (via third-party tools like DexGuard) can hide API keys and URLs. For L6: understand the limits — R8 obfuscation is not encryption. A determined attacker with tools like jadx, Frida, or Ghidra can still reverse-engineer the logic. R8 raises the bar but doesn't prevent reverse engineering. For sensitive logic (license checks, DRM), consider server-side enforcement. R8's real security value is removing unused code (reducing attack surface) and making automated analysis harder.
