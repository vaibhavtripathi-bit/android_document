2. OpenGL ES vs Vulkan

AspectOpenGL ESVulkan
API levelHigh-level, driver manages stateLow-level, app manages state explicitly
Driver overheadHigh (driver validates every call)Low (validation at pipeline creation)
ThreadingSingle-threaded (one GL context per thread)Multi-threaded command recording
Shader compilationAt draw time (can cause jank)At pipeline creation (pre-compiled)
Memory managementDriver-managedApp-managed (more control, more responsibility)
Android defaultAndroid 3.0 - 9.0Android 10+ (Skia uses Vulkan by default where available)
Staff-Level Insight — Vulkan's Impact on Android Rendering

Android 10+ uses Skia's Vulkan backend (called SkiaVulkan) by default on devices that support it. The key benefit: reduced driver overhead. OpenGL ES drivers do extensive validation on every draw call (checking state, validating shaders). Vulkan moves this validation to pipeline creation time — once a pipeline is created, draw calls are nearly zero-overhead. For UI rendering, this means the RenderThread spends less time in driver code and more time doing useful work. The result: lower CPU usage on the RenderThread, better battery life, and fewer dropped frames on complex UIs. In Systrace, you'll see shorter "flush commands" sections with Vulkan.

3. Shader Compilation Jank

// The #1 GPU-related jank cause: shader compilation
//
// When the GPU encounters a new combination of:
//   - Drawing operation (rect, circle, text, path)
//   - Blend mode (src_over, multiply, etc.)
//   - Color space
//   - Anti-aliasing mode
// ...it needs to compile a GPU shader program.
//
// Shader compilation is EXPENSIVE: 5-50ms per shader
// This causes visible jank on first render of new UI elements.

// OpenGL ES: shaders compiled at draw time → jank on first frame
// Vulkan: pipelines pre-compiled → jank at pipeline creation (can be deferred)

// Android's mitigations:
// 1. Shader cache: compiled shaders are cached on disk
//    Location: /data/user_de/0/com.example.app/code_cache/com.android.skia/
//    Survives app restarts, cleared on app update
//
// 2. Pre-warming: Android 12+ pre-compiles common shaders at install time
//
// 3. App-level mitigation: warm up shaders by drawing off-screen
//    val bitmap = Bitmap.createBitmap(1, 1, Bitmap.Config.ARGB_8888)
//    val canvas = Canvas(bitmap)
//    // Draw various shapes to trigger shader compilation
//    canvas.drawRoundRect(...) // Compiles rounded rect shader
//    canvas.drawCircle(...)    // Compiles circle shader
Shader Jank in Production

Shader compilation jank is most visible on: (1) first launch after install (empty shader cache), (2) after app update (cache cleared), (3) when navigating to a screen with novel drawing operations. In Systrace, look for long GrGLProgram or vkCreateGraphicsPipelines sections on the RenderThread. The Jetpack ProfileInstaller library helps by pre-compiling shaders during install, but it only covers Baseline Profiles (code paths), not shader compilation. For custom drawing, consider pre-warming shaders in a background thread during splash screen.

4. Texture Management & GPU Memory

// GPU textures are the primary memory consumer in rendering:

// Bitmap → GPU Texture:
// When a Bitmap is first drawn, HWUI uploads it to a GPU texture.
// The texture stays in GPU memory until evicted by the LRU cache.
// GPU texture cache size: typically 64-128MB (device-dependent)

// Text rendering:
// Glyphs are rasterized into a texture atlas (a large texture with all glyphs)
// Each glyph is rendered once, then reused via texture coordinates
// Atlas size: typically 2048x2048 (16MB for ARGB)
// If the atlas fills up, old glyphs are evicted → re-rasterized on next use

// Path rendering:
// Complex paths (rounded rects, custom shapes) are tessellated into triangles
// Tessellation result is cached in a path texture cache
// Cache miss → expensive tessellation → jank

// Monitoring GPU memory:
// adb shell dumpsys gfxinfo <package>
// Shows: total GPU memory, texture cache size, path cache, etc.

5. Overdraw

Overdraw occurs when the same pixel is drawn multiple times in a single frame. Each overdraw layer costs GPU fill rate:

// Overdraw levels (visible via Developer Options → Debug GPU Overdraw):
// No color:  0x overdraw (pixel drawn once) — ideal
// Blue:      1x overdraw (drawn twice) — acceptable
// Green:     2x overdraw (drawn three times) — concerning
// Pink:      3x overdraw (drawn four times) — bad
// Red:       4x+ overdraw — very bad

// Common causes:
// 1. Unnecessary backgrounds: Activity has a window background,
//    then a layout background, then a card background → 3x overdraw
//    Fix: theme.windowBackground = null, or remove redundant backgrounds

// 2. Overlapping views: multiple views stacked with opaque backgrounds
//    Fix: use clipToPadding, remove hidden view backgrounds

// 3. Complex custom drawing: drawing a background rect, then content on top
//    Fix: use clipRect() to skip drawing behind opaque content

// Removing window background:
// styles.xml:
// <item name="android:windowBackground">@null</item>
// Or in code:
window.setBackgroundDrawable(null)
Staff-Level Insight — Overdraw in Compose

Compose has less overdraw than the View system by default because it doesn't have the layered background problem (no XML backgrounds, no window background by default). However, Compose can still have overdraw from: (1) Box with overlapping children, (2) Modifier.background() on nested composables, (3) Surface composables that draw their own background. The same debugging tool works: Developer Options → Debug GPU Overdraw. The fix is the same: remove redundant backgrounds, use Modifier.drawBehind { } with clipRect() to skip drawing behind opaque content.
