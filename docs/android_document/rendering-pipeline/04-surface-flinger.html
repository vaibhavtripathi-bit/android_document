SurfaceFlinger & Layer Composition

The system compositor — BufferQueue producer/consumer, Surface and SurfaceControl, layer composition strategies, Hardware Composer (HWC), and BLAST BufferQueue

1. SurfaceFlinger's Role

SurfaceFlinger is Android's system-level compositor. It takes rendered buffers from all visible apps and system UI, composites them into a single image, and sends it to the display:

2. BufferQueue — The Producer/Consumer Pipeline

// BufferQueue is the core IPC mechanism between apps and SurfaceFlinger:
//
// Producer (App's RenderThread):
//   1. dequeueBuffer() — get an empty buffer from the queue
//   2. Render into the buffer (GPU commands)
//   3. queueBuffer() — return the filled buffer to the queue
//
// Consumer (SurfaceFlinger):
//   1. acquireBuffer() — get the latest filled buffer
//   2. Composite with other layers
//   3. releaseBuffer() — return the buffer to the pool
//
// Buffer states:
//   FREE → DEQUEUED → QUEUED → ACQUIRED → FREE
//
// With triple buffering, there are 3 buffers:
//   Buffer A: being displayed (ACQUIRED by SurfaceFlinger)
//   Buffer B: being rendered (DEQUEUED by app)
//   Buffer C: waiting in queue (QUEUED, ready for next frame)
Staff-Level Insight — BufferQueue and Memory

Each buffer in the BufferQueue is a GraphicBuffer — a GPU-accessible memory allocation (typically 4 bytes per pixel × screen resolution). For a 1080p display: 1920 × 1080 × 4 = ~8MB per buffer. With triple buffering: ~24MB per Surface. An app with multiple Surfaces (e.g., SurfaceView for video + main window) uses even more. This is why SurfaceView and TextureView are expensive — each creates its own BufferQueue. In interviews, this memory cost explains why you should minimize the number of Surfaces and prefer a single window Surface for most UI.

3. Surface, SurfaceControl, and Transactions

// Surface: the producer side of a BufferQueue
// - Each window (Activity) gets one Surface from WindowManager
// - SurfaceView creates an additional Surface (separate Z-order)
// - The app's RenderThread renders into the Surface's buffers

// SurfaceControl: metadata about a layer (position, size, Z-order, alpha)
// - Managed by WindowManager and SurfaceFlinger
// - Apps can create SurfaceControls for sub-layers (Android 10+)

// SurfaceControl.Transaction: atomic batch of layer changes
SurfaceControl.Transaction().apply {
    setPosition(surfaceControl, x, y)
    setBufferSize(surfaceControl, width, height)
    setAlpha(surfaceControl, 0.8f)
    setLayer(surfaceControl, zOrder)
    setCornerRadius(surfaceControl, 16f)
    apply() // All changes applied atomically on the next SF VSYNC
}

// BLAST (Android 12+):
// Before BLAST: app queues buffer, then sends a separate transaction for metadata
//   → Race condition: buffer and metadata could arrive at different times
// With BLAST: buffer and transaction are submitted together atomically
//   → No tearing, no mismatched metadata

4. Hardware Composer (HWC)

HWC is a HAL (Hardware Abstraction Layer) that composites layers using dedicated display hardware instead of the GPU:

Composition TypeDone ByCostWhen Used
HWC Overlay Display hardware Zero GPU cost Simple layers: solid color, video, no rotation/complex transforms
Client (GPU) Composition GPU via SurfaceFlinger GPU time + power Complex effects: rounded corners, blur, color matrix, many overlapping layers
Mixed Some HWC, some GPU Partial GPU cost Most common: simple layers via HWC, complex ones via GPU
Staff-Level Insight — HWC Limits and Debugging

Each SoC's HWC has limits: maximum number of overlay planes (typically 4-8), supported pixel formats, maximum layer size, supported transforms. When a layer exceeds these limits, it falls back to GPU composition. In Systrace/Perfetto, look for HWC::validate and HWC::present sections. If validate rejects layers (fallback to GPU), you'll see GPU composition sections. Common causes of GPU fallback: rounded corners on large layers, blur effects, too many overlapping transparent layers. For battery optimization, minimizing GPU composition is critical — HWC overlays are essentially free.

5. SurfaceView vs TextureView

AspectSurfaceViewTextureView
SurfaceSeparate Surface (own BufferQueue)Uses the window's Surface
Z-orderBehind or in front of the window (hole punched)Part of the View hierarchy (normal Z-order)
TransformsLimited (no rotation, scale, alpha on the Surface itself)Full View transforms (rotation, scale, alpha)
CompositionHWC overlay (free)GPU composition (costs GPU)
MemoryOwn BufferQueue (~24MB)Own BufferQueue + GPU texture copy
Use caseVideo playback, camera preview, gamesVideo with View transforms (rounded corners, animation)
ThreadingCan render from any threadMust render on UI thread (or use lockCanvas)
Staff-Level Insight — Prefer SurfaceView

Google's recommendation: always prefer SurfaceView over TextureView unless you specifically need View transforms. SurfaceView's separate Surface is composited by HWC (zero GPU cost), while TextureView requires GPU composition (copies the buffer to a GPU texture, then composites). For video playback, SurfaceView can use a dedicated HWC overlay plane — the video frames go directly from the decoder to the display without touching the GPU. TextureView forces every video frame through the GPU. The only valid reason for TextureView: you need to animate the video surface (rotation, scale) as part of the View hierarchy.
