Convention Plugins

Build variants, shared build logic, AGP extension API, custom tasks, and the convention plugin architecture used by Google's Now in Android and production-scale Android projects

1. The Problem: Duplicated Build Logic

In a modular Android project, every module's build.gradle.kts repeats the same boilerplate: compileSdk, minSdk, Kotlin compiler options, test runner setup, ProGuard configuration, KSP wiring. In a 50-module project, that's 50 copies of the same configuration — each slightly different because someone forgot to update one of them.

The Duplication Problem:

  :feature:home/build.gradle.kts     :feature:profile/build.gradle.kts
  ┌──────────────────────────────┐   ┌──────────────────────────────┐
  │ compileSdk = 35              │   │ compileSdk = 35              │
  │ minSdk = 26                  │   │ minSdk = 26                  │
  │ jvmTarget = "17"             │   │ jvmTarget = "17"             │
  │ testInstrumentationRunner =  │   │ testInstrumentationRunner =  │
  │   "...HiltTestRunner"        │   │   "...HiltTestRunner"        │
  │ consumerProguardFiles(...)   │   │ consumerProguardFiles(...)   │
  │ ksp { ... }                  │   │ ksp { ... }                  │
  └──────────────────────────────┘   └──────────────────────────────┘
           × 50 modules = maintenance nightmare

With Convention Plugins:

  :feature:home/build.gradle.kts     :feature:profile/build.gradle.kts
  ┌──────────────────────────────┐   ┌──────────────────────────────┐
  │ plugins {                    │   │ plugins {                    │
  │   id("myapp.android.feature")│   │   id("myapp.android.feature")│
  │ }                            │   │ }                            │
  │ // module-specific config    │   │ // module-specific config    │
  └──────────────────────────────┘   └──────────────────────────────┘
           All shared logic lives in ONE place

Three approaches have emerged to solve this:

Approach Mechanism Cache Impact Verdict
buildSrc Auto-compiled directory on build classpath Any change invalidates entire build Legacy — avoid for large projects
Convention plugins (composite build) Included build with plugin classes Only affected modules re-execute Recommended — Google's approach
Precompiled script plugins .gradle.kts files compiled as plugins Same as composite build Good for simple cases

2. buildSrc — The Legacy Approach

Gradle automatically compiles everything in the buildSrc/ directory at the project root and adds it to the build classpath. This lets you define version constants, custom tasks, and plugins in Kotlin that are available in every build.gradle.kts.

// buildSrc/src/main/kotlin/Dependencies.kt
object Versions {
    const val compileSdk = 35
    const val minSdk = 26
    const val targetSdk = 35
    const val kotlin = "2.0.21"
    const val compose = "1.7.5"
}

object Libs {
    const val coreKtx = "androidx.core:core-ktx:1.15.0"
    const val composeUi = "androidx.compose.ui:ui:${Versions.compose}"
}

// Usage in any module's build.gradle.kts:
android {
    compileSdk = Versions.compileSdk
}
dependencies {
    implementation(Libs.coreKtx)
}
The buildSrc Problem

Any change to buildSrc invalidates the entire build. Gradle considers buildSrc part of the build infrastructure. When its bytecode changes, every task in every module is considered out-of-date and re-executes. Change a version constant in Dependencies.kt → full rebuild of all 50 modules. On a large project, this turns a 30-second incremental build into a 5-minute full build.

Additionally, buildSrc is compiled before any project configuration begins. If it has compilation errors, you can't even run ./gradlew help.

L6 Insight — buildSrc Is an Anti-Pattern at Scale

buildSrc was the standard approach before Gradle 6.x introduced composite builds. It's still fine for small projects (<10 modules), but for anything at Google scale, it's an anti-pattern. The full-invalidation behavior means that a one-line version bump triggers a complete rebuild — destroying the value of incremental compilation, build cache, and configuration cache. The migration path is straightforward: move buildSrc/ contents into a build-logic/ included build. The effort is a few hours; the payoff is permanent.

3. Convention Plugins — The Modern Approach

A convention plugin is a Gradle plugin that encapsulates shared build configuration. Instead of each module configuring itself, it applies a convention plugin that does the configuration. The plugins live in a build-logic/ directory that is included as a composite build.

Directory Structure

project-root/
├── build-logic/
│   ├── settings.gradle.kts          ← declares repos for plugin dependencies
│   └── convention/
│       ├── build.gradle.kts         ← the plugin project
│       └── src/main/kotlin/
│           ├── AndroidLibraryConventionPlugin.kt
│           ├── AndroidApplicationConventionPlugin.kt
│           ├── KotlinLibraryConventionPlugin.kt
│           └── AndroidFeatureConventionPlugin.kt
├── settings.gradle.kts              ← includes build-logic
├── app/
│   └── build.gradle.kts             ← plugins { id("myapp.android.application") }
├── feature/
│   └── home/
│       └── build.gradle.kts         ← plugins { id("myapp.android.feature") }
└── core/
    └── domain/
        └── build.gradle.kts         ← plugins { id("myapp.kotlin.library") }

Root settings.gradle.kts

// settings.gradle.kts (project root)
pluginManagement {
    includeBuild("build-logic")  // ← this is the key line
    repositories {
        google()
        mavenCentral()
        gradlePluginPortal()
    }
}

dependencyResolutionManagement {
    repositoriesMode = RepositoriesMode.FAIL_ON_PROJECT_REPOS
    repositories {
        google()
        mavenCentral()
    }
}

include(":app")
include(":feature:home")
include(":feature:profile")
include(":core:domain")
include(":core:data")

build-logic/settings.gradle.kts

// build-logic/settings.gradle.kts
dependencyResolutionManagement {
    repositories {
        google()
        mavenCentral()
        gradlePluginPortal()
    }
    versionCatalogs {
        create("libs") {
            from(files("../gradle/libs.versions.toml"))
        }
    }
}

build-logic/convention/build.gradle.kts

// build-logic/convention/build.gradle.kts
plugins {
    `kotlin-dsl`
}

dependencies {
    compileOnly(libs.android.gradlePlugin)
    compileOnly(libs.kotlin.gradlePlugin)
    compileOnly(libs.ksp.gradlePlugin)
    compileOnly(libs.compose.gradlePlugin)
}

gradlePlugin {
    plugins {
        register("androidApplication") {
            id = "myapp.android.application"
            implementationClass = "AndroidApplicationConventionPlugin"
        }
        register("androidLibrary") {
            id = "myapp.android.library"
            implementationClass = "AndroidLibraryConventionPlugin"
        }
        register("androidFeature") {
            id = "myapp.android.feature"
            implementationClass = "AndroidFeatureConventionPlugin"
        }
        register("kotlinLibrary") {
            id = "myapp.kotlin.library"
            implementationClass = "KotlinLibraryConventionPlugin"
        }
    }
}

The compileOnly dependencies give the plugin code access to AGP, Kotlin, and KSP APIs without adding them to the runtime classpath of consuming modules. Each module applies the actual plugins through the convention plugin's apply method.

Why Composite Build > buildSrc

When you change a convention plugin, Gradle recompiles only build-logic/convention and re-configures only the modules that apply the changed plugin. Other modules remain UP-TO-DATE. With buildSrc, the same change would invalidate every task in every module.

4. Implementing Convention Plugins — Step by Step

Android Library Convention Plugin

// build-logic/convention/src/main/kotlin/AndroidLibraryConventionPlugin.kt
import com.android.build.gradle.LibraryExtension
import org.gradle.api.Plugin
import org.gradle.api.Project
import org.gradle.kotlin.dsl.configure
import org.gradle.kotlin.dsl.dependencies

class AndroidLibraryConventionPlugin : Plugin<Project> {
    override fun apply(target: Project) {
        with(target) {
            with(pluginManager) {
                apply("com.android.library")
                apply("org.jetbrains.kotlin.android")
            }

            extensions.configure<LibraryExtension> {
                compileSdk = 35

                defaultConfig {
                    minSdk = 26
                    testInstrumentationRunner = "androidx.test.runner.AndroidJUnitRunner"
                    consumerProguardFiles("consumer-rules.pro")
                }

                compileOptions {
                    sourceCompatibility = JavaVersion.VERSION_17
                    targetCompatibility = JavaVersion.VERSION_17
                }

                buildFeatures {
                    buildConfig = false
                }
            }

            configureKotlinOptions()

            dependencies {
                add("testImplementation", libs.findLibrary("junit").get())
                add("testImplementation", libs.findLibrary("truth").get())
            }
        }
    }
}

Android Application Convention Plugin

import com.android.build.api.dsl.ApplicationExtension
import org.gradle.api.Plugin
import org.gradle.api.Project

            extensions.configure<ApplicationExtension> {

                    minSdk = 26

                compileOptions {
                    sourceCompatibility = JavaVersion.VERSION_17
                    targetCompatibility = JavaVersion.VERSION_17

                            getDefaultProguardFile("proguard-android-optimize.txt"),

                buildFeatures {
                    buildConfig = true
                }

        }
    }

5. Precompiled Script Plugins

An alternative to writing Plugin<Project> classes: place .gradle.kts files in build-logic/convention/src/main/kotlin/. Gradle compiles them into plugin classes automatically. The file name becomes the plugin ID.

// build-logic/convention/src/main/kotlin/myapp.android.library.gradle.kts
plugins {
    id("com.android.library")
    id("org.jetbrains.kotlin.android")
}

android {
    compileSdk = 35

    defaultConfig {
        minSdk = 26
        testInstrumentationRunner = "androidx.test.runner.AndroidJUnitRunner"
        consumerProguardFiles("consumer-rules.pro")
    }

    compileOptions {
        sourceCompatibility = JavaVersion.VERSION_17
        targetCompatibility = JavaVersion.VERSION_17
    }
}

// Applied in modules as:
// plugins { id("myapp.android.library") }

Script Plugins vs Class Plugins

Aspect Precompiled Script Plugin Plugin<Project> Class
Syntax Direct build.gradle.kts style Kotlin class with apply() method
Testability Hard to unit test Testable with GradleRunner
Flexibility Limited — no constructor params, no inheritance Full Kotlin: conditionals, composition, reuse
IDE Support Weaker — some auto-complete issues Full Kotlin IDE support
Registration Automatic (file name = plugin ID) Manual in gradlePlugin { } block
Best For Simple, static configuration Complex logic, composition, testing
When to Use Which

Start with precompiled script plugins for simplicity. Migrate to class plugins when you need conditional logic (e.g., "apply Compose only if the module opts in"), composition (one plugin applying another), or testability. Most production projects end up with class plugins because the build logic inevitably grows beyond simple configuration.

6. Version Catalog Integration

Convention plugins should read versions and dependency coordinates from libs.versions.toml rather than hardcoding them. This keeps version management centralized while plugins handle structural configuration.

// build-logic/convention/src/main/kotlin/KotlinOptions.kt
internal val Project.libs
    get() = extensions.getByType(
        org.gradle.api.artifacts.VersionCatalogsExtension::class.java
    ).named("libs")

// Usage in a convention plugin:
class AndroidLibraryConventionPlugin : Plugin<Project> {
    override fun apply(target: Project) {
        with(target) {
            extensions.configure<LibraryExtension> {
                val catalog = libs
                compileSdk = catalog.findVersion("compileSdk")
                    .get().toString().toInt()

                defaultConfig {
                    minSdk = catalog.findVersion("minSdk")
                        .get().toString().toInt()
                }
            }

            dependencies {
                add("implementation",
                    libs.findLibrary("androidx-core-ktx").get())
                add("testImplementation",
                    libs.findLibrary("junit").get())
                add("androidTestImplementation",
                    libs.findLibrary("androidx-test-ext-junit").get())
            }
        }
    }
}

Sharing the Catalog with build-logic

The version catalog is defined in the main project's gradle/libs.versions.toml. To access it from build-logic, re-declare it in build-logic/settings.gradle.kts using from(files("../gradle/libs.versions.toml")). This ensures both the main project and the convention plugins use the same versions.

7. AGP Extension API — Variant Manipulation

The Android Gradle Plugin exposes a powerful extension API through androidComponents { }. This is the type-safe, stable API for manipulating build variants — far superior to the deprecated variantFilter approach.

beforeVariants — Modify or Disable Variants

// Disable variants you never build to reduce task count
import com.android.build.api.variant.ApplicationAndroidComponentsExtension

class AndroidApplicationConventionPlugin : Plugin<Project> {
    override fun apply(target: Project) {
        with(target) {
            // ... standard configuration ...

            extensions.configure<ApplicationAndroidComponentsExtension> {
                beforeVariants(selector().all()) { variantBuilder ->
                    // Disable all "staging + release" combinations
                    if (variantBuilder.buildType == "release" &&
                        variantBuilder.productFlavors.any { it.second == "staging" }
                    ) {
                        variantBuilder.enable = false
                    }
                }
            }
        }
    }
}

onVariants — Configure Finalized Variants

import com.android.build.api.variant.ApplicationAndroidComponentsExtension
import com.android.build.api.variant.BuildConfigField

extensions.configure<ApplicationAndroidComponentsExtension> {
    onVariants { variant ->
        // Add BuildConfig fields per variant
        variant.buildConfigFields.put(
            "API_BASE_URL",
            BuildConfigField(
                type = "String",
                value = when {
                    variant.name.contains("staging") ->
                        "\"https://staging-api.myapp.com\""
                    else ->
                        "\"https://api.myapp.com\""
                },
                comment = "Base URL for API calls"
            )
        )

        // Add a git commit hash to every variant
        variant.buildConfigFields.put(
            "GIT_HASH",
            BuildConfigField(
                type = "String",
                value = "\"${getGitHash()}\"",
                comment = "Git commit hash at build time"
            )
        )
    }
}

fun getGitHash(): String =
    Runtime.getRuntime()
        .exec("git rev-parse --short HEAD")
        .inputStream.bufferedReader().readText().trim()

Artifact Manipulation

import com.android.build.api.artifact.SingleArtifact
import com.android.build.api.variant.ApplicationAndroidComponentsExtension

extensions.configure<ApplicationAndroidComponentsExtension> {
    onVariants { variant ->
        // Wire a custom task to transform the merged manifest
        val manifestUpdater = tasks.register<ManifestVersionTask>(
            "${variant.name}UpdateManifest"
        ) {
            gitHash.set(provider { getGitHash() })
        }

        variant.artifacts
            .use(manifestUpdater)
            .wiredWithFiles(
                ManifestVersionTask::mergedManifest,
                ManifestVersionTask::updatedManifest
            )
            .toTransform(SingleArtifact.MERGED_MANIFEST)
    }
}
L6 Insight — beforeVariants Is a Build Performance Lever

Every enabled variant causes AGP to register hundreds of tasks: compile, merge resources, process manifest, run lint, package, sign, etc. A project with 2 build types × 3 flavors = 6 variants = roughly 1,200 AGP tasks. If you only ever build stagingDebug and productionRelease, the other 4 variants are pure waste. Using beforeVariants to disable them eliminates their tasks entirely — reducing configuration time by 30–60% in flavor-heavy projects. This is one of the highest-ROI optimizations you can put in a convention plugin.

Disabling Unused Variants

androidComponents {
    beforeVariants(selector().all()) { variantBuilder ->
        variantBuilder.enable = when {
            // Only allow staging+debug and production+release
            variantBuilder.buildType == "debug" &&
                variantBuilder.productFlavors.any { it.second == "production" } -> false
            variantBuilder.buildType == "release" &&
                variantBuilder.productFlavors.any { it.second == "staging" } -> false
            else -> true
        }
    }
}

Source Sets per Variant

src/
├── main/              ← shared across all variants
├── debug/             ← debug build type only
├── release/           ← release build type only
├── free/              ← free flavor only
├── paid/              ← paid flavor only
├── staging/           ← staging flavor only
├── production/        ← production flavor only
├── freeStagingDebug/  ← specific variant combination
└── paidProductionRelease/

AGP merges source sets in priority order: variant-specific > build-type > flavor > main. Resources in more specific source sets override less specific ones.

9. Custom Gradle Tasks

Lazy Task Registration

// GOOD — lazy registration: task is only created when needed
tasks.register<GenerateBuildInfoTask>("generateBuildInfo") {
    gitHash.set(provider { getGitHash() })
    buildTimestamp.set(provider { System.currentTimeMillis().toString() })
    outputFile.set(layout.buildDirectory.file("generated/build-info.txt"))
}

// BAD — eager creation: task is instantiated during configuration
tasks.create<GenerateBuildInfoTask>("generateBuildInfo") {
    // configured immediately, even if this task is never executed
}

Custom Task with Proper Inputs/Outputs

import org.gradle.api.DefaultTask
import org.gradle.api.file.RegularFileProperty
import org.gradle.api.provider.Property
import org.gradle.api.tasks.*

@CacheableTask
abstract class GenerateBuildInfoTask : DefaultTask() {

    @get:Input
    abstract val gitHash: Property<String>

    @get:Input
    abstract val buildTimestamp: Property<String>

    @get:OutputFile
    abstract val outputFile: RegularFileProperty

    @TaskAction
    fun generate() {
        val info = buildString {
            appendLine("git.hash=${gitHash.get()}")
            appendLine("build.timestamp=${buildTimestamp.get()}")
        }
        outputFile.get().asFile.writeText(info)
    }
}

// Registration with wiring
tasks.register<GenerateBuildInfoTask>("generateBuildInfo") {
    gitHash.set(provider {
        Runtime.getRuntime()
            .exec("git rev-parse --short HEAD")
            .inputStream.bufferedReader().readText().trim()
    })
    buildTimestamp.set(provider {
        java.time.Instant.now().toString()
    })
    outputFile.set(layout.buildDirectory.file("generated/build-info.txt"))
}

Key Annotations for Task Caching

Annotation Purpose Cache Impact
@Input Scalar value (String, Int, Boolean) Value is part of cache key
@InputFiles File-based inputs File content hash is part of cache key
@OutputFile / @OutputDirectory Where the task writes results Outputs are stored in cache
@PathSensitive(RELATIVE) Only relative paths matter Enables cache hits across machines
@CacheableTask Opt in to build cache Outputs can be restored from cache
@Incremental Receive only changed files Avoids reprocessing unchanged files
Configuration Avoidance in Tasks

Use Property<T> and Provider<T> for all task inputs — never resolve values during configuration. The provider { } lambda is evaluated lazily at execution time. This is critical for configuration cache compatibility: Gradle serializes the task graph, and Provider values are serialized as lazy references, not resolved values.

10. Staff-Level Corner Cases

Plugin Classpath Isolation

Each Gradle plugin runs in its own classloader. This means two plugins can depend on different versions of the same library (e.g., Plugin A uses Guava 31, Plugin B uses Guava 33) without conflicts. However, this isolation breaks down when plugins share types through the project's build script — if both plugins expose a Guava type in their API, the build script sees two incompatible versions of the same class.

// This is why convention plugins use compileOnly for AGP/Kotlin plugins:
dependencies {
    compileOnly(libs.android.gradlePlugin)  // available at compile time
    // NOT implementation — would add AGP to the plugin's runtime classpath
    // and potentially conflict with the version the project already has
}

afterEvaluate — The Anti-Pattern

// BAD — breaks configuration cache, creates ordering dependencies
project.afterEvaluate {
    tasks.named("compileDebugKotlin") {
        // This runs after ALL build scripts are evaluated
        // Order depends on project evaluation order — fragile
    }
}

// GOOD — use lazy configuration with providers
tasks.named<KotlinCompile>("compileDebugKotlin") {
    compilerOptions {
        freeCompilerArgs.add(provider {
            // Resolved at execution time, not evaluation time
            "-Xcontext-receivers"
        })
    }
}

afterEvaluate is a legacy escape hatch from the eager configuration era. It breaks configuration cache because it forces Gradle to evaluate the entire project before the block runs. In convention plugins, there is almost never a legitimate reason to use it — Provider<T> and lazy task configuration handle every use case better.

Convention Plugin Testing with GradleRunner

// build-logic/convention/src/test/kotlin/AndroidLibraryPluginTest.kt
import org.gradle.testkit.runner.GradleRunner
import org.gradle.testkit.runner.TaskOutcome
import org.junit.jupiter.api.Test
import org.junit.jupiter.api.io.TempDir
import java.io.File

class AndroidLibraryPluginTest {

    @TempDir
    lateinit var testProjectDir: File

    @Test
    fun `plugin applies without errors`() {
        File(testProjectDir, "settings.gradle.kts").writeText("")
        File(testProjectDir, "build.gradle.kts").writeText("""
            plugins {
                id("myapp.android.library")
            }
            android {
                namespace = "com.test.library"
            }
        """.trimIndent())

        val result = GradleRunner.create()
            .withProjectDir(testProjectDir)
            .withArguments("tasks", "--all")
            .withPluginClasspath()
            .build()

        assert(result.task(":tasks")?.outcome == TaskOutcome.SUCCESS)
    }

    @Test
    fun `compileSdk is set to 35`() {
        // ... setup ...
        val result = GradleRunner.create()

Plugin Marker Artifacts — How Plugin IDs Resolve

When you write: plugins { id("myapp.android.library") }

Gradle resolves the ID through this chain:

  1. Check pluginManagement { } repositories
  2. Look for a "plugin marker artifact":
     Maven coordinates: myapp.android.library:myapp.android.library.gradle.plugin:VERSION
     │
     └─► This POM/module metadata points to the actual plugin JAR
         (the one containing AndroidLibraryConventionPlugin.class)

  3. For included builds (composite build):
     Gradle skips Maven resolution entirely.
     It matches the plugin ID against gradlePlugin { plugins { register(...) } }
     in the included build's build.gradle.kts.

  4. The plugin class is loaded and apply(project) is called.

This is why convention plugins in an included build don't need
to be published — Gradle resolves them directly from source.

Composing Convention Plugins

// A feature plugin that composes library + compose + hilt plugins
class AndroidFeatureConventionPlugin : Plugin<Project> {
    override fun apply(target: Project) {
        with(target) {
            // Apply other convention plugins by ID
            pluginManager.apply("myapp.android.library")
            pluginManager.apply("myapp.android.compose")
            pluginManager.apply("myapp.android.hilt")

            dependencies {
                add("implementation", project(":core:ui"))
                add("implementation", project(":core:domain"))
                add("implementation", libs.findLibrary("hilt-navigation-compose").get())
                add("implementation", libs.findLibrary("lifecycle-viewmodel-compose").get())
            }
        }
    }
}

This composition pattern means a feature module's build file is a single line: id("myapp.android.feature"). The convention plugin handles Android configuration, Compose setup, Hilt wiring, and common dependencies. Module-specific build files only declare what's unique to that module.

L6 Insight — Convention Plugins Are Your Build's Type System

Think of convention plugins as a type system for your build. Just as Kotlin's type system prevents you from passing a String where an Int is expected, convention plugins prevent you from misconfiguring a module. A module that applies myapp.android.feature is guaranteed to have the correct compileSdk, Kotlin options, Compose setup, and Hilt wiring — there's no way to forget one of them. At Google scale, this is how teams maintain consistency across hundreds of modules owned by dozens of teams. The convention plugin is the single source of truth for "what does it mean to be a feature module in this project?" When that definition changes (e.g., bumping compileSdk), you change one file and every module picks it up automatically.

L6 Insight — Composite Build Cache Granularity

A subtle but important difference between buildSrc and composite builds: buildSrc is part of the "build classpath" — Gradle hashes the entire buildSrc output as a single input to every task's cache key. Change one file in buildSrc → every cache key changes → full rebuild. With a composite build, each convention plugin is a separate artifact. Gradle tracks which modules apply which plugins. If you change AndroidLibraryConventionPlugin, only modules using myapp.android.library are invalidated. Modules using myapp.kotlin.library remain cached. This granularity is what makes composite builds scale to hundreds of modules.
