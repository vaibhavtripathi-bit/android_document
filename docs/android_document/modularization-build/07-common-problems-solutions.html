Common Problems & Solutions

Circular dependencies, resource conflicts, manifest merging, R8 pitfalls, cross-module navigation, DI wiring, testing strategies, dynamic features, and staff-level corner cases in modular Android projects

1. Circular Dependencies

Gradle's dependency graph must be a DAG (directed acyclic graph). When :feature:home depends on :feature:profile (for user avatar) and :feature:profile depends on :feature:home (for navigation), Gradle rejects the build with Circular dependency between the following tasks.

Solution 1 — Extract Shared Code

Identify the shared concept and move it to a :core:* module. If both features need user data, create :core:user. If both need navigation, create :core:navigation. Both features depend on the core module; neither depends on each other.

  BEFORE (circular)              AFTER (DAG)

  :feature:home ──────┐          :feature:home ──┐
       ↑              ↓               ↓           ↓
  :feature:profile ───┘          :core:user   :core:navigation
                                      ↑           ↑
                                 :feature:profile ─┘

Solution 2 — Dependency Inversion

Define an interface in a shared module; each feature provides its own implementation. The features never reference each other — they only reference the abstraction.

// :core:navigation — shared abstraction
interface ProfileNavigator {
    fun navigateToProfile(userId: String)
}

interface HomeNavigator {
    fun navigateToHome()
}

// :feature:home — implements HomeNavigator, uses ProfileNavigator
class HomeScreen @Inject constructor(
    private val profileNavigator: ProfileNavigator
) { /* ... */ }

// :feature:profile — implements ProfileNavigator, uses HomeNavigator
class ProfileScreen @Inject constructor(
    private val homeNavigator: HomeNavigator
) { /* ... */ }

// :app — binds both implementations via Hilt
@Module
@InstallIn(SingletonComponent::class)
abstract class NavigationBindingsModule {
    @Binds abstract fun bindProfileNavigator(impl: ProfileNavigatorImpl): ProfileNavigator
    @Binds abstract fun bindHomeNavigator(impl: HomeNavigatorImpl): HomeNavigator
}

Solution 3 — Event Bus / Navigation Abstraction

Features communicate through a shared navigation contract using sealed classes or deep link URIs. No feature knows about any other feature's existence — they only emit navigation events that the app module routes.

Staff Insight
Circular dependencies are always a sign of a missing abstraction. The fix is never to hack around Gradle's restriction (e.g., merging modules, using reflection). The correct response is to identify the shared concept — user data, navigation contract, shared UI component — and extract it into a dedicated module. If you find yourself reaching for workarounds, your module boundaries are wrong.

2. Build Time Explosion

Adding modules increases configuration time because AGP registers hundreds of tasks per module (compile, merge resources, merge manifests, dex, lint, test — per variant). A 50-module project can spend 10+ seconds just configuring before any compilation starts.

Diagnosis

# Generate a build scan to identify bottlenecks
./gradlew assembleDebug --scan

# Quick local profile
./gradlew assembleDebug --profile
# → build/reports/profile/profile-*.html

Optimization Checklist

OptimizationSetting / CommandEstimated ImpactRisk
Configuration cache org.gradle.configuration-cache=true High 30-60% config phase Some plugins incompatible
Disable unused variants beforeVariants { if (...) enable = false } High ~50% fewer tasks None
Convention plugins Replace allprojects {} / subprojects {} Medium better cache hits Migration effort
kapt → KSP Migrate annotation processors to KSP High 25-40% compilation Not all processors support KSP
Parallel execution org.gradle.parallel=true Medium scales with cores Rare thread-safety issues in plugins
Daemon memory org.gradle.jvmargs=-Xmx4g Medium fewer GC pauses Consumes more RAM
Build cache org.gradle.caching=true High skips unchanged tasks Stale cache entries (rare) 
// Disable unused build variants in a library module
androidComponents {
    beforeVariants(selector().withBuildType("staging")) { variantBuilder ->
        variantBuilder.enable = false
    }
}

3. Resource Conflicts

When two modules define the same resource name (e.g., R.string.app_name or R.color.primary), the last module merged wins. This is unpredictable and leads to the wrong string or color appearing on some screens.

Solution 1 — Resource Prefix

// :feature:home/build.gradle.kts
android {
    resourcePrefix = "home_"
}

Lint will warn when resources don't start with the prefix. This is a lint check only — it doesn't prevent compilation. Enforce it in CI by failing on lint warnings.

Solution 2 — Non-Transitive R Classes

# gradle.properties (default since AGP 8.0)
android.nonTransitiveRClass=true

With non-transitive R classes, each module's R class contains only its own resources, not resources from its dependencies. To use a resource from :core:ui, you must explicitly reference com.example.core.ui.R.string.title instead of just R.string.title.

Staff Insight
Non-transitive R classes are the single most impactful change for modular resource management. They eliminate accidental resource shadowing, reduce R class size (fewer fields = smaller DEX), and make resource ownership explicit. If you're on AGP 8+ and haven't enabled this, you're carrying unnecessary risk. The migration cost is a one-time find-and-replace of unqualified R references.

4. Manifest Merging Issues

Each module has its own AndroidManifest.xml. AGP merges them into a single manifest during the build. Conflicts arise when modules or libraries declare conflicting attributes.

Common Conflicts & Fixes

ConflictManifest ToolExample
Library overrides app's theme tools:replace <application tools:replace="android:theme" android:theme="@style/AppTheme">
Library adds unwanted permission tools:remove <uses-permission android:name="android.permission.CAMERA" tools:node="remove"/>
Library has higher minSdk tools:overrideLibrary <uses-sdk tools:overrideLibrary="com.example.lib"/>
Duplicate activity declaration tools:node="remove" Remove the library's activity, declare your own

Debugging Manifest Merges

# Inspect the final merged manifest
cat app/build/intermediates/merged_manifests/debug/AndroidManifest.xml

# See merge decisions in detail
./gradlew :app:processDebugManifest --info

# Merge report (shows which manifest contributed each element)
cat app/build/outputs/logs/manifest-merger-debug-report.txt
Warning
tools:overrideLibrary silences the minSdk mismatch but doesn't fix it — your app will crash on older devices if the library uses APIs above the app's minSdk. Only use this when you've verified the library gracefully degrades or the code path is guarded by an API level check.

5. ProGuard / R8 with Modules

In a modular project, R8 runs only in the :app module during release builds. Library modules contribute their code, but R8 has no way of knowing which library classes are "public API" unless the library tells it.

Consumer ProGuard Rules

// :core:network/build.gradle.kts
android {
    defaultConfig {
        // These rules are automatically applied when any module depends on :core:network
        consumerProguardFiles("consumer-rules.pro")
    }
}

// :app/build.gradle.kts
android {
    buildTypes {
        release {
            isMinifyEnabled = true
            proguardFiles(
                getDefaultProguardFile("proguard-android-optimize.txt"),
                "proguard-rules.pro"
            )
        }
    }
}
# :core:network/consumer-rules.pro
# Keep all public API classes — R8 can't know these are needed
-keep class com.example.core.network.api.** { *; }

# Keep Retrofit interfaces (reflection-based)
-keep,allowobfuscation interface com.example.core.network.api.** {
    @retrofit2.http.* <methods>;
}

Debugging R8 Issues

# See what R8 removed
./gradlew :app:assembleRelease --info

# Add to proguard-rules.pro to dump removed classes
-printusage build/outputs/mapping/release/usage.txt

# Print the full configuration R8 used (merged from all modules)
-printconfiguration build/outputs/mapping/release/configuration.txt
Staff Insight
The most common R8 crash in modular projects is a library module that doesn't declare consumerProguardFiles for its public API. The app module's R8 pass strips the library's classes because nothing in the app directly references them (they're accessed via DI or reflection). The fix is simple: every library module must ship its own consumer ProGuard rules. Treat consumer-rules.pro as part of the module's public contract, not an afterthought.

6. Navigation Across Modules

Feature modules can't depend on each other (that would create tight coupling or circular dependencies), yet they need to navigate to each other's screens. This is one of the most frequently asked modularization questions in interviews.

Pattern 1 — Navigation Interface Abstraction

// :core:navigation — defines contracts
interface FeatureNavigator {
    fun navigateToProfile(userId: String)
    fun navigateToSettings()
    fun navigateToOrderDetail(orderId: String)
}

// :feature:home — consumes the navigator
class HomeViewModel @Inject constructor(
    private val navigator: FeatureNavigator
) : ViewModel() {
    fun onUserAvatarClicked(userId: String) {
        navigator.navigateToProfile(userId)
    }
}

// :app — provides the implementation, wiring NavController
class AppFeatureNavigator @Inject constructor(
    private val navController: Provider<NavController>
) : FeatureNavigator {
    override fun navigateToProfile(userId: String) {
        navController.get().navigate("profile/$userId")
    }
    override fun navigateToSettings() {
        navController.get().navigate("settings")
    }
    override fun navigateToOrderDetail(orderId: String) {
        navController.get().navigate("order/$orderId")
    }
}

Pattern 2 — Deep Link Navigation

// :core:navigation — route definitions
object DeepLinks {
    const val PROFILE = "myapp://profile/{userId}"
    const val SETTINGS = "myapp://settings"
    const val ORDER = "myapp://order/{orderId}"
}

// :feature:profile/AndroidManifest.xml declares the deep link
// <activity android:name=".ProfileActivity">
//   <intent-filter>
//     <data android:scheme="myapp" android:host="profile" />
//   </intent-filter>
// </activity>

// :feature:home — navigates via URI, no dependency on :feature:profile
fun navigateToProfile(context: Context, userId: String) {
    val uri = Uri.parse("myapp://profile/$userId")
    context.startActivity(Intent(Intent.ACTION_VIEW, uri))
}

Pattern 3 — Navigation Compose with Type-Safe Routes

// :core:navigation — shared route definitions
@Serializable data class ProfileRoute(val userId: String)
@Serializable data object SettingsRoute
@Serializable data class OrderRoute(val orderId: String)

// :feature:profile — registers its graph
fun NavGraphBuilder.profileGraph(onBack: () -> Unit) {
    composable<ProfileRoute> { backStackEntry ->
        val route = backStackEntry.toRoute<ProfileRoute>()
        ProfileScreen(userId = route.userId, onBack = onBack)
    }
}

// :app — assembles all feature graphs
NavHost(navController, startDestination = HomeRoute) {
    homeGraph(onProfileClick = { navController.navigate(ProfileRoute(it)) })
    profileGraph(onBack = { navController.popBackStack() })
    settingsGraph()
}
Staff Insight
Deep link navigation is the most scalable approach for large apps (50+ features). It decouples features completely, supports dynamic feature modules (which can't use compile-time dependencies), and works with external navigation (push notifications, web links). The trade-off is losing compile-time safety — a typo in a URI string fails at runtime. Type-safe Navigation Compose routes offer a middle ground: compile-time safety with route classes in a shared module, while keeping features decoupled.

7. Dependency Injection Across Modules

Hilt's component hierarchy is designed for multi-module projects. The key insight: Hilt merges all @Module and @InstallIn declarations from all modules into the app's generated component at compile time.

How Cross-Module Injection Works

  :core:domain                    :core:data                       :feature:home
  ┌──────────────────┐           ┌──────────────────────┐         ┌──────────────────┐
  │ interface UserRepo│           │ @Inject              │         │ @Inject           │
  │                  │           │ class UserRepoImpl   │         │ class HomeVM(     │
  │                  │           │   : UserRepo         │         │   repo: UserRepo  │
  └──────────────────┘           └──────────────────────┘         │ )                 │
         ↑                              ↑                         └──────────────────┘
         │                              │                                  │
         └──────────────────────────────┴──────────────────────────────────┘
                              :app (Hilt merges everything)
                              ┌─────────────────────────────┐
                              │ @HiltAndroidApp              │
                              │ Generated SingletonComponent │
                              │ includes all @Module from    │
                              │ every module on classpath    │
                              └─────────────────────────────┘

// :core:domain — pure Kotlin module
interface UserRepository {
    suspend fun getUser(id: String): User
}

// :core:data — Android library module
@Module
@InstallIn(SingletonComponent::class)
abstract class DataModule {
    @Binds abstract fun bindUserRepo(impl: UserRepositoryImpl): UserRepository
}

class UserRepositoryImpl @Inject constructor(
    private val api: UserApi,
    private val dao: UserDao
) : UserRepository {
    override suspend fun getUser(id: String): User = /* ... */
}

// :feature:home — Android library module
@HiltViewModel
class HomeViewModel @Inject constructor(
    private val userRepository: UserRepository  // injected from :core:data
) : ViewModel() { /* ... */ }

@EntryPoint for Non-Hilt Classes

// Access Hilt dependencies from a ContentProvider or dynamic feature
@EntryPoint
@InstallIn(SingletonComponent::class)
interface AnalyticsEntryPoint {
    fun analyticsTracker(): AnalyticsTracker
}

// Usage in a ContentProvider
class MyContentProvider : ContentProvider() {
    override fun onCreate(): Boolean {
        val entryPoint = EntryPointAccessors.fromApplication(
            context!!.applicationContext,
            AnalyticsEntryPoint::class.java
        )
        val tracker = entryPoint.analyticsTracker()
        return true
    }
}
Common Pitfalls
  • Missing @AndroidEntryPoint on Activity/Fragment — Hilt can't inject into it, crashes at runtime
  • Missing @HiltAndroidApp on Application — no component is generated, nothing works
  • KSP/kapt not applied in a library module — Hilt annotations are silently ignored, bindings are missing
  • Wrong component scope — installing a module in ViewModelComponent when it should be in SingletonComponent makes the binding unavailable to other ViewModels

8. Testing Across Modules

Modularization naturally enables better testing: each module is an isolated unit with a clear API boundary. The challenge is sharing test infrastructure and testing cross-module interactions.

Testing Strategy by Layer

Test TypeLocationScopeSpeed
Unit tests Each module's test/ Single module in isolation Fast
Integration tests :app or :integration-test module Cross-module interactions Medium
Robolectric tests Each module's test/ Android components without device Medium
Instrumented tests androidTest/ Full device/emulator, cross-module Slow

Test Fixtures — Sharing Test Utilities

// :core:data/build.gradle.kts
plugins {
    id("com.android.library")
    id("java-test-fixtures")  // enables testFixtures source set
}

// :core:data/testFixtures/kotlin/com/example/core/data/FakeUserRepository.kt
class FakeUserRepository : UserRepository {
    private val users = mutableMapOf<String, User>()

    fun addUser(user: User) { users[user.id] = user }

    override suspend fun getUser(id: String): User =
        users[id] ?: throw NoSuchElementException("User $id not found")
}

// :feature:home/build.gradle.kts — consume the test fixture
dependencies {
    testImplementation(testFixtures(project(":core:data")))
}

Shared Test Infrastructure Module

// :core:testing/build.gradle.kts — shared test utilities
dependencies {
    api(libs.junit)
    api(libs.kotlinx.coroutines.test)
    api(libs.turbine)
    api(libs.truth)
}

// :core:testing — custom test rule
class MainDispatcherRule(
    private val dispatcher: TestDispatcher = UnconfinedTestDispatcher()
) : TestWatcher() {
    override fun starting(description: Description) {
        Dispatchers.setMain(dispatcher)
    }
    override fun finished(description: Description) {
        Dispatchers.resetMain()
    }
}

// :feature:home/build.gradle.kts
dependencies {
    testImplementation(project(":core:testing"))
}

9. Dynamic Feature Module Issues

Dynamic feature modules use an inverted dependency: the dynamic feature depends on :app, not the other way around. This creates unique challenges.

Key Constraints

  • Inverted dependency — :app can't reference dynamic feature classes directly; use reflection or SplitInstallManager
  • Resources unavailable until installed — accessing a dynamic feature's resources before installation crashes with Resources.NotFoundException
  • ProGuard coordination — dynamic features have their own R8 configuration; rules must be coordinated with the base module to avoid stripping shared interfaces
  • Testing — standard androidTest can't test dynamic features; use bundletool for local testing
// Installing a dynamic feature at runtime
val splitInstallManager = SplitInstallManagerFactory.create(context)
val request = SplitInstallRequest.newBuilder()
    .addModule("premium_feature")
    .build()

splitInstallManager.startInstall(request)
    .addOnSuccessListener { sessionId -> /* monitor progress */ }
    .addOnFailureListener { exception -> /* handle error */ }

// Monitor installation state
splitInstallManager.registerListener { state ->
    when (state.status()) {
        SplitInstallSessionStatus.INSTALLED -> {
            // Safe to access the module's code and resources
            SplitCompat.installActivity(this)
        }
        SplitInstallSessionStatus.FAILED -> {
            // Handle failure — network error, insufficient storage, etc.
        }
    }
}

// Accessing dynamic feature code via reflection
val clazz = Class.forName("com.example.premium.PremiumActivity")
startActivity(Intent(this, clazz))
Critical
Your Application class must extend SplitCompatApplication (or call SplitCompat.install(this) in attachBaseContext) for dynamic feature resources and code to be accessible. Without this, ClassNotFoundException and Resources.NotFoundException will occur even after the module is installed.

10. Configuration Leaks

A configuration leak occurs when a plugin or build script in one module unintentionally affects other modules. This is one of the subtlest and hardest-to-debug problems in multi-module builds.

Common Leak Sources

Leak SourceProblemFix
allprojects { } Applies configuration to every module, even those that don't need it Use convention plugins per module type
subprojects { } Same as above, slightly narrower scope Convention plugins or explicit per-module config
buildscript { classpath(...) } Plugin classpath leaks to all subprojects Use plugins { } block with version catalogs
Root build.gradle.kts applying plugins Plugin side effects affect all modules Only apply plugins in the modules that need them 
// settings.gradle.kts — centralized plugin version management
pluginManagement {
    repositories {
        google()
        mavenCentral()
        gradlePluginPortal()
    }
}

// Each module applies only what it needs
// :feature:home/build.gradle.kts
plugins {
    id("com.android.library")
    id("org.jetbrains.kotlin.android")
    id("com.google.devtools.ksp")
    id("dagger.hilt.android.plugin")
}

Debugging Configuration Leaks

# If a build works without cache but fails with cache, suspect a leak
./gradlew assembleDebug --no-build-cache

# Compare configuration of two modules — they should differ only intentionally
./gradlew :feature:home:dependencies --configuration debugCompileClasspath
./gradlew :feature:profile:dependencies --configuration debugCompileClasspath

# Check which plugins are applied to a module
./gradlew :feature:home:buildEnvironment

11. Staff-Level Corner Cases

Gradle Module ≠ Android Module

Not every module needs the Android Gradle Plugin. A :core:domain module containing only Kotlin data classes and interfaces should use kotlin("jvm") — no Android dependencies, faster compilation, and testable on the JVM without Robolectric.

// :core:domain/build.gradle.kts — pure Kotlin, no Android
plugins {
    kotlin("jvm")
}

dependencies {
    implementation(libs.kotlinx.coroutines.core)
    // No Android dependencies — this module compiles in seconds
    // and tests run on the JVM without any Android test runner
}

// :core:data/build.gradle.kts — needs Android for Room, DataStore, etc.
plugins {
    id("com.android.library")
    kotlin("android")
}
Staff Insight
Every module that doesn't need Android APIs should be a pure Kotlin/JVM module. This isn't just a build speed optimization — it's an architectural signal. A :core:domain module with kotlin("jvm") cannot accidentally depend on Android, enforcing clean architecture at the build system level. In a 50-module project, having 15-20 pure Kotlin modules instead of Android libraries can cut total configuration time by 30% because AGP's task registration overhead is eliminated entirely.

includeBuild for Local Library Development

// settings.gradle.kts — substitute a published library with a local checkout
includeBuild("../my-shared-library") {
    dependencySubstitution {
        substitute(module("com.example:shared-lib"))
            .using(project(":shared-lib"))
    }
}

// Now any module that depends on "com.example:shared-lib:1.0.0"
// will use the local source instead — no SNAPSHOT publishing needed

dependencySubstitution for A/B Testing

// Swap implementations at build time based on a property
// settings.gradle.kts
if (providers.gradleProperty("useNewNetworkStack").getOrElse("false") == "true") {
    includeBuild("../network-v2") {
        dependencySubstitution {
            substitute(module("com.example:network"))
                .using(project(":network-v2"))
        }
    }
}

// Build with: ./gradlew assembleDebug -PuseNewNetworkStack=true

Gradle's --continue Flag

# Continue building other modules even if one fails
# Useful in CI to get maximum test coverage in a single run
./gradlew test --continue

# Without --continue: first failure stops the entire build
# With --continue: all modules run their tests, failures are reported at the end
Staff Insight
The --continue flag is essential for CI pipelines in large modular projects. Without it, a test failure in :core:data prevents :feature:home tests from running — even though they're independent. With --continue, you get the full picture of what's broken in a single CI run instead of playing whack-a-mole across multiple builds. Combine it with --parallel for maximum CI throughput.
