07 — RecyclerView Internals

Architecture, view recycling hierarchy, LayoutManager fill algorithm, DiffUtil, prefetch, and production-grade performance patterns

1. RecyclerView Architecture

RecyclerView extends ViewGroup and is fundamentally a scrolling container that recycles child Views instead of creating new ones for every item. Its power comes from a strict separation of concerns across several collaborating components.

1.1 Core Components

ComponentResponsibilityKey Methods
RecyclerViewScrolling container, orchestrates layout passesonLayout(), onTouchEvent()
Adapter<VH>Creates ViewHolders, binds data to themonCreateViewHolder(), onBindViewHolder()
ViewHolderHolds reference to item View and metadataitemView, bindingAdapterPosition
LayoutManagerMeasures, positions, and recycles child ViewsonLayoutChildren(), scrollVerticallyBy()
RecyclerManages the 4-level recycling cachegetViewForPosition(), recycleView()
RecycledViewPoolCross-RecyclerView pool of unbound ViewHoldersgetRecycledView(), putRecycledView()
ItemDecorationDraws dividers, spacing, overlaysonDraw(), onDrawOver(), getItemOffsets()
ItemAnimatorAnimates add/remove/move/changeanimateAdd(), animateRemove()

The key insight: RecyclerView itself knows almost nothing about how items are laid out. It delegates positioning entirely to the LayoutManager, which requests Views from the Recycler, which in turn calls the Adapter only when all caches miss. This inversion of control is what makes RecyclerView so extensible compared to the old ListView.

Staff Insight

RecyclerView's architecture is a textbook application of the Strategy pattern. LayoutManager, ItemAnimator, and ItemDecoration are all swappable strategies. In interviews, framing RecyclerView as "a ViewGroup with pluggable layout, animation, and decoration strategies" signals deep understanding. The old ListView baked all of these into one monolithic class — RecyclerView's design is the fix.

2. ViewHolder Pattern

RecyclerView.ViewHolder is the unit of recycling. It wraps a single item View and carries metadata about its position, state, and type.

2.1 Lifecycle

class UserViewHolder(view: View) : RecyclerView.ViewHolder(view) {
    val name: TextView = view.findViewById(R.id.name)
    val avatar: ImageView = view.findViewById(R.id.avatar)
}

class UserAdapter(private val users: List<User>) :
    RecyclerView.Adapter<UserViewHolder>() {

    override fun onCreateViewHolder(parent: ViewGroup, viewType: Int): UserViewHolder {
        val view = LayoutInflater.from(parent.context)
            .inflate(R.layout.item_user, parent, false)
        return UserViewHolder(view)
    }

    override fun onBindViewHolder(holder: UserViewHolder, position: Int) {
        val user = users[position]
        holder.name.text = user.name
        Glide.with(holder.avatar).load(user.avatarUrl).into(holder.avatar)
    }

    override fun getItemCount() = users.size
}

onCreateViewHolder() is called only when the Recycler has no available ViewHolder of the requested type. onBindViewHolder() is called every time a ViewHolder is about to become visible — this is where you bind data. The separation means inflation (expensive) happens rarely, while binding (cheap) happens frequently.

2.2 Position Types — The #1 Source of Bugs

ViewHolder exposes three different position accessors, and confusing them is the most common RecyclerView bug:

AccessorMeaningWhen to Use
bindingAdapterPosition Position in the adapter that last called onBindViewHolder() on this VH. Returns NO_POSITION (-1) if the VH is being animated out. Data access — items[holder.bindingAdapterPosition]
layoutPosition Position as of the last completed layout pass. May lag behind adapter changes by one frame. Layout calculations, ItemDecoration
absoluteAdapterPosition Global position across all adapters in a ConcatAdapter. Same as bindingAdapterPosition for single adapters. ConcatAdapter scenarios
Common Bug

Using adapterPosition (deprecated) or layoutPosition inside a click listener to index into your data list. If items were inserted/removed since the last layout, layoutPosition is stale. Always use bindingAdapterPosition and guard against NO_POSITION:

init {
    itemView.setOnClickListener {
        val pos = bindingAdapterPosition
        if (pos != RecyclerView.NO_POSITION) {
            onItemClick(items[pos])
        }
    }
}

2.3 Multi-Type RecyclerView

Override getItemViewType(position) to return different type constants. RecyclerView maintains separate recycling pools per view type, so a header ViewHolder is never reused for a content item.

override fun getItemViewType(position: Int): Int = when (items[position]) {
    is Header -> TYPE_HEADER
    is Content -> TYPE_CONTENT
    is Ad -> TYPE_AD
    else -> TYPE_UNKNOWN
}

override fun onCreateViewHolder(parent: ViewGroup, viewType: Int) = when (viewType) {
    TYPE_HEADER -> HeaderVH(inflate(R.layout.item_header, parent))
    TYPE_CONTENT -> ContentVH(inflate(R.layout.item_content, parent))
    TYPE_AD -> AdVH(inflate(R.layout.item_ad, parent))
    else -> throw IllegalArgumentException("Unknown viewType: $viewType")
}

3. The Recycler — View Recycling Internals

The Recycler is the heart of RecyclerView's performance. It manages a four-level cache hierarchy that determines whether a View needs to be created, rebound, or can be reused as-is.

3.1 Four-Level Cache Hierarchy

  Recycler Cache Hierarchy — View Resolution Order
  ══════════════════════════════════════════════════

  LayoutManager needs a View for position N
                    │
                    ▼
  ┌─────────────────────────────────────────────────────────┐
  │  Level 1: SCRAP  (mAttachedScrap / mChangedScrap)       │
  │                                                         │
  │  Views still attached to RV, marked for reuse during    │
  │  the current layout pass. No rebinding needed.          │
  │  mChangedScrap: items with payload changes              │
  │  Capacity: unbounded (all currently laid-out children)   │
  └──────────────────────┬──────────────────────────────────┘
                    miss │
                         ▼
  ┌─────────────────────────────────────────────────────────┐
  │  Level 2: CACHE  (mCachedViews)                         │
  │                                                         │
  │  Recently scrolled-off Views, still bound to their      │
  │  original position. If same position requested →        │
  │  return directly, NO rebinding.                         │
  │  Default capacity: 2                                     │
  └──────────────────────┬──────────────────────────────────┘
                    miss │
                         ▼
  ┌─────────────────────────────────────────────────────────┐
  │  Level 3: VIEW CACHE EXTENSION  (mViewCacheExtension)   │
  │                                                         │

3.3 The Resolution Algorithm (AOSP)

// Simplified from RecyclerView.Recycler.tryGetViewHolderForPositionByDeadline()

fun getViewForPosition(position: Int): ViewHolder {
    // 1. Check scrap (attached + changed)
    val fromScrap = getScrapOrHiddenOrCachedHolderForPosition(position, dryRun = false)
    if (fromScrap != null) return fromScrap

    // 2. Check cache by stable ID (if hasStableIds)
    if (adapter.hasStableIds()) {
        val fromId = getScrapOrCachedViewForId(adapter.getItemId(position))
        if (fromId != null) return fromId
    }

    // 3. Check ViewCacheExtension
    val fromExtension = viewCacheExtension?.getViewForPositionAndType(position, type)
    if (fromExtension != null) return getChildViewHolder(fromExtension)

    // 4. Check RecycledViewPool
    val fromPool = recycledViewPool.getRecycledView(viewType)
    if (fromPool != null) {
        fromPool.resetInternal()  // clear flags, position
        adapter.onBindViewHolder(fromPool, position)  // REBIND
        return fromPool
    }

    // 5. Create new
    val created = adapter.onCreateViewHolder(parent, viewType)
    adapter.onBindViewHolder(created, position)
    return created
}

3.4 Tuning the Cache

// Increase cache for fast-scrolling lists (default: 2)
recyclerView.setItemViewCacheSize(5)

// Increase pool for specific view types (default: 5 per type)
recyclerView.recycledViewPool.setMaxRecycledViews(TYPE_CONTENT, 20)

// Share pool across multiple RecyclerViews (e.g., ViewPager2 pages)
val sharedPool = RecyclerView.RecycledViewPool()
recyclerView1.setRecycledViewPool(sharedPool)
recyclerView2.setRecycledViewPool(sharedPool)
Staff Insight

The scrap list is the most misunderstood cache level. During every layout pass, RecyclerView detaches all children into scrap, then the LayoutManager re-requests them by position. If the item hasn't changed, it comes straight from scrap with zero cost — no rebinding. This is why notifyItemChanged() on a single position is so much cheaper than notifyDataSetChanged(): the latter invalidates all scrap entries, forcing a full rebind of every visible item. In interviews, explaining the scrap→cache→pool hierarchy with rebinding semantics is the gold standard answer for "how does RecyclerView recycling work?"

4.2 The Fill Algorithm (LinearLayoutManager)

The core of LinearLayoutManager is the fill() method. It's called during both initial layout and scroll events.

  LinearLayoutManager.fill() — Simplified Flow
  ═══════════════════════════════════════════════

  fill(recycler, layoutState, state, stopOnFocusable)
       │
       ▼
  remainingSpace = layoutState.available + layoutState.extra
       │
       ▼
  ┌──────────── LOOP while remainingSpace > 0 ────────────┐
  │                                                        │
  │  1. layoutChunk(recycler, state, layoutState, result)  │
  │     │                                                  │
  │     ├─ view = layoutState.next(recycler)               │
  │     │       → Recycler.getViewForPosition(pos)         │
  │     │                                                  │
  │     ├─ addView(view)  or  addDisappearingView(view)    │
  │     │                                                  │
  │     ├─ measureChildWithMargins(view, 0, 0)             │
  │     │                                                  │
  │     └─ layoutDecoratedWithMargins(view, l, t, r, b)    │
  │                                                        │
  │  2. remainingSpace -= result.consumed                   │
  │  3. layoutState.offset += result.consumed               │
  │  4. layoutState.currentPosition += layoutState.dir      │
  │                                                        │
  └────────────────────────────────────────────────────────┘
       │
       ▼
  return total space consumed (pixels filled)

Each iteration of the loop lays out one child. The LayoutState tracks the current fill position (which adapter position to fetch next) and direction (toward start or end). The loop stops when the remaining space is exhausted or there are no more items.

4.3 Scroll Handling

// LinearLayoutManager — simplified scroll logic
override fun scrollVerticallyBy(
    dy: Int,
    recycler: RecyclerView.Recycler,
    state: RecyclerView.State
): Int {
    // 1. Update layout state for scroll direction
    updateLayoutState(if (dy > 0) LAYOUT_END else LAYOUT_START, abs(dy), true, state)

    // 2. Fill new items in scroll direction
    val consumed = fill(recycler, layoutState, state, false)

    // 3. Offset all children by scroll amount
    offsetChildrenVertical(-scrolled)

    // 4. Recycle views that scrolled off-screen
    recycleViewsFromStart(recycler, scrollingOffset)  // or recycleViewsFromEnd

    return scrolled
}

The scroll handler fills new Views at the leading edge and recycles Views that have scrolled past the trailing edge. This is the core recycling loop — Views leaving one side are returned to the Recycler and reused for items entering from the other side.

4.4 Custom LayoutManager

Writing a custom LayoutManager requires implementing at minimum:

class CarouselLayoutManager : RecyclerView.LayoutManager() {

    override fun generateDefaultLayoutParams() =
        RecyclerView.LayoutParams(
            ViewGroup.LayoutParams.WRAP_CONTENT,
            ViewGroup.LayoutParams.WRAP_CONTENT
        )

    override fun onLayoutChildren(recycler: RecyclerView.Recycler, state: RecyclerView.State) {
        detachAndScrapAttachedViews(recycler)  // scrap all current children

        val itemCount = state.itemCount
        if (itemCount == 0) {
            removeAndRecycleAllViews(recycler)
            return
        }

        // Position each visible item
        var offsetX = paddingLeft
        for (i in 0 until itemCount) {
            val view = recycler.getViewForPosition(i)
            addView(view)
            measureChildWithMargins(view, 0, 0)

            val width = getDecoratedMeasuredWidth(view)
            val height = getDecoratedMeasuredHeight(view)
            layoutDecorated(view, offsetX, paddingTop, offsetX + width, paddingTop + height)

            offsetX += width
            if (offsetX > width + paddingRight) break  // stop when off-screen
        }
    }

    override fun canScrollHorizontally() = true

    override fun scrollHorizontallyBy(
        dx: Int, recycler: RecyclerView.Recycler, state: RecyclerView.State
    ): Int {
        offsetChildrenHorizontal(-dx)
        // Fill/recycle logic here...
        return dx
    }
}
Caution

Writing a production-quality custom LayoutManager is one of the hardest tasks in Android development. You must handle: initial layout, scrolling in both directions, predictive animations (pre-layout), accessibility traversal, focus search, saved state, and edge cases like empty adapter or single item. Most apps should use the built-in LayoutManagers or libraries like CarouselLayoutManager from Material Components.

5. DiffUtil — Efficient Updates

DiffUtil computes the minimum set of update operations to transform one list into another. It uses Eugene Myers' diff algorithm with O(N + D²) time complexity, where D is the number of edits.

5.1 The Algorithm

  DiffUtil Pipeline
  ═════════════════

  Old List: [A, B, C, D, E]
  New List: [A, C, D, F, E]

  Step 1: areItemsTheSame(old, new)
          → Identifies B removed, F inserted

  Step 2: areContentsTheSame(old, new)
          → For items present in both lists, checks if content changed

  Step 3: getChangePayload(old, new)
          → For changed items, returns the specific delta

  Step 4: DiffResult.dispatchUpdatesTo(adapter)
          → notifyItemRemoved(1)      // B removed
          → notifyItemInserted(3)     // F inserted
          → notifyItemMoved(...)      // if applicable

  Result: Only affected ViewHolders are rebound.
          Animations play automatically.

5.2 DiffUtil.Callback

class UserDiffCallback(
    private val old: List<User>,
    private val new: List<User>
) : DiffUtil.Callback() {

    override fun getOldListSize() = old.size
    override fun getNewListSize() = new.size

    override fun areItemsTheSame(oldPos: Int, newPos: Int): Boolean =
        old[oldPos].id == new[newPos].id  // same entity?

    override fun areContentsTheSame(oldPos: Int, newPos: Int): Boolean =
        old[oldPos] == new[newPos]  // same visual content?

    override fun getChangePayload(oldPos: Int, newPos: Int): Any? {
        val oldUser = old[oldPos]
        val newUser = new[newPos]
        val diff = mutableSetOf<String>()
        if (oldUser.name != newUser.name) diff.add("name")
        if (oldUser.avatarUrl != newUser.avatarUrl) diff.add("avatar")
        return diff.ifEmpty { null }
    }
}

5.3 ListAdapter — The Modern Approach

ListAdapter wraps AsyncListDiffer, which runs DiffUtil on a background thread and dispatches results on the main thread.

class UserAdapter : ListAdapter<User, UserViewHolder>(UserDiffItemCallback()) {

    override fun onCreateViewHolder(parent: ViewGroup, viewType: Int) =
        UserViewHolder(inflate(R.layout.item_user, parent))

    override fun onBindViewHolder(holder: UserViewHolder, position: Int) {
        holder.bind(getItem(position))
    }

    // Partial binding with payloads
    override fun onBindViewHolder(holder: UserViewHolder, position: Int, payloads: List<Any>) {
        if (payloads.isEmpty()) {
            onBindViewHolder(holder, position)
            return
        }
        @Suppress("UNCHECKED_CAST")
        val changes = payloads.first() as? Set<String> ?: return
        val user = getItem(position)
        if ("name" in changes) holder.name.text = user.name
        if ("avatar" in changes) Glide.with(holder.avatar).load(user.avatarUrl).into(holder.avatar)
    }
}

class UserDiffItemCallback : DiffUtil.ItemCallback<User>() {
    override fun areItemsTheSame(old: User, new: User) = old.id == new.id
    override fun areContentsTheSame(old: User, new: User) = old == new
}

// Usage — just submit a new list
viewModel.users.collect { users ->
    adapter.submitList(users)
}
Staff Insight

submitList() has a subtle gotcha: if you submit the same list instance after mutating it in place, DiffUtil sees old === new and short-circuits with no diff. Always submit a new list instance (e.g., list.toList() or use immutable data classes). This is the #2 most common RecyclerView bug after position confusion. In production, enforce immutability by using kotlinx.collections.immutable.PersistentList or always mapping from a Flow that emits new list objects.

Anti-Pattern

notifyDataSetChanged() is the nuclear option: it invalidates all ViewHolders, forces a full rebind of every visible item, disables all item animations, and defeats the scrap cache. Use it only as a last resort (e.g., when the entire dataset is replaced with no correlation to the old data).

6.2 Custom Divider with Type Awareness

class TypeAwareDivider(context: Context) : RecyclerView.ItemDecoration() {

    private val dividerPaint = Paint().apply {
        color = ContextCompat.getColor(context, R.color.divider)
        strokeWidth = 1f.dpToPx(context)
    }
    private val sectionPaint = Paint().apply {
        color = ContextCompat.getColor(context, R.color.section_divider)
        strokeWidth = 8f.dpToPx(context)
    }

    override fun getItemOffsets(
        outRect: Rect, view: View, parent: RecyclerView, state: RecyclerView.State
    ) {
        val position = parent.getChildAdapterPosition(view)
        if (position == RecyclerView.NO_POSITION) return

        val adapter = parent.adapter ?: return
        val currentType = adapter.getItemViewType(position)
        val nextType = if (position + 1 < adapter.itemCount)
            adapter.getItemViewType(position + 1) else -1

        outRect.bottom = if (currentType != nextType) 8.dp else 1.dp
    }

    override fun onDraw(canvas: Canvas, parent: RecyclerView, state: RecyclerView.State) {
        val left = parent.paddingLeft.toFloat()
        val right = (parent.width - parent.paddingRight).toFloat()

        for (i in 0 until parent.childCount - 1) {
            val child = parent.getChildAt(i)
            val pos = parent.getChildAdapterPosition(child)
            val adapter = parent.adapter ?: continue

            val currentType = adapter.getItemViewType(pos)
            val nextType = if (pos + 1 < adapter.itemCount)
                adapter.getItemViewType(pos + 1) else -1

            val paint = if (currentType != nextType) sectionPaint else dividerPaint
            val y = child.bottom + (child.layoutParams as ViewGroup.MarginLayoutParams).bottomMargin
            canvas.drawLine(left, y.toFloat(), right, y.toFloat(), paint)
        }
    }
}

7.2 Predictive Animations

RecyclerView supports predictive item animations: before a removal, it performs a "pre-layout" pass that includes the items that will slide into view after the removal. This lets the ItemAnimator animate those items from their predicted off-screen position into their final position, creating a smooth entrance effect.

  Predictive Animation — Item Removal
  ═════════════════════════════════════

  Pre-layout (includes items beyond viewport):
  ┌──────────────┐
  │  Item A       │  visible
  │  Item B       │  visible  ← will be removed
  │  Item C       │  visible
  │  Item D       │  visible
  │╌╌╌╌╌╌╌╌╌╌╌╌╌╌│  viewport edge
  │  Item E       │  off-screen but pre-laid-out
  └──────────────┘

  Post-layout (after removal):
  ┌──────────────┐
  │  Item A       │  stays
  │  Item C       │  slides up (animateMove)
  │  Item D       │  slides up (animateMove)
  │  Item E       │  slides in from below (animateAdd from predicted pos)
  │╌╌╌╌╌╌╌╌╌╌╌╌╌╌│
  └──────────────┘

  Item B: animateRemove (fade out)

7.3 Custom ItemAnimator

class SlideInAnimator : SimpleItemAnimator() {

    override fun animateAdd(holder: RecyclerView.ViewHolder): Boolean {
        holder.itemView.translationX = holder.itemView.width.toFloat()
        holder.itemView.alpha = 0f

        holder.itemView.animate()
            .translationX(0f)
            .alpha(1f)
            .setDuration(300)
            .setInterpolator(DecelerateInterpolator())
            .withEndAction { dispatchAddFinished(holder) }
            .start()
        return true
    }

    override fun animateRemove(holder: RecyclerView.ViewHolder): Boolean {
        holder.itemView.animate()
            .translationX(-holder.itemView.width.toFloat())
            .alpha(0f)
            .setDuration(250)
            .withEndAction { dispatchRemoveFinished(holder) }
            .start()
        return true
    }

    // Must implement — return false if no animation needed
    override fun animateMove(
        holder: RecyclerView.ViewHolder, fromX: Int, fromY: Int, toX: Int, toY: Int
    ): Boolean {
        // ... translate animation from (fromX,fromY) to (toX,toY) ...
        dispatchMoveFinished(holder)
        return false
    }

    override fun animateChange(
        oldHolder: RecyclerView.ViewHolder, newHolder: RecyclerView.ViewHolder,
        fromLeft: Int, fromTop: Int, toLeft: Int, toTop: Int
    ): Boolean {
        dispatchChangeFinished(oldHolder, true)
        dispatchChangeFinished(newHolder, false)
        return false
    }

    override fun runPendingAnimations() { /* batch animations if needed */ }
    override fun endAnimation(item: RecyclerView.ViewHolder) { item.itemView.animate().cancel() }
    override fun endAnimations() { /* cancel all */ }
    override fun isRunning() = false
}
Critical

You must call dispatchAddFinished(), dispatchRemoveFinished(), etc. when each animation completes. Failing to do so leaves RecyclerView in a broken state where it thinks animations are still running, blocking further layout passes and causing visual glitches.

7.4 Performance: Disabling Change Animations

Change animations create a second ViewHolder for the cross-fade, doubling memory for changed items. For frequently-updating lists (e.g., stock tickers, chat timestamps), disable them:

(recyclerView.itemAnimator as? SimpleItemAnimator)?.supportsChangeAnimations = false

8. ConcatAdapter

ConcatAdapter (formerly MergeAdapter) combines multiple adapters into a single RecyclerView. Each sub-adapter manages its own data and ViewHolder types independently.

8.1 Architecture

  ConcatAdapter — Position Mapping
  ═════════════════════════════════

  Global Position:  0    1    2    3    4    5    6    7

9. Prefetch and Performance

9.1 GapWorker — Prefetch During Idle

GapWorker is an internal class that prefetches ViewHolders during the idle phase between frames. When the user is scrolling, GapWorker predicts which items will be needed next and creates/binds them ahead of time, spreading the work across multiple frames to avoid jank.

  GapWorker Prefetch Timeline
  ════════════════════════════

  Frame N                    Frame N+1                  Frame N+2
  ┌────────────────────────┐ ┌────────────────────────┐ ┌──────────
  │ Input │ Layout │ Draw  │ │ Input │ Layout │ Draw  │ │ ...
  └───────┴────────┴───────┘ └───────┴────────┴───────┘ └──────────
                        │ idle │                   │ idle │
                        └──┬───┘                   └──┬───┘
                           │                          │
                    GapWorker creates            GapWorker binds
                    VH for pos N+5              VH for pos N+6

  Without prefetch: create + bind happens during Frame N+1's layout → jank
  With prefetch: work is spread across idle periods → smooth 60fps
// Prefetch is enabled by default
recyclerView.setItemPrefetchEnabled(true)

// For nested RecyclerViews: tell the outer LM how many inner items to prefetch
(innerRecyclerView.layoutManager as LinearLayoutManager)
    .initialPrefetchItemCount = 4  // prefetch 4 items of inner RV

9.3 setHasFixedSize — What It Actually Does

When setHasFixedSize(true) is set, adapter change notifications (notifyItemInserted(), etc.) call recyclerView.invalidate() instead of requestLayout(). This skips the full measure/layout pass on the RecyclerView itself (it already knows its own size won't change) and only re-lays out the children. For a RecyclerView that fills its parent with match_parent, this is always safe and always a win.

Staff Insight

The most impactful RecyclerView performance optimization at scale isn't any single API call — it's reducing onBindViewHolder() cost. In production, the bind method is called hundreds of times during a scroll. Every allocation, Spannable creation, date format, or image decode in onBindViewHolder() compounds. The staff-level approach: pre-compute display strings in the ViewModel/UseCase layer, use pre-sized image URLs, and keep onBindViewHolder() to pure property assignment. Profile with Systrace/Perfetto — look for repeated RV onBindView slices exceeding 2ms.

10. Staff-Level Corner Cases

10.1 The Layout-Time Mutation Crash

// CRASH: "Cannot call this method while RecyclerView is computing a layout or scrolling"
override fun onBindViewHolder(holder: VH, position: Int) {
    if (shouldRefresh) {
        notifyDataSetChanged()  // ← IllegalStateException
    }
}

RecyclerView sets a flag during layout/scroll that blocks any adapter mutation. If you need to trigger a data refresh from within a bind, post it:

holder.itemView.post { notifyDataSetChanged() }

10.2 RecyclerView Inside NestedScrollView — The Recycling Killer

Placing a RecyclerView with wrap_content height inside a NestedScrollView causes the LayoutManager to measure all items at once (the available space is effectively infinite). This defeats the entire purpose of recycling — all items are created and bound simultaneously.

<!-- ANTI-PATTERN: measures ALL items, no recycling -->
<NestedScrollView>
    <RecyclerView
        android:layout_height="wrap_content" />
</NestedScrollView>

<!-- FIX: give RecyclerView a fixed height or use ConcatAdapter -->
<RecyclerView
    android:layout_height="match_parent"
    android:nestedScrollingEnabled="true" />
Production Impact

This anti-pattern is extremely common and causes OOM crashes on lists with 1000+ items. In interviews, mentioning this immediately signals production experience. The fix is to use ConcatAdapter to combine the "above-the-fold" content with the list, giving RecyclerView full control of the scroll.

10.3 setIsRecyclable — Pinning ViewHolders

// Prevent recycling during an ongoing animation
holder.setIsRecyclable(false)
animateExpansion(holder.itemView) {
    holder.setIsRecyclable(true)  // re-enable after animation
}

Calls are ref-counted: if you call setIsRecyclable(false) twice, you must call setIsRecyclable(true) twice to re-enable recycling.

10.4 ItemTouchHelper — Swipe and Drag

val touchHelper = ItemTouchHelper(object : ItemTouchHelper.SimpleCallback(
    ItemTouchHelper.UP or ItemTouchHelper.DOWN,  // drag directions
    ItemTouchHelper.LEFT or ItemTouchHelper.RIGHT  // swipe directions
) {
    override fun onMove(
        rv: RecyclerView, source: RecyclerView.ViewHolder, target: RecyclerView.ViewHolder
    ): Boolean {
        adapter.moveItem(source.bindingAdapterPosition, target.bindingAdapterPosition)
        return true
    }

    override fun onSwiped(viewHolder: RecyclerView.ViewHolder, direction: Int) {
        adapter.removeItem(viewHolder.bindingAdapterPosition)
    }
})
touchHelper.attachToRecyclerView(recyclerView)

10.6 Memory Leak — ViewHolder Reference Escape

// LEAK: storing VH reference outside the adapter
class MyActivity : AppCompatActivity() {
    private var cachedHolder: RecyclerView.ViewHolder? = null  // ← LEAK

    // VH → itemView → RecyclerView → Activity
    // The VH is never released because the Activity holds it
}

ViewHolders should only be referenced by the Adapter and the Recycler. Any external reference creates a chain: ViewHolder → itemView → RecyclerView (parent) → Activity context.

Staff Insight

At Google scale, the biggest RecyclerView production issues are: (1) wrap_content inside scrolling parents defeating recycling, (2) submitList() with mutated-in-place lists causing silent no-ops, (3) onBindViewHolder() doing expensive work (date formatting, string concatenation, image decoding) that should be pre-computed in the domain layer, and (4) not sharing RecycledViewPool across ViewPager2 pages, causing redundant ViewHolder inflation. In system design interviews, propose: immutable list emissions from the ViewModel, pre-computed display models, shared pools for homogeneous nested lists, and Systrace profiling of bind times as your RecyclerView performance strategy.
