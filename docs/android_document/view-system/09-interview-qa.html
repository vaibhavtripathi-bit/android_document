09 — Interview Q&A — Android View System

65 Staff Engineer questions covering View hierarchy internals, measure & layout, drawing & Canvas, custom Views, touch & input dispatch, animations, RecyclerView, and performance & system design

View Hierarchy & Internals (Q1–Q10)

Q1. Explain the role of ViewRootImpl — how does it connect the View tree to the Window system?

ViewRootImpl is the bridge between the View tree and the WindowManagerService (WMS). Each window in an app has exactly one ViewRootImpl, created when WindowManagerGlobal.addView() is called. It holds the IWindowSession Binder proxy to WMS and the Surface that the View tree draws into. ViewRootImpl implements ViewParent and sits above the DecorView — when any View calls requestLayout() or invalidate(), the call propagates up through parents until it reaches ViewRootImpl, which schedules a traversal via Choreographer.postCallback(CALLBACK_TRAVERSAL). On the next VSYNC, performTraversals() runs the measure → layout → draw pipeline. ViewRootImpl also receives input events from the InputDispatcher (via an InputChannel socketpair), dispatches them through the View tree, manages window insets, handles configuration changes, and communicates window state (focus, visibility) back to WMS. It is not a View itself — it's an internal implementation class that apps never directly interact with, but it orchestrates everything the View tree does. In AOSP, ViewRootImpl is ~10,000 lines — it manages the ThreadedRenderer for hardware-accelerated drawing, the InputStage pipeline for event processing, accessibility event dispatch, and the SurfaceHolder callback for surface lifecycle. There's exactly one ViewRootImpl per window, but an Activity with dialogs can have multiple windows and thus multiple ViewRootImpl instances.

Q2. Walk through performTraversals() — what are the three phases and when is each triggered?

performTraversals() is the heart of the View rendering pipeline, called once per VSYNC frame when work is pending. It runs three phases in strict order: (1) Measure — performMeasure() calls mView.measure() on the DecorView with root MeasureSpecs derived from the window size (typically EXACTLY for both dimensions, matching the window frame from WMS). Each ViewGroup recursively measures its children. The result: every View has measuredWidth/measuredHeight set. (2) Layout — performLayout() calls mView.layout(), which assigns each View its final position (mLeft, mTop, mRight, mBottom) relative to its parent. (3) Draw — performDraw() calls draw(), which either uses the ThreadedRenderer (hardware-accelerated path: records DisplayList/RenderNode ops) or falls back to software Canvas rendering. Measure is triggered when any View calls requestLayout() or the window size changes. Layout always follows measure. Draw is triggered by invalidate() or when measure/layout changes require visual updates. ViewRootImpl coalesces multiple requests — if both requestLayout() and invalidate() are called before the next VSYNC, all three phases run in a single traversal. Between phases, performTraversals() also handles: applying window insets (dispatchApplyWindowInsets()), computing the visible rect, notifying ViewTreeObserver listeners, and checking if the Surface needs to be created or resized. The entire method is ~800 lines in AOSP — one of the most complex methods in the framework.

Q3. What is View.AttachInfo and why is it important?

View.AttachInfo is an internal data structure created by ViewRootImpl and shared across every View in the tree. It's assigned to each View during dispatchAttachedToWindow() and nulled out on detach.

AttachInfo carries critical state: the IWindowSession for communicating with WMS, the IBinder window token, the root View, the Handler for the UI thread, the ViewTreeObserver, the Surface, hardware rendering state, window insets, and the mScrollContainers list (Views that need to shrink when the soft keyboard appears).

When you call View.post(Runnable), the Runnable is posted to AttachInfo's Handler — if AttachInfo is null (View not yet attached), the Runnable is queued in a pending list (mRunQueue) and flushed when the View attaches. AttachInfo also stores the mHardwareAccelerated flag, which determines whether Canvas operations go through the GPU pipeline. Without AttachInfo, a View cannot draw, receive input, or communicate with the window system — it's the lifeline connecting each View to the infrastructure.

Q4. Explain the difference between invalidate() and requestLayout() — when do you use each?

invalidate() marks a View as needing to be redrawn without changing its size or position. It sets the PFLAG_DIRTY flag and propagates a dirty rect up to ViewRootImpl, which schedules a draw pass. In hardware-accelerated mode, only the invalidated View's RenderNode is re-recorded — parent RenderNodes are untouched, making it very efficient. requestLayout() marks a View as needing re-measurement and re-layout. It sets PFLAG_FORCE_LAYOUT and PFLAG_INVALIDATED, then walks up the parent chain calling requestLayout() on each parent until reaching ViewRootImpl, which schedules a full measure → layout → draw traversal. Use invalidate() when only visual appearance changes (color, alpha, drawing content) but dimensions stay the same. Use requestLayout() when size or position might change (text content changed, visibility toggled, LayoutParams modified). Calling requestLayout() when only a redraw is needed wastes CPU — the measure/layout phases run unnecessarily. Conversely, calling only invalidate() when size changed means the View draws with stale dimensions, causing visual glitches. There's also invalidate(Rect) for partial invalidation — in software rendering, only the dirty rect region is redrawn. In hardware rendering, the rect is ignored (the entire RenderNode is re-recorded), but the framework still uses it for damage tracking. A common mistake: calling both requestLayout() and invalidate() when only one is needed — requestLayout() already implies a draw pass, so the invalidate() is redundant.

Q5. How does the View lifecycle work from inflation to detachment? List all key callbacks in order.

The full lifecycle: (1) Constructor — View(Context, AttributeSet) called by the LayoutInflater. Custom attributes are read here via obtainStyledAttributes(). (2) onFinishInflate() — called after all children are inflated (for ViewGroups, this means child Views are available — safe to call findViewById()). (3) onAttachedToWindow() — the View is added to a window; AttachInfo is assigned, the mRunQueue is flushed, hardware rendering state is initialized, and the View registers with its parent's ViewTreeObserver. (4) onMeasure() — called during the measure pass; may be called multiple times per frame if parent needs multiple passes (RelativeLayout, LinearLayout with weights). (5) onSizeChanged(w, h, oldW, oldH) — called when measured dimensions actually change; ideal for precomputing drawing bounds. (6) onLayout(changed, left, top, right, bottom) — position assigned by parent; ViewGroups position their children here. (7) onDraw(Canvas) — the View renders itself to the Canvas. (8) onWindowFocusChanged(hasFocus) — window gains/loses focus; the first true callback is a reliable signal that the View is fully visible and measured. (9) onDetachedFromWindow() — the View is removed; clean up animations, handlers, listeners, and release resources here. Between steps 4–7, the View may go through many measure/layout/draw cycles. onVisibilityChanged() and onWindowVisibilityChanged() fire when visibility state changes. onSaveInstanceState()/onRestoreInstanceState() fire during configuration changes if the View has an ID. A subtle point: onAttachedToWindow() may be called multiple times if the View is removed and re-added (e.g., RecyclerView recycling), so avoid one-time initialization there — use the constructor or onFinishInflate() instead.

Q6. What is DecorView and how does setContentView() connect your layout to the window?

DecorView is the root View of every Activity window — a FrameLayout subclass that contains the system chrome (status bar background, navigation bar background, action bar) and a FrameLayout with ID android.R.id.content. When you call setContentView(R.layout.activity_main), the call goes to PhoneWindow.setContentView(), which: (1) creates the DecorView if it doesn't exist via installDecor(), selecting a system layout based on window features (no title, action bar, etc.) — the layout is chosen from com.android.internal.R.layout.screen_* templates, (2) inflates your layout XML into the content FrameLayout via LayoutInflater.inflate(layoutResID, mContentParent). The DecorView is not yet attached to the window at this point. Attachment happens later in ActivityThread.handleResumeActivity(), which calls WindowManagerImpl.addView(decor, layoutParams) → WindowManagerGlobal.addView() → creates ViewRootImpl → calls ViewRootImpl.setView(decor). This is when the IPC to WMS happens (mWindowSession.addToDisplayAsUser()), the Surface is created, and the first performTraversals() is scheduled. Your layout only becomes visible after this full chain completes. This explains why getWidth() returns 0 in onCreate() — the View tree hasn't been measured yet because ViewRootImpl doesn't exist until onResume(). The DecorView also handles system gesture insets, edge-to-edge rendering (when WindowCompat.setDecorFitsSystemWindows(window, false) is called), and dispatches WindowInsets down the tree.

Q7. How does ViewTreeObserver work? What are the common listeners and their use cases?

ViewTreeObserver is a notification hub attached to the View tree (stored in AttachInfo) that fires callbacks at specific points in the traversal pipeline. Key listeners:

OnPreDrawListener — fires after measure/layout but before draw; return false to cancel the draw (useful for deferring until data is ready, or for shared element transitions that need to postpone the first frame). OnGlobalLayoutListener — fires after layout completes; use it to read final dimensions (getWidth()/getHeight()) that aren't available in onCreate(). OnGlobalFocusChangeListener — fires when focus moves between Views. OnScrollChangedListener — fires when any View in the tree scrolls. OnWindowAttachListener — fires on attach/detach.

Internally, ViewRootImpl calls mTreeObserver.dispatchOnPreDraw() inside performDraw() and dispatchOnGlobalLayout() after performLayout(). A common pitfall: forgetting to remove OnGlobalLayoutListener after use — it fires on every layout pass, causing unnecessary work. Use doOnLayout {} or doOnPreDraw {} Kotlin extensions from AndroidX core-ktx, which auto-remove after the first invocation.

Q8. What happens when you call View.post(Runnable) before the View is attached to a window?

When a View is attached, View.post() delegates to mAttachInfo.mHandler.post(runnable), posting the Runnable to the UI thread's Handler. But before attachment, mAttachInfo is null. In this case, the View stores the Runnable in a HandlerActionQueue (mRunQueue) — an internal list of deferred (Runnable, delay) pairs.

When the View is later attached via dispatchAttachedToWindow(), the queue is flushed: all pending Runnables are posted to the AttachInfo Handler. This means View.post() is safe to call in constructors or onFinishInflate() — the Runnable will execute after the first traversal.

However, if the View is never attached (e.g., you inflate it but never add it to a parent), the Runnables are never executed and silently leak. Prior to API 24, there was a subtle bug: View.post() on a detached View would use ViewRootImpl.getRunQueue() (a static queue), which could execute the Runnable in the context of a different ViewRootImpl. This was fixed to use the per-View mRunQueue instead.

Q9. Explain how hardware-accelerated invalidation differs from software invalidation.

In software rendering, invalidate() marks a dirty rectangle that propagates up the tree. During draw, the software Canvas clips to the union of all dirty rects, and every View whose bounds intersect the dirty region must redraw — even if only a leaf View changed. The entire bitmap backing the window is partially redrawn.

In hardware-accelerated rendering, each View has a RenderNode containing a DisplayList (a sequence of recorded Canvas operations). When invalidate() is called, only the invalidated View's RenderNode is marked dirty and re-recorded during the next draw pass. Parent RenderNodes simply reference child RenderNodes — they don't need to re-record.

The ThreadedRenderer walks the RenderNode tree, finds dirty nodes, re-records only those, then sends the updated DisplayList to the RenderThread for GPU execution. This means invalidating a single leaf View is O(1) in the hardware path (re-record one RenderNode) vs O(depth) in software (redraw all ancestors intersecting the dirty rect). Property changes like setAlpha(), setTranslationX() are even cheaper — they modify RenderNode properties without re-recording the DisplayList at all.

Q10. How does the View system handle multiple requestLayout() calls in the same frame?

ViewRootImpl uses a flag-based coalescing mechanism. When the first requestLayout() reaches ViewRootImpl, it sets mLayoutRequested = true and posts a traversal callback to Choreographer for the next VSYNC. Subsequent requestLayout() calls from other Views in the same frame also propagate up and set PFLAG_FORCE_LAYOUT on each View in the chain, but ViewRootImpl sees mLayoutRequested is already true and doesn't post a duplicate Choreographer callback.

When performTraversals() runs on the next VSYNC, it performs a single measure pass starting from the root — every View with PFLAG_FORCE_LAYOUT set will be re-measured. After measure, a single layout pass runs. The flags are cleared after each phase.

This batching is critical for performance: if you change text on 10 TextViews in a loop, you get one traversal, not 10. However, calling requestLayout() inside onLayout() or onMeasure() is dangerous — it can trigger an infinite loop or cause the traversal to restart. ViewRootImpl has a safety check: if layout is requested during layout, it defers to the next frame and logs a warning ("requestLayout() improperly called during layout").

Measure & Layout (Q11–Q20)

Q11. Explain MeasureSpec — what are the three modes and how do parent LayoutParams map to child specs?

MeasureSpec is a 32-bit int packing a mode (top 2 bits) and a size (lower 30 bits), created via MeasureSpec.makeMeasureSpec(size, mode) and unpacked with getMode()/getSize(). The three modes: EXACTLY (0x40000000) — the parent has determined the exact size; the child must be this size (maps from match_parent when parent spec is EXACTLY, or from a fixed dp value). AT_MOST (0x80000000) — the child can be up to this size but no larger (maps from wrap_content, or match_parent when parent spec is AT_MOST). UNSPECIFIED (0x00000000) — no constraints; the child can be any size (used by ScrollView to let children be taller than the viewport, and during the first pass of some ViewGroups). The mapping logic lives in ViewGroup.getChildMeasureSpec(parentSpec, padding, childDimension): if childDimension is a fixed value (e.g., 200dp), the result is always EXACTLY with that size, regardless of parent mode. If childDimension is MATCH_PARENT, the child gets the parent's remaining size with the parent's mode. If childDimension is WRAP_CONTENT, the child gets AT_MOST with the parent's remaining size (or UNSPECIFIED if parent is UNSPECIFIED). The bit-packing design avoids object allocation — millions of MeasureSpec values are created per frame across the View tree, and using a primitive int instead of an object eliminates GC pressure entirely. Understanding this mapping is essential for implementing correct custom onMeasure().

Q12. What is the contract for onMeasure()? What happens if you don't call setMeasuredDimension()?

The contract: onMeasure(int widthMeasureSpec, int heightMeasureSpec) must call setMeasuredDimension(width, height) before returning. The measured dimensions should respect the MeasureSpec constraints — an EXACTLY spec means you must use that exact size, AT_MOST means your size must not exceed the spec size.

If you fail to call setMeasuredDimension(), the default View.onMeasure() implementation calls it with getDefaultSize(), but if you override onMeasure() and forget the call entirely, measure() throws an IllegalStateException: "onMeasure() did not set the measured dimension." This crash is intentional — without measured dimensions, the layout pass would use garbage values.

The helper resolveSize(desiredSize, measureSpec) (or resolveSizeAndState()) handles the spec logic: returns specSize for EXACTLY, min(desiredSize, specSize) for AT_MOST, and desiredSize for UNSPECIFIED. resolveSizeAndState() additionally sets the MEASURED_STATE_TOO_SMALL flag when the View is forced smaller than desired — parents can check this to decide whether to allocate more space. For ViewGroups, the contract extends further: you must measure all children by calling child.measure() with appropriate child MeasureSpecs, then compute your own size based on children's measured dimensions.

Q13. Why does RelativeLayout measure children twice? How does this cause exponential blowup with nesting?

RelativeLayout performs two measure passes per onMeasure() call. First pass: measures children with horizontal constraints resolved (left/right rules) but vertical dimensions set to UNSPECIFIED or a preliminary value. Second pass: measures children again with vertical constraints resolved, using the horizontal results from pass one. This is necessary because relative constraints create circular dependencies — a child's width might depend on a sibling's position, which depends on another child's height.

The problem: if you nest RelativeLayouts, each level doubles the measure calls. A RelativeLayout inside a RelativeLayout means the inner one's onMeasure() (2 passes) is called twice by the outer one = 4 child measurements. Three levels deep = 8 measurements. N levels = 2^N measurements per child — exponential blowup. With a tree of 20 Views at 4 levels of nesting, you can easily hit hundreds of measure calls per frame.

LinearLayout with layout_weight has the same double-measure behavior. This is why ConstraintLayout was created — it resolves all constraints in a single pass using a Cassowary-like solver.

Q14. How does ConstraintLayout solve the multi-pass measurement problem?

ConstraintLayout uses a linear constraint solver (based on the Cassowary algorithm, implemented in the androidx.constraintlayout.solver package) that resolves all horizontal and vertical constraints in a single measure/layout pass. Instead of iteratively measuring children to resolve dependencies (like RelativeLayout), it builds a system of linear equations from the declared constraints and solves them simultaneously using the simplex method. The solver operates on "anchors" (left, right, top, bottom, baseline) and produces exact positions for all children in O(n) time for typical layouts. This means nesting ConstraintLayouts doesn't cause exponential blowup — each level is still one pass. Additionally, ConstraintLayout is a flat layout: most UIs that would require 3-4 levels of nested LinearLayout/RelativeLayout can be expressed as a single ConstraintLayout with constraints, eliminating nesting entirely. The trade-off: the solver has higher constant-factor overhead than a simple LinearLayout for trivial cases (a vertical stack of 3 items is faster with LinearLayout). But for complex layouts with 10+ Views and cross-dependencies, ConstraintLayout is dramatically faster. It also supports advanced features — chains (spread, packed, weighted), barriers (virtual guidelines that track the edge of a set of Views), guidelines (percentage or fixed-offset reference lines), and Flow (a virtual layout helper that wraps children like a FlowLayout without nesting) — that would require custom ViewGroups in the old system. ConstraintLayout 2.0+ added MotionLayout for declarative animations between constraint sets.

Q15. What is the difference between getMeasuredWidth() and getWidth()? When can they differ?

getMeasuredWidth() returns the width set during the measure phase — the value passed to setMeasuredDimension(). It's available after onMeasure() completes. getWidth() returns mRight - mLeft, the actual width assigned during the layout phase. It's only valid after onLayout() completes.

They differ when a parent overrides the child's measured size during layout. For example, a parent might measure a child as 300px wide but then call child.layout(0, 0, 250, height), making getWidth() return 250 while getMeasuredWidth() still returns 300. This happens in practice with TableLayout (which stretches/shrinks columns), LinearLayout with weight (the weighted size may differ from the initial measurement), and custom ViewGroups that adjust child positions.

The rule: use getMeasuredWidth() inside onLayout() when deciding how to position children. Use getWidth() inside onDraw() or any code that runs after layout. Accessing getWidth() before layout returns 0, which is a common bug in onCreate().

Q16. Explain getChildMeasureSpec() — how does it combine parent spec and child LayoutParams?

ViewGroup.getChildMeasureSpec(int parentSpec, int padding, int childDimension) is the core function that translates a parent's constraints into a child's MeasureSpec. It computes availableSize = parentSpecSize - padding (accounting for parent padding and child margins), then applies a 3×3 matrix of (parentMode, childDimension) combinations:

When child has a fixed size (e.g., 100dp): result is always EXACTLY + that size, clamped to 0 minimum — the parent mode doesn't matter. When child is MATCH_PARENT: if parent is EXACTLY → child gets EXACTLY + availableSize; if parent is AT_MOST → child gets AT_MOST + availableSize; if parent is UNSPECIFIED → child gets UNSPECIFIED + 0. When child is WRAP_CONTENT: if parent is EXACTLY or AT_MOST → child gets AT_MOST + availableSize; if parent is UNSPECIFIED → child gets UNSPECIFIED + 0.

The key insight: WRAP_CONTENT children always get AT_MOST (not UNSPECIFIED) when the parent has a known size — this is why a wrap_content View inside a 300dp parent won't exceed 300dp. Custom ViewGroups that need different behavior (e.g., ScrollView giving children UNSPECIFIED height) must override this logic. The AOSP implementation is a single switch statement — deceptively simple but the source of most measurement bugs when developers don't understand the mapping.

Q17. What happens when you put a RecyclerView with wrap_content height inside a NestedScrollView?

This is a classic anti-pattern. NestedScrollView measures its child with UNSPECIFIED height (like ScrollView), telling the child "be as tall as you want." RecyclerView with wrap_content responds by measuring and laying out all items to determine its total height — defeating the entire purpose of recycling. If you have 10,000 items, all 10,000 ViewHolders are created and bound, consuming massive memory and causing multi-second jank. The RecyclerView becomes a very expensive LinearLayout. Internally, LinearLayoutManager.onMeasure() detects the UNSPECIFIED spec and calls fill() without a scroll limit, filling until all items are laid out. The fix: never use wrap_content RecyclerView inside a scrolling parent. Instead, use a single RecyclerView with multiple view types (headers, content sections, etc.) via ConcatAdapter or a custom adapter. Alternatively, set a fixed height on the RecyclerView, or use ConstraintLayout with 0dp height and constraints. If you absolutely must nest, RecyclerView.setNestedScrollingEnabled(false) fixes scroll coordination but doesn't prevent the full-layout problem. The same issue occurs with wrap_content RecyclerView inside a horizontal ScrollView — all columns are laid out. In Compose, LazyColumn inside a verticalScroll modifier throws an IllegalStateException at runtime, making this mistake impossible.

Q18. How does the measure cache (mMeasureCache) work? When is it invalidated?

Each View maintains a LongSparseLongArray called mMeasureCache that maps MeasureSpec pairs (width + height packed into a long key) to measured dimension pairs (measuredWidth + measuredHeight packed into a long value).

In View.measure(), before calling onMeasure(), the framework checks: if PFLAG_FORCE_LAYOUT is not set and the cache contains an entry for the current MeasureSpec pair, it restores the cached dimensions via setMeasuredDimensionRaw() and skips onMeasure() entirely. This is critical for ViewGroups like RelativeLayout that measure children multiple times — the second pass often hits the cache.

The cache is invalidated when: (1) requestLayout() is called, which sets PFLAG_FORCE_LAYOUT, bypassing the cache check. (2) forceLayout() is called, which sets the same flag. (3) The View is detached and reattached. The cache is not cleared on invalidate — since invalidate only affects drawing, not measurement. A subtle detail: the cache key includes the full MeasureSpec (mode + size), so a View measured with AT_MOST/300 and then AT_MOST/300 again hits the cache, but AT_MOST/300 followed by AT_MOST/250 does not.

Q19. What is forceLayout() and how does it differ from requestLayout()?

forceLayout() sets PFLAG_FORCE_LAYOUT and PFLAG_INVALIDATED on the View, marking it as needing re-measurement. But it does not propagate up the parent chain and does not schedule a traversal. It's a local flag-set only.

requestLayout() does the same flag-setting but also walks up the parent chain calling requestLayout() on each parent until reaching ViewRootImpl, which schedules a Choreographer traversal.

Use case for forceLayout(): when a parent ViewGroup knows it's about to re-measure children anyway (because it already called requestLayout() on itself), it can call forceLayout() on specific children to ensure they're re-measured without triggering redundant parent-chain walks. LinearLayout uses this internally — when its own onMeasure() runs, it calls forceLayout() on children that need the weight pass. Calling forceLayout() alone without a parent triggering a traversal does nothing visible — the flags sit there until the next traversal happens for some other reason. It's an optimization tool for ViewGroup implementers, not something app developers typically call directly.

Q20. Design the onMeasure() for a custom View that maintains a 16:9 aspect ratio regardless of parent constraints.

The key challenge is respecting MeasureSpec constraints while enforcing the ratio. The strategy: determine the maximum allowed width and height from the specs, then pick the largest 16:9 rectangle that fits:

override fun onMeasure(widthSpec: Int, heightSpec: Int) {
    val widthMode = MeasureSpec.getMode(widthSpec)
    val widthSize = MeasureSpec.getSize(widthSpec)
    val heightMode = MeasureSpec.getMode(heightSpec)
    val heightSize = MeasureSpec.getSize(heightSpec)

    val width: Int
    val height: Int

    when {
        widthMode == MeasureSpec.EXACTLY && heightMode == MeasureSpec.EXACTLY -> {
            width = widthSize
            height = heightSize // forced; ratio may break
        }
        widthMode == MeasureSpec.EXACTLY -> {
            width = widthSize
            height = (widthSize * 9f / 16f).toInt()
        }
        heightMode == MeasureSpec.EXACTLY -> {
            height = heightSize
            width = (heightSize * 16f / 9f).toInt()
        }
        else -> {
            val w = if (widthSize > 0) widthSize else 1920
            val hFromW = (w * 9f / 16f).toInt()
            if (heightSize > 0 && hFromW > heightSize) {
                height = heightSize
                width = (heightSize * 16f / 9f).toInt()
            } else {
                width = w; height = hFromW
            }
        }
    }
    setMeasuredDimension(width, height)
}

When both dimensions are EXACTLY, the parent forces the size and the ratio may not hold — this is correct behavior since the parent's constraints take priority. For AT_MOST/UNSPECIFIED combinations, we compute the ratio-preserving dimension from whichever axis is more constrained. This pattern generalizes to any aspect ratio.

Drawing & Canvas (Q21–Q30)

Q21. Walk through the 6 steps of View.draw() — what happens in each step?

The View.draw(Canvas) method executes 6 ordered steps: (1) Draw background — drawBackground(canvas) renders the View's background Drawable at the correct bounds, accounting for background padding and scroll offsets. The background is drawn at (mScrollX + mLeft, mScrollY + mTop). (2) Save canvas layers for fading edges — if the View has fading edges enabled (rare in modern code), a saveLayer() is called for the gradient effect. (3) Draw content — onDraw(canvas) is called; this is where your custom drawing logic goes. (4) Draw children — dispatchDraw(canvas) is called; ViewGroups override this to iterate children and call child.draw() with appropriate transformations (translation, scale, rotation from the child's RenderNode). For leaf Views, this is a no-op. (5) Draw fading edges — if saved in step 2, the fade gradients are drawn and the layer is restored. (6) Draw decorations — onDrawForeground(canvas) renders foreground Drawable, scrollbars, and any foreground overlay. The framework skips steps 2 and 5 when fading edges are disabled (the common case), making the effective pipeline: background → onDraw → dispatchDraw → foreground. In hardware-accelerated mode, these steps are recorded into a RenderNode's DisplayList rather than executed immediately on a bitmap. An important optimization: View.draw() checks the PFLAG_SKIP_DRAW flag (set by setWillNotDraw(true), which is the default for ViewGroups) — when set, it skips steps 1, 3, and 6, only calling dispatchDraw(). This avoids unnecessary background and foreground rendering for container Views that don't draw anything themselves.

Q22. Explain the difference between software Canvas and hardware RecordingCanvas.

A software Canvas wraps a Bitmap — every draw call (drawRect(), drawText(), drawPath()) immediately rasterizes pixels into the bitmap's memory using Skia's CPU renderer. The bitmap is then composited to the Surface by SurfaceFlinger. This is the pre-API 14 path and the fallback when hardware acceleration is disabled.

A RecordingCanvas (hardware path) doesn't rasterize anything — it records draw operations as a sequence of DisplayListOps stored in a RenderNode. Operations like drawRect() become "record a rect op with these coordinates and paint." The recorded DisplayList is later replayed by the RenderThread using Skia's GPU backend (OpenGL ES or Vulkan).

Benefits of recording: (1) the UI thread only records, which is fast — actual rendering happens on the RenderThread in parallel. (2) Only dirty RenderNodes need re-recording; unchanged subtrees are replayed from cache. (3) RenderNode property changes (alpha, translation, scale) don't require re-recording. The trade-off: some Canvas operations aren't supported in hardware mode (drawBitmapMesh() with colors, certain PathEffects), and saveLayer() allocates an offscreen GPU texture.

Q23. What is a RenderNode? How does it enable efficient property animations?

A RenderNode is a GPU-side representation of a View's drawing commands plus transformation properties. Each View has one RenderNode containing: (1) a DisplayList — the recorded sequence of Canvas operations from onDraw(), and (2) properties — translationX/Y/Z, scaleX/Y, rotationX/Y/Z, alpha, pivotX/Y, clip bounds, elevation, and camera distance.

When you call View.setTranslationX(100f), the framework calls mRenderNode.setTranslationX(100f) — this modifies a float on the RenderNode without touching the DisplayList. On the next frame, the RenderThread reads the updated property and applies it as a GPU transformation matrix when rendering the DisplayList. No onDraw() is called, no DisplayList is re-recorded, and the UI thread does minimal work.

This is why ViewPropertyAnimator and ObjectAnimator targeting View properties (alpha, translationX, scaleX, etc.) are so efficient — they're modifying GPU-side floats at 60fps. In contrast, animating a custom property that triggers invalidate() requires re-recording the DisplayList every frame. RenderNodes also form a tree mirroring the View tree, enabling the RenderThread to traverse and render the entire hierarchy independently of the UI thread. Since Android 10 (API 29), RenderNode is a public API — you can create standalone RenderNodes in custom Views for efficient sub-component caching without the overhead of child Views.

Q24. Why is saveLayer() expensive? When should you use it?

Canvas.saveLayer() allocates an offscreen buffer (a temporary texture in GPU memory) and redirects all subsequent draw operations into it until restore(). On restore, the offscreen buffer is composited back onto the main Canvas using the specified Paint (which can include alpha, blend modes, or color filters).

The cost: (1) Memory allocation — a full-size RGBA texture for the layer bounds (a 1080×1920 layer = ~8MB of GPU memory). (2) GPU state switch — changing render targets is expensive on mobile GPUs (tile-based renderers must flush). (3) Extra compositing pass — the layer must be drawn back onto the parent. (4) Overdraw — everything in the layer is drawn twice (once into the layer, once compositing back).

Use saveLayer() when you need the entire subtree to be composited as a unit — the most common case is applying alpha to a group of overlapping Views (without a layer, individual alpha on overlapping children causes visible blending artifacts at overlap regions). Also needed for certain PorterDuff blend modes (e.g., DST_OUT for cutout effects) and ComposeShader operations. Avoid it in scroll paths or animations. The View.LAYER_TYPE_HARDWARE flag is a persistent version of saveLayer — use it for animations, then remove it.

Q25. Explain Porter-Duff compositing — how would you implement circular image masking?

Porter-Duff modes define how source (Src) and destination (Dst) pixels combine based on their alpha values. The 12 standard modes include: SRC_OVER (default — source drawn over destination), DST_IN (keep destination only where source has alpha), SRC_IN (keep source only where destination has alpha), CLEAR (erase both). For circular masking, use SRC_IN with a layer:

override fun onDraw(canvas: Canvas) {
    val save = canvas.saveLayer(0f, 0f, width.toFloat(), height.toFloat(), null)
    // Step 1: Draw the circle (becomes Dst)
    canvas.drawCircle(width / 2f, height / 2f, width / 2f, circlePaint)
    // Step 2: Draw bitmap with SRC_IN (keeps bitmap only inside circle)
    maskPaint.xfermode = PorterDuffXfermode(PorterDuff.Mode.SRC_IN)
    canvas.drawBitmap(bitmap, null, bounds, maskPaint)
    maskPaint.xfermode = null
    canvas.restoreToCount(save)
}

The saveLayer() is required because Porter-Duff modes operate on the layer's pixel buffer — without it, SRC_IN would interact with whatever was previously drawn on the Canvas (including the window background). In practice, prefer BitmapShader with a circle for better performance — it avoids the saveLayer cost entirely by using the shader to sample the bitmap within a circle path. Glide and Coil use this approach for circular image transformations.

Q26. What is the difference between draw(), onDraw(), and dispatchDraw()? When do you override each?

draw(Canvas) is the framework method that orchestrates the full 6-step drawing pipeline (background → onDraw → dispatchDraw → foreground). You almost never override this — doing so skips the framework's background/foreground/scrollbar rendering. onDraw(Canvas) is the callback for drawing the View's own content — override this in custom Views to draw shapes, text, bitmaps, etc. It's called in step 3 of draw(). dispatchDraw(Canvas) is called in step 4 — ViewGroups override this to draw their children. The default ViewGroup implementation iterates mChildren and calls drawChild() on each. Override dispatchDraw() in a custom ViewGroup when you need to draw something between or after children (e.g., dividers, overlays, clip paths that affect children). A common pattern: override dispatchDraw() to apply a clip path before calling super.dispatchDraw(), creating a clipped container. If you need to draw both custom content and children, override onDraw() for content behind children and dispatchDraw() (after super) for content in front of children. Never forget to call setWillNotDraw(false) in a custom ViewGroup if you override onDraw() — ViewGroups skip onDraw() by default as an optimization.

Q27. How do hardware layers work? When are they beneficial and when are they harmful?

Setting View.setLayerType(LAYER_TYPE_HARDWARE, paint) tells the framework to render the View's RenderNode into an offscreen GPU texture (FBO) and cache it. On subsequent frames, if the View's DisplayList hasn't changed, the cached texture is composited directly — no re-recording or re-rendering of the View's draw commands.

Beneficial: during animations that only change RenderNode properties (alpha, translation, scale, rotation). The View is rendered once into the layer, and the animation just moves/fades the cached texture — dramatically reducing GPU work. ViewPropertyAnimator automatically sets and removes hardware layers via withLayer(). Also useful for complex Views with expensive onDraw() that don't change often (e.g., a complex chart that updates once per minute).

Harmful: when the View's content changes frequently (e.g., during scroll, text input, or content animations). Each change invalidates the layer, forcing a full re-render into the texture plus the compositing cost — worse than no layer at all. Layers also consume GPU memory (RGBA texture at the View's pixel dimensions). A 1080×500 View = ~2MB. Too many layers can cause GPU memory pressure and force texture eviction. Always remove layers after animations complete: setLayerType(LAYER_TYPE_NONE, null).

Q28. What Canvas operations are NOT supported with hardware acceleration?

Hardware-accelerated RecordingCanvas supports most operations, but several are unsupported or behave differently:

drawBitmapMesh() with the colors array is not supported. drawPicture() is not hardware-accelerated. Certain Paint features: setMaskFilter() with EmbossMaskFilter is unsupported; setShadowLayer() only works on text (not shapes). Canvas.drawTextOnPath() has limited support. PathEffect — DashPathEffect works, but DiscretePathEffect and PathDashPathEffect may not render correctly. ComposeShader with nested ComposeShaders is unsupported. clipPath() with non-rectangular paths doesn't support anti-aliasing in older API levels.

When an unsupported operation is used, the framework either silently ignores it, falls back to software rendering for that View (if LAYER_TYPE_SOFTWARE is set), or renders incorrectly. You can check support at runtime with Canvas.isHardwareAccelerated(). The fix for unsupported operations: set setLayerType(LAYER_TYPE_SOFTWARE, null) on that specific View — this forces software rendering only for that View while the rest of the tree stays hardware-accelerated. Over time, each Android release has expanded hardware support — most of these limitations are only relevant on older devices (API < 28).

Q29. Explain how View.setTranslationX() can be animated without triggering onDraw().

When you call View.setTranslationX(value), the View stores the value and calls mRenderNode.setTranslationX(value). Crucially, it does not call invalidate() — it calls invalidateViewProperty(false, false), which sets the PFLAG_DRAW_ANIMATION flag and notifies ViewRootImpl that a RenderNode property changed.

On the next frame, the ThreadedRenderer detects the dirty RenderNode property and updates the GPU-side transformation matrix. The DisplayList (recorded draw commands) is untouched — the same cached drawing is simply rendered at a different position.

This works because RenderNode properties are separate from the DisplayList: the DisplayList says "draw a rect at (0,0,100,100)" and the RenderNode's translation matrix shifts the entire output by the translation amount during GPU compositing. The same mechanism applies to setAlpha(), setScaleX(), setRotation(), setElevation(), and setPivotX/Y(). This is why ViewPropertyAnimator is the most efficient animation API — it only modifies these RenderNode-backed properties. Animating a property that isn't backed by RenderNode (e.g., background color, text size, padding) requires invalidate() and DisplayList re-recording every frame.

Q30. How does text rendering work on Canvas? What is the difference between getTextBounds() and measureText()?

Text rendering on Canvas goes through Skia's text shaping pipeline: Canvas.drawText() → Skia's SkFont → HarfBuzz shaper (handles ligatures, kerning, complex scripts like Arabic/Devanagari) → glyph IDs → rasterized into a glyph cache (texture atlas on GPU) → composited at the specified position.

Paint.measureText(String) returns the advance width — the total horizontal distance the cursor moves after drawing the text, including side bearings (spacing). This is what you use for layout calculations and text positioning.

Paint.getTextBounds(String, start, end, Rect) returns the tight bounding box of the actual glyph pixels — the smallest rectangle that encloses all drawn pixels. This is typically smaller than measureText() because it excludes leading/trailing side bearings. For example, the letter "T" has significant side bearings; measureText("T") might return 20px while getTextBounds() gives a rect 16px wide.

Use measureText() for positioning text sequentially (it accounts for proper spacing). Use getTextBounds() for centering text visually within a shape (it gives the actual pixel extent). Paint.getFontMetrics() gives ascent/descent/leading for vertical positioning — use -ascent to get the baseline offset from the top of the text.

Custom Views (Q31–Q38)

Q31. When should you extend View vs ViewGroup vs an existing View? Give examples for each.

Extend View when you need fully custom drawing with no children — a chart, a circular progress indicator, a signature pad, a custom gauge. You override onMeasure() and onDraw() and handle all rendering via Canvas. This is the lightest-weight option.

Extend ViewGroup when you need custom child arrangement logic — a flow layout that wraps children, a radial menu, a staggered grid. You override onMeasure() (measuring all children with appropriate MeasureSpecs), onLayout() (positioning children), and optionally onInterceptTouchEvent() for gesture handling. You must also override generateLayoutParams() if you need custom LayoutParams (e.g., with span or weight attributes).

Extend an existing View when you need to modify behavior of a standard widget — extend AppCompatEditText for a formatted currency input, extend AppCompatImageView for pinch-to-zoom, extend MaterialTextView for a collapsible text view. This gives you the existing measure/draw/accessibility behavior for free. Always extend the AppCompat/Material variant, not the platform class, to get backported features and consistent theming.

The decision matrix: Does it have children? → ViewGroup. Does it draw custom content from scratch? → View. Does it behave like an existing widget with modifications? → extend that widget. A common mistake is extending ViewGroup when you should extend View — if you're drawing everything yourself with Canvas and have no child Views, ViewGroup adds unnecessary overhead (child list management, layout dispatch). Another mistake: extending View instead of AppCompatTextView when you need text rendering — reimplementing text measurement, ellipsizing, and accessibility is extremely complex.

Q32. Implement onMeasure() for a custom circular gauge that respects MeasureSpec constraints.

A circular gauge needs equal width and height (square bounds) and should be as large as constraints allow:

override fun onMeasure(widthSpec: Int, heightSpec: Int) {
    val desiredSize = (200 * resources.displayMetrics.density).toInt()

    val width = resolveSize(desiredSize, widthSpec)
    val height = resolveSize(desiredSize, heightSpec)
    val size = minOf(width, height) // force square

    setMeasuredDimension(size, size)
}

override fun onSizeChanged(w: Int, h: Int, oldW: Int, oldH: Int) {
    val padding = maxOf(paddingLeft, paddingTop, paddingRight, paddingBottom).toFloat()
    val strokeHalf = arcPaint.strokeWidth / 2f
    val inset = padding + strokeHalf
    arcRect.set(inset, inset, w - inset, h - inset)
}

resolveSize() handles the MeasureSpec logic: for EXACTLY it returns the spec size, for AT_MOST it returns min(desired, specSize), for UNSPECIFIED it returns the desired size. Taking the minimum of width and height ensures the gauge is always circular. onSizeChanged() precomputes the arc bounds once rather than in every onDraw() call — this avoids object allocation in the draw path. The strokeHalf inset prevents the arc stroke from being clipped at the View edges. This pattern generalizes to any View that needs a square aspect ratio.

Q33. How do custom attributes work? Walk through the full flow from attrs.xml to reading values in the constructor.

The flow has four steps:

(1) Declare — define attributes in res/values/attrs.xml inside a <declare-styleable> block with name matching your View class. Each <attr> specifies a name and format (string, color, dimension, enum, flags, reference, etc.).

(2) Use in XML — in the layout file, add the app namespace (xmlns:app="http://schemas.android.com/apk/res-auto") and set attributes like app:gaugeColor="#FF0000".

(3) Read in constructor — in the View's (Context, AttributeSet) constructor, call context.obtainStyledAttributes(attrs, R.styleable.YourView), which returns a TypedArray. The TypedArray resolves theme references, applies default styles, and provides typed getters.

(4) Recycle — call typedArray.recycle() to return it to the pool.

init {
    context.obtainStyledAttributes(attrs, R.styleable.CircularGauge, defStyleAttr, 0).use { ta ->
        gaugeColor = ta.getColor(R.styleable.CircularGauge_gaugeColor, Color.BLUE)
        maxValue = ta.getFloat(R.styleable.CircularGauge_maxValue, 100f)
        strokeWidth = ta.getDimension(R.styleable.CircularGauge_strokeWidth, 8.dp)
    }
}

The defStyleAttr parameter allows a theme attribute to provide defaults (e.g., gaugeStyle in the theme). The .use {} extension auto-recycles. AAPT generates R.styleable.CircularGauge as an int array of attribute resource IDs, and R.styleable.CircularGauge_gaugeColor as the index into that array.

Q34. Implement a custom FlowLayout ViewGroup that wraps children to the next line.

The core logic: in onMeasure(), iterate children left-to-right; when a child would exceed the available width, wrap to the next row. Track row height and total height:

override fun onMeasure(widthSpec: Int, heightSpec: Int) {
    val maxWidth = MeasureSpec.getSize(widthSpec)
    var rowWidth = 0; var rowHeight = 0; var totalHeight = 0; var totalWidth = 0

    for (i in 0 until childCount) {
        val child = getChildAt(i)
        if (child.visibility == GONE) continue
        measureChildWithMargins(child, widthSpec, 0, heightSpec, totalHeight)
        val lp = child.layoutParams as MarginLayoutParams
        val childW = child.measuredWidth + lp.leftMargin + lp.rightMargin
        val childH = child.measuredHeight + lp.topMargin + lp.bottomMargin

        if (rowWidth + childW > maxWidth) {
            totalHeight += rowHeight; totalWidth = maxOf(totalWidth, rowWidth)
            rowWidth = childW; rowHeight = childH
        } else {
            rowWidth += childW; rowHeight = maxOf(rowHeight, childH)
        }
    }
    totalHeight += rowHeight; totalWidth = maxOf(totalWidth, rowWidth)
    setMeasuredDimension(
        resolveSize(totalWidth, widthSpec), resolveSize(totalHeight, heightSpec)
    )
}

onLayout() mirrors this logic, calling child.layout() with accumulated x/y positions, resetting x to 0 and advancing y by row height on each wrap. Override generateLayoutParams() to return MarginLayoutParams so margins work. This is the same algorithm Google's Flexbox library uses at its core, though Flexbox adds alignment, grow/shrink factors, and cross-axis distribution.

Q35. How do you save and restore state in a custom View? What happens with duplicate IDs?

Override onSaveInstanceState() to return a Parcelable containing your View's state, and onRestoreInstanceState() to read it back. The standard pattern uses a custom SavedState class extending BaseSavedState (which handles Parcelable boilerplate):

override fun onSaveInstanceState(): Parcelable {
    val superState = super.onSaveInstanceState()
    return SavedState(superState).apply { progress = this@Gauge.progress }
}

override fun onRestoreInstanceState(state: Parcelable) {
    val ss = state as SavedState
    super.onRestoreInstanceState(ss.superState)
    progress = ss.progress
}

The framework only calls these methods if the View has an android:id set. State is stored in a SparseArray<Parcelable> keyed by View ID.

Duplicate IDs cause state collision: if two Views share the same ID, the second View's state overwrites the first in the SparseArray during save, and both Views restore the second View's state. This is a common bug with Fragments that inflate the same layout — each instance has Views with identical IDs.

The fix: use Fragment.onSaveInstanceState() (which uses a separate SparseArray per Fragment) or ensure unique IDs. For RecyclerView items, state saving is handled by the adapter, not the View system — ViewHolder state must be managed manually since items are recycled.

Q36. What accessibility considerations must a custom View address? How do you implement them?

A custom View must provide: (1) Content description — set contentDescription for Views without text, so TalkBack can announce them. For Views with visible text, TalkBack reads the text automatically. (2) Role/class name — in onInitializeAccessibilityNodeInfo(), set info.className to the closest standard widget (e.g., "android.widget.Button" for a custom button) so screen readers apply correct interaction patterns (e.g., "double-tap to activate" for buttons).

(3) State information — report checked/selected/disabled state via info.isChecked, info.isEnabled, etc. (4) Actions — add supported actions via info.addAction(AccessibilityAction.ACTION_CLICK) and handle them in performAccessibilityAction(). Custom actions can have labels: AccessibilityAction(R.id.custom_action, "Delete item"). (5) Live regions — for dynamic content (counters, status), set accessibilityLiveRegion = ACCESSIBILITY_LIVE_REGION_POLITE so updates are announced without the user navigating to the View.

(6) Touch exploration — for compound Views, implement AccessibilityNodeProvider to expose virtual child nodes (e.g., a calendar grid where each day is a virtual accessible node that TalkBack can navigate individually). (7) Focus — ensure focusable = true and importantForAccessibility = IMPORTANT_FOR_ACCESSIBILITY_YES. Call sendAccessibilityEvent(TYPE_VIEW_SELECTED) when state changes. Test with TalkBack and the Accessibility Scanner. Compose handles most of this via the Modifier.semantics {} block, which is more declarative and less error-prone than the View accessibility API.

Q37. How do you support RTL in a custom View's onDraw() and onLayout()?

RTL support requires handling at multiple levels:

In onLayout(): check layoutDirection == LAYOUT_DIRECTION_RTL (or ViewCompat.getLayoutDirection()) and mirror child positions. Instead of placing children left-to-right starting at paddingStart, place them right-to-left starting at width - paddingEnd. Use getLayoutDirection() rather than checking locale — the system resolves this from the configuration.

In onDraw(): for Canvas operations, you can use canvas.save() + canvas.scale(-1f, 1f, width / 2f, 0f) to mirror the entire drawing, but this is heavy-handed and breaks text rendering. Better: use isLayoutRtl to conditionally flip x-coordinates for individual draw calls. For text, Canvas.drawText() handles RTL scripts automatically via HarfBuzz — you don't need to reverse strings. For progress bars or horizontal indicators, swap the start/end positions based on layout direction.

In XML: use marginStart/marginEnd instead of marginLeft/marginRight; use paddingStart/paddingEnd; use drawableStart/drawableEnd. For custom attributes, provide android:layoutDirection="locale" support. Override onRtlPropertiesChanged(int layoutDirection) to react to direction changes — recalculate any cached positions or paths. The ViewGroup helper getChildDrawingOrder() should also be RTL-aware if you override drawing order. Test with Developer Options → Force RTL layout, and test with actual RTL locales (Arabic, Hebrew) since some layout behaviors differ.

Q38. What is the performance impact of allocating objects in onDraw()? How do you avoid it?

Allocating objects in onDraw() is one of the most common custom View performance mistakes. onDraw() runs on every frame during animations (60+ times/second). Each allocation: (1) puts pressure on the garbage collector — ART's CC collector is concurrent but still has pause points, and high allocation rates increase GC frequency. (2) can trigger GC pauses that cause jank — even a 2ms GC pause during a 16ms frame budget is significant. (3) generates garbage that fills the young generation (RegionSpace TLAB), triggering minor GCs.

Common offenders: creating Paint, Rect, RectF, Path, Matrix, or PorterDuffXfermode objects inside onDraw(). Also: string formatting (String.format()), autoboxing primitives, and creating temporary arrays.

The fix: allocate all drawing objects in the constructor or init {} block, store them as fields, and reuse them in onDraw(). For Path, call path.reset() or path.rewind() instead of creating a new one. For Rect/RectF, mutate the existing instance with set(). Android Lint has the DrawAllocation check that flags allocations inside onDraw(). In Compose, the equivalent concern applies to DrawScope lambdas — avoid allocating inside Canvas draw calls.

Touch & Input (Q39–Q48)

Q39. Walk through the complete touch dispatch chain from Activity to the target View.

A touch event travels: InputDispatcher (native, in system_server) → InputChannel (Unix domain socket pair) → ViewRootImpl.WindowInputEventReceiver (via Looper fd callback) → ViewRootImpl.processPointerEvent() → DecorView.dispatchTouchEvent() → Activity.dispatchTouchEvent() → PhoneWindow.superDispatchTouchEvent() → DecorView.superDispatchTouchEvent() → ViewGroup.dispatchTouchEvent() (recursive descent through the View tree). At each ViewGroup level, dispatchTouchEvent(): (1) calls onInterceptTouchEvent() — if it returns true, the ViewGroup steals the event. (2) If not intercepted on ACTION_DOWN, iterates children in reverse drawing order (front-to-back via buildOrderedChildList() which respects elevation and translationZ), hit-tests each child (checking bounds + canReceivePointerEvents() + isTransformedTouchPointInView()), and calls child.dispatchTouchEvent(). (3) If a child returns true, it becomes the "touch target" and receives all subsequent events in the gesture. (4) If no child handles it, the ViewGroup's own onTouchEvent() is called. At the leaf View level, dispatchTouchEvent() calls onTouchListener.onTouch() first (if set and the View is enabled), then onTouchEvent(). If the entire tree returns false, the event bubbles back up to Activity.onTouchEvent(). The reverse drawing order ensures that visually front-most Views get first crack at events. The InputDispatcher in system_server has a 5-second ANR timeout — if the app doesn't consume the event within 5 seconds, an ANR is triggered.

Q40. Explain onInterceptTouchEvent() — when is it called and what happens when it returns true?

onInterceptTouchEvent() is called on a ViewGroup for every MotionEvent in a gesture, as long as the ViewGroup has not yet intercepted. On ACTION_DOWN, it's always called. On subsequent events (MOVE, UP), it's called only if: (1) the ViewGroup has a child touch target (a child claimed the DOWN), and (2) the child hasn't called requestDisallowInterceptTouchEvent(true).

When onInterceptTouchEvent() returns true: (1) the current child touch target receives ACTION_CANCEL — this tells the child the gesture is being stolen. (2) The child is removed from the touch target list. (3) All subsequent events in the gesture go directly to the ViewGroup's onTouchEvent() — onInterceptTouchEvent() is no longer called for this gesture.

This is how scrolling containers work: ScrollView.onInterceptTouchEvent() returns false on DOWN (letting children handle clicks), monitors MOVE events for vertical displacement exceeding touch slop, then returns true to steal the gesture and begin scrolling. The child receives CANCEL and stops its click/press handling. Once intercepted, the interception cannot be undone for the current gesture.

Q41. How does requestDisallowInterceptTouchEvent() work? When is the flag reset?

A child View calls parent.requestDisallowInterceptTouchEvent(true) to prevent its parent (and all ancestors) from intercepting touch events. Internally, this sets the FLAG_DISALLOW_INTERCEPT flag on the parent ViewGroup and propagates up the parent chain. When this flag is set, ViewGroup.dispatchTouchEvent() skips the onInterceptTouchEvent() call entirely — the parent cannot steal the gesture.

The flag is reset on every ACTION_DOWN. In ViewGroup.dispatchTouchEvent(), the first thing that happens on ACTION_DOWN is resetTouchState(), which clears FLAG_DISALLOW_INTERCEPT and all touch targets. This means the flag only affects MOVE/UP events within a gesture — the parent always gets to see ACTION_DOWN via onInterceptTouchEvent().

Common use case: a horizontal ViewPager inside a vertical ScrollView. The ViewPager detects horizontal movement and calls requestDisallowInterceptTouchEvent(true) on its parent to prevent ScrollView from stealing the horizontal swipe and interpreting it as a vertical scroll. NestedScrollView and RecyclerView use the newer NestedScrolling API instead, which provides more granular control.

Q42. What is the touch target mechanism in ViewGroup? How does multi-touch work with multiple targets?

ViewGroup maintains a linked list of TouchTarget objects, each containing a child View and a pointerIdBits bitmask. On ACTION_DOWN (first finger), the ViewGroup hit-tests children and creates a TouchTarget for the child that handles the event, with the pointer ID bit set.

On ACTION_POINTER_DOWN (additional fingers), the ViewGroup hit-tests again — if the new pointer lands on a different child, a new TouchTarget is created for that child. If it lands on the same child, the pointer ID is added to the existing TouchTarget's bitmask.

During ACTION_MOVE, the ViewGroup splits the MotionEvent: each TouchTarget receives only the pointers assigned to it, via MotionEvent.split(). This enables true multi-touch across different Views — e.g., pressing two buttons simultaneously, or pinch-zooming an image while scrolling a list with another finger.

When a pointer goes up (ACTION_POINTER_UP), its bit is removed from the TouchTarget. When the last pointer for a target goes up, the TouchTarget is removed. FLAG_SPLIT_MOTION_EVENTS (default true since API 11) controls whether a ViewGroup supports this splitting — without it, all pointers go to the first touch target.

Q43. Implement a zoomable ImageView using GestureDetector and ScaleGestureDetector.

The approach: use a Matrix to transform the image, ScaleGestureDetector for pinch zoom, and GestureDetector for pan and double-tap:

class ZoomableImageView(context: Context, attrs: AttributeSet?) : AppCompatImageView(context, attrs) {
    private val matrix = Matrix()
    private val scaleDetector = ScaleGestureDetector(context, object : ScaleGestureDetector.SimpleOnScaleGestureListener() {
        override fun onScale(detector: ScaleGestureDetector): Boolean {
            matrix.postScale(detector.scaleFactor, detector.scaleFactor,
                detector.focusX, detector.focusY)
            imageMatrix = matrix
            return true
        }
    })
    private val gestureDetector = GestureDetector(context, object : GestureDetector.SimpleOnGestureListener() {
        override fun onScroll(e1: MotionEvent?, e2: MotionEvent, dx: Float, dy: Float): Boolean {
            matrix.postTranslate(-dx, -dy)
            imageMatrix = matrix
            return true
        }
        override fun onDoubleTap(e: MotionEvent): Boolean {
            matrix.reset() // reset zoom
            imageMatrix = matrix
            return true
        }
    })

    init { scaleType = ScaleType.MATRIX }

    override fun onTouchEvent(event: MotionEvent): Boolean {
        scaleDetector.onTouchEvent(event)
        gestureDetector.onTouchEvent(event)
        return true
    }
}

Production code would add: scale limits (min/max zoom), boundary clamping (prevent panning beyond image bounds), fling support via OverScroller, and animated zoom transitions. ScaleGestureDetector handles the math of tracking two pointers and computing scale factor and focus point. The matrix approach is efficient because setImageMatrix() modifies the RenderNode transformation without re-decoding the bitmap.

Q44. Explain nested scrolling — how do NestedScrollingChild and NestedScrollingParent coordinate?

The nested scrolling protocol (introduced in API 21, backported via NestedScrollingChildHelper/NestedScrollingParentHelper) allows a child scrolling View to share scroll events with an ancestor without the parent intercepting the gesture. The flow: (1) Child calls startNestedScroll(axes) on ACTION_DOWN — this walks up the parent chain until a NestedScrollingParent accepts via onStartNestedScroll() returning true. (2) Before the child scrolls, it calls dispatchNestedPreScroll(dx, dy, consumed, offset) — the parent gets first chance to consume some or all of the scroll delta (e.g., collapsing a toolbar). The consumed array is filled by the parent with how much it consumed. (3) The child scrolls by whatever the parent didn't consume. (4) After scrolling, the child calls dispatchNestedScroll(dxConsumed, dyConsumed, dxUnconsumed, dyUnconsumed) — the parent can consume the remainder (e.g., stretching an overscroll effect). (5) On ACTION_UP, stopNestedScroll() is called. NestedScrollView, RecyclerView, and CoordinatorLayout all implement this protocol. The v2 API (NestedScrollingChild2/NestedScrollingParent2) added a type parameter distinguishing TYPE_TOUCH from TYPE_NON_TOUCH (fling), so parents can behave differently during flings. The v3 API (NestedScrollingChild3/NestedScrollingParent3) adds dispatchNestedScroll() with a consumed output array, enabling chained consumption across multiple nesting levels. This replaced the old requestDisallowInterceptTouchEvent() hack with a cooperative protocol. In Compose, Modifier.nestedScroll() provides the equivalent mechanism with a cleaner API.

Q45. How does CoordinatorLayout's Behavior system work? Give an example of a custom Behavior.

CoordinatorLayout is a FrameLayout that delegates child interaction logic to Behavior objects attached via app:layout_behavior or @DefaultBehavior annotation. During layout, CoordinatorLayout calls behavior.onMeasureChild() and behavior.onLayoutChild() — allowing the Behavior to override standard measurement and positioning.

For dependencies, behavior.layoutDependsOn(parent, child, dependency) declares which sibling a child depends on. When the dependency changes (moves, resizes), behavior.onDependentViewChanged() is called — this is how FloatingActionButton moves when a Snackbar appears.

For nested scrolling, CoordinatorLayout implements NestedScrollingParent3 and forwards scroll events to all children's Behaviors via onStartNestedScroll(), onNestedPreScroll(), onNestedScroll(), and onNestedFling(). A custom Behavior example — a FAB that hides on scroll down and shows on scroll up:

class HideFabOnScrollBehavior : FloatingActionButton.Behavior() {
    override fun onStartNestedScroll(parent: CoordinatorLayout, child: FloatingActionButton,
        directTarget: View, target: View, axes: Int, type: Int) = axes == ViewCompat.SCROLL_AXIS_VERTICAL

    override fun onNestedScroll(parent: CoordinatorLayout, child: FloatingActionButton,
        target: View, dxConsumed: Int, dyConsumed: Int, dxUnconsumed: Int, dyUnconsumed: Int, type: Int, consumed: IntArray) {
        if (dyConsumed > 0) child.hide() else if (dyConsumed < 0) child.show()
    }
}
Q46. What is ACTION_CANCEL and when is it sent? Why is it important to handle it?

ACTION_CANCEL indicates that the current gesture is being aborted — the View should undo any visual feedback (pressed state, drag highlights) and stop tracking the gesture. It's sent in these situations:

(1) A parent ViewGroup intercepts the touch — when onInterceptTouchEvent() returns true, the current child target receives ACTION_CANCEL. (2) The window loses focus during a gesture (e.g., a phone call interrupts, a dialog appears). (3) The parent is removed from the window during a gesture. (4) A system gesture (like the navigation bar swipe or edge-back gesture) overrides the app gesture.

Failing to handle ACTION_CANCEL causes bugs: a button stays in pressed state permanently, a drag operation leaves ghost artifacts, a scroll position gets stuck. In onTouchEvent(), handle it like ACTION_UP but without triggering the action:

MotionEvent.ACTION_CANCEL -> {
    isPressed = false       // clear pressed state
    removeCallbacks(longPressRunnable)
    velocityTracker?.recycle()
    velocityTracker = null
}

GestureDetector handles this internally, but custom touch handling must explicitly check for it. In Compose, the PointerInputScope cancellation is handled via coroutine cancellation — the awaitPointerEvent() loop terminates when the gesture is cancelled.

Q47. What is touch slop? How does it prevent accidental drags?

Touch slop is a distance threshold (in pixels) that a finger must move before the system considers it a drag rather than a tap. It's defined by ViewConfiguration.getScaledTouchSlop() — typically 8dp (scaled for screen density). The purpose: human fingers are imprecise; even a deliberate tap involves ~2-4px of movement. Without touch slop, every tap would also trigger a drag. The implementation pattern in custom touch handling:

val slop = ViewConfiguration.get(context).scaledTouchSlop
var downX = 0f; var downY = 0f; var isDragging = false

override fun onTouchEvent(event: MotionEvent): Boolean = when (event.action) {
    ACTION_DOWN -> { downX = event.x; downY = event.y; isDragging = false; true }
    ACTION_MOVE -> {
        if (!isDragging) {
            val dx = event.x - downX; val dy = event.y - downY
            if (dx * dx + dy * dy > slop * slop) isDragging = true
        }
        if (isDragging) handleDrag(event)
        true
    }
    else -> false
}

Different slop values exist for different gestures: scaledPagingTouchSlop (larger, for ViewPager — prevents accidental page swipes), scaledDoubleTapSlop (maximum distance between two taps for a double-tap). GestureDetector and ScaleGestureDetector use these internally. Accessibility mode increases touch slop to accommodate users with motor impairments.

Q48. How does VelocityTracker work? How is it used in fling animations?

VelocityTracker records touch positions over time and computes the velocity (pixels/second) of the finger movement. You obtain one from the pool via VelocityTracker.obtain(), feed it events with addMovement(event), then call computeCurrentVelocity(units, maxVelocity) to calculate velocity.

Internally, it uses a least-squares polynomial regression on the last ~20ms of touch samples to smooth out noise and produce accurate velocity estimates. The units parameter is typically 1000 (pixels per second). maxVelocity caps the result (from ViewConfiguration.getScaledMaximumFlingVelocity()) to prevent unreasonably fast flings. Usage for fling:

ACTION_UP -> {
    velocityTracker?.let { vt ->
        vt.computeCurrentVelocity(1000, maxFlingVelocity.toFloat())
        val velocityX = vt.xVelocity
        val velocityY = vt.yVelocity
        if (abs(velocityY) > minFlingVelocity) {
            scroller.fling(scrollX, scrollY, -velocityX.toInt(), -velocityY.toInt(),
                0, maxScrollX, 0, maxScrollY)
            ViewCompat.postInvalidateOnAnimation(this)
        }
        vt.recycle()
    }
}

OverScroller.fling() takes the velocity and computes a deceleration curve. On each frame, computeScrollOffset() returns the current position. RecyclerView's LinearLayoutManager uses this exact pattern for smooth fling scrolling. Always recycle the VelocityTracker to avoid pool exhaustion.

Animations (Q49–Q55)

Q49. Why is View Animation (legacy) broken? How does Property Animation fix it?

The legacy android.view.animation framework (Tween animations: TranslateAnimation, AlphaAnimation, etc.) has a fundamental flaw: it only transforms the drawing matrix of the View, not the View itself. When you translate a Button 200px to the right with a TranslateAnimation, the button visually moves, but its click target stays at the original position. The View's mLeft/mTop/mRight/mBottom are unchanged — getX() still returns the old value. Touch events are dispatched to the original bounds. This made animations purely visual — you couldn't interact with animated Views. Additionally, legacy animations call invalidate() on the parent every frame, forcing the entire parent subtree to redraw. The Property Animation framework (android.animation, API 11+) fixes this by actually modifying View properties. ObjectAnimator.ofFloat(view, "translationX", 200f) calls View.setTranslationX() on each frame, which updates the RenderNode's translation property. The View's hit rect includes the translation, so touch events work correctly at the animated position. Property animations also work on any object (not just Views), support custom interpolators, can animate any property via reflection or Property objects, and support AnimatorSet for choreographing complex sequences. A common migration pitfall: legacy AnimationListener.onAnimationEnd() fires before the final frame is drawn, while property animation's AnimatorListener.onAnimationEnd() fires after. Code that reads View position in the end callback may get different results.

Q50. Explain the difference between ValueAnimator, ObjectAnimator, and ViewPropertyAnimator — when do you use each?

ValueAnimator is the base class — it animates a value from start to end over time, calling an AnimatorUpdateListener on each frame with the interpolated value. You manually apply the value to whatever you want. Use it for animating non-View properties (a custom drawing parameter, a data model value, a gradient position) or when you need full control over what happens each frame.

ObjectAnimator extends ValueAnimator and automatically sets a named property on a target object using reflection or Property objects. ObjectAnimator.ofFloat(view, View.TRANSLATION_X, 100f) calls view.setTranslationX() each frame. Use it for animating a single property on any object. Using Property<View, Float> objects (like View.TRANSLATION_X) avoids reflection overhead — the Property object calls the setter directly.

ViewPropertyAnimator is a fluent API optimized specifically for View properties: view.animate().translationX(100f).alpha(0.5f).setDuration(300).start(). It batches multiple property changes into a single animation pass (one Choreographer callback updates all properties), automatically manages hardware layers (via withLayer()), and is the most efficient option for View animations. Use it whenever you're animating standard View properties (translation, scale, rotation, alpha). The hierarchy: ViewPropertyAnimator for View properties (most efficient, least boilerplate), ObjectAnimator for single properties on any object, ValueAnimator for full manual control.

Q51. How do interpolators work? Design a custom interpolator for a "rubber band" effect.

An Interpolator (or TimeInterpolator) maps elapsed time fraction (0.0 to 1.0) to an animation progress value. The input is linear time; the output controls the animation curve. LinearInterpolator: output = input. AccelerateDecelerateInterpolator: slow start, fast middle, slow end (cosine curve). OvershootInterpolator: goes past the target then settles back. The framework calls getInterpolation(float input) on each frame and uses the returned value to compute the animated property. A rubber band interpolator overshoots, bounces back past the start, then settles:

class RubberBandInterpolator(private val tension: Float = 2.5f) : Interpolator {
    override fun getInterpolation(t: Float): Float {
        // Damped sine wave: overshoots, oscillates, settles at 1.0
        val decay = (-tension * t).let { Math.exp(it.toDouble()).toFloat() }
        return 1f - decay * cos((2 * Math.PI * t).toFloat())
    }
}

The damped cosine produces oscillation that decays over time — the amplitude of each bounce decreases exponentially. The tension parameter controls how quickly oscillations decay. Higher tension = fewer bounces. This is mathematically similar to a spring-damper system, which is why SpringAnimation from the physics-based animation library is often a better choice — it models the physics directly and handles interruption gracefully (a spring can be redirected mid-animation without discontinuity).

Q52. Explain the Transition framework — how does beginDelayedTransition() work internally?

TransitionManager.beginDelayedTransition(sceneRoot, transition) captures the current state of all Views in the sceneRoot, then posts a callback to run after the current layout pass. When you modify View properties (visibility, bounds, text) after this call, the next layout pass computes the new state. The Transition framework then: (1) Captures end values — records the new position, size, visibility, and other properties of each View via Transition.captureEndValues(TransitionValues). (2) Compares start and end values — for each View that changed, creates appropriate Animators via Transition.createAnimator(). (3) Runs the animators — Views animate from their old state to their new state. Internally, beginDelayedTransition() calls sceneRoot.getViewTreeObserver().addOnPreDrawListener() — the pre-draw listener fires after measure/layout but before draw, giving the framework the new layout values. The Transition object (e.g., ChangeBounds, Fade, TransitionSet) defines which properties to capture and how to animate them. ChangeBounds captures left/top/right/bottom and animates position/size using a ValueAnimator that calls view.setLeftTopRightBottom(). Fade captures visibility and animates alpha. You can combine them with TransitionSet (sequential or together) for complex animations. The framework handles View identity matching via transitionName or View ID. A powerful pattern: AutoTransition combines Fade(OUT) → ChangeBounds → Fade(IN) in sequence, handling additions, removals, and moves in a single call. The Transition framework is also used internally by MotionLayout for constraint-based animations.

Q53. How do shared element transitions work? What are the common pitfalls?

Shared element transitions animate a View from one Activity/Fragment to another, creating visual continuity. The flow: (1) The calling Activity specifies shared elements via ActivityOptions.makeSceneTransitionAnimation(activity, view, transitionName). (2) The framework captures the shared element's start bounds, bitmap, and properties via SharedElementCallback.onCaptureSharedElementSnapshot(). (3) The target Activity starts; before drawing, it captures the shared element's end bounds via onSharedElementEnd(). (4) The framework creates a ChangeBounds + ChangeTransform + ChangeClipBounds + ChangeImageTransform animation from start to end, rendering the shared element in a ViewOverlay above both Activities. Common pitfalls: (1) Data not loaded — if the target screen loads an image asynchronously, the end bounds are wrong (0×0). Fix: call postponeEnterTransition() in onCreate() and startPostponedEnterTransition() after the image loads (with a timeout fallback to prevent indefinite postponement). (2) RecyclerView items — the shared element may be recycled or not yet laid out on return. Fix: scroll to the item and wait for layout via doOnPreDraw { startPostponedEnterTransition() } before starting the return transition. (3) Mismatched transitionNames — the names must match exactly between source and target; use ViewCompat.setTransitionName() for dynamic names in RecyclerView items. (4) Window background flicker — set window.sharedElementsUseOverlay = true (default) and ensure the target Activity's background is opaque. (5) Fragment transitions require setReorderingAllowed(true) to work correctly with postponed transitions — without it, the Fragment lifecycle callbacks execute in the wrong order.

Q54. When should you use physics-based animations (SpringAnimation, FlingAnimation) instead of property animations?

Physics-based animations (androidx.dynamicanimation) model real-world physics (spring force, friction) rather than following a fixed duration curve. Use them when: (1) Interruption is expected — a spring animation can be redirected to a new target mid-flight without visual discontinuity (the velocity is preserved and the spring naturally adjusts). Property animations with fixed duration produce jarring jumps when interrupted and restarted because velocity is lost. (2) Natural feel — springs produce overshoot and settle behavior that feels organic. The SpringForce is configured with stiffness (how fast it moves) and dampingRatio (how much it oscillates — 1.0 = critically damped, no overshoot; 0.5 = underdamped, bouncy). Fling animations produce deceleration that matches real-world friction. (3) Gesture-driven animations — after a drag release, FlingAnimation uses the finger's velocity (from VelocityTracker) to produce a natural continuation. SpringAnimation with the release velocity as initial velocity produces a spring-back effect. (4) No known end state — FlingAnimation decelerates to a stop based on friction coefficient; you don't need to specify an end value. Use property animations when: you need exact duration control, precise choreography with other animations (AnimatorSet), or the animation is non-interactive (a loading spinner, a one-shot entrance animation). In Compose, Animatable.animateTo(targetValue, spring()) and Animatable.animateDecay() are the equivalents, built into the animation system by default — Compose uses spring animations as the default for all animate*AsState() APIs.

Q55. How do hardware layers improve animation performance? When should you NOT use them?

During an animation, a View's onDraw() is called every frame to re-record the DisplayList. With a hardware layer (LAYER_TYPE_HARDWARE), the View is rendered once into a GPU texture, and subsequent frames just composite that texture with updated transformation properties (translation, alpha, scale, rotation). This eliminates per-frame DisplayList recording and GPU draw command execution for the View's content.

ViewPropertyAnimator.withLayer() automatically sets a hardware layer at animation start and removes it on end — this is the recommended pattern.

Do NOT use hardware layers when: (1) The View's content changes during the animation — each content change invalidates the layer, forcing a full re-render plus compositing (worse than no layer). Examples: a TextView with changing text, a ProgressBar, a RecyclerView scrolling. (2) The View is very large — the layer texture consumes GPU memory proportional to the View's pixel area. A full-screen layer on a 1440×3200 display = ~18MB. (3) Many Views have layers simultaneously — mobile GPUs have limited texture memory; too many layers cause eviction and re-rendering. (4) The animation is short (< 2 frames) — the cost of creating and destroying the layer exceeds the savings. Profile with GPU rendering profiler to verify layers actually help.

RecyclerView (Q56–Q62)

Q56. Explain RecyclerView's 4-level recycling hierarchy — scrap, cache, extension, pool.

When RecyclerView needs a ViewHolder for a position, it checks four levels in order via Recycler.tryGetViewHolderForPositionByDeadline(): (1) Scrap (mAttachedScrap + mChangedScrap) — ViewHolders that are still attached to the RecyclerView but being re-laid out during the current layout pass. mAttachedScrap holds items that haven't changed; mChangedScrap holds items that have changed (for change animations). Scrap items are returned without rebinding — they're considered valid for the same position. (2) Cache (mCachedViews) — recently scrolled-off ViewHolders, stored by position. Default size: 2 (configurable via setViewCacheSize()). Cache hits skip onBindViewHolder() — the ViewHolder is reused as-is for the same position. This makes reverse scrolling instant since the ViewHolder still has the correct data. (3) ViewCacheExtension — a developer-provided cache (rarely used). You can implement custom caching logic for specific positions — useful for header/footer Views that never change. (4) RecycledViewPool (mRecyclerPool) — ViewHolders stored by view type in ScrapData ArrayLists. Default: 5 per type. Pool items require rebinding via onBindViewHolder() because they may have held different data. The pool is shared across RecyclerViews via setRecycledViewPool() — useful for ViewPager2 tabs with similar content. If all levels miss, onCreateViewHolder() is called to inflate a new item. The scrap and cache are position-based (no rebind needed); the pool is type-based (rebind required). Understanding this hierarchy is key to optimizing scroll performance — increasing cache size helps oscillating scrolls, while increasing pool size helps fast forward scrolling.

Q57. What is the difference between bindingAdapterPosition, layoutPosition, and absoluteAdapterPosition?

bindingAdapterPosition — the position within the specific adapter that bound this ViewHolder. With ConcatAdapter, if adapter A has 5 items and adapter B has 10 items, a ViewHolder from adapter B at global position 7 has bindingAdapterPosition = 2 (index within B). This is what you use in click listeners to index into the adapter's data list.

absoluteAdapterPosition — the position within the outermost adapter (the one set on RecyclerView). In the ConcatAdapter example above, the same ViewHolder has absoluteAdapterPosition = 7. Use this for operations that need the global position (scrolling to a position, DiffUtil, smoothScrollToPosition()).

layoutPosition — the position as of the last layout pass. This can differ from adapter position when data changes have been dispatched (notifyItemInserted()) but layout hasn't run yet. Between notify*() and the next layout, layoutPosition reflects the old layout while adapter positions reflect the new data. All three return RecyclerView.NO_POSITION (-1) when the ViewHolder is not bound or is being removed — always check for this in click handlers to avoid ArrayIndexOutOfBoundsException. The common mistake: using adapterPosition (deprecated in favor of bindingAdapterPosition/absoluteAdapterPosition) or layoutPosition in click handlers — use bindingAdapterPosition for data access, absoluteAdapterPosition for RecyclerView operations.

Q58. How does DiffUtil work? What algorithm does it use and what is its complexity?

DiffUtil computes the minimum set of operations (insert, remove, move, change) to transform one list into another. It uses Eugene Myers' difference algorithm — the same algorithm behind Unix diff. The algorithm finds the shortest edit script (SES) between two sequences by exploring the edit graph diagonally. Complexity: O(N + D²) time and O(N + D²) space, where N is the total number of items and D is the number of edits (the "edit distance"). For small differences (D << N), this is nearly linear — ideal for typical UI updates where only a few items change. For completely different lists, it degrades to O(N²).

DiffUtil.calculateDiff(callback) takes a DiffUtil.Callback with areItemsTheSame(oldPos, newPos) (identity check — same entity, typically comparing IDs) and areContentsTheSame(oldPos, newPos) (equality check — same data, typically using data class equals). It also supports getChangePayload(oldPos, newPos) for partial updates — if only a "like count" changed, the payload carries just that delta, and onBindViewHolder(holder, position, payloads) can update only the like button instead of rebinding the entire item.

AsyncListDiffer and ListAdapter run the diff computation on a background thread via Dispatchers.Default and dispatch results to the main thread, preventing jank on large lists. Move detection is optional (detectMoves = true) and adds a second pass with O(N²) cost — disable it if items are never reordered. A subtle gotcha: areContentsTheSame() is only called when areItemsTheSame() returns true, so make sure your identity check is correct or you'll get stale items.

Q59. Explain RecyclerView prefetch — how does GapWorker improve scroll performance?

GapWorker is RecyclerView's prefetch mechanism (API 21+, enabled by default via setItemPrefetchEnabled(true)). During scrolling, the UI thread has idle time between VSYNC signals — after the current frame's layout/draw completes but before the next VSYNC arrives (the "gap"). GapWorker uses this idle time to pre-create and pre-bind ViewHolders that will be needed in upcoming frames.

The flow: (1) During onScrolled(), the LayoutManager predicts which positions will become visible based on scroll direction and velocity via collectAdjacentPrefetchPositions(). (2) GapWorker posts these prefetch requests to a priority queue sorted by distance from the viewport edge. (3) Between frames, GapWorker.prefetch() runs on the UI thread's idle time (registered via Choreographer.postCallback(CALLBACK_COMMIT)) — it calls onCreateViewHolder() and onBindViewHolder() for predicted positions, storing results in the cache. It checks the remaining time budget before each prefetch to avoid overrunning into the next frame. (4) When the user actually scrolls to those positions, the ViewHolders are already ready — no inflation or binding on the critical path.

For nested RecyclerViews (e.g., a vertical list of horizontal carousels), LinearLayoutManager.setInitialPrefetchItemCount() tells the inner LayoutManager how many items to prefetch. Without this, the inner RecyclerView inflates all visible items during the outer RecyclerView's bind call, causing frame drops. GapWorker typically eliminates 60-80% of scroll jank from ViewHolder creation.

Q60. How would you optimize a RecyclerView that displays 10 different view types?

Multiple view types stress the RecycledViewPool because each type has its own pool (default 5 per type). Optimizations:

(1) Increase pool size for frequently used types: recyclerView.recycledViewPool.setMaxRecycledViews(TYPE_MESSAGE, 15). Profile to find the right number — too high wastes memory, too low causes inflation during fast scrolls. (2) Share the pool across RecyclerViews with setRecycledViewPool() — useful in ViewPager2 where each page has similar content. (3) Reduce view types — if types differ only in minor ways (icon, color, visibility of a sub-view), use a single type with conditional binding instead of separate types. 10 types that are 90% identical should be 1-2 types with conditional logic.

(4) Use payloads — for items that change partially, notifyItemChanged(pos, payload) with a payload avoids full rebinding. (5) Lightweight inflation — avoid complex nested layouts in item XML. Use ConstraintLayout for flat hierarchies. Consider programmatic View creation (no XML inflation cost — LayoutInflater uses reflection to instantiate Views). (6) Prefetch tuning — set setInitialPrefetchItemCount() for nested RecyclerViews to prefetch the right number of inner items.

(7) setHasStableIds(true) — enables the RecyclerView to reuse ViewHolders more efficiently during animations and avoid unnecessary rebinds when items are reordered. (8) Avoid expensive onBind — defer image loading to post-bind (Glide/Coil handle this), avoid formatting strings or computing layouts in onBindViewHolder(). Pre-compute display strings in the data layer. (9) Use ConcatAdapter to compose adapters rather than managing type logic in a single monolithic adapter — this improves code organization and allows each adapter to have its own DiffUtil logic.

Q61. What happens when you call notifyDataSetChanged() vs notifyItemChanged() with payloads?

notifyDataSetChanged() is the nuclear option: it tells RecyclerView that the entire dataset is invalid. RecyclerView: (1) marks all ViewHolders as invalid (flags FLAG_UPDATE + FLAG_INVALID), (2) cannot run item animations (no way to know what changed — the ItemAnimator receives no add/remove/move/change info), (3) rebinds every visible ViewHolder via onBindViewHolder(), (4) cannot use the scrap cache (items go directly to the RecycledViewPool since their position validity is unknown), (5) may reset scroll position if the adapter size changed. It's O(visible items) in bind cost and provides the worst user experience — no animations, full rebind, potential flicker.

notifyItemChanged(position, payload) is surgical: it marks only that one ViewHolder with FLAG_UPDATE, preserves the scrap cache for all other items, enables change animations (crossfade by default via DefaultItemAnimator), and passes the payload to onBindViewHolder(holder, position, payloads). With a payload, you can update only the changed sub-view:

override fun onBindViewHolder(holder: VH, position: Int, payloads: List<Any>) {
    if (payloads.isEmpty()) { fullBind(holder, position); return }
    payloads.forEach { payload ->
        when (payload) {
            PAYLOAD_LIKE -> holder.likeButton.isSelected = items[position].isLiked
            PAYLOAD_COMMENT_COUNT -> holder.commentCount.text = items[position].comments.toString()
        }
    }
}

This avoids rebinding expensive elements (images, formatted text). Always prefer granular notify* methods or ListAdapter/DiffUtil over notifyDataSetChanged().

Q62. Design a RecyclerView setup for a chat app with messages, dates, typing indicators, and ads.

Use ConcatAdapter to compose specialized adapters: a TypingIndicatorAdapter (single item, shown/hidden), a MessagesAdapter (handles messages + date headers), and an AdsAdapter (injected at intervals). The MessagesAdapter uses multiple view types:

enum class ChatViewType { SENT_TEXT, RECEIVED_TEXT, SENT_IMAGE, RECEIVED_IMAGE, DATE_HEADER }

class MessagesAdapter : ListAdapter<ChatItem, RecyclerView.ViewHolder>(ChatDiffCallback()) {
    override fun getItemViewType(position: Int) = when (val item = getItem(position)) {
        is ChatItem.Message -> if (item.isMine) ChatViewType.SENT_TEXT.ordinal else ChatViewType.RECEIVED_TEXT.ordinal
        is ChatItem.DateHeader -> ChatViewType.DATE_HEADER.ordinal
        // ...
    }.also { /* image types similarly */ }
}

Key design decisions:

(1) Date headers are computed items inserted by the data layer (not the adapter) — when the date changes between consecutive messages, a ChatItem.DateHeader is inserted. (2) Use LinearLayoutManager with reverseLayout = true and stackFromEnd = true — new messages appear at the bottom without manual scrolling.

(3) DiffUtil with stable message IDs enables smooth animations when new messages arrive. (4) The typing indicator is a separate adapter at position 0 (bottom of reversed list) — toggling it doesn't affect message positions.

(5) Ads use ConcatAdapter.Config.Builder().setIsolateViewTypes(true) so ad ViewHolders aren't mixed into the message pool. (6) Set recycledViewPool.setMaxRecycledViews() higher for message types (users scroll through many) and lower for rare types (date headers, ads).

Performance & System Design (Q63–Q65)

Q63. You have a screen with 60fps jank — walk through your debugging workflow using Perfetto, GPU profiling, and Layout Inspector.

Step 1: Reproduce and capture. Enable GPU rendering profiling (Developer Options → Profile GPU rendering → On screen as bars). Green bars exceeding the 16ms line confirm jank. Record a Perfetto trace with atrace categories: view, gfx, input, dalvik, sched. Use adb shell perfetto or the Perfetto UI for trace capture.

Step 2: Analyze the trace. In Perfetto, find the janky frames in the "Expected/Actual Timeline" track — frames where actual duration exceeds expected deadline. Check if the UI thread or RenderThread is the bottleneck. Long Choreographer#doFrame slices indicate UI thread work (measure/layout/draw). Drill into sub-slices: measure, layout, draw to find which phase is slow. Long DrawFrame slices on RenderThread indicate GPU-side issues (shader compilation, large DisplayLists, texture uploads).

Step 3: Identify the cause. Common culprits: (a) Measure/layout — deep View hierarchies, double-measuring (RelativeLayout nesting), requestLayout() during scroll. Use Layout Inspector to check hierarchy depth and identify unnecessary nesting. Look for Measure(SPEC_MODE) slices that repeat excessively. (b) Draw — overdraw (Debug GPU overdraw overlay shows red = 4x overdraw), expensive onDraw() with allocations, missing hardware layers during animation. (c) RenderThread — large DisplayLists, excessive saveLayer(), shader compilation stalls (first-frame jank — look for shader_compile slices). (d) GC pressure — check dalvik track for GC pauses during animation frames; look for allocations in onDraw() or onBindViewHolder(). (e) Binder calls on UI thread — blocking IPC during scroll (e.g., ContentResolver.query()).

Step 4: Fix and verify. Flatten hierarchy with ConstraintLayout, eliminate allocations in onDraw(), add hardware layers for animations, reduce overdraw with clipRect() or removing redundant backgrounds, move Binder calls off the UI thread. Re-capture trace to confirm the fix reduced frame times below the VSYNC deadline.

Q64. Design the View architecture for a complex dashboard screen with charts, lists, and real-time updates — how do you minimize overdraw and optimize rendering?

Architecture: Use a single RecyclerView with ConcatAdapter as the root scrolling container — this gives you recycling for off-screen sections and a single scroll context. Adapter segments: header (summary cards), chart section, list section, real-time ticker. Each section is a different view type with ConcatAdapter.Config.Builder().setIsolateViewTypes(true) to prevent cross-section ViewHolder reuse.

Overdraw reduction: (1) Remove the window background (window.setBackgroundDrawable(null)) since the RecyclerView items cover the full screen — this eliminates one full-screen overdraw layer. (2) For chart Views, use canvas.clipRect() to avoid drawing outside visible bounds during partial invalidation. (3) Avoid overlapping backgrounds — if a card has a white background, don't also set a white background on its parent. Use Layout Inspector's "Show Overdraw" to verify — aim for mostly blue (1x) with minimal green (2x).

Real-time updates: Use notifyItemChanged(position, payload) with payloads for ticker updates — only rebind the changed text, not the entire item. For charts, use a custom View that draws directly to Canvas (not a library that inflates complex View hierarchies). Animate data changes with ValueAnimator driving invalidate() on the chart View — pre-compute interpolated data points in the animator callback and only draw in onDraw().

Performance: Charts should pre-compute Path objects on a background thread (using Dispatchers.Default) and swap the cached Path atomically for onDraw() to consume. Use setHasFixedSize(true) on the RecyclerView if the adapter changes don't affect RecyclerView's size. For the real-time ticker, consider a TextureView or SurfaceView if update frequency exceeds 10fps — this decouples its rendering from the main View tree's VSYNC cycle and avoids invalidating the RecyclerView item on every tick. For the list section, use setHasStableIds(true) and implement getItemId() to enable efficient animations and avoid unnecessary rebinds.

Q65. Compare the View system with Jetpack Compose — what are the architectural differences, trade-offs, and when would you choose one over the other at Google scale?

Architectural differences: The View system is object-oriented and mutable — each View is a long-lived object (~200 fields per View instance) with mutable state, and the framework mutates the tree in place via imperative calls (setText(), setVisibility()). Compose is declarative and functional — UI is described as a function of state, and the runtime diffs the output (slot table) to determine minimal changes. Views use XML inflation (reflection-based) + findViewById()/ViewBinding; Compose uses Kotlin DSL with compiler plugin-generated code (no reflection, no XML parsing).

The View system's measure/layout is top-down with MeasureSpec negotiation and allows multi-pass measurement (which causes the exponential blowup problem); Compose uses intrinsic measurements and enforces single-pass layout with constraints — children can only be measured once, preventing the exponential problem by design. Views dispatch invalidation up the tree to ViewRootImpl; Compose uses the snapshot system to automatically track state reads and invalidate only the reading composition scopes — far more granular than View-level invalidation.

Trade-offs: Views have 15+ years of battle-tested edge cases, full accessibility support, and extensive third-party library ecosystem. Compose has simpler state management, less boilerplate, better animation APIs (spring-based by default), and eliminates entire classes of bugs (View state restoration, Fragment lifecycle, RecyclerView adapter bugs). However, Compose has higher baseline memory (slot table overhead, Groups and Slots for every composable), longer initial compilation times (compiler plugin), and the debugging story is still maturing (recomposition counts, composition tracing). Compose's recomposition can be a performance trap if stability isn't managed — unstable classes (those without @Stable or @Immutable annotations, or with mutable properties) cause unnecessary recomposition cascading through the tree.

When to choose: At Google scale, Compose for all new features — it reduces code by 30-50%, eliminates XML/binding glue, and the declarative model scales better for complex UI state. The unidirectional data flow pattern (state hoisting) is natural in Compose but requires discipline in Views. Keep Views for: performance-critical custom drawing (Canvas-heavy Views with complex paths, shaders, and bitmap manipulation are still more straightforward than Compose's DrawScope), existing large codebases where migration cost exceeds benefit (a 500-screen app can't rewrite overnight), and libraries that must support minSdk < 21. The interop layer (AndroidView in Compose, ComposeView in Views) enables gradual migration — Google's own apps (Gmail, Maps, Play Store) use this hybrid approach. For a new app at Google scale: Compose-first with View interop for specialized components (MapView, WebView, camera preview, ExoPlayer's PlayerView). The key architectural win of Compose: it eliminates the View/ViewModel/XML/DataBinding/Navigation fragmentation — one language (Kotlin), one paradigm (declarative), one state model (snapshot system).
