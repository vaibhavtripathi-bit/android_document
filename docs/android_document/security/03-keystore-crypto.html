Android Keystore & Cryptography

Hardware-backed key storage, TEE vs StrongBox, key attestation, biometric-bound keys, encryption patterns, EncryptedSharedPreferences, Play Integrity API

1. Keystore Architecture

// Generating a key in Android Keystore:
val keyGenerator = KeyGenerator.getInstance(
    KeyProperties.KEY_ALGORITHM_AES,
    "AndroidKeyStore" // Provider name — routes to hardware
)

keyGenerator.init(
    KeyGenParameterSpec.Builder(
        "my_encryption_key",
        KeyProperties.PURPOSE_ENCRYPT or KeyProperties.PURPOSE_DECRYPT
    )
        .setBlockModes(KeyProperties.BLOCK_MODE_GCM)
        .setEncryptionPaddings(KeyProperties.ENCRYPTION_PADDING_NONE)
        .setKeySize(256)
        .setIsStrongBoxBacked(true) // Use StrongBox if available
        .setUserAuthenticationRequired(true) // Require biometric/PIN
        .setUserAuthenticationParameters(
            300, // Valid for 300 seconds after auth
            KeyProperties.AUTH_BIOMETRIC_STRONG or KeyProperties.AUTH_DEVICE_CREDENTIAL
        )
        .build()
)

val secretKey = keyGenerator.generateKey()
// Key is generated INSIDE the TEE/StrongBox
// The SecretKey object is a reference — the actual key material
// never enters the app's process memory
Staff-Level Insight — Key Material Never Leaves Hardware

When you call cipher.init(Cipher.ENCRYPT_MODE, secretKey), the Cipher implementation sends the data to the TEE/StrongBox via Binder IPC to the keystore daemon, which forwards it to the secure hardware. The encryption happens inside the secure enclave, and only the ciphertext is returned to the app. The private/secret key material never enters the app's process memory. This means even if an attacker has root access and can dump the app's memory, they cannot extract the key. They can only use the key through the Keystore API (which enforces authentication requirements).

2. Key Attestation

// Key attestation proves to a remote server that a key was generated
// in genuine Android hardware (TEE/StrongBox), not in software.

// Generate an attestation key:
val keyPairGenerator = KeyPairGenerator.getInstance(
    KeyProperties.KEY_ALGORITHM_EC,
    "AndroidKeyStore"
)

keyPairGenerator.initialize(
    KeyGenParameterSpec.Builder("attestation_key", KeyProperties.PURPOSE_SIGN)
        .setAlgorithmParameterSpec(ECGenParameterSpec("secp256r1"))
        .setDigests(KeyProperties.DIGEST_SHA256)
        .setAttestationChallenge(serverNonce) // Nonce from your server
        .build()
)

val keyPair = keyPairGenerator.generateKeyPair()

// Retrieve the attestation certificate chain:
val keyStore = KeyStore.getInstance("AndroidKeyStore")
keyStore.load(null)
val certificates = keyStore.getCertificateChain("attestation_key")
// certificates[0] = key certificate (contains attestation extension)
// certificates[1..n] = chain up to Google's root CA

// Send the certificate chain to your server.
// Server verifies:
// 1. Chain is valid and roots to Google's attestation root CA
// 2. The attestation extension contains the correct challenge nonce
// 3. The key was generated in TEE/StrongBox (securityLevel field)
// 4. Device boot state (verified boot, locked bootloader)
// 5. OS version and patch level
Staff-Level Insight — Attestation for Device Integrity

Key attestation is the foundation of device integrity verification. The attestation certificate's extension (OID 1.3.6.1.4.1.11129.2.1.17) contains: security level (Software/TEE/StrongBox), verified boot state, OS version, patch level, whether the bootloader is locked, and the attestation challenge. A server can verify that: (1) the key genuinely exists in hardware, (2) the device hasn't been rooted/unlocked, (3) the OS is up-to-date. This is used by banking apps, DRM systems, and enterprise MDM. Google's Play Integrity API (successor to SafetyNet) uses key attestation internally but provides a simpler API.

3. Biometric-Bound Keys

// Keys that require biometric authentication before each use:
val keyGenSpec = KeyGenParameterSpec.Builder(
    "biometric_key",
    KeyProperties.PURPOSE_ENCRYPT or KeyProperties.PURPOSE_DECRYPT
)
    .setBlockModes(KeyProperties.BLOCK_MODE_GCM)
    .setEncryptionPaddings(KeyProperties.ENCRYPTION_PADDING_NONE)
    .setUserAuthenticationRequired(true)
    .setUserAuthenticationParameters(
        0, // 0 = require auth for EVERY crypto operation
        KeyProperties.AUTH_BIOMETRIC_STRONG
    )
    .setInvalidatedByBiometricEnrollment(true) // Key invalidated if new fingerprint added
    .build()

// Using the key with BiometricPrompt:
val cipher = Cipher.getInstance("AES/GCM/NoPadding")
val secretKey = keyStore.getKey("biometric_key", null) as SecretKey

try {
    cipher.init(Cipher.ENCRYPT_MODE, secretKey)
} catch (e: UserNotAuthenticatedException) {
    // Expected — user must authenticate first
}

val biometricPrompt = BiometricPrompt(
    activity,
    executor,
    object : BiometricPrompt.AuthenticationCallback() {
        override fun onAuthenticationSucceeded(result: AuthenticationResult) {
            // result.cryptoObject.cipher is now unlocked
            val encryptedData = result.cryptoObject!!.cipher!!.doFinal(plaintext)
        }
    }
)

biometricPrompt.authenticate(
    BiometricPrompt.PromptInfo.Builder()
        .setTitle("Authenticate")
        .setNegativeButtonText("Cancel")
        .setAllowedAuthenticators(BiometricManager.Authenticators.BIOMETRIC_STRONG)
        .build(),
    BiometricPrompt.CryptoObject(cipher)
)
setInvalidatedByBiometricEnrollment — Critical for Banking Apps

When setInvalidatedByBiometricEnrollment(true) is set, the key becomes permanently unusable if the user enrolls a new fingerprint or face. This prevents an attacker who gains physical access from adding their own biometric and using the key. Banking and payment apps must use this. The trade-off: the user must re-enroll (re-enter their password) after adding a new biometric. If you don't set this, a new biometric can unlock existing keys — a security vulnerability.

4. Encryption Patterns

// AES-GCM encryption with Keystore key:
fun encrypt(plaintext: ByteArray, keyAlias: String): Pair<ByteArray, ByteArray> {
    val keyStore = KeyStore.getInstance("AndroidKeyStore").apply { load(null) }
    val key = keyStore.getKey(keyAlias, null) as SecretKey

    val cipher = Cipher.getInstance("AES/GCM/NoPadding")
    cipher.init(Cipher.ENCRYPT_MODE, key)
    val iv = cipher.iv // GCM generates a random IV
    val ciphertext = cipher.doFinal(plaintext)
    return Pair(iv, ciphertext) // Store both — IV is needed for decryption
}

fun decrypt(iv: ByteArray, ciphertext: ByteArray, keyAlias: String): ByteArray {
    val keyStore = KeyStore.getInstance("AndroidKeyStore").apply { load(null) }
    val key = keyStore.getKey(keyAlias, null) as SecretKey

    val cipher = Cipher.getInstance("AES/GCM/NoPadding")
    cipher.init(Cipher.DECRYPT_MODE, key, GCMParameterSpec(128, iv))
    return cipher.doFinal(ciphertext)
}

5. Jetpack Security — EncryptedSharedPreferences & EncryptedFile

// EncryptedSharedPreferences — drop-in replacement for SharedPreferences:
val masterKey = MasterKey.Builder(context)
    .setKeyScheme(MasterKey.KeyScheme.AES256_GCM)
    .setRequestStrongBoxBacked(true)
    .build()

val encryptedPrefs = EncryptedSharedPreferences.create(
    context,
    "secure_prefs",
    masterKey,
    EncryptedSharedPreferences.PrefKeyEncryptionScheme.AES256_SIV, // Keys encrypted
    EncryptedSharedPreferences.PrefValueEncryptionScheme.AES256_GCM // Values encrypted
)

// Usage is identical to regular SharedPreferences:
encryptedPrefs.edit()
    .putString("auth_token", token)
    .putBoolean("biometric_enabled", true)
    .apply()

val token = encryptedPrefs.getString("auth_token", null)

// EncryptedFile — encrypt entire files:
val encryptedFile = EncryptedFile.Builder(
    context,
    File(context.filesDir, "secret_data.bin"),
    masterKey,
    EncryptedFile.FileEncryptionScheme.AES256_GCM_HKDF_4KB
).build()

// Write:
encryptedFile.openFileOutput().use { output ->
    output.write(sensitiveData)
}

// Read:
encryptedFile.openFileInput().use { input ->
    val data = input.readBytes()
}
Staff-Level Insight — MasterKey and Tink Under the Hood

Jetpack Security uses Google's Tink cryptography library internally. MasterKey is an AES-256-GCM key stored in Android Keystore. EncryptedSharedPreferences uses a two-layer encryption scheme: (1) a keyset (collection of data encryption keys) is encrypted with the MasterKey and stored alongside the prefs file, (2) each preference key is encrypted with AES-256-SIV (deterministic — same key always produces same ciphertext, needed for lookup), (3) each preference value is encrypted with AES-256-GCM (randomized — different ciphertext each time). The keyset approach means rotating the MasterKey doesn't require re-encrypting all data — only the keyset wrapper is re-encrypted.

7. KeyMint HAL — The Hardware Interface

// The full path from your app to hardware:
// App (KeyStore API) → Binder IPC → keystored daemon → KeyMint HAL → TEE/StrongBox
//
// KeyMint (Android 12+, replaces Keymaster 4.x):
// - AIDL-based HAL (not HIDL)
// - Defines operations: generateKey, importKey, begin, update, finish, abort
// - Each operation is a Binder call to the HAL service
// - The HAL service runs in a separate process with minimal privileges
//
// Key blob format:
// When a key is generated, the TEE returns an encrypted "key blob":
// - The blob is encrypted with a hardware-bound key (never extractable)
// - Stored in /data/misc/keystore2/ (Keystore 2.0, Android 12+)
// - The blob is opaque to the app and to keystored
// - Only the TEE/StrongBox can decrypt and use the key material
//
// Keystore 2.0 (Android 12+):
// - Replaced the legacy keystored with a Rust-based daemon
// - Uses SQLite for key metadata storage
// - Binder-based API (IKeystoreService)
// - Better multi-user support: keys are scoped to user ID
// - Supports key upgrade (when TEE firmware is updated)
Staff-Level Insight — StrongBox Fallback Behavior

setIsStrongBoxBacked(true) does not fall back to TEE if StrongBox is unavailable — it throws StrongBoxUnavailableException. You must handle this explicitly: try StrongBox first, catch the exception, then fall back to TEE. This is intentional — if you request StrongBox, you're making a security decision that TEE isn't sufficient. Always check packageManager.hasSystemFeature(FEATURE_STRONGBOX_KEYSTORE) before requesting StrongBox. Also note: StrongBox is significantly slower than TEE (10-100x for RSA operations) because the secure element has a much slower processor.

8. GCM IV Semantics & Nonce Safety

// AES-GCM uses a 12-byte IV (Initialization Vector / nonce).
// CRITICAL: Never reuse an IV with the same key.
// IV reuse in GCM completely breaks confidentiality AND authenticity.

// Android Keystore handles this safely:
// When you call cipher.init(ENCRYPT_MODE, key) WITHOUT specifying an IV,
// the Keystore generates a random 12-byte IV internally.
// You MUST store the IV alongside the ciphertext for decryption.

// WRONG — reusing IV:
val fixedIv = ByteArray(12) // All zeros — NEVER do this
cipher.init(Cipher.ENCRYPT_MODE, key, GCMParameterSpec(128, fixedIv))

// CORRECT — let Keystore generate IV:
cipher.init(Cipher.ENCRYPT_MODE, key) // Random IV generated
val iv = cipher.iv // Save this with the ciphertext

// Why 12 bytes? GCM is optimized for 12-byte IVs.
// Other lengths require an extra GHASH computation.
// With random 12-byte IVs, the collision probability is negligible
// for up to 2^32 (~4 billion) encryptions with the same key.

9. Edge Cases — Production Gotchas

// 1. StrongBox fallback — MUST handle explicitly:
fun generateKey(): SecretKey {
    val baseSpec = KeyGenParameterSpec.Builder("my_key",
        KeyProperties.PURPOSE_ENCRYPT or KeyProperties.PURPOSE_DECRYPT)
        .setBlockModes(KeyProperties.BLOCK_MODE_GCM)
        .setEncryptionPaddings(KeyProperties.ENCRYPTION_PADDING_NONE)

    return try {
        val generator = KeyGenerator.getInstance("AES", "AndroidKeyStore")
        generator.init(baseSpec.setIsStrongBoxBacked(true).build())
        generator.generateKey() // StrongBox
    } catch (e: StrongBoxUnavailableException) {
        val generator = KeyGenerator.getInstance("AES", "AndroidKeyStore")
        generator.init(baseSpec.setIsStrongBoxBacked(false).build())
        generator.generateKey() // TEE fallback
    }
}

// 2. KeyPermanentlyInvalidatedException — biometric enrollment changed:
fun decryptData(iv: ByteArray, ciphertext: ByteArray): ByteArray? {
    val key = keyStore.getKey("biometric_key", null) as? SecretKey ?: return null
    return try {
        val cipher = Cipher.getInstance("AES/GCM/NoPadding")
        cipher.init(Cipher.DECRYPT_MODE, key, GCMParameterSpec(128, iv))
        cipher.doFinal(ciphertext)
    } catch (e: KeyPermanentlyInvalidatedException) {
        // User enrolled a new fingerprint — key is dead
        keyStore.deleteEntry("biometric_key")
        // Must re-authenticate with password and generate a new key
        null
    } catch (e: UserNotAuthenticatedException) {
        // Need biometric auth — show BiometricPrompt
        null
    }
}

// 3. Keys unavailable before first unlock (Direct Boot):
// Keys with setUserAuthenticationRequired(true) are in Credential Encrypted (CE) storage.
// They are UNAVAILABLE until the user unlocks the device after boot.
// If your app runs in Direct Boot (android:directBootAware="true"),
// it can only access Device Encrypted (DE) storage.
// Keystore keys in CE storage throw InvalidKeyException before first unlock.

// 4. RSA signing with Keystore:
val keyPairGenerator = KeyPairGenerator.getInstance("RSA", "AndroidKeyStore")
keyPairGenerator.initialize(
    KeyGenParameterSpec.Builder("signing_key", KeyProperties.PURPOSE_SIGN)
        .setKeySize(2048)
        .setDigests(KeyProperties.DIGEST_SHA256, KeyProperties.DIGEST_SHA512)
        .setSignaturePaddings(KeyProperties.SIGNATURE_PADDING_RSA_PKCS1)
        .build()
)
val keyPair = keyPairGenerator.generateKeyPair()

val signature = Signature.getInstance("SHA256withRSA")
signature.initSign(keyPair.private) // Private key stays in TEE
signature.update(dataToSign)
val signedBytes = signature.sign()

// Verify (can be done anywhere — public key is exportable):
signature.initVerify(keyPair.public)
signature.update(dataToSign)
val isValid = signature.verify(signedBytes)
