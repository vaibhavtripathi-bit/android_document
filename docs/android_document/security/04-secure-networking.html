Secure Networking

TLS on Android, Network Security Config, certificate pinning strategies, cleartext traffic blocking, OkHttp security, deep link validation, and Certificate Transparency

1. TLS on Android — Defaults by Version

Android VersionDefault TLSCleartext HTTPSystem CA Trust
Android 4.1-4.4TLS 1.0-1.2AllowedSystem + user CAs trusted
Android 5-6TLS 1.0-1.2AllowedSystem + user CAs trusted
Android 7 (API 24)TLS 1.0-1.2AllowedOnly system CAs trusted by default (user CAs no longer trusted)
Android 9 (API 28)TLS 1.2-1.3Blocked by defaultSystem CAs only
Android 10+TLS 1.2-1.3BlockedSystem CAs only. TLS 1.3 preferred.
Android 14+TLS 1.2-1.3BlockedSystem CAs only. TLS 1.0/1.1 removed.
Staff-Level Insight — Why User CAs Were Untrusted

Before Android 7, user-installed CA certificates were trusted for all apps. This meant corporate MDM solutions (or attackers with physical access) could install a CA and MITM all HTTPS traffic. Android 7 changed the default: apps targeting API 24+ only trust system CAs. User CAs are only trusted if the app explicitly opts in via Network Security Config. This broke corporate proxies (Charles, Fiddler) for debugging — which is intentional. For debug builds, you should explicitly trust user CAs. For release builds, never.

2. Network Security Config

// res/xml/network_security_config.xml — declarative network security policy:

// Production config — strict:
<?xml version="1.0" encoding="utf-8"?>
<network-security-config>
    <!-- Block cleartext (HTTP) for all domains -->
    <base-config cleartextTrafficPermitted="false">
        <trust-anchors>
            <certificates src="system" /> <!-- Only system CAs -->
        </trust-anchors>
    </base-config>

    <!-- Certificate pinning for your API -->
    <domain-config>
        <domain includeSubdomains="true">api.example.com</domain>
        <pin-set expiration="2025-06-01">
            <pin digest="SHA-256">base64EncodedSHA256OfSPKI=</pin>
            <pin digest="SHA-256">backupPinBase64Encoded=</pin> <!-- MUST have backup -->
        </pin-set>
    </domain-config>
</network-security-config>

// Debug config — allow inspection:
<network-security-config>
    <debug-overrides>
        <trust-anchors>
            <certificates src="user" /> <!-- Trust user CAs in debug builds -->
        </trust-anchors>
    </debug-overrides>
</network-security-config>

// AndroidManifest.xml:
// <application android:networkSecurityConfig="@xml/network_security_config" ...>

3. Certificate Pinning — Strategies

StrategyWhat's PinnedRotationRisk
Leaf pinning Server's certificate SPKI hash Must update app when cert rotates High — cert rotation breaks the app
Intermediate CA pinning Intermediate CA's SPKI hash Survives leaf cert rotation Medium — CA change breaks the app
Root CA pinning Root CA's SPKI hash Survives most rotations Low — but less protection (any cert from that CA is trusted) 
// OkHttp certificate pinning:
val client = OkHttpClient.Builder()
    .certificatePinner(
        CertificatePinner.Builder()
            .add(
                "api.example.com",
                "sha256/AAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAA=", // Primary
                "sha256/BBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBB="  // Backup
            )
            .build()
    )
    .build()

// Getting the pin hash for your server:
// openssl s_client -connect api.example.com:443 | \
//   openssl x509 -pubkey -noout | \
//   openssl pkey -pubin -outform der | \
//   openssl dgst -sha256 -binary | \
//   openssl enc -base64
Pinning Can Brick Your App

If you pin a certificate and it rotates without an app update, all network requests fail. Users on old app versions are locked out. Mitigations: (1) Always include a backup pin. (2) Set an expiration date (<pin-set expiration="...">) — after expiration, pinning is disabled. (3) Pin the intermediate CA instead of the leaf certificate. (4) Have a mechanism to force-update the app. (5) Consider using the Network Security Config approach (declarative, with expiration) over OkHttp pinning (code-level, no expiration). Google recommends against pinning for most apps — the system CA trust store is sufficient.

4. Certificate Transparency (CT)

// Certificate Transparency ensures CAs can't issue rogue certificates
// without detection. Every certificate must be logged in a public CT log.

// Android 10+ enforces CT by default:
// - All certificates issued after April 2018 must have SCTs
//   (Signed Certificate Timestamps) from at least 2 CT logs
// - If SCTs are missing, the connection fails
// - This prevents a compromised CA from issuing a fake cert for your domain
//   without it appearing in public logs

// As an app developer, CT is transparent — you don't need to do anything.
// But for L6: understand that CT is a defense against CA compromise,
// not against MITM. An attacker with a rogue cert from a compromised CA
// would be detected because the cert would appear in CT logs,
// and monitoring services would alert the domain owner.

5. Deep Link & App Link Security

// Deep links can be exploited if not validated properly:

// VULNERABLE: accepting any deep link without validation
// <intent-filter>
//     <action android:name="android.intent.action.VIEW" />
//     <data android:scheme="myapp" android:host="payment" />
// </intent-filter>
// Attacker creates: myapp://payment?amount=1000&to=attacker_account

// SECURE: validate deep link parameters
override fun onCreate(savedInstanceState: Bundle?) {
    super.onCreate(savedInstanceState)
    val uri = intent.data ?: return

    // Validate scheme and host
    if (uri.scheme != "https" || uri.host != "example.com") {
        finish()
        return
    }

    // Validate and sanitize parameters
    val amount = uri.getQueryParameter("amount")?.toDoubleOrNull()
    if (amount == null || amount <= 0 || amount > MAX_AMOUNT) {
        finish()
        return
    }

    // For App Links (verified): use autoVerify
    // <intent-filter android:autoVerify="true">
    //     <data android:scheme="https" android:host="example.com" />
    // </intent-filter>
    // Requires /.well-known/assetlinks.json on your server
    // Only YOUR app can handle these links (no disambiguation dialog)
}

// Android App Links verification:
// 1. At install time, Android fetches:
//    https://example.com/.well-known/assetlinks.json
// 2. The JSON must contain your app's package name and signing cert fingerprint
// 3. If verified, your app is the default handler — no other app can intercept
// 4. This prevents phishing apps from registering the same deep link scheme
Staff-Level Insight — Intent Scheme Vulnerability

A subtle attack vector: intent:// scheme URIs in WebViews. If a WebView loads a page containing intent://example.com#Intent;scheme=myapp;end, the WebView can launch arbitrary intents. Before Android 11, this could target unexported components. Mitigation: override shouldOverrideUrlLoading() in WebViewClient and validate all intent:// URIs. Android 12+ restricts intent:// to only target exported components, but you should still validate. Never load untrusted URLs in a WebView without a custom WebViewClient that filters navigation.

6. OkHttp & Retrofit Security Best Practices

// Secure OkHttp client configuration:
val client = OkHttpClient.Builder()
    // TLS 1.2+ only (drop TLS 1.0/1.1):
    .connectionSpecs(listOf(ConnectionSpec.MODERN_TLS))

    // Timeouts to prevent slowloris attacks:
    .connectTimeout(15, TimeUnit.SECONDS)
    .readTimeout(15, TimeUnit.SECONDS)
    .writeTimeout(15, TimeUnit.SECONDS)

    // Don't follow redirects to HTTP:
    .followSslRedirects(false)

    // Interceptor to add auth headers securely:
    .addInterceptor { chain ->
        val token = tokenProvider.getToken() // From Keystore-encrypted storage
        chain.proceed(
            chain.request().newBuilder()
                .header("Authorization", "Bearer $token")
                .build()
        )
    }

    // Don't log sensitive headers in production:
    .addInterceptor(
        HttpLoggingInterceptor().apply {
            level = if (BuildConfig.DEBUG)
                HttpLoggingInterceptor.Level.BODY
            else
                HttpLoggingInterceptor.Level.NONE
        }
    )
    .build()

7. Conscrypt — Android's TLS Engine

// Android uses Conscrypt as its TLS provider — a Java Security Provider
// backed by BoringSSL (Google's fork of OpenSSL).

// The TLS stack path:
// App (OkHttp / HttpURLConnection)
//   → javax.net.ssl.SSLSocket
//     → Conscrypt SSLEngine (Java)
//       → BoringSSL (native C/C++)
//         → Kernel socket

// Conscrypt is:
// - Pre-installed as a system provider (highest priority)
// - Updatable via Google Play (Mainline module since Android 10)
// - Handles TLS handshake, certificate verification, session management
// - Implements X509TrustManager that reads Network Security Config

// How Network Security Config becomes a TrustManager:
// 1. At app startup, the framework reads res/xml/network_security_config.xml
// 2. It creates a NetworkSecurityConfig object per domain
// 3. Each config creates a TrustManager with the specified trust anchors
// 4. When a TLS connection is made, the SSLSocket gets the TrustManager
//    for the target domain
// 5. The TrustManager validates the server's certificate chain against
//    the configured trust anchors (system CAs, user CAs, or pinned certs)
// 6. If pinning is configured, the TrustManager also checks SPKI hashes

// Conscrypt as Mainline module:
// Since Android 10, Conscrypt is updatable via Play Store.
// This means TLS vulnerabilities can be patched without a full OS update.
// The module includes BoringSSL, so cipher suite support and TLS version
// support can be updated independently of the device manufacturer.
Staff-Level Insight — Conscrypt and TrustManager Internals

When you set a custom SSLSocketFactory on OkHttp, you bypass the Network Security Config's TrustManager. This is a common mistake: developers add certificate pinning via OkHttp's CertificatePinner but also set a custom SSLSocketFactory that trusts all certificates — effectively disabling pinning. For L6: understand that NSC creates a RootTrustManager that delegates to per-domain NetworkSecurityTrustManager instances. Each instance wraps the configured trust anchors and pin set. The checkServerTrusted() method first validates the chain against CAs, then checks pins, then checks CT. If any step fails, the connection is rejected.

8. TLS Session Resumption & 0-RTT

// TLS session resumption avoids the full handshake on reconnection:

// TLS 1.2 Session Tickets:
// 1. First connection: full handshake (2 round trips)
// 2. Server sends a session ticket (encrypted session state)
// 3. Client stores the ticket
// 4. Next connection: client sends the ticket → 1 round trip handshake
// 5. Server decrypts the ticket, resumes the session

// TLS 1.3 0-RTT (Early Data):
// 1. Client sends data in the FIRST packet (zero round trips)
// 2. Server can process the data immediately
// 3. RISK: 0-RTT data is replayable — an attacker can capture and resend it
// 4. NEVER use 0-RTT for non-idempotent operations (purchases, transfers)
// 5. OkHttp does NOT enable 0-RTT by default for this reason

// Android's behavior:
// - Session tickets are cached per SSLContext (typically per OkHttp client)
// - Cache survives across connections to the same host
// - Cleared when the app process dies
// - TLS 1.3 is the default on Android 10+ (via Conscrypt)
// - 0-RTT is supported but disabled by default in Conscrypt

// Forward secrecy:
// TLS 1.3 mandates forward secrecy (ECDHE key exchange).
// Even if the server's private key is compromised later,
// past session data cannot be decrypted.
// TLS 1.2 supports forward secrecy but doesn't mandate it —
// cipher suites without ECDHE (e.g., RSA key exchange) lack it.
// Android's Conscrypt prefers ECDHE cipher suites.

9. WebView TLS — The Separate Attack Surface

// CRITICAL: WebView uses its OWN TLS stack, separate from OkHttp/HttpURLConnection.
// Network Security Config does NOT apply to WebView.loadUrl().
// WebView has its own trust manager and certificate handling.

// The most dangerous callback in Android:
class MyWebViewClient : WebViewClient() {

    // WRONG — accepting all SSL errors (MITM vulnerability):
    override fun onReceivedSslError(
        view: WebView, handler: SslErrorHandler, error: SslError
    ) {
        handler.proceed() // NEVER DO THIS — allows MITM attacks
    }

    // CORRECT — reject SSL errors and show user-friendly message:
    override fun onReceivedSslError(
        view: WebView, handler: SslErrorHandler, error: SslError
    ) {
        handler.cancel() // Always cancel
        view.loadData(
            "<h3>Connection not secure</h3>",
            "text/html", "utf-8"
        )
        // Log for debugging (not in production):
        Log.w("TLS", "SSL error: ${error.primaryError}")
    }
}

// Google Play rejects apps that call handler.proceed() unconditionally.
// If you MUST handle specific errors (e.g., dev environment):
// 1. Only in debug builds
// 2. Only for specific domains
// 3. Never in production APK

10. mTLS — Client Certificates

// Mutual TLS: server verifies client AND client verifies server.
// Used in enterprise apps, banking, device-to-server authentication.

// Loading client certificate from Android Keystore:
val keyStore = KeyStore.getInstance("AndroidKeyStore")
keyStore.load(null)

val keyManagerFactory = KeyManagerFactory.getInstance("X509")
keyManagerFactory.init(keyStore, null) // No password for AndroidKeyStore

val trustManagerFactory = TrustManagerFactory.getInstance("X509")
trustManagerFactory.init(null as KeyStore?) // System trust store

val sslContext = SSLContext.getInstance("TLS")
sslContext.init(
    keyManagerFactory.keyManagers,   // Client cert
    trustManagerFactory.trustManagers, // Server cert validation
    null
)

// OkHttp with mTLS:
val client = OkHttpClient.Builder()
    .sslSocketFactory(
        sslContext.socketFactory,
        trustManagerFactory.trustManagers[0] as X509TrustManager
    )
    .build()

// Alternative: use system KeyChain (user-installed client certs):
// KeyChain.getPrivateKey(context, alias)
// KeyChain.getCertificateChain(context, alias)

11. SSLSocketFactory Bypass — Anti-Pattern

// DANGEROUS: Custom TrustManager that trusts everything:
val unsafeTrustManager = object : X509TrustManager {
    override fun checkClientTrusted(chain: Array<X509Certificate>, type: String) {}
    override fun checkServerTrusted(chain: Array<X509Certificate>, type: String) {}
    override fun getAcceptedIssuers(): Array<X509Certificate> = arrayOf()
}

// This DISABLES all certificate validation:
// - No chain verification
// - No hostname verification
// - No pinning
// - Complete MITM vulnerability

// Common excuses (all wrong):
// "It's just for testing" → Use debug-overrides in NSC instead
// "Our server has a self-signed cert" → Add it to NSC trust-anchors
// "Charles Proxy doesn't work" → Use debug-overrides with user CAs

// How to detect this in code review:
// Search for: TrustManager, checkServerTrusted, ALLOW_ALL_HOSTNAME_VERIFIER
// Google Play automated scanning flags these patterns
Staff-Level Insight — Certificate Pinning Fail-Open vs Fail-Closed

NSC pin expiration is fail-open: when pins expire, the app falls back to normal CA validation. OkHttp CertificatePinner is fail-closed: expired pins cause connection failure. For critical apps, prefer fail-closed with a backup pin rotation strategy. Keep at least two pins (primary + backup) and rotate before expiry. A pin rotation cadence of 6 months with 12-month pin lifetime gives a safe overlap window.
