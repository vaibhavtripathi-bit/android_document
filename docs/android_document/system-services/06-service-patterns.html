06 — System Service Architectural Patterns

The recurring design patterns that every Android system service follows — from AIDL boundaries to permission enforcement, dumpsys debugging, caching, cross-service communication, and HAL integration. The playbook an Architect must internalize.

1. The AIDL System Service Pattern

Every system service in Android follows the same fundamental architecture: an AIDL interface defines the contract, a Stub implementation lives in system_server, and a Proxy is generated for the app process. A thin SDK Manager class wraps the proxy to provide a clean API. This pattern is the backbone of the entire framework.

1.1 Full Pattern Walk-Through: ClipboardService

// ── Step 1: AIDL Interface Definition ──
// frameworks/base/core/java/android/content/IClipboard.aidl

interface IClipboard {
    void setPrimaryClip(in ClipData clip, String callingPackage,
                        String attributionTag, int userId);
    ClipData getPrimaryClip(String pkg, String attributionTag, int userId);
    boolean hasPrimaryClip(String pkg, String attributionTag, int userId);
    void addPrimaryClipChangedListener(in IOnPrimaryClipChangedListener listener,
                                       String pkg, String attributionTag, int userId);
    void removePrimaryClipChangedListener(in IOnPrimaryClipChangedListener listener,
                                          String pkg, String attributionTag, int userId);
}

// ── Step 2: Stub Implementation in system_server ──
// frameworks/base/services/core/java/com/android/server/clipboard/ClipboardService.kt

class ClipboardService(context: Context) : SystemService(context) {

    private inner class ClipboardImpl : IClipboard.Stub() {

        override fun setPrimaryClip(
            clip: ClipData, callingPackage: String,
            attributionTag: String?, userId: Int
        ) {
            // Permission + identity checks happen HERE at the Binder boundary
            val callingUid = Binder.getCallingUid()
            val resolvedUserId = handleIncomingUser(userId, callingUid)
            enforceClipboardAccess(callingPackage, callingUid)

            val identity = Binder.clearCallingIdentity()
            try {
                setPrimaryClipInternal(clip, resolvedUserId)
            } finally {
                Binder.restoreCallingIdentity(identity)
            }
        }

        override fun getPrimaryClip(
            pkg: String, attributionTag: String?, userId: Int
        ): ClipData? {
            val callingUid = Binder.getCallingUid()
            clipboardAccessAllowed(AppOpsManager.OP_READ_CLIPBOARD,
                pkg, callingUid, userId) || return null
            // ... return clip data
        }
    }

    override fun onStart() {
        publishBinderService(Context.CLIPBOARD_SERVICE, ClipboardImpl())
    }
}

// ── Step 3: SDK Manager in app process ──
// frameworks/base/core/java/android/content/ClipboardManager.kt

class ClipboardManager(context: Context, private val service: IClipboard) {

    fun setPrimaryClip(clip: ClipData) {
        try {
            service.setPrimaryClip(clip, context.opPackageName,
                context.attributionTag, context.userId)
        } catch (e: RemoteException) {
            throw e.rethrowFromSystemServer()
        }
    }

    fun getPrimaryClip(): ClipData? {
        try {
            return service.getPrimaryClip(context.opPackageName,
                context.attributionTag, context.userId)
        } catch (e: RemoteException) {
            throw e.rethrowFromSystemServer()
        }
    }
}

// ── Step 4: App Usage ──
val clipboard = context.getSystemService(Context.CLIPBOARD_SERVICE) as ClipboardManager
clipboard.setPrimaryClip(ClipData.newPlainText("label", "Hello"))
val clip = clipboard.getPrimaryClip()

1.2 Why This Pattern Exists

Concern How the Pattern Addresses It
Process isolation App and service run in different processes. A crash in one doesn't affect the other.
Permission enforcement Every Binder call crosses a security boundary. The Stub checks Binder.getCallingUid() and enforces permissions before executing logic.
Versioning AIDL interfaces are stable contracts. New methods can be added with default implementations; old clients still work.
Testability The Manager wraps the proxy, making it easy to mock in tests. The Stub can be tested independently.
Encapsulation Internal state (locks, caches, multi-user data) stays in system_server. Apps only see the clean Manager API.
L6 Insight — The Dual-Interface Pattern

Every major system service publishes two interfaces: a public Binder interface (e.g., IClipboard) registered with ServiceManager for app access, and an internal interface (e.g., ClipboardInternal) registered with LocalServices for system_server-internal access. The internal interface exposes privileged operations that should never cross a Binder boundary — like force-clearing the clipboard during a user switch. This dual-interface pattern is the reason LocalServices exists: it provides a zero-overhead, type-safe way for services to call each other without Binder serialization, while keeping the public API surface minimal.

2. SystemService Lifecycle

The SystemService base class defines the lifecycle contract that every service in system_server must follow. Understanding this lifecycle is essential because it dictates when a service can safely interact with other services and the outside world.

2.1 Lifecycle Callbacks

// frameworks/base/services/core/java/com/android/server/SystemService.java

abstract class SystemService(val context: Context) {

    // ── Phase 1: Service creation ──
    // Constructor is called by SystemServiceManager via reflection.
    // Do minimal work here — other services may not exist yet.

    // ── Phase 2: Service start ──
    abstract fun onStart()
    // Contract: After onStart() returns, the service MUST be able to handle
    // incoming Binder calls. This is where you call:
    //   publishBinderService(name, binder)   → registers with ServiceManager
    //   publishLocalService(type, impl)      → registers with LocalServices

    // ── Phase 3: Boot phases (progressive initialization) ──
    open fun onBootPhase(phase: Int) {}
    // Called at each phase transition. Services defer heavy initialization
    // to the phase where their dependencies are guaranteed to be ready.

    // ── Phase 4: User lifecycle ──
    open fun onUserStarting(user: TargetUser) {}    // User is being started
    open fun onUserUnlocking(user: TargetUser) {}   // CE storage about to unlock
    open fun onUserUnlocked(user: TargetUser) {}    // CE storage available
    open fun onUserSwitching(from: TargetUser?, to: TargetUser) {}
    open fun onUserStopping(user: TargetUser) {}    // User is being stopped
    open fun onUserStopped(user: TargetUser) {}     // User fully stopped

    // ── Helpers ──
    protected fun publishBinderService(name: String, service: IBinder) {
        ServiceManager.addService(name, service)
    }

    protected fun <T> publishLocalService(type: Class<T>, service: T) {
        LocalServices.addService(type, service)
    }
}

Accessible from Any process (apps, other daemons) Only within system_server

2.3 Multi-User Lifecycle Contract

// Services must handle multiple users correctly. The callback order is strict:

// ── Boot (user 0 — system user) ──
// onStart()
// onBootPhase(100) → onBootPhase(480) → ... → onBootPhase(1000)
// onUserStarting(user=0)
// onUserUnlocking(user=0)    ← after lock screen credential entered
// onUserUnlocked(user=0)     ← credential-encrypted (CE) storage mounted

// ── Secondary user switch (e.g., user 10) ──
// onUserSwitching(from=0, to=10)
// onUserStarting(user=10)
// onUserUnlocking(user=10)
// onUserUnlocked(user=10)

// ── Work profile (user 11, running concurrently with user 0) ──
// onUserStarting(user=11)     ← started in background, no switch
// onUserUnlocking(user=11)
// onUserUnlocked(user=11)

// ── User removal ──
// onUserStopping(user=10)
// onUserStopped(user=10)

// Common mistakes:
// 1. Assuming userId is always 0
// 2. Not separating per-user state (files, databases, caches)
// 3. Leaking data across user boundaries (security violation)
// 4. Not cleaning up state in onUserStopped()
Interview Trap — onStart() vs onBootPhase()

A common mistake is doing too much in onStart(). The contract says the service must be callable after onStart(), but that doesn't mean it should be fully initialized. Heavy work — registering broadcast receivers, querying other services, loading data from disk — should be deferred to the appropriate onBootPhase(). Doing it in onStart() risks accessing services that haven't started yet, causing NullPointerException or stale data. The rule: onStart() publishes interfaces; onBootPhase() wires up dependencies.

3. Permission Enforcement Patterns

Every Binder call into system_server crosses a trust boundary. The calling app is untrusted; the service must verify the caller's identity and permissions before executing any operation. Android provides a layered enforcement model.

3.1 Identifying the Caller

// Every Binder transaction carries the caller's UID and PID,
// set by the kernel (unforgeable).

val callingUid = Binder.getCallingUid()   // UID of the calling app
val callingPid = Binder.getCallingPid()   // PID of the calling process

// Derive package name from UID:
val packages = context.packageManager.getPackagesForUid(callingUid)

// Derive userId from UID:
val userId = UserHandle.getUserId(callingUid)  // UID / 100000

3.2 Permission Check Methods

// ── Check vs Enforce ──
// check*() returns PERMISSION_GRANTED or PERMISSION_DENIED
// enforce*() throws SecurityException if denied

// Check the CALLING process's permission (most common)
val result = context.checkCallingPermission(Manifest.permission.READ_CONTACTS)
if (result != PackageManager.PERMISSION_GRANTED) {
    throw SecurityException("Caller lacks READ_CONTACTS")
}

// Or use the enforce shorthand (throws SecurityException automatically):
context.enforceCallingPermission(
    Manifest.permission.READ_CONTACTS,
    "Caller must hold READ_CONTACTS"
)

// ── The Dangerous checkCallingOrSelfPermission() ──
// Checks the caller's permission OR system_server's own permission.
// system_server holds almost all permissions, so this effectively
// BYPASSES the check when called from within system_server.
context.checkCallingOrSelfPermission(Manifest.permission.READ_CONTACTS)
// ⚠️ NEVER use this unless you intentionally want system_server
// to be able to call this code path without a permission check.
// Prefer checkCallingPermission() in almost all cases.

3.3 The clearCallingIdentity / restoreCallingIdentity Pattern

// When a system service needs to call ANOTHER service on behalf of the caller,
// it must temporarily assume system_server's identity. Otherwise, the downstream
// service would see the original app's UID and might deny the call.

override fun someServiceMethod(param: String, userId: Int) {
    // 1. Verify the caller has permission
    context.enforceCallingPermission(
        Manifest.permission.SOME_PERMISSION,
        "someServiceMethod requires SOME_PERMISSION"
    )

    // 2. Save the caller's identity and assume system_server's identity
    val token = Binder.clearCallingIdentity()
    try {
        // 3. Now Binder.getCallingUid() returns system_server's UID (1000)
        // Downstream calls will see us as the system, not the original app
        val pm = context.packageManager
        val info = pm.getApplicationInfo(param, 0)  // succeeds as system
        // ... do privileged work ...
    } finally {
        // 4. ALWAYS restore — forgetting this is a security vulnerability
        Binder.restoreCallingIdentity(token)
    }
}

// The token is a long that encodes both UID and PID of the original caller.
// clearCallingIdentity() replaces them with system_server's UID/PID.
// restoreCallingIdentity() puts the original values back.

3.4 AppOps — The Secondary Enforcement Layer

// Permissions are binary (granted or denied). AppOps adds fine-grained,
// runtime-revocable control on top of permissions.

// Pattern: check permission FIRST, then check AppOps
override fun getLocation(pkg: String, userId: Int): Location? {
    // Layer 1: Permission check
    context.enforceCallingPermission(
        Manifest.permission.ACCESS_FINE_LOCATION,
        "getLocation requires ACCESS_FINE_LOCATION"
    )

    // Layer 2: AppOps check (user may have revoked via Settings)
    val appOps = context.getSystemService(AppOpsManager::class.java)!!
    val mode = appOps.noteOp(
        AppOpsManager.OP_FINE_LOCATION,
        Binder.getCallingUid(),
        pkg
    )
    if (mode != AppOpsManager.MODE_ALLOWED) {
        return null  // silently deny — don't throw
    }

    // Both checks passed — return the data
    return getLocationInternal(userId)
}

// noteOp() vs checkOp():
// noteOp()  — records the access (visible in Settings → Privacy Dashboard)
// checkOp() — checks without recording (for pre-flight checks)

3.5 Cross-User Permission Enforcement

// Accessing data across user boundaries requires special permissions:

// INTERACT_ACROSS_USERS — allows interaction with other user profiles
//   (granted to system apps, requires signature|privileged protection level)

// INTERACT_ACROSS_USERS_FULL — unrestricted cross-user access
//   (granted only to core system components, signature protection level)

// The standard pattern for resolving userId:
fun handleIncomingUser(
    callingPid: Int, callingUid: Int,
    requestedUserId: Int, allowAll: Boolean,
    requireFull: Boolean, name: String
): Int {
    val callingUserId = UserHandle.getUserId(callingUid)

    if (requestedUserId == callingUserId) {
        return requestedUserId  // same user — no cross-user check needed
    }

    if (requestedUserId == UserHandle.USER_CURRENT) {
        // Resolve to the foreground user
        return ActivityManager.getCurrentUser()
    }

    // Cross-user access — check permission
    if (requireFull) {
        context.enforceCallingPermission(
            Manifest.permission.INTERACT_ACROSS_USERS_FULL, name
        )
    } else {
        context.enforceCallingPermission(
            Manifest.permission.INTERACT_ACROSS_USERS, name
        )
    }

    return requestedUserId
}
Security Vulnerability — Forgetting restoreCallingIdentity()

If a service calls clearCallingIdentity() and an exception causes the restoreCallingIdentity() in the finally block to be skipped (e.g., the thread is interrupted or a finally block is missing entirely), all subsequent Binder calls on that thread will appear to come from system_server (UID 1000) instead of the actual caller. This means any app could trigger a code path that permanently elevates the Binder thread's identity, bypassing permission checks for all future callers on that thread. This is why the try/finally pattern is non-negotiable.

4. dumpsys — The Universal Debugging Tool

Every system service implements the dump() method, making its internal state inspectable via adb shell dumpsys. This is the single most important debugging tool for system-level Android development.

4.1 The dump() Contract

// Every Binder object inherits dump() from IBinder.
// System services override it to expose their internal state.

class ExampleService : IExample.Stub() {

    override fun dump(fd: FileDescriptor, pw: PrintWriter, args: Array<String>?) {
        // Permission check — only shell/root/system can dump
        if (!DumpUtils.checkDumpPermission(context, TAG, pw)) return

        // Parse arguments for filtering
        val dumpAll = args == null || args.isEmpty()
        val dumpProto = args?.contains("--proto") == true

        if (dumpProto) {
            dumpProto(fd)  // machine-readable protobuf output
            return
        }

        pw.println("ExampleService state:")
        pw.println("  Active sessions: ${sessions.size}")
        pw.println("  Cache hit rate: ${cacheHits * 100 / totalRequests}%")

        if (dumpAll) {
            pw.println("  Detailed session list:")
            sessions.forEach { session ->
                pw.println("    $session")
            }
        }
    }
}

5. Service Caching and Performance

Looking up a system service involves Binder IPC to the native servicemanager. Android caches aggressively at multiple layers to avoid repeated lookups.

5.1 The Three-Layer Cache

  Service Lookup Caching — Three Layers
  ══════════════════════════════════════

  App calls: context.getSystemService(Context.CLIPBOARD_SERVICE)
                              │
                              ▼
  ┌─────────────────────────────────────────────────────────┐
  │  Layer 1: ContextImpl.mServiceCache[]                    │
  │  Per-Context array of cached Manager objects              │
  │  Hit? → Return cached ClipboardManager immediately        │
  │  Miss? ↓                                                  │
  └──────────────────────────┬──────────────────────────────┘
                             ▼
  ┌─────────────────────────────────────────────────────────┐
  │  Layer 2: SystemServiceRegistry.CachedServiceFetcher     │
  │  Lazy-initializes the Manager + its Binder proxy          │
  │  Creates: IClipboard.Stub.asInterface(binder)             │
  │  Wraps in: ClipboardManager(context, proxy)               │
  │  Stores result in Layer 1 cache                           │
  │  Miss? ↓                                                  │
  └──────────────────────────┬──────────────────────────────┘
                             ▼
  ┌─────────────────────────────────────────────────────────┐
  │  Layer 3: ServiceManager.getService(name)                │
  │  First call: Binder IPC to native servicemanager          │
  │  Returns: IBinder handle (BpBinder proxy)                 │
  │  Cached in ServiceManager's static sCache map             │
  │  Subsequent calls: return from sCache (no IPC)            │
  └─────────────────────────────────────────────────────────┘

5.2 CachedServiceFetcher — The Lazy Init Pattern

// frameworks/base/core/java/android/app/SystemServiceRegistry.java

// Each system service is registered with a fetcher that lazily creates
// the Manager object on first access.

abstract class CachedServiceFetcher<T> : ServiceFetcher<T> {
    private val cacheIndex: Int = sServiceCacheSize++

    override fun getService(ctx: ContextImpl): T {
        val cache = ctx.mServiceCache
        val gates = ctx.mServiceInitializationStateArray

        var service = cache[cacheIndex] as? T
        if (service != null) return service  // Layer 1 cache hit

        // Lazy initialization — only one thread creates the service
        synchronized(gates) {
            service = cache[cacheIndex] as? T
            if (service != null) return service  // double-check

            service = createService(ctx)
            cache[cacheIndex] = service
            return service!!
        }
    }

    abstract fun createService(ctx: ContextImpl): T
}

// Registration example:
registerService(Context.CLIPBOARD_SERVICE, ClipboardManager::class.java,
    object : CachedServiceFetcher<ClipboardManager>() {
        override fun createService(ctx: ContextImpl): ClipboardManager {
            val binder = ServiceManager.getService(Context.CLIPBOARD_SERVICE)
            val service = IClipboard.Stub.asInterface(binder)
            return ClipboardManager(ctx.outerContext, service)
        }
    }
)

5.3 ServiceManager Caching Internals

// frameworks/base/core/java/android/os/ServiceManager.java

object ServiceManager {
    private val sCache = HashMap<String, IBinder>()

    fun getService(name: String): IBinder? {
        // Check local cache first
        var service = sCache[name]
        if (service != null) return service

        // Cache miss — Binder IPC to native servicemanager
        service = rawGetService(name)
        if (service != null) {
            sCache[name] = service
        }
        return service
    }

    // Used during boot when services may not be registered yet
    fun waitForService(name: String): IBinder {
        // Blocks the calling thread until the service is registered.
        // Uses servicemanager's notification mechanism internally.
        // Timeout: typically 5-10 seconds before throwing.
        return rawWaitForService(name)
    }
}

5.4 Performance Characteristics

Operation Latency Notes
context.getSystemService() (cached) ~50ns Array lookup + cast. Virtually free.
ServiceManager.getService() (cached) ~100ns HashMap lookup. No IPC.
ServiceManager.getService() (cold) ~50-100μs Binder IPC to servicemanager. Happens once per service per process.
Binder call to system service ~100-500μs Parcel serialization + context switch + deserialization. Synchronous and blocking.
Binder call with large Parcel ~1-10ms Binder transaction buffer is 1MB. Large payloads approach this limit.
L6 Insight — Batching to Reduce Binder Round-Trips

Each Binder call to a system service costs ~100-500μs due to context switches and serialization. For operations that require multiple service calls (e.g., querying package info for 50 apps), the round-trip overhead dominates. This is why several services offer batch APIs: PackageManager.getInstalledPackages() returns all packages in a single Binder call instead of per-package queries, and NotificationManager supports posting multiple notifications atomically. When designing a new system service, always consider whether callers will need batch operations — adding them later requires an AIDL version bump and is much harder to retrofit.

6. Cross-Service Communication

System services within system_server constantly communicate with each other. The communication pattern depends on whether the services are in the same process and whether the call needs to cross a trust boundary.

6.1 LocalServices — The Fast Path

// LocalServices is an in-process registry for system_server-internal APIs.
// No Binder overhead — direct Java method calls.

// Publishing an internal interface:
class NotificationManagerService(context: Context) : SystemService(context) {

    override fun onStart() {
        // Public Binder interface — for apps
        publishBinderService(Context.NOTIFICATION_SERVICE, mService)

        // Internal interface — for other system services
        publishLocalService(
            NotificationManagerInternal::class.java,
            NotificationManagerInternalImpl()
        )
    }

    private inner class NotificationManagerInternalImpl
        : NotificationManagerInternal {

        override fun cancelNotificationForPackage(pkg: String, userId: Int) {
            // No permission check — caller is system_server
            // Direct method call — no Binder serialization
            cancelNotificationInternal(pkg, userId)
        }

        override fun getNotificationChannelCount(pkg: String, userId: Int): Int {
            return mChannels.getChannelCount(pkg, userId)
        }
    }
}

// Consuming the internal interface from another service:
class PowerManagerService(context: Context) : SystemService(context) {

    override fun onBootPhase(phase: Int) {
        if (phase == PHASE_SYSTEM_SERVICES_READY) {
            val nmi = LocalServices.getService(
                NotificationManagerInternal::class.java
            )
            // Direct call — no Binder, no Parcel, no permission check
            nmi.cancelNotificationForPackage("com.example", 0)
        }
    }
}

6.2 Handler Messaging — Async Within a Service

// Most services have their own Handler thread for async work.
// Binder calls arrive on Binder threads; services post to their Handler
// to serialize state access and avoid lock contention.

class LocationManagerService(context: Context) : SystemService(context) {

    private val handler = Handler(FgThread.getHandler().looper)

    private inner class LocationServiceImpl : ILocationManager.Stub() {

        override fun requestLocationUpdates(
            request: LocationRequest,
            listener: ILocationListener,
            pkg: String
        ) {
            // This runs on a Binder thread
            enforceCallingPermission(...)

            // Post to the service's Handler thread for state mutation
            handler.post {
                // This runs on the foreground thread — serialized access
                addLocationRequest(request, listener, pkg)
            }
        }
    }
}

// Why not just use synchronized blocks?
// 1. Handler serialization avoids lock contention entirely
// 2. Binder threads aren't blocked waiting for locks
// 3. Work can be prioritized and reordered via Message priorities
// 4. Delayed work is natural: handler.postDelayed(runnable, delayMs)

7.4 hwbinder — The HAL Binder Domain

// Android uses separate Binder domains for isolation:

// /dev/binder       — framework Binder (apps ↔ system_server)
//                     Managed by servicemanager
//                     Transaction buffer: 1MB per process

// /dev/hwbinder     — HAL Binder (system_server ↔ HAL processes)
//                     Managed by hwservicemanager
//                     Separate buffer, separate thread pool

// /dev/vndbinder    — vendor Binder (vendor process ↔ vendor process)
//                     Managed by vndservicemanager
//                     Isolated from framework entirely

// Why separate domains?
// 1. Security: HAL processes can't directly access framework services
// 2. Stability: HAL crash doesn't corrupt framework Binder state
// 3. Treble: vendor code (HAL) is isolated from framework code
//    Allows OS updates without vendor participation

// With AIDL HAL (Android 11+), the trend is toward unifying
// onto /dev/binder with @VintfStability annotations for stability.
Why HAL Matters at L6

An L6 candidate should understand the full path from an API call to hardware. When asked "what happens when an app takes a photo?", the answer spans: CameraManager.openCamera() → Binder to CameraService → HIDL/AIDL to Camera HAL → ioctl() to V4L2 driver → ISP hardware. Understanding this stack lets you diagnose latency issues (is it in the framework? the HAL? the driver?), design new features that require hardware interaction, and reason about the security boundaries at each layer.

// frameworks/base/core/java/android/app/IDeviceHealthManager.aidl

interface IDeviceHealthManager {
    DeviceHealthReport getHealthReport(String callingPackage, int userId);
    void registerHealthListener(in IDeviceHealthListener listener,
                                String callingPackage, int userId);

                                  String callingPackage, int userId);

// ── Step 2: Implement the Stub in system_server ──
// frameworks/base/services/core/.../DeviceHealthManagerService.kt

class DeviceHealthManagerService(context: Context) : SystemService(context) {

    private val lock = Any()
    private val listeners = RemoteCallbackList<IDeviceHealthListener>()
    private val handler = Handler(FgThread.getHandler().looper)

    // ── Binder interface implementation ──
    private val binder = object : IDeviceHealthManager.Stub() {

        override fun getHealthReport(pkg: String, userId: Int): DeviceHealthReport {
            // Permission enforcement
            context.enforceCallingPermission(
                "android.permission.READ_DEVICE_HEALTH",
                "getHealthReport"
            )
            val callingUid = Binder.getCallingUid()
            val resolvedUser = handleIncomingUser(userId, callingUid)

            // AppOps check
            val appOps = context.getSystemService(AppOpsManager::class.java)!!
            appOps.noteOp(OP_READ_DEVICE_HEALTH, callingUid, pkg)

            // Execute as system
            val token = Binder.clearCallingIdentity()
            try {
                return getHealthReportInternal(resolvedUser)
            } finally {
                Binder.restoreCallingIdentity(token)
            }
        }

        override fun registerHealthListener(
            listener: IDeviceHealthListener, pkg: String, userId: Int
        ) {
            context.enforceCallingPermission(
                "android.permission.READ_DEVICE_HEALTH",
                "registerHealthListener"
            )
            listeners.register(listener)
        }

        override fun unregisterHealthListener(
            listener: IDeviceHealthListener, pkg: String, userId: Int
        ) {
            listeners.unregister(listener)
        }

        override fun reportHealthEvent(event: HealthEvent) {
            // oneway — fire-and-forget, no return value
            context.enforceCallingPermission(
                "android.permission.REPORT_DEVICE_HEALTH",
                "reportHealthEvent"
            )
            handler.post { processHealthEvent(event) }
        }

        override fun dump(fd: FileDescriptor, pw: PrintWriter, args: Array<String>?) {
            if (!DumpUtils.checkDumpPermission(context, TAG, pw)) return
            synchronized(lock) {
                pw.println("DeviceHealthManagerService state:")
                pw.println("  Registered listeners: ${listeners.registeredCallbackCount}")
                pw.println("  Last report: $lastReport")
            }
        }
    }

    // ── SystemService lifecycle ──
    override fun onStart() {
        publishBinderService("device_health", binder)
        publishLocalService(
            DeviceHealthManagerInternal::class.java,
            DeviceHealthInternalImpl()
        )
    }

    override fun onBootPhase(phase: Int) {
        if (phase == PHASE_SYSTEM_SERVICES_READY) {
            val pm = context.getSystemService(PowerManager::class.java)!!
            // Register for power state changes, etc.
        }
    }

    override fun onUserSwitching(from: TargetUser?, to: TargetUser) {
        handler.post { switchUserData(from?.userHandle, to.userHandle) }
    }

    // ── Internal interface for other system services ──
    private inner class DeviceHealthInternalImpl : DeviceHealthManagerInternal {
        override fun forceHealthCheck(): DeviceHealthReport {
            return getHealthReportInternal(UserHandle.USER_SYSTEM)
        }
    }

    companion object {
        private const val TAG = "DeviceHealthMgr"
    }
}

Architect Decision — Service Granularity

A common design mistake is making a system service too broad (doing too many unrelated things) or too narrow (creating many tiny services that increase Binder overhead and boot time). The rule of thumb: one service per policy domain. NotificationManagerService owns all notification policy; AlarmManagerService owns all alarm scheduling. If two features share state and policy decisions, they belong in the same service. If they're independent, split them. Each new service adds ~1-5ms to boot time and one more entry in the Binder thread pool's contention space.

9. Staff-Level Corner Cases

9.1 Binder Thread Pool Exhaustion

// system_server has a fixed pool of 31 Binder threads (set during boot):
// BinderInternal.setMaxThreads(31)

// These 31 threads are SHARED across ALL 100+ system services.
// Every synchronous Binder call from an app occupies one thread
// until the call completes.

// Exhaustion scenario:
// 1. 31 apps simultaneously call ActivityManager.getRunningAppProcesses()
// 2. Each call acquires AMS's global lock
// 3. One thread holds the lock doing expensive work (e.g., computing OOM adj)
// 4. The other 30 threads block waiting for the lock
// 5. Thread 31 is now blocked too — ALL Binder threads are occupied
// 6. No app can communicate with ANY system service
// 7. The entire device appears frozen
// 8. After 60 seconds, Watchdog kills system_server

// Mitigations:
// - Services should minimize lock hold time
// - Use Handler posting instead of synchronized blocks where possible
// - oneway calls don't consume a thread in the caller's pool
// - Android 12+ added Binder thread priority inheritance to reduce
//   priority inversion issues

// Diagnosis:
// adb shell dumpsys activity binder
// Look for: "binder thread pool (31 threads) FULL"
// Also: /proc/<pid>/task/ — count threads named "Binder:xxx_N"

9.2 Lock Ordering Violations

// system_server has a strict lock ordering hierarchy:
// AMS.mGlobalLock → WMS.mGlobalLock → PMS.mLock → NMS.mNotificationLock

// Violation example (real AOSP bug pattern):

// Thread A (Binder thread handling activity launch):
synchronized(ams.mGlobalLock) {
    // AMS needs to update window state
    synchronized(wms.mGlobalLock) {  // OK: AMS → WMS
        wms.updateWindowState()
    }
}

// Thread B (Binder thread handling window focus change):
synchronized(wms.mGlobalLock) {
    // WMS needs to notify AMS about focus change
    synchronized(ams.mGlobalLock) {  // DEADLOCK: WMS → AMS (reverse order!)
        ams.notifyFocusChanged()
    }
}

// The fix in AOSP: WMS posts a message to AMS's Handler instead of
// calling directly while holding the WMS lock:
synchronized(wms.mGlobalLock) {
    // Don't call AMS directly — post to its Handler
    ams.handler.post { ams.notifyFocusChanged() }
}

// The ATMS refactoring (Android 10) introduced a shared mGlobalLock
// between AMS and ATMS to reduce cross-lock scenarios.

9.3 clearCallingIdentity Security Pitfalls

// Pattern 1: Missing finally block — SECURITY BUG
override fun dangerousMethod(param: String) {
    val token = Binder.clearCallingIdentity()
    doSomethingThatMightThrow(param)  // If this throws...
    Binder.restoreCallingIdentity(token)  // ...this never runs!
    // All subsequent calls on this Binder thread run as UID 1000 (system)
}

// Pattern 2: Nested clearCallingIdentity — SUBTLE BUG
override fun outerMethod() {
    val outerToken = Binder.clearCallingIdentity()
    try {
        innerMethod()  // This also calls clearCallingIdentity
    } finally {
        Binder.restoreCallingIdentity(outerToken)  // Restores original caller
    }
}

fun innerMethod() {
    val innerToken = Binder.clearCallingIdentity()
    try {
        // innerToken captured the SYSTEM identity (from outer clear)
        // not the original caller's identity
    } finally {
        Binder.restoreCallingIdentity(innerToken)
        // Restores system identity — outer finally will fix it,
        // but if inner is called independently, we lose the caller
    }
}

// Pattern 3: Correct — always use try/finally
override fun correctMethod(param: String) {
    context.enforceCallingPermission(PERMISSION, "correctMethod")
    val token = Binder.clearCallingIdentity()
    try {
        doWork(param)
    } finally {
        Binder.restoreCallingIdentity(token)
    }
}

// Using a service before its boot phase → crash or stale data

class BrokenService(context: Context) : SystemService(context) {

    override fun onStart() {

    override fun onBootPhase(phase: Int) {
        if (phase == PHASE_SYSTEM_SERVICES_READY) {
            // ContentService is guaranteed to be ready at phase 500
            context.contentResolver.registerContentObserver(
                Settings.System.CONTENT_URI, true, observer

        }
    }
}

// These bugs are timing-dependent: they may work on fast devices
// where services start quickly, but fail on slow emulators or
// devices with many pre-installed apps (slower PMS scan).

9.5 oneway Binder Calls — Fire-and-Forget Semantics

// AIDL methods marked "oneway" have special semantics:

// In AIDL:
oneway void notifyHealthEvent(in HealthEvent event);

// Behavior:
// 1. The call returns IMMEDIATELY — caller doesn't wait for execution
// 2. No return value is possible (void only)
// 3. Exceptions in the Stub are NOT propagated to the caller
// 4. Calls are queued in the Binder driver and delivered asynchronously
// 5. Multiple oneway calls to the same Binder are delivered IN ORDER
// 6. The caller's Binder thread is NOT consumed (no blocking)

// When services use oneway:
// - Callbacks to app listeners (ILocationListener.onLocationChanged)
// - Event notifications (IDeviceHealthListener.onHealthChanged)
// - Fire-and-forget commands (reportHealthEvent)

// Pitfall: oneway calls can be silently dropped if the target's
// Binder buffer is full (1MB limit). The caller has no way to know.
// For critical notifications, use synchronous calls with timeouts.

// Pitfall: oneway calls from system_server to apps can pile up
// if the app is slow to process them. The Binder driver queues
// them, but if the queue fills, subsequent calls fail silently.

9.6 The Hidden API Problem

// Apps historically used reflection to call @hide internal APIs:
val method = ActivityManager::class.java.getDeclaredMethod("forceStopPackage", String::class.java)
method.invoke(activityManager, "com.target.app")

// Android 9+ introduced API restrictions:
// - Light greylist: warning logged, access allowed (for now)
// - Dark greylist: access denied for targetSdk >= 28
// - Blacklist: access always denied

// Enforcement mechanism:
// 1. At runtime, ART checks every reflective call against the API list
// 2. The list is compiled into the boot image (boot.art)
// 3. Checked via art::hiddenapi::ShouldDenyAccessToMember()

// Android 12+ tightened further:
// - Most @hide APIs are now fully blocked
// - Exemption list is shrinking with each release
// - Apps must use SDK or @SystemApi alternatives

// Impact on system services:
// - Services can't assume apps won't call internal methods
// - Internal state must be robust against unexpected call patterns
// - @SystemApi provides a stable middle ground for privileged apps

// For system service developers:
// - Mark internal APIs as @hide (excluded from SDK)
// - Expose privileged APIs as @SystemApi with signature permissions
// - Public APIs go into the SDK with full compatibility guarantees
L6 Insight — The 31-Thread Bottleneck and System Design

The 31 Binder thread limit in system_server is one of the most consequential architectural constraints in Android. Every synchronous Binder call from any app to any system service consumes one of these threads until the call completes. When designing a new system service, you must consider: (1) Can your service's operations complete in under 1ms? If not, consider oneway calls with callbacks. (2) Does your service hold locks that other services also need? If so, you're contributing to the contention that can exhaust the pool. (3) Can you batch operations to reduce the number of Binder round-trips? The thread pool is the shared resource that makes system_server a single point of failure — and why Android's entire framework can freeze when one service misbehaves.

L6 Insight — RemoteCallbackList and Death Recipients

When a system service maintains listeners registered by apps (e.g., location updates, notification listeners), it must handle app death gracefully. RemoteCallbackList is the standard pattern: it automatically unregisters callbacks when the registering process dies, via Binder death recipients (IBinder.DeathRecipient). Without this, a dead app's callback would accumulate in the service, leaking memory and causing DeadObjectException on every broadcast attempt. Every service that accepts IInterface callbacks from apps must use RemoteCallbackList or manually link a death recipient — this is a non-negotiable pattern that interviewers expect L6 candidates to know cold.

Key Takeaways for Interview
  • AIDL pattern: AIDL interface → Stub (system_server) → Proxy (app) → Manager (SDK wrapper). Every service follows this.
  • Dual interfaces: Public Binder interface for apps + internal interface via LocalServices for system_server peers.
  • SystemService lifecycle: onStart() publishes interfaces; onBootPhase() wires dependencies; user callbacks handle multi-user state.
  • Permission enforcement: Binder.getCallingUid() → enforceCallingPermission() → AppOpsManager.noteOp() → clearCallingIdentity()/restoreCallingIdentity().
  • dumpsys: Every service implements dump(). Use --priority CRITICAL for fast triage, --proto for automation.
  • Three-layer cache: ContextImpl.mServiceCache → CachedServiceFetcher → ServiceManager.sCache.
  • Cross-service comms: LocalServices (zero-overhead), Handler posting (async), strict lock ordering (AMS → WMS → PMS).
  • HAL: HIDL (legacy) → AIDL HAL (modern). Separate Binder domains (/dev/hwbinder). Full stack: app → service → HAL → driver.
  • 31 Binder threads: Shared across all services. Exhaustion = system-wide freeze. Design for minimal lock hold time.

SELinux (Security-Enhanced Linux) provides mandatory access control that governs which processes can register, find, and call system services. This is the kernel-level enforcement layer that prevents privilege escalation even on rooted devices.

// 2. getService() — looking up a service:
//    Apps must have "find" permission for the service type.
//    Policy: allow untrusted_app activity_service:service_manager find;

SELinux Domains

Domain Description Service Access
system_server The system_server process Can register and access all services
untrusted_app Third-party apps (Play Store) Can find public services only
platform_app Apps signed with platform key Can find public + some privileged services
priv_app Privileged system apps (/system/priv-app/) Extended service access, can use @SystemApi

Neverallow Rules

// neverallow rules prevent policy violations at compile time:

// Untrusted apps can NEVER access HAL services directly:
// neverallow untrusted_app hwservice_manager:hwservice_manager find;

// Untrusted apps can NEVER register system services:
// neverallow untrusted_app service_manager_type:service_manager add;

// Untrusted apps can NEVER access vendor-specific Binder:
// neverallow untrusted_app vndbinder_device:chr_file rw_file_perms;

// Debugging SELinux denials:
// adb shell dmesg | grep "avc: denied"
//   → shows all SELinux denial logs
//
// audit2allow — generates policy from denial logs:
// adb shell dmesg | audit2allow -p policy
//   → outputs the allow rule that would permit the denied action
//   → useful for debugging, NOT for production policy
Staff-Level Insight — Why Root Can't Fake Services

SELinux is the reason a rooted app can't simply call ServiceManager.addService() to register a fake system service. Even with root (UID 0), the process runs in the untrusted_app or su SELinux domain, which lacks the add permission for service_manager. The kernel enforces the policy regardless of UID — this is mandatory access control, not discretionary. To register a service, you'd need to run in the system_server domain, which requires either modifying the SELinux policy (needs a custom ROM) or injecting code into the system_server process itself. This defense-in-depth is why Android's service architecture remains secure even on rooted devices.

11. TransactionTooLargeException & Binder Buffer

Every process has a 1MB Binder transaction buffer shared across all concurrent transactions. Understanding this limit and its failure modes is essential for diagnosing one of the most common production crashes.

Buffer Architecture

// The Binder buffer is allocated during process initialization:
// ProcessState::init() → mmap(1MB) on /dev/binder
//
// Key constraints:
// - 1MB total per process (shared across ALL concurrent transactions)
// - TransactionTooLargeException thrown when a single transaction
//   exceeds ~500KB (half the buffer, to leave room for other transactions)
// - The buffer is per-process, NOT per-transaction — multiple concurrent
//   small transactions can also exhaust it

// Buffer layout:
// ┌────────────────────────────────────────────┐
// │          1MB Binder Transaction Buffer       │
// │                                              │
// │  ┌──────────┐ ┌──────────┐ ┌──────────┐    │
// │  │ Active   │ │ Active   │ │ Active   │    │
// │  │ Txn #1   │ │ Txn #2   │ │ Txn #3   │    │
// │  │ 200KB    │ │ 300KB    │ │ ???      │    │
// │  └──────────┘ └──────────┘ └──────────┘    │
// │                                              │
// │  Only 500KB remaining — next large txn fails │
// └────────────────────────────────────────────┘

Common Triggers

Trigger Typical Size Mitigation
Large Bundle in onSaveInstanceState() 100KB-1MB Use ViewModel + SavedStateHandle with minimal state
Passing Bitmap via Intent extras 1-10MB Pass URI or file path instead
Large Cursor results via ContentProvider Variable Use CursorWindow (2MB default), paginate queries
Many concurrent Binder calls Cumulative Batch operations, reduce parallelism
Large Parcelable lists in AIDL Variable Use ContentProvider or file-based IPC for large data

Behavior Across Android Versions

// Pre-Android 7 (API < 24):
// TransactionTooLargeException was SILENT — the transaction simply
// returned null or an empty result. No crash, no log. Extremely
// difficult to debug because the app appeared to work but lost data.

// Android 7+ (API 24+):
// TransactionTooLargeException is now thrown as a RuntimeException.
// The app crashes with a clear stack trace.
// This is actually BETTER — silent data loss is worse than a crash.

// Detecting the problem:
// The exception message includes the transaction size:
// "data parcel size 1048576 bytes"
// Look for this in crash logs to identify oversized Parcels.

// Proactive monitoring:
// In debug builds, enable StrictMode:
StrictMode.setVmPolicy(
    StrictMode.VmPolicy.Builder()
        .detectAll()  // includes Binder transaction size warnings
        .penaltyLog()
        .build()
)
Staff-Level Insight — The 1MB Limit is Deliberate

The 1MB Binder buffer limit is a deliberate constraint to prevent apps from monopolizing Binder driver memory. The kernel allocates this buffer via mmap during ProcessState::init(), and it's shared across all concurrent transactions in the process. Increasing it would allow a single app to consume more kernel memory, potentially starving other processes. The architectural lesson: Binder is designed for control plane communication (small commands, metadata, handles), not data plane transfers. For large data, use ParcelFileDescriptor (shared memory / file descriptors), ContentProvider (cursor windows), or MemoryFile / SharedMemory — these bypass the Binder buffer entirely.

12. Versioned AIDL

Stable AIDL (Android 10+) adds version numbers and integrity hashes to AIDL interfaces, enabling backward-compatible evolution of service contracts. This is the mechanism that allows a new framework to talk to an old HAL and vice versa.

Versioning Mechanism

// Stable AIDL interfaces include auto-generated version methods:

// Server (HAL or system service) implements:
override fun getInterfaceVersion(): Int = VERSION  // e.g., 3
override fun getInterfaceHash(): String = HASH     // e.g., "a1b2c3d4..."

// Client checks before calling new methods:
val service: IDeviceHealth = ...
if (service.interfaceVersion >= 2) {
    // Safe to call methods added in version 2
    service.getDetailedReport()
} else {
    // Fall back to version 1 behavior
    service.getBasicReport()
}

// New methods added in newer versions return default values for old clients:
// If client calls a v3 method on a v2 server, the server's Stub
// returns the default value (0, null, empty list) without crashing.

// The hash provides integrity checking:
// If the interface definition changes without a version bump,
// the hash mismatch is detected at build time.

@VintfStability

// @VintfStability marks an interface as part of the Vendor Interface (VINTF):
@VintfStability
interface IDeviceHealthHal {
    DeviceHealthData getHealthData();          // v1
    DetailedHealthData getDetailedData();      // added in v2
}

// VINTF interfaces have strict requirements:
// 1. Must be versioned — every change requires a version bump
// 2. Must be backward compatible — no removing methods, no changing signatures
// 3. Registered in device manifest (VINTF manifest XML)
// 4. Checked at boot by vintf_object — mismatches prevent boot
//
// This is how Android maintains compatibility between:
// - New framework (Android 15) + old HAL (shipped with Android 13)
// - Old framework (Android 13) + new HAL (updated by vendor)

// Used for:
// - All HAL interfaces (camera, audio, sensors, power, etc.)
// - Some system service interfaces that cross the Treble boundary
// - APEX module interfaces (Mainline modules)
Staff-Level Insight — Versioned AIDL and Treble

Versioned AIDL is the cornerstone of Android's Treble architecture — it's how the platform maintains backward compatibility between framework and HAL across OS updates. Before Treble, a new Android version required vendor participation to update HAL implementations. With versioned AIDL, a new framework can talk to an old HAL (the Stub returns defaults for new methods) and an old framework can talk to a new HAL (the client simply doesn't call new methods). This decoupling is what enabled Google to ship Android updates faster — the framework and vendor images can evolve independently as long as the versioned AIDL contract is respected.

13. oneway Binder Semantics

oneway Binder calls are fire-and-forget: the caller doesn't block, no return value is possible, and exceptions are not propagated. They are the standard pattern for callbacks from system services to apps, but their silent failure mode is a common source of bugs.

oneway Behavior

// AIDL declaration:
interface ILocationListener {
    oneway void onLocationChanged(in Location location);
    oneway void onProviderEnabled(String provider);
    oneway void onProviderDisabled(String provider);
}

// Semantics:
// 1. Caller returns IMMEDIATELY — does not wait for execution
// 2. No return value (void only)
// 3. No exception propagation — if the Stub throws, caller never knows
// 4. Calls are queued in the target's Binder buffer
// 5. Delivered asynchronously on the target's Binder thread pool

// Ordering guarantee:
// oneway calls to the SAME Binder object from the SAME process
// are delivered IN ORDER (serialized per-interface per-process).
// Call A completes before call B starts.
//
// But oneway calls to DIFFERENT Binder objects may be reordered.

The Silent Drop Problem

// CRITICAL: If the target's Binder buffer is full, oneway calls
// are SILENTLY DROPPED. No exception, no error code, no log.

// Scenario:
// 1. System service sends rapid location updates to app via oneway
// 2. App's main thread is blocked (e.g., doing disk I/O)
// 3. oneway calls queue up in the app's 1MB Binder buffer
// 4. Buffer fills up → subsequent oneway calls are dropped
// 5. App misses location updates with NO indication of the loss

// This is a common source of "missing callback" bugs:
// - Location updates stop arriving during heavy processing
// - Notification listener misses notifications during GC pauses
// - Sensor callbacks are lost during ANR-like stalls

// Mitigations:
// 1. Process oneway callbacks quickly — don't block Binder threads
// 2. For critical data, use synchronous calls with timeouts instead
// 3. Include sequence numbers in callbacks to detect gaps
// 4. Use flow control: client signals readiness before server sends
Interview Pattern — oneway Debugging

When asked "why might an app stop receiving location updates even though the provider is active?", the oneway silent drop is a Staff-level answer. The typical junior answer focuses on permissions or provider state. The Staff answer: the app's Binder buffer is full because it's processing callbacks too slowly, and oneway calls are being silently dropped by the Binder driver. The fix: offload callback processing to a background thread immediately, keeping the Binder thread free to receive the next callback. This pattern applies to all oneway callbacks — location, sensors, notifications, and any custom AIDL listener.

14. CompatChanges Framework

Android 11+ introduced the CompatChanges framework — a centralized system for gating behavioral changes behind named change IDs instead of scattered targetSdkVersion checks. This enables per-app, per-change testing without modifying the APK.

How CompatChanges Works

// Instead of scattered SDK version checks:
// BAD (old pattern):
if (targetSdkVersion >= Build.VERSION_CODES.S) {
    enforceNewBehavior()
} else {
    legacyBehavior()
}

// GOOD (CompatChanges pattern):
// Each behavioral change gets a unique ID:
@ChangeId
@EnabledSince(targetSdkVersion = Build.VERSION_CODES.S)
val ENFORCE_NEW_BEHAVIOR: Long = 123456789L

// Services check the change ID:
if (CompatChanges.isChangeEnabled(ENFORCE_NEW_BEHAVIOR, callingUid)) {
    enforceNewBehavior()
} else {
    legacyBehavior()
}

// Benefits:
// 1. Each change is independently toggleable per app
// 2. Changes are named and documented (not magic SDK version numbers)
// 3. Can be tested without changing targetSdkVersion
// 4. Centralized registry of all behavioral changes

Testing with CompatChanges

// Toggle a specific change for an app (without changing targetSdkVersion):
// adb shell am compat enable <CHANGE_ID> <package>
// adb shell am compat disable <CHANGE_ID> <package>
// adb shell am compat reset <CHANGE_ID> <package>

// List all changes and their state for an app:
// adb shell am compat list <package>
//   → shows each change ID, name, enabled/disabled, and target SDK

// Example output:
// 123456789; ENFORCE_NEW_BEHAVIOR; enabledSinceTargetSdk=31; enabled
// 987654321; RESTRICT_BACKGROUND_FGS; enabledSinceTargetSdk=31; disabled (overridden)

// This allows QA to test new behavior on an app targeting SDK 30
// by enabling individual changes, without rebuilding the APK.
Staff-Level Insight — Replacing Scattered SDK Checks

CompatChanges is the modern replacement for the hundreds of scattered if (targetSdk >= N) checks throughout AOSP. The old pattern was fragile: changes were hard to find, impossible to test individually, and created a combinatorial explosion of behaviors based on SDK version. With CompatChanges, each behavioral change is a named, documented, independently toggleable entity. This is particularly valuable for the CTS (Compatibility Test Suite) — tests can enable/disable specific changes to verify both old and new behavior without maintaining separate APKs. For system service developers, the rule is simple: never check targetSdkVersion directly — always use CompatChanges.isChangeEnabled() with a registered change ID.
