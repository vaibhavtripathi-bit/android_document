08 — Interview Q&A — System Services & Platform Internals

70 Staff Engineer questions covering system_server boot, AMS/ATMS, WMS, PMS, power/alarm/notification services, AIDL service patterns, system-level design, failure modes, profiling, and platform evolution

system_server Boot (Q1–Q8)

Q1. Walk through the Android boot sequence from power-on to the first app Activity being displayed.

Bootloader → loads the Linux kernel into RAM and jumps to it. Kernel → initializes hardware, mounts initramfs, launches /init (PID 1). init → parses init.rc and init.zygote64_32.rc, starts native daemons (servicemanager, surfaceflinger, logd), then starts Zygote. Zygote → preloads ~10,000 framework classes and common resources into the Zygote heap, then forks system_server (UID 1000, PID typically ~500). system_server → SystemServer.run() creates the system context, then starts services in three batches via SystemServiceManager: bootstrap (AMS, PMS, PowerMS), core (BatteryService, UsageStats), and other (~80+ services including WMS, IMS, NMS). Each service receives onBootPhase() callbacks as the system progresses. At PHASE_ACTIVITY_MANAGER_READY (550), AMS sends Intent.ACTION_BOOT_COMPLETED and launches the Home activity. The Launcher process is forked from Zygote, ActivityThread.main() runs, the Launcher Activity inflates, and the first frame is rendered via Choreographer → RenderThread → SurfaceFlinger. The boot animation stops when system_server sets sys.boot_completed=1.

Q2. What is SystemServiceManager and how does it differ from ServiceManager?

SystemServiceManager is a Java class inside system_server that manages the lifecycle of system services — it instantiates them, calls their onStart(), and dispatches boot phase callbacks (onBootPhase(), onUserStarting(), onUserSwitching()). It holds a list of SystemService subclass instances and iterates them during phase transitions. ServiceManager is a native daemon (separate process) that acts as the Binder name registry — the "DNS of Android." Services register their IBinder with ServiceManager.addService(name, binder) and clients look them up with ServiceManager.getService(name). ServiceManager is handle 0 in the Binder driver, reachable by every process. The key distinction: SystemServiceManager manages lifecycle within system_server's JVM; ServiceManager manages cross-process Binder discovery. A service like AMS is both: SystemServiceManager creates and lifecycle-manages it, while ServiceManager publishes its IBinder so apps can find it.

Q3. Explain boot phases — why can't a service use another service before the right phase?

SystemServiceManager defines ordered phases delivered via onBootPhase(int phase): PHASE_WAIT_FOR_DEFAULT_DISPLAY (100) — display is ready. PHASE_LOCK_SETTINGS_READY (480) — lock screen settings loaded. PHASE_SYSTEM_SERVICES_READY (500) — core services initialized and can interact. PHASE_ACTIVITY_MANAGER_READY (550) — AMS can launch apps. PHASE_THIRD_PARTY_APPS_CAN_START (600) — safe to start third-party code. PHASE_BOOT_COMPLETED (1000) — user can interact. Services are created and onStart()'d in dependency order, but their internal state may not be fully initialized until a later phase. For example, WMS can't query PMS for package info until PMS has finished scanning packages (phase 500). Calling a service before its ready phase can return null, throw IllegalStateException, or silently return stale data. A new system service must document which phase it becomes usable at and which phase it requires from its dependencies.

Q4. What happens when Watchdog detects a hung thread in system_server?

The Watchdog thread in system_server monitors critical threads (main, display, animation, I/O, foreground) and locks (AMS lock, WMS mGlobalLock, PMS lock). Every 30 seconds it posts a HandlerChecker message to each monitored handler and checks if monitored locks can be acquired. If a checker doesn't complete within 60 seconds → half-watchdog: logs a warning with thread stacks. If still stuck at 120 seconds → full watchdog: (1) dumps all thread stacks to /data/anr/traces.txt, (2) dumps kernel thread info via /proc, (3) writes a dropbox entry, (4) calls Process.killProcess(Process.myPid()) to kill system_server. Since system_server is a child of Zygote with onrestart directives in init.rc, init detects the death and restarts Zygote, which re-forks system_server, which re-registers all services. All app processes are killed — effectively a soft reboot. Common triggers: deadlocks between AMS and WMS, Binder thread pool exhaustion (all 31 threads busy), excessive disk I/O on a monitored thread, or a vendor HAL call that never returns.

Q5. How does Zygote fork system_server? What capabilities and UID does it get?

In ZygoteInit.forkSystemServer(), Zygote calls Zygote.forkSystemServer() which invokes the native fork(). The arguments specify: UID 1000 (system), GID 1000, supplemental GIDs for privileged groups (inet, net_admin, etc.), runtime flags, and the entry point com.android.server.SystemServer. After fork, the child (system_server) retains Linux capabilities that Zygote held: CAP_NET_ADMIN, CAP_NET_RAW, CAP_SYS_NICE, CAP_SYS_PTRACE, CAP_SYS_TIME, CAP_BLOCK_SUSPEND, among others — these allow it to manage network, adjust process priorities, and hold wake locks. The SELinux context transitions from u:r:zygote:s0 to u:r:system_server:s0 via selinux_android_setcontext(). Unlike app processes, system_server does NOT drop capabilities or apply seccomp filters — it needs full system access. The Zygote server socket is closed in the child since system_server should never fork further processes. system_server then calls SystemServer.main() → run() to begin service initialization.

Q6. What is LocalServices and when would you use it instead of ServiceManager?

LocalServices is an in-process registry within system_server — a simple ArrayMap<Class<?>, Object>. System services register "Internal" interfaces (e.g., ActivityManagerInternal, DisplayManagerInternal) via LocalServices.addService(). Other services in the same process look them up via LocalServices.getService(). This avoids Binder overhead entirely: no Parcel serialization, no ioctl, no thread switch — just a direct Java method call. The "Internal" interfaces expose privileged operations that should never be accessible to apps (e.g., ActivityManagerInternal.killProcess()). Use LocalServices when: (1) the caller and callee are both in system_server, (2) you need methods not exposed in the public Binder API, (3) you want to avoid the ~50-100μs per-call Binder overhead for high-frequency internal calls. Without it, system services would make thousands of unnecessary Binder round-trips to each other per second, wasting CPU and Binder thread pool capacity.

Q7. How would you optimize system_server boot time? What techniques does AOSP use?

AOSP uses several techniques: (1) Parallel service initialization — services without dependencies start concurrently on separate threads. SystemServerInitThreadPool provides a fixed thread pool for parallel init work. (2) Deferred work — heavy initialization is postponed to later boot phases. PMS defers dex optimization to PHASE_BOOT_COMPLETED. (3) Zygote preloading optimization — classes and resources are preloaded in parallel where possible; the preload list is curated to include only classes actually used during boot. (4) Cached package scanning — PMS caches scan results in packages.xml; on subsequent boots, it only rescans if APK timestamps changed. (5) Profile-guided compilation — system_server has its own Baseline Profile, so critical boot paths are AOT-compiled. (6) Lazy service initialization — some services don't fully initialize until first use (e.g., BluetoothManagerService defers Bluetooth stack init). To optimize further: profile with Perfetto's android_startup metric, identify the longest-running onStart() and onBootPhase() methods, and parallelize or defer them. OEMs often add vendor services that serialize boot — auditing these is a common optimization.

Q8. What happens to all running apps when system_server crashes?

system_server is the parent of all Java system services and the coordinator of all app lifecycles. When it crashes: (1) init detects the death via SIGCHLD. (2) The onrestart directives in init.zygote*.rc trigger a restart of Zygote (which also kills system_server's sibling processes). (3) Since all app processes depend on system_server for lifecycle management, AMS connections, and window management, they become orphaned — the kernel sends SIGKILL to all app processes. (4) Zygote restarts, re-preloads classes, re-forks system_server. (5) system_server re-initializes all services, re-registers with ServiceManager, and reaches PHASE_BOOT_COMPLETED. (6) The Launcher is re-launched, and apps restart on demand. This is a "soft reboot" — the Linux kernel stays running, but the entire Android userspace restarts. Total recovery: ~15-30 seconds. Apps that saved state via onSaveInstanceState() can restore it, but anything in ViewModel or in-memory caches is lost. This is why system_server stability is paramount — the Watchdog exists specifically to kill-and-restart rather than let it run in a corrupt state.

AMS / ATMS (Q9–Q18)

Q9. Explain the AMS/ATMS split — why was it done and what does each handle?

Before Android 10, ActivityManagerService was a monolithic class (~30,000 lines) handling activities, services, broadcasts, processes, content providers, and OOM adjustment. Android 10 split activity/task management into ActivityTaskManagerService (ATMS). ATMS handles: activity lifecycle, task/back-stack management, recent tasks, multi-window, and configuration changes. It owns RootWindowContainer, ActivityStack, and ActivityRecord. AMS retains: process management (ProcessRecord), OOM adjustment (updateOomAdjLocked()), broadcast dispatch, service lifecycle, content provider management, and the am shell command. The split improved maintainability and enabled ATMS to share the WMS lock (mGlobalLock) instead of the AMS lock, reducing lock contention between activity transitions and process management. ATMS runs in the same process as AMS (system_server) and they communicate via direct Java calls through ActivityManagerInternal / ActivityTaskManagerInternal registered in LocalServices.

Q10. Walk through updateOomAdjLocked() — how does AMS compute process importance?

OomAdjuster.updateOomAdjLocked() is called on every lifecycle transition (activity start/stop, service bind/unbind, broadcast delivery). It iterates all ProcessRecords and computes each process's oom_score_adj based on the highest-priority component it hosts. The algorithm: (1) Start at CACHED_APP_MAX_ADJ (999). (2) If hosting a foreground Activity → FOREGROUND_APP_ADJ (0). (3) If hosting a visible Activity → VISIBLE_APP_ADJ (100). (4) If running a foreground service → PERCEPTIBLE_APP_ADJ (200). (5) If bound by a foreground app with BIND_IMPORTANT → inherit the client's adj. (6) If hosting a running service → SERVICE_ADJ (500). (7) Cached processes are ranked by LRU (900-999). The computed adj is written to /proc/<pid>/oom_score_adj via ProcessList.setOomAdj(). lmkd reads these values to decide kill order. The computation also considers: provider connections (a provider serving a foreground app elevates the provider's process), instrumentation, backup agents, and heavy-weight processes. This runs frequently — profiling shows it can consume 2-5% of system_server CPU under heavy app switching.

Q11. How does broadcast dispatch work internally? What's the difference between ordered and parallel broadcasts?

When an app calls sendBroadcast(), the Intent reaches AMS via Binder. AMS resolves receivers: (1) Dynamic receivers — registered via registerReceiver(), stored in mRegisteredReceivers HashMap. (2) Static receivers — declared in manifests, resolved by PMS's IntentResolver. Parallel (normal) broadcasts: all receivers are dispatched concurrently. AMS enqueues BroadcastRecords on the BroadcastQueue and delivers them to each receiver's process in parallel via one-way Binder calls. No ordering guarantee, no result propagation. Ordered broadcasts (sendOrderedBroadcast()): receivers are sorted by android:priority and dispatched one at a time. Each receiver must finish (return from onReceive() or call goAsync().finish()) before the next is dispatched. A receiver can abortBroadcast() to stop propagation or setResultData() to pass data to the next receiver. The ANR timeout applies per-receiver: 10s foreground, 60s background. AMS uses separate queues for foreground and background broadcasts to prevent slow background receivers from blocking urgent foreground broadcasts.

Q12. Explain the ContentProvider initialization ordering trap — why does it start before Application.onCreate()?

During app process startup, ActivityThread.handleBindApplication() follows this exact order: (1) Create the Application object. (2) Call Application.attachBaseContext(). (3) Call installContentProviders() — which instantiates and calls onCreate() on every ContentProvider declared in the manifest. (4) Call Application.onCreate(). This means ContentProvider.onCreate() runs before Application.onCreate(). If your ContentProvider depends on initialization done in Application.onCreate() (DI setup, logging init, config loading), it will crash or behave incorrectly. This ordering exists because the system may need to query your ContentProvider immediately after process start (e.g., another app's pending query triggered the process launch). Many libraries exploit this ordering for auto-initialization (Firebase, WorkManager, Lifecycle) — their ContentProviders run setup code before the app even knows it's alive. The App Startup library consolidates these into a single provider to reduce the cost.

Q13. What is BroadcastModernQueue (Android 14) and why was it introduced?

Android 14 introduced BroadcastModernQueue to replace the legacy BroadcastQueue implementation. The old system used two queues (foreground and background) with a simple FIFO dispatch model that had several problems: (1) A slow receiver in one app could delay broadcasts to all other apps in the same queue. (2) Ordered broadcasts blocked the entire queue until each receiver completed. (3) ANR detection was imprecise — the timeout started at dispatch, not at the receiver's actual processing start. BroadcastModernQueue introduces per-process dispatch: each process gets its own virtual queue, so a slow receiver only delays broadcasts to that same process. It also implements priority-based scheduling — foreground-visible broadcasts are dispatched before background ones regardless of arrival order. The new queue uses a runnable-list model where pending deliveries are sorted by priority and deadline, enabling better ANR detection with precise per-delivery timeouts. This significantly reduces broadcast latency for interactive apps when background apps are processing slow broadcasts.

Q14. How does forceStopPackage() work? What does it clean up?

AMS.forceStopPackageLocked() performs a comprehensive teardown: (1) Processes — kills all processes with the package's UID via Process.killProcessQuiet(). (2) Activities — finishes all Activities in all tasks via ATMS.finishActivitiesForPackage(). (3) Services — stops all running services and removes pending start requests. (4) Broadcasts — removes all pending broadcast deliveries for the package's receivers. (5) Alarms — AlarmManagerService removes all scheduled alarms for the package. (6) Jobs — JobSchedulerService cancels all pending jobs. (7) ContentProviders — kills provider connections and removes published providers. (8) Notifications — NotificationManagerService cancels all notifications. (9) Sets the package's "stopped" state in PMS — the app won't receive implicit broadcasts until the user manually launches it again. This stopped state is the mechanism behind "App info → Force Stop" preventing background execution. The package must be explicitly launched by the user to clear the stopped flag. This is also why FLAG_INCLUDE_STOPPED_PACKAGES exists — to override this for specific broadcasts.

Q15. Explain the app process attachment flow — what happens between fork and attachApplicationLocked()?

After Zygote forks a new process: (1) The child calls ActivityThread.main() which creates the main Looper, instantiates ActivityThread, and calls thread.attach(false). (2) attach() calls AMS.attachApplication(mAppThread) via Binder — passing the app's IApplicationThread stub (the callback interface AMS uses to control the app). (3) In system_server, AMS.attachApplicationLocked(): (a) finds the ProcessRecord created earlier during the start request, (b) records the IApplicationThread reference, (c) sets the process's oom_adj, (d) calls thread.bindApplication() via Binder — sending the app's ApplicationInfo, provider list, and instrumentation info. (4) Back in the app, handleBindApplication(): creates the Application object, calls attachBaseContext(), installs ContentProviders, calls Application.onCreate(). (5) AMS then delivers the pending component: realStartActivityLocked() for an Activity, realStartServiceLocked() for a Service, or delivers a pending broadcast. The entire flow from fork to first component callback takes ~200-500ms.

Q16. What causes Binder thread pool exhaustion in system_server? How do you diagnose it?

system_server has 31 Binder threads (set via BinderInternal.setMaxThreads(31)). Exhaustion occurs when all 31 are busy handling incoming calls from apps. Common causes: (1) Lock contention — a Binder thread acquires the AMS or WMS lock, but another thread holds it for a long operation (package scan, OOM adj computation). All subsequent Binder calls needing that lock queue up. (2) Disk I/O — a Binder thread performs synchronous I/O (writing packages.xml, reading settings). (3) Boot storm — dozens of apps start simultaneously after boot, all calling AMS/PMS/WMS. (4) Cascading delays — one slow HAL call blocks a Binder thread, causing other calls to queue, eventually exhausting the pool. Diagnosis: adb shell cat /proc/binder/proc/<system_server_pid> shows thread states and pending transactions. adb shell dumpsys activity processes shows per-process Binder stats. Perfetto's binder track shows transaction durations. Thread dumps (kill -3) show what each Binder thread is blocked on. When the pool is exhausted, ALL apps experience hangs — any getSystemService() call blocks indefinitely, causing cascading ANRs across the device.

Q17. How does AMS handle implicit broadcast restrictions (Android 8+)?

Android 8.0 (API 26) restricted implicit broadcasts to reduce background execution. The enforcement is in AMS.broadcastIntentLocked(): when dispatching a broadcast, AMS checks if the Intent is implicit (no explicit component) and whether the receiver is a static (manifest-declared) receiver. If both conditions are true, AMS skips the receiver unless the broadcast is on the exemption list (defined in Intent javadoc — e.g., ACTION_BOOT_COMPLETED, ACTION_LOCALE_CHANGED, ACTION_HEADSET_PLUG). Dynamic receivers (registered via registerReceiver()) are unaffected — they always receive implicit broadcasts because their process is already running. The exemption list is hardcoded in BroadcastQueue and checked via Intent.FLAG_RECEIVER_INCLUDE_BACKGROUND. Apps targeting API 26+ that rely on implicit broadcast receivers for background work must migrate to WorkManager (for deferred work), explicit broadcasts (for app-internal communication), or JobScheduler (for system event triggers). This restriction dramatically reduced background process launches — Google reported a 30-40% reduction in background activity on Oreo devices.

Q18. Design a system to detect and prevent broadcast storms in a large app.

Detection layer: Wrap LocalBroadcastManager (or your event bus) with a monitoring decorator that counts broadcasts per type per second. Log a warning when any broadcast type exceeds a threshold (e.g., 50/s). For system broadcasts, instrument sendBroadcast() calls with a centralized counter. In debug builds, use StrictMode custom policies to flag excessive broadcast sends. Prevention layer: (1) Debounce — for state-change broadcasts (connectivity, battery), debounce with a 200ms window so rapid toggles produce one broadcast. (2) Coalesce — batch multiple data-change notifications into a single broadcast carrying a list of changed URIs. (3) Replace with reactive streams — migrate from broadcasts to StateFlow/SharedFlow for intra-process communication; these naturally conflate rapid emissions. (4) Rate-limit senders — enforce per-component rate limits at the broadcast dispatch layer. Monitoring: In production, track broadcast counts per type via your analytics pipeline. Alert when any type exceeds 2x its baseline. On the platform side, Android 14's BroadcastModernQueue already throttles per-process delivery, but app-level storms still waste CPU in the dispatch path.

WMS (Q19–Q26)

Q19. Explain the WMS window hierarchy — from RootWindowContainer to WindowState.

The hierarchy is: RootWindowContainer → DisplayContent (one per display) → DisplayArea (groups windows by type) → TaskDisplayArea → Task (back stack) → ActivityRecord → WindowState. RootWindowContainer is the root of the entire window tree, owned by ATMS/WMS. DisplayContent represents a physical or virtual display. DisplayArea organizes windows into layers: below-app (wallpaper), app (activities), and above-app (status bar, navigation bar, system dialogs). Task corresponds to a back stack and contains ActivityRecords. Each ActivityRecord has one or more WindowStates (main window, sub-windows like dialogs, popup windows). WindowState holds the window's attributes (LayoutParams), surface reference (SurfaceControl), position, size, visibility, and input channel. WMS traverses this tree during layout computation, surface assignment, and input dispatch. The Z-ordering is determined by the window's position in the tree — higher in the above-app area means drawn on top. Android 12 refactored this hierarchy to use WindowContainer as a unified base class, enabling shell transitions to animate entire subtrees.

Q20. How does input dispatch work? Walk through the path from kernel event to View.onTouchEvent().

Kernel: the touch driver generates events on /dev/input/eventN. InputReader (native thread in system_server): reads raw events via EventHub, applies device-specific mapping (touch calibration, key layout), and produces NotifyMotionArgs. InputDispatcher (native thread): receives processed events, determines the target window by hit-testing against WMS's window list (checking bounds, visibility, and input flags). It sends the event through the window's input channel — a socketpair between system_server and the app process. App process: InputEventReceiver (on the UI thread's Looper) reads the event from the socket. ViewRootImpl.processInputEvent() → InputStage pipeline: (1) NativePreImeInputStage → (2) ViewPreImeInputStage → (3) ImeInputStage (offers to IME) → (4) EarlyPostImeInputStage → (5) NativePostImeInputStage → (6) ViewPostImeInputStage → DecorView.dispatchTouchEvent() → standard View dispatch: onInterceptTouchEvent() down the hierarchy, onTouchEvent() back up. After handling, the app sends a "finished" signal back through the socket. If InputDispatcher doesn't receive this within 5 seconds, it triggers an ANR.

Q21. What is SurfaceControl and how does WMS use it to manage window surfaces?

SurfaceControl is the Java wrapper around a native SurfaceControl object that represents a node in SurfaceFlinger's layer tree. Each WindowState in WMS has an associated SurfaceControl that controls the window's surface properties: position, size, Z-order, alpha, crop, and transform. WMS creates SurfaceControls via SurfaceControl.Builder and organizes them in a hierarchy mirroring the window tree. Changes are batched into a SurfaceControl.Transaction — an atomic set of property changes applied together in a single SurfaceFlinger composition cycle. This ensures visual consistency: a window move + resize + Z-order change all appear in the same frame. WMS uses transactions for: window animations (shell transitions), inset changes, display rotation, and multi-window layout updates. The app never directly manipulates its SurfaceControl — it only draws into the Surface (buffer producer side). WMS controls where and how that surface appears on screen via SurfaceControl transactions. This separation is fundamental to Android's security model: an app can't move its own window over the status bar or another app's window.

Q22. Explain Shell transitions (Android 12+) — how do they differ from legacy AppTransition?

Legacy AppTransition was tightly coupled to WMS internals: animation definitions were hardcoded, the animation ran on WMS's animation thread, and customization required framework modifications. Shell transitions (Android 12+) decouple transition logic into the SystemUI process (the "Shell"). The flow: (1) WMS collects all window changes that constitute a transition (activity open, close, task switch). (2) WMS creates a TransitionInfo describing the participating windows, their start/end states, and their SurfaceControls. (3) WMS sends this to the Shell via IShellTransitions Binder interface. (4) The Shell's Transitions handler chooses an animation (fade, slide, shared element), creates a SurfaceControl.Transaction animation using ValueAnimator or SurfaceAnimator, and applies frame-by-frame transforms. (5) When the animation completes, the Shell notifies WMS to finalize the transition. Benefits: animations are customizable without modifying the framework, they run in a separate process (Shell crash doesn't kill system_server), and they support complex multi-window animations (predictive back, split-screen entry). This is also how OEMs customize transition animations without forking WMS.

Q23. How does WMS handle window insets? Explain InsetsState and InsetsController.

InsetsState is a snapshot of all inset sources on a display — status bar, navigation bar, IME, display cutout, caption bar. Each source has a type, visibility, and frame (the area it occupies). WMS computes InsetsState during layout and sends it to each window via IWindow.insetsChanged(). On the app side, InsetsController (in ViewRootImpl) receives the state and applies it. When an app calls WindowInsetsControllerCompat.hide(Type.systemBars()), the controller sends a request to WMS, which updates the inset source visibility and re-dispatches the new InsetsState. The IME inset is special: when the keyboard appears, InputMethodManagerService notifies WMS, which creates an IME inset source with the keyboard's height. WMS then animates the inset change, and the app receives WindowInsetsAnimation callbacks for synchronized keyboard animations (Android 11+). The WindowInsets API exposes these as getInsets(Type.systemBars()), getInsets(Type.ime()), etc. The legacy fitSystemWindows() / systemWindowInsets approach is deprecated in favor of the typed insets API. In Compose, WindowInsets.ime, WindowInsets.systemBars, and WindowInsets.safeDrawing provide reactive inset values.

Q24. What is the mGlobalLock in WMS and why is it a performance concern?

mGlobalLock (also known as WindowManagerGlobalLock) is the primary synchronization lock shared between WMS and ATMS. It protects the window hierarchy, activity records, task stacks, and display state. Any operation that reads or modifies the window tree must hold this lock: layout computation, surface transactions, activity transitions, input focus changes, and configuration updates. The performance concern: this single lock serializes ALL window and activity operations across ALL apps. During heavy multitasking (split-screen entry, rapid app switching, rotation), multiple Binder threads contend for mGlobalLock, causing queueing delays. A long-running operation under the lock (e.g., complex layout computation for a multi-window configuration) blocks all other window operations. The Watchdog monitors mGlobalLock — if it can't be acquired within 60 seconds, it triggers a half-watchdog. Mitigation strategies in AOSP: minimize work done under the lock, use mGlobalLock only for the critical section (not for I/O or Binder calls to other services), and split read-heavy paths to use concurrent data structures where possible.

Q25. How do virtual displays work? Walk through createVirtualDisplay() to window rendering.

DisplayManager.createVirtualDisplay() → Binder call to DisplayManagerService → creates a VirtualDisplayAdapter.VirtualDisplayDevice with a Surface as the output. DMS notifies WMS, which creates a new DisplayContent in the window hierarchy for this virtual display. Apps can launch Activities on the virtual display via ActivityOptions.setLaunchDisplayId(). WMS manages windows on the virtual display identically to the physical display — layout, input dispatch, and surface management all work the same. SurfaceFlinger composites the virtual display's layers into the provided output Surface. Use cases: (1) Presentation API — rendering to an external screen (Chromecast, HDMI). (2) Screen recording — MediaProjection creates a virtual display whose output surface feeds MediaCodec. (3) Multi-window testing — creating isolated display contexts. (4) App cloning — running a second instance of an app on a virtual display. Security: virtual displays can be flagged as VIRTUAL_DISPLAY_FLAG_SECURE to prevent screenshots, or VIRTUAL_DISPLAY_FLAG_PUBLIC to allow any app to present content. The CAPTURE_VIDEO_OUTPUT permission is required for creating virtual displays that mirror the main display.

Q26. Explain the window leak problem — what causes it and how do you detect it?

A window leak occurs when a WindowManager.addView() call is not matched by a removeView() before the Activity is destroyed. The WindowManagerImpl tracks all added views per Activity. In Activity.onDestroy(), WindowManagerGlobal.closeAll() checks for remaining views — if found, it logs "Activity has leaked window" and force-removes them. Common causes: (1) Dialog not dismissed — showing a Dialog (which calls addView() internally) and not dismissing it before finish(). (2) PopupWindow not dismissed — same pattern. (3) Toast on destroyed Activity — custom Toast views added via WindowManager. (4) Floating overlay — TYPE_APPLICATION_OVERLAY windows not removed. Detection: the logcat message "WindowLeaked" is the primary signal. StrictMode.VmPolicy.Builder().detectLeakedClosableObjects() catches some cases. In testing, assert that WindowManagerGlobal.getInstance().getViewRootNames() returns an empty list after Activity destruction. The leak itself wastes native memory (each window has a Surface, SurfaceControl, and input channel) and can cause crashes if the leaked view tries to draw after its context is destroyed.

PMS (Q27–Q34)

Q27. Walk through the app installation flow from PackageInstaller to dex2oat.

(1) PackageInstaller (system app) receives the APK via a Session, validates the caller's permissions, and writes the APK to a staging directory. (2) It calls PMS.installPackageLI() which: (a) verifies the APK signature (v1/v2/v3/v4 scheme), (b) parses AndroidManifest.xml via PackageParser, (c) checks version codes, permissions, shared libraries, and signing certificate continuity for updates. (3) PMS copies the APK to /data/app/<package>/, extracts native libraries to lib/, and creates the data directory /data/data/<package>/ with the assigned UID. (4) PMS calls Installer.dexopt() → installd daemon (native, runs as root) → dex2oat compiles the dex bytecode to native code (.oat/.odex) using the device's ISA and any available profile (Baseline Profile from the APK's .dm file, or cloud profile from Play). (5) PMS updates packages.xml with the new package entry. (6) PMS sends ACTION_PACKAGE_ADDED broadcast. (7) If a Baseline Profile is present, dex2oat uses it for profile-guided compilation; otherwise, it does a speed-profile or verify compilation depending on the install reason.

Q28. How does PMS resolve intents? Explain the IntentResolver matching algorithm.

IntentResolver (and its subclass ActivityIntentResolver) matches Intents against registered IntentFilters using a multi-stage algorithm: (1) Action match — the filter must contain the Intent's action (or the filter has no actions, matching all). (2) Category match — the filter must contain ALL of the Intent's categories. CATEGORY_DEFAULT is implicitly added for implicit Intents. (3) Data match — scheme, authority (host:port), path, and MIME type are matched hierarchically. A filter specifying content://com.example/* matches content://com.example/items/42. (4) Package visibility (Android 11+) — the resolved target must be visible to the caller per <queries> rules. The resolver uses a HashMap indexed by action for O(1) action lookup, then filters by category and data. For queryIntentActivities(), all matching ResolveInfos are returned sorted by priority and preference. For startActivity() with multiple matches, the system shows the disambiguation dialog (or uses the user's default app choice). The resolver also handles MIME type wildcards (image/*), URI scheme-specific authorities, and path patterns (literal, prefix, glob via ssp matching).

Q29. Explain APK signature schemes v1-v4 — what does each protect against?

v1 (JAR signing): signs individual files in the APK. Verifier checks each file's hash against MANIFEST.MF and the signature in META-INF/*.SF. Weakness: doesn't protect the ZIP metadata (file ordering, extra fields), allowing Janus-style attacks where a DEX file is prepended to the APK. v2 (APK Signature Scheme, Android 7): signs the entire APK as a binary blob. A signing block is inserted between the ZIP entries and the central directory. Any modification to any byte invalidates the signature. Protects against Janus and ZIP metadata manipulation. v3 (Android 9): extends v2 with key rotation support. Includes a proof-of-rotation chain linking old signing keys to new ones, allowing apps to rotate their signing key while maintaining update trust. v3.1 (Android 13): allows targeting rotation to specific SDK versions. v4 (Android 11): produces a separate .idsig file containing a Merkle tree hash of the APK. Enables incremental installation (ADB Incremental) — the device can verify and run the APK while it's still being streamed, because each 4KB block can be independently verified. v4 doesn't replace v2/v3; it supplements them for streaming scenarios.

Q30. What is the Computer snapshot model in PMS (Android 12+) and why was it introduced?

Before Android 12, PMS held a single mutable state protected by mLock. Every query (getPackageInfo(), resolveActivity()) acquired this lock, creating severe contention — PMS's lock was one of the most contended in system_server. Android 12 introduced the Computer snapshot model: PMS periodically creates an immutable snapshot of its state (package list, component info, permissions) as a Computer object. Read-only queries use the snapshot without acquiring any lock — they're completely lock-free. Mutations (install, uninstall, permission grant) still acquire mLock, modify the mutable state, and then publish a new snapshot. The snapshot is a ComputerEngine that implements the Computer interface, holding immutable copies of package settings, shared libraries, and component resolvers. This is essentially a copy-on-write / MVCC pattern. The result: read throughput increased dramatically (no lock contention for queries), while write operations are slightly more expensive (snapshot creation). Since reads vastly outnumber writes (apps query PMS thousands of times per second), this is a massive net win for system_server responsiveness.

Q31. How does package visibility (Android 11+) work? What is the <queries> element?

Android 11 (API 30) restricts which packages an app can see via PackageManager APIs. By default, an app can only see: (1) its own package, (2) packages it interacts with via explicit Intents, (3) packages declared in <queries> in its manifest, (4) packages visible to all (e.g., launchers with QUERY_ALL_PACKAGES permission). The <queries> element declares visibility needs: <queries><package android:name="com.example.app"/></queries> for specific packages, or <queries><intent><action .../></intent></queries> for intent-based visibility. Enforcement is in PMS's AppsFilter: when getInstalledPackages() or resolveActivity() is called, PMS filters the results through AppsFilter.shouldFilterApplication(), which checks the caller's UID against the visibility rules. This prevents apps from enumerating all installed packages (a privacy concern — fingerprinting installed apps). The QUERY_ALL_PACKAGES permission is restricted by Google Play policy to specific app categories (launchers, device management, security apps). Internally, PMS builds a visibility graph at boot time and updates it on package install/uninstall.

Q32. Explain Mainline modules and APEX — how do they change the PMS model?

Project Mainline (Android 10+) allows Google to update core system components via Google Play without a full OTA. APEX (Android Pony EXpress) is the packaging format: an APEX file is a signed container with a filesystem image (ext4 or f2fs), a manifest, and a public key. At boot, apexd daemon mounts APEX packages at /apex/<name>/, making their contents available to the system. Mainline modules include: ART runtime, media codecs, DNS resolver, networking stack (Tethering, Connectivity), permission controller, and more. Impact on PMS: (1) APEX packages are not APKs — they're managed by apexd, not PMS. However, many APEX modules contain an APK component (e.g., the Tethering module has a system app) that PMS does manage. (2) Module updates can change system behavior between Android versions — an app compiled against Android 13 might see different networking behavior after a Mainline module update on Android 13. (3) PMS must handle the split: some components are updatable (Mainline), others are not (OEM partition). (4) The ModuleInfoProvider in PMS tracks which modules are installed and their versions. For Staff Engineers, the key implication is that you can no longer assume system behavior is fixed per API level — Mainline modules introduce mid-version behavioral changes.

Q33. What happens when packages.xml is corrupted? How does the system recover?

packages.xml (in /data/system/) is PMS's persistent state: all installed packages, their UIDs, permissions, signing certificates, and component states. PMS maintains a backup: packages-backup.xml. On every write, PMS first renames the current file to the backup, writes the new file, then deletes the backup (atomic via rename()). Recovery: (1) On boot, if packages.xml is missing or corrupt, PMS checks for packages-backup.xml and restores from it. (2) If both are corrupt, PMS falls back to a full rescan: it scans all APKs in /system/app/, /system/priv-app/, /data/app/, and /vendor/app/, re-parsing each manifest and rebuilding the package database from scratch. (3) During rescan, UIDs are reassigned — this breaks per-app data directories (which are owned by the old UID), potentially causing data loss. (4) Permission grants are lost — apps must re-request runtime permissions. This scenario is catastrophic in production and typically only occurs due to filesystem corruption (sudden power loss during write, flash storage failure). The atomic write pattern with backup is specifically designed to prevent this, but it's not immune to filesystem-level corruption.

Q34. How does PMS enforce hidden API restrictions? Walk through the enforcement path.

Hidden APIs are framework methods/fields annotated with @hide in AOSP and not part of the public SDK. Enforcement (Android 9+): (1) At build time, hiddenapi tool annotates each dex member with a restriction level in the dex file's access flags: whitelist (SDK), greylist (allowed but deprecated), greylist-max-o (blocked for targetSdk > 26), blacklist (always blocked). (2) At runtime, when an app accesses a member via reflection (Class.getDeclaredMethod()), JNI (GetMethodID), or direct invocation, ART's hiddenapi::ShouldDenyAccessToMember() checks the member's restriction level against the caller's domain (SDK app, platform, test). (3) For blocked access: ART returns null/throws NoSuchMethodException for reflection, or logs a warning and allows access for greylist (with a logcat warning). (4) The enforcement is per-member, checked at the ART runtime level — not in PMS. PMS's role is providing the app's targetSdkVersion which determines which greylist thresholds apply. (5) Apps can check their hidden API usage with veridex tool or StrictMode.VmPolicy.Builder().detectNonSdkApiUsage(). The restriction levels are updated each Android release, gradually moving greylist APIs to blacklist.

Other Services (Q35–Q42)

Q35. How does PowerManagerService manage the power state machine? What triggers state transitions?

PowerManagerService (PMS-power, not to be confused with PackageManagerService) manages the device's power state via a state machine with states: Awake (screen on, full CPU), Dreaming (screen saver / ambient display), Dozing (screen off, restricted CPU), Asleep (screen off, suspended). Transitions are triggered by: user interaction (touch → Awake), timeout (inactivity → Dreaming/Asleep), proximity sensor (phone call → screen off), power button press, and programmatic wake locks. Internally, updatePowerStateLocked() runs the state machine: it evaluates all active wake locks, user activity timestamps, proximity state, and policy flags to determine the desired state. Wake locks (PowerManager.PARTIAL_WAKE_LOCK, SCREEN_DIM_WAKE_LOCK, etc.) prevent specific transitions — a PARTIAL_WAKE_LOCK keeps the CPU running even when the screen is off. PMS communicates with the HAL via IPower AIDL interface to control hardware power states (CPU frequency, suspend). The batterystats service tracks wake lock usage per-app for battery attribution. Excessive wake lock holding is flagged in Battery Historian and can trigger Google Play policy violations.

Q36. Explain AlarmManagerService batching — how does the system reduce wake-ups?

AlarmManagerService implements alarm batching to reduce device wake-ups and save battery. When an app schedules an alarm via AlarmManager.set(), the service doesn't create a 1:1 kernel timer. Instead: (1) Inexact alarms (set(), setInexactRepeating()): the service groups alarms with similar trigger times into batches. All alarms in a batch fire together, requiring only one wake-up. The batch window is proportional to the alarm's futurity — an alarm 1 hour away may be shifted ±5 minutes. (2) Exact alarms (setExact(), setExactAndAllowWhileIdle()): fire at the specified time but are still subject to Doze restrictions. Android 12+ requires the SCHEDULE_EXACT_ALARM permission. (3) Doze mode: during Doze, all alarms are deferred to maintenance windows except setAndAllowWhileIdle() (rate-limited to once per 9 minutes) and setAlarmClock() (user-visible alarm clocks are exempt). (4) App Standby Buckets (Android 9+): apps in the "Rare" bucket have alarms deferred by up to 24 hours. Internally, the service uses a sorted list of Alarm objects and a Batch list. triggerAlarmsLocked() fires all alarms in the current batch and schedules the next kernel timer via timerfd or /dev/alarm.

Q37. How does JobSchedulerService interact with App Standby Buckets and Doze?

JobSchedulerService is the central scheduler for deferred background work. It evaluates job constraints (network, charging, idle, storage, content URI changes) and batches execution to minimize wake-ups. App Standby Buckets (Android 9+) impose per-bucket job limits: Active — no restrictions. Working Set — deferred up to 2 hours. Frequent — deferred up to 8 hours, max 10 jobs per day. Rare — deferred up to 24 hours, max 5 jobs per day. Restricted (Android 12+) — max 1 job per day, only during maintenance windows. Bucket assignment is based on app usage patterns tracked by UsageStatsService. Doze mode: all jobs are deferred until maintenance windows. setExpedited() jobs (Android 12+) run sooner but still respect Doze. Interaction with WorkManager: WorkManager delegates to JobScheduler on API 23+ — so WorkManager constraints are ultimately enforced by JobSchedulerService. The service maintains a priority queue sorted by constraint satisfaction, bucket priority, and scheduling time. JobConcurrencyManager limits concurrent jobs based on device thermal state and CPU load.

Q38. Walk through the notification posting pipeline from app to SystemUI.

(1) App calls NotificationManager.notify(id, notification) → Binder call to NotificationManagerService (NMS). (2) NMS validates: checks the caller's UID, enforces notification channel requirements (Android 8+), applies rate limiting (max 25 updates/s per package), checks POST_NOTIFICATIONS permission (Android 13+). (3) NMS creates a NotificationRecord with the notification's channel, importance, group, and ranking signals. (4) RankingHelper / NotificationComparator sorts all active notifications by: importance, priority, person association, conversation status, and timestamp. (5) NMS calls buzzBeepBlinkLocked() to determine if the notification should make sound, vibrate, or flash LED based on channel settings and DND (Do Not Disturb) policy. (6) NMS sends the ranked notification list to all registered NotificationListenerServices via INotificationListener.onNotificationPosted(). (7) SystemUI (the primary listener) receives the notification, inflates the RemoteViews layout, and displays it in the status bar and notification shade. (8) For heads-up notifications (importance HIGH+), SystemUI shows a floating panel. The entire pipeline is synchronous on the NMS Binder thread — a slow RemoteViews inflation in SystemUI can back-pressure NMS.

Q39. How does ConnectivityService select the "best" network? Explain network scoring.

ConnectivityService manages network selection using a scoring system. Each network (Wi-Fi, cellular, VPN, Ethernet) is represented by a NetworkAgent that reports a network score. The score is computed from: (1) Transport type — Wi-Fi and Ethernet get higher base scores than cellular. (2) Validation status — a validated network (passed captive portal check via NetworkMonitor) gets a bonus. (3) User preference — if the user explicitly selected a network, it gets a bonus. (4) Signal strength — RSSI for Wi-Fi, signal level for cellular. (5) Metered status — unmetered networks are preferred. When a NetworkRequest is filed (e.g., an app calls requestNetwork() or the system's default request), ConnectivityService matches it against available networks by capabilities (NET_CAPABILITY_INTERNET, NET_CAPABILITY_NOT_METERED, etc.) and selects the highest-scoring match. On network change, ConnectivityService calls rematchAllNetworksAndRequests(): it re-evaluates all networks against all requests, potentially switching the default network. Apps receive callbacks via ConnectivityManager.NetworkCallback: onAvailable(), onLost(), onCapabilitiesChanged(). The NetworkMonitor component periodically probes connectivitycheck.gstatic.com to detect captive portals and validate internet access.

Q40. Explain the location provider hierarchy — GPS vs Network vs Fused.

Android's location stack has three layers: (1) GPS Provider (GnssLocationProvider) — uses satellite signals (GPS, GLONASS, Galileo, BeiDou) via the GNSS HAL. High accuracy (1-5m) but high power consumption (~50mA) and slow first fix (10-30s cold, 1-5s warm). (2) Network Provider — uses Wi-Fi AP MAC addresses and cell tower IDs, sent to a location service (Google's or OEM's) that returns an approximate location. Lower accuracy (10-100m) but fast and low power. (3) Fused Location Provider (FLP, in Google Play Services) — combines GPS, network, sensors (accelerometer, gyroscope, barometer), and Bluetooth beacons using a Kalman filter. It intelligently switches between providers based on the requested accuracy and power budget. LocationManagerService manages provider registration and dispatches location updates to apps. Apps should use the FLP via FusedLocationProviderClient (Play Services) rather than LocationManager directly. The FLP batches location updates using the hardware sensor hub (if available) to avoid waking the AP. Android 12+ added UltraWideband as a location source. Background location access requires ACCESS_BACKGROUND_LOCATION permission (Android 10+) and is rate-limited to a few updates per hour for apps in the "Rare" standby bucket.

Q41. How does NotificationListenerService work? What are the security implications?

NotificationListenerService (NLS) is a bound service that receives all notification events system-wide. The user must explicitly grant access via Settings → Notification access. When granted, NMS binds to the NLS and calls onNotificationPosted() for every notification from every app, and onNotificationRemoved() on dismissal. The service receives the full StatusBarNotification including: package name, notification text, actions, extras (which may contain message content, sender info, images), and the notification channel. Security implications: (1) An NLS can read sensitive data — OTPs, banking notifications, private messages. (2) It can dismiss notifications (cancelNotification()), potentially hiding security alerts. (3) It can snooze notifications, affecting user awareness. (4) It persists across reboots once granted. (5) The permission is binary — all or nothing, no per-app granularity. Mitigations: Android 11+ added onNotificationPosted() with NotificationListenerService.FLAG_FILTER_TYPE_CONVERSATIONS for scoped access. Apps should use Notification.VISIBILITY_SECRET for sensitive notifications to prevent content from appearing on lock screen and in NLS extras. Google Play policy restricts NLS usage to specific app categories (accessibility, device management).

Q42. Design an alarm scheduling strategy that works reliably across Doze, App Standby, and battery optimization.

Tier 1 — User-visible alarms (alarm clock, medication reminders): Use AlarmManager.setAlarmClock(). This is exempt from Doze and App Standby because the system shows an alarm icon in the status bar, signaling user intent. The system wakes the device at the exact time. Tier 2 — Time-sensitive but not user-set (calendar event reminders, scheduled syncs): Use setExactAndAllowWhileIdle() with SCHEDULE_EXACT_ALARM permission. Rate-limited to once per 9 minutes during Doze, but fires during maintenance windows. For Android 14+, use USE_EXACT_ALARM for calendar/alarm apps. Tier 3 — Deferred work with time constraints: Use WorkManager with setInitialDelay(). WorkManager delegates to JobScheduler, which respects Doze and buckets but guarantees eventual execution. Add constraints (NetworkType.CONNECTED, BatteryNotLow) for efficiency. Tier 4 — Periodic background work: Use WorkManager.enqueueUniquePeriodicWork() with a minimum interval of 15 minutes. Accept that execution may be deferred by hours in the Rare bucket. Testing: Use adb shell dumpsys alarm to verify alarm scheduling. Use adb shell dumpsys deviceidle to simulate Doze. Test in all standby buckets via adb shell am set-standby-bucket.

Service Patterns (Q43–Q50)

Q43. Explain the AIDL system service pattern — from interface definition to app usage.

The full pattern: (1) Define the AIDL interface in frameworks/base/core/java/: IMyService.aidl with methods the app can call. (2) Implement the service: MyService extends IMyService.Stub in frameworks/base/services/core/. The Stub is a Binder object. (3) Register with SystemServiceManager: create a SystemService subclass whose onStart() calls publishBinderService("my_service", new MyService()) — this registers the IBinder with ServiceManager. (4) Create the manager class: MyManager in frameworks/base/core/java/ wraps the AIDL proxy with a friendlier API, handles caching, and translates RemoteExceptions. (5) Register in SystemServiceRegistry: add a factory lambda that creates MyManager from the IBinder. (6) App usage: context.getSystemService(MyManager::class.java) → SystemServiceRegistry calls the factory → factory calls ServiceManager.getService("my_service") → IMyService.Stub.asInterface(binder) → returns MyManager wrapping the proxy. (7) SELinux policy: add my_service to service_contexts with the appropriate domain. (8) Permissions: enforce via checkCallingPermission() in each Stub method.

Q44. How does permission enforcement work at the Binder boundary? Show the complete check flow.

When an app calls a system service method, the enforcement chain is: (1) App side: the Proxy marshals args and calls transact(). The kernel stamps the caller's UID/PID from current->cred. (2) Service side (in onTransact() or the method implementation): calls checkCallingPermission(permission) or enforceCallingPermission(permission, message). (3) checkCallingPermission() → ActivityManager.checkComponentPermission() → PermissionManagerService.checkPermission(). (4) PMS checks: (a) Is the caller UID 0 (root) or 1000 (system)? → granted. (b) Does the caller's package have the permission in its manifest AND has it been granted? For runtime permissions, check the grant state in PermissionState. For install-time permissions, check the protection level matches. (5) If denied, enforceCallingPermission() throws SecurityException. Key nuance: Binder.getCallingUid() returns the remote caller's UID only if the call is cross-process. For same-process calls (via asInterface() local path), it returns YOUR UID. Always use checkCallingOrSelfPermission() if the method might be called locally. The UID is unforgeable because it's set by the kernel, not userspace.

Q45. What is clearCallingIdentity/restoreCallingIdentity and when must you use it?

Binder.clearCallingIdentity() saves the caller's UID/PID (packed into a long) and replaces them with the current process's own identity. restoreCallingIdentity(token) restores the original caller's identity. You must use this when a system service needs to perform a privileged operation on behalf of a caller after already verifying their permissions. Example: an app calls your service to write a file. You check the app's permission, then need to call another system service (e.g., ContentResolver.insert()) that would check the caller's UID. Without clearCallingIdentity(), the nested call would check the app's UID (which might lack the permission for the internal operation). With it, the nested call sees system_server's UID (1000) and succeeds.

fun writeData(data: Bundle) {
    enforceCallingPermission("com.example.WRITE", "Need WRITE permission")
    val token = Binder.clearCallingIdentity()
    try {
        contentResolver.insert(internalUri, contentValuesFrom(data))
    } finally {
        Binder.restoreCallingIdentity(token)
    }
}

Forgetting to restore is a privilege escalation vulnerability: subsequent permission checks use system_server's UID, bypassing all restrictions. Always use try/finally. This pattern has been the root cause of multiple Android CVEs.

Q46. How does Context.getSystemService() work internally? Explain the caching mechanism.

Three caching layers make repeated calls nearly free: (1) SystemServiceRegistry (static initializer block): at class load time, registers a ServiceFetcher factory for each known service name. The factory knows how to create the manager wrapper (e.g., ActivityManager, WindowManager). (2) First call: ContextImpl.getSystemService(name) → SystemServiceRegistry.getSystemService(context, name) → calls the ServiceFetcher.getService(context). The fetcher calls ServiceManager.getService("activity") (one Binder transaction to ServiceManager) to get the IBinder, then IActivityManager.Stub.asInterface(binder) to create the proxy, then wraps it in ActivityManager. The result is cached per-Context in a Object[] array indexed by service ID. (3) Subsequent calls: return the cached wrapper directly — no Binder transaction, no object creation. ServiceManager itself caches IBinder lookups in a HashMap. BinderProxy instances are also cached globally. Result: the first getSystemService() call costs one Binder round-trip (~50μs); all subsequent calls are pure Java HashMap lookups (~100ns). The CachedServiceFetcher subclass handles per-Context caching, while StaticServiceFetcher caches globally (for services that don't need a Context).

Q47. Explain the HAL layer — how does a system service communicate with hardware?

The Hardware Abstraction Layer (HAL) provides a standardized interface between Android framework services and hardware-specific vendor code. The architecture (post-Treble, Android 8+): (1) System service (Java, in system_server) calls a HAL client library (native C++) via JNI. (2) The client library communicates with the HAL server (a vendor-provided daemon running in a separate process) via Binder IPC. (3) The HAL server implements the HAL interface and talks to the kernel driver via ioctl(), sysfs, or devfs. Example: SensorService (system_server) → JNI → ISensors HAL client → Binder → sensors.default HAL server (vendor process) → /dev/sensor_hub kernel driver. The HAL interface is defined in AIDL (Android 11+) or HIDL (Android 8-10) and versioned via VINTF (Vendor Interface) manifest. The VINTF manifest (/vendor/etc/vintf/manifest.xml) declares which HAL interfaces the vendor provides, their versions, and transport (hwbinder or binder). This separation allows framework updates without vendor changes — the HAL interface is the stability boundary.

Q48. What is the difference between HIDL and AIDL HAL? Why is Android migrating to AIDL?

HIDL (Hardware Interface Definition Language, Android 8-11): a separate IDL with its own compiler (hidl-gen), transport layer (/dev/hwbinder), service manager (hwservicemanager), and C++ code generation. HIDL was designed for the Treble separation but introduced significant complexity: a separate language to learn, separate tooling, separate Binder domain, and no Java backend (only C++). AIDL HAL (Android 11+): uses the standard AIDL compiler, runs on /dev/binder (the same domain as framework services), supports both Java and C++ backends, and leverages existing AIDL versioning infrastructure. Advantages of AIDL HAL: (1) Unified tooling — one IDL language for framework and HAL interfaces. (2) Versioned AIDL — built-in version management with getInterfaceVersion()/getInterfaceHash(). (3) Java support — HAL clients in system_server can use Java directly without JNI wrappers. (4) Better testing — standard AIDL mocking tools work for HAL interfaces. (5) Reduced attack surface — one fewer Binder domain to secure. Google mandates AIDL HAL for all new HALs starting Android 13, and is migrating existing HIDL HALs to AIDL. The /dev/hwbinder domain will eventually be deprecated.

Q49. How would you design a new system service from scratch? What are the key architectural decisions?

Key decisions: (1) Threading model — will the service handle requests on Binder threads (concurrent, needs synchronization) or dispatch to a dedicated handler thread (serialized, simpler but potentially slower)? For high-throughput services, use Binder threads with fine-grained locks. For state-machine services, use a handler thread. (2) State management — mutable state protected by locks (risk: contention, deadlocks) vs. snapshot/MVCC model (like PMS's Computer) for read-heavy workloads. (3) Persistence — use AtomicFile for crash-safe writes to XML/JSON config. Consider write-ahead logging for complex state. (4) Client API — define a clean AIDL interface. Use oneway callbacks for async notifications. Provide a Manager wrapper class for the public API. (5) Permission model — define custom permissions with appropriate protection levels. Enforce at every Binder entry point. Use clearCallingIdentity() for internal operations. (6) Boot phase — determine the earliest phase your service can start and the phase at which it becomes fully functional. (7) Watchdog integration — register your locks with Watchdog if they're held for extended periods. (8) Dump support — implement dump() for adb shell dumpsys diagnostics. (9) SELinux policy — define service context, allowed callers, and Binder permissions. (10) Testing — create a mock implementation for CTS tests and unit tests.

Q50. Explain cross-service communication patterns — LocalServices vs Binder vs Handler.

LocalServices: for intra-system_server calls. Zero overhead (direct Java method call). Used when both services are in the same process and the callee exposes an "Internal" interface. Example: WMS calls ActivityManagerInternal.updateOomAdj(). Pros: fastest, no serialization. Cons: only works within system_server, no cross-process. Binder: for cross-process communication. Used when the caller and callee are in different processes (app → system_server, system_server → HAL daemon). Overhead: ~50-100μs per call (Parcel serialization + ioctl + context switch). Pros: process isolation, kernel-enforced security, death notifications. Cons: serialization cost, thread pool limits. Handler messages: for intra-process async communication. Used when a service needs to defer work to a specific thread (e.g., WMS posts layout work to the display thread). Overhead: ~1-5μs (message enqueue + dequeue). Pros: thread confinement (avoids locks), ordered processing. Cons: single-threaded (can become a bottleneck), no cross-process. Pattern selection: same-process + synchronous → LocalServices. Same-process + async → Handler. Cross-process → Binder. High-frequency same-process → direct method call with shared lock (but beware contention). The choice directly impacts latency, throughput, and deadlock risk.

System Design / Staff-Level (Q51–Q58)

Q51. You notice system_server is consuming 80% CPU — walk through your investigation.

Step 1 — Identify the hot threads: adb shell top -H -p <system_server_pid> shows per-thread CPU usage. Identify which threads are consuming CPU: Binder threads (handling app requests), main thread (processing messages), or specific service threads (display, animation). Step 2 — Profile: capture a Perfetto trace with sched, am, wm, binder categories. Look for: (a) hot functions in the CPU profile, (b) excessive Binder transactions (high call rate from a misbehaving app), (c) lock contention (threads spinning on mGlobalLock or AMS lock). Step 3 — Check specific services: adb shell dumpsys activity — check if OOM adj computation is running excessively (many process state changes). dumpsys window — check for animation loops or layout thrashing. dumpsys package — check for ongoing package scans. Step 4 — Identify the root cause: common culprits: (a) an app sending thousands of Binder calls/second (check /proc/binder/stats), (b) broadcast storm (check dumpsys activity broadcasts), (c) WMS layout loop (a window continuously requesting layout), (d) GC pressure (check logcat | grep "GC" for system_server). Step 5 — Remediate: throttle the offending app, fix the service-side bottleneck, or file a platform bug if it's a framework issue.

Q52. Design the architecture for a new system service that manages device sensors with batching, power optimization, and multi-client support.

AIDL interface: ISensorService.aidl with methods: registerListener(ISensorCallback, sensorType, samplingPeriodUs, maxBatchReportLatencyUs), unregisterListener(ISensorCallback), flush(ISensorCallback). Use oneway for ISensorCallback.onSensorChanged(SensorEvent[]) to avoid blocking the HAL thread. Architecture: (1) HAL communication — bind to the ISensors AIDL HAL. Configure hardware batching via batch() — the sensor hub accumulates events and delivers them in batches, reducing AP wake-ups. (2) Multi-client multiplexing — maintain a HashMap<SensorType, List<ClientRegistration>>. When multiple clients request the same sensor at different rates, configure the HAL at the fastest rate and downsample for slower clients. (3) Power optimization — use the hardware FIFO for batching. When all clients allow high latency, increase batch size to reduce wake-ups. When the screen is off, switch to wake-up sensors only. (4) Client lifecycle — use DeathRecipient on each client's IBinder to auto-unregister dead clients. Use RemoteCallbackList for thread-safe callback dispatch. (5) Rate limiting — enforce minimum sampling period per sensor type to prevent battery drain. (6) Permissions — BODY_SENSORS for heart rate, HIGH_SAMPLING_RATE_SENSORS (Android 12+) for >200Hz. (7) Dump — implement dump() showing active clients, sensor rates, batch sizes, and power estimates.

Q53. How would you debug a deadlock between AMS and WMS?

Detection: The Watchdog will fire because it can't acquire the AMS lock or mGlobalLock within 60 seconds. The traces.txt dump shows the deadlock: Thread A (Binder thread) holds AMS lock, waiting for mGlobalLock. Thread B (display thread) holds mGlobalLock, waiting for AMS lock. Analysis: (1) Read traces.txt — find the two threads with circular lock dependencies. (2) Identify the call paths: Thread A is likely in an AMS method that calls into WMS (e.g., updateConfigurationLocked() → WMS.computeNewConfiguration()). Thread B is likely in a WMS method that calls back into AMS (e.g., relayoutWindow() → AMS.updateOomAdj()). (3) Check if this is a known pattern — the AMS/ATMS split was specifically designed to reduce these deadlocks by having ATMS share mGlobalLock with WMS instead of the AMS lock. Fix: (a) Ensure consistent lock ordering: always acquire mGlobalLock before AMS lock, never the reverse. (b) Reduce the scope of locked sections — don't hold one lock while calling into another service. (c) Use Handler.post() to defer the cross-service call outside the lock. (d) Use LocalServices with lock-free internal interfaces where possible. (e) Add the lock pair to Watchdog's monitor list for early detection.

Q54. Explain the security model for system services — SELinux, permissions, Binder identity.

Three layers of defense: (1) SELinux (Mandatory Access Control): every process has an SELinux context (e.g., u:r:untrusted_app:s0). The SELinux policy defines which contexts can perform Binder calls to which service contexts. service_contexts maps service names to SELinux types (e.g., activity u:object_r:activity_service:s0). A malicious app can't even reach a HAL service because SELinux denies the binder_call permission between untrusted_app and hal_sensors_default domains. (2) Android permissions: each service method checks the caller's permissions via enforceCallingPermission(). Permissions have protection levels: normal (auto-granted), dangerous (runtime prompt), signature (same signing key), signatureOrSystem (platform-signed or priv-app). (3) Binder identity: the kernel stamps caller UID/PID on every transaction from current->cred — unforgeable from userspace. The service reads it via Binder.getCallingUid(). Combined: SELinux prevents unauthorized IPC at the kernel level, permissions provide fine-grained access control at the application level, and Binder identity ensures the caller can't impersonate another app. clearCallingIdentity() is the controlled mechanism for privilege elevation within system_server, always paired with restoreCallingIdentity() in a finally block.

Q55. A Mainline module update broke your app's behavior — how do you investigate and work around it?

Investigation: (1) Identify the module: check adb shell pm list packages --apex-only and adb shell dumpsys package <module_package> for version info. Compare the module version on the broken device vs. a working device. (2) Check the module's changelog: Mainline modules publish release notes in the Google System Updates changelog. (3) Reproduce: use adb install-multi-package to install specific module versions (if you have the APEX files from a factory image). (4) Bisect: if you have access to multiple module versions, binary search for the breaking version. (5) Analyze the behavioral change: use Perfetto traces, logcat, and dumpsys for the affected service to understand what changed. Workaround: (1) Runtime version check: query the module version via PackageManager.getModuleInfo() and branch behavior accordingly. (2) Feature flag: gate the affected code path behind a feature flag so you can disable it for affected module versions via remote config. (3) Compatibility shim: wrap the affected API call in a try-catch and fall back to an alternative implementation. (4) Report the bug: file a bug on the Android issue tracker with the module version, device info, and reproduction steps. Mainline module bugs are prioritized because they affect all devices. (5) Long-term: avoid relying on undocumented behavior of system services — Mainline modules can change behavior between monthly updates.

Q56. Design a system to monitor and alert on system service health in production.

Data collection: (1) ApplicationExitInfo — on each app launch, query getHistoricalProcessExitReasons() for REASON_ANR, REASON_CRASH, REASON_LOW_MEMORY. Extract thread dumps from ANR exits. (2) Binder latency — instrument your getSystemService() calls with timing wrappers. Track p50/p95/p99 latency for each service (AMS, PMS, WMS). A sudden spike indicates system_server contention. (3) Broadcast delivery latency — timestamp when you send a broadcast and when onReceive() fires. Delays >1s indicate broadcast queue congestion. (4) Process state tracking — use ActivityManager.RunningAppProcessInfo to monitor your app's importance level. Unexpected transitions to IMPORTANCE_CACHED indicate aggressive LMK behavior. Alerting: (a) Binder p95 latency >500ms → alert (system_server likely under pressure). (b) ANR rate >0.2% per build → alert and block rollout. (c) Unexpected process kills >5% of sessions → investigate LMK thresholds. Dashboard: track metrics per device model, OS version, and Mainline module version. Correlate spikes with module updates or OS patches. This gives you early warning of platform regressions before they show up in Play Vitals (which has a 48-hour delay).

Q57. How does the platform handle backward compatibility when system service APIs change?

Multiple mechanisms: (1) SDK version gating: system services check ApplicationInfo.targetSdkVersion and branch behavior. Example: PMS enforces package visibility only for apps targeting API 30+. This is pervasive in AOSP — search for targetSdkVersion checks in any service. (2) Compatibility framework (Android 11+): CompatChanges defines named change IDs that can be toggled per-app. Services check CompatChanges.isChangeEnabled(CHANGE_ID, uid). This allows enabling new behavior for specific apps during testing without changing targetSdk. (3) Versioned AIDL: for HAL interfaces and some system service interfaces, versioned AIDL ensures old clients work with new servers. New methods return default values for old clients. (4) @Deprecated + @Removed: APIs are deprecated for 2+ versions before removal. Hidden API restrictions gradually block reflection access to removed APIs. (5) CTS (Compatibility Test Suite): ensures all OEM devices implement the same API behavior. CTS tests run against the public API contract. (6) Mainline module versioning: modules maintain backward compatibility within a major version. Breaking changes require a new module version with migration support. For Staff Engineers, the key insight is that behavioral changes are almost always gated on targetSdkVersion — upgrading targetSdk is when you must adapt to new platform behavior.

Q58. You're an Android platform architect — design the migration strategy for moving a system service into a Mainline module.

Phase 1 — Interface stabilization: Define a stable AIDL interface for the service. All app-facing APIs must go through this interface — no direct access to internal classes. Version the interface with getInterfaceVersion(). Create a "System API" surface for privileged callers (OEM apps, other system services) separate from the public API. Phase 2 — Decouple from system_server: Move the service implementation into a separate process. Replace LocalServices calls with Binder calls to the new process. This validates that the service can run independently. Update SELinux policies for the new process domain. Phase 3 — APEX packaging: Package the service as an APEX module with: the service binary, AIDL interface definitions, a manifest declaring the HAL/service interface version, and a preinstalled APK for the manager class. Register with apexd for mount-at-boot. Phase 4 — Compatibility testing: Run CTS and VTS (Vendor Test Suite) against the modularized service. Test upgrade/downgrade scenarios. Verify that apps compiled against the old monolithic service still work. Phase 5 — Rollout: Ship as a Mainline module update via Google Play. Monitor crash rates, ANR rates, and API latency. Maintain a rollback mechanism via apexd rollback. Key risks: increased IPC latency (was LocalServices, now Binder), boot ordering changes (APEX mounts before system_server starts), and version skew between the module and system_server.

Failure Modes, Profiling & Evolution (Q59–Q70)

Q59. What happens if a bootstrap service throws an exception in onStart()? How does the system recover?

system_server crashes because bootstrap services are critical — there's no graceful degradation. The exception propagates up through SystemServiceManager.startService() to SystemServer.run(), which has no catch block for bootstrap failures. Zygote detects the child death via waitpid(), kills all app processes (they depend on system_server for lifecycle management, Binder connections, and window management), and re-forks system_server. The device appears to "reboot" but the kernel stays running — this is a runtime restart, not a full reboot. If this happens repeatedly (typically 3 consecutive failures within a short window), the bootloader may trigger recovery mode to prevent a boot loop. The key insight is that system_server has no partial-failure model for bootstrap services — it's all-or-nothing. This is why bootstrap service code must be extremely defensive: validate all inputs, handle missing hardware gracefully, and never depend on external state that might be corrupt.

Q60. What happens when a ContentProvider's process dies while clients are waiting for query results?

It depends on whether the client acquired a stable or unstable provider reference. Stable clients (the default for ContentResolver.query()) crash along with the provider — AMS kills the client process because it assumes the client's state may be corrupted by the incomplete transaction. Unstable clients (acquired via ContentResolver.acquireUnstableContentProvider()) receive a DeadObjectException and can retry. MediaStore uses unstable providers specifically because the media scanner process can crash without killing the gallery app. The cleanup flow: AMS calls removeDyingProviderLocked() → removes the ContentProviderRecord from the registry → notifies all waiting clients → stable clients are killed via Process.killProcess(), unstable clients get DeadObjectException on their pending Binder call. This distinction is critical for system design: if your app queries a ContentProvider in another process, consider whether a provider crash should take your app down or whether you can recover gracefully.

Q61. Explain the PROC_START_TIMEOUT — what is it and what triggers it?

When AMS asks Zygote to fork a new app process, it sets a timeout: 10 seconds on normal devices, 20 seconds on slow devices, and 120 seconds when a debugger is attached. If the new process doesn't call AMS.attachApplicationLocked() within this timeout, AMS kills the process and logs "Timeout waiting for process to attach." Common causes: heavy Application.onCreate() work, slow ContentProvider initialization (all providers start before Application.onCreate()), excessive class loading or dex verification, and blocking I/O during static initializers. This is not an ANR — it's a process-level health check that fires before the app even reaches its first component callback. The fix is to defer heavy initialization to background threads, lazy-load components, use the App Startup library to consolidate ContentProvider initialization, and profile the cold start path with Perfetto's android_startup data source to identify the bottleneck.

Q62. How do you profile system_server boot time? What tools and metrics do you use?

The primary tool is Perfetto with the android_startup data source — it traces boot phases, per-service startup times, dex loading, and Binder calls during boot. Key metrics to track: (1) Time from Zygote fork to PHASE_BOOT_COMPLETED (total boot duration), (2) Per-service onStart() duration to identify slow services, (3) dex2oat time during boot for system_server and preloaded classes, (4) Time to first frame of Launcher (user-perceived boot time). Commands: adb shell dumpsys boottime shows per-phase timing with millisecond precision. Perfetto categories to enable: am, wm, pm, dalvik, binder_driver. For deeper analysis: start an atrace session before reboot with adb shell atrace --async_start -b 32768 am wm pm dalvik, then collect after boot completes. Google uses boot-time benchmarks as a release gate — regressions in boot time block Android releases. For OEM devices, vendor services added to system_server are the most common source of boot time regressions and should be the first audit target.

Q63. How do you use Perfetto to trace a Binder call from app to system service?

Enable Perfetto with binder_driver, am, and sched categories. The trace shows the complete round-trip: (1) App thread enters binder_transaction — the trace slice shows caller PID/UID and target service name. (2) Context switch to system_server's Binder thread — visible as a sched_switch event. (3) The service method execution appears as a slice in the system_server process track (e.g., AMS.getRunningAppProcesses). (4) Reply transaction back to the app — another binder_transaction slice. (5) App thread resumes with the result. Key metrics to extract: total round-trip time, time spent in the service method itself, time waiting for a free Binder thread (visible as a gap between the transaction start and the service method start — indicates thread pool pressure), and time waiting for locks within the service. For production monitoring, instrument your getSystemService() calls with timing wrappers and report p50/p95/p99 latency per service to your analytics pipeline — this gives early warning of system_server contention before it manifests as ANRs.

Q64. How do you debug lmkd killing your app's process? What tools reveal the kill reason?

Five tools provide visibility: (1) ApplicationExitInfo API (Android 11+): call getHistoricalProcessExitReasons() — returns REASON_LOW_MEMORY with the oom_score_adj at kill time, the PSI level, and the RSS at death. (2) logcat: search for "lowmemorykiller" or "lmkd" tags — shows PID, oom_adj, and the memory pressure level that triggered the kill. (3) dumpsys meminfo: shows per-process memory usage (PSS, RSS, private dirty) and current oom_adj scores for all processes. (4) dumpsys activity processes: shows all ProcessRecords with their current oom_adj, the reason for the adj assignment, and bound service connections. (5) /proc/pressure/memory: shows PSI (Pressure Stall Information) — the metric lmkd uses to decide when to kill. To prevent kills: reduce your app's memory footprint, respond to onTrimMemory() callbacks by releasing caches and image references, ensure your process has the highest possible oom_adj (maintain bound services, foreground work), and profile with heapprofd to find native memory leaks.

Q65. What changed in Android 12 for PackageManagerService? Explain the Computer snapshot model.

Before Android 12, PMS used a global mLock for all reads and writes — every resolveIntent(), getPackageInfo(), and checkPermission() call competed for the same lock, making it the biggest source of lock contention in system_server. Android 12 introduced the Computer snapshot model (essentially MVCC): reads operate on an immutable snapshot of package state (lock-free), while writes are serialized and create a new snapshot. The snapshot is a ComputerEngine implementing the Computer interface, holding immutable copies of package settings, component resolvers, and shared library state. It's invalidated on every mutation (install, uninstall, permission grant, component toggle). Trade-offs: memory overhead (~2-4MB per snapshot for 200 packages) and GC pressure from snapshot churn during high-frequency mutations like batch installs. This pattern is directly analogous to PostgreSQL's MVCC or Compose's snapshot system — connecting it to database concurrency patterns or Compose internals demonstrates systems-thinking beyond Android-specific knowledge.

Q66. How did BroadcastModernQueue (Android 14) change broadcast dispatch?

The legacy BroadcastQueue used a single ordered list per queue (foreground/background) — a slow receiver blocked all subsequent receivers in the same queue. BroadcastModernQueue introduces per-process queues: each receiving process has its own virtual queue, so a slow receiver in Process A doesn't block delivery to Process B. It also supports deferred delivery to cached processes — broadcasts are held until the process is needed for other reasons, reducing unnecessary process starts and saving battery. The modern queue improves ANR attribution by tracking per-receiver timing more precisely: the timeout starts when the receiver actually begins processing, not when the broadcast was dispatched. The migration was gradual — Android 14 uses the modern queue by default but falls back to the legacy queue for specific broadcast types that require strict global ordering guarantees (e.g., ACTION_BOOT_COMPLETED with ordered delivery). For app developers, the practical impact is reduced broadcast latency when other apps are slow, and more accurate ANR reporting in Play Vitals.

Q67. What is the CompatChanges framework and how does it replace targetSdkVersion checks?

Before Android 11, behavioral changes were gated by scattered if (targetSdk >= N) checks throughout AOSP — hundreds of them, each independently maintained, with no central registry. CompatChanges (Android 11+) replaces this with named change IDs that can be toggled per-app. Services check CompatChanges.isChangeEnabled(CHANGE_ID, uid) instead of raw SDK version checks. Benefits: (1) Individual changes can be tested without changing targetSdk — toggle one change at a time via adb shell am compat enable/disable CHANGE_ID package. (2) Per-app toggling enables gradual migration and targeted debugging. (3) A centralized registry of all behavioral changes makes it possible to audit what changes affect your app. (4) Better CTS coverage because each change ID has associated tests. For app developers: adb shell am compat list package shows all change IDs and their current state (enabled/disabled) for your app. This is invaluable when debugging behavioral differences after a targetSdk bump — you can bisect which specific change ID causes the regression.

Q68. Walk through what happens when a user sets an alarm clock — which system services are involved and how do they coordinate?

This single user action touches 6+ system services: (1) App calls AlarmManager.setAlarmClock() → Binder call to AlarmManagerService → stores the alarm with RTC_WAKEUP type → programs a kernel alarm timer via timerfd. (2) Device enters Doze → DeviceIdleController puts the system in IDLE state → but setAlarmClock() bypasses Doze restrictions because it's treated as a user-visible alarm (the alarm icon appears in the status bar). (3) At alarm time: the kernel RTC wakes the CPU → AlarmManagerService fires the PendingIntent → AMS starts the alarm app's BroadcastReceiver or Activity. (4) The alarm app posts a notification with fullScreenIntent → NotificationManagerService delivers it with HIGH importance → SystemUI shows a heads-up or full-screen notification. (5) PowerManagerService turns on the screen via userActivity() triggered by the notification. (6) AudioService plays the alarm sound at the alarm volume level, respecting Do Not Disturb exceptions. Understanding these cross-service coordination flows — and the failure modes at each step — is exactly what L6 interviews test.

Q69. How does SELinux enforce access to system services? What happens if an app tries to access a service it's not allowed to?

SELinux enforcement happens at the ServiceManager level. The service_contexts file maps service names to SELinux types (e.g., activity → u:object_r:activity_service:s0). When an app calls ServiceManager.getService("activity"), the native servicemanager daemon checks if the caller's SELinux domain (e.g., u:r:untrusted_app:s0) has find permission for the service's SELinux type. If denied: the call returns null (not a crash) — the app simply can't find the service, and getSystemService() returns null. Similarly, addService() checks add permission — only system_server's domain can register most services. For HAL services: a separate hwbinder domain with hwservicemanager prevents apps from directly accessing hardware services. The neverallow rules in the SELinux policy ensure that even if a vulnerability allows code execution in an app's context, it can't escalate to accessing privileged services — the kernel enforces the policy regardless of what userspace code does. This is defense-in-depth: SELinux blocks at the IPC layer, permissions block at the API layer, and Binder identity prevents impersonation.

Q70. Explain the lock ordering convention in system_server — why does it exist and what happens when it's violated?

system_server has a strict lock ordering convention: mGlobalLock (shared by ATMS and WMS) → AMS lock → PMS lock → other service locks. This ordering prevents deadlocks: if Thread A holds mGlobalLock and needs the AMS lock, it can safely acquire it because no other thread should hold the AMS lock while waiting for mGlobalLock. But if Thread B holds the AMS lock and tries to acquire mGlobalLock, it violates the ordering and risks deadlock. The convention exists because services frequently call each other: WMS calls AMS.updateOomAdj() during window operations, AMS calls WMS.computeNewConfiguration() during config changes. Violation detection: the Watchdog monitors lock acquisition — if a thread holds a lock for >60 seconds, it dumps all thread stacks to /data/anr/traces.txt, revealing the deadlock cycle. The AMS/ATMS split (Android 10) was specifically designed to reduce lock conflicts by having ATMS share mGlobalLock with WMS instead of using the separate AMS lock — this eliminated an entire class of AMS↔WMS deadlocks that plagued earlier Android versions.
