03 — Navigation API Reference

Every important Navigation Component API — NavController, NavHost, NavOptions, design patterns, known bugs, and the full evolution from manual FragmentTransactions to type-safe @Serializable routes

1. Navigation Component Evolution Timeline

🔄 API Evolution — Navigation Component (2010–2024)
Pre-Navigation (2010–2018): Manual FragmentTransaction.replace(), startActivity() with Intent flags, hand-rolled back stack management. No standard pattern — every team invented their own router. Back stack corruption was trivial, deep links required manual TaskStackBuilder, and testing navigation meant testing Fragment state machines by hand.
Navigation 1.0 (2018 — Google I/O): NavController, NavGraph (XML), NavHostFragment, Safe Args plugin. Standardized the single-Activity architecture. Solved: type-safe arguments, automatic deep link handling, back stack management. Problems: destination IDs were resource integers (fragile), no multiple back stacks, no Compose support, XML-only graph definition.
Navigation 2.0–2.3 (2019–2020): Dynamic feature module support via DynamicNavHostFragment. Kotlin DSL for programmatic graph building. Incremental stability improvements. Still no multiple back stacks — bottom navigation required ugly workarounds (Google's own NavigationExtensions.kt sample).
Navigation 2.4 (Jan 2022): Multiple back stacks — saveState/restoreState in NavOptions. Game-changer for bottom navigation. Also introduced NavigationUI integration with multiple back stacks. Two-pane layout support via AbstractListDetailFragment.
Navigation 2.5–2.6 (2022–2023): CreationExtras for ViewModel construction. Compose Navigation improvements — better integration with Lifecycle and SavedStateHandle. String-based routes became the Compose convention (replacing integer IDs), but with no compile-time checking.
Navigation 2.7 (2023): AnimatedNavHost merged into NavHost — all Compose navigation gets enter/exit animations by default. Predictive back gesture support for cross-fade and shared element transitions. NavHost now uses AnimatedContent internally.
Navigation 2.8 (2024): Type-safe routes with @Serializable. Replaced string-based "profile/{userId}" with compile-time-checked Kotlin objects. Kotlinx.serialization generates route strings automatically. String routes still work for backward compatibility, but new code should use @Serializable exclusively.

2. NavController — Core Methods

navigate(resId) / navigate(directions) / navigate(route)

fun navigate(@IdRes resId: Int, args: Bundle? = null, navOptions: NavOptions? = null)
fun navigate(directions: NavDirections, navOptions: NavOptions? = null)
fun navigate(route: T, navOptions: NavOptions? = null) // 2.8+ @Serializable

The primary navigation API. Three overloads reflecting three eras: resource IDs (XML graphs), Safe Args directions (generated code), and type-safe routes (Kotlin serialization).

Where it breaks

  • Double-navigation crash — tapping a button twice quickly fires two navigate() calls. The first changes currentDestination, so the second call's action ID is no longer valid on the new destination. Throws IllegalArgumentException: Navigation action/destination cannot be found from the current destination. No built-in protection.
  • "Action not found" from wrong destination — if you define an action only on Screen A but the user somehow triggers it after navigating to Screen B (race condition, delayed coroutine), the same crash occurs.
  • Global actions bypass this — global actions are defined at the graph level and are reachable from any destination. Use them for navigation triggered by asynchronous events.
// The double-navigation crash
binding.button.setOnClickListener {
    findNavController().navigate(R.id.action_home_to_detail) // first tap: OK
    // second tap (before Fragment transition completes):
    // currentDestination is now "detail", which doesn't have action_home_to_detail
    // → IllegalArgumentException
}

// Fix 1: check currentDestination before navigating
fun NavController.safeNavigate(actionId: Int) {
    currentDestination?.getAction(actionId)?.let { navigate(actionId) }
}

// Fix 2: use currentBackStackEntryAsState() in Compose
val navBackStackEntry by navController.currentBackStackEntryAsState()
val currentRoute = navBackStackEntry?.destination?.route
Button(onClick = {
    if (currentRoute == "home") {
        navController.navigate("detail")
    }
})

popBackStack()

fun popBackStack(): Boolean
fun popBackStack(destinationId: Int, inclusive: Boolean): Boolean
fun popBackStack(route: T, inclusive: Boolean): Boolean // 2.8+

Pops the back stack. Returns true if something was popped, false if the back stack was already at the start destination (nothing to pop).

Where it breaks

  • "Popped past start destination" trap — calling popBackStack(startDestId, inclusive = true) pops the start destination itself. The NavController now has an empty back stack and no current destination. Any subsequent navigate() call crashes because there's no graph context. You must call navigate() to re-establish a destination immediately.
  • Return value ignored — most code ignores the Boolean return. If popBackStack() returns false, the user is stuck. Always check the return value and call finish() or navigate elsewhere as a fallback.
// Dangerous: pops start destination, leaves NavController in broken state
navController.popBackStack(R.id.home, inclusive = true)
navController.navigate(R.id.detail) // may crash — no graph context

// Safe: check return value
if (!navController.popBackStack()) {
    // Nothing to pop — we're at the root. Finish the Activity.
    requireActivity().finish()
}

navigateUp()

fun navigateUp(): Boolean

Navigates "up" in the app hierarchy. Differs from popBackStack(): navigateUp() respects the logical parent defined in the nav graph (or manifest), while popBackStack() follows the temporal back stack. For deep links, navigateUp() recreates the parent stack if the user entered via a deep link and there's no back stack to pop.

Where it breaks

  • Toolbar integration — NavigationUI.setupActionBarWithNavController() wires the Up button to navigateUp(). If you also have a DrawerLayout, you must use NavigationUI.navigateUp(navController, appBarConfiguration) instead, which toggles the drawer on the start destination rather than calling navigateUp().
  • Deep link stack synthesis — when the user arrives via deep link, navigateUp() synthesizes the parent stack using the nav graph's hierarchy. If your graph doesn't define a proper parent chain, the synthesized stack is wrong.

getCurrentDestination()

val currentDestination: NavDestination?

Returns the current destination, or null if the NavController hasn't been initialized or the back stack is empty.

Where it breaks

  • Null during transitions — between the moment navigate() is called and the new destination's Fragment/Composable is created, currentDestination may briefly reference the old destination or be in a transitional state. Observing it synchronously after navigate() is unreliable.
  • Race with OnDestinationChangedListener — if you read currentDestination inside a listener callback, it's already updated. But if you read it from a coroutine launched in the same frame, it might not be updated yet.

currentBackStackEntryAsState() (Compose)

@Composable fun NavController.currentBackStackEntryAsState(): State<NavBackStackEntry?>

Reactive observation of the current back stack entry in Compose. Recomposes whenever the destination changes. The canonical way to derive UI state (like bottom bar selection) from navigation state.

Where it breaks

  • Recomposition during transitions — during animated transitions, both the old and new entry exist simultaneously. currentBackStackEntryAsState() returns the new entry immediately, even before the animation completes. If your bottom bar highlights based on this, it updates before the screen visually transitions.
// Bottom bar selection using currentBackStackEntryAsState
val navBackStackEntry by navController.currentBackStackEntryAsState()
val currentDestination = navBackStackEntry?.destination

NavigationBar {
    topLevelRoutes.forEach { route ->
        NavigationBarItem(
            selected = currentDestination?.hierarchy?.any {
                it.hasRoute(route::class)
            } == true,
            onClick = {
                navController.navigate(route) {
                    popUpTo(navController.graph.findStartDestination().id) {
                        saveState = true
                    }
                    launchSingleTop = true
                    restoreState = true
                }
            },
            icon = { Icon(route.icon, contentDescription = null) },
            label = { Text(route.label) }
        )
    }
}

previousBackStackEntry — Result passing

val previousBackStackEntry: NavBackStackEntry?

Accesses the previous destination's NavBackStackEntry. Primary use case: passing results back via SavedStateHandle (the Navigation equivalent of setFragmentResult()).

// Screen B sets a result for Screen A
navController.previousBackStackEntry
    ?.savedStateHandle
    ?.set("selected_item", selectedItem)
navController.popBackStack()

// Screen A observes the result
val result = navController.currentBackStackEntry
    ?.savedStateHandle
    ?.getStateFlow("selected_item", default = "")
    ?.collectAsState()

Where it breaks

previousBackStackEntry is null if the current destination is the start destination (nothing behind it). Also null during process death restoration until the back stack is fully rebuilt. Always null-check.

getBackStackEntry(route) — Graph-scoped ViewModel

fun getBackStackEntry(route: T): NavBackStackEntry // 2.8+
fun getBackStackEntry(@IdRes destinationId: Int): NavBackStackEntry

Retrieves a specific NavBackStackEntry from the back stack. The primary use case is obtaining a ViewModel scoped to a nested navigation graph — the entry for the graph itself serves as the ViewModelStoreOwner.

// ViewModel scoped to a nested graph (shared across screens in that graph)
@Serializable object CheckoutGraph  // nested graph route

// Inside any destination within the checkout graph:
val checkoutViewModel: CheckoutViewModel = viewModel(
    viewModelStoreOwner = navController.getBackStackEntry(CheckoutGraph)
)

// This ViewModel lives as long as the CheckoutGraph is on the back stack.
// When the user pops back past the graph, the ViewModel is cleared.

Where it breaks

Throws IllegalArgumentException if the route/ID is not on the back stack. This happens when you try to access a graph-scoped ViewModel from a destination that isn't inside that graph, or after the graph has been popped.

handleDeepLink(intent: Intent)

fun handleDeepLink(intent: Intent): Boolean

Processes a deep link Intent against the nav graph's declared deep links. Returns true if a matching deep link was found and navigated to.

Where it breaks

  • Clears the existing back stack — by default, handleDeepLink() pops the entire back stack and rebuilds it from the deep link's parent chain. If the user was mid-flow (e.g., filling a form), their state is destroyed with no warning.
  • No option to preserve state — there's no flag to say "navigate to this deep link but keep the existing back stack." You must manually parse the Intent and call navigate() yourself to preserve state.
  • Called automatically — NavHostFragment calls handleDeepLink() in onCreate() if the Activity's Intent contains a matching URI. This means deep links fire before your own onCreate() logic runs.
🐛 Bad Design — handleDeepLink() destroys user state

handleDeepLink() was designed for cold-start deep links where there's no existing state to preserve. But it's also called on warm starts (via onNewIntent()), where the user may have unsaved work. The API provides no way to merge a deep link into an existing back stack — it's all-or-nothing. Production apps must intercept the Intent before it reaches handleDeepLink() and decide whether to navigate manually or let the default behavior run.

addOnDestinationChangedListener()

fun addOnDestinationChangedListener(listener: OnDestinationChangedListener)

Registers a callback invoked every time the current destination changes. Used for updating UI elements (toolbar title, bottom bar visibility) in response to navigation.

Where it breaks

  • Not lifecycle-aware — unlike Compose's currentBackStackEntryAsState(), this listener is not automatically removed when the lifecycle owner is destroyed. You must call removeOnDestinationChangedListener() in onDestroy() or use a lifecycle observer. Forgetting this leaks the Activity/Fragment.
  • Fires immediately on registration — when you add a listener, it's called immediately with the current destination. This is by design (to initialize UI state), but surprises developers who expect it to fire only on future changes.

saveState() / restoreState()

open fun saveState(): Bundle?
open fun restoreState(navState: Bundle?)

Internal mechanism for persisting the NavController's state across process death. NavHostFragment calls these automatically in onSaveInstanceState() and onCreate(). You almost never call these directly — they exist for custom NavHost implementations.

What's saved

The entire back stack (destination IDs, arguments, SavedStateHandle contents), the current destination, and each Navigator's internal state. ViewModel state is NOT included — that's handled by ViewModelStore separately.

3. NavHost & NavGraph — Setup Methods

NavHostFragment.create(graphResId)

companion object fun create(@NavigationRes graphResId: Int, startDestinationArgs: Bundle? = null): NavHostFragment

Programmatic creation of a NavHostFragment with a navigation graph. Alternative to declaring <fragment> in XML with app:navGraph.

When to use

Dynamic graph selection at runtime (e.g., different flows for different user roles). Also required when the graph depends on data not available at XML inflation time.

// Programmatic NavHostFragment creation
val navHost = NavHostFragment.create(
    graphResId = if (isOnboarded) R.navigation.main_graph else R.navigation.onboarding_graph,
    startDestinationArgs = bundleOf("userId" to userId)
)
supportFragmentManager.beginTransaction()
    .replace(R.id.nav_host_container, navHost)
    .setPrimaryNavigationFragment(navHost) // required for back button handling
    .commit()
⚠ Gotcha — setPrimaryNavigationFragment

If you add a NavHostFragment programmatically without calling setPrimaryNavigationFragment(), the system back button won't be intercepted by the NavController. The Fragment's OnBackPressedCallback only fires if the Fragment is the primary navigation fragment. This is the #1 cause of "back button doesn't work" bugs with programmatic NavHost setup.

NavHost {} (Compose)

@Composable fun NavHost( navController: NavHostController, startDestination: T, // @Serializable route object (2.8+) builder: NavGraphBuilder.() -> Unit )

The Compose equivalent of NavHostFragment. Creates a navigation graph using the Kotlin DSL and hosts the current destination's composable content.

Where it breaks

  • NavController must be remembered — use rememberNavController(). Creating a new NavHostController() on every recomposition destroys the back stack.
  • startDestination must be stable — if you compute the start destination from a mutable state, changing it after initial composition has no effect. The graph is built once.
// Compose NavHost with type-safe routes (2.8+)
@Serializable object Home
@Serializable data class Detail(val itemId: String)

val navController = rememberNavController()

NavHost(navController = navController, startDestination = Home) {
    composable<Home> {
        HomeScreen(onItemClick = { id ->
            navController.navigate(Detail(itemId = id))
        })
    }
    composable<Detail> { backStackEntry ->
        val detail: Detail = backStackEntry.toRoute()
        DetailScreen(itemId = detail.itemId)
    }
}

navigation() — Nested Graphs

fun NavGraphBuilder.navigation<T>(startDestination: KClass<*>, builder: NavGraphBuilder.() -> Unit)

Defines a nested navigation graph. Nested graphs create a scoping boundary for ViewModels and encapsulate a sub-flow (e.g., checkout, onboarding).

Where it breaks

  • startDestination is required — a nested graph must declare its own start destination. Navigating to the graph navigates to its start destination. Forgetting this causes a runtime crash.
  • ViewModel scoping boundary — ViewModels scoped to a nested graph's NavBackStackEntry are shared across all destinations in that graph and cleared when the graph is popped. This is the primary mechanism for sharing state within a flow without leaking it globally.
// Nested graph for checkout flow with shared ViewModel
@Serializable object CheckoutGraph
@Serializable object Cart
@Serializable object Shipping
@Serializable object Payment

NavHost(navController, startDestination = Home) {
    composable<Home> { /* ... */ }
    navigation<CheckoutGraph>(startDestination = Cart::class) {
        composable<Cart> {
            val vm: CheckoutViewModel = viewModel(
                viewModelStoreOwner = navController.getBackStackEntry(CheckoutGraph)
            )
            CartScreen(vm)
        }
        composable<Shipping> {
            val vm: CheckoutViewModel = viewModel(
                viewModelStoreOwner = navController.getBackStackEntry(CheckoutGraph)
            )
            ShippingScreen(vm)
        }
        composable<Payment> {
            val vm: CheckoutViewModel = viewModel(
                viewModelStoreOwner = navController.getBackStackEntry(CheckoutGraph)
            )
            PaymentScreen(vm)
        }
    }
}

composable() / dialog() / navigation()

fun NavGraphBuilder.composable<T>(/* ... */) { content: @Composable (NavBackStackEntry) -> Unit }
fun NavGraphBuilder.dialog<T>(/* ... */) { content: @Composable (NavBackStackEntry) -> Unit }

Destination types in the Compose DSL. composable() renders full-screen content via ComposeNavigator. dialog() renders content in a dialog window via DialogNavigator — the previous destination remains visible behind it.

Where it breaks

dialog() destinations are on the back stack but don't trigger ON_STOP on the destination behind them (since it's still visible). If you collect Flows with repeatOnLifecycle(STARTED) on the screen behind a dialog, both the screen and the dialog receive emissions simultaneously.

NavDeepLink — URI, MIME, Action

navDeepLink { uriPattern = "app://example.com/profile/{userId}" }
navDeepLink { mimeType = "image/*" }
navDeepLink { action = "com.example.SHOW_PROFILE" }

Declares deep links on a destination. Supports URI patterns with placeholders ({arg}), wildcards (.*), optional query parameters, MIME type matching, and Intent action matching.

Where it breaks

  • Placeholder names must match argument names — {userId} in the URI must correspond to an argument named userId on the destination. A mismatch silently drops the value.
  • Query parameters are optional by default — app://example.com/profile?tab={tab} matches even without ?tab=.... The argument gets its default value. This is usually desirable but can mask bugs.
  • Multiple deep links on one destination — if two deep link patterns overlap, the first match wins. Order depends on declaration order in the graph, which is fragile.

4. NavOptions & NavArguments — Configuration

popUpTo / popUpToInclusive

navOptions { popUpTo(route) { inclusive = true } }

Pops all destinations up to (and optionally including) the specified destination before navigating. The most powerful and most misused NavOption.

Where it breaks

  • "Accidentally popped start destination" — popUpTo(startDest) { inclusive = true } removes the start destination. If the new destination is then popped, the NavController has an empty back stack and crashes.
  • The bottom-nav pattern — the canonical pattern uses popUpTo(graph.startDestination) { saveState = true } + launchSingleTop = true + restoreState = true. Getting any of these three wrong breaks bottom navigation. Missing saveState loses scroll position. Missing restoreState creates fresh instances. Missing launchSingleTop creates duplicate entries.
// The canonical bottom-nav pattern (2.4+)
fun NavController.navigateToTopLevel(route: Any) {
    navigate(route) {
        popUpTo(graph.findStartDestination().id) {
            saveState = true
        }
        launchSingleTop = true
        restoreState = true
    }
}
⚠ Gotcha — popUpTo with inclusive on start destination

If you write popUpTo(Home) { inclusive = true } and then navigate to Settings, the Home destination is removed from the back stack. Pressing Back from Settings exits the app instead of returning to Home. This is correct behavior — you asked to remove Home — but it surprises developers who wanted to "reset" the stack. To reset and keep the start destination, use inclusive = false.

launchSingleTop

navOptions { launchSingleTop = true }

If the destination is already at the top of the back stack, reuse it instead of creating a new instance. Prevents duplicate entries when the user taps a navigation item multiple times.

🐛 Bad Design — launchSingleTop doesn't update arguments

When launchSingleTop = true reuses an existing destination, the arguments are not updated. If you navigate to Detail(itemId = "A") and then Detail(itemId = "B") with launchSingleTop, the destination still shows item A. The new arguments are silently discarded. Workaround: write the new arguments to the destination's SavedStateHandle and observe them reactively, or avoid launchSingleTop when arguments change.

// launchSingleTop silently discards new arguments
navController.navigate(Detail(itemId = "A"))
navController.navigate(Detail(itemId = "B")) {
    launchSingleTop = true
}
// Detail screen still shows item "A" — "B" was discarded

// Workaround: observe arguments from SavedStateHandle
@Composable
fun DetailScreen(viewModel: DetailViewModel = viewModel()) {
    val itemId by viewModel.savedStateHandle
        .getStateFlow("itemId", "")
        .collectAsState()
    // Always reflects the latest value
}

saveState / restoreState — Multiple Back Stacks

popUpTo(dest) { saveState = true } navOptions { restoreState = true }

The mechanism behind multiple back stacks (Navigation 2.4+). When saveState = true, destinations being popped have their state (including ViewModel, SavedStateHandle, and view state) serialized and stored. When restoreState = true, navigating to a destination that was previously saved restores all that state.

🎯 Staff Insight — Multiple back stacks are a state serialization problem

Multiple back stacks aren't a navigation concept — they're a state persistence problem. Each "saved" back stack is essentially a process-death-style state bundle stored in memory. The NavController serializes NavBackStackEntry state (arguments, SavedStateHandle), the Navigator serializes its own state (Fragment view hierarchy for FragmentNavigator, nothing for ComposeNavigator), and the ViewModelStore is detached but kept alive. Restoring is the reverse. This is why non-serializable arguments (custom objects without Parcelable) break saveState — the system literally can't serialize them. In an L6 interview, explaining multiple back stacks as "in-memory process death simulation" demonstrates deep understanding.

Transitions — Fragment vs Compose

Fragment transitions

// In nav graph XML <action app:enterAnim="@anim/slide_in" app:exitAnim="@anim/slide_out" />

Compose transitions (2.7+)

composable<Detail>( enterTransition = { slideInHorizontally { it } }, exitTransition = { slideOutHorizontally { -it } } ) { /* ... */ }

Before Navigation 2.7, Compose required AnimatedNavHost from the Accompanist library for animated transitions. In 2.7, NavHost uses AnimatedContent internally and supports enterTransition/exitTransition/popEnterTransition/popExitTransition directly on each composable() destination.

Safe Args → @Serializable Routes

🔄 API Evolution — Type-safe navigation arguments
Navigation 1.0 (2018): Safe Args Gradle plugin. Generates Directions and Args classes from XML nav graph. Type-safe at compile time. Limitations: only supports primitive types, Parcelable, Serializable, and Enums. No sealed classes, no nullable complex types, no default values for complex types.
Navigation 2.4 (2022): String-based routes for Compose: composable("profile/{userId}"). No compile-time checking. Typos in route strings fail at runtime. Arguments extracted manually from NavBackStackEntry.arguments.
Navigation 2.8 (2024): @Serializable routes. Kotlin data classes/objects as route definitions. Kotlinx.serialization handles encoding/decoding. Compile-time type safety. Supports all Kotlin types that kotlinx.serialization supports (including enums, sealed classes, nullable types, default values).
// Safe Args (XML graph) — still works, but legacy
// nav_graph.xml defines: <argument name="userId" app:argType="string" />
// Generated: HomeFragmentDirections.actionHomeToProfile(userId = "123")
// Generated: ProfileFragmentArgs.fromBundle(arguments).userId

// String routes (Compose, pre-2.8) — no compile-time safety
composable("profile/{userId}") { entry ->
    val userId = entry.arguments?.getString("userId") ?: ""
    ProfileScreen(userId)
}

// @Serializable routes (2.8+) — compile-time safe
@Serializable data class Profile(
    val userId: String,
    val tab: ProfileTab = ProfileTab.POSTS  // default value works
)

@Serializable enum class ProfileTab { POSTS, FOLLOWERS, SETTINGS }

composable<Profile> { entry ->
    val profile: Profile = entry.toRoute()
    ProfileScreen(profile.userId, profile.tab)
}

// Navigate with full type safety:
navController.navigate(Profile(userId = "123", tab = ProfileTab.FOLLOWERS))

Edge cases with @Serializable routes

  • Enums — fully supported. Serialized as their name string in the route.
  • Nullable types — supported. val query: String? = null becomes an optional query parameter.
  • Default values — supported. Parameters with defaults become optional in the route URI.
  • Collections — List<String> is supported but serialized as repeated query parameters (?item=a&item=b). Large lists bloat the route string.
  • Custom serializers — you can provide custom NavType implementations for types that need special handling, but this breaks the simplicity of the @Serializable approach.

5. Design Patterns in Navigation

Command Pattern

NavController.navigate() is a Command: it encapsulates a navigation action with a destination, arguments, and NavOptions. The caller doesn't know how navigation is performed — FragmentNavigator, ComposeNavigator, or ActivityNavigator could handle it. NavOptions is the command's configuration (pop behavior, animations, single-top). The back stack is the command history, and popBackStack() is undo.

Mediator Pattern

NavController mediates between the UI layer (NavHost, buttons, bottom bars) and the navigation graph. No UI component talks directly to another — they all go through the NavController. This decouples "what to navigate to" from "how to navigate" and "where to render." The cost: the NavController becomes a God Object for navigation, accumulating methods for every use case.

Strategy Pattern

Navigator<T> is a Strategy. Each Navigator implementation handles one type of destination:

NavigatorHandlesMechanism
FragmentNavigator<fragment> destinationsFragmentTransaction.replace()
ComposeNavigatorcomposable {} destinationsCompose state + AnimatedContent
DialogNavigator<dialog> / dialog {}DialogFragment or Compose Dialog
ActivityNavigator<activity> destinationsstartActivity()

You can create custom Navigators (e.g., for bottom sheets) by extending Navigator and registering with NavigatorProvider. This is how libraries like Accompanist added bottom sheet navigation before official support.

Observer Pattern

OnDestinationChangedListener and currentBackStackEntryAsState() are Observer implementations. The NavController is the Subject, and UI components (toolbar, bottom bar, analytics) are Observers. The Compose variant uses Snapshot state internally, making it reactive without explicit listener management.

State Machine (Informal)

🎯 Staff Insight — The nav graph is a state machine, but not a strict one

A navigation graph defines states (destinations) and transitions (actions). This looks like a Finite State Machine — but it's not enforced as one. You can call navigate(anyDestination) from any current destination, even if no action connects them. Actions are suggestions, not constraints. A strict FSM would reject transitions not defined in the graph. Navigation's permissiveness is a design choice: it prioritizes flexibility over safety. In an L6 interview, you can argue that navigation should optionally support strict FSM mode — rejecting undefined transitions at runtime (or compile time with @Serializable routes) — to catch bugs in complex flows. This is a genuine architectural improvement that hasn't been implemented.

Deep Dive: Double-Navigation Crash

The root cause is a TOCTOU (Time-of-Check-Time-of-Use) race. The click handler checks the current destination implicitly (by referencing an action defined on that destination), but by the time the second click fires, the destination has changed. The NavController doesn't queue or debounce navigation calls — each one executes immediately and synchronously.

// Three defensive patterns:

// 1. Check currentDestination before navigating (Fragment)
fun NavController.safeNavigate(
    @IdRes currentDestId: Int,
    @IdRes actionId: Int
) {
    if (currentDestination?.id == currentDestId) {
        navigate(actionId)
    }
}

// 2. Throttle clicks at the UI layer
fun View.setThrottledClickListener(
    intervalMs: Long = 500,
    onClick: (View) -> Unit
) {
    var lastClickTime = 0L
    setOnClickListener { view ->
        val now = SystemClock.elapsedRealtime()
        if (now - lastClickTime >= intervalMs) {
            lastClickTime = now
            onClick(view)
        }
    }
}

// 3. Compose: derive click-ability from navigation state
@Composable
fun HomeScreen(navController: NavController) {
    val currentEntry by navController.currentBackStackEntryAsState()
    val isAtHome = currentEntry?.destination?.hasRoute<Home>() == true

    Button(onClick = {
        if (isAtHome) navController.navigate(Detail(id = "1"))
    }) { Text("Go to Detail") }
}

Deep Dive: FragmentNavigator View Leak

🐛 Bad Design — FragmentNavigator uses replace(), not remove()

FragmentNavigator uses FragmentTransaction.replace() to navigate between destinations. This destroys the Fragment's View (calls onDestroyView()) but keeps the Fragment instance alive on the back stack. If the Fragment holds references to Views (binding, RecyclerView adapter, custom views), those references become stale pointers to destroyed Views. They won't be garbage collected because the Fragment instance — still on the back stack — holds strong references. The fix: null out all View references in onDestroyView(). ViewBinding's _binding = null pattern exists specifically because of this design.

class ListFragment : Fragment(R.layout.fragment_list) {
    private var _binding: FragmentListBinding? = null
    private val binding get() = _binding!!

    override fun onViewCreated(view: View, savedInstanceState: Bundle?) {
        _binding = FragmentListBinding.bind(view)
        binding.recyclerView.adapter = MyAdapter()
    }

    override fun onDestroyView() {
        super.onDestroyView()
        _binding = null // REQUIRED: Fragment stays alive, View is destroyed
    }
}

Deep Dive: NavBackStackEntry Lifecycle Overlap

During animated transitions, the outgoing destination's NavBackStackEntry is moved to STARTED (from RESUMED) and the incoming entry is moved to STARTED (then RESUMED after the animation). For a brief period, both entries are in STARTED state. If both screens collect a shared Flow using repeatOnLifecycle(STARTED), both collectors are active simultaneously — causing duplicate side effects.

// Problem: both old and new screen collect during transition
viewLifecycleOwner.lifecycleScope.launch {
    viewLifecycleOwner.repeatOnLifecycle(Lifecycle.State.STARTED) {
        viewModel.events.collect { event ->
            // This fires on BOTH the outgoing and incoming screen
            // during the transition animation
            showSnackbar(event.message)
        }
    }
}

// Fix: use RESUMED instead of STARTED for one-shot events
viewLifecycleOwner.lifecycleScope.launch {
    viewLifecycleOwner.repeatOnLifecycle(Lifecycle.State.RESUMED) {
        viewModel.events.collect { event ->
            showSnackbar(event.message) // only the active screen collects
        }
    }
}
🔄 API Evolution — AnimatedNavHost → NavHost merge
Navigation 2.4–2.6: Compose NavHost had no animation support. Transitions were instant (hard cut). The Accompanist library provided AnimatedNavHost as a workaround, using AnimatedContent under the hood.
Navigation 2.7 (2023): AnimatedNavHost was merged into the official NavHost. The Accompanist version was deprecated. All Compose navigation now uses AnimatedContent internally, with configurable enterTransition/exitTransition per destination. Default transition is a fade.
Impact: Apps that depended on Accompanist's AnimatedNavHost had to migrate import paths. The API surface was nearly identical, but some edge cases around shared element transitions changed. Apps using plain NavHost got animations "for free" — which broke screenshot tests that expected instant transitions.
🔄 API Evolution — Multiple back stacks (the long road)
2018–2021: Navigation had no multiple back stack support. Google's own Android Architecture Components samples included a NavigationExtensions.kt file with a 200-line workaround that swapped NavHostFragment instances. This became the most-copied file in Android development — and it was full of bugs (state loss on process death, animation glitches, ViewModel leaks).
Navigation 2.4 (Jan 2022): Official multiple back stacks via saveState/restoreState in NavOptions. The NavigationExtensions.kt workaround was deprecated. Under the hood, each Navigator gained onSaveState()/onRestoreState() methods. FragmentNavigator saves Fragment state bundles. ComposeNavigator saves SavedStateHandle contents.
Impact: The migration from NavigationExtensions.kt to official multiple back stacks was non-trivial. The workaround used separate NavHostFragment instances per tab (each with its own NavController). The official solution uses a single NavController with saved/restored state. Apps had to restructure their navigation graphs to use a single graph with nested graphs per tab.
