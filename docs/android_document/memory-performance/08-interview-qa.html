08 — Interview Q&A — Memory Management & Performance

58 staff-level questions covering heap/stack/native memory, GC internals, memory leaks, ANR, OOM/LMK, app startup, and Baseline Profiles

Memory Model (Q1–Q8)

Q1. Explain the difference between Java heap, native heap, and stack memory in an Android process. Where does each live and what are the limits?

Every Android process has three primary memory regions. The Java/Dalvik heap is managed by ART, sized via dalvik.vm.heapsize (typically 256–512 MB), and holds all objects created with new. The native heap (malloc-backed) is limited only by the process's virtual address space and the kernel's RLIMIT_AS; it holds Bitmap pixel buffers (since Android 8.0), RenderNode display lists, SQLite page caches, and JNI-allocated memory. The stack is per-thread, typically 1 MB on Android (set by pthread_attr_setstacksize), and holds local primitives, method frames, and return addresses. A critical L6 nuance: dalvik.vm.heapgrowthlimit is the actual per-app cap (typically 256 MB); heapsize is the largeHeap cap. Native heap OOM manifests as std::bad_alloc or SIGABRT from jemalloc, not OutOfMemoryError, making it harder to diagnose in crash reports.

Q2. What are PSS, RSS, USS, and VSS? Which metric should you use when profiling an app's real memory footprint?

VSS (Virtual Set Size) is total virtual address space mapped — includes mmap'd files never touched, so it's misleadingly large. RSS (Resident Set Size) is physical pages currently in RAM, but double-counts shared pages (e.g., Zygote's preloaded classes shared across all apps). PSS (Proportional Set Size) divides each shared page's cost by the number of processes sharing it — if 3 apps share a 12 KB page, each is charged 4 KB. USS (Unique Set Size) counts only pages private to the process. For app profiling, PSS is the gold standard because it fairly attributes shared memory. dumpsys meminfo <pid> reports PSS broken down by category (Java Heap, Native Heap, Code, Stack, Graphics, etc.). USS is useful for detecting what memory is freed when your process dies. RSS is what the LMK actually considers when deciding which process to kill.

Q3. How does dumpsys meminfo break down an app's memory? What are the key categories to watch?

adb shell dumpsys meminfo <package> reports PSS in columns: Java Heap (ART-managed objects), Native Heap (malloc/jemalloc), Code (.dex, .oat, .so mmap'd), Stack (thread stacks), Graphics (GPU buffers, EGL mtrack), Private Other (ashmem, SharedMemory), System (overhead). Key signals: Java Heap growing after GC indicates a leak. Native Heap growing without corresponding Java allocations suggests Bitmap accumulation (pre-O), SQLite cache bloat, or JNI leaks. Graphics memory spiking indicates excessive Surface or TextureView usage. The "Objects" section shows View count, Activity count (should be 0 or 1 after navigation), and AppContext count — multiple Activities in a single-Activity app is a strong leak indicator. The "Total PSS by OOM Adjustment" section shows how the system categorizes your process priority.

Q4. Why did Android 8.0 move Bitmap pixel data from the Java heap to the native heap? What are the implications?

Before Android 8.0 (Oreo), Bitmap pixel buffers lived on the Java heap, counted against heapgrowthlimit. A 1080p ARGB_8888 image is ~8 MB — a few Bitmaps could exhaust the 256 MB Java heap. In Android 8.0, ART's Bitmap.cpp allocates pixels via calloc() on the native heap, with a reference tracked by a NativeAllocationRegistry so the GC knows the true memory cost and can trigger collection when native allocations grow. Implications: (1) Java heap pressure drops dramatically for image-heavy apps. (2) Runtime.maxMemory() no longer reflects actual memory pressure — you must check native heap via Debug.getNativeHeapAllocatedSize(). (3) OOM now manifests as native allocation failure or LMK kill rather than Java OutOfMemoryError. (4) NativeAllocationRegistry adds the native size to ART's internal accounting so GC_FOR_ALLOC triggers appropriately — without it, the GC wouldn't know about native memory pressure.

Q5. What is the Zygote heap and why does it matter for memory sharing?

When Zygote preloads ~10,000 framework classes, their class objects, static fields, and interned strings are allocated on the Zygote heap — a special region in ART's space::ZygoteSpace. After preloading, Zygote calls Runtime::PreZygoteFork() which marks these pages as the Zygote space and makes them read-only. When fork() creates a new app process, these pages are shared via copy-on-write. Because they're read-only, COW never triggers, so all apps share the same physical pages. This saves 50–80 MB per process. The GC treats ZygoteSpace specially: it never collects objects in it (they're immortal), and it never moves them (no compaction). If you use Class.forName() during Zygote preload to load your own classes (via a custom ROM), those classes also land in ZygoteSpace and are shared — a powerful optimization for OEM apps.

Q6. When should you use android:largeHeap="true"? What are the trade-offs?

largeHeap raises the Java heap limit from dalvik.vm.heapgrowthlimit (e.g., 256 MB) to dalvik.vm.heapsize (e.g., 512 MB). It does not increase native heap limits. Legitimate uses: photo/video editors, CAD apps, apps that must hold large datasets in memory. Trade-offs: (1) GC pauses scale with heap size — a 512 MB heap means the CC collector must scan more objects during the young-gen phase. (2) The app is more likely to be killed by LMK because it consumes more RSS, raising its oom_score_adj relative to other apps. (3) It masks real memory issues — developers use it as a band-aid instead of fixing leaks or reducing allocations. (4) On low-RAM devices (ActivityManager.isLowRamDevice()), largeHeap values are often the same as normal heap. Google's guidance: use it only after exhausting all optimization options, and pair it with aggressive onTrimMemory() handling.

Q7. How does ART decide when to grow the heap vs. triggering GC?

ART maintains a target footprint — the current allowed heap size. When an allocation would exceed this footprint, ART first attempts a concurrent GC. If the GC frees enough space, the allocation succeeds. If not, ART grows the footprint (up to heapgrowthlimit) using a growth multiplier configured by dalvik.vm.heaptargetutilization (default 0.75, meaning ART targets 75% heap utilization). The formula: new_footprint = live_bytes / target_utilization. So if 150 MB is live and utilization target is 0.75, the new footprint is 200 MB. If the footprint can't grow (at the limit), ART tries a more aggressive GC_FOR_ALLOC (blocking, compacting). If that fails, it throws OutOfMemoryError. The heapMinFree and heapMaxFree properties clamp the free space to prevent excessive growth or excessive GC. This adaptive sizing is why you see the heap "breathe" — growing under load, shrinking after GC.

Q8. What is NativeAllocationRegistry and how does it bridge native memory with GC awareness?

NativeAllocationRegistry (Android 8.0+) registers a native allocation size and a free function with a Java object. When the Java object becomes unreachable, the GC invokes the registered free function (via a reference queue and a Cleaner) to release the native memory. Critically, it also calls Runtime::RegisterNativeAllocation(size) which adds the size to ART's internal accounting. This means ART's GC trigger heuristic includes native memory — if native allocations grow rapidly, GC fires sooner even if Java heap is under-utilized. Without this, Bitmaps on the native heap would be invisible to the GC, and apps could allocate gigabytes of native memory before the GC noticed. The registry uses sun.misc.Cleaner (a PhantomReference subclass) rather than finalizers, avoiding the finalizer queue bottleneck and ensuring deterministic cleanup ordering. This is how Bitmap, Typeface, and HardwareBuffer manage their native backing memory.

GC Internals (Q9–Q18)

Q9. Describe ART's Concurrent Copying (CC) collector. What are its phases and why is it the default since Android 10?

The CC collector has three phases: (1) Initial Mark — a brief pause (~100 μs) where the GC marks roots (thread stacks, globals, JNI refs) and flips the "to-space" and "from-space" pointers. (2) Concurrent Copying — mutator threads continue running while GC threads copy live objects from from-space to to-space. Read barriers on every object reference ensure mutators always see the forwarded (to-space) copy. (3) Final Remark — a brief pause to drain the mark stack and handle reference processing (weak/soft/phantom refs). CC replaced the older CMS (Concurrent Mark-Sweep) as the default because it compacts the heap concurrently — eliminating fragmentation without a stop-the-world pause. CMS couldn't compact without pausing all threads. CC's pause times are typically under 200 μs regardless of heap size, compared to CMS's 5–50 ms pauses during compaction. The trade-off is ~5–10% throughput overhead from read barriers.

Q10. What is a read barrier? How does it differ from a write barrier, and why does ART's CC collector use read barriers?

A read barrier is a check inserted by the compiler before every object reference load: "is this reference pointing to from-space? If so, follow the forwarding pointer to to-space." ART's CC collector uses the Brooks forwarding pointer — every object has a word-sized field pointing to itself (or to its to-space copy if moved). The read barrier loads this pointer, adding ~1 instruction per reference load. A write barrier is a check on every reference store, used by generational and CMS collectors to track cross-generation or mutator-modified references. CMS uses write barriers to maintain the "dirty card table" — marking cards as modified so the remark phase can rescan them. CC uses read barriers instead because it needs to ensure mutators never see stale from-space pointers during concurrent copying. The throughput cost of read barriers (~5–10%) is offset by eliminating stop-the-world compaction pauses. ART's generational CC mode adds write barriers on top of read barriers to track young→old references.

Q11. How does generational GC work in ART? What triggers a young-gen vs. full collection?

ART's generational CC (default since Android 10) divides the heap into young generation (recently allocated objects) and old generation (objects that survived previous collections). Young-gen collections are triggered when the young-gen allocation threshold is reached (typically a few MB). They only scan and copy young-gen objects, using a remembered set (maintained via write barriers) to find old→young references without scanning the entire old generation. Full (major) collections are triggered when: (1) the overall heap footprint approaches the target, (2) an explicit System.gc() is called, or (3) native allocation pressure triggers it via NativeAllocationRegistry. Young-gen collections take ~1–3 ms (pause < 200 μs) because they scan a small region. Full collections take ~10–50 ms but are infrequent if most objects are short-lived (the generational hypothesis). The write barrier overhead for tracking old→young references is ~2–3% throughput, but it reduces full GC frequency by 5–10x.

Q12. What are GC roots in ART? List all root types and explain why missing a root causes a use-after-free.

GC roots are the starting points for reachability analysis. ART's roots include: (1) Thread stacks — local variables and parameters in every frame of every thread. (2) JNI global references — created via NewGlobalRef() in native code. (3) JNI local references — per-native-call frame references. (4) Static fields of loaded classes. (5) Monitor locks — objects currently locked by synchronized. (6) Interned strings — in the intern table. (7) Class objects of loaded classes (held by ClassLinker). (8) Finalizer references — objects with finalizers pending execution. If the GC misses a root (e.g., a JNI global ref not properly registered), it may collect a live object. The next access through the dangling reference reads freed memory — a use-after-free that typically manifests as SIGSEGV in ART's ReadBarrier::Barrier() or corrupted object headers. This is why JNI code must meticulously manage NewGlobalRef/DeleteGlobalRef.

Q13. How does allocation rate affect GC frequency and app performance? How do you measure it?

Allocation rate is the bytes allocated per second on the Java heap. High allocation rates (>50 MB/s) cause frequent young-gen GC cycles, each consuming CPU and introducing sub-millisecond pauses that accumulate into jank. In ART, every allocation checks the thread-local allocation buffer (TLAB) — if the TLAB is exhausted, ART requests a new TLAB from the RegionSpace, which may trigger a GC if the young-gen threshold is reached. Measure allocation rate with: (1) Android Studio Memory Profiler — shows allocations/s in real-time. (2) Perfetto — mem.rss.anon and ART heap counters. (3) Debug.getGlobalAllocCount() / Debug.getGlobalAllocSize() for programmatic measurement. (4) adb shell dumpsys meminfo <pid> — "Alloc" column shows cumulative allocations. Common high-allocation culprits: string concatenation in loops, autoboxing (Int → Integer), iterator() on collections in hot paths, lambda captures creating anonymous class instances, and String.format().

Q14. What are the different GC triggers in ART? Explain each one.

ART has several GC trigger reasons visible in logcat's GC lines: (1) Alloc — allocation failed because the heap footprint was reached; this is the most common trigger. (2) Background — app moved to background; ART performs a compacting GC to reduce memory footprint (switches from CC to a more aggressive collector). (3) Explicit — System.gc() or Runtime.gc() called; ART may ignore this if -XX:+DisableExplicitGC is set. (4) NativeAlloc — native allocations registered via NativeAllocationRegistry exceeded the native allocation GC trigger threshold. (5) CollectorTransition — switching between foreground (CC) and background (CC + compaction) collectors. (6) HomogeneousSpaceCompact — compacting the main space to reduce fragmentation (older ART versions). The background GC is particularly important: when onTrimMemory(TRIM_MEMORY_UI_HIDDEN) fires, ART aggressively compacts and releases pages back to the kernel via madvise(MADV_DONTNEED).

Q15. How do you read ART's GC log lines? Parse a real example.

A typical GC log: I/art: CC GC freed 42301(2MB) AllocSpace objects, 12(800KB) LOS objects, 49% free, 14MB/28MB, paused 112us total 8.234ms. Breakdown: CC = Concurrent Copying collector. freed 42301(2MB) AllocSpace = 42,301 objects totaling 2 MB freed from the main allocation space (RegionSpace). 12(800KB) LOS = 12 objects totaling 800 KB freed from the Large Object Space. 49% free = heap utilization is 51%. 14MB/28MB = 14 MB live out of 28 MB footprint. paused 112us = total stop-the-world pause time (initial mark + final remark). total 8.234ms = wall-clock time for the entire GC cycle. Red flags: pause > 1 ms (unusual for CC, investigate), freed very little (possible leak), LOS objects growing (large allocations not being freed), total time > 50 ms (heap too large or too many live objects).

Q16. What is RegionSpace in ART? How does it relate to TLAB allocation?

RegionSpace is ART's primary allocation space for the CC collector, dividing the heap into fixed-size regions (typically 256 KB each). Each region is tagged as: free, allocated (young-gen), to-space, or from-space. Thread-Local Allocation Buffers (TLABs) are carved from free regions — each thread gets a region (or portion) as its TLAB, enabling lock-free bump-pointer allocation. When a thread's TLAB is full, it requests a new region from RegionSpace (requires a lock, but infrequent). During GC, live objects are copied from from-space regions to to-space regions; from-space regions are then freed wholesale — no per-object deallocation. This region-based approach enables O(1) bulk freeing and naturally compacts the heap. Large objects (>75% of region size) bypass RegionSpace and go to the Large Object Space (LOS) to avoid wasting region space on a single large allocation.

Q17. What is the Large Object Space (LOS)? Why does it exist separately?

The LOS is a separate space in ART for objects larger than kLargeObjectThreshold (typically 12 KB, or 3 pages). Large objects are allocated directly via mmap() or from a free-list allocator, bypassing RegionSpace's bump-pointer. Rationale: (1) A 1 MB object in a 256 KB region space would waste the remainder of the region. (2) Large objects are expensive to copy during CC — the GC may choose to pin them instead. (3) mmap()-backed allocations can be returned to the OS immediately via munmap() when freed, unlike RegionSpace where freed regions are recycled internally. LOS objects are tracked in a separate mark bitmap. In GC logs, "LOS objects" freed indicates large allocations being reclaimed. Common LOS residents: large arrays, Bitmap backing arrays (pre-O), large StringBuilder buffers. Excessive LOS allocations indicate the app is creating many large temporary objects — a performance concern because each allocation involves a syscall.

Q18. How does ART's compaction work? Compare the foreground and background compaction strategies.

ART uses different compaction strategies based on app state. Foreground (CC): concurrent copying inherently compacts — live objects are copied to contiguous to-space regions during every GC cycle. No stop-the-world compaction needed. Read barriers ensure mutators see forwarded pointers. Background: when the app moves to background, ART performs a more aggressive compaction. It may use semi-space compaction (copy all live objects to one end of the heap) or homogeneous space compaction (compact the main space in-place). After compaction, ART calls madvise(MADV_DONTNEED) on freed pages, telling the kernel to reclaim the physical memory without unmapping the virtual addresses. This is why an app's RSS drops after backgrounding. The key trade-off: foreground CC trades ~5% throughput (read barriers) for zero-pause compaction; background compaction trades a brief pause for maximum memory return to the system. On Android 12+, ART can also use MADV_FREE (lazy reclaim) for better performance when the system isn't under memory pressure.

Memory Leaks (Q19–Q28)

Q19. What is the classic Activity leak pattern? Trace the reference chain that prevents GC.

The classic pattern: an inner class (e.g., Handler, Runnable, anonymous OnClickListener) holds an implicit reference to the enclosing Activity. If this inner class is posted to a Handler with a delayed message or registered as a listener on a long-lived object (e.g., a singleton, a static field, or a system service callback), the reference chain is: MessageQueue → Message → Runnable → Activity (or Singleton → Listener → Activity). Even after onDestroy(), the Activity cannot be GC'd because it's reachable from a GC root (the main thread's Looper/MessageQueue is always a root). The Activity retains its entire view hierarchy (mDecor), all Views' backing Bitmaps, Drawables, and any objects referenced by member fields — easily 10–50 MB per leaked Activity. Fix: use WeakReference in Handlers, remove callbacks in onDestroy(), use lifecycleScope for coroutines, and never store Activity references in singletons.

Q20. Explain the Fragment ViewBinding leak. Why does it happen and how do you fix it?

In a Fragment, the view lifecycle is shorter than the Fragment lifecycle — onDestroyView() is called when the Fragment's view is detached (e.g., navigating away), but the Fragment instance survives on the back stack. If you store a ViewBinding reference as a member field (private var _binding: FragmentXBinding? = null) and forget to null it in onDestroyView(), the binding holds a reference to the root View, which holds the entire view hierarchy. The chain: Fragment → _binding → rootView → all child Views → Drawables/Bitmaps. Fix: always null out the binding in onDestroyView():

override fun onDestroyView() {
    super.onDestroyView()
    _binding = null
}

Alternatively, use a delegate like autoCleared that automatically nulls the property when the view is destroyed. In Compose-based Fragments, this isn't an issue because ComposeView disposes its composition in onDestroyView().

Q21. How can a ViewModel leak an Activity? Isn't ViewModel supposed to survive configuration changes?

ViewModel survives configuration changes by being stored in ViewModelStore, which is retained by NonConfigurationInstances (or the SavedStateRegistryOwner). The leak occurs when a ViewModel holds a direct reference to an Activity, View, or Context (not Application context). After rotation, the old Activity is destroyed but the ViewModel still references it — the new Activity creates a new view hierarchy, but the old one is retained by the ViewModel. Chain: ViewModelStore → ViewModel → Activity → DecorView → all Views. This is why ViewModel's constructor should never accept Activity or Context — use AndroidViewModel (which holds Application context) or inject dependencies via Hilt's @HiltViewModel. A subtler variant: ViewModel holds a LiveData observer lambda that captures the Activity — the observer keeps the Activity alive even after onDestroy(). Always observe LiveData with a lifecycle-aware observe(viewLifecycleOwner, ...) call.

Q22. How do coroutine scope leaks happen? Compare GlobalScope, lifecycleScope, and viewModelScope.

GlobalScope launches coroutines that live until the process dies — if the coroutine captures an Activity (directly or via a lambda closure), the Activity leaks for the coroutine's entire lifetime. lifecycleScope (from lifecycle-runtime-ktx) is tied to the Activity/Fragment lifecycle — it cancels all coroutines in onDestroy(), preventing leaks. viewModelScope cancels in ViewModel.onCleared(). The leak mechanism: a coroutine's continuation object holds references to all captured variables in the lambda. If a GlobalScope.launch { delay(60_000); updateUI() } captures this@Activity, the Activity is retained for 60 seconds after destruction. Even lifecycleScope can leak if you launch in CREATED state and the coroutine captures Views that should be freed in onDestroyView() — use viewLifecycleOwner.lifecycleScope in Fragments. The repeatOnLifecycle(STARTED) pattern is the safest for Flow collection because it cancels and restarts the block as the lifecycle moves in and out of the target state.

Q23. What memory leaks are specific to Jetpack Compose? How does Compose's lifecycle model help or hurt?

Compose's lifecycle model largely prevents classic View-system leaks because there's no manual binding to null out — the composition is disposed when the Composable leaves the tree. However, Compose-specific leaks include: (1) Captured references in remember — if remember { SomeObject(activity) } captures an Activity reference, it lives as long as the Composable is in the composition. (2) rememberCoroutineScope() with long-running work — the scope is cancelled when the Composable leaves composition, but if you launch work that captures expensive objects and the Composable is in a lazy list, it may live longer than expected. (3) State hoisting to wrong scope — hoisting a MutableState holding a Bitmap to a ViewModel means the Bitmap lives until ViewModel clears, not until the screen is gone. (4) CompositionLocal leaking Context — providing an Activity as LocalContext is fine (it's the default), but storing it in a singleton from a Composable is not. Compose's DisposableEffect with onDispose is the correct place to unregister listeners — equivalent to onDestroyView() cleanup.

Q24. Why does WebView leak and how do you mitigate it?

WebView is one of the most notorious leakers on Android. Internally, WebView creates a Chromium renderer process and holds references to its parent Activity's Context through multiple paths: the ViewRootImpl, AccessibilityManager, InputMethodManager, and the Chromium AwContents native object. Even calling webView.destroy() doesn't fully sever all references in some Android versions. Mitigations: (1) Separate process — run the Activity containing WebView in a separate process (android:process=":webview"); when the Activity finishes, the entire process dies, reclaiming all memory. (2) Programmatic creation — create WebView with Application context (WebView(applicationContext)) instead of Activity context, then add it to the layout programmatically. (3) Manual cleanup — in onDestroy(): remove WebView from its parent, call webView.stopLoading(), webView.loadUrl("about:blank"), webView.destroy(), null the reference. (4) On Android 7+, WebView runs in a separate process by default (the "WebView renderer"), which helps isolate crashes but doesn't fully prevent the Java-side leak.

Q25. How does LeakCanary detect leaks internally? Walk through its detection pipeline.

LeakCanary's pipeline: (1) Object watching — it registers ActivityLifecycleCallbacks and FragmentLifecycleCallbacks. When onDestroy() fires, it wraps the Activity/Fragment in a KeyedWeakReference with a unique key and a ReferenceQueue. (2) Retained object detection — after a 5-second delay (configurable), it checks if the WeakReference was enqueued (meaning the object was GC'd). If not, it triggers Runtime.gc() and checks again. If still not enqueued, the object is "retained" (likely leaked). (3) Heap dump — when retained objects exceed a threshold (default 5), LeakCanary calls Debug.dumpHprofData() to capture a heap dump (.hprof file). (4) Heap analysis — the shark library parses the hprof, finds the shortest strong-reference path from any GC root to the retained object using BFS on the object graph. (5) Leak grouping — identical reference chains are grouped into a single leak signature. The key insight: LeakCanary doesn't just find retained objects — it finds the dominator in the reference chain (the object whose removal would free the leaked object), which is usually the actual bug location.

Q26. What are "architectural boundary" leaks? Give examples across Repository, UseCase, and ViewModel layers.

Architectural boundary leaks occur when a long-lived layer holds a reference to a short-lived layer. Examples: (1) Repository → Activity: a Repository singleton registers a callback on a system service and the callback lambda captures an Activity reference. The Repository outlives the Activity. (2) UseCase → ViewModel: a UseCase holds a reference to a ViewModel's MutableStateFlow and the ViewModel is cleared but the UseCase (in a DI singleton scope) keeps the StateFlow alive, which holds the last emitted value (possibly containing large data). (3) ViewModel → Fragment View: ViewModel stores a RecyclerView.Adapter reference to survive rotation — the adapter holds itemViews from the old layout. Fix: enforce unidirectional data flow (lower layers never reference upper layers), use Flow/LiveData for reactive communication (observer pattern breaks the reference), scope DI correctly (@ActivityScoped vs @Singleton), and use LeakCanary's custom ObjectWatcher to watch DI-scoped objects.

Q27. When is WeakReference the right tool for preventing leaks, and when is it misused?

WeakReference is appropriate when you need a non-owning reference to an object whose lifecycle you don't control — e.g., a cache that should not prevent GC, or a Handler in a static inner class that needs to reference its Activity. Misuse cases: (1) Using WeakReference instead of fixing the architecture — if your singleton holds a WeakReference<Activity> and checks for null everywhere, you've added complexity without fixing the real problem (the singleton shouldn't reference the Activity at all). (2) WeakReference in collections — a List<WeakReference<Listener>> accumulates null entries that must be cleaned up manually; use WeakHashMap or a proper listener registry with explicit unregister. (3) Assuming WeakReference prevents all leaks — if the object is also referenced strongly from another path, the WeakReference doesn't help. The correct fix is almost always lifecycle-aware components (lifecycleScope, LiveData, DisposableEffect) rather than WeakReference. WeakReference is a last resort for legacy code or framework-level utilities.

Q28. Explain the InputMethodManager leak. Why has it persisted across Android versions?

The InputMethodManager (IMM) singleton holds references to the currently focused View via mServedView, mNextServedView, and the mCurRootView field. When an Activity with an active input field is destroyed, IMM may retain a reference to the View (and thus the Activity) until another Activity takes focus. The chain: InputMethodManager (static singleton) → mServedView → View → Activity → DecorView → all Views. This has been a known issue since Android 4.x. Google has partially fixed it in various releases (Android 10 added InputMethodManager.clearConnectionLocked() on Activity destroy), but edge cases persist — especially with DialogFragments and multi-window. Mitigations: (1) In onDestroy(), call currentFocus?.let { imm.hideSoftInputFromWindow(it.windowToken, 0) }. (2) Clear focus from EditTexts before finishing. (3) LeakCanary specifically identifies this as a "known framework leak" and excludes it from leak reports by default via its AndroidReferenceMatchers list.

ANR Deep Dive (Q29–Q36)

Q29. What are the different ANR types and their timeout thresholds?

Android has four ANR types with different timeouts: (1) Input dispatching timeout — 5 seconds. If an input event (touch, key) is dispatched to a window but the app doesn't finish handling it within 5s, InputDispatcher in system_server triggers the ANR. (2) BroadcastReceiver timeout — 10 seconds for foreground broadcasts (Intent.FLAG_RECEIVER_FOREGROUND), 60 seconds for background broadcasts. Measured from AMS.broadcastIntentLocked() dispatch to onReceive() return. (3) Service timeout — 20 seconds for foreground services (SERVICE_TIMEOUT_MSG), 200 seconds for background services. Measured from AMS.realStartServiceLocked() to Service.onCreate()/onStartCommand() return. (4) ContentProvider timeout — 10 seconds for ContentProvider.onCreate() during process startup. These timeouts are hardcoded in ActivityManagerService and InputDispatcher.cpp. The 5-second input timeout is the most user-visible and the primary ANR metric in Google Play Console's vitals.

Q30. How does InputDispatcher detect an ANR internally?

InputDispatcher (in frameworks/native/services/inputflinger/) maintains a per-window wait queue of dispatched but unacknowledged events. When it dispatches an event via the input channel (a socket pair between system_server and the app), it records the dispatch time. The app's InputEventReceiver (on the UI thread's Looper) processes the event and sends a "finished" signal back through the socket. InputDispatcher::checkWindowReadyForMoreInputLocked() checks: if the oldest unacknowledged event is older than 5 seconds, it calls onAnrLocked(). This triggers InputManagerService → ActivityManagerService.inputDispatchingTimedOut() → ANR dialog. Key nuance: the timer starts when the event is dispatched, not when the user touches the screen. If the main thread is blocked before the event is even dequeued from the Looper, the 5 seconds still counts from dispatch time. This is why a 3-second disk I/O followed by a 3-second network call causes an ANR even though neither alone exceeds 5 seconds.

Q31. How do you analyze a traces.txt (ANR trace) file? What are the key things to look for?

When an ANR occurs, AMS sends SIGQUIT to the offending process, which triggers ART's signal handler to dump all thread stacks to /data/anr/traces.txt. Key analysis steps: (1) Find "main" thread — look for "main" prio=5 tid=1. Its stack trace shows what the UI thread was doing at ANR time. (2) Check thread state — Blocked (waiting for a monitor lock), Waiting (in Object.wait()), TimedWaiting (in Thread.sleep() or Object.wait(timeout)), Native (in JNI/syscall). (3) Look for lock contention — - waiting to lock <0x...> (a java.lang.Object) held by thread XX tells you exactly which thread holds the lock. Follow thread XX's stack to find why it's holding the lock. (4) Check Binder threads — if main thread shows android.os.BinderProxy.transactNative, it's blocked on a synchronous Binder call. (5) Check I/O — java.io.FileInputStream.read or SQLiteDatabase.rawQuery on main thread. (6) Check all threads — deadlocks show as two threads each waiting for a lock held by the other.

Q32. How can SharedPreferences.apply() cause an ANR? This is a common trap.

apply() is asynchronous — it writes to an in-memory map immediately and schedules a disk write on a background thread via QueuedWork. However, Activity.onPause() and Service.onStartCommand() call QueuedWork.waitToFinish(), which blocks the main thread until ALL pending apply() writes complete. If you called apply() multiple times (e.g., logging events to SharedPreferences), each write is queued. On onPause(), the main thread blocks waiting for all queued writes to flush to disk. On slow storage (eMMC, encrypted devices), this can take seconds — causing an ANR. The fix: (1) Batch writes — call apply() once with all changes, not per-field. (2) Use DataStore (Jetpack) which uses coroutines and doesn't block the main thread on lifecycle transitions. (3) Avoid writing large values to SharedPreferences. (4) On Android 12+, Google partially mitigated this by reducing QueuedWork blocking, but the fundamental issue remains on older versions. This is one of the most common ANR causes in production apps.

Q33. How does lock contention cause ANRs? Give a concrete Android example.

Lock contention ANR occurs when the main thread tries to acquire a lock held by a background thread doing slow work. Concrete example: a Room database with a synchronized write method. Background thread acquires the lock, starts a large transaction (inserting 1000 rows). Main thread calls a read method that also needs the lock — it blocks. If the transaction takes >5 seconds, the main thread can't process input events → ANR. The traces.txt shows: "main" ... - waiting to lock <0x1234> held by thread "DefaultDispatcher-worker-3", and thread "DefaultDispatcher-worker-3" shows SQLiteDatabase.insertWithOnConflict. Fix: (1) Use Room with @Transaction and suspend functions — Room uses its own internal executor and doesn't hold locks across suspension points. (2) Use ReentrantReadWriteLock — multiple readers can proceed concurrently. (3) Use Mutex from kotlinx.coroutines — it suspends instead of blocking. (4) Never expose raw synchronized blocks to code that runs on both main and background threads.

Q34. What is goAsync() in BroadcastReceiver? How does it prevent ANR and what are its limits?

goAsync() returns a PendingResult object that extends the BroadcastReceiver's lifecycle beyond onReceive(). Normally, the system considers the receiver "done" when onReceive() returns and may kill the process. With goAsync(), you can hand off work to a coroutine or thread and call pendingResult.finish() when done. However, the ANR timeout still applies — you still have 10 seconds (foreground) or 60 seconds (background) from dispatch time. goAsync() doesn't extend the timeout; it just lets you use the time asynchronously instead of blocking onReceive().

override fun onReceive(context: Context, intent: Intent) {
    val pendingResult = goAsync()
    CoroutineScope(Dispatchers.IO).launch {
        try {
            doWork()
        } finally {
            pendingResult.finish()
        }
    }
}

For work exceeding the timeout, use WorkManager instead. Also note: goAsync() keeps the process alive but doesn't prevent the system from killing it under extreme memory pressure — it's not a substitute for a foreground service.

Q35. How do you detect and prevent deadlocks in an Android app?

Deadlocks occur when two or more threads each hold a lock the other needs. Detection: (1) traces.txt analysis — look for circular lock dependencies: thread A waiting for lock X held by thread B, thread B waiting for lock Y held by thread A. (2) StrictMode — StrictMode.ThreadPolicy.Builder().detectResourceMismatches() can catch some lock ordering issues. (3) Perfetto — the "Thread State" track shows threads in "Blocked" state with lock contention markers. (4) Watchdog — for system_server, the Watchdog thread detects deadlocks by monitoring critical locks. Prevention: (1) Lock ordering — always acquire locks in a consistent global order. (2) Avoid nested locks — if you must, document the ordering. (3) Use coroutines — Mutex from kotlinx.coroutines suspends instead of blocking, and structured concurrency prevents orphaned coroutines. (4) Use tryLock() with timeout — ReentrantLock.tryLock(5, TimeUnit.SECONDS) fails gracefully instead of deadlocking. (5) Single-threaded confinement — confine mutable state to a single thread (e.g., Dispatchers.Main for UI state) eliminating the need for locks entirely.

Q36. How does the ANR reporting pipeline work from detection to Play Console?

The pipeline: (1) InputDispatcher or AMS detects the timeout and calls ProcessErrorStateRecord.appNotResponding(). (2) AMS sends SIGQUIT to the offending process → ART dumps thread stacks to /data/anr/traces.txt. (3) AMS also dumps stacks of other important processes (system_server, top app) for context. (4) AppErrors.handleShowAnrUi() shows the ANR dialog to the user ("Wait" or "Close"). (5) If the user has opted into usage diagnostics, the ANR report (including traces, process state, memory info) is uploaded to Google's servers. (6) Play Console aggregates ANR reports in Android Vitals. The metric is "ANR rate" = ANR sessions / total sessions. Google's bad behavior threshold is >0.47% for overall ANR rate and >0.24% for user-perceived ANR rate. Exceeding these thresholds reduces your app's visibility in Play Store search results. On Android 11+, ApplicationExitInfo provides programmatic access to ANR details via ActivityManager.getHistoricalProcessExitReasons() with reason REASON_ANR.

OOM & Low Memory Killer (Q37–Q42)

Q37. Compare the kernel OOM killer with Android's lmkd. Why does Android need its own?

The Linux kernel OOM killer is a last resort — it triggers when the system is completely out of memory (no free pages, no swap, all reclaim failed). It selects a process based on oom_score (heuristic: largest RSS, least important) and sends SIGKILL. This is too late for Android — by the time the kernel OOM fires, the system is thrashing, the UI is frozen, and the user experience is destroyed. Android's lmkd (Low Memory Killer Daemon, in system/memory/lmkd/) proactively kills processes before the system reaches critical memory. It uses oom_score_adj values set by ActivityManagerService (via ProcessList.setOomAdj()) to prioritize kills. lmkd monitors memory pressure via PSI (Pressure Stall Information) on Android 10+ or the legacy /sys/module/lowmemorykiller/parameters/ interface on older kernels. When free memory drops below configured thresholds (minfree levels), lmkd kills the process with the highest oom_score_adj first — typically empty/cached processes, then background, then services.

Q38. Explain oom_adj scores. How does AMS assign them and what do the values mean?

oom_score_adj ranges from -1000 (never kill) to 1000 (kill first). AMS assigns scores based on process importance in OomAdjuster.computeOomAdjLocked(): NATIVE_ADJ (-1000): native system daemons. SYSTEM_ADJ (-900): system_server. PERSISTENT_PROC_ADJ (-800): persistent apps (phone, system UI). PERSISTENT_SERVICE_ADJ (-700): persistent services. FOREGROUND_APP_ADJ (0): the app the user is interacting with. VISIBLE_APP_ADJ (100): visible but not focused (e.g., behind a dialog). PERCEPTIBLE_APP_ADJ (200): running a foreground service. PERCEPTIBLE_LOW_APP_ADJ (250): running a bound service the user can perceive. BACKUP_APP_ADJ (300): hosting a backup agent. SERVICE_ADJ (500): running a started service. HOME_APP_ADJ (600): launcher. PREVIOUS_APP_ADJ (700): previous foreground app. SERVICE_B_ADJ (800): old service. CACHED_APP_MIN_ADJ (900) to CACHED_APP_MAX_ADJ (999): cached processes, ordered by LRU. AMS recalculates oom_adj on every lifecycle transition (activity start/stop, service bind/unbind, broadcast delivery).

Q39. What are the onTrimMemory() levels? How should a well-behaved app respond to each?

ComponentCallbacks2.onTrimMemory(level) is called by AMS when the system needs memory. Levels in order of increasing severity: TRIM_MEMORY_UI_HIDDEN (20): your UI is no longer visible — release UI-only resources (image caches, view caches). TRIM_MEMORY_RUNNING_MODERATE (5): device is getting low, your app is running — trim non-critical caches. TRIM_MEMORY_RUNNING_LOW (10): device is low, your app is running — release more aggressively. TRIM_MEMORY_RUNNING_CRITICAL (15): device is critically low — release everything possible or you'll be killed. TRIM_MEMORY_BACKGROUND (40): your app is in the LRU cached list — release as much as possible to survive longer. TRIM_MEMORY_MODERATE (60): you're in the middle of the cached list. TRIM_MEMORY_COMPLETE (80): you're near the end of the cached list, about to be killed. Best practice: Glide/Coil listen to these and call trimMemory(level) on their caches. OkHttp's connection pool can be evicted. Room's invalidation tracker can be paused. Responding properly to level 20 alone can save 20–50 MB per backgrounded app.

Q40. How does native OOM differ from Java OOM? How do you diagnose native memory exhaustion?

Java OOM (OutOfMemoryError) is thrown by ART when the Java heap can't grow further (after GC). It's caught by the standard exception handler and produces a clear stack trace. Native OOM occurs when malloc()/mmap() fails — the process receives SIGABRT from jemalloc's abort() or the kernel sends SIGKILL via the OOM killer. Native OOM produces a tombstone (/data/tombstones/) with a native stack trace, not a Java stack trace. Diagnosis: (1) adb shell dumpsys meminfo <pid> — check "Native Heap" size. (2) Debug.getNativeHeapAllocatedSize() programmatically. (3) Malloc debug — adb shell setprop libc.debug.malloc.options backtrace enables allocation tracking with stack traces. (4) heapprofd (Perfetto) — continuous native heap profiling with minimal overhead, shows allocation call stacks. (5) AddressSanitizer (ASan) — detects use-after-free and buffer overflows in native code. Common native OOM causes: Bitmap accumulation (Android 8+), WebView Chromium renderer, RenderNode display lists, SQLite page cache, JNI-allocated buffers not freed.

Q41. What are PSI (Pressure Stall Information) metrics? How does lmkd use them?

PSI (Linux 4.20+, Android 10+) measures the percentage of time tasks are stalled waiting for memory resources. Three levels: some (at least one task stalled), full (all tasks stalled). Reported as rolling averages over 10-second, 60-second, and 300-second windows via /proc/pressure/memory. lmkd on Android 10+ uses PSI instead of the legacy vmpressure events. It registers PSI monitors via psi_monitor_open() with configurable thresholds — e.g., "trigger when 'some' stall exceeds 70ms in any 1-second window." When the threshold is breached, lmkd wakes up and kills the least-important process (highest oom_score_adj). PSI is more accurate than vmpressure because it measures actual task stall time rather than page reclaim activity. lmkd has three kill levels mapped to PSI thresholds: low (kill cached), medium (kill services), critical (kill perceptible). The thresholds are tuned per-device by OEMs based on RAM size and storage speed.

Q42. How does process priority affect survival during memory pressure? What can an app do to improve its priority?

Process priority is determined by oom_score_adj — lower values survive longer. Legitimate ways to improve priority: (1) Foreground service — moves you to PERCEPTIBLE_APP_ADJ (200); requires a visible notification. (2) Bound service from a foreground app — if a foreground Activity binds to your service with BIND_IMPORTANT, your process gets FOREGROUND_APP_ADJ (0). (3) Active BroadcastReceiver — while onReceive() executes, the process is temporarily foreground. (4) ContentProvider being queried — temporarily elevated while serving a query. Anti-patterns: (1) Empty foreground services just to stay alive — Google Play policy violation, detected by Play Protect. (2) Scheduling repeating alarms to restart killed services — wastes battery, killed by Doze. (3) Mutual binding between two apps to keep each other alive — detected and penalized since Android 10. The correct approach: design for process death. Use WorkManager for deferred work, ViewModel + SavedStateHandle for UI state, and accept that cached processes will be killed.

App Startup (Q43–Q47)

Q43. Define cold, warm, and hot start. What happens in each phase of a cold start?

Cold start: process doesn't exist. Phases: (1) Process creation (~100–300 ms) — Zygote fork, specialize (setuid, SELinux, seccomp). (2) Application.onCreate() — DI initialization (Dagger/Hilt component build), ContentProvider.onCreate() for all declared providers, library initializations. (3) Activity creation — Activity.onCreate(), layout inflation, view creation. (4) First frame — measure/layout/draw, RenderThread GPU upload. Total: 500 ms–3+ seconds. Warm start: process exists but Activity was destroyed (e.g., back press then relaunch). Skips Zygote fork and Application.onCreate(). Starts at Activity creation. Typically 200–500 ms. Hot start: process and Activity exist (e.g., home button then return). Only onRestart()/onStart()/onResume(). Typically <100 ms. Google measures TTID (Time to Initial Display) — first frame rendered, and TTFD (Time to Full Display) — app calls reportFullyDrawn(). Play Console's startup metric uses TTID. The Macrobenchmark library measures both with StartupTimingMetric.

Q44. Why are ContentProviders a major startup cost? How does the App Startup library fix this?

Every ContentProvider declared in the merged manifest has its onCreate() called during Application.attachBaseContext() → installContentProviders(), before Application.onCreate(). Many libraries abuse ContentProviders for auto-initialization (Firebase, WorkManager, Lifecycle, etc.) — each adds 2–10 ms. With 15+ libraries, this adds 50–150 ms to cold start. The App Startup library (androidx.startup) consolidates all initializations into a single ContentProvider (InitializationProvider). Libraries implement the Initializer<T> interface, declaring dependencies between initializers. App Startup topologically sorts them and runs them in order within the single provider's onCreate(). You can also disable automatic initialization and call AppInitializer.getInstance(context).initializeComponent(MyInitializer::class.java) lazily — deferring non-critical initialization to after the first frame. This can save 50–100 ms on cold start. To audit ContentProvider cost: adb shell am start -W reports TotalTime, and Perfetto's bindApplication slice shows per-provider timing.

Q45. What is reportFullyDrawn() and why does it matter for startup metrics?

Activity.reportFullyDrawn() signals to the system that the app has finished rendering all meaningful content — not just the first frame (which may be a skeleton/placeholder), but the actual data the user wants to see. This marks the TTFD (Time to Full Display) metric. Internally, it calls ActivityTaskManagerService.reportFullyDrawn() via Binder, which logs the timestamp. On Android 10+, Perfetto captures this as the reportFullyDrawn() slice. Google Play Console uses TTFD (when reported) alongside TTID for startup vitals. Without calling it, Play Console only has TTID, which may look fast even if the user stares at a loading spinner for 3 seconds. Best practice: call reportFullyDrawn() after your RecyclerView/LazyColumn has rendered its first page of real data. In Compose, use ReportDrawn() or ReportDrawnWhen { condition } from activity-compose. For lazy loading, call it after the first network response populates the UI. This metric directly influences your Play Store ranking.

Q46. How does the Splash Screen API (Android 12+) work internally? What happens during the splash phase?

The Splash Screen API (SplashScreen in androidx.core.splashscreen) replaces the old "window background theme" hack. Internally: (1) When AMS starts the Activity, it creates a SplashScreenView (a special window) before the app process is even ready. This window shows the app icon (from android:windowSplashScreenAnimatedIcon) with an optional brand image and background color. (2) The splash window is owned by system_server's WindowManagerService, not the app — so it appears instantly even during Zygote fork. (3) Once the app draws its first frame, SplashScreenView transitions to the app's window with a configurable exit animation. (4) The app can customize the exit via splashScreen.setOnExitAnimationListener { splashScreenView -> /* animate out */ }. (5) setKeepOnScreenCondition { !isReady } extends the splash until your data is loaded — but be careful: keeping it too long (>1 second) frustrates users. The key optimization: the splash is a system-rendered window, so it appears in <100 ms regardless of app initialization time, giving the perception of instant launch.

Q47. What are the top 5 cold start optimizations a Staff Engineer should implement?

(1) Defer initialization — use App Startup library to consolidate ContentProviders and lazy-initialize non-critical libraries (analytics, crash reporting) after the first frame. Measure each library's init cost with Perfetto. (2) Reduce main thread work in Application.onCreate() — move DI graph construction to a background thread where possible (Hilt's @EarlyEntryPoint for critical deps, defer the rest). (3) Optimize layout inflation — reduce view hierarchy depth, use ViewStub for below-the-fold content, avoid deeply nested ConstraintLayouts. In Compose, ensure the initial composition is minimal. (4) Baseline Profiles — pre-compile critical startup paths to avoid JIT compilation during launch. This alone can reduce TTID by 20–40%. (5) Async data loading with placeholder UI — render a skeleton screen immediately, load data in parallel via viewModelScope, update the UI reactively. Call reportFullyDrawn() when data arrives. Bonus: use R8 full mode to reduce dex size (fewer classes to load), and enable android:enableOnBackInvokedCallback to avoid loading the legacy back-press framework. Measure everything with Macrobenchmark's StartupTimingMetric.

Baseline Profiles (Q48–Q50)

Q48. Compare AOT, JIT, and interpreted execution in ART. How do Baseline Profiles fit into this picture?

ART has three execution tiers: (1) Interpreted — bytecode interpreted instruction-by-instruction; slowest but no compilation cost. Used for cold code paths. (2) JIT (Just-In-Time) — at runtime, ART's JIT compiler (jit::Jit) profiles method invocation counts and compiles hot methods to native code in a background thread. JIT-compiled code runs 5–10x faster than interpreted but has warmup cost (first invocations are slow). JIT profiles are saved to /data/misc/profiles/. (3) AOT (Ahead-Of-Time) — dex2oat compiles dex to native code during install or idle maintenance. AOT code is fastest (no warmup) but increases install time and storage. Baseline Profiles bridge the gap: they tell dex2oat which methods and classes to AOT-compile at install time, without compiling the entire app. This gives AOT speed for critical paths (startup, navigation) while keeping the APK small. Without Baseline Profiles, a fresh install runs entirely interpreted/JIT for the first few uses until the JIT profile accumulates enough data for bg-dexopt to AOT-compile. Baseline Profiles eliminate this "first run penalty."

Q49. How are Baseline Profiles generated, distributed, and applied? Walk through the full pipeline.

Generation: Use Macrobenchmark with BaselineProfileRule. The test runs critical user journeys (startup, navigation, scrolling) on a device/emulator. ART's JIT profiler records which methods are executed. The test extracts the profile via adb shell cmd package dump-profiles → produces a human-readable profile (HRF) file listing hot methods, startup methods, and post-startup methods. Packaging: The profile is placed in src/main/baseline-prof.txt (or generated into build/). AGP compiles it into a binary profile (.dm file) and includes it in the AAB/APK as assets/dexopt/baseline.prof. Distribution: Google Play delivers the .dm file alongside the APK. On install, PackageManagerService passes it to dex2oat. Application: dex2oat uses the profile to AOT-compile only the listed methods into the .odex file. Classes listed as "startup" are placed in a contiguous dex layout for better page cache locality. Cloud Profiles (Android 9+): Google Play aggregates JIT profiles from users who opted in, creating a "cloud profile" that's distributed to new installs — even without a developer-provided Baseline Profile. Cloud profiles take ~2 weeks to accumulate; Baseline Profiles provide day-one optimization.

Q50. What is dex layout optimization? How do Baseline Profiles and Startup Profiles improve page cache behavior?

Dex layout optimization reorders classes and methods within the .dex file so that startup-critical code is physically contiguous. When the app starts, the OS loads dex pages via mmap() — if startup classes are scattered across the dex file, the kernel must fault in many non-contiguous pages, causing I/O stalls. With dex layout optimization, startup classes are packed into the first few pages, so a sequential read brings in all needed code with minimal page faults. Startup Profiles (a subset of Baseline Profiles) specifically mark classes as "startup" (prefix S in the HRF). dex2oat uses this to: (1) reorder the .odex file so startup methods are contiguous, (2) place startup classes in the primary dex file (classes.dex) rather than secondary dex files. The impact is significant on cold start: 15–30% reduction in page faults, translating to 50–200 ms faster TTID on devices with slow storage (eMMC). R8's -optimizeartprofiles flag further optimizes the dex layout based on the profile. Combined with ProfileInstaller (from androidx.profileinstaller), which side-loads profiles on devices that don't receive them from Play (sideloaded APKs, alternative stores), Baseline Profiles provide consistent performance across all distribution channels.

Corner Cases & Advanced Topics (Q51–Q58)

Q51. How does ApplicationExitInfo (Android 11+) change production ANR/OOM debugging? What data does it provide that Play Vitals doesn't?

ActivityManager.getHistoricalProcessExitReasons() returns a list of ApplicationExitInfo objects describing why previous process instances died. For ANRs (REASON_ANR), it provides: (1) Full thread dump via traceInputStream — the exact same data as traces.txt, but accessible programmatically on the next app launch. (2) PSS and RSS at time of death — tells you if memory pressure contributed. (3) Process importance — whether the ANR was foreground or background. (4) Custom state via processStateSummary — up to 256 bytes you set via ActivityManager.setProcessStateSummary(), allowing you to record the current screen, pending operations, or cache sizes. Play Vitals only provides aggregated, sampled ANR clusters with partial stack traces. ApplicationExitInfo gives you every ANR for every user, with full thread dumps and custom context. At L6, you should architect a pipeline: on each app launch, query exit reasons, upload ANR traces to your backend, and build dashboards that track ANR rates per build, per screen, and per device model. This is how Google's internal apps achieve <0.1% ANR rates.

Q52. Explain TLAB (Thread-Local Allocation Buffer) in ART. Why does it make object allocation nearly free, and what happens when it's exhausted?

ART's RegionSpace assigns each thread a private chunk of memory (typically one 256 KB region) as its TLAB. Allocation within a TLAB is a simple bump-pointer increment: tlab_pos += size — no locks, no CAS, just 3 CPU instructions. This makes >99% of allocations lock-free and extremely fast. When the TLAB is exhausted: (1) The thread returns the current TLAB to RegionSpace. (2) It requests a new free region (requires a brief lock on RegionSpace). (3) If no free regions exist, ART triggers a young-gen GC to free regions. (4) If GC doesn't free enough, ART grows the heap footprint (up to heapgrowthlimit). (5) If at the limit, OutOfMemoryError is thrown. The key performance insight: TLAB exhaustion rate is directly proportional to allocation rate. An app allocating 50 MB/s exhausts a 256 KB TLAB every ~5 ms, triggering frequent slow-path calls and GC. This is why allocation rate is the #1 GC metric — not heap size, not object count, but bytes/second allocated. Reduce allocation rate by reusing objects, using @JvmInline value class, and avoiding allocations in onDraw() or Compose's composition phase.

Q53. How does Compose's recomposition affect GC? What allocation patterns does it introduce and how do you optimize them?

Compose recomposition creates allocations for: (1) Lambda instances — every onClick = { ... } creates a new lambda object unless the compiler memoizes it. (2) Modifier chains — Modifier.padding().background() creates new CombinedModifier objects. (3) State snapshot reads — reading MutableState registers the Composable as an observer, allocating SnapshotStateObserver entries. (4) Composer bookkeeping — groups, slots, and change lists. In LazyColumn, items entering/leaving the viewport cause composition/disposal, creating a high allocation rate during fast scrolling. Optimizations: (a) Strong Skipping Mode (Compose 1.5.4+) — compiler auto-memoizes lambdas and skips unstable parameters via equals(), reducing allocations by 30-50%. (b) Stability annotations — mark data classes as @Stable or @Immutable so the compiler can skip recomposition entirely. (c) remember {} — cache expensive objects in the slot table. (d) derivedStateOf {} — reduce recomposition scope. (e) contentType in LazyColumn — enables SubComposition reuse, reducing disposal/recomposition churn. Target: <2 MB/s allocation rate during steady-state scrolling.

Q54. How do you profile native memory leaks on Android? Compare heapprofd, malloc_debug, and Android Studio's native profiler.

Native memory leaks (Bitmaps, JNI, SQLite) are invisible to Java heap tools. Three approaches: (1) heapprofd (Perfetto) — the gold standard. Continuous native heap profiling with configurable sampling rate. Captures allocation call stacks with minimal overhead (~1-5%). Run via adb shell perfetto with the android.heapprofd data source, or from Android Studio's Profiler. Shows allocation flamegraphs, outstanding allocations, and growth over time. Works on non-debuggable release builds (Android 10+). (2) malloc_debug — set adb shell setprop libc.debug.malloc.options backtrace to enable per-allocation stack traces. High overhead (~10-50x slowdown), but captures every allocation. Useful for reproducing specific leaks in debug builds. (3) Android Studio Native Memory Profiler — GUI wrapper around heapprofd. Shows real-time native heap graph, allocation tracking, and leak detection. Requires a debuggable build. For production: use heapprofd via Perfetto with a low sampling rate (1 in 1000 allocations) to identify allocation sites. For development: use Android Studio's profiler for interactive investigation. Key metric: if Debug.getNativeHeapAllocatedSize() grows monotonically across Activity recreations, you have a native leak.

Q55. What is the FinalizerWatchdogDaemon timeout crash? Why does it happen and how do you prevent it?

ART runs a FinalizerWatchdogDaemon thread that monitors the FinalizerDaemon thread. If a single finalize() method takes longer than 10 seconds, the watchdog kills the process with TimeoutException. The stack trace typically shows: java.util.concurrent.TimeoutException: android.graphics.Bitmap.finalize() timed out after 10 seconds. This happens when: (1) The FinalizerDaemon thread is blocked — e.g., a finalizer calls a synchronized method that's locked by another thread. (2) The finalizer queue is backed up — too many finalizable objects were created, and the single-threaded FinalizerDaemon can't keep up. (3) The device is under extreme CPU pressure — the daemon thread doesn't get scheduled for 10+ seconds. Prevention: (a) Never use finalize() — use AutoCloseable + use {} for resources, or Cleaner/NativeAllocationRegistry for native memory. (b) Avoid creating large numbers of objects with finalizers. (c) If using JNI, use DeleteGlobalRef/DeleteLocalRef explicitly instead of relying on GC. (d) On Android 8+, Bitmap uses NativeAllocationRegistry (Cleaner-based), eliminating this crash for Bitmaps. Legacy libraries with finalize() are the remaining risk.

Q56. How does madvise() work in ART's memory management? Compare MADV_DONTNEED and MADV_FREE.

After GC frees regions in RegionSpace, the virtual address space is still mapped but the physical pages are no longer needed. ART uses madvise() to tell the kernel to reclaim the physical pages. MADV_DONTNEED: immediately releases the physical pages. Next access triggers a page fault and the kernel provides a new zero-filled page. This is the most aggressive option — RSS drops immediately. Used by ART after every GC and during background compaction. MADV_FREE (Android 12+, Linux 4.5+): marks pages as "lazily freeable." The kernel may reclaim them if under memory pressure, but if the app accesses them before reclaim, the original data is still there — no page fault, no zeroing. This is faster than MADV_DONTNEED when memory isn't tight because it avoids unnecessary page faults if the app reuses the memory soon. ART uses MADV_FREE for recently freed regions in foreground mode (where the memory might be reused quickly) and MADV_DONTNEED for background compaction (where returning memory to the system is the priority). The choice between them directly affects how quickly your app's RSS drops after GC — important for surviving lmkd's kill decisions.

Q57. Design a memory-efficient image pipeline for an app that displays thousands of photos. What are the key architectural decisions?

Key decisions for a staff-level image pipeline: (1) Three-tier cache: memory cache (LruCache sized to 1/8 of memoryClass), disk cache (DiskLruCache, 50-250 MB), and network. Memory cache uses Bitmap.Config.HARDWARE (GPU memory, not counted in Java or native heap) for display-only images. (2) Downsampling: decode at the display size, not the source size. Use BitmapFactory.Options.inSampleSize (power of 2 for efficiency) or ImageDecoder (Android 9+) with setTargetSize(). A 4000×3000 photo displayed in a 400×300 thumbnail should decode at 1/10th size = 400 KB instead of 46 MB. (3) Bitmap pooling: reuse Bitmap allocations via inBitmap (pre-O) or Glide's BitmapPool. Reduces allocation rate and GC pressure. (4) onTrimMemory response: register ComponentCallbacks2, evict memory cache on TRIM_MEMORY_UI_HIDDEN, trim to 50% on RUNNING_LOW. (5) Prefetching with backpressure: prefetch 2-3 screens ahead in RecyclerView/LazyColumn, but cancel prefetch requests when scrolling direction changes. (6) Monitoring: track native heap growth via Debug.getNativeHeapAllocatedSize(), set alerts when total PSS exceeds 60% of memoryClass. Libraries like Coil and Glide implement most of this — the L6 value is understanding why each decision matters and being able to diagnose when the pipeline breaks down.

Q58. An app has 2% ANR rate and 500 MB PSS on flagship devices. Walk through your investigation and remediation plan as a Staff Engineer.

Phase 1 — Data collection: (a) Query ApplicationExitInfo for all REASON_ANR exits — extract thread dumps and custom state. (b) Pull Play Vitals ANR clusters — group by stack trace signature. (c) Capture Perfetto traces during reproduction: sched, am, wm, input categories. (d) Run dumpsys meminfo to break down the 500 MB: Java heap, native heap, graphics, code, stack. Phase 2 — ANR triage: (a) Classify ANRs: input dispatch (5s), broadcast (10s), service (20s). (b) For each cluster, read the main thread state: Blocked (lock contention), Native (Binder/IO), Runnable (CPU-bound). (c) For lock contention: identify the holding thread and what it's doing. (d) For Binder: identify which system service is slow. Phase 3 — Memory triage: (a) If native heap is >200 MB: run heapprofd to find allocation sites (likely Bitmaps, WebView). (b) If Java heap is high: take heap dump, check for Activity/Fragment leaks. (c) If graphics is high: audit Surface/TextureView usage. Phase 4 — Remediation: (a) Replace synchronized blocks with Mutex or lock-free structures. (b) Move all Binder calls off main thread. (c) Implement onTrimMemory to release caches.
