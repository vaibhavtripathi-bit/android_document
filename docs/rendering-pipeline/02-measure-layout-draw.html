Measure / Layout / Draw

The View system's rendering traversal — ViewRootImpl.performTraversals(), the measure/layout/draw contracts, invalidation mechanics, MeasureSpec, and hardware-accelerated recording

1. ViewRootImpl — The Root of Everything

ViewRootImpl is the bridge between the WindowManager and the View tree. It owns the Surface, receives VSYNC callbacks from Choreographer, and drives the entire traversal:

// ViewRootImpl.performTraversals() — the heart of View rendering (simplified from AOSP):
private void performTraversals() {
    // 1. Apply pending window attributes (size, flags, etc.)
    // 2. Determine if layout is needed (mLayoutRequested flag)

    if (layoutRequested) {
        // === MEASURE PHASE ===
        // Calculate MeasureSpec from window size
        int childWidthMeasureSpec = getRootMeasureSpec(desiredWindowWidth, lp.width);
        int childHeightMeasureSpec = getRootMeasureSpec(desiredWindowHeight, lp.height);
        performMeasure(childWidthMeasureSpec, childHeightMeasureSpec);
        // Calls mView.measure(widthSpec, heightSpec) on the root DecorView
    }

    if (layoutRequested || didLayout) {
        // === LAYOUT PHASE ===
        performLayout(lp, desiredWindowWidth, desiredWindowHeight);
        // Calls mView.layout(0, 0, width, height) on the root DecorView
    }

    if (!cancelDraw) {
        // === DRAW PHASE ===
        performDraw();
        // With hardware acceleration: records DisplayList via RenderNode
        // Without: draws directly to Canvas (software rendering — rare today)
    }
}
Staff-Level Insight — performTraversals() Is Called by Choreographer

ViewRootImpl registers a traversal callback with Choreographer via scheduleTraversals(). When a view calls requestLayout() or invalidate(), it sets a flag and posts a sync barrier on the message queue. The sync barrier blocks all synchronous messages until the Choreographer's VSYNC callback (which is asynchronous) runs. This ensures the traversal happens at the right time in the frame — after input and animation callbacks but before the frame deadline. The sync barrier is removed after performTraversals() completes.

2. The Measure Phase

// MeasureSpec: a 32-bit int encoding mode (2 bits) + size (30 bits)
// Modes:
//   EXACTLY    — parent determined exact size (match_parent, or exact dp)
//   AT_MOST    — child can be up to this size (wrap_content)
//   UNSPECIFIED — no constraints (ScrollView children, etc.)

// View.measure() contract:
protected void onMeasure(int widthMeasureSpec, int heightMeasureSpec) {
    // 1. Extract mode and size from MeasureSpec
    int widthMode = MeasureSpec.getMode(widthMeasureSpec);
    int widthSize = MeasureSpec.getSize(widthMeasureSpec);

    // 2. Calculate desired size based on content
    int desiredWidth = calculateContentWidth();

    // 3. Resolve based on mode
    int finalWidth = switch (widthMode) {
        case MeasureSpec.EXACTLY -> widthSize;           // Use exact size
        case MeasureSpec.AT_MOST -> min(desiredWidth, widthSize); // Cap at max
        case MeasureSpec.UNSPECIFIED -> desiredWidth;    // Use desired
    };

    // 4. MUST call setMeasuredDimension()
    setMeasuredDimension(finalWidth, finalHeight);
}

// ViewGroup.measureChild() — how parents measure children:
protected void measureChild(View child, int parentWidthSpec, int parentHeightSpec) {
    LayoutParams lp = child.getLayoutParams();
    int childWidthSpec = getChildMeasureSpec(parentWidthSpec, padding, lp.width);
    int childHeightSpec = getChildMeasureSpec(parentHeightSpec, padding, lp.height);
    child.measure(childWidthSpec, childHeightSpec);
}
The Double-Measure Problem

Some ViewGroups measure children twice: LinearLayout with weight (first pass to measure, second to distribute remaining space), RelativeLayout (measures children that depend on each other). In a deeply nested hierarchy with double-measuring parents, the cost is O(2^depth). This is the primary reason the View system's layout can be slow — and why Compose enforces single-pass measurement. In interviews, this is the canonical example of why "flat hierarchies" matter in the View system.

3. The Layout Phase

// View.layout() assigns the final position:
public void layout(int l, int t, int r, int b) {
    // l, t, r, b = left, top, right, bottom relative to parent

    boolean changed = setFrame(l, t, r, b); // Store position

    if (changed || (mPrivateFlags & PFLAG_LAYOUT_REQUIRED) != 0) {
        onLayout(changed, l, t, r, b); // Let the view position its children
    }
}

4. The Draw Phase

// View.draw() has a strict order:
public void draw(Canvas canvas) {
    // 1. Draw the background
    drawBackground(canvas);

    // 2. Save canvas layers for fading edges (if needed)

    // 3. Draw the content (your custom drawing)
    onDraw(canvas);

    // 4. Draw children (ViewGroup dispatches to children)
    dispatchDraw(canvas);

    // 5. Draw decorations (scrollbars, foreground)
    onDrawForeground(canvas);

    // 6. Draw highlight/focus indicators
}

// With hardware acceleration (default since Android 3.0):
// draw() doesn't render pixels directly.
// Instead, it RECORDS drawing commands into a DisplayList (RenderNode).
// The RenderThread later replays these commands on the GPU.
Staff-Level Insight — Recording vs Rendering

With hardware acceleration, the UI thread's "draw" phase is actually a recording phase. Canvas operations (drawRect, drawText, drawBitmap) are recorded into a DisplayList (stored in a RenderNode). The actual GPU rendering happens on the RenderThread after the UI thread finishes. This is why View.draw() is usually fast — it's just appending commands to a list, not doing GPU work. The expensive part is on the RenderThread, which runs in parallel with the next frame's UI work.

5. Invalidation — requestLayout() vs invalidate()

MethodWhat It TriggersWhen to Use
invalidate() Draw only — re-records the DisplayList for this view Visual change that doesn't affect size/position (color, alpha, text content that fits)
requestLayout() Measure + Layout + Draw — full traversal from this view up to root Size or position change (text that changes length, visibility change, margin change)
postInvalidateOnAnimation() Draw on next VSYNC Continuous animation (custom view that redraws every frame) 
// How invalidation propagates:
// invalidate() sets PFLAG_DIRTY on the view
// Then walks UP the tree: parent.invalidateChild(this, dirtyRect)
// Each parent marks itself as having a dirty child
// Eventually reaches ViewRootImpl, which calls scheduleTraversals()

// requestLayout() sets PFLAG_FORCE_LAYOUT on the view
// Walks UP the tree: parent.requestLayout()
// Each parent marks itself as needing layout
// ViewRootImpl.scheduleTraversals() triggers full measure + layout + draw

// Key optimization: if only invalidate() was called (no requestLayout),
// performTraversals() skips measure and layout phases entirely
Staff-Level Insight — The Invalidation Bubble

Both invalidate() and requestLayout() propagate upward to the root. For requestLayout(), every ancestor is marked as needing layout — even if only a leaf view changed size. This means a single requestLayout() on a deeply nested view causes the entire path to root to be re-measured and re-laid-out. This is why ConstraintLayout (flat hierarchy) outperforms nested LinearLayouts — fewer ancestors to re-measure. In Compose, requestLayout() only affects the LayoutNode and its parent's measure policy, not the entire tree.

6. Hardware Acceleration — RenderNode & DisplayList

// Every View has a RenderNode that stores its DisplayList:
class View {
    final RenderNode mRenderNode; // Created in constructor

    // When draw() is called:
    void updateDisplayListIfDirty() {
        RecordingCanvas canvas = mRenderNode.beginRecording(width, height);
        try {
            draw(canvas); // Records commands into the DisplayList
        } finally {
            mRenderNode.endRecording();
        }
    }
}

// RenderNode properties that can be changed WITHOUT re-recording:
// (These are "render properties" — applied by RenderThread directly)
renderNode.setTranslationX(100f)  // No invalidate() needed
renderNode.setTranslationY(50f)
renderNode.setScaleX(1.5f)
renderNode.setRotation(45f)
renderNode.setAlpha(0.5f)
renderNode.setElevation(8f)

// This is why View.setTranslationX() is fast — it doesn't re-record
// the DisplayList. The RenderThread applies the transform on the GPU.
// This is also why ViewPropertyAnimator is fast — it only changes
// render properties, never triggers invalidate() or requestLayout().
Staff-Level Insight — RenderNode Properties Are the Fast Path

The fastest way to animate a View is through RenderNode properties: translation, scale, rotation, alpha, elevation. These bypass the UI thread entirely during animation — the RenderThread reads the property values and applies them as GPU transforms. No measure, no layout, no draw re-recording. This is why ViewPropertyAnimator and ObjectAnimator targeting these properties are the recommended animation APIs. Custom properties (like a view's internal color) require invalidate() → re-recording → more expensive. In Compose, Modifier.graphicsLayer { } achieves the same thing — it maps directly to RenderNode properties.

When Software Rendering Is Forced

Certain Canvas operations aren't supported by hardware acceleration (e.g., some PathEffects, ComposeShader with certain modes). When you use these, the system silently falls back to software rendering for that View. You can force software rendering with view.setLayerType(View.LAYER_TYPE_SOFTWARE, null). Check unsupported operations in the official docs. In practice, this is rare in modern apps — most drawing operations are hardware-accelerated.
