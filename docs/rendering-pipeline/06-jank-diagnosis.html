Jank Diagnosis & Optimization

Systrace/Perfetto, FrameMetrics API, JankStats library, common jank patterns, dropped frame analysis, and optimization strategies

1. What Is Jank

Jank is any visible stutter or hitch in the UI — the frame didn't arrive at the display in time. Formally: a frame that missed the VSYNC deadline.

When a frame takes longer than the display's refresh interval (16.67ms at 60Hz, 8.33ms at 120Hz), the display shows the previous frame again. The user perceives a stutter — content appears to jump or freeze instead of smooth motion.

Perception Thresholds

Frame durationEffect
< 16.67ms (60Hz)Smooth — no perceived jank
16–33msSingle dropped frame — subtle hitch, most users notice
33–50ms2–3 dropped frames — visible stutter
> 50msSevere jank — UI feels frozen

At 60Hz, each frame is ~16.67ms. If the UI thread takes 25ms for a single frame, the display shows the previous frame for 8ms extra — one dropped frame. If it takes 50ms, three frames are dropped.

Staff-Level Insight — Jank vs Jitter

Jank is a missed frame. Jitter is variance in frame timing — frames arrive on time but with irregular spacing. Jitter can feel worse than occasional jank because it disrupts the perceived smoothness of animations. For example, 10 frames at 14ms, 18ms, 12ms, 20ms all hit the deadline but feel less smooth than 10 frames at exactly 16.67ms. Production monitoring should track both: jank count (frames > 16.67ms) and frame time variance (e.g., 95th percentile minus median).

2. Systrace / Perfetto

Systrace is the legacy trace capture tool. Perfetto is its successor — same underlying infrastructure, but with a modern UI and better trace analysis. Both capture kernel-level events (scheduling, CPU, GPU) plus app-level instrumentation.

Capturing a Trace

// Command line (Perfetto):
// 1. Record a trace:
adb shell perfetto -o /data/misc/perfetto-traces/trace.pb -t 60000 \
  -c /data/misc/perfetto-traces/config.pbtxt

// Or use the Perfetto UI: https://ui.perfetto.dev
// Connect device, select "Android", enable tracing categories, start

// 2. Pull trace:
adb pull /data/misc/perfetto-traces/trace.pb

// 3. Open in trace viewer: https://ui.perfetto.dev
// Upload trace.pb or open locally

// Legacy systrace:
python systrace.py -o trace.html -t 5 sched gfx view wm am freq

Key Trace Sections to Inspect

SectionWhat it shows
Choreographer#doFrameFull frame work on UI thread — input, animation, measure, layout, draw
Choreographer#doFrame → DrawFrameRecording DisplayList to RenderThread
Choreographer#doFrame → traversalMeasure, layout, draw phases
RenderThreadGPU command recording, sync, flush
dequeueBuffer / queueBufferBuffer queue operations — stalls if GPU is busy
SurfaceFlingerComposition — usually not the bottleneck

When diagnosing jank: find the longest slice inside doFrame. If it's in traversal → measure/layout/draw issue. If it's in DrawFrame → recording or sync. If it's in RenderThread → GPU or shader compilation.

3. FrameMetrics API

Window.OnFrameMetricsAvailableListener (API 24) provides per-frame timing breakdowns directly in your app. It's the programmatic way to measure frame performance without external tools.

Available Metrics

MetricDescription
TOTAL_DURATIONTotal frame time (input to display)
LAYOUT_MEASURE_DURATIONMeasure + layout phase
DRAW_DURATIONRecording DisplayList to RenderThread
SYNC_DURATIONSync with RenderThread (upload bitmaps, etc.)
COMMAND_ISSUE_DURATIONRenderThread issuing GPU commands
SWAP_BUFFERS_DURATIONWaiting for GPU to finish, swap buffers
GPU_DURATIONActual GPU execution time (API 28+)
// Registering FrameMetrics listener (API 24+):
window.addOnFrameMetricsAvailableListener { window, frameMetrics, dropCount ->
    val totalDuration = frameMetrics.getMetric(FrameMetrics.TOTAL_DURATION)
    val layoutMeasure = frameMetrics.getMetric(FrameMetrics.LAYOUT_MEASURE_DURATION)
    val drawDuration = frameMetrics.getMetric(FrameMetrics.DRAW_DURATION)
    val syncDuration = frameMetrics.getMetric(FrameMetrics.SYNC_DURATION)
    val commandIssue = frameMetrics.getMetric(FrameMetrics.COMMAND_ISSUE_DURATION)
    val swapBuffers = frameMetrics.getMetric(FrameMetrics.SWAP_BUFFERS_DURATION)

    // All values in nanoseconds
    val totalMs = totalDuration / 1_000_000
    if (totalMs > 16) {
        // Jank detected — log which phase dominated
        val phase = when {
            layoutMeasure > drawDuration && layoutMeasure > syncDuration -> "Layout"
            drawDuration > layoutMeasure && drawDuration > syncDuration -> "Draw"
            syncDuration > layoutMeasure && syncDuration > drawDuration -> "Sync"
            swapBuffers > commandIssue -> "GPU"
            else -> "Command"
        }
        Log.w("FrameMetrics", "Jank: ${totalMs}ms, dominant phase: $phase")
    }
}

// Remove when done
window.removeOnFrameMetricsAvailableListener(listener)
Production Use

FrameMetrics has overhead — ~2–5% CPU on the UI thread when enabled. Use it only in debug builds or behind a feature flag. For production, prefer JankStats which samples frames and aggregates. For one-off diagnosis, Systrace/Perfetto is more accurate and has no app overhead.

4. JankStats Library (Jetpack)

JankStats (Jetpack) is the recommended way to track frame performance in production. It samples frames, aggregates jank stats, and integrates with analytics.

// build.gradle
implementation("androidx.metrics:metrics-performance:1.0.0")

// Create and track JankStats:
val jankStats = JankStats.createAndTrack(window, frameDataCallback = { frameData ->
    // FrameData contains: frameDurationUiNanos, frameDurationCpuNanos,
    // isJankFrame, totalFrameDuration
    if (frameData.isJankFrame) {
        Log.w("JankStats", "Jank: ${frameData.totalFrameDuration / 1_000_000}ms")
    }
})

// Optional: add state context
jankStats.addState("Screen", "Home")

// When navigating:
jankStats.addState("Screen", "Detail")

// Optional: report to analytics (state is included in frameData when available)
jankStats.addFrameDataListener { frameData ->
    if (frameData.isJankFrame) {
        analytics.logEvent("jank", mapOf(
            "duration_ms" to (frameData.totalFrameDuration / 1_000_000)
        ))
    }
}

// Release when done:
jankStats.release()

JankStats uses FrameMetrics internally but samples frames (e.g., every 100th frame) to reduce overhead. It defines jank as: frameDuration > frameDuration * 1.5 or > 700ms (configurable).

5. Common Jank Patterns

PatternCauseSystrace SignatureFix
Main thread blocking Network, disk I/O, heavy computation on UI thread Long gaps in doFrame, or doFrame delayed Move to coroutines/Dispatchers.IO, use ViewModel
Layout thrashing requestLayout in onDraw, nested double-measure Multiple traversal / measure / layout inside one frame Never call requestLayout in draw; use postOnAnimation
Bitmap allocation during scroll Decoding large bitmap in onBindViewHolder GC pauses, BitmapFactory.decode in frame Use Coil/Glide, preload, decode off main thread
RecyclerView item inflation jank Complex XML inflation, nested layouts Long inflate or measure in onCreateViewHolder ViewStub, setHasFixedSize(true), prefetch, AsyncLayoutInflater
Shader compilation jank First frame of new UI (rounded rect, custom draw) GrGLProgram or vkCreateGraphicsPipelines on RenderThread Pre-warm shaders off-screen, Baseline Profile, ProfileInstaller
Sync stall RenderThread busy (GPU upload, sync) Long sync or dequeueBuffer wait Reduce bitmap size, pre-upload textures, avoid large bitmap upload
Overdraw Same pixel drawn multiple times High GPU fill rate, long DrawFrame Remove redundant backgrounds, use clipRect, reduce view hierarchy
Large bitmap upload Uploading large bitmap to GPU texture Long sync or upload Downsample before display, use inSampleSize, BitmapFactory.Options
Staff-Level Insight — Layout Thrashing

Layout thrashing is one of the most insidious jank causes. The classic mistake: calling requestLayout() or invalidate() during onDraw(). That triggers another layout pass in the same frame — doubling the work. Worse: if a parent's layout depends on a child's size, and the child's size changes during layout, you get nested measure/layout. The fix: use View.post { requestLayout() } or View.postOnAnimation { requestLayout() } to defer to the next frame. Or use ConstraintLayout with flat hierarchy to avoid nested measure.

6. Optimization Strategies

Hierarchy Flattening

Deep view hierarchies cause multiple measure/layout passes. Use ConstraintLayout with flat structure, or Merge in XML to avoid redundant layouts.

ViewStub

// Defer inflation of rarely-used views:
<ViewStub
    android:id="@+id/stub_details"
    android:layout="@layout/details_panel"
    android:inflateId="@+id/details_panel" />

// Inflate only when needed:
findViewById<ViewStub>(R.id.stub_details).inflate()

RecyclerView Optimization

// setHasFixedSize — if item size doesn't change, skip layout pass
recyclerView.setHasFixedSize(true)

// DiffUtil — efficient updates without full rebind
val diff = DiffUtil.calculateDiff(object : DiffUtil.Callback() {
    override fun getOldListSize() = oldList.size
    override fun getNewListSize() = newList.size
    override fun areItemsTheSame(old: Int, new: Int) = oldList[old].id == newList[new].id
    override fun areContentsTheSame(old: Int, new: Int) = oldList[old] == newList[new]
})
adapter.submitList(newList, diff)

// Prefetch — RecyclerView prefetches items before they're visible
recyclerView.setItemViewCacheSize(20) // Default: 2

Async Inflation

// AsyncLayoutInflater — inflate on background thread
AsyncLayoutInflater(context).inflate(R.layout.complex_item, parent) { view, _, _ ->
    // Use view on main thread
    runOnUiThread { parent.addView(view) }
}

R8 Optimization

Enable R8 full mode, shrink resources, and obfuscation. R8 inlines small methods and removes dead code — reduces APK size and can improve JIT warmup.

7. Baseline Profiles

Baseline Profiles are Android's solution for eliminating JIT compilation jank. When a user first opens a screen, the JIT compiler compiles hot methods — causing 50–200ms of jank. Baseline Profiles pre-compile these methods at install time.

How They Work

A Baseline Profile is a list of class.method descriptors that should be compiled ahead of time (AOT). At install, the system compiles these methods — so when the user opens the app, the code is already compiled.

Creating a Baseline Profile

// 1. Add Macrobenchmark module (androidTest):
// androidTest/java/com/example/app/BaselineProfileGenerator.kt

@RunWith(AndroidJUnit4::class)
class BaselineProfileGenerator {
    @get:Rule
    val rule = BaselineProfileRule()

    @Test
    fun generateBaselineProfile() {
        rule.collect(
            packageName = "com.example.app",
            maxIterations = 10
        ) {
            pressHome()
            startActivityAndWait()
            device.wait(Until.hasObject(By.text("Home")), 5_000)
            // Navigate to key screens
            device.findObject(By.text("Detail")).click()
            device.wait(Until.hasObject(By.text("Detail")), 5_000)
            // Scroll
            device.findObject(By.scrollable(true)).scroll(Direction.DOWN)
        }
    }
}
// 2. Run: ./gradlew :app:generateBaselineProfile
// 3. Output: app/src/main/baseline-prof.txt
// 4. Add to build.gradle:
android {
    defaultConfig {
        baselineProfileFile = file("baseline-prof.txt")
    }
}
Staff-Level Insight — Baseline Profile vs ProfileInstaller

Baseline Profiles (Android 7+) target app code paths — the methods you exercise during profile generation. ProfileInstaller (Jetpack) downloads cloud profiles from Google Play — these profiles are aggregated from real users' usage patterns. The best approach: ship your own Baseline Profile (covers your critical paths) and add ProfileInstaller for cloud profiles (covers paths you might miss). Cloud profiles are updated over time as users use the app. For L6: understand that Baseline Profiles eliminate JIT jank on first launch — but they don't help with shader compilation (that's GPU-side) or layout complexity (that's CPU-side).
