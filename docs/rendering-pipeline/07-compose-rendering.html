How Compose Plugs Into the Rendering Pipeline

LayoutNode → RenderNode mapping, Compose's three-phase pipeline, how Compose uses Choreographer, AndroidComposeView as the bridge, and key differences from the View system

1. AndroidComposeView — The Single View

The entire Compose UI tree lives inside one Android ViewGroup: AndroidComposeView. When you call setContent { } in a ComponentActivity, the activity's root is a ComposeView (or ComposeView inside a layout), which internally creates an AndroidComposeView to host the Compose tree.

AndroidComposeView implements the Owner interface and bridges the View system's traversal to Compose's internal pipeline:

// AndroidComposeView (simplified from compose-ui source):
internal class AndroidComposeView(context: Context) : ViewGroup(context), Owner {
    val root: LayoutNode = LayoutNode()  // Root of the entire Compose tree
    private val recomposer: Recomposer

    // ViewRootImpl calls these during performTraversals():
    override fun onMeasure(widthMeasureSpec: Int, heightMeasureSpec: Int) {
        val constraints = convertToConstraints(widthMeasureSpec, heightMeasureSpec)
        root.measure(constraints)  // Triggers entire Compose layout pass
        setMeasuredDimension(root.width, root.height)
    }

    override fun onLayout(changed: Boolean, l: Int, t: Int, r: Int, b: Int) {
        root.place(0, 0)  // Place entire tree
    }

    override fun dispatchDraw(canvas: Canvas) {
        root.draw(canvas)  // Draw entire tree to Canvas
    }

    // LayoutNodes request work via Owner:
    override fun onRequestMeasure(layoutNode: LayoutNode) {
        requestLayout()  // Schedules View system layout — Choreographer CALLBACK_TRAVERSAL
    }

    override fun onRequestRelayout(layoutNode: LayoutNode) {
        invalidate()  // Just redraw, no measure/layout
    }
}

From ViewRootImpl's perspective, there is exactly one child to measure, layout, and draw: AndroidComposeView. The View system has no visibility into the hundreds of LayoutNodes inside — they're an implementation detail. Touch events flow through dispatchTouchEvent() into Compose's pointer input system, which hits tests against the LayoutNode tree.

Staff-Level Insight — Why One View Changes Everything

The View system's traversal cost is O(n) in the number of Views. With Compose, the View tree has one node (AndroidComposeView) for the entire UI. Measure, layout, and draw are each called once at the View level. All the real work happens inside Compose's LayoutNode tree, which has its own optimized pipeline. This is why Compose can handle complex UIs with less overhead — the View system's traversal is trivial. The trade-off: Compose must do its own layout/draw scheduling. When a LayoutNode requests measure, onRequestMeasure calls requestLayout(), which posts to Choreographer's CALLBACK_TRAVERSAL. The next frame, ViewRootImpl runs performTraversals(), and the single onMeasure call triggers the full Compose layout pass.

2. Compose's Three Phases vs View's Three Phases

Both systems have three phases, but they map differently and have different semantics:

AspectView SystemCompose
Phase 1 Measure — View.measure(MeasureSpec) Composition — run @Composable functions, update slot table
Phase 2 Layout — View.layout(l,t,r,b) Layout — measure + place LayoutNodes
Phase 3 Draw — View.draw(Canvas) → DisplayList Draw — render LayoutNodes to Canvas
Pass count Multi-pass: parent can measure child multiple times (e.g., wrap_content in ScrollView) Single-pass: each node measured exactly once (O(n))
Phase independence Measure/layout/draw coupled; invalidate() often triggers full traversal Phases are independent; state read in Draw skips Composition + Layout
Invalidation granularity View-level: invalidate() marks view dirty; parent traversal may re-measure siblings Scope-level: only invalidated recomposition scopes re-run; layout/draw only dirty nodes
When phases run All three in performTraversals() during CALLBACK_TRAVERSAL Composition during CALLBACK_TRAVERSAL (Recomposer); Layout+Draw when AndroidComposeView is traversed

Key difference: Composition is unique to Compose. It determines what to show (the node tree). The View system has no equivalent — the View tree is built imperatively (addView/removeView). Compose's Layout phase combines measure and place; the View system separates them but they run back-to-back.

// Compose: state read in Draw phase skips Composition + Layout
@Composable
fun AnimatedCard(alpha: Float) {
    // BAD: reads in Composition → recomposition every frame
    Box(Modifier.alpha(alpha))

    // GOOD: reads in Draw → only draw phase runs
    Box(Modifier.graphicsLayer { this.alpha = alpha })
}

// View system: invalidate() typically triggers measure+layout+draw from root
view.invalidate()  // Marks view dirty; next traversal re-measures if layout requested
view.requestLayout()  // Marks layout dirty; parent may re-measure this and siblings

3. LayoutNode → RenderNode

Compose uses the same underlying rendering infrastructure as the View system. Each LayoutNode that draws has an associated RenderNode. When Compose draws, it records into DisplayLists just like Views do.

Modifier.graphicsLayer { } is the key: it configures the RenderNode directly. Properties like alpha, scaleX, scaleY, translationX, translationY, rotationZ, elevation, and shadowElevation map to RenderNode setters. The DisplayList doesn't need to be re-recorded when these change — the RenderThread reads the property and applies it as a GPU transform.

// graphicsLayer maps to RenderNode — no re-recording per frame
Box(
    modifier = Modifier
        .graphicsLayer {
            alpha = animatedAlpha
            scaleX = scale; scaleY = scale
            translationX = offsetX
            transformOrigin = TransformOrigin.Center
        }
) { /* content */ }

// Equivalent to View system:
// view.animate().alpha(0.5f).scaleX(1.2f).translationX(100f)
// Both use RenderNode properties → RenderThread applies on GPU

When a LayoutNode draws, it obtains a Canvas from its RenderNode (via the graphics layer abstraction). Drawing commands are recorded into the DisplayList. The RenderThread later replays this — the same pipeline as the View system. Compose doesn't reinvent rendering; it uses HWUI/Skia underneath.

4. How Compose Uses Choreographer

Compose's Recomposer drives composition. It uses MonotonicFrameClock, which on Android wraps Choreographer. When state changes invalidate recomposition scopes, the Recomposer schedules work for the next frame.

// MonotonicFrameClock (Android) — from compose-ui:
// AndroidUiFrameClock implements MonotonicFrameClock
// withFrameNanos { } posts to Choreographer.postFrameCallback()
// The callback runs during CALLBACK_ANIMATION (before TRAVERSAL)

// Recomposer's frame loop (simplified):
suspend fun runRecomposeAndApplyChanges() {
    while (shouldKeepRecomposing) {
        awaitWorkAvailable()  // Wait for invalidations
        MonotonicFrameClock.withFrameNanos { frameTimeNanos ->
            // 1. Run recomposition (Composition phase)
            for (composition in invalidatedCompositions) {
                composition.recompose()
            }
            // 2. Apply changes to LayoutNode tree
            for (composition in invalidatedCompositions) {
                composition.applyChanges()
            }
        }
    }
}

The critical ordering: CALLBACK_ANIMATION runs before CALLBACK_TRAVERSAL. Recomposer uses the animation callback (or a similar timing), so composition completes before the View system's measure/layout/draw. When ViewRootImpl.performTraversals() runs during CALLBACK_TRAVERSAL, the Compose tree is already up to date. AndroidComposeView.onMeasure then measures the updated LayoutNode tree.

// State change flow:
// 1. User clicks → count++
// 2. Snapshot apply → invalidates recomposition scopes reading count
// 3. Recomposer has pending work → registers with Choreographer
// 4. Next VSYNC → Choreographer runs callbacks:
//    - CALLBACK_ANIMATION: Recomposer runs recompose + applyChanges
//    - CALLBACK_TRAVERSAL: ViewRootImpl.performTraversals()
//      → AndroidComposeView.onMeasure/onLayout/dispatchDraw
// 5. Frame rendered
Staff-Level Insight — Composition Before Traversal

Compose intentionally runs composition before the View traversal. This ensures that when AndroidComposeView is measured and drawn, the LayoutNode tree already reflects the latest state. If composition ran during or after traversal, you'd get a one-frame delay — the old tree would be drawn, and the new tree would appear on the next frame. By using Choreographer's animation callback (or equivalent), Compose guarantees that state changes from input and animation are fully applied before the View system draws. This is the same principle as the View system: input → animation → traversal. Compose's "composition" is logically part of the "prepare for draw" work that must complete before traversal.

5. Compose Rendering Optimizations

Less Overdraw

Compose doesn't add default backgrounds. The View system often has android:windowBackground, theme backgrounds on layouts, and View.setBackground() — each can cause overdraw. Compose's Box, Column, and Row don't draw unless you add Modifier.background() or Surface. Fewer layers = less overdraw.

Single-Pass Measurement O(n)

Compose enforces single-pass measurement. Each LayoutNode is measured exactly once per layout pass. The View system allows multiple measure passes (e.g., ScrollView measuring children with UNSPECIFIED, then again with a derived spec). Worst case for Views: O(2^n) if every wrap_content triggers re-measure. Compose: always O(n).

No Double-Measure Problem

In the View system, requestLayout() can cause a parent to re-measure all children. Nested wrap_content can cascade. Compose's constraint-based layout avoids this — constraints flow down, sizes flow up, one pass.

Phase-Independent Invalidation

If state is read only in the Draw phase (e.g., Modifier.graphicsLayer { alpha = x }), Composition and Layout are skipped. Only the affected LayoutNodes re-draw. In the View system, invalidate() skips measure/layout but the draw still traverses from the root to the invalidated view. Compose can skip more of the tree.

// Optimizing animations — push reads to Draw phase:
val animatedAlpha by animateFloatAsState(if (visible) 1f else 0f)

// Recomposition every frame (slow):
Box(Modifier.alpha(animatedAlpha)) { Content() }

// Draw phase only (fast):
Box(Modifier.graphicsLayer { alpha = animatedAlpha }) { Content() }

6. Compose vs View Performance

ScenarioFasterWhy
Complex hierarchies (many nested layouts) Compose Single-pass measure, lighter LayoutNode vs View, no XML inflation
Animations (alpha, scale, translation) Compose graphicsLayer → RenderNode; can skip recomposition with lambda modifiers
Simple static layouts (few views) Views Less overhead; no slot table, no Recomposer; direct View tree
Very large lists (10k+ items) RecyclerView View recycling; LazyColumn creates/destroys compositions, more memory per item
Memory per node Compose LayoutNode ~10x lighter than View; no ViewGroup overhead for containers

RecyclerView reuses View objects. LazyColumn disposes and recreates SubCompositions for items that scroll off-screen. For huge lists with simple items, RecyclerView's recycling can be more memory-efficient. For complex items or when you need Compose features (state hoisting, animations), LazyColumn is often better. Use key() and contentType to optimize LazyColumn recycling.

// LazyColumn with contentType helps reuse:
LazyColumn {
    items(
        items = list,
        key = { it.id },
        contentType = { it.type }  // Same type → reuse SubComposition
    ) { item ->
        ItemContent(item)
    }
}

7. Debugging Compose Rendering

Layout Inspector — Recomposition Counts

Android Studio Layout Inspector (Compose mode) shows recomposition counts per composable. Enable "Show Recomposition Counts" to find composables that recompose too often. High counts indicate stability issues or state reads that could be deferred to Layout/Draw.

Systrace / Perfetto Sections

// Key Compose trace sections:
// - Compose:recompose     — Composition phase (slot table walk, re-executing scopes)
// - Compose:applyChanges  — Applying node tree changes (Applier inserts/removes)
// - Choreographer#doFrame — Includes both View traversal and Compose composition
// - DrawFrame             — UI thread recording DisplayLists (includes Compose draw)

If Compose:recompose spans multiple VSYNC intervals (>16ms at 60Hz), you have recomposition jank. Fix: reduce scope, fix stability, use lambda modifiers. If Compose:applyChanges is slow, the node tree is large or changes are extensive.

Reading Perfetto Traces

  1. Record a trace with "Compose" tracing category enabled.
  2. Find the main thread; look for purple/orange "Compose" slices.
  3. Correlate with Choreographer frames — composition should fit within one frame.
  4. Check for "Compose:recompose" nested inside "Choreographer#doFrame" — that's the right order.
// Debug recomposition in code (dev only):
@Composable
fun RecompositionCounter(label: String) {
    val count = remember { mutableIntStateOf(0) }
    count.intValue++
    SideEffect { Log.d("Recomposition", "$label: ${count.intValue}") }
}

// Use sparingly — the counter itself causes recomposition when parent recomposes
