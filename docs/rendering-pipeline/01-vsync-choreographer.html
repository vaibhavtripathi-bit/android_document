VSYNC & Choreographer

The heartbeat of Android rendering — how VSYNC drives frame scheduling, Choreographer internals, triple buffering, frame callback ordering, and the 16ms budget

1. The Display Refresh Cycle

A display refreshes at a fixed rate — typically 60Hz (16.67ms per frame), 90Hz (11.11ms), or 120Hz (8.33ms). VSYNC (Vertical Sync) is the hardware signal that marks the start of each refresh cycle. Everything in Android rendering is synchronized to this signal.

2. Choreographer — The Frame Scheduler

Choreographer is the Android class that coordinates all frame work. It receives VSYNC signals and dispatches callbacks in a strict order:

// Choreographer callback types (in execution order):
// 1. CALLBACK_INPUT      — Input events (touch, key)
// 2. CALLBACK_ANIMATION  — Animation frame callbacks (ValueAnimator, etc.)
// 3. CALLBACK_INSETS_ANIMATION — Window inset animations
// 4. CALLBACK_TRAVERSAL  — View traversal (measure, layout, draw)
// 5. CALLBACK_COMMIT     — Post-traversal commit (frame completion bookkeeping)

// How Choreographer works internally (from AOSP):
public final class Choreographer {
    private final FrameDisplayEventReceiver mDisplayEventReceiver;
    private final CallbackQueue[] mCallbackQueues; // One per callback type

    // Called by native code when VSYNC arrives:
    void onVsync(long timestampNanos, long physicalDisplayId, int frame) {
        // Post a message to the UI thread's Looper
        Message msg = Message.obtain(mHandler, this::doFrame);
        msg.setAsynchronous(true); // Bypass sync barriers
        mHandler.sendMessageAtTime(msg, timestampNanos / TimeUtils.NANOS_PER_MS);
    }

    void doFrame(long frameTimeNanos) {
        // Execute callbacks in strict order:
        doCallbacks(CALLBACK_INPUT, frameTimeNanos);
        doCallbacks(CALLBACK_ANIMATION, frameTimeNanos);
        doCallbacks(CALLBACK_INSETS_ANIMATION, frameTimeNanos);
        doCallbacks(CALLBACK_TRAVERSAL, frameTimeNanos);
        doCallbacks(CALLBACK_COMMIT, frameTimeNanos);
    }
}
Staff-Level Insight — Why Callback Order Matters

The callback order is critical: input is processed first so touch events update state before animations run. Animations update values before traversal so the layout reflects the latest animated positions. Traversal (measure/layout/draw) happens last so it captures all state changes from input and animation. If you use View.post() to schedule work, it runs after the frame — potentially causing a one-frame delay. Use Choreographer.postFrameCallback() for frame-synchronized work, or View.postOnAnimation() which internally uses the animation callback type.

3. VSYNC Internals — From Hardware to App

// The VSYNC signal path:
// 1. Display hardware generates VSYNC interrupt
// 2. HWC (Hardware Composer) HAL receives the interrupt
// 3. SurfaceFlinger's EventThread distributes VSYNC to registered listeners
// 4. App process receives VSYNC via FrameDisplayEventReceiver (BitTube socket)
// 5. Choreographer.onVsync() is called on the UI thread
// 6. doFrame() dispatches all frame callbacks

// VSYNC offset:
// Apps don't receive VSYNC at the exact hardware time.
// There's a configurable offset (typically 1-4ms) so the app starts
// work slightly after VSYNC, giving SurfaceFlinger time to composite
// the previous frame before the app's new frame arrives.

// VSYNC phases (Android 11+):
// Phase 1: App VSYNC — triggers Choreographer (app produces frame)
// Phase 2: SF VSYNC — triggers SurfaceFlinger (composites layers)
// The offset between these phases is tuned per device.

4. Triple Buffering

Android uses triple buffering to keep the pipeline full even when a frame takes longer than 16ms:

BufferingBuffersBehavior
Single 1 App writes to the same buffer the display reads. Causes tearing.
Double 2 (front + back) App writes to back buffer while display reads front. If app misses deadline, it must wait a full frame before starting the next — causes visible stutter.
Triple 3 If the app misses one deadline, it can immediately start on the third buffer instead of waiting. Reduces stutter at the cost of one extra frame of latency. 
// Triple buffering timeline when Frame N is slow:
//
// VSYNC 0          VSYNC 1          VSYNC 2          VSYNC 3
//   |                |                |                |
//   |-- Frame N (UI) -------|         |                |
//   |                |-- Frame N (RT) ---|              |
//   |                |-- Frame N+1 (UI) -|              |
//   |                |                |-- Frame N+1 (RT)|
//   |                |                |                |
// Display: N-2      Display: N-1     Display: N       Display: N+1
//
// Without triple buffering, Frame N+1 couldn't start until Frame N's
// RenderThread work finished — wasting an entire VSYNC period.
// With triple buffering, Frame N+1 starts immediately on VSYNC 1.
Staff-Level Insight — Latency vs Throughput

Triple buffering is a throughput optimization at the cost of latency. With triple buffering, there's always one extra frame of latency between input and display (the frame in the third buffer). This is typically 16ms at 60Hz — imperceptible for most UI but noticeable for gaming. Android 12+ introduced BLAST (Buffer Layer Aggregation and Synchronization Transactions) which allows more intelligent buffer management, reducing latency when the pipeline isn't under pressure. In interviews, the key insight is: triple buffering prevents cascading frame drops (one slow frame doesn't cause two dropped frames) but adds baseline latency.

6. Frame Scheduling APIs

// Choreographer.postFrameCallback — run on next VSYNC
Choreographer.getInstance().postFrameCallback { frameTimeNanos ->
    // Called during CALLBACK_ANIMATION phase
    // frameTimeNanos = the VSYNC timestamp (not current time)
    updateAnimation(frameTimeNanos)
}

// View.postOnAnimation — convenience wrapper
view.postOnAnimation {
    // Same as postFrameCallback but scoped to the view
}

// ValueAnimator — uses Choreographer internally
ValueAnimator.ofFloat(0f, 1f).apply {
    duration = 300
    addUpdateListener { animator ->
        view.alpha = animator.animatedValue as Float
    }
    start() // Registers with Choreographer's CALLBACK_ANIMATION
}

// ViewTreeObserver.OnPreDrawListener — just before draw
view.viewTreeObserver.addOnPreDrawListener {
    // Called after measure/layout but before draw
    // Return false to cancel the draw (skip this frame)
    true
}

7. Variable Refresh Rate (VRR)

// Modern devices support variable refresh rates (60/90/120Hz).
// Android dynamically adjusts based on content:

// Checking current refresh rate:
val display = (context.getSystemService(Context.DISPLAY_SERVICE) as DisplayManager)
    .getDisplay(Display.DEFAULT_DISPLAY)
val refreshRate = display.refreshRate // e.g., 120.0

// Requesting a specific refresh rate (for games/video):
window.attributes = window.attributes.apply {
    preferredDisplayModeId = display.supportedModes
        .filter { it.refreshRate == 120f }
        .firstOrNull()?.modeId ?: 0
}

// Android 11+ Frame Rate API:
surface.setFrameRate(60f, Surface.FRAME_RATE_COMPATIBILITY_DEFAULT)
// Tells the system this surface only needs 60Hz
// System can lower refresh rate to save battery
Staff-Level Insight — VRR and Frame Pacing

With VRR displays, the frame budget changes dynamically. At 120Hz, you have only 8.33ms per frame. Android's FrameRateCategory (Android 15+) lets apps declare their frame rate needs per-surface. Static content (text, images) can run at 60Hz while animations run at 120Hz. The system aggregates all surface requests and picks the optimal refresh rate. For L6: understand that the Choreographer's VSYNC timing adapts to the current refresh rate — doFrame() is called at the display's actual interval, not a fixed 16ms.
