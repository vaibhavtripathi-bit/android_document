Common Problems & Solutions

The real-world pitfalls of modularization and Gradle at scale — circular dependencies, resource conflicts, manifest merging, cross-module navigation, DI wiring, R8 crashes, and the staff-level corner cases that separate senior from staff engineers

1. Circular Dependencies

Gradle enforces a strict DAG (directed acyclic graph) for project dependencies. If :feature:home depends on :feature:profile and :feature:profile depends on :feature:home, the build fails immediately with "Circular dependency between the following tasks".

Why It Happens

  • :feature:home needs a user avatar component that lives in :feature:profile
  • :feature:profile needs to navigate back to the home screen, so it depends on :feature:home
  • Two features share a data model that ended up in one of them instead of a shared module

Solution: Dependency Inversion

Extract the shared concept into a :core:* module. Both features depend on the abstraction, neither depends on the other.

  BEFORE (circular):                    AFTER (inverted):

  :feature:home ──────► :feature:profile    :feature:home ──────► :core:user
       ▲                     │                                        ▲
       └─────────────────────┘              :feature:profile ─────────┘

                                            :core:user contains:
                                              - UserAvatar composable
                                              - UserData model
                                              - NavigationTarget interface
// :core:navigation — shared navigation contract
interface NavigationTarget {
    val route: String
}

interface AppNavigator {
    fun navigateTo(target: NavigationTarget)
    fun navigateBack()
}

// :feature:home — declares its own target, uses navigator
object HomeTarget : NavigationTarget {
    override val route = "home"
}

class HomeViewModel @Inject constructor(
    private val navigator: AppNavigator
) : ViewModel() {
    fun goToProfile(userId: String) {
        navigator.navigateTo(ProfileTarget(userId))
    }
}

// :feature:profile — declares its own target, uses navigator
data class ProfileTarget(val userId: String) : NavigationTarget {
    override val route = "profile/$userId"
}

class ProfileViewModel @Inject constructor(
    private val navigator: AppNavigator
) : ViewModel() {
    fun goHome() {
        navigator.navigateTo(HomeTarget)
    }
}
L6 Staff Insight
Circular dependencies are always a symptom of a missing abstraction. The fix is never to hack around Gradle's restriction (e.g., using reflection or runtime class loading). When you encounter a cycle, ask: "What is the shared concept that both modules need?" Extract that concept into a :core:* module. At Google scale, circular dependency detection is automated in CI — a PR that introduces a cycle is rejected before review.

2. Build Time Explosion

Adding modules improves incremental build times (only changed modules recompile), but it increases configuration time. AGP registers hundreds of tasks per Android module — in a 50-module project, the task graph can contain 10,000+ tasks before any compilation starts.

Diagnosis

# Generate a build scan to identify bottlenecks
./gradlew assembleDebug --scan

# Profile configuration phase specifically
./gradlew assembleDebug --profile
# Check build/reports/profile/ for the HTML report

# Measure configuration time in isolation
./gradlew help --configuration-cache-problems=warn

3. Resource Conflicts

When two modules define a resource with the same name (e.g., R.string.app_name), the final merged value depends on dependency order — the last module processed wins. This is silent and unpredictable.

The Problem in Practice

// :feature:home/src/main/res/values/strings.xml
<string name="title">Home</string>

// :feature:profile/src/main/res/values/strings.xml
<string name="title">Profile</string>

// At runtime, R.string.title resolves to... whichever module
// was merged last. Change dependency order → different result.

Solution 1: Resource Prefix

// :feature:home/build.gradle.kts
android {
    namespace = "com.myapp.feature.home"
    resourcePrefix = "home_"
}

// Lint will warn if you create a resource without the prefix:
// "home_title" ✓
// "title" ← Lint warning (but still compiles!)
Important Caveat
resourcePrefix is a Lint check, not a compiler enforcement. Resources without the prefix still compile — the check only produces warnings. You need lintOptions { warningsAsErrors = true } or CI enforcement to make it a hard gate.

Solution 2: Non-Transitive R Classes

# gradle.properties (default since AGP 8.0)
android.nonTransitiveRClass=true

With non-transitive R classes, each module's R class contains only its own resources. To reference a resource from another module, you must use the fully qualified R class:

// Without non-transitive R: any module's resources are in your R class
val text = getString(R.string.some_core_string) // works, but fragile

// With non-transitive R: must qualify cross-module references
val text = getString(com.myapp.core.ui.R.string.some_core_string)
L6 Staff Insight
Non-transitive R classes are the single most impactful change for modular resource management. They eliminate accidental resource shadowing, reduce R class size (each module's R only has its own entries instead of the entire transitive closure), and make resource ownership explicit. If you're on AGP 8+, this is already the default. If you're migrating from AGP 7, enabling it is a high-priority task — it will surface every implicit cross-module resource reference that was silently working by accident.

4. Manifest Merging Issues

Every Android module has its own AndroidManifest.xml. AGP's manifest merger combines them into a single final manifest. When modules declare conflicting attributes, the merge fails — or worse, silently picks the wrong value.

Common Conflicts

ConflictSymptomFix
Duplicate <activity> Merge error: "Multiple entries with same key" Remove duplicate or use tools:node="remove"
Conflicting android:theme Wrong theme applied to activity tools:replace="android:theme" in app manifest
Unwanted <uses-permission> Permission requested that your app doesn't need tools:node="remove" on the permission element
Library with higher minSdkVersion Merge error: "uses-sdk:minSdkVersion X cannot be smaller than Y" tools:overrideLibrary="com.example.lib"

Manifest Merge Directives

<!-- App module AndroidManifest.xml -->
<manifest xmlns:android="http://schemas.android.com/apk/res/android"
    xmlns:tools="http://schemas.android.com/tools">

    <!-- Remove a permission that a library declares but we don't need -->
    <uses-permission
        android:name="android.permission.ACCESS_FINE_LOCATION"
        tools:node="remove" />

    <application>
        <!-- Replace the theme from a library's activity declaration -->
        <activity
            android:name="com.thirdparty.sdk.SdkActivity"
            android:theme="@style/MyApp.Theme"
            tools:replace="android:theme" />

        <!-- Remove an entire service declared by a library -->
        <service
            android:name="com.thirdparty.sdk.UnwantedService"
            tools:node="remove" />
    </application>

    <!-- Allow a library with minSdk 28 in our minSdk 26 app -->
    <uses-sdk tools:overrideLibrary="com.example.highminsdklib" />
</manifest>

Debugging the Merged Manifest

# Inspect the final merged manifest
cat app/build/intermediates/merged_manifests/debug/AndroidManifest.xml

# See detailed merge decisions (which attribute came from where)
./gradlew :app:processDebugManifest --info

# The merge report shows exactly which manifest contributed each element
cat app/build/outputs/logs/manifest-merger-debug-report.txt

5. ProGuard/R8 with Modules

In a modular project, R8 runs only in the :app module during release builds. It sees the entire dependency graph and strips/obfuscates code across all modules. The challenge: each library module must declare which of its classes should be kept.

Consumer ProGuard Rules

// :core:network/build.gradle.kts
android {
    defaultConfig {
        // These rules are automatically applied when any module
        // depends on :core:network
        consumerProguardFiles("consumer-rules.pro")
    }
}

// :app/build.gradle.kts
android {
    buildTypes {
        release {
            isMinifyEnabled = true
            isShrinkResources = true
            proguardFiles(
                getDefaultProguardFile("proguard-android-optimize.txt"),
                "proguard-rules.pro"  // app-level rules
            )
        }
    }
}
# :core:network/consumer-rules.pro
# Keep Retrofit interfaces (they're accessed via reflection)
-keep,allowobfuscation interface * {
    @retrofit2.http.* <methods>;
}

# Keep Moshi JSON adapters
-keep class com.myapp.core.network.model.** { *; }
-keepclassmembers class * {
    @com.squareup.moshi.Json <fields>;
}
The #1 R8 Crash in Modular Projects
A library module exposes a public API (data classes, interfaces) but doesn't declare consumer ProGuard rules. During the app's release build, R8 strips or obfuscates those classes because nothing in the app module directly references them by name — they're accessed through dependency injection or reflection. The debug build works perfectly; the release build crashes with ClassNotFoundException or NoSuchMethodError.

Debugging R8 Issues

# See what R8 removed
./gradlew :app:assembleRelease --info

# Add to proguard-rules.pro for diagnostics:
-printusage build/r8-usage.txt      # what was removed
-printseeds build/r8-seeds.txt      # what was kept (entry points)
-printmapping build/r8-mapping.txt  # obfuscation mapping

# Retrace a stack trace from a release crash
retrace build/r8-mapping.txt stacktrace.txt
L6 Staff Insight
Every library module should own its own consumer ProGuard rules. The rule of thumb: if a class is part of the module's public API or is accessed via reflection/serialization, it needs a keep rule in consumer-rules.pro. At scale, enforce this with a CI check: run assembleRelease + integration tests on every PR. The cost of catching an R8 stripping issue in production is orders of magnitude higher than catching it in CI.

6. Navigation Across Modules

Feature modules can't depend on each other (that creates coupling or cycles). But they need to navigate to each other. This is one of the most common modularization challenges.

Pattern 1: Navigation Abstraction with Hilt

// :core:navigation — defines the contract
interface FeatureNavigation {
    fun registerGraph(navGraphBuilder: NavGraphBuilder, navController: NavController)
}

// :feature:profile — implements the contract
class ProfileNavigation @Inject constructor() : FeatureNavigation {
    override fun registerGraph(
        navGraphBuilder: NavGraphBuilder,
        navController: NavController
    ) {
        navGraphBuilder.composable("profile/{userId}") { backStackEntry ->
            val userId = backStackEntry.arguments?.getString("userId") ?: return@composable
            ProfileScreen(userId = userId, onBack = { navController.popBackStack() })
        }
    }
}

// :feature:profile/di/NavigationModule.kt
@Module
@InstallIn(SingletonComponent::class)
abstract class ProfileNavigationModule {
    @Binds
    @IntoSet
    abstract fun bindProfileNavigation(impl: ProfileNavigation): FeatureNavigation
}

// :app — collects all feature navigations via multibinding
@Composable
fun AppNavHost(
    featureNavigations: Set<@JvmSuppressWildcards FeatureNavigation>,
    navController: NavHostController = rememberNavController()
) {
    NavHost(navController = navController, startDestination = "home") {
        featureNavigations.forEach { feature ->
            feature.registerGraph(this, navController)
        }
    }
}

Pattern 2: Deep Link Navigation

// :feature:profile/AndroidManifest.xml
<activity android:name=".ProfileActivity">
    <intent-filter>
        <action android:name="android.intent.action.VIEW" />
        <category android:name="android.intent.category.DEFAULT" />
        <data android:scheme="myapp" android:host="profile" />
    </intent-filter>
</activity>

// :feature:home — navigates without depending on :feature:profile
class HomeViewModel @Inject constructor(
    private val navigator: AppNavigator
) : ViewModel() {
    fun openProfile(userId: String) {
        navigator.navigateToUri("myapp://profile/$userId")
    }
}

// :core:navigation — navigator implementation
class AppNavigatorImpl @Inject constructor(
    private val context: Context
) : AppNavigator {
    override fun navigateToUri(uri: String) {
        val intent = Intent(Intent.ACTION_VIEW, Uri.parse(uri))
        intent.setPackage(context.packageName)
        context.startActivity(intent)
    }
}

Pattern 3: Type-Safe Routes with Navigation Compose

// :core:navigation — shared route definitions
@Serializable
data class ProfileRoute(val userId: String)

@Serializable
object HomeRoute

@Serializable
object SettingsRoute

// :feature:profile — registers its graph
fun NavGraphBuilder.profileGraph(navController: NavController) {
    composable<ProfileRoute> { backStackEntry ->
        val route = backStackEntry.toRoute<ProfileRoute>()
        ProfileScreen(userId = route.userId)
    }
}

// :feature:home — navigates using the type-safe route
class HomeViewModel @Inject constructor(
    private val navController: NavController
) : ViewModel() {
    fun openProfile(userId: String) {
        navController.navigate(ProfileRoute(userId = userId))
    }
}
L6 Staff Insight
Deep link navigation is the most scalable approach for large apps. It decouples features completely — a feature module doesn't even need to know other features exist. It also supports dynamic feature modules (which can't be depended on at compile time) and makes it trivial to add external deep link support. The trade-off is type safety: URI-based navigation is stringly-typed unless you layer a type-safe wrapper on top. The Hilt multibinding pattern (Pattern 1) is the best middle ground for most teams — type-safe, decoupled, and testable.

7. Dependency Injection Across Modules

Hilt's component hierarchy spans the entire app. When you apply @InstallIn(SingletonComponent::class) in a library module, Hilt's annotation processor in the :app module discovers it and merges it into the app's component. This works automatically — but there are pitfalls.

Cross-Module Interface Binding

// :core:domain — defines the interface
interface UserRepository {
    suspend fun getUser(id: String): User
    suspend fun updateUser(user: User)
}

// :core:data — implements the interface
class UserRepositoryImpl @Inject constructor(
    private val api: UserApi,
    private val dao: UserDao,
    private val dispatcher: CoroutineDispatcher
) : UserRepository {
    override suspend fun getUser(id: String): User =
        withContext(dispatcher) { /* ... */ }
    override suspend fun updateUser(user: User) =
        withContext(dispatcher) { /* ... */ }
}

// :core:data/di/DataModule.kt — binds implementation to interface
@Module
@InstallIn(SingletonComponent::class)
abstract class DataModule {
    @Binds
    abstract fun bindUserRepository(impl: UserRepositoryImpl): UserRepository
}

// :feature:profile — injects the interface (doesn't know about :core:data)
@HiltViewModel
class ProfileViewModel @Inject constructor(
    private val userRepository: UserRepository
) : ViewModel() { /* ... */ }

Common Multi-Module Hilt Pitfalls

PitfallSymptomFix
Missing @HiltAndroidApp No Hilt component generated Annotate your Application class in :app
Missing @AndroidEntryPoint UninitializedPropertyAccessException on @Inject fields Annotate every Activity/Fragment that uses injection
KSP/kapt not applied in library module Hilt modules in that library are silently ignored Apply id("com.google.devtools.ksp") in every module with Hilt annotations
Scoping mismatch New instance created each injection instead of singleton Add @Singleton to the @Provides/@Binds method
Dynamic feature modules Hilt can't discover modules in uninstalled features Use @EntryPoint to access the app's Hilt component

@EntryPoint for Non-Hilt Classes

// Accessing Hilt dependencies from a ContentProvider or dynamic feature
@EntryPoint
@InstallIn(SingletonComponent::class)
interface AnalyticsEntryPoint {
    fun analyticsTracker(): AnalyticsTracker
}

// Usage in a ContentProvider (which Hilt doesn't manage)
class MyContentProvider : ContentProvider() {
    override fun onCreate(): Boolean {
        val entryPoint = EntryPointAccessors.fromApplication(
            context!!.applicationContext,
            AnalyticsEntryPoint::class.java
        )
        val tracker = entryPoint.analyticsTracker()
        return true
    }
}

// :core:testing/src/main/kotlin/TestDispatchers.kt
class TestDispatcherRule(

9. Dynamic Feature Module Issues

Dynamic feature modules (delivered via Play Feature Delivery) have an inverted dependency: the dynamic feature depends on :app, not the other way around. This creates unique challenges.

Normal module:                    Dynamic feature module:
  :app ──────► :feature:home        :app ◄────── :dynamic:premium
  (app depends on feature)          (feature depends on app — INVERTED)

Key Constraints

ConstraintImplication
:app cannot reference dynamic feature classes Must use reflection or SplitInstallManager
Resources unavailable until installed Must check SplitInstallManager.installedModules before accessing
Separate ProGuard configuration Rules must be coordinated with base module to avoid stripping shared code
Can't use standard androidTest Use bundletool for local testing
Hilt can't discover modules in uninstalled features Use @EntryPoint to bridge into the app's DI graph

Installing and Accessing a Dynamic Feature

class DynamicFeatureLoader @Inject constructor(
    private val context: Context,
    private val splitInstallManager: SplitInstallManager
) {
    fun loadPremiumFeature(onSuccess: () -> Unit, onError: (Exception) -> Unit) {
        if ("premium" in splitInstallManager.installedModules) {
            launchPremiumActivity()
            onSuccess()
            return
        }

        val request = SplitInstallRequest.newBuilder()
            .addModule("premium")
            .build()

        splitInstallManager.startInstall(request)
            .addOnSuccessListener { sessionId ->
                splitInstallManager.registerListener { state ->
                    when (state.status()) {
                        SplitInstallSessionStatus.INSTALLED -> {
                            SplitCompat.installActivity(context as Activity)
                            launchPremiumActivity()
                            onSuccess()
                        }
                        SplitInstallSessionStatus.FAILED -> {
                            onError(RuntimeException("Install failed: ${state.errorCode()}"))
                        }
                    }
                }
            }
            .addOnFailureListener { onError(it) }
    }

    private fun launchPremiumActivity() {
        val intent = Intent()
            .setClassName(context.packageName, "com.myapp.premium.PremiumActivity")
        context.startActivity(intent)
    }
}

10. Configuration Leaks

A configuration leak occurs when build logic in one module or the root project unintentionally affects other modules. This breaks module isolation and causes hard-to-debug build failures.

Common Leak Sources

// ROOT build.gradle.kts — LEAKY PATTERNS

// BAD: applies a plugin to ALL subprojects, even those that don't need it
subprojects {
    apply(plugin = "org.jetbrains.kotlin.android")  // pure JVM modules don't want this
}

// BAD: configures ALL projects — couples every module to this block
allprojects {
    repositories {
        google()
        mavenCentral()
    }
    tasks.withType<org.jetbrains.kotlin.gradle.tasks.KotlinCompile>().configureEach {
        compilerOptions.jvmTarget.set(JvmTarget.JVM_17)
    }
}

// BAD: buildscript classpath leaks to all subprojects
buildscript {
    dependencies {
        classpath("com.android.tools.build:gradle:8.7.0")
    }
}

The Fix: Isolation at Every Level

// GOOD: Use plugins {} block (module-level, no leaking)
// Each module declares exactly what it needs
plugins {
    id("myapp.android.library")  // convention plugin — scoped to this module
}

// GOOD: pluginManagement in settings.gradle.kts (centralized, no leaking)
pluginManagement {
    repositories {
        google()
        mavenCentral()
        gradlePluginPortal()
    }
}

// GOOD: repositoriesMode prevents per-project repo declarations
dependencyResolutionManagement {
    repositoriesMode = RepositoriesMode.FAIL_ON_PROJECT_REPOS
    repositories {
        google()
        mavenCentral()
    }
}

Debugging Configuration Leaks

# If a build works without cache but fails with cache → likely a leak
./gradlew assembleDebug --no-build-cache

# Compare task inputs between two builds
./gradlew assembleDebug --scan  # run twice, compare scans

# Check which plugins are applied to a module (unexpected ones = leak)
./gradlew :core:domain:buildEnvironment

11. Staff-Level Corner Cases

Gradle Module ≠ Android Module

Not every Gradle module needs the Android plugin. A :core:domain module that contains only Kotlin interfaces and data classes should be a pure Kotlin/JVM module:

// :core:domain/build.gradle.kts
plugins {
    id("myapp.kotlin.library")  // uses kotlin("jvm"), NOT com.android.library
}

dependencies {
    implementation(libs.kotlinx.coroutines.core)
    // NO Android dependencies — this module is pure Kotlin
}

Benefits of pure Kotlin modules:

  • Faster compilation — no AGP overhead (no AAPT2, no resource merging, no manifest processing)
  • JVM-testable — tests run on the JVM without Robolectric or an emulator
  • Reusable — can be used in Kotlin Multiplatform, backend services, or CLI tools
  • Fewer tasks — a JVM module registers ~20 tasks vs. ~200+ for an Android library module

includeBuild for Local Library Development

// settings.gradle.kts — substitute a published library with a local checkout
includeBuild("../my-shared-library") {
    dependencySubstitution {
        substitute(module("com.mycompany:shared-library"))
            .using(project(":"))
    }
}

// Now any module that depends on "com.mycompany:shared-library:1.2.3"
// will use the local source instead — no need to publish a SNAPSHOT

dependencySubstitution for A/B Testing Implementations

// settings.gradle.kts — swap implementations at build time
if (providers.gradleProperty("useNewNetworkStack").getOrElse("false").toBoolean()) {
    includeBuild("../network-v2") {
        dependencySubstitution {
            substitute(module("com.myapp:network"))
                .using(project(":network-v2"))
        }
    }
}

// Build with: ./gradlew assembleDebug -PuseNewNetworkStack=true

Gradle's --continue Flag

# Default: Gradle stops at the first failure
./gradlew test
# :core:data:test FAILED → build stops, :feature:home:test never runs

# With --continue: build continues past failures
./gradlew test --continue
# :core:data:test FAILED → continues → :feature:home:test PASSED
# Reports ALL failures at the end — maximum test coverage per CI run
L6 Staff Insight
The most underrated modularization optimization is making :core:domain a pure Kotlin module. Teams reflexively apply com.android.library to every module, but domain logic (interfaces, use cases, data classes) has zero Android dependencies. Switching to kotlin("jvm") eliminates AGP's task overhead (~200 tasks → ~20), removes the need for Robolectric in tests, and makes the module eligible for Kotlin Multiplatform. In a 50-module project, converting even 10 modules from Android library to JVM library can cut configuration time by 20–30%.
