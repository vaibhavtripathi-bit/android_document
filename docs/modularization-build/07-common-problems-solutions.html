Common Problems & Solutions

The 10 most painful modularization and Gradle issues — circular dependencies, build time explosions, resource conflicts, manifest merging, R8, cross-module navigation, DI wiring, testing, dynamic features, and IDE performance

1. Circular Dependencies

The Problem

:feature:home depends on :feature:profile (to show a user card), and :feature:profile depends on :feature:home (to link back to the feed). Gradle rejects this at configuration time:

Circular dependency between the following tasks:
:feature:home:compileDebugKotlin
\--- :feature:profile:compileDebugKotlin
     \--- :feature:home:compileDebugKotlin (*)

This isn't a Gradle limitation — it's a fundamental constraint of directed acyclic graphs. The module dependency graph must be a DAG. Any cycle means the build system can't determine compilation order.

Solution A: Extract Shared Code into a Lower-Level Module

If both features need the same data class, UI component, or utility, it belongs in a shared module:

BEFORE (circular):                  AFTER (DAG):

:feature:home ←→ :feature:profile   :feature:home    :feature:profile
                                          ↓                ↓
                                       :core:shared-ui
                                    (UserCard, FeedLink)
// :core:shared-ui/build.gradle.kts
plugins {
    id("com.android.library")
    id("org.jetbrains.kotlin.android")
}

// :feature:home/build.gradle.kts
dependencies {
    implementation(project(":core:shared-ui"))
}

// :feature:profile/build.gradle.kts
dependencies {
    implementation(project(":core:shared-ui"))
}

Solution B: Interface Extraction + DI

When the dependency is behavioral (navigation, callbacks), define interfaces in a low-level module and implement them in each feature:

// :core:navigation — no feature dependencies
interface HomeNavigator {
    fun navigateToFeed(context: Context)
}

interface ProfileNavigator {
    fun navigateToProfile(context: Context, userId: String)
}

// :feature:home — depends on :core:navigation, NOT :feature:profile
class HomeScreen(private val profileNavigator: ProfileNavigator) {
    fun onUserCardClicked(userId: String) {
        profileNavigator.navigateToProfile(context, userId)
    }
}

// :feature:profile — depends on :core:navigation, NOT :feature:home
class ProfileScreen(private val homeNavigator: HomeNavigator) {
    fun onFeedLinkClicked() {
        homeNavigator.navigateToFeed(context)
    }
}

// :app — the composition root, depends on everything
// Hilt wires the implementations
@Module
@InstallIn(SingletonComponent::class)
abstract class NavigationModule {
    @Binds abstract fun bindProfileNavigator(impl: ProfileNavigatorImpl): ProfileNavigator
    @Binds abstract fun bindHomeNavigator(impl: HomeNavigatorImpl): HomeNavigator
}

Solution C: Event Bus / Navigation Abstraction

For loosely coupled communication, use a shared event bus or navigation command pattern. Each feature emits navigation events; the :app module observes and routes them:

// :core:navigation
sealed interface NavigationCommand {
    data class ToProfile(val userId: String) : NavigationCommand
    data object ToHomeFeed : NavigationCommand
}

class NavigationRouter {
    private val _commands = MutableSharedFlow<NavigationCommand>()
    val commands: SharedFlow<NavigationCommand> = _commands.asSharedFlow()

    suspend fun navigate(command: NavigationCommand) {
        _commands.emit(command)
    }
}
Staff Insight
Circular dependencies are always a design smell — they indicate that module boundaries are drawn incorrectly. The fix is never to "work around" the cycle but to extract the shared concern into a lower-level module. If you find yourself creating many shared modules to break cycles, your feature boundaries are too fine-grained. In an interview, immediately identifying a circular dependency as a boundary problem (not a tooling problem) demonstrates L6 thinking.

2. Build Time Explosion After Modularization

The Problem

You split a monolith into 20 modules expecting faster builds. Instead, build time got worse. This is the most common modularization disappointment.

Root Causes

CauseImpactHow to Detect
Configuration phase overheadEach module adds ~100–500ms to configuration--scan → Configuration tab
Excessive api dependenciesChange in module A recompiles B, C, D, E…--scan → Task execution timeline
allprojects {} / subprojects {}Forces configuration of ALL modules even if unused--scan → Configuration tab shows all modules
buildSrc changesAny change invalidates entire build cacheModify buildSrc → watch full recompilation
Missing parallelismModules compile sequentially--scan → Timeline shows serial execution

Solutions

A. Replace allprojects/subprojects with Convention Plugins

// BAD — configures every module even if not needed
// settings.gradle.kts or root build.gradle.kts
allprojects {
    repositories { mavenCentral() }
    tasks.withType<KotlinCompile>().configureEach {
        kotlinOptions.jvmTarget = "17"
    }
}

// GOOD — each module opts in via plugin
// build-logic/convention/src/main/kotlin/AndroidLibraryConventionPlugin.kt
class AndroidLibraryConventionPlugin : Plugin<Project> {
    override fun apply(target: Project) {
        with(target) {
            pluginManager.apply("com.android.library")
            pluginManager.apply("org.jetbrains.kotlin.android")
            extensions.configure<LibraryExtension> {
                compileSdk = 35
                defaultConfig.minSdk = 26
                compileOptions {
                    sourceCompatibility = JavaVersion.VERSION_17
                    targetCompatibility = JavaVersion.VERSION_17
                }
            }
        }
    }
}

B. Audit api vs implementation

// BAD — leaks Retrofit into every consumer
api(libs.retrofit)

// GOOD — Retrofit is an internal detail
implementation(libs.retrofit)

// Rule of thumb: use api ONLY when the dependency's types
// appear in your module's PUBLIC API (function signatures,
// return types, public class hierarchies)

C. Enable Configuration Cache

# gradle.properties
org.gradle.configuration-cache=true
org.gradle.configuration-cache.problems=warn

D. Migrate buildSrc to build-logic

# buildSrc change → EVERYTHING recompiles
# build-logic included build → only affected plugins recompile

# settings.gradle.kts
pluginManagement {
    includeBuild("build-logic")
}

E. Diagnostic Command

# Generate a build scan to identify bottlenecks
./gradlew assembleDebug --scan

# Check configuration time vs execution time
# If configuration > 10s with 20 modules, you have a problem
# If execution shows serial task chains, dependency graph is too linear
Staff Insight
The #1 mistake teams make: they modularize for "clean architecture" without profiling first. Modularization is a build performance tool — if your build isn't faster afterward, you did it wrong. Always run --scan before and after modularization. The ideal module graph is wide and shallow (many independent leaf modules), not deep and linear (A → B → C → D). A linear graph serializes compilation; a wide graph maximizes parallelism.

3. Resource Conflicts Across Modules

The Problem

Two modules define R.string.app_name or R.color.primary. During resource merging, the last one wins based on dependency order. This is silent — no build error, just wrong resources at runtime.

// :feature:home/src/main/res/values/strings.xml
<string name="title">Home</string>

// :feature:profile/src/main/res/values/strings.xml
<string name="title">Profile</string>

// At runtime, which "title" do you get? Depends on merge order.
// This is a silent, hard-to-debug bug.

Solution A: Resource Prefix Convention

// :feature:home/build.gradle.kts
android {
    resourcePrefix = "home_"
    // Lint will warn if any resource doesn't start with "home_"
    // Does NOT enforce at compile time — it's a lint check only
}

// Resources become: home_title, home_bg_color, home_ic_arrow
// Prevents collisions by convention

Solution B: Non-Transitive R Classes (The Real Fix)

# gradle.properties (default true in AGP 8+)
android.nonTransitiveRClass=true

With non-transitive R classes, each module's R class contains only its own resources. Module A cannot accidentally reference Module B's resources via R.string.xxx — it must use the fully qualified com.example.moduleb.R.string.xxx.

PropertyTransitive R (legacy)Non-Transitive R (AGP 8+)
R class contentsOwn + all transitive dependencies' resourcesOwn resources only
R class sizeHuge (grows with dependency count)Small (only own resources)
Accidental resource accessEasy — any resource accessible via RImpossible without explicit import
Build speedSlower (regenerate massive R class)Faster (small R classes, less work)
Migration effort—Fix all unqualified cross-module references
Migration Pain
Enabling nonTransitiveRClass on a large codebase breaks hundreds of resource references. Use Android Studio's refactoring tools or the resource-shrinker migration helper. Fix them module by module — don't try to do it all at once.

4. Manifest Merging Issues

The Problem

Every Android module has a AndroidManifest.xml. AGP merges them all into a single manifest for the APK. When two modules declare conflicting attributes on the same element, the merge fails:

Manifest merger failed:
  Attribute application@theme value=(@style/LibTheme) from [:library]
  is also present at [:app] value=(@style/AppTheme).
  Suggestion: add 'tools:replace="android:theme"' to <application>

Manifest Merge Markers

MarkerEffectUse Case
tools:replace="android:theme"Replace a specific attribute from a lower-priority manifestOverride library's theme with app's theme
tools:remove="android:allowBackup"Remove an attribute entirelyStrip unwanted attributes from a library
tools:node="remove"Remove an entire XML elementRemove an activity or permission from a library
tools:node="replace"Replace an entire element (all attributes)Completely override a library's activity declaration
tools:overrideLibrary="com.example.lib"Override minSdk conflict from a libraryLibrary requires API 26, your app targets API 24
<!-- :app/src/main/AndroidManifest.xml -->
<manifest xmlns:android="http://schemas.android.com/apk/res/android"
    xmlns:tools="http://schemas.android.com/tools">

    <!-- Override library's theme -->
    <application
        android:theme="@style/AppTheme"
        tools:replace="android:theme">

        <!-- Remove a library-declared activity -->
        <activity
            android:name="com.example.lib.DebugActivity"
            tools:node="remove" />

    </application>

    <!-- Override minSdk from a library -->
    <uses-sdk tools:overrideLibrary="com.example.legacy.lib" />
</manifest>

Debugging Manifest Merges

# Generate the merged manifest with source annotations
./gradlew :app:processDebugManifest

# Output location:
# app/build/intermediates/merged_manifests/debug/AndroidManifest.xml
#
# The merged manifest contains comments showing where each element
# and attribute came from — invaluable for debugging conflicts
Tip
Android Studio has a built-in Merged Manifest viewer: open any AndroidManifest.xml and click the "Merged Manifest" tab at the bottom. It color-codes elements by source module and highlights conflicts.

5. ProGuard/R8 with Modules

The Problem

R8 (the code shrinker/optimizer) runs only on the :app module. It processes the entire merged DEX as a single unit. Library modules don't run R8 independently — their code is shrunk as part of the app's R8 pass.

This creates a critical gap: if a library module uses reflection, annotation processing, or JNI but doesn't declare keep rules, R8 removes the reflected classes. The build succeeds, but the release APK crashes at runtime.

Consumer ProGuard Rules

// :core:network/build.gradle.kts
android {
    defaultConfig {
        // Rules automatically included when this module is consumed
        consumerProguardFiles("consumer-rules.pro")
    }
}

// :core:network/consumer-rules.pro
# Keep Retrofit interfaces (used via reflection)
-keep,allowobfuscation interface * {
    @retrofit2.http.* <methods>;
}

# Keep Moshi JSON adapters
-keep class com.example.network.model.** { *; }
-keepclassmembers class * {
    @com.squareup.moshi.Json <fields>;
}
// :app/build.gradle.kts
android {
    buildTypes {
        release {
            isMinifyEnabled = true
            proguardFiles(
                getDefaultProguardFile("proguard-android-optimize.txt"),
                "proguard-rules.pro"
            )
            // consumer-rules.pro from ALL library dependencies
            // are automatically merged — no manual action needed
        }
    }
}

R8 Debugging Outputs

FilePurposeFlag
mapping.txtObfuscation map (original → obfuscated names)-printmapping
usage.txtClasses/members removed by R8-printusage
seeds.txtClasses/members kept by keep rules-printseeds
configuration.txtAll merged ProGuard rules (from all sources)-printconfiguration
// proguard-rules.pro (app module, debug only)
-printusage build/outputs/r8/usage.txt
-printseeds build/outputs/r8/seeds.txt
-printconfiguration build/outputs/r8/configuration.txt
Common Crash Pattern
Debug build works perfectly. Release build crashes with ClassNotFoundException or NoSuchMethodError. The cause is almost always a missing keep rule in a library module. Always test the release build after adding or modifying modules. Run ./gradlew :app:assembleRelease and exercise every code path that uses reflection, serialization, or JNI.
Staff Insight
Every library module must ship consumer-rules.pro — missing rules are the #1 cause of release-only crashes in modular projects. At Google scale, teams enforce this with a CI check: if a module uses Retrofit, Moshi, Room, or any reflection-based library, the PR must include corresponding keep rules. The -printconfiguration output is your best friend — it shows the merged rules from all sources, so you can verify that every library's rules are actually being applied.

6. Navigation Across Modules

The Problem

:feature:home needs to navigate to :feature:profile, but adding a direct dependency creates coupling (or a cycle). Cross-module navigation is one of the most debated architectural problems in multi-module Android apps.

Solution 1: Deep Links

Each feature registers deep link URIs. Navigation uses URIs instead of direct class references:

// :feature:profile/src/main/AndroidManifest.xml
<activity android:name=".ProfileActivity">
    <intent-filter>
        <action android:name="android.intent.action.VIEW" />
        <category android:name="android.intent.category.DEFAULT" />
        <data android:scheme="myapp" android:host="profile" />
    </intent-filter>
</activity>

// :feature:home — navigates via URI, no dependency on :feature:profile
class HomeScreen @Inject constructor(
    private val context: Context
) {
    fun navigateToProfile(userId: String) {
        val uri = "myapp://profile?userId=$userId".toUri()
        context.startActivity(Intent(Intent.ACTION_VIEW, uri))
    }
}

Solution 2: Navigation Compose with Shared Routes

// :core:navigation — shared route definitions
@Serializable data class ProfileRoute(val userId: String)
@Serializable data object HomeRoute
@Serializable data object SettingsRoute

// :feature:home — uses routes, not direct references
fun NavGraphBuilder.homeScreen(
    onNavigateToProfile: (String) -> Unit
) {
    composable<HomeRoute> {
        HomeScreen(onNavigateToProfile = onNavigateToProfile)
    }
}

// :feature:profile
fun NavGraphBuilder.profileScreen() {
    composable<ProfileRoute> { backStackEntry ->
        val route = backStackEntry.toRoute<ProfileRoute>()
        ProfileScreen(userId = route.userId)
    }
}

// :app — wires everything together
@Composable
fun AppNavHost(navController: NavHostController) {
    NavHost(navController, startDestination = HomeRoute) {
        homeScreen(
            onNavigateToProfile = { userId ->
                navController.navigate(ProfileRoute(userId))
            }
        )
        profileScreen()
    }
}

Solution 3: Interface-Based Navigation with Hilt

// :core:navigation — defines the contract
interface AppNavigator {
    fun navigateToProfile(context: Context, userId: String)
    fun navigateToHome(context: Context)
    fun navigateToSettings(context: Context)
}

// :app — implements the contract (knows all features)
class AppNavigatorImpl @Inject constructor() : AppNavigator {
    override fun navigateToProfile(context: Context, userId: String) {
        context.startActivity(
            Intent(context, ProfileActivity::class.java)
                .putExtra("userId", userId)
        )
    }
    override fun navigateToHome(context: Context) {
        context.startActivity(Intent(context, HomeActivity::class.java))
    }
    override fun navigateToSettings(context: Context) {
        context.startActivity(Intent(context, SettingsActivity::class.java))
    }
}

// :app — Hilt binding
@Module
@InstallIn(SingletonComponent::class)
abstract class NavigatorModule {
    @Binds abstract fun bindNavigator(impl: AppNavigatorImpl): AppNavigator
}

// :feature:home — injects the interface, no feature dependencies
class HomeViewModel @HiltViewModel @Inject constructor(
    private val navigator: AppNavigator
) : ViewModel() {
    fun onProfileClicked(userId: String) {
        navigator.navigateToProfile(context, userId)
    }
}
ApproachProsCons
Deep linksNo shared module needed, works with ActivitiesString-based (no compile-time safety), parsing overhead
Compose routesType-safe, first-party supportCompose-only, shared module for route definitions
Interface + DIFully type-safe, testable (mock navigator)More boilerplate, :app becomes a routing hub

7. Dependency Injection Across Modules

The Problem

How does Hilt wire dependencies when @Provides is in :core:data, @Inject is in :feature:home, and the component is generated in :app?

How Hilt Spans Modules

Hilt's component hierarchy is app-wide. Every @Module with @InstallIn(SingletonComponent::class) is collected at compile time and merged into a single generated component in the :app module. The module boundary is invisible to Hilt's DI graph.

:core:network                :core:data                :feature:home
┌──────────────────┐   ┌──────────────────┐   ┌──────────────────────┐
│ @Module           │   │ @Module           │   │ @HiltViewModel       │
│ @InstallIn(       │   │ @InstallIn(       │   │ class HomeViewModel  │
│   SingletonComp)  │   │   SingletonComp)  │   │ @Inject constructor( │
│                   │   │                   │   │   repo: UserRepo     │
│ @Provides         │   │ @Binds            │   │ )                    │
│ fun api(): Api    │   │ fun repo(impl):   │   └──────────────────────┘
│                   │   │   UserRepo        │
└──────────────────┘   └──────────────────┘
         │                      │                        │
         └──────────────────────┼────────────────────────┘
                                ↓
                    :app (composition root)
              ┌──────────────────────────────┐
              │ Hilt generates:              │
              │ DaggerApp_HiltComponents     │
              │   .SingletonC                │
              │                              │
              │ Merges ALL @InstallIn modules │
              │ from ALL project modules     │
              └──────────────────────────────┘

Complete Example: 3-Module Hilt Setup

// :core:data/build.gradle.kts
plugins {
    id("com.android.library")
    id("dagger.hilt.android.plugin")
    id("com.google.devtools.ksp")
}
dependencies {
    implementation(libs.hilt.android)
    ksp(libs.hilt.compiler)
    implementation(project(":core:network"))
}

// :core:data — provides UserRepository
interface UserRepository {
    suspend fun getUser(id: String): User
}

class UserRepositoryImpl @Inject constructor(
    private val api: UserApi
) : UserRepository {
    override suspend fun getUser(id: String): User = api.fetchUser(id)
}

@Module
@InstallIn(SingletonComponent::class)
abstract class DataModule {
    @Binds abstract fun bindUserRepo(impl: UserRepositoryImpl): UserRepository
}
// :feature:home — consumes UserRepository
// Does NOT depend on :core:network — only on :core:data
@HiltViewModel
class HomeViewModel @Inject constructor(
    private val userRepo: UserRepository
) : ViewModel() {

    private val _user = MutableStateFlow<User?>(null)
    val user: StateFlow<User?> = _user.asStateFlow()

    fun loadUser(id: String) {
        viewModelScope.launch {
            _user.value = userRepo.getUser(id)
        }
    }
}
// :app — the composition root
@HiltAndroidApp
class MyApplication : Application()

// That's it. Hilt scans all modules for @InstallIn and generates
// the full component tree here. No manual wiring needed.

Common Pitfalls

  • Missing @InstallIn — the module is ignored silently in older Hilt versions; newer versions fail at compile time with "X cannot be provided without an @Provides-annotated method"
  • Wrong component scope — @InstallIn(ViewModelComponent::class) in a library module means the binding is only available to ViewModels, not to other classes
  • @EntryPoint for non-Hilt classes — ContentProviders, WorkManager workers (pre-Hilt 2.40), and custom Application subclasses need @EntryPoint to access the Hilt graph
  • Duplicate bindings — two modules both provide the same type without @Qualifier → compile error

8. Testing Across Modules

Module-Level Unit Tests

Each module has its own test/ source set. Tests run in isolation — a module's tests can only access classes visible to that module (its own code + implementation/api dependencies).

// :core:data/src/test/kotlin/UserRepositoryTest.kt
class UserRepositoryTest {
    private val fakeApi = FakeUserApi()
    private val repo = UserRepositoryImpl(fakeApi)

    @Test
    fun `getUser returns mapped domain object`() = runTest {
        fakeApi.stubUser("123", ApiUser("123", "Alice"))
        val user = repo.getUser("123")
        assertThat(user.name).isEqualTo("Alice")
    }
}

Shared Test Utilities: :core:testing

// :core:testing/build.gradle.kts
plugins {
    id("org.jetbrains.kotlin.jvm")
}
dependencies {
    implementation(libs.coroutines.test)
    implementation(libs.truth)
    implementation(libs.turbine)
}

// :core:testing — fakes, builders, rules shared across modules
class FakeUserRepository : UserRepository {
    private val users = mutableMapOf<String, User>()

    fun stubUser(id: String, user: User) { users[id] = user }

    override suspend fun getUser(id: String): User =
        users[id] ?: throw NoSuchElementException("No user $id")
}

// Consumed in feature modules:
// :feature:home/build.gradle.kts
dependencies {
    testImplementation(project(":core:testing"))
}

testFixtures: Share Test Code Without a Separate Module

// :core:data/build.gradle.kts
android {
    testFixtures { enable = true }
}

// :core:data/src/testFixtures/kotlin/FakeUserRepository.kt
class FakeUserRepository : UserRepository { /* ... */ }

// Consumer:
// :feature:home/build.gradle.kts
dependencies {
    testImplementation(testFixtures(project(":core:data")))
}

Integration Tests

LevelLocationScope
Unit tests:module/src/test/Single module, JVM, fast
Module integration:module/src/androidTest/Single module on device/emulator
App integration:app/src/androidTest/Full app, all modules wired
// Module-level integration test with Hilt
// :feature:home/src/androidTest/kotlin/HomeScreenTest.kt
@HiltAndroidTest
@RunWith(AndroidJUnit4::class)
class HomeScreenTest {

    @get:Rule val hiltRule = HiltAndroidRule(this)

    @BindValue
    val fakeRepo: UserRepository = FakeUserRepository().apply {
        stubUser("1", User("1", "Test User"))
    }

    @Test
    fun displaysUserName() {
        launchFragmentInHiltContainer<HomeFragment>()
        onView(withText("Test User")).check(matches(isDisplayed()))
    }
}
Staff Insight
The biggest testing win from modularization is parallel test execution. Gradle can run each module's test suite independently and in parallel. A monolith with 5,000 tests runs them serially in one JVM. Split across 20 modules, Gradle spawns 20 test JVMs (up to org.gradle.workers.max) and finishes in a fraction of the time. This alone justifies modularization for large codebases. Enable with org.gradle.parallel=true and ensure modules don't share mutable test state (files, databases, ports).

9. Dynamic Feature Module Issues

The Inverted Dependency Problem

Dynamic feature modules depend on :app, not the other way around. This is the opposite of normal module dependencies and creates unique challenges:

Normal modules:              Dynamic features:

:app                         :app
 ├── :feature:home            ↑       ↑
 ├── :feature:profile    :dynamic:ar  :dynamic:scanner
 └── :core:data          (depends     (depends
                          on :app)     on :app)

Accessing Code in Dynamic Features

Since :app can't depend on a dynamic feature, it can't reference its classes directly. Access patterns:

// Option A: Reflection (fragile)
val clazz = Class.forName("com.example.ar.ArActivity")
startActivity(Intent(this, clazz))

// Option B: SplitInstallManager + ServiceLoader (recommended)
// 1. Define interface in :app or :core module
interface FeatureEntryPoint {
    fun launch(context: Context)
}

// 2. Implement in dynamic feature
// :dynamic:ar/src/main/resources/META-INF/services/
//   com.example.app.FeatureEntryPoint
class ArEntryPoint : FeatureEntryPoint {
    override fun launch(context: Context) {
        context.startActivity(Intent(context, ArActivity::class.java))
    }
}

// 3. Load in :app after install
val entryPoints = ServiceLoader.load(FeatureEntryPoint::class.java)
entryPoints.firstOrNull()?.launch(context)
    ?: requestFeatureInstall("ar_feature")

Key Gotchas

  • Resources not available until installed — accessing a dynamic feature's resources before installation throws Resources.NotFoundException. Always check SplitInstallManager.installedModules first.
  • R8 runs separately — each dynamic feature module has its own R8 pass. Keep rules must be self-contained within the feature; they can't reference the base module's rules.
  • Testing limitations — dynamic features can't be tested with standard androidTest. Use bundletool to build a local APK set, or the Play Core testing API for on-device testing.
  • ClassNotFoundException — the most common crash: code tries to access a class in a feature that isn't installed yet. Always guard with install checks.
  • SplitCompat — dynamic feature Activities must call SplitCompat.installActivity(this) in attachBaseContext() to load the feature's code and resources into the running process.
// Proper dynamic feature Activity setup
class ArActivity : AppCompatActivity() {
    override fun attachBaseContext(newBase: Context) {
        super.attachBaseContext(newBase)
        SplitCompat.installActivity(this)
    }
}

10. IDE Performance with Many Modules

The Problem

Android Studio becomes sluggish with 50+ modules. Symptoms: Gradle sync takes minutes, indexing runs constantly, code completion lags, and "Searching for usages" hangs. This is the hidden cost of over-modularization.

Gradle-Level Optimizations

# gradle.properties — essential for multi-module projects
org.gradle.parallel=true
org.gradle.caching=true
org.gradle.configuration-cache=true
org.gradle.daemon=true

# Increase daemon heap for large projects
org.gradle.jvmargs=-Xmx4g -XX:+UseParallelGC

# Offline mode during development (skip dependency resolution)
# Use via command line: ./gradlew assembleDebug --offline

IDE-Level Optimizations

OptimizationHowImpact
Increase IDE heapHelp → Change Memory Settings → 4–8 GBReduces GC pauses during indexing
Disable unused pluginsSettings → Plugins → disable Firebase, Kotlin Notebook, etc.Faster startup and indexing
Disable Power Save Mode autoSettings → Appearance → disable power save on batteryPrevents throttled indexing on laptops
Use file nestingSettings → Editor → File NestingCleaner project view (cosmetic)

Advanced: Module Unloading

For very large projects (100+ modules), conditionally include only the modules you're actively working on:

// settings.gradle.kts
val activeModules: String by settings  // from gradle.properties or -P flag

include(":app")
include(":core:network")
include(":core:data")

// Only include feature modules you're working on
if (activeModules.contains("home")) {
    include(":feature:home")
}
if (activeModules.contains("profile")) {
    include(":feature:profile")
}
// ... other features excluded → faster sync, smaller index

// gradle.properties
activeModules=home,profile
Caveat
Module unloading breaks full-app builds and can cause confusing "unresolved reference" errors for excluded modules. Use it only for focused development. CI always builds with all modules included.

Gradle Sync Optimization

// settings.gradle.kts — use STABLE dependency resolution
dependencyResolutionManagement {
    repositoriesMode.set(RepositoriesMode.FAIL_ON_PROJECT_REPOS)
    repositories {
        google()
        mavenCentral()
    }
}

// This prevents individual modules from declaring their own
// repositories, which slows sync (each module checks each repo)
Staff Insight
IDE performance is the hidden cost of over-modularization. If developers can't navigate, build, and test efficiently, the architectural benefits are negated by lost productivity. The rule of thumb: if Gradle sync takes longer than 30 seconds or a clean build takes longer than 5 minutes, you've either over-modularized or under-optimized. At Google, the Gradle team tracks "developer inner loop time" (edit → build → test) as a key metric. Teams that modularize without investing in build infrastructure (remote cache, convention plugins, CI-optimized builds) see a net negative in developer velocity.

Key Takeaways for L6 Interviews

  • Circular dependencies are design problems, not tooling problems. Always extract shared concerns downward.
  • Modularization without build profiling is guesswork. Run --scan before and after. Measure configuration time, compilation parallelism, and cache hit rates.
  • nonTransitiveRClass is non-negotiable for any project with more than 5 modules. It prevents silent resource conflicts and reduces R class size.
  • Every library module must ship consumer-rules.pro. Missing rules are the #1 cause of release-only crashes.
  • Cross-module navigation has no single "right" answer. Deep links, shared routes, and interface-based navigation all have valid use cases. Pick based on your app's architecture (Activities vs Compose, single-activity vs multi-activity).
  • Hilt's component tree spans modules transparently. The :app module is the composition root where all @InstallIn modules are merged.
  • Parallel test execution is modularization's biggest testing win. Design modules so their tests are independent and share no mutable state.
  • IDE performance is a real constraint. Over-modularization with poor build infrastructure hurts developer velocity more than it helps architecture.
