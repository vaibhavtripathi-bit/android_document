Dependency Management

Version catalogs, BOMs, resolution strategies, dependency locking, analysis tooling, and avoiding dependency hell in multi-module Android projects

1. Version Catalogs (libs.versions.toml)

Before version catalogs, teams managed dependency versions through ext {} blocks in root build.gradle, buildSrc constants, or convention plugins. Each approach had drawbacks: ext blocks lacked type safety, buildSrc invalidated the entire build cache on any change, and convention plugins required boilerplate. Gradle 7.0 introduced version catalogs as a first-class solution.

The TOML File Structure

Consuming the Catalog

// app/build.gradle.kts
plugins {
    alias(libs.plugins.android.application)
    alias(libs.plugins.kotlin.android)
    alias(libs.plugins.kotlin.compose)
    alias(libs.plugins.hilt)
    alias(libs.plugins.ksp)
}

dependencies {
    // BOM — versions for all Compose libraries resolved from here
    implementation(platform(libs.compose.bom))
    implementation(libs.bundles.compose)
    debugImplementation(libs.bundles.compose.debug)

    // OkHttp BOM
    implementation(platform(libs.okhttp.bom))
    implementation(libs.bundles.networking)

    // Lifecycle bundle
    implementation(libs.bundles.lifecycle)

    // Navigation
    implementation(libs.navigation.compose)

    // Hilt
    implementation(libs.hilt.android)
    ksp(libs.hilt.compiler)
    implementation(libs.hilt.navigation.compose)

    // Room
    implementation(libs.bundles.room)
    ksp(libs.room.compiler)

    // Coroutines
    implementation(libs.coroutines.core)
    implementation(libs.coroutines.android)

    // Testing
    testImplementation(libs.junit)
    testImplementation(libs.truth)
    testImplementation(libs.turbine)
    testImplementation(libs.mockk)
    testImplementation(libs.coroutines.test)
}

Gradle generates type-safe accessors from the TOML file. The accessor name is derived from the library alias: dashes and dots become nested properties (libs.compose.ui.tooling), underscores become camelCase (libs.hiltNavigationCompose — though the convention is to use dashes).

Sharing Catalogs Across Projects

For organizations with multiple repositories, publish the catalog as a Gradle plugin:

// version-catalog/build.gradle.kts
plugins {
    `version-catalog`
    `maven-publish`
}

catalog {
    versionCatalog {
        from(files("gradle/libs.versions.toml"))
    }
}

publishing {
    publications {
        create<MavenPublication>("catalog") {
            from(components["versionCatalog"])
            groupId = "com.mycompany"
            artifactId = "version-catalog"
            version = "2024.12.1"
        }
    }
}
// Consumer project — settings.gradle.kts
dependencyResolutionManagement {
    versionCatalogs {
        create("libs") {
            from("com.mycompany:version-catalog:2024.12.1")
        }
    }
}
Staff Insight
Version catalogs replaced ext {} variables and buildSrc constants as the standard for dependency management. The key advantage over buildSrc is that changing a version in the TOML file does not invalidate the entire build cache — only the affected modules recompile. In large codebases (50+ modules), this difference saves minutes per build.

2. Bill of Materials (BOM)

A BOM (Bill of Materials) is a special POM file that declares a set of library versions known to be compatible with each other. Instead of specifying individual versions for every library in a family, you import the BOM and let it resolve versions for you.

How BOMs Work

platform() vs enforcedPlatform()

dependencies {
    // Recommendation — BOM versions used unless a dependency explicitly requests higher
    implementation(platform(libs.compose.bom))

    // Mandate — BOM versions forced even if transitive dependencies request higher
    implementation(enforcedPlatform(libs.compose.bom))
}
Behaviorplatform()enforcedPlatform()
Transitive dep requests higher versionHigher version winsBOM version forced
Direct dep specifies explicit versionExplicit version winsBOM version forced
Use caseDefault — safe for most projectsStrict version control (enterprise)
RiskVersion drift possibleBinary incompatibility if BOM is stale

Key Android BOMs

BOMArtifactCovers
Composeandroidx.compose:compose-bomAll Compose UI, Material, Foundation, Runtime
Firebasecom.google.firebase:firebase-bomAll Firebase SDKs
OkHttpcom.squareup.okhttp3:okhttp-bomOkHttp, MockWebServer, Logging Interceptor
Kotlinorg.jetbrains.kotlin:kotlin-bomkotlin-stdlib, kotlin-reflect
Coroutinesorg.jetbrains.kotlinx:kotlinx-coroutines-bomcoroutines-core, -android, -test, -rx

Creating Your Own BOM

// bom/build.gradle.kts
plugins {
    `java-platform`
    `maven-publish`
}

javaPlatform {
    allowDependencies()
}

dependencies {
    // Import upstream BOMs
    api(platform("androidx.compose:compose-bom:2024.12.01"))

    constraints {
        api("com.mycompany:core:1.5.0")
        api("com.mycompany:networking:2.1.0")
        api("com.mycompany:analytics:1.3.0")
        api("com.mycompany:design-system:3.0.0")
    }
}

publishing {
    publications {
        create<MavenPublication>("bom") {
            from(components["javaPlatform"])
            groupId = "com.mycompany"
            artifactId = "app-bom"
            version = "2024.12.1"
        }
    }
}
Staff Insight
BOMs solve the "which versions are compatible?" problem. Without a BOM, upgrading compose-ui to 1.7.6 while leaving compose-material3 at 1.2.0 can cause binary incompatibilities that manifest as NoSuchMethodError at runtime. Always use the official BOM for library families. For internal libraries at scale, publish your own BOM — it's the single source of truth for "these versions are tested together."

3. Dependency Configurations Deep Dive

Gradle configurations control when a dependency is available (compile time, runtime, or both) and whether it leaks to consumers of your module. Choosing the wrong configuration causes unnecessary recompilation, bloated APKs, or runtime crashes.

Configuration Reference Table

ConfigurationCompileRuntimeTransitive to ConsumersUse Case
implementationYesYesNoDefault — internal dependency
apiYesYesYesDependency exposed in public API
compileOnlyYesNoNoAnnotation processors, provided deps
runtimeOnlyNoYesNoSLF4J impl, DB drivers
testImplementationYes (tests)Yes (tests)NoUnit test dependencies
androidTestImplementationYes (androidTest)Yes (androidTest)NoInstrumented test dependencies
debugImplementationYes (debug)Yes (debug)NoDebug-only tools (LeakCanary)
ksp / kaptProcessorNoNoAnnotation/symbol processing
lintChecksNoNoNoCustom lint rules JAR

implementation vs api — The Critical Distinction

When to Use Each Configuration

// app/build.gradle.kts
dependencies {
    // implementation — Retrofit is used internally, not exposed
    implementation(libs.retrofit)

    // api — Room entities are part of the module's public API
    api(libs.room.runtime)

    // compileOnly — javax.inject annotations exist at compile time,
    // Hilt provides the actual implementation at runtime
    compileOnly("javax.inject:javax.inject:1")

    // runtimeOnly — Timber logging backend, never referenced in code directly
    runtimeOnly("com.jakewharton.timber:timber:5.0.1")

    // debugImplementation — LeakCanary only in debug builds
    debugImplementation("com.squareup.leakcanary:leakcanary-android:2.14")

    // ksp — Room annotation processor
    ksp(libs.room.compiler)

    // lintChecks — custom lint rules from an internal module
    lintChecks(project(":lint-rules"))
}
Common Mistake
Using api everywhere "just to be safe" is the #1 build performance killer in multi-module projects. Every api dependency creates a recompilation chain: if library X changes, every module that transitively depends on it recompiles. In a 60-module project, a single misplaced api can add 30+ seconds to incremental builds.

4. Dependency Resolution Strategy

When multiple modules request different versions of the same library, Gradle must decide which version to use. Understanding the resolution algorithm is essential for debugging version conflicts.

Default Behavior: Highest Version Wins

// Module A depends on OkHttp 4.11.0
// Module B depends on OkHttp 4.12.0
// Gradle resolves → OkHttp 4.12.0 (highest version wins)

Rich Version Constraints

Gradle supports fine-grained version declarations beyond simple version strings:

dependencies {
    implementation("com.example:lib") {
        version {
            // Hard constraint — reject anything outside this range
            strictly("[1.0, 2.0[")

            // Soft preference within the range
            prefer("1.5")
        }
    }

    // Shorthand forms
    implementation("com.example:lib:1.0!!") // strictly 1.0 (the !! suffix)
    implementation("com.example:lib") {
        version {
            require("1.0")    // minimum — allow higher
            prefer("1.2")     // ideal version
            reject("1.1")     // known broken version
        }
    }
}
ConstraintBehaviorWhen to Use
strictly("1.0")Reject any version other than 1.0, even if transitive deps request higherKnown incompatibility with newer versions
require("1.0")Minimum version — allow higher if another dep requires itDefault behavior, explicit minimum
prefer("1.0")Soft preference — easily overridden by any other constraintSuggestion, no hard requirement
reject("1.1")Exclude a specific version from resolutionKnown broken version
strictly("[1.0, 2.0[")Accept any version in range [1.0, 2.0), reject outsideMajor version boundary enforcement

ResolutionStrategy

// build.gradle.kts
configurations.all {
    resolutionStrategy {
        // Force a specific version globally
        force("com.squareup.okhttp3:okhttp:4.12.0")

        // Fail the build on ANY version conflict (strict mode)
        failOnVersionConflict()

        // Prefer project modules over external modules
        preferProjectModules()

        // Cache dynamic versions for 10 minutes (default: 24 hours)
        cacheDynamicVersionsFor(10, "minutes")

        // Don't cache changing modules (SNAPSHOTs)
        cacheChangingModulesFor(0, "seconds")

        // Substitute one dependency for another
        dependencySubstitution {
            substitute(module("com.example:old-lib"))
                .using(module("com.example:new-lib:2.0.0"))
                .because("old-lib was renamed to new-lib in v2")
        }
    }
}
Resolution Order
Gradle resolves versions in this priority: strictly > force() > enforcedPlatform() > require > platform() > prefer. When two strictly constraints conflict, the build fails — this is by design, forcing you to resolve the incompatibility explicitly.

5. Transitive Dependencies — The Hidden Danger

Transitive dependencies are libraries pulled in indirectly through your direct dependencies. They're the #1 source of mysterious runtime crashes in modular Android projects.

The Diamond Dependency Problem

Inspecting the Dependency Tree

# Full dependency tree for the runtimeClasspath configuration
./gradlew :app:dependencies --configuration runtimeClasspath

# Why was a specific version selected?
./gradlew :app:dependencyInsight --dependency com.google.code.gson:gson --configuration runtimeClasspath

# Output shows the resolution chain:
# com.google.code.gson:gson:2.10.1
#   variant "runtime" [
#     org.jetbrains.kotlin.platform.type = jvm
#   ]
#   Selection reasons:
#     - By conflict resolution: between versions 2.10.1 and 2.8.9

Excluding Transitive Dependencies

dependencies {
    // Exclude a specific transitive dependency
    implementation("com.example:analytics:3.0") {
        exclude(group = "com.google.code.gson", module = "gson")
    }

    // Exclude globally from a configuration
    configurations.implementation {
        exclude(group = "org.apache.httpcomponents")
    }

    // Provide the correct version yourself
    implementation("com.google.code.gson:gson:2.10.1")
}

Dependency Substitution

configurations.all {
    resolutionStrategy.dependencySubstitution {
        // Replace a published artifact with a local project (for development)
        substitute(module("com.mycompany:core"))
            .using(project(":core"))
            .because("Using local version during development")

        // Replace an old artifact with its successor
        substitute(module("com.android.support:support-v4"))
            .using(module("androidx.legacy:legacy-support-v4:1.0.0"))
            .because("Migrated to AndroidX")
    }
}
Runtime Crash Pattern
The most insidious transitive dependency bug: Library A uses Gson.fromJson(Reader, Type) which existed in Gson 2.8. Gson 2.10 changes the method signature. Gradle resolves to 2.10 (highest wins). The app compiles fine because your code doesn't call that method — but Library A does, and it crashes at runtime with NoSuchMethodError. The fix: pin Gson to 2.8 with strictly, or update Library A to a version compiled against Gson 2.10.

6. Dependency Locking

Without dependency locking, the same build.gradle.kts can produce different builds on different days. A transitive dependency releases a new patch version, Gradle resolves to it, and your app's behavior silently changes. Dependency locking pins every resolved version to a lockfile.

Enabling Dependency Locking

// build.gradle.kts
dependencyLocking {
    lockAllConfigurations()
}

// Or lock specific configurations
configurations {
    compileClasspath { resolutionStrategy.activateDependencyLocking() }
    runtimeClasspath { resolutionStrategy.activateDependencyLocking() }
}

Generating and Updating Lockfiles

# Generate lockfiles for all locked configurations
./gradlew :app:dependencies --write-locks

# Update locks for a specific dependency
./gradlew :app:dependencies --update-locks com.squareup.okhttp3:*

# Lockfiles are written to:
# gradle/dependency-locks/compileClasspath.lockfile
# gradle/dependency-locks/runtimeClasspath.lockfile

Lockfile Format

# This is a Gradle generated file for dependency locking.
# Manual edits can mess up your build. Be careful!
androidx.activity:activity-compose:1.9.3=runtimeClasspath
androidx.activity:activity-ktx:1.9.3=runtimeClasspath
androidx.compose.runtime:runtime:1.7.6=runtimeClasspath
com.google.code.gson:gson:2.10.1=runtimeClasspath
com.squareup.okhttp3:okhttp:4.12.0=runtimeClasspath
com.squareup.retrofit2:retrofit:2.11.0=runtimeClasspath
empty=

Lockfiles must be committed to version control. When a dependency version changes without updating the lockfile, the build fails — this is the safety net.

Workflow

  1. Enable locking, generate initial lockfiles, commit them
  2. When updating dependencies: change the version in libs.versions.toml, run --write-locks, review the diff, commit
  3. CI verifies: if lockfiles are stale (dependency changed without updating locks), the build fails
Staff Insight
Dependency locking is non-negotiable for production apps. Without it, a transitive dependency's patch release can silently change your app's behavior between Monday's build and Tuesday's build — with zero changes to your code. At Google scale, every build must be reproducible. Lockfiles are the mechanism that guarantees "same code + same lockfile = same binary."

7. Dependency Analysis Plugin

The Dependency Analysis Gradle Plugin by Tony Robalik is the industry-standard tool for detecting dependency misconfigurations. It catches problems that are invisible to the compiler but cause unnecessary recompilation, bloated APKs, and subtle bugs.

Setup

// settings.gradle.kts (or root build.gradle.kts)
plugins {
    id("com.autonomousapps.dependency-analysis") version "2.4.2"
}

// Optional: configure severity
dependencyAnalysis {
    issues {
        all {
            onUnusedDependencies { severity("fail") }
            onUsedTransitiveDependencies { severity("warn") }
            onIncorrectConfiguration { severity("fail") }
            onRedundantPlugins { severity("warn") }
        }
    }
}

Running the Analysis

# Full project health report
./gradlew buildHealth

# Per-module advice
./gradlew :app:projectHealth

# Auto-fix (applies suggested changes to build files)
./gradlew fixDependencies

What It Detects

IssueExampleImpact
Unused dependencyimplementation(libs.gson) but Gson is never importedBloated APK, slower builds
Used-but-undeclaredCode uses Gson but it's only available transitively through RetrofitFragile — removing Retrofit breaks Gson usage
api → implementationapi(libs.retrofit) but Retrofit types aren't in public APIUnnecessary recompilation chain
implementation → apiimplementation(libs.room.runtime) but Room entities are in public APIConsumer can't see Room types at compile time

CI Integration
Run ./gradlew buildHealth as a CI check. Configure onUnusedDependencies { severity("fail") } and onIncorrectConfiguration { severity("fail") } to fail the build on dependency hygiene violations. This prevents dependency rot from accumulating over time.

8. Avoiding Dependency Hell

Dependency hell is the state where version conflicts, transitive incompatibilities, and configuration mistakes compound until builds break in unpredictable ways. Here's the systematic approach to preventing it.

The Dependency Hygiene Checklist

#PracticeWhy
1Use version catalogs for all dependenciesSingle source of truth, type-safe, no cache invalidation
2Use BOMs for library familiesGuaranteed compatible versions within a family
3Default to implementation, use api only when requiredMinimizes recompilation chains
4Enable dependency lockingReproducible builds, no silent version drift
5Run dependency analysis in CICatches unused deps, wrong configurations
6Never use SNAPSHOT dependencies in release buildsSNAPSHOTs change without notice, break reproducibility
7Inspect dependency tree after adding new depsCatch transitive conflicts early
8Create a dedicated branch for dependency updatesIsolate risk, run full test suite
9Use reject() for known broken versionsPrevent Gradle from resolving to a buggy release
10Audit transitive deps with dependencyInsightUnderstand why a version was selected

Dependency Update Strategy

// Automate dependency updates with Renovate or Dependabot
// .github/renovate.json
{
  "extends": ["config:base"],
  "packageRules": [
    {
      "matchPackagePatterns": ["androidx.compose*"],
      "groupName": "Compose",
      "schedule": ["every 2 weeks"]
    },
    {
      "matchPackagePatterns": ["com.squareup.okhttp3*"],
      "groupName": "OkHttp"
    }
  ],
  "gradle": {
    "fileMatch": ["(^|/)build\\.gradle\\.kts$", "(^|/)libs\\.versions\\.toml$"]
  }
}
SNAPSHOT Dependencies
Never use -SNAPSHOT versions in production code. A SNAPSHOT artifact can change on every publish — the same version string resolves to different bytecode on different days. If you must use a SNAPSHOT for development, pin it with dependency locking and replace it with a stable version before merging to main.

9. Kotlin Multiplatform Dependencies

KMP modules introduce a source-set-based dependency model. Dependencies are declared per source set, and Gradle resolves platform-specific variants automatically.

Source Set Dependency Structure

// shared/build.gradle.kts
plugins {
    kotlin("multiplatform")
    id("com.android.library")
}

kotlin {
    androidTarget()
    iosX64()
    iosArm64()
    iosSimulatorArm64()

    sourceSets {
        commonMain.dependencies {
            implementation(libs.ktor.core)
            implementation(libs.ktor.content.negotiation)
            implementation(libs.kotlinx.serialization)
            implementation(libs.kotlinx.coroutines.core)
            implementation(libs.koin.core)
        }

        commonTest.dependencies {
            implementation(kotlin("test"))
            implementation(libs.kotlinx.coroutines.test)
        }

        androidMain.dependencies {
            implementation(libs.ktor.okhttp)     // Android-specific HTTP engine
            implementation(libs.koin.android)
        }

        iosMain.dependencies {
            implementation(libs.ktor.darwin)      // iOS-specific HTTP engine
        }
    }
}

expect/actual and Dependencies

// commonMain — declares the contract
expect class PlatformHttpEngine {
    fun create(): HttpClientEngine
}

// androidMain — uses OkHttp (declared in androidMain dependencies)
actual class PlatformHttpEngine {
    actual fun create(): HttpClientEngine = OkHttp.create()
}

// iosMain — uses Darwin (declared in iosMain dependencies)
actual class PlatformHttpEngine {
    actual fun create(): HttpClientEngine = Darwin.create()
}

Consuming KMP Modules from Android

// app/build.gradle.kts (pure Android module)
dependencies {
    // KMP module consumed as a regular Android library
    implementation(project(":shared"))

    // The KMP module's androidMain dependencies are transitively available
    // (if declared as api), or hidden (if declared as implementation)
}

KMP Dependency Resolution
KMP uses Gradle's variant-aware dependency resolution. When you declare implementation(libs.ktor.core) in commonMain, Gradle resolves the correct platform-specific artifact for each target (JVM JAR for Android, Kotlin/Native klib for iOS). This is powered by Gradle Module Metadata (.module files) which carry variant information that traditional POM files cannot express.

10. Staff-Level Corner Cases

classpath vs implementation in buildscript

// buildscript { } — loaded by Gradle's own classloader
buildscript {
    dependencies {
        classpath("com.android.tools.build:gradle:8.7.3")  // Gradle plugin
    }
}

// dependencies { } — loaded by the app's classloader
dependencies {
    implementation("com.android.tools.build:gradle:8.7.3") // WRONG — this is a plugin, not an app dependency
}

The classpath configuration loads classes into Gradle's classloader. The implementation configuration loads classes into the app's classloader. Mixing them causes ClassCastException — the same class loaded by two different classloaders are considered different types by the JVM.

Gradle Module Metadata (.module files)

Traditional POM files describe a single artifact with a flat list of dependencies. Gradle Module Metadata (.module files) are richer:

  • Variant-aware: describes multiple variants (debug/release, JVM/Android/iOS) in a single metadata file
  • Rich version constraints: strictly, require, prefer, reject — POM can only express a single version
  • Capabilities: declare what a module provides (e.g., "logging"), enabling conflict detection
  • Dependency constraints: recommend versions for transitive dependencies without adding them as direct dependencies
// Published .module file (simplified)
{
  "formatVersion": "1.1",
  "component": {

Capabilities — Detecting Conflicting Modules

// Two logging libraries that provide the same capability
// timber/build.gradle.kts
configurations.all {
    outgoing {
        capability("com.mycompany:logging:1.0")
    }
}

// custom-logger/build.gradle.kts
configurations.all {
    outgoing {
        capability("com.mycompany:logging:1.0")
    }
}

// If both are on the classpath, Gradle fails with:
// "Module 'timber' and 'custom-logger' both provide capability 'com.mycompany:logging:1.0'"
// Forces you to choose one — no silent conflicts

Composite Build Dependency Substitution

// settings.gradle.kts
includeBuild("../shared-library") {
    dependencySubstitution {
        substitute(module("com.mycompany:shared-library"))
            .using(project(":"))
    }
}

// Now any module that depends on "com.mycompany:shared-library:x.y.z"
// will use the local project instead — changes are reflected immediately
// without publishing to a Maven repository

This is essential for developing libraries alongside the app that consumes them. Without composite builds, you'd need to publish a SNAPSHOT after every change.

platform() vs enforcedPlatform() — The Subtle Difference

dependencies {
    // platform() — recommendation
    // If library X transitively brings Gson 2.11, Gson 2.11 wins
    implementation(platform("com.mycompany:app-bom:1.0"))

    // enforcedPlatform() — mandate
    // Even if library X transitively brings Gson 2.11, BOM's Gson 2.10 is forced
    implementation(enforcedPlatform("com.mycompany:app-bom:1.0"))
}

// Danger with enforcedPlatform(): if the BOM declares Gson 2.10 but library X
// was compiled against Gson 2.11 and uses a 2.11-only API, you get a runtime crash.
// enforcedPlatform() trades version drift for potential binary incompatibility.
Staff Insight
At Google scale, dependency management is a team sport. The combination of version catalogs (single source of truth), BOMs (compatible version sets), dependency locking (reproducible builds), and the dependency analysis plugin (hygiene enforcement) forms a four-layer defense against dependency hell. Miss any layer and problems compound: without catalogs, versions drift across modules; without BOMs, library families become incompatible; without locking, builds aren't reproducible; without analysis, unused and misconfigured dependencies accumulate silently. In an interview, demonstrate that you understand not just what each tool does, but why the combination matters.
