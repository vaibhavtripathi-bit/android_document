03 вҖ” Modularization Strategies

From monolith to modular вҖ” migration playbook, by-layer vs by-feature vs hybrid approaches, dependency graph shapes, anti-patterns, dynamic features, and real-world lessons from Uber, Airbnb, Cash App, and Google

1. Monolith вҶ’ Modular Migration

Every modularization journey starts with a monolith вҖ” a single :app module that contains everything. The migration must be incremental. Attempting a big-bang rewrite is the single most common reason modularization efforts fail.

The Six-Phase Playbook

Phase 1 вҖ” Extract :core:model: Start with pure data classes вҖ” API response models, database entities, domain objects. This module has zero Android dependencies (pure Kotlin), making it the safest extraction target.

// :core:model/build.gradle.kts
plugins {
    id("org.jetbrains.kotlin.jvm")  // pure Kotlin вҖ” no Android plugin
}

// :core:model/src/main/kotlin/com/example/model/User.kt
data class User(
    val id: String,
    val name: String,
    val email: String,
    val avatarUrl: String?,
)

// :core:model/src/main/kotlin/com/example/model/Article.kt
data class Article(
    val id: Long,
    val title: String,
    val body: String,
    val author: User,
    val publishedAt: Instant,
)

Phase 2 вҖ” Extract :core:network and :core:database: Infrastructure modules that wrap Retrofit, OkHttp, Room, and DataStore. They depend on :core:model for shared types.

// :core:network depends on :core:model
// :core:database depends on :core:model

// :core:network/build.gradle.kts
dependencies {
    implementation(project(":core:model"))
    implementation(libs.retrofit)
    implementation(libs.okhttp)
    implementation(libs.moshi)
}

// :core:database/build.gradle.kts
dependencies {
    implementation(project(":core:model"))
    implementation(libs.room.runtime)
    ksp(libs.room.compiler)
}

Phase 3 вҖ” Extract :core:domain: Use cases and business logic. Pure Kotlin, depends only on :core:model. Repositories are defined as interfaces here (not implementations).

// :core:domain вҖ” pure Kotlin, no Android dependencies
interface ArticleRepository {
    fun getArticles(): Flow<List<Article>>
    suspend fun getArticle(id: Long): Article
    suspend fun bookmarkArticle(id: Long)
}

class GetBookmarkedArticlesUseCase(
    private val articleRepo: ArticleRepository,
    private val userRepo: UserRepository,
) {
    operator fun invoke(): Flow<List<Article>> =
        articleRepo.getArticles()
            .combine(userRepo.bookmarks) { articles, bookmarks ->
                articles.filter { it.id in bookmarks }
            }
}

Phase 4 вҖ” Extract :core:data: Repository implementations that combine network + database. Depends on :core:domain (for interfaces), :core:network, and :core:database.

Phase 5 вҖ” Extract feature modules one by one: Start with the most independent feature вҖ” the one with the fewest cross-feature dependencies. Each feature module contains UI (Composables or Fragments), ViewModels, and navigation logic.

Phase 6 вҖ” Slim down :app: The final :app module should contain only DI wiring (Hilt @Modules), the main Activity, the top-level navigation graph, and the Application class.

Staff Insight вҖ” Incremental Migration

Never try to modularize everything at once. Extract one module at a time, verify the build compiles, run the full test suite, and merge to main before starting the next extraction. A failed modularization attempt вҖ” where half the codebase is in limbo between old and new structure вҖ” can set a team back months. At Google, modularization PRs are small and frequent, not large and infrequent.

Characteristics

  • Each module contains all features for that layer вҖ” every screen in :ui, every repository in :data
  • Dependencies flow strictly downward вҖ” :ui вҶ’ :domain вҶ’ :data вҶ’ :network/:database
  • Architecture rules are enforced by module boundaries вҖ” the UI layer physically cannot import database classes

Pros

  • Simple to understand вҖ” mirrors clean architecture layers directly
  • Clear separation of concerns at the module level
  • Easy to enforce with Gradle dependency rules

Cons

  • Modules grow massive вҖ” :ui eventually contains hundreds of screens
  • One feature change touches every module вҖ” adding a "bookmarks" feature requires changes in :ui, :domain, :data, :network, and :database
  • No build parallelism вҖ” the chain is strictly sequential, so Gradle can't build layers in parallel
  • Poor team ownership вҖ” no team "owns" a module because every module contains every team's code

When to Use

Small apps with 1вҖ“2 developers, prototypes, or apps with very few features where the overhead of more modules isn't justified.

Characteristics

  • Each feature module is self-contained вҖ” it has its own UI, ViewModel, domain logic, data layer, and even network calls
  • Features can be developed, tested, and deployed independently
  • Teams own entire features end-to-end

Pros

  • Team autonomy вҖ” a team can work on :feature:search without touching any other module
  • Parallel development вҖ” features don't block each other
  • Feature-level testing вҖ” run tests for just one feature in isolation
  • Build parallelism вҖ” Gradle can build all feature modules in parallel

Cons

  • Code duplication вҖ” each feature has its own Retrofit service, Room DAO, and repository. Shared API endpoints are duplicated.
  • Shared logic is hard to place вҖ” where does "user authentication" live if both :feature:home and :feature:profile need it?
  • Deep dependency graphs вҖ” features that need shared data end up depending on each other, creating coupling

When to Use

Medium-to-large apps with multiple teams and distinct, loosely-coupled features. Works well when features rarely share data.

4. Hybrid Approach (Recommended)

The hybrid approach combines the best of both worlds: vertical slicing for features (UI + ViewModel only) and horizontal slicing for shared infrastructure (:core:* modules). This is Google's recommended architecture, demonstrated in the Now in Android reference app.

        в””в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”ҳ

Key Rules

  • Feature modules contain only UI and presentation logic вҖ” Composables, ViewModels, UI state classes, navigation destinations
  • Feature modules depend on :core:data and :core:domain, never on other feature modules
  • :core:* modules are shared infrastructure вҖ” data, domain, model, network, database, UI (design system)
  • :core:domain is pure Kotlin вҖ” no Android dependencies, testable without Robolectric
  • :core:model is the leaf вҖ” depended on by everything, depends on nothing

Pros

  • Features are focused and small вҖ” only UI code, easy to reason about
  • Shared logic is centralized вҖ” no duplication of repositories or use cases
  • Clear, enforceable dependency rules вҖ” features вҶ’ core, never feature вҶ’ feature
  • Good build performance вҖ” features build in parallel, core modules are cached

Cons

  • More modules to manage вҖ” a medium app might have 20вҖ“40 modules
  • :core:data can become a bottleneck вҖ” if it grows too large, any change triggers recompilation of all features
  • Requires discipline to keep feature modules thin вҖ” tempting to put business logic in ViewModels

Comparison Table

5. Vertical vs Horizontal Slicing

These two terms capture the fundamental tension in modularization:

  • Vertical slice: a feature module that contains all layers (UI вҶ’ domain вҶ’ data) for one feature. Self-contained, independently testable, maps to team ownership.
  • Horizontal slice: a layer module that contains one layer for all features (all UI in :ui, all data in :data). Maximizes code reuse, minimizes duplication.

Most production projects use a mix: vertical slicing for features (each team owns their slice) and horizontal slicing for shared infrastructure (networking, database, design system). This is exactly the hybrid approach described above.

Staff Insight вҖ” Vertical Slicing at Google Scale

At Google scale, vertical slicing dominates because it aligns with team ownership. Each team can develop, test, and release their feature independently without coordinating with other teams. The cost of some code duplication is vastly outweighed by the velocity gain of independent deployment. When two teams need shared logic, it's extracted into a :core:* module with a clear owner вҖ” not dumped into a shared "commons" module.

6. The Dependency Graph вҖ” Shapes and Anti-Patterns

The shape of your module dependency graph determines build performance, maintainability, and how easily teams can work independently. Understanding graph shapes is a critical L6 skill.

Diamond Dependency

        :app
       /    \
      в–ј      в–ј
   :feat:A  :feat:B
       \    /
        в–ј  в–ј
     :core:data

Not inherently bad вҖ” this is normal in the hybrid approach. It becomes problematic when :feat:A and :feat:B transitively pull in different versions of the same library through :core:data. Gradle resolves this with "newest wins," but the mismatch can cause runtime surprises.

Circular Dependency

   :feature:home в”Җв”Җв”Җв”Җв”Җв”Җв–ә :feature:profile

Fix: Extract the shared code into a third module. If :feature:home needs to navigate to a profile screen and :feature:profile needs to display home widgets, extract the shared contract into :core:navigation or use an event bus / navigation abstraction.

// Before: circular dependency
// :feature:home depends on :feature:profile (for ProfileScreen)
// :feature:profile depends on :feature:home (for HomeWidget)

// After: extract shared contracts
// :core:navigation defines NavigationTarget sealed interface
// Both features depend on :core:navigation, not on each other

// :core:navigation
sealed interface NavigationTarget {
    data class Profile(val userId: String) : NavigationTarget
    data object Home : NavigationTarget
}

// :feature:home вҖ” depends on :core:navigation, NOT :feature:profile
class HomeViewModel(private val navigator: Navigator) {
    fun onProfileClick(userId: String) {
        navigator.navigate(NavigationTarget.Profile(userId))
    }
}

7. When NOT to Modularize

Modularization has real costs. It's not always the right investment.

Situation Why Modularization Hurts
App has < 5 screens, 1вҖ“2 developers Overhead of managing multiple modules exceeds any build or organizational benefit
Build time is already fast (< 30 seconds) Modularization adds configuration time that can offset compilation savings
Team lacks discipline for boundaries Module boundaries erode quickly вҖ” :core:common becomes a dumping ground, api is used everywhere
App is in maintenance mode The investment won't pay off if no new features are being added
Doing it for "clean architecture" alone Modularization should solve a real problem вҖ” build time, team scalability, code isolation вҖ” not satisfy an abstract ideal
Interview Warning

If asked "when would you modularize?", don't reflexively say "always." Demonstrating that you understand the costs of modularization вҖ” configuration overhead, increased complexity, steeper onboarding вҖ” shows L6-level judgment. The right answer is: "when the build time or team coordination cost of a monolith exceeds the overhead of managing modules."

8. Common Mistakes

Extracting Too Early

Modularizing before the architecture is stable leads to constant refactoring of module boundaries. If you're still figuring out which features exist and how they interact, keep it in :app and extract later when the boundaries are clear.

Circular Dependencies

Two features that depend on each other. Gradle rejects this at configuration time. The fix is always the same: extract the shared contract into a :core:* module that both features depend on.

God Modules

:core:common, :core:utils, or :core:shared вҖ” modules that everything depends on and that contain a grab-bag of unrelated utilities. Any change triggers a full rebuild. Split them into focused modules: :core:datetime, :core:formatting, :core:analytics, :core:testing.

Leaking Implementation Details

// BAD вҖ” leaks Retrofit and Moshi to all consumers of :core:network
// :core:network/build.gradle.kts
dependencies {
    api(libs.retrofit)        // вҶҗ consumers can see Retrofit types
    api(libs.moshi)           // вҶҗ consumers can see Moshi types
}

// GOOD вҖ” hides implementation details
dependencies {
    implementation(libs.retrofit)   // вҶҗ internal to :core:network
    implementation(libs.moshi)      // вҶҗ internal to :core:network
}
// Expose only your own interfaces/classes to consumers

Using api instead of implementation means consumers transitively depend on your internal libraries. This increases coupling, slows builds (more recompilation), and makes it impossible to swap implementations later.

Ignoring Build Performance

Adding modules without measuring build impact. Each module adds Gradle configuration time (~50вҖ“200ms per module). If you have 100 modules but most are tiny, configuration time can dominate compilation time. Measure with --profile and --scan before and after.

Not Using Convention Plugins

// BAD вҖ” duplicated across 30 modules
// :feature:home/build.gradle.kts
plugins {
    id("com.android.library")
    id("org.jetbrains.kotlin.android")
    id("com.google.devtools.ksp")
    id("dagger.hilt.android.plugin")
}
android {
    compileSdk = 34
    defaultConfig { minSdk = 26 }
    compileOptions {
        sourceCompatibility = JavaVersion.VERSION_17
        targetCompatibility = JavaVersion.VERSION_17
    }
    buildFeatures { compose = true }
    composeOptions { kotlinCompilerExtensionVersion = "1.5.8" }
}
// ... same 40 lines in every single module

// GOOD вҖ” convention plugin (covered in doc 05)
// :feature:home/build.gradle.kts
plugins {
    id("myapp.android.feature")  // one line, all config centralized
}
Staff Insight вҖ” Measure Before You Modularize

The most common modularization failure is not measuring. Teams modularize for "clean architecture" without measuring build time before and after. They're surprised when the build gets slower because Gradle configuration overhead (resolving dependencies, evaluating build scripts) for 80 modules exceeds the compilation savings from incremental builds. Always baseline your build time with ./gradlew --profile before starting, set a target (e.g., "reduce clean build from 4 min to 2 min"), and measure after each phase.

9. Real-World Case Studies

Uber вҖ” 1000+ Modules

  • One of the largest Android codebases in the world вҖ” over 1000 Gradle modules
  • Migrated from Buck to Bazel for build performance at scale (Gradle couldn't handle the graph size)
  • Strict module ownership вҖ” every module has a designated team owner in a CODEOWNERS-style system
  • Dependency rules enforced by CI вҖ” a PR that adds a dependency from :feature:rides to :feature:eats is automatically rejected
  • Custom api/impl split вҖ” public API modules expose interfaces, implementation modules are swappable

Airbnb вҖ” Strict API Boundaries

  • Feature modules with explicit API surfaces вҖ” each feature publishes a Kotlin interface that other features can depend on
  • Shared design system in a dedicated :design-system module вҖ” all UI components, tokens, and themes
  • Custom Gradle plugins enforce that feature modules don't depend on other feature modules directly вҖ” only through published APIs
  • Invested heavily in module-level testing вҖ” each feature module has its own test suite that runs independently

Cash App вҖ” api/impl Split

  • Pioneered the :feature:foo:api / :feature:foo:impl pattern вҖ” the API module contains interfaces and models, the impl module contains the actual implementation
  • Other features depend only on :feature:foo:api, never on :feature:foo:impl вҖ” this enforces encapsulation at the Gradle level
  • Extensive use of convention plugins вҖ” all build configuration centralized in build-logic/
  • Early adopter of Kotlin Multiplatform modules вҖ” shared business logic between Android and iOS
// Cash App api/impl pattern
// :feature:payments:api вҖ” interfaces only
interface PaymentsRepository {
    fun getPayments(): Flow<List<Payment>>
    suspend fun sendPayment(request: PaymentRequest): PaymentResult
}

// :feature:payments:impl вҖ” actual implementation
// Other features NEVER depend on this module
class PaymentsRepositoryImpl(
    private val api: PaymentsApi,
    private val dao: PaymentsDao,
) : PaymentsRepository {
    override fun getPayments(): Flow<List<Payment>> =
        dao.observePayments().map { entities ->
            entities.map { it.toDomain() }
        }
}

// :feature:home depends on :feature:payments:api (not :impl)
// Hilt wires the impl at the :app level

Google вҖ” Now in Android

  • Official reference architecture вҖ” github.com/android/nowinandroid
  • ~30 modules using the hybrid approach: :feature:* + :core:*
  • Convention plugins in build-logic/ вҖ” every module type (android-library, android-feature, kotlin-jvm) has a convention plugin
  • Version catalog (libs.versions.toml) for centralized dependency versions
  • Demonstrates the full pattern: feature modules depend on :core:data, :core:domain, :core:model, :core:ui

Lessons Learned Across All

Lesson Details
Start small Extract 2вҖ“3 core modules first, prove the pattern works, then scale
Measure everything Build time before/after, configuration time, compilation time, test execution time
Enforce with tooling Code review alone can't maintain boundaries вҖ” use CI checks, dependency-analysis plugin, custom lint rules
Invest in build infra early Convention plugins, version catalogs, and build cache pay for themselves quickly
Module ownership matters Every module needs a clear owner вҖ” unowned modules become dumping grounds

10. Dynamic Feature Modules (Play Feature Delivery)

Dynamic feature modules are delivered on-demand from the Play Store rather than included in the base APK. They use the com.android.dynamic-feature Gradle plugin.

Use Cases

  • Large features that not all users need вҖ” camera filters, AR experiences, offline maps, advanced editing tools
  • Region-specific features вҖ” payment methods, compliance modules
  • Reducing initial APK size вҖ” users download features only when they need them

The Inverted Dependency

Dynamic feature modules have an inverted dependency relationship compared to normal modules. The dynamic feature depends on :app, not the other way around. This is because the base APK must be installable without the dynamic feature.

NORMAL MODULES                      DYNAMIC FEATURE MODULES

     :app                                :app
      в”Ӯ                                   в–І
      в–ј                                   в”Ӯ (inverted!)
   :feature:foo                    :dynamic:camera_filters
                                          в”Ӯ
                                          в–ј
                                   :core:image_processing
// :dynamic:camera_filters/build.gradle.kts
plugins {
    id("com.android.dynamic-feature")
}
android {
    // dynamic features don't set compileSdk вҖ” inherited from :app
}
dependencies {
    implementation(project(":app"))  // depends on :app, not the other way around
}

// :app/build.gradle.kts
android {
    dynamicFeatures += setOf(":dynamic:camera_filters")
}

SplitInstallManager вҖ” Runtime Installation

class CameraActivity : AppCompatActivity() {

    private val splitInstallManager by lazy {
        SplitInstallManagerFactory.create(this)
    }

    fun requestCameraFilters() {
        val request = SplitInstallRequest.newBuilder()
            .addModule("camera_filters")
            .build()

        splitInstallManager.startInstall(request)
            .addOnSuccessListener { sessionId ->
                // Monitor installation progress
            }
            .addOnFailureListener { exception ->
                // Handle: network error, insufficient storage, etc.
            }
    }

    private val listener = SplitInstallStateUpdatedListener { state ->
        when (state.status()) {
            SplitInstallSessionStatus.DOWNLOADING -> {
                val total = state.totalBytesToDownload()
                val downloaded = state.bytesDownloaded()
                updateProgressBar(downloaded, total)
            }
            SplitInstallSessionStatus.INSTALLED -> {
                // Feature is ready вҖ” access via reflection or SplitCompat
                launchCameraFilters()
            }
            SplitInstallSessionStatus.FAILED -> {
                Log.e("DFM", "Install failed: ${state.errorCode()}")
            }
        }
    }

    private fun launchCameraFilters() {
        // Because :app can't depend on :dynamic:camera_filters,
        // you must use reflection or a service-locator pattern
        val intent = Intent()
            .setClassName(this, "com.example.camerafilters.CameraFiltersActivity")
        startActivity(intent)
    }
}

Limitations

  • No api dependencies вҖ” the base module can't see types from the dynamic feature at compile time
  • Reflection or service locator required вҖ” cross-module access must use Class.forName(), Intent with class names, or a shared interface in :app
  • Testing is complex вҖ” dynamic features aren't available in unit tests by default; you need SplitCompat and special test configurations
  • Play Store only вҖ” doesn't work with sideloaded APKs or alternative stores
  • Size overhead вҖ” each dynamic feature adds metadata and split APK overhead
Staff Insight вҖ” Dynamic Features: Power vs Complexity

Dynamic feature modules are powerful for APK size reduction вҖ” Uber reportedly saved 20+ MB by making rarely-used features on-demand. But they add significant engineering complexity: inverted dependencies, reflection-based access, complex testing, Play Store coupling, and user-facing download flows. Use them only when the size savings justify the cost. For most apps, Android App Bundles (which automatically split by density, ABI, and language) provide 80% of the size benefit with none of the complexity. Reserve dynamic features for truly large, optional features (50+ MB) that a minority of users need.

Alternative: Conditional Delivery

Instead of on-demand installation, you can use conditional delivery вҖ” features are automatically installed based on device properties (API level, device features, country). This avoids the runtime installation UX entirely while still reducing APK size for devices that don't need the feature.

<!-- In the dynamic feature's AndroidManifest.xml -->
<dist:module dist:title="@string/ar_features">
    <dist:delivery>
        <dist:install-time>
            <dist:conditions>
                <dist:device-feature dist:name="android.hardware.camera.ar"/>
                <dist:min-sdk dist:value="28"/>
            </dist:conditions>
        </dist:install-time>
    </dist:delivery>
</dist:module>
