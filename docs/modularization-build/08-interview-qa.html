08 — Interview Q&A — Modularization & Gradle Build System

60 Staff Engineer questions covering Gradle internals, module architecture, modularization strategies, dependency management, build performance, convention plugins, and common multi-module problems

Gradle Internals (Q1–Q10)

Q1. Explain Gradle's three build phases — what happens in each and why does the configuration phase matter for performance?

Initialization evaluates settings.gradle.kts to determine which projects participate in the build. Gradle creates a Settings object, resolves include() directives, and builds the project hierarchy — for a 200-module project this means instantiating 200 Project objects. Configuration executes every build.gradle.kts file, evaluating all plugin blocks, dependency declarations, and task registrations. This is where the directed acyclic graph (DAG) of tasks is constructed. Execution runs only the tasks required to satisfy the requested goal, determined by walking the DAG from the target task backward through its dependencies.

Configuration phase performance is critical because it runs on every single build invocation, even for a trivial ./gradlew help. In a 200-module project, if each build.gradle.kts takes 50ms to configure, that's 10 seconds before any real work begins. This is why tasks.register() (lazy) vs tasks.create() (eager) matters — eager creation forces task instantiation and configuration during the configuration phase. The configuration cache (Gradle 8+) serializes the task graph after the first build and skips the entire configuration phase on subsequent runs when inputs haven't changed, reducing incremental build startup from seconds to milliseconds. AGP 7+ is fully compatible with configuration caching.

Q2. What is the difference between tasks.register() and tasks.create()? Why does it matter?

tasks.create() eagerly instantiates the task object and runs its configuration closure during the configuration phase — even if the task will never be executed. tasks.register() returns a TaskProvider<T> and defers both instantiation and configuration until the task is actually needed (either directly requested or pulled into the DAG as a dependency). This is Gradle's "configuration avoidance" API, introduced in Gradle 4.9.

The impact is substantial: in a project with 500 registered tasks per module and 100 modules, eager creation instantiates 50,000 task objects during configuration. With lazy registration, only the ~50 tasks actually in the execution path are instantiated. This reduces configuration time, heap pressure, and GC pauses. AGP internally uses register() extensively — the variant API (onVariants {}, beforeVariants {}) is built around lazy Provider/Property wiring. The rule: never call tasks.create() in modern Gradle. Use tasks.register() and wire dependencies via TaskProvider references. If you need to configure a registered task, use tasks.named("taskName") { ... } which also returns a TaskProvider and defers configuration.

Q3. How does Gradle's dependency resolution work? What happens when two modules request different versions of the same library?

Gradle resolves dependencies in a multi-step process: (1) it collects all declared dependencies from the requesting configuration (e.g., implementation), (2) it resolves each dependency coordinate to a module version using repository metadata (POM/Gradle Module Metadata), (3) it builds a dependency graph by recursively resolving transitive dependencies, (4) it applies conflict resolution when multiple versions of the same module appear in the graph. The default strategy is highest version wins — if module A depends on OkHttp 4.9 and module B depends on OkHttp 4.12, Gradle selects 4.12 for the entire graph.

This can cause runtime failures if the higher version removes APIs the lower-version consumer relies on. Gradle Module Metadata (GMM) improves on Maven POMs by supporting variant-aware resolution — a library can publish different artifacts for different attributes (e.g., org.gradle.jvm.version), and Gradle selects the best match. You can override the default strategy with resolutionStrategy { force("com.squareup.okhttp3:okhttp:4.9.3") }, use strictly version constraints to reject upgrades, or implement a custom ComponentSelection rule. The dependencyInsight task (./gradlew :app:dependencyInsight --dependency okhttp --configuration runtimeClasspath) shows exactly why a particular version was selected, including the conflict resolution path.

Q4. Explain the build cache — how does it work, what is the cache key, and what causes cache misses?

Gradle's build cache stores task outputs keyed by a hash of the task's inputs. When a cacheable task is about to execute, Gradle computes a cache key by hashing: the task class implementation (bytecode hash), the task's input properties (file contents, primitive values, nested inputs), the task's classpath (plugin JARs), and the names of output properties. If a matching entry exists in the cache, outputs are restored from cache instead of executing the task — this is the "UP-TO-DATE" → "FROM-CACHE" distinction.

Common cache miss causes: (1) absolute paths leaking into inputs — if a task input includes /Users/alice/project/src/..., it won't match on Bob's machine; use @PathSensitive(PathSensitivity.RELATIVE) to normalize. (2) Non-deterministic inputs like timestamps, random values, or build-time-generated UUIDs embedded in outputs. (3) Volatile task inputs — a task that reads System.getenv("CI") without declaring it as an input will produce different outputs but the same cache key (or vice versa). (4) Task implementation changes — updating a plugin version changes the bytecode hash. The local cache lives in ~/.gradle/caches/build-cache-1/; remote caches use HTTP (Gradle Enterprise / Develocity) or custom backends. Use --scan with Gradle Enterprise to visualize cache hit rates and diagnose misses per task type.

Q5. What is the configuration cache? How does it differ from the build cache?

The build cache stores task outputs — the results of executing individual tasks (compiled classes, packaged JARs). The configuration cache (introduced in Gradle 6.6, stable in 8.1) stores the entire task graph — the serialized result of the configuration phase. On subsequent builds, if no build scripts, plugin code, or relevant system properties have changed, Gradle skips the configuration phase entirely and deserializes the cached task graph directly into the execution phase.

The configuration cache is invalidated when: build scripts change, settings.gradle.kts changes, plugin implementations change, system properties or environment variables read during configuration change, or gradle.properties values change. It imposes constraints: build scripts cannot reference Project objects at execution time (only Provider/Property values), cannot use Task.project in task actions, and cannot store non-serializable objects in task state. These constraints enforce a clean separation between configuration and execution. The performance impact is dramatic — a 300-module project that takes 15 seconds to configure can start execution in under 1 second with a warm configuration cache. Enable it with org.gradle.configuration-cache=true in gradle.properties. AGP 7.4+ and most Jetpack plugins are fully compatible.

Q6. How does incremental compilation work in Kotlin? What is ABI change detection?

Kotlin's incremental compilation tracks dependencies between source files and recompiles only the files affected by a change. The Kotlin compiler maintains a build history (stored in build/kotlin/) that records which files depend on which symbols. When a file changes, the compiler checks whether the change affects the file's ABI (Application Binary Interface) — the set of public/internal declarations visible to other files (function signatures, class hierarchies, property types). If only the method body changes (implementation detail), only that file is recompiled. If a public signature changes (ABI change), all files that reference that symbol are recompiled transitively.

Cross-module incremental compilation (enabled by default since Kotlin 1.7) extends this across module boundaries using IC (Incremental Compilation) build history. When module A's ABI changes, only the files in module B that actually import changed symbols are recompiled — not the entire module B. The kotlin.incremental.useClasspathSnapshot=true flag (default in Kotlin 1.8.20+) enables classpath snapshots that make cross-module IC more precise by comparing the actual class ABI rather than file timestamps. A common pitfall: adding a public extension function in a utility module can trigger widespread recompilation because Kotlin extension functions are resolved at compile time and many files may import the receiver type. Keeping utility modules' public API surface small is critical for incremental build performance.

Q7. Compare kapt vs KSP — why is KSP faster and when should you migrate?

kapt (Kotlin Annotation Processing Tool) works by generating Java stubs from Kotlin source, then running standard Java annotation processors (JSR 269) against those stubs. This requires a full Kotlin → Java stub generation pass before annotation processing can begin, effectively doubling the front-end compilation cost. Stubs must represent every Kotlin construct in Java (including Kotlin-specific features like default parameters, inline classes, and extension functions), which is lossy and slow. kapt also doesn't support incremental processing well — most processors are "aggregating" type, meaning any source change triggers full reprocessing.

KSP (Kotlin Symbol Processing) operates directly on the Kotlin compiler's symbol resolution output — it sees the Kotlin AST natively without generating stubs. This eliminates the stub generation pass entirely (typically 25-40% of kapt time). KSP supports true incremental processing: processors declare whether they're "aggregating" or "isolating", and KSP only re-invokes the processor for changed files when possible. KSP also supports Kotlin Multiplatform (kapt is JVM-only). Migrate when: your annotation processors have KSP support (Hilt/Dagger since 2.48, Room since 2.50, Moshi since 1.13). Keep kapt only for processors without KSP support (some legacy Dagger components, AutoValue). In a large project, migrating from kapt to KSP typically reduces clean build time by 15-25% and incremental build time by even more due to better incrementality.

Q8. What is the Gradle Daemon? How does it improve build performance?

The Gradle Daemon is a long-lived background JVM process that persists between build invocations. Without it, every ./gradlew call would cold-start a JVM, load all Gradle classes, initialize the classloader hierarchy, and apply plugins from scratch. The Daemon eliminates this overhead by keeping the JVM warm — class loading, JIT compilation, and in-memory caches persist across builds. The first build with a cold Daemon is slower (JVM warmup), but subsequent builds benefit from JIT-optimized hot paths and cached classloaders.

The Daemon also maintains in-memory caches: the file system snapshot cache (avoids re-scanning unchanged directories), the dependency resolution cache (avoids re-resolving unchanged dependency graphs), and the compilation avoidance cache. Daemon lifecycle: it starts on the first build, stays alive for 3 hours of inactivity (configurable via org.gradle.daemon.idletimeout), and is invalidated when the Gradle version changes, JVM arguments change, or org.gradle.jvmargs changes. Multiple Daemons can coexist for different Gradle versions. On CI, Daemon benefits are reduced because builds often start with a cold Daemon — this is why CI builds should focus on build cache hits rather than Daemon warmth. Use --status to list running Daemons and --stop to kill them. The Daemon's heap size (org.gradle.jvmargs=-Xmx4g) directly impacts build performance for large projects — too small causes GC thrashing, too large wastes memory.

Q9. Explain AGP's variant system — how do build types, flavors, and dimensions combine?

AGP's variant system creates a Cartesian product of build types (debug, release) and product flavors grouped into flavor dimensions. If you have 2 build types, a "tier" dimension with {free, paid}, and an "environment" dimension with {staging, production}, you get 2 × 2 × 2 = 8 variants: freeStagingDebug, freeStagingRelease, freeProductionDebug, etc. Each variant gets its own source set (src/freeStaging/), its own dependency configuration (freeStagingImplementation), its own manifest merging, and its own compile/package tasks.

Internally, AGP represents variants as ApplicationVariant/LibraryVariant objects created during configuration. Each variant has a VariantProperties that exposes lazy Provider-based APIs for modifying outputs. The new Variant API (androidComponents { onVariants { } }) lets you filter, modify, or disable variants before task creation. This is critical for build performance — if you only need 4 of 8 variants, use beforeVariants { if (it.name.contains("staging")) it.enable = false } to skip task registration for unused variants entirely. Each variant generates ~40-60 tasks (compile, merge resources, merge manifests, package, lint, test), so eliminating 4 unused variants saves 160-240 tasks from the configuration phase. Flavor dimensions also control dependency resolution: a freeStagingImplementation dependency is only included in that specific variant's classpath.

Q10. What is the Provider/Property API in Gradle? Why is lazy evaluation important?

Provider<T> is a read-only lazy value that is computed only when get() is called. Property<T> extends Provider with mutability — you can set() a value or set(provider) to wire it to another lazy source. ListProperty, SetProperty, MapProperty, and RegularFileProperty/DirectoryProperty handle collections and file system locations. These are created via ObjectFactory (injected by Gradle) and form the backbone of Gradle's lazy evaluation model.

Lazy evaluation matters because it enables configuration avoidance: task inputs and outputs are wired as Provider chains during configuration, but actual values aren't resolved until execution. This means a task's output directory (a DirectoryProperty) can be wired as another task's input without either task being configured — Gradle resolves the chain only when the consuming task executes. Without lazy evaluation, you'd need to eagerly resolve all values during configuration, which forces task instantiation and breaks caching. The Provider API also enables implicit task dependencies: when you wire taskB.inputFile.set(taskA.outputFile), Gradle automatically infers that taskB depends on taskA — no explicit dependsOn needed. This eliminates a major source of build script bugs (forgotten dependsOn declarations). In convention plugins, always use Property types for task inputs/outputs and wire them with .set(provider) rather than .set(value) to preserve laziness.

Module Architecture (Q11–Q20)

Q11. What are the benefits of modularization? When does it NOT make sense?

Benefits: (1) Build performance — Gradle compiles modules in parallel and caches outputs independently; changing one module doesn't recompile the entire app. (2) Encapsulation — internal visibility enforces API boundaries; modules expose only what they intend. (3) Team scalability — teams own modules independently, reducing merge conflicts and enabling parallel development. (4) Reusability — shared modules (networking, analytics, design system) can be consumed by multiple apps or features. (5) Testability — modules can be tested in isolation with fast unit test cycles. (6) Dynamic delivery — feature modules can be delivered on-demand via Play Feature Delivery.

When it does NOT make sense: (1) Small apps with 1-3 developers — the overhead of managing module boundaries, build-logic plugins, and inter-module navigation exceeds the benefit. (2) Prototypes and MVPs — speed of iteration matters more than architectural purity. (3) When the domain boundaries are unclear — premature modularization creates artificial boundaries that need constant refactoring. (4) When the team lacks Gradle expertise — misconfigured multi-module builds can be slower than a well-structured monolith (e.g., using api everywhere instead of implementation, or missing configuration avoidance). Start with a well-structured single module using package-by-feature, and extract modules when you feel concrete pain: slow builds, merge conflicts, or the need for code sharing across apps.

Q12. Describe the recommended module types for a large Android app and their dependency rules.

Google's recommended module taxonomy (from Now in Android): :app — the application module; depends on all feature modules and wires the navigation graph. Contains the Application class, Hilt entry point, and top-level navigation. :feature:* — feature modules (e.g., :feature:home, :feature:search); each owns a screen or flow. Depends on :core:* modules but never on other feature modules. :core:data — repository implementations; depends on :core:network, :core:database, :core:model. :core:domain — use cases / business logic; depends only on :core:model and :core:data interfaces (via dependency inversion). :core:model — pure Kotlin data classes shared across layers; no Android dependencies. :core:network — Retrofit interfaces, API models, serialization. :core:database — Room DAOs, entities, migrations. :core:ui — shared Compose components, theme, design tokens. :core:common — utilities, extensions, dispatchers.

Dependency rules: feature modules are leaf nodes — they depend downward on core modules but never sideways on other features. Core modules form a layered DAG: :core:data → :core:network + :core:database → :core:model. The :app module is the only module that depends on all features. This prevents circular dependencies and ensures features can be built/tested independently.

Q13. Explain api vs implementation — how does the choice affect build performance?

implementation means the dependency is used internally by the module but not exposed to consumers on their compile classpath. api means the dependency is part of the module's public API and is exposed transitively to consumers. Concretely: if :core:data uses implementation("com.squareup.retrofit2:retrofit:2.9.0"), then :feature:home (which depends on :core:data) cannot see Retrofit classes at compile time. If api were used instead, :feature:home would see Retrofit on its compile classpath.

The build performance impact is significant: when a module's implementation dependency changes (e.g., Retrofit version bump), only that module is recompiled — consumers are unaffected because the dependency wasn't on their compile classpath. With api, the change propagates: every consumer must be recompiled to verify compatibility. In a deep dependency graph, api creates a recompilation cascade — changing a leaf library can trigger recompilation of dozens of modules. The rule of thumb: use implementation by default; only use api when the dependency's types appear in your module's public API (return types, parameter types, superclasses). In practice, 90%+ of dependencies should be implementation. The dependency-analysis-gradle-plugin can automatically detect misused api/implementation declarations and suggest corrections.

Q14. What is the dependency inversion principle in the context of Android modules? Show an example.

Dependency inversion means high-level modules (features, domain) should not depend on low-level modules (network, database) directly — both should depend on abstractions (interfaces) defined in a shared module. In practice: :core:domain defines a UserRepository interface. :core:data implements it as UserRepositoryImpl using Retrofit and Room. :feature:profile depends on :core:domain (the interface) but not on :core:data (the implementation). The wiring happens in the :app module via Hilt: @Binds abstract fun bindUserRepository(impl: UserRepositoryImpl): UserRepository.

This has concrete build benefits: when :core:data's implementation changes (new caching layer, different ORM), :feature:profile doesn't recompile — it only depends on the stable interface in :core:domain. It also enables testing: features can be tested with fake repositories without pulling in Retrofit/Room dependencies. The pattern extends to navigation: define a Navigator interface in a shared module, implement it in :app, and inject it into features. This is how feature modules navigate to each other without direct dependencies. At Google scale, dependency inversion is non-negotiable — without it, a 200-module project devolves into a tightly coupled graph where every change triggers cascading recompilation.

Q15. Compare package-by-feature vs package-by-layer — which do you recommend and why?

Package-by-layer groups code by technical role: com.app.ui, com.app.data, com.app.domain. Every feature's UI code lives in the same package. This breaks down at scale: the ui package grows to hundreds of files, feature boundaries are invisible, and developers working on different features constantly conflict in the same directories. It also makes modularization harder — extracting a feature requires cherry-picking files from every layer package.

Package-by-feature groups code by business capability: com.app.feature.search, com.app.feature.checkout. Each package contains its own UI, domain logic, and data access. This maps naturally to modules — each feature package can become a module with minimal refactoring. It improves discoverability (everything related to search is in one place), reduces merge conflicts (teams work in separate packages), and aligns with how product teams think (features, not layers).

The recommendation for L6 projects: hybrid — package-by-feature at the top level (each feature is a module), with internal layering within each feature module (ui/, domain/, data/ sub-packages). Shared infrastructure (networking, database, design system) lives in horizontal :core:* modules organized by layer. This gives you feature isolation for business logic and layer separation for shared infrastructure. Google's Now in Android follows exactly this pattern.

Q16. Describe Google's Now in Android module architecture — what patterns does it use?

Now in Android (NiA) uses a feature-first modular architecture with ~30 modules. The key patterns: (1) Feature modules (:feature:foryou, :feature:interests, :feature:bookmarks, :feature:topic, :feature:search) — each contains a single screen's Compose UI, ViewModel, and navigation definition. Features never depend on each other. (2) Core modules (:core:data, :core:domain, :core:model, :core:network, :core:database, :core:datastore, :core:ui, :core:designsystem, :core:common, :core:testing) — shared infrastructure with strict layering. (3) Convention plugins in a build-logic composite build — AndroidApplicationConventionPlugin, AndroidLibraryConventionPlugin, AndroidFeatureConventionPlugin, etc., applied via plugins { id("nowinandroid.android.feature") }.

Key architectural decisions: :core:model is pure Kotlin (no Android dependencies), enabling it to be shared with KMP targets. :core:domain contains use cases that combine multiple repositories. The :app module owns the top-level navigation graph (using Navigation Compose) and is the Hilt composition root. :core:testing provides shared test utilities, fake repositories, and test dispatchers. Version catalogs (libs.versions.toml) centralize all dependency versions. The project uses nonTransitiveRClass for faster R class generation. NiA is the reference architecture Google recommends for production apps — it demonstrates every best practice in one cohesive project.

Q17. How do you determine the right module granularity? What are the signs of over-modularization?

Right granularity balances build parallelism (more modules = more parallel compilation) against configuration overhead (more modules = longer configuration phase) and cognitive load (more modules = harder to navigate). A good heuristic: one module per feature team's ownership boundary, plus shared modules for cross-cutting concerns. If a feature has 3-5 screens with shared state, it's one module. If those screens are independently deployable or owned by different teams, split them.

Signs of over-modularization: (1) Modules with fewer than 5-10 source files — the Gradle overhead (build script, plugin application, task registration) exceeds the benefit. (2) Excessive api dependencies — if most of a module's types are re-exported, the module isn't providing meaningful encapsulation. (3) Circular dependency pressure — if you constantly need module A to call module B and vice versa, they should probably be one module. (4) Configuration phase dominates build time — if ./gradlew help takes 20 seconds in a 500-module project, you've over-split. (5) Developers spend more time managing module boundaries (inter-module navigation, Hilt wiring, build script maintenance) than writing features. Signs of under-modularization: build times scale linearly with any change, merge conflicts are frequent, and internal visibility isn't providing encapsulation because everything is in one module. The sweet spot for most apps: 20-80 modules.

Q18. What is the api/impl module split pattern? When should you use it?

The api/impl split creates two modules for a single concern: :core:network-api (contains interfaces, data classes, and the public contract) and :core:network-impl (contains the implementation — Retrofit setup, interceptors, serialization). Consumer modules depend only on :core:network-api. The :app module (or a DI wiring module) depends on both and binds the implementation to the interface via Hilt.

Benefits: (1) Build performance — when the implementation changes (new interceptor, different HTTP client), only :core:network-impl recompiles. All consumers depend on the stable API module and are unaffected. (2) Strict encapsulation — implementation details (OkHttp, Retrofit, Moshi) are invisible to consumers; you can swap HTTP clients without touching any feature module. (3) Faster tests — feature module tests depend on :core:network-api and use fakes, never pulling in the real networking stack. Use this pattern for: infrastructure modules with complex implementations that change frequently (networking, database, analytics, auth). Don't use it for: simple modules where the interface and implementation are trivially coupled (a utility module with pure functions), or when you only have one consumer — the indirection adds complexity without benefit. At Google scale, the api/impl split is standard for all platform-level modules.

Q19. How do you enforce module boundaries? What tools and techniques are available?

Enforcement happens at multiple levels: (1) Kotlin internal visibility — classes and functions marked internal are invisible outside the module at compile time. This is the first line of defense. (2) Dependency constraints — Gradle's dependency configurations (implementation vs api) control what's on the compile classpath. A feature module simply cannot reference classes from another feature module if there's no dependency declared. (3) Architectural lint rules — custom Lint checks or tools like dependency-analysis-gradle-plugin can enforce rules like "feature modules must not depend on other feature modules" or "no Android dependencies in :core:model".

(4) Module graph validation — plugins like module-graph-assert let you declare allowed/forbidden dependency patterns: assert { module(":feature:.*") { shouldNotDependOn(":feature:.*") } }. (5) Convention plugins — by applying a nowinandroid.android.feature plugin that pre-configures allowed dependencies, you prevent ad-hoc dependency additions. (6) CI checks — run ./gradlew buildHealth (from dependency-analysis plugin) in CI to catch api/implementation misuse, unused dependencies, and boundary violations. (7) Code review discipline — CODEOWNERS files ensure module owners review any dependency changes. The most effective combination: internal visibility + strict implementation usage + CI-enforced module graph assertions. Without enforcement, module boundaries erode within months as developers add expedient cross-module dependencies.

Q20. How does internal visibility in Kotlin help with module encapsulation?

Kotlin's internal modifier restricts visibility to the same module — where "module" means a Gradle source set compiled together (a single :feature:search module). Any class, function, or property marked internal is accessible within the module but invisible to any module that depends on it. This is enforced at compile time by the Kotlin compiler — attempting to reference an internal symbol from another module produces a compilation error.

This is fundamentally more powerful than Java's package-private visibility, which only restricts access within a package (and packages can span modules). With internal, you can expose a clean public API from your module (interfaces, data classes, entry-point composables) while hiding all implementation details (repository implementations, mappers, internal state machines, helper functions). A practical pattern: in :feature:search, the SearchScreen composable and SearchRoute navigation definition are public, while the SearchViewModel, SearchRepository, and all internal state classes are internal. This means the :app module can wire navigation to SearchRoute but cannot accidentally couple to internal implementation details. The limitation: internal compiles to public in JVM bytecode (with name mangling) — reflection and Java code can bypass it. But for compile-time safety within a Gradle build, it's the strongest encapsulation tool available in Kotlin.

Modularization Strategies (Q21–Q28)

Q21. Walk through a monolith-to-modular migration — what are the phases and what do you extract first?

Phase 1 — Foundation: Set up build infrastructure first. Create a build-logic composite build with convention plugins, adopt version catalogs, and enable nonTransitiveRClass. This phase has zero architectural impact but establishes the tooling needed for modularization. Phase 2 — Extract leaf modules: Start with modules that have no upstream dependencies: :core:model (data classes), :core:common (utilities, extensions), :core:ui (shared Compose components, theme). These are easy wins — they break no existing dependencies and immediately enable parallel compilation.

Phase 3 — Extract infrastructure: Pull out :core:network, :core:database, :core:datastore. These require defining clean interfaces (repository pattern) and moving implementations behind them. This is where dependency inversion becomes critical. Phase 4 — Extract features: Starting with the most independent feature (fewest cross-feature dependencies), create :feature:* modules. Each extraction involves: moving source files, updating imports, replacing direct navigation with interface-based navigation, and wiring Hilt bindings. Phase 5 — Iterate: Extract remaining features one by one, resolving circular dependencies as they surface. The key principle: always extract bottom-up (leaves first, then mid-level, then features). Never try to extract a feature that depends on code still in the monolith — you'll create circular dependencies between the monolith and the new module.

Q22. Compare by-layer, by-feature, and hybrid modularization — trade-offs of each.

By-layer creates modules like :ui, :domain, :data, :network. Pros: simple structure, clear separation of concerns. Cons: poor build parallelism (changing a data class in :domain recompiles all of :ui), poor team scalability (every feature team touches every layer module), and the :ui module becomes a monolith itself. By-layer modularization provides almost no build performance benefit because the dependency chain is serial: :ui → :domain → :data → :network.

By-feature creates modules like :search, :checkout, :profile, each containing their own UI, domain, and data layers. Pros: excellent build parallelism (features compile independently), strong team ownership, natural code locality. Cons: code duplication across features (each might implement its own networking), harder to share infrastructure, and cross-feature interactions require careful design (navigation, shared state).

Hybrid (recommended) combines both: feature modules for business capabilities, core/infrastructure modules for shared technical concerns. :feature:search depends on :core:data, :core:ui, :core:model. This gives you feature-level parallelism with infrastructure reuse. The trade-off: more modules to manage and more complex dependency graph. But at Google scale, the hybrid approach is the only one that scales to 100+ modules with 20+ teams.

Q23. What is vertical vs horizontal slicing? When do you use each?

Vertical slicing cuts through all layers for a single feature: :feature:search contains its UI (Compose screens), ViewModel, domain logic (use cases), and data access (repository interface + implementation). The module is self-contained — it owns everything from the user interaction down to the data source. Horizontal slicing cuts across features at a single layer: :core:network contains all networking code for every feature, :core:database contains all persistence code.

Use vertical slicing for feature modules — this maximizes team autonomy, build parallelism, and encapsulation. A team can modify their entire feature stack without touching other modules. Use horizontal slicing for infrastructure modules — networking, database, analytics, auth, and design system are cross-cutting concerns that shouldn't be duplicated in every feature. The hybrid architecture is vertical features on top of horizontal infrastructure. A nuance: within a vertical feature module, you can still have internal layering (ui/, domain/, data/ packages) — the vertical slice is at the module level, the horizontal layers are at the package level within the module. For very large features (e.g., a checkout flow with 15 screens), you might further vertically slice into sub-features: :feature:checkout:cart, :feature:checkout:payment, :feature:checkout:confirmation.

Q24. What are the common dependency graph anti-patterns? How do you fix them?

(1) God module: One module (usually :app or :core:common) that everything depends on and that depends on everything. Fix: break it into focused modules; :core:common should contain only pure utilities with zero dependencies. (2) Circular dependencies: :feature:a → :feature:b → :feature:a. Gradle rejects this at configuration time. Fix: extract the shared code into a :core:* module, or use dependency inversion with interfaces. (3) Leaky abstractions via api: Using api everywhere creates implicit transitive dependencies — changing a leaf library triggers cascading recompilation. Fix: audit with dependency-analysis-gradle-plugin and switch to implementation.

(4) Diamond dependency with version conflicts: Module A depends on OkHttp 4.9 via :core:network, and also on OkHttp 4.12 via a third-party SDK. Gradle silects 4.12 silently, potentially breaking :core:network. Fix: use BOM or enforcedPlatform() to pin versions, and run dependencyInsight in CI. (5) Feature-to-feature dependencies: :feature:profile depends on :feature:settings because it needs a shared data class. Fix: move the shared type to :core:model. (6) Deep dependency chains: A → B → C → D → E means changing E recompiles everything. Fix: flatten the graph by having A depend directly on E (if appropriate), or use the api/impl split to insulate consumers from implementation changes.

Q25. When should you NOT modularize? Give concrete examples.

(1) Small teams (1-3 developers): The overhead of managing module boundaries, convention plugins, inter-module navigation, and Hilt wiring across modules exceeds the benefit. A well-structured single module with package-by-feature is simpler and builds fast enough. (2) Rapid prototyping: When validating a product idea, architectural purity is a liability. Ship the monolith, validate the concept, then modularize when the product stabilizes. (3) Unclear domain boundaries: If you don't know where features begin and end, premature modularization creates artificial boundaries that need constant refactoring. Let the domain model stabilize first.

(4) Build times are already acceptable: If your monolith builds incrementally in 15 seconds, modularization for build speed is premature optimization. (5) Tightly coupled features: If two "features" share 80% of their state and UI components, forcing them into separate modules creates excessive cross-module communication and boilerplate. They're one feature. (6) Library/SDK projects: A library published as a single artifact doesn't benefit from internal modularization — consumers see one dependency regardless. Internal modularization only helps if you publish multiple artifacts. The general principle: modularize when you feel concrete pain (slow builds, merge conflicts, inability to test in isolation, need for code sharing across apps), not because a blog post said you should.

Q26. What are the most common modularization mistakes? How do you avoid them?

(1) Modularizing by layer instead of feature: Creating :ui, :domain, :data modules gives almost no build parallelism — the dependency chain is serial. Fix: modularize by feature with shared core modules. (2) Using api everywhere: Developers default to api because implementation causes "unresolved reference" errors in consumers. This defeats the purpose of modularization by making every change cascade. Fix: use implementation by default, expose only what consumers need. (3) Putting everything in :core:common: This becomes a god module that every module depends on, destroying build parallelism. Fix: split into focused modules (:core:model, :core:network, :core:ui).

(4) Not investing in build infrastructure: Without convention plugins and version catalogs, each module's build.gradle.kts is a copy-pasted mess that drifts over time. Fix: create convention plugins from day one. (5) Over-modularizing: Creating 300 modules for a 10-person team means more time managing Gradle than writing features. Fix: start with 10-20 modules and split when needed. (6) Ignoring the configuration phase: More modules means more build scripts to evaluate. Without configuration cache and task avoidance, configuration time can exceed compilation time. Fix: enable configuration cache, use tasks.register(), filter unused variants. (7) Circular dependency hacks: Using reflection or runtime service loaders to work around circular dependencies instead of fixing the architecture. Fix: extract shared code or use dependency inversion.

Q27. How do dynamic feature modules work? What are their limitations?

Dynamic feature modules (DFMs) use Play Feature Delivery to download module code on-demand rather than including it in the base APK. They're declared with com.android.dynamic-feature plugin and have an inverted dependency: the DFM depends on :app (the base module), not the other way around. At build time, AGP packages each DFM as a separate split APK. At runtime, SplitInstallManager downloads and installs the split, and SplitCompatApplication loads the new classloader. Navigation to DFM code uses reflection or the Play Core SplitInstallHelper.

Limitations: (1) Inverted dependency direction — the base module cannot reference DFM classes at compile time, requiring reflection or service-loader patterns for communication. (2) Testing complexity — DFMs can't be tested with standard instrumentation tests; you need the bundletool fake install flow. (3) Hilt incompatibility — Hilt's component hierarchy assumes all modules are available at compile time; DFMs require @EntryPoint interfaces to access Hilt-provided dependencies from the base module. (4) Resource conflicts — DFM resources can clash with base module resources; namespace them carefully. (5) User experience — downloading a module on first access adds latency; you need loading UI and error handling for network failures. (6) Play Store only — DFMs don't work on alternative app stores or sideloaded APKs. (7) Size threshold — Google recommends DFMs only for features > 5MB; smaller features aren't worth the complexity. Most teams find that App Bundles (automatic density/ABI/language splits) provide sufficient size reduction without DFM complexity.

Q28. Design the module architecture for a super-app (e-commerce + payments + social + messaging).

A super-app requires a platform layer that multiple product verticals build on. Architecture: Platform modules: :platform:auth (login, session, tokens), :platform:network (HTTP client, interceptors, certificate pinning), :platform:analytics (event tracking, A/B testing), :platform:design-system (shared UI components, theme, typography), :platform:navigation (deep link registry, inter-feature navigation interfaces), :platform:storage (encrypted preferences, database), :platform:push (FCM, notification routing). These are owned by a platform team and versioned independently.

Product verticals: Each vertical is a group of feature modules: :ecommerce:catalog, :ecommerce:cart, :ecommerce:checkout, :ecommerce:orders; :payments:wallet, :payments:transfer, :payments:history; :social:feed, :social:profile, :social:groups; :messaging:chat, :messaging:calls. Each vertical has its own :*:core:data and :*:core:domain modules. Verticals depend on platform modules but never on each other — cross-vertical communication goes through :platform:navigation interfaces (deep links or a shared event bus).

The :app module is thin: it wires the navigation graph, initializes the platform, and provides the Hilt composition root. Each vertical can optionally be a dynamic feature module for on-demand delivery. Convention plugins enforce the rules: :ecommerce:* modules can depend on :platform:* but not on :social:*. This architecture supports 50+ developers across 8+ teams, with each team able to build, test, and release their vertical independently. Total module count: 40-80 modules, with the build-logic module providing 6-8 convention plugins for consistent configuration.

Dependency Management (Q29–Q36)

Q29. Explain version catalogs — how do they work and what problems do they solve?

Version catalogs (gradle/libs.versions.toml) provide a centralized, type-safe way to declare dependency coordinates and versions. The TOML file has four sections: [versions] (version strings), [libraries] (dependency coordinates referencing versions), [bundles] (groups of libraries applied together), and [plugins] (plugin coordinates). Gradle generates a type-safe accessor from the catalog: libs.versions.toml entry retrofit = { group = "com.squareup.retrofit2", name = "retrofit", version.ref = "retrofit" } becomes libs.retrofit in build scripts.

Problems solved: (1) Version drift — without a catalog, the same library can be declared at different versions in different modules' build scripts. The catalog is the single source of truth. (2) Discoverability — IDE autocompletion on libs. shows all available dependencies; no more searching build files for the right coordinate string. (3) Bundle management — bundles group related dependencies (e.g., compose = ["compose-ui", "compose-material3", "compose-tooling"]) so modules can add the full Compose stack with implementation(libs.bundles.compose). (4) Convention plugin integration — convention plugins can reference the catalog to apply standard dependencies. Version catalogs replaced the old patterns of ext { } blocks in root build.gradle and buildSrc constants. They're Gradle-native (no plugin needed since Gradle 7.0), support version ranges and rich constraints, and are shareable across projects via from(files("../shared-catalog.toml")) or published as a Gradle platform.

Q30. What is a BOM? How does platform() differ from enforcedPlatform()?

A BOM (Bill of Materials) is a special POM file that declares a set of dependency versions without providing any code. It acts as a version-alignment mechanism: when you import a BOM, all dependencies it declares use the BOM's versions unless explicitly overridden. In Gradle, you import a BOM with implementation(platform("androidx.compose:compose-bom:2024.01.00")), then declare individual dependencies without versions: implementation("androidx.compose.ui:ui") — the BOM supplies the version.

platform() imports the BOM as a recommendation — it sets default versions, but a direct version declaration in your build script or a higher transitive version wins. enforcedPlatform() imports the BOM as a constraint — it forces the BOM's versions even if a transitive dependency requests a higher version. This is equivalent to applying strictly constraints for every dependency in the BOM. Use platform() for library BOMs where you want alignment but allow overrides (Compose BOM, Firebase BOM). Use enforcedPlatform() when you need strict version pinning across the entire project — typically in a root build script to enforce organizational version policies. Be cautious with enforcedPlatform(): it can downgrade transitive dependencies, potentially causing runtime NoSuchMethodError if a library requires a newer version than the BOM specifies.

Q31. How do you handle transitive dependency conflicts? Walk through a diamond dependency scenario.

Diamond dependency: :app depends on :feature:a and :feature:b. :feature:a depends on :core:network which uses OkHttp 4.9. :feature:b depends on a third-party SDK that bundles OkHttp 4.12. Both OkHttp versions end up in the resolution graph. Gradle's default strategy selects the highest version (4.12). If OkHttp 4.12 is backward-compatible, this works. If it removed an API that :core:network uses, you get a runtime NoSuchMethodError.

Resolution strategies: (1) Upgrade the lower version — update :core:network to use OkHttp 4.12. This is the cleanest fix. (2) Force a version — configurations.all { resolutionStrategy { force("com.squareup.okhttp3:okhttp:4.9.3") } }. This pins the version but may break the third-party SDK. (3) Use strictly — implementation("com.squareup.okhttp3:okhttp") { version { strictly("4.9.3") } }. This fails the build if any transitive dependency requests a different version, making the conflict visible. (4) Exclude the transitive — implementation("com.thirdparty:sdk:1.0") { exclude(group = "com.squareup.okhttp3") }. Risky — the SDK may not work with the excluded dependency's replacement. (5) Use a BOM — align all OkHttp-related dependencies via enforcedPlatform(). Best practice: run ./gradlew :app:dependencies --configuration runtimeClasspath regularly in CI and fail on unexpected version upgrades using dependency locking.

Q32. What is dependency locking? Why is it important for production apps?

Dependency locking records the exact resolved versions of all dependencies (including transitives) in a lock file. Without locking, a build on Monday might resolve okhttp:4.12.0 and a build on Tuesday might resolve okhttp:4.12.1 (if a new version was published and your declaration uses a range or dynamic version). This makes builds non-reproducible — a bug might appear in Tuesday's build that didn't exist Monday, with no code change.

Enable locking per configuration: dependencyLocking { lockAllConfigurations() }. Generate lock files with ./gradlew dependencies --write-locks. Gradle creates gradle/dependency-locks/*.lockfile per configuration, listing every resolved coordinate. On subsequent builds, Gradle verifies that resolution matches the lock file — if a new transitive dependency appears or a version changes, the build fails with a clear error. You must explicitly run --write-locks to update. This is critical for: (1) Reproducible builds — the same commit always produces the same binary. (2) Security — prevents supply-chain attacks where a malicious library version is silently pulled in via transitive resolution. (3) Debugging — when a regression appears, you can diff lock files between commits to see exactly which dependency changed. Combine with dependencyInsight and Dependabot/Renovate for automated, auditable dependency updates.

Q33. How does the dependency-analysis plugin help? What issues does it detect?

The dependency-analysis-gradle-plugin (by Autonomousapps) performs bytecode analysis to determine actual dependency usage across your project. It detects: (1) Unused dependencies — declared in build.gradle.kts but no class from the dependency is referenced in source code. These add to the classpath, slow resolution, and increase APK size. (2) Used transitive dependencies — your code references a class from a library that isn't directly declared, only available transitively. This is fragile — if the intermediate dependency changes its transitives, your code breaks. (3) api vs implementation misuse — a dependency declared as api whose types never appear in the module's public API (should be implementation), or vice versa.

(4) Unused annotation processors — kapt/KSP processors declared but not generating any code. (5) Redundant plugins — plugins applied but not contributing to the build. Run ./gradlew buildHealth for a project-wide report, or ./gradlew :feature:search:projectHealth for a single module. The plugin outputs actionable advice: "Remove unused dependency: com.google.code.gson:gson", "Change to implementation: androidx.core:core-ktx (not part of ABI)". Integrating this into CI prevents dependency hygiene from degrading over time. In a 100-module project, the plugin typically finds 50-100 dependency issues on first run — fixing them measurably improves build times by reducing unnecessary compilation and classpath sizes.

Q34. Explain rich version constraints — strictly, require, prefer, reject.

Gradle's rich version constraints (available in Kotlin DSL and Gradle Module Metadata) give fine-grained control over version resolution: require (the default) — specifies the minimum acceptable version; Gradle can upgrade to a higher version if another dependency requests it. implementation("com.squareup.okhttp3:okhttp") { version { require("4.9.0") } } means "4.9.0 or higher." strictly — pins the exact version (or range) and fails the build if any transitive dependency requests an incompatible version. It's the strongest constraint — it rejects upgrades. Use it to enforce organizational version policies or prevent known-broken versions.

prefer — a soft preference that's used only if no other constraint specifies a version. It's the weakest constraint — any require or strictly from another module overrides it. Useful for suggesting a default version in a library that consumers can easily override. reject — explicitly excludes specific versions: version { require("[4.9, 5.0)"); reject("4.10.0") } means "any 4.x from 4.9 up, except 4.10.0." Use it to blacklist versions with known vulnerabilities or bugs. These constraints are published in Gradle Module Metadata (GMM), so they propagate to consumers — if your library publishes strictly("2.0"), consumers who transitively pull in your library will also be constrained. Maven POMs don't support rich constraints, which is why GMM is a significant improvement for dependency management.

Q35. How do you manage dependencies in a Kotlin Multiplatform module?

KMP modules declare dependencies in source set-specific blocks within kotlin { sourceSets { } }. commonMain dependencies are available on all platforms: commonMain.dependencies { implementation("org.jetbrains.kotlinx:kotlinx-coroutines-core:1.8.0") }. Platform-specific source sets (androidMain, iosMain, jvmMain) add platform-specific dependencies: androidMain.dependencies { implementation("io.ktor:ktor-client-okhttp:2.3.0") }, iosMain.dependencies { implementation("io.ktor:ktor-client-darwin:2.3.0") }.

Gradle resolves KMP dependencies using variant-aware resolution: when a KMP library publishes Gradle Module Metadata, it declares platform-specific variants. Gradle selects the correct variant based on the consuming source set's platform attributes. For example, kotlinx-coroutines-core publishes JVM, JS, and Native variants — Gradle automatically picks the JVM variant for androidMain and the Native variant for iosMain. Challenges: (1) Not all libraries publish KMP-compatible metadata — you may need expect/actual declarations with platform-specific implementations. (2) Version alignment across platforms is tricky — use a BOM or version catalog to keep Ktor/Serialization/Coroutines versions synchronized. (3) The androidMain source set can use Android-specific APIs (Context, SharedPreferences) but commonMain cannot — use expect/actual or dependency injection to abstract platform APIs. (4) Test dependencies follow the same pattern: commonTest for shared tests, androidUnitTest/iosTest for platform-specific tests.

Q36. What is the difference between classpath and implementation in buildscript {}?

classpath in the buildscript { dependencies { } } block adds dependencies to the build script's classloader — these are Gradle plugins and their dependencies needed to configure the build. classpath("com.android.tools.build:gradle:8.2.0") makes AGP available to apply in build scripts. implementation in a module's dependencies { } block adds dependencies to the project's compile and runtime classpath — these are libraries your application code uses.

They operate in completely different classloader hierarchies: build script dependencies are loaded by Gradle's build classloader and are available during the configuration phase. Project dependencies are resolved by Gradle's dependency engine and passed to the Kotlin/Java compiler during the execution phase. In modern Gradle (with the plugins { } DSL), you rarely need buildscript { } at all — plugins are declared as plugins { id("com.android.application") version "8.2.0" } and resolved from the pluginManagement { repositories { } } block in settings.gradle.kts. The buildscript { } block is legacy — it's still needed when a plugin doesn't publish a plugin marker artifact (required for the plugins { } DSL) or when you need to add a dependency to the build classpath that isn't a plugin (e.g., a custom task class from a JAR). Mixing up classpath and implementation is a common beginner mistake — adding a library to classpath makes it available in build scripts but not in application code.

Build Performance (Q37–Q44)

Q37. Your build takes 8 minutes — walk through your optimization strategy.

Step 1 — Profile: Run ./gradlew assembleDebug --scan and analyze the Gradle Enterprise build scan (or use --profile for a local HTML report). Identify the longest tasks, the configuration phase duration, and cache hit rates. Step 2 — Configuration phase: If configuration takes > 5 seconds, enable the configuration cache (org.gradle.configuration-cache=true), ensure all tasks use register() not create(), and filter unused build variants with beforeVariants { }. Remove allprojects { }/subprojects { } blocks that eagerly configure all modules.

Step 3 — Compilation: Migrate kapt to KSP (15-25% improvement). Enable Kotlin incremental compilation and cross-module IC. Check for ABI-breaking changes in frequently modified modules — consider the api/impl split for hot modules. Step 4 — Caching: Enable local build cache (org.gradle.caching=true). Audit cache miss rates per task type — fix non-cacheable tasks (absolute paths, non-deterministic outputs). Set up remote cache for CI-to-developer cache sharing. Step 5 — Parallelism: Enable org.gradle.parallel=true. Increase worker count: org.gradle.workers.max=8 (match CPU cores). Ensure modules don't have unnecessary dependencies that serialize the task graph. Step 6 — Resource optimization: Enable nonTransitiveRClass, use resourcePrefix to avoid conflicts, and enable PNG crunching only for release builds. Step 7 — Daemon tuning: Set org.gradle.jvmargs=-Xmx4g -XX:+UseG1GC to avoid GC pressure. After these steps, an 8-minute build typically drops to 2-3 minutes for clean builds and under 30 seconds for incremental builds.

Q38. How does parallel execution work in Gradle? What determines which tasks run in parallel?

With org.gradle.parallel=true, Gradle executes tasks from different projects in parallel using a worker thread pool (sized by org.gradle.workers.max, defaulting to CPU core count). Tasks within the same project execute sequentially by default (to avoid concurrent modification of shared project state). The task scheduler uses the DAG to determine which tasks have no unfinished dependencies and can run immediately — it's a topological sort with parallel dispatch.

What determines parallelism: (1) Module structure — independent modules (no dependency relationship) can compile in parallel. A flat dependency graph (many features depending on few core modules) enables more parallelism than a deep chain. (2) Task dependencies — :feature:a:compileKotlin and :feature:b:compileKotlin run in parallel if neither depends on the other. But both must wait for :core:model:compileKotlin if they depend on :core:model. (3) Worker API — tasks that use Gradle's Worker API can execute work items in parallel even within a single project. AGP uses this for resource merging, AAPT2 compilation, and dexing. (4) Shared output locks — tasks that write to the same output directory are serialized. The critical path (longest sequential chain of dependent tasks) determines the minimum possible build time regardless of parallelism. Use --scan to visualize the critical path and optimize the modules on it.

Q39. Explain the difference between local and remote build cache. How do you set up remote caching?

The local build cache stores task outputs on the developer's machine (~/.gradle/caches/build-cache-1/). It benefits repeated builds on the same machine — if you build, switch branches, then switch back, cached outputs from the first build are restored. The remote build cache stores outputs on a shared server (HTTP backend). It benefits the entire team: when developer A builds :core:network, the output is pushed to the remote cache. When developer B builds the same module with the same inputs, the output is pulled from the remote cache instead of recompiling. CI builds populate the cache; developer builds consume it.

Setup: In settings.gradle.kts: buildCache { remote<HttpBuildCache> { url = uri("https://cache.mycompany.com/cache/") push = System.getenv("CI") == "true" credentials { username = "..."; password = "..." } } }. Key decisions: (1) Only CI pushes to the remote cache (developers pull only) — this prevents cache poisoning from local modifications. (2) Use Gradle Enterprise / Develocity for the cache backend — it provides cache inspection, hit rate analytics, and access control. Alternatively, use a simple HTTP server (Nginx + filesystem) or cloud storage (GCS, S3) with the appropriate Gradle cache connector. (3) Cache key normalization is critical — ensure all tasks use relative paths (@PathSensitive(RELATIVE)) and don't embed machine-specific values. A well-configured remote cache typically achieves 70-90% hit rates on incremental CI builds and saves 40-60% of developer build time.

Q40. What is configuration avoidance? How do you achieve it?

Configuration avoidance means deferring the creation and configuration of Gradle objects (tasks, configurations, dependencies) until they're actually needed during execution. Without it, the configuration phase eagerly instantiates every task, evaluates every dependency block, and resolves every configuration — even for tasks that won't run. In a 200-module project, this can mean creating 10,000+ task objects just to run ./gradlew :app:assembleDebug.

How to achieve it: (1) Use tasks.register() instead of tasks.create() — defers task instantiation. (2) Use tasks.named() instead of tasks.getByName() — returns a TaskProvider without forcing instantiation. (3) Use Provider/Property API for task inputs/outputs — values are resolved lazily at execution time. (4) Avoid allprojects { }/subprojects { } — these eagerly configure every project. Use convention plugins instead, which are applied per-module. (5) Use the new AGP Variant API (androidComponents { onVariants { } }) instead of the legacy applicationVariants.all { } — the new API is lazy. (6) Filter unused variants with beforeVariants { if (...) it.enable = false } — prevents task registration for variants you'll never build. (7) Enable configuration cache — skips the entire configuration phase on cache hits. Measure with ./gradlew --dry-run assembleDebug — this runs only the configuration phase. If it takes more than 3-5 seconds for a 100-module project, you have configuration avoidance issues.

Q41. How do you optimize CI builds? What are the key techniques?

(1) Remote build cache: CI populates the cache; subsequent CI runs and developer builds consume it. This is the single highest-impact optimization — a well-configured remote cache can skip 60-80% of task execution on incremental CI builds. (2) Parallelism: CI machines typically have 8-16 cores — set org.gradle.workers.max to match. Enable org.gradle.parallel=true. (3) Gradle Daemon: On CI, the Daemon provides less benefit (cold start each run), but keeping it alive across pipeline stages (build → test → lint) avoids JVM restart overhead. (4) Dependency caching: Cache ~/.gradle/caches/ and ~/.gradle/wrapper/ between CI runs to avoid re-downloading dependencies.

(5) Selective testing: Use affected-module-detector or similar tools to run tests only in modules affected by the current changeset. In a 100-module project, a change to :feature:search only needs to test :feature:search and :app, not all 100 modules. (6) Test sharding: Split instrumented tests across multiple CI runners using -Pandroid.testInstrumentationRunnerArguments.numShards=4. (7) Avoid clean builds: Never run ./gradlew clean on CI unless necessary — it invalidates the local cache. Use --build-cache to leverage cached outputs. (8) Build only what's needed: For PR checks, run assembleDebug + testDebugUnitTest + lintDebug — skip release builds and instrumented tests (run those on merge to main). (9) Configuration cache: Serialize the task graph once and reuse across CI runs with the same build scripts. These techniques combined can reduce a 20-minute CI build to 5-7 minutes.

Q42. What causes a task to be not cacheable? How do you fix it?

A task is not cacheable when: (1) Not annotated as cacheable: The task class doesn't have @CacheableTask. Custom tasks must opt in explicitly. (2) Non-deterministic outputs: The task produces different outputs for the same inputs — timestamps in generated files, random UUIDs, build-time-dependent values. The cache stores outputs keyed by inputs; if the same inputs produce different outputs, cache entries are unreliable. (3) Absolute path inputs: If a task input includes an absolute path (/Users/alice/project/src/Main.kt), the cache key differs between machines. Fix with @PathSensitive(PathSensitivity.RELATIVE) to normalize paths.

(4) Undeclared inputs: The task reads files or system properties not declared as @Input/@InputFiles. The cache key doesn't include these, so the cached output may be stale. (5) Overlapping outputs: Two tasks write to the same output directory — Gradle can't cache them independently. Fix by giving each task a unique output directory. (6) Volatile inputs: Inputs that change every build (Git SHA, build timestamp, CI build number) cause cache misses on every run. Fix by moving volatile values to a separate, non-cacheable task and wiring the output to consumers. Diagnose with ./gradlew assembleDebug --scan — the build scan shows cache miss reasons per task. The -Dorg.gradle.caching.debug=true flag logs cache key computation details. Common offenders: custom code generation tasks, BuildConfig fields with timestamps, and annotation processors that embed absolute paths in generated code.

Q43. How does nonTransitiveRClass improve build performance?

By default, each module's R class contains resource IDs from the module itself and all its transitive dependencies. In a deep dependency graph, :app's R class contains every resource ID from every module — potentially thousands of fields. When any module adds or removes a resource, every downstream module's R class must be regenerated, causing cascading recompilation. With android.nonTransitiveRClass=true (in gradle.properties), each module's R class contains only its own resources. To reference a resource from a dependency, you use the dependency's R class: com.myapp.core.ui.R.drawable.icon instead of just R.drawable.icon.

The build performance impact: (1) Smaller R classes — fewer fields to generate, compile, and dex. (2) Reduced cascading recompilation — adding a resource to :core:ui only regenerates :core:ui's R class, not every consumer's. (3) Faster incremental builds — resource changes in one module don't trigger recompilation in unrelated modules. The trade-off: you must fully qualify resource references from dependencies, which is more verbose. IDE support handles this well — Android Studio auto-imports the correct R class. NiA and most Google sample projects enable this by default. In a 100-module project, enabling nonTransitiveRClass can reduce incremental build times by 10-20% for resource-heavy changes. AGP 8.0+ makes this the default for new projects.

Q44. What is the impact of allprojects {} / subprojects {} on build performance?

allprojects { } and subprojects { } in the root build.gradle.kts eagerly configure every project in the build during the configuration phase. Any code inside these blocks — repository declarations, dependency configurations, plugin applications, task registrations — runs for all modules regardless of whether they're needed for the current build target. In a 200-module project, this means 200 evaluations of the block's contents.

The problems: (1) Configuration time: Even ./gradlew :feature:search:assembleDebug configures all 200 modules because subprojects { } forces it. (2) Configuration cache incompatibility: allprojects { }/subprojects { } access the Project object of other projects, which violates configuration cache isolation requirements. (3) Coupling: All modules share the same configuration, making it impossible to customize individual modules without if (project.name == "...") { } hacks. The fix: replace allprojects { }/subprojects { } with convention plugins. Instead of subprojects { apply(plugin = "kotlin-android") }, create a convention plugin that each module applies explicitly: plugins { id("myapp.android.library") }. This is lazier (only applied to modules that need it), configuration-cache compatible, and more maintainable. For repository declarations, use dependencyResolutionManagement { repositories { } } in settings.gradle.kts — this is the modern, centralized approach that doesn't require allprojects { }.

Convention Plugins & Build Logic (Q45–Q52)

Q45. Why is buildSrc an anti-pattern for large projects? What should you use instead?

buildSrc is a special Gradle directory whose code is compiled and added to the build classpath of every module. The problem: any change to buildSrc invalidates the entire build. Gradle treats buildSrc as an input to every project's configuration — changing a single constant in buildSrc/src/main/kotlin/Versions.kt forces reconfiguration and recompilation of all modules. In a 200-module project, this means a full clean build for a one-line version bump. This also invalidates the configuration cache.

The replacement: a composite build with a build-logic directory. In settings.gradle.kts: pluginManagement { includeBuild("build-logic") }. The build-logic directory is a standalone Gradle project with its own settings.gradle.kts and build.gradle.kts. It publishes convention plugins that modules apply via the plugins { } DSL. Unlike buildSrc, a composite build is compiled independently — changes to build-logic only recompile the build-logic project itself, not all 200 modules. Gradle's composite build support also enables better caching: the convention plugin JARs are cached and reused across builds. Version catalogs further reduce the need for buildSrc — dependency versions live in gradle/libs.versions.toml instead of Kotlin constants. The migration path: move buildSrc code into build-logic, convert constants to version catalog entries, and convert shared configuration blocks into convention plugins.

Q46. Explain convention plugins — how do you set up a build-logic module?

Convention plugins encapsulate shared build configuration into reusable, composable plugins. Setup: (1) Create build-logic/ directory with settings.gradle.kts: dependencyResolutionManagement { repositories { google(); mavenCentral(); gradlePluginPortal() } } and build-logic/convention/build.gradle.kts that applies kotlin-dsl and declares dependencies on AGP and Kotlin Gradle plugins. (2) Include it in the root settings.gradle.kts: pluginManagement { includeBuild("build-logic") }.

(3) Create plugin classes in build-logic/convention/src/main/kotlin/: a plugin implements Plugin<Project> and configures the project in its apply() method. For example, AndroidLibraryConventionPlugin applies com.android.library and kotlin-android, sets compileSdk, minSdk, Java/Kotlin JVM target, enables nonTransitiveRClass, and configures common compiler options. (4) Register plugins in build-logic/convention/build.gradle.kts: gradlePlugin { plugins { register("androidLibrary") { id = "myapp.android.library"; implementationClass = "AndroidLibraryConventionPlugin" } } }. (5) Apply in modules: plugins { id("myapp.android.library") }. Convention plugins compose: AndroidFeatureConventionPlugin can apply myapp.android.library internally and add feature-specific configuration (Hilt, Navigation, Compose). This eliminates copy-paste across 100 module build scripts and ensures consistency. NiA's build-logic has ~10 convention plugins covering application, library, feature, Compose, Hilt, and testing configurations.

Q47. How do you access the AGP extension API from a convention plugin?

In a convention plugin, AGP's extensions are available after the Android plugin is applied. Access them via project.extensions.getByType<LibraryExtension>() (for com.android.library) or project.extensions.getByType<ApplicationExtension>() (for com.android.application). For the new Variant API, use project.extensions.getByType<AndroidComponentsExtension<*, *, *>>(). A common pattern in convention plugins:

class AndroidLibraryConventionPlugin : Plugin<Project> {
    override fun apply(target: Project) = with(target) {
        pluginManager.apply("com.android.library")
        pluginManager.apply("org.jetbrains.kotlin.android")

        extensions.configure<LibraryExtension> {
            compileSdk = 34
            defaultConfig.minSdk = 24
            compileOptions {
                sourceCompatibility = JavaVersion.VERSION_17
                targetCompatibility = JavaVersion.VERSION_17
            }
        }

        extensions.configure<AndroidComponentsExtension<*, *, *>> {
            beforeVariants(selector().withBuildType("release")) {
                (it as? HasUnitTestBuilder)?.enableUnitTest = false
            }
        }
    }
}

The key detail: your build-logic module must declare AGP as a compileOnly dependency (compileOnly("com.android.tools.build:gradle:8.2.0")) so the extension classes are available at compile time. At runtime, the actual AGP version comes from the consuming project's plugin resolution. This decoupling lets you update AGP in the root project without recompiling build-logic (as long as the API is compatible). Use compileOnly for the Kotlin Gradle plugin dependency too.

Q48. What are precompiled script plugins? When do you use them vs class-based plugins?

Precompiled script plugins are .gradle.kts files placed in the src/main/kotlin/ directory of a build-logic project. Gradle compiles them into plugin classes automatically — a file named myapp.android.library.gradle.kts becomes a plugin with ID myapp.android.library. Inside the script, you write the same DSL you'd write in a module's build.gradle.kts: plugins { id("com.android.library") }, android { compileSdk = 34 }, etc. No Plugin<Project> class needed.

Pros of precompiled scripts: (1) Familiar DSL syntax — no need to learn the Plugin<Project> API or extension access patterns. (2) Faster to write for simple conventions. (3) IDE support for the Gradle DSL works inside the script. Cons: (1) Limited composability — you can't easily parameterize a precompiled script (no constructor arguments). (2) Harder to test — class-based plugins can be unit tested with ProjectBuilder. (3) Less type-safe — extension access in scripts uses string-based lookups internally. Use precompiled scripts for simple, static conventions (setting compileSdk, enabling Compose, applying standard plugins). Use class-based plugins when you need: parameterization (a plugin that accepts configuration via a custom extension), complex logic (conditional plugin application, variant filtering), testability, or composition (one plugin applying another). In practice, most mature projects use class-based plugins for the flexibility and testability, with precompiled scripts for trivial cases.

Q49. How do you integrate version catalogs with convention plugins?

Version catalogs (gradle/libs.versions.toml) are accessible in convention plugins, but with a caveat: the type-safe libs. accessor generated by Gradle is not available inside Plugin<Project> classes (it's only available in build.gradle.kts scripts). In class-based plugins, access the catalog via the VersionCatalogsExtension:

val libs = project.extensions
    .getByType<VersionCatalogsExtension>()
    .named("libs")

dependencies {
    add("implementation", libs.findLibrary("androidx-core").get())
    add("implementation", platform(libs.findLibrary("compose-bom").get()))
}

For precompiled script plugins, the type-safe libs. accessor works directly because the script is compiled in the context of the consuming project. Best practices: (1) Use the catalog for all dependency versions — don't hardcode versions in convention plugins. (2) Use findVersion() for version strings needed in plugin configuration (e.g., jvmTarget = libs.findVersion("jvmTarget").get().toString()). (3) Use findBundle() to apply dependency bundles. (4) Share the catalog across projects by publishing it or referencing a shared file. The version catalog + convention plugin combination is the modern replacement for the old buildSrc/Dependencies.kt pattern — it's more maintainable, better cached, and natively supported by Gradle's tooling.

Q50. How do build variants and flavors work? How do you filter unused variants?

Build variants are the Cartesian product of build types and product flavors. Build types (debug, release) control optimization, signing, and debuggability. Product flavors (grouped into dimensions) control feature sets, branding, or environments. Each variant gets: its own source set (src/freeDebug/), its own merged manifest, its own resource set, its own dependency configuration (freeDebugImplementation), and its own set of ~40-60 AGP tasks. With 2 build types × 3 flavors, you get 6 variants and ~300 tasks.

Filtering unused variants is critical for build performance. Use the Variant API:

androidComponents {
    beforeVariants(selector().all()) { variant ->
        variant.enable = when {
            variant.buildType == "release" &&
                variant.flavorName == "staging" -> false
            variant.buildType == "debug" &&
                variant.flavorName == "production" -> false
            else -> true
        }
    }
}

Disabling a variant prevents AGP from registering its tasks entirely — they don't appear in the task graph and don't consume configuration time. For library modules, use publishing { singleVariant("release") } to publish only the release variant. In convention plugins, filter variants centrally: the feature convention plugin can disable all staging+release and production+debug combinations by default. This is especially impactful in CI: building only the variants you actually test or deploy can cut build time by 50% or more.

Q51. How do you write and test a custom Gradle plugin?

A custom Gradle plugin implements Plugin<Project>. Define a custom extension for configuration, register tasks lazily, and wire inputs/outputs with Provider/Property:

abstract class LicenseCheckExtension {
    abstract val allowedLicenses: ListProperty<String>
}

class LicenseCheckPlugin : Plugin<Project> {
    override fun apply(project: Project) {
        val extension = project.extensions.create<LicenseCheckExtension>("licenseCheck")
        project.tasks.register<LicenseCheckTask>("checkLicenses") {
            allowedLicenses.set(extension.allowedLicenses)
            dependencyFile.set(project.layout.buildDirectory.file("dependencies.txt"))
            reportFile.set(project.layout.buildDirectory.file("license-report.txt"))
        }
    }
}

Testing uses Gradle TestKit: create a temporary project directory, write a build.gradle.kts that applies your plugin, and run a build programmatically:

@Test fun `detects disallowed license`() {
    val projectDir = tempDir.newFolder()
    File(projectDir, "build.gradle.kts").writeText("""
        plugins { id("com.myapp.license-check") }
        licenseCheck { allowedLicenses.set(listOf("Apache-2.0")) }
    """)
    val result = GradleRunner.create()
        .withProjectDir(projectDir)
        .withArguments("checkLicenses")
        .withPluginClasspath()
        .buildAndFail()
    assertThat(result.output).contains("GPL-3.0 not in allowed list")
}

withPluginClasspath() adds the plugin's compiled classes to the test build's classpath. Use build() for expected success, buildAndFail() for expected failure. Test edge cases: missing extension configuration, empty inputs, up-to-date checking, and cache compatibility.

Q52. What is a composite build? How does it differ from buildSrc?

A composite build includes another Gradle project as a dependency via includeBuild("path") in settings.gradle.kts. The included build is compiled independently and its outputs (plugins, libraries) are substituted into the main build's dependency graph. buildSrc is a special, hardcoded composite build with unique behavior: it's always compiled first, its output is added to every project's build script classpath, and any change invalidates the entire build.

Key differences: (1) Invalidation scope: buildSrc changes invalidate all projects. Composite build changes only invalidate the included build itself — consuming projects are only affected if the included build's outputs (plugin bytecode) actually change. (2) Compilation independence: A composite build is a standalone Gradle project with its own Daemon, caching, and configuration. buildSrc is compiled as part of the main build's configuration phase. (3) Flexibility: You can have multiple composite builds (includeBuild("build-logic"), includeBuild("../shared-plugins")). buildSrc is singular. (4) Sharing: Composite builds can be shared across multiple projects (e.g., a company-wide plugin repository). buildSrc is project-local. (5) Configuration cache: Composite builds are compatible with the configuration cache. buildSrc has limitations. The migration from buildSrc to a composite build is straightforward: move the code to build-logic/, add a settings.gradle.kts and build.gradle.kts, register plugins with gradlePlugin { }, and add includeBuild("build-logic") to the root settings.

Common Problems & Solutions (Q53–Q60)

Q53. How do you solve circular dependencies between feature modules?

Circular dependencies (:feature:a → :feature:b → :feature:a) are rejected by Gradle at configuration time. They indicate an architectural problem — the modules aren't properly separated. Solutions: (1) Extract shared code: If both features need the same data class, use case, or utility, move it to a :core:* module that both depend on. (2) Dependency inversion: If :feature:a needs to call :feature:b's functionality, define an interface in a shared module (:core:navigation or :core:contracts), implement it in :feature:b, and inject it into :feature:a via Hilt. Neither feature depends on the other — both depend on the shared interface module.

(3) Event-based communication: Use a shared event bus or SharedFlow in a core module. :feature:a emits events, :feature:b observes them, with no direct dependency. (4) Navigation-based decoupling: Instead of :feature:a calling :feature:b's code directly, navigate to :feature:b's screen via deep links or a navigation interface. The :app module wires the navigation graph, so features don't need to know about each other. (5) Merge the modules: If two features are so intertwined that separating them requires excessive indirection, they might be one feature. Don't force modularization where the domain doesn't support it. The key principle: feature modules should be leaf nodes in the dependency graph — they depend downward on core modules, never sideways on other features.

Q54. How do you handle resource conflicts across modules?

Resource conflicts occur when two modules define resources with the same name and type (e.g., both :feature:a and :feature:b define R.string.title). AGP's resource merging follows a priority order: the consuming module's resources override dependency resources, and later dependencies override earlier ones. This can cause silent bugs — :feature:a's "title" string might be replaced by :feature:b's without any warning.

Solutions: (1) resourcePrefix — in each module's build.gradle.kts: android { resourcePrefix = "search_" }. This triggers a Lint warning (not a compilation error) when resources don't start with the prefix. It's advisory but effective with CI enforcement. (2) Naming conventions: Prefix all resources with the module name: search_title, checkout_title. This is the most reliable approach — it prevents conflicts by convention. (3) nonTransitiveRClass: With this enabled, each module's R class only contains its own resources. This doesn't prevent conflicts during merging, but it makes the conflict visible — you must explicitly reference com.myapp.feature.search.R.string.search_title. (4) Resource namespacing (experimental): AGP's android.experimental.enableResourceNamespacing=true fully namespaces resources per module, preventing conflicts entirely. However, it's still experimental and requires all resource references to be fully qualified. (5) Design system module: Shared resources (colors, dimensions, styles, themes) live in :core:designsystem — features reference them instead of defining their own.

Q55. Explain manifest merging — what happens when two modules declare conflicting attributes?

AGP merges manifests from all modules, libraries, and build variants into a single final manifest. The merge follows a priority order: build variant manifest (highest) > main source set manifest > library manifests (in dependency order). When two manifests declare the same element with different attribute values, the merger applies rules: higher-priority values override lower-priority ones for most attributes. For <activity>, <service>, <receiver>, and <provider> elements, entries from all manifests are merged by android:name — if two libraries declare the same component, attributes are merged with higher-priority values winning.

Conflicts arise when: (1) Two libraries set different minSdk values — the merger takes the highest. If your app's minSdk is lower than a library's, you get a build error unless you add tools:overrideLibrary="com.library.package". (2) Two libraries declare the same permission with different maxSdkVersion — use tools:remove or tools:replace to resolve. (3) Conflicting android:allowBackup or android:supportsRtl — use tools:replace="android:allowBackup" in the app manifest. Debugging: run ./gradlew :app:processDebugMainManifest and inspect app/build/intermediates/merged_manifests/debug/AndroidManifest.xml. The merge report at build/outputs/logs/manifest-merger-debug-report.txt shows exactly which manifest contributed each element and how conflicts were resolved. Common pitfall: a library adding INTERNET permission that you didn't expect — always review the merged manifest in CI.

Q56. How does R8/ProGuard work with multi-module projects? What is consumerProguardFiles?

R8 (the successor to ProGuard) runs only on the final application module (:app) during release builds. It processes the entire app's bytecode — from all modules and all dependencies — in a single pass. This whole-program analysis enables aggressive optimizations: unused classes from any module are removed, cross-module method inlining is performed, and the entire class hierarchy is analyzed for devirtualization. Library modules (com.android.library) are not individually shrunk — their bytecode is passed to R8 as-is when the app module is built.

consumerProguardFiles is how library modules contribute keep rules to the final R8 pass. In a library's build.gradle.kts: android { defaultConfig { consumerProguardFiles("consumer-rules.pro") } }. These rules are packaged inside the library's AAR and automatically applied when R8 processes the app. This is how libraries protect their public API, reflection-accessed classes, and serialization models from being removed or renamed. The distinction: proguardFiles are used when the module itself is the application being shrunk. consumerProguardFiles are rules that travel with the library and are applied by the consuming app's R8 pass. A common multi-module pattern: each :core:* module has a consumer-rules.pro that keeps its public API, Hilt-generated classes, Room entities, and Retrofit interfaces. The :app module's proguard-rules.pro handles app-specific rules (entry points, reflection). Missing consumerProguardFiles is a top cause of release-build crashes — R8 strips classes that are only accessed via reflection or serialization.

Q57. How do you implement navigation across feature modules that can't depend on each other?

Feature modules must navigate to each other without direct dependencies. The main patterns: (1) Deep link-based navigation: Each feature registers deep links in its navigation graph. Navigation uses URIs: navController.navigate("myapp://feature/search?query=kotlin"). The :app module's top-level NavGraph includes all feature graphs, and deep link resolution handles routing. Pros: simple, works with Navigation Component. Cons: string-based URIs are fragile, no compile-time safety. (2) Navigation interface pattern: Define a Navigator interface in a shared module (:core:navigation) with methods like navigateToSearch(query: String). Implement it in :app (which has access to all feature routes). Inject the interface into features via Hilt. Pros: type-safe, refactor-friendly. Cons: the interface grows with every new destination.

(3) Route registry pattern: Each feature module registers its routes in a shared registry (via Hilt multibindings or a ServiceLoader). The :app module collects all registered routes and builds the NavGraph dynamically. Pros: decentralized, each feature owns its routes. Cons: runtime wiring, harder to debug. (4) Type-safe routes with @Serializable: Navigation Compose 2.8+ supports @Serializable route objects. Define route data classes in :core:navigation, and features reference them without depending on each other. The :app module maps routes to composable destinations. This is the modern recommended approach — it combines type safety with decoupling. Regardless of pattern, the :app module is always the composition root that wires features together.

Q58. How does Hilt work across module boundaries? What is the composition root?

Hilt generates a single component hierarchy rooted in the @HiltAndroidApp-annotated Application class (the composition root). At compile time, Hilt processes @Module and @InstallIn annotations from all modules on the classpath and generates the Dagger_MyApplication_HiltComponents class in the :app module. This means the :app module must have visibility into all Hilt modules — it depends on all feature and core modules (directly or transitively).

Cross-module patterns: (1) Feature modules declare @Module @InstallIn(ViewModelComponent::class) for ViewModel-scoped bindings and @InstallIn(SingletonComponent::class) for app-scoped bindings. These modules are discovered by Hilt when :app compiles. (2) Interface binding: :core:data declares @Binds abstract fun bindUserRepo(impl: UserRepoImpl): UserRepo in a @Module @InstallIn(SingletonComponent::class). Features inject UserRepo (the interface) without knowing about UserRepoImpl. (3) Component dependencies: Hilt's predefined components (SingletonComponent, ActivityComponent, ViewModelComponent) span module boundaries — a binding in :core:network scoped to @Singleton is available in any feature module's @HiltViewModel. (4) Testing: @TestInstallIn in test modules replaces production bindings with fakes. @UninstallModules removes specific modules. The composition root pattern means Hilt is inherently centralized — the :app module is the single point where all bindings are assembled. This is a trade-off: it ensures consistency but means :app must depend on everything.

Q59. Design a testing strategy for a 50-module project — unit tests, integration tests, and CI.

Unit tests (per module): Each module has src/test/ with fast JVM tests. :core:domain tests use cases with fake repositories. :core:data tests repository logic with fake data sources and Room's in-memory database. :feature:* modules test ViewModels with fake use cases and TestDispatcher. Compose UI tests use createComposeRule() with fake data. Target: 80%+ coverage per module, < 2 minutes per module. Use a :core:testing module for shared test utilities (fake dispatchers, test data builders, custom assertions).

Integration tests: :app module has src/androidTest/ with end-to-end flows using Hilt's @HiltAndroidTest. Replace real network/database with @TestInstallIn fakes or MockWebServer. Test critical user journeys: login → browse → checkout → confirmation. Target: 10-20 critical path tests, < 10 minutes total.

CI strategy: (1) PR checks (fast, < 10 min): Run testDebugUnitTest only for affected modules (use affected-module-detector). Run lintDebug for affected modules. Run assembleDebug for :app to verify compilation. (2) Merge to main (thorough, < 20 min): Run all unit tests across all 50 modules (parallelized across CI runners). Run integration tests. Generate coverage report with JaCoCo. (3) Nightly (comprehensive, < 60 min): Full instrumented test suite on multiple API levels. Full Lint analysis. Dependency vulnerability scan. Build scan analysis for cache hit rates. Infrastructure: Remote build cache (CI pushes, developers pull). Test result caching — unchanged modules' test results are cached. Parallel test execution with maxParallelForks = Runtime.getRuntime().availableProcessors() / 2. Flaky test quarantine — automatically retry failed tests and flag flaky ones.

Q60. Your modularized project's IDE is slow — how do you diagnose and fix it?

Diagnosis: (1) Check module count — Android Studio's indexer scales poorly beyond 100-150 modules. Each module adds overhead for: source indexing, resource indexing, Gradle sync, and symbol resolution. (2) Run Help → Diagnostic Tools → Activity Monitor to identify CPU-bound operations. (3) Check Gradle sync time in the Build Output window — if sync takes > 30 seconds, the configuration phase is the bottleneck. (4) Check memory: Help → Change Memory Settings — Android Studio with 200 modules needs 4-8 GB heap. (5) Profile with Help → Diagnostic Tools → Analyze Stack Trace during a hang to identify the bottleneck (indexing, Gradle sync, code analysis).

Fixes: (1) Increase IDE memory: Set -Xmx6g in studio.vmoptions. (2) Reduce loaded modules: Use settings.gradle.kts to conditionally include only the modules you're working on: if (System.getenv("LOAD_ALL") != "true") { include(":feature:search", ":core:data", ":core:model") }. (3) Enable configuration cache: Reduces Gradle sync time dramatically. (4) Disable unused inspections: Settings → Editor → Inspections — disable inspections you don't use (e.g., spelling, some Kotlin inspections). (5) Use nonTransitiveRClass: Reduces R class indexing overhead. (6) Exclude build directories from indexing: Mark build/ directories as excluded in Project Structure. (7) Upgrade Android Studio: Each release improves multi-module performance — Electric Eel and later versions have significant indexing optimizations. (8) Use unloadModules: Right-click modules in the Project view → "Unload Modules" to exclude them from indexing without modifying settings.gradle.kts. (9) SSD and fast hardware: Multi-module indexing is I/O-intensive — an NVMe SSD and 32 GB RAM make a measurable difference. The root cause is usually too many modules for the IDE to handle — aim for < 100 modules, and use the conditional include pattern for larger projects.
