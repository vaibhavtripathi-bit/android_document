08 — Interview Q&A — Modularization & Gradle

60 Staff Engineer questions covering Gradle internals, module architecture, modularization strategies, dependency management, build performance, convention plugins, and common problems at scale

Gradle Internals (Q1–Q10)

Q1. Explain Gradle's three build phases — what happens in each and why does it matter for performance?

Initialization evaluates settings.gradle.kts to determine which projects participate in the build. Gradle creates a Settings object, resolves include() statements, and builds the project hierarchy. Configuration evaluates every build.gradle.kts for every included project, executing plugin apply blocks, registering tasks, and wiring up dependency configurations — even for projects you aren't building. Execution runs only the tasks in the task graph that are required by the requested target, respecting dependencies and parallelism.

Performance impact is enormous: a 50-module project configures all 50 modules even when you run :app:assembleDebug. This is why configuration avoidance matters — using tasks.register() instead of tasks.create(), lazy Provider/Property APIs, and the configuration cache (which serializes the task graph and skips configuration entirely on subsequent runs). At Google-scale projects, configuration alone can take 10–30 seconds without the configuration cache. The configuration cache reduces this to near-zero on cache hits by replaying the serialized task graph.

Q2. What is the difference between tasks.register() and tasks.create()? Why does it matter?

tasks.create() eagerly instantiates and configures the task during the configuration phase — its constructor runs, its configuration closure executes, and the task object is fully materialized in memory. tasks.register() returns a TaskProvider<T> and defers instantiation until the task is actually needed for execution or queried by another task. The configuration closure only runs if the task is part of the execution graph.

In a project with 200 registered tasks where you only execute one, create() materializes all 200 during configuration while register() materializes only the task you requested plus its dependencies. AGP registers hundreds of tasks per variant (compile, merge resources, dex, package) — with 4 build variants that's 800+ tasks. Eager creation wastes CPU and memory during configuration. The Gradle team measured 20–40% configuration time reduction by migrating to lazy task registration. The TaskProvider also enables type-safe task wiring: taskB.configure { input.set(taskA.flatMap { it.output }) } connects tasks without materializing either during configuration.

Q3. How does Gradle's dependency resolution work? What happens when two modules request different versions of the same library?

Gradle resolves dependencies in a multi-step process: (1) build a dependency graph from all declared dependencies and their transitives, (2) identify conflicts where multiple versions of the same module appear, (3) apply conflict resolution — by default, Gradle selects the highest version (optimistic upgrade). This differs from Maven, which uses nearest-wins. The resolution engine walks the graph, collects all requested versions of each module coordinate (group:name), and picks the winner.

You can override this with resolutionStrategy.force() (pins a specific version), failOnVersionConflict() (fails the build instead of auto-resolving), or dependencySubstitution rules. Rich version declarations add nuance: strictly rejects any other version (acts like a veto), require is the default preference, and prefer is a soft hint. BOMs (platform()) inject version constraints without adding dependencies. When resolution completes, Gradle caches the result in .gradle/caches/modules-2 and creates a resolved dependency graph that tasks like compileKotlin consume. Use ./gradlew :app:dependencies or --scan to inspect the resolved graph and understand why a particular version was selected.

Q4. Explain the difference between api and implementation — how does it affect compilation and build performance?

implementation makes a dependency available to the declaring module at compile time and runtime, but hides it from modules that depend on the declaring module. api makes the dependency available transitively — any module depending on the declaring module also sees the api dependency on its compile classpath. This distinction comes from the Gradle Java Library plugin and is critical for build performance.

When a library exposed via implementation changes its ABI, only the declaring module recompiles. When a library exposed via api changes, every module in the transitive dependency chain must recompile. In a deep dependency graph (:app → :feature → :core → :network), if :network uses api for OkHttp, changing OkHttp's version triggers recompilation of :core, :feature, and :app. With implementation, only :network recompiles. The rule of thumb: use implementation by default, use api only when your module's public API exposes types from the dependency (e.g., a function returns OkHttpClient). Misusing api everywhere effectively disables the compilation avoidance that modularization provides — you get the complexity of modules without the build speed benefit.

Q5. What is the Gradle Daemon? How does it improve build performance and when should you disable it?

The Gradle Daemon is a long-lived JVM process that persists between builds. On the first invocation, Gradle spawns a daemon that loads the Gradle runtime, classloaders, and build scripts into memory. Subsequent builds reuse this warm JVM — the JIT compiler has already optimized hot paths, class metadata is cached, and the file system snapshot from the previous build is available. This typically saves 2–5 seconds of JVM startup plus significant time from JIT warmup.

The daemon also maintains an in-memory file system snapshot (via the Virtual File System / VFS) that tracks which files changed between builds, enabling faster up-to-date checks. Daemons are version-specific — if you upgrade Gradle, a new daemon spawns. They expire after 3 hours of inactivity by default. You should disable the daemon (--no-daemon) in CI environments where each build runs in a fresh container (no reuse benefit, and the daemon's memory overhead is wasted). However, if your CI reuses build agents, keeping the daemon alive across builds provides the same warm-JVM benefits as local development. Gradle 8+ also supports --daemon=false for single-use mode. Memory issues (OOM) in the daemon are common — tune with org.gradle.jvmargs=-Xmx4g in gradle.properties.

Q6. How does incremental compilation work in Kotlin? What is ABI change detection?

The Kotlin compiler tracks dependencies between source files at the class/function level. When you edit a file, the compiler determines which other files depend on the changed declarations and recompiles only those. This is incremental compilation — instead of recompiling all 500 files in a module, it might recompile only 3. The Kotlin compiler maintains an incremental cache (in build/kotlin/compileKotlin/cacheable) that stores the dependency graph between files.

ABI (Application Binary Interface) change detection is the cross-module optimization. When module A depends on module B, Gradle doesn't need to recompile A every time B changes — only when B's ABI changes. The ABI includes public/protected class signatures, method signatures, and constant values. If you change only a private method body in B, the ABI is unchanged and A is skipped. Kotlin uses kotlin-build-statistics to track this. ABI detection breaks when you use internal visibility with api configuration (the internal symbol is ABI-visible to same-module consumers via mangled names). Annotation processors (kapt) disable incremental compilation for affected source sets because generated code can depend on any input — KSP improves this with fine-grained input tracking.

Q7. Explain the build cache — how does it work, what is the cache key, and how do local and remote caches differ?

The build cache stores task outputs keyed by a hash of the task's inputs. When a task runs, Gradle computes a cache key from: the task class name, its input properties (file contents, string values, classpath), and the task's implementation (class bytecode hash). If a cache entry with that key exists, Gradle skips execution and copies the cached outputs. The cache key is deterministic — identical inputs always produce the same key.

The local cache (default: ~/.gradle/caches/build-cache-1) stores entries on disk. It helps when you switch branches and switch back — outputs from the previous branch are cached. The remote cache (HTTP backend, Develocity, or custom) is shared across the team. Developer A builds :core:network, the output is pushed to the remote cache. Developer B pulls the cached output instead of recompiling. This is transformative for large teams — common modules are built once and shared. Cache misses happen when: inputs aren't properly declared (absolute paths leak into the key), tasks use non-deterministic outputs (timestamps in generated files), or the JDK/toolchain version differs. Debug with --scan → Build Cache tab, which shows exactly which input changed and caused the miss. Remote caches require careful access control — read-only for developers, read-write for CI.

Q8. What is the configuration cache? How does it differ from the build cache?

The build cache caches task outputs — the files produced by task execution. The configuration cache caches the task graph itself — the result of the configuration phase. On the first run, Gradle evaluates all build.gradle.kts files, resolves plugins, registers tasks, and wires the dependency graph. It then serializes this entire task graph to disk. On subsequent runs (if no build scripts, gradle.properties, or included builds changed), Gradle deserializes the cached graph and jumps straight to execution, skipping configuration entirely.

This is especially impactful for large projects where configuration takes 10–30 seconds. The configuration cache imposes strict requirements: build logic cannot reference mutable global state, Project objects cannot be accessed at execution time (only during configuration), and all task inputs must be serializable. Violations are reported as problems — Gradle tracks them and fails the build if the count exceeds the threshold. The configuration cache also enables parallel configuration in Gradle 8.x — since each project's configuration is isolated, they can be evaluated concurrently. Enabling it: org.gradle.configuration-cache=true in gradle.properties. The build cache and configuration cache are complementary — use both for maximum performance.

Q9. How does AGP (Android Gradle Plugin) work internally? What tasks does it register and how do build variants affect task count?

AGP is a Gradle plugin (com.android.application / com.android.library) that registers Android-specific tasks during configuration. For each build variant (combination of build type × product flavors), AGP registers ~40–60 tasks: compile{Variant}Kotlin, merge{Variant}Resources, process{Variant}Manifest, merge{Variant}Assets, dexBuilder{Variant}, merge{Variant}Dex, package{Variant}, lintAnalyze{Variant}, and many more. With 2 build types (debug/release) × 2 flavors (free/paid), you get 4 variants and ~200 tasks per module.

AGP uses the Variant API (stabilized in AGP 7.0+) to let plugins customize variants. Internally, AGP creates a VariantBuilder during configuration (where you can disable variants via beforeVariants { if (name == "freeRelease") enable = false }) and a Variant object during finalization (where you can wire custom tasks into the build pipeline via artifacts API). AGP also manages the Android SDK, NDK, build tools, and R8/ProGuard integration. The task graph is complex — assembleDebug depends on ~30 tasks in a single module. Across 50 modules, that's 1500+ tasks in the execution graph. Minimizing variants (disable unused flavor combinations) and using tasks.register() in custom plugins is critical to keep configuration time manageable.

Q10. What is the Provider/Property API in Gradle? Why is it important for configuration avoidance?

Provider<T> is a read-only lazy value that is computed only when queried (.get()). Property<T> extends Provider with a setter (.set(), .convention()). Together they form the backbone of Gradle's configuration avoidance strategy. Instead of resolving values eagerly during configuration, you wire providers together — the actual computation happens during execution when the task runs.

For example, a task's output directory is a DirectoryProperty. Another task's input can be wired to it: inputDir.set(otherTask.flatMap { it.outputDir }). This creates a lazy dependency chain — neither task is materialized during configuration, and the path is resolved only at execution time. The flatMap/map operators let you transform providers without triggering evaluation. ListProperty, SetProperty, and MapProperty handle collections lazily. AGP's Variant API exposes everything as providers — variant.applicationId is a Property<String>, not a raw string. Without this API, build scripts would eagerly resolve all values during configuration, defeating task avoidance and making the configuration cache impossible (since resolved values might reference non-serializable objects like Project).

Module Architecture (Q11–Q20)

Q11. What are the benefits of modularization? When does it NOT make sense?

Benefits: (1) Build speed — Gradle compiles modules in parallel and caches outputs independently; changing one module doesn't recompile unrelated ones. (2) Encapsulation — internal visibility enforces boundaries; modules expose only their public API. (3) Team scalability — teams own modules independently, reducing merge conflicts and enabling parallel development. (4) Reusability — shared modules (:core:network, :core:design-system) are consumed by multiple apps. (5) Testability — modules can be tested in isolation with fake dependencies.

When it doesn't make sense: Small apps with 1–2 developers gain minimal build speed improvement but pay the complexity tax of managing module boundaries, dependency wiring, and navigation abstractions. If your monolith builds in under 30 seconds, modularization adds overhead without proportional benefit. Over-modularization (100+ modules for a medium app) increases configuration time, complicates the dependency graph, and creates a "module per file" anti-pattern where the abstraction cost exceeds the benefit. The sweet spot depends on team size, build time, and feature independence — modularize when build times exceed 2–3 minutes or when team contention on shared code becomes a bottleneck.

Q12. Describe the module types in a well-architected Android app — what does each contain and what are the dependency rules?

:app — the application module. Contains the Application class, Hilt entry point (@HiltAndroidApp), navigation graph wiring, and top-level DI bindings. Depends on all feature modules. :feature:* — one per user-facing feature (e.g., :feature:home, :feature:search). Contains UI (Composables/Fragments), ViewModels, and feature-specific DI. Depends on :core:* modules but never on other features. :core:data — repository implementations, data sources, mappers. Depends on :core:network, :core:database, :core:domain. :core:domain — pure Kotlin module with use cases, domain models, and repository interfaces. Zero Android dependencies. :core:network — Retrofit/Ktor setup, API definitions, DTOs. :core:database — Room DAOs, entities, migrations. :core:ui / :core:design-system — shared Compose components, theme, typography.

The dependency rule: dependencies flow downward (:app → :feature → :core). Features never depend on each other. Core modules may depend on other core modules but never on features. :core:domain depends on nothing (or only :core:model). This DAG structure ensures maximum parallelism and cache reuse.

Q13. Explain the difference between package-by-feature and package-by-layer — which do you prefer and why?

Package-by-layer groups code by technical role: ui/, data/, domain/, di/. This is the traditional approach — easy to understand but scales poorly. Adding a feature touches every layer package, and unrelated features are interleaved in the same directories. Package-by-feature groups code by business capability: feature/search/, feature/checkout/. Each package contains its own UI, ViewModel, repository, and DI — a vertical slice. This maps naturally to team ownership and makes feature deletion trivial (delete one package).

For modular Android apps, package-by-feature at the module level with package-by-layer inside each module is the preferred hybrid. Each :feature:* module is a feature boundary, and within it you organize by layer (ui/, data/, domain/). This gives you the encapsulation benefits of feature isolation (Kotlin internal visibility at the module boundary) with the readability of layered organization within each feature. Pure package-by-layer in a monolith leads to "shotgun surgery" — every change touches multiple packages. Pure package-by-feature without modules lacks compile-time boundary enforcement.

Q14. What is Google's recommended module architecture (Now in Android)? Walk through the dependency graph.

The Now in Android (NIA) project uses a layered modular architecture: :app → :feature:* → :core:*. The :app module depends on all feature modules and wires the top-level navigation (NavHost). Feature modules (:feature:foryou, :feature:interests, :feature:bookmarks, :feature:topic) contain Composable screens, ViewModels, and navigation definitions. They depend on :core:data (repositories), :core:domain (use cases), :core:ui (shared components), and :core:designsystem (theme/tokens).

:core:data implements repository interfaces from :core:domain and depends on :core:network (Retrofit + kotlinx.serialization) and :core:database (Room). :core:domain is a pure Kotlin module — no Android dependencies — containing use cases and model interfaces. :core:model holds shared data classes used across layers. :core:common provides utilities (dispatchers, result wrappers). NIA also has :core:testing (shared test utilities), :core:datastore (Proto DataStore), and :sync (WorkManager sync). Convention plugins in build-logic/ standardize module configuration. The key insight: features are independent vertical slices, core modules are shared horizontal layers, and the dependency graph is a strict DAG with no cycles.

Q15. How do you decide the right module granularity? What are the signs of over-modularization vs under-modularization?

Right granularity balances build parallelism against dependency management overhead. A good heuristic: one module per independently deployable/testable feature, plus shared core modules for cross-cutting concerns. If two features always change together and are owned by the same team, they might belong in one module. If a module has only 2–3 files, it's probably too granular.

Over-modularization signs: Configuration phase takes longer than execution. Developers spend more time managing build.gradle.kts files than writing feature code. Navigation between modules requires excessive abstraction layers. The dependency graph is so complex that no one understands it. Build cache hit rates drop because too many small modules create excessive cache entries. Under-modularization signs: Incremental builds are slow because one large module recompiles entirely. Teams block each other with merge conflicts. internal visibility is meaningless because everything is in one module. Test suites are slow because the entire module must be compiled to test one feature. A practical metric: if a module takes more than 30 seconds to compile incrementally, consider splitting it. If configuration time exceeds 15% of total build time, you may have too many modules.

Q16. Explain the api/impl module split pattern — when and why would you use it?

The pattern splits a module into two: :feature:search:api (contains only interfaces, data classes, and navigation routes) and :feature:search:impl (contains the actual implementation). Other modules depend only on :api, never on :impl. The :app module depends on all :impl modules to wire the DI bindings.

This achieves compile-time decoupling: when the implementation changes (new UI, refactored ViewModel), only :impl recompiles. Modules depending on :api are unaffected because the interface didn't change. It also enables parallel development — Team A defines the :api contract, Team B implements it, and Team C consumes it, all simultaneously. The trade-off is complexity: you double the module count, need DI to bind interfaces to implementations, and must carefully design the API surface. Use this pattern for modules with many consumers (e.g., :core:analytics:api consumed by every feature) or when implementation churn is high. Don't use it for leaf modules with a single consumer — the indirection adds cost without benefit. At scale (50+ modules), the compile-time savings from this pattern can reduce incremental build times by 30–50%.

Q17. How do you enforce module boundaries? What tools and techniques prevent boundary violations?

Kotlin internal visibility is the first line of defense — classes marked internal are invisible to other modules at compile time. This is enforced by the Kotlin compiler, not just convention. Gradle dependency rules prevent illegal dependencies: if :feature:home doesn't declare a dependency on :feature:profile, it simply can't import its classes. You can add custom validation in settings.gradle.kts or a convention plugin that fails the build if a feature module depends on another feature module.

Architecture linters like dependency-analysis-gradle-plugin detect unused dependencies, misused api/implementation configurations, and transitive dependency leaks. Module graph validation tools (e.g., module-graph-assert) let you declare rules like "no feature-to-feature dependencies" and fail CI on violations. ArchUnit or konsist can enforce package-level rules within modules. At the code review level, CODEOWNERS files assign module ownership — changes to :core:network require network team approval. The most robust approach combines compile-time enforcement (Gradle dependencies + internal), static analysis (dependency-analysis plugin), and CI gates (module graph assertions + CODEOWNERS).

Q18. What is the role of :core:domain and why should it be a pure Kotlin module?

:core:domain contains the business logic layer: use cases (interactors), domain models, and repository interfaces. It represents the "what" of the application — business rules that are independent of Android framework, UI framework, or data source implementation. A GetSearchResultsUseCase defines the business operation; it doesn't know whether data comes from Retrofit, Room, or a fake.

Making it a pure Kotlin module (kotlin("jvm") instead of com.android.library) has concrete benefits: (1) Fastest compilation — no Android resource merging, no AIDL, no manifest processing. Kotlin JVM modules compile 2–5x faster than Android library modules. (2) No AGP overhead — no variant multiplication, no R class generation. (3) Testability — unit tests run on the JVM without Robolectric or instrumentation. (4) Reusability — can be shared with a KMP backend or Compose Desktop app. (5) Dependency hygiene — forces you to keep Android framework types out of the domain layer, which is architecturally correct. The domain module should depend only on :core:model (shared data classes), Kotlin coroutines, and possibly javax.inject for DI annotations.

Q19. How do you handle shared resources (strings, colors, drawables) across feature modules?

Create a :core:ui or :core:design-system module that owns shared resources. This module contains the app theme (Theme.kt), color tokens, typography, shared drawables/icons, and common string resources. Feature modules depend on :core:ui and reference its resources. For strings, only truly shared strings (app name, common actions like "Cancel", "OK") belong in :core:ui — feature-specific strings stay in the feature module.

With non-transitive R classes (android.nonTransitiveRClass=true), each module's R class contains only its own resources. To reference a resource from :core:ui, you use the fully qualified R class: com.example.core.ui.R.string.app_name. This prevents accidental resource collisions and makes resource ownership explicit. For icons, consider a dedicated :core:icons module with vector drawables — this keeps the design system module lean. Resource conflicts (two modules defining R.string.title) are resolved by AGP's resource merging: the consuming module's resource wins, but this is fragile. Prefix resources with the module name (home_title, search_title) to avoid collisions. At scale, some teams use a resource linter that fails CI when resource names don't follow the naming convention.

Q20. Design the module architecture for a super-app with 20 features, 5 teams, and a shared design system.

Structure into four layers: App shell (:app) — navigation host, Hilt setup, feature wiring. Feature modules (:feature:payments, :feature:rides, :feature:food, etc.) — one per feature, owned by a single team. Use the api/impl split for features with cross-feature contracts (e.g., :feature:payments:api exposes PaymentResult that :feature:food:impl consumes). Core modules (:core:network, :core:auth, :core:analytics, :core:domain, :core:data, :core:database) — shared infrastructure. Design system (:core:design-system) — tokens, components, theme, owned by the platform/design team.

Team ownership: each team owns their feature modules and contributes to core modules via PRs with CODEOWNERS review. Enforce rules: no feature-to-feature dependencies (use :api modules for contracts), :core:domain is pure Kotlin, all core modules use implementation (not api) for their internal dependencies. Convention plugins in build-logic/ standardize Android SDK versions, Compose compiler, Kotlin options, and testing setup. Remote build cache is essential — 5 teams means lots of redundant compilation without it. Module graph assertions in CI prevent architectural drift. Each feature module should be independently buildable and testable: ./gradlew :feature:payments:impl:testDebugUnitTest should work without building unrelated features. Target: 40–60 modules total (20 features × 2 for api/impl + 10–15 core modules).

Modularization Strategies (Q21–Q28)

Q21. Walk through a monolith-to-modular migration strategy — what do you extract first and why?

Start with the lowest-risk, highest-value extractions. Phase 1: Extract :core:model (pure data classes — zero risk, establishes the pattern) and :core:network (Retrofit interfaces + DTOs — well-bounded, few inbound dependencies). Phase 2: Extract :core:domain (use cases + repository interfaces) and :core:database (Room DAOs + entities). Phase 3: Extract feature modules one at a time, starting with the most independent feature (fewest dependencies on other features).

The key principle: extract leaves first, then work inward. Modules with no dependents are safe to extract — you're only changing the extracted code and its consumers in the monolith. Each extraction follows a pattern: (1) create the new module with build.gradle.kts, (2) move source files, (3) add the dependency from the monolith to the new module, (4) fix compilation errors (usually visibility — make public what was package-private), (5) verify tests pass. Use the dependency-analysis-gradle-plugin to identify actual dependency boundaries before extracting. Don't try to extract everything at once — incremental migration over weeks/months is safer. Set up convention plugins early so each new module is configured consistently. Track progress with a module dependency graph visualization — the monolith should shrink visibly with each extraction.

Q22. Compare by-layer, by-feature, and hybrid modularization — when would you use each?

By-layer creates modules for each architectural layer: :ui, :domain, :data, :network. Simple to understand, but every feature change touches every module, killing parallel compilation. Build cache is ineffective because the :ui module changes on every feature PR. Works only for very small apps (under 5 features) where the simplicity outweighs the build cost.

By-feature creates one module per feature: :feature:home, :feature:search. Each module contains its own UI, domain, and data layers. Maximizes parallelism and cache reuse — changing :feature:search doesn't recompile :feature:home. But shared code (networking, auth, analytics) gets duplicated or requires extracting shared modules anyway. Hybrid (recommended) combines both: feature modules for vertical slices, core modules for shared horizontal layers. :feature:* modules contain feature-specific code, :core:* modules contain shared infrastructure. This is what NIA, Tivi, and most production apps use. The hybrid approach maps naturally to team ownership (team owns features + contributes to core) and maximizes both build performance and code reuse.

Q23. What is vertical vs horizontal slicing? How does it relate to team ownership?

Vertical slicing cuts through all layers for a single feature: :feature:checkout contains the checkout UI, ViewModel, use case, repository, and data source. The slice is self-contained — a team can develop, test, and deploy the checkout feature independently. Horizontal slicing cuts across features at a single layer: :core:network contains all networking logic for every feature. Changes to the network layer affect all features.

For team ownership, vertical slices map to feature teams (Squad Checkout owns :feature:checkout) while horizontal slices map to platform teams (Platform team owns :core:network, :core:auth, :core:analytics). The ideal structure has feature teams owning vertical slices with autonomy to ship independently, and a platform team owning horizontal slices that provide stable, well-documented APIs. Conflicts arise when a feature team needs a change in a horizontal module — this is where the api/impl pattern helps: the platform team defines the :api contract, feature teams can contribute :impl changes via PRs. CODEOWNERS enforces review gates. The ratio matters: too many horizontal modules create a platform team bottleneck; too few create duplication across feature modules.

Q24. What are the most common modularization mistakes? How do you avoid them?

(1) God :core:common module — a dumping ground for anything "shared." It becomes a dependency of everything, changes constantly, and destroys build cache effectiveness. Fix: split into focused modules (:core:network, :core:analytics, :core:model). (2) Overusing api configuration — exposing transitive dependencies everywhere, which means a change in a leaf module triggers recompilation across the entire graph. Fix: default to implementation, use api only when your public API exposes the dependency's types.

(3) Feature-to-feature dependencies — :feature:home depends on :feature:profile for a user avatar widget. This creates coupling and potential cycles. Fix: extract shared components to :core:ui or use the api/impl pattern. (4) Ignoring configuration time — 100 modules with eager task creation and heavy build.gradle.kts logic can make configuration slower than the monolith's full build. Fix: convention plugins, configuration cache, tasks.register(). (5) No enforcement — module boundaries exist in theory but developers add shortcuts. Fix: CI gates with module graph assertions, dependency-analysis plugin, CODEOWNERS. (6) Extracting too early — modularizing before the architecture stabilizes leads to constant module restructuring. Fix: start with package-by-feature in the monolith, extract to modules once boundaries are clear.

Q25. How do you break a circular dependency between two feature modules?

Circular dependencies (:feature:home → :feature:profile → :feature:home) violate Gradle's DAG requirement and fail the build. The root cause is always shared concepts that ended up in the wrong module. Three strategies to break the cycle:

(1) Extract to a shared module — if both features need a UserAvatar composable, move it to :core:ui. Both features depend on :core:ui instead of each other. (2) Dependency inversion — if :feature:home needs to navigate to profile, define a ProfileNavigator interface in :core:navigation. :feature:profile implements it, :feature:home consumes it, and DI binds them in :app. (3) Event/callback pattern — :feature:home emits a NavigateToProfile(userId) event. The :app module's navigation host observes events and routes to the correct feature. No direct dependency between features. The general principle: when two modules need each other, they're sharing a concept that should be extracted to a lower-level module, or they should communicate through an abstraction that lives below both of them.

Q26. What are dynamic feature modules? When should you use them and what are the trade-offs?

Dynamic feature modules (DFMs) use Play Feature Delivery to download module code on demand rather than including it in the initial APK. They're declared with com.android.dynamic-feature plugin and delivered via the Play Core SplitInstallManager API. The base :app module doesn't depend on DFMs — instead, DFMs depend on :app (inverted dependency). Code in DFMs is loaded at runtime via SplitCompat or the SplitInstallHelper.

Use when: A feature is large (10+ MB), used by a small percentage of users (e.g., AR camera, advanced editor), or required only in specific markets. DFMs reduce initial download size, which improves install conversion rates (Google data: every 6 MB reduction increases installs by 1%). Trade-offs: (1) Inverted dependency graph — DFMs depend on :app, so they can't expose types to the base module without reflection or ServiceLoader. (2) Complex navigation — you can't reference DFM classes directly; use deep links or reflection. (3) Testing complexity — DFMs require special test infrastructure. (4) Play Store dependency — only works with Play Store distribution, not sideloading. (5) Runtime failures — if the download fails, you need graceful fallback UI. For most apps, regular modularization with App Bundles (which splits by ABI, density, and language automatically) provides sufficient size reduction without DFM complexity.

Q27. How do you measure the success of a modularization effort? What metrics do you track?

Build metrics: (1) Incremental build time (the primary metric — should decrease as modules enable parallel compilation and caching). (2) Configuration time (should stay flat or decrease with convention plugins + configuration cache). (3) Build cache hit rate (should increase as modules stabilize — target 70%+ for CI). (4) Clean build time (may increase slightly due to module overhead, but incremental is what matters). Track these with Develocity (Gradle Enterprise) or --scan data over time.

Developer productivity metrics: (5) PR build time on CI (should decrease). (6) Merge conflict rate (should decrease as teams work in separate modules). (7) Time to onboard a new developer to a feature (should decrease — smaller, focused modules are easier to understand). Architecture metrics: (8) Module coupling — measure with the dependency-analysis plugin; track the ratio of api to implementation dependencies (lower is better). (9) Module cohesion — a module that changes in every PR has low cohesion and should be split. (10) Dependency graph depth — deeper graphs increase incremental build risk; keep the critical path short. Set baselines before modularization and track trends weekly. A successful modularization should show 40–60% reduction in incremental build time within the first quarter.

Q28. Your team's modular build is slower than the monolith was — what went wrong and how do you fix it?

Common causes: (1) Configuration overhead — 80 modules with eager task creation, heavy buildSrc (which invalidates all modules on any change), and no configuration cache. Fix: migrate to build-logic/ convention plugins, enable configuration cache, use tasks.register(). (2) Overuse of api — transitive dependencies mean changing one module recompiles dozens. Fix: audit with dependency-analysis-gradle-plugin, switch to implementation. (3) No build cache — without caching, modular builds do more work (module overhead + per-module compilation). Fix: enable local + remote build cache.

(4) Too many variants — 4 build types × 3 flavors = 12 variants × 80 modules = 960 variant configurations. Fix: disable unused variants in non-app modules (beforeVariants { enable = false }). (5) kapt in every module — kapt disables incremental compilation and runs a full annotation processing round per module. Fix: migrate to KSP. (6) Monolithic :core:common — one module that everything depends on, changing frequently, invalidating the entire graph. Fix: split into focused modules. Diagnose with ./gradlew assembleDebug --scan — the build scan shows configuration time, task execution timeline, cache hit rates, and the critical path. Focus optimization on the critical path — the longest chain of sequential tasks determines your minimum build time.

Dependency Management (Q29–Q36)

Q29. Explain version catalogs (libs.versions.toml) — how do they work and why are they better than ext {} variables?

Version catalogs (Gradle 7.0+, stable in 7.4) define dependencies in a TOML file (gradle/libs.versions.toml) with three sections: [versions] (version numbers), [libraries] (dependency coordinates with version references), and [plugins] (plugin IDs with versions). Gradle generates a type-safe accessor (libs.androidx.core.ktx) available in all build.gradle.kts files. Bundles group related dependencies: [bundles] compose = ["compose-ui", "compose-material3", "compose-tooling"].

Advantages over ext {}: (1) Type-safe accessors — IDE autocomplete and compile-time checking; ext variables are untyped strings. (2) Centralized — one file for all dependency versions, not scattered across build.gradle.kts files. (3) Tooling support — Dependabot, Renovate, and gradle-versions-plugin can parse TOML natively. (4) Shareable — catalogs can be published as a Gradle platform and consumed by other projects. (5) No buildSrc invalidation — changing a version in ext {} inside buildSrc invalidates all modules; changing libs.versions.toml only affects modules using the changed dependency. The TOML format is also more readable and diffable than Kotlin DSL code. Convention plugins access the catalog via project.extensions.getByType<VersionCatalogsExtension>().named("libs").

Q30. What is a BOM? How does platform() differ from enforcedPlatform()?

A BOM (Bill of Materials) is a POM file that declares dependencyManagement entries — version constraints for a set of libraries without adding them as actual dependencies. When you use platform("androidx.compose:compose-bom:2024.01.00"), Gradle imports the version constraints. You then declare individual dependencies without versions: implementation("androidx.compose.ui:ui") — the BOM provides the version.

platform() applies constraints as preferences — if another dependency brings in a higher version, Gradle's default highest-wins resolution picks the higher version. enforcedPlatform() applies constraints as forced versions — it overrides any other version, even higher ones. This is equivalent to wrapping every BOM entry in strictly. Use platform() for most cases (Compose BOM, Firebase BOM) — it ensures compatible versions while allowing legitimate upgrades. Use enforcedPlatform() when you need absolute version pinning (e.g., a security-critical library where you must control the exact version). Be cautious with enforcedPlatform() — it can downgrade transitive dependencies, causing runtime NoSuchMethodError if a library expects a newer version of its dependency.

Q31. How do you handle transitive dependency conflicts? Walk through a diamond dependency scenario.

Diamond dependency: :app → :feature:home → okhttp:4.12.0 and :app → :feature:search → okhttp:4.11.0. Both paths converge on OkHttp with different versions. Gradle's default resolution picks the highest version (4.12.0). This usually works because libraries maintain backward compatibility, but not always — 4.12.0 might remove a deprecated API that 4.11.0's consumer relies on.

Diagnosis: ./gradlew :app:dependencies --configuration runtimeClasspath shows the resolved graph with version selections marked (e.g., okhttp:4.11.0 -> 4.12.0). ./gradlew :app:dependencyInsight --dependency okhttp explains why a specific version was selected. Resolution strategies: (1) Align versions — use a BOM or version catalog to ensure all modules use the same version. (2) failOnVersionConflict() — makes the build fail on any conflict, forcing explicit resolution. (3) resolutionStrategy.force("com.squareup.okhttp3:okhttp:4.12.0") — pins a specific version globally. (4) strictly version constraint — implementation("okhttp:4.12.0!!") rejects any other version. (5) Dependency locking — locks the entire resolved graph to a file, making resolution reproducible. For large projects, use a combination of BOMs (for library families) and the dependency-analysis plugin (to detect conflicts early).

Q32. What is dependency locking? When and why should you use it?

Dependency locking captures the exact resolved versions of all dependencies into a lock file (gradle.lockfile per module or a single file). On subsequent builds, Gradle verifies that resolution produces the same versions — if a new transitive dependency appears or a version changes, the build fails until you explicitly update the lock file with ./gradlew dependencies --write-locks.

Enable per configuration: dependencyLocking { lockAllConfigurations() }. The lock file records every resolved artifact coordinate and version. Use when: (1) You need reproducible builds — without locking, a new transitive dependency published upstream can silently change your dependency graph. (2) Security-sensitive projects — locking ensures no unexpected dependency changes sneak in. (3) Compliance requirements — auditors need to verify exact dependency versions. Trade-offs: Lock files create merge conflicts when multiple PRs update dependencies simultaneously. They add maintenance overhead — every dependency update requires regenerating locks. For most Android projects, version catalogs + BOMs provide sufficient version control without the overhead of full locking. Locking is most valuable in enterprise environments or libraries where reproducibility is critical. Gradle also supports dependency verification (checksums + PGP signatures in verification-metadata.xml) for supply-chain security.

Q33. Explain the dependency analysis plugin — what problems does it detect?

The com.autonomousapps.dependency-analysis plugin (DAGP) analyzes bytecode and source code to determine actual dependency usage. It detects: (1) Unused dependencies — declared in build.gradle.kts but no classes from the dependency are used. These waste compile time and bloat the classpath. (2) Used transitive dependencies — your code uses classes from a library that you didn't declare directly (it came transitively). This is fragile — if the intermediate dependency removes its transitive, your build breaks.

(3) api vs implementation misuse — a dependency declared as api but none of its types appear in your module's public API (should be implementation), or declared as implementation but its types leak through your public API (should be api). (4) compileOnly candidates — dependencies used only for annotations or compile-time checks. (5) Processor dependencies — annotation processors declared as implementation instead of kapt/ksp. Run with ./gradlew buildHealth to get a project-wide report. The plugin is essential for maintaining dependency hygiene in large projects — without it, dependency declarations drift from actual usage over time, accumulating unused dependencies and hidden transitive usages that eventually cause build failures during refactoring.

Q34. What is the difference between compileOnly, runtimeOnly, and implementation? Give examples for each.

implementation — available at compile time and runtime. The standard configuration for most dependencies. Example: implementation(libs.retrofit) — you call Retrofit APIs in your code and need it at runtime to make HTTP calls.

compileOnly — available at compile time only, not packaged into the APK. The dependency must be provided by the runtime environment. Examples: compileOnly(libs.javax.inject) for annotation-only dependencies where Hilt provides the runtime implementation; compileOnly(libs.auto.service.annotations) for annotation processor annotations; compileOnly(libs.lombok) (though rarely used in Kotlin). Also useful for optional integrations: a library module can compile against an API without forcing consumers to include it.

runtimeOnly — not available at compile time, only included at runtime. You can't import classes from it in your source code. Examples: runtimeOnly(libs.slf4j.android) — SLF4J logging implementation selected at runtime; runtimeOnly(libs.leakcanary) in some setups where LeakCanary auto-initializes via ContentProvider; JDBC drivers in backend projects. In Android, runtimeOnly is less common but useful for logging backends, crash reporting SDKs that initialize via manifest-merged ContentProviders, and test runtime dependencies.

Q35. How do you manage dependencies in a Kotlin Multiplatform module?

KMP modules use source sets (commonMain, androidMain, iosMain) with per-source-set dependency declarations. Common dependencies go in commonMain: sourceSets { commonMain.dependencies { implementation(libs.kotlinx.coroutines.core) } }. Platform-specific dependencies go in their source set: androidMain.dependencies { implementation(libs.retrofit) }, iosMain.dependencies { implementation(libs.ktor.client.darwin) }.

The KMP Gradle plugin resolves dependencies per target — commonMain dependencies are available in all targets, platform source sets add target-specific implementations. For expect/actual patterns, the common source set declares expect classes with common dependencies, and platform source sets provide actual implementations with platform-specific dependencies. Version catalogs work seamlessly with KMP — define all versions in libs.versions.toml and reference them in source set blocks. Challenges: (1) Not all libraries have KMP artifacts — you may need expect/actual wrappers. (2) Dependency resolution differs per target — a version conflict might exist only on iOS. (3) Test dependencies need separate declaration per source set (commonTest, androidUnitTest). (4) The android() target in KMP uses AGP under the hood, so AGP version compatibility with the KMP plugin matters. Use ./gradlew :shared:dependencies to inspect per-configuration resolution.

Q36. Design a dependency management strategy for a 50-module project with 3 apps sharing common modules.

Structure: Version catalog (gradle/libs.versions.toml) as the single source of truth for all dependency versions across all 3 apps. Group related libraries into bundles (compose, networking, testing). Use BOMs for library families (Compose BOM, Firebase BOM, OkHttp BOM) to ensure internal consistency. Convention plugins in build-logic/ apply dependencies consistently — AndroidFeatureConventionPlugin adds Compose, Hilt, and navigation dependencies automatically.

Shared modules (:core:*) are consumed by all 3 apps. Publish them as a versioned internal library (Maven local or Artifactory) if the apps are in separate repositories, or use Gradle composite builds if they're in a monorepo. Dependency analysis: run DAGP on CI weekly to detect drift. Version updates: use Renovate or Dependabot with auto-merge for patch versions, manual review for minor/major. Conflict resolution: failOnVersionConflict() in the :app modules to catch conflicts early. Lock files for the 3 app modules only (not library modules) to ensure reproducible releases. Remote build cache shared across all 3 apps — since they share core modules, one app's CI run populates the cache for the others. This strategy scales because version decisions are centralized (TOML), application is automated (convention plugins), and drift is detected (DAGP + CI).

Build Performance (Q37–Q46)

Q37. Your build takes 8 minutes — walk through your investigation and optimization strategy.

Step 1: Profile — run ./gradlew assembleDebug --scan and analyze the build scan. Check the timeline: is configuration slow (>10s)? Is one task dominating execution? What's the critical path? Step 2: Identify the bottleneck category — configuration time, compilation, resource processing, dexing, or linking. Step 3: Apply targeted fixes.

Configuration (>10s): Enable configuration cache, migrate from buildSrc to build-logic/, use tasks.register(), disable unused variants. Compilation (>3min): Check for kapt (migrate to KSP), audit api vs implementation (DAGP), enable incremental compilation, check for ABI-breaking changes in frequently-modified modules. Resource processing (>1min): Use resourcePrefix to catch conflicts early, disable unused resource configurations (resConfigs "en"). Dexing (>1min): Ensure incremental dexing is enabled, check for legacy multidex issues. Cache misses: Enable local + remote build cache, debug misses with --scan. General: Increase daemon memory (-Xmx4g), enable parallel execution (org.gradle.parallel=true), use file system watching. Prioritize by impact — the build scan's critical path shows which optimizations yield the most time savings.

Q38. Explain configuration avoidance — what patterns should you avoid in build.gradle.kts?

Configuration avoidance means deferring work until it's actually needed. During configuration, Gradle evaluates every build.gradle.kts — any expensive operation here runs on every build, even if the affected tasks never execute. Avoid: (1) tasks.create() — use tasks.register() instead. (2) Resolving dependencies during configuration — configurations.getByName("implementation").resolve() triggers full resolution. Use Provider-based APIs instead. (3) Reading files — file("version.txt").readText() runs every configuration. Use providers.fileContents().

(4) Accessing Task properties eagerly — tasks.getByName("compileKotlin").outputs materializes the task. Use tasks.named("compileKotlin").flatMap { it.outputs }. (5) Heavy computation in build.gradle.kts — git hash computation, network calls, or file parsing. Wrap in a ValueSource for configuration cache compatibility. (6) afterEvaluate {} — often used to work around ordering issues but forces eager evaluation. Use Provider chains instead. (7) Iterating over all tasks — tasks.forEach {} materializes every task. Use tasks.withType<T>().configureEach {} for lazy configuration. The configuration cache enforces many of these rules — enabling it is the best way to find violations, as Gradle reports each one as a "configuration cache problem."

Q39. How does the remote build cache work? Design the cache strategy for a team of 20 developers.

The remote build cache is an HTTP-based key-value store. When a cacheable task executes, Gradle computes the cache key (hash of inputs) and stores the output as a tar archive at that key. Before executing a task, Gradle checks the remote cache — if a hit, it downloads and unpacks the output, skipping execution. The cache backend can be Develocity (Gradle Enterprise), a generic HTTP server, or cloud storage (GCS/S3 with a proxy).

Strategy for 20 developers: (1) CI writes, developers read — CI builds push to the remote cache (read-write access), developer machines only pull (read-only). This ensures cache entries are produced from clean, reproducible CI builds. (2) Cache per branch type — main branch builds populate the primary cache; feature branch builds read from main's cache but write to a separate namespace (prevents feature branch pollution). (3) Local cache as L1, remote as L2 — developers hit local cache first (instant), then remote (network latency). (4) Cache key hygiene — normalize file paths (use relative paths), pin JDK version, ensure deterministic outputs. Run --scan weekly to audit cache hit rates — target 70%+ on CI, 50%+ locally. (5) Retention policy — expire entries after 7–14 days to limit storage. (6) Monitoring — Develocity dashboards show hit rates, miss reasons, and cache size trends. The ROI is significant: with 20 developers, shared core modules are compiled once on CI and reused by everyone.

Q40. Compare kapt and KSP — why is KSP faster and how do you migrate?

kapt (Kotlin Annotation Processing Tool) works by generating Java stubs from Kotlin source, then running the Java annotation processing pipeline (javax.annotation.processing) on those stubs. This is inherently slow: (1) stub generation is a full Kotlin compilation pass, (2) the Java AP infrastructure processes all sources (not just changed ones), and (3) kapt disables Kotlin incremental compilation for the affected source set because generated code can depend on any input.

KSP (Kotlin Symbol Processing) operates directly on Kotlin compiler symbols — no stub generation, no Java AP overhead. KSP processors receive a Resolver that provides the Kotlin AST, and they declare their input dependencies explicitly, enabling incremental processing — only changed files and their dependents are reprocessed. KSP is typically 2–3x faster than kapt. Migration: (1) Check processor support — Hilt, Room, Moshi, and most major libraries support KSP. (2) Replace kapt(libs.hilt.compiler) with ksp(libs.hilt.compiler). (3) Replace id("kotlin-kapt") with id("com.google.devtools.ksp"). (4) Remove kapt {} configuration blocks. (5) Test thoroughly — KSP and kapt can produce subtly different generated code. Migrate one processor at a time. Once all processors are on KSP, remove the kapt plugin entirely to eliminate stub generation.

Q41. What is the impact of build variants on build performance? How do you minimize it?

Each build variant (build type × product flavor dimensions) multiplies the number of tasks AGP registers. With 2 build types and 2 flavor dimensions of 3 flavors each, you get 2 × 3 × 3 = 18 variants per module. Each variant registers ~40–60 tasks, so a single module has 720–1080 tasks. Across 50 modules, that's 36,000–54,000 tasks — most of which are never executed. Even with lazy registration, the Variant objects are created during configuration, consuming memory and time.

Minimization strategies: (1) Disable unused variants — in library modules, you typically only need debug and release: androidComponents { beforeVariants { if (buildType == "benchmark") enable = false } }. (2) Minimize flavor dimensions — use build config fields or runtime configuration instead of flavors when possible. (3) Use variantFilter (deprecated) or beforeVariants to disable invalid combinations (e.g., freeRelease if free tier has no release build). (4) Limit flavors to :app — library modules rarely need flavors; use missingDimensionStrategy to provide defaults. (5) Single-variant development — configure IDE to sync only one variant: android.settings.executionProfile= or use android.buildVariants in gradle.properties. The build scan's "Task creation" section shows exactly how many tasks are registered — if it's over 10,000, variant reduction is a high-priority optimization.

Q42. How do you optimize Gradle for CI? What settings differ from local development?

CI-specific settings: (1) Daemon — disable if containers are ephemeral (--no-daemon); keep enabled if build agents are reused. (2) Parallel execution — match org.gradle.workers.max to CI machine CPU cores (often higher than developer laptops). (3) Memory — CI machines often have more RAM; increase -Xmx6g for large projects. (4) Build cache — CI should write to the remote cache; configure push credentials only in CI environment variables.

(5) Configuration cache — enable with org.gradle.configuration-cache=true; CI benefits most because it runs the same tasks repeatedly. (6) Dependency caching — mount ~/.gradle/caches as a persistent volume or CI cache to avoid re-downloading dependencies. (7) Variant filtering — CI might build all variants for release but only debug for PR checks. (8) Test sharding — split instrumented tests across multiple CI jobs. (9) Build scan — always publish scans on CI for debugging slow builds. (10) Avoid clean — many CI pipelines run clean build unnecessarily, destroying incremental build benefits. Only clean when you suspect cache corruption. (11) Gradle wrapper — always use ./gradlew to ensure consistent Gradle version. The goal: CI builds should be faster than local builds because they have more resources and a warm remote cache.

Q43. What causes a build cache miss? How do you debug cache misses?

Cache misses occur when the cache key differs from any stored entry. The cache key is a hash of: task class implementation (bytecode), task input properties (file contents, strings, classpath entries), and input normalization settings. Common causes: (1) Absolute paths — a file input contains /Users/alice/project/... on one machine and /Users/bob/project/... on another. Fix: use PathSensitivity.RELATIVE on file inputs. (2) Timestamps — generated files include build timestamps, making outputs non-deterministic. Fix: strip timestamps from generated code.

(3) Non-deterministic task outputs — a code generator produces output in random order (e.g., iterating a HashSet). Fix: sort outputs deterministically. (4) JDK version mismatch — different JDK versions produce different bytecode. Fix: pin JDK via Gradle toolchains. (5) Undeclared inputs — the task reads a file not declared as an input; the cache key doesn't include it, so the cached output is stale. Fix: declare all inputs. Debugging: Run ./gradlew assembleDebug --scan and open the Build Cache section. For each missed task, the scan shows which input changed. Locally, use -Dorg.gradle.caching.debug=true to log cache key computation. Compare cache keys between two builds with ./gradlew :module:compileDebugKotlin --build-cache -Dorg.gradle.caching.debug=true 2>&1 | grep "cache key". Develocity provides a "cache miss comparison" tool that diffs two builds' cache keys input by input.

Q44. Explain parallel execution in Gradle — at what level does it parallelize and what are the limitations?

Gradle parallelizes at the task level within the execution phase. With org.gradle.parallel=true, Gradle runs tasks from different projects concurrently (up to org.gradle.workers.max threads, defaulting to CPU core count). Tasks within the same project run sequentially by default (the "project lock"), though tasks can opt into cross-project parallelism via the Worker API. The task scheduler respects the dependency graph — a task only starts when all its dependencies have completed.

Limitations: (1) Project lock — by default, only one task per project runs at a time. This prevents race conditions on shared project state but limits parallelism within a module. (2) Critical path — the longest chain of dependent tasks determines the minimum build time regardless of parallelism. If :core:network:compileKotlin → :core:data:compileKotlin → :feature:home:compileKotlin → :app:compileKotlin is a 4-minute chain, adding more cores won't help. (3) I/O contention — too many parallel tasks can saturate disk I/O, especially on spinning disks. (4) Memory pressure — each parallel task consumes heap; too many concurrent Kotlin compilations can OOM the daemon. (5) Non-parallelizable tasks — some tasks (like lint) acquire exclusive locks. The Worker API enables finer-grained parallelism within a task (e.g., compiling multiple source sets concurrently), but adoption is limited. Modularization improves parallelism by creating more independent projects that can compile simultaneously.

Q45. How does file system watching improve build performance?

File system watching (enabled by default since Gradle 7.0) uses OS-level file system notifications (inotify on Linux, FSEvents on macOS, ReadDirectoryChangesW on Windows) to track file changes between builds. Without it, Gradle must scan the entire project directory tree at the start of each build to determine which files changed — this "file system snapshotting" can take several seconds for large projects with thousands of files.

With file system watching, the Gradle daemon maintains a Virtual File System (VFS) — an in-memory representation of the file tree. Between builds, the OS notifies the daemon of file changes, and the VFS is updated incrementally. When the next build starts, Gradle already knows which files changed without scanning. This is especially impactful for incremental builds where only a few files changed — instead of scanning 50,000 files to find 3 changes, the VFS already has the answer. The improvement is typically 0.5–3 seconds per build. Caveats: (1) The VFS is per-daemon — if the daemon restarts, the snapshot is lost. (2) Some file system events can be missed (e.g., changes made by external tools while the daemon is paused), causing Gradle to fall back to a full scan. (3) Network file systems (NFS, SMB) don't support reliable watching. Configure with org.gradle.vfs.watch=true (default) and monitor with --watch-fs debug logging.

Q46. Your team reports that clean builds are fast but incremental builds are slow — what's wrong?

This is counterintuitive — incremental builds should be faster. Common causes: (1) Broken incremental compilation — kapt disables Kotlin incremental compilation. If every module uses kapt, every change triggers full recompilation. Check with --info log: look for "Incremental compilation is disabled." Fix: migrate to KSP. (2) ABI-breaking changes in hot modules — if :core:model changes frequently and uses api configuration, every change recompiles the entire downstream graph. Fix: stabilize the model module, use implementation.

(3) Build cache corruption — stale cache entries cause incorrect up-to-date checks, forcing Gradle to re-execute tasks that should be cached. Fix: clear local cache (rm -rf ~/.gradle/caches/build-cache-1). (4) Timestamp-based inputs — a task input includes a file that changes on every build (e.g., BuildConfig with a build timestamp). This invalidates the task and all downstream tasks on every build. Fix: remove dynamic values from build inputs or isolate them in a leaf module. (5) clean in the workflow — developers running clean assembleDebug out of habit. Fix: educate the team that clean destroys incremental build state. (6) IDE sync issues — Android Studio sometimes triggers full rebuilds after sync. Fix: update AGP/AS to latest stable. Diagnose with --scan — compare the "avoided" vs "executed" task counts between clean and incremental builds. In a healthy incremental build, 80%+ of tasks should be UP-TO-DATE or FROM-CACHE.

Convention Plugins & Build Logic (Q47–Q52)

Q47. Why is buildSrc an anti-pattern for large projects? What should you use instead?

buildSrc is a special Gradle directory that's compiled before any project's build.gradle.kts and added to the build script classpath. The problem: any change to any file in buildSrc invalidates the entire build script classpath, forcing reconfiguration of every module. Change a single constant in buildSrc/src/main/kotlin/Versions.kt and all 50 modules reconfigure from scratch. This destroys the configuration cache and makes incremental builds significantly slower.

The alternative: composite builds with a build-logic/ included build. Create a build-logic/ directory with its own settings.gradle.kts and build.gradle.kts. Define convention plugins as standalone Gradle plugins in this included build. In the root settings.gradle.kts, add pluginManagement { includeBuild("build-logic") }. Now, changing a convention plugin only invalidates modules that apply that specific plugin — not the entire project. The build-logic/ build is also independently cacheable and can use the configuration cache. NIA, Tivi, and most modern Android projects use this pattern. Additional benefit: convention plugins in build-logic/ can be unit-tested with GradleRunner, which is difficult with buildSrc. Migration: move buildSrc code to build-logic/, convert objects/functions to Gradle plugins, and apply them via plugins { id("my.convention.plugin") }.

Q48. Walk through implementing a convention plugin — from project setup to applying it in a module.

Step 1: Project setup. Create build-logic/convention/build.gradle.kts:

plugins { `kotlin-dsl` }
dependencies {
    compileOnly(libs.android.gradlePlugin)
    compileOnly(libs.kotlin.gradlePlugin)
}
gradlePlugin {
    plugins {
        register("androidLibrary") {
            id = "myapp.android.library"
            implementationClass = "AndroidLibraryConventionPlugin"
        }
    }
}

Step 2: Implement the plugin. Create AndroidLibraryConventionPlugin.kt:

class AndroidLibraryConventionPlugin : Plugin<Project> {
    override fun apply(target: Project) = with(target) {
        pluginManager.apply("com.android.library")
        pluginManager.apply("org.jetbrains.kotlin.android")
        extensions.configure<LibraryExtension> {
            compileSdk = 35
            defaultConfig { minSdk = 26 }
            compileOptions {
                sourceCompatibility = JavaVersion.VERSION_17
                targetCompatibility = JavaVersion.VERSION_17
            }
        }
        extensions.configure<KotlinAndroidProjectExtension> {
            compilerOptions { jvmTarget.set(JvmTarget.JVM_17) }
        }
    }
}

Step 3: Include in root. In settings.gradle.kts: pluginManagement { includeBuild("build-logic") }. Step 4: Apply in a module. In :core:network/build.gradle.kts: plugins { id("myapp.android.library") }. The module's build file shrinks from 30 lines to 5 — all configuration is centralized and consistent.

Q49. How do you access the version catalog from a convention plugin?

Convention plugins in build-logic/ don't have direct access to the type-safe libs accessor because they're compiled in a separate build. You access the catalog through the VersionCatalogsExtension:

class AndroidLibraryConventionPlugin : Plugin<Project> {
    override fun apply(target: Project) = with(target) {
        val libs = extensions
            .getByType<VersionCatalogsExtension>()
            .named("libs")

        dependencies {
            add("implementation",
                libs.findLibrary("androidx.core.ktx").get())
            add("testImplementation",
                libs.findLibrary("junit").get())
        }
    }
}

The findLibrary(), findVersion(), findBundle(), and findPlugin() methods return Optional wrappers. Call .get() to unwrap (throws if not found) or .orElse() for a fallback. The string key matches the alias in libs.versions.toml with dots replaced by dots (e.g., androidx-core-ktx in TOML → "androidx.core.ktx" in findLibrary). An alternative approach: make the build-logic/ build share the same version catalog by adding versionCatalogs { create("libs") { from(files("../gradle/libs.versions.toml")) } } in build-logic/settings.gradle.kts. This gives you type-safe accessors within the convention plugin's own build.gradle.kts (for plugin dependencies), but you still need the extension API inside the Plugin.apply() method for the target project's dependencies.

Q50. Explain build variants, flavors, and dimensions — how do they interact and what is the variant matrix?

Build types define how the app is built: debug (debuggable, no minification), release (minified, signed), custom types like benchmark. Product flavors define what the app contains: different branding, feature sets, or API endpoints. Flavors are grouped into dimensions — each dimension is an axis of variation. The variant matrix is the Cartesian product: buildTypes × dimension1 × dimension2 × ....

Example: build types = {debug, release}, dimension "tier" = {free, paid}, dimension "env" = {staging, prod}. Variants: freeStagingDebug, freeStagingRelease, freeProdDebug, freeProdRelease, paidStagingDebug, paidStagingRelease, paidProdDebug, paidProdRelease — 8 variants. Each variant gets its own source set (src/freeDebug/), resource merging, manifest merging, and task set. Source set priority (highest wins): variant > build type > flavor > main. Flavors within a dimension are mutually exclusive; flavors across dimensions combine. In library modules, variants are published — consumers must match or use matchingFallbacks. The variant count directly impacts configuration time and task count, so minimize dimensions and disable invalid combinations with beforeVariants { if (name.contains("stagingRelease")) enable = false }.

Q51. How do you test a custom Gradle plugin?

Use Gradle TestKit with GradleRunner. Create a test that sets up a temporary project directory with build.gradle.kts and settings.gradle.kts, applies your plugin, runs a task, and asserts on the output:

@Test fun `plugin applies android library configuration`() {
    val projectDir = tempDir.newFolder("test-project")
    File(projectDir, "settings.gradle.kts").writeText(
        """pluginManagement { /* ... */ }"""
    )
    File(projectDir, "build.gradle.kts").writeText(
        """plugins { id("myapp.android.library") }"""
    )
    val result = GradleRunner.create()
        .withProjectDir(projectDir)
        .withArguments("tasks", "--all")
        .withPluginClasspath()
        .build()
    assertThat(result.output).contains("compileDebugKotlin")
}

withPluginClasspath() adds your plugin's classes to the test build's classpath. Test at multiple levels: (1) Unit tests — test helper functions and configuration logic without Gradle. (2) Integration tests — use GradleRunner to verify the plugin applies correctly, tasks are registered, and builds succeed. (3) Functional tests — create realistic multi-module test projects and verify end-to-end behavior. Test edge cases: plugin applied to wrong module type (application vs library), missing required properties, interaction with other plugins. Use GradleRunner.withGradleVersion() to test compatibility across Gradle versions. For convention plugins, test that the convention is actually applied: verify compileSdk, Kotlin options, and dependency configurations match expectations.

Q52. Design the build logic architecture for a multi-app project (main app + wear app + TV app) sharing common modules.

Structure: build-logic/ as an included build with convention plugins for each module type. Define plugins: myapp.android.application (base app config), myapp.android.library (library modules), myapp.android.feature (feature modules with Compose + Hilt), myapp.kotlin.library (pure Kotlin modules), myapp.android.compose (Compose compiler setup), myapp.android.wear (Wear OS specifics), myapp.android.tv (Leanback/TV specifics).

Shared modules (:core:network, :core:data, :core:domain, :core:model) apply myapp.android.library or myapp.kotlin.library. Feature modules apply myapp.android.feature. App modules compose their specific plugin: :app:mobile applies myapp.android.application, :app:wear applies myapp.android.application + myapp.android.wear, :app:tv applies myapp.android.application + myapp.android.tv. Version catalog is shared across all apps. Each app module depends on the features it needs — the mobile app might include all features while the wear app includes only :feature:notifications and :feature:health. Build variants: mobile has free/paid flavors, wear and TV have only debug/release. Use missingDimensionStrategy in shared modules to handle flavor mismatches. Remote build cache is shared — a core module compiled by the mobile CI is reused by the wear CI.

Common Problems & System Design (Q53–Q60)

Q53. How do you handle navigation between feature modules that can't depend on each other?

Three approaches, from simplest to most robust: (1) Deep link / URI-based navigation — each feature registers a deep link route (e.g., myapp://profile/{userId}). Features navigate by URI: navController.navigate("myapp://profile/$userId"). No compile-time dependency needed. Works with Navigation Component out of the box. Downside: no type safety, string-based routing is error-prone.

(2) Navigation interface pattern — define a :core:navigation module with interfaces: interface ProfileNavigator { fun navigateToProfile(userId: String) }. :feature:profile implements it, :feature:home injects it via Hilt. :app provides the binding. Type-safe, testable (inject a fake navigator in tests), but requires DI wiring. (3) Shared route definitions — :core:navigation contains route sealed classes or @Serializable route objects (Navigation Compose 2.8+). Features reference routes without depending on each other: navController.navigate(ProfileRoute(userId)). The :app module's NavHost maps routes to feature composables. This is the recommended approach for Navigation Compose — type-safe, no DI overhead, and the navigation graph is centrally defined. For large apps, combine approaches 2 and 3: shared routes for simple navigation, navigator interfaces for complex flows with callbacks.

Q54. Explain manifest merging — what are common conflicts and how do you resolve them?

AGP merges manifests from all modules, build variants, dependencies, and the main source set into a single AndroidManifest.xml. The merge priority (highest wins): build variant manifest > main source set > library dependencies > transitive dependencies. The merger uses a set of rules: elements with the same android:name are merged (attributes combined), conflicting attributes cause build errors unless resolved with merge markers.

Common conflicts: (1) Duplicate <activity> or <service> — two libraries declare the same component. Fix: tools:node="remove" on the unwanted one. (2) minSdkVersion mismatch — a library requires minSdk 26 but your app targets 21. Fix: tools:overrideLibrary="com.example.library" (risky — you must handle the API gap at runtime). (3) Conflicting android:allowBackup — your app sets false, a library sets true. Fix: tools:replace="android:allowBackup" in your manifest. (4) Duplicate permissions — multiple libraries request the same permission. This is harmless (duplicates are merged), but unwanted permissions need tools:node="remove". (5) ContentProvider authority conflicts — two libraries use the same authority string. Fix: use ${applicationId} placeholder in authorities. Debug with ./gradlew :app:processDebugManifest and inspect build/intermediates/merged_manifests/debug/AndroidManifest.xml. The merge report at build/outputs/logs/manifest-merger-debug-report.txt shows exactly how each element was resolved.

Q55. How does R8/ProGuard work with multiple modules? What are consumer ProGuard rules?

R8 (the default shrinker/optimizer since AGP 3.4) runs only in the application module during release builds. It processes the entire app's bytecode — all modules merged together. Library modules are not individually shrunk; their code is included in the app's R8 pass. This means R8 has full visibility of the entire call graph and can aggressively remove unused code across module boundaries.

Consumer ProGuard rules (consumerProguardFiles) are keep rules bundled inside a library's AAR. When R8 processes the app, it automatically includes these rules. Libraries use them to protect their public API, reflection-based code, and serialization classes. For example, a :core:network module with Retrofit interfaces needs: -keep interface com.example.network.api.** { *; } in its consumer rules. Without this, R8 would remove the interface methods (they're never called directly — Retrofit uses reflection/proxy). Each module declares consumer rules in build.gradle.kts: android { defaultConfig { consumerProguardFiles("consumer-rules.pro") } }. Common pitfall: putting keep rules in the app's proguard-rules.pro instead of the library's consumer rules — this works but doesn't travel with the library if it's published or reused. Another pitfall: overly broad keep rules (-keep class ** { *; }) that disable shrinking entirely. Use -printusage to see what R8 removes and -whyareyoukeeping to debug unexpected keeps.

Q56. How does Hilt work across modules? What are the common pitfalls?

Hilt generates a single component hierarchy in the :app module (the @HiltAndroidApp entry point). Each module contributes bindings via @Module/@InstallIn — Hilt's annotation processor (kapt/KSP) scans each module independently and generates metadata. At app compilation, Hilt aggregates all module metadata and generates the final DaggerMyApp_HiltComponents class. This means modules don't need to know about each other's bindings — Hilt wires everything at the app level.

Pitfalls: (1) Missing Hilt plugin in a module — if a module provides @Module classes but doesn't apply the Hilt Gradle plugin, the metadata isn't generated and bindings are silently missing. Runtime crash: MissingBinding. (2) Scope mismatch — a @Singleton binding in :feature:home that depends on a @ViewModelScoped binding. Hilt fails at compile time with a scope violation. (3) Duplicate bindings — two modules provide the same type without @Qualifier. Fix: use @Named or custom qualifiers. (4) kapt performance — Hilt with kapt runs annotation processing in every module. Migrate to KSP for 2x speedup. (5) Test module isolation — @UninstallModules in tests requires the test to know about the production module, creating a coupling. Use @TestInstallIn instead for global test replacements. (6) Dynamic feature modules — Hilt doesn't natively support DFMs; you need @EntryPoint interfaces to access the app's component from DFM code.

Q57. How do you share test utilities across modules without polluting the main source set?

Three approaches: (1) :core:testing module — a dedicated module containing shared test fakes, builders, rules, and utilities. Modules depend on it with testImplementation(project(":core:testing")). This is the NIA approach — :core:testing contains TestDispatcherRule, fake repositories, and Compose test helpers. The module uses implementation (not testImplementation) for its own dependencies since it's a regular module whose consumers use it in their test configurations.

(2) testFixtures source set — Gradle's built-in test fixtures feature. Enable with java { withTestFixtures() } (or android { testFixtures { enable = true } } for Android modules). Create src/testFixtures/kotlin/ with fake implementations. Consumers use testImplementation(testFixtures(project(":core:data"))). This keeps fakes co-located with the module they fake, which is better for maintenance. (3) Shared test artifacts — publish test JARs from modules and consume them. This is the pre-testFixtures approach and is more complex. Recommendation: use testFixtures for module-specific fakes (a fake UserRepository lives in :core:data's test fixtures) and :core:testing for cross-cutting utilities (test rules, Compose helpers, coroutine test setup). Never put test utilities in the main source set — they'll be included in the production APK.

Q58. What are non-transitive R classes? Why are they important for modular projects?

Historically, each module's R class contained all resource IDs — its own plus all transitive dependencies'. A module with 10 dependencies would have a massive R class with thousands of fields from libraries it doesn't directly use. This caused: (1) slow compilation (generating and compiling huge R classes), (2) resource name collisions (two dependencies defining R.string.title silently shadow each other), and (3) unclear resource ownership.

Non-transitive R classes (android.nonTransitiveRClass=true in gradle.properties, default since AGP 8.0) restrict each module's R class to only its own resources. To reference a resource from a dependency, you use the dependency's fully qualified R class: com.example.core.ui.R.drawable.ic_logo instead of just R.drawable.ic_logo. Benefits: (1) Faster builds — R class generation is O(n) instead of O(n²) across the module graph. (2) Explicit ownership — you always know which module owns a resource. (3) No silent shadowing — resource collisions become compile errors instead of silent bugs. (4) Smaller R classes — less bytecode to compile and dex. The migration can be tedious — you need to add explicit imports for cross-module resources. Use Android Studio's refactoring tools or the resource-prefix convention to make the transition smoother. This is one of the most impactful build performance improvements for modular projects.

Q59. A library module works fine in debug but crashes in release — how do you debug it?

Release-only crashes are almost always caused by R8/ProGuard code shrinking. R8 removes classes, methods, and fields it considers unused — but reflection, serialization, and framework callbacks aren't visible in the static call graph. Debugging steps:

(1) Check the stack trace — ClassNotFoundException, NoSuchMethodError, or NoSuchFieldException point to R8 removing something needed at runtime. (2) Deobfuscate — use the mapping.txt file (at build/outputs/mapping/release/) with retrace to get readable stack traces. (3) Identify the missing class/method — add a targeted keep rule: -keep class com.example.MyClass { *; }. (4) Check consumer ProGuard rules — the library module might be missing rules for its reflection-based APIs (Retrofit interfaces, Room DAOs, Kotlin serialization classes). (5) Use -printusage — outputs all removed code; search for the missing class. (6) Use -whyareyoukeeping — if a class is unexpectedly kept or removed, this shows the reasoning. (7) Test with minifyEnabled true in debug — create a debugMinified build type to catch R8 issues during development. Other release-only causes: missing debuggable false assumptions, StrictMode disabled, different code paths for BuildConfig.DEBUG, and native library stripping. Always test release builds in CI before shipping.

Q60. Design a testing strategy for a 30-module project — unit tests, integration tests, and CI pipeline.

Unit tests (per module): Each module has src/test/ with JVM-based tests. :core:domain tests use cases with fake repositories. :core:data tests repositories with fake data sources. :feature:* tests ViewModels with fake use cases. Use :core:testing for shared fakes and testFixtures for module-specific fakes. Run with ./gradlew testDebugUnitTest. Target: every module has >80% coverage on business logic.

Integration tests: :core:database tests Room DAOs with in-memory database (Robolectric or instrumented). :core:network tests API parsing with MockWebServer. Feature modules test screen-level behavior with Compose testing (createComposeRule) + fake dependencies. End-to-end tests: In :app's androidTest/, test critical user flows across features. Use Hilt test rules with @TestInstallIn to swap real backends for fakes.

CI pipeline: (1) PR checks (fast, <5 min) — run only affected module tests using ./gradlew test --changed or a custom task that detects changed modules from git diff. Lint check on changed modules. (2) Merge to main (medium, <15 min) — full unit test suite across all modules, parallelized across CI workers. Build release APK to catch R8 issues. (3) Nightly (thorough, <45 min) — instrumented tests on Firebase Test Lab, full lint, dependency analysis, coverage report. Optimization: Remote build cache shared between CI jobs — test compilation is cached, only test execution runs. Shard instrumented tests across multiple devices. Use Gradle's --parallel and --max-workers tuned to CI machine specs. Track test execution time trends — a test suite that grows from 5 to 15 minutes signals the need for better test isolation or parallelization.
