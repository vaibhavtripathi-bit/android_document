60 Staff Engineer questions covering Gradle internals, module architecture, modularization strategies, dependency management, build performance, convention plugins, and common problems & solutions

Gradle Internals (Q1–Q10)

Q1. Explain Gradle's three build phases — what happens in each and why does it matter for performance?

Initialization evaluates settings.gradle.kts to determine which projects participate in the build — it creates a Settings object, resolves include() calls, and builds the project hierarchy. Configuration evaluates every build.gradle.kts for every included project: plugins are applied, dependencies are declared, and tasks are registered (but not executed). Execution runs only the tasks in the requested task graph, determined by the task you invoked and its dependency chain.

Performance-wise, initialization is usually fast (milliseconds), but configuration is the silent killer — it runs for every project on every build, even if only one module changed. In a 200-module project, configuration can take 10–20 seconds. This is why configuration avoidance (using tasks.register() instead of tasks.create(), lazy Property/Provider APIs) is critical — it defers work from configuration to execution time, meaning tasks that aren't in the execution graph never pay the configuration cost. The configuration cache (Gradle 8+) serializes the entire task graph after the first build and skips both initialization and configuration on subsequent builds when inputs haven't changed, reducing overhead to near-zero.

A common anti-pattern is doing I/O during configuration — resolving dependencies, reading files, or executing shell commands. These run on every build invocation regardless of which task you're running. At L6, you should know that AGP registers hundreds of tasks per variant during configuration, so minimizing variant count directly reduces configuration time. Use ./gradlew assembleDebug --scan to see the phase breakdown — the build scan's "Performance" tab shows exact time spent in each phase.

Q2. What is the difference between tasks.register() and tasks.create()? Why does it matter?

tasks.create() eagerly instantiates and configures the task during the configuration phase — the task object is allocated, its @TaskAction closure is set up, and all configuration lambdas run immediately. tasks.register() returns a TaskProvider — the task is only instantiated when something actually needs it (another task depends on it, or it's in the execution graph).

In a 200-module project with ~40 tasks per module, eager creation means 8,000 task objects are allocated and configured on every build, even if you're only running :app:assembleDebug which touches maybe 400 tasks. With lazy registration, the other 7,600 tasks are never created. Google measured 10–25% configuration time reduction just from switching AGP's internal tasks to lazy registration. The TaskProvider also participates in Gradle's Provider chain — you can wire task outputs to other task inputs without materializing either task until execution:

val generateTask = tasks.register<GenerateSourcesTask>("generate") {
    outputDir.set(layout.buildDirectory.dir("generated"))
}
tasks.register<CompileTask>("compile") {
    sourceDir.set(generateTask.flatMap { it.outputDir })
}

Convention plugins should always use tasks.register(). The only reason tasks.create() still exists is backward compatibility — it's effectively deprecated for new code.

Q3. How does Gradle's dependency resolution work? What happens when two modules request different versions of the same library?

Gradle resolves dependencies in a multi-step pipeline. First, it builds a dependency graph from all declared dependencies across configurations (implementation, api, etc.), following transitive edges. When it encounters a conflict — say :feature-a requests OkHttp 4.11 and :feature-b requests OkHttp 4.12 — it applies a conflict resolution strategy. The default strategy is newest-wins: Gradle selects 4.12 and uses it everywhere. This differs from Maven's "nearest-first" approach.

You can override this with resolutionStrategy.force() or failOnVersionConflict() (which makes the build fail instead of silently upgrading). Gradle also supports rich version declarations: strictly rejects any upgrade beyond the declared version, require is the default behavior, and prefer is a soft hint. BOMs (platform()) inject version constraints without adding dependencies. After conflict resolution, Gradle computes a resolved dependency graph, downloads missing artifacts from declared repositories (checking local cache first via ~/.gradle/caches/modules-2), and makes them available on the appropriate classpaths.

The dependencyInsight task is invaluable for debugging: ./gradlew :app:dependencyInsight --dependency okhttp --configuration runtimeClasspath shows exactly why a particular version was selected and which path forced the upgrade. For large projects, centralize repository declarations in settings.gradle.kts using dependencyResolutionManagement { repositoriesMode.set(RepositoriesMode.FAIL_ON_PROJECT_REPOS) } to prevent modules from adding rogue repositories. Gradle Module Metadata (GMM) provides richer constraint information than POM files, including variant-aware resolution and capability conflicts.

Q4. Explain the difference between api and implementation — how does it affect compilation and build performance?

Both api and implementation add a dependency to the module's compile and runtime classpaths, but they differ in transitive visibility. api exposes the dependency to any module that depends on the declaring module — if :core:network declares api(libs.retrofit), then :feature:search (which depends on :core:network) can directly use Retrofit classes without declaring its own dependency. implementation hides the dependency — consumers can't see it at compile time (though it's still on the runtime classpath).

The build performance impact is significant: when a library exposed via api changes its ABI (public method signatures), all transitive consumers must be recompiled. With implementation, only the declaring module recompiles. In a deep dependency graph :app → :feature → :core, if :core uses api(retrofit), changing Retrofit's version recompiles all three modules; with implementation(retrofit), only :core recompiles. The rule of thumb: use implementation by default, switch to api only when the dependency's types appear in your module's public API (return types, parameter types, supertypes):

// :core:network — Retrofit type exposed in public API → use api
fun getClient(): OkHttpClient  // OkHttpClient is in the public signature
api(libs.okhttp)

// :core:network — Retrofit used internally only → use implementation
internal fun fetchData(): MyModel  // Retrofit types not in public API
implementation(libs.retrofit)

In practice, 90%+ of dependencies should be implementation. Misusing api is one of the most common causes of slow incremental builds in modular projects — the dependency analysis plugin (com.autonomousapps.dependency-analysis) can detect this automatically and suggest the correct configuration.

Q5. What is the Gradle Daemon? How does it improve build performance and when should you disable it?

The Gradle Daemon is a long-lived JVM process that persists between builds. Without it, every ./gradlew invocation would cold-start a JVM, load all Gradle classes, and apply plugins from scratch — easily adding 5–10 seconds. The Daemon keeps the JVM warm: class metadata is loaded, JIT compilation has optimized hot paths, and the file system snapshot is cached in memory. After the first build, subsequent builds benefit from JIT-compiled code paths, in-memory dependency resolution caches, and file system watching state.

The Daemon times out after 3 hours of inactivity by default. You might disable it in CI environments where each build runs in a fresh container (no reuse benefit) and the Daemon's memory overhead is wasted — though modern CI practice is to keep Daemons alive across builds on the same agent. Gradle 8+ also supports file system watching via the Daemon, which uses OS-level file watchers (inotify on Linux, FSEvents on macOS) to track file changes between builds instead of re-scanning the entire file tree. The Daemon's heap is controlled by org.gradle.jvmargs in gradle.properties — typically 2–4 GB for large Android projects. Multiple Daemon instances can coexist if different projects use different Gradle versions or JVM arguments. Use gradle --status to check running Daemons and gradle --stop to kill them. Watch for Daemon expiration due to memory pressure — if the Daemon runs out of heap, Gradle kills it and starts a new one, losing all JIT and cache benefits.

Q6. How does incremental compilation work in Kotlin? What is ABI change detection?

The Kotlin compiler tracks dependencies between source files at the class level. When you modify a file, the compiler checks whether the change affects the ABI (Application Binary Interface) — the set of public/internal declarations visible to other files: class signatures, method signatures, public property types, and supertype lists. If the change is ABI-compatible (e.g., you modified a private method body or changed a local variable), only the modified file is recompiled. If the ABI changed (e.g., you added a public method or changed a return type), the compiler identifies all files that depend on the changed declarations and recompiles them too — this is the "changed files + dependents" strategy.

The dependency information is stored in .kotlin/build-history and .kotlin/caches directories. Kotlin's incremental compilation is more conservative than Java's — it uses a coarser dependency model (file-level rather than method-level), so a single ABI change in a heavily-imported utility file can trigger widespread recompilation. The kotlin.incremental.useClasspathSnapshot=true flag (default since Kotlin 1.7) enables classpath snapshots, which detect ABI changes in upstream modules more precisely — instead of recompiling everything when a dependency module changes, it compares the ABI snapshot and only recompiles files that actually used the changed declarations. This is critical for modular builds where upstream module changes are frequent.

A common pitfall: using kapt disables incremental compilation for files with annotations unless kapt.incremental.apt=true is set and the processor supports it — this is one reason KSP (which is fully incremental by default) is strongly preferred. Another subtlety: adding a public function to a heavily-imported file (like a utility class in :core:common) is an ABI change that triggers recompilation of every file that imports from that file — even if no file uses the new function. This is why minimizing the public API surface of shared modules is critical for build performance.

Q7. Explain the build cache — how does it work, what is the cache key, and how do local and remote caches differ?

The build cache stores task outputs keyed by a hash of the task's inputs. The cache key is computed from: the task class name, the task's input properties (file contents, not paths — so relocatable), the task's classpath (implementation), and the task action's implementation hash. When a task is about to execute, Gradle computes the cache key and checks the cache — if a hit is found, outputs are restored from cache instead of executing the task.

The local cache lives on disk (~/.gradle/caches/build-cache-1/) and benefits a single developer — switching branches and rebuilding hits the local cache for unchanged modules. The remote cache (backed by an HTTP service like Gradle Enterprise or a simple HTTP server) is shared across the team and CI — when CI builds a commit, all developers pulling that commit get cache hits for every module. The remote cache is transformative for large projects: a developer's first build after git pull can be 80–90% cache hits.

Cache misses happen when: inputs change (obvious), the task implementation changes (plugin upgrade), absolute paths leak into inputs (non-relocatable), or system properties/environment variables are accidentally captured as inputs. Tasks must be annotated with @CacheableTask and use proper input/output annotations (@InputFiles, @OutputDirectory, @PathSensitive) for caching to work correctly. The --scan flag with Gradle Enterprise shows exactly why each task missed the cache.

Q8. What is the configuration cache? How does it differ from the build cache?

The build cache caches task outputs — it skips execution of individual tasks whose inputs haven't changed. The configuration cache caches the entire configured task graph — it skips the initialization and configuration phases entirely. On the first run, Gradle evaluates all build scripts, resolves plugins, configures all tasks, builds the task graph, and serializes the result to .gradle/configuration-cache/. On subsequent runs (if no build scripts, gradle.properties, or included builds changed), Gradle deserializes the cached graph and jumps straight to execution.

For a 200-module project where configuration takes 15 seconds, the configuration cache reduces that to ~1 second. The configuration cache imposes constraints: build scripts cannot reference mutable global state, Project objects cannot be stored in task state (they're not serializable), and all task inputs must be captured via the Provider/Property API. Common violations include:

// BAD — captures Project in task action (not serializable)
tasks.register("myTask") {
    doLast { println(project.name) }
}
// GOOD — capture value at configuration time, store as String
tasks.register("myTask") {
    val name = project.name
    doLast { println(name) }
}
// BEST — use Provider for deferred resolution
tasks.register("myTask") {
    val dir = project.layout.buildDirectory
    doLast { println(dir.get()) }
}

AGP has supported the configuration cache since AGP 7.4+. Enable with org.gradle.configuration-cache=true in gradle.properties. Use --configuration-cache-problems=warn during migration to identify violations without failing the build. The cache is invalidated when build scripts change, gradle.properties changes, or included build logic changes — but not when source code changes, which is the common case for incremental builds.

Q9. How does AGP (Android Gradle Plugin) work internally? What tasks does it register and how do build variants affect task count?

AGP is a Gradle plugin (com.android.application / com.android.library) that registers Android-specific tasks during the configuration phase. For each build variant (combination of build type × product flavors), AGP registers a full pipeline: generate{Variant}Sources (AIDL, BuildConfig, data binding), merge{Variant}Resources (AAPT2 compile + link), compile{Variant}Kotlin, merge{Variant}Assets, package{Variant}Resources, dex{Variant} (D8/R8), merge{Variant}Dex, package{Variant} (APK assembly), and sign{Variant}.

A single module with 2 build types (debug/release) and 2 flavors (free/paid) has 4 variants, each generating ~30–40 tasks — that's 120–160 tasks per module. In a 50-module project, you're looking at 6,000–8,000 tasks. AGP uses the Variant API (stabilized in AGP 7.0+) to let plugins hook into the variant pipeline:

androidComponents {
    onVariants(selector().withBuildType("release")) { variant ->
        variant.packaging.resources.excludes.add("META-INF/*.kotlin_module")
    }
    beforeVariants(selector().withFlavor("environment" to "staging")) {
        enable = buildType != "release"  // disable stagingRelease
    }
}

Internally, AGP uses a TaskManager hierarchy (ApplicationTaskManager, LibraryTaskManager) that creates task instances and wires their inputs/outputs. The variant-aware dependency resolution ensures that debugImplementation dependencies only appear on the debug variant's classpath. Minimizing variants via beforeVariants { enable = false } is one of the most effective build performance optimizations. AGP also provides the artifacts API for transforming intermediate build artifacts (manifests, classes, resources) without internal API hacks — this is the supported way to add custom build steps per variant.

Q10. What is the Provider/Property API in Gradle? Why is it important for configuration avoidance?

The Provider<T> API represents a value that is computed lazily — it's not resolved until .get() is called, which ideally happens at execution time, not configuration time. Property<T> extends Provider and adds mutability — you can .set() it during configuration and .get() it during execution. Specialized variants include RegularFileProperty, DirectoryProperty, ListProperty, MapProperty, and SetProperty.

The key benefit is configuration avoidance: when you wire task A's output to task B's input via Providers, Gradle establishes a dependency without resolving either value. If task B isn't in the execution graph, task A's output is never computed. The Provider API also enables task dependency inference — Gradle automatically adds a task dependency when you connect a Provider from one task to another's input:

abstract class GenerateTask : DefaultTask() {
    @get:OutputDirectory abstract val outputDir: DirectoryProperty
}
abstract class CompileTask : DefaultTask() {
    @get:InputDirectory abstract val sourceDir: DirectoryProperty
}
val gen = tasks.register<GenerateTask>("generate") {
    outputDir.set(layout.buildDirectory.dir("gen"))
}
tasks.register<CompileTask>("compile") {
    sourceDir.set(gen.flatMap { it.outputDir }) // auto-dependency, no dependsOn()
}

For the configuration cache, Providers are essential because they're serializable — Gradle can serialize the wiring and restore it without re-evaluating build scripts. Convention plugins should exclusively use the Provider API for all file paths, properties, and inter-task wiring. The convention() method sets a default value that can be overridden — it replaces the old pattern of setting defaults in the task constructor. The API also includes map() and flatMap() for transformations, zip() for combining providers, and fileProperty()/directoryProperty() for file-based values. Configuration avoidance via the Provider API is the single most impactful pattern for build performance in large projects.

Module Architecture (Q11–Q20)

Q11. What are the benefits of modularization? When does it NOT make sense?

Modularization provides: (1) Build performance — Gradle compiles modules in parallel and caches outputs independently, so changing one module doesn't recompile the entire app. (2) Code ownership — teams own specific modules with clear boundaries, reducing merge conflicts. (3) Encapsulation — internal visibility in Kotlin enforces that implementation details don't leak across module boundaries. (4) Reusability — shared modules can be consumed by multiple apps (phone, wear, TV). (5) Testability — modules can be tested in isolation with fake implementations. (6) Dynamic delivery — feature modules can be delivered on-demand via Play Feature Delivery.

However, modularization does NOT make sense when: the project is small (under ~15K lines) and a single team maintains it — the overhead of managing module boundaries, build configuration, and inter-module navigation outweighs the benefits. Over-modularization creates its own problems: explosion of build.gradle.kts files, longer configuration times (each module adds ~50–100ms), complex dependency graphs that are hard to reason about, and excessive boilerplate for inter-module communication. The sweet spot is typically 10–30 modules for a medium app, 30–100 for a large app. Start with a monolith, extract modules when you feel pain (slow builds, merge conflicts, unclear ownership). A well-structured single module with clean package boundaries and internal visibility can be just as maintainable as a multi-module project for a small team — modularization is a tool, not a goal.

Q12. Describe the module types in a well-architected Android app — what does each contain and what are the dependency rules?

A well-structured modular app typically has four layers: (1) :app — the application module that wires everything together; it depends on all feature modules, provides the Application class, Hilt's @HiltAndroidApp, and the navigation graph. It should contain minimal code. (2) Feature modules (:feature:search, :feature:profile) — each contains UI (Composables/Fragments), ViewModels, and feature-specific logic. Feature modules depend on :core modules but never on other feature modules — this prevents horizontal coupling.

(3) Core modules (:core:network, :core:database, :core:ui, :core:domain, :core:data) — shared infrastructure and business logic. :core:domain is a pure Kotlin module (no Android dependencies) containing use cases and repository interfaces. :core:data implements those interfaces using :core:network and :core:database. (4) Common/shared modules (:core:common, :core:model) — pure data classes, utility functions, and shared types. (5) Testing modules (:core:testing, :core:testing-android) — shared test utilities, fakes, and test rules consumed via testImplementation.

The dependency rule is strictly unidirectional: :app → :feature:* → :core:* → :core:model. No cycles, no feature-to-feature dependencies. This ensures that changes in one feature module never trigger recompilation of another feature module. The graph should be as wide and shallow as possible to maximize build parallelism — all feature modules compile simultaneously once their core dependencies are ready.

Q13. Explain the difference between package-by-feature and package-by-layer — which do you prefer and why?

Package-by-layer groups code by technical role: all repositories in one package, all ViewModels in another, all UI in a third. This was the default Android approach for years but scales poorly — a single feature's code is scattered across many packages, making it hard to understand, modify, or extract. Package-by-feature groups all code for a feature together: the search feature's ViewModel, repository, UI, and models live in the same package (or module). This aligns with how developers think about changes ("I'm working on search") and maps naturally to module boundaries.

For modular projects, package-by-feature is strictly superior because each feature module already enforces the grouping at the module level, and within a module you can further organize by layer if needed. The hybrid approach — feature modules containing internal layer packages — gives you the best of both: feature cohesion at the module level, clean internal organization at the package level. Google's Now in Android sample uses exactly this pattern: :feature:search contains ui/, domain/, and data/ packages internally.

The key insight is that package-by-feature maximizes cohesion (related code is together) and minimizes coupling (features don't know about each other), which are the two fundamental principles of good modular design. Kotlin's internal visibility modifier is particularly powerful in this context — it restricts classes to their module, providing true encapsulation that Java's package-private could never achieve across packages.

Q14. What is Google's recommended module architecture (Now in Android)? Walk through the dependency graph.

Google's Now in Android (NiA) uses a layered modular architecture with strict dependency rules. The graph flows: :app → :feature:* → :core:*. Feature modules include :feature:foryou, :feature:interests, :feature:bookmarks, :feature:topic, and :feature:search. Each feature depends on :core:data (repositories), :core:designsystem (shared Compose components), :core:ui (shared UI utilities), and :core:model (data classes).

:core:data implements repository interfaces from :core:domain and depends on :core:network (Retrofit) and :core:database (Room). :core:domain is a pure Kotlin module — no Android dependencies — containing use cases and repository interfaces, depending only on :core:model. :core:designsystem contains the Material3 theme, custom components, and icons. NiA also has :core:testing (shared test utilities), :core:datastore (Proto DataStore), and :sync (WorkManager sync). Convention plugins live in build-logic/ as a composite build.

The key principle: no feature module depends on another feature module, and the dependency graph is a DAG. This maximizes parallel compilation — all feature modules compile simultaneously once their core dependencies are ready. The architecture scales because adding a new feature is just adding a new :feature:X module with the same dependency pattern — no existing modules need modification. Dependency inversion is central: feature modules depend on abstractions (:core:domain interfaces), not implementations (:core:data classes), enabling testing with fake repositories without pulling in Retrofit, Room, or any infrastructure dependency.

Q15. How do you decide the right module granularity? What are the signs of over-modularization vs under-modularization?

Signs of under-modularization: build times are long because a small change triggers recompilation of a large module; merge conflicts are frequent because multiple teams edit the same module; you can't test a feature in isolation; internal visibility is useless because everything is in one module. Signs of over-modularization: you have modules with only 2–3 files; configuration time dominates build time (each module adds ~50–100ms of configuration overhead); the dependency graph is so complex that no one can reason about it; inter-module communication requires excessive boilerplate for trivial interactions.

A good heuristic: each module should represent a deployable concern — something a team could own, test, and release independently. For features, one module per screen or user flow is typical. For core modules, group by technical domain (networking, database, UI components). Measure the impact: if splitting a module doesn't improve parallel build time (because the two halves are always built together anyway) and doesn't improve team autonomy, the split adds complexity without benefit. Tools like Gradle's --scan and module graph visualization help identify modules that are always on the critical path together and could be merged. Google's internal guidance suggests 10–50 modules for a medium app, 50–200 for a large app — beyond 200, you need significant build infrastructure investment (remote cache, convention plugins, module ownership tooling, custom CI).

Q16. Explain the api/impl module split pattern — when and why would you use it?

The api/impl split separates a module's public contract from its implementation. :core:network-api contains interfaces (ApiService, NetworkResult) and data classes, while :core:network-impl contains the Retrofit/OkHttp implementation. Feature modules depend on :core:network-api (lightweight, stable), and only :app depends on :core:network-impl. Benefits: (1) Build performance — when the Retrofit implementation changes, only :core:network-impl and :app recompile; feature modules are untouched. (2) Testability — tests provide fake implementations without pulling in the real implementation's dependencies. (3) Substitutability — swap implementations by changing only the DI wiring:

// In :core:network-impl
@Module @InstallIn(SingletonComponent::class)
abstract class NetworkModule {
    @Binds abstract fun bindApiService(impl: RetrofitApiService): ApiService
}

The trade-off is more modules and more boilerplate — every public type needs to be in the API module. Use this pattern for modules with heavy implementation dependencies (networking, database, analytics) or modules that change frequently. Don't use it for lightweight modules like :core:model where the "implementation" is just data classes. At Google scale, the api/impl split is standard for infrastructure modules — it's one of the most effective techniques for reducing the build critical path in large projects.

Q17. How do you enforce module boundaries? What tools and techniques prevent boundary violations?

(1) Kotlin internal visibility — the most powerful tool; internal classes/functions are invisible outside their module, enforced by the compiler. This is strictly stronger than Java's package-private because it works at the module level. (2) Dependency rules — if :feature:search doesn't declare a dependency on :feature:profile, it physically cannot import classes from it. Gradle enforces this at compile time. (3) Architectural fitness functions — the dependency analysis plugin detects unused dependencies, misused api vs implementation, and undeclared transitive dependencies.

(4) Custom lint checks — you can write Lint rules that flag forbidden imports or dependency patterns at the IDE level. (5) Module graph validation — CI scripts can parse settings.gradle.kts and all build.gradle.kts files to verify the dependency graph matches the intended architecture (no feature→feature edges, no cycles). Tools like gradle-module-dependency-graph can generate visual graphs for review. (6) Convention plugins — by centralizing build configuration, you ensure all modules follow the same rules.

(7) Konsist architectural tests — write test assertions that verify module boundaries:

@Test fun `domain module has no Android imports`() {
    Konsist.scopeFromModule(":core:domain")
        .files
        .assertFalse {
            it.hasImport { import ->
                import.name.startsWith("android.")
            }
        }
}

@Test fun `feature modules don't depend on other features`() {
    Konsist.scopeFromModule(":feature:search")
        .files
        .assertFalse {
            it.hasImport { import ->
                import.name.contains(".feature.") &&
                    !import.name.contains(".feature.search")
            }
        }
}

(8) Code review discipline — automated checks catch most violations, but architectural intent (should this class be in :core:domain or :feature:search?) requires human judgment. At Google scale, module owners are defined in OWNERS files, and changes to a module's public API require approval from the module owner.

Q18. What is the role of :core:domain and why should it be a pure Kotlin module?

:core:domain contains the app's business logic: use cases (interactors), repository interfaces, and domain-level models. It defines what the app does without specifying how. Making it a pure Kotlin module (org.jetbrains.kotlin.jvm plugin, no com.android.library) has several benefits: (1) Fast compilation — no AAPT2, no resource merging, no manifest processing. (2) Testability — unit tests run on the JVM without Robolectric. (3) Enforced separation — you physically cannot import android.content.Context. (4) Reusability — the domain module can be shared with KMP targeting iOS, desktop, or server. (5) Stability — domain logic changes less frequently, so fewer recompilations cascade through the graph.

// :core:domain — pure Kotlin, no Android
class GetArticlesUseCase(
    private val articleRepo: ArticleRepository,
    private val userRepo: UserRepository,
) {
    suspend operator fun invoke(filter: Filter): List<Article> {
        val user = userRepo.getCurrentUser()
        return articleRepo.getArticles(filter, user.preferences)
    }
}
interface ArticleRepository {
    suspend fun getArticles(filter: Filter, prefs: UserPrefs): List<Article>
}

The domain module typically depends only on :core:model and kotlinx-coroutines-core. It should never depend on Hilt, Retrofit, Room, or any Android library. Repository interfaces defined here follow the Dependency Inversion Principle — :core:data implements them, and feature modules depend on the interfaces. This means features can be tested with fake repositories running on the JVM in milliseconds, without any Android framework involvement.

Q19. How do you handle shared resources (strings, colors, drawables) across feature modules?

Shared resources live in a dedicated module — typically :core:ui or :core:designsystem. This module contains the app's theme, shared drawables/icons, common strings, and reusable Compose components. For strings, there are two strategies: (1) Centralized — all strings in :core:ui. Simple but creates a bottleneck: any string change recompiles :core:ui and all its consumers. (2) Distributed — each feature module owns its own strings, with only truly shared strings in :core:ui. Better for build performance and team ownership.

Resource name collisions are a real risk: if :feature:search and :feature:profile both define R.string.title, the merged result is unpredictable. Solutions: use module-prefixed resource names (search_title, profile_title) or enable android.nonTransitiveRClass=true (default since AGP 8.0) so each module's R class only contains its own resources. The resourcePrefix DSL option in AGP can enforce naming conventions at build time:

android {
    resourcePrefix = "search_"  // lint error if resources don't start with "search_"
}

For Compose, the design system module exposes a custom theme object wrapping Material3 with app-specific tokens:

// :core:designsystem
@Composable
fun AppTheme(content: @Composable () -> Unit) {
    MaterialTheme(
        colorScheme = AppColorScheme,
        typography = AppTypography,
        content = content
    )
}

For colors and theme, centralization in :core:designsystem is standard — the design system changes infrequently and must be consistent across features.

Q20. Design the module architecture for a super-app with 20 features, 5 teams, and a shared design system.

The architecture has four tiers. Tier 1 — App shells: :app (main phone app), potentially :app:wear and :app:auto. Each app module wires features together via navigation and Hilt. Tier 2 — Feature modules (20 modules): grouped by team ownership — e.g., Team Payments owns :feature:payments, :feature:wallet, :feature:transactions, :feature:receipts. No feature-to-feature dependencies — cross-feature communication goes through :core:navigation (deep link routes) and :core:domain (shared use cases).

Tier 3 — Core modules: :core:designsystem (shared Compose theme + components, owned by platform/design team), :core:network + :core:network-api, :core:database, :core:domain (pure Kotlin), :core:data, :core:auth, :core:analytics + :core:analytics-api, :core:navigation, :core:model. Use the api/impl split for network and analytics to isolate implementation churn. Tier 4 — Build logic: build-logic/convention with convention plugins ensuring consistent configuration across all 35–40 modules.

Remote build cache is essential at this scale — 5 teams means frequent CI builds, and cache sharing dramatically reduces build times. CODEOWNERS file maps modules to teams for PR review routing. Use the dependency analysis plugin in CI to prevent boundary violations. Each team can build and test their features independently with ./gradlew :feature:payments:testDebugUnitTest, and the :app module assembles everything for integration testing.

Governance: enforce a module registry — a modules.json or Gradle convention that declares each module's owner, tier, and allowed dependencies. CI validates the dependency graph against these rules. For the design system, publish a component catalog (Compose previews or a standalone showcase app) so teams can discover and reuse components without duplicating UI code. Total: ~35–40 modules, manageable complexity, maximum parallelism, clear ownership boundaries.

Modularization Strategies (Q21–Q28)

Q21. Walk through a monolith-to-modular migration strategy — what do you extract first and why?

Start bottom-up with the leaf dependencies. Phase 1: Extract :core:model (shared data classes) and :core:common (utility functions). These have zero dependencies on the rest of the app, so extraction is safe and mechanical. Phase 2: Extract :core:network, :core:database, and :core:datastore — infrastructure modules that depend only on :core:model. This forces you to define clean interfaces for data access. Phase 3: Extract :core:domain — pull out use cases and repository interfaces into a pure Kotlin module.

Phase 4: Extract :core:ui / :core:designsystem — shared UI components, theme, and resources. Phase 5: Extract feature modules one at a time, starting with the most independent feature (fewest dependencies on other features). Each extraction follows the same pattern: identify the feature's boundary, move code to the new module, replace direct references with interface-based dependencies, and verify the build. Throughout the migration, maintain a single :app module that depends on everything — it's the integration point.

The key principle: never break the build — each extraction is a self-contained PR that compiles and passes tests. Set up convention plugins early (Phase 1) so new modules get consistent configuration. Use the dependency analysis plugin to identify actual dependency edges before cutting. Measure build times at each phase — if extraction doesn't improve build parallelism, reconsider the split. Use the strangler fig pattern: new features go in modules from day one, legacy code migrates gradually. Expect 3–6 months for a medium app (50K lines), 6–12 months for a large app (200K+ lines).

Q22. Compare by-layer, by-feature, and hybrid modularization — when would you use each?

By-layer creates modules for each architectural layer: :ui, :domain, :data, :network. Pros: simple to understand, clear separation of concerns. Cons: a single feature change touches every layer module, destroying build parallelism; teams can't own a vertical slice; the :ui module becomes a monolith. By-layer works only for very small apps (under 10K lines). By-feature creates one module per feature, each containing its own UI, domain, and data layers. Pros: maximum build parallelism, clear team ownership. Cons: code duplication if features share data access patterns; no shared infrastructure.

Hybrid (the industry standard) combines both: feature modules for vertical slices, core modules for shared horizontal infrastructure. :feature:search depends on :core:network and :core:domain. This gives you feature isolation with shared infrastructure, maximum build parallelism (features compile in parallel, core modules are cached), and clean team ownership (feature teams own feature modules, platform team owns core modules).

Google's Now in Android, Square's Cash App, and Airbnb all use the hybrid approach. The key insight is that pure by-layer fails at team scale (everyone edits :ui), pure by-feature fails at code reuse (duplicated networking logic), but hybrid gives you both. Use pure by-feature only if features are truly independent (SDK-style modules with no shared backend).

Q23. What is vertical vs horizontal slicing? How does it relate to team ownership?

Vertical slicing cuts the app by feature — each slice contains all layers (UI, domain, data) for a single feature. A vertical slice for "search" includes the search screen, search ViewModel, search use case, and search repository. Horizontal slicing cuts by technical layer — the network layer, the database layer, the UI layer. In modular architecture, feature modules are vertical slices and core modules are horizontal slices.

The relationship to team ownership is direct: vertical slices map to feature teams (Team Search owns :feature:search end-to-end), while horizontal slices map to platform teams (the Platform team owns :core:network, :core:database, :core:designsystem). This alignment is critical for organizational scaling — a feature team can ship a feature without coordinating with other feature teams.

Conway's Law applies: the module structure should mirror the team structure. If you have 5 feature teams and 1 platform team, you should have ~5 feature module groups and a set of core modules. Misalignment (e.g., two teams co-owning one module) leads to merge conflicts, unclear responsibility, and slow velocity. Conflicts arise at the boundary: when a feature team needs a new API endpoint, do they add it to :core:network (horizontal, requires platform team review) or create a feature-specific data source (vertical, duplicates infrastructure)? The answer depends on reusability — if only one feature uses it, keep it vertical; if multiple features need it, promote it to a horizontal module.

Q24. What are the most common modularization mistakes? How do you avoid them?

(1) God module — a :core:common that everything depends on and that changes frequently, invalidating the entire build graph. Fix: split into focused modules (:core:model, :core:extensions). (2) Leaky abstractions — using api instead of implementation everywhere, so a change in a transitive dependency recompiles the world. Fix: default to implementation, use the dependency analysis plugin. (3) Feature-to-feature dependencies — :feature:checkout depends on :feature:cart, creating coupling. Fix: extract shared logic to :core:domain, use navigation routes.

(4) Over-modularization — 200 modules with 5 files each; configuration time exceeds compilation time. Fix: merge small modules, aim for 500–2000 lines per module minimum. (5) Ignoring build performance — modularizing without measuring; the modular build is slower because of configuration overhead and poor cache hit rates. Fix: measure with --scan before and after every extraction, optimize configuration avoidance, enable build cache. (6) No convention plugins — each module has a unique build.gradle.kts with copy-pasted configuration that drifts over time. Fix: create convention plugins in build-logic/ from day one.

(7) Circular dependencies — usually caused by shared models or navigation. Fix: extract the shared types into a common module that both sides depend on. (8) No enforcement — module boundaries exist but nothing prevents violations. Fix: CI checks with dependency analysis plugin + architectural lint rules. Avoid all of these by measuring first, splitting second, and automating enforcement from the start.

Q25. How do you break a circular dependency between two feature modules?

Circular dependencies (:feature:A → :feature:B → :feature:A) are a build error in Gradle — the dependency graph must be a DAG. Solution 1 — Extract shared types: move shared data classes, interfaces, or navigation routes into :core:model or :core:navigation. Solution 2 — Dependency inversion: define an interface in :feature:A that :feature:B implements, inject via DI. The :app module wires them via Hilt. Solution 3 — Event bus / mediator: use a shared event system (SharedFlow in :core:common).

Solution 4 — Navigation routes: instead of direct class references, features navigate via route objects defined in :core:navigation:

// :core:navigation — shared route definitions
@Serializable data class ProfileRoute(val userId: String)
@Serializable data class SearchRoute(val query: String = "")

// :feature:search — navigates without depending on :feature:profile
navController.navigate(ProfileRoute(userId = "123"))

The best approach depends on the coupling type: shared types → extract; behavioral coupling → dependency inversion; navigation → routes. The root cause is usually a design problem: features that are "circular" often share a concept that should be its own module. For example, if :feature:profile and :feature:settings both need user data, extract :core:user with the user model and repository interface. At L6, you should recognize circular dependencies as an architectural smell and address the underlying coupling, not just work around the Gradle error.

Q26. What are dynamic feature modules? When should you use them and what are the trade-offs?

Dynamic feature modules (DFMs) use Play Feature Delivery to download module code on-demand rather than including it in the base APK. They're declared with the com.android.dynamic-feature plugin and have an inverted dependency: the DFM depends on :app, not the other way around. This means :app cannot directly reference classes in the DFM — you must use reflection, the SplitInstallManager API, or a service-locator pattern.

Use DFMs when: a feature is used by a small percentage of users, the feature adds significant APK size (5+ MB), or you need conditional delivery based on device features (AR module only for ARCore devices). Trade-offs are significant: complexity — the inverted dependency model, runtime installation checks, and error handling add significant overhead. Testing — DFMs require bundletool or Play Store for testing. Navigation — you can't use type-safe navigation; deep links or reflection are required. Hilt — DFMs can't use standard Hilt; they need @EntryPoint. User experience — download failures and slow networks create edge cases.

For most apps, the complexity isn't worth it — App Bundles already provide per-device optimization (density splits, ABI splits, language splits). DFMs make sense only for genuinely large, rarely-used features. Google has been de-emphasizing DFMs in recent guidance, favoring regular modules with good R8 optimization. If your APK size concern is about total install size rather than initial download, consider using install-time delivery (included in the initial install but as a separate split) rather than on-demand delivery.

Q27. How do you measure the success of a modularization effort? What metrics do you track?

Track metrics across three dimensions. Build performance: (1) clean build time, (2) incremental build time (the most important — measure after a single-file change in a leaf module vs a core module), (3) configuration time, (4) build cache hit rate (local and remote), (5) CI pipeline duration. Target: incremental builds under 30 seconds, cache hit rate above 80%.

Developer productivity: (1) time from code change to running app, (2) merge conflict frequency (should decrease as modules isolate team changes), (3) PR review turnaround (smaller, focused PRs in owned modules), (4) onboarding time for new developers. Code quality: (1) module coupling metrics (afferent/efferent coupling), (2) module cohesion, (3) dependency graph depth (deep chains hurt incremental builds), (4) ratio of api vs implementation dependencies, (5) boundary violation count (detected by CI).

Use Gradle build scans for build metrics, the dependency analysis plugin for coupling metrics, and custom scripts to visualize the module graph. Set baselines before modularization and track trends. Concrete targets: incremental build under 30 seconds, cache hit rate >80% on CI, zero circular dependencies, no module with >50 direct dependents. A common mistake is celebrating module count as a success metric — 100 modules means nothing if builds are slower. The only metric that matters is developer iteration speed: how fast can a developer make a change, build, test, and ship?

Q28. Your team's modular build is slower than the monolith was — what went wrong and how do you fix it?

(1) Configuration overhead — each module adds 50–100ms of configuration time. With 150 modules and no configuration cache, that's 7–15 seconds before any compilation starts. Fix: enable the configuration cache (org.gradle.configuration-cache=true), minimize variant count with variantFilter, use convention plugins to reduce script complexity. (2) No build cache — without caching, every module recompiles from scratch. Fix: enable local and remote build cache, ensure tasks are cacheable (no absolute paths in inputs). (3) Overuse of api — if core modules expose dependencies via api, a single change cascades through the entire graph. Fix: audit with the dependency analysis plugin and switch to implementation.

(4) God module — a :core:common that every module depends on and that changes frequently. Fix: split the god module into focused modules. (5) kapt in every module — annotation processing runs per-module and doesn't parallelize well. Fix: migrate to KSP, minimize modules that need annotation processing. (6) Not enough parallelism — if the dependency graph is a long chain (A→B→C→D), modules compile sequentially regardless of CPU cores. Fix: flatten the graph, use the api/impl split to reduce the critical path. (7) Deep dependency chains — changes to :core:common cascade through everything. Fix: use implementation for all dependencies in shared modules, minimize public API surface.

Diagnosis workflow: run ./gradlew :app:assembleDebug --scan to see the critical path, identify the slowest tasks, measure cache hit rates, and verify that parallel execution is actually happening. Compare the build scan timeline with the monolith's — often the issue is that the monolith had one 60-second compilation, while the modular build has 40 sequential 3-second compilations due to a deep dependency chain. The fix is always the same: widen the graph, enable caching, reduce configuration overhead, and minimize the public API surface of shared modules.

Dependency Management (Q29–Q36)

Q29. Explain version catalogs (libs.versions.toml) — how do they work and why are they better than ext {} variables?

Version catalogs (Gradle 7.0+, stable in 7.4) are a declarative, type-safe way to manage dependency versions. The gradle/libs.versions.toml file has sections: [versions], [libraries], [bundles], and [plugins]. Gradle generates type-safe accessors:

# gradle/libs.versions.toml
[versions]
retrofit = "2.9.0"
okhttp = "4.12.0"
compose-bom = "2024.02.00"

[libraries]
retrofit-core = { module = "com.squareup.retrofit2:retrofit", version.ref = "retrofit" }
okhttp-bom = { module = "com.squareup.okhttp3:okhttp-bom", version.ref = "okhttp" }
compose-bom = { module = "androidx.compose:compose-bom", version.ref = "compose-bom" }

[bundles]
networking = ["retrofit-core", "okhttp-bom"]

[plugins]
android-application = { id = "com.android.application", version = "8.2.0" }

Compared to ext {}: (1) Type safety — typos are compile errors (libs.retorfit won't compile). (2) IDE support — auto-completion, navigation, refactoring. (3) Bundles — group related dependencies, reducing boilerplate: implementation(libs.bundles.networking) adds Retrofit + OkHttp + Moshi in one line. (4) Centralization — single source of truth, easily diffable in code review. (5) Tooling — Renovate and Dependabot understand TOML catalogs and can auto-update versions. (6) Publishable — catalogs can be published as a Maven artifact and shared across projects. (7) No configuration cache issues — unlike ext {} which captures Project references. The main limitation: accessing catalogs from convention plugins requires the VersionCatalogsExtension workaround (see Q49).

Q30. What is a BOM? How does platform() differ from enforcedPlatform()?

A BOM (Bill of Materials) is a POM file that declares <dependencyManagement> entries — version constraints for a set of libraries, without adding them as actual dependencies. In Gradle, you consume a BOM with implementation(platform(libs.compose.bom)), which imports the version constraints. You then declare individual libraries without versions: implementation(libs.compose.ui), and the BOM supplies the version.

platform() applies the BOM's versions as preferences — if another dependency in your graph brings in a higher version, the higher version wins (standard Gradle conflict resolution). enforcedPlatform() applies versions as strict constraints — it forces the BOM's version even if a transitive dependency requests a higher one. This is useful when you need exact version alignment (e.g., all Compose libraries must be the same version), but dangerous if a transitive dependency genuinely needs a newer version for a bug fix — it can cause runtime NoSuchMethodError if a library requires a newer version than the BOM specifies.

In practice, use platform() for most BOMs (Compose BOM, Firebase BOM). Use enforcedPlatform() only when you need to pin versions for reproducibility or security (e.g., pinning OkHttp across all transitives to a specific patched version). You can also create your own BOM as a java-platform module to align internal library versions across modules:

// :bom/build.gradle.kts
plugins { `java-platform` }
dependencies {
    constraints {
        api(libs.retrofit)
        api(libs.okhttp)
        api(libs.moshi)
    }
}

// :feature:search/build.gradle.kts
dependencies {
    implementation(platform(project(":bom")))
    implementation(libs.retrofit)  // version from BOM
}
Q31. How do you handle transitive dependency conflicts? Walk through a diamond dependency scenario.

The diamond dependency problem: :app depends on :feature:A and :feature:B. :feature:A depends on :core:network which uses OkHttp 4.11. :feature:B depends on a third-party SDK that bundles OkHttp 4.9. Gradle's default resolution selects 4.11 (newest wins). But if OkHttp 4.11 removed a method that the SDK uses, you get a runtime NoSuchMethodError.

Debugging: run ./gradlew :app:dependencyInsight --dependency okhttp --configuration runtimeClasspath to see the resolution chain. Solutions:

// (1) Force a specific version globally
configurations.all {
    resolutionStrategy.force("com.squareup.okhttp3:okhttp:4.11.0")
}
// (2) Exclude the transitive from a specific dependency
implementation("com.example:sdk:1.0") {
    exclude(group = "com.squareup.okhttp3")
}
// (3) Use strictly for hard constraints
implementation("com.squareup.okhttp3:okhttp") {
    version { strictly("4.11.0") }
}
// (4) Fail on any conflict (forces explicit resolution)
configurations.all {
    resolutionStrategy.failOnVersionConflict()
}

The ideal fix is always to upgrade the SDK to use a compatible version. For prevention at scale: use dependency locking to detect version changes in CI, the dependency analysis plugin to catch undeclared transitive usages before they become runtime failures, and failOnVersionConflict() to make conflicts build errors rather than silent upgrades. For systematic management, use a BOM or a version catalog with strictly constraints for critical dependencies.

Q32. What is dependency locking? When and why should you use it?

Dependency locking records the exact resolved versions of all dependencies into a lock file (gradle.lockfile per module). When locking is active, Gradle verifies that resolved versions match the lock file — if a transitive dependency version changes, the build fails with a clear diff. Enable with dependencyLocking { lockAllConfigurations() } and generate lock files with ./gradlew dependencies --write-locks.

Use dependency locking when: (1) Reproducibility — you need identical builds across machines and time (required for security audits, compliance). (2) Supply chain security — detect unexpected transitive changes that could indicate a compromised dependency. (3) Large teams — prevent one developer's library upgrade from silently changing transitive versions for everyone. (4) CI stability — dynamic versions (1.+, latest.release) or SNAPSHOT dependencies can cause flaky CI; locking pins them.

The trade-off is maintenance: every intentional dependency upgrade requires regenerating lock files. Combine locking with Dependabot/Renovate — the bot creates PRs that update both the version catalog and the lock files, making version changes explicit and reviewable. Lock files should always be committed to version control. An alternative is strict version declarations in the version catalog — using exact versions everywhere effectively achieves the same reproducibility without lock files. At Google scale, dependency locking is standard practice; for smaller teams, strict version catalogs are often sufficient.

Q33. Explain the dependency analysis plugin — what problems does it detect?

The com.autonomousapps.dependency-analysis plugin (DAGP) analyzes bytecode and source code to detect dependency declaration problems. It catches: (1) Unused dependencies — you declared implementation(libs.gson) but no code uses Gson classes. (2) Used-but-undeclared dependencies — you use a class from a transitive dependency without declaring it directly. This is fragile — if the intermediate dependency changes its transitives, your code breaks. (3) api vs implementation misuse — you declared implementation but expose the dependency's types in your public API (should be api), or vice versa.

(4) compileOnly candidates — dependencies only used for annotations or compile-time checks. (5) Redundant plugins — kotlin-kapt applied but no annotation processors configured. DAGP works by analyzing compiled bytecode to determine which classes are actually referenced, then comparing against declared dependencies.

# Project-wide health report
./gradlew buildHealth

# Per-module report
./gradlew :feature:search:projectHealth

# Example output:
# Unused dependencies:
#   implementation("com.google.code.gson:gson:2.10")
# Used transitive (should be declared):
#   com.squareup.okio:okio:3.6.0
# api should be implementation:
#   api("org.jetbrains.kotlinx:kotlinx-coroutines-core:1.7.3")

In CI, configure DAGP to fail the build on violations. For large projects, DAGP is essential — it prevents dependency rot and keeps the graph clean, directly improving build performance by ensuring correct api/implementation usage.

Q34. What is the difference between compileOnly, runtimeOnly, and implementation? Give examples for each.

implementation adds the dependency to both compile and runtime classpaths — the standard choice. Example: implementation(libs.retrofit) — you call Retrofit APIs in your code and need it at runtime. compileOnly adds to the compile classpath only — available during compilation but not packaged into the APK. Examples: annotation libraries like compileOnly(libs.javax.inject) where only annotations are needed at compile time; or a library guaranteed to be provided by the runtime environment.

runtimeOnly adds to the runtime classpath only — you can't import its classes in source code, but it's packaged into the APK. Examples: runtimeOnly(libs.slf4j.android) (SLF4J binding discovered via ServiceLoader); database drivers; or debugRuntimeOnly(libs.leakcanary). There's also testImplementation, testCompileOnly, and testRuntimeOnly with the same semantics scoped to tests.

Choosing correctly matters for build performance — compileOnly dependencies don't trigger recompilation of consumers, and runtimeOnly dependencies don't affect the compile classpath at all. Understanding these configurations is critical for the api/impl module split pattern — the API module can use compileOnly for heavy dependencies it doesn't want to force on consumers, while the impl module uses implementation for the actual libraries.

Q35. How do you manage dependencies in a Kotlin Multiplatform module?

KMP modules use the kotlin("multiplatform") plugin and declare dependencies per source set. commonMain depends on KMP-compatible libraries; platform-specific source sets add platform implementations:

kotlin {
    androidTarget()
    iosX64(); iosArm64(); iosSimulatorArm64()
    sourceSets {
        commonMain.dependencies {
            implementation(libs.ktor.client.core)
            implementation(libs.kotlinx.serialization.json)
        }
        androidMain.dependencies {
            implementation(libs.ktor.client.okhttp)
        }
        iosMain.dependencies {
            implementation(libs.ktor.client.darwin)
        }
    }
}

The expect/actual mechanism lets you declare platform-specific APIs in commonMain and implement them per platform:

// commonMain — expect declaration
expect class PlatformLogger() {
    fun log(message: String)
}

// androidMain — actual implementation
actual class PlatformLogger {
    actual fun log(message: String) = Log.d("App", message)
}

// iosMain — actual implementation
actual class PlatformLogger {
    actual fun log(message: String) = NSLog(message)
}

The version catalog works across all source sets. The challenge is that not all libraries have KMP artifacts — you may need to wrap platform-specific libraries behind expect/actual interfaces. The dependency graph is more complex because Gradle resolves per variant × platform, creating a larger variant matrix. In a modular Android project, a KMP module (e.g., :core:domain) can be consumed by the Android app via its androidMain artifact while also being compiled for iOS. A key challenge: Android-specific dependencies (Jetpack, AGP) can only appear in androidMain, so your :core:domain module must be genuinely platform-agnostic.

Q36. Design a dependency management strategy for a 50-module project with 3 apps sharing common modules.

Four pillars. (1) Version catalog: a single gradle/libs.versions.toml consumed by all modules. Group related libraries into bundles (compose, testing, networking). Use BOMs for library families. (2) Convention plugins: build-logic/convention with plugins like AndroidLibraryConventionPlugin, AndroidFeatureConventionPlugin, AndroidApplicationConventionPlugin, KotlinLibraryConventionPlugin. Each plugin applies the correct Kotlin/AGP version, sets SDK levels, configures common dependencies.

(3) Dependency analysis: run DAGP in CI to enforce correct api/implementation usage, detect unused dependencies. Fail the build on violations. (4) Dependency locking + automated updates: enable locking for reproducibility, use Renovate to create PRs for version updates with lock file regeneration. Group updates by ecosystem (all Compose together, all Kotlin together) to reduce PR noise. (5) Repository management: dependencyResolutionManagement in settings.gradle.kts with repositoriesMode.set(RepositoriesMode.FAIL_ON_PROJECT_REPOS) to prevent modules from adding unauthorized repositories.

For the 3 apps: the common modules (:core:*) are in the same Gradle build, and each app module (:app:phone, :app:wear, :app:tv) depends on the subset of feature and core modules it needs. This avoids publishing internal modules to a Maven repository. If the apps are in separate repositories, publish common modules to an internal Maven repository with version alignment via a shared BOM. Use strictly constraints for security-critical dependencies (OkHttp, BouncyCastle) to prevent transitive downgrades.

Additional governance: establish a dependency review process — any new third-party dependency requires a lightweight RFC covering license compatibility, maintenance health, binary size impact, and security posture. Use ./gradlew dependencies --configuration releaseRuntimeClasspath in CI to generate a dependency report, and diff it against the previous release to catch unexpected transitive additions. For the Wear and TV apps, be especially aggressive about excluding unnecessary transitive dependencies (e.g., Wear doesn't need material3 full, TV doesn't need compose-foundation gestures) to keep APK sizes minimal. This setup ensures consistency, security, and maintainability across all 50 modules and 3 apps.

Build Performance (Q37–Q46)

Q37. Your build takes 8 minutes — walk through your investigation and optimization strategy.

Start with a build scan (./gradlew assembleDebug --scan). Step 1 — Identify the phase: check configuration time vs execution time. If configuration is 30+ seconds, enable the configuration cache and audit for eager task creation. Step 2 — Find the critical path: the build scan shows the longest chain of sequential tasks. If :core:network:compileDebugKotlin → :feature:search:compileDebugKotlin → :app:compileDebugKotlin takes 4 minutes, that's your bottleneck — no amount of parallelism helps. Consider the api/impl split to shorten the chain.

Step 3 — Check cache hit rates: if most tasks show "executed" instead of "FROM-CACHE", enable the build cache and investigate misses (absolute paths, non-deterministic outputs). Step 4 — Audit annotation processing: kapt tasks are often the slowest. Migrate to KSP (2–3× faster). Step 5 — Check variant count: disable unused variants with beforeVariants { enable = false }. Step 6 — Resource processing: enable resource shrinking, use WebP, remove unused resources, use vectorDrawables.useSupportLibrary = true to avoid PNG generation.

Step 7 — Gradle settings: ensure org.gradle.parallel=true, adequate heap (-Xmx4g), file system watching enabled. Step 8 — Dependency graph: check for api leaks causing cascade recompilation, split god modules, use the api/impl pattern for heavy infrastructure modules. Step 9 — CI-specific: use remote build cache, run only needed variant, skip lint/test for non-PR builds. Target: incremental builds under 30s, clean builds under 3 minutes for a 50-module project.

Q38. Explain configuration avoidance — what patterns should you avoid in build.gradle.kts?

Configuration avoidance means deferring work from the configuration phase to the execution phase. Avoid: tasks.create() — use tasks.register(). tasks.all { } or tasks.withType<T>().all { } — forces all matching tasks to be realized; use configureEach { }. Resolving file paths eagerly: val output = file("${buildDir}/generated") — use layout.buildDirectory.dir("generated"). Calling .get() on a Provider during configuration. Using afterEvaluate { } — it runs during configuration and masks ordering issues.

// BAD — captures Project in task action (breaks configuration cache)
tasks.register("bad") {
    doLast { println(project.buildDir) }
}
// GOOD — capture Provider, resolve at execution time
tasks.register("good") {
    val dir = project.layout.buildDirectory
    doLast { println(dir.get()) }
}
// BAD — eagerly resolves all tasks
tasks.withType<KotlinCompile>().all { kotlinOptions.jvmTarget = "17" }
// GOOD — lazily configures only realized tasks
tasks.withType<KotlinCompile>().configureEach { kotlinOptions.jvmTarget = "17" }

A well-configured build script should do almost nothing during configuration: declare plugins, register tasks lazily, wire inputs/outputs via Providers, and declare dependencies. All actual work happens during execution. A useful diagnostic: --dry-run shows configuration time without execution — if it's slow, you have configuration-phase problems. Every millisecond spent in configuration runs on every single build invocation, regardless of what task you're running.

Q39. How does the remote build cache work? Design the cache strategy for a team of 20 developers.

The remote build cache is an HTTP-based key-value store where keys are task input hashes and values are task output archives. When a task is about to execute, Gradle checks: local cache → remote cache → execute. Architecture: deploy a Gradle Enterprise instance (or compatible HTTP cache) accessible to all developers and CI. Push policy: only CI pushes to the remote cache — this ensures cache entries are built from clean, reproducible environments. Developers pull but don't push (prevents polluted entries from local environment differences).

CI pipeline: main branch CI builds push to cache after every merge. PR builds pull from cache (hitting entries from the base branch) and optionally push to a separate namespace. Cache key hygiene: ensure tasks use @PathSensitive(PathSensitivity.RELATIVE) so different machine paths don't cause misses. Avoid capturing environment variables or timestamps in task inputs. Monitoring: track cache hit rates per module — a healthy setup shows 70–90% hit rates. Eviction: TTL-based (7–14 days), with size-based eviction as a fallback. Network: cache nodes should be geographically close; for distributed teams, deploy regional nodes with replication.

Common pitfalls: developers pushing to the cache (pollutes it with local-only artifacts), non-deterministic tasks poisoning the cache (timestamps in generated files), and network latency making cache reads slower than local compilation for small tasks. The ROI is massive — a 50-module project with 80% cache hit rate can reduce incremental build times from 4 minutes to under 1 minute. For 20 developers doing 10 builds/day each, that's 200 builds × 3 minutes saved = 10 hours of developer time saved daily.

Q40. Compare kapt and KSP — why is KSP faster and how do you migrate?

kapt works by generating Java stubs from Kotlin source, then running Java's javax.annotation.processing API over those stubs. This means: (1) the Kotlin compiler runs to generate stubs (a partial compilation), (2) Java annotation processors run, (3) the Kotlin compiler runs again to compile actual source plus generated code. The stub generation step is essentially a full Kotlin compilation. KSP operates directly on the Kotlin compiler's symbol resolution phase — no stubs generated. KSP is typically 2× faster.

// Before (kapt)
plugins {
    kotlin("kapt")
}
dependencies {
    kapt(libs.hilt.compiler)
    kapt(libs.room.compiler)
}

// After (KSP)
plugins {
    id("com.google.devtools.ksp")
}
dependencies {
    ksp(libs.hilt.compiler)
    ksp(libs.room.compiler)
}

Check that your processors support KSP — Hilt (2.48+), Room (2.50+), Moshi (1.13+), Kotlin Serialization (native KSP). Migrate one processor at a time — a project can use both kapt and KSP simultaneously during transition. Once all processors are on KSP, remove the kapt plugin entirely — this eliminates stub generation from every module. KSP also sees Kotlin-specific features (extension functions, data classes, sealed classes) that kapt can't represent in Java stubs. Having both plugins applied negates some performance gains, so complete the migration as quickly as possible. Move annotation processor arguments from kapt { arguments { } } to ksp { arg("key", "value") }.

Q41. What is the impact of build variants on build performance? How do you minimize it?

Each build variant generates a complete set of tasks: compilation, resource processing, dexing, packaging. With 2 build types × 3 flavors = 6 variants, each module has 6× the tasks. In a 50-module project, that's 300 variant-module combinations, each with ~30 tasks = 9,000 tasks. Even with lazy registration, the configuration phase must evaluate variant-aware dependency resolution for all variants.

Minimization strategies:

// Disable unused variants
androidComponents {
    beforeVariants(selector().withName("freeRelease")) {
        enable = false  // never build freeRelease
    }
    beforeVariants(selector().withBuildType("benchmark")) {
        enable = buildType == "benchmark" &&
            project.path == ":app"  // only :app needs benchmark
    }
}

// Library modules: ignore app-level flavor dimensions
android {
    missingDimensionStrategy("environment", "production")
}

(1) Minimize flavor dimensions — each dimension multiplies the variant count. (2) For library modules, consider disabling the release build type during development — CI builds the release variant. (3) Use missingDimensionStrategy in modules that don't need all flavors. (4) Use the Variant API for fine-grained control — disable specific tasks per variant without disabling the entire variant. A common pattern: use build types for debug/release differences and flavors only for genuinely different product configurations. Don't create flavors for things that can be handled by build config fields or runtime flags.

Q42. How do you optimize Gradle for CI? What settings differ from local development?

CI environments have different characteristics: no warm Daemon, no local cache history, but access to remote cache and more CPU cores. (1) Daemon: keep enabled if the CI agent persists between builds. (2) Remote build cache: CI should pull and push — buildCache { remote<HttpBuildCache> { isPush = System.getenv("CI") != null } }. (3) Parallelism: set org.gradle.workers.max to match CI CPU cores. (4) Memory: increase -Xmx to 4–6 GB. (5) Configuration cache: enable it.

(6) Skip unnecessary tasks: for PR validation, run assembleDebug + testDebugUnitTest + lintDebug. Don't run all variants on every PR. (7) Build scans: use --scan on CI to track performance trends. (8) Build only affected modules: use affected-module-detector to identify changed modules and only build/test those. (9) Distribute tests: split instrumented tests across multiple CI agents using test sharding. (10) Dependency caching: pre-populate the Gradle dependency cache in a Docker layer to avoid re-downloading on every build. Avoid: running clean before every CI build (defeats incremental compilation), using --refresh-dependencies (bypasses dependency cache), or building all variants when only one is needed.

Q43. What causes a build cache miss? How do you debug cache misses?

Cache misses occur when the cache key (hash of task inputs) doesn't match any stored entry. Common causes: (1) Absolute paths — if a task input includes /Users/alice/project/src/main.kt instead of a relative path. Fix: use @PathSensitive(PathSensitivity.RELATIVE). (2) Timestamps/build IDs — if BuildConfig includes a build timestamp, every build produces a unique output. Fix: use a fixed value for debug. (3) Environment variables — task accidentally captures $HOME or CI-specific variables.

(4) Non-deterministic outputs — timestamps in generated files, random ordering in collections. (5) Task implementation changes — plugin upgrade changes the task class hash, invalidating all cached entries. (6) JVM version differences — different JDKs produce different bytecode. (7) Volatile inputs — BuildConfig.VERSION_CODE changes every build, invalidating the cache for compilation tasks. Fix: move version info to a separate task or use a fixed value for debug builds.

Debugging: Gradle Enterprise's build scan shows cache miss reasons per task. The -Dorg.gradle.caching.debug=true flag shows cache key computation details. Compare cache keys between two builds to find the differing input. The "Build Cache" tab in Gradle Enterprise shows input comparisons between the current build and the closest cache hit, highlighting exactly which input differed. For custom tasks, implement @CacheableTask and annotate all inputs/outputs correctly — missing annotations mean the task is never cached.

Q44. Explain parallel execution in Gradle — at what level does it parallelize and what are the limitations?

Gradle parallelizes at the task level within the execution phase. When org.gradle.parallel=true, Gradle's task scheduler executes independent tasks concurrently using a worker pool (sized by org.gradle.workers.max, defaulting to CPU core count). Tasks from different modules can run in parallel if they have no dependency relationship. Within a single module, tasks are typically sequential (compile → dex → package), but tasks from different modules on independent branches of the dependency graph run simultaneously.

Limitations: (1) Critical path — the longest chain of dependent tasks determines minimum build time regardless of parallelism. If :core:model → :core:data → :feature:search → :app takes 3 minutes, adding 100 CPU cores won't help. (2) Memory pressure — parallel Kotlin compilations are memory-intensive; 8 parallel compilations each using 512 MB = 4 GB. Insufficient heap causes GC thrashing. (3) Configuration is single-threaded — the configuration phase evaluates build scripts sequentially (Gradle is working on parallel configuration for future versions). (4) Shared resources — tasks accessing shared state may contend. (5) Worker count — org.gradle.workers.max defaults to CPU core count; setting it too high causes context-switching overhead.

The module dependency graph shape matters more than core count — a wide, shallow graph parallelizes well; a deep, narrow chain doesn't. Use build scans to visualize the timeline — gaps between parallel tasks indicate dependency bottlenecks. The ideal setup: wide dependency graph (many leaf modules that compile in parallel), SSD storage, sufficient heap (4GB+), and worker count matching physical cores.

Practical tip: use ./gradlew :app:assembleDebug --scan and examine the "Timeline" tab in the build scan. Look for tasks that block the critical path — if a single :core module's compilation takes 40 seconds while everything else takes 5 seconds, that module is the bottleneck. Consider splitting it or optimizing its compilation (reduce annotation processing, minimize ABI surface). Also note that Gradle's Worker API enables parallelism within a single task — useful for custom tasks that process independent inputs (e.g., processing 100 images in parallel within one task).

Q45. How does file system watching improve build performance?

File system watching (default since Gradle 7.0) uses OS-level file watchers (inotify on Linux, FSEvents on macOS, ReadDirectoryChangesW on Windows) to track file changes between builds. Without it, Gradle must scan the entire project directory tree at the start of each build — 2–5 seconds in a large project with thousands of files. With file system watching, the Gradle Daemon maintains an in-memory Virtual File System (VFS) reflecting the project's current state.

Between builds, the OS notifies the Daemon of changed files, and the VFS is updated incrementally. When the next build starts, Gradle already knows which files changed without scanning. The benefit is most pronounced for incremental builds — instead of scanning 50,000 files to find the 3 that changed, the Daemon already knows. This typically saves 1–5 seconds per build, which adds up significantly over hundreds of daily builds.

The feature requires the Gradle Daemon and works best when the Daemon stays alive between builds. It can be disabled with org.gradle.vfs.watch=false if the OS watcher causes issues (some network file systems or Docker volume mounts don't support it). Limitations: OS watcher limits can be exceeded on very large projects (increase fs.inotify.max_user_watches on Linux), and some file operations (like git checkout touching thousands of files) can overwhelm the watcher, causing a full rescan. On CI with ephemeral VMs, file system watching provides no benefit since the Daemon doesn't persist between builds. The VFS also improves up-to-date checking — Gradle can quickly determine that a task's inputs haven't changed without re-hashing files.

Q46. Your team reports that clean builds are fast but incremental builds are slow — what's wrong?

This is counterintuitive — incremental builds should be faster. Likely causes: (1) Cache invalidation cascade — a frequently-changed module is depended on by everything via api. Every change invalidates the entire graph, making incremental builds effectively clean builds. Fix: audit api vs implementation, split the god module. (2) Non-incremental annotation processing — kapt processors that aren't incremental force full recompilation on any change. Fix: migrate to KSP. (3) Build cache disabled — clean builds benefit from parallelism, but incremental builds benefit from caching.

(4) Kotlin incremental compilation broken — if kotlin.incremental=false or the incremental state is corrupted (common after build failures), every compilation is full. Check .kotlin directories for corruption — deleting them forces a clean rebuild of IC state. (5) Resource changes triggering full recompilation — changing a string regenerates R classes, triggering recompilation of all files importing R. Fix: enable non-transitive R classes (android.nonTransitiveRClass=true). (6) BuildConfig or generated code with timestamps — if generated files change on every build, downstream tasks always re-execute. Fix: make generated code deterministic or move volatile values to a separate, isolated module.

(7) Daemon restart — if the Daemon dies between builds (OOM), the next build is effectively clean because all in-memory caches (file system snapshot, JIT compilation) are lost. Check gradle --status and increase heap.

Diagnose with --scan — compare a clean build's task execution graph with an incremental build's to see which tasks are unexpectedly re-executing. The build scan's "Performance" tab shows whether tasks were executed, up-to-date, or from-cache. Focus on tasks marked "executed" that should have been "up-to-date" — these reveal the root cause of the incremental build regression.

Convention Plugins & Build Logic (Q47–Q52)

Q47. Why is buildSrc an anti-pattern for large projects? What should you use instead?

buildSrc is a special Gradle directory compiled before any build script and added to the build script classpath. The problem: any change to buildSrc invalidates the entire build. Gradle treats buildSrc as an input to every build script — if you change a single line, every module's configuration is invalidated, the configuration cache is busted, and the build cache for configuration-dependent tasks is invalidated. For a 100-module project, this means a full reconfiguration (10–20 seconds) plus potential recompilation of everything.

The alternative is composite builds with an included build (typically build-logic/convention). Changes to convention plugins only invalidate modules that apply the changed plugin — not the entire project. The migration is straightforward:

// settings.gradle.kts
pluginManagement {
    includeBuild("build-logic")
}

// build-logic/convention/build.gradle.kts
plugins { `kotlin-dsl` }
dependencies {
    compileOnly(libs.android.gradlePlugin)
    compileOnly(libs.kotlin.gradlePlugin)
}
gradlePlugin {
    plugins {
        register("androidLibrary") {
            id = "myapp.android.library"
            implementationClass = "AndroidLibraryConventionPlugin"
        }
    }
}

Additionally, composite builds support the build cache for the convention plugins themselves, and they can be developed and tested independently (with GradleRunner functional tests). The composite build approach also enables sharing build logic across multiple projects — publish the plugins to a Maven repository and consume them in other projects. Google's Now in Android uses exactly this pattern. buildSrc is fine for small projects (<5 modules) where the invalidation cost is negligible, but for anything larger, composite builds are strictly superior.

Q48. Walk through implementing a convention plugin — from project setup to applying it in a module.

Step 1 — Project setup: create build-logic/convention/build.gradle.kts applying kotlin-dsl. Add AGP and Kotlin Gradle plugin as compileOnly dependencies. Step 2 — Plugin class:

class AndroidLibraryConventionPlugin : Plugin<Project> {
    override fun apply(target: Project) = with(target) {
        pluginManager.apply("com.android.library")
        pluginManager.apply("org.jetbrains.kotlin.android")
        extensions.configure<LibraryExtension> {
            compileSdk = 34
            defaultConfig.minSdk = 24
            compileOptions {
                sourceCompatibility = JavaVersion.VERSION_17
                targetCompatibility = JavaVersion.VERSION_17
            }
        }
    }
}

Step 3 — Register in gradlePlugin { plugins { register("androidLibrary") { id = "myapp.android.library"; implementationClass = "AndroidLibraryConventionPlugin" } } }. Step 4 — Include in root settings.gradle.kts: pluginManagement { includeBuild("build-logic") }. Step 5 — Apply in any module:

// :feature:search/build.gradle.kts — entire file
plugins {
    id("myapp.android.feature")  // applies library + kotlin + hilt + compose
}
dependencies {
    implementation(project(":core:domain"))
    implementation(project(":core:ui"))
    testImplementation(project(":core:testing"))
}

The module's build file shrinks from 30+ lines to under 10 lines. Create layered variants: AndroidFeatureConventionPlugin adds Hilt + Navigation + Compose on top of the library plugin. KotlinLibraryConventionPlugin applies kotlin("jvm") for pure Kotlin modules. Convention plugins eliminate boilerplate, enforce consistency, and make build configuration changes a single-point update across all modules.

Q49. How do you access the version catalog from a convention plugin?

Convention plugins in an included build don't have direct access to the version catalog's type-safe accessors (those are generated for the consuming project). The workaround uses VersionCatalogsExtension:

class AndroidLibraryConventionPlugin : Plugin<Project> {
    override fun apply(target: Project) = with(target) {
        val libs = extensions.getByType<VersionCatalogsExtension>()
            .named("libs")
        dependencies {
            add("implementation",
                libs.findLibrary("kotlinx-coroutines-core").get())
            add("testImplementation",
                libs.findLibrary("junit").get())
        }
    }
}

The findLibrary(), findVersion(), findBundle(), and findPlugin() methods return Optional values looked up by the alias in libs.versions.toml. Since Gradle 8.0, you can share the catalog with the included build by referencing it in the included build's settings.gradle.kts: dependencyResolutionManagement { versionCatalogs { create("libs") { from(files("../../gradle/libs.versions.toml")) } } }. Some teams create extension functions to reduce boilerplate:

// build-logic/convention/src/main/kotlin/Extensions.kt
val Project.libs: VersionCatalog
    get() = extensions.getByType<VersionCatalogsExtension>()
        .named("libs")

fun VersionCatalog.library(alias: String) =
    findLibrary(alias).get()

fun VersionCatalog.version(alias: String): String =
    findVersion(alias).get().requiredVersion

The version catalog remains the single source of truth, even when accessed indirectly through convention plugins.

Q50. Explain build variants, flavors, and dimensions — how do they interact and what is the variant matrix?

Build variants are the Cartesian product of build types and product flavors. Build types (typically debug and release) control debuggability, minification, and signing. Product flavors define different versions: free/paid, staging/production. Flavors are organized into dimensions — each dimension represents an independent axis of variation.

If you have dimension tier (free, paid) and dimension environment (staging, production), the variant matrix is: freeStagingDebug, freeStagingRelease, freeProductionDebug, freeProductionRelease, paidStagingDebug, paidStagingRelease, paidProductionDebug, paidProductionRelease — 8 variants from 2×2×2. Each variant gets its own source set (src/freeStagingDebug/), resource merging, manifest merging, and AGP task pipeline. The merging priority (highest wins): variant-specific → flavor-combination → individual flavor → build type → main.

For dependency management, each variant has its own configuration: freeStagingDebugImplementation. The variant explosion is a real performance concern — each variant adds ~50 AGP tasks per module. With 8 variants and 30 modules: 12,000 tasks. Minimize dimensions and disable unused combinations with androidComponents.beforeVariants. Library modules typically don't need flavors — they publish a single variant and let the consuming app's variant selection drive the build. Use missingDimensionStrategy in library modules to prevent them from inheriting app-level dimensions they don't care about. Don't create flavors for things that can be handled by build config fields or runtime flags.

Q51. How do you test a custom Gradle plugin?

Three levels. (1) Unit tests: test individual functions that don't depend on Gradle APIs — standard JUnit/Kotlin test. (2) Integration tests with ProjectBuilder: creates an in-memory Project instance to verify task registration and configuration:

@Test fun `plugin registers expected tasks`() {
    val project = ProjectBuilder.builder().build()
    project.plugins.apply("myapp.android.library")
    assertNotNull(project.tasks.findByName("compileDebugKotlin"))
}

(3) Functional tests with GradleRunner (from gradle-test-kit): executes a real Gradle build in a temporary directory — the most thorough approach:

@Test fun `assembleDebug succeeds`() {
    val result = GradleRunner.create()
        .withProjectDir(tempDir)
        .withArguments("assembleDebug")
        .withPluginClasspath()
        .build()
    assertEquals(TaskOutcome.SUCCESS,
        result.task(":assembleDebug")?.outcome)
}

Functional tests catch real-world issues: configuration cache compatibility, task dependency wiring, and interaction with other plugins. They're slower (seconds per test) but essential. Test against multiple Gradle versions using parameterized tests to ensure compatibility. In CI, run plugin tests before the main build — a broken convention plugin breaks every module. For configuration cache testing, add .withArguments("assembleDebug", "--configuration-cache") to verify your plugin doesn't capture non-serializable state.

Q52. Design the build logic architecture for a multi-app project (main app + wear app + TV app) sharing common modules.

Build logic (build-logic/convention/): convention plugins for each module type — AndroidApplicationConventionPlugin (shared app config), AndroidLibraryConventionPlugin, AndroidFeatureConventionPlugin (Compose + Hilt + Navigation), KotlinLibraryConventionPlugin. Each app type gets a specialized plugin: PhoneAppPlugin extends the base with phone-specific config (adaptive icons, deep links), WearAppPlugin adds Wear Compose and Tiles, TvAppPlugin adds Leanback/TV Compose.

Common modules: :core:model, :core:domain (pure Kotlin), :core:data, :core:network, :core:database, :core:designsystem (shared theme tokens — colors, typography — but platform-specific components in each app). App modules: :app:phone depends on :feature:* phone features + :core:*. :app:wear depends on :feature:wear:* + :core:*. :app:tv depends on :feature:tv:* + :core:*. Feature modules are platform-specific (different UI) but share domain logic via :core:domain. The version catalog is shared across all apps. Build variants: each app has its own flavor dimensions. Convention plugins handle the differences:

// :feature:search/build.gradle.kts — just 3 lines
plugins {
    id("myapp.android.feature")
}
dependencies {
    implementation(project(":core:search-api"))
}

// :app:wear/build.gradle.kts
plugins {
    id("myapp.wear.app")
}
dependencies {
    implementation(project(":feature:wear:workout"))
    implementation(project(":feature:wear:health"))
}

Key design decisions: CI optimization — use affected-module-detector so changes to :core:network trigger tests for all three apps, but changes to :feature:wear:workout only trigger the Wear app build. Resource management — each platform has its own design system module (:core:designsystem-phone, :core:designsystem-wear, :core:designsystem-tv) extending shared tokens from :core:designsystem. Build performance — disable unused build types per app (Wear and TV likely don't need a benchmark build type), and use beforeVariants to prune the variant matrix aggressively. Common Problems & Solutions (Q53–Q60)

Q53. How do you handle navigation between feature modules that can't depend on each other?

Feature modules must not depend on each other, but users navigate between features constantly. Solutions: (1) Shared navigation routes: define all routes in :core:navigation. Each route is a sealed class or @Serializable data class. Feature modules depend on :core:navigation to declare their routes and navigate to other features' routes. The :app module registers the actual screen composables in the NavHost. (2) Deep links: features navigate via URIs. No compile-time dependency needed. (3) Navigator interface pattern: define a FeatureNavigator interface in :core:navigation, implemented by :app, injected via Hilt.

(4) Type-safe routes with Kotlin Serialization (Navigation 2.8+):

// :core:navigation — shared route definitions
@Serializable data class ProfileRoute(val userId: String)
@Serializable data class SearchRoute(val query: String = "")

// :feature:search — navigates without depending on :feature:profile
navController.navigate(ProfileRoute(userId = "123"))

// :app — registers the destination
composable<ProfileRoute> { backStackEntry ->
    val route = backStackEntry.toRoute<ProfileRoute>()
    ProfileScreen(userId = route.userId)
}

The key principle: the route definition lives in a shared module; the route registration (connecting a route to a screen) lives in :app. This separation means adding a new feature module only requires: (1) defining its routes in :core:navigation, (2) implementing the screen in the feature module, and (3) registering the destination in :app's NavHost — no other module needs to change.

Q54. Explain manifest merging — what are common conflicts and how do you resolve them?

Android's build system merges manifests from all modules, libraries, and build variants into a single final manifest. Merge priority (highest wins): build variant → app module → library modules (in dependency order) → AAR libraries. Common conflicts: (1) minSdk mismatch — a library declares minSdk 26 but your app targets minSdk 24. Fix: tools:overrideLibrary="com.example.library". (2) Duplicate components — two libraries declare the same activity/service. Fix: tools:node="remove" or tools:node="replace".

(3) Permission conflicts — a library adds unwanted permissions. Fix: <uses-permission android:name="..." tools:node="remove" />. (4) allowBackup conflict — Fix: tools:replace="android:allowBackup" on <application>. (5) Intent filter collisions — multiple modules declare the same deep link pattern.

In modular projects, each feature module's manifest should declare only its own components and permissions — the :app module handles application-level attributes and resolves conflicts. Debug with ./gradlew :app:processDebugManifest and inspect build/intermediates/merged_manifests/ to see the final result. Android Studio's merged-manifest view highlights conflicts with clickable resolution suggestions. The tools:node, tools:replace, tools:remove, and tools:strict attributes give you fine-grained control over the merge behavior.

Q55. How does R8/ProGuard work with multiple modules? What are consumer ProGuard rules?

R8 runs only in the app module during release builds — it processes the entire app's bytecode (all modules merged) in a single pass. This means R8 has full visibility: it can inline across module boundaries, remove unused classes from any module, and optimize the entire call graph. Library modules don't run R8 themselves — they contribute their bytecode to the app module's R8 pass. Consumer ProGuard rules (consumerProguardFiles) are rules that a library module packages into its AAR:

// :core:network/build.gradle.kts
android {
    defaultConfig {
        consumerProguardFiles("consumer-rules.pro")
    }
}

// :core:network/consumer-rules.pro
-keep interface com.example.network.ApiService { *; }
-keepclassmembers class * implements com.example.network.ApiService { *; }

Common pitfalls: (1) forgetting consumer rules — works in debug (no R8) but crashes in release with ClassNotFoundException. (2) Overly broad keep rules (-keep class ** { *; }) that disable optimization — your APK doesn't shrink at all. (3) Missing rules for Kotlin serialization, Retrofit interfaces, or Hilt-generated code. (4) Not testing with minification during development — create a staging build type with isMinifyEnabled = true to catch issues early. Use -printusage to see what R8 removed and -printseeds to see what was kept. Remember that R8 sees all modules together — a class "unused" in its own module might be used by another module, so per-module analysis is misleading.

Q56. How does Hilt work across modules? What are the common pitfalls?

Hilt generates a single component hierarchy rooted in @HiltAndroidApp in the :app module. Each module contributes @Module-annotated classes and @Inject-annotated constructors. At compile time, Hilt processes each module independently, generating component code. At the app level, Hilt aggregates all modules' contributions into the final component. The typical pattern for modular Hilt:

// :core:data — provides repository binding
@Module @InstallIn(SingletonComponent::class)
abstract class DataModule {
    @Binds abstract fun bindArticleRepo(
        impl: ArticleRepositoryImpl
    ): ArticleRepository
}

// :feature:search — consumes via constructor injection
@HiltViewModel
class SearchViewModel @Inject constructor(
    private val articleRepo: ArticleRepository
) : ViewModel()

Pitfalls: (1) Missing @InstallIn — a @Module without @InstallIn(SingletonComponent::class) is silently ignored; no error, just missing bindings at runtime. (2) Scope mismatches — a singleton depending on ViewModel-scoped fails at compile time. (3) @EntryPoint for non-Hilt classes — ContentProviders, DFMs need @EntryPoint to access the graph. (4) Visibility — Hilt needs to see @Module classes at compile time; internal modules can't be aggregated in the app component. (5) Test isolation — @UninstallModules removes bindings, but you must provide replacements via @BindValue or @TestInstallIn. (6) Duplicate bindings — two modules provide the same type without @Qualifier, causing "is bound multiple times" errors. (7) Build performance — Hilt's KSP/kapt processing runs in every module with @Inject or @Module. Minimize the number of modules that apply the Hilt plugin by centralizing DI wiring in :core:data and :core:network.

Q57. How do you share test utilities across modules without polluting the main source set?

Two approaches. (1) Dedicated testing module: :core:testing contains shared test utilities — fake implementations, custom JUnit rules, Compose test helpers, test dispatchers, data factories. Modules declare testImplementation(project(":core:testing")). For Android-specific utilities, create :core:testing-android with androidTestImplementation.

(2) Test fixtures (Gradle's java-test-fixtures plugin): a module publishes a testFixtures source set alongside its main source set:

// :core:network/build.gradle.kts
plugins { id("java-test-fixtures") }

// :core:network/src/testFixtures/kotlin/FakeApiService.kt
class FakeApiService : ApiService {
    override suspend fun search(query: String) = emptyList<Result>()
}

// :feature:search/build.gradle.kts
dependencies {
    testImplementation(testFixtures(project(":core:network")))
}

Test fixtures keep fakes co-located with the real implementation, while :core:testing is better for generic utilities (test dispatchers, assertion helpers, data factories). Avoid: putting test utilities in src/main/ with @VisibleForTesting — this ships test code to production. Avoid: putting fakes in src/test/ of a module — test source sets aren't published, so other modules can't access them. Avoid: duplicating fakes across modules — centralize them in :core:testing or use test fixtures. For Hilt testing, :core:testing can provide @TestInstallIn modules that replace real bindings with fakes project-wide.

Q58. What are non-transitive R classes? Why are they important for modular projects?

Historically, Android's R class was transitive: each module's R class contained not only its own resources but also all resources from its dependencies. If :core:ui defined R.string.app_name, then :feature:search's R class also contained R.string.app_name. Problems: (1) Build performance — changing a string in :core:ui regenerated R classes for every transitive consumer, triggering recompilation across the entire graph. (2) APK size — duplicate R class entries bloated DEX files. (3) Accidental coupling — modules could reference resources from transitive dependencies without declaring a direct dependency.

Non-transitive R classes (android.nonTransitiveRClass=true, default since AGP 8.0) make each module's R class contain only its own resources. To use a resource from :core:ui, you must explicitly import its R class:

// Before (transitive R) — implicit, fragile
import com.example.feature.search.R
textView.setText(R.string.app_name)  // actually from :core:ui!

// After (non-transitive R) — explicit, correct
import com.example.core.ui.R as CoreR
textView.setText(CoreR.string.app_name)

This means resource changes in :core:ui only regenerate :core:ui's R class — not every consumer's. The migration can be tedious — you must update all resource references to use fully qualified R class names — but the build performance improvement in large projects (30%+ faster resource processing) makes it essential. It also makes resource dependencies explicit and prevents accidental coupling on transitive resources.

Q59. A library module works fine in debug but crashes in release — how do you debug it?

Almost always an R8/ProGuard issue — code needed at runtime is being stripped or obfuscated. Step 1 — Get the stack trace: typically ClassNotFoundException, NoSuchMethodError, or NullPointerException from a reflection-based framework (Retrofit, Moshi, Hilt). Step 2 — Deobfuscate: use mapping.txt (in app/build/outputs/mapping/release/) with retrace. Step 3 — Check what R8 removed: run with -printusage usage.txt. Search for the missing class.

Step 4 — Add keep rules: the library module needs consumer ProGuard rules for reflection-dependent code. Step 5 — Test with minification in debug:

buildTypes {
    create("staging") {
        initWith(getByName("debug"))
        isMinifyEnabled = true
        proguardFiles(
            getDefaultProguardFile("proguard-android-optimize.txt"),
            "proguard-rules.pro"
        )
    }
}

Step 6 — Check Kotlin-specific issues: R8 may remove companion object members, default parameter values, or @JvmStatic bridges. Any use of Class.forName(), KClass.memberProperties, or annotation-based frameworks (Room, Moshi, Gson, Kotlin Serialization) needs keep rules. The root cause is that R8 performs whole-program optimization — it can't know that a class is used via reflection unless you tell it via keep rules. Add targeted keep rules in the library's consumer-rules.pro rather than broad -keep class ** { *; } rules that disable optimization entirely.

Q60. Design a testing strategy for a 30-module project — unit tests, integration tests, and CI pipeline.

Unit tests: every module has src/test/ with JVM-based tests. Pure Kotlin modules (:core:domain) use JUnit 5 + Turbine + coroutines-test. Android modules use Robolectric for ViewModel/Repository tests. Target: 80%+ line coverage per module. Shared test utilities live in :core:testing (fakes, test dispatchers, data factories). Each module's tests run independently — ./gradlew :feature:search:testDebugUnitTest — enabling parallel execution across modules.

Integration tests: src/androidTest/ in feature modules for Compose UI tests (ComposeTestRule), navigation tests, and Hilt-injected component tests. Use :core:testing-android for shared Hilt test rules and mock server setup. Database modules get Room migration tests. Network modules get MockWebServer integration tests.

CI pipeline — Tier 1 (every PR): run unit tests for affected modules only (use affected-module-detector to identify changed modules and their dependents). Run lint on affected modules. Build assembleDebug for the app module. Target: 3–5 minutes.

# CI script — Tier 1
./gradlew testDebugUnitTest \
    -Paffected_module_detector.enable \
    --parallel \
    --build-cache \
    --configuration-cache

Tier 2 (merge to main): run all unit tests, all lint checks, build release APK, push to remote build cache. Target: 8–12 minutes. Tier 3 (nightly): run instrumented tests on Firebase Test Lab with test sharding (4–8 devices), full coverage report (JaCoCo aggregated across modules), dependency analysis (buildHealth), Compose screenshot tests, security scanning. Target: 20–30 minutes. Use Gradle's --parallel and remote build cache on all tiers. Track test execution time trends — slow tests degrade developer experience and should be optimized or moved to a lower tier. Share test results via remote build cache so developers don't re-run tests CI already passed.
