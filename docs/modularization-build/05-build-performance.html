Build Performance

Profiling, configuration avoidance, task caching, incremental compilation, kapt→KSP migration, daemon tuning, and CI optimization for large multi-module Android projects

1. Build Profiling — Measure Before You Optimize

Build performance work without profiling is guesswork. Gradle provides multiple profiling tools, each at a different level of detail. The discipline is always the same: measure → identify bottleneck → fix → measure again.

Profiling Tools

Tool Command / Flag Output Best For
HTML Profile --profile Local HTML in build/reports/profile/ Quick local overview
Build Scan --scan Online interactive report Deep analysis, sharing with team
Debug Logging -Dorg.gradle.debug=true Verbose console output Debugging specific issues
Info Logging --info Cache hit/miss reasons per task Diagnosing cache misses
Dry Run --dry-run Task graph without execution Verifying task dependencies

Running a Build Scan

# Generate a local HTML profile
./gradlew assembleDebug --profile

# Generate an interactive Build Scan (uploads to scans.gradle.com)
./gradlew assembleDebug --scan

A Build Scan breaks the build into phases and shows:

  • Configuration time — how long Gradle spent evaluating build.gradle.kts files
  • Dependency resolution time — network calls to Maven repositories
  • Task execution time — per-task breakdown with outcomes (EXECUTED, UP-TO-DATE, FROM-CACHE)
  • Cache hit rate — percentage of tasks served from local or remote cache
  • Critical path — the longest chain of dependent tasks that determines total build time

Key Metrics to Track

│                    Build Scan Summary                        │
├─────────────────────────────────────────────────────────────┤
│  Total build time .............. 2m 34s                     │
│  Configuration phase ........... 12.3s  (target: <5s)      │
│  Dependency resolution ......... 3.1s                       │
│  Task execution ................ 2m 18s                     │
│                                                             │
│  Tasks executed ................ 47                          │
│  Tasks UP-TO-DATE .............. 128                         │
│  Tasks FROM-CACHE .............. 34                          │
│  Tasks SKIPPED ................. 12                          │
│                                                             │
│  Cache hit rate ................ 72%  (target: >85%)       │
└─────────────────────────────────────────────────────────────┘
L6 Insight — Profile First, Always

The most common mistake in build optimization is assuming the bottleneck. Teams spend weeks migrating to KSP when their real problem is a 15-second configuration phase caused by allprojects {} eagerly configuring 200 modules. A Build Scan takes 30 seconds to generate and immediately shows where time is spent. Never skip this step.

2. Configuration Avoidance

The configuration phase runs for every single build invocation, including ./gradlew help and ./gradlew tasks. In a project with 300 modules, eager configuration can take 15–30 seconds before any real work begins. Configuration avoidance is about deferring work until it's actually needed.

Lazy Task Registration

The single most impactful configuration avoidance technique is using tasks.register instead of tasks.create:

// BAD — task is created and configured immediately, even if never executed
tasks.create("generateDocs") {
    doLast { /* ... */ }
}

// GOOD — task is only created when something depends on it or it's requested
tasks.register("generateDocs") {
    doLast { /* ... */ }
}

With register, Gradle knows the task exists but doesn't instantiate or configure it until the task graph requires it. For a build with hundreds of custom tasks, this saves seconds of configuration time.

Lazy Properties — Provider<T> and Property<T>

abstract class GenerateApiDocsTask : DefaultTask() {
    // Lazy — value is resolved only during execution, not configuration
    @get:Input
    abstract val apiVersion: Property<String>

    @get:InputDirectory
    abstract val sourceDir: DirectoryProperty

    @get:OutputDirectory
    abstract val outputDir: DirectoryProperty

    @TaskAction
    fun generate() {
        val version = apiVersion.get()  // resolved NOW, at execution time
        // ...
    }
}

// Registration with lazy wiring
tasks.register<GenerateApiDocsTask>("generateApiDocs") {
    apiVersion.set(provider { project.findProperty("api.version") as String })
    sourceDir.set(layout.projectDirectory.dir("src/main/api"))
    outputDir.set(layout.buildDirectory.dir("generated/api-docs"))
}

Anti-Patterns That Break Configuration Avoidance

Avoid These in build.gradle.kts
  • allprojects { } / subprojects { } — forces configuration of every project in the build, even those not involved in the current task
  • configurations.getByName("implementation").resolve() — resolves dependencies during configuration (triggers network calls)
  • File I/O at configuration time — reading version files, parsing JSON, etc.
  • Accessing Task.project in task actions — breaks configuration cache (see Section 5)
  • tasks.getByName("compile...") — eagerly realizes the task; use tasks.named("compile...") instead
// BAD — forces all subprojects to configure
subprojects {
    apply(plugin = "com.android.library")  // not every subproject is a library!
    dependencies {
        implementation("com.google.code.gson:gson:2.10")
    }
}

// GOOD — use convention plugins applied selectively
// In each module's build.gradle.kts:
plugins {
    id("myapp.android.library")
}

3. Task Avoidance — UP-TO-DATE and FROM-CACHE

Gradle's core performance model is avoidance: the fastest task is one that doesn't run. Gradle achieves this by tracking every task's inputs and outputs, then skipping tasks whose inputs haven't changed.

Task Outcomes

Outcome Meaning Performance Impact
EXECUTED Task ran normally Full cost
UP-TO-DATE Inputs unchanged since last local execution Near-zero (input hash check only)
FROM-CACHE Outputs restored from build cache Low (cache lookup + output copy)
SKIPPED Task disabled via onlyIf { } or similar Zero
NO-SOURCE Task has no input files (e.g., empty source set) Zero

Declaring Inputs and Outputs

@CacheableTask
abstract class ProcessTemplatesTask : DefaultTask() {

    @get:InputFiles
    @get:PathSensitive(PathSensitivity.RELATIVE)
    abstract val templates: ConfigurableFileCollection

    @get:Input
    abstract val appVersion: Property<String>

    @get:OutputDirectory
    abstract val outputDir: DirectoryProperty

    @TaskAction
    fun process() {
        val out = outputDir.get().asFile
        templates.files.forEach { template ->
            val content = template.readText()
                .replace("{{VERSION}}", appVersion.get())
            File(out, template.name).writeText(content)
        }
    }
}

Key annotations:

  • @Input — scalar values (strings, numbers, enums)
  • @InputFiles / @InputDirectory — file-based inputs
  • @OutputFile / @OutputDirectory — where the task writes results
  • @PathSensitive(PathSensitivity.RELATIVE) — tells Gradle that only relative paths matter, not absolute. Critical for cache hits across different machines.
  • @CacheableTask — opts the task into the build cache (not just UP-TO-DATE checks)

Incremental Task Inputs

@CacheableTask
abstract class IncrementalProcessTask : DefaultTask() {

    @get:Incremental
    @get:InputFiles
    @get:PathSensitive(PathSensitivity.RELATIVE)
    abstract val sources: ConfigurableFileCollection

    @get:OutputDirectory
    abstract val outputDir: DirectoryProperty

    @TaskAction
    fun execute(inputChanges: InputChanges) {
        // Only process files that actually changed
        inputChanges.getFileChanges(sources).forEach { change ->
            when (change.changeType) {
                ChangeType.ADDED, ChangeType.MODIFIED -> {
                    processFile(change.file, outputDir.get().asFile)
                }
                ChangeType.REMOVED -> {
                    File(outputDir.get().asFile, change.file.name).delete()
                }
            }
        }
    }
}
Non-Cacheable Tasks

Tasks without declared outputs, or tasks that produce non-deterministic outputs (e.g., embedding timestamps, random UUIDs), cannot be cached effectively. A single non-cacheable task on the critical path can negate the benefit of caching everything else.

4. Build Cache — Local and Remote

UP-TO-DATE checks only work locally — they compare against the previous execution on the same machine. The build cache extends this across machines and across clean builds by storing task outputs keyed by their inputs.

How Cache Keys Work

Cache Key = hash(
    task implementation class bytecode,
    task action class bytecode,
    names + values of all @Input properties,
    hashes of all @InputFiles content,
    Gradle version,
    JVM version
)

Cache Lookup:
  1. Compute cache key from current inputs
  2. Check local cache (~/.gradle/caches/build-cache-1/)
  3. If miss → check remote cache (HTTP GET)
  4. If hit  → download outputs, skip task execution
  5. If miss → execute task, store outputs in local cache
  6. If push enabled → upload to remote cache (HTTP PUT)

Enabling the Build Cache

# gradle.properties
org.gradle.caching=true

For a remote cache, configure it in settings.gradle.kts:

// settings.gradle.kts
buildCache {
    local {
        isEnabled = true
        directory = file("${rootDir}/.gradle/build-cache")
    }
    remote<HttpBuildCache> {
        url = uri("https://cache.mycompany.com/cache/")
        isPush = System.getenv("CI") != null  // only CI pushes
        credentials {
            username = System.getenv("CACHE_USER") ?: ""
            password = System.getenv("CACHE_PASS") ?: ""
        }
    }
}

Remote Cache Workflow

┌──────────────┐     push outputs     ┌──────────────────┐
│   CI Server  │ ──────────────────▶  │   Remote Cache    │
│  (full build)│                      │  (HTTP server /   │
└──────────────┘                      │   Gradle Ent.)    │
                                      └────────┬─────────┘
                                               │
                                      pull cached outputs
                                               │
                                      ┌────────▼─────────┐
                                      │  Developer        │
                                      │  (incremental     │
                                      │   build)          │
                                      └──────────────────┘

Result: developer's "clean" build behaves like an incremental build
        because most task outputs are already in the remote cache.

Debugging Cache Misses

# Show cache hit/miss reasons for every task
./gradlew assembleDebug --info 2>&1 | grep -E "cache|UP-TO-DATE|FROM-CACHE"

# Common output:
# Task ':app:compileDebugKotlin' is not up-to-date because:
#   Value of input property 'classpath' has changed for task ':app:compileDebugKotlin'

Common cache miss reasons:

  • Different JDK version — JDK is part of the cache key; CI on JDK 17.0.9, developer on 17.0.11 → miss
  • Different OS — line endings, path separators can affect input hashes
  • Absolute paths in inputs — missing @PathSensitive(RELATIVE) on file inputs
  • Non-deterministic outputs — timestamps in generated code, random ordering in output files
  • Gradle version mismatch — Gradle version is part of the cache key
  • buildSrc changes — any change to buildSrc invalidates every task in the build
L6 Insight — Build Scan Comparison

Gradle Enterprise (now Develocity) lets you compare two Build Scans side-by-side. This is the fastest way to diagnose "why did CI take 8 minutes today when it took 3 minutes yesterday?" You can diff task outcomes, input changes, and cache hit rates between the two builds. Without this tool, debugging cache regressions on a 300-module project is nearly impossible.

5. Configuration Cache

While the build cache caches task outputs, the configuration cache caches the entire task graph from the configuration phase. On subsequent builds, Gradle deserializes the task graph directly, skipping all build.gradle.kts evaluation, plugin application, and dependency resolution.

How It Works

First Build (cold):
  settings.gradle.kts → build.gradle.kts (all modules) → task graph
  │                                                          │
  │         serialize task graph to .gradle/configuration-cache/
  │                                                          │
  ▼                                                          ▼
  Configuration: 12s                              Execution: 2m 18s

Subsequent Build (warm):
  deserialize task graph from .gradle/configuration-cache/
  │
  ▼
  Configuration: 0.3s  (skipped!)                 Execution: 2m 18s

Savings: 5–30 seconds per build on large projects

Enabling Configuration Cache

# gradle.properties
org.gradle.configuration-cache=true

# During migration — warn instead of fail on problems
org.gradle.configuration-cache.problems=warn

Restrictions

The configuration cache serializes the task graph and all task inputs. This means certain patterns that worked before are now forbidden:

// BREAKS configuration cache — accessing Project at execution time
abstract class BadTask : DefaultTask() {
    @TaskAction
    fun run() {
        val version = project.version  // ERROR: cannot access Project
        println("Building version $version")
    }
}

// FIXED — pass the value as an input property
abstract class GoodTask : DefaultTask() {
    @get:Input
    abstract val version: Property<String>

    @TaskAction
    fun run() {
        println("Building version ${version.get()}")
    }
}

tasks.register<GoodTask>("printVersion") {
    version.set(project.version.toString())  // resolved at configuration time, serialized
}

Other restrictions:

  • Cannot use Task.project in task actions
  • Cannot read system properties or environment variables without declaring them as build inputs
  • Cannot use Gradle.buildFinished { } or Gradle.taskGraph.whenReady { }
  • Some older plugins break configuration cache — check plugin compatibility before enabling
Migration Strategy

Enable with problems=warn first. Fix violations one plugin/module at a time. The most common violations are Task.project access in task actions and build-finished listeners. AGP and most Jetpack plugins are configuration-cache compatible since AGP 8.0+.

6. Parallel Execution

Gradle can execute tasks from independent modules simultaneously. This is the simplest optimization to enable and one of the most impactful for multi-module projects.

Enabling Parallel Execution

# gradle.properties
org.gradle.parallel=true
org.gradle.workers.max=8  # default: number of CPU cores

How Parallel Execution Works

Module dependency graph:

  :app ──▶ :feature-home ──▶ :core-data ──▶ :core-network
    │                            │
    ├──▶ :feature-search ───────┘
    │
    └──▶ :feature-profile ──▶ :core-ui

Sequential execution:
  :core-network → :core-data → :core-ui → :feature-home →
  :feature-search → :feature-profile → :app
  Total: sum of all module build times

Parallel execution (4 workers):
  Worker 1: :core-network → :core-data → :feature-home ──┐
  Worker 2: :core-ui → :feature-profile ─────────────────┤
  Worker 3: (idle, waiting for :core-data) → :feature-search ──┤
  Worker 4: (idle) ──────────────────────────────────────┤
                                                         ▼
                                                       :app
  Total: length of critical path (much shorter)

Key points:

  • Parallelism works at the module level — tasks within the same module still execute sequentially
  • The speedup is bounded by the critical path — the longest chain of dependent modules
  • More independent modules = more parallelism = faster builds
  • A monolithic single-module app gets zero benefit from parallel execution
Parallel + Modularization Synergy

This is why modularization and build performance are deeply connected. A well-modularized project with a wide, shallow dependency graph maximizes parallel execution. A deep, narrow graph (A→B→C→D→E) serializes the build regardless of how many cores you have.

7. Incremental Compilation — Kotlin and Java

Even when a compilation task must execute (not UP-TO-DATE or FROM-CACHE), Kotlin and Java compilers can limit recompilation to only the files affected by a change.

Kotlin Incremental Compilation

# gradle.properties (both enabled by default)
kotlin.incremental=true
kotlin.incremental.useClasspathSnapshot=true

Kotlin's incremental compiler uses ABI (Application Binary Interface) change detection:

Change in UserRepository.kt:

  Case 1: Implementation-only change (private method body)
  ┌──────────────────────────────────────────────────────┐
  │  ABI unchanged → only UserRepository.kt recompiles   │
  │  Dependents: NOT recompiled                           │
  └──────────────────────────────────────────────────────┘

  Case 2: Public API change (new public method signature)
  ┌──────────────────────────────────────────────────────┐
  │  ABI changed → UserRepository.kt + all files that    │
  │  import UserRepository recompile                      │
  └──────────────────────────────────────────────────────┘

  Case 3: Change to a public inline function
  ┌──────────────────────────────────────────────────────┐
  │  Inline functions are copied into call sites →        │
  │  ALL callers must recompile (even across modules)     │
  └──────────────────────────────────────────────────────┘

Classpath Snapshots

kotlin.incremental.useClasspathSnapshot=true enables more precise change detection. Instead of treating the entire classpath as a single input, Kotlin takes a snapshot of the ABI of each dependency. If a dependency's implementation changes but its ABI stays the same, downstream modules skip recompilation entirely.

What Breaks Incremental Compilation

  • Annotation processors (kapt) — kapt generates Java stubs for the entire module, then runs processors on all stubs. Any change triggers full reprocessing.
  • Public inline functions — inlined into call sites, so changes propagate to all callers
  • Constant values — const val is inlined by the compiler; changing the value requires recompiling all users
  • Sealed class hierarchies — adding a subclass changes the ABI of the sealed class, triggering recompilation of all when expressions
L6 Insight — API vs Implementation Modules

This is why the api/implementation dependency split matters for build performance, not just encapsulation. When module A depends on module B via implementation, changes to B's transitive dependencies don't appear on A's compile classpath — so A doesn't recompile. With api, B's transitive dependencies leak onto A's classpath, and any ABI change in those transitives triggers recompilation of A. Overusing api creates a "recompilation cascade" that defeats incremental compilation.

8. kapt vs KSP — Annotation Processing Performance

Annotation processing is often the single largest contributor to slow incremental builds. Understanding why kapt is slow and how KSP fixes it is essential for L6-level build optimization.

How kapt Works (and Why It's Slow)

kapt pipeline:

  Kotlin source files
       │
       ▼
  ┌─────────────────────────┐
  │  Generate Java stubs    │  ← Kotlin compiler creates .java stubs
  │  (ALL Kotlin files)     │     for every Kotlin class (slow, non-incremental)
  └────────────┬────────────┘
               │
               ▼
  ┌─────────────────────────┐
  │  Run Java annotation    │  ← Standard javac annotation processing
  │  processors on stubs    │     (processes ALL stubs, not just changed ones)
  └────────────┬────────────┘
               │
               ▼
  ┌─────────────────────────┐
  │  Compile Kotlin with    │  ← Normal Kotlin compilation
  │  generated sources      │
  └─────────────────────────┘

Problems:
  1. Stub generation processes ALL files on every change
  2. Java AP runs on ALL stubs (even if only one file changed)
  3. Three compilation passes instead of one
  4. Stubs are Java → loses Kotlin-specific information

How KSP Works (and Why It's Fast)

KSP pipeline:

  Kotlin source files
       │
       ▼
  ┌─────────────────────────┐
  │  Kotlin compiler with   │  ← KSP runs as a compiler plugin
  │  KSP plugin             │     (no stub generation, no javac)
  │                         │
  │  • Processes Kotlin     │  ← Direct access to Kotlin symbols
  │    symbols directly     │     (no information loss)
  │  • Incremental by       │  ← Only processes changed files +
  │    default              │     files that depend on changes
  └─────────────────────────┘

Advantages:
  1. No stub generation step
  2. Incremental — only processes changed files
  3. Single compilation pass
  4. Native Kotlin support (sealed classes, inline, etc.)
  5. 2–5x faster than kapt for most processors

Migrating from kapt to KSP

// build.gradle.kts — BEFORE (kapt)
plugins {
    id("org.jetbrains.kotlin.kapt")
}

dependencies {
    kapt("com.google.dagger:hilt-compiler:2.51.1")
    kapt("androidx.room:room-compiler:2.6.1")
    kapt("com.squareup.moshi:moshi-kotlin-codegen:1.15.1")
}

// build.gradle.kts — AFTER (KSP)
plugins {
    id("com.google.devtools.ksp")
}

dependencies {
    ksp("com.google.dagger:hilt-compiler:2.51.1")
    ksp("androidx.room:room-compiler:2.6.1")
    ksp("com.squareup.moshi:moshi-kotlin-codegen:1.15.1")
}
Library KSP Support Notes
Dagger / Hilt Since 2.48 Full KSP support, recommended by Google
Room Since 2.5.0 Full KSP support
Moshi Since 1.13.0 Full KSP support
Glide Since 4.14.0 KSP for generated API
Auto-Service Via auto-service-ksp Third-party KSP adapter
L6 Insight — kapt Is the #1 Build Performance Kill

On a typical 50-module Android project, kapt accounts for 30–50% of incremental build time. Migrating to KSP is the single highest-ROI build performance optimization for most teams. The migration is mechanical (change kapt() to ksp() in dependency declarations), and the payoff is immediate: 2–5x faster annotation processing, proper incremental support, and elimination of the stub generation step. If you can only do one build optimization, do this one.

9. Gradle Daemon Tuning

The Gradle Daemon is a long-lived JVM process that stays alive between builds, avoiding the 5–15 second JVM startup cost. Tuning the daemon's JVM settings can significantly impact build performance, especially for large projects.

Recommended gradle.properties

# gradle.properties — optimized for a large multi-module project

# Daemon JVM settings
org.gradle.jvmargs=-Xmx4g \
  -XX:MaxMetaspaceSize=512m \
  -XX:+HeapDumpOnOutOfMemoryError \
  -XX:+UseG1GC \
  -Dfile.encoding=UTF-8

# Parallel execution
org.gradle.parallel=true
org.gradle.workers.max=8

# Caching
org.gradle.caching=true
org.gradle.configuration-cache=true

# File system watching (Gradle 7+)
org.gradle.vfs.watch=true

# Daemon lifecycle
org.gradle.daemon.idletimeout=10800000

# Kotlin
kotlin.incremental=true
kotlin.incremental.useClasspathSnapshot=true

JVM Settings Explained

Setting Default Recommended Why
-Xmx 512m 4g (large project) Prevents OOM during compilation of large modules; too high wastes memory
-XX:MaxMetaspaceSize Unlimited 512m Caps class metadata memory; prevents runaway metaspace from many plugins
-XX:+UseG1GC G1GC (JDK 11+) G1GC Good throughput/latency balance for large heaps; default on modern JDKs
-XX:+HeapDumpOnOutOfMemoryError Off On Produces a heap dump for post-mortem analysis when OOM occurs

File System Watching

# gradle.properties
org.gradle.vfs.watch=true

Without file system watching, Gradle scans the entire project directory tree at the start of every build to detect changes. With VFS watching enabled, Gradle uses OS-level file watchers (inotify on Linux, FSEvents on macOS) to track changes in real time. The daemon maintains a virtual file system (VFS) snapshot between builds and only updates entries for files that actually changed.

Impact: UP-TO-DATE checks become near-instant instead of requiring a full directory scan. Most noticeable on projects with thousands of source files.

10. CI Optimization

CI builds have different characteristics than developer builds: they often start from a clean state, run on shared infrastructure, and execute the full test suite. Optimizing CI requires a different set of strategies.

Remote Build Cache — The Biggest CI Win

// settings.gradle.kts — CI cache configuration
buildCache {
    local {
        isEnabled = true
    }
    remote<HttpBuildCache> {
        url = uri("https://cache.mycompany.com/cache/")
        // CI pushes to cache; developers only pull
        isPush = System.getenv("CI") != null
    }
}

The remote build cache transforms the CI/developer workflow:

  • CI builds execute all tasks and push outputs to the remote cache
  • Developer builds pull cached outputs — a "clean" build on a developer machine behaves like an incremental build
  • PR builds benefit from the main branch cache — only tasks affected by the PR's changes actually execute

Test Parallelization and Sharding

// In a module's build.gradle.kts
tasks.withType<Test>().configureEach {
    // Run test classes in parallel within this module
    maxParallelForks = (Runtime.getRuntime().availableProcessors() / 2)
        .coerceAtLeast(1)

    // Restart the test JVM after N tests to prevent memory leaks
    forkEvery = 100
}

For large test suites, parallelize across CI nodes using test sharding:

// Custom test sharding — split tests across N CI nodes
tasks.withType<Test>().configureEach {
    val totalShards = System.getenv("CI_TOTAL_SHARDS")?.toIntOrNull() ?: 1
    val shardIndex = System.getenv("CI_SHARD_INDEX")?.toIntOrNull() ?: 0

    filter {
        // Include only tests assigned to this shard
        // (hash test class name to assign to shard deterministically)
        setIncludePatterns(
            testClassNames
                .filter { name ->
                    abs(name.hashCode()) % totalShards == shardIndex
                }
                .map { "$it.*" }
        )
    }
}

CI Anti-Patterns

Stop Doing These on CI
  • ./gradlew clean assembleDebug — clean destroys all incremental build state. Remove it unless you're debugging a specific caching issue.
  • Ephemeral containers without remote cache — every build starts cold, no daemon warmth, no local cache. Either use persistent agents or a remote cache.
  • --no-daemon on persistent agents — the daemon startup cost is amortized over multiple builds on persistent agents. Only use --no-daemon on truly ephemeral containers.
  • Running all tests on every PR — use test impact analysis or affected-module detection to only run tests for changed modules.

Affected Module Detection

// Pseudo-code for affected module detection in CI
// Only build/test modules that changed or depend on changed modules

val changedFiles = getGitDiff("origin/main", "HEAD")
val changedModules = changedFiles.map { findOwningModule(it) }.toSet()
val affectedModules = changedModules.flatMap { getTransitiveDependents(it) }.toSet()

// Run only affected tests
// ./gradlew :feature-home:testDebug :feature-search:testDebug

Tools like AffectedModuleDetector (Dropbox) and Gradle Enterprise's predictive test selection automate this.

L6 Insight — The CI Cache Flywheel

The remote build cache creates a virtuous cycle: CI builds on main populate the cache with outputs for every task. PR builds share most inputs with main, so they get 80–95% cache hit rates and only execute tasks affected by the PR's changes. Developer builds pulling from the same cache turn 5-minute clean builds into 30-second incremental builds. The ROI compounds — every CI build makes every subsequent build faster. This is the single most impactful infrastructure investment for build performance at scale.

11. Staff-Level Corner Cases and Gotchas

buildSrc — The Silent Build Killer

buildSrc is a special Gradle directory that's compiled before any build script. The problem: any change to buildSrc invalidates the entire build. Every task in every module re-executes because Gradle considers buildSrc part of the build logic, and the build logic changed.

// The problem:
// buildSrc/src/main/kotlin/Dependencies.kt
object Versions {
    const val compose = "1.7.0"  // change this...
}
// → EVERY task in EVERY module re-executes

// The solution: move build logic to an included build
// build-logic/convention/src/main/kotlin/...
// settings.gradle.kts:
pluginManagement {
    includeBuild("build-logic")
}

With an included build (composite build), only modules that actually depend on the changed plugin re-execute. This is why the Now in Android project and most modern Android projects use build-logic/ instead of buildSrc/.

Gradle Version Upgrades and Cache Invalidation

The Gradle version is part of every build cache key. Upgrading Gradle from 8.9 to 8.10 invalidates every cached task output. The first build after an upgrade is effectively a clean build. Plan Gradle upgrades for low-traffic times and ensure CI populates the cache with the new version before developers pull.

Non-Deterministic Tasks

// BAD — timestamp in output makes the task non-deterministic
abstract class GenerateBuildInfo : DefaultTask() {
    @get:OutputFile
    abstract val outputFile: RegularFileProperty

    @TaskAction
    fun generate() {
        outputFile.get().asFile.writeText("""
            buildTime=${System.currentTimeMillis()}
            version=${project.version}
        """.trimIndent())
    }
}
// This task will NEVER get a cache hit because the output
// is different every time (different timestamp)

// FIXED — use a reproducible build timestamp
abstract class GenerateBuildInfo : DefaultTask() {
    @get:Input
    abstract val version: Property<String>

    @get:Input
    abstract val gitCommit: Property<String>

    @get:OutputFile
    abstract val outputFile: RegularFileProperty

    @TaskAction
    fun generate() {
        outputFile.get().asFile.writeText("""
            gitCommit=${gitCommit.get()}
            version=${version.get()}
        """.trimIndent())
    }
}

Build Performance Optimization Checklist

Optimization Phase Typical Savings Effort
kapt → KSP migration Execution 30–50% of incremental build Medium (mechanical)
Remote build cache Execution 60–90% of clean build Medium (infra setup)
Configuration cache Configuration 5–30s per build High (plugin compat)
Parallel execution Execution 20–50% (depends on DAG) Low (one flag)
Configuration avoidance Configuration 5–15s per build Medium (refactoring)
buildSrc → included build All Eliminates full rebuilds on build logic changes Medium (migration)
File system watching UP-TO-DATE checks 1–5s per build Low (one flag)
Daemon JVM tuning All Prevents OOM, reduces GC pauses Low (properties)
Affected module detection CI 50–80% of CI test time High (tooling)
Test sharding CI Linear with shard count Medium (CI config)
L6 Insight — The Optimization Order

When asked "how would you speed up our Android build?" in an interview, don't jump to solutions. Start with: "I'd run a Build Scan first." Then prioritize by ROI: (1) kapt→KSP migration (highest ROI, mechanical), (2) remote build cache (highest absolute savings), (3) parallel execution (one flag), (4) configuration cache (high effort but big payoff), (5) buildSrc→included build (if applicable). The order matters because each optimization's benefit depends on the others — there's no point optimizing configuration time if execution takes 10x longer.
