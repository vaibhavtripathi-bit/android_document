Build Performance

Profiling, configuration avoidance, build cache, incremental compilation, kapt vs KSP, daemon tuning, and CI optimization for large Android projects

1. Build Profiling — Measure Before You Optimize

L6 Staff Insight
Never optimize without profiling first. The bottleneck is rarely where you think it is. Teams routinely waste weeks "optimizing" configuration time when the real problem is a non-incremental annotation processor running on every build.

Profiling Tools

ToolCommand / SetupWhat It Shows
--profile ./gradlew assembleDebug --profile HTML report in build/reports/profile/ — task execution times, configuration time, dependency resolution
--scan ./gradlew assembleDebug --scan Detailed online report — cache hit rates, critical path, dependency insights, build comparison
--info ./gradlew assembleDebug --info Cache key computation, task skip reasons, dependency resolution details
Gradle Enterprise / Develocity Plugin in settings.gradle.kts Historical trends, build comparison, flaky test detection, cache performance dashboards

Key Metrics to Track

  • Total build time — wall-clock time from invocation to completion
  • Configuration phase time — time spent evaluating build.gradle.kts files (runs on every build)
  • Task execution time — time in the Execution phase (compilation, resource processing, dexing)
  • Cache hit rate — percentage of tasks served from cache (local + remote)
  • Critical path — the longest chain of dependent tasks that determines minimum build time

Reading a Build Scan

# Generate a build scan
./gradlew assembleDebug --scan

# The scan URL is printed at the end:
# Publishing build scan...
# https://gradle.com/s/abc123xyz

In the scan, focus on these tabs:

  • Performance → Configuration — time per project. If a single module takes 2s to configure, that's a red flag.
  • Performance → Task execution — sort by duration. The top 5 tasks are your optimization targets.
  • Build cache — hit/miss breakdown. Look for tasks that should be cached but aren't.
  • Timeline — visual Gantt chart of task execution. Reveals parallelism gaps and serial bottlenecks.

2. Configuration Avoidance

The Configuration phase runs for every build invocation — even ./gradlew help or ./gradlew tasks. In a 200-module project, eager configuration can add 10-30 seconds before any real work begins.

Lazy Task Registration

// BAD — eager: task is created and configured immediately
tasks.create("generateDocs") {
    doLast { /* ... */ }
}

// GOOD — lazy: task is created only when needed
tasks.register("generateDocs") {
    doLast { /* ... */ }
}

tasks.register() returns a TaskProvider<T> — a lazy reference. The task object is only instantiated when something in the build actually depends on it. For a ./gradlew assembleDebug invocation, tasks like generateDocs are never materialized.

Avoid allprojects / subprojects

// BAD — forces configuration of ALL modules
// root build.gradle.kts
allprojects {
    repositories {
        mavenCentral()
        google()
    }
}

subprojects {
    tasks.withType<KotlinCompile>().configureEach {
        kotlinOptions.jvmTarget = "17"
    }
}

// GOOD — use convention plugins (applied only where needed)
// build-logic/convention/src/main/kotlin/AndroidLibraryConvention.kt
class AndroidLibraryConvention : Plugin<Project> {
    override fun apply(project: Project) {
        project.tasks.withType<KotlinCompile>().configureEach {
            kotlinOptions.jvmTarget = "17"
        }
    }
}

Provider / Property API

abstract class ProcessDataTask : DefaultTask() {
    // Lazy — value resolved at execution time, not configuration time
    @get:Input
    abstract val inputCount: Property<Int>

    @get:InputFile
    abstract val dataFile: RegularFileProperty

    @get:OutputDirectory
    abstract val outputDir: DirectoryProperty

    @TaskAction
    fun process() {
        val count = inputCount.get()  // resolved NOW, at execution time
        val file = dataFile.get().asFile
        // ...
    }
}

// Wiring with lazy providers
tasks.register<ProcessDataTask>("processData") {
    inputCount.set(provider { computeExpensiveCount() })
    dataFile.set(layout.projectDirectory.file("data/input.json"))
    outputDir.set(layout.buildDirectory.dir("processed"))
}
Common Mistake
Calling .get() on a Provider during configuration forces eager resolution and defeats the purpose. Always pass Provider objects around — only call .get() inside @TaskAction methods.

What NOT to Do in build.gradle.kts

// DON'T — reads file at configuration time (runs on every build)
val versionName = File("version.txt").readText().trim()

// DO — defer to execution time
val versionName: Provider<String> = providers.fileContents(
    layout.projectDirectory.file("version.txt")
).asText.map { it.trim() }

// DON'T — network call at configuration time
val latestVersion = URL("https://api.example.com/version").readText()

// DON'T — expensive computation at configuration time
val generatedSources = fileTree("src").filter { it.extension == "kt" }.count()

3. Task Avoidance and UP-TO-DATE Checks

Gradle tracks the inputs and outputs of every task. If inputs haven't changed since the last successful execution, the task is marked UP-TO-DATE and skipped entirely.

Task Outcomes

OutcomeMeaning
EXECUTEDTask ran — inputs changed or no previous execution
UP-TO-DATEInputs unchanged since last execution — skipped
FROM-CACHEOutputs restored from build cache (local or remote)
SKIPPEDTask was disabled via onlyIf { } or enabled = false
NO-SOURCETask has @SkipWhenEmpty input files and none exist

Declaring Inputs and Outputs

@CacheableTask
abstract class TransformResourcesTask : DefaultTask() {

    @get:InputFiles
    @get:PathSensitive(PathSensitivity.RELATIVE)
    abstract val inputResources: ConfigurableFileCollection

    @get:Input
    abstract val minSdkVersion: Property<Int>

    @get:OutputDirectory
    abstract val outputDir: DirectoryProperty

    @TaskAction
    fun transform() {
        val outDir = outputDir.get().asFile
        outDir.deleteRecursively()
        inputResources.forEach { file ->
            // transform and write to outDir
        }
    }
}

Path Sensitivity

@PathSensitive controls what part of a file's path is included in the cache key:

SensitivityCache Key IncludesUse When
ABSOLUTEFull absolute pathAlmost never — breaks cache portability
RELATIVEPath relative to the containing directorySource files, resources — most common choice
NAME_ONLYJust the filenameFiles where only content + name matters, not directory structure
NONENothing (only file content)Files where path is irrelevant — e.g., a config file

Incremental Task Processing

abstract class IncrementalCompileTask : DefaultTask() {

    @get:Incremental
    @get:InputFiles
    @get:PathSensitive(PathSensitivity.RELATIVE)
    abstract val sources: ConfigurableFileCollection

    @get:OutputDirectory
    abstract val outputDir: DirectoryProperty

    @TaskAction
    fun compile(inputChanges: InputChanges) {
        if (!inputChanges.isIncremental) {
            // Full rebuild — clean outputs and reprocess everything
            outputDir.get().asFile.deleteRecursively()
        }

        inputChanges.getFileChanges(sources).forEach { change ->
            when (change.changeType) {
                ChangeType.ADDED, ChangeType.MODIFIED -> {
                    processFile(change.file)
                }
                ChangeType.REMOVED -> {
                    deleteOutputFor(change.file)
                }
            }
        }
    }
}
Key Concept
@Incremental + InputChanges lets a task process only changed files rather than the entire input set. This is the difference between a 200ms incremental build and a 20-second full reprocess.

4. Build Cache Deep Dive

How It Works

Task Inputs                    Cache Key                    Cache Entry
┌──────────────────┐    ┌──────────────────────┐    ┌──────────────────┐
│ Task class        │    │                      │    │                  │
│ Task inputs hash  │───▶│  SHA-256 composite   │───▶│  Stored outputs  │
│ Gradle version    │    │  hash = cache key    │    │  (tar archive)   │
│ JVM version       │    │                      │    │                  │
│ Classpath         │    └──────────────────────┘    └──────────────────┘
└──────────────────┘              │
                                  ▼
                    ┌─────────────────────────────┐
                    │  Lookup: local → remote      │
                    │  Hit? → restore outputs      │
                    │  Miss? → execute → store     │
                    └─────────────────────────────┘

Local vs Remote Cache

AspectLocal CacheRemote Cache
Location~/.gradle/caches/build-cache-1/HTTP server (Develocity, custom, GCS/S3)
ScopeSingle machineShared across team + CI
SpeedDisk I/O (fast)Network I/O (slower, but still faster than recompilation)

Enabling the Build Cache

# gradle.properties
org.gradle.caching=true
// settings.gradle.kts — remote cache configuration
buildCache {
    local {
        isEnabled = true
        directory = File(rootDir, ".gradle/build-cache")
        removeUnusedEntriesAfterDays = 7
    }
    remote<HttpBuildCache> {
        url = uri("https://cache.example.com/cache/")
        isPush = System.getenv("CI") != null  // only CI pushes
        credentials {
            username = System.getenv("CACHE_USER") ?: ""
            password = System.getenv("CACHE_PASS") ?: ""
        }
    }
}
L6 Staff Insight
Remote build cache can cut clean build times by 60-80%. A developer running ./gradlew assembleDebug on a fresh checkout reuses cached outputs from the last CI build. The key design decision is push policy: only CI should push to remote cache (to avoid polluting it with developer-specific outputs), while developers only pull.

Making Tasks Cacheable

abstract class GenerateApiClientTask : DefaultTask() {

    abstract val specFile: RegularFileProperty

    @get:Input
    abstract val packageName: Property<String>

    @TaskAction
    fun generate() { /* ... */ }
}

Debugging Cache Misses

# Show cache key computation and miss reasons
./gradlew assembleDebug --info 2>&1 | grep -i "cache"

# Common output:
# Appending taskClass to build cache key: ...
# Appending inputPropertyHash for 'sources' to build cache key: abc123
# Build cache key for task ':app:compileDebugKotlin' is def456
# Task ':app:compileDebugKotlin' is not up-to-date because:
#   Value of input property 'sources' has changed for task

Common cache miss reasons:

  • Absolute paths in inputs — different machines have different project paths. Fix: use @PathSensitive(RELATIVE) or @PathSensitive(NONE)
  • Timestamps in outputs — generated files include build timestamps. Fix: make outputs deterministic
  • System properties leaking into inputs — System.getProperty() at configuration time. Fix: use explicit @Input properties
  • Non-deterministic task outputs — HashMap iteration order, random UUIDs. Fix: use sorted collections, deterministic IDs
  • Volatile inputs — Git SHA, build number changing on every commit. Fix: exclude from cache key or use @Internal

5. Configuration Cache

While the build cache stores task outputs, the configuration cache stores the entire task graph from the Configuration phase. On subsequent builds, Gradle deserializes the cached graph and skips Configuration entirely.

Impact

Without Configuration Cache:
  Init (0.5s) → Configuration (15s) → Execution (45s) = 60.5s total

With Configuration Cache (cache hit):
  Init (0.5s) → Deserialize graph (1s) → Execution (45s) = 46.5s total
                                                            ─────────
                                                            23% faster

For large projects (200+ modules), configuration can take 15-30 seconds. The configuration cache eliminates this on repeat builds.

Enabling

# gradle.properties
org.gradle.configuration-cache=true

# Optional: max entries (default 512)
org.gradle.configuration-cache.max-problems=20

Restrictions and Migration

The configuration cache serializes the task graph. Anything that can't be serialized or that depends on mutable state breaks it:

// BREAKS — accessing Project at execution time
abstract class BadTask : DefaultTask() {
    @TaskAction
    fun run() {
        val buildDir = project.buildDir  // ERROR: can't access Project
        val version = project.version    // ERROR
    }
}

// FIXED — declare inputs explicitly
abstract class GoodTask : DefaultTask() {
    @get:Input
    abstract val version: Property<String>

    @get:OutputDirectory
    abstract val outputDir: DirectoryProperty

    @TaskAction
    fun run() {
        val dir = outputDir.get().asFile
        val ver = version.get()
    }
}

Common migration fixes:

  • Replace project.buildDir with layout.buildDirectory
  • Replace project.version with an explicit @Input property
  • Replace project.rootDir with layout.projectDirectory
  • Remove afterEvaluate { } blocks — use lazy configuration instead
  • Avoid storing Project, Gradle, or Settings references in task fields
Migration Pain
Third-party plugins are the biggest blocker for configuration cache adoption. Check plugin compatibility before enabling. AGP has full support since 8.0. Many popular plugins (Hilt, Room, Firebase) now support it, but some smaller plugins may not.

6. Parallel Execution

# gradle.properties
org.gradle.parallel=true
org.gradle.workers.max=8  # default: number of CPU cores

How Parallel Execution Works

Module Dependency Graph:
                    :app
                   / | \
                  /  |  \
          :feature-a |  :feature-b
                |    |    |
                └────┼────┘
                     |
                  :core-data
                     |
                  :core-network

Parallel Execution Timeline:
  Worker 1: ──[:core-network]──[:core-data]──[:app]──────────▶
  Worker 2: ────────────────────[:feature-a]──────────────────▶
  Worker 3: ────────────────────[:feature-b]──────────────────▶
                                                    time ───▶

:feature-a and :feature-b build simultaneously once :core-data finishes.

Gradle determines parallelism from the task dependency graph. Two tasks can run in parallel if neither depends on the other (directly or transitively). This is why modularization matters for build performance — more independent modules means more parallelism.

Worker API for Intra-Task Parallelism

abstract class ParallelProcessTask : DefaultTask() {

    @get:Inject
    abstract val workerExecutor: WorkerExecutor

    @get:InputFiles
    @get:PathSensitive(PathSensitivity.RELATIVE)
    abstract val inputFiles: ConfigurableFileCollection

    @get:OutputDirectory
    abstract val outputDir: DirectoryProperty

    @TaskAction
    fun process() {
        inputFiles.forEach { file ->
            workerExecutor.noIsolation().submit(ProcessAction::class) {
                inputFile.set(file)
                outputDir.set(this@ParallelProcessTask.outputDir)
            }
        }
    }
}

abstract class ProcessAction : WorkAction<ProcessAction.Params> {
    interface Params : WorkParameters {
        val inputFile: RegularFileProperty
        val outputDir: DirectoryProperty
    }
    override fun execute() {
        // runs in parallel with other work items
    }
}
L6 Staff Insight
Parallel execution benefits scale with the number of independent modules. A monolithic :app module with everything in it gets zero parallelism — Gradle can only run one compilation task at a time. Breaking it into 20 feature modules with a shared :core layer lets Gradle compile all features simultaneously. This is the single strongest argument for modularization from a build-time perspective.

7. Incremental Compilation — Kotlin & Java

How Kotlin Incremental Compilation Works

Change in UserRepository.kt (implementation only, no ABI change)
                │
                ▼
  ┌─────────────────────────────┐
  │ Kotlin Compiler: ABI check  │
  │ Public API changed? NO      │
  │ → Recompile ONLY changed    │
  │   file: UserRepository.kt   │
  └─────────────────────────────┘

Change in UserRepository.kt (public method signature changed)
                │
                ▼
  ┌─────────────────────────────┐
  │ Kotlin Compiler: ABI check  │
  │ Public API changed? YES     │
  │ → Recompile changed file    │
  │   + all files that import   │
  │   UserRepository             │
  └─────────────────────────────┘
# gradle.properties — Kotlin incremental compilation settings
kotlin.incremental=true                          # default: true
kotlin.incremental.usePreciseJavaTracking=true   # track Java→Kotlin deps precisely
kotlin.incremental.useClasspathSnapshot=true     # ABI-based classpath tracking

ABI vs Implementation Changes

Change TypeExampleRecompilation Scope
Implementation only Change the body of a private function Only the changed file
ABI change (same module) Add a parameter to a public function Changed file + files in the same module that reference it
ABI change (cross-module) Change a public function in :core Changed file + all dependent modules that use the changed API
Key Insight
This is why implementation vs api dependency configuration matters for build performance. With implementation, an ABI change in :core-network only triggers recompilation in :core-data (its direct dependent). With api, it cascades to :feature-a, :feature-b, and :app — everything that transitively depends on it.

What Breaks Incrementality

  • Annotation processors (kapt) — kapt generates Java stubs from all Kotlin files, then runs annotation processors on everything. Any change triggers full reprocessing.
  • Constant changes — changing a const val in a companion object forces recompilation of all files that inline the constant.
  • Adding/removing files — can invalidate the incremental compilation cache in some edge cases.
  • Compiler plugin changes — updating a compiler plugin version invalidates all cached outputs.

8. kapt vs KSP

How kapt Works (and Why It's Slow)

kapt Pipeline:
  Kotlin sources ──▶ Generate Java stubs ──▶ Run Java AP ──▶ Compile Kotlin
                     (ALL files, every     (ALL stubs,
                      time)                 every time)

  Problem: stub generation is O(all Kotlin files), not O(changed files)
  Problem: Java annotation processors see stubs, not real Kotlin code

How KSP Works

KSP Pipeline:
  Kotlin sources ──▶ KSP processes Kotlin AST directly ──▶ Compile Kotlin
                     (only changed files in
                      incremental mode)

  No stub generation. No Java interop layer. Native Kotlin support.

Comparison

AspectkaptKSP
Stub generationRequired (slow)Not needed
IncrementalNo — full reprocess on any changeYes — processes only changed files
Speed (typical)Baseline2-3x faster
Kotlin supportSees Java stubs (lossy)Native Kotlin AST (full fidelity)
MultiplatformJVM onlyMultiplatform supported
Dagger/HiltSupportedSupported (Hilt 2.48+)
RoomSupportedSupported (Room 2.5.0+)
MoshiSupportedSupported (Moshi 1.13+)

Migration

// BEFORE: kapt
plugins {
    id("org.jetbrains.kotlin.kapt")
}

dependencies {
    kapt("com.google.dagger:hilt-compiler:2.51")
    kapt("androidx.room:room-compiler:2.6.1")
}

// AFTER: KSP
plugins {
    id("com.google.devtools.ksp") version "2.0.0-1.0.22"
}

dependencies {
    ksp("com.google.dagger:hilt-compiler:2.51")
    ksp("androidx.room:room-compiler:2.6.1")
}

You can use both kapt and KSP in the same module during migration — migrate one processor at a time, validate, then remove kapt when all processors are migrated.

L6 Staff Insight
Migrating from kapt to KSP is often the single biggest build performance win for Android projects. In a 50-module project, kapt can account for 30-40% of total build time because it forces full recompilation on every change. KSP's incremental processing means only changed files are reprocessed. At Google scale, this translates to saving thousands of developer-hours per quarter.

9. Gradle Daemon Tuning

JVM Arguments

# gradle.properties
org.gradle.jvmargs=-Xmx4g \
  -XX:+UseG1GC \
  -XX:MaxMetaspaceSize=512m \
  -XX:+HeapDumpOnOutOfMemoryError \
  -Dfile.encoding=UTF-8
SettingSmall Project (<20 modules)Medium (20-80 modules)Large (80+ modules)
-Xmx2g4g6-8g
-XX:MaxMetaspaceSize256m512m1g
workers.maxCPU coresCPU coresCPU cores - 2

Why G1GC

The default GC (Parallel GC) optimizes for throughput but causes long pauses on large heaps. G1GC divides the heap into regions and collects incrementally, keeping pause times short. For Gradle builds with 4GB+ heaps, G1GC reduces GC-induced stalls that show up as mysterious "idle time" in build scans.

Daemon Lifecycle

# gradle.properties
org.gradle.daemon=true                      # default: true
org.gradle.daemon.idletimeout=10800000      # 3 hours (default)

# Check running daemons
./gradlew --status

# Stop all daemons
./gradlew --stop

The daemon keeps the JVM warm between builds — JIT-compiled code, loaded classes, and resolved dependencies stay in memory. The first build after a daemon restart is 20-40% slower due to JIT warmup.

CI Consideration
On ephemeral CI containers (Docker, Kubernetes pods), the daemon provides no benefit because each build gets a fresh container. Use --no-daemon and allocate higher heap to compensate. On persistent CI agents, keep the daemon running for warmth.

Parallel Test Execution

// Module-level build.gradle.kts
tasks.withType<Test>().configureEach {
    maxParallelForks = (Runtime.getRuntime().availableProcessors() / 2)
        .coerceAtLeast(1)

    // Forked JVM settings for test workers
    jvmArgs("-Xmx512m")

    // Fail fast — stop on first failure in CI
    failFast = true
}

Test Sharding Across CI Agents

// Split tests across N CI agents using a simple hash-based approach
tasks.withType<Test>().configureEach {
    val shardIndex = System.getenv("SHARD_INDEX")?.toIntOrNull() ?: 0
    val shardCount = System.getenv("SHARD_COUNT")?.toIntOrNull() ?: 1

    filter {
        // Include only tests whose hash maps to this shard
        setIncludePatterns(
            "**/*Test*"
        )
    }

    // Use Gradle's built-in test filtering with a custom shard selector
    if (shardCount > 1) {
        exclude { element ->
            val hash = element.name.hashCode().absoluteValue
            hash % shardCount != shardIndex
        }
    }
}

L6 Staff Insight
The biggest CI optimization is remote build cache + persistent agents. Ephemeral containers throw away all caching benefits — every build starts cold. At scale, the cost of longer CI builds (more compute minutes) often exceeds the cost of maintaining persistent agents. The hybrid approach (persistent pool for main branch, ephemeral for PR builds) gives the best tradeoff: fast main builds with isolated PR validation.

L6 Staff Insight
android.nonTransitiveRClass=true is a sleeper optimization. By default, each module's R class includes all resource IDs from its transitive dependencies — meaning a change to a string in :core-ui triggers recompilation in every module that depends on it. Non-transitive R classes make each module's R class contain only its own resources, dramatically reducing recompilation scope. This is the default in new AGP projects but must be explicitly enabled in older ones.

Optimization Priority Order

If you're starting from scratch, apply optimizations in this order for maximum ROI:

  1. Profile first — run --scan to identify the actual bottleneck
  2. Enable parallel + caching — two lines in gradle.properties, massive impact
  3. Migrate kapt → KSP — often the single biggest win
  4. Tune JVM — right-size heap, switch to G1GC
  5. Enable non-transitive R classes — reduces recompilation cascade
  6. Configuration cache — high impact but requires migration effort
  7. Remote build cache — infrastructure investment, but transforms CI performance
  8. Modularize — the long game, but enables all other optimizations to compound
