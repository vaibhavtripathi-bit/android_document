Convention Plugins, Build Variants & Sharing Build Logic

Convention plugins via composite builds, buildSrc pitfalls, precompiled script plugins, AGP extension APIs, build variants and flavor dimensions, custom Gradle plugins, version catalog integration, and the staff-level corner cases that separate Architects from seniors

1. The Problem: Duplicated Build Logic

In a modular Android project, every module's build.gradle.kts repeats the same configuration: compileSdk, minSdk, Kotlin compiler options, test runners, lint rules, Compose compiler flags, and dependency declarations. This duplication is the single largest source of build maintenance pain in multi-module projects.

The Duplication Tax

// :feature:home/build.gradle.kts — one of 30 modules
android {
    compileSdk = 35
    defaultConfig {
        minSdk = 26
        testInstrumentationRunner = "androidx.test.runner.AndroidJUnitRunner"
    }
    compileOptions {
        sourceCompatibility = JavaVersion.VERSION_17
        targetCompatibility = JavaVersion.VERSION_17
    }
    kotlinOptions { jvmTarget = "17" }
    buildFeatures { compose = true }
}

dependencies {
    implementation(libs.hilt.android)
    ksp(libs.hilt.compiler)
    testImplementation(libs.junit)
    testImplementation(libs.mockk)
    testImplementation(libs.turbine)
    testImplementation(libs.coroutines.test)
}

// Now copy-paste this into 29 more modules...

The consequences compound as the project grows:

ProblemImpactReal-World Example
Consistency driftModules diverge silentlyOne module has minSdk = 24, another minSdk = 21 — crash on API 22 devices
Change amplificationOne logical change → N file editsBumping compileSdk from 34 to 35 requires editing 30 files
Onboarding frictionNew modules copy from random existing modulesDeveloper copies from an outdated module, misses Compose compiler flag
Review burdenBuild file PRs are noise30-file PR to add a test dependency — reviewers rubber-stamp it

The solution: extract shared build logic into convention plugins — reusable, composable, testable build configuration that lives in one place.

2. buildSrc — The Legacy Approach

Before convention plugins, buildSrc/ was the standard way to share build logic. It's still common in older projects and worth understanding — both for migration purposes and to explain why it was replaced.

How buildSrc Works

A buildSrc/ directory at the project root is automatically compiled by Gradle and added to the build classpath of every module. It can contain Kotlin/Java code, Gradle plugins, and dependency version constants.

project-root/
├── buildSrc/
│   ├── build.gradle.kts          // buildSrc's own build file
│   └── src/main/kotlin/
│       ├── Versions.kt           // Version constants
│       ├── Dependencies.kt       // Dependency coordinates
│       └── AndroidConventions.kt // Shared configuration functions
├── app/
│   └── build.gradle.kts          // Can reference buildSrc classes
├── feature/home/
│   └── build.gradle.kts          // Can reference buildSrc classes
└── settings.gradle.kts
// buildSrc/src/main/kotlin/Versions.kt
object Versions {
    const val compileSdk = 35
    const val minSdk = 26
    const val targetSdk = 35
    const val kotlin = "2.0.21"
}

// buildSrc/src/main/kotlin/Dependencies.kt
object Dependencies {
    const val hiltAndroid = "com.google.dagger:hilt-android:${Versions.hilt}"
    const val hiltCompiler = "com.google.dagger:hilt-compiler:${Versions.hilt}"
}

The Fatal Flaw

buildSrc Cache Invalidation
Any change to any file in buildSrc/ invalidates the entire build cache. Every module recompiles from scratch. In a 30-module project, changing a single version constant in buildSrc can turn a 30-second incremental build into a 5-minute full rebuild. Gradle treats buildSrc as a classpath dependency of every build script — if the classpath changes, all scripts must be re-evaluated and all task outputs are considered stale.

Additional buildSrc problems:

  • Compiled before everything — buildSrc is compiled before any build script is evaluated, adding startup latency on every build
  • Single project scope — can't share buildSrc across repositories without copy-pasting
  • No version isolation — all modules see the same buildSrc classpath, no way to version plugins independently
Staff Insight — buildSrc Is an Anti-Pattern for Large Projects
Use buildSrc only for trivial constants in small projects (under 5 modules). Never put plugins, custom tasks, or complex logic in buildSrc. The cache invalidation penalty is devastating at scale — a team of 10 engineers each making 20 builds/day loses hundreds of engineer-hours per month to unnecessary full rebuilds. In an interview, if you mention buildSrc, immediately follow with "but we migrated to convention plugins because of cache invalidation."
AspectbuildSrcConvention Plugins (build-logic)
Setup complexityTrivial — just create the directoryModerate — composite build setup
IDE supportExcellent — auto-detectedExcellent — after initial setup
Cache impactAny change invalidates everythingChanges only invalidate build-logic
ReusabilitySingle project onlyShareable across projects
RecommendationSmall projects, rarely-changed constantsAny project with > 5 modules

3. Convention Plugins — The Modern Approach

Convention plugins are precompiled script plugins that live in a dedicated build-logic/ module, included via Gradle's composite build mechanism. This is the approach used by Google's Now in Android project and recommended by the Gradle team.

Core Idea

Each plugin configures exactly one concern. Modules compose them together:

  • AndroidApplicationConventionPlugin — base Android app configuration (compileSdk, minSdk, Java/Kotlin options)
  • AndroidLibraryConventionPlugin — base Android library configuration
  • AndroidComposeConventionPlugin — Compose compiler + BOM + dependencies
  • AndroidFeatureConventionPlugin — composes library + compose + hilt + common feature deps
  • KotlinLibraryConventionPlugin — pure Kotlin/JVM modules (no Android)
  • AndroidTestConventionPlugin — test runner, test dependencies, coverage

Project Structure

project-root/
├── build-logic/
│   ├── convention/
│   │   ├── build.gradle.kts                        ← Declares plugin dependencies (AGP, KGP, etc.)
│   │   └── src/main/kotlin/
│   │       ├── AndroidApplicationConventionPlugin.kt
│   │       ├── AndroidLibraryConventionPlugin.kt
│   │       ├── AndroidFeatureConventionPlugin.kt
│   │       ├── AndroidComposeConventionPlugin.kt
│   │       ├── KotlinLibraryConventionPlugin.kt
│   │       ├── AndroidTestConventionPlugin.kt
│   │       └── KotlinAndroid.kt                     ← Shared extension functions
│   └── settings.gradle.kts                          ← build-logic's own settings
├── app/
│   └── build.gradle.kts                             ← plugins { id("myapp.android.application") }
├── feature/home/
│   └── build.gradle.kts                             ← plugins { id("myapp.android.feature") }
├── core/network/
│   └── build.gradle.kts                             ← plugins { id("myapp.android.library") }
└── settings.gradle.kts                              ← includeBuild("build-logic")

Step 3: build-logic/convention/build.gradle.kts

// build-logic/convention/build.gradle.kts
plugins {
    `kotlin-dsl`  // Enables writing Gradle plugins in Kotlin
}

dependencies {
    compileOnly(libs.android.gradlePlugin)  // AGP classes (ApplicationExtension, LibraryExtension)
    compileOnly(libs.kotlin.gradlePlugin)   // Kotlin Gradle Plugin classes (KotlinCompile)
    compileOnly(libs.compose.gradlePlugin)  // Compose compiler plugin classes
    compileOnly(libs.ksp.gradlePlugin)      // KSP plugin classes
}

// Register all convention plugins
gradlePlugin {
    plugins {
        register("androidApplication") {
            id = "myapp.android.application"
            implementationClass = "AndroidApplicationConventionPlugin"
        }
        register("androidLibrary") {
            id = "myapp.android.library"
            implementationClass = "AndroidLibraryConventionPlugin"
        }
        register("androidFeature") {
            id = "myapp.android.feature"
            implementationClass = "AndroidFeatureConventionPlugin"
        }
        register("androidCompose") {
            id = "myapp.android.compose"
            implementationClass = "AndroidComposeConventionPlugin"
        }
        register("androidHilt") {
            id = "myapp.android.hilt"
            implementationClass = "AndroidHiltConventionPlugin"
        }
        register("kotlinLibrary") {
            id = "myapp.kotlin.library"
            implementationClass = "KotlinLibraryConventionPlugin"
        }
    }
}
Staff Insight — Why compileOnly for Plugin Dependencies?
Convention plugin dependencies on AGP/KGP/Compose must be compileOnly, not implementation. The actual plugin JARs are already on the classpath when the main build applies them via plugins { }. Using implementation would create duplicate classpath entries and version conflicts. The compileOnly scope gives you compile-time access to extension types (ApplicationExtension, LibraryExtension) without bundling the JARs into the plugin output. This is the same pattern as a servlet container — you compile against the API, but the runtime provides the implementation.

Step 5: Shared Kotlin Configuration

// build-logic/convention/src/main/kotlin/KotlinAndroid.kt
import com.android.build.api.dsl.CommonExtension
import org.gradle.api.JavaVersion
import org.gradle.api.Project
import org.gradle.api.artifacts.VersionCatalog
import org.gradle.api.artifacts.VersionCatalogsExtension
import org.gradle.kotlin.dsl.assign
import org.gradle.kotlin.dsl.configure
import org.gradle.kotlin.dsl.getByType
import org.gradle.kotlin.dsl.withType
import org.jetbrains.kotlin.gradle.dsl.JvmTarget
import org.jetbrains.kotlin.gradle.tasks.KotlinCompile

internal val Project.libs: VersionCatalog
    get() = extensions.getByType<VersionCatalogsExtension>().named("libs")

internal fun Project.configureKotlinAndroid(
    commonExtension: CommonExtension<*, *, *, *, *, *>,
) {
    commonExtension.apply {
        compileSdk = 35

        defaultConfig {
            minSdk = 26
            testInstrumentationRunner = "androidx.test.runner.AndroidJUnitRunner"
        }

        compileOptions {
            sourceCompatibility = JavaVersion.VERSION_17
            targetCompatibility = JavaVersion.VERSION_17
        }
    }

    configureKotlin()
}

internal fun Project.configureKotlin() {
    tasks.withType<KotlinCompile>().configureEach {
        compilerOptions {
            jvmTarget = JvmTarget.JVM_17
            freeCompilerArgs.addAll(
                "-opt-in=kotlinx.coroutines.ExperimentalCoroutinesApi",
                "-opt-in=kotlinx.coroutines.FlowPreview",
            )
        }
    }
}

Step 6: The Plugin Classes

// AndroidApplicationConventionPlugin.kt
import com.android.build.api.dsl.ApplicationExtension
import org.gradle.api.Plugin
import org.gradle.api.Project
import org.gradle.kotlin.dsl.configure

class AndroidApplicationConventionPlugin : Plugin<Project> {
    override fun apply(target: Project) {
        with(target) {
            pluginManager.apply("com.android.application")
            pluginManager.apply("org.jetbrains.kotlin.android")

            extensions.configure<ApplicationExtension> {
                configureKotlinAndroid(this)
                defaultConfig.targetSdk = 35
                buildFeatures.buildConfig = true
            }
        }
    }
}
// AndroidLibraryConventionPlugin.kt
import com.android.build.gradle.LibraryExtension
import org.gradle.api.Plugin
import org.gradle.api.Project
import org.gradle.kotlin.dsl.configure

class AndroidLibraryConventionPlugin : Plugin<Project> {
    override fun apply(target: Project) {
        with(target) {
            pluginManager.apply("com.android.library")
            pluginManager.apply("org.jetbrains.kotlin.android")

            extensions.configure<LibraryExtension> {
                configureKotlinAndroid(this)
                buildFeatures.buildConfig = false
            }
        }
    }
}
// AndroidFeatureConventionPlugin.kt — composes other convention plugins
class AndroidFeatureConventionPlugin : Plugin<Project> {
    override fun apply(target: Project) {
        with(target) {
            pluginManager.apply("myapp.android.library")
            pluginManager.apply("myapp.android.compose")
            pluginManager.apply("myapp.android.hilt")

            dependencies {
                add("implementation", project(":core:ui"))
                add("implementation", project(":core:data"))
                add("implementation", project(":core:navigation"))
                add("implementation", libs.findLibrary("lifecycle.viewmodel.compose").get())
                add("implementation", libs.findLibrary("navigation.compose").get())
                add("implementation", libs.findLibrary("hilt.navigation.compose").get())
            }
        }
    }
}

5. Precompiled Script Plugins

An alternative to class-based plugins: place .gradle.kts files directly in build-logic/convention/src/main/kotlin/. The file name becomes the plugin ID.

How It Works

// build-logic/convention/src/main/kotlin/myapp.android.library.gradle.kts
// File name → plugin ID: "myapp.android.library"

plugins {
    id("com.android.library")
    id("org.jetbrains.kotlin.android")
}

android {
    compileSdk = 35
    defaultConfig {
        minSdk = 26
        testInstrumentationRunner = "androidx.test.runner.AndroidJUnitRunner"
    }
    compileOptions {
        sourceCompatibility = JavaVersion.VERSION_17
        targetCompatibility = JavaVersion.VERSION_17
    }
}

tasks.withType<org.jetbrains.kotlin.gradle.tasks.KotlinCompile>().configureEach {
    compilerOptions {
        jvmTarget.set(org.jetbrains.kotlin.gradle.dsl.JvmTarget.JVM_17)
    }
}

Precompiled Script vs. Class-Based: When to Use Which

CriterionPrecompiled Script PluginClass-Based Plugin
SyntaxLooks like a regular build.gradle.ktsImplements Plugin<Project>
Plugin IDDerived from filenameRegistered in gradlePlugin { }
Conditional logicLimited — no constructor args, no genericsFull Kotlin power
Accessing AGP extensionsDirect via android { } blockVia extensions.configure<T> { }
ComposabilityCan apply other plugins, but can't be parameterizedCan accept configuration, compose flexibly
Best forSimple, static configurationComplex logic, shared across concerns
Precompiled Script Gotcha
Precompiled script plugins can't easily access the version catalog's libs accessor — the same limitation as class-based plugins. You must use the VersionCatalogsExtension API. Additionally, precompiled scripts are compiled into classes under the hood, so they have the same classpath requirements as class-based plugins. The "simpler syntax" is syntactic sugar, not a fundamentally different mechanism.

6. AGP Extension API in Convention Plugins

Convention plugins configure Android modules by accessing AGP's extension objects. Understanding which extension to use and how to access it is critical.

Extension Type Hierarchy

CommonExtension<*, *, *, *, *, *>
├── ApplicationExtension        ← com.android.application modules
├── LibraryExtension             ← com.android.library modules
├── TestExtension                ← com.android.test modules
└── DynamicFeatureExtension      ← com.android.dynamic-feature modules

Use CommonExtension when your plugin should work with ANY Android module type.
Use ApplicationExtension/LibraryExtension when you need type-specific APIs.

Compose Convention Plugin (Using CommonExtension)

// AndroidComposeConventionPlugin.kt
import com.android.build.api.dsl.CommonExtension
import org.gradle.api.Plugin
import org.gradle.api.Project
import org.gradle.kotlin.dsl.configure
import org.gradle.kotlin.dsl.dependencies

class AndroidComposeConventionPlugin : Plugin<Project> {
    override fun apply(target: Project) {
        with(target) {
            pluginManager.apply("org.jetbrains.kotlin.plugin.compose")

            extensions.configure<CommonExtension<*, *, *, *, *, *>> {
                buildFeatures {
                    compose = true
                }
            }

            dependencies {
                val bom = libs.findLibrary("compose.bom").get()
                add("implementation", platform(bom))
                add("androidTestImplementation", platform(bom))
                add("implementation", libs.findLibrary("compose.ui").get())
                add("implementation", libs.findLibrary("compose.ui.tooling.preview").get())
                add("implementation", libs.findLibrary("compose.material3").get())
                add("debugImplementation", libs.findLibrary("compose.ui.tooling").get())
                add("debugImplementation", libs.findLibrary("compose.ui.test.manifest").get())
            }
        }
    }
}

7. Build Variants, Flavors, and Dimensions

Build variants are the mechanism for producing different versions of your app from the same codebase. Understanding the combinatorial math and performance implications is critical for Architect-level decisions.

Build Types

android {
    buildTypes {
        debug {
            isDebuggable = true
            isMinifyEnabled = false
            applicationIdSuffix = ".debug"
            versionNameSuffix = "-debug"
        }
        release {
            isMinifyEnabled = true
            isShrinkResources = true
            proguardFiles(
                getDefaultProguardFile("proguard-android-optimize.txt"),
                "proguard-rules.pro"
            )
            signingConfig = signingConfigs.getByName("release")
        }
        create("staging") {
            initWith(getByName("debug"))
            applicationIdSuffix = ".staging"
            versionNameSuffix = "-staging"
            matchingFallbacks += listOf("debug")
        }
        create("benchmark") {
            initWith(getByName("release"))
            signingConfig = signingConfigs.getByName("debug")
            matchingFallbacks += listOf("release")
            isDebuggable = false
        }
    }
}

Product Flavors and Dimensions

android {
    flavorDimensions += listOf("tier", "environment")

    productFlavors {
        create("free") {
            dimension = "tier"
            applicationIdSuffix = ".free"
            buildConfigField("Boolean", "IS_PREMIUM", "false")
            buildConfigField("Int", "MAX_DOWNLOADS", "5")
        }
        create("paid") {
            dimension = "tier"
            buildConfigField("Boolean", "IS_PREMIUM", "true")
            buildConfigField("Int", "MAX_DOWNLOADS", "Integer.MAX_VALUE")
        }

        create("staging") {
            dimension = "environment"
            buildConfigField("String", "API_URL", "\"https://staging-api.example.com\"")
            manifestPlaceholders["appLabel"] = "MyApp (Staging)"
        }
        create("production") {
            dimension = "environment"
            buildConfigField("String", "API_URL", "\"https://api.example.com\"")
            manifestPlaceholders["appLabel"] = "MyApp"
        }
    }
}

Variant Math

Variants = buildTypes × flavor_dim_1 × flavor_dim_2 × ...

Example: 4 build types × 2 tiers × 2 environments = 16 variants

  freeStagingDebug       paidStagingDebug
  freeStagingRelease     paidStagingRelease
  freeStagingStaging     paidStagingStaging       ← confusing name, but valid
  freeStagingBenchmark   paidStagingBenchmark
  freeProductionDebug    paidProductionDebug
  freeProductionRelease  paidProductionRelease
  freeProductionStaging  paidProductionStaging
  freeProductionBenchmark paidProductionBenchmark

Each variant generates:
  • Its own source set (src/freeStagingDebug/kotlin/)
  • Its own resources (src/freeStagingDebug/res/)
  • Its own manifest merge
  • Its own R class generation
  • Its own compilation tasks (Kotlin, Java, resource processing)
  • Its own APK/AAB output

Source Set Merging Order

// For variant "freeStagingDebug", source sets merge in priority order:
// 1. src/main/              ← base (lowest priority)
// 2. src/free/              ← tier flavor
// 3. src/staging/           ← environment flavor
// 4. src/freeStaging/       ← multi-flavor combination
// 5. src/debug/             ← build type
// 6. src/freeStagingDebug/  ← full variant (highest priority)

// Resources: later values override earlier (string in src/debug/ overrides src/main/)
// Kotlin/Java: source sets are additive — duplicate class = compile error
// Manifests: merged with priority order above

buildConfigField and manifestPlaceholders

android {
    defaultConfig {
        buildConfigField("String", "BASE_URL", "\"https://api.example.com\"")
        manifestPlaceholders["deepLinkScheme"] = "myapp"
    }

    buildTypes {
        debug {
            buildConfigField("String", "BASE_URL", "\"https://debug-api.example.com\"")
            manifestPlaceholders["deepLinkScheme"] = "myapp-debug"
        }
    }
}

val url = BuildConfig.BASE_URL

// In AndroidManifest.xml:

Variant Filtering — Remove What You Don't Ship

androidComponents {
    beforeVariants { variantBuilder ->
        // Nobody ships freeStagingRelease — disable it
        if (variantBuilder.productFlavors.containsAll(
                listOf("tier" to "free", "environment" to "staging")
            ) && variantBuilder.buildType == "release"
        ) {
            variantBuilder.enable = false
        }

        // Benchmark only makes sense for production release
        if (variantBuilder.buildType == "benchmark" &&
            variantBuilder.productFlavors.any { it.second != "production" }
        ) {
            variantBuilder.enable = false
        }
    }
}
Staff Insight — Variant Explosion Is a Hidden Performance Tax
Every variant multiplies the number of AGP tasks: resource merging, AIDL compilation, R class generation, Kotlin compilation, dexing, packaging. With 16 variants across 30 modules, Gradle must configure 480 sets of tasks — even if you're only building one variant. This is why beforeVariants { variantBuilder.enable = false } is not optional at scale. In a real project I've seen configuration time drop from 18 seconds to 6 seconds just by disabling 8 unused variant combinations. Always audit: if you never build freeProductionBenchmark, disable it.

8. Custom Gradle Plugins

When convention plugins aren't enough — custom code generation, bytecode transformation, artifact manipulation, or publishing reusable plugins to Maven — you need a full class-based Gradle plugin.

Plugin Implementation

// build-logic/convention/src/main/kotlin/BuildMetadataPlugin.kt
import org.gradle.api.DefaultTask
import org.gradle.api.Plugin
import org.gradle.api.Project
import org.gradle.api.file.RegularFileProperty
import org.gradle.api.provider.Property
import org.gradle.api.tasks.Input
import org.gradle.api.tasks.OutputFile
import org.gradle.api.tasks.TaskAction

abstract class GenerateBuildMetadataTask : DefaultTask() {
    @get:Input
    abstract val gitSha: Property<String>

    @get:Input
    abstract val buildTimestamp: Property<String>

    @get:Input
    abstract val variantName: Property<String>

    @get:OutputFile
    abstract val outputFile: RegularFileProperty

    @TaskAction
    fun generate() {
        val content = buildString {
            appendLine("package com.myapp.buildmeta")
            appendLine()
            appendLine("object BuildMetadata {")
            appendLine("    const val GIT_SHA = \"${gitSha.get()}\"")
            appendLine("    const val BUILD_TIME = \"${buildTimestamp.get()}\"")
            appendLine("    const val VARIANT = \"${variantName.get()}\"")
            appendLine("}")
        }
        outputFile.get().asFile.writeText(content)
    }
}

class BuildMetadataPlugin : Plugin<Project> {
    override fun apply(target: Project) {
        with(target) {
            val gitSha = providers.exec {
                commandLine("git", "rev-parse", "--short", "HEAD")
            }.standardOutput.asText.map { it.trim() }

            val timestamp = providers.provider {
                java.time.Instant.now().toString()
            }

            // Use the Variant API — NOT afterEvaluate
            plugins.withId("com.android.application") {
                val androidComponents = extensions
                    .getByType(com.android.build.api.variant.ApplicationAndroidComponentsExtension::class.java)

                androidComponents.onVariants { variant ->
                    val taskName = "generate${variant.name.replaceFirstChar { it.uppercase() }}BuildMetadata"
                    val task = tasks.register(taskName, GenerateBuildMetadataTask::class.java) {
                        this.gitSha.set(gitSha)
                        this.buildTimestamp.set(timestamp)
                        this.variantName.set(variant.name)
                        this.outputFile.set(
                            layout.buildDirectory.file(
                                "generated/buildmeta/${variant.name}/BuildMetadata.kt"
                            )
                        )
                    }

                    variant.sources.java?.addGeneratedSourceDirectory(
                        task, GenerateBuildMetadataTask::outputFile
                    )
                }
            }
        }
    }
}

9. Version Catalog Integration

Convention plugins can't use the libs.xxx accessor syntax available in regular build scripts. The libs accessor is generated code that exists in each project's build script scope — it's not available inside plugin classes because plugins are compiled separately.

Accessing the Catalog from Plugin Code

import org.gradle.api.artifacts.VersionCatalogsExtension
import org.gradle.kotlin.dsl.getByType

class AndroidComposeConventionPlugin : Plugin<Project> {
    override fun apply(target: Project) {
        with(target) {
            val libs = extensions.getByType<VersionCatalogsExtension>().named("libs")

            dependencies {
                // findLibrary → Optional<Provider<MinimalExternalModuleDependency>>
                add("implementation", libs.findLibrary("compose.ui").get())
                add("implementation", libs.findLibrary("compose.material3").get())

                // findVersion → Optional<VersionConstraint>
                val compileSdk = libs.findVersion("compileSdk").get().requiredVersion.toInt()

                // findBundle → Optional<Provider<ExternalModuleDependencyBundle>>
                add("implementation", libs.findBundle("compose").get())
            }
        }
    }
}

Helper Extension for Cleaner Access

// build-logic/convention/src/main/kotlin/CatalogExtensions.kt
import org.gradle.api.Project
import org.gradle.api.artifacts.MinimalExternalModuleDependency
import org.gradle.api.artifacts.VersionCatalog
import org.gradle.api.artifacts.VersionCatalogsExtension
import org.gradle.api.provider.Provider
import org.gradle.kotlin.dsl.getByType

internal val Project.libs: VersionCatalog
    get() = extensions.getByType<VersionCatalogsExtension>().named("libs")

internal fun VersionCatalog.library(alias: String): Provider<MinimalExternalModuleDependency> =
    findLibrary(alias).orElseThrow {
        NoSuchElementException("Library '$alias' not found in version catalog")
    }

internal fun VersionCatalog.version(alias: String): String =
    findVersion(alias).orElseThrow {
        NoSuchElementException("Version '$alias' not found in version catalog")
    }.requiredVersion

// Usage in plugins becomes clean:
// dependencies { add("implementation", libs.library("compose.ui")) }
Staff Insight — Single Source of Truth
Combining convention plugins with version catalogs gives you a single source of truth for both build configuration and dependency versions. The version catalog owns all version numbers and dependency coordinates. Convention plugins own all build configuration logic. Module build.gradle.kts files contain only the module's namespace and its unique dependencies. This separation means a new module requires exactly 3 things: a build.gradle.kts with the right plugin ID, a namespace, and an entry in settings.gradle.kts. Everything else is inherited.

10. Staff-Level Corner Cases

Convention Plugin Classpath Isolation

Plugins in build-logic/ have their own classpath, completely separate from the main project. If your convention plugin needs to configure AGP, it must declare compileOnly(libs.android.gradlePlugin) in its own dependencies. Forgetting this produces a cryptic ClassNotFoundException at configuration time — the error message won't mention build-logic.

// build-logic/convention/build.gradle.kts
dependencies {
    // These are the plugin's OWN dependencies — not the app's
    compileOnly(libs.android.gradlePlugin)   // For ApplicationExtension, LibraryExtension
    compileOnly(libs.kotlin.gradlePlugin)    // For KotlinCompile task type
    compileOnly(libs.compose.gradlePlugin)   // For ComposeCompilerGradlePluginExtension
    compileOnly(libs.ksp.gradlePlugin)       // For KspExtension

    // Version catalog entries:
    // [libraries]
    // android-gradlePlugin = { group = "com.android.tools.build", name = "gradle", version.ref = "agp" }
}

Plugin Ordering Matters

// WRONG — Hilt plugin applied before Android plugin
plugins {
    id("dagger.hilt.android.plugin")  // Fails: no Android extension yet
    id("com.android.library")
}

// CORRECT — Android plugin first, then Hilt
plugins {
    id("com.android.library")
    id("dagger.hilt.android.plugin")
}

// In convention plugins, control ordering explicitly:
class AndroidHiltConventionPlugin : Plugin<Project> {
    override fun apply(target: Project) {
        with(target) {
            pluginManager.apply("dagger.hilt.android.plugin")
            pluginManager.apply("com.google.devtools.ksp")

            dependencies {
                add("implementation", libs.library("hilt.android"))
                add("ksp", libs.library("hilt.compiler"))
            }
        }
    }
}
// Safe because AndroidFeatureConventionPlugin applies myapp.android.library FIRST,
// which applies com.android.library — so by the time Hilt runs, the Android extension exists.
Staff Insight — The Transitive Plugin Ordering Trap
When convention plugins compose each other (feature = library + compose + hilt), the pluginManager.apply() calls execute in declaration order. If plugin B depends on extensions created by plugin A, A must be applied first. This is straightforward in a single plugin, but becomes subtle when convention plugins compose each other — you must reason about the transitive application order. The safe pattern: always apply the Android plugin first, then language plugins (Kotlin), then annotation processors (KSP), then framework plugins (Hilt, Room), then feature-level plugins.

buildSrc + build-logic Coexistence

// Possible but not recommended during migration:
// - buildSrc still compiles first and invalidates everything on change
// - build-logic plugins are isolated from buildSrc changes
// - Migrate incrementally: move one plugin at a time from buildSrc → build-logic
// - Once buildSrc is empty, delete it

// Migration checklist:
// 1. Create build-logic/ structure
// 2. Move one plugin from buildSrc to build-logic
// 3. Update modules to use the new plugin ID
// 4. Verify build cache behavior (build-logic changes shouldn't invalidate all modules)
// 5. Repeat until buildSrc is empty
// 6. Delete buildSrc/

Composite Build Caching Advantage

// buildSrc change → ALL modules recompile (classpath changed)
// build-logic change → only build-logic recompiles
//   → main project re-evaluates build scripts (fast)
//   → task outputs remain cached if inputs didn't change

// This is the key architectural difference:
// buildSrc is a classpath dependency of every build script
// build-logic is a separate build whose outputs (plugin JARs) are inputs to the main build
// Gradle tracks inputs at the task level, so unchanged task inputs = cache hit

Settings Plugins

// Some plugins MUST be applied in settings.gradle.kts, not build.gradle.kts:
// - Gradle Enterprise / Develocity (build scan + cache)

// settings.gradle.kts

Non-Transitive R Classes

// gradle.properties — default in AGP 8.0+ for new projects
android.nonTransitiveRClass=true

// With transitive R (legacy):
//   :feature:home's R class contains ALL resources from ALL dependencies
//   Changing a resource in :core:ui recompiles R in every dependent module

// With non-transitive R:
//   :feature:home's R class contains ONLY its own resources
//   Must use fully qualified R references for dependency resources:
//     com.myapp.core.ui.R.drawable.ic_logo  (not just R.drawable.ic_logo)
//   Dramatically reduces R class size and incremental compilation scope

Build Features — Disable What You Don't Use

// In a convention plugin, disable unused features by default:
internal fun Project.configureAndroidDefaults(
    commonExtension: CommonExtension<*, *, *, *, *, *>,
) {
    commonExtension.buildFeatures {
        viewBinding = false
        compose = false
        buildConfig = false
        aidl = false
        renderScript = false
        resValues = false
        shaders = false
    }
}
// Modules opt IN to features they need (e.g., Compose convention plugin enables compose = true)
// Each enabled feature adds tasks to configuration and execution phases
// In a 50-module project, disabling unused features saves measurable configuration time
