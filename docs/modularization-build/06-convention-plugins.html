Convention Plugins, Build Variants & Shared Build Logic

Eliminating duplicated build configuration across modules with convention plugins, composite builds, version catalog integration, AGP variant API, custom tasks, and the staff-level patterns used in production-scale Android projects

1. The Problem: Duplicated Build Logic

In a modular Android project, every module's build.gradle.kts repeats the same boilerplate: compileSdk, minSdk, Kotlin compiler options, testing dependencies, Compose compiler configuration, R8 rules. Copy-pasting this across 50+ modules creates a maintenance nightmare.

:feature:home/build.gradle.kts        :feature:search/build.gradle.kts
┌──────────────────────────────┐      ┌──────────────────────────────┐
│ android {                    │      │ android {                    │
│   compileSdk = 35            │      │   compileSdk = 35            │
│   minSdk = 26                │      │   minSdk = 26                │
│   kotlinOptions {            │      │   kotlinOptions {            │
│     jvmTarget = "17"         │      │     jvmTarget = "17"         │
│   }                          │      │   }                          │
│   buildFeatures {            │      │   buildFeatures {            │
│     compose = true           │      │     compose = true           │
│   }                          │      │   }                          │
│ }                            │      │ }                            │
│ dependencies {               │      │ dependencies {               │
│   // same testing deps...    │      │   // same testing deps...    │
│ }                            │      │ }                            │
└──────────────────────────────┘      └──────────────────────────────┘
          × 50 modules = 50 places to update when compileSdk changes

The consequences are real:

  • Inconsistency — one module on minSdk 24, another on 26, a third on 21. Nobody notices until a runtime crash.
  • Upgrade friction — bumping compileSdk from 34 to 35 requires touching 50 files. Reviewers stop reading after the third file.
  • Drift — new modules get slightly different configurations because the developer copied from a stale template.

Three solutions exist, in order of increasing sophistication: buildSrc, precompiled script plugins, and convention plugins in a composite build.

2. buildSrc — The Legacy Approach

Gradle automatically compiles anything in the buildSrc/ directory at the project root and adds it to the build classpath. This lets you define shared constants, custom tasks, and even plugins.

// buildSrc/src/main/kotlin/Versions.kt
object Versions {
    const val compileSdk = 35
    const val minSdk = 26
    const val targetSdk = 35
    const val kotlin = "2.1.0"
    const val compose = "1.7.6"
}

// Any module's build.gradle.kts can reference:
android {
    compileSdk = Versions.compileSdk
    defaultConfig {
        minSdk = Versions.minSdk
    }
}

Why buildSrc Is an Anti-Pattern for Large Projects

Critical Problem

Any change to any file in buildSrc/ invalidates the entire build. Every task in every module is marked as out-of-date and re-executes. Change a version constant? Full rebuild. Fix a typo in a comment? Full rebuild. This is because Gradle treats buildSrc as part of the build logic classpath — if the classpath changes, all tasks that depend on it (which is all of them) must re-run.

Additional problems:

  • buildSrc is compiled before any project configuration — it adds a fixed cost to every build invocation, including ./gradlew help
  • No isolation — buildSrc classes are visible to all build scripts, creating implicit coupling
  • Breaks configuration cache in many scenarios because the entire build classpath changes
  • Cannot be shared across multiple projects (it's embedded in one repository)
L6 Insight — buildSrc Is a Migration Target

If you inherit a project using buildSrc, migrating to a composite build (build-logic/) is one of the highest-ROI build improvements you can make. The migration is mechanical: move buildSrc/src/ into build-logic/convention/src/, add a settings.gradle.kts and build.gradle.kts to build-logic/, and include it via pluginManagement { includeBuild("build-logic") }. The payoff is immediate: changes to build logic only invalidate modules that use the changed plugin, not the entire build. On a 50-module project, this can turn a 5-minute full rebuild into a 30-second incremental build.

3. Convention Plugins — The Modern Approach

Convention plugins live in a separate Gradle project (typically build-logic/) that's included as a composite build. Each plugin encapsulates a specific build configuration pattern — "Android library module", "Compose feature module", "Kotlin-only module" — and modules opt in by applying the plugin.

Project Structure

project-root/
├── build-logic/
│   ├── settings.gradle.kts          ← standalone Gradle project
│   └── convention/
│       ├── build.gradle.kts         ← plugin project (depends on AGP, Kotlin plugin)
│       └── src/main/kotlin/
│           ├── AndroidLibraryConventionPlugin.kt
│           ├── AndroidApplicationConventionPlugin.kt
│           ├── AndroidComposeConventionPlugin.kt
│           ├── AndroidFeatureConventionPlugin.kt
│           ├── JvmLibraryConventionPlugin.kt
│           └── KotlinAndroid.kt     ← shared configuration helpers
├── settings.gradle.kts              ← includes build-logic
├── app/
│   └── build.gradle.kts             ← plugins { id("myapp.android.application") }
├── feature/home/
│   └── build.gradle.kts             ← plugins { id("myapp.android.feature") }
└── core/domain/
    └── build.gradle.kts             ← plugins { id("myapp.jvm.library") }

Wiring the Composite Build

// build-logic/settings.gradle.kts
dependencyResolutionManagement {
    repositories {
        google()
        mavenCentral()
    }
    versionCatalogs {
        create("libs") {
            from(files("../gradle/libs.versions.toml"))
        }
    }
}

rootProject.name = "build-logic"
include(":convention")

// Root settings.gradle.kts
pluginManagement {
    includeBuild("build-logic")
    repositories {
        google()
        mavenCentral()
        gradlePluginPortal()
    }
}

The key difference from buildSrc: changes to build-logic/ only invalidate modules that apply the changed plugin. If you modify the Compose convention plugin, only Compose modules rebuild — pure Kotlin modules are unaffected.

Shared Kotlin Configuration Helper

// build-logic/convention/src/main/kotlin/KotlinAndroid.kt
import com.android.build.api.dsl.CommonExtension
import org.gradle.api.Project
import org.gradle.api.plugins.JavaPluginExtension
import org.gradle.kotlin.dsl.configure
import org.gradle.kotlin.dsl.dependencies
import org.jetbrains.kotlin.gradle.dsl.JvmTarget
import org.jetbrains.kotlin.gradle.dsl.KotlinAndroidProjectExtension

internal fun Project.configureKotlinAndroid(
    commonExtension: CommonExtension<*, *, *, *, *, *>,
) {
    val libs = versionCatalogs.named("libs")

    commonExtension.apply {
        compileSdk = libs.findVersion("compileSdk").get().toString().toInt()

        defaultConfig {
            minSdk = libs.findVersion("minSdk").get().toString().toInt()
        }

        compileOptions {
            sourceCompatibility = JavaVersion.VERSION_17
            targetCompatibility = JavaVersion.VERSION_17
        }

        testOptions {
            unitTests.isIncludeAndroidResources = true
        }
    }

    configure<KotlinAndroidProjectExtension> {
        compilerOptions {
            jvmTarget.set(JvmTarget.JVM_17)
            freeCompilerArgs.addAll(
                "-opt-in=kotlinx.coroutines.ExperimentalCoroutinesApi",
            )
        }
    }
}

internal fun Project.configureKotlinJvm() {
    configure<JavaPluginExtension> {
        sourceCompatibility = JavaVersion.VERSION_17
        targetCompatibility = JavaVersion.VERSION_17
    }
}

Android Library Convention Plugin

// build-logic/convention/src/main/kotlin/AndroidLibraryConventionPlugin.kt
import com.android.build.gradle.LibraryExtension
import org.gradle.api.Plugin
import org.gradle.api.Project
import org.gradle.kotlin.dsl.configure
import org.gradle.kotlin.dsl.dependencies

class AndroidLibraryConventionPlugin : Plugin<Project> {
    override fun apply(target: Project) = with(target) {
        val libs = versionCatalogs.named("libs")

        pluginManager.apply("com.android.library")
        pluginManager.apply("org.jetbrains.kotlin.android")

        configure<LibraryExtension> {
            configureKotlinAndroid(this)

            defaultConfig {
                testInstrumentationRunner = "androidx.test.runner.AndroidJUnitRunner"
                consumerProguardFiles("consumer-rules.pro")
            }
        }

        dependencies {
            add("testImplementation", libs.findLibrary("junit").get())
            add("testImplementation", libs.findLibrary("truth").get())
            add("androidTestImplementation", libs.findLibrary("androidx-test-ext").get())
        }
    }
}

5. Precompiled Script Plugins

An alternative to writing Plugin<Project> classes: place .gradle.kts files directly in build-logic/convention/src/main/kotlin/. The filename becomes the plugin ID.

// build-logic/convention/src/main/kotlin/myapp.android.library.gradle.kts
plugins {
    id("com.android.library")
    id("org.jetbrains.kotlin.android")
}

android {
    compileSdk = 35

    defaultConfig {
        minSdk = 26
        testInstrumentationRunner = "androidx.test.runner.AndroidJUnitRunner"
        consumerProguardFiles("consumer-rules.pro")
    }

    compileOptions {
        sourceCompatibility = JavaVersion.VERSION_17
        targetCompatibility = JavaVersion.VERSION_17
    }
}

kotlin {
    compilerOptions {
        jvmTarget.set(org.jetbrains.kotlin.gradle.dsl.JvmTarget.JVM_17)
    }
}

dependencies {
    "testImplementation"("junit:junit:4.13.2")
}

// Applied in modules the same way:
// plugins { id("myapp.android.library") }

Class Plugins vs Precompiled Script Plugins

Aspect Class Plugin (Plugin<Project>) Precompiled Script (.gradle.kts)
Syntax Standard Kotlin class Build script DSL (no class boilerplate)
Testability Testable with GradleRunner Harder to unit test
Composition Can call shared helper functions Limited — can apply other plugins but can't share code easily
IDE support Full autocomplete and navigation Autocomplete can be unreliable
Version catalog access Via versionCatalogs.named("libs") Direct libs.xyz accessor (auto-generated)
Speed to write More boilerplate Faster for simple configurations
Recommendation

Use class plugins for anything non-trivial. Precompiled scripts are fine for simple, stable configurations (e.g., a Detekt or Spotless setup), but class plugins are easier to test, compose, and refactor. Google's Now in Android project uses class plugins exclusively.

Accessing the Catalog from a Convention Plugin

// Inside a Plugin<Project> class
class AndroidComposeConventionPlugin : Plugin<Project> {
    override fun apply(target: Project) = with(target) {
        val libs = versionCatalogs.named("libs")

        pluginManager.apply("org.jetbrains.kotlin.plugin.compose")

        configure<LibraryExtension> {
            buildFeatures {
                compose = true
            }
        }

        dependencies {
            val bom = libs.findLibrary("compose-bom").get()
            add("implementation", platform(bom))
            add("implementation", libs.findLibrary("compose-ui").get())
            add("implementation", libs.findLibrary("compose-material3").get())
            add("implementation", libs.findLibrary("compose-ui-tooling-preview").get())
            add("debugImplementation", libs.findLibrary("compose-ui-tooling").get())
        }
    }
}

// Extension property for cleaner access (define once, use everywhere)
internal val Project.versionCatalogs
    get() = extensions.getByType<VersionCatalogsExtension>()
Gotcha — Catalog Sharing

The build-logic/ project is a separate Gradle build — it doesn't automatically see the root project's version catalog. You must explicitly share it via the versionCatalogs { create("libs") { from(files("../gradle/libs.versions.toml")) } } block in build-logic/settings.gradle.kts. Forgetting this results in "catalog 'libs' not found" errors.

7. AGP Extension API — Variant Manipulation

The Android Gradle Plugin exposes a type-safe androidComponents extension for manipulating build variants. This is the modern replacement for the deprecated variantFilter API.

Lifecycle: beforeVariants vs onVariants

AGP Variant API Lifecycle:

  ┌─────────────────────────────────┐
  │  finalizeDsl { }                │  ← Last chance to modify the DSL
  │  (android { } block is locked   │     (compileSdk, buildTypes, etc.)
  │   after this point)             │
  └───────────────┬─────────────────┘
                  │
  ┌───────────────▼─────────────────┐
  │  beforeVariants { builder -> }  │  ← Modify/disable variants BEFORE creation
  │  • builder.enable = false       │     (variant objects don't exist yet)
  │  • builder.minSdk = ...         │
  └───────────────┬─────────────────┘
                  │
  ┌───────────────▼─────────────────┐
  │  onVariants { variant -> }      │  ← Read/modify finalized variants
  │  • variant.buildConfigFields    │     (variant objects are created)
  │  • variant.artifacts.use(...)   │
  │  • variant.packaging            │
  └─────────────────────────────────┘

Disabling Unused Variants

// In a convention plugin — disable variants that no module needs
androidComponents {
    beforeVariants(selector().withBuildType("staging")) { variantBuilder ->
        variantBuilder.enable = false
    }

    // Disable specific flavor + build type combinations
    beforeVariants(
        selector()
            .withFlavor("environment" to "qa")
            .withBuildType("release")
    ) { variantBuilder ->
        variantBuilder.enable = false
    }
}

Adding BuildConfig Fields Per Variant

androidComponents {
    onVariants { variant ->
        variant.buildConfigFields.put(
            "API_BASE_URL",
            com.android.build.api.variant.BuildConfigField(
                type = "String",
                value = when {
                    variant.name.contains("staging") -> "\"https://staging.api.example.com\""
                    variant.name.contains("production") -> "\"https://api.example.com\""
                    else -> "\"https://dev.api.example.com\""
                },
                comment = "Base URL for API calls"
            )
        )
    }
}

Artifact Transformation

// Transform the merged manifest — e.g., inject a build timestamp
abstract class ManifestTransformTask : DefaultTask() {
    @get:InputFile
    abstract val mergedManifest: RegularFileProperty

    @get:OutputFile
    abstract val updatedManifest: RegularFileProperty

    @TaskAction
    fun transform() {
        val content = mergedManifest.get().asFile.readText()
        val updated = content.replace(
            "</application>",
            """    <meta-data android:name="build.commit" android:value="${gitCommit()}" />
    </application>"""
        )
        updatedManifest.get().asFile.writeText(updated)
    }
}

androidComponents {
    onVariants { variant ->
        val manifestTask = tasks.register<ManifestTransformTask>(
            "${variant.name}ManifestTransform"
        )
        variant.artifacts
            .use(manifestTask)
            .wiredWithFiles(
                ManifestTransformTask::mergedManifest,
                ManifestTransformTask::updatedManifest
            )
            .toTransform(com.android.build.api.artifact.SingleArtifact.MERGED_MANIFEST)
    }
}
L6 Insight — Variant Explosion Is a Hidden Build Killer

Every variant multiplies the number of AGP tasks. A project with 2 build types × 3 flavors = 6 variants = roughly 6× the tasks AGP registers during configuration. With 50 modules, that's 300 variant configurations. If you only ever build debug and release, those extra 4 variants per module are pure waste. Aggressively disable unused variants in your convention plugins — it's one of the simplest ways to cut configuration time. The Now in Android project disables all non-essential variants in its convention plugins for exactly this reason.

8. Build Variants, Flavors, and Dimensions

Core Concepts

Concept Purpose Example
Build type Debuggability, minification, signing debug, release
Product flavor App variations with different features/config free/paid, staging/production
Flavor dimension Groups flavors into orthogonal axes tier (free/paid), environment (staging/prod)
Variant Cartesian product of build type × all dimensions freeProductionDebug, paidStagingRelease 
android {
    flavorDimensions += listOf("tier", "environment")

    productFlavors {
        create("free") {
            dimension = "tier"
            applicationIdSuffix = ".free"
        }
        create("paid") {
            dimension = "tier"
        }
        create("staging") {
            dimension = "environment"
            applicationIdSuffix = ".staging"
        }
        create("production") {
            dimension = "environment"
        }
    }

    buildTypes {
        debug { /* ... */ }
        release { isMinifyEnabled = true }
    }
}

// Resulting variants (2 tiers × 2 environments × 2 build types = 8):
// freeStagingDebug, freeStagingRelease,
// freeProductionDebug, freeProductionRelease,
// paidStagingDebug, paidStagingRelease,
// paidProductionDebug, paidProductionRelease

Source Sets Per Variant

src/
├── main/              ← all variants
├── debug/             ← debug build type only
├── release/           ← release build type only
├── free/              ← free flavor only
├── paid/              ← paid flavor only
├── staging/           ← staging flavor only
├── production/        ← production flavor only
├── freeDebug/         ← free + debug combination
├── paidProduction/    ← paid + production combination
└── paidProductionRelease/  ← paid + production + release (most specific)

Merge order (most specific wins):
  variant > buildType > flavor > main

Per-Variant Configuration Values

android {
    productFlavors {
        create("staging") {
            dimension = "environment"
            buildConfigField("String", "API_URL", "\"https://staging.api.example.com\"")
            resValue("string", "app_name", "MyApp Staging")
        }
        create("production") {
            dimension = "environment"
            buildConfigField("String", "API_URL", "\"https://api.example.com\"")
            resValue("string", "app_name", "MyApp")
        }
    }
}
Variant Matching in Library Modules

Library modules don't need to declare the same flavors as the app module. AGP uses variant-aware dependency resolution: it matches the app's variant to the library's best-matching variant based on build type and flavor dimensions. If the library doesn't have a matching flavor, AGP falls back to the default. Use matchingFallbacks to control this: create("staging") { matchingFallbacks += "debug" }.

9. Custom Gradle Tasks

Lazy Task Registration

// Always use register (lazy) instead of create (eager)
tasks.register<GenerateBuildInfoTask>("generateBuildInfo") {
    gitCommit.set(provider { execGitCommand("rev-parse", "--short", "HEAD") })
    buildTimestamp.set(provider { java.time.Instant.now().toString() })
    outputFile.set(layout.buildDirectory.file("generated/build-info.json"))
}

Custom Task with Proper Inputs/Outputs

@CacheableTask
abstract class GenerateBuildInfoTask : DefaultTask() {

    @get:Input
    abstract val gitCommit: Property<String>

    @get:Input
    abstract val buildTimestamp: Property<String>

    @get:OutputFile
    abstract val outputFile: RegularFileProperty

    @TaskAction
    fun generate() {
        val info = buildString {
            appendLine("{")
            appendLine("""  "commit": "${gitCommit.get()}",""")
            appendLine("""  "timestamp": "${buildTimestamp.get()}",""")
            appendLine("""  "builtBy": "Gradle ${GradleVersion.current().version}" """)
            appendLine("}")
        }
        outputFile.get().asFile.writeText(info)
    }
}

// Helper to run git commands
fun Project.execGitCommand(vararg args: String): String {
    val output = java.io.ByteArrayOutputStream()
    exec {
        commandLine("git", *args)
        standardOutput = output
    }
    return output.toString().trim()
}

Task Input/Output Annotations

Annotation Type Purpose
@Input Scalar (String, Int, Boolean) Simple value that affects output
@InputFile RegularFileProperty Single file input
@InputFiles ConfigurableFileCollection Multiple file inputs
@InputDirectory DirectoryProperty Directory input (all files within)
@OutputFile RegularFileProperty Single file output
@OutputDirectory DirectoryProperty Directory output
@PathSensitive Modifier on file inputs Controls whether absolute/relative path matters for cache key
@CacheableTask Class annotation Opts task into build cache (not just UP-TO-DATE)
Configuration Avoidance

Use Property<T> and Provider<T> for all task inputs — never raw values. Properties are resolved lazily at execution time, not configuration time. This means the provider { execGitCommand(...) } call above only runs when the task actually executes, not when Gradle configures the build. For a 300-module project, this avoidance of unnecessary work during configuration is critical.

10. Staff-Level Corner Cases

Plugin Classpath Isolation

Each plugin in a composite build runs in its own classloader. This means two convention plugins can depend on different versions of the same library without conflict — Gradle isolates them. However, this also means you can't share runtime objects between plugins; they communicate only through Gradle's extension and property mechanisms.

// build-logic/convention/build.gradle.kts
dependencies {
    compileOnly(libs.android.gradlePlugin)  // AGP 8.7.3
}

// build-logic/analysis/build.gradle.kts
dependencies {
    compileOnly(libs.android.gradlePlugin)  // Could be a different version
}
// No conflict — each plugin project has its own classloader

Avoid afterEvaluate in Plugins

Anti-Pattern

afterEvaluate { } in convention plugins is a code smell. It runs after the project's build.gradle.kts is fully evaluated, which means: (1) it breaks configuration cache because it creates ordering dependencies between projects, (2) it makes the build non-deterministic — the behavior depends on plugin application order, (3) it prevents Gradle from optimizing configuration. Use the AGP Variant API (androidComponents { onVariants { } }) instead — it provides proper lifecycle hooks that work with configuration cache.

// BAD — breaks configuration cache, order-dependent
class BadPlugin : Plugin<Project> {
    override fun apply(target: Project) = with(target) {
        afterEvaluate {
            val android = extensions.getByType<LibraryExtension>()
            android.buildTypes.forEach { buildType ->
                tasks.register("custom${buildType.name.capitalize()}Task") { /* ... */ }
            }
        }
    }
}

// GOOD — uses AGP's lifecycle API
class GoodPlugin : Plugin<Project> {
    override fun apply(target: Project) = with(target) {
        pluginManager.withPlugin("com.android.library") {
            configure<LibraryExtension> {
                // Configure DSL directly — no afterEvaluate needed
            }
        }
        extensions.getByType<AndroidComponentsExtension<*, *, *>>().apply {
            onVariants { variant ->
                tasks.register("custom${variant.name}Task") { /* ... */ }
            }
        }
    }
}

Plugin Marker Artifacts — How Plugin IDs Resolve

When you write plugins { id("myapp.android.library") }, Gradle resolves the ID to a plugin class through plugin marker artifacts. For convention plugins in a composite build, the gradlePlugin { plugins { register(...) } } block generates these markers automatically.

Resolution: plugins { id("myapp.android.library") }

  1. Gradle looks for a plugin marker artifact:
     group = "myapp.android.library"
     name  = "myapp.android.library.gradle.plugin"

  2. For composite builds, Gradle checks included builds first:
     build-logic/convention registered "myapp.android.library"
     → maps to AndroidLibraryConventionPlugin class

  3. For external plugins, Gradle checks pluginManagement repositories:
     gradlePluginPortal(), google(), mavenCentral()
     → downloads the marker POM → resolves the implementation JAR

  4. Plugin class is instantiated and apply(project) is called

Composing Convention Plugins

Convention plugins can apply other convention plugins, creating a layered configuration system:

// AndroidFeatureConventionPlugin — composes library + compose + hilt
class AndroidFeatureConventionPlugin : Plugin<Project> {
    override fun apply(target: Project) = with(target) {
        pluginManager.apply("myapp.android.library")   // base Android config
        pluginManager.apply("myapp.android.compose")    // Compose setup
        pluginManager.apply("myapp.android.hilt")       // Hilt + KSP

        val libs = versionCatalogs.named("libs")

        dependencies {
            add("implementation", project(":core:ui"))
            add("implementation", project(":core:domain"))
            add("implementation", libs.findLibrary("lifecycle-viewmodel-compose").get())
            add("implementation", libs.findLibrary("hilt-navigation-compose").get())
        }
    }
}

L6 Insight — Convention Plugins Are Architecture Enforcement

Convention plugins aren't just about reducing boilerplate — they're an architectural enforcement mechanism. When a module applies id("myapp.android.feature"), it gets a specific, opinionated configuration: Compose enabled, Hilt wired, standard dependencies included, unused variants disabled. Developers can't accidentally create a feature module without Hilt, or with a different minSdk, or with Compose disabled. The plugin is the architecture. This is why Google's Now in Android project uses convention plugins — not to save keystrokes, but to make the wrong thing hard to do. In an interview, frame convention plugins as an architecture tool, not a build tool.

Composite Build vs buildSrc — Decision Matrix

Criterion buildSrc Composite Build (build-logic/)
Cache invalidation Any change → full rebuild Only affected modules rebuild
Configuration cache Often incompatible Fully compatible
Compilation overhead Compiled on every build Compiled only when changed
Reusability Single project only Can be shared across projects via Git submodule or published artifact
IDE support Good Good (since Gradle 7+)
Setup complexity Zero (just create the directory) Requires settings.gradle.kts + build.gradle.kts
Recommendation Small projects (<5 modules) Always for 10+ modules
L6 Insight — Convention Plugin Testing Is a Signal

In an interview, mentioning that you test convention plugins with GradleRunner is a strong staff-level signal. Most teams treat build logic as "infrastructure that just works" and never test it. But convention plugins are code that affects every module in the project — a bug in a convention plugin breaks the entire build. Testing that the plugin applies the correct SDK versions, enables the right features, and registers the expected tasks catches regressions before they reach 50 modules. It's the same principle as testing any other shared library.
