02 — Module Architecture

Module types, dependency rules, api vs implementation, package-by-feature, Google's Now in Android reference architecture, granularity trade-offs, encapsulation patterns, and staff-level corner cases

1. Why Modularize?

A monolithic Android app — one :app module containing everything — works fine for a hackathon. It does not work for a team of 20 engineers shipping weekly. Modularization is the structural investment that converts a codebase from "one person can hold it in their head" to "fifty people can work on it simultaneously without stepping on each other."

The Six Motivations

  • Build speed — Gradle can compile independent modules in parallel and skip unchanged ones entirely. A change in :feature:search doesn't recompile :feature:profile.
  • Team scalability — Each team owns a set of modules. Merge conflicts drop because teams rarely touch the same files. Code review scope shrinks to the module boundary.
  • Code isolation — Module boundaries enforce architectural rules at compile time. A feature module literally cannot import another feature's internal class if the dependency isn't declared.
  • Reusability — A :core:designsystem module can be shared across multiple apps (main app, Wear OS companion, Automotive variant) without copy-pasting.
  • Dynamic delivery — Play Feature Delivery lets you ship feature modules on demand, reducing initial install size. Only possible if the feature is a separate module.
  • Testability — Modules can be tested in isolation with fast unit tests. No need to build the entire app to test a single repository.

Monolith vs Modular — Quantified

DimensionMonolith (:app only)Modular (30+ modules)
Clean build ~5 min (single compilation unit) ~5 min (similar — all modules compile)
Incremental build (1 file changed) ~60–90s (recompiles entire app) ~10–20s (recompiles only affected module + link)
Parallel compilation 1 task (no parallelism) N independent tasks (saturates CPU cores)
Team velocity (10+ engineers) Frequent merge conflicts, long CI Independent feature lanes, fast CI per module
Code coupling Any class can import any other class Compile-time enforcement of boundaries
Test execution Run all tests (slow) Run only affected module's tests (fast)
APK size (with dynamic delivery) Everything bundled On-demand features reduce initial install
Staff Insight — The Real Reason to Modularize

Build speed gets all the attention, but the most impactful benefit at scale is enforced isolation. In a monolith, "don't depend on that package" is a code review comment. In a modular project, it's a compiler error. This shifts architectural enforcement from human discipline to tooling — and tooling doesn't get tired on Friday afternoons. At Google scale, this is non-negotiable: teams must be able to change their module's internals without breaking other teams, and the build system must guarantee it.

3. Dependency Rules — The Dependency Inversion Principle

The module graph above isn't arbitrary — it's a direct application of the Dependency Inversion Principle (the "D" in SOLID). High-level modules (features, domain) don't depend on low-level modules (network, database). Both depend on abstractions (interfaces in :core:domain).

The Five Rules

  1. Horizontal isolation — Feature modules NEVER depend on other feature modules. :feature:home cannot import anything from :feature:search. If they need to communicate, they go through a shared :core:* module or navigation.
  2. Vertical dependency — Feature modules depend downward on :core:* modules. Never upward.
  3. Domain purity — :core:domain depends ONLY on :core:model. No Android framework, no Retrofit, no Room. It's a pure Kotlin/JVM module (id("java-library") + id("org.jetbrains.kotlin.jvm")).
  4. Dependency inversion — :core:domain defines repository interfaces. :core:data implements them. The feature module depends on the interface (via :core:domain), not the implementation.
  5. Composition root — :app is the only module that sees everything. It wires interface → implementation bindings via Hilt modules. No other module should have this god-view.

Dependency Inversion in Practice

// :core:domain — defines the contract (pure Kotlin, no Android)
// build.gradle.kts: plugins { id("java-library"); kotlin("jvm") }

interface ArticleRepository {
    fun getArticles(): Flow<List<Article>>
    suspend fun refreshArticles()
    suspend fun toggleBookmark(articleId: String)
}

class GetBookmarkedArticlesUseCase(
    private val articleRepo: ArticleRepository
) {
    operator fun invoke(): Flow<List<Article>> =
        articleRepo.getArticles()
            .map { articles -> articles.filter { it.isBookmarked } }
}
// :core:data — implements the contract
// build.gradle.kts: dependencies { implementation(project(":core:domain")) }

class ArticleRepositoryImpl(
    private val api: ArticleApiService,
    private val dao: ArticleDao,
    private val dispatcher: CoroutineDispatcher
) : ArticleRepository {

    override fun getArticles(): Flow<List<Article>> =
        dao.observeAll().map { entities -> entities.map { it.toDomain() } }

    override suspend fun refreshArticles() = withContext(dispatcher) {
        val dtos = api.fetchArticles()
        dao.upsertAll(dtos.map { it.toEntity() })
    }

    override suspend fun toggleBookmark(articleId: String) {
        dao.toggleBookmark(articleId)
    }
}
// :app — wires interface to implementation via Hilt
@Module
@InstallIn(SingletonComponent::class)
abstract class DataModule {
    @Binds
    abstract fun bindArticleRepository(
        impl: ArticleRepositoryImpl
    ): ArticleRepository
}
  Dependency flow (compile-time):

  :feature:home ──depends──▶ :core:domain ◀──depends── :core:data
       │                        │                          │
       │ uses                   │ defines                  │ implements
       ▼                        ▼                          ▼
  GetBookmarkedArticles    ArticleRepository         ArticleRepositoryImpl
       UseCase              (interface)               (class)

  At runtime, Hilt injects ArticleRepositoryImpl where ArticleRepository is needed.
  :feature:home never knows about :core:data — it only sees the interface.
Why This Matters

Without dependency inversion, :feature:home would depend directly on :core:data, which depends on :core:network and :core:database. A change in Retrofit's API would recompile the feature module. With inversion, the feature depends only on the stable interface in :core:domain — insulating it from infrastructure changes entirely.

4. api vs implementation — Visibility at Module Boundaries

Gradle's dependency configurations control what consumers of your module can see. Getting this wrong is the #1 cause of unnecessary recompilation in modular projects.

The Two Core Configurations

implementation (default — use this)

The dependency is available only within the declaring module. Consumers cannot see it.

If module B implementation-depends on library X, and module A depends on B — A cannot see X's classes.

Build impact: Changes to X recompile B only. A is untouched.

api (use sparingly)

The dependency is exposed transitively to consumers. If B api-depends on X, then A can see X's classes through B.

Build impact: Changes to X recompile both B and A (and anything depending on A).

When to Use api

Use api only when the dependency is part of your module's public API surface. If a type from the dependency appears in your module's public method signatures, return types, or superclass — it must be api. Otherwise, implementation. 

// :core:network/build.gradle.kts

dependencies {
    // Retrofit types appear in public API (ApiService interfaces return Call<T>)
    api(libs.retrofit)

    // OkHttp is used internally for interceptors — never exposed
    implementation(libs.okhttp)

    // Moshi is used internally for serialization — consumers don't see it
    implementation(libs.moshi)
}

All Dependency Configurations

ConfigurationCompile ClasspathRuntime ClasspathTransitive to ConsumersUse Case
implementation Yes Yes No Default for all dependencies
api Yes Yes Yes (compile + runtime) Types exposed in public API
compileOnly Yes No No Annotation processors, compile-time checks (e.g., @RestrictTo)
runtimeOnly No Yes No SLF4J implementation, database drivers, Conscrypt provider
testImplementation Test only Test only No JUnit, Mockk, Turbine — test dependencies
androidTestImplementation AndroidTest only AndroidTest only No Espresso, Compose testing, Hilt testing
debugImplementation Debug variant Debug variant No LeakCanary, debug drawer, Flipper

Transitive Dependency Propagation

  Module A ──depends──▶ Module B ──depends──▶ Library X

  If B uses "implementation(X)":
    A's compile classpath:  B ✓   X ✗  ← A cannot see X
    A's runtime classpath:  B ✓   X ✓  ← X is available at runtime (transitive)

  If B uses "api(X)":
    A's compile classpath:  B ✓   X ✓  ← A can see X
    A's runtime classpath:  B ✓   X ✓  ← X is available at runtime

  Key insight: "implementation" still includes X at runtime — it only hides
  it from the compile classpath. This prevents A from accidentally using X's
  classes, and prevents recompilation of A when X changes.
Staff Insight — api Leaks Destroy Build Performance

Overusing api is the single most common build performance mistake in modular projects. Every api dependency creates a transitive compile dependency that propagates through the entire graph. In a project with 50 modules, one careless api at the bottom of the graph can cause 40+ modules to recompile when that library updates. The fix is mechanical: audit every api declaration and ask "does this type appear in my module's public method signatures?" If not, switch to implementation. Google's Now in Android project has ~90% implementation and ~10% api — that's the right ratio.

Staff Insight — Package-by-Feature Aligns with Team Topology

The hybrid approach isn't just a code organization preference — it mirrors how high-performing organizations structure teams. Conway's Law says system architecture mirrors communication structure. When each team owns a feature module, the module boundary is the team boundary. Cross-team dependencies are explicit (Gradle dependency declarations), cross-team contracts are explicit (public API of the module), and cross-team changes require explicit coordination (changing a module's public API). This is exactly how Google, Netflix, and Uber structure their Android codebases.

Convention Plugin Example

// build-logic/convention/src/main/kotlin/AndroidFeatureConventionPlugin.kt
class AndroidFeatureConventionPlugin : Plugin<Project> {
    override fun apply(target: Project) = with(target) {
        pluginManager.apply {
            apply("nowinandroid.android.library")
            apply("nowinandroid.android.hilt")
        }
        dependencies {
            add("implementation", project(":core:ui"))
            add("implementation", project(":core:designsystem"))
            add("implementation", libs.findLibrary("androidx.hilt.navigation.compose").get())
            add("implementation", libs.findLibrary("androidx.lifecycle.runtimeCompose").get())
            add("implementation", libs.findLibrary("androidx.lifecycle.viewModelCompose").get())
            add("testImplementation", project(":core:testing"))
        }
    }
}

7. Module Granularity — When to Split, When Not To

Modularization is not "more modules = better." There's a sweet spot between under-modularization (monolith) and over-modularization (100 modules with 3 files each). Finding it requires judgment.

When to Create a New Module

  • Clear boundary — The code has a well-defined responsibility that can be described in one sentence.
  • Different team — A different team will own and maintain this code.
  • Independent release cycle — The module could theoretically be versioned and released independently.
  • Reuse across apps — The module is needed by multiple applications (main app, Wear OS, Automotive).
  • Dynamic delivery — The feature should be downloadable on demand to reduce APK size.
  • Build isolation — Changes to this code shouldn't trigger recompilation of unrelated modules.

When NOT to Create a New Module

  • Tiny scope — The module would contain 1-2 files. The configuration overhead (build.gradle.kts, manifest, package) isn't worth it.
  • Unclear boundary — You can't describe the module's responsibility without saying "and also..."
  • Circular dependency — Splitting would create A → B → A cycles. This is a sign the boundary is wrong.
  • Single consumer — If only one module would ever use it, keep it as a package within that module.

Granularity Guidelines

App SizeTeam SizeRecommended ModulesRationale
Small (MVP) 1-3 engineers 3-5 (:app, :core:data, :core:domain, :core:model) Overhead of many modules outweighs benefits. Focus on shipping.
Medium 5-15 engineers 10-30 Feature modules per team, shared core modules. Build speed matters now.
Large 15-50 engineers 50-100 Fine-grained feature modules, api/impl splits, convention plugins essential.
Super-app 50+ engineers 200+ Dynamic feature modules, build caching critical, dedicated build team.

The Cost of Over-Modularization

  • Configuration overhead — Each module needs a build.gradle.kts, AndroidManifest.xml, package structure. Convention plugins mitigate this but don't eliminate it.
  • Dependency graph complexity — More modules = more edges in the graph. Harder to reason about transitive dependencies.
  • IDE performance — Android Studio's indexing and sync time scales with module count. 200+ modules can make sync take 2-3 minutes.
  • Configuration phase — Gradle evaluates every module's build.gradle.kts during configuration. More modules = longer configuration phase (mitigated by configuration cache).
Common Anti-Pattern — The 3-File Module

If you find yourself creating a module with a build.gradle.kts that's longer than the actual source code, you've over-modularized. A module with one interface, one implementation, and one DI binding should probably be a package within a larger module. The rule of thumb: if the module wouldn't have its own meaningful test suite, it's too small.

8. Module Boundaries and Encapsulation

A module's value comes from its boundary — the clear line between what's public (the API) and what's private (the implementation). Kotlin and Gradle provide several mechanisms to enforce this boundary.

internal Visibility Modifier

Kotlin's internal modifier restricts access to the declaring module (compilation unit). This is the primary tool for encapsulating implementation details within a module. 

// :core:network module

// Public — consumers can see this
interface ArticleApiService {
    suspend fun fetchArticles(): List<ArticleDto>
}

// Internal — only visible within :core:network
internal class RetrofitArticleApiService(
    private val retrofit: Retrofit
) : ArticleApiService {
    override suspend fun fetchArticles(): List<ArticleDto> =
        retrofit.create<ArticleApi>().getArticles()
}

// Internal — implementation detail
internal interface ArticleApi {
    @GET("articles")
    suspend fun getArticles(): List<ArticleDto>
}

// Internal — interceptor is an implementation detail
internal class AuthInterceptor(
    private val tokenProvider: TokenProvider
) : Interceptor {
    override fun intercept(chain: Interceptor.Chain): Response {
        val request = chain.request().newBuilder()
            .addHeader("Authorization", "Bearer ${tokenProvider.getToken()}")
            .build()
        return chain.proceed(request)
    }
}

The api/impl Module Split

For large-scale projects, the gold standard is splitting each feature into two modules: an :api module (public interfaces) and an :impl module (implementation). Consumers depend only on :api. 

  Traditional (single module):
  :feature:search ──depends──▶ :feature:home  ← tight coupling

  api/impl split:
  :feature:search ──depends──▶ :feature:home:api     ← depends on interface only
                               :feature:home:impl    ← wired by :app via DI

  Benefits:
  • :feature:search compiles against :feature:home:api (tiny, stable)
  • :feature:home:impl can change freely without recompiling :feature:search
  • Different teams can work on :api and :impl independently
  • :impl can be swapped (e.g., mock impl for testing)
// :feature:home:api/build.gradle.kts
plugins { id("nowinandroid.android.library") }
// Minimal dependencies — just :core:model

// :feature:home:api — public contract
interface HomeNavigator {
    fun navigateToArticleDetail(articleId: String)
}

data class HomeUiState(
    val articles: List<Article>,
    val isLoading: Boolean
)

// :feature:home:impl/build.gradle.kts
plugins { id("nowinandroid.android.feature") }
dependencies {
    implementation(project(":feature:home:api"))
    implementation(project(":core:data"))
}

// :feature:home:impl — implementation
internal class HomeNavigatorImpl(
    private val navController: NavController
) : HomeNavigator {
    override fun navigateToArticleDetail(articleId: String) {
        navController.navigate("article/$articleId")
    }
}

@RestrictTo — Lint-Enforced Visibility

When internal is too coarse (you want to restrict within a module) or not available (Java interop), @RestrictTo provides lint-enforced visibility restrictions. 

// Only usable within the same library group (e.g., :core:* modules)
@RestrictTo(RestrictTo.Scope.LIBRARY_GROUP)
fun dangerousInternalHelper() { /* ... */ }

// Only usable within the same library
@RestrictTo(RestrictTo.Scope.LIBRARY)
class InternalDataSource { /* ... */ }

// Only usable in tests
@RestrictTo(RestrictTo.Scope.TESTS)
class FakeArticleRepository : ArticleRepository { /* ... */ }
internal vs @RestrictTo

internal is enforced by the Kotlin compiler — it's impossible to bypass without reflection. @RestrictTo is enforced by Android Lint — it produces warnings/errors in the IDE but can be suppressed. For hard boundaries, prefer internal. For advisory boundaries (especially in library groups), use @RestrictTo.

Staff Insight — The api/impl Split is the Gold Standard

The :api/:impl split is how Google structures modules in their internal monorepo. It enables three critical workflows: (1) parallel development — teams agree on the :api contract, then implement independently; (2) build speed — consumers compile against the tiny :api module, not the large :impl; (3) testing — test doubles implement the :api interfaces, so integration tests don't need the real implementation. The overhead is real (twice as many modules), so reserve this pattern for modules with multiple consumers or modules that change frequently.

9. Staff-Level Corner Cases

Module Initialization Order

Modules are compiled in dependency order, but runtime initialization doesn't follow module boundaries — it follows Android component lifecycle. ContentProvider instances are initialized before Application.onCreate(), and libraries like Firebase and WorkManager use this trick for auto-initialization via androidx.startup. 

// Problem: Library A (in :core:analytics) auto-initializes via ContentProvider
// Library B (in :core:network) also auto-initializes via ContentProvider
// Order between them is UNDEFINED — depends on manifest merge order

// If A depends on B being initialized first → crash on cold start

// Fix: Use App Startup library to declare explicit dependencies
class AnalyticsInitializer : Initializer<Analytics> {
    override fun create(context: Context): Analytics {
        // This runs AFTER NetworkInitializer because of dependencies()
        return Analytics.init(context)
    }
    override fun dependencies(): List<Class<out Initializer<*>>> =
        listOf(NetworkInitializer::class.java)  // explicit ordering
}

Resource Conflicts Across Modules

Each Android library module generates its own R class. When two modules define a resource with the same name (e.g., R.string.app_name), the last one wins — determined by the dependency order in the consuming module's build.gradle.kts. 

// :core:ui defines R.string.error_message = "Something went wrong"
// :feature:home defines R.string.error_message = "Home error"
// :app depends on both → which one wins?

// Answer: the one declared LAST in :app's dependencies block
// This is fragile and non-obvious

// Fix 1: Prefix resources with module name
//   :core:ui → R.string.core_ui_error_message
//   :feature:home → R.string.feature_home_error_message

// Fix 2: Use resourcePrefix in build.gradle.kts (lint enforcement)
android {
    resourcePrefix = "feature_home_"  // lint warns if resources don't start with this
}

BuildConfig Conflicts

Each Android module generates its own BuildConfig class with the module's applicationId, versionCode, etc. Library modules get their package's BuildConfig, not the app's. 

// In :feature:home (com.example.feature.home)
val isDebug = BuildConfig.DEBUG  // ← This is com.example.feature.home.BuildConfig
                                  //    NOT com.example.app.BuildConfig

// BuildConfig.APPLICATION_ID in a library module = the library's package name
// BuildConfig.VERSION_NAME in a library module = the library's version (often "unspecified")

// Fix: Pass app-level config through DI or a :core:common config object
// AGP 8.0+ deprecated non-constant BuildConfig fields in library modules
AGP 8.0 Change — BuildConfig Off by Default

Starting with AGP 8.0, BuildConfig generation is disabled by default for library modules. You must explicitly opt in with android.buildFeatures.buildConfig = true. This was done to reduce build overhead — generating BuildConfig for every module adds up. The recommended alternative is to use buildConfigField only in the :app module and pass values down via DI.

Manifest Merging

Each module has its own AndroidManifest.xml. AGP merges them using a priority system: :app manifest has highest priority, then library modules in dependency order. Conflicts require explicit resolution. 

<!-- :core:network/AndroidManifest.xml -->
<uses-permission android:name="android.permission.INTERNET" />

<!-- :feature:camera/AndroidManifest.xml -->
<uses-permission android:name="android.permission.CAMERA" />

<!-- These merge cleanly into :app's manifest -->

<!-- Conflict example: two modules declare the same activity -->
<!-- :app/AndroidManifest.xml — resolve with tools:replace -->
<application
    tools:replace="android:theme"
    android:theme="@style/AppTheme">

    <!-- Remove a permission added by a library you don't want -->
    <uses-permission
        android:name="android.permission.ACCESS_FINE_LOCATION"
        tools:node="remove" />
</application>

ProGuard/R8 Rules Across Modules

Each module can include its own proguard-rules.pro (for app modules) or consumer-proguard-rules.pro (for library modules — automatically included by consumers). Missing rules in a library module cause runtime crashes that only appear in release builds. 

// :core:network/build.gradle.kts
android {
    defaultConfig {
        // These rules are automatically included when any module depends on :core:network
        consumerProguardFiles("consumer-proguard-rules.pro")
    }
}

// :core:network/consumer-proguard-rules.pro
-keep class com.example.core.network.model.** { *; }
-keepclassmembers class * {
    @com.squareup.moshi.Json <fields>;
}

// Common mistake: putting rules in proguard-rules.pro instead of consumer-proguard-rules.pro
// proguard-rules.pro → only applies when THIS module is the app module
// consumer-proguard-rules.pro → applies when a CONSUMER includes this library
The Release-Only Crash

The most insidious modularization bug: everything works in debug, but the release build crashes with ClassNotFoundException or NoSuchMethodError. This happens when a library module uses reflection (Moshi, Retrofit, Room) but doesn't include consumer-proguard-rules.pro to keep the reflected classes. R8 strips them because it can't see the reflective access. Always test release builds with minifyEnabled = true in CI.

Test Fixtures

Gradle's testFixtures source set (Gradle 5.6+) lets a module expose test utilities without polluting its main source set. Consumers add testImplementation(testFixtures(project(":core:data"))) to access them. 

// :core:data/build.gradle.kts
plugins {
    id("java-test-fixtures")  // enables testFixtures source set
}

// :core:data/src/testFixtures/kotlin/com/example/core/data/FakeArticleRepository.kt
class FakeArticleRepository : ArticleRepository {
    private val articles = MutableStateFlow<List<Article>>(emptyList())

    override fun getArticles(): Flow<List<Article>> = articles

    override suspend fun refreshArticles() {
        articles.value = listOf(
            Article(id = "1", title = "Test Article", isBookmarked = false)
        )
    }

    override suspend fun toggleBookmark(articleId: String) {
        articles.update { list ->
            list.map { if (it.id == articleId) it.copy(isBookmarked = !it.isBookmarked) else it }
        }
    }

    fun emit(vararg items: Article) { articles.value = items.toList() }
}

// :feature:home/build.gradle.kts
dependencies {
    testImplementation(testFixtures(project(":core:data")))
    // Now HomeViewModelTest can use FakeArticleRepository
}
Staff Insight — testFixtures vs :core:testing

There are two strategies for sharing test utilities: testFixtures and a dedicated :core:testing module. Use testFixtures when the fake is tightly coupled to the module it fakes (e.g., FakeArticleRepository lives in :core:data's testFixtures because it implements ArticleRepository). Use :core:testing for cross-cutting test infrastructure (test dispatchers, custom assertions, test rules) that isn't tied to any specific module. Google's NiA uses both — :core:testing for shared infrastructure and testFixtures for module-specific fakes.

Key Takeaways

  • Module taxonomy: :app (shell) → :feature:* (UI + ViewModel) → :core:* (shared infrastructure) → :core:model (leaf).
  • Horizontal isolation: Feature modules never depend on each other. Communication goes through :core:* or navigation.
  • Dependency inversion: :core:domain defines interfaces, :core:data implements them, :app wires them via DI.
  • implementation by default: Use api only when the dependency's types appear in your module's public API. Overusing api destroys incremental build performance.
  • Hybrid organization: Modules by feature, packages by layer within each module. Aligns with team ownership.
  • Granularity sweet spot: 10-30 modules for medium apps, 50-100 for large. Over-modularization has real costs (IDE sync, configuration phase, cognitive overhead).
  • api/impl split: The gold standard for large-scale projects — consumers depend on :api, implementation lives in :impl, DI wires them.
  • consumer-proguard-rules.pro: Library modules must ship their own R8 rules. Missing rules = release-only crashes.
