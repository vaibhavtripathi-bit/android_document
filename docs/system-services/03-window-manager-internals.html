03 — WindowManagerService Internals

The system service that manages every window on every display — hierarchy, surfaces, input dispatch, transitions, insets, and the global lock that governs it all

1. WMS Architecture Overview

WindowManagerService (WMS) runs inside system_server and is the single authority over all windows on all displays. Every pixel you see on an Android screen — Activities, dialogs, toasts, the status bar, the keyboard — is a window managed by WMS.

// WMS is created during system_server boot (SystemServer.java):
//   startOtherServices() {
//       wm = WindowManagerService.main(context, inputManager, ...);
//       ServiceManager.addService(Context.WINDOW_SERVICE, wm);
//   }
//
// WMS holds the global lock (mGlobalLock) shared with ATMS.
// Almost every WMS operation acquires this lock.
// Source: frameworks/base/services/core/java/com/android/server/wm/

// Key responsibilities:
// 1. Maintain the window hierarchy (which windows exist, their z-order)
// 2. Allocate and manage SurfaceControl for each window
// 3. Relay layout information so windows know their size/position
// 4. Coordinate with InputManagerService for input dispatch
// 5. Drive window transitions and animations
// 6. Manage window insets (status bar, nav bar, IME, cutout)
// 7. Handle multi-display window placement

The Window Hierarchy

WMS organizes windows in a tree structure. Every class in this tree extends WindowContainer, which provides child management, z-ordering, and surface hierarchy operations:

      ├── DisplayContent (#0 — built-in)    ← one per physical/virtual display

      │    │    │    ├── WindowToken (StatusBar)

Core Classes

// WindowContainer — abstract base for every node in the tree
// Provides: addChild(), removeChild(), forAllWindows(), assignLayer()
// Each WindowContainer has a SurfaceControl that forms the surface tree

// WindowState — the per-window metadata object
class WindowState extends WindowContainer<WindowState> {
    final WindowManager.LayoutParams mAttrs;  // type, flags, format
    final IWindow mClient;                    // Binder to the app's ViewRootImpl
    int mBaseLayer;                           // z-order base (derived from window type)
    int mSubLayer;                            // sub-window offset within parent
    final Rect mRequestedFrame;               // what the app asked for
    final Rect mFrame;                        // what WMS assigned
    boolean mHasSurface;                      // whether a surface is allocated
    WindowSurfaceController mWinAnimator;     // manages the SurfaceControl
    InsetsState mRequestedInsetsState;        // insets the app sees
}

// WindowToken — groups related windows
// For Activities, ActivityRecord extends WindowToken
// For system windows (toast, status bar), a plain WindowToken is used

// ActivityRecord — extends WindowToken
// Links the activity lifecycle (ATMS) to the window hierarchy (WMS)
// Contains: starting window state, letterbox config, transition info

// DisplayContent — one per display, holds the full window tree for that display
// Contains: DisplayPolicy, DisplayRotation, InsetsStateController
Key Insight
ActivityRecord extends WindowToken is a deliberate design choice. It means Activities are window tokens — there's no separate lookup table mapping activities to their windows. This was refactored in Android 10 (project "WM Unification") to merge the previously separate AMS and WMS hierarchies into a single tree.

Z-Order Calculation

// Z-ordering is determined by two values:
//   mBaseLayer — derived from window type via WindowManagerPolicy
//   mSubLayer  — offset for sub-windows relative to parent

// WindowManagerPolicy.getWindowLayerFromTypeLw(type, canAddInternalSystemWindow):
//   TYPE_WALLPAPER            → layer 1
//   TYPE_BASE_APPLICATION     → layer 2
//   TYPE_APPLICATION          → layer 2
//   TYPE_APPLICATION_PANEL    → layer 2  (sub-window, mSubLayer differentiates)
//   TYPE_TOAST                → layer 8
//   TYPE_INPUT_METHOD         → layer 13
//   TYPE_STATUS_BAR           → layer 15
//   TYPE_NAVIGATION_BAR       → layer 17
//   TYPE_SYSTEM_ALERT         → layer 9
//   TYPE_APPLICATION_OVERLAY  → layer 12

// The actual z-order in the surface tree is set by:
//   WindowContainer.assignLayer(SurfaceControl.Transaction t, int layer)
//
// For app windows within the same Task, z-order follows insertion order.
// DisplayArea boundaries enforce that system windows always appear
// above/below task windows regardless of individual layer values.

// Sub-window layers (relative to parent):
//   TYPE_APPLICATION_PANEL          → mSubLayer = 1  (above parent)
//   TYPE_APPLICATION_MEDIA          → mSubLayer = -2 (below parent)
//   TYPE_APPLICATION_MEDIA_OVERLAY  → mSubLayer = -1 (below parent, above media)
//   TYPE_APPLICATION_ATTACHED_DIALOG → mSubLayer = 1 (above parent)

DisplayPolicy

DisplayPolicy (formerly PhoneWindowManager in the WMS context) makes policy decisions about window placement, visibility, and system UI behavior. It decides:

  • Which system bars are visible and where they're placed
  • How windows interact with the status bar and navigation bar
  • Fullscreen and immersive mode behavior
  • Keyguard (lock screen) window policy
  • Screen rotation and orientation lock

3. SurfaceControl and the Surface Hierarchy

Every WindowContainer in the WMS tree has a corresponding SurfaceControl — a handle to a layer in SurfaceFlinger. The surface tree mirrors the window tree:

// SurfaceControl — a reference to a SurfaceFlinger layer
// Created via SurfaceControl.Builder:
//   new SurfaceControl.Builder(session)
//       .setName("WindowState{com.example/.MainActivity}")
//       .setParent(parentSurfaceControl)
//       .setBufferSize(width, height)
//       .build()

// SurfaceControl.Transaction — batched surface operations
// All surface mutations go through transactions for atomicity:
val t = SurfaceControl.Transaction()
t.setPosition(surfaceControl, x, y)
t.setBufferSize(surfaceControl, width, height)
t.setVisibility(surfaceControl, true)
t.setLayer(surfaceControl, zOrder)
t.setAlpha(surfaceControl, 0.8f)
t.setCrop(surfaceControl, cropRect)
t.apply()  // commits all changes atomically to SurfaceFlinger

// Why transactions matter:
// Without batching, setting position then size would cause a visible
// glitch between the two operations. Transactions ensure atomicity.

WindowSurfaceController

// WindowSurfaceController manages the SurfaceControl for a WindowState.
// It handles:
//   - Surface creation and destruction
//   - Position/size updates via SurfaceControl.Transaction
//   - Surface visibility (show/hide)
//   - Surface format and flags

// Lifecycle:
// 1. createSurfaceLocked() — called when window first needs a surface
//    Creates the SurfaceControl, sets initial properties
// 2. setSurfaceBoundariesLocked() — updates position/size during relayout
// 3. hide() / show() — visibility changes
// 4. destroySurfaceLocked() — cleanup when window is removed

Leash Surfaces and SurfaceAnimator

// A "leash" is an intermediate SurfaceControl inserted between a
// window's parent and the window itself. It's used for animations.

// Normal hierarchy:
//   ParentSurface → WindowSurface

// During animation:
//   ParentSurface → LeashSurface → WindowSurface
//                    ↑ animator controls this

// Why leashes?
// The animator manipulates the leash (position, scale, alpha) without
// touching the window's own surface properties. When the animation ends,
// the leash is removed and the window reparents back to the original parent.
// This prevents animation state from "leaking" into the window's real state.

// SurfaceAnimator drives animations:
class SurfaceAnimator {
    // startAnimation() — creates a leash, starts the animation adapter
    // onAnimationFinished() — removes leash, reparents window back
    // transferAnimation() — hands off animation to another container
    //   (used when an activity finishes during a transition)
}

// Animation adapters:
//   LocalAnimationAdapter — runs animation on system_server (legacy)
//   RemoteAnimationAdapter — delegates to an app or SystemUI
//   SurfaceAnimationRunner — executes ValueAnimator on a separate thread
Staff Insight — Surface Tree Debugging

dumpsys SurfaceFlinger --list prints every layer in the surface tree. Cross-reference this with dumpsys window to verify the WMS window tree matches the SurfaceFlinger layer tree. Mismatches indicate leaked surfaces — a window was removed from WMS but its SurfaceControl wasn't destroyed. This is a real production issue: leaked surfaces consume GPU memory and SurfaceFlinger composition time. In AOSP, WindowState.destroySurface() has multiple code paths (normal removal, process death, surface error) and bugs in any path cause leaks.

4. Window Addition and Removal

addWindow() — Creating a Window

// Entry point: WindowManagerService.addWindow()
// Called when ViewRootImpl.setView() triggers Session.addToDisplayAsUser()

// Simplified flow:
fun addWindow(session: Session, client: IWindow, attrs: LayoutParams,
              viewVisibility: Int, displayId: Int, ...) {

    // 1. Permission check
    //    System window types (>= FIRST_SYSTEM_WINDOW) require either:
    //    - INTERNAL_SYSTEM_WINDOW (signature-level, for system apps)
    //    - SYSTEM_ALERT_WINDOW (for TYPE_APPLICATION_OVERLAY)
    //    Toast windows (TYPE_TOAST) are special-cased: allowed without
    //    permission but limited to the app's own UID

    // 2. Find or create WindowToken
    //    For app windows: token must already exist (created by ATMS
    //    when the Activity was launched)
    //    For system windows: WMS creates a new WindowToken

    // 3. Validate the token
    //    TYPE_APPLICATION windows must have an ActivityRecord token
    //    Sub-windows must find their parent WindowState

    // 4. Create WindowState
    val win = WindowState(this, session, client, token, attrs, ...)
    //    WindowState constructor computes mBaseLayer from the type

    // 5. Policy check
    //    DisplayPolicy.adjustWindowParamsLw() — may modify layout params
    //    DisplayPolicy.prepareAddWindowLw() — can reject the window

    // 6. Register the window
    mWindowMap.put(client.asBinder(), win)  // global IBinder → WindowState map
    token.addWindow(win)                    // add to parent token

    // 7. Assign input channel
    //    Creates a socket pair: one end for InputDispatcher, one for the app
    win.openInputChannel(outInputChannel)

    // 8. Assign layers and request layout
    //    Triggers a full relayout pass
    displayContent.assignWindowLayers(transaction)
}

removeWindow() — Destroying a Window

// Triggered by ViewRootImpl.die() → Session.remove() → WMS.removeWindow()

// Simplified flow:
fun removeWindow(session: Session, client: IWindow) {
    val win = mWindowMap.get(client.asBinder()) ?: return

    // 1. Remove from parent token
    win.removeIfPossible()
    //    If an exit animation is playing, removal is deferred
    //    until the animation completes (removeImmediately() later)

    // 2. Destroy surface
    win.destroySurface()
    //    Releases the SurfaceControl → SurfaceFlinger destroys the layer

    // 3. Close input channel
    win.mInputChannel?.dispose()

    // 4. Recalculate focus
    //    The focused window may have changed
    updateFocusedWindowLocked()

    // 5. Recalculate insets
    //    If the removed window was an inset source (e.g., IME),
    //    all other windows need updated insets
    displayContent.insetsStateController.onWindowRemoved(win)

    // 6. Remove from global map
    mWindowMap.remove(client.asBinder())
}

relayoutWindow() — The Hot Path

// relayoutWindow() is called whenever a window's size, position,
// visibility, or surface needs updating. This is the MOST FREQUENTLY
// called WMS method — it runs every frame during animations.

// Entry: IWindowSession.relayout() → WMS.relayoutWindow()

fun relayoutWindow(session: Session, client: IWindow,
                   attrs: LayoutParams, requestedWidth: Int,
                   requestedHeight: Int, viewVisibility: Int, ...) {

    // 1. Update window attributes if changed
    // 2. Compute new window frame (position + size)
    //    WindowLayout.computeFrames() — considers display bounds,
    //    insets, gravity, requested size, compatibility mode
    // 3. Create or update surface
    //    If the window is becoming visible and has no surface → create
    //    If size changed → resize the surface buffer
    // 4. Send updated frame and insets back to the app
    //    The app's ViewRootImpl uses this to trigger measure/layout/draw
    // 5. Apply surface transaction
    //    Position, size, visibility, crop — all in one transaction
}

// Performance: relayoutWindow() acquires mGlobalLock, computes layout,
// and applies a SurfaceControl.Transaction. During window animations,
// this runs 60-120 times per second PER ANIMATING WINDOW.
Performance Warning
relayoutWindow() holds mGlobalLock for its entire duration. During complex animations with multiple windows (e.g., split-screen transition), this creates significant lock contention. This is why Android 12+ moved animation execution to Shell transitions — the animation runs on a separate thread in SystemUI, only calling back to WMS for the final state commit.

5. Input Dispatch

WMS is the bridge between InputManagerService (which receives raw events from the kernel) and the app windows that consume them.

The Full Input Pipeline

  Linux Kernel (/dev/input/eventN)
       │
       ▼
  InputReader (native thread)
  Reads raw events, classifies device type (touch, key, etc.)
  Applies input mapping, produces MotionEvent / KeyEvent
       │
       ▼
  InputDispatcher (native thread)
  Determines target window via hit-testing or focus
  Sends event through InputChannel (Unix socket pair)
       │
       ├── ANR detection: tracks pending events per window
       │   If app doesn't acknowledge within 5s → ANR
       │
       ▼
  InputChannel (socket pair)
  One end in InputDispatcher (server), one in app process (client)
       │
       ▼
  App Process: InputEventReceiver (on Looper thread)
  Receives event, posts to Choreographer for batching
       │
       ▼
  ViewRootImpl.deliverInputEvent()
       │
       ▼
  InputStage pipeline (in ViewRootImpl):
    NativePreImeInputStage → ViewPreImeInputStage →
    ImeInputStage → EarlyPostImeInputStage →
    NativePostImeInputStage → ViewPostImeInputStage →
    SyntheticInputStage
       │
       ▼
  View.dispatchTouchEvent() / View.dispatchKeyEvent()

Hit Testing

// InputDispatcher.findTouchedWindowAtLocked() determines which window
// receives a touch event:

// 1. Walk the window list top-to-bottom (highest z-order first)
// 2. For each window, check:
//    a. Is the window visible? (InputWindowInfo.visible)
//    b. Is the touch point within the window's touchable region?
//       (InputWindowInfo.touchableRegion — may be smaller than the frame
//        due to FLAG_NOT_TOUCH_MODAL or custom touchable regions)
//    c. Does the window have FLAG_NOT_TOUCHABLE? → skip
//    d. Does the window have FLAG_NOT_FOCUSABLE? → skip for key events
// 3. First matching window wins

// Special cases:
// - FLAG_NOT_TOUCH_MODAL: window only receives touches within its bounds
//   (default for most windows). Without this flag, the window receives
//   ALL touches even outside its bounds.
// - FLAG_WATCH_OUTSIDE_TOUCH: window receives ACTION_OUTSIDE for touches
//   that land on other windows (used by dropdown menus to dismiss)
// - Obscured touch rejection: if a window is covered by another window
//   from a different UID, the touch may be filtered (tapjacking protection)

Focus Management

// WMS maintains the "focused window" — the window that receives key events
// and IME input. Focus is recalculated whenever windows are added, removed,
// or change visibility.

// InputMonitor.updateInputWindowsLw():
//   Builds the complete window list with input properties and sends it
//   to InputDispatcher via InputManagerService.setInputWindows()
//
//   This is called after EVERY window hierarchy change.
//   InputDispatcher uses this list for hit-testing and focus routing.

// Focus calculation (simplified):
// 1. Find the top-most focusable window in the focused app's task
// 2. A window is focusable if:
//    - It's visible
//    - It doesn't have FLAG_NOT_FOCUSABLE
//    - Its app is not frozen (e.g., during config change)
// 3. If no focusable window exists, focus goes to the next app in z-order

// The focused window is critical for:
// - Key event delivery (hardware keys, keyboard input)
// - IME visibility and targeting
// - Accessibility focus tracking

ANR Detection in InputDispatcher

// InputDispatcher tracks dispatched-but-unacknowledged events per window.
// The app must call InputEventReceiver.finishInputEvent() to acknowledge.

// ANR triggers:
// 1. Key event: no focused window, or focused window hasn't consumed
//    the previous key event within 5 seconds
// 2. Touch event: touched window hasn't consumed the previous touch
//    event within 5 seconds
// 3. No focused window for a key event (focus timeout)

// When ANR fires:
//   InputDispatcher → InputManagerService.notifyANR()
//   → ActivityManagerService.inputDispatchingTimedOut()
//   → Shows the "App isn't responding" dialog

// The 5-second timeout is hardcoded in InputDispatcher.cpp:
//   constexpr nsecs_t DEFAULT_INPUT_DISPATCHING_TIMEOUT = 5000 * 1000000LL;

6. Window Transitions and Animations

Shell Transitions (Android 12+)

// The modern transition system decouples animation from WMS.
// WMS collects what changed; SystemUI's Shell library animates it.

// Architecture:
//   WMS (TransitionController)
//     ↕ Binder IPC
//   SystemUI Shell (Transitions)
//     → uses SurfaceControl.Transaction to animate surfaces

// Transition lifecycle:
// 1. COLLECT — WMS detects window changes (open, close, resize, move)
//    TransitionController.collectExistingOpenTransition()
//    Participants are added: the windows/tasks involved
//
// 2. READY — all participants have drawn their first frame
//    WMS calls Transitions.onTransitionReady() in Shell
//
// 3. PLAY — Shell creates animation (fade, slide, shared element)
//    Shell manipulates SurfaceControl leashes directly
//    No mGlobalLock needed during animation playback
//
// 4. FINISH — Shell calls back to WMS
//    WMS applies the final state (visibility, z-order)
//    Leashes are removed, surfaces reparented

// Why Shell transitions are better:
// - Animation runs in SystemUI process, not system_server
// - No mGlobalLock contention during animation
// - SystemUI can customize animations without modifying WMS
// - Easier to implement complex multi-window transitions

Legacy Transitions (pre-Android 12)

// AppTransition — the old animation system, still partially used
// Animations run inside system_server on the animation thread

// AppTransition states:
//   IDLE → TIMEOUT_SET → READY → RUNNING → IDLE

// When an activity opens/closes:
// 1. AppTransition.prepareAppTransitionLocked(TRANSIT_OPEN/CLOSE)
// 2. WMS waits for the new window to draw (or timeout after 5s)
// 3. AppTransition.goodToGo() → starts the animation
// 4. Animation runs via WindowAnimator on the animation thread
// 5. On completion, final state is applied

// Remote animations (Android 9+):
// Apps can provide their own transition via RemoteAnimationAdapter
// Used by Launcher for the app-open animation:
//   val adapter = RemoteAnimationAdapter(runner, duration, statusBarDelay)
//   ActivityOptions.makeRemoteAnimation(adapter)

TransitionController Internals

// TransitionController orchestrates transitions across multiple windows.
// A single transition can involve many participants:
//   - Opening activity's window
//   - Closing activity's window
//   - The task container
//   - Wallpaper (if transitioning to/from launcher)
//   - Navigation bar (may need to animate color change)

// Transition merging:
// If a new transition starts while one is playing, they may merge.
// Example: user rapidly taps back → close transition merges with
// the previous close transition to avoid visual glitches.

// Transition types (Transition.java):
//   TRANSIT_OPEN, TRANSIT_CLOSE
//   TRANSIT_TO_FRONT, TRANSIT_TO_BACK
//   TRANSIT_CHANGE (bounds change, e.g., entering split-screen)
//   TRANSIT_KEYGUARD_GOING_AWAY
//   TRANSIT_PIP

Virtual Displays

Foldable Display Handling

8. Window Insets

Window insets describe the regions of the screen occupied by system UI (status bar, navigation bar, IME, display cutout). The insets system was completely rewritten in Android 11.

InsetsState and InsetsSource

// InsetsState — the complete insets state for a window
// Contains an array of InsetsSource, one per inset type:
//   - STATUS_BAR (top)
//   - NAVIGATION_BAR (bottom or side)
//   - IME (keyboard)
//   - DISPLAY_CUTOUT (notch/hole-punch)
//   - SYSTEM_GESTURES (edge gesture regions)
//   - MANDATORY_SYSTEM_GESTURES (non-overridable gesture regions)
//   - CAPTION_BAR (freeform window title bar)

// Each InsetsSource has:
class InsetsSource {
    val type: Int           // which inset (STATUS_BAR, IME, etc.)
    val frame: Rect         // the region this inset occupies
    var isVisible: Boolean  // whether the inset is currently shown
    // The "frame" is in screen coordinates.
    // A window computes its usable area by subtracting visible inset frames.
}

// InsetsStateController (in WMS, per-display):
// Tracks all inset sources for a display.
// When a source changes (e.g., IME shows), it recomputes InsetsState
// and dispatches it to all affected windows.

InsetsController — The App Side

// InsetsController is the app-side API for controlling insets.
// Accessed via: window.insetsController (or WindowCompat.getInsetsController())

// Show/hide system bars:
val controller = WindowCompat.getInsetsController(window, view)
controller.hide(WindowInsetsCompat.Type.systemBars())
controller.show(WindowInsetsCompat.Type.ime())

// Control behavior when hidden:
controller.systemBarsBehavior =
    WindowInsetsControllerCompat.BEHAVIOR_SHOW_TRANSIENT_BARS_BY_SWIPE

// Under the hood:
// 1. App calls InsetsController.hide(types)
// 2. InsetsController sends a request to WMS via IWindowSession
// 3. WMS updates the InsetsSource visibility
// 4. WMS dispatches updated InsetsState to all affected windows
// 5. Each window's ViewRootImpl applies the new insets via
//    View.onApplyWindowInsets() / WindowInsetsAnimation callbacks

Edge-to-Edge

// "Edge-to-edge" means the app draws behind system bars.
// The modern approach (Android 15+ default, backported via AndroidX):

// Step 1: Tell the system the app handles insets itself
WindowCompat.setDecorFitsSystemWindows(window, false)
// Internally: sets LAYOUT_IN_DISPLAY_CUTOUT_MODE_ALWAYS
// and removes SYSTEM_UI_FLAG_LAYOUT_* flags

// Step 2: Make system bars transparent
window.statusBarColor = Color.TRANSPARENT
window.navigationBarColor = Color.TRANSPARENT

// Step 3: Handle insets in your layout
ViewCompat.setOnApplyWindowInsetsListener(view) { v, insets ->
    val systemBars = insets.getInsets(WindowInsetsCompat.Type.systemBars())
    v.setPadding(systemBars.left, systemBars.top,
                 systemBars.right, systemBars.bottom)
    insets
}

// What happens internally:
// 1. setDecorFitsSystemWindows(false) tells WMS to NOT shrink the
//    window frame to avoid system bars
// 2. WMS sends the full display frame as the window frame
// 3. InsetsState is still sent — the app knows WHERE the bars are
// 4. The app is responsible for padding/offsetting its own content

// In Compose:
Scaffold(
    modifier = Modifier.windowInsetsPadding(WindowInsets.systemBars)
) { paddingValues ->
    // content with proper inset handling
}
WindowInsets API Evolution

Pre-Android 11: systemUiVisibility flags (SYSTEM_UI_FLAG_FULLSCREEN, SYSTEM_UI_FLAG_HIDE_NAVIGATION, etc.) — a bitmask nightmare with 15+ flags and complex interactions. Android 11+: WindowInsetsController with simple show()/hide() methods and typed inset queries. The old flags still work but are deprecated and internally translated to the new system.

9. Staff-Level Corner Cases

The Global Lock Problem

// WMS and ATMS share a single lock: mGlobalLock (WindowManagerGlobalLock)
// This was introduced in Android 10 when AMS and WMS hierarchies merged.

// The problem:
// - ATMS operations (activity start, lifecycle transitions) hold mGlobalLock
// - WMS operations (relayout, surface updates) hold mGlobalLock
// - Input dispatch needs to read the window list (holds mGlobalLock briefly)
// - Animation frame callbacks need layout info (hold mGlobalLock)

// All of these contend on the SAME lock. On a busy system:
//   Thread A: ATMS starting an activity → holds mGlobalLock
//   Thread B: WMS relayoutWindow() → BLOCKED waiting for mGlobalLock
//   Thread C: InputDispatcher updating windows → BLOCKED
//   Thread D: Animation thread → BLOCKED

// This is the #1 source of jank in system_server.
// Mitigation strategies in AOSP:
// 1. Shell transitions (move animation off the lock)
// 2. Async surface operations (apply transactions without holding the lock)
// 3. Read-only snapshots for InputDispatcher (copy window list under lock,
//    InputDispatcher reads the copy without locking)
// 4. Ongoing work to split the lock into finer-grained locks
Staff Insight — mGlobalLock Contention as a System Design Question

In an L6 interview, you might be asked: "How would you reduce jank caused by system_server lock contention?" The answer involves understanding that mGlobalLock serializes all window and activity operations. The key insight is separating read paths from write paths: InputDispatcher only needs a consistent snapshot of the window list (read), while activity launches mutate the hierarchy (write). Android's evolution has been moving toward this separation — Shell transitions moved animation reads out, and InputMonitor now builds a snapshot under the lock and hands it off. A staff engineer would propose further splitting: a reader-writer lock, or lock-free snapshots using immutable data structures for the window list, with copy-on-write semantics for mutations.

Window Leaks

// A "window leak" occurs when an app creates a window (via WindowManager.addView)
// but never removes it. When the Activity is destroyed, ViewRootImpl detects
// orphaned windows and throws WindowLeaked:

// ViewRootImpl.die(immediate):
//   if (mView != null && mAdded) {
//       // Window still attached after Activity destroyed
//       Log.e(TAG, "WindowLeaked: " + mView + " was originally added here",
//              mLocation);  // mLocation = stack trace from addView()
//   }

// Common causes:
// 1. Dialog.show() without Dialog.dismiss() before Activity.onDestroy()
// 2. PopupWindow not dismissed
// 3. Custom floating view added via WindowManager.addView()
// 4. Toast with custom view (pre-Android 11, toasts were real windows)

// The leak is in the app process (ViewRootImpl), but WMS also has the
// WindowState. When the app process dies, WMS cleans up via:
//   AppWindowToken.destroySurfaces() → removes all surfaces
//   WindowState.removeIfPossible() → removes from hierarchy
// But if the process is still alive, the leaked window persists.

Overlay Attacks (TYPE_SYSTEM_ALERT)

// TYPE_SYSTEM_ALERT (and TYPE_APPLICATION_OVERLAY on API 26+) creates
// windows that float above other apps. This enables overlay attacks:

// Attack scenario (StrandHogg-style):
// 1. Malicious app creates an overlay matching the target app's UI
// 2. Overlay captures user input (credentials, permissions)
// 3. User thinks they're interacting with the real app

// Android's defenses:
// - API 23+: SYSTEM_ALERT_WINDOW requires runtime permission
// - API 26+: TYPE_SYSTEM_ALERT deprecated, TYPE_APPLICATION_OVERLAY
//   introduced with stricter z-ordering (below critical system windows)
// - API 31+: apps targeting S+ can't create overlays that cover
//   permission dialogs (FLAG_NOT_TOUCHABLE enforced)
// - Touch filtering: if a touch passes through an overlay from a
//   different UID, the underlying window receives FLAG_WINDOW_IS_OBSCURED
//   in the MotionEvent, and can reject the touch

// Detecting overlays:
// MotionEvent.FLAG_WINDOW_IS_OBSCURED — set when an overlay is above
// MotionEvent.FLAG_WINDOW_IS_PARTIALLY_OBSCURED — partial coverage
// Apps handling sensitive input should check these flags.

// 1. WMS removes the WindowState from the source DisplayContent

Staff Insight — Debugging Production Window Issues

When debugging window-related issues in production (e.g., a window not appearing, wrong z-order, input not reaching the right window), the workflow is: (1) dumpsys window windows to see the full window list and verify your window exists with correct visibility and frame, (2) dumpsys window input to verify the focused window and input channel state, (3) dumpsys SurfaceFlinger to verify the surface exists and has the correct layer, (4) dumpsys input to see InputDispatcher's view of the window list and any pending/dropped events. The gap between what WMS thinks (dumpsys window) and what InputDispatcher thinks (dumpsys input) is where most input-routing bugs live.

10. DisplayArea Hierarchy (Android 12+)

Android 12 replaced the flat window list with a tree-structured DisplayArea hierarchy. This enables flexible window organization for foldables, multi-display devices, and OEM customization.

The DisplayArea Tree

DisplayContent (#0 — built-in display)
 └── DisplayArea.Root
      │
      ├── DisplayArea (ABOVE_TASKS)              ← z-order above all app windows
      │    ├── WindowToken (StatusBar)
      │    │    └── WindowState (status_bar)
      │    ├── WindowToken (NavigationBar)
      │    │    └── WindowState (navigation_bar)
      │    └── WindowToken (NotificationShade)
      │         └── WindowState (shade)
      │
      ├── TaskDisplayArea (DEFAULT_TASK_DISPLAY_AREA)  ← all app Tasks live here
      │    ├── Task (stack #1 — fullscreen)
      │    │    └── Task (leaf) → ActivityRecord → WindowState
      │    ├── Task (stack #2 — split-screen primary)
      │    │    └── ...
      │    └── Task (stack #3 — split-screen secondary)
      │         └── ...
      │
      └── DisplayArea (BELOW_TASKS)              ← z-order below all app windows
           └── WindowToken (Wallpaper)
                └── WindowState (wallpaper)

// DisplayAreaPolicy defines HOW the DisplayArea tree is structured.
// OEMs can provide a custom policy to reorganize the hierarchy.

// Default policy (DisplayAreaPolicyBuilder):
// Root → ABOVE_TASKS / DEFAULT_TASK_DISPLAY_AREA / BELOW_TASKS
// This is sufficient for standard phones.

// OEM customization examples:
// - Foldable: separate TaskDisplayAreas for inner and outer screens
// - Automotive: dedicated DisplayArea for instrument cluster vs infotainment
// - Dual-screen: two TaskDisplayAreas, one per physical screen half

// The policy is set per-display during DisplayContent initialization:
// DisplayContent.configureSurfaces(Transaction t) {
//     mDisplayAreaPolicy = DisplayAreaPolicy.Provider
//         .fromResources(mWmService.mContext.getResources())
//         .instantiate(this);
// }

// DisplayAreaGroup — coordinates operations across multiple DisplayAreas:
// When entering split-screen, a DisplayAreaGroup ensures both halves
// resize atomically and maintain consistent z-ordering boundaries.
Staff Insight — Why DisplayArea Matters for Large Screens

The flat window list worked fine when Android ran on single-screen phones. But foldables, tablets, and desktop modes need regions within a display where different sets of tasks live. DisplayArea provides this: a TaskDisplayArea is a bounded region where tasks can be placed, and the tree structure ensures system windows (status bar, nav bar) always appear at the correct z-order relative to app windows — regardless of how many task regions exist. Without this, implementing Activity Embedding or split-screen on foldables would require brittle z-order hacks.

11. Window Layout Computation

Every time relayoutWindow() runs, WMS must compute the exact position and size of the window. This is the hot path that runs 60–120 times per second during animations.

WindowLayout.computeFrames()

// Source: frameworks/base/services/core/java/com/android/server/wm/WindowLayout.java

// computeFrames() calculates the final window frame given:
// - Parent bounds (the containing DisplayArea or Task bounds)
// - Requested size (what the app asked for via LayoutParams)
// - Gravity (how to position within the parent)
// - Insets (status bar, nav bar, IME, cutout)
// - Compatibility mode (scaling for old apps)

// Simplified logic:
fun computeFrames(
    attrs: WindowManager.LayoutParams,
    parentBounds: Rect,
    displayFrame: Rect,
    compatScale: Float,
    outFrame: Rect,          // → final window frame
    outDisplayFrame: Rect,   // → available display area
    outInsetsFrame: Rect     // → frame after insets applied
) {
    // Step 1: Resolve requested width/height
    // MATCH_PARENT → use parent bounds
    // WRAP_CONTENT → use requested size (from measure pass)
    // Fixed value → use as-is

    // Step 2: Apply gravity
    // Gravity.apply(attrs.gravity, requestedWidth, requestedHeight,
    //               parentBounds, outFrame)
    // CENTER → centered in parent
    // TOP|FILL_HORIZONTAL → full width, pinned to top
    // BOTTOM|CENTER_HORIZONTAL → centered at bottom

    // Step 3: Apply insets
    // Subtract visible inset regions from the available frame
    // Unless the window opts out (edge-to-edge)

    // Step 4: Apply compatibility scaling
    // For apps targeting old SDKs on high-density displays,
    // WMS scales the frame and applies letterboxing
}

Gravity and Positioning

// Gravity determines how a window is positioned within its parent bounds.
// Common gravity combinations for system windows:

// Status bar:
//   Gravity.TOP | Gravity.FILL_HORIZONTAL
//   → pinned to top, stretches full width

// Navigation bar (gesture nav):
//   Gravity.BOTTOM | Gravity.FILL_HORIZONTAL
//   → pinned to bottom, stretches full width

// Dialog:
//   Gravity.CENTER
//   → centered both horizontally and vertically

// Toast:
//   Gravity.CENTER_HORIZONTAL | Gravity.BOTTOM
//   → centered horizontally, offset from bottom

// PiP window:
//   Gravity.NO_GRAVITY (position set explicitly via x, y offsets)
//   → WMS computes position based on PiP bounds and stash state
Staff Insight — Layout Computation Performance

Layout computation runs on every relayoutWindow() call — which happens every frame during window animations. On a device running at 120Hz with 3 animating windows (e.g., split-screen transition), that's 360 computeFrames() calls per second, all under mGlobalLock. This is why the method is carefully optimized: no allocations, no floating-point math, pure integer rect arithmetic. Adding even a single object allocation to this path would cause measurable GC pressure during animations. When profiling system_server jank, computeFrames() and its callers are one of the first places to look.

12. InputDispatcher Focus & Timeout

InputDispatcher (native C++ code in frameworks/native/services/inputflinger/) is the component that decides which window receives each input event. Its focus management and ANR timeout logic are critical to understand.

Focus Window Per Display

// InputDispatcher maintains a focused window for EACH display.
// The focused window receives all key events and IME input for that display.

// Focus is set by WMS via:
// InputMonitor.updateInputWindowsLw()
//   → builds InputWindowInfo list from the WMS window tree
//   → calls InputManagerService.setInputWindows(displayId, windowInfoList)
//   → InputDispatcher stores the window list and focused window per display

// The window list includes for each window:
// - InputWindowInfo.token (IBinder identifying the window)
// - InputWindowInfo.frame (touchable bounds)
// - InputWindowInfo.touchableRegion (may be smaller than frame)
// - InputWindowInfo.visible
// - InputWindowInfo.focusable
// - InputWindowInfo.inputFeatures (e.g., SPY window for gestures)

findFocusedWindowTargetLocked()

// When a key event arrives, InputDispatcher calls:
// findFocusedWindowTargetLocked(currentTime, entry, nextWakeupTime)

// Logic:
// 1. Get the focused window for the event's display
// 2. If no focused window → wait (with timeout) for WMS to set one
// 3. If focused window exists but has pending unacknowledged events:
//    → Check if the timeout (5s) has elapsed
//    → If elapsed → trigger ANR
//    → If not → wait for the app to finish processing
// 4. If focused window is ready → dispatch the event

// The 5s timeout is the INPUT_DISPATCHING_TIMEOUT:
// constexpr nsecs_t DEFAULT_INPUT_DISPATCHING_TIMEOUT = 5000 * 1000000LL;
// (defined in InputDispatcher.cpp)

Focus Change Race Condition

// There's an inherent race between WMS updating focus and InputDispatcher
// receiving the update:

// Timeline:
// T0: User taps on App B (currently App A is focused)
// T1: WMS processes the touch → determines App B should be focused
// T2: WMS calls setInputWindows() with App B as focused
// T3: InputDispatcher receives the update
//
// Between T0 and T3, key events still go to App A (the old focused window).
// This is usually imperceptible (< 1 frame), but under heavy load
// (mGlobalLock contention), the gap can grow to 50-100ms.
//
// Mitigation: InputDispatcher uses "focus transfer" — when focus changes,
// any in-flight key events for the old window are cancelled (ACTION_CANCEL)
// and re-dispatched to the new focused window.

ANR Notification Flow

// When InputDispatcher detects an ANR (5s timeout):

// InputDispatcher (native)
//   → InputDispatcher::onAnrLocked()
//     → mPolicy->notifyANR(connection, reason)
//       → InputManagerService.notifyANR() (JNI callback to Java)
//         → ActivityManagerService.inputDispatchingTimedOut()
//           → ProcessRecord.appNotResponding()
//             → Dump traces (kill -3)
//             → Show "App isn't responding" dialog
//             → Log to dropbox for later analysis

// The ANR dialog gives the user two choices:
// "Wait" → resets the timeout, gives the app more time
// "Close app" → force-stops the process
Production Impact

The 5-second input dispatching timeout is the most common ANR type in production — accounting for ~70% of all ANRs reported via Play Vitals. It's triggered whenever the main thread is blocked: database queries, synchronous network calls, heavy JSON parsing, lock contention, or even excessive SharedPreferences.apply() calls that queue up on the main thread's finalizer. Unlike broadcast ANRs (10s) or service ANRs (20s), the input ANR has the tightest timeout because users directly feel input unresponsiveness.

13. Predictive Back Gesture

Android 13 introduced predictive back: the system shows a preview of the back destination before the user completes the back gesture. This requires deep coordination between WMS, the back animation system, and the app.

Architecture

// The predictive back system involves:
// 1. WMS — coordinates window surfaces for the animation
// 2. BackAnimationController — orchestrates the animation lifecycle
// 3. BackNavigationInfo — describes the back destination
// 4. The app's OnBackAnimationCallback — optional custom animation

// When the user starts a back gesture (swipe from edge):
// InputDispatcher → SystemUI (gesture detection)
//   → BackAnimationController.onBackGestureStarted()
//     → WMS provides the current and destination window surfaces
//     → Animation begins immediately (predictive preview)

The Three Phases

// Predictive back has three distinct phases:

// Phase 1: onBackStarted(backEvent: BackEvent)
//   Gesture begins. The system creates a temporary animation surface
//   (separate from the window's actual surface) and starts showing
//   the back destination preview.
//   backEvent.touchX / touchY — initial touch position
//   backEvent.progress — 0.0 (just started)

// Phase 2: onBackProgressed(backEvent: BackEvent)
//   Called continuously as the user's finger moves.
//   backEvent.progress — 0.0 to 1.0 (how far the gesture has progressed)
//   The app can use this to animate its own UI (e.g., shrink the card,
//   reveal the previous screen underneath).

// Phase 3a: onBackInvoked()
//   User completed the gesture → navigate back.
//   The animation finishes and the current screen is removed.

// Phase 3b: onBackCancelled()
//   User cancelled the gesture (moved finger back).
//   The animation reverses and the current screen is restored.

// Registration:
// onBackPressedDispatcher.addCallback(
//     owner = this,
//     onBackPressedCallback = object : OnBackPressedCallback(true) {
//         override fun handleOnBackStarted(backEvent: BackEvent) { ... }
//         override fun handleOnBackProgressed(backEvent: BackEvent) { ... }
//         override fun handleOnBackPressed() { ... }
//         override fun handleOnBackCancelled() { ... }
//     }
// )

WMS Surface Coordination

// WMS creates temporary surfaces for the predictive back animation:

// 1. Snapshot the current window → create a screenshot surface
// 2. Prepare the destination window → either:
//    a. The previous Activity in the back stack (cross-activity back)
//    b. The home screen (back from root activity)
//    c. A custom destination (app-defined)
// 3. BackAnimationController manipulates both surfaces:
//    - Current window: scale down, translate, fade out
//    - Destination window: scale up, translate in, fade in
// 4. On completion: remove temporary surfaces, apply final state

// The temporary surfaces are critical — they prevent the animation
// from interfering with the window's real surface state. If the user
// cancels the gesture, the temporary surfaces are simply discarded.
Compatibility Edge Case

If an app doesn't opt into predictive back (no OnBackAnimationCallback registered and android:enableOnBackInvokedCallback="false" in the manifest), the system falls back to the legacy onBackPressed() behavior — no preview animation, immediate back navigation on gesture completion. During the transition period (Android 13–15), apps must explicitly opt in. Android 16+ is expected to enforce predictive back for all apps targeting that SDK level, which will break apps that override onBackPressed() without migrating to the callback API.

14. TaskFragment (Android 12L)

TaskFragment enables Activity Embedding — showing two Activities side-by-side within a single Task on large screens. It's a container in the WMS hierarchy between Task and ActivityRecord.

Hierarchy Position

TaskDisplayArea
 └── Task (root task)
      ├── TaskFragment (left pane — 50% width)
      │    └── ActivityRecord (ListActivity)
      │         └── WindowState
      │
      └── TaskFragment (right pane — 50% width)
           └── ActivityRecord (DetailActivity)
                └── WindowState

Without Activity Embedding (traditional):
TaskDisplayArea
 └── Task
      └── ActivityRecord (only one visible at a time)
           └── WindowState

TaskFragmentOrganizer API

// TaskFragmentOrganizer is the app-side API for controlling
// how Activities are arranged within TaskFragments.

// The Jetpack WindowManager library uses this internally:
// SplitController → TaskFragmentOrganizer → WMS

// Split rules define which Activities should be embedded:
// SplitPairRule — two specific Activities side-by-side
val splitPairRule = SplitPairRule.Builder(
    setOf(SplitPairFilter(
        ComponentName(this, ListActivity::class.java),
        ComponentName(this, DetailActivity::class.java),
        null  // intent action filter
    ))
)
    .setMinWidthDp(600)         // minimum width to split
    .setMinSmallestWidthDp(600) // minimum smallest width
    .setSplitRatio(0.33f)       // 1/3 + 2/3 split
    .setFinishPrimaryWithSecondary(SplitRule.FinishBehavior.NEVER)
    .setFinishSecondaryWithPrimary(SplitRule.FinishBehavior.ALWAYS)
    .build()

// SplitPlaceholderRule — show a placeholder until the detail Activity launches
val placeholderRule = SplitPlaceholderRule.Builder(
    setOf(ActivityFilter(ComponentName(this, ListActivity::class.java), null)),
    Intent(this, PlaceholderActivity::class.java)
)
    .setMinWidthDp(600)
    .build()

// Register rules:
RuleController.getInstance(this).addRule(splitPairRule)
RuleController.getInstance(this).addRule(placeholderRule)

WMS Management of TaskFragments

// WMS manages TaskFragment as a WindowContainer in the hierarchy:
// - TaskFragment extends WindowContainer
// - It sits between Task and ActivityRecord
// - Each TaskFragment has its own bounds (a sub-region of the Task)
// - WMS computes layout for each TaskFragment independently

// Key operations:
// createTaskFragment(token, ownerToken, bounds)
//   → WMS creates a new TaskFragment within the specified Task
// reparentActivityToTaskFragment(activityToken, taskFragmentToken)
//   → moves an ActivityRecord into a TaskFragment
// deleteTaskFragment(token)
//   → removes the TaskFragment, reparents Activities back to Task

// Surface hierarchy:
// Task SurfaceControl
//   ├── TaskFragment SurfaceControl (left, cropped to left bounds)
//   │    └── ActivityRecord SurfaceControl
//   └── TaskFragment SurfaceControl (right, cropped to right bounds)
//        └── ActivityRecord SurfaceControl
Collapse Behavior

When a TaskFragment is too narrow (e.g., phone in portrait, or the user resizes a freeform window below the minWidthDp threshold), the split collapses to single-Activity mode. The secondary Activity is either finished or moved to the back of the stack, depending on the FinishBehavior configured in the split rule. This transition must be seamless — WMS animates the collapse using Shell transitions, and the remaining Activity receives a configuration change for the new (full-width) bounds. Testing this collapse/expand cycle across configuration changes is critical for large-screen apps.
