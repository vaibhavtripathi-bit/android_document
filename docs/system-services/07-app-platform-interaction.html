07 — App-Platform Interaction

The boundary between app code and system services — how Context.getSystemService() resolves to a Binder proxy, how Manager classes abstract IPC, and how the platform controls what apps can reach

1. Context.getSystemService() — The App's Entry Point

Every Android app interacts with the platform through a single gateway: Context.getSystemService(). This deceptively simple call hides a multi-layered resolution chain that spans from a static registry in the app process, through Binder IPC, to the system_server.

1.1 SystemServiceRegistry — The Static Factory Map

SystemServiceRegistry is a static class initialized once per process. It maps service name strings to ServiceFetcher factory lambdas. Every system service the SDK exposes is registered here at class-load time.

// frameworks/base/core/java/android/app/SystemServiceRegistry.java

object SystemServiceRegistry {

    // The master map: service name → factory
    private val SYSTEM_SERVICE_FETCHERS =
        ArrayMap<String, ServiceFetcher<*>>()

    // The reverse map: manager class → service name
    private val SYSTEM_SERVICE_NAMES =
        ArrayMap<Class<*>, String>()

    init {
        // Registered at class-load time via static block
        // ~120 services registered here

        registerService(
            Context.ALARM_SERVICE,
            AlarmManager::class.java,
            CachedServiceFetcher { ctx ->
                val binder = ServiceManager.getService(Context.ALARM_SERVICE)
                val service = IAlarmManager.Stub.asInterface(binder)
                AlarmManager(service, ctx)
            }
        )

        registerService(
            Context.ACTIVITY_SERVICE,
            ActivityManager::class.java,
            CachedServiceFetcher { ctx ->
                ActivityManager(ctx)
            }
        )

        registerService(
            Context.WINDOW_SERVICE,
            WindowManager::class.java,
            CachedServiceFetcher { ctx ->
                // WindowManager is special — it's per-Display
                ctx.display?.let { display ->
                    WindowManagerImpl(ctx)
                }
            }
        )

        // ... 117 more registrations
    }

    fun <T> getSystemService(ctx: ContextImpl, name: String): T? {
        val fetcher = SYSTEM_SERVICE_FETCHERS[name] ?: return null
        @Suppress("UNCHECKED_CAST")
        return fetcher.getService(ctx) as? T
    }
}

1.2 ServiceFetcher Variants

Not all services are created equal. The registry uses three fetcher strategies depending on whether the service needs per-Context state, is a true singleton, or is created once per process.

Fetcher Type Caching Example Services When Used
CachedServiceFetcher Per-Context (stored in ContextImpl.mServiceCache[]) AlarmManager, NotificationManager, ConnectivityManager Service needs Context reference (most common)
StaticServiceFetcher Singleton — one instance shared across all Contexts InputMethodManager, AccessibilityManager Service is truly global, no per-Context state
StaticApplicationContextServiceFetcher Singleton — created with Application context ConnectivityManager (Android 12+) Prevents Context leaks from Activity references 
// CachedServiceFetcher — the most common pattern

abstract class CachedServiceFetcher<T> : ServiceFetcher<T> {
    private val cacheIndex: Int = sServiceCacheSize++

    override fun getService(ctx: ContextImpl): T? {
        val cache = ctx.mServiceCache
        val gates = ctx.mServiceInitializationStateArray

        var service = cache[cacheIndex] as? T
        if (service != null) return service  // Cache hit

        // Prevent double-init with a gate (compare-and-swap)
        if (gates[cacheIndex].compareAndSet(
                ContextImpl.STATE_UNINITIALIZED,
                ContextImpl.STATE_INITIALIZING
            )) {
            try {
                service = createService(ctx)
                cache[cacheIndex] = service
                gates[cacheIndex].set(ContextImpl.STATE_READY)
            } catch (e: Exception) {
                gates[cacheIndex].set(ContextImpl.STATE_NOT_FOUND)
                throw e
            }
        } else {
            // Another thread is initializing — spin-wait
            while (gates[cacheIndex].get() == ContextImpl.STATE_INITIALIZING) {
                Thread.yield()
            }
            service = cache[cacheIndex] as? T
        }
        return service
    }

    abstract fun createService(ctx: ContextImpl): T
}

// StaticServiceFetcher — true singleton

abstract class StaticServiceFetcher<T> : ServiceFetcher<T> {
    @Volatile private var cachedInstance: T? = null

    override fun getService(ctx: ContextImpl): T {
        cachedInstance?.let { return it }
        synchronized(this) {
            cachedInstance?.let { return it }
            val instance = createService()
            cachedInstance = instance
            return instance
        }
    }

    abstract fun createService(): T
}
Interview Trap — Context Type Matters

getSystemService() returns different objects depending on the Context. WindowManager obtained from an Activity context is bound to that Activity's window and display. WindowManager from Application context uses the default display. Calling applicationContext.getSystemService(LAYOUT_INFLATER_SERVICE) returns a LayoutInflater without the Activity's theme — inflated views will use the default theme. This is a common source of subtle UI bugs.

2. ServiceManager — The Binder Directory

ServiceManager is the phone book of Android IPC. It's the very first Binder service registered in the system — assigned handle 0 in the Binder driver, which means every process can reach it without needing to look it up first.

2.1 The Native servicemanager Daemon

The actual directory is maintained by a native C++ daemon (servicemanager) started by init before Zygote. The Java ServiceManager class is a thin wrapper that sends Binder transactions to this daemon.

// The native servicemanager process lifecycle:
//
// 1. init starts servicemanager (defined in servicemanager.rc)
//    → PID ~2, runs as UID 1000 (system)
//    → Registers itself as the "context manager" with the Binder driver
//    → This gives it the special handle 0
//
// 2. servicemanager opens /dev/binder and calls:
//    ioctl(fd, BINDER_SET_CONTEXT_MGR_EXT, &obj)
//    → The kernel Binder driver now routes all handle-0 transactions here
//
// 3. servicemanager enters its event loop:
//    → Waits for addService / getService / listServices transactions
//    → Maintains an internal map: name (String) → IBinder handle
//
// 4. SELinux enforcement:
//    → Every addService/getService call is checked against SELinux policy
//    → service_contexts file maps service names to SELinux labels
//    → Example: "alarm" → u:object_r:alarm_service:s0
//    → An app can only getService() if its SELinux domain has
//      "find" permission for that service's label

L6 Insight — ServiceManager Caching and the Binder Proxy Lifecycle

The Java ServiceManager maintains a local sCache (a HashMap<String, IBinder>) to avoid repeated IPC to the native servicemanager. But this cache is never invalidated during normal operation — it assumes system services don't die. If system_server crashes and restarts, all cached proxies become stale DeadObjectException bombs. This is acceptable because a system_server crash triggers a runtime restart that kills all app processes anyway. However, for HAL services (registered via hwservicemanager), the proxy can become stale if the HAL process crashes independently. Apps using Camera2 or MediaCodec must handle DeadObjectException from HAL proxies — the Manager classes do this internally with DeathRecipient callbacks that reconnect on death.

3. Manager Classes — The App-Facing API

Apps never interact with raw AIDL proxies directly. Each system service has a corresponding Manager class that wraps the Binder proxy and provides a clean, versioned, Kotlin/Java-friendly API.

3.1 What Managers Do

Responsibility Example
Wrap AIDL proxy with clean API AlarmManager.setExactAndAllowWhileIdle() wraps IAlarmManager.set() with correct flags
Argument validation NotificationManager.notify() validates channel ID exists before IPC
Exception translation Converts RemoteException to RuntimeException or swallows it (rethrow as RuntimeException since system_server death is unrecoverable)
Client-side caching PackageManager.getInstalledPackages() may cache results to avoid repeated IPC
Binder death handling LocationManager re-registers listeners if the location service restarts
Backward compatibility NotificationManager.createNotificationChannel() is a no-op below API 26
Thread safety Managers are generally thread-safe — the underlying Binder proxy handles synchronization

3.2 Anatomy of a Manager Class

// Simplified AlarmManager showing the Manager pattern

class AlarmManager(
    private val service: IAlarmManager,  // AIDL proxy
    private val context: Context
) {

    // Clean API that hides AIDL complexity
    fun setExactAndAllowWhileIdle(
        type: Int,
        triggerAtMillis: Long,
        operation: PendingIntent
    ) {
        // Argument validation — fail fast in the app process
        require(triggerAtMillis >= 0) { "Invalid trigger time" }

        try {
            // Delegate to AIDL proxy → Binder IPC → system_server
            service.set(
                context.packageName,
                type,
                triggerAtMillis,
                0,          // windowMillis (exact = 0)
                0,          // interval
                FLAG_ALLOW_WHILE_IDLE or FLAG_STANDALONE,
                operation,
                null,       // listener
                null,       // listenerTag
                null        // workSource
            )
        } catch (e: RemoteException) {
            // system_server is dead — nothing we can do
            throw e.rethrowFromSystemServer()
        }
    }

    // Cancellation wraps the proxy call
    fun cancel(operation: PendingIntent) {
        try {
            service.remove(operation, null)
        } catch (e: RemoteException) {
            throw e.rethrowFromSystemServer()
        }
    }

    // Some Managers provide listener-based async APIs
    fun setExact(
        type: Int,
        triggerAtMillis: Long,
        tag: String,
        listener: OnAlarmListener,
        handler: Handler?
    ) {
        val listenerWrapper = ListenerWrapper(listener)
        try {
            service.set(
                context.packageName,
                type,
                triggerAtMillis,
                0, 0, 0,
                null,             // no PendingIntent
                listenerWrapper,  // Binder callback
                tag,
                null
            )
        } catch (e: RemoteException) {
            throw e.rethrowFromSystemServer()
        }
    }
}

3.3 Singleton vs Per-Context Managers

// Most Managers are per-Context (CachedServiceFetcher):
// Each Activity/Service gets its own AlarmManager, NotificationManager, etc.
// The Manager holds a reference to the Context that created it.

// Some Managers are singletons (StaticServiceFetcher):
// InputMethodManager — one per process, manages the soft keyboard
// AccessibilityManager — one per process, manages accessibility state

// Why the distinction?
// Per-Context: Manager behavior depends on the Context
//   → NotificationManager needs the calling package name from Context
//   → WindowManager needs the Activity's window token
//   → LayoutInflater needs the Activity's theme
//
// Singleton: Manager is truly process-global
//   → InputMethodManager: one keyboard per process
//   → AccessibilityManager: accessibility state is global

// Android 12+ trend: migrating from per-Context to singleton
// using StaticApplicationContextServiceFetcher to prevent
// Activity Context leaks in long-lived Manager references

4. System API Boundaries

Android's API surface is layered. Not every method on a system service is available to all apps. Understanding these layers is critical for L6 because it explains why certain APIs exist and who can use them.

API Level Annotation Who Can Access Stability Guarantee Example
Public SDK (none — default) All apps Full backward compatibility AlarmManager.set()
System API @SystemApi Privileged apps (system partition, platform-signed) Stable across releases, may evolve ActivityManager.forceStopPackage()
Module Library API @SystemApi(client=MODULE_LIBRARIES) Mainline modules only Stable within module boundary Networking stack internals
Module App API @SystemApi(client=MODULE_APPS) Apps bundled with Mainline modules Stable within module boundary Permission controller APIs
Test API @TestApi Test builds only (userdebug/eng) No stability guarantee ActivityManager.forceStopPackage() in tests
Hidden API @hide (Javadoc) Framework internals only None — can change any time ActivityManagerService.dumpStackTraces() 
// How the API boundary is enforced at build time:

// 1. The Android SDK (android.jar) only contains public + @SystemApi methods
//    → Apps compiled against the SDK simply can't reference @hide methods

// 2. For @SystemApi access, apps must:
//    a. Be installed in /system/priv-app/ (privileged app)
//    b. Hold the required signature-level permission
//    c. Compile against the full framework (not the SDK stub)

// 3. At runtime, the actual framework classes contain ALL methods
//    → Reflection can reach @hide methods (this is the problem)
//    → Android 9+ added runtime restrictions (next section)

// Example: the difference between SDK and full framework

// In android.jar (SDK stub):
class ActivityManager {
    fun getRunningAppProcesses(): List<RunningAppProcessInfo>  // public
    // forceStopPackage() is NOT here — it's @SystemApi
    // getProcessPss() is NOT here — it's @hide
}

// In the actual framework on device:
class ActivityManager {
    fun getRunningAppProcesses(): List<RunningAppProcessInfo>  // public
    @SystemApi
    fun forceStopPackage(packageName: String)  // system apps only
    /** @hide */
    fun getProcessPss(pids: IntArray): LongArray  // internal only
}

5. Hidden API Restrictions

Before Android 9, apps could freely use reflection to access @hide APIs. This created a massive compatibility problem: apps broke on every Android upgrade because internal APIs changed. Android 9 introduced a graduated enforcement system.

5.1 The Greylist System

// Hidden APIs are categorized into lists based on usage and risk:

// ┌──────────────────────────────────────────────────────────────┐
// │ List              │ Behavior                                  │
// ├──────────────────────────────────────────────────────────────┤
// │ Whitelist         │ Public SDK — always accessible             │
// │ Light greylist    │ @UnsupportedAppUsage — allowed but warned  │
// │                   │ (logcat warning, may be blocked later)     │
// │ Dark greylist     │ Blocked for apps targeting ≥ certain SDK   │
// │                   │ (greylist-max-o: blocked if targetSdk ≥ 28)│
// │ Blacklist         │ Always blocked — throws exception          │
// └──────────────────────────────────────────────────────────────┘

// The lists are defined in:
// frameworks/base/config/hiddenapi-greylist.txt
// frameworks/base/config/hiddenapi-greylist-max-o.txt
// frameworks/base/config/hiddenapi-greylist-max-p.txt
// etc.

5.2 Enforcement Mechanism

// The enforcement happens in ART (the runtime), not in the framework:

// 1. When an app calls Class.getDeclaredMethod() or Field.get():
//    → ART intercepts the reflection call
//    → Checks the target member against the hidden API lists
//    → Checks the calling app's targetSdkVersion

// 2. The check flow:
//    App calls Method.invoke() on a @hide method
//        │
//        ▼
//    ART: hiddenapi::ShouldDenyAccessToMember()
//        │
//        ├── Is caller exempt? (system app, platform-signed, test build)
//        │   → YES: allow
//        │
//        ├── Is member on the whitelist?
//        │   → YES: allow
//        │
//        ├── Is member on the light greylist?
//        │   → Log warning, allow (for now)
//        │
//        ├── Is member on dark greylist for this targetSdk?
//        │   → Block: throw NoSuchMethodException / NoSuchFieldException
//        │
//        └── Is member on the blacklist?
//            → Block: throw NoSuchMethodException / NoSuchFieldException

// 3. The policy is stored in the app's runtime flags:
//    HiddenApiEnforcementPolicy.ENABLED  → full enforcement
//    HiddenApiEnforcementPolicy.DISABLED → no enforcement (system apps)
//    Set by PMS based on the app's signing certificate and install location

5.3 Evolution of Enforcement

9 (Pie) Warning + partial block Light greylist warns; dark greylist blocks for targetSdk ≥ 28
10 Stricter greylist greylist-max-o blocked for targetSdk ≥ 29; more APIs moved to blacklist
11 Tighter greylist-max-p blocked; JNI access to hidden APIs also restricted
12 Near-complete Most light greylist APIs moved to dark greylist or blacklist
13+ Full enforcement Very few greylist exceptions remain; @UnsupportedAppUsage sunset

Real-World Breakage

The hidden API restrictions broke thousands of apps. Notable examples: apps using ActivityThread.currentApplication() (light greylist, later blocked), apps accessing View.mListenerInfo for click interception, and apps calling Telephony.Sms.moveMessageToFolder() (hidden API). The @UnsupportedAppUsage annotation was Google's compromise — it marked APIs that were widely used and gave developers time to migrate. But each Android release shrinks the greylist further.

6.2 APEX Packages

// APEX (Android Pony EXpress) is the packaging format for native modules.
// Unlike APKs, APEX packages can contain:
//   - Native shared libraries (.so)
//   - Java libraries (JAR/DEX)
//   - Configuration files
//   - Precompiled binaries

// APEX lifecycle:
// 1. Downloaded via Google Play (as a special update)
// 2. Verified (APK signature + dm-verity for the payload)
// 3. Staged for next boot
// 4. At boot: ApexManager mounts the APEX at /apex/<name>/
//    → Old version at /apex/<name>@<old_version>/
//    → New version at /apex/<name>@<new_version>/ (active)

// Example: Conscrypt module
// /apex/com.android.conscrypt/
//   ├── javalib/conscrypt.jar      → TLS provider
//   ├── lib64/libcrypto.so         → BoringSSL
//   ├── lib64/libssl.so            → SSL library
//   └── apex_manifest.json         → version, dependencies

// Key constraint: APEX modules can only replace code, not data.
// They can't modify /data — only /apex mount points.
// This means module updates are atomic and rollback-safe.

6.3 Impact on App Developers

// Before Mainline: behavior changes only arrived with OS updates
// After Mainline: behavior changes can arrive ANY TIME via module updates

// This creates a new compatibility challenge:

// 1. TLS behavior changes (Conscrypt module)
//    → A Conscrypt update could change TLS version defaults

// 2. Media codec changes

//    → Mitigation: don't assume DNS resolution time

The New Fragmentation

Mainline was designed to reduce fragmentation, but it introduces a new kind: module version fragmentation. Two devices running Android 13 might have different Conscrypt versions, different media codec versions, and different networking stack versions. Your app can no longer assume that "Android 13 = consistent behavior." The mitigation is to code defensively, use Jetpack abstractions where possible, and test against a range of module versions.

7. Jetpack Libraries as Platform Abstraction

Many Jetpack libraries exist specifically to abstract away the differences between system service APIs across Android versions. They provide a stable API surface that works from API 21+ while internally dispatching to the correct system service.

7.1 Abstraction Layers

Jetpack Library Wraps System Service Why Needed
WorkManager JobScheduler (23+), AlarmManager + BroadcastReceiver (pre-23), GCM (pre-21) Background work API changed 4 times across OS versions
CameraX Camera2 API → CameraService (HAL) Camera2 is complex, device-specific quirks need workarounds
BiometricPrompt BiometricManager (28+), FingerprintManager (23-27) Biometric API was completely redesigned in API 28
WindowManager (Jetpack) DisplayManager + WindowMetrics Foldable/multi-display support varies wildly across OEMs
NotificationCompat NotificationManager Notification API changed significantly in API 26 (channels) and 33 (permission)

7.2 How WorkManager Abstracts the Platform

// WorkManager is the canonical example of Jetpack platform abstraction.
// It provides ONE API but dispatches to DIFFERENT system services:

// Internal delegation chain:
class WorkManagerImpl : WorkManager() {

    // Scheduler selection based on API level and device
    private fun createSchedulers(): List<Scheduler> {
        val schedulers = mutableListOf<Scheduler>()

        if (Build.VERSION.SDK_INT >= 23) {
            // API 23+: Use JobScheduler (preferred)
            // → Wraps IJobScheduler via JobScheduler Manager
            schedulers.add(SystemJobScheduler(context, this))
        }

        // Always add the greedy scheduler as fallback
        // → Uses AlarmManager + BroadcastReceiver
        schedulers.add(GreedyScheduler(context, this))

        return schedulers
    }
}

// SystemJobScheduler internally:
class SystemJobScheduler(context: Context, workManager: WorkManagerImpl) {
    private val jobScheduler =
        context.getSystemService(Context.JOB_SCHEDULER_SERVICE) as JobScheduler

    fun schedule(workSpec: WorkSpec) {
        // Convert WorkRequest constraints → JobInfo constraints
        val jobInfoBuilder = JobInfo.Builder(jobId, componentName)
            .setRequiredNetworkType(mapNetworkType(workSpec.constraints))
            .setRequiresCharging(workSpec.constraints.requiresCharging)
            .setRequiresBatteryNotLow(workSpec.constraints.requiresBatteryNotLow)
            // ...

        jobScheduler.schedule(jobInfoBuilder.build())
        // → IPC to JobSchedulerService in system_server
    }
}

7.3 The Abstraction Cost

// Jetpack abstractions are not free. Trade-offs:

// 1. Behavior differences from raw API
//    WorkManager adds its own retry/backoff logic on top of JobScheduler's
//    → A WorkRequest might execute at different times than a raw JobInfo
//    → WorkManager's 10-minute minimum periodic interval vs
//       JobScheduler's 15-minute minimum

// 2. Additional indirection
//    CameraX → Camera2 → CameraService (HAL) → kernel driver
//    vs.
//    Camera2 → CameraService (HAL) → kernel driver
//    → Extra layer means extra latency for camera open/close

// 3. Dependency size
//    WorkManager pulls in Room (for persistence), Lifecycle, etc.
//    → Adds ~300KB to APK size (after R8 optimization)

// 4. Debugging complexity
//    When a WorkManager job fails, you need to understand:
//    → WorkManager's internal state machine
//    → The underlying JobScheduler behavior
//    → Doze/App Standby bucket interactions
//    → The system service's rate limiting

// When to use raw system APIs instead of Jetpack:
// - You need precise control over timing (AlarmManager.setExact)
// - You need HAL-specific features (Camera2 manual controls)
// - APK size is critical (games, instant apps)
// - You only support recent API levels (no backward compat needed)

8. App Process and System Service Threading

Understanding the threading model of app-to-system-service calls is essential for avoiding ANRs and jank. Every system service call from an app is a synchronous Binder transaction by default.

8.2 Why This Causes ANRs

// Common ANR pattern: system service call on main thread

class MyActivity : AppCompatActivity() {

    override fun onResume() {
        super.onResume()

        // BAD: synchronous Binder call on main thread
        val pm = packageManager
        val packages = pm.getInstalledPackages(PackageManager.GET_META_DATA)
        // → IPC to PackageManagerService
        // → PMS acquires its internal lock
        // → PMS iterates all packages (could be 200+)
        // → Serializes result into Parcel
        // → Returns via Binder
        // Total time: 50-500ms depending on device and package count
        // If PMS lock is contended: could be SECONDS

        // BAD: multiple system service calls in sequence
        val am = getSystemService(ACTIVITY_SERVICE) as ActivityManager
        val memInfo = ActivityManager.MemoryInfo()
        am.getMemoryInfo(memInfo)  // IPC #1
        val processes = am.runningAppProcesses  // IPC #2
        val tasks = am.getRunningTasks(10)  // IPC #3
        // Each call is a separate Binder transaction
        // Total: 3 round-trips × latency per call
    }
}

// GOOD: move system service calls off the main thread

class MyActivity : AppCompatActivity() {

    override fun onResume() {
        super.onResume()

        lifecycleScope.launch(Dispatchers.Default) {
            val packages = packageManager
                .getInstalledPackages(PackageManager.GET_META_DATA)
            // Process on background thread

            withContext(Dispatchers.Main) {
                // Update UI with results
                updatePackageList(packages)
            }
        }
    }
}

// Some system services provide async/callback-based APIs
// to avoid blocking the calling thread:

// 1. ConnectivityManager — callback-based network monitoring
val cm = getSystemService(CONNECTIVITY_SERVICE) as ConnectivityManager

// Callback arrives on a Binder thread — post to main if needed

// 2. LocationManager — listener-based

    10f,    // min distance meters

            // Arrives on the calling thread's Looper (if it has one)
            // or on a Binder thread
        }
    },
    Looper.getMainLooper()  // Specify which Looper to deliver on
)

// 3. PackageManager — since Android 13, some queries return via callback
//    to avoid blocking on large result sets

// 4. SensorManager — always callback-based
val sm = getSystemService(SENSOR_SERVICE) as SensorManager
sm.registerListener(
    object : SensorEventListener {
        override fun onSensorChanged(event: SensorEvent) { /* async */ }
        override fun onAccuracyChanged(sensor: Sensor, accuracy: Int) {}

    sm.getDefaultSensor(Sensor.TYPE_ACCELEROMETER),
    SensorManager.SENSOR_DELAY_NORMAL,
    handler  // Deliver on this Handler's thread

L6 Insight — The Binder Thread Pool Starvation Cascade

system_server's 31 Binder threads serve all apps simultaneously. When a system service acquires a lock in its Binder thread handler and that lock is held by the main thread (which is itself waiting on another Binder call), you get a cross-thread deadlock that starves the entire thread pool. Here's the cascade: App A calls PackageManager.getInstalledPackages() → PMS Binder thread acquires mPackages lock → PMS main thread is doing a package scan holding the same lock → Binder thread blocks → 30 more Binder calls from other apps pile up → all 31 threads exhausted → every app on the device hangs on any system service call → Watchdog fires after 60s → runtime restart. This is why system service implementations must never hold locks across Binder calls and must minimize lock scope. Google's internal code review guidelines for system services mandate lock-ordering documentation and prohibit nested lock acquisition across service boundaries.

9.1 getSystemService() Before Application.onCreate()

// Some services aren't fully available during early app initialization.
// This catches developers who call getSystemService() in static
// initializers or ContentProvider.onCreate().

// ContentProvider.onCreate() runs BEFORE Application.onCreate():
//
// App process start sequence:
// 1. ActivityThread.handleBindApplication()
// 2. Application object created
// 3. ContentProvider.onCreate() for ALL declared providers
// 4. Application.onCreate()
//
// In ContentProvider.onCreate():
// ✅ getSystemService(ALARM_SERVICE) — works (AlarmManager is ready)
// ✅ getSystemService(CONNECTIVITY_SERVICE) — works

//     (SearchManager needs Activity context)

9.2 Binder Death and DeadObjectException

// If system_server crashes, every Binder proxy in every app becomes invalid.
// Any subsequent call throws DeadObjectException.

// But system_server crashes are rare. More common: HAL service death.
// Camera HAL, Media HAL, Sensor HAL — these run in separate processes
// and CAN crash independently.

// How Manager classes handle death:

// Pattern 1: Reconnect on death (CameraManager)
class CameraManager {
    private var cameraService: ICameraService? = null

    private val deathRecipient = IBinder.DeathRecipient {
        // HAL process died — clear cached proxy
        synchronized(lock) {
            cameraService = null
        }
        // Next camera operation will re-acquire the proxy
    }

    private fun getCameraService(): ICameraService {
        synchronized(lock) {
            cameraService?.let { return it }

            val binder = ServiceManager.getService("media.camera")
                ?: throw CameraAccessException(CameraAccessException.CAMERA_DISCONNECTED)

            binder.linkToDeath(deathRecipient, 0)
            val service = ICameraService.Stub.asInterface(binder)
            cameraService = service
            return service
        }
    }
}

// Pattern 2: Rethrow as RuntimeException (most Managers)
class AlarmManager {
    fun set(type: Int, triggerAtMillis: Long, operation: PendingIntent) {
        try {
            service.set(/* ... */)
        } catch (e: RemoteException) {
            // system_server dead — rethrow; app will be killed anyway
            throw e.rethrowFromSystemServer()
        }
    }
}

// RemoteException.rethrowFromSystemServer() wraps the exception:
// → If the dead service IS system_server: throws RuntimeException
//    (because the app process will be killed momentarily anyway)
// → If it's a different service: rethrows as-is

9.3 TransactionTooLargeException

// The Binder transaction buffer is 1MB per process (shared across
// ALL concurrent transactions). Exceeding it throws
// TransactionTooLargeException.

// Common triggers:
// 1. Passing large Bitmaps via Intent extras
//    → Solution: use FileProvider or content:// URIs
//
// 2. onSaveInstanceState() with too much data
//    → The saved state is passed to AMS via Binder
//    → Solution: keep saved state < 50KB, use ViewModel for large data
//
// 3. getInstalledPackages() with GET_META_DATA on a device with 200+ apps
//    → The result Parcel can exceed 1MB
//    → Solution: use GET_META_DATA only when needed, paginate if possible

// The 1MB limit is SHARED across all concurrent transactions:
// If Thread A is mid-transaction using 500KB,
// Thread B only has 500KB left for its transaction.
// This makes the exception non-deterministic and hard to reproduce.

// Debugging:
// adb shell dumpsys activity processes | grep "Transaction"
// Look for: "!@TransactionTooLargeException: data parcel size XXXXX bytes"

// Android 8+: the exception includes the parcel size in the message
// Android 11+: onSaveInstanceState() logs a warning if state > 32KB

9.4 System Service Rate Limiting

// System services protect themselves from misbehaving apps
// with various rate-limiting mechanisms:

// 1. Location — throttled by app standby bucket
//    Active bucket: full-rate updates
//    Working set: reduced rate
//    Frequent: further reduced
//    Rare: very infrequent
//    Restricted (Android 12+): no background location at all

// 2. Alarms — throttled starting Android 12
//    Apps can only set a limited number of exact alarms
//    Must declare SCHEDULE_EXACT_ALARM permission
//    System can defer non-exact alarms during Doze

// 3. Notifications — rate limited per channel
//    Posting too many notifications too quickly → some silently dropped
//    Toast rate limiting: max 5 toasts per 20 seconds per UID

// 4. JobScheduler — limited by app standby bucket
//    Active: 10 minutes of execution per 10 minutes
//    Working set: 10 minutes per 2 hours
//    Frequent: 10 minutes per 8 hours
//    Rare: 10 minutes per 24 hours

// 5. Broadcast delivery — prioritized
//    Foreground apps get broadcasts first
//    Background apps may have broadcasts deferred
//    Implicit broadcasts restricted to manifest receivers (Android 8+)

// 6. WakeLock acquisition — monitored
//    Long-held wakelocks trigger battery blame
//    System can force-release wakelocks from misbehaving apps

9.5 RECEIVER_EXPORTED / RECEIVER_NOT_EXPORTED (Android 13+)

// Android 13 (API 33) requires explicit export declaration
// for dynamically registered BroadcastReceivers.

// Before Android 13:

// → Implicitly exported — any app can send broadcasts to this receiver
// → Security risk: malicious apps can trigger your receiver

// Android 13+:
// Must specify RECEIVER_EXPORTED or RECEIVER_NOT_EXPORTED

// For receivers that should only receive system broadcasts:
context.registerReceiver(
    receiver,
    intentFilter,
    Context.RECEIVER_NOT_EXPORTED  // Only system + same-app can reach it
)

// For receivers that need broadcasts from other apps:
context.registerReceiver(
    receiver,
    intentFilter,
    Context.RECEIVER_EXPORTED  // Any app can send broadcasts to it
)

// If you don't specify on Android 13+:
// → SecurityException: "Need to specify RECEIVER_EXPORTED or
//    RECEIVER_NOT_EXPORTED when registering receiver"

// Protected broadcasts (ACTION_BOOT_COMPLETED, ACTION_BATTERY_CHANGED, etc.)
// are always safe with RECEIVER_NOT_EXPORTED because only the system
// can send them. Using RECEIVER_EXPORTED for these is unnecessary
// but not harmful.

// Compat library:
// ContextCompat.registerReceiver(context, receiver, filter, flags)
// Handles the flag requirement on 33+ and ignores it on older versions

L6 Insight — Mainline Module Updates as the New Fragmentation Vector

At Google scale, Mainline module updates are tested against a matrix of device configurations before rollout. But the combinatorial explosion of module versions creates a testing challenge that didn't exist before. Consider: a device running Android 13 with Conscrypt v42, Media v38, and Networking v45 might behave differently from the same Android 13 with Conscrypt v44, Media v38, and Networking v47. Google uses staged rollouts (1% → 10% → 50% → 100%) and automatic rollback (if crash rate exceeds threshold) to mitigate this. For app developers, the implication is that you can no longer reproduce a user's exact platform configuration by just knowing their Android version — you also need module versions. PackageManager.getModuleInfo() and Build.VERSION.SECURITY_PATCH together give you the full picture, and crash reporting tools (Firebase Crashlytics) now include module versions in crash reports.

  • getSystemService() resolves through SystemServiceRegistry → CachedServiceFetcher → ServiceManager → Binder proxy → Manager object
  • ServiceManager is handle 0 in the Binder driver — the universal directory; SELinux enforces access per service name
  • Manager classes wrap AIDL proxies with validation, caching, death handling, and backward compatibility

10. ServiceManager Cache Invalidation & HAL Death

ServiceManager caches Binder proxies in a static sCache (HashMap<String, IBinder>) to avoid repeated IPC to the native servicemanager daemon. But what happens when the target service dies?

10.1 System Service Death — Non-Issue

For system services running in system_server, a crash kills the entire process. Since all app processes depend on system_server, Zygote kills them all and re-forks system_server. When apps restart, they get fresh sCache entries — stale proxies are never observed because the entire Java heap is gone.

10.2 HAL Service Death — The Real Problem

HAL processes (camera, media, sensors) run in separate processes and can crash independently of system_server. When a HAL dies:

  1. servicemanager detects the death (the HAL's Binder node is destroyed in the kernel driver)
  2. servicemanager removes the registration — the next getService() call will block until the HAL restarts and re-registers
  3. But the app's cached Manager object still holds the dead Binder proxy
  4. Any call on that proxy throws DeadObjectException
// The death notification mechanism

val binder: IBinder = ServiceManager.getService("media.camera")

// Register for death notification
binder.linkToDeath(object : IBinder.DeathRecipient {
    override fun binderDied() {
        // HAL process crashed — the proxy is now dead
        // Clear any cached references
        synchronized(lock) {
            cachedCameraService = null
        }
        // Next operation will re-acquire via getService()
    }
}, 0)

// Recovery pattern for app code:
fun safeCameraOperation() {
    try {
        cameraManager.openCamera(cameraId, callback, handler)
    } catch (e: DeadObjectException) {
        // HAL died — clear cached manager, re-acquire
        cameraManager = context.getSystemService(Context.CAMERA_SERVICE)
            as CameraManager
        // Retry after a delay (HAL needs time to restart)
        handler.postDelayed({ safeCameraOperation() }, 1000)
    }
}
L6 Insight — Built-in Reconnection in Manager Classes

Some Manager classes (e.g., CameraManager, MediaCodecList) have built-in reconnection logic — they register DeathRecipient on the HAL's IBinder and automatically re-bind when the HAL restarts. This is why CameraManager.openCamera() can throw CameraAccessException(CAMERA_DISCONNECTED) instead of a raw DeadObjectException — the Manager detects the death and translates it into a domain-specific exception. Understanding this pattern is critical for designing robust Manager wrappers: always register DeathRecipient, always clear cached proxies on death, and always provide a domain-specific error path for clients.

11. ContextImpl Double-Init Race

CachedServiceFetcher.getService() uses a gate mechanism to prevent double initialization when multiple threads call getSystemService() simultaneously for the same service.

11.1 The Gate Mechanism

// ContextImpl maintains a per-service-index state array:
// mServiceInitializationStateArray: AtomicInteger[]

// States:
// STATE_UNINITIALIZED (0) — no thread has started creating this service
// STATE_INITIALIZING  (1) — a thread is currently in createService()
// STATE_READY         (2) — service is created and cached
// STATE_NOT_FOUND     (3) — createService() threw an exception

// The race scenario:
//
// Thread A                          Thread B
// ────────                          ────────
// getService("alarm")               getService("alarm")
//   │                                 │
//   ▼                                 ▼
// CAS(UNINITIALIZED → INITIALIZING)  CAS fails (already INITIALIZING)
//   │                                 │
//   ▼                                 ▼
// createService(ctx)                 spin-wait: while(INITIALIZING) yield()
//   │                                 │
//   ▼                                 │ (blocked)
// cache[index] = manager             │
// state = READY                       │
//   │                                 ▼
//   │                               state == READY → return cache[index]
//   ▼
// return manager

11.2 The Exception Edge Case

If Thread A's createService() throws an exception, the gate transitions to STATE_NOT_FOUND. Thread B, which was spin-waiting, unblocks and finds cache[index] is null — it returns null to the caller. Subsequent calls will also see STATE_NOT_FOUND and return null without retrying.

// What happens when createService() fails:
//
// Thread A: CAS(UNINITIALIZED → INITIALIZING) ✓
// Thread A: createService() throws ServiceNotFoundException
// Thread A: state = NOT_FOUND (in catch block)
// Thread A: exception propagates to caller
//
// Thread B: was spinning on INITIALIZING
// Thread B: sees NOT_FOUND → exits spin
// Thread B: cache[index] is null → returns null
//
// Thread C (later): sees NOT_FOUND → returns null immediately
//
// The service is permanently "not found" for this Context.
// Only creating a new Context would reset the state.
L6 Insight — Concurrent Initialization Pattern

This is a classic double-checked locking pattern adapted for service initialization with three states instead of two. The compareAndSet on AtomicInteger replaces synchronized for the fast path (cache hit), while the spin-wait with Thread.yield() handles the contention case. The trade-off: spin-waiting wastes CPU cycles but avoids the overhead of a wait()/notify() mechanism for what is typically a very short initialization. Understanding this pattern demonstrates knowledge of concurrent initialization beyond basic synchronization — interviewers look for awareness of CAS operations, spin-wait trade-offs, and failure-state propagation in concurrent systems.

12. @SystemApi Enforcement

@SystemApi methods exist in the real framework on every device but are deliberately excluded from the public SDK stubs that apps compile against. The enforcement is asymmetric — strong at compile time, weaker at runtime.

12.1 Compile-Time Enforcement

// The public SDK (android.jar) is a stub JAR containing only:
//   - Public methods (no annotation)
//   - @SystemApi methods are EXCLUDED from the stubs
//
// When your app compiles against android.jar:
//   ActivityManager.forceStopPackage("com.example")
//   → Compile error: "Cannot resolve method 'forceStopPackage'"
//   → The method simply doesn't exist in the stubs
//
// For system apps that need @SystemApi access:
//   - Compile against the full framework (system-api stubs)
//   - Must be installed in /system/priv-app/
//   - Must be signed with the platform key (or hold signature permissions)

12.2 Runtime Enforcement

// At runtime, the real framework classes contain ALL methods:
//   - Public, @SystemApi, @TestApi, and @hide methods all exist
//
// An app using reflection CAN call @SystemApi methods:
//   val method = ActivityManager::class.java
//       .getDeclaredMethod("forceStopPackage", String::class.java)
//   method.invoke(activityManager, "com.example")
//
// BUT: hidden API restrictions (Android 9+) block this:
//   - @SystemApi methods are on the greylist/blacklist
//   - ART intercepts the reflection call
//   - Throws NoSuchMethodException for blacklisted methods
//
// Exceptions — who CAN access @SystemApi at runtime:
//   1. System apps in /system/priv-app/
//   2. Apps signed with the platform signing key
//   3. Apps with HiddenApiEnforcementPolicy.DISABLED
//   4. Shell (UID 2000) — for adb commands

12.3 priv-app Permissions

// Privileged apps must also have their permissions whitelisted:
// /etc/permissions/privapp-permissions-platform.xml
//
// <privapp-permissions package="com.android.settings">
//     <permission name="android.permission.MANAGE_USERS"/>
//     <permission name="android.permission.WRITE_SECURE_SETTINGS"/>
//     <permission name="android.permission.FORCE_STOP_PACKAGES"/>
// </privapp-permissions>
//
// Without this whitelist entry, even a priv-app can't use the permission.
// Android 9+ enforces this strictly — missing entries cause boot-time errors.
L6 Insight — Asymmetric Enforcement

The enforcement is intentionally asymmetric: compile-time enforcement is strong (methods don't exist in stubs — you literally can't reference them), but runtime enforcement relies on hidden API restrictions which have exemptions for privileged apps. This design allows the platform to expose powerful APIs to system apps while keeping them invisible to third-party developers. The implication for app developers: don't rely on reflection to access @SystemApi methods — the hidden API restrictions will block you, and the greylist/blacklist evolves with each Android release. For platform engineers: always pair @SystemApi with a permission check in the service implementation — the annotation alone is not a security boundary.

13. veridex Migration Workflow

veridex is Google's static analysis tool for detecting hidden API usage in APKs. It scans dex bytecode for references to non-SDK APIs and reports them with severity levels.

13.1 Running veridex

// Basic usage:
$ ./veridex --dex-file=app.apk

// Output:
// #1: Linking greylist Landroid/app/ActivityThread;
//         →currentApplication()Landroid/app/Application;
//     in: Lcom/example/MyUtils;→getApp()Landroid/app/Application;
//
// #2: Linking blacklist Landroid/view/View;→mListenerInfo:
//         Landroid/view/View$ListenerInfo;
//     in: Lcom/example/ClickInterceptor;→intercept(Landroid/view/View;)V
//
// Summary:
//   2 greylist references
//   1 blacklist reference
//   0 greylist-max-o references

// Severity levels:
// ┌─────────────────────┬───────────────────────────────────────────┐
// │ Level               │ Meaning                                   │
// ├─────────────────────┼───────────────────────────────────────────┤
// │ whitelist           │ Public SDK — safe                         │
// │ greylist            │ Warning — may be blocked in future        │
// │ greylist-max-o      │ Blocked for targetSdk ≥ 26 (Oreo)        │
// │ greylist-max-p      │ Blocked for targetSdk ≥ 28 (Pie)         │
// │ greylist-max-q      │ Blocked for targetSdk ≥ 29 (Q)           │
// │ blacklist           │ Always blocked — throws exception         │
// └─────────────────────┴───────────────────────────────────────────┘

13.2 Migration Workflow

// Step 1: Run veridex on your APK
$ ./veridex --dex-file=app-release.apk > veridex-report.txt

// Step 2: Triage the report
//   - blacklist: MUST fix — these will crash on affected API levels
//   - greylist-max-*: MUST fix before raising targetSdk
//   - greylist: SHOULD fix — will be blocked in future releases

// Step 3: Replace hidden API calls with public alternatives
//   ActivityThread.currentApplication()
//     → Pass Application via DI, or use ProcessLifecycleOwner
//
//   View.mListenerInfo (field access)
//     → View.getOnClickListener() (public since API 15)
//
//   Telephony.Sms.moveMessageToFolder()
//     → ContentResolver operations with Telephony.Sms.CONTENT_URI

// Step 4: If no public alternative exists
//   → File a feature request on the Android issue tracker
//   → Google tracks these to prioritize SDK API additions

// Step 5: Test with strict enforcement
$ adb shell settings put global hidden_api_policy 1
// 0 = disabled, 1 = just warn, 2 = dark greylist enforced, 3 = all enforced

// The appcompat.sh wrapper script:
$ appcompat.sh --dex-file=app.apk --imprecise
// Formats veridex output for easier triage with categorized sections
L6 Insight — Google's Play Store Enforcement

Google runs veridex on all Play Store apps and sends pre-launch warnings to developers whose apps use hidden APIs that will be blocked in the next Android release. This is how they drive migration before tightening restrictions — developers get 6-12 months of warnings before an API moves from greylist to blacklist. For large apps with extensive hidden API usage, the migration can take multiple release cycles. The strategic approach: run veridex as part of your CI pipeline, fail the build on any blacklist reference, and treat greylist references as tech debt to be resolved before the next targetSdk bump.

14. Android 14+ Platform Changes

Each Android version tightens the contract between apps and the platform. Understanding the evolution helps predict future restrictions and design forward-compatible apps.

14.1 Key Platform Changes (Android 13–15)

Change Version Impact Migration
POST_NOTIFICATIONS permission Android 13 (API 33) Apps must request runtime permission to post notifications Add permission request flow; handle denial gracefully
RECEIVER_EXPORTED / RECEIVER_NOT_EXPORTED Android 13–14 Dynamic receivers must declare export status; Android 14 enforces for all apps Audit all registerReceiver() calls; add the flag
Foreground service types Android 14 (API 34) Must declare foregroundServiceType in manifest Add type attribute: camera, location, dataSync, etc.
USE_EXACT_ALARM permission Android 14 Alarm clock apps can use exact alarms without user approval Use instead of SCHEDULE_EXACT_ALARM for clock/timer apps
Photo Picker Android 13+ ACTION_PICK_IMAGES replaces READ_MEDIA_IMAGES for selecting photos Use PickVisualMedia ActivityResult contract
Edge-to-edge enforcement Android 15 (API 35) Apps targeting 15 must handle insets — no default padding Use WindowInsets API; test with WindowInsetsCompat
Predictive back Android 14–15 System back gesture shows preview; apps must opt in Use OnBackPressedCallback; enable android:enableOnBackInvokedCallback

14.2 The Pattern of Tightening

// The platform-app contract tightens in a predictable pattern:
//
// Phase 1: New behavior introduced, opt-in via targetSdk
//   Android 13: POST_NOTIFICATIONS required for targetSdk 33+
//   Android 13: RECEIVER_EXPORTED optional for targetSdk 33
//
// Phase 2: Enforcement broadened
//   Android 14: RECEIVER_EXPORTED required for ALL dynamic receivers
//   Android 14: foregroundServiceType required for targetSdk 34+
//
// Phase 3: No escape — enforced for all apps
//   Android 15: edge-to-edge enforced for targetSdk 35+
//   Future: likely enforced for all apps in Android 16+
//
// Prediction strategy:
//   1. Watch the "behavior changes for all apps" section in each release
//   2. Anything that's "opt-in for targetSdk N" will become mandatory
//      for targetSdk N+1 or N+2
//   3. Design your app to handle the restriction NOW, even if you
//      haven't raised targetSdk yet
L6 Insight — Forward-Compatible Design

Each Android version tightens the platform-app contract — understanding the evolution helps predict future restrictions. The pattern is consistent: introduce a new restriction gated on targetSdkVersion, give developers 1-2 releases to migrate, then enforce for all apps. Staff engineers should design apps that are forward-compatible by default: always declare export status on receivers, always specify foreground service types, always handle insets explicitly, and always request permissions before using restricted features. This proactive approach means your app doesn't break when you raise targetSdkVersion — a common source of regressions in large codebases where the targetSdk bump touches hundreds of behaviors simultaneously.
