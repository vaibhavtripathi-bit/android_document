05 — Other Critical System Services

PowerManagerService, AlarmManagerService, JobSchedulerService, NotificationManagerService, ConnectivityService, LocationManagerService, and StorageManagerService — the services that govern power, scheduling, notifications, networking, location, and storage across the entire platform

1. PowerManagerService — The Power State Machine

PowerManagerService (PMS — not to be confused with PackageManagerService) controls the device's power state, manages wake locks, and coordinates with DeviceIdleController (Doze) and BatteryStatsService to enforce power policy. It is the single authority that decides whether the screen is on, the CPU stays awake, or the device enters deep sleep.

Power State Machine

  Power States (PowerManagerService internal state machine)
  ═══════════════════════════════════════════════════════════

  ┌──────────┐   user activity    ┌──────────┐
  │  ASLEEP  │ ──────────────────→│  AWAKE   │
  │ (suspend)│                    │(screen on│
  └────┬─────┘                    │ CPU on)  │
       ↑                          └────┬─────┘
       │                               │ screen timeout
       │                               ▼
       │                          ┌──────────┐
       │   dream ends / timeout   │ DREAMING │
       │ ←────────────────────────│(Daydream/│
       │                          │ screensav)│
       │                          └────┬─────┘
       │                               │ dream ends
       │                               ▼
       │                          ┌──────────┐
       │   no wake locks held     │   DOZE   │
       └──────────────────────────│(low-power│
                                  │ idle)    │
                                  └──────────┘

  Transitions are driven by updatePowerStateLocked(), which
  evaluates ALL inputs on every state change trigger.

The Central Method: updatePowerStateLocked()

// Source: frameworks/base/services/core/java/com/android/server/power/PowerManagerService.java

// updatePowerStateLocked() is called on EVERY power-relevant event:
// - Wake lock acquired/released
// - User activity (touch, button press)
// - Proximity sensor state change
// - Battery state change
// - Display policy change
// - Device admin policy change

// The method evaluates inputs in a strict order:
private fun updatePowerStateLocked() {
    // Phase 1: Determine desired wakefulness
    //   Inputs: user activity timeout, wake locks, stay-on-while-plugged,
    //           proximity sensor, lid switch, device admin policy
    updateWakefulnessLocked(dirtyPhase1)

    // Phase 2: Update display power state
    //   Sends request to DisplayPowerController:
    //   - screen brightness (auto/manual)
    //   - screen state (ON, DIM, DOZE, OFF)
    //   - proximity sensor override
    updateDisplayPowerStateLocked(dirtyPhase2)

    // Phase 3: Update dream state
    //   If wakefulness == DREAMING → start DreamManagerService
    //   If wakefulness changed away from DREAMING → stop dream
    updateDreamLocked(dirtyPhase3)

    // Phase 4: Send notifications
    //   Broadcasts ACTION_SCREEN_ON / ACTION_SCREEN_OFF
    //   Updates BatteryStatsService with screen state
    finishWakefulnessChangeIfNeededLocked()

    // Phase 5: Update suspend blocker
    //   If any PARTIAL_WAKE_LOCK held → acquire CPU suspend blocker
    //   Otherwise → release suspend blocker → kernel can enter suspend
    updateSuspendBlockerLocked()
}

Wake Lock Types

Wake Lock CPU Screen Status
PARTIAL_WAKE_LOCK ON Off allowed Only valid type for background work
SCREEN_DIM_WAKE_LOCK ON Dim Deprecated API 17 — use FLAG_KEEP_SCREEN_ON
SCREEN_BRIGHT_WAKE_LOCK ON Bright Deprecated API 17 — use FLAG_KEEP_SCREEN_ON
FULL_WAKE_LOCK ON Bright + keyboard Deprecated API 17

Wake Lock Internals

// When an app acquires a wake lock:
// 1. App: PowerManager.WakeLock.acquire()
// 2. Binder call → PowerManagerService.acquireWakeLockInternal()
// 3. PMS creates a WakeLock record:
class WakeLock {
    val lock: IBinder          // Binder token for death tracking
    val flags: Int             // PARTIAL, SCREEN_DIM, etc.
    val tag: String            // Developer-provided tag (e.g., "MyDownloadWakeLock")
    val ownerUid: Int          // Calling app's UID
    val ownerPid: Int          // Calling app's PID
    val workSource: WorkSource // Optional — attribute to a different UID
    var isDisabled: Boolean    // Disabled by DeviceIdleController in Doze
}

// 4. PMS calls updatePowerStateLocked() → re-evaluates state
// 5. If PARTIAL_WAKE_LOCK → acquires native suspend blocker
//    (writes "PowerManagerService.WakeLocks" to /sys/power/wake_lock)
// 6. Kernel stays awake as long as any suspend blocker is held

// Death tracking:
// PMS links to the app's Binder token via linkToDeath().
// If the app process dies, PMS automatically releases all its wake locks.
// This prevents leaked wake locks from dead processes.

Doze Mode Interaction

// PowerManagerService → DeviceIdleController (Doze) coordination:
//
// DeviceIdleController monitors:
// - Screen off + stationary + on battery → enters IDLE mode (Doze)
// - Periodic maintenance windows allow brief work bursts
//
// When Doze activates:
// 1. DeviceIdleController tells PowerManagerService to disable non-exempt wake locks
// 2. PMS sets WakeLock.isDisabled = true for non-whitelisted apps
// 3. AlarmManagerService defers non-exempt alarms
// 4. JobSchedulerService defers non-exempt jobs
// 5. Network access is blocked for non-exempt apps
//
// Exemptions (can still run in Doze):
// - Apps on the battery optimization whitelist
// - High-priority FCM messages (trigger a temporary maintenance window)
// - setAlarmClock() alarms (user-visible alarm clocks)
// - setExactAndAllowWhileIdle() (limited to 1 per 9 minutes in Doze)
// - Foreground services with active notification

// App Standby Buckets (Android 9+):
// PowerManagerService + UsageStatsService assign apps to buckets:
//   ACTIVE       → app in foreground, no restrictions
//   WORKING_SET  → recently used, mild restrictions
//   FREQUENT     → used regularly, deferred jobs/alarms
//   RARE         → seldom used, heavily deferred
//   RESTRICTED   → (Android 12+) never used, almost no background work
//
// Bucket determines: job frequency, alarm frequency, network access,
// FCM priority, and foreground service launch restrictions
Battery Drain Root Cause

Wake lock bugs are the #1 cause of battery drain in production Android apps. The most common pattern: acquiring a PARTIAL_WAKE_LOCK in a BroadcastReceiver or Service, then failing to release it on error paths. Use dumpsys power to list all held wake locks, and dumpsys batterystats --charged to identify the top wake lock holders. WorkManager's internal wake lock management is one of the key reasons Google recommends it over raw AlarmManager + Service patterns.

2. AlarmManagerService — Alarm Scheduling

AlarmManagerService manages time-based event delivery. It interacts directly with the kernel's RTC and timer hardware, and coordinates with DeviceIdleController to batch and defer alarms for power efficiency.

Alarm Types

Type Time Base Wakes Device? Use Case
RTC Wall clock (UTC) No UI updates when screen is on
RTC_WAKEUP Wall clock (UTC) Yes Calendar reminders, scheduled syncs
ELAPSED_REALTIME Since boot (incl. sleep) No Relative delays, polling intervals
ELAPSED_REALTIME_WAKEUP Since boot (incl. sleep) Yes Periodic background tasks

The Precision Hierarchy

// Alarm precision from LEAST to MOST exact:

// 1. set() — inexact, system can defer up to the full batch window
//    AlarmManager.set(RTC_WAKEUP, triggerTime, pendingIntent)
//    The system batches this with other alarms to reduce wake-ups.
//    Actual delivery: triggerTime to triggerTime + batchWindow

// 2. setWindow() — bounded inexact
//    AlarmManager.setWindow(RTC_WAKEUP, triggerTime, windowMs, pendingIntent)
//    Fires within [triggerTime, triggerTime + windowMs]

// 3. setExact() — exact, but deferred in Doze
//    AlarmManager.setExact(RTC_WAKEUP, triggerTime, pendingIntent)
//    Fires at exactly triggerTime... unless device is in Doze.
//    Android 12+: requires SCHEDULE_EXACT_ALARM permission.

// 4. setExactAndAllowWhileIdle() — exact, fires in Doze (throttled)
//    Fires even in Doze, but limited to 1 per 9 minutes per app.
//    Intended for messaging apps that need timely delivery.

// 5. setAlarmClock() — exact, fires in Doze, shows alarm icon
//    AlarmManager.setAlarmClock(AlarmClockInfo(triggerTime, showIntent), pendingIntent)
//    Always fires on time. System shows alarm icon in status bar.
//    The ONLY alarm type that is truly unrestricted.
//    Intended for user-facing alarm clock apps.

Alarm Batching Internals

// Source: frameworks/base/services/core/java/com/android/server/alarm/AlarmManagerService.java

// AlarmManagerService groups alarms into Batch objects:
class Batch {
    val alarms: ArrayList<Alarm>
    val start: Long    // earliest delivery time in this batch
    val end: Long      // latest delivery time in this batch
    val isStandalone: Boolean  // true for exact alarms (own batch)
}

// Batching algorithm:
// 1. Inexact alarms: system computes a batch window based on:
//    - futurity (how far in the future) — longer = wider window
//    - fuzz factor: 75% of the interval for repeating alarms
// 2. The system finds an existing batch whose [start, end] overlaps
//    with the new alarm's [triggerTime, triggerTime + window]
// 3. If overlap found → add to that batch (coalesced wake-up)
// 4. If no overlap → create new batch
// 5. Exact alarms get their own standalone batch (no coalescing)

// The kernel timer:
// AlarmManagerService programs the next wake-up via:
//   /dev/alarm (legacy) or timerfd_create() + CLOCK_BOOTTIME_ALARM
// Only ONE kernel timer is set — for the earliest pending batch.
// When it fires, AlarmManagerService delivers all alarms in that batch,
// then programs the next timer for the next batch.

// Android 12+ SCHEDULE_EXACT_ALARM:
// setExact() and setExactAndAllowWhileIdle() now require:
//   <uses-permission android:name="android.permission.SCHEDULE_EXACT_ALARM" />
// This is a special permission: auto-granted on install for most apps,
// but user can revoke it in Settings → Apps → Alarms & reminders.
// Apps must check AlarmManager.canScheduleExactAlarms() before use.
// USE_EXACT_ALARM (Android 14+): for alarm clock apps, cannot be revoked.

The Alarm → WorkManager Chain

// The common production pattern for reliable background work:
//
// AlarmManager (trigger)
//   → PendingIntent → BroadcastReceiver
//     → enqueue WorkManager OneTimeWorkRequest
//       → WorkManager acquires wake lock
//         → Worker.doWork() executes
//           → WorkManager releases wake lock
//
// Why not just use AlarmManager directly?
// 1. AlarmManager delivers a broadcast — you get ~10s to complete work
// 2. No automatic retry on failure
// 3. No constraint awareness (network, charging, etc.)
// 4. No chaining or dependency management
// 5. Wake lock management is manual and error-prone
//
// WorkManager wraps JobScheduler (API 23+) or AlarmManager + BroadcastReceiver
// (API 14-22) and handles all of the above automatically.

3. JobSchedulerService — Background Job Management

JobSchedulerService is the system's constraint-aware background work scheduler. It evaluates job constraints (network, charging, idle, storage), manages concurrency, and coordinates with Doze and App Standby Buckets to balance app needs against battery life.

JobInfo Constraints

// JobInfo.Builder defines the constraints for a job:
val jobInfo = JobInfo.Builder(JOB_ID, ComponentName(context, MyJobService::class.java))
    .setRequiredNetworkType(JobInfo.NETWORK_TYPE_UNMETERED) // WiFi only
    .setRequiresCharging(true)                              // plugged in
    .setRequiresDeviceIdle(true)                            // device idle
    .setRequiresStorageNotLow(true)                         // storage OK
    .addTriggerContentUri(                                  // content change trigger
        JobInfo.TriggerContentUri(
            MediaStore.Images.Media.EXTERNAL_CONTENT_URI,
            JobInfo.TriggerContentUri.FLAG_NOTIFY_FOR_DESCENDANTS
        )
    )
    .setPeriodic(15 * 60 * 1000L)         // minimum 15 minutes
    .setPersisted(true)                    // survive reboot (needs RECEIVE_BOOT_COMPLETED)
    .setBackoffCriteria(30_000L, JobInfo.BACKOFF_POLICY_EXPONENTIAL)
    .build()

// Constraint evaluation:
// JobSchedulerService continuously monitors constraint state via:
// - ConnectivityService → network state
// - BatteryService → charging state
// - DeviceIdleController → idle state
// - StorageManagerService → storage state
// - ContentObserver → content URI changes
//
// When ALL constraints for a job are satisfied → job becomes READY
// JobConcurrencyManager decides when to actually start it

JobStore — Persistence

// Source: frameworks/base/services/core/java/com/android/server/job/JobStore.java
// Persists to: /data/system/job/jobs.xml

// JobStore persists all scheduled jobs to XML so they survive reboot.
// Only jobs with setPersisted(true) are written to disk.
//
// On boot, JobSchedulerService reads jobs.xml and re-evaluates constraints.
// Jobs whose constraints are immediately met may fire shortly after boot.
//
// XML structure:
// <job-info version="0">
//   <job jobid="42" package="com.example.app" class=".MyJobService"
//        uid="10123" priority="0" flags="0">
//     <constraints connectivity="unmetered" charging="true" />
//     <periodic period="900000" flex="300000" />
//     <extras>...PersistableBundle...</extras>
//   </job>
// </job-info>
//
// Limitation: extras must be PersistableBundle (primitives + String only)
// because they're serialized to XML. No Parcelable, no Serializable.

JobConcurrencyManager — Slot Allocation

// JobConcurrencyManager controls how many jobs run simultaneously.
// Source: frameworks/base/services/core/java/com/android/server/job/JobConcurrencyManager.java

// Concurrency slots are categorized by work type:
//   TOP       → job from the foreground app (highest priority)
//   FGS       → job from an app with a foreground service
//   EJ        → expedited job
//   BG        → regular background job
//   BGUSER    → background job from a user-initiated work request

// System-wide limits (vary by device, typical values):
//   Total concurrent jobs: ~16
//   Per-app limit: ~4-8 depending on work type and standby bucket
//   TOP app gets priority allocation

// When a slot opens:
// 1. JobConcurrencyManager picks the highest-priority READY job
// 2. Creates a JobServiceContext → binds to the app's JobService
// 3. Calls onStartJob(params) via Binder
// 4. Starts a timeout timer (default 10 minutes, 4 min for expedited)
// 5. If job doesn't call jobFinished() before timeout → onStopJob()

// Expedited Jobs (Android 12+):
// JobInfo.Builder.setExpedited(true)
// - Run immediately regardless of Doze/Standby
// - Shorter execution window (1-4 minutes depending on bucket)
// - Quota-limited per app
// - Replaced the deprecated setImportantWhileForeground()
// - WorkManager maps OneTimeWorkRequest with OutOfQuotaPolicy to this

Doze and Standby Bucket Interaction

// JobScheduler respects both Doze and App Standby Buckets:

// Doze impact:
// - IDLE state: all non-exempt jobs deferred until maintenance window
// - Maintenance windows: system briefly exits Doze to run pending work
// - Expedited jobs can run in Doze (within quota)

// App Standby Bucket impact on job frequency:
//   ACTIVE       → no restrictions
//   WORKING_SET  → deferred up to 2 hours
//   FREQUENT     → deferred up to 8 hours
//   RARE         → deferred up to 24 hours
//   RESTRICTED   → deferred up to 48 hours, max 1 job per day

// WorkManager uses JobScheduler under the hood (API 23+):
// - OneTimeWorkRequest → one-shot JobInfo
// - PeriodicWorkRequest → periodic JobInfo (min 15 min)
// - Constraints map directly to JobInfo constraints
// - WorkManager adds: chaining, retry, unique work, LiveData observation

// Debugging:
// adb shell dumpsys jobscheduler                    // full state
// adb shell cmd jobscheduler run -f com.example 42  // force-run job ID 42
// adb shell cmd jobscheduler get-job-state com.example 42

Notification Channels (Android 8+)

// Apps MUST assign every notification to a channel (targetSdk 26+).

// Creating a channel:

// - If notification references a non-existent channel → silently dropped
// - User can modify channel importance, sound, vibration in Settings
// - App CANNOT programmatically change a channel after creation
//   (only name and description can be updated)
// - Deleting and recreating a channel with same ID preserves user settings

NotificationRecord and Ranking

// NMS wraps each posted notification in a NotificationRecord:
// Source: frameworks/base/services/core/java/com/android/server/notification/NotificationRecord.java

class NotificationRecord {
    val sbn: StatusBarNotification    // the notification + metadata
    val channel: NotificationChannel  // resolved channel
    val importanceExplanation: String // why this importance was assigned
    var importance: Int               // final computed importance
    var isIntercepted: Boolean        // suppressed by DND?
    var isSeen: Boolean               // user has seen it?
    var stats: NotificationStats      // interaction tracking

    // Ranking signals (evaluated by RankingHelper / NotificationSignalExtractors):
    // 1. Channel importance (user-controlled)
    // 2. Notification priority (app-set, capped by channel)
    // 3. Category (CATEGORY_MESSAGE, CATEGORY_ALARM, etc.)
    // 4. Person association (conversations get boosted)
    // 5. Conversation shortcut (Android 11+, bubbles)
    // 6. Freshness (newer notifications rank higher)
    // 7. Ongoing flag (sticky notifications rank lower)
}

// Importance levels:
//   IMPORTANCE_NONE (0)    → blocked, not shown
//   IMPORTANCE_MIN (1)     → no sound, no peek, collapsed in shade
//   IMPORTANCE_LOW (2)     → no sound, no peek, shown in shade
//   IMPORTANCE_DEFAULT (3) → sound, no peek, shown in shade
//   IMPORTANCE_HIGH (4)    → sound + heads-up (peek) display

// Heads-up trigger conditions:
// - importance >= HIGH, OR
// - fullScreenIntent is set (e.g., incoming call), OR
// - notification is from a foreground service that just started

Rate Limiting and Abuse Prevention

// NMS protects the user from notification spam:

// 1. Per-app rate limiting:
//    Default: 50 notifications per second (toast: 5 per second)
//    Exceeding the limit → notifications silently dropped
//    System apps and foreground apps get higher limits

// 2. Notification grouping (Android 7+):
//    If an app posts 4+ notifications without a group → NMS auto-groups them
//    Apps should use Notification.Builder.setGroup() proactively

// 3. Snoozing:
//    User can snooze notifications → NMS stores snooze duration
//    Re-posts the notification after the snooze period

// 4. Notification listeners rate limiting:
//    NotificationListenerService callbacks are batched
//    Listeners that process too slowly get disconnected

// 5. Do Not Disturb (ZenMode):
//    ZenModeHelper evaluates DND rules:
//    - Priority mode: only alarms, reminders, starred contacts
//    - Total silence: nothing
//    - Alarms only: only CATEGORY_ALARM
//    NMS checks ZenMode before buzzing/beeping/blinking

NetworkCapabilities and NetworkRequest

// NetworkCapabilities describes what a network can do:
class NetworkCapabilities {
    // Transport types:
    //   TRANSPORT_WIFI, TRANSPORT_CELLULAR, TRANSPORT_VPN,
    //   TRANSPORT_ETHERNET, TRANSPORT_BLUETOOTH

    // Capabilities (boolean flags):
    //   NET_CAPABILITY_INTERNET          → can reach the internet
    //   NET_CAPABILITY_NOT_METERED       → unmetered (WiFi, typically)
    //   NET_CAPABILITY_VALIDATED         → internet connectivity verified
    //   NET_CAPABILITY_NOT_VPN           → not a VPN
    //   NET_CAPABILITY_NOT_RESTRICTED    → not a restricted network
    //   NET_CAPABILITY_TRUSTED           → trusted network (not captive portal)
    //   NET_CAPABILITY_NOT_ROAMING       → not roaming
    //   NET_CAPABILITY_FOREGROUND        → foreground network (not background)
    //   NET_CAPABILITY_TEMPORARILY_NOT_METERED → carrier promo, etc.

    // Bandwidth estimates:
    val linkUpstreamBandwidthKbps: Int
    val linkDownstreamBandwidthKbps: Int
}

// Apps request networks with specific capabilities:
val request = NetworkRequest.Builder()
    .addCapability(NetworkCapabilities.NET_CAPABILITY_INTERNET)
    .addCapability(NetworkCapabilities.NET_CAPABILITY_NOT_METERED)
    .addTransportType(NetworkCapabilities.TRANSPORT_WIFI)
    .build()

connectivityManager.registerNetworkCallback(request, object : ConnectivityManager.NetworkCallback() {
    override fun onAvailable(network: Network) {
        // Unmetered WiFi network is available — safe to do large downloads
    }
    override fun onCapabilitiesChanged(network: Network, caps: NetworkCapabilities) {
        val isValidated = caps.hasCapability(NetworkCapabilities.NET_CAPABILITY_VALIDATED)
        // Network capabilities changed (e.g., lost internet validation)
    }
    override fun onLost(network: Network) {
        // Network lost — switch to offline mode or queue work
    }
})

Network Scoring and Selection

// ConnectivityService picks the "best" network using a scoring system:
//
// Each NetworkAgent reports a score (int):
//   WiFi:     60 (connected) / 56 (no internet)
//   Cellular: 50 (connected) / 10 (emergency only)
//   VPN:      101 (always preferred when active)
//   Ethernet: 70
//
// Selection algorithm:
// 1. For each NetworkRequest, find all networks that satisfy capabilities
// 2. Among satisfying networks, pick the one with highest score
// 3. That network becomes the "default" for matching requests
// 4. Routing table updated: default route → selected network's interface
//
// Score adjustments:
// - Validated (internet works): +0 (but unvalidated gets penalized)
// - Captive portal detected: score drops → cellular may take over
// - Network agent can update score dynamically (e.g., WiFi signal drops)
//
// Multi-network routing (Android 5+):
// Apps can bind to a specific network:
//   network.bindSocket(socket)
//   network.openConnection(url)
// This bypasses the default network selection — useful for VPN split tunneling

VPN Integration

// VPN apps use VpnService to create a TUN interface:
// 1. App calls VpnService.Builder.establish() → returns a ParcelFileDescriptor
// 2. System creates a TUN device (tun0)
// 3. VpnService registers a NetworkAgent with ConnectivityService
// 4. ConnectivityService routes traffic through the VPN's TUN interface
// 5. VPN app reads packets from the TUN fd, encrypts, sends to VPN server

// ConnectivityService VPN handling:
// - VPN NetworkAgent has score 101 → always preferred over WiFi/cellular
// - Per-app VPN: VpnService.Builder.addAllowedApplication() / addDisallowedApplication()
// - Always-on VPN: Settings → Network → VPN → always-on
//   ConnectivityService blocks all non-VPN traffic until VPN connects
// - Lockdown mode: no traffic at all if VPN disconnects (even to local network)

// DNS resolution (Android 10+):
// DnsResolver API replaces the old InetAddress.getByName():
//   DnsResolver.getInstance().query(network, hostname, ...)
// Private DNS (DNS-over-TLS):
//   Settings → Network → Private DNS → hostname (e.g., dns.google)
//   ConnectivityService configures DNS-over-TLS for all networks
//   Fallback: if DoT fails, system can fall back to cleartext DNS
//   (configurable per network policy)

6. LocationManagerService — Location Providers

LocationManagerService (LMS) manages location providers, enforces location permissions, and coordinates with the Fused Location Provider (Google Play Services) to deliver location to apps.

Provider Architecture

// LocationManagerService delegates to LocationProviderManager per provider:
//
// Built-in providers:
//   "gps"     → GnssLocationProvider (raw satellite fixes)
//   "network" → NetworkLocationProvider (WiFi + cell tower triangulation)
//   "passive" → PassiveLocationProvider (piggybacks on other requests)
//   "fused"   → FusedLocationProvider (Google Play Services, if available)
//
// Provider registration:
// Each provider implements the LocationProviderManager interface.
// GnssLocationProvider talks to the GNSS HAL via JNI → native → kernel GPS driver.
// NetworkLocationProvider is typically a bound service (GMS NetworkLocation).

// LocationRequest (app-side):
val request = LocationRequest.Builder(Priority.PRIORITY_HIGH_ACCURACY, 5000L)
    .setMinUpdateDistanceMeters(10f)     // minimum displacement
    .setMaxUpdateDelayMillis(30_000L)    // batch delivery window
    .setWaitForAccurateLocation(true)    // wait for GPS fix, not just network
    .build()

// Priority levels:
//   PRIORITY_HIGH_ACCURACY   → GPS + network + sensors (highest power)
//   PRIORITY_BALANCED_POWER  → network + WiFi (no GPS)
//   PRIORITY_LOW_POWER       → cell tower only (coarse)
//   PRIORITY_PASSIVE         → no active requests, receive from others

// Fused Location Provider (Google Play Services):
// Combines GPS + WiFi + cell + accelerometer + gyroscope + barometer
// Uses sensor fusion algorithms to provide:
// - Faster first fix (WiFi/cell while waiting for GPS)
// - Smoother location updates (sensor dead-reckoning between GPS fixes)
// - Lower power (can use sensors instead of GPS for walking detection)
// LMS delegates to FLP when available; falls back to raw providers otherwise

Background Location Restrictions

// Android has progressively restricted background location access:

// Android 8 (API 26):
// - Background apps receive location updates only a few times per hour
// - Foreground service with FOREGROUND_SERVICE_TYPE_LOCATION bypasses this

// Android 10 (API 29):
// - New permission: ACCESS_BACKGROUND_LOCATION
// - Apps must request foreground location first, then background separately
// - User sees a separate prompt for background location
// - "Allow all the time" grants background; "While using the app" does not

// Android 11 (API 30):
// - Background location prompt removed from the runtime dialog
// - User must go to Settings to grant "Allow all the time"
// - App can only request foreground location in the dialog

// Android 12 (API 31):
// - Approximate vs precise: ACCESS_COARSE_LOCATION vs ACCESS_FINE_LOCATION
// - User can choose "Approximate" even when app requests precise
// - New runtime dialog shows both options side by side
// - Apps must handle receiving coarse location even when requesting fine

// LocationManagerService enforcement:
// LMS checks the calling app's permission + AppOps state:
// 1. checkCallingPermission(ACCESS_FINE_LOCATION)
// 2. AppOpsManager.noteOp(OP_FINE_LOCATION) → may return MODE_FOREGROUND
// 3. If MODE_FOREGROUND → LMS checks if app is in foreground
//    (via ActivityManagerService.getUidProcessState())
// 4. If background + no ACCESS_BACKGROUND_LOCATION → throttled/denied

Geofencing

// Geofencing allows apps to be notified when the device enters/exits a region.
// Two implementations:

// 1. Platform GeofenceManager (LocationManagerService):
//    locationManager.addGeofence(geofence, pendingIntent)
//    Limited: uses raw GPS, high power consumption, max ~100 geofences

// 2. Google Play Services GeofencingClient (recommended):
//    geofencingClient.addGeofences(geofencingRequest, pendingIntent)
//    Uses Fused Location Provider — much lower power
//    Combines WiFi + cell + activity recognition to trigger geofences
//    Supports up to 100 geofences per app
//    Triggers: GEOFENCE_TRANSITION_ENTER, EXIT, DWELL
//    Dwell: fires only after staying in the region for a specified duration

// Geofence limitations:
// - Minimum radius: 100m (smaller radii unreliable)
// - Background restrictions apply (Android 10+)
// - Geofences cleared on reboot → must re-register in BOOT_COMPLETED
// - Play Services geofences survive app death (managed by GMS process)

7. StorageManagerService — Storage and Volumes

StorageManagerService manages storage volumes (internal, adoptable, portable), coordinates with vold (the volume daemon), and provides storage statistics. It works alongside MediaProvider and the FUSE layer to enforce Scoped Storage.

Volume Management

// StorageManagerService communicates with vold (volume daemon) via Binder.
// vold handles the low-level mount/unmount/format operations.

// Volume types:
// 1. Internal (emulated):
//    /data/media/0/ → mounted as /storage/emulated/0/ via FUSE
//    Always available, encrypted with FBE (File-Based Encryption)

// 2. Adoptable storage (Android 6+):
//    SD card formatted as internal storage (ext4, encrypted)
//    Merged with internal storage — apps can be moved to it
//    Tied to the device — cannot be read on another device

// 3. Portable storage:
//    SD card with FAT32/exFAT — readable on any device
//    Apps cannot be installed on it
//    Mounted at /storage/XXXX-XXXX/

// StorageManagerService tracks volume state:
//   UNMOUNTED → CHECKING → MOUNTED → EJECTING → UNMOUNTED
//   MOUNTED → BAD_REMOVAL (physically removed without ejecting)

// StorageVolume API (app-facing):
val volumes = storageManager.storageVolumes
for (volume in volumes) {
    val desc = volume.getDescription(context)  // "Internal storage", "SD card"
    val uuid = volume.uuid                      // null for primary
    val state = volume.state                    // "mounted", "unmounted", etc.
    val isPrimary = volume.isPrimary
    val isRemovable = volume.isRemovable
    val isEmulated = volume.isEmulated
}

Scoped Storage Enforcement

// Scoped Storage (Android 10+, enforced Android 11+):
// StorageManagerService delegates enforcement to MediaProvider + FUSE.

// The FUSE layer:
// /storage/emulated/0/ is a FUSE mount backed by /data/media/0/
// MediaProvider registers as the FUSE handler:
// - Every file operation (open, read, write, stat) goes through MediaProvider
// - MediaProvider checks permissions per-file based on the calling app's UID
//
// Access rules:
// 1. App-specific directory (/storage/emulated/0/Android/data/<pkg>/):
//    Full access, no permission needed
// 2. Media files (images, video, audio):
//    READ_MEDIA_IMAGES / READ_MEDIA_VIDEO / READ_MEDIA_AUDIO (Android 13+)
//    Or legacy READ_EXTERNAL_STORAGE (Android 12 and below)
// 3. Other files (PDFs, docs):
//    Must use Storage Access Framework (SAF) / document picker
//    Or MediaStore.Downloads for files the app created
// 4. All files access:
//    MANAGE_EXTERNAL_STORAGE (special permission, Play policy restricted)

// StorageStatsManager — per-app usage:
val stats = storageStatsManager.queryStatsForPackage(
    StorageManager.UUID_DEFAULT, packageName, userHandle
)

val dataSize = stats.dataBytes     // /data/data/<pkg>/
val cacheSize = stats.cacheBytes   // cache directories

DeviceStorageMonitorService

// DeviceStorageMonitorService monitors /data partition free space.
// Thresholds (configurable via sys.sysctl.extra_free_kbytes):
//
// LOW:      500MB free → sends ACTION_DEVICE_STORAGE_LOW broadcast
//           Apps should reduce cache, defer downloads
//           PackageManagerService refuses new app installs
//
// CRITICAL: 50MB free → system starts aggressively clearing caches

//           → sends ACTION_DEVICE_STORAGE_OK

// Monitoring interval: checks every 60 seconds (configurable)
// Uses StatFs to query filesystem stats on /data

// Impact on apps:
// - SQLite writes may fail with SQLITE_FULL
// - SharedPreferences.commit() may return false
// - File.createNewFile() throws IOException
// - Apps should handle storage-full gracefully, especially for databases

8. Staff-Level Patterns Across Services

All the services covered here — and indeed every system service in system_server — follow consistent architectural patterns. Understanding these patterns lets you reason about any system service, even ones you haven't studied directly.

The Universal Service Pattern

// Every system service follows the same Binder pattern:

// 1. AIDL interface definition:
//    frameworks/base/core/java/android/os/IPowerManager.aidl
//    Defines the public API surface (acquireWakeLock, goToSleep, etc.)

// 2. Stub implementation in system_server:
//    PowerManagerService extends IPowerManager.Stub
//    Runs in system_server process (UID 1000, PID = system_server PID)

// 3. Proxy in app process:
//    PowerManager wraps IPowerManager.Stub.Proxy
//    Obtained via context.getSystemService(Context.POWER_SERVICE)
//    → ServiceManager.getService("power") → returns IBinder
//    → IPowerManager.Stub.asInterface(binder) → returns proxy

// 4. Permission enforcement at the Binder boundary:
class PowerManagerService : IPowerManager.Stub() {
    override fun acquireWakeLock(
        lock: IBinder, flags: Int, tag: String, packageName: String,
        ws: WorkSource?, historyTag: String?
    ) {
        // FIRST: check permission
        context.enforceCallingOrSelfPermission(
            android.Manifest.permission.WAKE_LOCK,
            "acquireWakeLock"
        )
        // THEN: validate inputs, enforce policy, do the work
        val callingUid = Binder.getCallingUid()
        val callingPid = Binder.getCallingPid()
        // ...
    }
}

// 5. dump() for debugging:
//    Every service implements dump(fd, pw, args) for dumpsys
//    dumpsys power → calls PowerManagerService.dump()
//    dumpsys alarm → calls AlarmManagerService.dump()
//    dumpsys jobscheduler → calls JobSchedulerService.dump()

Service-to-Service Communication

// Services within system_server communicate via two mechanisms:

// 1. LocalServices (in-process, no Binder overhead):
//    For system_server-internal communication between services.
//    Uses a simple static registry — no IPC, no serialization.
//
//    // Publishing:
//    LocalServices.addService(PowerManagerInternal::class.java, powerManagerInternal)
//
//    // Consuming:
//    val pmi = LocalServices.getService(PowerManagerInternal::class.java)
//    pmi.setDeviceIdleMode(true)  // direct method call, same process
//
//    PowerManagerInternal exposes methods that should NOT be in the public AIDL
//    (e.g., setDeviceIdleMode is called by DeviceIdleController, not by apps)

// 2. Binder (cross-process, for app → system_server):
//    The standard AIDL path described above.
//    Also used for system_server → app callbacks (e.g., onStartJob())

// Thread model:
// system_server has a Binder thread pool of 31 threads (default max).
// ALL incoming Binder calls from ALL apps land on these 31 threads.
// If all 31 threads are blocked (e.g., waiting on a lock), new calls queue.
// This is why lock contention in system services (like pre-Android 12 PMS)
// can cause system-wide slowdowns — it's not just one service that suffers,
// it's every Binder call to system_server.

// Handler threads:
// Many services also have their own Handler + Looper for async work:
//   PowerManagerService → mHandler (PowerManagerHandler)
//   AlarmManagerService → mHandler (AlarmHandler)
//   JobSchedulerService → mHandler (JobHandler)
// Binder thread receives the call → posts to Handler → Handler processes on its thread
// This avoids holding Binder threads for long operations.

Essential Debugging Commands

// adb shell cmd notification post -t "Test" "tag" "body"  // post test notification

Staff-Level Insight — The 31-Thread Bottleneck

system_server's Binder thread pool is capped at 31 threads (configurable but rarely changed). Every app-to-system call — getPackageInfo(), acquireWakeLock(), notify(), requestLocationUpdates() — competes for these 31 threads. When a system service holds a lock for too long (e.g., PMS's global lock pre-Android 12, or NMS ranking during a notification storm), it doesn't just slow down that service — it starves all other services of Binder threads. This is the root cause of many "system_server ANR" reports in the field. At Staff level, you should understand that optimizing a single system service's lock contention has system-wide impact, and that the Handler thread pattern (receive on Binder thread → post to Handler → release Binder thread) is the standard mitigation. Services that do synchronous work on Binder threads are architectural red flags.

Staff-Level Insight — Wake Lock Bugs as a System Design Problem

Wake lock bugs are not just app-level issues — they're a system design problem. The kernel's suspend mechanism is all-or-nothing: if any wake lock is held, the entire SoC stays awake. A single misbehaving app holding a PARTIAL_WAKE_LOCK for 8 hours drains the battery for every app on the device. This is why Android evolved from raw wake locks → JobScheduler → WorkManager → Doze → App Standby Buckets → foreground service restrictions. Each layer adds constraints to prevent apps from keeping the device awake unnecessarily. The architectural lesson: when a low-level primitive (wake locks) is too dangerous for direct use, the platform must build progressively higher-level abstractions that enforce correct behavior by construction. At Google scale, battery drain from wake lock bugs is measured in aggregate device-hours and directly impacts user retention metrics.

Interview Pattern — Cross-Service Coordination

A common L6 interview question: "Walk me through what happens when a user sets an alarm in a clock app." The answer touches every service in this document: AlarmManagerService schedules the alarm → PowerManagerService keeps the device in suspend until the alarm time → kernel RTC wakes the CPU → AlarmManagerService delivers the PendingIntent → NotificationManagerService posts the alarm notification with fullScreenIntent → PowerManagerService turns on the screen → the alarm app's activity launches via AMS. Understanding how services compose to deliver user-facing features is the hallmark of Staff-level systems thinking.

9. DeviceIdleController — Doze State Machine

DeviceIdleController manages Doze mode — Android's aggressive power-saving state that suspends most background activity when the device is idle, stationary, and unplugged. It implements a multi-stage state machine that progressively restricts app behavior to maximize battery life during extended idle periods.

Staff-Level Insight — Maintenance Window Intervals

The increasing maintenance window intervals (1hr → 2hr → 4hr → 6hr max) are deliberately designed to minimize wake-ups during overnight charging. As the device spends more time in deep IDLE, the gaps between maintenance windows grow, so the device progressively spends more time in true deep sleep. This is a classic exponential backoff pattern applied to power management — the same principle behind TCP retransmission timers. At Google scale, even small improvements to this algorithm affect aggregate battery life across billions of devices.

10. App Standby Buckets — Assignment Algorithm

UsageStatsService tracks app usage patterns and assigns each app to a standby bucket that determines how aggressively its background work is restricted. This system replaced the binary "app is active / app is idle" model with a graduated approach that better reflects real usage patterns.

Bucket Assignment Criteria

// UsageStatsService evaluates multiple signals to assign buckets:
//
// ACTIVE — app is currently in use or was just used
//   Triggers: app in foreground, notification tapped, sync adapter running
//
// WORKING_SET — app used recently, likely to be used again soon
//   Triggers: used in last few hours, widget visible on home screen
//
// FREQUENT — app used regularly but not recently
//   Triggers: used in last few days, periodic engagement pattern
//
// RARE — app installed but seldom used
//   Triggers: not used in weeks, no notification interaction
//
// RESTRICTED (Android 12+) — app never opened or misbehaving
//   Triggers: never launched by user, excessive wake locks,
//             repeated background crashes, excessive battery usage
//
// Bucket transitions are one-directional toward restriction:
// ACTIVE → WORKING_SET → FREQUENT → RARE → RESTRICTED
// Any user interaction immediately promotes back to ACTIVE.

Per-Bucket Restrictions

Bucket Jobs Alarms Network FCM
ACTIVE No limits No limits No limits No limits
WORKING_SET Deferred 2hr Deferred No limits No limits
FREQUENT Deferred 8hr Deferred Limited 10/hr
RARE Deferred 24hr Max 1/day Limited 5/day
RESTRICTED Max 1/day Max 1/day Limited 1/day

Debugging Standby Buckets

// Check current bucket for an app:
// adb shell am get-standby-bucket com.example.app
//   → returns: ACTIVE, WORKING_SET, FREQUENT, RARE, or RESTRICTED

// Force a bucket for testing:
// adb shell am set-standby-bucket com.example.app rare

// View all bucket assignments:
// adb shell dumpsys usagestats

// Programmatic check (from within the app):
val usageStatsManager = context.getSystemService(UsageStatsManager::class.java)!!
val bucket = usageStatsManager.appStandbyBucket
// Returns: STANDBY_BUCKET_ACTIVE (10), STANDBY_BUCKET_WORKING_SET (20),
//          STANDBY_BUCKET_FREQUENT (30), STANDBY_BUCKET_RARE (40),
//          STANDBY_BUCKET_RESTRICTED (45)
Staff-Level Insight — RESTRICTED Bucket

The RESTRICTED bucket (Android 12+) is the most aggressive tier. Apps that the user has never opened, or that exhibit bad behavior (excessive wake locks, repeated background crashes, abnormal battery usage), are placed here. In RESTRICTED, an app gets at most 1 job per day, 1 alarm per day, and 1 FCM message per day — effectively making it dormant. This bucket was introduced because RARE wasn't restrictive enough for apps that users install but never use (e.g., pre-installed bloatware). The combination of standby buckets + Doze creates a multi-level restriction hierarchy that's one of the most complex power management systems in any mobile OS.

11. JobScheduler Slot Types & Concurrency

JobConcurrencyManager controls how many jobs can run simultaneously and how slots are allocated across different priority levels. This is the mechanism that determines whether your background job actually executes or waits in a queue.

Slot Architecture (Android 12+)

  JobConcurrencyManager Slot Allocation
  ═══════════════════════════════════════

  Total concurrent slots: ~8-16 (device-dependent)

  ┌─────────────────────────────────────────────┐
  │              Slot Priority Tiers             │
  │                                             │
  │  ┌─────┐  ┌─────┐  ┌─────┐  ┌─────┐       │
  │  │ TOP │  │ FGS │  │  EJ │  │  BG │       │
  │  │     │  │     │  │     │  │     │       │
  │  │ FG  │  │ App │  │Expe-│  │Back-│       │
  │  │ app │  │ w/  │  │dited│  │grnd │       │
  │  │ jobs│  │ FGS │  │ job │  │ job │       │
  │  └─────┘  └─────┘  └─────┘  └─────┘       │
  │  Highest ──────────────────→ Lowest        │
  │                                             │
  │  When all slots full:                       │
  │  Lower-priority jobs preempted for higher   │
  └─────────────────────────────────────────────┘

Slot Types and Limits

Slot Type Description Max Execution Time Priority
TOP Job from the foreground app 10 minutes Highest — always gets a slot
FGS Job from app with foreground service 10 minutes High — preferred allocation
EJ Expedited job (setExpedited(true)) 1-4 minutes (bucket-dependent) Medium-high — quota-limited
BG Regular background job 10 minutes Lowest — preempted when slots scarce

Concurrency Internals

// Source: frameworks/base/services/core/java/com/android/server/job/JobConcurrencyManager.java

// JobSchedulerService.Constants — configurable limits via DeviceConfig:
//   KEY_CONCURRENCY_LIMIT = "concurrency_limit"  // total slots
//   KEY_SCREEN_OFF_ADJUSTMENT = "screen_off_adjustment"  // reduce when screen off

// Slot assignment algorithm:
// 1. Count available slots (total - currently running)
// 2. For each READY job, determine its work type (TOP/FGS/EJ/BG)
// 3. Assign slots in priority order: TOP first, then FGS, EJ, BG
// 4. If no slots available for a lower-priority job:
//    a. Check if any running job has lower priority
//    b. If yes → preempt (call onStopJob()) and assign slot to new job
//    c. If no → job waits in queue

// Preemption:
// When a TOP job arrives but all slots are occupied by BG jobs,
// JobConcurrencyManager calls onStopJob() on the lowest-priority
// running job to free a slot. The preempted job is rescheduled
// with its original constraints.
Staff-Level Insight — Multi-Level Scheduling Hierarchy

The slot system creates a multi-level scheduling hierarchy when combined with App Standby Buckets. A background job from a RARE-bucket app might wait hours even if the device is idle — first the bucket imposes a 24-hour deferral, then even when the job becomes READY, it competes for BG slots against jobs from higher-bucket apps. This means a RARE-bucket app's job could theoretically wait 24+ hours before execution. Understanding this compound effect is critical for diagnosing "my job never runs" bugs in production — it's rarely a single restriction, but the intersection of bucket + slot priority + Doze state.

12. NotificationListenerService Security

NotificationListenerService (NLS) is one of the most powerful APIs in Android — it can read all notifications from all apps, including their content, actions, and inline reply text. This power comes with significant security implications.

NLS Capabilities

// A NotificationListenerService receives:
class MyNotificationListener : NotificationListenerService() {

    override fun onNotificationPosted(sbn: StatusBarNotification) {
        val notification = sbn.notification
        val pkg = sbn.packageName              // source app
        val extras = notification.extras

        val title = extras.getString(Notification.EXTRA_TITLE)
        val text = extras.getCharSequence(Notification.EXTRA_TEXT)
        val bigText = extras.getCharSequence(Notification.EXTRA_BIG_TEXT)

        // Can read MessagingStyle messages (private conversations!)
        val messages = Notification.MessagingStyle.Message
            .getMessagesFromBundleArray(
                extras.getParcelableArray(Notification.EXTRA_MESSAGES)
            )

        // Can read actions (including inline reply RemoteInput)
        val actions = notification.actions
    }

    // NLS can also:
    override fun onNotificationRemoved(sbn: StatusBarNotification) { }

    // Dismiss any notification
    fun dismissNotification(key: String) = cancelNotification(key)

    // Snooze notifications
    fun snoozeNotif(key: String) = snoozeNotification(key, 60_000L)

    // Modify notification ranking
    override fun onNotificationRankingUpdate(rankingMap: RankingMap) { }
}

Security Model

// NLS requires explicit user opt-in:
// Settings → Apps → Special app access → Notification access
// The user must manually toggle the switch for each NLS app.

// Manifest declaration (Android 13+):
// <service
//     android:name=".MyNotificationListener"
//     android:permission="android.permission.BIND_NOTIFICATION_LISTENER_SERVICE"
//     android:exported="true">
//     <intent-filter>
//         <action android:name="android.service.notification.NotificationListenerService" />
//     </intent-filter>
// </service>

// NMS binds to NLS via INotificationListener.aidl:
// NotificationManagerService → bindService() → app's NLS
// NMS pushes notifications to the listener via oneway Binder calls

// Security implications of a malicious NLS:
// - Read 2FA codes from SMS/authenticator notifications
// - Read private messages from messaging apps
// - Read financial alerts (bank transactions, payment confirmations)
// - Read email subjects and previews
// - Dismiss security-critical notifications silently
// - Track app usage patterns via notification metadata
Security Risk — NLS Abuse

A malicious NotificationListenerService is essentially a keylogger for notifications. It can silently read 2FA codes, private messages, and financial alerts from every app on the device. Google Play policy restricts NLS usage to specific categories — accessibility tools, companion device apps (e.g., smartwatch), and notification management utilities. Apps that request NLS access without clear justification are rejected during review. Despite this, NLS remains one of the most abused APIs by sideloaded malware.

13. MediaProvider & FUSE — Scoped Storage Enforcement

Scoped Storage (Android 10+) is enforced by MediaProvider acting as a FUSE (Filesystem in Userspace) handler. Every file operation on shared storage is intercepted and permission-checked by MediaProvider before the kernel grants access.

The FUSE Enforcement Flow

  Scoped Storage — FUSE Enforcement Path
  ═════════════════════════════════════════

  App opens /sdcard/DCIM/photo.jpg
                │
                ▼
  ┌──────────────────────────────────────┐
  │  Linux Kernel — VFS Layer            │
  │  Detects FUSE mount at /sdcard/      │
  │  Redirects to userspace handler      │
  └──────────────┬───────────────────────┘
                 │ FUSE protocol
                 ▼
  ┌──────────────────────────────────────┐
  │  MediaProvider (FUSE handler)        │
  │                                      │
  │  1. Resolve calling app UID          │
  │  2. Check READ_MEDIA_IMAGES perm     │
  │  3. Check AppOps for fine-grained    │
  │     access control                   │
  │  4. Query MediaStore for file owner  │
  │                                      │
  │  Permitted? → Return file descriptor │
  │  Denied?    → Return EACCES          │
  └──────────────┬───────────────────────┘
                 │
                 ▼
  ┌──────────────────────────────────────┐
  │  /data/media/0/DCIM/photo.jpg        │
  │  (actual file on ext4/f2fs)          │
  └──────────────────────────────────────┘

Access Rules

Storage Location Permission Required Notes
App-specific dir (Android/data/<pkg>/) None Full access, no FUSE check
Media files (images, video, audio) READ_MEDIA_IMAGES / VIDEO / AUDIO (API 33+) Legacy: READ_EXTERNAL_STORAGE
Non-media files (PDFs, docs) Storage Access Framework (SAF) Or MediaStore.Downloads for own files
All files MANAGE_EXTERNAL_STORAGE Special permission, Play policy restricted

Bulk Operations (Android 11+)

// For modifying files owned by other apps, MediaProvider requires user consent:

// Request write access to a set of files:
val uris = listOf(uri1, uri2, uri3)
val pendingIntent = MediaStore.createWriteRequest(contentResolver, uris)
startIntentSenderForResult(pendingIntent.intentSender, REQUEST_CODE, null, 0, 0, 0)

// Other bulk operations:
// MediaStore.createDeleteRequest()  — delete files owned by other apps
// MediaStore.createTrashRequest()   — move to trash
// MediaStore.createFavoriteRequest() — mark as favorite

// These all show a system dialog asking the user to confirm the operation.
// The app cannot bypass this dialog — MediaProvider enforces it.
Staff-Level Insight — FUSE Performance Trade-off

The FUSE layer adds overhead to every file operation — each open(), read(), stat() requires a context switch from kernel space to the MediaProvider userspace process for permission checking. This overhead is why Android 11 introduced MediaStore.setRequireOriginal() and direct file path access for apps with READ_MEDIA_* permissions — a pragmatic trade-off between security and performance. For apps that process many files (photo editors, file managers), the FUSE overhead can be significant. The platform team's solution was to allow direct path access while still enforcing permissions via the FUSE layer's initial open() check, rather than checking every individual read() call.

14. BatteryStatsService

BatteryStatsService tracks per-app battery usage across multiple dimensions — wake locks, CPU time, network bytes, sensor usage, and GPS time. It is the data source behind the "Battery usage" screen in Settings and the essential tool for diagnosing battery drain in production.

What BatteryStats Tracks

Metric Granularity Source
Wake locks Per-app, per-tag, duration PowerManagerService
CPU time Per-UID, user + system time /proc/uid_stat/
Network bytes Per-UID, tx + rx, WiFi + cellular NetworkStatsService
Sensor usage Per-app, per-sensor, duration SensorService
GPS time Per-app, duration LocationManagerService
Camera usage Per-app, duration CameraService
Bluetooth scans Per-app, count + duration BluetoothManagerService

Debugging with BatteryStats

// The essential battery debugging commands:

// Full battery stats (human-readable):
// adb shell dumpsys batterystats

// Machine-readable format for Battery Historian:
// adb shell dumpsys batterystats --checkin > batterystats.txt
// Then upload to Battery Historian (https://bathist.ef.lc/)

// Reset stats (start fresh measurement):
// adb shell dumpsys batterystats --reset

// Per-app stats:
// adb shell dumpsys batterystats <package-name>

// BatteryStats data structure:
// Persisted to: /data/system/batterystats.bin
// In-memory: BatteryStatsImpl — tracks all metrics since last charge
// On each charge cycle, stats are rolled over and archived

// BatteryStatsHelper computes per-app power estimates using:
// - Device-specific power_profile.xml (mA per component)
// - Measured usage durations from BatteryStats
// - Formula: energy = power (mA) × duration (hours)
//
// Example from power_profile.xml:
// <item name="screen.on">200</item>        <!-- mA when screen on -->
// <item name="wifi.active">120</item>      <!-- mA during WiFi tx -->
// <item name="gps.on">50</item>            <!-- mA during GPS fix -->
// <item name="cpu.active">100</item>       <!-- mA at base CPU freq -->
Staff-Level Insight — power_profile.xml Accuracy

power_profile.xml is OEM-specific and often inaccurate — the values are typically measured once during device bring-up and never updated. This means the battery estimates shown in Settings → Battery are approximations, not measurements. Some OEMs provide wildly inaccurate profiles (e.g., listing GPS power consumption at 50mA when the actual hardware draws 150mA), which leads to misleading battery usage attribution. At Google scale, this inaccuracy is a known limitation — the Android team has explored on-device power measurement via fuel gauge ICs, but the power_profile.xml approach remains the standard because it works across all hardware without special sensor support.
