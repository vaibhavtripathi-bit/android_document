AMS & ATMS Internals

Process management, broadcast dispatch, ContentProvider lifecycle, instrumentation, and the critical code paths inside Android's most important system services

1. AMS vs ATMS — The Android 10 Split

Before Android 10 (Q), ActivityManagerService was a monolithic god-class — over 25,000 lines managing processes, activities, broadcasts, services, content providers, and tasks. The Android 10 refactor extracted activity and task management into a separate ActivityTaskManagerService.

Why the Split Happened

  • Lock contention — AMS's synchronized(this) lock was a global bottleneck. Every operation (process management, activity transitions, broadcast dispatch) competed for the same lock.
  • Separation of concerns — Activity lifecycle and task management are logically distinct from process lifecycle and broadcast dispatch.
  • Multi-display / foldables — ATMS manages per-display task stacks, which required a cleaner abstraction than the monolithic AMS could offer.
  • Testability — Smaller classes with defined boundaries are easier to unit test in AOSP.

Responsibility Split

Responsibility AMS (ActivityManagerService) ATMS (ActivityTaskManagerService)

OOM adjustment OomAdjuster, updateOomAdjLocked() —

Cross-Service Communication

AMS and ATMS live in the same process (system_server) but need to call each other frequently. They use internal interfaces registered via LocalServices — direct Java method calls, no Binder IPC:

// ATMS exposes ActivityTaskManagerInternal to AMS
LocalServices.addService(
    ActivityTaskManagerInternal::class.java,
    atmsInternalImpl
)

// AMS exposes ActivityManagerInternal to ATMS
LocalServices.addService(
    ActivityManagerInternal::class.java,
    amsInternalImpl
)

// Usage from AMS when it needs activity info:
val atmi = LocalServices.getService(ActivityTaskManagerInternal::class.java)
val topActivity = atmi.getTopActivity()

// Usage from ATMS when it needs process info:
val ami = LocalServices.getService(ActivityManagerInternal::class.java)
ami.killProcess(pid, reason)
Cross-Reference

For a deep dive into the ATMS side — ActivityRecord, ClientTransaction, TransactionExecutor, and the activity lifecycle state machine — see the Lifecycle & Navigation series. This document focuses on the AMS side: processes, broadcasts, providers, and instrumentation.

The Lock Hierarchy

Post-split, the locking changed from a single synchronized(this) on AMS to a more granular scheme:

// Pre-Android 10: everything under one lock
synchronized(this) { // "this" = AMS instance
    // ALL operations: process, activity, broadcast, provider
}

// Post-Android 10:
// ATMS uses WindowManagerGlobalLock (mGlobalLock)
// AMS uses its own lock for process/broadcast/provider operations
// Lock ordering: mGlobalLock → AMS lock (never the reverse)

// mGlobalLock is shared between ATMS and WMS
// because activity transitions and window operations are tightly coupled

2. Process Management

ProcessRecord — The In-Memory Process Representation

Every running app process is tracked by a ProcessRecord in AMS. This is the single most important data structure for understanding how Android manages app processes.

// Source: frameworks/base/services/core/java/com/android/server/am/ProcessRecord.java

class ProcessRecord {
    // Identity
    val info: ApplicationInfo       // Package/app metadata
    val processName: String         // e.g., "com.example.app"
    val uid: Int                    // Linux UID (unique per app)
    var pid: Int                    // Linux PID (assigned after fork)

    // Binder connection to the app process
    var thread: IApplicationThread? // Proxy to ActivityThread in the app
    // null until attachApplicationLocked() completes

    // Component tracking
    val activities: ArrayList
    val services: ArraySet
    val executingServices: ArraySet  // Currently running onStart/onBind
    val connections: ArrayMap
    val receivers: ArraySet
    val pubProviders: ArrayMap

    // OOM adjustment state
    var curAdj: Int = UNKNOWN_ADJ           // Current computed adj
    var setAdj: Int = UNKNOWN_ADJ           // Last value written to kernel
    var curProcState: Int                    // PROCESS_STATE_* constant
    var curCapability: Int                   // PROCESS_CAPABILITY_* flags
    var killedByAm: Boolean = false          // Killed intentionally by AMS
    var killed: Boolean = false              // Process is dead

    // Crash/ANR tracking
    var crashHandler: CrashHandler?
    var lastCrashTime: Long
    var crashCount: Int
    var anrAnnotation: String?

    // Timing
    var lastActivityTime: Long              // Last time an Activity was in foreground
    var lastProviderTime: Long              // Last time a provider was used
    var curRawAdj: Int                      // Raw adj before clamping
}

updateOomAdjLocked() — The Critical Recalculation

This is the single most performance-critical method in all of AMS. It recalculates the oom_adj score for every process whenever any component state changes.

// Source: frameworks/base/services/core/java/com/android/server/am/OomAdjuster.java

// Simplified logic of updateOomAdjLocked():
fun updateOomAdjLocked(app: ProcessRecord): Boolean {
    var adj = CACHED_APP_MAX_ADJ  // Start at lowest priority

    // 1. Check for foreground activity
    if (app.activities.any { it.isVisible }) {
        adj = FOREGROUND_APP_ADJ
    }

    // 2. Check for foreground service
    if (app.hasForegroundServices()) {
        adj = minOf(adj, PERCEPTIBLE_APP_ADJ)
    }

    // 3. Check for executing service (onStartCommand running)
    if (app.executingServices.isNotEmpty()) {
        adj = minOf(adj, FOREGROUND_APP_ADJ)
    }

    // 4. Check for recent service (<30 min)
    if (app.services.isNotEmpty() && !isServiceBProcess(app)) {
        adj = minOf(adj, SERVICE_ADJ)
    }

    // 5. Check for active content provider clients
    if (app.pubProviders.isNotEmpty()) {
        for (provider in app.pubProviders.values) {
            for (conn in provider.connections) {
                // Elevate based on client's importance
                adj = minOf(adj, conn.client.curAdj + 1)
            }
        }
    }

    // 6. Check for bound service clients (transitive dependency)
    for (sr in app.services) {
        for (conn in sr.getConnections()) {
            val client = conn.binding.client
            // The binding flags matter:
            // BIND_ABOVE_CLIENT → adj = client adj - 1
            // BIND_IMPORTANT → adj = client adj
            // BIND_NOT_FOREGROUND → don't elevate to FG
            adj = computeBindingAdj(adj, conn, client)
        }
    }

    // 7. Write to kernel if changed
    if (adj != app.setAdj) {
        ProcessList.setOomAdj(app.pid, app.uid, adj)
        // Writes to /proc/{pid}/oom_score_adj
        app.setAdj = adj
    }
    return true
}
Staff-Level Insight — updateOomAdjLocked() Performance

updateOomAdjLocked() is called hundreds of times per second on a busy device — every activity start/stop, service bind/unbind, broadcast delivery, and provider access triggers a recalculation. The method walks the entire process list and follows transitive binding chains (process A binds to B which binds to C). In AOSP, this is the #1 source of lock contention in AMS. Android 12 introduced OomAdjuster as a separate class (extracted from AMS) with optimizations: partial updates (only recalculate affected processes), caching of intermediate results, and batching of /proc writes. A Staff Engineer should know that adding a new component type that affects oom_adj is extremely risky — it can degrade system-wide responsiveness.

// If a receiver doesn't call goAsync().finish() or return from
// onReceive() within the timeout, AMS triggers a broadcast ANR:
// 1. Logs "Timeout of broadcast" to logcat
// 2. Skips the stuck receiver, moves to next
// 3. For ordered broadcasts: propagates current result to next receiver
// 4. Reports ANR to the stuck app's process
Interview Gotcha — goAsync()

BroadcastReceiver.goAsync() returns a PendingResult that lets you move work off the main thread, but the ANR timeout still applies. You get the same 10s/60s window — goAsync() just lets you use that time on a background thread instead of blocking the main thread. You must call pendingResult.finish() when done. If you need more than 60 seconds, you must delegate to a Service or WorkManager.

Implicit Broadcast Restrictions (Android 8+)

Starting with Android 8 (Oreo), most implicit broadcasts are not delivered to manifest-declared receivers. Only dynamically registered receivers receive them. This was a critical change to reduce background process launches.

// Exempt implicit broadcasts (still delivered to manifest receivers):
// ACTION_BOOT_COMPLETED, ACTION_LOCALE_CHANGED, ACTION_USB_ACCESSORY_ATTACHED,
// ACTION_HEADSET_PLUG, ACTION_CONNECTION_STATE_CHANGED (Bluetooth),
// and a few dozen others listed in the framework

// Non-exempt: ACTION_POWER_CONNECTED, ACTION_SCREEN_ON, CONNECTIVITY_ACTION, etc.
// These only reach registerReceiver() receivers

// Why? Before Android 8, a single CONNECTIVITY_ACTION broadcast could wake up
// hundreds of apps that declared  for it in their manifest.
// Each app's process had to be started just to deliver the broadcast.

Android 14: BroadcastModernQueue

Android 14 introduced BroadcastModernQueue, a ground-up rewrite of the broadcast dispatch infrastructure:

// Source: frameworks/base/services/core/java/com/android/server/am/BroadcastQueueModernImpl.java

// Key changes from legacy BroadcastQueue:
// 1. Per-process delivery queues instead of global ordered/parallel queues
//    → A slow receiver in process A doesn't block delivery to process B
// 2. Concurrent delivery to multiple processes
//    → Legacy: one ordered broadcast at a time across ALL processes
//    → Modern: parallel delivery to different processes, ordered within each
// 3. Prioritized urgent broadcasts (e.g., PHONE_STATE)
//    → Can jump ahead of pending non-urgent broadcasts
// 4. Better ANR attribution
//    → Tracks exactly which process is slow, not just "broadcast timed out"
// 5. Deferred delivery for cached processes
//    → Don't wake a cached process just for a broadcast; deliver when it wakes
Staff-Level Insight — Broadcast Storms

A single sendBroadcast() can trigger delivery to hundreds of receivers across dozens of processes. In production, broadcast storms are a real problem: an app sends ACTION_SOMETHING, which wakes 50 processes, each of which does work that triggers more broadcasts. The legacy BroadcastQueue serialized ordered broadcasts globally — one slow receiver blocked all ordered broadcasts system-wide. BroadcastModernQueue fixes this by isolating delivery per-process, but the fundamental problem remains: broadcasts are a fan-out mechanism with no backpressure. At Google scale, the guidance is to prefer direct Binder calls or JobScheduler over broadcasts for inter-component communication.

Sticky Broadcasts

Deprecated API 21 — Sticky broadcasts (sendStickyBroadcast()) are cached by AMS and replayed to any future registerReceiver() call matching the intent. They still exist in AOSP (e.g., ACTION_BATTERY_CHANGED) but are deprecated for third-party use due to security concerns — any app can read or overwrite them.

4. ContentProvider Lifecycle

The Initialization Trap

ContentProviders are initialized before Application.onCreate(). This is one of the most important (and surprising) ordering guarantees in Android:

// App process startup order:
// 1. Zygote forks
// 2. ActivityThread.main()
// 3. ActivityThread.handleBindApplication()
//    a. Create Application instance
//    b. installContentProviders()     ← Providers initialized HERE
//       → ContentProvider.onCreate()  ← BEFORE Application.onCreate()!
//    c. Application.onCreate()        ← After all providers are ready
//    d. Deliver pending components (Activities, Services, etc.)

// This means: if your ContentProvider.onCreate() depends on
// Application.onCreate() initialization, it will fail.
// Libraries like Firebase, WorkManager, and AndroidX Startup
// exploit this ordering by using ContentProviders as auto-init hooks.

ContentProviderRecord in AMS

// Source: frameworks/base/services/core/java/com/android/server/am/ContentProviderRecord.java

class ContentProviderRecord {
    val info: ProviderInfo                // Manifest metadata
    val name: ComponentName               // authority → component mapping
    val appInfo: ApplicationInfo
    var proc: ProcessRecord?              // Hosting process
    var provider: IContentProvider?       // Binder to the actual provider instance
    val connections: ArrayList
    var externalProcessNoHandleCount: Int  // External process references
    var launchingApp: ProcessRecord?       // Process being started for this provider
}

// ContentProviderConnection tracks each client's reference
class ContentProviderConnection {
    val provider: ContentProviderRecord
    val client: ProcessRecord
    var stableCount: Int = 0    // Stable references: provider death kills client
    var unstableCount: Int = 0  // Unstable references: provider death is survivable
    var waiting: Boolean        // Client is waiting for provider to publish
}

acquireProvider() / releaseProvider() — Reference Counting

// When an app calls contentResolver.query(uri, ...):
// 1. ContentResolver looks up the authority
// 2. Calls ActivityThread.acquireProvider(authority)
//    → If provider is in same process: return local reference
//    → If remote: AMS.getContentProvider(callingPackage, authority)
//      → AMS finds or starts the provider process
//      → Returns IContentProvider binder proxy
// 3. Query executes via Binder IPC to the provider process
// 4. ContentResolver calls releaseProvider() when done

// Reference counting matters for process lifecycle:
// - stableCount > 0: provider death kills the CLIENT process too
//   (because the client may have an open Cursor with stale data)
// - unstableCount > 0: provider death is survivable
//   (client gets DeadObjectException, can retry)

// ContentResolver.query() uses UNSTABLE provider by default (since API 16)
// ContentResolver.insert/update/delete use STABLE provider
ContentProvider Deadlock

A classic deadlock scenario: Process A's ContentProvider calls Process B's ContentProvider, which calls back to Process A's ContentProvider. Both processes are waiting for each other on their Binder threads. AMS cannot detect this because the deadlock happens at the Binder layer, not the AMS lock layer. The only resolution is a timeout (ANR) or process kill. Avoid cross-provider calls in production code — use a mediator pattern or async callbacks instead.

Multi-Process Providers

// android:multiprocess="true" in manifest
// → Provider is instantiated in EACH client process (no IPC needed)
// → Faster, but each instance has its own state
// → Rarely used: only makes sense for read-only, stateless providers

// android:multiprocess="false" (default)
// → Single instance in the provider's own process
// → All clients talk to it via Binder IPC
// → Consistent state, but IPC overhead per call

5. Instrumentation

The Instrumentation Class

Instrumentation is the hook point through which the system creates Activities, calls lifecycle methods, and dispatches input events. Every app process has exactly one Instrumentation instance, held by ActivityThread.

How ActivityThread Uses Instrumentation

// Inside ActivityThread.performLaunchActivity():
fun performLaunchActivity(r: ActivityClientRecord): Activity {
    // 1. Create the Activity via Instrumentation
    val activity = mInstrumentation.newActivity(
        cl, component.className, r.intent
    )

    // 2. Create/get the Application
    val app = r.packageInfo.makeApplication(false, mInstrumentation)

    // 3. Attach the Activity (sets base context, window, etc.)
    activity.attach(context, this, mInstrumentation, ...)

    // 4. Call onCreate via Instrumentation
    mInstrumentation.callActivityOnCreate(activity, r.state)

    return activity
}

6. App Process Attachment

attachApplicationLocked() — The Handshake

When a newly forked process calls ActivityThread.attach(), it triggers a Binder call back to AMS. attachApplicationLocked() is where AMS completes the handshake and sends all pending work to the new process.

// Source: frameworks/base/services/core/java/com/android/server/am/ActivityManagerService.java

fun attachApplicationLocked(thread: IApplicationThread, callingPid: Int) {
    // 1. Find the ProcessRecord (created earlier by startProcessLocked)
    val app = mPidsSelfLocked.get(callingPid)
        ?: throw SecurityException("No process record for pid $callingPid")

    // 2. Store the Binder proxy
    app.thread = thread
    app.curAdj = INVALID_ADJ  // Will be recalculated

    // 3. Tell the process about itself (bindApplication)
    thread.bindApplication(
        processName = app.processName,
        appInfo = app.info,
        providers = app.pubProviders.values.toList(),  // ContentProviders to install
        instrumentationName = app.instrumentationClass,
        profilerInfo = app.profilerInfo,
        config = mConfiguration,                        // Current device config
        compatInfo = app.compat,
        services = getCommonServicesLocked(),            // Cached service binders
        coreSettings = coreSettings,
        buildSerial = Build.SERIAL,
        autofillOptions = ...,
        contentCaptureOptions = ...,
        disabledCompatChanges = ...,
        serializedSystemFontMap = ...
    )

    // 4. Deliver pending Activities (via ATMS)
    val atm = LocalServices.getService(ActivityTaskManagerInternal::class.java)
    val didSomething = atm.attachApplication(app.getWindowProcessController())

    // 5. Deliver pending Services
    mServices.attachApplicationLocked(app, processName)

    // 6. Deliver pending Broadcasts
    sendPendingBroadcastsLocked(app)

    // 7. Deliver pending Backup agent
    mBackupTargets.get(app.userId)?.let { backupTarget ->
        if (backupTarget.processName == processName) {
            thread.scheduleCreateBackupAgent(...)
        }
    }

    // 8. Update OOM adj now that we know what the process is doing
    updateOomAdjLocked(app, OOM_ADJ_REASON_PROCESS_BEGIN)
}

Staff-Level Insight — The Attach Window

The gap between fork() and attachApplicationLocked() is typically 50–200ms on modern devices, but can be much longer on low-end hardware or during boot (when dozens of processes start simultaneously). During this window, AMS queues all pending work. If the process takes too long to attach (>10 seconds), AMS considers it a "process not responding" situation and may kill it. This is a common source of startup failures on slow devices — the process is alive but hasn't attached yet, and AMS times out. The PROC_START_TIMEOUT constant controls this (10 seconds by default, 120 seconds during boot).

system_server Binder Thread Pool Exhaustion

Critical — 31 Threads for Everything

system_server has a Binder thread pool of 31 threads, shared across ALL 100+ system services. When all 31 threads are busy handling app requests (e.g., during boot when dozens of apps start simultaneously), new Binder calls from apps block until a thread frees up. This manifests as system-wide hangs: apps freeze, touch events stop being dispatched, and the Watchdog may trigger. Common culprits: (1) a system service making a synchronous Binder call to an app process that's slow to respond (holding a system_server Binder thread hostage), (2) ContentProvider queries that do disk I/O on the Binder thread, (3) PackageManager scanning during app install. The mitigation is to never make synchronous calls from system_server to app processes — use oneway Binder calls or post to a handler thread.

updateOomAdjLocked() and Transitive Bindings

// The most subtle corner case in OOM adj calculation:
// transitive binding chains.

// Process A (foreground activity, adj=0)
//   └─ binds to Process B (service, BIND_ABOVE_CLIENT)
//       └─ binds to Process C (service, BIND_IMPORTANT)
//           └─ binds to Process D (service, default flags)

// OOM adj propagation:
// A: adj=0 (foreground)
// B: adj=-1 (BIND_ABOVE_CLIENT → above client's adj)
//    → Clamped to 0 (can't go below FOREGROUND_APP_ADJ for non-system)
// C: adj=0 (BIND_IMPORTANT → same as client B)
// D: adj=1 (default binding → client adj + 1)

// The problem: if A stops, ALL of B, C, D lose their elevated adj.
// A single Activity finish can cascade oom_adj changes across
// dozens of processes. This is why updateOomAdjLocked() must
// walk the entire dependency graph on every state change.
Staff-Level Insight — Why AMS Is Android's Achilles' Heel

AMS (combined with ATMS) is the single most complex subsystem in Android. It manages the lifecycle of every component in every app, computes process priorities for the entire system, dispatches all broadcasts, and mediates all ContentProvider access. Despite the Android 10 split, the core challenge remains: global state with fine-grained locking. Every optimization in AMS (batched oom_adj updates, per-process broadcast queues, deferred delivery) is a response to the fundamental tension between correctness (every state change must be reflected immediately) and performance (recalculating everything on every change is too expensive). In interviews, demonstrating understanding of this tension — and the specific mechanisms AMS uses to manage it — is what separates L6 from L5 answers.

L6 Insight — Why PSI Is Superior

PSI-based lmkd is more responsive than legacy because it reacts to actual memory pressure, not just free memory thresholds. A device can have low free memory but no pressure (because the page cache is reclaimable). Conversely, a device can have "enough" free memory but high pressure (because of heavy I/O causing page faults). PSI captures the ground truth: how much time tasks actually spend stalled waiting for memory. This results in fewer unnecessary kills (better user experience) and faster response to genuine pressure (fewer OOM crashes).

ContentProvider death handling is one of the most nuanced areas of AMS — the behavior depends on whether the client holds a stable or unstable reference to the provider.

// Why the distinction?
// Stable reference: client depends on provider's transactional state.
//   If provider dies mid-transaction, client's state is inconsistent.
//   Safest to kill the client too and let it restart fresh.
//
// Unstable reference: client can handle provider death gracefully.
//   A query() result is read-only — if provider dies, client just gets
//   DeadObjectException and can retry the query.

// ContentResolver uses unstable providers for query() by default:
// ContentResolver.query() → acquireUnstableProvider() → IContentProvider.query()
// ContentResolver.insert() → acquireProvider() (STABLE) → IContentProvider.insert()

11.2 AMS Cleanup on Provider Death

// When a provider process dies, AMS calls:
fun removeDyingProviderLocked(proc: ProcessRecord, cpr: ContentProviderRecord) {
    // 1. Mark provider as dead
    cpr.provider = null

    // 2. Handle stable connections — kill those clients
    for (conn in cpr.connections) {
        if (conn.stableCount > 0) {
            // Client has stable reference → must die too
            if (conn.client != proc) {  // Don't kill the already-dead provider process
                conn.client.thread?.let {
                    // Kill the client process
                    Process.killProcess(conn.client.pid)
                }
            }
        }
        // 3. Set dead flag for unstable connections
        conn.dead = true
    }

    // 4. If provider was being launched, notify waiting clients
    if (cpr.launchingApp == proc) {
        cpr.launchingApp = null
        cpr.notifyAll()  // Wake up threads waiting in getContentProvider()
    }
}
L6 Insight — MediaStore Uses Unstable Providers

MediaStore uses acquireUnstableContentProvider() because the media scanner process (com.android.providers.media) can crash without killing the gallery app. If the gallery held a stable reference, a media scanner crash would kill the gallery — terrible UX. With unstable references, the gallery gets a DeadObjectException, shows an error, and retries when the media scanner restarts. This pattern is critical for any app that queries ContentProviders hosted in potentially unstable processes.

12. cleanUpApplicationRecordLocked() — Process Death Cleanup

When a process dies — whether killed by lmkd, crashed, or force-stopped — AMS must clean up all state associated with it. cleanUpApplicationRecordLocked() is the comprehensive cleanup method that handles this.

12.1 The Cleanup Cascade

// Source: frameworks/base/services/core/java/com/android/server/am/ActivityManagerService.java

fun cleanUpApplicationRecordLocked(
    app: ProcessRecord,
    restarting: Boolean,
    allowRestart: Boolean,
    index: Int,
    replacingPid: Boolean
) {
    // 1. ACTIVITIES — remove all ActivityRecords
    //    Delegates to ATMS: WindowProcessController.clearActivities()
    //    → Finishes all activities, removes from task stacks
    //    → May trigger task removal if last activity in task

    // 2. SERVICES — clean up all ServiceRecords
    mServices.killServicesLocked(app, allowRestart)
    //    → Stops all started services
    //    → Disconnects all bound services
    //    → Notifies clients via onServiceDisconnected()
    //    → May restart services with START_STICKY

    // 3. BROADCASTS — remove pending broadcast deliveries
    //    → Cancel any broadcasts queued for this process
    //    → If process was current receiver in ordered broadcast,
    //      skip to next receiver

    // 4. CONTENT PROVIDERS — clean up all ContentProviderRecords
    for (cpr in app.pubProviders.values) {
        removeDyingProviderLocked(app, cpr)
        //    → Kill stable clients (see Section 11)
        //    → Notify unstable clients
    }

    // 5. RECEIVERS — unregister all dynamic BroadcastReceivers
    for (rl in app.receivers) {
        removeReceiverLocked(rl)
    }

    // 6. ALARMS — cancel pending alarms
    //    AlarmManagerService.removeLocked(app.uid)

    // 7. JOBS — cancel pending jobs
    //    JobSchedulerService.cancelJobsForUid(app.uid)

    // 8. NOTIFICATIONS — remove notifications if process was force-stopped
    //    (not on normal death — notifications survive process death)

    // 9. OOM ADJ — recalculate for ALL processes that had bindings
    //    to the dead process. A single death can trigger cascading
    //    oom_adj changes across dozens of other processes.
    updateOomAdjLocked(OOM_ADJ_REASON_PROCESS_END)

    // 10. SEND BROADCAST — if force-stopped:
    //     Send ACTION_PACKAGE_RESTARTED so other apps can clean up
    //     their references to this package
}
L6 Insight — Cascading State Changes

cleanUpApplicationRecordLocked() is one of the most complex methods in AMS. A single process death can trigger cascading state changes across dozens of other processes: bound service clients get onServiceDisconnected(), stable ContentProvider clients are killed, oom_adj is recalculated for every process that had a binding to the dead process, and ordered broadcast chains are advanced. In extreme cases (e.g., a process that hosts providers used by 20 other apps), a single death can cause a "death cascade" where multiple processes are killed and restarted in rapid succession. This is why AMS batches oom_adj updates — processing each death individually would cause O(n²) recalculations.

12.2 Process Restart After Death

// After cleanup, AMS may restart the process if:
// 1. It had a Service with START_STICKY or START_REDELIVER_INTENT
//    → ActiveServices schedules a restart after a backoff delay
//    → Delay increases with each crash: 1s → 4s → 16s → ... (exponential)
//    → Max delay: 60 seconds

// 2. It had a persistent flag (android:persistent="true")
//    → Restarted immediately (only for system apps)

// 3. It's needed for a pending component (Activity, broadcast, etc.)
//    → Restarted on demand when the component is next requested

// AMS does NOT restart processes that:
// - Were force-stopped by the user
// - Had only cached/empty state (no active components)
// - Crashed too many times (crash loop detection)

13.2 The Dispatch Loop

// Source: frameworks/base/services/core/java/com/android/server/am/BroadcastQueue.java

fun processNextBroadcastLocked(fromMsg: Boolean, skipOomAdj: Boolean) {
    // 1. Deliver all parallel broadcasts first (no timeout per-receiver)
    while (mParallelBroadcasts.isNotEmpty()) {
        val r = mParallelBroadcasts.removeAt(0)
        for (i in 0 until r.receivers.size) {
            deliverToRegisteredReceiverLocked(r, r.receivers[i] as BroadcastFilter, false, i)
        }
    }

    // 2. Process ordered broadcasts one receiver at a time
    val r = mOrderedBroadcasts.firstOrNull() ?: return

    // 3. Check for timeout on current receiver
    if (r.receiverTime + mConstants.TIMEOUT > now) {
        // Still within timeout — wait
        return
    }

    // 4. TIMEOUT — current receiver is too slow
    broadcastTimeoutLocked(false)
    //    → Logs ANR for the RECEIVING process
    //    → Skips to next receiver
    //    → Current result propagated to next receiver
}

13.3 The goAsync() Trap

goAsync() Does NOT Extend the Timeout

A common misconception: BroadcastReceiver.goAsync() does not give you more time. It returns a PendingResult that lets you move processing off the main thread, but the same 10s/60s ANR timeout still applies. If you call goAsync() and never call PendingResult.finish(), the broadcast queue blocks until the timeout fires, then:

  • ANR is attributed to your process
  • For ordered broadcasts: the entire chain is stalled — no other receiver gets the broadcast until your timeout expires
  • The broadcast queue skips your receiver and moves to the next one
// The goAsync() leak pattern:
class LeakyReceiver : BroadcastReceiver() {
    override fun onReceive(context: Context, intent: Intent) {
        val pending = goAsync()  // Get PendingResult

        thread {
            doLongWork()
            // BUG: forgot to call pending.finish()
            // → Broadcast queue blocks for 10s/60s
            // → ANR attributed to THIS process
            // → For ordered broadcasts: blocks ALL subsequent receivers
        }
    }
}

// Correct pattern:
class CorrectReceiver : BroadcastReceiver() {
    override fun onReceive(context: Context, intent: Intent) {
        val pending = goAsync()

        CoroutineScope(Dispatchers.IO).launch {
            try {
                doWork()
            } finally {
                pending.finish()  // ALWAYS call finish()
            }
        }
    }
}

13.4 Ordered Broadcast Starvation

// Edge case: ordered broadcast with many receivers
// Receiver A (priority=100): takes 9 seconds (just under timeout)
// Receiver B (priority=50):  takes 9 seconds
// Receiver C (priority=0):   takes 9 seconds

// Total time: 27 seconds for ONE broadcast to complete
// No individual receiver ANRs (each is under 10s)
// But the broadcast queue is blocked for 27 seconds
// Other ordered broadcasts are STARVED

// This is why BroadcastModernQueue (Android 14) uses per-process queues:
// Slow receivers in Process A don't block delivery to Process B
L6 Insight — Debugging Broadcast ANRs in Production

When investigating broadcast ANRs from crash reports, check: (1) traces.txt for the receiving process's main thread stack — it shows what the receiver was doing when the timeout fired, (2) dumpsys activity broadcasts to see the current broadcast queue state and which receiver is active, (3) the BroadcastRecord.dispatchTime vs receiverTime to distinguish between "receiver is slow" (long gap between receiverTime and now) vs "dispatch is slow" (long gap between dispatchTime and receiverTime — indicates the process was being started). A common false positive: the ANR is attributed to Receiver B, but the real culprit is Receiver A which took 9.9 seconds, leaving Receiver B only 0.1 seconds before the cumulative timeout fires.
