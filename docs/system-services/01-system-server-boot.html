01 — system_server Boot & SystemServiceManager

From power button to a fully running Android — how system_server is born, how 100+ system services start in the right order, and what happens when things go wrong

1. The Android Boot Sequence — Power On to system_server

Every Android device follows the same fundamental boot chain. Understanding this sequence is critical because it explains why system_server exists as a single process hosting all framework services, and why its startup order is so carefully orchestrated.

  Android Boot Sequence — From Hardware to Framework
  ═══════════════════════════════════════════════════

  ┌─────────────┐
  │  POWER ON   │  Hardware reset vector
  └──────┬──────┘
         │
         ▼
  ┌─────────────┐  ROM code → loads bootloader from boot partition
  │ BOOTLOADER  │  (aboot / ABL on Qualcomm, cboot on Tegra)
  │  (Stage 1)  │  Verifies boot.img signature (Verified Boot / AVB)
  └──────┬──────┘  Sets up minimal hardware: DRAM, UART, storage
         │
         ▼
  ┌─────────────┐  Decompresses and loads kernel + ramdisk
  │ BOOTLOADER  │  Passes device tree (DTB) and kernel command line
  │  (Stage 2)  │  Hands off to kernel entry point
  └──────┬──────┘
         │
         ▼
  ┌─────────────┐  Mounts rootfs, initializes drivers, starts scheduler
  │   LINUX     │  Sets up SELinux in enforcing mode
  │   KERNEL    │  Launches PID 1: /init
  └──────┬──────┘
         │
         ▼
  ┌─────────────┐  Parses init.rc + device-specific .rc files
  │    INIT     │  Mounts filesystems (/system, /vendor, /data)
  │   (PID 1)   │  Starts property service, SELinux policy load
  │             │  Launches native daemons: servicemanager, surfaceflinger,
  └──────┬──────┘  logd, adbd, healthd, vold, etc.
         │
         │  Starts app_process64 with --zygote flag
         ▼
  ┌─────────────┐  ZygoteInit.main():
  │   ZYGOTE    │    1. Preloads ~15,000 classes (android.*, java.*, etc.)
  │  (PID ~200) │    2. Preloads shared libraries (ICU, OpenGL, etc.)
  │             │    3. Preloads resources (framework-res.apk drawables)
  │             │    4. Opens /dev/binder
  │             │    5. Creates server socket /dev/socket/zygote
  └──────┬──────┘    6. Calls forkSystemServer()
         │
         │  fork() — child becomes system_server
         ▼
  ┌─────────────┐  UID=1000 (system), GID=1000

  │ (PID ~500)  │  Runs SystemServer.main() → starts 100+ services
  └──────┬──────┘
         │
         │  After PHASE_BOOT_COMPLETED (1000):
         ▼
  ┌─────────────┐
  │  LAUNCHER   │  AMS starts the Home activity
  │   + APPS    │  Boot animation dismissed
  └─────────────┘  Device is ready for user interaction

1.1 init.rc — Service Definitions

Android's init process reads .rc files that define services and their startup triggers. Zygote is defined in init.zygote64_32.rc (or init.zygote64.rc on 64-bit-only devices):

// system/core/rootdir/init.zygote64_32.rc (simplified)

service zygote /system/bin/app_process64 -Xzygote /system/bin \
        --zygote --start-system-server --socket-name=zygote
    class main
    priority -20
    user root
    group root readproc reserved_disk
    socket zygote stream 660 root system
    socket usap_pool_primary stream 660 root system
    onrestart exec_background - system system -- /system/bin/vdc checkpoint resetCheckpoint
    onrestart write /sys/power/state on
    onrestart restart audioserver
    onrestart restart cameraserver
    onrestart restart media
    onrestart restart netd
    onrestart restart wificond

// Key observations:
// 1. --start-system-server flag tells Zygote to fork system_server
// 2. socket zygote — the Unix domain socket apps connect to for forking
// 3. onrestart — if Zygote dies, these services ALSO restart
//    This is the "runtime restart" — NOT a full kernel reboot

1.2 Zygote's Role — The Process Factory

Zygote preloads everything that app processes will need, so that fork() gives each child a pre-warmed address space via copy-on-write (COW). system_server is the first child Zygote forks.

// frameworks/base/core/java/com/android/internal/os/ZygoteInit.java

class ZygoteInit {
    fun main(argv: Array<String>) {
        // 1. Preload phase — shared across ALL forked processes
        preloadClasses()       // ~15,000 classes from /system/etc/preloaded-classes
        preloadResources()     // Framework drawables, layouts, colors
        preloadSharedLibraries() // libandroid_runtime.so, libicu, etc.
        preloadTextResources() // Hyphenation patterns, etc.

        // 2. Create the Zygote server socket
        val zygoteServer = ZygoteServer(isPrimaryZygote = true)

        // 3. Fork system_server if --start-system-server was passed
        if (startSystemServer) {
            val pid = forkSystemServer(
                uid = 1000,           // AID_SYSTEM
                gid = 1000,
                gids = intArrayOf(    // supplementary groups
                    1001, 1002, 1003, 1004, 1005, 1006, 1007,
                    1008, 1009, 1010, 1018, 1021, 1023, 1024,
                    1032, 1065, 3001, 3002, 3003, 3006, 3007,
                    3009, 3010, 3011
                ),
                runtimeFlags = 0,
                rlimits = null,
                permittedCapabilities = capabilities,
                effectiveCapabilities = capabilities
            )
            // Parent (Zygote) continues; child becomes system_server
        }

        // 4. Zygote enters its select loop, waiting for fork requests
        zygoteServer.runSelectLoop(abiList)
    }
}

1.3 forkSystemServer() — The Critical Fork

// frameworks/base/core/java/com/android/internal/os/ZygoteInit.java

private fun forkSystemServer(): Runnable? {
    // Arguments that will be passed to SystemServer
    val args = arrayOf(
        "--setuid=1000",
        "--setgid=1000",
        "--setgroups=1001,1002,...",
        "--capabilities=$capabilities,$capabilities",
        "--nice-name=system_server",
        "--runtime-args",
        "--target-sdk-version=${Build.VERSION_CODES.CUR_DEVELOPMENT}",
        "com.android.server.SystemServer"  // ← the class to invoke
    )

    val parsedArgs = ZygoteArguments(args)

    // Native fork via Zygote.forkSystemServer() → nativeForkSystemServer()
    // This calls fork() at the native level
    val pid = Zygote.forkSystemServer(
        parsedArgs.mUid, parsedArgs.mGid,
        parsedArgs.mGids, parsedArgs.mRuntimeFlags,
        null, parsedArgs.mPermittedCapabilities,
        parsedArgs.mEffectiveCapabilities
    )

    if (pid == 0) {
        // Child process — we ARE system_server now
        // Close the Zygote socket (child doesn't need it)
        zygoteServer.closeServerSocket()
        // Continue to SystemServer.main()
        return handleSystemServerProcess(parsedArgs)
    }

    // Parent (Zygote) — record child PID for death monitoring
    return null
}
L6 Insight — Why Fork Instead of exec()?

system_server is forked from Zygote, not exec()'d as a fresh process. This means it inherits Zygote's preloaded classes and resources via COW pages. On a typical device, this saves ~30MB of shared memory and ~2 seconds of class loading time. The trade-off: system_server's address space layout is predictable (same as Zygote), which is a minor ASLR concern — mitigated by the fact that system_server runs as UID 1000 with restricted capabilities, not as root.

2. system_server Startup — SystemServer.main()

Once forked, system_server's entry point is SystemServer.main(), which immediately calls run(). This method orchestrates the entire framework bring-up.

// frameworks/base/services/java/com/android/server/SystemServer.java

class SystemServer {
    companion object {
        @JvmStatic
        fun main(args: Array<String>) {
            SystemServer().run()
        }
    }

    private fun run() {
        // 1. Set system properties
        SystemProperties.set("persist.sys.dalvik.vm.lib.2", VMRuntime.getRuntime().vmLibrary())

        // 2. Enforce time — if system clock is before 1970, set to build timestamp
        if (System.currentTimeMillis() < EARLIEST_SUPPORTED_TIME) {
            SystemClock.setCurrentTimeMillis(EARLIEST_SUPPORTED_TIME)
        }

        // 3. Set default timezone if not set
        val timezoneProperty = SystemProperties.get("persist.sys.timezone")
        if (timezoneProperty.isNullOrEmpty()) {
            SystemProperties.set("persist.sys.timezone", "GMT")
        }

        // 4. Language/locale defaults
        if (!SystemProperties.get("persist.sys.language").isNullOrEmpty()) {
            // Migrate old locale properties to new format
        }

        // 5. Configure the VM for system_server
        VMRuntime.getRuntime().clearGrowthLimit()  // Remove heap growth limit
        Build.ensureFingerprintProperty()

        // 6. Enforce strict mode for system_server debugging
        Environment.setUserRequired(true)
        BaseBundle.setShouldDefuse(true)

        // 7. Ensure binder calls go to the foreground priority
        BinderInternal.disableBackgroundScheduling(true)
        BinderInternal.setMaxThreads(sMaxBinderThreads) // 31 threads

        // 8. Prepare the main Looper
        Looper.prepareMainLooper()
        Looper.getMainLooper().setSlowLogThresholdMs(
            SLOW_DISPATCH_THRESHOLD_MS,  // 10s
            SLOW_DELIVERY_THRESHOLD_MS   // 10s
        )

        // 9. Load native services library
        System.loadLibrary("android_servers")

        // 10. Initialize native services
        nativeInit()

        // 11. Create the system context
        createSystemContext()

        // 12. Create SystemServiceManager
        mSystemServiceManager = SystemServiceManager(mSystemContext)
        mSystemServiceManager.setStartInfo(
            mRuntimeRestart, mRuntimeStartElapsedTime, mRuntimeStartUptime
        )
        LocalServices.addService(SystemServiceManager::class.java, mSystemServiceManager)

        // 13. Start services in three phases
        startBootstrapServices()   // ~25 critical services
        startCoreServices()        // ~5 core services
        startOtherServices()       // ~80+ remaining services

        // 14. Start the Looper
        Looper.loop()
        throw RuntimeException("Main thread loop unexpectedly exited")
    }
}

3. SystemServiceManager — The Service Registry

SystemServiceManager is the orchestrator that instantiates, registers, and lifecycle-manages all system services. It's not the same as ServiceManager (the Binder registry) — it manages the Java objects within the system_server process.

3.1 Starting a Service

// frameworks/base/services/core/java/com/android/server/SystemServiceManager.java

class SystemServiceManager(private val context: Context) {
    private val services = ArrayList<SystemService>()

    // Primary way to start a service — by class
    fun <T : SystemService> startService(serviceClass: Class<T>): T {
        val name = serviceClass.name

        // 1. Instantiate via reflection — constructor must take (Context)
        val constructor = serviceClass.getConstructor(Context::class.java)
        val service = constructor.newInstance(context)

        // 2. Register in our internal list
        startService(service)
        return service
    }

    fun startService(service: SystemService) {
        // Add to the list of managed services
        services.add(service)

        // 3. Call onStart() — service publishes its Binder to ServiceManager here
        try {
            service.onStart()
        } catch (e: RuntimeException) {
            throw RuntimeException(
                "Failed to start service ${service.javaClass.name}: onStart threw", e
            )
        }
    }

    // Notify all services of a boot phase transition
    fun startBootPhase(t: TimingsTraceAndSlog, phase: Int) {
        for (service in services) {
            try {
                service.onBootPhase(phase)
            } catch (e: Exception) {
                throw RuntimeException(
                    "Failed to boot service ${service.javaClass.name}: " +
                    "onBootPhase($phase) threw", e
                )
            }
        }
    }
}

3.2 SystemService Base Class — Lifecycle Callbacks

// frameworks/base/services/core/java/com/android/server/SystemService.java

abstract class SystemService(val context: Context) {

    // Called when the service is first started — publish Binder interfaces here
    abstract fun onStart()

    // Called at each boot phase — initialize progressively
    open fun onBootPhase(phase: Int) {}

    // Multi-user lifecycle callbacks
    open fun onUserStarting(user: TargetUser) {}
    open fun onUserUnlocking(user: TargetUser) {}
    open fun onUserUnlocked(user: TargetUser) {}
    open fun onUserSwitching(from: TargetUser?, to: TargetUser) {}
    open fun onUserStopping(user: TargetUser) {}
    open fun onUserStopped(user: TargetUser) {}

    // Helper to publish a Binder service
    protected fun publishBinderService(name: String, service: IBinder) {
        ServiceManager.addService(name, service)
    }

    // Helper to publish a local (in-process) service
    protected fun  publishLocalService(type: Class<T>, service: T) {
        LocalServices.addService(type, service)
    }
}

3.3 Boot Phases — The Ordering Contract

Boot phases solve a fundamental dependency problem: services start in a fixed order, but many services need to interact with services that start later. Boot phases let services defer initialization until their dependencies are ready.

Phase Constant Meaning
100 PHASE_WAIT_FOR_DEFAULT_DISPLAY DisplayManagerService has provided the default display. Services that need display info can proceed.
480 PHASE_LOCK_SETTINGS_READY LockSettingsService is ready. Services that need credential-encrypted storage can proceed.
500 PHASE_SYSTEM_SERVICES_READY Core system services are ready. Most services register their receivers and observers here.
520 PHASE_DEVICE_SPECIFIC_SERVICES_READY Device-specific (vendor) services are ready.
550 PHASE_ACTIVITY_MANAGER_READY AMS is fully ready. Services can now interact with the activity lifecycle.
600 PHASE_THIRD_PARTY_APPS_CAN_START Third-party apps can now be launched. ContentProviders are queryable.
1000 PHASE_BOOT_COMPLETED Boot is complete. ACTION_BOOT_COMPLETED broadcast is sent. Home screen is visible. 
// Example: How a service uses boot phases for progressive initialization

class ExampleSystemService(context: Context) : SystemService(context) {

    override fun onStart() {
        // Minimal initialization — publish Binder interface
        publishBinderService("example", ExampleServiceImpl())
    }

    override fun onBootPhase(phase: Int) {
        when (phase) {
            PHASE_SYSTEM_SERVICES_READY -> {
                // NOW we can safely look up other services
                val am = context.getSystemService(Context.ACTIVITY_SERVICE)
                        as ActivityManager
                val pm = context.getSystemService(Context.POWER_SERVICE)
                        as PowerManager
                // Register broadcast receivers, content observers, etc.
            }
            PHASE_THIRD_PARTY_APPS_CAN_START -> {
                // NOW we can interact with app-installed content providers
                // or start binding to app services
            }
            PHASE_BOOT_COMPLETED -> {
                // Final initialization — everything is up
                // Schedule deferred work, start background tasks
            }
        }
    }
}
Common Interview Trap

Candidates often confuse SystemServiceManager (the Java-level lifecycle manager inside system_server) with ServiceManager (the Binder-level name registry that maps string names to IBinder objects). They serve completely different purposes: SystemServiceManager manages the lifecycle of SystemService objects; ServiceManager is the Binder directory that apps use to look up remote service proxies. A service is registered with both: SystemServiceManager manages its lifecycle, and ServiceManager publishes its Binder interface.

3.4 LocalServices — The In-Process Fast Path

// frameworks/base/core/java/com/android/server/LocalServices.java

// LocalServices is a simple in-process registry for system_server-internal APIs.
// No Binder overhead — direct Java method calls.

object LocalServices {
    private val services = ArrayMap<Class<*>, Any>()

    fun <T> addService(type: Class<T>, service: T) {
        synchronized(services) {
            check(!services.containsKey(type)) {
                "Override of existing service: $type"
            }
            services[type] = service as Any
        }
    }

    fun <T> getService(type: Class<T>): T? {
        synchronized(services) {
            @Suppress("UNCHECKED_CAST")
            return services[type] as? T
        }
    }
}

// Usage within system_server:
// val ams = LocalServices.getService(ActivityManagerInternal::class.java)
// This is a DIRECT Java call — no Binder transaction, no Parcel serialization.
// Much faster for intra-system_server communication.

4. Bootstrap Services — The Foundation

Bootstrap services are started first because every other service depends on them. If any bootstrap service fails to start, system_server crashes and the device reboots.

// frameworks/base/services/java/com/android/server/SystemServer.java

private fun startBootstrapServices() {
    // Watchdog FIRST — must monitor from the very beginning
    val watchdog = Watchdog.getInstance()
    watchdog.start()

    // Installer — talks to installd daemon for dexopt, mkdir, etc.
    val installer = mSystemServiceManager.startService(Installer::class.java)

    // DeviceIdentifiersPolicyService — controls access to device IDs
    mSystemServiceManager.startService(DeviceIdentifiersPolicyService::class.java)

    // UriGrantsManagerService — manages URI permission grants
    mSystemServiceManager.startService(UriGrantsManagerService.Lifecycle::class.java)

    // PowerManagerService — wake locks, power state; needed by almost everything
    mPowerManagerService = mSystemServiceManager
        .startService(PowerManagerService::class.java)

    // ThermalManagerService — thermal throttling
    mSystemServiceManager.startService(ThermalManagerService::class.java)

    // RecoverySystemService — OTA updates
    mSystemServiceManager.startService(RecoverySystemService.Lifecycle::class.java)

    // DisplayManagerService — must start before WMS/AMS
    // because they need display info during their initialization
    mDisplayManagerService = mSystemServiceManager
        .startService(DisplayManagerService::class.java)

    // ── Phase 100: WAIT_FOR_DEFAULT_DISPLAY ──
    mSystemServiceManager.startBootPhase(PHASE_WAIT_FOR_DEFAULT_DISPLAY)

    // PackageManagerService — scans all APKs, resolves permissions
    // This is one of the most time-consuming steps (~3-5s on first boot)
    mPackageManagerService = PackageManagerService.main(
        mSystemContext, installer,
        domainVerificationService,
        mFactoryTestMode != FactoryTest.FACTORY_TEST_OFF
    )

    // UserManagerService — multi-user support
    mSystemServiceManager.startService(UserManagerService.LifeCycle::class.java)

    // ActivityTaskManagerService — Activity lifecycle, task management
    // Split from AMS in Android 10 for better separation of concerns
    val atm = mSystemServiceManager
        .startService(ActivityTaskManagerService.Lifecycle::class.java)
        .service

    // ActivityManagerService — process management, broadcasts, providers
    mActivityManagerService = ActivityManagerService.Lifecycle
        .startService(mSystemServiceManager, atm)

    // Set AMS as the system process
    mActivityManagerService.setSystemProcess()

    // SensorPrivacyService — sensor toggle (camera/mic kill switch)
    mSystemServiceManager.startService(SensorPrivacyService::class.java)
}

7. ServiceManager Registration — The Binder Directory

While SystemServiceManager manages lifecycle within system_server, ServiceManager is the global Binder directory that lets any process look up a service by name.

// How a system service registers its Binder interface:

class ActivityManagerService : IActivityManager.Stub() {

    fun setSystemProcess() {
        // Register with ServiceManager — apps can now find us
        ServiceManager.addService(
            Context.ACTIVITY_SERVICE,  // "activity"
            this,                       // the Binder object
            true,                       // allowIsolated
            DUMP_FLAG_PRIORITY_CRITICAL or DUMP_FLAG_PRIORITY_NORMAL or DUMP_FLAG_PROTO
        )

        // Also register sub-services
        ServiceManager.addService("procstats", mProcessStats)
        ServiceManager.addService("meminfo", MemBinder(this))
        ServiceManager.addService("gfxinfo", GraphicsBinder(this))
        ServiceManager.addService("dbinfo", DbBinder(this))
        ServiceManager.addService("cpuinfo", CpuBinder(this))
    }
}

// How an app looks up a service:

// Layer 1: Raw Binder lookup (rarely used directly by apps)
val binder: IBinder = ServiceManager.getService("activity")
val ams: IActivityManager = IActivityManager.Stub.asInterface(binder)

// Layer 2: Context.getSystemService() — the standard app API
// This caches the proxy object so repeated calls don't re-query ServiceManager

8. Watchdog — Detecting system_server Hangs

The Watchdog is a critical safety mechanism that detects when system_server's threads are stuck. If the main thread (or any monitored thread) is blocked for more than 60 seconds, Watchdog kills system_server, triggering a runtime restart.

// frameworks/base/services/core/java/com/android/server/Watchdog.java

class Watchdog : Thread("watchdog") {
    companion object {
        const val DEFAULT_TIMEOUT = 60_000L   // 60 seconds
        const val HALF_TIMEOUT = DEFAULT_TIMEOUT / 2  // 30s — warning threshold
    }

    // Monitored handler threads
    private val handlerCheckers = ArrayList<HandlerChecker>()

    init {
        // Monitor these critical threads:
        addThread(handler = FgThread.getHandler(),     name = "foreground thread")
        addThread(handler = MainThread.getHandler(),   name = "main thread")
        addThread(handler = UiThread.getHandler(),     name = "ui thread")
        addThread(handler = IoThread.getHandler(),     name = "i/o thread")
        addThread(handler = DisplayThread.getHandler(), name = "display thread")
        addThread(handler = AnimationThread.getHandler(), name = "animation thread")
        addThread(handler = SurfaceAnimationThread.getHandler(),
                  name = "surface animation thread")
    }

    override fun run() {
        while (true) {
            // 1. Post a check message to each monitored thread
            for (checker in handlerCheckers) {
                checker.scheduleCheckLocked()
            }

            // 2. Wait for HALF_TIMEOUT (30s)
            Thread.sleep(HALF_TIMEOUT)

            // 3. Evaluate results
            val waitState = evaluateCheckerCompletionLocked()

            when (waitState) {
                COMPLETED -> continue  // All threads responded — healthy

                WAITING -> continue    // Still within timeout — wait more

                WAITED_HALF -> {
                    // 30s elapsed, some threads haven't responded
                    // Log a warning + collect stack traces
                    // (This is the "half watchdog" — not fatal yet)
                    ActivityManagerService.dumpStackTraces(
                        pids, processCpuTracker, null, null
                    )
                }

                OVERDUE -> {
                    // 60s elapsed — thread is STUCK
                    // This is FATAL for system_server
                    val subject = "Watchdog: ${describeBlockedThreads()}"

                    // Dump all thread stacks to /data/anr/traces.txt
                    ActivityManagerService.dumpStackTraces(
                        pids, processCpuTracker, null, null
                    )

                    // Write to dropbox for later analysis
                    doSysRq('w')  // Show blocked tasks
                    doSysRq('l')  // Show backtrace of all active CPUs

                    // KILL system_server
                    Process.killProcess(Process.myPid())
                    System.exit(10)
                }
            }
        }
    }
}

            // Just acquiring the lock is the health check
        }

Trigger Root Cause How to Debug

Watchdog Kill → Runtime Restart

When Watchdog kills system_server, the consequences cascade: Zygote detects its child (system_server) has died, kills all app processes, and restarts itself. This triggers a runtime restart (also called "soft reboot") — the Linux kernel stays running, but the entire Android framework restarts from Zygote. Users see the boot animation again. This is NOT a full hardware reboot — the kernel, init, and native daemons survive. On a Pixel device, a runtime restart takes ~15-20 seconds vs ~45-60 seconds for a full reboot.

9. Staff-Level Corner Cases

9.1 system_server Crash Recovery

// When system_server dies (Watchdog kill, uncaught exception, OOM):

// 1. Zygote detects child death via waitpid()
//    (Zygote is the PARENT process of system_server)

// 2. Zygote's reaction (ZygoteInit.java):
//    - Kills ALL app processes (they can't function without system services)
//    - Zygote itself exits

// 3. init detects Zygote death (init monitors all services)
//    - Triggers the "onrestart" commands from init.zygote64_32.rc:
//      → restart audioserver, cameraserver, media, netd, wificond
//    - Restarts Zygote

// 4. New Zygote:
//    - Preloads classes/resources again
//    - Forks a NEW system_server
//    - system_server starts all services from scratch

// 5. Apps are re-launched as needed:
//    - Launcher is started first (Home intent)
//    - Other apps are started on demand
//    - Apps with saved state can restore via onSaveInstanceState/ViewModel

// Total time: ~15-20 seconds on modern hardware
// The kernel, init, and native daemons (surfaceflinger, etc.) survive

// This is why Android apps MUST handle process death gracefully —
// a system_server crash kills every app process on the device.

9.2 Boot Time Optimization

// Google has invested heavily in reducing boot time. Key techniques:

// 1. Parallel service startup
//    Some services in startOtherServices() are started on background threads:

// 4. PackageManagerService optimization
//    PMS is the biggest bottleneck on first boot (scanning all APKs).
//    Optimizations:
//    - Parallel package scanning (multi-threaded dexopt)

L6 Insight — system_server Memory and GC Impact

system_server runs with a 512MB+ heap (configurable via dalvik.vm.heapsize for system_server). Because it hosts 100+ services with complex object graphs, GC pressure is significant. A major GC pause in system_server can cause system-wide jank — every app's Binder calls to system services will stall during the pause. This is why ART uses the concurrent copying (CC) collector with read barriers: it minimizes pause times to <1ms for most collections. However, under extreme memory pressure (e.g., after an OTA when PMS is rescanning all packages), system_server can hit multi-millisecond pauses. Google monitors system_server GC metrics in production via BinderCallsStatsService and LooperStats to detect regressions.

// 1. @PublicApi (SDK) — available to all apps

9.5 Multi-User Boot

// Android supports multiple users (profiles, work profiles, guest).
// system_server must handle user lifecycle carefully:

// User 0 (system user) is ALWAYS started during boot.
// Secondary users are started on demand (user switch) or at boot
// (if "start user in background" is configured).

// The boot sequence for users:
// 1. PHASE_BOOT_COMPLETED fires
// 2. AMS calls UserController.startUser(userId=0)
// 3. SystemServiceManager notifies all services:
//    - onUserStarting(user=0)
//    - onUserUnlocking(user=0)  (after credential entry)
//    - onUserUnlocked(user=0)   (CE storage available)

// When a secondary user is switched to:
// 4. onUserSwitching(from=0, to=10)
// 5. onUserStarting(user=10)
// 6. onUserUnlocking(user=10)
// 7. onUserUnlocked(user=10)

// Services MUST handle per-user state correctly:
// - PMS: separate installed packages per user
// - NotificationManagerService: separate notification channels per user
// - AccountManagerService: separate accounts per user
// - ContentService: separate sync adapters per user

// Common bug: services that assume userId=0 and break on multi-user devices.
// Work profiles (managed profiles) are especially tricky because they
// run concurrently with the parent user.

// If all 31 threads are blocked (e.g., waiting for a lock held by the

L6 Insight — Boot Phase Ordering Bugs

One of the most subtle bugs in system_server development is a service trying to use another service before the right boot phase. For example, a service that registers a ContentObserver in onStart() instead of onBootPhase(PHASE_SYSTEM_SERVICES_READY) will crash because ContentService isn't ready yet. These bugs are hard to reproduce because they depend on exact startup timing — they might work on a fast device but fail on a slow one where services start in a slightly different order. The fix is always the same: defer initialization to the appropriate boot phase. Google enforces this with strict testing on slow emulators during CTS.

Key Takeaways for Interview
  • Boot chain: Bootloader → Kernel → init → Zygote → system_server → Launcher
  • system_server is forked from Zygote (COW optimization), runs as UID 1000
  • Three startup phases: bootstrap (critical deps), core (infrastructure), other (everything else)
  • SystemServiceManager manages Java lifecycle; ServiceManager is the Binder directory
  • Boot phases (100→1000) solve service dependency ordering
  • LocalServices provides fast in-process communication between system services
  • Watchdog kills system_server after 60s thread hang → runtime restart (not full reboot)
  • Mainline modules allow updating services independently via Google Play

10. Failure Modes & Recovery

Understanding how system_server fails — and how the system recovers — is critical for diagnosing production issues and designing resilient platform code.

10.1 Bootstrap Service Crash in onStart()

Fatal — Runtime Restart

If a bootstrap service throws in onStart(), SystemServiceManager.startService() re-throws as a RuntimeException, which is uncaught on the main thread. system_server crashes. Zygote detects its child's death via waitpid(), kills all app processes, and exits. init detects Zygote's death and restarts it. The device appears to "reboot" but the kernel stays running — this is a runtime restart, not a full reboot.

// SystemServiceManager.startService() — the crash path:
fun startService(service: SystemService) {
    services.add(service)
    try {
        service.onStart()
    } catch (e: RuntimeException) {
        // This propagates up to SystemServer.run() → uncaught → process death
        throw RuntimeException(
            "Failed to start service ${service.javaClass.name}: onStart threw", e
        )
    }
}

// Recovery chain:
// system_server crashes
//   → Zygote waitpid() returns
//     → Zygote kills all app processes (SIGKILL)
//       → Zygote exits
//         → init detects Zygote death
//           → init triggers onrestart commands (audioserver, cameraserver, etc.)
//             → init restarts Zygote
//               → Zygote preloads, forks new system_server
//                 → Boot animation plays again (~15-20s total)

10.2 Boot Phase Crash

// If a service throws in onBootPhase(), the same crash cascade occurs:
fun startBootPhase(t: TimingsTraceAndSlog, phase: Int) {
    for (service in services) {
        try {
            service.onBootPhase(phase)
        } catch (e: Exception) {
            throw RuntimeException(
                "Failed to boot service ${service.javaClass.name}: " +
                "onBootPhase($phase) threw", e
            )
        }
    }
}
Defensive Boot Phase Code

Boot phase code must be defensive — catch exceptions, log, and degrade gracefully rather than crashing. A single uncaught exception in any service's onBootPhase() takes down the entire framework. In practice, Google's CTS tests run on slow emulators specifically to catch timing-dependent boot phase failures.

11. USAP (Unspecialized App Process) Pool

Android 10 introduced the USAP pool — pre-forked processes from Zygote that sit idle until an app needs to launch. This is one of the most impactful cold-start optimizations in modern Android.

11.1 How USAP Works

  USAP Pool Architecture
  ═══════════════════════

  ┌──────────────┐
  │    Zygote     │
  │  (PID ~200)   │
  └──────┬───────┘
         │ Pre-forks N processes during idle time
         │
         ├──→ ┌──────────┐  ┌──────────┐  ┌──────────┐
         │    │  USAP 1  │  │  USAP 2  │  │  USAP 3  │
         │    │ (idle)   │  │ (idle)   │  │ (idle)   │
         │    │ ~2-4 MB  │  │ ~2-4 MB  │  │ ~2-4 MB  │
         │    └──────────┘  └──────────┘  └──────────┘
         │         Pool of pre-warmed processes
         │
         │  AMS: "Start com.example.app"
         │
         ├──→ Grab USAP 1 (instead of fork())
         │    │
         │    ├─ Specialize: set UID, GID, SELinux context
         │    ├─ Load app's classes and resources
         │    └─ Becomes the app process
         │
         └──→ Refill pool: fork a replacement USAP

11.2 Configuration and Tuning

// USAP pool system properties:
// persist.device_config.runtime_native.usap_pool_enabled     — enable/disable (default: true on Android 10+)
// persist.device_config.runtime_native.usap_pool_size_max    — max pool size (default: 3)
// persist.device_config.runtime_native.usap_pool_size_min    — min pool size (default: 1)
// persist.device_config.runtime_native.usap_refill_threshold — refill when pool drops below this

// - system_server (always a direct fork with special capabilities)
// - Isolated processes (WebView sandbox — need special UID allocation)
// - App Zygote processes (custom preloading)
L6 Insight — USAP Performance Impact

USAP saves ~20-50ms per app cold start by eliminating the fork() latency from Zygote. On a busy device where Zygote is handling multiple fork requests sequentially (Zygote is single-threaded for fork), the savings compound: without USAP, the 5th app in a launch queue waits for 4 forks to complete. With USAP, each app grabs a pre-forked process instantly. The trade-off is memory: each idle USAP consumes ~2-4MB (COW pages from Zygote that haven't been touched yet). On low-RAM devices, the pool size is reduced or disabled entirely. USAP is the primary reason cold start times improved significantly in Android 10+.

12. SELinux Domain Transition During Boot

SELinux (Security-Enhanced Linux) enforces mandatory access control at the kernel level. Every process runs in a specific SELinux domain that determines what resources it can access.

12.1 Domain Transitions at Fork

Process SELinux Domain Key Permissions
Zygote u:r:zygote:s0 Fork processes, preload classes, bind to Zygote socket
system_server u:r:system_server:s0 Register services with ServiceManager, access /dev/binder, read/write system data dirs
Regular app u:r:untrusted_app:s0:c[uid] Access own data dir, use granted permissions, Binder IPC to services
Platform-signed app u:r:platform_app:s0 Access platform APIs, broader filesystem access
Privileged app u:r:priv_app:s0 System APIs, can hold signature-level permissions 
// Domain transition happens during Zygote.forkAndSpecialize():
// 1. Zygote calls fork() — child inherits u:r:zygote:s0
// 2. Child calls selinux_android_setcontext():
//    - For system_server: transitions to u:r:system_server:s0
//    - For apps: transitions to u:r:untrusted_app:s0 (or platform_app, priv_app)
// 3. Kernel enforces the new domain immediately
//    - system_server can now register services, access system dirs
//    - Apps CANNOT register system services or access system_server files

// The transition is irreversible — a process cannot change its own domain
// after specialization (no privilege escalation possible)

12.2 service_contexts — ServiceManager Access Control

// /system/etc/selinux/plat_service_contexts maps service names to SELinux types:
//
// activity                u:object_r:activity_service:s0
// window                  u:object_r:window_service:s0
// package                 u:object_r:package_service:s0
// notification            u:object_r:notification_service:s0
// connectivity            u:object_r:connectivity_service:s0

// ServiceManager checks this on EVERY addService() and getService() call:
// 1. Caller's SELinux domain is checked against the service's SELinux type
// 2. Only system_server (u:r:system_server:s0) can addService() for system services
// 3. Apps can getService() for services they have SELinux permission to access
// 4. Denied access → SecurityException + SELinux audit log (avc: denied)
SELinux Debugging

SELinux denials are logged to logcat with the tag avc. Use adb logcat | grep avc to find them. The denial message includes the source domain, target type, and denied permission — enough to write a policy rule. In userdebug builds, you can temporarily set SELinux to permissive mode (adb shell setenforce 0) to test whether SELinux is the root cause of a failure.

L6 Insight — Boot Time as a Release Gate

Google runs automated boot-time benchmarks on every Android build. The key metric is time from Zygote fork to PHASE_BOOT_COMPLETED. A regression of even 500ms triggers investigation. The biggest bottlenecks are typically: (1) PMS package scanning on first boot (~3-5s), (2) dex2oat compilation during OTA (~10-30s on first boot after update), (3) service onStart() methods that do synchronous I/O. Platform engineers use Perfetto traces to identify which service is slow and optimize it — often by deferring work to a later boot phase or moving it to a background thread.

14. PROC_START_TIMEOUT — Process Health Check

When AMS asks Zygote to fork a new app process, it sets a timeout for the process to "check in" by calling attachApplicationLocked(). If the process doesn't attach in time, AMS kills it.

14.1 Timeout Values

// Source: frameworks/base/services/core/java/com/android/server/am/ActivityManagerService.java

companion object {
    const val PROC_START_TIMEOUT = 10_000L          // 10 seconds (normal)
    const val PROC_START_TIMEOUT_WITH_WRAPPER = 1200_000L  // 20 minutes (with debug wrapper)
}

// The timeout is also extended during boot:
// - During system boot: effectively longer because AMS is lenient
// - Under debugger: 120 seconds (to allow breakpoint attachment)
// - On slow/emulator devices: OEMs may increase this

14.2 What Happens on Timeout

// AMS posts a delayed message when starting a process:
mHandler.sendMessageDelayed(
    mHandler.obtainMessage(PROC_START_TIMEOUT_MSG, app),
    PROC_START_TIMEOUT
)

// If attachApplicationLocked() is called in time:
//   → Message is removed, timeout cancelled

// If timeout fires (process didn't attach):
fun processStartTimedOutLocked(app: ProcessRecord) {
    // 1. Log the failure
    Slog.e(TAG, "Process ${app.processName} failed to attach in time")

    // 2. Kill the stuck process
    Process.killProcessQuiet(app.pid)

    // 3. Clean up the ProcessRecord
    //    (but DON'T remove it — AMS may retry starting the process)

    // 4. If this was for an Activity, show "App isn't responding" dialog
    // 5. If this was for a Service, the caller gets a failure callback
}

14.3 Common Causes of Timeout

Cause Why It's Slow Mitigation
Heavy Application.onCreate() Synchronous initialization of SDKs, databases, DI frameworks Defer to background thread, use App Startup library
ContentProvider initialization Providers init before Application.onCreate() — each one adds latency Minimize providers, use lazy initialization
Slow class loading / verification Large DEX files, missing Baseline Profiles → interpreted execution Baseline Profiles, R8 optimization, reduce DEX size
Disk I/O during startup SharedPreferences.apply() blocking, database migrations Async I/O, deferred migrations
Zygote contention Multiple processes forking simultaneously (boot time) USAP pool, staggered process starts
L6 Insight — PROC_START_TIMEOUT vs ANR

This timeout is separate from ANR detection. ANR monitors whether a running process responds to input or completes broadcasts/services in time. PROC_START_TIMEOUT monitors whether a newly forked process successfully initializes and connects back to AMS. A process can pass the start timeout (attach successfully) but then immediately ANR if its main thread is blocked. In production crash reports, a PROC_START_TIMEOUT failure appears as "Process didn't attach" in logcat, while ANR appears as "Input dispatching timed out" or "Broadcast of Intent timed out".
