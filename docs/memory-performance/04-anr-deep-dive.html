04 — ANR Deep Dive

ANR taxonomy, AOSP internals (InputDispatcher, AMS timeouts), traces.txt analysis, and prevention strategies

1. ANR Taxonomy — Every Timeout the System Enforces

An Application Not Responding (ANR) is triggered when the system detects that an app's main thread has failed to process a message within a deadline. The system enforces different timeouts depending on the component type and its foreground/background state.

ANR Type Timeout Trigger Condition AOSP Enforcer
Input Dispatch 5 s Key or touch event not consumed; no finishInputEvent() callback InputDispatcher.cpp (native, system_server)
BroadcastReceiver (foreground) 10 s onReceive() doesn't return within deadline BroadcastQueue.java
BroadcastReceiver (background) 60 s Same, but for broadcasts sent to background queue BroadcastQueue.java
Service (foreground) 20 s onCreate() / onStartCommand() / onBind() doesn't complete ActiveServices.java
Service (background) 200 s Same, but for services in background processes ActiveServices.java
ContentProvider 10 s Provider publish timeout — ContentProvider.onCreate() doesn't return during app startup ActivityManagerService.java
JobService ~10 min (soft) Job execution limit exceeded; onStopJob() called, then force-stop if unresponsive JobServiceContext.java
Watch Out — Background Timeouts Are Generous but Still Fatal

The 200s background service timeout and 60s background broadcast timeout are much longer, but they still produce ANRs visible in Play Vitals. Users on slow devices hit these more often than you'd expect — especially during cold starts when the system is under memory pressure and your onCreate() chain blocks on disk I/O.

2. ANR Internals — How the System Detects ANRs

Each ANR type has a distinct detection mechanism inside system_server. Understanding these paths is critical for diagnosing why an ANR was triggered and what the system was waiting for.

2.1 Input Dispatch Timeout — InputDispatcher.cpp

// InputDispatcher lives in system_server's native layer.
// Source: frameworks/native/services/inputflinger/dispatcher/InputDispatcher.cpp

// Flow:
// 1. InputReader reads raw events from /dev/input/* (EventHub)
// 2. InputDispatcher receives processed events via InputListener
// 3. Dispatcher finds the focused window (via WindowManagerService)
// 4. Dispatcher sends the event to the app via an InputChannel (Unix socket pair)
// 5. Dispatcher starts a timer: DEFAULT_INPUT_DISPATCHING_TIMEOUT (5s)
// 6. App's main thread Looper picks up the event → ViewRootImpl → View hierarchy
// 7. App calls finishInputEvent() → signal sent back through InputChannel
// 8. Dispatcher cancels the timer

// If step 7 never happens within 5s:
// - InputDispatcher::onAnrLocked() is called
// - It notifies InputManagerService (Java) via JNI callback
// - InputManagerService calls AMS.inputDispatchingTimedOut()
// - AMS delegates to AppErrors.appNotResponding()

// Key detail: the timer starts when the event is DISPATCHED to the app,
// not when the event arrives at InputDispatcher. If the app's window
// is not focused or the previous event hasn't been consumed yet,
// the dispatcher WAITS (doesn't start the timer until dispatch).

// This means: if your main thread is blocked BEFORE the event arrives,
// the 5s timer starts only when InputDispatcher actually sends the event.
// The real wall-clock delay the user experiences can be much longer.

2.2 Broadcast Timeout — BroadcastQueue.java

// Source: frameworks/base/services/core/java/com/android/server/am/BroadcastQueue.java

// Two queues: foreground (10s) and background (60s)
// Ordered broadcasts are delivered sequentially; each receiver gets a timeout.

// Flow:
// 1. AMS.broadcastIntentLocked() enqueues BroadcastRecord
// 2. BroadcastQueue.processNextBroadcastLocked() picks next receiver
// 3. Sets alarm: setBroadcastTimeoutLocked(timeoutTime)
//    → Handler.sendMessageAtTime(BROADCAST_TIMEOUT_MSG, timeoutTime)
// 4. Delivers to app via IApplicationThread.scheduleReceiver()
// 5. App's main thread runs onReceive() in ReceiverDispatcher
// 6. App calls AMS.finishReceiver() when done
// 7. BroadcastQueue cancels the timeout alarm

// If finishReceiver() never comes:
// 8. BROADCAST_TIMEOUT_MSG fires → broadcastTimeoutLocked()
// 9. Checks if the app is still processing (hasn't finished)
// 10. Calls AMS → AppErrors.appNotResponding()

// Gotcha: for manifest-registered receivers, the system must LAUNCH
// the process first. The timeout includes process start time.
// On a cold start under memory pressure, just Zygote fork + class loading
// can eat 3-5 seconds of your 10-second budget.

2.3 Service Timeout — ActiveServices.java

// Source: frameworks/base/services/core/java/com/android/server/am/ActiveServices.java

// Flow:
// 1. AMS calls ActiveServices.realStartServiceLocked()
// 2. Calls bumpServiceExecutingNoBatteryStatsLocked() which posts:
//    → SERVICE_TIMEOUT_MSG with delay = SERVICE_TIMEOUT (20s or 200s)
// 3. Delivers to app via IApplicationThread.scheduleCreateService()
// 4. App's main thread calls Service.onCreate() then onStartCommand()/onBind()
// 5. App calls AMS.serviceDoneExecuting() when lifecycle method returns
// 6. ActiveServices cancels the timeout message

// If serviceDoneExecuting() never comes:
// 7. SERVICE_TIMEOUT_MSG fires → serviceTimeout()
// 8. Calls AppErrors.appNotResponding()

// The timeout covers the ENTIRE lifecycle callback chain:
// onCreate() + onStartCommand() must both complete within the window.
// If onCreate() takes 15s and onStartCommand() takes 6s = 21s → ANR.

2.4 The ANR Dialog Flow — AppErrors.java

// Source: frameworks/base/services/core/java/com/android/server/am/AppErrors.java

// AppErrors.appNotResponding(ProcessRecord app, ...) does:
// 1. Dumps thread stacks → /data/anr/traces.txt (via dumpStackTraces())
//    - Dumps the offending process FIRST

//    - Uses Process.sendSignal(pid, SIGNAL_QUIT) → SIGQUIT triggers VM dump

// On Android 11+, multiple ANR trace files are kept:

3. traces.txt Analysis — Reading ANR Thread Dumps

When an ANR occurs, the system sends SIGQUIT to the offending process, which triggers ART to dump all thread stacks to /data/anr/traces.txt. Knowing how to read this file is the single most important ANR debugging skill.

3.1 Thread Dump Format

// Each thread entry looks like this:

  | group="main" sCount=1 ucsCount=0 flags=1 obj=0x72a01e18 self=0xb400007a
  | sysTid=12345 nice=-10 cgrp=top-app sched=0/0 handle=0x7b1234
  | state=S schedstat=( 12345678 9876543 210 ) utm=100 stm=50 core=3 HZ=100
  | stack=0x7fc0000000-0x7fc0002000 stackSize=8192KB
  | held mutexes=

  at android.app.Activity.performCreate(Activity.java:8051)

  at android.app.servertransaction.LaunchActivityItem.execute(LaunchActivityItem.java:85)
  at android.app.ActivityThread.main(ActivityThread.java:7656)

// Key fields to examine:
// "main"       → thread name ("main" = UI thread — the one that matters for ANRs)
// tid=1        → ART thread ID (tid=1 is always the main thread)
// Blocked      → thread STATE — this is the critical field
// sysTid=12345 → Linux kernel thread ID (for systrace/perfetto correlation)
// waiting to lock <0x...> held by thread 15 → LOCK CONTENTION

3.2 Thread States — What They Mean for ANRs

State Meaning ANR Implication
Runnable Thread is executing or ready to execute CPU-bound work on main thread (heavy computation, layout inflation)
Blocked Waiting to acquire a monitor lock Lock contention — another thread holds the lock
Waiting Waiting on Object.wait() with no timeout Waiting for a condition that may never come (potential deadlock)
TimedWaiting Waiting with a timeout (Thread.sleep(), Object.wait(timeout)) Sleeping on main thread, or waiting for a Binder reply with timeout
Native Executing in native code (JNI) Blocked in a native call — Binder IPC, file I/O, SQLite, DNS lookup
Suspended Suspended by debugger or GC GC pause (rare on ART with concurrent GC, but possible under heap pressure)

  - waiting to lock <0x0d4e5f6a> (a java.lang.Object) held by thread 22
  at com.example.app.HomeFragment.onViewCreated(HomeFragment.kt:45)
  at androidx.fragment.app.Fragment.performViewCreated(Fragment.java:3128)

"CacheWriter-1" prio=5 tid=22 Runnable
  at com.example.app.CacheManager.writeToDisk(CacheManager.kt:89)
  - locked <0x0d4e5f6a> (a java.lang.Object)
  at com.example.app.CacheManager.persist(CacheManager.kt:72)
  ...

// Analysis:
// 1. Main thread (tid=1) is in state "Blocked"
// 2. It's trying to lock object 0x0d4e5f6a
// 3. That lock is held by thread 22 ("CacheWriter-1")
// 4. Thread 22 is "Runnable" — doing disk I/O in writeToDisk()
// 5. Main thread is stuck until the disk write completes

// Use a ConcurrentHashMap or copy-on-write pattern instead.

4. Common ANR Causes — With Code Examples

4.1 Synchronous Binder Call on Main Thread

// BAD: Synchronous IPC on main thread
class MainActivity : AppCompatActivity() {
    override fun onCreate(savedInstanceState: Bundle?) {
        super.onCreate(savedInstanceState)

        // getInstalledPackages() is a Binder call to PackageManagerService.
        // On a device with 200+ apps, this can take 2-8 seconds.
        val packages = packageManager.getInstalledPackages(0) // ANR risk!

        // ContentResolver.query() is a Binder call to the ContentProvider's process.
        // If the provider is in another process, this is cross-process IPC.
        val cursor = contentResolver.query(
            ContactsContract.Contacts.CONTENT_URI,
            null, null, null, null
        ) // ANR risk!
    }
}

// GOOD: Move to background
class MainActivity : AppCompatActivity() {
    override fun onCreate(savedInstanceState: Bundle?) {
        super.onCreate(savedInstanceState)
        setContentView(R.layout.activity_main)

        lifecycleScope.launch {
            val packages = withContext(Dispatchers.IO) {
                packageManager.getInstalledPackages(0)
            }
            updateUI(packages)
        }
    }
}

4.2 SharedPreferences.commit() on Main Thread

// BAD: commit() is synchronous — writes to disk and BLOCKS until done
class SettingsActivity : AppCompatActivity() {
    fun saveUserPreference(key: String, value: String) {
        val prefs = getSharedPreferences("settings", MODE_PRIVATE)
        prefs.edit().putString(key, value).commit() // Blocks main thread!
        // commit() calls:
        // 1. commitToMemory() — fast, in-memory
        // 2. enqueueDiskWrite() with isSync=true
        // 3. writeToFile() — fsync() to disk — can take 50-500ms on slow storage
    }
}

// GOOD: apply() is asynchronous — writes to memory immediately, disk later
class SettingsActivity : AppCompatActivity() {
    fun saveUserPreference(key: String, value: String) {
        val prefs = getSharedPreferences("settings", MODE_PRIVATE)
        prefs.edit().putString(key, value).apply()
        // apply() calls:
        // 1. commitToMemory() — instant
        // 2. enqueueDiskWrite() with isSync=false
        // 3. Disk write happens on QueuedWork's HandlerThread
    }
}

// DANGER: apply() has its own ANR trap!
// ActivityThread.handleStopActivity() calls QueuedWork.waitToFinish()
// which blocks the main thread until ALL pending apply() writes complete.
// If you call apply() many times rapidly, the pending writes queue up,
// and the next Activity stop/pause blocks until they all flush.
// This is a VERY common ANR cause that most developers don't know about.
Danger — apply() Can Still Cause ANRs

SharedPreferences.apply() queues disk writes on a background thread, but ActivityThread.handleStopActivity() and handlePauseActivity() call QueuedWork.waitToFinish(), which blocks the main thread until all pending apply() writes are flushed. If you batch many apply() calls (e.g., in a loop), the next activity transition blocks until all writes complete. On Android 8+, Google optimized this to only wait for the last write, but on older devices this is a major ANR source. Consider DataStore or batching edits into a single apply().

4.3 Database Query on Main Thread

// BAD: Room DAO called on main thread
@Dao
interface UserDao {
    @Query("SELECT * FROM users WHERE active = 1")
    fun getActiveUsers(): List<User>  // Returns synchronously
}

class UserListFragment : Fragment() {
    override fun onViewCreated(view: View, savedInstanceState: Bundle?) {
        // This executes a SQLite query on the main thread.
        // Room actually throws IllegalStateException by default,
        // but allowMainThreadQueries() disables this safety check.
        val users = database.userDao().getActiveUsers() // ANR risk!
    }
}

// GOOD: Use suspend functions or Flow
@Dao
interface UserDao {
    @Query("SELECT * FROM users WHERE active = 1")
    suspend fun getActiveUsers(): List<User>

    @Query("SELECT * FROM users WHERE active = 1")
    fun observeActiveUsers(): Flow<List<User>>
}

class UserListFragment : Fragment() {
    override fun onViewCreated(view: View, savedInstanceState: Bundle?) {
        viewLifecycleOwner.lifecycleScope.launch {
            val users = database.userDao().getActiveUsers() // Runs on IO dispatcher
            adapter.submitList(users)
        }
    }
}

4.4 Lock Contention — Synchronized Block Held by Background Thread

// BAD: Shared lock between main thread and I/O thread
class ConfigManager {
    private val lock = Any()
    private var config: Map<String, String> = emptyMap()

    fun getConfig(key: String): String? {
        synchronized(lock) {  // Main thread acquires lock
            return config[key]
        }
    }

    fun refreshFromNetwork() {
        // Called from background thread
        val newConfig = api.fetchConfig() // Takes 2-5 seconds
        synchronized(lock) {  // Holds lock during slow parsing
            config = parseConfig(newConfig) // 500ms of parsing
        }
    }
}

// If main thread calls getConfig() while refreshFromNetwork() holds the lock,
// main thread blocks for the duration of parseConfig() → ANR if > 5s total.

// GOOD: Lock-free reads with atomic reference swap
class ConfigManager {
    @Volatile
    private var config: Map<String, String> = emptyMap()

    fun getConfig(key: String): String? = config[key]  // No lock needed

    fun refreshFromNetwork() {
        val newConfig = api.fetchConfig()
        val parsed = parseConfig(newConfig)
        config = parsed // Atomic reference swap — readers see old or new, never partial
    }
}

4.5 BroadcastReceiver Doing Heavy Work in onReceive()

// BAD: Heavy work in onReceive() — runs on main thread
class SyncReceiver : BroadcastReceiver() {
    override fun onReceive(context: Context, intent: Intent) {
        // onReceive() runs on the MAIN THREAD of the app process.
        // You have 10 seconds (foreground) before ANR.
        val db = Room.databaseBuilder(context, AppDb::class.java, "app.db").build()
        val data = api.fetchSync() // Network call — blocks main thread!
        db.syncDao().insertAll(data) // Disk I/O — blocks main thread!
    }
}

// GOOD: Use goAsync() to extend the deadline, then move work off main thread
class SyncReceiver : BroadcastReceiver() {
    override fun onReceive(context: Context, intent: Intent) {
        val pendingResult = goAsync() // Tells system: "I'm not done yet"
        // goAsync() prevents the system from calling finishReceiver()
        // You now have the full broadcast timeout to complete.

        CoroutineScope(Dispatchers.IO).launch {
            try {
                val data = api.fetchSync()
                database.syncDao().insertAll(data)
            } finally {
                pendingResult.finish() // MUST call this or the system leaks the broadcast
            }
        }
    }
}

// Even with goAsync(), you're still bound by the broadcast timeout (10s/60s).
// For truly long-running work, enqueue a WorkManager job instead.

4.6 Deadlock Between Main Thread and Background Thread

// Classic deadlock: two threads acquire two locks in opposite order
class DataManager {
    private val lockA = Any()
    private val lockB = Any()

    // Called on main thread
    fun readData(): String {
        synchronized(lockA) {
            synchronized(lockB) {
                return cache.get("data") ?: ""
            }
        }
    }

    // Called on background thread
    fun writeData(value: String) {
        synchronized(lockB) {      // Acquires B first
            synchronized(lockA) {  // Then tries to acquire A
                cache.put("data", value)
            }
        }
    }
}

// Deadlock scenario:
// T1 (main):       acquires lockA → tries to acquire lockB → BLOCKED (held by T2)
// T2 (background): acquires lockB → tries to acquire lockA → BLOCKED (held by T1)
// Neither thread can proceed → main thread blocked forever → ANR

// traces.txt will show:
// "main" tid=1 Blocked
//   - waiting to lock <0xAAA> (lockB) held by thread 15
// "Worker-1" tid=15 Blocked
//   - waiting to lock <0xBBB> (lockA) held by thread 1

// Fix: Always acquire locks in the same order, or use lock-free data structures.

5. Prevention Strategies

5.1 StrictMode — Detect Violations During Development

class MyApp : Application() {
    override fun onCreate() {
        super.onCreate()

        if (BuildConfig.DEBUG) {
            StrictMode.setThreadPolicy(
                StrictMode.ThreadPolicy.Builder()
                    .detectDiskReads()       // File I/O on main thread
                    .detectDiskWrites()      // File writes on main thread
                    .detectNetwork()         // Network calls on main thread
                    .detectCustomSlowCalls() // Custom slow-call markers
                    .penaltyLog()            // Log to Logcat
                    .penaltyDeath()          // Crash on violation (aggressive but effective)
                    .build()
            )

            StrictMode.setVmPolicy(
                StrictMode.VmPolicy.Builder()
                    .detectLeakedClosableObjects()
                    .detectLeakedSqlLiteObjects()
                    .detectActivityLeaks()
                    .penaltyLog()
                    .build()
            )
        }
    }
}

5.2 Move Work Off Main Thread — Coroutines Pattern

// Pattern: wrap any potentially blocking call in withContext(Dispatchers.IO)
class Repository(
    private val dao: UserDao,
    private val api: ApiService,
    private val prefs: SharedPreferences
) {
    suspend fun loadUserProfile(userId: String): UserProfile {
        return withContext(Dispatchers.IO) {
            val cached = dao.getUser(userId)
            if (cached != null && !cached.isStale()) {
                cached.toProfile()
            } else {
                val remote = api.fetchUser(userId)
                dao.insertUser(remote)
                remote.toProfile()
            }
        }
    }

    suspend fun savePreference(key: String, value: String) {
        withContext(Dispatchers.IO) {
            prefs.edit().putString(key, value).apply()
        }
    }
}

// In ViewModel — collect on main, work on IO:
class ProfileViewModel(private val repo: Repository) : ViewModel() {
    private val _state = MutableStateFlow<UiState>(UiState.Loading)
    val state: StateFlow<UiState> = _state

    fun loadProfile(userId: String) {
        viewModelScope.launch {
            _state.value = UiState.Loading
            try {
                val profile = repo.loadUserProfile(userId)
                _state.value = UiState.Success(profile)
            } catch (e: Exception) {
                _state.value = UiState.Error(e.message ?: "Unknown error")
            }
        }
    }
}

5.3 goAsync() for BroadcastReceiver

// goAsync() returns a PendingResult that keeps the broadcast "alive"
// while you move work to a background thread.
// CRITICAL: you MUST call pendingResult.finish() or the broadcast leaks.

class BootReceiver : BroadcastReceiver() {
    override fun onReceive(context: Context, intent: Intent) {
        if (intent.action != Intent.ACTION_BOOT_COMPLETED) return

        val pendingResult = goAsync()

        // Use a coroutine scope that won't be cancelled by the receiver lifecycle
        CoroutineScope(Dispatchers.IO + SupervisorJob()).launch {
            try {
                WorkManager.getInstance(context).enqueue(
                    OneTimeWorkRequestBuilder<BootSyncWorker>().build()
                )
            } finally {
                pendingResult.finish()
            }
        }
    }
}

5.4 Avoid Holding Locks Across Binder Calls

// BAD: Lock held while making a Binder call
class ServiceProxy {
    private val lock = Any()
    private var cachedResult: List<PackageInfo>? = null

    fun getPackages(pm: PackageManager): List<PackageInfo> {
        synchronized(lock) {
            if (cachedResult == null) {
                // Binder call while holding lock — if another thread needs this lock,
                // it blocks for the entire duration of the IPC round-trip.
                cachedResult = pm.getInstalledPackages(0)
            }
            return cachedResult!!
        }
    }
}

// GOOD: Double-checked locking with volatile, or compute outside lock
class ServiceProxy {
    @Volatile
    private var cachedResult: List<PackageInfo>? = null

    fun getPackages(pm: PackageManager): List<PackageInfo> {
        cachedResult?.let { return it }
        val result = pm.getInstalledPackages(0) // Binder call outside lock
        cachedResult = result
        return result
    }
}

5.5 ANR Monitoring in Production

// Google Play Vitals: automatically reports ANR rates
// Threshold: > 0.47% ANR rate = "bad behavior" flag in Play Console
// > 8% = risk of search ranking penalty

// Firebase Crashlytics ANR reporting (enabled by default in recent SDKs):
// Captures ANR stack traces and reports them alongside crashes.
// Includes device state, memory pressure, and battery level.

// Custom ANR watchdog — detect ANRs before the system does:
class AnrWatchdog(private val timeoutMs: Long = 5000L) : Thread("ANR-Watchdog") {
    @Volatile private var tick = 0L
    @Volatile private var reported = false
    private val handler = Handler(Looper.getMainLooper())

    private val ticker = Runnable { tick = System.currentTimeMillis() }

    override fun run() {
        while (!isInterrupted) {
            tick = 0
            handler.post(ticker)
            Thread.sleep(timeoutMs)
            if (tick == 0L && !reported) {
                // Main thread didn't process our Runnable within timeoutMs
                reported = true
                val stackTrace = Looper.getMainLooper().thread.stackTrace
                reportAnrToBackend(stackTrace)
            } else {
                reported = false
            }
        }
    }
}

5.6 Perfetto / Systrace for ANR Investigation

// Perfetto (modern) or Systrace (legacy) captures system-wide traces
// that show exactly what the main thread was doing during an ANR.

// Capture a trace:
// $ adb shell perfetto -o /data/misc/perfetto-traces/trace.perfetto-trace \
//     -t 10s sched freq idle am wm input view

// In the trace, look for:
// 1. The main thread's scheduling state (Running, Sleeping, Blocked)
// 2. Binder transactions — which service the main thread called
// 3. Lock contention events (futex waits)
// 4. Disk I/O on the main thread (f2fs_write, ext4_da_write)
// 5. GC pauses (concurrent-GC, young-GC)

// Perfetto SQL query to find main thread blocks:
// SELECT ts, dur, name FROM slice
// WHERE track_id = (SELECT id FROM thread_track WHERE name = 'main')
// AND dur > 1000000000  -- longer than 1 second
// ORDER BY dur DESC

// For ANR-specific traces on Android 11+:
// $ adb bugreport
// The bugreport includes ANR traces, system health data, and Perfetto traces.
Staff-Level Insight — ANRs Are Often Lock Contention, Not Just Heavy Work

Junior engineers assume ANRs are caused by "doing too much on the main thread." At L6, you know the most insidious ANRs come from lock contention: the main thread does almost no work itself, but blocks waiting for a lock held by a background thread doing I/O. The traces.txt shows the main thread in Blocked state with waiting to lock <0x...> held by thread N. The fix isn't to "move work off main thread" — the work is already off the main thread. The fix is to eliminate shared mutable state, use lock-free data structures (ConcurrentHashMap, AtomicReference, @Volatile), or redesign the synchronization so the main thread never contends. At Google scale, this pattern accounts for ~30-40% of ANRs that aren't simple "network on main thread" mistakes.

Summary — ANR Prevention Checklist
  • StrictMode in debug builds to catch disk/network on main thread
  • Coroutines + Dispatchers.IO for all blocking operations
  • apply() over commit() for SharedPreferences (but beware QueuedWork.waitToFinish())
  • goAsync() for BroadcastReceivers, or delegate to WorkManager
  • Lock-free patterns — @Volatile, ConcurrentHashMap, AtomicReference
  • Never hold locks across Binder calls
  • Monitor in production — Play Vitals, Crashlytics, custom watchdog
  • Perfetto traces for deep investigation of specific ANR reports

6. ApplicationExitInfo — Programmatic ANR Detection (Android 11+)

// Android 11 (API 30) introduced ApplicationExitInfo — a programmatic API
// to query WHY your app's process was killed in previous sessions.
// This is a game-changer for ANR and OOM diagnosis in production.

// Query exit reasons:
val am = getSystemService(Context.ACTIVITY_SERVICE) as ActivityManager
val exitInfos = am.getHistoricalProcessExitReasons(
    packageName,  // null for all packages (requires permission)
    0,            // pid (0 = all)
    10            // max results
)

for (info in exitInfos) {
    when (info.reason) {
        ApplicationExitInfo.REASON_ANR -> {
            // ANR detected — get the traces
            val trace = info.traceInputStream?.bufferedReader()?.readText()
            // trace contains the full thread dump (same as traces.txt)
            // Send to your analytics backend for analysis
            analytics.reportAnr(
                timestamp = info.timestamp,
                description = info.description,  // e.g., "Input dispatching timed out"
                importance = info.importance,     // process importance at time of ANR
                pss = info.pss,                   // PSS at time of death
                rss = info.rss,                   // RSS at time of death
                trace = trace
            )
        }
        ApplicationExitInfo.REASON_LOW_MEMORY -> {
            // Killed by lmkd — log the memory state
            analytics.reportLmkKill(
                pss = info.pss,
                rss = info.rss,
                importance = info.importance
            )
        }
        ApplicationExitInfo.REASON_CRASH -> {
            // Unhandled exception
        }
        ApplicationExitInfo.REASON_CRASH_NATIVE -> {
            // Native crash (SIGSEGV, SIGABRT)
        }
        ApplicationExitInfo.REASON_EXCESSIVE_RESOURCE_USAGE -> {
            // System killed for using too much CPU/memory
        }
    }
}

// Key fields on ApplicationExitInfo:
// .reason          → REASON_ANR, REASON_LOW_MEMORY, REASON_CRASH, etc.
// .status          → exit code or signal number
// .importance      → RunningAppProcessInfo.IMPORTANCE_* at time of death
// .pss / .rss      → memory usage at time of death (in KB)
// .timestamp       → wall-clock time of death
// .description     → human-readable description (e.g., ANR message)
// .traceInputStream → full thread dump for ANR (null for non-ANR exits)
// .processStateSummary → custom byte[] you set via setProcessStateSummary()

Custom State for Crash Diagnostics

// You can attach custom state that survives process death:
// This is invaluable for debugging — attach your app's current screen,
// pending operations, or memory snapshot.

class MyApplication : Application() {
    override fun onCreate() {
        super.onCreate()

        // Set custom state that will be available in ApplicationExitInfo
        // after the process is killed. Max 256 bytes.
        val state = "screen=home,pending_uploads=3,cache_mb=45"
        ActivityManager.setProcessStateSummary(state.toByteArray())
    }

    fun onScreenChanged(screen: String) {
        val state = "screen=$screen,heap_pct=${getHeapUsagePercent()}"
        ActivityManager.setProcessStateSummary(state.toByteArray())
    }
}

// On next launch, read it:
val exitInfos = am.getHistoricalProcessExitReasons(packageName, 0, 1)
exitInfos.firstOrNull()?.let { info ->
    val customState = info.processStateSummary?.let { String(it) }
    // customState = "screen=home,pending_uploads=3,cache_mb=45"
    // Now you know exactly what the app was doing when it was killed
}
Staff-Level Insight — ApplicationExitInfo Replaces Guesswork

Before Android 11, diagnosing production ANRs required: (1) hoping the user opted into Play Vitals, (2) parsing aggregated ANR clusters in Play Console with limited stack traces, (3) guessing from Crashlytics ANR reports which often lack the full thread dump. ApplicationExitInfo gives you the exact thread dump (via traceInputStream), the memory state (PSS/RSS), and your custom diagnostic state — all queryable programmatically on the next app launch. At Google, this API is used to build internal ANR dashboards with per-build regression tracking. For L6 interviews, knowing this API signals you understand production debugging at scale, not just development-time tooling.
