03 — Memory Leaks

Leak taxonomy at architectural boundaries — Activity, Fragment, View, Listener, Coroutine, Compose leaks, detection tools, and prevention patterns

1. GC Root Retention Path — How Leaks Form

A memory leak occurs when an object that should be garbage-collected is kept alive by an unintended strong reference chain from a GC root. The GC cannot reclaim the object because it's still "reachable" — even though the application no longer needs it.

// GC roots in the JVM / ART runtime:
//
// 1. Thread stacks — local variables and parameters of all live threads
// 2. Static fields — fields of loaded classes (ClassLoader keeps classes alive)
// 3. JNI global references — native code holding Java object refs
// 4. Monitor objects — objects used as synchronized locks
// 5. GC-internal roots — runtime bookkeeping objects
//
// A leak forms when:
// GC Root → ... → Retaining Object → Leaked Object
//
// The "retaining object" is the nearest object in the chain that
// SHOULD have released its reference but didn't.
// LeakCanary calls this the "leak trace" — the shortest path from
// a GC root to the leaked object.

Typical Activity Leak Chain

  GC Root                    Retaining Object              Leaked Object
  ──────                     ────────────────              ─────────────

  Thread Stack               Anonymous Runnable            Destroyed Activity
  (Main Thread)              (posted to Handler)           (should be GC'd)
       │                           │                            │
       ▼                           ▼                            ▼
  ┌──────────┐    strong     ┌───────────┐    strong     ┌──────────────┐
  │  Handler  │─────────────►│ Runnable  │──────────────►│  Activity    │
  │ (static   │              │ (inner    │  implicit     │  (destroyed, │
  │  in Main  │              │  class of │  outer ref)   │   onDestroy  │
  │  Looper)  │              │  Activity)│               │   called)    │
  └──────────┘               └───────────┘               └──────────────┘
       │                                                        │
       │                                                        ▼
       │                                                 ┌──────────────┐
  GC root: MessageQueue                                  │  View tree   │
  holds the Handler which                                │  Bitmaps     │
  holds the Runnable which                               │  Adapters    │
  holds the Activity                                     │  ~10-50 MB   │
                                                         └──────────────┘

  The Activity's entire View hierarchy, all Bitmaps, Drawables,
  and Adapters are retained — potentially tens of megabytes.
Key Insight

The leaked Activity itself may be small, but it retains its entire View tree, all Bitmaps, Drawables, and any objects referenced by its fields. A single Activity leak can retain 10–50 MB depending on UI complexity. On a device with 256 MB heap, that's catastrophic after a few configuration changes.

3. Leak Type: Static Reference to Activity/Context

BAD — Static holds Activity Context

class LeakyActivity : AppCompatActivity() {

    companion object {
        // Static reference survives Activity destruction.
        // After rotation, the OLD Activity is retained forever.
        lateinit var instance: LeakyActivity
    }

    override fun onCreate(savedInstanceState: Bundle?) {
        super.onCreate(savedInstanceState)
        instance = this // Leak: static → destroyed Activity
    }
}

GOOD — Use Application Context or WeakReference only where appropriate

class SafeActivity : AppCompatActivity() {

    companion object {
        // If you truly need a static context, use Application context.
        // Application context lives for the entire process lifetime —
        // it cannot leak an Activity.
        lateinit var appContext: Context
    }

    override fun onCreate(savedInstanceState: Bundle?) {
        super.onCreate(savedInstanceState)
        appContext = applicationContext
    }
}

4. Leak Type: Inner Class / Anonymous Class

In Kotlin, inner classes (and anonymous object expressions) capture an implicit reference to the enclosing class. This is the single most common source of Android memory leaks.

BAD — Anonymous Runnable captures Activity

class LeakyActivity : AppCompatActivity() {

    override fun onCreate(savedInstanceState: Bundle?) {
        super.onCreate(savedInstanceState)

        // This anonymous Runnable is an inner class of LeakyActivity.
        // It holds an implicit reference to `this` (the Activity).
        // If the Activity is destroyed before 30s, the Runnable
        // keeps the Activity alive until it executes.
        Handler(Looper.getMainLooper()).postDelayed({
            updateUI() // `this@LeakyActivity` is captured
        }, 30_000)
    }

    private fun updateUI() { /* ... */ }
}

GOOD — Remove callbacks on destroy

class SafeActivity : AppCompatActivity() {

    private val handler = Handler(Looper.getMainLooper())
    private val updateRunnable = Runnable { updateUI() }

    override fun onCreate(savedInstanceState: Bundle?) {
        super.onCreate(savedInstanceState)
        handler.postDelayed(updateRunnable, 30_000)
    }

    override fun onDestroy() {
        super.onDestroy()
        handler.removeCallbacks(updateRunnable)
    }

    private fun updateUI() { /* ... */ }
}

5. Leak Type: Handler/Runnable Holding Activity

The Handler leak is a variant of the inner-class problem, but it's so pervasive it deserves its own section. The MessageQueue on the main Looper is a GC root — any Message in the queue retains its target Handler, which retains the Activity.

BAD — Handler as inner class with delayed message

class LeakyActivity : AppCompatActivity() {

    // Inner class Handler holds implicit ref to LeakyActivity.
    // Messages in the queue → Handler → Activity.
    private val handler = object : Handler(Looper.getMainLooper()) {
        override fun handleMessage(msg: Message) {
            // `this@LeakyActivity` is captured
            textView.text = "Updated"
        }
    }

    override fun onCreate(savedInstanceState: Bundle?) {
        super.onCreate(savedInstanceState)
        handler.sendMessageDelayed(Message.obtain(), 60_000)
    }
    // Activity destroyed → message still in queue → leak for up to 60s
}

GOOD — Static Handler with WeakReference, or lifecycle-aware removal

class SafeActivity : AppCompatActivity() {

    // Approach 1: Remove all messages on destroy
    private val handler = Handler(Looper.getMainLooper())

    override fun onCreate(savedInstanceState: Bundle?) {
        super.onCreate(savedInstanceState)
        handler.postDelayed({ safeUpdate() }, 60_000)
    }

    override fun onDestroy() {
        super.onDestroy()
        handler.removeCallbacksAndMessages(null) // Nuclear option: clear everything
    }

    // Approach 2: Use lifecycleScope (preferred in modern Android)
    // No Handler needed — coroutine is cancelled when Activity is destroyed
    private fun scheduleUpdate() {
        lifecycleScope.launch {
            delay(60_000)
            textView.text = "Updated"
        }
    }

    private fun safeUpdate() { /* ... */ }
}

6. Leak Type: Listener/Callback Not Unregistered

System services hold strong references to registered listeners. If you register a listener in onCreate/onResume and forget to unregister in the symmetric lifecycle callback, the system service retains your Activity.

BAD — LocationManager listener never unregistered

class LeakyActivity : AppCompatActivity() {

    private val locationListener = object : LocationListener {
        override fun onLocationChanged(location: Location) {
            // `this@LeakyActivity` captured via outer ref
            updateMap(location)
        }
    }

    override fun onResume() {
        super.onResume()
        val lm = getSystemService(LOCATION_SERVICE) as LocationManager
        lm.requestLocationUpdates(
            LocationManager.GPS_PROVIDER, 1000, 10f, locationListener
        )
    }

    // Missing onPause() → locationListener → Activity leaked
    // LocationManager lives in system_server, holds a Binder callback
    // to our listener. The listener's anonymous class holds the Activity.
}

GOOD — Symmetric register/unregister

class SafeActivity : AppCompatActivity() {

    private val locationListener = object : LocationListener {
        override fun onLocationChanged(location: Location) {
            updateMap(location)
        }
    }

    override fun onResume() {
        super.onResume()
        val lm = getSystemService(LOCATION_SERVICE) as LocationManager
        lm.requestLocationUpdates(
            LocationManager.GPS_PROVIDER, 1000, 10f, locationListener
        )
    }

    override fun onPause() {
        super.onPause()
        val lm = getSystemService(LOCATION_SERVICE) as LocationManager
        lm.removeUpdates(locationListener) // Symmetric unregister
    }
}

Other common offenders

// SensorManager — register/unregister in onResume/onPause
sensorManager.registerListener(listener, sensor, SENSOR_DELAY_NORMAL)
sensorManager.unregisterListener(listener)

// BroadcastReceiver — register/unregister
registerReceiver(receiver, IntentFilter("ACTION"))
unregisterReceiver(receiver)

// OnSharedPreferenceChangeListener — the classic silent leak
// SharedPreferences stores listeners in a WeakHashMap internally,
// BUT if you register an anonymous listener, nothing else holds a
// strong ref → it gets GC'd immediately and never fires.
// If you store it in a field, the Activity holds it, and
// SharedPreferences holds the Activity via the listener.
// Fix: unregister in onDestroy.
prefs.registerOnSharedPreferenceChangeListener(listener)
prefs.unregisterOnSharedPreferenceChangeListener(listener)

7. Leak Type: ViewModel Holding View/Activity Reference

ViewModel survives configuration changes — it outlives the Activity that created it. If a ViewModel holds a reference to a View or Activity, the destroyed Activity is retained until the ViewModel is cleared.

BAD — ViewModel holds Activity reference

class LeakyViewModel(
    private val activity: Activity // ViewModel outlives Activity → leak
) : ViewModel() {

    fun loadData() {
        // Uses activity to show a Toast — seems harmless, but
        // after rotation the OLD activity is retained
        Toast.makeText(activity, "Loading...", Toast.LENGTH_SHORT).show()
    }
}

// Also bad: ViewModel holding a View reference
class AlsoLeakyViewModel : ViewModel() {
    var recyclerView: RecyclerView? = null // Retains Activity's View tree
}

GOOD — ViewModel never touches Views; use Application context if needed

class SafeViewModel(application: Application) : AndroidViewModel(application) {

    // AndroidViewModel provides getApplication() — Application context
    // that lives for the process lifetime. Safe to hold.
    fun loadData() {
        Toast.makeText(getApplication(), "Loading...", Toast.LENGTH_SHORT).show()
    }
}

// Even better: ViewModel exposes state, UI observes it
class BetterViewModel : ViewModel() {
    private val _uiState = MutableStateFlow(UiState())
    val uiState: StateFlow<UiState> = _uiState.asStateFlow()

    fun loadData() {
        viewModelScope.launch {
            _uiState.value = UiState(message = "Loading...")
        }
    }
}

// Activity observes — ViewModel has zero knowledge of Views
class SafeActivity : AppCompatActivity() {
    private val viewModel: BetterViewModel by viewModels()

    override fun onCreate(savedInstanceState: Bundle?) {
        super.onCreate(savedInstanceState)
        lifecycleScope.launch {
            repeatOnLifecycle(Lifecycle.State.STARTED) {
                viewModel.uiState.collect { state ->
                    textView.text = state.message
                }
            }
        }
    }
}

8. Leak Type: Fragment Retained Beyond Activity Lifecycle

// Pre-AndroidX problem: setRetainInstance(true) kept the Fragment
// across configuration changes, but its View was destroyed.
// If the Fragment held View references, the old Activity's Views leaked.

// BAD — retained Fragment holding View references
class LeakyFragment : Fragment() {
    private var binding: FragmentDetailBinding? = null

    override fun onCreate(savedInstanceState: Bundle?) {
        super.onCreate(savedInstanceState)
        retainInstance = true // Deprecated — Fragment survives config change
    }

    override fun onCreateView(inflater: LayoutInflater, container: ViewGroup?, s: Bundle?): View {
        binding = FragmentDetailBinding.inflate(inflater, container, false)
        return binding!!.root
    }
    // After rotation: Fragment is retained, but its View is destroyed and recreated.
    // `binding` still points to the OLD View → old Activity's View tree leaked.
}

// GOOD — clear View references in onDestroyView
class SafeFragment : Fragment() {
    private var _binding: FragmentDetailBinding? = null
    private val binding get() = _binding!!

    override fun onCreateView(inflater: LayoutInflater, container: ViewGroup?, s: Bundle?): View {
        _binding = FragmentDetailBinding.inflate(inflater, container, false)
        return binding.root
    }

    override fun onDestroyView() {
        super.onDestroyView()
        _binding = null // Release View references — prevents leak
    }
}
Fragment ViewBinding Leak — The #1 Fragment Leak

The Fragment outlives its View (Fragment is in CREATED state but View is destroyed during back-stack transactions). If you hold a ViewBinding reference without nulling it in onDestroyView(), the entire View tree of the previous Activity is retained. This is so common that it's the default LeakCanary finding in most Android apps. Navigation Component makes this worse because Fragments on the back stack have their Views destroyed but the Fragment instance stays alive.

9. Leak Type: Coroutine Scope Leak

BAD — GlobalScope captures Activity

class LeakyActivity : AppCompatActivity() {

    override fun onCreate(savedInstanceState: Bundle?) {
        super.onCreate(savedInstanceState)

        // GlobalScope is NEVER cancelled — it lives for the process lifetime.
        // The lambda captures `this` (the Activity).
        // If the Activity is destroyed, the coroutine keeps it alive
        // until the network call completes.
        GlobalScope.launch {
            val data = repository.fetchData() // 30-second timeout
            withContext(Dispatchers.Main) {
                textView.text = data // `this@LeakyActivity` captured
            }
        }
    }
}

BAD — Custom scope without cancellation

class LeakyActivity : AppCompatActivity() {

    // Custom scope but never cancelled → same as GlobalScope
    private val scope = CoroutineScope(Dispatchers.Main + Job())

    override fun onCreate(savedInstanceState: Bundle?) {
        super.onCreate(savedInstanceState)
        scope.launch {
            val data = repository.fetchData()
            textView.text = data
        }
    }
    // Missing: scope.cancel() in onDestroy()
}

GOOD — lifecycleScope or viewModelScope

class SafeActivity : AppCompatActivity() {

    override fun onCreate(savedInstanceState: Bundle?) {
        super.onCreate(savedInstanceState)

        // lifecycleScope is cancelled when Activity is destroyed.
        // All child coroutines are cancelled → no leak.
        lifecycleScope.launch {
            val data = repository.fetchData()
            textView.text = data
        }

        // Even better: repeatOnLifecycle for flow collection
        lifecycleScope.launch {
            repeatOnLifecycle(Lifecycle.State.STARTED) {
                viewModel.dataFlow.collect { data ->
                    textView.text = data
                }
            }
        }
    }
}

// In ViewModel — viewModelScope is cancelled in onCleared()
class SafeViewModel : ViewModel() {
    fun loadData() {
        viewModelScope.launch {
            // Cancelled when ViewModel is cleared
            val data = repository.fetchData()
            _state.value = data
        }
    }
}
Coroutine Leak Internals

When a coroutine suspends, its continuation object holds references to all captured variables in the lambda. The continuation is kept alive by the Job in the coroutine's scope. If the Job is never cancelled (GlobalScope, orphaned custom scope), the continuation — and everything it captures — lives until the coroutine completes or the process dies. lifecycleScope uses LifecycleCoroutineScope which registers a LifecycleObserver that calls cancel() on ON_DESTROY.

10. Leak Type: Compose — Remembering Activity/Context

BAD — Capturing Activity in remember

@Composable
fun LeakyScreen() {
    // LocalContext.current returns the Activity in an Activity-hosted composition.
    // Storing it in remember{} keeps the reference across recompositions.
    // If the Activity is recreated (config change), the old Activity is retained
    // by the Compose slot table until the composition is disposed.
    val context = LocalContext.current
    val manager = remember {
        // This captures `context` (which is the Activity) in the slot table.
        // The slot table survives recomposition but the Activity doesn't.
        SomeManager(context) // Leak if SomeManager stores the context
    }
}

GOOD — Use Application context or remember with keys

@Composable
fun SafeScreen() {
    val context = LocalContext.current

    // Option 1: Use applicationContext — safe, never leaks an Activity
    val manager = remember {
        SomeManager(context.applicationContext)
    }

    // Option 2: Use DisposableEffect for cleanup
    DisposableEffect(Unit) {
        val listener = SomeListener()
        someService.register(listener)
        onDispose {
            someService.unregister(listener)
        }
    }

    // Option 3: For system services, use rememberUpdatedState
    // to always reference the current context without capturing it
    val currentContext by rememberUpdatedState(context)
    LaunchedEffect(Unit) {
        // currentContext always points to the latest Activity
        // No stale reference retained
        doSomethingWith(currentContext)
    }
}
Compose Slot Table Retention

The Compose slot table stores all remember{} values. During recomposition, the slot table is reused — values persist across recompositions. If a remember{} block captures an Activity reference, that reference lives in the slot table until the composition is disposed (the Composable leaves the tree). For Composables in a NavHost, back-stack destinations keep their composition alive, meaning remembered Activity references survive even when the screen isn't visible.

11. Leak Type: Thread Holding Activity Reference

BAD — Long-running thread captures Activity

class LeakyActivity : AppCompatActivity() {

    override fun onCreate(savedInstanceState: Bundle?) {
        super.onCreate(savedInstanceState)

        // Thread is a GC root while running.
        // The lambda captures `this` (Activity).
        // If the Activity is destroyed before the thread finishes,
        // the Activity is retained for up to 60 seconds.
        Thread {
            val result = heavyComputation() // Takes 60 seconds
            runOnUiThread {
                textView.text = result // `this@LeakyActivity` captured
            }
        }.start()
    }
}

GOOD — Use structured concurrency

class SafeActivity : AppCompatActivity() {

    override fun onCreate(savedInstanceState: Bundle?) {
        super.onCreate(savedInstanceState)

        // lifecycleScope cancels when Activity is destroyed.
        // Dispatchers.Default runs on a thread pool — no new thread created.
        lifecycleScope.launch {
            val result = withContext(Dispatchers.Default) {
                heavyComputation()
            }
            textView.text = result
        }
    }
}

12. Leak Type: Singleton Holding Activity Context

BAD — Singleton stores Activity context

object Analytics {
    private lateinit var context: Context

    fun init(context: Context) {
        // If called with an Activity context, the Activity is retained
        // for the entire process lifetime (singleton never dies).
        this.context = context
    }

    fun trackEvent(event: String) {
        // Uses context to get package info, etc.
        val packageName = context.packageName
    }
}

// In Activity:
class LeakyActivity : AppCompatActivity() {
    override fun onCreate(savedInstanceState: Bundle?) {
        super.onCreate(savedInstanceState)
        Analytics.init(this) // Passes Activity context → leaked
    }
}

GOOD — Always use Application context in singletons

object Analytics {
    private lateinit var context: Context

    fun init(context: Context) {
        // applicationContext is safe — it lives for the process lifetime,
        // same as the singleton. No leak possible.
        this.context = context.applicationContext
    }

    fun trackEvent(event: String) {
        val packageName = context.packageName
    }
}

// Even better: inject via Hilt
@Singleton
class Analytics @Inject constructor(
    @ApplicationContext private val context: Context
) {
    fun trackEvent(event: String) { /* ... */ }
}

13. Architectural Boundary Leaks

Service → Activity Reference

// BAD: Service holds a direct reference to Activity
class MyService : Service() {
    var activity: MainActivity? = null // Service outlives Activity → leak

    override fun onBind(intent: Intent): IBinder {
        return LocalBinder()
    }

    inner class LocalBinder : Binder() {
        fun getService(): MyService = this@MyService
    }
}

// In Activity:
// serviceConnection callback stores `service.activity = this` → leak

// GOOD: Use callbacks/interfaces, clear on unbind
class MyService : Service() {
    private var callback: ServiceCallback? = null

    fun setCallback(callback: ServiceCallback?) {
        this.callback = callback
    }

    override fun onBind(intent: Intent): IBinder = LocalBinder()

    inner class LocalBinder : Binder() {
        fun getService(): MyService = this@MyService
    }
}

class SafeActivity : AppCompatActivity(), ServiceCallback {
    private var service: MyService? = null

    private val connection = object : ServiceConnection {
        override fun onServiceConnected(name: ComponentName, binder: IBinder) {
            service = (binder as MyService.LocalBinder).getService()
            service?.setCallback(this@SafeActivity)
        }
        override fun onServiceDisconnected(name: ComponentName) {
            service = null
        }
    }

    override fun onStop() {
        super.onStop()
        service?.setCallback(null) // Clear callback before unbind
        unbindService(connection)
    }
}

BroadcastReceiver — Registered but Not Unregistered

// BAD: registered in onCreate, never unregistered
class LeakyActivity : AppCompatActivity() {
    private val receiver = object : BroadcastReceiver() {
        override fun onReceive(context: Context, intent: Intent) {
            updateUI(intent) // captures outer Activity
        }
    }

    override fun onCreate(savedInstanceState: Bundle?) {
        super.onCreate(savedInstanceState)
        registerReceiver(receiver, IntentFilter("com.example.ACTION"))
    }
    // Missing unregisterReceiver → Activity leaked
    // Also causes "Activity has leaked IntentReceiver" crash on destroy
}

// GOOD: symmetric register/unregister
class SafeActivity : AppCompatActivity() {
    private val receiver = object : BroadcastReceiver() {
        override fun onReceive(context: Context, intent: Intent) {
            updateUI(intent)
        }
    }

    override fun onStart() {
        super.onStart()
        registerReceiver(receiver, IntentFilter("com.example.ACTION"))
    }

    override fun onStop() {
        super.onStop()
        unregisterReceiver(receiver)
    }
}

WebView — Notorious Activity Leak

// WebView is one of the worst leakers in Android.
// It holds references to the Activity context internally,
// and its native layer (Chromium) has its own reference counting
// that doesn't align with Android's lifecycle.

// BAD: WebView in XML layout
// <WebView android:id="@+id/webview" ... />
// The WebView holds the Activity context from inflation.
// Even calling webView.destroy() doesn't fully release it
// on some Android versions.

// GOOD: Create WebView programmatically with Application context,
// add to layout, and aggressively clean up
class SafeWebViewActivity : AppCompatActivity() {

    private var webView: WebView? = null

    override fun onCreate(savedInstanceState: Bundle?) {
        super.onCreate(savedInstanceState)
        setContentView(R.layout.activity_webview)

        // Create with Application context to avoid Activity leak
        webView = WebView(applicationContext).apply {
            layoutParams = ViewGroup.LayoutParams(MATCH_PARENT, MATCH_PARENT)
            settings.javaScriptEnabled = true
        }
        findViewById<ViewGroup>(R.id.webview_container).addView(webView)
        webView?.loadUrl("https://example.com")
    }

    override fun onDestroy() {
        webView?.let { wv ->
            // Remove from parent first
            (wv.parent as? ViewGroup)?.removeView(wv)
            wv.stopLoading()
            wv.settings.javaScriptEnabled = false
            wv.clearHistory()
            wv.clearCache(true)
            wv.loadUrl("about:blank")
            wv.onPause()
            wv.removeAllViews()
            wv.destroyDrawingCache()
            wv.destroy()
        }
        webView = null
        super.onDestroy()
    }
}

InputMethodManager Leak (System Bug)

// InputMethodManager (IMM) is a system singleton that holds a reference
// to the currently focused View's "served view" — which references
// the Activity. This is a known Android framework bug that has existed
// since early Android versions.
//
// The leak path:
// InputMethodManager (static singleton)
//   → mServedView (the focused EditText)
//     → mContext (the Activity)
//
// This leak is NOT your fault — it's a framework bug.
// It's been partially fixed in newer Android versions but still
// appears on older devices.

// Workaround: clear focus and hide keyboard in onDestroy
class SafeActivity : AppCompatActivity() {

    override fun onDestroy() {
        // Clear focus from any EditText
        currentFocus?.let { view ->
            val imm = getSystemService(INPUT_METHOD_SERVICE) as InputMethodManager
            imm.hideSoftInputFromWindow(view.windowToken, 0)
            view.clearFocus()
        }
        // On older devices, you may need to use reflection to null out
        // InputMethodManager.mServedView — but this is fragile and
        // not recommended for production. LeakCanary will flag this
        // as a "known framework leak" and exclude it from reports.
        super.onDestroy()
    }
}
ContentProvider Context Leak

ContentProvider is initialized before Application.onCreate() and receives the Application context via getContext(). However, if a ContentProvider stores a reference to a Context passed via a query caller (e.g., from ContentResolver in an Activity), it can leak that Activity. The fix: ContentProviders should only use getContext() (Application context) and never store references from external callers. Libraries that abuse ContentProvider for initialization (like Firebase, WorkManager) are careful about this — but custom providers often aren't.

14. Detection Tools

LeakCanary — How It Works Internally

// LeakCanary is the gold standard for leak detection in Android.
// Add to debug builds only:
// debugImplementation("com.squareup.leakcanary:leakcanary-android:2.14")
//
// No code changes needed — it auto-installs via ContentProvider.

// Internal architecture:
//
// 1. ObjectWatcher
//    - Watches objects that should be GC'd (Activities, Fragments, Views)
//    - Uses WeakReference + ReferenceQueue pattern:
//
//      val weakRef = WeakReference(activity, referenceQueue)
//      watchedObjects[uuid] = KeyedWeakReference(weakRef, description)
//
//    - After 5 seconds, checks if the WeakReference was enqueued
//      (enqueued = object was GC'd = no leak)
//    - If NOT enqueued after GC trigger → object is "retained"
//
// 2. Retained Object Threshold
//    - Default: 5 retained objects trigger a heap dump
//    - Configurable via LeakCanary.config
//    - Avoids dumping on every single retained object (expensive)
//
// 3. HeapDumper
//    - Calls Debug.dumpHprofData(file) — freezes the app briefly
//    - Produces an .hprof file (heap dump)
//    - On Android 11+, uses non-blocking heap dump API
//
// 4. Shark (Smart Heap Analysis Reports for Kotlin)
//    - LeakCanary's heap analysis engine
//    - Parses the .hprof file
//    - Finds the shortest strong-reference path from GC roots
//      to each retained object
//    - Identifies the "leak trace" — the chain of references
//    - Classifies leaks: Application leak vs Library leak vs Framework leak
//    - Runs in a separate process to avoid OOM during analysis

// LeakCanary hooks into:
// - Application.registerActivityLifecycleCallbacks() → watches Activities
// - FragmentManager.registerFragmentLifecycleCallbacks() → watches Fragments
// - ViewTreeLifecycleOwner → watches root Views
// - ViewModel.onCleared() → watches ViewModels (via reflection)

Android Studio Memory Profiler

// Android Studio Profiler → Memory tab
//
// Key capabilities:
//
// 1. Real-time memory graph
//    - Shows Java heap, native heap, graphics, stack, code, others
//    - Useful for spotting memory growth patterns (sawtooth = healthy GC,
//      staircase = leak)
//
// 2. Heap Dump
//    - Capture button → takes a snapshot of all live objects
//    - Filter by class name, package, or "Activity" / "Fragment"

15. WeakReference Is Not a Fix for Leaks

Staff-Level Insight — WeakReference Is a Band-Aid, Not a Cure

A common junior-to-mid-level "fix" for memory leaks is to wrap the problematic reference in a WeakReference. This masks the symptom but doesn't fix the architectural problem. Here's why it's dangerous:

  • Non-deterministic behavior — The GC can collect the weakly-referenced object at any time. Your code must handle null on every access, leading to silent failures and hard-to-reproduce bugs.
  • Hides the real bug — The leak exists because a lifecycle boundary is violated. WeakReference papers over it instead of fixing the ownership model.
  • Testing becomes impossible — You can't reliably test code that depends on GC timing. The behavior differs between debug and release builds, between devices, and between ART versions.
  • Performance overhead — WeakReference objects add GC pressure. The GC must process the ReferenceQueue for every collected weak ref. At scale, this is measurable.

The correct fix is always structural: use lifecycle-aware components (lifecycleScope, viewModelScope, DisposableEffect), unregister callbacks symmetrically, and never pass Activity context where Application context suffices. If you find yourself reaching for WeakReference, step back and ask: "Who owns this object, and when should the reference be released?" Fix the ownership, not the reference type.

// BAD — WeakReference as a "fix"
class AnalyticsTracker {
    // "Fixed" the leak by using WeakReference. But now:
    // - trackEvent() silently does nothing if Activity was GC'd
    // - No guarantee WHEN the Activity will be collected
    // - Behavior differs between debug (GC aggressive) and release
    private var activityRef: WeakReference<Activity>? = null

    fun attach(activity: Activity) {
        activityRef = WeakReference(activity)
    }

    fun trackEvent(event: String) {
        val activity = activityRef?.get() ?: return // Silent failure
        // What if this was a critical analytics event? It's just lost.
    }
}

// GOOD — Fix the architecture
class AnalyticsTracker(
    private val context: Context // Application context — no leak possible
) {
    fun trackEvent(event: String) {
        // Always works, no null checks, deterministic behavior
        val packageName = context.packageName
        // ... send event
    }
}

16. Leak Prevention Cheat Sheet

Rule Why
Never store Activity context in singletons or companion objects Singleton outlives Activity → retains destroyed Activity forever
Use applicationContext in anything that outlives an Activity Application context lives for the process — cannot leak an Activity
Null out Fragment ViewBinding in onDestroyView() Fragment outlives its View on back stack → retains old View tree
Use lifecycleScope / viewModelScope instead of GlobalScope Structured concurrency cancels coroutines at lifecycle boundaries
Unregister listeners in the symmetric lifecycle callback System services hold strong refs to listeners → retains Activity
Never pass View or Activity to ViewModel ViewModel survives config changes → retains destroyed Activity
Remove Handler callbacks in onDestroy() MessageQueue is a GC root → pending messages retain Handler → Activity
Create WebView with Application context WebView's Chromium internals hold context references aggressively
Use DisposableEffect for cleanup in Compose Ensures resources are released when Composable leaves the tree
Run LeakCanary in every debug build Catches leaks early — before they compound in production
