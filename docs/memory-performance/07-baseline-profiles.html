07 — Baseline Profiles & ART Compilation

AOT vs JIT vs interpreted execution, profile-guided optimization, Baseline Profiles, Startup Profiles, Cloud Profiles, dex layout optimization, and Macrobenchmark generation

1. ART Compilation Modes — The Full Picture

ART (Android Runtime) supports three execution modes, each with different trade-offs. Understanding these is essential because Baseline Profiles exist to solve the fundamental tension between them.

Interpreted Execution

The ART interpreter reads DEX bytecode instruction by instruction and executes it on the CPU. No machine code is generated. This is the slowest execution mode — roughly 10-100x slower than compiled code — but it requires zero compilation time and zero additional storage. Every method starts here on a fresh install.

JIT (Just-In-Time) Compilation

When ART detects a method is "hot" (executed frequently or in a loop), the JIT compiler compiles it to native machine code at runtime. The compiled code is stored in a per-process JIT code cache (~2MB by default, configurable via dalvik.vm.jit.codecachesize). JIT compilation happens on a background thread but still consumes CPU and causes brief pauses when the compiled code is installed.

// JIT code cache lifecycle:
// 1. Method invoked → interpreter executes bytecode
// 2. Method hits "hot" threshold (~10,000 invocations or loop iterations)
// 3. JIT compiler thread compiles method → native code
// 4. Next invocation uses compiled code (10-100x faster)
// 5. Process dies → JIT cache is lost (volatile, per-process)

AOT (Ahead-Of-Time) Compilation

The dex2oat tool compiles DEX bytecode to native machine code before execution — either at install time or during idle maintenance. The result is an .odex (optimized DEX) file stored on disk. AOT code is the fastest execution path: no interpretation overhead, no JIT compilation pauses.

Profile-Guided Compilation (PGO)

The key insight: most apps use only 10-20% of their code in any given session. Full AOT compilation wastes storage and install time compiling dead code. Profile-guided compilation compiles only the methods that are actually used — the best of both worlds.

Mode Execution Speed Install Time Storage Startup Impact
Interpreted Slowest (1x baseline) None None Slow — every method interpreted
JIT Fast after warmup (50-100x) None ~2MB code cache (volatile) Slow first launch, improves over time
Full AOT Fastest (50-100x) Very long (minutes) Large (.odex can be 2-5x DEX size) Fast — all code pre-compiled
Profile-guided AOT Fast for hot paths (50-100x) Moderate (seconds) Small (only hot methods) Fast for profiled paths

ART Compilation History

Why Full AOT Was Abandoned

Android 5-6's full AOT approach caused 3-10 minute install times and consumed 2-5x the DEX file size in storage. System updates triggered re-compilation of every app, sometimes taking 30+ minutes. Users saw "Optimizing apps..." for extended periods after OTA updates. The shift to JIT + profile-guided AOT in Android 7 was a direct response to these UX disasters.

2. How Profile-Guided Compilation Works

Profile-guided compilation is the mechanism that makes Baseline Profiles possible. Understanding the pipeline is critical for knowing what Baseline Profiles actually do at install time.

The JIT compiler tracks method execution in a profile info structure. When a method is invoked enough times (or contains a hot loop), it's marked as "hot". The runtime periodically flushes this profile to disk:

// Profile file location (per-user, per-package):
// /data/misc/profiles/cur/0/com.example.app/primary.prof

// Inspect a profile (requires root or debuggable build):
adb shell cmd package dump-profiles com.example.app

// Profile contains:
// - Hot methods (frequently executed)
// - Startup methods (executed during app startup)
// - Post-startup methods (executed after startup completes)
// - Classes used during startup (for dex layout optimization)

The bg-dexopt (background dex optimization) job runs when the device is idle and charging. It reads the profile and invokes dex2oat with the speed-profile compiler filter — compiling only the profiled methods to native code. Cold methods remain as DEX bytecode, interpreted or JIT-compiled on demand.

// dex2oat compiler filters (from least to most aggressive):
// verify        — only verify DEX bytecode, no compilation
// quicken       — verify + minor optimizations (inline checks)
// speed-profile — compile only profiled methods (default for bg-dexopt)
// speed         — compile everything (used for system apps)
// everything    — compile + optimize everything (debug/testing only)

// Check current compilation state:
adb shell dumpsys package com.example.app | grep -A5 "Dexopt state"
// Output: [status=speed-profile, reason=bg-dexopt]

3. The First-Launch Problem

This is the core problem that Baseline Profiles solve. On a fresh install, the profile-guided pipeline hasn't run yet:

  1. Fresh install — no primary.prof exists. The .odex file contains only verified bytecode (compiler filter: verify or quicken).
  2. First launch — every method is interpreted. The JIT kicks in for hot methods, but compilation takes time and CPU.
  3. After several sessions — the profile accumulates hot methods. But bg-dexopt hasn't run yet.
  4. After idle + charging — bg-dexopt compiles profiled methods. The app is now fast.
  5. Day 3-5 — the app finally reaches peak performance.
The Business Problem

Users judge your app by the first launch, not the 10th. App store ratings, retention, and conversion are all measured in the first session. A 500ms cold start that becomes 200ms after a week doesn't help — the user already formed their opinion (or uninstalled). Google's data shows that every 100ms of startup latency costs measurable retention. Baseline Profiles exist to make the first launch as fast as the 100th.

4. Baseline Profiles — Solving the First-Launch Problem

A Baseline Profile is a pre-built method profile that ships inside your APK/AAB. Instead of waiting days for the runtime to discover hot methods, you tell the system upfront: "these methods are critical — compile them at install time."

How It Works End-to-End

  1. Developer generates a profile by exercising critical user journeys in a Macrobenchmark test
  2. Profile is compiled into a binary format and bundled in the APK at assets/dexopt/baseline.prof (binary) and baseline-prof.txt (human-readable rules in the source)
  3. On install: the Play Store (or ProfileInstaller on first launch) feeds the profile to dex2oat
  4. dex2oat compiles the profiled methods AOT — producing an .odex with hot methods pre-compiled
  5. First launch: critical paths execute compiled native code instead of interpreted bytecode

Typical Performance Improvements

MetricWithout Baseline ProfileWith Baseline ProfileImprovement
Cold start (first install)~800ms~450ms~40% faster
Scroll jank (first session)15-20 janky frames/s3-5 janky frames/s~70% fewer janks
Screen transition~300ms~200ms~30% faster
JIT compilation pausesFrequent in first 30sMinimalNear-elimination

5. Baseline Profile Format

Baseline Profiles use a human-readable rule format in source (baseline-prof.txt) that gets compiled to a binary format (baseline.prof) during the build.

Rule Syntax

// Each line describes a method or class with flags:
// Flags: H = Hot, S = Startup, P = Post-startup, L = Library

// Hot method called during startup:
HSPLcom/example/app/MainActivity;->onCreate(Landroid/os/Bundle;)V

// Hot startup method in a library:
HSPLandroidx/compose/runtime/ComposerImpl;->startGroup(I)V

// Post-startup method (used after initial render):
HPLcom/example/app/DetailScreen;->loadData()V

// Class rule (ensures class is in startup dex layout):
Lcom/example/app/MainActivity;

// Wildcard rules — match all methods in a package:
HSPLcom/example/app/**;->**(**)**

// Wildcard — all methods in a specific class:
HSPLcom/example/app/Repository;->**(**)**

Flag Meanings

FlagMeaningEffect
HHotMethod is AOT-compiled with full optimization
SStartupMethod is needed during app startup — prioritized in compilation and dex layout
PPost-startupMethod is used after startup — compiled but not prioritized for dex layout
LLibraryMethod belongs to a library (informational, used for profile merging)
Binary vs Text Format

The baseline-prof.txt in your source is human-readable rules. During the build, the AGP (Android Gradle Plugin) compiles this into a binary .prof format using profgen. The binary format is what ships in the APK and what dex2oat reads. You never edit the binary format directly. The text format supports wildcards; the binary format resolves them to concrete method references.

6. Generating Baseline Profiles with Macrobenchmark

The recommended way to generate Baseline Profiles is using the Macrobenchmark library and the androidx.baselineprofile Gradle plugin. This automates the entire pipeline: run instrumented tests on a device, collect the profile, and merge it into your build.

Project Setup

// settings.gradle.kts
include(":app")
include(":baselineprofile")  // Separate module for profile generation

// build.gradle.kts (project-level)
plugins {
    id("com.android.application") version "8.2.0" apply false
    id("androidx.baselineprofile") version "1.3.0" apply false
}

// build.gradle.kts (:baselineprofile module)
plugins {
    id("com.android.test")
    id("androidx.baselineprofile")
}

android {
    namespace = "com.example.app.baselineprofile"
    compileSdk = 34
    targetProjectPath = ":app"

    defaultConfig {
        minSdk = 28  // Macrobenchmark requires API 28+
        testInstrumentationRunner = "androidx.test.runner.AndroidJUnit4"
    }
}

baselineProfile {
    managedDevices += "pixel6Api34"  // Gradle Managed Device
    useConnectedDevices = true
}

dependencies {
    implementation("androidx.test.ext:junit:1.1.5")
    implementation("androidx.test.uiautomator:uiautomator:2.3.0")
    implementation("androidx.benchmark:benchmark-macro-junit4:1.2.3")
}

// build.gradle.kts (:app module)
plugins {
    id("com.android.application")
    id("androidx.baselineprofile")
}

dependencies {
    baselineProfile(project(":baselineprofile"))

}

Profile Generator

// baselineprofile/src/main/java/.../BaselineProfileGenerator.kt

@RunWith(AndroidJUnit4::class)
class BaselineProfileGenerator {

    @get:Rule
    val rule = BaselineProfileRule()

    @Test
    fun generateProfile() {
        rule.collect(
            packageName = "com.example.app",
            maxIterations = 15,
            stableIterations = 3,
            includeInStartupProfile = true
        ) {
            // Critical user journey 1: App startup
            pressHome()
            startActivityAndWait()

            // Critical user journey 2: Main list scroll
            device.wait(Until.hasObject(By.res("main_list")), 10_000)
            val list = device.findObject(By.res("main_list"))
            list.setGestureMargin(device.displayWidth / 5)
            list.fling(Direction.DOWN)
            list.fling(Direction.UP)

            // Critical user journey 3: Navigate to detail
            device.findObject(By.res("item_card")).click()
            device.wait(Until.hasObject(By.res("detail_screen")), 5_000)

            // Critical user journey 4: Search
            device.pressBack()
            device.findObject(By.res("search_icon")).click()
            device.wait(Until.hasObject(By.res("search_field")), 3_000)
            device.findObject(By.res("search_field")).text = "test query"
            device.wait(Until.hasObject(By.res("search_results")), 5_000)
        }
    }
}

Running the Generator

./gradlew :app:generateBaselineProfile

Library Profiles

Libraries can ship their own Baseline Profiles. When your app depends on a library with a profile, the AGP automatically merges the library's profile with your app's profile. Major Jetpack libraries already ship profiles:

  • Compose Runtime/UI/Foundation — profiles for recomposition, layout, drawing
  • Navigation Compose — profiles for screen transitions
  • RecyclerView — profiles for scrolling, view recycling
  • Fragment — profiles for fragment transactions
  • Room — profiles for database operations

7. Startup Profiles (Android 13+)

Startup Profiles are a subset of Baseline Profiles that specifically target methods and classes needed during app startup. They enable a critical optimization that Baseline Profiles alone cannot: dex layout optimization.

What Startup Profiles Add

  • Class ordering — classes needed during startup are grouped contiguously in the DEX file
  • Method ordering — startup methods are placed together for better code locality
  • Reduced page faults — instead of loading scattered 4KB pages from disk, startup classes are in adjacent pages
// Startup Profile is generated alongside Baseline Profile:
rule.collect(
    packageName = "com.example.app",
    includeInStartupProfile = true  // This flag enables Startup Profile generation
) {
    // Only the startup journey contributes to the Startup Profile
    pressHome()
    startActivityAndWait()
    device.wait(Until.hasObject(By.res("main_content")), 10_000)
}

// Output files:
// baseline-prof.txt    → all hot methods (startup + post-startup)
// startup-prof.txt     → only startup methods (subset)

8. Cloud Profiles (Android 12+)

Cloud Profiles are Google Play's server-side solution. Play collects aggregated, anonymized runtime profiles from users who have opted in, then distributes these profiles to new installs.

How Cloud Profiles Work

  1. Collection: devices upload anonymized profile data to Play servers (methods + classes, no user data)
  2. Aggregation: Play merges profiles from thousands of users, keeping methods that appear in >threshold% of profiles
  3. Distribution: when a new user installs the app, Play downloads the aggregated profile alongside the APK
  4. Compilation: the profile is fed to dex2oat at install time, same as a Baseline Profile

Cloud Profiles vs Baseline Profiles

AspectCloud ProfilesBaseline Profiles
SourceAggregated from real usersDeveloper-generated from Macrobenchmark
Coverage~80% of hot methodsCovers exactly what you profile
AvailabilityDays after release (needs user data)Immediate (ships with APK)
DistributionPlay Store onlyAny install method
Android version12+ (API 31+)7+ (API 24+) via ProfileInstaller
Developer effortZero (automatic)Requires Macrobenchmark setup
Update latencyDays to propagate after app updateAvailable on same-day install
Cloud Profiles Are Not Enough

Cloud Profiles have a critical gap: they take days to aggregate after a new app version is published. Every user who installs in that window gets no profile. They also don't work for sideloaded apps, enterprise deployments, or pre-Play Store testing. Baseline Profiles fill these gaps — they're available from the moment the APK is built. The recommended strategy: ship Baseline Profiles AND rely on Cloud Profiles. They complement each other; Cloud Profiles may cover user journeys you didn't test in Macrobenchmark.

9. Dex Layout Optimization

Dex layout optimization reorders classes and methods within the DEX file to improve I/O locality during startup. This is a separate optimization from AOT compilation — it affects how quickly classes can be loaded from disk, not how fast they execute.

The Page Fault Problem

DEX files are memory-mapped (mmap) into the process address space. When a class is first accessed, the kernel loads the 4KB page containing that class from flash storage. If startup classes are scattered across the DEX file, each class load triggers a separate page fault — a random I/O operation on flash.

// Without dex layout optimization:
// Startup loads classes A, B, C, D (scattered in dex)
//
// DEX file on disk:
// [page 0: class X] [page 1: class A] [page 2: class Y]
// [page 3: class Z] [page 4: class B] [page 5: class W]
// [page 6: class C] [page 7: class V] [page 8: class D]
//
// Loading A, B, C, D = 4 page faults (random I/O)
// Each page fault: 0.1-1ms on flash → 0.4-4ms total

// With dex layout optimization (startup classes grouped):
// DEX file on disk:
// [page 0: class A] [page 1: class B] [page 2: class C]
// [page 3: class D] [page 4: class X] [page 5: class Y]
// [page 6: class Z] [page 7: class W] [page 8: class V]
//
// Loading A, B, C, D = 1 sequential read (pages 0-3)
// Sequential I/O: ~0.1ms total (kernel read-ahead kicks in)

How It's Applied

The dex2oat compiler uses the Startup Profile to determine class ordering. When compiling with a profile that includes class entries (lines starting with L), it reorders the DEX file so startup classes appear first. This happens at install time alongside AOT compilation.

// Startup Profile entries that drive dex layout:
// Class entries (no method, just the class descriptor):
Lcom/example/app/App;
Lcom/example/app/MainActivity;
Lcom/example/app/MainViewModel;
Lcom/example/app/data/Repository;
Lcom/example/app/di/AppModule;

// These classes will be grouped at the beginning of the DEX file
// Kernel read-ahead (typically 128KB) will prefetch adjacent pages
// Result: most startup classes loaded in 1-2 I/O operations
Staff-Level Insight — Quantifying Page Faults

You can measure page fault impact using adb shell am start -W --start-profiler combined with Perfetto's mm_filemap_add_to_page_cache tracepoint. On a typical app with 500+ classes loaded during startup, dex layout optimization reduces page faults from ~200 to ~30-50, saving 10-50ms on cold start depending on flash speed. On low-end devices with slower NAND, the savings can exceed 100ms. This is why Startup Profiles are a separate concept from Baseline Profiles — they serve a different optimization (I/O locality vs compilation).

10. Measuring Impact

Measuring the impact of Baseline Profiles requires controlled benchmarking with different compilation modes. The Macrobenchmark library provides this out of the box.

Benchmark with Compilation Modes

@RunWith(AndroidJUnit4::class)
class StartupBenchmark {

    @get:Rule
    val rule = MacrobenchmarkRule()

    @Test
    fun startupNoCompilation() {
        rule.measureRepeated(
            packageName = "com.example.app",
            metrics = listOf(StartupTimingMetric()),
            compilationMode = CompilationMode.None(),  // Interpreted + JIT only
            startupMode = StartupMode.COLD,
            iterations = 10
        ) {
            pressHome()
            startActivityAndWait()
        }
    }

    @Test
    fun startupBaselineProfile() {
        rule.measureRepeated(
            packageName = "com.example.app",
            metrics = listOf(StartupTimingMetric()),
            compilationMode = CompilationMode.Partial(
                baselineProfileMode = BaselineProfileMode.Require
            ),
            startupMode = StartupMode.COLD,
            iterations = 10
        ) {
            pressHome()
            startActivityAndWait()
        }
    }

    @Test
    fun startupFullCompilation() {
        rule.measureRepeated(
            packageName = "com.example.app",
            metrics = listOf(StartupTimingMetric()),
            compilationMode = CompilationMode.Full(),  // Full AOT (upper bound)
            startupMode = StartupMode.COLD,
            iterations = 10
        ) {
            pressHome()
            startActivityAndWait()
        }
    }
}

Compilation Modes Explained

Partial(Require)Installs Baseline Profile, compiles profiled methodsInstall with Baseline Profile
Partial(Suggest)Uses profile if available, falls back to NoneReal-world install behavior

Staff-Level Insight

At Google, Baseline Profiles are mandatory for all user-facing apps. The 30-50% cold start improvement is the single highest-ROI performance optimization available. If you're not shipping a Baseline Profile, you're leaving performance on the table for every new install and every app update. The cost is minimal — a one-time Macrobenchmark setup and a CI job that regenerates the profile periodically. The Compose team's own Baseline Profile reduced LazyColumn first-scroll jank by 40%. For L6 candidates: be prepared to explain the full pipeline from profile generation → binary compilation → install-time dex2oat → runtime execution, and articulate why Cloud Profiles alone are insufficient.
