01 — Android Memory Model

How memory is organized in an Android process — Java heap, native heap, stack, mmap regions, and how the OS accounts for it all

1. Process Memory Layout

  Virtual Address Space of an Android App Process
  ┌─────────────────────────────────────────────────┐  High address
  │  Kernel Space (inaccessible to userspace)        │
  ├─────────────────────────────────────────────────┤
  │  Stack (main thread)     ~1MB default            │  ← grows down
  │  Stack (thread 2)        ~1MB per thread         │
  │  Stack (thread N)                                │

  │  mmap regions                                    │

  │  ├── ashmem / SharedMemory                       │
  │  ├── APK (memory-mapped, read-only)              │
  │  ├── .so libraries (native code)                 │
  │  ├── .dex / .oat files (compiled code)           │
  │  ├── Font files, resource tables                 │
  │  └── GPU memory mappings                         │
  ├─────────────────────────────────────────────────┤
  │  Native Heap (malloc / jemalloc)                 │  ← grows up

  │  .text (executable code)                         │
  └─────────────────────────────────────────────────┘  Low address

2. Java/ART Heap — Spaces

// The ART runtime divides the managed heap into several spaces:

// 1. RegionSpace (main allocation space, Android 8+):
// - Used by the Concurrent Copying (CC) collector
// - Divided into 256KB regions
// - New objects allocated via bump pointer (fast)
// - During GC: live objects copied to new regions, old regions freed
// - This is where most of your objects live

// 2. Large Object Space (LOS):
// - Objects > 12KB (3 pages) go here
// - Allocated individually via mmap
// - Not compacted — freed individually
// - Large arrays, big Bitmaps (pre-Android 8), large strings

// 3. Zygote Space:
// - Objects allocated during Zygote preloading (before fork)
// - Shared across all app processes via COW
// - Never collected by app's GC (belongs to Zygote)
// - ~30-50MB of framework classes, resources, etc.

// 4. Image Space:
// - Pre-compiled .art files (boot image)
// - Memory-mapped, shared across processes
// - Contains pre-initialized objects (String pool, Class objects)
// - Read-only after loading

// 5. Non-Moving Space:
// - Objects that can't be moved (JNI global refs, interned strings)
// - Used by mark-sweep, not copying collector
// - Fragmentation risk — kept small

// Heap size limits:
// Per-app heap limit set by the device manufacturer:
// dalvik.vm.heapsize = 256m (typical)
// dalvik.vm.heapgrowthlimit = 128m (default limit)
// android:largeHeap="true" → uses heapsize instead of heapgrowthlimit
// Check at runtime: Runtime.getRuntime().maxMemory()
Staff-Level Insight — largeHeap is a Code Smell

android:largeHeap="true" doubles the heap limit but signals to the system that your app is memory-hungry. The OOM killer will target your app more aggressively when memory is low. It also increases GC pause times (more heap to scan). At Google, using largeHeap requires justification — photo editors and video apps may need it, but most apps should optimize their memory usage instead. If you need largeHeap, you likely have a leak or are caching too aggressively.

3. Native Heap — The Unmanaged Side

// Native heap is allocated via malloc() (jemalloc on Android).
// NOT managed by the GC — you must free it yourself (or use RAII in C++).

// What lives on the native heap:
// 1. Bitmap pixel data (Android 8+ moved from Java heap to native)
//    - Bitmap.createBitmap(1000, 1000, ARGB_8888) = 4MB native memory
//    - Freed when Bitmap is GC'd (weak reference + native destructor)
//    - Before Android 8: Bitmap pixels were on the Java heap (counted toward limit)

// 2. RenderNode / DisplayList data
//    - Hardware-accelerated drawing commands
//    - Created by View.draw() → Canvas → RenderNode
//    - Freed when View is detached and GC'd

// 3. SQLite page cache
//    - Default page cache: 2000 pages × 4KB = ~8MB per database
//    - Plus WAL file mapping

// 4. NDK allocations (your native code)
// 5. WebView renderer (Chromium)
//    - Can use 50-200MB of native memory
//    - Separate process on Android 8+ (isolated)

// 6. ART internal structures
//    - JIT code cache (~2MB)
//    - Class metadata, method tables
//    - Thread stacks for GC workers

// Native memory is NOT subject to the Java heap limit.
// An app with 128MB Java heap limit can use 500MB+ of native memory.
// This is a common source of OOM on low-memory devices.

4. Stack Memory

// Each thread gets its own stack (default ~1MB on Android):
// - Local variables (primitives, object references)
// - Method call frames
// - Return addresses

// Stack is allocated when the thread is created:
// pthread_attr_setstack() in native, Thread() in Java
// Default sizes:
// - Main thread: ~8MB (set by Zygote)
// - Java threads: ~1MB (configurable via -Xss, rarely changed)
// - Coroutine: NO dedicated stack — uses the dispatcher thread's stack
//   (this is why coroutines are lightweight — no 1MB stack per coroutine)

// StackOverflowError:
// - Deep recursion exhausts the stack
// - Common with: recursive layout inflation, deep View hierarchies,
//   recursive JSON parsing, unbounded tree traversal
// - Fix: convert recursion to iteration, or increase stack size for specific threads

// Stack vs Heap for objects:
// Java: ALL objects are heap-allocated (even small ones)
// Kotlin: value classes with @JvmInline can be stack-allocated (scalar replacement)
// ART optimization: escape analysis can stack-allocate objects that don't escape
//   the method — but this is limited and not guaranteed

5. Memory Metrics — PSS, RSS, USS, VSS

// The OS tracks memory usage with several metrics:

// VSS (Virtual Set Size):
// Total virtual address space mapped (including unmapped pages)
// Includes: code, data, mmap, heap, stack, shared libs
// NOT useful for real memory usage — can be 2GB+ for a simple app

// RSS (Resident Set Size):
// Physical pages currently in RAM (not swapped/paged out)
// Includes shared pages at FULL size (double-counted across processes)
// Useful for: "how much RAM is this process touching right now?"

// PSS (Proportional Set Size):
// Like RSS but shared pages are divided by the number of sharing processes
// If 3 processes share a 12KB page, each is charged 4KB
// This is what Android uses for memory accounting and OOM decisions
// "dumpsys meminfo" reports PSS

// USS (Unique Set Size) / Private Dirty:
// Pages used ONLY by this process (not shared with anyone)
// This is the "real cost" of your app — memory freed if you kill the process
// Most important metric for optimization

// Viewing memory:
// adb shell dumpsys meminfo 
// Shows: Java Heap, Native Heap, Code, Stack, Graphics, Private Dirty, Shared Clean
// adb shell cat /proc//smaps  — detailed per-mapping breakdown
// adb shell procrank — sorted by PSS across all processes
Metric Shared Pages Use Case Command
VSS Full count Virtual space (not useful) cat /proc/pid/status
RSS Full count Physical pages in RAM cat /proc/pid/status
PSS Proportional Fair accounting (OOM decisions) dumpsys meminfo
USS Excluded True per-app cost dumpsys meminfo (Private Dirty)

Staff-Level Insight — Bitmap Migration (Android 8)

Before Android 8, Bitmap pixel data lived on the Java heap and counted toward the heap limit. This meant a 128MB heap could be exhausted by a few large images. Android 8 moved Bitmap pixels to the native heap — they no longer count toward the Java heap limit. This dramatically reduced OOM crashes for image-heavy apps but created a new problem: native memory leaks from Bitmaps are invisible to the Java heap monitor. You must track native heap growth separately. Use Debug.getNativeHeapAllocatedSize() to monitor.

7. Memory Accounting in Practice

// Programmatic memory checks:

// Java heap:
val runtime = Runtime.getRuntime()
val maxMemory = runtime.maxMemory()         // Heap limit (128MB typical)
val totalMemory = runtime.totalMemory()     // Current heap size
val freeMemory = runtime.freeMemory()       // Free within current heap
val usedMemory = totalMemory - freeMemory   // Currently allocated

// Native heap:
val nativeHeapSize = Debug.getNativeHeapSize()
val nativeHeapAlloc = Debug.getNativeHeapAllocatedSize()
val nativeHeapFree = Debug.getNativeHeapFreeSize()

// Full memory info (requires Debug permission or same process):
val memInfo = Debug.MemoryInfo()
Debug.getMemoryInfo(memInfo)
val totalPss = memInfo.totalPss              // Total PSS in KB
val javaPss = memInfo.dalvikPss              // Java heap PSS
val nativePss = memInfo.nativePss            // Native heap PSS
val otherPss = memInfo.otherPss              // Everything else

// ActivityManager memory class (device capability):
val am = getSystemService(ACTIVITY_SERVICE) as ActivityManager
val memoryClass = am.memoryClass             // MB (e.g., 128, 256)
val largeMemoryClass = am.largeMemoryClass   // MB with largeHeap
// Use memoryClass to size your caches:
// val cacheSize = memoryClass * 1024 * 1024 / 8  // 1/8 of heap for image cache

8. NativeAllocationRegistry — Bridging Native Memory with GC

// Problem: Android 8+ moved Bitmap pixels to native heap.
// The GC only tracks Java heap — it has no idea about native allocations.
// Without intervention, an app could allocate 500MB of Bitmaps
// and the GC would never trigger because the Java heap is only 30MB.

// Solution: NativeAllocationRegistry (android.os.NativeAllocationRegistry)
// Registers a native allocation size + free function with a Java object.
// When the Java object is GC'd, the native memory is freed.
// CRITICALLY: it also tells ART's GC about the native memory size.

// How it works internally:
// 1. Java object (e.g., Bitmap) is created
// 2. Native code allocates pixel buffer via malloc (e.g., 4MB)
// 3. NativeAllocationRegistry.registerNativeAllocation(size) is called
//    → ART adds 4MB to its internal "native bytes allocated" counter
//    → If native bytes exceed the GC trigger threshold → GC fires
// 4. A Cleaner (PhantomReference) is attached to the Java object
// 5. When the Java object becomes unreachable:
//    → GC enqueues the PhantomReference
//    → Cleaner thread calls the native free function (e.g., free(pixelBuffer))
//    → ART decrements the native bytes counter

// This is why Bitmaps on Android 8+ don't cause Java OOM
// but DO trigger GC when you allocate too many of them.

// Used by: Bitmap, Typeface, HardwareBuffer, NativeAllocationRegistry
// Source: frameworks/base/core/java/android/os/NativeAllocationRegistry.java

// Kotlin example — how Bitmap uses it:
// (simplified from frameworks/base/graphics/java/android/graphics/Bitmap.java)
//
// class Bitmap {
//     init {
//         val registry = NativeAllocationRegistry(
//             Bitmap::class.java.classLoader,
//             nativeGetNativeFinalizer(),  // pointer to C++ destructor
//             allocationByteCount.toLong()
//         )
//         registry.registerNativeAllocation(this, nativePtr)
//     }
// }
Staff-Level Insight — Cleaner vs finalize()

Before Android 8, native cleanup used finalize() — a method called by the GC before reclaiming an object. Problems: (1) Finalizers run on a single FinalizerDaemon thread — if one finalizer blocks, ALL finalization stalls, causing memory buildup. (2) Finalizable objects require two GC cycles to reclaim (first GC marks for finalization, second GC actually frees). (3) The FinalizerWatchdogDaemon kills the process if a finalizer takes >10 seconds. NativeAllocationRegistry uses sun.misc.Cleaner (a PhantomReference subclass) instead — Cleaners run on the ReferenceQueueDaemon thread, don't delay GC, and objects are reclaimable in a single GC cycle. At L6, you should know: never use finalize() in modern Android code. Use Cleaner (Java 9+) or NativeAllocationRegistry for native resource cleanup.

9. TLAB — Thread-Local Allocation Buffers

// TLAB is ART's fast-path allocation mechanism.
// Each thread gets a private chunk of RegionSpace (typically one 256KB region).
// Allocation within a TLAB is a simple bump-pointer increment — no locks needed.

// Allocation fast path (pseudocode from ART):
// Object* Allocate(size_t size) {
//     if (tlab_pos + size <= tlab_end) {
//         Object* obj = tlab_pos;
//         tlab_pos += size;
//         return obj;  // No lock, no CAS, just pointer bump
//     }
//     return SlowPathAllocate(size);  // Request new TLAB from RegionSpace
// }

// SlowPathAllocate:
// 1. Return current TLAB to RegionSpace
// 2. Request a new free region (requires lock, but infrequent)
// 3. If no free regions → trigger GC
// 4. If GC doesn't free enough → grow heap (up to max)
// 5. If at max → OutOfMemoryError

// Why this matters for performance:
// - Most allocations (>99%) hit the TLAB fast path
// - Cost: ~3 CPU instructions (load, add, store)
// - No synchronization needed (thread-local)
// - This is why object allocation in ART is nearly free
// - BUT: high allocation RATE still causes frequent TLAB exhaustion
//   → frequent slow-path calls → frequent GC triggers

10. Page Reclaim — madvise and Memory Return

// After GC, ART has freed regions in RegionSpace.
// But the OS still thinks those pages are "used" (RSS stays high).
// ART must explicitly tell the kernel to reclaim the physical pages.

// ART uses madvise() system call:
// madvise(addr, length, MADV_DONTNEED)
//   → Kernel marks pages as "not needed"
//   → Physical pages are freed immediately
//   → Virtual address mapping remains (no munmap needed)
//   → Next access triggers a page fault → kernel allocates a new zero page

// When ART calls madvise:
// 1. After every GC: freed RegionSpace regions → MADV_DONTNEED
// 2. On background transition: aggressive compaction + MADV_DONTNEED
// 3. Heap trim: if free space > heapMaxFree → shrink and MADV_DONTNEED

// Android 12+ also uses:
// madvise(addr, length, MADV_FREE)
//   → "Lazy" reclaim: kernel MAY reclaim the pages if under pressure
//   → If not reclaimed, the data is still there (faster re-access)
//   → Better performance than MADV_DONTNEED when memory isn't tight

// This is why an app's RSS drops after going to background:
// ART compacts the heap and calls madvise to return pages to the OS.
// The app's virtual address space stays the same, but physical RAM is freed.
