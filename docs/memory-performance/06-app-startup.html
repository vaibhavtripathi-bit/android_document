06 — App Startup Optimization

Cold, warm, and hot start internals — Zygote fork to first frame, ContentProvider cost, App Startup library, Splash Screen API, reportFullyDrawn, and Macrobenchmark measurement

1. Cold / Warm / Hot Start Definitions

Android distinguishes three startup modes based on how much of the app's process and Activity state already exists in memory. The distinction matters because each mode exercises a completely different code path through the framework.

Start TypeProcess StateActivity StateWhat HappensTypical Time
Cold Not running Does not exist Zygote fork → Application.attachBaseContext → ContentProviders → Application.onCreate → Activity.onCreate → onStart → onResume → first frame 300ms–1.5s+
Warm Alive in memory Destroyed (finished or killed) Activity.onCreate → onStart → onResume → first frame (skips process creation, Application init, ContentProviders) 150ms–500ms
Hot Alive in memory Stopped (in background) Activity.onRestart → onStart → onResume → first frame (skips onCreate entirely) 50ms–150ms 
// How the system decides which start type to use:
//
// 1. User taps the launcher icon
// 2. System (AMS) checks: is the app's process alive?
//    NO  → COLD START (fork from Zygote, full initialization)
//    YES → Is the Activity instance still in memory?
//           NO  → WARM START (process alive, but Activity was destroyed)
//           YES → HOT START (Activity is stopped, just bring to front)
//
// Key distinction for warm start:
// The process survived (Application object, singletons, static state all intact),
// but the Activity was destroyed — either by the user pressing Back, or by the
// system reclaiming memory (onSaveInstanceState was called).
//
// What "first frame" means:
// The system measures startup time from the intent to the first
// ViewTreeObserver.OnDrawListener callback after the Activity's window
// becomes visible. This is the TTID (Time To Initial Display) metric.
//
// Google Play Vitals thresholds:
// Cold start: excessive if > 5s (warning at > 3s)
// Warm start: excessive if > 2s
// Hot start:  excessive if > 1.5s
//
// Google's internal target for flagship apps: < 500ms cold start
Warm Start Is Not What Most Developers Think

Many developers assume "warm start" means the app was recently in the background. That's actually a hot start. A warm start specifically means the process is alive but the Activity was destroyed — the system must call Activity.onCreate() again, re-inflate the layout, and re-bind data. This happens when the user presses Back and then re-launches, or when the system destroyed the Activity to reclaim memory while the process survived. The process-level state (Application, singletons, cached data) is still in memory, which is why it's faster than cold.

The 500ms Budget

Google's target for cold start is under 500ms from process creation to first frame. On a Pixel device with a release build, this is achievable. But consider what eats into that budget: Zygote fork (~50ms, fixed), ActivityThread.main() setup (~30ms, fixed), ContentProvider initialization (~100-300ms, you control this), Application.onCreate() (~50-200ms, you control this), Activity creation + layout inflation (~100-200ms, you control this), first draw (~50ms, mostly fixed). The framework overhead is ~130ms. That leaves you ~370ms for everything your app does before the first frame.

3. ContentProvider Init Cost

ContentProviders are the silent killer of startup performance. They're initialized before Application.onCreate(), on the main thread, with no way to defer or parallelize them through the standard manifest mechanism.

// Every ContentProvider declared in the merged manifest is instantiated
// and has its onCreate() called during handleBindApplication().
//
// The problem: libraries use ContentProviders for auto-initialization.
// Each library adds its own ContentProvider to your merged manifest
// via manifest merger, and each one runs before your code.
//
// Common offenders and their approximate cost:
// ┌────────────────────────────────────┬──────────────┬──────────────────────┐
// │ Library                            │ ContentProvider                     │
// │                                    │ Approx. Cost │ What It Initializes  │
// ├────────────────────────────────────┼──────────────┼──────────────────────┤
// │ Firebase Core                      │ ~50-100ms    │ FirebaseApp, analytics│
// │ Firebase Crashlytics               │ ~30-60ms     │ NDK crash handler     │
// │ WorkManager                        │ ~20-40ms     │ WorkManagerImpl       │
// │ AndroidX Lifecycle                 │ ~10-20ms     │ ProcessLifecycleOwner │
// │ AndroidX Startup                   │ ~5-10ms      │ InitializationProvider│
// │ Facebook SDK                       │ ~40-80ms     │ FB App Events         │
// │ Google Play Services (Ads)         │ ~30-50ms     │ Ad initialization     │
// │ Leak Canary (debug)                │ ~20-40ms     │ Watcher installation  │
// │ Room (if auto-migrating)           │ ~10-30ms     │ Database verification │
// └────────────────────────────────────┴──────────────┴──────────────────────┘
//
// Worst case: 10 libraries × 50ms each = 500ms BEFORE Application.onCreate()
// Your entire startup budget is gone before a single line of your code runs.

4. Jetpack App Startup Library

The App Startup library solves the ContentProvider proliferation problem by consolidating all library initializations into a single ContentProvider and providing a dependency-aware lazy initialization mechanism.

// Dependency:
// implementation "androidx.startup:startup-runtime:1.1.1"

// How it works:
// 1. App Startup provides ONE ContentProvider: InitializationProvider
// 2. Libraries implement the Initializer<T> interface instead of their own CP
// 3. InitializationProvider reads metadata from the manifest and calls
//    each Initializer in dependency order
// 4. Libraries can be lazily initialized (not at startup) if desired

// Step 1: Implement an Initializer for your library
class WorkManagerInitializer : Initializer<WorkManager> {

    override fun create(context: Context): WorkManager {
        val config = Configuration.Builder()
            .setMinimumLoggingLevel(Log.INFO)
            .build()
        WorkManager.initialize(context, config)
        return WorkManager.getInstance(context)
    }

    // Declare dependencies: what must be initialized first?
    override fun dependencies(): List<Class<out Initializer<*>>> {
        // WorkManager depends on nothing — empty list
        return emptyList()
    }
}

class AnalyticsInitializer : Initializer<AnalyticsTracker> {

    override fun create(context: Context): AnalyticsTracker {
        // This runs AFTER WorkManagerInitializer because we declared the dependency
        val tracker = AnalyticsTracker.Builder(context)
            .setEndpoint(BuildConfig.ANALYTICS_URL)
            .build()
        tracker.initialize()
        return tracker
    }

    override fun dependencies(): List<Class<out Initializer<*>>> {
        // Analytics depends on WorkManager (for batched event upload)
        return listOf(WorkManagerInitializer::class.java)
    }
}

Manifest Configuration

// In your AndroidManifest.xml, declare initializers as metadata
// on the InitializationProvider:

// <provider
//     android:name="androidx.startup.InitializationProvider"
//     android:authorities="${applicationId}.androidx-startup"
//     android:exported="false"
//     tools:node="merge">
//
//     <meta-data
//         android:name="com.example.WorkManagerInitializer"
//         android:value="androidx.startup" />
//
//     <meta-data
//         android:name="com.example.AnalyticsInitializer"
//         android:value="androidx.startup" />
// </provider>

// REMOVING a library's auto-init ContentProvider:
// Many libraries (WorkManager, Lifecycle) already support App Startup.
// To disable their standalone ContentProvider and use App Startup instead:

// <provider
//     android:name="androidx.work.impl.WorkManagerInitializer"
//     android:authorities="${applicationId}.workmanager-init"
//     tools:node="remove" />
//
// tools:node="remove" tells the manifest merger to strip this provider
// from the final merged manifest. Now WorkManager won't auto-init via
// its own ContentProvider — you control when it initializes.

Lazy (Manual) Initialization

// For libraries that aren't needed at startup, skip auto-init entirely
// and initialize them on-demand:

// 1. Disable auto-init in the manifest:
// <provider
//     android:name="androidx.startup.InitializationProvider"
//     android:authorities="${applicationId}.androidx-startup"
//     android:exported="false"
//     tools:node="merge">
//     <meta-data
//         android:name="com.example.AnalyticsInitializer"
//         android:value="androidx.startup"
//         tools:node="remove" />   <!-- Remove from auto-init -->
// </provider>

// 2. Initialize manually when needed:
class HomeFragment : Fragment() {
    override fun onViewCreated(view: View, savedInstanceState: Bundle?) {
        super.onViewCreated(view, savedInstanceState)

        // Initialize analytics lazily — only when the user reaches Home
        val analytics = AppInitializer.getInstance(requireContext())
            .initializeComponent(AnalyticsInitializer::class.java)

        analytics.trackScreenView("home")
    }
}

// Benefits:
// - Startup cost reduced: analytics init moved from cold start to first screen
// - Dependencies are still resolved: if AnalyticsInitializer depends on
//   WorkManagerInitializer, App Startup initializes WorkManager first
// - Thread-safe: AppInitializer handles concurrent initialization calls
// - Idempotent: calling initializeComponent() twice returns the cached result
Measuring the Impact

A real-world app with Firebase, WorkManager, Lifecycle, Crashlytics, and a custom analytics SDK had 5 ContentProviders adding ~280ms to cold start. After migrating to App Startup with lazy init for analytics and Crashlytics, cold start dropped to ~120ms of ContentProvider overhead — a single InitializationProvider running only the two initializers needed before first frame. The remaining three initialized lazily within the first 2 seconds of user interaction.

5. Splash Screen API (Android 12+)

Before Android 12, the system showed a blank window (with the app's windowBackground theme) during cold start. Developers hacked around this with custom splash Activities or themed windows. Android 12 introduced a proper Splash Screen API that the system controls, ensuring a consistent, fast experience.

// The Splash Screen API works at the WINDOW level, not the Activity level.
// The system draws the splash screen BEFORE your Activity's first frame.
//
// How it works internally:
// 1. AMS creates a "starting window" (splash) immediately when launching the app
// 2. The splash is drawn by the system (WindowManagerService), not your app
// 3. Your app process forks, initializes, creates the Activity
// 4. When the Activity's first frame is ready, the system cross-fades
//    from the splash to your Activity's content
//
// This means the user sees the splash INSTANTLY — no blank screen.

// Theme configuration (in res/values/themes.xml):
// <style name="Theme.App" parent="Theme.MaterialComponents.DayNight">
//     <!-- Splash screen background color -->
//     <item name="android:windowSplashScreenBackground">@color/brand_dark</item>
//
//     <!-- Center icon (adaptive icon or AnimatedVectorDrawable) -->
//     <item name="android:windowSplashScreenAnimatedIcon">@drawable/splash_icon</item>
//
//     <!-- Icon animation duration (max 1000ms) -->
//     <item name="android:windowSplashScreenAnimationDuration">500</item>
//
//     <!-- Branding image at the bottom (optional, NOT recommended by Google) -->
//     <item name="android:windowSplashScreenBrandingImage">@drawable/brand</item>
// </style>

// Compat library for Android 5+ (backports the API):
// implementation "androidx.core:core-splashscreen:1.0.1"

Controlling the Splash Screen from Code

class MainActivity : AppCompatActivity() {

    private val viewModel: MainViewModel by viewModels()

    override fun onCreate(savedInstanceState: Bundle?) {
        // MUST be called before super.onCreate()
        val splashScreen = installSplashScreen()

        super.onCreate(savedInstanceState)
        setContentView(R.layout.activity_main)

        // Keep the splash screen visible while loading initial data.
        // The splash stays until this condition returns false.
        splashScreen.setKeepOnScreenCondition {
            viewModel.isLoading.value
        }
        // How it works internally:
        // The system checks this condition on every frame via a
        // ViewTreeObserver.OnPreDrawListener. As long as it returns true,
        // the Activity's content is not drawn (the splash stays).
        // When it returns false, the system removes the splash and
        // draws the Activity's first frame.

        // Custom exit animation — animate the splash screen away
        splashScreen.setOnExitAnimationListener { splashScreenView ->
            // splashScreenView is the actual View of the splash screen
            // You can animate it however you want

            val slideUp = ObjectAnimator.ofFloat(
                splashScreenView,
                View.TRANSLATION_Y,
                0f,
                -splashScreenView.height.toFloat()
            ).apply {
                interpolator = AnticipateInterpolator()
                duration = 300L
            }

            val fadeOut = ObjectAnimator.ofFloat(
                splashScreenView,
                View.ALPHA,
                1f,
                0f
            ).apply {
                duration = 300L
            }

            AnimatorSet().apply {
                playTogether(slideUp, fadeOut)
                doOnEnd { splashScreenView.remove() }
                start()
            }
        }
    }
}

class MainViewModel : ViewModel() {
    val isLoading = MutableStateFlow(true)

    init {
        viewModelScope.launch {
            // Load critical data needed for first screen
            val config = async { repository.fetchConfig() }
            val user = async { repository.getCurrentUser() }
            config.await()
            user.await()
            isLoading.value = false
        }
    }
}
Don't Abuse setKeepOnScreenCondition

Holding the splash screen for too long defeats its purpose. The splash should only stay while loading data that is absolutely required for the first frame — authentication state, critical config, or feature flags. If you hold it for 3+ seconds to load a full feed, users perceive the app as slow. Instead, show a skeleton UI or shimmer placeholder and load data asynchronously. Google Play Vitals measures TTID (Time To Initial Display) from the splash, so a long splash still counts as slow startup.

6. reportFullyDrawn()

Activity.reportFullyDrawn() tells the system when your app is actually usable — not just when the first frame was drawn, but when the meaningful content is visible and interactive. This is the TTFD (Time To Full Display) metric.

// TTID vs TTFD:
// TTID (Time To Initial Display): first frame rendered — may be a skeleton/placeholder
// TTFD (Time To Full Display): app is usable — real content is visible
//
// The system automatically measures TTID. YOU must report TTFD.
//
// When you call reportFullyDrawn():
// 1. A "fully drawn" timestamp appears in Perfetto/Systrace
// 2. Google Play Vitals records it as your app's TTFD metric
// 3. Macrobenchmark can measure it via StartupTimingMetric
//
// In logcat, you'll see:
// I/ActivityTaskManager: Fully drawn com.example.app/.MainActivity: +1s200ms

class FeedActivity : AppCompatActivity() {

    private var reportedFullyDrawn = false

    override fun onCreate(savedInstanceState: Bundle?) {
        super.onCreate(savedInstanceState)
        setContentView(R.layout.activity_feed)

        val recyclerView = findViewById<RecyclerView>(R.id.feed_list)
        val adapter = FeedAdapter()
        recyclerView.adapter = adapter

        lifecycleScope.launch {
            val feedItems = repository.loadFeed()
            adapter.submitList(feedItems)

            // Report fully drawn AFTER the data is loaded AND rendered.
            // Post to ensure the RecyclerView has laid out the items.
            recyclerView.post {
                if (!reportedFullyDrawn) {
                    reportFullyDrawn()
                    reportedFullyDrawn = true
                }
            }
        }
    }
}

// More precise: wait for the RecyclerView to actually render items
class FeedActivity : AppCompatActivity() {

    override fun onCreate(savedInstanceState: Bundle?) {
        super.onCreate(savedInstanceState)
        setContentView(R.layout.activity_feed)

        val recyclerView = findViewById<RecyclerView>(R.id.feed_list)

        // Use a pre-draw listener to detect when content is actually on screen
        recyclerView.viewTreeObserver.addOnPreDrawListener(
            object : ViewTreeObserver.OnPreDrawListener {
                override fun onPreDraw(): Boolean {
                    if (recyclerView.childCount > 0) {
                        recyclerView.viewTreeObserver.removeOnPreDrawListener(this)
                        reportFullyDrawn()
                        return true
                    }
                    return true
                }
            }
        )

        lifecycleScope.launch {
            val items = repository.loadFeed()
            (recyclerView.adapter as FeedAdapter).submitList(items)
        }
    }
}

FullyDrawnReporter for Jetpack Compose

// Starting with Activity 1.7+ and Compose, use FullyDrawnReporter
// to coordinate multiple async data sources before reporting.

// FullyDrawnReporter tracks multiple "reporters" — it calls
// reportFullyDrawn() only when ALL reporters have completed.

class ComposeActivity : ComponentActivity() {

    override fun onCreate(savedInstanceState: Bundle?) {
        super.onCreate(savedInstanceState)
        setContent {
            MyApp()
        }
    }
}

@Composable
fun MyApp() {
    val feedState by feedViewModel.uiState.collectAsStateWithLifecycle()
    val profileState by profileViewModel.uiState.collectAsStateWithLifecycle()

    // ReportDrawnWhen — reports fully drawn when the condition is true
    ReportDrawnWhen {
        feedState is FeedUiState.Success && profileState is ProfileUiState.Success
    }

    // Alternatively, for more control:
    // ReportDrawnAfter — reports after the suspending block completes
    ReportDrawnAfter {
        feedViewModel.waitForInitialLoad()
    }

    Column {
        ProfileHeader(profileState)
        FeedList(feedState)
    }
}

// Under the hood, ReportDrawnWhen uses FullyDrawnReporter:
// 1. It registers a "reporter" with the Activity's FullyDrawnReporter
// 2. When the condition becomes true, it removes the reporter
// 3. When ALL reporters are removed, FullyDrawnReporter calls reportFullyDrawn()
//
// This is critical for Compose screens with multiple async data sources:
// you don't want to report "fully drawn" until ALL sections have data.

7. Measuring Startup with Macrobenchmark

Macrobenchmark is the gold standard for measuring startup performance. It launches your app from outside the process (like a real user), measures real timing, and captures Perfetto traces for analysis.

// Module setup: create a :macrobenchmark module (or :benchmark)
// build.gradle.kts:
// plugins {
//     id("com.android.test")
//     id("org.jetbrains.kotlin.android")
// }
// android {
//     targetProjectPath = ":app"
//     experimentalProperties["android.experimental.self-instrumenting"] = true
// }
// dependencies {
//     implementation("androidx.benchmark:benchmark-macro-junit4:1.2.4")
// }

@RunWith(AndroidJUnit4::class)
class StartupBenchmark {

    @get:Rule
    val benchmarkRule = MacrobenchmarkRule()

    @Test
    fun startupCold() = benchmarkRule.measureRepeated(
        packageName = "com.example.app",
        metrics = listOf(StartupTimingMetric()),
        iterations = 10,
        startupMode = StartupMode.COLD,
        setupBlock = {
            // Runs before each iteration
            pressHome()
            // For COLD: the framework kills the process automatically
        }
    ) {
        // measureBlock: the actual startup action
        startActivityAndWait()
        // startActivityAndWait() launches the default Activity and waits
        // until the first frame is drawn (TTID).
        // If reportFullyDrawn() is called, it also captures TTFD.
    }

    @Test
    fun startupWarm() = benchmarkRule.measureRepeated(
        packageName = "com.example.app",
        metrics = listOf(StartupTimingMetric()),
        iterations = 10,
        startupMode = StartupMode.WARM
    ) {
        startActivityAndWait()
    }

    @Test
    fun startupHot() = benchmarkRule.measureRepeated(
        packageName = "com.example.app",
        metrics = listOf(StartupTimingMetric()),
        iterations = 10,
        startupMode = StartupMode.HOT
    ) {
        startActivityAndWait()
    }
}

// StartupTimingMetric captures:
// - timeToInitialDisplayMs: time from intent to first frame (TTID)
// - timeToFullDisplayMs: time from intent to reportFullyDrawn() (TTFD)
//
// Output example:
// StartupBenchmark_startupCold
//   timeToInitialDisplayMs   min 412.3, median 438.7, max 521.1

    startupMode = StartupMode.COLD

8. Startup Optimization Techniques

Lazy Initialization in Application.onCreate()

// BAD: initializing everything eagerly in Application.onCreate()
class MyApplication : Application() {
    override fun onCreate() {
        super.onCreate()
        Firebase.initialize(this)          // ~80ms
        CrashReporter.init(this)           // ~40ms
        AnalyticsTracker.init(this)        // ~30ms
        ImageLoader.init(this)             // ~20ms
        NetworkClient.init(this)           // ~15ms
        FeatureFlags.init(this)            // ~25ms (network call!)
        AdSdk.init(this)                   // ~50ms
        // Total: ~260ms before Activity.onCreate() even starts
    }
}

// GOOD: only initialize what's needed for the first frame
class MyApplication : Application() {

    override fun onCreate() {
        super.onCreate()
        // Only critical-path items: crash reporting + image loader
        CrashReporter.init(this)           // ~40ms — need this for safety
        ImageLoader.init(this)             // ~20ms — needed for first frame
        // Total: ~60ms — 200ms saved
    }

    // Lazy singletons for everything else
    val analytics: AnalyticsTracker by lazy { AnalyticsTracker(this) }
    val networkClient: NetworkClient by lazy { NetworkClient(this) }
    val featureFlags: FeatureFlags by lazy { FeatureFlags(this) }
}

Background Thread Initialization

// Move non-UI initialization off the main thread.
// Use a background thread or coroutine for anything that doesn't
// need to touch Views or the main thread's Looper.

class MyApplication : Application() {

    private val initScope = CoroutineScope(
        SupervisorJob() + Dispatchers.Default + CoroutineName("AppInit")
    )

    override fun onCreate() {
        super.onCreate()

        // Critical path (main thread): only what's needed before first frame
        CrashReporter.init(this)

        // Background initialization: everything else
        initScope.launch {
            // These run concurrently on background threads
            launch { Firebase.initialize(this@MyApplication) }
            launch { AnalyticsTracker.init(this@MyApplication) }
            launch { AdSdk.init(this@MyApplication) }
            launch {
                // Preload Dagger component graph on background thread
                // so it's ready when the first injection happens
                DaggerAppComponent.create()
            }
        }
    }
}

// CAUTION: some libraries REQUIRE main thread initialization.
// Firebase, for example, must be initialized on the main thread.
// Always check library documentation before moving init to background.
// If unsure, wrap in a try-catch and fall back to main thread init.

Reducing Layout Inflation Cost

// Layout inflation is a significant part of Activity.onCreate() cost.
// Complex view hierarchies with deep nesting are expensive to inflate.

// Techniques to reduce inflation cost:

// 1. Use ViewStub for views not needed immediately
// <ViewStub
//     android:id="@+id/error_stub"
//     android:layout="@layout/error_view"
//     android:layout_width="match_parent"
//     android:layout_height="wrap_content" />

// ViewStub costs ~0ms at inflation time. Inflate only when needed:
val errorView = findViewById<ViewStub>(R.id.error_stub).inflate()

// 2. Flatten view hierarchy — use ConstraintLayout instead of nested LinearLayouts
// Deep nesting: measure/layout cost grows exponentially
// ConstraintLayout: flat hierarchy, single measure/layout pass

// 3. Use Compose — no XML inflation at all
// Compose skips the entire XML parsing + View instantiation pipeline.
// First composition is typically faster than XML inflation for complex layouts.

// 4. AsyncLayoutInflater for non-critical UI
val asyncInflater = AsyncLayoutInflater(this)
asyncInflater.inflate(R.layout.heavy_layout, parentView) { view, _, parent ->
    parent?.addView(view)
}

Dagger/Hilt: Lazy Injection

// Dagger creates ALL injected dependencies eagerly by default.
// For startup-critical paths, use Provider<T> or Lazy<T> to defer creation.

@HiltViewModel
class MainViewModel @Inject constructor(
    private val userRepo: UserRepository,             // Created immediately
    private val analytics: Lazy<AnalyticsTracker>,    // Created on first .get()
    private val adManager: Provider<AdManager>        // Created on each .get()
) : ViewModel() {

    fun onScreenVisible() {
        // analytics is only created NOW, not when the ViewModel was constructed
        analytics.get().trackScreenView("main")
    }

    fun loadAd() {
        // adManager creates a new instance each time (Provider)
        adManager.get().loadBanner()
    }
}

// Lazy<T> vs Provider<T>:
// Lazy<T>: creates the instance on first .get(), caches it for subsequent calls
// Provider<T>: creates a new instance on every .get() call
//
// For startup optimization, Lazy<T> is what you want — defer creation
// but still get a singleton-like behavior within the injection scope.

// Hilt entry point for manual lazy access:
@EntryPoint
@InstallIn(SingletonComponent::class)
interface LazyEntryPoint {
    fun analyticsTracker(): AnalyticsTracker
}

// Access lazily from anywhere:
val entryPoint = EntryPointAccessors.fromApplication(
    context, LazyEntryPoint::class.java
)
val analytics = entryPoint.analyticsTracker()

Avoid Disk I/O on Main Thread During Startup

// SharedPreferences.getXxx() can block the main thread on first access.
// The first call to getSharedPreferences() triggers a file read from disk.
// If the file is large or the disk is slow, this can take 10-50ms.

// BAD: blocking main thread during startup
class MyApplication : Application() {
    override fun onCreate() {
        super.onCreate()
        val prefs = getSharedPreferences("config", MODE_PRIVATE)
        val theme = prefs.getString("theme", "light")  // Blocks until file is read!
        applyTheme(theme)
    }
}

// GOOD: use apply() instead of commit(), and preload on background thread
class MyApplication : Application() {
    lateinit var prefs: SharedPreferences

    override fun attachBaseContext(base: Context) {
        super.attachBaseContext(base)
        // Trigger the file load early — it happens asynchronously
        // but the first getString() will still block if not yet loaded.
        // At least we start the I/O as early as possible.
        Thread { getSharedPreferences("config", MODE_PRIVATE) }.start()
    }

    override fun onCreate() {
        super.onCreate()
        prefs = getSharedPreferences("config", MODE_PRIVATE)
        // By now, the background thread likely finished loading the file
    }
}

// BEST: use DataStore (async by design, no main-thread blocking)
val Context.configStore by preferencesDataStore(name = "config")

class MainViewModel @Inject constructor(
    private val dataStore: DataStore<Preferences>
) : ViewModel() {
    val theme: StateFlow<String> = dataStore.data
        .map { prefs -> prefs[THEME_KEY] ?: "light" }
        .stateIn(viewModelScope, SharingStarted.Eagerly, "light")
}
Staff-Level Insight

ContentProvider.onCreate() is the hidden startup tax — at Google, we audit every ContentProvider and remove or defer any that aren't needed before first frame. A single library adding a ContentProvider that does disk I/O or network in its onCreate() can add 100ms+ to cold start, and developers never see it because it runs before their code. The fix is systematic: (1) inspect the merged manifest for all <provider> tags, (2) profile each one with Perfetto to measure its cost, (3) migrate to App Startup or remove via tools:node="remove", (4) add a Macrobenchmark startup test to CI that fails on regression. At Google scale, we track P50 and P90 cold start times per release and investigate any regression > 20ms. The startup budget is a shared resource — every library, every ContentProvider, every eager singleton competes for the same 500ms window.
