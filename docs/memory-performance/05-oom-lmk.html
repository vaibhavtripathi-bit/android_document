05 — OOM & Low Memory Killer

Linux OOM killer, Android's lmkd, oom_adj scores, process priority, onTrimMemory, and surviving low-memory conditions

1. Linux OOM Killer vs Android LMK

Standard Linux and Android take fundamentally different approaches to memory exhaustion. Linux treats process killing as a last resort; Android treats it as a normal, continuous optimization.

Linux OOM Killer — Last Resort

// The Linux kernel OOM killer activates when:
// 1. All physical RAM is exhausted
// 2. All swap space is exhausted
// 3. The kernel cannot reclaim any more pages
//
// At this point the kernel MUST kill something or the entire system hangs.
//
// Selection algorithm (simplified):
// For each process:
//   oom_score = (process_RSS / total_RAM) * 1000
//   oom_score += oom_score_adj  (userspace hint: -1000 to +1000)
//   // Higher score = more likely to be killed
//
// The process with the HIGHEST oom_score is killed via SIGKILL.
//
// Problems for mobile:
// - Too late: by the time OOM killer fires, the system is already thrashing
// - Too coarse: kills one process, but the system may need to free much more
// - No awareness of Android's component model (Activities, Services, etc.)
// - No concept of "foreground" vs "background" in the Android sense

Android's lmkd — Proactive Killing

// Android replaces the reactive Linux OOM killer with a proactive daemon: lmkd.
// lmkd kills processes BEFORE the system reaches critical memory pressure.
//
// Key differences:
// ┌────────────────────────┬──────────────────────┬─────────────────────────────┐
// │                        │ Linux OOM Killer      │ Android lmkd                │
// ├────────────────────────┼──────────────────────┼─────────────────────────────┤
// │ When it kills          │ Last resort (no RAM)  │ Proactively (thresholds)    │
// │ Selection criteria     │ oom_score (RSS-based)  │ oom_adj (component-aware)   │
// │ Who sets priority      │ Kernel heuristics     │ ActivityManagerService       │
// │ Awareness of UI state  │ None                  │ Full (fg/bg/cached/service) │
// │ Runs as                │ Kernel code           │ Userspace daemon             │
// │ Trigger mechanism      │ Page allocation fail  │ Memory pressure thresholds  │
// └────────────────────────┴──────────────────────┴─────────────────────────────┘

// lmkd evolution:
// Android 1-8: kernel lowmemorykiller module (drivers/staging/android/)
//   - Kernel module that hooks into the page allocator
//   - Kills processes when free pages drop below configured thresholds
//   - Simple but effective; ran entirely in kernel space
//   - Deprecated because kernel staging drivers are hard to maintain
//
// Android 9+: lmkd userspace daemon (/system/bin/lmkd)
//   - Moved to userspace for better maintainability
//   - Communicates with AMS via a Unix domain socket
//   - Polls /proc/meminfo or uses vmpressure events
//   - More sophisticated kill strategies
//
// Android 10+: lmkd with PSI (Pressure Stall Information)
//   - Uses kernel PSI metrics instead of free-memory thresholds
//   - PSI measures actual stall time caused by memory pressure
//   - More accurate: "how much is memory pressure hurting performance?"
//   - vs old approach: "how many free pages remain?" (misleading with caches)

PSI — Pressure Stall Information

// PSI provides three metrics in /proc/pressure/memory:
//
// some avg10=0.50 avg60=1.20 avg300=0.80 total=12345678
// full avg10=0.00 avg60=0.10 avg300=0.05 total=1234567
//
// "some": percentage of time at least SOME tasks are stalled on memory
// "full": percentage of time ALL tasks are stalled (system completely blocked)
//
// avg10/avg60/avg300: moving averages over 10s, 60s, 300s windows
//
// lmkd uses these to make kill decisions:
// - Low pressure (some avg10 > threshold): kill cached processes
// - Medium pressure (some avg60 > threshold): kill heavier cached processes
// - Critical pressure (full avg10 > threshold): kill service processes
//
// Why PSI is better than free-memory thresholds:
// Free memory can be low but the system is fine (pages are reclaimable cache).
// Free memory can be "adequate" but the system is thrashing (fragmentation).
// PSI directly measures the IMPACT of memory pressure on task execution.

2. oom_adj Scores — The Priority System

Every Android process has an oom_adj score that determines its kill priority. Lower scores = higher importance = killed last. ActivityManagerService continuously recomputes these scores as the app's state changes.

How AMS Computes oom_adj — updateOomAdjLocked()

// AMS recomputes oom_adj for ALL processes whenever significant state changes:
// - Activity lifecycle transitions (resume, pause, stop, destroy)
// - Service start/stop/bind/unbind
// - Broadcast receiver registration
// - Content provider acquisition/release
// - Process importance changes from client bindings

// Source: frameworks/base/services/core/java/com/android/server/am/OomAdjuster.java

// Simplified logic of updateOomAdjLocked():
fun computeOomAdj(app: ProcessRecord): Int {
    // 1. Start with the worst priority
    var adj = CACHED_APP_MAX_ADJ

    // 2. Check for foreground Activity
    if (app.hasActivityInState(RESUMED)) {
        adj = FOREGROUND_APP_ADJ  // 0
    }

    // 3. Check for visible Activity (not resumed but visible)
    else if (app.hasActivityInState(PAUSING, PAUSED) && app.isVisible) {
        adj = VISIBLE_APP_ADJ  // 100
    }

    // 4. Check for foreground service
    else if (app.hasForegroundService()) {
        adj = PERCEPTIBLE_APP_ADJ  // 200
    }

    // 5. Check for active instrumentation (testing)
    else if (app.hasActiveInstrumentation()) {
        adj = FOREGROUND_APP_ADJ  // 0
    }

    // 6. Check for active broadcast receiver
    else if (app.isReceivingBroadcast()) {
        adj = FOREGROUND_APP_ADJ  // 0 (temporarily, while receiving)
    }

    // 7. Check for running service
    else if (app.hasStartedService()) {
        adj = SERVICE_ADJ  // 500
        // Demote to SERVICE_B_ADJ (800) if running > 30 min
        if (app.serviceStartTime + 30.minutes < now) {
            adj = SERVICE_B_ADJ
        }
    }

    // 8. Check bindings — a bound service inherits client's priority
    for (connection in app.serviceConnections) {
        val clientAdj = connection.client.currentAdj
        if (connection.hasFlag(BIND_ABOVE_CLIENT)) {
            adj = minOf(adj, clientAdj)  // At least as important as client
        } else if (connection.hasFlag(BIND_IMPORTANT)) {
            adj = minOf(adj, clientAdj + 1)  // Slightly below client
        } else if (connection.hasFlag(BIND_NOT_FOREGROUND)) {
            // Don't promote above PERCEPTIBLE
        } else {
            adj = minOf(adj, maxOf(clientAdj, PERCEPTIBLE_APP_ADJ))
        }
    }

    // 9. Check content provider clients
    for (provider in app.publishedProviders) {
        for (client in provider.connections) {
            adj = minOf(adj, client.currentAdj + 1)
        }
    }

    // 10. Apply to the process
    return adj
}

// After computing, AMS writes the value:
// ProcessList.setOomAdj(pid, uid, adj)
//   → writes to /proc/{pid}/oom_score_adj
//   → also notifies lmkd via socket: "pid X has adj Y"
Binding Flags Matter Enormously

The flags you pass to bindService() directly control whether the service's process gets promoted. BIND_ABOVE_CLIENT makes the service at least as important as the binding Activity. BIND_NOT_FOREGROUND prevents the service from being promoted above PERCEPTIBLE_APP_ADJ. BIND_WAIVE_PRIORITY means the binding doesn't affect the service's priority at all. Choosing the wrong flag can either cause your service to be killed too easily or prevent the system from reclaiming memory from processes that should be killable.

3. Process Priority — What Affects It

The Five Process Buckets

// Android documentation describes five process importance levels.
// Internally these map to ranges of oom_adj values.

// 1. FOREGROUND PROCESS (adj = 0)
//    - Has a resumed Activity (user is interacting with it)
//    - Is executing a BroadcastReceiver.onReceive()
//    - Is executing a Service lifecycle callback (onCreate, onStart, onBind)
//    - Has active instrumentation
//    Kill policy: NEVER killed unless absolutely necessary (system is dying)

// 2. VISIBLE PROCESS (adj = 100)
//    - Has a visible but not resumed Activity (behind a transparent Activity/dialog)
//    - Has a foreground Service (startForeground() called with notification)
//    Kill policy: Only killed if necessary to keep foreground process alive

// 3. SERVICE PROCESS (adj = 500)
//    - Running a started Service (startService()) that is not foreground
//    - Doing background work the user doesn't directly see
//    Kill policy: Killed when memory is needed; demoted after 30 min
//    Note: Android 8+ restricts background services heavily

// 4. CACHED (BACKGROUND) PROCESS (adj = 900-999)
//    - Not currently needed — no active components
//    - Kept alive for faster re-launch (warm start)
//    - Organized in an LRU list — most recently used = lower adj
//    Kill policy: Freely killed, starting from highest adj (oldest)
//    System maintains ~16-32 cached processes (device-dependent)

// 5. EMPTY PROCESS (adj = 999)
//    - No active components at all
//    - Exists only as a process cache
//    Kill policy: First to be killed

How Foreground Services Change oom_adj

// Starting a foreground service dramatically changes process priority:

class UploadService : Service() {
    override fun onStartCommand(intent: Intent?, flags: Int, startId: Int): Int {
        // Before startForeground(): process adj = SERVICE_ADJ (500)
        // or CACHED_APP_MAX_ADJ (999) if no other components

        val notification = NotificationCompat.Builder(this, CHANNEL_ID)
            .setContentTitle("Uploading...")
            .setSmallIcon(R.drawable.ic_upload)
            .build()

        startForeground(NOTIFICATION_ID, notification)
        // After startForeground(): process adj = PERCEPTIBLE_APP_ADJ (200)
        // This is a HUGE jump — from "freely killable" to "user-perceptible"

        return START_STICKY
    }
}

// Timeline of oom_adj changes:
// 1. App in foreground:           adj = 0   (FOREGROUND_APP_ADJ)
// 2. User presses Home:           adj = 700 (PREVIOUS_APP_ADJ)
// 3. Another app opens:           adj = 900 (CACHED_APP_MIN_ADJ)
// 4. Service.startForeground():   adj = 200 (PERCEPTIBLE_APP_ADJ)
// 5. Service.stopForeground():    adj = 900 (back to CACHED)
// 6. Time passes, more apps open: adj = 950 → 999 (slides toward MAX)

How Binding Elevates a Service Process

// When an Activity binds to a Service in another process,
// the Service's process priority is elevated based on the binding flags.

// Scenario: Activity (adj=0) binds to a Service (adj=900)

// Default binding (no special flags):
bindService(intent, connection, Context.BIND_AUTO_CREATE)
// Service adj becomes: max(clientAdj, PERCEPTIBLE_APP_ADJ) = max(0, 200) = 200

// BIND_ABOVE_CLIENT: service is at least as important as the client
bindService(intent, connection, BIND_AUTO_CREATE or BIND_ABOVE_CLIENT)
// Service adj becomes: clientAdj = 0 (same as the Activity!)

// BIND_IMPORTANT: slightly below client
bindService(intent, connection, BIND_AUTO_CREATE or BIND_IMPORTANT)
// Service adj becomes: clientAdj + 1 = 1

// BIND_NOT_FOREGROUND: don't promote above perceptible
bindService(intent, connection, BIND_AUTO_CREATE or BIND_NOT_FOREGROUND)
// Service adj becomes: max(clientAdj, PERCEPTIBLE_APP_ADJ) = 200

// BIND_WAIVE_PRIORITY: binding doesn't affect priority at all
bindService(intent, connection, BIND_AUTO_CREATE or BIND_WAIVE_PRIORITY)
// Service adj stays at 900 — binding is "invisible" to OOM adjuster

// When the Activity unbinds or is destroyed:
// Service adj reverts to its own intrinsic priority (SERVICE_ADJ or CACHED)

Cached Processes and the LRU List

// Cached processes are organized in an LRU (Least Recently Used) list.
// AMS maintains this list in ProcessList.mLruProcesses.

// When a process transitions to cached:
// - It's placed at the "recent" end of the LRU list (adj = 900)
// - As more processes become cached, older ones slide toward adj = 999
// - lmkd kills from the 999 end first

// The system maintains a target number of cached processes:
// CachedAppOptimizer in AMS manages this:
//   - Typical target: 32 cached processes (varies by device RAM)
//   - When count exceeds target: oldest cached processes are killed
//   - On low-RAM devices (< 2GB): target may be as low as 8-16

// Cached process compaction (Android 10+):
// Instead of killing, AMS first tries to COMPACT cached processes:
// - Sends SIGSTOP to freeze the process
// - Calls process_madvise(MADV_COLD) to hint pages as reclaimable
// - Kernel can reclaim those pages without killing the process
// - If the process is resumed, pages are faulted back in (slower but alive)

// Freezer (Android 11+):
// Cached processes can be FROZEN using cgroup v2 freezer:
// - Process is completely stopped (no CPU time)
// - Memory pages are marked as candidates for swap/zRAM
// - Unfreezing is faster than cold-starting the process
// echo 1 > /sys/fs/cgroup/uid_{uid}/pid_{pid}/cgroup.freeze

4. onTrimMemory Callbacks

Android notifies apps about memory pressure via the ComponentCallbacks2 interface. The system sends increasingly urgent levels, giving apps a chance to release resources before being killed.

// ComponentCallbacks2 is implemented by Application, Activity, Service, etc.
// Override onTrimMemory(level: Int) to respond to memory pressure.

class MyApplication : Application(), ComponentCallbacks2 {

    private val imageCache = LruCache<String, Bitmap>(50 * 1024 * 1024) // 50MB
    private val dataCache = mutableMapOf<String, Any>()

    override fun onTrimMemory(level: Int) {
        super.onTrimMemory(level)

        when (level) {
            // ── App is RUNNING (foreground) ──────────────────────

            TRIM_MEMORY_RUNNING_MODERATE -> {  // level = 5
                // System is getting low on memory. Your app is running
                // and not killable, but consider releasing non-critical resources.
                // Other background apps are being killed.
                dataCache.entries
                    .filter { it.value is SoftReference<*> }
                    .forEach { dataCache.remove(it.key) }
            }

            TRIM_MEMORY_RUNNING_LOW -> {  // level = 10
                // System is considerably low on memory while your app is running.
                // Release unnecessary resources NOW to avoid performance degradation.
                imageCache.trimToSize(imageCache.maxSize() / 2)
                dataCache.clear()
            }

            TRIM_MEMORY_RUNNING_CRITICAL -> {  // level = 15
                // System is critically low on memory. Your app is not yet killable
                // but the system WILL kill background processes and may kill yours
                // if you don't free memory immediately.
                imageCache.evictAll()
                dataCache.clear()
                // Release any recreatable resources: decoded bitmaps,
                // layout caches, view pools, typeface caches
            }

            // ── App's UI is HIDDEN ──────────────────────────────

            TRIM_MEMORY_UI_HIDDEN -> {  // level = 20
                // App moved to background — UI is no longer visible.
                // This is NOT a memory pressure signal — it's a lifecycle hint.
                // Release UI-specific resources: image caches, view references,
                // large layout hierarchies that can be recreated.
                imageCache.evictAll()
                // This is the most important callback for most apps.
                // Glide/Coil listen for this to clear their memory caches.
            }

            // ── App is CACHED (background, killable) ────────────

            TRIM_MEMORY_BACKGROUND -> {  // level = 40
                // App is in the cached LRU list near the beginning (recently used).
                // Free resources that can be efficiently rebuilt if the app returns.
                imageCache.evictAll()
                dataCache.clear()
            }

            TRIM_MEMORY_MODERATE -> {  // level = 60
                // App is in the middle of the cached LRU list.
                // More aggressive cleanup — you're getting closer to being killed.
                imageCache.evictAll()
                dataCache.clear()
                // Release database connections, close cursors,
                // trim thread pools, release WebView if possible
            }

            TRIM_MEMORY_COMPLETE -> {  // level = 80
                // App is at the END of the cached LRU list — next to be killed.
                // Release EVERYTHING that isn't absolutely needed to restart.
                imageCache.evictAll()
                dataCache.clear()
                // Last chance: release all caches, pools, buffers.
                // If you don't free enough, you'll be killed.
            }
        }
    }
}

UI_HIDDEN Is Not Memory Pressure

TRIM_MEMORY_UI_HIDDEN (level 20) is sent every time your app goes to background, regardless of memory conditions. It's a lifecycle signal, not a pressure signal. The RUNNING_* levels (5, 10, 15) are sent while your app is in the foreground and indicate actual system-wide memory pressure. The BACKGROUND/MODERATE/COMPLETE levels (40, 60, 80) are sent to cached apps and indicate your position in the kill queue. Many developers confuse these categories.

How Image Libraries Handle onTrimMemory

// Glide's implementation (simplified):
// Glide registers a ComponentCallbacks2 on the Application:

class GlideMemoryTrimmer : ComponentCallbacks2 {
    override fun onTrimMemory(level: Int) {
        when {
            level >= TRIM_MEMORY_MODERATE -> {
                // Clear both memory cache and bitmap pool entirely
                memoryCache.clearMemory()
                bitmapPool.clearMemory()
            }
            level >= TRIM_MEMORY_BACKGROUND || level == TRIM_MEMORY_RUNNING_CRITICAL -> {
                // Trim to 50% of max size
                memoryCache.trimMemory(level)
                bitmapPool.trimMemory(level)
            }
            level == TRIM_MEMORY_UI_HIDDEN -> {
                // App went to background — clear memory cache
                // but keep bitmap pool (bitmaps are expensive to allocate)
                memoryCache.clearMemory()
            }
        }
    }
}

5. OutOfMemoryError — Java vs Native

Java Heap OOM

// Java OOM occurs when the Dalvik/ART heap cannot grow further.
// Each app has a per-app heap limit set by the device manufacturer.

// Check your heap limits:
val runtime = Runtime.getRuntime()
val maxHeap = runtime.maxMemory()       // Hard limit (e.g., 256MB, 512MB)
val totalHeap = runtime.totalMemory()   // Currently allocated heap size
val freeHeap = runtime.freeMemory()     // Free within current allocation
val usedHeap = totalHeap - freeHeap     // Actually used

// Typical limits (varies by device):
// Low-RAM devices:  128MB - 192MB
// Mid-range:        256MB - 384MB
// Flagships:        512MB - 1GB
// largeHeap=true:   2x-4x the normal limit (NOT recommended)

// The limit is set in /system/build.prop:
// dalvik.vm.heapsize=512m          (max heap per app)
// dalvik.vm.heapgrowthlimit=256m   (default limit)
// dalvik.vm.heapstartsize=8m       (initial heap)
// dalvik.vm.heapmaxfree=8m         (max free before shrinking)
// dalvik.vm.heapminfree=512k       (min free after GC)
// dalvik.vm.heaptargetutilization=0.75  (target 75% utilization)

// When usedHeap approaches maxMemory():
// 1. ART triggers GC (concurrent, then stop-the-world if needed)
// 2. If GC can't free enough → ART tries to grow the heap
// 3. If heap is at maxMemory() → java.lang.OutOfMemoryError

// Common causes of Java heap OOM:
// - Holding too many Bitmaps in memory (pre-Android 8)
// - Memory leaks (Activity references held by static fields, singletons)
// - Unbounded caches (HashMap growing without eviction)
// - Large JSON/XML parsing (loading entire response into memory)
// - Too many threads (each thread stack = 1-8MB)

Native OOM

// Native OOM is fundamentally different from Java OOM.
// Native memory (malloc/mmap) has NO per-app heap limit.
// It's bounded only by:
// 1. Device physical RAM + swap/zRAM
// 2. Virtual address space (practically unlimited on 64-bit)
// 3. Per-process limits (RLIMIT_AS, rarely set on Android)

// Native malloc failure:
// - malloc() returns NULL (C/C++)
// - If not checked → SIGSEGV (crash) when dereferencing NULL
// - Unlike Java, there's no exception — just a NULL pointer

// Major native memory consumers in Android apps:
// 1. Bitmaps (Android 8+): pixel data stored in native heap
//    - A 4K photo: 4032 × 3024 × 4 bytes = ~46MB native memory
//    - 10 such photos = 460MB — no Java OOM, but device runs out of RAM

// 2. WebView / Chromium renderer:
//    - Each WebView tab: 50-200MB native memory
//    - DOM, render tree, composited layers, V8 JS heap
//    - Completely invisible to Java heap monitoring

// 3. SQLite:
//    - Page cache, WAL file mapping, prepared statements
//    - Typically 2-20MB but can grow with large transactions

// 4. Graphics / GPU buffers:
//    - EGL surfaces, texture uploads, shader programs
//    - Managed by the GPU driver, not the Java heap

// 5. RenderScript / Vulkan compute buffers

// Monitoring native memory:
// Debug.getNativeHeapSize()         // Total native heap
// Debug.getNativeHeapAllocatedSize() // Used native heap
// Debug.getNativeHeapFreeSize()     // Free native heap
// Or: adb shell dumpsys meminfo <package>

Bitmap OOM — The Great Migration

// Pre-Android 8 (API < 26):
// Bitmap pixel data stored on the JAVA heap.
// - Counted against Runtime.maxMemory()
// - Easy to trigger OOM with a few large images
// - Bitmap.recycle() was critical to free memory promptly
// - Bitmap pixel data backed by byte[] on the Java heap

// Android 8+ (API 26+):
// Bitmap pixel data stored on the NATIVE heap.
// - NOT counted against Runtime.maxMemory()
// - Java heap only holds the Bitmap object (~64 bytes)
// - Pixel data allocated via native malloc (hardware bitmaps use GPU memory)
// - Bitmap.recycle() still frees native memory but is less critical
//   (GC + native finalizer will eventually free it)

// The catch: native memory is invisible to most monitoring tools.
// An app can use 500MB of Bitmap native memory and show only 30MB Java heap.
// This is why apps "mysteriously" get killed without hitting Java OOM.

// Hardware Bitmaps (Android 8+):
// Bitmap.Config.HARDWARE — pixel data stored in GPU memory
// - Cannot be read/modified on CPU (getPixels() throws)
// - Most memory-efficient for display-only bitmaps
// - Memory is managed by the graphics driver
// - Glide uses HARDWARE config by default

// Monitoring bitmap memory:
val bitmap = BitmapFactory.decodeResource(resources, R.drawable.large_image)
val byteCount = bitmap.byteCount        // Pixel data size
val allocationByteCount = bitmap.allocationByteCount  // Allocated buffer size
// On Android 8+: this memory is in native heap, not Java heap

Handling OOM Gracefully

// Strategy 1: Catch OOM for non-critical allocations
fun loadBitmapSafely(path: String, reqWidth: Int, reqHeight: Int): Bitmap? {
    return try {
        val options = BitmapFactory.Options().apply {
            inJustDecodeBounds = true
        }
        BitmapFactory.decodeFile(path, options)

        options.inSampleSize = calculateInSampleSize(options, reqWidth, reqHeight)
        options.inJustDecodeBounds = false
        BitmapFactory.decodeFile(path, options)
    } catch (e: OutOfMemoryError) {
        // Log, report to analytics, return placeholder
        System.gc()  // Hint to GC (not guaranteed)
        null
    }
}

// Strategy 2: Use inSampleSize to downsample large images
fun calculateInSampleSize(
    options: BitmapFactory.Options,
    reqWidth: Int,
    reqHeight: Int
): Int {
    val (height, width) = options.outHeight to options.outWidth
    var inSampleSize = 1
    if (height > reqHeight || width > reqWidth) {
        val halfHeight = height / 2
        val halfWidth = width / 2
        while (halfHeight / inSampleSize >= reqHeight &&
               halfWidth / inSampleSize >= reqWidth) {
            inSampleSize *= 2
        }
    }
    return inSampleSize
}

// Strategy 3: Monitor memory proactively
class MemoryMonitor(private val context: Context) {
    fun isMemoryLow(): Boolean {
        val activityManager = context.getSystemService<ActivityManager>()!!
        val memInfo = ActivityManager.MemoryInfo()
        activityManager.getMemoryInfo(memInfo)
        return memInfo.lowMemory  // true when available RAM < threshold
    }

    fun getAvailableMemoryMB(): Long {
        val memInfo = ActivityManager.MemoryInfo()
        context.getSystemService<ActivityManager>()!!.getMemoryInfo(memInfo)
        return memInfo.availMem / (1024 * 1024)
    }

    fun getHeapUsagePercent(): Float {
        val runtime = Runtime.getRuntime()
        val used = runtime.totalMemory() - runtime.freeMemory()
        return used.toFloat() / runtime.maxMemory()
    }
}

lmkd Communication with AMS

// AMS and lmkd communicate via a Unix domain socket: /dev/socket/lmkd

// AMS → lmkd messages:
// LMK_TARGET: set memory thresholds for each adj level
//   Format: LMK_TARGET <minfree_1> <adj_1> <minfree_2> <adj_2> ...
//   Example: LMK_TARGET 18432 0 23040 100 27648 200 32256 300 55296 900 80640 999
//   (minfree values in pages: 18432 pages × 4KB = 72MB for adj 0)

// LMK_PROCPRIO: update a process's oom_adj
//   Format: LMK_PROCPRIO <pid> <uid> <adj>
//   Sent every time AMS recomputes oom_adj for a process

// LMK_PROCREMOVE: process has died, remove from lmkd's tracking
//   Format: LMK_PROCREMOVE <pid>

// lmkd → AMS messages:
// LMK_PROCKILL: lmkd killed a process
//   Format: LMK_PROCKILL <pid> <uid>
//   AMS uses this to clean up the ProcessRecord

// lmkd configuration (system properties):
// ro.lmk.low=1001           (adj threshold for low pressure kills)
// ro.lmk.medium=800         (adj threshold for medium pressure kills)
// ro.lmk.critical=0         (adj threshold for critical pressure kills)
// ro.lmk.use_psi=true       (use PSI instead of vmpressure)
// ro.lmk.psi_partial_stall_ms=70  (PSI threshold for partial stall)
// ro.lmk.psi_complete_stall_ms=700 (PSI threshold for complete stall)

Architecture-Level Strategies

// 1. Saved instance state — survive process death
class DetailActivity : AppCompatActivity() {
    private var scrollPosition = 0
    private var formData: FormState? = null

    override fun onSaveInstanceState(outState: Bundle) {
        super.onSaveInstanceState(outState)
        outState.putInt("scroll_position", scrollPosition)
        outState.putParcelable("form_data", formData)
    }

    override fun onCreate(savedInstanceState: Bundle?) {
        super.onCreate(savedInstanceState)
        savedInstanceState?.let {
            scrollPosition = it.getInt("scroll_position")
            formData = it.getParcelable("form_data")
        }
    }
}

// 2. ViewModel + SavedStateHandle — survive both config change AND process death
class DetailViewModel(private val savedState: SavedStateHandle) : ViewModel() {
    // Survives config change (retained in memory)
    private val _uiState = MutableStateFlow(DetailUiState())
    val uiState: StateFlow<DetailUiState> = _uiState.asStateFlow()

    // Survives process death (persisted to Bundle)
    var searchQuery: String
        get() = savedState["query"] ?: ""
        set(value) { savedState["query"] = value }

    // getStateFlow() for reactive saved state
    val savedQuery: StateFlow<String> = savedState.getStateFlow("query", "")
}

// 3. Proactive memory management in custom views
class ThumbnailGridView(context: Context) : RecyclerView(context) {
    override fun onDetachedFromWindow() {
        super.onDetachedFromWindow()
        // Release adapter references to prevent holding bitmaps
        adapter = null
    }
}

// 4. WorkManager for critical background work
// If your process is killed during background work, WorkManager

val uploadWork = OneTimeWorkRequestBuilder<UploadWorker>()

WorkManager.getInstance(context).enqueue(uploadWork)

Staff-Level Insight — The Real OOM Killer at Scale

The real OOM killer at Google scale is not the Java heap — it's native memory from Bitmaps, WebView, and SQLite that grows invisibly. A production app might show 80MB Java heap usage but consume 400MB total (PSS) because of native allocations. Runtime.maxMemory() only guards the Java heap; native memory is bounded only by device RAM. This is why apps get killed by lmkd without ever throwing OutOfMemoryError. At L6, you must monitor total PSS (via Debug.getMemoryInfo() or dumpsys meminfo), not just Java heap. The biggest offenders: (1) Bitmap pixel buffers on Android 8+ (native heap), (2) WebView renderer processes (50-200MB each), (3) SQLite page caches and memory-mapped WAL files, (4) Protobuf/FlatBuffer deserialization buffers, (5) JNI global references holding native objects. Google's internal memory tooling tracks RSS/PSS per-process with per-allocation-site attribution — something you can approximate with malloc_debug or Android Studio's native memory profiler.
