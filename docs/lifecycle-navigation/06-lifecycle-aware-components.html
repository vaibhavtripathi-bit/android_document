06 — Lifecycle-Aware Components

LifecycleRegistry internals, ViewModel survival mechanics, SavedStateHandle plumbing, LiveData dispatch, ProcessLifecycleOwner, and Compose integration

1. Lifecycle & LifecycleOwner Internals

1.1 The State Machine

The Lifecycle abstract class models a finite state machine. States are nodes; events are edges:

        Events (edges)                    States (nodes)
  ┌─────────────────────────────────────────────────────────────┐
  │                                                             │
  │   ON_CREATE        ON_START        ON_RESUME                │
  │  ──────────►      ──────────►     ──────────►               │
  │  INITIALIZED ──► CREATED ──────► STARTED ──────► RESUMED    │
  │              ◄──         ◄──────          ◄──────            │
  │   ON_DESTROY       ON_STOP        ON_PAUSE                  │
  │                                                             │
  │  DESTROYED ◄── (from CREATED via ON_DESTROY)                │
  │                                                             │
  └─────────────────────────────────────────────────────────────┘

  Key insight: Events move between ADJACENT states only.
  ON_CREATE moves INITIALIZED → CREATED
  ON_DESTROY moves CREATED → DESTROYED
  There is no direct jump from RESUMED → DESTROYED.

The State enum is comparable — isAtLeast(State) is just an ordinal comparison. This is what makes lifecycle-aware collection work: you check isAtLeast(STARTED) to know if the UI is visible.

1.2 LifecycleRegistry — The Default Implementation

LifecycleRegistry is the concrete implementation used by ComponentActivity, Fragment, and NavBackStackEntry. Its internals are critical to understand:

// Simplified LifecycleRegistry internals
class LifecycleRegistry(private val provider: LifecycleOwner) : Lifecycle() {

    // Observers stored in insertion-order map that is safe to iterate
    // while being modified — crucial for re-entrant additions
    private var observerMap = FastSafeIterableMap<LifecycleObserver, ObserverWithState>()

    private var state: State = State.INITIALIZED

    fun handleLifecycleEvent(event: Event) {
        enforceMainThreadIfNeeded("handleLifecycleEvent")
        moveToState(event.targetState)
    }

    private fun moveToState(next: State) {
        if (state == next) return
        state = next
        sync()  // ← the core dispatch loop
    }

    private fun sync() {
        while (!isSynced) {
            // If any observer is AHEAD of current state → dispatch downward events
            if (state < observerMap.eldest().value.state) {
                backwardPass(observerMap)
            }
            // If any observer is BEHIND current state → dispatch upward events
            val newest = observerMap.newest()
            if (!newEventOccurred && state > newest.value.state) {
                forwardPass(observerMap)
            }
        }
    }
}

FastSafeIterableMap — a linked-list-backed map that allows modification during iteration. This is not a ConcurrentHashMap; it's single-threaded but re-entrant-safe. When you add an observer inside an ON_CREATE callback, the map handles it without ConcurrentModificationException.

1.3 The Sync Mechanism

The sync() loop guarantees that all observers reach the same state before returning. If the lifecycle owner is in STARTED and an observer is still at INITIALIZED, the forward pass dispatches ON_CREATE then ON_START to that observer before moving on.

The isSynced check: the eldest observer's state equals the newest observer's state equals the registry's current state. Only then does the loop exit.

1.4 Re-entrant Observer Addition

If observer A's ON_CREATE callback adds observer B, the registry handles this by:

  1. B is added with state INITIALIZED
  2. The newEventOccurred flag is set, breaking the current forward pass
  3. sync() re-enters its while loop, now seeing B needs catch-up
  4. B receives ON_CREATE (and ON_START if the owner is already STARTED)

1.5 How ComponentActivity Reports Events

ComponentActivity uses two mechanisms depending on API level:

API LevelMechanismHow It Works
< 29ReportFragmentA headless fragment injected into the Activity. Its onStart()/onStop()/etc. dispatch events to the registry. This avoids requiring subclasses to call super in the right order.
≥ 29Application.ActivityLifecycleCallbacksRegistered in ReportFragment.injectIfNeededIn(). Uses registerActivityLifecycleCallbacks() on the Activity itself — no fragment needed.
Staff-Level Insight

The ReportFragment trick predates the Lifecycle library — it was first used by ActivityCompat for permission callbacks. The pattern of "inject a headless fragment to intercept lifecycle" is a recurring Android architecture hack. On API 29+, the framework added Activity.registerActivityLifecycleCallbacks() (per-Activity, not just per-Application), making the fragment unnecessary. The library still adds ReportFragment on API 29+ but its lifecycle methods are no-ops — only the callback path fires.

2.4 Observer Dispatch Ordering

This is a subtle but important detail:

DirectionEventsDispatch Order
Upward (gaining state)ON_CREATE, ON_START, ON_RESUMEFIFO — first added, first notified
Downward (losing state)ON_PAUSE, ON_STOP, ON_DESTROYLIFO — last added, first notified

This mirrors constructor/destructor ordering — components initialized first are torn down last. The forwardPass() iterates eldest→newest; backwardPass() iterates newest→eldest.

3. ProcessLifecycleOwner

3.1 App-Level Lifecycle

ProcessLifecycleOwner provides a single Lifecycle for the entire application process. It reaches STARTED when the first Activity starts and drops to CREATED when the last Activity stops.

  Process Lifecycle State Machine
  ────────────────────────────────────────────────
  App launch
    │
    ▼
  CREATED ──── first Activity.onStart() ────► STARTED
    ▲                                            │
    │                                            ▼
    │                                         RESUMED
    │                                            │
    │         last Activity.onPause()            │
    │         ◄──────────────────────────────────┘
    │                    │
    │                    ▼
    │                 STARTED
    │                    │
    │         last Activity.onStop()
    │         + 700ms delay (no new Activity)
    │         ◄──────────────────────────────────┘
    │
  CREATED  (app is "backgrounded")

  ✗ Never reaches DESTROYED — the process may be killed at any time

3.2 The 700ms Delay

When the last Activity stops, ProcessLifecycleOwner waits 700 milliseconds before dispatching ON_STOP. Why?

During a configuration change, the old Activity is destroyed and a new one is created. There's a brief window where zero Activities are in STARTED state. Without the delay, the process lifecycle would incorrectly report the app as backgrounded during every rotation.

// From ProcessLifecycleOwner source
companion object {
    internal const val TIMEOUT_MS = 700L
}

private val delayedPauseRunnable = Runnable {
    dispatchPauseIfNeeded()
    dispatchStopIfNeeded()
}

// Posted with delay; cancelled if a new Activity starts within 700ms
handler.postDelayed(delayedPauseRunnable, TIMEOUT_MS)

3.3 Detection Mechanism

ProcessLifecycleOwner registers an Application.ActivityLifecycleCallbacks via ProcessLifecycleInitializer (an androidx.startup.Initializer). It maintains counters:

  • startedCounter — incremented on each Activity's onStart(), decremented on onStop()
  • resumedCounter — same pattern for onResume()/onPause()
  • When a counter transitions from 0→1, dispatch the corresponding upward event
  • When a counter transitions from 1→0, schedule the delayed downward event

3.4 Use Cases and Limitations

Use CaseExample
AnalyticsTrack app foreground/background sessions
Connection managementOpen/close WebSocket when app is visible
Rate limitingPause background sync when app is foregrounded
Warning

ProcessLifecycleOwner does not track individual Activities, does not dispatch ON_DESTROY, and cannot distinguish between "user left the app" and "system killed the process." For process death, use ApplicationExitInfo.

4. ViewModel Deep Dive

4.1 ViewModelStore and ViewModelStoreOwner

A ViewModelStore is a HashMap<String, ViewModel>. The key is typically the ViewModel's canonical class name (or a custom key). ViewModelStoreOwner is implemented by ComponentActivity, Fragment, and NavBackStackEntry.

4.2 How ViewModel Survives Configuration Change

  Config Change: Activity A₁ destroyed, Activity A₂ created
  ──────────────────────────────────────────────────────────

  Activity A₁                          Activity A₂
  ┌──────────────┐                     ┌──────────────┐
  │ ViewModelStore│─── retained via ──►│ ViewModelStore│ (same instance!)
  └──────┬───────┘   NonConfig-       └──────┬───────┘
         │           Instances                │
    ┌────┴────┐                          ┌────┴────┐
    │ViewModel│ ◄── same object ──────► │ViewModel│
    └─────────┘                          └─────────┘

  Mechanism (ComponentActivity):
  1. onRetainNonConfigurationInstance() saves ViewModelStore
  2. Framework stores it in ActivityClientRecord.lastNonConfigurationInstances
  3. New Activity retrieves it via getLastNonConfigurationInstance()
  4. ViewModelStore (and all ViewModels) survive untouched

The NonConfigurationInstances is a framework-level object stored in ActivityThread.ActivityClientRecord. It's not serialized — it's an in-memory reference, which is why ViewModels can hold arbitrary objects (unlike onSaveInstanceState).

4.6 onCleared() and viewModelScope

abstract class ViewModel {
    // Called when the ViewModel is no longer used and will be destroyed.
    // NOT called on config change — only when the owner is permanently finished.
    protected open fun onCleared() {}

    // Auto-cancelled scope — uses Dispatchers.Main.immediate + SupervisorJob
    val viewModelScope: CoroutineScope
        // Stored as a Closeable in the ViewModel's internal closeables bag
        // When onCleared() runs, all Closeables are closed → scope is cancelled
}

// Cancellation order:
// 1. ViewModel.onCleared() is called
// 2. All registered Closeable/AutoCloseable instances are closed
// 3. viewModelScope.coroutineContext[Job]!!.cancel() fires
// 4. Child coroutines receive CancellationException
Staff-Level Insight

Since lifecycle 2.8+, ViewModel accepts Closeable and AutoCloseable resources via addCloseable(). The viewModelScope itself is stored as a Closeable. This means onCleared() fires before the scope is cancelled — if your onCleared() launches a coroutine on viewModelScope, it will be immediately cancelled. Use addCloseable() for cleanup that needs to outlive onCleared(), or use a NonCancellable context for critical cleanup work.

5. SavedStateHandle

5.1 Bridging ViewModel and Saved State

SavedStateHandle gives ViewModels access to the saved-state Bundle — the same mechanism that survives process death. It's a key-value map that auto-serializes to the Activity's Bundle.

  SavedStateHandle Data Flow
  ──────────────────────────────────────────────────────────

  ViewModel                    SavedStateHandle
  ┌──────────────┐            ┌──────────────────┐
  │ val query =  │◄──────────│ regular: Map<K,V> │
  │  handle["q"] │            │                  │
  │              │───────────►│ savedState-      │
  │ handle["q"]  │            │ Provider:        │──► Activity
  │   = "cats"   │            │ Bundle           │    .onSaveInstanceState()
  └──────────────┘            │                  │
                              │ liveDatas: Map   │
                              │ flows: Map       │
                              └──────────────────┘

  On process death:
  1. System calls Activity.onSaveInstanceState(outBundle)
  2. SavedStateRegistry iterates all registered providers
  3. SavedStateHandle serializes its map into the Bundle
  4. Bundle is written to the system_server process via Binder

  On restore:
  1. Activity.onCreate(savedInstanceState) receives the Bundle
  2. SavedStateRegistry restores providers
  3. SavedStateHandle is reconstructed from the Bundle
  4. ViewModel receives the restored handle via CreationExtras

5.2 SavedStateHandleSupport (Modern Integration)

Since lifecycle 2.5+, SavedStateHandle is created via CreationExtras rather than a special factory:

// Inside viewModelFactory { initializer { ... } }
val handle = createSavedStateHandle()
// This calls SavedStateHandleSupport.createSavedStateHandle(extras)
// which reads from:
//   extras[SAVED_STATE_REGISTRY_OWNER_KEY]  → SavedStateRegistryOwner
//   extras[VIEW_MODEL_STORE_OWNER_KEY]      → ViewModelStoreOwner
//   extras[DEFAULT_ARGS_KEY]                → Intent extras / Fragment args

5.3 Reading and Writing

class SearchViewModel(private val handle: SavedStateHandle) : ViewModel() {

    // Direct get/set
    val query: String? = handle.get<String>("query")
    fun setQuery(q: String) { handle["query"] = q }

    // StateFlow — survives process death, lifecycle-aware in Compose
    val queryFlow: StateFlow<String> = handle.getStateFlow("query", "")

    // LiveData — for View-system codebases
    val queryLiveData: MutableLiveData<String> = handle.getLiveData("query", "")
}

5.4 DEFAULT_ARGS_KEY

Intent extras (for Activities) and Fragment arguments are automatically available in SavedStateHandle:

// If Activity launched with: intent.putExtra("userId", 42L)
// Then in ViewModel:

// If Fragment created with: arguments = bundleOf("itemId" to "abc")
// Then in ViewModel:

6. LiveData Internals

6.1 Observer Pattern

LiveData wraps a value and dispatches changes only to STARTED+ observers. When an observer transitions from CREATED to STARTED, it receives the latest value if it hasn't seen it yet.

6.2 The mVersion Counter

// Simplified LiveData internals
abstract class LiveData<T> {
    private var mVersion = START_VERSION  // -1
    private var mData: T? = null

    // Each observer wrapper tracks its own lastVersion
    inner class ObserverWrapper {
        var mLastVersion = START_VERSION
    }

    private fun considerNotify(observer: ObserverWrapper) {
        if (!observer.mActive) return
        // Stale check: has observer already seen this version?
        if (observer.mLastVersion >= mVersion) return
        observer.mLastVersion = mVersion
        observer.mObserver.onChanged(mData as T)
    }
}

The version counter prevents stale delivery: if the observer already saw version N, it won't be notified again. But this creates the infamous "event" problem — on config change, the new observer starts with mLastVersion = -1, so it receives the current value even if it was set before the config change.

6.3 setValue() vs postValue()

// setValue — main thread only, synchronous dispatch
fun setValue(value: T) {
    assertMainThread("setValue")
    mVersion++
    mData = value
    dispatchingValue(null)  // notify all active observers immediately
}

// postValue — any thread, asynchronous
fun postValue(value: T) {
    val postTask: Boolean
    synchronized(mDataLock) {
        postTask = mPendingData === NOT_SET
        mPendingData = value  // ← overwrites previous pending value!
    }
    if (!postTask) return  // Runnable already posted, it will pick up new value
    ArchTaskExecutor.postToMainThread(mPostValueRunnable)
}

// The Runnable:
private val mPostValueRunnable = Runnable {
    val newValue: Any?
    synchronized(mDataLock) {
        newValue = mPendingData
        mPendingData = NOT_SET
    }
    setValue(newValue as T)  // calls setValue on main thread
}
Danger — postValue() Race

If you call postValue("A") then postValue("B") from a background thread before the main thread processes the Runnable, only "B" is delivered. The first value is silently lost. This is a common source of bugs in apps that post rapid updates from coroutines or RxJava streams.

6.6 The SingleLiveEvent Problem

LiveData re-delivers its current value to new observers (e.g., after config change). For one-shot events (navigation, snackbar), this causes duplicate handling. Common workarounds:

// Workaround 1: Event wrapper (Google sample)
class Event<out T>(private val content: T) {
    private var hasBeenHandled = false
    fun getContentIfNotHandled(): T? =
        if (hasBeenHandled) null
        else { hasBeenHandled = true; content }
}

// Workaround 2 (modern): Use Kotlin Channel + Flow
class MyViewModel : ViewModel() {
    private val _events = Channel<UiEvent>(Channel.BUFFERED)
    val events: Flow<UiEvent> = _events.receiveAsFlow()

    fun onAction() {
        viewModelScope.launch { _events.send(UiEvent.ShowSnackbar("Done")) }
    }
}

// Collect in Fragment/Activity with repeatOnLifecycle
lifecycleScope.launch {
    repeatOnLifecycle(Lifecycle.State.STARTED) {
        viewModel.events.collect { event -> handleEvent(event) }
    }
}

7.1 LocalLifecycleOwner

LocalLifecycleOwner is a CompositionLocal that provides the nearest LifecycleOwner. In an Activity, this is the Activity itself. Inside a NavHost, each destination's NavBackStackEntry is the lifecycle owner.

@Composable
fun MyScreen() {

    // lifecycleOwner.lifecycle is the NavBackStackEntry's lifecycle
    // when inside NavHost, NOT the Activity's lifecycle
}

7.2 LifecycleEventEffect

// Runs a side effect when a specific lifecycle event occurs
@Composable
fun AnalyticsScreen(viewModel: AnalyticsViewModel) {
    LifecycleEventEffect(Lifecycle.Event.ON_RESUME) {
        viewModel.trackScreenView()
    }

    LifecycleEventEffect(Lifecycle.Event.ON_PAUSE) {
        viewModel.trackScreenExit()
    }
}

// LifecycleStartEffect / LifecycleResumeEffect — paired start/stop
@Composable
fun LocationScreen() {
    LifecycleStartEffect(Unit) {
        val listener = startLocationUpdates()
        onStopOrDispose {
            stopLocationUpdates(listener)
        }
    }
}

7.3 collectAsStateWithLifecycle()

@Composable
fun SearchScreen(viewModel: SearchViewModel) {
    // Lifecycle-aware: stops collecting when lifecycle drops below STARTED
    val results by viewModel.results.collectAsStateWithLifecycle()

    // Under the hood, this uses repeatOnLifecycle:
    // 1. When lifecycle reaches STARTED → starts collecting the Flow
    // 2. When lifecycle drops below STARTED → cancels collection
    // 3. On recomposition → reuses the same collection if lifecycle hasn't changed
}

7.4 repeatOnLifecycle Internals

// Simplified implementation
suspend fun Lifecycle.repeatOnLifecycle(
    state: Lifecycle.State,
    block: suspend CoroutineScope.() -> Unit
) {
    if (currentState === Lifecycle.State.DESTROYED) return

    coroutineScope {
        withContext(Dispatchers.Main.immediate) {
            val observer = object : LifecycleEventObserver {
                var job: Job? = null

                override fun onStateChanged(source: LifecycleOwner, event: Event) {
                    if (currentState >= state) {
                        // Launch block when reaching target state
                        if (job == null) {
                            job = this@coroutineScope.launch { block() }
                        }
                    } else {
                        // Cancel block when dropping below target state
                        job?.cancel()
                        job = null
                    }
                }
            }
            lifecycle.addObserver(observer)
            try {
                // Suspend until the lifecycle is DESTROYED
                suspendCancellableCoroutine<Unit> { /* never resumed */ }
            } finally {
                lifecycle.removeObserver(observer)
            }
        }
    }
}
Staff-Level Insight

repeatOnLifecycle uses Dispatchers.Main.immediate to ensure the observer is added on the main thread synchronously. The block is launched in a new coroutine each time the state is reached and cancelled each time it drops below — this is not a pause/resume, it's a full restart. This means any local state inside the block is lost on each cycle. If you need to preserve state across stop/start cycles, hoist it to the ViewModel or use a StateFlow that survives the cancellation.

8. Staff-Level Corner Cases

8.1 Observer Catch-Up on Late Addition

When you add an observer to a lifecycle that's already in STARTED state, the observer immediately receives ON_CREATE and ON_START events (catch-up). This happens synchronously inside addObserver():

// Lifecycle is in STARTED state
lifecycle.addObserver(myObserver)
// myObserver.onCreate() called ← catch-up
// myObserver.onStart() called  ← catch-up
// ...later when lifecycle reaches RESUMED:
// myObserver.onResume() called ← normal dispatch

This is why the sync() mechanism exists — it walks the observer through every intermediate state, never skipping one.

8.2 ProcessLifecycleOwner False Positive

During a config change, there's a ~700ms window where ProcessLifecycleOwner still considers the app foregrounded. But if your Activity takes longer than 700ms to recreate (heavy onCreate, slow inflation), the process lifecycle will briefly report ON_STOP then ON_START. This can trigger false "app backgrounded" analytics events or disconnect WebSockets unnecessarily.

8.3 ViewModel Leaks

Danger — ViewModel Leak Patterns

ViewModels outlive Activities during config changes. Storing any of these in a ViewModel creates a leak:

  • Activity/Fragment reference — the old instance can't be GC'd
  • View reference — Views hold a reference to their Context (the Activity)
  • Context (non-Application) — same as Activity reference
  • Adapter with Context — RecyclerView.Adapter often holds Activity context

Use AndroidViewModel if you need Application context, but prefer injecting dependencies via constructor.

8.4 SavedStateHandle Serialization Cost

Every value in SavedStateHandle is serialized to a Bundle on every onSaveInstanceState() call. Large Parcelable objects (bitmap data, large lists) cause:

  • TransactionTooLargeException — the Bundle is sent via Binder to system_server, which has a 1MB transaction limit
  • Main thread jank — serialization happens on the main thread during onStop()
  • Rule of thumb: keep saved state under 50KB total; store IDs, not objects

8.5 postValue() Race Condition

// Background thread:
liveData.postValue("Loading")   // posts Runnable to main
liveData.postValue("Success")   // overwrites mPendingData, Runnable already posted

// Main thread eventually runs the Runnable:
// Observer sees ONLY "Success" — "Loading" is silently dropped

// Fix: use setValue() from main thread, or use StateFlow which is conflated
// but at least guarantees the latest value is always delivered

8.6 viewModelScope Cancellation Timing

class MyViewModel : ViewModel() {
    override fun onCleared() {
        // ⚠ viewModelScope is still active HERE
        // but Closeables (including the scope) are closed AFTER onCleared() returns
        viewModelScope.launch {
            // This coroutine WILL start but will be cancelled almost immediately
            // when the Closeable cleanup runs after onCleared()
        }
    }
}

// Safe pattern for cleanup work:
override fun onCleared() {
    // Use NonCancellable for critical cleanup
    viewModelScope.launch(NonCancellable) {
        repository.flushPendingWrites()
    }
    // ⚠ But even this is risky — the scope's Job is cancelled after onCleared()

}

8.7 Fragment viewModels() vs activityViewModels() with Navigation

Warning — Scoping Pitfall

In a Navigation-based app, by activityViewModels() scopes the ViewModel to the Activity, which means it lives for the entire app session. If you want to share a ViewModel between fragments in a single navigation flow (e.g., checkout), use by navGraphViewModels(R.id.checkout_graph) instead. The nav-graph-scoped ViewModel is cleared when the user navigates out of that graph, preventing memory waste from long-lived ViewModels holding stale data.

Staff-Level Insight

NavBackStackEntry implements both LifecycleOwner and ViewModelStoreOwner. When you use navGraphViewModels(), the delegate finds the NavBackStackEntry for the navigation graph (not the individual destination) and uses its ViewModelStore. This is why the ViewModel is shared across all destinations within that graph and cleared when the graph is popped. Understanding this three-way relationship (Lifecycle + ViewModelStore + SavedStateRegistry, all on NavBackStackEntry) is essential for designing correct state scoping in production apps.

9. LiveData → StateFlow Migration

// Step-by-step migration pattern:

// BEFORE (LiveData):
class MyViewModel : ViewModel() {
    private val _state = MutableLiveData<UiState>(UiState.Loading)
    val state: LiveData<UiState> = _state

    fun load() {
        viewModelScope.launch {
            _state.value = UiState.Loading
            val data = repository.fetch()
            _state.value = UiState.Success(data)
        }
    }
}

// In Fragment:
viewModel.state.observe(viewLifecycleOwner) { state -> render(state) }

// AFTER (StateFlow):
class MyViewModel : ViewModel() {
    private val _state = MutableStateFlow<UiState>(UiState.Loading)
    val state: StateFlow<UiState> = _state.asStateFlow()

    fun load() {
        viewModelScope.launch {
            _state.value = UiState.Loading
            val data = repository.fetch()
            _state.value = UiState.Success(data)
        }
    }
}

// In Fragment (lifecycle-aware collection):
viewLifecycleOwner.lifecycleScope.launch {
    viewLifecycleOwner.repeatOnLifecycle(Lifecycle.State.STARTED) {
        viewModel.state.collect { state -> render(state) }
    }
}

// In Compose:
val state by viewModel.state.collectAsStateWithLifecycle()

// For one-shot events, use SharedFlow (NOT StateFlow):
private val _events = MutableSharedFlow<Event>() // replay = 0
val events: SharedFlow<Event> = _events.asSharedFlow()

// WhileSubscribed(5000) pattern for upstream efficiency:
val uiState: StateFlow<UiState> = repository.dataFlow
    .map { UiState.Success(it) }
    .stateIn(
        scope = viewModelScope,
        started = SharingStarted.WhileSubscribed(5_000), // 5s grace for config change
        initialValue = UiState.Loading
    )

10. ProcessLifecycleOwner Edge Cases

// PiP + ProcessLifecycleOwner:
// When app enters PiP, the PiP Activity is still visible
// ProcessLifecycleOwner stays in STARTED/RESUMED state
// It does NOT fire ON_STOP even though the app appears "backgrounded"

// Multi-window + ProcessLifecycleOwner:
// Both Activities are RESUMED (Android 10+)
// ProcessLifecycleOwner is RESUMED as long as ANY Activity is RESUMED
// ON_STOP fires only when ALL Activities are stopped

// Transparent Activity overlay:
// If Activity B (transparent) launches over Activity A:
//   Activity A: onPause (but NOT onStop — still visible behind B)
//   ProcessLifecycleOwner: stays RESUMED (A is still started)

// ON_DESTROY is NEVER dispatched by ProcessLifecycleOwner
// If the process is killed, no callback fires
// Use ApplicationExitInfo (API 30+) to detect this on next launch

// ProcessLifecycleOwner initialization:
// Uses androidx.startup (ProcessLifecycleInitializer)
// Registers ActivityLifecycleCallbacks on Application
// Counts started/resumed Activities to determine app state

11. Observer Removal During Dispatch

// What happens if an observer removes itself during a lifecycle event?
val observer = object : DefaultLifecycleObserver {
    override fun onStart(owner: LifecycleOwner) {
        // Self-removal during dispatch is SAFE
        // LifecycleRegistry uses FastSafeIterableMap which supports
        // concurrent modification during iteration
        owner.lifecycle.removeObserver(this)
    }
}

// What happens if observer A removes observer B during dispatch?
// Also safe — FastSafeIterableMap handles this
// But B may or may not receive the current event depending on
// whether it was already visited in the iteration

// Adding an observer during dispatch:
// The new observer receives catch-up events to reach current state
// But it does NOT receive the currently-dispatching event
// Example: adding observer in ON_START callback → new observer
// receives ON_CREATE catch-up but NOT the current ON_START
