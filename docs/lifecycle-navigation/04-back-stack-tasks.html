04 — Back Stack & Tasks

AOSP task hierarchy, launch modes resolved inside ActivityStarter, Intent flag arithmetic, task affinity, Recents internals, and the predictive back gesture model

1. Task & Back Stack Fundamentals

A Task is the unit of user-perceived work. In AOSP it is represented by the Task class (renamed from TaskRecord → ActivityStack → Task across Android 12–14 refactors). Each Task holds a LIFO stack of ActivityRecord objects — this is the "back stack."

AOSP Window Hierarchy

Activities don't float in a vacuum. They live inside a strict containment hierarchy managed by WindowManagerService:

  RootWindowContainer                        ← singleton, root of all display state
  └── DisplayContent  (display 0)            ← one per physical/virtual display
      ├── TaskDisplayArea  ("DefaultTaskDisplayArea")
      │   ├── Task  (id=42, affinity="com.app")        ← a "root task" (≈ Recents entry)
      │   │   ├── ActivityRecord  C  (RESUMED)          ← top of back stack
      │   │   ├── ActivityRecord  B  (STOPPED)
      │   │   └── ActivityRecord  A  (STOPPED)          ← bottom / root activity
      │   │
      │   └── Task  (id=43, affinity="com.other")
      │       └── ActivityRecord  X  (STOPPED)
      │
      ├── TaskDisplayArea  ("pip-task-display-area")    ← PiP gets its own TDA
      │   └── Task  (id=44)
      │       └── ActivityRecord  P  (PAUSED)
      │
      └── ImeContainer                                  ← input method window
          └── WindowState  (IME)

Key relationships:

  • RootWindowContainer → manages all DisplayContent objects (one per display)
  • DisplayContent → contains one or more TaskDisplayArea containers
  • TaskDisplayArea → holds root-level Task objects (what the user sees in Recents)
  • Task → ordered list of ActivityRecord entries; the top one is the visible activity
  • ActivityRecord → wraps the app-side Activity instance; holds state, intent, saved instance bundle reference, and the associated WindowState

Back Stack Behavior

When the user launches Activity B from Activity A, B is pushed onto the same Task's stack. Pressing Back pops B (destroying it) and resumes A. The root activity's Back press finishes the Task — it moves to the background and may appear in Recents.

  Launch A        Launch B from A      Launch C from B      Back from C
  ┌───────┐       ┌───────┐            ┌───────┐            ┌───────┐
  │   A   │       │   B   │ ← top      │   C   │ ← top      │   B   │ ← top (resumed)
  │       │       │   A   │            │   B   │            │   A   │
  └───────┘       └───────┘            │   A   │            └───────┘
   Task 42         Task 42             └───────┘             Task 42
                                        Task 42
                                                          C is destroyed

2. Launch Modes — Deep Internals

Launch modes are declared in the manifest via android:launchMode and resolved at runtime inside ActivityStarter.computeLaunchingTaskFlags() and ActivityStarter.recycleTask(). The five modes:

standard (default)

A new ActivityRecord is created every time, placed in the caller's task. Multiple instances can coexist in the same task or across tasks. This is what ActivityStarter does when no special flags or modes apply.

singleTop

If an instance of the target Activity is already at the top of the target task, no new instance is created. Instead, onNewIntent() is delivered to the existing instance. If it's anywhere else in the stack, a new instance is created normally.

// AOSP: ActivityStarter.recycleTask()
// Simplified logic for singleTop:
if (targetActivity != null && targetActivity == top) {
    deliverNewIntent(targetActivity, intent)  // → onPause → onNewIntent → onResume
} else {
    startNewActivity(intent)
}

singleTask

Only one instance of this Activity can exist in the entire system. If the Activity already exists in any task:

  1. That task is brought to the foreground
  2. Everything above the target Activity in the stack is destroyed (CLEAR_TOP semantics)
  3. onNewIntent() is delivered to the existing instance

If no instance exists, a new task is created (or the Activity is placed in a task matching its taskAffinity).

singleInstance

Like singleTask, but the Activity is the only member of its task. No other Activity can ever be launched into the same task. Any Activity launched from a singleInstance Activity goes into a different task.

singleInstancePerTask (Android 12+)

Allows one instance per task, but multiple tasks can each have their own instance. This was added to support multi-window scenarios where you want the same Activity type in two split-screen tasks. Unlike singleInstance, other activities can exist in the same task.

Staff-Level Insight — Launch Mode Resolution Order

When both a manifest launchMode and Intent flags are present, Intent flags win for most behaviors. Specifically, ActivityStarter.computeLaunchingTaskFlags() applies Intent flags after reading the manifest mode. The one exception: singleInstance in the manifest cannot be overridden by flags — the system always enforces the "alone in its task" constraint. This means a caller can effectively upgrade standard to singleTask-like behavior using FLAG_ACTIVITY_NEW_TASK | FLAG_ACTIVITY_CLEAR_TOP, but cannot downgrade singleInstance to standard.

3. Intent Flags — Complete Reference

Core Flags

FLAG_ACTIVITY_NEW_TASK — Launch into a task matching the target's taskAffinity. If such a task exists, the Activity is placed there; otherwise a new task is created. Required when launching from a non-Activity context (Service, BroadcastReceiver, Application) because there's no existing task to inherit.

FLAG_ACTIVITY_CLEAR_TOP — If the target Activity already exists in the task, destroy every Activity above it. The critical subtlety: by default the target itself is also destroyed and recreated (not just delivered onNewIntent()). To get onNewIntent() delivery, combine with FLAG_ACTIVITY_SINGLE_TOP.

FLAG_ACTIVITY_SINGLE_TOP — Equivalent to singleTop launch mode when set as a flag. Most useful combined with CLEAR_TOP to achieve singleTask-like behavior from code.

FLAG_ACTIVITY_CLEAR_TASK — Wipe the entire task before launching the target Activity. Must be used with FLAG_ACTIVITY_NEW_TASK. The target becomes the new root of a fresh task.

FLAG_ACTIVITY_NO_HISTORY — The Activity is not retained in the back stack. Once the user navigates away, it's finished. Useful for splash screens or intermediate auth screens.

FLAG_ACTIVITY_REORDER_TO_FRONT — If the target Activity exists anywhere in the task, move it to the top without destroying anything. The stack order changes but no Activities are killed.

Recents & Multi-Window Flags

FLAG_ACTIVITY_EXCLUDE_FROM_RECENTS — The task won't appear in the Overview/Recents screen. Equivalent to android:excludeFromRecents="true" on the root Activity.

FLAG_ACTIVITY_MULTIPLE_TASK — Used with NEW_TASK: always creates a new task even if one with matching affinity exists. Without this flag, NEW_TASK reuses an existing task.

FLAG_ACTIVITY_NEW_DOCUMENT — Creates a new document-centric task in Recents. Each launch gets its own Recents entry. Used by apps like Chrome (each tab = separate Recents card).

FLAG_ACTIVITY_LAUNCH_ADJACENT — In multi-window mode, launch the Activity in the adjacent split. Has no effect outside multi-window.

Flag Combination Matrix

Flag CombinationResulting Behavior
NEW_TASKFind or create task by affinity; place Activity there
NEW_TASK | CLEAR_TOPFind task by affinity, clear above target, recreate target
NEW_TASK | CLEAR_TOP | SINGLE_TOPFind task by affinity, clear above target, deliver onNewIntent() (≈ singleTask)
NEW_TASK | CLEAR_TASKWipe entire task, launch as new root
NEW_TASK | MULTIPLE_TASKAlways create a brand-new task (ignore affinity match)

NEW_DOCUMENT | MULTIPLE_TASKAlways new Recents entry (even if same document URI)
Warning — CLEAR_TOP Without SINGLE_TOP

FLAG_ACTIVITY_CLEAR_TOP alone does not deliver onNewIntent(). It destroys the target Activity and recreates it. This catches many developers off guard. The AOSP code path: ActivityStarter calls Task.performClearTop() which finishes the target, then starts a fresh instance. Only when SINGLE_TOP is also set (or the manifest declares singleTop) does it skip destruction and deliver onNewIntent() instead.

4. Task Affinity

Every Activity has a taskAffinity — a string that declares which task the Activity "prefers" to belong to. By default, all Activities in an app share the same affinity: the application's package name.

How Affinity Works

Affinity matters in two situations:

  1. With FLAG_ACTIVITY_NEW_TASK — ActivityStarter searches for an existing root task whose affinity matches the target Activity's taskAffinity. If found, the Activity is placed in that task. If not, a new task is created with that affinity.
  2. With allowTaskReparenting="true" — When a task with matching affinity comes to the foreground, the Activity migrates from its current task to the preferred one.
// Manifest: Activity with custom affinity
<activity
    android:name=".settings.SettingsActivity"
    android:taskAffinity="com.myapp.settings"
    android:launchMode="singleTask" />

// Launching from anywhere with NEW_TASK → goes to "com.myapp.settings" task
val intent = Intent(context, SettingsActivity::class.java).apply {
    addFlags(Intent.FLAG_ACTIVITY_NEW_TASK)
}
context.startActivity(intent)

Empty Affinity

Setting taskAffinity="" means the Activity has no affinity. With FLAG_ACTIVITY_NEW_TASK, this always creates a new task because no existing task can match an empty affinity.

Affinity Resolution in AOSP

Inside ActivityStarter.getReusableTask(), the system iterates through all root tasks in the TaskDisplayArea, comparing each task's root affinity against the target Activity's taskAffinity. The first match wins. If FLAG_ACTIVITY_MULTIPLE_TASK is set, this search is skipped entirely — a new task is always created.

allowTaskReparenting

When set to true, an Activity can move between tasks. Classic example: App A launches a map Activity from App B. The map Activity initially lives in App A's task. When the user later opens App B directly, the map Activity migrates to App B's task (because its affinity matches App B).

  Before reparenting:                    After App B comes to foreground:

  Task (App A)          Task (App B)     Task (App A)          Task (App B)
  ┌──────────┐          ┌──────────┐     ┌──────────┐          ┌──────────┐
  │  MapAct  │ ← B's    │          │     │          │          │  MapAct  │ ← migrated
  │  HomeA   │  affinity │  HomeB   │     │  HomeA   │          │  HomeB   │
  └──────────┘          └──────────┘     └──────────┘          └──────────┘
Staff-Level Insight — Task Affinity as an Attack Vector

A malicious app can declare android:taskAffinity="com.target.app" on its Activity. If the target app launches an Activity with FLAG_ACTIVITY_NEW_TASK, the system may find the malicious app's task first (since affinity matching is a simple string comparison on root tasks). This is the "task hijacking" or "StrandHogg" vulnerability. The attacker's Activity appears on top, potentially showing a phishing UI. Mitigations: Android 9+ added restrictions on cross-app task reuse. Android 12+ further tightened this by requiring the caller and target to share the same UID or the target task to be truly empty. At L6, you should know this exists and recommend explicit task affinity + CLEAR_TASK for sensitive flows like login.

5. Recent Tasks (Overview Screen)

The Recents screen (Overview) is powered by RecentTasks in system_server. Each entry corresponds to a root-level Task object.

How Recents Reads Tasks

RecentTasks maintains a list of Task objects ordered by last-active time. When the user opens Recents, SystemUI calls ActivityTaskManager.getRecentTasks() which returns RecentTaskInfo objects containing the task's base intent, description, screenshot, and task ID.

Document Launch Mode

By default, each app gets one Recents entry. documentLaunchMode changes this:

ModeBehavior
none (default)Single Recents entry per task affinity
intoExistingReuse existing document task if same URI
alwaysAlways create a new Recents entry
neverNever create separate document entries

Controlling Recents Behavior

// Manifest attributes on the root Activity of a task:
<activity
    android:name=".MainActivity"
    android:autoRemoveFromRecents="true"   <!-- auto-remove when task finishes -->
    android:maxRecents="3"                 <!-- max Recents entries for this app -->
    android:excludeFromRecents="false" />

Programmatic Task Management

// Get all tasks belonging to this app
val appTasks: List<ActivityManager.AppTask> =
    activityManager.appTasks

for (task in appTasks) {
    val info = task.taskInfo
    Log.d("Tasks", "Task ${info.taskId}: base=${info.baseIntent}")

    // Move task to front
    task.moveToFront()

    // Finish and remove from Recents entirely
    task.finishAndRemoveTask()
}

// From within an Activity — remove current task from Recents
finishAndRemoveTask()
How finishAndRemoveTask() Differs from finish()

finish() destroys the current Activity but leaves the task in Recents (other Activities in the stack may still be there, or the task persists as a recent entry). finishAndRemoveTask() finishes all Activities in the task and removes the task from Recents entirely. Under the hood, it calls ActivityTaskManagerService.finishActivity() with the REMOVE_TASK flag, which triggers Task.removeIfPossible() and removes the task from RecentTasks.

6. Back Navigation Internals

Back Press Dispatch Chain

When the user presses Back (hardware button or gesture), the event travels through a well-defined chain:

  User presses Back
       │
       ▼
  InputDispatcher (system_server)
       │  dispatches KeyEvent(KEYCODE_BACK) to focused window
       ▼
  ViewRootImpl.processKeyEvent()
       │
       ▼
  Activity.onKeyDown(KEYCODE_BACK)
       │
       ▼
  Activity.onBackPressed()  [deprecated API 33+]
       │
       ▼
  OnBackPressedDispatcher.onBackPressed()
       │  iterates enabled callbacks in reverse-add order
       ▼
  ┌─ OnBackPressedCallback (app-registered, e.g. "save draft?")
  │     if enabled → handleOnBackPressed() → CONSUMED
  │     if disabled → fall through
  ├─ FragmentManager's callback (pops fragment back stack)
  │     if fragment back stack non-empty → pop → CONSUMED
  │     if empty → fall through
  └─ Default callback → Activity.finish()
       │
       ▼
  Task back stack pops the ActivityRecord
  If task is now empty → task moves to background

Predictive Back Gesture (Android 13+)

Android 13 introduced the ahead-of-time back navigation model. Instead of the system only knowing what happens after the user completes the back gesture, apps now declare their back handling before the gesture starts. This enables the system to show a preview animation (e.g., peeking at the home screen behind the app).

Platform API: OnBackInvokedCallback

// Platform API (no AndroidX dependency)
val callback = OnBackInvokedCallback {
    // Handle back — save state, dismiss dialog, etc.
    handleBack()
}

// Register with priority
onBackInvokedDispatcher.registerOnBackInvokedCallback(
    OnBackInvokedDispatcher.PRIORITY_DEFAULT,
    callback
)

// Unregister when no longer needed (e.g., dialog dismissed)
onBackInvokedDispatcher.unregisterOnBackInvokedCallback(callback)

AndroidX API: OnBackPressedCallback

// AndroidX — lifecycle-aware, works on older APIs too
val callback = object : OnBackPressedCallback(enabled = true) {
    override fun handleOnBackPressed() {
        if (hasUnsavedChanges) {
            showSaveDialog()
        } else {
            isEnabled = false       // disable self
            onBackPressedDispatcher.onBackPressed()  // re-dispatch
        }
    }
}

// Tied to lifecycle — auto-removed when DESTROYED
onBackPressedDispatcher.addCallback(viewLifecycleOwner, callback)

System Back vs Activity.finish()

These are different code paths:

  • System Back: Goes through OnBackPressedDispatcher → callbacks → fragment back stack → then finish(). The predictive back animation plays. ActivityRecord.finishing is set by ActivityTaskManagerService.
  • Activity.finish(): Directly calls ActivityTaskManagerService.finishActivity() via Binder. Skips the dispatcher chain entirely. No predictive back animation. The Activity is scheduled for destruction immediately.
Warning — Opting Into Predictive Back

Predictive back animations require android:enableOnBackInvokedCallback="true" in the manifest. Without this, the system falls back to the legacy onBackPressed() path and no preview animation is shown. If your app overrides onBackPressed() without migrating to OnBackPressedCallback, enabling this flag will break your custom back handling — the system will ignore onBackPressed() entirely and use the dispatcher chain.

7. Staff-Level Corner Cases

singleTask + CLEAR_TOP — The "Nuke from Orbit"

Combining singleTask launch mode with FLAG_ACTIVITY_CLEAR_TOP is redundant but harmless — singleTask already implies CLEAR_TOP behavior. However, developers sometimes add CLEAR_TOP explicitly "for safety." The real danger is combining singleTask with FLAG_ACTIVITY_CLEAR_TASK: this wipes the entire task including the singleTask Activity itself, then recreates it as the new root. This is the nuclear option for resetting navigation state.

// "Reset to login" pattern — wipe everything
val intent = Intent(this, LoginActivity::class.java).apply {
    addFlags(
        Intent.FLAG_ACTIVITY_NEW_TASK or
        Intent.FLAG_ACTIVITY_CLEAR_TASK
    )
}
startActivity(intent)
// LoginActivity is now the only Activity in a fresh task

Launching from Non-Activity Contexts

Starting an Activity from a Service, BroadcastReceiver, or Application context requires FLAG_ACTIVITY_NEW_TASK. Without it, you get:

// Crash: "Calling startActivity() from outside of an Activity context
// requires the FLAG_ACTIVITY_NEW_TASK flag."
// This is enforced in ContextImpl.startActivity():
if ((intent.flags & Intent.FLAG_ACTIVITY_NEW_TASK) == 0) {
    throw AndroidRuntimeException(
        "Calling startActivity() from outside of an Activity context..."
    )
}

On Android 10+, there are additional restrictions: background activity starts are blocked unless the app has a visible window, a foreground service with the right type, or a PendingIntent from a visible app.

Task Reparenting Race Conditions

allowTaskReparenting can cause confusing behavior when combined with singleTask. If Activity B (from App B, with allowTaskReparenting="true") is in App A's task, and the user launches App B which has a singleTask root Activity — the reparenting and the singleTask resolution can conflict. The system resolves this by performing reparenting first (moving B to App B's task), then applying singleTask logic. But the timing depends on which task comes to the foreground first, leading to non-deterministic stack states in edge cases.

Deep Links & Synthetic Back Stacks

When a deep link lands the user in the middle of your app (e.g., a notification opens OrderDetailActivity), pressing Back should navigate to OrderListActivity → MainActivity, not exit the app. TaskStackBuilder synthesizes this back stack:

// Build a synthetic back stack for a deep link
val pendingIntent = TaskStackBuilder.create(context)
    .addNextIntentWithParentStack(
        Intent(context, OrderDetailActivity::class.java).apply {
            putExtra("orderId", orderId)
        }
    )
    .getPendingIntent(
        requestCode,
        PendingIntent.FLAG_UPDATE_CURRENT or PendingIntent.FLAG_IMMUTABLE
    )

// This creates: MainActivity → OrderListActivity → OrderDetailActivity
// The parent chain is read from android:parentActivityName in the manifest:
// <activity android:name=".OrderDetailActivity"
//     android:parentActivityName=".OrderListActivity" />
// <activity android:name=".OrderListActivity"
//     android:parentActivityName=".MainActivity" />

Under the hood, TaskStackBuilder creates an array of Intents and calls ContextImpl.startActivities(), which sends them all to ActivityTaskManagerService in a single transaction. The system creates the Activities in order, building the back stack bottom-up.

CLEAR_TOP Without SINGLE_TOP — The Destroy-Recreate Trap

This is one of the most common bugs in Android navigation:

  Stack: A → B → C → D
  Launch B with FLAG_ACTIVITY_CLEAR_TOP (no SINGLE_TOP)

  Step 1: Clear above B → C and D are destroyed
  Step 2: B is ALSO destroyed (not reused!)
  Step 3: New instance of B is created

  Result: A → B'  (B' is a NEW instance, B's state is lost)

  vs. with CLEAR_TOP | SINGLE_TOP:

  Step 1: Clear above B → C and D are destroyed
  Step 2: B receives onNewIntent() (same instance)

  Result: A → B   (same B, state preserved)

Multiple Tasks with Same Affinity

FLAG_ACTIVITY_MULTIPLE_TASK combined with FLAG_ACTIVITY_NEW_TASK bypasses affinity matching and always creates a new task. This means you can end up with multiple tasks that have the same affinity string. The Recents screen shows them as separate entries. This is intentional for document-centric apps but can be confusing if used accidentally.

// Creates a new task every time, even if one with matching affinity exists
val intent = Intent(this, EditorActivity::class.java).apply {
    addFlags(
        Intent.FLAG_ACTIVITY_NEW_TASK or
        Intent.FLAG_ACTIVITY_MULTIPLE_TASK
    )
    data = documentUri  // each document gets its own task
}
startActivity(intent)
Staff-Level Insight — Debugging Task State in Production

Use adb shell dumpsys activity activities to inspect the full task hierarchy at runtime. The output shows every Task, its affinity, and the ordered list of ActivityRecord entries with their states. For programmatic inspection, ActivityManager.getRunningTasks() is deprecated but ActivityManager.getAppTasks() still works for your own app's tasks. In production, log taskId (from Activity.getTaskId()) and intent.flags in your analytics to diagnose navigation bugs. At Google scale, task/back-stack bugs are the #1 source of "user ended up on a blank screen" reports — they're notoriously hard to reproduce because they depend on the exact sequence of foreground/background transitions, process death timing, and which Recents entry the user tapped.

Danger — Background Activity Start Restrictions (Android 10+)

Starting with Android 10, apps cannot start Activities from the background unless they meet specific exemptions (visible window, foreground service with type, system UID, SYSTEM_ALERT_WINDOW permission, companion device manager, etc.). Attempting to do so silently fails — no crash, no log, the Activity just doesn't appear. This breaks many legacy patterns like "show a full-screen alarm from a Service." The fix is to use a high-priority notification with a full-screen intent (setFullScreenIntent()), which the system will either show as a heads-up notification or launch as a full-screen Activity depending on the device state.

8. TaskFragment (Android 12L+)

Android 12L introduced TaskFragment as a subdivision of Task. A Task can now contain multiple TaskFragments, each holding a group of Activities. This enables the system to display two Activities side-by-side within the same Task (used by Jetpack WindowManager's Activity Embedding):

// TaskFragment hierarchy:
//   Task
//   ├── TaskFragment (left pane)
//   │   └── ActivityRecord (ListActivity)
//   └── TaskFragment (right pane)
//       └── ActivityRecord (DetailActivity)
//
// Key implications:
// - Both Activities share the same Task (same taskId, same affinity)
// - Each TaskFragment has its own lifecycle management
// - Back press pops from the focused TaskFragment first
// - This is the foundation for large-screen Activity Embedding
//
// Jetpack WindowManager Activity Embedding API:
val splitPairRule = SplitPairRule.Builder(
    setOf(SplitPairFilter(
        ComponentName(this, ListActivity::class.java),
        ComponentName(this, DetailActivity::class.java),
        null
    ))
).setMinWidthDp(600)
 .setFinishPrimaryWithSecondary(SplitRule.FinishBehavior.NEVER)
 .build()

RuleController.getInstance(this).addRule(splitPairRule)

9. Predictive Back — Animation Phases

// Android 14+ predictive back with animation callbacks:
// The system provides gesture progress so you can animate the back transition

// Platform API (for custom animations):
onBackInvokedDispatcher.registerOnBackInvokedCallback(
    OnBackInvokedDispatcher.PRIORITY_DEFAULT,
    object : OnBackAnimationCallback {
        override fun onBackStarted(backEvent: BackEvent) {
            // Gesture started — begin animation setup
            // backEvent.touchX, touchY, progress (0.0)
        }
        override fun onBackProgressed(backEvent: BackEvent) {
            // Gesture in progress — update animation
            // backEvent.progress: 0.0 → 1.0
            // backEvent.swipeEdge: EDGE_LEFT or EDGE_RIGHT
            view.scaleX = 1f - (backEvent.progress * 0.1f)
        }
        override fun onBackInvoked() {
            // Gesture completed — commit the back navigation
            finish()
        }
        override fun onBackCancelled() {
            // Gesture cancelled — restore original state
            view.scaleX = 1f
        }
    }
)

// AndroidX (for Fragment/Navigation integration):
// OnBackPressedCallback already supports predictive back
// when handleOnBackStarted/handleOnBackProgressed/handleOnBackCancelled
// are overridden

// Manifest opt-in required:
// android:enableOnBackInvokedCallback="true"
//
// Without opt-in, the system falls back to the legacy onBackPressed() path
// and predictive animations are disabled

// FLAG_ACTIVITY_REQUIRE_NON_BROWSER (Android 12+):
// Prevents the intent from resolving to a browser app
// Useful when you expect a specific app to handle the intent
