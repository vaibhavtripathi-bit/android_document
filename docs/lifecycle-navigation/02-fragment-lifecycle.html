02 — Fragment Lifecycle Deep Dive

Dual lifecycles, FragmentManager state machines, BackStackRecord internals, SpecialEffectsController, and every corner case that trips up L6 candidates

1. The Dual Lifecycle — Fragment vs Fragment View

The single most important concept in Fragment internals: a Fragment has TWO independent lifecycles. The Fragment object lifecycle and the Fragment View lifecycle are decoupled. When a Fragment is placed on the back stack, its View is destroyed but the Fragment object remains in CREATED state.

  Fragment Object Lifecycle          Fragment View Lifecycle
  ═══════════════════════           ═══════════════════════

       │

       │                                  │

Staff-Level Insight — The ViewBinding Leak

The most common Fragment memory leak at scale: holding a ViewBinding reference as a class field. When the Fragment goes on the back stack, onDestroyView() fires but the Fragment object stays alive. If you hold private var binding: MyBinding? = myBinding without nulling it in onDestroyView(), the entire View tree is retained by the living Fragment object. The fix is either null the binding in onDestroyView() or use a delegate that auto-clears. Google's own architecture samples use the null-in-onDestroyView pattern. For Flow collection, always use viewLifecycleOwner.lifecycleScope — never the Fragment's own lifecycleScope — to ensure collection stops when the View is destroyed.

2. FragmentManager Internals

Architecture Overview

  FragmentManager (FragmentManagerImpl)
  ├── FragmentStore
  │   ├── mAdded: ArrayList<Fragment>         ← currently added fragments
  │   ├── mActive: HashMap<String, FragmentStateManager>  ← all active (incl. back stack)
  │   └── mSavedState: HashMap<String, Bundle>  ← saved state for removed fragments
  │
  ├── mBackStack: ArrayList<BackStackRecord>  ← back stack entries
  │
  ├── FragmentLifecycleCallbacksDispatcher    ← observer pattern for lifecycle events
  │
  ├── SpecialEffectsController                ← animation/transition orchestration
  │
  └── mHost: FragmentHostCallback             ← bridge to Activity/other host
      └── provides: Context, Handler, LoaderManager, ViewModelStore

FragmentStateManager — Per-Fragment State Machine

Each active Fragment has a FragmentStateManager that tracks its current state and drives transitions. The states form an ordered enum:

// From FragmentStateManager / Fragment.java (AOSP)
// These are integer constants — ordering matters for moveToState()

static final int INITIALIZING = -1;  // not yet attached
static final int ATTACHED = 0;       // onAttach() called
static final int CREATED = 1;        // onCreate() called
static final int VIEW_CREATED = 2;   // onCreateView + onViewCreated done
static final int AWAITING_EXIT_EFFECTS = 3;  // waiting for exit animation
static final int STARTED = 4;        // onStart() called
static final int RESUMED = 5;        // onResume() called
static final int AWAITING_ENTER_EFFECTS = 6; // waiting for enter animation

// The state machine moves UP (attach→resume) or DOWN (resume→destroy)
// moveToState() compares current state vs target state and calls
// the appropriate callbacks in sequence

moveToState() — The Core Engine

FragmentStateManager.moveToState() is the single method that drives all Fragment lifecycle transitions. FragmentManager calls it for each Fragment whenever the host Activity's state changes or a transaction executes.

// Simplified flow of FragmentManager.moveToExpectedState()
// Called after every transaction execution and on Activity lifecycle changes

fun moveFragmentToExpectedState(f: Fragment) {
    val stateManager = mActive[f.mWho] ?: return
    
    // Target state = min(FragmentManager's max state, fragment's own max)
    // setMaxLifecycle() can cap a fragment below the FM's state
    val targetState = min(
        computeExpectedState(),       // from host Activity state
        f.mMaxState.ordinal           // from setMaxLifecycle()
    )
    
    if (f.mState < targetState) {
        // Moving UP: INITIALIZING → ATTACHED → CREATED → VIEW_CREATED → STARTED → RESUMED
        stateManager.moveToExpectedState()
    } else if (f.mState > targetState) {
        // Moving DOWN: RESUMED → STARTED → CREATED → VIEW_CREATED(destroy) → DESTROYED
        stateManager.moveToExpectedState()
    }
}

// Inside FragmentStateManager.moveToExpectedState():
// Each step checks current state, calls the next callback, updates mState
// Example: CREATED → VIEW_CREATED triggers:
//   1. ensureInflatedView() or onCreateView()
//   2. onViewCreated()
//   3. onViewStateRestored()
//   4. mState = VIEW_CREATED

executePendingTransactions() — Batching

// FragmentManager maintains a queue of pending operations:
//   mPendingActions: ArrayList<OpGenerator>

// commit() posts to the Handler → async execution on next message loop
// commitNow() → executes immediately, synchronously

// executePendingTransactions() flow:
fun executePendingTransactions(): Boolean {
    // Guard against re-entrant execution
    if (mExecutingActions) {
        throw IllegalStateException(
            "FragmentManager is already executing transactions"
        )
    }
    
    mExecutingActions = true
    try {
        // 1. Generate BackStackRecords from pending actions
        //    (each commit() added an OpGenerator to mPendingActions)
        generateOpsForPendingActions(mTmpRecords, mTmpIsPop)
        
        // 2. Optimize: merge redundant operations
        //    e.g., add(A) + remove(A) in same batch → no-op
        optimizeAndExecuteOps(mTmpRecords, mTmpIsPop)
        
        // 3. For each record: executeOps() → modifies FragmentStore
        // 4. moveToState() for all affected fragments
        // 5. Fire onBackStackChanged listeners
    } finally {
        mExecutingActions = false
    }
}

commit() vs commitNow() vs commitAllowingStateLoss()

MethodExecutionState-Loss Safe?Back Stack?Internals
commit()Async (posted to Handler)No — throws after onSaveInstanceStateYesAdds OpGenerator to mPendingActions, posts MSG_EXEC to Handler
commitNow()SynchronousNoNo (throws if addToBackStack used)Calls executePendingTransactions() inline; disallows back stack to prevent ordering issues
commitAllowingStateLoss()AsyncYes — silently drops stateYesSame as commit() but sets mAllowStateLoss = true, skipping the state check
commitNowAllowingStateLoss()SynchronousYesNoCombination: immediate + no state check
Warning — commitNow() and Back Stack

commitNow() throws IllegalStateException if you call addToBackStack() on the same transaction. The reason: synchronous execution would interleave with async commits already in the queue, making back stack ordering unpredictable. If you need both sync execution and back stack, use commit() followed by executePendingTransactions().

3. Fragment Transactions — BackStackRecord

BackStackRecord Internals

// FragmentTransaction is abstract. The concrete implementation is BackStackRecord.
// Each BackStackRecord holds a list of Op objects:

class BackStackRecord : FragmentTransaction(), BackStackEntry {
    val mOps = ArrayList<Op>()
    var mAddToBackStack = false
    var mName: String? = null  // optional back stack name for popBackStack(name)
    
    class Op {
        var mCmd: Int = 0          // ADD, REMOVE, REPLACE, HIDE, SHOW, etc.
        var mFragment: Fragment? = null
        var mEnterAnim: Int = 0
        var mExitAnim: Int = 0
        var mPopEnterAnim: Int = 0
        var mPopExitAnim: Int = 0
    }
}

// Operation constants:
// OP_NULL = 0, OP_ADD = 1, OP_REPLACE = 2, OP_REMOVE = 3,
// OP_HIDE = 4, OP_SHOW = 5, OP_DETACH = 6, OP_ATTACH = 7,
// OP_SET_PRIMARY_NAV = 8, OP_UNSET_PRIMARY_NAV = 9,
// OP_SET_MAX_LIFECYCLE = 10

How REPLACE Works Internally

// replace(containerViewId, fragment) is NOT atomic — it expands to:
//   1. REMOVE every existing fragment in that container
//   2. ADD the new fragment

// From BackStackRecord.executeOps():
fun replace(containerViewId: Int, fragment: Fragment): FragmentTransaction {
    // Internally, the FragmentManager iterates mAdded:
    //   for (f in fragmentStore.mAdded) {
    //       if (f.mContainerId == containerViewId) {
    //           addOp(Op(OP_REMOVE, f))  // remove each existing
    //       }
    //   }
    //   addOp(Op(OP_ADD, fragment))       // then add new one
    
    // When addToBackStack() is set, popping reverses this:
    //   REMOVE the new fragment, then RE-ADD all the old ones
    //   Old fragments' saved state is restored from FragmentStore.mSavedState
    return this
}

addToBackStack() — What Gets Saved

// When addToBackStack() is called:
// 1. The BackStackRecord is pushed onto FragmentManager.mBackStack
// 2. For each REMOVED fragment, its state is saved:
//    - Fragment.SavedState (view hierarchy state)
//    - Fragment.mSavedFragmentState (Bundle from onSaveInstanceState)
//    - Arguments bundle reference (preserved, not copied)
//    - The fragment's mWho identifier (UUID string)
// 3. The fragment is moved to CREATED state (view destroyed)
//    but kept in FragmentStore.mActive

// On popBackStack():
// 1. The BackStackRecord operations are REVERSED
// 2. Removed fragments are re-instantiated from saved state
// 3. Added fragments are removed
// 4. View hierarchy is recreated via onCreateView()

popBackStack() — Async vs Sync

// popBackStack() is ASYNCHRONOUS — posts to Handler like commit()
// popBackStackImmediate() is SYNCHRONOUS — executes inline

// Both support name and flags:
fragmentManager.popBackStack("detail", FragmentManager.POP_BACK_STACK_INCLUSIVE)
// Pops everything up to AND including the entry named "detail"

// Without INCLUSIVE flag: pops up to but NOT including the named entry

// popBackStack(null, POP_BACK_STACK_INCLUSIVE) → clears entire back stack

// The pop operation:
// 1. Finds the target BackStackRecord in mBackStack
// 2. Generates reverse operations (add↔remove, show↔hide)
// 3. Executes them through the same moveToState() pipeline

4. SpecialEffectsController — Animation Orchestration

Fragment animations and transitions are not simple view animations — they are deeply integrated with the lifecycle state machine through SpecialEffectsController.

  Transaction executes REPLACE(A → B)
  ┌──────────────────────────────────────────────────────────────┐
  │                                                              │
  │  Fragment A (exiting)              Fragment B (entering)     │
  │  ─────────────────                 ──────────────────        │
  │  State: STARTED                    State: CREATED            │
  │       │                                  │                   │
  │  moveToState(CREATED)              moveToState(STARTED)      │
  │       │                                  │                   │
  │       ▼                                  ▼                   │
  │  AWAITING_EXIT_EFFECTS ◄──┐    ┌──► AWAITING_ENTER_EFFECTS  │
  │       │                   │    │         │                   │
  │       │    SpecialEffectsController      │                   │
  │       │    orchestrates both together    │                   │
  │       │         │                        │                   │
  │       │    ┌────┴────┐                   │                   │
  │       │    │ Runs:   │                   │                   │
  │       │    │ - Exit animation for A      │                   │
  │       │    │ - Enter animation for B     │                   │
  │       │    │ - Shared element transitions │                  │
  │       │    └────┬────┘                   │                   │
  │       │         │ (all complete)         │                   │
  │       ▼         ▼                        ▼                   │
  │  onDestroyView()               onStart() / onResume()       │
  │                                                              │
  └──────────────────────────────────────────────────────────────┘
// SpecialEffectsController manages Operation objects:

class SpecialEffectsController {
    // Each fragment transition creates an Operation:
    class Operation(
        val fragment: Fragment,
        val lifecycleImpact: LifecycleImpact,  // ADDING, REMOVING, NONE
        val effects: List<Effect>               // animations, transitions
    ) {
        // The Operation holds a Signal that blocks lifecycle progression
        // Until ALL effects complete, the fragment stays in AWAITING_* state
        
        fun complete() {
            // Called when all animations/transitions finish
            // Unblocks the fragment's lifecycle to proceed to final state
        }
    }
    
    // Three types of Effect run per Operation:
    // 1. AnimationEffect — old-style view animations (R.anim.*)
    // 2. TransitionEffect — AndroidX Transition framework
    // 3. SharedElementEffect — shared element transitions
    
    // All three must complete before Operation.complete() fires
}

setMaxLifecycle() — External Lifecycle Control

// setMaxLifecycle() caps a fragment's lifecycle state from outside.
// Primary consumer: ViewPager2's FragmentStateAdapter

// ViewPager2 uses BEHAVIOR_RESUME_ONLY_CURRENT_FRAGMENT:
class FragmentStateAdapter(fm: FragmentManager, lifecycle: Lifecycle) {
    // For the CURRENT page:
    transaction.setMaxLifecycle(fragment, Lifecycle.State.RESUMED)
    
    // For OFF-SCREEN pages:
    transaction.setMaxLifecycle(fragment, Lifecycle.State.STARTED)
    
    // This means off-screen fragments are STARTED but never RESUMED
    // Their views exist (for smooth swiping) but they don't get onResume()
}

// Internally, setMaxLifecycle() stores the cap in Fragment.mMaxState
// moveToState() checks: targetState = min(hostState, mMaxState)
// The fragment simply won't advance past the cap

5. Fragment View Lifecycle — Deep Dive

getViewLifecycleOwner() vs this (Fragment as LifecycleOwner)

// Fragment implements LifecycleOwner TWICE:
// 1. Fragment itself:       getLifecycle()          → tracks CREATED..DESTROYED
// 2. Fragment's View:       getViewLifecycleOwner() → tracks VIEW_CREATED..VIEW_DESTROYED

// CRITICAL: For UI-related observation, ALWAYS use viewLifecycleOwner

// WRONG — leaks observation across back stack navigation:
class MyFragment : Fragment() {
    override fun onViewCreated(view: View, savedInstanceState: Bundle?) {
        // This observer survives onDestroyView() because Fragment lifecycle
        // is still CREATED on the back stack. When the fragment returns,
        // a SECOND observer is registered → duplicate updates
        viewModel.data.observe(this) { updateUI(it) }
    }
}

// CORRECT — observation scoped to View lifetime:
class MyFragment : Fragment() {
    override fun onViewCreated(view: View, savedInstanceState: Bundle?) {
        viewModel.data.observe(viewLifecycleOwner) { updateUI(it) }
    }
}

// For Flow collection:
override fun onViewCreated(view: View, savedInstanceState: Bundle?) {
    viewLifecycleOwner.lifecycleScope.launch {
        viewLifecycleOwner.repeatOnLifecycle(Lifecycle.State.STARTED) {
            viewModel.uiState.collect { state -> render(state) }
        }
    }
    // Collection automatically cancels in onDestroyView()
    // Re-starts when fragment returns from back stack and gets a new View
}

onViewStateRestored() and SavedStateRegistry

// onViewStateRestored() fires AFTER onViewCreated() and AFTER the framework
// has restored all View state (EditText text, ScrollView position, etc.)

// The restoration flow:
// 1. onCreateView() → inflate views
// 2. onViewCreated() → bind views, set up listeners
// 3. Framework calls view.onRestoreInstanceState() for each View with an ID
// 4. onViewStateRestored() → all view state is now restored
//    Safe to read EditText.text, RecyclerView scroll position, etc.

// SavedStateRegistry integration:
// Fragment's SavedStateRegistryController creates a registry scoped to
// the Fragment. SavedStateHandle (in ViewModel) uses this registry.
// The registry is restored in onCreate(), but View-related state
// is restored in onViewStateRestored().

// This is why SavedStateHandle works across process death:
// ViewModel.SavedStateHandle → Fragment.SavedStateRegistry → 
//   Activity.SavedStateRegistry → Activity.onSaveInstanceState() Bundle

6. Child Fragments and Nested FragmentManagers

The FragmentManager Hierarchy

  Activity
  └── supportFragmentManager (Activity-level)
      ├── FragmentA
      │   └── childFragmentManager (FragmentA-level)
      │       ├── ChildFragment1
      │       └── ChildFragment2
      │           └── childFragmentManager (ChildFragment2-level)
      │               └── GrandchildFragment
      └── FragmentB
          └── childFragmentManager (FragmentB-level)
              └── ChildFragment3

  Key relationships:
  ─────────────────
  FragmentA.parentFragmentManager  == Activity.supportFragmentManager
  ChildFragment1.parentFragmentManager == FragmentA.childFragmentManager
  FragmentA.childFragmentManager   != FragmentA.parentFragmentManager
// Lifecycle dispatch order: ALWAYS parent before child (top-down)
// 
// Activity.onStart()
//   → FragmentManager dispatches onStart() to FragmentA
//     → FragmentA.childFragmentManager dispatches onStart() to ChildFragment1
//     → FragmentA.childFragmentManager dispatches onStart() to ChildFragment2
//       → ChildFragment2.childFragmentManager dispatches onStart() to Grandchild
//   → FragmentManager dispatches onStart() to FragmentB
//     → FragmentB.childFragmentManager dispatches onStart() to ChildFragment3
//
// Destruction is REVERSE: child before parent (bottom-up)
// GrandchildFragment.onStop() → ChildFragment2.onStop() → FragmentA.onStop()

setPrimaryNavigationFragment()

// The primary navigation fragment receives back button events first.
// Only ONE fragment per FragmentManager can be primary.

// When the user presses Back:
// 1. Activity.onBackPressed() → FragmentManager.popBackStackImmediate()
// 2. FM checks: is there a primaryNavigationFragment?
// 3. If yes → delegate to that fragment's childFragmentManager first
// 4. If child FM handled it (had back stack entries) → done
// 5. If not → FM pops its own back stack
// 6. If FM has nothing to pop → Activity finishes

// NavHostFragment sets itself as primaryNavigationFragment
// This is how Navigation Component intercepts back presses

// In a transaction:
transaction.setPrimaryNavigationFragment(navHostFragment)
// Internally sets OP_SET_PRIMARY_NAV operation
Danger — Wrong FragmentManager

Using parentFragmentManager from inside a child fragment to show a DialogFragment will attach the dialog to the parent's FragmentManager. If the parent fragment is destroyed, the dialog leaks or crashes. Always use childFragmentManager for fragments that should be scoped to the current fragment's lifetime. Conversely, using childFragmentManager when you intend to navigate away from the parent will cause the child transaction to be lost.

7. Fragment Factory and Constructor Arguments

The No-Arg Constructor Requirement

// When Android recreates a Fragment (config change, process death),
// it uses reflection:
//   fragment = fragmentClass.getConstructor().newInstance()
//
// If there's no no-arg constructor → InstantiationException
//
// The arguments Bundle survives recreation:
//   1. Fragment.setArguments(bundle) stores in Fragment.mArguments
//   2. On save: mArguments is written to the saved state Bundle
//   3. On recreate: new instance created, then mArguments is restored
//   4. arguments is available in onAttach() and later

// WRONG — will crash on recreation:
class UserFragment(private val userId: String) : Fragment()

// CORRECT — use companion factory + arguments:
class UserFragment : Fragment() {
    companion object {
        fun newInstance(userId: String) = UserFragment().apply {
            arguments = bundleOf("userId" to userId)
        }
    }
    
    private val userId: String by lazy {
        requireArguments().getString("userId")!!
    }
}

FragmentFactory — Custom Instantiation

// FragmentFactory replaces the default reflection-based instantiation.
// Registered on FragmentManager before any fragments are created.

class MyFragmentFactory(
    private val repository: UserRepository
) : FragmentFactory() {
    
    override fun instantiate(classLoader: ClassLoader, className: String): Fragment {
        return when (loadFragmentClass(classLoader, className)) {
            UserFragment::class.java -> UserFragment(repository)
            else -> super.instantiate(classLoader, className)
        }
    }
}

// Register BEFORE super.onCreate() in Activity:
class MyActivity : AppCompatActivity() {
    override fun onCreate(savedInstanceState: Bundle?) {
        supportFragmentManager.fragmentFactory = MyFragmentFactory(repo)
        super.onCreate(savedInstanceState)  // fragments may be recreated here
    }
}

// How it works internally:
// FragmentManager.instantiate() → calls fragmentFactory.instantiate()
// instead of Class.newInstance(). The factory is also propagated to
// child FragmentManagers.

Hilt @AndroidEntryPoint and Fragment Recreation

// Hilt generates a superclass that installs a custom FragmentFactory:
//
// @AndroidEntryPoint
// class MyFragment : Fragment()
//
// Generates: Hilt_MyFragment extends Fragment
// MyFragment actually extends Hilt_MyFragment

8. Staff-Level Corner Cases

Committing After onSaveInstanceState

// After Activity.onSaveInstanceState(), the framework has captured state.
// Any fragment transaction committed now would be LOST on process death.

// commit() → throws IllegalStateException("Can not perform this action
//             after onSaveInstanceState")
// commitAllowingStateLoss() → silently proceeds, state may be lost

// When is commitAllowingStateLoss() acceptable?
// 1. UI updates that don't need to survive process death
//    (e.g., showing a transient error banner fragment)
// 2. Analytics/logging fragments that don't hold user state
// 3. NEVER for navigation that the user expects to persist

// What "state loss" actually means:
// The transaction executes and the UI updates, but if the process is killed
// before the next onSaveInstanceState(), the transaction is not in the
// saved state. On restoration, the fragment won't be there.

setRetainInstance() — Deprecated, Use ViewModel

// setRetainInstance(true) kept the Fragment object alive across config changes.
// The Fragment was detached from the old Activity and re-attached to the new one.
// Callbacks skipped: onDestroy() and onCreate() — went directly from
// onDestroyView() → onCreateView()

// DEPRECATED in AndroidX Fragment 1.3.0 because:
// 1. Retained fragments bypass the normal save/restore mechanism
// 2. They hold references to the old Activity's View hierarchy (leak-prone)
// 3. ViewModel provides the same benefit without the lifecycle complexity
// 4. Retained fragments don't work with Navigation Component

// Migration: move retained data to ViewModel
// ViewModel survives config change via ViewModelStore held by
// the Activity's NonConfigurationInstances (same mechanism, cleaner API)

DialogFragment Lifecycle Quirks

// DialogFragment has additional lifecycle complexity:
// 1. It creates a Dialog in onCreateDialog() OR a View in onCreateView()
// 2. The Dialog has its own Window and DecorView
// 3. dismiss() triggers a fragment transaction (remove)

// Dangerous pattern — dismiss during onSaveInstanceState:
class MyDialog : DialogFragment() {
    fun safeDismiss() {
        // WRONG: dismiss() calls commitInternal() which may throw
        // if called after onSaveInstanceState
        // dismiss()
        
        // CORRECT: use dismissAllowingStateLoss()
        dismissAllowingStateLoss()
    }
}

// Another quirk: DialogFragment.show(fm, tag) internally calls:
//   fm.beginTransaction().add(this, tag).commit()
// There's no container view — the fragment is added without a container ID.
// The Dialog's window is separate from the Activity's view hierarchy.

// showNow() variant uses commitNow() — synchronous but no back stack

postponeEnterTransition() — Shared Element Timing

// Shared element transitions need the target View to be laid out
// before the transition can calculate start/end bounds.
// Problem: RecyclerView items aren't laid out when the fragment starts.

class DetailFragment : Fragment() {
    override fun onCreate(savedInstanceState: Bundle?) {
        super.onCreate(savedInstanceState)
        // Tell the framework: DON'T start the enter transition yet
        postponeEnterTransition()
        
        // Safety timeout — if we forget to start, don't block forever
        postponeEnterTransition(2000, TimeUnit.MILLISECONDS)
    }
    
    override fun onViewCreated(view: View, savedInstanceState: Bundle?) {
        val imageView = view.findViewById<ImageView>(R.id.hero_image)
        imageView.transitionName = "hero_${args.itemId}"
        
        // Load image, then start transition when ready
        Glide.with(this)
            .load(args.imageUrl)
            .listener(object : RequestListener<Drawable> {
                override fun onResourceReady(...): Boolean {
                    // Image loaded — View is ready for transition
                    startPostponedEnterTransition()
                    return false
                }
                override fun onLoadFailed(...): Boolean {
                    // Still start transition on failure to avoid blocking
                    startPostponedEnterTransition()
                    return false
                }
            })
            .into(imageView)
    }
}

// Internally:
// postponeEnterTransition() sets Fragment.mPostponedAlpha to 0 (invisible)
// The SpecialEffectsController sees the postpone flag and waits
// startPostponedEnterTransition() clears the flag → effects proceed
// If timeout fires first, the transition starts with whatever state exists
Staff-Level Insight — Fragment State Restoration Ordering

When the system recreates fragments after process death, the ordering is subtle and frequently misunderstood. The Activity's onCreate(savedInstanceState) calls super.onCreate() which triggers FragmentManager.restoreSaveState(). This recreates all fragments synchronously during super.onCreate() — before your Activity's onCreate() body runs. This means: (1) FragmentFactory must be set before super.onCreate(), (2) any data the fragment needs from the Activity must come through arguments or SavedStateHandle, not Activity fields that are initialized after super.onCreate(), and (3) findFragmentById() returns the recreated fragment immediately after super.onCreate() — you should check for an existing fragment before adding a new one to avoid duplicates.

Fragment in ViewPager2 — BEHAVIOR_RESUME_ONLY_CURRENT_FRAGMENT

// ViewPager2's FragmentStateAdapter manages fragment lifecycle via setMaxLifecycle():
//
// Page 0 (current):  setMaxLifecycle(RESUMED)   → fully interactive
// Page 1 (adjacent): setMaxLifecycle(STARTED)   → view exists, not resumed
// Page 3 (far off):  fragment destroyed entirely → recycled
//
// This replaced the old ViewPager's setUserVisibleHint() hack.
//
// The adapter also handles:
// - Fragment recreation after config change (via FragmentManager saved state)
// - Stable IDs for correct fragment-to-position mapping
// - containsItem() / getItemId() for dynamic dataset changes
//
// Common bug: not implementing getItemId()/containsItem() when the dataset
// changes → stale fragments shown for wrong positions

Fragment Added to Destroyed Activity

// If you hold a reference to FragmentManager and commit after the Activity
// is destroyed (e.g., in an async callback):

viewModel.result.observe(this) { result ->
    // If Activity is finishing/destroyed when this fires:
    if (!isAdded || isDetached) return@observe
    
    // Even with the check, parentFragmentManager may throw:
    // "FragmentManager has been destroyed"
    
    // Defensive pattern:
    if (parentFragmentManager.isDestroyed) return@observe
    
    parentFragmentManager.commit {
        replace(R.id.container, ResultFragment.newInstance(result))
    }
}

// The root cause: FragmentManager.mDestroyed is set to true in
// Activity.onDestroy(). Any commit after that throws.
// isStateSaved checks are separate from isDestroyed checks.

9. Fragment Result API

The modern replacement for setTargetFragment() (deprecated) and onActivityResult():

// Receiving fragment — registers a listener
override fun onCreate(savedInstanceState: Bundle?) {
    super.onCreate(savedInstanceState)
    parentFragmentManager.setFragmentResultListener("requestKey", this) { key, bundle ->
        val result = bundle.getString("resultValue")
        // Handle result — this callback is lifecycle-aware
        // It only fires when this Fragment is at least STARTED
    }
}

// Sending fragment — sets the result
parentFragmentManager.setFragmentResult("requestKey", bundleOf("resultValue" to "data"))

// Key behaviors:
// - Result is stored in FragmentManager until consumed
// - Survives config change (stored in FragmentManager's saved state)
// - Only ONE listener per requestKey per FragmentManager
// - For child → parent communication, use parentFragmentManager
// - For parent → child, use childFragmentManager
// - The listener is automatically removed when the Fragment reaches DESTROYED

11. ComposeView in Fragments

// When hosting Compose inside a Fragment, ViewCompositionStrategy is critical:
override fun onCreateView(inflater: LayoutInflater, container: ViewGroup?, savedInstanceState: Bundle?): View {
    return ComposeView(requireContext()).apply {
        // DEFAULT: DisposeOnDetachedFromWindow — WRONG for Fragments!
        // Fragment view can be detached/reattached (e.g., ViewPager2)

        // CORRECT for Fragments:
        setViewCompositionStrategy(
            ViewCompositionStrategy.DisposeOnViewTreeLifecycleDestroyed
        )
        setContent {
            MyTheme {
                MyScreen(viewModel = viewModel())
            }
        }
    }
}

// Why DisposeOnViewTreeLifecycleDestroyed?
// - Disposes composition when the Fragment's VIEW lifecycle is destroyed
// - NOT when the Fragment itself is destroyed (it may be on back stack)
// - Matches Fragment's dual lifecycle model perfectly
// - Without this, Compose may be disposed prematurely when Fragment
//   is added to back stack, losing all state
Staff-Level Insight — FragmentManagerViewModel

FragmentManagerViewModel is an internal ViewModel that FragmentManager uses to retain non-configuration state across config changes. It stores: (1) child FragmentManager non-config state, (2) child ViewModelStore instances, and (3) retained Fragment instances (legacy setRetainInstance). It's scoped to the parent's ViewModelStore — for the top-level FragmentManager, that's the Activity's ViewModelStore. This is why Fragment ViewModels survive config changes: they're stored in the Fragment's own ViewModelStore, which is held by FragmentManagerViewModel, which is in the Activity's ViewModelStore, which survives via NonConfigurationInstances. Understanding this chain is essential for debugging ViewModel scoping issues.
