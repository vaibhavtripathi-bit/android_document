02 — Fragment Methods Reference

Every callback, every manager method, every transaction operation — what it does, where it breaks, how the API evolved, and why the design is bad

1. Fragment Lifecycle Callbacks — The Dual Lifecycle Problem

A Fragment has two independent lifecycles: the Fragment object lifecycle and the Fragment View lifecycle. Every callback below belongs to one or both. Misunderstanding which lifecycle you're in is the #1 source of Fragment bugs at scale.

  Fragment Object Lifecycle          Fragment View Lifecycle
  ═══════════════════════           ═══════════════════════

  onAttach(context)
       │
  onCreate(savedState)
       │

       │                                  │

onAttach(context: Context)

fun onAttach(context: Context)

Purpose: First callback. The Fragment is associated with its host. getContext(), requireContext(), and requireActivity() become non-null. Use this to grab host-level dependencies or validate that the host implements a required interface.

Where it breaks: Casting context to an interface the host must implement. If the host doesn't implement it, you get a ClassCastException at runtime with no compile-time safety.

// Old pattern — fragile, no compile-time safety
override fun onAttach(context: Context) {
    super.onAttach(context)
    // Crashes at runtime if host doesn't implement the interface
    callback = context as MyCallback
}

// Modern replacement — Fragment Result API or shared ViewModel
// No interface casting needed, decoupled communication
API Evolution — onAttach signature

onAttach(Activity) was the original signature (API 11). Deprecated in API 23 and replaced with onAttach(Context) because Fragments can now be hosted by non-Activity contexts (e.g., FragmentController in custom hosts). The old signature still fires after the new one for backward compatibility, but relying on it means your Fragment can't be hosted outside an Activity.

onCreate(savedInstanceState: Bundle?)

fun onCreate(savedInstanceState: Bundle?)

Purpose: Initialize non-UI state. Restore from savedInstanceState. arguments is available. The View does not exist yet — do not touch any UI here.

Where it breaks: Accessing view or binding here crashes with NPE. Also, savedInstanceState is null on first creation but non-null after config change — code that assumes one or the other fails.

override fun onCreate(savedInstanceState: Bundle?) {
    super.onCreate(savedInstanceState)
    // Safe: arguments, context, non-UI initialization
    userId = requireArguments().getString("userId")!!

    // WRONG: view is null here
    // binding.textView.text = userId  // NPE!

    // Restore non-UI state
    if (savedInstanceState != null) {
        pendingAction = savedInstanceState.getString("pending_action")
    }
}

onCreateView(inflater, container, savedInstanceState)

fun onCreateView(inflater: LayoutInflater, container: ViewGroup?, savedInstanceState: Bundle?): View?

Purpose: Inflate or programmatically create the Fragment's View hierarchy. Return the root View, or null for headless Fragments.

Bug — The attachToRoot Trap

The container parameter is provided so the inflater can read the parent's LayoutParams. You must pass attachToRoot = false:

// WRONG — crashes with "The specified child already has a parent"
return inflater.inflate(R.layout.fragment_main, container, true)

// CORRECT — inflater reads LayoutParams from container but doesn't attach
return inflater.inflate(R.layout.fragment_main, container, false)

When attachToRoot = true, the inflater immediately adds the View to the container. But FragmentManager also tries to add it, causing a duplicate-parent crash. This is a design flaw: the API signature invites you to pass true because the parameter is named container, not parentForLayoutParams.

onViewCreated(view: View, savedInstanceState: Bundle?)

fun onViewCreated(view: View, savedInstanceState: Bundle?)

Purpose: The View exists but hasn't been state-restored yet. Bind views, set up RecyclerView adapters, register observers on viewLifecycleOwner, configure click listeners.

Why here and not onCreateView: onCreateView should only inflate. Putting setup in onCreateView mixes creation with configuration, and the returned View isn't yet attached to the Fragment's internal state. onViewCreated receives the already-assigned view parameter, guaranteeing getView() is non-null.

onViewStateRestored(savedInstanceState: Bundle?)

fun onViewStateRestored(savedInstanceState: Bundle?)

Purpose: All View state has been restored by the framework (EditText text, ScrollView position, CheckBox state). Safe to read restored View values.

Where it breaks: Setting an EditText's text in onViewCreated gets overwritten by the framework's restoration in onViewStateRestored. If you need to override restored state, do it here.

override fun onViewCreated(view: View, savedInstanceState: Bundle?) {
    // WRONG: this text will be overwritten by state restoration
    binding.searchField.setText("default query")
}

override fun onViewStateRestored(savedInstanceState: Bundle?) {
    super.onViewStateRestored(savedInstanceState)
    // CORRECT: override restored state here if needed
    if (shouldResetSearch) {
        binding.searchField.setText("default query")
    }
    // Safe to read restored values
    val restoredText = binding.searchField.text.toString()
}

onStart() / onResume()

fun onStart()  /  fun onResume()

Purpose: onStart = Fragment visible. onResume = Fragment interactive. Start animations in onStart, acquire camera/sensors in onResume.

Where it breaks: In multi-window mode (API 24+), a visible-but-unfocused Fragment is STARTED but not RESUMED. Code that assumes "visible = resumed" breaks. ViewPager2 uses setMaxLifecycle(STARTED) for off-screen pages, so off-screen Fragments never reach RESUMED.

onPause() / onStop()

fun onPause()  /  fun onStop()

Purpose: onPause = losing focus. onStop = no longer visible. Release camera in onPause, stop heavy work in onStop.

Where it breaks: After API 24, onSaveInstanceState is called after onStop (not after onPause as before). Code that commits Fragment transactions in onStop may throw IllegalStateException on older API levels where state is already saved by then.

onDestroyView()

fun onDestroyView()

Purpose: The View hierarchy is being destroyed. The Fragment object may survive (back stack). Clean up View references, cancel view-scoped coroutines, release adapters.

Bug — The ViewBinding Leak

The most common Fragment memory leak in production. When a Fragment is placed on the back stack, onDestroyView fires but the Fragment object stays alive in CREATED state. If you hold a ViewBinding reference as a class field, the entire View tree is retained by the living Fragment:

class MyFragment : Fragment(R.layout.fragment_my) {
    // WRONG — binding survives onDestroyView, leaks the entire View tree
    private var binding: FragmentMyBinding? = null

    override fun onViewCreated(view: View, savedInstanceState: Bundle?) {
        binding = FragmentMyBinding.bind(view)
    }

    // FIX — null the binding when the View is destroyed
    override fun onDestroyView() {
        super.onDestroyView()
        binding = null
    }
}

This also applies to Flow collection: using lifecycleScope (Fragment's scope) instead of viewLifecycleOwner.lifecycleScope means collection continues after the View is destroyed, updating a dead View tree.

onDestroy() / onDetach()

fun onDestroy()  /  fun onDetach()

Purpose: onDestroy = Fragment being permanently removed. onDetach = Fragment disassociated from host. After onDetach, getActivity() returns null.

Bug — getActivity() Returns Null

Any async callback (network response, coroutine completion, Handler post) that fires after onDetach will find getActivity() == null. Calling requireActivity() doesn't fix the problem — it just converts a silent null into an explicit IllegalStateException.

// The classic crash — async callback after detach
apiService.fetchUser(userId) { user ->
    // Fragment may be detached by the time this fires
    requireActivity().runOnUiThread {  // IllegalStateException!
        binding.name.text = user.name  // NPE — binding is null!
    }
}

// Defensive but ugly — null checks everywhere
apiService.fetchUser(userId) { user ->
    activity?.runOnUiThread {
        view?.let { binding.name.text = user.name }
    }
}

// Correct — lifecycle-aware, auto-cancels on destroy
viewLifecycleOwner.lifecycleScope.launch {
    val user = apiService.fetchUser(userId)
    binding.name.text = user.name
}

2. FragmentManager Methods — Evolution of Transaction Management

Transaction Commit Methods

Gotcha — The Fundamental Tension

commit() is async to allow batching multiple transactions in one frame. But async means the transaction hasn't executed when the next line of code runs. commitNow() is sync but disallows addToBackStack() because synchronous execution would interleave with already-queued async commits, making back stack ordering unpredictable. There is no method that is both synchronous AND supports back stack. The workaround — commit() followed by executePendingTransactions() — forces all pending transactions to execute, not just yours.

executePendingTransactions()

fun executePendingTransactions(): Boolean

Forces all pending commit() calls to execute synchronously. Returns true if there were pending transactions. Throws IllegalStateException if called re-entrantly (FragmentManager is already executing). Use sparingly — it defeats the batching optimization that commit() provides.

findFragmentById / findFragmentByTag

fun findFragmentById(@IdRes id: Int): Fragment?

fun findFragmentByTag(tag: String): Fragment?

Searches mAdded list (currently added Fragments). findFragmentById matches the container ID the Fragment was added to. Returns null if not found — always null-check or use the as? safe cast. After super.onCreate() in Activity, recreated Fragments are already findable — check before adding to avoid duplicates.

popBackStack / popBackStackImmediate

fun popBackStack()  /  fun popBackStack(name: String?, flags: Int)

fun popBackStackImmediate(): Boolean

popBackStack() is async (like commit()). popBackStackImmediate() is sync. With a name and POP_BACK_STACK_INCLUSIVE, pops everything up to and including the named entry. popBackStack(null, POP_BACK_STACK_INCLUSIVE) clears the entire back stack.

Where it breaks: Calling popBackStack() followed by commit() in the same frame — the pop hasn't executed yet, so the commit operates on stale state. Use popBackStackImmediate() or chain both in a single executePendingTransactions() call.

Fragment Result API

fun setFragmentResult(requestKey: String, result: Bundle)

fun setFragmentResultListener(requestKey: String, lifecycleOwner: LifecycleOwner, listener: FragmentResultListener)

API Evolution — Fragment Communication

Phase 1 — setTargetFragment (API 11): Created a strong reference between two Fragments. Broken by design: the reference didn't survive process death, the requestCode could collide with Activity's requestCode space, and it coupled Fragments tightly. Deprecated in AndroidX Fragment 1.3.0.

Phase 2 — Shared ViewModel: Two Fragments scoped to the same Activity ViewModel. Works but couples both Fragments to the ViewModel's API and doesn't work across different FragmentManagers (parent ↔ child).

Phase 3 — Fragment Result API (AndroidX 1.3.0): Lifecycle-aware, survives config change (result stored in FragmentManager's saved state), only one listener per key, automatically removed on DESTROYED. Use parentFragmentManager for sibling/parent communication, childFragmentManager for parent→child.

registerFragmentLifecycleCallbacks

fun registerFragmentLifecycleCallbacks(cb: FragmentLifecycleCallbacks, recursive: Boolean)

Observer pattern for Fragment lifecycle events. When recursive = true, also observes child FragmentManagers. Useful for logging, analytics, or framework-level hooks (Hilt uses this internally to inject dependencies). The callback fires after the Fragment's own callback.

// Logging all Fragment lifecycle events — useful for debugging
supportFragmentManager.registerFragmentLifecycleCallbacks(
    object : FragmentManager.FragmentLifecycleCallbacks() {
        override fun onFragmentCreated(fm: FragmentManager, f: Fragment, s: Bundle?) {
            Log.d("FragmentLifecycle", "${f::class.simpleName} created")
        }
        override fun onFragmentResumed(fm: FragmentManager, f: Fragment) {
            Log.d("FragmentLifecycle", "${f::class.simpleName} resumed")
        }
        override fun onFragmentDestroyed(fm: FragmentManager, f: Fragment) {
            Log.d("FragmentLifecycle", "${f::class.simpleName} destroyed")
        }
    },
    true  // recursive — also observes child FragmentManagers
)

3. Fragment Transaction Operations — add vs replace vs hide vs show vs attach vs detach

Lifecycle Impact Comparison

OperationFragment State AfterView Exists?Receives Lifecycle Events?
add()RESUMED (if host is resumed)YesYes
remove()DESTROYED (no back stack) / CREATED (with back stack)NoNo (destroyed) / Partial (back stack)
replace()Old: same as remove(). New: same as add()New: YesNew: Yes
hide()RESUMED (unchanged)Yes (invisible)Yes — no lifecycle change at all
show()RESUMED (unchanged)Yes (visible)Yes — no lifecycle change at all
detach()CREATEDNo (destroyed)Partial (no view callbacks)
attach()RESUMED (if host is resumed)Yes (recreated)Yes (view callbacks re-fire)

add(containerViewId, fragment, tag)

Adds the Fragment to the container. The Fragment goes through the full lifecycle up to the host's current state. Multiple Fragments can coexist in the same container — they stack in Z-order (last added is on top).

// Multiple fragments in the same container — they overlap
supportFragmentManager.commit {
    add(R.id.container, FragmentA())  // bottom layer
    add(R.id.container, FragmentB())  // on top of A
    // Both are RESUMED, both receive touch events (B first)
    // A's views are behind B's — may still be partially visible
}

replace(containerViewId, fragment)

Internally expands to: remove every existing Fragment in that container, then add the new one. It's not an atomic swap — it's a loop + add. With addToBackStack(), popping reverses this: removes the new Fragment and re-adds all the old ones from saved state.

// replace() removes ALL fragments in the container, not just one
// If container has FragmentA and FragmentB:
supportFragmentManager.commit {
    replace(R.id.container, FragmentC())
    // Internally: remove(A) + remove(B) + add(C)
    // With addToBackStack: popping re-adds BOTH A and B
    addToBackStack("to_c")
}

hide(fragment) / show(fragment)

Sets View.GONE / View.VISIBLE. No lifecycle change whatsoever. The Fragment stays RESUMED. onHiddenChanged(boolean) fires as a notification, but it's not a lifecycle callback. Use this when you need instant switching (e.g., bottom navigation tabs) and can afford to keep all Fragments in memory.

// Bottom navigation pattern with hide/show
private val fragments = mapOf(
    R.id.nav_home to HomeFragment(),
    R.id.nav_search to SearchFragment(),
    R.id.nav_profile to ProfileFragment()
)
private var activeFragment: Fragment = fragments.values.first()

fun switchTab(itemId: Int) {
    val target = fragments[itemId] ?: return
    supportFragmentManager.commit {
        hide(activeFragment)
        show(target)
        // Both fragments stay RESUMED — no lifecycle change
        // onHiddenChanged(boolean) fires on both
    }
    activeFragment = target
}
Gotcha — hide/show and LiveData

Because hidden Fragments are still RESUMED, LiveData observers continue to fire. A hidden Fragment updating its UI in response to LiveData is wasted work. There's no built-in way to pause observation for hidden Fragments — you'd need to manually check isHidden in the observer or use setMaxLifecycle(STARTED) instead of hide().

detach(fragment) / attach(fragment)

detach() destroys the View (calls onDestroyView) but keeps the Fragment object in CREATED state. attach() recreates the View (onCreateView → onViewCreated → ...). This is confusing because the method names sound like onAttach/onDetach callbacks, but they operate at a completely different level.

// detach/attach — confusing naming
supportFragmentManager.commit {
    detach(myFragment)
    // Calls: onPause → onStop → onDestroyView
    // Does NOT call: onDestroy, onDetach
    // Fragment.isDetached == true, but onDetach() was NOT called
}

supportFragmentManager.commit {
    attach(myFragment)
    // Calls: onCreateView → onViewCreated → onViewStateRestored → onStart → onResume
    // The Fragment object was alive the whole time — only the View was destroyed/recreated
}
Gotcha — detach() vs onDetach()

The transaction operation detach() and the lifecycle callback onDetach() are completely different things. detach() destroys the View but keeps the Fragment. onDetach() fires when the Fragment is fully removed from the host. The naming collision has caused countless bugs where developers confuse the two. Fragment.isDetached refers to the transaction operation, not the lifecycle state.

addToBackStack(name)

Pushes the transaction onto the back stack so it can be reversed with popBackStack(). The name parameter is optional — passing null means you can't pop to this specific entry by name (only by index or "pop one"). Removed Fragments are kept in CREATED state (View destroyed, object alive) rather than fully destroyed.

setMaxLifecycle(fragment, state)

Caps the Fragment's lifecycle at the given state. The Fragment won't advance past this cap regardless of the host's state. ViewPager2 uses STARTED for off-screen pages (view exists for smooth swiping, but onResume never fires). Internally stores the cap in Fragment.mMaxState; moveToState() computes targetState = min(hostState, mMaxState).

// Manual tab management with setMaxLifecycle — better than hide/show
// because off-screen tabs don't waste resources in RESUMED state
supportFragmentManager.commit {
    setMaxLifecycle(previousTab, Lifecycle.State.STARTED)  // demote
    setMaxLifecycle(newTab, Lifecycle.State.RESUMED)        // promote
}

// ViewPager2 does this internally via FragmentStateAdapter:
// Current page:   setMaxLifecycle(fragment, RESUMED)
// Adjacent pages: setMaxLifecycle(fragment, STARTED)
// Far-off pages:  Fragment destroyed entirely (recycled)

setPrimaryNavigationFragment(fragment)

Designates which Fragment receives back-button events first. Only one per FragmentManager. When the user presses Back, the primary navigation Fragment's childFragmentManager gets first chance to pop its own back stack. NavHostFragment sets itself as primary — this is how Navigation Component intercepts back presses.

Staff Insight — Three Visibility Mechanisms, Zero Compile-Time Safety

Android provides three ways to manage Fragment visibility — add/remove, hide/show, and detach/attach — each with completely different lifecycle implications. There is no type system or lint rule that prevents you from using the wrong one. At Google scale, this leads to entire categories of bugs: teams using hide/show for a ViewPager-like pattern (wasting memory on all pages), or using detach/attach when they meant remove/add (Fragment object leaks). The Navigation Component and Compose navigation both eliminate this choice entirely by managing Fragment/composable lifecycle internally.

4. View & State Access Methods

getViewLifecycleOwner() vs this

A Fragment implements LifecycleOwner twice: this (Fragment object lifecycle) and getViewLifecycleOwner() (View lifecycle). Using the wrong one is the single most common source of Fragment leaks and duplicate-observer bugs.

// WRONG — observer survives onDestroyView, duplicates on back stack return
viewModel.data.observe(this) { updateUI(it) }

// CORRECT — observer dies with the View, re-registered on new View
viewModel.data.observe(viewLifecycleOwner) { updateUI(it) }

// WRONG — Flow collection continues after View is destroyed
lifecycleScope.launch {
    repeatOnLifecycle(Lifecycle.State.STARTED) {
        viewModel.uiState.collect { render(it) }
    }
}

// CORRECT — collection scoped to View lifetime
viewLifecycleOwner.lifecycleScope.launch {
    viewLifecycleOwner.repeatOnLifecycle(Lifecycle.State.STARTED) {
        viewModel.uiState.collect { render(it) }
    }
}

When viewLifecycleOwner is available: Between onCreateView and onDestroyView. Accessing it outside this window throws IllegalStateException. In onCreate or onDestroy, you must use this as the LifecycleOwner (for non-UI work only).

requireX() vs getX()

NullableNon-nullThrowsUse When
getActivity()requireActivity()IllegalStateExceptionrequireActivity() in lifecycle callbacks; getActivity() in async code with null check
getContext()requireContext()IllegalStateExceptionSame pattern as Activity
getView()requireView()IllegalStateExceptionrequireView() in onViewCreated..onDestroyView; getView() elsewhere
getArguments()requireArguments()IllegalStateExceptionrequireArguments() when Fragment contract mandates args

Design principle: requireX() is fail-fast — it converts a subtle null-related bug later into an immediate crash at the point of misuse. Prefer requireX() in lifecycle callbacks where the value is guaranteed non-null. Use getX() with null checks in async callbacks where the Fragment may have been detached.

Arguments Survive Config Change, Constructor Params Don't

// WRONG — userId lost on config change / process death
class UserFragment(private val userId: String) : Fragment()

// CORRECT — arguments are saved/restored by FragmentManager
class UserFragment : Fragment() {
    companion object {
        fun newInstance(userId: String) = UserFragment().apply {
            arguments = bundleOf("userId" to userId)
        }
    }
    private val userId: String by lazy { requireArguments().getString("userId")!! }
}

arguments is written to the Fragment's saved state Bundle during onSaveInstanceState. On recreation, the system creates a new Fragment instance via the no-arg constructor, then restores mArguments before onAttach. Constructor parameters are lost because the system uses reflection: fragmentClass.getConstructor().newInstance().

5. Design Patterns in Fragment

Composite Pattern — Child FragmentManagers

Every Fragment has its own childFragmentManager, creating a tree of FragmentManagers. Lifecycle dispatch is top-down (parent before child); destruction is bottom-up. This is the Composite pattern — each node (Fragment) can contain sub-trees. But the nesting introduces combinatorial complexity: a child Fragment's effective state is min(parentState, ownState, maxLifecycleCap). Bugs in nested Fragments are notoriously hard to reproduce because they depend on the parent's lifecycle timing.

Template Method — Doubled Complexity

Like Activity, Fragment uses the Template Method pattern: subclasses override callbacks (onCreate, onCreateView, etc.) while the framework controls invocation order. But Fragment has more callbacks (13 vs Activity's 7) across two lifecycles. The template is so complex that even experienced developers put code in the wrong callback.

Mediator — FragmentManager as God Object

FragmentManager mediates between Fragments, but it's also responsible for: lifecycle management, transaction execution, back stack management, state save/restore, animation orchestration (SpecialEffectsController), and Fragment instantiation. This violates Single Responsibility and makes FragmentManager one of the most complex classes in AndroidX (~3000 lines in FragmentManager.java). The AndroidX team has been decomposing it (FragmentStateManager, FragmentStore, SpecialEffectsController) but the central Mediator remains.

The "Mini-Activity" Anti-Pattern

Fragments were designed as reusable UI components — modular pieces of an Activity's UI. But they evolved into mini-Activities with their own back stack, lifecycle, state save/restore, permission handling, and result APIs. This scope creep means Fragments carry Activity-level complexity without Activity-level isolation (they share the host's process, window, and configuration). This is widely considered a design mistake and is the primary motivation behind Jetpack Compose's simpler composable model, where UI components are functions with no lifecycle of their own.

State Pattern (Broken)

Fragment has internal states (INITIALIZING → ATTACHED → CREATED → VIEW_CREATED → STARTED → RESUMED) but these were not exposed until FragmentStateManager was introduced internally. You couldn't query "what state am I in?" without checking Lifecycle.State, which doesn't map 1:1 (e.g., there's no Lifecycle.State.VIEW_CREATED). The internal state machine exists but is invisible to consumers — a broken State pattern where the states are implementation details rather than part of the public contract.

Staff Insight — Why Compose Doesn't Have Fragments

Every design pattern above is either broken or over-complicated in Fragment. Compose's answer: composable functions have no lifecycle of their own — they exist when they're in the composition and don't when they're not. There's no dual lifecycle, no FragmentManager god object, no transaction commit timing issues, no back stack state loss. The Composition itself is the "FragmentManager," and recomposition replaces the transaction model. The trade-off is that Compose requires a fundamentally different mental model, and interop with the existing Fragment-based ecosystem (Navigation, ViewPager2, DialogFragment) still requires Fragments under the hood.

6. Known Bugs & Bad Design Decisions

1. The Dual Lifecycle

The problem: Fragment object lifecycle and View lifecycle are independent. When a Fragment goes on the back stack, the View is destroyed but the Fragment lives on. Any reference from the Fragment object to the View (binding, adapter, listener) is a memory leak. Any observation scoped to the Fragment (not the View) duplicates on back stack return.

Partial fix: viewLifecycleOwner (AndroidX) lets you scope observations to the View. But the fundamental complexity remains — developers must constantly reason about which lifecycle they're in.

2. setRetainInstance(true)

The problem: Retained the Fragment object across config changes. The Fragment was detached from the old Activity and re-attached to the new one, skipping onDestroy/onCreate. Caused memory leaks (retained Fragment holding View references from the old Activity), confusion about what's retained vs recreated, and didn't work with Navigation Component.

// The old pattern — deprecated, leak-prone
class DataFragment : Fragment() {
    var expensiveData: LargeDataset? = null

    override fun onCreate(savedInstanceState: Bundle?) {
        super.onCreate(savedInstanceState)
        retainInstance = true  // Fragment survives config change
        // Problem: if this Fragment holds any View reference,
        // the old Activity's entire View tree is leaked
    }
}

// The replacement — ViewModel
class DataViewModel : ViewModel() {
    val expensiveData = MutableLiveData<LargeDataset>()
    // Survives config change via ViewModelStore
    // No View references, no lifecycle confusion
}

Fix: Deprecated in AndroidX Fragment 1.3.0. Replaced by ViewModel, which provides the same config-change survival via ViewModelStore held in NonConfigurationInstances — same mechanism, cleaner separation.

3. setTargetFragment()

The problem: Created a strong reference between two Fragments. Didn't survive process death (the target Fragment's reference was lost on recreation). The requestCode shared the same space as Activity's startActivityForResult codes, causing collisions. Tightly coupled Fragments that should communicate through a shared abstraction.

Fix: Deprecated in AndroidX Fragment 1.3.0. Replaced by Fragment Result API (lifecycle-aware, survives config change, decoupled via string keys).

4. onActivityResult() in Fragments

The problem: Fragment results went through the host Activity. The Activity used bit manipulation on the requestCode: bits 16-31 encoded the Fragment's index in the FragmentManager. Nested Fragments broke this because child FragmentManagers had their own indices. ViewPager Fragments received wrong results because their index changed on page swipes.

// How the old system encoded Fragment index into requestCode:
// requestCode = ((fragmentIndex + 1) << 16) + (userRequestCode & 0xffff)
// This meant: max 65535 request codes, max 65534 fragments
// Nested fragments? Their index was in the CHILD FragmentManager,
// but the Activity only knew about the PARENT FragmentManager's indices
// → child fragments never received their results

// Modern replacement — Activity Result API
val launcher = registerForActivityResult(
    ActivityResultContracts.StartActivityForResult()
) { result ->
    if (result.resultCode == Activity.RESULT_OK) {
        val data = result.data
        // Type-safe, no requestCode, works in nested Fragments
    }
}
launcher.launch(intent)
API Evolution — Activity Results

API 11: Fragment.startActivityForResult() — delegates to Activity with encoded requestCode. AndroidX Activity 1.2.0: Activity Result API with registerForActivityResult() — type-safe contracts, no requestCode, works in Fragments without Activity delegation. The old API is deprecated but not removed because millions of apps depend on it.

5. getActivity() Returning Null

The problem: After onDetach(), getActivity() returns null. Every async callback in a Fragment needs a null check. requireActivity() just makes the crash explicit (throws instead of NPE). There's no lifecycle-aware mechanism that automatically cancels arbitrary callbacks on detach.

Mitigation (not fix): Coroutines with lifecycleScope auto-cancel on destroy. LiveData stops delivering on destroy. But raw callbacks (Retrofit, Handler.post, BroadcastReceiver) still need manual null checks.

6. No-Arg Constructor Requirement

The problem: The system recreates Fragments via reflection with Class.newInstance(). Constructor injection is impossible. The arguments Bundle is the only way to pass data that survives recreation, but it's limited to Bundle-serializable types.

API Evolution — Fragment Instantiation

API 11: No-arg constructor + setArguments(). AndroidX Fragment 1.1.0: FragmentFactory — register a factory on FragmentManager that can create Fragments with constructor parameters. Must be set before super.onCreate() because Fragments may be recreated during super.onCreate(). Hilt: @AndroidEntryPoint generates a superclass that installs a Hilt-aware FragmentFactory automatically, enabling @Inject constructor parameters. Each step reduced boilerplate but added its own complexity layer.

7. commitAllowingStateLoss()

The problem: The name implies safety ("allowing" sounds permissive). In reality, it means "this transaction might be silently dropped on process death." The transaction executes and the UI updates, but if the process is killed before the next onSaveInstanceState(), the transaction isn't in the saved state. On restoration, the Fragment won't be there — the user sees a different UI than what they left.

// The timeline of state loss:
// 1. User navigates to DetailFragment via commitAllowingStateLoss()
// 2. User presses Home → onSaveInstanceState() fires
//    BUT the transaction wasn't in the saved state
// 3. System kills the process
// 4. User returns → Activity restores from saved state
// 5. DetailFragment is GONE — user sees the previous screen
//    No crash, no error, just silently wrong UI

// When it IS acceptable:
supportFragmentManager.commitNowAllowingStateLoss {
    replace(R.id.banner_container, ErrorBannerFragment())
    // Transient UI — if lost on process death, that's fine
}

When it's acceptable: Transient UI that doesn't need to survive process death (error banners, analytics Fragments). Never for navigation the user expects to persist.

8. Three Visibility Mechanisms

The problem: add/remove, hide/show, and detach/attach all manage Fragment visibility with completely different lifecycle implications. No compile-time guidance on which to use. Easy to pick the wrong one and get subtle bugs (hidden Fragment still consuming resources, detached Fragment leaking its object).

9. Nested Fragment Back Stack

The problem: A child FragmentManager has its own back stack that doesn't integrate with the parent's. Pressing Back doesn't automatically pop the child's back stack — you need to wire it up via setPrimaryNavigationFragment(). Without this, the Back press skips the child entirely and pops the parent's back stack or finishes the Activity.

// Parent Fragment hosts child Fragments with their own back stack
class ContainerFragment : Fragment() {
    override fun onViewCreated(view: View, savedInstanceState: Bundle?) {
        childFragmentManager.commit {
            replace(R.id.child_container, ChildA())
        }
    }

    fun navigateToChildB() {
        childFragmentManager.commit {
            replace(R.id.child_container, ChildB())
            addToBackStack("child_b")
        }
    }
    // Problem: pressing Back does NOT pop ChildB → ChildA
    // Instead, it pops the PARENT's back stack or finishes the Activity
    // Fix: setPrimaryNavigationFragment in the parent's transaction,
    // or use OnBackPressedDispatcher:

    override fun onCreate(savedInstanceState: Bundle?) {
        super.onCreate(savedInstanceState)
        requireActivity().onBackPressedDispatcher.addCallback(this) {
            if (childFragmentManager.backStackEntryCount > 0) {
                childFragmentManager.popBackStack()
            } else {
                isEnabled = false
                requireActivity().onBackPressedDispatcher.onBackPressed()
            }
        }
    }
}

Fix: Navigation Component abstracts this away by managing a single logical back stack. OnBackPressedDispatcher (AndroidX Activity 1.0.0) provides a callback-based mechanism, but you still need to register it manually for custom child back stacks.

10. Fragment Class Hierarchy Evolution

API Evolution — The Three Fragments

API 11 (Honeycomb): android.app.Fragment — platform Fragment, tied to OS release cycle. Support Library v4: android.support.v4.app.Fragment — backported to API 4, decoupled from OS updates. AndroidX (2018): androidx.fragment.app.Fragment — same code, new package, semantic versioning. Platform Fragment deprecated in API 28 (Android 9). Migration required changing imports across every Fragment in the app — a mechanical but error-prone process. The platform version received no further updates, so apps that didn't migrate missed years of bug fixes (viewLifecycleOwner, Fragment Result API, FragmentFactory, setMaxLifecycle). Today, using android.app.Fragment is a red flag in code review.

Staff Insight — The Meta-Problem

The 10 issues above aren't independent bugs — they're symptoms of a single architectural decision: making Fragment a lifecycle-aware, state-saving, back-stack-managing, view-owning component that lives inside another lifecycle-aware, state-saving, back-stack-managing, view-owning component (Activity). Every "fix" (viewLifecycleOwner, Fragment Result API, FragmentFactory, ViewModel) adds a new abstraction layer on top of the broken foundation rather than replacing it. This is why Compose navigation doesn't use Fragments at all — it's a ground-up rethink. In an L6 interview, demonstrating that you understand this meta-pattern (accumulated complexity from backward-compatible fixes) shows architectural maturity beyond just knowing the APIs.
