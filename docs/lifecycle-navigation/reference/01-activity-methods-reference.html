01 — Activity Methods Reference

Every important Activity callback: what it does, where it breaks, design patterns, bad design decisions, and how the API evolved over 15 years

1. Lifecycle Callbacks — Methods, Gotchas, Evolution

onCreate(savedInstanceState: Bundle?)

override fun onCreate(savedInstanceState: Bundle?)

Called when the Activity is first created, or re-created after process death / config change. This is where you inflate layout, bind ViewModel, and restore saved state.

When called

TransactionExecutor dispatches a LaunchActivityItem → ActivityThread.handleLaunchActivity() → Activity.performCreate(). The system has already attached the Window, Context, and base Application before this fires.

Where it breaks

  • Cold start bottleneck — heavy work here (DB init, large layout inflation, synchronous network) directly increases Time-to-Initial-Display. Google's threshold: <500ms for cold start.
  • Fragment commit after super.onCreate() — if you commit a Fragment in onCreate() after super.onCreate() restores Fragments from saved state, you get duplicates on config change.
  • savedInstanceState is null on first launch — but also null if the system killed the process and the Activity wasn't in the back stack (no state was saved).
// BAD: heavy init blocks cold start
override fun onCreate(savedInstanceState: Bundle?) {
    super.onCreate(savedInstanceState)
    val db = Room.databaseBuilder(this, AppDb::class.java, "app.db")
        .allowMainThreadQueries() // blocks main thread
        .build()
    db.userDao().getAll() // synchronous query in onCreate
    setContentView(R.layout.activity_main)
}

// FIX: defer heavy work, use App Startup or lazy init
override fun onCreate(savedInstanceState: Bundle?) {
    super.onCreate(savedInstanceState)
    setContentView(R.layout.activity_main)
    lifecycleScope.launch {
        val users = withContext(Dispatchers.IO) {
            database.userDao().getAll()
        }
        adapter.submitList(users)
    }
}

onStart()

override fun onStart()

Activity becomes visible but not yet interactive. Corresponds to Lifecycle.State.STARTED. Called after onCreate() on first launch, or after onRestart() when returning from stopped state.

When called

Activity.performStart() → dispatches Lifecycle.Event.ON_START. The window is being made visible but hasn't received focus yet.

Where it breaks

  • Not safe for UI measurement — the View hierarchy is not yet laid out. view.width returns 0 here.
  • Multi-window confusion — in split-screen, an Activity can be STARTED (visible) but not RESUMED (not focused). Before Android 10's multi-resume, only one Activity was RESUMED at a time.

onResume()

override fun onResume()

Activity is in the foreground and interactive. This is the RESUMED state — the only state where the Activity is fully usable.

When called

Activity.performResume() → ON_RESUME event. Called after onStart() on launch, or after onPause() returns (e.g., dialog dismissed, returning from another Activity).

Where it breaks

  • Not "fully drawn" — onResume() fires before the first frame is rendered. Use onWindowFocusChanged(true) or reportFullyDrawn() for actual visibility.
  • Camera/sensor registration here causes multi-window bugs — in split-screen, the non-focused Activity is STARTED, not RESUMED. If you only register sensors in onResume(), the visible-but-unfocused Activity loses sensor data. Android 10+ multi-resume helps, but you must handle both behaviors.

onPause()

override fun onPause()

Activity is losing focus. Another Activity is coming to the foreground, or a dialog/overlay appeared. The Activity is still partially visible.

When called

Activity.performPause() → ON_PAUSE event. The new Activity's onCreate()/onStart()/onResume() will not run until this method returns.

Where it breaks

  • The 500ms ANR trap — if onPause() takes >500ms, the system shows an ANR dialog. This is a hard limit from ActivityManagerService. Never do synchronous I/O here.
  • Pre-API 28 onSaveInstanceState ordering — before API 28, onSaveInstanceState() was called before onPause(). Code that saved state in onPause() and expected it to override onSaveInstanceState() broke silently.
⚠ Gotcha — onPause blocks the next Activity

The launching Activity's onResume() is gated on the current Activity's onPause() completing. A slow onPause() makes the entire transition feel sluggish. Persist data asynchronously — use lifecycleScope.launch or WorkManager, not synchronous writes.

onStop()

override fun onStop()

Activity is no longer visible. Safe to release heavy resources (camera, GPS, large bitmaps). Corresponds to Lifecycle.State.CREATED (yes, CREATED — there is no STOPPED state in Lifecycle.State).

When called

Activity.performStop() → ON_STOP event. Called when another Activity fully covers this one, or the user presses Home.

Where it breaks

  • Not called on process kill — if the system kills the process while the Activity is in the background, onStop() and onDestroy() are never called. Don't rely on them for critical persistence.
  • Fragment transactions after onStop() — committing a FragmentTransaction after onStop() throws IllegalStateException (state loss). Use commitAllowingStateLoss() as a last resort.

onDestroy()

override fun onDestroy()

Final cleanup. Called when the Activity is finishing (finish() called) or being destroyed by the system (config change, process death).

When called

Activity.performDestroy() → ON_DESTROY event. Check isFinishing to distinguish user-initiated finish from system-initiated destruction.

Where it breaks

  • Not guaranteed — the system can kill the process without calling onDestroy(). Never put critical cleanup here without a fallback.
  • Leaked references — if you registered a callback in onCreate() and unregister in onDestroy(), a process kill leaks the registration until the process dies (which is fine for in-process callbacks, but not for system-level ones like ContentObserver).

onRestart()

override fun onRestart()

Called when a stopped Activity is about to restart. Only called when returning from stopped state — never on first launch or after config change. Followed by onStart().

Where it breaks

Rarely used directly. The main gotcha: developers put refresh logic here, but it doesn't fire after config change (which goes through onCreate() → onStart(), skipping onRestart()). Put refresh logic in onStart() instead.

onSaveInstanceState(outState: Bundle)

override fun onSaveInstanceState(outState: Bundle)

Save transient UI state (scroll position, text input, selection) to a Bundle that survives config change and process death.

When called

API 28+: after onStop(). Pre-API 28: before onStop(), with no guaranteed ordering relative to onPause().

Where it breaks

  • Bundle size limit — the Binder transaction buffer is ~1MB shared across all transactions. Large Bundles cause TransactionTooLargeException. Keep saved state under 50KB.
  • Not called on explicit finish() — if the user presses Back or you call finish(), onSaveInstanceState() is NOT called. The system only saves state when it might need to restore later.
🔄 API Evolution — onSaveInstanceState timing
API 1–27: Called before onStop(), sometimes before onPause(). No guaranteed ordering.
API 28+ (P): Always called after onStop(). This made the lifecycle deterministic — you know the Activity is no longer visible when state is saved.
Impact: Code that assumed onSaveInstanceState ran before onPause and used it to pass data to onPause broke on API 28. Code that committed Fragment transactions in onSaveInstanceState now ran after onStop, causing state-loss exceptions.

onRestoreInstanceState(savedInstanceState: Bundle)

override fun onRestoreInstanceState(savedInstanceState: Bundle)

Called after onStart() with the same Bundle passed to onCreate(). Only called if there is saved state to restore (unlike onCreate which always receives the parameter, possibly null).

Where it breaks

Developers duplicate restoration logic in both onCreate() and onRestoreInstanceState(). Pick one. onCreate() is more common because you often need the state during initial setup. onRestoreInstanceState() is useful when you want to restore state after onStart() has run (e.g., after Fragments are attached).

2. Intent & Result Callbacks — The Evolution Story

onNewIntent(intent: Intent)

override fun onNewIntent(intent: Intent)

Called when a singleTop, singleTask, or singleInstance Activity receives a new Intent instead of being re-created. Also called when FLAG_ACTIVITY_SINGLE_TOP or FLAG_ACTIVITY_CLEAR_TOP routes to an existing instance.

Where it breaks

  • getIntent() returns the OLD intent — you must call setIntent(intent) manually if you want getIntent() to reflect the new one. This is a 15-year-old API wart that was never fixed.
  • Lifecycle ordering — onNewIntent() is called before onResume() but the Activity might be in any state (STARTED or RESUMED depending on the scenario). Don't assume the UI is fully ready.
override fun onNewIntent(intent: Intent) {
    super.onNewIntent(intent)
    setIntent(intent) // without this, getIntent() returns stale data
    handleDeepLink(intent)
}

onActivityResult → registerForActivityResult

// DEPRECATED override fun onActivityResult(requestCode: Int, resultCode: Int, data: Intent?)
// REPLACEMENT val launcher = registerForActivityResult(StartActivityForResult()) { result -> ... }
🔄 API Evolution — Activity Results
API 1: startActivityForResult(intent, requestCode) + onActivityResult(). Request codes are plain integers — collision-prone, no type safety.
Fragment complication: When a Fragment called startActivityForResult(), the Activity used bits 16–31 of the requestCode to encode the Fragment's index in FragmentManager. This left only 16 bits (0–65535) for your actual code. Nested Fragments made the bit-shifting even more fragile.
AndroidX Activity 1.2.0 (2020): ActivityResultContract + registerForActivityResult(). Type-safe, lifecycle-aware, no request codes. The launcher is registered at creation time and can be called from anywhere.
API 33 (Android 13): onActivityResult() formally deprecated.
🐛 Bad Design — Fragment requestCode bit-shifting

When FragmentActivity dispatched results, it used (fragmentIndex + 1) << 16 | (requestCode & 0xffff) to encode the Fragment index into the upper 16 bits. This meant: (1) you couldn't use request codes > 65535, (2) having >65534 Fragments was undefined, (3) ViewPager Fragments received results for the wrong Fragment because the index shifted when pages were destroyed/recreated. The Activity Result API eliminated this entirely by using a registry keyed on the contract, not an integer.

// OLD: collision-prone, not lifecycle-safe
class OldActivity : AppCompatActivity() {
    fun pickImage() {
        startActivityForResult(Intent(Intent.ACTION_PICK).apply {
            type = "image/*"
        }, REQUEST_PICK_IMAGE) // magic number
    }

    override fun onActivityResult(requestCode: Int, resultCode: Int, data: Intent?) {
        super.onActivityResult(requestCode, resultCode, data)
        if (requestCode == REQUEST_PICK_IMAGE && resultCode == RESULT_OK) {
            handleImage(data?.data)
        }
    }
    companion object { const val REQUEST_PICK_IMAGE = 1001 }
}

// NEW: type-safe, lifecycle-aware
class NewActivity : AppCompatActivity() {
    private val pickImage = registerForActivityResult(
        ActivityResultContracts.GetContent()
    ) { uri: Uri? ->
        uri?.let { handleImage(it) }
    }

    fun onPickClicked() = pickImage.launch("image/*")
}

3. Configuration & Window Callbacks

onConfigurationChanged(newConfig: Configuration)

override fun onConfigurationChanged(newConfig: Configuration)

Called when a configuration change occurs and you've declared android:configChanges in the manifest for that change type. The Activity is NOT destroyed — you handle the change yourself.

Where it breaks

  • Missing config types — if you declare configChanges="orientation" but not screenSize, the Activity still gets destroyed on rotation (since API 13, orientation changes also change screen size).
  • Compose doesn't need this — Compose recomposes on config change automatically. Declaring configChanges to avoid recreation is a View-system pattern that's unnecessary with Compose (and actually harmful because it prevents proper resource re-resolution).

onMultiWindowModeChanged(isInMultiWindowMode: Boolean, newConfig: Configuration)

override fun onMultiWindowModeChanged(isInMultiWindowMode: Boolean, newConfig: Configuration)

Called when the Activity enters or exits multi-window (split-screen or freeform). Added in API 24.

Where it breaks

On API 24–25, entering multi-window destroyed and recreated the Activity (triggering a full lifecycle restart). On API 26+, the Activity receives this callback without destruction if configChanges includes screenSize|screenLayout|smallestScreenSize. Many apps broke on Nougat because they didn't handle the destruction path.

onPictureInPictureModeChanged(isInPiPMode: Boolean, newConfig: Configuration)

override fun onPictureInPictureModeChanged(isInPiPMode: Boolean, newConfig: Configuration)

Called when the Activity enters or exits PiP. In PiP, the Activity is in PAUSED state (pre-Android 10) or RESUMED (Android 10+ with multi-resume).

Where it breaks

Pre-Android 10: the Activity was PAUSED in PiP, so onPause() paused video playback. Google had to add a special isInPictureInPictureMode check to keep playback running. Android 10's multi-resume fixed this by keeping PiP Activities RESUMED.

onTopResumedActivityChanged(isTopResumedActivity: Boolean)

override fun onTopResumedActivityChanged(isTopResumedActivity: Boolean)

Added in API 29 (Android 10) for multi-resume. Indicates whether this Activity is the top resumed Activity. In multi-window, multiple Activities can be RESUMED, but only one is "top resumed" (has window focus).

When useful

Exclusive resources like the camera should be acquired only by the top-resumed Activity. Use this callback instead of onResume()/onPause() for camera access in multi-window scenarios.

onWindowFocusChanged(hasFocus: Boolean)

override fun onWindowFocusChanged(hasFocus: Boolean)

Called when the Activity's window gains or loses focus. This is the real "fully visible and interactive" signal — it fires after the first layout pass, after the window is drawn and composited.

When useful

  • Measuring views — view.width/view.height are valid here (unlike in onResume()).
  • Starting animations that should play when the user can actually see them.
  • Detecting when a dialog or system window steals focus.

Where it breaks

Called with false when a dialog appears (even your own AlertDialog), when the notification shade is pulled down, or when a permission dialog shows. Don't use focus loss as a signal to pause critical work — it's too noisy.

onContentChanged()

override fun onContentChanged()

Called after setContentView() or addContentView(). Rarely used directly — it's a hook from the View system era for frameworks like Butter Knife that needed to know when the view hierarchy changed. Compose makes this irrelevant.

onAttachedToWindow() / onDetachedFromWindow()

override fun onAttachedToWindow()

Called when the Activity's window is attached to the WindowManager. Useful for setting window flags that must be applied after attachment (e.g., FLAG_DISMISS_KEYGUARD). Rarely needed in modern code.

onUserInteraction()

override fun onUserInteraction()

Called on every user touch, key press, or trackball event dispatched to the Activity. Fires before the event reaches any View. Useful for resetting inactivity timers (e.g., auto-logout in banking apps).

onUserLeaveHint()

override fun onUserLeaveHint()

Called when the user navigates away from the Activity by choice (Home press, task switch) — but NOT when another Activity launches in front. This is the canonical entry point for PiP.

override fun onUserLeaveHint() {
    super.onUserLeaveHint()
    if (shouldEnterPiP()) {
        enterPictureInPictureMode(pipParams)
    }
}

// API 31+ auto-enter PiP replaces this pattern:
setPictureInPictureParams(
    PictureInPictureParams.Builder()
        .setAutoEnterEnabled(true)
        .build()
)

4. Permission & Back Navigation — Evolution Stories

onRequestPermissionsResult → registerForActivityResult

// DEPRECATED override fun onRequestPermissionsResult(requestCode: Int, permissions: Array<String>, grantResults: IntArray)
// REPLACEMENT val permLauncher = registerForActivityResult(RequestPermission()) { granted -> ... }
🔄 API Evolution — Runtime Permissions
API 1–22: All permissions granted at install time. No runtime API needed.
API 23 (M): Runtime permissions introduced. requestPermissions() + onRequestPermissionsResult() with raw integer request codes. Same collision problems as onActivityResult().
AndroidX Activity 1.2.0: RequestPermission() and RequestMultiplePermissions() contracts. Type-safe, lifecycle-aware, no request codes.
API 33 (Android 13): onRequestPermissionsResult() formally deprecated.
API 34 (Android 14): Partial photo/video access. READ_MEDIA_VISUAL_USER_SELECTED permission — user can grant access to specific photos, not all. The old binary grant/deny model doesn't fit anymore.
// OLD: raw request codes
override fun onRequestPermissionsResult(
    requestCode: Int, permissions: Array<String>, grantResults: IntArray
) {
    if (requestCode == REQ_CAMERA && grantResults[0] == PERMISSION_GRANTED) {
        openCamera()
    }
}

// NEW: type-safe contract
private val cameraPermission = registerForActivityResult(
    ActivityResultContracts.RequestPermission()
) { granted ->
    if (granted) openCamera()
    else showRationale()
}

fun onCameraClicked() = cameraPermission.launch(Manifest.permission.CAMERA)

onBackPressed → OnBackPressedDispatcher → Predictive Back

// DEPRECATED (API 33) override fun onBackPressed()
// REPLACEMENT onBackPressedDispatcher.addCallback(this) { /* handle back */ }
🔄 API Evolution — Back Navigation
API 1: onBackPressed() — synchronous, single override point. Subclasses fight over who handles back.
AndroidX Activity 1.1.0 (2019): OnBackPressedDispatcher with OnBackPressedCallback. Multiple callbacks in priority order, lifecycle-aware. Fragments and NavController register their own callbacks.
API 33 (Android 13): OnBackInvokedCallback at the platform level. onBackPressed() deprecated. The system needs to know ahead of time whether the app will handle back, to support predictive back animations.
API 34 (Android 14): Predictive back animations enabled for system transitions (back-to-home). Apps must opt in via enableOnBackInvokedCallback="true" in manifest.
API 35 (Android 15): Predictive back becomes mandatory. Apps that override onBackPressed() without the dispatcher break the gesture preview animation.
🐛 Bad Design — onBackPressed() is fundamentally incompatible with predictive back

onBackPressed() is synchronous and runs after the back gesture completes. Predictive back needs to know before the gesture whether the app will handle it (to show the correct animation preview). A synchronous callback can't answer a question about the future. This is why the entire API had to be replaced — it wasn't fixable, only replaceable. The OnBackPressedDispatcher solves this by registering callbacks with isEnabled flags that the system queries during the gesture.

// OLD: synchronous, can't support gesture preview
override fun onBackPressed() {
    if (viewModel.hasUnsavedChanges) {
        showDiscardDialog()
    } else {
        super.onBackPressed()
    }
}

// NEW: lifecycle-aware, supports predictive back
val callback = object : OnBackPressedCallback(enabled = true) {
    override fun handleOnBackPressed() {
        if (viewModel.hasUnsavedChanges) {
            showDiscardDialog()
        } else {
            isEnabled = false
            onBackPressedDispatcher.onBackPressed()
        }
    }
}
onBackPressedDispatcher.addCallback(this, callback)

// Observe changes to enable/disable the callback dynamically
viewModel.hasUnsavedChanges.observe(this) { hasChanges ->
    callback.isEnabled = hasChanges
}

5. Design Patterns in Activity

Template Method Pattern

The lifecycle callbacks are a textbook Template Method: the framework (ActivityThread) controls the algorithm (launch → start → resume → pause → stop → destroy), and you override individual steps. The problem: callback ordering matters but is not enforced at compile time. Nothing prevents you from calling finish() in onCreate() before setContentView(), or committing a Fragment transaction in onSaveInstanceState(). The compiler can't help — these are runtime ordering constraints.

God Object Anti-Pattern

🎯 Staff Insight — Activity is a God Object

A single ComponentActivity is simultaneously: Context, LifecycleOwner, ViewModelStoreOwner, SavedStateRegistryOwner, OnBackPressedDispatcherOwner, ActivityResultRegistryOwner, MenuHost, ContextAware, and FullyDrawnReporterOwner. This violates SRP so thoroughly that the entire Jetpack library exists to decompose it. ViewModel extracts state management. SavedStateRegistry extracts persistence. OnBackPressedDispatcher extracts navigation. Each Jetpack component peels off one responsibility from the God Object. In an L6 interview, framing Activity's design problems and Jetpack's solutions as "decomposing a God Object" shows architectural maturity.

Observer Pattern (Bolted On)

Activity didn't originally support the Observer pattern. LifecycleOwner was retrofitted via ComponentActivity in AndroidX. The class hierarchy evolution tells the story:

🔄 API Evolution — Activity class hierarchy
Pre-AndroidX: Activity → FragmentActivity → AppCompatActivity. Lifecycle observation required manual registration.
AndroidX 1.0: ComponentActivity inserted as the base class. Hierarchy became: Activity → ComponentActivity → FragmentActivity → AppCompatActivity. ComponentActivity implements LifecycleOwner, ViewModelStoreOwner, SavedStateRegistryOwner.
Impact: Code using reflection to access Activity internals (e.g., activity.javaClass.superclass expecting FragmentActivity) broke when ComponentActivity was inserted. Libraries that checked instanceof FragmentActivity still worked, but the class hierarchy assumption changed.

State Machine (Implicit → Explicit)

The original Activity had callbacks without explicit states. Lifecycle.State made the state machine explicit — but the mapping isn't clean:

Lifecycle.StateActivity callbacks in this stateOddity
INITIALIZEDBefore onCreate()Exists only briefly
CREATEDAfter onCreate() / after onStop()Two different lifecycle positions map to the same state
STARTEDAfter onStart() / after onPause()No PAUSED state — STARTED covers both "started-not-yet-resumed" and "paused"
RESUMEDAfter onResume()Clean mapping
DESTROYEDAfter onDestroy()Terminal state
🎯 Staff Insight — Why there's no PAUSED state

Lifecycle.State was designed as a monotonically increasing then decreasing state machine. States go up (INITIALIZED → CREATED → STARTED → RESUMED) and back down the same path. A separate PAUSED state would create an asymmetry: going up you'd skip it (onCreate → onStart → onResume), but going down you'd hit it (onPause → PAUSED → onStop → CREATED). The Jetpack team chose simplicity over completeness. The cost: you can't observe "paused" as a distinct state with Lifecycle.State — you have to observe ON_PAUSE as an event instead.

Deep Dive: singleInstance + onActivityResult

When you launch an Activity with launchMode="singleInstance" using startActivityForResult(), the result is delivered immediately as RESULT_CANCELED — before the target Activity even launches. This happens because singleInstance Activities run in their own task, and the result mechanism requires same-task communication. The system can't defer the result across tasks, so it fails fast. This was never "fixed" because it's architecturally correct — but it surprises everyone.

Deep Dive: recreate() inconsistency

🐛 Bad Design — recreate() lies about config change

Activity.recreate() destroys and recreates the Activity, preserving NonConfigurationInstances (which includes ViewModel). But isChangingConfigurations() returns false during recreate(). This means: (1) ViewModel survives (good), (2) code that checks isChangingConfigurations() to decide whether to clean up resources will clean up even though ViewModel is retained (bad), (3) onRetainNonConfigurationInstance() IS called (ViewModel is retained), but the Activity doesn't "know" it's a config change. Libraries that branch on isChangingConfigurations() behave differently for recreate() vs actual config change.

Deep Dive: finish() in onCreate() — the trampoline

The trampoline pattern: Activity A launches, checks a condition in onCreate(), starts Activity B, and calls finish(). The Activity goes through the full lifecycle (create → start → resume → pause → stop → destroy) even though it was never meaningfully visible. ViewModel is created in onCreate() and cleared in onDestroy() — wasteful.

// Trampoline pattern — now restricted on Android 12+
class SplashActivity : AppCompatActivity() {
    override fun onCreate(savedInstanceState: Bundle?) {
        super.onCreate(savedInstanceState)
        // Don't setContentView — we're just routing
        if (isLoggedIn()) {
            startActivity(Intent(this, HomeActivity::class.java))
        } else {
            startActivity(Intent(this, LoginActivity::class.java))
        }
        finish()
    }
}

// Android 12+ blocks background trampolines:
// If SplashActivity is started from a background context (e.g., notification),
// the trampoline finish() may prevent the target Activity from appearing.
// Fix: use PendingIntent pointing directly to the target Activity.
⚠ Android 12 Trampoline Restriction

Starting Android 12 (API 31), Activities that call finish() in onCreate() when launched from a notification trampoline are blocked. The system prevents the target Activity from launching if the trampoline Activity finishes before it's fully displayed. This broke many apps that used a routing Activity to decide where to navigate from a notification. The fix: use a PendingIntent that targets the final destination directly, or use conditional navigation in the destination Activity itself.
