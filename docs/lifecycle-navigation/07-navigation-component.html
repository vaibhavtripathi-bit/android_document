07 — Navigation Component (Fragment-Based)

NavController internals, NavBackStackEntry lifecycle ownership, Navigator abstraction, multiple back stacks, deep link handling, ViewModel scoping, and staff-level pitfalls

1. Navigation Architecture

The Navigation Component is an opinionated framework that manages Fragment transactions, back stack, argument passing, deep links, and ViewModel scoping through a single NavController. Understanding the layered architecture is essential — the library is not a thin wrapper over FragmentManager; it introduces its own back stack model that delegates to FragmentManager under the hood.

NavBackStackEntry — Triple Owner

Each NavBackStackEntry is a self-contained lifecycle scope. This is the key architectural insight — it decouples the destination's lifecycle from the Fragment's lifecycle.

// NavBackStackEntry implements:
// 1. LifecycleOwner    → its Lifecycle tracks CREATED → STARTED → RESUMED → DESTROYED
// 2. ViewModelStoreOwner → ViewModels scoped to this entry survive config changes
// 3. SavedStateRegistryOwner → SavedStateHandle persists across process death

// The lifecycle of a NavBackStackEntry:
//   navigate(A → B):
//     Entry A: RESUMED → STARTED (still on back stack, but not visible)
//     Entry B: CREATED → STARTED → RESUMED
//
//   popBackStack() (B popped):
//     Entry B: STARTED → CREATED → DESTROYED (ViewModelStore cleared)
//     Entry A: STARTED → RESUMED
//
// This is distinct from Fragment lifecycle:
//   Fragment for A may have its VIEW destroyed (onDestroyView) when B is shown,
//   but the NavBackStackEntry for A stays in STARTED state.
//   The Fragment INSTANCE stays alive if FragmentNavigator uses replace + addToBackStack.

4. NavController Internals

This section covers what happens inside NavController when you call navigate() and popBackStack(). Understanding these internals is what separates L6 answers from surface-level ones.

navigate() — Step by Step

// NavController.navigate(resId, args, navOptions, navigatorExtras)
//
// Step 1: Resolve the destination
//   → findDestination(resId) walks the NavGraph tree
//   → If resId is an action ID, resolve the action's destination
//   → If destination is a NavGraph, resolve to its startDestination
//
// Step 2: Apply NavOptions (popUpTo, singleTop, etc.)
//   → If popUpTo is set:
//     → Pop entries from the back stack up to (and optionally including) the target
//     → Each popped NavBackStackEntry transitions to DESTROYED
//     → Its ViewModelStore is cleared
//   → If launchSingleTop and destination is already at top:
//     → Don't create a new entry; update the existing entry's arguments
//     → (Pre-2.4: arguments were NOT updated — see corner cases)
//
// Step 3: Create NavBackStackEntry
//   → new NavBackStackEntry(destination, args, hostLifecycleState, viewModelStore)
//   → Entry starts in CREATED state
//
// Step 4: Delegate to Navigator
//   → navigator.navigate(entry, navOptions, navigatorExtras)
//   → FragmentNavigator: creates FragmentTransaction, replace(), addToBackStack()
//   → DialogFragmentNavigator: calls dialogFragment.show()
//   → ActivityNavigator: calls startActivity()
//
// Step 5: Update back stack
//   → Push the new NavBackStackEntry onto NavController's internal back stack
//   → Transition entry lifecycle: CREATED → STARTED → RESUMED
//   → Previous entry: RESUMED → STARTED (if still on back stack)
//
// Step 6: Notify listeners
//   → OnDestinationChangedListener callbacks fire
//   → NavigationUI updates toolbar title, bottom nav selection, etc.

popBackStack() — Step by Step

// NavController.popBackStack()
//
// Step 1: Get the current back stack entry (top of stack)
// Step 2: Determine the Navigator that owns this destination
// Step 3: Call navigator.popBackStack(entry, savedState)
//   → FragmentNavigator: FragmentManager.popBackStack()
//   → DialogFragmentNavigator: dialogFragment.dismiss()
// Step 4: Transition the popped entry:
//   → RESUMED → STARTED → CREATED → DESTROYED
//   → ViewModelStore.clear() — all ViewModels for this entry are cleared
// Step 5: Remove entry from NavController's back stack
// Step 6: The entry below becomes the new top:
//   → STARTED → RESUMED
// Step 7: Notify OnDestinationChangedListener

The Navigator Abstraction

// Navigator<D : NavDestination> is the SPI that decouples NavController
// from the actual navigation mechanism.

// Key methods:
abstract class Navigator<D : NavDestination> {
    // Create a new destination instance (used by XML inflation)
    abstract fun createDestination(): D

    // Perform the navigation — return the NavDestination if successful
    abstract fun navigate(
        entries: List<NavBackStackEntry>,
        navOptions: NavOptions?,
        navigatorExtras: Extras?
    )

    // Pop the given entry from this navigator's back stack
    abstract fun popBackStack(popUpTo: NavBackStackEntry, savedState: Boolean)
}

// FragmentNavigator.navigate() internally does:
//   val ft = fragmentManager.beginTransaction()
//   ft.setReorderingAllowed(true)
//   ft.replace(containerId, fragment)  // NOT add — replace
//   ft.setPrimaryNavigationFragment(fragment)
//   ft.addToBackStack(entry.id)
//   ft.commit()
//
// setPrimaryNavigationFragment() is critical — it ensures the child
// FragmentManager handles back button presses before the parent.

NavOptions

OptionEffectGotcha
popUpToPop all entries above the specified destination before navigatingIf the destination isn't on the stack, pops everything
inclusiveAlso pop the popUpTo destination itselfIf inclusive pops the start destination, the graph is empty — next back press exits
launchSingleTopIf destination is already at the top, don't create a new instancePre-2.4: didn't update arguments. Post-2.4: creates a new entry but reuses the ViewModel
restoreStateRestore previously saved back stack state for the destinationOnly works with saveState = true on the corresponding popUpTo
saveStateSave the back stack state when popping (for multiple back stacks)Saved state is keyed by destination ID — only one saved state per destination

5. Fragment Back Stack in Navigation

How Navigation Uses FragmentTransaction

// NavController maintains its OWN back stack (List<NavBackStackEntry>)
// FragmentNavigator ALSO adds entries to FragmentManager's back stack
//
// These two back stacks are kept in sync:
//
// NavController back stack:     [Entry:Home] [Entry:Detail] [Entry:Settings]
// FragmentManager back stack:   [Home→Detail] [Detail→Settings]
//                                (FragmentTransaction records)
//
// When NavController.popBackStack() is called:
//   1. NavController pops its entry
//   2. FragmentNavigator calls fragmentManager.popBackStack()
//   3. Both stacks stay in sync
//
// The Fragment lifecycle during navigation:
//   navigate(Home → Detail):
//     Home Fragment: onPause → onStop → onDestroyView (view destroyed!)
//     Detail Fragment: onAttach → onCreate → onCreateView → onStart → onResume
//
//   popBackStack() (back to Home):
//     Detail Fragment: onPause → onStop → onDestroyView → onDestroy → onDetach
//     Home Fragment: onCreateView → onStart → onResume (view RE-created!)
//
// Key insight: the Home Fragment INSTANCE stays alive (it's on the back stack),
// but its VIEW is destroyed. This is why you must inflate views in
// onCreateView() and not cache view references across view lifecycles.
Staff-Level Insight — setPrimaryNavigationFragment and Back Press Dispatch

FragmentNavigator calls setPrimaryNavigationFragment() on every transaction. This is the mechanism that makes the system Back button work correctly with nested navigation. When the user presses Back, FragmentManager.popBackStackImmediate() is called on the primary navigation fragment's child FragmentManager first. If the child FM has entries to pop, it handles the back press. Only when the child FM's back stack is empty does the parent FM handle it. Without setPrimaryNavigationFragment(), pressing Back would pop from the wrong FragmentManager, causing the entire NavHostFragment to be removed instead of navigating within it. This is also why setReorderingAllowed(true) is always set — it enables the framework to optimize fragment lifecycle transitions and is required for setPrimaryNavigationFragment() to work correctly with postponed transitions.

Multiple Back Stacks (Navigation 2.4+)

// The killer feature for bottom navigation: each tab maintains its own back stack.
//
// Before 2.4: switching tabs destroyed the previous tab's back stack.
// After 2.4: FragmentManager.saveBackStack() / restoreBackStack() enable
//            saving and restoring entire back stack segments.
//
// How it works under the hood:
//
// Tab A stack: [Home] → [Detail] → [SubDetail]
// Tab B stack: [Search] → [Results]
//
// User switches from Tab A to Tab B:
//   1. NavController pops Tab A's entries with saveState = true
//   2. FragmentManager.saveBackStack("tab_a") saves:
//      - Fragment instances (as SavedState)
//      - FragmentTransaction records
//      - ViewModel states (via SavedStateHandle)
//   3. NavController navigates to Tab B's start destination with restoreState = true
//   4. FragmentManager.restoreBackStack("tab_b") restores the saved stack
//
// User switches back to Tab A:
//   → Tab A's full stack [Home → Detail → SubDetail] is restored
//   → Fragment views are recreated, ViewModels are restored

// The NavigationUI helper does this automatically:
NavigationUI.setupWithNavController(bottomNav, navController)

// Equivalent manual implementation:
bottomNav.setOnItemSelectedListener { item ->
    NavigationUI.onNavDestinationSelected(item, navController)
}
// This internally calls navigate() with:
//   popUpTo(graph.startDestinationId) { saveState = true }
//   launchSingleTop = true
//   restoreState = true

6. ViewModel Scoping with Navigation

Destination-Scoped ViewModel

// Default: ViewModel scoped to the current NavBackStackEntry
class DetailFragment : Fragment() {
    // This ViewModel lives as long as DetailFragment's NavBackStackEntry
    // is on the back stack. Cleared when the entry is popped.
    private val viewModel: DetailViewModel by viewModels()
}

// Under the hood:
// by viewModels() uses the Fragment as the ViewModelStoreOwner.
// But the Fragment's ViewModelStore is actually backed by the
// NavBackStackEntry's ViewModelStore (when using Navigation).
//
// Lifecycle:
//   navigate(Home → Detail): DetailViewModel created
//   navigate(Detail → SubDetail): DetailViewModel stays alive
//   popBackStack() to Detail: DetailViewModel still alive
//   popBackStack() to Home: DetailViewModel.onCleared() called

Graph-Scoped ViewModel (Shared Across Destinations)

// ViewModel scoped to a nested navigation graph — shared by all
// destinations within that graph.

class LoginFragment : Fragment() {
    // Scoped to the auth_graph — shared with RegisterFragment
    private val authViewModel: AuthViewModel by navGraphViewModels(R.id.auth_graph)
}

class RegisterFragment : Fragment() {
    // Same ViewModel instance as LoginFragment's authViewModel
    private val authViewModel: AuthViewModel by navGraphViewModels(R.id.auth_graph)
}

// How it works:
// navGraphViewModels() finds the NavBackStackEntry for the specified graph ID
// and uses IT as the ViewModelStoreOwner.
//
// NavBackStackEntry for auth_graph exists as long as any destination
// within auth_graph is on the back stack.
//
// When the last destination in auth_graph is popped, the graph's
// NavBackStackEntry is also popped → AuthViewModel.onCleared()

// Common use cases:
// 1. Multi-step forms (wizard): share form data across steps
// 2. Auth flows: share credentials/tokens across login/register/verify
// 3. Checkout flows: share cart data across address/payment/confirm

How NavBackStackEntry Acts as ViewModelStoreOwner

// NavBackStackEntry implements ViewModelStoreOwner:
class NavBackStackEntry : LifecycleOwner, ViewModelStoreOwner,
    SavedStateRegistryOwner {

    // Each entry has its own ViewModelStore
    override val viewModelStore: ViewModelStore

    // The ViewModelStore is created lazily and stored in
    // NavControllerViewModel — a ViewModel scoped to the Activity.
    // This is how ViewModel state survives config changes:
    //
    // Activity (ViewModelStoreOwner)
    //   └── NavControllerViewModel (survives config change)
    //         ├── Entry A → ViewModelStore { DetailViewModel }
    //         ├── Entry B → ViewModelStore { SettingsViewModel }
    //         └── Entry auth_graph → ViewModelStore { AuthViewModel }
    //
    // On config change:
    //   Activity is destroyed and recreated
    //   NavControllerViewModel survives (it's in Activity's ViewModelStore)
    //   All NavBackStackEntry ViewModelStores survive
    //   NavController restores its back stack from SavedState
    //   Each restored entry reconnects to its ViewModelStore
}

// ViewModel cleared when:
// 1. NavBackStackEntry is popped from the back stack (user navigates back)
// 2. NavBackStackEntry is popped via popUpTo with inclusive=true
// 3. NavController is destroyed (Activity finishes permanently)

7. Deep Links

Implicit Deep Links

// Implicit deep links match incoming Intents by URI, action, or MIME type.
// Defined in the nav graph, they generate <intent-filter> entries in the manifest.

// URI matching supports:
// - Exact paths:    "myapp://detail/42"
// - Placeholders:   "myapp://detail/{itemId}" → extracted as argument
// - Wildcards:      "myapp://detail/.*" → regex matching
// - Query params:   "myapp://search?query={query}" → extracted as argument
// - Optional params: query parameters not in the URI are optional

// Matching priority (when multiple destinations match):
// 1. Exact URI match
// 2. URI with fewer placeholders
// 3. URI + action + MIME type (most specific wins)
// 4. If still tied, the deepest destination in the graph wins

Explicit Deep Links — NavDeepLinkBuilder

// NavDeepLinkBuilder creates a PendingIntent that navigates to a specific
// destination with a synthetic back stack — used for notifications.

val pendingIntent = NavDeepLinkBuilder(context)
    .setGraph(R.navigation.main_graph)
    .setDestination(R.id.detailFragment)
    .setArguments(bundleOf("itemId" to 42L))
    .createPendingIntent()

// What happens when the PendingIntent fires:
// 1. If the app is not running: Activity is launched fresh
// 2. NavController processes the deep link Intent
// 3. A synthetic back stack is built:
//    [startDestination] → [intermediate destinations] → [detailFragment]
//    The user can press Back through the synthetic stack naturally
//
// The synthetic back stack follows the graph structure:
//   main_graph (startDest = home) → detailFragment
//   Back stack created: [Home] → [Detail]

// For nested graphs:
val pendingIntent = NavDeepLinkBuilder(context)
    .setGraph(R.navigation.main_graph)
    .setDestination(R.id.registerFragment) // inside auth_graph
    .createPendingIntent()
// Back stack: [Home] → [Login] → [Register]
// (Login is auth_graph's startDestination)

handleDeepLink() — Intent Processing

// NavController.handleDeepLink(intent) is called automatically by
// NavHostFragment in its onCreate(). The flow:
//
// 1. Extract the deep link URI from the Intent
// 2. Match against all NavDeepLink entries in the graph
// 3. If matched:
//    a. Pop the entire current back stack
//    b. Build the synthetic back stack from the graph root to the destination
//    c. Navigate to each destination in sequence
//    d. The final destination receives the deep link arguments
//
// 4. If the Activity was already running (onNewIntent):
//    a. The existing back stack is cleared
//    b. New synthetic stack is built
//    c. This can be surprising — deep linking destroys existing navigation state
//
// Handling deep links in an existing Activity:
override fun onNewIntent(intent: Intent) {
    super.onNewIntent(intent)
    navController.handleDeepLink(intent)
}

// Process death and deep links:
// - NavController saves its state in onSaveInstanceState()
// - On restore, it rebuilds the back stack from saved state
// - If the app was deep-linked and then process-killed, the restored
//   state includes the synthetic back stack
// - SavedStateHandle in ViewModels preserves deep link arguments
Warning — Deep Link Clears Existing Back Stack

When handleDeepLink() processes an incoming Intent, it clears the entire existing back stack and builds a new synthetic one. If the user was mid-flow (e.g., filling out a form), that state is lost. For apps that need to preserve existing state on deep link, you must intercept the Intent before it reaches NavController and handle it manually — for example, by navigating to the deep link destination without clearing the stack.

8. Staff-Level Corner Cases

Double-Navigation Crash

// The most common Navigation crash: tapping a button twice quickly.
//
// What happens:
//   1. First tap: navigate(HomeFragmentDirections.actionHomeToDetail(42))
//      → NavController is now at DetailFragment
//   2. Second tap (before Fragment transaction commits):
//      → navigate(HomeFragmentDirections.actionHomeToDetail(43))
//      → NavController tries to find action_home_to_detail from current destination
//      → Current destination is now DetailFragment, which has no such action
//      → IllegalArgumentException: "navigation destination is not a direct child
//         of the current NavGraph"

// Fix 1: Check current destination before navigating
fun navigateSafe(directions: NavDirections) {
    val currentDest = findNavController().currentDestination
    if (currentDest?.id == R.id.homeFragment) {
        findNavController().navigate(directions)
    }
}

// Fix 2: Use navigate with action ID and catch the exception
// (Not recommended — masks real bugs)

// Fix 3: Debounce clicks at the View level
button.setOnClickListener {
    it.isEnabled = false
    findNavController().navigate(directions)
    // Re-enable after a delay or in onResume
}

launchSingleTop with Arguments

// Question: If destination A is at the top and you navigate to A with
// launchSingleTop=true and new arguments, are the arguments updated?
//
// Answer: It depends on the Navigation version.
//
// Pre-2.4: Arguments were NOT updated. The existing entry kept its old args.
//   This was a known pain point — no way to "refresh" a singleTop destination.
//
// Post-2.4: A NEW NavBackStackEntry is created with the new arguments,
//   but the ViewModel is REUSED from the old entry.
//   The old entry is removed, the new entry takes its place.
//   This means:
//     - Fragment.arguments will have the new values
//     - ViewModel survives (same instance)
//     - SavedStateHandle is recreated with new args
//     - Lifecycle observers see DESTROYED on old entry, CREATED→RESUMED on new
//
// To observe argument changes in the ViewModel:
class DetailViewModel(savedStateHandle: SavedStateHandle) : ViewModel() {
    // This LiveData updates when arguments change via singleTop re-navigation
    val itemId: StateFlow<Long> = savedStateHandle.getStateFlow("itemId", 0L)
}

Navigation During Config Change

// During config change, NavController's state is saved and restored:
//
// 1. Activity.onSaveInstanceState()
//    → NavController.saveState() saves:
//      - Current back stack (destination IDs + arguments + state)
//      - Navigator state (FragmentNavigator saves Fragment states)
//      - Deep link state
//
// 2. Activity recreated → NavHostFragment.onCreate()
//    → NavController.restoreState() rebuilds:
//      - NavBackStackEntry list from saved state
//      - Reconnects entries to their ViewModelStores (via NavControllerViewModel)
//      - FragmentNavigator restores Fragments from FragmentManager's saved state
//
// Corner case: what if a navigate() call is in-flight during config change?
//   → The FragmentTransaction may not have committed yet
//   → setReorderingAllowed(true) ensures transactions are reordered correctly
//   → The saved state captures the post-navigation state because
//     onSaveInstanceState() runs after all pending transactions execute

Nested Graph Scoping Subtleties

// Nested graph ViewModel scoping has a subtle lifecycle:
//
// Graph: main_graph
//   ├── homeFragment
//   └── auth_graph (startDest = loginFragment)
//         ├── loginFragment
//         └── registerFragment
//
// Scenario:
//   1. Navigate: home → login (enters auth_graph)
//      → NavBackStackEntry for auth_graph is created
//      → NavBackStackEntry for loginFragment is created
//   2. Navigate: login → register
//      → NavBackStackEntry for registerFragment is created
//      → auth_graph entry stays alive
//   3. Pop: register → login
//      → registerFragment entry destroyed
//      → auth_graph entry still alive (loginFragment is still in it)
//   4. Pop: login → home
//      → loginFragment entry destroyed
//      → auth_graph entry ALSO destroyed (no more children)
//      → Graph-scoped ViewModel is cleared
//
// Gotcha: if you popUpTo(auth_graph, inclusive=true), the graph entry
// AND all its children are destroyed in one operation.

popUpTo with inclusive=true — Accidentally Emptying the Stack

// A common mistake with bottom navigation or login flows:
navController.navigate(R.id.homeFragment, null, navOptions {
    popUpTo(R.id.homeFragment) { inclusive = true }
})

// If homeFragment is the start destination and inclusive=true:
//   → The start destination is popped
//   → The graph's back stack is now empty
//   → Navigating to homeFragment creates a new entry
//   → But pressing Back now EXITS the app (no start destination to fall back to)
//
// The correct pattern for "clear stack and go to start":
navController.navigate(R.id.homeFragment, null, navOptions {
    popUpTo(navController.graph.startDestinationId) { inclusive = true }
    launchSingleTop = true
})

DialogFragment Navigation Lifecycle

// DialogFragment destinations have a unique lifecycle interaction:
//
// When navigating to a dialog destination:
//   → The dialog's NavBackStackEntry is pushed onto the back stack
//   → The dialog Fragment is shown via show()
//   → The PREVIOUS destination stays RESUMED (not paused!)
//   → This is different from fragment destinations where the previous
//     destination moves to STARTED
//
// Back stack state:
//   [Home: RESUMED] [ConfirmDialog: RESUMED]
//   Both entries are RESUMED simultaneously!
//
// When the dialog is dismissed (back press or dismiss()):
//   → Dialog's NavBackStackEntry is popped
//   → Dialog's ViewModel is cleared
//   → Home stays RESUMED (was never paused)
//
// Gotcha: if you observe NavController.currentBackStackEntryAsFlow(),
// the dialog IS the current entry while shown. If you use
// currentDestination to conditionally navigate, you must account for
// dialog destinations being "current".
Staff-Level Insight — NavControllerViewModel and Config Change Survival

The entire Navigation back stack's ViewModel state survives configuration changes through a clever indirection: NavControllerViewModel is a ViewModel scoped to the Activity. It holds a Map<UUID, ViewModelStore> where each UUID corresponds to a NavBackStackEntry. When the Activity is recreated, NavControllerViewModel survives (it's in the Activity's ViewModelStore). NavController restores its back stack from SavedState (which contains the UUIDs), then reconnects each restored NavBackStackEntry to its ViewModelStore via the UUID lookup. This is why graph-scoped ViewModels survive config changes even though the Fragments are destroyed and recreated — the ViewModelStore is owned by the NavBackStackEntry, not the Fragment, and the entry's store is held by NavControllerViewModel. If you call ViewModelProvider(navBackStackEntry) directly, you're accessing this same store.

Danger — Fragment Leaks with Navigation

A subtle leak pattern: holding a reference to a Fragment's View in the Fragment class field after onDestroyView(). When Navigation replaces Fragment A with Fragment B, A's view is destroyed but A's instance stays alive on the back stack. If A holds a reference to its binding or a view, that entire view hierarchy is leaked until A is popped. Always null out view references in onDestroyView(), or use viewLifecycleOwner-scoped observers that auto-clean up.

9. Deep Link Edge Cases

// Ambiguous deep links — multiple destinations match the same URI:
// NavController matches in order: explicit URI > implicit URI > MIME type
// If two destinations have the same URI pattern, the FIRST one in the
// nav graph XML wins. This is a common source of bugs in large apps.

// Deep link + existing back stack:
// handleDeepLink() CLEARS the existing back stack by default
// This means if user has A → B → C and a deep link arrives for D:
//   Result: startDestination → D (A, B, C are gone)
//
// To preserve existing state, DON'T use handleDeepLink().
// Instead, manually navigate:
intent.data?.let { uri ->
    val matched = navController.graph.matchDeepLink(
        NavDeepLinkRequest.Builder.fromUri(uri).build()
    )
    if (matched != null) {
        navController.navigate(matched.destination.id, matched.matchingArgs)
    }
}

// Deep link + process death:
// If the app is killed while displaying a deep-linked destination:
// 1. NavController saves its back stack to savedInstanceState
// 2. On restore, the synthetic back stack is recreated
// 3. Arguments are preserved in the NavBackStackEntry's SavedStateHandle
// 4. But any in-memory state (ViewModel without SavedStateHandle) is LOST

10. Back Stack Corruption Recovery

// FragmentManager and NavController back stacks can diverge in edge cases:
// - Rapid navigation during config change
// - commitAllowingStateLoss after onSaveInstanceState
// - Manual FragmentTransaction alongside NavController

// Detection: NavController throws IllegalStateException with
// "FragmentManager is already executing transactions"

// Prevention:
// 1. NEVER mix manual FragmentTransactions with NavController
// 2. Use navController.navigate() exclusively
// 3. Guard navigation with lifecycle checks:
fun safeNavigate(navController: NavController, directions: NavDirections) {
    val currentDest = navController.currentDestination
    if (currentDest != null) {
        // Only navigate if the action exists on the current destination
        val action = currentDest.getAction(directions.actionId)
        if (action != null) {
            navController.navigate(directions)
        }
    }
}

// Recovery: if the back stack IS corrupted, the nuclear option:
navController.popBackStack(navController.graph.startDestinationId, false)
// This pops everything back to the start destination
