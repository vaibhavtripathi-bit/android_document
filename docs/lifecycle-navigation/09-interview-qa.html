09 — Lifecycle & Navigation Interview Q&A

75 L6 Staff Engineer questions covering Activity/Fragment lifecycle internals, configuration changes, process death, back stack & tasks, multi-window, lifecycle-aware components, Navigation Component, Compose Navigation, and system design

Activity Lifecycle (Q1–Q10)

Q1. Walk through what happens in AOSP when you call startActivity() — from Intent to first frame visible.

The call traverses: Activity.startActivity() → Instrumentation.execStartActivity() → Binder IPC to ActivityTaskManagerService.startActivity() in system_server. ATMS delegates to ActivityStarter, which resolves the Intent via PackageManagerService, checks permissions, evaluates launch flags and launch modes, and finds or creates a Task. It creates an ActivityRecord and calls RootWindowContainer.resumeFocusedTasksTopActivities(). The current foreground Activity receives a pause transaction via ClientTransaction containing a PauseActivityItem. Once the old Activity reports paused (via activityPaused() Binder callback), ATMS sends a LaunchActivityItem + ResumeActivityItem bundled in a single ClientTransaction to the target process. If the process doesn't exist, Process.start() forks via Zygote first. In the app process, TransactionExecutor processes the items: LaunchActivityItem calls ActivityThread.handleLaunchActivity() → performLaunchActivity() (instantiates Activity, calls attach(), onCreate(), onStart()), then ResumeActivityItem drives onResume(). After onResume(), the first ViewRootImpl.performTraversals() runs measure/layout/draw, RenderThread submits the frame, and SurfaceFlinger composites it. The system calls reportFullyDrawn() or the window becomes visible via WMS.addWindow().

Q2. Explain ClientTransaction and TransactionExecutor — how does system_server drive Activity lifecycle?

ClientTransaction is a Parcelable container sent from system_server to an app process via the IApplicationThread Binder interface. It holds a list of callback items (e.g., LaunchActivityItem, NewIntentItem, ActivityConfigurationChangeItem) and an optional lifecycle state request (e.g., ResumeActivityItem, PauseActivityItem, DestroyActivityItem). On the app side, ActivityThread.scheduleTransaction() posts it to the main thread's Handler via EXECUTE_TRANSACTION message. TransactionExecutor.execute() first processes all callback items in order, then drives the lifecycle to the requested final state by calling the necessary intermediate transitions. For example, if the current state is CREATED and the request is RESUMED, the executor calls onStart() then onResume() automatically. This design replaced the older scattered scheduleXxxActivity() Binder methods with a unified transaction model, making lifecycle transitions atomic and ordered.

Q3. What is the difference between isFinishing() and isChangingConfigurations() in onDestroy()? Why does it matter?

isFinishing() returns true when the Activity is being permanently destroyed — the user pressed back, you called finish(), or the system is clearing the task. isChangingConfigurations() returns true when the Activity is being destroyed solely to be recreated due to a configuration change (rotation, locale, etc.). The distinction matters for resource cleanup: if isFinishing() is true, you should release all resources permanently (cancel network requests, close database connections, unregister global listeners). If isChangingConfigurations() is true, you should retain expensive resources via ViewModel or onRetainNonConfigurationInstance() because the Activity will be immediately recreated. A common bug is unconditionally releasing resources in onDestroy(), causing unnecessary re-initialization after rotation. Note that both can be false simultaneously — when the system kills the Activity for memory reclamation without calling onDestroy() at all.

Q4. An Activity is visible but NOT in RESUMED state — give 3 scenarios where this happens.

(1) Multi-window mode (pre-Android 10): Before multi-resume was introduced, only the last-touched Activity was RESUMED; the other visible Activity was in STARTED (paused but visible). (2) Transparent/dialog Activity overlay: When a dialog-themed Activity or a translucent Activity launches on top, the underlying Activity receives onPause() but NOT onStop() because it remains partially visible — it sits in the STARTED state. (3) During the pause-resume transition gap: Between the moment the system sends PauseActivityItem and the new Activity reaches RESUMED, the old Activity is in STARTED state but still rendering on screen. A fourth edge case: Picture-in-Picture mode — the PiP Activity is visible in a small window but enters PAUSED state (STARTED on Android 10+ with multi-resume). This distinction is critical for camera apps, video players, and any resource that should only be active when truly focused.

Q5. Explain the "trampoline Activity" pattern and why Android 12 restricted it.

A trampoline Activity is a transparent or no-UI Activity that exists solely to redirect — it launches in onCreate(), calls startActivity() for the real destination, then immediately calls finish(). Common uses: notification click handlers that route based on payload, deep link dispatchers, and authentication gates. Android 12 (API 31) restricted this for notification trampolines specifically: if an Activity started from a notification PendingIntent calls startActivity() and finish() without becoming visible, the system blocks the secondary launch. The restriction exists because trampolines were abused to circumvent background Activity launch restrictions — an app could use a trampoline to start an arbitrary Activity from a notification even when the app was in the background. The fix is to launch the final destination directly from the PendingIntent, or use a Service or BroadcastReceiver trampoline (which were also restricted in Android 12 for services, and Android 14 for receivers).

Q6. What happens to Activity lifecycle when the system kills the process? Which callbacks are NOT called?

When the system kills a process (via lmkd or kill -9), no lifecycle callbacks are called — no onPause(), onStop(), or onDestroy(). The process is terminated at the kernel level. However, the system does call onSaveInstanceState() before the process becomes eligible for killing — specifically, it's called before or after onStop() (after onStop() on API 28+, before onStop() on earlier versions). The ActivityRecord in system_server survives the process death and retains the saved Bundle. When the user returns to the task, the system forks a new process via Zygote and delivers the saved state Bundle to onCreate(savedInstanceState) and onRestoreInstanceState(). This is why you must never rely on onDestroy() for persisting critical data — use onStop() or onSaveInstanceState() instead.

Q7. Explain onNewIntent() timing — when is it called relative to other lifecycle callbacks?

onNewIntent() is called when an existing Activity instance is reused instead of creating a new one — triggered by singleTop/singleTask/singleInstance launch modes or FLAG_ACTIVITY_SINGLE_TOP with FLAG_ACTIVITY_CLEAR_TOP. The callback order depends on the Activity's current state. If the Activity is already resumed (foreground): onPause() → onNewIntent() → onResume(). If the Activity is stopped (background): onNewIntent() → onRestart() → onStart() → onResume(). Critically, getIntent() still returns the original Intent after onNewIntent() — you must call setIntent(intent) explicitly if you want getIntent() to reflect the new one. This is a common bug: reading deep link data in onResume() via getIntent() without updating it in onNewIntent() causes stale data to be processed on every resume.

Q8. How does moveTaskToBack(true) differ from pressing Home in terms of lifecycle callbacks?

Both result in the same lifecycle sequence: onPause() → onStop() → onSaveInstanceState(). The difference is in task ordering. Pressing Home brings the launcher task to the front, pushing the current task down in the recents stack. moveTaskToBack(true) moves the entire task (all Activities in the back stack) to the bottom of the task stack without bringing any other task forward — the user sees whatever was behind it. The true parameter means "move the root task"; if false, it only works if the calling Activity is the root of the task. From the system's perspective, both result in the Activity entering STOPPED state and becoming eligible for process death. The practical difference matters for multi-task apps: moveTaskToBack() preserves the previous task's z-order, while Home always foregrounds the launcher.

Q9. What is ActivityRecord in AOSP and what state does it hold?

ActivityRecord is the system_server-side representation of an Activity instance, living in com.android.server.wm. Each ActivityRecord holds: the ComponentName (package + class), the Intent that launched it, the resultTo (who to deliver onActivityResult to), the icicle (saved instance state Bundle), the current lifecycle state (an enum: INITIALIZING, STARTED, RESUMED, PAUSING, PAUSED, STOPPING, STOPPED, FINISHING, DESTROYING, DESTROYED), the task affinity, launch mode, window configuration, and the app WindowToken for WMS. It also tracks launchedFromUid/launchedFromPackage for security checks, visible and nowVisible flags, and the ProcessRecord of the hosting process. ActivityRecord survives process death — this is how the system restores the back stack when the user returns. It's the authoritative source of truth; the app-side Activity object is just a projection of this server-side record.

Q10. Explain the pause-then-launch race condition — what happens if a new Activity is launched while the current one is still pausing?

ATMS enforces a strict pause-before-launch protocol. When Activity A is being paused and a new startActivity() for Activity C arrives, the system queues the launch. TaskFragment.schedulePauseActivity() sends a PauseActivityItem to A's process and sets a pause timeout (500ms). If A reports paused within the timeout, the system proceeds to launch B (the pending Activity). If A fails to report paused (ANR-like), the timeout fires in ActivityRecord.schedulePauseTimeout(), the system force-transitions A to PAUSED state and proceeds anyway. The race condition becomes problematic with rapid launches: if you call startActivity() twice in quick succession, the second launch is deferred until the first Activity's pause completes, which can cause visible jank or unexpected ordering. This is also why onPause() must be fast — a slow onPause() delays the next Activity's launch by up to 500ms before the timeout kicks in.

Fragment Lifecycle (Q11–Q20)

Q11. Explain the dual lifecycle in Fragments — Fragment lifecycle vs Fragment View lifecycle. Why are they separate?

A Fragment has two distinct lifecycles: the Fragment instance lifecycle (onCreate → onDestroy) and the Fragment View lifecycle (onCreateView → onDestroyView). They're separate because a Fragment's instance can outlive its View. When a Fragment is placed on the back stack via replace(), the Fragment instance stays alive (in CREATED state) but its View is destroyed — onDestroyView() is called without onDestroy(). When the user presses back, the same Fragment instance gets a new View via onCreateView(). This means the Fragment instance lifecycle spans multiple View creation/destruction cycles. The practical consequence: any observer registered on the Fragment's this lifecycle persists across view recreations, while observers on viewLifecycleOwner are automatically cleaned up when the View is destroyed. This is why you must use viewLifecycleOwner for UI observations (LiveData, Flow collection) — using this causes duplicate observers after back stack restoration.

Q12. Walk through FragmentManager.moveToState() — how does it drive Fragment state transitions?

moveToState(Fragment f, int newState) is the central state machine in FragmentManager. It compares the Fragment's current mState (an int: INITIALIZING=0, ATTACHED=1, CREATED=2, VIEW_CREATED=3, AWAITING_EXIT_EFFECTS=4, ACTIVITY_CREATED=5, STARTED=6, AWAITING_ENTER_EFFECTS=7, RESUMED=8) with the target state and walks through each intermediate state, calling the appropriate callbacks. Moving up: INITIALIZING→ATTACHED calls onAttach(), ATTACHED→CREATED calls onCreate(), CREATED→VIEW_CREATED calls onCreateView()+onViewCreated(), and so on up to RESUMED. Moving down: RESUMED→STARTED calls onPause(), STARTED→ACTIVITY_CREATED calls onStop(), VIEW_CREATED→CREATED calls onDestroyView(), CREATED→ATTACHED calls onDestroy(), ATTACHED→INITIALIZING calls onDetach(). The Fragment can never skip states — each transition is sequential. The host Activity's state caps the maximum state a Fragment can reach; if the Activity is STARTED, no Fragment can be RESUMED.

Q13. What is SpecialEffectsController and how does it delay Fragment lifecycle during animations?

SpecialEffectsController (introduced in Fragment 1.4.0) coordinates animations, transitions, and lifecycle for Fragments entering/exiting. When a Fragment transaction includes animations (e.g., setCustomAnimations() or shared element transitions), the controller delays the Fragment's lifecycle progression until effects complete. For an entering Fragment, the view is created and added to the container, but moveToState() is paused at AWAITING_ENTER_EFFECTS until the enter animation finishes — only then does it proceed to STARTED/RESUMED. For an exiting Fragment, the lifecycle moves to AWAITING_EXIT_EFFECTS, the exit animation plays, and only after completion does onDestroyView() fire. This prevents the old problem where a Fragment's view was removed mid-animation. The controller uses Operation objects that track each Fragment's pending effects and calls complete() when all animations finish, unblocking the lifecycle state machine.

Q14. commit() vs commitNow() vs commitAllowingStateLoss() — explain the internals of each.

commit() schedules the transaction asynchronously by posting it to the FragmentManager's internal Handler via mHost.getHandler().post(mExecCommit). It executes on the next main thread message loop iteration. Multiple commit() calls batch together. commitNow() executes the transaction synchronously on the calling thread — it calls FragmentManager.execSingleAction() immediately. The tradeoff: commitNow() cannot be added to the back stack (throws IllegalStateException) because back stack operations require ordered execution that async scheduling guarantees. commitAllowingStateLoss() is identical to commit() but sets mAllowStateLoss = true, suppressing the IllegalStateException that fires if you commit after onSaveInstanceState(). The risk: if the process is killed and restored, this transaction is lost because it wasn't included in the saved state. Use it only for UI updates that are acceptable to lose (e.g., showing a transient dialog). There's also commitNowAllowingStateLoss() — synchronous + state-loss tolerant.

Q15. How does setMaxLifecycle() work and why does ViewPager2 use it?

setMaxLifecycle(fragment, Lifecycle.State) caps the maximum lifecycle state a Fragment can reach, regardless of the host Activity's state. Internally, FragmentManager.moveToState() checks fragment.mMaxState and clamps the target state: newState = Math.min(newState, fragment.mMaxState). ViewPager2 uses this to implement RESUMED-only-when-visible behavior. Its FragmentStateAdapter calls setMaxLifecycle(STARTED) for off-screen Fragments and setMaxLifecycle(RESUMED) for the currently visible Fragment. This replaced the old setUserVisibleHint() hack from ViewPager1, which was unreliable because it was a separate signal disconnected from the lifecycle. With setMaxLifecycle(), an off-screen Fragment's viewLifecycleOwner is in STARTED state, so launchWhenResumed blocks and collectAsStateWithLifecycle() with RESUMED minimum state stops collecting — giving correct lifecycle-aware behavior for free.

Q16. Explain BackStackRecord — what operations does it store and how does popBackStack() reverse them?

BackStackRecord extends FragmentTransaction and implements FragmentManager.BackStackEntry. It stores an ArrayList<Op> where each Op has a cmd (ADD, REMOVE, REPLACE, HIDE, SHOW, DETACH, ATTACH, SET_PRIMARY_NAV, SET_MAX_LIFECYCLE, UNSET) and a reference to the target Fragment. When addToBackStack() is called, the record is saved in FragmentManager.mBackStack. On popBackStack(), the FragmentManager calls BackStackRecord.generateOps() which creates a reverse record: ADD becomes REMOVE, REMOVE becomes ADD, REPLACE reverses to re-add the removed Fragment and remove the added one, SHOW becomes HIDE, etc. The reversed operations are then executed via moveToState(). This is why replace() + addToBackStack() works correctly — the system records both the removal of the old Fragment and the addition of the new one, and reverses both on pop.

Q17. What is the ViewBinding leak in Fragments and how do you fix it?

When a Fragment is on the back stack, its View is destroyed (onDestroyView()) but the Fragment instance stays alive. If you hold a reference to a ViewBinding in a Fragment field (private var binding: MyBinding? = null) and don't null it in onDestroyView(), the binding holds a strong reference to the entire View hierarchy, preventing garbage collection. This leaks the Views, their Drawables, Bitmaps, and anything attached to them. The fix:

class MyFragment : Fragment(R.layout.my_fragment) {
    private var _binding: MyFragmentBinding? = null
    private val binding get() = _binding!!

    override fun onViewCreated(view: View, savedInstanceState: Bundle?) {
        _binding = MyFragmentBinding.bind(view)
    }

    override fun onDestroyView() {
        super.onDestroyView()
        _binding = null  // Critical: break the reference
    }
}

Alternatively, use a delegate like autoCleared or scope observations to viewLifecycleOwner. In Compose-based Fragments using ComposeView, this is less of an issue because ComposeView disposes its composition in onDestroyView() via the ViewCompositionStrategy.

Q18. How do child FragmentManagers work? What is setPrimaryNavigationFragment()?

Each Fragment has its own childFragmentManager — a separate FragmentManager instance that manages Fragments nested inside it. The child FM's lifecycle is tied to the parent Fragment: when the parent reaches RESUMED, child Fragments can also reach RESUMED; when the parent is destroyed, all children are destroyed. setPrimaryNavigationFragment() designates one child Fragment as the primary navigation host. This matters for back button dispatch: when the user presses back, FragmentManager.popBackStackImmediate() first checks the primary navigation Fragment's child FM for back stack entries before checking its own. This creates a delegation chain: Activity FM → primary nav Fragment's child FM → that Fragment's primary nav child FM, etc. NavHostFragment sets itself as the primary navigation Fragment so that Navigation's back stack is popped before the Activity's FragmentManager back stack. Without this, pressing back would pop the Activity's FM first, potentially removing the entire NavHostFragment.

Q19. Why must Fragments have a no-arg constructor? How does FragmentFactory solve this?

The system recreates Fragments via reflection using Class.newInstance() (which requires a public no-arg constructor) during configuration changes and process death restoration. FragmentManager.restoreAllState() → FragmentFactory.instantiate(classLoader, className) → Class.forName(className).newInstance(). If you add constructor parameters, the reflective instantiation throws InstantiationException. FragmentFactory (added in Fragment 1.1.0) replaces the default reflection with a customizable factory:

class MyFragmentFactory(
    private val repository: Repository
) : FragmentFactory() {
    override fun instantiate(classLoader: ClassLoader, className: String): Fragment =
        when (className) {
            MyFragment::class.java.name -> MyFragment(repository)
            else -> super.instantiate(classLoader, className)
        }
}

// Install before super.onCreate() in Activity
supportFragmentManager.fragmentFactory = MyFragmentFactory(repo)

In practice, most modern apps use Hilt's @AndroidEntryPoint which installs its own FragmentFactory via HiltFragmentFactory, making manual factory setup unnecessary.

Q20. What happens if you commit a Fragment transaction after onSaveInstanceState()?

FragmentManager.enqueueAction() checks mStateSaved flag. If true, it throws IllegalStateException("Can not perform this action after onSaveInstanceState"). This exists because any transaction committed after state save won't be included in the saved Bundle — if the process is killed and restored, the transaction is silently lost, leading to inconsistent UI state. The flag is set in saveAllState() (called from Activity.onSaveInstanceState()) and cleared in noteStateNotSaved() (called from onStart()). Common triggers: committing in an async callback (network response, coroutine) that fires after the Activity has stopped. Solutions: (1) use commitAllowingStateLoss() if the transaction is non-critical, (2) check lifecycle.currentState.isAtLeast(STARTED) before committing, (3) use lifecycle-aware coroutine scopes like viewLifecycleOwner.lifecycleScope.launchWhenStarted (deprecated) or repeatOnLifecycle(STARTED).

Config Change & Process Death (Q21–Q30)

Q21. Walk through the full sequence of callbacks during a configuration change.

When a configuration change occurs (e.g., rotation), the system sends ActivityConfigurationChangeItem via ClientTransaction. If the Activity does NOT handle the config change (no android:configChanges for that type), AMS triggers recreation: onPause() → onStop() → onSaveInstanceState() → onDestroy() (with isChangingConfigurations() == true) → onCreate(savedInstanceState) → onStart() → onRestoreInstanceState() → onResume(). During destruction, onRetainNonConfigurationInstance() is called to save the ViewModelStore. The new Activity instance retrieves it via getLastNonConfigurationInstance(). Fragment lifecycle mirrors this: all Fragments go through destroy/create, but their ViewModels survive via the Activity's NonConfigurationInstances. If the Activity DOES declare android:configChanges="orientation|screenSize", only onConfigurationChanged() is called — no destruction/recreation occurs.

Q22. How does AMS decide whether to recreate an Activity on config change?

When a configuration change is detected (from WindowManagerService, DisplayManagerService, or LocaleManager), ActivityRecord.shouldRelaunchLocked() computes the diff between the old and new Configuration objects using Configuration.diff(), which returns a bitmask of changed fields (e.g., CONFIG_ORIENTATION, CONFIG_SCREEN_SIZE, CONFIG_LOCALE). This bitmask is ANDed with the complement of the Activity's declared android:configChanges from the manifest. If any bits remain set (meaning the Activity doesn't handle those changes), the Activity is destroyed and recreated. If all bits are handled, only onConfigurationChanged() is dispatched. The computation is: changesToHandle = configDiff & ~handledConfigChanges. If changesToHandle != 0, recreate. This is why declaring configChanges="orientation" without screenSize still causes recreation on rotation for API 13+ — rotation changes both orientation AND screen size.

Q23. Explain SavedStateRegistry architecture — how does it wrap the platform Bundle mechanism?

SavedStateRegistry (from androidx.savedstate) is a component-level abstraction over the platform's onSaveInstanceState(Bundle). It implements SavedStateRegistryOwner (both ComponentActivity and Fragment implement this). Multiple components register SavedStateProvider instances under unique keys via registerSavedStateProvider(key, provider). When the system calls onSaveInstanceState(), SavedStateRegistry.performSave(outBundle) iterates all registered providers, calls provider.saveState() on each, and stores the returned Bundles as sub-Bundles under their keys in the outgoing Bundle. On restoration, performRestore(savedState) unpacks the sub-Bundles, and components call consumeRestoredStateForKey(key) to retrieve their data. This decouples state saving from the Activity/Fragment — any component (ViewModel, custom view, library) can participate in state saving without the Activity knowing about it. SavedStateHandle in ViewModel is built on top of this mechanism.

Q24. How does ViewModel survive configuration changes? Explain the NonConfigurationInstances mechanism.

During a configuration change, before the Activity is destroyed, ActivityThread.performDestroyActivity() calls activity.retainNonConfigurationInstances(), which calls onRetainNonConfigurationInstance(). ComponentActivity overrides this to return a NonConfigurationInstances object containing the ViewModelStore. This object is stored in the ActivityClientRecord (the app-process-side record, NOT the system_server ActivityRecord). When the new Activity instance is created, ActivityThread.performLaunchActivity() passes the retained NonConfigurationInstances to activity.attach(), making it available via getLastNonConfigurationInstance(). ComponentActivity.ensureViewModelStore() checks for a retained ViewModelStore first, and only creates a new one if none exists. This is purely an in-process, in-memory mechanism — it does NOT survive process death. The ViewModelStore holds a HashMap<String, ViewModel>, so all ViewModels survive together.

Q25. What is the difference between config change and process death in terms of what survives?

Config change: ViewModel survives (via NonConfigurationInstances), saved instance state Bundle survives, static/singleton state survives (same process), retained Fragments survive. The process stays alive; only the Activity/Fragment instances are recreated. Process death: Only the saved instance state Bundle survives (stored in system_server's ActivityRecord.icicle). ViewModel is destroyed (it's in-process memory). Static/singleton state is lost. All in-memory caches, repositories, and dependency injection graphs are recreated from scratch. This distinction drives the state restoration pyramid: UI state (scroll position, text input) goes in savedInstanceState Bundle; expensive-to-recreate data (network results, computed data) goes in ViewModel; persistent data (user preferences, form drafts) goes in DataStore/Room. Testing process death requires actually killing the process (adb shell am kill), not just rotating the device.

Q26. Explain SavedStateHandle — how does it make ViewModel state survive process death?

SavedStateHandle bridges ViewModel (survives config change only) with SavedStateRegistry (survives process death). It's a key-value map backed by a Bundle. Internally, SavedStateHandleController registers a SavedStateProvider with the SavedStateRegistry for each ViewModel that has a SavedStateHandle. On save, the handle's current state is serialized to a Bundle. On restoration, the Bundle is passed to the SavedStateHandle constructor, pre-populating it. Usage:

class MyViewModel(
    private val savedStateHandle: SavedStateHandle
) : ViewModel() {
    // Survives both config change AND process death
    val query = savedStateHandle.getStateFlow("query", "")

    fun setQuery(q: String) {
        savedStateHandle["query"] = q
    }
}

With Hilt, @HiltViewModel automatically provides SavedStateHandle via CreationExtras. The handle also receives the Activity's intent extras and Fragment arguments as initial values, making it a unified source for both navigation arguments and saved state.

Q27. What causes TransactionTooLargeException and how do you diagnose/fix it?

TransactionTooLargeException is thrown when the total size of all Parcels in a Binder transaction exceeds the 1MB buffer (actually ~500KB in practice because the buffer is shared). For saved state, it fires during onSaveInstanceState() when the Bundle (which includes all Fragment saved states, View hierarchy state, and SavedStateHandle data) exceeds the limit. Common causes: storing large Bitmaps, entire API responses, or large lists in saved state; Fragments with setArguments() containing large Bundles (arguments are re-saved every time). Diagnosis: use TooLargeTool library or log Bundle sizes in onSaveInstanceState() via Bundle.toByteArray().size (using Parcel marshalling). Fix: store only IDs/keys in saved state, not data; use ViewModel for large in-memory data; persist large data to disk and save only a reference; avoid passing large argument Bundles to Fragments. On API 24+, the exception is thrown; on earlier versions, the transaction silently fails.

Q28. How do you properly test process death? What are the differences between adb shell am kill and "Don't keep activities"?

adb shell am kill <package> kills the process only when it's in the background (not foreground). It simulates real process death: the process is terminated, all in-memory state is lost, but ActivityRecord in system_server retains the saved Bundle. When the user returns, the Activity is recreated with savedInstanceState != null. "Don't keep activities" (Developer Options) is fundamentally different: it destroys the Activity immediately when the user leaves, but the process stays alive. This means ViewModel, singletons, and static state survive — it only tests onSaveInstanceState()/onRestoreInstanceState(), not full process death. For proper testing: (1) put app in background, (2) run adb shell am kill <package>, (3) return to app from Recents. You can also use Android Studio's "Terminate Application" button in the process list. For automated testing, use Instrumentation.callActivityOnSaveInstanceState() or Robolectric's ActivityController.recreate().

Q29. Explain the state restoration pyramid — what goes in Bundle vs SavedStateHandle vs persistent storage?

The pyramid has three tiers based on survival scope and size constraints:

  • Bundle / onSaveInstanceState(): Tiny UI state that must survive process death — scroll position, selected tab index, text input cursor position. Size limit: keep under 50KB total. The View system automatically saves/restores some state (EditText text, RecyclerView scroll position) if views have IDs.
  • SavedStateHandle (in ViewModel): Small navigation arguments and user-entered data that must survive process death — search query, form field values, selected filter IDs. Still Bundle-backed, same size constraints, but scoped to ViewModel lifecycle.
  • ViewModel (without SavedStateHandle): Expensive-to-recreate data that only needs to survive config changes — API responses, computed/filtered lists, pagination state. Lost on process death; must be re-fetched from the persistent layer.
  • Persistent storage (Room, DataStore, files): Data that must survive app termination — user preferences, cached API data, draft content. No size limit. Source of truth for ViewModel re-population after process death.
Q30. How does rememberSaveable in Compose hook into SavedStateRegistry?

rememberSaveable uses the SaveableStateRegistry provided by LocalSaveableStateRegistry, which is backed by the platform's SavedStateRegistry. When a composable calls rememberSaveable { mutableStateOf("") }, the runtime registers a saver with the registry under a key derived from the composable's position in the composition (using the key from the slot table). On onSaveInstanceState(), the registry iterates all registered saveables, calls their Saver.save() function (default: autoSaver which uses Bundle-compatible types), and stores the results. On restoration, the saved values are provided to rememberSaveable before the composable re-executes. For custom types, you provide a Saver:

val colorSaver = Saver<Color, Int>(
    save = { it.toArgb() },
    restore = { Color(it) }
)
var color by rememberSaveable(stateSaver = colorSaver) {
    mutableStateOf(Color.Red)
}

rememberSaveable with listSaver or mapSaver handles complex types. For NavBackStackEntry-scoped composables, the saveable state is tied to the entry's SavedStateHandle, so it survives navigation and process death.

Back Stack & Tasks (Q31–Q38)

Q31. Explain all 5 launch modes and how ActivityStarter resolves them.

ActivityStarter.startActivityUnchecked() evaluates launch mode (from manifest + Intent flags) to determine task and instance behavior:

  • standard: Always creates a new instance in the caller's task. Multiple instances can exist in the same task.
  • singleTop: If the target Activity is already at the top of the task, onNewIntent() is delivered instead of creating a new instance. If it's not at the top, a new instance is created (same as standard).
  • singleTask: The system looks for an existing task with matching taskAffinity. If found and the Activity exists in it, all Activities above it are cleared and onNewIntent() is delivered. If the task exists but the Activity doesn't, a new instance is created in that task. If no matching task exists, a new task is created.
  • singleInstance: Like singleTask, but the Activity is the only Activity in its task — no other Activities can be launched into it. Any startActivity() from it creates a new task.
  • singleInstancePerTask (Android 12+): Like singleInstance but allows one instance per task. Different tasks can each have one instance. Useful for multi-window scenarios where you want the same Activity in separate tasks.

Intent flags (FLAG_ACTIVITY_NEW_TASK, FLAG_ACTIVITY_CLEAR_TOP, etc.) can override manifest launch modes. When both are specified, flags take precedence for most behaviors.

Q32. FLAG_ACTIVITY_CLEAR_TOP without FLAG_ACTIVITY_SINGLE_TOP — what happens and why is it surprising?

If Activity B exists in the back stack [A → B → C → D] and you launch B with only FLAG_ACTIVITY_CLEAR_TOP, the system clears C and D, destroys the existing B instance, and creates a new B instance. The surprising part: B is destroyed and recreated, not reused. onNewIntent() is NOT called. The stack becomes [A → B(new)]. This happens because the default launch mode is standard, which always creates a new instance. To get the "reuse existing instance" behavior (clear top + deliver new intent), you must combine FLAG_ACTIVITY_CLEAR_TOP | FLAG_ACTIVITY_SINGLE_TOP. With both flags, the existing B is reused: C and D are destroyed, and B receives onNewIntent(). This is one of the most common launch-mode bugs — developers expect CLEAR_TOP alone to reuse the instance, but it doesn't.

Q33. What is task affinity and how does allowTaskReparenting work?

Task affinity is a string (default: the app's package name) declared via android:taskAffinity in the manifest. It determines which task an Activity "belongs to." It's used in two scenarios: (1) When FLAG_ACTIVITY_NEW_TASK is used, the system looks for an existing task with matching affinity before creating a new one. (2) With allowTaskReparenting="true", an Activity can move between tasks. Example: App A launches Activity B (from App B's package) into App A's task. Later, the user launches App B from the launcher. Because B has allowTaskReparenting="true" and its affinity matches App B's task, the system reparents B from App A's task to App B's task. Internally, RootWindowContainer.moveActivityToPinnedRootTask() or the task-reparenting logic in ActivityRecord.reparent() handles this. Setting taskAffinity="" (empty string) means the Activity has no affinity to any task, which is useful for preventing unintended task association.

Q34. Explain the "task hijacking" (StrandHogg) vulnerability — how does it work and how do you prevent it?

StrandHogg exploits taskAffinity + allowTaskReparenting. A malicious app declares an Activity with android:taskAffinity="com.victim.app" and android:allowTaskReparenting="true". When the victim app's task is brought to the foreground, the malicious Activity reparents itself into the victim's task and appears on top, displaying a phishing UI (e.g., a fake login screen). The user believes they're interacting with the victim app. StrandHogg 2.0 was worse — it used FLAG_ACTIVITY_NEW_TASK to inject into any task without needing allowTaskReparenting. Prevention: (1) Set android:taskAffinity="" on sensitive Activities. (2) Set android:allowTaskReparenting="false" (default). (3) Use android:launchMode="singleTask" for your main Activity to ensure it controls its task. (4) Check getCallingActivity() or getReferrer() for unexpected launchers. Android 12+ mitigated StrandHogg 2.0 by restricting background Activity launches and adding stricter task affinity checks.

Q35. How does the predictive back gesture work internally? Explain OnBackPressedDispatcher.

OnBackPressedDispatcher (in ComponentActivity) maintains a LinkedList<OnBackPressedCallback>. When back is pressed, it iterates from the last-added callback, finding the first one where isEnabled == true, and calls its handleOnBackPressed(). If no enabled callback exists, it falls through to Activity.onBackPressed() → finish(). For predictive back (Android 14+), the system needs to know ahead of time whether the app will handle back. The dispatcher reports this via IOnBackInvokedCallback registered with WindowOnBackInvokedDispatcher. When the user starts the back gesture, the system calls onBackStarted(), then onBackProgressed() with progress updates (0.0 to 1.0), and finally onBackInvoked() on commit or onBackCancelled() on cancel. The system uses this to show the predictive animation (peeking the previous Activity/home screen). Apps must opt in via android:enableOnBackInvokedCallback="true" in the manifest. Navigation Component and Fragment back stack integrate automatically.

Q36. How do you synthesize a back stack for deep links using TaskStackBuilder?

When a user opens a deep link directly to a detail screen, pressing back should navigate to the parent screen, not exit the app. TaskStackBuilder creates a synthetic back stack:

val detailIntent = Intent(context, DetailActivity::class.java).apply {
    putExtra("id", itemId)
}
TaskStackBuilder.create(context)
    .addNextIntentWithParentStack(detailIntent)
    .startActivities()
// Creates: MainActivity → ListActivity → DetailActivity

Internally, addNextIntentWithParentStack() reads android:parentActivityName from the manifest and recursively adds parent Intents. It sets FLAG_ACTIVITY_NEW_TASK on the root and FLAG_ACTIVITY_CLEAR_TASK to ensure a fresh task. For PendingIntents (notifications), use getPendingIntent() instead of startActivities(). With Navigation Component, NavDeepLinkBuilder does this automatically — it creates a PendingIntent that synthesizes the back stack based on the nav graph's parent hierarchy. The key insight: the intermediate Activities are not actually created until the user navigates back to them — only the topmost Activity is launched immediately.

Q37. What is singleInstancePerTask (Android 12+) and when would you use it?

singleInstancePerTask allows exactly one instance of the Activity per task, but unlike singleInstance, multiple instances can exist across different tasks. If you launch the Activity into Task A, it becomes the root. Launching it again in Task A delivers onNewIntent(). But launching it with FLAG_ACTIVITY_NEW_TASK | FLAG_ACTIVITY_MULTIPLE_TASK creates a new instance in a new Task B. Use cases: (1) Multi-window — a document editor that should have one instance per window/task, allowing the user to open multiple documents in split-screen, each in its own task. (2) Dynamic shortcuts — each shortcut opens the same Activity type but in a separate task. (3) Launcher-like behavior — apps that manage multiple independent workspaces. It solves the problem where singleInstance was too restrictive (only one instance globally) and singleTask didn't prevent multiple instances when launched with NEW_TASK from different affinities.

Q38. How does finishAndRemoveTask() differ from finish()?

finish() destroys only the current Activity. If it's the last Activity in the task, the task is also removed from Recents. But if other Activities remain in the task's back stack, the task persists and the previous Activity becomes visible. finishAndRemoveTask() destroys the current Activity AND removes the entire task from the Recents screen, regardless of how many Activities are in the back stack. Internally, it calls ActivityTaskManagerService.finishActivity() with the FINISH_TASK_WITH_ROOT_ACTIVITY flag, which triggers Task.removeTask() in the window manager. All Activities in the task receive onDestroy(). Use cases: (1) logout flows where you want to clear the entire navigation history, (2) notification-launched tasks that should be ephemeral, (3) OAuth callback Activities that should not appear in Recents. Note: calling this from a non-root Activity still removes the entire task.

Multi-Window & PiP (Q39–Q44)

Q39. Explain multi-resume (Android 10+) — how does it change the lifecycle model?

Before Android 10, only one Activity could be in RESUMED state at a time, even in split-screen. The focused Activity was RESUMED; the other visible Activity was PAUSED (technically STARTED). This broke apps that released resources in onPause() — a video player in split-screen would stop playback when the user tapped the other app. Multi-resume (Android 10+, mandatory for OEMs on Android 12+) allows all visible Activities to be RESUMED simultaneously. The lifecycle model changes: both split-screen Activities are RESUMED, and the system introduces onTopResumedActivityChanged(isTopResumed: Boolean) to distinguish the focused Activity from other resumed Activities. This callback replaces the old "only one RESUMED" signal. For exclusive resources (camera, microphone), apps should acquire in onTopResumedActivityChanged(true) and release in onTopResumedActivityChanged(false), not in onResume()/onPause(). The system still guarantees that onTopResumedActivityChanged(false) is called on the old Activity before onTopResumedActivityChanged(true) on the new one.

Q40. How should you handle camera access contention in multi-window?

With multi-resume, multiple RESUMED Activities may compete for the camera. The camera framework uses a priority system: the top-resumed Activity gets priority. If Activity A holds the camera and Activity B becomes top-resumed, the system disconnects A's camera (triggering CameraDevice.StateCallback.onDisconnected()) and grants it to B. Best practice:

class CameraFragment : Fragment() {
    override fun onStart() {
        super.onStart()
        // Register for camera availability, but don't open yet
    }

    // Use this instead of onResume/onPause for camera
    override fun onTopResumedActivityChanged(isTopResumed: Boolean) {
        if (isTopResumed) {
            openCamera()
        } else {
            // Don't close immediately — you might still be visible
            // Only close if camera is disconnected
        }
    }

    private val stateCallback = object : CameraDevice.StateCallback() {
        override fun onDisconnected(camera: CameraDevice) {
            // Another app took the camera — release gracefully
            camera.close()
        }
    }
}

Handle onDisconnected() gracefully — show a "camera in use" message rather than crashing. Re-attempt opening when onTopResumedActivityChanged(true) fires again.

Q41. Walk through PiP internals — PinnedTaskController, PipTaskOrganizer, SurfaceControl.

When enterPictureInPictureMode(params) is called, the system transitions through several layers. In system_server, ActivityRecord.pictureInPictureArgs is updated and RootWindowContainer.moveActivityToPinnedRootTask() moves the Activity's task into the pinned root task (a special Task with windowing mode WINDOWING_MODE_PINNED). PinnedTaskController manages the pinned task's bounds, aspect ratio constraints, and snap-to-edge behavior. On the SystemUI side, PipTaskOrganizer (a TaskOrganizer) receives callbacks when the pinned task is created/changed and drives the PiP window animations. It uses SurfaceControl.Transaction to directly manipulate the task's surface — resizing, repositioning, and animating without going through the normal layout system. The sourceRectHint from PictureInPictureParams tells the system which region of the Activity's window contains the video, enabling a seamless zoom-out animation from that rect to the PiP window. The Activity receives onPictureInPictureModeChanged() and should hide non-essential UI (controls, text) since the window is tiny.

Q42. What is onTopResumedActivityChanged() and when should you use it?

onTopResumedActivityChanged(isTopResumed: Boolean) is called when the Activity gains or loses the "top resumed" status — meaning it's the single Activity that has user focus among all RESUMED Activities. It was introduced in Android 10 alongside multi-resume. Use it for exclusive resources: camera, microphone, or any hardware that only one app should use at a time. The callback guarantees ordering: the losing Activity receives onTopResumedActivityChanged(false) before the gaining Activity receives true, with a 500ms timeout if the losing Activity doesn't release promptly. Do NOT use it for general UI updates (use onResume()/onPause()) or for pausing video playback (users expect video to continue in split-screen). The callback is only meaningful in multi-window; in single-window mode, it mirrors onResume()/onPause(). In Compose, you can observe this via a custom LifecycleObserver since there's no built-in Compose API for it.

Q43. How does PiP affect process priority (oom_adj)?

A PiP Activity is visible but not interactive (it's in a small overlay window). The system assigns it a visible process priority — oom_adj of VISIBLE_APP_ADJ (100), which is lower priority than a foreground Activity (FOREGROUND_APP_ADJ = 0) but much higher than a background process (CACHED_APP_MIN_ADJ = 900+). This means PiP apps are unlikely to be killed by lmkd under normal memory pressure, but they will be killed before the foreground app. The Activity's lifecycle state in PiP is PAUSED (pre-Android 10) or STARTED (Android 10+ with multi-resume for PiP, though this varies by OEM). When the user dismisses PiP, the Activity transitions to STOPPED and the process drops to cached priority. For video playback in PiP, use a foreground Service with a media notification to ensure the process stays at PERCEPTIBLE_APP_ADJ (200) even if the PiP window is dismissed.

Q44. Explain auto-enter PiP (Android 12+) and the sourceRectHint for seamless animation.

Auto-enter PiP (Android 12+) allows the Activity to enter PiP automatically when the user navigates away (Home gesture, Recents), without requiring an explicit enterPictureInPictureMode() call. Enable it via:

setPictureInPictureParams(
    PictureInPictureParams.Builder()
        .setAutoEnterEnabled(true)
        .setAspectRatio(Rational(16, 9))
        .setSourceRectHint(videoViewBounds) // Rect of the video surface
        .setSeamlessResizeEnabled(true)
        .build()
)

sourceRectHint is a Rect in window coordinates that tells the system where the "interesting content" (video) is. The PiP enter animation zooms from this rect to the PiP window position, creating a seamless transition where the video appears to shrink from its original position. Without it, the entire Activity window shrinks, which looks jarring. setSeamlessResizeEnabled(true) (Android 12+) enables cross-fade instead of sharp resize during PiP transitions. Internally, PipTaskOrganizer uses the sourceRectHint to calculate the animation start bounds and uses SurfaceControl.Transaction.setMatrix() to scale the surface. Update the sourceRectHint whenever the video view's position changes (e.g., after layout changes).

Lifecycle-Aware Components (Q45–Q50)

Q45. How does LifecycleRegistry dispatch events to observers? Explain the sync mechanism.

LifecycleRegistry maintains a FastSafeIterableMap<LifecycleObserver, ObserverWithState> where each entry tracks the observer and its current state. When handleLifecycleEvent(event) is called, it sets the new state and calls sync(). The sync algorithm ensures all observers reach the target state by iterating in two passes: backward sync (if moving to a lower state) iterates from newest to oldest observer, calling dispatchEvent() to move each down; forward sync (if moving to a higher state) iterates from oldest to newest. This ordering guarantees that newer observers (added later, typically children) are destroyed before older ones (parents), and created after them — matching the intuitive parent-before-child lifecycle. The mHandlingEvent flag prevents re-entrant dispatching. If an observer is added during dispatch, it's moved to the current target state after the current sync completes. The mNewEventOccurred flag handles the case where a new event arrives during dispatch — it restarts the sync loop.

Q46. How does ProcessLifecycleOwner detect app foreground/background? Why the 700ms delay?

ProcessLifecycleOwner uses ActivityLifecycleCallbacks registered on the Application to track all Activities. It counts started/resumed Activities: when the first Activity starts, it dispatches ON_START; when the last Activity stops, it dispatches ON_STOP. The 700ms delay on ON_PAUSE/ON_STOP exists to handle Activity transitions. When navigating from Activity A to Activity B, there's a brief moment where A is paused/stopped but B hasn't started yet. Without the delay, the process would appear to go "background" during every Activity transition. The 700ms window (defined as TIMEOUT_MS in ProcessLifecycleOwner) allows B's onStart()/onResume() to fire and cancel the pending background event. If no Activity starts within 700ms, the process is genuinely in the background. This means ON_STOP fires ~700ms after the last Activity actually stops. ON_DESTROY is never dispatched — the process lifecycle has no destroy event because process death is not observable from within the process.

Q47. Explain ViewModelProvider resolution with CreationExtras — how does the modern factory pattern work?

The modern ViewModel creation (lifecycle-viewmodel 2.5+) uses CreationExtras — a type-safe map of dependencies passed to the factory. When you call ViewModelProvider(owner)[MyViewModel::class], the provider calls factory.create(modelClass, extras). The extras are populated by the ViewModelStoreOwner: ComponentActivity provides APPLICATION_KEY, SAVED_STATE_REGISTRY_OWNER_KEY, VIEW_MODEL_STORE_OWNER_KEY, and DEFAULT_ARGS_KEY (from Intent extras). This replaces the old AbstractSavedStateViewModelFactory approach. Custom factories use:

class MyViewModel(
    private val repo: Repository,
    private val savedStateHandle: SavedStateHandle
) : ViewModel() {
    companion object {
        val Factory: ViewModelProvider.Factory = viewModelFactory {
            initializer {
                val app = this[APPLICATION_KEY] as MyApp
                val handle = createSavedStateHandle()
                MyViewModel(app.repository, handle)
            }
        }
    }
}

createSavedStateHandle() is an extension on CreationExtras that extracts the SavedStateRegistryOwner and default args to construct the handle. Hilt bypasses this entirely with its own HiltViewModelFactory that uses Dagger's dependency graph.

Q48. LiveData postValue() race condition — what happens when you call it multiple times rapidly?

postValue() is designed for background threads — it posts a Runnable to the main thread's Handler. Internally, it stores the value in a mPendingData volatile field and posts a Runnable that calls setValue(mPendingData). The critical detail: if you call postValue() multiple times before the Runnable executes, each call overwrites mPendingData, but only one Runnable is posted (guarded by a flag). This means intermediate values are lost — only the last value is dispatched to observers. Example: postValue("A") then postValue("B") rapidly — observers only see "B". This is by design (not a bug) to coalesce rapid updates, but it can cause issues if you need every value (e.g., event-style usage). Solutions: (1) use setValue() from the main thread if you need every emission, (2) use Channel or SharedFlow for event streams, (3) use StateFlow which has the same "latest value wins" semantics but is more explicit about it.

Q49. How does collectAsStateWithLifecycle() work under the hood?

collectAsStateWithLifecycle() (from lifecycle-runtime-compose) is a Compose function that collects a Flow into a Compose State while respecting the lifecycle. Internally, it calls produceState() with a lifecycle-aware collector:

// Simplified internal implementation
@Composable
fun <T> Flow<T>.collectAsStateWithLifecycle(
    initialValue: T,
    lifecycle: Lifecycle,
    minActiveState: Lifecycle.State = Lifecycle.State.STARTED
): State<T> {
    val state = remember { mutableStateOf(initialValue) }
    LaunchedEffect(this, lifecycle, minActiveState) {
        lifecycle.repeatOnLifecycle(minActiveState) {
            this@collectAsStateWithLifecycle.collect {
                state.value = it
            }
        }
    }
    return state
}

The key is repeatOnLifecycle(STARTED): it starts collection when the lifecycle reaches STARTED and cancels the collection coroutine when it drops below STARTED (i.e., on STOPPED). When the lifecycle returns to STARTED, a new collection coroutine is launched. This prevents wasted work (processing Flow emissions while the UI is invisible) and avoids crashes from updating a non-visible UI. It uses the viewLifecycleOwner's lifecycle in Fragments, ensuring collection stops when the Fragment's view is destroyed.

Q50. Explain observer catch-up: what happens when you add a LifecycleObserver in STARTED state?

When you add a LifecycleObserver to a LifecycleRegistry that's already in STARTED state, the observer doesn't just receive the current state — it receives all intermediate events to catch up. The observer starts at INITIALIZED and the registry dispatches: ON_CREATE (→ CREATED), then ON_START (→ STARTED). If the lifecycle is in RESUMED, it additionally dispatches ON_RESUME. This catch-up happens synchronously in addObserver() via a while loop that compares the observer's state with the target state and dispatches events until they match. This is why DefaultLifecycleObserver.onCreate() fires even if you register the observer in onResume() — the observer catches up through all prior states. The implication: observers must be idempotent for lifecycle events they might receive during catch-up. If your ON_CREATE handler allocates a resource, it will allocate it even when added late. This catch-up mechanism is also why LiveData delivers the current value to new observers — it observes the lifecycle and receives the catch-up events.

Navigation (Q51–Q58)

Q51. How does NavController manage the back stack internally? Explain NavBackStackEntry.

NavController maintains its own back stack as a ArrayDeque<NavBackStackEntry> (in mBackStack). Each NavBackStackEntry represents a destination on the stack and is a LifecycleOwner, ViewModelStoreOwner, and SavedStateRegistryOwner. When you call navigate(), the controller creates a new NavBackStackEntry, pushes it onto its back stack, and delegates to the appropriate Navigator (e.g., FragmentNavigator, ComposeNavigator) to perform the actual UI transition. The entry's lifecycle is managed by NavController: it moves to RESUMED when the destination is visible and focused, STARTED when visible but not focused (e.g., a dialog destination is on top), and CREATED when on the back stack but not visible. On popBackStack(), the entry's lifecycle moves to DESTROYED and its ViewModelStore is cleared. The entry also holds a SavedStateHandle for passing results back and arguments Bundle for navigation arguments. This dual back stack (NavController's + Navigator's) is the source of many subtle bugs when they get out of sync.

Q52. Explain multiple back stacks in Navigation 2.4+ — how do saveBackStack() and restoreBackStack() work?

Multiple back stacks enable bottom navigation tabs to each maintain their own navigation history. When the user switches tabs, the current tab's back stack is saved and the new tab's is restored. Internally, NavController.saveBackStack(route) pops all entries above (and including) the specified route, saves their state (arguments, SavedStateHandle, ViewModel state) into an internal Map<String, ArrayDeque<NavBackStackEntryState>>, and tells the Navigator to save its state (e.g., FragmentNavigator calls FragmentManager.saveBackStack()). restoreBackStack(route) retrieves the saved state and recreates the entries. The Fragment-based implementation uses FragmentManager.saveBackStack(name)/restoreBackStack(name) (added in Fragment 1.4.0), which saves Fragment instances, view state, and ViewModel state as a unit. Usage with bottom nav:

NavigationBarItem(
    selected = currentRoute == tab.route,
    onClick = {
        navController.navigate(tab.route) {
            popUpTo(navController.graph.findStartDestination().id) {
                saveState = true
            }
            launchSingleTop = true
            restoreState = true
        }
    }
)

The saveState/restoreState flags in NavOptions trigger the save/restore mechanism.

Q53. How does ViewModel scoping work with Navigation? Destination-scoped vs graph-scoped.

Each NavBackStackEntry is a ViewModelStoreOwner, so ViewModels can be scoped to a navigation destination. Destination-scoped: calling viewModel() or hiltViewModel() inside a composable destination creates a ViewModel tied to that entry's ViewModelStore. It's cleared when the destination is popped from the back stack. Graph-scoped: you can scope a ViewModel to a navigation graph (nested or root) by passing the graph's back stack entry as the owner:

// In a composable destination
val sharedViewModel: SharedViewModel = hiltViewModel(
    navController.getBackStackEntry("nested_graph_route")
)
// Or in Fragment:
val sharedVm: SharedViewModel by navGraphViewModels(R.id.nested_graph)

Graph-scoped ViewModels live as long as the graph is on the back stack — they're shared across all destinations within that graph. This is the recommended pattern for sharing state between screens (e.g., a multi-step wizard). The ViewModel is cleared when the entire graph is popped. Internally, NavController creates a NavBackStackEntry for the graph itself (not just leaf destinations), and this entry owns the shared ViewModelStore.

Q54. Explain type-safe navigation in Compose (Navigation 2.8+) — how are @Serializable routes resolved?

Navigation 2.8+ replaces string-based routes with Kotlin @Serializable classes and objects. You define routes as data classes:

@Serializable data class Detail(val id: String)
@Serializable data object Home
@Serializable data object Settings

NavHost(navController, startDestination = Home) {
    composable<Home> { HomeScreen() }
    composable<Detail> { backStackEntry ->
        val detail: Detail = backStackEntry.toRoute()
        DetailScreen(detail.id)
    }
}

Under the hood, the library uses kotlinx.serialization to generate a route pattern from the class. Detail(id = "123") serializes to a URI-like string "Detail/123". The composable<T> function extracts the serializer for T, generates the route pattern with placeholders (e.g., "Detail/{id}"), and registers NavArguments with the correct types. When you call navController.navigate(Detail(id = "123")), the route object is serialized to the full route string and matched against registered patterns. backStackEntry.toRoute<Detail>() deserializes the arguments back into the data class. This provides compile-time safety — no more typos in route strings or argument key mismatches. Nullable parameters become optional query parameters, and default values are supported via Kotlin defaults.

Q55. How does ComposeNavigator differ from FragmentNavigator?

FragmentNavigator delegates to FragmentManager for transactions: navigate() calls FragmentManager.commit { replace(containerId, fragment) }, and the Fragment's lifecycle is managed by the FM. Back stack operations use FragmentManager.popBackStack(). Each destination is a real Fragment instance with its own view hierarchy. ComposeNavigator is fundamentally different — it doesn't use Fragments or Views at all. It maintains a list of NavBackStackEntry objects and exposes them to the NavHost composable, which renders the current entry's content composable. Navigation is a state change: updating the back stack triggers recomposition of NavHost, which swaps the displayed composable. There's no FragmentManager, no Fragment transactions, no view inflation. Animations are handled by Compose's animation APIs (e code>AnimatedContent inside AnimatedNavHost). The lifecycle of each entry is managed directly by NavController. This makes Compose navigation lighter weight but means you lose Fragment-specific features like setMaxLifecycle() and FragmentManager-based state saving (replaced by rememberSaveable).

Q56. What is the double-navigation crash and how do you prevent it in both Fragment and Compose navigation?

The double-navigation crash occurs when navigate() is called twice rapidly (e.g., user double-taps a button), and the second call fails because the current destination has already changed. In Fragment navigation, the second navigate() tries to find an action from a destination that's no longer current, throwing IllegalArgumentException("Navigation action/destination cannot be found"). In Compose, it can cause duplicate entries or similar crashes. Prevention strategies:

// 1. Check current destination before navigating
fun NavController.navigateSafe(route: String) {
    currentDestination?.let { current ->
        if (current.route != route) navigate(route)
    }
}

// 2. Debounce clicks (Compose)
@Composable
fun SingleClickButton(onClick: () -> Unit, content: @Composable () -> Unit) {
    var lastClickTime by remember { mutableLongStateOf(0L) }
    Button(onClick = {
        val now = System.currentTimeMillis()
        if (now - lastClickTime > 300) {
            lastClickTime = now
            onClick()
        }
    }) { content() }
}

// 3. Use Lifecycle-aware navigation (best approach)
fun NavController.navigateOnce(route: String) {
    if (currentBackStackEntry?.lifecycle?.currentState == Lifecycle.State.RESUMED) {
        navigate(route)
    }
}

Option 3 is the most robust: it only navigates if the current entry is RESUMED, which means no transition is in progress. During a transition, the entry is in STARTED state, so the check prevents the second navigation.

Q57. How do you structure navigation in a 100+ destination multi-module app?

At scale, navigation architecture requires careful modularization. The recommended approach:

  • Feature modules own their nav graphs: Each feature module defines a nested NavGraph with its internal destinations. The module exposes only its entry-point route (e.g., ProfileGraph) and hides internal routes.
  • Navigation API module: A shared module (e.g., :core:navigation) contains all route definitions (@Serializable classes) so any module can reference any route without depending on the feature module directly.
  • App module composes graphs: The app module's NavHost includes all feature nav graphs. Each feature provides an extension function on NavGraphBuilder:
// :feature:profile module
fun NavGraphBuilder.profileGraph(navController: NavController) {
    navigation<ProfileGraph>(startDestination = ProfileHome) {
        composable<ProfileHome> { ... }
        composable<EditProfile> { ... }
        composable<ProfileSettings> { ... }
    }
}

// :app module
NavHost(navController, startDestination = HomeGraph) {
    homeGraph(navController)
    profileGraph(navController)
    settingsGraph(navController)
    // ... 10+ feature graphs
}
  • Cross-module navigation: Features navigate to other features' entry points via routes from the shared navigation module. No feature depends on another feature's implementation.
  • Graph-scoped ViewModels: Shared state within a feature flow uses graph-scoped ViewModels. Cross-feature state uses Activity-scoped ViewModels or a shared data layer.
Q58. Explain shared element transitions in Compose Navigation — how does SharedTransitionLayout work?

SharedTransitionLayout (from animation:animation and integrated with Navigation Compose) coordinates shared element animations between navigation destinations. It works by matching elements across screens using a shared key:

SharedTransitionLayout {
    AnimatedNavHost(navController, startDestination = "list") {
        composable("list") {
            LazyColumn {
                items(items) { item ->
                    Image(
                        modifier = Modifier.sharedElement(
                            rememberSharedContentState(key = "image-${item.id}"),
                            animatedVisibilityScope = this@composable
                        )
                    )
                }
            }
        }
        composable("detail/{id}") { entry ->
            val id = entry.arguments?.getString("id")
            Image(
                modifier = Modifier.sharedElement(
                    rememberSharedContentState(key = "image-$id"),
                    animatedVisibilityScope = this@composable
                )
            )
        }
    }
}

Internally, SharedTransitionLayout creates an overlay layer above both the entering and exiting content. When a transition starts, it identifies matched elements (same key) in both screens, captures their bounds, and animates a single element from the source bounds to the target bounds on the overlay layer. The original elements in both screens are made invisible during the transition. animatedVisibilityScope ties the animation to the navigation transition's lifecycle. The system uses LookaheadScope to pre-calculate target positions before the animation starts, enabling smooth interpolation. Unlike the old View-based shared element transitions (which used Window overlay and TransitionManager), Compose's implementation is purely compositional — no window-level hacks.

System Design & Staff-Level Scenarios (Q59–Q65)

Q59. Design the lifecycle management for a video calling app that supports PiP, multi-window, and background audio.

This requires careful layering of lifecycle-aware components:

  • Video rendering: Bind to onTopResumedActivityChanged() for camera acquisition. In multi-window, both participants see video (both RESUMED), but only the top-resumed Activity holds the local camera. Use CameraDevice.StateCallback.onDisconnected() to gracefully handle camera loss.
  • PiP mode: When entering PiP, hide call controls, show only the remote video feed. Use setAutoEnterEnabled(true) so swiping home enters PiP automatically. Set sourceRectHint to the remote video surface bounds. Add PiP actions (mute, end call) via PictureInPictureParams.Builder.setActions().
  • Background audio: Use a foreground Service with FOREGROUND_SERVICE_TYPE_MICROPHONE | FOREGROUND_SERVICE_TYPE_CAMERA to keep audio/video processing alive when the Activity is stopped. The Service holds the WebRTC peer connection. The Activity binds to the Service and receives video frames.
  • State management: Call state (participants, mute status, duration) lives in a ViewModel scoped to the Activity. The Service holds the actual media session. Use ProcessLifecycleOwner to detect app background and switch to audio-only mode to save battery.
  • Process death: Save call ID in SavedStateHandle. On restoration, reconnect to the call via the Service (which may still be running if the process was killed and restarted for the Service).
Q60. You're seeing ANRs caused by Fragment transactions on the main thread in a large app — how do you investigate and fix?

Investigation steps:

  • Analyze ANR traces: Pull /data/anr/traces.txt or use ApplicationExitInfo.getTraceInputStream() (API 30+). Look for the main thread blocked in FragmentManager.execPendingActions() or moveToState().
  • Identify the bottleneck: Common causes: (1) Fragment onCreateView() inflating complex layouts synchronously — profile with Systrace/Perfetto, look for long inflate slices. (2) commitNow() triggering synchronous execution of expensive transactions. (3) Multiple commit() calls batching into a single expensive execution. (4) FragmentFactory.instantiate() doing heavy work (DI initialization).
  • Fixes: (1) Use ViewStub or lazy inflation for complex views. (2) Replace commitNow() with commit() where possible to spread work across frames. (3) Use postponeEnterTransition() to defer heavy work until data is ready. (4) Move to Compose for destinations with complex UI — composable inflation is cheaper than XML inflation. (5) Use FragmentManager.setStrictMode() in debug to catch expensive operations. (6) Profile with FragmentStrictMode.Policy.Builder().detectFragmentReuse().detectSetUserVisibleHint().penaltyLog().build().
  • Systemic fix: Migrate to single-Activity + Navigation Component to reduce the number of concurrent Fragment transactions. Use AsyncLayoutInflater for the heaviest layouts.
Q61. Design a state restoration strategy for a complex form (20+ fields) that survives both config change and process death.

Layer the state across the restoration pyramid:

@HiltViewModel
class FormViewModel @Inject constructor(
    private val savedStateHandle: SavedStateHandle,
    private val formRepository: FormRepository
) : ViewModel() {
    // Each field backed by SavedStateHandle (survives process death)
    val name = savedStateHandle.getStateFlow("name", "")
    val email = savedStateHandle.getStateFlow("email", "")
    // ... 20+ fields

    // Validation state in ViewModel memory (survives config change only)
    private val _validationErrors = MutableStateFlow<Map<String, String>>(emptyMap())
    val validationErrors = _validationErrors.asStateFlow()

    fun updateField(key: String, value: String) {
        savedStateHandle[key] = value
        validateField(key, value)
        // Auto-save draft to persistent storage
        draftSaveJob?.cancel()
        draftSaveJob = viewModelScope.launch {
            delay(500) // Debounce
            formRepository.saveDraft(collectFormData())
        }
    }

    // Restore from persistent storage on fresh start
    init {
        if (savedStateHandle.get<String>("name").isNullOrEmpty()) {
            viewModelScope.launch {
                formRepository.loadDraft()?.let { draft ->
                    draft.fields.forEach { (k, v) -> savedStateHandle[k] = v }
                }
            }
        }
    }
}

Key decisions: (1) SavedStateHandle for all field values — survives process death, auto-saved. (2) Debounced draft persistence to Room/DataStore — survives app uninstall/clear data scenarios and provides cross-device sync potential. (3) Validation state in plain ViewModel memory — cheap to recompute, no need to persist. (4) Keep total SavedStateHandle size small — 20 string fields is well under the Bundle limit. (5) Use getStateFlow() for reactive UI updates in Compose.

Q62. How would you migrate a 50-Activity app to single-Activity + Navigation Component architecture?

This is a multi-quarter migration that must be done incrementally:

  • Phase 1 — Audit: Map all Activities, their launch modes, deep links, task affinities, and inter-Activity communication patterns (startActivityForResult, Intent extras). Identify Activities that MUST remain separate (different tasks, process isolation, third-party SDK requirements).
  • Phase 2 — Scaffold: Create a single host Activity with NavHostFragment. Define the nav graph with all destinations. Create Fragment equivalents for each Activity, initially just wrapping the existing Activity's layout and logic.
  • Phase 3 — Incremental migration: Migrate one flow at a time (e.g., onboarding, settings). Convert each Activity to a Fragment, replace startActivity() calls with navController.navigate(), replace startActivityForResult() with Navigation's SavedStateHandle result pattern, replace Intent extras with Safe Args / serializable routes.
  • Phase 4 — Deep links: Migrate deep link handling from Activity intent-filters to Navigation's <deepLink> declarations. Use NavDeepLinkBuilder for notification PendingIntents.
  • Phase 5 — Cleanup: Remove migrated Activities from the manifest. Consolidate shared UI (toolbar, bottom nav) into the host Activity. Migrate ViewModel scoping from Activity-scoped to destination/graph-scoped.

Key risks: launch mode behavior changes (singleTask Activities become singleTop destinations), back stack differences, and third-party SDKs that expect Activity contexts. Keep a compatibility layer that can launch legacy Activities during the transition.

Q63. Design a navigation architecture for a super-app with 10 feature modules, each with its own nav graph.

Architecture layers:

  • :core:navigation module: Contains all route definitions as @Serializable classes/objects. Every feature module depends on this. No feature depends on another feature.
  • Feature modules: Each exposes a NavGraphBuilder extension function (e.g., fun NavGraphBuilder.paymentGraph(navController: NavController)). Internal destinations are private to the module. Each feature has its own nested NavGraph with a defined start destination.
  • :app module: Composes all feature graphs into the root NavHost. Manages bottom navigation with multiple back stacks (saveState/restoreState).
  • Cross-module navigation: Feature A navigates to Feature B's entry point via routes from :core:navigation. The NavController is passed down, not stored globally.
  • Deep links: Each feature declares its deep links in its nav graph. The app module's Activity intent-filter catches all deep link schemes and NavController handles routing.
  • Shared ViewModels: Use Activity-scoped ViewModels for truly global state (auth, user profile). Use graph-scoped ViewModels for multi-screen flows within a feature. Never use destination-scoped ViewModels for cross-destination sharing.
  • Dynamic feature modules: Use Navigation's DynamicNavHostFragment or conditional graph inclusion. When a feature isn't installed, navigate to a "download module" placeholder that triggers on-demand delivery via Play Feature Delivery.

Testing: Each feature module has its own TestNavHostController tests for internal navigation. Integration tests in :app verify cross-module navigation.

Q64. Explain how you'd debug a ViewModel that isn't being cleared when expected, causing memory leaks.

Systematic debugging approach:

  • Step 1 — Verify the leak: Use LeakCanary or Android Studio Memory Profiler. Capture a heap dump after the expected clear point. Search for the ViewModel class — if instances exist with a GC root path, it's leaked.
  • Step 2 — Check the ViewModelStoreOwner: The ViewModel is cleared when its owner's ViewModelStore.clear() is called. For Activity-scoped VMs, this happens in ComponentActivity.onDestroy() when isChangingConfigurations() is false. For Fragment-scoped VMs, in FragmentManagerViewModel.clearNonConfigState() when the Fragment is destroyed (not just its view). For Navigation-scoped VMs, when the NavBackStackEntry is popped.
  • Step 3 — Common causes: (1) ViewModel scoped to Activity but should be scoped to Fragment/NavBackStackEntry — it lives too long. (2) Navigation: using activityViewModels() instead of navGraphViewModels() — the VM lives for the entire Activity instead of the nav graph. (3) Fragment on back stack: the Fragment instance is alive (and its VM), but the developer expected it to be destroyed. (4) ViewModelStore retained via NonConfigurationInstances across config changes — this is expected, not a leak.
  • Step 4 — Check ViewModel internals: The VM itself may hold references that prevent cleanup: a coroutine in viewModelScope that never completes (holding a reference to a repository that holds a Context), a LiveData observer registered without a lifecycle owner, or a callback registered with a system service.
  • Fix: Scope VMs correctly, cancel long-running work in onCleared(), use viewModelScope (auto-cancelled), and avoid storing Context references in VMs.
Q65. A user reports that your app loses all state when they switch to another app and come back — walk through your investigation.

This is a classic process death issue. Investigation steps:

  • Step 1 — Confirm process death: Check ApplicationExitInfo via ActivityManager.getHistoricalProcessExitReasons(). Look for REASON_LOW_MEMORY or REASON_OTHER. Also check logcat for ActivityManager: Killing <pid> messages.
  • Step 2 — Check saved state: Add logging in onCreate() to check if savedInstanceState is null. If it's null after process death, the state wasn't saved. If it's non-null but the app still shows fresh state, the restoration logic is broken.
  • Step 3 — Audit state storage: Identify what state is lost. Common culprits: (1) State stored only in ViewModel without SavedStateHandle — lost on process death. (2) State in static/singleton objects (DI graph, in-memory caches) — lost on process death. (3) Fragment arguments not properly saved. (4) Custom Views without onSaveInstanceState()/onRestoreInstanceState() implementations.
  • Step 4 — Reproduce: Use adb shell am kill <package> (not "Don't keep activities") to simulate real process death. Verify with Strict Mode or a debug build that logs all save/restore calls.
  • Step 5 — Fix: Move critical UI state to SavedStateHandle. Ensure ViewModels re-fetch data from the persistent layer (Room/DataStore) on initialization, not just from in-memory caches. Use rememberSaveable in Compose. Test every screen with the process death flow as part of CI (using Macrobenchmark or custom instrumentation tests).

Additional L6 Corner Cases & Edge Cases (Q66–Q75)

Q66. How does handleDeepLink() interact with an existing back stack? When would you avoid using it?

NavController.handleDeepLink() clears the existing back stack and builds a synthetic one from the start destination to the deep-linked destination. If the user was at Screen A → B → C and a deep link to D arrives, the result is StartDest → D — screens A, B, C are destroyed. This is correct for cold-start deep links (app launched from a notification) but destructive for in-app deep links where you want to preserve context. For the latter, use navController.navigate() directly with the resolved destination and arguments, which pushes onto the existing back stack. You can resolve the deep link manually: navController.graph.matchDeepLink(NavDeepLinkRequest.Builder.fromUri(uri).build()) returns the matched destination and arguments without clearing anything.

Q67. Implement an auth gate pattern in Compose Navigation that redirects unauthenticated users to login while preserving their intended destination.

The key challenge is intercepting navigation before the destination composable renders. The cleanest pattern: use a splash/loading screen as the start destination that checks auth state and navigates accordingly. If the user deep-links to /profile/123 while logged out: (1) NavController processes the deep link and builds the synthetic back stack. (2) The profile destination's composable checks isLoggedIn. (3) If false, a LaunchedEffect navigates to the login screen, passing the intended route as an argument. (4) After login, navigate to the saved route with popUpTo(loginRoute) { inclusive = true }. The critical detail: don't use startDestination = if (loggedIn) home else login because the start destination is baked into the nav graph at creation time and won't update reactively. Instead, always start at a splash destination that observes auth state via collectAsStateWithLifecycle() and navigates imperatively.

Q68. How does ComposeNavigator persist and restore state compared to FragmentNavigator? What survives process death?

FragmentNavigator delegates state persistence to FragmentManager, which saves Fragment state (arguments, view hierarchy, onSaveInstanceState) into the Activity's Bundle. ComposeNavigator takes a different approach: each NavBackStackEntry has its own SavedStateRegistry, and rememberSaveable values are stored there. On process death, NavController serializes the entire back stack (destination IDs, arguments, saved state bundles) into the Activity's savedInstanceState. On restoration, it recreates NavBackStackEntry objects with the saved state, and when the composable re-executes, rememberSaveable reads from the restored registry. ViewModels are not restored (they don't survive process death), but SavedStateHandle values are (they're part of the saved state bundle). The key difference: Fragment state includes the view hierarchy state automatically; Compose state only includes explicitly saved values (rememberSaveable, SavedStateHandle).

Q69. How does Navigation Component integrate with the predictive back gesture (Android 13+)?

Navigation 2.7+ supports predictive back natively. NavHostFragment registers an OnBackPressedCallback with the Activity's OnBackPressedDispatcher. When the user starts a back gesture, the system calls handleOnBackStarted(), which begins the back animation (the current Fragment starts scaling/fading). As the gesture progresses, handleOnBackProgressed(backEvent) updates the animation based on backEvent.progress (0.0→1.0). If the gesture completes, handleOnBackPressed() commits the pop. If cancelled, handleOnBackCancelled() reverses the animation. In Compose Navigation, AnimatedNavHost uses popExitTransition and popEnterTransition driven by the gesture progress. The manifest must declare android:enableOnBackInvokedCallback="true". Without this, the legacy onBackPressed() path is used and no predictive animation occurs. For custom back handling, use PredictiveBackHandler in Compose or override handleOnBackStarted/Progressed/Cancelled in OnBackPressedCallback.

Q70. How do you handle accessibility (TalkBack, focus management) across navigation transitions?

When navigating to a new destination, TalkBack should announce the new screen and focus should move to the primary content. NavigationUI handles toolbar title updates automatically, which TalkBack reads. For custom announcements, add an OnDestinationChangedListener that calls window.decorView.announceForAccessibility(destination.label). Focus management: FragmentNavigator requests focus on the new Fragment's root view after the transaction, which causes TalkBack to start reading from the top. In Compose, use Modifier.semantics { contentDescription = "Screen title" } on the root composable and FocusRequester to direct focus. Common pitfalls: (1) shared element transitions can confuse TalkBack because the focused element moves between destinations. (2) Bottom sheet destinations may not receive focus automatically — use Modifier.focusRequester() with LaunchedEffect. (3) Back navigation should restore focus to the element that triggered the forward navigation, which Navigation handles for Fragments but requires manual work in Compose.

Q71. Design navigation for an app with dynamic feature modules (on-demand delivery via Play Feature Delivery).

Architecture: (1) :core:navigation module defines all route interfaces/sealed classes — no implementation, just contracts. Every feature module depends on this. (2) Each feature module defines its own NavGraphBuilder extension function (e.g., fun NavGraphBuilder.checkoutGraph(navController: NavController)) that registers its destinations. (3) The :app module assembles the full nav graph by calling each feature's extension. (4) For dynamic features not yet installed, use DynamicNavHostFragment or a placeholder destination that triggers SplitInstallManager.startInstall(). After installation, navigate to the real destination. (5) Handle installation failure gracefully — show a retry screen, not a crash. (6) Route versioning: since dynamic modules update independently, use stable route contracts in :core:navigation and avoid breaking changes. (7) Testing: each feature module tests its nav graph in isolation with TestNavHostController. Integration tests in :app verify cross-module navigation with all features installed.

Q72. What are the edge cases of type-safe @Serializable routes in Navigation 2.8+? How do you handle route schema changes?

Edge cases: (1) object vs data class: @Serializable object HomeRoute produces route string "HomeRoute" (no parameters). data class ProfileRoute(val id: String) produces "ProfileRoute/{id}". (2) Default values: parameters with defaults become query parameters (optional), not path parameters. data class SearchRoute(val query: String = "") → "SearchRoute?query={query}". (3) Enums: serialized as their name string. (4) Malformed deep links: if a deep link provides a non-numeric value for an Int parameter, toRoute<T>() throws IllegalArgumentException — always wrap in try-catch. (5) Schema changes: if you rename a parameter or change its type, existing deep link bookmarks and pending intents break. Mitigation: register multiple deepLinks patterns (old and new) on the destination, and use a migration function to map old parameters to new ones. (6) Nullable parameters: use val id: String? = null — serialized as an optional query parameter. Null values are omitted from the route string entirely.

Q73. Explain the AMS vs ATMS split in Android 10. Why was it done and what are the implications?

Before Android 10, ActivityManagerService was a monolithic class (20,000+ lines) handling Activity lifecycle, process management, broadcast dispatch, content providers, and more. Android 10 extracted Activity/Task management into ActivityTaskManagerService (ATMS). AMS retained process lifecycle, broadcasts, and providers. The split was motivated by: (1) Multi-display support: ATMS manages per-display RootWindowContainer → DisplayContent → TaskDisplayArea hierarchies, which was impractical in the monolithic AMS. (2) Code maintainability: separating concerns made both services easier to reason about. (3) Testability: smaller, focused services are easier to unit test. Both services run in system_server. ATMS exposes ActivityTaskManagerInternal for AMS to query Activity state (e.g., determining process importance based on Activity visibility). For L6, the key implication is that startActivity() calls now go through ATMS, while getRunningAppProcesses() and forceStopPackage() still go through AMS.

Q74. How does TaskFragment (Android 12L) enable Activity Embedding, and what are the lifecycle implications?

TaskFragment subdivides a Task into regions, each hosting one or more Activities. This enables two Activities to display side-by-side within the same Task (e.g., list + detail on a tablet). Jetpack WindowManager's Activity Embedding API uses SplitPairRule to define which Activity pairs should split. When a split is active: (1) Both Activities are in the same Task (same taskId). (2) Each Activity has its own lifecycle — both can be RESUMED simultaneously (like multi-window). (3) Back press pops from the focused TaskFragment. (4) If the window narrows (foldable fold, window resize), the system collapses the split — the secondary Activity stacks on top of the primary. (5) SplitRule.FinishBehavior controls what happens when one Activity finishes: NEVER (other stays), ALWAYS (both finish), ADJACENT (finish only if in same split). The key L6 insight: TaskFragment is the foundation for large-screen support without rewriting apps to use Fragments or Compose — existing multi-Activity apps get split-screen behavior through manifest rules and the Jetpack library.

Q75. Design a migration strategy from Fragment Navigation to Compose Navigation for a large production app (50+ screens).

Phase 1 — Coexistence: Keep the existing NavHostFragment and nav graph. Convert individual Fragment destinations to Compose by replacing the Fragment's layout with a ComposeView using ViewCompositionStrategy.DisposeOnViewTreeLifecycleDestroyed. The Fragment becomes a thin shell hosting Compose. This requires zero navigation changes. Phase 2 — Hybrid navigation: Add navigation-compose alongside navigation-fragment. New features are pure Compose destinations. Use a shared :core:navigation module with route definitions. The Fragment nav graph and Compose nav graph can coexist in the same NavController by registering both FragmentNavigator and ComposeNavigator. Phase 3 — Migration: Convert Fragment shells to pure Compose destinations one by one. Move the destination from the XML nav graph to the Kotlin DSL nav graph. Phase 4 — Cleanup: Remove NavHostFragment, switch to NavHost composable, remove Fragment dependencies. Key risks: (1) ViewModel scoping changes — Fragment-scoped VMs become destination-scoped. (2) Shared element transitions need reimplementation. (3) Deep link patterns must be preserved. (4) Testing: maintain both Fragment and Compose test suites during migration. Budget 6-12 months for a 50-screen app with a team of 4-6 engineers.
