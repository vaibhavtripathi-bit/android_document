03 — Configuration Changes & Process Death

The two destruction events every Android engineer must master — what triggers them, what survives each, and how the system restores your app from the ashes

1. Configuration Change Internals

What Triggers a Configuration Change

A configuration change occurs whenever the device's Configuration object changes in a way that affects the current Activity's resources. The system must recreate the Activity so it can re-inflate layouts and reload resources matching the new configuration.

The Configuration Object and AMS Detection

In system_server, ActivityTaskManagerService (ATMS) maintains the global Configuration. When a system event changes it, the flow is:

  System Event (e.g., rotation sensor)
       │
       ▼
  WindowManagerService.updateRotation()
       │
       ▼
  DisplayContent.updateDisplayInfo()
       │
       ▼
  ActivityTaskManagerService.updateGlobalConfigurationLocked()
       │  ← computes new Configuration, diffs against current
       ▼
  For each ActivityRecord in the affected Task:
       │
       ├── Does manifest declare android:configChanges matching ALL changed bits?
       │     YES → deliver onConfigurationChanged() — no recreate
       │     NO  → schedule Activity destroy + recreate
       │
       ▼
  ActivityRecord.ensureActivityConfiguration()
       │
       ▼
  relaunchActivityLocked() → sends RELAUNCH_ACTIVITY transaction to app process

The key AOSP class is ActivityRecord.ensureActivityConfiguration(). It computes the diff between the Activity's current Configuration and the new global one using Configuration.diff(), which returns a bitmask of changed fields. If the Activity's android:configChanges manifest attribute covers all changed bits, the system skips recreation.

android:configChanges — What It Actually Does

Declaring android:configChanges="orientation|screenSize" does not prevent the configuration from changing. It tells the system: "I'll handle this myself — don't destroy and recreate me." The Activity receives onConfigurationChanged(newConfig) instead.

⚠ Common L6 Trap

Declaring android:configChanges="orientation|screenSize" is almost always wrong for production apps. It means you are responsible for reloading every resource — layouts, drawables, strings, dimensions — that might differ for the new configuration. You'll also miss density, locale, uiMode (dark mode), and fontScale changes. The system's recreate mechanism exists precisely because manual handling is error-prone at scale.

Why is onSaveInstanceState() called before onDestroy()? Because the saved state Bundle is passed through the ActivityClientRecord in the app process and mirrored in ActivityRecord in system_server. The system needs this data before it can schedule the relaunch transaction. During config change, isChangingConfigurations() returns true — you can use this to skip expensive cleanup in onDestroy().

2. onSaveInstanceState Deep Dive

When It's Called

API LevelTimingRationale
Pre-28 (before P)Before onStop(), after onPause()Historical behavior; no guarantee of order relative to onStop()
28+ (P and later)After onStop()Guarantees the Activity is no longer visible when state is saved, preventing state loss from user interactions during the save window

The change in API 28 is significant: it means FragmentTransaction.commit() is safe to call in onStop() on API 28+ (the state hasn't been saved yet), but would cause state loss on pre-28 devices.

The Bundle Mechanism — Three Layers of Saving

When the system calls Activity.onSaveInstanceState(outState), three distinct save operations happen:

// Layer 1: View hierarchy auto-save
// Activity.onSaveInstanceState() calls:
//   window.saveHierarchyState(outState)
//     → iterates all Views with android:id
//     → calls View.onSaveInstanceState() on each
//     → stores in a SparseArray<Parcelable> keyed by view ID

// Layer 2: Fragment state save
// FragmentController.saveAllState() is called
//   → each Fragment's onSaveInstanceState() is invoked
//   → Fragment arguments, back stack, and view state are bundled

// Layer 3: Your custom state
// Whatever you put in the Bundle in your override:
override fun onSaveInstanceState(outState: Bundle) {
    super.onSaveInstanceState(outState) // triggers Layer 1 + 2
    outState.putInt("selectedTab", currentTabIndex)
    outState.putString("searchQuery", searchBar.text.toString())
}

What's Auto-Saved (and What's Not)

The View hierarchy auto-save only works for Views with an android:id. Each View type saves specific state:

ViewAuto-Saved StateRequires android:id
EditTextText content, cursor positionYes
ScrollViewScroll positionYes
CheckBox / SwitchChecked stateYes
RecyclerViewScroll position (via LayoutManager)Yes
TabLayoutSelected tab indexYes
TextViewNothing (text is static)—
ImageViewNothing (image source is static)—

Bundle Size Limits — TransactionTooLargeException

The saved state Bundle is transmitted via Binder IPC to system_server. Binder has a 1 MB per-process transaction buffer shared across all concurrent transactions. If the Bundle (plus any other in-flight Binder data) exceeds this, you get TransactionTooLargeException.

🔴 TransactionTooLargeException

This crash is not thrown at save time — it's thrown when the system tries to write the Bundle to the Binder buffer, which may happen asynchronously. On API 24+ it crashes immediately; on older versions the data is silently dropped and the Activity restores with a null savedInstanceState. The worst kind of bug: silent data loss.

// What should NOT go in savedInstanceState:
// ✗ Large bitmaps or byte arrays
// ✗ Full API response objects
// ✗ Lists of complex Parcelable objects (e.g., 500 items)
// ✗ Fragment arguments with large payloads

// What SHOULD go in savedInstanceState:
// ✓ IDs, keys, positions (Int, Long, String)
// ✓ Small enum/state values
// ✓ Search queries, filter selections
// ✓ Scroll positions, selected tab indices

Architecture

The Jetpack SavedStateRegistry is an abstraction over the platform Bundle mechanism that allows any component — not just Activities and Fragments — to participate in state saving and restoration.

  └─────────────────────────────────────────────────────┘

How It Wraps the Platform Bundle

During onSaveInstanceState(), SavedStateRegistryController.performSave(outBundle) iterates all registered SavedStateProvider instances, calls saveState() on each, and writes the results into the platform Bundle under a reserved key ("androidx.lifecycle.BundlableSavedStateRegistry.key").

// Registering a custom SavedStateProvider
class MyCustomComponent(registry: SavedStateRegistry) {

    init {
        registry.registerSavedStateProvider("my_component") {
            bundleOf(
                "scroll_position" to currentScrollY,
                "expanded_ids" to expandedItemIds.toIntArray()
            )
        }
    }

    fun restore(registry: SavedStateRegistry) {
        val bundle = registry.consumeRestoredStateForKey("my_component")
        bundle?.let {
            currentScrollY = it.getInt("scroll_position")
            expandedItemIds = it.getIntArray("expanded_ids")?.toSet() ?: emptySet()
        }
    }
}

Integration with SavedStateHandle

SavedStateHandle is the ViewModel-facing API built on top of SavedStateRegistry. When a ViewModel is created via SavedStateViewModelFactory, the factory registers a SavedStateProvider with the registry. The SavedStateHandle acts as a key-value map that automatically participates in the save/restore cycle.

class SearchViewModel(private val savedStateHandle: SavedStateHandle) : ViewModel() {

    // Survives both config change AND process death
    val query: StateFlow<String> = savedStateHandle.getStateFlow("query", "")

    fun updateQuery(newQuery: String) {
        savedStateHandle["query"] = newQuery
    }

    // Survives config change only (regular ViewModel field)
    private val _results = MutableStateFlow<List<Result>>(emptyList())
    val results: StateFlow<List<Result>> = _results

    fun search() {
        viewModelScope.launch {
            _results.value = repository.search(query.value)
        }
    }
}

4. ViewModel Retention Across Config Changes

The Underlying Mechanism: NonConfigurationInstances

ViewModel survival across config changes is not magic — it relies on a platform mechanism that predates Jetpack: Activity.onRetainNonConfigurationInstance().

  ViewModel Retention During Config Change
  ═════════════════════════════════════════

  OLD Activity                              NEW Activity
  ────────────                              ────────────
  onPause()
  onStop()
  onSaveInstanceState()
  onRetainNonConfigurationInstance()
    │  returns ViewModelStore ──────┐
  onDestroy()                       │
                                    │  ViewModelStore held by
                                    │  ActivityClientRecord
                                    │  in the app process
                                    │  (NOT sent to system_server)
                                    │
                                    └──▶ getLastNonConfigurationInstance()
                                         │
                                         ▼
                                    onCreate()
                                      ViewModelProvider checks for
                                      existing ViewModelStore
                                      → finds retained VMs
                                      → skips creation

The critical detail: ViewModelStore is retained in the app process memory via ActivityClientRecord.lastNonConfigurationInstances. It never crosses a Binder boundary. This is why ViewModels can hold large objects (unlike Bundles) — they're just Java objects on the heap.

🟣 Staff-Level Insight — Why ViewModels Don't Survive Process Death

NonConfigurationInstances are stored in the app process's ActivityThread.mActivities map. When the process dies, all heap memory is gone — including the ViewModelStore. The ActivityRecord in system_server survives (it's in a different process), and it holds the saved state Bundle, but it has no reference to the ViewModelStore. This is the fundamental architectural reason ViewModels survive config changes but not process death — they live in different processes.

ViewModelProvider and Factory

// Modern ViewModel creation with CreationExtras (Lifecycle 2.5+)
class MyViewModel(
    private val repository: Repository,
    private val savedStateHandle: SavedStateHandle
) : ViewModel() {

    companion object {
        val Factory: ViewModelProvider.Factory = viewModelFactory {
            initializer {
                val app = this[APPLICATION_KEY] as MyApplication
                val savedStateHandle = createSavedStateHandle()
                MyViewModel(app.repository, savedStateHandle)
            }
        }
    }
}

// In Activity:
val viewModel: MyViewModel by viewModels { MyViewModel.Factory }

// What happens internally:
// 1. viewModels {} checks ViewModelStoreOwner (the Activity)
// 2. ViewModelProvider looks in ViewModelStore for existing "MyViewModel" key
// 3. If found → return existing instance (config change case)
// 4. If not → call Factory.create() with CreationExtras
// 5. CreationExtras includes: APPLICATION_KEY, SAVED_STATE_REGISTRY_OWNER_KEY,
//    VIEW_MODEL_STORE_OWNER_KEY, DEFAULT_ARGS_KEY

🟣 Staff-Level Insight — ContentProvider Initialization Trap

After process death, ContentProvider.onCreate() runs before Activity.onCreate() and even before Application.onCreate() completes (ContentProviders are initialized during Application.attachBaseContext() → installContentProviders()). If your ContentProvider depends on DI initialization that happens in Application.onCreate(), you'll crash on process restoration. This is a common production bug. Libraries like Firebase and WorkManager use ContentProviders for auto-initialization, which is why they work without explicit Application.onCreate() calls — but it also means they run before your DI setup. The App Startup library (androidx.startup) was created specifically to solve initialization ordering.

Process Recreation Sequence

  1. Zygote fork — new process with preloaded framework classes (COW pages)
  2. Application.attachBaseContext() — earliest app code execution
  3. ContentProvider.onCreate() — all declared providers initialized
  4. Application.onCreate() — your DI setup, crash reporting init, etc.
  5. Activity.onCreate(savedInstanceState) — the Bundle from ActivityRecord is passed in
  6. Activity.onStart() → onRestoreInstanceState() → onResume()

6. Saved State Strategies at Scale

The State Restoration Pyramid

  State Restoration Pyramid — What Goes Where
  ════════════════════════════════════════════

              ┌───────────────┐
              │  Persistent   │  Room, DataStore, SharedPrefs
              │   Storage     │  User data, cached API responses
              │               │  Survives: everything (even uninstall for backup)
              ├───────────────┤
              │ SavedState    │  SavedStateHandle, onSaveInstanceState
              │  Handle /     │  UI state: scroll pos, search query, selected tab
              │  Bundle       │  Survives: config change + process death
              ├───────────────┤
              │  ViewModel    │  In-memory business state
              │  (in-memory)  │  API results, computed data, running operations
              │               │  Survives: config change only
              ├───────────────┤
              │  View /       │  Transient interaction state
              │  Compose      │  Animation progress, touch state, focus
              │  State        │  Survives: nothing (or config change via remember)
              └───────────────┘

  Rule of thumb: store the MINIMUM needed to reconstruct the screen.
  Save IDs and queries, not full data objects.

Compose rememberSaveable

rememberSaveable hooks into SavedStateRegistry under the hood. It uses the composition's SaveableStateRegistry (which delegates to the platform SavedStateRegistry) to persist values across both config changes and process death.

@Composable
fun SearchScreen(viewModel: SearchViewModel = viewModel()) {
    // Survives config change + process death (via SavedStateRegistry)
    var query by rememberSaveable { mutableStateOf("") }

    // Survives config change only (via ViewModel retention)
    val results by viewModel.results.collectAsStateWithLifecycle()

    // Survives nothing — reset on any recreation
    var isAnimating by remember { mutableStateOf(false) }

    // Best practice: use SavedStateHandle in ViewModel instead of
    // rememberSaveable for state that drives business logic
    // Use rememberSaveable for purely UI-level state
}

7. Staff-Level Corner Cases

Diagnosing TransactionTooLargeException

// The TooLargeTool approach — log Bundle sizes in debug builds
class DebugActivity : ComponentActivity() {

    override fun onSaveInstanceState(outState: Bundle) {
        super.onSaveInstanceState(outState)
        if (BuildConfig.DEBUG) {
            logBundleSize(outState, "Activity")
        }
    }

    private fun logBundleSize(bundle: Bundle, tag: String) {
        val parcel = android.os.Parcel.obtain()
        try {
            parcel.writeBundle(bundle)
            val sizeKb = parcel.dataSize() / 1024
            Log.d("BundleSize", "$tag: ${sizeKb}KB")
            if (sizeKb > 500) {
                Log.w("BundleSize", "$tag approaching limit! Keys:")
                bundle.keySet().forEach { key ->
                    val value = bundle.get(key)
                    val valueParcel = android.os.Parcel.obtain()
                    try {
                        valueParcel.writeValue(value)
                        Log.w("BundleSize",
                            "  $key: ${valueParcel.dataSize() / 1024}KB")
                    } finally {
                        valueParcel.recycle()
                    }
                }
            }
        } finally {
            parcel.recycle()
        }
    }
}

// Common culprits:
// 1. Fragment arguments with large Parcelable lists
// 2. NavController saving entire back stack with large args
// 3. RecyclerView adapter saving state for thousands of items
// 4. WebView saving full page state

ViewModel + SavedStateHandle During Background Work

class UploadViewModel(private val savedStateHandle: SavedStateHandle) : ViewModel() {

    // 1. viewModelScope is cancelled → coroutine is cancelled

            .build()

    }

    // On restoration, check WorkManager status instead of restarting
    fun checkPendingUpload() {
        val pendingUri = savedStateHandle.get<String>("pending_uri") ?: return
        // Observe WorkManager for the existing work status
    }
}
🟣 Staff-Level Insight — The isChangingConfigurations() Optimization

In onDestroy(), isChangingConfigurations() returns true if the Activity is being destroyed for a config change (and will be immediately recreated). Use this to avoid expensive cleanup that's pointless during recreation — closing database connections, unregistering system-level listeners, or releasing resources that the new Activity instance will immediately re-acquire. Conversely, if isFinishing() is true, the Activity is genuinely going away (user pressed back, finish() was called). The combination of these two flags gives you a complete picture: isFinishing() = user-initiated destruction, isChangingConfigurations() = system-initiated temporary destruction, neither = system killing the Activity for resources (process death path).

Multi-Process Apps

// If you declare android:process=":background" on a Service or Activity,
// it runs in a SEPARATE process. Each process:
//   - Has its own Application.onCreate() call
//   - Has its own static fields and singletons
//   - Has its own DI graph
//   - Has its own ViewModelStore (if it has Activities)
//
// Process death affects processes independently:
//   - Main process can die while :background lives on
//   - Each process has its own OOM adj score

//
// Common trap: assuming a singleton initialized in the main process

    scenario.recreate() // triggers full destroy → create cycle

    scenario.onActivity { activity ->

⚠ Don't Keep Activities ≠ Process Death

The "Don't keep activities" developer option destroys Activities when they go to the background, but the process stays alive. This means static fields, singletons, and in-memory caches survive. It's useful for testing onSaveInstanceState/onRestoreInstanceState paths, but it does not test the full process death scenario. For true process death testing, use adb shell am kill after backgrounding the app.

9. Configuration.diff() and Granular Change Detection

// When handling config changes manually, you can detect WHICH config changed:
override fun onConfigurationChanged(newConfig: Configuration) {
    super.onConfigurationChanged(newConfig)

    // Configuration.diff() returns a bitmask of what changed
    // (internal API, but the concept matters for L6)
    //
    // In practice, check specific fields:
    val isNightMode = newConfig.uiMode and Configuration.UI_MODE_NIGHT_MASK ==
        Configuration.UI_MODE_NIGHT_YES

    val isLandscape = newConfig.orientation == Configuration.ORIENTATION_LANDSCAPE

    // Per-app language (Android 13+):
    val currentLocale = newConfig.locales[0]
}

// android:configChanges gotcha — you must handle ALL declared configs:
// If you declare android:configChanges="orientation|screenSize"
// you STILL get destroyed for: locale, fontScale, uiMode (dark mode),
// density, keyboard, layoutDirection, etc.
//
// Common mistake: declaring orientation|screenSize and assuming
// the Activity never recreates. Then a user changes dark mode
// and the Activity recreates without proper handling.

10. PersistableBundle and Persistent Saved State

// Activities can opt into persistent saved state that survives REBOOTS:
// AndroidManifest.xml: android:persistableMode="persistAcrossReboots"

// This gives you a second callback:
override fun onSaveInstanceState(outState: Bundle, outPersistentState: PersistableBundle) {
    super.onSaveInstanceState(outState, outPersistentState)
    outPersistentState.putString("lastDocument", currentDocId)
}

override fun onCreate(savedInstanceState: Bundle?, persistentState: PersistableBundle?) {
    super.onCreate(savedInstanceState, persistentState)
    // persistentState survives device reboot
    // savedInstanceState survives process death but NOT reboot
    val lastDoc = persistentState?.getString("lastDocument")
}

// Limitations:
// - PersistableBundle only supports primitive types (no Parcelable)
// - Written to disk by system_server (XML in /data/system/recent_tasks/)
// - Rarely used in practice — DataStore/SharedPreferences are more flexible
// - But it's an L6 interview topic because it reveals understanding of
//   the state persistence hierarchy

11. Activity.recreate() vs Config Change

// Activity.recreate() programmatically triggers a destroy-recreate cycle
// It follows the SAME path as a config change:
// onPause → onStop → onSaveInstanceState → onDestroy → onCreate → ...

// Key difference from actual config change:
// - isChangingConfigurations() returns FALSE
// - ViewModel IS still retained (NonConfigurationInstances is preserved)
// - This is because recreate() internally uses the same
//   ActivityClientController.activityRelaunch() path

// Use case: applying a theme change at runtime
fun applyTheme(isDark: Boolean) {
    AppCompatDelegate.setDefaultNightMode(
        if (isDark) AppCompatDelegate.MODE_NIGHT_YES
        else AppCompatDelegate.MODE_NIGHT_NO

    // This triggers recreate() on all visible Activities
}
