1. Compose Navigation Architecture

The navigation-compose artifact is a thin Compose-specific layer on top of navigation-runtime. It does not replace the core NavController вҖ” it provides a ComposeNavigator that knows how to render composable destinations, replacing FragmentNavigator's role. The same NavBackStackEntry model, lifecycle ownership, and ViewModel scoping from fragment-based navigation carry over unchanged.

ComposeNavigator Internals

// ComposeNavigator extends Navigator<ComposeNavigator.Destination>
// Unlike FragmentNavigator, it does NOT perform transactions.
// Instead, it manages a list of NavBackStackEntry objects and
// exposes them as State for the NavHost composable to observe.

// When navigate() is called:
//   1. ComposeNavigator.navigate(entries, navOptions, extras)
//   2. Adds entries to its internal backStack StateFlow
//   3. NavHost observes this StateFlow via collectAsState()
//   4. Recomposition triggers вҶ’ new composable destination is rendered
//   5. Previous destination's composable leaves composition
//      (unless animating вҖ” see Section 6)

// Key difference from FragmentNavigator:
// - No FragmentTransaction, no FragmentManager involvement
// - Destination switching is purely a recomposition
// - The composable for a destination only exists in composition
//   while visible (or animating out) вҖ” there is no equivalent
//   of a Fragment instance surviving on the back stack

NavBackStackEntry in Compose вҖ” Triple Owner

// Each NavBackStackEntry implements:
//   LifecycleOwner          вҶ’ tracks CREATED вҶ’ STARTED вҶ’ RESUMED вҶ’ DESTROYED
//   ViewModelStoreOwner     вҶ’ ViewModels scoped to this destination
//   SavedStateRegistryOwner вҶ’ SavedStateHandle survives process death

// Inside composable {} block, the entry is provided via CompositionLocals:
//   LocalViewModelStoreOwner.current     == NavBackStackEntry
//   LocalLifecycleOwner.current          == NavBackStackEntry
//   LocalSavedStateRegistryOwner.current == NavBackStackEntry

// This means:
//   viewModel()      вҶ’ scoped to NavBackStackEntry (not Activity)
//   hiltViewModel()  вҶ’ scoped to NavBackStackEntry (not Activity)
//   rememberSaveable вҶ’ saved in NavBackStackEntry's SavedStateRegistry
//   collectAsStateWithLifecycle() вҶ’ respects entry's lifecycle

2. Defining Routes and Destinations

String-Based Routes (Legacy)

// The original routing model вҖ” still works but lacks compile-time safety
val navController = rememberNavController()

NavHost(navController = navController, startDestination = "home") {
    composable("home") { HomeScreen(navController) }

    composable(
        route = "profile/{userId}",
        arguments = listOf(
            navArgument("userId") { type = NavType.StringType }
        )
    ) { backStackEntry ->
        val userId = backStackEntry.arguments?.getString("userId")
        ProfileScreen(userId = userId!!)
    }

    // Optional query parameters
    composable(
        route = "search?query={query}&sort={sort}",
        arguments = listOf(
            navArgument("query") { defaultValue = "" },
            navArgument("sort") {
                type = NavType.StringType
                nullable = true
                defaultValue = null
            }
        )
    ) { SearchScreen(it) }
}

// Navigating:
navController.navigate("profile/user_123")
navController.navigate("search?query=kotlin&sort=date")

Type-Safe Navigation (Navigation 2.8+)

// Type-safe routes use @Serializable data classes/objects as route definitions.
// The compiler plugin generates serialization logic that maps fields to
// route path segments and query parameters.

@Serializable
object HomeRoute                          // no arguments вҶ’ "HomeRoute"

@Serializable
data class ProfileRoute(val userId: String) // вҶ’ "ProfileRoute/{userId}"

@Serializable
data class SearchRoute(                   // optional args become query params
    val query: String = "",
    val sort: String? = null
)                                         // вҶ’ "SearchRoute?query={query}&sort={sort}"

// Building the graph:
NavHost(navController = navController, startDestination = HomeRoute) {
    composable<HomeRoute> { HomeScreen(navController) }

    composable<ProfileRoute> { backStackEntry ->
        val route: ProfileRoute = backStackEntry.toRoute<ProfileRoute>()
        ProfileScreen(userId = route.userId)
    }

    composable<SearchRoute> { backStackEntry ->
        val route: SearchRoute = backStackEntry.toRoute<SearchRoute>()
        SearchScreen(query = route.query, sort = route.sort)
    }
}

// Navigating вҖ” fully type-safe, no string construction:
navController.navigate(ProfileRoute(userId = "user_123"))
navController.navigate(SearchRoute(query = "kotlin", sort = "date"))

// Under the hood:
// 1. @Serializable generates a KSerializer for the route class
// 2. Navigation uses RouteSerializer to convert the object to a route string:
//    ProfileRoute(userId = "user_123") вҶ’ "ProfileRoute/user_123"
// 3. On the receiving end, the route string is deserialized back:
//    "ProfileRoute/user_123" вҶ’ ProfileRoute(userId = "user_123")
// 4. Required params (no default) вҶ’ path segments
//    Optional params (has default) вҶ’ query parameters

Custom NavType

// For complex types not natively supported by Navigation:
@Serializable
data class Filter(val category: String, val minPrice: Int)

val FilterNavType = object : NavType<Filter>(isNullableAllowed = false) {
    override fun get(bundle: Bundle, key: String): Filter? =
        bundle.getString(key)?.let { Json.decodeFromString(it) }

    override fun parseValue(value: String): Filter =
        Json.decodeFromString(Uri.decode(value))

    override fun serializeAsValue(value: Filter): String =
        Uri.encode(Json.encodeToString(value))

    override fun put(bundle: Bundle, key: String, value: Filter) {
        bundle.putString(key, Json.encodeToString(value))
    }
}

// With type-safe routes, custom types are handled via typeMap:
composable<SearchRoute>(
    typeMap = mapOf(typeOf<Filter>() to FilterNavType)
) { ... }

Dialog Destinations and Nested Graphs

// Dialog destinations вҖ” rendered as overlays, previous destination stays RESUMED
NavHost(navController, startDestination = HomeRoute) {

3. NavBackStackEntry Lifecycle in Compose

Understanding the lifecycle states of NavBackStackEntry during navigation is critical. In Compose, the entry lifecycle controls when composables are in composition, when collectAsStateWithLifecycle() collects, and when ViewModels are cleared.

Lifecycle State Transitions

// Scenario: Navigate Home вҶ’ Profile вҶ’ Settings

// After navigating to Home:
//   Home entry:     RESUMED  (visible, in composition)

// After navigating to Profile:
//   Home entry:     STARTED  (on back stack, NOT in composition unless animating)
//   Profile entry:  RESUMED  (visible, in composition)

// After navigating to Settings:
//   Home entry:     CREATED  (deep in back stack)
//   Profile entry:  STARTED  (one level back)
//   Settings entry: RESUMED  (visible)

// Pop Settings:
//   Settings entry: DESTROYED (removed, ViewModel cleared)
//   Profile entry:  RESUMED   (back to top)
//   Home entry:     STARTED   (moves up one level)

// Key rule: only the TOP entry is RESUMED.
// Entries one level below are STARTED (visible during animation).
// Deeper entries are CREATED.

// With AnimatedNavHost (default since Nav 2.7):
//   During enter/exit animation, BOTH the outgoing and incoming
//   entries are STARTED. The incoming entry moves to RESUMED only
//   after the animation completes.

Lifecycle and collectAsStateWithLifecycle()

// collectAsStateWithLifecycle() respects the NavBackStackEntry lifecycle.
// Default minActiveState = Lifecycle.State.STARTED

@Composable
fun ProfileScreen(viewModel: ProfileViewModel = hiltViewModel()) {
    val uiState by viewModel.uiState.collectAsStateWithLifecycle()
    // Collection is active when entry is STARTED or RESUMED.
    // When entry drops to CREATED (deep in back stack), collection stops.
    // This prevents unnecessary recompositions for invisible destinations.
}

// Why this matters at scale:
// In a bottom-nav app with 5 tabs, each tab's composable might observe
// multiple Flows. Without lifecycle-aware collection, ALL tabs would
// keep collecting even when invisible вҖ” wasting CPU and memory.
// collectAsStateWithLifecycle() automatically pauses collection
// when the entry drops below STARTED.

CompositionLocal Providers per Entry

// NavHost wraps each destination's content in a CompositionLocalProvider:
//
// Simplified NavHost internals:
// visibleEntries.forEach { entry ->
//     CompositionLocalProvider(
//         LocalViewModelStoreOwner provides entry,
//         LocalLifecycleOwner provides entry,
//         LocalSavedStateRegistryOwner provides entry
//     ) {
//         entry.destination.content(entry)  // your composable
//     }
// }
//
// This is why viewModel() inside a composable {} block returns a
// ViewModel scoped to the NavBackStackEntry вҖ” not the Activity.
// The LocalViewModelStoreOwner is the entry, and ViewModelProvider
// uses that owner's ViewModelStore.

4. State Management in Compose Navigation

rememberSaveable вҖ” Survives Config Change and Process Death

@Composable
fun ProfileScreen() {
    // Saved in NavBackStackEntry's SavedStateRegistry.
    // Survives: config change, process death (while on back stack).
    // Lost: when NavBackStackEntry is popped (destination removed).
    var searchQuery by rememberSaveable { mutableStateOf("") }

    // Custom Saver for complex types:
    var filter by rememberSaveable(stateSaver = FilterSaver) {
        mutableStateOf(Filter(category = "all", minPrice = 0))
    }
}

object FilterSaver : Saver<Filter, String> {
    override fun save(value: Filter): String = Json.encodeToString(value)
    override fun restore(value: String): Filter = Json.decodeFromString(value)
}

// How it works under the hood:
// 1. rememberSaveable registers with the SavedStateRegistry of the
//    current LocalSavedStateRegistryOwner (= NavBackStackEntry)
// 2. On config change: NavBackStackEntry's SavedStateRegistry saves
//    all registered values into a Bundle
// 3. NavControllerViewModel preserves the entry's ViewModelStore
// 4. On restore: the new composition reads from the restored Bundle
// 5. On process death: the Bundle is serialized to disk via
//    Activity.onSaveInstanceState() вҶ’ NavController.saveState()

ViewModel Scoping Patterns

// Pattern 1: ViewModel per destination (default)
@Composable
fun ProfileScreen(viewModel: ProfileViewModel = hiltViewModel()) {
    // Scoped to this NavBackStackEntry.
    // Created when first entering this destination.
    // Cleared when this entry is popped from the back stack.
}

// Pattern 2: ViewModel scoped to a nested nav graph (shared state)
@Composable
fun CheckoutStepOne(navController: NavController) {
    val parentEntry = remember(navController) {
        navController.getBackStackEntry<CheckoutGraphRoute>()
    }
    val sharedVm: CheckoutViewModel = hiltViewModel(parentEntry)
    // Same instance across all destinations in the checkout graph.
    // Cleared when the entire checkout graph is popped.
}

@Composable
fun CheckoutStepTwo(navController: NavController) {
    val parentEntry = remember(navController) {
        navController.getBackStackEntry<CheckoutGraphRoute>()
    }
    val sharedVm: CheckoutViewModel = hiltViewModel(parentEntry)
    // Exact same CheckoutViewModel instance as StepOne.
}

// Pattern 3: ViewModel scoped to Activity (escape hatch)
@Composable
fun AnyScreen() {
    val activityVm: AppViewModel = viewModel(
        viewModelStoreOwner = LocalContext.current as ComponentActivity
    )
    // Lives as long as the Activity вҖ” survives all navigation.
    // Use sparingly: this is essentially a global singleton per Activity.
}

SavedStateHandle and Route Arguments

// Route arguments are automatically available in SavedStateHandle.
// This is the bridge between navigation arguments and ViewModel state.

@Serializable
data class ProfileRoute(val userId: String)

@HiltViewModel
class ProfileViewModel @Inject constructor(
    savedStateHandle: SavedStateHandle
) : ViewModel() {
    // Type-safe access with Navigation 2.8+:
    val route: ProfileRoute = savedStateHandle.toRoute<ProfileRoute>()

    // Or traditional string-key access (works with both old and new routes):
    val userId: String = savedStateHandle["userId"]!!

    // SavedStateHandle also persists across process death.
    // You can write to it and the values survive:
    val scrollPosition = savedStateHandle.getStateFlow("scroll_pos", 0)

    fun saveScroll(pos: Int) {
        savedStateHandle["scroll_pos"] = pos
    }
}
Staff-Level Insight вҖ” NavBackStackEntry's ViewModelStore Indirection

In Compose Navigation, when a destination leaves composition (user navigates forward), its composable is removed from the composition tree вҖ” all remember state is lost. But the NavBackStackEntry stays alive on the back stack with its ViewModelStore intact. The ViewModelStore is not held by the entry itself вҖ” it's stored in NavControllerViewModel, a ViewModel scoped to the Activity. This double indirection (Activity вҶ’ NavControllerViewModel вҶ’ per-entry ViewModelStore) is why ViewModels survive config changes: the Activity's ViewModelStore survives, which preserves NavControllerViewModel, which preserves all entry ViewModelStores. When an entry is popped, NavControllerViewModel removes that entry's ViewModelStore and calls clear() on it. A subtle consequence: if you hold a reference to a popped entry's ViewModel elsewhere, the ViewModel instance exists but its onCleared() has already been called вҖ” any coroutine scope from viewModelScope is cancelled.

// When a deep link fires, NavController builds a synthetic back stack

// Graph structure:

// Deep link: myapp://notifications
// Synthetic back stack built:

Staff-Level Insight вҖ” Scaling to 100+ Destinations

At Google-scale apps, the single NavHost with all destinations registered upfront becomes a bottleneck вҖ” not for runtime performance, but for build times and code ownership. The recommended architecture: each feature module exposes a NavGraphBuilder extension function. Route classes live in a shared :core:navigation module that is lightweight (no UI dependencies). The :app module's only job is to compose these graphs. For teams with strict module boundaries, consider a navigator abstraction layer: feature modules declare navigation intents (e.g., "go to profile") via an interface, and the app module provides the implementation that calls navController.navigate(). This fully decouples features from the navigation framework вҖ” you could swap Compose Navigation for a custom solution without touching feature code. The trade-off is indirection and boilerplate; for most apps under 50 destinations, direct NavController usage with shared routes is simpler and sufficient.

9. Staff-Level Corner Cases

Double-Navigation Guard in Compose

// The Compose equivalent of the double-tap crash from Fragment navigation.
// In Compose, rapid recomposition can trigger navigate() multiple times.

// Problem:
@Composable
fun HomeScreen(navController: NavController) {
    Button(onClick = {
        // If the user taps rapidly, this fires multiple times
        // before the first navigation completes
        navController.navigate(ProfileRoute(userId = "123"))
    }) { Text("Go to Profile") }
}

// Fix: guard with currentBackStackEntryAsState()
@Composable
fun HomeScreen(navController: NavController) {
    val currentEntry by navController.currentBackStackEntryAsState()

    Button(onClick = {
        // Only navigate if we're still on the Home destination
        if (currentEntry?.destination?.hasRoute<HomeRoute>() == true) {
            navController.navigate(ProfileRoute(userId = "123"))
        }
    }) { Text("Go to Profile") }
}

NavBackStackEntry Lifecycle During Shared Element Transitions

// During a shared element transition between Screen A and Screen B:
//
//   Screen A entry: RESUMED вҶ’ STARTED (NOT DESTROYED)
//   Screen B entry: CREATED вҶ’ STARTED
//
// Both entries are STARTED for the duration of the animation.
// Screen A's composable is still in composition (needed for the
// shared element to animate from its position).
//
// Only after the animation completes:
//   Screen A entry: STARTED вҶ’ CREATED (or STARTED if one level back)
//   Screen B entry: STARTED вҶ’ RESUMED
//
// This means:
// - ViewModels for Screen A are NOT cleared during the transition
// - collectAsStateWithLifecycle() keeps collecting on both screens
//   (STARTED meets the default minActiveState)
// - Side effects in Screen A (LaunchedEffect) continue running
//   until the entry drops below STARTED

Graph-Scoped ViewModel Outliving Destinations

Back Handling in Compose

// BackHandler {} composable вҖ” the Compose equivalent of OnBackPressedCallback
@Composable
fun EditScreen(navController: NavController, hasUnsavedChanges: Boolean) {
    // Only intercept back when there are unsaved changes
    BackHandler(enabled = hasUnsavedChanges) {
        // Show confirmation dialog instead of navigating back
        showDiscardDialog = true
    }

    if (showDiscardDialog) {
        AlertDialog(
            onDismissRequest = { showDiscardDialog = false },
            confirmButton = {
                TextButton(onClick = { navController.popBackStack() }) {
                    Text("Discard")
                }
            },
            text = { Text("Discard unsaved changes?") }
        )
    }
}

// BackHandler uses OnBackPressedDispatcher under the hood.
// Multiple BackHandlers can be active вҖ” the most recently composed
// enabled one takes priority (LIFO order).
// When enabled = false, the handler is removed from the dispatcher.

Process Death вҖ” What Survives and What Doesn't

// After process death and restoration:
//
// SURVIVES:
//   вң“ NavController back stack (destination IDs + arguments)
//   вң“ rememberSaveable values (in SavedStateRegistry)
//   вң“ SavedStateHandle contents in ViewModels
//   вң“ Route arguments (serialized in the back stack state)
//
// DOES NOT SURVIVE:
//   вң— ViewModel instances (recreated fresh)
//   вң— remember {} values (not saved)
//   вң— In-memory caches, repositories, singletons
//   вң— Coroutine jobs in viewModelScope (new scope created)
//   вң— Composition state not in rememberSaveable
//
// The restoration flow:
//   1. Activity.onCreate(savedInstanceState) вҖ” bundle is non-null
//   2. NavController.restoreState() rebuilds back stack entries
//   3. NavHost recomposes with restored entries
//   4. Each destination's composable runs fresh
//   5. rememberSaveable restores from SavedStateRegistry
//   6. viewModel() / hiltViewModel() creates NEW ViewModel instances
//   7. SavedStateHandle in new ViewModels has the restored values
//   8. ViewModel.init {} can read SavedStateHandle to restore state

Danger вҖ” rememberSaveable with Mutable Collections in Navigation

Using rememberSaveable with mutable collections (e.g., mutableListOf()) inside a navigation destination is a trap. The default Saver for List uses ArrayList serialization, which works вҖ” but mutations to the list do not trigger recomposition because rememberSaveable stores the value, not a MutableState. Worse, if you wrap it in mutableStateOf(mutableListOf()), mutations to the inner list still don't trigger recomposition because the State object reference hasn't changed. The correct pattern: use rememberSaveable { mutableStateOf(listOf(...)) } with immutable list operations (list + newItem, list.filter {}) that produce new list instances, or use a SnapshotStateList with a custom Saver.

10. ComposeNavigator State Flow

// How ComposeNavigator actually manages state:
//
// NavController.navigate(route)
//   вҶ’ Navigator.navigate(destination, args, navOptions)
//     вҶ’ ComposeNavigator creates a NavBackStackEntry
//     вҶ’ Entry is added to ComposeNavigator's internal backStack StateFlow
//     вҶ’ NavHost observes this StateFlow
//     вҶ’ Recomposition triggers AnimatedContent with new entry
//     вҶ’ CompositionLocalProvider sets up:
//        - LocalViewModelStoreOwner вҶ’ NavBackStackEntry
//        - LocalLifecycleOwner вҶ’ NavBackStackEntry
//        - LocalSavedStateRegistryOwner вҶ’ NavBackStackEntry
//     вҶ’ The composable lambda for the destination is invoked
//
// On popBackStack():
//   вҶ’ ComposeNavigator removes the entry from backStack
//   вҶ’ AnimatedContent animates the exit transition
//   вҶ’ After animation completes, the entry's Lifecycle moves to DESTROYED
//   вҶ’ ViewModelStore is cleared вҶ’ ViewModel.onCleared()
//   вҶ’ SavedStateRegistry is cleared
//
// Key difference from FragmentNavigator:
// - No Fragment transactions, no FragmentManager involvement
// - State is driven by Compose's snapshot system
// - Transitions are Compose animations, not Fragment transitions
// - The NavBackStackEntry IS the composition scope

13. Conditional Navigation (Auth Gate Pattern)

// Common pattern: redirect unauthenticated users to login
// while preserving their intended destination

@Composable
fun AppNavHost(isLoggedIn: Boolean) {
    val navController = rememberNavController()

    NavHost(
        navController = navController,
        startDestination = if (isLoggedIn) HomeRoute else LoginRoute
    ) {
        composable<LoginRoute> {
            LoginScreen(
                onLoginSuccess = {
                    navController.navigate(HomeRoute) {
                        popUpTo<LoginRoute> { inclusive = true }
                    }
                }
            )
        }
        composable<HomeRoute> { HomeScreen() }
        composable<ProfileRoute> { backStackEntry ->
            // Guard: if not logged in, redirect to login
            // and save intended destination
            if (!isLoggedIn) {
                LaunchedEffect(Unit) {
                    navController.navigate(LoginRoute) {
                        // Don't add to back stack вҖ” after login,
                        // navigate directly to intended destination
                    }
                }
                return@composable
            }
            ProfileScreen(backStackEntry.toRoute<ProfileRoute>().userId)
        }
    }
}
