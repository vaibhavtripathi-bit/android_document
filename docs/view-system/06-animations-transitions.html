Animations & Transitions

Property Animation, Transition Framework, MotionLayout, Physics-Based Motion — from the broken legacy API to production-grade animation systems

1. Animation Framework Evolution

Android's animation story is one of successive replacements, each fixing fundamental flaws in the previous generation. Understanding why each API was introduced — not just how to use it — is critical for L6 interviews.

2. View Animation (Legacy) — Why It's Broken

The original Animation class (API 1) is fundamentally flawed in a way that every Android developer should understand — it only changes the visual rendering of a View, not its actual properties.

The Core Problem

// Move a button 300px to the right using View Animation
val anim = TranslateAnimation(0f, 300f, 0f, 0f)
anim.duration = 500
anim.fillAfter = true  // keep the visual at the end position
button.startAnimation(anim)

// After animation completes:
// - The button APPEARS at x+300
// - But button.x is still at the ORIGINAL position
// - Click events are dispatched to the ORIGINAL position
// - The user taps the visible button → nothing happens
// - The user taps empty space at the original position → click fires!
View Animation — The Visual vs Reality Gap

Before animation:
  ┌──────────┐
  │  Button  │  ← visual position = actual position = click target
  └──────────┘

After TranslateAnimation with fillAfter=true:
  ┌ ─ ─ ─ ─ ┐                        ┌──────────┐
  │  (ghost) │  ← click target HERE   │  Button  │  ← visual position HERE
  └ ─ ─ ─ ─ ┘    (actual bounds)      └──────────┘    (but can't click!)

Property Animation fixes this:
                                      ┌──────────┐
                                      │  Button  │  ← visual = actual = click target
                                      └──────────┘

Why It Still Exists

  • overridePendingTransition(enterAnim, exitAnim) — Activity enter/exit animations still use the legacy Animation resource format (R.anim.*)
  • WindowAnimation styles — system-level window animations use the same format
  • layoutAnimation — staggered entrance animations for ViewGroup children use LayoutAnimationController, which wraps legacy Animation
  • Some third-party libraries still use it for compatibility
Never Use View Animation for Interactive Elements
If the animated View is clickable, draggable, or otherwise interactive, never use Animation. The touch target stays at the original position. Use ObjectAnimator or ViewPropertyAnimator instead — they change the actual translationX, translationY, scaleX, etc. properties, so the View's hit rect moves with the visual.

3. Property Animation — ValueAnimator

ValueAnimator is the foundation of the entire property animation system. It doesn't animate anything by itself — it produces interpolated values over time, and you decide what to do with them.

Basic Usage

// Animate a float from 0 to 1 over 300ms
val animator = ValueAnimator.ofFloat(0f, 1f).apply {
    duration = 300
    addUpdateListener { animation ->
        val fraction = animation.animatedValue as Float
        // Do anything with this value — update a View, a Paint, a path, etc.
        customView.progress = fraction
        customView.invalidate()
    }
}
animator.start()

Value Types and TypeEvaluators

// Built-in value types
ValueAnimator.ofFloat(0f, 1f)           // Float interpolation
ValueAnimator.ofInt(0, 255)             // Int interpolation
ValueAnimator.ofArgb(Color.RED, Color.BLUE) // ARGB color interpolation (API 21)

// Custom TypeEvaluator for arbitrary types
val pointEvaluator = TypeEvaluator<PointF> { fraction, startValue, endValue ->
    PointF(
        startValue.x + fraction * (endValue.x - startValue.x),
        startValue.y + fraction * (endValue.y - startValue.y)
    )
}

val animator = ValueAnimator.ofObject(
    pointEvaluator,
    PointF(0f, 0f),
    PointF(300f, 500f)
).apply {
    duration = 400
    addUpdateListener {
        val point = it.animatedValue as PointF
        view.translationX = point.x
        view.translationY = point.y
    }
}

Lifecycle Callbacks

animator.addListener(object : AnimatorListenerAdapter() {
    override fun onAnimationStart(animation: Animator) {
        // Animation started (after startDelay)
    }
    override fun onAnimationEnd(animation: Animator) {
        // Animation completed or was cancelled then ended
    }
    override fun onAnimationCancel(animation: Animator) {
        // cancel() was called — onAnimationEnd() still fires after this
    }
    override fun onAnimationRepeat(animation: Animator) {
        // One repeat cycle completed
    }
})

// Repeat configuration
animator.repeatCount = ValueAnimator.INFINITE  // or a specific count
animator.repeatMode = ValueAnimator.REVERSE    // RESTART or REVERSE
Key Detail
onAnimationEnd() is called even after cancel(). If you call cancel(), the sequence is: onAnimationCancel() → onAnimationEnd(). If you only want to run cleanup on natural completion, check the isCancel parameter in the onAnimationEnd(animation, isReverse) overload (API 26+), or track cancellation state manually.

4. ObjectAnimator — Property-Based Animation

ObjectAnimator extends ValueAnimator and automatically applies the animated value to a named property on a target object via reflection (or Property objects).

String-Based Properties (Reflection)

// Animates view.setTranslationX() from 0 to 100
val anim = ObjectAnimator.ofFloat(view, "translationX", 0f, 100f).apply {
    duration = 300
}
anim.start()

// Under the hood, ObjectAnimator:
// 1. Looks for setTranslationX(float) via reflection
// 2. If start value omitted, looks for getTranslationX() to read current value
// 3. On each frame, calls setTranslationX(interpolatedValue)
R8/ProGuard Trap
String-based property names use reflection. R8 can strip or rename the setter/getter methods, breaking the animation silently (the View just doesn't move — no crash). Always annotate custom property setters with @Keep, or better yet, use Property objects.

Property Objects (No Reflection)

// Type-safe, no reflection, R8-safe
val anim = ObjectAnimator.ofFloat(view, View.TRANSLATION_X, 0f, 100f)

// Built-in Property constants on View:
// View.TRANSLATION_X, View.TRANSLATION_Y, View.TRANSLATION_Z
// View.SCALE_X, View.SCALE_Y
// View.ROTATION, View.ROTATION_X, View.ROTATION_Y
// View.ALPHA
// View.X, View.Y, View.Z  (translationX + left, etc.)

// Custom Property for your own classes:
val progressProperty = object : Property<CircularProgressView, Float>(
    Float::class.java, "progress"
) {
    override fun get(view: CircularProgressView): Float = view.progress
    override fun set(view: CircularProgressView, value: Float) {
        view.progress = value
        view.invalidate()
    }
}

ObjectAnimator.ofFloat(progressView, progressProperty, 0f, 1f).start()
Staff Insight — Property vs String Performance
Property objects bypass reflection entirely. With string-based names, ObjectAnimator calls Class.getMethod() during setup and uses Method.invoke() on every frame — both are expensive. Property objects use direct virtual dispatch. In a benchmark with 20 simultaneous animators, Property objects reduce per-frame overhead by ~40%. Always use View.TRANSLATION_X etc. for View properties, and define custom Property objects for your own animatable classes.

PropertyValuesHolder — Multiple Properties, Single Animator

// Animate translationX and alpha simultaneously with ONE animator
val pvhX = PropertyValuesHolder.ofFloat(View.TRANSLATION_X, 0f, 200f)
val pvhAlpha = PropertyValuesHolder.ofFloat(View.ALPHA, 1f, 0f)

val anim = ObjectAnimator.ofPropertyValuesHolder(view, pvhX, pvhAlpha).apply {
    duration = 400
    interpolator = FastOutSlowInInterpolator()
}
anim.start()

// Why use PropertyValuesHolder instead of two ObjectAnimators?
// - Single animator = single listener, single timing source
// - Slightly less overhead than two separate animators
// - Guarantees both properties update on the exact same frame

5. AnimatorSet — Composing Animations

AnimatorSet orchestrates multiple animators with dependency relationships — parallel, sequential, or arbitrary dependency graphs.

// Parallel execution
AnimatorSet().apply {
    playTogether(fadeIn, slideUp, scaleUp)
    duration = 300  // overrides individual durations
    start()
}

// Sequential execution
AnimatorSet().apply {
    playSequentially(fadeIn, slideUp, scaleUp)
    start()
}

// Dependency graph — the most powerful form
AnimatorSet().apply {
    play(slideUp).with(fadeIn)          // slideUp and fadeIn run together
    play(scaleUp).after(slideUp)        // scaleUp starts after slideUp ends
    play(colorChange).after(200)        // colorChange starts after 200ms delay
    play(bounceBack).before(fadeOut)    // bounceBack must finish before fadeOut starts
    start()
}

Nested AnimatorSets

// AnimatorSets can contain other AnimatorSets
val enterSet = AnimatorSet().apply {
    playTogether(fadeIn, slideUp)
    duration = 250
}

val emphasizeSet = AnimatorSet().apply {
    playSequentially(scaleUp, scaleDown)
    duration = 150
}

val masterSet = AnimatorSet().apply {
    playSequentially(enterSet, emphasizeSet)
    interpolator = FastOutSlowInInterpolator()
    start()
}
Key Detail
Setting interpolator on an AnimatorSet applies it to all child animators, overriding their individual interpolators. If you need different interpolators per child, set them on the individual animators and leave the AnimatorSet's interpolator unset.

6. ViewPropertyAnimator — The Fluent API

ViewPropertyAnimator is the most efficient way to animate common View properties. It batches multiple property changes into a single native call on the RenderThread.

// Fluent API — chains property changes
view.animate()
    .translationX(100f)
    .translationY(50f)
    .alpha(0.5f)
    .scaleX(1.2f)
    .rotation(45f)
    .setDuration(300)
    .setInterpolator(FastOutSlowInInterpolator())
    .setStartDelay(100)
    .withStartAction { /* runs on animation start */ }
    .withEndAction { /* runs on animation end */ }
    .start()  // optional — starts automatically on next frame if omitted

How It Works Internally

// ViewPropertyAnimator batches all property changes and applies them
// as a single RenderNode update. Internally:
//
// 1. Each .translationX(), .alpha(), etc. sets a bitmask flag
// 2. On the next animation frame, ALL flagged properties are read,
//    interpolated, and written in a single pass
// 3. The update goes directly to the RenderNode — no invalidate(),
//    no display list re-recording, no measure/layout
//
// This is fundamentally different from ObjectAnimator, which:
// - Creates a separate animator per property
// - Calls setTranslationX() individually (which calls invalidate())
// - Each invalidate() marks the RenderNode dirty

// Cancel all pending ViewPropertyAnimator animations on a View
view.animate().cancel()
Staff Insight — ViewPropertyAnimator vs ObjectAnimator
ViewPropertyAnimator is the most efficient animation API for View properties because it operates directly on the RenderNode without going through the normal invalidation path. When you call view.animate().translationX(100f).alpha(0.5f), both properties are batched into a single RenderNode property update that the RenderThread applies without re-recording the display list. In contrast, two ObjectAnimators for the same properties would each call setTranslationX() and setAlpha() separately, each triggering invalidate() and potentially re-recording the display list. For simple View property animations, always prefer ViewPropertyAnimator.

7. Interpolators

Interpolators map the linear progression of time (0.0 → 1.0) to a non-linear animation fraction. They are the single most important factor in making animations feel natural.

Built-In Interpolators

InterpolatorCurveUse Case
LinearInterpolatorConstant speedProgress bars, continuous rotation
AccelerateInterpolatorSlow → fastElements leaving the screen
DecelerateInterpolatorFast → slowElements entering the screen
AccelerateDecelerateInterpolatorSlow → fast → slowDefault; generic movement
OvershootInterpolatorOvershoots end, bounces backPlayful UI, FAB reveal
AnticipateInterpolatorPulls back, then forwardThrowing/launching motion
AnticipateOvershootInterpolatorPull back → overshoot → settleDramatic emphasis
BounceInterpolatorBounces at the endDrop/landing effects
CycleInterpolatorSine wave (n cycles)Shake/vibrate effects
FastOutSlowInInterpolatorcubic-bezier(0.4, 0, 0.2, 1)Material Design standard — use this by default
FastOutLinearInInterpolatorcubic-bezier(0.4, 0, 1, 1)Elements leaving the screen (Material)
LinearOutSlowInInterpolatorcubic-bezier(0, 0, 0.2, 1)Elements entering the screen (Material)
PathInterpolatorCustom cubic Bézier or PathAny custom curve

Custom Interpolators

// PathInterpolator — define any curve via a Path
val path = Path().apply {
    moveTo(0f, 0f)
    cubicTo(0.2f, 0f, 0.0f, 1f, 1f, 1f)  // custom cubic bezier
}
val customInterpolator = PathInterpolator(path)

// Or use the 4-parameter cubic-bezier shorthand
val materialStandard = PathInterpolator(0.4f, 0f, 0.2f, 1f)

// Implement TimeInterpolator directly for full control
val elasticInterpolator = TimeInterpolator { input ->
    // input: 0.0 → 1.0 (linear time)
    // return: animation fraction (can exceed 0..1 for overshoot)
    val p = 0.3f
    Math.pow(2.0, -10.0 * input).toFloat() *
        Math.sin((input - p / 4) * (2 * Math.PI) / p).toFloat() + 1f
}
Material Design Guidance
Material Design 3 uses emphasized easing with two-segment curves: a faster initial segment and a slower final segment. The standard curve is cubic-bezier(0.2, 0, 0, 1) for M3 (different from M2's 0.4, 0, 0.2, 1). Always check the current Material spec — the curves evolve between versions.

8. Transition Framework

The Transition Framework (API 19) automates the animation of layout changes. Instead of manually creating animators for each View, you describe what should animate and the framework figures out how.

beginDelayedTransition — The Simple Path

// Animate the next layout change automatically
TransitionManager.beginDelayedTransition(
    parentViewGroup,
    AutoTransition()  // Fade out removed → move/resize remaining → fade in added
)

// Now make layout changes — the framework captures before/after states
textView.visibility = View.GONE
button.layoutParams.width = 400
button.requestLayout()

// The framework:
// 1. Captures the "before" state of all children (bounds, visibility, alpha)
// 2. Lets the layout change happen
// 3. Captures the "after" state
// 4. Creates animators for every View whose state changed
// 5. Runs the animations

Built-In Transitions

TransitionWhat It Animates
FadeAlpha (fade in/out for appearing/disappearing Views)
ChangeBoundsPosition and size (left, top, right, bottom)
SlideSlides Views in/out from an edge (Gravity.START, .TOP, etc.)
ExplodeViews fly outward from the center (or inward)
ChangeTransformScale and rotation
ChangeClipBoundsClip bounds (View.setClipBounds())
ChangeImageTransformImageView's matrix (scale type changes)
AutoTransitionFade out → ChangeBounds → Fade in (sequential)

TransitionSet — Composing Transitions

val transition = TransitionSet().apply {
    ordering = TransitionSet.ORDERING_TOGETHER  // or ORDERING_SEQUENTIAL
    addTransition(Fade(Fade.OUT).apply {
        duration = 150
    })
    addTransition(ChangeBounds().apply {
        duration = 300
        interpolator = FastOutSlowInInterpolator()
    })
    addTransition(Fade(Fade.IN).apply {
        duration = 150
        startDelay = 150  // start after fade-out
    })
}

TransitionManager.beginDelayedTransition(container, transition)

Scene Transitions

// Scenes represent complete layout states
val scene1 = Scene.getSceneForLayout(container, R.layout.scene_1, context)
val scene2 = Scene.getSceneForLayout(container, R.layout.scene_2, context)

// Animate between scenes
TransitionManager.go(scene2, ChangeBounds())

// Scene enter/exit actions
scene2.setEnterAction { /* setup after transition */ }
scene2.setExitAction { /* cleanup before leaving */ }

Targeting and Excluding

val transition = ChangeBounds().apply {
    // Only animate specific Views
    addTarget(R.id.card_view)
    addTarget(R.id.title_text)

    // Or exclude Views from animation
    excludeTarget(R.id.background, true)
    excludeTarget(RecyclerView::class.java, true)  // exclude by type

    // Exclude children of a ViewGroup
    excludeChildren(R.id.toolbar, true)
}

Shared Element Transitions

// Activity → Activity shared element transition
val options = ActivityOptions.makeSceneTransitionAnimation(
    this,
    Pair(imageView, "hero_image"),     // View → transitionName
    Pair(titleView, "hero_title")
)
startActivity(intent, options.toBundle())

// In the target Activity (before setContentView or in theme):
window.sharedElementEnterTransition = ChangeTransform()
window.sharedElementReturnTransition = ChangeTransform()

// Fragment shared element transitions
parentFragmentManager.commit {
    setReorderingAllowed(true)  // REQUIRED for shared elements
    addSharedElement(imageView, "hero_image")
    replace(R.id.container, DetailFragment())
    addToBackStack(null)
}

// In the destination Fragment:
sharedElementEnterTransition = TransitionInflater.from(requireContext())
    .inflateTransition(R.transition.shared_image)
Staff Insight — Shared Element Fragility
Shared element transitions are notoriously fragile in production. They depend on the target View being laid out and measured before the transition starts. Common breakage scenarios:
  • RecyclerView: The target item may not be laid out yet (scrolled off-screen). Use postponeEnterTransition() + startPostponedEnterTransition() after the adapter binds the target item.
  • Async image loading: Glide/Coil loads the image after layout. The transition animates a placeholder → loaded image jump. Use postponeEnterTransition() and start it in the image load callback.
  • ViewPager2: The shared element may be inside a Fragment managed by the ViewPager adapter. You need to set the transition name dynamically in onBindViewHolder() and coordinate with setExitSharedElementCallback().
  • Timing: If the destination layout takes too long (complex ConstraintLayout, nested RecyclerView), the transition may start before layout completes, causing the shared element to animate to the wrong position.

})

10. Physics-Based Animation

Physics-based animations (API 26, backported via AndroidX) use real physics equations instead of fixed durations and interpolators. They produce natural motion that respects the user's input velocity.

SpringAnimation

// Spring animation — bounces to the target value
val springAnim = SpringAnimation(view, DynamicAnimation.TRANSLATION_Y, 0f).apply {
    spring = SpringForce(0f).apply {
        stiffness = SpringForce.STIFFNESS_MEDIUM       // how tight the spring is
        dampingRatio = SpringForce.DAMPING_RATIO_LOW_BOUNCY  // how bouncy
    }
}
springAnim.start()

// Stiffness constants:
// STIFFNESS_HIGH (10000f)   — very tight, fast settle
// STIFFNESS_MEDIUM (1500f)  — default
// STIFFNESS_LOW (200f)      — loose, slow settle
// STIFFNESS_VERY_LOW (50f)  — very loose, long settle

// Damping ratio constants:
// DAMPING_RATIO_HIGH_BOUNCY (0.2f)     — lots of bouncing
// DAMPING_RATIO_MEDIUM_BOUNCY (0.5f)   — moderate bounce
// DAMPING_RATIO_LOW_BOUNCY (0.75f)     — slight bounce
// DAMPING_RATIO_NO_BOUNCY (1f)         — critically damped, no overshoot

FlingAnimation

// Fling animation — decelerates from an initial velocity
val flingAnim = FlingAnimation(view, DynamicAnimation.TRANSLATION_X).apply {
    setStartVelocity(velocityTracker.xVelocity)  // from user gesture
    friction = 1.5f  // how quickly it decelerates (default: 1.0)
    setMinValue(0f)  // clamp to bounds
    setMaxValue(maxScrollX)
}
flingAnim.start()

// Chaining: fling → spring (natural gesture dismissal)
flingAnim.addEndListener { _, canceled, value, velocity ->
    if (!canceled) {
        // Fling ended — spring back to nearest snap point
        SpringAnimation(view, DynamicAnimation.TRANSLATION_X, snapPoint).apply {
            setStartVelocity(velocity)  // preserve the fling's remaining velocity
            spring = SpringForce(snapPoint).apply {
                stiffness = SpringForce.STIFFNESS_MEDIUM
                dampingRatio = SpringForce.DAMPING_RATIO_NO_BOUNCY
            }
        }.start()
    }
}

Gesture-Driven Spring

// Complete drag → spring-back pattern
var springAnim: SpringAnimation? = null

view.setOnTouchListener { v, event ->
    when (event.action) {
        MotionEvent.ACTION_DOWN -> {
            springAnim?.cancel()  // cancel any running spring
            true
        }
        MotionEvent.ACTION_MOVE -> {
            v.translationY = event.rawY - startY
            true
        }
        MotionEvent.ACTION_UP -> {
            velocityTracker.computeCurrentVelocity(1000)
            springAnim = SpringAnimation(v, DynamicAnimation.TRANSLATION_Y, 0f).apply {
                setStartVelocity(velocityTracker.yVelocity)
                spring = SpringForce(0f).apply {
                    stiffness = SpringForce.STIFFNESS_LOW
                    dampingRatio = SpringForce.DAMPING_RATIO_MEDIUM_BOUNCY
                }
            }
            springAnim?.start()
            true
        }
        else -> false
    }
}
Staff Insight — Why Physics Beats Duration-Based
Duration-based animations have a fundamental problem with interruption. If a user flings a View at high velocity and the animation has a fixed 300ms duration, the motion feels sluggish. If they barely nudge it, the same 300ms feels too fast. Physics-based animations solve this because the velocity is an input, not a derived value. A fast fling produces a long, sweeping motion; a slow drag produces a short, gentle one. Additionally, interrupting a spring animation mid-flight is seamless — you just update the target or start velocity, and the physics naturally adjusts. With duration-based animations, interruption requires cancelling and restarting, often causing a visible jump.

11. Staff-Level Corner Cases

Animations on Detached Views

// Animators keep running even after the View is detached from the window.
// The animator holds a strong reference to the target View, preventing GC.

class LeakyView(context: Context) : View(context) {
    private val pulseAnimator = ObjectAnimator.ofFloat(this, View.ALPHA, 0.5f, 1f).apply {
        repeatCount = ValueAnimator.INFINITE
        repeatMode = ValueAnimator.REVERSE
        duration = 1000
    }

    override fun onAttachedToWindow() {
        super.onAttachedToWindow()
        pulseAnimator.start()
    }

    // MUST cancel in onDetachedFromWindow — otherwise the animator
    // keeps running, holding a reference to this View (and its Context)
    override fun onDetachedFromWindow() {
        pulseAnimator.cancel()
        super.onDetachedFromWindow()
    }
}

cancel() vs end()

// animator.cancel() — stops at the CURRENT value
val anim = ObjectAnimator.ofFloat(view, View.ALPHA, 0f, 1f)
anim.start()
// ... at alpha = 0.3 ...
anim.cancel()  // alpha stays at 0.3
// Fires: onAnimationCancel() → onAnimationEnd()

// animator.end() — jumps to the FINAL value
anim.start()
// ... at alpha = 0.3 ...
anim.end()  // alpha jumps to 1.0
// Fires: onAnimationEnd() (no onAnimationCancel)

// Common pattern: end all animations before starting new ones
// to avoid visual conflicts
ViewCompat.animate(view).cancel()  // cancel ViewPropertyAnimator
existingAnimator?.end()             // jump to final state
newAnimator.start()                 // start fresh

Hardware Layers During Animation

// Hardware layers cache a View's rendering as a GPU texture.
// During animation, this avoids re-recording the display list every frame.

// Manual approach:
view.setLayerType(View.LAYER_TYPE_HARDWARE, null)
val anim = ObjectAnimator.ofFloat(view, View.ALPHA, 0f, 1f).apply {
    addListener(object : AnimatorListenerAdapter() {
        override fun onAnimationEnd(animation: Animator) {
            // MUST reset — hardware layers consume GPU memory
            view.setLayerType(View.LAYER_TYPE_NONE, null)
        }
    })
}
anim.start()

// ViewPropertyAnimator does this automatically:
view.animate()
    .alpha(0f)
    .withLayer()  // enables hardware layer for the duration of the animation
    .start()
// withLayer() sets LAYER_TYPE_HARDWARE on start and LAYER_TYPE_NONE on end
Hardware Layer Trap
Hardware layers are only beneficial for animations that don't change the View's content — translation, rotation, scale, alpha. If the View's content changes during animation (e.g., a TextView whose text updates, or a ProgressBar), the layer must be re-created every frame, which is more expensive than not using a layer. Only use withLayer() for transform/alpha animations.

Staff Insight — Animation and the Choreographer
All property animations are driven by the Choreographer's CALLBACK_ANIMATION callback type. On each VSYNC, the AnimationHandler (a singleton per Looper) iterates all active ValueAnimator instances, computes their current fraction based on elapsed time, applies the interpolator, evaluates the TypeEvaluator, and calls the update listener. This means all animators on the same thread share the same frame clock — they are perfectly synchronized. It also means that if the main thread is blocked (heavy work in onDraw(), GC pause), all animations stutter together. This is why ViewPropertyAnimator is superior for View properties — it pushes the actual property updates to the RenderThread, which runs independently of the main thread's workload.

Animator Pause and Resume (API 19+)

// Pause/resume preserves the animator's current state
animator.pause()   // freezes at current value, stops consuming frames
animator.resume()  // continues from where it paused

// AnimatorListenerAdapter has pause/resume callbacks
animator.addPauseListener(object : AnimatorListenerAdapter() {
    override fun onAnimationPause(animation: Animator) { /* paused */ }
    override fun onAnimationResume(animation: Animator) { /* resumed */ }
})

// Use case: pause animations when the Activity is stopped
// to avoid wasting CPU on invisible animations
override fun onStop() {
    super.onStop()
    activeAnimators.forEach { it.pause() }
}

override fun onStart() {
    super.onStart()
    activeAnimators.forEach { it.resume() }
}

Danger — Animation Scale = 0 in Production
Some users disable animations in Accessibility settings (or Developer Options). When the animator duration scale is 0, all ValueAnimator-based animations complete instantly — onAnimationEnd() fires on the same frame as start(). If your code depends on animation duration for timing (e.g., delaying a state change until animation completes), it will break. Always handle the zero-duration case: check ValueAnimator.areAnimatorsEnabled() (API 26+) or use withEndAction / doOnEnd instead of postDelayed.
