Drawing & Canvas Internals

The Draw phase pipeline, Canvas and Paint internals, hardware vs software rendering, RenderNode / DisplayList architecture, Porter-Duff compositing, Path drawing, and text rendering — everything that turns a View tree into pixels

1. The Draw Phase — How Views Render

After measure and layout determine where every View goes, the draw phase determines what they look like. The framework calls View.draw(Canvas) on the root View, which cascades through the entire tree.

The 6 Steps of View.draw()

View.draw(Canvas) is the entry point called by the framework. You almost never override it — instead you override onDraw(). The method performs exactly 6 steps in a fixed order:

// AOSP View.draw() — simplified from android.view.View
public void draw(Canvas canvas) {
    // Step 1: Draw the background
    drawBackground(canvas)

    // Step 2: Save canvas layer for fading edges (if needed)
    val horizontalEdges = isHorizontalFadingEdgeEnabled
    val verticalEdges = isVerticalFadingEdgeEnabled
    if (!verticalEdges && !horizontalEdges) {
        // Fast path — no fading edges, skip save/restore
        onDraw(canvas)           // Step 3
        dispatchDraw(canvas)     // Step 4
        drawAutofilledHighlight(canvas)
        onDrawForeground(canvas) // Step 6 (scrollbars, foreground)
        return
    }

    // Slow path: save layer, draw content, draw fading edges, restore
    canvas.saveUnclippedLayer(left, top, right, bottom)
    onDraw(canvas)               // Step 3
    dispatchDraw(canvas)         // Step 4
    // Step 5: draw fading edges using gradient shaders
    drawFadingEdges(canvas)
    canvas.restoreUnclippedLayer()
    onDrawForeground(canvas)     // Step 6
}

Drawing Order in ViewGroups

Children are drawn in the order they appear in the ViewGroup — index 0 is the bottom-most (drawn first, painted over by later children), and the last child is the top-most. This is the painter's algorithm: later draws paint over earlier ones.

// ViewGroup.dispatchDraw() — simplified
override fun dispatchDraw(canvas: Canvas) {
    for (i in 0 until childCount) {
        val childIndex = getChildDrawingOrder(childCount, i)
        val child = getChildAt(childIndex)
        drawChild(canvas, child, drawingTime)
    }
}

// Custom drawing order:
class ReorderingLayout(context: Context) : FrameLayout(context) {
    init {
        isChildrenDrawingOrderEnabled = true
    }

    override fun getChildDrawingOrder(childCount: Int, drawingPosition: Int): Int {
        // Reverse drawing order: last child drawn first (bottom)
        return childCount - 1 - drawingPosition
    }
}

Z-Order and Elevation

View.setZ() and View.setElevation() affect drawing order. Views with higher Z values are drawn later (on top) and also cast shadows. The total Z = elevation + translationZ.

// Z affects both drawing order and shadow casting
view.elevation = 8f.dp     // Permanent elevation (e.g., FAB = 6dp, AppBar = 4dp)
view.translationZ = 4f.dp  // Temporary Z offset (e.g., pressed state lift)

// ViewGroup sorts children by Z for drawing:
// 1. Children with Z <= 0 drawn in layout order
// 2. Children with Z > 0 drawn after, sorted by Z value
// This is why elevated Views always appear above non-elevated siblings
Staff-Level Insight — draw() vs onDraw() vs dispatchDraw()

A common mistake is overriding draw() instead of onDraw(). If you override draw() without calling super.draw(), you break the entire 6-step pipeline — no background, no children, no scrollbars. Override onDraw() for leaf View drawing, and dispatchDraw() if you need to draw something over all children (e.g., a custom overlay in a ViewGroup). For drawing between the background and content, some developers override draw(), call super.draw(), and insert custom drawing at the right point — but this is fragile and breaks if the framework changes the step order. The correct pattern is to use onDrawForeground() (API 23+) for overlays, or a separate overlay View.

2. Canvas Internals

Canvas is the drawing surface — the abstraction through which all rendering commands flow. What backs the Canvas determines whether you're doing software or hardware rendering.

Two Kinds of Canvas

Canvas TypeBackingRenderingWhen Used
Canvas(Bitmap) CPU-side Bitmap Software (Skia CPU) Offscreen rendering, LAYER_TYPE_SOFTWARE, bitmap manipulation
RecordingCanvas RenderNode (DisplayList) Hardware (GPU via RenderThread) Default for all Views since API 14+ 
// Software Canvas — draws directly to a Bitmap
val bitmap = Bitmap.createBitmap(400, 400, Bitmap.Config.ARGB_8888)
val canvas = Canvas(bitmap)
canvas.drawCircle(200f, 200f, 100f, paint)
// bitmap now contains the circle — immediate execution

// Hardware Canvas — records commands for later GPU execution
// You never create RecordingCanvas directly; the framework provides it:
override fun onDraw(canvas: Canvas) {
    // 'canvas' is actually a RecordingCanvas when hardware accelerated
    canvas.drawCircle(200f, 200f, 100f, paint)
    // Nothing is drawn yet! The command is recorded into the RenderNode's DisplayList.
    // RenderThread will execute it on the GPU later.
}

Canvas State Stack

Canvas maintains a stack of transform and clip states. save() pushes the current state, restore() pops it. This lets you apply temporary transforms without affecting subsequent drawing.

override fun onDraw(canvas: Canvas) {
    // Draw a rotated rectangle without affecting other drawing
    canvas.save()                          // Push state
    canvas.translate(width / 2f, height / 2f)  // Move origin to center
    canvas.rotate(45f)                     // Rotate 45 degrees
    canvas.drawRect(-50f, -50f, 50f, 50f, paint)
    canvas.restore()                       // Pop state — rotation is gone

    // This text is NOT rotated — the restore() undid the transform
    canvas.drawText("Not rotated", 10f, 30f, textPaint)
}

Key Canvas Operations

CategoryMethodsNotes
Primitives drawRect(), drawCircle(), drawOval(), drawLine(), drawArc(), drawRoundRect() Basic shapes. All take a Paint parameter.
Complex drawPath(), drawBitmap(), drawText(), drawVertices() Paths for arbitrary shapes, bitmaps for images, text for typography.
Transform translate(), rotate(), scale(), skew(), concat(Matrix) Transform the coordinate system. Applied to all subsequent draws.
Clip clipRect(), clipPath(), clipOutRect(), clipOutPath() Restrict drawing to a region. clipOut* variants (API 26+) exclude a region.
State save(), restore(), saveLayer(), getSaveCount() save() is cheap; saveLayer() allocates an offscreen buffer.

save() vs saveLayer() — A Critical Distinction

// save() — saves transform and clip state only (CHEAP)
canvas.save()
canvas.clipRect(100f, 100f, 300f, 300f)
canvas.drawBitmap(bitmap, 0f, 0f, null)
canvas.restore()

// saveLayer() — allocates an offscreen buffer (EXPENSIVE)
// Required for Porter-Duff compositing, alpha blending of groups, etc.
canvas.saveLayer(bounds, paint)
// Everything drawn here goes to a temporary offscreen bitmap/texture
canvas.drawCircle(200f, 200f, 100f, dstPaint)
canvas.drawRect(150f, 150f, 250f, 250f, srcPaint) // Composited with circle
canvas.restore() // Offscreen buffer composited back to the main canvas
Performance Warning — saveLayer() Cost

saveLayer() allocates a temporary texture the size of the layer bounds. On a 1080×1920 device, a full-screen saveLayer() allocates ~8MB of GPU memory (1080 × 1920 × 4 bytes). Each saveLayer() also forces a GPU state flush. In a 60fps loop, this means allocating and deallocating 8MB sixty times per second. Always pass the tightest possible bounds to saveLayer(), and avoid it entirely if you can achieve the same effect with clipRect() or Paint.setAlpha().

3. Paint — The Drawing Brush

Paint describes how to draw: color, stroke, anti-aliasing, shaders, compositing modes, and more. It's the most parameter-rich object in the drawing pipeline.

Core Properties

val paint = Paint().apply {
    color = Color.BLUE
    style = Paint.Style.STROKE       // FILL, STROKE, or FILL_AND_STROKE
    strokeWidth = 4f
    isAntiAlias = true               // Smooth edges (always enable for production)
    strokeCap = Paint.Cap.ROUND      // BUTT, ROUND, SQUARE
    strokeJoin = Paint.Join.ROUND    // MITER, ROUND, BEVEL
    textSize = 48f
    typeface = Typeface.DEFAULT_BOLD
}

Filters and Effects

    0f                                  // Phase offset

paint.pathEffect = CornerPathEffect(15f)  // Round all corners
Staff-Level Insight — Paint Allocation in onDraw()

Creating Paint objects inside onDraw() is one of the most common performance mistakes in custom Views. onDraw() is called 60+ times per second during animations. Each Paint() allocation triggers a JNI call to create a native Skia SkPaint, plus the Kotlin object allocation itself. The garbage collector then has to clean these up, potentially causing GC pauses during animation frames. Always allocate Paint objects as class fields and reuse them. If you need to change a property per-frame, mutate the existing Paint — don't create a new one. Android Lint rule DrawAllocation catches this, but it only flags new Paint() — it won't catch allocations hidden inside helper methods.

4. Hardware Acceleration — DisplayList / RenderNode

Since Android 3.0, hardware acceleration is the default. Instead of drawing directly to pixels, the framework records drawing commands into a data structure that the GPU executes later on the RenderThread.

The Evolution: DisplayList → RenderNode

API LevelMechanismKey Change
11–20 (3.0–4.4)DisplayListDrawing commands recorded as a flat list. Entire list re-recorded on any change.
21+ (5.0+)RenderNodeEach View owns a RenderNode. Properties (alpha, translation, rotation) can change without re-recording the display list.

How RenderNode Recording Works

// AOSP View.updateDisplayListIfDirty() — simplified
fun updateDisplayListIfDirty() {
    val renderNode = mRenderNode

    if (mPrivateFlags and PFLAG_DRAWING_CACHE_VALID == 0  // Dirty?
        || !renderNode.hasDisplayList()) {

        // Re-record the DisplayList
        val canvas: RecordingCanvas = renderNode.beginRecording(width, height)
        try {
            if (this is ViewGroup) {
                dispatchDraw(canvas)   // Record children
            } else {
                draw(canvas)           // Record this View's drawing
            }
        } finally {
            renderNode.endRecording()  // Seal the DisplayList
        }
    }
    // If not dirty, the existing DisplayList is reused — zero work!
}

RenderNode Properties — The "Free" Animations

RenderNode stores certain properties separately from the DisplayList. Changing these properties does NOT require re-recording the DisplayList — the RenderThread simply applies them when traversing the tree.

RenderNode PropertyView APIRe-records DisplayList?
translationX / translationYsetTranslationX(), setTranslationY()No
scaleX / scaleYsetScaleX(), setScaleY()No
rotation / rotationX / rotationYsetRotation()No
alphasetAlpha()No
elevation / translationZsetElevation(), setTranslationZ()No
pivotX / pivotYsetPivotX(), setPivotY()No
clipToBoundssetClipToOutline()No
Content (onDraw output)invalidate()Yes
// ViewPropertyAnimator — exploits RenderNode properties
view.animate()
    .translationX(200f)
    .alpha(0.5f)
    .rotation(90f)
    .setDuration(300)
    .start()
// All three properties animate on the RenderThread.
// The UI thread is NOT involved per-frame — only the initial setup.
// No invalidate(), no onDraw(), no DisplayList re-recording.
Staff-Level Insight — Why invalidate() Is Expensive and setTranslationX() Is Free

When you call invalidate(), the View is marked dirty, and on the next frame the framework calls updateDisplayListIfDirty() which re-records the entire DisplayList by calling draw(canvas). This involves executing all your onDraw() code, creating Skia commands, and uploading them to the RenderThread. In contrast, setTranslationX() simply writes a float to the RenderNode's native struct — the RenderThread reads this value when traversing the tree and applies it as a matrix transform. No Java code runs, no DisplayList is re-recorded. This is why ViewPropertyAnimator and ObjectAnimator targeting these properties can achieve 60fps even when the UI thread is busy — the animation runs entirely on the RenderThread. In interviews, this is the key insight: RenderNode property changes are O(1); invalidate() is O(n) where n is the complexity of onDraw().

5. Software vs Hardware Rendering

Every View can be rendered in software mode, hardware mode, or with a hardware layer. Understanding the differences is critical for debugging rendering issues.

Comparison

AspectSoftware RenderingHardware Rendering

ExecutionImmediate, on UI threadDeferred, on RenderThread (GPU)
SpeedSlow for complex scenes10-100× faster for typical UI

Layer Types

// LAYER_TYPE_NONE (default) — no layer, normal rendering
view.setLayerType(View.LAYER_TYPE_NONE, null)

// LAYER_TYPE_SOFTWARE — forces software rendering for this View
// Use when you need an unsupported Canvas operation
view.setLayerType(View.LAYER_TYPE_SOFTWARE, null)

// LAYER_TYPE_HARDWARE — renders View to an offscreen GPU texture
// The texture is reused until the View is invalidated
view.setLayerType(View.LAYER_TYPE_HARDWARE, null)

// Hardware layer with a Paint — apply effects to the cached texture
val layerPaint = Paint().apply {
    colorFilter = ColorMatrixColorFilter(ColorMatrix().apply {
        setSaturation(0f)  // Grayscale the entire View
    })
}
view.setLayerType(View.LAYER_TYPE_HARDWARE, layerPaint)

When to Use Hardware Layers

// GOOD: Complex View that doesn't change during animation
// The View is rendered to a texture ONCE, then the texture is
// translated/rotated/faded — no re-drawing needed
view.setLayerType(View.LAYER_TYPE_HARDWARE, null)
view.animate()
    .translationX(300f)
    .alpha(0f)
    .withEndAction {
        view.setLayerType(View.LAYER_TYPE_NONE, null) // Remove layer when done
    }
    .start()

// BAD: View that changes every frame (e.g., progress bar, video)
// The hardware layer texture must be re-created every frame,
// DOUBLING the GPU work (render to texture + composite texture)
progressBar.setLayerType(View.LAYER_TYPE_HARDWARE, null) // WASTEFUL!
Hardware Layer Gotcha — The Double-Edged Sword

Hardware layers save GPU time only if the View doesn't change. If the View is invalidated every frame (e.g., a RecyclerView scrolling, a ProgressBar animating), the layer texture must be re-rendered every frame — you pay the cost of rendering to the texture plus compositing the texture, which is worse than no layer at all. Use "Show Hardware Layers Updates" in Developer Options — green flashes indicate the layer is being re-rendered. If it flashes every frame, remove the layer.

6. Porter-Duff Compositing

Porter-Duff modes define how source (what you're drawing) and destination (what's already on the canvas) pixels are combined. They're the foundation of all compositing in Android.

Key Modes

Common Compositing Patterns

// Pattern 1: Circular image masking (avatar)
class CircularImageView(context: Context, attrs: AttributeSet?) : ImageView(context, attrs) {
    private val maskPaint = Paint(Paint.ANTI_ALIAS_FLAG).apply {
        xfermode = PorterDuffXfermode(PorterDuff.Mode.SRC_IN)
    }
    private val circlePaint = Paint(Paint.ANTI_ALIAS_FLAG).apply {
        color = Color.WHITE
    }

    override fun onDraw(canvas: Canvas) {
        // Must use saveLayer for Porter-Duff to work correctly
        canvas.saveLayer(0f, 0f, width.toFloat(), height.toFloat(), null)

        // Step 1: Draw the circle (destination)
        val cx = width / 2f
        val cy = height / 2f
        val radius = minOf(cx, cy)
        canvas.drawCircle(cx, cy, radius, circlePaint)

        // Step 2: Draw the image with SRC_IN (source clipped to circle)
        canvas.saveLayer(0f, 0f, width.toFloat(), height.toFloat(), maskPaint)
        super.onDraw(canvas)
        canvas.restore()

        canvas.restore()
    }
}

// Pattern 2: Punch-hole / cutout effect using DST_OUT
fun drawWithCutout(canvas: Canvas, holePath: Path) {
    canvas.saveLayer(bounds, null)
    drawContent(canvas)  // Draw the full content

    val cutoutPaint = Paint().apply {
        xfermode = PorterDuffXfermode(PorterDuff.Mode.DST_OUT)
        color = Color.WHITE
    }
    canvas.drawPath(holePath, cutoutPaint)  // Erase the hole
    canvas.restore()
}

7. Path and Shape Drawing

Path defines arbitrary shapes using a sequence of line, arc, and curve commands. It's the most flexible drawing primitive.

Building Paths

val path = Path().apply {
    moveTo(100f, 100f)           // Starting point
    lineTo(300f, 100f)           // Straight line
    quadTo(350f, 200f, 300f, 300f) // Quadratic Bézier curve (1 control point)
    cubicTo(250f, 350f, 150f, 350f, 100f, 300f) // Cubic Bézier (2 control points)
    arcTo(RectF(50f, 200f, 150f, 300f), 180f, 90f) // Arc segment
    close()                      // Close the path back to the start
}

// Path directions matter for fill rules
val outerRect = Path().apply {
    addRect(0f, 0f, 200f, 200f, Path.Direction.CW)
}
val innerRect = Path().apply {
    addRect(50f, 50f, 150f, 150f, Path.Direction.CCW) // Counter-clockwise = hole
}
outerRect.op(innerRect, Path.Op.DIFFERENCE) // Creates a frame shape

PathMeasure — Animating Along a Path

// Measure a path and get positions along it
val pathMeasure = PathMeasure(curvePath, false)
val totalLength = pathMeasure.length

// Get position and tangent at 50% of the path
val pos = FloatArray(2)
val tan = FloatArray(2)
pathMeasure.getPosTan(totalLength * 0.5f, pos, tan)
// pos[0], pos[1] = x, y coordinates at the midpoint
// tan[0], tan[1] = tangent vector (for rotation)

// Animate an object along a path
val animator = ValueAnimator.ofFloat(0f, 1f).apply {
    duration = 2000
    addUpdateListener { anim ->
        val fraction = anim.animatedValue as Float
        pathMeasure.getPosTan(totalLength * fraction, pos, tan)
        movingView.translationX = pos[0]
        movingView.translationY = pos[1]
        movingView.rotation = Math.toDegrees(
            Math.atan2(tan[1].toDouble(), tan[0].toDouble())
        ).toFloat()
    }
}

// Extract a segment of a path (useful for progress indicators)
val segmentPath = Path()
pathMeasure.getSegment(0f, totalLength * progress, segmentPath, true)

clipPath() — Clipping to Arbitrary Shapes

// Clip to a rounded rectangle
override fun onDraw(canvas: Canvas) {
    val clipPath = Path().apply {
        addRoundRect(
            RectF(0f, 0f, width.toFloat(), height.toFloat()),
            cornerRadius, cornerRadius,
            Path.Direction.CW
        )
    }
    canvas.save()
    canvas.clipPath(clipPath)
    super.onDraw(canvas)  // Content is clipped to rounded rect
    canvas.restore()
}

// Performance note: clipPath() with anti-aliasing requires a saveLayer()
// on older APIs. On API 18+, hardware-accelerated clipPath() is supported
// but without anti-aliased edges. For smooth rounded corners, prefer
// OutlineProvider + setClipToOutline(true) which uses the GPU clip.
Best Practice — Rounded Corners

For simple rounded corners, avoid clipPath(). Instead use ViewOutlineProvider + setClipToOutline(true), which clips on the GPU with anti-aliased edges and zero saveLayer() cost. For ShapeableImageView from Material Components, this is handled automatically. Reserve clipPath() for truly arbitrary shapes that can't be expressed as an Outline.

8. Text Rendering

Text is one of the most complex and expensive things to render. The pipeline involves shaping (converting characters to glyphs), layout (line breaking, bidi), and rasterization (drawing glyphs to pixels).

Low-Level Text Drawing

// drawText() draws at the BASELINE, not the top-left
val textPaint = TextPaint().apply {
    textSize = 48f
    color = Color.BLACK
    isAntiAlias = true
    typeface = Typeface.create("sans-serif-medium", Typeface.NORMAL)
}

override fun onDraw(canvas: Canvas) {
    // y parameter is the baseline position, NOT the top of the text
    canvas.drawText("Hello", 20f, 100f, textPaint)

    // To position text by its top edge:
    val metrics = textPaint.fontMetrics
    val topY = 100f - metrics.ascent  // ascent is negative
    canvas.drawText("Hello", 20f, topY, textPaint)
}

Measuring Text

// measureText() — advance width (includes inter-character spacing)
val advanceWidth = paint.measureText("Hello World")

// getTextBounds() — tight bounding box around the glyphs
val bounds = Rect()
paint.getTextBounds("Hello World", 0, 11, bounds)
// bounds.width() <= advanceWidth (no trailing space)
// bounds.height() = tight height around actual glyphs

// FontMetrics — the font's vertical metrics
val fm = paint.fontMetrics
// fm.top      — highest point of any glyph (negative, above baseline)
// fm.ascent   — recommended top of text (negative)
// fm.descent  — recommended bottom of text (positive)
// fm.bottom   — lowest point of any glyph (positive)
// fm.leading  — extra space between lines

// Line height = fm.descent - fm.ascent + fm.leading

Staff-Level Insight — Text Rendering Cost and Caching

Text rendering is the single most expensive Canvas operation in typical Android UIs. The pipeline is: Unicode → ICU shaping → Harfbuzz glyph selection → FreeType rasterization → glyph cache lookup → GPU texture upload. TextView internally caches its StaticLayout and only re-creates it when text, width, or text appearance changes. If you're building a custom View that draws text, you must cache your StaticLayout similarly — recreating it in onDraw() is catastrophically expensive. The glyph cache (TextLayoutCache in AOSP) is an LRU cache that stores shaped text runs. Cache misses trigger the full shaping pipeline, which can take 1-5ms for complex scripts (Arabic, Thai, Devanagari). This is why scrolling through text-heavy lists with diverse scripts can cause jank on lower-end devices.

9. Staff-Level Corner Cases & Deep Dives

RecordingCanvas Silent Failures

When hardware acceleration is enabled, onDraw() receives a RecordingCanvas. Unsupported operations don't throw exceptions — they silently do nothing or produce incorrect results. This is one of the most confusing debugging scenarios.

override fun onDraw(canvas: Canvas) {
    // This works in software, silently fails in hardware:
    canvas.drawPicture(myPicture)  // Nothing drawn, no exception

    // Check if hardware accelerated:
    if (canvas.isHardwareAccelerated) {
        // Use alternative rendering path
        drawPictureFallback(canvas)
    }

    // Or force software rendering for this View only:
    // setLayerType(View.LAYER_TYPE_SOFTWARE, null)
}

Canvas Save/Restore Mismatch

// DANGEROUS: mismatched save/restore corrupts the state stack
override fun onDraw(canvas: Canvas) {
    canvas.save()
    canvas.save()
    canvas.rotate(45f)
    // ... drawing ...
    canvas.restore()
    // Forgot one restore()! The next View drawn on this Canvas
    // will inherit the extra save state, causing subtle rendering bugs.

    // SAFE: use restoreToCount()
    val saveCount = canvas.save()
    canvas.save()
    canvas.rotate(45f)
    // ... drawing ...
    canvas.restoreToCount(saveCount) // Restores ALL saves back to this point
}

setWillNotDraw() Optimization

// ViewGroup sets this to true by default — it assumes ViewGroups
// don't draw themselves, only their children.
// If your custom ViewGroup needs to draw (e.g., custom background),
// you MUST clear this flag:
class DrawingViewGroup(context: Context) : ViewGroup(context) {
    init {
        setWillNotDraw(false) // Required! Otherwise onDraw() is never called
    }

    override fun onDraw(canvas: Canvas) {
        // This is only called if setWillNotDraw(false)
        canvas.drawColor(Color.LTGRAY)
    }
}

// Example of 4× overdraw:
//   1. Window background (white)
//   2. Activity background (white)     ← redundant!
//   3. FrameLayout background (white)  ← redundant!
//   4. CardView background (white)     ← the only one that matters

// Fix: remove redundant backgrounds
// In your theme:

Staff-Level Insight — The Full Draw Pipeline in One Frame

Here's the complete journey of a single frame's draw phase at the AOSP level: (1) ViewRootImpl.performDraw() is called during the Choreographer CALLBACK_TRAVERSAL. (2) It calls ThreadedRenderer.draw() which begins a frame on the RenderThread. (3) updateDisplayListIfDirty() walks the View tree top-down, calling RenderNode.beginRecording() → View.draw(RecordingCanvas) → RenderNode.endRecording() for each dirty View. (4) The RenderNode tree is synced to the RenderThread (the "sync" phase — UI thread is briefly blocked). (5) RenderThread walks the RenderNode tree, flattens it into Skia draw ops, batches/reorders them for efficiency, and issues OpenGL/Vulkan commands. (6) The completed frame buffer is submitted to the BufferQueue, and SurfaceFlinger composites it on the next VSYNC. Understanding this pipeline is essential for diagnosing where jank occurs — is it in recording (step 3), sync (step 4), or GPU execution (step 5)?

Direct RenderNode Usage (API 29+)

// Since API 29, RenderNode is a public API — you can create
// and manage them directly for advanced rendering scenarios
val renderNode = RenderNode("custom-node").apply {
    setPosition(0, 0, 400, 400)
    translationX = 50f
    alpha = 0.8f
}

// Record drawing commands
val recordingCanvas = renderNode.beginRecording()
recordingCanvas.drawCircle(200f, 200f, 100f, paint)
renderNode.endRecording()

// Draw the RenderNode in onDraw()
override fun onDraw(canvas: Canvas) {
    if (canvas.isHardwareAccelerated) {
        canvas.drawRenderNode(renderNode)
        // The RenderNode's transform (translation, alpha) is applied
        // automatically by the GPU — no CPU cost
    }
}

// Animate RenderNode properties directly — no invalidate() needed
renderNode.translationX = 100f  // Cheap property change
invalidate()  // Still need to trigger a frame, but the DisplayList
              // inside the RenderNode is NOT re-recorded
Staff-Level Insight — RenderNode as a Micro-Optimization

Direct RenderNode usage (API 29+) lets you create sub-trees within a single View's drawing that can be independently transformed without re-recording. Imagine a custom View with 50 chart bars — instead of re-recording all 50 bars when one animates, you give each bar its own RenderNode. Animating one bar's translationY doesn't re-record the other 49. This is exactly what RecyclerView's item animator does internally, and it's how Compose's Modifier.graphicsLayer works under the hood — each graphicsLayer creates a separate RenderNode. In an L6 interview, connecting View system RenderNode to Compose's graphicsLayer demonstrates deep cross-system understanding.
