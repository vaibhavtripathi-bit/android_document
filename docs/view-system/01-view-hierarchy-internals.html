01 — View Hierarchy Internals

ViewRootImpl, DecorView, AttachInfo, the traversal pipeline, invalidation mechanics, and how the View tree connects to the Window system — the foundation everything else builds on

1. The View Tree Architecture

Every screen in Android is a tree of View and ViewGroup objects. This tree is the fundamental data structure of the entire UI framework — measure, layout, draw, touch dispatch, accessibility, and focus traversal all operate by walking this tree.

DecorView — The Root of Every Window

The root of the View tree is always a DecorView, which extends FrameLayout. It is the top-level View in every Activity window and is owned by the PhoneWindow (the sole concrete implementation of the abstract Window class).

DecorView is responsible for:

  • Hosting the system chrome — status bar background, navigation bar background, caption bar in freeform mode
  • Providing the content frame (android.R.id.content) where your layout lives
  • Handling window-level insets (system bars, display cutouts, IME)
  • Dispatching key events and back-press handling

From setContentView() to a Live Tree

When you call setContentView(R.layout.main), the following chain executes:

// Activity.setContentView() delegates to PhoneWindow
Activity.setContentView(layoutResID)
  → PhoneWindow.setContentView(layoutResID)
    → installDecor()                          // creates DecorView if needed
      → generateDecor()                       // new DecorView(context)
      → generateLayout(mDecor)                // inflates system layout into DecorView
        // system layout depends on window features (action bar, title, etc.)
        // always contains a FrameLayout with id android.R.id.content
    → mLayoutInflater.inflate(layoutResID, mContentParent)
      // mContentParent = the FrameLayout (android.R.id.content)
      // LayoutInflater recursively parses XML → creates View objects
      // → attaches them as children of mContentParent

Staff Insight
The system layout inflated into DecorView varies based on window feature flags (requestWindowFeature()). There are ~15 different system layout resources in AOSP (screen_simple.xml, screen_action_bar.xml, screen_toolbar.xml, etc.). The generateLayout() method in PhoneWindow selects the appropriate one based on the theme attributes and requested features. This is why requestWindowFeature() must be called before setContentView() — the system layout hasn't been inflated yet.

2. ViewRootImpl — The Bridge Between View and Window

ViewRootImpl is the most important class in the View system that most developers never directly interact with. It is not a View — it implements ViewParent and acts as the bridge between the View tree and the WindowManagerService (WMS).

One ViewRootImpl Per Window

Every window in Android has its own ViewRootImpl. An Activity has one, but so does every Dialog, Toast, PopupWindow, and system overlay. They are independent rendering pipelines:

Window TypeViewRootImplSurfaceInput Channel
Activity1 per ActivityOwn SurfaceOwn InputChannel
DialogSeparate from ActivityOwn SurfaceOwn InputChannel
ToastSeparateOwn SurfaceNone (no input)
PopupWindowSeparateOwn SurfaceOwn InputChannel
System AlertSeparateOwn SurfaceOwn InputChannel

Creation Path

ViewRootImpl is created when a View tree is added to the window system:

// The chain that creates ViewRootImpl:
WindowManager.addView(view, layoutParams)
  → WindowManagerImpl.addView(view, params)
    → WindowManagerGlobal.addView(view, params, display, parentWindow, userId)
      → root = new ViewRootImpl(view.getContext(), display)
      → root.setView(view, params, panelParentView, userId)

// WindowManagerGlobal maintains parallel arrays:
//   mViews[]  — the root Views (DecorViews)
//   mRoots[]  — the ViewRootImpl instances
//   mParams[] — the WindowManager.LayoutParams

What ViewRootImpl Holds

ViewRootImpl is the central coordinator. It owns or manages:

  • mView — reference to the root View (typically DecorView)
  • mSurface / mSurfaceControl — the drawing target allocated by SurfaceFlinger
  • mChoreographer — schedules work on VSYNC boundaries
  • mAttachInfo — shared context for all Views in the tree (see Section 3)
  • mInputChannel / mInputEventReceiver — receives touch/key events from InputDispatcher
  • mDirty — the dirty region that needs redrawing
  • mTraversalBarrier — a sync barrier posted to the MessageQueue to prioritize traversal messages
  • mWinFrame — the window's frame as assigned by WMS
  • IWindowSession — Binder proxy to WMS for window operations

ViewRootImpl.setView() — The Connection Point

setView() is where the View tree becomes a live window. This single method call triggers a cascade of critical setup:

// Simplified ViewRootImpl.setView() — AOSP
void setView(View view, WindowManager.LayoutParams attrs, View panelParentView) {
    synchronized (this) {
        if (mView == null) {
            mView = view;

            // 1. Schedule the first traversal
            requestLayout();   // → scheduleTraversals()

            // 2. Set up input event pipeline
            mInputChannel = new InputChannel();

            // 3. Add window to WMS via Binder IPC
            //    This is the cross-process call that registers the window
            res = mWindowSession.addToDisplayAsUser(
                mWindow,           // IWindow — WMS's callback handle
                mWindowAttributes,
                getHostVisibility(),
                mDisplay.getDisplayId(),
                userId,
                mInsetsController,
                mInputChannel,     // WMS fills this with the input pipe
                mTempInsets,
                mTempControls
            );

            // 4. Set up input event receiver on the Looper
            mInputEventReceiver = new WindowInputEventReceiver(
                mInputChannel, Looper.myLooper()
            );

            // 5. Assign the View tree's parent to this ViewRootImpl
            view.assignParent(this);
        }
    }
}
Staff Insight
ViewRootImpl posts a sync barrier to the MessageQueue before scheduling a traversal via Choreographer. A sync barrier blocks all synchronous messages (regular Handler.post()) and only allows asynchronous messages through. The Choreographer callback is posted as an async message, so it jumps ahead of any pending synchronous work. This is why View.post() runnables execute after the traversal — they are synchronous messages blocked by the barrier until the traversal completes and the barrier is removed.

3. View.AttachInfo — The Shared Context

View.AttachInfo is a package-private inner class of View that holds shared state for every View in a window. There is exactly one instance per ViewRootImpl, and every View in the tree shares the same reference.

What AttachInfo Contains

FieldPurpose
mViewRootImplBack-reference to the ViewRootImpl
mHandlerHandler bound to the UI thread's Looper — used by View.post()
mRootViewThe root View (DecorView)
mSurfaceThe Surface this window draws to
mWindowTokenIBinder token identifying this window to WMS
mTreeObserverThe ViewTreeObserver for this tree
mHardwareAcceleratedWhether hardware-accelerated rendering is enabled
mDisplayIdWhich display this window is on
mContentInsetsInsets consumed by system bars
mHasWindowFocusWhether this window currently has focus
mWindowVisibilityCurrent visibility of the window

How AttachInfo Propagates

When a View is added to the tree, AttachInfo flows down from the root:

// ViewGroup.addViewInner() — simplified
void addViewInner(View child, int index, LayoutParams params, boolean preventRequestLayout) {
    // ... parameter validation ...

    if (mAttachInfo != null) {
        // Propagate AttachInfo down to the new child (and its subtree)
        child.dispatchAttachedToWindow(mAttachInfo, mViewFlags & VISIBILITY_MASK);
    }
    // ...
}

// View.dispatchAttachedToWindow()
void dispatchAttachedToWindow(AttachInfo info, int visibility) {
    mAttachInfo = info;

    // Execute any pending View.post() runnables that were queued before attach
    if (mRunQueue != null) {
        mRunQueue.executeActions(info.mHandler);
        mRunQueue = null;
    }

    onAttachedToWindow();
    // ...
}

// ViewGroup overrides to recurse into children:
void dispatchAttachedToWindow(AttachInfo info, int visibility) {
    super.dispatchAttachedToWindow(info, visibility);
    for (int i = 0; i < mChildrenCount; i++) {
        children[i].dispatchAttachedToWindow(info, visibility);
    }
}
Common Trap
View.mAttachInfo is null until the View is attached to a window. Many View operations silently fail or throw when called before attach. For example, View.getLocationOnScreen() returns [0, 0], View.getWindowToken() returns null, and View.startAnimation() won't schedule properly. Always check View.isAttachedToWindow() or use View.doOnAttach { } (Kotlin extension) when you need guaranteed attach state.

4. The Traversal — performTraversals()

ViewRootImpl.performTraversals() is the single most important method in the entire View system. It orchestrates the three phases of rendering — Measure, Layout, Draw — in a single pass triggered by VSYNC.

How a Traversal Gets Scheduled

// Something triggers a traversal (requestLayout, invalidate, etc.)
ViewRootImpl.scheduleTraversals()
  → mTraversalBarrier = mHandler.getLooper().getQueue().postSyncBarrier()
  → mChoreographer.postCallback(
        Choreographer.CALLBACK_TRAVERSAL,
        mTraversalRunnable,    // → doTraversal()
        null
    )

// On next VSYNC:
mTraversalRunnable.run()
  → doTraversal()
    → mHandler.getLooper().getQueue().removeSyncBarrier(mTraversalBarrier)
    → performTraversals()

performTraversals() Flow

This method is ~900 lines in AOSP. Here is the essential flow:

performTraversals()
│
├─ 1. Window size / configuration changed?
│     → relayoutWindow()  ← Binder IPC to WMS
│     → WMS assigns window frame, allocates/resizes Surface
│     → mSurface is now valid for drawing
│
├─ 2. MEASURE PHASE
│     → desiredWindowWidth/Height from WMS frame
│     → childWidthMeasureSpec = getRootMeasureSpec(windowWidth, lp.width)
│     → childHeightMeasureSpec = getRootMeasureSpec(windowHeight, lp.height)
│     → performMeasure(childWidthMeasureSpec, childHeightMeasureSpec)
│        → mView.measure(widthSpec, heightSpec)
│           → onMeasure() — recursive down the entire tree
│
├─ 3. LAYOUT PHASE
│     → performLayout(lp, desiredWindowWidth, desiredWindowHeight)
│        → mView.layout(0, 0, mView.getMeasuredWidth(), mView.getMeasuredHeight())
│           → onLayout() — recursive, each ViewGroup positions children
│
├─ 4. Report draw if first traversal
│     → pendingDrawFinished or reportNextDraw
│
└─ 5. DRAW PHASE
      → performDraw()
         → draw(fullRedrawNeeded)
            → Hardware path: mAttachInfo.mThreadedRenderer.draw(mView, ...)
            │  → builds/updates RenderNode display list
            │  → hands off to RenderThread
            │
            └─ Software path: drawSoftware(surface, ...)
               → surface.lockCanvas(dirty)
               → mView.draw(canvas)  — recursive
               → surface.unlockCanvasAndPost(canvas)

The Three Phases at a Glance

PhaseTriggerWalks Tree?What It Does
MeasurerequestLayout()Top-downEach View calculates its desired width/height via MeasureSpec negotiation
LayoutrequestLayout()Top-downEach ViewGroup assigns final positions (left, top, right, bottom) to children
Drawinvalidate() or requestLayout()Top-downEach View renders itself to the Canvas (or updates its RenderNode)
Key Detail
invalidate() only triggers the draw phase. requestLayout() triggers all three phases. This distinction is critical for performance — if only visual appearance changed (color, alpha, text content), use invalidate(). If size or position might change, you need requestLayout().

5. Invalidation — Triggering Redraws

Invalidation is the mechanism by which a View tells the system "I need to be redrawn." It is the most common trigger for rendering work.

The Invalidation Chain

// View.invalidate() — simplified AOSP flow
public void invalidate() {
    invalidate(true);  // invalidateCache = true
}

void invalidateInternal(int l, int t, int r, int b,
                         boolean invalidateCache, boolean fullInvalidate) {
    // Skip if View is not visible and not animating
    if (skipInvalidate()) return;

    // Set dirty flags
    if (invalidateCache) {
        mPrivateFlags |= PFLAG_INVALIDATED;    // mark display list as invalid
    }
    mPrivateFlags |= PFLAG_DIRTY;

    // Propagate up to parent
    final AttachInfo ai = mAttachInfo;
    final ViewParent p = mParent;
    if (p != null && ai != null) {
        final Rect damage = ai.mTmpInvalRect;
        damage.set(l, t, r, b);
        p.invalidateChild(this, damage);
    }
}

// ViewGroup.invalidateChild() propagates up the tree:
// child → parent → parent → ... → ViewRootImpl

// ViewRootImpl.invalidateChildInParent()
//   → union the dirty rect into mDirty
//   → scheduleTraversals()  (only draw phase will execute)

Hardware-Accelerated Invalidation

With hardware acceleration (default since Android 4.1), invalidation is fundamentally different:

  • Each View has a RenderNode (display list) that records its drawing commands
  • invalidate() marks only that View's RenderNode as dirty
  • The parent's RenderNode still references the child's RenderNode — it does not need to re-record
  • Only the invalidated View's display list is rebuilt; the rest of the tree is untouched
  • This is O(1) per invalidated View, not O(n) for the tree

In software rendering (legacy), invalidation uses dirty rectangles. The union of all dirty rects is computed, and the entire draw pass clips to that region. Every View that intersects the dirty rect must redraw — much more expensive.

invalidate() vs postInvalidate()

// invalidate() — must be called on the UI thread
view.invalidate()

// postInvalidate() — safe from any thread
// Posts to the UI thread's Handler via AttachInfo
view.postInvalidate()

// Under the hood:
public void postInvalidate() {
    val attachInfo = mAttachInfo ?: return
    attachInfo.mViewRootImpl.dispatchInvalidateDelayed(this, 0)
}
// → Handler.sendMessageDelayed(MSG_INVALIDATE, view, delayMillis)

6. requestLayout() — Triggering Re-measurement

requestLayout() is the heavier sibling of invalidate(). It signals that a View's size or position may have changed, requiring the full measure → layout → draw pipeline.

The requestLayout() Chain

// View.requestLayout() — simplified AOSP
public void requestLayout() {
    if (mMeasureCache != null) mMeasureCache.clear();

    // Set flags that force measure and layout
    mPrivateFlags |= PFLAG_FORCE_LAYOUT;
    mPrivateFlags |= PFLAG_INVALIDATED;

    // Propagate up to parent
    if (mParent != null && !mParent.isLayoutRequested()) {
        mParent.requestLayout();
    }
}

// This propagates all the way up:
// View → ViewGroup → ViewGroup → ... → ViewRootImpl.requestLayout()

// ViewRootImpl.requestLayout()
public void requestLayout() {
    if (!mHandlingLayoutInLayoutRequest) {
        checkThread();                    // must be UI thread
        mLayoutRequested = true;
        scheduleTraversals();             // full traversal: measure + layout + draw
    }
}

The PFLAG_FORCE_LAYOUT Optimization

Not every View in the tree gets re-measured on every requestLayout(). The flag propagation creates a dirty path from the requesting View up to the root. During the measure pass:

// View.measure() — simplified
public final void measure(int widthMeasureSpec, int heightMeasureSpec) {
    // Only re-measure if forced OR if the MeasureSpec changed
    boolean forceLayout = (mPrivateFlags & PFLAG_FORCE_LAYOUT) == PFLAG_FORCE_LAYOUT;
    boolean specChanged = widthMeasureSpec != mOldWidthMeasureSpec
                       || heightMeasureSpec != mOldHeightMeasureSpec;

    if (forceLayout || specChanged) {
        onMeasure(widthMeasureSpec, heightMeasureSpec);
        mPrivateFlags |= PFLAG_LAYOUT_REQUIRED;  // ensure layout() runs too
    }

    mOldWidthMeasureSpec = widthMeasureSpec;
    mOldHeightMeasureSpec = heightMeasureSpec;
}

Views not on the dirty path skip onMeasure() entirely because their PFLAG_FORCE_LAYOUT is not set and their MeasureSpec hasn't changed. This is a critical optimization for deep trees.

Danger — Infinite Layout Loop
Calling requestLayout() from inside onLayout() or onMeasure() creates an infinite loop. The framework detects this and logs "requestLayout() improperly called during layout", but in older API levels it could cause ANRs. A common trigger: setting text on a TextView inside onLayout(), which internally calls requestLayout() because the text bounds changed. Always separate layout-triggering logic from the layout pass itself.
Staff Insight
Multiple requestLayout() calls within the same frame are coalesced. Once ViewRootImpl.scheduleTraversals() has been called, the mTraversalScheduled flag is set to true, and subsequent calls to scheduleTraversals() are no-ops until the traversal executes. This means you can freely call requestLayout() on multiple Views in the same frame without triggering multiple traversals — they all collapse into a single measure/layout/draw pass.

7. ViewTreeObserver — Lifecycle Callbacks

ViewTreeObserver provides a set of callbacks for observing events in the View tree's lifecycle. It is the primary mechanism for reacting to layout completion, draw scheduling, and scroll events.

Key Listeners

ListenerWhen FiredCommon Use
OnPreDrawListenerAfter layout, before drawShared element transitions, last-minute adjustments; return false to cancel the draw
OnGlobalLayoutListenerAfter layout completesGetting final View dimensions, keyboard visibility detection
OnScrollChangedListenerAny View in the tree scrollsParallax effects, scroll-linked animations
OnWindowFocusChangeListenerWindow gains/loses focusPausing animations when backgrounded
OnWindowAttachListenerView tree attached/detachedCleanup, resource management
OnDrawListenerEvery draw passFrame-level monitoring (use sparingly)

Getting Dimensions After Layout

// The classic pattern — get dimensions after layout completes
view.viewTreeObserver.addOnGlobalLayoutListener(object : ViewTreeObserver.OnGlobalLayoutListener {
    override fun onGlobalLayout() {
        // Remove immediately to avoid repeated calls
        view.viewTreeObserver.removeOnGlobalLayoutListener(this)

        val width = view.width
        val height = view.height
        // Now safe to use dimensions
    }
})

// Modern Kotlin alternative using doOnLayout extension:
view.doOnLayout {
    val width = it.width
    val height = it.height
}
// doOnLayout uses OnPreDrawListener internally and auto-removes

The Floating Observer Problem

View.getViewTreeObserver() behaves differently depending on attach state:

  • Before attach: returns a "floating" ViewTreeObserver owned by the View itself
  • After attach: returns the ViewTreeObserver from AttachInfo (shared by all Views in the tree)
  • When the View is attached, listeners from the floating observer are merged into the real observer
  • When the View is detached, the observer is marked as "dead" — adding listeners to a dead observer throws IllegalStateException
// Safe pattern — always check isAlive()
val vto = view.viewTreeObserver
if (vto.isAlive) {
    vto.addOnGlobalLayoutListener { /* ... */ }
}
Memory Leak Trap
ViewTreeObserver listeners hold a strong reference to their enclosing scope. If you add an OnGlobalLayoutListener from an Activity or Fragment and forget to remove it, the listener keeps the entire View tree alive through the ViewTreeObserver → AttachInfo → ViewRootImpl → DecorView chain. Always remove listeners when done, or use the one-shot pattern shown above. The doOnLayout / doOnPreDraw Kotlin extensions handle removal automatically.

Key Lifecycle Details

// onAttachedToWindow — start ongoing work
override fun onAttachedToWindow() {
    super.onAttachedToWindow()
    // Safe to register observers, start animations, acquire resources
    // mAttachInfo is now non-null
    // View.post() will execute on the UI thread
}

// onDetachedFromWindow — the View's "onDestroy"
override fun onDetachedFromWindow() {
    super.onDetachedFromWindow()
    // MUST cleanup here:
    // - Cancel running animations (animator.cancel())
    // - Remove ViewTreeObserver listeners
    // - Unregister broadcast receivers
    // - Release hardware resources (Bitmap, SurfaceTexture)
    // - Cancel pending Handler callbacks
}

// onVisibilityChanged — called for THIS View and ancestors
override fun onVisibilityChanged(changedView: View, visibility: Int) {
    super.onVisibilityChanged(changedView, visibility)
    // changedView may be an ancestor, not necessarily this View
    // visibility: VISIBLE (0), INVISIBLE (4), GONE (8)
    // Use to pause/resume expensive operations like video playback
}
Staff Insight
onDetachedFromWindow() is not guaranteed to be called if the process is killed. For Activity-scoped cleanup, prefer Lifecycle observers or onDestroy(). However, for View-internal resources (animations, hardware layers, registered listeners), onDetachedFromWindow() is the correct place because it fires when the View is removed from the tree — which happens during Activity.onDestroy() when the window is removed, during Fragment transactions, or when a parent ViewGroup calls removeView().

Activity.makeVisible() — When the Window Actually Appears

The View tree is created during onCreate() via setContentView(), but the window doesn't appear on screen until makeVisible() is called during onResume():

// Activity.makeVisible() — called by ActivityThread after onResume()
void makeVisible() {
    if (!mWindowAdded) {
        val wm = getWindowManager()
        wm.addView(mDecor, getWindow().getAttributes())
        // ↑ This creates ViewRootImpl, calls WMS.addWindow(),
        //   allocates Surface, schedules first traversal
        mWindowAdded = true
    }
    mDecor.visibility = View.VISIBLE
}

// Timeline:
// onCreate()  → setContentView() → DecorView + View tree created (no window yet)
// onStart()   → (nothing window-related)
// onResume()  → (nothing yet)
// makeVisible() → WindowManager.addView() → ViewRootImpl → WMS.addWindow()
//              → Surface allocated → first performTraversals() → pixels on screen

Surface Allocation

The Surface that the View tree draws to is not created by the app — it is allocated by SurfaceFlinger and passed to ViewRootImpl via WMS:

  1. ViewRootImpl.setView() → mWindowSession.addToDisplayAsUser() → WMS creates WindowState
  2. First performTraversals() → relayoutWindow() → WMS asks SurfaceFlinger to create a SurfaceControl
  3. SurfaceFlinger creates a Layer backed by a BufferQueue
  4. WMS returns the SurfaceControl to ViewRootImpl, which wraps it in a Surface
  5. ViewRootImpl draws to the Surface (dequeues a buffer, draws, queues it back)
  6. SurfaceFlinger composites all Layers and sends the final frame to the display via HWC

10. Staff-Level Corner Cases

Single-Threaded Rendering

ViewRootImpl runs entirely on the main (UI) thread. All traversals — measure, layout, draw (the recording phase) — are single-threaded. The only concurrency is with the RenderThread, which takes the recorded display list and renders it on the GPU asynchronously. This means:

  • You cannot call View.invalidate() or View.requestLayout() from a background thread (throws CalledFromWrongThreadException)
  • The checkThread() in ViewRootImpl.requestLayout() compares Thread.currentThread() against the thread that created the ViewRootImpl
  • Technically, if you create a ViewRootImpl on a non-main thread (via WindowManager.addView() from a HandlerThread with a Looper), that View tree runs on that thread — but this is extremely rare and fragile

View.post() Before Attach

Calling View.post(Runnable) before the View is attached to a window doesn't crash — the Runnable is queued in a HandlerActionQueue (formerly RunQueue):

// View.post() — simplified
public boolean post(Runnable action) {
    val attachInfo = mAttachInfo
    if (attachInfo != null) {
        // Attached — post directly to the UI thread Handler
        return attachInfo.mHandler.post(action)
    }

    // Not attached yet — queue for later
    getRunQueue().post(action)
    return true
}

// When the View is finally attached:
// dispatchAttachedToWindow() → mRunQueue.executeActions(info.mHandler)
// All queued Runnables are posted to the Handler at that point
Subtle Timing
Runnables queued via View.post() before attach execute after the View is attached but before the first traversal completes. They are posted to the Handler during dispatchAttachedToWindow(), but the first traversal hasn't happened yet (it was scheduled via Choreographer). So view.post { view.width } in onCreate() will return 0 because layout hasn't run yet. Use view.doOnLayout { } instead.

Multiple Windows in One Activity

An Activity can have multiple windows, each with its own ViewRootImpl:

// Each of these creates a separate ViewRootImpl:
val dialog = AlertDialog.Builder(this).setTitle("Test").create()
dialog.show()  // → WindowManager.addView() → new ViewRootImpl

val popup = PopupWindow(contentView, width, height)
popup.showAtLocation(anchor, gravity, x, y)  // → new ViewRootImpl

// Toast (before API 30) also created a separate window:
Toast.makeText(this, "Hello", Toast.LENGTH_SHORT).show()  // → new ViewRootImpl

// You can inspect all windows:
// WindowManagerGlobal.getInstance().mViews  — all root Views
// WindowManagerGlobal.getInstance().mRoots  — all ViewRootImpls

DecorView vs Window.getDecorView()

Window.getDecorView() is not a simple getter — it triggers installDecor() if the DecorView hasn't been created yet:

// PhoneWindow.getDecorView()
public final View getDecorView() {
    if (mDecor == null || mForceDecorInstall) {
        installDecor()  // creates DecorView + inflates system layout
    }
    return mDecor
}

// This means calling getDecorView() before setContentView() will:
// 1. Create the DecorView with the DEFAULT system layout
// 2. Your subsequent setContentView() will inflate into the already-created content frame
// 3. But if you called requestWindowFeature() AFTER getDecorView(), it's too late —
//    the system layout was already chosen

ViewRootImpl and Configuration Changes

During a configuration change that triggers Activity recreation:

  1. Old Activity's onDestroy() → WindowManager.removeView(decorView) → ViewRootImpl.die()
  2. ViewRootImpl.die() → doDie() → dispatchDetachedFromWindow() → mWindowSession.remove(mWindow) (WMS removes the window)
  3. New Activity created → setContentView() → new DecorView
  4. makeVisible() → new ViewRootImpl → WMS.addWindow() → new Surface
  5. The old and new windows briefly overlap — WMS handles the transition animation between them
Staff Insight — Thread Affinity Trick
The "wrong thread" check in ViewRootImpl compares against mThread, which is set in the constructor to Thread.currentThread(). This is not hardcoded to the main thread — it's whatever thread created the ViewRootImpl. In AOSP system UI code (SystemUI, Launcher), you'll occasionally see View trees created on dedicated HandlerThreads for isolation. The entire View tree then runs on that thread, including traversals and input dispatch. This is an advanced pattern used for system overlays that must not block the main thread, but it's extremely fragile — any cross-thread View access will crash.

The isLayoutRequested() Check

When requestLayout() propagates up the tree, each ViewGroup checks isLayoutRequested() to avoid redundant propagation:

// View.requestLayout()
if (mParent != null && !mParent.isLayoutRequested()) {
    mParent.requestLayout();
}

// If the parent already has PFLAG_FORCE_LAYOUT set (from a previous
// requestLayout() in the same frame), propagation stops.
// This prevents O(n²) propagation when multiple children request layout.

View IDs and Tag System

// View.findViewById() walks the tree recursively — O(n)
// For deep trees, this is expensive. Cache references in onViewCreated() or use ViewBinding.

// View tags: arbitrary key-value storage on any View
view.setTag(R.id.my_key, someObject)
val obj = view.getTag(R.id.my_key)

// Tags use a SparseArray internally — O(log n) lookup
// Common use: ViewHolder pattern (pre-RecyclerView), avoiding redundant work in adapters
Danger — Leaking via View Tags
Storing Activity or Fragment references in View tags creates a leak path. The View may outlive the Activity (e.g., if it's in a static cache or a RecyclerView.RecycledViewPool shared across Activities). Always store lightweight data in tags, never lifecycle-scoped objects. The same applies to View.setTag(Object) — the tag holds a strong reference.
