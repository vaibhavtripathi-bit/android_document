05 — Touch & Input System

MotionEvent internals, the dispatch chain algorithm, interception, touch targets, gesture detection, nested scrolling, CoordinatorLayout behaviors, and multi-touch — the complete input pipeline from finger to callback

1. MotionEvent — The Touch Data

Every touch, stylus stroke, mouse click, and trackball movement arrives as a MotionEvent. This object carries the full state of all active pointers at a given instant.

Action Types

ActionWhenNotes
ACTION_DOWNFirst finger touchesStarts a new gesture. Resets all state.
ACTION_MOVEAny finger movesContains positions for all active pointers
ACTION_UPLast finger liftsEnds the gesture
ACTION_CANCELGesture invalidatedParent intercepted, window lost focus, etc.
ACTION_POINTER_DOWN2nd+ finger touchesPointer index encoded in upper bits
ACTION_POINTER_UPNon-last finger liftsPointer index encoded in upper bits

Coordinate Systems

  • getX() / getY() — coordinates relative to the receiving View's top-left corner (after scroll and translation transforms)
  • getRawX() / getRawY() — coordinates relative to the screen (display origin)

When a parent dispatches to a child, it adjusts the event coordinates by the child's position and scroll offset. This is why getX() gives you local coordinates — the framework has already done the math.

getActionMasked() vs getAction()

getAction() returns the raw action integer, which for multi-touch events encodes the pointer index in the upper 8 bits. getActionMasked() strips the index, giving you the pure action constant. Always use getActionMasked() — using getAction() with a when block will silently fail to match ACTION_POINTER_DOWN/ACTION_POINTER_UP for any pointer index > 0.

Pointer Tracking

  • getPointerCount() — number of active fingers right now
  • getPointerId(pointerIndex) — stable ID for a finger across the entire gesture
  • findPointerIndex(pointerId) — reverse lookup: ID → current index
  • Pointer index can change between events (when a finger lifts, indices compact). Pointer ID is stable.

Historical Data

The input system batches touch samples between VSYNC frames. A single ACTION_MOVE event may contain multiple intermediate positions:

// Smooth drawing: process all historical points, then the current one
override fun onTouchEvent(event: MotionEvent): Boolean {
    when (event.actionMasked) {
        MotionEvent.ACTION_MOVE -> {
            for (h in 0 until event.historySize) {
                val hx = event.getHistoricalX(h)
                val hy = event.getHistoricalY(h)
                path.lineTo(hx, hy)
            }
            path.lineTo(event.x, event.y)
            invalidate()
        }
    }
    return true
}

Object Pooling

MotionEvent uses an internal object pool to avoid allocation pressure. MotionEvent.obtain() pulls from the pool; recycle() returns it. If you copy an event (e.g., for deferred processing), use MotionEvent.obtain(event) to get a pooled copy, and always recycle it when done. Forgetting to recycle causes pool exhaustion and falls back to allocation.

2. Touch Dispatch Chain — The Core Algorithm

This is the single most important algorithm in the Android View system for input handling. Every touch event walks a well-defined chain from the Activity down to the target View and back up.

┌─────────────────────────────────────────────────────────────────────┐
│                     TOUCH DISPATCH CHAIN                            │
│                                                                     │
│  Activity.dispatchTouchEvent(ev)                                    │
│    │                                                                │
│    ├─→ PhoneWindow.superDispatchTouchEvent(ev)                      │
│    │     │                                                          │
│    │     └─→ DecorView.dispatchTouchEvent(ev)                       │
│    │           │                                                    │
│    │           └─→ ViewGroup.dispatchTouchEvent(ev)  ◄── THE CORE   │
│    │                 │                                              │
│    │                 ├─ 1. onInterceptTouchEvent(ev)?               │
│    │                 │     ├─ true  → steal event, send CANCEL      │
│    │                 │     │         to child, handle self          │
│    │                 │     └─ false → continue to children          │
│    │                 │                                              │
│    │                 ├─ 2. Dispatch to child (reverse draw order)   │
│    │                 │     └─→ child.dispatchTouchEvent(ev)         │

│    │                 │           │                                  │
│    │                 │           └─ onTouchEvent(ev)?               │
│    │                 │               ├─ true  → consumed            │
│    │                 │               └─ false → not handled         │
│    │                 │                                              │
│    │                 └─ 3. No child handled → own onTouchEvent(ev)  │
│    │                                                                │
│    └─→ If nobody handled → Activity.onTouchEvent(ev)               │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘

ViewGroup.dispatchTouchEvent() — Step by Step

The AOSP implementation in ViewGroup.java (~300 lines) does the following on each event:

  1. Security check — if the event is obscured by another window and the View has filterTouchesWhenObscured, drop it
  2. On ACTION_DOWN: clear all previous touch targets and reset the disallowIntercept flag
  3. Intercept check: call onInterceptTouchEvent() (unless child called requestDisallowInterceptTouchEvent(true))
  4. If not intercepted and ACTION_DOWN: iterate children in reverse drawing order (topmost first), find the child whose bounds contain the touch point, call child.dispatchTouchEvent(). If the child returns true, add it to the touch target list.
  5. If intercepted or no touch target: call super.dispatchTouchEvent() (which calls View.dispatchTouchEvent() on itself)
  6. For non-DOWN events: dispatch directly to the existing touch target (no hit testing)

View.dispatchTouchEvent() — The Leaf

// Simplified AOSP View.dispatchTouchEvent()
fun dispatchTouchEvent(event: MotionEvent): Boolean {
    var result = false

    // 1. OnTouchListener gets first crack
    if (onTouchListener != null
        && (viewFlags and ENABLED_MASK) == ENABLED
        && onTouchListener.onTouch(this, event)) {
        result = true
    }

    // 2. onTouchEvent() only if listener didn't consume
    if (!result && onTouchEvent(event)) {
        result = true
    }

    return result
}
L6 Staff Insight
The priority order is: OnTouchListener.onTouch() → onTouchEvent() → OnClickListener (inside onTouchEvent on ACTION_UP). This means setting an OnTouchListener that returns true will completely suppress click/long-press callbacks. This is a common source of bugs when developers add touch listeners for analytics tracking and accidentally swallow all events.

The Critical ACTION_DOWN Rule

If a View's onTouchEvent() returns false for ACTION_DOWN, it will never receive any subsequent events in that gesture (MOVE, UP, etc.). The ViewGroup removes it from the touch target list. This is the #1 mistake in custom touch handling — forgetting to return true for DOWN.

3. onInterceptTouchEvent() — Parent Stealing

Only ViewGroup has onInterceptTouchEvent(). It's the mechanism by which a parent can "steal" a touch stream from its children mid-gesture.

The Interception Sequence

  1. User touches down on a child inside a ScrollView
  2. ScrollView.onInterceptTouchEvent(DOWN) returns false — let the child have it
  3. Child receives ACTION_DOWN, returns true — becomes touch target
  4. User starts dragging vertically
  5. ScrollView.onInterceptTouchEvent(MOVE) detects the drag exceeds touch slop → returns true
  6. Child receives ACTION_CANCEL — the gesture was taken away
  7. All subsequent MOVE/UP events go directly to ScrollView.onTouchEvent()
  8. onInterceptTouchEvent() is no longer called for this gesture — the ViewGroup already owns it

requestDisallowInterceptTouchEvent()

A child can call parent.requestDisallowInterceptTouchEvent(true) to prevent the parent from intercepting. This sets a flag that propagates up the entire View tree. The flag is reset on every new gesture (ACTION_DOWN clears it in the parent's dispatchTouchEvent()).

// Child ViewPager inside a ScrollView — prevent vertical scroll from stealing horizontal swipes
override fun onInterceptTouchEvent(ev: MotionEvent): Boolean {
    when (ev.actionMasked) {
        MotionEvent.ACTION_DOWN -> {
            startX = ev.x
            startY = ev.y
            parent.requestDisallowInterceptTouchEvent(true)
        }
        MotionEvent.ACTION_MOVE -> {
            val dx = abs(ev.x - startX)
            val dy = abs(ev.y - startY)
            if (dy > dx) {
                // Vertical movement — let parent scroll
                parent.requestDisallowInterceptTouchEvent(false)
            }
        }
    }
    return super.onInterceptTouchEvent(ev)
}
L6 Staff Insight
The intercept mechanism is the root cause of most touch conflict bugs in Android. The classic pattern — ViewPager inside ScrollView, RecyclerView inside ViewPager, map inside scroll — all reduce to the same problem: two ViewGroups both want to intercept the same gesture. The solution is always a combination of onInterceptTouchEvent() direction detection and requestDisallowInterceptTouchEvent(). At L6 you should be able to diagnose and fix these conflicts from first principles, not just copy StackOverflow patterns.

4. Touch Target Management

Internally, ViewGroup maintains a linked list of touch targets — children that are actively handling a touch gesture.

The TouchTarget Data Structure

// AOSP ViewGroup.java (simplified)
private static final class TouchTarget {
    var child: View
    var pointerIdBits: Int    // which pointer IDs this target owns
    var next: TouchTarget?    // linked list

    companion object {
        private val sRecycleBin: TouchTarget? = null  // object pool
        fun obtain(child: View, pointerIdBits: Int): TouchTarget { ... }
    }
    fun recycle() { ... }
}

Lifecycle of a Touch Target

EventTouch Target Action
ACTION_DOWNClear all targets. Find child under touch point. If child handles it (returns true), create a TouchTarget and add to the head of the list.
ACTION_POINTER_DOWNFind child under the new pointer. May create a new TouchTarget or add the pointer ID to an existing one.
ACTION_MOVEDispatch to each TouchTarget with only the pointer IDs it owns (using MotionEvent.split()).
ACTION_POINTER_UPRemove the pointer ID from its TouchTarget. If the target has no more pointers, remove it from the list.
ACTION_UPDispatch to the target, then clear all targets.
ACTION_CANCELDispatch CANCEL to all targets, then clear the list.

Event Splitting

Since Android 3.0, ViewGroup supports split motion events — different pointers can be dispatched to different children simultaneously. This is controlled by setMotionEventSplittingEnabled() (default: true). When splitting is enabled, the framework uses MotionEvent.split(idBits) to create a new event containing only the pointers relevant to each child.

Gotcha
If you disable event splitting (android:splitMotionEvents="false"), all pointers go to the first child that received ACTION_DOWN. This can cause confusing behavior where tapping a second button while holding the first does nothing.

5. GestureDetector and Gesture Recognition

Raw MotionEvent processing is tedious. GestureDetector encapsulates the state machine for recognizing common gestures.

Recognized Gestures

CallbackGestureTrigger Condition

onSingleTapUp()TapACTION_UP without exceeding touch slop or long press timeout
onSingleTapConfirmed()Confirmed single tapAfter double-tap timeout expires (300ms)
onDoubleTap()Double tapTwo taps within DOUBLE_TAP_TIMEOUT (300ms)

Zoomable/Pannable ImageView

class ZoomableImageView @JvmOverloads constructor(
    context: Context, attrs: AttributeSet? = null
) : AppCompatImageView(context, attrs) {

    private val matrix = Matrix()
    private var scaleFactor = 1f
    private var translateX = 0f
    private var translateY = 0f

    private val gestureDetector = GestureDetector(context,
        object : GestureDetector.SimpleOnGestureListener() {
            override fun onDown(e: MotionEvent) = true

            override fun onScroll(
                e1: MotionEvent?, e2: MotionEvent,
                distanceX: Float, distanceY: Float
            ): Boolean {
                translateX -= distanceX
                translateY -= distanceY
                updateMatrix()
                return true
            }

            override fun onDoubleTap(e: MotionEvent): Boolean {
                scaleFactor = if (scaleFactor > 1f) 1f else 2.5f
                updateMatrix()
                return true
            }

            override fun onFling(
                e1: MotionEvent?, e2: MotionEvent,
                velocityX: Float, velocityY: Float
            ): Boolean {
                // Animate fling with OverScroller
                return true
            }
        })

    private val scaleDetector = ScaleGestureDetector(context,
        object : ScaleGestureDetector.SimpleOnScaleGestureListener() {
            override fun onScale(detector: ScaleGestureDetector): Boolean {
                scaleFactor *= detector.scaleFactor
                scaleFactor = scaleFactor.coerceIn(0.5f, 5f)
                updateMatrix()
                return true
            }
        })

    override fun onTouchEvent(event: MotionEvent): Boolean {
        scaleDetector.onTouchEvent(event)
        gestureDetector.onTouchEvent(event)
        return true
    }

    private fun updateMatrix() {
        matrix.reset()
        matrix.postScale(scaleFactor, scaleFactor,
            width / 2f, height / 2f)
        matrix.postTranslate(translateX, translateY)
        imageMatrix = matrix
        scaleType = ScaleType.MATRIX
    }
}

6. VelocityTracker — Measuring Speed

VelocityTracker computes the velocity of a pointer's movement over recent touch events. It's the backbone of fling detection, scroll physics, and swipe-to-dismiss.

Usage Pattern

private var velocityTracker: VelocityTracker? = null

override fun onTouchEvent(event: MotionEvent): Boolean {
    when (event.actionMasked) {
        MotionEvent.ACTION_DOWN -> {
            velocityTracker?.recycle()
            velocityTracker = VelocityTracker.obtain()
            velocityTracker?.addMovement(event)
        }
        MotionEvent.ACTION_MOVE -> {
            velocityTracker?.addMovement(event)
        }
        MotionEvent.ACTION_UP -> {
            velocityTracker?.let { vt ->
                vt.addMovement(event)
                vt.computeCurrentVelocity(
                    1000, // units: pixels per second
                    ViewConfiguration.get(context).scaledMaximumFlingVelocity.toFloat()
                )
                val vx = vt.xVelocity
                val vy = vt.yVelocity

                val minFling = ViewConfiguration.get(context)
                    .scaledMinimumFlingVelocity
                if (abs(vy) > minFling) {
                    onFling(vy)
                }
            }
            velocityTracker?.recycle()
            velocityTracker = null
        }
        MotionEvent.ACTION_CANCEL -> {
            velocityTracker?.recycle()
            velocityTracker = null
        }
    }
    return true
}

Key Parameters

Method / ConstantValue (typical)Purpose
computeCurrentVelocity(units)1000Compute in pixels/second (1000ms units)
scaledMinimumFlingVelocity~50 dp/sBelow this, it's a drag not a fling
scaledMaximumFlingVelocity~8000 dp/sClamp velocity to prevent absurd flings

Internally, VelocityTracker uses a least-squares polynomial fit over the last ~100ms of touch samples to compute velocity. It's not a simple delta/time calculation — the polynomial fit smooths out jitter from noisy touch hardware.

7. Nested Scrolling

The classic intercept mechanism breaks down when you need cooperative scrolling — a child and parent both scrolling in the same direction, sharing the scroll delta. Nested scrolling (API 21+) solves this.

The Problem

A ScrollView containing a RecyclerView: the inner RecyclerView consumes all vertical scroll events. The outer ScrollView never sees them. onInterceptTouchEvent() can't help because you don't want the parent to steal the gesture — you want them to share it.

The Protocol

  CHILD (NestedScrollingChild)              PARENT (NestedScrollingParent)
  ─────────────────────────                 ──────────────────────────────
  1. startNestedScroll(axes)  ──────────→   onStartNestedScroll() → true
                                            onNestedScrollAccepted()

  2. dispatchNestedPreScroll   ──────────→  onNestedPreScroll(dx, dy, consumed[])
     (dx, dy, consumed[])                    ↑ parent consumes part of scroll
                                             │ writes to consumed[]

  3. Child scrolls the remainder (dx - consumed[0], dy - consumed[1])

  4. dispatchNestedScroll      ──────────→  onNestedScroll(dxConsumed, dyConsumed,
     (consumed, unconsumed)                   dxUnconsumed, dyUnconsumed)
                                             ↑ parent scrolls any unconsumed

  5. stopNestedScroll()        ──────────→  onStopNestedScroll()

Implementations in the Framework

  • RecyclerView — implements NestedScrollingChild3 (dispatches scroll to parent)
  • NestedScrollView — implements both child and parent (replaces ScrollView for nested scenarios)
  • CoordinatorLayout — implements NestedScrollingParent3 (delegates to Behaviors)
  • SwipeRefreshLayout — implements parent (consumes overscroll at top)

Version Evolution

InterfaceAPIAddition
NestedScrollingChild21Basic nested scroll dispatch
NestedScrollingChild226Adds type parameter (touch vs fling)
NestedScrollingChild3AndroidXAdds consumed array to dispatchNestedScroll
L6 Staff Insight
The nested scrolling protocol is fundamentally different from the intercept mechanism. Intercept is adversarial — parent steals from child. Nested scrolling is cooperative — child offers scroll to parent before and after consuming it. At L6, you need to know when each is appropriate: use intercept for conflicting gestures (horizontal vs vertical), use nested scrolling for same-direction cooperative scrolling (collapsing toolbar + list). Mixing them up causes subtle bugs that are hard to diagnose.

8. CoordinatorLayout and Behaviors

CoordinatorLayout is a FrameLayout that adds two capabilities: dependency-based layout and nested scroll delegation to Behaviors.

Behavior<V> — The Extension Point

abstract class Behavior<V : View> {
    // Dependency graph
    open fun layoutDependsOn(parent: CoordinatorLayout,
        child: V, dependency: View): Boolean = false
    open fun onDependentViewChanged(parent: CoordinatorLayout,
        child: V, dependency: View): Boolean = false
    open fun onDependentViewRemoved(parent: CoordinatorLayout,
        child: V, dependency: View) {}

    // Nested scroll delegation
    open fun onStartNestedScroll(coordinatorLayout: CoordinatorLayout,
        child: V, directTargetChild: View, target: View,
        axes: Int, type: Int): Boolean = false
    open fun onNestedPreScroll(coordinatorLayout: CoordinatorLayout,
        child: V, target: View, dx: Int, dy: Int,
        consumed: IntArray, type: Int) {}
    open fun onNestedScroll(coordinatorLayout: CoordinatorLayout,
        child: V, target: View, dxConsumed: Int, dyConsumed: Int,
        dxUnconsumed: Int, dyUnconsumed: Int, type: Int,
        consumed: IntArray) {}

    // Touch interception
    open fun onInterceptTouchEvent(parent: CoordinatorLayout,
        child: V, ev: MotionEvent): Boolean = false
    open fun onTouchEvent(parent: CoordinatorLayout,
        child: V, ev: MotionEvent): Boolean = false

    // Layout
    open fun onMeasureChild(...): Boolean = false
    open fun onLayoutChild(...): Boolean = false
}

How CoordinatorLayout Routes Nested Scroll

  1. A scrollable child (e.g., RecyclerView) starts a nested scroll
  2. CoordinatorLayout.onStartNestedScroll() iterates all children, calling each child's Behavior.onStartNestedScroll()
  3. Any Behavior that returns true will receive subsequent onNestedPreScroll() and onNestedScroll() calls
  4. Multiple Behaviors can consume scroll simultaneously — the consumed array accumulates

Built-in Behaviors

BehaviorWhat It Does
AppBarLayout.ScrollingViewBehaviorPositions the scrolling view below the AppBarLayout and triggers collapse/expand
AppBarLayout.BehaviorHandles the actual collapse/expand animation of the AppBarLayout in response to nested scroll
FloatingActionButton.BehaviorHides FAB on scroll down, shows on scroll up; also dodges Snackbar
BottomSheetBehaviorDraggable bottom sheet with peek/expanded/hidden states
Key Insight
CoordinatorLayout's Behavior system is essentially a dependency graph for layout and scroll. Instead of building a monolithic custom ViewGroup that knows about all its children, you attach modular Behaviors to individual children. This is the Composition pattern applied to layout — and it's why Material Design's scrolling effects (collapsing toolbar, FAB hide, bottom sheet) all work together without custom ViewGroup code.

Multi-Touch Drawing Canvas

class MultiTouchDrawView(context: Context, attrs: AttributeSet?) :
    View(context, attrs) {

    private data class Stroke(
        val path: Path = Path(),
        val paint: Paint = Paint().apply {
            style = Paint.Style.STROKE
            strokeWidth = 6f
            isAntiAlias = true
            strokeCap = Paint.Cap.ROUND
            color = COLORS[colorIndex++ % COLORS.size]
        }
    )

    private val activeStrokes = SparseArray<Stroke>() // pointerId → Stroke
    private val completedStrokes = mutableListOf<Stroke>()

    override fun onTouchEvent(event: MotionEvent): Boolean {
        when (event.actionMasked) {
            MotionEvent.ACTION_DOWN,
            MotionEvent.ACTION_POINTER_DOWN -> {
                val idx = event.actionIndex
                val id = event.getPointerId(idx)
                val stroke = Stroke()
                stroke.path.moveTo(event.getX(idx), event.getY(idx))
                activeStrokes.put(id, stroke)
            }
            MotionEvent.ACTION_MOVE -> {
                for (i in 0 until event.pointerCount) {
                    val id = event.getPointerId(i)
                    activeStrokes[id]?.path?.lineTo(
                        event.getX(i), event.getY(i)
                    )
                }
            }
            MotionEvent.ACTION_UP,
            MotionEvent.ACTION_POINTER_UP -> {
                val idx = event.actionIndex
                val id = event.getPointerId(idx)
                activeStrokes[id]?.let { completedStrokes.add(it) }
                activeStrokes.remove(id)
            }
            MotionEvent.ACTION_CANCEL -> {
                for (i in 0 until activeStrokes.size()) {
                    completedStrokes.add(activeStrokes.valueAt(i))
                }
                activeStrokes.clear()
            }
        }
        invalidate()
        return true
    }

    override fun onDraw(canvas: Canvas) {
        completedStrokes.forEach { canvas.drawPath(it.path, it.paint) }
        for (i in 0 until activeStrokes.size()) {
            val s = activeStrokes.valueAt(i)
            canvas.drawPath(s.path, s.paint)
        }
    }

    companion object {
        private val COLORS = intArrayOf(
            0xFF2563EB.toInt(), 0xFF7C3AED.toInt(),
            0xFF16803C.toInt(), 0xFFDC2626.toInt(),
            0xFFC2410C.toInt()
        )
        private var colorIndex = 0
    }
}

10. Staff-Level Corner Cases

ACTION_CANCEL — When and Why

ACTION_CANCEL is sent in these situations:

  • A parent ViewGroup intercepts the gesture mid-stream (the child gets CANCEL)
  • The window loses focus (e.g., a dialog appears, notification shade pulled down)
  • The gesture is invalidated by the system (e.g., palm rejection on stylus devices)
  • The View is removed from the hierarchy during an active gesture

You must handle ACTION_CANCEL. Failing to do so causes stuck states — pressed highlights that never clear, drag operations that never end, velocity trackers that never reset.

Touch Slop

val touchSlop = ViewConfiguration.get(context).scaledTouchSlop // ~8dp

override fun onTouchEvent(event: MotionEvent): Boolean {
    when (event.actionMasked) {
        MotionEvent.ACTION_DOWN -> {
            downX = event.x; downY = event.y
            isDragging = false
        }
        MotionEvent.ACTION_MOVE -> {
            if (!isDragging) {
                val dx = event.x - downX
                val dy = event.y - downY
                if (dx * dx + dy * dy > touchSlop * touchSlop) {
                    isDragging = true
                    // Adjust start point to avoid the initial "jump"
                    downX = event.x; downY = event.y
                }
            }
            if (isDragging) { /* handle drag */ }
        }
    }
    return true
}

setClickable(true) — The Hidden Touch Target

Calling setClickable(true), setLongClickable(true), or setContextClickable(true) makes the View's onTouchEvent() return true for ACTION_DOWN — even without any click listener. Setting an OnClickListener implicitly calls setClickable(true). This means:

  • A TextView with no listener: onTouchEvent() returns false for DOWN → events pass through to parent
  • A TextView with setOnClickListener {}: onTouchEvent() returns true for DOWN → it becomes a touch target and consumes all events
Common Bug
Adding a click listener to a View inside a scrollable container can break scrolling. The View now consumes ACTION_DOWN, becomes the touch target, and the parent's onInterceptTouchEvent() must detect a drag before it can steal the gesture. If the parent's intercept logic is broken (or missing), the scroll stops working.

setEnabled(false) — Still Consumes Events

A disabled View (isEnabled = false) still receives touch events. Its onTouchEvent() still returns true if it's clickable — it just doesn't fire click callbacks. The View silently swallows the events. To make a View truly transparent to touch, set isClickable = false and isLongClickable = false, or override onTouchEvent() to return false.

TouchDelegate — Expanding Touch Targets

// Expand a small button's touch area by 48dp on all sides
val parent = button.parent as View
parent.post {
    val rect = Rect()
    button.getHitRect(rect)
    val extra = (48 * resources.displayMetrics.density).toInt()
    rect.inset(-extra, -extra)
    parent.touchDelegate = TouchDelegate(rect, button)
}

TouchDelegate is limited to one delegate per parent View. For multiple expanded targets, you need a custom TouchDelegate subclass that manages a list. Material Design guidelines require 48dp minimum touch targets — TouchDelegate lets you meet this without inflating the View's actual bounds.

WindowInsets and Touch

System bars (status bar, navigation bar) consume touch events in their area. When using edge-to-edge layouts with WindowCompat.setDecorFitsSystemWindows(window, false), your Views may extend behind system bars but touch events in those regions are consumed by the system. Apply WindowInsetsCompat padding to keep interactive elements within the safe area.

L6 Staff Insight
The most subtle touch bug at scale is the phantom touch target: a View that's invisible (GONE or zero alpha) but still clickable, consuming events and preventing the View behind it from receiving touches. This happens when you animate a View to alpha 0 but forget to set isClickable = false, or when a View.GONE View is still in the touch target list because it was the target when the gesture started (the framework dispatches to the target, not by hit-testing on every MOVE). The fix: always clear click listeners and set isClickable = false when hiding interactive Views, and handle ACTION_CANCEL to clean up gesture state.
