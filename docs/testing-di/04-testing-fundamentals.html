04 — Testing Fundamentals

The test pyramid, source sets, frameworks, test doubles taxonomy, MockK vs Mockito, fakes vs mocks, and designing for testability at staff level

1. The Test Pyramid for Android

The test pyramid is not a suggestion — it's an economic model. Each layer trades off speed, confidence, and maintenance cost. At L6, your test strategy is a design decision that shapes the entire architecture.

                    The Android Test Pyramid

                           ╱╲
                          ╱  ╲          UI / E2E Tests (~10%)
                         ╱    ╲         Espresso, Compose Testing
                        ╱ SLOW ╲        Real device / emulator
                       ╱ BRITTLE╲       Minutes to run
                      ╱──────────╲
                     ╱            ╲     Integration Tests (~20%)
                    ╱  MEDIUM COST ╲    Robolectric, instrumented
                   ╱   MEDIUM SPEED ╲   Component interactions
                  ╱────────────────────╲
                 ╱                      ╲   Unit Tests (~70%)
                ╱     FAST · CHEAP ·     ╲  JVM-only, no framework
               ╱     HIGH CONFIDENCE      ╲ Business logic, mappers
              ╱────────────────────────────╲

    Speed:    ██████████ ← Unit    Integration →  ████     UI → ██
    Cost:     ██         ← Unit    Integration →  ████     UI → ██████████
    Flakiness:██         ← Unit    Integration →  ████     UI → ████████
LayerTarget %Runs OnSpeedWhat It Validates
Unit~70%JVM (local)ms per testBusiness logic, mappers, state machines, ViewModels with fakes
Integration~20%Robolectric / devicesecondsComponent wiring, Room DAOs, Navigation graphs, DI modules
UI / E2E~10%Device / emulatorseconds–minutesCritical user flows, accessibility, visual regression
Staff-Level Insight — The Pyramid Is a Design Decision

At L6, you don't just write tests — you design the architecture so the pyramid is achievable. If your codebase can't reach 70% unit tests, the problem isn't testing — it's coupling. A ViewModel that directly calls Room or Retrofit forces you into integration tests. A ViewModel that depends on a Repository interface lets you unit test with a fake. The test pyramid is a consequence of good architecture, not something you bolt on afterwards.

2. Test Source Sets in Android

Android projects have three source sets for test code, each with different capabilities and trade-offs.

Source SetDirectoryRuns OnUse For
test/src/test/java/Local JVMUnit tests, ViewModel tests, business logic, mappers
androidTest/src/androidTest/java/Device / emulatorUI tests, instrumented integration tests, Room DAO tests
testFixtures/src/testFixtures/java/Shared across bothFakes, builders, test data factories

Gradle Dependency Configuration

dependencies {
    // Local JVM tests (src/test/)
    testImplementation("junit:junit:4.13.2")
    testImplementation("com.google.truth:truth:1.4.2")
    testImplementation("io.mockk:mockk:1.13.10")
    testImplementation("org.jetbrains.kotlinx:kotlinx-coroutines-test:1.8.1")

    // Instrumented tests (src/androidTest/)
    androidTestImplementation("androidx.test.ext:junit:1.1.5")
    androidTestImplementation("androidx.test.espresso:espresso-core:3.5.1")
    androidTestImplementation("androidx.compose.ui:ui-test-junit4")

    // Shared test utilities (src/testFixtures/)
    testFixturesImplementation("org.jetbrains.kotlinx:kotlinx-coroutines-core:1.8.1")
}

When to Put Tests Where

  • test/ — anything that doesn't need Context, Android framework classes, or a real database. This is your workhorse: ViewModels, UseCases, mappers, validators, state machines.
  • androidTest/ — anything that needs a real Android environment: Espresso UI tests, Room migration tests, Compose UI tests, WorkManager integration tests.
  • testFixtures/ — fakes, builders, and test data that both test/ and androidTest/ need. Requires java-test-fixtures plugin. This eliminates duplication of test doubles across source sets.
Common Mistake

Putting ViewModel tests in androidTest/ because the ViewModel uses SavedStateHandle. You don't need a device for this — SavedStateHandle can be constructed directly in JVM tests: SavedStateHandle(mapOf("key" to "value")). Every test you move from androidTest/ to test/ saves 10–30 seconds of execution time.

3. Testing Frameworks — Evolution and Comparison

🔄 API Evolution — JUnit on Android (2006–2024)

JUnit 4 (2006): The Android default for nearly two decades. @Rule, @Before, @After, @Test. Limitations: no parameterized tests without custom runners, no nested test classes, no lifecycle extensions. Android's entire testing ecosystem — ActivityScenarioRule, HiltAndroidRule, InstantTaskExecutorRule — is built on JUnit 4 @Rule.

JUnit 5 (2017): Modern test framework with @BeforeEach/@AfterEach, @Nested classes, @ParameterizedTest, @ExtendWith for composable extensions. But Android support lagged — required the third-party android-junit5 Gradle plugin. Instrumented test support was experimental for years.

Current state (2024+): JUnit 4 remains the default in Android Studio templates. JUnit 5 is gaining adoption for local JVM tests, but most Android testing libraries (Hilt testing, Robolectric, Compose testing) still assume JUnit 4 runners. Pragmatic choice: JUnit 4 for androidTest/, JUnit 5 for test/ if your team wants modern features.

JUnit 4 — The Android Standard

class UserRepositoryTest {

    @get:Rule
    val instantExecutor = InstantTaskExecutorRule()

    private lateinit var repository: UserRepository
    private lateinit var fakeDao: FakeUserDao

    @Before
    fun setup() {
        fakeDao = FakeUserDao()
        repository = UserRepository(fakeDao)
    }

    @After
    fun tearDown() {
        // cleanup if needed
    }

    @Test
    fun `getUser returns cached user when available`() {
        fakeDao.insertUser(User(id = 1, name = "Alice"))
        val result = repository.getUser(1)
        assertThat(result).isEqualTo(User(id = 1, name = "Alice"))
    }
}

JUnit 5 — Modern Features

class UserRepositoryTest {

    private lateinit var repository: UserRepository
    private lateinit var fakeDao: FakeUserDao

    @BeforeEach
    fun setup() {
        fakeDao = FakeUserDao()
        repository = UserRepository(fakeDao)
    }

    @Nested
    inner class GetUser {
        @Test
        fun `returns cached user when available`() {
            fakeDao.insertUser(User(id = 1, name = "Alice"))
            val result = repository.getUser(1)
            assertThat(result).isEqualTo(User(id = 1, name = "Alice"))
        }

        @Test
        fun `returns null when user not found`() {
            val result = repository.getUser(999)
            assertThat(result).isNull()
        }
    }

    @ParameterizedTest
    @ValueSource(strings = ["", " ", "  "])
    fun `rejects blank usernames`(name: String) {
        val result = repository.validateUsername(name)
        assertThat(result).isFalse()
    }
}

Truth — Google's Recommendation

// Basic assertions
assertThat(user.name).isEqualTo("Alice")
assertThat(users).hasSize(3)

// Collection assertions — readable and precise
assertThat(users.map { it.name })
    .containsExactly("Alice", "Bob", "Charlie")
    .inOrder()

    repository.getUser(-1)
}

Recommendation: Use Truth for Android projects (Google ecosystem, first-class Android support). Use kotlin.test if you're building KMP libraries. Avoid mixing assertion libraries within a project — pick one and standardize.

5. Test Doubles — The Complete Taxonomy

Most developers use "mock" as a catch-all term. At L6, precision matters — each test double type serves a different purpose and has different trade-offs.

                        Test Double Taxonomy

    ┌──────────┐   Simplest — passed but never actually used
    │  Dummy   │   e.g., NoOpLogger(), object : Callback {}
    └──────────┘
    ┌──────────┐   Returns predefined values, no verification
    │  Stub    │   e.g., returns hardcoded user from getUser()
    └──────────┘
    ┌──────────┐   Working implementation with shortcuts
    │  Fake    │   e.g., in-memory HashMap instead of Room DB
    └──────────┘
    ┌──────────┐   Records interactions, verifies method calls
    │  Mock    │   e.g., verify(analytics).trackEvent("login")
    └──────────┘
    ┌──────────┐   Wraps real object, records calls, delegates
    │  Spy     │   e.g., spyk(realRepo) — real logic + recording
    └──────────┘

Dummy — Satisfies a Parameter

// A dummy is passed to satisfy a constructor but never called
val dummyLogger = object : Logger {
    override fun log(message: String) { /* never called in this test */ }
    override fun error(message: String, t: Throwable) { /* never called */ }
}

// Used when testing something that requires a Logger but the test
// doesn't exercise any logging paths
val processor = DataProcessor(repository = fakeRepo, logger = dummyLogger)

Stub — Returns Predefined Values

// A stub returns canned responses — no behavior, no verification
class StubUserApi : UserApi {
    override suspend fun getUser(id: Int): UserDto =
        UserDto(id = 1, name = "Stub User", email = "stub@test.com")

    override suspend fun getUsers(): List<UserDto> =
        listOf(UserDto(id = 1, name = "Stub User", email = "stub@test.com"))
}

Fake — Working Implementation with Shortcuts

// A fake has real behavior but takes shortcuts (in-memory instead of DB)
class FakeUserRepository : UserRepository {
    private val users = mutableListOf<User>()
    private var shouldFail = false

    fun setShouldFail(fail: Boolean) { shouldFail = fail }

    override suspend fun getUser(id: Int): User? {
        if (shouldFail) throw IOException("Fake network error")
        return users.find { it.id == id }
    }

    override suspend fun saveUser(user: User) {
        if (shouldFail) throw IOException("Fake network error")
        users.removeAll { it.id == user.id }
        users.add(user)
    }

    override suspend fun getAllUsers(): List<User> = users.toList()

    override suspend fun deleteUser(id: Int) {
        users.removeAll { it.id == id }
    }
}

Mock — Records and Verifies Interactions

// MockK example — records calls, verifies interactions
@Test
fun `login tracks analytics event`() {
    val analytics = mockk<Analytics>(relaxed = true)
    val authService = mockk<AuthService>()
    coEvery { authService.login("user", "pass") } returns AuthResult.Success

    val viewModel = LoginViewModel(authService, analytics)
    viewModel.login("user", "pass")

    verify { analytics.trackEvent("login_success") }
    verify(exactly = 0) { analytics.trackEvent("login_failure") }
}

Spy — Wraps a Real Object

// A spy delegates to the real implementation but records calls
@Test
fun `caching repository delegates to real repo on cache miss`() {
    val realRepo = InMemoryUserRepository()
    val spiedRepo = spyk(realRepo)

    val user = spiedRepo.getUser(1)

    // Verify the real method was called
    verify(exactly = 1) { spiedRepo.getUser(1) }
    // The real implementation ran — we get real behavior + call recording
}

6. MockK vs Mockito-Kotlin — Deep Comparison

FeatureMockKMockito-Kotlin
Kotlin-firstYes — built for KotlinWrapper around Java Mockito
Coroutine supportcoEvery, coVerifywhenever + runTest
Final class mockingYes (default — Kotlin classes are final)Needs mock-maker-inline artifact
Extension function mockingYes (mockkStatic)No
Object / companion mockingYes (mockkObject)No
Relaxed mocksrelaxed = true (returns defaults)No equivalent
DSL qualityExcellent — every { x.foo() } returns yGood — whenever(x.foo()).thenReturn(y)
PerformanceSlightly slower (byte-code instrumentation)Faster (proxy-based)
MultiplatformNo (JVM only)No (JVM only)
Inline class supportYes (since 1.13)Partial

MockK in Practice

// MockK — Kotlin-idiomatic DSL
val repo = mockk<UserRepository>()

// Stubbing
every { repo.getUser(1) } returns User(1, "Alice")
every { repo.getUser(any()) } returns null
coEvery { repo.saveUser(any()) } just Runs

// Verification
verify(exactly = 1) { repo.getUser(1) }
coVerify { repo.saveUser(match { it.name == "Alice" }) }

// Mocking Kotlin objects and companions
mockkObject(NetworkConfig)
every { NetworkConfig.baseUrl } returns "https://test.api.com"

// Mocking extension functions
mockkStatic("com.example.ExtensionsKt")
every { any<String>().toSlug() } returns "test-slug"

Mockito-Kotlin in Practice

// Mockito-Kotlin — familiar to Java developers
val repo = mock<UserRepository>()

// Stubbing
whenever(repo.getUser(1)).thenReturn(User(1, "Alice"))
whenever(repo.getUser(any())).thenReturn(null)

// Verification
verify(repo, times(1)).getUser(1)
verify(repo, never()).deleteUser(any())

// Argument captor
val captor = argumentCaptor<User>()
verify(repo).saveUser(captor.capture())
assertThat(captor.firstValue.name).isEqualTo("Alice")

When to choose MockK: Kotlin-heavy codebase, need to mock objects/companions/extension functions, want a Kotlin-idiomatic DSL.

When to choose Mockito: Java interop, existing Mockito codebase, performance-critical test suites with thousands of tests.

7. Fakes vs Mocks — The Staff-Level Debate

Staff-Level Insight — Google's Internal Guidance: Prefer Fakes

Google's internal testing best practices document explicitly recommends fakes over mocks. The reasoning: mocks verify implementation details (which methods were called, in what order), while fakes verify behavior (given this input, what's the output?). When you refactor the internals of a class, mock-based tests break even though behavior is unchanged. Fake-based tests survive refactoring because they test the contract, not the wiring. At Google scale, mock-heavy test suites become a maintenance burden that slows down refactoring — the exact opposite of what tests should enable.

The Comparison — Same Test, Two Approaches

// ═══════════════════════════════════════════════════════
// Approach 1: Mock-based — tests implementation details
// ═══════════════════════════════════════════════════════
@Test
fun `refreshUsers fetches from API and saves to DB`() {
    val api = mockk<UserApi>()
    val dao = mockk<UserDao>(relaxed = true)
    coEvery { api.getUsers() } returns listOf(UserDto(1, "Alice", "a@b.com"))

    val repo = UserRepositoryImpl(api, dao)
    repo.refreshUsers()

    // Verifying HOW it works — brittle
    coVerify(exactly = 1) { api.getUsers() }
    coVerify { dao.deleteAll() }           // what if we switch to upsert?
    coVerify { dao.insertAll(any()) }      // breaks if we batch differently
}

// ═══════════════════════════════════════════════════════
// Approach 2: Fake-based — tests behavior / contract
// ═══════════════════════════════════════════════════════
@Test
fun `refreshUsers makes remote users available locally`() {
    val fakeApi = FakeUserApi(users = listOf(UserDto(1, "Alice", "a@b.com")))
    val fakeDao = FakeUserDao()

    val repo = UserRepositoryImpl(fakeApi, fakeDao)
    repo.refreshUsers()

    // Verifying WHAT it does — resilient
    val users = fakeDao.getAllUsers()
    assertThat(users).hasSize(1)
    assertThat(users[0].name).isEqualTo("Alice")
    // Survives refactoring: upsert vs delete+insert, batching changes, etc.
}

When Mocks Are Still Appropriate

  • Side effects that matter: verifying analytics events were tracked, logs were written, notifications were sent
  • Interaction order matters: verifying a cache is checked before a network call
  • External systems you don't control: verifying the correct HTTP method/headers/body were sent to a third-party API
  • Cost of building a fake is too high: complex systems where a faithful fake would be hundreds of lines
🔄 API Evolution — Test Doubles in the Android Ecosystem

Early Android (2010–2015): Mockito was the only game in town. Everything was mocked — repositories, DAOs, even Android framework classes. Tests were brittle and broke on every refactor.

Architecture Components era (2017–2020): Google introduced Room with an in-memory database option, effectively providing a framework-level fake. The FakeRepository pattern emerged in official samples. Testing guidance shifted toward fakes for data layers.

Modern era (2021+): MockK became the Kotlin-first alternative. Google's Now in Android sample uses fakes exclusively for repositories. The community consensus solidified: fakes for data layer, mocks only for verifying side effects. Compose testing further reinforced this — ComposeTestRule works with real composables, not mocks.

8. Testing Architecture — Designing for Testability

Testability is not something you add after writing code — it's a property of the architecture. Every decision that improves testability also improves modularity, readability, and maintainability.

Principle 1: Depend on Interfaces, Not Implementations

// ✗ Hard to test — ViewModel creates its own dependency
class UserViewModel : ViewModel() {
    private val repo = UserRepositoryImpl(
        RetrofitClient.userApi,
        AppDatabase.getInstance().userDao()
    )
}

// ✓ Testable — dependency is injected via constructor
class UserViewModel(
    private val repo: UserRepository  // interface, not impl
) : ViewModel() {
    // In tests: pass FakeUserRepository
    // In production: Hilt provides UserRepositoryImpl
}

Principle 2: Inject Dispatchers

// ✗ Hardcoded dispatcher — can't control timing in tests
class UserRepository(private val api: UserApi) {
    suspend fun getUser(id: Int): User = withContext(Dispatchers.IO) {
        api.getUser(id).toDomain()
    }
}

// ✓ Injected dispatcher — tests use TestDispatcher
class UserRepository(
    private val api: UserApi,
    private val ioDispatcher: CoroutineDispatcher = Dispatchers.IO
) {
    suspend fun getUser(id: Int): User = withContext(ioDispatcher) {
        api.getUser(id).toDomain()
    }
}

// In tests:
val testDispatcher = StandardTestDispatcher()
val repo = UserRepository(fakeApi, testDispatcher)

Principle 3: The Repository Pattern Enables the Pyramid

// Interface — the contract
interface UserRepository {
    suspend fun getUser(id: Int): User?
    suspend fun saveUser(user: User)
    fun observeUsers(): Flow<List<User>>
}

// Production implementation
class UserRepositoryImpl(
    private val api: UserApi,
    private val dao: UserDao,
    private val ioDispatcher: CoroutineDispatcher = Dispatchers.IO
) : UserRepository {
    override suspend fun getUser(id: Int): User? = withContext(ioDispatcher) {
        dao.getUser(id) ?: api.getUser(id).toDomain().also { dao.insert(it) }
    }
    override suspend fun saveUser(user: User) = withContext(ioDispatcher) {
        dao.insert(user)
        api.updateUser(user.toDto())
    }
    override fun observeUsers(): Flow<List<User>> = dao.observeAll()
}

// Fake for tests — lives in testFixtures/
class FakeUserRepository : UserRepository {
    private val users = MutableStateFlow<List<User>>(emptyList())
    var failWith: Throwable? = null

    override suspend fun getUser(id: Int): User? {
        failWith?.let { throw it }
        return users.value.find { it.id == id }
    }
    override suspend fun saveUser(user: User) {
        failWith?.let { throw it }
        users.update { list -> list.filter { it.id != user.id } + user }
    }
    override fun observeUsers(): Flow<List<User>> = users

    // Test helpers
    fun addUser(user: User) { users.update { it + user } }
    fun clear() { users.value = emptyList() }
}

Principle 4: ViewModel as the Testability Boundary

// ViewModel is the ideal unit test target:
// - Takes interfaces as constructor params → inject fakes
// - Exposes StateFlow → assert on state emissions
// - Contains business logic → high value per test

class UserListViewModel(
    private val repo: UserRepository,
    private val analytics: Analytics
) : ViewModel() {

    private val _uiState = MutableStateFlow<UiState>(UiState.Loading)
    val uiState: StateFlow<UiState> = _uiState.asStateFlow()

    fun loadUsers() {
        viewModelScope.launch {
            _uiState.value = UiState.Loading
            try {
                val users = repo.getAllUsers()
                _uiState.value = UiState.Success(users)
            } catch (e: Exception) {
                _uiState.value = UiState.Error(e.message ?: "Unknown error")
            }
        }
    }
}

// Test — fast, deterministic, no Android framework
@Test
fun `loadUsers emits Success with users`() = runTest {
    val fakeRepo = FakeUserRepository()
    fakeRepo.addUser(User(1, "Alice"))
    val viewModel = UserListViewModel(fakeRepo, mockk(relaxed = true))

    viewModel.loadUsers()
    advanceUntilIdle()

    assertThat(viewModel.uiState.value)
        .isEqualTo(UiState.Success(listOf(User(1, "Alice"))))
}
Staff-Level Insight — The Two-Boundary Testing Strategy

At L6, think of your app as having two testability boundaries: (1) ViewModel ↔ Data Layer — test ViewModels with fake repositories. This validates business logic, state management, and error handling without any UI framework. (2) UI ↔ ViewModel — test Compose/Views with a fake ViewModel (or a real ViewModel with fake repos). This validates that the UI correctly renders state and dispatches events. Together, these two boundaries give you near-complete coverage with mostly fast JVM tests. The only thing left for E2E tests is verifying that DI wires everything correctly and that navigation flows work end-to-end.

@Test fun getUser_whenUserExists_returnsUser() { ... }
@Test fun getUser_whenUserNotFound_returnsNull() { ... }

@Test fun `should throw IOException when network fails`() { ... }

// Pick ONE convention per project and enforce it.

Test Data Builders

// Builder pattern for test data — avoids brittle constructors
fun buildUser(
    id: Int = 1,
    name: String = "Test User",
    email: String = "test@example.com",

// Usage — only specify what matters for this test

Flaky Tests Are a Staff-Level Problem

At Google scale, a test suite with 0.1% flake rate and 10,000 tests means ~10 spurious failures per CI run. This erodes trust in the test suite and leads to developers ignoring failures. Staff engineers own the reliability of the test infrastructure, not just individual test correctness. Strategies: quarantine flaky tests, require flake-rate SLOs, invest in deterministic test infrastructure (hermetic builds, fixed-seed randomness, controlled clocks).
