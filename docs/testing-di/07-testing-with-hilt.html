07 — Testing with Hilt

Replacing bindings, @UninstallModules vs @BindValue vs @TestInstallIn, testing ViewModels / Fragments / Compose with Hilt, E2E patterns, and CI integration

1. Why Hilt + Testing Matters

Dependency injection exists to make testing easy. If your DI framework makes testing harder, something is fundamentally wrong with how you're using it. Hilt's testing story is one of its strongest selling points over raw Dagger — it provides first-class APIs for replacing production bindings with test doubles at the component level.

Hilt's testing philosophy rests on a single principle: the production object graph is the default, and tests surgically replace only what they need. You don't rebuild the entire graph for every test — you start with the real graph and swap out specific nodes.

Two Core Approaches

ApproachAnnotationScopeUse When
Replace entire module@UninstallModulesPer-test classReplacing all bindings from a module (e.g., all network bindings)
Replace single binding@BindValuePer-test classSurgical replacement of one or two dependencies
Replace globally@TestInstallInAll tests in moduleAlways using a fake database or mock server across all tests

Custom Test Runner

Hilt needs to replace the Application class with HiltTestApplication during tests. This requires a custom AndroidJUnitRunner.

class HiltTestRunner : AndroidJUnitRunner() {

        className: String?,

Anatomy of a Hilt Test Class

@HiltAndroidTest                          // 1. Marks this class for Hilt injection
@UninstallModules(NetworkModule::class)   // 2. (Optional) Remove production modules
class UserListScreenTest {

    @get:Rule(order = 0)
    val hiltRule = HiltAndroidRule(this)   // 3. Manages Hilt component lifecycle

    @get:Rule(order = 1)
    val composeRule = createAndroidComposeRule<HiltTestActivity>()

    @BindValue                             // 4. (Optional) Replace individual bindings
    @JvmField
    val repo: UserRepository = FakeUserRepository()

    @Inject
    lateinit var analytics: Analytics      // 5. Can inject real bindings too

    @Before
    fun setup() {
        hiltRule.inject()                  // 6. MUST call — triggers field injection
    }

    @Test
    fun showsUserList() { /* ... */ }
}
Critical — Rule Ordering

HiltAndroidRule must execute before any rule that needs the Hilt component (like ActivityScenarioRule or createAndroidComposeRule). Use @get:Rule(order = 0) for the Hilt rule and higher order numbers for dependent rules. Getting this wrong produces cryptic IllegalStateException errors about uninitialized components.

3. Replacing Bindings — @UninstallModules

@UninstallModules removes an entire production module from the Hilt component graph for a specific test class. You then provide replacement bindings — either via a nested @Module inside the test class or via @BindValue.

// Production module being replaced
@Module
@InstallIn(SingletonComponent::class)
object NetworkModule {
    @Provides @Singleton
    fun provideOkHttp(): OkHttpClient = OkHttpClient.Builder()
        .addInterceptor(AuthInterceptor())
        .build()

    @Provides @Singleton
    fun provideRetrofit(client: OkHttpClient): Retrofit = Retrofit.Builder()
        .baseUrl("https://api.example.com/")
        .client(client)
        .addConverterFactory(GsonConverterFactory.create())
        .build()

    @Provides @Singleton
    fun provideUserApi(retrofit: Retrofit): UserApi =
        retrofit.create(UserApi::class.java)
}

// Test class — replaces the entire NetworkModule
@HiltAndroidTest
@UninstallModules(NetworkModule::class)
class UserApiIntegrationTest {

    @get:Rule(order = 0)
    val hiltRule = HiltAndroidRule(this)

    @Inject lateinit var userApi: UserApi

    private lateinit var mockWebServer: MockWebServer

    @Module
    @InstallIn(SingletonComponent::class)
    inner class TestNetworkModule {
        @Provides @Singleton
        fun provideMockWebServer(): MockWebServer = mockWebServer

        @Provides @Singleton
        fun provideUserApi(server: MockWebServer): UserApi =
            Retrofit.Builder()
                .baseUrl(server.url("/"))
                .addConverterFactory(GsonConverterFactory.create())
                .build()
                .create(UserApi::class.java)
    }

    @Before
    fun setup() {
        mockWebServer = MockWebServer()
        mockWebServer.start()
        hiltRule.inject()
    }

    @After
    fun tearDown() {
        mockWebServer.shutdown()
    }

    @Test
    fun fetchesUsersFromApi() {
        mockWebServer.enqueue(MockResponse()
            .setBody("""[{"id":1,"name":"Alice"}]""")
            .setResponseCode(200))

        val users = runBlocking { userApi.getUsers() }
        assertThat(users).hasSize(1)
        assertThat(users[0].name).isEqualTo("Alice")
    }
}
When @UninstallModules Goes Wrong

If you uninstall a module but forget to provide a replacement for every binding it contributed, you'll get a compile-time error like "UserApi cannot be provided without an @Provides-annotated method". The error message names the missing binding but doesn't tell you which @UninstallModules caused it. When debugging, check which module provided the missing type and ensure your test module covers all its bindings.

4. Replacing Bindings — @BindValue

@BindValue replaces a single binding in the Hilt component with a value from the test class. It's the surgical scalpel to @UninstallModules's sledgehammer.

@HiltAndroidTest
class UserProfileTest {

    @get:Rule(order = 0)
    val hiltRule = HiltAndroidRule(this)

    @get:Rule(order = 1)
    val composeRule = createAndroidComposeRule<HiltTestActivity>()

    // Replace ONLY the repository — everything else stays real
    @BindValue
    @JvmField
    val userRepo: UserRepository = FakeUserRepository().apply {
        addUser(User(id = 1, name = "Alice", email = "alice@test.com"))
    }

    // Can combine with qualifiers
    @BindValue
    @Named("api_base_url")
    @JvmField
    val baseUrl: String = "https://test.example.com/"

    @Before
    fun setup() {
        hiltRule.inject()
    }

    @Test
    fun displaysUserProfile() {
        composeRule.setContent {
            UserProfileScreen(userId = 1)
        }
        composeRule.onNodeWithText("Alice").assertIsDisplayed()
        composeRule.onNodeWithText("alice@test.com").assertIsDisplayed()
    }
}

@BindValue Rules

  • @JvmField is required — Hilt reads the field directly, not through a getter
  • The field type determines which binding is replaced — if the production graph binds UserRepository, your @BindValue field must be typed as UserRepository, not FakeUserRepository
  • Works with @Named, @Qualifier, and custom qualifier annotations
  • The field must be initialized before hiltRule.inject() is called — lateinit var is allowed but must be set in @Before before inject()

5. @TestInstallIn — Global Test Replacements

@TestInstallIn replaces a module for all tests in the entire test module — not just one test class. It's the right choice when you want every test to use a fake database or a mock server by default.

// src/androidTest/java/com/example/app/di/FakeRepositoryModule.kt
// This applies to ALL tests — no per-test annotation needed
@Module
@TestInstallIn(
    components = [SingletonComponent::class],
    replaces = [RepositoryModule::class]
)
object FakeRepositoryModule {

    @Provides @Singleton
    fun provideUserRepository(): UserRepository = FakeUserRepository()

    @Provides @Singleton
    fun provideOrderRepository(): OrderRepository = FakeOrderRepository()
}

Staff-Level Insight — Layered Replacement Strategy

At scale, the most maintainable pattern is a three-tier replacement strategy: (1) @TestInstallIn for infrastructure that's always fake — network layer pointing to MockWebServer, analytics pointing to a no-op, crash reporting disabled. These live in a shared di/ package in androidTest/. (2) @BindValue for per-test customization — a specific test needs a repository pre-loaded with certain data, or needs a mock to verify interactions. (3) @UninstallModules as a last resort when a test needs a radically different wiring that conflicts with the global @TestInstallIn. This layering keeps 90% of tests boilerplate-free while allowing surgical overrides when needed.

6. Testing ViewModels with Hilt

Option 1: Direct Instantiation (Preferred for Unit Tests)

ViewModels are plain Kotlin classes. For unit tests, skip Hilt entirely and construct them directly with fake dependencies. This is faster, simpler, and runs on the JVM.

// src/test/java/com/example/app/ui/UserListViewModelTest.kt
class UserListViewModelTest {

    @get:Rule
    val mainDispatcherRule = MainDispatcherRule()

    private val fakeRepo = FakeUserRepository()
    private val fakeAnalytics = FakeAnalytics()

    private lateinit var viewModel: UserListViewModel

    @Before
    fun setup() {
        viewModel = UserListViewModel(
            repository = fakeRepo,
            analytics = fakeAnalytics,
            savedStateHandle = SavedStateHandle(mapOf("userId" to 42))
        )
    }

    @Test
    fun `loading users emits Success state`() = runTest {
        fakeRepo.addUser(User(id = 1, name = "Alice"))

        viewModel.loadUsers()
        advanceUntilIdle()

        assertThat(viewModel.uiState.value)
            .isEqualTo(UiState.Success(listOf(User(id = 1, name = "Alice"))))
    }

    @Test
    fun `network error emits Error state`() = runTest {
        fakeRepo.failWith = IOException("No network")

        viewModel.loadUsers()
        advanceUntilIdle()

        val state = viewModel.uiState.value
        assertThat(state).isInstanceOf(UiState.Error::class.java)
        assertThat((state as UiState.Error).message).contains("No network")
    }
}

Option 2: Hilt-Injected ViewModel (Instrumented Tests)

When you need to test the ViewModel as Hilt wires it — verifying that @HiltViewModel injection works correctly with the real or partially-real graph.

@HiltAndroidTest
class UserListViewModelHiltTest {

    @get:Rule(order = 0)
    val hiltRule = HiltAndroidRule(this)

    @get:Rule(order = 1)
    val activityRule = ActivityScenarioRule(HiltTestActivity::class.java)

    @BindValue @JvmField
    val repo: UserRepository = FakeUserRepository().apply {
        addUser(User(id = 1, name = "Alice"))
    }

    @Before
    fun setup() {
        hiltRule.inject()
    }

    @Test
    fun viewModelReceivesInjectedDependencies() {
        activityRule.scenario.onActivity { activity ->
            val viewModel: UserListViewModel by activity.viewModels()
            // ViewModel was created by Hilt with our @BindValue repo
            runBlocking {
                viewModel.loadUsers()
                assertThat(viewModel.uiState.value)
                    .isInstanceOf(UiState.Success::class.java)
            }
        }
    }
}
Prefer Option 1

Direct instantiation (Option 1) should be your default for ViewModel tests. It runs on the JVM (milliseconds, not seconds), doesn't require an emulator, and tests the ViewModel's logic in isolation. Use Option 2 only when you specifically need to verify that Hilt's injection wiring is correct — typically in a small number of integration smoke tests, not for every ViewModel test.

7. Testing Fragments with Hilt

The Problem: launchFragmentInContainer Doesn't Work

launchFragmentInContainer from androidx.fragment:fragment-testing launches the Fragment inside a generic EmptyFragmentActivity — which is not annotated with @AndroidEntryPoint. Hilt requires the host Activity to be a Hilt-injected Activity, so Fragment injection fails.

The Solution: HiltTestActivity + Custom Launcher

// src/debug/java/com/example/app/HiltTestActivity.kt
// Lives in debug/ so it's available in androidTest but not in release
@AndroidEntryPoint
class HiltTestActivity : AppCompatActivity()

// src/debug/AndroidManifest.xml
<activity
    android:name=".HiltTestActivity"
    android:exported="false" />
// src/androidTest/java/com/example/app/launchFragmentInHiltContainer.kt
inline fun <reified T : Fragment> launchFragmentInHiltContainer(
    fragmentArgs: Bundle? = null,
    themeResId: Int = R.style.Theme_App,
    initialState: Lifecycle.State = Lifecycle.State.RESUMED,
    crossinline action: T.() -> Unit = {}
): ActivityScenario<HiltTestActivity> {
    val intent = Intent.makeMainActivity(
        ComponentName(
            ApplicationProvider.getApplicationContext(),
            HiltTestActivity::class.java
        )
    ).putExtra("androidx.fragment.app.testing.FragmentScenario.EmptyFragmentActivity.THEME_EXTRAS_BUNDLE_KEY", themeResId)

    return ActivityScenario.launch<HiltTestActivity>(intent).onActivity { activity ->
        val fragment = activity.supportFragmentManager.fragmentFactory.instantiate(
            T::class.java.classLoader!!,
            T::class.java.name
        ).apply { arguments = fragmentArgs }

        activity.supportFragmentManager.commitNow {
            add(android.R.id.content, fragment)
            setMaxLifecycle(fragment, initialState)
        }

        (fragment as T).action()
    }
}

Complete Fragment Test

@HiltAndroidTest
class UserListFragmentTest {

    @get:Rule(order = 0)

    }
}

8. Testing Compose with Hilt

@HiltAndroidTest
class UserListScreenTest {

    @get:Rule(order = 0)
    val hiltRule = HiltAndroidRule(this)

    @get:Rule(order = 1)
    val composeRule = createAndroidComposeRule<HiltTestActivity>()

    @BindValue @JvmField
    val repo: UserRepository = FakeUserRepository().apply {
        addUser(User(id = 1, name = "Alice"))
    }

    @Before
    fun setup() {
        hiltRule.inject()
    }

    @Test
    fun displaysUsersFromHiltViewModel() {
        composeRule.setContent {
            // hiltViewModel() picks up our @BindValue repo via Hilt
            val viewModel: UserListViewModel = hiltViewModel()
            UserListScreen(viewModel = viewModel)
        }

        composeRule.onNodeWithText("Alice").assertIsDisplayed()
    }

    @Test
    fun showsLoadingThenContent() {
        val delayedRepo = FakeUserRepository(simulateDelay = 500L).apply {
            addUser(User(id = 1, name = "Alice"))
        }

        composeRule.setContent {
            UserListScreen(viewModel = UserListViewModel(delayedRepo, FakeAnalytics()))
        }

        composeRule.onNodeWithTag("loading_indicator").assertIsDisplayed()
        composeRule.waitUntil(timeoutMillis = 2000) {
            composeRule.onAllNodesWithText("Alice").fetchSemanticsNodes().isNotEmpty()
        }
        composeRule.onNodeWithText("Alice").assertIsDisplayed()
    }
}

The key insight: createAndroidComposeRule<HiltTestActivity>() gives you a Hilt-aware Activity that hosts your Compose content. Any call to hiltViewModel() inside setContent resolves through the Hilt component — which includes your @BindValue replacements.

9. Testing with Fake vs Mock — Hilt Patterns

The Fake Pattern

// Fake lives in testFixtures/ — shared across test/ and androidTest/
class FakeUserRepository : UserRepository {
    private val users = mutableListOf<User>()
    var failWith: Throwable? = null
    var simulateDelay: Long = 0L

    override suspend fun getUser(id: Int): User? {
        failWith?.let { throw it }
        if (simulateDelay > 0) delay(simulateDelay)
        return users.find { it.id == id }
    }

    override suspend fun getAllUsers(): List<User> {
        failWith?.let { throw it }
        if (simulateDelay > 0) delay(simulateDelay)
        return users.toList()
    }

    override suspend fun saveUser(user: User) {
        failWith?.let { throw it }
        users.removeAll { it.id == user.id }
        users.add(user)
    }

    override fun observeUsers(): Flow<List<User>> = flow {
        emit(users.toList())
    }

    fun addUser(user: User) { users.add(user) }
    fun clear() { users.clear() }
}

// Register in a test module
@Module
@TestInstallIn(
    components = [SingletonComponent::class],
    replaces = [RepositoryModule::class]
)
object FakeRepositoryModule {
    @Provides @Singleton
    fun provideUserRepository(): UserRepository = FakeUserRepository()
}

The Mock Pattern

@HiltAndroidTest
class UserProfileTest {

    @get:Rule(order = 0)
    val hiltRule = HiltAndroidRule(this)

    // MockK mock injected via @BindValue
    @BindValue @JvmField
    val repo: UserRepository = mockk(relaxed = true)

    @BindValue @JvmField
    val analytics: Analytics = mockk(relaxed = true)

    @Before
    fun setup() {
        coEvery { repo.getUser(1) } returns User(1, "Alice", "alice@test.com")
        hiltRule.inject()
    }

    @Test
    fun tracksProfileViewEvent() {
        // ... trigger profile load ...
        verify { analytics.trackEvent("profile_viewed", mapOf("userId" to "1")) }
    }
}

When to Use Which

CriterionFakeMock
What you're testingBehavior / output correctnessInteractions / side effects
Refactoring resilienceHigh — survives internal changesLow — breaks when call patterns change

Staff-Level Insight — Fakes as a Testing Contract

At Google, teams that own an API are expected to also ship a canonical fake alongside it. If you own UserRepository, you own FakeUserRepository. The fake lives in testFixtures/ and is published as part of the module's test artifact. This means consumers of your API don't have to write their own fakes or resort to mocks — they get a tested, maintained fake that faithfully implements the contract. At L6, you should be driving this pattern across your team: every interface in the data layer ships with a fake. The fake itself should have its own tests to verify it behaves identically to the real implementation for the contract's invariants.

10. End-to-End Testing with Hilt

E2E tests use the real Activity, real Navigation, real ViewModel, and real database — but replace the network layer with a MockWebServer to control responses.

@HiltAndroidTest
@UninstallModules(NetworkModule::class)
class UserFlowE2ETest {

    @get:Rule(order = 0)
    val hiltRule = HiltAndroidRule(this)

    @get:Rule(order = 1)
    val activityRule = ActivityScenarioRule(MainActivity::class.java)

    private val mockWebServer = MockWebServer()

    @Module
    @InstallIn(SingletonComponent::class)
    inner class TestNetworkModule {
        @Provides @Singleton
        fun provideRetrofit(): Retrofit = Retrofit.Builder()
            .baseUrl(mockWebServer.url("/"))
            .addConverterFactory(GsonConverterFactory.create())
            .build()

        @Provides @Singleton
        fun provideUserApi(retrofit: Retrofit): UserApi =
            retrofit.create(UserApi::class.java)
    }

    @Before
    fun setup() {
        mockWebServer.start()
        hiltRule.inject()
    }

    @After
    fun tearDown() {
        mockWebServer.shutdown()
    }

    @Test
    fun fullUserFlow_fromLaunchToDetail() {
        // Enqueue list response
        mockWebServer.enqueue(MockResponse()
            .setBody("""[{"id":1,"name":"Alice","email":"alice@test.com"}]""")
            .setResponseCode(200))

        // Enqueue detail response
        mockWebServer.enqueue(MockResponse()
            .setBody("""{"id":1,"name":"Alice","email":"alice@test.com","bio":"Engineer"}""")
            .setResponseCode(200))

        // Verify list screen
        onView(withText("Alice")).check(matches(isDisplayed()))

        // Navigate to detail
        onView(withText("Alice")).perform(click())

        // Verify detail screen
        onView(withText("Engineer")).check(matches(isDisplayed()))

        // Verify correct API calls were made
        val listRequest = mockWebServer.takeRequest()
        assertThat(listRequest.path).isEqualTo("/users")

        val detailRequest = mockWebServer.takeRequest()
        assertThat(detailRequest.path).isEqualTo("/users/1")
    }

    @Test
    fun showsErrorScreen_whenServerReturns500() {
        mockWebServer.enqueue(MockResponse().setResponseCode(500))

        onView(withText("Something went wrong")).check(matches(isDisplayed()))
        onView(withText("Retry")).check(matches(isDisplayed()))
    }
}

Evolution — Gradle Managed Devices (AGP 7.1+)

Before GMD: CI pipelines required pre-configured emulators or Firebase Test Lab. Emulator setup was flaky, slow, and environment-dependent. Teams maintained custom scripts to boot emulators, wait for them, and tear them down.

After GMD (2022+): Gradle Managed Devices define emulators declaratively in build.gradle.kts. Gradle handles downloading system images, creating AVDs, booting, running tests, and cleanup. The aosp-atd (Automated Test Device) images are stripped-down for CI — no Play Store, no Google apps, faster boot. Combined with test sharding (numShards), a 20-minute instrumented test suite can run in 5 minutes across 4 parallel emulators. This is the modern standard for Android CI.

        viewModel.loadUsers()

The @TestInstallIn + @UninstallModules Conflict

If you have a @TestInstallIn that globally replaces NetworkModule, and a specific test class also uses @UninstallModules(NetworkModule::class), the behavior depends on whether the @TestInstallIn module is also uninstalled. If you uninstall the original module but the @TestInstallIn replacement is still installed, you may get duplicate bindings. The safest pattern: if you use @TestInstallIn globally, individual tests should use @BindValue for per-test overrides rather than @UninstallModules.
