06 — Testing Android Components

Robolectric, ActivityScenario, FragmentScenario, ViewModel testing, Room, Navigation, Compose testing, Espresso — and when to use each at L6 scale

1. Robolectric — Android Without a Device

Robolectric lets you run tests that touch Android framework APIs on the local JVM — no emulator, no device, no 30-second boot time. It's the bridge between fast unit tests and accurate Android behavior.

API Evolution — Robolectric (2010–2024)

Pre-Robolectric: All Android tests required a device or emulator. The feedback loop was measured in minutes. Teams avoided writing tests because the infrastructure was too slow.

Robolectric 1–3 (2010–2017): Introduced shadow classes — hand-written JVM reimplementations of Android framework classes. ShadowActivity, ShadowLooper, ShadowApplication simulated Android behavior. Fast, but shadows were incomplete and diverged from real framework behavior. Tests passed on Robolectric but failed on device.

Robolectric 4+ (2018–present): Pivotal rewrite. Uses real Android framework code compiled from AOSP for the JVM. Shadows still exist for hardware-dependent APIs, but the core framework (Activity lifecycle, Intent resolution, SharedPreferences, resource loading) runs real code. Much more accurate. Integrated with AndroidX Test APIs so the same test class works on both Robolectric and a real device.

Current (4.x): @Config(sdk = [34]) targets specific API levels. Qualifiers simulate device configurations. First-class Compose support via createComposeRule(). Roborazzi adds screenshot testing on JVM.

@RunWith(AndroidJUnit4::class)
@Config(sdk = [34])
class SharedPrefsManagerTest {

    private lateinit var manager: SharedPrefsManager

    @Before fun setup() {
        val context = ApplicationProvider.getApplicationContext<Application>()
        manager = SharedPrefsManager(context)
    }

    @Test fun `saves and retrieves user token`() {
        manager.saveToken("abc-123")
        assertThat(manager.getToken()).isEqualTo("abc-123")
    }

    @Test fun `clearAll removes all stored data`() {
        manager.saveToken("abc-123")
        manager.saveUserId(42)
        manager.clearAll()
        assertThat(manager.getToken()).isNull()
    }
}

2. ActivityScenario — Testing Activities

ActivityScenario is the AndroidX API for launching and controlling Activities in tests. It replaces the deprecated ActivityTestRule and works identically on Robolectric and instrumented tests.

Core API

@RunWith(AndroidJUnit4::class)
class UserProfileActivityTest {

    @Test fun `displays user name from intent`() {
        val intent = Intent(
            ApplicationProvider.getApplicationContext(),
            UserProfileActivity::class.java
        ).putExtra("USER_NAME", "Alice")

        ActivityScenario.launch<UserProfileActivity>(intent).use { scenario ->
            scenario.onActivity { activity ->
                val nameView = activity.findViewById<TextView>(R.id.tvUserName)
                assertThat(nameView.text.toString()).isEqualTo("Alice")
            }
        }
    }

    @Test fun `survives configuration change`() {
        ActivityScenario.launch(UserProfileActivity::class.java).use { scenario ->
            scenario.onActivity { it.findViewById<EditText>(R.id.etSearch).setText("query") }
            scenario.recreate()
            scenario.onActivity {
                assertThat(it.findViewById<EditText>(R.id.etSearch).text.toString())
                    .isEqualTo("query")
            }
        }
    }

    @Test fun `lifecycle state transitions`() {
        ActivityScenario.launch(UserProfileActivity::class.java).use { scenario ->
            assertThat(scenario.state).isEqualTo(Lifecycle.State.RESUMED)
            scenario.moveToState(Lifecycle.State.CREATED)
            assertThat(scenario.state).isEqualTo(Lifecycle.State.CREATED)
            scenario.moveToState(Lifecycle.State.DESTROYED)
        }
    }
}
Common Mistake — onActivity Threading

scenario.onActivity { } runs the lambda on the main thread and blocks the test thread until it completes. Never do long-running work inside it. For assertions that don't need the main thread, extract values inside onActivity and assert outside it. Also note: onActivity throws if the Activity is in DESTROYED state — always check the lifecycle state first if you've been moving it around.

3. FragmentScenario — Testing Fragments

FragmentScenario launches a Fragment in isolation, inside a bare-bones host Activity. This lets you test Fragment behavior without building the full Activity hierarchy.

@RunWith(AndroidJUnit4::class)
class UserListFragmentTest {

    @Test fun `displays users from ViewModel`() {
        val args = bundleOf("FILTER" to "active")
        launchFragmentInContainer<UserListFragment>(args).use { scenario ->
            scenario.onFragment { fragment ->
                val rv = fragment.view?.findViewById<RecyclerView>(R.id.rvUsers)
                assertThat(rv?.adapter?.itemCount).isGreaterThan(0)
            }
        }
    }

    @Test fun `survives process death`() {
        launchFragmentInContainer<UserListFragment>().use { scenario ->
            scenario.onFragment { it.viewModel.selectUser(User(1, "Alice")) }
            scenario.recreate()
            scenario.onFragment {
                assertThat(it.viewModel.selectedUser.value?.name).isEqualTo("Alice")
            }
        }
    }

    @Test fun `lifecycle control — view availability`() {
        launchFragmentInContainer<UserListFragment>().use { scenario ->
            scenario.moveToState(Lifecycle.State.CREATED)
            scenario.onFragment { assertThat(it.view).isNull() }
            scenario.moveToState(Lifecycle.State.RESUMED)
            scenario.onFragment { assertThat(it.view).isNotNull() }
        }
    }
}

FragmentScenario with Custom FragmentFactory

When Fragments have constructor dependencies (common with DI), you need a custom FragmentFactory to provide them in tests.

class TestFragmentFactory(
    private val fakeRepo: FakeUserRepository
) : FragmentFactory() {
    override fun instantiate(classLoader: ClassLoader, className: String) = when (className) {
        UserListFragment::class.java.name -> UserListFragment(fakeRepo)
        else -> super.instantiate(classLoader, className)
    }
}

@Test fun `fragment uses injected repository`() {
    val fakeRepo = FakeUserRepository().apply { addUser(User(1, "Alice")) }
    launchFragmentInContainer<UserListFragment>(factory = TestFragmentFactory(fakeRepo))
        .use { scenario ->
            scenario.onFragment { assertThat(it.adapter.itemCount).isEqualTo(1) }
        }
}

4. Testing ViewModels

ViewModel tests are the highest-value tests in your suite. They validate business logic, state management, and error handling — all on the JVM, in milliseconds. No Android framework needed.

The Setup Pattern

class UserListViewModelTest {

    @get:Rule val mainDispatcherRule = MainDispatcherRule()

    private val fakeRepo = FakeUserRepository()
    private val fakeAnalytics = FakeAnalytics()
    private val viewModel = UserListViewModel(fakeRepo, fakeAnalytics)

    @Test fun `initial state is Loading`() {
        assertThat(viewModel.uiState.value).isEqualTo(UiState.Loading)
    }

    @Test fun `loadUsers emits Success`() = runTest {
        fakeRepo.addUser(User(1, "Alice"))
        viewModel.loadUsers()
        advanceUntilIdle()
        val state = viewModel.uiState.value as UiState.Success
        assertThat(state.users).hasSize(1)
    }

    @Test fun `loadUsers emits Error on failure`() = runTest {
        fakeRepo.failWith = IOException("Network error")
        viewModel.loadUsers()
        advanceUntilIdle()
        val state = viewModel.uiState.value as UiState.Error
        assertThat(state.message).contains("Network error")
    }

    @Test fun `deleteUser removes user and tracks analytics`() = runTest {
        fakeRepo.addUser(User(1, "Alice"))
        viewModel.loadUsers()
        advanceUntilIdle()
        viewModel.deleteUser(1)
        advanceUntilIdle()
        assertThat(fakeRepo.getAllUsers()).isEmpty()
        assertThat(fakeAnalytics.trackedEvents).contains("user_deleted")
    }
}

class MainDispatcherRule(
    private val dispatcher: TestDispatcher = UnconfinedTestDispatcher()
) : TestWatcher() {
    override fun starting(description: Description) { Dispatchers.setMain(dispatcher) }
    override fun finished(description: Description) { Dispatchers.resetMain() }
}

Testing SavedStateHandle

class DetailViewModel(
    private val savedStateHandle: SavedStateHandle,
    private val repository: UserRepository
) : ViewModel() {
    val userId: Int = savedStateHandle["userId"] ?: error("userId required")
    val userName: StateFlow<String> = savedStateHandle.getStateFlow("userName", "")
}

@Test fun `reads userId from SavedStateHandle`() {
    val vm = DetailViewModel(SavedStateHandle(mapOf("userId" to 42)), FakeUserRepository())
    assertThat(vm.userId).isEqualTo(42)
}

@Test fun `persists userName across process death`() {
    val handle = SavedStateHandle(mapOf("userId" to 1, "userName" to "Alice"))
    assertThat(DetailViewModel(handle, FakeUserRepository()).userName.value).isEqualTo("Alice")
}
Staff-Level Insight — ViewModel Tests Are Your Highest-ROI Tests

At L6, you recognize that ViewModel tests sit at the sweet spot of the testing pyramid: they validate business logic and state management (high value), run on the JVM in milliseconds (low cost), and are resilient to UI refactoring (low maintenance). A well-tested ViewModel with a fake repository gives you more confidence than a hundred Espresso tests. The key insight: if your ViewModel is hard to test, the problem is the ViewModel's design — it has too many responsibilities or too many concrete dependencies. Testability is a design signal.

5. Testing Room Database

Room provides first-class testing support with in-memory databases. DAO tests validate your queries, type converters, and migration logic without touching disk.

DAO Testing with In-Memory Database

@RunWith(AndroidJUnit4::class)
class UserDaoTest {

    private lateinit var db: AppDatabase
    private lateinit var dao: UserDao

    @Before fun setup() {
        db = Room.inMemoryDatabaseBuilder(
            ApplicationProvider.getApplicationContext(), AppDatabase::class.java
        ).allowMainThreadQueries().build()
        dao = db.userDao()
    }

    @After fun tearDown() { db.close() }

    @Test fun `insert and retrieve`() {
        val user = UserEntity(id = 1, name = "Alice", email = "alice@test.com")
        dao.insert(user)
        assertThat(dao.getUserById(1)).isEqualTo(user)
    }

    @Test fun `upsert replaces existing`() {
        dao.insert(UserEntity(id = 1, name = "Alice", email = "old@test.com"))
        dao.upsert(UserEntity(id = 1, name = "Alice", email = "new@test.com"))
        assertThat(dao.getUserById(1)?.email).isEqualTo("new@test.com")
    }

    @Test fun `observeUsers emits on changes`() = runTest {
        val emissions = mutableListOf<List<UserEntity>>()
        val job = launch(UnconfinedTestDispatcher()) {
            dao.observeAllUsers().toList(emissions)
        }
        dao.insert(UserEntity(1, "Alice", "a@test.com"))
        dao.insert(UserEntity(2, "Bob", "b@test.com"))
        assertThat(emissions.last()).hasSize(2)
        job.cancel()
    }
}

Testing Migrations

@RunWith(AndroidJUnit4::class)
class MigrationTest {

    @get:Rule val helper = MigrationTestHelper(
        InstrumentationRegistry.getInstrumentation(), AppDatabase::class.java
    )

    @Test fun `v1 to v2 adds email column`() {
        helper.createDatabase("test-db", 1).apply {
            execSQL("INSERT INTO users (id, name) VALUES (1, 'Alice')")
            close()
        }
        helper.runMigrationsAndValidate("test-db", 2, true, MIGRATION_1_2).apply {
            val cursor = query("SELECT * FROM users WHERE id = 1")
            cursor.moveToFirst()
            assertThat(cursor.getString(cursor.getColumnIndex("name"))).isEqualTo("Alice")
            assertThat(cursor.getString(cursor.getColumnIndex("email"))).isNull()
            cursor.close(); close()
        }
    }

    @Test fun `full migration path v1 to latest`() {
        helper.createDatabase("test-db", 1).close()
        Room.databaseBuilder(
            ApplicationProvider.getApplicationContext(), AppDatabase::class.java, "test-db"
        ).addMigrations(*ALL_MIGRATIONS).build()
            .apply { openHelper.writableDatabase; close() }
    }
}
Common Mistake — allowMainThreadQueries

allowMainThreadQueries() is for tests only. It disables Room's main-thread check so you can write synchronous test code. Never use it in production. For Flow-based DAO methods, you still need runTest with a TestDispatcher. Also: always call database.close() in @After — in-memory databases leak native memory if not closed.

6. Testing Navigation

TestNavHostController lets you verify navigation actions without launching real Fragments or running animations. You assert on the current destination, back stack state, and argument passing.

@RunWith(AndroidJUnit4::class)
class NavigationTest {

    private lateinit var navController: TestNavHostController

    @Before fun setup() {
        navController = TestNavHostController(ApplicationProvider.getApplicationContext())
        navController.setGraph(R.navigation.main_graph)
    }

    @Test fun `navigates to detail on item click`() {
        launchFragmentInContainer<UserListFragment>().use { scenario ->
            scenario.onFragment {
                Navigation.setViewNavController(it.requireView(), navController)
            }
            onView(withId(R.id.rvUsers)).perform(
                RecyclerViewActions.actionOnItemAtPosition<ViewHolder>(0, click())
            )
            assertThat(navController.currentDestination?.id).isEqualTo(R.id.userDetailFragment)
        }
    }

    @Test fun `passes correct arguments`() {
        navController.navigate(R.id.userDetailFragment, bundleOf("userId" to 42))
        assertThat(navController.currentBackStackEntry?.arguments?.getInt("userId")).isEqualTo(42)
    }

    @Test fun `deep link resolves correctly`() {
        val intent = Intent(Intent.ACTION_VIEW, Uri.parse("myapp://users/42"))
        assertThat(navController.handleDeepLink(intent)).isTrue()
        assertThat(navController.currentDestination?.id).isEqualTo(R.id.userDetailFragment)
    }

    @Test fun `back pops to list`() {
        navController.navigate(R.id.userDetailFragment)
        navController.popBackStack()
        assertThat(navController.currentDestination?.id).isEqualTo(R.id.userListFragment)
    }
}

Limitations of TestNavHostController

  • No Fragment lifecycle: Fragments aren't actually created or destroyed — you're testing the navigation graph logic, not Fragment transitions
  • No animations/transitions: Shared element transitions, enter/exit animations are not executed
  • No real back stack management: The back stack is simulated, not the real FragmentManager back stack
  • Best for: verifying that user actions trigger the correct navigation actions with the correct arguments

7. Compose Testing

Compose testing operates on the semantics tree, not the view hierarchy. This makes tests more resilient to layout changes and closer to how accessibility services interact with your UI.

API Evolution — UI Testing on Android (2013–2024)

Espresso (2013): View-based UI testing. onView(withId(R.id.button)).perform(click()).check(matches(isDisplayed())). Operates on the View hierarchy. Requires IdlingResource for async synchronization. Still the standard for View-based UI.

UI Automator (2014): Cross-app testing. Can interact with system UI, notifications, other apps. Used for E2E tests that span app boundaries.

Compose Testing (2021): Semantic-based testing. onNodeWithText("Hello").performClick(). No view hierarchy — operates on the semantics tree (the same tree used by accessibility services). Built-in synchronization — automatically waits for recomposition and animations. No IdlingResource needed for most cases.

Current: Espresso for Views, Compose Testing for Compose. Both coexist in hybrid apps. Compose tests can run on JVM (with Robolectric) or on device.

Core API

class UserListScreenTest {

    @get:Rule val composeTestRule = createComposeRule()

    @Test fun `displays user list`() {
        composeTestRule.setContent {
            UserListScreen(
                uiState = UiState.Success(listOf(User(1, "Alice"), User(2, "Bob"))),
                onUserClick = {}, onRefresh = {}
            )
        }
        composeTestRule.onNodeWithText("Alice").assertIsDisplayed()
        composeTestRule.onNodeWithText("Bob").assertIsDisplayed()
    }

    @Test fun `click triggers callback`() {
        var clicked: Int? = null
        composeTestRule.setContent {
            UserListScreen(
                uiState = UiState.Success(listOf(User(1, "Alice"))),
                onUserClick = { clicked = it }, onRefresh = {}
            )
        }
        composeTestRule.onNodeWithText("Alice").performClick()
        assertThat(clicked).isEqualTo(1)
    }

    @Test fun `search filters users`() {
        composeTestRule.setContent {
            UserListScreen(
                uiState = UiState.Success(listOf(User(1, "Alice"), User(2, "Bob"))),
                onUserClick = {}, onRefresh = {}
            )
        }
        composeTestRule.onNodeWithTag("searchField").performTextInput("Ali")
        composeTestRule.onNodeWithText("Alice").assertIsDisplayed()
        composeTestRule.onNodeWithText("Bob").assertDoesNotExist()
    }

    @Test fun `error state shows retry`() {
        var retried = false
        composeTestRule.setContent {
            UserListScreen(
                uiState = UiState.Error("Network error"),
                onUserClick = {}, onRefresh = { retried = true }
            )
        }
        composeTestRule.onNodeWithText("Retry").performClick()
        assertThat(retried).isTrue()
    }
}

class ComposeNavigationTest {

Espresso — Quick Reference

@RunWith(AndroidJUnit4::class)
class LoginActivityTest {

    @get:Rule val activityRule = ActivityScenarioRule(LoginActivity::class.java)

    @Test fun loginWithValidCredentials() {
        onView(withId(R.id.etEmail)).perform(typeText("alice@test.com"), closeSoftKeyboard())
        onView(withId(R.id.etPassword)).perform(typeText("password123"), closeSoftKeyboard())
        onView(withId(R.id.btnLogin)).perform(click())
        onView(withId(R.id.tvWelcome)).check(matches(withText("Welcome, Alice")))
    }
}

// Custom IdlingResource — needed for async operations in Espresso
class OkHttpIdlingResource(private val client: OkHttpClient) : IdlingResource {
    private var callback: IdlingResource.ResourceCallback? = null
    override fun getName() = "OkHttp"
    override fun isIdleNow(): Boolean {
        val idle = client.dispatcher.runningCallsCount() == 0
        if (idle) callback?.onTransitionToIdle()
        return idle
    }
    override fun registerIdleTransitionCallback(cb: ResourceCallback) { callback = cb }
}
Staff-Level Insight — Choosing Your UI Testing Strategy

At L6, the question isn't "Espresso or Compose Testing?" — it's "what's the right testing strategy for a hybrid codebase in transition?" The answer: test new Compose screens with Compose Testing, keep existing Espresso tests for View-based screens, and never rewrite working tests just to change frameworks. For hybrid screens (Compose inside Fragments), use createAndroidComposeRule<MyActivity>() which gives you both Espresso and Compose testing APIs in the same test. The migration path is organic: as you rewrite screens in Compose, the tests naturally migrate too. Forcing a test framework migration ahead of the UI migration is wasted effort.

The Testing Diamond Anti-Pattern

The "testing diamond" occurs when teams write too many integration tests and too few unit tests. The shape inverts: wide in the middle (integration), narrow at the top (E2E) and bottom (unit).

    Healthy Pyramid              Testing Diamond (Anti-Pattern)

         ╱╲                              ╱╲
        ╱E2E╲                           ╱E2E╲
       ╱──────╲                        ╱──────╲
      ╱ Integr. ╲                    ╱            ╲
     ╱────────────╲                ╱  Integration   ╲
    ╱   Unit Tests  ╲            ╱────────────────────╲
   ╱──────────────────╲          ╲    Unit Tests     ╱
                                   ╲────────────────╱

   ✓ Fast feedback loop          ✗ Slow CI (integration tests dominate)
   ✓ Cheap to maintain           ✗ Brittle (testing wiring, not logic)
   ✓ Refactor-friendly           ✗ Hard to refactor (tests break)

Root cause: ViewModels that directly depend on concrete classes (Room DAOs, Retrofit services) instead of interfaces. When you can't inject a fake, you're forced into integration tests.

Fix: Introduce repository interfaces, inject dispatchers, use constructor injection. Every dependency you make injectable is a test you can move from integration to unit.

Code Coverage — Meaningful vs Vanity Metrics

Danger — 80% Coverage ≠ 80% Confidence

Line coverage measures which lines were executed, not which behaviors were verified. A test that calls viewModel.loadUsers() without asserting anything achieves 100% coverage of that method with 0% confidence. At L6, push for mutation testing (Pitest) over line coverage — it measures whether your tests actually detect bugs. A mutation that changes > to >= and no test fails means your tests have a blind spot, regardless of coverage percentage.

Staff-Level Insight — The Test Infrastructure Is a Product

At L6, you treat test infrastructure as a product with its own users (your team). Shared fakes, custom rules, test data builders, and CI configuration are force multipliers — every hour invested saves hundreds of hours across the team. The sign of a mature test infrastructure: a new team member can write a well-structured test for any component in under 10 minutes, because the fakes, rules, and patterns are already in place. If writing a test requires 30 minutes of boilerplate setup, the infrastructure has failed.
