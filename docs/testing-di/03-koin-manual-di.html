03 — Koin, Manual DI & Framework Comparison

Runtime DI with Koin, the no-framework approach, kotlin-inject, and a decision framework for choosing the right tool

1. Koin — Runtime DI for Kotlin

Koin is a lightweight dependency injection framework that uses Kotlin DSL instead of annotation processing. There is no code generation — resolution happens entirely at runtime through a service locator backed by a HashMap.

Core Concepts

// 1. Define modules using Kotlin DSL
val networkModule = module {
    single { OkHttpClient.Builder().build() }
    single { Retrofit.Builder()
        .client(get())   // ← resolves OkHttpClient from the container
        .baseUrl("https://api.example.com/")
        .build()
    }
    single<ApiService> { get<Retrofit>().create(ApiService::class.java) }
}

val repositoryModule = module {
    single<UserRepository> { UserRepositoryImpl(api = get()) }
    factory { FetchUserUseCase(repo = get()) }
}

val viewModelModule = module {
    viewModel { UserListViewModel(fetchUser = get()) }
    viewModel { (userId: String) ->  // ← parameterized
        UserDetailViewModel(userId = userId, repo = get())
    }
}

// 2. Start Koin in Application
class MyApp : Application() {
    override fun onCreate() {
        super.onCreate()
        startKoin {
            androidContext(this@MyApp)
            modules(networkModule, repositoryModule, viewModelModule)
        }
    }
}

// 3. Inject in Activity/Fragment
class UserListActivity : AppCompatActivity() {
    private val viewModel: UserListViewModel by viewModel()
    private val repo: UserRepository by inject()  // lazy injection
}

// 4. Inject in Composable
@Composable
fun UserListScreen(
    viewModel: UserListViewModel = koinViewModel()
) {
    val users by viewModel.users.collectAsState()
    // ...
}

Key DSL primitives:

  • single { } — singleton scoped to the Koin container lifetime
  • factory { } — new instance on every get() call
  • viewModel { } — tied to ViewModelStore, survives configuration changes
  • get() — resolves a dependency from the container (inside a definition)
  • by inject() — lazy property delegate for injection outside definitions

Scoping

val scopedModule = module {
    scope<MainActivity> {
        scoped { SessionManager(context = get()) }  // lives as long as the scope
        factory { AnalyticsTracker(session = get()) }
    }
}

// In Activity — create and manage the scope
class MainActivity : AppCompatActivity(), KoinScopeComponent {
    override val scope: Scope by activityScope()

    private val session: SessionManager by inject()  // resolved from this scope

    override fun onDestroy() {
        super.onDestroy()
        // scope auto-closed by activityScope() delegate
    }
}
⚠️ Scoping Pitfall
Koin scopes are manually managed. If you forget to close a scope, its singletons leak. Hilt scopes are tied to Android component lifecycles automatically — Koin requires you to wire this yourself (or use the activityScope() / fragmentScope() delegates which handle it).

2. Koin Internals

Understanding what happens behind the DSL is essential for debugging resolution failures and reasoning about performance at scale.

  Koin Resolution Flow
  ═══════════════════════════════════════════════════════════════

  startKoin { modules(m1, m2) }
       │
       ▼
  ┌─────────────────────┐
  │   KoinApplication    │  ← entry point, holds global state
  │  ┌─────────────────┐ │
  │  │   Koin instance  │ │  ← the actual container
  │  │  ┌─────────────┐ │ │
  │  │  │ InstanceReg- │ │ │  ← HashMap<IndexKey, InstanceFactory>
  │  │  │ istry        │ │ │     IndexKey = (qualifier, primaryType)
  │  │  └─────────────┘ │ │
  │  │  ┌─────────────┐ │ │
  │  │  │ ScopeReg-   │ │ │  ← registered scope definitions
  │  │  │ istry       │ │ │
  │  │  └─────────────┘ │ │
  │  └─────────────────┘ │
  └─────────────────────┘

  get<UserRepository>()
       │
       ▼
  1. Build IndexKey(qualifier=null, type=UserRepository::class)
  2. Look up in InstanceRegistry HashMap
  3. Found? → InstanceFactory.get(InstanceContext)
       │
       ├── SingleInstanceFactory → return cached instance (or create + cache)
       ├── FactoryInstanceFactory → create new instance every time
       └── ScopedInstanceFactory → check scope's instance map
  4. Not found? → throw NoBeanDefFoundException (RUNTIME CRASH)

Key Internal Classes

  • BeanDefinition<T> — metadata for a binding: the Kotlin type, qualifier, Kind (Singleton/Factory/Scoped), and the lambda that creates the instance
  • InstanceFactory<T> — wraps a BeanDefinition and manages instance lifecycle. SingleInstanceFactory caches in a volatile var; FactoryInstanceFactory invokes the lambda every time
  • Scope — a named container that holds scoped instances. The root scope (_root_) is where single { } definitions live. Custom scopes (scope<Activity> { }) create child scopes with their own instance maps
  • IndexKey — the lookup key: data class IndexKey(val primaryType: KClass<*>, val qualifier: Qualifier?). This is why you can have two bindings of the same type if they have different qualifiers

Resolution Algorithm

// Simplified resolution logic inside Scope.resolveInstance()
internal fun <T> resolveInstance(
    qualifier: Qualifier?,
    clazz: KClass<*>,
    parameters: ParametersHolder?
): T {
    // 1. Check THIS scope's registry
    val indexKey = indexKey(clazz, qualifier)
    val factory = _instanceRegistry[indexKey]

    if (factory != null) {
        return factory.get(InstanceContext(this, parameters)) as T
    }

    // 2. Walk up to parent scopes (linked scopes)
    _linkedScopes.forEach { linked ->
        val result = linked.resolveInstance<T>(qualifier, clazz, parameters)
        if (result != null) return result
    }

    // 3. Fall back to root scope
    val rootFactory = _koin.instanceRegistry[indexKey]
    if (rootFactory != null) {
        return rootFactory.get(InstanceContext(this, parameters)) as T
    }

    // 4. Nothing found → crash
    throw NoBeanDefFoundException(
        "No definition found for '${clazz.qualifiedName}' with qualifier '$qualifier'"
    )
}
🎯 L6 Insight — Koin's Achilles Heel at Scale
Koin's HashMap lookup is O(1) per resolution, so raw performance isn't the problem. The real issue is error detection surface area. In a 500-module app, a missing binding is a NoBeanDefFoundException that only surfaces when that specific code path executes. Hilt catches this at compile time. At Google scale, a single missed binding in a rarely-tested code path can cause a production crash that slips through CI. This is why compile-time DI is non-negotiable for large apps — it shifts the entire class of "missing binding" bugs from runtime to build time.

Module Loading Performance

// Koin module loading is O(n) where n = total definitions
// Each definition: allocate BeanDefinition + InstanceFactory + HashMap.put()
//
// Measured on Pixel 6, 200 definitions:
//   startKoin { modules(...) }  →  ~8-15ms
//
// 1000 definitions:
//   startKoin { modules(...) }  →  ~40-80ms
//
// This happens on the main thread during Application.onCreate()
// For comparison, Hilt's generated code: ~0ms (already compiled)

// Lazy modules (Koin 3.2+) defer loading:
val heavyModule = lazyModule {
    single { ExpensiveService() }
}
// Loaded on first access, not at startup
koin.loadModules(heavyModule)
🔄 Evolution — Koin Verify (3.3+)
Koin addressed the "runtime crash" criticism by introducing verify() — a unit test utility that walks all module definitions and checks that every get() call can be resolved. This doesn't match Hilt's compile-time guarantee (it requires you to write and run the test), but it closes the gap significantly for teams that integrate it into CI. Usage: @Test fun verifyModules() { myModule.verify() }. Limitation: it can't verify parameterized definitions or Android-specific types (Context, SavedStateHandle) without extra configuration.

When to Choose Koin

  • Kotlin Multiplatform projects — Koin runs on JVM, iOS, JS, and native targets; Hilt is Android-only
  • Small to medium apps — fewer bindings means the "runtime crash" risk is manageable
  • Rapid prototyping — zero build overhead, instant feedback loop
  • Teams unfamiliar with Dagger — the DSL is immediately readable to any Kotlin developer
  • Build time is critical — kapt adds 20-40% to build times in large projects; Koin adds nothing

When to Choose Hilt

  • Large apps — compile-time safety catches missing bindings before they reach production
  • Google-stack projects — Hilt integrates with ViewModel, WorkManager, Navigation, Compose
  • Teams already on Dagger — Hilt is a natural migration path with incremental adoption
  • Runtime DI crashes are unacceptable — in apps with hundreds of screens, untested paths exist
  • Google L6 interviews — Hilt/Dagger is what Google uses internally; demonstrating deep knowledge signals alignment
🎯 L6 Insight — The Real Trade-off
The Koin-vs-Hilt debate is really about where you want to pay the cost of correctness. Koin pays at runtime (fast builds, potential crashes). Hilt pays at build time (slow builds, guaranteed correctness). In a mature CI pipeline with comprehensive test coverage, Koin's risk is mitigated. But at Google scale — thousands of modules, hundreds of contributors — you can't rely on test coverage alone to catch every missing binding. The compile-time guarantee isn't just convenient; it's a structural property of the build system that scales independently of test quality.

4. Manual DI — The No-Framework Approach

Manual DI means constructing and wiring your dependency graph by hand, without any framework. Android's official documentation describes this as the AppContainer pattern.

The AppContainer Pattern

// Application-scoped container — lives for the entire process
class AppContainer(private val context: Context) {
    // Singletons
    private val okHttpClient = OkHttpClient.Builder()
        .addInterceptor(AuthInterceptor { authTokenProvider.token })
        .build()

    private val retrofit = Retrofit.Builder()
        .client(okHttpClient)
        .baseUrl("https://api.example.com/")
        .addConverterFactory(MoshiConverterFactory.create())
        .build()

    val apiService: ApiService = retrofit.create(ApiService::class.java)
    val database: AppDatabase = Room.databaseBuilder(
        context, AppDatabase::class.java, "app.db"
    ).build()

    val authTokenProvider = AuthTokenProvider(
        prefs = context.getSharedPreferences("auth", Context.MODE_PRIVATE)
    )

    // Factory methods for non-singleton dependencies
    fun userRepository(): UserRepository = UserRepositoryImpl(
        api = apiService,
        dao = database.userDao()
    )
}

// Activity-scoped container — created per Activity instance
class UserActivityContainer(private val app: AppContainer) {
    private val repo = app.userRepository()

    val fetchUserUseCase = FetchUserUseCase(repo)

    fun viewModelFactory() = object : ViewModelProvider.Factory {
        override fun <T : ViewModel> create(modelClass: Class<T>): T {
            @Suppress("UNCHECKED_CAST")
            return UserListViewModel(fetchUserUseCase) as T
        }
    }
}

// Wire it up
class MyApp : Application() {
    lateinit var container: AppContainer
    override fun onCreate() {
        super.onCreate()
        container = AppContainer(this)
    }
}

class UserListActivity : AppCompatActivity() {
    private lateinit var activityContainer: UserActivityContainer

    private val viewModel: UserListViewModel by viewModels {
        activityContainer.viewModelFactory()
    }

    override fun onCreate(savedInstanceState: Bundle?) {
        activityContainer = UserActivityContainer(
            (application as MyApp).container
        )
        super.onCreate(savedInstanceState)
    }
}

Testing with Manual DI

// The beauty of manual DI: testing is trivial
// Just pass fakes directly — no framework magic needed

class FakeUserRepository : UserRepository {
    var users = listOf(User("1", "Alice"), User("2", "Bob"))
    var shouldFail = false

    override suspend fun getUsers(): List<User> {
        if (shouldFail) throw IOException("Network error")
        return users
    }
}

class FetchUserUseCaseTest {
    @Test
    fun `returns users from repository`() = runTest {
        val fakeRepo = FakeUserRepository()
        val useCase = FetchUserUseCase(fakeRepo)

        val result = useCase()

        assertEquals(2, result.size)
        assertEquals("Alice", result[0].name)
    }

    @Test
    fun `propagates repository errors`() = runTest {
        val fakeRepo = FakeUserRepository().apply { shouldFail = true }
        val useCase = FetchUserUseCase(fakeRepo)

        assertThrows<IOException> { useCase() }
    }
}

// No @UninstallModules, no loadModules override, no test rules.
// Constructor injection + fakes = the simplest possible test setup.

When Manual DI Is the Right Choice

  • Libraries/SDKs — you can't force consumers to use Hilt or Koin; manual DI keeps your library framework-agnostic
  • Very small apps — under ~20 classes, a framework adds complexity without proportional benefit
  • Learning — building a manual DI container teaches the concepts that frameworks abstract away
  • Performance-critical paths — zero overhead, zero reflection, zero startup cost
⚠️ Manual DI Breaks Down
Manual DI doesn't scale. At ~50+ classes, the AppContainer becomes a god object. You end up reinventing scoping, lazy initialization, and circular dependency detection — poorly. The inflection point is when you find yourself writing lazy { } wrappers and scope-tracking logic inside your container. At that point, you've built a worse version of Koin.

5. Kotlin Inject — The Emerging Alternative

kotlin-inject is a compile-time DI framework built natively for Kotlin using KSP. It's conceptually similar to Dagger but designed from scratch for Kotlin's type system.

// Define a component (similar to Dagger's @Component)
@Component
@Singleton
abstract class AppComponent(
    @get:Provides val context: Application
) {
    abstract val userRepository: UserRepository

    // Binds implementation to interface
    @Provides
    fun provideApiService(client: OkHttpClient): ApiService =
        Retrofit.Builder().client(client).build().create(ApiService::class.java)

    // Singleton scope
    @Provides @Singleton
    fun provideOkHttpClient(): OkHttpClient = OkHttpClient.Builder().build()

    // Bind interface to implementation
    val UserRepositoryImpl.bind: UserRepository
        @Provides get() = this
}

// Constructor injection — same as Dagger
@Inject
class UserRepositoryImpl(
    private val api: ApiService,
    private val dao: UserDao
) : UserRepository {
    override suspend fun getUsers() = api.fetchUsers()
}

// Generated code creates the component
val component = AppComponent::class.create(application)
val repo = component.userRepository

🔄 Evolution — The KSP Shift
The DI landscape is shifting toward KSP. Hilt added KSP support (still experimental as of 2024), kotlin-inject is KSP-native, and even Koin introduced koin-annotations with KSP for optional compile-time verification. kapt's days are numbered — it runs Kotlin → Java stubs → annotation processing, adding significant build overhead. KSP processes Kotlin directly, cutting annotation processing time by 2-3x. For new projects, prefer KSP-based solutions. For existing Dagger/Hilt projects, the kapt → KSP migration is worth the effort.

When to Consider kotlin-inject

  • New KMP projects wanting compile-time safety without Dagger's Java heritage
  • Teams that want Dagger-like guarantees but find Hilt's Android coupling too opinionated
  • Library authors who need DI internally but can't depend on Hilt
  • Not yet recommended for: large production Android apps (limited Android lifecycle integration, smaller ecosystem)

7. Migration Patterns

Dagger 2 → Hilt (Incremental)

// Step 1: Add @HiltAndroidApp to Application
// This is backward-compatible — existing Dagger components still work
@HiltAndroidApp
class MyApp : Application() {
    // Keep existing AppComponent for now
    lateinit var appComponent: AppComponent
}

// Step 2: Migrate one Activity at a time
// Before (Dagger):
class UserActivity : AppCompatActivity() {
    @Inject lateinit var viewModelFactory: ViewModelFactory
    override fun onCreate(savedInstanceState: Bundle?) {
        (application as MyApp).appComponent.inject(this)
        super.onCreate(savedInstanceState)
    }
}

// After (Hilt):
@AndroidEntryPoint
class UserActivity : AppCompatActivity() {
    @Inject lateinit var viewModelFactory: ViewModelFactory
    // No manual injection needed — Hilt handles it
}

// Step 3: Move modules to @InstallIn
// Before:
@Module
class NetworkModule {
    @Provides @Singleton
    fun provideOkHttp(): OkHttpClient = OkHttpClient.Builder().build()
}

// After:
@Module
@InstallIn(SingletonComponent::class)
object NetworkModule {  // note: object, not class
    @Provides @Singleton
    fun provideOkHttp(): OkHttpClient = OkHttpClient.Builder().build()
}

// Step 4: Once all Activities/Fragments are @AndroidEntryPoint,
// remove the old AppComponent and @Component interface entirely.

Koin → Hilt

// Koin module:
val appModule = module {
    single { OkHttpClient.Builder().build() }
    single<UserRepository> { UserRepositoryImpl(api = get()) }
    viewModel { UserListViewModel(repo = get()) }
}

// Equivalent Hilt:
@Module
@InstallIn(SingletonComponent::class)
object NetworkModule {
    @Provides @Singleton
    fun provideOkHttp(): OkHttpClient = OkHttpClient.Builder().build()
}

@Module
@InstallIn(SingletonComponent::class)
abstract class RepositoryModule {
    @Binds @Singleton
    abstract fun bindUserRepo(impl: UserRepositoryImpl): UserRepository
}

// Koin: by inject()  →  Hilt: @Inject lateinit var
// Koin: by viewModel()  →  Hilt: @HiltViewModel + hiltViewModel()
// Koin: get()  →  Hilt: @Inject constructor()

// ViewModel migration:
// Koin:
class UserListViewModel(private val repo: UserRepository) : ViewModel()

// Hilt:
@HiltViewModel
class UserListViewModel @Inject constructor(
    private val repo: UserRepository
) : ViewModel()

Manual DI → Hilt

// Manual:
class AppContainer(context: Context) {
    val okHttp = OkHttpClient.Builder().build()
    val apiService: ApiService = Retrofit.Builder()
        .client(okHttp).build().create(ApiService::class.java)
    fun userRepo(): UserRepository = UserRepositoryImpl(apiService)
}

// Step 1: Replace AppContainer fields with @Module @Provides
@Module
@InstallIn(SingletonComponent::class)
object AppModule {
    @Provides @Singleton
    fun provideOkHttp(): OkHttpClient = OkHttpClient.Builder().build()

    @Provides @Singleton
    fun provideApiService(client: OkHttpClient): ApiService =
        Retrofit.Builder().client(client).build().create(ApiService::class.java)
}

// Step 2: Add @Inject to implementation constructors
class UserRepositoryImpl @Inject constructor(
    private val api: ApiService
) : UserRepository

// Step 3: Bind interfaces
@Module
@InstallIn(SingletonComponent::class)
abstract class BindingsModule {
    @Binds abstract fun bindUserRepo(impl: UserRepositoryImpl): UserRepository
}

// Step 4: Replace (application as MyApp).container.xxx
//         with @Inject or @AndroidEntryPoint

// Step 5: Delete AppContainer, ActivityContainer, and the
//         lateinit var container in Application
🚨 Migration Anti-Pattern
Never run two DI frameworks simultaneously in production (e.g., Koin for old screens, Hilt for new ones). The two containers don't share instances — you'll get duplicate singletons, inconsistent state, and impossible-to-debug issues. Migrate module by module, but keep a single source of truth for each binding. During Dagger→Hilt migration, this works because Hilt is Dagger under the hood and can coexist with raw Dagger components.
