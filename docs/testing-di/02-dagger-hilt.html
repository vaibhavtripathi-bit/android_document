02 — Dagger 2 & Hilt Deep Dive

Code generation internals, the annotation processing pipeline, component hierarchy, scoping mechanics, and the evolution from manual Dagger to Hilt

1. Dagger 2 Internals — How Code Generation Works

The Annotation Processing Pipeline

Dagger 2 is a fully compile-time DI framework. There is zero reflection at runtime. The entire dependency graph is resolved, validated, and turned into plain Kotlin/Java code by an annotation processor before your app ever runs.

The pipeline has two paths depending on your build configuration:

ProcessorMechanismSpeedKotlin Support
kaptGenerates Java stubs from Kotlin, then runs Java annotation processorSlow — requires stub generation passFull (via stubs)
KSPDirectly analyzes Kotlin symbols, no stub generation~2x faster than kaptNative

As of Dagger 2.48+, KSP is fully supported. Hilt also supports KSP. For new projects, always prefer KSP over kapt.

What Gets Generated — The Four Factory Types

Dagger generates exactly four kinds of classes. Understanding these is the key to debugging any Dagger issue:

// 1. @Component → DaggerXxxComponent
//    The root factory. Implements your component interface.
//    Contains all wiring logic as Provider<T> fields.

@Component(modules = [NetworkModule::class])
interface AppComponent {
    fun inject(activity: MainActivity)
    fun repository(): UserRepository
}
// → Generates: DaggerAppComponent (implements AppComponent)

// 2. @Module + @Provides → XxxModule_ProvideYyyFactory
//    One factory per @Provides method.

@Module
class NetworkModule {
    @Provides
    fun provideOkHttpClient(): OkHttpClient = OkHttpClient.Builder().build()
}
// → Generates: NetworkModule_ProvideOkHttpClientFactory

// 3. @Inject constructor → Xxx_Factory
//    One factory per constructor-injected class.

class UserRepository @Inject constructor(
    private val api: UserApi,
    private val db: UserDao
)
// → Generates: UserRepository_Factory

// 4. @Component + inject(target) → XxxComponent_MembersInjector
//    For field injection targets.
// → Generates: MainActivity_MembersInjector

Inside the Generated Component

The generated DaggerAppComponent is where the magic happens. Here's a simplified view of what it actually looks like:

// Generated by Dagger (simplified)
class DaggerAppComponent private constructor(
    private val networkModule: NetworkModule
) : AppComponent {

    // Each dependency is a Provider<T> — a lazy factory
    private val okHttpClientProvider: Provider<OkHttpClient> =
        NetworkModule_ProvideOkHttpClientFactory.create(networkModule)

    private val userApiProvider: Provider<UserApi> =
        RetrofitUserApi_Factory.create(okHttpClientProvider)

    private val userDaoProvider: Provider<UserDao> =
        RoomUserDao_Factory.create(databaseProvider)

    // For @Singleton: DoubleCheck wraps the provider
    private val userRepositoryProvider: Provider<UserRepository> =
        DoubleCheck.provider(
            UserRepository_Factory.create(userApiProvider, userDaoProvider)
        )

    override fun repository(): UserRepository =
        userRepositoryProvider.get()

    override fun inject(activity: MainActivity) {
        // Calls MembersInjector to set @Inject fields
        MainActivity_MembersInjector.injectViewModel(
            activity, userRepositoryProvider.get()
        )
    }

    class Builder {
        private var networkModule: NetworkModule? = null
        fun networkModule(module: NetworkModule) = apply { this.networkModule = module }
        fun build(): AppComponent = DaggerAppComponent(
            networkModule ?: NetworkModule()
        )
    }
}
Staff-Level Insight — Provider<T> Chains

Every dependency in Dagger is a Provider<T> — a get() call that returns an instance. Unscoped providers create a new instance every call. DoubleCheck wraps a provider to cache the first result (thread-safe via double-checked locking — hence the name). Lazy<T> injection uses the same DoubleCheck internally. This means @Singleton @Provides fun provideFoo(): Foo and @Inject constructor(val foo: Lazy<Foo>) use the exact same caching mechanism under the hood.

The Dependency Graph as a DAG

        Compile-Time Dependency Graph (Directed Acyclic Graph)

    AppComponent
        │
        ├── UserRepository ──────┬── UserApi ──── OkHttpClient
        │   (DoubleCheck)        │                (from @Provides)
        │                        └── UserDao ──── AppDatabase
        │                                         (from @Provides)
        ├── AuthService ─────────┬── TokenStore
        │                        └── UserApi (shared reference ↑)
        │
        └── AnalyticsTracker ──── OkHttpClient (shared reference ↑)

    Key insight: Dagger resolves this entire graph at COMPILE TIME.
    If any edge is missing (no binding for a type), you get a
    compile error — not a runtime crash.

This compile-time validation is Dagger's killer feature. The annotation processor walks the DAG, verifies every node has a binding, detects cycles, and emits an error with the full dependency chain if anything is missing.

2. Dagger 2 Core Concepts

@Component — The Root of the Graph

A @Component is an interface that defines the entry points into the dependency graph. Dagger generates the implementation.

@Singleton
@Component(modules = [NetworkModule::class, DatabaseModule::class])
interface AppComponent {
    // Provision method — returns a dependency directly
    fun userRepository(): UserRepository

    // Members-injection method — injects @Inject fields into target
    fun inject(activity: MainActivity)

    @Component.Factory
    interface Factory {
        fun create(@BindsInstance app: Application): AppComponent
    }
}

@Component.Factory (preferred) and @Component.Builder let you pass runtime instances (like Application) into the graph via @BindsInstance.

@Module + @Provides vs @Binds

Modules supply bindings that can't come from constructor injection — interfaces, third-party classes, or configured instances.

@Module
abstract class RepositoryModule {
    // @Binds: tells Dagger "when someone asks for UserRepository,
    // give them UserRepositoryImpl". No factory class generated —
    // Dagger just uses UserRepositoryImpl_Factory directly.
    @Binds
    abstract fun bindUserRepository(impl: UserRepositoryImpl): UserRepository

    companion object {
        // @Provides: for things you need to configure or can't
        // constructor-inject. Generates a factory class.
        @Provides
        fun provideRetrofit(client: OkHttpClient): Retrofit =
            Retrofit.Builder()
                .baseUrl("https://api.example.com/")
                .client(client)
                .addConverterFactory(GsonConverterFactory.create())
                .build()
    }
}
Aspect@Provides@Binds
Use caseThird-party classes, configured instancesInterface → implementation mapping
Generated codeFull factory class (Module_ProvideXxxFactory)No factory — reuses impl's factory
Module typeConcrete class or companion objectabstract class or interface
PerformanceOne extra class + method callZero overhead

@Scope — Lifecycle-Bound Caching

Scopes control how long an instance lives. A scoped binding creates exactly one instance per component instance.

@Scope
@Retention(AnnotationRetention.RUNTIME)
annotation class ActivityScope

@ActivityScope
@Component(dependencies = [AppComponent::class])
interface ActivityComponent { ... }

// This class gets ONE instance per ActivityComponent instance
@ActivityScope
class Navigator @Inject constructor(private val router: Router)

Internally, scoping wraps the Provider<T> in DoubleCheck. Without a scope annotation, every get() call creates a new instance.

Multibindings — Plugin Architectures

@Module
abstract class AnalyticsModule {
    @Binds @IntoSet
    abstract fun bindFirebaseTracker(impl: FirebaseTracker): AnalyticsTracker

    @Binds @IntoSet
    abstract fun bindMixpanelTracker(impl: MixpanelTracker): AnalyticsTracker
}

// Now you can inject a Set of all trackers:
class AnalyticsDispatcher @Inject constructor(
    private val trackers: Set<@JvmSuppressWildcards AnalyticsTracker>
) {
    fun track(event: Event) = trackers.forEach { it.track(event) }
}

// @IntoMap for keyed access:
@Module
abstract class ViewModelModule {
    @Binds @IntoMap
    @ClassKey(HomeViewModel::class)
    abstract fun bindHomeVm(vm: HomeViewModel): ViewModel
}

@JvmSuppressWildcards is required because Kotlin generics compile to Set<? extends AnalyticsTracker> in Java, which Dagger can't match. This is a common source of cryptic compile errors.

3. Evolution: Dagger 2 → Hilt

Evolution — The Dagger 2 Pain Points That Created Hilt

Before Hilt, every Android team using Dagger had to solve the same problems independently:

  • Manual component creation: You had to create DaggerAppComponent in Application.onCreate(), then somehow make it accessible to Activities and Fragments — usually via casting application as MyApp.
  • Manual injection in Activities/Fragments: Every Activity needed (application as MyApp).appComponent.activityComponentFactory().create(this).inject(this) in onCreate(). Forget one line and you get a UninitializedPropertyAccessException at runtime.
  • No standard component hierarchy: Some teams used subcomponents per Activity, others used component dependencies. Some scoped to Activity, others to Fragment. No two Dagger setups looked alike.
  • ViewModel injection was a hack: You needed a custom ViewModelProvider.Factory that delegated to Dagger's multibinding map — 50+ lines of boilerplate that every team copy-pasted.
  • Testing was painful: Replacing a single binding for tests meant creating a test component, test modules, and wiring them manually.

Google's internal Android teams had the same problems. Hilt was born from their internal solution (dagger.android was the first attempt, but it was too complex). Hilt standardizes the component hierarchy and automates the boilerplate.

What Hilt Actually Generates

When you annotate an Activity with @AndroidEntryPoint, Hilt generates a base class that handles injection automatically:

// Your code:
@AndroidEntryPoint
class MainActivity : AppCompatActivity() {
    @Inject lateinit var analytics: AnalyticsTracker
}

// What Hilt generates (simplified):

// 1. Hilt_MainActivity — base class with injection logic
abstract class Hilt_MainActivity : AppCompatActivity() {
    private var injected = false

    override fun onCreate(savedInstanceState: Bundle?) {
        // Inject BEFORE super.onCreate() so dependencies are
        // available when Fragment.onAttach() is called
        inject()
        super.onCreate(savedInstanceState)
    }

    private fun inject() {
        if (!injected) {
            val component = EntryPoints.get(
                applicationContext, MainActivity_GeneratedInjector::class.java
            )
            component.injectMainActivity(this as MainActivity)
            injected = true
        }
    }
}

// 2. MainActivity_GeneratedInjector — entry point interface
@EntryPoint
@InstallIn(ActivityComponent::class)
interface MainActivity_GeneratedInjector {
    fun injectMainActivity(activity: MainActivity)
}

// Your MainActivity now extends Hilt_MainActivity (bytecode transform)
Staff-Level Insight — The Bytecode Transform

Hilt uses a Gradle transform (via the hilt-android-gradle-plugin) to rewrite your class's superclass. Your source says class MainActivity : AppCompatActivity(), but after the transform, the bytecode says class MainActivity : Hilt_MainActivity(). This is why you don't see Hilt_ in your source code but it appears in stack traces. The transform also handles the tricky ordering: injection happens in onCreate() before super.onCreate(), which is critical because super.onCreate() restores Fragments, and those Fragments might access injected fields during onAttach().

4. Hilt — The Android DI Standard

@HiltAndroidApp — The Root

@HiltAndroidApp
class MyApplication : Application() {
    // Hilt generates SingletonComponent and attaches it to this Application.

Under the hood, @HiltAndroidApp generates a Hilt_MyApplication base class that creates the SingletonComponent in onCreate() and stores it as a field. All other components are children of this root.

@InstallIn — Placing Modules in the Hierarchy

// This module's bindings are available to SingletonComponent
// and ALL its children (Activity, Fragment, Service, etc.)
@Module
@InstallIn(SingletonComponent::class)
object NetworkModule {
    @Provides @Singleton
    fun provideOkHttpClient(): OkHttpClient = OkHttpClient()
}

// This module's bindings are only available to FragmentComponent
// and its children (ViewComponent, ViewWithFragmentComponent)
@Module
@InstallIn(FragmentComponent::class)
object FragmentAnalyticsModule {
    @Provides
    fun provideFragmentTracker(fragment: Fragment): FragmentTracker =
        FragmentTracker(fragment.javaClass.simpleName)
}

@HiltViewModel — ViewModel Injection

@HiltViewModel
class UserViewModel @Inject constructor(
    private val repository: UserRepository,
    private val savedStateHandle: SavedStateHandle  // auto-provided by Hilt
) : ViewModel() {

    private val userId: String = savedStateHandle["userId"]
        ?: throw IllegalArgumentException("userId required")

    val user = repository.getUser(userId)
        .stateIn(viewModelScope, SharingStarted.WhileSubscribed(5000), null)
}

// In Activity/Fragment — no factory needed:
@AndroidEntryPoint
class UserFragment : Fragment() {
    private val viewModel: UserViewModel by viewModels()
}

Before Hilt, this required a custom ViewModelProvider.Factory backed by Dagger's @IntoMap multibinding — roughly 50 lines of boilerplate that Hilt eliminates entirely.

@EntryPoint — Escaping the Hilt World

// For classes that Hilt can't inject (ContentProvider, third-party code):
@EntryPoint
@InstallIn(SingletonComponent::class)
interface AnalyticsEntryPoint {
    fun analyticsTracker(): AnalyticsTracker
}

// Usage in a ContentProvider (created before Application.onCreate):
class MyContentProvider : ContentProvider() {
    override fun onCreate(): Boolean {
        val entryPoint = EntryPointAccessors.fromApplication(
            context!!.applicationContext,
            AnalyticsEntryPoint::class.java
        )
        val tracker = entryPoint.analyticsTracker()
        return true
    }
}
Warning — ContentProvider Timing

ContentProvider.onCreate() runs before Application.onCreate(). If your @EntryPoint depends on something initialized in Application.onCreate(), you'll get a crash. Hilt handles this by initializing the component eagerly during ContentProvider creation, but your own initialization code in Application.onCreate() hasn't run yet. Design your singleton bindings to be lazily initialized or avoid depending on Application-level setup from ContentProviders.

5. Hilt Scoping Deep Dive

Unscoped vs Scoped — The Critical Distinction

// UNSCOPED: new instance every time it's injected
class AnalyticsLogger @Inject constructor() {
    init { Log.d("DI", "New AnalyticsLogger: ${hashCode()}") }
}

// If injected into 3 different classes → 3 different instances

// SCOPED: one instance per component instance
@Singleton
class AnalyticsLogger @Inject constructor() {
    init { Log.d("DI", "Singleton AnalyticsLogger: ${hashCode()}") }
}

// If injected into 3 different classes → same instance

Scope-to-Component Mapping

@ActivityScopedActivityComponentActivity instanceNo
@FragmentScopedFragmentComponentFragment instanceNo

The Scoping Trap

Danger — Over-Scoping Wastes Memory

A common mistake is making everything @Singleton "for safety." Every @Singleton binding lives for the entire process lifetime — it's never garbage collected. If you scope a 2MB image cache as @Singleton when it's only needed in one screen, that memory is held forever. Rule of thumb: scope stateful objects that must be shared (repositories, caches, connection pools). Don't scope stateless objects (mappers, validators, formatters) — creating new instances is cheap and they'll be GC'd when no longer referenced.

How Scoping Works Internally

// Unscoped binding in generated code:
// Every get() creates a new instance
private val loggerProvider: Provider<Logger> =
    Logger_Factory.create()

// Scoped binding in generated code:
// DoubleCheck caches the first result
private val loggerProvider: Provider<Logger> =
    DoubleCheck.provider(Logger_Factory.create())

// DoubleCheck internals (simplified):
class DoubleCheck<T>(private val provider: Provider<T>) : Provider<T> {
    @Volatile private var instance: Any? = UNINITIALIZED

    override fun get(): T {
        var result = instance
        if (result == UNINITIALIZED) {
            synchronized(this) {
                result = instance
                if (result == UNINITIALIZED) {
                    result = provider.get()
                    instance = result
                }
            }
        }
        @Suppress("UNCHECKED_CAST")
        return result as T
    }
}

6. Advanced Hilt Patterns

Multibindings with @IntoSet / @IntoMap

// Plugin architecture: register interceptors from different modules
@Module
@InstallIn(SingletonComponent::class)
abstract class InterceptorModule {
    @Binds @IntoSet
    abstract fun bindAuthInterceptor(impl: AuthInterceptor): Interceptor

    @Binds @IntoSet
    abstract fun bindLoggingInterceptor(impl: LoggingInterceptor): Interceptor

@Module
@InstallIn(SingletonComponent::class)
object NetworkModule {
    @Provides @Singleton
    fun provideOkHttpClient(
        interceptors: Set<@JvmSuppressWildcards Interceptor>
    ): OkHttpClient = OkHttpClient.Builder()
        .apply { interceptors.forEach { addInterceptor(it) } }
        .build()

@AssistedInject — Runtime + DI Parameters

// Some classes need both DI-provided and runtime-provided parameters
class PaymentProcessor @AssistedInject constructor(
    private val api: PaymentApi,           // from DI
    private val logger: Logger,            // from DI
    @Assisted private val orderId: String  // from caller at runtime
) {
    suspend fun process(): PaymentResult = api.charge(orderId)
}

@AssistedFactory
interface PaymentProcessorFactory {
    fun create(orderId: String): PaymentProcessor
}

// Usage:
@AndroidEntryPoint
class CheckoutFragment : Fragment() {
    @Inject lateinit var processorFactory: PaymentProcessorFactory

    fun onCheckout(orderId: String) {
        val processor = processorFactory.create(orderId)
        // ...
    }
}
Evolution — From Manual Factories to @AssistedInject

Before @AssistedInject (added in Dagger 2.31), mixing DI and runtime parameters required writing a manual factory class: an interface with a create() method, a concrete implementation that took DI params in its constructor and runtime params in create(), and a @Binds binding for the factory interface. This was ~30 lines of boilerplate per class. Square's AssistedInject library solved this first, and Dagger later adopted the pattern natively. The @AssistedFactory annotation generates the factory implementation at compile time.

class SyncWorker @AssistedInject constructor(

Multi-Module Hilt

In a multi-module Gradle project, each module can declare @InstallIn modules independently. Hilt aggregates them at build time using the Gradle plugin:

// :feature:auth module
@Module
@InstallIn(SingletonComponent::class)
abstract class AuthModule {
    @Binds
    abstract fun bindAuthRepo(impl: AuthRepositoryImpl): AuthRepository
}

// :feature:profile module
@Module
@InstallIn(SingletonComponent::class)
abstract class ProfileModule {
    @Binds
    abstract fun bindProfileRepo(impl: ProfileRepositoryImpl): ProfileRepository
}

// :app module — no explicit wiring needed!
// Hilt's Gradle plugin scans all modules and aggregates @InstallIn
// declarations into the generated component at build time.

This works via Hilt's aggregation mechanism: each module generates metadata classes annotated with @OriginatingElement and @InstallIn. The app module's Hilt processor collects all of these and generates the final component.

7. When Dagger 2 Is Better Than Hilt

Hilt is the right choice for most Android apps, but there are cases where raw Dagger 2 is preferable:

ScenarioWhy Dagger 2 Wins
Kotlin MultiplatformHilt depends on Android-specific classes (Application, Activity). KMP shared modules can't use Hilt. Use Dagger 2 or Koin for shared code.
Custom component hierarchiesHilt's hierarchy is fixed to Android lifecycle. If your app needs a UserSessionComponent that outlives Activities but isn't a ViewModel, raw Dagger gives you full control.
Android librariesLibraries shouldn't force Hilt on consumers. Use Dagger 2 internally and expose a manual API. Hilt's @EntryPoint can bridge the gap for Hilt-using consumers.
Massive monoreposHilt's aggregation step touches the app module on every change. In 500+ module projects, this can add minutes to incremental builds. Raw Dagger with component dependencies allows more granular compilation.
Component dependencies over subcomponentsHilt exclusively uses subcomponents. If you need the stronger encapsulation of component dependencies (child only sees explicitly exposed types), use raw Dagger.
Staff-Level Insight — The Build Time Trade-off

In Google's internal monorepo, Hilt's aggregation step was a significant build bottleneck for large apps. The hilt-android-gradle-plugin must process every @InstallIn module across all transitive dependencies to generate the final component. For a 200-module app, this can take 30-60 seconds on every incremental build that touches a module. Google addressed this with Hilt's classpath aggregation (enabled by default since AGP 7.1), which moves aggregation to the classpath level and avoids reprocessing unchanged modules. If you're on an older AGP, set hilt.enableAggregatingTask=true in gradle.properties.

8. Common Pitfalls

Field Injection Misuse

// WRONG: field injection in a regular class
// Dagger can't inject here because it doesn't control construction
class UserMapper {
    @Inject lateinit var formatter: DateFormatter  // never injected!
}

// RIGHT: constructor injection for regular classes
class UserMapper @Inject constructor(
    private val formatter: DateFormatter
)

// Field injection is ONLY for Android entry points where the
// framework controls construction:
@AndroidEntryPoint
class MainActivity : AppCompatActivity() {
    @Inject lateinit var mapper: UserMapper  // OK — Hilt injects this
}

Missing @InstallIn

// Forgetting @InstallIn gives a cryptic error:
// "XxxModule is missing an @InstallIn annotation"
@Module  // Compile error!
object BadModule {
    @Provides fun provideFoo(): Foo = Foo()
}

// Fix:
@Module
@InstallIn(SingletonComponent::class)
object GoodModule {
    @Provides fun provideFoo(): Foo = Foo()
}

Circular Dependencies

// Dagger detects cycles at compile time:
class A @Inject constructor(val b: B)
class B @Inject constructor(val a: A)
// Error: [Dagger/DependencyCycle] Found a dependency cycle:
//   A is injected at B(a)
//   B is injected at A(b)

// Fix: break the cycle with Lazy or Provider
class A @Inject constructor(val b: Lazy<B>)  // B created on first access
class B @Inject constructor(val a: A)

Wrong Scope → Memory Leak or Stale Data

// DANGER: Activity-specific data scoped to Singleton
@Singleton  // Lives forever — holds reference to first Activity's data!
class ScreenState @Inject constructor() {
    var currentUser: User? = null
}

// FIX: scope to the component that matches the data's lifetime
@ActivityScoped
class ScreenState @Inject constructor() {
    var currentUser: User? = null
}

@Provides When @Binds Is Better

// Wasteful: generates a factory class for a simple interface binding
@Module
@InstallIn(SingletonComponent::class)
object WastefulModule {
    @Provides
    fun bindRepo(impl: UserRepositoryImpl): UserRepository = impl
}

// Better: no factory generated, Dagger reuses UserRepositoryImpl_Factory
@Module
@InstallIn(SingletonComponent::class)
abstract class EfficientModule {
    @Binds
    abstract fun bindRepo(impl: UserRepositoryImpl): UserRepository
}

Qualifier Confusion

// Two OkHttpClient instances — how does Dagger know which to inject?
@Module
@InstallIn(SingletonComponent::class)
object NetworkModule {
    @Provides @Named("auth")
    fun provideAuthClient(): OkHttpClient =
        OkHttpClient.Builder().addInterceptor(AuthInterceptor()).build()

    @Provides @Named("public")
    fun providePublicClient(): OkHttpClient =
        OkHttpClient.Builder().build()
}

// Better: custom qualifiers are type-safe (no string typos)
@Qualifier @Retention(AnnotationRetention.BINARY)
annotation class AuthClient

@Qualifier @Retention(AnnotationRetention.BINARY)
annotation class PublicClient

@Module
@InstallIn(SingletonComponent::class)
object NetworkModule {
    @Provides @AuthClient
    fun provideAuthClient(): OkHttpClient = /* ... */

    @Provides @PublicClient
    fun providePublicClient(): OkHttpClient = /* ... */
}

// Injection site:
class ApiService @Inject constructor(
    @AuthClient private val authClient: OkHttpClient,
    @PublicClient private val publicClient: OkHttpClient
)
Warning — @Named vs Custom Qualifiers

@Named("auth") uses a string key — a typo like @Named("Auth") silently creates a different binding and you get a missing binding error. Custom qualifier annotations (@AuthClient) are checked at compile time. Always prefer custom qualifiers in production code. Reserve @Named for quick prototyping only.
