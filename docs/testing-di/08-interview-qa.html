58 L6 Staff Engineer questions covering DI fundamentals, Dagger/Hilt internals, Koin & framework comparison, testing fundamentals, coroutine/Flow testing, Android component testing, Hilt testing, and system design

DI Fundamentals (Q1–Q8)

Q1. What is the difference between Dependency Injection and Service Locator? Which does Koin actually implement?

Dependency Injection pushes dependencies into a class from the outside — the class declares what it needs (via constructor parameters) and an external entity provides them. The class has zero knowledge of where dependencies come from. Service Locator is a pull pattern — the class actively asks a global registry for its dependencies at runtime (e.g., ServiceLocator.get<UserRepository>()). The key distinction is control flow direction: DI inverts control (the framework calls you), while Service Locator keeps control with the consumer. Koin is technically a Service Locator — when you call inject() or get() inside an Activity, the class is actively pulling from Koin's global registry. However, Koin also supports constructor injection in its DSL (factory { UserRepo(get()) }), where the resolution happens at the module definition level. Martin Fowler's original distinction matters at L6: Service Locator makes dependencies implicit (harder to trace), while true DI makes them explicit in the constructor signature. In practice, Koin's constructor DSL gives you most of DI's testability benefits while retaining Service Locator's runtime resolution model.

Q2. Explain the Dependency Inversion Principle and how DI enforces it.

The Dependency Inversion Principle (the "D" in SOLID) states: (1) high-level modules should not depend on low-level modules — both should depend on abstractions, and (2) abstractions should not depend on details — details should depend on abstractions. Without DI, a PaymentViewModel might directly instantiate StripePaymentGateway, creating a compile-time dependency from the high-level policy layer to a low-level implementation detail. DI enforces the principle by requiring the ViewModel to depend on an interface PaymentGateway and having the DI framework bind the concrete implementation. The wiring happens in the composition root — the single place (Hilt module, Koin module, or manual factory) where abstractions are mapped to implementations. This is not just academic: in Android, it means your :feature:checkout module depends on :core:payment-api (interfaces only), not on :impl:stripe. The implementation module is only known at the app level. This enables parallel team development, faster incremental builds (changing Stripe impl doesn't recompile checkout), and trivial test substitution.

Q3. Constructor injection vs field injection vs method injection — when to use each in Android?

Constructor injection is the gold standard: dependencies are passed as constructor parameters, making them explicit, immutable (val), and impossible to forget. The object is fully initialized after construction — no partially-constructed states. Use it for all classes you control: ViewModels, repositories, use cases, data sources. Field injection (Dagger's @Inject lateinit var) is necessary when you don't control construction — Activities, Fragments, Services, and BroadcastReceivers are instantiated by the Android framework via reflection. Hilt's @AndroidEntryPoint generates a base class that calls inject() in onCreate(), populating @Inject-annotated fields. The downside: fields must be lateinit var (mutable, nullable before injection), and the dependency graph is invisible from the class signature. Method injection (@Inject fun setDeps(repo: Repo)) is rare in Android — it's useful when you need to inject after construction but before a specific lifecycle point, or when a dependency is optional. In practice, 90% of Android DI is constructor injection for business logic and field injection for framework components.

Q4. Why is Activity/Fragment field injection necessary? Why can't we use constructor injection?

Activities and Fragments are instantiated by the Android framework, not by your code. The system uses reflection — Instrumentation.newActivity() calls Class.newInstance() (the no-arg constructor) for Activities. Fragments require a no-arg constructor so the system can recreate them after process death via FragmentFactory.instantiate() which also uses Class.newInstance(). If you add constructor parameters, the system throws InstantiationException. This is why Hilt uses field injection: @AndroidEntryPoint generates a Hilt_MyActivity superclass that overrides onCreate() to call the generated component's inject(this) method, which sets all @Inject lateinit var fields via generated MembersInjector code. For Fragments specifically, FragmentFactory (Jetpack) offers an escape hatch — you can register a custom factory that does use constructor injection. Hilt doesn't use this approach because it would require replacing the factory in every FragmentManager, including child fragment managers, and it doesn't work for Fragments restored from saved state in older AndroidX versions.

Q5. Explain DI scopes in Android — how do they map to component lifecycles?

A scope in DI means "one instance per lifetime of this component." Hilt's component hierarchy maps directly to Android lifecycles: @Singleton → SingletonComponent (Application lifetime — one instance for the entire process), @ActivityRetainedScoped → ActivityRetainedComponent (survives configuration changes, tied to ViewModel store — destroyed only on Activity finish), @ActivityScoped → ActivityComponent (one per Activity instance — destroyed on rotation), @ViewModelScoped → ViewModelComponent (one per ViewModel instance — survives config changes), @FragmentScoped → FragmentComponent (one per Fragment instance). Each child component inherits bindings from its parent — a @FragmentScoped binding can inject @Singleton dependencies but not vice versa. The critical nuance: @ActivityRetainedScoped uses ViewModel internally to survive configuration changes, so it has the same lifecycle as ViewModelStoreOwner. Unscoped bindings (no annotation) create a new instance on every injection — this is the default and is correct for stateless objects like mappers or validators. Over-scoping is a common mistake: making everything @Singleton wastes memory and creates hidden state coupling.

Q6. What is the "scope mismatch" problem? Give an example of a memory leak caused by incorrect scoping.

Scope mismatch occurs when a longer-lived scope holds a reference to a shorter-lived object. Classic example: a @Singleton-scoped EventBus that accepts listener registrations, and an @ActivityScoped presenter registers itself as a listener. The EventBus lives for the entire process, so it holds a reference to the presenter, which holds a reference to the Activity's Context — the Activity can never be garbage collected. In Dagger/Hilt, the type system partially prevents this: you cannot inject an @ActivityScoped binding into a @Singleton component because the Activity component is a child of the Singleton component, not the other way around. However, the protection breaks down with indirect references: a @Singleton cache that stores Any objects, and someone puts an Activity-scoped object into it. Another subtle case: @Singleton-scoped SharedPreferences.OnSharedPreferenceChangeListener that captures an Activity in its lambda closure. The fix is always the same: the shorter-lived object must unregister, or use WeakReference, or restructure so the longer-lived scope only holds scope-appropriate references.

Q7. When should you NOT use a DI framework?

DI frameworks add complexity that isn't always justified. Skip them when: (1) Small apps or libraries — a library SDK should not force Hilt/Koin on consumers; use manual constructor injection with a simple factory or builder pattern. (2) Kotlin Multiplatform shared modules — Hilt is Android-only; Koin works but adds a runtime dependency. Manual DI with expect/actual or a simple factory is cleaner. (3) Performance-critical paths — Koin's runtime resolution adds microseconds per get() call; in a tight loop or hot path, direct construction is faster. Dagger/Hilt have zero runtime overhead after compilation, so this concern is Koin-specific. (4) Prototype/hackathon code — the ceremony of modules, components, and scopes slows initial development. (5) When the team doesn't understand it — a poorly configured Dagger graph with incorrect scopes causes worse bugs than no DI at all. The alternative is always manual constructor injection — pass dependencies through constructors, wire them in a CompositionRoot class. You get testability without framework coupling. Google's "Guide to app architecture" explicitly shows manual DI as a valid approach for small-to-medium apps.

Q8. How does DI improve testability? Show a concrete example.

Without DI, a class creates its own dependencies, making them impossible to replace in tests:

class OrderViewModel {
    private val repo = OrderRepository(RetrofitClient.api, RoomDb.instance.orderDao())
    fun placeOrder(order: Order) = viewModelScope.launch {
        repo.submit(order) // hits real network + database
    }
}

You cannot test placeOrder() without a running server and database. With DI:

class OrderViewModel @Inject constructor(
    private val repo: OrderRepository
) : ViewModel() {
    fun placeOrder(order: Order) = viewModelScope.launch { repo.submit(order) }
}

// In test:
val fakeRepo = FakeOrderRepository()
val vm = OrderViewModel(fakeRepo)
vm.placeOrder(testOrder)
assertThat(fakeRepo.submittedOrders).containsExactly(testOrder)

The ViewModel now depends on an abstraction passed via constructor. Tests supply a FakeOrderRepository that records calls in memory — no network, no database, deterministic, fast. This is the fundamental value proposition: DI separates object creation from object use, and tests exploit that separation by providing controlled implementations. At L6 scale, this extends to replacing entire subsystems: swap the real analytics SDK with a fake that asserts event names, swap the feature flag service with one that returns deterministic values, swap the auth token provider with one that returns expired tokens to test refresh logic.

Dagger & Hilt (Q9–Q20)

Q9. How does Dagger 2 generate code at compile time? Walk through the annotation processing pipeline.

Dagger uses JSR-269 annotation processing (or KSP since Dagger 2.48+). The pipeline: (1) The compiler invokes Dagger's ComponentProcessor, which scans for @Component, @Module, @Inject, and @Provides annotations. (2) Dagger builds a binding graph — a directed acyclic graph where nodes are types and edges are "provides" relationships. Each @Inject constructor registers a binding; each @Provides/@Binds method registers a binding. (3) Dagger validates the graph: checks for missing bindings, duplicate bindings, scope violations, and cycles. If validation fails, you get a compile error with the full dependency chain. (4) For each @Component, Dagger generates a DaggerXxxComponent class implementing the component interface. This class contains Provider<T> fields for each binding — scoped bindings use DoubleCheck.provider() (lazy singleton with double-checked locking), unscoped bindings use direct Provider lambdas. (5) For each @Inject constructor, Dagger generates a XxxFactory implementing Provider<T>. (6) For field injection targets, Dagger generates Xxx_MembersInjector classes. The generated code is plain Java/Kotlin with no reflection — it's as fast as hand-written factory code.

Q10. Explain @Provides vs @Binds — when to use which and why @Binds is more efficient.

@Provides is a concrete method in a @Module that creates and returns an instance. Dagger generates a Factory that calls this method. @Binds is an abstract method that tells Dagger "when someone asks for this interface, give them this implementation" — it must take exactly one parameter (the implementation) and return the interface type. @Binds is more efficient because Dagger doesn't generate a separate Factory class for it — it reuses the implementation's existing Factory directly. With @Provides, Dagger generates: (1) a ModuleName_ProvideXxxFactory class, (2) a method call that instantiates the module and calls the provides method. With @Binds, Dagger simply aliases the binding — no extra class, no extra method call. In a large app with hundreds of bindings, this reduces generated code size and method count. Use @Binds whenever you're mapping an interface to an implementation that already has an @Inject constructor. Use @Provides when you need custom creation logic, third-party classes without @Inject, or builder/factory patterns (e.g., Retrofit.Builder().build()).

Q11. What is the Hilt component hierarchy? Draw it and explain the lifecycle of each component.

Hilt's component hierarchy mirrors Android's lifecycle nesting:

SingletonComponent (@Singleton)                    ← Application
  └─ ActivityRetainedComponent (@ActivityRetainedScoped)  ← survives config change
       └─ ActivityComponent (@ActivityScoped)              ← Activity instance
            ├─ FragmentComponent (@FragmentScoped)          ← Fragment instance
            └─ ViewComponent (@ViewScoped)                  ← View instance
                 └─ ViewWithFragmentComponent               ← View in Fragment
       └─ ViewModelComponent (@ViewModelScoped)            ← ViewModel instance
  └─ ServiceComponent (@ServiceScoped)                     ← Service instance

SingletonComponent is created when the Application starts and destroyed when the process dies. ActivityRetainedComponent is backed by a ViewModel internally — it survives rotation but is destroyed on finish(). ActivityComponent is created in Activity.onCreate() and destroyed in onDestroy() — a new one is created on every configuration change. ViewModelComponent is created per-ViewModel instance and destroyed when the ViewModel is cleared. Each child component can access all bindings from its parent components. The hierarchy is enforced at compile time: you cannot inject an @ActivityScoped binding into SingletonComponent because the Activity component is a subcomponent of Singleton, not the other way around.

Q12. How does @HiltViewModel work internally? How does Hilt inject SavedStateHandle?

@HiltViewModel triggers Hilt's code generation to create a custom ViewModelProvider.Factory. At compile time, Hilt generates a ViewModelComponent subcomponent for each @HiltViewModel class. At runtime, when ViewModelProvider requests the ViewModel, Hilt's HiltViewModelFactory intercepts the creation. It creates a ViewModelComponent instance, which has access to two special bindings: SavedStateHandle and ViewModelLifecycle. The SavedStateHandle is obtained from SavedStateHandleSupport — Hilt's factory implements CreationExtras-aware creation (AndroidX Lifecycle 2.5+), extracting the SavedStateHandle from CreationExtras which the Activity/Fragment provides via defaultViewModelCreationExtras. Before CreationExtras, Hilt used AbstractSavedStateViewModelFactory which obtained the handle from SavedStateRegistryOwner. The generated factory calls the ViewModel's @Inject constructor with all resolved dependencies, including SavedStateHandle. This is why SavedStateHandle "just works" in Hilt ViewModels without any module — it's a built-in binding of ViewModelComponent.

Q13. Explain @EntryPoint — when do you need it and how does it work?

@EntryPoint provides access to the Hilt dependency graph from code that Hilt doesn't manage. You need it for: (1) ContentProviders — they initialize before Application.onCreate(), so @AndroidEntryPoint doesn't work. (2) Third-party libraries that create objects via their own factories (e.g., WorkManager's ListenableWorker before @HiltWorker existed). (3) Dagger components from non-Hilt modules in a migration. (4) Custom Views that need dependencies but aren't annotated with @AndroidEntryPoint. You define an @EntryPoint interface inside the class that needs access, annotated with @InstallIn(SingletonComponent::class). At runtime, you call EntryPointAccessors.fromApplication(context, MyEntryPoint::class.java), which casts the Application's Hilt component to your interface and returns it. Under the hood, the generated SingletonComponent implements all @EntryPoint interfaces, so the cast succeeds. The entry point interface methods are essentially component provision methods — Dagger generates implementations that return the bound instances.

Q14. What is @AssistedInject and when would you use it? How does it differ from @Provides with runtime parameters?

@AssistedInject handles the case where a class needs both DI-provided dependencies and runtime parameters that aren't known at graph construction time. Example: a PaymentProcessor needs a DI-injected PaymentGateway but also a runtime orderId: String. You annotate the constructor with @AssistedInject, mark runtime params with @Assisted, and define an @AssistedFactory interface:

class PaymentProcessor @AssistedInject constructor(
    private val gateway: PaymentGateway,
    @Assisted private val orderId: String
)

@AssistedFactory
interface PaymentProcessorFactory {
    fun create(orderId: String): PaymentProcessor
}

Dagger generates the factory implementation that injects PaymentGateway from the graph and passes orderId from the caller. The alternative — a @Provides method — doesn't work because @Provides runs at injection time with no way to pass runtime parameters. You'd have to create the object manually and lose DI for its other dependencies. @AssistedInject is commonly used for: Workers (@HiltWorker uses it internally for Context and WorkerParameters), dialog/bottom-sheet presenters with runtime arguments, and any object whose identity depends on runtime data.

Q15. How does Hilt handle multi-module projects? What is module aggregation?

In a multi-module Gradle project, each module defines its own @Modules and @InstallIn annotations independently. Hilt uses module aggregation at the app module level to discover and combine all modules from all transitive dependencies. The mechanism: each @InstallIn module generates a metadata class annotated with @HiltModules (or similar marker). At app compilation, Hilt's processor in the :app module scans the classpath for all these metadata markers and aggregates them into the generated component. Pre-Hilt 2.40, aggregation used a Gradle transform (HiltTransformClassesTask) that scanned bytecode. Post-2.40 with aggregating KSP/KAPT, it uses classpath scanning during annotation processing. The key constraint: @Module @InstallIn(SingletonComponent::class) in :feature:auth is automatically included in the app's SingletonComponent — no manual wiring needed. This is Hilt's killer feature over raw Dagger for multi-module: with raw Dagger, you'd manually list every module in the component definition, creating a dependency from :app to every feature module. Hilt's aggregation inverts this — feature modules declare their own bindings, and the app module discovers them automatically.

Q16. Explain the difference between Dagger subcomponents and component dependencies.

Subcomponents are child components that have full access to their parent's bindings. The parent component declares a factory method or @Subcomponent.Factory, and the subcomponent can inject any binding from the parent without explicit exposure. Internally, the generated subcomponent holds a reference to the parent component instance. Component dependencies use the @Component(dependencies = [ParentComponent::class]) syntax. The child can only access bindings that the parent explicitly exposes via provision methods on the component interface. If ParentComponent doesn't have a fun userRepo(): UserRepository method, the child can't access it — even if it's bound in the parent. Subcomponents are tighter coupling (child knows parent's full graph), component dependencies are looser (explicit API surface). Hilt exclusively uses subcomponents — each level in the hierarchy (ActivityComponent, FragmentComponent, etc.) is a subcomponent of its parent. This is why any @Singleton binding is automatically available in @ActivityScoped code. For multi-module architectures without Hilt, component dependencies are often preferred because they enforce explicit API boundaries between modules.

Q17. What happens if you have a circular dependency in Dagger? How do you fix it?

Dagger detects circular dependencies at compile time and fails with an error like: Found a dependency cycle: A → B → A. This is a major advantage over runtime DI frameworks that would stack-overflow at runtime. To fix it: (1) Lazy<T> or Provider<T> — inject Lazy<B> in A instead of B directly. Dagger breaks the cycle by deferring B's creation until lazy.get() is called. This works when only one side of the cycle needs the other during construction. (2) Redesign — circular dependencies usually indicate a design problem. Extract the shared logic into a third class that both A and B depend on, breaking the cycle. For example, if AuthManager depends on NetworkClient and NetworkClient depends on AuthManager (for token refresh), extract a TokenProvider interface that NetworkClient depends on and AuthManager implements. (3) Method injection — use @Inject fun setB(b: B) on A, which runs after construction. This is the least clean option. At L6, the answer is almost always option 2 — circular dependencies are a code smell indicating violated single-responsibility.

Q18. How does Hilt's @AndroidEntryPoint work under the hood? What does the bytecode transform do?

When you annotate class MyActivity : AppCompatActivity() with @AndroidEntryPoint, Hilt's Gradle transform (or bytecode transformation via ASM) modifies the class hierarchy. It generates a Hilt_MyActivity class that extends AppCompatActivity, and rewrites MyActivity to extend Hilt_MyActivity instead. The generated base class overrides onCreate() to: (1) obtain the parent component (e.g., ActivityRetainedComponent from the Application), (2) create the ActivityComponent subcomponent, (3) call inject(this) on the component, which invokes the generated MembersInjector to set all @Inject fields. For Fragments, the transform is similar but injection happens in onAttach(). The bytecode transform is necessary because Java/Kotlin doesn't support changing a class's superclass via annotation processing alone — APT can only generate new classes, not modify existing ones. This is why Hilt requires the hilt-android-gradle-plugin — it registers the ASM transform. With AGP 7.2+, Hilt also supports local test mode where the transform is skipped and a different code path uses reflection-based injection for Robolectric tests.

Q19. When would you choose raw Dagger 2 over Hilt?

Choose raw Dagger when: (1) Non-Android targets — Dagger works in any JVM project (backend, CLI tools, Kotlin Multiplatform JVM target). Hilt is Android-specific. (2) Custom component hierarchies — Hilt enforces its fixed hierarchy (Singleton → ActivityRetained → Activity → Fragment). If you need a custom scope like @UserSessionScoped that spans multiple Activities but isn't tied to the Application, raw Dagger lets you define arbitrary component relationships. (3) Library SDKs — libraries shouldn't force Hilt on consumers. Use raw Dagger internally or manual DI. (4) Gradle plugin restrictions — some build systems or CI environments can't use Hilt's Gradle plugin. (5) Fine-grained control over component creation — Hilt creates components automatically at lifecycle events; with raw Dagger, you control exactly when components are created and destroyed. The trade-off is significant boilerplate: with raw Dagger, you manually wire every module into every component, handle Activity/Fragment injection yourself, and manage the component lifecycle. For most Android apps, Hilt's conventions eliminate 60-80% of Dagger boilerplate.

Q20. Explain multibindings (@IntoSet, @IntoMap) — design a plugin architecture using them.

Multibindings let multiple modules contribute to a single Set<T> or Map<K, V> without knowing about each other. @IntoSet adds a binding to a Set; @IntoMap adds a key-value pair to a Map (with @StringKey, @ClassKey, or custom key annotations). Plugin architecture example:

interface FeatureInitializer {
    fun initialize(app: Application)
    val priority: Int
}

// In :feature:analytics module
@Module @InstallIn(SingletonComponent::class)
object AnalyticsModule {
    @Provides @IntoSet
    fun provideAnalyticsInit(analytics: AnalyticsService): FeatureInitializer =
        object : FeatureInitializer {
            override fun initialize(app: Application) = analytics.init(app)
            override val priority = 1
        }
}

// In Application
@AndroidEntryPoint
class MyApp : Application() {
    @Inject lateinit var initializers: Set<@JvmSuppressWildcards FeatureInitializer>
    override fun onCreate() {
        super.onCreate()
        initializers.sortedBy { it.priority }.forEach { it.initialize(this) }
    }
}

Each feature module contributes its initializer independently. The app module doesn't know which features exist — it just iterates the set. @IntoMap is even more powerful for routing: map screen names to Fragment factories, map intent actions to handlers, or map analytics event types to processors. The @JvmSuppressWildcards annotation is needed because Kotlin generics produce wildcard types that Dagger can't match.

Koin & Framework Comparison (Q21–Q26)

Q21. How does Koin resolve dependencies at runtime? What data structure does it use?

Koin maintains a global registry called InstanceRegistry backed by a ConcurrentHashMap<IndexKey, InstanceFactory<*>>. The IndexKey is a combination of the Kotlin KClass and an optional qualifier (named string or custom qualifier). When you call get<UserRepository>(), Koin: (1) computes the IndexKey from the reified type parameter, (2) looks up the InstanceFactory in the map, (3) if it's a single { } definition, returns the cached instance (double-checked locking via synchronized); if it's a factory { }, invokes the lambda to create a new instance. Dependencies within the lambda are resolved recursively — factory { UserRepo(get(), get()) } triggers two more get() calls. The resolution is entirely at runtime with no compile-time validation. If a binding is missing, you get a NoBeanDefFoundException at the call site — which could be deep in a user flow, not at app startup. Koin 3.2+ added module includes for better organization, and Koin 3.3+ added verify() which performs a dry-run resolution of all definitions at test time, catching missing bindings before production.

Q22. What are the risks of runtime DI resolution at scale? How does Koin 3.3+ Verify help?

Runtime resolution risks: (1) Silent failures — a missing binding only crashes when that code path executes. If a rarely-used screen injects a dependency that was removed in a refactor, it passes all common tests and crashes in production. (2) Type erasure — Koin uses KClass for lookup, so List<String> and List<Int> resolve to the same key (List::class). You must use qualifiers to distinguish them, and forgetting to do so causes silent wrong-type injection. (3) Startup performance — in a 200-module app with 500+ definitions, Koin's module loading and lambda allocation at startup adds measurable time (10-50ms). (4) Refactoring fragility — renaming a class doesn't produce a compile error if the Koin module still references the old name via a lambda. Koin 3.3+ verify() mitigates risk (1) by iterating all registered definitions and attempting to resolve their dependency chains in a test. It catches missing bindings and circular dependencies. However, it doesn't catch type-erasure issues, scoping errors, or bindings that depend on Android runtime state (Context, Activity). It's a significant improvement but still weaker than Dagger's compile-time graph validation.

Q23. Compare Hilt vs Koin for a 50-module app — build time, runtime safety, developer experience.

Build time: Hilt adds 10-30 seconds to a clean build due to annotation processing (KAPT) or KSP. Incremental builds are fast if bindings don't change. Koin has zero build-time overhead — it's just Kotlin lambdas. However, Hilt's KSP backend (Dagger 2.48+) is 2x faster than KAPT. Runtime safety: Hilt validates the entire dependency graph at compile time — missing bindings, scope violations, and duplicate bindings are compile errors. Koin validates nothing until runtime (unless you run verify() in tests). In a 50-module app with hundreds of bindings, Hilt's compile-time safety prevents entire categories of production crashes. Developer experience: Koin's DSL is more Kotlin-idiomatic and has a gentler learning curve. Hilt requires understanding annotations, generated code, and the component hierarchy. Koin's error messages at runtime are clear but come too late. Hilt's compile errors can be cryptic (especially with missing bindings in transitive modules) but come at the right time. Verdict for 50-module app: Hilt wins on safety and scales better. The build-time cost is amortized by the time saved not debugging runtime DI failures. Koin is better for small teams, rapid prototyping, or KMP projects.

Q24. What is kotlin-inject and when would you consider it over Hilt?

kotlin-inject is a compile-time DI framework written in pure Kotlin, using KSP for code generation. It's conceptually similar to Dagger but designed for Kotlin from the ground up — it uses @Inject annotations, @Component interfaces, and @Provides functions, but with Kotlin-native features like @Scope as a typealias and multi-platform support. Consider it over Hilt when: (1) Kotlin Multiplatform — kotlin-inject works on JVM, iOS, JS, and Native targets. Hilt is Android-only. (2) Pure Kotlin modules — if your business logic is in platform-agnostic Kotlin modules, kotlin-inject provides compile-time safety without Android dependencies. (3) Faster KSP processing — kotlin-inject's KSP processor is lighter than Dagger's, resulting in faster builds for pure Kotlin modules. (4) Simpler mental model — no bytecode transforms, no Gradle plugin, no component hierarchy imposed by the framework. The trade-off: kotlin-inject has no Android-specific integration (no @AndroidEntryPoint, no @HiltViewModel, no automatic component lifecycle management). You'd pair it with manual injection in Android entry points. For a KMP project with shared business logic, kotlin-inject for shared code + Hilt for Android-specific wiring is a viable architecture.

Q25. You're starting a new Kotlin Multiplatform project — which DI framework do you choose and why?

For KMP, the viable options are: Koin, kotlin-inject, or manual DI. Hilt is eliminated because it's Android-only. Koin is the most popular choice for KMP — it's pure Kotlin, works on all targets (JVM, iOS, JS, Native), and its DSL is simple. The risk is runtime resolution, but verify() in tests mitigates it. kotlin-inject provides compile-time safety on all KMP targets via KSP, but its ecosystem is smaller and Android integration requires more manual work. Manual DI (constructor injection + factory classes) is the simplest and has zero dependencies — it works everywhere, is fully type-safe, and has no framework learning curve. My recommendation for a new KMP project: manual DI for the shared module (keep it dependency-free, use a simple SharedDependencies factory class), and Hilt on Android / manual DI on iOS for platform-specific wiring. This avoids forcing a DI framework on the shared code, keeps the shared module lightweight, and lets each platform use its preferred patterns. If the shared module grows beyond 50+ classes with complex graphs, migrate to kotlin-inject for compile-time safety.

Q26. Design a migration strategy from Koin to Hilt for a medium-sized app.

Phase 1 — Preparation (1-2 weeks): Add Hilt Gradle plugin and dependencies alongside Koin. Annotate Application with @HiltAndroidApp. Create empty Hilt modules that mirror Koin modules. Ensure all classes use constructor injection (refactor any that call get() directly). Phase 2 — Dual-wiring (2-4 weeks): For each Koin module, create the equivalent Hilt @Module with @Provides/@Binds. Use a feature flag to switch between Koin and Hilt at the Application level. Migrate one feature module at a time — annotate its Activities/Fragments with @AndroidEntryPoint, replace by inject() with @Inject lateinit var, and remove the corresponding Koin definitions. Phase 3 — Validation (1 week): Run the full test suite with Hilt enabled. Use @HiltAndroidTest for instrumented tests. Compare crash rates between Koin and Hilt builds via staged rollout. Phase 4 — Cleanup (1 week): Remove all Koin dependencies, delete Koin modules, remove the feature flag. Key risks: (1) Koin's scope { } definitions don't map 1:1 to Hilt scopes — audit each. (2) Koin's named() qualifiers must become @Named or custom qualifier annotations. (3) Test modules need complete rewrite from Koin's declareMock to Hilt's @UninstallModules/@BindValue.

Testing Fundamentals (Q27–Q34)

Q27. Explain the test pyramid for Android — what goes in each layer and why.

The test pyramid has three layers. Unit tests (base, ~70%): fast, run on JVM, test individual classes in isolation. Cover ViewModels, repositories, use cases, mappers, validators, and business logic. Use fakes/mocks for dependencies. Run in milliseconds. Integration tests (middle, ~20%): test interactions between components. Room DAO tests with in-memory database, Navigation tests with TestNavHostController, Repository tests with MockWebServer, ViewModel + Repository integration. Can run on JVM (Robolectric) or device. End-to-end / UI tests (top, ~10%): test full user flows through the real app. Espresso or Compose Testing with real Activities, real navigation, real (or fake) backend. Run on device/emulator, slow (seconds per test). The pyramid shape matters because cost increases upward: unit tests are cheap to write, fast to run, and easy to debug. E2E tests are expensive, slow, and flaky. A common anti-pattern is the "ice cream cone" — mostly E2E tests, few unit tests — which results in slow CI, flaky builds, and hard-to-diagnose failures. At L6, you design the architecture to maximize the unit test layer: clean separation of concerns means most logic is testable without Android framework dependencies.

Q28. Fake vs Mock vs Stub — when to use each? Why does Google prefer fakes?

A stub returns hardcoded values — fun getUser() = User("test"). It doesn't verify interactions. A mock records interactions and verifies them — verify(repo).save(any()). It tests how the code interacts with dependencies. A fake is a working implementation with shortcuts — FakeUserRepository backed by a MutableList instead of a real database. It behaves like the real thing but is faster and deterministic. Google prefers fakes because: (1) Tests verify behavior, not implementation — mock-heavy tests break when you refactor internals even if behavior is unchanged. (2) Fakes are reusable — one FakeUserRepository serves all tests, while mocks are configured per-test. (3) Fakes catch integration bugs — a fake that maintains state (add user, then query) catches bugs that a mock returning hardcoded values would miss. (4) Fakes are readable — test setup is minimal (val repo = FakeUserRepository()) vs. mock setup (whenever(repo.getUser(any())).thenReturn(user)). The trade-off: fakes require maintenance — when the interface changes, the fake must be updated. At Google scale, each team maintains fakes for their public APIs, and these fakes are tested themselves.

Q29. Compare MockK vs Mockito-Kotlin — when would you choose each?

Mockito-Kotlin is a Kotlin wrapper around Mockito. It works well for most cases but has limitations: it can't mock final classes by default (Kotlin classes are final) — you need the mockito-inline artifact or MockitoExtension with mock-maker-inline. It can't mock object declarations, top-level functions, or extension functions. MockK is built for Kotlin from scratch. It mocks final classes natively, supports object mocking (mockkObject), static/top-level function mocking (mockkStatic), extension function mocking, coroutine support (coEvery, coVerify), and has a more Kotlin-idiomatic DSL. Choose Mockito-Kotlin when: the team is already familiar with Mockito, you're in a mixed Java/Kotlin codebase, or you want lighter test dependencies (Mockito is smaller). Choose MockK when: you're in a pure Kotlin codebase, you need to mock Kotlin-specific constructs (objects, extensions, coroutines), or you want a more expressive DSL. At L6, the deeper answer is: prefer fakes over both. Use mocks sparingly for verifying interactions with side-effecting dependencies (analytics, logging) where a fake would be overkill.

Q30. How do you test a class that depends on Dispatchers.IO? What is the dispatcher injection pattern?

Hardcoding Dispatchers.IO makes tests non-deterministic and slow (real threading). The dispatcher injection pattern replaces hardcoded dispatchers with injected ones:

class UserRepository(
    private val api: UserApi,
    private val ioDispatcher: CoroutineDispatcher = Dispatchers.IO
) {
    suspend fun getUser(id: String): User = withContext(ioDispatcher) {
        api.fetchUser(id)
    }
}

// In test:
@Test
fun `getUser returns mapped user`() = runTest {
    val repo = UserRepository(fakeApi, StandardTestDispatcher(testScheduler))
    val user = repo.getUser("123")
    assertThat(user.name).isEqualTo("Test")
}

In production, Hilt provides the real dispatcher; in tests, you inject StandardTestDispatcher which uses virtual time. The pattern extends to a DispatcherProvider interface with io, main, default, and unconfined properties, bound via a Hilt module. An alternative is Dispatchers.setMain() for Dispatchers.Main only (via kotlinx-coroutines-test), but this doesn't work for IO or Default. The injection pattern is strictly superior because it's explicit, works for all dispatchers, and doesn't rely on global mutable state.

Q31. Explain runTest — how does it manage virtual time? What is StandardTestDispatcher vs UnconfinedTestDispatcher?

runTest (from kotlinx-coroutines-test) creates a TestScope with a TestCoroutineScheduler that controls virtual time. Instead of waiting real milliseconds for delay(1000), the scheduler advances its virtual clock instantly. runTest automatically advances time until all coroutines complete, and fails if coroutines are still running after 60 seconds of virtual time (configurable). StandardTestDispatcher queues coroutines without executing them immediately — you must call advanceUntilIdle(), advanceTimeBy(), or runCurrent() to execute them. This gives you precise control over execution order and timing. UnconfinedTestDispatcher executes coroutines eagerly — as soon as they're launched, they run to the first suspension point. This is simpler for tests that don't care about timing but can cause unexpected ordering. runTest uses StandardTestDispatcher by default. Key nuance: runTest calls advanceUntilIdle() at the end of the test block, so with StandardTestDispatcher, coroutines launched inside the block do complete — but coroutines launched in child scopes (like viewModelScope) need the dispatcher to be shared via injection.

Q32. How do you test a ViewModel that uses viewModelScope? Show the complete setup.

viewModelScope uses Dispatchers.Main.immediate internally. In a JVM test, there's no Main dispatcher, so you must install one:

class SearchViewModelTest {
    @get:Rule
    val mainDispatcherRule = MainDispatcherRule()

    private lateinit var viewModel: SearchViewModel
    private val fakeRepo = FakeSearchRepository()

    @Before
    fun setup() {
        viewModel = SearchViewModel(fakeRepo)
    }

    @Test
    fun `search emits results`() = runTest {
        fakeRepo.setResults(listOf("Kotlin", "Compose"))
        viewModel.search("Ko")
        advanceUntilIdle()
        assertThat(viewModel.results.value).containsExactly("Kotlin", "Compose")
    }
}

class MainDispatcherRule(
    private val dispatcher: TestDispatcher = UnconfinedTestDispatcher()
) : TestWatcher() {
    override fun starting(description: Description) {
        Dispatchers.setMain(dispatcher)
    }
    override fun finished(description: Description) {
        Dispatchers.resetMain()
    }
}

MainDispatcherRule replaces Dispatchers.Main with a test dispatcher before each test and resets it after. Using UnconfinedTestDispatcher for Main means viewModelScope.launch executes eagerly, simplifying assertions. If the ViewModel also uses Dispatchers.IO, inject it via constructor so the test can provide StandardTestDispatcher(testScheduler) sharing the same scheduler as runTest.

Q33. What is Turbine and why is it better than manual Flow collection in tests?

Turbine is a testing library for Kotlin Flows that provides a structured API for asserting emissions. Without Turbine, testing a Flow requires launching a coroutine to collect, storing emissions in a list, and manually managing timing:

// Without Turbine — fragile and verbose
val emissions = mutableListOf<UiState>()
val job = launch { viewModel.state.collect { emissions.add(it) } }
advanceUntilIdle()
assertThat(emissions[0]).isEqualTo(UiState.Loading)
assertThat(emissions[1]).isEqualTo(UiState.Success(data))
job.cancel()

With Turbine:

viewModel.state.test {
    assertThat(awaitItem()).isEqualTo(UiState.Loading)
    assertThat(awaitItem()).isEqualTo(UiState.Success(data))
    cancelAndIgnoreRemainingEvents()
}

Turbine is better because: (1) awaitItem() suspends until the next emission with a configurable timeout — no manual advanceUntilIdle(). (2) It fails fast if an unexpected item arrives or the Flow completes prematurely. (3) awaitError() asserts Flow errors cleanly. (4) expectNoEvents() verifies no emissions within a timeout — critical for testing debounce. (5) It handles StateFlow's initial value correctly. (6) cancelAndConsumeRemainingEvents() prevents test leaks. Turbine integrates with runTest's virtual time, so awaitItem() advances the test scheduler automatically.

Q34. How do you test a debounce operator in a Flow? Show the code.

Testing debounce requires controlling virtual time precisely:

class SearchViewModel(private val repo: SearchRepository) : ViewModel() {
    private val _query = MutableStateFlow("")
    val results: StateFlow<List<String>> = _query
        .debounce(300)
        .filter { it.length >= 2 }
        .flatMapLatest { repo.search(it) }
        .stateIn(viewModelScope, SharingStarted.Lazily, emptyList())
}

@Test
fun `debounce waits 300ms before searching`() = runTest {
    val fakeRepo = FakeSearchRepository()
    val vm = SearchViewModel(fakeRepo)

    vm.setQuery("Ko")
    advanceTimeBy(200) // only 200ms elapsed
    assertThat(fakeRepo.searchCount).isEqualTo(0) // not yet

    advanceTimeBy(150) // now 350ms total, past debounce
    assertThat(fakeRepo.searchCount).isEqualTo(1)
}

@Test
fun `rapid typing only triggers one search`() = runTest {
    val fakeRepo = FakeSearchRepository()
    val vm = SearchViewModel(fakeRepo)

    vm.setQuery("K")
    advanceTimeBy(100)
    vm.setQuery("Ko")
    advanceTimeBy(100)
    vm.setQuery("Kot")
    advanceTimeBy(400) // 400ms after last input

    assertThat(fakeRepo.searchCount).isEqualTo(1)
    assertThat(fakeRepo.lastQuery).isEqualTo("Kot")
}

The key is that runTest's StandardTestDispatcher controls the virtual clock. advanceTimeBy() moves time forward without real waiting. The debounce operator checks the scheduler's virtual time, so it behaves identically to production but executes instantly. This is why dispatcher injection matters — if the Flow used Dispatchers.Default internally, the debounce would use real time and the test would be flaky.

Testing Android Components (Q35–Q44)

Q35. How does Robolectric work? What are shadow classes? When should you NOT use Robolectric?

Robolectric runs Android code on the JVM by replacing the Android SDK's native implementations with Java-based shadow classes. When your code calls Context.getSystemService(), Robolectric intercepts it via bytecode instrumentation (using ClassLoader manipulation) and routes it to ShadowContextImpl.getSystemService(), which returns a fake implementation. Shadows exist for most framework classes: ShadowActivity, ShadowApplication, ShadowLooper, ShadowSQLiteDatabase, etc. They simulate Android behavior without an emulator. Don't use Robolectric when: (1) Testing rendering fidelity — shadows approximate layout/drawing but don't match real device behavior. (2) Testing hardware-dependent features — camera, sensors, Bluetooth have incomplete shadows. (3) Testing platform-specific behavior across API levels — shadows may not accurately replicate OEM-specific quirks. (4) Testing performance — Robolectric's overhead makes it unsuitable for benchmarking. (5) Testing WebView — ShadowWebView is extremely limited. Use Robolectric for: lifecycle tests, Intent resolution, SharedPreferences, ContentProvider queries, and anything where you need Android APIs but not a real device. It's 10-100x faster than instrumented tests.

Q36. Explain ActivityScenario — how do you test configuration changes and saved state?

ActivityScenario (from androidx.test) provides a controlled API to launch, drive, and inspect Activities. It works on both real devices and Robolectric. To test configuration changes:

@Test
fun `state survives rotation`() {
    val scenario = ActivityScenario.launch(MyActivity::class.java)
    scenario.onActivity { activity ->
        activity.viewModel.setQuery("test")
    }
    scenario.recreate() // simulates config change
    scenario.onActivity { activity ->
        assertThat(activity.viewModel.query.value).isEqualTo("test")
    }
}

For saved state / process death simulation:

@Test
fun `state survives process death`() {
    val scenario = ActivityScenario.launch(MyActivity::class.java)
    scenario.onActivity { it.savedField = "important" }
    val bundle = Bundle()
    scenario.onActivity { it.onSaveInstanceState(bundle) }
    scenario.moveToState(Lifecycle.State.DESTROYED)
    // Re-launch with saved state
    val restored = ActivityScenario.launch(MyActivity::class.java, bundle)
    restored.onActivity {
        assertThat(it.savedField).isEqualTo("important")
    }
}

scenario.moveToState() drives the Activity to any lifecycle state. scenario.recreate() destroys and recreates the Activity, preserving onSaveInstanceState data but not ViewModel (simulating config change). The scenario handles all the ActivityThread machinery that normally requires a running system.

Q37. How do you test a Fragment in isolation? What is launchFragmentInContainer?

launchFragmentInContainer (from androidx.fragment:fragment-testing) launches a Fragment inside a minimal empty Activity, isolating it from the real host Activity's logic:

@Test
fun `displays user name`() {
    val bundle = bundleOf("userId" to "123")
    val scenario = launchFragmentInContainer<ProfileFragment>(bundle)
    scenario.onFragment { fragment ->
        assertThat(fragment.binding.nameText.text).isEqualTo("Test User")
    }
}

It creates an EmptyFragmentActivity (a bare FragmentActivity with a FrameLayout container), adds the Fragment via FragmentManager, and returns a FragmentScenario with the same API as ActivityScenario — moveToState(), recreate(), onFragment {}. You can pass a FragmentFactory for constructor injection. Key limitations: (1) The host Activity is not your real Activity, so Fragment code that casts requireActivity() to a specific type will crash. (2) Navigation tests don't work because there's no NavHostFragment. (3) For Hilt, launchFragmentInContainer doesn't work because the empty Activity isn't annotated with @AndroidEntryPoint — you need launchFragmentInHiltContainer from Hilt's testing library.

Q38. How do you test Room DAOs? What is the in-memory database pattern?

Room DAOs are tested using an in-memory database that exists only in RAM, is fast, and is destroyed after each test:

@RunWith(AndroidJUnit4::class)
class UserDaoTest {
    private lateinit var db: AppDatabase
    private lateinit var dao: UserDao

    @Before
    fun setup() {
        db = Room.inMemoryDatabaseBuilder(
            ApplicationProvider.getApplicationContext(),
            AppDatabase::class.java
        ).allowMainThreadQueries().build()
        dao = db.userDao()
    }

    @After
    fun teardown() { db.close() }

    @Test
    fun `insert and query returns user`() = runTest {
        val user = UserEntity(id = "1", name = "Alice")
        dao.insert(user)
        val result = dao.getById("1")
        assertThat(result).isEqualTo(user)
    }

    @Test
    fun `flow emits on update`() = runTest {
        dao.insert(UserEntity("1", "Alice"))
        dao.observeById("1").test {
            assertThat(awaitItem()?.name).isEqualTo("Alice")
            dao.update(UserEntity("1", "Bob"))
            assertThat(awaitItem()?.name).isEqualTo("Bob")
            cancelAndIgnoreRemainingEvents()
        }
    }
}

allowMainThreadQueries() is needed because tests run on the main thread. The in-memory database tests the actual SQL queries Room generates — this catches typos in @Query annotations, migration issues, and type converter bugs. These are integration tests (they need Android context) but run fast on Robolectric (~50ms each). For migration testing, use MigrationTestHelper which creates a real SQLite database file, applies migrations sequentially, and validates the schema.

Q39. How do you test Navigation? What are the limitations of TestNavHostController?

TestNavHostController is a special NavController for tests that tracks navigation state without actually performing Fragment transactions:

@Test
fun `clicking login navigates to home`() {
    val navController = TestNavHostController(ApplicationProvider.getApplicationContext())
    navController.setGraph(R.navigation.main_nav)
    navController.setCurrentDestination(R.id.loginFragment)

    val scenario = launchFragmentInContainer<LoginFragment>()
    scenario.onFragment { fragment ->
        Navigation.setViewNavController(fragment.requireView(), navController)
    }

    onView(withId(R.id.loginButton)).perform(click())
    assertThat(navController.currentDestination?.id).isEqualTo(R.id.homeFragment)
}

Limitations: (1) It doesn't perform actual Fragment transactions — you can't verify that the destination Fragment renders correctly. (2) Deep link testing is limited — handleDeepLink() works but doesn't simulate the full Intent resolution pipeline. (3) It doesn't test popUpTo with inclusive correctly in all edge cases — the back stack manipulation is simplified. (4) Shared element transitions and animations are not tested. (5) For Compose Navigation, you use createComposeRule() with a real NavHost instead — TestNavHostController is Fragment-centric. For comprehensive navigation testing, combine TestNavHostController for unit-level route assertions with Espresso/Compose E2E tests for full flow verification.

Q40. Compare Espresso vs Compose Testing — when to use each in a hybrid app.

Espresso tests the View system: onView(withId(R.id.button)).perform(click()).check(matches(isDisplayed())). It uses IdlingResource for async synchronization and operates on the View hierarchy via ViewInteraction. Compose Testing uses ComposeTestRule: onNodeWithText("Submit").performClick().assertIsDisplayed(). It operates on the semantics tree (accessibility nodes), not the View hierarchy. In a hybrid app: use Espresso for screens built with Views/XML, Compose Testing for Compose screens. For screens mixing both (e.g., ComposeView inside a Fragment), you can use both in the same test — Espresso for the Fragment's View hierarchy and createAndroidComposeRule for the Compose content. Key differences: (1) Compose Testing doesn't need IdlingResource — it automatically waits for recomposition to settle. (2) Espresso uses view IDs; Compose uses testTag or semantic matchers. (3) Compose Testing supports StateRestorationTester for process death simulation — Espresso has no equivalent. Strategy: as you migrate to Compose, migrate tests from Espresso to Compose Testing. Don't maintain parallel test suites for the same screen.

Q41. How do you test rememberSaveable across process death? What is StateRestorationTester?

StateRestorationTester simulates state save/restore without destroying the Activity — it triggers SaveableStateRegistry to save all rememberSaveable values, clears the composition, and restores from the saved state:

@Test
fun `counter survives state restoration`() {
    val restorationTester = StateRestorationTester(composeTestRule)
    restorationTester.setContent {
        var count by rememberSaveable { mutableIntStateOf(0) }
        Button(onClick = { count++ }) { Text("Count: $count") }
    }

    composeTestRule.onNodeWithText("Count: 0").performClick()
    composeTestRule.onNodeWithText("Count: 1").assertIsDisplayed()

    restorationTester.emulateSavedInstanceStateRestore()

    composeTestRule.onNodeWithText("Count: 1").assertIsDisplayed()
}

Under the hood, emulateSavedInstanceStateRestore() calls SaveableStateRegistry.performSave() to serialize all saveable state into a Map, disposes the composition entirely, then recreates it with the saved map as the restored state. This catches bugs where: (1) state isn't marked as rememberSaveable (only remember), (2) custom Saver implementations lose data during serialization, (3) state depends on non-saveable objects that become null after restoration. It's faster than ActivityScenario.recreate() because it doesn't involve Activity lifecycle — just Compose's state system.

Q42. How do you test a composable that uses hiltViewModel()?

Composables using hiltViewModel() require a Hilt-managed Activity because hiltViewModel() internally accesses the ViewModelStoreOwner's Hilt component. In tests:

@HiltAndroidTest
@RunWith(AndroidJUnit4::class)
class SearchScreenTest {
    @get:Rule(order = 0)
    val hiltRule = HiltAndroidRule(this)

    @get:Rule(order = 1)
    val composeRule = createAndroidComposeRule<HiltComponentActivity>()

    @BindValue
    val fakeRepo: SearchRepository = FakeSearchRepository()

    @Test
    fun `displays search results`() {
        (fakeRepo as FakeSearchRepository).setResults(listOf("Result 1"))
        composeRule.setContent {
            SearchScreen() // internally calls hiltViewModel()
        }
        composeRule.onNodeWithText("Result 1").assertIsDisplayed()
    }
}

HiltComponentActivity (from hilt-android-testing) is a minimal Activity annotated with @AndroidEntryPoint. createAndroidComposeRule launches it, and @BindValue replaces the real SearchRepository binding with the fake. The order parameter on rules ensures Hilt initializes before Compose. Alternative approach: extract the ViewModel's state and callbacks as parameters to the composable, making it a "stateless" composable that doesn't call hiltViewModel() — then test it with a plain createComposeRule() without Hilt. This is the preferred pattern: test the composable UI with plain Compose testing, test the ViewModel separately as a unit test.

Q43. What is screenshot testing? Compare Roborazzi, Paparazzi, and Dropshots.

Screenshot testing renders a UI component and compares the output image against a stored reference (golden) image. Any pixel difference beyond a threshold fails the test, catching visual regressions. Paparazzi (by Cash App): runs on JVM using Android's layoutlib (the same renderer Android Studio uses for XML previews). No device/emulator needed. Fast (~1s per screenshot). Supports Views and Compose. Limitation: layoutlib rendering doesn't perfectly match device rendering, and it can't test device-specific features. Roborazzi: runs on JVM using Robolectric's native rendering. Integrates with Robolectric's shadow system, so it can test more complex scenarios (Activities, Fragments with lifecycle). Supports Compose via captureRoboImage(). Slightly slower than Paparazzi but more accurate for full-screen captures. Dropshots (by Dropbox): runs on a real device/emulator, capturing actual rendered pixels. Most accurate but slowest — requires CI with emulators. Best for pixel-perfect validation. Recommendation: Paparazzi or Roborazzi for component-level screenshot tests in CI (fast, no device). Dropshots for critical screens where pixel accuracy matters. All three integrate with CI via golden file comparison and PR diff images.

Q44. How do you handle flaky tests at scale? What strategies does Google use?

Flaky tests — tests that pass and fail non-deterministically — are a top productivity killer at scale. Strategies: (1) Quarantine — automatically detect flaky tests (run each failing test 3x; if it passes once, it's flaky) and move them to a separate suite that doesn't block CI. Track flaky rate per team. (2) Deterministic concurrency — inject TestDispatcher everywhere, use runTest with virtual time, never use delay() or Thread.sleep() in tests. (3) Hermetic tests — no shared state between tests. Each test creates its own database, its own fake server, its own DI graph. Use @Before/@After to reset singletons. (4) Idling resources — for Espresso, register IdlingResource for all async operations (network, database, animations). For Compose, the framework handles this automatically. (5) Test sharding — run tests in parallel on multiple emulators via AndroidJUnitRunner sharding or Gradle's maxParallelForks. (6) Deflake tooling — Google's internal TAP (Test Automation Platform) runs each test 100x on commit to detect flakiness before merge. Open-source equivalent: run tests with --runs-per-test=5 in CI. (7) Screenshot diffing — replace pixel-exact assertions with perceptual diff thresholds (allow 0.1% pixel difference for anti-aliasing). At Google scale, the flaky test rate is tracked as a team-level metric, and teams with >1% flaky rate are required to fix or delete offending tests.

Testing with Hilt (Q45–Q52)

Q45. Explain the three ways to replace bindings in Hilt tests: @UninstallModules, @BindValue, @TestInstallIn.

@UninstallModules removes a specific module from the Hilt component for that test class. You then provide replacement bindings via an inner @Module or @BindValue. It's per-test-class — different test classes can uninstall different modules. @BindValue binds a field in the test class directly into the Hilt component. It's the simplest way to replace a single binding: @BindValue val repo: UserRepository = FakeUserRepository(). It works with @Named and qualifiers. Under the hood, Hilt generates a module that provides the field's value. @TestInstallIn is a global replacement — it replaces a module across all test classes in the test source set. Defined as a top-level module: @TestInstallIn(components = [SingletonComponent::class], replaces = [NetworkModule::class]). Use it for bindings that should always be faked in tests (e.g., analytics, crash reporting). The hierarchy: @TestInstallIn applies globally, @UninstallModules applies per-class, @BindValue applies per-field. Combine them: use @TestInstallIn for infrastructure fakes, @UninstallModules + @BindValue for test-specific overrides.

Q46. How do you set up Hilt for instrumented tests? Walk through the complete setup.

Complete setup requires four pieces: (1) Dependencies: add androidTestImplementation("com.google.dagger:hilt-android-testing:$version") and kaptAndroidTest("com.google.dagger:hilt-android-compiler:$version") (or kspAndroidTest). (2) Custom test runner: create a class that extends AndroidJUnitRunner and overrides newApplication() to return HiltTestApplication:

class HiltTestRunner : AndroidJUnitRunner() {
    override fun newApplication(cl: ClassLoader?, name: String?, context: Context?): Application {
        return super.newApplication(cl, HiltTestApplication::class.java.name, context)
    }
}

Register it in build.gradle: testInstrumentationRunner = "com.example.HiltTestRunner". (3) Test class annotations: annotate each test class with @HiltAndroidTest and add @get:Rule val hiltRule = HiltAndroidRule(this). Call hiltRule.inject() in @Before if you need field injection in the test class itself. (4) For Compose tests: use createAndroidComposeRule<HiltComponentActivity>() and ensure rule ordering with @get:Rule(order = 0) for Hilt, order = 1 for Compose. The HiltTestApplication replaces your real Application class, generating a test component that supports @UninstallModules, @BindValue, and @TestInstallIn.

Q47. How do you test a Fragment with Hilt? Why doesn't launchFragmentInContainer work?

launchFragmentInContainer launches the Fragment inside an EmptyFragmentActivity — a bare-bones Activity that is not annotated with @AndroidEntryPoint. Since Hilt requires the host Activity to be a Hilt-managed entry point (to provide the ActivityComponent), injection fails with IllegalStateException: Hilt Fragments must be attached to an @AndroidEntryPoint Activity. The solution is launchFragmentInHiltContainer, a utility function from Hilt's testing documentation:

inline fun <reified T : Fragment> launchFragmentInHiltContainer(
    fragmentArgs: Bundle? = null,
    themeResId: Int = R.style.FragmentScenarioEmptyFragmentActivityTheme,
    crossinline action: Fragment.() -> Unit = {}
) {
    val startActivityIntent = Intent.makeMainActivity(
        ComponentName(ApplicationProvider.getApplicationContext(), HiltTestActivity::class.java)
    ).putExtra("theme", themeResId)

    ActivityScenario.launch<HiltTestActivity>(startActivityIntent).onActivity { activity ->
        val fragment: Fragment = activity.supportFragmentManager.fragmentFactory.instantiate(
            Preconditions.checkNotNull(T::class.java.classLoader), T::class.java.name
        )
        fragment.arguments = fragmentArgs
        activity.supportFragmentManager.beginTransaction()
            .add(android.R.id.content, fragment, "")
            .commitNow()
        fragment.action()
    }
}

HiltTestActivity is a minimal Activity annotated with @AndroidEntryPoint that you add to your debug/AndroidManifest.xml. This provides the Hilt component hierarchy that the Fragment needs for injection.

Q48. Design an E2E test that uses Hilt with MockWebServer — show the architecture.

The architecture layers: MockWebServer replaces the real API, Hilt replaces the base URL binding, and the test drives the full UI flow:

@HiltAndroidTest
@UninstallModules(NetworkModule::class)
class CheckoutFlowTest {
    @get:Rule(order = 0) val hiltRule = HiltAndroidRule(this)
    @get:Rule(order = 1) val composeRule = createAndroidComposeRule<MainActivity>()

    private val mockServer = MockWebServer()

    @BindValue val api: ProductApi = Retrofit.Builder()
        .baseUrl(mockServer.url("/"))
        .addConverterFactory(MoshiConverterFactory.create())
        .build()
        .create(ProductApi::class.java)

    @Before fun setup() {
        mockServer.enqueue(MockResponse().setBody("""[{"id":"1","name":"Widget","price":9.99}]"""))
        mockServer.enqueue(MockResponse().setBody("""{"orderId":"ORD-123","status":"confirmed"}"""))
    }

    @After fun teardown() { mockServer.shutdown() }

    @Test
    fun `complete checkout flow`() {
        composeRule.onNodeWithText("Widget").performClick()
        composeRule.onNodeWithText("Add to Cart").performClick()
        composeRule.onNodeWithText("Checkout").performClick()
        composeRule.onNodeWithText("ORD-123").assertIsDisplayed()
        assertThat(mockServer.requestCount).isEqualTo(2)
    }
}

Key design decisions: (1) @UninstallModules(NetworkModule::class) removes the production Retrofit binding. (2) @BindValue provides a Retrofit instance pointing at MockWebServer's local URL. (3) MockWebServer queues responses in order — the first request gets the product list, the second gets the order confirmation. (4) The test verifies the full user journey through real Activities, real ViewModels, real navigation — only the network layer is faked. This catches integration bugs that unit tests miss.

Q49. When should you use @TestInstallIn (global) vs @UninstallModules (per-test)?

Use @TestInstallIn for bindings that should always be replaced in every test: analytics (you never want real analytics firing in tests), crash reporting (no real Crashlytics), feature flags (deterministic values), and network interceptors (logging, auth tokens). These are "infrastructure" concerns that no test should use the real implementation of. Use @UninstallModules + @BindValue for bindings that vary per test: a repository that returns success in one test and error in another, a clock that returns different times, or a feature flag that's enabled in one test and disabled in another. The key trade-off: @TestInstallIn creates a single test component shared across all tests — this is faster (component is generated once) but inflexible. @UninstallModules creates a unique component per test class — more flexible but slower compilation (Hilt generates a new component for each unique combination of uninstalled modules). At scale, minimize @UninstallModules usage: define a comprehensive @TestInstallIn suite with configurable fakes (e.g., FakeUserRepository with a setResponse() method), and use @BindValue only for the few bindings that truly need per-test customization.

Q50. How do you configure JaCoCo to exclude Hilt-generated code from coverage reports?

Hilt generates substantial code (Hilt_*, *_Factory, *_MembersInjector, DaggerXxx_HiltComponents_*) that inflates your codebase and deflates coverage percentages. Configure JaCoCo exclusions in build.gradle:

tasks.withType<JacocoReport> {
    afterEvaluate {
        classDirectories.setFrom(files(classDirectories.files.map {
            fileTree(it) {
                exclude(
                    "**/Hilt_*.*",
                    "**/*_HiltModules*.*",
                    "**/*_Factory.*",
                    "**/*_MembersInjector.*",
                    "**/Dagger*Component*.*",
                    "**/BuildConfig.*",
                    "**/*_Impl.*", // Room-generated
                    "**/hilt_aggregated_deps/**"
                )
            }
        }))
    }
}

Also exclude from SonarQube/Codecov via their respective configuration files. The exclusion patterns should cover: Dagger/Hilt generated components, factories, member injectors, and the aggregated deps metadata. Additionally, exclude Room-generated _Impl classes and DataBinding/ViewBinding generated code. The goal is to measure coverage of your code, not framework-generated boilerplate. Track coverage at the module level — a :core:data module should have 80%+ coverage of its repository and data source classes, independent of how much generated code exists.

Q51. How do you test a CoroutineWorker that uses Hilt (@HiltWorker)?

@HiltWorker uses @AssistedInject under the hood — Context and WorkerParameters are assisted, and all other constructor parameters are injected from the Hilt graph. To test:

@HiltAndroidTest
class SyncWorkerTest {
    @get:Rule val hiltRule = HiltAndroidRule(this)

    @BindValue val syncRepo: SyncRepository = FakeSyncRepository()

    @Test
    fun `sync worker uploads pending data`() = runTest {
        val context = ApplicationProvider.getApplicationContext<Context>()
        val worker = TestListenableWorkerBuilder<SyncWorker>(context)
            .setWorkerFactory(hiltRule.workerFactory()) // Hilt-aware factory
            .build()

        (syncRepo as FakeSyncRepository).setPendingItems(listOf(item1, item2))
        val result = worker.doWork()

        assertThat(result).isEqualTo(ListenableWorker.Result.success())
        assertThat((syncRepo as FakeSyncRepository).uploadedItems).hasSize(2)
    }
}

The critical piece is TestListenableWorkerBuilder from work-testing — it creates the Worker with test parameters (no real WorkManager scheduling). Combined with Hilt's @BindValue to replace the repository, you test the Worker's logic in isolation. For testing scheduling behavior (constraints, backoff, chaining), use TestDriver from work-testing which lets you simulate constraint satisfaction and trigger period work. Note: TestListenableWorkerBuilder calls the Worker's constructor directly, so the Hilt worker factory must be set to resolve @Inject dependencies.

Q52. What are the common pitfalls when using Hilt for testing? List at least 5.

(1) Rule ordering: HiltAndroidRule must execute before ActivityScenarioRule or ComposeTestRule. Use @get:Rule(order = 0) for Hilt, order = 1 for others. Wrong order causes UninitializedPropertyAccessException. (2) @UninstallModules + @TestInstallIn conflict: if a @TestInstallIn globally replaces ModuleA, and a test also @UninstallModules(ModuleA::class), you get duplicate binding errors because the @TestInstallIn replacement is still installed. (3) Forgetting HiltTestApplication: without the custom test runner returning HiltTestApplication, Hilt's test component isn't created and all injection fails silently or with cryptic errors. (4) Scoping leaks in tests: @Singleton-scoped fakes retain state across tests if the Application isn't recreated. Always reset fake state in @Before or use @BindValue which creates fresh instances per test class. (5) Slow compilation: each unique @UninstallModules combination generates a separate Hilt component. 50 test classes with different uninstall sets = 50 generated components. Minimize unique combinations. (6) Robolectric compatibility: Hilt's bytecode transform requires specific Robolectric configuration (@Config(application = HiltTestApplication::class)) and the robolectric-hilt artifact. (7) @BindValue with interfaces: the field type must match the binding type exactly — @BindValue val repo: UserRepository = FakeUserRepository() binds UserRepository, not FakeUserRepository.

System Design / Staff-Level (Q53–Q58)

Q53. Design the testing strategy for a new feature (search with autocomplete, debounce, pagination) — what tests at each layer?

Unit tests (ViewModel layer): (1) Test debounce behavior — verify that rapid query changes result in a single API call after 300ms using advanceTimeBy(). (2) Test pagination — verify that loadMore() appends results, handles last page, and doesn't duplicate requests. (3) Test error states — network error shows retry, empty results shows empty state. (4) Test query filtering — queries under 2 characters don't trigger search. Use FakeSearchRepository and Turbine for Flow assertions. Unit tests (Repository layer): (5) Test API response mapping — verify JSON → domain model conversion. (6) Test caching — second identical query returns cached results. (7) Test pagination token handling — verify next page token is passed correctly. Integration tests: (8) Room DAO test — verify search history insert/query/delete. (9) Repository + MockWebServer — verify HTTP request format, headers, error code handling. UI tests: (10) Compose test — verify search field input, loading indicator, result list rendering, scroll-to-load-more trigger. (11) Screenshot test — golden images for empty state, loading, results, error. E2E test: (12) Full flow with Hilt + MockWebServer — type query, see results, scroll for pagination, tap result, verify navigation. Total: ~15-20 tests, 80% unit, 15% integration, 5% E2E.

Q54. You're the tech lead for a 200-module app with 5000 tests. Tests take 45 minutes on CI. How do you reduce this to 15 minutes?

Phase 1 — Measure (Week 1): Profile the test suite. Categorize: how many are unit (JVM), Robolectric, instrumented? What's the time distribution? Usually 80% of time is spent on 20% of tests. Identify the slowest 50 tests. Phase 2 — Quick wins (Weeks 2-3): (1) Test sharding — split instrumented tests across 4-8 emulators using numShards in AndroidJUnitRunner or Gradle Managed Devices. This alone can cut instrumented test time by 4-8x. (2) Migrate Robolectric → pure JVM — many tests use Robolectric unnecessarily. If a test only needs Context for string resources, use a fake. Robolectric startup adds 3-5 seconds per test class. (3) Parallel execution — set maxParallelForks = Runtime.getRuntime().availableProcessors() / 2 for JVM tests. Phase 3 — Architecture (Weeks 4-8): (4) Affected module detection — only run tests in modules affected by the PR's changes. Use Gradle's -p flag or tools like Dropbox's AffectedModuleDetector. In a 200-module app, most PRs touch 2-5 modules. (5) Test caching — enable Gradle remote build cache so unchanged modules' test results are reused. (6) Move to Gradle Managed Devices with ATD (Automated Test Device) images — 30-50% faster than standard emulators. Phase 4 — Ongoing: (7) Quarantine flaky tests — they waste retry time. (8) Set CI time budgets per module. (9) Track test time as a metric in engineering dashboards.

Q55. Design the DI architecture for a super-app with 10 feature modules, each with its own dependencies.

Architecture: three-layer module structure with Hilt. Layer 1 — Core modules: :core:network (Retrofit, OkHttp, interceptors), :core:database (Room), :core:auth (token management), :core:analytics. Each provides a Hilt @Module @InstallIn(SingletonComponent::class) with interface bindings. Layer 2 — Feature API modules: :feature:payments-api exposes interface PaymentNavigator and interface PaymentRepository — no implementations, no Hilt modules. Layer 3 — Feature implementation modules: :feature:payments-impl depends on :feature:payments-api and :core:*. Contains @Module @InstallIn(SingletonComponent::class) object PaymentsModule { @Binds fun bindRepo(impl: PaymentRepositoryImpl): PaymentRepository }. The :app module depends on all :feature:*-impl modules. Hilt's module aggregation discovers all modules automatically. Cross-feature communication: features depend on each other's API modules, never impl modules. :feature:checkout-impl depends on :feature:payments-api to inject PaymentNavigator. Scoping: shared infrastructure is @Singleton. Feature-specific state uses @ActivityRetainedScoped or @ViewModelScoped. Dynamic feature modules: use @EntryPoint to access the base app's graph, since dynamic modules can't use standard Hilt aggregation.

Q56. A team member argues "we don't need DI, we can just use object singletons." How do you respond?

The argument has surface appeal — object UserRepository is simple, no framework needed. But it fails on four axes: (1) Testability: you cannot replace an object singleton in tests. Every test that touches code using UserRepository hits the real implementation — real network, real database. You'd need mockkObject which is fragile and couples tests to implementation. With DI, tests inject FakeUserRepository trivially. (2) Lifecycle management: an object lives for the entire process. If UserRepository holds a database connection, it's never closed. If it caches user data, it persists across logout/login. With DI scoping, you can create a @UserSessionScoped repository that's destroyed on logout. (3) Dependency ordering: object initialization is lazy but unordered. If UserRepository depends on AuthManager which depends on NetworkClient, you get implicit initialization chains that are hard to reason about and can cause ExceptionInInitializerError. DI makes the graph explicit. (4) Compile-time isolation: every module that uses UserRepository has a compile-time dependency on its implementation. Changing the constructor requires recompiling all consumers. With DI + interfaces, only the module defining the binding recompiles. The response isn't "use Hilt" — it's "use constructor injection." Even manual DI with a factory class solves all four problems without a framework.

Q57. You're migrating from Koin to Hilt in a 100-screen app. Design the migration plan — phases, risks, rollback strategy.

Phase 0 — Assessment (1 week): Audit all Koin modules — count definitions, identify scopes, map qualifiers, find get()/inject() call sites. Create a spreadsheet: module name → Koin definitions → equivalent Hilt scope → migration complexity (low/medium/high). Identify blockers: classes with Koin-specific patterns (scope { }, parametersOf()). Phase 1 — Coexistence setup (1 week): Add Hilt dependencies alongside Koin. Annotate Application with @HiltAndroidApp. Create a KoinBridge module that exposes Koin-managed singletons to Hilt via @Provides methods calling KoinJavaComponent.get(). This lets Hilt-migrated code consume Koin-managed dependencies during transition. Phase 2 — Module-by-module migration (6-8 weeks): Migrate one feature module per sprint. Order: start with leaf modules (no dependents), work toward core. For each module: (a) create Hilt @Module, (b) annotate Activities/Fragments with @AndroidEntryPoint, (c) replace by inject() with @Inject, (d) remove Koin definitions, (e) run tests. Phase 3 — Remove Koin (1 week): Delete all Koin modules, remove startKoin, remove Koin dependencies. Rollback strategy: feature-flag the migration at the Application level — if (useHilt) startHilt() else startKoin(). If crash rate increases, disable the flag via remote config. Risks: (1) Koin's parametersOf() → Hilt's @AssistedInject is not 1:1. (2) Koin scopes tied to custom lifecycle → need custom Hilt components. (3) Test migration is as much work as production code.

Q58. How do you balance test coverage with development velocity? What metrics do you track?

The goal is not 100% coverage — it's high-confidence coverage of critical paths with minimal maintenance burden. Metrics to track: (1) Line/branch coverage by module — set thresholds per module type: :core:data ≥ 80%, :feature:* ≥ 60%, :app ≥ 40%. Enforce via CI gates that fail the build if coverage drops below threshold. (2) Mutation testing score — tools like PIT (pitest) inject bugs into your code and check if tests catch them. Coverage can be 80% but mutation score 30% if tests don't assert meaningful behavior. (3) Test execution time — track P50 and P99 test suite duration. Set a budget (e.g., 15 minutes max). (4) Flaky test rate — percentage of tests that fail non-deterministically. Target < 0.5%. (5) Defect escape rate — bugs found in production that should have been caught by tests. This is the ultimate metric. Balancing velocity: (a) Require tests for bug fixes (regression tests) and new features, but not for refactors that don't change behavior. (b) Use code review to enforce test quality, not just coverage numbers. (c) Invest in test infrastructure (fakes, test utilities, CI speed) so writing tests is fast. (d) Delete tests that are more expensive to maintain than the bugs they prevent. At L6, you own the testing culture — define the strategy, build the infrastructure, and mentor the team on writing effective tests rather than chasing coverage numbers.
