Concurrency Patterns

Mutex, Semaphore, parallel decomposition, fan-out/fan-in, withTimeout, race patterns, thread confinement, and the actor model

1. Mutex — Coroutine-Safe Mutual Exclusion

import kotlinx.coroutines.sync.Mutex
import kotlinx.coroutines.sync.withLock

class BankAccount {
    private val mutex = Mutex()
    private var balance = 0.0

    suspend fun deposit(amount: Double) {
        mutex.withLock {
            // Only one coroutine can execute this block at a time
            val current = balance
            delay(10) // Simulate processing
            balance = current + amount
        }
    }

    suspend fun getBalance(): Double = mutex.withLock { balance }
}

Mutex vs synchronized

PropertysynchronizedMutex
Blocks thread?Yes — thread is held hostageNo — coroutine suspends, thread is free
Reentrant?Yes — same thread can re-enterNo — same coroutine re-entering deadlocks
Works across dispatchers?No — tied to thread identityYes — works regardless of which thread resumes
Use in coroutines?Dangerous — can starve thread poolSafe — designed for coroutines
Mutex Is Not Reentrant

Unlike synchronized, Mutex does not allow the same coroutine to acquire the lock twice. This will deadlock:

val mutex = Mutex()
suspend fun outer() = mutex.withLock {
    inner() // DEADLOCK — inner tries to acquire the same mutex
}
suspend fun inner() = mutex.withLock { /* ... */ }

The fix: restructure to avoid nested locking, or extract the shared logic into a private function called within a single lock scope.

2. Semaphore — Limiting Concurrency

import kotlinx.coroutines.sync.Semaphore
import kotlinx.coroutines.sync.withPermit

// Allow at most 5 concurrent network requests
val networkSemaphore = Semaphore(permits = 5)

suspend fun fetchUrl(url: String): Response {
    return networkSemaphore.withPermit {
        // At most 5 coroutines execute this block concurrently
        httpClient.get(url)
    }
}

// Process 1000 URLs with max 5 concurrent requests
suspend fun fetchAll(urls: List<String>): List<Response> = coroutineScope {
    urls.map { url ->
        async { fetchUrl(url) }
    }.awaitAll()
}
Staff-Level Insight — Semaphore vs limitedParallelism

Both limit concurrency, but at different levels:

  • Semaphore — limits how many coroutines enter a critical section. The coroutines can run on any dispatcher/thread.
  • limitedParallelism(n) — limits how many threads a dispatcher uses. Affects all coroutines on that dispatcher.

Use Semaphore when you want to limit a specific operation (e.g., max 5 API calls). Use limitedParallelism when you want to limit a resource (e.g., max 4 database connections).

3. Parallel Decomposition

Basic Pattern — async/await

suspend fun loadDashboard(userId: String): Dashboard = coroutineScope {
    // Launch all requests in parallel
    val user = async { userApi.getUser(userId) }
    val posts = async { postApi.getPosts(userId) }
    val notifications = async { notifApi.getUnread(userId) }

    // Wait for all results
    Dashboard(
        user = user.await(),
        posts = posts.await(),
        notifications = notifications.await()
    )
    // If any fails → all cancelled (coroutineScope uses Job, not SupervisorJob)
}

Fan-Out — Distribute Work to Multiple Workers

suspend fun processImages(images: List<Image>): List<ProcessedImage> = coroutineScope {
    images.map { image ->
        async(Dispatchers.Default) {
            processImage(image) // CPU-intensive
        }
    }.awaitAll()
}

// With concurrency limit (avoid overwhelming CPU)
suspend fun processImages(images: List<Image>): List<ProcessedImage> = coroutineScope {
    val semaphore = Semaphore(4) // Max 4 concurrent
    images.map { image ->
        async(Dispatchers.Default) {
            semaphore.withPermit { processImage(image) }
        }
    }.awaitAll()
}

Fan-In — Multiple Producers, Single Consumer

suspend fun mergeDataSources(): Flow<DataItem> {
    val source1 = databaseFlow()
    val source2 = networkFlow()
    val source3 = cacheFlow()
    return merge(source1, source2, source3)
    // Values from all sources interleaved as they arrive
}

4. withTimeout / withTimeoutOrNull

// Throws TimeoutCancellationException if timeout exceeded
suspend fun fetchWithTimeout(): Data {
    return withTimeout(5000L) { // 5 seconds
        api.fetchData() // Cancelled if takes too long
    }
}

// Returns null instead of throwing
suspend fun fetchWithTimeoutOrNull(): Data? {
    return withTimeoutOrNull(5000L) {
        api.fetchData()
    }
}

// Practical pattern: timeout with fallback
suspend fun getUserName(userId: String): String {
    return withTimeoutOrNull(3000L) {
        userApi.getUser(userId).name
    } ?: cachedUserName(userId) ?: "Unknown User"
}
TimeoutCancellationException Is a CancellationException

TimeoutCancellationException extends CancellationException. This means a generic catch (e: CancellationException) will catch timeouts too. And if you're inside a coroutineScope, the timeout cancellation propagates to siblings. Be precise with your catches.

5. Race Pattern — First Wins

// Return the first successful result, cancel the rest
suspend fun <T> raceAll(vararg tasks: suspend () -> T): T = coroutineScope {
    val result = CompletableDeferred<T>()
    tasks.forEach { task ->
        launch {
            try {
                result.complete(task())
            } catch (e: Exception) {
                // Ignore individual failures; if all fail, result never completes
            }
        }
    }
    result.await()
    // coroutineScope waits for all children, but we can cancel the scope
}

// Practical: race cache vs network
suspend fun getUser(id: String): User = coroutineScope {
    select {
        async { cache.getUser(id) }.onAwait { it }
        async { network.getUser(id) }.onAwait { it }
    }
    // select returns the first completed result, but doesn't cancel the other
    // To cancel: wrap in a separate scope and cancel after first result
}

The select Expression

import kotlinx.coroutines.selects.select

suspend fun fetchFastest(): Data = coroutineScope {
    val fromCache = async { cacheRepo.get() }
    val fromNetwork = async { networkRepo.get() }

    val result = select<Data> {
        fromCache.onAwait { it }
        fromNetwork.onAwait { it }
    }
    // Cancel the slower one
    coroutineContext.cancelChildren()
    result
}

6. Thread Confinement

Instead of using locks, confine mutable state to a single thread. No concurrent access = no synchronization needed.

// Single-thread confinement using limitedParallelism
class UserCache {
    private val cacheDispatcher = Dispatchers.Default.limitedParallelism(1)
    private val cache = mutableMapOf<String, User>()

    suspend fun get(id: String): User? = withContext(cacheDispatcher) {
        cache[id]
    }

    suspend fun put(id: String, user: User) = withContext(cacheDispatcher) {
        cache[id] = user
    }

    suspend fun getOrFetch(id: String, fetch: suspend () -> User): User {
        return withContext(cacheDispatcher) {
            cache.getOrPut(id) {
                // DANGER: fetch() might suspend and switch threads
                // But we're inside withContext(cacheDispatcher), so we return to this thread
                withContext(Dispatchers.IO) { fetch() }.also { cache[id] = it }
            }
        }
    }
}
Staff-Level Insight — Thread Confinement vs Mutex

Thread confinement is simpler and often faster than Mutex because there's no lock acquisition overhead. The tradeoff: all access to the confined state must go through withContext(singleThreadDispatcher), which adds a context switch. For high-contention scenarios (many coroutines accessing the same state), thread confinement is more predictable. For low-contention scenarios, Mutex has less overhead. In practice, most Android apps use thread confinement for caches and Mutex for counters/flags.

7. The Actor Pattern

An actor is a coroutine that processes messages from a channel sequentially. It combines thread confinement with message passing:

// The actor pattern (manual implementation — the actor() builder is @ObsoleteCoroutinesApi)
sealed class CounterMsg {
    object Increment : CounterMsg()
    class GetCount(val response: CompletableDeferred<Int>) : CounterMsg()
}

fun CoroutineScope.counterActor(): Channel<CounterMsg> {
    val channel = Channel<CounterMsg>()
    launch {
        var count = 0
        for (msg in channel) { // Sequential processing — no concurrency issues
            when (msg) {
                is CounterMsg.Increment -> count++
                is CounterMsg.GetCount -> msg.response.complete(count)
            }
        }
    }
    return channel
}

// Usage
val counter = scope.counterActor()
counter.send(CounterMsg.Increment)
counter.send(CounterMsg.Increment)
val response = CompletableDeferred<Int>()
counter.send(CounterMsg.GetCount(response))
println(response.await()) // 2
Staff-Level Insight — When to Use Actors

The actor() builder is marked @ObsoleteCoroutinesApi because the Kotlin team plans to replace it with a better API. But the pattern is still valid and widely used. The manual implementation above (Channel + launch) is the recommended approach. Use actors when:

  • You have mutable state that needs sequential access
  • Multiple coroutines need to send commands to a single state owner
  • You want to decouple the "what" (message) from the "when" (processing)

In Android, the ViewModel is essentially an actor — it receives UI events (messages) and processes them sequentially to update state. MVI architecture formalizes this pattern.

8. Retry with Exponential Backoff

suspend fun <T> retryWithBackoff(
    maxRetries: Int = 3,
    initialDelay: Long = 1000L,
    maxDelay: Long = 16000L,
    factor: Double = 2.0,
    block: suspend () -> T
): T {
    var currentDelay = initialDelay
    repeat(maxRetries) { attempt ->
        try {
            return block()
        } catch (e: Exception) {
            if (e is CancellationException) throw e // Never retry cancellation
            if (attempt == maxRetries - 1) throw e  // Last attempt, give up
        }
        delay(currentDelay)
        currentDelay = (currentDelay * factor).toLong().coerceAtMost(maxDelay)
    }
    error("Unreachable")
}

// Usage
val data = retryWithBackoff(maxRetries = 3) {
    api.fetchData() // Retries up to 3 times with 1s, 2s, 4s delays
}

9. Debounce and Throttle

// Debounce — wait for silence, then act (search-as-you-type)
fun debounceSearch(queryFlow: Flow<String>): Flow<List<Result>> {
    return queryFlow
        .debounce(300L)
        .filter { it.length >= 2 }
        .distinctUntilChanged()
        .flatMapLatest { query -> searchApi.search(query) }
}

// Throttle first — act immediately, then ignore for a period (button clicks)
fun <T> Flow<T>.throttleFirst(windowMs: Long): Flow<T> = flow {
    var lastEmitTime = 0L
    collect { value ->
        val now = System.currentTimeMillis()
        if (now - lastEmitTime >= windowMs) {
            lastEmitTime = now
            emit(value)
        }
    }
}

// Throttle latest — emit the most recent value at fixed intervals
fun <T> Flow<T>.throttleLatest(windowMs: Long): Flow<T> {
    return this.sample(windowMs)
}

10. Concurrency Correctness — Deadlocks, Starvation, Priority Inversion

Priority Inversion

Priority inversion occurs when a low-priority task holds a resource needed by a high-priority task, effectively blocking the high-priority task at the low-priority level.

val mutex = Mutex()

// Low-priority: long-running computation on Default (limited to CPU cores)
scope.launch(Dispatchers.Default) {
    mutex.withLock {
        heavyComputation() // Holds the lock for 500ms
    }
}

// High-priority: UI update on Main — BLOCKED waiting for the mutex
scope.launch(Dispatchers.Main) {
    mutex.withLock {
        // Must wait for Default coroutine to finish
        // Main thread is suspended (not blocked), but the UI update is delayed
        updateUI(data)
    }
}

With Mutex, the Main coroutine suspends (thread is free), so it's not an ANR — but the UI update is delayed until the Default coroutine releases the lock. With synchronized, it's worse: the main thread is blocked, causing jank or ANR.

The Three Concurrency Hazards

HazardWhat HappensCoroutine ExampleFix
Deadlock Two tasks each hold a resource the other needs Non-reentrant Mutex: outer() locks M, calls inner() which also locks M → deadlock Avoid nested locking. Use a single lock scope. Or use thread confinement.
Starvation A task never gets to run because others monopolize the resource synchronized inside coroutines on Dispatchers.Default — blocks all CPU-core threads, no coroutine can make progress Use Mutex (suspends, doesn't block). Or move blocking work to Dispatchers.IO.
Priority Inversion High-priority task waits for low-priority task holding a shared resource Main coroutine waiting on Mutex held by Default coroutine doing slow work Minimize critical section duration. Use thread confinement (no shared locks). Or use lock-free structures (AtomicReference, StateFlow.update()).
Staff-Level Insight — Priority Inversion in Android

Classic priority inversion is rare in coroutines because Mutex suspends rather than blocks — the Main thread stays free for frame rendering. But the logical priority inversion still exists: a UI state update is delayed because a background coroutine holds the lock. The real-world impact: the user taps a button, the ViewModel tries to update state, but a background sync holds the Mutex for 200ms → the UI feels laggy.

The L6 fix: avoid shared Mutex between Main and background coroutines entirely. Use thread confinement (limitedParallelism(1)) so all state mutations happen on a single dedicated thread, or use lock-free updates (StateFlow.update { } with CAS). If you must share a Mutex, keep the critical section under 1ms.
