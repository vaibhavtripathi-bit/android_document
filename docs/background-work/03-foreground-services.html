Foreground Services

The only reliable way to do visible, ongoing work — and the increasingly strict rules around them

1. Why Foreground Services Exist

A foreground service tells the system: "The user is aware this work is happening." In exchange for showing a persistent notification, the service gets:

  • OOM adj = 0 (FOREGROUND_APP_ADJ) — same as a visible Activity. Almost never killed.
  • Exemption from background execution limits (Android 8+)
  • Exemption from Doze restrictions (while running)

Use cases: music playback, navigation, file uploads/downloads, fitness tracking, ongoing calls.

2. Basic Implementation

class UploadService : Service() {

    override fun onStartCommand(intent: Intent?, flags: Int, startId: Int): Int {
        val notification = createNotification()
        // Must call within 5 seconds of startForegroundService()
        startForeground(NOTIFICATION_ID, notification)

        CoroutineScope(Dispatchers.IO + SupervisorJob()).launch {
            performUpload()
            stopSelf()
        }

        return START_NOT_STICKY
    }

    private fun createNotification(): Notification {
        val channel = NotificationChannel(
            "upload_channel",
            "Uploads",
            NotificationManager.IMPORTANCE_LOW
        )
        getSystemService(NotificationManager::class.java)
            .createNotificationChannel(channel)

        return NotificationCompat.Builder(this, "upload_channel")
            .setContentTitle("Uploading files...")
            .setSmallIcon(R.drawable.ic_upload)
            .setOngoing(true)
            .build()
    }

    override fun onBind(intent: Intent?): IBinder? = null

    companion object {
        private const val NOTIFICATION_ID = 1
    }
}

Starting from the caller

// From an Activity (foreground)
val intent = Intent(this, UploadService::class.java)
startForegroundService(intent)  // Safe from foreground context

// OR using ContextCompat for backward compat
ContextCompat.startForegroundService(this, intent)
5-Second Rule

After calling startForegroundService(), the service MUST call startForeground(id, notification) within 5 seconds. If it doesn't, the system throws a ForegroundServiceDidNotStartInTimeException and kills the service. This is the #1 crash source for foreground services in production.

Staff-Level Insight — The 5-Second Race

The 5-second timer starts when startForegroundService() is called, but onCreate() and onStartCommand() run on the main thread. If your main thread is blocked (heavy initialization, disk I/O on main thread, slow DI injection), the 5-second window can expire before startForeground() is reached. Solutions:

  • Call startForeground() as the very first line of onStartCommand()
  • Never do heavy work before startForeground()
  • On Android 12+, use ServiceInfo.FOREGROUND_SERVICE_TYPE_* to declare type upfront

When startForeground() succeeds, AMS updates the process's oom_adj from SERVICE (500) to FOREGROUND_APP (0) via OomAdjuster.updateOomAdjLocked(). This makes the process virtually unkillable — the same priority as the foreground Activity. This is why foreground services are the strongest process-survival mechanism, and why Google restricts them heavily.

3. The Evolution of Restrictions (Timeline)

4. Foreground Service Types (Android 14+)

Breaking Change — API 34

Android 14 requires you to declare what your foreground service does. You must declare the type in both the manifest AND at runtime when calling startForeground().

Type Permission Required Timeout Use Case
camera FOREGROUND_SERVICE_CAMERA None Video recording
connectedDevice FOREGROUND_SERVICE_CONNECTED_DEVICE None Bluetooth, USB, companion devices
dataSync FOREGROUND_SERVICE_DATA_SYNC 6 hours Upload/download. Prefer WorkManager instead.
health FOREGROUND_SERVICE_HEALTH None Fitness, heart rate tracking
location FOREGROUND_SERVICE_LOCATION None Navigation, geofencing
mediaPlayback FOREGROUND_SERVICE_MEDIA_PLAYBACK None Music, podcasts, audio books
mediaProjection FOREGROUND_SERVICE_MEDIA_PROJECTION None Screen sharing, screen recording
microphone FOREGROUND_SERVICE_MICROPHONE None Voice recording, voice calls
phoneCall FOREGROUND_SERVICE_PHONE_CALL None Ongoing phone/VoIP calls
remoteMessaging FOREGROUND_SERVICE_REMOTE_MESSAGING None Messaging on secondary devices
shortService None 3 minutes Quick tasks that must finish. Can't be stopped by user.
specialUse FOREGROUND_SERVICE_SPECIAL_USE None Escape hatch — requires Play Store justification
systemExempted System apps only None System-level services 
// Manifest
<uses-permission android:name="android.permission.FOREGROUND_SERVICE" />
<uses-permission android:name="android.permission.FOREGROUND_SERVICE_LOCATION" />

<service
    android:name=".NavigationService"
    android:foregroundServiceType="location"
    android:exported="false" />

// Runtime — must pass the type
startForeground(
    NOTIFICATION_ID,
    notification,
    ServiceInfo.FOREGROUND_SERVICE_TYPE_LOCATION
)
Staff-Level Insight — dataSync Timeout

The dataSync type has a 6-hour hard timeout on Android 14+. After 6 hours, the system calls onTimeout() and you must stop within seconds or the system force-stops you. Google's message is clear: use WorkManager for data sync, not foreground services. The dataSync type is being deprecated in Android 15.

5. Background Start Restrictions (Android 12+)

Breaking — API 31

Apps targeting Android 12+ cannot start foreground services from the background. The system throws ForegroundServiceStartNotAllowedException.

Exemptions (when you CAN start from background)

  • Handling a high-priority FCM push (RemoteMessage.getPriority() == PRIORITY_HIGH)
  • User tapped a notification action / PendingIntent
  • App has SYSTEM_ALERT_WINDOW permission
  • App is a device owner / profile owner
  • Exact alarm (ACTION_SCHEDULE_EXACT_ALARM) fires
  • App is on the battery optimization exemption list
  • Companion device manager role
Staff-Level Insight — The Real Impact

This restriction broke many apps that relied on BroadcastReceiver → startForegroundService() patterns. The fix is usually one of:

  • WorkManager with setExpedited() — runs immediately, no notification needed on API 31+
  • High-priority FCM → service start (still allowed)
  • User-initiated data transfer (Android 14+) — new WorkManager feature for large uploads/downloads the user explicitly triggered

At Google scale, the migration from FGS to WorkManager for background-initiated work was a multi-quarter effort for many teams.

6. Stopping a Foreground Service

// Remove from foreground but keep service running
stopForeground(STOP_FOREGROUND_REMOVE)  // removes notification
stopForeground(STOP_FOREGROUND_DETACH)  // keeps notification but it becomes dismissible

// Actually stop the service
stopSelf()
Interview Gotcha

Q: "Does stopForeground() stop the service?"
No. It only removes the foreground state (and optionally the notification). The service keeps running as a regular background service — which means it's now subject to background execution limits and can be killed. You must still call stopSelf() to actually stop.

7. Android 15 Changes (Preview)

Upcoming

  • dataSync foreground service type is deprecated — use WorkManager's user-initiated data transfer
  • mediaProcessing type added (6-hour timeout) for transcoding, encoding
  • Tighter enforcement on shortService — system more aggressively stops at 3 minutes
  • New restrictions on apps that start too many foreground services
Staff-Level Insight — The Trend

Google is systematically eliminating foreground services as a general-purpose background execution mechanism. The direction is clear: foreground services should only be used for work that is genuinely user-facing and ongoing (music, navigation, calls). Everything else should use WorkManager. When an interviewer asks about background work, demonstrating awareness of this trend shows L6-level platform understanding.
