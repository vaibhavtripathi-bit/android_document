Android Services — Fundamentals

What they are, why they exist, lifecycle internals, and what L6 engineers actually need to know

1. Why Services Exist

A Service is an application component that performs work without a UI. But that one-liner hides the real purpose:

  • Process priority elevation — A Service tells the system "this process is doing useful work, don't kill it yet." Without a Service, a backgrounded app's process can be killed at any time.
  • Component lifecycle contract — Unlike spawning a raw thread (which the system knows nothing about), a Service is a managed component with lifecycle callbacks the system can reason about.
Staff-Level Insight

The #1 misconception: "Service = background thread." A Service runs on the main thread by default. It's not a threading mechanism — it's a process-priority mechanism. The system uses the presence of a running Service to decide OOM adjustment scores (via ActivityManagerService). A started service bumps the process to SERVICE_ADJ (500), which is significantly safer than CACHED_APP_MIN_ADJ (900+).

3. Service Lifecycle — The Real Picture

4. onCreate() — What Actually Happens

Called once when the Service is first created. The system calls ActivityThread.handleCreateService() which:

  1. Loads the Service class via the app's ClassLoader
  2. Calls attach() to bind the base Context
  3. Calls onCreate()
Staff-Level Insight

If your Service is already created and you call startService() again, onCreate() is NOT called again — only onStartCommand(). This is a common interview trap. The lifecycle is: onCreate → onStartCommand → onStartCommand → ... → onDestroy.

5. onStartCommand() — The Return Value That Matters

This is the most interview-critical callback. It receives the Intent and must return one of three constants that tell the system what to do if the process is killed:

Return Value Behavior After Process Kill Intent on Restart Use Case
START_NOT_STICKY System does NOT recreate the service N/A Work that can be restarted by the app itself (e.g., periodic polling that will fire again)
START_STICKY System recreates the service null intent Long-running services like music players — you want the service back but don't need the original intent
START_REDELIVER_INTENT System recreates the service Original intent redelivered Work that MUST complete — file uploads, transactions. The intent carries the work payload. 
class MyService : Service() {

    override fun onStartCommand(intent: Intent?, flags: Int, startId: Int): Int {
        // flags tells you HOW this was started:
        //   0                    = normal start
        //   START_FLAG_REDELIVERY = redelivered after kill (only with START_REDELIVER_INTENT)
        //   START_FLAG_RETRY     = retried after abnormal kill

        // startId: unique ID for this particular start request
        // Use with stopSelfResult(startId) to safely stop only if no new work arrived

        return START_REDELIVER_INTENT
    }

    override fun onBind(intent: Intent?): IBinder? = null
}
Staff-Level Insight — stopSelf(startId) vs stopSelf()

stopSelf() stops immediately. stopSelfResult(startId) stops only if startId matches the most recent start request. This prevents a race condition where you finish processing request #1 and call stopSelf(), but request #2 already arrived. Using stopSelfResult(1) would correctly NOT stop the service because startId is now 2. This is exactly what IntentService did internally.

Interview Gotcha

Q: "What happens if onStartCommand receives a null intent?"
This happens when the system recreates a START_STICKY service after process death. Your code MUST handle intent == null. Many production crashes come from force-unwrapping the intent here.

6. IntentService — Why It Existed and Why It Died

Deprecated API 30

IntentService was a convenience wrapper that solved two problems with plain Service:

  1. Automatically created a worker thread (via HandlerThread)
  2. Automatically called stopSelf(startId) after processing each intent

How It Worked Internally

// Simplified AOSP source of IntentService
class IntentService : Service() {
    private lateinit var serviceLooper: Looper
    private lateinit var serviceHandler: ServiceHandler

    private inner class ServiceHandler(looper: Looper) : Handler(looper) {
        override fun handleMessage(msg: Message) {
            onHandleIntent(msg.obj as? Intent)
            stopSelf(msg.arg1) // ← uses startId, not stopSelf()
        }
    }

    override fun onCreate() {
        super.onCreate()
        val thread = HandlerThread("IntentService[$name]")
        thread.start()
        serviceLooper = thread.looper
        serviceHandler = ServiceHandler(serviceLooper)
    }

    override fun onStartCommand(intent: Intent?, flags: Int, startId: Int): Int {
        val msg = serviceHandler.obtainMessage()
        msg.arg1 = startId
        msg.obj = intent
        serviceHandler.sendMessage(msg)
        return START_REDELIVER_INTENT // default, subclasses could override
    }
}

Why Deprecated

  • Background execution limits (Android 8+): You can't reliably call startService() from the background anymore. The system throws IllegalStateException.
  • No foreground service support: IntentService had no built-in way to call startForeground() before the system killed it for violating background limits.
  • Single-threaded queue: Only one intent processed at a time — no parallelism.
  • Replaced by: JobIntentService (also deprecated now) → WorkManager
Staff-Level Insight

The real killer was Android 8's startService() restriction. If your app is in the background and calls context.startService(), it crashes. The alternatives are startForegroundService() (must show notification within 5 seconds) or using WorkManager/JobScheduler. This single change made IntentService and plain started services unreliable for background work.

7. The Main Thread Trap

This deserves its own section because it's the most common misconception:

All Service callbacks — onCreate(), onStartCommand(), onBind(), onDestroy() — run on the main (UI) thread. If you do heavy work directly in onStartCommand(), you'll get an ANR after 20 seconds (foreground) or 200 seconds (background, varies by OEM).

class CorrectService : Service() {
    private val scope = CoroutineScope(SupervisorJob() + Dispatchers.IO)

    override fun onStartCommand(intent: Intent?, flags: Int, startId: Int): Int {
        scope.launch {
            doHeavyWork()
            stopSelf(startId)
        }
        return START_REDELIVER_INTENT
    }

    override fun onDestroy() {
        scope.cancel()
        super.onDestroy()
    }

    override fun onBind(intent: Intent?): IBinder? = null
}

8. Process Priority & OOM Killer

This is the real reason Services exist. The Linux OOM killer uses oom_adj scores (managed by ActivityManagerService) to decide which processes to kill under memory pressure:

Process State OOM Adj Score Kill Priority
Foreground Activity / Foreground Service 0 (FOREGROUND_APP_ADJ) Last to die
Visible Activity 100 (VISIBLE_APP_ADJ) Very safe
Started Service (recent) 500 (SERVICE_ADJ) Moderate
Cached / Empty process 900+ (CACHED_APP_*) First to die
Staff-Level Insight

Service aging: After ~30 minutes, a started service's process gets demoted from SERVICE_ADJ (500) to SERVICE_B_ADJ (800), making it nearly as killable as cached apps. This is why long-running started services are unreliable — the system actively works against them. This was intentional to prevent apps from abusing startService() to stay alive forever. It's also why Foreground Services (with notification) exist — they stay at adj 0.

AttributeWhy It Matters
exported If true, other apps can start/bind to it. Default changed to require explicit declaration in Android 12 (API 31). Always set this explicitly.
process Runs the service in a separate process. Useful for isolation (crash won't kill main app) but adds IPC overhead. Prefix with : for private process.

Android 12+ Breaking Change

If your targetSdk >= 31 and your service has an <intent-filter>, you must explicitly set android:exported="true" or "false". The system no longer infers it. Apps that didn't update crashed on install.

AOSP Deep Dive — How the System Manages Service Processes

ProcessRecord and oom_adj

Every app process is tracked by a ProcessRecord in ActivityManagerService (AMS). AMS continuously recalculates each process's oom_adj score — a number that tells the Linux kernel's low-memory killer (lmkd) which processes to kill first:

oom_adjProcess StateExample
0 (FOREGROUND_APP)Foreground Activity or Foreground ServiceUser is looking at the app, or music is playing with notification
100 (VISIBLE_APP)Visible but not foregroundActivity behind a dialog
200 (PERCEPTIBLE_APP)Doing something the user is aware ofBackground music without FGS (rare)
500 (SERVICE)Running a started serviceService started <30 min ago
700 (SERVICE_B)Old serviceService running >30 min, demoted
900 (CACHED)No active componentsApp in recents but nothing running
Staff-Level Insight — The Kill Path

When memory is low, lmkd (low-memory killer daemon) reads /proc/<pid>/oom_score_adj and sends SIGKILL to the highest-scored processes first. The process gets no callback — no onDestroy(), no onTerminate(). It's an instant kill. This is why you can never rely on onDestroy() for saving critical state. AMS recalculates oom_adj via OomAdjuster.updateOomAdjLocked() whenever a component state changes (Activity paused, Service started, etc.).

The 30-minute service demotion (500 → 700) is AMS's way of penalizing long-running services that don't promote to foreground. After 30 minutes, your service process is nearly as killable as a cached process. This is the system pressure that pushes developers toward WorkManager or Foreground Services.

bindService() — The Binder IPC Flow

Staff-Level Insight — Why Binding Is Async

bindService() returns immediately (boolean success). The onServiceConnected() callback arrives later because: (1) the request goes through Binder IPC to AMS, (2) AMS may need to start the service process, (3) AMS calls onBind() on the service's main thread, (4) the IBinder is delivered back through Binder IPC to the client. This is why you must handle the gap between bindService() and onServiceConnected() — the service isn't available yet. A common race: the Activity is destroyed before onServiceConnected() fires, causing a crash if you don't null-check.
