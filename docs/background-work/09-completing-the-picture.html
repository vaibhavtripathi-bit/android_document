1. FCM as a Background Work Trigger

Most Reliable Cross-OEM Mechanism

Firebase Cloud Messaging is referenced throughout this series as "the most reliable way to trigger background work." Here's the complete picture of how it works as a background execution mechanism.

FCM Delivery Architecture

FCM Message Types

TypePayloadDelivery When BackgroundedWakes From Doze?
Notification message notification key System tray automatically (no app code runs) No
Data message data key only onMessageReceived() called in your FirebaseMessagingService Only if high-priority
Both notification + data Notification shown by system; data available when user taps Only if high-priority
Staff-Level Insight — Always Use Data-Only Messages

For background work triggers, always use data-only messages. Notification messages are handled by the system when the app is backgrounded — your onMessageReceived() is never called. With data-only messages, you control the entire flow: receive data → decide what to do → enqueue WorkManager / start FGS / show custom notification.

The FCM → WorkManager Pattern

class MyFirebaseService : FirebaseMessagingService() {

    override fun onMessageReceived(remoteMessage: RemoteMessage) {
        // This runs on a BACKGROUND THREAD (not main thread)
        // You have ~20 seconds before the system kills this

        val syncType = remoteMessage.data["sync_type"] ?: return

        when (remoteMessage.priority) {
            RemoteMessage.PRIORITY_HIGH -> {
                // High-priority: runs immediately, even in Doze
                val request = OneTimeWorkRequestBuilder<SyncWorker>()
                    .setExpedited(OutOfQuotaPolicy.RUN_AS_NON_EXPEDITED_WORK_REQUEST)
                    .setInputData(workDataOf("sync_type" to syncType))
                    .build()
                WorkManager.getInstance(this).enqueue(request)
            }
            else -> {
                // Normal priority: may be batched/deferred
                val request = OneTimeWorkRequestBuilder<SyncWorker>()
                    .setInputData(workDataOf("sync_type" to syncType))
                    .build()
                WorkManager.getInstance(this).enqueue(request)
            }
        }
    }

    override fun onNewToken(token: String) {
        // Send token to your server
        val request = OneTimeWorkRequestBuilder<TokenUploadWorker>()
            .setInputData(workDataOf("token" to token))
            .setConstraints(Constraints.Builder()
                .setRequiredNetworkType(NetworkType.CONNECTED).build())
            .build()
        WorkManager.getInstance(this).enqueue(request)
    }
}

FCM Priority & Quotas

PriorityBehaviorQuotaUse Case
High Wakes device from Doze, delivered immediately ~10/min per device (Google throttles abuse) Chat messages, urgent sync, incoming calls
Normal (default) Batched, deferred in Doze, respects App Standby No hard limit Non-urgent updates, content refresh
Staff-Level Insight — High-Priority FCM Is the Only Reliable Cross-OEM Wake

High-priority FCM is the only mechanism that reliably wakes apps on all OEMs including Xiaomi, Huawei, and Samsung. This is because FCM uses Google Play Services, which OEMs cannot kill (it's a system-level service). The flow is: Google server → GCM/FCM connection (maintained by Play Services) → your app's FirebaseMessagingService. Even MIUI's aggressive battery manager doesn't block this path.

This is why at Google scale, the architecture for reliable background sync is: server detects data change → sends high-priority FCM → app receives → enqueues expedited WorkManager request. The periodic WorkManager sync is a fallback, not the primary trigger.

Android 12+ Caveat

High-priority FCM is one of the exemptions that allows starting a foreground service from the background on Android 12+. But there's a catch: the exemption lasts only for a short window (~10 seconds) after onMessageReceived() is called. If you need to start a FGS, do it immediately. Better yet, use WorkManager.setExpedited() which handles the API-level differences for you.

2. Exact Alarm Restrictions

Exact alarms are the last remaining way to execute code at a precise time. Google has been progressively restricting them:

The Two Permissions

PermissionIntroducedWho Gets ItUser Can Revoke?
SCHEDULE_EXACT_ALARM Android 12 (API 31) Any app that declares it. Granted by default on API 31-32. Yes, on Android 13+ (Settings → Alarms & Reminders)
USE_EXACT_ALARM Android 14 (API 34) Only alarm/timer/calendar apps. Auto-granted, cannot be revoked. No 
// Check if you can schedule exact alarms
val alarmManager = getSystemService(AlarmManager::class.java)
if (alarmManager.canScheduleExactAlarms()) {
    alarmManager.setExactAndAllowWhileIdle(
        AlarmManager.RTC_WAKEUP,
        triggerTimeMillis,
        pendingIntent
    )
} else {
    // Redirect user to settings
    startActivity(Intent(Settings.ACTION_REQUEST_SCHEDULE_EXACT_ALARM))
}

// Listen for permission grant/revoke
class AlarmPermissionReceiver : BroadcastReceiver() {
    override fun onReceive(context: Context, intent: Intent) {
        if (intent.action == AlarmManager.ACTION_SCHEDULE_EXACT_ALARM_PERMISSION_STATE_CHANGED) {
            // Re-check canScheduleExactAlarms() and reschedule if needed
        }
    }
}
Staff-Level Insight — When to Use Which

USE_EXACT_ALARM is for apps whose core purpose is time-based — alarm clocks, timers, calendar apps. Google Play will reject apps that declare it without justification.

SCHEDULE_EXACT_ALARM is for everything else that needs exact timing — medication reminders, scheduled tasks, etc. But since Android 13, users can revoke it, so your app must handle the revocation gracefully (fall back to inexact alarms or WorkManager).

The interview-critical point: most apps should NOT use exact alarms. If your work is "sync every hour," use WorkManager. Exact alarms are only for user-facing, time-critical events where a 10-minute delay is unacceptable.

3. Background Location Access

Location is the most restricted background capability in modern Android. The restrictions evolved across multiple API levels:

API LevelChange
API 29 (Android 10)ACCESS_BACKGROUND_LOCATION introduced. Must be requested separately from foreground location. User sees "Allow all the time" option.
API 30 (Android 11)Can't request background location in the same dialog as foreground. Must request foreground first, then separately request background. System shows "Allow all the time" in Settings only.
API 31 (Android 12)Foreground service with location type can access location while-in-use. Background location still requires ACCESS_BACKGROUND_LOCATION.
Play Store PolicyApps must justify background location access. Google reviews and can reject apps that don't have a clear user-facing reason.

The Three Location Access Tiers

// Step 1: Request foreground location first
val foregroundRequest = arrayOf(
    Manifest.permission.ACCESS_FINE_LOCATION,
    Manifest.permission.ACCESS_COARSE_LOCATION
)
requestPermissions(foregroundRequest, FOREGROUND_LOCATION_CODE)

// Step 2: ONLY AFTER foreground is granted, request background
// (Must be a separate request on API 30+)
if (shouldRequestBackground) {
    requestPermissions(
        arrayOf(Manifest.permission.ACCESS_BACKGROUND_LOCATION),
        BACKGROUND_LOCATION_CODE
    )
    // On API 30+, this opens Settings instead of a dialog
}
Staff-Level Insight — The Navigation App Pattern

Most navigation apps (Google Maps, Waze) use Tier 2: while-in-use with a foreground service. They don't need ACCESS_BACKGROUND_LOCATION because the FGS with foregroundServiceType="location" provides continuous location access as long as the notification is showing. This is the recommended pattern for any app that needs location while the user is actively using a feature but the screen might be off.

Geofencing is the main use case for Tier 3 (true background location). Google's Geofencing API (GeofencingClient) handles this with minimal battery impact by using cell tower and WiFi signals instead of GPS.

4. The Master Decision Matrix

This is the single most important table for interview preparation. When an interviewer says "how would you do X in the background?", this is your mental model:

Scenario API Thread? Survives Process Death? Survives Reboot?
Quick computation while UI is visible Coroutine in ViewModel Yes (Dispatchers.Default/IO) No No
Network call while UI is visible Coroutine in ViewModel Yes (Dispatchers.IO) No No
Must finish even if user leaves screen WorkManager (OneTime) Yes (managed) Yes Yes
Must finish immediately + survive death WorkManager (Expedited) Yes (managed) Yes Yes
Periodic sync (every 1h+) WorkManager (Periodic) Yes (managed) Yes Yes
User-visible ongoing work (music, nav, calls) Foreground Service You manage Yes (with START_STICKY) With BOOT_COMPLETED
Long-running work (>10 min, user-triggered) WorkManager + setForeground() Yes (managed) Yes Yes
User-facing alarm (alarm clock, reminder) AlarmManager (exact) No (triggers receiver) Yes (persisted) With BOOT_COMPLETED
Server-initiated urgent sync FCM (high) → WorkManager Yes (managed) Yes N/A (server triggers)
Cross-process communication Bound Service (AIDL) Binder thread pool Depends on binding No
Simple timer/delay in UI Handler.postDelayed() / delay() Main or specified No No
Staff-Level Insight — The Anti-Patterns

Common wrong answers in interviews:

  • "Use a Service for network calls" — No. If the UI is visible, use a coroutine in ViewModel. If it must survive, use WorkManager. A raw Service adds complexity with no benefit.
  • "Use a Thread for background work" — Never use raw Thread in modern Android. Use coroutines (structured concurrency, cancellation, exception handling). The system doesn't know about your threads and will kill the process.
  • "Use AlarmManager for periodic sync" — No. AlarmManager is for user-facing time-critical events. WorkManager is for periodic background work.
  • "Use a foreground service for data sync" — Only if the user explicitly triggered it and needs to see progress. Otherwise, WorkManager.

5. Implicit Intent Restrictions for Services

Since Android 5.0 (API 21), you cannot start a service with an implicit intent. The system throws IllegalArgumentException.

// BAD — crashes on API 21+
val intent = Intent("com.example.MY_ACTION")
startService(intent)

// GOOD — explicit intent
val intent = Intent(this, MyService::class.java)
startService(intent)

// GOOD — explicit with component name (for cross-app)
val intent = Intent().apply {
    component = ComponentName("com.other.app", "com.other.app.MyService")
}
startService(intent)

Why This Matters

  • Security: Implicit intents could be intercepted by malicious apps that register matching intent filters
  • Determinism: Multiple services could match an implicit intent — the system can't choose
  • Android 12+ exported requirement: If your service has an intent-filter and exported="true", any app can start it. Without an intent-filter, only your app can start it (explicit intent required).
Staff-Level Insight

This restriction applies to startService() and bindService(), but NOT to sendBroadcast() (implicit broadcasts are still allowed at runtime, just restricted from manifest registration on API 26+). The distinction matters because the security model is different: a broadcast can be received by multiple apps (by design), but a service should only be started by the intended caller.

6. Legacy APIs — Know They Exist, Know They're Dead

An L6 candidate should recognize these and explain why they were replaced:

SyncAdapter + AccountManager

Effectively Dead

The original Android sync framework. You'd create an AccountAuthenticator, a SyncAdapter, and a stub ContentProvider. The system would batch syncs across apps and manage scheduling. It was powerful but absurdly complex — required 3 components, XML declarations, and a fake ContentProvider even if you didn't use one.

Replaced by: WorkManager with PeriodicWorkRequest. Does everything SyncAdapter did with 90% less boilerplate.

DownloadManager

Still Valid (Niche)

System service for downloading files with built-in retry, notification progress, and WiFi-only support. Still valid for simple file downloads where you want system-managed UI. But for anything custom (upload, bidirectional sync, custom progress), use WorkManager + setForeground().

val request = DownloadManager.Request(Uri.parse(url))
    .setTitle("Downloading update")
    .setNotificationVisibility(DownloadManager.Request.VISIBILITY_VISIBLE_NOTIFY_COMPLETED)
    .setAllowedOverMetered(false)
    .setDestinationInExternalPublicDir(Environment.DIRECTORY_DOWNLOADS, "file.zip")

val downloadManager = getSystemService(DownloadManager::class.java)
val downloadId = downloadManager.enqueue(request)

AsyncTask

Deprecated API 30

Single-line mention: replaced by coroutines. If you see it in an interview question, say "deprecated, use coroutines with viewModelScope/lifecycleScope" and move on.

Loader / AsyncTaskLoader

Deprecated

Replaced by ViewModel + LiveData/Flow + coroutines. Not relevant to background work discussion.

7. Final Interview Q&A — Remaining Gaps

Q48 Your app needs to sync data when the server has updates. Design the push notification → sync architecture. L6+

Architecture:

  1. Server detects data change → sends high-priority FCM data message with sync metadata (type, timestamp, delta ID)
  2. App's FirebaseMessagingService.onMessageReceived() fires (even in Doze)
  3. Enqueue expedited WorkManager request with the sync metadata as input data
  4. Worker performs delta sync — fetches only changed data since last sync timestamp
  5. Fallback: Periodic WorkManager request (every 1-2 hours) catches anything missed

Why not just use periodic sync?

  • App Standby Buckets can defer periodic work by up to 24 hours for rarely-used apps
  • Users expect near-real-time updates for messaging/social apps
  • FCM high-priority bypasses Doze and most OEM restrictions
Staff-level addition: The server should track which sync messages were acknowledged by the client. If a client goes silent for too long (FCM delivery failed, device off), the server queues updates and sends a "catch-up" FCM when the device comes back online. This is the pattern used by Gmail, WhatsApp, and most Google apps. Also mention: FCM data messages have a 4KB payload limit — send sync metadata (what changed), not the actual data.
Q49 When should you use a coroutine vs a Service vs WorkManager? Give concrete examples. L5
UseExampleWhy Not the Others
Coroutine (viewModelScope) Fetching a list of items when user opens a screen UI is visible, no need to survive process death. Service is overkill. WorkManager is for guaranteed/deferred work.
WorkManager Uploading a photo the user just took Must complete even if user leaves app. Coroutine dies with ViewModel. Service doesn't survive background limits.
Foreground Service Playing music while user browses other apps User-facing ongoing work. WorkManager isn't for continuous tasks. Coroutine dies when UI dies.

The rule of thumb:

  • If the user is looking at the screen → coroutine
  • If the work must finish regardless → WorkManager
  • If the user should see it's happening → Foreground Service
What juniors miss: They default to Service for everything. The modern answer is: you almost never need a raw Service. Coroutines handle in-process work, WorkManager handles guaranteed background work, and Foreground Services handle user-visible ongoing work. A raw started Service is only needed for very specific cases (like a music player that's also a bound service for UI control).
Q50 What is the difference between SCHEDULE_EXACT_ALARM and USE_EXACT_ALARM? When would you use each? L6

SCHEDULE_EXACT_ALARM (API 31+): For apps that need exact timing but aren't primarily alarm/timer apps. Granted by default on API 31-32, but user-revocable on API 33+. Your app must handle revocation gracefully.

USE_EXACT_ALARM (API 34+): For apps whose core function is time-based (alarm clocks, timers, calendar). Auto-granted and cannot be revoked. But Google Play rejects apps that declare it without justification.

Decision:

  • Building an alarm clock app? → USE_EXACT_ALARM
  • Building a medication reminder? → SCHEDULE_EXACT_ALARM + handle revocation
  • Building a sync feature? → Neither. Use WorkManager.
Staff-level addition: When SCHEDULE_EXACT_ALARM is revoked, your pending exact alarms are cancelled silently. You must listen for ACTION_SCHEDULE_EXACT_ALARM_PERMISSION_STATE_CHANGED broadcast and reschedule. If you can't get exact alarms, fall back to setWindow() (inexact but within a window) or WorkManager with setInitialDelay().
Q51 How does background location access work on Android 11+? Walk through the permission flow. L6

The two-step flow (API 30+):

  1. Request foreground location first: ACCESS_FINE_LOCATION / ACCESS_COARSE_LOCATION. User sees standard permission dialog.
  2. After foreground is granted, request background: ACCESS_BACKGROUND_LOCATION. On API 30+, this does NOT show a dialog — it opens the app's Settings page where the user must manually select "Allow all the time."

You cannot request both in the same call. If you include ACCESS_BACKGROUND_LOCATION with foreground permissions, the system silently ignores the background request.

Three tiers of location access:

  • Foreground only: Location available only while Activity is visible
  • While-in-use (FGS): Foreground service with foregroundServiceType="location" — location available while notification is showing
  • Background (always): ACCESS_BACKGROUND_LOCATION — location available anytime, no FGS needed
Staff-level addition: Google Play requires a privacy policy and in-app disclosure for background location. Apps must explain why they need "all the time" access. Google reviews this manually and rejects apps without justification. The practical advice: use while-in-use (FGS) for navigation/fitness, and Geofencing API for location-triggered actions (it uses background location internally but is more battery-efficient than continuous GPS).
Q52 What was SyncAdapter and why was it replaced? If you see it in a legacy codebase, how do you migrate? L5

SyncAdapter was Android's original sync framework. It required three components: an AccountAuthenticator (even if your app doesn't use accounts), a ContentProvider (even if you don't expose data), and the SyncAdapter itself. The system batched syncs across apps for battery efficiency.

Why it died:

  • Absurd boilerplate — 3 components + XML for a simple periodic sync
  • Tied to the Account system — meaningless for most apps
  • No constraint support beyond "network available"
  • WorkManager does everything better with 10x less code

Migration path:

  1. Replace the sync logic with a CoroutineWorker
  2. Replace ContentResolver.requestSync() with WorkManager.enqueueUniquePeriodicWork()
  3. Remove the AccountAuthenticator, stub ContentProvider, and SyncAdapter XML
  4. If the app uses accounts for authentication, keep AccountManager but decouple it from sync
What juniors miss: They've never seen SyncAdapter and can't explain what it was. At L6, you should know it existed, why it was the right design in 2010 (battery-efficient batched sync before JobScheduler existed), and why WorkManager replaced it. Showing you understand the historical evolution demonstrates platform maturity.
Q53 Summarize the complete evolution of Android background work restrictions from API 21 to API 34 in under 60 seconds. L6+

The 60-second answer:

API 21: JobScheduler introduced constraint-based scheduling. API 23: Doze mode defers all background work when device is idle. API 26: the big one — background services killed after 1 minute, startService() from background throws, implicit broadcasts blocked from manifest. This forced the industry to move from "I'll run whenever I want" to "I'll declare constraints and let the system schedule." API 28: WorkManager launched as the unified Jetpack solution. API 29: background location restricted. API 30: IntentService deprecated, camera/mic FGS restricted from background. API 31: foreground services can't start from background at all (with exemptions), expedited jobs added. API 33: notification permission required (affects FGS visibility), exact alarms revocable. API 34: foreground service types mandatory, dataSync gets 6-hour timeout, SCHEDULE_EXACT_ALARM vs USE_EXACT_ALARM split.

The theme: Google systematically moved Android from "apps control their own execution" to "apps declare intent, system controls execution." The only surviving escape hatches are foreground services (for genuinely user-facing work) and high-priority FCM (for server-initiated urgency).

Staff-level addition: Being able to tell this story coherently in an interview — connecting the dots across API levels and explaining the why behind each restriction — is what separates L6 from L5. It shows you understand Android as an evolving platform with design philosophy, not just a collection of APIs.
