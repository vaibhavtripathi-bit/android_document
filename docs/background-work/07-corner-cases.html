Corner Cases & Production Traps

27 questions covering edge cases, OEM nightmares, process death, BroadcastReceiver traps, wakelocks, PendingIntent security, debugging tools, and advanced WorkManager patterns. These are the questions that separate L5 from L6.

Section A: Service Lifecycle Traps

Q21 What happens when you call startService() on an already-running service? What about bindService()? L5

startService() on a running service: onCreate() is NOT called again. Only onStartCommand() is called with the new intent and a new startId. The service is the same instance.

bindService() on a running service: If the service was already created (via start or prior bind), onCreate() is skipped. onBind() is called only for the first binding client. Subsequent clients get the cached IBinder via onServiceConnected() without triggering onBind() again — unless onUnbind() previously returned true, in which case onRebind() is called.

What juniors miss: They think every startService() call creates a new service instance. It doesn't. There's only ever ONE instance. Multiple startService() calls result in multiple onStartCommand() calls with incrementing startId values — which is why stopSelfResult(startId) exists.
Q22 What is onTaskRemoved() and why does it matter for services? L6

onTaskRemoved(Intent rootIntent) is called when the user swipes the app away from the Recents screen. This is NOT the same as onDestroy().

The subtle behavior:

  • On stock Android: the service continues running after task removal (if started). onTaskRemoved() is a notification, not a kill signal.
  • On many OEM ROMs (Xiaomi, Huawei, Samsung, Oppo): the system force-kills the entire process on task removal, regardless of running services. onTaskRemoved() may or may not be called before the kill.
  • For START_STICKY services: the system should recreate the service after the kill, but OEMs often prevent this too.
class MyService : Service() {
    override fun onTaskRemoved(rootIntent: Intent?) {
        super.onTaskRemoved(rootIntent)
        // Last chance to save state, reschedule via WorkManager,
        // or set an alarm as a backup restart mechanism
    }
}
Staff-level addition: This is one of the most painful production issues in Android. At Google scale, teams track service survival rates by OEM and Android version. The mitigation is to never rely on a service staying alive — always persist critical state and use WorkManager for guaranteed re-execution. Some apps use onTaskRemoved() to enqueue a WorkManager request as a "dead man's switch."
Q23 What happens if you bind to a service from an Activity and the Activity is destroyed by a configuration change? L5

If you bound in onCreate() and unbind in onDestroy(), the config change (rotation) triggers onDestroy() → unbind → onStop() → new Activity onCreate() → rebind. If the service has no other clients and isn't started, it gets destroyed and recreated. This causes unnecessary service restarts.

Worse scenario: If you forget to unbind in onDestroy(), the system logs a ServiceConnectionLeaked error and force-unbinds. The service may be destroyed unexpectedly.

Correct pattern:

  • Bind in onStart(), unbind in onStop() — symmetric lifecycle pair that survives config changes
  • Or bind from a ViewModel / Application scope if the service must outlive the Activity
What juniors miss: They bind in onCreate() and unbind in onDestroy(), thinking these are symmetric. They are for Fragments, but for Activities, config changes call both, causing unnecessary service churn. The correct pair for Activities is onStart()/onStop().
Q24 Can a Service start an Activity? What are the restrictions? L5

Technically yes, but with severe restrictions since Android 10 (API 29):

  • Android 10+: Background activity starts are blocked. A service running in the background cannot call startActivity() — the system silently drops it.
  • Exemptions: The app has a visible window, was recently in the foreground, has SYSTEM_ALERT_WINDOW, or the activity start is from a PendingIntent created by the system (notification tap).
  • Alternative: Show a high-priority notification with a full-screen intent (setFullScreenIntent()) — this is how incoming call screens work.
// The correct pattern for "launching UI from background"
val fullScreenIntent = Intent(this, IncomingCallActivity::class.java)
val fullScreenPendingIntent = PendingIntent.getActivity(
    this, 0, fullScreenIntent,
    PendingIntent.FLAG_UPDATE_CURRENT or PendingIntent.FLAG_IMMUTABLE
)

val notification = NotificationCompat.Builder(this, CHANNEL_ID)
    .setSmallIcon(R.drawable.ic_call)
    .setContentTitle("Incoming call")
    .setPriority(NotificationCompat.PRIORITY_HIGH)
    .setCategory(NotificationCompat.CATEGORY_CALL)
    .setFullScreenIntent(fullScreenPendingIntent, true)
    .build()
Staff-level addition: The background activity start restriction broke many VoIP and messaging apps. Google's prescribed pattern is setFullScreenIntent() which shows the Activity when the screen is off, or heads-up notification when the screen is on. On Android 14+, even USE_FULL_SCREEN_INTENT permission is restricted to specific app categories (calls, alarms). Apps must request the permission and can be revoked.

Section B: OEM & Device-Specific Nightmares

Q25 Why do background services and WorkManager behave differently on Xiaomi, Samsung, and Huawei devices? L6

OEMs add their own battery optimization layers on top of stock Android's Doze and App Standby:

OEMSystemImpact
XiaomiMIUI Battery Saver, Autostart ManagerKills background processes aggressively. Apps must be manually whitelisted in "Autostart" settings. START_STICKY services are NOT restarted.
HuaweiEMUI Power ManagerThree tiers: "Don't optimize", "Optimize", "Force close". Default is "Optimize" which kills services after screen off. Also blocks RECEIVE_BOOT_COMPLETED.
SamsungDevice Care / Sleeping Apps"Deep sleeping apps" list prevents any background execution. Less aggressive than Xiaomi/Huawei but still breaks START_STICKY and alarms.
Oppo/VivoColorOS/FuntouchOS BatterySimilar to Xiaomi. Kills on swipe from recents. Autostart permission required.
OnePlusOxygenOS Battery OptimizationAggressive in recent versions. "Optimized" mode delays jobs significantly.
Staff-level addition: This is arguably the biggest real-world challenge in Android background work. Stock AOSP behavior is well-documented, but 70%+ of devices run OEM-modified ROMs. Mitigations:
  • Guide users to whitelist your app (show OEM-specific instructions — libraries like AutoStarter detect the OEM and deep-link to the right settings page)
  • Use FCM high-priority messages as the most reliable cross-OEM wake mechanism
  • Track background execution success rates by OEM in your analytics
  • Never assume START_STICKY will actually restart — always have a WorkManager fallback

At Google, the Android team maintains dontkillmyapp.com which ranks OEMs by how badly they break background execution.

Q26 How does WorkManager handle device-specific quirks internally? L6

WorkManager includes internal workarounds for known OEM issues:

  • Delegation fallback chain: If JobScheduler fails (some OEMs have buggy implementations), WorkManager falls back to AlarmManager + BroadcastReceiver
  • RescheduleReceiver: Listens for BOOT_COMPLETED and TIME_SET to re-register work after reboot or time changes
  • ForceStopRunnable: On initialization, WorkManager checks if the app was force-stopped and reschedules any interrupted work
  • Room DB persistence: Even if the process is killed by an OEM battery manager, the work is still in the DB. Next time the process starts (via any trigger — notification tap, JobScheduler, alarm), WorkManager reads the DB and re-executes
Staff-level addition: WorkManager's source code (it's open-source in AOSP/Jetpack) contains specific workarounds for Samsung, Huawei, and other OEMs. The WorkManagerImpl.isDeviceIdleMode() check and the ConstraintTrackingWorker wrapper exist specifically to handle OEM-specific constraint-checking bugs. This is a major reason Google recommends WorkManager over raw JobScheduler — the library absorbs the OEM fragmentation pain.

Section C: Process Death & State

Q27 What is the difference between a service running in a separate process vs the main process? When would you choose each? L6

Declaring android:process=":background" in the manifest runs the service in a separate Linux process with its own:

  • Dalvik/ART VM instance
  • Memory heap
  • Application instance (onCreate() called again)
  • Static variables (NOT shared with main process)

Advantages:

  • Crash isolation: Service crash doesn't kill the UI process
  • Memory: Separate OOM adj — system can kill the service process without killing the UI
  • Security: Different Linux UID possible (though rare)

Disadvantages:

  • IPC overhead: All communication goes through Binder (serialization/deserialization)
  • No shared memory: Can't share objects, singletons, or static state directly
  • Double initialization: Application class, DI graph, etc. all initialize twice → slower startup, more memory
  • Debugging complexity: Two processes to attach debugger to
Staff-level addition: Multi-process services are used at Google scale for crash isolation in critical paths (e.g., WebView runs in a separate process since Android 8). But for most apps, the IPC overhead and double-initialization cost outweigh the benefits. A common interview mistake is suggesting a separate process without acknowledging the Application.onCreate() double-call — your DI setup, analytics init, and crash reporting all run twice. Use ProcessLifecycleOwner or check getProcessName() to conditionally initialize.
Q28 A WorkManager worker is running when the user force-stops the app. What happens? L6

Force-stop (Settings → App → Force Stop) is the nuclear option. It:

  • Kills all processes immediately
  • Cancels ALL pending alarms
  • Cancels ALL pending JobScheduler jobs
  • Puts the app in a "stopped state" — it won't receive any implicit broadcasts (including BOOT_COMPLETED)
  • The app stays stopped until the user manually launches it again

WorkManager's Room DB still has the WorkSpecs, but the system schedulers (JobScheduler, AlarmManager) are all cancelled. The work will NOT execute until the user opens the app again, at which point WorkManager re-initializes, reads the DB, and re-schedules.

What juniors miss: They say "WorkManager guarantees execution." It does — except after force-stop. Force-stop is the one scenario where nothing can save you. The work isn't lost (it's in the DB), but it won't run until the user manually opens the app. This is by design — Google considers force-stop as the user explicitly saying "I don't want this app doing anything."
Q29 What happens to a bound service when the binding Activity's process is killed by the system (low memory)? L6

If the Activity's process is killed (not the service's process):

  • The system automatically unbinds the dead client
  • If this was the last client and the service isn't started, the service is destroyed
  • onServiceDisconnected() is NOT called on the dead client (the client's process is gone)
  • If the service is in a separate process, it continues running (if also started) or gets destroyed (if only bound)

When the Activity is recreated (user returns), it must re-bind. There's no automatic reconnection.

Staff-level addition: This is why the started+bound hybrid pattern exists for critical services (like music players). The startService() keeps it alive independently of any bindings. If all UI clients die, the service persists. When the UI comes back, it re-binds and gets the existing service instance. Without the started component, the service dies with its last client.

Section D: WorkManager Edge Cases

Q30 What happens to WorkManager's pending work when the user clears app data? L5

All pending work is lost. WorkManager persists WorkSpecs in a Room database stored in the app's internal storage (/data/data/<package>/databases/). Clearing app data deletes this database.

Unlike force-stop (where the DB survives but schedulers are cancelled), clearing data destroys the DB itself. There is no recovery.

Staff-level addition: This also happens on app updates if the app uses allowBackup="false" and the update triggers a data migration that fails. In practice, critical work should have a server-side fallback — the server should know what work is pending and can re-trigger via FCM if the client goes silent. Never rely solely on client-side persistence for business-critical operations.
Q31 Can a PeriodicWorkRequest run more frequently than every 15 minutes? What are the workarounds? L5

No. The minimum interval is 15 minutes, enforced by both WorkManager and the underlying JobScheduler. If you pass a smaller value, it's silently clamped to 15 minutes.

Workarounds for more frequent execution:

  • Foreground service with a coroutine loop — runs as frequently as you want, but requires a notification
  • Self-rescheduling OneTimeWorkRequest: At the end of doWork(), enqueue a new OneTimeWorkRequest with setInitialDelay(5, TimeUnit.MINUTES). This is a hack and has caveats (no flex window, retry logic is manual)
  • AlarmManager with setExactAndAllowWhileIdle() — but this is heavily restricted in Doze and on OEMs
What juniors miss: The self-rescheduling pattern seems clever but has a critical flaw: if the worker fails and returns Result.failure(), the chain breaks — no new work is enqueued. You need explicit failure handling to re-enqueue even on failure. Also, each execution is a separate WorkSpec in the DB, so you lose the "periodic" semantics (no ExistingPeriodicWorkPolicy).
Q32 What is the WorkManager 10-minute execution limit and how does it interact with process death? L6

When WorkManager delegates to JobScheduler, the job inherits JobScheduler's 10-minute execution limit. After 10 minutes, the system calls onStopJob() / the Worker's onStopped().

The interaction with process death:

  • If the worker is stopped at 10 minutes and returns Result.retry(), it's re-enqueued with backoff
  • If the process is killed during execution (before 10 min), the work is treated as a retry — runAttemptCount increments
  • If you use setForeground(), the 10-minute limit is removed because the work is promoted to a foreground service
  • On Android 12+, expedited jobs have their own quota-based limits separate from the 10-minute rule
Staff-level addition: The 10-minute limit is actually a JobScheduler limit, not a WorkManager limit. When the GreedyScheduler runs work in-process (app is alive, constraints met), there is no 10-minute limit — the work runs until completion. The limit only applies when the SystemJobScheduler triggers the work. This means the same worker can have different time limits depending on which scheduler triggered it. This is a subtle but important distinction for designing long-running workers.
Q33 How do you cancel WorkManager work safely? What are the pitfalls? L5 
// By ID
WorkManager.getInstance(context).cancelWorkById(workRequest.id)

// By unique name
WorkManager.getInstance(context).cancelUniqueWork("sync_work")

// By tag
WorkManager.getInstance(context).cancelAllWorkByTag("upload")

// Nuclear option
WorkManager.getInstance(context).cancelAllWork()

Pitfalls:

  • Cancellation is not immediate: If doWork() is running, onStopped() is called but the coroutine/thread isn't automatically killed. You must check isStopped periodically in your work loop.
  • Chained work: Cancelling one worker in a chain cancels ALL dependent (downstream) workers
  • Periodic work: cancelWorkById() cancels the entire periodic schedule, not just the current execution
  • Race condition: If you cancel and immediately re-enqueue with the same unique name, the cancel might arrive after the enqueue, killing the new work
class CancellableWorker(...) : CoroutineWorker(...) {
    override suspend fun doWork(): Result {
        for (item in largeList) {
            if (isStopped) return Result.failure() // check cancellation
            processItem(item)
        }
        return Result.success()
    }
}
What juniors miss: They call cancelWorkById() and assume the work stops instantly. It doesn't. The worker must cooperatively check isStopped. If your doWork() is a single long-running network call with no cancellation check, the work continues until the call completes or the process is killed. For coroutines, the job is cancelled but only if you're using suspending functions that check for cancellation (like delay(), yield(), or ensureActive()).

Section E: Scenario-Based (L6+ Design)

Q34 Your app's periodic sync worker has a 40% failure rate on Xiaomi devices. How do you debug and fix this? L6+

Debugging:

  1. Check if it's autostart-related: Xiaomi's MIUI blocks background execution for apps not in the "Autostart" whitelist. Check if the worker is being scheduled but never executed (WorkSpec stays ENQUEUED).
  2. Check battery optimization: MIUI has its own battery saver that kills processes independently of Android's Doze. Check if the process is being killed before the worker completes.
  3. Check runAttemptCount: If it's high, the worker is being interrupted repeatedly (process kills mid-execution).
  4. Server-side monitoring: Track sync completion rate by device manufacturer, model, and OS version.

Fixes:

  • Guide users to whitelist: Detect Xiaomi devices (Build.MANUFACTURER) and show a one-time prompt linking to Autostart settings
  • Use FCM high-priority as trigger: Instead of relying solely on periodic work, have the server send a high-priority FCM that triggers an expedited WorkManager request. FCM high-priority bypasses most OEM restrictions.
  • Reduce work duration: Shorter workers are less likely to be killed. Break large syncs into smaller chunks.
  • Implement idempotent, resumable sync: Each execution picks up where the last one left off, so partial kills don't lose progress.
Staff-level addition: At Google, teams maintain OEM-specific compatibility matrices. The dontkillmyapp.com project (endorsed by Google) documents these issues. For critical apps, the solution is usually a combination of: (1) FCM-triggered sync as primary, (2) periodic WorkManager as fallback, (3) user-facing whitelist prompt, and (4) server-side monitoring with automatic FCM retry if the client goes silent for too long.
Q35 You need to run a service that plays music, allows the user to control it from a notification, AND survives the app being swiped from recents. Design this. L6+

Architecture:

  1. Foreground Service with foregroundServiceType="mediaPlayback"
  2. Started + Bound hybrid: startForegroundService() to keep it alive, bindService() from the UI for control
  3. MediaSession: Use MediaSessionCompat for lock screen controls, Bluetooth, and notification media controls
  4. MediaStyle notification: With play/pause/skip actions via PendingIntent

Surviving swipe from recents:

  • The foreground service with notification keeps the process alive on stock Android
  • Override onTaskRemoved() — on OEM ROMs that kill the process, use it to save playback state
  • Return START_STICKY from onStartCommand() so the system recreates the service
  • On recreation, read saved state and resume playback

Edge cases:

  • Audio focus: Request audio focus via AudioManager. Handle AUDIOFOCUS_LOSS_TRANSIENT (duck or pause) vs AUDIOFOCUS_LOSS (stop)
  • Becoming noisy: Register ACTION_AUDIO_BECOMING_NOISY receiver to pause when headphones are unplugged
  • Android 14+: Must declare FOREGROUND_SERVICE_MEDIA_PLAYBACK permission
  • Android 13+: Must have POST_NOTIFICATIONS permission or the foreground notification is invisible (service still runs but user can't see it)
Staff-level addition: The modern approach (2024+) is to use Media3 (androidx.media3) with MediaSessionService, which handles most of this boilerplate — foreground service management, MediaSession, notification with controls, audio focus, and becoming-noisy handling. Mentioning Media3 in an interview shows you're current. Also discuss the MediaBrowserServiceCompat pattern for Android Auto / Wear OS integration, where external clients browse and control your media library.

Section F: BroadcastReceivers & Service Interaction

Q36 How do implicit broadcast restrictions (Android 8+) affect the BroadcastReceiver → Service pattern? L6

Before Android 8, a common pattern was registering a BroadcastReceiver in the manifest for CONNECTIVITY_CHANGE, BOOT_COMPLETED, etc., and starting a service from onReceive(). Android 8 broke this in two ways:

  • Most implicit broadcasts are blocked for manifest-registered receivers. CONNECTIVITY_CHANGE, ACTION_NEW_PICTURE, ACTION_NEW_VIDEO, and many others no longer wake your app.
  • startService() from background throws IllegalStateException. Even if your receiver fires, you can't start a regular service.

Exempted broadcasts (still work from manifest):

  • BOOT_COMPLETED, LOCALE_CHANGED, MY_PACKAGE_REPLACED
  • ACTION_USB_DEVICE_ATTACHED, ACTION_HEADSET_PLUG
  • SMS_RECEIVED_ACTION, WAP_PUSH_RECEIVED_ACTION

Modern alternatives:

  • Use registerReceiver() at runtime (only works while app is alive)
  • Use WorkManager with NetworkType.CONNECTED constraint instead of listening for connectivity changes
  • Use startForegroundService() from BOOT_COMPLETED (still allowed)
Staff-level addition: The implicit broadcast ban was the other half of Android 8's background execution overhaul (alongside the startService() restriction). Together, they killed the "wake up on event → start service → do work" pattern. The replacement is declarative: tell WorkManager your constraints, and the system handles the triggering. In an interview, connecting these two restrictions as parts of the same design philosophy shows architectural understanding.
Q37 A BroadcastReceiver's onReceive() has a ~10-second limit. How do you handle work that takes longer? L5

If onReceive() doesn't return within ~10 seconds, the system considers the receiver ANR'd. You cannot start a coroutine or thread from onReceive() and expect it to complete — once onReceive() returns, the process may be killed (the BroadcastReceiver no longer keeps it alive).

Solutions (in order of preference):

  1. WorkManager: Enqueue a OneTimeWorkRequest from onReceive(). Survives process death, constraint-aware.
  2. goAsync(): Returns a PendingResult that extends the timeout to ~30 seconds. You must call pendingResult.finish() when done. Still limited — not for heavy work.
  3. Foreground service: Call startForegroundService() from onReceive() (only works if the broadcast is from an exempted source on API 31+).
class MyReceiver : BroadcastReceiver() {
    override fun onReceive(context: Context, intent: Intent) {
        val pendingResult = goAsync() // extends to ~30 seconds
        CoroutineScope(Dispatchers.IO).launch {
            try {
                doQuickWork()
            } finally {
                pendingResult.finish()
            }
        }
    }
}
What juniors miss: They launch a coroutine from onReceive() without goAsync(), thinking it will complete. It won't — the process can be killed the moment onReceive() returns. goAsync() is the minimum viable fix, but WorkManager is the correct answer for anything non-trivial.

Section G: Wakelocks & Power

Q38 What is a Wakelock, why is it dangerous, and when is it still necessary? L6

A WakeLock prevents the CPU (and optionally the screen) from sleeping. Acquired via PowerManager:

val wakeLock = (getSystemService(POWER_SERVICE) as PowerManager)
    .newWakeLock(PowerManager.PARTIAL_WAKE_LOCK, "myapp:sync")
    .apply { acquire(10 * 60 * 1000L) } // 10-min timeout

Why dangerous:

  • Forgotten wakelocks drain battery — the CPU never sleeps
  • Doze mode ignores wakelocks (they're released by the system)
  • Play Store's Android Vitals flags apps with excessive wakelock usage
  • A wakelock without a timeout is a ticking time bomb

When still necessary:

  • Inside a BroadcastReceiver with goAsync() — the system may sleep before your async work completes
  • Custom alarm-based scheduling (legacy code)
  • Bluetooth/NFC communication that must complete without CPU sleeping

In modern code, you almost never need a wakelock directly. WorkManager, JobScheduler, and foreground services all manage wakelocks internally.

Staff-level addition: The key insight: PARTIAL_WAKE_LOCK keeps the CPU on but lets the screen off. FULL_WAKE_LOCK, SCREEN_DIM_WAKE_LOCK, and SCREEN_BRIGHT_WAKE_LOCK are all deprecated — use WindowManager.LayoutParams.FLAG_KEEP_SCREEN_ON instead. Also, WakefulBroadcastReceiver (which auto-managed wakelocks for receiver→service handoff) is deprecated — use WorkManager. See 08 — Power Management for the full deep-dive.

Section H: PendingIntent & Security

Q39 Explain PendingIntent mutability changes in Android 12 and why they matter for services. L6

Android 12 (API 31) requires you to explicitly declare PendingIntent mutability:

  • PendingIntent.FLAG_IMMUTABLE — the intent cannot be modified by the receiving app. Use this by default.
  • PendingIntent.FLAG_MUTABLE — the intent can be modified. Required only when the PendingIntent needs to be filled in later (e.g., inline reply in notifications, bubble intents).

If you don't specify either flag on API 31+, the system throws IllegalArgumentException.

How this affects services:

  • AlarmManager PendingIntents: Must be FLAG_IMMUTABLE unless you need the alarm to modify the intent
  • Notification actions: Most should be FLAG_IMMUTABLE. Inline reply actions need FLAG_MUTABLE.
  • WorkManager: Handles this internally — no action needed
  • Custom service triggers via PendingIntent: Always use FLAG_IMMUTABLE to prevent intent hijacking
Staff-level addition: The security concern: a mutable PendingIntent can be intercepted and modified by a malicious app. If you create a mutable PendingIntent that starts a service with extras, a malicious app could modify the extras to inject data. This is a real attack vector that Google patched. In an interview, connecting this to the broader Android security model (intent spoofing, permission enforcement) shows L6 thinking.

  • Battery Historian: Visualizes wakelock timelines, job execution, network activity per app

Staff-level addition: At Google scale, teams build dashboards that track: (1) sync success rate by OEM and API level, (2) average job latency by standby bucket, (3) wakelock duration percentiles, (4) FGS crash rate (ForegroundServiceDidNotStartInTimeException). Q41 How do you test WorkManager workers that depend on constraints like network or charging? L5

WorkManager provides a testing artifact (work-testing) with utilities:

// build.gradle
androidTestImplementation "androidx.work:work-testing:2.9.0"

// In test setup
@Before
fun setup() {
    val config = Configuration.Builder()
        .setMinimumLoggingLevel(Log.DEBUG)
        .setExecutor(SynchronousExecutor())
        .build()
    WorkManagerTestInitHelper.initializeTestWorkManager(context, config)
}

// Test: simulate constraints being met
@Test
fun testConstrainedWork() {
    val request = OneTimeWorkRequestBuilder<SyncWorker>()
        .setConstraints(Constraints.Builder()
            .setRequiredNetworkType(NetworkType.CONNECTED)
            .build())
        .build()

    WorkManager.getInstance(context).enqueue(request).result.get()

    // Worker is ENQUEUED (constraints not met in test)
    val testDriver = WorkManagerTestInitHelper.getTestDriver(context)!!
    testDriver.setAllConstraintsMet(request.id) // simulate constraints met

    // Now verify it ran
    val info = WorkManager.getInstance(context)
        .getWorkInfoById(request.id).get()
    assertThat(info.state).isEqualTo(WorkInfo.State.SUCCEEDED)
}
What juniors miss: They try to test WorkManager by actually toggling WiFi or plugging in the charger on a device. The TestDriver API lets you simulate constraint changes programmatically. Also, TestListenableWorkerBuilder lets you unit-test the doWork() logic directly without the scheduling layer.

Section J: Notification & FGS Interaction

Q42 What happens to a foreground service if the user denies POST_NOTIFICATIONS permission on Android 13+? L5

The foreground service still runs. The notification is simply not shown to the user. This creates a paradox: the service is "foreground" (elevated priority, wakelock) but the user has no visibility into it.

Implications:

  • The service gets foreground priority (OOM adj 0) regardless of notification visibility
  • The user can't dismiss or interact with the notification (they can't see it)
  • Play Store policy may flag this as deceptive behavior for certain app categories
  • The notification still appears in the notification shade under "silenced" if the channel exists
Staff-level addition: This is a policy gap that Google is aware of. The practical advice: always request POST_NOTIFICATIONS before starting a foreground service. If denied, consider whether the work can be done via WorkManager instead (no notification needed for expedited work on API 31+). For truly essential foreground services (navigation, calls), explain to the user why the notification matters.
Q43 Can you use notification ID 0 for a foreground service? What are the notification ID rules? L5

No. The notification ID passed to startForeground(id, notification) must be non-zero. Passing 0 silently fails on some Android versions or throws on others.

Other rules:

  • The ID must be unique per app (not per service). Two services using the same ID will overwrite each other's notification.
  • You can update the notification by calling NotificationManager.notify(sameId, updatedNotification)
  • If you call startForeground() with a new ID, the old notification is replaced
  • On Android 13+, the notification channel must exist before startForeground() is called
What juniors miss: Using the same notification ID across multiple foreground services. If Service A and Service B both use ID 1, stopping Service A's foreground state removes the notification for Service B too, potentially causing a crash or the 5-second timeout on the next startForegroundService() call.

Section K: Advanced WorkManager

Q44 How does WorkManager's multiprocess support work? When would you use RemoteListenableWorker? L6+

WorkManager 2.6+ added RemoteListenableWorker and RemoteCoroutineWorker for running workers in a different process than the one that scheduled them.

// In manifest — declare the service in the target process
<service
    android:name="androidx.work.multiprocess.RemoteWorkerService"
    android:process=":worker"
    android:exported="false" />

// Your worker
class CrossProcessWorker(
    context: Context,
    params: WorkerParameters
) : RemoteCoroutineWorker(context, params) {

    override suspend fun doRemoteWork(): Result {
        // Runs in :worker process
        return Result.success()
    }
}

// Schedule from main process — specify the target process
val request = OneTimeWorkRequestBuilder<CrossProcessWorker>()
    .build()
WorkManager.getInstance(context).enqueue(request)

Use cases:

  • Crash isolation: Heavy/risky work (native code, ML inference) in a separate process so crashes don't kill the UI
  • Memory isolation: Memory-intensive work gets its own heap
  • Multi-module apps: Different modules run workers in their own processes
Staff-level addition: The multiprocess API uses AIDL internally to communicate between the scheduling process and the worker process. This means all input/output data must be Parcelable (which Data already is). The main gotcha: Application.onCreate() runs in both processes, so your initialization code (DI, analytics, crash reporting) must handle multi-process correctly or use conditional initialization based on getProcessName().
Q45 What is WorkManager's Data size limit and how do you pass large payloads? L5

WorkManager's Data object has a 10KB limit. If you exceed it, workDataOf() throws IllegalStateException.

Workarounds for large payloads:

  • Pass a URI or file path instead of the data itself. The worker reads the file.
  • Pass a database ID. The worker queries the database for the full payload.
  • Use a shared file in internal storage. Pass the filename as input data.
// Instead of passing the entire image bytes:
val request = OneTimeWorkRequestBuilder<UploadWorker>()
    .setInputData(workDataOf(
        "file_uri" to savedFileUri.toString() // pass URI, not bytes
    ))
    .build()
What juniors miss: They try to serialize large objects into Data and hit the 10KB limit in production. The limit exists because Data is stored in SQLite (WorkManager's Room DB) and passed through Binder for IPC — both have size constraints. Design your workers to accept references (URIs, IDs), not payloads.
Q46 How do you observe and react to ContentProvider changes to trigger background work? L6

JobScheduler supports addTriggerContentUri() to fire a job when a ContentProvider's data changes:

val jobInfo = JobInfo.Builder(JOB_ID, componentName)
    .addTriggerContentUri(
        JobInfo.TriggerContentUri(
            MediaStore.Images.Media.EXTERNAL_CONTENT_URI,
            JobInfo.TriggerContentUri.FLAG_NOTIFY_FOR_DESCENDANTS
        )
    )
    .setTriggerContentUpdateDelay(1000) // debounce: wait 1s after last change
    .setTriggerContentMaxDelay(5000)    // but no more than 5s total
    .build()

WorkManager does NOT directly support content URI triggers. Workarounds:

  • Use ContentObserver registered at runtime + enqueue WorkManager request on change
  • Use JobScheduler for the trigger, then delegate actual work to WorkManager
  • For photo/video capture: use ActivityResultContracts.TakePicture and enqueue work after the result
Staff-level addition: Content URI triggers were designed for apps like Google Photos that need to detect new photos/videos immediately. The debounce parameters (setTriggerContentUpdateDelay / setTriggerContentMaxDelay) are critical — without them, a burst of inserts (e.g., restoring a photo backup) fires your job hundreds of times. The FLAG_NOTIFY_FOR_DESCENDANTS flag means changes to any URI under the specified path trigger the job. WorkManager's lack of content URI support is a known gap — the team recommends the ContentObserver + WorkManager pattern.
Q47 Your WorkManager worker needs to make 3 sequential API calls where each depends on the previous result. How do you handle partial failure? L6+

Option A: Single worker with checkpoint state

class MultiStepWorker(...) : CoroutineWorker(...) {
    override suspend fun doWork(): Result {
        val lastStep = inputData.getInt("last_completed_step", 0)

        return try {
            if (lastStep < 1) {
                val result1 = api.step1()
                saveCheckpoint(1, result1)
            }
            if (lastStep < 2) {
                val result2 = api.step2(loadCheckpoint(1))
                saveCheckpoint(2, result2)
            }
            if (lastStep < 3) {
                api.step3(loadCheckpoint(2))
            }
            clearCheckpoints()
            Result.success()
        } catch (e: IOException) {
            // Re-enqueue with last completed step so we resume
            Result.retry()
        }
    }
}

Option B: Chained workers with data passing

WorkManager.getInstance(context)
    .beginWith(step1Request)
    .then(step2Request)  // gets step1's output as input
    .then(step3Request)
    .enqueue()

Trade-offs:

ApproachProsCons
Single worker + checkpointsSimpler, one WorkSpec in DB, custom retry per stepMust manage checkpoint state manually (SharedPrefs/DB)
Chained workersClean separation, each step independently retryable, observable per stepIf step 2 fails permanently, step 3 is cancelled. Data passing via 10KB Data objects only.
Staff-level addition: At Google scale, the single-worker-with-checkpoints pattern is more common for tightly coupled steps because it avoids the chain-cancellation problem. The checkpoint state is stored in a local database (not WorkManager's Data) to survive the 10KB limit. For loosely coupled steps (filter → compress → upload), chaining is cleaner. The key interview insight: always discuss what happens on partial failure and how each approach handles it.
