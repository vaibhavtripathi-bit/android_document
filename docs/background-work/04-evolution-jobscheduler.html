The Evolution of Background Work

Why Android kept deprecating things — the path from AlarmManager hacks to constraint-based scheduling

1. The Problem Google Was Solving

Before Android 5.0, apps abused background execution freely:

  • AlarmManager with setRepeating() to wake the device every N minutes
  • startService() from BroadcastReceiver on CONNECTIVITY_CHANGE
  • Wakelocks held for minutes while syncing data
  • Multiple apps doing the same thing simultaneously, draining battery

The result: terrible battery life. Google's response was a multi-year effort to move apps from "I'll run whenever I want" to "I'll tell the system what I need, and it'll batch and schedule intelligently."

2. AlarmManager — The Old Way

Not a scheduling API — it's a time-based trigger. Apps abused it for periodic background work:

// The bad old pattern (don't do this for background work)
val alarmManager = getSystemService(AlarmManager::class.java)
val intent = PendingIntent.getBroadcast(this, 0, Intent(this, SyncReceiver::class.java), PendingIntent.FLAG_IMMUTABLE)

alarmManager.setRepeating(
    AlarmManager.ELAPSED_REALTIME_WAKEUP,
    SystemClock.elapsedRealtime(),
    15 * 60 * 1000, // every 15 min — WAKES THE DEVICE
    intent
)

Problems:

  • No batching — each app wakes the device independently
  • No constraint awareness — fires even on metered network, low battery
  • Doze mode kills it — inexact alarms deferred indefinitely in Doze
  • Exact alarms restricted — Android 12+ requires SCHEDULE_EXACT_ALARM permission, Android 13+ can revoke it
Staff-Level Insight

AlarmManager still has valid uses: user-facing alarms (alarm clock apps, calendar reminders, medication reminders). These are time-critical and user-initiated. For everything else (sync, upload, cleanup), use WorkManager. In an interview, showing you understand this distinction matters.

3. JobScheduler (API 21+)

Current (platform API)

Google's first constraint-based scheduler. Instead of "run at time X", you say "run when these conditions are met":

class SyncJobService : JobService() {

    // Runs on MAIN THREAD — you must offload work
    override fun onStartJob(params: JobParameters): Boolean {
        CoroutineScope(Dispatchers.IO).launch {
            performSync(params)
            // MUST call jobFinished() when done
            jobFinished(params, false) // false = don't reschedule
        }
        return true // true = work is ongoing (async)
    }

    // Called if system needs to preempt your job (e.g., constraints no longer met)
    override fun onStopJob(params: JobParameters): Boolean {
        // Cancel your coroutine/thread here
        return true // true = reschedule, false = drop
    }
}

// Scheduling
val jobScheduler = getSystemService(JobScheduler::class.java)
val jobInfo = JobInfo.Builder(JOB_ID, ComponentName(this, SyncJobService::class.java))
    .setRequiredNetworkType(JobInfo.NETWORK_TYPE_UNMETERED)
    .setRequiresCharging(true)
    .setRequiresDeviceIdle(false)
    .setPersisted(true) // survives reboot (needs RECEIVE_BOOT_COMPLETED)
    .setPeriodic(15 * 60 * 1000L) // minimum 15 min
    .setBackoffCriteria(30_000L, JobInfo.BACKOFF_POLICY_EXPONENTIAL)
    .build()

jobScheduler.schedule(jobInfo)

Staff-Level Insight — jobFinished() Is Critical

If you return true from onStartJob() (async work), you MUST call jobFinished(params, needsReschedule) when done. If you forget:

  • The system thinks the job is still running
  • It holds a wakelock on your behalf (battery drain)
  • The job slot is occupied — your next scheduled run may be skipped

This is a production bug that's hard to catch in testing because it manifests as battery drain and missed syncs days later.

Interview Gotcha

Q: "What happens when onStopJob() is called?"
The system calls this when constraints are no longer met (e.g., device went off WiFi) OR under memory pressure. Your job has a few seconds to clean up. If you return true, the system reschedules with back-off. If false, the job is dropped. You do NOT need to call jobFinished() inside onStopJob() — the system already considers the job stopped.

4. Doze Mode & App Standby Buckets

Understanding these is essential because they affect ALL background work APIs:

Doze Mode (API 23+)

When the device is stationary, screen off, and on battery for a while, the system enters Doze. In Doze:

  • Network access is blocked
  • Wakelocks are ignored
  • AlarmManager alarms are deferred
  • JobScheduler jobs are deferred
  • Syncs are deferred

The system opens brief maintenance windows where deferred work can execute, with increasing intervals (1 min → 3 min → 9 min → ... up to hours).

App Standby Buckets (API 28+)

Apps are categorized based on how recently/frequently the user interacted:

BucketJob FrequencyAlarm FrequencyCriteria
Active No restrictions No restrictions Currently in use or very recently used
Working Set Deferred up to 2 hours Deferred up to 6 min Used regularly but not currently active
Frequent Deferred up to 8 hours Deferred up to 30 min Used often but not daily
Rare Deferred up to 24 hours Deferred up to 2 hours Rarely used
Restricted (API 31+) 1 job per day 1 alarm per day User-restricted or never used
Staff-Level Insight — Bucket Assignment

The system uses UsageStatsManager to assign buckets. But here's what juniors miss: push notifications that the user taps count as interaction and can promote your app to a better bucket. FCM high-priority messages also temporarily lift Doze restrictions. At Google scale, teams optimize their notification strategy partly to maintain favorable standby bucket placement.

5. JobIntentService — The Bridge That Also Died

Deprecated

JobIntentService (AndroidX) was a compatibility shim:

  • On API 26+: delegates to JobScheduler
  • On API < 26: uses startService() + wakelock (like old IntentService)
class SyncJobIntentService : JobIntentService() {

    override fun onHandleWork(intent: Intent) {
        // Runs on background thread (like IntentService)
        // System holds a wakelock for you
        performSync()
    }

    companion object {
        fun enqueueWork(context: Context, intent: Intent) {
            enqueueWork(context, SyncJobIntentService::class.java, JOB_ID, intent)
        }
    }
}

Why It Died

  • Unreliable on API 26+: When using JobScheduler internally, the system could stop the job at any time (constraints changed, memory pressure). But JobIntentService gave no way to handle onStopJob() — your work just got killed.
  • No constraint support: You couldn't specify "only on WiFi" or "only when charging"
  • WorkManager does everything better: Same backward compatibility, plus constraints, chaining, observability
Do Not Use

If you see JobIntentService in a codebase, it should be migrated to WorkManager. There is no valid reason to use it in new code. In an interview, mentioning it shows you know the history, but always follow up with "...which is now replaced by WorkManager."

6. GCMNetworkManager — The Forgotten One

Before JobScheduler existed, Google Play Services provided GCMNetworkManager for constraint-based scheduling on older devices. It's relevant because WorkManager uses it internally as a fallback on devices with Play Services but below API 23.

You'll never use it directly, but knowing it exists explains one of WorkManager's internal delegation paths.

7. Decision Matrix — What to Use When

Scenario API Why
User-facing ongoing work (music, navigation) Foreground Service Needs persistent notification, highest priority
Deferrable background work (sync, upload, cleanup) WorkManager Constraint-aware, survives process death, backward compatible
Exact-time user alarms (alarm clock, reminders) AlarmManager Only API for exact timing guarantees
Immediate short task from foreground Coroutine in ViewModel No need for a Service if the UI is visible
Immediate task that must survive process death WorkManager (expedited) setExpedited() runs ASAP with process-death guarantee
Cross-process communication Bound Service + AIDL Only option for rich IPC (deep-dive doc coming)
