L6 Interview Q&A — Background Work

20 curated questions spanning Services → WorkManager. Click to reveal answers. Difficulty tagged by Google level. See also 07 — Corner Cases for 15 more edge-case questions.

Section A: Service Fundamentals

Q1 Does a Service run on a background thread? L4

No. All Service callbacks (onCreate, onStartCommand, onBind, onDestroy) run on the main (UI) thread. A Service provides process-priority elevation, not threading. You must create your own threads or use coroutines for heavy work.

What juniors miss: They say "Service runs in the background" and mean "background thread." The correct framing is: a Service runs without a UI, but on the main thread. "Background" refers to the absence of UI, not the thread.
Q2 Explain the three return values of onStartCommand and when you'd use each. L5

START_NOT_STICKY: System does NOT recreate the service after process kill. Use for work that will be retried by the app itself (e.g., periodic polling that fires again anyway).

START_STICKY: System recreates the service but delivers a null intent. Use for long-running services like music players where you want the service back but don't need the original command.

START_REDELIVER_INTENT: System recreates and redelivers the original intent. Use for work that MUST complete — the intent carries the work payload (file URI, transaction ID).

Staff-level addition: Mention the flags parameter — START_FLAG_REDELIVERY tells you this is a redelivered intent (only with START_REDELIVER_INTENT), and START_FLAG_RETRY indicates the system is retrying after an abnormal kill. Also mention stopSelfResult(startId) for safe stopping without race conditions.
Q3 What is the actual purpose of a Service from the system's perspective? L6

A Service is a process-priority signal to the ActivityManagerService. When a Service is running, the system adjusts the process's OOM adjustment score:

  • Started service: SERVICE_ADJ (500) — safer than cached apps (900+)
  • Foreground service: FOREGROUND_APP_ADJ (0) — almost never killed
  • Bound service: inherits the binding client's priority (with flags like BIND_IMPORTANT)

Without a Service, a backgrounded app's process sits at CACHED_APP_MIN_ADJ (900+) and is first in line for the OOM killer.

Staff-level addition: Mention service aging — after ~30 minutes, a started service is demoted from SERVICE_ADJ (500) to SERVICE_B_ADJ (800), making it nearly as killable as cached processes. This is why long-running started services are unreliable and why foreground services exist.
Q4 What was IntentService, why was it deprecated, and what replaced it? L5

IntentService was a convenience wrapper that created a HandlerThread, processed intents serially on it, and auto-called stopSelf(startId) when the queue drained.

Why deprecated (API 30): Android 8's background execution limits made startService() from background throw IllegalStateException. IntentService had no built-in way to call startForeground() before the system killed it. It was also single-threaded with no parallelism.

Replaced by: JobIntentService (also deprecated) → WorkManager.

What juniors miss: They know it's deprecated but can't explain the internal mechanism (HandlerThread + Handler + Message queue) or why the background execution limits specifically broke it.

Section B: Foreground Services

Q5 What happens if you call startForegroundService() but don't call startForeground() within 5 seconds? L5

The system throws ForegroundServiceDidNotStartInTimeException (on Android 12+, previously an ANR). The service is killed. This is the #1 crash source for foreground services in production.

Staff-level addition: The 5-second timer starts at startForegroundService() call, but onStartCommand() runs on the main thread. If the main thread is blocked (heavy DI initialization, disk I/O, slow ContentProvider queries in onCreate), the window expires. Best practice: call startForeground() as the absolute first line of onStartCommand(), before any other initialization.
Q6 Can an app start a foreground service from the background on Android 12+? What are the alternatives? L6

No — the system throws ForegroundServiceStartNotAllowedException. There are specific exemptions: high-priority FCM, user tapping a notification/PendingIntent, exact alarm firing, SYSTEM_ALERT_WINDOW permission, device/profile owner, companion device manager.

Alternatives:

  • WorkManager.setExpedited() — runs immediately, uses JobScheduler's expedited jobs on API 31+, falls back to FGS on older APIs
  • High-priority FCM → foreground service (still allowed)
  • User-initiated data transfer (WorkManager 2.9+, Android 14+)
Staff-level addition: Explain that this restriction was Google's response to apps abusing BroadcastReceiver → startForegroundService() to run background work with a barely-visible notification. The architectural shift is from "I'll start a service" to "I'll declare work and let the system schedule it."
Q7 Explain foreground service types in Android 14. Why were they introduced? L6

Android 14 requires every foreground service to declare a foregroundServiceType in both the manifest and at runtime. Types include: camera, connectedDevice, dataSync, health, location, mediaPlayback, microphone, phoneCall, shortService, specialUse, etc.

Why: Google wanted to audit what foreground services actually do. Before this, any app could show a notification and run indefinitely. Now the system can enforce type-specific rules: dataSync gets a 6-hour timeout, shortService gets 3 minutes, and each type requires a corresponding FOREGROUND_SERVICE_* permission.

Staff-level addition: The dataSync type is being deprecated in Android 15 — Google wants all data sync to go through WorkManager. The specialUse type is an escape hatch but requires Play Store justification. Mention that this is part of Google's multi-year strategy to make foreground services auditable and eventually limit them to truly user-facing ongoing work only.
Q8 Does stopForeground() stop the service? L4

No. stopForeground() only removes the foreground state (and optionally the notification). The service continues running as a regular background service — which means it's now subject to background execution limits and the OOM killer. You must call stopSelf() to actually stop the service.

What juniors miss: They assume stopForeground() is the counterpart to startForeground() and stops everything. It only demotes the service from foreground to background priority.

Section C: Scheduling & Doze

Q9 What is Doze mode and how does it affect background work? L5

When the device is stationary, screen off, and on battery, the system enters Doze. In Doze: network access is blocked, wakelocks are ignored, AlarmManager alarms and JobScheduler jobs are deferred, syncs are deferred. The system opens brief maintenance windows with increasing intervals.

Staff-level addition: Explain the two tiers: light Doze (API 24+, triggers faster, less restrictive) and deep Doze (original, full restrictions). Also mention that high-priority FCM messages can temporarily lift Doze restrictions — this is why push notification architecture matters for background work reliability. At Google scale, teams design their sync strategy around Doze maintenance windows.
Q10 Explain App Standby Buckets and their impact on job scheduling. L6

Apps are categorized into buckets based on recency of user interaction: Active (no restrictions), Working Set (jobs deferred up to 2h), Frequent (up to 8h), Rare (up to 24h), Restricted (1 job/day on API 31+).

The system uses UsageStatsManager to assign buckets. Push notifications that the user taps count as interaction and promote the app to a better bucket.

Staff-level addition: At Google scale, teams optimize notification engagement partly to maintain favorable bucket placement. Also, the Restricted bucket (API 31+) is devastating — essentially 1 job per day. Apps that the user installs but rarely opens quickly fall into Rare/Restricted. This is why WorkManager's constraint-based approach matters: the system batches your work into the allowed windows.
Q11 What happens if you forget to call jobFinished() in a JobService? L5

If onStartJob() returns true (async work) and you never call jobFinished():

  • The system thinks the job is still running
  • It holds a wakelock on your behalf → battery drain
  • The job slot is occupied → your next scheduled run may be skipped
  • Eventually the system may force-stop the job (implementation-dependent)
What juniors miss: This is a production bug that's invisible in testing. It manifests as battery drain and missed syncs days later. WorkManager handles this internally, which is one reason it's preferred over raw JobScheduler.

Section D: WorkManager

Q12 How does WorkManager guarantee work execution across process death and reboot? L6

WorkManager persists all work metadata (WorkSpecs) in a Room database (SQLite). When you enqueue work:

  1. WorkSpec is written to the DB
  2. The Scheduler delegates to a system-level scheduler (JobScheduler on API 23+, AlarmManager + BroadcastReceiver below)
  3. On process death: the system scheduler fires, WorkManager's process starts, it reads pending WorkSpecs from the DB, and executes them
  4. On reboot: SystemForegroundService + RECEIVE_BOOT_COMPLETED triggers re-scheduling from the DB
Staff-level addition: WorkManager runs TWO schedulers simultaneously — the GreedyScheduler (in-process, runs immediately if constraints met) and a system scheduler (for process-death survival). The Processor deduplicates. This dual-scheduler design is why WorkManager feels instant when the app is alive but still guarantees execution after death. Mention this architecture to show deep understanding.
Q13 What's the difference between setExpedited() and setForeground() in WorkManager? L6

setExpedited() is set on the WorkRequest at enqueue time. It tells the system to run this work ASAP. On API 31+, it uses JobScheduler's expedited job quota. On older APIs, it falls back to a foreground service (requires getForegroundInfo()).

setForeground() is called inside the Worker's doWork(). It promotes the WorkManager-managed service to a foreground service at runtime. This removes the 10-minute execution limit and shows a notification.

Key difference: setExpedited() = "schedule me immediately." setForeground() = "I'm already running and need to become a foreground service for long-running work."

What juniors miss: They conflate the two. setExpedited() is about scheduling priority. setForeground() is about runtime execution mode. You can use both together for work that needs to start immediately AND run for a long time.
Q14 How do you inject dependencies into a WorkManager Worker? L5

Workers are created via reflection by default (no-arg constructor). For DI with Hilt:

  1. Annotate Worker with @HiltWorker
  2. Use @AssistedInject constructor with @Assisted on Context and WorkerParameters
  3. Inject HiltWorkerFactory in Application
  4. Implement Configuration.Provider and set the custom worker factory
  5. Disable default WorkManager initialization in the manifest
Staff-level addition: Explain why @AssistedInject is needed — Context and WorkerParameters are provided by the framework at runtime, not by the DI graph. The HiltWorkerFactory intercepts Worker creation, injects dependencies, and passes the assisted parameters. Without custom initialization, WorkManager uses DefaultWorkerFactory which only supports no-arg constructors.
Q15 What happens when parallel workers in a chain produce output data with the same key? L6

By default, OverwritingInputMerger is used — if two parallel workers output the same key, one overwrites the other non-deterministically. The downstream worker gets only one value.

To collect all values, use ArrayCreatingInputMerger: it combines values for the same key into an array.

val downstream = OneTimeWorkRequestBuilder<CombineWorker>()
    .setInputMerger(ArrayCreatingInputMerger::class)
    .build()
What juniors miss: They don't know about input mergers at all. This is a subtle production bug — parallel workers silently lose data because the default merger overwrites. In an interview, mentioning this unprompted shows real WorkManager experience.
Q16 What's the difference between KEEP, REPLACE, and UPDATE policies for unique work? L5

KEEP: If work with this name is pending/running, ignore the new request. Use for idempotent periodic work.

REPLACE: Cancel existing work, enqueue new. Use when parameters changed and old work is stale.

UPDATE (WorkManager 2.8+): Modify constraints/data of existing work without cancelling. The running instance continues; changes apply on next execution.

Staff-level addition: UPDATE was a major addition. Before it, changing a periodic worker's interval required REPLACE, which cancelled the running instance and reset the period timer. UPDATE modifies in-place. At Google scale, this enables A/B testing different sync intervals without disrupting active work.

Section E: Architecture & Design (L6 Level)

Q17 Design a reliable background sync system for an app with millions of users. What would you use and why? L6+

Architecture:

  • WorkManager with PeriodicWorkRequest for regular sync (e.g., every 1 hour)
  • High-priority FCM for server-initiated urgent sync (new messages, critical updates)
  • Expedited work for user-triggered sync (pull-to-refresh)
  • Unique work with KEEP policy to prevent duplicate syncs

Constraints:

  • Network: CONNECTED (not UNMETERED — users expect sync on cellular too)
  • Battery: setRequiresBatteryNotLow(true) for periodic, no constraint for user-triggered

Resilience:

  • Exponential backoff with max retries
  • Delta sync (only changed data) to minimize payload
  • Conflict resolution strategy (server wins, client wins, or merge)
Staff-level addition: Discuss App Standby Buckets impact — users who don't open the app daily will have sync deferred up to 24 hours. Mitigation: engaging notifications that promote bucket placement. Also discuss Doze maintenance windows and how FCM high-priority messages temporarily lift restrictions. Mention monitoring: track sync success rate by bucket, device manufacturer (OEMs add their own restrictions), and Android version.
Q18 Why did Google create WorkManager when JobScheduler already existed? L6

Three fundamental problems with JobScheduler alone:

  1. API level fragmentation: JobScheduler is API 21+, but many features (expedited, network type) were added in later APIs. No backward compatibility.
  2. No persistence layer: JobScheduler doesn't survive app updates or data clears. WorkManager's Room DB does.
  3. No work chaining or observability: JobScheduler is fire-and-forget. WorkManager provides LiveData/Flow observation, chaining, unique work policies.

WorkManager is a Jetpack abstraction that uses JobScheduler (API 23+), GCMNetworkManager (Play Services), or AlarmManager + BroadcastReceiver as internal delegates. It adds the persistence, chaining, and observability layer on top.

Staff-level addition: The key architectural insight is that WorkManager is a library (ships with your APK, you control the version) while JobScheduler is a platform API (tied to the device's Android version). This means WorkManager can add features and fix bugs via Jetpack updates without waiting for OS updates — critical given Android's fragmentation.
Q19 How would you handle a task that needs to upload 500MB of data reliably? L6

Approach:

  1. User-initiated data transfer (WorkManager 2.9+ / Android 14+) if user explicitly triggered it
  2. Otherwise: CoroutineWorker with setForeground() to promote to foreground service (removes 10-min limit)
  3. Declare foregroundServiceType="dataSync" in manifest
  4. Implement chunked upload with resume support (track last uploaded byte in WorkManager's progress data)
  5. Use Result.retry() on network failures with exponential backoff
  6. Report progress via setProgress() for UI updates

Edge cases:

  • Handle onStopped() in the Worker — save progress so the next attempt resumes
  • On Android 14+, dataSync has a 6-hour timeout — ensure upload completes or saves progress before onTimeout()
  • Network constraint: CONNECTED (not UNMETERED, unless the user chose WiFi-only in settings)
Staff-level addition: Discuss the dataSync deprecation in Android 15 and how user-initiated data transfer is the replacement. Also mention that at Google scale, large uploads use server-side resumable upload protocols (like Google's resumable upload API) where the server tracks progress, making client-side resume trivial.
Q20 Walk me through the complete lifecycle of a WorkManager request from enqueue to completion, including what happens on process death. L6+

Happy path (app alive):

  1. WorkManager.enqueue(request) → WorkSpec written to Room DB
  2. GreedyScheduler checks constraints immediately — if met, sends work to Processor
  3. Simultaneously, SystemJobScheduler schedules a JobScheduler job as backup
  4. Processor creates Worker instance via WorkerFactory
  5. doWork() executes on background thread
  6. Worker returns Result.success() → Processor updates WorkSpec in DB to SUCCEEDED
  7. If chained, next WorkSpec's constraints are evaluated

Process death scenario:

  1. App process is killed. GreedyScheduler dies with it.
  2. WorkSpec is still in Room DB with state ENQUEUED or RUNNING.
  3. SystemJobScheduler's job fires (JobScheduler wakes the process).
  4. WorkManager initializes, reads pending WorkSpecs from DB.
  5. RUNNING work is reset to ENQUEUED (treated as a retry).
  6. Processor executes the Worker again.

Reboot scenario:

  1. Device reboots. RECEIVE_BOOT_COMPLETED broadcast fires.
  2. WorkManager's RescheduleReceiver triggers.
  3. Reads all pending WorkSpecs from DB, re-registers with system scheduler.
Staff-level addition: The subtle detail: when a RUNNING worker is interrupted by process death, it's treated as a retry — runAttemptCount increments. If you have max retry logic, process death counts against it. Design your retry budget accordingly. Also, the Room DB is the single source of truth — if the DB is corrupted or cleared (app data clear), all pending work is lost. WorkManager does NOT survive adb shell pm clear.
