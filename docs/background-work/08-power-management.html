3. Wakelocks — The Full Picture

Types of Wakelocks

TypeCPUScreenStatus
PARTIAL_WAKE_LOCK On Off Only type still recommended (when absolutely necessary)
SCREEN_DIM_WAKE_LOCK On Dim Deprecated — use FLAG_KEEP_SCREEN_ON
SCREEN_BRIGHT_WAKE_LOCK On Bright Deprecated

Safe Wakelock Pattern

val wakeLock: PowerManager.WakeLock =
    (getSystemService(POWER_SERVICE) as PowerManager).run {
        newWakeLock(PowerManager.PARTIAL_WAKE_LOCK, "myapp:mytag").apply {
            acquire(10 * 60 * 1000L) // ALWAYS set a timeout
        }
    }

// In finally block or onDestroy:
if (wakeLock.isHeld) {
    wakeLock.release()
}
The #1 Wakelock Bug

Acquiring a wakelock without a timeout. If your code crashes or takes an unexpected path before release(), the CPU stays awake indefinitely. Always pass a timeout to acquire(timeoutMs). The system auto-releases after the timeout.

Staff-Level Insight — Who Holds Wakelocks For You

In modern Android, you rarely need explicit wakelocks because the framework manages them:

  • JobScheduler: Holds a wakelock for the duration of onStartJob() → jobFinished()
  • WorkManager: Holds a wakelock for the duration of doWork()
  • Foreground Service: The foreground state itself prevents CPU sleep
  • BroadcastReceiver: System holds a wakelock during onReceive() (but NOT during goAsync() — you need your own)

If you're using any of these APIs correctly, you don't need a wakelock. The only remaining use case is custom alarm-based scheduling in legacy code.

Staff-Level Insight — The 9-Minute Throttle

setAndAllowWhileIdle() and setExactAndAllowWhileIdle() are throttled to at most once per 9 minutes per app in deep Doze. If you fire one, the next one is delayed by 9 minutes regardless of when you scheduled it. This prevents apps from using exact alarms to circumvent Doze. The only way to get truly unrestricted alarm delivery is to be on the battery optimization whitelist — which requires user action.

5. Battery Optimization Whitelist

Users can exempt apps from Doze and App Standby via Settings → Battery → Battery Optimization. Apps can request this programmatically:

// Check if whitelisted
val pm = getSystemService(POWER_SERVICE) as PowerManager
val isWhitelisted = pm.isIgnoringBatteryOptimizations(packageName)

// Request whitelist (shows system dialog)
if (!isWhitelisted) {
    val intent = Intent(Settings.ACTION_REQUEST_IGNORE_BATTERY_OPTIMIZATIONS).apply {
        data = Uri.parse("package:$packageName")
    }
    startActivity(intent)
}
Play Store Policy

Google Play restricts the use of REQUEST_IGNORE_BATTERY_OPTIMIZATIONS. Only apps whose core function is adversely affected by Doze may use it (e.g., messaging apps, VoIP, health monitoring). Using it for general sync or analytics will get your app rejected. The alternative is guiding users to the battery settings page manually.

Staff-Level Insight — Whitelisting Doesn't Fix Everything

Battery optimization whitelisting exempts your app from stock Android's Doze and App Standby. It does NOT exempt you from:

  • OEM battery managers (Xiaomi, Huawei, Samsung) — these have their own whitelists
  • App Standby Buckets (API 28+) — whitelisting doesn't change your bucket
  • Background execution limits (API 26+) — startService() from background still throws
  • Foreground service restrictions (API 31+) — still can't start FGS from background

It's a partial fix, not a silver bullet. The only truly reliable cross-device mechanism is high-priority FCM.

Staff-Level Insight — Battery Historian

Battery Historian is Google's tool for visualizing battery drain. It parses bugreport output and shows a timeline of wakelocks, jobs, alarms, network activity, and CPU usage per app. To use it:

# Capture bugreport
adb bugreport bugreport.zip

# Run Battery Historian (Docker)
docker run -p 9999:9999 gcr.io/android-battery-historian/stable:3.1 --port 9999

# Upload bugreport.zip to http://localhost:9999

In an interview, mentioning Battery Historian shows you've actually profiled battery usage in production, not just read about it.

7. The Complete Decision Framework

Combining everything from this series — here's how to think about background work at L6 level:

Staff-Level Insight — The Interview Answer

When asked "how do you handle background work in Android?", the L6 answer isn't just "use WorkManager." It's:

  1. Classify the work: Is it user-facing? Deferrable? Time-sensitive? Must it survive process death?
  2. Pick the right API based on the classification (the decision tree above)
  3. Account for power management: Doze, App Standby Buckets, OEM restrictions
  4. Design for failure: Process death, force-stop, constraint changes, OEM kills
  5. Monitor in production: Success rates by OEM/API/bucket, Battery Historian, Android Vitals

This five-step framework demonstrates the systems thinking that Google expects at L6.
