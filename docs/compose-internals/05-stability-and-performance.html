Stability & Performance

The stability system, @Stable and @Immutable contracts, skippability rules, strong skipping mode, lambda memoization, derivedStateOf, and performance profiling

1. What Is Stability?

Stability is the Compose compiler's way of determining whether a type's equals() is reliable for skipping recomposition. If the compiler can prove that a.equals(b) == true means "a and b will produce identical UI," it marks the type as stable and generates skip logic.

The Stability Contract

A type is stable if:

  1. equals() will always return the same result for the same two instances (consistent)
  2. If a public property changes, the Compose runtime is notified (via snapshot state)
  3. All public properties are themselves stable

What the Compiler Considers Stable

TypeStable?Why
Primitives (Int, Boolean, Float, etc.)YesImmutable, reliable equals
StringYesImmutable
data class with all stable propertiesYesCompiler can verify all fields
MutableState<T>YesChanges notify the snapshot system
List<T>, Map<K,V>, Set<T>NoCould be MutableList at runtime — compiler can't verify immutability
Classes from external modulesNoCompiler can't inspect source — assumes unstable
var propertiesNoCan change without notifying Compose
LambdasDependsNon-capturing or capturing only stable values → stable. Otherwise unstable.

2. @Stable and @Immutable

// @Immutable: promise that ALL properties will NEVER change after construction
@Immutable
data class User(
    val id: String,
    val name: String,
    val avatarUrl: String
)

// @Stable: promise that if properties change, Compose will be notified
// (via snapshot state). equals() is consistent.
@Stable
class UserState {
    var name by mutableStateOf("")        // Changes notify Compose ✓
    var isOnline by mutableStateOf(false) // Changes notify Compose ✓
}

// @Stable on an interface: promise that all implementations will be stable
@Stable
interface Theme {
    val primaryColor: Color
    val backgroundColor: Color
}
Critical — These Are Contracts, Not Enforcement

@Stable and @Immutable are promises you make to the compiler. The compiler trusts you and generates skip logic based on these promises. If you lie (e.g., mark a class @Immutable but mutate its properties), the UI will show stale data because Compose will skip recomposition when it shouldn't. There is no runtime verification.

4. Fixing Stability Issues

// Problem: List is unstable
data class ScreenState(
    val items: List<Item>  // List is unstable → ScreenState is unstable
)

// Fix 1: Use kotlinx.collections.immutable
data class ScreenState(
    val items: ImmutableList<Item>  // ImmutableList is @Stable → ScreenState is stable
)

// Fix 2: Annotate with @Immutable (if you guarantee immutability)
@Immutable
data class ScreenState(
    val items: List<Item>  // You promise this List is never mutated
)

// Fix 3: Stability configuration file (compose compiler 1.5.5+)
// Create: stability-config.txt
// Content:
// java.util.List
// java.util.Map
// com.example.MyExternalClass
// This tells the compiler to treat these types as stable

// In build.gradle:
// composeCompiler {
//     stabilityConfigurationFile = rootProject.layout.projectDirectory.file("stability-config.txt")
// }

// Fix 4: Wrap in a stable wrapper
@Stable
class StableList<T>(val list: List<T>)

// Fix 5: For lambdas — use remember
@Composable
fun Parent() {
    val viewModel: MyViewModel = viewModel()

    // BAD: new lambda instance every recomposition → child can't skip
    ChildComposable(onClick = { viewModel.doSomething() })

    // GOOD: remembered lambda → same instance → child can skip
    val onClick = remember { { viewModel.doSomething() } }
    ChildComposable(onClick = onClick)

    // ALSO GOOD (with strong skipping): lambda is compared by ===
    // If captures don't change, same instance is reused automatically
}
Staff-Level Insight — Lambda Memoization

The Compose compiler automatically memoizes lambdas that capture only stable values. If you write Button(onClick = { count++ }) where count is a MutableState, the compiler wraps the lambda in remember(count) { { count++ } } internally. But if the lambda captures an unstable value, the compiler can't memoize it — a new lambda instance is created every recomposition. With strong skipping mode, this is less of a problem because lambdas are compared by ===, and the compiler's memoization ensures the same instance is reused when captures haven't changed.

6. Performance Anti-Patterns and Fixes

Anti-PatternWhy It's BadFix
Reading state in Composition when only Draw needs it Triggers full recomposition + layout + draw instead of just draw Use Modifier.drawBehind { } or graphicsLayer { }
Passing unstable types as parameters Composable can never be skipped Use @Immutable, ImmutableList, or stability config
Creating objects in Composition New object every recomposition → child sees "changed" param Use remember { } to cache objects
Reading LazyListState.firstVisibleItemScrollOffset in Composition Changes every frame during scroll → recompose every frame Use snapshotFlow or read in Layout/Draw phase
Large composable functions that read state Entire function re-executes even if only one part needs the state Extract state-reading parts into smaller composables
derivedStateOf with parameters (not state) Doesn't track parameter changes — stale values Use remember(key) for parameters, derivedStateOf for state
Staff-Level Insight — The Phase Optimization Strategy

The most impactful L6 optimization: defer state reads to the latest possible phase.

  • Composition phase: Box(Modifier.offset(x = animatedX.value.dp)) — reads state in Composition → triggers recomposition + layout + draw every frame.
  • Layout phase: Box(Modifier.offset { IntOffset(animatedX.value, 0) }) — reads state in Layout → skips recomposition, only layout + draw.
  • Draw phase: Box(Modifier.graphicsLayer { translationX = animatedX.value }) — reads state in Draw → skips recomposition AND layout, only draw. This is the fastest.

For animations, always use the lambda versions of modifiers (offset { }, graphicsLayer { }) — they defer reads to Layout/Draw phase.

7. Benchmarking with Macrobenchmark

@LargeTest
@RunWith(AndroidJUnit4::class)
class ScrollBenchmark {
    @get:Rule
    val benchmarkRule = MacrobenchmarkRule()

    @Test
    fun scrollList() = benchmarkRule.measureRepeated(
        packageName = "com.example.app",
        metrics = listOf(
            FrameTimingMetric(),  // Measures jank (missed frames)
            TraceSectionMetric("Compose:recompose"), // Recomposition time
        ),
        iterations = 5,
        startupMode = StartupMode.WARM,
        setupBlock = {
            pressHome()
            startActivityAndWait()
        }
    ) {
        val list = device.findObject(By.res("item_list"))
        list.setGestureMargin(device.displayWidth / 5)
        list.fling(Direction.DOWN)
        device.waitForIdle()
    }
}
