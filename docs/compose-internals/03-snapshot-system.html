The Snapshot System

How Compose tracks state reads and writes, MutableState internals, snapshot isolation, MVCC, read/write observers, and how state invalidation triggers recomposition

1. The Big Picture — Why Snapshots Exist

The snapshot system is Compose's reactive engine. It solves two problems:

  1. Automatic tracking: When a composable reads state.value, the runtime automatically knows which composable depends on which state — no manual subscription needed.
  2. Isolation: During recomposition, the runtime needs a consistent view of state. If state changes mid-recomposition, you'd get torn reads (half old, half new). Snapshots provide MVCC (Multi-Version Concurrency Control) — like database transactions for state.

2. MutableState Internals

// What mutableStateOf() actually creates (simplified from source):
fun <T> mutableStateOf(
    value: T,
    policy: SnapshotMutationPolicy<T> = structuralEqualityPolicy()
): MutableState<T> = ParcelableSnapshotMutableState(value, policy)

// The actual state holder:
internal class SnapshotMutableStateImpl<T>(
    value: T,
    override val policy: SnapshotMutationPolicy<T>
) : StateObject, MutableState<T> {

    // Each snapshot sees its own version of the value
    override var firstStateRecord: StateRecord = StateStateRecord(value)

    override var value: T
        get() {
            // 1. Find the correct record for the current snapshot
            val record = readable(firstStateRecord, this)
            // 2. Notify the snapshot that this state was READ
            //    → This is how the Composer tracks dependencies
            Snapshot.current.readObserver?.invoke(this)
            return record.value
        }
        set(value) {
            // 1. Get a writable record for the current snapshot
            val record = writable(firstStateRecord, this)
            // 2. Check if value actually changed (using the policy)
            if (!policy.equivalent(record.value, value)) {
                record.value = value
                // 3. Notify the snapshot that this state was WRITTEN
                //    → This triggers invalidation
            }
        }
}

SnapshotMutationPolicy — Controlling When State "Changes"

PolicyComparisonWhen to Use
structuralEqualityPolicy() equals() Default. Recompose only if old.equals(new) is false.
referentialEqualityPolicy() === Recompose only if the object reference changes. Useful for large objects where equals() is expensive.
neverEqualPolicy() Always different Always triggers recomposition on every write. Useful for mutable objects where you can't detect changes.
Staff-Level Insight — The StateRecord Chain

Each StateObject maintains a linked list of StateRecords — one per active snapshot. When you read state, the runtime walks this chain to find the record valid for the current snapshot ID. When you write, it creates a new record (or reuses an old one). This is how MVCC works: each snapshot sees its own version of the data without locks. The chain is typically short (1-3 records) because old records are garbage collected when their snapshot is no longer active. This is the same pattern databases use for snapshot isolation.

3. Snapshot Isolation — MVCC for UI

// Snapshots provide isolation — like database transactions:

// Global snapshot (the "current" state of the world)
val name = mutableStateOf("Alice")

// During recomposition, the Composer takes a READ snapshot:
// It sees a frozen view of all state — even if another thread writes
// to 'name' during recomposition, the Composer still sees "Alice"

// After recomposition, changes are applied atomically:
Snapshot.withMutableSnapshot {
    // Inside this block, writes are isolated
    name.value = "Bob"
    // Other threads still see "Alice" until this snapshot is applied
}
// Now everyone sees "Bob"

Snapshot Types

TypeCreated ByPurpose
GlobalSnapshot System (always active) The "current truth." All non-snapshot reads/writes go here.
MutableSnapshot Snapshot.takeMutableSnapshot() Isolated read-write snapshot. Changes are invisible until apply().
ReadonlySnapshot Snapshot.takeSnapshot() Frozen view of state at a point in time. Used during recomposition.
NestedMutableSnapshot Inside another mutable snapshot Nested transaction. Can be applied or disposed independently.

4. Read and Write Observers

The snapshot system uses observers to track reads and writes. During composition, the Composer installs a read observer:

// How the Composer tracks state reads (simplified):
class ComposerImpl : Composer {
    private val currentRecompositionScope: RecomposeScopeImpl

    fun startComposition() {
        // Take a snapshot with a read observer
        snapshot = Snapshot.takeMutableSnapshot(
            readObserver = { stateObject ->
                // Every time ANY state is read during composition,
                // this callback fires.
                // Record: "currentRecompositionScope reads stateObject"
                currentRecompositionScope.recordRead(stateObject)
            },
            writeObserver = { stateObject ->
                // If state is written during composition (rare but possible),
                // schedule re-invalidation
            }
        )
    }
}

// The mapping is bidirectional:
// StateObject → Set<RecompositionScope>  (which scopes read this state?)
// RecompositionScope → Set<StateObject>  (which states does this scope read?)
Staff-Level Insight — The Invalidation Table

The runtime maintains a bidirectional mapping between state objects and recomposition scopes. When state is written, it looks up all scopes that read that state and marks them invalid. This is O(1) per scope — no tree diffing, no dirty checking. This is fundamentally different from React (which diffs virtual DOM subtrees) or Angular (which runs change detection on the component tree). Compose's approach is more like a reactive database: subscriptions are automatic and invalidation is precise.

5. GlobalSnapshotManager — The Android Bridge

On Android, the GlobalSnapshotManager connects the snapshot system to the Choreographer:

// GlobalSnapshotManager (simplified from AndroidX source):
internal object GlobalSnapshotManager {
    fun ensureStarted() {
        Snapshot.registerGlobalWriteObserver { changedObjects ->
            // A state object was written in the global snapshot.
            // Post to the main thread to apply changes:
            channel.trySend(Unit)
        }

        // On the main thread, process pending changes:
        scope.launch {
            channel.consumeEach {
                Snapshot.sendApplyNotifications()
                // This notifies all Recomposers that state changed
                // → Recomposers schedule recomposition via Choreographer
            }
        }
    }
}
Staff-Level Insight — Why sendApplyNotifications Matters

Snapshot.sendApplyNotifications() is the bridge between "state changed" and "recomposition happens." It processes the global snapshot's pending changes, notifies all registered apply observers (which include the Recomposer), and those observers schedule recomposition on the next frame. If you call mutableStateOf outside of composition (e.g., in a ViewModel), the write goes to the global snapshot. The GlobalSnapshotManager picks it up and routes it to the Recomposer. This is why state updates from ViewModels automatically trigger recomposition — no LiveData.observe() or Flow.collect() ceremony needed.

6. derivedStateOf — Computed State

@Composable
fun FilteredList(items: List<Item>, query: String) {
    // BAD: filteredItems recalculated on EVERY recomposition
    val filteredItems = items.filter { it.name.contains(query) }

    // GOOD: only recalculated when items or query actually change
    val filteredItems by remember(items, query) {
        derivedStateOf { items.filter { it.name.contains(query) } }
    }
    // Wait — this is also wrong! derivedStateOf already tracks its inputs.

    // CORRECT: derivedStateOf reads 'items' and 'query' inside the lambda,
    // so it automatically recalculates when they change.
    // But items and query are parameters, not state objects — they can't be tracked.
    // So we need remember with keys:
    val filteredItems = remember(items, query) {
        items.filter { it.name.contains(query) }
    }

    // derivedStateOf is for when inputs ARE state objects:
    val searchQuery = remember { mutableStateOf("") }
    val itemsState = remember { mutableStateOf(listOf<Item>()) }

    val filteredItems by remember {
        derivedStateOf {
            // Reads searchQuery.value and itemsState.value
            // Automatically recalculates when either changes
            // AND: only triggers recomposition if the RESULT changes
            itemsState.value.filter { it.name.contains(searchQuery.value) }
        }
    }
}
Staff-Level Insight — derivedStateOf vs remember(key)

derivedStateOf is NOT a replacement for remember(key). It's for a specific case: when you want to reduce recomposition frequency. derivedStateOf creates a state object that only triggers recomposition when its output changes, not when its inputs change. Example: if scrollState.value changes from 100 to 101, but derivedStateOf { scrollState.value > 0 } still returns true, no recomposition happens. Use derivedStateOf when: (1) inputs are state objects (not parameters), and (2) the derived value changes less frequently than the inputs. For parameters, use remember(key).

7. snapshotFlow — Bridging Snapshots to Coroutines

// snapshotFlow converts snapshot state reads into a Flow:
@Composable
fun ScrollTracker(listState: LazyListState) {
    LaunchedEffect(listState) {
        snapshotFlow { listState.firstVisibleItemIndex }
            .distinctUntilChanged()
            .collect { index ->
                analytics.trackScroll(index)
            }
    }
}

// How snapshotFlow works internally:
// 1. Takes a read snapshot
// 2. Executes the lambda, tracking all state reads
// 3. Emits the result
// 4. Suspends until any tracked state changes
// 5. Re-executes the lambda in a new snapshot
// 6. If result changed → emit. If same → skip (built-in distinctUntilChanged)
Staff-Level Insight — snapshotFlow Is Not collectAsState

snapshotFlow goes from Compose state → Flow (for side effects). collectAsState goes from Flow → Compose state (for UI). They're inverses. Use snapshotFlow when you need to react to state changes in a coroutine (analytics, logging, network calls). Use collectAsState when you need to display Flow data in UI. Never use both together — that creates a circular dependency.

8. Thread Safety of Snapshots

The snapshot system is designed for concurrent access:

  • Reads are lock-free: Reading state walks the StateRecord chain, which uses atomic operations.
  • Writes to the global snapshot are synchronized: The global snapshot uses a lock for writes, but it's very brief (just updating a record pointer).
  • Mutable snapshots are thread-local: Each mutable snapshot is used by one thread, so no synchronization needed.
  • Apply is atomic: When a mutable snapshot is applied, all its changes become visible at once.
Common Snapshot Pitfalls
  • Reading state outside composition: If you read state.value in a regular function (not @Composable), the read is NOT tracked. Changes won't trigger recomposition.
  • Mutating state during composition: Writing to state during composition causes a recomposition loop. The snapshot system detects this and throws.
  • Forgetting that by delegates track reads: var x by mutableStateOf(0) — reading x triggers tracking because the delegate calls .value. But val x = mutableStateOf(0) — reading x gives you the State object, not the value. You must read x.value for tracking.
  • Stale captures in lambdas: A lambda that captures state.value (not state) captures the value at creation time, not a live reference. The lambda won't see future changes.

9. rememberSaveable — Surviving Config Changes & Process Death

remember stores values in the slot table, which survives recomposition but is destroyed on configuration changes (Activity recreation) and process death. rememberSaveable persists values to the Bundle:

// remember vs rememberSaveable:
var count by remember { mutableStateOf(0) }           // Lost on config change
var count by rememberSaveable { mutableStateOf(0) }   // Survives config change + process death

// How rememberSaveable works internally:
@Composable
fun <T : Any> rememberSaveable(
    vararg inputs: Any?,
    saver: Saver<T, out Any> = autoSaver(),
    key: String? = null,
    init: () -> T
): T {
    val registry = LocalSaveableStateRegistry.current
    // 1. On first composition: call init(), register with SaveableStateRegistry
    // 2. On restoration: read saved value from Bundle, pass through Saver.restore()
    // 3. On save (config change / process death): call Saver.save(), write to Bundle
}

SaveableStateRegistry — The Persistence Engine

// SaveableStateRegistry is provided by the Activity/Fragment:
// ComponentActivity.setContent() installs it via CompositionLocalProvider

// The registry maintains a map: String (key) → List<Any?> (saved values)
// Keys are auto-generated from the composable's position in the slot table

// When the system calls Activity.onSaveInstanceState(outState):
// 1. Registry walks all registered saveable values
// 2. Each value is passed through its Saver.save() → produces a Bundle-compatible type
// 3. Values are written to the Activity's Bundle
// 4. On restoration, the Bundle is read and values are restored via Saver.restore()

// Custom Saver example:
data class City(val name: String, val country: String)

val CitySaver = Saver<City, List<String>>(
    save = { listOf(it.name, it.country) },
    restore = { City(it[0], it[1]) }
)

@Composable
fun CityPicker() {
    var city by rememberSaveable(stateSaver = CitySaver) {
        mutableStateOf(City("London", "UK"))
    }
}

Staff-Level Insight — Config Changes in Compose

When a configuration change occurs (rotation, locale, etc.), the Activity is recreated. The Composition is disposed and recreated from scratch — the slot table is destroyed. All remember values are lost. rememberSaveable values survive because they're serialized to the Activity's Bundle before destruction and restored after. ViewModel survives config changes (held by ViewModelStore) but NOT process death unless it uses SavedStateHandle. The L6 strategy: use ViewModel for screen state (survives config changes cheaply), SavedStateHandle/rememberSaveable for user input that must survive process death, and remember for transient UI state (animations, scroll position via rememberLazyListState which internally uses rememberSaveable).

10. Snapshot Apply & Conflict Resolution

// When a MutableSnapshot is applied, conflicts can occur:
val snapshot = Snapshot.takeMutableSnapshot()

// Thread A (inside snapshot):
snapshot.enter {
    state.value = "A"
}

// Thread B (global snapshot, concurrent):
state.value = "B"

// When snapshot.apply() is called:
// 1. Runtime checks if any StateObject written in the snapshot
//    was ALSO written in the global snapshot since the snapshot was taken
// 2. If conflict detected → apply() returns ApplyResult.Failure
// 3. The caller must handle the failure (retry, merge, or abort)

// In practice, Compose's Recomposer handles this automatically:
// - Recomposition runs in a mutable snapshot
// - If a conflict occurs during apply, the Recomposer retries recomposition
// - This is rare because recomposition is fast and state writes are batched

// The conflict check uses snapshot IDs:
// Each snapshot has an ID. Each StateRecord has the ID of the snapshot that wrote it.
// On apply: for each written state, check if a newer record exists in the global snapshot
// that wasn't visible when this snapshot was taken. If yes → conflict.
Staff-Level Insight — Why Conflicts Are Rare

In practice, snapshot conflicts almost never happen because: (1) recomposition runs on the main thread and completes within one frame, (2) state writes from background threads go to the global snapshot and are batched via sendApplyNotifications(), and (3) the Recomposer takes a fresh snapshot for each recomposition, so it always sees the latest state. The conflict resolution mechanism exists as a safety net for edge cases where concurrent mutable snapshots write the same state object. The retry cost is minimal — just re-execute the affected recomposition scopes.
