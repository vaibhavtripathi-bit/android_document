Recomposition Deep Dive

Recomposition scopes, smart recomposition, the $changed bitmask, donut-hole skipping, positional memoization, and why your composable recomposes (or doesn't)

1. What Is a Recomposition Scope?

A recomposition scope is the smallest unit of code that can be independently re-executed. The compiler creates a scope for every @Composable function that:

  1. Has a non-Unit return type, OR
  2. Reads state that might change

When state changes, only the scopes that read that state are invalidated and re-executed. Everything else is skipped.

@Composable
fun ParentScreen() {                    // Scope A
    var name by remember { mutableStateOf("World") }

    Header()                             // Scope B — does NOT read 'name'
    Greeting(name)                       // Scope C — reads 'name' via parameter
    Footer()                             // Scope D — does NOT read 'name'
}

// When 'name' changes:
// Scope A is invalidated (it reads 'name' to pass to Greeting)
// But Header() and Footer() are SKIPPED if their params haven't changed
// Greeting() re-executes with the new 'name'
Staff-Level Insight — Scope Granularity

The recomposition scope is the @Composable function, not individual lines. If a function reads state anywhere in its body, the entire function re-executes. This is why extracting state-reading code into smaller composables improves performance — it creates more granular scopes. The compiler wraps each @Composable call in startRestartGroup/endRestartGroup, which defines the scope boundary.

2. Smart Recomposition — The Skipping Mechanism

When a parent recomposes, it re-invokes all child composables. But the compiler-generated code checks whether parameters have changed before executing the body:

// What the compiler generates for a child call:
fun Greeting(name: String, $composer: Composer, $changed: Int) {
    $composer.startRestartGroup(0xABCD)

    // The skip check:
    val nameChanged = $changed and 0b0001 != 0
    if (nameChanged || !$composer.skipping) {
        // name changed → execute body
        Text("Hello, $name", $composer, 0)
    } else {
        // name unchanged → skip ENTIRE body
        $composer.skipToGroupEnd()
    }

    $composer.endRestartGroup()?.updateScope { c, _ ->
        Greeting(name, c, $changed or 0b0001)
    }
}

When Can a Composable Be Skipped?

ConditionSkippable?Why
All parameters are stable and equals() returns true Yes Compiler generates skip check using $changed bitmask + equals()
Any parameter is unstable No (before strong skipping) Compiler can't guarantee equals() is meaningful, so it always re-executes
Function has no parameters Yes (always skipped on recomposition) Nothing to compare — body only runs on initial composition
Function returns non-Unit No Return value might be needed by parent — must always execute
Strong skipping mode (Compose 1.5.4+) Yes, even with unstable params Uses referential equality (===) for unstable params instead of skipping skip-check entirely
Staff-Level Insight — Strong Skipping Mode

Before strong skipping (default since Compose compiler 1.5.4), any unstable parameter made the entire composable non-skippable. This meant passing a List<T> (unstable because mutable) would force recomposition every time the parent recomposed, even if the list was the same object. Strong skipping mode changes this: unstable parameters are compared with referential equality (===) instead of structural equality (equals()). Same object reference → skip. This dramatically reduces unnecessary recompositions in real apps.

3. Donut-Hole Skipping

This is the most elegant optimization in Compose and the most frequently asked about in L6 interviews:

@Composable
fun UserCard(user: User) {                    // Scope A
    Card {                                     // Scope B (Card's content lambda)
        Text(user.name)                        // Reads user.name
        ProfileImage(user.avatarUrl)           // Reads user.avatarUrl
        StatusBadge(isOnline = user.isOnline)  // Reads user.isOnline
    }
}

// If ONLY user.isOnline changes:
// 1. Scope A (UserCard) is invalidated because 'user' changed
// 2. UserCard re-executes, calling Card { ... }
// 3. Card itself is SKIPPED (its params didn't change — it's the same lambda)
// 4. But INSIDE Card's lambda, StatusBadge is re-invoked with new isOnline
// 5. Text and ProfileImage are SKIPPED (their params didn't change)

// This is "donut-hole skipping" — the outer composable (Card) is skipped,
// but the INNER content (the hole) is re-executed selectively
Staff-Level Insight — How Donut-Hole Works Internally

The content lambda ({ Text(); ProfileImage(); StatusBadge() }) is a separate recomposition scope from Card itself. When Card is skipped, the runtime still walks into the lambda's scope to check if any of its children need recomposition. This works because the slot table stores the lambda's content as a nested group — the Composer can skip Card's own logic but still enter the content group. This is why @Composable content lambdas are special — they create their own scopes, enabling this "skip the wrapper, update the content" pattern.

4. Positional Memoization — remember

remember stores a value in the slot table at the current position. The position is determined by the call site (source location), not the value:

@Composable
fun MyList(items: List<String>) {
    Column {
        items.forEach { item ->
            // DANGER: remember's position is based on the forEach index
            // If items are reordered, remembered values don't follow the items
            val color = remember { randomColor() }
            Text(item, color = color)
        }
    }
}

// Fix: use key() to give each item a stable identity
@Composable
fun MyList(items: List<String>) {
    Column {
        items.forEach { item ->
            key(item) { // Stable identity — remembered values follow the item
                val color = remember { randomColor() }
                Text(item, color = color)
            }
        }
    }
}

How remember Works Internally

// Simplified from compose-runtime source:
@Composable
inline fun <T> remember(calculation: () -> T): T {
    val composer = currentComposer
    // On first composition: calculate and store in slot table
    // On recomposition: read from slot table (skip calculation)
    return if (composer.inserting) {
        val value = calculation()
        composer.updateRememberedValue(value)
        value
    } else {
        composer.rememberedValue() as T
    }
}

// remember with keys:
@Composable
inline fun <T> remember(key1: Any?, calculation: () -> T): T {
    val composer = currentComposer
    return if (composer.inserting || composer.changed(key1)) {
        // Key changed → recalculate
        val value = calculation()
        composer.updateRememberedValue(value)
        value
    } else {
        composer.rememberedValue() as T
    }
}
Common remember Pitfalls
  • remember without key: remember { expensiveCalc(param) } — the calculation runs only once, even if param changes. Fix: remember(param) { expensiveCalc(param) }
  • remember in loops without key(): Values are tied to position, not identity. Reordering items shuffles remembered values.
  • remember with unstable keys: If the key object's equals() is broken, remember recalculates every recomposition.
  • remember { mutableStateOf() } vs rememberSaveable: remember survives recomposition but NOT process death or config change. rememberSaveable survives both.

5. Keys — Controlling Identity

// Without key: identity is based on position in the slot table
LazyColumn {
    items(users) { user ->
        UserRow(user) // Position 0, 1, 2, ... — reordering breaks state
    }
}

// With key: identity is based on the key value
LazyColumn {
    items(users, key = { it.id }) { user ->
        UserRow(user) // Identity tied to user.id — reordering preserves state
    }
}

// key() composable for non-lazy contexts
@Composable
fun ChatMessages(messages: List<Message>) {
    Column {
        messages.forEach { msg ->
            key(msg.id) {
                MessageBubble(msg) // State follows the message, not the position
            }
        }
    }
}
Staff-Level Insight — How Keys Work in the Slot Table

Without keys, the Composer matches composables by position in the parent's child list. Position 0 maps to slot 0, position 1 to slot 1, etc. If you insert an item at position 0, every subsequent item shifts — all their remembered state is now associated with the wrong item.

With keys, the Composer uses the key value to find the correct slot, regardless of position. Internally, it uses a hash map to look up slots by key. This is more expensive (hash lookup vs array index), but correct. The rule: always use keys when items have stable identity and can be reordered, inserted, or removed.

6. The Recomposition Loop

Recomposition doesn't happen immediately when state changes. It's batched and scheduled:

  1. State write: count++ triggers the snapshot system
  2. Invalidation: All recomposition scopes that read count are marked invalid
  3. Scheduling: A recomposition is scheduled for the next frame (via Choreographer on Android)
  4. Recomposition: On the next frame, the Composer re-executes all invalidated scopes
  5. Apply changes: The Applier applies node tree changes (inserts, removes, moves)
  6. Layout + Draw: Changed LayoutNodes are re-measured and re-drawn
// Multiple state changes in the same frame are batched:
var a by mutableStateOf(0)
var b by mutableStateOf(0)

fun onClick() {
    a = 1  // Invalidates scopes reading 'a'
    b = 2  // Invalidates scopes reading 'b'
    // Only ONE recomposition happens (on next frame), not two
}
Staff-Level Insight — Recomposition Is Always on the Main Thread

Recomposition (the Composition phase) always runs on the main thread because it accesses the slot table, which is not thread-safe. However, state writes can happen from any thread — the snapshot system is thread-safe. The invalidation is posted to the main thread's Choreographer, which triggers recomposition on the next VSYNC. This is why mutableStateOf is safe to update from a background coroutine — the write is thread-safe, and the resulting recomposition is automatically scheduled on the main thread.

7. Debugging Recomposition

// Method 1: Recomposition counter (debug only)
@Composable
fun RecompositionCounter(label: String) {
    val count = remember { mutableIntStateOf(0) }
    SideEffect { count.intValue++ }
    Log.d("Recomposition", "$label: ${count.intValue}")
}

// Method 2: Layout Inspector
// Android Studio → Layout Inspector → Show Recomposition Counts
// Shows how many times each composable recomposed and how many were skipped

//             project.buildDir.absolutePath + "/compose_reports"
//     )
// }
// Generates: *-composables.txt (skippability), *-classes.txt (stability)
Staff-Level Insight — Reading Compiler Reports

The compiler reports show two critical things: (1) which composables are skippable (can be skipped on recomposition) and (2) which classes are stable (safe for equality comparison). If a composable shows as "not skippable," check its parameters — one of them is likely unstable. If a class shows as "unstable," it either has mutable properties, uses a type the compiler can't verify (like List<T>), or comes from a non-Compose module. These reports are the primary tool for diagnosing unnecessary recompositions.
