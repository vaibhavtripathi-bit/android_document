Custom Layouts & Drawing

The Layout phase internals, building custom Layout composables, Modifier chain mechanics, Canvas drawing, SubcomposeLayout, intrinsic measurements, and how to build anything from scratch

1. The Layout Phase — How Compose Measures and Places

Constraints

// Constraints define the min/max width and height a child can be:
data class Constraints(
    val minWidth: Int = 0,
    val maxWidth: Int = Constraints.Infinity,
    val minHeight: Int = 0,
    val maxHeight: Int = Constraints.Infinity
)

// Examples:
// Constraints(0, 1080, 0, 1920) → "be whatever size you want up to screen size"
// Constraints(500, 500, 300, 300) → "be exactly 500x300" (tight constraints)
// Constraints(0, Infinity, 0, 50) → "any width, but max 50px tall"

2. Building a Custom Layout

The Layout composable is the foundation for all custom layouts:

@Composable
fun VerticalStack(
    modifier: Modifier = Modifier,
    spacing: Dp = 0.dp,
    content: @Composable () -> Unit
) {
    Layout(
        content = content,
        modifier = modifier,
        measurePolicy = { measurables, constraints ->
            // measurables = list of children to measure
            // constraints = size constraints from parent

            val spacingPx = spacing.roundToPx()

            // Step 1: Measure each child
            val placeables = measurables.map { measurable ->
                measurable.measure(constraints.copy(minWidth = 0, minHeight = 0))
            }

            // Step 2: Calculate total size
            val width = placeables.maxOf { it.width }
            val height = placeables.sumOf { it.height } +
                         spacingPx * (placeables.size - 1).coerceAtLeast(0)

            // Step 3: Return layout result with placement logic
            layout(width, height) {
                var yOffset = 0
                placeables.forEach { placeable ->
                    placeable.placeRelative(x = 0, y = yOffset)
                    yOffset += placeable.height + spacingPx
                }
            }
        }
    )
}
Staff-Level Insight — Why measure() Can Only Be Called Once

Compose enforces single-pass measurement to guarantee O(n) layout complexity. In the View system, a parent can call child.measure() multiple times (e.g., LinearLayout with weight does two passes), leading to O(2^depth) worst case in deeply nested layouts. Compose prevents this at the API level — calling measurable.measure() twice throws an IllegalStateException. If you need to "pre-measure" (e.g., to know a child's size before deciding constraints for another child), use intrinsics or SubcomposeLayout.

3. Real-World Custom Layout — Flow Layout

@Composable
fun FlowRow(
    modifier: Modifier = Modifier,
    horizontalSpacing: Dp = 8.dp,
    verticalSpacing: Dp = 8.dp,
    content: @Composable () -> Unit
) {
    Layout(content = content, modifier = modifier) { measurables, constraints ->
        val hSpacing = horizontalSpacing.roundToPx()
        val vSpacing = verticalSpacing.roundToPx()

        val placeables = measurables.map { it.measure(constraints.copy(minWidth = 0, minHeight = 0)) }

        data class RowInfo(val items: MutableList<Placeable> = mutableListOf(), var width: Int = 0, var height: Int = 0)

        val rows = mutableListOf(RowInfo())
        var currentRow = rows.first()

        placeables.forEach { placeable ->
            val newWidth = currentRow.width + placeable.width + if (currentRow.items.isNotEmpty()) hSpacing else 0
            if (newWidth > constraints.maxWidth && currentRow.items.isNotEmpty()) {
                currentRow = RowInfo()
                rows.add(currentRow)
            }
            currentRow.items.add(placeable)
            currentRow.width += placeable.width + if (currentRow.items.size > 1) hSpacing else 0
            currentRow.height = maxOf(currentRow.height, placeable.height)
        }

        val totalHeight = rows.sumOf { it.height } + vSpacing * (rows.size - 1).coerceAtLeast(0)
        val totalWidth = rows.maxOfOrNull { it.width } ?: 0

        layout(totalWidth.coerceIn(constraints.minWidth, constraints.maxWidth),
               totalHeight.coerceIn(constraints.minHeight, constraints.maxHeight)) {
            var yOffset = 0
            rows.forEach { row ->
                var xOffset = 0
                row.items.forEach { placeable ->
                    placeable.placeRelative(xOffset, yOffset)
                    xOffset += placeable.width + hSpacing
                }
                yOffset += row.height + vSpacing
            }
        }
    }
}

4. The Modifier Chain — Internals

Modifiers are a linked list that wraps around a LayoutNode. Each modifier creates a layer:

// What you write:
Box(
    modifier = Modifier
        .padding(16.dp)
        .background(Color.Blue)
        .size(100.dp)
        .clickable { }
)

// What the system sees (outside-in order):
// LayoutNode
//   └── PaddingModifier(16.dp)
//       └── BackgroundModifier(Color.Blue)
//           └── SizeModifier(100.dp)
//               └── ClickableModifier
//                   └── [content]

// Measure flows INWARD: padding → background → size → content
// Draw flows OUTWARD: content → size → background → padding

Modifier.Node Architecture (Compose 1.3+)

// Old: Modifier.Element — created new objects every recomposition
// New: Modifier.Node — persistent, updateable, much more efficient

// Creating a custom modifier with Modifier.Node:
class CircleClipNode : Modifier.Node(), DrawModifierNode {
    override fun ContentDrawScope.draw() {
        clipPath(Path().apply { addOval(Rect(0f, 0f, size.width, size.height)) }) {
            this@draw.drawContent()
        }
    }
}

fun Modifier.circleClip() = this then object : ModifierNodeElement<CircleClipNode>() {
    override fun create() = CircleClipNode()
    override fun update(node: CircleClipNode) { /* nothing to update */ }
    override fun hashCode() = "circleClip".hashCode()
    override fun equals(other: Any?) = other is CircleClipNode
}
Staff-Level Insight — Modifier.Node vs Modifier.Element

The old Modifier.composed { } API created a new modifier instance on every recomposition, which meant re-allocating objects and re-attaching to the LayoutNode. Modifier.Node is persistent — it's created once and updated in-place when parameters change. This eliminates allocation churn and enables the runtime to diff modifier chains efficiently. For L6 interviews: always use Modifier.Node for custom modifiers. Modifier.composed is effectively deprecated for new code.

5. Custom Drawing with Canvas

@Composable
fun PieChart(
    data: List<PieSlice>,
    modifier: Modifier = Modifier
) {
    Canvas(modifier = modifier.aspectRatio(1f)) {
        val total = data.sumOf { it.value.toDouble() }.toFloat()
        var startAngle = -90f

        data.forEach { slice ->
            val sweepAngle = (slice.value / total) * 360f
            drawArc(
                color = slice.color,
                startAngle = startAngle,
                sweepAngle = sweepAngle,
                useCenter = true,
                size = size
            )
            startAngle += sweepAngle
        }
    }
}

data class PieSlice(val value: Float, val color: Color)

drawBehind vs drawWithContent vs drawWithCache

// drawBehind: draw BEHIND the content (background)
Modifier.drawBehind {
    drawCircle(Color.Red, radius = size.minDimension / 2)
}

// drawWithContent: control draw order (before/after content)
Modifier.drawWithContent {
    drawContent()  // Draw the composable's content first
    // Then draw overlay on top:
    drawRect(Color.Black.copy(alpha = 0.3f))
}

// drawWithCache: cache expensive drawing operations
Modifier.drawWithCache {
    val path = Path().apply {
        // Expensive path calculation — only recalculated when size changes
        moveTo(0f, size.height)
        cubicTo(size.width * 0.25f, size.height * 0.5f,
                size.width * 0.75f, size.height * 0.8f,
                size.width, 0f)
    }
    onDrawBehind {
        drawPath(path, Color.Blue, style = Stroke(width = 4f))
    }
}

6. SubcomposeLayout — Measure-Then-Compose

SubcomposeLayout lets you compose children during the layout phase, after you know the available size. This is how LazyColumn, BoxWithConstraints, and Scaffold work.

@Composable
fun TwoPane(
    first: @Composable () -> Unit,
    second: @Composable () -> Unit,
    modifier: Modifier = Modifier,
    splitRatio: Float = 0.5f
) {
    SubcomposeLayout(modifier) { constraints ->
        val firstWidth = (constraints.maxWidth * splitRatio).roundToInt()
        val secondWidth = constraints.maxWidth - firstWidth

        // Compose and measure first pane with exact width
        val firstPlaceables = subcompose("first") { first() }
            .map { it.measure(constraints.copy(minWidth = firstWidth, maxWidth = firstWidth)) }

        // Compose and measure second pane with remaining width
        val secondPlaceables = subcompose("second") { second() }
            .map { it.measure(constraints.copy(minWidth = secondWidth, maxWidth = secondWidth)) }

        val height = maxOf(
            firstPlaceables.maxOfOrNull { it.height } ?: 0,
            secondPlaceables.maxOfOrNull { it.height } ?: 0
        )

        layout(constraints.maxWidth, height) {
            firstPlaceables.forEach { it.placeRelative(0, 0) }
            secondPlaceables.forEach { it.placeRelative(firstWidth, 0) }
        }
    }
}
Staff-Level Insight — SubcomposeLayout Cost

SubcomposeLayout defers composition to the layout phase, which means it can't be parallelized with other compositions and adds latency. It also creates a separate Composition for each subcompose() call, with its own slot table. Use it only when you genuinely need to know the size before composing children. For most cases, regular Layout with constraints is sufficient and more performant. BoxWithConstraints is a common misuse — developers use it to read screen size, but LocalConfiguration.current or window size classes are cheaper alternatives.

7. Intrinsic Measurements

Intrinsics let a parent ask "how big would you like to be?" without actually measuring. This enables two-pass-like behavior within the single-pass constraint:

// IntrinsicSize.Min — set width/height to the minimum intrinsic size
Row(Modifier.height(IntrinsicSize.Min)) {
    Text("Short")
    Divider(
        Modifier
            .fillMaxHeight()  // Fill the Row's height
            .width(1.dp)
    )
    Text("This is a longer text\nthat spans two lines")
}
// Without IntrinsicSize.Min: Row doesn't know how tall to be before measuring children
// With IntrinsicSize.Min: Row asks children for their minimum intrinsic height,
// uses the max as its height, then measures children with that constraint

// Providing intrinsics in a custom layout:
@Composable
fun MyCustomLayout(
    modifier: Modifier = Modifier,
    content: @Composable () -> Unit
) {
    Layout(
        content = content,
        modifier = modifier,
        measurePolicy = object : MeasurePolicy {
            override fun MeasureScope.measure(
                measurables: List<Measurable>,
                constraints: Constraints
            ): MeasureResult {
                // Normal measure logic...
                val placeables = measurables.map { it.measure(constraints) }
                return layout(constraints.maxWidth, constraints.maxHeight) {
                    placeables.forEach { it.placeRelative(0, 0) }
                }
            }

            override fun IntrinsicMeasureScope.minIntrinsicHeight(
                measurables: List<IntrinsicMeasurable>,
                width: Int
            ): Int {
                // Return the minimum height this layout needs for the given width
                return measurables.maxOfOrNull { it.minIntrinsicHeight(width) } ?: 0
            }

            override fun IntrinsicMeasureScope.maxIntrinsicWidth(
                measurables: List<IntrinsicMeasurable>,
                height: Int
            ): Int {
                return measurables.sumOf { it.maxIntrinsicWidth(height) }
            }
        }
    )
}
Staff-Level Insight — Intrinsics vs SubcomposeLayout

Both solve the "I need to know child sizes before deciding layout" problem, but differently:

  • Intrinsics: Cheap. Just asks for a size estimate without actually composing or measuring. O(n) walk of the tree. Use when you need approximate sizes (e.g., "make all items the same height as the tallest").
  • SubcomposeLayout: Expensive. Actually composes and measures children during layout. Use when you need exact sizes or when children depend on available space (e.g., LazyColumn only composes visible items).

Rule of thumb: try intrinsics first. Only use SubcomposeLayout if intrinsics can't express what you need.

8. ParentData — Communicating from Child to Parent

// ParentData lets children pass data to their parent's measure policy.
// This is how Column's Modifier.weight() works:

// Step 1: Define the parent data
data class MyLayoutParentData(val weight: Float = 0f)

// Step 2: Create a modifier that attaches parent data
fun Modifier.weight(weight: Float) = this.then(
    object : ParentDataModifier {
        override fun Density.modifyParentData(parentData: Any?): Any {
            return (parentData as? MyLayoutParentData ?: MyLayoutParentData()).copy(weight = weight)
        }
    }
)

// Step 3: Read parent data in the measure policy
Layout(content = content, modifier = modifier) { measurables, constraints ->
    val totalWeight = measurables.sumOf {
        ((it.parentData as? MyLayoutParentData)?.weight ?: 0f).toDouble()
    }.toFloat()

    // Distribute space based on weights...
}

9. Lookahead Layout — Animation-Aware Measurement

// LookaheadScope (Compose 1.4+) enables measure-and-animate patterns:
// It measures the FINAL layout first (lookahead), then animates FROM current TO final.

LookaheadScope {
    Box(
        Modifier
            .animateBounds(this@LookaheadScope) // Animates size/position changes
            .then(if (expanded) Modifier.fillMaxWidth() else Modifier.width(100.dp))
    ) {
        Text("Content")
    }
}

// How it works internally:
// 1. Lookahead pass: measures all children at their FINAL size (after state change)
// 2. Approach pass: measures at the CURRENT animated size
// 3. Each frame: approach sizes interpolate toward lookahead sizes
// This enables layout animations without SubcomposeLayout

10. Lazy Layout Internals

LazyColumn/LazyRow are the most complex layouts in Compose. Understanding their internals is critical for L6 interviews:

// LazyColumn is a SubcomposeLayout that:
// 1. Knows the viewport size (from constraints)
// 2. Composes ONLY visible items + a small prefetch buffer
// 3. Disposes items that scroll off-screen
// 4. Reuses SubCompositions for items of the same contentType

// Internal architecture:
// LazyColumn
//   └── SubcomposeLayout (measures viewport)
//       └── LazyListMeasurePolicy
//           ├── itemProvider: LazyListItemProvider (maps index → content lambda)
//           ├── prefetchState: LazyListPrefetchState (prefetches upcoming items)
//           └── beyondBoundsInfo: tracks items composed outside visible bounds

// Key internal types:
// LazyListState — scroll position, visible items, animation
// LazyListItemProvider — maps data index to composable content
// LazyMeasuredItem — a measured item with size and offset
// LazyListMeasureResult — the final layout result (visible items + their positions)

Prefetching

// LazyColumn prefetches items BEFORE they become visible:
// 1. During scroll, it predicts which items will be visible next
// 2. It schedules composition of those items on idle frames
// 3. When the item scrolls into view, it's already composed → no jank

// contentType enables efficient reuse:
LazyColumn {
    items(messages, key = { it.id }, contentType = { it.type }) { msg ->
        when (msg.type) {
            MessageType.TEXT -> TextMessage(msg)
            MessageType.IMAGE -> ImageMessage(msg)
        }
    }
}
// Items with the same contentType can reuse each other's SubComposition
// Different contentTypes get separate SubCompositions
// This is analogous to RecyclerView's viewType
Staff-Level Insight — LazyColumn vs RecyclerView

LazyColumn doesn't "recycle" views like RecyclerView — it reuses SubCompositions. When an item scrolls off-screen, its SubComposition is kept alive (not disposed) if another item of the same contentType needs it. The new item's content is composed into the existing SubComposition, updating the slot table in-place. This is cheaper than creating a new SubComposition but more expensive than RecyclerView's view binding (which just rebinds data to existing Views). For very large lists (10K+ items), RecyclerView with Compose items (via ComposeView in ViewHolder) can still outperform pure LazyColumn due to RecyclerView's more aggressive recycling. But for most apps, LazyColumn with proper keys and contentType is sufficient.

11. The Semantics Tree — Accessibility & Testing

Compose maintains a semantics tree parallel to the LayoutNode tree. It describes the meaning of UI elements for accessibility services (TalkBack) and UI tests:

// Every composable can contribute semantics:
Text("Hello") // Automatically adds: text = "Hello", role = null
Button(onClick = { }) { Text("Submit") }
// Adds: role = Button, onClick action, text = "Submit" (merged from child)

// Custom semantics:
Box(
    Modifier.semantics {
        contentDescription = "Profile picture"
        testTag = "profile_avatar"
        role = Role.Image
    }
)

// The semantics tree has two views:
// 1. Merged tree — what TalkBack sees. Parent merges children's semantics.
//    Button { Text("Submit"); Icon(check) } → single "Submit" node
// 2. Unmerged tree — what tests see. Every semantics node is separate.
//    Button → child Text("Submit") → child Icon

// Modifier.semantics(mergeDescendants = true) creates a merge boundary
// This is how Button, Card, ListItem work — they merge children into one accessible node
Staff-Level Insight — Semantics for Testing

The semantics tree is the foundation of Compose UI testing. onNodeWithText("Submit") searches the semantics tree, not the LayoutNode tree. This means: (1) you must add testTag or meaningful text/contentDescription for testability, (2) merged semantics can hide individual elements from tests (use useUnmergedTree = true to access them), and (3) the semantics tree is built during composition and updated during recomposition — it's always in sync with the UI. For accessibility, ensure every interactive element has a content description and every image has either a description or null (decorative).

12. Gesture System Internals

// Compose's gesture system is built on PointerInputScope:
Modifier.pointerInput(Unit) {
    detectTapGestures(
        onTap = { offset -> /* tap at offset */ },
        onLongPress = { offset -> /* long press */ },
        onDoubleTap = { offset -> /* double tap */ }
    )
}

// How hit testing works:
// 1. Touch event arrives at AndroidComposeView.dispatchTouchEvent()
// 2. Converted to Compose PointerInputEvent
// 3. Hit test: walk the LayoutNode tree from root, checking bounds
//    - Nodes are tested in reverse child order (last child = on top)
//    - Each LayoutNode with a PointerInputModifier is a candidate
// 4. Matching nodes receive the event in order (parent before child for Initial pass)
// 5. After all nodes process: Main pass (child before parent)
// 6. Finally: Final pass (parent before child, for cleanup)

// Event propagation phases:
// Initial → Main → Final
// Similar to View's onInterceptTouchEvent → onTouchEvent pattern
// But more flexible: any node can consume the event at any phase

// Nested scrolling:
Modifier.nestedScroll(connection, dispatcher)
// Compose has built-in nested scroll support:
// Child scrollable → overflow → parent scrollable → overflow → ...
// No need for NestedScrollView like in the View system
Staff-Level Insight — Gesture Conflicts

The most common L6 gesture question: "How do you handle conflicting gestures?" In Compose, gestures compete via the awaitPointerEventScope coroutine. When multiple gesture detectors match (e.g., tap vs drag), the first one to call event.consume() wins. For complex cases, use detectTransformGestures (handles pan + zoom + rotate simultaneously) or build custom gesture detectors with awaitFirstDown() + awaitTouchSlopOrCancellation(). The key difference from Views: Compose gestures are coroutine-based, so you can express complex gesture sequences as sequential code instead of state machines.

13. Animation Internals

// Compose animations are built on a frame-by-frame dispatch system:

// Core type: Animatable — a value holder that animates between values
val alpha = remember { Animatable(0f) }
LaunchedEffect(Unit) {
    alpha.animateTo(1f, animationSpec = tween(durationMillis = 300))
}

// How it works internally:
// 1. animateTo() registers with the MonotonicFrameClock
// 2. Each frame (VSYNC), the clock calls back with the frame time
// 3. Animatable calculates the current value using the AnimationSpec
// 4. The value is stored in a MutableState → triggers recomposition
// 5. But smart usage reads the value in Draw phase → no recomposition

// AnimationSpec hierarchy:
// AnimationSpec<T> → VectorizedAnimationSpec<V>
//   ├── TweenSpec (duration-based, easing curve)
//   ├── SpringSpec (physics-based, damping + stiffness)
//   ├── KeyframesSpec (keyframe-based)
//   ├── SnapSpec (instant)
//   └── RepeatableSpec / InfiniteRepeatableSpec (wraps another spec)

// Performance-critical pattern:
// BAD: animate in Composition phase
val animatedAlpha by animateFloatAsState(targetValue = if (visible) 1f else 0f)
Box(Modifier.alpha(animatedAlpha)) // Reads in Composition → recompose every frame

// GOOD: animate in Draw phase
val animatedAlpha by animateFloatAsState(targetValue = if (visible) 1f else 0f)
Box(Modifier.graphicsLayer { alpha = animatedAlpha }) // Reads in Draw → skip recomposition
Staff-Level Insight — Transition vs animateAsState vs Animatable

animateXAsState: simplest API, fire-and-forget, state-driven. Use for single-value animations triggered by state changes. updateTransition: coordinates multiple animations together, ensures they stay in sync. Use for complex state-based animations (e.g., expanding a card changes size, color, and elevation simultaneously). Animatable: imperative, coroutine-based, most control. Use when you need to chain animations sequentially, handle gesture-driven animations, or implement custom physics. For L6: know that all three ultimately use the same MonotonicFrameClock and AnimationState internally — the difference is the API surface, not the performance.
