Slot Table Mechanics

The gap buffer data structure, groups and slots, how the compiler generates group calls, how remember stores data, and how keys enable identity

1. What Is the Slot Table?

The slot table is the persistent data structure that stores everything about a composition:

  • The tree structure of composable calls (which composable called which)
  • Parameter values passed to each composable
  • remember values
  • State objects
  • Node references (LayoutNodes)
  • Recomposition scope lambdas

It's implemented as a gap buffer — the same data structure used in text editors (like Emacs and VS Code) for efficient insertion and deletion at the cursor position.

2. The Gap Buffer

A gap buffer is an array with a "gap" (empty space) that can be moved to any position. Insertions and deletions at the gap position are O(1). Moving the gap is O(n) but amortized O(1) because compositions typically process nodes sequentially.

Staff-Level Insight — Why a Gap Buffer?

The Compose team chose a gap buffer over alternatives (linked list, tree, hash map) because composition is sequential — the Composer walks the tree top-to-bottom, left-to-right, which is exactly the access pattern gap buffers optimize for. Insertions and deletions at the current position are O(1). The gap buffer also has excellent cache locality (contiguous memory) compared to tree structures. The slot table actually uses two parallel arrays: an IntArray for group metadata (keys, sizes, flags) and an Array<Any?> for slot data (values, state objects, nodes). This separation reduces memory overhead because group metadata is all primitives.

3. Groups and Slots

The slot table stores data in groups. Each group represents a composable call or a structural element:

Group TypeCreated ByWhat It Stores
RestartGroup startRestartGroup(key) A recomposition scope. Contains the updateScope lambda. This is the unit of recomposition.
ReplaceGroup startReplaceGroup(key) Conditional content (if/else, when). If the key changes, the entire group is replaced.
MovableGroup startMovableGroup(key) Keyed content (key() composable). Can be reordered in the slot table without recreating.
Node startNode() A node in the tree (LayoutNode on Android). Links the slot table to the actual UI tree. 
// What the compiler generates for:
@Composable
fun MyScreen() {
    Column {
        if (showHeader) {
            Header()
        }
        Text("Hello")
    }
}

// Pseudo-code of generated group calls:
fun MyScreen($composer: Composer, $changed: Int) {
    $composer.startRestartGroup(0x1234)     // RestartGroup for MyScreen

    Column($composer, 0) {                  // Column creates a Node group internally
        $composer.startReplaceGroup(0x5678) // ReplaceGroup for the if/else
        if (showHeader) {
            Header($composer, 0)            // RestartGroup for Header
        }
        $composer.endReplaceGroup()

        Text("Hello", $composer, 0)         // RestartGroup for Text
    }

    $composer.endRestartGroup()
}

Slots Within Groups

Each group can have slots — data stored alongside the group. This is where remember values, state objects, and parameters live:

// remember stores its value as a slot in the current group:
@Composable
fun Counter() {
    // Slot table for this group:
    // Group: RestartGroup(key=0xABCD)
    //   Slot 0: MutableState(value=0)    ← from remember { mutableStateOf(0) }
    //   Slot 1: "Count: 0"               ← parameter to Text
    //   Child Group: RestartGroup for Text(...)
    //   Child Group: RestartGroup for Button(...)

    var count by remember { mutableStateOf(0) }
    Text("Count: $count")
    Button(onClick = { count++ }) { Text("+") }
}

4. How if/else and when Work

Conditional content uses ReplaceGroups. When the condition changes, the old group is removed and a new one is inserted:

@Composable
fun ConditionalContent(showA: Boolean) {
    // Compiler generates:
    $composer.startReplaceGroup(0x1111)
    if (showA) {
        $composer.startReplaceGroup(0x2222) // Key for the "true" branch
        ComponentA($composer, 0)
        $composer.endReplaceGroup()
    } else {
        $composer.startReplaceGroup(0x3333) // Different key for "false" branch
        ComponentB($composer, 0)
        $composer.endReplaceGroup()
    }
    $composer.endReplaceGroup()
}

// When showA changes from true to false:
// 1. Composer sees key 0x3333 but slot table has key 0x2222
// 2. Keys don't match → REPLACE the group
// 3. Old group (ComponentA) is removed: all its remembered values, state, and nodes are disposed
// 4. New group (ComponentB) is inserted: fresh composition, new remembered values, new nodes
Critical — State Loss on Branch Change

When an if/else branch changes, all state in the old branch is lost. remember values, mutableStateOf, LaunchedEffect coroutines — everything is disposed. This is by design (the old UI no longer exists), but it catches developers off guard. If you need state to survive branch changes, hoist it above the conditional.

5. How Lists and key() Work

// Without key: positional identity
@Composable
fun ItemList(items: List<String>) {
    Column {
        items.forEachIndexed { index, item ->
            // Compiler uses index as implicit key
            // Position 0 → slot 0, Position 1 → slot 1, etc.
            ItemRow(item)
        }
    }
}

// With key: explicit identity
@Composable
fun ItemList(items: List<String>) {
    Column {
        items.forEach { item ->
            key(item) {
                // Compiler generates a MovableGroup with this key
                // The group can be found by key regardless of position
                ItemRow(item)
            }
        }
    }
}

// What happens when items are reordered (without key):
// Before: [A, B, C] → Slot 0=A, Slot 1=B, Slot 2=C
// After:  [C, A, B] → Slot 0 gets C (was A), Slot 1 gets A (was B), Slot 2 gets B (was C)
// All three ItemRows RECOMPOSE because their params changed
// All remembered state is now associated with the WRONG item

// What happens when items are reordered (with key):
// Before: [A, B, C] → Key "A"=slot X, Key "B"=slot Y, Key "C"=slot Z
// After:  [C, A, B] → Key "C" found at slot Z, moved to position 0
//                      Key "A" found at slot X, moved to position 1
//                      Key "B" found at slot Y, moved to position 2
// No recomposition needed — just MOVE the groups in the slot table
// All remembered state follows the correct item
Staff-Level Insight — The Cost of key()

Using key() adds overhead: the Composer must maintain a hash map of key → slot position, and moving groups in the gap buffer requires shifting data. For small, static lists, positional identity (no key) is faster. For dynamic lists with reordering, insertion, or deletion, key() is essential for correctness. LazyColumn with key = { it.id } is the canonical example — without keys, scrolling and item animations break because state doesn't follow items.

6. Slot Table During Recomposition

During recomposition, the Composer walks the slot table with a cursor:

  1. Start group: Move cursor to the group's position. Check if the key matches.
  2. Key matches: The group exists from last composition. Check if parameters changed.
  3. Parameters unchanged: Skip the group (advance cursor past all its slots and children).
  4. Parameters changed: Re-execute the composable body, updating slots in-place.
  5. Key doesn't match: The old group is removed, a new one is inserted at the gap.
  6. End group: Advance cursor past the group.
// Recomposition walk (pseudo-code):
fun recompose(scope: RecompositionScope) {
    composer.startGroup(scope.key)

    if (composer.inserting) {
        // First composition — everything is new
        scope.block(composer)
    } else if (composer.skipping) {
        // Parameters unchanged — skip entire group
        composer.skipToGroupEnd()
    } else {
        // Parameters changed — re-execute
        scope.block(composer)
        // Composer compares new output with old slots
        // Unchanged children are skipped
        // Changed children are updated in-place
        // New children are inserted at the gap
        // Removed children are deleted (gap expands)
    }

    composer.endGroup()
}
Staff-Level Insight — Why Slot Table Is O(changed) Not O(total)

The key insight: recomposition cost is proportional to what changed, not the total tree size. When the Composer skips a group, it advances the cursor past all its slots and children in O(1) — it knows the group's size from the metadata. Only invalidated scopes are re-executed. This is why a 10,000-node Compose tree can recompose in microseconds if only one Text changed — the Composer skips 9,999 groups and re-executes one.

7. The Composition Object

// The Composition ties everything together:
class CompositionImpl(
    private val parent: CompositionContext,
    private val applier: Applier<*>,       // UiApplier on Android
    private val slotTable: SlotTable,       // The gap buffer
    private val recomposer: Recomposer      // Schedules recomposition
) {
    fun setContent(content: @Composable () -> Unit) {
        // Initial composition:
        // 1. Create a Composer with the slot table
        // 2. Execute content() — populates the slot table
        // 3. Apply changes via the Applier — builds the node tree
    }

    fun recompose() {
        // 1. Get invalidated scopes from the snapshot system
        // 2. For each scope, re-execute its updateScope lambda
        // 3. Composer walks the slot table, updating changed groups
        // 4. Apply changes via the Applier — updates the node tree
    }

    fun dispose() {
        // 1. Walk the slot table
        // 2. Dispose all remembered values (call onDispose for DisposableEffect, etc.)
        // 3. Remove all nodes via the Applier
        // 4. Clear the slot table
    }
}
