The Compose Runtime

What the compiler plugin actually does, how @Composable functions are transformed, the node tree, Applier, Composer, and the architecture that makes declarative UI possible

1. Compose Is Not a UI Framework

The most important L6 insight: Compose Runtime is a general-purpose tree management library. It knows nothing about Android, Views, or pixels. The runtime manages a tree of nodes, and the UI layer (compose-ui) provides Android-specific nodes (LayoutNode) and rendering.

Staff-Level Insight — Why This Architecture Matters

Because the runtime is decoupled from the UI, Compose can manage any tree structure — not just UI. Google uses the Compose runtime for vector graphics (compose-animation), navigation graphs, and even server-side rendering experiments. In an interview, saying "Compose runtime manages a generic node tree via the Applier pattern, and compose-ui provides the Android-specific LayoutNode implementation" demonstrates deep architectural understanding.

2. The Compiler Plugin — What @Composable Really Does

Compiler Internals

The @Composable annotation is not a regular annotation — it triggers the Compose compiler plugin to transform the function at compile time. Here's what happens:

What You Write

@Composable
fun Greeting(name: String) {
    Text("Hello, $name")
}

What the Compiler Generates (Simplified)

fun Greeting(name: String, $composer: Composer, $changed: Int) {
    $composer.startRestartGroup(0x1a2b3c4d) // Unique key for this call site

    if ($changed and 0b0001 != 0 || !$composer.skipping) {
        // Parameters changed or first composition — execute the body
        Text("Hello, $name", $composer, 0)
    } else {
        // Parameters unchanged — SKIP the body entirely
        $composer.skipToGroupEnd()
    }

    $composer.endRestartGroup()?.updateScope { composer, _ ->
        // Lambda that re-invokes this function on recomposition
        Greeting(name, composer, $changed or 0b0001)
    }
}

What the Compiler Adds

AdditionPurpose
$composer: Composer parameter Thread-local object that manages the slot table, tracks state reads, and orchestrates recomposition. Every @Composable call passes this through.
$changed: Int bitmask Encodes which parameters changed since last composition. Each parameter gets bits. Used for smart recomposition — skip if nothing changed.
startRestartGroup(key) Marks the beginning of a recomposition scope in the slot table. The key is a hash of the source location (file + line + column).
endRestartGroup()?.updateScope { } Registers a lambda that can re-invoke this function. This is how the runtime knows how to recompose — it calls this lambda.
Skip logic If all parameters are unchanged (checked via $changed bitmask + equals()), the entire function body is skipped.
Staff-Level Insight — The $changed Bitmask

The $changed parameter uses 2 bits per parameter: one for "definitely changed" and one for "definitely unchanged." If the compiler can prove at compile time that a parameter is stable and unchanged (e.g., a literal or a val from an outer scope), it sets the "unchanged" bit. At runtime, the generated code checks these bits before calling equals() — avoiding the cost of equality checks when the compiler already knows the answer. This is why stability annotations (@Stable, @Immutable) matter so much for performance.

3. The Composer — Orchestrator of Everything

The Composer is the central object that:

  • Manages the slot table (stores the composition's data)
  • Tracks which state objects were read during composition (for invalidation)
  • Decides whether to skip or execute a composable's body
  • Coordinates with the Applier to update the node tree
// The Composer interface (simplified from compose-runtime source)
interface Composer {
    val inserting: Boolean       // true during first composition, false during recomposition
    val skipping: Boolean        // true when current group can be skipped

    fun startRestartGroup(key: Int): Composer
    fun endRestartGroup(): ScopeUpdateScope?

    fun startNode()              // Begin a node in the tree
    fun endNode()                // End a node

    fun <T> remember(calculation: () -> T): T  // Positional memoization
    fun changed(value: Any?): Boolean           // Has this value changed since last composition?

    fun skipToGroupEnd()         // Skip the current group (no changes)

    fun <T> createNode(factory: () -> T)  // Create a new node via the Applier
    fun useNode()                           // Reuse existing node (no changes)
}

4. The Applier — How Nodes Get Into the Tree

The Applier is the bridge between the Compose runtime and the actual node tree. It defines how to insert, move, and remove nodes:

// The Applier interface (from compose-runtime)
interface Applier<N> {
    val current: N                    // Current node being built

    fun onBeginChanges() {}
    fun onEndChanges() {}

    fun insertTopDown(index: Int, instance: N)   // Insert child at index
    fun insertBottomUp(index: Int, instance: N)  // Insert after children are built
    fun remove(index: Int, count: Int)           // Remove children
    fun move(from: Int, to: Int, count: Int)     // Reorder children

    fun clear()                       // Remove all children
    fun up()                          // Navigate to parent
    fun down(node: N)                 // Navigate to child
}

// Android's implementation:
internal class UiApplier(root: LayoutNode) : AbstractApplier<LayoutNode>(root) {
    override fun insertBottomUp(index: Int, instance: LayoutNode) {
        current.insertAt(index, instance) // Adds LayoutNode to parent
    }
    override fun remove(index: Int, count: Int) {
        current.removeAt(index, count)
    }
    override fun move(from: Int, to: Int, count: Int) {
        current.move(from, to, count)
    }
}
Staff-Level Insight — insertTopDown vs insertBottomUp

The Applier has two insertion strategies: insertTopDown (parent first, then children) and insertBottomUp (children first, then attach to parent). Android's UiApplier uses bottom-up because it's more efficient — children are fully measured and laid out before being attached to the parent, avoiding redundant layout passes. This is the opposite of the View system, where addView() triggers a layout pass on every addition. The bottom-up strategy is a key reason Compose layout is faster than Views for complex hierarchies.

5. The Node Tree — LayoutNode

On Android, the Compose node tree is made of LayoutNodes. Each LayoutNode holds:

  • Modifier chain — the linked list of modifiers (padding, click, background, etc.)
  • Measure policy — how to measure and place children (from Layout() or built-in composables)
  • Draw operations — the list of draw commands (from drawBehind, drawWithContent, etc.)
  • Children — child LayoutNodes
  • Semantics — accessibility and test info
// What happens when you write:
@Composable
fun MyScreen() {
    Column {          // Creates a LayoutNode with vertical measure policy
        Text("Hello") // Creates a LayoutNode with text draw + measure
        Button(       // Creates a LayoutNode with click modifier
            onClick = {}
        ) {
            Text("Click me") // Child LayoutNode inside Button's LayoutNode
        }
    }
}

// Resulting node tree:
// LayoutNode (Column — vertical measure policy)
//   ├── LayoutNode (Text "Hello" — text measure + draw)
//   └── LayoutNode (Button — click modifier)
//       └── LayoutNode (Text "Click me")
Staff-Level Insight — LayoutNode vs View

A LayoutNode is dramatically lighter than a View. A View has ~30 fields (padding, margins, scroll, focus, animation state, etc.) and inherits from a deep class hierarchy. A LayoutNode is a flat data structure with only what's needed: a modifier chain, a measure policy, and children. Modifiers are composed as a linked list, not inherited. This is why Compose can handle thousands of nodes without the memory overhead of the View system. In benchmarks, a LayoutNode is ~10x smaller than a View in memory.

6. Composition vs Recomposition — The Two Phases

PhaseWhenWhat Happens
Initial Composition First time a composable tree is rendered Composer is in inserting mode. Every composable executes. Slot table is populated. Nodes are created via Applier. State reads are tracked.
Recomposition When a read state value changes Composer is NOT inserting. Only invalidated scopes re-execute. Unchanged composables are skipped. Slot table is updated in-place. Applier applies diffs to the node tree. 
@Composable
fun Counter() {
    var count by remember { mutableStateOf(0) }  // State stored in slot table

    // This entire Column is a recomposition scope
    Column {
        Text("Count: $count")  // Reads 'count' — tracked by snapshot system
        Button(onClick = { count++ }) {
            Text("Increment")  // Does NOT read 'count' — not invalidated
        }
    }
}

// When count changes:
// 1. Snapshot system detects write to 'count'
// 2. Finds all scopes that READ 'count' during last composition
// 3. Invalidates those scopes (marks them for recomposition)
// 4. On next frame, Composer re-executes only invalidated scopes
// 5. Text("Count: $count") re-executes with new value
// 6. Button's Text("Increment") is SKIPPED (didn't read 'count')
Staff-Level Insight — Recomposition Is Not Re-rendering

A critical distinction: recomposition (re-executing @Composable functions) is separate from layout (measuring/placing nodes) and drawing (rendering pixels). Recomposition only updates the slot table and tells the Applier what changed. Layout and draw happen in a separate phase, and only for nodes that actually changed. This three-phase pipeline (Composition → Layout → Draw) is analogous to React's virtual DOM diff, but more efficient because Compose tracks state reads at a granular level instead of diffing entire subtrees.

7. The Three Phases of a Frame

  • Composition only: A state change that updates text content triggers recomposition but may not change layout (if the text fits in the same space).
  • Layout only: A size change (e.g., Modifier.size() animation) triggers layout but not composition.
  • Draw only: A color change (e.g., Modifier.drawBehind { }) triggers draw but not composition or layout.
Staff-Level Insight — Optimizing by Phase

The L6 performance optimization strategy: push state reads as late as possible in the pipeline. If you can read state in the Draw phase (via drawBehind) instead of Composition, you skip recomposition and layout entirely. If you can read state in the Layout phase (via Modifier.layout { }), you skip recomposition. This is why Modifier.graphicsLayer { alpha = animatedAlpha.value } is faster than Box(modifier = Modifier.alpha(animatedAlpha.value)) — the former reads state in the Draw phase, the latter in Composition.

8. The Recomposer — State Machine & Frame Loop

The Recomposer is the orchestrator that drives all compositions. It manages a state machine and coordinates with the platform's frame clock (Choreographer on Android).

Recomposer State Machine

// Recomposer states (from source):
enum class State {
    ShutDown,           // Terminal state — Recomposer is dead
    ShuttingDown,       // Disposing all compositions
    Inactive,           // Created but not yet running (no frame clock)
    InactivePendingWork,// Has invalidated scopes but no frame clock to process them
    Idle,               // Running, no pending work — waiting for invalidations
    PendingWork         // Has invalidated scopes — will recompose on next frame
}

// Transitions:
// Inactive → Idle (when frame clock is available, e.g. window attached)
// Idle → PendingWork (when a state write invalidates a scope)
// PendingWork → Idle (after recomposition completes)
// Idle → ShuttingDown → ShutDown (when Recomposer is cancelled)
// InactivePendingWork → PendingWork (when frame clock becomes available)

The Frame Loop

// Simplified from Recomposer.runRecomposeAndApplyChanges():
suspend fun runRecomposeAndApplyChanges() {
    val toRecompose = mutableListOf<ControlledComposition>()

    while (shouldKeepRecomposing) {
        // 1. Wait for the next frame (suspends until Choreographer VSYNC)
        awaitWorkAvailable()
        frameClockMark = MonotonicFrameClock.withFrameNanos { frameTimeNanos ->

            // 2. Collect all compositions with invalidated scopes
            synchronized(stateLock) {
                toRecompose.addAll(compositionsWithInvalidations)
                compositionsWithInvalidations.clear()
            }

            // 3. Take a global snapshot (consistent view of all state)
            Snapshot.sendApplyNotifications()

            // 4. Recompose each invalidated composition
            for (composition in toRecompose) {
                composition.recompose() // Walks slot table, re-executes scopes
            }

            // 5. Apply changes to node trees
            for (composition in toRecompose) {
                composition.applyChanges() // Applier inserts/removes/moves nodes
            }

            toRecompose.clear()
        }
    }
}
Staff-Level Insight — Frame Overrun

If recomposition takes longer than 16ms (one frame), the Recomposer doesn't panic — it simply misses the current frame and continues on the next VSYNC. The Choreographer callback is re-registered. This means janky frames are possible if recomposition is expensive. The key diagnostic: in a Perfetto trace, look for "Compose:recompose" sections that span more than one VSYNC interval. The fix is always to reduce recomposition scope (extract smaller composables, fix stability, defer state reads to later phases).

9. Composition vs SubComposition

A Composition is the top-level container that owns a slot table and is driven by a Recomposer. A SubComposition is a child composition created during the layout phase:

AspectCompositionSubComposition
Created by setContent { } / ComposeView SubcomposeLayout, LazyColumn, AndroidView
Slot table Own slot table Own separate slot table
Recomposer Owns or inherits one Shares parent's Recomposer
When composed During Composition phase During Layout phase (after parent is measured)
CompositionLocals Provides to children Inherits from parent via CompositionContext
Invalidation Scheduled on next frame Can trigger parent relayout (since it runs during layout) 
// How LazyColumn uses SubComposition:
// 1. LazyColumn is a SubcomposeLayout
// 2. During layout, it knows the viewport size
// 3. It subcomposes ONLY the visible items (e.g., items 5-15 of 1000)
// 4. Each subcompose() call creates/reuses a SubComposition with its own slot table
// 5. When the user scrolls, new items are subcomposed, off-screen items are disposed
// 6. This is why LazyColumn is O(visible) not O(total)
Staff-Level Insight — SubComposition Cost

Each SubComposition has its own slot table, which means memory overhead per sub-composition. LazyColumn mitigates this by reusing SubCompositions for items of the same contentType. But BoxWithConstraints creates a SubComposition even for static content — it's a common performance trap. Prefer LocalConfiguration.current or window size classes over BoxWithConstraints when you just need screen dimensions. The SubComposition also can't be parallelized with the parent's composition — it runs sequentially during layout, adding latency to the frame.

10. AndroidComposeView — The Bridge to the View System

AndroidComposeView is the single Android View that hosts the entire Compose tree. It implements the Owner interface:

// AndroidComposeView (simplified):
internal class AndroidComposeView(context: Context) : ViewGroup(context), Owner {
    val root: LayoutNode = LayoutNode()  // Root of the Compose node tree
    private val recomposer: Recomposer

    // Called by the View system when layout is needed:
    override fun onMeasure(widthMeasureSpec: Int, heightMeasureSpec: Int) {
        // Convert View MeasureSpec → Compose Constraints
        val constraints = convertToConstraints(widthMeasureSpec, heightMeasureSpec)
        // Measure the root LayoutNode (triggers entire Compose layout)
        root.measure(constraints)
        setMeasuredDimension(root.width, root.height)
    }

    override fun onLayout(changed: Boolean, l: Int, t: Int, r: Int, b: Int) {
        // Place the root LayoutNode
        root.place(0, 0)
    }

    override fun dispatchDraw(canvas: Canvas) {
        // Draw the entire Compose tree
        root.draw(canvas)
    }

    // Owner interface — how LayoutNodes request work:
    override fun onRequestMeasure(layoutNode: LayoutNode) {
        // Mark node as needing measure, schedule layout pass
        requestLayout() // Triggers View system's layout cycle
    }

    override fun onRequestRelayout(layoutNode: LayoutNode) {
        // Mark node as needing placement only
    }
}
Staff-Level Insight — One View, Entire Tree

The entire Compose UI is rendered inside a single AndroidComposeView. There's no View-per-composable — all LayoutNodes are managed within this one ViewGroup. This is why Compose has dramatically less overhead than the View system for complex UIs. The View system's measure()/layout()/draw() cycle is called exactly once on AndroidComposeView, which then dispatches to the Compose layout/draw pipeline internally. Touch events flow through AndroidComposeView.dispatchTouchEvent() into Compose's pointer input system.
