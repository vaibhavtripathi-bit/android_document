Compose ↔ View Interop

ComposeView, AndroidView, Fragment interop, hybrid navigation, performance pitfalls, and the migration strategy for large codebases

1. The Two Directions

DirectionAPIUse Case
Compose in Views ComposeView Add Compose to an existing View-based Activity/Fragment. The starting point for migration.
Views in Compose AndroidView Use a View (MapView, WebView, AdView, custom View) inside a Compose screen.

2. ComposeView — Compose Inside Views

// In an Activity:
class MainActivity : ComponentActivity() {
    override fun onCreate(savedInstanceState: Bundle?) {
        super.onCreate(savedInstanceState)
        setContent {  // This creates a ComposeView internally
            MyApp()
        }
    }
}

// In a Fragment (XML layout):
// fragment_profile.xml:
// <androidx.compose.ui.platform.ComposeView
//     android:id="@+id/compose_view"
//     android:layout_width="match_parent"
//     android:layout_height="match_parent" />

class ProfileFragment : Fragment() {
    override fun onCreateView(inflater: LayoutInflater, container: ViewGroup?, savedInstanceState: Bundle?): View {
        return ComposeView(requireContext()).apply {
            setViewCompositionStrategy(ViewCompositionStrategy.DisposeOnViewTreeLifecycleDestroyed)
            setContent {
                ProfileScreen()
            }
        }
    }
}

ViewCompositionStrategy — Critical for Fragments

StrategyDisposes WhenUse Case
DisposeOnDetachedFromWindow ComposeView detaches from window Default. Fine for Activities. Breaks in Fragments with ViewPager/transitions.
DisposeOnViewTreeLifecycleDestroyed Fragment's view lifecycle is destroyed Use this for Fragments. Handles Fragment view recreation correctly.
DisposeOnLifecycleDestroyed(lifecycle) Specific lifecycle is destroyed Custom lifecycle-aware disposal.
Critical — Fragment ViewPager Bug

The default DisposeOnDetachedFromWindow strategy disposes the Composition when the ComposeView detaches from the window. In a ViewPager, fragments are detached (but not destroyed) when swiped away. This disposes all Compose state. When the user swipes back, the Composition is recreated from scratch — losing all state. Always use DisposeOnViewTreeLifecycleDestroyed in Fragments.

3. AndroidView — Views Inside Compose

@Composable
fun MapScreen(location: LatLng) {
    AndroidView(
        factory = { context ->
            // Called ONCE — create the View
            MapView(context).apply {
                onCreate(Bundle())
                getMapAsync { map ->
                    map.uiSettings.isZoomControlsEnabled = true
                }
            }
        },
        update = { mapView ->
            // Called on EVERY recomposition — update the View
            mapView.getMapAsync { map ->
                map.moveCamera(CameraUpdateFactory.newLatLng(location))
            }
        },
        modifier = Modifier.fillMaxSize()
    )
}

// For Views that need lifecycle management:
@Composable
fun WebViewScreen(url: String) {
    var webView by remember { mutableStateOf<WebView?>(null) }

    AndroidView(
        factory = { context ->
            WebView(context).apply {
                settings.javaScriptEnabled = true
                webViewClient = WebViewClient()
                loadUrl(url)
            }.also { webView = it }
        },
        update = { view ->
            if (view.url != url) {
                view.loadUrl(url)
            }
        }
    )

    DisposableEffect(Unit) {
        onDispose {
            webView?.destroy()
        }
    }
}
Staff-Level Insight — AndroidView Performance

AndroidView creates a real Android View and embeds it in the Compose LayoutNode tree via a special ViewGroup wrapper. This has overhead: the View goes through the full View measure/layout/draw pipeline, which is separate from Compose's pipeline. For simple views, consider reimplementing in Compose. For complex views (MapView, WebView, camera preview), AndroidView is the only option. Key optimization: make the update lambda as cheap as possible — it runs on every recomposition. Don't recreate objects or call expensive View methods unless the data actually changed.

5. Hybrid Navigation — The Migration Challenge

// Strategy 1: Fragment-based navigation with Compose screens
// Keep Navigation Component (Fragments), replace Fragment content with Compose
class HomeFragment : Fragment() {
    override fun onCreateView(inflater: LayoutInflater, container: ViewGroup?, savedInstanceState: Bundle?): View {
        return ComposeView(requireContext()).apply {
            setViewCompositionStrategy(ViewCompositionStrategy.DisposeOnViewTreeLifecycleDestroyed)
            setContent {
                HomeScreen(
                    onNavigateToDetail = {
                        findNavController().navigate(R.id.action_home_to_detail)
                    }
                )
            }
        }
    }
}

// Strategy 2: Compose Navigation with View destinations
// Use Compose NavHost, embed Views via AndroidView where needed
NavHost(navController, startDestination = "home") {
    composable("home") { HomeScreen() }
    composable("legacy_detail") {
        AndroidView(factory = { context ->
            LegacyDetailView(context) // Old View-based screen
        })
    }
}

// Strategy 3: Single-Activity Compose Navigation (target state)
// Full Compose with Navigation Compose
NavHost(navController, startDestination = "home") {
    composable("home") { HomeScreen() }
    composable("detail/{id}") { backStackEntry ->
        DetailScreen(id = backStackEntry.arguments?.getString("id") ?: "")
    }
}
Staff-Level Insight — Migration Strategy for Large Apps

The Google-recommended migration path for large apps:

  1. Bottom-up: Start with leaf composables (buttons, cards, list items). Replace XML includes with Compose.
  2. Screen-by-screen: Replace Fragment content with ComposeView. Keep Fragment navigation.
  3. Navigation migration: Move from Fragment Navigation to Compose Navigation. This is the hardest step — do it last.
  4. Remove Fragments: Once all screens are Compose, remove Fragment dependencies.

Key principle: never rewrite — migrate incrementally. Both systems coexist. The interop APIs are designed for long-term hybrid codebases, not just temporary bridges.

6. State Sharing Between Compose and Views

// Sharing state via ViewModel (recommended):
class SharedViewModel : ViewModel() {
    val userName = MutableStateFlow("Alice")
}

// In Compose:
@Composable
fun ComposeScreen(viewModel: SharedViewModel = viewModel()) {
    val name by viewModel.userName.collectAsState()
    Text("Hello, $name")
}

// In View (Fragment):
class ViewFragment : Fragment() {
    private val viewModel: SharedViewModel by activityViewModels()

    override fun onViewCreated(view: View, savedInstanceState: Bundle?) {
        viewLifecycleOwner.lifecycleScope.launch {
            viewModel.userName.collect { name ->
                binding.nameText.text = "Hello, $name"
            }
        }
    }
}

// Sharing Compose state with Views via ComposeView:
@Composable
fun BridgeComponent(onStateChanged: (String) -> Unit) {
    var text by remember { mutableStateOf("") }
    TextField(value = text, onValueChange = {
        text = it
        onStateChanged(it) // Push to View world
    })
}

7. Theming Interop

// Bridge Material XML theme to Compose Material theme:
// Use the MDC-Compose theme adapter:
// implementation "com.google.android.material:compose-theme-adapter:1.x.x"

@Composable
fun AppWithBridgedTheme() {
    MdcTheme { // Reads colors, typography, shapes from XML theme
        // Compose components use the XML theme values
        Surface {
            Text("Themed text") // Uses XML-defined textAppearance
        }
    }
}

// For Material3:
// implementation "com.google.android.material:compose-theme-adapter-3:1.x.x"
@Composable
fun AppWithM3BridgedTheme() {
    Mdc3Theme {
        Surface {
            Text("M3 themed text")
        }
    }
}

8. Performance Considerations

ScenarioImpactMitigation
Multiple ComposeViews in one layout Each creates a separate Composition with its own slot table and Recomposer Minimize the number of ComposeViews. Prefer one large ComposeView over many small ones.
AndroidView in a LazyColumn Views are created/destroyed on scroll (no recycling like RecyclerView) Use AndroidView with onReset parameter (Compose 1.4+) for view recycling. Or use AndroidViewBinding.
Nested scrolling (Compose inside View scroller) Gesture conflicts between Compose and View touch handling Use NestedScrollInterop modifier: Modifier.nestedScroll(rememberNestedScrollInteropConnection())
ComposeView in RecyclerView items Composition created/disposed on every bind/recycle Use ViewCompositionStrategy.DisposeOnViewTreeLifecycleDestroyed and keep Compose content minimal.
Staff-Level Insight — The Rendering Pipeline Boundary

When Compose and Views coexist, there are two rendering pipelines running in the same frame: the View system's measure/layout/draw and Compose's composition/layout/draw. They synchronize via the Choreographer (both run on the same VSYNC), but they're independent. A Compose recomposition doesn't trigger View relayout, and vice versa. The boundary is the ComposeView (View → Compose) or the wrapper ViewGroup inside AndroidView (Compose → View). Understanding this boundary is critical for debugging layout issues in hybrid apps — a size change in Compose won't automatically resize the parent View unless you explicitly request it.
