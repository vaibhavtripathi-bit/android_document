1. App Thinning

App thinning is Apple's umbrella term for three technologies that reduce the size of the app delivered to each device. Understanding these is essential because the size users see on the App Store is the thinned size, not your archive size.

1.1 Three Components

App Thinning Components: ┌─────────────────────────────────────────────────────────────────┐ │ App Thinning │ ├──────────────────┬──────────────────┬───────────────────────────┤ │ │ │ │ │ App Slicing │ On-Demand │ Bitcode │ │ │ Resources │ (deprecated Xcode 14) │ │ │ │ │ │ Device-specific │ Download assets │ Server-side │ │ variants: │ only when │ recompilation │ │ • Architecture │ needed │ (was: Apple optimizes │ │ • GPU assets │ • Game levels │ for each device) │ │ • Display scale │ • Tutorials │ │ │ │ • Rare features │ │ └──────────────────┴──────────────────┴───────────────────────────┘

1.2 App Slicing

App slicing creates device-specific variants of your app. The App Store delivers only the variant matching the user's device.

Slicing Dimension What Gets Removed Typical Savings
Architecture Unused CPU architectures (e.g., x86_64 for device builds) ~50% of executable size
Asset catalogs Unused scale variants (@1x on @3x device) ~30-60% of asset size
GPU resources Metal shaders for other GPU families Variable
Device class iPad-only or iPhone-only resources Variable

1.3 On-Demand Resources (ODR)

// Tag resources in asset catalog or build phases
// Tags: "level1", "level2", "tutorial", etc.

// Request resources at runtime
let request = NSBundleResourceRequest(tags: ["level5"])
request.loadingPriority = NSBundleResourceRequestLoadingPriorityUrgent

request.beginAccessingResources { error in
    if let error = error {
        print("Failed to download: \(error)")
        return
    }
    // Resources are now available
    let levelData = NSDataAsset(name: "level5_data")
}

// Release resources when done
request.endAccessingResources()

// Check available disk space before downloading
request.conditionallyBeginAccessingResources { available in
    if available {
        // Already downloaded
    } else {
        // Need to download — show progress UI
        request.beginAccessingResources { error in /* ... */ }
    }
}
ODR Hosting

On-demand resources are hosted by Apple's CDN at no additional cost. The system manages a local cache with automatic eviction. You can tag resources as "initial install" (downloaded with app), "prefetch" (downloaded after install), or "on-demand" (downloaded when requested). Total ODR limit is 20GB per app, with 512MB initial install limit for non-streaming resources.

2. Bitcode Removal (Xcode 14)

Bitcode was Apple's intermediate representation that allowed server-side recompilation for each device architecture. It was deprecated in Xcode 14 and removed from the App Store pipeline.

2.1 What Changed

Aspect Before Xcode 14 Xcode 14+
ENABLE_BITCODE YES (default for iOS) Deprecated, ignored
Archive size Larger (includes bitcode) Smaller (no bitcode)
Upload time Longer Shorter
Server-side optimization Apple could recompile No recompilation
Third-party SDKs Must include bitcode No bitcode required 
# Remove bitcode from existing frameworks (if migrating)
xcrun bitcode_strip -r MyFramework.framework/MyFramework -o MyFramework_stripped

# Check if binary contains bitcode
otool -l MyFramework | grep __LLVM

3. Asset Catalog Optimization

Asset catalogs (.xcassets) are compiled into a single Assets.car file by actool. Optimizing assets is often the highest-impact size reduction for content-heavy apps.

3.1 Image Optimization Strategies

Strategy Savings Tradeoff
Use PDF vectors instead of PNGs 60-80% Only for simple icons/illustrations
Use SVG (iOS 13+) 60-80% Limited to simple graphics
Use HEIF instead of JPEG 40-50% iOS 11+ only
Compress PNGs with pngquant/ImageOptim 30-70% Lossy compression
Remove unused assets Variable Need tooling to find unused
Provide only @2x and @3x (drop @1x) ~15% No @1x devices in active use
Use SF Symbols instead of custom icons 100% per replaced icon Limited to Apple's icon set 
# Analyze Assets.car contents
xcrun --sdk iphoneos assetutil --info Assets.car | python3 -m json.tool

# Find large assets
xcrun --sdk iphoneos assetutil --info Assets.car | \
    python3 -c "
import json, sys
assets = json.load(sys.stdin)
for a in sorted(assets, key=lambda x: x.get('SizeOnDisk', 0), reverse=True)[:20]:
    print(f\"{a.get('SizeOnDisk', 0):>10} bytes  {a.get('Name', 'unknown')}\")"

4. __TEXT Segment Analysis

The __TEXT segment contains executable code, string literals, and other read-only data. It's typically the largest segment in an iOS binary and the primary target for size optimization.

4.1 Mach-O Segment Layout

Mach-O Binary Layout: ┌─────────────────────────────────────┐ │ Mach-O Header │ Load commands, magic number ├─────────────────────────────────────┤ │ __TEXT Segment │ Read-only, executable │ ┌─────────────────────────────┐ │ │ │ __text │ │ Machine code instructions │ │ __stubs │ │ Stub functions for lazy binding │ │ __stub_helper │ │ Helper code for stubs │ │ __cstring │ │ C string literals │ │ __swift5_typeref │ │ Swift type reference strings │ │ __swift5_reflstr │ │ Swift reflection strings │ │ __swift5_proto │ │ Swift protocol conformances │ │ __objc_methname │ │ ObjC method name strings │ │ __objc_classname │ │ ObjC class name strings │ │ __const │ │ Constant data │ └─────────────────────────────┘ │ ├─────────────────────────────────────┤ │ __DATA Segment │ Read-write │ ┌─────────────────────────────┐ │ │ │ __objc_classlist │ │ ObjC class references │ │ __objc_protolist │ │ ObjC protocol references │ │ __swift5_types │ │ Swift type metadata │ │ __got │ │ Global offset table │ │ __la_symbol_ptr │ │ Lazy symbol pointers │ │ __data │ │ Initialized global data │ │ __bss │ │ Uninitialized global data │ └─────────────────────────────┘ │ ├─────────────────────────────────────┤ │ __LINKEDIT Segment │ Linker metadata │ ┌─────────────────────────────┐ │ │ │ Symbol table │ │ Symbol names and addresses │ │ String table │ │ Symbol name strings │ │ Code signature │ │ Code signing data │ │ Fixup chains │ │ dyld binding info │ └─────────────────────────────┘ │ └─────────────────────────────────────┘

4.2 Analyzing Segment Sizes

# Detailed section sizes

5. Swift Metadata Size Impact

Swift's runtime requires type metadata, protocol conformance records, and reflection strings that add to binary size. Understanding this overhead helps you make informed API design decisions.

5.1 Swift Metadata Sections

Section Contents Size Impact
__swift5_types Type descriptor records for every Swift type ~100-200 bytes per type
__swift5_typeref Mangled type name strings for reflection ~50-100 bytes per type
__swift5_reflstr Field name strings for Mirror reflection ~20-50 bytes per stored property
__swift5_proto Protocol conformance records ~12 bytes per conformance
__swift5_protos Protocol descriptor records ~40 bytes per protocol
__swift5_fieldmd Field metadata (types and names) Variable per type

5.2 Reducing Swift Metadata

// Each type generates metadata. More types = more metadata.

// HIGH metadata cost: many small types
struct UserName { let value: String }
struct UserEmail { let value: String }
struct UserAge { let value: Int }
struct UserID { let value: UUID }
// 4 type descriptors, 4 type refs, 4 field metadata records

// LOWER metadata cost: fewer types with more properties
struct User {
    let name: String
    let email: String
    let age: Int
    let id: UUID
}
// 1 type descriptor, 1 type ref, 1 field metadata record

// Generic specializations also add metadata
func process<T: Codable>(_ value: T) { /* ... */ }
// Each concrete specialization (process(Int), process(String), etc.)
// generates a separate function body + metadata
# Measure Swift metadata size
size -m MyApp | grep swift5

# Disable reflection metadata (reduces size but breaks Mirror)
# In OTHER_SWIFT_FLAGS:
-Xfrontend -disable-reflection-metadata

# Disable reflection names only (keeps type metadata)
-Xfrontend -disable-reflection-names
Disabling Reflection Metadata

Disabling reflection metadata can save 5-15% of Swift code size, but it breaks Mirror, String(describing:) for custom types, and any code that relies on runtime type introspection. Test thoroughly before enabling in production. Codable still works because it uses compile-time generated code, not runtime reflection.

6. Link-Time Optimization (LTO)

LTO allows the compiler to optimize across translation unit boundaries during the link phase. It can significantly reduce binary size by enabling cross-module dead code elimination and inlining.

6.1 LTO Modes

Mode Build Setting Description Impact
None LLVM_LTO = NO No cross-module optimization Fastest build
Monolithic LLVM_LTO = YES Merges all LLVM IR, optimizes as one unit Best size reduction, very slow build
Thin LTO LLVM_LTO = YES_THIN Parallel optimization with cross-module info Good size reduction, moderate build time
Thin LTO Pipeline: Compile Phase: A.swift → A.o (with LLVM bitcode summary) B.swift → B.o (with LLVM bitcode summary) C.swift → C.o (with LLVM bitcode summary) Link Phase: 1. Read all summaries → build cross-module call graph 2. Decide: which functions to inline across modules 3. Import needed function bodies 4. Optimize each module in parallel (with cross-module info) 5. Link final binary vs Monolithic LTO: 1. Merge ALL LLVM IR into one giant module 2. Run all optimizations on the merged module (single-threaded) 3. Much slower but slightly better optimization 
// Enable Thin LTO for release builds
// In Release.xcconfig:
LLVM_LTO = YES_THIN

// Typical size savings: 5-15% of __TEXT segment
// Build time increase: 20-50% for Thin LTO, 2-5x for Monolithic

7. -Osize vs -O

Swift provides two optimization levels that trade off between runtime performance and binary size.

7.1 Comparison

Aspect -O (Speed) -Osize (Size)
Inlining threshold Aggressive (inline more) Conservative (inline less)
Loop unrolling Yes Minimal
Generic specialization Aggressive Selective (only hot paths)
Code duplication Allowed for speed Minimized
Typical size difference Baseline 10-30% smaller
Typical speed difference Baseline 0-5% slower 
// Per-module optimization level
// Use -O for performance-critical modules, -Osize for the rest

// In Package.swift:
.target(
    name: "ImageProcessing",
    swiftSettings: [
        .unsafeFlags(["-O"], .when(configuration: .release))
    ]
)

.target(
    name: "Settings",
    swiftSettings: [
        .unsafeFlags(["-Osize"], .when(configuration: .release))
    ]
)

8. Strip Symbols

Symbol stripping removes debug symbols and symbol names from the binary, reducing the __LINKEDIT segment size.

8.1 Stripping Levels

Setting Value Effect
STRIP_INSTALLED_PRODUCT YES Strip the final product
STRIP_STYLE all Remove all symbols (smallest binary)
STRIP_STYLE non-global Keep global symbols (needed for dynamic libs)
STRIP_STYLE debugging Remove debug symbols only
STRIP_SWIFT_SYMBOLS YES Strip Swift-specific symbols
COPY_PHASE_STRIP YES Strip during copy phases 
# Manual stripping
strip -x MyApp                # Remove local symbols
strip -S MyApp                # Remove debug symbols
strip MyApp                   # Remove all symbols (default)

# View symbols before/after
nm MyApp | wc -l              # Count symbols before
strip -x MyApp
nm MyApp | wc -l              # Count symbols after

# Generate dSYM before stripping (for crash symbolication)
dsymutil MyApp -o MyApp.dSYM
strip MyApp
Always Generate dSYMs Before Stripping

If you strip symbols without first generating a dSYM file, you lose the ability to symbolicate crash reports. Always run dsymutil before strip, or use Xcode's DEBUG_INFORMATION_FORMAT = dwarf-with-dsym which does this automatically during archive builds.

9. Binary Analysis Tools

These tools help you understand what's in your binary and where the size is coming from.

9.1 nm — Symbol Table

# List all symbols with sizes
nm -S MyApp | sort -k2 -rn | head -20

# List only undefined symbols (external dependencies)
nm -u MyApp

# List only global (exported) symbols
nm -g MyApp

# Demangle Swift symbols
nm MyApp | swift demangle | head -20

# Find largest symbols
nm -S --size-sort MyApp | tail -20

9.2 otool — Object File Tool

# View load commands (segments, dylibs, rpaths)
otool -l MyApp

# View linked dynamic libraries
otool -L MyApp

# Disassemble a specific function
otool -tV MyApp | grep -A 50 "_main:"

# View Objective-C metadata
otool -oV MyApp

# View section contents
otool -s __TEXT __cstring MyApp

9.3 bloaty — Binary Size Profiler

# Install bloaty (Google's binary size profiler)
brew install bloaty

# Analyze by section
bloaty MyApp

# Analyze by symbol
bloaty -d symbols MyApp

# Analyze by compile unit (source file)
bloaty -d compileunits MyApp

# Compare two binaries (before/after optimization)
bloaty MyApp_new -- MyApp_old

# Example output:
#     FILE SIZE        VM SIZE
# +++++++++++++++  +++++++++++++++
#   +12%  +1.2Mi   +12%  +1.2Mi    __TEXT,__text
#   -45%  -234Ki   -45%  -234Ki    __TEXT,__swift5_reflstr

11. Architect: 200MB to Under 50MB

Architect Must-Know

You've been asked to reduce a 200MB app to under 50MB (thinned download size). Here's a systematic approach that has worked for real production apps.

11.1 Phase 1: Measure (Week 1)

# Step 1: Get baseline sizes
size -m MyApp.app/MyApp
# Note: __TEXT, __DATA, __LINKEDIT sizes

# Step 2: Generate link map
# Build with GENERATE_LINK_MAP = YES
# Analyze top contributors

# Step 3: Analyze asset catalog
xcrun --sdk iphoneos assetutil --info MyApp.app/Assets.car

# Step 4: List all files in app bundle by size
find MyApp.app -type f -exec ls -la {} \; | sort -k5 -rn | head -30

# Step 5: Check for duplicate resources
find MyApp.app -type f -exec md5 {} \; | sort | uniq -d -w 32

11.2 Phase 2: Quick Wins (Week 2-3)

Action Expected Savings Risk
Enable -Osize for non-critical modules 10-20% of code size Low (minimal perf impact)
Enable dead code stripping 5-15% of code size Low
Switch to static linking 10-30% (dead code stripped) Medium (test thoroughly)
Compress/optimize images 30-60% of asset size Low
Remove unused assets Variable (often 10-20%) Low (use tooling to verify)
Strip symbols 5-10% of total Low (keep dSYMs)
Enable Thin LTO 5-10% of code size Low (longer builds)

11.3 Phase 3: Structural Changes (Week 4-8)

// Move large assets to on-demand resources
// Candidates: tutorial images, rarely-used features, large data files

// Replace embedded ML models with on-device download
func loadModel() async throws -> MLModel {
    let config = MLModelConfiguration()
    config.computeUnits = .all

    // Download model on first use instead of bundling
    let modelURL = try await ModelDownloader.download(
        model: .imageClassifier,
        conditions: .init(allowsCellularAccess: false)
    )

    return try MLModel(contentsOf: modelURL, configuration: config)
}

// Replace large JSON/plist configs with server-fetched configs
// Replace bundled fonts with system fonts where possible

11.4 Phase 4: Advanced Optimization (Week 8+)

// Audit generic specializations
// Each specialization duplicates the function body
// Consider using existentials for cold paths

// BEFORE: Generic (generates specialization per type)
func log<T: CustomStringConvertible>(_ value: T) {
    print("[\(Date())] \(value.description)")
}

// AFTER: Existential (single function, no duplication)
func log(_ value: any CustomStringConvertible) {
    print("[\(Date())] \(value.description)")
}

// Disable reflection metadata if not needed
// OTHER_SWIFT_FLAGS = -Xfrontend -disable-reflection-names

// Use @_specialize to control which specializations are generated
@_specialize(where T == Int)
@_specialize(where T == Double)
func compute<T: Numeric>(_ values: [T]) -> T { /* ... */ }
Binary Size Reduction Checklist
  • Measure first — use link maps, bloaty, and size reports
  • Enable -Osize for non-performance-critical modules
  • Enable dead code stripping and Thin LTO
  • Switch to static linking where possible
  • Optimize and compress all image assets
  • Remove unused code and assets (use tooling)
  • Move large resources to on-demand resources
  • Strip all symbols (keep dSYMs for crash reports)
  • Audit generic specializations on cold paths
  • Consider disabling reflection metadata
  • Set up CI tracking to prevent size regressions
