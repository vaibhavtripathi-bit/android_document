1. Extension Architecture Overview

App Extensions are separate binaries that run in their own process, distinct from the containing app. They provide functionality in specific system contexts (notification, share sheet, widget, etc.) and have strict resource limits.

┌────────────────────────────────────────────────────────────┐ │ iOS EXTENSION MODEL │ ├────────────────────────────────────────────────────────────┤ │ │ │ ┌──────────────┐ ┌──────────────┐ │ │ │ Containing │ │ Host App │ │ │ │ App Process │ │ (e.g. Mail) │ │ │ │ │ │ │ │ │ │ MyApp.app │ NO │ Launches │ │ │ │ │◄──────►│ extension │ │ │ │ (may not be │ direct │ via system │ │ │ │ running) │ IPC │ │ │ │ └──────┬───────┘ └──────┬───────┘ │ │ │ │ │ │ │ App Groups │ Extension point API │ │ │ (shared container) │ │ │ ▼ ▼ │ │ ┌──────────────────────────────────────┐ │ │ │ Extension Process │ │ │ │ (separate binary) │ │ │ │ │ │ │ │ MyApp.app/PlugIns/Share.appex │ │ │ │ │ │ │ │ Memory limit: 40-120MB │ │ │ │ No background execution │ │ │ │ Limited API access │ │ │ └──────────────────────────────────────┘ │ │ │ └────────────────────────────────────────────────────────────┘

1.1 Key Principles

  • Separate process — extensions run in their own process, not the containing app's process
  • No direct communication — extension and containing app cannot communicate directly (no shared memory, no direct IPC)
  • Shared data via App Groups — use shared containers, UserDefaults suites, and Keychain groups
  • Strict resource limits — lower memory limits, no background execution, limited API access
  • Host app launches extension — the host app (e.g., Mail, Photos) launches the extension, not the containing app

2.3 Notification Service Extension

class NotificationService: UNNotificationServiceExtension {
    var contentHandler: ((UNNotificationContent) -> Void)?
    var bestAttemptContent: UNMutableNotificationContent?
    
    override func didReceive(
        _ request: UNNotificationRequest,
        withContentHandler contentHandler: @escaping (UNNotificationContent) -> Void
    ) {
        self.contentHandler = contentHandler
        bestAttemptContent = request.content.mutableCopy() as? UNMutableNotificationContent
        
        guard let content = bestAttemptContent else {
            contentHandler(request.content)
            return
        }
        
        // ~30 seconds to modify the notification
        // Common uses: download image, decrypt content, localize
        
        if let imageURLString = content.userInfo["image_url"] as? String,
           let imageURL = URL(string: imageURLString) {
            downloadImage(from: imageURL) { attachment in
                if let attachment = attachment {
                    content.attachments = [attachment]
                }
                contentHandler(content)
            }
        } else {
            contentHandler(content)
        }
    }
    
    // Called if time expires before contentHandler is called
    override func serviceExtensionTimeWillExpire() {
        if let content = bestAttemptContent {
            contentHandler?(content)
        }
    }
}

2.4 Share Extension

class ShareViewController: SLComposeServiceViewController {
    
    override func isContentValid() -> Bool {
        return !contentText.isEmpty
    }
    
    override func didSelectPost() {
        guard let items = extensionContext?.inputItems as? [NSExtensionItem] else {
            extensionContext?.completeRequest(returningItems: [], completionHandler: nil)
            return
        }
        
        for item in items {
            for provider in item.attachments ?? [] {
                if provider.hasItemConformingToTypeIdentifier("public.url") {
                    provider.loadItem(forTypeIdentifier: "public.url") { [weak self] item, error in
                        if let url = item as? URL {
                            self?.saveToSharedContainer(url: url, text: self?.contentText ?? "")
                        }
                        self?.extensionContext?.completeRequest(
                            returningItems: [],
                            completionHandler: nil
                        )
                    }
                }
            }
        }
    }
    
    private func saveToSharedContainer(url: URL, text: String) {
        // Write to App Group shared container
        let defaults = UserDefaults(suiteName: "group.com.app.shared")
        var pending = defaults?.array(forKey: "pendingShares") as? [[String: String]] ?? []
        pending.append(["url": url.absoluteString, "text": text])
        defaults?.set(pending, forKey: "pendingShares")
    }
}

3. Memory Limits

Today Widget (Legacy) ~40 MB Very strict; avoid images
WidgetKit Widget ~30 MB Timeline entries are archived

Notification Content ~120 MB More generous for custom UI
Share Extension ~120 MB Watch out for large attachments
Keyboard Extension ~48 MB (no network) / ~64 MB (with network) Depends on RequestsOpenAccess

File Provider ~40 MB Stream files, don't load entirely
Common Pitfall: Extension Memory Crashes

Extension memory limits are hard limits — Jetsam kills the extension immediately when exceeded, with no warning. The most common cause: loading a full-resolution image into memory. A single 12MP photo decoded as a bitmap uses ~48MB. In a Notification Service Extension with a 24MB limit, this is instant death. Always downsample images.

// Downsample images to avoid memory spikes in extensions
func downsampledImage(at url: URL, maxDimension: CGFloat) -> UIImage? {
    let options: [CFString: Any] = [
        kCGImageSourceCreateThumbnailFromImageAlways: true,
        kCGImageSourceCreateThumbnailWithTransform: true,
        kCGImageSourceThumbnailMaxPixelSize: maxDimension,
        kCGImageSourceShouldCacheImmediately: true
    ]
    
    guard let source = CGImageSourceCreateWithURL(url as CFURL, nil),
          let cgImage = CGImageSourceCreateThumbnailAtIndex(source, 0, options as CFDictionary)
    else { return nil }
    
    return UIImage(cgImage: cgImage)
}

4. Process Model & IPC

4.1 No Shared Address Space

Extensions and their containing app are separate processes. They cannot:

  • Share memory (no shared singletons, no in-memory caches)
  • Use NotificationCenter across processes (Darwin notifications excepted)
  • Access each other's files directly (without App Groups)
  • Call each other's code directly

4.2 Darwin Notifications (Cross-Process)

import Foundation

// Darwin notifications work across processes (unlike NSNotificationCenter)
let notificationName = "com.app.dataChanged" as CFString

// Post from extension
let center = CFNotificationCenterGetDarwinNotifyCenter()
CFNotificationCenterPostNotification(center, CFNotificationName(notificationName), nil, nil, true)

// Observe in containing app
CFNotificationCenterAddObserver(
    center,
    nil,
    { center, observer, name, object, userInfo in
        // Data changed in extension — reload from shared container
        DispatchQueue.main.async {
            SharedDataManager.shared.reload()
        }
    },
    notificationName,
    nil,
    .deliverImmediately
)

4.3 Opening the Containing App

// Extensions cannot open the containing app directly.
// Today widgets (legacy) could use extensionContext?.open(url:)
// Other extensions must use indirect methods.

// WidgetKit: use widgetURL or Link
struct MyWidget: Widget {
    var body: some WidgetConfiguration {
        StaticConfiguration(kind: "MyWidget", provider: Provider()) { entry in
            WidgetView(entry: entry)
                .widgetURL(URL(string: "myapp://widget/\(entry.itemID)")!)
        }
    }
}

// In widget view, use Link for tappable regions
struct WidgetView: View {
    var body: some View {
        Link(destination: URL(string: "myapp://action")!) {
            Text("Open App")
        }
    }
}

5. App Groups for Data Sharing

5.1 Shared UserDefaults

// Both app and extension use the same suite name
let sharedDefaults = UserDefaults(suiteName: "group.com.app.shared")

// In the app: write data
sharedDefaults?.set(username, forKey: "currentUsername")
sharedDefaults?.set(unreadCount, forKey: "unreadCount")

// In the extension: read data
let username = sharedDefaults?.string(forKey: "currentUsername")
let unreadCount = sharedDefaults?.integer(forKey: "unreadCount")

5.2 Shared Container (File System)

// Get the shared container URL
let containerURL = FileManager.default.containerURL(
    forSecurityApplicationGroupIdentifier: "group.com.app.shared"
)

// Write a file in the app
let fileURL = containerURL?.appendingPathComponent("shared_data.json")
try? data.write(to: fileURL!, options: .atomic)

// Read the file in the extension
if let data = try? Data(contentsOf: fileURL!) {
    let model = try? JSONDecoder().decode(SharedModel.self, from: data)
}

5.3 Shared Core Data Store

// Both app and extension point to the same SQLite file
final class SharedPersistenceController {
    static let shared = SharedPersistenceController()
    let container: NSPersistentContainer
    
    init() {
        container = NSPersistentContainer(name: "Model")
        
        // Point to shared container
        let storeURL = FileManager.default
            .containerURL(forSecurityApplicationGroupIdentifier: "group.com.app.shared")!
            .appendingPathComponent("Model.sqlite")
        
        let description = NSPersistentStoreDescription(url: storeURL)
        container.persistentStoreDescriptions = [description]
        
        container.loadPersistentStores { _, error in
            if let error = error {
                fatalError("Core Data failed: \(error)")
            }
        }
        
        // Enable remote change notifications for cross-process sync
        container.viewContext.automaticallyMergesChangesFromParent = true
    }
}
Common Pitfall: Core Data Concurrent Access

When both the app and extension access the same Core Data store simultaneously, you must handle merge conflicts. Use NSMergeByPropertyObjectTrumpMergePolicy and enable NSPersistentStoreRemoteChangeNotificationPostOptionKey to detect cross-process changes. Without this, you'll get silent data corruption.

6. Extension vs Containing App

6.1 API Restrictions

Extensions have access to a subset of iOS APIs. Unavailable APIs are marked with NS_EXTENSION_UNAVAILABLE:

API Available in Extension? Reason
UIApplication.shared No Extension is not an app
HealthKit No (most) Privacy restrictions
CallKit No Telephony access
Background tasks No Extensions don't background
Camera/Microphone Limited Only some extension types
UserNotifications Yes Can schedule local notifications
CoreData Yes Via shared container
URLSession Yes Background sessions supported

6.2 Conditional Compilation

// Check if running in extension context
var isExtension: Bool {
    Bundle.main.bundlePath.hasSuffix(".appex")
}

// Or use the preprocessor macro
#if !EXTENSION
// Code only for the main app
UIApplication.shared.open(url)
#endif

7. Architect Considerations

Architect Must-Know: Sharing Code Without Bloating Extensions

The biggest architectural challenge with extensions is code sharing. You want to reuse networking, models, and business logic between the app and extensions, but every byte linked into the extension increases its memory footprint and launch time.

7.1 Framework Architecture

┌──────────────────────────────────────────────────────┐ │ CODE SHARING STRATEGY │ ├──────────────────────────────────────────────────────┤ │ │ │ ┌─────────────┐ ┌─────────────┐ │ │ │ Main App │ │ Extension │ │ │ │ │ │ (.appex) │ │ │ └──────┬──────┘ └──────┬──────┘ │ │ │ │ │ │ ▼ ▼ │ │ ┌─────────────────────────────────┐ │ │ │ SharedKit.framework │ ◀── Shared │ │ │ (embedded in both) │ framework │ │ │ │ │ │ │ • Models (Codable structs) │ │ │ │ • Networking (URLSession) │ │ │ │ • Persistence (Core Data) │ │ │ │ • Business logic │ │ │ └─────────────────────────────────┘ │ │ │ │ ┌─────────────────────────────────┐ │ │ │ AppOnlyKit.framework │ ◀── App only │ │ │ (NOT in extension) │ │ │ │ │ │ │ │ • Heavy UI components │ │ │ │ • Analytics SDKs │ │ │ │ • Feature flags │ │ │ │ • UIApplication-dependent code │ │ │ └─────────────────────────────────┘ │ │ │ └──────────────────────────────────────────────────────┘

7.2 Lean Extension Pattern

// SharedKit: minimal framework for extension use
// Keep dependencies minimal — no heavy SDKs

// Protocol-based abstraction for extension-safe code
protocol DataFetching {
    func fetchItems() async throws -> [Item]
}

// Lightweight implementation for extensions
struct ExtensionDataFetcher: DataFetching {
    func fetchItems() async throws -> [Item] {
        // Read from shared container (no network)
        let url = SharedContainer.url.appendingPathComponent("items.json")
        let data = try Data(contentsOf: url)
        return try JSONDecoder().decode([Item].self, from: data)
    }
}

// Full implementation for the app
struct AppDataFetcher: DataFetching {
    let networkClient: NetworkClient
    let cache: CacheManager
    
    func fetchItems() async throws -> [Item] {
        // Full network + cache + sync logic
        if let cached = try? cache.get(key: "items") {
            return cached
        }
        let items = try await networkClient.fetch("/items")
        try? cache.set(key: "items", value: items)
        // Also write to shared container for extension
        try? writeToSharedContainer(items)
        return items
    }
}

7.3 WidgetKit Timeline Architecture

// WidgetKit uses a timeline model — not live data
struct Provider: TimelineProvider {
    func placeholder(in context: Context) -> SimpleEntry {
        SimpleEntry(date: Date(), items: Item.placeholders)
    }
    
    func getSnapshot(in context: Context,
                     completion: @escaping (SimpleEntry) -> Void) {
        let entry = SimpleEntry(date: Date(), items: loadFromSharedContainer())
        completion(entry)
    }
    
    func getTimeline(in context: Context,
                     completion: @escaping (Timeline<SimpleEntry>) -> Void) {
        let items = loadFromSharedContainer()
        let entry = SimpleEntry(date: Date(), items: items)
        
        // Refresh every 30 minutes
        let nextUpdate = Calendar.current.date(byAdding: .minute, value: 30, to: Date())!
        let timeline = Timeline(entries: [entry], policy: .after(nextUpdate))
        completion(timeline)
    }
}

Best Practice: Extension Architecture Checklist
  • Minimize shared framework size — only include what the extension actually needs
  • Use static linking for shared frameworks to avoid dylib loading overhead in extensions
  • Prefetch data in the main app — write to shared container so extensions read locally
  • Use Darwin notifications for cross-process signaling (not NSNotificationCenter)
  • Handle Core Data merge conflicts when both app and extension write simultaneously
  • Test memory limits — use Instruments to profile extension memory usage
  • Audit dependencies — third-party SDKs often exceed extension memory budgets
  • Use #if !EXTENSION to exclude app-only code from extension targets
