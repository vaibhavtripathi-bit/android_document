AppDelegate & Boot Sequence

Q1. Why does AppDelegate inherit from UIResponder instead of NSObject? Senior

AppDelegate inherits from UIResponder so it participates in the responder chain. It sits at the very end of the chain: UIView → UIViewController → UIWindow → UIWindowScene → UIApplication → AppDelegate. This means any event (key commands, remote control events, motion events, menu actions) that isn't handled by any view or controller bubbles up to the app delegate. This is how the app delegate can handle hardware keyboard shortcuts, shake gestures, and remote audio control events. Without UIResponder inheritance, the app delegate would be invisible to the event delivery system.

Q2. What is the difference between +load and +initialize, and why does +load impact launch time? Senior

+load is called by the ObjC runtime during dyld initialization, before main(). It runs for every class and category that implements it, in a defined order (superclass before subclass, class before category). It runs with the runtime lock held, so it's serial and blocks the entire launch process. Every +load method directly adds to pre-main time.

+initialize is called lazily — the first time a message is sent to a class. It's thread-safe (uses a dispatch_once-like mechanism) and inherited (a subclass may trigger the superclass's +initialize). Because it's lazy, it has zero impact on launch time.

The key insight: +load is the only safe place for method swizzling (because it runs before any code uses the class), but every +load adds to pre-main time. At scale, auditing and eliminating unnecessary +load methods is a high-ROI optimization.

Q3. Walk through the complete boot sequence from icon tap to first visible frame. L6

1. Kernel: XNU creates the process (fork+exec), maps the Mach-O binary, starts dyld.

2. dyld4 (pre-main): Loads the shared cache (pre-linked system frameworks), maps app dylibs, performs rebasing (ASLR slide adjustment), binds external symbols, registers ObjC classes, runs +load methods and C++ static constructors, then jumps to main().

3. UIApplicationMain: Creates the UIApplication singleton, instantiates the AppDelegate, sets it as the delegate, starts the main RunLoop.

4. willFinishLaunchingWithOptions: First callback. App state is .inactive. Set up essential infrastructure (Core Data, DI container).

5. State restoration: If using legacy NSCoder restoration, it happens between will and did finish.

6. didFinishLaunchingWithOptions: Second callback. Analytics, feature flags, push registration.

7. Scene connection (iOS 13+): scene(_:willConnectTo:options:) — create UIWindow, set rootViewController, makeKeyAndVisible().

8. Layout pass: Auto Layout constraint solving, layoutSubviews cascade.

9. First render: CATransaction commit → render server (backboardd) → GPU compositing → display. Launch storyboard is dismissed, first app frame appears.

10. Active: sceneDidBecomeActive (or applicationDidBecomeActive pre-iOS 13).

Target: cold launch under 400ms. Watchdog kills at 20s.

Q4. What does UIApplicationMain actually do, and why does it never return? Senior

UIApplicationMain creates the UIApplication singleton (or a custom subclass if specified), creates an instance of the delegate class, sets the delegate, and then starts the main RunLoop via CFRunLoopRun(). The RunLoop is an infinite loop that processes events (touches, timers, display link callbacks, etc.) and never exits under normal operation. The function signature returns Int for C compatibility, but it never actually returns — the process is terminated by the system, not by returning from main.

Q5. What is the difference between willFinishLaunchingWithOptions and didFinishLaunchingWithOptions? Senior

willFinishLaunchingWithOptions is called first. State restoration (NSCoder-based) happens between will and did. Use willFinish for setup that state restoration depends on (Core Data stack, DI container). Use didFinish for everything else (analytics, push registration, feature flags, appearance customization). Both are called while the app is in the .inactive state. Returning false from willFinish tells the system you can't handle the launch URL, but does NOT prevent the app from launching.

Q6. How does @main differ from @UIApplicationMain, and how does it work with SwiftUI? Senior

@UIApplicationMain was UIKit-specific — it generated a main() that called UIApplicationMain(). @main (Swift 5.3+) is generalized — it works with any type that has a static main() method. For UIKit, the UIApplicationDelegate protocol extension provides a default main() that calls UIApplicationMain. For SwiftUI, the App protocol provides a different main() that bootstraps the SwiftUI runtime, creates the app struct, evaluates the body to get Scene declarations, maps them to UIScene configurations, and starts the RunLoop. This decouples the entry point from UIKit entirely.

Q7. What is the dyld shared cache and why does it matter for launch performance? L6

Apple pre-links all system frameworks (UIKit, Foundation, CoreGraphics, etc.) into a single shared cache file. This cache is mapped into every process at the same virtual address, enabling page sharing across all running apps. Without it, each app would need to individually load, rebase, and bind hundreds of system dylibs — adding seconds to launch time. The shared cache is rebuilt during OS updates. Your app's own dylibs are NOT in the shared cache (unless you use mergeable libraries in Xcode 15+). This is why reducing the number of embedded frameworks improves launch time — each additional dylib requires separate loading, rebasing, and binding.

Q8. How would you architect a phased launch system to minimize time-to-first-frame? Architect

Divide launch work into three phases: Critical (before first frame: DI container, Core Data stack, auth token check), Important (first idle after frame: analytics SDK, feature flags, prefetch), and Deferred (after 2+ seconds: non-essential SDKs, prewarming caches). Use a LaunchCoordinator that registers tasks by phase. Critical tasks run synchronously in didFinishLaunching. Important tasks dispatch to the next main RunLoop iteration. Deferred tasks use asyncAfter with a delay. Instrument each phase with os_signpost. Track p50/p95 launch times via MetricKit's MXAppLaunchMetric. The key insight: anything that doesn't affect the first visible frame should be deferred.

Q9. What are dyld closures and how do they improve subsequent launches? L6

dyld closures are pre-computed data structures that cache the results of the first launch's loading process — which dylibs to load, where to find symbols, what rebasing/binding to perform. After the first launch, dyld4 writes these closures to disk. On subsequent launches, dyld reads the closure instead of re-parsing Mach-O headers and re-resolving symbols, significantly reducing pre-main time. Closures are invalidated when the app binary changes (update) or when a dylib is updated. This is why the first launch after an install or update is slower than subsequent launches.

Q10. How do Swift global lets differ from ObjC/C++ globals in terms of initialization timing? Senior

Swift global let constants are lazily initialized using a dispatch_once-like mechanism. They're only initialized on first access, so they don't add to pre-main time. ObjC/C++ static globals and constructors (__attribute__((constructor))) run during dyld initialization, before main(), and directly add to pre-main time. This means Swift's lazy initialization is inherently safer for launch performance, but you need to be careful when bridging to ObjC or using C++ dependencies that may have static initializers.

Q11. What happens if didFinishLaunchingWithOptions takes more than 20 seconds? Senior

The system watchdog (watchdogd) will kill the app with a 0x8badf00d ("ate bad food") exception code. This appears as a watchdog timeout in crash reports. The watchdog monitors the time from process start to the first frame being rendered. The exact timeout varies by device and iOS version but is approximately 20 seconds. This is a hard kill — no callback, no chance to save state. The fix is to defer all non-essential work past the first frame.

Q12. How would you measure and optimize pre-main time in a large app with 50+ dylibs? Architect

Measure: Set DYLD_PRINT_STATISTICS=1 in the Xcode scheme to get a breakdown (dylib loading, rebase/binding, ObjC setup, initializer time). Use Instruments App Launch template for detailed profiling. In production, use MetricKit's MXAppLaunchMetric.

Optimize: (1) Reduce dylib count — merge frameworks using static linking or Xcode 15 mergeable libraries. Each dylib adds ~1-5ms. (2) Eliminate +load methods — move to +initialize or lazy initialization. (3) Reduce ObjC class count — each class registration adds to ObjC setup time. (4) Use Swift over ObjC where possible — Swift doesn't have +load. (5) Audit C++ static constructors in dependencies. (6) Consider using dyld closures by ensuring binary stability between launches.

Scene Lifecycle & Multi-Window

Q13. What is the difference between UIScene, UISceneSession, and UIWindowScene? Senior

UISceneSession is the persistent identity — it survives process death and represents a "slot" for a scene. It has a persistentIdentifier that's stable across launches. UIScene is the runtime instance attached to a session — it's created when the scene connects and destroyed when it disconnects. UIWindowScene is the concrete subclass of UIScene that hosts UIWindows. Think of it as: the session is the parking spot, the scene is the car currently in it, and the window scene is a specific type of car that can display UI.

Q14. What is the critical difference between scene disconnection and scene destruction? L6

Disconnection (sceneDidDisconnect): The scene object is released but the session persists. The system does this to reclaim memory for background scenes. The scene may reconnect later to the same session. You should release heavy resources but NOT delete user data.

Destruction (application(_:didDiscardSceneSessions:)): The session itself is permanently destroyed. This happens when the user swipes away in the app switcher. Called on the AppDelegate, not SceneDelegate. NOW you should delete all data associated with this scene. Critical: if the app wasn't running when the user swiped away, this callback is deferred to the next launch.

Q15. Why was UIScreen.main deprecated, and what replaced it? Senior

With multi-window support, there's no single "main" screen. Different scenes can be on different screens (iPad + external display). Use windowScene.screen instead. Similarly, UIApplication.shared.keyWindow is deprecated — use windowScene.keyWindow or iterate UIApplication.shared.connectedScenes to find the relevant scene.

Q16. How do you programmatically create a new window on iPad? Senior

Call UIApplication.shared.requestSceneSessionActivation(nil, userActivity: activity, options: options, errorHandler: handler). Pass nil for the first parameter to create a new session. The userActivity carries the content to display. The system will call your AppDelegate's configurationForConnecting to get the scene configuration, then call the SceneDelegate's scene(_:willConnectTo:options:) where you read the activity and set up the UI. To close a window, use requestSceneSessionDestruction.

Q17. How do scene activation conditions work, and when would you use them? L6

Activation conditions control which scene handles incoming activities. Each scene has prefersToActivateForTargetContentIdentifierPredicate (this scene is the best match) and canActivateForTargetContentIdentifierPredicate (this scene can handle it as a fallback). When the system receives a notification or activity with a targetContentIdentifier, it evaluates these predicates to route to the correct scene. Use case: a messaging app with multiple chat windows — each scene sets its predicate to match its chat ID, so notifications route to the correct window.

Q18. What lifecycle callbacks does a scene NOT receive? Senior

There is no callback when a scene moves from Background to Suspended — the system simply stops executing code. There is no callback when Jetsam kills a suspended app. applicationWillTerminate is NOT reliably called for scene-based apps. The only reliable save point is sceneDidEnterBackground — you must save all critical state there.

Q19. How did the scene-based lifecycle break singleton patterns? Architect

Pre-iOS 13, singletons like AppState.shared mapped naturally to the single-window model. With multi-window, you have multiple scenes with independent lifecycles showing potentially different content. What broke: UIScreen.main (which screen?), keyWindow (which window?), singleton state managers (which scene's state?), deep link handlers (which navigation stack?), alert presenters (which scene to present on?). The fix: separate process-scoped state (auth, caches — singleton OK) from scene-scoped state (navigation, UI — one per scene). Use a SceneStateRegistry keyed by session.persistentIdentifier.

Q20. How would you design a data layer for a multi-scene iPad app? Architect

Three layers: (1) Process-scoped shared state — singleton, shared across all scenes. User session, auth tokens, network cache, Core Data persistent container. Changes propagate to all scenes via Combine/Observation. (2) Scene-scoped state — one instance per scene, keyed by persistentIdentifier. Navigation path, selected tab, scroll positions, editing context. (3) Coordination layer — handles cross-scene events (user logged out → all scenes reset, document deleted → scenes showing it navigate away). For Core Data, use a shared viewContext for reading but per-scene child contexts for editing to avoid conflicts.

Q21. What is the difference between static and runtime scene configuration? Senior

Static configuration is declared in Info.plist under UIApplicationSceneManifest. It specifies the delegate class, storyboard, and configuration name. Runtime configuration is provided by implementing application(_:configurationForConnecting:options:) in the AppDelegate. Runtime configuration is more flexible — you can choose different delegate classes based on the connection options (e.g., a compose scene vs a browse scene). If you provide runtime configuration, it overrides the Info.plist configuration for that session.

Q22. How does the responder chain change with scenes? L6

Pre-iOS 13: UIView → UIViewController → UIWindow → UIApplication → AppDelegate. iOS 13+: UIView → UIViewController → UIWindow → UIWindowScene → UIApplication → AppDelegate. The UIWindowScene is inserted into the chain, meaning scene delegates can handle events that views and controllers don't. This is important for multi-window apps where different scenes might handle the same key command differently.

Q23. What happens if UIApplicationSupportsMultipleScenes is false? Senior

The system will only ever create one scene session. The app behaves like a single-window app. However, you still use the scene-based lifecycle (SceneDelegate). The old AppDelegate lifecycle methods (applicationDidBecomeActive, etc.) are ignored if a SceneDelegate is present. Setting this to false is the default for new projects and is appropriate for iPhone-only apps or apps that don't need multi-window.

Q24. How do you handle a push notification that should open in a specific scene? L6

Set targetContentIdentifier on the UNNotificationContent when creating the push payload. In each scene's scene(_:willConnectTo:options:), set scene.activationConditions.prefersToActivateForTargetContentIdentifierPredicate to match the scene's content (e.g., a chat ID). When the user taps the notification, the system evaluates the predicates and routes to the matching scene. If no scene matches, a new scene is created. This is the only reliable way to route notifications to specific windows in a multi-scene app.

Process Lifecycle & Background Execution

Q25. What is the critical mistake with beginBackgroundTask, and what happens if you make it? Senior

The critical mistake is calling beginBackgroundTask without ever calling endBackgroundTask. When the background time expires, the system calls your expiration handler. If you don't call endBackgroundTask in the expiration handler either, the system kills your app immediately with a 0x8badf00d watchdog exception. This appears as a crash in your logs. The fix: always call endBackgroundTask in both the completion path and the expiration handler. Use a guard pattern: store the task ID, check for .invalid before ending, and set to .invalid after ending.

Q26. How did the background time budget change in iOS 13, and why? Senior

Pre-iOS 13, beginBackgroundTask gave approximately 180 seconds (3 minutes). iOS 13 reduced this to approximately 30 seconds. Apple made this change to improve battery life and system responsiveness. Many apps were abusing the 3-minute window for long-running background work that should have used proper background modes or BGTaskScheduler. The reduction was silent — no API change, just a behavioral change — which broke many apps that relied on the longer window.

Q27. Explain the difference between BGAppRefreshTask and BGProcessingTask. Senior

BGAppRefreshTask: ~30 seconds of execution, system-scheduled based on usage patterns, suitable for fetching new content or syncing small amounts of data. No special constraints required. BGProcessingTask: several minutes of execution, can require external power and/or network connectivity, suitable for heavy work like ML model training, database maintenance, or large syncs. Both are one-shot (must re-schedule after each execution) and not guaranteed to run at any specific time. Register handlers in didFinishLaunching because the system may launch your app directly into a background task.

Q28. What is Jetsam, and how does it differ from a regular crash? L6

Jetsam is iOS's memory management daemon (analogous to Linux's OOM killer). It monitors system memory pressure and kills processes to free memory. Key differences from crashes: Jetsam sends SIGKILL (unblockable — no signal handler), provides no callback (no didReceiveMemoryWarning guarantee, no applicationWillTerminate), generates a Jetsam event report (different format from crash reports), and kills based on priority bands (background apps first, foreground last). Prevention is about reducing memory footprint, not handling a callback. Background apps are limited to ~50-80MB; exceeding this triggers Jetsam even without system-wide pressure.

Q29. Why is didReceiveMemoryWarning unreliable? L6

Three reasons: (1) It runs on the main thread — if the main thread is blocked (large layout pass, synchronous network call), the warning can't be delivered before Jetsam acts. (2) Under extreme memory pressure, Jetsam may kill the app without sending any warning. (3) For suspended apps, no code runs at all — the warning can't be delivered to a process that isn't executing. Use DispatchSource.makeMemoryPressureSource for more reliable monitoring, and os_proc_available_memory() to check available memory proactively.

Q30. What survives process suspension vs termination? Senior

Suspension (in memory, not executing): All memory intact, file handles open, view hierarchy preserved. Timers paused, GCD work paused, RunLoop not running. Network connections may timeout. When resumed, everything continues from where it stopped.

Termination (process killed): All memory freed, file handles closed, view hierarchy destroyed. What survives: files on disk, UserDefaults (if synchronized), Keychain items, Core Data persistent store, UISceneSession (persisted by system), NSUserActivity (if saved to session). Design your app assuming termination can happen at any time with no callback.

Q31. When is applicationWillTerminate actually called? Senior

Only when the app is in the foreground and the system needs to terminate it (rare), or for legacy apps that haven't adopted the scene lifecycle. It is NOT called when: Jetsam kills a suspended app (most common termination), the user force-quits from the app switcher, or the device reboots. For scene-based apps, save state in sceneDidEnterBackground instead — it's the last reliable callback.

Q32. How do background URL sessions survive process death? L6

Background URLSession (created with URLSessionConfiguration.background(withIdentifier:)) hands the transfer to the system daemon (nsurlsessiond). The daemon manages the transfer independently of your app process. When the transfer completes, the system relaunches your app in the background and calls application(_:handleEventsForBackgroundURLSession:completionHandler:). You recreate the URLSession with the same identifier, and the delegate callbacks fire with the results. The key: the identifier must be unique and consistent across launches.

Q33. What are Jetsam priority bands, and how do they affect your app? L6

Jetsam assigns priority bands (0-40+) to processes. Band 0-10: idle/suspended apps (killed first, in LRU order). Band 10-20: low priority (background tasks, extensions). Band 20-30: default (foreground inactive). Band 30-40: high priority (foreground active app, SpringBoard). Band 40+: critical (kernel, core daemons, killed last). Your foreground app is in the highest non-system band, so it's the last user app to be killed. But background apps are in the lowest bands and are killed aggressively. This is why background memory limits (~50-80MB) are much lower than foreground limits (~1-2GB).

Q34. How would you design a reliable background sync system? Architect

Core principle: every operation is a state machine persisted to disk, resumable after process death. (1) Model each sync operation as a Codable struct with states (pending, uploading, confirming, completed, failed). (2) Persist the operation queue to disk (SQLite or JSON file). (3) On app launch, resume incomplete operations. (4) Use background URLSession for large transfers (survives process death). (5) Use BGAppRefreshTask for periodic sync checks. (6) Use BGProcessingTask for heavy reconciliation (while charging). (7) Make all operations idempotent — the server must handle duplicate requests gracefully. (8) Implement exponential backoff for failures. (9) Use os_signpost to instrument sync performance in production via MetricKit.

Q35. What is the 0x8badf00d exception code? Senior

"Ate bad food" — a watchdog timeout. The system killed your app because it didn't complete a required operation in time. Common causes: (1) Launch took more than 20 seconds. (2) beginBackgroundTask expiration handler didn't call endBackgroundTask. (3) Synchronous work on the main thread during a state transition. (4) Background task exceeded its time budget. This is a SIGKILL — no crash handler can catch it. Fix by ensuring all time-critical operations complete within their budgets.

Q36. How does silent push notification rate limiting work? L6

Silent pushes (content-available: 1) wake your app for ~30 seconds. The system applies rate limiting based on: (1) Energy budget — if your app uses too much CPU/energy during wakeups, the system throttles delivery. (2) Frequency — too many silent pushes in a short period triggers throttling. (3) User behavior — if the user hasn't opened your app recently, the system may stop delivering silent pushes. (4) Force-quit — if the user force-quit the app, silent pushes are not delivered until the user manually opens the app again. There's no API to check your current rate limit status. Design your system to work even if silent pushes are delayed or dropped.

Q37. How do you debug BGTaskScheduler tasks? Senior

Use LLDB commands while the app is running: e -l objc -- (void)[[BGTaskScheduler sharedScheduler] _simulateLaunchForTaskWithIdentifier:@"com.app.refresh"] to trigger a task, and _simulateExpirationForTaskWithIdentifier: to test expiration handling. You can also set breakpoints in your task handlers and use the Xcode debugger. For production debugging, log task execution to a shared file or analytics. Note: BGTaskScheduler tasks are not guaranteed to run at any specific time — the system considers battery, network, and usage patterns.

State Restoration

Q38. Compare the three generations of state restoration APIs. L6

Gen 1 (iOS 6-12): NSCoder-based. System walks the VC hierarchy by restorationIdentifier, calls encodeRestorableState/decodeRestorableState. Complex (requires restoration classes, identifier chains), tightly coupled to VC hierarchy, doesn't support multi-scene.

Gen 2 (iOS 13+): NSUserActivity-based. SceneDelegate returns an NSUserActivity from stateRestorationActivity(for:). Simpler, decoupled from VC hierarchy, supports multi-scene, doubles as Handoff/Spotlight/Siri activity.

Gen 3 (iOS 14+): @SceneStorage. SwiftUI property wrapper that automatically persists per-scene values. Simplest API, limited to basic types, SwiftUI only, not suitable for complex state.

Q39. Why did NSCoder-based restoration break with scene-based apps? L6

NSCoder restoration was designed for a single-window app where the AppDelegate owned the window. It ran between willFinish and didFinish, restoring the entire VC hierarchy. With scenes: (1) Window creation moved to SceneDelegate, but restoration timing didn't change. (2) Each scene needs independent restoration, but the old system was app-wide. (3) The old save trigger (applicationDidEnterBackground) is per-app, not per-scene. (4) The NSCoder methods simply aren't called for scene-based apps. Apple replaced it with the NSUserActivity approach where each scene saves/restores its own state via the session.

Q40. What are the limitations of @SceneStorage? Senior

Limited type support (Bool, Int, Double, String, URL, Data, RawRepresentable with Int/String raw value). Not shared between scenes. Data may be purged by the system at any time. No encryption — don't store sensitive data. Size should be kept small (kilobytes). Not a general persistence mechanism — only for UI state. For complex types, you must conform to RawRepresentable and encode/decode manually (typically via JSON).

Q41. How would you build a state restoration system that works across app updates? Architect

Key principles: (1) Restore intent, not UI structure. Save "user was viewing item 123" not "user was on DetailViewController." (2) Version your state format. Include a version number and write migration code for each version bump. (3) Use route descriptors, not class names. VCs may be renamed or restructured. (4) Graceful degradation. If restored state references deleted content, fall back to a reasonable default. (5) Test with old state fixtures. Keep serialized state from previous versions as test data. (6) Use Codable enums for user intent. This gives you compile-time safety and easy migration. (7) Separate restoration from navigation. The restoration system produces an intent; a separate navigator executes it.

Q42. What is the restorationIdentifier chain and why does every VC in the chain need one? Senior

In NSCoder-based restoration, the system identifies each VC by its path of restoration identifiers: e.g., ["TabBar", "Nav", "Detail"]. If any VC in the chain is missing an identifier, all its children are excluded from restoration. The identifier only needs to be unique among siblings. The system uses this path to reconstruct the VC hierarchy during restoration. This was fragile — adding a container VC or changing the hierarchy broke restoration for all descendants.

Q43. How does NSUserActivity serve double duty for restoration and Handoff? Senior

The same NSUserActivity object can power state restoration, Handoff, Spotlight indexing, and Siri Shortcuts. Set isEligibleForHandoff = true for Handoff, isEligibleForSearch = true for Spotlight, and isEligibleForPrediction = true for Siri. The activity's userInfo carries the state data. For restoration, it's saved to the scene session. For Handoff, it's advertised to nearby devices. This is a multiplier on engineering effort — one activity definition serves four purposes.

Q44. What is the security concern with the legacy shouldSaveApplicationState API? L6

The original shouldSaveApplicationState/shouldRestoreApplicationState used insecure NSCoder decoding, vulnerable to object substitution attacks. If the restoration archive on disk was tampered with, an attacker could inject arbitrary objects during decoding. Apple deprecated these in iOS 13.2 and replaced them with shouldSaveSecureApplicationState/shouldRestoreSecureApplicationState which use NSSecureCoding — requiring you to specify the expected class when decoding.

Q45. How do you test state restoration? Senior

(1) In Xcode: background the app, then stop it from Xcode (simulates process death). Relaunch to test restoration. (2) Unit test encode/decode round-trips for your state models. (3) Keep old state snapshots as test fixtures to verify migration. (4) Test graceful degradation: what happens when restored state references deleted content? (5) Use XCUIApplication.terminate() in UI tests. (6) Test that incoming URLs/activities take priority over restoration (they should).

Q46. When does stateRestorationActivity(for:) get called? Senior

The system calls it when it wants to save the scene's state — typically when the scene enters the background or when the system is about to disconnect the scene. It may be called multiple times. The returned NSUserActivity is stored on the scene session's stateRestorationActivity property and persists across process death. On the next connection, you read it from session.stateRestorationActivity in scene(_:willConnectTo:options:).

Deep Links & Universal Links

Q47. Why are Universal Links more secure than custom URL schemes? Senior

Custom URL schemes have no ownership verification — any app can register the same scheme, and the system's behavior when two apps claim the same scheme is undefined. A malicious app could intercept OAuth callbacks. Universal Links use domain verification via the AASA file: only the app whose team ID is listed in the AASA file served over HTTPS from that domain can handle those links. The AASA file proves domain ownership. Additionally, Universal Links don't show a confirmation dialog — they open directly in the app, providing a better UX.

Q48. How does Apple's CDN affect AASA file updates? L6

Since iOS 14, devices never contact your server directly for the AASA file. Apple's CDN (app-site-association.cdn-apple.com) fetches and caches it. Updates propagate with a delay of hours to days. You cannot force a refresh. IP-based access control doesn't work (Apple's bot fetches it). For development, use the ?mode=developer associated domains entitlement to bypass the CDN and fetch directly from your server. The AASA file must be at /.well-known/apple-app-site-association, served over HTTPS with no redirects, Content-Type application/json, and under 128KB.

Q49. How do deep links arrive differently in scene-based vs pre-scene apps? Senior

Pre-scene: URL schemes arrive via application(_:open:options:). Universal Links arrive via application(_:continue:restorationHandler:). Scene-based: URL schemes arrive via scene(_:openURLContexts:) (or in connectionOptions.urlContexts during connection). Universal Links arrive via scene(_:continue:) (or in connectionOptions.userActivities). The key difference: scene-based delivery includes UIOpenURLContext which wraps the URL with metadata (source app, open-in-place flag).

Q50. How would you architect a deep link router for an app with 100+ screens? Architect

Registration-based pattern: (1) URL Parser normalizes URLs from different sources (scheme, universal link, notification). (2) Middleware pipeline runs auth checks, feature flags, analytics, A/B test routing. (3) Route matcher uses pattern matching ("/products/:id") against registered routes. (4) Route handlers create VCs and extract parameters. (5) Navigator handles push/present/reset. Each feature module registers its own routes via a protocol. Auth-guarded routes queue the URL and present login; after auth, the queued URL is replayed. Use middleware for cross-cutting concerns (analytics, feature flags). Test the route matcher with unit tests covering all URL patterns.

Q51. What are deferred deep links and how do you implement them? L6

Deferred deep links handle the case where a user clicks a link but doesn't have the app installed. After installing from the App Store, the app should navigate to the linked content. Implementation options: (1) App Clip → Full App handoff via shared Keychain (most reliable). (2) Server-side fingerprinting matching device characteristics (IP, user agent) — privacy concerns. (3) Clipboard-based (write URL to pasteboard) — broken by iOS 16 paste permission. (4) Third-party SDKs (Branch, Firebase Dynamic Links) that combine multiple techniques. The App Clip approach is best: the clip stores the deep link context in a shared Keychain access group, and the full app reads it on first launch.

Q52. What is the AASA "components" format and how does it differ from the legacy "paths" format? Senior

The legacy format used a "paths" array with simple glob patterns: "paths": ["/products/*", "NOT /admin/*"]. The newer "components" format (iOS 13+) uses objects with richer matching: path patterns, query parameters, fragment matching, and exclude flags. Components support "/" for path, "?" for query, "#" for fragment, and "exclude": true for exclusion. The components format is more expressive and is the recommended approach for new apps.

Q53. How do you handle a deep link that requires authentication? Architect

Use a middleware pattern in your router: (1) The auth middleware checks if the user is authenticated. (2) If not, it saves the pending URL to a queue (UserDefaults or in-memory). (3) It presents the login flow. (4) After successful authentication, the router replays the queued URL. (5) If auth fails or is cancelled, the queued URL is discarded. Mark routes as requiresAuth during registration so the middleware knows which routes to guard. This pattern also works for onboarding flows — queue the deep link until onboarding is complete.

Q54. What are App Clips and what are their lifecycle restrictions? Senior

App Clips are lightweight app experiences (max 10MB initial, 50MB with ODR) invoked via NFC, QR, Safari, Maps, or Messages. Restrictions: no background execution (except audio), no tracking without ATT, limited Keychain access, data deleted after inactivity period, no access to CallKit/CareKit/CloudKit/HealthKit. They use the same scene lifecycle as regular apps. The invocation URL arrives as an NSUserActivity. App Clips can share data with the full app via shared Keychain (App Groups). They're ideal for deferred deep linking — the clip captures context that the full app reads after installation.

Q55. How do you test Universal Links during development? Senior

(1) Use ?mode=developer in the associated domains entitlement to bypass Apple's CDN. (2) Test with xcrun simctl openurl booted "https://example.com/path". (3) Validate AASA with Apple's Search API Validation Tool. (4) Check Apple's CDN directly: curl https://app-site-association.cdn-apple.com/a/v1/example.com. (5) On device, use the Associated Domains diagnostic profile from Apple. (6) Write unit tests for your route matcher with all URL patterns. (7) Test both cold launch (app not running) and warm launch (app backgrounded) — they use different code paths (connectionOptions vs openURLContexts).

Extensions

Q56. Why do extensions run in a separate process from the containing app? Senior

Security and stability. Extensions are loaded by host apps (Mail, Photos, etc.) — if they shared the containing app's process, a crash in the extension would crash the host app. Separate processes provide: (1) Memory isolation — extension crash doesn't affect host or containing app. (2) Security sandboxing — extension can't access host app's data or vice versa. (3) Resource limits — the system can enforce strict memory limits per extension. (4) Independent lifecycle — extension can be terminated without affecting the containing app. The trade-off: no shared memory, requiring IPC via App Groups.

Q57. What are the memory limits for different extension types, and why do they matter? L6

Notification Service: ~24MB. Today Widget: ~40MB. WidgetKit: ~30MB. Keyboard: ~48-64MB. Share/Notification Content: ~120MB. These are hard limits — Jetsam kills immediately on violation, no warning. The most common cause of extension crashes: loading a full-resolution image. A 12MP photo decoded as RGBA bitmap = 12M × 4 bytes = ~48MB — exceeding most extension limits. Always downsample images using CGImageSource thumbnail APIs. Audit third-party SDK memory usage — many SDKs are designed for apps, not extensions, and may allocate too much memory.

Q58. How do you share data between an app and its extensions? Senior

Via App Groups, which provide: (1) Shared UserDefaults — UserDefaults(suiteName: "group.com.app.shared"). (2) Shared file container — FileManager.containerURL(forSecurityApplicationGroupIdentifier:). (3) Shared Core Data store — point NSPersistentStoreDescription to a URL in the shared container. (4) Shared Keychain — use kSecAttrAccessGroup with the shared group identifier. For real-time signaling (extension tells app data changed), use Darwin notifications (CFNotificationCenterGetDarwinNotifyCenter) — these work across processes unlike NSNotificationCenter.

Q59. What happens when both the app and extension write to the same Core Data store simultaneously? L6

Without proper handling, you get merge conflicts and potential data corruption. Solutions: (1) Set NSMergeByPropertyObjectTrumpMergePolicy on both contexts. (2) Enable NSPersistentStoreRemoteChangeNotificationPostOptionKey on the store description to detect cross-process changes. (3) Set automaticallyMergesChangesFromParent = true on the viewContext. (4) Use NSPersistentHistoryTracking to process changes made by the other process. (5) Consider using WAL (Write-Ahead Logging) mode for SQLite to allow concurrent reads and writes. The key insight: Core Data's in-memory cache doesn't know about changes made by another process — you must explicitly merge them.

Q60. How would you architect code sharing between app and extensions without bloating the extension? Architect

Create a layered framework architecture: (1) SharedKit — minimal framework embedded in both app and extensions. Contains only models, networking, persistence, and core business logic. No heavy SDKs. (2) AppOnlyKit — framework embedded only in the app. Contains heavy UI, analytics SDKs, feature flags, UIApplication-dependent code. (3) Use static linking for SharedKit to avoid dylib loading overhead in extensions. (4) Use protocol-based abstractions so extensions can use lightweight implementations (read from shared container) while the app uses full implementations (network + cache). (5) Use #if !EXTENSION to exclude app-only code. (6) Audit SharedKit's dependency tree — a single heavy transitive dependency can blow the extension's memory budget.

Q61. How does the Notification Service Extension lifecycle work? Senior

When a push notification with mutable-content: 1 arrives, the system launches the Notification Service Extension and calls didReceive(_:withContentHandler:). You have ~30 seconds to modify the notification content (download images, decrypt payload, localize text) and call the content handler. If time expires, the system calls serviceExtensionTimeWillExpire() — you must call the content handler with your best attempt. If you still don't call it, the system displays the original unmodified notification. The extension process is terminated after the handler is called. Memory limit is ~24MB — be very careful with image downloads.

Q62. Why can't extensions use UIApplication.shared? Senior

Extensions are not apps — they don't have a UIApplication instance. The UIApplication.shared accessor is marked NS_EXTENSION_UNAVAILABLE. This means extensions can't: open URLs, register for push notifications, check application state, use beginBackgroundTask, or access the keyWindow. If your shared code needs to call UIApplication APIs, use conditional compilation (#if !EXTENSION) or protocol abstractions that provide different implementations for app vs extension contexts.

Q63. How does WidgetKit's timeline model work? Senior

WidgetKit doesn't show live data — it displays pre-rendered timeline entries. Your TimelineProvider returns an array of entries with dates, and the system displays the appropriate entry at each time. Refresh policies: .atEnd (refresh when last entry expires), .after(date) (refresh at a specific time), .never (manual refresh only). The main app triggers refresh via WidgetCenter.shared.reloadTimelines(ofKind:). The widget reads data from the shared container — it should never make network requests in the timeline provider (the system may not grant network access). Budget: the system limits how often your timeline provider runs.

Q64. How do Darwin notifications differ from NSNotificationCenter notifications? L6

NSNotificationCenter is process-local — notifications only reach observers within the same process. Darwin notifications (via CFNotificationCenterGetDarwinNotifyCenter) are system-wide — they cross process boundaries. This makes Darwin notifications the only notification mechanism that works between an app and its extensions. Limitations: Darwin notifications carry no payload (just the notification name), they're not guaranteed to be delivered if the receiving process is suspended, and they use string-based names (no type safety). Use them for signaling ("data changed"), then have the receiver read the actual data from the shared container.

Q65. How would you design a Share Extension that handles large files without exceeding memory limits? Architect

(1) Never load the entire file into memory. Use NSItemProvider.loadFileRepresentation to get a file URL instead of loadItem which may load into memory. (2) Copy the file to the shared container using FileManager's streaming copy (not Data(contentsOf:)). (3) Write a "pending upload" record to shared UserDefaults. (4) Call completeRequest to dismiss the extension quickly. (5) The main app picks up the pending upload on next launch or via Darwin notification. (6) Use a background URLSession in the main app for the actual upload (survives process death). (7) For images, downsample thumbnails for the extension UI using CGImageSource. (8) Monitor memory with os_proc_available_memory() and abort gracefully if running low.
