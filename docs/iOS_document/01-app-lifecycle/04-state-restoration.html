1. Why State Restoration Matters

State restoration is the mechanism by which an app recreates its UI state after process termination. On iOS, this is not a nice-to-have — it's essential because the system routinely kills background apps via Jetsam. Without state restoration, users return to your app and find themselves at the root screen instead of where they left off.

User is on Screen C → Switches to Safari → Jetsam kills your app │ ▼ ┌─────────────────────┐ │ WITHOUT RESTORATION │ │ User returns → Root │ │ screen. Frustrating.│ └─────────────────────┘ │ ┌─────────────────────┐ │ WITH RESTORATION │ │ User returns → │ │ Screen C restored. │ │ Seamless. │ └─────────────────────┘

1.1 Three Eras of State Restoration

Era API iOS Version Status
Generation 1 NSCoder-based (UIStateRestoring) iOS 6–12 Deprecated for scene-based apps
Generation 2 NSUserActivity-based iOS 13+ Current UIKit approach
Generation 3 @SceneStorage iOS 14+ SwiftUI native approach

2. NSCoder-Based Restoration (Legacy)

The original state restoration system was deeply integrated into UIKit. The system would walk the view controller hierarchy, encode each controller's state using NSCoder, and save it to disk.

2.1 How It Worked

SAVE (entering background): UIApplication └── shouldSaveApplicationState → true └── Walk VC hierarchy (by restorationIdentifier) ├── encodeRestorableState(with: NSCoder) ├── encodeRestorableState(with: NSCoder) └── encodeRestorableState(with: NSCoder) └── Write archive to disk RESTORE (next launch): UIApplication └── shouldRestoreApplicationState → true └── Read archive from disk └── For each saved VC: ├── viewControllerWithRestorationIdentifierPath → create VC └── decodeRestorableState(with: NSCoder)

2.2 Implementation

// AppDelegate: opt in to restoration
func application(_ app: UIApplication,
    shouldSaveSecureApplicationState coder: NSCoder) -> Bool {
    return true
}

func application(_ app: UIApplication,
    shouldRestoreSecureApplicationState coder: NSCoder) -> Bool {
    // Check version compatibility
    let savedVersion = coder.decodeObject(
        of: NSString.self,
        forKey: UIApplication.stateRestorationBundleVersionKey
    ) as? String
    let currentVersion = Bundle.main.infoDictionary?["CFBundleVersion"] as? String
    return savedVersion == currentVersion
}

// View Controller: encode/decode state
class DetailViewController: UIViewController {
    var itemID: String?
    
    override var restorationIdentifier: String? {
        get { return "DetailVC" }
        set { }
    }
    
    override var restorationClass: UIViewControllerRestoration.Type? {
        return DetailViewController.self
    }
    
    override func encodeRestorableState(with coder: NSCoder) {
        super.encodeRestorableState(with: coder)
        coder.encode(itemID, forKey: "itemID")
    }
    
    override func decodeRestorableState(with coder: NSCoder) {
        super.decodeRestorableState(with: coder)
        itemID = coder.decodeObject(
            of: NSString.self,
            forKey: "itemID"
        ) as? String
    }
}

// UIViewControllerRestoration: factory method
extension DetailViewController: UIViewControllerRestoration {
    static func viewController(
        withRestorationIdentifierPath path: [String],
        coder: NSCoder
    ) -> UIViewController? {
        return DetailViewController()
    }
}
Common Pitfall: NSCoder Security

The original shouldSaveApplicationState / shouldRestoreApplicationState methods used insecure NSCoder decoding. Apple deprecated these in iOS 13.2 and replaced them with shouldSaveSecureApplicationState / shouldRestoreSecureApplicationState which use NSSecureCoding. The insecure versions are vulnerable to object substitution attacks if the restoration archive is tampered with.

3. NSUserActivity-Based Restoration

iOS 13's scene-based lifecycle introduced a simpler, more flexible approach using NSUserActivity. Instead of encoding the entire view controller hierarchy, you encode a high-level description of the user's activity.

3.1 Saving State

class SceneDelegate: UIResponder, UIWindowSceneDelegate {
    
    // Called when the system wants to save state for this scene
    func stateRestorationActivity(for scene: UIScene) -> NSUserActivity? {
        let activity = NSUserActivity(activityType: "com.app.browsing")
        
        // Encode current state
        guard let navController = window?.rootViewController as? UINavigationController else {
            return nil
        }
        
        var state: [String: Any] = [:]
        state["selectedTab"] = tabController?.selectedIndex ?? 0
        
        // Encode navigation stack as route descriptors
        let routes = navController.viewControllers.compactMap { vc -> [String: Any]? in
            guard let restorable = vc as? StateRestorable else { return nil }
            return restorable.restorationState
        }
        state["navigationStack"] = routes
        
        activity.userInfo = state
        activity.title = "App State"
        return activity
    }
}

3.2 Restoring State

func scene(_ scene: UIScene,
    willConnectTo session: UISceneSession,
    options: UIScene.ConnectionOptions) {
    
    guard let windowScene = scene as? UIWindowScene else { return }
    
    // Create window and basic UI
    let window = UIWindow(windowScene: windowScene)
    let rootVC = MainTabController()
    window.rootViewController = rootVC
    window.makeKeyAndVisible()
    self.window = window
    
    // Restore state from session's activity
    if let activity = session.stateRestorationActivity,
       activity.activityType == "com.app.browsing" {
        restore(from: activity, into: rootVC)
    }
    
    // Handle incoming URL/activity (takes priority over restoration)
    if let urlContext = options.urlContexts.first {
        handle(url: urlContext.url, in: rootVC)
    } else if let userActivity = options.userActivities.first {
        handle(userActivity: userActivity, in: rootVC)
    }
}

private func restore(from activity: NSUserActivity, into tabController: MainTabController) {
    guard let userInfo = activity.userInfo else { return }
    
    if let tabIndex = userInfo["selectedTab"] as? Int {
        tabController.selectedIndex = tabIndex
    }
    
    if let routes = userInfo["navigationStack"] as? [[String: Any]] {
        let viewControllers = routes.compactMap { routeData -> UIViewController? in
            ViewControllerFactory.create(from: routeData)
        }
        if let nav = tabController.selectedViewController as? UINavigationController {
            nav.setViewControllers(viewControllers, animated: false)
        }
    }
}

4. SwiftUI @SceneStorage

SwiftUI provides the simplest state restoration API yet: @SceneStorage. It automatically persists values per-scene and restores them when the scene reconnects.

4.1 Basic Usage

struct ContentView: View {
    // Automatically saved and restored per-scene
    @SceneStorage("selectedTab") private var selectedTab: String = "home"
    @SceneStorage("searchText") private var searchText: String = ""
    @SceneStorage("scrollPosition") private var scrollPosition: Double = 0
    
    var body: some View {
        TabView(selection: $selectedTab) {
            HomeView(searchText: $searchText)
                .tag("home")
            SettingsView()
                .tag("settings")
        }
    }
}

4.3 Custom Types via RawRepresentable

// Use RawRepresentable to store complex types
enum AppTab: String {
    case home, search, profile, settings
}

struct ContentView: View {
    @SceneStorage("tab") private var tab: AppTab = .home
    // Works because AppTab: String (RawRepresentable)
}

// For complex state, encode as Data
struct NavigationState: Codable {
    var path: [String]
    var selectedID: String?
}

extension NavigationState: RawRepresentable {
    var rawValue: String {
        guard let data = try? JSONEncoder().encode(self),
              let string = String(data: data, encoding: .utf8)
        else { return "" }
        return string
    }
    
    init?(rawValue: String) {
        guard let data = rawValue.data(using: .utf8),
              let decoded = try? JSONDecoder().decode(NavigationState.self, from: data)
        else { return nil }
        self = decoded
    }
}

struct DetailView: View {
    @SceneStorage("navState") private var navState = NavigationState(path: [], selectedID: nil)
}
Common Pitfall: @SceneStorage Limitations

@SceneStorage is not a general-purpose persistence mechanism. It's specifically for scene state restoration. Key limitations:

  • Data is not shared between scenes (each scene has its own storage)
  • Data may be purged by the system at any time
  • Not suitable for user data — only UI state
  • No encryption — don't store sensitive information
  • Size should be kept small (kilobytes, not megabytes)

5. restorationIdentifier Chain

In the legacy NSCoder system, every view controller in the hierarchy needed a restorationIdentifier. The system used the chain of identifiers to uniquely identify each controller:

// The restoration identifier path for a deeply nested VC:
// ["TabBarController", "NavigationController", "DetailVC"]

// If ANY controller in the chain is missing an identifier,
// all its children are excluded from restoration.

class MainTabController: UITabBarController {
    override func viewDidLoad() {
        super.viewDidLoad()
        restorationIdentifier = "MainTab"
        
        let nav = UINavigationController(rootViewController: HomeViewController())
        nav.restorationIdentifier = "HomeNav"  // Must set on nav too!
        
        viewControllers = [nav]
    }
}

class HomeViewController: UIViewController {
    override func viewDidLoad() {
        super.viewDidLoad()
        restorationIdentifier = "HomeVC"
    }
}
Deep Dive: Identifier Uniqueness

The restorationIdentifier only needs to be unique among siblings — not globally. The system builds a path like ["TabBar", "Nav", "Detail"]. However, if you have multiple instances of the same VC class (e.g., in a tab bar), each needs a distinct identifier. A common pattern is to use the tab index or a content identifier as part of the restoration identifier.

6. Why State Restoration Broke in Scene-Based Apps

The transition to scene-based lifecycle in iOS 13 broke the legacy NSCoder restoration system in several ways:

6.1 The Breaking Changes

Issue Pre-iOS 13 iOS 13+ (Scene-based)
Restoration timing Between willFinish and didFinish During scene connection
Restoration scope Entire app (one window) Per-scene
Window ownership AppDelegate creates window SceneDelegate creates window
Save trigger applicationDidEnterBackground stateRestorationActivity(for:)
NSCoder methods Called automatically Not called for scene-based apps

6.2 Migration Path

// If you had NSCoder-based restoration and adopted scenes,
// you need to migrate to NSUserActivity-based restoration.

class SceneDelegate: UIResponder, UIWindowSceneDelegate {
    
    func stateRestorationActivity(for scene: UIScene) -> NSUserActivity? {
        // Replace the entire NSCoder chain with a single activity
        let activity = NSUserActivity(activityType: "com.app.state")
        
        // Encode your state as a simple dictionary
        var state = AppStateSnapshot()
        state.selectedTab = tabController.selectedIndex
        state.navigationStack = captureNavigationStack()
        state.scrollPositions = captureScrollPositions()
        
        if let data = try? JSONEncoder().encode(state) {
            activity.addUserInfoEntries(from: ["state": data])
        }
        
        return activity
    }
}

7. Codable State Serialization

7.1 Designing Restorable State

// Define a Codable state model for your entire app
struct AppStateSnapshot: Codable {
    var version: Int = 1  // For migration
    var selectedTab: Int = 0
    var navigationStacks: [String: [RouteDescriptor]] = [:]
    var scrollPositions: [String: CGFloat] = [:]
    var searchQueries: [String: String] = [:]
    var draftContent: [String: String] = [:]
}

struct RouteDescriptor: Codable {
    let type: String       // "detail", "profile", "settings"
    let parameters: [String: String]  // ["id": "123"]
}

// Protocol for view controllers that can save/restore state
protocol StateRestorable {
    var routeDescriptor: RouteDescriptor { get }
    func restore(from descriptor: RouteDescriptor)
}

// Factory to recreate VCs from descriptors
enum ViewControllerFactory {
    static func create(from descriptor: RouteDescriptor) -> UIViewController? {
        switch descriptor.type {
        case "detail":
            guard let id = descriptor.parameters["id"] else { return nil }
            return DetailViewController(itemID: id)
        case "profile":
            guard let userID = descriptor.parameters["userID"] else { return nil }
            return ProfileViewController(userID: userID)
        default:
            return nil
        }
    }
}

7.2 Version Migration

struct StateMigrator {
    static func migrate(_ data: Data) -> AppStateSnapshot? {
        let decoder = JSONDecoder()
        
        // Try current version first
        if let state = try? decoder.decode(AppStateSnapshot.self, from: data),
           state.version == AppStateSnapshot.currentVersion {
            return state
        }
        
        // Try v1 and migrate
        if let v1 = try? decoder.decode(AppStateSnapshotV1.self, from: data) {
            return migrateV1ToV2(v1)
        }
        
        // Unrecognized version — return nil (start fresh)
        return nil
    }
}

8. Architect Considerations

Architect Must-Know: Building Robust State Restoration Across App Updates

State restoration that works within a single app version is easy. Building a system that works across app updates — where view controllers may be renamed, navigation flows restructured, and data models changed — is an architectural challenge.

8.1 Design Principles

// Principle 1: Restore INTENT, not UI structure
// BAD: "User was on DetailViewController with itemID 123"
// GOOD: "User was viewing item 123"

// Principle 2: Version your state format
struct RestorationState: Codable {
    static let currentVersion = 3
    let version: Int
    let intent: UserIntent  // What the user was doing
}

// Principle 3: Use route descriptors, not class names
enum UserIntent: Codable {
    case viewingItem(id: String)
    case composingMessage(draftID: String?)
    case browsing(category: String, page: Int)
    case searching(query: String)
}

// Principle 4: Graceful degradation
func restore(intent: UserIntent) {
    switch intent {
    case .viewingItem(let id):
        if canNavigateToItem(id) {
            navigateToItem(id)
        } else {
            // Item may have been deleted — fall back gracefully
            navigateToHome()
        }
    case .composingMessage(let draftID):
        if let draftID, draftExists(draftID) {
            openComposer(draftID: draftID)
        } else {
            openComposer(draftID: nil)
        }
    default:
        navigateToHome()
    }
}

// Principle 5: Test restoration with process kill
// In Xcode: Debug → Simulate Background Fetch
// Or: stop the app from Xcode while it's backgrounded

8.2 Unified Restoration System

// A unified system that works for both UIKit and SwiftUI
@MainActor
final class RestorationManager: ObservableObject {
    @Published var currentIntent: UserIntent?
    
    private let encoder = JSONEncoder()
    private let decoder = JSONDecoder()
    
    // Save to NSUserActivity for UIKit scene restoration
    func activity() -> NSUserActivity? {
        guard let intent = currentIntent,
              let data = try? encoder.encode(intent)
        else { return nil }
        
        let activity = NSUserActivity(activityType: "com.app.restore")
        activity.addUserInfoEntries(from: ["intent": data])
        
        // Bonus: make it eligible for Handoff and Spotlight
        activity.isEligibleForHandoff = true
        activity.isEligibleForSearch = true
        activity.title = intent.displayTitle
        
        return activity
    }
    
    // Restore from NSUserActivity
    func restore(from activity: NSUserActivity) {
        guard let data = activity.userInfo?["intent"] as? Data,
              let intent = try? decoder.decode(UserIntent.self, from: data)
        else { return }
        
        currentIntent = intent
    }
}
Best Practice: State Restoration Testing

State restoration is notoriously hard to test because it requires process death. Strategies:

  • In Xcode: background your app, then stop it from Xcode. Relaunch to test restoration.
  • Write unit tests that encode/decode your state models to verify round-trip fidelity
  • Test migration by keeping old state snapshots as test fixtures
  • Test graceful degradation: what happens when restored state references deleted content?
  • Use XCUIApplication.terminate() in UI tests to simulate process death
