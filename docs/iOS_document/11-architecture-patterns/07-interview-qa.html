MVC / MVVM / Architecture Patterns (Questions 1–12)

1. Explain the difference between Smalltalk's original MVC and Apple's Cocoa MVC. Why did Apple change it? Senior

In Smalltalk's MVC (1979), the View directly observes the Model via the Observer pattern and updates itself. The Controller only handles user input translation. Each widget (text field, button) has its own MVC triad.

Apple's Cocoa MVC makes the Controller a mediator between View and Model. The View never talks to the Model directly. Apple changed this for practical reasons:

  • Interface Builder — Views designed in IB need to be generic and reusable. A UITableView shouldn't know about your User model.
  • View lifecycle — iOS views have complex lifecycles (viewDidLoad, rotation, trait changes) that need management.
  • Memory management — Pre-ARC, direct View→Model observation created retain cycle nightmares.
  • Reusability — Generic cells (UITableViewCell) can be reused across different data types.

The tradeoff: Apple's MVC concentrates responsibilities in the Controller, leading to the Massive View Controller problem.

2. What is the Massive View Controller problem? Is it a code size issue or a design issue? Senior

It's a design issue, not a code size issue. A 200-line VC that handles networking, formatting, navigation, and error presentation is already "massive" in terms of coupling. The problem is responsibility accumulation:

  • UIViewController owns the view hierarchy (self.view)
  • Delegation patterns (UITableViewDataSource, UITextFieldDelegate) point back to the VC
  • Lifecycle hooks (viewDidLoad, viewWillAppear, traitCollectionDidChange) live on the VC
  • Navigation (present, push) is called from the VC
  • Without a binding mechanism, the VC manually shuttles data between Model and View

Consequences: untestable (can't test formatting without UIKit), merge conflicts (multiple devs editing same file), hidden dependencies (VC depends on 10+ services), and no reuse (logic trapped inside VC).

3. Compare MVVM with ObservableObject vs @Observable. When would you choose each? L6

ObservableObject + @Published (iOS 13+): Uses objectWillChange publisher. ANY @Published property change fires objectWillChange, causing ALL subscribing views to re-evaluate their body. This is object-level granularity.

@Observable (iOS 17+): Uses the Observation framework with per-property tracking via compiler-generated accessors. Only views that actually READ a changed property re-evaluate. This is property-level granularity.

Choose ObservableObject when: Supporting iOS 13-16, using Combine pipelines extensively, or when the ViewModel has few properties (granularity difference is negligible).

Choose @Observable when: iOS 17+ minimum, ViewModel has many properties (avoids unnecessary re-renders), or when you want simpler code (no @StateObject/@ObservedObject distinction — just @State or let).

Key difference: With @Observable, you don't need @StateObject vs @ObservedObject. The ownership model is simpler — use @State for owned instances, plain let for injected ones.

4. You're building a new app from scratch with a team of 8 iOS developers. The app has 40 screens. Which architecture would you choose and why? L6

For 8 developers and 40 screens, I'd choose MVVM + Coordinators:

  • MVVM provides testable ViewModels without excessive boilerplate. At 8 developers, VIPER's 5-7 files per screen is overkill.
  • Coordinators separate navigation from view logic, enabling deep linking and A/B tested flows.
  • Modularization: Split into 5-8 SPM packages (Domain, Network, DesignSystem, 3-4 feature modules). This gives each pair of developers ownership of 1-2 modules.

Why not TCA: Learning curve is steep. With 8 developers of varying experience, MVVM is universally understood. TCA would slow onboarding.

Why not VIPER: 40 screens × 5-7 files = 200-280 architecture files. The boilerplate cost doesn't justify the separation for this team size.

Why not plain MVC: 8 developers will step on each other's toes in monolithic VCs. MVVM's separation enables parallel work.

5. Explain TCA's Reducer, Store, Effect, and how composition works. What problem does @Dependency solve? L6

Reducer: A pure function that takes (inout State, Action) and returns an Effect. It describes how state changes in response to actions. "Pure" means no side effects in the state mutation — side effects are returned as Effects.

Store: Holds the current state and processes actions by forwarding them to the reducer. The Store is the single source of truth. Views observe the Store's state.

Effect: Represents a side effect (API call, timer, database write). Effects are async operations that eventually produce new actions, which are fed back into the Store. This creates the unidirectional loop: View → Action → Reducer → State → View, with Effects as the async escape hatch.

Composition: Parent reducers compose child reducers using Scope. A parent's state contains child state as properties, and parent actions contain child actions as enum cases. The Scope reducer focuses the parent's state/action down to the child's domain.

@Dependency: TCA's dependency injection system. It solves the problem of passing dependencies through the entire reducer tree. Instead of constructor injection (which would require threading dependencies through every parent), @Dependency uses a context-based system. In tests, you override dependencies in the TestStore's withDependencies closure, making tests deterministic.

6. What are the failure modes of VIPER at scale? When does VIPER's overhead become justified? L6

Failure modes:

  • Boilerplate explosion: 5-7 files and 6+ protocols per screen. A 100-screen app has 500-700 architecture files. Without code generation (Sourcery), this is unsustainable.
  • Protocol ceremony: Changing a simple interaction requires updating 3-4 protocol definitions and their implementations.
  • Navigation rigidity: The Router is tightly coupled to UIKit. SwiftUI adoption is awkward.
  • Over-separation: Simple screens (settings, about) don't need Interactor/Presenter separation. VIPER forces it anyway.
  • Onboarding cost: New developers need 2-4 weeks to understand the pattern and its conventions.

Justified when: 20+ developers (strict separation prevents conflicts), complex business logic per screen (Interactor isolation is valuable), and when combined with code generation. Uber's RIBs (a VIPER variant) works because they have dedicated tooling and 100+ iOS developers.

7. How would you incrementally migrate a 500K-line MVC codebase to MVVM without a rewrite? Architect

Use the Strangler Fig Pattern — wrap new architecture around old code incrementally:

  1. Phase 1 (weeks 1-4): Extract ViewModels from existing VCs. Start with the most-changed screens. Move business logic, formatting, and state management into a ViewModel. The VC becomes a thin binding layer. No navigation changes yet.
  2. Phase 2 (weeks 4-8): Add Coordinator layer. Extract navigation from VCs into coordinators. Start with one flow (e.g., checkout). VCs delegate navigation decisions upward.
  3. Phase 3 (weeks 8-16): Define module boundaries. Create SPM packages for the domain layer (models, protocols) and shared infrastructure (networking, analytics). Feature modules come later.
  4. Phase 4 (ongoing): New features use MVVM+C. Legacy screens stay as MVC until they need changes. When touching old code for bugs, migrate that screen.

Key rules: Never do a "big bang" rewrite. Each PR should be shippable. Maintain a compatibility layer so old MVC screens and new MVVM screens coexist. Track migration progress (% of screens migrated) as a team metric.

8. Compare Clean Architecture's dependency rule with VIPER's component structure. Can you use Clean Architecture without VIPER? L6

Clean Architecture defines concentric layers with a strict dependency rule: dependencies point inward. Inner layers (Entities, Use Cases) know nothing about outer layers (UI, Frameworks). It's a principle, not a specific component structure.

VIPER is a specific component structure (View, Interactor, Presenter, Entity, Router) that happens to follow Clean Architecture's dependency rule. The Interactor is the Use Case layer, the Presenter is the Interface Adapter layer, and the View/Router are the Frameworks layer.

Yes, you can use Clean Architecture without VIPER. MVVM + Clean Architecture is common: the ViewModel acts as the Interface Adapter, Use Cases are separate classes injected into the ViewModel, and the Repository pattern implements the data layer. This gives you Clean Architecture's testability benefits without VIPER's boilerplate.

The key Clean Architecture rule to enforce: your Domain layer (entities + use case protocols) has zero framework imports. No UIKit, no SwiftUI, no Combine. This makes it testable with pure XCTest.

9. What is the role of a ViewModel in SwiftUI vs UIKit? How does the binding mechanism differ? Senior

The ViewModel's role is the same in both: transform Model data into display-ready state, handle user actions, and manage side effects. The binding mechanism differs significantly:

UIKit: Manual binding. The VC subscribes to ViewModel's @Published properties using Combine's sink or assign. The VC must manually update UI elements when state changes. Cancellables must be managed explicitly.

SwiftUI: Automatic binding. SwiftUI's runtime observes the ViewModel (via ObservableObject or @Observable) and automatically re-evaluates the View's body when observed state changes. No manual subscription management needed.

Key differences:

  • SwiftUI ViewModels don't need to worry about threading for UI updates (body is always on MainActor)
  • UIKit ViewModels must ensure .receive(on: DispatchQueue.main) before UI-bound publishers
  • SwiftUI's @Observable provides per-property tracking; UIKit's Combine binding is per-subscription
  • In SwiftUI, the View IS the binding layer. In UIKit, the VC is an additional binding layer between View and ViewModel
10. Your team is debating TCA vs MVVM for a new SwiftUI app. The team has 5 senior devs and 10 mid-level devs. What do you recommend? Architect

I'd recommend MVVM with @Observable for the majority of the app, with TCA considered for specific complex features.

Reasoning:

  • Onboarding: 10 mid-level devs need to be productive quickly. MVVM is universally understood. TCA requires 2-4 weeks of learning (Reducers, Effects, @Dependency, TestStore).
  • Hiring: Most iOS candidates know MVVM. Few know TCA. This affects your hiring pipeline.
  • Debugging: TCA's action log is powerful but the indirection (Action → Reducer → Effect → Action) can confuse mid-level devs.
  • Flexibility: MVVM doesn't lock you into a framework. TCA is a significant dependency.

Where TCA shines: If you have specific features with complex state machines (e.g., a multi-step form with validation, undo, and offline support), TCA's deterministic testing is invaluable. Consider using TCA for those features while keeping MVVM for simpler screens.

Compromise: Adopt TCA's ideas (unidirectional flow, explicit state, action-based mutations) without the framework. Build a lightweight Store that your senior devs maintain.

11. How does Conway's Law apply to iOS architecture decisions? Architect

Conway's Law states: "Organizations design systems that mirror their communication structures." For iOS architecture:

  • Team per feature → Module per feature: If you have a Payments team, a Search team, and a Profile team, your codebase should have PaymentsModule, SearchModule, and ProfileModule. Trying to have shared ownership of a monolithic codebase with feature teams creates constant conflicts.
  • Platform team → Platform SDK: If you have a platform/infrastructure team, they should own shared modules (networking, analytics, design system) as internal SDKs with versioned APIs.
  • Cross-functional teams → Vertical slices: If teams include iOS, Android, and backend devs, architecture should enable vertical feature development (shared API contracts, feature flags).

Practical implication: Before choosing an architecture, map your team structure. If you have 3 teams of 5, you need 3 feature modules with clear interfaces. If you have 1 team of 15, a monolithic MVVM app might work. Architecture follows organization, not the other way around.

12. What are the tradeoffs between protocol-oriented and class-based architecture in Swift? L6

Protocol-oriented: Dependencies are defined as protocols. Implementations are injected. This enables testing with mocks and swapping implementations (e.g., real vs fake network layer).

Tradeoffs:

  • Pro: Testability — inject MockUserService in tests
  • Pro: Flexibility — swap implementations at runtime (A/B test different backends)
  • Con: Existential overhead — any Protocol uses existential containers (24-40 bytes on stack, heap allocation for large types)
  • Con: Protocol witness table dispatch is slower than direct/vtable dispatch
  • Con: Protocol proliferation — every service gets a protocol even if there's only one implementation

Pragmatic approach: Use protocols for boundaries (module interfaces, external services, anything you need to mock in tests). Use concrete types internally within a module where there's only one implementation. The package access level (Swift 5.9) helps — expose protocols publicly, use concrete types within the package.

Coordinator Pattern (Questions 13–20)

13. What problem does the Coordinator pattern solve? Why can't you just use segues or NavigationLink? Senior

The Coordinator pattern solves navigation coupling. Without coordinators, each VC/View knows about its destination screens, creating tight coupling that prevents reuse, complicates deep linking, and makes A/B testing navigation flows impossible.

Why not segues: Segues embed navigation in storyboards. They can't be A/B tested, can't handle conditional navigation (login required?), and create implicit dependencies between VCs.

Why not NavigationLink: Value-based NavigationLink (iOS 16+) is better, but the View still knows about its destination type. For deep linking, you need to construct the entire NavigationPath programmatically — which is exactly what a coordinator does.

Coordinators centralize navigation decisions, making it possible to: route deep links through a tree, A/B test different flows, gate navigation on authentication, and track screen transitions for analytics.

14. How do you detect when a user pops a view controller (back button or swipe) in a coordinator? Senior

The coordinator must be the UINavigationController's delegate and implement navigationController(_:didShow:animated:):

  1. In the delegate method, get the "from" VC via transitionCoordinator?.viewController(forKey: .from)
  2. Check if the "from" VC is still in the navigation stack: !navigationController.viewControllers.contains(fromVC)
  3. If it's NOT in the stack, it was popped
  4. Look up the completion handler registered for that VC and call it

This is necessary because UIKit doesn't have a "didPop" callback. The coordinator maintains a dictionary of [UIViewController: () -> Void] completion handlers, registered when pushing each VC. When a pop is detected, the handler is called to clean up the child coordinator.

SwiftUI alternative: With NavigationStack, observe the NavigationPath's count. When it decreases, a pop occurred. Or use .onDisappear on destination views.

15. How do you implement deep linking through a coordinator tree? L6

Deep linking through a coordinator tree follows a top-down routing pattern:

  1. Parse: Convert the URL into a typed DeepLink enum (e.g., .product(id: "123"))
  2. Route to root: AppCoordinator receives the deep link. It ensures prerequisites (authentication, onboarding) are met.
  3. Traverse tree: AppCoordinator routes to the appropriate child (e.g., TabCoordinator → HomeCoordinator)
  4. Reset if needed: The target coordinator pops to root, then pushes the deep link destination
  5. Handle pending: If auth is required, store the deep link as pending and process it after login

Each coordinator implements a DeepLinkable protocol with handle(deepLink:) -> Bool. The parent tries each child; if none handles it, the parent creates the appropriate child coordinator.

Key challenge: Ensuring the navigation stack is in the correct state before pushing the deep link destination. You may need to dismiss modals, pop to root, and switch tabs before navigating.

16. What is the most common memory leak in coordinator-based architectures? Senior

The most common leak is child coordinators not being removed from the parent's array when their flow completes. If a user navigates back (pop) and the child coordinator isn't removed, it leaks along with all its VCs, ViewModels, and captured closures.

Causes:

  • Forgetting to implement the UINavigationControllerDelegate pop detection
  • Strong reference cycles in coordinator closures (not using [weak self])
  • Child coordinator's onFinish closure not being called

Prevention: Use a dictionary ([UUID: Coordinator]) instead of an array for child coordinators. Set the onFinish closure automatically in addChild. In debug builds, add a deinit print to verify coordinators are deallocated. Use Xcode's Memory Graph Debugger to detect leaked coordinators.

17. How do you implement coordinators in SwiftUI with NavigationStack? L6

SwiftUI coordinators use NavigationPath as the navigation state and a coordinator object that manipulates it:

  1. Create an @Observable coordinator class that owns a NavigationPath
  2. Define a Route enum (Hashable) for all possible destinations
  3. The coordinator exposes push(_:), pop(), popToRoot() methods
  4. The root view uses NavigationStack(path: $coordinator.path)
  5. Use .navigationDestination(for: Route.self) to map routes to views
  6. For sheets/covers, use separate @Published properties with .sheet(item:)

Key difference from UIKit coordinators: SwiftUI coordinators are value-driven (manipulate a path) rather than imperative (call pushViewController). Deep linking is simpler — just set the path to the desired stack. But detecting "pop" is harder — you need to observe path changes.

18. Design a coordinator architecture for an app with 100+ screens, deep linking, and A/B tested flows. Architect

At 100+ screens, the coordinator tree needs infrastructure:

  • CoordinatorFactory: Creates coordinators with injected dependencies. Respects feature flags and A/B test assignments. The factory decides which variant of a flow to instantiate.
  • DeepLinkRouter: Parses URLs into typed routes, then traverses the coordinator tree top-down. Handles auth gates (pending deep links) and tab switching.
  • AnalyticsMiddleware: Wraps coordinator transitions to automatically track screen views, navigation timing, and funnel progression.
  • Coordinator Registry: In debug builds, maintains a live tree visualization for diagnosing navigation issues.

Module boundaries: Each feature module exposes a coordinator protocol. The app target's composition root wires them together. Feature modules don't know about each other — they communicate through the parent coordinator.

A/B testing: The CoordinatorFactory checks experiment assignments and creates the appropriate variant. E.g., makeCheckoutCoordinator() returns either NewCheckoutCoordinator or LegacyCheckoutCoordinator based on the user's experiment group.

19. How should coordinators communicate with each other? Delegation, closures, or NotificationCenter? Senior

Closures are the best default for coordinator communication. Each child coordinator exposes completion closures (e.g., onFinish, onLoginSuccess) that the parent sets when creating the child.

Why not delegation: Delegation requires defining a protocol for each coordinator pair. With 30+ coordinators, that's 30+ delegate protocols. Closures are more concise and don't require protocol definitions.

Why not NotificationCenter: Notifications are untyped, global, and hard to trace. They create invisible coupling. If coordinator A posts a notification and coordinator B observes it, there's no compile-time guarantee that B exists or handles it correctly.

Exception: For truly cross-cutting events (user logged out, app entering background), a typed event bus (Combine PassthroughSubject) is appropriate. But coordinator-to-coordinator communication should be parent-child via closures.

20. How do you handle authentication gates in a coordinator tree? L6

Authentication gates are handled at the AppCoordinator level:

  1. AppCoordinator checks auth state on launch. If not authenticated, show AuthCoordinator.
  2. When a deep link arrives that requires auth, store it as pendingDeepLink and show AuthCoordinator.
  3. AuthCoordinator's onLoginSuccess closure triggers AppCoordinator to: remove AuthCoordinator, show MainApp, and process the pending deep link.
  4. For mid-session auth requirements (e.g., "login to purchase"), the feature coordinator calls up to the parent, which presents AuthCoordinator modally. On success, the feature coordinator continues.

Key principle: Feature coordinators should never know about the auth flow. They signal "authentication required" upward, and the parent handles it. This keeps auth logic centralized and enables changing the auth flow (add biometric, add SSO) without touching feature code.

Reactive Architecture (Questions 21–28)

21. Compare Combine, AsyncSequence, and the Observation framework. When would you use each? L6

Combine: Best for complex event stream processing — debounce, combineLatest, retry, throttle. Use for search-as-you-type, real-time data streams, and complex multi-source data pipelines. Has built-in backpressure via Demand.

AsyncSequence: Best for consuming async data sequences — for-await loops, bridging callback APIs. Natural backpressure via await. Simpler mental model than Combine. Use for iterating over paginated API results, WebSocket messages, or file system events.

Observation (@Observable): Best for UI state binding in SwiftUI. Per-property tracking eliminates unnecessary re-renders. Not a stream — it's synchronous property observation. Use for ViewModel → View binding on iOS 17+.

Decision tree: Need operators like debounce/combineLatest? → Combine. Processing async values sequentially? → AsyncSequence. Simple UI state binding? → @Observable. Supporting iOS 13-16? → Combine + ObservableObject.

22. What is unidirectional data flow? Why does it make debugging easier? Senior

Unidirectional data flow (UDF) ensures state changes follow a single direction: View → Action → Reducer → State → View. The View never mutates state directly — it sends actions, and the reducer produces new state.

Why debugging is easier:

  • Single source of truth: State lives in one place (the Store). No scattered mutable state across VCs.
  • Action log: Every state change is triggered by a named action. You can log all actions to see exactly what happened.
  • Time-travel debugging: Since state is a value type, you can snapshot it at any point and replay actions.
  • Deterministic: Given the same initial state and sequence of actions, you always get the same final state. No race conditions in state mutations.
  • No "who changed this?": In bidirectional binding (two-way), any component can mutate state, making it hard to trace changes. In UDF, only the reducer mutates state.
23. How does withObservationTracking work? What is its limitation? L6

withObservationTracking takes two closures: an apply closure and an onChange closure. During the apply closure, the Observation framework records which @Observable properties are accessed. When any of those properties change, the onChange closure fires.

Key limitation: onChange fires exactly once, then stops observing. You must re-register to continue observing. This is by design — it prevents accidental infinite observation loops.

SwiftUI handles this automatically (it re-registers after each body evaluation). For UIKit integration, you need a manual re-registration loop:

func observe() {
    withObservationTracking {
        updateUI(with: model.name)
    } onChange: {
        DispatchQueue.main.async { self.observe() }
    }
}

The onChange closure is called on an arbitrary thread, so you must dispatch to main for UI updates.

24. What are the risks of using Combine extensively in a large codebase? L6

Debugging difficulty: Long Combine chains produce incomprehensible stack traces. When a flatMap inside a combineLatest inside a debounce fails, the error is nearly impossible to trace.

Subscription leaks: Forgotten AnyCancellable storage leads to memory leaks. Subscriptions that outlive their intended scope cause stale data and unexpected behavior.

Thread confusion: Combine operators can switch threads implicitly. Without explicit .receive(on:), UI updates may happen on background threads.

Infinite loops: Two publishers that observe each other create infinite update cycles. Hard to detect until CPU spikes.

Testing complexity: Testing time-dependent Combine chains requires TestScheduler injection. Without it, tests are flaky.

Mitigation: Limit chain length to 5-6 operators. Use the "reactive boundary" pattern — Combine at system boundaries, imperative logic inside. Name intermediate publishers. Always specify schedulers explicitly.

25. Explain state machines in iOS. When should you use one instead of boolean flags? Senior

A state machine formalizes the valid states and transitions of a system. Instead of multiple booleans (isLoading, hasError, hasData) that can be in invalid combinations (loading AND error), you use an enum with explicit states.

Use a state machine when:

  • You have 3+ states with specific valid transitions
  • Invalid state combinations are possible with booleans (isLoading && hasError)
  • State transitions have side effects (entering "loading" starts a network request)
  • You need to audit or log state transitions

Example: A payment flow has states: Idle → Validating → Processing → Succeeded/Failed. Using booleans (isValidating, isProcessing, hasSucceeded, hasFailed) allows 16 combinations, but only 5 are valid. An enum enforces exactly those 5 states.

Swift enums with associated values are natural state machines. For complex cases, use GKStateMachine from GameplayKit or a custom state machine with a transition table.

26. How would you design a reactive architecture that scales to 20+ feature teams without creating debugging nightmares? Architect

The key principle is reactive at boundaries, imperative inside:

  • Module boundaries: Use Combine or AsyncSequence at module interfaces (network → ViewModel, user input → search). These are natural stream boundaries.
  • Inside modules: Use async/await for business logic. It's imperative, debuggable, and has clear stack traces.
  • UI binding: Use @Observable (iOS 17+) or ObservableObject. Simple property observation, not complex chains.

Guidelines for teams:

  1. No Combine chain longer than 5 operators. Break into named intermediate publishers.
  2. Every AnyCancellable must be stored in a named Set. No orphaned subscriptions.
  3. All schedulers must be explicit — no implicit thread switching.
  4. No nested subscriptions (subscribing inside a sink). Use flatMap or switchToLatest.
  5. Inject schedulers for testability. Never use DispatchQueue.main directly in Combine chains.

Enforcement: SwiftLint custom rules for chain length, mandatory .receive(on:) before sink, and CI checks for subscription leak patterns.

27. What is the "reactive boundary" pattern? L6

The reactive boundary pattern uses reactive programming only at system boundaries — where data enters or leaves a component — while keeping internal logic imperative.

Boundaries where reactive is valuable: Network responses arriving over time, user input events (search-as-you-type), real-time data (WebSocket, push notifications), timer-based updates.

Inside the boundary: Business logic uses async/await. Formatting uses pure functions. State mutations are direct assignments.

Example: A ViewModel uses Combine to debounce search input (boundary), but the actual search logic is an async function. The ViewModel exposes @Published properties (boundary to View), but internal state management is imperative.

This gives you reactive's benefits (declarative event handling) without its costs (debugging complexity, learning curve) for the majority of your code.

28. How do you bridge Combine publishers to AsyncSequence and vice versa? Senior

Combine → AsyncSequence: Use the .values property on any publisher (iOS 15+):

let publisher = NotificationCenter.default.publisher(for: .init("event"))
for await notification in publisher.values {
    // Process each notification
}

AsyncSequence → Combine: No built-in bridge. Create a custom publisher or use PassthroughSubject:

let subject = PassthroughSubject<Value, Never>()
Task {
    for await value in asyncSequence {
        subject.send(value)
    }
    subject.send(completion: .finished)
}

Key consideration: Combine publishers are multi-subscriber (multiple sinks share one upstream). AsyncSequence is single-consumer (each for-await loop gets its own iteration). When bridging, be aware of this semantic difference.

Modularization (Questions 29–38)

29. What is the difference between a static library, dynamic library, and framework on iOS? Senior

Static library (.a): Object code copied into the app binary at link time. Zero runtime loading cost. But code is duplicated if multiple targets link the same library (app + extension).

Dynamic library (.dylib): Loaded by dyld at runtime. Shared between app and extensions (reducing total size). But each dynamic library adds ~2-5ms to launch time. Apple recommends ≤6 dynamic frameworks.

Framework (.framework): A bundle containing a dynamic library (or static library) plus headers, modules, and resources. It's the standard distribution format for iOS libraries.

SPM default: SPM uses static linking by default (type: .automatic resolves to static). This is correct for most cases — faster launch, no dyld overhead.

Xcode 15 Mergeable Libraries: Build as dynamic during development (fast incremental builds) and merge into static for release (fast launch). Best of both worlds.

30. How does the `package` access level (Swift 5.9) change module design? L6

Before package, you had two choices for cross-target visibility within a package: public (visible to everyone) or internal (visible only within the target). This forced you to make implementation details public just so sibling targets could access them.

package adds a middle ground: visible to all targets within the same Swift package, but invisible to external consumers. This enables:

  • Multi-target packages: A package with CheckoutUI and CheckoutDomain targets can share types without making them public.
  • Cleaner API surface: Only truly external APIs need to be public. Internal coordination types use package access.
  • Testing: Test targets in the same package can access package-level types without @testable import.

This is particularly valuable for the "interface module" pattern — a package can have a public interface target and a package-access implementation target.

31. What is @_exported import? What are its risks? L6

@_exported import re-exports a module's symbols. When module A does @_exported import B, anyone importing A also gets B's symbols without explicitly importing B.

Use case: Umbrella modules. A CoreKit module that re-exports DomainModels, CoreUtils, and DesignSystem so feature modules only need import CoreKit.

Risks:

  • Not officially supported: The underscore prefix means it's not part of Swift's stable API. It can break between Swift versions.
  • Hidden dependencies: A module appears to not need an import, but it actually does through transitive re-export. If the umbrella module removes the re-export, all dependents break.
  • Build graph impact: Changes to the re-exported module trigger recompilation of everything that imports the umbrella module.
  • Name collisions: Re-exporting multiple modules increases the chance of symbol name conflicts.

Recommendation: Use sparingly. Only for intentional umbrella modules where the re-export is documented and expected.

32. Compare Tuist, XcodeGen, and Bazel for iOS project generation. When would you choose each? L6

Tuist: Swift-based manifests, generates .xcodeproj. Offers Tuist Cloud for remote caching. Best for 10-100 developer teams that want project generation + build optimization without Bazel's complexity. Learning curve: medium.

XcodeGen: YAML-based manifests, generates .xcodeproj. Simpler than Tuist but fewer features (no caching, no graph optimization). Best for 5-50 developer teams that just want to eliminate .xcodeproj merge conflicts. Learning curve: low.

Bazel: Starlark-based BUILD files. Hermetic builds, remote caching, remote execution. Best for 50+ developer teams (Google, Uber, Lyft) where build time is a critical bottleneck. Requires dedicated build infrastructure team. Learning curve: very high.

SPM alone: For 1-20 developer teams, SPM packages with Xcode's native support may be sufficient. No additional tooling needed.

Decision: Start with SPM. When .xcodeproj conflicts become painful, add XcodeGen or Tuist. When build times exceed 15 minutes and you have a team to maintain it, consider Bazel.

33. What is the "interface module" pattern and why is it important for large codebases? Architect

The interface module pattern creates a companion module for each feature that contains only protocols and data types — no implementations. Feature modules depend on each other's interfaces, never implementations.

Example: FeatureHomeInterface (protocols: HomeModuleInterface, HomeDependencies) and FeatureHome (implementation: HomeModule, HomeViewModel, HomeView). FeatureSearch depends on FeatureHomeInterface to navigate to home, but not on FeatureHome's implementation.

Benefits:

  • Breaks circular dependencies: Home can navigate to Search and Search can navigate to Home — both depend on each other's interfaces, not implementations.
  • Faster compilation: Interface modules are tiny (just protocols). Changes to implementation don't trigger recompilation of dependents.
  • Independent development: Teams can work on their feature without waiting for other teams' implementations. Mock the interface for development.
  • Enforced boundaries: The interface module IS the API contract. Changes require explicit API review.
34. How would you modularize a 2M+ line iOS codebase? What's the migration strategy? Architect

Phase 1 (weeks 1-4): Extract Foundation Layer. Move pure utilities, extensions, and helpers into a CoreUtils SPM package. Zero dependencies on app code. This is the safest first step.

Phase 2 (weeks 4-8): Extract Domain Models. Move all data models and repository protocols into a DomainModels package. Depends only on Foundation and CoreUtils.

Phase 3 (weeks 8-16): Extract Infrastructure. Network layer, storage, analytics as separate packages. These implement domain protocols.

Phase 4 (weeks 8-16, parallel): Extract Design System. Shared UI components, colors, typography.

Phase 5 (weeks 16-52): Extract Features. One feature at a time, starting with the most independent (fewest cross-feature dependencies). Each feature becomes a separate SPM package.

Phase 6 (ongoing): Enforce Boundaries. CI checks for layer violations. CODEOWNERS per module. Module API review process.

Key metrics: Module count (target 20-50), max fan-out (≤8 dependencies), incremental build time (<30s), cross-module PRs (<10% of all PRs).

35. What are the tradeoffs between mono-repo and poly-repo for iOS modules? L6

Mono-repo (all code in one repo): Atomic changes across modules in one PR. Always uses latest version of every module. Easy code discovery (grep everything). But requires sophisticated build tooling (Bazel/Tuist) and CI that can filter affected modules.

Poly-repo (each module in its own repo): Natural access control (repo permissions). Simple CI (per-repo). But cross-module changes require cascading PRs. Version pinning needed. Refactoring across modules is painful.

Recommendation for most teams: Start with mono-repo + SPM local packages. This gives you the benefits of modularization (build isolation, code ownership) without the overhead of managing multiple repos. Move to poly-repo only if you have teams in different organizations or need strict access control.

Google, Meta, and Uber use mono-repos. Most mid-size companies use poly-repos. The choice depends more on tooling investment capacity than technical merit.

36. How do you prevent circular dependencies between feature modules? L6

Three strategies:

  1. Interface module pattern: Features depend on each other's interface modules (protocols only), not implementations. FeatureHome depends on FeatureSearchInterface, and FeatureSearch depends on FeatureHomeInterface.
  2. Event bus / delegation to parent: Features don't depend on each other at all. They emit events or delegate to the parent coordinator, which routes between features.
  3. Shared domain module: Extract shared types into a DomainModels module that both features depend on. The shared module has no feature-specific logic.

Enforcement: Define module layers (Foundation → Domain → Infrastructure → Features → App). CI checks that features never import other features directly. SPM's dependency resolution will fail on circular dependencies, providing compile-time enforcement.

37. How do dynamic frameworks affect app launch time? What's Apple's recommendation? Senior

Each dynamic framework adds ~2-5ms to launch time because dyld must: find the framework, map it into memory, perform symbol binding (rebasing and binding), and run initializers. With 50 dynamic frameworks, that's 100-250ms just for framework loading.

Apple's recommendation: Keep dynamic frameworks to ≤6 for fast launch. Use static linking for everything else.

Xcode 15 Mergeable Libraries: The best solution. During development, libraries are dynamic (fast incremental builds). For release, Xcode merges them into the main binary (static linking behavior, fast launch). Set "Create Merged Binary" in build settings.

Measurement: Use the DYLD_PRINT_STATISTICS environment variable or Instruments' App Launch template to measure dyld time.

38. How do you handle resources (images, strings, xibs) in SPM packages? Senior

SPM packages access resources via Bundle.module — an auto-generated accessor for the package's resource bundle.

In Package.swift, declare resources in the target: .target(name: "DesignSystem", resources: [.process("Resources")]). SPM processes resources (optimizes images, compiles strings) or copies them verbatim (.copy).

Access in code: UIImage(named: "icon", in: .module, with: nil) or Image("icon", bundle: .module) in SwiftUI.

Gotcha: Bundle.module only exists in SPM targets. If you have code shared between an SPM package and a non-SPM target, you need a bundle abstraction. Also, Bundle.module is generated at compile time — it won't exist in Objective-C code without bridging.

Dependency Management (Questions 39–44)

39. Explain SPM's PubGrub resolution algorithm at a high level. What happens when resolution fails? L6

PubGrub is a conflict-driven dependency resolver. It works by:

  1. Starting with the root package's dependency constraints
  2. Selecting the highest compatible version for each package
  3. Fetching each selected version's Package.swift to discover transitive dependencies
  4. Computing the intersection of all constraints for each package
  5. If a conflict is found (no version satisfies all constraints), PubGrub backtracks and tries alternative versions
  6. If no solution exists, it produces a human-readable "derivation tree" explaining the conflict chain

When resolution fails: PubGrub produces an error like "Because LibA 2.0 depends on LibC 3.x, and LibB 1.0 depends on LibC 2.x, and your app depends on both LibA 2.0 and LibB 1.0, version solving failed." This derivation tree helps you understand which constraints conflict.

Resolution strategies: Widen constraints (change exact to range), update the conflicting dependency, fork and patch, or replace the dependency entirely.

40. What is the difference between `pod install` and `pod update`? When should you use each? Senior

pod install: Respects Podfile.lock. For existing pods, uses the locked versions. Only resolves NEW pods not in Podfile.lock. This ensures reproducible builds — every developer and CI gets the same versions.

pod update: Ignores Podfile.lock. Resolves ALL dependencies to the latest compatible versions. Writes a new Podfile.lock.

pod update SomePod: Only updates SomePod (and its dependencies). Keeps all other pods at locked versions.

Rules:

  • Use pod install for CI, new team members, and after pulling changes
  • Use pod update only when intentionally upgrading dependencies
  • Use pod update SomePod to update a single dependency
  • Always commit Podfile.lock to source control
41. What is an XCFramework? How does BUILD_LIBRARY_FOR_DISTRIBUTION enable module stability? L6

An XCFramework bundles pre-compiled binaries for multiple platforms (iOS device, simulator, macOS) and architectures (arm64, x86_64) in a single distributable artifact. It replaced "fat/universal" frameworks that couldn't contain both device and simulator arm64 slices.

BUILD_LIBRARY_FOR_DISTRIBUTION=YES generates .swiftinterface files — textual descriptions of the module's public API in a stable format. These enable module stability: a framework compiled with Swift 5.8 can be used by an app compiled with Swift 5.9+.

Without this flag, the binary contains .swiftmodule files that are tied to the exact Swift compiler version. Upgrading Xcode would require recompiling all binary dependencies.

Tradeoff: Module stability restricts what you can expose publicly (no @inlinable functions that use internal types, no @frozen enums with internal cases). It also slightly increases binary size due to the textual interface files.

42. How would you design a dependency management strategy for an enterprise iOS app with 50+ developers? Architect

Governance:

  • Approved list: Security team reviews each dependency for vulnerabilities, license compatibility (no GPL in proprietary apps), and maintenance health.
  • SBOM (Software Bill of Materials): Track all dependencies for compliance (SOC2, HIPAA). Enables vulnerability scanning.
  • Private mirror: Mirror approved packages internally. Protects against upstream deletion and enables air-gapped builds.

Technical:

  • SPM for all new dependencies. Migrate CocoaPods dependencies to SPM over time.
  • Version pinning: Use from: for internal packages (trust your own semver), exact: for critical external dependencies.
  • Binary caching: Pre-build dependencies for CI using Tuist Cloud or custom Artifactory. Saves 5-15 min per build.
  • Automated updates: Renovate/Dependabot for patch/minor updates. Manual review for major updates.

Principle: Every dependency is a liability. Before adding one, ask: Can we implement this in <200 lines? Is this a core competency we should own? What's the transitive dependency cost?

43. How do you migrate from CocoaPods to SPM in a large project? L6

Migrate incrementally, not all at once:

  1. Audit: List all pods, check SPM availability, identify pods with no SPM support (plan alternatives or forks).
  2. Parallel setup: Add SPM packages alongside CocoaPods. Both systems resolve independently. Verify no symbol conflicts.
  3. Migrate leaf dependencies first: Start with dependencies that have no dependents in your codebase. Remove from Podfile, add to Package.swift.
  4. Migrate core dependencies: Networking, storage, analytics. These affect many modules — test thoroughly.
  5. Remove CocoaPods: Delete Podfile, Podfile.lock, Pods/, .xcworkspace. Update CI scripts.

Gotchas: Some pods use CocoaPods-specific features (subspecs, build phase scripts) that don't translate directly to SPM. Resource bundles work differently (Bundle.module vs pod bundle). ObjC pods may need bridging headers.

44. What is the diamond dependency problem? How does SPM handle it? Senior

The diamond dependency problem occurs when: App depends on LibA and LibB. LibA depends on LibC 1.x. LibB depends on LibC 2.x. The dependency graph forms a diamond shape with LibC at the bottom.

SPM's approach: SPM requires a single version of each package that satisfies ALL constraints. If LibA says from: "1.0.0" and LibB says from: "2.0.0", SPM tries to find a version ≥2.0.0 that also satisfies ≥1.0.0. Since 2.x satisfies both (2.0.0 ≥ 1.0.0), it resolves to 2.x.

When it fails: If LibA says from: "1.0.0" (meaning 1.0.0..<2.0.0) and LibB says from: "2.0.0", there's no version that's both <2.0.0 and ≥2.0.0. Resolution fails.

Solutions: Update LibA to support LibC 2.x. Fork LibA and widen its constraint. Replace LibA with an alternative. Or vendor LibC's source directly.

Large-Scale Patterns (Questions 45–55)

45. How do you implement feature flags that support gradual rollout and instant kill switches? L6

Gradual rollout: Use deterministic bucketing based on user ID hash. hash(userId + flagName) % 100 < rolloutPercentage. This ensures the same user always gets the same result (no flickering), and you can increase the percentage gradually (1% → 5% → 25% → 100%).

Kill switches: Ops flags that default to "enabled" and can be remotely disabled. The flag evaluation priority is: debug override > remote config > local cache > default value. Kill switches use the remote config path with aggressive polling (every 5 minutes) or push-based updates.

Implementation: A FeatureFlagProvider protocol with isEnabled(_:) and observe(_:) (AsyncStream for real-time updates). The provider checks remote config first, falls back to cached values, then defaults. Feature code wraps functionality in if featureFlags.isEnabled(.newCheckout).

Hygiene: Release flags must be removed within 2 sprints of full rollout. Track flag age in a dashboard. Alert when flags exceed their expected lifetime. Every flag is technical debt.

46. Design an analytics abstraction layer that supports multiple providers and can be tested. L6

Architecture: Three layers — event definitions (type-safe enums), an analytics service (fan-out to providers), and concrete providers (Firebase, Amplitude, etc.).

Event definitions: Use an enum conforming to AnalyticsEvent protocol with name and parameters properties. This gives compile-time safety — you can't track an event with the wrong parameters.

Analytics service: Takes an array of AnalyticsProvider and fans out each event to all providers. Supports middleware (enrichment, filtering, sampling).

Testing: Inject a MockAnalyticsProvider that records all tracked events. In tests, assert that specific events were tracked with correct parameters.

Key design decisions: Events are value types (Sendable). The service is thread-safe. Events are queued and batched for network efficiency. The abstraction layer means switching from Firebase to Amplitude requires changing one line in the composition root.

47. Explain the difference between os_log levels (debug, info, default, error, fault). When do logs persist to disk? Senior

Debug (.debug): Memory only. Not persisted. Visible only when attached to debugger or using log stream. Use for development-time details.

Info (.info): Memory only by default. Persisted only during log collect (when actively capturing). Use for helpful but non-essential information.

Notice/Default (.default): Persisted to disk. Survives app restart. Use for normal operations worth noting (user actions, state transitions).

Error (.error): Persisted to disk. Use for recoverable errors (network timeout, invalid input).

Fault (.fault): Persisted to disk with stack trace capture. Use for unrecoverable errors and bugs (assertion failures, impossible states).

Privacy: Dynamic values are redacted by default in release builds. Use privacy: .public for values safe to log, privacy: .private(mask: .hash) for identifiers you need to correlate without exposing.

48. How do you design a crash reporting system that provides useful context without leaking PII? L6

Breadcrumbs: Record navigation events, user actions, and state transitions as breadcrumbs. These provide context for crash reports without containing user data. E.g., "Navigated to ProductDetail", "Tapped Add to Cart", "Network request failed (status 500)".

Custom keys: Set non-PII metadata: app version, build number, OS version, device model, feature flags, experiment assignments. These help reproduce the crash.

User identifier: Set an opaque user ID (not email or name) so you can count affected users and correlate crashes. Clear it on logout.

PII protection: Never include user input (search queries, messages) in crash metadata. Never include authentication tokens. Use hashed identifiers where correlation is needed. Comply with GDPR — provide a way to delete crash data for a specific user.

Abstraction: Use a CrashReporter protocol so you can swap providers (Crashlytics, Sentry, Bugsnag) without touching feature code. The abstraction also enables a mock reporter for tests.

49. What is the role of a platform team in a 20+ team iOS organization? Architect

The platform team builds and maintains the internal SDKs that all feature teams consume. Their output is developer productivity — every hour they save across 20 teams compounds massively.

Responsibilities:

  • Networking SDK: Shared HTTP client with auth, retry, caching, certificate pinning
  • Analytics SDK: Event tracking, user properties, provider abstraction
  • Feature Flags: Flag evaluation, gradual rollout, kill switches
  • Logging: Structured logging, log aggregation, privacy compliance
  • Crash Reporting: Symbolication, alerting, breadcrumbs
  • Design System: Shared components, tokens, accessibility
  • Build Infrastructure: CI/CD, build caching, module graph optimization

SDK design principles: Zero configuration (one-line init), protocol-based (feature teams depend on interfaces), fail-safe (SDK failures never crash the app), testable (mock implementations provided), and observable (health metrics exposed).

50. How do you implement A/B testing without creating unmaintainable code? L6

Architecture: Experiments are defined server-side with variants and weights. The client SDK assigns users to variants deterministically (hash-based bucketing). Feature code checks the variant and renders accordingly.

Avoiding unmaintainable code:

  • Coordinator-level routing: A/B test entire flows at the coordinator level, not inside views. The CoordinatorFactory creates variant A or variant B coordinator.
  • Experiment lifecycle: Every experiment has a defined end date. When concluded, the winning variant becomes the default and the losing variant's code is deleted.
  • Maximum concurrent experiments: Limit to 3-5 experiments per feature area. More creates combinatorial explosion.
  • Exposure logging: Automatically log when a user is exposed to a variant (sees the UI). This is critical for statistical validity.
  • Clean separation: Each variant should be a separate View/ViewModel, not if/else branches within one view.
51. How do you design a remote config system that's resilient to network failures? L6

Layered fallback: Remote config → Cached config (disk) → Bundled defaults (compiled into app).

Fetch strategy: Fetch on app launch (non-blocking). Cache to disk. Activate on next launch (fetch-then-activate pattern) to avoid UI flicker. For critical configs, activate immediately with a loading state.

Staleness handling: Cache configs with a TTL (e.g., 12 hours). If the cache is stale and network is unavailable, use stale cache (better than defaults). Log staleness for monitoring.

Validation: Validate fetched config against a schema before caching. Reject malformed configs to prevent corrupted state.

Rollback: Server-side config changes should be reversible. If a bad config ships, the server can push the previous version. The client's aggressive polling (every 5 minutes for critical configs) ensures quick recovery.

52. How do you ensure feature flag hygiene? What happens when flags accumulate? Senior

When flags accumulate: Code becomes littered with if/else branches. Test matrix explodes (N flags = 2^N combinations). Dead code paths remain. New developers can't tell which code paths are active.

Hygiene practices:

  • Expiration dates: Every release flag has a removal deadline (2 sprints after full rollout)
  • Flag audit dashboard: Track flag age, rollout percentage, and owner
  • Automated alerts: Notify the flag owner when it exceeds its expected lifetime
  • CI enforcement: Lint rule that flags older than X days generate warnings
  • Categorization: Release flags (temporary), experiment flags (temporary), ops flags (permanent). Only ops flags are allowed to be permanent.
  • Removal PRs: When removing a flag, delete both code paths (the flag check and the disabled path)
53. Design the platform infrastructure for a company with 20+ iOS feature teams shipping weekly. Architect

Module architecture: Mono-repo with 40-60 SPM packages. Each team owns 2-3 feature modules. Platform team owns 8-10 infrastructure modules. Strict layer enforcement via CI.

Build system: Bazel or Tuist with remote caching. Incremental builds <30 seconds. CI runs only affected module tests. Full test suite runs nightly.

Release process: Trunk-based development. All code ships behind feature flags. Release train every week (automated). Feature teams control their own rollout via feature flags. Kill switches for every critical path.

Platform SDK: PlatformInterfaces (protocols, consumed by features), PlatformSDK (implementations, consumed by app target), PlatformMocks (test doubles, consumed by test targets).

Observability: Structured logging with os_log. Crash reporting with breadcrumbs. Analytics with type-safe events. Performance monitoring with MetricKit. All accessible via a unified dashboard.

Developer experience: New developer productive in 2 weeks. Module-level documentation. Architecture decision records (ADRs). Automated code generation for boilerplate (Sourcery/Swift macros).

54. How do you handle backward compatibility when your internal SDK has 20+ consumer teams? Architect

Semantic versioning: Strictly follow semver. Breaking changes = major version bump. New features = minor. Bug fixes = patch.

Deprecation policy: Deprecated APIs remain for 2 major versions. Mark with @available(*, deprecated, message: "Use newAPI instead"). Compiler warnings guide migration.

Migration guides: Every major version includes a migration guide with before/after code examples and a codemod script where possible.

Canary releases: New SDK versions are tested by 1-2 volunteer teams before wide rollout. This catches integration issues early.

API review process: All public API changes go through a review board. Once published, APIs are contracts. Changing them affects 20+ teams.

Feature flags for SDK features: New SDK capabilities can be gated behind flags, allowing gradual adoption.

55. You're joining a company with a 1.5M line iOS codebase, no modularization, 15-minute build times, and 30 developers. What's your first 90-day plan? Architect

Days 1-30: Assess and Quick Wins

  • Map the codebase: identify natural module boundaries, hotspot files (most edited), and dependency clusters
  • Measure: build times (clean/incremental), test suite duration, CI queue times
  • Quick win: Extract CoreUtils (pure extensions/helpers) into an SPM package. Zero risk, immediate build time improvement for that module.
  • Quick win: Enable Xcode's "Parallelize Build" and "Build Active Architecture Only" for debug

Days 30-60: Foundation

  • Extract DomainModels (data models + repository protocols) into SPM package
  • Extract DesignSystem (shared UI components) into SPM package
  • Set up module dependency rules and CI enforcement
  • Write Architecture Decision Record (ADR) for the modularization strategy

Days 60-90: First Feature Module

  • Extract the most independent feature into its own SPM package (proof of concept)
  • Measure build time improvement
  • Document the extraction process as a playbook for other teams
  • Present results to engineering leadership with a roadmap for remaining modules

Key principle: Show value early. The first extracted module proves the approach works and builds organizational buy-in for the larger effort.
