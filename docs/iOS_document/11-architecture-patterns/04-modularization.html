1. Why Modularize

Modularization is the single most impactful architectural decision for large iOS codebases. It affects build time, team autonomy, code ownership, testability, and app launch time.

1.1 Problems Modularization Solves

ProblemMonolithic AppModularized App
Build time15-30 min full build2-5 min (only changed modules)
Team conflictsConstant merge conflictsTeams own separate modules
Code ownershipUnclear ("who owns this?")CODEOWNERS per module
TestingRun all tests (40 min)Run affected module tests (3 min)
OnboardingUnderstand entire codebaseUnderstand your module + interfaces
ReuseCopy-paste between targetsImport shared module
Access controlEverything is internalExplicit public API surface

2. Swift Package Manager Modules

2.1 Local Package Structure

// Package.swift for a local module
// swift-tools-version: 5.9
import PackageDescription

let package = Package(
    name: "FeatureHome",
    platforms: [.iOS(.v17)],
    products: [
        .library(name: "FeatureHome", targets: ["FeatureHome"]),

        .target(
            name: "FeatureHome",
            dependencies: [
                "DomainModels",

                "DesignSystem",

            dependencies: ["FeatureHome"],
            path: "Tests"

3. Framework vs Static Library vs Dynamic Library

3.1 Comparison

Aspect Static Library (.a) Dynamic Library (.dylib) Framework (.framework)
Linking Copied into binary at build time Loaded at runtime by dyld Bundle containing dylib + headers + resources
App size Larger (code duplicated) Smaller (shared) Depends on static/dynamic
Launch time Faster (no runtime loading) Slower (dyld must load each) Depends on type
Resources Cannot contain resources Cannot contain resources Can contain resources (images, xibs)
Swift modules Yes (.swiftmodule) Yes Yes (bundled)
SPM default Yes (automatic = static) No No (use .dynamic explicitly)
SPM Library Types

SPM's .library(type:) can be .static, .dynamic, or omitted (automatic). When automatic, SPM decides based on the dependency graph. For most local packages, static linking is preferred — it produces faster launch times and avoids the dyld overhead of loading many dynamic frameworks.

3.2 Launch Time Impact

// dyld loads dynamic frameworks at launch
// Each dynamic framework adds ~2-5ms to launch time
// 50 dynamic frameworks = 100-250ms just for loading
//
// Apple's recommendation: ≤ 6 dynamic frameworks for fast launch
//
// Static libraries: zero launch time cost (already in binary)
// But: larger binary size, no code sharing between app + extensions

// Xcode 15+ Mergeable Libraries: best of both worlds
// Build as dynamic for development (fast incremental builds)
// Merge into static for release (fast launch)

4. @_exported import

@_exported import is an underscored (unofficial) Swift attribute that re-exports a module's symbols. When module A does @_exported import B, anyone who imports A also gets B's symbols without explicitly importing B.

// In DesignSystem module:
@_exported import SwiftUI  // Anyone importing DesignSystem also gets SwiftUI

// In FeatureHome:
import DesignSystem  // Also gets SwiftUI for free

// Umbrella module pattern:
// CoreKit/Sources/CoreKit.swift
@_exported import DomainModels
@_exported import CoreUtils
@_exported import DesignSystem
@_exported Risks

@_exported is not officially supported and can break between Swift versions. It also creates hidden dependencies — a module appears to not need an import, but it actually does through a transitive re-export. Use sparingly and only for umbrella modules where the intent is explicit.

5. Module Boundaries & API Surface

5.1 Access Control as Module Boundary

// Swift access levels enforce module boundaries
public    // Visible to other modules — your API surface
package   // Visible within the same package (Swift 5.9+)
internal  // Visible within the module (default)
fileprivate // Visible within the file
private   // Visible within the enclosing declaration

// Module boundary example
// DomainModels module:
public struct User: Sendable, Equatable {
    public let id: String
    public let name: String
    public let email: String

    public init(id: String, name: String, email: String) {
        self.id = id
        self.name = name
        self.email = email
    }
}

public protocol UserRepository: Sendable {
    func fetchUser(id: String) async throws -> User
    func searchUsers(query: String) async throws -> [User]
}

// NetworkLayer module — implements the protocol
public final class APIUserRepository: UserRepository {
    private let client: APIClient  // internal — not exposed

    public init(client: APIClient) {
        self.client = client
    }

    public func fetchUser(id: String) async throws -> User {
        let dto: UserDTO = try await client.request(.user(id: id))
        return dto.toDomain()
    }
}

5.2 The package Access Level (Swift 5.9+)

// Package.swift with multiple targets sharing package access
let package = Package(
    name: "FeatureCheckout",
    targets: [
        .target(name: "CheckoutUI", dependencies: ["CheckoutDomain"]),

// In CheckoutDomain:
package struct CartCalculation {
    // Visible to CheckoutUI (same package) but NOT to other packages
    package func calculateTotal(items: [CartItem]) -> Decimal { ... }

6. API Design Guidelines for Internal Modules

6.1 Module Interface Contract

// Good: Protocol-based module interface
// FeatureHome exposes only what's needed
public protocol HomeModuleInterface {
    func makeHomeViewController() -> UIViewController
    func makeHomeView() -> AnyView
}

public final class HomeModule: HomeModuleInterface {
    private let dependencies: HomeDependencies

    public init(dependencies: HomeDependencies) {
        self.dependencies = dependencies
    }

    public func makeHomeViewController() -> UIViewController {
        let vm = HomeViewModel(service: dependencies.productService)
        return HomeViewController(viewModel: vm)
    }

    public func makeHomeView() -> AnyView {
        let vm = HomeViewModel(service: dependencies.productService)
        return AnyView(HomeView(viewModel: vm))
    }
}

// Dependencies struct — explicit about what the module needs
public struct HomeDependencies {
    public let productService: ProductServiceProtocol
    public let analyticsService: AnalyticsProtocol
    public let featureFlags: FeatureFlagProtocol

    public init(
        productService: ProductServiceProtocol,
        analyticsService: AnalyticsProtocol,
        featureFlags: FeatureFlagProtocol
    ) {
        self.productService = productService
        self.analyticsService = analyticsService
        self.featureFlags = featureFlags
    }
}

7. Tuist, XcodeGen, and Bazel

7.1 Tuist

Tuist generates Xcode projects from Swift manifests. It eliminates .xcodeproj merge conflicts and provides build graph optimization.

// Tuist/Project.swift
import ProjectDescription

let project = Project(
    name: "MyApp",
    targets: [
        .target(
            name: "MyApp",
            destinations: .iOS,
            product: .app,
            bundleId: "com.example.myapp",
            sources: ["App/Sources/**"],
            dependencies: [
                .project(target: "FeatureHome", path: "Modules/FeatureHome"),
                .project(target: "FeatureSearch", path: "Modules/FeatureSearch"),
                .project(target: "DesignSystem", path: "Modules/DesignSystem"),
            ]
        ),
    ]
)

7.2 XcodeGen

XcodeGen uses YAML to define projects. Simpler than Tuist but less powerful.

# project.yml
name: MyApp
targets:
  MyApp:
    type: application
    platform: iOS
    sources: App/Sources
    dependencies:
      - target: FeatureHome
      - target: DesignSystem
    settings:
      SWIFT_VERSION: "5.9"

  FeatureHome:
    type: framework
    platform: iOS
    sources: Modules/FeatureHome/Sources
    dependencies:
      - target: DomainModels
      - target: NetworkLayer

7.3 Bazel for iOS

Bazel (used by Google, Uber, Lyft) provides hermetic builds, remote caching, and remote execution. It's the most powerful option but has the steepest learning curve.

# BUILD.bazel for an iOS module
load("@build_bazel_rules_apple//apple:ios.bzl", "ios_application")
load("@build_bazel_rules_swift//swift:swift.bzl", "swift_library")

swift_library(
    name = "FeatureHome",
    srcs = glob(["Sources/**/*.swift"]),
    deps = [
        "//Modules/DomainModels",
        "//Modules/NetworkLayer",
        "//Modules/DesignSystem",
    ],
    module_name = "FeatureHome",
    visibility = ["//visibility:public"],
)

swift_test(
    name = "FeatureHomeTests",
    srcs = glob(["Tests/**/*.swift"]),
    deps = [":FeatureHome"],
)

8. Mono-repo vs Poly-repo

8.1 Mono-repo

All code in one repository. Used by Google, Meta, Uber.

mono-repo/ ├── ios-app/ │ ├── App/ │ ├── Modules/ │ │ ├── FeatureHome/ │ │ ├── FeatureSearch/ │ │ ├── NetworkLayer/ │ │ └── DesignSystem/ │ └── Package.swift ├── android-app/ ├── backend/ ├── shared-protos/ └── tools/ ├── linters/ └── ci-scripts/

8.2 Poly-repo

Each module in its own repository. Used by many mid-size companies.

github.com/company/ ├── ios-app (main app repo) ├── ios-design-system (SPM package) ├── ios-networking (SPM package) ├── ios-analytics (SPM package) ├── ios-feature-home (SPM package) └── ios-feature-search (SPM package)

8.3 Tradeoff Analysis

DimensionMono-repoPoly-repo
Atomic changesEasy (one PR changes all modules)Hard (cascading PRs)
Dependency managementAlways latest (source of truth)Version pinning needed
Build toolingComplex (Bazel/Tuist needed)Simple (SPM resolves)
CI timeLong without smart filteringShort (per-repo CI)
Code discoveryEasy (grep everything)Hard (search across repos)
Access controlNeeds CODEOWNERSNatural (repo permissions)
RefactoringEasy (change + dependents in one PR)Painful (version bumps)

9. Module Dependency Graph Management

9.1 Dependency Graph Rules

The Diamond Dependency Problem

If module A depends on B and C, and both B and C depend on D but at different versions, you have a diamond dependency. SPM resolves this by requiring a single version of D that satisfies both B and C's constraints. If no such version exists, resolution fails.

// Enforce no circular dependencies with a build script
// This runs in CI to verify the dependency graph is a DAG

// Module layer rules:
// Layer 0 (Foundation): CoreUtils, Extensions
// Layer 1 (Domain): DomainModels, DomainProtocols
// Layer 2 (Infrastructure): NetworkLayer, StorageLayer, AnalyticsSDK
// Layer 3 (Shared UI): DesignSystem, UIComponents
// Layer 4 (Features): FeatureHome, FeatureSearch, FeatureCart
// Layer 5 (App): MainApp (composition root)
//
// Rule: Layer N can only depend on layers 0...(N-1)
// Features (Layer 4) NEVER depend on other features

10. Architect: Modularization for 2M+ Lines

Architect Must-Know: Modularization Strategy for 2M+ Line Codebases

At 2M+ lines, modularization is not optional — it's survival. Without it, build times exceed 30 minutes, merge conflicts are constant, and no single developer can understand the codebase. The architect must design a module graph that enables 50+ developers to work independently.

10.1 Migration Strategy

// Phase 1: Extract Foundation Layer (weeks 1-4)
// Move pure utilities, extensions, and helpers into CoreUtils
// Zero dependencies on app code

// Phase 2: Extract Domain Models (weeks 4-8)
// Move all data models and repository protocols
// Depends only on Foundation and CoreUtils

// Phase 3: Extract Infrastructure (weeks 8-16)
// Network layer, storage, analytics
// Implements domain protocols

// Phase 4: Extract Design System (weeks 8-16, parallel with Phase 3)
// Shared UI components, colors, typography

// Phase 5: Extract Features (weeks 16-52)
// One feature at a time, starting with the most independent
// Each feature is a separate SPM package

// Phase 6: Enforce Boundaries (ongoing)
// CI checks for layer violations
// CODEOWNERS per module
// Module API review process

Best Practice: The Interface Module Pattern

For each feature module, create a companion interface module containing only protocols and data types. Feature modules depend on each other's interfaces, never implementations. This breaks circular dependencies and enables independent compilation.
