1. SPM Internals

1.1 Package.swift Manifest

The Package.swift manifest is a Swift script interpreted by the Swift Package Manager. It uses the PackageDescription library to declare the package's name, platforms, products, dependencies, and targets.

// swift-tools-version: 5.9
import PackageDescription

let package = Package(
    name: "NetworkKit",
    platforms: [
        .iOS(.v16),
        .macOS(.v13),
    ],
    products: [
        .library(
            name: "NetworkKit",
            type: .static,
            targets: ["NetworkKit"]
        ),
    ],
    dependencies: [
        .package(
            url: "https://github.com/Alamofire/Alamofire.git",
            from: "5.8.0"
        ),
        .package(
            url: "https://github.com/apple/swift-log.git",
            .upToNextMinor(from: "1.5.0")
        ),
        .package(
            url: "https://github.com/company/internal-sdk.git",
            exact: "2.3.1"
        ),
    ],
    targets: [
        .target(
            name: "NetworkKit",
            dependencies: [
                .product(name: "Alamofire", package: "Alamofire"),
                .product(name: "Logging", package: "swift-log"),
            ]
        ),
        .testTarget(
            name: "NetworkKitTests",
            dependencies: ["NetworkKit"]
        ),
    ]
)

1.3 PubGrub Resolution Algorithm

SPM uses the PubGrub algorithm (developed by Natalie Weizenbaum for Dart's pub) for dependency resolution. PubGrub is a conflict-driven solver that provides clear error messages when resolution fails.

┌─────────────────────────────────────────────────────────────┐ │ PubGrub Resolution Process │ │ │ │ 1. Start with root package's dependencies │ │ │ │ 2. For each dependency, fetch available versions │ │ (git tags matching semver) │ │ │ │ 3. Select highest compatible version for each package │ │ │ │ 4. Fetch selected version's Package.swift │ │ → Discover transitive dependencies │ │ │ │ 5. Check for conflicts: │ │ App needs Alamofire ≥5.8 AND ≥5.9 │ │ → Resolve to ≥5.9 (intersection) │ │ │ │ 6. If conflict is unsolvable: │ │ App needs LibA 1.x, LibA needs LibB 2.x │ │ App needs LibB 1.x │ │ → CONFLICT: LibB needs to be both 1.x and 2.x │ │ → PubGrub backtracks and tries different versions │ │ │ │ 7. If no solution exists, PubGrub produces a │ │ human-readable "derivation tree" explaining why │ │ │ │ 8. Write resolved versions to Package.resolved │ └─────────────────────────────────────────────────────────────┘

1.4 Package.resolved

// Package.resolved — locks exact versions for reproducible builds
// MUST be committed to source control
{
  "pins" : [
    {
      "identity" : "alamofire",
      "kind" : "remoteSourceControl",
      "location" : "https://github.com/Alamofire/Alamofire.git",
      "state" : {
        "revision" : "f455c2975872ccd2d9c81594c658af65716e9b9a",
        "version" : "5.9.1"
      }
    },
    {
      "identity" : "swift-log",
      "kind" : "remoteSourceControl",
      "location" : "https://github.com/apple/swift-log.git",
      "state" : {
        "revision" : "e97a6fcb1ab07462881ac165fdbb37f067e205d5",
        "version" : "1.5.4"
      }
    }
  ],
  "version" : 3
}
Package.resolved vs Podfile.lock

Both serve the same purpose — locking dependency versions for reproducible builds. The key difference: Package.resolved is generated by SPM and uses JSON format with git revision hashes. Podfile.lock is generated by CocoaPods and uses YAML. Both should be committed to source control.

2. CocoaPods Deep Dive

CocoaPods is a Ruby-based dependency manager that has been the iOS ecosystem's dominant tool since 2011. While SPM is replacing it for new projects, millions of existing projects still use CocoaPods.

2.1 Podspec Anatomy

# NetworkKit.podspec
Pod::Spec.new do |s|
  s.name             = 'NetworkKit'
  s.version          = '2.3.1'
  s.summary          = 'Networking layer for iOS apps'
  s.homepage         = 'https://github.com/company/NetworkKit'
  s.license          = { :type => 'MIT', :file => 'LICENSE' }
  s.author           = { 'Company' => 'ios@company.com' }
  s.source           = { :git => 'https://github.com/company/NetworkKit.git',
                          :tag => s.version.to_s }

  s.ios.deployment_target = '16.0'
  s.swift_version = '5.9'

  s.source_files = 'Sources/**/*.swift'
  s.resource_bundles = {
    'NetworkKit' => ['Sources/**/*.{xib,storyboard,xcassets}']
  }

  s.dependency 'Alamofire', '~> 5.8'

  s.subspec 'Core' do |core|
    core.source_files = 'Sources/Core/**/*.swift'
  end

  s.subspec 'Mocking' do |mock|
    mock.source_files = 'Sources/Mocking/**/*.swift'
    mock.dependency 'NetworkKit/Core'
  end

  s.test_spec 'Tests' do |test|
    test.source_files = 'Tests/**/*.swift'
    test.dependency 'NetworkKit/Mocking'
  end
end

2.2 pod install vs pod update

┌─────────────────────────────────────────────────────────────┐ │ pod install vs pod update │ │ │ │ pod install: │ │ ┌─────────────────────────────────────────────────────┐ │ │ │ 1. Read Podfile │ │ │ │ 2. If Podfile.lock exists: │ │ │ │ → Use locked versions for existing pods │ │ │ │ → Only resolve NEW pods not in Podfile.lock │ │ │ │ 3. If Podfile.lock doesn't exist: │ │ │ │ → Resolve all dependencies from scratch │ │ │ │ 4. Download pods to Pods/ directory │ │ │ │ 5. Generate Pods.xcodeproj │ │ │ │ 6. Update Podfile.lock │ │ │ └─────────────────────────────────────────────────────┘ │ │ │ │ pod update: │ │ ┌─────────────────────────────────────────────────────┐ │ │ │ 1. Read Podfile │ │ │ │ 2. IGNORE Podfile.lock │ │ │ │ 3. Resolve ALL dependencies to latest compatible │ │ │ │ 4. Download updated pods │ │ │ │ 5. Regenerate Pods.xcodeproj │ │ │ │ 6. Write new Podfile.lock │ │ │ └─────────────────────────────────────────────────────┘ │ │ │ │ pod update Alamofire: │ │ → Only update Alamofire (and its dependencies) │ │ → Keep all other pods at locked versions │ │ │ │ Rule: Use `pod install` for CI and team members │ │ Use `pod update` only when intentionally upgrading │ └─────────────────────────────────────────────────────────────┘

3. Carthage

Carthage is a decentralized dependency manager that builds frameworks but doesn't modify your Xcode project. It follows the Unix philosophy: do one thing well.

3.1 How Carthage Works

┌─────────────────────────────────────────────────────────────┐ │ Carthage Build Process │ │ │ │ 1. Read Cartfile │ │ github "Alamofire/Alamofire" ~> 5.8 │ │ github "company/NetworkKit" "main" │ │ │ │ 2. Resolve versions → write Cartfile.resolved │ │ │ │ 3. Clone repos to Carthage/Checkouts/ │ │ │ │ 4. Build frameworks: │ │ carthage build --platform iOS --use-xcframeworks │ │ → xcodebuild each dependency │ │ → Output to Carthage/Build/ │ │ │ │ 5. YOU manually: │ │ → Drag .xcframework into Xcode project │ │ → Add to "Frameworks, Libraries, and Embedded Content" │ │ │ │ Key difference from CocoaPods: │ │ Carthage NEVER modifies your .xcodeproj │ │ You have full control over project configuration │ └─────────────────────────────────────────────────────────────┘

4. Binary Frameworks (XCFramework)

XCFrameworks (Xcode 11+) bundle pre-compiled binaries for multiple platforms and architectures in a single distributable artifact. They solve the "fat binary" problem that plagued universal frameworks.

4.1 XCFramework Structure

NetworkKit.xcframework/ ├── Info.plist ├── ios-arm64/ │ └── NetworkKit.framework/ │ ├── NetworkKit (binary) │ ├── Headers/ │ ├── Modules/ │ │ ├── module.modulemap │ │ └── NetworkKit.swiftmodule/ │ │ ├── arm64-apple-ios.swiftinterface │ │ └── arm64-apple-ios.private.swiftinterface │ └── Info.plist ├── ios-arm64_x86_64-simulator/ │ └── NetworkKit.framework/ │ └── ... (simulator binary) └── macos-arm64_x86_64/ └── NetworkKit.framework/ └── ... (macOS binary)

4.2 Creating an XCFramework

# Step 1: Build for each platform
xcodebuild archive \
  -scheme NetworkKit \
  -destination "generic/platform=iOS" \
  -archivePath build/NetworkKit-iOS \
  SKIP_INSTALL=NO \
  BUILD_LIBRARY_FOR_DISTRIBUTION=YES

xcodebuild archive \
  -scheme NetworkKit \
  -destination "generic/platform=iOS Simulator" \
  -archivePath build/NetworkKit-Simulator \
  SKIP_INSTALL=NO \
  BUILD_LIBRARY_FOR_DISTRIBUTION=YES

# Step 2: Create XCFramework
xcodebuild -create-xcframework \
  -archive build/NetworkKit-iOS.xcarchive \
    -framework NetworkKit.framework \
  -archive build/NetworkKit-Simulator.xcarchive \
    -framework NetworkKit.framework \
  -output build/NetworkKit.xcframework

# BUILD_LIBRARY_FOR_DISTRIBUTION=YES generates .swiftinterface files
# These enable module stability — the binary works across Swift versions

4.3 Distributing via SPM

// Package.swift with binary target
let package = Package(
    name: "NetworkKit",
    products: [
        .library(name: "NetworkKit", targets: ["NetworkKit"]),
    ],
    targets: [
        .binaryTarget(
            name: "NetworkKit",
            url: "https://releases.company.com/NetworkKit-2.3.1.xcframework.zip",
            checksum: "a1b2c3d4e5f6..."
        ),
        // Or local path during development:
        // .binaryTarget(name: "NetworkKit", path: "NetworkKit.xcframework")
    ]
)
Module Stability

BUILD_LIBRARY_FOR_DISTRIBUTION=YES generates .swiftinterface files — textual descriptions of the module's public API. These enable module stability: a framework compiled with Swift 5.8 can be used by an app compiled with Swift 5.9+. Without this flag, the binary is tied to the exact Swift compiler version.

5. Version Resolution Conflicts

5.1 Common Conflict Scenarios

// Scenario 1: Diamond dependency with incompatible versions
// App → LibA (needs LibC 1.x)
// App → LibB (needs LibC 2.x)
// Resolution: IMPOSSIBLE — must update LibA or LibB

// Scenario 2: Transitive dependency conflict
// App → Alamofire ~> 5.8
// App → SomeSDK → Alamofire ~> 5.4 (pinned by SDK vendor)
// Resolution: SPM resolves to 5.8.x (satisfies both)
//             IF SomeSDK's constraint is "from: 5.4" not "exact: 5.4"

// Scenario 3: Platform mismatch
// App targets iOS 16+
// LibA requires iOS 17+
// Resolution: SPM error — platform requirement not met

6. Private Registry & Distribution

6.1 SPM Private Registry

// SPM supports package registries (SE-0292, Swift 5.7+)
// Configure in .swiftpm/configuration/registries.json
{
    "registries": {
        "company.com": {
            "url": "https://spm-registry.company.com"
        }
    },
    "version": 1
}

// Reference packages by identity instead of URL
let package = Package(
    dependencies: [
        .package(id: "company.com.network-kit", from: "2.3.0"),
    ]
)

6.2 CocoaPods Private Spec Repo

# Add private spec repo
pod repo add company-specs https://github.com/company/podspecs.git

# Podfile references both public and private sources
source 'https://cdn.cocoapods.org/'
source 'https://github.com/company/podspecs.git'

target 'MyApp' do
  pod 'Alamofire', '~> 5.8'          # Public
  pod 'CompanyNetworkKit', '~> 2.3'   # Private
end

# Push to private spec repo
pod repo push company-specs NetworkKit.podspec --allow-warnings

8. Architect: Enterprise Dependency Strategy

Architect Must-Know: Dependency Management for Enterprise Apps

At enterprise scale, dependency management is a security, compliance, and reliability concern. Every third-party dependency is a supply chain risk. The architect must balance developer productivity (easy to add dependencies) with organizational control (security review, license compliance, version stability).

8.1 Enterprise Dependency Policy

// Enterprise dependency governance:
//
// 1. APPROVED LIST: Maintain a list of approved dependencies
//    - Security team reviews each dependency
//    - License compatibility checked (no GPL in proprietary apps)
//    - Maintenance health assessed (last commit, open issues)
//
// 2. VERSION PINNING: Pin major versions, allow patch updates
//    - from: "5.8.0" allows 5.8.0 ..< 6.0.0
//    - Automated PR for minor/patch updates (Dependabot/Renovate)
//    - Manual review for major updates
//
// 3. PRIVATE MIRROR: Mirror approved packages internally
//    - Protects against upstream deletion (left-pad incident)
//    - Enables air-gapped builds
//    - Adds security scanning layer
//
// 4. BINARY CACHING: Pre-build dependencies for CI
//    - Tuist Cloud or custom Artifactory
//    - Saves 5-15 min per CI build
//
// 5. SBOM (Software Bill of Materials): Track all dependencies
//    - Required for compliance (SOC2, HIPAA)
//    - Enables vulnerability scanning

8.2 Migration Path: CocoaPods → SPM

┌─────────────────────────────────────────────────────────────┐ │ CocoaPods → SPM Migration │ │ │ │ Phase 1: Audit (1-2 weeks) │ │ ├── List all CocoaPods dependencies │ │ ├── Check SPM availability for each │ │ ├── Identify pods with no SPM support │ │ └── Plan for alternatives or forks │ │ │ │ Phase 2: Parallel Setup (1 week) │ │ ├── Add SPM packages alongside CocoaPods │ │ ├── Both systems resolve independently │ │ └── Verify no symbol conflicts │ │ │ │ Phase 3: Migrate Leaf Dependencies (2-4 weeks) │ │ ├── Start with dependencies that have no dependents │ │ ├── Remove from Podfile, add to Package.swift │ │ └── Run full test suite after each migration │ │ │ │ Phase 4: Migrate Core Dependencies (2-4 weeks) │ │ ├── Migrate networking, storage, analytics │ │ └── These affect many modules — test thoroughly │ │ │ │ Phase 5: Remove CocoaPods (1 week) │ │ ├── Delete Podfile, Podfile.lock, Pods/ │ │ ├── Remove .xcworkspace (use .xcodeproj directly) │ │ └── Update CI scripts │ └─────────────────────────────────────────────────────────────┘
Best Practice: Dependency Minimalism

Every dependency is a liability. Before adding one, ask: (1) Can we implement this in <200 lines? (2) Is this a core competency we should own? (3) What's the maintenance health? (4) What's the transitive dependency cost? The best dependency is the one you don't add.
