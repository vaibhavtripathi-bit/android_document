UIKit ↔ SwiftUI Interop

Staff-level prep for bridging UIKit into SwiftUI and SwiftUI into UIKit: representables, coordinators, UIHostingController, sizing semantics, and migration strategy at the scale of large legacy codebases. The goal is not “I wrapped a view once”—it is to explain lifecycle contracts, ownership, and where bugs hide when two layout systems meet.

Framing for interviews

Interop questions often test whether you understand reference semantics (UIKit) vs value-typed descriptions (SwiftUI), and whether you can reason about update scheduling: SwiftUI may call updateUIView frequently; UIKit callbacks must not assume a 1:1 pairing with SwiftUI “renders.”

1. UIViewRepresentable / UIViewControllerRepresentable

What a representable is

A representable is SwiftUI’s adapter protocol for embedding UIKit objects inside a SwiftUI hierarchy. SwiftUI owns the slot in the layout/render graph; UIKit owns the UIView instance’s imperative behavior. Your type’s job is to translate Swift state into UIKit mutations and UIKit events into Swift state—without fighting either framework’s lifecycle.

Protocol UIKit object Typical use
UIViewRepresentable UIView (or subclass) Controls: maps, text views, pickers, complex custom views, third-party UIKit SDKs.
UIViewControllerRepresentable UIViewController System controllers (UIImagePickerController), document pickers, share sheets, storyboard-backed flows.

UIViewRepresentable protocol requirements

The core contract has two required methods. Everything else is optional sugar—until you need delegates, then makeCoordinator() becomes practically required.

  • makeUIView(context:) -> UIViewType: create the view once (per representable instance lifecycle).
  • updateUIView(_ uiView: UIViewType, context: Context): synchronize Swift inputs into UIKit state.

Optional hooks you should know cold:

  • static func dismantleUIView(_ uiView: UIViewType, coordinator: Coordinator): tear down timers, KVO, notification observers, gesture recognizers that outlive the view.
  • func makeCoordinator() -> Coordinator: produce a reference type for delegate/target-action storage.
  • sizeThatFits(_ proposal: ProposedViewSize, uiView: UIViewType, context: Context) -> CGSize? (iOS 16+): opt into SwiftUI-driven measurement for the wrapped view.

UIViewControllerRepresentable protocol requirements

The controller-shaped twin mirrors the same lifecycle split:

  • makeUIViewController(context:) -> UIViewControllerType
  • updateUIViewController(_:context:)
  • Optional: static func dismantleUIViewController(_:coordinator:), makeCoordinator(), and sizing helpers where applicable.
Common mistake

Treating updateUIView as “initialize.” Heavy one-time setup belongs in makeUIView (or the coordinator’s initializer). updateUIView can run often and should be idempotent and cheap enough to repeat.

Context: what you actually get

The Context type is SwiftUI’s bundle of handles for bridging work:

Context surface Why it matters
context.coordinator Your Coordinator reference; stable across updates for delegate callbacks.
context.transaction Animation transaction metadata propagating from SwiftUI (disables animations, duration, etc.).
context.environment Bridged environment values (size classes, layout direction, color scheme)—useful for configuring UIKit appearance.

Lifecycle: when each method is called (mental model)

SwiftUI inserts Representable in hierarchy | +-- makeUIView / makeUIViewController (once per backing instance*) | +-- makeCoordinator (if implemented; early in lifecycle) | +-- updateUIView / updateUIViewController (0..N times; state/env changes) | +-- dismantleUIView / dismantleUIViewController (on removal/teardown) * Identity churn (e.g. unstable .id) can force recreation—treat as expensive.

In interviews, emphasize: SwiftUI drives the cadence. UIKit callbacks (e.g. map delegate) are asynchronous with respect to SwiftUI’s update pass; you coordinate through bindings and main-queue discipline.

Complete example: wrapping MKMapView

This example shows region synchronization SwiftUI → UIKit, and map camera changes UIKit → SwiftUI via MKMapViewDelegate. The coordinator holds the delegate and a weak map reference to avoid retain cycles. 

import SwiftUI
import MapKit

struct MapKitView: UIViewRepresentable {
    @Binding var region: MKCoordinateRegion
    var showsUserLocation: Bool = false

    func makeCoordinator() -> Coordinator {
        Coordinator(self)
    }

    func makeUIView(context: Context) -> MKMapView {
        let map = MKMapView()
        map.delegate = context.coordinator
        map.showsUserLocation = showsUserLocation
        map.setRegion(region, animated: false)
        context.coordinator.mapView = map
        return map
    }

    func updateUIView(_ map: MKMapView, context: Context) {
        map.showsUserLocation = showsUserLocation
        // Avoid feedback loops: only setRegion when SwiftUI region meaningfully changes.
        if !context.coordinator.isDragging,
           !regionsApproximatelyEqual(map.region, region) {
            map.setRegion(region, animated: context.transaction.disablesImplicitAnimation ? false : true)
        }
    }

    static func dismantleUIView(_ map: MKMapView, coordinator: Coordinator) {
        map.delegate = nil
        coordinator.mapView = nil
    }

    final class Coordinator: NSObject, MKMapViewDelegate {
        var parent: MapKitView
        weak var mapView: MKMapView?
        var isDragging = false

        init(_ parent: MapKitView) {
            self.parent = parent
        }

        func mapView(_ mapView: MKMapView, regionWillChangeAnimated animated: Bool) {
            isDragging = true
        }

        func mapView(_ mapView: MKMapView, regionDidChangeAnimated animated: Bool) {
            isDragging = false
            parent.region = mapView.region
        }
    }

    private func regionsApproximatelyEqual(_ a: MKCoordinateRegion, _ b: MKCoordinateRegion) -> Bool {
        abs(a.center.latitude - b.center.latitude) < 0.000_1
        && abs(a.center.longitude - b.center.longitude) < 0.000_1
        && abs(a.span.latitudeDelta - b.span.latitudeDelta) < 0.000_1
        && abs(a.span.longitudeDelta - b.span.longitudeDelta) < 0.000_1
    }
}

Complete example: wrapping UITextView with a delegate

Text views are a classic stress test: attributed strings, caret, scrolling, keyboard, and delegate callbacks. The pattern below keeps one source of truth in SwiftUI (text) while still allowing UIKit to drive typing. 

import SwiftUI
import UIKit

struct AttributedTextEditor: UIViewRepresentable {
    @Binding var text: NSAttributedString
    var isEditable: Bool = true

    func makeCoordinator() -> Coordinator {
        Coordinator(self)
    }

    func makeUIView(context: Context) -> UITextView {
        let tv = UITextView()
        tv.delegate = context.coordinator
        tv.isEditable = isEditable
        tv.isScrollEnabled = true
        tv.backgroundColor = .clear
        tv.attributedText = text
        return tv
    }

    func updateUIView(_ uiView: UITextView, context: Context) {
        uiView.isEditable = isEditable

        // Avoid clobbering the caret while typing: compare value, not object identity.
        if uiView.attributedText.string != text.string {
            uiView.attributedText = text
        }
    }

    final class Coordinator: NSObject, UITextViewDelegate {
        var parent: AttributedTextEditor

        init(_ parent: AttributedTextEditor) {
            self.parent = parent
        }

        func textViewDidChange(_ textView: UITextView) {
            parent.text = textView.attributedText
        }

        func textView(_ textView: UITextView, shouldChangeTextIn range: NSRange, replacementText text: String) -> Bool {
            // Return false to veto edits; return true to allow.
            return true
        }
    }
}
Interview sound bite

“Representables are bridges with a strict lifecycle: create once, update many times, dismantle on removal. Delegates belong in a Coordinator because SwiftUI’s View is a struct and cannot reliably be a UITextViewDelegate.”

2. Coordinator pattern for delegates

Why Coordinator exists

UIKit’s delegate/data-source protocols are overwhelmingly class-constrained reference semantics. SwiftUI views are struct values recreated as the tree is re-evaluated. If you assign textView.delegate = self from inside a struct, you do not get a stable object identity across updates unless you introduce a reference type. The coordinator is that stable object.

Problem Without coordinator With coordinator
Delegate must be a class instance Temptation to use singletons / globals One coordinator per representable instance, owned by SwiftUI’s bridge machinery
Callbacks need “current props” Stale captures from makeUIView closures Coordinator holds parent (the struct) and reads fresh values each callback
Retain cycles UIKit holds delegate strongly; delegate holds UIKit Use weak back-references to views/controllers when needed

Coordinator roles beyond “delegate”

  • Delegate / data source: UITableViewDataSource, MKMapViewDelegate, etc.
  • Target-action: UIControl events wired with addTarget(_:action:for:) where the target is the coordinator.
  • Gesture recognizers: attach to the view, implement UIGestureRecognizerDelegate on the coordinator.
  • Combine / async glue: store cancellables or tasks; clear them in dismantleUIView.

Parent reference pattern

The usual pattern stores var parent: MyRepresentable on the coordinator. On each UIKit callback, you read parent to decide how to update bindings. When SwiftUI recreates the representable struct, it updates the bridge such that the coordinator’s parent tracks the latest props—think “refresh the adapter’s configuration,” not “recreate the coordinator every time.”

Example: multiple delegate methods (UIScrollViewDelegate)

import SwiftUI
import UIKit

struct ScrollOffsetReader: UIViewRepresentable {
    @Binding var offset: CGFloat
    @Binding var isDragging: Bool

    func makeCoordinator() -> Coordinator { Coordinator(self) }

    func makeUIView(context: Context) -> UIScrollView {
        let scroll = UIScrollView()
        scroll.delegate = context.coordinator
        scroll.alwaysBounceVertical = true
        return scroll
    }

    func updateUIView(_ uiView: UIScrollView, context: Context) {
        context.coordinator.parent = self   // explicit refresh pattern (also see note below)
    }

    final class Coordinator: NSObject, UIScrollViewDelegate {
        var parent: ScrollOffsetReader

        init(_ parent: ScrollOffsetReader) {
            self.parent = parent
        }

        func scrollViewDidScroll(_ scrollView: UIScrollView) {
            parent.offset = scrollView.contentOffset.y
        }

        func scrollViewWillBeginDragging(_ scrollView: UIScrollView) {
            parent.isDragging = true
        }

        func scrollViewDidEndDragging(_ scrollView: UIScrollView, willDecelerate decelerate: Bool) {
            if !decelerate { parent.isDragging = false }
        }

        func scrollViewDidEndDecelerating(_ scrollView: UIScrollView) {
            parent.isDragging = false
        }
    }
}

Note: some teams assign context.coordinator.parent = self inside updateUIView to guarantee the coordinator always references the latest struct values without relying on implicit update mechanics—either approach is interview-defensible if you articulate why (avoid stale configuration during rapid state churn).

3. UIHostingController embedding

Embedding SwiftUI in UIKit

UIHostingController is a UIViewController whose root view hosts a SwiftUI hierarchy. It is the primary vehicle for incremental adoption inside existing UIKit apps: keep navigation chrome in UIKit, drop SwiftUI for isolated features, or wrap design-system experiments.

Child view controller containment (the correct way)

For production UIKit parents, you should use full containment so lifecycle events propagate:

  1. addChild(hosting)
  2. attach hosting.view to the hierarchy with Auto Layout (or frames, if you must)
  3. hosting.didMove(toParent: self)

On teardown, reverse with willMove(toParent: nil), remove constraints, removeFromParent(). 

import UIKit
import SwiftUI

final class ProfileViewController: UIViewController {
    private let userID: String
    private var hosting: UIHostingController<ProfileSwiftUIView>?

    init(userID: String) {
        self.userID = userID
        super.init(nibName: nil, bundle: nil)
    }

    required init?(coder: NSCoder) { fatalError("init(coder:) has not been implemented") }

    override func viewDidLoad() {
        super.viewDidLoad()
        view.backgroundColor = .systemBackground

        let root = ProfileSwiftUIView(userID: userID)
        let host = UIHostingController(rootView: root)
        hosting = host

        addChild(host)
        host.view.translatesAutoresizingMaskIntoConstraints = false
        view.addSubview(host.view)

        NSLayoutConstraint.activate([
            host.view.leadingAnchor.constraint(equalTo: view.leadingAnchor),
            host.view.trailingAnchor.constraint(equalTo: view.trailingAnchor),
            host.view.topAnchor.constraint(equalTo: view.safeAreaLayoutGuide.topAnchor),
            host.view.bottomAnchor.constraint(equalTo: view.bottomAnchor),
        ])

        host.didMove(toParent: self)
    }
}

Sizing: intrinsic content size vs explicit constraints

SwiftUI views do not automatically behave like a UILabel with a meaningful intrinsicContentSize unless the hosted hierarchy exposes a compact layout. In UIKit tables and stacks, “zero height hosting controller” is a recurring incident.

  • Pin edges when the SwiftUI content should fill a known rect (most screens).
  • Let height be intrinsic when embedding small widgets (cells, banners) and use iOS 16+ sizingOptions to translate SwiftUI’s ideal size into UIKit measurement.

sizingOptions (iOS 16+)

UIHostingController gained sizing hooks so UIKit layout can ask SwiftUI “how big would you like to be?” without manual magic. The flags that show up most in real apps:

Option Meaning (practical)
.preferredContentSize Updates preferredContentSize for presentation contexts (popovers/sheets) based on SwiftUI content.
.intrinsicContentSize Feeds Auto Layout’s intrinsic size pipeline so parent constraints can hug SwiftUI content. 
let host = UIHostingController(rootView: BannerView(message: message))
host.sizingOptions = [.intrinsicContentSize]   // iOS 16+
// In a cell: pin leading/trailing/top, height can be driven by content + priorities
Sizing footgun

Intrinsic sizing still requires sane SwiftUI layout: ambiguous SwiftUI layouts produce unstable or zero sizes, and UIKit will faithfully propagate that ambiguity as constraint conflicts or clipping.

4. Incremental adoption strategy (UIKit → SwiftUI)

Bottom-up: leaf views first

Replace self-contained controls and cells where boundaries are narrow: labels with rich layout, badges, small compound components. Risk is lower because navigation and business orchestration remain in UIKit.

Top-down: new screens in SwiftUI

Ship greenfield flows as SwiftUI inside a UIKit shell (UIHostingController) or SwiftUI app entry with UIKit islands (UIViewRepresentable). This accelerates team skill growth and avoids rewriting stable legacy first.

Feature-by-feature migration

Carve vertical slices: one product area end-to-end (UI + state + analytics + tests). Slices prevent “half migrated” modules that compile but behave inconsistently across navigation paths.

Shared state between UIKit and SwiftUI

Prefer a small number of well-defined state carriers:

  • ObservableObject + @Published bridged into SwiftUI with @ObservedObject/@StateObject.
  • @Observable (iOS 17) with observation in SwiftUI and explicit refresh hooks in UIKit if needed.
  • Unidirectional stores (TCA-like, Redux-like) where UIKit subscribers and SwiftUI views read the same store.

Timeline and risk assessment (talk track)

Phase Scope Risk Success signal
0–6 weeks Interop scaffolding, design tokens, navigation bridges Medium: wrong sizing/containment patterns spread fast One production feature flag using SwiftUI in UIKit
2–4 months Vertical slices, shared VM layer, snapshot tests High: state duplication bugs Parity metrics (crashes, latency, QA escapes)
6–18 months Broad migration + deprecation of legacy UI Org risk: skill drift, review churn Reduced UI defect rate / faster iteration on migrated areas

ASCII diagram: migration phases

Phase A: UIKit App (monolith) +---------------------------+ | UINavigationController | | - Legacy feature screens | +---------------------------+ Phase B: Islands +---------------------------+ | UIKit shell | | - UIHostingController ----> SwiftUI feature (new) | - Legacy UIViewController | +---------------------------+ Phase C: Dual runtime (long coast) +---------------------------+ | SwiftUI root (new) | | - UIViewRepresentable ---> critical UIKit SDKs | - Native SwiftUI | +---------------------------+ Phase D: Consolidation +---------------------------+ | SwiftUI-first + thin UIKit| +---------------------------+

5. Two-way data flow between UIKit and SwiftUI

SwiftUI → UIKit: updateUIView on state change

When bindings or observed models change, SwiftUI schedules a pass that includes your representable’s updateUIView. This is the “push props down” direction. Keep updates fast; defer heavy work to async queues but mutate UIKit on the main thread.

UIKit → SwiftUI: coordinator + Binding

UIKit callbacks run on the main thread; write into @Binding, ObservableObject, or call a closure injected from SwiftUI. Avoid calling SwiftUI-specific assumptions from background threads. 

struct UIKitSearchField: UIViewRepresentable {
    @Binding var query: String
    var onSubmit: () -> Void = {}

    func makeCoordinator() -> Coordinator { Coordinator(self) }

    func makeUIView(context: Context) -> UISearchBar {
        let bar = UISearchBar()
        bar.delegate = context.coordinator
        return bar
    }

    func updateUIView(_ uiView: UISearchBar, context: Context) {
        if uiView.text != query {
            uiView.text = query
        }
    }

    final class Coordinator: NSObject, UISearchBarDelegate {
        var parent: UIKitSearchField
        init(_ parent: UIKitSearchField) { self.parent = parent }

        func searchBar(_ searchBar: UISearchBar, textDidChange searchText: String) {
            parent.query = searchText
        }

        func searchBarSearchButtonClicked(_ searchBar: UISearchBar) {
            parent.onSubmit()
        }
    }
}

ObservableObject shared between both worlds

A reference-type model owned by UIKit (e.g. view model created in UIViewController) can be passed into SwiftUI via UIHostingController(rootView:) initializer capture or environment injection. SwiftUI observes changes; UIKit can mutate the same object and call objectWillChange.send() when needed. 

final class SessionStore: ObservableObject {
    @Published var username: String = ""
}

struct HeaderView: View {
    @ObservedObject var session: SessionStore

    var body: some View {
        Text("Hello, \(session.username)")
    }
}

// UIKit creates the store once; SwiftUI observes it.
let store = SessionStore()
let host = UIHostingController(rootView: HeaderView(session: store))

@Observable shared (iOS 17+)

With the Observation framework, SwiftUI views use @Bindable/@State patterns against @Observable models. From UIKit, you still hold the same reference-typed model; updates can flow through method calls. Ensure UIKit-side code triggers observation notifications appropriately when mutating fields that SwiftUI reads—in practice, prefer narrow mutation APIs on the model rather than scattered field writes. 

import Observation

@Observable final class Cart {
    var itemCount: Int = 0
}

struct CartBadge: View {
    @Bindable var cart: Cart

    var body: some View {
        Text("\(cart.itemCount) items")
    }
}
Direction Mechanism Watch for
SwiftUI → UIKit updateUIView, environment-driven config Feedback loops (map region, text caret)
UIKit → SwiftUI Coordinator writes bindings / store Main-thread violations, re-entrancy
Both Shared reference model Over-wide observation causing excessive body recomputation

6. Sizing challenges (intrinsic content size bridging)

UIKit intrinsicContentSize vs SwiftUI sizing

UIKit Auto Layout asks views for an intrinsic size when constraints are underdetermined. SwiftUI instead uses proposals and responses in its layout protocol; bridging layers must translate between the two paradigms.

UIHostingController inside UIKit containers

  • Table/collection cells: pin lateral edges; choose a height strategy (fixed, self-sizing with sizingOptions, or explicit calculation).
  • Stack views: remember UIStackView uses intrinsic sizes heavily—test on largest dynamic type.
  • Navigation transitions: snapshot-based animations may not play nicely with highly dynamic SwiftUI sizes; stabilize identity and avoid layout thrash during transitions.

UIViewRepresentable inside SwiftUI

By default, many UIKit views return UIView.noIntrinsicMetric for one or both axes. SwiftUI may expand the view to fill proposed space unless you constrain it with SwiftUI layout (e.g. frame, fixedSize, parent stack rules).

Common sizing bugs and fixes

Symptom Likely cause Fix direction
Zero-height SwiftUI in UIKit No intrinsic content + missing height constraint Pin height, or enable sizingOptions + hugging priorities
UIKit view fills too much space in SwiftUI UIKit intrinsic width/height is large or unspecified Use frame(maxWidth:), fixedSize, or implement sizeThatFits (iOS 16+)
Scroll views “fight” the parent Nested scrollable axes with ambiguous constraints Split responsibilities; use SwiftUI.ScrollView or UIKit scroll, not both competing

sizeThatFits in UIViewRepresentable (iOS 16+)

Implementing sizeThatFits lets SwiftUI query the UIKit view for a natural size given a proposal. This is the bridge answer to “my representable always expands to infinity.” 

struct MeasuredLabel: UIViewRepresentable {
    var text: String

    func makeUIView(context: Context) -> UILabel {
        let label = UILabel()
        label.numberOfLines = 0
        return label
    }

    func updateUIView(_ uiView: UILabel, context: Context) {
        uiView.text = text
    }

    func sizeThatFits(_ proposal: ProposedViewSize, uiView: UILabel, context: Context) -> CGSize? {
        let width = proposal.width ?? UIView.layoutFittingExpandedSize.width
        let target = CGSize(width: width, height: .greatestFiniteMagnitude)
        return uiView.systemLayoutSizeFitting(
            target,
            withHorizontalFittingPriority: .required,
            verticalFittingPriority: .fittingSizeLevel
        )
    }
}
Staff-level summary

Interop sizing is translation work: UIKit wants intrinsic metrics and constraints; SwiftUI wants proposals. When something looks “infinite” or “zero,” you are usually missing a translator—either explicit frames, UIHostingController.sizingOptions, or sizeThatFits on the representable.

7. Architect callout: migrating a 500K LOC UIKit app to SwiftUI

500K LOC UIKit → SwiftUI (6–18 month horizon)

At this scale, migration is a program, not a refactor. Success is measured by risk-adjusted throughput: shipping customer value while steadily reducing the percentage of new work that deepens UIKit coupling.

Phase planning.

  • Months 0–3: interop standards (hosting containment, representable templates, state bridges), CI snapshots for SwiftUI, performance baselines.
  • Months 3–9: vertical slices through high-churn product areas; shared domain/view-model layer; navigation adapters.
  • Months 9–18: consolidate duplicated UI paths; retire worst legacy modules; train review rubrics for SwiftUI performance and identity.

Shared architecture layer.

Extract UI-agnostic use cases (networking, validation, feature flags, analytics events) so both UIKit and SwiftUI call the same services. Ban “SwiftUI-only” business rules unless you enjoy split-brain bugs.

Navigation bridging.

Introduce a small routing abstraction: deep links and in-app routes resolve to either a UIKit stack push or a SwiftUI navigation path. Avoid N×N ad hoc segues between runtimes.

State management bridging.

Standardize on one observable store style per app (Observation vs Combine vs external store). Document how UIKit view controllers subscribe—via callbacks, Combine, or explicit presenter methods.

Testing strategy during migration.

  • Snapshot tests for SwiftUI screens (dark mode, dynamic type, RTL).
  • Integration tests for navigation bridges and feature flags.
  • Performance tests on representative devices for scroll-heavy SwiftUI lists.

Risk mitigation.

  • Feature flags per screen/flow; instant rollback to UIKit path.
  • Crash bucketing split by runtime (SwiftUI vs UIKit) to spot bridge regressions quickly.
  • Memory leak audits around coordinators, hosting controllers, and delegate cycles.

Decision framework: what to migrate first

Migrate earlier Defer
High churn UI (frequent product iteration) Stable, rarely touched settings/admin surfaces
Self-contained visuals with few system edge cases Heavy UIKit text stacks with custom IME behavior
Areas with poor test coverage that SwiftUI snapshots can secure Modules tightly coupled to complex UIView subclass hierarchies
Closing narrative

Staff interviews reward systems thinking: you can name the protocols, but you win by explaining lifecycle, thread model, sizing translation, and how you would run a multi-quarter migration without freezing feature development.
