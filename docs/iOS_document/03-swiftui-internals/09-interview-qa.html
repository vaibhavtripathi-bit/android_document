1. AttributeGraph & Runtime

Q1: Explain how AttributeGraph tracks dependencies between views and state. L6

SwiftUI builds a persistent graph of attributes (the AttributeGraph) rather than re-executing your entire UI tree as a plain function on every frame. Each node can declare inputs—bindings to model values, environment, layout results, and child attributes. When an input changes, the graph marks affected nodes dirty and schedules incremental updates. The framework walks only the subgraph that depends on the changed attribute, which is why small state changes do not force a full view hierarchy rebuild in the naive sense. Conceptually, your body produces a description; the runtime materializes and updates graph nodes that correspond to stable identity and type information.

State change → dirty attributes → AG invalidation → layout / display list update → CA commit

At Staff level you should mention that this is similar in spirit to reactive dependency graphs in other UI systems, but SwiftUI’s graph is tightly integrated with Swift’s value semantics and type metadata.

Q2: What happens internally when a @State value changes? Senior

@State is backed by storage that lives with the view’s identity in the graph, not inside the struct value of the view itself. When you mutate the wrapped value, SwiftUI records the change, marks the owning view subtree as needing an update, and coalesces work into the current transaction or the next run loop turn. The view’s body is re-evaluated for that identity; the diffing layer compares the new description to the previous one and updates only what changed. Wrapping mutations in withAnimation attaches animation metadata so the same invalidation drives animated transitions on the Core Animation side.

struct Counter: View {
    @State private var count = 0
    var body: some View {
        Button("Tap") { count += 1 }  // invalidates this view’s body in the graph
    }
}
Q3: How does the SwiftUI render loop relate to Core Animation’s commit cycle? Architect

SwiftUI batches updates: model changes invalidate graph nodes, layout produces resolved frames, and the rendering backend translates stable representations into Core Animation layer trees (or Metal-backed content where applicable). On each frame, UIKit/AppKit still drives display links; SwiftUI’s work typically completes before or during the commit that flushes layers to the render server. Heavy synchronous work on the main thread delays both SwiftUI’s graph pass and CA’s commit, which shows up as dropped frames. Understanding this separation helps you decide when to move work off the main actor and when to use explicit drawingGroup or pre-rendering strategies.

Q4: What is the ViewGraph, and how does it differ from the AttributeGraph? L6

Interviewers often conflate these terms. Practically, the AttributeGraph is the lower-level dependency and invalidation structure; the “view graph” language in documentation refers to the tree of view values and their identities that SwiftUI maintains as it applies diffs. You can describe the ViewGraph as the logical hierarchy of view identities and their associated attributes, while the AttributeGraph is the engine that stores attributes, tracks edges between them, and propagates invalidation. A strong answer ties user-visible behavior—preserved scroll position, focus, animations—to identity in the view graph and to attribute invalidation for recomputation.

Q5: Why are SwiftUI updates transactional, and what breaks if you ignore that model? Senior

Multiple state writes in one event handler are coalesced into a single consistent update pass. That avoids inconsistent intermediate frames where one @Published property has updated but another has not. If you fight the model—for example by forcing UIKit updates from random threads without hopping to the main actor—you can see tearing between SwiftUI and hosted UIKit views. Prefer batching related mutations, using objectWillChange.send() carefully with ObservableObject, and relying on SwiftUI’s transaction system for animation grouping.

var transaction = Transaction(animation: .easeInOut)
transaction.disablesAnimations = false
withTransaction(transaction) {
    flagA = true
    flagB = true  // one coherent update
}
Q6: How does the runtime use type metadata when updating the view hierarchy? Architect

Each concrete view type encodes how to build and update its corresponding graph nodes. When the type of a child slot stays the same, SwiftUI can often perform fine-grained updates. When the type changes—after an if branch switches between different view types—the old subtree is torn down and a new one is created unless you stabilize identity with id or AnyView (with tradeoffs). At L6 depth, you might discuss how opaque return types (some View) still preserve concrete type information at compile time for specialization, while type erasure hides that from the compiler and can reduce optimization opportunities.

Q7: What causes “AttributeGraph: cycle detected” style issues, and how do you debug them? L6

These arise when layout or preference resolution creates a circular dependency: a parent’s size depends on a child’s reported size, which depends on the parent’s offered size, without a fixed point. GeometryReader in the middle of flexible stacks is a common trigger when combined with unconditional reads of geometry that feed back into the same subtree’s layout. Mitigations include restructuring so one direction of information flow is explicit, using fixed sizes or Layout protocols with well-defined proposals, or breaking the cycle with anchors and two-pass patterns. Debugging involves Instruments (SwiftUI template), simplifying the view tree, and logging proposed vs reported sizes.

Warning

Avoid reading geometry.size to set the same container’s height that wraps the GeometryReader unless you have a clear constraint chain.

Q8: How does environment propagation interact with the AttributeGraph? Senior

Environment values are part of the attribute system. When you inject .environment(\.key, value), descendants that read that key subscribe to it. Changing an environment value invalidates dependent subtrees similarly to state, but the storage is scoped along the hierarchy rather than owned by a single view. This is why deep environment changes can be expensive if they force large subtrees to refresh—prefer narrow placement of modifiers or explicit parameters for hot paths.

Q9: Describe the lifecycle from user event to pixels on screen in SwiftUI terms. Architect

Gesture or control action runs (typically on the main actor), mutating state or calling async callbacks that eventually mutate state. SwiftUI schedules an update; the AttributeGraph marks dirty nodes. Layout runs with parent-to-child proposals and child-to-parent size reporting. Style and animation attributes resolve. The rendering backend updates layers and textures. Core Animation composites and the render server presents. For Staff interviews, mention async image decoding and how it can complete off-main but must publish results on the main actor for safe UI updates.

Q10: Why is “views are cheap” only partly true at scale? L6

Value-typed view structs are cheap to allocate, but graph nodes, layout passes, and diffing are not free. Deep trees, heavy body work, dynamic member lookups, and type erasure all shift cost from allocation to CPU time in the framework. Profiling real devices matters: Instruments shows whether you are bound by SwiftUI body time, layout, or GPU. The mental model is cheap descriptions, potentially expensive materialization and update—especially when identity churns or layout is under-constrained.

2. View Identity & Diffing

Q11: Why can if/else inside a ViewBuilder destroy view state? Senior

ViewBuilder builds a conditional as different view types at compile time: either the if branch’s type or the else branch’s type occupies the same structural slot. When the condition flips, SwiftUI sees a different type at that position and treats it as a new subtree, resetting @State stored on the old identity. Text field focus, scroll position, and internal control state disappear unless you hoist state above the branch or use stable identity with .id applied thoughtfully.

if showLogin {
    LoginView()   // identity A
} else {
    HomeView()    // identity B — not the same node as A
}

Hoisting shared state to a parent or using a wrapper view with constant type avoids the reset.

Q12: What is the performance cost of AnyView, and why? L6

AnyView type-erases the concrete view type, so the compiler cannot specialize updates and the runtime must use dynamic dispatch and heap allocation for the boxed content. Each rebuild may pay extra for existential containers and lose opportunities for static optimization. It is acceptable at boundaries (small conditional chrome) but harmful inside tight loops or large lists. Prefer @ViewBuilder generics, Group, TupleView, or splitting into separate containers with stable structure.

Q13: Contrast structural identity with explicit identity (.id). Senior

Structural identity is derived from the view tree’s type-aligned positions: same parent, same index, same type implies continuity. Explicit identity overrides that with a user-supplied value; when id changes, SwiftUI discards the old subtree and creates a new one—useful for forcing reset when model primary keys change, or harmful if overused causing unnecessary teardown. Staff candidates explain tradeoffs: .id(object.id) for list detail when switching items, versus avoiding churn on every minor model tick.

Q14: How does ForEach use identity, and what breaks with duplicate IDs? L6

ForEach requires stable, unique identifiers (via Identifiable or id: closures). Identity tells SwiftUI which rows correspond across data changes for diffing and animation. Duplicate IDs violate assumptions: updates may apply to the wrong row, animations glitch, and state can leak between cells. Always key on primary keys, not display strings that can collide.

ForEach(items) { item in  // items: [Identifiable]
    RowView(item: item)
}
Q15: What are ViewBuilder limitations and how does the builder transform control flow? Architect

@ViewBuilder supports a subset of Swift: limited if, optional unwrapping, switch (with version caveats), and no arbitrary loops producing a single view without ForEach. The builder rewrites statements into nested generic conditional types (ConditionalContent, etc.). That is why branching affects type identity and why complex control flow sometimes compiles to unexpected depths. At L6, you might discuss how variadic generics in newer Swift versions affect tuple views and builder performance.

Q16: When should you prefer Group versus a container like VStack for identity? Senior

Group is transparent for layout—it does not introduce a layout node of its own in the same way a stack does—but it still participates in the type structure of the builder output. Use it to apply modifiers to multiple views or to satisfy API that expects a single View without changing layout semantics. Changing between Group { a; b } and VStack { a; b } changes structure and can affect identity of children relative to ancestors; prefer the minimal wrapper that matches layout intent.

Q17: How do opaque return types (some View) help the diffing engine? L6

From the caller’s perspective the type is opaque, but inside the module the concrete type is known at compile time, enabling specialization and avoiding existential overhead in many hot paths. The concrete type of body is fixed per view struct declaration; if you need to return different concrete types, you use Group, AnyView, or @ViewBuilder conditionals—each choice has identity implications. Changing a view’s body return type (e.g., refactoring modifiers) can alter the structural identity of anonymous inner types.

Q18: Explain why EquatableView or custom equatable() can reduce updates. Senior

When the wrapped view’s content compares equal to the previous value, SwiftUI can skip descending into that subtree’s body, reducing work for expensive children. This is valuable for large static regions adjacent to frequently changing siblings. The pitfall is incorrect Equatable implementations that return true when underlying dependencies (environment, preferences) changed, causing stale UI. Pair with clear input models and test edge cases.

Q19: How does @ViewBuilder content in library APIs affect binary stability and diffing? Architect

Public APIs that take @ViewBuilder closures freeze client usage patterns into the type system of the call site. From an architecture angle, heavy use of generics in public modifiers can increase compile times and ABI surface for libraries. For app teams, the concern is more about encapsulation: exposing builders encourages flexible composition but makes it harder to enforce consistent identity and performance inside black-box components. Document when children are expected to be lazy (LazyVStack) vs eager.

Q20: What happens to animation and transitions when identity is unstable? L6

Transitions animate insertion and removal of views with stable identity relative to their container. If identity flickers—e.g., list rows keyed by index after a sort—SwiftUI may interpret rows as remove+insert rather than move, producing wrong animations and lost focus. Fix keys, use animation(_:value:) with meaningful values, and prefer withAnimation around batched data updates for coherent transitions.

3. State Management

Q21: Explain the critical difference between @StateObject and @ObservedObject. Senior

@StateObject creates and owns the observable object for the lifetime of the view that declares it; SwiftUI keeps the instance across view struct value copies. @ObservedObject expects an object injected from outside; the view does not own creation. Using @ObservedObject for something you initialize in the view’s initializer can recreate the object on every parent refresh, breaking invariants and causing bugs like reset network clients or lost form data. Use @StateObject for view-owned view models and @ObservedObject for models owned by a parent or the environment.

struct Screen: View {
    @StateObject private var vm = ScreenModel()  // stable
    var body: some View { Content(vm: vm) }
}
Q22: How does @Observable (Observation framework) achieve per-property tracking? L6

The macro-driven @Observable transforms stored properties so reads and writes register with an observation registrar. SwiftUI can subscribe to fine-grained changes instead of invalidating the whole view on any objectWillChange pulse. That reduces over-updating compared to coarse ObservableObject when many unrelated properties live on one model. Views that use @Bindable or bind directly to observable models still re-render when touched properties change; the win is narrower invalidation and less manual publisher wiring.

Q23: When would you still choose ObservableObject over @Observable? Senior

Interop with Combine pipelines, existing view models, or APIs that expect ObservableObject and objectWillChange are common reasons. Older OS targets before Observation is fully available in your deployment target also matter. Migration can be incremental: wrap legacy objects or bridge publishers into observable models. Team consistency and testing infrastructure sometimes favor staying on Combine until a platform baseline allows a clean cut.

Q24: How does @EnvironmentObject differ from passing @ObservedObject down manually? Senior

Both hold a reference to a shared object, but @EnvironmentObject retrieves the instance from the environment chain, avoiding prop drilling through intermediaries that do not care about the model. The risk is runtime failure if the object was not injected—prefer explicit parameters for library components and environment for app shells. Deep environment changes can invalidate large subtrees; scope injections as narrowly as practical.

Q25: Describe main-actor isolation patterns for SwiftUI view models. Architect

Mark UI-facing models with @MainActor so published state mutations always run on the main actor, matching SwiftUI’s expectations. Long-running work uses Task or async functions that hop to background executors and then await MainActor.run to publish results. Mixing nonisolated mutations into observable state is a common source of purple runtime warnings and subtle races. Document concurrency boundaries for large teams so feature modules do not accidentally call view model setters from background queues.

@MainActor
final class ProfileVM: ObservableObject {
    @Published private(set) var name = ""
    func load() async {
        let value = await fetchOnBackground()
        name = value  // main actor
    }
}
Q26: What are pitfalls of binding to optional or computed state? L6

Binding unwraps and writes back through projected values; if the storage is ephemeral or recreated each body evaluation, writes may not persist or may target the wrong owner. Use stable reference types or Binding(get:set:) with explicit mutation paths. For optionals, combine if let / guard let with a hand-built binding to the underlying store, or use small wrapper types—ensure set mutates the authoritative store, not a copy.

Q27: How do you test observable view models that drive SwiftUI? Senior

Keep business logic in testable types; use ObservableObject’s publishers or Observation’s testing hooks to assert state transitions. For SwiftUI integration, ViewInspector (third-party) or snapshot tests on key screens complement unit tests. Prefer verifying outputs and side effects (network mocks, analytics) over testing framework internals. Async tests should use await fulfillment(of:) or Swift Testing with concurrency discipline.

Q28: Explain @Bindable with @Observable models in child views. L6

@Bindable wraps an observable model so you can derive Bindings to its properties for controls like TextField. Without it, value-type observation wrappers may not expose two-way bindings ergonomically. The model remains reference-identical while bindings propagate writes through observation, keeping the graph and controls in sync.

struct Editor: View {
    @Bindable var model: NoteModel
    var body: some View {
        TextField("Title", text: $model.title)
    }
}
Q29: How should global app state interact with feature modules? Architect

Inject a narrow protocol or facade through the environment at the composition root; avoid singletons that hide dependencies. Feature modules receive only the capabilities they need (session, flags, navigation router), which improves testability and prevents accidental cross-feature coupling. For very large apps, consider a lightweight store per feature with a shared event bus or coordinator pattern for cross-cutting concerns like deep links.

Pattern

Composition root owns concrete types; features depend on protocols and factory closures.

Q30: What is the role of transaction and animation in state-driven UI updates? Senior

Transaction carries animation curves, disables animations for atomic updates, and propagates with state changes. Multiple mutations in one transaction animate as a group. Understanding transactions helps debug “why didn’t this animate” issues—often a child used withAnimation while a parent disabled animations, or implicit animations conflicted with explicit ones.

4. Layout System

Q31: Describe the three-phase layout negotiation in SwiftUI. L6

Parents propose a size to children (proposal), children choose their size based on constraints and content (placement sizing), then parents position children in their coordinate space (placement). This repeats down the tree. Understanding “proposal down, size up, position down” explains stack behavior, flexible vs fixed spacers, and why infinity proposals from scroll views affect text wrapping. Custom Layout types implement the same protocol-driven conversation explicitly.

Q32: When should you use the Layout protocol instead of GeometryReader? Senior

Use Layout when you can express child arrangement purely through the layout protocol’s sizeThatFits and placeSubviews without needing arbitrary reads of global geometry mid-tree. GeometryReader always takes all offered space in its axis—often a layout smell—and can introduce secondary layout passes. Layout is more composable, testable, and participates cleanly in the proposal system; reserve GeometryReader for truly geometry-dependent drawing or one-off measurements.

Q33: What is ProposedViewSize and how do width/height nil values behave? L6

A proposal may be unspecified along an axis (nil), meaning “ideal size” or flexibility depending on context—e.g., vertical stacks may propose nil height to text so it chooses its intrinsic height. Concrete numeric proposals constrain maximum (and sometimes minimum) space. Misunderstanding nil vs infinity leads to bugs like text not wrapping when width is unbounded. Read proposals in custom layouts and forward them intentionally to children.

Q34: How does safeAreaInsets participate in layout? Senior

The safe area shrinks the region where content is laid out by default; modifiers like ignoresSafeArea expand the drawable region. Layout proposals account for safe area unless ignored, which is why full-bleed backgrounds need explicit ignores while primary content stays inset. Keyboard and accessory bars publish inset changes that animate; views relayout when insets update.

Q35: Explain preference keys and bottom-up data flow for layout. Architect

Preference keys let children report values up the tree; ancestors reduce them with associative combining. Used for aligning overlays, custom tab bars, or collecting child sizes. Unlike environment (top-down), preferences bubble up and can trigger parent layout when values change. Avoid feedback loops where a parent’s layout depends on a preference that depends on the parent’s size without a stable algorithm.

struct HeightKey: PreferenceKey {
    static var defaultValue: CGFloat = 0
    static func reduce(value: inout CGFloat, nextValue: () -> CGFloat) {
        value = max(value, nextValue())
    }
}
Q36: Why can LazyVStack inside ScrollView still cause measurement issues? L6

Lazy containers only materialize visible children, but scroll views may still need approximate sizing for scroll indicators, rubber-banding, and accessibility. Very dynamic content heights can cause scroll position jumps if estimates change. Mitigations include fixed row heights, caching heights, or using scrollTargetLayout APIs appropriate to your OS version. Profile with Instruments to see if time is in measurement or rendering.

Q37: How do you implement a custom equal-width column grid with Layout? Senior

In sizeThatFits, measure children with flexible proposals, compute the maximum of child widths (or distribute width evenly), then multiply by row count for total size. In placeSubviews, assign frames on a grid. Using the layout protocol keeps the logic pure and avoids GeometryReader expansion. For variable row heights, track per-row maxima similarly.

Q38: What is the relationship between layout and accessibility frame updates? Architect

After layout, accessibility nodes expose frames in screen space; dynamic type and layout changes require VoiceOver to refresh. Breaking layout caching or using transforms without updating accessibility can produce mismatches. Use accessibilityElement merging and labels intentionally after custom layouts so assistive tech matches visual placement.

5. Rendering & Core Animation

Q39: How does SwiftUI bridge to Core Animation? L6

SwiftUI maps view hierarchies to layer-backed representations managed by the framework. Animatable properties participate in implicit animations via transactions; the engine interpolates layer properties or uses action dictionaries similarly to UIKit’s model layer. You rarely manipulate CALayers directly in pure SwiftUI; instead you influence the graph and let the bridge update layers. Hosting UIViewRepresentable merges UIKit’s layer tree under SwiftUI’s composition.

Q40: When would you use drawingGroup() and what does it do? Senior

drawingGroup(opaque:colorMode:) renders the subtree into an offscreen image (often Metal-backed) and composites it as a single layer. Benefits include flattening deep hierarchies for effects like blur and potentially improving animation performance for complex vector content. Costs include memory for the buffer and loss of per-view layer inspectability; hit testing and accessibility can be affected if not paired with appropriate modifiers. Use when profiling shows layer explosion or expensive per-frame redraws of static decorative content.

Q41: Compare Canvas with imperative draw(_:) style rendering. L6

Canvas is declarative: you describe drawing operations in a closure that receives a GraphicsContext and size. It integrates with SwiftUI’s invalidation—when inputs change, the canvas redraws. For highly dynamic scenes, combine with TimelineView or explicit state. It is preferable to UIKit draw(_:) inside SwiftUI when you want unified state and less manual setNeedsDisplay juggling, though UIKit still wins for some legacy drawing stacks.

Canvas { context, size in
    var path = Path()
    path.addEllipse(in: CGRect(origin: .zero, size: size))
    context.fill(path, with: .color(.blue))
}
Q42: What role does Metal play in SwiftUI rendering on modern iOS? Architect

Many compositing paths use GPU acceleration through Metal for efficiency, especially for effects, large images, and drawingGroup. You typically do not write Metal shaders for standard apps, but understanding GPU memory pressure helps explain jetsam under heavy image pipelines. Custom Metal interop is advanced and usually done via UIViewRepresentable or platform interop layers.

Q43: How do blend modes and compositing groups affect performance? Senior

Non-default blending often forces offscreen passes because the GPU must resolve intermediate pixels. Nesting many blended overlays in scrolling lists multiplies fill rate cost. Prefer pre-rendered assets or simplified visuals on older devices; use Instruments’ Core Animation instrument to detect offscreen rendering (yellow flashes in the simulator’s color overlays).

Q44: Explain vector vs bitmap resolution in SwiftUI images and scaling. Senior

SF Symbols and vector PDFs scale crisply; raster images need correct asset scales (@2x/@3x). SwiftUI’s Image respects rendering mode and template behavior for tinting. Improper resampling (scaling small bitmaps up) causes blur and GPU filter cost. Use resizable(capInsets:resizingMode:) intentionally for stretchable assets.

Q45: What are common causes of implicit animation not running? L6

The changing property may not be animatable in the bridge, animations may be disabled in the current transaction, or the view identity may reset so the change is treated as a new insertion. Conditional application of .animation on the wrong branch also fails. Use withAnimation around the state mutation and ensure the view’s type and identity remain stable for the property you animate.

Q46: How does TimelineView interact with power and refresh policies? Architect

TimelineView schedules updates based on a schedule (e.g., animation, every second). Frequent schedules keep the CPU and GPU awake—bad for battery if overused on always-visible widgets or backgrounds. Choose the coarsest schedule that meets UX needs; for clocks, use pausable schedules when off-screen if APIs allow, or move to lower-frequency updates when approximate time suffices.

6. Navigation

Q47: Design a navigation system for a 100+ screen app using NavigationStack. Architect

Use a router type that maps enum cases to destinations, backed by NavigationPath for programmatic push/pop. Split features into modules that register route builders into a central registry (type-safe enums per feature, not one giant enum if compile time suffers). Persist path codable representation for state restoration. Keep view construction lazy via factories so the stack does not instantiate all screens eagerly. Deep links parse URL → route array → assign path. Test push/pop sequences and memory retention of view models tied to stack entries.

enum AppRoute: Hashable {
    case profile(UUID)
    case settings
}
@State private var path = NavigationPath()
Q48: How would you implement deep linking with NavigationStack? L6

Parse universal links into a sequence of Hashable routes, then replace or append to NavigationPath on appear or in onOpenURL. Ensure tab selection happens before stack mutation if using tab bar + nested stacks. Handle invalid URLs gracefully and align with server-driven feature flags. For complex apps, maintain a single source of truth for navigation state observable by analytics.

Q49: Compare value-type navigation state with binding-driven navigation. Senior

Storing NavigationPath or route arrays in a view model or app store enables testing and persistence; purely binding-driven stacks are simpler but harder to log and replay. Hybrid patterns expose bindings to SwiftUI while deriving them from a central router. Choose based on QA needs, crash reporting (breadcrumbs), and whether you must synchronize navigation across multiple windows on iPad.

Q50: How do you handle modal presentation alongside stack navigation? Senior

Use separate boolean or item bindings for sheets and full-screen covers; avoid duplicating the same destination in both stack and sheet unless UX requires it. Dismiss policies should be explicit (user cancel vs completion). For iPad, consider navigationSplitView with column selection instead of stacked modals. State restoration must encode both path and presented modals if you support multitasking restore.

Q51: What breaks when migrating from NavigationView to NavigationStack? L6

Legacy double-column behavior on iPad and implicit root handling differ. NavigationView’s quirks masked missing data flow; stacks require explicit paths and navigationDestination registrations. Test split view, compact horizontal size classes, and orientation changes. Update any code that relied on NavigationLink(isActive:) two-way bindings to path-based APIs.

Q52: How would you coordinate navigation with authentication state? Architect

Use a root coordinator that switches between login and main tab structures at a high level—avoid pushing dozens of login-related routes onto the same stack as post-login content. When session expires, reset paths atomically and present re-auth modally if needed. Persist partial user context securely, not navigation stacks containing PII in route parameters. Log transitions for fraud and support tooling.

Q53: Explain navigationDestination(for:) and type erasure pitfalls. Senior

Register destinations per type pushed onto the path. Multiple registrations for related types must be consistent; overload ambiguity can route incorrectly. Keep destination views lightweight wrappers that fetch models by ID from a store rather than capturing large value payloads in the route enum, reducing diff churn and memory.

Q54: How do you test navigation-heavy SwiftUI flows? L6

Unit-test the router: given URL or user action, assert resulting route stack. UI tests exercise taps and back gestures. Snapshot tests are brittle for navigation; prefer asserting view model state. For NavigationPath, encode/decode round-trip tests catch Codable breaking changes across releases.

7. Performance

Q55: Walk through your process for debugging a slow SwiftUI screen. Architect

Reproduce on the lowest supported device with Instruments SwiftUI profiling. Check time in body evaluation, layout, and Core Animation commits. Inspect for accidental onAppear work, synchronous decoding on main, and list identity churn. Binary-search the view tree by commenting regions or using equatable() probes. Once hot spots are found, apply targeted fixes: hoist state, lazy containers, image downsampling, or move networking off main. Verify with before/after traces and memory footprint (jetsam risk).

Signal

High body count per second often correlates with unstable identity or over-frequent published updates.

Q56: When are lazy stacks preferable to regular stacks? Senior

Use lazy stacks for long scrolling content where off-screen views should not be constructed. Regular stacks eagerly build all children—fine for small forms, expensive for hundreds of rows. Lazy stacks trade initial work for scrolling measurement complexity; pick based on count and dynamism. Inside non-scrolling parents, lazy often adds overhead without benefit.

Q57: How does image loading affect scroll performance? L6

Decoding large images on the main thread stalls frames. Use downsampling (UIImage thumbnail APIs, ImageIO), async loaders with placeholders, and correct asset sizes. In lists, cancel in-flight loads when cells disappear. Combine with LazyVStack and reasonable row heights to keep memory predictable.

Q58: What is the impact of frequent @Published emissions? Senior

Each emission can invalidate every observing view even if only one field matters. Throttle or batch updates, split models by concern, or migrate to @Observable for finer tracking. For high-frequency streams (progress, timers), derive display strings at lower cadence or use TimelineView instead of pushing every tick through ObservableObject.

Q59: How do you profile SwiftUI memory in a list of rich cells? L6

Use Allocations and Leaks instruments; watch transient allocations during scroll. Ensure view models are not captured strongly by async closures without cancellation. Avoid storing UIImages in model layers; use identifiers and caches with eviction. Test navigation push/pop for retained cycles via UIViewController representables.

Q60: Describe using redacted(reason:) and placeholder UI for perceived performance. Senior

Skeleton placeholders preserve layout and reduce layout thrash when content arrives asynchronously. Pair with stable container sizes to minimize cumulative layout shift. Redacted placeholders should match final typography roughly so Dynamic Type transitions are smooth. This is UX performance as much as raw frame time.

8. UIKit Interop

Q61: Design a migration strategy for a large UIKit app to SwiftUI. Architect

Adopt a strangler pattern: wrap new SwiftUI screens in UIKit via UIHostingController; embed legacy UIKit in SwiftUI via UIViewRepresentable. Migrate leaf flows first (settings, simple lists) where navigation maps cleanly to stacks. Establish design system components in SwiftUI early to avoid duplicate styling. Define module boundaries so shared view models are UI-framework agnostic. Track metrics (crash rate, performance) per migrated screen and roll back risky areas. Train the team on identity and state pitfalls before tackling complex gestures or collection views.

Q62: Explain the responsibilities of UIViewRepresentable lifecycle methods. L6

makeUIView(context:) constructs the view once; updateUIView(_:context:) syncs SwiftUI state to UIKit whenever the struct’s properties change; dismantleUIView(_:coordinator:) tears down. The coordinator holds delegates and targets to avoid retain cycles. Do expensive setup in makeUIView; keep updateUIView idempotent and fast. Missing coordinator wiring is a top cause of delegate callbacks not reaching SwiftUI state.

struct MapWrapper: UIViewRepresentable {
    func makeUIView(context: Context) -> MKMapView { MKMapView() }
    func updateUIView(_ map: MKMapView, context: Context) {
        // apply region, annotations from SwiftUI state
    }
}
Q63: How do you handle intrinsic content size and layout in representables? Senior

SwiftUI asks UIKit views for sizing via the layout integration layer. If intrinsic size is ambiguous, provide fixedSize() or explicit frames in SwiftUI, or implement preferred layout constraints inside the UIView. Mismatches cause clipping or excess padding. For wrapping scroll views, communicate content size changes with invalidation when content updates.

Q64: What are threading rules when bridging UIKit and SwiftUI? L6

All UIKit view mutations must occur on the main thread. SwiftUI’s update pass is main-thread; still avoid dispatching async callbacks that touch UIKit without mainActor hops. Combine publishers should use receive(on: DispatchQueue.main) or @MainActor annotations. Data races between representable updates and UIKit delegates indicate missing serialization.

Q65: When would you use UIViewControllerRepresentable instead of a hosting controller from UIKit? Architect

Use it when the UIViewController lifecycle, child view controllers, or system view controllers (camera, document picker) must participate in SwiftUI composition. Hosting controller is for presenting SwiftUI from UIKit root; representable wraps UIKit for SwiftUI parents. For complex flows (nav controller inside SwiftUI), ensure environment objects and safe area propagate correctly—often requiring manual child containment and callback bridges to SwiftUI state.

Staff takeaway

Choose the thinnest bridge that preserves the lifecycle you need; every layer is a maintenance and testing surface.
