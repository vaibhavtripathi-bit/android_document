1. @State — Heap-backed, view-scoped storage

@State is the primitive for private mutable state that belongs to a view subtree, not to the Swift struct instance you type in source code. That distinction is the heart of almost every “why did my state reset?” question in SwiftUI.

1.1 What @State compiles to (conceptual model)

At compile time, @State private var count: Int = 0 lowers to a property wrapper whose wrappedValue reads and writes through SwiftUI’s runtime. The actual Int (or whatever value type you store) lives in heap-allocated storage managed by the framework and keyed off the view’s identity in the attribute graph — not off the ephemeral struct value passed to body.

┌─────────────────────────────────────────────────────────────────┐ │ Your source: struct CounterView: View { @State private var n } │ │ │ │ Each body evaluation: NEW struct instance (value semantics) │ │ │ │ │ ▼ │ │ Attribute Graph node for this View identity │ │ │ │ │ └──► Heap slot: StateBox holds actual storage for `n` │ │ Survives struct copies / body re-invocation │ └─────────────────────────────────────────────────────────────────┘
Interview sound bite

SwiftUI views are structs; structs are recreated constantly. Persistent UI state cannot live “inside” the struct’s inline memory the way it would in a UIKit UIView subclass instance. @State relocates that state into runtime-owned storage associated with a stable graph identity.

1.2 wrappedValue vs projectedValue ($)

The property wrapper exposes two interfaces:

  • wrappedValue — the current value (T). Reading it during body establishes a dependency so the view invalidates when the value changes.
  • projectedValue — a Binding<T> (spelled $name) that forwards reads and writes back into the same underlying storage without copying ownership of the object.
struct CounterView: View {
    @State private var count: Int = 0

    var body: some View {
        VStack {
            Text("\(count)")           // reads wrappedValue
            Stepper("Adjust", value: $count, in: 0...100)  // Binding via $
        }
    }
}

Here, Stepper mutates through the binding; SwiftUI writes the new integer into the same heap-backed slot and schedules a targeted invalidation for views that depended on count.

1.3 Why @State should be private

Marking @State private is not a mere style guideline. If you expose mutable state as internal or public, other types can write to it without going through SwiftUI’s observation machinery in the intended way, and you encourage architectural leaks where parent views reach into children to flip bits. The framework also assumes state is encapsulated so dependency edges remain local and predictable.

Staff-level trap

Exposing @State breaks encapsulation and makes refactors painful. Interviewers often probe whether you understand that child state is not a substitute for a model object — it is UI-local transient state (selection, expansion, field draft text) with a narrow API surface.

1.4 Struct recreation vs persistent state

Every time SwiftUI calls body, it constructs a fresh CounterView value. If count lived only in that struct’s memory, it would reset to 0 constantly. Because the attribute graph associates storage with the view’s identity (type + position in the hierarchy + explicit id when used), the heap slot survives those re-instantiations.

struct LifecycleDemo: View {
    @State private var tick: Int = 0
    @State private var refreshToken: Int = 0

    var body: some View {
        VStack {
            Text("tick = \(tick)")
            Button("Bump parent token") { refreshToken += 1 }
            ChildWithLocalState(parentToken: refreshToken)
        }
        .onAppear { tick += 1 }  // runs when inserted; state persists across parent redraws
    }
}

struct ChildWithLocalState: View {
    let parentToken: Int
    @State private var local: Int = 0

    var body: some View {
        Button("local=\(local), token=\(parentToken)") { local += 1 }
    }
}

Changing refreshToken forces the parent to rebuild; ChildWithLocalState receives a new struct instance, but local remains tied to the child’s graph identity and does not reset unless the child is removed from the hierarchy or its identity changes (for example via .id(...)).

1.5 When @State is the wrong tool

Use @State for small value-semantic UI state owned by a single view. For reference-type models shared across screens, reach for @StateObject, @Observable, or dependency injection — not a growing pile of @State in the root view.

1.6 DynamicProperty, container relationships, and why “the struct is not the view”

Under the ABI surface, SwiftUI’s stateful property wrappers conform to protocols such as DynamicProperty. Before body runs, SwiftUI walks the view’s stored properties and calls into the wrapper so each one can update its connection to graph-backed storage. That is how a brand-new struct value still “finds” the same heap slot: the wrapper participates in a setup phase that binds the ephemeral value to a persistent graph location.

Ephemeral value semantics Persistent graph semantics -------------------------- --------------------------- let v1 = CounterView() Attribute graph node ID: N let v2 = CounterView() // new struct State storage keyed by N | ^ | same logical view in hierarchy | +---- DynamicProperty update() -----------+---- resolves to same box

Interviewers often expect you to articulate the consequence: you cannot rely on self identity in a View struct for persistence. Only framework-managed storage (@State, @StateObject, external models, environment) survives. This is also why capturing self incorrectly inside escaping closures can produce stale reads unless you capture the underlying value or use the binding.

Implementation note

Exact method names and call order are private to SwiftUI and change between OS releases. The durable mental model is: wrappers reconcile value-type views with reference-type graph state. Saying “@State is stored in the struct” is a red flag at staff level.

1.7 Initialization rules and the single source of truth

@State initializers run when SwiftUI first allocates storage for that identity. Supplying @State private var text = "" sets the initial heap value once. Later, if you attempt to “re-initialize” by changing code and hot-reloading, or by reconstructing a view with a different generic parameter so its identity changes, you will see resets — not because @State is broken, but because identity changed or storage was recreated.

// Initial value only matters on first creation of this view identity.
struct KeyedEditor: View {
    let recordID: UUID
    @State private var draft: String = ""

    var body: some View {
        TextField("Draft", text: $draft)
            .onChange(of: recordID) { _, _ in
                draft = ""   // explicit reset when logical record changes
            }
    }
}

Staff-level apps almost always pair identity changes (.id(recordID)) with explicit model-driven resets, rather than expecting magic synchronization.

1.8 Heap allocation and the “first touch” timeline (mental model)

SwiftUI does not document the private box type that backs @State, but the observable behavior implies a lazy, identity-keyed allocation: when a view node is first inserted into the live graph, the runtime reserves storage for each DynamicProperty field. Subsequent body evaluations run against new struct values, yet the wrapper’s update hook reattaches to the same box.

Timeline: @State private var n = 0 --------------------------------- [Insert] View identity N appears in graph | +--> allocate State storage for `n` (initial value 0) | [Body #1] CounterView() value created on stack | +--> @State wrapper update: bind stack value -> heap box for N | [User taps] set n = 1 via Binding / direct write | +--> mark dependents dirty; schedule re-render | [Body #2] NEW CounterView() stack value | +--> @State wrapper update: SAME heap box, n still 1

The interview-critical phrase: ownership of the value is transferred to the framework for the lifetime of that graph identity. You should not think “my struct holds an Int,” but “my struct holds a handle that reads through to graph storage.”

1.9 Property wrapper spelling: wrappedValue, projectedValue, and underscored access

Under the hood, @State var x is lowered to a stored property of type State<T> (name illustrative). The compiler synthesizes accessors so that x reads wrappedValue and $x reads projectedValue. In advanced scenarios (custom initializers, property wrapper composition), you may see _x — the wrapper’s projected base — exactly as with any Swift property wrapper.

// Conceptual de-sugaring (simplified)
struct CounterView: View {
    private var _count = State<Int>(initialValue: 0)

    private var count: Int {
        get { _count.wrappedValue }
        nonmutating set { _count.wrappedValue = newValue }
    }

    // $count == _count.projectedValue  (Binding<Int>)
}
Why interviewers probe this

Demonstrating that @State is a property wrapper with wrapper/projected values proves you understand Swift’s lowering — and explains why $ is not “magic syntax” but projectedValue vending a Binding.

2. @Binding — Projected two-way connection

A Binding<T> is a lightweight pair of closures (get + set) that describes how to read and write a piece of data without owning it. The projected value of @State, @Bindable wrappers, and many other sources produces such a binding.

2.1 Binding as a struct

Conceptually (simplified), Binding forwards to underlying storage:

// Conceptual mental model — not exact ABI
struct Binding<Value> {
    var get: () -> Value
    var set: (Value) -> Void

    var wrappedValue: Value {
        get { get() }
        nonmutating set { set(newValue) }
    }
}

Controls like Toggle, TextField, and Sheet modifiers accept Binding so they can both display and mutate source-of-truth state held elsewhere.

2.2 Custom bindings with Binding(get:set:)

struct SettingsPanel: View {
    @Binding var fontSize: CGFloat

    var body: some View {
        Slider(
            value: Binding(
                get: { fontSize },
                set: { fontSize = CGFloat(round($0)) }
            ),
            in: 12...24
        )
    }
}

Custom bindings are how you implement derived UI state, clamping, side effects on write, or bridging to non-SwiftUI APIs — while keeping a single authoritative store.

Pattern

Prefer pushing Binding downward rather than callbacks for simple two-way sync; it composes with SwiftUI controls and keeps mutation sites uniform.

2.3 $state ergonomics: projected value in practice

For @State, @FocusState, and other wrappers that vend a Binding, the prefix $ is pure sugar for the property wrapper’s projectedValue. It does not allocate a new model; it hands the control a capability to mutate the same storage SwiftUI already tracks.

struct FormDemo: View {
    @State private var username: String = ""
    @State private var accepted: Bool = false

    var body: some View {
        VStack {
            TextField("Username", text: $username)   // Binding<String>
            Toggle("Accept", isOn: $accepted)        // Binding<Bool>
        }
    }
}

2.4 Constant and optional bindings

.constant(true) fixes a value for previews or disabled controls. Binding helpers unwrap optionals or supply defaults — useful when bridging legacy models into SwiftUI without rewriting everything at once.

2.5 Binding transformations: map, optional chaining, and transaction

Binding is a small value type; you can derive new bindings that apply forward/backward transforms. This is the SwiftUI analogue of lens-style updates in other reactive UI frameworks — keep the authoritative store untouched while presenting a control-friendly projection.

extension Binding {
    func map<T>(
        get: @escaping (Value) -> T,
        set: @escaping (T, Value) -> Value
    ) -> Binding<T> {
        Binding<T>(
            get: { get(self.wrappedValue) },
            set: { self.wrappedValue = set($0, self.wrappedValue) }
        )
    }
}

In application code, prefer standard library / SwiftUI helpers where available; in interviews, articulating that Binding is get/set closures proves you can implement validation, clamping, and reversible edits without duplicating source-of-truth fields.

3. @StateObject vs @ObservedObject — Ownership and the classic bug

Both wrappers subscribe to ObservableObject’s objectWillChange publisher (Combine) and invalidate dependent views. The critical difference is who creates and owns the object instance across view updates.

3.1 @StateObject — SwiftUI owns the lifetime

When you write @StateObject private var model = MyModel(), SwiftUI constructs (or retrieves) the object once per view identity and retains it for you. Subsequent body evaluations reuse that instance. This mirrors @State but for reference types conforming to ObservableObject.

3.2 @ObservedObject — You own the lifetime

@ObservedObject var model: MyModel expects the caller to supply a stable object. SwiftUI does not promise to preserve whatever instance you passed in the initializer across arbitrary graph updates; the view’s struct is recreated, and if your initializer runs again and constructs a new MyModel(), you have a bug.

The classic bug

Using @ObservedObject with a default value or inline construction in the child’s memberwise initializer causes a fresh model on every parent re-render — resetting UI, losing in-flight work, duplicating network requests, and confusing testers.

// BUGGY: new ViewModel every time Parent redraws
struct BrokenChild: View {
    @ObservedObject var vm: ItemViewModel

    init(itemID: UUID) {
        _vm = ObservedObject(wrappedValue: ItemViewModel(id: itemID))
    }

    var body: some View { Text(vm.title) }
}

// FIXED: parent owns creation; child observes
struct Parent: View {
    @StateObject private var vm = ItemViewModel(id: UUID())
    var body: some View { HealthyChild(vm: vm) }
}

struct HealthyChild: View {
    @ObservedObject var vm: ItemViewModel
    var body: some View { Text(vm.title) }
}

3.3 Memory and lifecycle

  • @StateObject — retained by the view graph; released when the view is torn down (subject to navigation stack and hosting rules).
  • @ObservedObject — SwiftUI holds a weak-ish conceptual relationship to “whatever you passed”; if you recreate models carelessly, ARC may deallocate old instances while subscribers still expect them — or you may leak if bindings keep cycles alive.

ARC angle: the observable object is a class; someone must keep it alive. @StateObject ensures the owning view subtree participates in that retention. @ObservedObject does not transfer ownership — if the parent stops retaining the model and nothing else holds it, the instance can deallocate even though a child view’s last body still referenced it. In practice, parent-created models are retained by the parent’s property list or environment; the wrapper only subscribes to changes.

Re-creation angle: if a child incorrectly constructs ItemViewModel() inside init for @ObservedObject, every parent invalidation produces a new object graph: new Combine subscriptions, new network tasks, lost form state. @StateObject’s closure is evaluated once per view identity (with documented exceptions around explicit id changes), which is why Apple positions it as the fix for “created in the view.”

3.4 @StateObject(wrappedValue:) vs injected instances — initializer recipes

// StateObject: pass factory OR value — still "owned" by view graph rules
struct Screen: View {
    let id: UUID
    @StateObject private var vm: ItemViewModel

    init(id: UUID) {
        self.id = id
        _vm = StateObject(wrappedValue: ItemViewModel(id: id))
    }

    var body: some View { Text(vm.title) }
}

// ObservedObject: NEVER allocate here unless idempotently cached
struct Row: View {
    @ObservedObject var vm: RowViewModel
    var body: some View { Text(vm.title) }
}

3.5 Decision flowchart (ASCII)

Who creates the ObservableObject? | +--- The CHILD view creates it? | | | YES --> @StateObject (SwiftUI retains) | | | NO --> continue | +--- The PARENT (or DI container) creates it? | YES --> Pass in; child uses @ObservedObject | NO --> Rethink architecture (singleton? environment?)
Navigation stacks

With NavigationStack and value-based navigation, view identity rules still apply. If you hoist a @StateObject at the wrong level, you can accidentally share one view model across destinations — or recreate it too often. Always tie object lifetime to the intended screen scope.

4. @EnvironmentObject — Type-keyed ambient state

@EnvironmentObject is a specialized path into SwiftUI’s Environment: at runtime, SwiftUI looks up a value by dynamic type (the concrete ObservableObject type) and injects it into body evaluation.

4.1 Injection and lookup

@main
struct MyApp: App {
    @StateObject private var session = SessionStore()

    var body: some Scene {
        WindowGroup {
            RootView()
                .environmentObject(session)
        }
    }
}

struct ProfileTab: View {
    @EnvironmentObject private var session: SessionStore

    var body: some View {
        Text(session.userName)
    }
}

If no ancestor called .environmentObject(session) (or equivalent), accessing @EnvironmentObject triggers a runtime failure — SwiftUI cannot synthesize a default.

Crash if missing

Unlike @Environment values with defaults, @EnvironmentObject is effectively required. Previews and tests must inject mocks explicitly.

4.2 Scoping and overrides

Environment propagates down the tree. A subtree can replace an object by applying another .environmentObject — useful for feature-local session overrides or A/B test buckets. The nearest ancestor wins, similar to CSS cascade (conceptually).

FeatureRoot()
    .environmentObject(scopedStore)  // shadows outer store for this subtree

4.3 When not to use environment objects

Global ambient state hides dependencies (harder to test, harder to reason about data flow). Prefer explicit initializer parameters for shallow trees; reserve EnvironmentObject for truly cross-cutting concerns (session, theme, feature flags) with narrow protocols or concrete types per domain.

5. @Observable macro (iOS 17+) — Under the hood

Swift 5.9 introduced the Observation framework. The @Observable macro transforms a class at compile time so accesses to individual stored properties can be tracked with lower granularity than objectWillChange.send() blasting every subscriber.

5.1 What the macro generates (illustrative)

Exact synthesized code evolves with compiler versions; interview answers should describe behavior and registrar participation rather than memorizing every line. A faithful mental model:

// You write:
import Observation

@Observable final class Cart {
    var items: [Item] = []
    var couponCode: String?
}

// Compiler conceptually expands toward (simplified pseudocode):
final class Cart {
    private let _$observationRegistrar = ObservationRegistrar()

    var items: [Item] {
        get {
            _$observationRegistrar.access(self, keyPath: \.items)
            return _items
        }
        set {
            _$observationRegistrar.willSet(self, keyPath: \.items)
            _items = newValue
            _$observationRegistrar.didSet(self, keyPath: \.items)
        }
    }
    private var _items: [Item] = []
    // ... similar for couponCode ...
}

5.2 ObservationRegistrar

The registrar records which observers touched which properties during a tracking pass. On mutation, only observers that depended on the mutated key path are notified — enabling fine-grained invalidation when SwiftUI integrates with this machinery.

5.3 Property-level vs object-level observation

@Published on an ObservableObject still ultimately funnels through objectWillChange before the concrete property change is visible — subscribers often broaden their invalidation. @Observable aims for per-field dependency lists in the observation system, which SwiftUI can use to narrow updates.

Integration point

SwiftUI’s runtime registers dependencies while reading body. When those reads touch @Observable properties, the framework links the view’s graph attributes to observation records. After a write, only affected views revisit dependent subtrees.

5.4 ObservationRegistrar, access lists, and how withObservationTracking fits

The registrar maintains associations between an observable instance and the observers that have read particular storage. Conceptually, each tracked property access during a withObservationTracking apply pass appends the observer to an access list for the touched key paths. When the runtime executes a setter, it consults those lists to decide which observers need notification.

withObservationTracking { apply } onChange: { ... } | |-- apply() runs once immediately | | | +--> reads cart.items --> registrar notes: observer O depends on \.items | +-- later: cart.items = ... --> onChange fires for O | +--> IMPORTANT: tracking is NOT automatically re-armed SwiftUI re-establishes dependencies on next body evaluation Your manual code must recurse if you want continuous watching

SwiftUI’s integration is analogous to “always recurse”: every time a view body is evaluated, dependency discovery runs again, so you inherit correct incremental behavior without writing the recursive harness yourself. That is the bridge between low-level Observation APIs and day-to-day declarative UI.

Do not confuse layers

In interviews, clearly separate: (1) the macro-generated accessors calling into ObservationRegistrar, (2) withObservationTracking for ad-hoc observation outside SwiftUI, and (3) SwiftUI’s own body-driven re-tracking. Mixing these levels signals confusion about ownership of the subscription lifecycle.

6. @Bindable — Bindings for @Observable models

Reference-type @Observable models are not property wrappers on the struct; you typically hold them as let model: Cart or @State / @Bindable depending on ownership. SwiftUI introduced @Bindable to opt into binding projection for observable objects without adopting the older ObservableObject + @Published pattern.

6.1 Why @Bindable exists

Controls want Binding. Your model is a class instance with mutable properties. @Bindable tells the compiler to synthesize accessors that produce Bindings to individual properties while preserving observation.

struct CartEditor: View {
    @Bindable var cart: Cart

    var body: some View {
        TextField("Coupon", text: Binding(
            get: { cart.couponCode ?? "" },
            set: { cart.couponCode = $0.isEmpty ? nil : $0 }
        ))
    }
}

When properties are non-optional value types, @Bindable also enables the same $model.property spelling people associate with @State, but the backing mechanism is different: you are projecting into an observable object’s storage, not a dedicated per-view heap box.

@Observable final class UserDraft {
    var displayName: String = ""
}

struct ProfileForm: View {
    @Bindable var draft: UserDraft

    var body: some View {
        TextField("Name", text: $draft.displayName)  // Binding from @Bindable
    }
}
Ownership nuance

@Bindable enables projections at the use site; it does not answer “who retains this model?” If a parent recreates the observable instance on every outer state change, children still bind correctly to the current instance — but you may wipe in-flight edits or duplicate side effects. Pair @Bindable with deliberate scoping: feature roots own models; leaves receive stable references.

6.2 Difference from @Binding

Wrapper What it wraps Typical use
@Binding An existing Binding<T> passed from parent Child receives two-way linkage without owning storage
@Bindable An @Observable (or bindable) object Derive $property bindings from reference model fields

7. Observation tracking — withObservationTracking

Lower-level APIs in the Observation framework let you run a closure and record which observable properties it read. SwiftUI uses analogous machinery internally when evaluating view bodies against @Observable types.

7.1 Single-shot tracking

withObservationTracking runs the apply closure once to establish dependencies, and registers an onChange handler to fire when tracked properties mutate. After that callback runs, you must re-enter tracking if you want future notifications — tracking is not a persistent Combine-style subscription unless you re-arm it.

Timeline (manual Observation tracking) -------------------------------------- t0: withObservationTracking { read A,B } onChange: handler | +--> dependencies {A,B} registered for this generation t1: write to A only | +--> onChange(handler) fires ONCE | +--> unless handler calls withObservationTracking again, further writes will NOT notify this observer 
import Observation

func watchCart(_ cart: Cart, on update: @escaping () -> Void) {
    func recurse() {
        withObservationTracking {
            _ = cart.items.count   // establish dependency
        } onChange: {
            update()
            recurse()          // re-register for next generation
        }
    }
    recurse()
}

7.2 How SwiftUI re-registers

Each time SwiftUI evaluates a view’s body, it effectively performs a fresh dependency discovery pass for observable models touched during that evaluation. That is why you do not manually call withObservationTracking in typical app code — the framework wraps your body evaluation.

Mental model

Think “pull-based + cached incremental”: a body read pulls current model fields; the runtime remembers which fields participated so subsequent writes enqueue minimal recomputation.

8. Why @Published tends to fan out invalidation

Combine-driven ObservableObject publishes objectWillChange before mutations visible to observers. Many SwiftUI view bodies capture the whole object or re-read multiple properties; in practice, dependency tracking may be coarser than the ideal “only the label that changed.”

8.1 ObservableObject + @Published: broadcast flavor

ObservableObject root | | objectWillChange.send() | v +-----+-----+-----+ | | | | View View View View (may all re-evaluate body) ...

8.2 @Observable: property-scoped flavor (conceptual)

@Observable Cart |-- items .............. observers A, C '-- couponCode .......... observer B mutation to couponCode | v only subtree B invalidated (idealized; implementation may vary slightly)

8.3 Performance narrative for interviews

Benchmarks depend on view complexity, but the asymptotic story matters: coarse invalidation multiplies work with large body trees; granular observation reduces redundant layout and diffing. Modern SwiftUI prefers @Observable for new model code partly for this scalability win.

Same screen, two child views ═══════════════════════════════ ObservableObject + @Published on "ProfileModel" ───────────────────────────────────────────── @Published var avatarURL @Published var bio │ │ changing bio triggers objectWillChange v Parent body + AvatarView + BioEditor may all reconsider work (exact fan-out depends on what parent read) @Observable Profile ─────────────────── avatarURL ◄── AvatarView body depends only on this bio ◄── BioEditor depends only on this mutating bio │ v AvatarView ideally skips (no read dependency on bio)
Nuance

Even with @Observable, if a view’s body reads ten properties, it still depends on all ten. The win is that a different view that only read couponCode will not churn when items alone changes.

8.4 Reducing accidental “read the world” dependencies

Staff-level discussions often turn to dependency minimization: pass only the sub-values a child needs (strings, IDs, lightweight structs) rather than the entire model, so body evaluation does not eagerly touch unrelated observable fields. With @Observable, splitting read surfaces is even more valuable because it maps directly to finer invalidation.

10. State architecture for large SwiftUI apps

Architect must-know: avoid the god object

At L6, interviewers probe whether you can scale SwiftUI beyond demo apps. A single AppState with one hundred @Published properties becomes a change amplifier: any mutation risks fan-out, merge conflicts in a large team, and untestable entanglement. Split state into focused domains — authentication, navigation, feature modules — with explicit composition at the root.

  • Domain stores: one observable object or @Observable class per bounded context (e.g., CartStore, InboxStore).
  • Routing state: keep navigation models separate from business entities; use value types where possible.
  • Environment vs injection: reserve environment for truly global services; pass explicit dependencies into feature roots for clarity.
  • Feature modules: expose narrow protocols to siblings; never reach through the graph to “the one true singleton.”

For 50+ screens, adopt a pattern the team can repeat: a feature root View owns a @StateObject coordinator, children use @ObservedObject or bindings, and cross-feature events flow through an app-level event bus or coordinator — not ad-hoc environment writes from deep leaves.

10.2 Migration strategy (ObservableObject → @Observable)

Migrate leaf models first where win is clearest; keep adapters at module boundaries until Combine-based SDKs disappear. Run both styles temporarily behind protocols if you must share code with extensions or widgets.

Closing framing

Staff engineers explain state as a graph problem: identities own storage, bindings are projections, objects have lifetimes, and observation determines work. Master those four axes and SwiftUI’s property wrappers become predictable rather than magical.

10.3 Quick reference — interview questions

Question Answer kernel
Why doesn’t my struct reset lose @State? State is keyed by view identity in the attribute graph, not by struct instance address.
When must I use @StateObject? When the view introduces the observable object instance that must survive parent redraws.
What does $foo mean? Projected Binding (or bindable projection) for two-way UI linkage.
How does @Observable differ from @Published? Field-level access tracking vs object-broadcast objectWillChange semantics.
Why did my preview crash? Missing .environmentObject / required environment value.
