1. SwiftData Overview & Architecture

SwiftData, introduced at WWDC 2023 (iOS 17+), is Apple's declarative persistence framework that replaces the verbose Core Data API with Swift macros, native types, and first-class SwiftUI integration. Under the hood, it is built entirely on Core Data — the same NSPersistentContainer, NSManagedObjectContext, and SQLite store power the engine.

Core Principles

  • Macro-driven schema — @Model replaces .xcdatamodeld files and NSManagedObject subclasses
  • Type-safe queries — #Predicate macro replaces NSPredicate string-based queries
  • Swift-native types — use Array, Set, Optional, enums, and Codable structs directly
  • SwiftUI-first — @Query property wrapper auto-refreshes views on data changes
  • Observable integration — models conform to Observable, enabling fine-grained SwiftUI updates

Deep Dive — Same SQLite Under the Hood

SwiftData's ModelContainer wraps NSPersistentContainer. You can access the underlying Core Data stack via container.configurations. The SQLite file is identical in format — you can even open a SwiftData store with Core Data tooling. This means WAL journaling, VACUUM, and all SQLite pragmas still apply.

        }

    }
}

2. @Model Macro Internals

The @Model macro is the cornerstone of SwiftData. Applied to a Swift class, it transforms the class into a persistent model by generating conformance to PersistentModel and Observable, synthesizing backing storage, and registering the schema with the runtime.

What the Macro Generates

Given this source declaration:

@Model
final class Article {
    var title: String
    var body: String
    var publishedAt: Date?
    var viewCount: Int = 0
}

The macro expands roughly to:

// Expanded @Model macro output (simplified)
final class Article: PersistentModel, Observable {

    // Backing storage managed by SwiftData runtime
    private var _backingData: any BackingData<Article>

    // Each property becomes a computed property with get/set
    var title: String {
        get {
            _$observationRegistrar.access(self, keyPath: \.title)
            return getValue(forKey: "title")
        }
        set {
            _$observationRegistrar.withMutation(of: self, keyPath: \.title) {
                setValue(forKey: "title", to: newValue)
            }
        }
    }

    // ... similar for body, publishedAt, viewCount

    static var schemaMetadata: [Schema.PropertyMetadata] {
        [
            .init(name: "title", keypath: \Article.title),
            .init(name: "body", keypath: \Article.body),
            .init(name: "publishedAt", keypath: \Article.publishedAt, isOptional: true),
            .init(name: "viewCount", keypath: \Article.viewCount, defaultValue: 0),
        ]
    }
}

@Attribute Options

Fine-tune property storage with @Attribute:

@Model
final class User {
    // Unique constraint — upsert on conflict
    @Attribute(.unique)
    var email: String

    var name: String

    // Store large data externally (binary blob outside SQLite row)
    @Attribute(.externalStorage)
    var avatarData: Data?

    // Encrypt at rest using data protection
    @Attribute(.encrypt)
    var socialSecurityNumber: String?

    // Index for Spotlight search
    @Attribute(.spotlight)
    var displayName: String

    // Custom value transformer for non-standard types
    @Attribute(.transformable(by: "ColorTransformer"))
    var favoriteColor: UIColor?

    // Combine multiple attributes
    @Attribute(.unique, .spotlight)
    var username: String
}

@Transient — Excluding Properties from Persistence

@Model
final class Player {
    var name: String
    var highScore: Int

    // Not persisted — computed at runtime only
    @Transient
    var isCurrentlyPlaying: Bool = false

    // @Transient properties MUST have a default value
    @Transient
    var cachedRank: Int? = nil
}
Pitfall — @Transient Requires Default Values

Every @Transient property must have a default value. Without one, the compiler will error because SwiftData cannot hydrate the property from the store. This is a common source of confusion when migrating from Core Data where transient properties don't require defaults.

3. ModelContainer & ModelContext

ModelContainer is the top-level object that owns the persistence stack. ModelContext is the scratchpad where you create, read, update, and delete model instances. Together they mirror Core Data's NSPersistentContainer and NSManagedObjectContext.

ModelContainer Configuration

// Basic — auto-infers schema from model types
let container = try ModelContainer(
    for: Article.self, User.self, Tag.self
)

// Full configuration
let schema = Schema([Article.self, User.self, Tag.self])

let config = ModelConfiguration(
    "Production",
    schema: schema,
    isStoredInMemoryOnly: false,
    allowsSave: true,
    groupContainer: .identifier("group.com.myapp.shared"),
    cloudKitDatabase: .private("iCloud.com.myapp.data")
)

let container = try ModelContainer(
    for: schema,
    configurations: [config]
)

ModelContext — The Scratchpad

let context = container.mainContext

// INSERT
let article = Article(title: "SwiftData Deep Dive", body: "...")
context.insert(article)

// DELETE
context.delete(article)

// FETCH
let descriptor = FetchDescriptor<Article>(
    predicate: #Predicate { $0.viewCount > 100 },
    sortBy: [SortDescriptor(\.publishedAt, order: .reverse)]
)
let popular = try context.fetch(descriptor)

// SAVE — explicit save (usually not needed with autosave)
try context.save()

// COUNT — without fetching objects
let count = try context.fetchCount(descriptor)

Autosave Behavior

Deep Dive — Autosave

The mainContext provided by SwiftUI's .modelContainer() modifier has autosave enabled by default. This means changes are automatically persisted at system-determined intervals (typically on runloop idle). You do not need to call save() in normal SwiftUI usage. However, background contexts created manually have autosave disabled — you must call save() explicitly.

SwiftUI Setup

@main
struct MyApp: App {
    var body: some Scene {
        WindowGroup {
            ContentView()
        }
        // Simple — single model type
        .modelContainer(for: Article.self)
    }
}

// Access in any SwiftUI view
struct ContentView: View {
    @Environment(\.modelContext) private var context

    var body: some View {
        Button("Add Article") {
            let article = Article(title: "New", body: "Content")
            context.insert(article)
        }
    }
}

UIKit Setup

class AppDelegate: UIResponder, UIApplicationDelegate {
    var container: ModelContainer!

    func application(
        _ application: UIApplication,
        didFinishLaunchingWithOptions launchOptions: [UIApplication.LaunchOptionsKey: Any]?
    ) -> Bool {
        container = try! ModelContainer(for: Article.self)
        return true
    }
}

class ArticleListViewController: UIViewController {
    private var context: ModelContext!

    override func viewDidLoad() {
        super.viewDidLoad()
        let appDelegate = UIApplication.shared.delegate as! AppDelegate
        context = ModelContext(appDelegate.container)
    }

    func fetchArticles() throws -> [Article] {
        let descriptor = FetchDescriptor<Article>(
            sortBy: [SortDescriptor(\.publishedAt, order: .reverse)]
        )
        return try context.fetch(descriptor)
    }
}

Background Context for Heavy Work

// Create a background context for batch operations
let backgroundContext = ModelContext(container)
backgroundContext.autosaveEnabled = false

Task.detached {
    for data in largeDataset {
        let article = Article(title: data.title, body: data.body)
        backgroundContext.insert(article)
    }
    try backgroundContext.save()
}
Pitfall — ModelContext Is Not Sendable

ModelContext is not Sendable. You must create a new ModelContext on the thread/task where you intend to use it. Passing a context across actor boundaries will cause data races. In Swift 6 strict concurrency mode, the compiler will flag this as an error.

4. @Query Property Wrapper

@Query is SwiftData's SwiftUI-integrated property wrapper that automatically fetches data, observes changes, and triggers view updates. It replaces the combination of NSFetchedResultsController + @FetchRequest from Core Data.

Basic Usage

struct ArticleListView: View {
    // Fetch all articles, sorted by date descending
    @Query(sort: \.publishedAt, order: .reverse)
    private var articles: [Article]

    var body: some View {
        List(articles) { article in
            ArticleRow(article: article)
        }
    }
}

Various @Query Configurations

// 1. Simple — all items, no filter
@Query var allItems: [Item]

// 2. With predicate filter
@Query(
    filter: #Predicate<Article> { $0.viewCount > 50 },
    sort: \.publishedAt,
    order: .reverse
)
private var popularArticles: [Article]

// 3. Multiple sort descriptors
@Query(sort: [
    SortDescriptor(\Article.publishedAt, order: .reverse),
    SortDescriptor(\Article.title)
])
private var sortedArticles: [Article]

// 4. With fetch limit
@Query(
    filter: #Predicate<Article> { $0.viewCount > 1000 },
    sort: \.viewCount,
    order: .reverse
)
private var topArticles: [Article]

// 5. With animation
@Query(sort: \.title, animation: .default)
private var animatedArticles: [Article]

// 6. Using FetchDescriptor for full control
@Query(FetchDescriptor<Article>(
    predicate: #Predicate { $0.publishedAt != nil },
    sortBy: [SortDescriptor(\.publishedAt, order: .reverse)],
    fetchLimit: 20
))
private var recentPublished: [Article]

Dynamic Queries with Init

struct FilteredArticleList: View {
    @Query private var articles: [Article]

    init(minViews: Int, sortOrder: SortOrder) {
        let predicate = #Predicate<Article> { article in
            article.viewCount >= minViews
        }
        _articles = Query(
            filter: predicate,
            sort: \.publishedAt,
            order: sortOrder
        )
    }

    var body: some View {
        List(articles) { article in
            Text(article.title)
        }
    }
}

Comparison with NSFetchedResultsController

Feature @Query (SwiftData) NSFetchedResultsController
UI framework SwiftUI only UIKit (with delegate)
Change tracking Automatic via Observable Delegate callbacks
Sections Limited (iOS 17 lacks built-in) Full section support
Batch updates Automatic diff Manual with beginUpdates/endUpdates
Animation Built-in animation parameter Manual UITableView animations
Predicate type #Predicate (type-safe) NSPredicate (string-based)
Best Practice — Keep @Query in Leaf Views

Place @Query in the smallest possible view to minimize re-renders. If a parent view has a @Query, every child rebuilds when data changes. Extract query-dependent UI into a dedicated child view. This mirrors the Core Data pattern of keeping NSFetchedResultsController close to the table/collection view.

5. Predicate Macro

The #Predicate macro provides type-safe, compile-time-checked query predicates. It replaces NSPredicate's stringly-typed format strings with native Swift expressions that the compiler validates.

Basic Syntax

// Simple comparison
let highViewCount = #Predicate<Article> { article in
    article.viewCount > 100
}

// String operations
let titleSearch = #Predicate<Article> { article in
    article.title.contains("Swift")
}

// Optional handling
let published = #Predicate<Article> { article in
    article.publishedAt != nil
}

// Compound predicates with logical operators
let featured = #Predicate<Article> { article in
    article.viewCount > 500 && article.publishedAt != nil
}

// OR conditions
let interesting = #Predicate<Article> { article in
    article.viewCount > 1000 || article.title.contains("SwiftUI")
}

Complex Predicate Examples

// Date range query
let startDate = Calendar.current.date(byAdding: .month, value: -1, to: Date())!
let recentArticles = #Predicate<Article> { article in
    if let date = article.publishedAt {
        return date > startDate
    }
    return false
}

// Nested relationship predicate
let authorArticles = #Predicate<Article> { article in
    article.author?.name == "John"
}

// Variable capture from outside scope
func searchPredicate(query: String, minViews: Int) -> Predicate<Article> {
    #Predicate<Article> { article in
        article.title.localizedStandardContains(query) &&
        article.viewCount >= minViews
    }
}

// Enum-based filtering (raw value comparison)
let draftFilter = #Predicate<Article> { article in
    article.statusRawValue == "draft"
}

Supported Operations

Category Supported Operations
Comparison ==, !=, <, >, <=, >=
Logical &&, ||, !
String contains, localizedStandardContains, hasPrefix, hasSuffix
Optional != nil, == nil, optional chaining
Collections contains on arrays
Arithmetic +, -, *, /

Comparison with NSPredicate

// NSPredicate — stringly typed, crashes at runtime on typos
let nsPredicate = NSPredicate(
    format: "viewCount > %d AND title CONTAINS[cd] %@",
    100, "Swift"
)

// #Predicate — type-safe, compile-time checked
let predicate = #Predicate<Article> { article in
    article.viewCount > 100 && article.title.contains("Swift")
}
// Typo in property name → compiler error, not runtime crash
Pitfall — #Predicate Limitations

Not all Swift expressions work inside #Predicate. Key limitations:

  • No switch statements — only if/else
  • No custom function calls — only built-in operations
  • No map, filter, reduce on collections
  • No string interpolation
  • No regex matching
  • Enum comparisons require raw value workarounds

These limitations exist because the macro must translate Swift expressions into SQL-compatible operations that SQLite can execute.

6. SortDescriptor

SwiftData uses the Swift standard library's SortDescriptor type (not NSSortDescriptor). It provides KeyPath-based, type-safe sorting that integrates with @Query and FetchDescriptor.

Basic Usage

// Single sort — ascending by title
let byTitle = SortDescriptor(\Article.title)

// Descending order
let byDateDesc = SortDescriptor(\Article.publishedAt, order: .reverse)

// Multiple sort descriptors — primary then secondary
let sorts: [SortDescriptor<Article>] = [
    SortDescriptor(\.viewCount, order: .reverse),  // primary: most viewed
    SortDescriptor(\.title)                         // secondary: alphabetical
]

// In @Query
@Query(sort: [
    SortDescriptor(\Article.publishedAt, order: .reverse),
    SortDescriptor(\Article.title)
])
private var articles: [Article]

// In FetchDescriptor
var descriptor = FetchDescriptor<Article>(
    sortBy: [SortDescriptor(\.publishedAt, order: .reverse)]
)
descriptor.fetchLimit = 50

SortDescriptor vs NSSortDescriptor

Feature SortDescriptor (SwiftData) NSSortDescriptor (Core Data)
Type safety KeyPath-based, compile-time String key, runtime
Generic Yes — SortDescriptor<T> No — untyped
Custom comparator Via SortComparator protocol Via selector or comparator block
Foundation Swift standard library Objective-C Foundation
Bridging Can convert to NSSortDescriptor Native

Dynamic Sort Switching

struct ArticleListView: View {
    @State private var sortOrder: SortOrder = .forward
    @State private var sortBy: SortOption = .date

    enum SortOption {
        case date, title, views
    }

    var body: some View {
        SortedArticleList(sortBy: sortBy, order: sortOrder)
    }
}

struct SortedArticleList: View {
    @Query private var articles: [Article]

    init(sortBy: ArticleListView.SortOption, order: SortOrder) {
        let descriptors: [SortDescriptor<Article>]
        switch sortBy {
        case .date:
            descriptors = [SortDescriptor(\.publishedAt, order: order)]
        case .title:
            descriptors = [SortDescriptor(\.title, order: order)]
        case .views:
            descriptors = [SortDescriptor(\.viewCount, order: order)]
        }
        _articles = Query(sort: descriptors)
    }

    var body: some View {
        List(articles) { article in
            Text(article.title)
        }
    }
}

7. Relationship Management

SwiftData infers relationships from model properties. The @Relationship macro provides explicit control over delete rules, inverse relationships, and cascade behavior.

Defining Relationships

@Model
final class Author {
    var name: String
    var email: String

    // To-many: one author has many articles
    @Relationship(deleteRule: .cascade, inverse: \Article.author)
    var articles: [Article] = []

    init(name: String, email: String) {
        self.name = name
        self.email = email
    }
}

@Model
final class Article {
    var title: String
    var body: String
    var publishedAt: Date?
    var viewCount: Int = 0

    // To-one: each article belongs to one author
    var author: Author?

    // Many-to-many: articles can have multiple tags
    @Relationship(inverse: \Tag.articles)
    var tags: [Tag] = []

    init(title: String, body: String) {
        self.title = title
        self.body = body
    }
}

@Model
final class Tag {
    @Attribute(.unique)
    var name: String

    var articles: [Article] = []

    init(name: String) {
        self.name = name
    }
}

Delete Rules

Delete Rules — What happens to related objects when the source is deleted ┌──────────┬────────────────────────────────────────────────────────┐ │ .cascade │ Delete the source → all related objects are deleted │ │ │ Author deleted → all their Articles deleted │ ├──────────┼────────────────────────────────────────────────────────┤ │ .nullify │ Delete the source → relationship set to nil │ │ │ Author deleted → article.author = nil │ ├──────────┼────────────────────────────────────────────────────────┤ │ .deny │ Prevent deletion if related objects exist │ │ │ Cannot delete Author while they have Articles │ ├──────────┼────────────────────────────────────────────────────────┤ │ .noAction│ Do nothing — leaves orphaned references (dangerous) │ │ │ Author deleted → article.author points to nothing │ └──────────┴────────────────────────────────────────────────────────┘ 
@Model
final class Department {
    var name: String

    // Cascade: deleting department deletes all employees
    @Relationship(deleteRule: .cascade)
    var employees: [Employee] = []

    init(name: String) {
        self.name = name
    }
}

@Model
final class Employee {
    var name: String

    // Nullify: deleting employee sets department.manager to nil
    @Relationship(deleteRule: .nullify, inverse: \Department.manager)
    var managedDepartment: Department?

    var department: Department?

    init(name: String) {
        self.name = name
    }
}

Comparison with Core Data Relationships

Aspect SwiftData Core Data

Best Practice — Always Specify Inverse Relationships

SwiftData can infer inverse relationships in simple cases, but explicit inverse: parameters prevent ambiguity when a model has multiple relationships to the same type. Always specify inverses to avoid runtime surprises and ensure correct cascade behavior.

Pitfall — Relationship Arrays Must Have Default Values

To-many relationship properties must be initialized with an empty array (= []). Omitting the default value causes a runtime crash when SwiftData tries to hydrate the object. This is different from Core Data where NSSet relationships are implicitly empty.

8. Migration Strategies

SwiftData provides a structured migration system through VersionedSchema and SchemaMigrationPlan. Lightweight (additive) migrations happen automatically; destructive or transformative changes require explicit migration stages.

Schema Versioning

// Version 1 — original schema
enum SchemaV1: VersionedSchema {
    static var versionIdentifier = Schema.Version(1, 0, 0)

    static var models: [any PersistentModel.Type] {
        [ArticleV1.self]
    }

    @Model
    final class ArticleV1 {
        var title: String
        var body: String

        init(title: String, body: String) {
            self.title = title
            self.body = body
        }
    }
}

// Version 2 — added viewCount and publishedAt
enum SchemaV2: VersionedSchema {
    static var versionIdentifier = Schema.Version(2, 0, 0)

    static var models: [any PersistentModel.Type] {
        [ArticleV2.self]
    }

    @Model
    final class ArticleV2 {
        var title: String
        var body: String
        var viewCount: Int = 0
        var publishedAt: Date?

        init(title: String, body: String) {
            self.title = title
            self.body = body
        }
    }
}

// Version 3 — split body into summary + content
enum SchemaV3: VersionedSchema {
    static var versionIdentifier = Schema.Version(3, 0, 0)

    static var models: [any PersistentModel.Type] {
        [Article.self]
    }

    @Model
    final class Article {
        var title: String
        var summary: String
        var content: String
        var viewCount: Int = 0
        var publishedAt: Date?

        init(title: String, summary: String, content: String) {
            self.title = title
            self.summary = summary
            self.content = content
        }
    }
}

SchemaMigrationPlan

enum ArticleMigrationPlan: SchemaMigrationPlan {
    static var schemas: [any VersionedSchema.Type] {
        [SchemaV1.self, SchemaV2.self, SchemaV3.self]
    }

    static var stages: [MigrationStage] {
        [migrateV1toV2, migrateV2toV3]
    }

    // V1 → V2: lightweight (just adding new columns with defaults)
    static let migrateV1toV2 = MigrationStage.lightweight(
        fromVersion: SchemaV1.self,
        toVersion: SchemaV2.self
    )

    // V2 → V3: custom (splitting body into summary + content)
    static let migrateV2toV3 = MigrationStage.custom(
        fromVersion: SchemaV2.self,
        toVersion: SchemaV3.self,
        willMigrate: { context in
            // Fetch all V2 articles
            let articles = try context.fetch(
                FetchDescriptor<SchemaV2.ArticleV2>()
            )
            for article in articles {
                // Split body: first 200 chars → summary, full → content
                let bodyText = article.body
                article.summary = String(bodyText.prefix(200))
                article.content = bodyText
            }
            try context.save()
        },
        didMigrate: nil
    )
}

Applying the Migration Plan

@main
struct MyApp: App {
    let container: ModelContainer

    init() {
        do {
            container = try ModelContainer(
                for: SchemaV3.Article.self,
                migrationPlan: ArticleMigrationPlan.self
            )
        } catch {
            fatalError("Failed to configure ModelContainer: \(error)")
        }
    }

    var body: some Scene {
        WindowGroup {
            ContentView()
        }
        .modelContainer(container)
    }
}
Migration Flow SchemaV1 SchemaV2 SchemaV3 ┌──────┐ lightweight ┌──────┐ custom ┌──────┐ │ body │ ──────────────→ │ body │ ────────────→ │summary│ │ │ (add cols) │views │ (split) │content│ │ │ │date │ │views │ └──────┘ └──────┘ │date │ └──────┘
Deep Dive — Lightweight vs Custom Migration

Lightweight migrations handle additive changes automatically: adding properties with defaults, making properties optional, adding new model types. Custom migrations are required for: renaming properties, splitting/merging properties, changing types, removing properties without defaults, or any data transformation. Under the hood, SwiftData delegates to Core Data's NSMappingModel and NSMigrationManager.

Pitfall — Migration Testing

Always test migrations with real production data snapshots. A migration that works on a fresh database may fail on a store with millions of rows, NULL values in unexpected columns, or data from older schema versions. Ship migration code behind feature flags and include rollback strategies for critical apps.

SwiftData and Core Data share the same SQLite foundation but differ significantly in API design, developer experience, and maturity. This section provides a comprehensive comparison for architectural decision-making.

Deep Dive — Interoperability

Because SwiftData and Core Data share the same SQLite store format, you can open a SwiftData-created database with Core Data and vice versa. This enables incremental migration: keep Core Data for UIKit screens while adopting SwiftData for new SwiftUI features. Access the underlying NSPersistentContainer via ModelContainer when you need batch operations or other Core Data-only features.

10. Current Limitations

SwiftData is powerful but young. Understanding its limitations is critical for making informed architectural decisions at the Staff/L6 level.

No Batch Operations

Core Data offers NSBatchInsertRequest, NSBatchDeleteRequest, and NSBatchUpdateRequest that operate directly on the SQLite store without loading objects into memory. SwiftData has no equivalent — every operation requires fetching objects into the context, mutating them, and saving. For datasets with 100K+ rows, this is a significant performance gap.

// Core Data — batch delete 50K rows in ~100ms
let batchDelete = NSBatchDeleteRequest(
    fetchRequest: Article.fetchRequest()
)
try context.execute(batchDelete)

// SwiftData — must fetch all objects first (slow for large sets)
let articles = try context.fetch(FetchDescriptor<Article>())
for article in articles {
    context.delete(article)
}
try context.save()

Limited Predicate Operations

  • No SUBQUERY equivalent for complex relationship queries
  • No aggregate functions (@sum, @avg, @count) in predicates
  • No regex matching
  • No ANY/ALL/NONE collection operators
  • Enum filtering requires raw value workarounds

No NSFetchedResultsController Equivalent

@Query works only in SwiftUI views. For UIKit, there is no built-in mechanism for observing query results with section support, incremental updates, or diffable data source integration. You must manually observe context changes or fall back to Core Data's NSFetchedResultsController.

CloudKit Limitations

  • No support for CKShare (shared databases) — only private database
  • No fine-grained conflict resolution — last-write-wins only
  • No progress reporting for sync operations
  • Debugging sync issues requires Core Data CloudKit logging

Performance with Large Datasets

  • No prefetching hints (Core Data's relationshipKeyPathsForPrefetching)
  • No control over faulting behavior
  • No fetchBatchSize equivalent for memory-efficient iteration
  • No NSAsynchronousFetchRequest equivalent

Concurrency Model Differences

// Core Data — explicit context confinement
let bgContext = persistentContainer.newBackgroundContext()
bgContext.perform {
    // Safe: block runs on context's queue
    let articles = try? bgContext.fetch(request)
}

// SwiftData — ModelContext is not Sendable
// Must create context on the target actor/thread
let container = self.container
Task.detached {
    let context = ModelContext(container)
    // Use context only within this task
    let descriptor = FetchDescriptor<Article>()
    let articles = try context.fetch(descriptor)
}
Pitfall — ModelActor for Background Work

For structured background work, use the @ModelActor macro instead of manually creating contexts. It provides proper actor isolation and a dedicated context:

@ModelActor
actor DataImporter {
    // modelExecutor and modelContext are synthesized

    func importArticles(from data: [ArticleDTO]) throws {
        for dto in data {
            let article = Article(title: dto.title, body: dto.body)
            modelContext.insert(article)
        }
        try modelContext.save()
    }
}

// Usage
let importer = DataImporter(modelContainer: container)
try await importer.importArticles(from: dtos)

Other Notable Gaps

  • No abstract entities or entity inheritance
  • No derived attributes
  • No fetch request templates
  • No NSExpressionDescription for computed fetches
  • No NSFetchIndexDescription for custom indexes (beyond .unique)
  • Limited testing support — no built-in in-memory store toggle for unit tests (must use isStoredInMemoryOnly)
Best Practice — Unit Testing with SwiftData

For unit tests, always create an in-memory container to avoid file system side effects and ensure test isolation:

func makeTestContainer() throws -> ModelContainer {
    let config = ModelConfiguration(
        isStoredInMemoryOnly: true
    )
    return try ModelContainer(
        for: Article.self, Author.self, Tag.self,
        configurations: config
    )
}

func testInsertArticle() throws {
    let container = try makeTestContainer()
    let context = ModelContext(container)

    let article = Article(title: "Test", body: "Body")
    context.insert(article)
    try context.save()

    let fetched = try context.fetch(FetchDescriptor<Article>())
    XCTAssertEqual(fetched.count, 1)
    XCTAssertEqual(fetched.first?.title, "Test")
}

11. When to Choose SwiftData vs Core Data in 2024+

Architect Must-Know — SwiftData vs Core Data Decision Matrix

Choose SwiftData when:

  • New greenfield project targeting iOS 17+ with SwiftUI-first architecture
  • Small-to-medium datasets (under ~50K objects) where batch operations aren't critical
  • Team expertise is Swift-native — developers unfamiliar with Objective-C patterns and KVC
  • Rapid prototyping — SwiftData's concise API accelerates development significantly
  • Simple CloudKit sync — private database sync with last-write-wins is sufficient
  • Type safety is a priority — compile-time schema and predicate validation prevents runtime crashes

Choose Core Data when:

  • Existing Core Data codebase — migration risk outweighs SwiftData benefits
  • Large datasets (100K+ objects) requiring batch insert/delete/update operations
  • UIKit-heavy app relying on NSFetchedResultsController for table/collection views
  • Advanced CloudKit — shared databases, CKShare, custom conflict resolution
  • Performance-critical paths — fine-grained faulting, prefetching, and fetch batch size control
  • Must support iOS < 17
  • Complex queries — subqueries, aggregate functions, expression descriptions

Hybrid approach (recommended for large teams):

  • Use SwiftData for new SwiftUI features; keep Core Data for existing UIKit screens
  • Both frameworks read/write the same SQLite store — interop is possible
  • Gradually migrate screens as they are rewritten in SwiftUI
  • Use Core Data's batch operations via the underlying NSPersistentContainer when needed

Migration path from Core Data to SwiftData:

  1. Ensure your Core Data model has no unsupported features (abstract entities, derived attributes)
  2. Create @Model classes matching your existing entity descriptions
  3. Use the same SQLite store URL in ModelConfiguration
  4. Migrate one screen at a time — both stacks can coexist
  5. Write integration tests verifying data round-trips between both APIs

Key interview signal: A Staff engineer recognizes that SwiftData is not a replacement for Core Data in all scenarios. The decision depends on dataset size, team composition, deployment target, and feature requirements. Recommending SwiftData for a performance-critical app with 500K+ records or a UIKit-heavy codebase is a red flag. Recommending Core Data for a new iOS 17+ SwiftUI app with simple persistence needs is equally misguided.
