1. Core Data Stack Architecture

Core Data is not a database. It is an object-graph management and persistence framework. The stack is composed of four primary objects that collaborate to move managed objects between memory and the persistent store.

┌─────────────────────────────────────────────────────────┐ │ NSPersistentContainer │ │ (Convenience wrapper — owns the entire stack) │ │ │ │ ┌───────────────────────────────────────────────────┐ │ │ │ NSManagedObjectModel (.xcdatamodeld) │ │ │ │ Entities, Attributes, Relationships, Fetch Reqs │ │ │ └──────────────────────┬────────────────────────────┘ │ │ │ │ │ ┌──────────────────────▼────────────────────────────┐ │ │ │ NSPersistentStoreCoordinator │ │ │ │ Mediates between context(s) and store(s) │ │ │ │ Serializes read/write to the backing store │ │ │ └──────────┬───────────────────────┬────────────────┘ │ │ │ │ │ │ ┌──────────▼──────────┐ ┌─────────▼───────────────┐ │ │ │ NSPersistentStore │ │ NSPersistentStore │ │ │ │ (SQLite — default) │ │ (Binary / In-Memory) │ │ │ └─────────────────────┘ └─────────────────────────┘ │ │ │ │ ┌───────────────────────────────────────────────────┐ │ │ │ viewContext (NSManagedObjectContext) │ │ │ │ Main-queue context for UI reads │ │ │ └───────────────────────────────────────────────────┘ │ │ │ │ newBackgroundContext() → private-queue contexts │ └─────────────────────────────────────────────────────────┘

Component Breakdown

Component Responsibility Key Details
NSManagedObjectModel Schema definition Compiled from .xcdatamodeld. Entities, attributes, relationships, fetch request templates, configurations.
NSPersistentStoreCoordinator Store broker Serializes context requests to the store. Supports multiple stores (e.g., one SQLite + one in-memory). Thread-safe internally via its own lock.
NSPersistentStore Storage backend SQLite (default, recommended), Binary, In-Memory. SQLite uses WAL mode by default for concurrent reads.
NSManagedObjectContext Scratchpad / staging area Tracks insertions, updates, deletions. Must be used on its designated queue. Supports undo.
NSPersistentContainer Convenience wrapper (iOS 10+) Creates and owns the model, coordinator, and a viewContext. Factory for background contexts.

Modern Stack Setup

import CoreData

final class PersistenceController {

    static let shared = PersistenceController()

    let container: NSPersistentContainer

    var viewContext: NSManagedObjectContext {
        container.viewContext
    }

    init(inMemory: Bool = false) {
        container = NSPersistentContainer(name: "Model")

        if inMemory {
            container.persistentStoreDescriptions.first?.url =
                URL(fileURLWithPath: "/dev/null")
        }

        // Enable persistent history tracking for background merging
        let description = container.persistentStoreDescriptions.first
        description?.setOption(
            true as NSNumber,
            forKey: NSPersistentHistoryTrackingKey
        )
        description?.setOption(
            true as NSNumber,
            forKey: NSPersistentStoreRemoteChangeNotificationPostOptionKey
        )

        container.loadPersistentStores { storeDescription, error in
            if let error = error as NSError? {
                fatalError("Unresolved error \(error)")
            }
        }

        container.viewContext.automaticallyMergesChangesFromParent = true
        container.viewContext.mergePolicy = NSMergeByPropertyObjectTrumpMergePolicy
        container.viewContext.name = "viewContext"
    }

    func newBackgroundContext() -> NSManagedObjectContext {
        let context = container.newBackgroundContext()
        context.mergePolicy = NSMergeByPropertyObjectTrumpMergePolicy
        context.undoManager = nil
        return context
    }
}
Deep Dive — Store Types

SQLite is the only store type that supports faulting, making it suitable for large datasets. Binary stores load the entire object graph into memory. In-Memory stores are useful for unit tests and caches. You can attach multiple stores to one coordinator using configurations — for example, a transient cache in-memory and persistent data in SQLite.

2. NSManagedObjectContext Deep Dive

An NSManagedObjectContext is a scratchpad — an in-memory staging area where you create, fetch, modify, and delete managed objects. Nothing touches the persistent store until you call save().

Internal State Tracking

NSManagedObjectContext ├── registeredObjects ← all objects this context knows about ├── insertedObjects ← newly created, not yet saved ├── updatedObjects ← existing objects with pending changes ├── deletedObjects ← marked for deletion on next save └── hasChanges ← true if any of the above are non-empty

Every managed object has an objectID (NSManagedObjectID). Temporary IDs are assigned at insertion and become permanent after the first save. You can force permanent IDs early with obtainPermanentIDs(for:) — essential when passing object references across contexts.

save() Mechanics

  1. Validation — Core Data calls validateForInsert(), validateForUpdate(), or validateForDelete() on each changed object.
  2. Will-save notification — NSManagedObjectContextWillSaveNotification fires. Objects receive willSave().
  3. Conflict detection — If the store version differs from the snapshot, the merge policy resolves conflicts.
  4. Push to coordinator — The coordinator serializes the save to the SQLite store.
  5. Did-save notification — NSManagedObjectContextDidSaveNotification fires with inserted/updated/deleted object IDs.
  6. Reset change tracking — insertedObjects, updatedObjects, deletedObjects are cleared.

Undo Manager Integration

let context = NSManagedObjectContext(concurrencyType: .mainQueueConcurrencyType)
context.undoManager = UndoManager()

// Group a set of changes
context.undoManager?.beginUndoGrouping()
user.name = "Alice"
user.email = "alice@example.com"
context.undoManager?.endUndoGrouping()

// Undo reverts both changes atomically
context.undoManager?.undo()

// Redo re-applies them
context.undoManager?.redo()
Best Practice

Set undoManager = nil on background contexts. Undo tracking adds memory overhead and is only useful for UI-driven editing contexts. The viewContext should have an undo manager only if your app supports undo (e.g., document-based apps).

Context Creation Patterns

// 1. Via NSPersistentContainer (preferred)
let bgContext = container.newBackgroundContext()

// 2. Via performBackgroundTask (auto-creates and auto-releases)
container.performBackgroundTask { context in
    // Already on the correct queue
    let user = User(context: context)
    user.name = "Bob"
    try? context.save()
}

// 3. Child context (pushes to parent, not to store)
let childContext = NSManagedObjectContext(concurrencyType: .mainQueueConcurrencyType)
childContext.parent = container.viewContext

3. Concurrency Model

Core Data contexts are not thread-safe. Each context is bound to a specific queue, and all access must happen on that queue. The two concurrency types are:

Concurrency Type Queue Usage
.mainQueueConcurrencyType Main thread UI reads, viewContext
.privateQueueConcurrencyType Private serial queue Background imports, syncing, heavy fetches

perform vs performAndWait

let bgContext = container.newBackgroundContext()

// Asynchronous — enqueues block on context's queue
bgContext.perform {
    let request = User.fetchRequest()
    let users = try? bgContext.fetch(request)
    // Process on background queue
}

// Synchronous — blocks calling thread until complete
// Safe to call from any thread; re-entrant on the context's own queue
bgContext.performAndWait {
    let count = try? bgContext.count(for: User.fetchRequest())
    print("User count: \(count ?? 0)")
}
Warning — Thread Violation

Accessing a context or its managed objects from the wrong thread causes undefined behavior — crashes, data corruption, or silent data loss. Enable -com.apple.CoreData.ConcurrencyDebug 1 in your scheme's launch arguments during development. This triggers an assertion on any illegal cross-thread access.

Parent-Child Context Architecture

┌──────────────────────────────────┐ │ Persistent Store (SQLite) │ └──────────────┬───────────────────┘ │ save() ┌──────────────▼───────────────────┐ │ Root Context (private queue) │ │ parent: nil │ │ persistentStoreCoordinator: PSC │ └──────────────┬───────────────────┘ │ save() pushes up ┌──────────────▼───────────────────┐ │ Main Context (main queue) │ │ parent: Root Context │ │ Used for UI binding │ └──────────────┬───────────────────┘ │ save() pushes up ┌──────────────▼───────────────────┐ │ Editing Context (main queue) │ │ parent: Main Context │ │ Used for modal editing screens │ └──────────────────────────────────┘

When a child context calls save(), changes are pushed to the parent context in memory — they do not hit the persistent store. Only when the root context (the one connected to the coordinator) calls save() do changes reach SQLite. This gives you a transactional editing pattern: discard the child context to cancel, or save up the chain to commit.

Multi-Context Architecture Code

final class CoreDataStack {

    private let container: NSPersistentContainer

    // Root context writes to store on a private queue
    private lazy var writerContext: NSManagedObjectContext = {
        let ctx = NSManagedObjectContext(concurrencyType: .privateQueueConcurrencyType)
        ctx.persistentStoreCoordinator = container.persistentStoreCoordinator
        ctx.mergePolicy = NSMergeByPropertyObjectTrumpMergePolicy
        ctx.undoManager = nil
        ctx.name = "writerContext"
        return ctx
    }()

    // Main-queue context for UI — child of writer
    lazy var mainContext: NSManagedObjectContext = {
        let ctx = NSManagedObjectContext(concurrencyType: .mainQueueConcurrencyType)
        ctx.parent = writerContext
        ctx.automaticallyMergesChangesFromParent = true
        ctx.name = "mainContext"
        return ctx
    }()

    init() {
        container = NSPersistentContainer(name: "Model")
        container.loadPersistentStores { _, error in
            if let error { fatalError("Store load failed: \(error)") }
        }
    }

    // Editing context for modal forms — child of main
    func newEditingContext() -> NSManagedObjectContext {
        let ctx = NSManagedObjectContext(concurrencyType: .mainQueueConcurrencyType)
        ctx.parent = mainContext
        ctx.undoManager = UndoManager()
        ctx.name = "editingContext"
        return ctx
    }

    // Background import context — child of writer
    func newImportContext() -> NSManagedObjectContext {
        let ctx = NSManagedObjectContext(concurrencyType: .privateQueueConcurrencyType)
        ctx.parent = writerContext
        ctx.undoManager = nil
        ctx.name = "importContext"
        return ctx
    }

    func saveAll() {
        mainContext.performAndWait {
            guard mainContext.hasChanges else { return }
            try? mainContext.save()
        }
        writerContext.perform {
            guard self.writerContext.hasChanges else { return }
            try? self.writerContext.save()
        }
    }
}
Deep Dive — Why Parent-Child?

The parent-child pattern avoids merge conflicts entirely for editing workflows. A child context's save() is a fast in-memory push — no disk I/O. The root writer context batches disk writes on its private queue, keeping the main thread free. This is the classic pattern for document-based apps and complex editing flows. However, for simple CRUD apps, NSPersistentContainer's viewContext + newBackgroundContext() with automaticallyMergesChangesFromParent is simpler and sufficient.

4. NSFetchedResultsController

NSFetchedResultsController (FRC) bridges Core Data and UIKit by monitoring a fetch request and notifying its delegate of changes. It provides:

  • Sections — automatic grouping via sectionNameKeyPath
  • Caching — optional on-disk section cache (keyed by a name you provide)
  • Efficient updates — only reports inserted/deleted/moved/updated index paths

Complete UITableView Binding

final class UsersViewController: UITableViewController {

    private lazy var fetchedResultsController: NSFetchedResultsController<User> = {
        let request = User.fetchRequest()
        request.sortDescriptors = [
            NSSortDescriptor(keyPath: \User.lastName, ascending: true),
            NSSortDescriptor(keyPath: \User.firstName, ascending: true)
        ]
        request.fetchBatchSize = 20

        let frc = NSFetchedResultsController(
            fetchRequest: request,
            managedObjectContext: PersistenceController.shared.viewContext,
            sectionNameKeyPath: "sectionLetter",
            cacheName: "UsersCache"
        )
        frc.delegate = self
        return frc
    }()

    override func viewDidLoad() {
        super.viewDidLoad()
        try? fetchedResultsController.performFetch()
    }

    // MARK: - UITableViewDataSource

    override func numberOfSections(in tableView: UITableView) -> Int {
        fetchedResultsController.sections?.count ?? 0
    }

    override func tableView(
        _ tableView: UITableView,
        numberOfRowsInSection section: Int
    ) -> Int {
        fetchedResultsController.sections?[section].numberOfObjects ?? 0
    }

    override func tableView(
        _ tableView: UITableView,
        cellForRowAt indexPath: IndexPath
    ) -> UITableViewCell {
        let cell = tableView.dequeueReusableCell(
            withIdentifier: "UserCell",
            for: indexPath
        )
        let user = fetchedResultsController.object(at: indexPath)
        cell.textLabel?.text = "\(user.firstName ?? "") \(user.lastName ?? "")"
        cell.detailTextLabel?.text = user.email
        return cell
    }

    override func tableView(
        _ tableView: UITableView,
        titleForHeaderInSection section: Int
    ) -> String? {
        fetchedResultsController.sections?[section].name
    }
}

FRC Delegate — Classic Approach

extension UsersViewController: NSFetchedResultsControllerDelegate {

    func controllerWillChangeContent(
        _ controller: NSFetchedResultsController<NSFetchRequestResult>
    ) {
        tableView.beginUpdates()
    }

    func controller(
        _ controller: NSFetchedResultsController<NSFetchRequestResult>,
        didChange anObject: Any,
        at indexPath: IndexPath?,
        for type: NSFetchedResultsChangeType,
        newIndexPath: IndexPath?
    ) {
        switch type {
        case .insert:
            guard let newIndexPath else { return }
            tableView.insertRows(at: [newIndexPath], with: .automatic)
        case .delete:
            guard let indexPath else { return }
            tableView.deleteRows(at: [indexPath], with: .automatic)
        case .update:
            guard let indexPath else { return }
            tableView.reloadRows(at: [indexPath], with: .automatic)
        case .move:
            guard let indexPath, let newIndexPath else { return }
            tableView.moveRow(at: indexPath, to: newIndexPath)
        @unknown default:
            tableView.reloadData()
        }
    }

    func controllerDidChangeContent(
        _ controller: NSFetchedResultsController<NSFetchRequestResult>
    ) {
        tableView.endUpdates()
    }
}

Modern Diffable Data Source Integration (iOS 13+)

extension UsersViewController: NSFetchedResultsControllerDelegate {

    func controller(
        _ controller: NSFetchedResultsController<NSFetchRequestResult>,
        didChangeContentWith snapshot: NSDiffableDataSourceSnapshotReference
    ) {
        let snapshot = snapshot as NSDiffableDataSourceSnapshot<String, NSManagedObjectID>
        dataSource.apply(snapshot, animatingDifferences: true)
    }
}
Best Practice

Prefer the didChangeContentWith snapshot: delegate method over the classic begin/end updates pattern. It eliminates the common "invalid update" crashes caused by simultaneous inserts and deletes. The snapshot-based approach also works naturally with UICollectionViewDiffableDataSource.

5. Lightweight vs Heavyweight Migration

When your data model changes between app versions, Core Data must migrate the existing store to match the new schema. The two strategies differ dramatically in complexity.

Lightweight Migration (Automatic)

Core Data can infer the mapping automatically for these changes:

Change Supported? Notes
Add attribute Yes Must have a default value or be optional
Remove attribute Yes Data is dropped silently
Add entity Yes New table created
Remove entity Yes Table dropped
Rename attribute/entity Yes Set Renaming ID in model inspector
Add relationship Yes Must be optional or have a default
Change optional ↔ required Yes Existing nils may cause failures
Add/remove index Yes Transparent
Change attribute type No Requires heavyweight migration
Split/merge entities No Requires heavyweight migration

Enabling Lightweight Migration

let description = NSPersistentStoreDescription()
description.url = storeURL
description.shouldMigrateStoreAutomatically = true
description.shouldInferMappingModelAutomatically = true

// These are the defaults — shown here for clarity
container.persistentStoreDescriptions = [description]

Heavyweight (Custom) Migration

When lightweight migration cannot handle the transformation, you must provide an explicit NSMappingModel and optionally a custom NSEntityMigrationPolicy.

class UserToProfileMigrationPolicy: NSEntityMigrationPolicy {

    override func createDestinationInstances(
        forSource sInstance: NSManagedObject,
        in mapping: NSEntityMapping,
        manager: NSMigrationManager
    ) throws {
        // Create the destination "Profile" entity from the source "User"
        let profile = NSEntityDescription.insertNewObject(
            forEntityName: "Profile",
            into: manager.destinationContext
        )

        // Map old fields to new structure
        let firstName = sInstance.value(forKey: "firstName") as? String ?? ""
        let lastName = sInstance.value(forKey: "lastName") as? String ?? ""
        profile.setValue("\(firstName) \(lastName)", forKey: "fullName")

        // Carry over unchanged fields
        profile.setValue(
            sInstance.value(forKey: "email"),
            forKey: "email"
        )
        profile.setValue(
            sInstance.value(forKey: "createdAt"),
            forKey: "createdAt"
        )

        manager.associate(
            sourceInstance: sInstance,
            withDestinationInstance: profile,
            for: mapping
        )
    }
}

Programmatic Migration Manager

func migrateStore(
    at sourceURL: URL,
    sourceModel: NSManagedObjectModel,
    destinationModel: NSManagedObjectModel,
    mappingModel: NSMappingModel
) throws {
    let manager = NSMigrationManager(
        sourceModel: sourceModel,
        destinationModel: destinationModel
    )

    let destinationURL = sourceURL
        .deletingLastPathComponent()
        .appendingPathComponent("Model_migrated.sqlite")

    try manager.migrateStore(
        from: sourceURL,
        type: NSSQLiteStoreType,
        mapping: mappingModel,
        to: destinationURL,
        type: NSSQLiteStoreType
    )

    // Swap files atomically
    let fm = FileManager.default
    try fm.removeItem(at: sourceURL)
    try fm.moveItem(at: destinationURL, to: sourceURL)
}
Warning — Migration Pitfalls

1) Heavyweight migration loads the entire store into memory. For large databases (100k+ rows), this can cause OOM crashes. Consider staged migration or progressive lazy migration instead.
2) Never delete old model versions from your bundle. Core Data needs the source model to detect which migration path to take.
3) Test migrations with production-sized data. A migration that works on 100 rows may fail or take minutes on 500k rows.

6. Batch Operations

Batch operations execute directly against the SQLite store, bypassing the managed object context entirely. This makes them orders of magnitude faster for bulk operations but requires explicit merging back to in-memory contexts.

Normal save(): Context → Coordinator → SQLite (one row at a time, fires KVO, validations) Batch operation: Request → Coordinator → SQLite (direct SQL, no context involvement) ↓ Must manually merge changes back into context(s)

NSBatchInsertRequest (iOS 13+)

func importUsers(_ records: [UserDTO]) throws {
    let context = PersistenceController.shared.newBackgroundContext()

    try context.performAndWait {
        var index = 0

        let batchInsert = NSBatchInsertRequest(
            entity: User.entity(),
            managedObjectHandler: { managedObject in
                guard index < records.count else { return true }

                let user = managedObject as! User
                let record = records[index]
                user.id = record.id
                user.firstName = record.firstName
                user.lastName = record.lastName
                user.email = record.email
                user.createdAt = record.createdAt

                index += 1
                return false  // false = continue inserting
            }
        )

        batchInsert.resultType = .statusOnly

        let result = try context.execute(batchInsert) as! NSBatchInsertResult

        if let success = result.result as? Bool, success {
            print("Batch insert of \(records.count) records succeeded")
        }
    }
}

NSBatchDeleteRequest

func deleteOldUsers(olderThan date: Date) throws {
    let fetchRequest: NSFetchRequest<NSFetchRequestResult> = User.fetchRequest()
    fetchRequest.predicate = NSPredicate(
        format: "lastActiveAt < %@",
        date as NSDate
    )

    let batchDelete = NSBatchDeleteRequest(fetchRequest: fetchRequest)
    batchDelete.resultType = .resultTypeObjectIDs

    let context = PersistenceController.shared.viewContext
    let result = try context.execute(batchDelete) as! NSBatchDeleteResult
    let objectIDs = result.result as? [NSManagedObjectID] ?? []

    // Merge deletions into the view context
    NSManagedObjectContext.mergeChanges(
        fromRemoteContextSave: [NSDeletedObjectsKey: objectIDs],
        into: [context]
    )
}

NSBatchUpdateRequest

func markAllUsersInactive() throws {
    let batchUpdate = NSBatchUpdateRequest(entityName: "User")
    batchUpdate.propertiesToUpdate = ["isActive": false]
    batchUpdate.resultType = .updatedObjectIDsResultType

    let context = PersistenceController.shared.viewContext
    let result = try context.execute(batchUpdate) as! NSBatchUpdateResult
    let objectIDs = result.result as? [NSManagedObjectID] ?? []

    NSManagedObjectContext.mergeChanges(
        fromRemoteContextSave: [NSUpdatedObjectsKey: objectIDs],
        into: [context]
    )
}

Importing 100k Records — Production Pattern

func importLargeDataset(_ records: [UserDTO]) async throws {
    let container = PersistenceController.shared.container
    let batchSize = 5000

    try await container.performBackgroundTask { context in
        context.mergePolicy = NSMergeByPropertyObjectTrumpMergePolicy
        context.undoManager = nil

        for chunk in stride(from: 0, to: records.count, by: batchSize) {
            let end = min(chunk + batchSize, records.count)
            let slice = Array(records[chunk..<end])

            var idx = 0
            let request = NSBatchInsertRequest(
                entity: User.entity(),
                managedObjectHandler: { obj in
                    guard idx < slice.count else { return true }
                    let user = obj as! User
                    user.id = slice[idx].id
                    user.firstName = slice[idx].firstName
                    user.lastName = slice[idx].lastName
                    user.email = slice[idx].email
                    idx += 1
                    return false
                }
            )
            request.resultType = .statusOnly
            try context.execute(request)
        }

        // Notify view context of all changes
        try context.save()
    }
}
Deep Dive — Why Bypass the Context?

A normal context.save() for 100k objects would: (1) create 100k NSManagedObject instances in memory, (2) run validation on each, (3) fire KVO for every property change, (4) generate change notifications. Batch operations skip all of this and execute raw SQL against the store. The trade-off: no validation, no KVO, no relationship cascade, and no automatic context refresh. You must merge changes manually.

7. NSMergePolicy Types

Merge conflicts occur when two contexts modify the same object and one saves before the other. The merge policy determines how Core Data resolves these conflicts.

Policy Behavior Use Case
NSErrorMergePolicy Throws an error on conflict. No automatic resolution. Default. Use when you want full control over conflict resolution.
NSMergeByPropertyStoreTrumpMergePolicy Store wins on a per-property basis. Unchanged in-memory properties are preserved. Background sync where server data should take precedence.
NSMergeByPropertyObjectTrumpMergePolicy In-memory object wins on a per-property basis. Unchanged store properties are preserved. User edits that should override background imports. Most commonly used.
NSOverwriteMergePolicy In-memory object wins entirely. All store values are overwritten. Full-replace sync patterns (e.g., re-downloading entire records).
NSRollbackMergePolicy Store wins entirely. All in-memory changes are discarded. Read-only contexts that should always reflect the latest store state.

Per-Property Resolution Example

Object in Store: { name: "Alice", email: "old@mail.com", age: 30 } Object in Context: { name: "Bob", email: "old@mail.com", age: 30 } ↑ changed ↑ unchanged ↑ unchanged NSMergeByPropertyObjectTrumpMergePolicy result: { name: "Bob", email: "old@mail.com", age: 30 } ↑ context wins ↑ no conflict ↑ no conflict NSMergeByPropertyStoreTrumpMergePolicy result: { name: "Alice", email: "old@mail.com", age: 30 } ↑ store wins ↑ no conflict ↑ no conflict 
// Setting merge policy on contexts
let viewContext = container.viewContext
viewContext.mergePolicy = NSMergeByPropertyObjectTrumpMergePolicy

let syncContext = container.newBackgroundContext()
syncContext.mergePolicy = NSMergeByPropertyStoreTrumpMergePolicy

// Custom merge policy (advanced)
class TimestampMergePolicy: NSMergePolicy {
    override func resolve(
        constraintConflicts list: [NSConstraintConflict]
    ) throws {
        for conflict in list {
            guard let dbObject = conflict.databaseObject,
                  let conflicting = conflict.conflictingObjects.first
            else { continue }

            let dbDate = dbObject.value(forKey: "updatedAt") as? Date ?? .distantPast
            let newDate = conflicting.value(forKey: "updatedAt") as? Date ?? .distantPast

            if dbDate > newDate {
                // Store is newer — discard in-memory changes
                conflicting.setValuesForKeys(dbObject.dictionaryWithValues(
                    forKeys: conflict.constraint
                ))
            }
        }
        try super.resolve(constraintConflicts: list)
    }
}
Warning — Default Merge Policy

The default merge policy is NSErrorMergePolicy, which throws on any conflict. If you have multiple contexts writing to the same store (which is common), you must set an explicit merge policy or your saves will fail silently when you catch the error without handling it. Always set NSMergeByPropertyObjectTrumpMergePolicy as a baseline.

8. Faulting & Uniquing

Faulting and uniquing are Core Data's two primary memory optimization strategies. Understanding them is critical for performance tuning.

What Are Faults?

A fault is a placeholder object that represents a managed object whose attribute data has not been loaded from the store. It has the correct objectID and class, but accessing any attribute triggers a fault fire — a trip to the store to load the data.

Fault (unfired): ┌──────────────────────────────┐ │ NSManagedObject (User) │ │ objectID: 0x...4a2 │ │ isFault: true │ │ name: ??? (not loaded) │ │ email: ??? (not loaded) │ └──────────────────────────────┘ │ access .name ▼ Materialized (fault fired): ┌──────────────────────────────┐ │ NSManagedObject (User) │ │ objectID: 0x...4a2 │ │ isFault: false │ │ name: "Alice" │ │ email: "alice@example.com" │ └──────────────────────────────┘

The N+1 Query Problem

If you fetch 100 User objects (1 query) and then iterate to access each user's department relationship, Core Data fires a separate fault for each department — 100 additional queries. This is the classic N+1 problem.

// BAD: N+1 queries — 1 fetch + N fault fires
let users = try context.fetch(User.fetchRequest())
for user in users {
    print(user.department?.name ?? "None")  // fires fault each time
}

// GOOD: Prefetch relationships to avoid N+1
let request = User.fetchRequest()
request.relationshipKeyPathsForPrefetching = ["department"]
let users = try context.fetch(request)
for user in users {
    print(user.department?.name ?? "None")  // already loaded
}

Batch Faulting

let request = User.fetchRequest()

// fetchBatchSize causes Core Data to fault objects in batches
// instead of materializing the entire result set at once
request.fetchBatchSize = 20

// Only fetch specific properties (partial faulting)
request.propertiesToFetch = ["firstName", "lastName"]

// Return faults instead of materialized objects
request.returnsObjectsAsFaults = true  // default

let users = try context.fetch(request)
// Only 20 objects are materialized at a time as you iterate

Uniquing

Uniquing guarantees that within a single context, there is only ever one object instance per row in the store. If you fetch the same user twice, you get the same Swift object reference.

let request1 = User.fetchRequest()
request1.predicate = NSPredicate(format: "id == %@", userId)
let user1 = try context.fetch(request1).first

let request2 = User.fetchRequest()
request2.predicate = NSPredicate(format: "id == %@", userId)
let user2 = try context.fetch(request2).first

// Uniquing guarantees identity
assert(user1 === user2)  // same object in memory
Warning — Faulting Pitfalls

1) Accessing a fault's properties on the wrong thread fires the fault on that thread, causing a concurrency violation.
2) Calling refreshObject(_:mergeChanges: false) turns a materialized object back into a fault, releasing its memory — but any unsaved changes are lost.
3) Uniquing only applies within a single context. The same row in two different contexts produces two distinct object instances with different lifecycles.

Advanced Prefetching for Table Views

final class OptimizedFRC {

    static func makeUserFRC(
        context: NSManagedObjectContext
    ) -> NSFetchedResultsController<User> {
        let request = User.fetchRequest()

        // Sort for sections + ordering
        request.sortDescriptors = [
            NSSortDescriptor(keyPath: \User.department?.name, ascending: true),
            NSSortDescriptor(keyPath: \User.lastName, ascending: true)
        ]

        // Prefetch the department relationship to avoid N+1
        request.relationshipKeyPathsForPrefetching = ["department"]

        // Batch fault 50 at a time for smooth scrolling
        request.fetchBatchSize = 50

        return NSFetchedResultsController(
            fetchRequest: request,
            managedObjectContext: context,
            sectionNameKeyPath: "department.name",
            cacheName: nil
        )
    }
}
Best Practice

Always set fetchBatchSize on fetch requests used with table/collection views. A batch size of 2× the visible row count is a good starting point. Combine with relationshipKeyPathsForPrefetching for any relationships displayed in cells. Profile with Instruments → Core Data template to verify fault-firing patterns.

Deep Dive — History Tracking

NSPersistentCloudKitContainer relies on persistent history tracking (NSPersistentHistoryTransaction) to detect local changes that need syncing. This creates a transaction log in the SQLite store. You should periodically purge old history to prevent unbounded growth. Use NSPersistentHistoryChangeRequest.deleteHistory(before:) to clean up transactions older than the last successful sync token.

10. Core Data Stack for 1M+ Records

Architect Must-Know — Production-Scale Core Data

At L6/Staff level, you must be able to design a Core Data stack that handles millions of records without degrading UI responsiveness. This requires combining batch operations, background contexts, fetch optimization, and denormalization into a cohesive architecture.

Architecture Principles for Large Datasets

  1. Never fetch all records — Use fetchLimit, fetchOffset, and fetchBatchSize to paginate.
  2. Import on background contexts — Use NSBatchInsertRequest for bulk imports, never insert one-by-one on the main context.
  3. Denormalize for read performance — Store computed/derived values (e.g., fullName, sectionLetter) as attributes to avoid expensive predicate computations.
  4. Index aggressively — Add indexes on any attribute used in predicates or sort descriptors.
  5. Minimize context memory — Use refreshObject(_:mergeChanges:) to re-fault objects after processing. Set stalenessInterval on background contexts.
  6. Profile with Instruments — Use the Core Data template to identify excessive faulting, slow fetches, and memory bloat.

Production Stack — Complete Implementation

import CoreData
import os

final class ProductionCoreDataStack {

    static let shared = ProductionCoreDataStack()

    private let logger = Logger(
        subsystem: "com.example.app",
        category: "CoreData"
    )

    let container: NSPersistentContainer

    var viewContext: NSManagedObjectContext {
        container.viewContext
    }

    private init() {
        container = NSPersistentContainer(name: "Model")

        guard let description = container.persistentStoreDescriptions.first
        else { fatalError("Missing store description") }

        // Enable WAL checkpoint for better read concurrency
        description.setValue(
            "WAL" as NSString,
            forPragmaNamed: "journal_mode"
        )

        // History tracking for background merge awareness
        description.setOption(
            true as NSNumber,
            forKey: NSPersistentHistoryTrackingKey
        )
        description.setOption(
            true as NSNumber,
            forKey: NSPersistentStoreRemoteChangeNotificationPostOptionKey
        )

        container.loadPersistentStores { [logger] desc, error in
            if let error {
                logger.critical("Store load failed: \(error.localizedDescription)")
                fatalError("Store load failed: \(error)")
            }
            logger.info("Store loaded: \(desc.url?.lastPathComponent ?? "unknown")")
        }

        // Configure view context for UI
        container.viewContext.automaticallyMergesChangesFromParent = true
        container.viewContext.mergePolicy = NSMergeByPropertyObjectTrumpMergePolicy
        container.viewContext.undoManager = nil
        container.viewContext.shouldDeleteInaccessibleFaults = true
        container.viewContext.name = "viewContext"
    }

    // MARK: - Background Contexts

    func newBackgroundContext() -> NSManagedObjectContext {
        let context = container.newBackgroundContext()
        context.mergePolicy = NSMergeByPropertyObjectTrumpMergePolicy
        context.undoManager = nil
        context.name = "backgroundContext"
        return context
    }

    // MARK: - Batch Import

    func batchImport<T>(
        entity: NSEntityDescription,
        records: [T],
        batchSize: Int = 5000,
        configure: @escaping (NSManagedObject, T) -> Void
    ) async throws {
        let total = records.count
        logger.info("Starting batch import of \(total) records")

        try await container.performBackgroundTask { [logger] context in
            context.mergePolicy = NSMergeByPropertyObjectTrumpMergePolicy
            context.undoManager = nil

            for chunkStart in stride(from: 0, to: total, by: batchSize) {
                let chunkEnd = min(chunkStart + batchSize, total)
                let chunk = Array(records[chunkStart..<chunkEnd])

                var idx = 0
                let request = NSBatchInsertRequest(
                    entity: entity,
                    managedObjectHandler: { obj in
                        guard idx < chunk.count else { return true }
                        configure(obj, chunk[idx])
                        idx += 1
                        return false
                    }
                )
                request.resultType = .statusOnly
                try context.execute(request)

                logger.debug("Imported chunk \(chunkStart)-\(chunkEnd)")
            }

            try context.save()
            logger.info("Batch import complete: \(total) records")
        }
    }

    // MARK: - Optimized Fetch

    func fetchPage<T: NSManagedObject>(
        _ type: T.Type,
        predicate: NSPredicate? = nil,
        sortDescriptors: [NSSortDescriptor],
        page: Int,
        pageSize: Int = 50,
        propertiesToFetch: [String]? = nil
    ) throws -> [T] {
        let request = T.fetchRequest()
        request.predicate = predicate
        request.sortDescriptors = sortDescriptors
        request.fetchLimit = pageSize
        request.fetchOffset = page * pageSize
        request.fetchBatchSize = pageSize
        request.returnsObjectsAsFaults = false

        if let properties = propertiesToFetch {
            request.propertiesToFetch = properties
        }

        return try viewContext.fetch(request) as? [T] ?? []
    }

    // MARK: - Count Without Loading

    func count<T: NSManagedObject>(
        _ type: T.Type,
        predicate: NSPredicate? = nil
    ) throws -> Int {
        let request = T.fetchRequest()
        request.predicate = predicate
        return try viewContext.count(for: request)
    }

    // MARK: - History Cleanup

    func purgeHistory(before date: Date) throws {
        let purgeRequest = NSPersistentHistoryChangeRequest.deleteHistory(
            before: date
        )
        let bgContext = newBackgroundContext()
        try bgContext.performAndWait {
            try bgContext.execute(purgeRequest)
        }
        logger.info("Purged history before \(date)")
    }
}

Fetch Request Optimization Checklist

Optimization API Effect
Limit results fetchLimit Adds LIMIT to SQL — stops scanning after N rows
Paginate fetchOffset Adds OFFSET — skip rows for pagination
Batch faulting fetchBatchSize Materializes objects in batches as you iterate
Partial fetch propertiesToFetch Only loads specified columns — reduces memory
Count only context.count(for:) Executes SELECT COUNT(*) — no objects created
Dictionary result resultType = .dictionaryResultType Returns dictionaries instead of managed objects — no faulting overhead
Prefetch relations relationshipKeyPathsForPrefetching Batch-loads related objects — eliminates N+1
Distinct results returnsDistinctResults Adds SELECT DISTINCT — useful with propertiesToFetch

Best Practice — Memory Management at Scale

When processing large result sets in a background context, call context.reset() after processing each batch to release all registered objects. Alternatively, use refreshObject(_:mergeChanges: false) on individual objects to re-fault them. For truly massive imports (1M+), use NSBatchInsertRequest exclusively — it never creates managed objects in memory at all.

Architect Must-Know — Interview Talking Points

When discussing Core Data at scale in an L6 interview, demonstrate understanding of these trade-offs:

  • Context-per-operation vs shared contexts — Creating a new background context per import operation provides isolation but increases coordinator contention. A pooled approach with 2–3 reusable background contexts reduces overhead.
  • SQLite WAL mode — Core Data uses WAL by default, enabling concurrent reads during writes. However, long-running read transactions can prevent WAL checkpointing, growing the WAL file unboundedly.
  • Batch operations vs context saves — Batch operations are 10–100× faster for bulk mutations but skip validation, KVO, and relationship cascades. Use them for imports; use context saves for user-driven edits.
  • Denormalization vs normalization — Normalized schemas reduce storage but increase join complexity. For read-heavy apps (most iOS apps), denormalize frequently-accessed computed values into indexed attributes.
  • When to move beyond Core Data — If you need full-text search (consider SQLite FTS5 via GRDB), complex aggregate queries, or cross-platform sync (consider CRDTs), Core Data may not be the right tool. Know its boundaries.
