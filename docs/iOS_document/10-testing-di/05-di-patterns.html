1. DI Principles & Why It Matters

Dependency Injection is the practice of providing a component's dependencies from the outside rather than having the component create them internally. At Staff level, DI is not just about testability — it's about module boundaries, build times, and team velocity.

1.1 The Dependency Inversion Principle

WITHOUT DI (tight coupling): ┌──────────────┐ ┌──────────────────┐ │ UserViewModel │─────▶│ URLSessionClient │ │ │ │ (concrete) │ └──────────────┘ └──────────────────┘ Problem: ViewModel directly depends on concrete implementation. Cannot test without real network. Cannot swap implementations. WITH DI (inverted dependency): ┌──────────────┐ ┌──────────────────┐ │ UserViewModel │─────▶│ NetworkClient │ │ │ │ (protocol) │ └──────────────┘ └────────▲─────────┘ │ ┌──────────┴──────────┐ │ │ ┌───────────────┐ ┌─────────────────┐ │ URLSession │ │ MockNetwork │ │ Client │ │ Client │ └───────────────┘ └─────────────────┘

1.2 DI Benefits at Scale

Benefit Impact Scale Factor
Testability Every component can be tested in isolation Linear with module count
Modularity Modules depend on abstractions, not implementations Enables parallel team development
Build time Changing an implementation doesn't recompile dependents Exponential savings at 100+ modules
Flexibility Swap implementations (debug/release, A/B tests) Enables feature flags at DI level

2. Constructor Injection

Constructor injection is the gold standard of DI. Dependencies are passed through the initializer, making them explicit, immutable, and impossible to forget.

2.1 Basic Pattern

protocol UserRepository {
    func fetchUser(id: String) async throws -> User
    func saveUser(_ user: User) async throws
}

protocol AnalyticsTracking {
    func track(_ event: AnalyticsEvent)
}

protocol Logger {
    func log(_ message: String, level: LogLevel)
}

final class UserProfileViewModel: ObservableObject {
    @Published private(set) var user: User?
    @Published private(set) var isLoading = false
    @Published private(set) var error: Error?

    private let repository: UserRepository
    private let analytics: AnalyticsTracking
    private let logger: Logger

    init(
        repository: UserRepository,
        analytics: AnalyticsTracking,
        logger: Logger
    ) {
        self.repository = repository
        self.analytics = analytics
        self.logger = logger
    }

    func loadUser(id: String) async {
        isLoading = true
        defer { isLoading = false }

        do {
            user = try await repository.fetchUser(id: id)
            analytics.track(.profileViewed(userId: id))
        } catch {
            self.error = error
            logger.log("Failed to load user \(id): \(error)", level: .error)
        }
    }
}

2.2 Testing with Constructor Injection

@Test func loadUserSuccess() async {
    let mockRepo = StubUserRepository(
        user: User(id: "1", name: "Alice")
    )
    let spyAnalytics = SpyAnalytics()
    let spyLogger = SpyLogger()

    let vm = UserProfileViewModel(
        repository: mockRepo,
        analytics: spyAnalytics,
        logger: spyLogger
    )

    await vm.loadUser(id: "1")

    #expect(vm.user?.name == "Alice")
    #expect(vm.error == nil)
    #expect(spyAnalytics.events.count == 1)
    #expect(spyLogger.messages.isEmpty)
}

2.3 Default Parameter Values

// Provide production defaults to reduce boilerplate at call sites
final class UserProfileViewModel: ObservableObject {
    init(
        repository: UserRepository = APIUserRepository(),
        analytics: AnalyticsTracking = FirebaseAnalytics.shared,
        logger: Logger = OSLogger.default
    ) {
        // ...
    }
}
Pitfall: Default Values Create Hidden Dependencies

Default parameter values are convenient but create implicit coupling. The ViewModel module must now import the modules containing APIUserRepository, FirebaseAnalytics, and OSLogger. In a modular codebase, this defeats the purpose of DI. Use factories or containers instead.

3. Factory Pattern

Factories centralize object creation, keeping dependency wiring out of feature code. They're especially useful when objects need runtime parameters alongside injected dependencies.

3.1 Simple Factory

protocol ViewModelFactory {
    func makeUserProfileViewModel(userId: String) -> UserProfileViewModel
    func makeOrderListViewModel() -> OrderListViewModel
    func makeSettingsViewModel() -> SettingsViewModel
}

class AppViewModelFactory: ViewModelFactory {
    private let networkClient: NetworkClient
    private let database: Database
    private let analytics: AnalyticsTracking

    init(networkClient: NetworkClient, database: Database, analytics: AnalyticsTracking) {
        self.networkClient = networkClient
        self.database = database
        self.analytics = analytics
    }

    func makeUserProfileViewModel(userId: String) -> UserProfileViewModel {
        UserProfileViewModel(
            repository: APIUserRepository(client: networkClient, db: database),
            analytics: analytics,
            logger: OSLogger(subsystem: "UserProfile")
        )
    }

    func makeOrderListViewModel() -> OrderListViewModel {
        OrderListViewModel(
            repository: APIOrderRepository(client: networkClient),
            analytics: analytics
        )
    }

    func makeSettingsViewModel() -> SettingsViewModel {
        SettingsViewModel(database: database)
    }
}

3.2 Closure-Based Factory

struct UserProfileFactory {
    var makeViewModel: (String) -> UserProfileViewModel
}

extension UserProfileFactory {
    static func live(client: NetworkClient, analytics: AnalyticsTracking) -> Self {
        Self { userId in
            UserProfileViewModel(
                repository: APIUserRepository(client: client),
                analytics: analytics,
                logger: OSLogger.default
            )
        }
    }

    static var preview: Self {
        Self { userId in
            UserProfileViewModel(
                repository: StubUserRepository(user: .mock),
                analytics: NoOpAnalytics(),
                logger: PrintLogger()
            )
        }
    }
}

4. Environment-Based DI in SwiftUI

SwiftUI provides a built-in DI system through the environment. This is the idiomatic way to inject dependencies in SwiftUI apps.

4.1 @Environment with Custom Keys

// Define the environment key
private struct UserRepositoryKey: EnvironmentKey {
    static let defaultValue: any UserRepository = APIUserRepository()
}

extension EnvironmentValues {
    var userRepository: any UserRepository {
        get { self[UserRepositoryKey.self] }
        set { self[UserRepositoryKey.self] = newValue }
    }
}

// Use in views
struct UserListView: View {
    @Environment(\.userRepository) private var repository
    @State private var users: [User] = []

    var body: some View {
        List(users) { user in
            Text(user.name)
        }
        .task {
            users = (try? await repository.fetchUsers()) ?? []
        }
    }
}

// Inject at any level of the view hierarchy
struct MyApp: App {
    var body: some Scene {
        WindowGroup {
            ContentView()
                .environment(\.userRepository, APIUserRepository(client: .live))
        }
    }
}

// Override for previews
#Preview {
    UserListView()
        .environment(\.userRepository, StubUserRepository(users: User.mockList))
}

4.2 @EnvironmentObject

// Observable object as dependency
class AuthManager: ObservableObject {
    @Published var currentUser: User?
    @Published var isAuthenticated = false

    func login(email: String, password: String) async throws { /* ... */ }
    func logout() { /* ... */ }
}

struct ProfileView: View {
    @EnvironmentObject var auth: AuthManager

    var body: some View {
        if let user = auth.currentUser {
            Text("Welcome, \(user.name)")
        } else {
            Text("Not logged in")
        }
    }
}

// Inject
ContentView()
    .environmentObject(AuthManager())
Pitfall: @EnvironmentObject Crashes

@EnvironmentObject crashes at runtime if the object isn't provided by an ancestor view. There's no compile-time safety. Prefer @Environment with custom keys (which have default values) or the @Observable + @Environment pattern in iOS 17+.

4.3 iOS 17+: @Observable with @Environment

@Observable
class AppState {
    var currentUser: User?
    var theme: Theme = .system
}

// Direct environment injection (no EnvironmentKey needed)
struct ContentView: View {
    @Environment(AppState.self) var appState

    var body: some View {
        Text(appState.currentUser?.name ?? "Guest")
    }
}

// Provide
ContentView()
    .environment(AppState())

5. Swinject Container

Swinject is the most popular DI container for Swift. It provides runtime registration and resolution of dependencies with support for scoping, circular dependencies, and auto-injection.

5.1 Basic Usage

import Swinject

let container = Container()

// Register dependencies
container.register(NetworkClient.self) { _ in
    URLSessionNetworkClient(session: .shared)
}

container.register(UserRepository.self) { resolver in
    APIUserRepository(client: resolver.resolve(NetworkClient.self)!)
}

container.register(UserProfileViewModel.self) { resolver in
    UserProfileViewModel(
        repository: resolver.resolve(UserRepository.self)!,
        analytics: resolver.resolve(AnalyticsTracking.self)!,
        logger: resolver.resolve(Logger.self)!
    )
}

// Resolve
let viewModel = container.resolve(UserProfileViewModel.self)!

5.2 Scoping

// Transient (new instance every time) — default
container.register(Logger.self) { _ in OSLogger() }

// Singleton (shared instance)
container.register(Database.self) { _ in SQLiteDatabase() }
    .inObjectScope(.container)

// Weak singleton (shared while someone holds a reference)
container.register(ImageCache.self) { _ in InMemoryImageCache() }
    .inObjectScope(.weak)

5.3 Assemblies for Modular Registration

class NetworkAssembly: Assembly {
    func assemble(container: Container) {
        container.register(NetworkClient.self) { _ in
            URLSessionNetworkClient()
        }.inObjectScope(.container)

        container.register(ImageDownloader.self) { r in
            NetworkImageDownloader(client: r.resolve(NetworkClient.self)!)
        }
    }
}

class UserAssembly: Assembly {
    func assemble(container: Container) {
        container.register(UserRepository.self) { r in
            APIUserRepository(client: r.resolve(NetworkClient.self)!)
        }
    }
}

// Compose assemblies
let assembler = Assembler([
    NetworkAssembly(),
    UserAssembly(),
    OrderAssembly(),
    AnalyticsAssembly(),
])
Pitfall: Force Unwrapping Resolved Dependencies

Swinject's resolve() returns an optional. Force-unwrapping (!) moves errors from compile-time to runtime. In production, a missing registration causes a crash. Mitigate with: (1) integration tests that resolve all registrations, (2) a debug-mode assertion that logs unresolved types, (3) consider compile-time DI alternatives for critical paths.

6. @Dependency in TCA

The Composable Architecture (TCA) by Point-Free uses a sophisticated dependency management system based on Swift's task-local values.

6.1 Defining Dependencies

import Dependencies

// Define the dependency interface
struct UserClient {
    var fetch: (String) async throws -> User
    var save: (User) async throws -> Void
    var delete: (String) async throws -> Void
}

// Register with the dependency system
extension UserClient: DependencyKey {
    static let liveValue = UserClient(
        fetch: { id in
            let (data, _) = try await URLSession.shared.data(from: URL(string: "https://api.example.com/users/\(id)")!)
            return try JSONDecoder().decode(User.self, from: data)
        },
        save: { user in
            // ...
        },
        delete: { id in
            // ...
        }
    )

    static let testValue = UserClient(
        fetch: unimplemented("UserClient.fetch"),
        save: unimplemented("UserClient.save"),
        delete: unimplemented("UserClient.delete")
    )
}

extension DependencyValues {
    var userClient: UserClient {
        get { self[UserClient.self] }
        set { self[UserClient.self] = newValue }
    }
}

6.2 Using Dependencies in Reducers

@Reducer
struct UserProfile {
    @ObservableState
    struct State {
        var user: User?
        var isLoading = false
    }

    enum Action {
        case loadUser(String)
        case userLoaded(Result<User, Error>)
    }

    @Dependency(\.userClient) var userClient

    var body: some ReducerOf<Self> {
        Reduce { state, action in
            switch action {
            case .loadUser(let id):
                state.isLoading = true
                return .run { send in
                    let result = await Result { try await userClient.fetch(id) }
                    await send(.userLoaded(result))
                }
            case .userLoaded(.success(let user)):
                state.isLoading = false
                state.user = user
                return .none
            case .userLoaded(.failure):
                state.isLoading = false
                return .none
            }
        }
    }
}

6.3 Testing with @Dependency

@Test func loadUserSuccess() async {
    let store = TestStore(initialState: UserProfile.State()) {
        UserProfile()
    } withDependencies: {
        $0.userClient.fetch = { id in
            User(id: id, name: "Alice")
        }
    }

    await store.send(.loadUser("1")) {
        $0.isLoading = true
    }

    await store.receive(.userLoaded(.success(User(id: "1", name: "Alice")))) {
        $0.isLoading = false
        $0.user = User(id: "1", name: "Alice")
    }
}

7. Protocol Witnesses

Protocol witnesses replace protocols with structs containing closures. This pattern (popularized by Point-Free) avoids the overhead of protocol existentials and enables more flexible composition.

7.1 Protocol vs Witness

// Protocol approach
protocol Logging {
    func log(_ message: String)
}

struct ConsoleLogger: Logging {
    func log(_ message: String) { print(message) }
}

// Protocol witness approach
struct Logger {
    var log: (String) -> Void
}

extension Logger {
    static let console = Logger { message in
        print(message)
    }

    static let noop = Logger { _ in }

    static func file(path: String) -> Logger {
        Logger { message in
            try? message.write(toFile: path, atomically: true, encoding: .utf8)
        }
    }

    // Composition: log to multiple destinations
    func combined(with other: Logger) -> Logger {
        Logger { message in
            self.log(message)
            other.log(message)
        }
    }
}

7.2 Advantages Over Protocols

Aspect Protocol Protocol Witness
Composition Requires wrapper types Natural (combine closures)
Partial conformance Must implement all methods Can leave closures nil or default
Existential overhead Box allocation for any Protocol Struct (stack-allocated)
Retroactive conformance Extension on type Just create a new instance
Discoverability Xcode autocomplete Less discoverable

8. Reader Monad Pattern

The reader monad pattern threads dependencies through a computation without explicitly passing them as parameters. In Swift, this is typically implemented with a generic type that wraps a function from an environment to a result.

struct Reader<Environment, Value> {
    let run: (Environment) -> Value

    func map<NewValue>(_ transform: @escaping (Value) -> NewValue) -> Reader<Environment, NewValue> {
        Reader<Environment, NewValue> { env in
            transform(self.run(env))
        }
    }

    func flatMap<NewValue>(_ transform: @escaping (Value) -> Reader<Environment, NewValue>) -> Reader<Environment, NewValue> {
        Reader<Environment, NewValue> { env in
            transform(self.run(env)).run(env)
        }
    }
}

// Define the environment
struct AppEnvironment {
    var userClient: UserClient
    var analytics: AnalyticsClient
    var logger: Logger
    var date: () -> Date
    var uuid: () -> UUID
}

// Functions that read from the environment
func fetchAndLogUser(id: String) -> Reader<AppEnvironment, async () throws -> User> {
    Reader { env in
        {
            env.logger.log("Fetching user \(id)")
            let user = try await env.userClient.fetch(id)
            env.analytics.track(.userFetched)
            return user
        }
    }
}
Deep Dive: Reader Monad in Practice

The reader monad is more common in functional programming communities. In mainstream iOS development, it's rarely used directly. However, TCA's @Dependency system is essentially a reader monad implemented with Swift's task-local values. Understanding the concept helps you appreciate why TCA's dependency system works the way it does.

9. Service Locator Anti-Pattern

The service locator pattern uses a global registry to resolve dependencies. While it appears simpler than DI, it creates hidden dependencies and makes code harder to test and reason about.

// SERVICE LOCATOR (anti-pattern)
class ServiceLocator {
    static let shared = ServiceLocator()
    private var services: [String: Any] = [:]

    func register<T>(_ service: T) {
        services[String(describing: T.self)] = service
    }

    func resolve<T>() -> T {
        services[String(describing: T.self)] as! T
    }
}

// Usage (looks simple, but is problematic)
class UserViewModel {
    func loadUser() {
        let repo: UserRepository = ServiceLocator.shared.resolve()
        // Hidden dependency! Not visible in initializer.
    }
}

9.1 Why It's an Anti-Pattern

Problem Explanation
Hidden dependencies Can't see what a class needs by looking at its initializer
Global mutable state Any code can register/override services at any time
Runtime failures Missing registration crashes at runtime, not compile time
Test isolation Tests must carefully set up and tear down the global locator
Dependency graph invisible No way to visualize or validate the dependency graph

11. Architect: DI for 50+ Feature Modules

Architect Must-Know

In a modular app with 50+ feature modules, the DI architecture determines build times, team autonomy, and testability. The wrong choice creates a tangled dependency graph that slows everyone down.

11.1 Modular DI Architecture

┌─────────────────────────────────────────────────────────────┐ │ Modular DI Architecture │ │ │ │ Layer 1: Interface Modules (protocols only) │ │ ┌──────────┐ ┌──────────┐ ┌──────────┐ ┌──────────┐ │ │ │UserAPI │ │OrderAPI │ │PaymentAPI│ │AnalytAPI │ │ │ │(protocol) │ │(protocol)│ │(protocol)│ │(protocol)│ │ │ └──────────┘ └──────────┘ └──────────┘ └──────────┘ │ │ ▲ ▲ ▲ ▲ │ │ │ │ │ │ │ │ Layer 2: Feature Modules (depend on interfaces only) │ │ ┌──────────┐ ┌──────────┐ ┌──────────┐ │ │ │UserFeature│ │OrderFeat │ │PaymentFt │ │ │ │(views+VM) │ │(views+VM)│ │(views+VM)│ │ │ └──────────┘ └──────────┘ └──────────┘ │ │ ▲ ▲ ▲ │ │ │ │ │ │ │ Layer 3: Implementation Modules │ │ ┌──────────┐ ┌──────────┐ ┌──────────┐ │ │ │UserImpl │ │OrderImpl │ │PaymentImp│ │ │ │(concrete) │ │(concrete)│ │(concrete)│ │ │ └──────────┘ └──────────┘ └──────────┘ │ │ ▲ ▲ ▲ │ │ │ │ │ │ │ Layer 4: App Module (wires everything together) │ │ ┌─────────────────────────────────────────────┐ │ │ │ AppAssembly / AppDIContainer │ │ │ │ (knows all implementations, creates graph) │ │ │ └─────────────────────────────────────────────┘ │ └─────────────────────────────────────────────────────────────┘

11.2 Rules for Modular DI

Best Practices
  • Feature modules never import implementation modules — they depend only on interface (protocol) modules
  • Only the app module knows all implementations — it's the composition root
  • Each module has its own assembly/registration — modules register their own types
  • Interface modules are tiny — protocols + data types only, fast to compile
  • Test targets depend on interface modules — create mocks from protocols

11.3 Build Time Impact

Build time comparison (50 modules, change in UserImpl): Without modular DI: UserImpl changed → recompile UserFeature → recompile OrderFeature → recompile PaymentFeature → ... → recompile App Total: 45 modules recompiled, 8 minutes With modular DI: UserImpl changed → recompile UserImpl → recompile App (link only) Total: 2 modules recompiled, 45 seconds
