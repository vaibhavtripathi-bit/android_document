1. Pipeline Overview

The Core Animation rendering pipeline is a two-process architecture. Your app prepares the layer tree, then hands it off to a separate system process (the render server) that does the actual GPU compositing. This split is fundamental to understanding iOS rendering performance.

Deserialize tree │ │ b. Build render tree │ │ c. Submit to GPU │ └───────────┬───────────┘ │ ▼ ┌─────────────────────┐ │ 4. GPU COMPOSITING │ │ │ │ a. Vertex shaders │ │ b. Fragment shaders │ │ c. Why Two Processes?

Benefit Explanation
Crash isolation If your app crashes, the render server keeps the last frame on screen (no black flash)
Animation continuity Core Animation animations run in the render server — they continue even if your app's main thread is blocked
Security Apps can't directly access the GPU or frame buffer — the render server mediates all access
Efficiency The render server batches work from multiple apps into a single GPU submission

2. CATransaction: Implicit vs Explicit

Every change to a CALayer property happens within a CATransaction. Transactions batch changes and commit them together at the end of the RunLoop iteration.

Implicit Transactions

// Every RunLoop iteration has an implicit transaction
// Any layer property change is automatically captured

layer.opacity = 0.5          // Captured in implicit transaction
layer.position = newPosition  // Also captured
layer.backgroundColor = UIColor.red.cgColor  // Also captured

// At the end of the RunLoop iteration:
// 1. Implicit transaction commits
// 2. All changes are batched and sent to render server
// 3. Default animation duration: 0.25s for standalone layers
//    (UIView-backed layers suppress implicit animations by default)

Explicit Transactions

// Explicit transactions give you control over commit timing and animation

CATransaction.begin()

// Disable implicit animations (important for performance)
CATransaction.setDisableActions(true)

// Set animation duration for this transaction
CATransaction.setAnimationDuration(0.5)

// Set completion handler
CATransaction.setCompletionBlock {
    print("Animation complete")
}

// Batch multiple layer changes
layer1.opacity = 0
layer2.position = newPosition
layer3.transform = CATransform3DMakeScale(2, 2, 1)

CATransaction.commit()

// Nested transactions
CATransaction.begin()
CATransaction.setAnimationDuration(1.0)

layer1.opacity = 0.5  // 1.0s animation

CATransaction.begin()
CATransaction.setAnimationDuration(0.3)
layer2.opacity = 0.5  // 0.3s animation (inner transaction)
CATransaction.commit()

CATransaction.commit()

Why UIView Suppresses Implicit Animations

// UIView-backed layers suppress implicit animations by default.
// This is because UIView has its own animation system (UIView.animate).

// The mechanism: UIView implements the CALayerDelegate method
// action(for:forKey:) and returns NSNull() for most properties,
// which tells Core Animation "no animation."

// Inside a UIView.animate block, UIView temporarily returns
// actual CAAction objects instead of NSNull(), enabling animations.

// This is why this doesn't animate:
view.layer.opacity = 0.5  // No animation — UIView suppresses it

// But this does:
UIView.animate(withDuration: 0.3) {
    view.layer.opacity = 0.5  // Animates — UIView provides CAAction
}

// And standalone layers always animate implicitly:
let standaloneLayer = CALayer()
view.layer.addSublayer(standaloneLayer)
standaloneLayer.opacity = 0.5  // Animates with 0.25s default duration

3. CALayer Display Cycle

// The layer display cycle determines how content is rendered

// Option 1: Direct contents assignment (most efficient)
layer.contents = image.cgImage  // No drawing — just compositing

// Option 2: Layer delegate drawing
class MyView: UIView {
    override func draw(_ rect: CGRect) {
        // Called when layer needs to redraw its contents
        // UIKit creates a CGContext, calls this method,
        // then assigns the resulting bitmap to layer.contents

        let context = UIGraphicsGetCurrentContext()!
        context.setFillColor(UIColor.blue.cgColor)
        context.fill(rect)
    }
}

// The display cycle:
// 1. Something calls setNeedsDisplay() (or a property change triggers it)
// 2. At commit time, CALayer.display() is called
// 3. display() checks if delegate implements display(_:)
//    → If yes: delegate.display(layer) — delegate provides contents directly
//    → If no: layer creates a CGContext and calls delegate.draw(_:in:)
// 4. For UIView: draw(_:) is called with the CGContext
// 5. The resulting bitmap is assigned to layer.contents

// IMPORTANT: draw(_:) creates a backing store (bitmap in memory)
// A 375×812 @3x view = 1125×2436 pixels × 4 bytes = ~10.4 MB!
// Only override draw(_:) when you actually need custom drawing.
Common Pitfall

Overriding draw(_:) allocates a backing store bitmap. A plain UIView with just backgroundColor set does NOT have a backing store — the color is composited directly by the GPU. The moment you override draw(_:), UIKit allocates a bitmap the size of the view (× screen scale). For a full-screen view on iPhone 15 Pro Max at 3x, that's ~10 MB. Don't override draw(_:) just to set a background color.

4. Off-Screen Rendering Triggers

Off-screen rendering occurs when the GPU can't composite a layer in a single pass and needs to render to a temporary buffer first. This is expensive because it requires a context switch on the GPU.

Normal Rendering (single pass): Layer → GPU → Frame Buffer → Screen (fast — single pass compositing) Off-Screen Rendering (multi-pass): Layer → GPU → Off-Screen Buffer → GPU → Frame Buffer → Screen (slow — extra render target, context switch, memory allocation) Cost of off-screen rendering: • GPU context switch (~0.1ms per layer) • Temporary buffer allocation (layer size × scale × 4 bytes) • Extra compositing pass • At 60fps with 20 off-screen layers: ~2ms wasted (12% of frame budget)

What Triggers Off-Screen Rendering

Trigger Why Mitigation
cornerRadius + masksToBounds GPU must clip content to rounded rect Use layer.cornerRadius without masksToBounds if possible; or pre-render rounded images
shadow without shadowPath GPU must trace the layer's alpha channel to compute shadow shape Always set shadowPath explicitly
mask (CALayer mask) Requires compositing the mask with the layer content Use cornerRadius instead of mask for simple shapes
shouldRasterize = true Intentionally renders to off-screen buffer (cached) Use when the layer content is static and complex
Group opacity (allowsGroupOpacity) Must composite sublayers together before applying opacity Set layer.allowsGroupOpacity = false if not needed
CALayer.mask Arbitrary mask shape requires off-screen compositing Avoid when possible; use cornerRadius for simple cases 
// BAD: Off-screen rendering for rounded corners
imageView.layer.cornerRadius = 20
imageView.layer.masksToBounds = true  // Triggers off-screen render!

// GOOD: cornerRadius without masksToBounds (iOS 13+)
imageView.layer.cornerRadius = 20
imageView.layer.cornerCurve = .continuous  // Smooth Apple-style corners
// No masksToBounds — background rounds but content doesn't clip
// Works for solid-color views but not for images that need clipping

// GOOD: Pre-render rounded image on background thread
func roundedImage(_ image: UIImage, radius: CGFloat) -> UIImage {
    let renderer = UIGraphicsImageRenderer(size: image.size)
    return renderer.image { context in
        let rect = CGRect(origin: .zero, size: image.size)
        UIBezierPath(roundedRect: rect, cornerRadius: radius).addClip()
        image.draw(in: rect)
    }
}

// BAD: Shadow without path
view.layer.shadowColor = UIColor.black.cgColor
view.layer.shadowOffset = CGSize(width: 0, height: 2)
view.layer.shadowRadius = 4
view.layer.shadowOpacity = 0.3
// GPU must trace alpha channel every frame!

// GOOD: Shadow with explicit path
view.layer.shadowPath = UIBezierPath(
    roundedRect: view.bounds,
    cornerRadius: view.layer.cornerRadius
).cgPath
// GPU knows the shadow shape — no off-screen render needed
// IMPORTANT: Update shadowPath in layoutSubviews() when bounds change

shouldRasterize — Intentional Caching

// shouldRasterize renders the layer tree to an off-screen bitmap
// and caches it. Subsequent frames reuse the cached bitmap.

cell.layer.shouldRasterize = true
cell.layer.rasterizationScale = UIScreen.main.scale  // MUST set this!

// When to use shouldRasterize:
// ✓ Complex static layer trees (many sublayers, effects)
// ✓ Layers that don't change content frequently
// ✓ During scroll animations (cache cell appearance)

// When NOT to use:
// ✗ Layers that change every frame (animation, video)
// ✗ Very large layers (cache bitmap is expensive in memory)
// ✗ Layers with frequently changing content

// The cache is invalidated when:
// • Any sublayer property changes
// • The layer's bounds change
// • The cache hasn't been used for 100ms (auto-eviction)

// Debugging: Enable "Color Off-screen Rendered" in Instruments
// Yellow highlight = off-screen rendered layers

5. Compositing

Compositing is the process of combining multiple layer bitmaps into a single frame buffer image. The GPU does this using the painter's algorithm (back to front).

// Compositing properties that affect performance:

// 1. Opaque layers are faster to composite
view.isOpaque = true  // Tells compositor: no need to blend with layers behind
view.backgroundColor = .white  // Fully opaque background

// 2. Blending is expensive — minimize transparent layers
// Instruments → "Color Blended Layers" shows blending in red

// 3. Misaligned layers cause extra work
// If a layer's position doesn't align to pixel boundaries,
// the GPU must anti-alias the edges
view.frame = CGRect(x: 10.5, y: 20.3, width: 100, height: 50)  // BAD
view.frame = CGRect(x: 10, y: 20, width: 100, height: 50)      // GOOD

// 4. Layer count matters — each layer is a separate GPU draw call
// Flatten complex static content into a single layer when possible

6. Metal Backing

Since iOS 12, Core Animation uses Metal as its GPU backend (replacing OpenGL ES). This is transparent to most developers but has implications for advanced rendering.

// CAMetalLayer — direct Metal rendering in a layer
class MetalView: UIView {
    override class var layerClass: AnyClass {
        return CAMetalLayer.self

    var metalLayer: CAMetalLayer {
        return layer as! CAMetalLayer

    func configure() {
        metalLayer.device = MTLCreateSystemDefaultDevice()
        metalLayer.pixelFormat = .bgra8Unorm
        metalLayer.framebufferOnly = true
        metalLayer.contentsScale = UIScreen.main.scale

    }
}

7. CADisplayLink & ProMotion

// CADisplayLink synchronizes drawing with the display refresh rate
class AnimationController {
    private var displayLink: CADisplayLink?

    func startAnimation() {
        displayLink = CADisplayLink(target: self, selector: #selector(tick))

        // ProMotion (120Hz) support — iOS 15+
        if #available(iOS 15.0, *) {
            // Request specific frame rates
            displayLink?.preferredFrameRateRange = CAFrameRateRange(
                minimum: 60,      // Don't go below 60fps
                maximum: 120,     // Up to 120fps on ProMotion
                preferred: 120    // Prefer 120fps
            )
        }

        displayLink?.add(to: .main, forMode: .common)
    }

    @objc func tick(_ link: CADisplayLink) {
        let timestamp = link.timestamp           // When this frame started
        let targetTimestamp = link.targetTimestamp // When next frame should display
        let duration = link.duration              // Time between frames

        // Calculate animation progress based on timestamps
        let dt = targetTimestamp - timestamp
        updateAnimation(deltaTime: dt)
    }

    func stopAnimation() {
        displayLink?.invalidate()
        displayLink = nil
    }
}

// Frame pacing on ProMotion displays:
// The display can run at: 120Hz, 80Hz, 60Hz, 48Hz, 40Hz, 30Hz, 24Hz, 20Hz
// (divisors of 120)
//
// If your frame takes 9ms → 120fps (8.33ms budget)
// If your frame takes 12ms → 80fps (12.5ms budget)
// If your frame takes 17ms → 60fps (16.67ms budget)
//
// The system dynamically adjusts refresh rate based on content
// Static content → 24Hz or lower (saves battery)
// Scrolling → 120Hz
// Animation → matches animation frame rate
Deep Dive

ProMotion adaptive refresh rate: On ProMotion displays (iPhone 13 Pro+, iPad Pro), the system dynamically adjusts the refresh rate. When nothing is animating, it drops to 24Hz or even 10Hz to save battery. When scrolling or animating, it ramps up to 120Hz. Your app doesn't need to do anything special — Core Animation handles this automatically. However, if you use CADisplayLink for custom animations, set preferredFrameRateRange to hint at your desired frame rate.

8. Architect: Achieving Consistent 120fps

Architect Must-Know

Achieving consistent 120fps on ProMotion displays requires understanding the full pipeline:

  • Frame budget at 120fps: 8.33ms — This is half the 60fps budget. Every millisecond counts.
  • Commit phase must be fast — Layout + display + prepare must complete in ~4ms to leave room for the render server and GPU.
  • Eliminate off-screen rendering — Each off-screen layer costs ~0.1ms on the GPU. With 20 visible cells, that's 2ms — 24% of your frame budget.
  • Pre-decode images — Image decoding on the main thread during scroll is the #1 cause of 120fps drops. Decode on a background queue and cache the decoded bitmap.
  • Minimize layer count — Each layer is a separate GPU draw call. Flatten static content into fewer layers.
// 120fps optimization checklist:

// 1. Pre-decode images on background thread
func preDecodeImage(_ image: UIImage) -> UIImage {
    let renderer = UIGraphicsImageRenderer(size: image.size)
    return renderer.image { context in
        image.draw(at: .zero)
    }
}

// 2. Set shadowPath for all shadows
override func layoutSubviews() {
    super.layoutSubviews()
    layer.shadowPath = UIBezierPath(
        roundedRect: bounds,
        cornerRadius: layer.cornerRadius
    ).cgPath
}

// 3. Mark opaque views
view.isOpaque = true
view.backgroundColor = .systemBackground

// 4. Align frames to pixel boundaries
extension CGRect {
    var pixelAligned: CGRect {
        let scale = UIScreen.main.scale
        return CGRect(
            x: (origin.x * scale).rounded() / scale,
            y: (origin.y * scale).rounded() / scale,
            width: (size.width * scale).rounded() / scale,
            height: (size.height * scale).rounded() / scale
        )
    }
}
