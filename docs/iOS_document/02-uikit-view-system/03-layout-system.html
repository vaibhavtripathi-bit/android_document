1. Auto Layout & the Cassowary Algorithm

Auto Layout is a constraint-based layout system built on the Cassowary linear arithmetic constraint-solving algorithm, originally developed at the University of Washington in 1998. It solves a system of linear equations and inequalities to determine view positions and sizes.

How Cassowary Works

Each constraint is a linear equation or inequality of the form:

view1.attribute = multiplier × view2.attribute + constant Examples: button.leading = container.leading + 16 label.width ≤ 200 imageView.height = imageView.width × 0.75 button.centerY = container.centerY + 0 In matrix form, the solver finds values for all view attributes (x, y, width, height) that satisfy ALL constraints simultaneously. The Cassowary algorithm uses an incremental simplex method: ┌─────────────────────────────────────────────────────┐ │ 1. Convert constraints to a tableau (matrix) │ │ 2. Add slack variables for inequalities │ │ 3. Use simplex pivoting to find feasible solution │ │ 4. Optimize for "stay" constraints (minimize change)│ │ 5. When constraints change, incrementally update │ │ (don't resolve from scratch) │ └─────────────────────────────────────────────────────┘ 
// Each NSLayoutConstraint maps to one Cassowary equation
let constraint = NSLayoutConstraint(
    item: button,                        // view1
    attribute: .leading,                 // view1.attribute
    relatedBy: .equal,                   // = (or ≤, ≥)
    toItem: container,                   // view2
    attribute: .leading,                 // view2.attribute
    multiplier: 1.0,                     // multiplier
    constant: 16                          // constant
)
// Equation: button.leading = 1.0 × container.leading + 16

// Modern anchor syntax (preferred)
button.leadingAnchor.constraint(
    equalTo: container.leadingAnchor,
    constant: 16
).isActive = true

// The solver needs enough constraints to determine all 4 attributes:
// x (leading/trailing/centerX), y (top/bottom/centerY),
// width, height
// Underconstrained → ambiguous layout
// Overconstrained → unsatisfiable (constraint conflict)

Constraint Activation

// Batch activation is more efficient than individual activation
// because the solver processes all constraints in one pass

// BAD: Each activation triggers a solver pass
constraint1.isActive = true
constraint2.isActive = true
constraint3.isActive = true

// GOOD: Single solver pass for all constraints
NSLayoutConstraint.activate([
    button.leadingAnchor.constraint(equalTo: container.leadingAnchor, constant: 16),
    button.trailingAnchor.constraint(equalTo: container.trailingAnchor, constant: -16),
    button.topAnchor.constraint(equalTo: label.bottomAnchor, constant: 12),
    button.heightAnchor.constraint(equalToConstant: 44)
])

// Deactivation + reactivation for layout changes
NSLayoutConstraint.deactivate(portraitConstraints)
NSLayoutConstraint.activate(landscapeConstraints)

2. Constraint Priorities

Every constraint has a priority from 1 to 1000. The solver tries to satisfy all constraints, but when conflicts arise, lower-priority constraints are broken first.

Priority Name Value Usage
.required Required 1000 Must be satisfied — unsatisfiable if conflicting
.defaultHigh Default High 750 Default content hugging priority
.dragThatCanResizeScene Drag Resize 510 Window resizing on macOS/iPad
.sceneSizeStayPut Scene Stay 500 Scene size preference
.defaultLow Default Low 250 Default compression resistance
.fittingSizeLevel Fitting Size 50 Used by systemLayoutSizeFitting 
// Setting constraint priority
let widthConstraint = view.widthAnchor.constraint(equalToConstant: 200)
widthConstraint.priority = .defaultHigh  // 750 — can be broken if needed
widthConstraint.isActive = true

// Custom priority values
widthConstraint.priority = UILayoutPriority(rawValue: 999)

// CRITICAL: You cannot change a required (1000) constraint's priority
// after activation. Set it to 999 if you might need to change it later.
widthConstraint.priority = UILayoutPriority(rawValue: 999)  // Can change later
widthConstraint.isActive = true
widthConstraint.priority = .defaultLow  // OK — was 999, not 1000
Common Pitfall

Never change a required constraint's priority at runtime. If a constraint is activated with priority 1000 (required), attempting to change its priority crashes. Always use 999 for constraints you might need to adjust. The solver treats 999 and 1000 nearly identically, but 999 allows runtime modification.

3. intrinsicContentSize Contract

intrinsicContentSize is a view's natural size based on its content. It acts as implicit width and/or height constraints.

// Views with intrinsic content size:
// UILabel       — width and height (based on text + font)
// UIButton      — width and height (based on title + image + insets)
// UIImageView   — width and height (based on image size)
// UITextField   — height only (width is ambiguous)
// UITextView    — neither (scrollable content)
// UISwitch      — width and height (fixed size)
// UISlider      — height only
// Plain UIView  — neither (noIntrinsicMetric for both)

class BadgeView: UIView {
    var text: String = "" {
        didSet {
            // MUST invalidate when content changes
            invalidateIntrinsicContentSize()
            setNeedsDisplay()
        }
    }

    override var intrinsicContentSize: CGSize {
        let textSize = (text as NSString).size(
            withAttributes: [.font: UIFont.systemFont(ofSize: 12)]
        )
        return CGSize(
            width: textSize.width + 16,  // padding
            height: textSize.height + 8
        )
    }
}

// intrinsicContentSize creates implicit constraints:
// width constraint at contentHuggingPriority(.horizontal)
// height constraint at contentHuggingPriority(.vertical)
// These are "soft" constraints — they can be overridden by explicit constraints

4. Content Hugging & Compression Resistance

Content Hugging vs Compression Resistance: Content Hugging — "Don't stretch me beyond my intrinsic size" ═══════════════════════════════════════════════════════════════ intrinsicContentSize.width = 100 High hugging (750): [ Hello ] ← stays at 100pt Low hugging (250): [ Hello ] ← stretches to fill Compression Resistance — "Don't shrink me below my intrinsic size" ═══════════════════════════════════════════════════════════════ intrinsicContentSize.width = 200, available space = 100 High resistance (750): [This is a long te…] ← truncates at 200pt min Low resistance (250): [This…] ← compressed to 100pt 
// Two labels side by side — which one stretches?
let titleLabel = UILabel()
let dateLabel = UILabel()

// Title should stretch, date should stay at intrinsic size
titleLabel.setContentHuggingPriority(.defaultLow, for: .horizontal)    // 250
dateLabel.setContentHuggingPriority(.defaultHigh, for: .horizontal)    // 750

// If space is tight, title should compress first
titleLabel.setContentCompressionResistancePriority(.defaultLow, for: .horizontal)
dateLabel.setContentCompressionResistancePriority(.required, for: .horizontal)

// The solver creates these implicit constraints from intrinsicContentSize:
// Hugging:     view.width ≤ intrinsicWidth  @ huggingPriority
// Compression: view.width ≥ intrinsicWidth  @ compressionResistancePriority

// So for titleLabel (hugging=250, compression=250):
// titleLabel.width ≤ intrinsicWidth @ 250  (easily broken → stretches)
// titleLabel.width ≥ intrinsicWidth @ 250  (easily broken → compresses)

// For dateLabel (hugging=750, compression=1000):
// dateLabel.width ≤ intrinsicWidth @ 750   (hard to break → won't stretch)
// dateLabel.width ≥ intrinsicWidth @ 1000  (required → never compresses)

5. The Layout Cycle

// The layout cycle has three key methods:

// 1. setNeedsLayout() — marks the view as needing layout
//    Does NOT trigger layout immediately
//    Coalesces multiple calls into a single layout pass
view.setNeedsLayout()

// 2. layoutIfNeeded() — triggers layout IMMEDIATELY if needed
//    Synchronous — blocks until layout is complete
//    Used to animate constraint changes
view.layoutIfNeeded()

// 3. layoutSubviews() — called by the system to perform layout
//    Override this to do custom layout
//    NEVER call this directly — use setNeedsLayout()

class CustomLayoutView: UIView {
    override func layoutSubviews() {
        super.layoutSubviews()  // MUST call super — triggers Auto Layout

        // At this point, Auto Layout has resolved all constraints
        // Subview frames are set — do manual adjustments here

        // Example: round a view's corners based on its resolved height
        avatarView.layer.cornerRadius = avatarView.bounds.height / 2
    }
}

// Animating constraint changes:
heightConstraint.constant = 200
UIView.animate(withDuration: 0.3) {
    // layoutIfNeeded() inside animation block captures the layout change
    self.view.layoutIfNeeded()
}

6. updateConstraints() Cycle

class AdaptiveView: UIView {
    private var compactConstraints: [NSLayoutConstraint] = []
    private var regularConstraints: [NSLayoutConstraint] = []

    override func updateConstraints() {
        // Called bottom-up (leaves first, then parents)
        // Batch constraint changes here for efficiency

        if traitCollection.horizontalSizeClass == .compact {
            NSLayoutConstraint.deactivate(regularConstraints)
            NSLayoutConstraint.activate(compactConstraints)
        } else {
            NSLayoutConstraint.deactivate(compactConstraints)
            NSLayoutConstraint.activate(regularConstraints)
        }

        super.updateConstraints()  // MUST call super at the END
    }

    func someStateChange() {
        // Trigger constraint update on next layout pass
        setNeedsUpdateConstraints()
    }
}

// IMPORTANT: Apple recommends AGAINST overriding updateConstraints()
// for most cases. It's only beneficial when you have a large number
// of constraints that change together and you want to batch the update.
// For simple constraint changes, modify constraints directly.

7. Layout Passes: Constraints → Layout → Display

The Three Passes in Detail: PASS 1: CONSTRAINT UPDATE (bottom-up) ───────────────────────────────────── Leaf views → Parent views → Root view • updateConstraints() called on each view that needs it • Solver incorporates constraint changes • Triggered by setNeedsUpdateConstraints() PASS 2: LAYOUT (top-down) ───────────────────────────────────── Root view → Parent views → Leaf views • layoutSubviews() called on each view that needs it • Solver results applied to view frames • Triggered by setNeedsLayout() PASS 3: DISPLAY (top-down) ───────────────────────────────────── Root view → Parent views → Leaf views • draw(_:) called on each view that needs it • Views render their content to backing layers • Triggered by setNeedsDisplay()

8. UIStackView Internals

UIStackView is a non-rendering view that manages Auto Layout constraints for its arranged subviews. It doesn't draw anything — it has no backing layer content.

// UIStackView manages constraints automatically
let stack = UIStackView(arrangedSubviews: [label1, label2, button])
stack.axis = .vertical
stack.spacing = 12
stack.alignment = .fill
stack.distribution = .fill

// What UIStackView creates internally (approximately):
// For .vertical axis, .fill distribution, .fill alignment:
//
// label1.top = stack.top
// label2.top = label1.bottom + 12 (spacing)
// button.top = label2.bottom + 12
// button.bottom = stack.bottom
//
// label1.leading = stack.leading
// label1.trailing = stack.trailing
// (same for label2, button)

// Distribution options and their constraint implications:
// .fill           — one view stretches (lowest hugging priority wins)
// .fillEqually    — all views same size (equal height/width constraints)
// .fillProportionally — sizes proportional to intrinsicContentSize
// .equalSpacing   — equal spacing between views
// .equalCentering  — equal spacing between view centers
// Hiding views in a stack view — constraints automatically update
UIView.animate(withDuration: 0.3) {
    label2.isHidden = true  // Stack view removes spacing + constraints
}

// Adding/removing arranged subviews
stack.addArrangedSubview(newView)
stack.removeArrangedSubview(label1)  // Removes from arrangement but NOT from superview
label1.removeFromSuperview()          // Must also remove from superview

// Custom spacing (iOS 11+)
stack.setCustomSpacing(24, after: label1)  // Extra space after label1
Deep Dive

UIStackView doesn't render. It overrides layerClass to return CATransformLayer, which is a non-rendering layer that exists only to host sublayers. This means backgroundColor, cornerRadius, etc. have no effect on a stack view. If you need a background, add a background view behind the stack view or embed the stack in a container view.

9. translatesAutoresizingMaskIntoConstraints

// When you create a view in code and want to use Auto Layout:
let view = UIView()
view.translatesAutoresizingMaskIntoConstraints = false

// WHY: By default, UIKit creates constraints from the autoresizing mask
// (the old springs-and-struts system). These auto-generated constraints
// conflict with your manual Auto Layout constraints.

// When translatesAutoresizingMaskIntoConstraints = true (default):
// Setting view.frame creates constraints like:
//   view.width = 200 @ 1000
//   view.height = 100 @ 1000
//   view.leading = superview.leading + 50 @ 1000
//   view.top = superview.top + 100 @ 1000
// These are REQUIRED constraints — they conflict with your constraints!

// Views created in Interface Builder automatically have this set to false
// Views created in code have it set to true by default

// UIStackView's arranged subviews: translates is set to false automatically
// when you call addArrangedSubview()

10. Performance at Scale

Constraint Complexity

The Cassowary solver's time complexity is approximately O(n²) in the number of constraints, where n is the number of constraints that change. This means constraint churn (frequently adding/removing constraints) is expensive.

Constraint Count Layout Time (approximate) Impact
< 100 < 1ms Negligible
100–500 1–5ms Fine for static layouts
500–1000 5–16ms Can cause frame drops during animation
> 1000 > 16ms Guaranteed frame drops at 60fps 
// Measuring layout performance
let start = CFAbsoluteTimeGetCurrent()
view.layoutIfNeeded()
let elapsed = CFAbsoluteTimeGetCurrent() - start
print("Layout took \(elapsed * 1000)ms")

// systemLayoutSizeFitting — measures a view's size under constraints
let size = cell.contentView.systemLayoutSizeFitting(
    CGSize(width: tableView.bounds.width, height: 0),
    withHorizontalFittingPriority: .required,
    verticalFittingPriority: .fittingSizeLevel
)
// This is expensive — it creates a temporary constraint environment
// Cache the result when possible

Optimization Strategies

// 1. Modify constraint constants instead of deactivating/reactivating
// BAD: Constraint churn
NSLayoutConstraint.deactivate([oldConstraint])
let newConstraint = view.heightAnchor.constraint(equalToConstant: 200)
newConstraint.isActive = true

// GOOD: Modify constant (incremental solver update)
heightConstraint.constant = 200

// 2. Use constraint references for dynamic layouts
class ExpandableView: UIView {
    private var heightConstraint: NSLayoutConstraint!

    func setup() {
        heightConstraint = heightAnchor.constraint(equalToConstant: 44)
        heightConstraint.isActive = true
    }

    func expand() {
        heightConstraint.constant = 200  // Cheap — just updates the constant
    }
}

// 3. For collection view cells, prefer manual layout over Auto Layout
class PerformantCell: UICollectionViewCell {
    override func layoutSubviews() {
        super.layoutSubviews()
        let bounds = contentView.bounds
        // Manual frame calculation — O(1) per cell, no solver involved
        imageView.frame = CGRect(x: 16, y: 12, width: 48, height: 48)
        titleLabel.frame = CGRect(
            x: 76, y: 12,
            width: bounds.width - 92, height: 22
        )
    }

    override func sizeThatFits(_ size: CGSize) -> CGSize {
        // For self-sizing cells without Auto Layout
        let titleSize = titleLabel.sizeThatFits(
            CGSize(width: size.width - 92, height: .greatestFiniteMagnitude)
        )
        return CGSize(width: size.width, height: max(72, titleSize.height + 24))
    }
}
Architect Must-Know

Layout performance at scale — avoiding constraint explosion:

  • Flatten hierarchies — Deep nesting multiplies constraint count. A 5-level deep hierarchy with 4 constraints per level = 20 constraints per leaf path. Flatten to 2 levels = 8 constraints.
  • Use manual layout for cells — In a table/collection view with 50+ visible cells, Auto Layout constraints can exceed 1000. Manual layoutSubviews() is O(1) per cell.
  • Cache sizing calculations — systemLayoutSizeFitting is expensive. Cache cell heights keyed by content hash.
  • Profile with Instruments — Use the "Layout" instrument to see constraint solving time per frame. Target < 4ms for 60fps (leaving 12ms for rendering).
  • Consider UIStackView carefully — Each stack view adds constraints proportional to its arranged subview count. Nested stacks multiply this. For complex cells, manual layout is often faster.
