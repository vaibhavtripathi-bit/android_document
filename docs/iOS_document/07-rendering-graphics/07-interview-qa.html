Core Animation Pipeline

Q1: Explain the two-process rendering architecture on iOS. Why does Apple split rendering between the app process and the render server? Architect

iOS rendering is split between two separate processes:

  • App process: Your code runs here. You modify the CALayer model tree, and at the end of each RunLoop cycle, Core Animation commits the layer tree via Mach port IPC.
  • Render server (backboardd): A system daemon that receives the serialized layer tree, builds its own render tree, executes animations, generates Metal command buffers, and submits GPU work for compositing.

Why the split?

  • Security: Apps never get direct GPU access. The render server mediates all GPU operations, preventing one app from reading another's framebuffer.
  • Stability: If your app crashes, the render server continues. System UI keeps rendering.
  • Animation continuity: Once committed, Core Animation animations run in the render server at full frame rate, even if your app's main thread is blocked.
  • Power efficiency: The render server can batch and optimize GPU work across all visible apps.

Performance implication: The commit phase serializes the entire dirty layer tree across the process boundary. Large layer trees = expensive serialization. This is why minimizing layer count matters.

Q2: What are the three layer trees in Core Animation? When would you access the presentation tree? Senior

Core Animation maintains three parallel layer trees:

  1. Model tree: The target values you set in code (layer.opacity = 0.5). Updated immediately when you set properties. Represents the final state.
  2. Presentation tree: Read-only snapshot of interpolated values during animations. Accessed via layer.presentation(). Shows what's currently visible on screen.
  3. Render tree: Private copy in the render server process. Never directly accessible. Created by deserializing the committed model tree.

When to access the presentation tree:

  • Hit testing during animation: The model tree has the final position, but the layer is visually at the presentation tree's position. Use layer.presentation()?.hitTest(point).
  • Interrupting animations: Read the current presentation value to start a new animation from the current visual state.
  • Coordinating with non-animated elements: Position a non-animated view relative to an animating layer's current position.
Q3: Describe the four phases of the Core Animation render loop. Which phase is most commonly the performance bottleneck? Senior

Each frame, Core Animation executes four phases on the main thread:

  1. Layout: Auto Layout constraints are resolved. layoutSubviews() is called on views marked with setNeedsLayout(). The Cassowary simplex solver runs here.
  2. Display: Layers marked with setNeedsDisplay() get their backing stores rendered. draw(_:) is called. Core Graphics drawing happens here.
  3. Prepare: Core Animation prepares image data for the GPU. Compressed images (JPEG, PNG) are decoded into bitmap data. This is a hidden cost — large images decoded here cause frame drops.
  4. Commit: The layer tree is serialized and sent to the render server via Mach port IPC. Cost scales with layer count and changed data.

Most common bottleneck: The Prepare phase is the most insidious because it's invisible in your code — it's the system decoding images you've assigned to layers. The Layout phase is the most common code-level bottleneck, especially with complex Auto Layout constraint hierarchies.

Q4: What's the difference between implicit and explicit CATransactions? How does UIView.animate relate to them? Senior

Implicit transactions: When you modify layer properties outside of an explicit transaction, Core Animation wraps them in an implicit transaction that commits at the end of the current RunLoop iteration. All changes within one RunLoop cycle are batched together. Default animation duration is 0.25s.

Explicit transactions: Created with CATransaction.begin()/.commit(). Give you control over animation duration, timing, completion callbacks, and the ability to disable animations entirely with setDisableActions(true). Can be nested.

UIView.animate: Creates an explicit CATransaction under the hood. Inside the animation block, property changes on UIView animatable properties generate implicit CABasicAnimation objects. The transaction configures the duration, curve, and other parameters.

UIView.performWithoutAnimation is equivalent to wrapping changes in a CATransaction with setDisableActions(true).

Q5: Why do Core Animation animations continue smoothly even when the main thread is blocked? Senior

Because animations are executed by the render server, not by your app process. When you add a CAAnimation to a layer, the animation parameters (from/to values, duration, timing function) are serialized and sent to the render server during the commit phase.

The render server then independently interpolates the property values each frame and submits GPU commands. Your app process doesn't need to wake up for each frame of the animation. This is why a spinning activity indicator continues rotating even during a brief main thread block.

This only applies to Core Animation animations (CABasicAnimation, CAKeyframeAnimation, etc.) and UIView.animate blocks. If you're driving animations manually via CADisplayLink or Timer, those callbacks run on your main thread and will stall if the thread is blocked.

Q6: How does ProMotion adaptive frame rate work? How do you request a specific frame rate for your animations? Senior

ProMotion displays (iPhone 13 Pro+, iPad Pro) support variable refresh rates from 1Hz to 120Hz. The system dynamically adjusts based on content: static content drops to 10Hz to save power, scrolling ramps to 80-120Hz, Apple Pencil input runs at 120Hz, and 24fps video matches exactly at 24Hz.

Requesting specific frame rates (iOS 15+):

  • CADisplayLink.preferredFrameRateRange: Set minimum, maximum, and preferred frame rates for display link callbacks.
  • UIViewPropertyAnimator.preferredFrameRateRange: Request frame rates for property animations.
  • CAAnimation.preferredFrameRateRange: Set on individual Core Animation animations.

The system uses these hints along with its own heuristics to choose the actual refresh rate. It may choose a lower rate than requested to save power if the content doesn't benefit from higher rates.

Q7: What are the common pitfalls with CADisplayLink? How do you avoid the retain cycle? Senior

Pitfalls:

  1. Retain cycle: CADisplayLink strongly retains its target. If the target (e.g., a view controller) also retains the display link, neither will be deallocated. deinit won't fire because the display link keeps the target alive.
  2. RunLoop mode: Adding to .default mode pauses the display link during UIScrollView tracking. Use .common to fire during both default and tracking modes.
  3. Assuming fixed frame rate: On ProMotion displays, the frame interval varies. Always compute delta time from targetTimestamp - timestamp, never assume 16.67ms.
  4. Not invalidating: Forgetting to call invalidate() when done leaks the display link and continues consuming CPU.

Retain cycle solution: Use a proxy object pattern. Create a lightweight object that holds a weak reference to the actual target. Set the proxy as the display link's target. When the real target deallocates, the proxy detects the nil weak reference and invalidates the display link.

Q8: What's the "flash-back" bug with CAAnimation, and how do you prevent it? Senior

The flash-back bug occurs when you add a CAAnimation to a layer without also updating the model tree's property. The animation plays (using the presentation tree), but when it completes and is removed, the layer snaps back to the model tree's old value — creating a visible "flash."

Prevention: Always set the final value on the model layer before adding the animation:

layer.opacity = 0.0  // Set model tree to final value
let fade = CABasicAnimation(keyPath: "opacity")
fade.fromValue = 1.0
fade.toValue = 0.0
fade.duration = 0.3
layer.add(fade, forKey: "fadeOut")

The alternative approach — setting fillMode = .forwards and isRemovedOnCompletion = false — is generally discouraged because it leaves the animation in memory and the model/presentation trees out of sync, which can cause bugs with hit testing and subsequent animations.

Q9: How does the commit phase serialize the layer tree? What factors increase commit cost? Architect

During the commit phase, Core Animation traverses the dirty portion of the layer tree and serializes all changed properties into a binary format that's sent to the render server via Mach port IPC (inter-process communication).

Factors that increase commit cost:

  • Layer count: More layers = more data to serialize. Each layer has dozens of properties.
  • Backing store data: Layers with custom draw(_:) send their bitmap data (width × height × 4 bytes × scale²).
  • New image contents: Newly assigned contents images must be transferred across the process boundary.
  • Deep hierarchy: Deeply nested trees take longer to traverse.
  • Animation data: New animations add to the serialized payload.

Target: Keep the total commit phase under 4-5ms to leave sufficient time for the render server within the 16.67ms frame budget.

Q10: How would you design a rendering performance monitoring system for a large iOS app? L6

A comprehensive rendering performance monitoring system would include:

  1. MetricKit integration: Subscribe to MXMetricManager for scroll hitch rate (scrollHitchTimeRatio), animation metrics, and hang diagnostics. This provides production-level data aggregated by Apple.
  2. CADisplayLink-based hitch detection: In debug/internal builds, use a CADisplayLink to detect when frame intervals exceed the expected budget. Log the call stack when hitches occur.
  3. os_signpost instrumentation: Add signpost intervals around critical rendering paths (cell configuration, image loading, layout) for Instruments analysis.
  4. Automated UI tests: Performance XCTests that scroll through key screens and assert FPS stays above a threshold using XCTMetric.
  5. Dashboard and alerting: Aggregate MetricKit data server-side. Alert when scroll hitch rate exceeds baseline. Track trends across releases.
  6. Debug overlay: In internal builds, show a real-time FPS counter and hitch indicator overlay.

Off-Screen Rendering

Q11: What is off-screen rendering and why is it expensive? Senior

Off-screen rendering occurs when the GPU cannot composite a layer in a single pass directly to the framebuffer. It must allocate a temporary off-screen buffer, render into it, then composite that buffer back into the framebuffer.

Why it's expensive:

  • GPU context switching: Switching render targets (framebuffer → off-screen buffer → framebuffer) flushes the GPU pipeline and invalidates tile caches.
  • Memory allocation: Temporary buffers consume GPU memory.
  • Double rendering: Content is effectively rendered twice — once to the off-screen buffer, then composited to the framebuffer.
  • Compounds in scroll views: 10 visible cells × 2 off-screen triggers each = 20 extra GPU passes per frame.

The cost is approximately 2-3ms per off-screen pass on older devices, which can easily consume the entire frame budget.

Q13: Does cornerRadius alone trigger off-screen rendering? What changed in iOS 13? Senior

cornerRadius alone does not trigger off-screen rendering. It only rounds the layer's own background and border. The problem arises when combined with masksToBounds = true (or clipsToBounds), because the GPU must clip all sublayer content to the rounded rect.

iOS 13 optimization: Apple added a specialized GPU shader for the common case: a single layer with cornerRadius + masksToBounds that has contents (an image) but no sublayers that extend beyond the corner radius. In this case, no off-screen pass is needed. However, if you have sublayers that need clipping (e.g., a label overlapping the corner area), off-screen rendering still occurs.

Q14: Why must you always provide shadowPath? What happens if you don't? Senior

Without shadowPath, the GPU must compute the shadow outline by examining the layer's alpha channel pixel by pixel, every frame. This is extremely expensive because it requires an off-screen pass to trace the layer's visible shape.

With shadowPath, you provide the outline explicitly as a CGPath. The GPU uses this path directly to render the shadow — no tracing needed, no off-screen pass.

Caveat: shadowPath is not automatically updated when the layer's bounds change. You must update it manually in layoutSubviews(). Forgetting this causes the shadow to appear misaligned after layout changes.

Q15: When is shouldRasterize a good optimization, and when does it hurt performance? Architect

Good: When a layer subtree is complex (many sublayers, shadows, gradients) but doesn't change frequently. The first frame pays the off-screen rendering cost, but subsequent frames reuse the cached bitmap — much cheaper than re-compositing the entire subtree.

Bad: When the layer content changes every frame (progress bars, timers, live content). Each change invalidates the cache, so you pay the off-screen cost every frame plus the cache management overhead. This is worse than not rasterizing.

Verification: Use Instruments' "Color Hits Green and Misses Red" debug option. Green = cache hit (good). Red = cache miss (bad). If you see mostly red, remove shouldRasterize.

Common pitfalls: Forgetting to set rasterizationScale = UIScreen.main.scale (defaults to 1.0, causing blurry content on Retina). Cache size limit of ~2.5× screen size. Cache expires after ~100ms of non-use.

Q16: How would you audit an entire app for off-screen rendering issues? L6

Systematic audit process:

  1. Instrument: Run Core Animation template with "Color Offscreen-Rendered Yellow" enabled. Navigate through every screen, especially scroll views.
  2. Prioritize: Focus on views in scroll views and animated views. A static header with one off-screen trigger matters less than a table cell with triggers.
  3. Categorize: Group findings by trigger type to apply batch fixes (e.g., all missing shadowPaths, all cornerRadius+masksToBounds).
  4. Fix and measure: Fix one category at a time, re-measuring FPS after each fix to quantify improvement.
  5. Automate: Write a recursive view tree audit function that checks for known triggers. Run it in debug builds.
  6. Prevent: Add team coding standards requiring shadowPath in code review. Provide utility classes for common patterns (rounded image views, shadowed containers).
  7. CI integration: Add performance UI tests that scroll through critical screens and assert FPS thresholds.
Q17: Explain group opacity. Why does it cause off-screen rendering? Senior

When allowsGroupOpacity is true (default for UIView-backed layers) and the layer's opacity is less than 1.0, the GPU must composite all sublayers into a temporary buffer first, then apply opacity to the entire group as a unit.

Without group opacity, each sublayer would have opacity applied individually. Where sublayers overlap, you'd see incorrect visual artifacts (the overlap region would appear more transparent than intended because opacity is applied twice).

Avoidance: Set layer.allowsGroupOpacity = false if the visual difference is acceptable, or apply opacity to individual child elements instead of the container.

Q18: Can you move off-screen rendering work to a background thread? Senior

No. Off-screen rendering happens in the render server's GPU pass, not in your app process. It's a GPU operation, not a CPU operation. You can't move it to a background thread because it doesn't happen on any of your threads — it happens in a separate system process (backboardd) on the GPU.

The only way to address off-screen rendering is to eliminate the trigger (provide shadowPath, remove masksToBounds, etc.) or cache the result with shouldRasterize. You can also pre-render the visual effect on a background thread using Core Graphics (e.g., pre-rounding images), which avoids the GPU off-screen pass entirely.

Metal & GPU

Q19: What is Metal and how does it relate to Core Animation? Senior

Metal is Apple's low-level GPU programming framework providing near-direct access to GPU hardware. It's the foundation for all GPU work on Apple platforms.

Core Animation uses Metal internally for layer compositing. When the render server composites your layer tree, it generates Metal command buffers and submits them to the GPU. Core Image uses Metal for filter processing. SceneKit and RealityKit use Metal for 3D rendering. Even SwiftUI's drawingGroup() modifier renders to a Metal texture.

Most iOS developers never write Metal code directly because Core Animation, Core Image, and MPS provide high-level abstractions. You drop to Metal when these abstractions can't express your algorithm or when you need frame-level GPU control.

Q20: Explain the Metal command model: device, command queue, command buffer, and encoders. Senior
  • MTLDevice: Represents the GPU. Singleton per GPU. Factory for all GPU resources (buffers, textures, pipeline states, command queues).
  • MTLCommandQueue: Serial queue of command buffers. Create once and reuse (expensive to create). Typically one per app.
  • MTLCommandBuffer: Container for a batch of GPU commands. Created per frame. Lightweight. Contains one or more encoders.
  • Encoders: Write GPU commands into a command buffer:
    • MTLRenderCommandEncoder: Draw calls (vertex + fragment shaders)
    • MTLComputeCommandEncoder: Parallel compute work (compute shaders)
    • MTLBlitCommandEncoder: Memory copy operations (texture/buffer transfers)

Flow: Create command buffer → create encoder → encode commands → end encoding → present drawable → commit buffer → GPU executes.

Q21: What is the advantage of Apple Silicon's unified memory for Metal? Senior

Apple Silicon uses unified memory architecture — the CPU and GPU share the same physical memory. This eliminates the need to copy data between separate CPU and GPU memory pools.

With MTLBuffer created using .storageModeShared, both CPU and GPU access the same memory address. There's zero copy overhead. On traditional discrete GPU architectures (like desktop GPUs), data must be explicitly copied across the PCIe bus.

This is particularly beneficial for:

  • Texture uploads: No need to copy image data from CPU to GPU memory
  • Compute results: GPU compute output is immediately accessible to CPU
  • Dynamic vertex data: CPU can update vertex buffers that GPU reads without explicit synchronization copies
Q22: What are Metal Performance Shaders (MPS)? When should you use them over custom shaders? Senior

MPS is a collection of highly optimized GPU kernels for common operations: image processing (blur, resize, histogram), matrix math, neural network inference, and ray tracing.

Use MPS when: You need standard operations like Gaussian blur, Lanczos scaling, histogram computation, or CNN inference. MPS kernels are optimized by Apple for each GPU generation and automatically benefit from hardware improvements.

Use custom shaders when: You need a novel algorithm that MPS doesn't provide, or you need to combine operations in a way MPS doesn't support efficiently.

MPS is always the first choice for standard operations because Apple engineers have optimized them with knowledge of GPU microarchitecture details that aren't publicly documented.

Q23: How does Core Image use Metal? What's the advantage of CIContext(mtlDevice:)? Senior

Core Image is a high-level image processing framework that builds a filter graph (DAG of operations). When you chain filters, Core Image doesn't execute them immediately — it builds the graph lazily. When you render, it optimizes the graph (fusing operations, eliminating redundancies) and executes it on the GPU via Metal.

CIContext(mtlDevice:) creates a Core Image context backed by a specific Metal device. This enables:

  • GPU acceleration: All filter operations run on the GPU via Metal compute shaders
  • Direct texture rendering: Render CIImage directly to an MTLTexture without CPU roundtrip
  • Command buffer integration: Render into a Metal command buffer alongside your own Metal work
  • Shared resources: Core Image and your Metal code share the same GPU device and memory
Q24: What is GPU Frame Capture and how do you use it to debug rendering issues? Senior

GPU Frame Capture is an Xcode debugging tool that captures every GPU command for a single frame. You trigger it by clicking the camera icon in the debug bar while your Metal app is running.

What you can inspect:

  • Command buffer timeline: Every GPU command in execution order
  • Shader debugger: Step through vertex/fragment/compute shaders line by line, inspecting variable values
  • Texture viewer: Inspect every texture at every pipeline stage
  • Performance counters: GPU utilization, bandwidth, shader occupancy
  • Dependency viewer: Resource dependencies between render passes
  • Memory viewer: GPU memory allocations and usage

It's essential for diagnosing issues like incorrect rendering output, performance bottlenecks in shaders, and excessive GPU memory usage.

Q25: When should an iOS architect decide to use Metal directly instead of higher-level frameworks? L6

The decision framework, from highest to lowest level:

  1. UIKit/SwiftUI + Core Animation: Standard UI. Already Metal-backed. No benefit from going lower.
  2. Core Image: Image filter chains. 200+ built-in filters with automatic GPU optimization.
  3. Metal Performance Shaders: Standard GPU operations (blur, resize, ML). Apple-optimized per GPU generation.
  4. Metal directly: Only when the above can't express your algorithm.

Justified Metal use cases: Custom real-time visual effects (AR filters), games requiring full render pipeline control, novel compute algorithms (custom ML operators), real-time video processing with custom shaders, and scientific visualization.

Rule of thumb: Use the highest-level API that meets your requirements. Every level you drop adds maintenance burden, requires GPU expertise, and loses automatic optimizations from Apple's frameworks.

Text Rendering

Q26: Describe the TextKit 1 architecture. What are the roles of NSTextStorage, NSLayoutManager, and NSTextContainer? Senior

TextKit 1 follows an MVC pattern with three collaborating objects:

  • NSTextStorage (Model): Subclass of NSMutableAttributedString. Stores the text content and attributes. Notifies its layout managers when text changes via edited(_:range:changeInLength:). You subclass it for custom text processing (e.g., syntax highlighting in processEditing()).
  • NSLayoutManager (Controller): The layout engine. Converts characters to glyphs, computes line fragment rects, handles line breaking and hyphenation. Maintains the character-to-glyph mapping. One text storage can have multiple layout managers (e.g., for split-view editing).
  • NSTextContainer (View geometry): Defines the shape of the region where text is laid out. Has a size, line fragment padding, and exclusion paths (for text wrapping around images). One layout manager can have multiple containers (for multi-page/multi-column layout).

The relationships are 1:N — one storage can have many layout managers, and one layout manager can have many containers.

Q27: How does TextKit 2 differ from TextKit 1? What problem does non-contiguous layout solve? Architect

Key differences:

  • Non-contiguous layout: TextKit 2 only lays out text that's visible in the viewport. TextKit 1 lays out the entire document contiguously. For a 100,000-line document, TextKit 1 must process all lines; TextKit 2 only processes the ~50 visible lines.
  • No direct glyph access: TextKit 2 abstracts away glyphs behind NSTextLayoutFragment objects. Code that manipulates glyphs directly won't port.
  • NSTextLayoutManager replaces NSLayoutManager. NSTextContentStorage replaces NSTextStorage as the backing store interface.
  • Viewport-based rendering: Uses NSTextViewportLayoutController to manage which text layout fragments are visible and need layout.

Non-contiguous layout solves: Performance degradation with large documents. In TextKit 1, opening a 1MB text file causes a multi-second layout pass. TextKit 2 opens it instantly because it only lays out the visible portion.

Q28: What is Core Text and when would you use it instead of TextKit? Senior

Core Text is the lowest-level text layout API available to app developers. It's a C API (with Swift bridging) built on Core Graphics. The hierarchy is: CTFramesetter → CTTypesetter → CTFrame → CTLine → CTRun.

Use Core Text when:

  • Maximum performance: Core Text is thread-safe for read operations. You can compute text layout on background threads, which TextKit cannot do.
  • Custom layout: Multi-column text, text in arbitrary shapes (circles, paths), magazine-style layouts.
  • Chat app message rendering: For maximum scroll performance, render text directly with Core Text in a custom CALayer subclass.
  • PDF generation: Direct CGContext drawing with precise typographic control.

Core Text provides no editing support — it's read-only rendering. For editable text, use TextKit.

Q29: Why is emoji rendering complex? What challenges does it create for text measurement? Senior

Emoji rendering is complex because of the disconnect between Unicode code points and visual glyphs:

  • ZWJ sequences: "👨‍👩‍👧‍👦" (family) is 7 Unicode code points joined by Zero-Width Joiners, rendered as 1 glyph.
  • Skin tone modifiers: "👍🏽" is 2 code points (thumbs up + skin tone modifier) → 1 glyph.
  • Flag sequences: "🏴󠁧󠁢󠁳󠁣󠁴󠁿" (Scotland flag) is 8 code points → 1 glyph.
  • Bitmap font: Apple Color Emoji is a bitmap font, not vector. Scaling is more expensive.

Challenges for measurement: Swift's String.count counts grapheme clusters (visual characters), but Core Text and NSString work with UTF-16 code units. A family emoji is 1 Character in Swift but 11 UTF-16 code units. Splitting at the wrong UTF-16 offset produces invalid Unicode. Always use NSRange(range, in: string) for safe conversion.

Q30: How would you optimize text rendering performance for a chat app with millions of messages? L6

Architecture-level optimizations:

  1. Cache text measurements: Use CTFramesetterSuggestFrameSizeWithConstraints and cache results keyed by (text hash, width, font config). Invalidate on Dynamic Type changes.
  2. Background computation: Pre-compute all cell heights on a background queue when a page of messages loads. Core Text is thread-safe for reads.
  3. Avoid NSAttributedString for simple messages: Plain text messages (no formatting) should use UILabel with a plain string. NSAttributedString creation is expensive.
  4. Custom text rendering layer: For maximum performance, use a custom CALayer subclass that renders text with Core Text directly, bypassing UILabel/UITextView overhead.
  5. Batch layout: When loading 50 messages, compute all heights in one background pass before updating the collection view.
  6. Incremental layout: For very long messages, consider TextKit 2's non-contiguous layout to only render visible portions.
Q31: What happens when you access UITextView.layoutManager on iOS 16+? Senior

On iOS 16+, UITextView uses TextKit 2 by default. When you access the layoutManager property, UITextView performs a one-way, irreversible fallback to TextKit 1. It creates an NSLayoutManager, migrates the text content, and switches the text view's internal implementation.

This is a compatibility mechanism, but it has consequences: you lose TextKit 2's non-contiguous layout benefits, and the fallback itself has a performance cost. Apple logs a console warning when this happens.

To use TextKit 2 properly, access textLayoutManager instead of layoutManager. Check for TextKit 2 availability and use the new API.

Image Handling

Q33: A 2MB JPEG file consumes 48MB of RAM when displayed. Explain why. Senior

The file size (2MB) is the compressed data on disk. When displayed, the image must be decoded into a raw bitmap that the GPU can render. The decoded bitmap size is determined by pixel dimensions, not compression:

Memory = width × height × bytesPerPixel

For a 12MP iPhone photo (4032 × 3024 pixels) in sRGB (4 bytes per pixel): 4032 × 3024 × 4 = 48,770,832 bytes ≈ 48MB

JPEG compression achieves ~25:1 ratio, so 48MB compresses to ~2MB on disk. But the GPU needs the uncompressed bitmap to render pixels. This is why you must downsample images to the display size — a 200×200 point thumbnail on a 3x display only needs 600×600 × 4 = 1.4MB, not 48MB.

Q34: When does image decoding happen in the render pipeline, and why is this a problem? Senior

Image decoding happens during the Prepare phase of the Core Animation render loop, which runs on the main thread. When you assign a UIImage to a UIImageView, the image data stays compressed in memory. During the next render loop, Core Animation detects that the image hasn't been decoded and decodes it synchronously.

This is problematic because: (1) decoding a 12MP JPEG can take 50-100ms, (2) it happens on the main thread, blocking all UI interaction, and (3) it's invisible in your code — there's no explicit decode call to profile.

Solution: Force decode on a background thread by drawing the image into a CGContext (which triggers decode), or better, use ImageIO's CGImageSourceCreateThumbnailAtIndex with kCGImageSourceShouldCacheImmediately: true to decode at a reduced resolution.

Q35: Why is ImageIO downsampling superior to creating a UIImage and drawing it at a smaller size? Architect

The critical difference is peak memory usage:

  • ImageIO downsampling: Reads only the pixels needed from compressed data. For JPEG, it can use DCT coefficients to decode at a reduced resolution. Peak memory is proportional to the output size. A 12MP image downsampled to 600×600 peaks at ~1.4MB.
  • UIImage + draw at smaller size: Must decode the entire image into a full-resolution bitmap first (48MB), then create a second bitmap at the target size (1.4MB), then discard the full bitmap. Peak memory is 48MB + 1.4MB ≈ 49.4MB.

For a photo gallery showing 10 thumbnails, ImageIO approach peaks at ~14MB total. The UIImage approach peaks at ~48MB per image being processed (sequential) or ~490MB (parallel). The difference can mean the difference between smooth operation and Jetsam termination.

Q36: What's the difference between UIGraphicsImageRenderer and UIGraphicsBeginImageContext? Senior

UIGraphicsImageRenderer (iOS 10+) is the modern replacement:

  • Automatic pixel format: Chooses optimal format. For grayscale content, uses 1 byte/pixel instead of 4 — up to 75% memory savings.
  • Wide color support: Automatically uses Display P3 color space on capable devices.
  • Block-based API: No begin/end pairs to manage. Context is automatically cleaned up.
  • Automatic scale: Handles Retina scale automatically.

UIGraphicsBeginImageContext (legacy): Always 4 bytes/pixel, always sRGB, requires manual begin/end pairs, no wide color support. Still works but wastes memory for non-color content.

Q37: How does PHCachingImageManager improve photo grid scrolling performance? Senior

PHCachingImageManager pre-generates and caches thumbnail images for PHAssets before they're requested. You call startCachingImages(for:targetSize:contentMode:options:) with the assets that are about to become visible.

When you later call requestImage(for:targetSize:...) for a cached asset, the thumbnail is returned immediately from the cache instead of being generated on-demand. This eliminates the decode latency during scroll.

Best practice: Update the cache window as the user scrolls. Call startCachingImages for assets approaching the viewport and stopCachingImages for assets moving away. This keeps memory usage bounded while ensuring thumbnails are ready when needed.

Q38: Describe the multi-tier caching architecture used by SDWebImage/Kingfisher. Architect

Production image caching follows a three-tier architecture:

  1. Memory cache (NSCache): Stores decoded UIImage objects. ~50-100MB limit. Fastest access. Automatically evicted under memory pressure. Keyed by URL + target size.
  2. Disk cache (FileManager): Stores compressed image data on disk. ~200-500MB limit with LRU eviction. Survives app restarts. Requires background decode before display.
  3. Network (URLSession): Fetches from server on cache miss. Result is stored in both disk and memory caches.

Additional optimizations: Request coalescing (multiple views requesting the same URL share one network request), background decoding (decode on a concurrent queue, deliver on main), request cancellation (cancel when cell scrolls off-screen), and progressive loading (display partial image as data arrives).

Q39: How would you architect an image pipeline for an app like Instagram? L6

Key architectural decisions:

  1. Request deduplication: Coalesce multiple requests for the same URL into a single network fetch. Use a dictionary of URL → [completion handlers].
  2. Multi-resolution support: Cache thumbnails and full-resolution images separately. Generate thumbnails server-side. Request appropriate resolution based on display context.
  3. Progressive loading: Show low-quality placeholder immediately, then progressive JPEG scan passes, then final image. Use CGImageSourceCreateIncremental.
  4. Memory-aware caching: Monitor os_proc_available_memory() and reduce cache size under pressure. Respond to didReceiveMemoryWarning by clearing memory cache.
  5. Disk cache management: LRU eviction with configurable size limit. Background cleanup on app launch. Store metadata (URL, size, last access) in SQLite for fast lookups.
  6. Processing pipeline: Downsample → decode → apply transforms (round corners, filters) → cache processed result. All on background concurrent queue.
  7. Cancellation: Cancel in-flight requests when cells scroll off-screen. Use operation dependencies or structured concurrency for clean cancellation.
  8. Monitoring: Track cache hit rates, decode times, network latency. Alert on degradation.
Q40: What are the trade-offs between HEIF and JPEG for an iOS app? Senior

HEIF advantages: ~50% smaller file size at equivalent quality. Supports alpha channel, depth data, and animation. Hardware-accelerated encode/decode on A10+ chips.

HEIF disadvantages: Slower decode than JPEG (even with hardware acceleration). Not universally supported outside Apple ecosystem. Sharing with Android/web users may require JPEG conversion. Older devices (pre-A10) can't decode in hardware.

Key insight: Decoded memory is identical — both formats decompress to the same raw bitmap (width × height × 4 bytes). HEIF only saves disk/network space, not RAM. For an image pipeline, the format choice affects storage and transfer costs but not display memory.

Recommendation: Use HEIF for local storage and Apple-to-Apple sharing. Convert to JPEG for cross-platform sharing and web upload.

Scrolling Performance

Q41: Walk through the complete lifecycle of a UICollectionView cell during a scroll, identifying every performance-critical point. Architect
  1. Prefetch (optional): prefetchItemsAt called ~1-2 screens ahead. Start network requests and background decoding here.
  2. Dequeue: dequeueReusableCell retrieves a recycled cell from the reuse pool. prepareForReuse() is called — reset state, cancel pending image loads.
  3. Configure: cellForItemAt — set text, images, visibility. This is the most critical path. Every millisecond here directly impacts frame budget.
  4. Layout: Auto Layout resolves constraints. layoutSubviews() called. Update shadow paths here.
  5. Display: If any subview overrides draw(_:), backing stores are rendered.
  6. Prepare: Undecoded images are decoded on main thread. This is the hidden killer.
  7. Commit: Layer tree serialized to render server. More layers = higher cost.
  8. GPU compositing: Render server composites. Off-screen triggers add passes here.

Performance-critical points: Configure (CPU), Prepare (image decode), and GPU compositing (off-screen rendering).

Q42: How does UICollectionViewDataSourcePrefetching work internally? What determines which cells get prefetched? Senior

UICollectionView predicts which cells will become visible based on scroll velocity and direction. It calls prefetchItemsAt for index paths that are approximately 1-2 screens away from the current viewport.

If the user changes scroll direction, it calls cancelPrefetchingForItemsAt for index paths that are no longer approaching the viewport. This prevents wasted work.

Implementation best practices:

  • Start network requests and image decoding in prefetchItemsAt
  • Cancel in-flight requests in cancelPrefetchingForItemsAt
  • Store prefetch tasks keyed by index path for cancellation
  • Don't do synchronous work in the prefetch callback — it runs on the main thread
Q43: What are blended layers and why do they impact scroll performance? Senior

Blended layers are layers where the GPU must perform alpha compositing — reading the destination pixel, computing the blend equation, and writing the result. This happens when a layer is not fully opaque (has transparency).

For opaque layers, the GPU simply overwrites the destination pixel (one write). For blended layers, it must read + compute + write (read + compute + write = 3 operations vs 1).

Impact on scrolling: In a feed with 10 visible cells, each with 5 blended layers (transparent labels, images with alpha), that's 50 blend operations per frame. On older GPUs, this measurably impacts frame rate.

Fix: Set backgroundColor to an opaque color on all labels and views. Set isOpaque = true. Use Instruments' "Color Blended Layers" to identify: green = opaque (good), red = blended (fix it).

Q44: What causes misaligned images and how do you fix them? Senior

Misaligned images occur when an image's pixel grid doesn't align with the screen's pixel grid. Two causes:

  1. Fractional point values: A frame at (10.3, 20.7) doesn't align to pixel boundaries on a 3x display (pixels are at 1/3 point intervals). The GPU must anti-alias, which is expensive.
  2. Image size mismatch: A 200×200 pixel image displayed in a 100×100 point view on a 3x display means 200px in a 300px space. The GPU must resample every frame.

Fixes: Snap frames to pixel boundaries using round(value * scale) / scale. Downsample images to the exact display pixel size (pointSize × screenScale). Use Instruments' "Color Misaligned Images" to detect: magenta = misaligned position, yellow = stretched/compressed.

Q45: Describe a systematic approach to diagnosing scroll jank in a production app. L6

Phase 1 — Measure: Profile on the oldest supported device. Use Instruments Core Animation template. Record a 30-second scroll. Note: how many drops, how severe, when they occur (initial load vs steady scroll).

Phase 2 — Identify root cause: Is it CPU-bound or GPU-bound? Check GPU utilization — if high, problem is compositing (off-screen rendering, blending). If low, problem is CPU (cell configuration, image decode, layout). Use Time Profiler for CPU bottlenecks. Enable all Core Animation debug overlays.

Phase 3 — Fix in priority order: (1) Move image decode off main thread, (2) Downsample to display size, (3) Add shadowPath to all shadows, (4) Set opaque backgrounds, (5) Pre-compute cell heights, (6) Enable prefetching, (7) Consider shouldRasterize for complex static cells.

Phase 4 — Verify and prevent: Re-measure to confirm improvement. Set up MetricKit monitoring for production scroll hitch rate. Add performance assertions in debug builds. Document team guidelines. Add CI performance tests.

Q46: How do you pre-compute and cache cell heights for a collection view with variable-height cells? Senior

Create view models with pre-computed heights on a background thread:

  1. When data arrives (network response), dispatch to a background queue.
  2. For each item, create an NSAttributedString and use CTFramesetterSuggestFrameSizeWithConstraints to compute text height. Add padding for images, margins, etc.
  3. Store the computed height in the view model.
  4. In sizeForItemAt, return the cached height directly — no computation needed.

Cache invalidation: Invalidate when (1) Dynamic Type size changes, (2) device rotation changes available width, (3) data updates. Use a dictionary keyed by (item ID, width) for the cache.

For DiffableDataSource, compute heights during snapshot preparation, before applying the snapshot.

Q47: What's the difference between CPU-bound and GPU-bound scroll jank? How do you tell them apart? Architect

CPU-bound jank: The app process takes too long in the render loop phases (layout, display, prepare, commit). Caused by: expensive cell configuration, main-thread image decoding, complex Auto Layout, heavy draw(_:) overrides.

GPU-bound jank: The render server's GPU compositing takes too long. Caused by: off-screen rendering triggers, excessive blending, too many layers, large textures.

How to tell them apart: In Instruments, check GPU utilization. If GPU is near 100% during jank, it's GPU-bound. If GPU is low but frames are still dropping, it's CPU-bound (the app isn't committing fast enough for the render server to have work to do). Time Profiler shows CPU bottlenecks; Core Animation debug overlays show GPU issues.

Often it's a combination of both. Fix CPU issues first (they're usually more impactful), then address GPU issues.

Q48: How would you establish team-wide practices to prevent scroll performance regressions in a large iOS codebase? L6

Prevention framework:

  1. Coding standards: Document rendering performance guidelines. Require shadowPath for all shadows. Require opaque backgrounds. Require background image decoding. Include in onboarding.
  2. Reusable components: Provide optimized base classes: ShadowedView (auto-updates shadowPath), AsyncImageView (background decode + downsample), OptimizedCell (opaque, rasterized).
  3. Code review checklist: Automated lint rules or PR template checklist for rendering performance. Flag shadowOpacity without shadowPath, cornerRadius + masksToBounds in cells, UIImage(contentsOfFile:) on main thread.
  4. CI performance tests: XCUITest that scrolls through key screens on a low-end device simulator, measuring FPS with XCTMetric. Fail the build if FPS drops below threshold.
  5. Production monitoring: MetricKit scroll hitch rate dashboard. Alert when hitch rate exceeds baseline. Track per-release trends.
  6. Performance budget: Define per-cell budgets (e.g., < 4ms configuration, < 50 layers, zero off-screen triggers). Debug assertions enforce in development.
