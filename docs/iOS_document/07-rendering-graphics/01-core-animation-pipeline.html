1. The Big Picture: Two-Process Rendering

Unlike many rendering architectures, iOS splits rendering across two separate processes. Your app process prepares the layer tree and commits it; a separate system process — the render server — performs the actual GPU compositing. This is the single most important architectural fact about iOS rendering.

┌────────────────────────────────────┐ ┌────────────────────────────────────┐ │ APP PROCESS (your app) │ │ RENDER SERVER (backboardd) │ │ │ │ │ │ RunLoop tick │ │ Receives serialized layer tree │ │ ├─ Layout phase │ │ ├─ Decodes layer properties │ │ ├─ Display phase │ │ ├─ Builds render tree │ │ ├─ Prepare phase │ │ ├─ Submits GPU commands │ │ └─ Commit phase ─────────────────────── └─ Composites to framebuffer │ │ │ │ │ │ CALayer model tree │ │ Render tree (private copy) │ │ CALayer presentation tree │ │ GPU command buffers │ └────────────────────────────────────┘ └────────────────────────────────────┘ │ │ │ Mach port IPC │ │ (serialized layer tree) │ └──────────────────────────────────────────────┘
Architect Must-Know

Why the process boundary matters for performance:

  • The commit phase serializes your entire layer tree and sends it via Mach port IPC to the render server. Large layer trees = expensive serialization.
  • Once committed, the render server can composite frames independently of your app. This is why simple animations (opacity, transform) continue smoothly even if your main thread is briefly blocked.
  • Off-screen rendering happens in the render server's GPU pass, not in your app process — you can't "fix" it with background threads.
  • The render server is shared across all apps. A misbehaving app's complex layer tree can theoretically impact system-wide rendering.

Why Two Processes?

Apple designed this split for several critical reasons:

ReasonExplanation
Security Apps never get direct GPU access. The render server mediates all GPU operations, preventing one app from reading another's framebuffer.
Stability If your app crashes, the render server continues running. System UI (SpringBoard) keeps rendering.
Animation continuity Core Animation animations are executed by the render server. Once committed, they run at full frame rate even if your app's main thread is busy.
Power efficiency The render server can batch and optimize GPU work across all visible apps, reducing redundant draw calls.

2. The Three Layer Trees

Core Animation maintains three parallel representations of your layer hierarchy. Understanding these is essential for debugging animation glitches and understanding the commit cycle.

┌─────────────────┐ ┌─────────────────────┐ ┌─────────────────────┐ │ MODEL TREE │ │ PRESENTATION TREE │ │ RENDER TREE │ │ │ │ │ │ │ │ Your code sets │ │ Interpolated values │ │ Private copy in │ │ properties on │ │ during animations. │ │ render server │ │ these layers. │ │ Read-only snapshot. │ │ process. Never │ │ │ │ │ │ directly accessible.│ │ layer.position │ │ layer.presentation │ │ │ │ layer.opacity │ │ .position │ │ Serialized from │ │ layer.transform │ │ .opacity │ │ model tree at │ │ │ │ .transform │ │ commit time. │ └─────────────────┘ └─────────────────────┘ └─────────────────────┘

Model Tree

The model tree reflects the target values you set in code. When you write layer.opacity = 0.5, the model tree updates immediately, even if an animation is in flight. After an animation completes, the model tree represents the final state.

let layer = CALayer()
layer.opacity = 1.0

// Start a fade animation
let fade = CABasicAnimation(keyPath: "opacity")
fade.fromValue = 1.0
fade.toValue = 0.0
fade.duration = 0.3
layer.add(fade, forKey: "fadeOut")

// Model tree: opacity is STILL 1.0 (unless you also set it)
// You must set the final value on the model tree:
layer.opacity = 0.0
Common Pitfall

Forgetting to update the model tree when adding explicit CAAnimations causes the "flash-back" bug: the animation plays, then the layer snaps back to the model tree's old value. Always set the final value on the model layer before adding the animation, or use fillMode + isRemovedOnCompletion = false (less recommended).

Presentation Tree

The presentation tree provides a read-only snapshot of interpolated values at the current moment during an animation. Access it via layer.presentation().

// During an animation, read the current on-screen position:
if let presentationLayer = myLayer.presentation() {
    let currentPosition = presentationLayer.position
    let currentOpacity = presentationLayer.opacity
    print("Currently at: \(currentPosition), opacity: \(currentOpacity)")
}

// Common use: hit-testing during animation
override func hitTest(_ point: CGPoint, with event: UIEvent?) -> UIView? {
    guard let presentation = layer.presentation() else {
        return super.hitTest(point, with: event)
    }
    let convertedPoint = layer.convert(point, from: layer.superlayer)
    if presentation.contains(convertedPoint) {
        return self
    }
    return nil
}

Render Tree

The render tree is a private copy maintained by the render server process. You never access it directly. It's created by deserializing the data your app sends during the commit phase. The render server uses this tree to generate GPU commands for compositing.

3. Render Loop Phases

Each frame, Core Animation executes four phases within your app process before committing to the render server. These phases run on the main thread as part of the RunLoop.

┌──────────────────────────────────────────────────────────────────┐ │ ONE FRAME (16.67ms at 60Hz) │ │ │ │ ┌─────────┐ ┌─────────┐ ┌─────────┐ ┌─────────┐ │ │ │ LAYOUT │──▶│ DISPLAY │──▶│ PREPARE │──▶│ COMMIT │ │ │ │ │ │ │ │ │ │ │ │ │ │ Auto │ │ draw() │ │ Image │ │ Encode │ │ │ │ Layout │ │ calls │ │ decode │ │ & send │ │ │ │ resolve │ │ backing │ │ layer │ │ to │ │ │ │ │ │ store │ │ setup │ │ render │ │ │ │ │ │ render │ │ │ │ server │ │ │ └─────────┘ └─────────┘ └─────────┘ └─────────┘ │ │ │ │ layoutSubviews drawRect GPU texture Mach port IPC │ │ updateConstr. CALayerDel. preparation serialization │ └──────────────────────────────────────────────────────────────────┘

Phase 1: Layout

Auto Layout constraints are resolved, and layoutSubviews() is called on views that have been marked with setNeedsLayout(). This is where the Cassowary simplex solver runs.

class CustomView: UIView {
    override func layoutSubviews() {
        super.layoutSubviews()
        // Called during Layout phase
        // Expensive constraint resolution has already happened
        // Final frames are available here
        gradientLayer.frame = bounds
        shadowLayer.frame = bounds
    }
}

Phase 2: Display

Layers that have been marked with setNeedsDisplay() get their backing stores rendered. For views that override draw(_:in:), this is when Core Graphics drawing happens. Most views don't override this — UIKit handles it.

class ChartView: UIView {
    override func draw(_ rect: CGRect) {
        // Called during Display phase
        // Creates a backing store bitmap for this layer
        guard let ctx = UIGraphicsGetCurrentContext() else { return }
        ctx.setFillColor(UIColor.systemBlue.cgColor)
        ctx.fill(CGRect(x: 0, y: 0, width: 100, height: 50))
    }
}
Performance Warning

Every layer that has a backing store (from draw(_:)) consumes memory: width × height × 4 bytes × contentsScale². On a 3x Retina display, a 375×812 full-screen backing store uses 375 × 3 × 812 × 3 × 4 = ~27MB. Avoid unnecessary draw(_:) overrides.

Phase 3: Prepare

Core Animation prepares image data for the GPU. This includes decoding compressed images (JPEG, PNG) into bitmap data if they haven't been decoded yet. This is a hidden performance cost — large images decoded on the main thread during this phase cause frame drops.

Phase 4: Commit

The layer tree is serialized and sent to the render server via Mach port IPC. This includes all layer properties, transforms, and backing store data. The cost scales with the number of layers and the amount of changed data.

Deep Dive: Commit Transaction Cost

The commit phase must serialize every dirty layer property across the process boundary. Factors that increase commit cost:

  • Layer count — more layers = more data to serialize
  • Backing store size — layers with custom draw(_:) send their bitmap data
  • Image data — newly assigned contents images must be transferred
  • Complex layer hierarchies — deeply nested trees take longer to traverse

Target: keep the total commit phase under 4-5ms to leave room for the render server's work within the frame budget.

4. CATransaction: Implicit vs Explicit

Every change to a CALayer property happens within a CATransaction. Core Animation creates implicit transactions automatically, but you can also create explicit ones for fine-grained control.

Implicit Transactions

When you modify layer properties outside of an explicit transaction, Core Animation wraps them in an implicit transaction that commits at the end of the current RunLoop iteration.

// These three changes are batched into one implicit transaction
// and committed together at the end of the RunLoop cycle
myLayer.position = CGPoint(x: 100, y: 200)
myLayer.opacity = 0.5
myLayer.backgroundColor = UIColor.red.cgColor

// All three changes are committed in a single transaction
// The render server receives them atomically

Explicit Transactions

Explicit transactions give you control over animation parameters and allow you to disable implicit animations.

// Disable implicit animations for immediate property changes
CATransaction.begin()
CATransaction.setDisableActions(true)
myLayer.position = CGPoint(x: 100, y: 200)
myLayer.opacity = 0.5
CATransaction.commit()

// Custom animation duration for implicit animations
CATransaction.begin()
CATransaction.setAnimationDuration(0.5)
myLayer.cornerRadius = 20
CATransaction.setCompletionBlock {
    print("Animation completed")
}
CATransaction.commit()

// Nested transactions
CATransaction.begin()
CATransaction.setAnimationDuration(1.0)
outerLayer.opacity = 0.0

    CATransaction.begin()
    CATransaction.setAnimationDuration(0.25)
    innerLayer.opacity = 0.0  // This animates at 0.25s, not 1.0s
    CATransaction.commit()

CATransaction.commit()

UIView Animation Blocks and Transactions

UIView.animate(withDuration:) creates an explicit CATransaction under the hood and configures the animation parameters. Inside the block, property changes on UIView animatable properties generate implicit CAAnimation objects.

// Under the hood, UIView.animate creates a CATransaction
UIView.animate(withDuration: 0.3) {
    // CATransaction.begin() has been called
    // Animation duration set to 0.3
    self.myView.alpha = 0.0  // Generates CABasicAnimation on opacity
    self.myView.frame = newFrame  // Generates animations on position + bounds
    // CATransaction.commit() happens after block
}

5. The Render Server (backboardd)

The render server is a system daemon called backboardd (on iOS 6+; previously SpringBoard handled rendering). It runs in a separate process with elevated privileges for GPU access.

What the Render Server Does

  1. Deserializes the committed layer tree from your app
  2. Builds the render tree — its own optimized representation
  3. Executes animations — interpolates property values each frame
  4. Generates GPU commands — creates Metal command buffers for compositing
  5. Submits to GPU — the GPU composites layers into the framebuffer
Render Server Frame Cycle ───────────────────────── ┌──────────────┐ │ Receive layer │◀── Mach port message from app │ tree update │ └──────┬───────┘ ▼ ┌──────────────┐ │ Update render │ Apply property changes │ tree │ Advance animations └──────┬───────┘ ▼ ┌──────────────┐ │ Traverse tree │ For each visible layer: │ generate GPU │ - Determine compositing order │ commands │ - Handle opacity, masks, shadows └──────┬───────┘ - Off-screen passes if needed ▼ ┌──────────────┐ │ Submit to GPU │ Metal command buffer └──────┬───────┘ ▼ ┌──────────────┐ │ GPU composite │ Hardware compositing │ to framebuf. │ Display controller scanout └──────────────┘

Animations in the Render Server

This is why Core Animation animations are "free" from your app's perspective. Once you commit an animation, the render server handles all interpolation. Your app doesn't need to wake up each frame.

// This animation runs entirely in the render server
let rotation = CABasicAnimation(keyPath: "transform.rotation.z")
rotation.fromValue = 0
rotation.toValue = CGFloat.pi * 2
rotation.duration = 2.0
rotation.repeatCount = .infinity
spinnerLayer.add(rotation, forKey: "spin")

// Even if you block the main thread for 1 second,
// the spinner continues rotating smoothly because
// the render server drives the animation independently.

6. GPU Compositing

The GPU composites layers using a painter's algorithm — back-to-front rendering. Each layer is a textured quad that the GPU draws in order.

What the GPU Does Per Layer

OperationGPU CostNotes
Draw textured quad Low Basic compositing — very fast on modern GPUs
Apply transform Low Matrix multiplication in vertex shader
Alpha blending Low-Medium Requires reading destination pixel (blended layers)
Mask application High Requires off-screen pass to render mask, then composite
Shadow rendering High Without shadowPath: must compute layer outline first
Corner radius + clip Medium-High With masksToBounds: off-screen pass for clipping
Deep Dive: Opaque vs Transparent Layers

When a layer is marked isOpaque = true, the GPU can skip reading the destination pixel during compositing (no alpha blending needed). This is a significant optimization for layers that cover their entire bounds with content. UIView.backgroundColor with alpha 1.0 automatically sets isOpaque. Transparent backgrounds force blending, which shows up as red in Instruments' "Color Blended Layers" debug option.

7. ProMotion & Adaptive Frame Rates

ProMotion displays (iPhone 13 Pro+, iPad Pro) support adaptive refresh rates from 10Hz to 120Hz. The system dynamically adjusts based on content.

How ProMotion Decides Frame Rate

Content TypeFrame RateRationale
Static content 10Hz Nothing changing — minimum refresh to save power
Scrolling text 80-120Hz Smooth scrolling requires high refresh
Standard animations 60-120Hz Depends on animation complexity
Apple Pencil input 120Hz Maximum responsiveness for drawing
Video playback (24fps) 24Hz Matches content frame rate exactly
Always-On Display 1Hz iPhone 14 Pro+ lock screen

Requesting Specific Frame Rates

// Request a preferred frame rate range for animations
let animator = UIViewPropertyAnimator(duration: 0.5, curve: .easeInOut) {
    view.frame = targetFrame
}

// iOS 15+: Set preferred frame rate range
if #available(iOS 15.0, *) {
    let preference = CAFrameRateRange(
        minimum: 60,
        maximum: 120,
        preferred: 120  // Request highest available
    )
    animator.preferredFrameRateRange = preference
}
animator.startAnimation()

8. CADisplayLink Deep Dive

CADisplayLink is a timer synchronized to the display's refresh rate. It fires a callback before each frame is rendered, making it the correct tool for frame-by-frame animations and custom rendering loops.

class CustomAnimationController {
    private var displayLink: CADisplayLink?
    private var startTime: CFTimeInterval = 0

    func startAnimation() {
        displayLink = CADisplayLink(
            target: self,
            selector: #selector(tick)
        )

        // ProMotion: request specific frame rate range
        if #available(iOS 15.0, *) {
            displayLink?.preferredFrameRateRange = CAFrameRateRange(
                minimum: 30,
                maximum: 120,
                preferred: 120
            )
        }

        displayLink?.add(to: .main, forMode: .common)
        startTime = CACurrentMediaTime()
    }

    @objc private func tick(_ link: CADisplayLink) {
        let elapsed = link.targetTimestamp - startTime
        let frameDuration = link.targetTimestamp - link.timestamp

        // Use targetTimestamp for when this frame will appear on screen
        // Use timestamp for when the previous frame appeared
        // frameDuration tells you the actual frame interval

        updateAnimation(progress: elapsed / totalDuration)

        if elapsed >= totalDuration {
            displayLink?.invalidate()
            displayLink = nil
        }
    }

    func stopAnimation() {
        displayLink?.invalidate()
        displayLink = nil
    }
}
CADisplayLink Pitfalls
  • Retain cycle: CADisplayLink strongly retains its target. Use a proxy object or invalidate in deinit (but deinit won't fire if the display link retains you).
  • RunLoop mode: Adding to .default mode pauses during scrolling. Use .common to fire during scroll tracking.
  • Don't assume frame rate: On ProMotion displays, the interval varies. Always compute delta time from timestamps.

Proxy Pattern to Avoid Retain Cycles

class DisplayLinkProxy {
    weak var target: AnyObject?
    let selector: Selector

    init(target: AnyObject, selector: Selector) {
        self.target = target
        self.selector = selector
    }

    @objc func tick(_ link: CADisplayLink) {
        guard let target = target else {
            link.invalidate()
            return
        }
        _ = target.perform(selector, with: link)
    }
}

// Usage: display link won't retain the view controller
let proxy = DisplayLinkProxy(target: self, selector: #selector(handleFrame))
displayLink = CADisplayLink(target: proxy, selector: #selector(DisplayLinkProxy.tick))

9. Frame Pacing & Dropped Frames

A frame is "dropped" when your app doesn't finish its work before the display needs the next frame. Understanding frame budgets is critical.

DisplayRefresh RateFrame BudgetDevices
Standard 60Hz 16.67ms iPhone SE, iPhone 11, iPad (non-Pro)
ProMotion 120Hz 8.33ms iPhone 13 Pro+, iPad Pro
Frame Budget Breakdown (16.67ms at 60Hz) ────────────────────────────────────────── ┌──────────┬──────────┬──────────┬──────────┬──────────┐ │ Layout │ Display │ Prepare │ Commit │ Render │ │ ~2ms │ ~2ms │ ~1ms │ ~2ms │ Server │ │ │ │ │ │ ~8ms │ └──────────┴──────────┴──────────┴──────────┴──────────┘ ◀─── App Process (~7ms) ───▶◀── Render Server (~8ms) ──▶ If app takes > ~8ms, render server doesn't have enough time → frame drop → visible jank

Detecting Dropped Frames

// Monitor for hitches using MetricKit (iOS 14+)
class MetricsSubscriber: NSObject, MXMetricManagerSubscriber {
    func didReceive(_ payloads: [MXMetricPayload]) {
        for payload in payloads {
            if let animationMetrics = payload.animationMetrics {
                let scrollHitchRate = animationMetrics.scrollHitchTimeRatio
                print("Scroll hitch rate: \(scrollHitchRate)")
            }
        }
    }

    func didReceive(_ payloads: [MXDiagnosticPayload]) {
        for payload in payloads {
            if let hangDiagnostics = payload.hangDiagnostics {
                for hang in hangDiagnostics {
                    print("Hang duration: \(hang.hangDuration)")
                }
            }
        }
    }
}

// Register subscriber
let subscriber = MetricsSubscriber()
MXMetricManager.shared.add(subscriber)

11. Architect-Level Patterns

Architect Must-Know: Designing for the Pipeline

At Staff level, you should be able to design rendering strategies for an entire app. Key principles:

  1. Minimize layer count — flatten hierarchies where possible. Use shouldRasterize to collapse complex subtrees into single cached bitmaps.
  2. Keep the commit phase lean — avoid changing thousands of layers per frame. Batch updates.
  3. Leverage the render server — use Core Animation animations (transform, opacity, position) that run in the render server, not frame-by-frame UIView updates.
  4. Decode images off the main thread — the Prepare phase decodes images synchronously. Pre-decode on a background queue.
  5. Profile on real hardware — the Simulator uses the Mac's GPU, which has vastly different performance characteristics.

Layer Flattening Strategy

// For complex cell layouts that don't change often,
// rasterize the entire subtree into a single cached bitmap
func configureForPerformance(_ cell: UICollectionViewCell) {
    cell.layer.shouldRasterize = true
    cell.layer.rasterizationScale = UIScreen.main.scale

    // Mark all sublayers as opaque where possible
    cell.contentView.layer.isOpaque = true
    cell.contentView.backgroundColor = .systemBackground
}
Best Practice Summary
  • Understand that your app and the render server are separate processes connected by Mach port IPC
  • Keep all four render loop phases (layout, display, prepare, commit) within the frame budget
  • Use explicit CATransaction to batch changes and control animation timing
  • Leverage the render server for animations — prefer CAAnimation over per-frame property updates
  • On ProMotion devices, use CAFrameRateRange to request appropriate frame rates
  • Always profile on real devices — Simulator GPU characteristics are misleading
