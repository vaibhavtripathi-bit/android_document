1. ASLR (Address Space Layout Randomization)

ASLR is the foundational memory-layout defense on iOS. It randomizes the virtual addresses at which the executable, shared libraries, stack, heap, and mmap regions are loaded each time a process starts. An attacker who discovers a gadget address in one run cannot reuse it in the next — the entire address space has shifted.

1.1 What ASLR Randomizes

  • Main executable base — the __TEXT segment of your Mach-O binary is loaded at a random base address (requires PIE)
  • dyld_shared_cache — the giant pre-linked blob of all system frameworks (~1.5 GB) is mapped at a random address chosen at boot time (not per-process — shared across all processes for memory efficiency)
  • Stack — the initial stack pointer is randomized with a random gap at the top of the stack region
  • Heap — malloc zones and mmap regions are placed at randomized base addresses
  • dyld itself — the dynamic linker's load address is randomized

1.2 Entropy Levels on iOS

RegionEntropy (bits)Possible PositionsNotes

Deep Dive — dyld_shared_cache Randomization

The shared cache is randomized once per boot, not per process launch. This is a deliberate trade-off: per-process randomization would prevent page sharing across processes, wasting hundreds of megabytes of RAM. The boot-time randomization still defeats remote attacks (attacker can't predict the layout of a target device) but means a local attacker who leaks one process's shared cache address knows it for all processes until reboot.

1.3 PIE (Position Independent Executable)

ASLR for the main executable requires the binary to be compiled as a Position Independent Executable (PIE). Since iOS 4.3, Apple has required all App Store binaries to be PIE. The compiler generates position-independent code using PC-relative addressing — no absolute addresses are baked into the __TEXT segment.

1.4 How ASLR Defeats ROP/JOP Attacks

Return-Oriented Programming (ROP) chains together small instruction sequences ("gadgets") ending in ret instructions, found at known addresses in libraries. ASLR breaks this by making gadget addresses unpredictable. The attacker must first leak a pointer to de-randomize the layout — turning a single vulnerability into a two-stage exploit.

WITHOUT ASLR — Fixed Layout (Predictable) ┌──────────────────────────────────────────────────┐ │ 0x100000000 │ __TEXT (executable code) │ ├──────────────┼───────────────────────────────────┤ │ 0x100080000 │ __DATA (globals, GOT) │ ├──────────────┼───────────────────────────────────┤ │ 0x180000000 │ dyld_shared_cache (UIKit, etc.) │ ├──────────────┼───────────────────────────────────┤ │ 0x16FDFC000 │ Stack ↓ │ ├──────────────┼───────────────────────────────────┤ │ 0x600000000 │ Heap ↑ │ └──────────────┴───────────────────────────────────┘ Attacker knows: gadget at 0x180042A3C → always valid WITH ASLR — Randomized Layout (Unpredictable) ┌──────────────────────────────────────────────────┐ │ 0x102A40000 │ __TEXT (+0x2A40000 slide) │ ├──────────────┼───────────────────────────────────┤ │ 0x102AC0000 │ __DATA (+0x2A40000 slide) │ ├──────────────┼───────────────────────────────────┤ │ 0x1837B0000 │ dyld_shared_cache (+0x37B0000) │ ├──────────────┼───────────────────────────────────┤ │ 0x16F8A4000 │ Stack (random gap) │ ├──────────────┼───────────────────────────────────┤ │ 0x60013C000 │ Heap (random base) │ └──────────────┴───────────────────────────────────┘ Attacker's gadget at 0x180042A3C → INVALID (SIGBUS/SIGSEGV)
Pitfall — ASLR Is Not Enough Alone

ASLR is probabilistic, not deterministic. On 32-bit devices (now obsolete), entropy was as low as 8 bits — brute-forceable in seconds. Even on 64-bit, a single pointer leak (e.g., via a format string bug or an info-leak vulnerability) de-randomizes the entire region. ASLR must be combined with PAC, stack canaries, and W^X for meaningful protection.

2. Stack Canaries

A stack canary (also called a stack cookie or stack guard) is a random value placed on the stack between local variables and the saved return address. Before a function returns, the compiler-inserted code checks whether the canary has been modified. If it has, a buffer overflow has occurred, and the process is immediately terminated — preventing the attacker from hijacking control flow via a corrupted return address.

2.1 How Stack Canaries Work

Stack Frame Layout (ARM64, with canary) High Address ┌─────────────────────────────────┐ │ Caller's Frame │ ├─────────────────────────────────┤ │ Return Address (LR saved) │ ← attacker wants to overwrite this ├─────────────────────────────────┤ │ Saved Frame Pointer (FP/x29) │ ├─────────────────────────────────┤ │ ▓▓▓ STACK CANARY ▓▓▓ │ ← random value, checked before ret ├─────────────────────────────────┤ │ Local Variable N │ │ ... │ │ Local Variable 2 │ │ Local Variable 1 │ ← buffer overflow starts here ├─────────────────────────────────┤ │ [buffer being overflowed] │ ← write goes UP toward canary └─────────────────────────────────┘ Low Address Overflow path: buffer → locals → CANARY (corrupted!) → FP → LR ↑ Detection point: __stack_chk_fail()

2.2 Canary Lifecycle

  1. Process start: The dynamic linker generates a random canary value and stores it in a thread-local variable (__stack_chk_guard)
  2. Function prologue: The compiler inserts code to load the canary and push it onto the stack after the saved registers
  3. Function epilogue: Before the function returns, the compiler inserts code to compare the stack canary with the stored reference value
  4. Mismatch: If the values differ, __stack_chk_fail() is called, which calls abort() — the process is killed immediately
// Pseudo-assembly of canary check (ARM64)
// Function prologue:
adrp    x8, ___stack_chk_guard@PAGE
ldr     x8, [x8, ___stack_chk_guard@PAGEOFF]
stur    x8, [x29, #-8]          // store canary on stack

// ... function body ...

// Function epilogue:
ldur    x8, [x29, #-8]          // load canary from stack
adrp    x9, ___stack_chk_guard@PAGE
ldr     x9, [x9, ___stack_chk_guard@PAGEOFF]
cmp     x8, x9                     // compare with reference
b.ne    __stack_chk_fail_abort     // MISMATCH → abort()
ret

Pitfall — Canaries Don't Protect Everything

Stack canaries only detect linear buffer overflows that write sequentially past the buffer. An attacker who can write to an arbitrary offset (e.g., via an index-out-of-bounds bug) can skip over the canary and directly overwrite the return address. Canaries also don't protect heap allocations at all. They are one layer in a defense-in-depth strategy.

3. Pointer Authentication Codes (PAC) — A12+

Pointer Authentication is Apple's most significant hardware security feature since ASLR. Introduced with the A12 Bionic (2018), PAC uses the unused upper bits of 64-bit pointers to store a cryptographic signature (the PAC). Any attempt to tamper with a signed pointer — even by a single bit — invalidates the signature, causing a fault when the pointer is authenticated. This defeats ROP, JOP, and other code-reuse attacks at the hardware level.

3.1 How PAC Works

On ARM64, virtual addresses on iOS use only 36–39 bits (depending on the page size and virtual address space configuration). The upper bits (bits 47–63 in a 48-bit VA space) are unused. PAC repurposes these bits to store a truncated QARMA cryptographic hash of:

  1. The pointer value itself (the address)
  2. A context value (a modifier — typically the stack pointer or a type discriminator)
  3. A secret key stored in a system register, inaccessible to userspace
PAC Pointer Format (ARM64e, 48-bit VA) 63 56 55 48 47 0 ┌──────────┬─────────┬──────────────────────────────────────────┐ │ PAC │ PAC │ Virtual Address │ │ (high) │ (low) │ (actual pointer value, 48 bits) │ │ 8 bits │ 8 bits │ │ └──────────┴─────────┴──────────────────────────────────────────┘ ↑ └── Cryptographic signature = QARMA(pointer, context, key) Signing: raw_ptr ──→ PACIA x0, sp ──→ signed_ptr (PAC in upper bits) Verifying: signed_ptr ──→ AUTIA x0, sp ──→ raw_ptr (or FAULT if tampered) If attacker modifies any bit of the pointer: signed_ptr' ──→ AUTIA x0, sp ──→ ❌ PAC mismatch → upper bits corrupted → dereference causes SIGSEGV

3.4 Context Diversification

The context value is critical for PAC security. Using the stack pointer (SP) as context for return addresses means a signed return address is only valid for the specific stack frame that created it. An attacker who captures a signed pointer from one call site cannot replay it at another — the SP will differ, causing authentication to fail.

// PAC-protected function prologue/epilogue (ARM64e)
// Prologue: sign LR with SP as context
pacibsp                            // x30 = PACIB(x30, sp)
stp     x29, x30, [sp, #-16]!   // save signed LR to stack
mov     x29, sp

// ... function body ...

// Epilogue: authenticate LR before returning
ldp     x29, x30, [sp], #16     // restore signed LR
retab                              // AUTIB(x30, sp) + ret
                                   // if tampered → fault before branch

Deep Dive — ARM64e ABI

The arm64e ABI is a separate ABI from arm64. Binaries compiled for arm64e embed PAC signing/authentication instructions that would be illegal on pre-A12 hardware. Apple ships system frameworks as arm64e on supported devices, but third-party apps still target arm64 (the PAC instructions are transparently inserted by the system's arm64e dyld for framework code). Apple has not yet opened the arm64e ABI to third-party developers — your app binary runs as arm64, but all system calls and framework code it invokes run as arm64e with full PAC protection.

Pitfall — PAC Bypass Research

PAC is not unbreakable. Researchers have demonstrated bypasses via: (1) PAC forgery using signing gadgets — finding existing code that signs attacker-controlled values; (2) context confusion — using a pointer signed with one context in a location expecting a different context; (3) brute-forcing the PAC (feasible for short PAC widths on some configurations). Apple continuously hardens PAC by increasing context diversity and reducing signing gadget availability in system frameworks.

4. __DATA_CONST Segment Hardening

The __DATA_CONST segment is a Mach-O segment that contains data which must be writable during process startup (for dyld fixups — rebasing and binding) but is constant for the rest of the process's lifetime. After dyld completes its fixups, it marks __DATA_CONST pages as read-only using mprotect(), preventing any further modification.

4.1 What Lives in __DATA_CONST

__cfstringCoreFoundation string objectsPrevents string replacement attacks
__objc_classlistObjC class list pointersPrevents class substitution
__objc_protolistObjC protocol list pointersPrevents protocol hijacking

4.2 __DATA_CONST vs __DATA

Mach-O Segment Layout (Simplified) ┌─────────────────────────────────────────────────────────────┐ │ __TEXT │ r-x │ Always read-only + executable │ │ │ │ Code, string literals, const data │ ├──────────────────┼───────┼───────────────────────────────────┤ │ __DATA_CONST │ rw- │ Writable during dyld fixup ONLY │ │ │→ r-- │ Then mprotect'd to read-only │ │ │ │ GOT, const data, ObjC metadata │ ├──────────────────┼───────┼───────────────────────────────────┤ │ __DATA │ rw- │ Always writable │ │ │ │ Global vars, ObjC ivar storage, │ │ │ │ non-const statics, __bss │ ├──────────────────┼───────┼───────────────────────────────────┤ │ __LINKEDIT │ r-- │ Always read-only │ │ │ │ Symbol tables, code signatures │ └──────────────────┴───────┴───────────────────────────────────┘ Timeline: dyld load → fixup rebases/binds (__DATA_CONST is rw-) → mprotect(r--) → main() ↑ GOT is now immutable

4.3 Security Impact

Before __DATA_CONST, the GOT lived in __DATA and was writable for the entire process lifetime. A memory corruption bug could overwrite a GOT entry to redirect any function call — a classic exploitation primitive. With __DATA_CONST:

  • GOT overwrites trigger a SIGBUS (writing to read-only memory) instead of silently succeeding
  • ObjC class metadata can't be modified after launch, preventing runtime class injection
  • vtable pointers for C++ objects in __DATA_CONST are immutable, defeating vtable hijacking
// Inspecting __DATA_CONST with otool
$ otool -l MyApp | grep -A5 __DATA_CONST

Best Practice — Minimize __DATA Usage

Move as much data as possible into __DATA_CONST by declaring globals as let (Swift) or const (C/ObjC). The linker automatically places constant-after-init data in __DATA_CONST. Audit your binary with otool -l or size -m to verify that sensitive metadata (GOT entries, class lists) resides in __DATA_CONST rather than __DATA.

5. Memory Tagging Extension (MTE)

ARM Memory Tagging Extension (MTE), introduced in ARMv8.5-A, is a hardware feature that assigns a 4-bit tag to every 16-byte granule of memory and to every pointer. On every memory access, the hardware checks that the pointer's tag matches the memory's tag. A mismatch indicates a spatial (buffer overflow) or temporal (use-after-free) memory safety violation.

5.1 How MTE Works

MTE Pointer Format 63 60 59 0 ┌──────┬──────────────────────────────────────────────────────┐ │ TAG │ Virtual Address (60 bits) │ │4 bits│ │ └──┬───┴──────────────────────────────────────────────────────┘ │ │ Memory Granules (each 16 bytes has a 4-bit tag) │ ┌─────┬─────┬─────┬─────┬─────┬─────┬─────┬─────┐ │ │ T=3 │ T=3 │ T=3 │ T=7 │ T=7 │ T=2 │ T=2 │ T=2 │ │ └─────┴─────┴─────┴─────┴─────┴─────┴─────┴─────┘ │ ↑ ↑ │ │ │ └──────┘ │ Tag=3 matches T=3 ✓ Tag=3 vs T=7 ✗ → FAULT Buffer Overflow Detection: malloc(48) → returns ptr with tag=3, memory tagged [3,3,3,7,7,...] ptr[48] → tag=3 vs memory tag=7 → TAG MISMATCH → fault Use-After-Free Detection: ptr = malloc(32) → tag=5 free(ptr) → memory re-tagged to 9 *ptr → tag=5 vs memory tag=9 → TAG MISMATCH → fault

Deep Dive — Why Apple Hasn't Adopted MTE

As of 2025, Apple's A-series and M-series chips implement ARMv8.6-A but do not enable MTE. Likely reasons: (1) MTE's 4-bit tag provides weaker probabilistic guarantees than PAC's ~16-bit signature; (2) MTE requires additional memory for tag storage (~3% overhead), which is significant on memory-constrained iPhones; (3) Apple's investment in PAC, __DATA_CONST, and Swift's memory safety may provide sufficient coverage; (4) MTE is primarily valuable for detecting bugs during development — Android has adopted it for this purpose via the Scudo allocator. Apple may adopt MTE in future chips for defense-in-depth, especially as the memory overhead becomes less significant.

6. Swift vs ObjC Runtime Security

The choice of language runtime has profound security implications. Objective-C's dynamic runtime — while powerful for flexibility — exposes a large attack surface. Swift's design choices (static dispatch, value types, bounds checking) eliminate entire classes of vulnerabilities.

6.1 Objective-C Runtime Attack Surface

Dynamic Dispatch via objc_msgSend

Every ObjC method call goes through objc_msgSend, which performs a runtime lookup: read the object's isa pointer → find the class → walk the method cache/list → jump to the IMP (function pointer). Each step is an attack opportunity:

// What happens when you call [obj doSomething]:
// 1. Read obj->isa to get the Class
// 2. Look up "doSomething" in class's method cache
// 3. If miss, walk method_list, superclass chain
// 4. Jump to the IMP (function pointer)

// Attack vectors:
// - Corrupt isa → redirect to fake class with attacker's method table
// - Corrupt method cache → point selector to attacker's function
// - Swizzle method → replace IMP at runtime (legitimate API!)

// Method swizzling — a "feature" that's also an attack vector
@implementation NSURLSession (Intercept)
+ (void)load {
    Method original = class_getInstanceMethod(
        self, @selector(dataTaskWithURL:completionHandler:));
    Method swizzled = class_getInstanceMethod(
        self, @selector(evil_dataTaskWithURL:completionHandler:));
    method_exchangeImplementations(original, swizzled);
    // All network requests now go through attacker's code
}
@end

isa Pointer Manipulation

Every ObjC object's first word is the isa pointer — a pointer to its class. Corrupting this pointer redirects all method dispatch for that object. On pre-A12 devices without PAC, this is a single 8-byte write. On A12+, the isa is PAC-signed with the DA key, making corruption detectable.

Runtime Method Resolution

ObjC provides +resolveInstanceMethod:, -forwardInvocation:, and -forwardingTargetForSelector: — mechanisms that let objects dynamically add or redirect methods at runtime. While powerful, these create an unpredictable control flow that static analysis tools cannot fully reason about.

6.2 Swift Runtime Security Advantages

Static and VTable Dispatch

// Swift dispatch mechanisms (from most to least secure):

// 1. Static/Direct dispatch — compile-time resolved, inlined
struct Calculator {
    func add(_ a: Int, _ b: Int) -> Int { a + b }
    // Compiled to a direct function call — no indirection, no attack surface
}

// 2. VTable dispatch — class methods (not @objc)
class Renderer {
    func draw() { }
    // Dispatched via vtable (like C++), NOT objc_msgSend
    // vtable is in __DATA_CONST → read-only after launch
}

// 3. Witness table dispatch — protocol conformance
protocol Drawable { func draw() }
// Protocol witness table is also in __DATA_CONST

// 4. ObjC-compatible dispatch — only when explicitly requested
class ViewController: UIViewController {
    @objc dynamic func handleTap() { }
    // Only this method uses objc_msgSend
}

Value Types on Stack

Swift structs and enums are value types allocated on the stack (or inline in their container). They have no heap allocation, no reference counting, no isa pointer — eliminating heap corruption, use-after-free, and isa manipulation for these types entirely.

Bounds Checking

// Swift arrays have bounds checking by default
let array = [1, 2, 3]
let value = array[5]  // → fatal error: Index out of range
                       // Process is killed, NOT undefined behavior

// In C/ObjC:
int arr[3] = {1, 2, 3};
int value = arr[5];  // → undefined behavior, reads garbage
                      // Attacker can use this to leak memory

Best Practice — Maximize Swift, Minimize @objc

For security-critical code paths (authentication, encryption, payment processing), use pure Swift with structs and protocols. Avoid @objc dynamic unless required for UIKit interop. Every @objc annotation re-introduces the ObjC runtime attack surface for that method. Use final on classes to enable static dispatch and prevent subclass-based attacks.

7. Hardened Runtime

The Hardened Runtime is a set of security policies that restrict what a process can do at runtime. While primarily a macOS feature (required for notarization since macOS 10.14.5), many of its protections are the default and mandatory behavior on iOS. Understanding the Hardened Runtime is essential for cross-platform development and for understanding why iOS is more locked down than macOS.

Deep Dive — Why iOS Is Stricter

iOS was designed as a closed platform from day one. macOS inherited decades of UNIX openness and had to retrofit security via opt-in mechanisms. The Hardened Runtime on macOS is essentially "make this app behave like an iOS app." On iOS, there's no opt-in — the kernel enforces these policies for all processes. This is why iOS has never had a widespread malware problem despite being the world's most targeted platform: the attack surface is fundamentally smaller.

8. W^X (Write XOR Execute)

W^X (Write XOR Execute) is a memory protection policy that ensures no memory page is simultaneously writable and executable. A page can be r-x (readable + executable) or rw- (readable + writable), but never rwx. This prevents an attacker who achieves a write primitive from injecting and executing shellcode in the same memory region.

8.1 How iOS Enforces W^X

On iOS, W^X is enforced at the kernel level by XNU. The mmap() and mprotect() system calls reject any request that would create a page with both PROT_WRITE and PROT_EXEC:

// This FAILS on iOS (returns MAP_FAILED / EPERM):
void *page = mmap(NULL, 4096,
    PROT_READ | PROT_WRITE | PROT_EXEC,  // ← W+X rejected
    MAP_ANON | MAP_PRIVATE, -1, 0);

// This also FAILS — can't add EXEC to a writable page:
void *page = mmap(NULL, 4096,
    PROT_READ | PROT_WRITE,
    MAP_ANON | MAP_PRIVATE, -1, 0);
// Write shellcode to page...
mprotect(page, 4096, PROT_READ | PROT_EXEC);  // ← DENIED

// The ONLY way to get executable memory on iOS:
// 1. Load a signed Mach-O (dyld verifies code signature per-page)
// 2. WebKit's JIT (com.apple.jit entitlement — Apple-only)

8.2 The JIT Exception — WebKit Only

Just-In-Time compilation requires generating machine code at runtime and executing it — inherently violating W^X. On iOS, Apple grants a special exception only to WebKit (Safari's JavaScript engine, JavaScriptCore):

  • WebKit uses the com.apple.jit private entitlement
  • It uses a dual-mapping technique: the JIT region is mapped twice at different virtual addresses — one mapping is rw- (for writing generated code) and the other is r-x (for executing it). The same physical page backs both mappings
  • The writable mapping is accessible only from a dedicated JIT compilation thread, and its address is randomized
  • Third-party browsers on iOS must use WebKit (WKWebView) — they cannot ship their own JS engine with JIT. This is why Chrome on iOS uses WebKit, not V8

Pitfall — App Review and Dynamic Code

Apple's App Review guidelines (section 2.5.2) explicitly prohibit downloading and executing code. Even if you could somehow bypass W^X (you can't on non-jailbroken devices), doing so would result in App Store rejection. The only exceptions are: (1) WebKit JavaScript execution, (2) JavaScriptCore's interpreter mode. React Native and Flutter work because they either interpret JS (React Native in debug) or compile to native ARM64 ahead of time (Flutter, React Native Hermes).

9. Architect: Defense-in-Depth Security Architecture

Architect Must-Know

At Staff/L6 level, you must understand how all runtime security mechanisms compose into a layered defense. No single mechanism is sufficient — ASLR can be bypassed with an info leak, PAC can be bypassed with signing gadgets, stack canaries don't protect the heap, W^X doesn't prevent code reuse. The architect's job is to design systems that require an attacker to chain multiple bypasses, making exploitation exponentially harder.

9.3 Runtime Integrity Verification

import Foundation
import DeviceCheck
import CryptoKit

enum RuntimeIntegrity {

    static func performChecks() -> [String: Bool] {
        var results: [String: Bool] = [:]

        results["debugger"] = !isDebuggerAttached()
        results["sandbox"] = isSandboxIntact()
        results["codeSignature"] = isCodeSignatureValid()
        results["dynamicLibraries"] = !hasInjectedLibraries()

        return results
    }

    private static func isDebuggerAttached() -> Bool {
        var info = kinfo_proc()
        var size = MemoryLayout<kinfo_proc>.stride
        var mib: [Int32] = [CTL_KERN, KERN_PROC, KERN_PROC_PID, getpid()]
        let result = sysctl(&mib, UInt32(mib.count), &info, &size, nil, 0)
        guard result == 0 else { return false }
        return (info.kp_proc.p_flag & P_TRACED) != 0
    }

    private static func isSandboxIntact() -> Bool {
        let jailbreakPaths = [
            "/Applications/Cydia.app",
            "/usr/sbin/sshd",
            "/bin/bash",
            "/etc/apt",
            "/private/var/lib/apt"
        ]
        for path in jailbreakPaths {
            if FileManager.default.fileExists(atPath: path) {
                return false
            }
        }
        // Verify sandbox by attempting a forbidden operation
        let canWrite = FileManager.default
            .isWritableFile(atPath: "/private/var/mobile")
        return !canWrite
    }

    private static func isCodeSignatureValid() -> Bool {
        guard let executableURL = Bundle.main.executableURL,
              let data = try? Data(contentsOf: executableURL) else {
            return false
        }
        let hash = SHA256.hash(data: data)
        let expected = expectedExecutableHash()
        return hash.description == expected
    }

    private static func hasInjectedLibraries() -> Bool {
        let expectedCount = expectedDylibCount()
        let actualCount = _dyld_image_count()
        if actualCount > expectedCount { return true }

        for i in 0..<actualCount {
            guard let name = _dyld_get_image_name(i) else { continue }
            let path = String(cString: name)
            if path.contains("MobileSubstrate") ||
               path.contains("cycript") ||
               path.contains("frida") ||
               path.contains("SSLKillSwitch") {
                return true
            }
        }
        return false
    }

    private static func expectedExecutableHash() -> String {
        // In production: fetch from server or embed encrypted
        return "..."
    }

    private static func expectedDylibCount() -> UInt32 {
        // Baseline dylib count for your app — update per release
        return 250
    }
}

9.5 Designing for Financial / Health Apps

Architect — Security Architecture for Regulated Apps

For apps handling financial transactions (PCI-DSS), health data (HIPAA), or government data (FedRAMP), the runtime security architecture must address:

  • Device integrity: Use App Attest to verify the device hasn't been compromised. Fall back to heuristic checks (sandbox integrity, dylib count, debugger detection) on older devices. Never trust client-side checks alone — validate server-side.
  • Memory hygiene: Zero sensitive data (keys, tokens, PII) immediately after use. Use Data with resetBytes(in:) instead of String for secrets (String may be interned or copied by the runtime). Consider using UnsafeMutableRawBufferPointer with explicit zeroing for cryptographic keys.
  • Anti-tampering: Verify the executable hash at launch. Monitor for injected dylibs. Use server-driven feature gating so compromised clients can be remotely disabled.
  • Secure communication: Certificate pinning (via URLSessionDelegate or NSPinnedDomains), mutual TLS for high-value endpoints, encrypted payloads for defense-in-depth against TLS interception.
  • Assume breach: Design server APIs so that even a fully compromised client cannot cause unauthorized transactions. Use server-side validation, rate limiting, anomaly detection, and step-up authentication for sensitive operations.

Best Practice — The 80/20 of Runtime Security

For most apps, the highest-impact security decisions are: (1) Write in Swift, not ObjC — this alone eliminates most memory corruption vulnerabilities; (2) Use the latest deployment target — this ensures your users' devices have PAC, latest ASLR entropy, and current kernel mitigations; (3) Don't disable compiler protections — leave -fstack-protector-strong enabled, don't use -Ounchecked in release builds; (4) Use App Attest for server communication — hardware-backed device integrity is far more reliable than client-side jailbreak detection. The OS provides an extraordinary security baseline — your job is to not weaken it.
