1. App Transport Security (ATS)

App Transport Security is a system-level policy introduced in iOS 9 that forces all HTTP connections made through URLSession, NSURLConnection, and any framework built on top of them (Alamofire, SDWebImage, etc.) to meet a minimum set of security requirements. ATS operates at the SecureTransport / Network.framework layer — it is not a simple URL filter but a TLS policy enforcer embedded in the networking stack.

1.1 Default ATS Requirements

When ATS is enabled (the default since iOS 9), every HTTPS connection must satisfy all of the following:

  • TLS 1.2 or later — connections using TLS 1.0 or 1.1 are rejected at the handshake level
  • Forward secrecy — the cipher suite must use an ephemeral key exchange (ECDHE). Static RSA key exchange is blocked because compromise of the server's private key would decrypt all past traffic
  • Certificate requirements — the server certificate must be signed with SHA-256 or stronger, and the RSA key must be at least 2048 bits (or ECC 256 bits)
  • No plaintext HTTP — http:// URLs are blocked entirely unless explicitly exempted

The specific cipher suites allowed by ATS are:

// ATS-compliant cipher suites (as of iOS 17)
TLS_ECDHE_ECDSA_WITH_AES_256_GCM_SHA384
TLS_ECDHE_ECDSA_WITH_AES_128_GCM_SHA256
TLS_ECDHE_RSA_WITH_AES_256_GCM_SHA384
TLS_ECDHE_RSA_WITH_AES_128_GCM_SHA256
TLS_ECDHE_ECDSA_WITH_CHACHA20_POLY1305_SHA256
TLS_ECDHE_RSA_WITH_CHACHA20_POLY1305_SHA256

// TLS 1.3 cipher suites (always ATS-compliant)
TLS_AES_256_GCM_SHA384
TLS_AES_128_GCM_SHA256
TLS_CHACHA20_POLY1305_SHA256

Pitfall: App Review and NSAllowsArbitraryLoads

Since January 2017, Apple requires a justification for NSAllowsArbitraryLoads = YES during App Review. Apps that set this without a valid reason (e.g., a web browser that must load arbitrary content) are routinely rejected. Even with a justification, Apple may request that you use NSExceptionDomains for specific hosts instead of the global kill switch. Starting with iOS 10, if you set both NSAllowsArbitraryLoads and any of the granular keys (NSAllowsArbitraryLoadsForMedia, NSAllowsArbitraryLoadsInWebContent), the granular keys take precedence and NSAllowsArbitraryLoads is ignored.

1.4 ATS and Third-Party SDKs

ATS applies process-wide — it affects every framework and SDK linked into your app. This creates friction with third-party ad SDKs, analytics providers, and legacy backend services that may not support TLS 1.2. The recommended approach:

// Diagnosing ATS failures in the console
// Set CFNETWORK_DIAGNOSTICS=1 in your scheme's environment variables
// This logs detailed TLS negotiation info including:
//   - Offered cipher suites
//   - Server-selected cipher suite
//   - TLS version negotiated
//   - Certificate chain details
//   - ATS policy evaluation result

// You can also use nscurl to test ATS compliance from Terminal:
// $ /usr/bin/nscurl --ats-diagnostics https://your-server.com
Deep Dive: ATS Internals

ATS is implemented in the CFNetwork framework, specifically in the _CFNetworkHTTPConnectionCacheSetLimit and SSLSetProtocolVersionMin code paths. When URLSession creates a connection, it reads the ATS policy from the app's Info.plist (cached at launch), resolves per-domain exceptions, and configures the underlying sec_protocol_options_t object accordingly. The policy is evaluated before the TLS handshake begins — the system sets the minimum TLS version and allowed cipher suites on the socket, so non-compliant servers fail at the protocol level, not via an application-layer check.

2. TLS 1.2 vs TLS 1.3

iOS 12.2 added TLS 1.3 support, and iOS 13.4 enabled it by default. Understanding the differences is critical for diagnosing connection failures and making informed security architecture decisions.

2.1 TLS 1.2 Handshake (2-RTT)

TLS 1.2 Full Handshake (2 Round Trips): Client Server │ │ │──── ClientHello ──────────────────────────────▶│ RTT 1 │ (TLS version, random, cipher suites, │ start │ extensions, SNI) │ │ │ │◀─── ServerHello ──────────────────────────────│ │ (chosen cipher, random, session ID) │ │◀─── Certificate ──────────────────────────────│ │ (server cert chain) │ │◀─── ServerKeyExchange ────────────────────────│ RTT 1 │ (ECDHE params, signature) │ end │◀─── ServerHelloDone ──────────────────────────│ │ │ │──── ClientKeyExchange ────────────────────────▶│ RTT 2 │ (ECDHE public key) │ start │──── ChangeCipherSpec ─────────────────────────▶│ │──── Finished (encrypted) ─────────────────────▶│ │ │ │◀─── ChangeCipherSpec ─────────────────────────│ RTT 2 │◀─── Finished (encrypted) ─────────────────────│ end │ │ │◀═══ Application Data (encrypted) ═════════════▶│

2.2 TLS 1.3 Handshake (1-RTT)

TLS 1.3 Full Handshake (1 Round Trip): Client Server │ │ │──── ClientHello ──────────────────────────────▶│ RTT 1 │ (TLS version, random, cipher suites, │ start │ key_share (ECDHE public key), │ │ supported_groups, signature_algorithms) │ │ │ │◀─── ServerHello ──────────────────────────────│ │ (chosen cipher, key_share) │ │◀─── {EncryptedExtensions} ────────────────────│ │◀─── {Certificate} ───────────────────────────│ │◀─── {CertificateVerify} ─────────────────────│ RTT 1 │◀─── {Finished} ──────────────────────────────│ end │ │ │──── {Finished} ───────────────────────────────▶│ │ │ │◀═══ Application Data (encrypted) ═════════════▶│ {} = encrypted with handshake keys

2.3 TLS 1.3 0-RTT Resumption

TLS 1.3 0-RTT Resumption (Zero Round Trips for first data): Client Server │ │ │──── ClientHello ──────────────────────────────▶│ │ + early_data extension │ │ + pre_shared_key (from previous session) │ │──── (0-RTT) Application Data ─────────────────▶│ ← Data sent │ (encrypted with PSK-derived key) │ BEFORE │ │ handshake! │◀─── ServerHello ──────────────────────────────│ │◀─── {EncryptedExtensions} ────────────────────│ │◀─── {Finished} ──────────────────────────────│ │ │ │──── {Finished} ───────────────────────────────▶│ │ │ │◀═══ Application Data (1-RTT keys) ════════════▶│ ⚠️ 0-RTT data is NOT forward-secret and is REPLAYABLE
Pitfall: 0-RTT Replay Attacks

TLS 1.3's 0-RTT early data is vulnerable to replay attacks. An attacker who captures the ClientHello + early data can replay it to the server, potentially causing duplicate side effects (double purchases, duplicate API calls). The server has no cryptographic guarantee that 0-RTT data is fresh. Apple's URLSession does not enable 0-RTT by default. If you use Network.framework directly, you must explicitly opt in via sec_protocol_options_set_tls_early_data_enabled and ensure your server implements replay protection (e.g., single-use tickets, strike registers, or limiting 0-RTT to idempotent requests).

2.4 Detailed Comparison

Deep Dive: Why TLS 1.3 Removed Static RSA Key Exchange

In TLS 1.2 with static RSA, the client encrypts the pre-master secret with the server's RSA public key. If the server's private key is later compromised (or obtained via legal compulsion), an attacker who recorded past traffic can decrypt every session. TLS 1.3 mandates ephemeral Diffie-Hellman: each session generates a unique key pair that is discarded after the session ends. Even if the server's long-term key is compromised, past sessions remain confidential. This property — forward secrecy — is why TLS 1.3 is a strict security improvement.

3. Certificate Pinning Implementation

Certificate pinning restricts which certificates your app trusts for a given domain, beyond the system's default trust store. Instead of trusting any certificate signed by any of the ~150 root CAs in Apple's trust store, you pin to a specific certificate or public key. This defends against compromised CAs, rogue certificates, and man-in-the-middle attacks on corporate networks with TLS inspection proxies.

3.2 Complete Certificate Pinning Implementation

import Foundation
import Security
import CommonCrypto

final class CertificatePinningManager: NSObject, URLSessionDelegate {

    private let pinnedKeyHashes: [String: Set<String>]

    /// Initialize with a mapping of hostname → set of SHA-256 hashes
    /// of the Subject Public Key Info (SPKI).
    init(pinnedKeyHashes: [String: Set<String>]) {
        self.pinnedKeyHashes = pinnedKeyHashes
        super.init()
    }

    func urlSession(
        _ session: URLSession,
        didReceive challenge: URLAuthenticationChallenge,
        completionHandler: @escaping (URLSession.AuthChallengeDisposition, URLCredential?) -> Void
    ) {
        guard challenge.protectionSpace.authenticationMethod
                == NSURLAuthenticationMethodServerTrust,
              let serverTrust = challenge.protectionSpace.serverTrust,
              let expectedHashes = pinnedKeyHashes[challenge.protectionSpace.host]
        else {
            completionHandler(.cancelAuthenticationChallenge, nil)
            return
        }

        // Step 1: Standard X.509 trust evaluation
        var error: CFError?
        let trustValid = SecTrustEvaluateWithError(serverTrust, &error)

        guard trustValid else {
            completionHandler(.cancelAuthenticationChallenge, nil)
            return
        }

        // Step 2: Extract public keys from the certificate chain
        let certCount = SecTrustGetCertificateCount(serverTrust)
        var foundMatch = false

        for index in 0..<certCount {
            guard let certificate = SecTrustGetCertificateAtIndex(serverTrust, index),
                  let publicKey = SecCertificateCopyKey(certificate),
                  let publicKeyData = SecKeyCopyExternalRepresentation(publicKey, nil) as? Data
            else { continue }

            // Step 3: Hash the public key and compare against pins
            let keyHash = sha256(data: publicKeyData)
            let hashString = keyHash.base64EncodedString()

            if expectedHashes.contains(hashString) {
                foundMatch = true
                break
            }
        }

        if foundMatch {
            let credential = URLCredential(trust: serverTrust)
            completionHandler(.useCredential, credential)
        } else {
            completionHandler(.cancelAuthenticationChallenge, nil)
        }
    }

    private func sha256(data: Data) -> Data {
        var hash = [UInt8](repeating: 0, count: Int(CC_SHA256_DIGEST_LENGTH))
        data.withUnsafeBytes { buffer in
            _ = CC_SHA256(buffer.baseAddress, CC_LONG(data.count), &hash)
        }
        return Data(hash)
    }
}

// Usage
let pinManager = CertificatePinningManager(pinnedKeyHashes: [
    "api.example.com": [
        "BBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBB=",  // current key
        "CCCCCCCCCCCCCCCCCCCCCCCCCCCCCCCCCCCCCCCCCCC=",  // backup key
    ]
])

let session = URLSession(
    configuration: .default,
    delegate: pinManager,
    delegateQueue: nil
)
Pitfall: SecTrustGetCertificateAtIndex Deprecation

SecTrustGetCertificateAtIndex was deprecated in iOS 15. On iOS 15+, use SecTrustCopyCertificateChain to get the full chain as a CFArray of SecCertificate objects. The old API still works but will eventually be removed. Always check availability:

if #available(iOS 15.0, *) {
    guard let chain = SecTrustCopyCertificateChain(serverTrust) as? [SecCertificate]
    else { return }
    for cert in chain { /* pin check */ }
} else {
    for i in 0..<SecTrustGetCertificateCount(serverTrust) {
        guard let cert = SecTrustGetCertificateAtIndex(serverTrust, i)
        else { continue }
        /* pin check */
    }
}

3.3 Extracting Pin Hashes from a Live Server

// Terminal command to extract SPKI hash for pinning:
// $ openssl s_client -connect api.example.com:443 -servername api.example.com \
//     | openssl x509 -pubkey -noout \
//     | openssl pkey -pubin -outform DER \
//     | openssl dgst -sha256 -binary \
//     | base64

// Programmatic extraction for debugging:
func extractPinHash(from certificate: SecCertificate) -> String? {
    guard let publicKey = SecCertificateCopyKey(certificate),
          let keyData = SecKeyCopyExternalRepresentation(publicKey, nil) as? Data
    else { return nil }

    var hash = [UInt8](repeating: 0, count: Int(CC_SHA256_DIGEST_LENGTH))
    keyData.withUnsafeBytes { _ = CC_SHA256($0.baseAddress, CC_LONG(keyData.count), &hash) }
    return Data(hash).base64EncodedString()
}

4. Public Key Pinning vs Certificate Pinning

There are two fundamental approaches to pinning, and the choice has major operational implications for certificate rotation and incident response.

Best Practice: Always Pin the Public Key

Public key pinning is the industry standard for mobile apps. It decouples your pinning from the certificate lifecycle: you can renew certificates, switch CAs, and even switch between RSA and ECDSA — as long as you reuse the same key pair (or pre-pin the new key), the app continues to work. Certificate pinning is fragile and has caused major outages at companies that forgot to update pins before cert expiry.

4.3 HPKP: A Cautionary Tale

HTTP Public Key Pinning (HPKP) was a browser-based pinning mechanism defined in RFC 7469. The server sent a Public-Key-Pins header telling browsers to remember specific key hashes for a given duration. It was deprecated in 2018 and removed from Chrome 72 because:

  • Hostile pinning — an attacker who compromised a site could set HPKP headers pinning to their own keys, permanently locking out the legitimate owner
  • Self-denial-of-service — operators who lost their pinned keys (or forgot to rotate) bricked their own sites for the pin duration (up to 60 days)
  • Low adoption — the risk/reward ratio was too unfavorable; fewer than 0.1% of sites used it
  • Certificate Transparency provided a less dangerous alternative for detecting rogue certificates
Deep Dive: Why Mobile Pinning Is Different from HPKP

Mobile certificate pinning avoids HPKP's worst problems because the pins are embedded in the app binary, not received from the server. An attacker cannot inject hostile pins. The risk of self-denial-of-service remains, but it is mitigated by app update mechanisms and remote kill switches. This is why pinning is still recommended for mobile apps even though HPKP was abandoned for browsers.

5. SecTrust API Deep Dive

The SecTrust API is the foundation of all certificate evaluation on Apple platforms. It lives in the Security framework and provides a rich interface for building custom trust policies beyond what ATS offers.

5.2 Custom Trust Evaluation

func evaluateServerTrust(
    _ trust: SecTrust,
    forHost host: String
) -> Bool {
    // Create an SSL policy that validates the hostname
    let policy = SecPolicyCreateSSL(true, host as CFString)
    SecTrustSetPolicies(trust, policy)

    // Evaluate against the system trust store
    var error: CFError?
    let isValid = SecTrustEvaluateWithError(trust, &error)

    if !isValid {
        let nsError = error as? NSError
        print("Trust evaluation failed: \(nsError?.localizedDescription ?? "unknown")")
        print("Error code: \(nsError?.code ?? -1)")
    }

    return isValid
}

5.3 Custom Anchor Certificates (Self-Signed / Private CA)

func evaluateWithCustomCA(
    trust: SecTrust,
    customCACertificate: SecCertificate
) -> Bool {
    // Replace the system trust store with our custom CA
    let anchors = [customCACertificate] as CFArray
    SecTrustSetAnchorCertificates(trust, anchors)

    // IMPORTANT: By default, SetAnchorCertificates disables the
    // system trust store. To trust BOTH custom + system roots:
    SecTrustSetAnchorCertificatesOnly(trust, false)

    var error: CFError?
    return SecTrustEvaluateWithError(trust, &error)
}

// Loading a CA certificate from the app bundle
func loadCertificate(named name: String) -> SecCertificate? {
    guard let url = Bundle.main.url(forResource: name, withExtension: "der"),
          let data = try? Data(contentsOf: url),
          let certificate = SecCertificateCreateWithData(nil, data as CFData)
    else { return nil }
    return certificate
}

5.4 Inspecting the Evaluation Result

func inspectTrustResult(_ trust: SecTrust) {
    var error: CFError?
    let isValid = SecTrustEvaluateWithError(trust, &error)

    // Get detailed result dictionary
    if let resultDict = SecTrustCopyResult(trust) as? [String: Any] {
        print("Trust result: \(resultDict)")
        // Contains keys like:
        //   "TrustResultValue" → 4 (kSecTrustResultUnspecified = valid)
        //   "TrustRevocationChecked" → true/false
        //   "TrustExpirationDate" → date
        //   "TrustCertificateTransparency" → true/false
    }

    // Extract the leaf certificate's public key
    if let publicKey = SecTrustCopyPublicKey(trust) {
        let attributes = SecKeyCopyAttributes(publicKey) as? [String: Any]
        print("Key type: \(attributes?[kSecAttrKeyType as String] ?? "unknown")")
        print("Key size: \(attributes?[kSecAttrKeySizeInBits as String] ?? "unknown")")
    }

    // Get certificate chain (iOS 15+)
    if #available(iOS 15.0, *) {
        if let chain = SecTrustCopyCertificateChain(trust) as? [SecCertificate] {
            for (i, cert) in chain.enumerated() {
                let summary = SecCertificateCopySubjectSummary(cert) as? String
                print("  [\(i)] \(summary ?? "unknown")")
            }
        }
    }
}
Pitfall: SecTrustEvaluate (Deprecated) vs SecTrustEvaluateWithError

The old SecTrustEvaluate(_:_:) function (deprecated in iOS 15) returned a SecTrustResultType enum that was notoriously confusing. Both .unspecified and .proceed meant "trusted," while .recoverableTrustFailure meant "untrusted but you can override." Many developers incorrectly treated .unspecified as a failure. Use SecTrustEvaluateWithError (iOS 12+) which returns a simple Bool — true means trusted, false means not trusted, with a descriptive CFError.

6. Certificate Transparency

Certificate Transparency (CT) is a public auditing framework (RFC 6962) that makes it possible to detect misissued or rogue certificates. It was created by Google after the DigiNotar breach (2011) where a compromised CA issued fraudulent certificates for google.com and other high-profile domains.

6.3 CT on iOS

Apple has required Certificate Transparency for all publicly-trusted certificates since October 15, 2018. Any certificate issued after this date by a CA in Apple's trust store must have at least two SCTs from approved CT logs. Certificates without valid SCTs are treated as untrusted.

// Checking CT status in SecTrust evaluation
func checkCertificateTransparency(_ trust: SecTrust) {
    var error: CFError?
    let isValid = SecTrustEvaluateWithError(trust, &error)

    if let result = SecTrustCopyResult(trust) as? [String: Any] {
        let ctStatus = result["TrustCertificateTransparency"] as? Bool
        print("CT compliant: \(ctStatus ?? false)")
    }
}

// CT enforcement means:
// - Rogue CAs cannot secretly issue certs (they'd be logged)
// - Domain owners can monitor CT logs for unauthorized certs
// - Misissued certs are detectable within hours, not months
Deep Dive: CT and Pinning Are Complementary

Certificate Transparency and certificate pinning solve different problems. CT ensures that every certificate is publicly logged and auditable — it detects rogue certificates after issuance. Pinning ensures your app only trusts specific certificates — it prevents MITM even if a rogue cert is issued. For maximum security, use both: CT catches CA compromises at the ecosystem level, while pinning provides app-level defense-in-depth. Apple's CT requirement applies to the system trust evaluation; your pinning logic runs on top of it.

7. Declarative Pinning — NSPinnedDomains (iOS 14+)

iOS 14 introduced NSPinnedDomains, a declarative way to configure certificate pinning directly in Info.plist without writing any URLSessionDelegate code. This is Apple's recommended approach for new projects.

7.1 Info.plist Configuration

<!-- NSPinnedDomains in Info.plist -->
<key>NSAppTransportSecurity</key>
<dict>
    <key>NSPinnedDomains</key>
    <dict>
        <key>api.example.com</key>
        <dict>
            <!-- Include subdomains like staging.api.example.com -->
            <key>NSIncludesSubdomains</key>
            <true/>

Best Practice: Layered Pinning Strategy

For production apps, use both declarative and programmatic pinning. Set NSPinnedDomains as a baseline — it catches cases where developers forget to implement the delegate, and it protects WKWebView traffic automatically. Then add programmatic pinning in your URLSessionDelegate for custom error reporting, telemetry on pin failures, and the ability to dynamically update pins via remote configuration.

8. Architect: Certificate Pinning Strategy for Graceful Rotation

Architect Must-Know

At the Staff/L6 level, you must design a pinning strategy that balances security with operational resilience. A naive pinning implementation is a ticking time bomb — when the certificate rotates and the app hasn't been updated, every user is locked out. This section covers the full architecture for a production-grade pinning system.

8.1 The Multi-Pin Architecture

Certificate Pinning Rotation Architecture: ┌─────────────────────────────────────────────────────────────────────┐ │ App Binary (v3.2) │ │ │ │ Embedded Pins: │ │ ┌───────────────────────────────────────────────────────────────┐ │ │ │ Pin Set for api.example.com: │ │ │ │ │ │ │ │ 1. Current Key → SHA256/AAAA... (active server key) │ │ │ │ 2. Next Key → SHA256/BBBB... (pre-generated backup) │ │ │ │ 3. Backup Key → SHA256/CCCC... (offline cold storage) │ │ │ │ 4. (intermediate CA fallback) │ │ │ └───────────────────────────────────────────────────────────────┘ │ │ │ │ Remote Config Override: │ │ ┌───────────────────────────────────────────────────────────────┐ │ │ │ Fetch from: https://config.example.com/pins.json │ │ │ │ (This endpoint is NOT pinned — uses system trust only) │ │ │ │ │ │ │ │ Can: add new pins, disable pinning entirely │ │ │ │ Cannot: remove embedded pins (defense in depth) │ │ │ └───────────────────────────────────────────────────────────────┘ │ │ │ │ Telemetry: │ │ ┌───────────────────────────────────────────────────────────────┐ │ │ │ On pin failure: report to analytics (unpinned endpoint) │ │ │ │ Metrics: pin_match_index, server_cert_hash, failure_count │ │ │ └───────────────────────────────────────────────────────────────┘ │ └─────────────────────────────────────────────────────────────────────┘ Rotation Timeline: Phase 1: Normal Operation Server uses Key A ──▶ App matches Pin #1 (Current) ✓ Phase 2: Key Rotation Server switches to Key B ──▶ App matches Pin #2 (Next) ✓ Ship app update with: Pin #1 = Key B (now current) Pin #2 = Key D (new next) Pin #3 = Key C (unchanged backup) Phase 3: Emergency (Key Compromise) Server switches to Key C (cold backup) ──▶ App matches Pin #3 ✓ OR: Remote config disables pinning temporarily

8.2 Complete Production Implementation

import Foundation
import Security
import CommonCrypto

final class ProductionPinningManager: NSObject, URLSessionDelegate, Sendable {

    struct PinSet: Codable, Sendable {
        let hostname: String
        let spkiHashes: [String]
        let includesSubdomains: Bool
        let enforcementMode: EnforcementMode
        let expiresAt: Date?
    }

    enum EnforcementMode: String, Codable, Sendable {
        case enforce
        case reportOnly
        case disabled
    }

    struct PinFailureReport: Codable, Sendable {
        let hostname: String
        let serverKeyHash: String
        let expectedHashes: [String]
        let certChainSummary: [String]
        let timestamp: Date
        let appVersion: String
        let osVersion: String
    }

    private let embeddedPins: [PinSet]
    private let remotePinOverrides: RemotePinStore
    private let telemetry: PinningTelemetry

    init(
        embeddedPins: [PinSet],
        remotePinStore: RemotePinStore,
        telemetry: PinningTelemetry
    ) {
        self.embeddedPins = embeddedPins
        self.remotePinOverrides = remotePinStore
        self.telemetry = telemetry
        super.init()
    }

    func urlSession(
        _ session: URLSession,
        didReceive challenge: URLAuthenticationChallenge,
        completionHandler: @escaping (URLSession.AuthChallengeDisposition, URLCredential?) -> Void
    ) {
        let host = challenge.protectionSpace.host

        guard challenge.protectionSpace.authenticationMethod
                == NSURLAuthenticationMethodServerTrust,
              let serverTrust = challenge.protectionSpace.serverTrust
        else {
            completionHandler(.performDefaultHandling, nil)
            return
        }

        // Resolve effective pin set: remote overrides take precedence
        let effectivePinSet = resolveEffectivePins(for: host)

        guard let pinSet = effectivePinSet else {
            completionHandler(.performDefaultHandling, nil)
            return
        }

        // Check if pinning is disabled via remote config
        if pinSet.enforcementMode == .disabled {
            completionHandler(.performDefaultHandling, nil)
            return
        }

        // Check pin expiration
        if let expiresAt = pinSet.expiresAt, Date() > expiresAt {
            telemetry.reportExpiredPins(hostname: host)
            completionHandler(.performDefaultHandling, nil)
            return
        }

        // Step 1: Standard trust evaluation
        var error: CFError?
        guard SecTrustEvaluateWithError(serverTrust, &error) else {
            completionHandler(.cancelAuthenticationChallenge, nil)
            return
        }

        // Step 2: Extract and hash server public keys from the chain
        let serverKeyHashes = extractKeyHashes(from: serverTrust)

        // Step 3: Check if any server key matches any pin
        let matchingHash = serverKeyHashes.first { pinSet.spkiHashes.contains($0) }

        if let _ = matchingHash {
            let credential = URLCredential(trust: serverTrust)
            completionHandler(.useCredential, credential)
        } else {
            // Pin mismatch — report and potentially block
            let report = buildFailureReport(
                host: host,
                serverKeyHashes: serverKeyHashes,
                expectedHashes: pinSet.spkiHashes,
                trust: serverTrust
            )
            telemetry.reportPinFailure(report)

            switch pinSet.enforcementMode {
            case .enforce:
                completionHandler(.cancelAuthenticationChallenge, nil)
            case .reportOnly:
                let credential = URLCredential(trust: serverTrust)
                completionHandler(.useCredential, credential)
            case .disabled:
                completionHandler(.performDefaultHandling, nil)
            }
        }
    }

    private func resolveEffectivePins(for host: String) -> PinSet? {
        if let remotePins = remotePinOverrides.pins(for: host) {
            // Merge: remote can ADD pins or change mode, but
            // cannot REMOVE embedded pins (defense in depth)
            if let embedded = findEmbeddedPinSet(for: host) {
                let mergedHashes = Array(Set(embedded.spkiHashes + remotePins.spkiHashes))
                return PinSet(
                    hostname: host,
                    spkiHashes: mergedHashes,
                    includesSubdomains: embedded.includesSubdomains,
                    enforcementMode: remotePins.enforcementMode,
                    expiresAt: remotePins.expiresAt
                )
            }
            return remotePins
        }
        return findEmbeddedPinSet(for: host)
    }

    private func findEmbeddedPinSet(for host: String) -> PinSet? {
        embeddedPins.first { pinSet in
            if pinSet.hostname == host { return true }
            if pinSet.includesSubdomains && host.hasSuffix(".\(pinSet.hostname)") {
                return true
            }
            return false
        }
    }

    private func extractKeyHashes(from trust: SecTrust) -> [String] {
        var hashes: [String] = []

        if #available(iOS 15.0, *) {
            guard let chain = SecTrustCopyCertificateChain(trust) as? [SecCertificate]
            else { return [] }
            for cert in chain {
                if let hash = hashPublicKey(of: cert) {
                    hashes.append(hash)
                }
            }
        } else {
            for i in 0..<SecTrustGetCertificateCount(trust) {
                guard let cert = SecTrustGetCertificateAtIndex(trust, i)
                else { continue }
                if let hash = hashPublicKey(of: cert) {
                    hashes.append(hash)
                }
            }
        }
        return hashes
    }

    private func hashPublicKey(of certificate: SecCertificate) -> String? {
        guard let key = SecCertificateCopyKey(certificate),
              let keyData = SecKeyCopyExternalRepresentation(key, nil) as? Data
        else { return nil }

        var hash = [UInt8](repeating: 0, count: Int(CC_SHA256_DIGEST_LENGTH))
        keyData.withUnsafeBytes { buffer in
            _ = CC_SHA256(buffer.baseAddress, CC_LONG(keyData.count), &hash)
        }
        return Data(hash).base64EncodedString()
    }

    private func buildFailureReport(
        host: String,
        serverKeyHashes: [String],
        expectedHashes: [String],
        trust: SecTrust
    ) -> PinFailureReport {
        var chainSummary: [String] = []
        if #available(iOS 15.0, *) {
            if let chain = SecTrustCopyCertificateChain(trust) as? [SecCertificate] {
                chainSummary = chain.compactMap {
                    SecCertificateCopySubjectSummary($0) as? String
                }
            }
        }

        return PinFailureReport(
            hostname: host,
            serverKeyHash: serverKeyHashes.first ?? "unknown",
            expectedHashes: expectedHashes,
            certChainSummary: chainSummary,
            timestamp: Date(),
            appVersion: Bundle.main.infoDictionary?["CFBundleShortVersionString"] as? String ?? "?",
            osVersion: ProcessInfo.processInfo.operatingSystemVersionString
        )
    }
}

8.3 Remote Pin Configuration Service

final class RemotePinConfigService: RemotePinStore {

    private struct RemoteConfig: Codable {
        let version: Int
        let pins: [ProductionPinningManager.PinSet]
        let signature: String
    }

    private let configURL: URL
    private let signaturePublicKey: SecKey
    private var cachedPins: [String: ProductionPinningManager.PinSet] = [:]
    private let queue = DispatchQueue(label: "com.app.remotepins")

    init(configURL: URL, signaturePublicKey: SecKey) {
        self.configURL = configURL
        self.signaturePublicKey = signaturePublicKey
    }

    func pins(for hostname: String) -> ProductionPinningManager.PinSet? {
        queue.sync { cachedPins[hostname] }
    }

    func refresh() async {
        do {
            // Use a SEPARATE unpinned session for config fetches
            let (data, _) = try await URLSession.shared.data(from: configURL)
            let config = try JSONDecoder().decode(RemoteConfig.self, from: data)

            // Verify the config signature to prevent tampering
            guard verifySignature(config.signature, for: data) else {
                return
            }

            queue.async { [weak self] in
                self?.cachedPins = Dictionary(
                    uniqueKeysWithValues: config.pins.map { ($0.hostname, $0) }
                )
            }
        } catch {
            // Config fetch failure is non-fatal; embedded pins remain active
        }
    }

    private func verifySignature(_ signature: String, for data: Data) -> Bool {
        guard let sigData = Data(base64Encoded: signature) else { return false }
        let algorithm: SecKeyAlgorithm = .ecdsaSignatureMessageX962SHA256
        var error: Unmanaged<CFError>?
        let isValid = SecKeyVerifySignature(
            signaturePublicKey, algorithm,
            data as CFData, sigData as CFData, &error
        )
        return isValid
    }
}

8.5 Gradual Rollout Strategy

enum PinningRolloutPhase {
    case reportOnly
    case enforceForBeta
    case enforceForPercentage(Int)
    case enforceForAll
}

final class PinningRolloutController {

    private let phase: PinningRolloutPhase
    private let deviceID: String

    func shouldEnforce() -> Bool {
        switch phase {
        case .reportOnly:
            return false

        case .enforceForBeta:
            return Bundle.main.appStoreReceiptURL?.lastPathComponent
                == "sandboxReceipt"

        case .enforceForPercentage(let percent):
            let hash = abs(deviceID.hashValue) % 100
            return hash < percent

        case .enforceForAll:
            return true
        }
    }
}
Best Practice: Pinning Architecture Checklist
  • Pin the public key, not the certificate — survives cert renewal with same key pair
  • Always have at least 2 backup pins — one pre-generated backup key, one intermediate CA pin
  • Implement remote pin updates — signed configuration fetched over an unpinned (but TLS-validated) channel
  • Start in report-only mode — collect telemetry for 2+ weeks before enforcing
  • Gradual rollout — beta → 1% → 10% → 50% → 100%, monitoring failure rates at each stage
  • Emergency kill switch — remote config can disable pinning entirely within minutes
  • Monitor pin match index — alert if traffic shifts from Pin #1 to Pin #2 unexpectedly
  • Set pin expiration dates — pins that auto-expire prevent permanent lockout if remote config is unreachable
  • Never pin the root CA alone — root CAs sign millions of certificates; pinning them provides minimal security
  • Test with Charles/Proxyman — verify that pinning actually blocks proxy interception in debug builds

8.6 Interview Questions L6 Architect

Q: Your app uses certificate pinning and a certificate rotation is scheduled for next week. 30% of your users are on app versions older than 6 months. How do you handle this?

This is a classic Staff-level question that tests operational thinking. The answer involves multiple layers:

  1. If you pinned the public key and are reusing the same key pair for the new certificate, no action is needed — all app versions will continue to work.
  2. If the key is changing, check whether the new key's hash was embedded as a backup pin in older app versions. If yes, the rotation is safe.
  3. If older versions don't have the new pin, use the remote config channel to push the new pin hash to all connected clients before rotating the server certificate.
  4. For truly old versions that predate the remote config feature, you have two options: (a) keep the old key active on a parallel server endpoint during a transition period, or (b) accept that those users will be temporarily blocked and will need to update the app.
  5. Monitor the rollout: deploy the new certificate to a canary server first, watch pin failure telemetry for 24 hours, then roll out to production.

Q: An attacker has compromised your intermediate CA. How does your pinning architecture respond? L6

If you pinned the leaf public key, the CA compromise has no immediate impact — the attacker cannot forge a certificate with your pinned key. If you pinned the CA key, you are vulnerable: the attacker can issue a certificate for your domain that matches the CA pin. The correct architecture pins the leaf key as primary and the CA key only as a last-resort fallback. Additionally, Certificate Transparency will expose the rogue certificate in CT logs within hours, giving you time to respond.

Q: Why shouldn't the remote pin configuration endpoint itself be pinned? Senior

Circular dependency: if the config endpoint is pinned and the pin becomes invalid (because the server's certificate rotated), you cannot fetch the updated pins. The config endpoint must use standard system trust evaluation (ATS) without pinning. To prevent tampering, the config payload is cryptographically signed with a key embedded in the app binary — an attacker who intercepts the config cannot modify it without the signing key.
