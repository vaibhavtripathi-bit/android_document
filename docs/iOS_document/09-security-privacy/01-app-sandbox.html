1. iOS Sandbox Model

Every third-party app on iOS runs inside a sandbox — a kernel-enforced jail that restricts what system resources the process can access. Unlike advisory security mechanisms (which can be bypassed by a determined process), the iOS sandbox is mandatory: it is enforced by the XNU kernel itself, and no userspace code can opt out of it.

1.1 Kernel-Level Enforcement: TrustedBSD MAC Framework

The iOS sandbox is built on the TrustedBSD Mandatory Access Control (MAC) framework, which Apple integrated into XNU. This is the same framework used in FreeBSD and macOS. The key components are:

  • MAC Framework — a kernel subsystem that intercepts every system call (open, read, write, socket, mmap, exec, etc.) and checks it against a policy before allowing it to proceed
  • Sandbox.kext (kernel extension) — Apple's MAC policy module, historically called "seatbelt." It registers hooks with the MAC framework for every security-relevant operation
  • Sandbox profiles — declarative policy files (written in a Scheme-like DSL called SBPL — Sandbox Profile Language) that define what a process can and cannot do
  • libsandbox — userspace library that parses profiles and communicates them to the kernel via sandbox_init() or the MAC_SANDBOX policy
Deep Dive — Seatbelt Internals

The name "seatbelt" comes from the original macOS sandbox implementation (circa 10.5 Leopard). On iOS, the sandbox profile for third-party apps is called container (previously mobile). System daemons each have their own profile — securityd, mediaserverd, SpringBoard, etc. You can find macOS profiles at /System/Library/Sandbox/Profiles/; iOS profiles are compiled into the kernel cache.

1.2 How the Sandbox Is Applied at Process Creation

The sandbox is not something an app "enters" — it is imposed before the app's code ever runs. The sequence is:

  1. launchd (PID 1) receives a request to launch the app (from SpringBoard via XPC)
  2. posix_spawn() is called with the POSIX_SPAWN_SETEXEC attribute, which includes sandbox configuration
  3. The kernel's exec handler checks the binary's code signature and extracts entitlements
  4. Based on the entitlements (specifically com.apple.security.app-sandbox and the app's bundle ID), the kernel applies the container sandbox profile
  5. The MAC framework hooks are now active — every subsequent system call is filtered through the sandbox policy
  6. The app's main() function executes inside the sandbox — it has never had unrestricted access

1.3 What the Sandbox Restricts

The container sandbox profile for third-party apps enforces these restrictions:

Interview Pitfall

Many candidates say "the sandbox prevents apps from accessing other apps' data." This is true but incomplete. The sandbox also prevents fork(), direct hardware access, kernel interaction, and writable+executable memory mapping. The W^X (write XOR execute) policy is particularly important — it prevents JIT compilation exploits and is why only Apple's own JavaScriptCore has the dynamic-codesigning entitlement.

2. iOS Sandbox vs Android Isolation

Both platforms isolate apps, but the mechanisms are fundamentally different. Understanding this comparison is valuable for Staff-level interviews at companies like Google that work on both platforms.

Dimension iOS (Sandbox/Seatbelt) Android (UID-based + SELinux)
Isolation Mechanism TrustedBSD MAC + sandbox profiles Linux UID per app + SELinux MAC
File System Path-based rules in sandbox profile Unix DAC (file ownership by app UID) + SELinux labels
Process Isolation fork() denied by sandbox policy Zygote forks for each app; seccomp-bpf filters syscalls
IPC Mach ports (kernel-mediated) Binder (kernel driver)
Dynamic Code W^X enforced; no JIT except JSCore ART compiles to native; JIT allowed in app process
Permissions Entitlements (compile-time) + runtime prompts Manifest permissions + runtime prompts (API 23+)
Code Signing Every page verified at load time (AMFI) APK signature checked at install time only
Escape Difficulty Requires kernel exploit (very hard) Requires SELinux bypass + root (hard but more surface area)
Architect Must-Know

The key architectural difference: iOS verifies code signatures at page fault time via AMFI (Apple Mobile File Integrity). Every memory page loaded from disk is checked against the code signature. Android checks the APK signature only at install time. This means a modified iOS binary is caught the moment the tampered page is loaded, while a modified Android binary (if you bypass install verification) runs unchecked. This is why iOS jailbreaks must patch the kernel — there's no userspace-only path to running unsigned code.

3. Entitlements System

Entitlements are key-value pairs embedded in the app binary's code signature that declare what capabilities the app requires. They are the primary mechanism by which the sandbox is customized per-app. Think of them as a "capability passport" — the kernel checks this passport at every security-relevant operation.

3.1 How Entitlements Work

  1. Declaration — the developer declares entitlements in an .entitlements plist file in Xcode
  2. Provisioning Profile — Apple's signing infrastructure verifies the developer is allowed to use each entitlement and includes them in the provisioning profile
  3. Code Signing — during codesign, entitlements are embedded in the binary's code signature blob (specifically in the CodeDirectory as a DER-encoded entitlements blob)
  4. Kernel Extraction — at process creation, AMFI extracts entitlements from the code signature and attaches them to the process's task structure in the kernel
  5. Runtime Checks — when the sandbox or any kernel subsystem needs to check a capability, it queries the process's entitlements via IOTaskHasEntitlement() or the MAC framework

3.3 Reading Entitlements at Runtime

While you cannot modify entitlements at runtime, you can inspect them. This is useful for debugging and for verifying your app's configuration:

import Foundation
import Security

// Read the current process's entitlements using Security framework
func readEntitlements() -> [String: Any]? {
    var code: SecStaticCode?
    let status = SecCodeCopySelf([], &code)
    guard status == errSecSuccess, let code = code else {
        return nil
    }

    var info: CFDictionary?
    let infoStatus = SecCodeCopySigningInformation(
        code,
        SecCSFlags(rawValue: kSecCSSigningInformation),
        &info
    )
    guard infoStatus == errSecSuccess,
          let dict = info as? [String: Any],
          let entitlements = dict[kSecCodeInfoEntitlementsDict as String]
              as? [String: Any] else {
        return nil
    }
    return entitlements
}

// Check a specific entitlement
func hasAppGroupEntitlement() -> Bool {
    guard let entitlements = readEntitlements(),
          let groups = entitlements["com.apple.security.application-groups"]
              as? [String] else {
        return false
    }
    return !groups.isEmpty
}

From the command line, you can inspect an app's entitlements with:

# Extract entitlements from a signed binary
codesign -d --entitlements :- /path/to/YourApp.app

# From an IPA (extract first, then inspect)
unzip YourApp.ipa -d /tmp/app
codesign -d --entitlements :- /tmp/app/Payload/YourApp.app

# Using ldid on a jailbroken device
ldid -e /var/containers/Bundle/Application/.../YourApp.app/YourApp
Deep Dive — Entitlement Encoding

Entitlements are stored in two formats within the code signature: a plist-encoded XML blob (for legacy compatibility) and a DER-encoded blob (used by the kernel since iOS 15 for faster parsing). The DER format is critical for performance — the kernel parses entitlements on every exec(), and DER parsing is O(1) for key lookup vs O(n) for plist XML. This is why Apple introduced the com.apple.developer.entitlements-validation entitlement to enforce DER-only validation.

4. Container Structure

Every iOS app gets its own container — an isolated directory tree that the sandbox restricts the app to. The container is created by installd when the app is first installed and has a UUID-based path that changes on every installation.

4.1 Container Layout

4.2 Bundle Container (Read-Only)

The bundle container holds the signed app binary and all resources. It is read-only at runtime — the sandbox denies all write operations to this directory. This is critical for code integrity: since every page is signature-verified at load time (AMFI), any modification would invalidate the signature and crash the app.

  • The UUID path changes on every app update (the old bundle is replaced atomically)
  • The data container UUID remains stable across updates
  • Bundle.main.bundlePath returns the path to this container

4.3 Data Container (Read-Write)

The data container is the app's writable storage. It has a different UUID from the bundle container (this is a common interview question). The key directories:

Directory Purpose Backed Up Purged by System
Documents/ User-generated content, files visible in Files app (if UIFileSharingEnabled) Yes No
Library/Application Support/ App-specific data not visible to user Yes No
Library/Caches/ Regenerable cached data No Yes (low disk space)
Library/Preferences/ UserDefaults plist files Yes No
tmp/ Temporary files No Yes (app not running)

4.4 Group Containers (Shared)

App Groups allow multiple apps and extensions from the same development team to share a container. This is the only sanctioned way for apps to share files on iOS (besides the pasteboard and document providers).

import Foundation

// Accessing container directories
let fm = FileManager.default

// Bundle container (read-only)
let bundlePath = Bundle.main.bundlePath
let infoPlist = Bundle.main.url(forResource: "Info", withExtension: "plist")

// Data container — Documents
let documentsURL = fm.urls(for: .documentDirectory, in: .userDomainMask).first!

// Data container — Library/Application Support
let appSupportURL = fm.urls(
    for: .applicationSupportDirectory,
    in: .userDomainMask
).first!

// Data container — Library/Caches
let cachesURL = fm.urls(for: .cachesDirectory, in: .userDomainMask).first!

// Data container — tmp
let tmpURL = fm.temporaryDirectory

// Group container (requires App Groups entitlement)
if let groupURL = fm.containerURL(
    forSecurityApplicationGroupIdentifier: "group.com.yourteam.shared"
) {
    let sharedDB = groupURL.appendingPathComponent("shared.sqlite")
    print("Shared DB at: \(sharedDB.path)")
}

// Shared UserDefaults via App Groups
let sharedDefaults = UserDefaults(suiteName: "group.com.yourteam.shared")
sharedDefaults?.set("value", forKey: "sharedKey")
Common Pitfall — Container UUID Confusion

The bundle container and data container have different UUIDs. Never hardcode or cache absolute paths — they change on every install/update (bundle) or device restore (data). Always use FileManager APIs to resolve paths at runtime. Also note: on the iOS Simulator, the container structure is different from a real device (paths are under ~/Library/Developer/CoreSimulator/).

5. File Protection Classes

iOS provides four Data Protection classes that control when files can be read and written based on the device's lock state. Each class has a corresponding encryption key that is derived from the user's passcode and a hardware key in the Secure Enclave. When the key is unavailable (because the device is locked), the file is cryptographically inaccessible.

5.1 The Four Protection Classes

NSFileProtectionComplete

The most secure class. The file's class key is evicted from memory 10 seconds after the device locks. The file is completely inaccessible until the user unlocks the device again.

  • Key available: Only while device is unlocked (or within ~10s grace period after lock)
  • Use for: Highly sensitive data — health records, financial data, authentication tokens
  • Limitation: Background operations cannot read/write these files

NSFileProtectionCompleteUnlessOpen

Uses an asymmetric key scheme. A file can be created while locked (using the public key), but can only be opened for reading while unlocked. If a file handle is already open when the device locks, it remains accessible.

  • Key available: Public key always available (for writes); private key only while unlocked
  • Use for: Files that need background writes but sensitive reads — email attachments downloading in background, photo uploads
  • Mechanism: Per-file key wrapped with ECDH — public key encrypts, private key (protected) decrypts

NSFileProtectionCompleteUntilFirstUserAuthentication

The default protection class for most app files. The class key is loaded into memory at first unlock after boot and remains available until the device is rebooted. This is sometimes called "After First Unlock" (AFU).

  • Key available: From first unlock until reboot
  • Use for: Most app data — databases, cached content, preferences
  • Why it's the default: Balances security with usability — background operations work, but data is protected at rest before first unlock (e.g., if device is seized while powered off)

NSFileProtectionNone

The file is encrypted with a key that is always available, even before first unlock. The key is derived from the hardware UID only (no passcode component). This provides protection against physical extraction of the flash storage but not against a running device.

  • Key available: Always (even before first unlock)
  • Use for: Data that must be accessible immediately at boot — device management profiles, certain system databases
  • Warning: Avoid for any sensitive data

5.2 Protection Class Comparison

CompleteUnlessOpen No Write-only (new files) Yes Write-only
CompleteUntilFirstAuth (default) No Yes Yes Yes

5.3 Setting File Protection in Swift

import Foundation

// Method 1: Set protection when creating a file
let sensitiveData = "secret-token-12345".data(using: .utf8)!
let fileURL = FileManager.default
    .urls(for: .documentDirectory, in: .userDomainMask)[0]
    .appendingPathComponent("secrets.dat")

try sensitiveData.write(
    to: fileURL,
    options: [.completeFileProtection]  // NSFileProtectionComplete
)

// Method 2: Set protection on an existing file
try FileManager.default.setAttributes(
    [.protectionKey: FileProtectionType.complete],
    ofItemAtPath: fileURL.path
)

// Method 3: Set default protection for a directory
let secureDir = FileManager.default
    .urls(for: .documentDirectory, in: .userDomainMask)[0]
    .appendingPathComponent("secure")

try FileManager.default.createDirectory(
    at: secureDir,
    withIntermediateDirectories: true,
    attributes: [.protectionKey: FileProtectionType.complete]
)

// Method 4: Verify protection class of a file
let attrs = try FileManager.default.attributesOfItem(atPath: fileURL.path)
if let protection = attrs[.protectionKey] as? FileProtectionType {
    switch protection {
    case .complete:
        print("NSFileProtectionComplete")
    case .completeUnlessOpen:
        print("NSFileProtectionCompleteUnlessOpen")
    case .completeUntilFirstUserAuthentication:
        print("NSFileProtectionCompleteUntilFirstUserAuthentication")
    case .none:
        print("NSFileProtectionNone")
    default:
        print("Unknown protection: \(protection)")
    }
}

// Method 5: Handle protection errors in background
func readSecureFile(at url: URL) -> Data? {
    do {
        return try Data(contentsOf: url)
    } catch let error as NSError
        where error.domain == NSCocoaErrorDomain
           && error.code == NSFileReadNoPermissionError {
        // File is protected and device is locked
        print("File inaccessible — device locked, protection class active")
        return nil
    } catch {
        print("Read error: \(error)")
        return nil
    }
}
Best Practice — Protection Class Selection

Use NSFileProtectionComplete for authentication tokens, encryption keys, health data, and financial records. Use CompleteUnlessOpen for files that need background download completion. Use the default CompleteUntilFirstUserAuthentication for general app data. Never use None unless you have a specific technical reason (e.g., boot-time configuration). Also set the NSFileProtectionComplete capability in your app's entitlements to change the default protection class for all new files.

6.2 How File Encryption Works

  1. File creation: A new random 256-bit AES per-file key is generated
  2. Key wrapping: The per-file key is wrapped (encrypted) with the appropriate class key based on the file's protection class
  3. Metadata storage: The wrapped per-file key is stored in the file's metadata (in the APFS filesystem metadata, not in the file content)
  4. Content encryption: The file content is encrypted with AES-XTS using the per-file key
  5. On read: The class key unwraps the per-file key, which decrypts the file content
  6. On lock: Class A keys are evicted from memory — files in that class become cryptographically inaccessible

6.3 Secure Enclave Integration

The Secure Enclave Processor (SEP) is a hardware coprocessor with its own encrypted memory, secure boot chain, and hardware random number generator. Its role in Data Protection:

  • UID key storage: The 256-bit UID key is fused into the SEP at manufacturing and can never be read — only used for cryptographic operations inside the SEP
  • Passcode verification: The SEP enforces escalating delays between passcode attempts (1s, 5s, 15s, 1min, 1hr) and can be configured to wipe after 10 failed attempts
  • Key derivation: The passcode-to-key derivation happens inside the SEP, entangling the passcode with the UID key — this means brute-forcing the passcode requires the physical device
  • Effaceable Storage: The SEP manages a region of flash storage that can be instantly wiped, destroying all class keys and making all data irrecoverable
Architect Must-Know — Why This Matters

The key insight for architects: because the UID key never leaves the Secure Enclave and is entangled with the passcode, an attacker who extracts the flash storage cannot brute-force the passcode offline. They must use the physical device, which enforces rate limiting. This is fundamentally different from full-disk encryption on a laptop, where the encryption key can be brute-forced on any hardware. This is why iOS devices are considered secure even with a 6-digit passcode — the SEP makes each guess take 80ms minimum, making brute-force impractical.

7. Jailbreak Detection Techniques

Jailbreak detection attempts to determine whether the device has been compromised — whether the sandbox and code signing enforcement have been bypassed. While these checks are common in banking and enterprise apps, understanding their limitations is more important than knowing the techniques themselves.

7.1 Common Detection Techniques

import Foundation
import UIKit
import Darwin

struct JailbreakDetector {

    // Technique 1: Check for known jailbreak file paths
    static func checkJailbreakFiles() -> Bool {
        let suspiciousPaths = [
            "/Applications/Cydia.app",
            "/Applications/Sileo.app",
            "/Library/MobileSubstrate/MobileSubstrate.dylib",
            "/usr/sbin/sshd",
            "/usr/bin/ssh",
            "/etc/apt",
            "/private/var/lib/apt/",
            "/usr/lib/TweakInject",
            "/var/jb",                    // rootless jailbreaks (iOS 15+)
            "/var/binpack",
            "/Library/PreferenceBundles",
        ]
        for path in suspiciousPaths {
            if FileManager.default.fileExists(atPath: path) {
                return true
            }
        }
        return false
    }

    // Technique 2: Check if we can write outside our sandbox
    static func checkSandboxViolation() -> Bool {
        let testPath = "/private/jailbreak_test_\(UUID().uuidString)"
        do {
            try "test".write(toFile: testPath, atomically: true, encoding: .utf8)
            try FileManager.default.removeItem(atPath: testPath)
            return true  // Should not be possible in sandbox
        } catch {
            return false
        }
    }

    // Technique 3: Check for fork() ability
    static func checkForkAbility() -> Bool {
        let pid = fork()
        if pid >= 0 {
            if pid > 0 {
                // Parent: kill the child
                kill(pid, SIGTERM)
                var status: Int32 = 0
                waitpid(pid, &status, 0)
            }
            return true  // fork() succeeded — sandbox is broken
        }
        return false
    }

    // Technique 4: Check for injected dylibs
    static func checkDyldInjection() -> Bool {
        let suspiciousLibs = [
            "SubstrateLoader",
            "MobileSubstrate",
            "TweakInject",
            "libhooker",
            "substitute",
            "ellekit",       // modern jailbreak hooking framework
        ]
        let imageCount = _dyld_image_count()
        for i in 0..<imageCount {
            guard let name = _dyld_get_image_name(i) else { continue }
            let imageName = String(cString: name)
            for lib in suspiciousLibs {
                if imageName.contains(lib) {
                    return true
                }
            }
        }
        return false
    }

    // Technique 5: Check for symbolic links
    static func checkSymbolicLinks() -> Bool {
        let pathsToCheck = [
            "/Applications",
            "/var/stash",
            "/Library/Ringtones",
            "/Library/Wallpaper",
        ]
        for path in pathsToCheck {
            var statInfo = stat()
            if lstat(path, &statInfo) == 0 {
                if (statInfo.st_mode & S_IFLNK) == S_IFLNK {
                    return true
                }
            }
        }
        return false
    }

    // Technique 6: Check URL scheme for Cydia/Sileo
    static func checkJailbreakURLSchemes() -> Bool {
        let schemes = [
            "cydia://package/com.example",
            "sileo://package/com.example",
            "zbra://packages",
        ]
        for scheme in schemes {
            if let url = URL(string: scheme),
               UIApplication.shared.canOpenURL(url) {
                return true
            }
        }
        return false
    }

    // Technique 7: Check environment variables for injection
    static func checkEnvironmentVariables() -> Bool {
        let suspiciousVars = [
            "DYLD_INSERT_LIBRARIES",
            "_MSSafeMode",
            "DYLD_LIBRARY_PATH",
        ]
        for envVar in suspiciousVars {
            if getenv(envVar) != nil {
                return true
            }
        }
        return false
    }

    // Combined check
    static func isJailbroken() -> Bool {
        return checkJailbreakFiles()
            || checkSandboxViolation()
            || checkForkAbility()
            || checkDyldInjection()
            || checkSymbolicLinks()
            || checkJailbreakURLSchemes()
            || checkEnvironmentVariables()
    }
}

8. Why Jailbreak Detection Is Unreliable

This is one of the most important sections for Staff-level interviews. The question isn't "how do you detect jailbreaks?" — it's "why is jailbreak detection fundamentally a losing game?"

8.1 The Cat-and-Mouse Game

Every jailbreak detection technique listed above has a known bypass. The jailbreak community maintains tools specifically designed to defeat these checks:

Detection Technique Bypass Method Tools
File path checks Hook stat(), access(), fileExistsAtPath: to return "not found" for known paths Shadow, A-Bypass, Hestia
Sandbox write test Hook open() with O_CREAT to deny writes to test paths Liberty Lite, Choicy
fork() check Hook fork() to return -1 (EPERM) for the target app KernBypass, vnodebypass
dyld image check Hook _dyld_image_count() and _dyld_get_image_name() to hide injected libraries Shadow, FlyJB
URL scheme check Hook canOpenURL: to return NO for jailbreak URLs Shadow, A-Bypass
Environment variables Hook getenv() to return NULL for suspicious variables Most bypass tweaks
Symbolic link check Hook lstat() to report regular files instead of symlinks Shadow, vnodebypass

8.2 The Fundamental Problem

Architect Must-Know — The Halting Problem of Integrity

Jailbreak detection is fundamentally impossible to do reliably because of a simple principle: you cannot verify the integrity of a system from within that system if the system is compromised. This is analogous to the halting problem in computer science.

When your app calls stat("/Applications/Cydia.app"), that call goes through the kernel. On a jailbroken device, the kernel itself may be patched — your system call never reaches the real filesystem. You're asking a potentially compromised system "are you compromised?" and trusting the answer.

Even "advanced" techniques like checking code signatures, verifying kernel integrity, or using syscall timing analysis can be defeated because the jailbreak operates at a privilege level above your app. The attacker controls the kernel; you're in userspace.

8.3 Why Modern Jailbreaks Are Harder to Detect

  • Rootless jailbreaks (iOS 15+): Tools like Dopamine and palera1n don't modify the root filesystem at all. They use /var/jb as a prefix, making traditional path checks useless
  • Kernel-level hooking: Modern jailbreaks patch the kernel's MAC framework hooks directly, so sandbox checks return "clean" results even though the sandbox is compromised
  • Per-app bypass: Tools like Choicy and Shadow can selectively disable jailbreak tweaks for specific apps, so the app sees a "clean" environment while the rest of the system is jailbroken
  • Kernel memory protection: Some jailbreaks use KPP/KTRR bypass techniques that make kernel patches invisible to integrity checks

8.4 What To Do Instead

Rather than trying to detect jailbreaks (which is unreliable), focus on protecting your data assuming the device is compromised:

  • Use Apple's App Attest (DCAppAttestService) — hardware-backed attestation that the app binary is genuine and running on a non-compromised device. The attestation is verified server-side, not on-device
  • Use DeviceCheck — server-side per-device bits that persist across app installs
  • Implement server-side validation — never trust client-side security decisions
  • Use Secure Enclave keys — even a jailbroken kernel cannot extract keys from the SEP
  • Encrypt sensitive data with keys derived from server-provided secrets, not just local storage
import DeviceCheck

// App Attest — hardware-backed integrity verification
func attestApp() async throws {
    let service = DCAppAttestService.shared

    guard service.isSupported else {
        throw AttestError.unsupported
    }

    // Step 1: Generate a key pair in the Secure Enclave
    let keyId = try await service.generateKey()

    // Step 2: Get a server challenge
    let challenge = try await fetchChallengeFromServer()
    let challengeHash = Data(SHA256.hash(data: challenge))

    // Step 3: Attest the key (sends to Apple for verification)
    let attestation = try await service.attestKey(
        keyId,
        clientDataHash: challengeHash
    )

    // Step 4: Send attestation to YOUR server for verification
    // Server verifies with Apple's attestation root CA
    try await sendAttestationToServer(
        keyId: keyId,
        attestation: attestation
    )
}

// For subsequent requests, generate assertions
func generateAssertion(keyId: String, request: Data) async throws -> Data {
    let requestHash = Data(SHA256.hash(data: request))
    return try await DCAppAttestService.shared.generateAssertion(
        keyId,
        clientDataHash: requestHash
    )
}
Best Practice — Server-Side Attestation

App Attest moves the trust decision to the server. The Secure Enclave generates a key pair, Apple's attestation service verifies the device and app integrity, and your server validates the attestation certificate chain. A jailbroken device cannot forge a valid attestation because it would need to compromise the Secure Enclave hardware — which is a fundamentally different (and much harder) attack than patching the kernel.

9. Sandbox Escape History

Understanding historical sandbox escapes is important for two reasons: it demonstrates the real-world effectiveness of defense in depth, and it shows what attack surfaces exist. These are not theoretical — they were used in the wild.

9.1 Notable Sandbox Escapes

CVE / Name Year Attack Vector What Was Exploited Apple's Fix
CVE-2016-4655/4656/4657 (Trident / Pegasus) 2016 Safari WebKit → kernel WebKit memory corruption → kernel info leak → kernel code execution. Used by NSO Group's Pegasus spyware Patched WebKit, added KPP (Kernel Patch Protection)

9.2 Common Attack Surfaces

Sandbox escapes typically exploit one of these attack surfaces:

  • IOKit user clients: GPU drivers, USB, audio — complex C/C++ code with large attack surface accessible from the sandbox
  • Mach port IPC: Messages to system services (securityd, mediaserverd) can trigger parsing bugs in the receiving daemon
  • XPC services: Deserialization of XPC messages can trigger type confusion or buffer overflows
  • Kernel sockets: The networking stack is complex and has historically had use-after-free and race condition bugs
  • WebKit: The JavaScript engine (JSC) has JIT compilation, which is inherently a large attack surface (writable+executable memory)
Interview Pitfall — "iOS Is Secure"

Never say "iOS is secure" in an interview without qualification. iOS has a strong security architecture, but sandbox escapes happen regularly (Apple patches them in every iOS update). The correct framing is: "iOS provides defense in depth — even when one layer is breached, other layers (Data Protection, Secure Enclave, code signing) continue to protect user data." This demonstrates architectural thinking rather than blind trust in a platform.

9.3 Why This Matters for App Developers

You cannot control whether the OS has a zero-day vulnerability. But you can design your app so that a sandbox escape doesn't automatically compromise user data:

  • Encrypt sensitive data with Secure Enclave keys (survives kernel compromise)
  • Don't store secrets in plaintext files (even within the sandbox)
  • Use NSFileProtectionComplete so data is inaccessible when the device is locked
  • Implement server-side validation for security-critical operations
  • Use App Attest to detect compromised devices server-side

10.2 Practical Implementation

Here's how to implement defense in depth for a banking app — a common Staff-level interview scenario:

import Foundation
import CryptoKit
import Security
import LocalAuthentication

// Defense in Depth: Secure Data Manager
final class SecureDataManager {

    // LAYER 5: Application-level encryption using Secure Enclave key
    private let encryptionKeyTag = "com.bank.app.encryption.key"

    func getOrCreateSecureEnclaveKey() throws -> SecKey {
        let tag = encryptionKeyTag.data(using: .utf8)!

        // Try to load existing key
        let query: [String: Any] = [
            kSecClass as String: kSecClassKey,
            kSecAttrApplicationTag as String: tag,
            kSecAttrKeyType as String: kSecAttrKeyTypeECSECPrimeRandom,
            kSecReturnRef as String: true,
        ]
        var item: CFTypeRef?
        if SecItemCopyMatching(query as CFDictionary, &item) == errSecSuccess {
            return item as! SecKey
        }

        // Create new key in Secure Enclave with biometric protection
        var error: Unmanaged<CFError>?
        guard let accessControl = SecAccessControlCreateWithFlags(
            kCFAllocatorDefault,
            kSecAttrAccessibleWhenUnlockedThisDeviceOnly,
            [.privateKeyUsage, .biometryCurrentSet],
            &error
        ) else {
            throw error!.takeRetainedValue() as Error
        }

        let attributes: [String: Any] = [
            kSecAttrKeyType as String: kSecAttrKeyTypeECSECPrimeRandom,
            kSecAttrKeySizeInBits as String: 256,
            kSecAttrTokenID as String: kSecAttrTokenIDSecureEnclave,
            kSecPrivateKeyAttrs as String: [
                kSecAttrIsPermanent as String: true,
                kSecAttrApplicationTag as String: tag,
                kSecAttrAccessControl as String: accessControl,
            ],
        ]

        guard let privateKey = SecKeyCreateRandomKey(
            attributes as CFDictionary, &error
        ) else {
            throw error!.takeRetainedValue() as Error
        }
        return privateKey
    }

    // LAYER 4: Write encrypted data with file protection
    func secureWrite(_ data: Data, to filename: String) throws {
        // Encrypt with app-level key (Layer 5)
        let symmetricKey = SymmetricKey(size: .bits256)
        let sealed = try AES.GCM.seal(data, using: symmetricKey)
        guard let encrypted = sealed.combined else {
            throw SecureError.encryptionFailed
        }

        // Write with file-level protection (Layer 4)
        let url = secureDirectory().appendingPathComponent(filename)
        try encrypted.write(to: url, options: [.completeFileProtection])

        // Store symmetric key in Keychain with biometric protection
        try storeKeyInKeychain(symmetricKey, forFile: filename)
    }

    private func secureDirectory() -> URL {
        let dir = FileManager.default
            .urls(for: .applicationSupportDirectory, in: .userDomainMask)[0]
            .appendingPathComponent("secure_vault")
        try? FileManager.default.createDirectory(
            at: dir,
            withIntermediateDirectories: true,
            attributes: [.protectionKey: FileProtectionType.complete]
        )
        return dir
    }

    private func storeKeyInKeychain(
        _ key: SymmetricKey,
        forFile filename: String
    ) throws {
        let keyData = key.withUnsafeBytes { Data($0) }
        let query: [String: Any] = [
            kSecClass as String: kSecClassGenericPassword,
            kSecAttrService as String: "com.bank.app.filekeys",
            kSecAttrAccount as String: filename,
            kSecValueData as String: keyData,
            kSecAttrAccessible as String:
                kSecAttrAccessibleWhenUnlockedThisDeviceOnly,
        ]
        let status = SecItemAdd(query as CFDictionary, nil)
        guard status == errSecSuccess || status == errSecDuplicateItem else {
            throw SecureError.keychainWriteFailed(status)
        }
    }
}

enum SecureError: Error {
    case encryptionFailed
    case keychainWriteFailed(OSStatus)
    case attestationFailed
}

Architect Must-Know — The Golden Rule

The golden rule of iOS security architecture: never rely on a single layer. The sandbox might be escaped (CVEs happen every year). File protection might be bypassed (device is unlocked when stolen). The Keychain might be dumped (on a jailbroken device with the right tools). But if you layer all of these together — sandbox + file protection + Keychain + Secure Enclave + server-side validation — an attacker must breach every layer simultaneously. That's the power of defense in depth.

In an interview, when asked "how would you secure sensitive data?", walk through each layer and explain what threat it defends against. This demonstrates architectural thinking, not just API knowledge.

10.4 Don't Trust Local State

A critical principle that many apps violate: never make security decisions based on local state alone.

// BAD: Security decision based on local state
func checkPremiumAccess() -> Bool {
    return UserDefaults.standard.bool(forKey: "isPremium")
    // Trivially bypassed: modify the plist on a jailbroken device
}

// GOOD: Server-validated with signed token
func checkPremiumAccess() async throws -> Bool {
    let token = try getAuthToken()
    let response = try await apiClient.verifyEntitlement(
        token: token,
        entitlement: "premium"
    )
    // Server checks subscription status, returns signed response
    return response.isValid && verifyServerSignature(response)
}

// BAD: Jailbreak check gates functionality
func performTransaction() {
    if JailbreakDetector.isJailbroken() {
        showError("Device compromised")
        return
    }
    // Bypassed by any jailbreak hiding tool
    processPayment()
}

// GOOD: Server validates device integrity via App Attest
func performTransaction(_ transaction: Transaction) async throws {
    let requestData = try JSONEncoder().encode(transaction)
    let assertion = try await generateAssertion(
        keyId: attestKeyId,
        request: requestData
    )
    // Server verifies assertion, checks device integrity,
    // validates transaction server-side
    try await apiClient.submitTransaction(
        transaction,
        assertion: assertion
    )
}

11. Interview Questions

Q1: Explain how the iOS sandbox is enforced at the kernel level. What prevents an app from bypassing it? L6

The iOS sandbox is enforced by the TrustedBSD MAC (Mandatory Access Control) framework integrated into XNU. The Sandbox.kext kernel extension registers hooks for every security-relevant system call — open, read, write, socket, mmap, fork, exec, iokit, etc. When a sandboxed process makes any system call, the MAC framework invokes the sandbox policy check before the operation proceeds. The policy is defined by a sandbox profile (written in SBPL — Sandbox Profile Language) that specifies allow/deny rules based on the operation type, file paths, and the process's entitlements.

An app cannot bypass the sandbox because: (1) the sandbox is applied by the kernel at process creation via posix_spawn() — the app's code never runs without it; (2) the MAC framework hooks are in the kernel, not in userspace libraries that could be swapped out; (3) code signing enforcement (AMFI) prevents loading modified binaries; (4) the W^X policy prevents generating new executable code at runtime. To bypass the sandbox, you would need a kernel exploit — which is why sandbox escapes are high-severity vulnerabilities.

Q2: What is the difference between the bundle container and the data container? Why do they have different UUIDs? Senior

The bundle container holds the signed app binary and resources (read-only). The data container holds the app's writable data (Documents, Library, tmp). They have different UUIDs for several reasons: (1) the bundle container is replaced atomically on every app update (new UUID), while the data container persists across updates (stable UUID) — this ensures user data survives updates; (2) separating them allows the system to verify code integrity independently of data; (3) it enables the system to purge the bundle container (e.g., app offloading) while preserving user data.

Q3: Explain all four Data Protection classes and when you would use each one. Senior

NSFileProtectionComplete: Key evicted from memory ~10s after lock. Use for highly sensitive data (auth tokens, health records, financial data). Cannot be accessed in background. NSFileProtectionCompleteUnlessOpen: Uses asymmetric crypto — can create new files while locked (public key), but can only open existing files while unlocked (private key). Open file handles remain valid. Use for background downloads of sensitive content. NSFileProtectionCompleteUntilFirstUserAuthentication: (Default) Key loaded at first unlock, stays in memory until reboot. Use for general app data — balances security with background access. NSFileProtectionNone: Key always available, derived from hardware UID only. Use only when data must be accessible before first unlock (rare). The key architectural insight is that each class represents a different tradeoff between security and availability, and you should choose based on the sensitivity of the data and whether background access is needed.

Q4: Why is jailbreak detection fundamentally unreliable? What should you do instead? L6

Jailbreak detection is unreliable because it attempts to verify system integrity from within a potentially compromised system. Every detection technique (file checks, fork tests, dylib enumeration, URL scheme checks) relies on system calls that go through the kernel — and on a jailbroken device, the kernel is compromised. Jailbreak bypass tools (Shadow, Liberty Lite, Choicy) hook these system calls to return "clean" results. It's a cat-and-mouse game that the defender always loses because the attacker operates at a higher privilege level.

Instead, use: (1) App Attest — hardware-backed attestation verified server-side, not on-device; (2) Server-side validation — never trust client-side security decisions; (3) Secure Enclave keys — even a compromised kernel can't extract SEP keys; (4) Defense in depth — encrypt data with multiple layers so a single compromise doesn't expose everything.

Q5: How does the Data Protection key hierarchy work? Why can't an attacker brute-force the passcode offline? L6

The key hierarchy has four levels: (1) Hardware UID key — 256-bit AES key fused into the Secure Enclave at manufacturing, never readable by software; (2) Passcode key — derived from the user's passcode entangled with the UID key via PBKDF2 inside the Secure Enclave; (3) Class keys — one per protection class, wrapped by the passcode key; (4) Per-file keys — unique 256-bit AES key per file, wrapped by the appropriate class key.

Offline brute-force is impossible because the passcode derivation happens inside the Secure Enclave and requires the hardware UID key. An attacker who extracts the flash storage has encrypted data + wrapped keys, but cannot derive the passcode key without the UID key, which is physically fused into the SEP chip. They must use the physical device, which enforces escalating delays (1s → 5s → 15s → 1min → 1hr) and optional wipe after 10 attempts. Even a 6-digit passcode (1M combinations) would take ~22 hours at minimum with the rate limiting.

Q6: How would you architect the security for a banking app that handles sensitive financial transactions? Architect

I would implement defense in depth across all layers: Transport: TLS 1.3 with public key pinning (not certificate pinning — easier rotation), backup pins, and a max-age policy. Authentication: OAuth 2.0 + PKCE with short-lived access tokens stored in the Keychain with kSecAttrAccessibleWhenUnlockedThisDeviceOnly and biometric protection via SecAccessControl. Refresh tokens in Keychain with the same protection. Data at rest: All financial data encrypted with AES-GCM using keys derived from a Secure Enclave P-256 key. File protection class set to NSFileProtectionComplete. Transaction integrity: App Attest for device integrity verification. Each transaction request signed with the attested key and verified server-side. Server-side: All business logic validation server-side. Rate limiting. Anomaly detection. Never trust client-reported state (premium status, account balance, etc.). No jailbreak detection: Instead, rely on App Attest and server-side validation. If App Attest fails, degrade gracefully (e.g., require additional authentication steps) rather than blocking entirely.

Q7: Compare iOS sandbox enforcement with Android's app isolation. What are the architectural tradeoffs? L6

iOS uses TrustedBSD MAC (Mandatory Access Control) with sandbox profiles — a policy-based system where the kernel checks every syscall against a declarative profile. Android uses Linux UID-based isolation (each app gets a unique UID) combined with SELinux MAC policies. Key differences: (1) Code signing: iOS verifies code signatures at page-fault time (AMFI), while Android verifies at install time only — this means iOS catches binary tampering at runtime. (2) Dynamic code: iOS enforces W^X (no JIT except JSCore), while Android allows JIT in ART — this gives iOS a smaller attack surface for code injection. (3) Process model: iOS denies fork() entirely, while Android's Zygote model forks for every app launch — Android uses seccomp-bpf to filter syscalls post-fork. (4) IPC: iOS uses Mach ports (kernel-mediated), Android uses Binder (kernel driver) — both are kernel-level but have different attack surfaces. The tradeoff: iOS is more restrictive (better security) but less flexible (no sideloading, no JIT, limited IPC). Android is more open but has a larger attack surface.

Q8: What are entitlements and how does the kernel use them? Give examples of entitlements that change sandbox behavior. Senior

Entitlements are key-value pairs embedded in the app's code signature that declare capabilities. At process creation, AMFI extracts them from the code signature and attaches them to the kernel's task structure. When the sandbox evaluates a policy rule, it can check entitlements via IOTaskHasEntitlement(). Examples: keychain-access-groups allows access to shared Keychain groups; com.apple.security.application-groups enables shared containers; aps-environment enables push notifications; com.apple.developer.associated-domains enables Universal Links. The critical insight is that entitlements are verified at two levels: Apple's signing infrastructure ensures you're allowed to claim them (provisioning profile), and the kernel enforces them at runtime. You cannot add entitlements without re-signing, and re-signing requires Apple's approval chain.

Q9: A background download completes while the device is locked. The file needs NSFileProtectionComplete. How do you handle this? Senior

You can't directly write a file with NSFileProtectionComplete while the device is locked — the class key isn't available. Solutions: (1) Use NSFileProtectionCompleteUnlessOpen for the download destination — this allows writing via the public key while locked. When the device is unlocked, re-encrypt with NSFileProtectionComplete by reading and rewriting the file. (2) Write to a temporary file with NSFileProtectionCompleteUntilFirstUserAuthentication, then listen for protectedDataDidBecomeAvailableNotification and move/re-protect the file. (3) If the data is highly sensitive, keep it in memory (if small enough) and write it only when UIApplication.shared.isProtectedDataAvailable is true. The architectural choice depends on the data size and sensitivity — option 1 is best for large files, option 3 for small secrets.

Q10: Explain the Triangulation attack (CVE-2023-32434). What does it teach us about defense in depth? Architect

Triangulation was a zero-click exploit chain discovered by Kaspersky in 2023. It used an iMessage attachment to trigger a WebKit vulnerability, then chained to a kernel exploit via undocumented MMIO (Memory-Mapped I/O) hardware registers to achieve full device compromise — all without user interaction. It was attributed to a sophisticated state actor and was active for years before discovery.

Lessons for defense in depth: (1) Even Apple's sandbox can be bypassed by a sufficiently motivated attacker with zero-day exploits. (2) Data encrypted with Secure Enclave keys would have remained protected even after kernel compromise — the SEP is a separate processor with its own security boundary. (3) Server-side detection (anomalous network traffic, unexpected API patterns) might catch compromised devices even when on-device detection fails. (4) Apple's response — Lockdown Mode — demonstrates that reducing attack surface (disabling iMessage attachment previews, WebKit JIT, etc.) is a valid security strategy for high-risk users. (5) This is why we encrypt sensitive data even within the sandbox — the sandbox is a defense layer, not an absolute guarantee.
