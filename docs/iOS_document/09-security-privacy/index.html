Series Overview

Security and privacy are not optional features вҖ” they are architectural constraints that shape every design decision in iOS development. At Google L6/Staff level, you're expected to understand not just the APIs, but the threat models they defend against, the hardware primitives they rely on, and the tradeoffs between security, usability, and performance.

Apple's security model is a layered defense-in-depth system. Each layer вҖ” from the Secure Enclave hardware to the app sandbox to App Transport Security вҖ” assumes the layer below it might be compromised. This series dissects every layer, from kernel-enforced sandboxing to the privacy manifests introduced in iOS 17.

Architect Must-Know

At L6/Staff level, security questions aren't about memorizing API calls. They test whether you can design secure systems вҖ” choosing the right protection class for keychain items, implementing certificate pinning that survives rotation, building privacy-respecting analytics, and understanding why jailbreak detection is fundamentally a losing game. You must reason about threat models, not just implementations.

Threat Model Hierarchy

Understanding iOS security requires thinking in terms of threat models вҖ” what attacks each layer defends against:

Threat Level Attacker Capability Defense Layers Example Scenario
L1: Network Can observe/modify network traffic (coffee shop WiFi) ATS, TLS, certificate pinning MITM attack on public WiFi
L2: Physical Access Has the device but not the passcode Data Protection classes, Secure Enclave, biometric auth Stolen device
L3: Malicious App Another app on the same device Sandbox, entitlements, keychain access groups Sideloaded malware (jailbroken device)
L4: Compromised OS Kernel-level access (jailbreak) Secure Enclave, hardware encryption, PAC State-sponsored attack, jailbroken device
L5: Supply Chain Can modify the app binary before installation Code signing, provisioning profiles, App Store review Compromised CI/CD pipeline
Common Interview Pitfall

Many candidates focus only on L1 (network security) and ignore L2-L5. Staff-level interviews expect you to reason about the full threat model. When asked "How would you secure sensitive data in your app?", the answer should address multiple layers: encryption at rest (L2), transport security (L1), sandbox isolation (L3), and Secure Enclave for key material (L4).
