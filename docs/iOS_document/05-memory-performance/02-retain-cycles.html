What Is a Retain Cycle?

A retain cycle occurs when two or more objects hold strong references to each other, forming a cycle that prevents any of them from being deallocated. Since ARC cannot detect cycles (unlike tracing GC), the memory is leaked for the lifetime of the process.

Retain Cycle: ┌──────────┐ strong ┌──────────┐ │ Object A │─────────▶│ Object B │ │ (RC = 1) │ │ (RC = 1) │ │ │◀─────────│ │ └──────────┘ strong └──────────┘ Neither can reach RC = 0 → Memory leaked forever Broken Cycle (with weak): ┌──────────┐ strong ┌──────────┐ │ Object A │─────────▶│ Object B │ │ (RC = 1) │ │ (RC = 0) │ ← can be freed │ │◀ ─ ─ ─ ─ │ │ └──────────┘ weak └──────────┘

The insidious nature of retain cycles: they don't crash. They silently consume memory until Jetsam kills the app. In long-running apps (messaging, navigation, media), even small leaks compound into hundreds of megabytes.

Closure Capture Lists

Closures are the #1 source of retain cycles in Swift. When a closure captures self (or any reference type), it creates a strong reference by default.

How Capture Works

class ViewController: UIViewController {
    var name = "Home"
    var onTap: (() -> Void)?

    func setupHandler() {
        // 💥 RETAIN CYCLE: closure captures self strongly
        // self → onTap (strong) → closure → self (strong)
        onTap = {
            print(self.name)
        }
    }
}
The Cycle: ┌──────────────────┐ │ ViewController │ │ ┌─────────────┐ │ ┌─────────────────┐ │ │ onTap ──────│─│────▶│ Closure │ │ └─────────────┘ │ │ ┌─────────────┐ │ │ │ │ │ self (strong)│──│──┐ │ │◀────│─│─────────────│──│──┘ │ │ │ └─────────────┘ │ └──────────────────┘ └─────────────────┘

Capture List Syntax

// Capture list goes before parameters
let closure = { [weak self, capturedValue = someValue] (param: Int) in
    guard let self else { return }
    print(self.name, capturedValue, param)
}

// Capture semantics:
var x = 10
let byReference = { print(x) }  // Captures x by reference
x = 20
byReference()                    // Prints 20

let byValue = { [x] in print(x) } // Captures x by value (copy)
x = 30
byValue()                        // Prints 20 (captured at creation)

When to Use [weak self] vs [unowned self]

Scenario Use Reason
Network completion handler [weak self] VC may be dismissed before response arrives
Animation completion [weak self] View may be removed during animation
Timer callback [weak self] Timer outlives the object that created it
Parent-child (child references parent) [unowned self] Parent always outlives child by design
Lazy property initialization [unowned self] Property can't exist without self
Combine sink on self's publisher [weak self] Subscription may outlive subscriber
DispatchQueue.async [weak self] Block may execute after deallocation
Escaping closure stored as property [weak self] Creates direct retain cycle

The guard let self Pattern

// Modern Swift (5.7+): guard let self
class ProfileViewModel {
    var name: String = ""

    func fetchProfile() {
        APIClient.fetch("/profile") { [weak self] result in
            guard let self else { return }
            // self is now a strong reference for the rest of the closure
            self.name = result.name
            self.updateUI()
        }
    }
}
Pitfall: guard let self Extends Lifetime

After guard let self, you hold a strong reference to self for the remainder of the closure. If the closure does long-running work after the guard, the object stays alive for that entire duration even if it would otherwise be deallocated. For short callbacks this is fine; for long-running operations, consider checking self at multiple points.

Delegate Pattern Leaks

The delegate pattern is a classic source of retain cycles. The delegating object holds a reference to its delegate, and if that reference is strong, a cycle forms when the delegate also holds the delegating object.

// ❌ WRONG: Strong delegate creates retain cycle
protocol DataManagerDelegate: AnyObject {
    func didLoadData(_ data: [Item])
}

class DataManager {
    var delegate: DataManagerDelegate?  // 💥 Strong reference!
}

class ViewController: UIViewController, DataManagerDelegate {
    let manager = DataManager()  // VC → manager (strong)

    override func viewDidLoad() {
        super.viewDidLoad()
        manager.delegate = self   // manager → VC (strong) 💥 CYCLE
    }
}

// ✅ CORRECT: Weak delegate
class DataManager {
    weak var delegate: DataManagerDelegate?  // Weak breaks cycle
}
Best Practice: Protocol Constraints for Weak Delegates

Always constrain delegate protocols to AnyObject (or class in older Swift). This is required for weak references and also documents that the delegate must be a reference type. Without this constraint, you can't declare the delegate property as weak.

NotificationCenter Observer Leaks

Pre-iOS 9: Manual Removal Required

// Pre-iOS 9: Forgetting to remove observer → crash (dangling pointer)
class OldViewController: UIViewController {
    override func viewDidLoad() {
        super.viewDidLoad()
        NotificationCenter.default.addObserver(
            self,
            selector: #selector(handleNotification),
            name: .someNotification,
            object: nil
        )
    }

    deinit {
        // REQUIRED pre-iOS 9, or crash when notification fires
        NotificationCenter.default.removeObserver(self)
    }
}

Post-iOS 9: Automatic Removal (Mostly)

Starting with iOS 9, NotificationCenter uses weak references for selector-based observers, so they're automatically removed on deallocation. However, block-based observers still require manual management:

class ModernViewController: UIViewController {
    var observer: NSObjectProtocol?

    override func viewDidLoad() {
        super.viewDidLoad()

        // ❌ Block-based: closure captures self, observer token retained
        observer = NotificationCenter.default.addObserver(
            forName: .someNotification,
            object: nil,
            queue: .main
        ) { [weak self] notification in
            self?.handleNotification(notification)
        }
    }

    deinit {
        // Still needed for block-based observers
        if let observer = observer {
            NotificationCenter.default.removeObserver(observer)
        }
    }
}
Deep Dive: Why Block-Based Observers Leak

The block-based addObserver(forName:object:queue:using:) returns an opaque NSObjectProtocol token. NotificationCenter holds a strong reference to this token, and the token holds a strong reference to the closure. If the closure captures self strongly and self holds the token, you get: self → token → closure → self. Using [weak self] in the closure breaks the cycle, but you still need to remove the observer to stop receiving notifications after deallocation.

Timer Retain Cycles

Timer (NSTimer) is one of the most common sources of retain cycles because the RunLoop holds a strong reference to the timer, and the timer holds a strong reference to its target.

Timer Retain Cycle: ┌──────────┐ strong ┌──────────┐ strong ┌──────────┐ │ RunLoop │─────────▶│ Timer │─────────▶│ self │ │ (main) │ │ │ │ (VC) │ └──────────┘ └──────────┘ └──────────┘ │ strong │ ▼ ┌──────────┐ │ Timer │ │ (stored) │ └──────────┘ Even if VC is dismissed, RunLoop → Timer → VC keeps it alive! 
class PollingViewController: UIViewController {
    var timer: Timer?

    override func viewDidLoad() {
        super.viewDidLoad()

        // ❌ LEAKS: Timer retains self, RunLoop retains timer
        timer = Timer.scheduledTimer(
            timeInterval: 5.0,
            target: self,
            selector: #selector(poll),
            userInfo: nil,
            repeats: true
        )
    }

    deinit {
        print("Never called! 💥")
    }
}

// ✅ Solution 1: Block-based timer with [weak self]
class FixedPollingVC: UIViewController {
    var timer: Timer?

    override func viewDidLoad() {
        super.viewDidLoad()
        timer = Timer.scheduledTimer(withTimeInterval: 5.0, repeats: true) { [weak self] _ in
            self?.poll()
        }
    }

    override func viewDidDisappear(_ animated: Bool) {
        super.viewDidDisappear(animated)
        timer?.invalidate()
    }
}

// ✅ Solution 2: Proxy pattern (for target/selector API)
class WeakTimerProxy {
    weak var target: AnyObject?
    let selector: Selector

    init(target: AnyObject, selector: Selector) {
        self.target = target
        self.selector = selector
    }

    @objc func fire(_ timer: Timer) {
        if let target = target {
            _ = target.perform(selector, with: timer)
        } else {
            timer.invalidate()
        }
    }
}

Combine Sink Leaks

Combine's sink operator returns an AnyCancellable that must be stored. If the closure captures self strongly and the cancellable is stored on self, you get a retain cycle.

class SearchViewModel: ObservableObject {
    @Published var query = ""
    @Published var results: [SearchResult] = []
    private var cancellables = Set<AnyCancellable>()

    init() {
        // ❌ RETAIN CYCLE:
        // self → cancellables → AnyCancellable → closure → self
        $query
            .debounce(for: .milliseconds(300), scheduler: RunLoop.main)
            .sink { query in
                self.performSearch(query)  // 💥 Strong capture
            }
            .store(in: &cancellables)

        // ✅ FIXED: [weak self]
        $query
            .debounce(for: .milliseconds(300), scheduler: RunLoop.main)
            .sink { [weak self] query in
                self?.performSearch(query)
            }
            .store(in: &cancellables)
    }
}

// ✅ Alternative: Use .assign(to:) for @Published properties
// .assign(to: &$results) uses an unowned reference internally
// and doesn't return an AnyCancellable (no storage needed)
Pitfall: .assign(to:on:) vs .assign(to:)

.assign(to:on:) captures the object strongly and returns an AnyCancellable — it can create retain cycles. .assign(to: &$property) (introduced in iOS 14) uses an unowned reference and doesn't return a cancellable — it's cycle-safe. Always prefer the &$ variant when assigning to a @Published property on self.

@MainActor Closure Captures

With Swift concurrency, @MainActor closures introduce subtle capture behavior:

class ViewModel {
    var data: [String] = []

    func loadData() {
        // Task captures self strongly by default
        Task { @MainActor in
            // self is captured strongly for the Task's lifetime
            self.data = await fetchData()
        }

        // ✅ Better: use [weak self]
        Task { [weak self] @MainActor in
            guard let self else { return }
            self.data = await fetchData()
        }
    }
}

// Note: Task doesn't create a retain cycle per se (it's not stored
// on self), but it extends self's lifetime until the task completes.
// For short-lived tasks this is fine; for long-running ones, use weak.
Deep Dive: Task Capture Semantics

A Task { } closure is @Sendable and @escaping. It captures variables by value for Sendable types and by strong reference for reference types. Unlike stored closures, a Task doesn't create a retain cycle because it's not stored on self — but it does keep self alive until the task completes. If the task takes 30 seconds and the user navigates away, the view controller stays in memory for those 30 seconds. Use [weak self] to allow early deallocation.

Common UIKit Retain Cycle Patterns

1. UIAlertAction Closures

// UIAlertController retains its actions, actions retain closures
func showAlert() {
    let alert = UIAlertController(title: "Error", message: "Retry?", preferredStyle: .alert)

    // ✅ Safe: UIAlertController is presented briefly, then dismissed
    // The cycle (self → present alert → action → closure → self)
    // is broken when the alert is dismissed and released.
    // But if you store the alert as a property, use [weak self].
    alert.addAction(UIAlertAction(title: "Retry", style: .default) { _ in
        self.retryRequest()
    })
    present(alert, animated: true)
}

2. UIView.animate Closures

// ✅ Safe: animation closures are non-escaping in practice
// They execute and are released by the animation system
UIView.animate(withDuration: 0.3) {
    self.view.alpha = 0  // No [weak self] needed
}

// ⚠️ But completion handlers can be delayed:
UIView.animate(withDuration: 0.3, animations: {
    self.view.alpha = 0
}) { _ in
    // This runs after animation completes — self kept alive
    // Usually fine, but use [weak self] for long animations
    self.view.removeFromSuperview()
}

3. Lazy Properties with Closures

class ProfileView: UIView {
    // ❌ RETAIN CYCLE: lazy closure captures self strongly
    lazy var nameLabel: UILabel = {
        let label = UILabel()
        label.textColor = self.tintColor  // captures self
        return label
    }()
    // Actually ✅: The closure executes once and is released.
    // The closure is not stored — it's called immediately on first access.
    // This is NOT a retain cycle despite capturing self.

    // ❌ REAL CYCLE: lazy property that stores a closure
    lazy var formatter: () -> String = {
        return self.name.uppercased()  // 💥 stored closure → self
    }
}

SwiftUI Retain Cycle Patterns

SwiftUI's value-type views largely avoid retain cycles, but ObservableObject classes and closures can still leak:

// ❌ ObservableObject with Combine cycle
class ContentViewModel: ObservableObject {
    @Published var items: [Item] = []
    let service: DataService
    var cancellables = Set<AnyCancellable>()

    init(service: DataService) {
        self.service = service
        service.itemsPublisher
            .sink { [weak self] items in  // ✅ weak self
                self?.items = items
            }
            .store(in: &cancellables)
    }
}

// ⚠️ SwiftUI .task modifier: captures are handled by the runtime
// The task is cancelled when the view disappears, so strong capture
// of the view model is usually fine (task lifetime ≤ view lifetime)
struct ContentView: View {
    @StateObject var viewModel = ContentViewModel()

    var body: some View {
        List(viewModel.items) { item in
            Text(item.name)
        }
        .task {
            await viewModel.loadItems()
        }
    }
}

Leak Detection Strategies

1. Xcode Memory Graph Debugger

The Memory Graph Debugger (Debug Navigator → Memory Graph) shows all live objects and their reference relationships. Look for:

  • Objects with unexpected strong references
  • Cycles highlighted with purple indicators
  • Objects that should have been deallocated (e.g., dismissed view controllers)

2. Instruments Leaks Template

The Leaks instrument periodically scans the heap for objects that are unreachable from any root but still allocated. It catches cycles that the Memory Graph Debugger might miss during manual inspection.

3. deinit Assertions

class BaseViewController: UIViewController {
    deinit {
        #if DEBUG
        print("✅ \(type(of: self)) deallocated")
        #endif
    }
}

// In tests: verify deallocation
func testViewControllerDeallocates() {
    weak var weakVC: MyViewController?
    autoreleasepool {
        let vc = MyViewController()
        weakVC = vc
        vc.loadViewIfNeeded()
        vc.viewDidLoad()
    }
    XCTAssertNil(weakVC, "ViewController should be deallocated")
}

4. Runtime Leak Checker

class LeakDetector {
    static let shared = LeakDetector()
    private var trackedObjects: [String: WeakBox] = [:]

    func track(_ object: AnyObject, file: String = #file, line: Int = #line) {
        let key = "\(type(of: object))_\(file):\(line)"
        trackedObjects[key] = WeakBox(object)
    }

    func expectDeallocation(after delay: TimeInterval = 2.0) {
        DispatchQueue.main.asyncAfter(deadline: .now() + delay) { [weak self] in
            guard let self else { return }
            for (key, box) in self.trackedObjects {
                if box.object != nil {
                    assertionFailure("🔴 LEAK: \(key) not deallocated")
                }
            }
            self.trackedObjects.removeAll()
        }
    }
}

private class WeakBox {
    weak var object: AnyObject?
    init(_ object: AnyObject) { self.object = object }
}

Building Leak Detection into CI

Architect Must-Know: CI-Integrated Leak Detection

At the Staff level, you should be able to design a leak detection pipeline that catches retain cycles before they reach production:

  1. Unit test deallocation assertions — every view controller and view model gets a test that verifies deallocation after use. Use the weak var + autoreleasepool pattern shown above.
  2. UI test memory snapshots — run UI tests that navigate through every screen, then check that dismissed screens are deallocated. Use XCTMemoryMetric to track peak memory.
  3. Instruments automation — use xctrace CLI to run the Leaks template in CI: 
    xctrace record --template Leaks \
  --launch MyApp.app \
  --output leaks.trace
  4. Memory Graph export — use leaks command-line tool on the simulator process to detect cycles programmatically.
  5. Production monitoring — use MetricKit's MXMemoryMetric to track peak memory and Jetsam events in production. Alert on regressions.

The goal: no retain cycle survives past code review. Make leak detection as automated as linting.

Automated Deallocation Test Pattern

import XCTest

class MemoryLeakTests: XCTestCase {

    func assertDeallocated<T: AnyObject>(
        _ factory: () -> T,
        file: StaticString = #file,
        line: UInt = #line
    ) {
        weak var weakRef: T?
        autoreleasepool {
            let instance = factory()
            weakRef = instance
        }
        XCTAssertNil(weakRef,
            "Expected \(T.self) to be deallocated",
            file: file, line: line)
    }

    func testProfileViewModelDeallocates() {
        assertDeallocated {
            let vm = ProfileViewModel()
            vm.loadProfile()
            return vm
        }
    }

    func testSettingsViewControllerDeallocates() {
        assertDeallocated {
            let vc = SettingsViewController()
            vc.loadViewIfNeeded()
            return vc
        }
    }
}
