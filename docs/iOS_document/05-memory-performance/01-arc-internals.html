ARC vs Garbage Collection

ARC and GC solve the same problem — automatic memory management — but with fundamentally different tradeoffs. Understanding these tradeoffs is critical for explaining why iOS chose ARC and what constraints it imposes.

Dimension ARC (Swift/ObjC) Tracing GC (Java/Go/C#)
When it runs Deterministic — at compile-inserted points Non-deterministic — GC decides when
Deallocation timing Immediate when refcount hits 0 Deferred until GC cycle
Pause times None (no stop-the-world) GC pauses (mitigated in modern GCs)
Cycles Cannot detect — programmer must break Automatically collected
Overhead Per retain/release call (atomic increment) Per GC cycle (mark/sweep/compact)
Memory high-water mark Lower — objects freed immediately Higher — objects linger until GC
Predictability High — deinit runs at known points Low — finalizers run at GC's discretion
Thread safety cost Atomic refcount operations Write barriers, card tables
Deep Dive: Why Apple Chose ARC Over GC

Apple actually shipped a tracing GC for Objective-C on macOS (10.5 Leopard, 2007) called libauto. It was deprecated in 10.8 and removed in 10.11. The reasons: GC pauses were unacceptable for UI responsiveness, memory overhead was too high for mobile devices, and the non-deterministic deallocation made resource management (file handles, GPU buffers) unreliable. ARC, introduced in 2011, gave the benefits of automatic memory management with deterministic deallocation and zero pause times — perfect for the resource-constrained, latency-sensitive iOS environment.

The Fundamental ARC Contract

ARC's contract is simple: the compiler inserts retain (increment refcount) and release (decrement refcount) calls at compile time. When the strong reference count reaches zero, the object is deallocated immediately. This happens:

  • When a variable goes out of scope
  • When a property is reassigned
  • When a collection element is removed
  • When an optional is set to nil
class Document {
    let name: String
    init(name: String) {
        self.name = name
        print("Document '\(name)' created")
    }
    deinit {
        print("Document '\(name)' deallocated")
    }
}

func example() {
    let doc = Document(name: "Report")  // refcount = 1
    let copy = doc                            // refcount = 2 (retain)
    print(copy.name)
    // end of scope: copy released (refcount = 1)
    //               doc released  (refcount = 0 → deinit)
}

Retain/Release Under the Hood

At the runtime level, Swift uses swift_retain and swift_release functions (not ObjC's objc_retain/objc_release, though those are used for @objc classes). These are implemented in the Swift runtime library.

The Atomic Increment

The strong reference count is modified using atomic operations to ensure thread safety. On ARM64 (all modern iOS devices), this uses the LDXR/STXR (load-exclusive/store-exclusive) instruction pair:

swift_retain(object): ┌─────────────────────────────────────────┐ │ 1. Load object's refcount (atomic) │ │ 2. Increment strong count by 1 │ │ 3. Store back (atomic, may retry on │ │ contention via LL/SC loop) │ │ 4. Return object pointer │ └─────────────────────────────────────────┘ swift_release(object): ┌─────────────────────────────────────────┐ │ 1. Load object's refcount (atomic) │ │ 2. Decrement strong count by 1 │ │ 3. If strong count == 0: │ │ a. Call object's deinit │ │ b. If no weak/unowned refs: │ │ → Free memory immediately │ │ c. If weak/unowned refs exist: │ │ → Mark as deallocating │ │ → Memory freed when last │ │ unowned ref is accessed or │ │ last weak ref is zeroed │ │ 4. Store back (atomic) │ └─────────────────────────────────────────┘

Three Reference Count Types

Swift objects maintain three separate reference counts, not just one:

Count Incremented By When Zero Effect
Strong Strong references (let x = obj) Object is deinitialized deinit runs, stored properties released
Unowned unowned references Object memory is freed Heap allocation returned to allocator
Weak weak references (via side table) Side table entry freed Side table deallocated
Pitfall: Unowned Keeps Memory Alive

A common misconception: unowned references don't prevent deinitialization, but they do prevent the memory from being freed. The object's memory stays allocated (in a "zombie" state) until all unowned references are gone. This means unowned can cause memory to linger longer than expected in pathological cases. With weak, the side table mechanism avoids this — the object memory is freed as soon as strong count hits zero.

Swift Object Layout

Every Swift class instance on the heap has a specific memory layout. Understanding this layout is essential for reasoning about memory overhead and performance.

Swift Heap Object Layout (64-bit): ┌────────────────────────────────────────────────┐ │ Metadata Pointer (8 bytes) │ ← ISA / type metadata ├────────────────────────────────────────────────┤ │ Reference Count (8 bytes) │ ← Inline refcount word ├────────────────────────────────────────────────┤ │ Stored Property 1 │ ├────────────────────────────────────────────────┤ │ Stored Property 2 │ ├────────────────────────────────────────────────┤ │ ... │ ├────────────────────────────────────────────────┤ │ Stored Property N │ └────────────────────────────────────────────────┘ Minimum object size: 16 bytes (metadata + refcount) Alignment: 16 bytes on ARM64

Metadata Pointer

The first 8 bytes point to the type metadata structure, which contains:

  • Value witness table — size, alignment, stride, copy/destroy functions
  • Virtual method table (vtable) — for class method dispatch
  • Superclass pointer — for inheritance chain
  • Type descriptor — name, generic parameters, field descriptors

For @objc classes, this pointer serves as the ISA pointer that the Objective-C runtime uses for message dispatch.

Reference Count Word

The second 8 bytes store the reference counts in a packed format:

Inline Reference Count Word (64 bits): ┌───────────────────────────────────────────────────────────┐ │ Bit 63 │ Bits 62-33 │ Bit 32 │ Bits 31-1 │ Bit 0 │ │ UseSlowRC │ Strong RC │ isDeinit│ Unowned RC│ Immortal │ │ (1 bit) │ (30 bits) │ (1 bit) │ (31 bits) │ (1 bit) │ └───────────────────────────────────────────────────────────┘ UseSlowRC = 1 → refcounts are in a side table (for weak refs) Immortal = 1 → object is never deallocated (e.g., static strings) isDeinit = 1 → object is currently being deinitialized

The inline format packs strong and unowned counts into a single 64-bit word, enabling atomic operations on both simultaneously. When weak references exist, the UseSlowRC bit is set and counts move to a side table.

Reference Count Storage

Inline (Fast Path)

When an object has only strong and unowned references (no weak), the reference counts are stored inline in the object header. This is the fast path — a single atomic read-modify-write on the 8-byte refcount word.

// Conceptual: what the runtime does for retain
func swift_retain(_ object: HeapObject) {
    // Single atomic increment of the strong count field
    let oldBits = object.refCounts.atomicLoad()
    let newBits = oldBits.incrementStrongCount()
    object.refCounts.atomicStore(newBits)
}

// Fast path: ~2-3 nanoseconds on A-series chips
// Compare to ObjC objc_retain: ~5-10ns (hash table lookup)

Out-of-Line (Slow Path — Side Table)

When the first weak reference to an object is created, the runtime:

  1. Allocates a side table entry on the heap
  2. Copies the current strong/unowned counts to the side table
  3. Replaces the inline refcount word with a pointer to the side table
  4. Sets the UseSlowRC bit

All subsequent retain/release operations go through the side table (the "slow path").

Side Tables for Weak References

The side table is a critical piece of Swift's memory management that enables weak reference zeroing without keeping the object's memory alive.

Without weak references (inline): ┌──────────────────┐ │ HeapObject │ │ ┌──────────────┐ │ │ │ metadata ptr │ │ │ ├──────────────┤ │ │ │ refcount │ │──── strong + unowned counts inline │ ├──────────────┤ │ │ │ properties │ │ │ └──────────────┘ │ └──────────────────┘ With weak references (side table): ┌──────────────────┐ ┌────────────────────┐ │ HeapObject │ │ Side Table Entry │ │ ┌──────────────┐ │ │ ┌────────────────┐ │ │ │ metadata ptr │ │ │ │ strong count │ │ │ ├──────────────┤ │ │ ├────────────────┤ │ │ │ side table ──│─│────────▶│ │ unowned count │ │ │ │ pointer │ │ │ ├────────────────┤ │ │ ├──────────────┤ │ │ │ weak count │ │ │ │ properties │ │ │ ├────────────────┤ │ │ └──────────────┘ │ │ │ object pointer │──│──▶ back to HeapObject └──────────────────┘ │ └────────────────┘ │ └────────────────────┘

Why Side Tables?

The side table design solves a critical problem: when an object is deallocated, weak references must be set to nil. Without side tables, you'd need to either:

  • Keep the object memory alive until all weak references are gone (wastes memory)
  • Maintain a list of all weak reference locations in the object (expensive to update)

With side tables, the object memory can be freed immediately when the strong count hits zero. Weak references point to the side table, not the object. When a weak reference is loaded, the runtime checks the side table — if the object is gone, it returns nil and decrements the weak count.

Weak Reference Zeroing Mechanism

When you read a weak reference, the runtime performs a multi-step process:

Loading a weak reference: ┌─────────────────────────────────────────────────┐ │ 1. Load side table pointer from weak variable │ │ 2. Lock the side table entry (spin lock) │ │ 3. Check if object is still alive: │ │ ├── YES: Increment strong count (retain) │ │ │ Return strong reference │ │ └── NO: Set weak variable to nil │ │ Decrement weak count │ │ Return nil │ │ 4. Unlock side table entry │ └─────────────────────────────────────────────────┘

This is why accessing a weak reference always produces an Optional — the runtime must check if the object is still alive, and it might not be.

class Cache {
    weak var delegate: CacheDelegate?

    func notifyDelegate() {
        // Runtime: load side table → check alive → retain or nil
        guard let delegate = delegate else {
            return // Object was deallocated
        }
        // delegate is now a strong reference (retained)
        delegate.cacheDidUpdate(self)
        // Strong reference released at end of scope
    }
}
Deep Dive: Weak Reference Performance Cost

Loading a weak reference is significantly more expensive than loading a strong reference: it involves a side table lookup, a spin lock, and a conditional retain. In hot loops, this matters. Profile before using weak in performance-critical paths — sometimes restructuring to avoid the weak reference entirely is better. The cost is approximately 10-20x a simple strong reference load.

    weak var delegate: NetworkDelegate?
}
Pitfall: unowned in Async Contexts

Never use unowned in closures that execute asynchronously (network callbacks, timers, GCD dispatches) unless you have an ironclad guarantee the referenced object outlives the closure. In practice, this is almost never true for async work. Use weak instead — the optional unwrapping cost is negligible compared to a crash in production.

withExtendedLifetime

withExtendedLifetime prevents the compiler from releasing an object before a certain point. This is necessary when you're working with unsafe pointers or bridging to C APIs that hold raw pointers to Swift objects.

class AudioBuffer {
    var samples: [Float] = []

    func processWithCAPI() {
        // Without withExtendedLifetime, the compiler might release
        // self before c_process_audio returns, because 'self' has
        // no more strong references after the pointer is taken.
        withExtendedLifetime(self) {
            samples.withUnsafeMutableBufferPointer { buffer in
                c_process_audio(buffer.baseAddress, buffer.count)
            }
        }
    }
}

// Also useful for keeping objects alive during observation:
let observer = NotificationCenter.default.addObserver(...)
withExtendedLifetime(observer) {
    // observer guaranteed alive for this block
    performWork()
}

When the Compiler Can Release Early

The Swift compiler's optimizer can release objects as soon as they're no longer used, even before the end of their lexical scope. This is called lifetime shortening and is usually beneficial, but can cause issues with unsafe code:

func dangerous() {
    let data = Data(repeating: 0, count: 1024)
    let ptr = (data as NSData).bytes
    // ⚠️ Compiler may release 'data' here — it's not used again
    // ptr is now a dangling pointer!
    memcpy(destination, ptr, 1024) // 💥 Use-after-free
}

func safe() {
    let data = Data(repeating: 0, count: 1024)
    withExtendedLifetime(data) {
        let ptr = (data as NSData).bytes
        memcpy(destination, ptr, 1024) // ✅ data alive
    }
}

Copy-on-Write for Value Types

Swift's standard library collections (Array, Dictionary, Set, String) use copy-on-write (CoW) to avoid unnecessary copying. The value type holds a reference to a heap-allocated buffer; copies share the buffer until one of them mutates.

Copy-on-Write Lifecycle: 1. Initial: var a = [1, 2, 3] ┌───┐ ┌──────────────┐ │ a │────▶│ Buffer (RC=1)│ └───┘ │ [1, 2, 3] │ └──────────────┘ 2. After copy (no mutation): var b = a ┌───┐ ┌──────────────┐ ┌───┐ │ a │────▶│ Buffer (RC=2)│◀────│ b │ └───┘ │ [1, 2, 3] │ └───┘ └──────────────┘ 3. After mutation (copy triggered): b.append(4) ┌───┐ ┌──────────────┐ ┌───┐ ┌──────────────┐ │ a │────▶│ Buffer (RC=1)│ │ b │────▶│ Buffer (RC=1)│ └───┘ │ [1, 2, 3] │ └───┘ │ [1, 2, 3, 4] │ └──────────────┘ └──────────────┘

Implementing Custom CoW

final class StorageBuffer<T> {
    var elements: [T]
    init(_ elements: [T]) { self.elements = elements }
}

struct CoWCollection<T> {
    private var storage: StorageBuffer<T>

    init(_ elements: [T]) {
        storage = StorageBuffer(elements)
    }

    var elements: [T] {
        storage.elements
    }

    mutating func append(_ element: T) {
        // isKnownUniquelyReferenced checks if RC == 1
        if !isKnownUniquelyReferenced(&storage) {
            storage = StorageBuffer(storage.elements)
        }
        storage.elements.append(element)
    }
}
Best Practice: isKnownUniquelyReferenced

Always use isKnownUniquelyReferenced for custom CoW types. It checks if the reference count is exactly 1 without any overhead beyond a single atomic load. The storage class must be a Swift native class (not an @objc class) — isKnownUniquelyReferenced always returns false for Objective-C objects.

Tagged Pointers

Tagged pointers are an Objective-C runtime optimization where small objects are stored directly in the pointer value instead of on the heap. On ARM64, the top bit indicates a tagged pointer, and the next 3 bits encode the class.

Regular pointer (heap object): ┌─────────────────────────────────────────────────────────────────┐ │ 0 [63 bits of heap address] │ └─────────────────────────────────────────────────────────────────┘ Tagged pointer (inline value): ┌─────────────────────────────────────────────────────────────────┐ │ 1 [3-bit class tag] [60 bits of payload data] │ └─────────────────────────────────────────────────────────────────┘ Tagged classes (ARM64): Tag 0: NSAtom (unused) Tag 1: NSNumber (int/float that fits in 60 bits) Tag 2: NSString (up to ~7 ASCII characters) Tag 3: NSDate Tag 4: NSData (small) Tag 5: NSManagedObjectID Tag 6: NSIndexPath Tag 7: Extended tag (8 more bits → 256 more classes)

Tagged pointers are significant for performance because they avoid heap allocation entirely — no malloc, no reference counting, no deallocation. They're particularly impactful for NSNumber (extremely common in ObjC bridging) and short strings.

// These NSNumber values are likely tagged pointers:
let n1 = NSNumber(value: 42)       // Fits in 60 bits → tagged
let n2 = NSNumber(value: 3.14)     // Float fits → tagged

// This NSString is likely a tagged pointer:
let s1 = "Hello" as NSString       // 5 ASCII chars → tagged

// This is NOT a tagged pointer (too long):
let s2 = "This is a longer string" as NSString  // Heap allocated
Deep Dive: Swift Strings and Tagged Pointers

Swift's String has its own small string optimization separate from NSString tagged pointers. A Swift String uses 16 bytes inline (two words). If the string content fits in 15 bytes (small ASCII or UTF-8), it's stored directly in those 16 bytes with no heap allocation. This is independent of the ObjC tagged pointer mechanism and works for pure Swift strings that never bridge to NSString.

ISA Pointer Optimization

On ARM64, Apple optimizes the ISA pointer (first word of every ObjC object) to pack additional information into unused bits. This is called the non-pointer ISA or optimized ISA.

Non-Pointer ISA (ARM64): ┌─────────────────────────────────────────────────────────────────┐ │ Bit 63 │ Bit 62 │ Bits 61-46 │ Bits 45-3 │ Bits 2-1│Bit 0│ │ nonptr │ has_assoc│ has_cxx_dtr│ shiftcls │ magic │ idx │ │ ISA flag │ objects? │ destructor?│ class ptr │ │ │ └─────────────────────────────────────────────────────────────────┘ Key fields packed into the ISA: - nonpointer (1 bit): 1 = optimized ISA - has_assoc (1 bit): Has associated objects? - has_cxx_dtor (1 bit): Has C++ destructor? - shiftcls (33 bits): Class pointer (shifted) - magic (6 bits): Identifies initialized objects - weakly_referenced (1 bit): Has weak references? - deallocating (1 bit): Currently in dealloc? - has_sidetable_rc (1 bit): Refcount overflowed to side table? - extra_rc (19 bits): Inline reference count (0-524287)

The key optimization: the reference count is stored inline in the ISA for ObjC objects (up to 524,287 in the 19-bit extra_rc field). This avoids a separate side table lookup for the vast majority of objects. Only when the count overflows does it spill to the ObjC side table (different from Swift's side table).

ARC at the Compiler Level

Understanding where the compiler inserts retain/release calls is essential for reasoning about performance. The Swift compiler operates on SIL (Swift Intermediate Language), and ARC operations are visible in the SIL output.

SIL ARC Instructions

SIL Instruction Meaning Runtime Call
strong_retain Increment strong refcount swift_retain
strong_release Decrement strong refcount swift_release
copy_value Retain (ownership transfer) swift_retain
destroy_value Release (end of ownership) swift_release
load_weak Load weak ref (may be nil) swift_weakLoadStrong
store_weak Store to weak variable swift_weakAssign

Viewing SIL Output

# Generate raw SIL (before optimization)
swiftc -emit-sil MyFile.swift

# Generate optimized SIL (after ARC optimization)
swiftc -emit-sil -O MyFile.swift

# Example: see ARC operations for a simple function
swiftc -emit-sil -O -parse-as-library example.swift | grep -E "retain|release|copy_value|destroy_value"
Architect Must-Know: ARC at the Compiler Level

At the Staff level, you should be able to explain the ARC optimization pipeline:

  1. SILGen — inserts conservative retain/release at every ownership transfer
  2. Mandatory passes — semantic checks, definite initialization
  3. ARC optimization passes — the key optimizations:
    • Retain/release pairing — removes retain immediately followed by release of the same value
    • Copy forwarding — eliminates copies when the source is not used after the copy
    • Owned-to-guaranteed conversion — passes parameters as borrowed instead of owned, eliminating retain/release at call sites
    • Dead object elimination — removes allocations whose results are never used
    • Semantic ARC optimization — uses ownership SSA to prove larger regions where retain/release can be eliminated
  4. LLVM lowering — SIL ARC instructions become calls to swift_retain/swift_release

The optimizer can typically eliminate 50-80% of the retain/release calls that SILGen initially inserts. You can verify this by comparing -emit-sil (unoptimized) vs -emit-sil -O (optimized) output.

Retain/Release Optimization

Example: Redundant Retain/Release Elimination

// Source code:
func getName(_ person: Person) -> String {
    let name = person.name
    return name
}

// Unoptimized SIL (conceptual):
// strong_retain %person          ← retain parameter
// %name = load %person.name
// strong_retain %name            ← retain loaded value
// strong_release %person         ← release parameter
// return %name

// Optimized SIL (conceptual):
// %name = load %person.name      ← parameter is borrowed, no retain
// strong_retain %name            ← only retain the return value
// return %name

Guaranteed (Borrowed) Parameters

One of the most impactful optimizations: the Swift calling convention uses guaranteed (borrowed) parameters by default. The caller retains the argument for the duration of the call, so the callee doesn't need to retain/release it:

// The caller guarantees 'person' is alive for the call duration
// So getName doesn't need to retain/release the parameter
func getName(_ person: Person) -> String {
    return person.name  // No retain of 'person' needed
}

// Compare with @owned convention (explicit):
// func getName(_ person: __owned Person) → caller transfers ownership
// callee must release when done

Escape Analysis and Stack Promotion

The optimizer can determine if a class instance never escapes the current function. If so, it can allocate it on the stack instead of the heap, eliminating all reference counting:

func computeArea(width: Double, height: Double) -> Double {
    // If Rect is a class, the optimizer may stack-promote this
    // allocation because 'r' never escapes the function
    let r = Rect(width: width, height: height)
    return r.area
    // No heap allocation, no refcounting
}
Best Practice: Help the Optimizer

To maximize ARC optimization:

  • Use final on classes that won't be subclassed — enables devirtualization and better escape analysis
  • Prefer struct over class when you don't need reference semantics
  • Use private/internal access control — the optimizer can see all uses within the module
  • Enable Whole Module Optimization (-whole-module-optimization) for release builds
  • Avoid unnecessary closures that capture self — each capture is a retain
