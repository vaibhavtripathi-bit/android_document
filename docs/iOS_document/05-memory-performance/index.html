Series 5: Memory Management & Performance

From ARC compiler internals and Jetsam OOM kills to app startup optimization and energy-aware design — the complete memory and performance story every Staff iOS engineer must master.

Memory management on iOS is fundamentally different from garbage-collected platforms. There is no GC pause — but there is also no safety net. ARC inserts retain/release calls at compile time, and getting them wrong means leaks, crashes, or the dreaded Jetsam termination with no warning. Performance profiling on iOS spans CPU time, memory footprint (dirty, compressed, clean), energy budget, thermal state, and launch time — all of which Apple measures in production via MetricKit and uses to rank your app.

This series covers the full depth expected at a Google L6 / Staff-level interview: not just "use [weak self]" but "explain the side table mechanism for weak references, how the Swift optimizer eliminates redundant retain/release pairs, and how you'd build a memory budget system for an app processing 4K video frames."

What Makes This Series L6-Level

  • Compiler-level ARC understanding — where retain/release are inserted in SIL, how the optimizer removes redundant ones, and how copy-on-write is implemented for value types
  • Jetsam internals — priority bands, memory footprint categories, why didReceiveMemoryWarning is unreliable, and how to build proactive memory budgets
  • Launch time mastery — pre-main (dyld4, shared cache, +load) and post-main (first frame) with real strategies to go from 3s to under 400ms
  • Production profiling pipeline — from Instruments in development to MetricKit and os_signpost in production, with CI integration
  • Energy and thermal awareness — designing for sustained workloads under thermal throttling, background energy budgets, and network coalescing

Prerequisites

This series assumes familiarity with:

  • Swift value types and reference types (structs, classes, enums)
  • Basic closure syntax and capture semantics
  • UIKit or SwiftUI app structure
  • Xcode development environment

For concurrency-related memory topics (actor isolation, Sendable, Combine), see Series 4: Concurrency & Threading.

Architect-Level Expectation

At the Staff/L6 level, you're expected to not just avoid memory issues but to design systems that make memory issues structurally impossible. This means building memory budgets, integrating leak detection into CI, designing data pipelines that respect Jetsam limits, and choosing architectures that minimize retain cycle risk. Each document in this series includes an Architect Callout with these system-level concerns.
