UIKit Evolution: iOS 2 through iOS 17+

A year-by-year chronicle of UIKit's most architecturally significant changes — from the original iPhone SDK through modern compositional layouts, content configurations, and the trait system overhaul. Each era introduced patterns that Staff engineers must understand to maintain, migrate, and architect large-scale iOS applications.

1. Foundation Era: iOS 2–6 (2008–2012)

iOS 2.0 (2008) — The Original SDK

The iPhone SDK introduced UIKit as a simplified descendant of AppKit (macOS). Key architectural decisions made here still affect us today:

  • UIView/CALayer split — Views handle events, layers handle rendering. This AppKit heritage means every UIView has a backing CALayer.
  • UITableView — Cell reuse pattern (dequeueReusableCell) became the foundation of all list UIs for 15+ years.
  • UINavigationController / UITabBarController — Container view controller pattern that still dominates app architecture.
  • Target-action pattern — addTarget(_:action:for:) using Objective-C selectors.
  • Delegate pattern — Protocol-based callbacks that became UIKit's primary communication mechanism.

iOS 3.0 (2009) — Core Data & Push

  • Core Data brought to iOS (previously macOS-only)
  • Push notifications via APNs
  • MapKit, In-App Purchase, copy/paste

iOS 4.0 (2010) — Multitasking & Blocks

  • Multitasking — 7 background modes (audio, location, VoIP, etc.)
  • Blocks (closures) — Enabled GCD and completion handler patterns
  • Grand Central Dispatch — Replaced manual thread management
  • Retina display — @2x assets, UIScreen.scale

iOS 5.0 (2011) — ARC & Storyboards

  • ARC (Automatic Reference Counting) — Compiler-inserted retain/release. Eliminated most manual memory management.
  • Storyboards — Visual UI design with segues. Would later be deprecated in favor of programmatic approaches.
  • UIAppearance — Global theming proxy (detailed in Section 6)
  • NSURLSession predecessor work, iCloud sync

iOS 6.0 (2012) — Auto Layout & UICollectionView

  • Auto Layout — Constraint-based layout using the Cassowary algorithm. Replaced autoresizingMask for complex layouts.
  • UICollectionView — Generalized grid/custom layout. UICollectionViewFlowLayout as default.
  • State Restoration — restorationIdentifier chain for preserving UI state across app kills.
Deep Dive: Why UICollectionView Was Revolutionary

Before iOS 6, developers built grids by embedding UIScrollView with manual layout or abusing UITableView. UICollectionView introduced a layout object abstraction — separating data (data source) from presentation (layout). This architectural pattern was so successful that it was later extended with CompositionalLayout (iOS 13) and DiffableDataSource (iOS 13), and conceptually influenced SwiftUI's LazyVGrid/LazyHGrid.

2. Modern UIKit: iOS 7–12 (2013–2018)

iOS 7.0 (2013) — The Flat Design Revolution

iOS 7 was the most dramatic visual overhaul in iOS history, requiring virtually every app to be redesigned:

  • Flat design — Removed skeuomorphism. Every custom UI needed updating.
  • UIKit Dynamics — Physics-based animations (UIDynamicAnimator)
  • TextKit — NSLayoutManager, NSTextStorage, NSTextContainer — replaced Core Text for most uses
  • View controller transitions — UIViewControllerTransitioningDelegate, custom modal/navigation transitions
  • Background fetch — application(_:performFetchWithCompletionHandler:)
  • JavaScriptCore — Embedded JavaScript engine

iOS 8.0 (2014) — Adaptive UI & Extensions

iOS 8 introduced the adaptive UI paradigm that remains foundational:

  • Size Classes — Compact/Regular for width/height (detailed in Section 7)
  • Trait Collections — UITraitCollection encapsulating device characteristics
  • App Extensions — Share, Today Widget, custom keyboard, photo editing
  • WKWebView — Replaced UIWebView (multi-process, JIT compilation)
  • Self-sizing cells — UITableViewAutomaticDimension for Auto Layout-driven cell heights
  • UIVisualEffectView — System blur effects
  • Presentation controllers — UIPresentationController for custom modal presentations

iOS 9.0 (2015) — UIStackView & iPad Multitasking

  • UIStackView — Simplified linear layouts without manual constraints
  • iPad multitasking — Split View, Slide Over (required adaptive UI)
  • App Transport Security — HTTPS required by default
  • SFSafariViewController — In-app browser with Safari features
  • Contacts framework — Replaced AddressBook (C-based API)

iOS 10.0 (2016) — UIViewPropertyAnimator

  • UIViewPropertyAnimator — Interruptible, reversible, scrubbable animations. Replaced many uses of UIView.animate.
  • UserNotifications framework — Unified local/remote notification API
  • SiriKit — Intent-based Siri integration
  • UIPreviewInteraction — 3D Touch peek/pop

iOS 11.0 (2017) — Safe Area & Drag-and-Drop

  • Safe area insets — safeAreaLayoutGuide replaced topLayoutGuide/bottomLayoutGuide (iPhone X notch)
  • Drag and Drop — UIDragInteraction/UIDropInteraction
  • Large titles — prefersLargeTitles on UINavigationBar
  • ARKit, Core ML, PDFKit, Files app integration

iOS 12.0 (2018) — Performance Focus

  • Performance improvements — Older devices (iPhone 5s, 6) ran significantly faster
  • Siri Shortcuts — NSUserActivity + INIntent for custom shortcuts
  • Grouped notifications
  • Network framework — Modern networking (replaces some CFNetwork uses)

3. Scene & Compositional Era: iOS 13 (2019)

iOS 13 was the most significant API shift since iOS 7, introducing three paradigm changes simultaneously:

iOS 13: The Triple Disruption

iOS 13 introduced (1) UIScene lifecycle, (2) CompositionalLayout + DiffableDataSource, and (3) SwiftUI — all in the same release. This created the largest migration burden in iOS history. Many apps are still completing this transition years later.

UIScene Lifecycle

Multi-window support required a fundamental rethinking of app architecture. See Section 8 for full details.

UICollectionViewCompositionalLayout

Replaced UICollectionViewFlowLayout for complex layouts. See Section 9.

DiffableDataSource

enum Section { case main }

var dataSource: UICollectionViewDiffableDataSource<Section, Item>!

// Apply a snapshot — no more reloadData() / performBatchUpdates()
var snapshot = NSDiffableDataSourceSnapshot<Section, Item>()
snapshot.appendSections([.main])
snapshot.appendItems(items)
dataSource.apply(snapshot, animatingDifferences: true)

Other iOS 13 Changes

  • Dark Mode — UIUserInterfaceStyle, dynamic colors, trait-based assets
  • SF Symbols — 1,500+ vector symbols with weight/scale variants
  • UIMenu / Context Menus — Replaced 3D Touch peek/pop (deprecated)
  • Modal presentation — Default changed to .pageSheet (not .fullScreen)

4. Configuration Era: iOS 14–15 (2020–2021)

iOS 14 (2020)

  • UICollectionView.CellRegistration — Type-safe cell registration replacing string-based identifiers
  • UICollectionViewListCell — Table view-style cells in collection views
  • UIColorPickerViewController
  • PHPickerViewController — Privacy-focused photo picker (replaces UIImagePickerController)
  • WidgetKit — SwiftUI-based widgets (replaced Today Extension)
// iOS 14: Type-safe cell registration
let cellRegistration = UICollectionView.CellRegistration<UICollectionViewListCell, Item> {

    content.secondaryText = item.subtitle
    content.image = item.icon

dataSource = UICollectionViewDiffableDataSource(

iOS 15 (2021)

  • UIButton.Configuration — Declarative button styling (see Section 10)
  • UISheetPresentationController — System half-sheet presentations (see Section 12)
  • UIContentConfiguration — Unified content configuration protocol (see Section 11)
  • UIListContentConfiguration — Standard list cell configurations
  • AttributedString — Swift-native attributed strings (replaces NSAttributedString for new code)
  • async/await URLSession — data(from:), bytes(from:), download(from:)

6. UIAppearance Proxy Deep Dive

Introduced in iOS 5, UIAppearance provides a proxy-based mechanism for global theming. It uses Objective-C's message forwarding to record method invocations and replay them on matching instances.

// Global appearance customization
UINavigationBar.appearance().tintColor = .systemBlue
UINavigationBar.appearance().barTintColor = .systemBackground
UINavigationBar.appearance().titleTextAttributes = [
    .foregroundColor: UIColor.label,
    .font: UIFont.systemFont(ofSize: 17, weight: .semibold)
]

// Scoped appearance — only when contained in specific containers
UIBarButtonItem.appearance(
    whenContainedInInstancesOf: [UINavigationBar.self]
).tintColor = .systemRed

// Custom view appearance (requires @objc dynamic)
class CustomButton: UIButton {
    @objc dynamic var customColor: UIColor? {
        didSet { backgroundColor = customColor }
    }
}
CustomButton.appearance().customColor = .systemTeal
UIAppearance Limitations
  • Only works with @objc dynamic properties — pure Swift properties are invisible to the proxy
  • Applied at view insertion time — changing appearance after views are on screen requires removing and re-adding them
  • No animation support — changes are immediate
  • Container-based scoping is limited to containment hierarchy, not arbitrary conditions
  • In iOS 13+, many appearance customizations moved to UINavigationBarAppearance, UITabBarAppearance, etc.

iOS 13+ Appearance Objects

// Modern appearance API (iOS 13+)
let appearance = UINavigationBarAppearance()
appearance.configureWithOpaqueBackground()
appearance.backgroundColor = .systemBackground
appearance.titleTextAttributes = [.foregroundColor: UIColor.label]
appearance.largeTitleTextAttributes = [.foregroundColor: UIColor.label]

// Apply to all states
navigationBar.standardAppearance = appearance
navigationBar.scrollEdgeAppearance = appearance
navigationBar.compactAppearance = appearance
navigationBar.compactScrollEdgeAppearance = appearance

7. Size Classes & Trait Collections

Size classes (iOS 8) abstracted device dimensions into two categories — Compact and Regular — for width and height independently. This enabled truly adaptive layouts.

Size Class Matrix (2024): Device / Orientation Width Height ───────────────────────────────────────────────── iPhone Portrait Compact Regular iPhone Landscape Compact* Compact iPhone Plus/Max Landscape Regular Compact iPad Portrait Regular Regular iPad Landscape Regular Regular iPad Split View (1/3) Compact Regular iPad Split View (1/2) Compact* Regular iPad Split View (2/3) Regular Regular iPad Slide Over Compact Regular Mac Catalyst Regular Regular * Some models vary. Always test with actual trait collections. 
// Responding to size class changes
override func traitCollectionDidChange(
    _ previousTraitCollection: UITraitCollection?
) {
    super.traitCollectionDidChange(previousTraitCollection)

    if traitCollection.horizontalSizeClass != previousTraitCollection?.horizontalSizeClass {
        updateLayoutForSizeClass()
    }
}

func updateLayoutForSizeClass() {
    switch traitCollection.horizontalSizeClass {
    case .compact:
        // Single-column layout
        stackView.axis = .vertical
        sidebarWidthConstraint.isActive = false
    case .regular:
        // Multi-column layout
        stackView.axis = .horizontal
        sidebarWidthConstraint.isActive = true
    default:
        break
    }
}

// Trait collection includes much more than size classes
let traits = traitCollection
traits.horizontalSizeClass    // .compact or .regular
traits.verticalSizeClass      // .compact or .regular
traits.userInterfaceStyle     // .light or .dark (iOS 13+)
traits.userInterfaceIdiom     // .phone, .pad, .mac, .tv, .vision
traits.displayScale           // 1.0, 2.0, 3.0
traits.preferredContentSizeCategory  // Dynamic Type size
traits.accessibilityContrast  // .normal or .high
traits.activeAppearance       // .active or .inactive (iOS 13+)
traits.layoutDirection        // .leftToRight or .rightToLeft

8. UIScene Architecture

iOS 13 introduced UIScene to support multiple windows. This fundamentally changed how apps manage their lifecycle and state.

Pre-iOS 13 Architecture: ┌──────────────────────────────┐ │ UIApplication │ │ (singleton, one window) │ │ │ │ ┌────────────────────────┐ │ │ │ UIApplicationDelegate│ │ │ │ - lifecycle events │ │ │ │ - state restoration │ │ │ │ - window management │ │ │ └────────────────────────┘ │ │ │ │ ┌────────────────────────┐ │ │ │ UIWindow │ │ │ │ (single instance) │ │ │ └────────────────────────┘ │ └──────────────────────────────┘ Post-iOS 13 Architecture: ┌──────────────────────────────────────────────────┐ │ UIApplication │ │ (singleton, shared) │ │ │ │ ┌─────────────────────────────────────────────┐ │ │ │ UIApplicationDelegate │ │ │ │ - process-level events only │ │ │ │ - scene configuration │ │ │ │ - NO window management │ │ │ └─────────────────────────────────────────────┘ │ │ │ │ ┌──────────────────┐ ┌──────────────────┐ │ │ │ UISceneSession │ │ UISceneSession │ │ │ │ ┌──────────────┐│ │ ┌──────────────┐│ │ │ │ │ UIWindowScene││ │ │ UIWindowScene││ │ │ │ │ ┌──────────┐││ │ │ ┌──────────┐││ │ │ │ │ │ UIWindow │││ │ │ │ UIWindow │││ │ │ │ │ └──────────┘││ │ │ └──────────┘││ │ │ │ └──────────────┘│ │ └──────────────┘│ │ │ │ UISceneDelegate │ │ UISceneDelegate │ │ │ └──────────────────┘ └──────────────────┘ │ └──────────────────────────────────────────────────┘ 
// Scene-based lifecycle (iOS 13+)
class SceneDelegate: UIResponder, UIWindowSceneDelegate {
    var window: UIWindow?

    func scene(
        _ scene: UIScene,
        willConnectTo session: UISceneSession,
        options connectionOptions: UIScene.ConnectionOptions
    ) {
        guard let windowScene = scene as? UIWindowScene else { return }
        window = UIWindow(windowScene: windowScene)
        window?.rootViewController = makeRootViewController()
        window?.makeKeyAndVisible()
    }

    func sceneDidBecomeActive(_ scene: UIScene) { }
    func sceneWillResignActive(_ scene: UIScene) { }
    func sceneDidEnterBackground(_ scene: UIScene) { }
    func sceneWillEnterForeground(_ scene: UIScene) { }
    func sceneDidDisconnect(_ scene: UIScene) {
        // Scene destroyed — clean up heavy resources
        // But session persists for state restoration
    }
}
Scene Lifecycle Gotcha: Disconnection ≠ Termination

When a scene is disconnected (sceneDidDisconnect), the session still exists. The system may reconnect the scene later with the same session. This means you should release heavy resources in sceneDidDisconnect but preserve state in the session's userInfo or stateRestorationActivity. Many apps incorrectly treat disconnection as termination, losing user state.

9. UICollectionViewCompositionalLayout

CompositionalLayout (iOS 13) replaced FlowLayout with a composable, section-based layout system. Each section can have a completely different layout.

CompositionalLayout Hierarchy: ┌─────────────────────────────────────────────────────┐ │ CompositionalLayout │ │ │ │ ┌─────────────────────────────────────────────────┐ │ │ │ Section 0 (e.g., Featured — horizontal scroll) │ │ │ │ ┌─────────────────────────────────────────────┐│ │ │ │ │ Group (horizontal) ││ │ │ │ │ ┌──────┐ ┌──────┐ ┌──────┐ ┌──────┐ ││ │ │ │ │ │ Item │ │ Item │ │ Item │ │ Item │ ... ││ │ │ │ │ └──────┘ └──────┘ └──────┘ └──────┘ ││ │ │ │ └─────────────────────────────────────────────┘│ │ │ └─────────────────────────────────────────────────┘ │ │ │ │ ┌─────────────────────────────────────────────────┐ │ │ │ Section 1 (e.g., Grid — 3 columns) │ │ │ │ ┌──────┐ ┌──────┐ ┌──────┐ │ │ │ │ │ Item │ │ Item │ │ Item │ │ │ │ │ ├──────┤ ├──────┤ ├──────┤ │ │ │ │ │ Item │ │ Item │ │ Item │ │ │ │ │ └──────┘ └──────┘ └──────┘ │ │ │ └─────────────────────────────────────────────────┘ │ │ │ │ ┌─────────────────────────────────────────────────┐ │ │ │ Section 2 (e.g., List — full width rows) │ │ │ │ ┌─────────────────────────────────────────────┐│ │ │ │ │ Item (full width) ││ │ │ │ ├─────────────────────────────────────────────┤│ │ │ │ │ Item (full width) ││ │ │ │ └─────────────────────────────────────────────┘│ │ │ └─────────────────────────────────────────────────┘ │ └─────────────────────────────────────────────────────┘ 
// CompositionalLayout with multiple section types
func createLayout() -> UICollectionViewCompositionalLayout {
    UICollectionViewCompositionalLayout { sectionIndex, environment in
        switch sectionIndex {
        case 0: return self.createFeaturedSection()
        case 1: return self.createGridSection(environment: environment)
        default: return self.createListSection()
        }
    }
}

func createFeaturedSection() -> NSCollectionLayoutSection {
    let itemSize = NSCollectionLayoutSize(
        widthDimension: .fractionalWidth(1.0),
        heightDimension: .fractionalHeight(1.0)
    )
    let item = NSCollectionLayoutItem(layoutSize: itemSize)
    item.contentInsets = NSDirectionalEdgeInsets(
        top: 4, leading: 4, bottom: 4, trailing: 4
    )

    let groupSize = NSCollectionLayoutSize(
        widthDimension: .fractionalWidth(0.85),
        heightDimension: .absolute(250)
    )
    let group = NSCollectionLayoutGroup.horizontal(
        layoutSize: groupSize, subitems: [item]
    )

    let section = NSCollectionLayoutSection(group: group)
    section.orthogonalScrollingBehavior = .groupPagingCentered
    return section
}

func createGridSection(
    environment: NSCollectionLayoutEnvironment
) -> NSCollectionLayoutSection {
    let columns = environment.container.effectiveContentSize.width > 500 ? 4 : 2
    let itemSize = NSCollectionLayoutSize(
        widthDimension: .fractionalWidth(1.0 / CGFloat(columns)),
        heightDimension: .fractionalWidth(1.0 / CGFloat(columns))
    )
    let item = NSCollectionLayoutItem(layoutSize: itemSize)
    item.contentInsets = .init(top: 2, leading: 2, bottom: 2, trailing: 2)

    let groupSize = NSCollectionLayoutSize(
        widthDimension: .fractionalWidth(1.0),
        heightDimension: .fractionalWidth(1.0 / CGFloat(columns))
    )
    let group = NSCollectionLayoutGroup.horizontal(
        layoutSize: groupSize, subitems: [item]
    )
    return NSCollectionLayoutSection(group: group)
}

10. UIButton.Configuration

iOS 15 introduced UIButton.Configuration, replacing the scattered property-based button customization with a unified, declarative configuration struct.

// Before iOS 15 — scattered properties
let button = UIButton(type: .system)
button.setTitle("Submit", for: .normal)
button.setImage(UIImage(systemName: "checkmark"), for: .normal)
button.backgroundColor = .systemBlue
button.setTitleColor(.white, for: .normal)
button.layer.cornerRadius = 8
button.contentEdgeInsets = UIEdgeInsets(top: 8, left: 16, bottom: 8, right: 16)

// iOS 15+ — Configuration API
var config = UIButton.Configuration.filled()
config.title = "Submit"
config.image = UIImage(systemName: "checkmark")
config.imagePadding = 8
config.cornerStyle = .medium
config.baseBackgroundColor = .systemBlue
config.baseForegroundColor = .white
config.contentInsets = NSDirectionalEdgeInsets(
    top: 8, leading: 16, bottom: 8, trailing: 16
)

// Dynamic configuration updates via configurationUpdateHandler
config.showsActivityIndicator = false

let button = UIButton(configuration: config)
button.configurationUpdateHandler = { button in
    var config = button.configuration
    config?.showsActivityIndicator = button.isSelected
    config?.title = button.isSelected ? "Loading..." : "Submit"
    button.configuration = config
}

// Built-in styles
let filled = UIButton.Configuration.filled()        // Solid background
let tinted = UIButton.Configuration.tinted()        // Translucent background
let gray = UIButton.Configuration.gray()            // Gray background
let plain = UIButton.Configuration.plain()          // No background
let bordered = UIButton.Configuration.bordered()    // Bordered style
let borderedTinted = UIButton.Configuration.borderedTinted()
Best Practice: Configuration vs Legacy Properties

When using UIButton.Configuration, do not mix it with legacy property setters (setTitle, setImage, backgroundColor). The configuration API and legacy API are mutually exclusive — setting a configuration clears legacy properties, and vice versa. For new code targeting iOS 15+, always use Configuration.

11. UIContentConfiguration Protocol

The UIContentConfiguration protocol (iOS 14+) unified how cells, headers, and other views configure their content. It replaces the old pattern of directly setting properties on cell subviews.

// The UIContentConfiguration protocol
protocol UIContentConfiguration {
    func makeContentView() -> UIView & UIContentView
    func updated(for state: UIConfigurationState) -> Self
}

// Using built-in UIListContentConfiguration
var content = UIListContentConfiguration.subtitleCell()
content.text = "Primary Text"
content.secondaryText = "Secondary Text"
content.image = UIImage(systemName: "star.fill")
content.imageProperties.tintColor = .systemYellow
content.textProperties.font = .preferredFont(forTextStyle: .headline)
cell.contentConfiguration = content

// Custom UIContentConfiguration
struct ProfileContentConfiguration: UIContentConfiguration {
    var name: String?
    var email: String?
    var avatarURL: URL?
    var isOnline: Bool = false

    func makeContentView() -> UIView & UIContentView {
        ProfileContentView(configuration: self)
    }

    func updated(for state: UIConfigurationState) -> ProfileContentConfiguration {
        var updated = self
        if let cellState = state as? UICellConfigurationState {
            if cellState.isHighlighted {
                // Adjust for highlighted state
            }
        }
        return updated
    }
}

13. Trait System Evolution (iOS 17)

iOS 17 overhauled the trait system with UIMutableTraits, custom trait definitions, and UITraitOverrides. This was the most significant trait system change since iOS 8.

Before iOS 17: Limited Trait Customization

// Before iOS 17: Override trait collection for child VCs
let childTraits = UITraitCollection(traitsFrom: [
    UITraitCollection(horizontalSizeClass: .compact),
    UITraitCollection(userInterfaceStyle: .dark)
])
setOverrideTraitCollection(childTraits, forChild: childVC)

// Problem: No way to define custom traits
// Problem: traitCollectionDidChange called for ANY trait change

iOS 17: Modern Trait System

// Define custom traits
struct ThemeTrait: UITraitDefinition {
    static let defaultValue: String = "default"
    static let affectsColorAppearance = true
}

extension UITraitCollection {
    var theme: String { self[ThemeTrait.self] }
}

extension UIMutableTraits {
    var theme: String {
        get { self[ThemeTrait.self] }
        set { self[ThemeTrait.self] = newValue }
    }
}

// Apply trait overrides at any level
view.traitOverrides.theme = "premium"
viewController.traitOverrides.horizontalSizeClass = .compact

// Register for specific trait changes (replaces traitCollectionDidChange)
registerForTraitChanges(
    [UITraitHorizontalSizeClass.self, UITraitUserInterfaceStyle.self]
) { (self: Self, previousTraitCollection: UITraitCollection) in
    self.updateLayout()
}

// Register for custom trait changes
registerForTraitChanges([ThemeTrait.self]) { (self: Self, _) in
    self.applyTheme(self.traitCollection.theme)
}
Deep Dive: Why traitCollectionDidChange Was Deprecated

The old traitCollectionDidChange(_:) was called for any trait change — size class, dark mode, Dynamic Type, accessibility, and more. This meant every view controller had to check which trait actually changed, leading to unnecessary work and bugs. The iOS 17 registration-based API lets you subscribe to specific traits, improving both correctness and performance.

Trait System Evolution Summary

Era Mechanism Limitation
iOS 8–12 traitCollectionDidChange Called for all trait changes, no custom traits
iOS 13 Added userInterfaceStyle, activeAppearance Still no custom traits, still global callback
iOS 17 registerForTraitChanges, UITraitDefinition, traitOverrides Requires iOS 17+ (N-2 adoption ~2026)

14. Architect: N-2 Migration Strategy

Architect Must-Know: Supporting iOS N-2

Large apps typically support the current iOS version and two prior (N-2). This creates a rolling window where new APIs become available. Staff engineers must plan adoption timelines years in advance.

Migration Decision Framework

Phased Migration Strategy for Large Codebases

// Phase 1: Abstraction layer (do this immediately)
protocol CollectionLayoutProvider {
    func createLayout() -> UICollectionViewLayout
}

// Phase 2: Dual implementation
struct ModernLayoutProvider: CollectionLayoutProvider {
    func createLayout() -> UICollectionViewLayout {
        if #available(iOS 13, *) {
            return createCompositionalLayout()
        } else {
            return createFlowLayout()
        }
    }
}

// Phase 3: When minimum target reaches iOS 13, remove fallback
struct ModernLayoutProvider: CollectionLayoutProvider {
    func createLayout() -> UICollectionViewLayout {
        createCompositionalLayout()
    }
}

Best Practice: Migration Playbook
  1. Audit — Identify all uses of deprecated APIs with -Wdeprecated compiler flag
  2. Abstract — Create protocol/wrapper layers around APIs that will change
  3. Implement — Build new implementation behind #available checks
  4. Test — Run full test suite on both old and new code paths
  5. Feature flag — Gate new implementation behind a remote feature flag for gradual rollout
  6. Monitor — Track crash rates and performance metrics per code path
  7. Clean up — When minimum deployment target passes the API version, remove old code path
