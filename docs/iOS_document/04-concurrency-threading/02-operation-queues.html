1. OperationQueue / Operation Architecture

OperationQueue and Operation (formerly NSOperationQueue and NSOperation) provide an object-oriented abstraction over GCD. While GCD works with blocks and queues, the Operation framework works with objects that encapsulate work — giving you dependency management, cancellation, KVO observation, and reusability that raw GCD blocks don't provide.

1.1 Class Hierarchy

Operation (abstract base class) ├── BlockOperation (convenience for block-based work) ├── Your custom Operation subclass │ ├── Synchronous operation (override main()) │ └── Asynchronous operation (override start(), manage state) └── (Internal) _NSOperationInternal OperationQueue ├── .main (singleton — dispatches to main thread) └── Custom instances (dispatch to background) └── Backed by a GCD concurrent queue internally

1.2 Relationship to GCD

OperationQueue uses GCD internally. Each OperationQueue has an underlying DispatchQueue that it dispatches operations to. You can even access it via underlyingQueue:

let opQueue = OperationQueue()
opQueue.name = "com.app.imageProcessing"
opQueue.maxConcurrentOperationCount = 4
opQueue.qualityOfService = .userInitiated

// Access the underlying GCD queue
let gcdQueue = opQueue.underlyingQueue
// You can set a custom underlying queue:
opQueue.underlyingQueue = DispatchQueue(
    label: "com.app.imageProcessing.gcd",
    qos: .userInitiated,
    attributes: .concurrent
)
Deep Dive: Why Operation Exists When We Have GCD

GCD is low-level and block-based — perfect for simple async dispatch but awkward for complex workflows. Operation adds: (1) Dependencies — "don't start B until A finishes"; (2) Cancellation — propagated through the dependency graph; (3) State observation — KVO on isReady/isExecuting/isFinished; (4) Reusability — encapsulate complex work in a class; (5) maxConcurrentOperationCount — built-in concurrency limiting without semaphores. These features make Operation the right choice for complex data processing pipelines.

2. Operation Lifecycle & State Machine

Every Operation follows a strict state machine. Understanding this is critical for building custom async operations correctly.

Operation State Machine: ┌──────────┐ │ Pending │ isReady = false │ │ (waiting for dependencies) └─────┬────┘ │ All dependencies finished ▼ ┌──────────┐ │ Ready │ isReady = true │ │ (waiting for queue slot) └─────┬────┘ │ Queue picks it up, calls start() ▼ ┌──────────┐ │ Executing│ isExecuting = true │ │ (main() running) └─────┬────┘ │ main() returns (sync) or │ you set isFinished = true (async) ▼ ┌──────────┐ │ Finished │ isFinished = true │ │ (cleanup, dependents notified) └──────────┘ Cancellation can happen at ANY state: - Pending → cancel() → still moves to Finished (skips execution) - Ready → cancel() → still starts, but isCancelled = true - Executing → cancel() → isCancelled = true (must check manually)

2.1 Key Properties

Property Type Description
isReady Bool (KVO) True when all dependencies are finished. Queue checks this before starting.
isExecuting Bool (KVO) True while the operation is actively running.
isFinished Bool (KVO) True when the operation is complete. Triggers dependent operations.
isCancelled Bool (KVO) True after cancel() is called. Must be checked cooperatively.
isAsynchronous Bool If true, the operation manages its own execution lifecycle.
queuePriority QueuePriority Scheduling priority within the queue (not system QoS).
qualityOfService QualityOfService Maps to GCD QoS for thread scheduling.

3. Dependencies Between Operations

Dependencies are the killer feature of Operation. You can express "B depends on A" — meaning B won't start until A's isFinished becomes true. Dependencies work across queues.

let downloadOp = DownloadOperation(url: imageURL)
let decodeOp = DecodeOperation()
let filterOp = FilterOperation(filter: .sepia)
let displayOp = DisplayOperation(imageView: imageView)

// Build the dependency chain
decodeOp.addDependency(downloadOp)   // Decode after download
filterOp.addDependency(decodeOp)     // Filter after decode
displayOp.addDependency(filterOp)    // Display after filter

// Operations can run on different queues!
let networkQueue = OperationQueue()
let processingQueue = OperationQueue()

networkQueue.addOperation(downloadOp)
processingQueue.addOperation(decodeOp)
processingQueue.addOperation(filterOp)
OperationQueue.main.addOperation(displayOp)

// Dependency graph:
// Download → Decode → Filter → Display
//  (network)  (bg)     (bg)    (main)

3.1 Passing Data Between Dependent Operations

Operations don't have a built-in data passing mechanism. You need to design this yourself:

class DownloadOperation: Operation {
    let url: URL
    var downloadedData: Data?  // Output property

    init(url: URL) { self.url = url; super.init() }

    override func main() {
        guard !isCancelled else { return }
        downloadedData = try? Data(contentsOf: url)
    }
}

class DecodeOperation: Operation {
    var inputData: Data?     // Input property
    var decodedImage: UIImage? // Output property

    override func main() {
        guard !isCancelled else { return }

        // Pull data from dependency
        let data = inputData ?? {
            let downloadOp = dependencies
                .compactMap { $0 as? DownloadOperation }
                .first
            return downloadOp?.downloadedData
        }()

        guard let data = data else { return }
        decodedImage = UIImage(data: data)
    }
}

// Adapter pattern for cleaner data passing
class AdapterOperation<Input, Output>: Operation {
    let transform: (Input) -> Output
    var input: Input?
    var output: Output?

    init(transform: @escaping (Input) -> Output) {
        self.transform = transform
        super.init()
    }

    override func main() {
        guard !isCancelled, let input = input else { return }
        output = transform(input)
    }
}
Common Pitfall: Circular Dependencies

If operation A depends on B and B depends on A, neither will ever execute. The queue will deadlock silently — no crash, no warning, just operations stuck in the "pending" state forever. Always validate your dependency graph is a DAG (Directed Acyclic Graph). For complex pipelines, consider building a validation step that detects cycles before adding operations to the queue.

4. Cooperative Cancellation

Cancellation in the Operation framework is cooperative — calling cancel() sets a flag, but the operation must check it and respond appropriately.

class LargeFileProcessor: Operation {
    let fileURL: URL

    init(fileURL: URL) {
        self.fileURL = fileURL
        super.init()
    }

    override func main() {
        // Check before starting
        guard !isCancelled else { return }

        let data = try? Data(contentsOf: fileURL)
        guard let data = data else { return }

        let chunkSize = 1024 * 1024  // 1MB chunks
        var offset = 0

        while offset < data.count {
            // Check cancellation at each iteration
            guard !isCancelled else {
                cleanupPartialWork()
                return
            }

            let end = min(offset + chunkSize, data.count)
            let chunk = data[offset..<end]
            processChunk(chunk)
            offset = end
        }
    }
}

// Cancellation propagation through the queue
let queue = OperationQueue()
queue.addOperations([op1, op2, op3, op4], waitUntilFinished: false)

// Cancel everything
queue.cancelAllOperations()
// This calls cancel() on each operation in the queue.
// Pending operations will have isCancelled = true when they start.
// Running operations must check isCancelled themselves.
Deep Dive: Cancellation and Dependencies

When you cancel an operation, its dependents are not automatically cancelled. A cancelled operation still transitions to isFinished = true (after its start() is called), which satisfies the dependency. The dependent operation will then start — with its own isCancelled still false. If you want cascading cancellation, you must implement it yourself, either by checking the dependency's isCancelled in the dependent's main(), or by cancelling the entire queue.

5. KVO on isFinished / isExecuting / isReady

The Operation framework relies heavily on KVO (Key-Value Observing) for its state machine. The queue observes isReady, isExecuting, and isFinished to know when to schedule and clean up operations.

5.1 Why KVO Matters for Custom Operations

For synchronous operations (override main()), the base class handles all KVO notifications. For asynchronous operations (override start()), you must manually fire KVO notifications when state changes — otherwise the queue won't know your operation finished.

class AsyncNetworkOperation: Operation {
    private var _isExecuting = false
    private var _isFinished = false

    override var isExecuting: Bool { _isExecuting }
    override var isFinished: Bool { _isFinished }
    override var isAsynchronous: Bool { true }

    override func start() {
        guard !isCancelled else {
            finish()
            return
        }

        willChangeValue(forKey: "isExecuting")
        _isExecuting = true
        didChangeValue(forKey: "isExecuting")

        // Start async work
        URLSession.shared.dataTask(with: request) { [weak self] data, _, error in
            self?.handleResponse(data: data, error: error)
            self?.finish()
        }.resume()
    }

    private func finish() {
        willChangeValue(forKey: "isExecuting")
        willChangeValue(forKey: "isFinished")
        _isExecuting = false
        _isFinished = true
        didChangeValue(forKey: "isFinished")
        didChangeValue(forKey: "isExecuting")
    }
}
Common Pitfall: Missing KVO Notifications

If you forget to fire willChangeValue/didChangeValue for isFinished, the queue will never remove your operation. It will stay in the queue's operations array forever, and any operations that depend on it will never start. This is one of the most common bugs with custom async operations. Always use a helper method like finish() above to ensure KVO is fired correctly.

5.2 Thread Safety for KVO Properties

// Thread-safe async operation base class
class AsyncOperation: Operation {
    private let stateQueue = DispatchQueue(
        label: "com.app.operation.state",
        attributes: .concurrent
    )

    private var _isExecuting = false
    private var _isFinished = false

    override var isExecuting: Bool {
        stateQueue.sync { _isExecuting }
    }

    override var isFinished: Bool {
        stateQueue.sync { _isFinished }
    }

    override var isAsynchronous: Bool { true }

    override func start() {
        guard !isCancelled else {
            markFinished()
            return
        }

        willChangeValue(forKey: "isExecuting")
        stateQueue.sync(flags: .barrier) { _isExecuting = true }
        didChangeValue(forKey: "isExecuting")

        execute()  // Subclasses override this
    }

    func execute() {
        fatalError("Subclasses must override execute()")
    }

    func markFinished() {
        willChangeValue(forKey: "isExecuting")
        willChangeValue(forKey: "isFinished")
        stateQueue.sync(flags: .barrier) {
            _isExecuting = false
            _isFinished = true
        }
        didChangeValue(forKey: "isFinished")
        didChangeValue(forKey: "isExecuting")
    }
}

6. Building Custom Async Operations

// Real-world example: Image download + cache operation
final class ImageFetchOperation: AsyncOperation {
    let url: URL
    private let cache: NSCache<NSURL, UIImage>
    private var task: URLSessionDataTask?

    var fetchedImage: UIImage?
    var fetchError: Error?

    init(url: URL, cache: NSCache<NSURL, UIImage>) {
        self.url = url
        self.cache = cache
        super.init()
    }

    override func execute() {
        // Check cache first
        if let cached = cache.object(forKey: url as NSURL) {
            fetchedImage = cached
            markFinished()
            return
        }

        task = URLSession.shared.dataTask(with: url) { [weak self] data, _, error in
            guard let self = self else { return }

            if self.isCancelled {
                self.markFinished()
                return
            }

            if let data = data, let image = UIImage(data: data) {
                self.cache.setObject(image, forKey: self.url as NSURL)
                self.fetchedImage = image
            } else {
                self.fetchError = error
            }

            self.markFinished()
        }
        task?.resume()
    }

    override func cancel() {
        task?.cancel()
        super.cancel()
    }
}

7. BlockOperation

BlockOperation is a concrete subclass that wraps one or more closures. It's the simplest way to use the Operation framework without subclassing.

// Simple BlockOperation
let op = BlockOperation {
    print("Processing data...")
    processData()
}

// Multiple execution blocks (all run concurrently!)
let multiOp = BlockOperation()
multiOp.addExecutionBlock { processChunk1() }
multiOp.addExecutionBlock { processChunk2() }
multiOp.addExecutionBlock { processChunk3() }
// The operation finishes when ALL blocks complete

// Completion block (runs after all execution blocks)
multiOp.completionBlock = {
    print("All chunks processed")
}

// Using with dependencies
let fetchOp = BlockOperation { fetchData() }
let parseOp = BlockOperation { parseData() }
let saveOp = BlockOperation { saveToDatabase() }

parseOp.addDependency(fetchOp)
saveOp.addDependency(parseOp)

let queue = OperationQueue()
queue.addOperations([fetchOp, parseOp, saveOp], waitUntilFinished: false)
Common Pitfall: BlockOperation Execution Blocks are Concurrent

When you add multiple execution blocks to a BlockOperation, they run concurrently, not sequentially. The operation is only considered finished when all blocks complete. If your blocks share mutable state, you'll have a data race. Use separate operations with dependencies for sequential execution.

8. When to Use OperationQueue over GCD

Feature GCD (DispatchQueue) OperationQueue async/await
Dependencies Manual (chaining blocks) Built-in, cross-queue Implicit (await)
Cancellation DispatchWorkItem (limited) Built-in, cooperative Task.cancel() cooperative
Max concurrency Semaphore (manual) maxConcurrentOperationCount TaskGroup (cooperative)
State observation None KVO on all state properties None (structured)
Reusability Closures (not reusable) Operation subclasses async functions
Overhead Minimal (C-level) Higher (ObjC objects, KVO) Moderate (continuations)
Thread explosion Possible Controlled via maxConcurrent Impossible (cooperative)
Best for Simple async dispatch Complex dependency graphs Most new code
Best Practice: Decision Framework
  • Use OperationQueue when: You need complex dependency graphs, KVO observation of task state, or you're working with legacy code that expects Operations
  • Use GCD when: You need simple async dispatch, barriers for reader-writer patterns, or DispatchSource for system events
  • Use async/await when: Writing new code — it's safer (no thread explosion), more readable, and has better cancellation support
  • Avoid: Mixing all three in the same subsystem — pick one primary concurrency model per module

9. Architect: Complex Dependency Graphs for Data Pipelines

Architect Must-Know

At the Staff level, you'll design data processing pipelines that involve downloading, parsing, transforming, and persisting data — often with complex dependencies, error handling, and cancellation requirements. OperationQueue excels here because it makes the dependency graph explicit and observable.

9.1 Real-World Pipeline: Social Feed Sync

Social Feed Sync Pipeline: ┌─────────────┐ ┌─────────────┐ ┌─────────────┐ │ Fetch Posts │ │ Fetch Users │ │ Fetch Media │ │ (network) │ │ (network) │ │ (network) │ └──────┬──────┘ └──────┬──────┘ └──────┬──────┘ │ │ │ └────────┬───────┘ │ │ │ ▼ │ ┌────────────────┐ │ │ Merge Posts │ │ │ + User Data │ │ └───────┬────────┘ │ │ │ └──────────┬─────────────┘ │ ▼ ┌────────────────┐ │ Resolve Media │ │ References │ └───────┬────────┘ │ ┌─────────┼─────────┐ │ │ │ ▼ ▼ ▼ ┌──────────┐ ┌────────┐ ┌──────────┐ │ Save to │ │ Update │ │ Notify │ │ Core Data│ │ Cache │ │ UI │ └──────────┘ └────────┘ └──────────┘ 
final class FeedSyncPipeline {
    private let networkQueue: OperationQueue
    private let processingQueue: OperationQueue
    private let persistenceQueue: OperationQueue

    init() {
        networkQueue = OperationQueue()
        networkQueue.name = "com.app.feed.network"
        networkQueue.maxConcurrentOperationCount = 3
        networkQueue.qualityOfService = .userInitiated

        processingQueue = OperationQueue()
        processingQueue.name = "com.app.feed.processing"
        processingQueue.maxConcurrentOperationCount = 2
        processingQueue.qualityOfService = .utility

        persistenceQueue = OperationQueue()
        persistenceQueue.name = "com.app.feed.persistence"
        persistenceQueue.maxConcurrentOperationCount = 1
        persistenceQueue.qualityOfService = .utility
    }

    func syncFeed() {
        let fetchPosts = FetchPostsOperation()
        let fetchUsers = FetchUsersOperation()
        let fetchMedia = FetchMediaMetadataOperation()

        let mergePosts = MergePostsUsersOperation()
        mergePosts.addDependency(fetchPosts)
        mergePosts.addDependency(fetchUsers)

        let resolveMedia = ResolveMediaOperation()
        resolveMedia.addDependency(mergePosts)
        resolveMedia.addDependency(fetchMedia)

        let saveToDB = SaveToCoreDataOperation()
        let updateCache = UpdateCacheOperation()
        let notifyUI = BlockOperation {
            DispatchQueue.main.async {
                NotificationCenter.default.post(
                    name: .feedDidSync, object: nil
                )
            }
        }

        saveToDB.addDependency(resolveMedia)
        updateCache.addDependency(resolveMedia)
        notifyUI.addDependency(saveToDB)
        notifyUI.addDependency(updateCache)

        // Dispatch to appropriate queues
        networkQueue.addOperations(
            [fetchPosts, fetchUsers, fetchMedia],
            waitUntilFinished: false
        )
        processingQueue.addOperations(
            [mergePosts, resolveMedia],
            waitUntilFinished: false
        )
        persistenceQueue.addOperations(
            [saveToDB, updateCache],
            waitUntilFinished: false
        )
        OperationQueue.main.addOperation(notifyUI)
    }

    func cancelSync() {
        networkQueue.cancelAllOperations()
        processingQueue.cancelAllOperations()
        persistenceQueue.cancelAllOperations()
    }
}

9.2 Design Principles for Operation Pipelines

Best Practice: Pipeline Architecture
  • Separate queues by concern: Network, processing, and persistence queues with different concurrency limits and QoS
  • Operations as value transformers: Each operation takes input, produces output — pure transformation where possible
  • Error propagation: Store errors on operations; dependents check dependency errors before proceeding
  • Cancellation at every level: Check isCancelled at the start of main() and at every significant checkpoint
  • Progress reporting: Use Progress objects (composable, KVO-observable) for UI feedback
  • Testability: Inject queues and dependencies; test operations in isolation
