1. libdispatch Architecture

Grand Central Dispatch (GCD) is Apple's C-based concurrency library, open-sourced as libdispatch. It was introduced in Mac OS X 10.6 (2009) and iOS 4 (2010). Unlike thread-based concurrency where you manage threads directly, GCD uses a work-submission model: you submit blocks of work to queues, and libdispatch manages a pool of kernel threads to execute them.

1.1 Core Components

┌─────────────────────────────────────────────────────────────────┐ │ Your Application │ │ │ │ dispatch_async(queue, block) dispatch_sync(queue, block) │ └──────────────────────┬───────────────────────────────────────────┘ │ ▼ ┌─────────────────────────────────────────────────────────────────┐ │ libdispatch │ │ │ │ ┌──────────────┐ ┌──────────────┐ ┌──────────────────────┐ │ │ │ Queue Manager│ │ Work Items │ │ Thread Pool Manager │ │ │ │ │ │ (blocks) │ │ (workqueue kext) │ │ │ └──────┬───────┘ └──────┬───────┘ └──────────┬───────────┘ │ │ │ │ │ │ │ └─────────────────┼──────────────────────┘ │ │ │ │ └───────────────────────────┼──────────────────────────────────────┘ │ ▼ ┌─────────────────────────────────────────────────────────────────┐ │ XNU Kernel │ │ │ │ ┌──────────────────┐ ┌────────────────────────────────────┐ │ │ │ pthread_workqueue│ │ Mach Scheduler │ │ │ │ (kernel ext) │ │ (priority bands, time quanta) │ │ │ └──────────────────┘ └────────────────────────────────────┘ │ └─────────────────────────────────────────────────────────────────┘

The key insight is that libdispatch doesn't create threads itself. It communicates with a kernel extension (pthread_workqueue) that manages a pool of worker threads. This is why GCD is more efficient than manual thread creation — the kernel can make globally optimal scheduling decisions.

1.2 The Dispatch Object Hierarchy

All GCD objects inherit from dispatch_object_t and are reference-counted (even in Swift's ARC world, GCD objects have their own retain/release):

dispatch_object_t ├── dispatch_queue_t │ ├── dispatch_queue_serial_t │ ├── dispatch_queue_concurrent_t │ └── dispatch_queue_main_t (singleton) ├── dispatch_source_t ├── dispatch_group_t ├── dispatch_semaphore_t ├── dispatch_data_t ├── dispatch_io_t └── dispatch_workloop_t (iOS 12+)

1.3 How dispatch_async Works Internally

When you call dispatch_async(queue, block), here's what happens at the libdispatch level:

  1. Block copy: The block is copied from stack to heap (if not already there). Captured variables are retained.
  2. Enqueue: The block is wrapped in a dispatch_continuation_t struct and appended to the queue's FIFO list (lock-free MPSC queue for serial, lock-free MPMC for concurrent).
  3. Drain signal: libdispatch signals the kernel workqueue that work is available.
  4. Thread wakeup: The kernel either wakes an idle thread or creates a new one (up to limits).
  5. Dequeue & execute: The worker thread dequeues the continuation and invokes the block.
  6. Cleanup: The block is released (and captured objects with it).
// What you write:
DispatchQueue.global(qos: .userInitiated).async {
    let data = processData()
    DispatchQueue.main.async {
        self.updateUI(with: data)
    }
}

// What libdispatch does internally (pseudocode):
// 1. _dispatch_continuation_init(block) → heap-allocate continuation
// 2. _dispatch_queue_push(queue, continuation) → MPSC enqueue
// 3. _dispatch_queue_wakeup(queue) → signal kernel workqueue
// 4. Worker thread: _dispatch_queue_drain(queue) → dequeue + invoke
Deep Dive: Lock-Free Queues

libdispatch uses lock-free data structures internally. Serial queues use an MPSC (Multiple Producer, Single Consumer) atomic linked list. The enqueue operation uses os_atomic_xchg (compare-and-swap) to atomically append to the tail. This means dispatch_async itself never blocks — it's O(1) and wait-free for the caller. The drain operation on the consumer side uses atomic load-acquire to process items in FIFO order.

2. Dispatch Queues: Serial vs Concurrent

2.1 Serial Queues

A serial queue executes one block at a time, in FIFO order. It acts as a mutual exclusion mechanism — a lightweight alternative to locks for protecting shared state.

// Creating a serial queue
let serialQueue = DispatchQueue(label: "com.app.cache")

// This is equivalent to:
let serialQueue = DispatchQueue(
    label: "com.app.cache",
    attributes: []  // no .concurrent → serial by default
)

// Using as a synchronization mechanism
class ThreadSafeCache<Key: Hashable, Value> {
    private var storage: [Key: Value] = [:]
    private let queue = DispatchQueue(label: "com.app.cache")

    func get(_ key: Key) -> Value? {
        queue.sync { storage[key] }
    }

    func set(_ key: Key, value: Value) {
        queue.async { [weak self] in
            self?.storage[key] = value
        }
    }
}
Common Pitfall: dispatch_sync Deadlock

Calling sync on the current serial queue causes a deadlock. The current block holds the queue's "execution token," and sync waits for that token — classic deadlock. This also applies transitively through target queues.

// DEADLOCK! Never do this:
let q = DispatchQueue(label: "com.app.serial")
q.async {
    q.sync {  // 💀 Deadlock: waiting for self
        print("never reached")
    }
}

// Also deadlocks on main queue:
DispatchQueue.main.sync {  // 💀 if called from main thread
    print("never reached")
}

2.2 Concurrent Queues

A concurrent queue can execute multiple blocks simultaneously. The queue still dequeues in FIFO order, but it doesn't wait for one block to finish before starting the next.

let concurrentQueue = DispatchQueue(
    label: "com.app.processing",
    attributes: .concurrent
)

// Reader-writer pattern using barrier
class ThreadSafeDictionary<Key: Hashable, Value> {
    private var storage: [Key: Value] = [:]
    private let queue = DispatchQueue(
        label: "com.app.dict",
        attributes: .concurrent
    )

    // Multiple readers execute concurrently
    func get(_ key: Key) -> Value? {
        queue.sync { storage[key] }
    }

    // Writer uses barrier: waits for all readers, then exclusive access
    func set(_ key: Key, value: Value) {
        queue.async(flags: .barrier) { [weak self] in
            self?.storage[key] = value
        }
    }
}
Concurrent Queue with Barrier: Time ──────────────────────────────────────────────────────────▶ Thread 1: ┃ Read A ┃ ┃ Read D ┃ Thread 2: ┃ Read B ┃ ┃ Read E ┃ Thread 3: ┃ Read C ┃ ┃ Read F ┃ ┃ BARRIER ┃ ┃ Write X ┃ ┃(exclusive)┃ All reads before barrier complete → barrier executes alone → reads after barrier start

2.3 Global Queues

Apple provides pre-created global concurrent queues at each QoS level. These are shared across the entire process:

// The four global concurrent queues
DispatchQueue.global(qos: .userInteractive)  // Highest priority
DispatchQueue.global(qos: .userInitiated)    // High priority
DispatchQueue.global(qos: .utility)           // Medium priority
DispatchQueue.global(qos: .background)        // Lowest priority

// Plus two special ones:
DispatchQueue.global(qos: .default)            // Between userInitiated and utility
DispatchQueue.global(qos: .unspecified)       // Inherits from context
Common Pitfall: Barrier on Global Queues

Using .barrier on a global queue has no barrier effect. Barriers only work on custom concurrent queues you create. Since global queues are shared by the entire process, allowing barriers would let any code block the entire system. The barrier flag is silently ignored on global queues.

2.4 Custom vs Global: When to Use Each

Aspect Custom Queue Global Queue
Barrier support Yes — reader/writer pattern No — barrier silently ignored
Debugging Custom label in crash logs Generic label, hard to trace
Target queue Configurable Fixed to root queue
Thread explosion risk Lower (controllable via target) Higher (shared, no control)

3. Thread Pool & the 64-Thread Limit

3.1 How the Thread Pool Works

GCD manages a per-process thread pool through the kernel's pthread_workqueue facility. The pool has these characteristics:

  • Lazy creation: Threads are created on demand, not upfront
  • Reuse: Idle threads are kept alive briefly for reuse
  • Per-workqueue limit: Each workqueue (mapped to a QoS class) can have up to 64 threads
  • Overcommit queues: Some queues (like serial queues) use "overcommit" workqueues that can exceed the 64-thread limit — this is how thread explosion happens

3.2 Thread Explosion: The #1 GCD Problem

Thread explosion occurs when GCD creates too many threads, exhausting system resources. Each thread costs ~512KB of stack space (1MB on macOS), plus kernel resources.

// 🚨 Thread explosion pattern — DO NOT DO THIS
for i in 0..<1000 {
    DispatchQueue.global().async {
        // Each block calls a synchronous API that blocks the thread
        let data = try? Data(contentsOf: urls[i])  // Blocks!
        process(data)
    }
}
// GCD sees 1000 blocks, each blocking → creates up to 64+ threads
// Each thread: 512KB stack → 32MB+ just for stacks
// Plus context switching overhead → app becomes unresponsive
Thread Explosion Timeline: Time 0ms: Block 0 submitted → Thread 1 created, starts executing Time 1ms: Block 1 submitted → Thread 1 busy (blocked on I/O) → Thread 2 created Time 2ms: Block 2 submitted → Threads 1,2 busy → Thread 3 created ... Time 63ms: Block 63 submitted → Thread 64 created (LIMIT HIT) Time 64ms: Block 64 submitted → ⚠️ Overcommit: Thread 65 created anyway ... Time 200ms: 200+ threads, each blocked on I/O Stack memory: 200 × 512KB = 100MB Context switches: thousands per second App: unresponsive, possible Jetsam kill

3.3 Preventing Thread Explosion

// ✅ Solution 1: Use a serial queue for serialized access
let processingQueue = DispatchQueue(label: "com.app.processing")
for url in urls {
    processingQueue.async {
        let data = try? Data(contentsOf: url)
        process(data)
    }
}

// ✅ Solution 2: Use a semaphore to limit concurrency
let semaphore = DispatchSemaphore(value: 4)  // Max 4 concurrent
for url in urls {
    DispatchQueue.global().async {
        semaphore.wait()
        defer { semaphore.signal() }
        let data = try? Data(contentsOf: url)
        process(data)
    }
}

// ✅ Solution 3: Use OperationQueue with maxConcurrentOperationCount
let opQueue = OperationQueue()
opQueue.maxConcurrentOperationCount = 4

// ✅ Solution 4 (Modern): Use TaskGroup with limited concurrency
await withTaskGroup(of: Data?.self) { group in
    for url in urls {
        group.addTask {
            try? await URLSession.shared.data(from: url).0
        }
    }
    // Swift concurrency uses cooperative threading — no explosion
}
Best Practice

The modern solution to thread explosion is Swift concurrency. The cooperative thread pool used by async/await is limited to the number of CPU cores (typically 6-8 on modern iPhones). Tasks suspend at await points instead of blocking threads, so you can have thousands of in-flight tasks without creating thousands of threads.

4. Quality of Service (QoS) Classes

QoS classes tell the system the intent of your work, allowing it to make intelligent scheduling decisions about CPU priority, I/O priority, timer coalescing, and CPU throughput vs efficiency core usage.

QoS Class Priority Use Case Scheduling Behavior
.userInteractive Highest (33) Main thread work, animations, event handling Performance cores, highest CPU priority, immediate scheduling
.userInitiated High (25) User tapped something, waiting for result Performance cores, high priority, minimal latency
.default Medium (21) Default when no QoS specified Balanced between performance and efficiency
.utility Low (17) Long-running tasks: downloads, imports, indexing Efficiency cores preferred, timer coalescing, I/O throttled
.background Lowest (9) Prefetching, maintenance, backups Efficiency cores only, heavily throttled, may be deferred

4.1 QoS Propagation Rules

QoS propagates through the system in specific ways:

// Rule 1: Blocks inherit the QoS of the submitting thread
DispatchQueue.global(qos: .userInitiated).async {
    // This block runs at .userInitiated

    DispatchQueue.global().async {
        // This inherits .userInitiated from the submitting context
        // NOT .default (the queue's own QoS)
    }
}

// Rule 2: Queue QoS acts as a floor
let utilityQueue = DispatchQueue(
    label: "com.app.utility",
    qos: .utility
)
// Submitting .background work to a .utility queue → runs at .utility
// Submitting .userInitiated work to a .utility queue → runs at .userInitiated

// Rule 3: dispatch_sync propagates QoS to the target queue
// If main thread (userInteractive) calls sync on a background queue,
// the block executes at userInteractive priority

5. Priority Inversion & Resolution

Priority inversion occurs when a high-priority task is blocked waiting for a low-priority task that can't run because medium-priority tasks are consuming CPU time.

5.1 Classic Priority Inversion Scenario

Priority Inversion: High Priority (H): ──────┃ BLOCKED waiting for lock ┃──────────▶ held by L Medium Priority (M): ──────┃ RUNNING (preempted L) ┃──────────────▶ Low Priority (L): ──┃RUN┃ preempted by M... can't release lock ┃ Problem: H waits for L, but L can't run because M keeps preempting it. H is effectively running at L's priority — INVERTED.

5.2 How GCD Resolves Priority Inversion

GCD uses priority boosting (also called priority inheritance) to resolve inversions:

  • dispatch_sync: When a high-QoS thread calls sync on a queue, GCD temporarily boosts the priority of the currently executing block on that queue to match the waiter's QoS.
  • DispatchSemaphore: When wait() is called, the system can boost the priority of the thread that will eventually call signal().
  • Serial queues: The queue's effective QoS is raised to the highest QoS of any pending block.
// Priority inversion scenario and GCD's resolution
let serialQueue = DispatchQueue(label: "com.app.shared")

// Low-priority work is running on the serial queue
serialQueue.async(qos: .background) {
    heavyComputation()  // Takes 5 seconds
}

// High-priority work needs the same queue
DispatchQueue.global(qos: .userInteractive).async {
    serialQueue.sync {
        // GCD detects: userInteractive is waiting for background work
        // Resolution: boost background block to userInteractive priority
        quickUIUpdate()
    }
}
Common Pitfall: os_unfair_lock vs DispatchSemaphore for Priority Inversion

os_unfair_lock (the recommended low-level lock on Apple platforms) supports priority inheritance natively through the kernel. DispatchSemaphore does not fully support priority inheritance in all cases — Apple's own documentation warns about this. For critical sections protecting shared state, prefer os_unfair_lock or serial dispatch queues over semaphores.

5.3 Priority Inversion with dispatch_semaphore

// ⚠️ Semaphore-based priority inversion (hard to detect)
let semaphore = DispatchSemaphore(value: 1)

// Background thread acquires semaphore
DispatchQueue.global(qos: .background).async {
    semaphore.wait()
    slowDatabaseWrite()  // Holds semaphore for a long time
    semaphore.signal()
}

// Main thread waits on semaphore — PRIORITY INVERSION
// The background thread holding the semaphore won't get boosted
// because semaphore doesn't track which thread holds it
semaphore.wait()  // 💀 Main thread blocked, UI frozen

6. DispatchWorkItem

DispatchWorkItem wraps a block with additional capabilities: cancellation, notification on completion, and QoS assignment.

// Creating and using a DispatchWorkItem
let workItem = DispatchWorkItem(qos: .userInitiated) {
    let result = performExpensiveSearch(query)
    DispatchQueue.main.async {
        updateSearchResults(result)
    }
}

// Submit to a queue
DispatchQueue.global().async(execute: workItem)

// Cancel if user types a new query
workItem.cancel()  // Sets isCancelled = true

// Notification when complete
workItem.notify(queue: .main) {
    print("Search complete (or cancelled)")
}

// Wait synchronously (with timeout)
let result = workItem.wait(timeout: .now() + 5.0)
switch result {
case .success: print("Completed")
case .timedOut: print("Timed out")
}
Common Pitfall: DispatchWorkItem Cancellation is Cooperative

Calling cancel() on a work item does not stop it from executing if it has already started. It only sets the isCancelled flag. If the block hasn't started yet, it will still be dequeued and "executed" — but the block body won't run. You must check isCancelled inside long-running work items for cancellation to be effective.

// Proper cancellation checking
let workItem = DispatchWorkItem {
    for i in 0..<10000 {
        guard !Thread.current.isCancelled else { return }
        // Note: DispatchWorkItem.isCancelled is not directly accessible
        // inside the block. You need to capture the item:
        processItem(i)
    }
}

// Better pattern: capture the work item for cancellation checking
var item: DispatchWorkItem?
item = DispatchWorkItem {
    for i in 0..<10000 {
        guard item?.isCancelled == false else { return }
        processItem(i)
    }
}
DispatchQueue.global().async(execute: item!)

7. DispatchGroup

DispatchGroup allows you to aggregate multiple asynchronous tasks and get notified when all of them complete. Internally, it uses an atomic counter.

let group = DispatchGroup()

// Pattern 1: Using enter/leave for async APIs
for url in imageURLs {
    group.enter()
    downloadImage(url) { image in
        processImage(image)
        group.leave()  // Must balance every enter()
    }
}

// Async notification when all complete
group.notify(queue: .main) {
    print("All images processed")
    updateUI()
}

// Pattern 2: Using dispatch_group_async (simpler for sync work)
let group2 = DispatchGroup()
let queue = DispatchQueue.global(qos: .userInitiated)

queue.async(group: group2) { task1() }
queue.async(group: group2) { task2() }
queue.async(group: group2) { task3() }

// Synchronous wait with timeout
let result = group2.wait(timeout: .now() + 10.0)
if result == .timedOut {
    print("Some tasks didn't complete in time")
}
Common Pitfall: Unbalanced enter/leave

Every enter() must have a matching leave(). If you call leave() more times than enter(), your app crashes (EXC_BAD_INSTRUCTION). If you call enter() without a matching leave() (e.g., in an error path), the group never completes. Always use defer { group.leave() } to ensure balance.

8. DispatchSemaphore

A counting semaphore that controls access to a resource by multiple threads. Initialized with a count; wait() decrements, signal() increments. When count reaches 0, wait() blocks.

// Limiting concurrent network requests
let maxConcurrent = DispatchSemaphore(value: 4)

for url in urls {
    DispatchQueue.global().async {
        maxConcurrent.wait()    // Blocks if 4 already running
        defer { maxConcurrent.signal() }

        let data = synchronousDownload(url)
        process(data)
    }
}

// Binary semaphore (mutex equivalent)
let mutex = DispatchSemaphore(value: 1)

mutex.wait()
// Critical section — only one thread at a time
sharedResource.modify()
mutex.signal()

// Making async API synchronous (use sparingly!)
func synchronousWrapper() -> Data? {
    let semaphore = DispatchSemaphore(value: 0)
    var result: Data?

    asyncFetch { data in
        result = data
        semaphore.signal()
    }

    semaphore.wait()
    return result
}
Common Pitfall: Semaphore on Main Thread

Never call semaphore.wait() on the main thread. It blocks the thread, freezing the UI. If the work that will call signal() also needs the main thread (e.g., a UIKit callback), you get a deadlock. This is a common source of app freezes in production.

9. DispatchSource

DispatchSource monitors low-level system events and delivers them as blocks on a dispatch queue. It's the bridge between kernel events and GCD.

9.1 Source Types

Source Type Monitors Use Case
.timer Time intervals Repeating tasks, debouncing
.read File descriptor readability Socket/pipe data available
.write File descriptor writability Socket/pipe ready for write
.vnode File system changes File modified/deleted/renamed
.process Process events Process exit, fork, signal
.signal UNIX signals SIGTERM, SIGHUP handling
.memoryPressure System memory warnings Cache eviction 
// DispatchSource timer — more precise than Timer
let timer = DispatchSource.makeTimerSource(queue: .global())
timer.schedule(
    deadline: .now(),
    repeating: .seconds(1),
    leeway: .milliseconds(100)  // Allows coalescing for energy
)
timer.setEventHandler {
    print("Tick at \(Date())")
}
timer.setCancelHandler {
    print("Timer cancelled")
}
timer.resume()  // Sources start suspended!

// File system monitoring
let fd = open("/path/to/file", O_EVTONLY)
let source = DispatchSource.makeFileSystemObjectSource(
    fileDescriptor: fd,
    eventMask: [.write, .delete, .rename],
    queue: .global()
)
source.setEventHandler {
    let flags = source.data
    if flags.contains(.write) { print("File modified") }
    if flags.contains(.delete) { print("File deleted") }
}
source.resume()
Deep Dive: DispatchSource vs Timer vs asyncAfter

DispatchSource.makeTimerSource is the most precise and configurable timer in GCD. Unlike Timer (which requires a RunLoop), dispatch timers work on any queue. Unlike asyncAfter (which is one-shot), dispatch timers support repeating with configurable leeway for energy optimization. The leeway parameter lets the system coalesce timer firings across the process, reducing CPU wakeups.

10. Target Queue Hierarchy

Every dispatch queue has a target queue — the queue where its blocks are ultimately executed. This forms a hierarchy that controls execution context, QoS, and mutual exclusion.

10.1 Default Target Queues

Target Queue Hierarchy (Default): ┌──────────────────────┐ │ Root Queue │ │ (per QoS level) │ │ Managed by kernel │ └──────────┬───────────┘ │ ┌────────────────┼────────────────┐ │ │ │ ┌─────────▼──────┐ ┌──────▼───────┐ ┌──────▼───────┐ │ Global Queue │ │ Global Queue │ │ Global Queue │ │ .userInitiated │ │ .utility │ │ .background │ └─────────┬──────┘ └──────┬───────┘ └──────┬───────┘ │ │ │ ┌─────────▼──────┐ ┌──────▼───────┐ ┌──────▼───────┐ │ Your Custom │ │ Your Custom │ │ Your Custom │ │ Serial Queue │ │ Concurrent Q │ │ Serial Queue │ └────────────────┘ └──────────────┘ └──────────────┘ By default, custom queues target the default-priority global concurrent queue.

10.2 Custom Target Queues

// Creating a queue hierarchy for mutual exclusion
let targetQueue = DispatchQueue(label: "com.app.subsystem")

let queue1 = DispatchQueue(
    label: "com.app.subsystem.cache",
    target: targetQueue
)
let queue2 = DispatchQueue(
    label: "com.app.subsystem.network",
    target: targetQueue
)

// queue1 and queue2 are MUTUALLY EXCLUSIVE because they share
// a serial target queue. Blocks from either queue will never
// execute concurrently with each other.

// This is powerful for subsystem-level synchronization:
// Each module gets its own queue for debugging/labeling,
// but they all serialize through the shared target.

10.3 Target Queue and QoS Interaction

// The target queue's QoS acts as a ceiling
let bgTarget = DispatchQueue(
    label: "com.app.bg",
    qos: .background
)

let childQueue = DispatchQueue(
    label: "com.app.child",
    qos: .userInitiated,  // Requested QoS
    target: bgTarget       // But target is background!
)

// The child's effective QoS will be resolved considering both
// its own QoS and the target's. The system takes the HIGHER
// of the two to prevent priority inversion within the hierarchy.
Deep Dive: DispatchWorkloop (iOS 12+)

DispatchWorkloop is a specialized serial queue that provides priority-ordered execution instead of FIFO. Work items are dequeued in priority order, and the workloop supports full priority inversion avoidance. This is what Swift concurrency's actor executor uses under the hood. It's not commonly used directly, but understanding it helps explain actor scheduling behavior.

11. Architect: Designing a Thread-Safe Work Scheduling System

Architect Must-Know

At the Staff level, you'll be asked to design concurrency architectures for entire subsystems. Here's a real-world design for a thread-safe work scheduler that handles prioritization, cancellation, and bounded concurrency.

11.1 Design Requirements

  • Process work items with configurable max concurrency
  • Support priority-based scheduling (higher priority items execute first)
  • Support cancellation of pending and in-flight work
  • Prevent thread explosion
  • Provide completion callbacks
  • Be testable (injectable queue for deterministic tests)
final class WorkScheduler<Result: Sendable> {

    enum Priority: Int, Comparable {
        case low = 0, normal = 1, high = 2, critical = 3
        static func < (lhs: Priority, rhs: Priority) -> Bool {
            lhs.rawValue < rhs.rawValue
        }
    }

    private struct WorkItem {
        let id: UUID
        let priority: Priority
        let work: () -> Result
        let completion: (Result?) -> Void
        var isCancelled = false
    }

    private var pendingItems: [WorkItem] = []
    private var activeCount = 0
    private let maxConcurrency: Int

    private let stateQueue = DispatchQueue(
        label: "com.app.scheduler.state"
    )
    private let executionQueue: DispatchQueue

    init(
        maxConcurrency: Int = 4,
        executionQueue: DispatchQueue = .global(qos: .userInitiated)
    ) {
        self.maxConcurrency = maxConcurrency
        self.executionQueue = executionQueue
    }

    func schedule(
        priority: Priority = .normal,
        work: @escaping () -> Result,
        completion: @escaping (Result?) -> Void
    ) -> UUID {
        let id = UUID()
        let item = WorkItem(
            id: id, priority: priority,
            work: work, completion: completion
        )

        stateQueue.async { [weak self] in
            self?.pendingItems.append(item)
            self?.pendingItems.sort { $0.priority > $1.priority }
            self?.drainQueue()
        }
        return id
    }

    func cancel(id: UUID) {
        stateQueue.async { [weak self] in
            if let index = self?.pendingItems.firstIndex(where: { $0.id == id }) {
                self?.pendingItems[index].isCancelled = true
                self?.pendingItems.remove(at: index)
            }
        }
    }

    private func drainQueue() {
        // Called on stateQueue — safe to access state
        while activeCount < maxConcurrency, !pendingItems.isEmpty {
            let item = pendingItems.removeFirst()
            guard !item.isCancelled else {
                item.completion(nil)
                continue
            }

            activeCount += 1
            executionQueue.async { [weak self] in
                let result = item.work()
                item.completion(result)

                self?.stateQueue.async {
                    self?.activeCount -= 1
                    self?.drainQueue()
                }
            }
        }
    }
}

11.2 Design Analysis

Design Decision Rationale
Serial stateQueue All state mutations are serialized — no locks needed, no data races

Best Practice: GCD Architecture Guidelines
  • One serial queue per subsystem for state protection — simpler than locks, no deadlock risk
  • Never block the main queue — use async dispatch for all non-trivial work
  • Use target queues to create subsystem-level mutual exclusion without a single bottleneck queue
  • Always specify QoS — let the system make intelligent scheduling decisions
  • Prefer custom queues over global queues — better debugging, barrier support, target queue control
  • Limit concurrency explicitly — use semaphores, OperationQueue, or TaskGroup instead of unbounded dispatch_async
  • Consider migrating to async/await — cooperative threading eliminates thread explosion by design
