1. RunLoop Fundamentals

A RunLoop is an event-processing loop that manages input sources (events) for a thread. Every thread can have at most one RunLoop. The main thread's RunLoop is created automatically and runs for the lifetime of the app — it's what keeps the app alive and responsive.

1.1 The RunLoop Cycle

Main RunLoop Cycle (simplified): ┌─────────────────────────────────────────────────────────┐ │ │ │ ┌──────────────────────┐ │ │ │ 1. Process Timers │ Fire any timers whose │ │ │ │ fire date has passed │ │ └──────────┬───────────┘ │ │ │ │ │ ▼ │ │ ┌──────────────────────┐ │ │ │ 2. Process Sources │ Handle input sources: │ │ │ (Source0) │ - Touch events │ │ │ │ - performSelector:onThread: │ │ └──────────┬───────────┘ │ │ │ │ │ ▼ │ │ ┌──────────────────────┐ │ │ │ 3. Process Sources │ Handle port-based sources: │ │ │ (Source1) │ - Mach port messages │ │ │ │ - CFMachPort callbacks │ │ └──────────┬───────────┘ │ │ │ │ │ ▼ │ │ ┌──────────────────────┐ │ │ │ 4. Process Blocks │ Execute blocks added via │ │ │ │ CFRunLoopPerformBlock │ │ └──────────┬───────────┘ │ │ │ │ │ ▼ │ │ ┌──────────────────────┐ │ │ │ 5. Sleep / Wait │ Block until: │ │ │ │ - A source fires │ │ │ │ - A timer fires │ │ │ │ - RunLoop timeout │ │ └──────────┬───────────┘ │ │ │ │ │ └────────────────────────────────────────────┘ │ (loop back to step 1)

1.2 RunLoop in Swift

// The main RunLoop (always exists)
let mainRunLoop = RunLoop.main
let cfMainRunLoop = CFRunLoopGetMain()

// Current thread's RunLoop (created on first access)
let currentRunLoop = RunLoop.current

// Running a RunLoop on a background thread (rare but possible)
let thread = Thread {
    let runLoop = RunLoop.current
    // Add sources/timers to the RunLoop
    runLoop.add(timer, forMode: .default)
    // Run the loop
    runLoop.run(until: Date.distantFuture)
}
thread.start()
Deep Dive: Why Does the App Need a RunLoop?

Without a RunLoop, the main thread would execute main() and exit — the app would terminate immediately. The RunLoop keeps the main thread alive by blocking (sleeping) when there's no work, and waking up to process events (touches, timers, network callbacks). It's the heartbeat of every iOS app. The UIApplicationMain function (or @main) sets up the main RunLoop and never returns.

2. RunLoop Modes: default, tracking, common

A RunLoop mode determines which sources and timers are processed during a given iteration of the loop. The RunLoop runs in exactly one mode at a time.

2.1 Standard Modes

Tracking .tracking / UITrackingRunLoopMode During UIScrollView scrolling Only scroll-tracking sources

RunLoop Mode Switching During Scroll: Normal operation: User starts scrolling: Scroll ends: ┌──────────────────┐ ┌──────────────────┐ ┌──────────────────┐ │ Mode: .default │───▶│ Mode: .tracking │────────▶│ Mode: .default │ │ │ │ │ │ │ │ ✅ Timers fire │ │ ❌ Timers pause │ │ ✅ Timers resume │ │ ✅ Sources fire │ │ ✅ Scroll events │ │ ✅ Sources fire │ │ ✅ Perform selectors│ │ ❌ Other sources │ │ ✅ Perform selectors│ └──────────────────┘ └──────────────────┘ └──────────────────┘ Solution: Add timer to .common mode → fires in BOTH modes

2.2 Custom RunLoop Modes

// You can create custom modes (rare but powerful)

3. CFRunLoop Sources

The RunLoop processes two types of sources, plus timers:

3.1 Source Types

Type Mechanism Examples
Source0 (non-port) Application-triggered. Must be manually signaled via CFRunLoopSourceSignal + CFRunLoopWakeUp Touch events, performSelector:onThread:, custom sources
Source1 (port-based) Kernel-triggered via Mach ports. Automatically wakes the RunLoop. Mach port messages, CFMachPort, system events from SpringBoard
Timers Fire at scheduled times. Not real-time — fire at the next RunLoop iteration after the fire date. Timer, CFRunLoopTimer, performSelector:afterDelay:
How Touch Events Reach Your Code: IOKit (hardware) → SpringBoard → Mach port (Source1) → Main RunLoop │ ▼ UIApplication.sendEvent() │ ▼ UIWindow.sendEvent() │ ▼ Hit testing → UIResponder chain

3.2 RunLoop Observers

// Observing RunLoop activity (useful for profiling)
let observer = CFRunLoopObserverCreateWithHandler(
    nil,  // allocator
    CFRunLoopActivity.allActivities.rawValue,  // activities to observe
    true,  // repeats
    0,     // order
    { observer, activity in
        switch activity {
        case .entry:
            print("RunLoop entered")
        case .beforeTimers:
            print("About to process timers")
        case .beforeSources:
            print("About to process sources")
        case .beforeWaiting:
            print("About to sleep")
        case .afterWaiting:
            print("Woke up")
        case .exit:
            print("RunLoop iteration complete")
        default: break
        }
    }
)

CFRunLoopAddObserver(CFRunLoopGetMain(), observer, CFRunLoopMode.defaultMode)
Deep Dive: RunLoop Observers for Jank Detection

A common Staff-level technique for detecting UI jank is to install a RunLoop observer that measures the time between .beforeSources and .afterWaiting. If a single RunLoop iteration takes longer than 16.67ms (for 60fps) or 8.33ms (for 120fps ProMotion), you've dropped a frame. This is how tools like Facebook's FBRetainCycleDetector and custom performance monitors work.

4. Why Timer Doesn't Fire During Scrolling

This is one of the most frequently asked iOS interview questions, and understanding it requires knowing RunLoop modes.

4.1 The Problem

// This timer STOPS firing when the user scrolls a UIScrollView
let timer = Timer.scheduledTimer(
    withTimeInterval: 1.0,
    repeats: true
) { _ in
    print("Tick")  // Stops printing during scroll!
}
// Timer.scheduledTimer adds to RunLoop.current in .default mode

4.2 Why It Happens

Timer in .default mode only: Normal: During scroll: ┌─────────────────────┐ ┌─────────────────────┐ │ RunLoop mode: │ │ RunLoop mode: │ │ .default │ │ .tracking │ │ │ │ │ │ Timer in .default ✅ │ │ Timer in .default ❌ │ │ Timer fires! │ │ Timer NOT processed! │ └─────────────────────┘ └─────────────────────┘ When UIScrollView begins tracking, the RunLoop switches to .tracking mode. Timers added only to .default mode are not processed in .tracking mode.

4.3 The Fix

// Fix 1: Add timer to .common modes (includes both .default and .tracking)
let timer = Timer(timeInterval: 1.0, repeats: true) { _ in
    print("Tick")  // Fires during scrolling too!
}
RunLoop.main.add(timer, forMode: .common)

// Fix 2: Use DispatchSourceTimer instead (not RunLoop-dependent)
let timer = DispatchSource.makeTimerSource(queue: .main)
timer.schedule(deadline: .now(), repeating: 1.0)
timer.setEventHandler {
    print("Tick")  // Always fires — not affected by RunLoop modes
}
timer.resume()

// Fix 3: Use Combine Timer publisher
Timer.publish(every: 1.0, on: .main, in: .common)
    .autoconnect()
    .sink { _ in
        print("Tick")  // Fires during scrolling
    }
Common Pitfall: Timer.scheduledTimer Default Mode

Timer.scheduledTimer always adds the timer to the current RunLoop in .default mode. If you call it from a background thread, it adds to that thread's RunLoop (which may not be running). If you call it from the main thread, it works but pauses during scrolling. Always explicitly add timers to .common mode if you need them to fire during user interaction.

5. RunLoop.main vs DispatchQueue.main

This is a subtle but critical distinction that trips up even experienced developers. They are not the same thing.

Aspect RunLoop.main DispatchQueue.main
What it is Event processing loop on main thread Serial dispatch queue bound to main thread
Mode-aware Yes — respects RunLoop modes No — always executes regardless of mode
When blocks execute During specific RunLoop phases Between RunLoop iterations (Source0)
Timer behavior Mode-dependent (may pause during scroll) DispatchSourceTimer: always fires
Combine scheduler RunLoop.main — mode-aware DispatchQueue.main — always fires
Use case Timer scheduling, RunLoop-aware APIs General main-thread dispatch 
// These behave differently during scrolling!

// RunLoop.main: respects modes
RunLoop.main.perform {
    print("RunLoop perform")
    // Executes in .default mode only
    // May be delayed during scrolling
}

// DispatchQueue.main: always executes
DispatchQueue.main.async {
    print("Dispatch async")
    // Executes regardless of RunLoop mode
    // Fires even during scrolling
}

// Combine: the scheduler matters
publisher
    .receive(on: RunLoop.main)  // May delay during scroll
    .sink { ... }

publisher
    .receive(on: DispatchQueue.main)  // Always delivers
    .sink { ... }
Deep Dive: How DispatchQueue.main Integrates with RunLoop

GCD's main queue is implemented as a Source0 on the main RunLoop. When you call DispatchQueue.main.async, the block is enqueued and the RunLoop is woken up via CFRunLoopWakeUp. The block executes during the Source0 processing phase of the RunLoop iteration. Because Source0 processing happens in all modes (it's not mode-specific like timers), main queue blocks execute even during scrolling. This is why DispatchQueue.main is generally more reliable than RunLoop.main.perform.

6. Main Actor Relationship

The @MainActor in Swift concurrency is backed by the main thread's executor, which uses DispatchQueue.main (not RunLoop.main). This means @MainActor code behaves like DispatchQueue.main.async, not like RunLoop-scheduled work.

// @MainActor uses DispatchQueue.main's executor
@MainActor
func updateUI() {
    // This runs on the main thread via the main dispatch queue
    // NOT via RunLoop.main.perform
    // Therefore: it executes even during scrolling
    label.text = "Updated"
}

// MainActor.run also uses the main dispatch queue
await MainActor.run {
    // Equivalent to DispatchQueue.main.async in terms of scheduling
    tableView.reloadData()
}

7. UI Thread Safety

All UIKit and AppKit operations must happen on the main thread. Violating this causes crashes, visual glitches, or undefined behavior.

7.1 Why Main Thread Only?

  • UIKit is not thread-safe: UIView, UIViewController, and all their subclasses use non-atomic properties and are not designed for concurrent access
  • Rendering pipeline: Core Animation commits happen on the main thread at the end of each RunLoop iteration. Modifying the layer tree from a background thread races with the commit
  • Event handling: Touch events are delivered on the main thread. Modifying the view hierarchy from another thread while touches are being processed causes crashes

7.2 Detecting Main Thread Violations

// Runtime detection: Main Thread Checker (Xcode)
// Enable in: Edit Scheme → Run → Diagnostics → Main Thread Checker
// Pauses on UIKit calls from background threads

// Compile-time detection: @MainActor (Swift 6)
@MainActor
func updateLabel(_ text: String) {
    label.text = text  // Compiler guarantees main thread
}

// Manual assertion
func unsafeUIUpdate() {
    dispatchPrecondition(condition: .onQueue(.main))
    // or
    assert(Thread.isMainThread)
    label.text = "Updated"
}

// Common patterns for ensuring main thread

// Pattern 1: DispatchQueue.main.async
DispatchQueue.global().async {
    let data = processData()
    DispatchQueue.main.async {
        self.label.text = data.title
    }
}

// Pattern 2: @MainActor (modern)
Task {
    let data = await processData()
    await MainActor.run {
        label.text = data.title
    }
}

// Pattern 3: @MainActor on the function
@MainActor
func display(_ data: ProcessedData) {
    label.text = data.title
    imageView.image = data.image
}

7.3 Common Main Thread Violations

// ❌ Violation 1: URLSession callback updating UI
URLSession.shared.dataTask(with: url) { data, _, _ in
    // This callback is on a background thread!
    self.imageView.image = UIImage(data: data!)  // 💀 Crash risk
}

// ❌ Violation 2: NotificationCenter observer on wrong thread
NotificationCenter.default.addObserver(
    forName: .dataDidUpdate,
    object: nil,
    queue: nil  // nil = fires on posting thread, which may be background!
) { _ in
    self.tableView.reloadData()  // 💀 May be background thread
}

// ✅ Fix: Specify .main queue
NotificationCenter.default.addObserver(
    forName: .dataDidUpdate,
    object: nil,
    queue: .main  // ✅ Always delivers on main thread
) { _ in
    self.tableView.reloadData()
}

8. Architect: RunLoop-Aware Programming for Responsive UIs

Architect Must-Know

Understanding the RunLoop is essential for building responsive UIs at scale. The main RunLoop has a budget of ~16.67ms per iteration (60fps) or ~8.33ms (120fps ProMotion). Every piece of work on the main thread eats into this budget. Staff engineers must design systems that respect this constraint.

8.1 Frame Budget

Frame Budget (60fps): One RunLoop iteration = One frame = 16.67ms ┌──────────────────────────────────────────────────────────────┐ │ 16.67ms budget │ │ │ │ ┌──────┐ ┌──────┐ ┌──────────┐ ┌────────┐ ┌─────────────┐ │ │ │Touch │ │Timer │ │ Layout │ │ Display│ │ CA Commit │ │ │ │events│ │fire │ │ pass │ │ pass │ │ (to render │ │ │ │ 1ms │ │ 2ms │ │ 4ms │ │ 3ms │ │ server) │ │ │ └──────┘ └──────┘ └──────────┘ └────────┘ │ 5ms │ │ │ └─────────────┘ │ │ Total: 15ms ✅ (within budget) │ └──────────────────────────────────────────────────────────────┘ If total > 16.67ms → DROPPED FRAME (jank)

8.2 Techniques for Staying Within Budget

// Technique 1: Defer expensive work to idle time
func deferToIdleTime(_ work: @escaping () -> Void) {
    // CFRunLoopPerformBlock schedules for the next idle iteration
    CFRunLoopPerformBlock(CFRunLoopGetMain(), CFRunLoopMode.defaultMode.rawValue) {
        work()
    }
    CFRunLoopWakeUp(CFRunLoopGetMain())
}

// Technique 2: Chunk work across multiple RunLoop iterations
func processInChunks(items: [Item], chunkSize: Int = 10) {
    var index = 0

    func processNextChunk() {
        let end = min(index + chunkSize, items.count)
        for i in index..<end {
            processItem(items[i])
        }
        index = end

        if index < items.count {
            // Schedule next chunk on next RunLoop iteration
            DispatchQueue.main.async {
                processNextChunk()
            }
        }
    }

    processNextChunk()
}

// Technique 3: Use CADisplayLink for frame-synchronized work
class SmoothAnimator {
    private var displayLink: CADisplayLink?

    func start() {
        displayLink = CADisplayLink(target: self, selector: #selector(tick))
        displayLink?.add(to: .main, forMode: .common)  // .common for scroll
    }

    @objc func tick(_ link: CADisplayLink) {
        let frameDuration = link.targetTimestamp - link.timestamp
        // frameDuration tells you the budget for this frame
        updateAnimation(dt: frameDuration)
    }
}

8.3 RunLoop Monitoring for Production

// Production jank detector using RunLoop observer
final class JankDetector {
    private var lastActivity: CFAbsoluteTime = 0
    private let threshold: CFAbsoluteTime = 0.05  // 50ms = 3 dropped frames

    func start() {
        let observer = CFRunLoopObserverCreateWithHandler(
            nil,
            CFRunLoopActivity.beforeWaiting.rawValue |
            CFRunLoopActivity.afterWaiting.rawValue,
            true, 0
        ) { [weak self] _, activity in
            guard let self = self else { return }

            if activity == .afterWaiting {
                self.lastActivity = CFAbsoluteTimeGetCurrent()
            } else if activity == .beforeWaiting {
                let elapsed = CFAbsoluteTimeGetCurrent() - self.lastActivity
                if elapsed > self.threshold {
                    reportJank(duration: elapsed)
                }
            }
        }

        CFRunLoopAddObserver(
            CFRunLoopGetMain(),
            observer,
            CFRunLoopMode.commonModes
        )
    }
}
Best Practice: Main Thread Guidelines
  • Never block the main thread: No synchronous network calls, no heavy computation, no file I/O
  • Use @MainActor for compile-time main thread safety instead of runtime assertions
  • Prefer DispatchQueue.main over RunLoop.main for general main-thread dispatch (mode-independent)
  • Add timers to .common mode if they must fire during scrolling
  • Chunk expensive work across multiple RunLoop iterations to maintain 60fps
  • Use CADisplayLink for frame-synchronized animations (add to .common mode)
  • Monitor jank in production with RunLoop observers and MetricKit
