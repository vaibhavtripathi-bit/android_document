1. async/await Under the Hood

Swift concurrency (introduced in Swift 5.5 / iOS 15) fundamentally changes how concurrent code executes. Instead of blocking threads, async functions suspend at await points, freeing the thread to do other work. The runtime uses a cooperative thread pool limited to the number of CPU cores.

1.1 The Cooperative Thread Pool

GCD Thread Model (old): Swift Concurrency Thread Model (new): ┌─────────────────────────┐ ┌─────────────────────────┐ │ Thread 1: Task A (blocked on I/O)│ │ Core 1: Task A → suspend │ │ Thread 2: Task B (blocked on I/O)│ │ → Task C runs │ │ Thread 3: Task C (running) │ │ → Task A resumes │ │ Thread 4: Task D (blocked on I/O)│ │ Core 2: Task B → suspend │ │ Thread 5: Task E (blocked on I/O)│ │ → Task D runs │ │ ... │ │ → Task B resumes │ │ Thread 64: Task ZZ (blocked) │ │ Core 3: Task E (running) │ │ ⚠️ Thread explosion! │ │ ... │ └─────────────────────────┘ │ Core 6: Task F (running) │ │ ✅ Only 6 threads total! │ 64+ threads, most blocked └─────────────────────────┘ 512KB × 64 = 32MB stack memory 6 threads, tasks suspend/resume Massive context switching Minimal context switching

1.2 How an async Function Compiles

The Swift compiler transforms async functions into a state machine. Each await point becomes a state transition. The function's local variables are stored in a heap-allocated async frame (not on the stack), so they survive suspension.

// What you write:
func fetchUserProfile(id: String) async throws -> UserProfile {
    let userData = try await api.fetchUser(id: id)       // Suspension point 1
    let avatar = try await api.fetchAvatar(url: userData.avatarURL) // Suspension point 2
    return UserProfile(user: userData, avatar: avatar)
}

// What the compiler generates (conceptually):
// A state machine with states: .entry, .afterFetchUser, .afterFetchAvatar
//
// State .entry:
//   Call api.fetchUser(id:) with continuation → suspend
//   Thread is freed to run other tasks
//
// State .afterFetchUser (resumed by continuation):
//   Store userData in async frame
//   Call api.fetchAvatar(url:) with continuation → suspend
//   Thread is freed again
//
// State .afterFetchAvatar (resumed by continuation):
//   Create UserProfile, return to caller's continuation
Deep Dive: Async Frames vs Stack Frames

Traditional function calls use stack frames — allocated on the thread's stack, automatically freed when the function returns. Async functions use async frames — heap-allocated structures that hold the function's local state across suspension points. This is why async functions can suspend without holding a thread: their state lives on the heap, not on any thread's stack. The tradeoff is heap allocation overhead, but this is amortized by the massive reduction in thread count.

2. Continuations & Suspension Points

Continuations are the mechanism that bridges callback-based APIs to async/await. They represent "the rest of the computation" after a suspension point.

2.1 withCheckedContinuation

// Bridging a callback-based API to async/await
func fetchImage(url: URL) async throws -> UIImage {
    try await withCheckedThrowingContinuation { continuation in
        URLSession.shared.dataTask(with: url) { data, response, error in
            if let error = error {
                continuation.resume(throwing: error)
                return
            }
            guard let data = data, let image = UIImage(data: data) else {
                continuation.resume(throwing: ImageError.decodingFailed)
                return
            }
            continuation.resume(returning: image)
        }.resume()
    }
}
Common Pitfall: Continuation Must Be Resumed Exactly Once

A continuation must be resumed exactly once. Resuming zero times leaks the task (it hangs forever). Resuming more than once is undefined behavior — with withCheckedContinuation, it triggers a runtime crash; with withUnsafeContinuation, it's a memory corruption bug. Always ensure every code path through your callback resumes the continuation.

// ❌ BUG: Missing resume on error path
func broken() async -> Data? {
    await withCheckedContinuation { continuation in
        fetchData { result in
            switch result {
            case .success(let data):
                continuation.resume(returning: data)
            case .failure:
                // 💀 Forgot to resume! Task hangs forever.
                break
            }
        }
    }
}

2.2 Checked vs Unsafe Continuations

Aspect withCheckedContinuation withUnsafeContinuation
Double-resume Runtime crash (detectable) Undefined behavior (memory corruption)
Never-resume Runtime warning in debug Silent hang
Performance Slight overhead for checking Zero overhead
Use case Development, most production code Hot paths where overhead matters

3. Structured Concurrency: TaskGroup

Structured concurrency means that concurrent tasks form a tree — child tasks are scoped to their parent, and the parent doesn't complete until all children complete (or are cancelled). This eliminates an entire class of resource leak bugs.

3.1 withTaskGroup

// Parallel fan-out: fetch multiple images concurrently
func fetchAllImages(urls: [URL]) async throws -> [UIImage] {
    try await withThrowingTaskGroup(of: (Int, UIImage).self) { group in
        for (index, url) in urls.enumerated() {
            group.addTask {
                let (data, _) = try await URLSession.shared.data(from: url)
                guard let image = UIImage(data: data) else {
                    throw ImageError.decodingFailed
                }
                return (index, image)
            }
        }

        // Collect results, maintaining original order
        var images = [UIImage?](repeating: nil, count: urls.count)
        for try await (index, image) in group {
            images[index] = image
        }
        return images.compactMap { $0 }
    }
}

3.2 TaskGroup Properties

Structured Concurrency Tree: Parent Task ┌────┼────┐ │ │ │ Child Child Child Task1 Task2 Task3 │ ┌──┼──┐ Grand Grand child child Rules: 1. Parent waits for ALL children before completing 2. If parent is cancelled → all children are cancelled 3. If a child throws → other children are cancelled (ThrowingTaskGroup) 4. Children inherit parent's priority and task-local values 5. Children cannot outlive their parent scope

3.3 Limiting Concurrency in TaskGroup

// Process 1000 items with max 10 concurrent tasks
func processWithLimit(items: [Item]) async -> [Result] {
    await withTaskGroup(of: Result.self) { group in
        var results: [Result] = []
        var iterator = items.makeIterator()

        // Seed with initial batch
        for _ in 0..<min(10, items.count) {
            if let item = iterator.next() {
                group.addTask { await process(item) }
            }
        }

        // As each completes, add the next
        for await result in group {
            results.append(result)
            if let item = iterator.next() {
                group.addTask { await process(item) }
            }
        }

        return results
    }
}

4. Task vs Task.detached

This is one of the most critical distinctions in Swift concurrency. Getting it wrong leads to subtle bugs with actor isolation, priority, and cancellation.

Aspect Task { } Task.detached { }
Actor context Inherits — runs on same actor Does not inherit — runs on default executor
Priority Inherits from creating context Must specify explicitly (or .medium)
Task-local values Inherits from parent Does not inherit
Cancellation Not automatically cancelled with parent Not automatically cancelled with parent
Structured? No (unstructured) No (unstructured) 
@MainActor
class ViewModel: ObservableObject {
    @Published var data: [Item] = []

    func loadData() {
        // Task inherits @MainActor context
        Task {
            // This closure runs on MainActor
            let items = try await api.fetchItems()
            data = items  // ✅ Safe: we're on MainActor
        }

        // Task.detached does NOT inherit @MainActor
        Task.detached {
            let items = try await self.api.fetchItems()
            await MainActor.run {
                self.data = items  // Must explicitly hop to MainActor
            }
        }
    }
}
Common Pitfall: Task { } is NOT Structured

Despite being called "Task," Task { } and Task.detached { } are both unstructured concurrency. They create top-level tasks that are not children of any parent. Only tasks created inside withTaskGroup or withThrowingTaskGroup are truly structured. Unstructured tasks can outlive their creating scope, which means they can leak if you don't store and cancel the Task handle.

// Storing task handles for cancellation
class SearchViewModel {
    private var searchTask: Task<Void, Never>?

    func search(query: String) {
        // Cancel previous search
        searchTask?.cancel()

        searchTask = Task {
            try? await Task.sleep(for: .milliseconds(300))  // Debounce
            guard !Task.isCancelled else { return }
            let results = try? await api.search(query)
            guard !Task.isCancelled else { return }
            await updateResults(results ?? [])
        }
    }

    func cleanup() {
        searchTask?.cancel()
        searchTask = nil
    }
}

5. Task Cancellation (Cooperative Model)

Like Operation, Swift concurrency uses cooperative cancellation. Calling task.cancel() sets a flag; the task must check it and respond.

5.1 Cancellation Checking Patterns

func processLargeDataset(items: [Item]) async throws -> [Result] {
    var results: [Result] = []

    for item in items {
        // Method 1: Check and throw
        try Task.checkCancellation()  // Throws CancellationError

        // Method 2: Check and handle gracefully
        guard !Task.isCancelled else {
            return results  // Return partial results
        }

        let result = await process(item)
        results.append(result)
    }

    return results
}

// Cancellation-aware sleep
func pollWithBackoff() async throws {
    var delay: Duration = .seconds(1)

    while true {
        let status = try await checkStatus()
        if status == .complete { return }

        // Task.sleep throws CancellationError if task is cancelled
        try await Task.sleep(for: delay)
        delay = min(delay * 2, .seconds(60))
    }
}

5.2 Cancellation Propagation in TaskGroup

// In ThrowingTaskGroup: one child throws → others cancelled
func fetchAllOrFail(urls: [URL]) async throws -> [Data] {
    try await withThrowingTaskGroup(of: Data.self) { group in
        for url in urls {
            group.addTask {
                let (data, _) = try await URLSession.shared.data(from: url)
                return data
            }
        }

        var results: [Data] = []
        for try await data in group {
            results.append(data)
        }
        return results
        // If ANY child throws, the group cancels remaining children
        // and rethrows the error
    }
}

// In non-throwing TaskGroup: cancellation must be manual
func fetchFirstSuccess(urls: [URL]) async -> Data? {
    await withTaskGroup(of: Data?.self) { group in
        for url in urls {
            group.addTask {
                try? await URLSession.shared.data(from: url).0
            }
        }

        for await result in group {
            if let data = result {
                group.cancelAll()  // Cancel remaining tasks
                return data
            }
        }
        return nil
    }
}

6. Task Priority & Priority Escalation

Tasks have priorities that map to GCD QoS classes. The runtime supports priority escalation — if a high-priority task awaits a low-priority task, the low-priority task's priority is boosted.

// Task priorities
Task(priority: .high) { ... }       // Maps to .userInitiated
Task(priority: .medium) { ... }     // Maps to .default
Task(priority: .low) { ... }        // Maps to .utility
Task(priority: .background) { ... } // Maps to .background
Task(priority: .userInitiated) { ... }

// Priority escalation example
let lowPriorityTask = Task(priority: .background) {
    await heavyComputation()  // Running at .background
}

// High-priority task awaits the low-priority one
Task(priority: .userInitiated) {
    let result = await lowPriorityTask.value
    // Runtime detects: .userInitiated waiting on .background
    // → Escalates lowPriorityTask to .userInitiated
    updateUI(result)
}
Deep Dive: Priority Escalation Mechanics

Priority escalation in Swift concurrency works through the executor. When a high-priority task calls await on a lower-priority task's value, the runtime walks the dependency chain and boosts the priority of all tasks in the chain. For actors, this means the actor's executor will schedule the boosted job ahead of other same-priority jobs. This is similar to GCD's priority inheritance but works at the task level rather than the thread level.

7. TaskLocal Values

TaskLocal values are the Swift concurrency equivalent of thread-local storage. They propagate automatically through the task hierarchy — child tasks inherit their parent's task-local values.

enum RequestContext {
    @TaskLocal static var requestID: String?
    @TaskLocal static var userID: String?
    @TaskLocal static var traceID: String?
}

// Setting task-local values
func handleRequest(_ request: Request) async {
    await RequestContext.$requestID.withValue(request.id) {
        await RequestContext.$traceID.withValue(UUID().uuidString) {
            // All code in this scope (and child tasks) sees these values
            await processRequest(request)
        }
    }
}

func processRequest(_ request: Request) async {
    // Task-local values are available here
    log("Processing request \(RequestContext.requestID ?? "unknown")")

    await withTaskGroup(of: Void.self) { group in
        group.addTask {
            // Child tasks inherit task-local values
            log("Child task sees: \(RequestContext.requestID ?? "none")")
        }
    }
}

// Task.detached does NOT inherit task-local values
Task.detached {
    log("Detached: \(RequestContext.requestID ?? "none")")  // "none"
}
Best Practice: TaskLocal for Distributed Tracing

TaskLocal values are ideal for propagating request context through a call chain — request IDs, trace IDs, user context, feature flags. They replace the thread-local storage pattern that was common with GCD. The key advantage: they propagate through async boundaries automatically, which thread-local storage cannot do (since async code can resume on any thread).

8. Sendable Protocol & @Sendable Closures

Sendable is a marker protocol that indicates a type is safe to share across concurrency domains (actors, tasks). It's the foundation of Swift's compile-time data race safety.

8.1 What Is Sendable?

// Types that are implicitly Sendable:
// - Value types (struct, enum) where all stored properties are Sendable
// - Actor types (isolated by definition)
// - Final classes with only let properties that are Sendable
// - Tuples of Sendable types
// - Metatypes (String.self)

struct UserData: Sendable {  // ✅ All properties are Sendable
    let name: String
    let age: Int
}

final class ImmutableConfig: Sendable {  // ✅ Final + all let + Sendable
    let apiKey: String
    let timeout: TimeInterval
}

class MutableState: Sendable {  // ❌ Compiler error: has mutable state
    var count = 0  // Not safe to share across tasks
}

// @unchecked Sendable: "trust me, I handle synchronization"
final class ThreadSafeCounter: @unchecked Sendable {
    private var _count = 0
    private let lock = NSLock()

    var count: Int {
        lock.lock()
        defer { lock.unlock() }
        return _count
    }

    func increment() {
        lock.lock()
        _count += 1
        lock.unlock()
    }
}

8.2 @Sendable Closures

// @Sendable closures cannot capture mutable state
var counter = 0

Task { @Sendable in
    counter += 1  // ❌ Error: mutation of captured var in @Sendable closure
}

// Correct: capture immutable value
let currentCount = counter
Task {
    print(currentCount)  // ✅ Captures a let binding
}

// @Sendable function types
func runInBackground(_ work: @Sendable @escaping () async -> Void) {
    Task.detached(operation: work)
}

9. Architect: Migrating from GCD to Structured Concurrency

Architect Must-Know

Migrating a large codebase from GCD to async/await is a multi-quarter effort that requires careful planning. Here's a battle-tested strategy used by teams at scale.

9.1 Migration Strategy: Bottom-Up

Migration Order (Bottom → Top): Phase 1: Leaf functions (no dependencies) ┌──────────────────────────────────────────────┐ │ Network layer: URLSession → async/await │ │ File I/O: Data(contentsOf:) → async │ │ Database: Core Data perform → async │ └──────────────────────────────────────────────┘ │ Phase 2: Service layer ┌──────────────────────────────────────────────┐ │ Repositories: callback → async │ │ Use cases: DispatchGroup → TaskGroup │ │ Caches: serial queue → actor │ └──────────────────────────────────────────────┘ │ Phase 3: Presentation layer ┌──────────────────────────────────────────────┐ │ ViewModels: Combine → @MainActor + async │ │ Coordinators: callback → async │ │ UI bindings: @Published → @Observable │ └──────────────────────────────────────────────┘

9.2 Bridge Patterns

// Pattern 1: Wrap callback API in async
extension LegacyNetworkClient {
    func fetch(url: URL) async throws -> Data {
        try await withCheckedThrowingContinuation { continuation in
            self.fetch(url: url) { result in
                continuation.resume(with: result)
            }
        }
    }
}

// Pattern 2: Expose async API with callback wrapper for legacy callers
class ModernService {
    func fetchData() async throws -> Data {
        // Modern implementation
    }

    // Legacy bridge for callers not yet migrated
    func fetchData(completion: @escaping (Result<Data, Error>) -> Void) {
        Task {
            do {
                let data = try await fetchData()
                completion(.success(data))
            } catch {
                completion(.failure(error))
            }
        }
    }
}

// Pattern 3: Replace serial queue with actor
// Before:
class CacheManager {
    private var cache: [String: Data] = [:]
    private let queue = DispatchQueue(label: "cache")

    func get(_ key: String) -> Data? {
        queue.sync { cache[key] }
    }
}

// After:
actor CacheManager {
    private var cache: [String: Data] = [:]

    func get(_ key: String) -> Data? {
        cache[key]
    }
}

9.3 Migration Checklist

Best Practice: Migration Checklist
  • Enable strict concurrency warnings first (-strict-concurrency=complete) to see the scope of changes needed
  • Migrate bottom-up: leaf functions first, then services, then presentation
  • Use withCheckedContinuation to bridge callback APIs — don't rewrite everything at once
  • Replace serial queues with actors — but beware of reentrancy (see next document)
  • Replace DispatchGroup with TaskGroup — structured concurrency handles cancellation automatically
  • Mark types as Sendable progressively — use @unchecked Sendable as a temporary bridge
  • Test thoroughly: async code has different timing characteristics than GCD code
  • Keep both APIs during transition: provide async + callback versions until all callers migrate
