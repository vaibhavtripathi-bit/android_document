Introduction

This series maps the substrate beneath UIKit and SwiftUI: the Darwin/XNU kernel and userspace services, the Objective-C runtime that still underpins much of Apple’s stack, the Swift runtime and ABI guarantees that affect binary compatibility and performance, and the platform frameworks that cut across every shipping app—push notifications, inter-process communication, accessibility, and localization.

Staff interviews rarely stop at API names. Interviewers probe boundaries (app vs. extension vs. daemon), ownership (who allocates, who invalidates), failure modes (Jetsam, XPC errors, notification delivery), and tradeoffs (runtime dynamism vs. stability, IPC patterns vs. latency). The goal here is not to memorize trivia, but to build narratives you can use when architecting features, debugging production issues, or comparing approaches across teams.

What You’ll Learn

Across the eight documents below, you will develop working knowledge of:

  • Darwin & XNU — Mach tasks and threads, virtual memory basics, launchd and daemons, and how userland services relate to what your process can see and do.
  • Objective-C runtime — messaging, method resolution, associated objects, swizzling (and when it is indefensible), and interaction with Swift bridging.
  • Swift runtime & ABI — metadata, witness tables, resilience, calling conventions, and implications for library evolution and binary size.
  • Notifications & push — APNs flow, notification service extensions, categories and actions, silent pushes, and reliability vs. battery tradeoffs.
  • Inter-process communication — XPC, Mach ports, URL schemes vs. universal links, app groups, and extension boundaries.
  • Accessibility — VoiceOver and the accessibility API surface, custom elements, focus order, and building inclusive features without bolting on a11y late.
  • Internationalization — Locale, formatting, RTL, string catalogs, and shipping localized products at scale.
  • Interview synthesis — 55+ Staff-level questions tying these threads together, with patterns for structuring answers under time pressure.

Prerequisites

Recommended Background

You will get the most from this series if you already have:

  • Solid iOS app development experience — view lifecycles, background modes, and common system frameworks (Foundation, UIKit or SwiftUI).
  • Comfort with Swift and Objective-C interop — bridging headers, @objc, and where dynamism still appears in production APIs.
  • Basic concurrency and memory context — main-thread rules, ARC ownership, and why “crash in extension” or “Jetsam” are different failure classes.
  • Familiarity with prior series in this prep set — especially memory/performance and rendering/graphics when topics touch Jetsam, IPC to the render server, or process boundaries.

If a topic feels opaque, skim the linked document’s introduction first, then circle back after reading the prerequisite bullet that matches your gap.

How to Use This Series

Study Tips
  1. Read for invariants, not flashcards. Note what the kernel, runtime, or service guarantees versus what is implementation detail—interviewers reward that distinction.
  2. Draw boundaries. For each major idea (push delivery, XPC, sandbox), sketch which process owns which resource and what happens when it dies or is suspended.
  3. Pair each chapter with one war story. Even hypothetical: “We missed a silent push edge case because…” anchors abstraction to behavior.
  4. Cross-link to UI and product. Ask how VoiceOver order or RTL would change your navigation model—Staff loops often connect platform to UX.
  5. Use the Q&A document actively. Answer aloud with a timer, then read the write-up; mark only the gaps that surprised you for review.
  6. Schedule spaced review. Runtime and IPC details fade quickly; a second pass a week later pays disproportionate dividends.

Why This Matters at Staff Level

Architect Perspective

At L6/Staff, you are expected to own decisions that span teams and releases: notification strategy for a new surface, extension architecture for a feature that must stay sandbox-safe, or a Swift package layout that will ship inside multiple apps. Those choices sit on top of platform contracts—ABI stability, process isolation, push policies, and accessibility expectations enforced by App Review and public trust.

Interviewers use system-framework depth as a signal for risk awareness: can you anticipate Jetsam under memory pressure, explain why an XPC helper failed closed, or argue for incremental adoption of Swift without breaking binary compatibility? Fluency here turns “I would use the API” into “I would shape the system this way, and here is what breaks if we do not.” That is the bar this series is written for.
