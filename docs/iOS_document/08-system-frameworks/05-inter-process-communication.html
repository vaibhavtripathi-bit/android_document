Deep Dive — Why IPC Dominates Staff Conversations

On iOS, every meaningful interaction outside your process—Keychain, Photo Library, extensions, Siri intents, background downloads—crosses a trust boundary. Interviewers want you to name the mechanism, describe serialization and lifecycle, and reason about sandboxing, entitlements, and what happens when the remote side dies mid-flight.

1. XPC Services & NSXPCConnection

XPC (Cross-Process Communication) is Apple’s structured IPC layer. Conceptually it sits above raw Mach messaging: connections are authenticated, messages are typed, and the runtime can enforce interface contracts. NSXPCConnection exposes this to Cocoa as an object-oriented API built on Objective-C protocols and proxies.

┌──────────────────────────────────────────────────────────────────────────┐ │ NSXPCConnection lifecycle │ ├──────────────────────────────────────────────────────────────────────────┤ │ │ │ Client (e.g. app) Service (XPC service / helper) │ │ │ │ │ │ │ 1. Create connection │ │ │ │ (named Mach service or │ │ │ │ anonymous listener) │ │ │ ├──────────────────────────────────────►│ │ │ │ 2. Mach-level handshake + │ │ │ │ audit token / code signature │ │ │ │◄──────────────────────────────────────┤ │ │ │ 3. Export remoteObjectInterface │ │ │ │ (client → server protocol) │ │ │ │ remoteObjectProxy │ │ │ ├ ─ ─ ─ async messages ─ ─ ─ ─ ─ ─ ─ ─►│ │ │ │◄─ ─ ─ replies / errors ─ ─ ─ ─ ─ ─ ─ ─┤ │ │ │ 4. interruptionHandler / │ │ │ │ invalidationHandler │ │ │ ▼ ▼ │ │ Connection invalidated on crash, Service may be │ │ jetsam, or explicit invalidate() relaunched by launchd │ └──────────────────────────────────────────────────────────────────────────┘

Exported interface (protocol)

The server advertises an object that conforms to a protocol marked for XPC. The client receives a proxy that implements the same protocol; method calls are asynchronous unless you wrap them explicitly. Only types that XPC can serialize cross the boundary: primitives, collections, NSData, NSURL, NSUUID, NSDate, and a small set of toll-free bridged / security types, plus object graphs that satisfy NSSecureCoding when configured.

Pitfall — NSSecureCoding & Secure Coding Classes

Staff-level answers should mention class allowlists: without secure coding, a compromised or buggy peer could deserialize attacker-controlled class names. Modern XPC interfaces prefer explicit secure coding on both sides and minimal surface area on the protocol.

import Foundation

// Server-side object must inherit NSObject and use @objc protocol.
@objc protocol ImageProcessorXPC {
    process(data: Data, reply: @escaping (Data?, Error?) -> Void)
}

final class ImageProcessorService: NSObject, ImageProcessorXPC {
    func process(data: Data, reply: @escaping (Data?, Error?) -> Void) {
        // Heavy work on service queue — never block the main thread.
        DispatchQueue.global(qos: .userInitiated).async {
            let out = data /* transform */
            reply(out, nil)
        }
    }
}

// Client: NSXPCConnection(serviceName:) for embedded XPC services.
func makeClientConnection() -> NSXPCConnection {
    let conn = NSXPCConnection(serviceName: "com.example.app.ImageProcessor")
    conn.remoteObjectInterface = NSXPCInterface(with: ImageProcessorXPC.self)
    conn.invalidationHandler = {
        // Permanent: service gone, connection unusable.
    }
    conn.interruptionHandler = {
        // Transient: may reconnect (e.g. service restarted).
    }
    conn.resume()
    return conn
}

func invokeRemote(connection: NSXPCConnection, payload: Data) {
    let proxy = connection.remoteObjectProxyWithErrorHandler { err in
        // Delivery or execution failed on remote side.
    } as? ImageProcessorXPC
    proxy?.process(data: payload) { result, err in
        // Handle on calling queue unless you hop explicitly.
    }
}

Connection lifecycle

  • Creation — Named service (init(serviceName:)), Mach service name, or listener for anonymous endpoints.
  • resume() — Starts the connection; until resumed, no messages flow.
  • invalidate() — Tears down the connection; proxies become invalid.

Invalidation vs interruption

The interruption handler runs when the remote side disappears in a way that might be recoverable—think transient crashes or service recycling. The invalidation handler runs when the connection is definitively dead from the client’s perspective. In practice you implement both: interruption for “maybe recreate connection,” invalidation for “drop state and surface error.”

Security: entitlements & audit token

XPC peers are not arbitrary processes: the kernel and launchd participate in matching service names to signed bundles and entitlements. The server can inspect the client’s audit token (and code signing identity) to authorize operations—this is how system daemons reject unprivileged callers. Your own helper should validate expected bundle IDs and team IDs when exposing sensitive work.

XPC in app extensions

Extensions often host or consume XPC to talk to their containing app’s helper services. The same rules apply: minimal protocols, explicit error handling, and awareness that extensions have tighter memory and CPU budgets—large payloads should be file-backed in a shared container rather than inline Data.

Anonymous vs named connections

Named connections resolve through a well-known Mach service name registered with the system (your XPC service bundle). Anonymous connections use listener endpoints created at runtime; the server passes a capability to the client through a side channel (often a file descriptor or inherited XPC endpoint). Named is simpler for app-embedded services; anonymous supports tighter coupling when both ends are under your control.

Best Practice — Protocol Shape

Keep XPC protocols small, prefer async reply blocks over synchronous RPC, and use file URLs in a shared group for bulk data. Staff interviewers reward this as “understands Jetsam and message size limits.”

2. Mach Ports for IPC

At the Darwin layer, a Mach port is a kernel-managed right to send or receive messages. Everything from XPC to parts of IOKit ultimately reduces to mach_msg() and port rights. Understanding ports explains why “IPC” on Apple platforms is not sockets-first.

Port rights

  • Receive right — Owns a queue; only one holder per port.
  • Send right — Can enqueue messages to that port.
  • Send-once right — A single-use send right, consumed on first successful send—used for replies.

Port sets

A port set multiplexes multiple receive rights: mach_msg can block on the set, which is how run loops and dispatch sources wait on many channels. Higher frameworks hide this, but the model is “wait on many receive queues.”

Mach message layout

A Mach message has a fixed header (ports, size, id, complexity), a descriptor section for out-of-line memory and port attachments, and an inline body. Port descriptors transfer send rights to another task atomically—this is how capabilities move between processes.

Out-of-line (OOL) data

Large buffers can be sent as OOL memory regions referenced by descriptors rather than copied inline, reducing copies for bulk transfers—subject to policy and sandbox checks. XPC builds structured, typed messages on top of this substrate.

Port lifecycle

Ports are reference-counted kernel objects. Dropping the last send right while no messages are queued can destroy the port; receive rights must be deallocated explicitly or leak kernel resources. Bugs here manifest as port leaks in long-running daemons.

How CFMessagePort and XPC relate

CFMessagePort wraps Mach ports with a run-loop source. XPC uses Mach messaging internally but adds authentication, type information, object serialization, and service management via launchd.

Pitfall — Raw mach_* in App Store Apps

Direct use of mach_port_allocate / mach_msg in third-party iOS apps is rare, tightly sandboxed, and easy to get wrong. Staff answers should emphasize: prefer NSXPC / higher APIs; cite Mach when explaining why XPC is fast and kernel-mediated.

// Illustrative C — not a template for production iOS app code.

3. CFMessagePort

CFMessagePort is Core Foundation IPC over Mach ports, usually paired with a local name (string registered in the bootstrap namespace) or a remote port discovered by name. Callbacks integrate with CFRunLoop.

Local vs remote

  • Local — Creates a receive port and registers a name; others create a remote port referencing that name.
  • Remote — Sends CFMessagePortSendRequest to deliver a buffer synchronously (with timeout).

Synchronous messaging & callbacks

CFMessagePortSendRequest blocks until a reply or timeout. Alternatively, install a callback on the local port for asynchronous handling. This is simpler than XPC but lacks rich object graphs and secure typing.

Limitations on iOS

Sandboxed App Store applications cannot use CFMessagePort for arbitrary peer-to-peer IPC the way macOS daemons might. Practical iOS use is narrow—often legacy, internal, or unavailable between unrelated apps. Interviewers want you to state plainly: prefer App Groups + XPC / file coordination / NSFileCoordinator for sanctioned cross-target communication.

Deep Dive — Same User, Bootstrap Namespace

Historically, CFMessagePort names were resolved through bootstrap (launchd). On iOS, the sandbox prevents most interesting registrations between unrelated vendors. When CFMessagePort is “still useful,” it is typically within tightly controlled environments (simulators, tests, or specific internal tools)—not as a general inter-app bus.

import CoreFoundation

let portName = "com.example.internal.port" as CFString

// Local listener (conceptual — often not viable sandbox-to-sandbox).
var context = CFMessagePortContext(version: 0, info: nil, retain: nil, release: nil, copyDescription: nil)

let callback: CFMessagePortCallBack = { _, msgid, data, info in
    // Handle message id + CFDataRef payload
    return nil // reply data as Unmanaged<CFData>?
}

if let local = CFMessagePortCreateLocal(nil, portName, callback, &context, nil) {
    let source = CFMessagePortCreateRunLoopSource(nil, local, 0)
    CFRunLoopAddSource(CFRunLoopGetMain(), source, CFRunLoopMode.defaultMode)
}

// Remote synchronous send (returns Data?)
if let remote = CFMessagePortCreateRemote(nil, portName) {
    var returnData: Unmanaged<CFData>?
    let status = CFMessagePortSendRequest(
        remote,
        1001,
        "ping".data(using: .utf8)! as CFData,
        1.0, 1.0, 0,
        &returnData
    )
    _ = status
}

4. App Groups for Data Sharing

App Groups carve out a shared filesystem container for executables that declare the same group identifier and are signed with the same team. This is the sanctioned way to share files and preferences between an app and its extensions.

Entitlement & provisioning

Add the App Groups capability in Xcode for each target; the identifier looks like group.com.company.product. Profiles must include the group or installation fails.

Container URL

import Foundation

let groupId = "group.com.example.shared"
let folder = FileManager.default.containerURL(
    forSecurityApplicationGroupIdentifier: groupId
)
let sharedDB = folder?.appendingPathComponent("Library/Application Support/Model.sqlite")

Sharing with extensions

Widgets, share extensions, and notification service extensions read the same SQLite store or exported files. Use atomic writes (write to temp, replaceItemAt) to avoid torn reads.

Shared Core Data / SQLite

Point persistent store URLs at the group container. Use a single writer model or SQLite WAL with careful coordination; Core Data can use NSPersistentContainer with a custom store URL.

NSFileCoordinator

When multiple processes may touch the same file, NSFileCoordinator and NSFilePresenter coordinate reads/writes and broadcast changes—critical for avoiding corruption when the app and extension are both alive.

Best Practice — Single Source of Truth

Treat the group container as a contract: version your schema, document file layout, and handle “extension runs while app suspended” as the normal case—not an edge case.

import Foundation

func coordinateRead(url: URL, completion: @escaping (Data?) -> Void) {
    let coordinator = NSFileCoordinator()
    var error: NSError?
    coordinator.coordinate(readingItemAt: url, options: [], error: &error) { readURL in
        completion(try? Data(contentsOf: readURL))
    }
    if let error = error {
        completion(nil)
    }
}

5. UserDefaults(suiteName:)

Standard UserDefaults.standard is per-process (per-target). UserDefaults(suiteName:) backed by an App Group plist lives in the shared container, so the app and extensions see the same keys.

import Foundation

let shared = UserDefaults(suiteName: "group.com.example.shared")!
shared.set("dark", forKey: "theme")
shared.synchronize() // Usually unnecessary; flush is asynchronous by design.

let theme = shared.string(forKey: "theme")

Synchronization considerations

There is no cross-process KVO on UserDefaults. Another process may change plist-backed keys while you hold a cached instance; rely on Darwin notifications, polling on UIApplication foreground, or explicit handshake files.

Pitfall — Assuming Instant Consistency

Heavy write churn to shared defaults can be slower than you expect and is a poor substitute for a real datastore. Large blobs belong in files or SQLite, not defaults.

Darwin notifications for change signals

Post a Darwin notification (signal-only) after committing important defaults so extensions wake to refresh UI. See §11 Darwin Notifications.

6. Keychain Sharing

Keychain items are isolated by default. Keychain access groups let multiple executables from the same developer team share credentials when entitlements and provisioning align.

kSecAttrAccessGroup

import Security

let group = "TEAMID.com.example.sharedkeychain"
let tokenData = "opaque-token".data(using: .utf8)!
let query: [String: Any] = [
    kSecClass as String: kSecClassGenericPassword,
    kSecAttrAccount as String: "session",
    kSecAttrAccessGroup as String: group,
    kSecValueData as String: tokenData,
    kSecAttrAccessible as String: kSecAttrAccessibleAfterFirstUnlock
]

SecItemAdd(query as CFDictionary, nil)

Keychain sharing vs App Groups

They solve different problems: App Groups share files and defaults; keychain groups share small secrets with hardware-backed protection. You often use both: tokens in Keychain, bulk state in the group container.

Deep Dive — Access Group Prefix

Xcode may prepend the team ID automatically in entitlements. Mismatched strings between targets are a top cause of “works in app, fails in extension” bugs.

7. UIDocumentInteractionController

UIDocumentInteractionController previews documents and offers “Open in …” style actions to other apps that declare compatible UTIs. It is part of the document interaction family that predates some modern sharing APIs but remains relevant for file-centric workflows.

UTI system

Uniform Type Identifiers describe file semantics (public.png, com.adobe.pdf). Declare imported/exported types in Info.plist so the system can route documents to your app.

Preview & options

import UIKit

final class PreviewHost: UIViewController, UIDocumentInteractionControllerDelegate {
    var docController: UIDocumentInteractionController?

    func presentPreview(fileURL: URL) {
        docController = UIDocumentInteractionController(url: fileURL)
        docController?.delegate = self
        docController?.presentPreview(animated: true)
    }
}

Modern alternatives

For many flows, UIActivityViewController or drag-and-drop are more appropriate. For in-place editing, UIDocumentPickerViewController / security-scoped bookmarks matter more than document interaction.

Pitfall — Security-Scoped URLs

Files outside your sandbox need appropriate access: open via picker, use security-scoped bookmarks, and stop access when finished.

8. UIActivityViewController

The share sheet presents activities (AirDrop, Mail, custom share extensions) for arbitrary items described by UIActivityItemSource or plain objects.

import UIKit
import UniformTypeIdentifiers

func presentShareSheet(from vc: UIViewController, text: String, image: UIImage) {
    let items: [Any] = [text, image]
    let avc = UIActivityViewController(activityItems: items, applicationActivities: nil)
    avc.excludedActivityTypes = [.assignToContact]
    vc.present(avc, animated: true)
}

NSItemProvider & rich data

import UniformTypeIdentifiers

let provider = NSItemProvider(item: image, typeIdentifier: UTType.image.identifier)
provider.registerDataRepresentation(for: .png, visibility: .all) { completion in
    completion(image.pngData(), nil)
    return nil
}

Custom UIActivity

Subclass UIActivity for in-app actions that should appear beside system activities—useful for branded workflows that are not full share extensions.

Best Practice — Lazy Providers

Large exports should load on demand via NSItemProvider registration blocks so the share sheet opens instantly.

9. URL Schemes

Custom URL schemes (myapp://) register your app as a handler for links. They are simple but coarse: any app can attempt to open them, and users cannot see hostname-level trust like HTTPS.

Info.plist registration

<!-- CFBundleURLTypes -->
<key>CFBundleURLTypes</key>
<array>
  <dict>
    <key>CFBundleURLSchemes</key>
    <array><string>myapp</string></array>
  </dict>
</array>

canOpenURL & LSApplicationQueriesSchemes

iOS restricts probing for other apps’ schemes. Declare allowed schemes under LSApplicationQueriesSchemes or canOpenURL returns false—even if the app is installed.

import UIKit

let url = URL(string: "myapp://open/profile?id=42")!
if UIApplication.shared.canOpenURL(url) {
    UIApplication.shared.open(url)
}

Security: scheme hijacking

Malicious apps could register the same scheme on jailbroken or misconfigured devices; on stock iOS, collisions are rarer but universal links are preferred for domain-bound deep links.

10. Universal Links

Universal Links associate HTTPS URLs with your app via the Associated Domains entitlement and an apple-app-site-association (AASA) JSON file served by your domain.

┌─────────────────────────────────────────────────────────────────────────┐ │ Universal Link resolution (simplified) │ ├─────────────────────────────────────────────────────────────────────────┤ │ │ │ User taps https://app.example.com/item/7 │ │ │ │ │ ▼ │ │ iOS fetches AASA from https://app.example.com/.well-known/ │ │ apple-app-site-association (or root path; CDN caching applies) │ │ │ │ │ ▼ │ │ Validate JSON: appIDs, paths, components │ │ │ │ │ ┌────┴────┐ │ │ │ │ │ │ Match No match │ │ │ │ │ │ ▼ ▼ │ │ Open app Fall through to Safari (or continue user activity) │ │ (handoff to scene / app delegate) │ └─────────────────────────────────────────────────────────────────────────┘

AASA essentials

  • Served over HTTPS with a valid certificate.
  • application/json or application/pkcs7-mime (signed) formats per Apple docs.
  • Contains applinks details including appID (TEAMID.bundleid) and path rules.

Entitlement

// Associated Domains capability
applinks:app.example.com

CDN & caching

Apple caches AASA aggressively. Use correct Content-Type, avoid redirects when possible, and plan for propagation delay after changes—this is a common production footgun.

Debugging

Use the Associated Domains Development workflow, swcd logs, and verify the AASA with curl -v. Mismatched team ID or bundle ID in appID is the #1 misconfiguration.

Architect Must Know — Universal Links vs URL Schemes

Universal links require domain control, survive app uninstall fallbacks to web, and avoid naked scheme conflicts. They integrate with NSUserActivity and scene continuation. Schemes remain useful for narrow in-app triggers but are weak for identity-bound navigation.

import UIKit

func application(_: UIApplication,
                   continue userActivity: NSUserActivity,
                   restorationHandler: @escaping ([UIUserActivityRestoring]?) -> Void) -> Bool {
    if userActivity.activityType == NSUserActivityTypeBrowsingWeb,
       let url = userActivity.webpageURL {
        // Route url.pathComponents, restore state
        return true
    }
    return false
}

11. Darwin Notifications

CFNotificationCenterGetDarwinNotifyCenter broadcasts signal-only events system-wide to interested observers. There is no payload—only the notification name. Pair it with shared files, defaults, or Keychain updates.

import Foundation

let name = "com.example.shared.refresh" as CFString

// Post (from app or extension)
CFNotificationCenterPostNotification(
    CFNotificationCenterGetDarwinNotifyCenter(),
    CFNotificationName(name),
    nil,
    nil,
    true
)

// Observe
let observer = UnsafeMutableRawPointer(Unmanaged.passUnretained(self).toOpaque())

CFNotificationCenterAddObserver(
    CFNotificationCenterGetDarwinNotifyCenter(),
    observer,
    { _, _, _, _, _ in
        // Read updated state from App Group container
    },
    name,
    nil,
    .deliverImmediately
)

notify_post / notify_register_dispatch

The lower-level notify(3) API registers string tokens and can deliver to dispatch queues—useful in daemons and some system-adjacent code. Same constraint: notifications are tiny signals, not data buses.

Pitfall — Thundering Herd & Privacy

Every observer wakes and must re-read shared state. Names are global strings—choose unique reverse-DNS tokens to avoid accidental collisions with other processes.

12. Choosing the Right IPC Mechanism

Architect Must Know — Decision Checklist
  1. Data size — Inline RPC vs file handoff vs database in group container.
  2. Security — Secrets in Keychain; never in URL schemes; universal links for verified navigation.
  3. Latency — XPC for interactive helpers; Darwin notify for wake signals; avoid synchronous CFMessagePort on UI.
  4. Directionality — One-way signals (Darwin), request/reply (XPC), shared mutable state (files + coordination).
  5. Process relationship — Same team App Group + Keychain vs arbitrary apps (share sheet / open URL only).
Best Practice — Defense in Depth

Combine mechanisms: store authoritative state in SQLite under App Group, notify via Darwin, expose privileged operations only through an XPC service that checks caller identity, and deep-link users with universal links—not opaque custom schemes—for anything security-sensitive.
