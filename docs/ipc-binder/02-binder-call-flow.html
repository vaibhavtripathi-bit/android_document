02 — Binder Call Flow

End-to-end trace of a Binder RPC — from Proxy to Stub, Parcel serialization, thread pool dispatch, synchronous vs oneway semantics, and death notifications

1. The IBinder Hierarchy — Java/Kotlin Layer

// The Java Binder framework mirrors the kernel structures:

// IBinder (interface)
//   ├── Binder (local implementation — "BBinder" in native)
//   │     └── Your AIDL Stub extends this
//   └── BinderProxy (remote handle — "BpBinder" in native)
//         └── Your AIDL Proxy wraps this

// Key IBinder methods:
interface IBinder {
    fun transact(code: Int, data: Parcel, reply: Parcel?, flags: Int): Boolean
    fun linkToDeath(recipient: DeathRecipient, flags: Int)
    fun unlinkToDeath(recipient: DeathRecipient, flags: Int): Boolean
    fun isBinderAlive(): Boolean
    fun getInterfaceDescriptor(): String?
    fun pingBinder(): Boolean
}

// Binder (local — runs in the server process):
open class Binder : IBinder {
    // transact() calls onTransact() — your implementation
    override fun transact(code: Int, data: Parcel, reply: Parcel?, flags: Int): Boolean {
        // ... validation ...
        return onTransact(code, data, reply, flags) // Dispatches to your code
    }

    // Override this in your Stub:
    protected open fun onTransact(code: Int, data: Parcel, reply: Parcel?, flags: Int): Boolean
}

// BinderProxy (remote — runs in the client process):
// - Created by the framework when you receive an IBinder from another process
// - transact() calls into native BpBinder::transact() → ioctl(BINDER_WRITE_READ)
// - You never create BinderProxy directly; the framework does it

3. Parcel — The Serialization Format

// Parcel is Binder's serialization container. It's NOT a general-purpose
// serialization format — it's designed specifically for IPC.

// Parcel is a flat buffer of typed data:
val parcel = Parcel.obtain()

// Primitives (written directly, no overhead):
parcel.writeInt(42)
parcel.writeLong(123456789L)
parcel.writeFloat(3.14f)
parcel.writeDouble(2.718)
parcel.writeString("hello")       // UTF-16 encoded, length-prefixed
parcel.writeByteArray(byteArrayOf(1, 2, 3))

// Parcelable objects (custom serialization):
parcel.writeParcelable(myObject, 0)
// Calls myObject.writeToParcel(parcel, flags)

// IBinder references (translated by kernel):
parcel.writeStrongBinder(myBinder)
// Kernel translates: local BBinder → binder_node → binder_ref for receiver

// File descriptors (dup'd by kernel):
parcel.writeFileDescriptor(fd)
// Kernel dups the fd into the target process

// CRITICAL: Parcel is NOT safe for persistent storage
// - Format is not versioned
// - Depends on platform implementation
// - Use only for IPC, never for disk/network
Warning — Parcel Size Limits

The Binder transaction buffer is ~1MB per process, shared across all concurrent transactions. A single transaction that exceeds the remaining buffer space throws TransactionTooLargeException. In practice, keep individual transactions under 100KB. For large data, use: (1) ashmem / SharedMemory for bulk byte arrays, (2) ContentProvider with a cursor for large result sets, (3) file-based transfer with a content URI.

4. Binder Thread Pool

// Each process that uses Binder has a thread pool for handling incoming transactions.

// Pool configuration (native, ProcessState.cpp):
// - Maximum threads: 15 (BINDER_DEFAULT_MAX_THREADS, set via ioctl)
//   + 1 main Binder thread = 16 total threads that can handle transactions
// - Threads are spawned on demand by the kernel (BR_SPAWN_LOOPER)
// - Once spawned, threads loop in IPCThreadState::joinThreadPool()

// Thread lifecycle:
// 1. Process starts → main thread calls startThreadPool()
//    → spawns one thread that calls joinThreadPool()
// 2. Incoming transaction arrives, no free thread →
//    kernel sends BR_SPAWN_LOOPER to the process
// 3. Process spawns a new thread (up to max)
// 4. Thread enters joinThreadPool() → loops on ioctl(BINDER_WRITE_READ)
// 5. Thread receives BR_TRANSACTION → executes onTransact()
// 6. Thread sends BC_REPLY → goes back to waiting

// IMPORTANT: These are NOT your app's worker threads.
// Binder threads are managed by the framework, separate from:
// - Main/UI thread (Looper thread)
// - Coroutine dispatchers
// - ExecutorService threads

// When a Binder call arrives, it runs on a BINDER THREAD.
// If you need to touch the UI, you must post to the main thread:
// Handler(Looper.getMainLooper()).post { updateUI() }
  App Process Thread Pool
  ┌─────────────────────────────────────────────────┐
  │                                                   │
  │  Main Thread (UI)          ← Looper/Handler       │
  │  ─────────────                                    │
  │                                                   │
  │  Binder Thread 1  ← joinThreadPool() ← waiting   │
  │  Binder Thread 2  ← joinThreadPool() ← handling  │
  │  Binder Thread 3  ← joinThreadPool() ← waiting   │
  │  ...                                              │
  │  Binder Thread 15 ← joinThreadPool() ← waiting   │
  │  + Main Binder    ← startThreadPool()             │
  │  ─────────────────                                │
  │  Total: up to 16 Binder threads                   │
  │                                                   │
  │  Worker threads (coroutines, executors) ← yours   │
  │  ─────────────────────────────────                │
  │                                                   │
  └─────────────────────────────────────────────────┘
Staff-Level Insight — system_server Thread Pool

system_server increases its Binder thread pool to 31 (via BinderInternal.setMaxThreads(31)) because it handles transactions from ALL apps simultaneously. Even so, under heavy load (e.g., many apps starting at boot), the pool can be exhausted, causing Binder calls to block. This is one reason cold boot is slow — apps compete for system_server's Binder threads. Android optimizes this with transaction batching and oneway calls where possible.

5. Synchronous vs Oneway Transactions

// Synchronous (default):
// - Client thread BLOCKS until server processes and replies
// - Client sleeps in kernel (wait_event_interruptible)
// - Server's onTransact() runs, writes reply, sends BC_REPLY
// - Client wakes up, reads reply
// - Guarantees ordering: call A completes before call B starts (per-thread)

// Oneway (FLAG_ONEWAY / IBinder.FLAG_ONEWAY):
// - Client does NOT block — returns immediately after kernel accepts
// - No reply Parcel (reply parameter is null)
// - Server processes asynchronously
// - Transactions are queued and processed in order (per binder_node)
// - Used for: callbacks, notifications, fire-and-forget operations

// AIDL syntax:
// oneway interface ICallback {
//     void onResult(in String data);  // All methods are oneway
// }
// OR per-method:
// interface IService {
//     String getData();               // Synchronous
//     oneway void notifyChange();     // Oneway
// }

// Oneway gotchas:
// 1. No return value — can't get a result
// 2. No exception propagation — server crash is silent
// 3. Transactions can queue up — if server is slow, buffer fills
// 4. Ordering is guaranteed per-node, but NOT across different nodes
// 5. If the target process dies, queued oneway transactions are lost
Aspect Synchronous Oneway
Client blocks? Yes — until reply No — returns immediately
Reply Parcel? Yes No (null)
Exception propagation? Yes (RemoteException) No
Ordering guarantee? Sequential per-thread FIFO per-node
Buffer pressure? Low (one at a time) High (can queue many)
Use case getData(), query() Callbacks, events

6. Death Notifications — DeathRecipient

// When a remote process dies, all its binder_nodes are destroyed.
// Clients holding references get notified via DeathRecipient.

// Registration:
val deathRecipient = IBinder.DeathRecipient {
    // Called on a Binder thread (NOT the main thread)
    Log.w(TAG, "Remote service died — cleaning up")
    cleanupConnection()
    scheduleReconnect()
}

// Link to the remote binder:
remoteBinder.linkToDeath(deathRecipient, 0)

// Unlink when done (e.g., in onServiceDisconnected or cleanup):
remoteBinder.unlinkToDeath(deathRecipient, 0)

// How it works internally:
// 1. linkToDeath() → BC_REQUEST_DEATH_NOTIFICATION to kernel
// 2. Kernel adds a binder_ref_death to the binder_ref
// 3. When target process exits → binder_release() frees binder_proc
// 4. For each binder_node: iterate all refs, find death notifications
// 5. Send BR_DEAD_BINDER to each registered process
// 6. Framework calls DeathRecipient.binderDied() on a Binder thread

// ServiceConnection also uses this internally:
// onServiceDisconnected() is triggered by the death notification
// But onServiceDisconnected ≠ binderDied:
// - onServiceDisconnected: called on main thread, after framework cleanup
// - binderDied: called on Binder thread, immediately when death detected
Staff-Level Insight — Death Race Condition

There's a subtle race: you receive an IBinder from onServiceConnected() and call linkToDeath(). But the remote process could have died between the bind and your linkToDeath() call. Defense: always check binder.pingBinder() after linking, or wrap the entire setup in a try-catch for RemoteException. Also, binderDied() is called on a Binder thread — if you update UI or shared state, you need synchronization.

7. ServiceManager — The Binder Name Registry

// ServiceManager is the central registry for system services.
// It runs as a native daemon (not Java) and holds handle 0.

// Registration (system_server only):
// ServiceManager.addService("activity", activityManagerService)
//
// Internally:
// 1. system_server calls ServiceManager.addService(name, binder)
// 2. This calls IServiceManager.Stub.Proxy.addService()
// 3. Which calls transact() on handle 0 (ServiceManager's BinderProxy)
// 4. ServiceManager receives BR_TRANSACTION
// 5. Stores name → IBinder mapping in a flat list
// 6. SELinux check: only allowed contexts can register specific services

// Lookup (any process):
// val binder = ServiceManager.getService("activity")
//
// Internally:
// 1. App calls ServiceManager.getService("activity")
// 2. Transact to handle 0 with the service name
// 3. ServiceManager looks up the name → returns the IBinder
// 4. Kernel translates: ServiceManager's binder_ref → new binder_ref for caller
// 5. Caller gets a BinderProxy wrapping the handle

// ServiceManager.waitForService(name):
// Blocks until the service is registered (used during boot)

// Caching:
// ServiceManagerNative caches lookups in a HashMap.
// SystemServiceRegistry (in the framework) caches even more aggressively:
// getSystemService("activity") → cached ActivityManager wrapper object
// The IBinder is fetched once, then the wrapper is reused.

8. Binder Context Propagation

// When a Binder call arrives, the framework provides context about the caller:

// On the server side (inside onTransact or AIDL implementation):
val callerUid = Binder.getCallingUid()    // UID of the calling process
val callerPid = Binder.getCallingPid()    // PID of the calling process

// These are set by the KERNEL (from current->cred) — unforgeable.

// Clearing identity for privileged operations:
fun handleClientRequest() {
    val callerUid = Binder.getCallingUid()
    // Check caller's permission first:
    enforceCallingPermission("com.example.PERMISSION", "Denied")

    // Now do work on behalf of the caller using OUR privileges:
    val token = Binder.clearCallingIdentity()
    try {
        // getCallingUid() now returns OUR UID (system_server's UID)
        // This lets us call other system services without the caller's restrictions
        performPrivilegedWork()
    } finally {
        Binder.restoreCallingIdentity(token) // ALWAYS restore in finally
    }
}

// How clearCallingIdentity works:
// - Saves current (caller's) UID/PID in a long (high 32 = PID, low 32 = UID)
// - Sets calling UID/PID to this process's own UID/PID
// - restoreCallingIdentity() reverses it
// - Thread-local storage — only affects the current thread

// CRITICAL: forgetting restoreCallingIdentity is a privilege escalation bug.
// Subsequent permission checks would use the server's UID instead of the caller's.
