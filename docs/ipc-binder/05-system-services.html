05 — System Services Architecture

How system_server boots, registers 100+ services, and how apps interact with AMS, WMS, PMS through Binder — plus the HAL layer beneath

1. system_server — The Heart of Android

// system_server is a single process (UID 1000) that hosts ALL core system services.
// It's the first child forked from Zygote during boot.

// Source: frameworks/base/services/java/com/android/server/SystemServer.java

// Boot sequence:
// ZygoteInit.forkSystemServer()
//   → Zygote.forkSystemServer(uid=1000, gid=1000, ...)
//   → fork() + specialize
//   → SystemServer.main()
//     → SystemServer.run()
//       → createSystemContext()        // Create the system's Context
//       → startBootstrapServices()     // Critical services (AMS, PMS, etc.)
//       → startCoreServices()          // BatteryService, UsageStatsService, etc.
//       → startOtherServices()         // Everything else (~80+ services)
//       → Looper.loop()               // Enter the main message loop

// Service start order matters — dependencies:
// Bootstrap (must start first):
//   Installer → DeviceIdentifiersPolicyService → UriGrantsManagerService
//   → ActivityTaskManagerService → ActivityManagerService
//   → PowerManagerService → RecoverySystemService
//   → PackageManagerService → UserManagerService
//   → OverlayManagerService

// Core:
//   BatteryService → UsageStatsService → WebViewUpdateService

// Other (after boot):
//   WindowManagerService → InputManagerService → AlarmManagerService
//   → NetworkManagementService → ConnectivityService → NotificationManagerService
//   → LocationManagerService → ... (~80 more)

3. getSystemService() — The App-Side Lookup

// When an app calls context.getSystemService(Context.ACTIVITY_SERVICE):

// Step 1: SystemServiceRegistry (client-side cache)
// Source: frameworks/base/core/java/android/app/SystemServiceRegistry.java
//
// Static initializer registers ALL service factories:
// registerService(Context.ACTIVITY_SERVICE, ActivityManager::class.java) {
//     context -> ActivityManager(context, /* handler */)
// }
//
// The factory creates a WRAPPER object (e.g., ActivityManager)
// that holds a cached Binder proxy to the real service.

// Step 2: First call triggers Binder lookup
// ActivityManager constructor:
//   IBinder b = ServiceManager.getService("activity")
//   IActivityManager service = IActivityManager.Stub.asInterface(b)
//   // This BinderProxy is cached — subsequent calls reuse it

// Step 3: Wrapper methods delegate to the Binder proxy
// activityManager.getRunningAppProcesses()
//   → IActivityManager.Stub.Proxy.getRunningAppProcesses()
//     → transact(TRANSACTION_getRunningAppProcesses, ...)
//       → ioctl(BINDER_WRITE_READ)
//         → system_server handles on a Binder thread
//           → AMS.getRunningAppProcesses()
//             → reply with List

// Caching layers:
// 1. SystemServiceRegistry: caches the wrapper object per-Context
// 2. ServiceManager: caches the IBinder per service name
// 3. BinderProxy: cached in the framework (BinderProxy.getInstance())
// Result: getSystemService() is cheap after the first call

4. AMS Deep Dive — Process & Activity Management

// ActivityManagerService is the most complex system service.
// Source: frameworks/base/services/core/java/com/android/server/am/

// Key responsibilities:
// 1. Process management: start, stop, OOM adjustment
// 2. Broadcast dispatch: ordered, sticky, permission-checked
// 3. Content provider lifecycle
// 4. Service lifecycle (startService, bindService)
// 5. App standby, battery optimization

// Process records:
// AMS maintains a ProcessRecord for every running process:
class ProcessRecord {
    val info: ApplicationInfo       // Package info
    val pid: Int                    // Linux PID
    val uid: Int                    // Linux UID
    var thread: IApplicationThread? // Binder to the app's ActivityThread
    var curAdj: Int                 // Current OOM adjustment score
    var setAdj: Int                 // Last set OOM adj
    val activities: ArrayList       // Activities in this process
    val services: HashSet           // Services in this process
    val connections: HashSet        // Service connections
    val receivers: HashSet          // Registered broadcast receivers
    val providers: ArrayList        // Content providers
    // ...
}

// OOM adjustment (oom_adj):
// AMS continuously recalculates each process's importance:
// FOREGROUND_APP_ADJ = 0      (visible Activity, foreground Service)
// VISIBLE_APP_ADJ = 100       (visible but not focused)
// PERCEPTIBLE_APP_ADJ = 200   (playing audio, has FGS)
// BACKUP_APP_ADJ = 300        (backup in progress)
// HEAVY_WEIGHT_APP_ADJ = 400  (heavy-weight process)
// SERVICE_ADJ = 500           (running a Service)
// HOME_APP_ADJ = 600          (launcher)
// PREVIOUS_APP_ADJ = 700      (previous app)
// SERVICE_B_ADJ = 800         (old Service)
// CACHED_APP_MIN_ADJ = 900    (cached, eligible for kill)
// CACHED_APP_MAX_ADJ = 999    (lowest priority cached)

// The Linux kernel's lowmemorykiller uses these values to decide
// which processes to kill when memory is low.
// AMS writes adj values to /proc//oom_score_adj

6. HAL Layer — Hardware Abstraction via Binder

// System services often need to talk to hardware.
// This goes through the HAL (Hardware Abstraction Layer).

// HAL evolution:
// Legacy HAL: .so libraries loaded into system_server (unsafe, no isolation)
// HIDL HAL (Android 8-11): separate processes, /dev/hwbinder, HIDL IDL
// AIDL HAL (Android 11+): separate processes, /dev/binder, standard AIDL

// Example: Camera
// App → CameraManager (Binder) → CameraService (native, system_server)
//   → ICameraProvider (AIDL HAL) → camera HAL process → kernel driver

// Example: Sensors
// App → SensorManager (Binder) → SensorService (native)
//   → ISensors (AIDL HAL) → sensors HAL process → kernel driver

// AIDL HAL advantages over HIDL:
// 1. Same language as framework Binder (no separate IDL)
// 2. Uses /dev/binder (unified with framework)
// 3. Versioned interfaces (stable across Android releases)
// 4. Better tooling (same aidl compiler)

// VINTF (Vendor Interface):
// Declares which HAL interfaces a device provides
// /vendor/etc/vintf/manifest.xml:
// 
//     android.hardware.camera.provider
//     2
//     ICameraProvider/default
// 
// The framework checks VINTF at boot to verify HAL availability
Staff-Level Insight — system_server is a Single Point of Failure

If system_server crashes, the entire user-space restarts (similar to Zygote death). This is by design — system services have complex interdependencies that can't be partially recovered. Android mitigates this with: (1) Watchdog thread that monitors system_server's main thread for ANRs, (2) native crash handlers that generate tombstones, (3) init restart policies. At Google scale, system_server stability is monitored via system_server_watchdog metrics. A Staff Engineer should understand that adding work to system_server's main thread (e.g., synchronous Binder calls from a system service to another) can cascade into system-wide hangs.

7. Watchdog — Detecting system_server Hangs

// The Watchdog runs inside system_server as a separate thread.
// Source: frameworks/base/services/core/java/com/android/server/Watchdog.java

// It monitors critical threads and locks:
// - Main thread (foreground handler)
// - Display thread
// - Animation thread
// - Surface animation thread
// - I/O thread
// - Critical locks (AMS lock, WMS lock, PMS lock)

// How it works:
// 1. Every 30 seconds, Watchdog posts a message to each monitored handler
// 2. If the message isn't processed within 60 seconds → half-watchdog
// 3. If not processed within 120 seconds → full watchdog
// 4. Full watchdog:
//    a. Dumps all thread stacks (like ANR)
//    b. Dumps kernel thread info
//    c. Kills system_server (Process.killProcess(Process.myPid()))
//    d. init restarts everything

// Common causes of system_server watchdog:
// 1. Lock contention (AMS lock held too long)
// 2. Binder thread pool exhaustion (all 31 threads busy)
// 3. Disk I/O on the main thread (PackageManager scanning)
// 4. Deadlock between system services

8. Boot Phases — SystemServiceManager

// SystemServiceManager controls the boot lifecycle of all system services.
// Services receive callbacks at specific boot phases:

// Phase 0:    PHASE_WAIT_FOR_DEFAULT_DISPLAY (100)
//             → Display is ready (WMS can create windows)
// Phase 1:    PHASE_LOCK_SETTINGS_READY (480)
//             → Lock settings (PIN/pattern) are available
// Phase 2:    PHASE_SYSTEM_SERVICES_READY (500)
//             → Core services are ready, can interact with each other
// Phase 3:    PHASE_ACTIVITY_MANAGER_READY (550)
//             → AMS is ready, can start launching apps
// Phase 4:    PHASE_THIRD_PARTY_APPS_CAN_START (600)
//             → Safe to start third-party apps
// Phase 5:    PHASE_BOOT_COMPLETED (1000)
//             → Boot animation finished, user can interact

// Each SystemService overrides onBootPhase():
class MySystemService : SystemService(context) {
    override fun onStart() {
        // Register with ServiceManager
        publishBinderService("my_service", MyServiceImpl())
    }

    override fun onBootPhase(phase: Int) {
        when (phase) {
            PHASE_SYSTEM_SERVICES_READY -> {
                // Safe to call other system services
            }
            PHASE_BOOT_COMPLETED -> {
                // Boot done, start background work
            }
        }
    }

    override fun onUserStarting(user: TargetUser) { /* per-user init */ }
    override fun onUserStopping(user: TargetUser) { /* per-user cleanup */ }
}

9. LocalServices — In-Process Service Lookup

// Problem: system services within system_server often call each other.
// Using Binder for same-process calls is wasteful (Parcel, ioctl, thread switch).

// Solution: LocalServices — a simple in-process registry
// Source: frameworks/base/core/java/com/android/server/LocalServices.java

// Registration (during service init):
// LocalServices.addService(DisplayManagerInternal::class.java, displayManagerImpl)

// Lookup (from another service in the same process):
// val display = LocalServices.getService(DisplayManagerInternal::class.java)
// display.setDisplayPower(...)  // Direct method call — no IPC

// Key properties:
// 1. No Binder, no Parcel, no serialization — direct Java method call
// 2. Only works within system_server (same process)
// 3. Uses a simple ArrayMap<Class, Object> internally
// 4. The "Internal" interfaces expose more than the public Binder API
//    (e.g., DisplayManagerInternal has methods not in IDisplayManager)
// 5. Thread safety: the caller runs on whatever thread it's on
//    (no Binder thread dispatch)

10. Binder Identity — Security Pattern in System Services

// Every system service method that handles app requests must:
// 1. Check the caller's identity
// 2. Optionally elevate to system privileges for internal work
// 3. Always restore the original identity

// Pattern used throughout AOSP:
fun handleAppRequest(data: String): Result {
    // Step 1: Check caller permission
    val callerUid = Binder.getCallingUid()
    val callerPid = Binder.getCallingPid()
    context.enforceCallingPermission(
        Manifest.permission.ACCESS_FINE_LOCATION,
        "Caller $callerUid lacks location permission"
    )

    // Step 2: Clear identity for privileged internal work
    val token = Binder.clearCallingIdentity()
    try {
        // Now getCallingUid() returns system_server's UID (1000)
        // This lets us call other system services without the caller's restrictions
        val location = locationService.getLastLocation() // Internal call
        val result = processData(data, location)
        return result
    } finally {
        // Step 3: ALWAYS restore in finally block
        Binder.restoreCallingIdentity(token)
    }
}

// enforceCallingPermission vs enforceCallingOrSelfPermission:
// enforceCallingPermission()       → checks ONLY the remote caller's UID
// enforceCallingOrSelfPermission() → checks caller OR self (if local call)
// Use enforceCallingPermission() for security — enforceCallingOrSelfPermission()
// can be a confused deputy vulnerability if the service itself has the permission
