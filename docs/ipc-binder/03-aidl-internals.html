03 — AIDL Internals

What the AIDL compiler generates, how Proxy/Stub work under the hood, in/out/inout semantics, oneway trade-offs, Parcelable design, and versioned AIDL for stable interfaces

1. What AIDL Actually Generates

// Given this AIDL file:
// IMyService.aidl
// interface IMyService {
//     String getData(in String key);
//     oneway void notifyUpdate(in Bundle data);
// }

// The AIDL compiler generates a single Java/Kotlin file with:
// 1. IMyService interface (extends IInterface)
// 2. IMyService.Stub abstract class (extends Binder, implements IMyService)
// 3. IMyService.Stub.Proxy class (implements IMyService, wraps BinderProxy)

// Simplified generated code:

interface IMyService : IInterface {
    fun getData(key: String): String
    fun notifyUpdate(data: Bundle)

    abstract class Stub : Binder(), IMyService {
        companion object {
            const val DESCRIPTOR = "com.example.IMyService"
            const val TRANSACTION_getData = IBinder.FIRST_CALL_TRANSACTION + 0
            const val TRANSACTION_notifyUpdate = IBinder.FIRST_CALL_TRANSACTION + 1

            fun asInterface(obj: IBinder?): IMyService? {
                if (obj == null) return null
                // Check if it's a local Binder (same process):
                val iin = obj.queryLocalInterface(DESCRIPTOR)
                if (iin is IMyService) return iin  // Same process — no IPC!
                // Different process — wrap in Proxy:
                return Proxy(obj)
            }
        }

        override fun onTransact(code: Int, data: Parcel, reply: Parcel?, flags: Int): Boolean {
            when (code) {
                TRANSACTION_getData -> {
                    data.enforceInterface(DESCRIPTOR)
                    val _arg0 = data.readString()!!
                    val _result = this.getData(_arg0)
                    reply!!.writeNoException()
                    reply.writeString(_result)
                    return true
                }
                TRANSACTION_notifyUpdate -> {
                    data.enforceInterface(DESCRIPTOR)
                    val _arg0 = data.readBundle(javaClass.classLoader)!!
                    this.notifyUpdate(_arg0)
                    // No reply — oneway
                    return true
                }
            }
            return super.onTransact(code, data, reply, flags)
        }

        private class Proxy(private val mRemote: IBinder) : IMyService {
            override fun getData(key: String): String {
                val _data = Parcel.obtain()
                val _reply = Parcel.obtain()
                try {
                    _data.writeInterfaceToken(DESCRIPTOR)
                    _data.writeString(key)
                    mRemote.transact(TRANSACTION_getData, _data, _reply, 0)
                    _reply.readException()
                    return _reply.readString()!!
                } finally {
                    _reply.recycle()
                    _data.recycle()
                }
            }

            override fun notifyUpdate(data: Bundle) {
                val _data = Parcel.obtain()
                try {
                    _data.writeInterfaceToken(DESCRIPTOR)
                    _data.writeBundle(data)
                    mRemote.transact(TRANSACTION_notifyUpdate, _data, null,
                        IBinder.FLAG_ONEWAY)  // Oneway — no reply
                } finally {
                    _data.recycle()
                }
            }

            override fun asBinder(): IBinder = mRemote
        }
    }
}
Staff-Level Insight — asInterface() Short-Circuit

Stub.asInterface() calls queryLocalInterface(DESCRIPTOR). If the IBinder is a local Binder (same process), it returns the Stub directly — no IPC, no Parcel, no kernel. This is why binding to a Service in the same process is nearly free. The framework automatically optimizes same-process calls to direct method invocations. This is also why you should never assume your AIDL implementation runs on a Binder thread — in the same-process case, it runs on the caller's thread.

2. in / out / inout — Direction Tags

// AIDL direction tags control how Parcelable parameters are marshalled:

// in (default for primitives, required for Parcelable):
// - Data is sent FROM client TO server
// - Server gets a copy; modifications don't propagate back
// void updateUser(in User user);
// Proxy: _data.writeParcelable(user)
// Stub:  val user = data.readParcelable(...)

// out:
// - Data is sent FROM server TO client (in the reply)
// - Client sends an empty/default object; server fills it
// void getUser(out User user);
// Proxy: (writes nothing for the parameter)
//        After transact: user.readFromParcel(_reply)
// Stub:  this.getUser(user)
//        reply.writeParcelable(user)

// inout:
// - Data goes BOTH ways — client sends, server can modify, changes come back
// void processUser(inout User user);
// Proxy: _data.writeParcelable(user)
//        After transact: user.readFromParcel(_reply)
// Stub:  val user = data.readParcelable(...)
//        this.processUser(user)
//        reply.writeParcelable(user)

// Performance implications:
// in    = 1 serialization (client writes)
// out   = 1 serialization (server writes reply)
// inout = 2 serializations (client writes + server writes reply)
//
// ALWAYS use "in" unless you need the server to modify the parameter.
// "inout" doubles the Parcel size and serialization cost.
Tag Client → Server Server → Client Parcel Cost Use Case
in Yes No 1x Most parameters (default)
out No Yes 1x Server fills a result object
inout Yes Yes 2x Modify-in-place (rare, avoid)

3. Parcelable Design — Performance Patterns

// AIDL Parcelable (defined in .aidl file — structured Parcelable):
// parcelable User {
//     int id;
//     String name;
//     long createdAt;
// }

// Kotlin Parcelable (using @Parcelize):
@Parcelize
data class User(
    val id: Int,
    val name: String,
    val createdAt: Long
) : Parcelable

// Performance rules:
// 1. Keep Parcelables small — every field is serialized/deserialized
// 2. Avoid nested Parcelables when possible (each adds overhead)
// 3. Use primitive types over objects (Int vs Integer)
// 4. NEVER put Bitmaps in Parcelables — use content URIs or SharedMemory
// 5. Lists: use typed lists (writeTypedList) not generic (writeList)

// Bundle vs Parcelable:
// Bundle: key-value map, flexible, but slower (HashMap + type checking)
// Parcelable: fixed schema, faster (direct read/write, no key lookup)
// Rule: use Parcelable for structured data, Bundle for ad-hoc extras

// Parcel.writeTypedList vs writeList:
// writeTypedList: knows the type at compile time, no class name written
// writeList: writes class name for EACH element (huge overhead for large lists)

// Serialization cost comparison (approximate):
// writeInt:          ~4 bytes, ~10 ns
// writeString:       ~(4 + 2*len) bytes, ~50 ns for short strings
// writeParcelable:   ~(4 + class_name + fields) bytes, ~500 ns+
// writeBundle:       ~(4 + map overhead + values) bytes, ~1 μs+

4. Oneway — Deep Dive

// Oneway transactions have subtle behaviors that matter at scale:

// 1. BUFFER PRESSURE:
// Synchronous: client blocks → only 1 transaction in flight per client thread
// Oneway: client returns immediately → can fire many transactions rapidly
// Each queued oneway consumes buffer in the TARGET's 1MB mmap space
// If the target is slow, the buffer fills → FAILED BINDER TRANSACTION

// 2. ORDERING:
// Oneway transactions to the SAME binder_node are processed in FIFO order.
// But oneway to DIFFERENT nodes (even in the same process) have NO ordering.
// This means: callbacks from different services can arrive out of order.

// 3. THREAD DISPATCH:
// Synchronous: the reply goes back to the SAME thread that sent the request
// Oneway: each transaction can be dispatched to ANY free Binder thread
// This means: oneway callbacks can arrive on different threads each time

// 4. EXCEPTION HANDLING:
// Synchronous: RemoteException propagates to the caller
// Oneway: exceptions in the server are SILENTLY SWALLOWED
// The client has no way to know if the server threw an exception
// Defense: use death notifications to detect process death

// 5. WHEN TO USE ONEWAY:
// - Callbacks from server to client (ICallback pattern)
// - Event notifications (broadcast-like)
// - Logging / analytics (fire-and-forget)
// - Any call where you don't need a return value or error

// 6. WHEN NOT TO USE ONEWAY:
// - Queries (need return value)
// - Operations where failure must be reported
// - High-frequency calls to a slow server (buffer exhaustion)
Staff-Level Insight — Oneway Buffer Exhaustion in Production

A common production issue: a foreground service sends rapid oneway callbacks to a client app that's been throttled by the scheduler. The client's Binder threads can't keep up, the 1MB buffer fills, and subsequent transactions fail silently. The server has no feedback that callbacks are being dropped. Defense: (1) rate-limit oneway calls, (2) use a single batched callback instead of per-item callbacks, (3) monitor FAILED BINDER TRANSACTION in logcat, (4) consider using ResultReceiver or Messenger for flow-controlled callbacks.

5. Versioned AIDL — Stable Interfaces

// Problem: AIDL interfaces are compiled into both client and server.
// If the server adds a new method, old clients crash (unknown transaction code).
// If the server removes a method, old clients get unexpected behavior.

// Solution: Versioned/Stable AIDL (used by system services and HALs)

// Version declaration:
// interface IMyService {
//     String getData(in String key);           // Added in version 1
//     oneway void notifyUpdate(in Bundle data); // Added in version 1
//     int getVersion();                         // Auto-generated
//     String getHash();                         // Auto-generated
// }

// In the build file:
// aidl_interface {
//     name: "my_service",
//     stability: "vintf",  // or "local"
//     versions: ["1", "2"],
//     // Each version has a frozen snapshot in aidl_api/
// }

// How versioning works:
// 1. Each version is a frozen snapshot of the .aidl files
// 2. New methods can only be ADDED (never removed or changed)
// 3. New Parcelable fields can only be APPENDED
// 4. getVersion() returns the server's version
// 5. Client checks version before calling new methods:

val version = service.version
if (version >= 2) {
    service.newMethodAddedInV2()
} else {
    // Fallback for old servers
}

// Default values for new Parcelable fields:
// parcelable Config {
//     int timeout;          // v1
//     boolean useCache;     // v1
//     String region = "us"; // v2 — has default, old clients don't send it
// }
// Old server reading new client's Parcel: ignores extra fields
// New server reading old client's Parcel: uses default values

6. AIDL vs Messenger vs ContentProvider vs Broadcast

Mechanism Direction Data Format Thread Best For
AIDL Bidirectional RPC Parcel (typed) Binder thread pool Complex APIs, high-frequency calls
Messenger Message queue Message + Bundle Handler thread Simple, serialized requests
ContentProvider CRUD + query Cursor / Bundle Binder thread pool Structured data, cross-app sharing
Broadcast One-to-many Intent + extras Main thread (default) System events, loose coupling 
// Messenger — single-threaded Binder wrapper:
// Uses a Handler internally → all messages processed sequentially
// No need for thread synchronization
// But: no return values (must send a reply Message back)
// Under the hood: Messenger wraps a Binder (IMessenger.aidl)

// When to choose AIDL over Messenger:
// 1. You need concurrent request handling (Messenger is single-threaded)
// 2. You need typed return values (Messenger only has Message)
// 3. You need complex API surface (Messenger is just sendMessage)
// 4. You need oneway callbacks (AIDL supports per-method oneway)

// When Messenger is better:
// 1. Simple request/response (don't need full AIDL)
// 2. You want automatic serialization (no thread safety concerns)
// 3. Client doesn't need to handle concurrent callbacks

7. AIDL Performance Optimization

// 1. MINIMIZE TRANSACTION COUNT:
// BAD: calling getField1(), getField2(), getField3() separately (3 IPCs)
// GOOD: getSnapshot() returning a Parcelable with all fields (1 IPC)

// 2. MINIMIZE PARCEL SIZE:
// BAD: sending a 500KB Bitmap in a Parcelable
// GOOD: sending a content URI, let the receiver load it via ContentResolver

// 3. USE ONEWAY FOR CALLBACKS:
// Callbacks don't need return values — always make them oneway
// Saves a kernel round-trip (no BC_REPLY)

// 4. BATCH OPERATIONS:
// BAD: inserting 1000 rows one at a time via AIDL
// GOOD: sending a List in a single transaction
// BETTER: using ContentProvider.bulkInsert() or applyBatch()

// 5. AVOID BINDER IN TIGHT LOOPS:
// Each Binder call: ~50-100 μs overhead (context switch + copy)
// 1000 calls = 50-100 ms of pure IPC overhead
// Solution: batch, cache, or use shared memory

// 6. CACHE REMOTE OBJECTS:
// Don't call getSystemService() repeatedly — cache the result
// val am = getSystemService(ACTIVITY_SERVICE) as ActivityManager // Cache this

// 7. USE SharedMemory FOR BULK DATA:
// val sharedMemory = SharedMemory.create("data", size)
// val buffer = sharedMemory.mapReadWrite()
// // Write data to buffer
// sharedMemory.setProtect(OsConstants.PROT_READ) // Make read-only
// // Pass sharedMemory via AIDL (it's Parcelable)
// // Receiver maps it read-only — zero-copy after initial write

8. Complete AIDL Example — Service + Client

// === Step 1: Define the AIDL interface ===
// src/main/aidl/com/example/IDataService.aidl

// package com.example;
// import com.example.UserData;
//
// interface IDataService {
//     UserData getUser(in String userId);
//     List<UserData> searchUsers(in String query);
//     oneway void notifyUpdate(in String userId);
// }

// === Step 2: Define the Parcelable ===
// src/main/aidl/com/example/UserData.aidl
// parcelable UserData;

// src/main/java/com/example/UserData.kt
@Parcelize
data class UserData(
    val id: String,
    val name: String,
    val email: String?  // Nullable — Parcel handles null strings
) : Parcelable

// === Step 3: Implement the Service ===
class DataService : Service() {
    private val binder = object : IDataService.Stub() {
        override fun getUser(userId: String): UserData {
            // Runs on a BINDER THREAD (not main thread)
            // Safe to do DB/network here
            return repository.getUser(userId)
        }

        override fun searchUsers(query: String): List<UserData> {
            return repository.search(query)
        }

        override fun notifyUpdate(userId: String) {
            // Oneway — no return, no exception to caller
            broadcastUpdate(userId)
        }
    }

    override fun onBind(intent: Intent): IBinder = binder
}

// === Step 4: Client binds and calls ===
class ClientActivity : AppCompatActivity() {
    private var service: IDataService? = null
    private var bound = false

    private val deathRecipient = IBinder.DeathRecipient {
        service = null
        bound = false
        // Reconnect on a background thread
        lifecycleScope.launch { bindToService() }
    }

    private val connection = object : ServiceConnection {
        override fun onServiceConnected(name: ComponentName, binder: IBinder) {
            service = IDataService.Stub.asInterface(binder)
            binder.linkToDeath(deathRecipient, 0)
            bound = true
        }

        override fun onServiceDisconnected(name: ComponentName) {
            service = null
            bound = false
        }
    }

    private fun bindToService() {
        val intent = Intent().apply {
            component = ComponentName("com.example", "com.example.DataService")
        }
        bindService(intent, connection, Context.BIND_AUTO_CREATE)
    }

    // Call from a coroutine (NOT main thread for cross-process):
    suspend fun fetchUser(id: String): UserData? = withContext(Dispatchers.IO) {
        try {
            service?.getUser(id)
        } catch (e: RemoteException) {
            null // Remote process died
        }
    }

    override fun onDestroy() {
        if (bound) {
            service?.asBinder()?.unlinkToDeath(deathRecipient, 0)
            unbindService(connection)
        }
        super.onDestroy()
    }
}

9. Null Handling & List/Map in AIDL

// Null handling in Parcel:
// - writeString(null) → writes -1 length marker → readString() returns null
// - writeParcelable(null, 0) → writes 0 marker → readParcelable() returns null
// - AIDL-generated code uses @nullable annotation (Android 11+):
//   @nullable String getName();  // Return can be null
//   void setName(@nullable String name);  // Parameter can be null
// - Without @nullable, generated code uses !! (non-null assertion)
//   which throws NullPointerException if null is received

// List in AIDL:
// List<String> getNames();
// → Generated: reply.writeStringList(result)
// → Receiver: _reply.createStringArrayList()

// List<Parcelable> in AIDL:
// List<UserData> getUsers();
// → Generated: reply.writeTypedList(result)  // Efficient — no class name per item
// → Receiver: _reply.createTypedArrayList(UserData.CREATOR)

// Map in AIDL (limited):
// Map is supported but only Map<String, T> where T is a supported type
// → Generated: reply.writeMap(result)  // Less efficient than typed lists
// Prefer: return a List<Parcelable> with key/value fields instead

// CREATOR for custom Parcelable (non-@Parcelize):
// companion object CREATOR : Parcelable.Creator<UserData> {
//     override fun createFromParcel(parcel: Parcel): UserData {
//         return UserData(
//             id = parcel.readString()!!,
//             name = parcel.readString()!!,
//             email = parcel.readString()  // nullable
//         )
//     }
//     override fun newArray(size: Int): Array<UserData?> = arrayOfNulls(size)
// }
// @Parcelize eliminates this boilerplate entirely — always prefer it
Staff-Level Insight — ClassLoader in readParcelable

Parcel.readParcelable(classLoader) requires the correct ClassLoader to deserialize custom Parcelables. If you pass null, it uses the boot classloader which doesn't know your app's classes — you get BadParcelableException. Always use javaClass.classLoader or UserData::class.java.classLoader. This is a common production bug when reading Parcelables from Bundles received via IPC (e.g., Intent.getParcelableExtra() on Android 13+ requires the class parameter).
