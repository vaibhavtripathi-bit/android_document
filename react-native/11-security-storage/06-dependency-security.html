1. React Native Dependency Attack Surface

A typical React Native app has dependencies across three package ecosystems — npm (JavaScript), CocoaPods (iOS native), and Gradle/Maven (Android native). Each ecosystem has its own vulnerability database, update mechanisms, and attack vectors. The total dependency count often exceeds 1,000 packages.

React Native Dependency Graph ═════════════════════════════ Your App ├── package.json (npm) │ ├── react-native (+ 50 transitive deps) │ ├── @react-navigation/native (+ 20 deps) │ ├── react-native-reanimated (+ native code) │ ├── ... (typically 50-200 direct deps) │ └── Total: 500-1500 npm packages │ ├── ios/Podfile (CocoaPods) │ ├── React-Core, React-RCTNetwork, ... │ ├── Flipper, OpenSSL-Universal, ... │ └── Total: 50-150 pods │ └── android/build.gradle (Gradle/Maven) ├── com.facebook.react:react-android ├── com.google.android.material ├── ... (transitive deps) └── Total: 100-300 artifacts Attack surface = npm + CocoaPods + Gradle + native C/C++ libs
Architect Insight

React Native's dependency surface is 3-5x larger than a pure native app because it spans three ecosystems. A supply chain attack on any single ecosystem compromises the app. Staff engineers must establish security processes that cover all three — most teams only audit npm and miss CocoaPods/Gradle entirely.

2. npm Audit & Vulnerability Scanning

# Built-in npm audit
npm audit
npm audit --production    # Only production deps (skip devDependencies)
npm audit fix             # Auto-fix compatible updates
npm audit fix --force     # Force major version updates (DANGEROUS)

# Better alternative: npm audit with JSON output for CI
npm audit --json | jq '.vulnerabilities | to_entries[] | select(.value.severity == "critical")'

# Socket.dev — detects supply chain attacks, not just CVEs
npx socket report

# Snyk — comprehensive vulnerability scanning
npx snyk test
npx snyk monitor    # Continuous monitoring

# OSV-Scanner (Google) — cross-ecosystem scanning
osv-scanner --lockfile=package-lock.json
osv-scanner --lockfile=ios/Podfile.lock
osv-scanner --lockfile=android/gradle.lockfile

2.1 CI Integration

# .github/workflows/security.yml
name: Security Audit
on: [push, pull_request]

jobs:
  audit:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: npm audit
        run: npm audit --audit-level=high --production

      - name: Snyk scan
        uses: snyk/actions/node@master
        env:
          SNYK_TOKEN: ${{ secrets.SNYK_TOKEN }}
        with:
          args: --severity-threshold=high

      - name: License check
        run: npx license-checker --failOn "GPL-3.0;AGPL-3.0"

4. Autolinking Security Implications

React Native's autolinking automatically links native modules found in node_modules. This means any npm package with native code is automatically compiled into your app binary — including packages you didn't explicitly install (transitive dependencies).

Autolinking Risk

A malicious npm package with native code (iOS/Android) could be autolinked into your app without your knowledge. The native code runs with full app permissions — it can access the file system, network, keychain, and device sensors. Always audit new dependencies for native code: check for ios/ and android/ directories, react-native.config.js, and podspec files.

// react-native.config.js — Explicitly control autolinking
module.exports = {
  dependencies: {
    // Disable autolinking for specific packages
    'react-native-suspicious-package': {
      platforms: {
        ios: null,     // Don't link on iOS
        android: null, // Don't link on Android
      },
    },
  },
};

// Library vetting checklist (automate in CI)
interface LibraryAudit {
  name: string;
  checks: {
    maintainerCount: number;          // > 2 preferred
    lastPublish: Date;               // Within 6 months
    weeklyDownloads: number;         // Popularity signal
    openIssues: number;              // Maintenance signal
    hasNativeCode: boolean;          // Higher risk
    license: string;                 // Compatible license
    knownVulnerabilities: number;    // From npm audit
    typescriptSupport: boolean;      // Quality signal
    testCoverage: number | null;     // Quality signal
    installScripts: boolean;         // postinstall = risk
    dependencyCount: number;         // Fewer = better
    sourceAvailable: boolean;        // Can audit code
  };
}

Supply Chain Attack Vectors for React Native ═════════════════════════════════════════════ 1. Typosquatting react-native-keychain → react-native-keychaln (malicious) 2. Dependency Confusion Internal @company/utils → public @company/utils (malicious) 3. Maintainer Account Compromise Legitimate package updated with malicious code 4. Malicious postinstall Scripts npm install triggers arbitrary code execution 5. Protestware / Sabotage Maintainer intentionally breaks package 6. Binary Substitution Pre-built .framework/.aar replaced with trojanized version

6.1 Mitigations

// .npmrc — Security-hardened npm configuration
// Prevent dependency confusion attacks
@company:registry=https://npm.company.com/
registry=https://registry.npmjs.org/

// Disable postinstall scripts for untrusted packages
ignore-scripts=true

// Pin exact versions (no ranges)
save-exact=true

// Require package-lock.json
package-lock=true

7. Lock File Integrity

# Verify lock file integrity in CI
# npm ci (clean install) fails if lock file doesn't match package.json
npm ci

# Detect lock file manipulation
# lockfile-lint checks for suspicious registry URLs
npx lockfile-lint \
  --path package-lock.json \
  --type npm \
  --allowed-hosts npm \
  --validate-https \
  --validate-integrity

# Verify no packages were added from unexpected registries
npx lockfile-lint \
  --path package-lock.json \
  --type npm \
  --allowed-urls https://registry.npmjs.org

10. Binary Dependency Risks

Pre-built Binary Risks

Some React Native packages ship pre-built binaries (.framework, .xcframework, .aar, .so) instead of source code. These cannot be audited without reverse engineering. Examples include Hermes engine, some analytics SDKs, and proprietary native modules. Prefer packages that build from source. If you must use pre-built binaries, verify checksums and use only from trusted publishers.

12. Architect Considerations

Dependency Security Strategy

A Staff-level dependency security strategy includes: (1) automated scanning across all three ecosystems (npm, CocoaPods, Gradle), (2) lock file integrity verification in CI, (3) a library vetting process for new dependencies, (4) automated updates with security-focused grouping, (5) license compliance checking, (6) supply chain attack mitigations (scoped registries, disabled scripts), and (7) regular dependency audits with a clear remediation SLA.
