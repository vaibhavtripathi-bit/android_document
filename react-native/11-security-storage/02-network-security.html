1. TLS Fundamentals for Mobile

TLS (Transport Layer Security) is the foundation of all network security in mobile apps. React Native's networking stack delegates to platform-native HTTP clients — NSURLSession on iOS and OkHttp on Android — which handle TLS negotiation. Understanding the TLS handshake is critical for diagnosing certificate pinning failures and configuring security policies.

TLS 1.3 Handshake (0-RTT not shown) ════════════════════════════════════ Client (RN App) Server (API) │ │ │──── ClientHello ────────────────▶│ │ (supported ciphers, │ │ key share, SNI) │ │ │ │◀─── ServerHello ────────────────│ │ (selected cipher, │ │ key share, certificate) │ │ │ │ [Client verifies certificate │ │ against trust store AND │ │ pinned certificates/keys] │ │ │ │──── Finished ──────────────────▶│ │ (encrypted with │ │ derived session key) │ │ │ │◀═══ Application Data ══════════▶│ │ (AES-256-GCM encrypted) │ │ │
TLS 1.3 vs 1.2 on Mobile

iOS 12.2+ and Android 10+ support TLS 1.3 by default. TLS 1.3 eliminates the RSA key exchange, removes vulnerable cipher suites, and reduces the handshake to 1-RTT (vs 2-RTT for TLS 1.2). For Staff-level interviews, know that TLS 1.3's 0-RTT mode trades security for latency — it's vulnerable to replay attacks and should be disabled for state-changing requests.

2. HTTPS Enforcement

Both iOS and Android enforce HTTPS by default through platform-level policies. However, developers can (and sometimes do) weaken these protections. A Staff engineer must ensure these policies are correctly configured and not bypassed.

3. Certificate Pinning

Certificate pinning restricts which certificates the app trusts for a given domain, beyond the system trust store. This prevents MITM attacks even if an attacker has a CA-signed certificate (e.g., corporate proxy, compromised CA).

3.1 Pinning Strategies

Certificate Pinning Strategies ══════════════════════════════ Strategy What's Pinned Rotation Impact Security ───────────────────────────────────────────────────────────────── Leaf Certificate Server's exact cert Must update app Highest on cert renewal Intermediate CA Issuing CA cert Survives leaf High cert rotation Public Key (SPKI) SubjectPublicKeyInfo Survives cert High hash of the key renewal if key is reused Root CA Root CA cert Survives all Lower rotations under same root
Architect Insight

Public key (SPKI) pinning is the recommended approach for production apps. It survives certificate renewal as long as the same key pair is used (which is standard practice). Pin at least two keys — the current key and a backup key — to avoid bricking the app if the primary key is compromised and must be rotated.

4. Pinning Implementation in React Native

4.1 react-native-ssl-pinning

import { fetch as sslFetch } from 'react-native-ssl-pinning';

const response = await sslFetch('https://api.example.com/data', {
  method: 'GET',
  headers: {
    'Authorization': `Bearer ${token}`,
  },
  sslPinning: {
    certs: ['api-cert'], // .cer files in app bundle
  },
  timeoutInterval: 30000,
});

// Public key pinning (preferred)
const response2 = await sslFetch('https://api.example.com/data', {
  method: 'GET',
  sslPinning: {
    certs: ['api-cert'],
    publicKeys: [
      'sha256/AAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAA=',
      'sha256/BBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBB=', // Backup pin
    ],
  },
});

5. API Key Protection

API keys in React Native apps are inherently exposed — the JavaScript bundle is readable on-device. Staff engineers must design systems that minimize the impact of key exposure.

5.1 Protection Strategies

// BAD: API key in JS bundle (easily extracted)
const API_KEY = 'sk_live_abc123';

// BETTER: Environment variable at build time (still in bundle, but not in source)
const API_KEY = process.env.API_KEY; // Replaced by Metro transformer

// GOOD: Fetch from server after authentication
const getApiKey = async () => {
  const response = await fetch('https://api.example.com/config', {
    headers: { 'Authorization': `Bearer ${userToken}` },
  });
  return response.json();
};

// BEST: Proxy through your backend — client never sees third-party keys
// Client calls your API → Your server adds the third-party API key
const response = await fetch('https://api.example.com/maps/geocode', {
  headers: { 'Authorization': `Bearer ${userToken}` },
  body: JSON.stringify({ address: '123 Main St' }),
});
Native Module Key Storage

Storing API keys in native code (Swift/Kotlin) and exposing them via a native module is marginally better than JS — the key is in compiled binary rather than a readable JS bundle — but it's still extractable via reverse engineering. The only truly secure approach is to never put secret keys on the client device. Use your backend as a proxy for third-party API calls.

6. Token Storage & Refresh

import * as Keychain from 'react-native-keychain';

class TokenManager {
  private static ACCESS_SERVICE = 'com.myapp.access-token';
  private static REFRESH_SERVICE = 'com.myapp.refresh-token';
  private refreshPromise: Promise<string> | null = null;

  async storeTokens(accessToken: string, refreshToken: string) {
    await Promise.all([
      Keychain.setGenericPassword('access', accessToken, {
        service: TokenManager.ACCESS_SERVICE,
        accessible: Keychain.ACCESSIBLE.WHEN_UNLOCKED_THIS_DEVICE_ONLY,
      }),
      Keychain.setGenericPassword('refresh', refreshToken, {
        service: TokenManager.REFRESH_SERVICE,
        accessible: Keychain.ACCESSIBLE.AFTER_FIRST_UNLOCK_THIS_DEVICE_ONLY,
      }),
    ]);
  }

  async getAccessToken(): Promise<string | null> {
    const creds = await Keychain.getGenericPassword({
      service: TokenManager.ACCESS_SERVICE,
    });
    if (!creds) return null;

    const payload = JSON.parse(atob(creds.password.split('.')[1]));
    if (payload.exp * 1000 < Date.now() - 30000) {
      return this.refreshAccessToken();
    }
    return creds.password;
  }

  async refreshAccessToken(): Promise<string> {
    if (this.refreshPromise) return this.refreshPromise;

    this.refreshPromise = (async () => {
      try {
        const refreshCreds = await Keychain.getGenericPassword({
          service: TokenManager.REFRESH_SERVICE,
        });
        if (!refreshCreds) throw new Error('No refresh token');

        const response = await fetch('https://api.example.com/auth/refresh', {
          method: 'POST',
          headers: { 'Content-Type': 'application/json' },
          body: JSON.stringify({ refreshToken: refreshCreds.password }),
        });

        const { accessToken, refreshToken } = await response.json();
        await this.storeTokens(accessToken, refreshToken);
        return accessToken;
      } finally {
        this.refreshPromise = null;
      }
    })();

    return this.refreshPromise;
  }
}

7. Man-in-the-Middle Prevention

MITM attacks on mobile apps typically exploit: (1) compromised or rogue CAs, (2) corporate proxy certificates installed on the device, (3) user-installed CA certificates, or (4) ARP spoofing on shared networks. Certificate pinning is the primary defense, but additional layers are needed.

Certificate Pinning (SPKI) │ │ 2. Proxy Detection (reject proxied requests) │ │ 3. Request Signing (HMAC body + timestamp) │ │ 4. Mutual TLS (client certificate) │ │ 5. 8. Android Network Security Config

<!-- android/app/src/main/res/xml/network_security_config.xml -->
<network-security-config>
    <!-- Base config: no cleartext, system CAs only -->
    <base-config cleartextTrafficPermitted="false">
        <trust-anchors>
            <certificates src="system" />
        </trust-anchors>
    </base-config>

    <!-- Debug overrides: allow user CAs for Charles/Flipper -->
    <debug-overrides>
        <trust-anchors>
            <certificates src="user" />
            <certificates src="system" />
        </trust-anchors>
    </debug-overrides>

    <!-- Domain-specific pinning -->
    <domain-config cleartextTrafficPermitted="false">
        <domain includeSubdomains="true">api.example.com</domain>
        <pin-set expiration="2025-12-31">
            <pin digest="SHA-256">base64_primary_pin=</pin>
            <pin digest="SHA-256">base64_backup_pin=</pin>
        </pin-set>
    </domain-config>
</network-security-config>
Debug Overrides Leak

The <debug-overrides> section only applies to debug builds (when android:debuggable="true"). However, if your build configuration accidentally sets debuggable to true in release, user-installed CAs will be trusted — completely defeating certificate pinning. Always verify debuggable is false in release builds.

10. Proxy Detection

import { NativeModules, Platform } from 'react-native';

const detectProxy = async (): Promise<boolean> => {
  if (Platform.OS === 'ios') {
    const proxySettings = await NativeModules.ProxyDetector.getProxySettings();
    return proxySettings.HTTPEnable === 1 || proxySettings.HTTPSEnable === 1;
  }

  if (Platform.OS === 'android') {
    const host = await NativeModules.ProxyDetector.getSystemProxyHost();
    const port = await NativeModules.ProxyDetector.getSystemProxyPort();
    return host !== null && host !== '' && port > 0;
  }

  return false;
};

// Usage in API client
const apiClient = {
  async request(url: string, options: RequestInit) {
    if (__DEV__ === false) {
      const isProxied = await detectProxy();
      if (isProxied) {
        throw new Error('Network proxy detected — request blocked');
      }
    }
    return fetch(url, options);
  },
};

11. Request Signing

Request signing ensures that API requests haven't been tampered with in transit. The client computes an HMAC over the request body, timestamp, and nonce, then includes the signature in a header. The server recomputes the HMAC and rejects mismatches.

import { HmacSHA256 } from 'crypto-js';

interface SignedRequest {
  url: string;
  method: string;
  body?: string;
  headers: Record<string, string>;
}

function signRequest(
  url: string,
  method: string,
  body: string | undefined,
  signingKey: string,
): SignedRequest {
  const timestamp = Math.floor(Date.now() / 1000).toString();
  const nonce = Math.random().toString(36).substring(2);

  const payload = [method, url, body || '', timestamp, nonce].join('\n');
  const signature = HmacSHA256(payload, signingKey).toString();

  return {
    url,
    method,
    body,
    headers: {
      'X-Timestamp': timestamp,
      'X-Nonce': nonce,
      'X-Signature': signature,
    },
  };
}

12. Architect Considerations

Network Security Architecture Checklist

A Staff-level network security design must address: (1) TLS version enforcement, (2) certificate pinning with rotation strategy, (3) API key isolation via backend proxy, (4) token lifecycle with secure storage, (5) request signing for tamper detection, (6) proxy detection for sensitive operations, and (7) platform-specific configuration (ATS, Network Security Config) that doesn't weaken defaults.

Certificate Pinning Rotation Strategy

Pin Rotation Without App Update ═══════════════════════════════ Time ──────────────────────────────────────────▶ Pin Set v1: [Key A (active), Key B (backup)] │ ▼ Key A compromised Server rotates to Key B Pin Set v2: [Key B (active), Key C (new backup)] │ ▼ App update ships with v2 pins Pin Set v2 active in all app versions
  • Always pin at least 2 keys — current and backup. This prevents app lockout during emergency key rotation.
  • Use SPKI pinning over certificate pinning — keys survive certificate renewal.
  • Set pin expiration dates — Android's <pin-set expiration> provides a safety valve. After expiration, the system trust store is used as fallback.
  • Monitor pinning failures — implement reporting for pinning failures to detect attacks or misconfigurations before they cause outages.
  • OTA pin updates — consider a mechanism to update pins via a trusted channel (e.g., CodePush) for emergency rotation without a full app release.
