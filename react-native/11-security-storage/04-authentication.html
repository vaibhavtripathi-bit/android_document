1. JWT Token Management

JSON Web Tokens are the standard bearer token format for mobile API authentication. A Staff engineer must understand the full JWT lifecycle — issuance, validation, refresh, revocation — and the security implications of each design choice.

JWT Token Lifecycle in React Native ════════════════════════════════════ Login API Requests Token Refresh ───── ──────────── ───────────── User credentials Access token in Refresh token │ Authorization header exchange ▼ │ │ ┌─────────┐ ┌─────▼─────┐ ┌─────▼─────┐ │ Auth │ tokens │ API │ 401 │ Auth │ │ Server │────────────▶│ Server │───────────▶│ Server │ └─────────┘ └───────────┘ └───────────┘ │ │ ▼ ▼ Store in Keychain: New access token - Access token (short-lived, 15min) + rotated refresh token - Refresh token (long-lived, 30 days) stored in Keychain

1.1 JWT Structure and Validation

interface JWTPayload {
  sub: string;       // Subject (user ID)
  iat: number;       // Issued at (Unix timestamp)
  exp: number;       // Expiration (Unix timestamp)
  iss: string;       // Issuer
  aud: string;       // Audience
  scope: string;     // Permissions
  jti: string;       // Unique token ID (for revocation)
}

function decodeJWT(token: string): JWTPayload {
  const [, payload] = token.split('.');
  return JSON.parse(atob(payload.replace(/-/g, '+').replace(/_/g, '/')));
}

function isTokenExpired(token: string, bufferSeconds = 30): boolean {
  const payload = decodeJWT(token);
  return payload.exp * 1000 < Date.now() + bufferSeconds * 1000;
}
Never Validate JWT Signatures on the Client

The client should never verify JWT signatures — it doesn't have the server's signing key (and must not). Client-side JWT "validation" is limited to checking expiration and decoding the payload for display purposes. Actual validation happens server-side on every API request. If you need to verify token authenticity on the client, use an opaque token with a server introspection endpoint.

2. OAuth 2.0 with PKCE

PKCE (Proof Key for Code Exchange) is mandatory for mobile OAuth flows. Mobile apps cannot securely store a client secret (the JS bundle is readable), so PKCE replaces the client secret with a dynamically generated code verifier/challenge pair.

2.1 Implementation with react-native-app-auth

import { authorize, refresh, revoke } from 'react-native-app-auth';

const config = {
  issuer: 'https://auth.example.com',
  clientId: 'com.myapp.mobile',
  redirectUrl: 'com.myapp://oauth/callback',
  scopes: ['openid', 'profile', 'email', 'offline_access'],
  usePKCE: true,
  serviceConfiguration: {
    authorizationEndpoint: 'https://auth.example.com/authorize',
    tokenEndpoint: 'https://auth.example.com/token',
    revocationEndpoint: 'https://auth.example.com/revoke',
  },
  additionalParameters: {
    prompt: 'login',
  },
  iosPrefersEphemeralSession: true, // Don't share cookies with Safari
};

// Login
const login = async () => {
  const result = await authorize(config);
  await tokenManager.storeTokens(
    result.accessToken,
    result.refreshToken,
  );
  return result;
};

// Token refresh
const refreshTokens = async (refreshToken: string) => {
  const result = await refresh(config, { refreshToken });
  await tokenManager.storeTokens(
    result.accessToken,
    result.refreshToken || refreshToken,
  );
  return result;
};

// Logout with token revocation
const logout = async (token: string) => {
  await revoke(config, { tokenToRevoke: token, sendClientId: true });
  await tokenManager.clearTokens();
};
Architect Insight — Why System Browser?

react-native-app-auth uses ASWebAuthenticationSession (iOS) and Chrome Custom Tabs (Android) — the system browser, not a WebView. This is critical for security: (1) the app cannot inject JS to steal credentials, (2) the user sees the real URL bar, (3) existing browser sessions/password managers are available, and (4) it's required by OAuth 2.0 for Native Apps (RFC 8252).

3. Biometric Authentication

import ReactNativeBiometrics, { BiometryTypes } from 'react-native-biometrics';

const rnBiometrics = new ReactNativeBiometrics({
  allowDeviceCredentials: true, // Fall back to PIN/pattern/password
});

class BiometricAuth {
  async checkAvailability() {
    const { available, biometryType } = await rnBiometrics.isSensorAvailable();
    return {
      available,
      type: biometryType, // BiometryTypes.FaceID | TouchID | Biometrics
    };
  }

  async enrollBiometric(userId: string) {
    const { publicKey } = await rnBiometrics.createKeys();
    await api.registerBiometricKey(userId, publicKey);
  }

  async authenticateWithBiometric() {
    const { success, signature } = await rnBiometrics.createSignature({
      promptMessage: 'Sign in',
      payload: JSON.stringify({
        timestamp: Date.now(),
        action: 'authenticate',
      }),
    });

    if (success) {
      const response = await api.verifyBiometricSignature(signature);
      return response.tokens;
    }
    throw new Error('Biometric authentication failed');
  }
}

// Apple Sign-In (required if you offer any social login on iOS)
const appleLogin = async () => {
  const response = await appleAuth.performRequest({
    requestedOperation: appleAuth.Operation.LOGIN,
    requestedScopes: [appleAuth.Scope.EMAIL, appleAuth.Scope.FULL_NAME],
  });

  const credentialState = await appleAuth.getCredentialStateForUser(response.user);
  if (credentialState === appleAuth.State.AUTHORIZED) {
    const tokens = await api.exchangeAppleToken(
      response.identityToken,
      response.authorizationCode,
    );
    return tokens;
  }
};

5. Session Management

import { AppState, AppStateStatus } from 'react-native';

class SessionManager {
  private sessionTimeout = 30 * 60 * 1000; // 30 minutes
  private lastActiveTime: number = Date.now();
  private appStateSubscription: any;

  startMonitoring() {
    this.appStateSubscription = AppState.addEventListener(
      'change',
      this.handleAppStateChange,
    );
  }

  private handleAppStateChange = (state: AppStateStatus) => {
    if (state === 'active') {
      const elapsed = Date.now() - this.lastActiveTime;
      if (elapsed > this.sessionTimeout) {
        this.handleSessionExpired();
      }
    } else if (state === 'background') {
      this.lastActiveTime = Date.now();
    }
  };

  private async handleSessionExpired() {
    const biometricAvailable = await biometricAuth.checkAvailability();
    if (biometricAvailable.available) {
      await biometricAuth.authenticateWithBiometric();
    } else {
      navigation.navigate('LockScreen');
    }
  }
}

6. Token Refresh Strategies

Token Refresh Strategies ════════════════════════ Strategy 1: Proactive Refresh (Recommended) ───────────────────────────────────────────── Check token expiry BEFORE each request. Refresh if within buffer window (e.g., 30s before expiry). + No 401 errors from expired tokens + Seamless UX - Slightly more complex Strategy 2: Reactive Refresh (401 Interceptor) ───────────────────────────────────────────── Send request → receive 401 → refresh → retry. + Simple implementation - User sees brief delay on first 401 - Must queue concurrent requests during refresh Strategy 3: Background Refresh ───────────────────────────────────────────── Timer-based refresh before expiry. + Always has valid token ready - Wastes refresh tokens if app is idle - Complex lifecycle management 
class AuthenticatedFetch {
  private refreshPromise: Promise<string> | null = null;
  private requestQueue: Array<() => void> = [];

  async fetch(url: string, options: RequestInit = {}): Promise<Response> {
    let token = await tokenManager.getAccessToken();

    if (!token || isTokenExpired(token)) {
      token = await this.refreshWithDedup();
    }

    const response = await fetch(url, {
      ...options,
      headers: {
        ...options.headers,
        'Authorization': `Bearer ${token}`,
      },
    });

    if (response.status === 401) {
      token = await this.refreshWithDedup();
      return fetch(url, {
        ...options,
        headers: { ...options.headers, 'Authorization': `Bearer ${token}` },
      });
    }

    return response;
  }

  private async refreshWithDedup(): Promise<string> {
    if (!this.refreshPromise) {
      this.refreshPromise = tokenManager.refreshAccessToken()
        .finally(() => { this.refreshPromise = null; });
    }
    return this.refreshPromise;
  }
}

7. Secure Token Storage

Token Storage Decision Matrix

Access tokens (short-lived): Keychain/Keystore with WHEN_UNLOCKED. These are needed only when the app is active. Refresh tokens (long-lived): Keychain/Keystore with AFTER_FIRST_UNLOCK so background refresh works. Never store tokens in AsyncStorage, MMKV (even encrypted), UserDefaults, or SharedPreferences.

8. Multi-Factor Authentication

class MFAManager {
  async enrollTOTP(): Promise<{ secret: string; qrUri: string }> {
    const response = await api.post('/mfa/totp/enroll');
    return {
      secret: response.secret,
      qrUri: response.otpauth_uri,
    };
  }

  async verifyTOTP(code: string): Promise<boolean> {
    const response = await api.post('/mfa/totp/verify', { code });
    return response.verified;
  }

  async requestSMSCode(phoneNumber: string): Promise<void> {
    await api.post('/mfa/sms/send', { phoneNumber });
  }

  async verifySMSCode(code: string): Promise<boolean> {
    const response = await api.post('/mfa/sms/verify', { code });
    return response.verified;
  }

  async challengeWithPush(): Promise<boolean> {
    const { challengeId } = await api.post('/mfa/push/challenge');
    return new Promise((resolve) => {
      const poll = setInterval(async () => {
        const status = await api.get(`/mfa/push/status/${challengeId}`);
        if (status.resolved) {
          clearInterval(poll);
          resolve(status.approved);
        }
      }, 2000);
    });
  }
}

9. Passkeys / WebAuthn

Passkeys are the future of authentication — phishing-resistant, passwordless credentials backed by platform authenticators (Secure Enclave on iOS, Android Keystore). React Native support is emerging through native modules and libraries.

Passkey Authentication Flow ═══════════════════════════ Registration: RN App ──▶ Server: "I want to register" Server ──▶ RN App: challenge + relying party info RN App ──▶ OS: navigator.credentials.create() OS ──▶ Secure Enclave: Generate key pair Secure Enclave ──▶ OS: Public key + attestation OS ──▶ RN App: credential RN App ──▶ Server: public key + attestation Server: stores public key Authentication: RN App ──▶ Server: "I want to sign in" Server ──▶ RN App: challenge RN App ──▶ OS: navigator.credentials.get() OS ──▶ User: Biometric prompt User ──▶ Secure Enclave: Biometric verified Secure Enclave ──▶ OS: Signed challenge OS ──▶ RN App: assertion RN App ──▶ Server: signed challenge Server: verifies with stored public key

11. Architect Considerations

Authentication Architecture Checklist

A Staff-level auth design must address: (1) token lifecycle (issuance, storage, refresh, revocation), (2) PKCE for all OAuth flows, (3) biometric as a convenience layer (not a replacement for server auth), (4) graceful degradation when biometrics are unavailable, (5) session timeout with re-authentication, (6) concurrent request handling during token refresh, and (7) logout that revokes tokens server-side.

  • Token refresh deduplication: Multiple concurrent requests hitting a 401 must share a single refresh operation. Without this, you'll burn refresh tokens and potentially trigger rate limits.
  • Apple Sign-In requirement: If your app offers any third-party social login, Apple requires you to also offer Sign in with Apple. Rejection is common for missing this.
  • Biometric re-enrollment: When a user adds a new fingerprint, BIOMETRY_CURRENT_SET keychain items are invalidated. Your app must handle this gracefully — prompt for password re-authentication.
  • Refresh token rotation: Issue a new refresh token with each use. If a refresh token is used twice, revoke all tokens for that user (indicates token theft).
