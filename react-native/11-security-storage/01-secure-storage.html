1. Storage Landscape in React Native

React Native applications have access to multiple storage mechanisms, each with different security guarantees. The critical distinction is between general-purpose storage (AsyncStorage, MMKV, SQLite) and credential storage (Keychain, Keystore). A Staff engineer must understand that these are fundamentally different systems with different threat models.

Storage Security Hierarchy in React Native ═══════════════════════════════════════════ HIGHEST SECURITY ┌─────────────────────────────────────────────┐ │ Secure Enclave / StrongBox │ │ Hardware-isolated key storage │ │ Keys never leave secure processor │ ├─────────────────────────────────────────────┤ │ Keychain (iOS) / Keystore (Android) │ │ OS-managed credential storage │ │ Hardware-backed on modern devices │ ├─────────────────────────────────────────────┤ │ Encrypted MMKV / Encrypted AsyncStorage │ │ App-level encryption with key in Keychain │ ├─────────────────────────────────────────────┤ │ SQLCipher (Encrypted SQLite) │ │ Database-level AES-256 encryption │ ├─────────────────────────────────────────────┤ │ Plain AsyncStorage / MMKV / SQLite │ │ No encryption — plaintext on disk │ └─────────────────────────────────────────────┘ LOWEST SECURITY
Critical Misconception

AsyncStorage stores data as plaintext in SQLite (Android) or plist files (iOS). On a rooted/jailbroken device, or via a device backup, this data is trivially readable. Never store tokens, passwords, or PII in plain AsyncStorage.

2. iOS Keychain Services

The iOS Keychain is a system-level encrypted database managed by the securityd daemon. It uses AES-256-GCM encryption with keys derived from the device's hardware UID and the user's passcode. Keychain items are stored in a SQLite database at /var/Keychains/keychain-2.db, encrypted with a class key hierarchy.

2.1 Keychain Architecture

iOS Keychain Key Hierarchy ══════════════════════════ Device Hardware UID (burned into Secure Enclave) │ ▼ ┌──────────────┐ ┌──────────────┐ │ Device Key │ │ Passcode Key │ └──────┬───────┘ └──────┬───────┘ │ │ ▼ ▼ ┌──────────────────────────────────┐ │ Class Key (per access │ │ control level) │ └──────────────┬───────────────────┘ │ ▼ ┌──────────────────────────────────┐ │ Per-Item Key │ │ (wraps individual keychain item)│ └──────────────────────────────────┘

2.2 Keychain Access Groups

Keychain items are scoped by access groups, which control sharing between apps. The default access group is the app's bundle identifier. Apps in the same team can share keychain items via a shared access group defined in entitlements.

// Keychain access group configuration (Xcode entitlements)
// com.apple.security.application-groups
// $(AppIdentifierPrefix)com.company.shared

// In React Native via react-native-keychain:
import * as Keychain from 'react-native-keychain';

await Keychain.setGenericPassword('user', 'token_value', {
  accessGroup: '$(AppIdentifierPrefix)com.company.shared',
  service: 'com.company.auth',
});

2.3 Keychain Accessibility Levels

Accessibility Constant Available When Persists Backup Use Case
kSecAttrAccessibleWhenUnlocked Device unlocked Yes Default — auth tokens
kSecAttrAccessibleAfterFirstUnlock After first unlock Yes Background refresh tokens
kSecAttrAccessibleWhenPasscodeSetThisDeviceOnly Passcode set, unlocked No High-security credentials
kSecAttrAccessibleAlways Always (deprecated) Yes Avoid — insecure
Architect Insight

The ThisDeviceOnly variants prevent keychain items from being included in device backups (iCloud or iTunes). For tokens that should not survive device migration, always use ThisDeviceOnly. This is a common L6 interview question — "What happens to keychain items when a user restores a backup to a new device?"

3. Android Keystore System

The Android Keystore system provides a container for cryptographic keys that are bound to the device's hardware security module (TEE or StrongBox). Unlike iOS Keychain, Android Keystore stores keys rather than arbitrary credentials — you generate or import keys, then use them to encrypt/decrypt data stored elsewhere.

3.1 Keystore Architecture

Android Keystore Architecture ═════════════════════════════ ┌─────────────────────────────────────────┐ │ Application Layer │ │ KeyStore API / Cipher / KeyGenerator │ └──────────────────┬──────────────────────┘ │ ┌──────────────────▼──────────────────────┐ │ Keystore Daemon │ │ (keystore2 service — Android 12+) │ └──────────────────┬──────────────────────┘ │ ┌──────────────────▼──────────────────────┐ │ Hardware Abstraction Layer (HAL) │ │ KeyMint (Android 12+) / Keymaster │ └──────────────────┬──────────────────────┘ │ ┌──────────────────▼──────────────────────┐ │ Trusted Execution Environment (TEE) │ │ or StrongBox Secure Element │ │ Keys never leave this boundary │ └─────────────────────────────────────────┘

3.2 Key Properties

// Android Keystore key generation (Java/Kotlin — underlying native code)
val keyGenerator = KeyGenerator.getInstance(
    KeyProperties.KEY_ALGORITHM_AES, "AndroidKeyStore"
)

keyGenerator.init(
    KeyGenParameterSpec.Builder(
        "my_key_alias",
        KeyProperties.PURPOSE_ENCRYPT or KeyProperties.PURPOSE_DECRYPT
    )
    .setBlockModes(KeyProperties.BLOCK_MODE_GCM)
    .setEncryptionPaddings(KeyProperties.ENCRYPTION_PADDING_NONE)
    .setKeySize(256)
    .setUserAuthenticationRequired(true)
    .setUserAuthenticationValidityDurationSeconds(30)
    .setIsStrongBoxBacked(true) // Requires StrongBox hardware
    .build()
)

val secretKey = keyGenerator.generateKey()

4. react-native-keychain Bridge

react-native-keychain is the standard library for bridging to platform-native credential storage. It abstracts Keychain (iOS) and Keystore-backed EncryptedSharedPreferences (Android) behind a unified API.

4.1 Core API

import * as Keychain from 'react-native-keychain';

// Store credentials
await Keychain.setGenericPassword('username', 'access_token_here', {
  accessible: Keychain.ACCESSIBLE.WHEN_UNLOCKED_THIS_DEVICE_ONLY,
  accessControl: Keychain.ACCESS_CONTROL.BIOMETRY_CURRENT_SET,
  securityLevel: Keychain.SECURITY_LEVEL.SECURE_HARDWARE,
  storage: Keychain.STORAGE_TYPE.AES,
  service: 'com.myapp.auth',
});

// Retrieve credentials
const credentials = await Keychain.getGenericPassword({
  service: 'com.myapp.auth',
  authenticationPrompt: {
    title: 'Authenticate to access credentials',
    cancel: 'Cancel',
  },
});

if (credentials) {
  console.log('Token:', credentials.password);
}

// Check supported biometry type
const biometryType = await Keychain.getSupportedBiometryType();
// Returns: 'FaceID' | 'TouchID' | 'Fingerprint' | 'Face' | 'Iris' | null

// Reset credentials
await Keychain.resetGenericPassword({ service: 'com.myapp.auth' });

5. Encrypted AsyncStorage

For bulk data that needs encryption but doesn't fit the credential model, encrypted wrappers around AsyncStorage provide a middle ground. The pattern is: store the encryption key in Keychain/Keystore, encrypt data before writing to AsyncStorage.

import AsyncStorage from '@react-native-async-storage/async-storage';
import * as Keychain from 'react-native-keychain';
import CryptoJS from 'crypto-js';

class EncryptedStorage {
  private encryptionKey: string | null = null;

  async initialize(): Promise<void> {
    const creds = await Keychain.getGenericPassword({
      service: 'com.myapp.storage-key',
    });

    if (creds) {
      this.encryptionKey = creds.password;
    } else {
      this.encryptionKey = CryptoJS.lib.WordArray.random(256 / 8).toString();
      await Keychain.setGenericPassword('storage', this.encryptionKey, {
        service: 'com.myapp.storage-key',
        accessible: Keychain.ACCESSIBLE.WHEN_UNLOCKED_THIS_DEVICE_ONLY,
      });
    }
  }

  async setItem(key: string, value: string): Promise<void> {
    const encrypted = CryptoJS.AES.encrypt(value, this.encryptionKey!).toString();
    await AsyncStorage.setItem(key, encrypted);
  }

  async getItem(key: string): Promise<string | null> {
    const encrypted = await AsyncStorage.getItem(key);
    if (!encrypted) return null;
    return CryptoJS.AES.decrypt(encrypted, this.encryptionKey!)
      .toString(CryptoJS.enc.Utf8);
  }
}
Warning: CryptoJS Limitations

CryptoJS is a pure-JS implementation — it's slow and not audited for mobile use. For production, prefer react-native-quick-crypto (OpenSSL bindings via JSI) or platform-native encryption through a native module. CryptoJS also lacks streaming support, making it unsuitable for large payloads.

6. MMKV with Encryption

MMKV (from WeChat/Tencent) supports AES-128 CFB encryption natively. When used via react-native-mmkv, encryption is handled in C++ with no JS-side crypto overhead. The encryption key should still be stored in Keychain/Keystore.

import { MMKV } from 'react-native-mmkv';

// Create encrypted MMKV instance
const secureStorage = new MMKV({
  id: 'secure-storage',
  encryptionKey: 'key-from-keychain', // Retrieved from Keychain at startup
});

secureStorage.set('user.token', 'eyJhbGciOiJSUzI1NiIs...');
secureStorage.set('user.refreshToken', 'dGhpcyBpcyBhIHJlZnJl...');

const token = secureStorage.getString('user.token');

// Re-key: change encryption key (e.g., after password change)
secureStorage.recrypt('new-key-from-keychain');

// Remove encryption entirely
secureStorage.recrypt(undefined);
MMKV vs Keychain/Keystore

MMKV encryption protects data at rest on disk but the encryption key must be loaded into memory. Keychain/Keystore items are decrypted by the OS only when accessed, and hardware-backed keys never leave the TEE. For high-value credentials (auth tokens, API keys), always prefer Keychain/Keystore. Use encrypted MMKV for bulk sensitive data like cached user profiles.

7. Biometric-Protected Storage

Both platforms support gating credential access on biometric authentication. The key insight is that biometric protection happens at the OS level — the Keychain/Keystore item is encrypted with a key that requires biometric authentication to unwrap.

import * as Keychain from 'react-native-keychain';

// Store with biometric protection
await Keychain.setGenericPassword('user', 'sensitive_token', {
  accessControl: Keychain.ACCESS_CONTROL.BIOMETRY_CURRENT_SET_OR_DEVICE_PASSCODE,
  accessible: Keychain.ACCESSIBLE.WHEN_PASSCODE_SET_THIS_DEVICE_ONLY,
  securityLevel: Keychain.SECURITY_LEVEL.SECURE_HARDWARE,
  service: 'com.myapp.biometric-auth',
});

// Retrieve — OS will prompt for biometric automatically
try {
  const credentials = await Keychain.getGenericPassword({
    service: 'com.myapp.biometric-auth',
    authenticationPrompt: {
      title: 'Verify your identity',
      subtitle: 'Access your secure credentials',
      description: 'Use Face ID or your passcode',
      cancel: 'Cancel',
    },
  });
} catch (error) {
  if (error.message === 'User canceled the operation') {
    // Handle cancellation
  } else if (error.message.includes('Biometry is locked out')) {
    // Too many failed attempts — fall back to passcode
  }
}

Access Control Behavior Fallback

BIOMETRY_ANY_OR_DEVICE_PASSCODE Biometric or passcode Device passcode
BIOMETRY_CURRENT_SET_OR_DEVICE_PASSCODE Current biometric or passcode Device passcode, invalidated on biometric change

8. Secure Enclave & StrongBox

The Secure Enclave (iOS) and StrongBox (Android) are dedicated security processors physically isolated from the main CPU. Keys generated in these processors never leave the hardware boundary — all cryptographic operations happen inside the secure processor.

Secure Enclave / StrongBox Architecture ════════════════════════════════════════ ┌──────────────────────────────────────────┐ │ Main Processor │ │ ┌────────────┐ ┌───────────────────┐ │ │ │ App Process │ │ OS Kernel │ │ │ │ (RN JS + │ │ Keychain/ │ │ │ │ Native) │──│ Keystore daemon │ │ │ └────────────┘ └────────┬──────────┘ │ └───────────────────────────┼─────────────┘ │ Mailbox protocol ┌───────────────────────────▼─────────────┐ │ Secure Enclave / StrongBox │ │ ┌─────────────────────────────────┐ │ │ │ Dedicated AES Engine │ │ │ │ True Random Number Generator │ │ │ │ Secure Boot ROM │ │ │ │ Encrypted Memory │ │ │ │ Anti-replay Counter │ │ │ └─────────────────────────────────┘ │ │ Keys are generated and used HERE │ │ Only results (ciphertext) leave │ └─────────────────────────────────────────┘
Architect Insight

Secure Enclave keys support only elliptic curve operations (P-256). You cannot store arbitrary data in the Secure Enclave — you generate an EC key pair, then use it to encrypt/sign data. For "storing" a secret in the Secure Enclave, the pattern is: generate an SE key, encrypt your secret with the SE public key, store the ciphertext in Keychain, decrypt on demand using the SE private key (which never leaves the enclave).

9. iOS Data Protection Classes

iOS Data Protection encrypts files on disk using keys derived from the device passcode and hardware UID. Each file is assigned a protection class that determines when its class key is available in memory.

Protection Class Key Available After Reboot Use Case
Complete Protection Only when unlocked After first unlock Health data, financial records
Protected Unless Open Until file is closed After first unlock Background downloads
Protected Until First User Auth After first unlock After first unlock Default — most app data
No Protection Always Always Non-sensitive system data

10. Encrypted Databases (SQLCipher)

SQLCipher is a fork of SQLite that provides transparent AES-256 encryption of the entire database file. In React Native, it's available through react-native-sqlcipher-storage or op-sqlite with encryption support.

import SQLite from 'react-native-sqlcipher-storage';

// Open encrypted database
const db = await SQLite.openDatabase({
  name: 'secure.db',
  key: 'encryption-key-from-keychain', // Retrieved from Keychain
  location: 'default',
});

// Re-key database (change encryption key)
await db.executeSql("PRAGMA rekey = 'new-key-from-keychain'");

// Verify encryption is active
const [result] = await db.executeSql('PRAGMA cipher_version');
console.log('SQLCipher version:', result.rows.item(0).cipher_version);

// Performance: SQLCipher adds ~5-15% overhead for read/write operations
// Use PRAGMA cipher_memory_security = OFF for better performance
// (disables memory wiping — acceptable for most mobile apps)

12. Architect Considerations

Storage Strategy Decision Framework

Rule of thumb: Credentials (tokens, passwords, API keys) → Keychain/Keystore. Bulk sensitive data (user profiles, messages) → Encrypted MMKV or SQLCipher. Non-sensitive preferences → Plain MMKV or AsyncStorage. The encryption key for any app-level encryption must always be stored in Keychain/Keystore — never hardcoded or derived from predictable inputs.

Common L6 Interview Pitfall

Candidates often conflate "encrypted storage" with "secure storage." Encryption protects data at rest, but if the encryption key is stored alongside the data (or hardcoded in the JS bundle), the encryption is theater. True secure storage requires the key to be managed by a hardware-backed system (Keychain/Keystore/TEE) that enforces access control independently of the application.

Key Architectural Decisions

  • Key rotation strategy: How do you rotate encryption keys without data loss? MMKV supports recrypt(); SQLCipher supports PRAGMA rekey; Keychain items must be re-stored.
  • Backup behavior: ThisDeviceOnly items don't survive backup/restore. Design your auth flow to handle credential loss gracefully.
  • Multi-user support: Use separate Keychain services or MMKV instances per user to prevent credential leakage between accounts.
  • Wipe on jailbreak: Consider clearing sensitive storage if jailbreak/root is detected — but balance security with user experience.
  • Migration path: When upgrading storage (e.g., AsyncStorage → encrypted MMKV), migrate data atomically and verify before deleting the old store.
