Security & Storage — Staff Interview Q&A

Fifty-six deep-dive questions on React Native mobile security: platform primitives, threat models, and the architectural trade-offs expected at L6/Staff level.

Secure Storage

Q1: How do iOS Keychain and Android Keystore differ at the OS primitive level, and what does that imply for a cross-platform RN app?

On iOS, the Keychain is a centralized credential store with access control lists tied to kSecAttrAccessible* and optional LocalAuthentication constraints. On Android, the Keystore backs hardware-backed keys (TEE or StrongBox) and EncryptedSharedPreferences uses keys that never leave secure hardware in the ideal case. For RN, you abstract via libraries like react-native-keychain but must still reason about backup (kSecAttrSynchronizable vs Android backup rules), key invalidation on biometric enrollment changes, and first-unlock availability on Android file-based encryption.

Q2: Explain Keychain accessibility classes and when you would choose kSecAttrAccessibleAfterFirstUnlockThisDeviceOnly over WhenUnlockedThisDeviceOnly.

WhenUnlocked* requires the device to be currently unlocked, minimizing exposure if the phone is seized while locked. AfterFirstUnlock* allows background reads after the first unlock following reboot—needed for push-handled refreshes or background tasks. ThisDeviceOnly excludes iCloud Keychain sync, reducing cross-device leakage. Staff-level answer: align with threat model—high-value tokens often use WhenUnlockedThisDeviceOnly; background-capable services may accept AfterFirstUnlockThisDeviceOnly with compensating server-side controls.

Q3: What role does the Secure Enclave (and Android StrongBox) play, and can JavaScript ever “see” private keys?

The Secure Enclave / StrongBox perform cryptographic operations inside an isolated coprocessor; private key material is not exportable to app address space. In RN, JS triggers native modules that call SecKey / KeyStore APIs—JS sees ciphertext, handles, and operation results, not raw keys. If your design passes PEM private keys through the bridge, you have already lost hardware isolation; the correct pattern is key generation and signing inside Keystore/Keychain with minimal data crossing the bridge.

Q4: How does MMKV encryption work relative to AsyncStorage, and what are the residual risks?

MMKV uses mmap’d files with optional AES encryption keyed from a secret you supply (often combined with secure storage for the key). It is faster and more predictable than SQLite-backed AsyncStorage. Residual risks: key provisioning (if the encryption key is hardcoded or derivable, mmap only obfuscates), forensics on unlocked devices, and lack of OS-level access control comparable to Keychain unless you integrate with platform secure storage for the master key.

Q5: How would you implement biometric-gated secret retrieval without storing secrets in memory longer than necessary?

Use Keychain/Keystore items protected with LAContext (iOS) or setUserAuthenticationRequired (Android) so decryption requires a fresh biometric gate. Fetch tokens just-in-time per request, zero sensitive buffers after use where APIs allow, and prefer short-lived session keys derived server-side. Avoid keeping decrypted JWTs in global JS singletons; pair with memory pressure awareness knowing JS runtimes do not guarantee immediate zeroing.

Q6: What happens to secure storage on Android when the user enables backup or switches devices?

Android backups can exfiltrate app data unless you exclude paths via android:fullBackupContent / dataExtractionRules. Keystore keys marked non-exportable will not serialize into backup blobs the same way plaintext files will. Staff answer: audit Auto Backup, test restore flows, and ensure server-side session invalidation on suspicious device changes—client storage is never sufficient alone.

Q7: Compare “encrypt everything in AsyncStorage” vs “store only a refresh token in Keychain and keep state ephemeral.”

App-level encryption of AsyncStorage still hinges on where the DEK lives; if it is in the JS bundle or predictable, it is theater. Keychain-first design reduces blast radius and aligns with least privilege: minimize persisted PII, prefer opaque refresh tokens with rotation, and use encrypted containers only for non-secret UI cache. Trade-off: more round trips and tighter native dependency.

Q8: How do you reason about jailbroken/rooted devices in relation to secure storage guarantees?

OS security boundaries are weakened—hooks, kernel patches, and elevated processes can read app sandbox memory or intercept Keychain APIs. Secure storage still raises cost but is not absolute. Combine integrity signals (SafetyNet/Play Integrity, DeviceCheck/App Attest), server-side risk scoring, and reduced functionality on high-risk devices rather than pretending Keychain is magic.

Network Security

Q9: Describe certificate pinning in RN: public key vs leaf pinning, and rotation strategy.

SPKI pinning hashes the public key so you survive leaf cert rotation that preserves the same key pair. Pinning only the leaf ties you to shorter-lived certs and increases outage risk. Staff pattern: pin an intermediate or use multiple pins (backup keys), ship updates before expiry, and implement emergency pin override only with extreme care (often server-driven feature flags, not weaker TLS).

Q10: How does TLS 1.3 change threat modeling versus TLS 1.2 for mobile clients?

TLS 1.3 removes obsolete ciphers, enforces forward secrecy for most handshakes, and shortens the handshake—fewer downgrade edges. For RN, ensure your native stack (NSURLSession/OkHttp) is current; legacy proxies or corporate MITM boxes may break—plan pinning exceptions only for documented enterprise deployments, not general users.

Q11: What is the limitation of pinning against a sophisticated MITM on a compromised device?

Pinning stops network-path adversaries and mis-issued CAs in the honest device model. On a device with installed user CAs and forced proxying (common in pentests) or runtime hooking of SSLContext, pinning can be bypassed in process. Treat pinning as defense in depth with app attestation and behavioral monitoring, not a sole guarantee.

Q12: When would you add request signing (HMAC) on top of HTTPS, and what keys must never live in JS?

Use signing to mitigate replay within narrow windows, bind requests to device context, or authenticate non-TLS channels (e.g., WebSockets uplift). Long-lived symmetric master secrets in the bundle are extractable via reverse engineering—prefer ephemeral negotiated keys, attestation-backed enrollment, or server-issued per-device secrets stored in Keystore. If signing is client-only theater, skip it.

Q13: How can you detect proxy-based traffic interception in production without harming legitimate users?

Signals include proxy system settings, unexpected CA chains, TLS handshake anomalies, and timing fingerprints—each has false positives (corporate networks, parental controls). Staff approach: graded responses (warn vs block), correlate with Play Integrity / DeviceCheck, and avoid brittle hard-fail rules unless the app is high-risk (banking).

Q14: Explain App Transport Security (ATS) and Android Network Security Config interplay in an RN monorepo.

ATS in Info.plist governs iOS outbound TLS; network security config governs Android cleartext and user CA trust. RN’s fetch uses native stacks—both platforms must align. Document exceptions (dev-only domains) and ensure release builds strip debug allowances; CI should grep for NSAllowsArbitraryLoads and cleartextTrafficPermitted.

Q15: Where should API keys live in an RN app, and what is the realistic threat model?

Anything shipped to the client is recoverable. Prefer proxying through your backend for paid/third-party APIs. If a client key is unavoidable (maps, analytics), treat it as rate-limited and scoped, rotate often, and pair with referrer/bundle-id restrictions on the provider side. Interviewers expect honesty: process.env is not secrecy.

Q16: How do WebSockets and GraphQL subscriptions change your network security posture?

Long-lived connections complicate token refresh, pinning rotation, and idle timeouts. You must re-run auth on reconnect, avoid leaking tokens in query strings, and ensure subscription payloads are authorized server-side—GraphQL’s flexibility increases risk of overfetch and field-level auth bugs independent of transport crypto.

Code Protection

Q17: What does Hermes bytecode buy you for IP protection versus plain bundled JS?

Hermes ships precompiled bytecode, raising the bar versus readable minified JS but not providing cryptographic secrecy. Determined attackers decompile or instrument the runtime. Combine with platform obfuscation (R8/ProGuard on Android), splitting critical logic server-side, and license/legal protections for real risk reduction.

Q18: How do ProGuard/R8 rules interact with React Native’s reflection-heavy native modules?

R8 may strip or rename classes referenced from JNI or reflection unless you add keep rules. Misconfigured shrinking causes runtime crashes, not subtle security wins. Maintain consumer ProGuard files in libraries, test release builds aggressively, and treat mapping files as sensitive build artifacts (they ease attacker decompilation).

Q19: Differentiate jailbreak detection, root detection, and emulator detection—what are the ethical and reliability issues?

Each heuristic (binary presence, syscall behavior, SafetyNet signals) can false-positive on OEM ROMs or enterprise devices. Over-blocking angers users and may violate store policies if implemented as discrimination without disclosure. Staff stance: risk-based friction, transparent messaging, and never sole reliance for financial guarantees.

Q20: How would you implement tamper detection for the JS bundle and native binary?

Compute signatures at build time, verify at runtime against embedded or remotely provisioned expectations, and integrate Play Integrity / App Attest for device honesty. Understand limits: attackers who control the device can patch your checks—server-side attestation and continuous auth matter more than a single CRC in JS.

Q21: What is the threat model of remote JS execution (e.g., OTA bundles) to code integrity?

OTA changes attack surface and incident response: a compromised CDN or signing key pivots to all clients. Mitigations: code signing for bundles, TLS + pinning for delivery, version pinning, staged rollouts, and kill switches. Align with store policies—Apple scrutinizes executable content updates.

Q22: Can obfuscation replace proper crypto or secure storage?

No—obfuscation is entropy against casual inspection. Secrets in strings remain extractable via memory dumps and dynamic instrumentation. Use OS keystores, short-lived tokens, and server authority for secrets; obfuscate only as a supplementary control.

Q23: What are practical anti-debugging measures on mobile, and why might you avoid them?

Techniques include ptrace tricks, timing checks, and isDebuggerAttached flags—they complicate QA, crash under legitimate debuggers, and anger store reviewers if they break development workflows. Prefer split build flavors (strict checks only in release) and measure ROI vs maintenance.

Q24: How does React Native’s bridge surface affect code protection planning?

The JSI/TurboModules boundary exposes attackable native entry points. Large, permissive native modules increase call surface for hooking. Minimize privileged operations callable from JS, validate arguments in native code, and assume JS is fully hostile in high-threat models.

Authentication

Q25: Design JWT lifecycle management for RN with refresh rotation—where do tokens live and how do you handle race conditions?

Store refresh tokens in Keychain/Keystore; keep access tokens memory-first with short TTL. Use a single-flight refresh mutex so concurrent 401s do not stampede refresh endpoints. On rotation failure, hard logout and clear secure storage; log metrics for replay detection server-side.

Q26: Why is OAuth 2.0 PKCE mandatory for public mobile clients, and what breaks without it?

Without PKCE, intercepted authorization codes can be exchanged by an attacker who lacks the client secret (mobile apps cannot keep secrets). code_verifier binds the token exchange to the same app instance that started the flow. Staff detail: still validate state, use HTTPS, and prefer ASWebAuthenticationSession / Custom Tabs over embedded WebViews where possible.

Q27: How do Passkeys/WebAuthn change your RN auth architecture compared to passwords?

Passkeys move credentials into the platform authenticator with phishing-resistant origin binding. RN typically integrates via native modules or react-native-passkey wrappers. You must handle multi-device sync (iCloud/Google Password Manager), account recovery, and fallback factors for legacy users.

Q28: Compare biometric unlock for the app shell vs cryptographic authentication for transactions.

App unlock reuses OS session convenience and can be replayed if the device is already compromised at the UI layer. Transaction signing should use Keystore user-auth keys with gating per operation for high-value actions. Interviewers want separation of UX friction and security guarantees.

Q29: What are common JWT anti-patterns in mobile apps?

Storing long-lived JWTs in AsyncStorage, trusting client-side exp without server validation, disabling signature verification “temporarily,” and bloating claims with PII. Prefer opaque session IDs for server-side revocation when requirements demand instant kill.

Q30: How would you implement SSO across a mobile app and a WebView-heavy flow without token leakage?

Avoid postMessage of raw tokens without strict origin validation. Prefer ASWebAuthenticationSession / Custom Tabs with deep link callbacks, ephemeral cookies where applicable, and server-issued one-time exchange codes bound to attested clients. Never log redirect URLs containing secrets.

Q31: Describe step-up authentication and where it fits in RN navigation.

Step-up re-authenticates for sensitive routes (wire transfers, profile changes). Implement as a navigator guard consulting a risk engine response header, triggering native biometric or FIDO2 prompts before pushing certain screens. Cache step-up validity with short TTL server-side.

Q32: How do you handle auth when the OS kills your JS context while native keeps credentials?

On resume, rehydrate session state idempotently: read secure storage, silent refresh if needed, and avoid duplicating navigators. Use AppState listeners and central auth providers to prevent split-brain between native modules and JS trees.

Data Protection

Q33: What does “encryption at rest” mean on mobile for SQLite, images, and logs?

Modern devices use FDE/FBE, but unlocked phones decrypt transparently—app-level encryption still matters for device-loss and backup exfiltration. Encrypt SQLite with SQLCipher or similar using keys from Keystore, scrub logs of PII, and route crash reports through PII filters.

Q34: Compare GDPR and CCPA implications for mobile analytics in practical engineering terms.

Both require disclosure, lawful basis/consent, and access/deletion pathways; California adds opt-out of sale/share nuances. Implement consent gating before SDK init, data inventory per SDK, and server endpoints honoring erasure propagating to warehouses. RN-specific: delay Firebase/Segment init until legal flags resolve.

Q35: How do you mitigate screenshots and screen recording for sensitive screens?

iOS offers limited APIs (e.g., UITextField.isSecureTextEntry for fields, snapshot-aware flags); true screenshot blocking is constrained. Android can set FLAG_SECURE on windows via native modules. Communicate honestly: these are deterrents, not solutions—complement with watermarking, short timeouts, and server policies.

Q36: What is your strategy for PII in React Native form state and navigation params?

Avoid passing national IDs or account numbers through deep link query params or persistent navigation state. Keep sensitive fields uncontrolled where possible, clear on blur/unmount, and ensure Redux/RTK stores are not logging middleware in production.

Q37: How would you design remote wipe for a BYOD app?

Server pushes a revocation signal via push or next API call; client deletes Keychain items, SQLite, and cache; optionally wipe derived keys only if crypto scheme allows crypto-shredding. For MDM-managed devices, complement with enterprise wipe policies—consumer apps cannot kernel-wipe hardware.

Q38: What is data minimization in push notification payloads?

APNs/FCM payloads are visible in notification centers and logs—never include full account numbers or medical details. Send opaque IDs and fetch in-app after unlock with authorized APIs. This reduces shoulder surfing and vendor exposure.

Q39: How do you handle clipboard security for OTPs and crypto addresses?

Prefer autofill APIs and in-app OTP fields. If clipboard is used, set expiring clears where OS supports (iOS pasteboard TTL patterns), warn users, and avoid global listeners that leak secrets across apps.

Q40: Explain secure deletion challenges on flash storage.

SSD wear-leveling means overwrite patterns are unreliable. Rely on Keystore deletion, file removal plus DB vacuum for apps, and rotating keys instead of Hollywood secure erase. For highly sensitive caches, use RAM-only buffers with minimal persistence.

Dependency Security

Q41: Walk through a supply-chain attack scenario against an RN app and your defenses.

A compromised npm package adds a postinstall exfiltration script or patches metro behavior. Defenses: lockfile integrity, npm audit/pnpm audit, dependency pinning, SBOM reviews, private registries with provenance, and CI rules blocking install scripts. Runtime: network egress allowlists for servers where feasible.

Q42: What are autolinking risks when native pods or Gradle deps pull transitive libraries?

Autolinking accelerates integration but obscures transitive native code—a forgotten pod may enable Bluetooth or location APIs that trigger privacy manifest obligations. Maintain dependency graphs for iOS and Android separately; diff them each release.

Q43: How do you validate lock file integrity across a team?

Commit package-lock.json/yarn.lock/pnpm-lock.yaml, enforce CI install --frozen-lockfile, and alert on unexpected lock churn. For native, commit Gemfile.lock and Podfile.lock where applicable. Optionally verify with sigstore/npm provenance as tooling matures.

Q44: What is the limitation of npm audit for RN security decisions?

Audit focuses on known CVEs in JS deps, not malicious maintainer behavior, and severity may not reflect reachable code paths in your bundle. Pair with static analysis, manual review of native bridges, and runtime permission audits.

Q45: How would you vet a new native module before production?

Review JNI/C++ surface, permissions declared in AndroidManifest.xml and Info.plist, ProGuard consumer rules, prior CVEs, maintainer responsiveness, and whether functionality duplicates a well-audited alternative. Build a threat model for data crossing the bridge.

Q46: Describe Renovate/Dependabot strategy trade-offs for mobile apps.

Auto-bumps reduce CVE exposure but can break native compile chains. Use grouping, staggered schedules, and mandatory device QA for minor/patch RN upgrades. Staff answer: balance MTTR for security vs release stability with risk tiers.

Q47: What risks do remote config SDKs introduce?

They can toggle features without store review—great for ops, dangerous if mis-governed. Compromise of the remote config console becomes remote code execution if configs fetch JS. Enforce schema validation, auth on admin consoles, and policy alignment with App Review 2.5.2.

Q48: How do monorepos affect supply-chain visibility for RN?

Shared packages may drag web-only deps into mobile bundles if Metro resolves incorrectly—use package exports and platform extensions. Centralize security scanning but scope findings to the mobile subgraph to avoid alert fatigue.

App Store Security

Q49: What are Apple Privacy Manifests and why do RN apps struggle with them?

Manifests declare required reason APIs and third-party SDK data collection. RN apps aggregate many pods; transitive SDKs may require merged privacy reports. Fix by upgrading SDKs, adding PrivacyInfo.xcprivacy aggregates, and validating with Xcode report generation pre-submission.

Q50: How does CodePush intersect with Apple Guideline 2.5.2?

Apple tolerates JS updates that fix bugs within the app’s general scope but not material new functionality that circumvents review. Operationalize with change management, legal/product alignment, and feature flags that cannot alter binary entitlements without a store build.

Q51: Compare Apple App Attest and Play Integrity for backend API gating.

Both provide attestation artifacts verifying genuine OS/app instances. App Attest ties to Apple’s ecosystem; Play Integrity covers Android variants with device integrity fields. Neither replaces user auth—use for risk scoring, rate limit tiers, and fraud reduction, expecting patched devices to bypass low-effort checks.

Q52: What are Required Reason APIs and how do you discover unexpected usage?

Apple mandates declarations for sensitive APIs (e.g., disk space, boot time). Discover usage via SDK binary scans, Xcode privacy reports, and grep across pods. Missing reasons cause rejections—treat findings as a cross-functional release gate.

Q53: How should Google Play Data Safety forms map to your RN analytics stack?

Create a data dictionary per SDK: collected fields, purposes, encryption in transit, deletion. RN engineers must sync with legal/privacy because declarations are legally attested—mismatches risk takedowns and regulatory exposure.

Q54: What store pitfalls arise from ATS exceptions and custom URL schemes?

Broad ATS exceptions trigger review questions; document business need. Custom URL schemes for OAuth redirects can be hijacked if not validated—prefer universal links/app links with verified domain ownership.

Q55: How do you operationalize privacy nutrition labels and Play data cards across releases?

Embed a release checklist tying SDK upgrades to label diffs, require PM/legal sign-off when new data types appear, and version-control questionnaires alongside changelog entries. Automate diffing of Info.plist usage descriptions and manifest merges.

Q56: As Staff engineer, how do you align mobile security architecture with org-wide IAM and compliance?

Publish reference architectures for auth, storage, and logging; run threat modeling sessions per major feature; interface IAM on OAuth scopes, session policies, and SIEM ingestion from mobile clients; and translate store obligations into CI guardrails the whole team cannot accidentally bypass.
