1. Apple App Review for React Native

Apple's App Review guidelines have specific implications for React Native apps. The key concern is Guideline 2.5.2 — apps must be self-contained and may not download or execute code that changes the app's features or functionality. React Native's JavaScript bundle is pre-packaged, which is compliant, but OTA updates (CodePush) exist in a gray area.

1.1 Key Guidelines for RN Apps

Guideline Requirement RN Impact
2.5.2 No downloading executable code CodePush must not change features/functionality
2.5.6 Apps must use public APIs No private iOS API usage in native modules
3.3.2 No hidden features All features must be reviewable
4.0 Design guidelines Must feel native, not web-like
5.1.1 Data collection consent Privacy manifest + ATT framework
5.1.2 Data use and sharing Privacy nutrition labels must be accurate
Guideline 2.5.2 and CodePush

Apple's interpretation: JavaScript executed by a built-in framework (JavaScriptCore/Hermes) is acceptable. OTA updates that fix bugs or make minor improvements are generally tolerated. However, OTA updates that add new features, change the app's purpose, or bypass the review process will result in rejection or removal. The safest approach: use CodePush only for bug fixes and minor UI tweaks, not feature launches.

3. Privacy Manifests (iOS 17+)

Starting with iOS 17, Apple requires privacy manifests (PrivacyInfo.xcprivacy) that declare what data your app and its third-party SDKs collect, why certain APIs are used, and what tracking domains are contacted. This is enforced at App Review time.

<!-- PrivacyInfo.xcprivacy — Required for iOS 17+ -->
<?xml version="1.0" encoding="UTF-8"?>
<!DOCTYPE plist PUBLIC "-//Apple//DTD PLIST 1.0//EN"
  "http://www.apple.com/DTDs/PropertyList-1.0.dtd">
<plist version="1.0">
<dict>
    <key>NSPrivacyTracking</key>
    <false/>

    <key>NSPrivacyTrackingDomains</key>
    <array/>

    <key>NSPrivacyCollectedDataTypes</key>
    <array>
        <dict>
            <key>NSPrivacyCollectedDataType</key>
            <string>NSPrivacyCollectedDataTypeEmailAddress</string>
            <key>NSPrivacyCollectedDataTypeLinked</key>
            <true/>
            <key>NSPrivacyCollectedDataTypeTracking</key>
            <false/>
            <key>NSPrivacyCollectedDataTypePurposes</key>
            <array>
                <string>NSPrivacyCollectedDataTypePurposeAppFunctionality</string>
            </array>
        </dict>
    </array>

    <key>NSPrivacyAccessedAPITypes</key>
    <array>
        <dict>
            <key>NSPrivacyAccessedAPIType</key>
            <string>NSPrivacyAccessedAPICategoryUserDefaults</string>
            <key>NSPrivacyAccessedAPITypeReasons</key>
            <array>
                <string>CA92.1</string>
            </array>
        </dict>
    </array>
</dict>
</plist>
React Native and Privacy Manifests

React Native itself uses several "required reason APIs" (UserDefaults, file timestamp, system boot time, disk space). The React Native community maintains privacy manifests for the core framework. However, every third-party library with native iOS code must also include its own privacy manifest. Check each dependency's iOS pod for a PrivacyInfo.xcprivacy file — missing manifests will cause App Review rejection starting Spring 2024.

CodePush Compliance Spectrum ════════════════════════════ SAFE (Bug fixes, minor UI tweaks) ├── Fix crash on specific device ├── Update error messages ├── Fix layout issues ├── Update API endpoint URLs └── Minor copy changes GRAY AREA (Proceed with caution) ├── A/B test UI variations ├── Update business logic ├── Change navigation flow └── Add analytics events RISKY (May trigger review/rejection) ├── Add new screens/features ├── Change app's core functionality ├── Add new permissions usage ├── Change monetization model └── Add new third-party SDKs

6. App Attestation

6.1 iOS DeviceCheck & App Attest

import { NativeModules } from 'react-native';

class AppAttestManager {
  async generateAttestation(): Promise<string> {
    const keyId = await NativeModules.AppAttest.generateKey();
    const challenge = await api.getAttestationChallenge();
    const attestation = await NativeModules.AppAttest.attestKey(
      keyId,
      challenge,
    );
    return attestation;
  }

  async generateAssertion(request: string): Promise<string> {
    const assertion = await NativeModules.AppAttest.generateAssertion(
      this.keyId,
      request,
    );
    return assertion;
  }
}

6.2 Android Play Integrity API

// Android Play Integrity (replaces SafetyNet)
class PlayIntegrityManager {
  async requestIntegrityToken(): Promise<string> {
    const nonce = await api.getIntegrityNonce();
    const token = await NativeModules.PlayIntegrity
      .requestIntegrityToken(nonce);
    return token;
  }
}

8. Required Reason APIs

iOS 17 introduced "required reason APIs" — system APIs that can be used for fingerprinting. Apps must declare why they use these APIs in their privacy manifest.

API Category Used By RN Required Reason
NSUserDefaults Yes (AsyncStorage on iOS) CA92.1 (access app's own data)
File timestamp APIs Yes (Metro, RNFS) DDA9.1 (display to user) or C617.1
System boot time Possible (performance monitoring) 35F9.1 (measure elapsed time)
Disk space APIs Possible (storage management) E174.1 (check available space)
Active keyboard Rare 3EC4.1 (customize for keyboard)

9. Common RN Rejection Reasons

Top React Native App Review Rejections

(1) Missing privacy manifest for third-party SDKs, (2) NSAllowsArbitraryLoads=true in production, (3) Missing Sign in with Apple when other social logins are offered, (4) Missing account deletion capability, (5) WebView-heavy UI that doesn't feel native, (6) Missing purpose strings for permissions (camera, location, etc.), (7) CodePush used to add features post-review, (8) Missing ATT (App Tracking Transparency) prompt when using IDFA.

10. Architect Considerations

App Store Compliance Architecture

Build compliance into your CI/CD pipeline: (1) automated privacy manifest generation from dependency analysis, (2) pre-submission checklist validation, (3) data collection audit on each release, (4) CodePush policy enforcement (no feature changes via OTA), (5) permission usage documentation, (6) app attestation integration for sensitive API endpoints.
