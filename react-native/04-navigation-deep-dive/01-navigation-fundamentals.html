Navigation Fundamentals

Navigation in React Native is fundamentally different from web navigation. Mobile apps manage their own navigation stacks, handle platform-specific transitions, and must integrate with OS-level deep linking. Understanding why mobile navigation is complex is the foundation for all navigation architecture decisions.

1. Why Mobile Navigation Differs from Web

On the web, the browser manages navigation: URL bar, history stack, back/forward buttons, and page lifecycle are all built-in. Mobile apps have none of this infrastructure — the app must build and manage its own navigation system.

Web Navigation: Mobile Navigation: ═══════════════ ══════════════════ Browser manages: App must manage: ├─ URL bar ├─ Screen stack (push/pop) ├─ History stack ├─ Tab bar state ├─ Back/forward buttons ├─ Drawer state ├─ Page transitions ├─ Transition animations ├─ Deep linking (URLs) ├─ Deep linking (URL schemes) └─ Memory (pages unloaded) ├─ Gesture-based navigation (swipe back) ├─ Screen lifecycle (focus/blur) ├─ Memory management (screen retention) └─ Platform-specific behavior

1.1 Key Differences

AspectWebMobile (React Native)
Navigation modelURL-based, statelessStack-based, stateful
Screen retentionPrevious page destroyedPrevious screen stays mounted
TransitionsInstant (or CSS transitions)Native animated transitions
Back navigationBrowser back buttonGesture (swipe), hardware button (Android)
Deep linkingURLs (built-in)URL schemes, Universal/App Links (configured)
Parallel screensOne page at a timeTabs show multiple screens simultaneously
MemoryBrowser managesApp must manage (screens stay in memory)
Architect Insight

The fundamental difference is screen retention. On the web, navigating to a new page destroys the previous one. On mobile, pushing a new screen onto the stack keeps the previous screen alive (mounted) underneath. This means mobile apps accumulate memory as users navigate deeper, and "going back" doesn't re-create the screen — it reveals the existing one. This has profound implications for state management, data freshness, and memory budgeting.

3. JavaScript-Based Navigation

React Native navigation libraries implement navigation logic in JavaScript, using React components to manage screens and transitions.

JS-Based Navigation (React Navigation): NavigationContainer (root) └─ Stack.Navigator ├─ Stack.Screen name="Home" │ └─ HomeScreen component (React) ├─ Stack.Screen name="Details" │ └─ DetailsScreen component (React) └─ Stack.Screen name="Settings" └─ SettingsScreen component (React) All screens are React components Transitions are Animated/Reanimated animations State is a JS object describing the navigation tree 
import { NavigationContainer } from '@react-navigation/native';
import { createNativeStackNavigator } from '@react-navigation/native-stack';

const Stack = createNativeStackNavigator();

const App = () => (
  <NavigationContainer>
    <Stack.Navigator initialRouteName="Home">
      <Stack.Screen name="Home" component={HomeScreen} />
      <Stack.Screen name="Details" component={DetailsScreen} />
    </Stack.Navigator>
  </NavigationContainer>
);

// Navigation actions:
const HomeScreen = ({ navigation }) => (
  <Button
    title="Go to Details"
    onPress={() => navigation.navigate('Details', { id: 42 })}
  />
);

const DetailsScreen = ({ route, navigation }) => (
  <View>
    <Text>Item ID: {route.params.id}</Text>
    <Button title="Go Back" onPress={() => navigation.goBack()} />
  </View>
);

4. Navigation State as Data

React Navigation represents the entire navigation state as a serializable JavaScript object. This is a powerful concept that enables state persistence, deep linking, and programmatic navigation.

// Navigation state is a tree of route objects:
const navigationState = {
  type: 'stack',
  key: 'stack-1',
  routeNames: ['Home', 'Details', 'Settings'],
  routes: [
    { key: 'home-1', name: 'Home', params: {} },
    { key: 'details-1', name: 'Details', params: { id: 42 } },
  ],
  index: 1,  // Currently focused route (Details)
  stale: false,
};

// Nested navigators create nested state:
const nestedState = {
  type: 'tab',
  routes: [
    {
      key: 'home-tab',
      name: 'HomeTab',
      state: {
        type: 'stack',
        routes: [
          { key: 'home', name: 'Home' },
          { key: 'details', name: 'Details', params: { id: 42 } },
        ],
        index: 1,
      },
    },
    { key: 'settings-tab', name: 'SettingsTab' },
  ],
  index: 0,
};

// State is serializable → can be persisted to AsyncStorage
// State is declarative → can be restored from deep link

5. Screen Lifecycle

// Screen lifecycle in React Navigation:

// 1. Mount: Screen component first rendered
// 2. Focus: Screen becomes the active/visible screen
// 3. Blur: Screen loses focus (another screen pushed on top)
// 4. Unmount: Screen removed from navigator (popped from stack)

// Key insight: screens stay MOUNTED when blurred!
// Pushing a new screen does NOT unmount the previous one

import { useFocusEffect, useIsFocused } from '@react-navigation/native';

const ProfileScreen = () => {
  // useEffect runs on mount/unmount — NOT on focus/blur
  useEffect(() => {
    console.log('Mounted');
    return () => console.log('Unmounted');
  }, []);

  // useFocusEffect runs on focus/blur
  useFocusEffect(
    useCallback(() => {
      console.log('Focused — fetch fresh data');
      fetchProfile();

      return () => {
        console.log('Blurred — cancel pending requests');
      };
    }, [])
  );

  // useIsFocused for conditional rendering
  const isFocused = useIsFocused();
  // Only render expensive content when focused

  return <View>{isFocused && <ExpensiveChart />}</View>;
};
Common Mistake

Using useEffect to fetch data on screen "appearance" is wrong in React Navigation. useEffect only runs on mount/unmount. If a user navigates away and comes back (the screen was blurred then focused), useEffect does NOT re-run because the component was never unmounted. Use useFocusEffect for data fetching that should happen every time the screen is visible.

6. Deep Linking Introduction

// Deep linking maps URLs to navigation states
// URL: myapp://details/42
// → Navigate to Details screen with params { id: 42 }

// React Navigation linking configuration:
const linking = {
  prefixes: ['myapp://', 'https://myapp.com'],
  config: {
    screens: {
      Home: 'home',
      Details: 'details/:id',
      Settings: 'settings',
      Profile: {
        path: 'user/:userId',
        screens: {
          Posts: 'posts',
          Followers: 'followers',
        },
      },
    },
  },
};

<NavigationContainer linking={linking}>
  {/* navigators */}
</NavigationContainer>

// Deep link → navigation state conversion:
// "myapp://details/42" →
// { routes: [{ name: 'Home' }, { name: 'Details', params: { id: '42' } }] }

7. Navigation Container

// NavigationContainer is the root component
// It manages:
// 1. Navigation state
// 2. Deep link handling
// 3. State persistence
// 4. Navigation ref (for programmatic navigation)

const App = () => {
  const [isReady, setIsReady] = useState(false);
  const [initialState, setInitialState] = useState();

  // State persistence
  useEffect(() => {
    const restoreState = async () => {
      const savedState = await AsyncStorage.getItem('NAV_STATE');
      if (savedState) setInitialState(JSON.parse(savedState));
      setIsReady(true);
    };
    restoreState();
  }, []);

  if (!isReady) return <SplashScreen />;

  return (
    <NavigationContainer
      initialState={initialState}
      onStateChange={(state) => {
        AsyncStorage.setItem('NAV_STATE', JSON.stringify(state));
      }}
      linking={linking}
      fallback={<LoadingScreen />}
    >
      <RootNavigator />
    </NavigationContainer>
  );
};

8. Navigation Ref

// Navigation ref enables navigation from outside React components
import { createNavigationContainerRef } from '@react-navigation/native';

export const navigationRef = createNavigationContainerRef();

// Use in NavigationContainer:
<NavigationContainer ref={navigationRef}>
  {/* navigators */}
</NavigationContainer>

// Navigate from anywhere (e.g., push notification handler):
export function navigateFromNotification(screen, params) {
  if (navigationRef.isReady()) {
    navigationRef.navigate(screen, params);
  }
}

// Common use cases for navigation ref:
// 1. Push notification deep linking
// 2. Analytics screen tracking
// 3. Authentication redirect (from API interceptor)
// 4. Global error handling (navigate to error screen)

// Screen tracking with navigation ref:
<NavigationContainer
  ref={navigationRef}
  onStateChange={() => {
    const currentRoute = navigationRef.getCurrentRoute();
    analytics.trackScreen(currentRoute.name, currentRoute.params);
  }}
>

10. Architect Considerations

Staff-Level Mental Model

Navigation is the skeleton of a mobile app — every feature, every user flow, every deep link passes through the navigation system. A Staff engineer's navigation architecture decisions include: (1) Library choice — React Navigation is the default; Wix's react-native-navigation for apps requiring fully native navigation; expo-router for Expo-first apps. (2) State management integration — navigation state should be the source of truth for "where the user is," not Redux/Zustand. (3) Deep linking from day one — retrofitting deep links is extremely painful. (4) Screen lifecycle awareness — every screen must handle focus/blur correctly for data freshness and resource management. (5) Type safety — navigation params should be fully typed to prevent runtime crashes.
