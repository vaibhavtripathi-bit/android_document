Deep Linking

Deep linking connects URLs to specific screens in your app. Understanding URL scheme configuration, Universal Links (iOS), App Links (Android), and navigation integration is essential for building apps that integrate with the broader mobile ecosystem.

1. URL Scheme Configuration

Custom URL schemes (e.g., myapp://) are the simplest form of deep linking. They allow other apps and the OS to open your app to a specific screen.

1.1 iOS Configuration (Info.plist)

<!-- ios/MyApp/Info.plist -->
<key>CFBundleURLTypes</key>
<array>
  <dict>
    <key>CFBundleURLSchemes</key>
    <array>
      <string>myapp</string>
    </array>
    <key>CFBundleURLName</key>
    <string>com.mycompany.myapp</string>
  </dict>
</array>

<!-- Now myapp://details/42 will open the app -->
<!-- AppDelegate handles the URL via:
     application:openURL:options: -->

1.2 Android Configuration (AndroidManifest.xml)

<!-- android/app/src/main/AndroidManifest.xml -->
<activity
  android:name=".MainActivity"
  android:launchMode="singleTask">

  <intent-filter>
    <action android:name="android.intent.action.VIEW" />
    <category android:name="android.intent.category.DEFAULT" />
    <category android:name="android.intent.category.BROWSABLE" />
    <data android:scheme="myapp" />
  </intent-filter>

</activity>

<!-- Now myapp://details/42 will open the app -->
<!-- launchMode="singleTask" ensures existing instance receives the intent -->
<!-- Without singleTask, a new activity instance is created -->
URL Scheme Limitations

Custom URL schemes are not secure — any app can register the same scheme. If two apps register myapp://, the OS decides which one opens (undefined behavior on iOS, user choice on Android). For secure deep linking, use Universal Links (iOS) or App Links (Android), which verify domain ownership.

2. Universal Links (iOS)

Universal Links use standard HTTPS URLs that open your app instead of Safari. They require domain verification — Apple checks that you own the domain.

// Step 1: Host apple-app-site-association (AASA) file
// at https://myapp.com/.well-known/apple-app-site-association

{
  "applinks": {
    "apps": [],
    "details": [
      {
        "appID": "TEAMID.com.mycompany.myapp",
        "paths": [
          "/details/*",
          "/user/*",
          "/settings"
        ]
      }
    ]
  }
}

// Step 2: Enable Associated Domains in Xcode
// Signing & Capabilities → + Capability → Associated Domains
// Add: applinks:myapp.com

// Step 3: Handle in AppDelegate
// application:continueUserActivity:restorationHandler:
// React Native's Linking module handles this automatically

// Now https://myapp.com/details/42 opens the app directly
// If app not installed → opens in Safari (graceful fallback)

3. App Links (Android)

// Step 1: Host Digital Asset Links file
// at https://myapp.com/.well-known/assetlinks.json

[{
  "relation": ["delegate_permission/common.handle_all_urls"],
  "target": {
    "namespace": "android_app",
    "package_name": "com.mycompany.myapp",
    "sha256_cert_fingerprints": [
      "AA:BB:CC:DD:..."
    ]
  }
}]

// Step 2: Configure intent-filter with autoVerify
<intent-filter android:autoVerify="true">
  <action android:name="android.intent.action.VIEW" />
  <category android:name="android.intent.category.DEFAULT" />
  <category android:name="android.intent.category.BROWSABLE" />
  <data android:scheme="https"
       android:host="myapp.com"
       android:pathPrefix="/details" />
</intent-filter>

// autoVerify="true" triggers Android to verify domain ownership
// Verified links open app directly (no disambiguation dialog)
// Unverified links show "Open with..." chooser

4. Linking API

import { Linking } from 'react-native';

// Get the URL that launched the app (cold start)
const initialUrl = await Linking.getInitialURL();
if (initialUrl) {
  handleDeepLink(initialUrl);
}

// Listen for URLs while app is running (warm start)
const subscription = Linking.addEventListener('url', ({ url }) => {
  handleDeepLink(url);
});

// Cleanup
subscription.remove();

// Open external URLs
await Linking.openURL('https://google.com');
await Linking.openURL('tel:+1234567890');
await Linking.openURL('mailto:user@example.com');

// Check if URL can be opened
const canOpen = await Linking.canOpenURL('myapp://details/42');

// Open app settings
await Linking.openSettings();

6. Deferred Deep Links

// Deferred deep links work even when the app is NOT installed
// Flow:
// 1. User clicks link on web/email
// 2. App not installed → redirect to App Store / Play Store
// 3. User installs app
// 4. On first launch, app receives the original deep link

// This requires a third-party service that:
// - Stores the deep link intent
// - Identifies the user after install (fingerprinting or clipboard)
// - Delivers the link on first app open

// Implementation with Branch.io:
import branch from 'react-native-branch';

branch.subscribe(({ error, params, uri }) => {
  if (error) return;

  if (params['+clicked_branch_link']) {
    // User came from a Branch link
    const screen = params.screen;     // e.g., 'Details'
    const id = params.id;             // e.g., '42'
    navigation.navigate(screen, { id });
  }

  if (params['+is_first_session']) {
    // This is a DEFERRED deep link (first install)
    // Show onboarding first, then navigate
  }
});

9. Authentication and Deep Links

// Challenge: deep link to authenticated screen when user is not logged in

// Pattern: store pending deep link, process after auth
const App = () => {
  const { isAuthenticated } = useAuth();
  const [pendingDeepLink, setPendingDeepLink] = useState(null);

  const linking = {
    prefixes: ['myapp://'],
    config: { screens: { /* ... */ } },
    getStateFromPath(path, config) {
      const state = defaultGetStateFromPath(path, config);

      if (!isAuthenticated && requiresAuth(path)) {
        setPendingDeepLink(path);
        return undefined; // Don't navigate yet
      }

      return state;
    },
  };

  // After successful login, process pending deep link
  useEffect(() => {
    if (isAuthenticated && pendingDeepLink) {
      const state = getStateFromPath(pendingDeepLink, linking.config);
      navigationRef.reset(state);
      setPendingDeepLink(null);
    }
  }, [isAuthenticated, pendingDeepLink]);

  return (
    <NavigationContainer ref={navigationRef} linking={linking}>
      {isAuthenticated ? <MainNavigator /> : <AuthNavigator />}
    </NavigationContainer>
  );
};

10. Architect Considerations

Staff-Level Mental Model

Deep linking should be designed from day one, not retrofitted. Every screen in your app should have a URL — this forces clean navigation architecture and enables: (1) Push notification routing. (2) Email/SMS marketing links. (3) Cross-app integration. (4) QR code scanning. (5) Sharing. (6) Testing (open any screen directly). The linking config is essentially a URL router — treat it with the same rigor as a web API's route definitions. Use Universal Links/App Links for production (secure, fallback to web), URL schemes for development convenience.

10.1 Deep Link Architecture Checklist

// 1. Every screen has a URL path
// 2. URL params are minimal (IDs, not full objects)
// 3. Authentication-gated links are handled gracefully
// 4. Invalid URLs show a "Not Found" screen
// 5. Universal Links / App Links configured for production
// 6. URL scheme configured for development
// 7. Deep links tested in all app states (cold/warm/foreground)
// 8. Analytics track deep link attribution
// 9. Deferred deep links handled for install flows
// 10. getStateFromPath unit tested for all URL patterns
