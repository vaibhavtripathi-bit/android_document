Navigation Fundamentals

Q1: Why is mobile navigation fundamentally different from web navigation?

On the web, the browser provides navigation infrastructure: URL bar, history stack, back/forward buttons, and page lifecycle. Mobile apps have none of this — they must build and manage their own navigation system. The key differences are: (1) Screen retention — web destroys previous pages, mobile keeps them mounted. (2) Transitions — mobile requires animated transitions between screens. (3) Gestures — swipe-back on iOS, hardware back on Android. (4) Parallel screens — tabs show multiple screens simultaneously. (5) Deep linking requires explicit configuration (URL schemes, Universal/App Links) rather than being built-in.

Q2: Explain the concept of "navigation state as data" in React Navigation.

React Navigation represents the entire navigation state as a serializable JavaScript object tree. Each navigator owns a state slice containing: type (stack/tab/drawer), routes array, current index, and optionally nested navigator states. This design enables: (1) State persistence — save/restore navigation state across app restarts. (2) Deep linking — convert URLs to navigation state objects. (3) Time-travel debugging — inspect and replay navigation history. (4) Programmatic navigation — construct any navigation state from code. (5) Testing — assert navigation state without rendering screens.

Q3: What is the difference between navigation.navigate() and navigation.push()?

navigate() checks if a screen with the given name already exists in the stack. If it does, it navigates to the existing instance (updating params). If not, it pushes a new screen. push() always creates a new screen instance, even if one with the same name exists. Use navigate() for singleton screens (Settings, Profile) where you want to reuse the existing instance. Use push() when you need multiple instances of the same screen (e.g., nested Detail screens in a social feed where each post opens a new Detail).

Q4: Why should you use useFocusEffect instead of useEffect for data fetching in screens?

useEffect runs on mount/unmount, but screens in a stack navigator stay mounted when another screen is pushed on top (they're blurred, not unmounted). When the user goes back, the screen is focused again but useEffect doesn't re-run because the component was never unmounted. useFocusEffect runs every time the screen gains focus, making it correct for: fetching fresh data, starting subscriptions, and resuming animations. It also runs cleanup when the screen loses focus, which is correct for pausing subscriptions and canceling requests.

Q5: How does screen lifecycle differ between stack and tab navigators?

Stack: screens mount when pushed, stay mounted when blurred (new screen pushed on top), unmount when popped. The previous screen's state is preserved. Tab: all tab screens mount on first visit (or immediately if lazy=false), stay mounted forever (never unmount), and toggle between focused/blurred states. Tab screens are never unmounted during normal use — switching tabs only changes focus. This means tab screens accumulate memory and must manage their own resource lifecycle via focus/blur events.

Q6: Compare React Navigation, react-native-navigation (Wix), and expo-router.

React Navigation: JS-based with native-stack option, largest community, good TypeScript support, works with Expo. Wix's react-native-navigation: fully native (each screen is a native ViewController/Activity), best performance for complex apps, steeper learning curve, limited Expo support. expo-router: file-based routing (like Next.js) built on React Navigation, automatic deep linking from file structure, excellent TypeScript (auto-typed routes), designed for Expo. Choice: React Navigation for most apps, Wix for apps requiring fully native navigation, expo-router for Expo-first projects.

Q7: What is the NavigationContainer and what does it manage?

NavigationContainer is the root component that manages: (1) Global navigation state — the single source of truth for all navigator states. (2) Deep link handling — converts incoming URLs to navigation state via the linking config. (3) State persistence — saves/restores state via onStateChange/initialState. (4) Navigation ref — provides programmatic navigation access from outside React components. (5) State change callbacks — enables analytics screen tracking. It must wrap all navigators and there should be exactly one NavigationContainer per app.

Q8: When would you use the navigation ref for programmatic navigation?

Navigation ref enables navigation from outside React component tree: (1) Push notification handlers — navigate to the relevant screen when a notification is tapped. (2) API interceptors — redirect to login when a 401 response is received. (3) Analytics — track current screen name on state changes. (4) Global error handlers — navigate to error screen from a centralized error boundary. (5) Background task completion — navigate to results screen when a long-running task finishes. Always check navigationRef.isReady() before navigating, as the ref may not be initialized during app startup.

Q9: How does the Android hardware back button interact with React Navigation?

React Navigation automatically handles the Android back button by popping the current screen from the stack. For tab navigators, it goes to the previously focused tab (using the history array in state). For drawers, it closes the drawer first. Custom behavior is implemented via BackHandler.addEventListener('hardwareBackPress') — return true to prevent default behavior, false to allow it. The beforeRemove event also intercepts back button presses, enabling "unsaved changes" dialogs. useFocusEffect should wrap BackHandler listeners to ensure they only apply when the screen is focused.

React Navigation Internals

Q10: Explain the Router concept in React Navigation.

Each navigator type has a Router — a pure function that takes the current state and an action, and returns the next state. StackRouter handles push, pop, replace, reset actions. TabRouter handles jumpTo, tabPress actions. DrawerRouter handles openDrawer, closeDrawer, toggleDrawer actions. Routers are composable — a tab navigator's router delegates to child stack routers for actions it doesn't handle. Custom navigators can use built-in routers or create custom ones by implementing the router protocol (getStateForAction, getInitialState, etc.).

Q11: How does React Navigation handle stale state during rehydration?

When navigation state is restored from persistence or deep link, it's marked as stale: true. Stale state may reference routes that no longer exist (removed screens) or miss new routes (added screens). React Navigation rehydrates stale state by: (1) Removing routes not in the current navigator's routeNames. (2) Adding missing routes with default params. (3) Adjusting the index if it's out of bounds. (4) Recursively rehydrating nested navigator states. This makes state persistence resilient to app updates that change the navigation structure.

Q12: How do you create a custom navigator?

Use the useNavigationBuilder hook with a router (StackRouter, TabRouter, or custom). The hook returns: state (current navigation state), navigation (navigation object), descriptors (screen render functions and options), and NavigationContent (wrapper component). You render screens using descriptors[route.key].render() and build custom UI (tab bars, sidebars) that dispatch navigation actions. createNavigatorFactory wraps your navigator component for the standard Screen/Navigator API. Custom navigators are useful for non-standard patterns like bottom sheets, floating panels, or wizard flows.

Q13: What is the beforeRemove event and how does it work?

beforeRemove fires when a screen is about to be removed from the navigator (popped from stack, tab removed). It can be prevented with e.preventDefault(), which stops the navigation action. Common use: "unsaved changes" confirmation dialog. The event's data.action contains the original navigation action, which can be re-dispatched after user confirmation. It works with all removal triggers: back button, swipe gesture, hardware back, and programmatic navigation. It does NOT fire when the entire navigator is unmounted (e.g., auth state change).

Q14: How does TypeScript integration work with React Navigation?

Define a ParamList type mapping screen names to their parameter types. Create typed navigators with createNativeStackNavigator<ParamList>(). Screen components receive typed props via NativeStackScreenProps<ParamList, 'ScreenName'>. The useNavigation hook can be typed with NativeStackNavigationProp<ParamList>. This provides: compile-time checking of screen names, parameter types, and navigation actions. For nested navigators, use CompositeScreenProps to combine parent and child param lists. Type errors catch common bugs like misspelled screen names or missing required params.

Q15: Why should route params be serializable?

Route params must be serializable (strings, numbers, booleans, plain objects) because: (1) State persistence — navigation state is JSON.stringified to AsyncStorage. Non-serializable values (functions, class instances, Dates) are lost. (2) Deep linking — URLs can only encode serializable data. (3) State rehydration — stale state is parsed from JSON. (4) Debugging — navigation state should be inspectable as plain data. Best practice: pass IDs as params and fetch full objects in the destination screen. This also prevents stale data — the destination always fetches the latest version.

Q16: How do navigation events propagate through nested navigators?

Navigation events (focus, blur, beforeRemove) propagate from child to parent navigators. When a screen in a nested stack gains focus, the focus event fires on the screen, then on the parent tab screen that contains the stack. This means a tab screen receives focus events when any screen in its nested stack is focused. Events can be listened to at any level using navigation.addListener(). The navigation.getParent() method accesses parent navigator's navigation object for cross-navigator event handling.

Q17: How do you implement screen options that depend on route params?

Three approaches: (1) Function options — options={({ route }) => ({ title: route.params.title })} on Stack.Screen. Evaluated when screen is added. (2) navigation.setOptions() — call from within the screen component to dynamically update options. Use useLayoutEffect to avoid a flash of default options. (3) screenOptions on the navigator — function form receives { route, navigation } for per-screen customization. setOptions is most flexible because it can react to state changes within the screen (e.g., showing a "Save" button only when form is dirty).

Q18: How does navigator composition work for cross-navigator navigation?

When you call navigation.navigate('ScreenName'), React Navigation traverses up the navigator tree to find the navigator that owns that screen name. If the current navigator doesn't have it, the action bubbles to the parent. This enables navigating from a deeply nested screen to a screen in a sibling navigator (e.g., from HomeStack's DetailScreen to ProfileTab). For ambiguous names, use the navigator's name as a prefix or pass a nested state object. navigation.getParent() provides explicit access to parent navigators for targeted actions.

Native Stack

Q19: What is the key architectural difference between JS stack and native stack?

JS stack renders all screens as React Native Views stacked with transforms and animates transitions using the Animated API on the JS thread. Native stack wraps each screen in a native container (UIViewController on iOS, Fragment on Android) managed by the platform's navigation controller (UINavigationController, FragmentManager). Native transitions run on the UI thread, independent of JS. This means native stack transitions are guaranteed 60fps even when the JS thread is busy, while JS stack transitions can stutter if JS is processing heavy work.

Q20: What does react-native-screens provide and why is it needed?

react-native-screens provides native screen containers (Screen, ScreenStack, ScreenStackHeaderConfig) that integrate with platform navigation controllers. It enables: (1) Native stack navigation via UINavigationController/Fragment. (2) Screen detaching — removing native views of inactive screens to save memory. (3) Screen freezing — preventing re-renders of inactive screens using React Suspense. (4) Native header configuration. Without react-native-screens, native-stack navigator cannot function. enableScreens() must be called at app startup before any navigation renders.

Q21: What are the limitations of native stack compared to JS stack?

Native stack limitations: (1) Custom transitions — only platform-provided animations (slide, fade, modal), no arbitrary custom animations. (2) Shared element transitions — limited support compared to JS stack with react-native-shared-element. (3) Header customization — native headers are platform UI, not arbitrary React components. Workaround: headerShown: false with custom header. (4) Screen count — each screen creates a native VC/Fragment with overhead (~50-100KB). (5) Intermediate screen access — can't easily render content between screens during transitions like JS stack can.

Q22: How does modal presentation work in native stack?

Native stack supports multiple presentation styles: 'card' (standard push), 'modal' (iOS sheet presentation), 'transparentModal' (transparent background), 'fullScreenModal', 'formSheet', 'containedModal'. On iOS 15+, modal presentation supports sheet detents (sheetAllowedDetents: ['medium', 'large']) with a grabber handle (sheetGrabberVisible). These map directly to UIKit's UIModalPresentationStyle. On Android, modal presentation uses a slide-up animation. The key advantage over JS-based modals is native gesture handling — the iOS sheet can be dismissed by swiping down with native physics.

Q23: How does screen detaching work in native stack?

When a new screen is pushed, react-native-screens can detach the native views of the screen underneath (remove from native hierarchy). The React component stays mounted — useState, useRef, and all JS state is preserved. When the user goes back, native views are re-created from the React component's current state. Benefits: saves ~2-5MB of native memory per detached screen. Trade-offs: brief re-creation cost when going back, potential flash as views re-attach. Controlled via detachPreviousScreen option on individual screens or detachInactiveScreens on tab navigators.

Q24: How does enableFreeze() work under the hood?

enableFreeze() uses React Suspense to prevent re-renders of inactive screens. When a screen becomes inactive, react-native-screens wraps it in a Suspense boundary and throws a special "freeze" promise. React suspends the subtree — no re-renders occur, even if context values change or parent components re-render. When the screen becomes active again, the promise resolves and React resumes rendering. This is a significant optimization: in an app with 5 tabs each with 3 screens, a theme change re-renders only the 3 active screens instead of all 15.

Q25: When should you use JS stack instead of native stack?

Use JS stack when you need: (1) Custom transition animations — e.g., a card flip, zoom, or parallax effect that platform transitions don't support. (2) Shared element transitions — react-native-shared-element works with JS stack. (3) Fully custom headers that are React components with complex interactions. (4) Content rendered between screens during transitions. (5) Cross-platform transition consistency — JS stack transitions look identical on both platforms. A common hybrid approach: native stack for the main navigation skeleton, JS stack for specific flows that need custom transitions.

Q26: How do you handle memory in deep native stack navigation?

Each native screen adds ~50-100KB overhead plus its content (views, images, state). Strategies: (1) Use navigation.replace() for sequential flows (onboarding, checkout) to avoid stack growth. (2) Reset the stack after completing a flow: navigation.reset({ index: 0, routes: [{ name: 'Home' }] }). (3) Enable screen detaching to free native views of inactive screens. (4) Clean up resources in useFocusEffect cleanup (cancel subscriptions, clear large datasets). (5) Monitor memory with Xcode Instruments / Android Profiler. (6) Set maximum stack depth limits in your navigation architecture.

Deep Linking

Q27: What are the three types of deep links and when do you use each?

Custom URL schemes (myapp://): simplest setup, not secure (any app can claim the scheme), no fallback if app not installed. Use for development and internal testing. Universal Links (iOS) / App Links (Android): use standard HTTPS URLs, require domain verification (AASA/assetlinks.json), secure (verified ownership), graceful fallback to web if app not installed. Use for production. Deferred deep links: work even when app isn't installed — store the intent, redirect to app store, deliver the link on first launch. Use for marketing campaigns, referral programs, and user acquisition flows.

Q28: How do Universal Links work on iOS?

Three components: (1) Apple App Site Association (AASA) file hosted at https://domain.com/.well-known/apple-app-site-association — contains app ID and supported URL paths. (2) Associated Domains entitlement in Xcode — applinks:domain.com. (3) AppDelegate handler — application:continueUserActivity:restorationHandler:. When a user taps an HTTPS link matching the AASA configuration, iOS checks if the associated app is installed. If yes, opens the app directly (no browser). If no, opens Safari. Apple caches the AASA file at install time and periodically refreshes it.

Q29: How does React Navigation's linking config convert URLs to navigation state?

The linking config defines a mapping from URL paths to screen names. When a URL is received, React Navigation: (1) Matches the URL against the config's screen paths. (2) Extracts path parameters (e.g., /details/:id → { id: '42' }). (3) Constructs a navigation state tree that includes all parent navigators needed to reach the target screen. (4) Resets the navigation state to the constructed state. For nested navigators, the config mirrors the navigator hierarchy. Custom getStateFromPath/getPathFromState functions can override the default conversion logic for complex URL structures.

Q30: How do you handle deep links to authenticated screens?

When a deep link targets an authenticated screen but the user isn't logged in: (1) Store the pending deep link URL. (2) Show the login screen. (3) After successful authentication, process the stored deep link. Implementation: override getStateFromPath in the linking config to check auth state. If not authenticated, save the path and return undefined (don't navigate). After login, use the navigation ref to reset state to the saved deep link's target. This ensures the user always authenticates first, then lands on the intended screen.

Q31: What is the difference between getInitialURL and the Linking 'url' event?

getInitialURL() returns the URL that launched the app from a cold start (app was not running). It's called once during app initialization. The Linking 'url' event fires when the app receives a URL while already running (warm start — app in background or foreground). Both are needed for complete deep link handling: getInitialURL for cold starts, addEventListener('url') for warm starts. React Navigation's linking config handles both automatically when configured. Common bug: only handling one case, causing deep links to fail in certain app states.

Q32: How do you test deep links across all app states?

Test matrix: (1) Cold start — app not running, URL launches app. Test with: xcrun simctl openurl (iOS), adb shell am start (Android). (2) Background — app in background, URL brings it to foreground. (3) Foreground — app active, URL received via Linking event. (4) Not installed — URL should redirect to app store (Universal/App Links) or show error (URL scheme). (5) Invalid URLs — should show a "Not Found" screen. (6) Auth-gated URLs — should redirect to login first. Unit test getStateFromPath with various URL patterns. Integration test the full flow on real devices.

Q33: Why are URL schemes not secure and what's the alternative?

Custom URL schemes (myapp://) are not secure because any app can register the same scheme. On iOS, if two apps register myapp://, behavior is undefined (Apple doesn't guarantee which app opens). On Android, the user sees a disambiguation dialog. A malicious app could intercept deep links intended for your app, potentially stealing auth tokens or sensitive data from URL parameters. The alternative: Universal Links (iOS) and App Links (Android) use HTTPS URLs with domain verification. The OS verifies that the app's developer owns the domain, preventing impersonation.

Q34: How do deferred deep links work?

Deferred deep links bridge the install gap: (1) User clicks a link (web, email, ad). (2) App not installed → redirect to App Store/Play Store. (3) User installs and opens app. (4) Third-party service (Branch.io, etc.) delivers the original link intent. The service identifies the user post-install using: device fingerprinting (IP, user agent, screen size), clipboard (iOS pasteboard), or Google Play Install Referrer (Android). The app's SDK checks for pending deep links on first launch and navigates accordingly. This enables attribution (which campaign drove the install) and personalized onboarding.

Q35: How would you architect deep linking for a large app from scratch?

Architecture: (1) Every screen gets a URL path — define this in the linking config alongside the navigator structure. (2) Use Universal Links/App Links for production, URL schemes for development. (3) Implement auth-gated deep link handling with pending link storage. (4) Add a NotFound screen as catch-all for invalid paths. (5) Unit test getStateFromPath for all URL patterns. (6) Integrate with push notification service — notification payloads include deep link URLs. (7) Add analytics tracking for deep link attribution. (8) Document all deep link paths for marketing and QA teams. (9) Set up deferred deep links for install campaigns.

Navigation Patterns

Q36: What is the recommended pattern for authentication flows?

Use conditional rendering based on auth state: {isSignedIn ? <MainScreen /> : <AuthScreen />}. This is declarative — auth state drives navigation state. Benefits: (1) Users can't navigate back to auth screens after login. (2) Token expiry automatically shows login. (3) No imperative navigation.navigate('Login') calls scattered through the app. (4) animationTypeForReplace: 'pop' makes the transition feel natural. The auth state should be managed in a context/store, with token persistence in SecureStore (not AsyncStorage). Check token validity on app launch before showing the main screen.

Q37: How do you implement a "reset tab stack on tab press" pattern?

Use the tabPress listener on Tab.Screen. Check if the tab's nested stack has screens beyond the initial screen (state.index > 0). If so, dispatch StackActions.popToTop() to reset to the root screen. This is the standard iOS behavior — tapping the active tab scrolls to top or pops to root. Implementation: access the tab's state via navigation.getState(), find the current tab's route, check its nested state index, and conditionally dispatch popToTop. Some apps also scroll FlatList to top on tab re-press using a ref.

Q38: When should you use modal presentation vs push navigation?

Modal: for creation/editing flows (new post, edit profile), settings/preferences, alerts/confirmations, content that can be dismissed (image viewer, share sheet). Modals communicate "this is a temporary context you can dismiss." Push: for drill-down into content (list → detail), sequential flows (onboarding steps), content exploration. Push communicates "you're going deeper into the content hierarchy." Technical difference: modals slide up (or use sheet presentation on iOS 15+), push slides from right. Modals can be dismissed by swiping down, push by swiping from left edge.

Q39: How do you prevent back navigation for critical flows?

Three mechanisms: (1) beforeRemove event — most flexible, can show confirmation dialog and conditionally prevent navigation. Works with all back triggers (gesture, button, hardware back). (2) gestureEnabled: false — disables swipe-back gesture on native stack. Combine with headerBackVisible: false to hide the back button. (3) BackHandler on Android — return true from hardwareBackPress listener to intercept hardware back. Use useFocusEffect to scope the listener to the active screen. Best practice: always provide an alternative way to leave the screen (Cancel button, explicit navigation) — never trap the user.

Q40: How do you implement shared element transitions in React Native?

Use react-native-shared-element with react-navigation-shared-element. Wrap shared elements in SharedElement components with matching IDs on source and destination screens. Define sharedElements static method on the destination screen to specify which elements animate and how (move, fade, morph). Requires JS stack (not native stack) because it needs control over the transition animation. Limitations: complex to set up, can cause performance issues with many shared elements, doesn't work with native stack. Alternative: use Reanimated's layout animations for simpler shared-element-like effects.

Q41: What is the recommended nesting depth for navigators?

Maximum 3 levels: typically Drawer → Tabs → Stack. Deeper nesting causes: (1) Complex navigation logic — actions must bubble through multiple levels. (2) Difficult debugging — state tree becomes deeply nested. (3) Performance overhead — each navigator adds state management overhead. (4) TypeScript complexity — composing param lists across many levels is verbose. If you need more levels, consider flattening: put all screens in a single root stack and use conditional rendering or custom navigators for non-standard patterns. Global modals should be at the root level, not nested inside tabs.

Q42: How do you implement screen tracking for analytics?

Centralized approach: use onStateChange on NavigationContainer with navigationRef.getCurrentRoute(). Track when the current route name changes. Store the previous route name in a ref. This captures all navigation changes in one place. Per-screen approach: use useFocusEffect in individual screens. More granular but requires adding tracking to every screen. Best practice: centralized tracking for screen views, per-screen tracking for specific interactions. Include route params in tracking data (sanitized — no PII). Use a middleware pattern to enrich tracking with user context.

Q43: How do you handle bottom sheet navigation?

Two approaches: (1) Navigation-integrated — present bottom sheet as a transparentModal screen with @gorhom/bottom-sheet inside. The sheet is part of the navigation state, supports deep linking, and is dismissed via navigation.goBack(). (2) Standalone — use bottom-sheet as a regular component controlled by local state, not part of navigation. Better for contextual sheets (action menus, filters) that don't need deep link support. The navigation approach is better for sheets that represent distinct "screens" (comments, share options), while standalone is better for UI overlays.

Navigation Performance

Q44: What is the single most impactful navigation performance optimization?

Using native stack (@react-navigation/native-stack) instead of JS stack. Native transitions run on the UI thread, independent of JS thread load. This guarantees 60fps transitions even when the JS thread is busy with data fetching, rendering, or computation. The difference is dramatic on lower-end devices where JS thread contention is more severe. Secondary to this: enabling screen freezing (enableFreeze) to prevent unnecessary re-renders of inactive screens, which reduces JS thread work across the entire app.

Q45: How does InteractionManager help with navigation performance?

InteractionManager.runAfterInteractions() defers execution until all animations and transitions complete. During a push transition, the JS thread is busy with the animation (JS stack) and new screen rendering. Heavy work (data fetching, complex computations, large list rendering) competes for JS thread time, causing animation stutter. runAfterInteractions queues the heavy work until the transition finishes. Pattern: render a lightweight skeleton immediately, then use runAfterInteractions to load data and render full content. This ensures smooth transitions at the cost of slightly delayed content appearance.

Q46: How does screen freezing reduce JS thread work?

Without freezing, inactive screens still re-render when: context values change (theme, locale, auth state), parent components re-render, or global state updates. In an app with 5 tabs each with 3 screens, a theme change triggers 15 re-renders. With freezing (enableFreeze), only the 3 active screens re-render — an 80% reduction. Freezing uses React Suspense to suspend inactive screen subtrees. The trade-off: frozen screens show stale data when unfocused. Use useFocusEffect to refresh data when a frozen screen becomes active.

Q47: What are the memory implications of deep navigation stacks?

Each screen adds: React component tree (~5-15MB), native views (~2-5MB), images/media (~5-50MB), data/state (~1-10MB). A 10-screen deep stack can consume 100-500MB. iOS kills apps at ~1-2GB. Mitigations: (1) Screen detaching — frees native views of inactive screens. (2) navigation.replace() for sequential flows. (3) navigation.reset() after completing flows. (4) Clean up resources on blur (cancel subscriptions, clear large datasets). (5) Use thumbnails instead of full images in background screens. (6) Set architectural limits on stack depth.

Q48: How do you preload data for the next screen to hide latency?

Prefetch on press-in (not press): when the user touches a list item, start fetching the detail data immediately. By the time the press completes and the transition starts, data may already be cached. Use React Query's prefetchQuery or SWR's preload. For images: call Image.prefetch() or FastImage.preload() for the next screen's hero images. For complex screens: consider pre-rendering a hidden instance. The key insight: the ~200ms between press-in and navigation gives enough time to start (and often complete) a network request.

Q49: What performance budgets should you set for navigation?

Screen mount → first paint: < 100ms (show skeleton). Screen mount → interactive: < 300ms (data loaded, content rendered). Transition animation: 60 FPS (16.6ms per frame). Tab switch (already mounted): < 50ms. Deep link → target screen: < 500ms. Memory per screen: < 20MB average. Stack depth: < 8 screens before reset. These budgets should be monitored in CI with performance tests. Profile on the lowest-spec target device (not the developer's latest iPhone). Set up alerts for budget violations.

Q50: How do you profile navigation performance?

Tools: (1) React DevTools Profiler — record navigation actions, check which components rendered and how long. (2) Flipper Performance plugin — monitor JS thread FPS during transitions. (3) Xcode Instruments (iOS) — Time Profiler for JS/native thread work, Allocations for memory. (4) Android Studio Profiler — CPU, memory, and frame rendering. (5) Custom timing — measure mount-to-interactive with useRef(Date.now()) and InteractionManager. (6) React Navigation's onStateChange for transition timing. Focus on: transition FPS, time-to-interactive for new screens, memory growth during navigation.

Q51: How does lazy loading work for tab navigators and what are the trade-offs?

With lazy: true, tab content is only rendered when the user first visits that tab. Without lazy, all tab contents render on initial mount. Benefits: faster initial app load (only one tab renders), lower initial memory usage. Trade-offs: first visit to a tab shows a loading state (use lazyPlaceholder for a smooth experience), data fetching starts later (on first visit instead of app launch). Optimization: prefetch data for likely-to-be-visited tabs in the background after initial render, so data is ready when the user switches tabs.

Q52: How would you architect navigation for a large-scale app with 50+ screens?

Architecture: (1) Feature-based navigator modules — each feature (auth, home, profile, settings) owns its own stack navigator, exported as a component. (2) Root navigator composes feature navigators — Drawer → Tabs → Feature stacks. (3) Centralized linking config — one file mapping all URLs to screens. (4) Typed navigation — strict ParamList types per feature, composed at the root. (5) Navigation utilities — shared helpers for common patterns (reset, replace, deep link handling). (6) Performance defaults — native stack, screen freezing, lazy tabs enabled globally. (7) Testing — unit tests for linking config, integration tests for critical flows. (8) Documentation — navigation map showing all screens and their relationships.
