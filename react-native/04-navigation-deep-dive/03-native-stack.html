Native Stack

@react-navigation/native-stack leverages native navigation controllers (UINavigationController on iOS, Fragment on Android) for truly native transitions and performance. Understanding the trade-offs between native and JS stacks is critical for navigation architecture decisions.

1. Native Stack Overview

JS Stack (@react-navigation/stack): ═══════════════════════════════════ React Components (all screens mounted as React views) ├─ Screen A (visible, z-index: 0) ├─ Screen B (visible, z-index: 1) └─ Screen C (visible, z-index: 2, current) Transitions: Animated API / Reanimated (JS-driven) All screens are React Native Views stacked with transforms Native Stack (@react-navigation/native-stack): ═══════════════════════════════════════════════ iOS: UINavigationController ├─ UIViewController A (Screen A) ├─ UIViewController B (Screen B) └─ UIViewController C (Screen C, current) Android: FragmentManager ├─ Fragment A (Screen A) ├─ Fragment B (Screen B) └─ Fragment C (Screen C, current) Transitions: Native platform animations (60fps guaranteed) Each screen is a native view controller / fragment

1.1 Setup

import { createNativeStackNavigator } from '@react-navigation/native-stack';

// Requires react-native-screens to be installed and enabled
import { enableScreens } from 'react-native-screens';
enableScreens();

const Stack = createNativeStackNavigator();

<Stack.Navigator>
  <Stack.Screen name="Home" component={HomeScreen} />
  <Stack.Screen name="Details" component={DetailsScreen} />
</Stack.Navigator>

// Each Stack.Screen creates a native screen container:
// iOS: RNSScreenView wrapping a UIViewController
// Android: ScreenFragment wrapping a Fragment

3. Android: Fragment

// Native stack on Android uses Fragment transactions
// Each screen is a ScreenFragment managed by FragmentManager

// - Material Design transitions
// - Status bar color per screen

// Android-specific screen options:
<Stack.Screen
  name="Home"
  component={HomeScreen}

5. Performance vs JS Stack

MetricJS Stack (@react-navigation/stack)Native Stack (native-stack)
Transition FPS30-60 FPS (depends on JS thread)60 FPS (native, guaranteed)
Gesture responsivenessMay lag if JS busyAlways responsive (native)
Screen mountingAll screens are React viewsEach screen is native container
Memory per screenLower (shared view hierarchy)Higher (native VC/Fragment overhead)
Custom transitionsFull control (any animation)Limited to platform options
Shared elementsPossible (with effort)Limited support
Header customizationFull (React component)Limited (native header)
JS thread dependencyHigh (transitions block on JS)Low (transitions are native)
Architect Insight

Native stack should be the default choice for most apps. The 60fps guaranteed transitions and native gesture handling provide a significantly better user experience. Only fall back to JS stack when you need: (1) fully custom transition animations, (2) shared element transitions, or (3) complete header customization that native headers can't provide. Many apps use native stack for the main navigation and JS stack for specific flows that need custom transitions.

6. Limitations

// Limitation 1: Custom transitions
// Native stack only supports platform-provided animations
// You cannot create arbitrary custom transition animations
// like you can with JS stack's TransitionPresets

// Limitation 2: Shared element transitions
// Native stack has limited shared element support
// For complex shared element transitions, use:
// - react-native-shared-element (with JS stack)
// - react-navigation-shared-element

// Limitation 3: Header customization
// Native headers are platform-native UI
// You can customize title, buttons, colors
// But cannot use arbitrary React components as headers
// Workaround: headerShown: false + custom header component

<Stack.Screen
  name="Custom"
  component={CustomScreen}
  options={{ headerShown: false }}
/>

const CustomScreen = () => (
  <SafeAreaView>
    <CustomHeader />
    <ScreenContent />
  </SafeAreaView>
);

// Limitation 4: Screen count
// Each native screen creates a UIViewController/Fragment
// Deep stacks (10+ screens) consume more memory than JS stack
// Monitor memory in deep navigation flows

7. react-native-screens

react-native-screens is the native module that powers native-stack. It provides native screen containers that integrate with platform navigation controllers.

// react-native-screens provides:
// 1. Screen — native screen container (UIViewController/Fragment)
// 2. ScreenStack — native navigation stack
// 3. ScreenStackHeaderConfig — native header configuration
// 4. enableScreens() — opt-in to native screens
// 5. enableFreeze() — opt-in to screen freezing

import { enableScreens, enableFreeze } from 'react-native-screens';

// Enable at app startup (before any navigation renders)
enableScreens(true);

// Enable screen freezing (prevents re-renders of inactive screens)
enableFreeze(true);

// Screen states in react-native-screens:
// 'active'     — fully rendered, visible
// 'transitioning' — during push/pop animation
// 'detached'   — native view removed, React component alive

// Detaching inactive screens saves native memory:
// When Screen B is pushed on top of Screen A,
// Screen A's native view can be detached (removed from hierarchy)
// but its React component stays mounted (state preserved)
// When user goes back, native view is re-attached

8. Screen Lifecycle with Native Stack

Native Stack Screen Lifecycle: Push Screen B on top of Screen A: Screen A: focus → blur → (native view may detach) Screen B: mount → focus → (native view attached) Pop Screen B (go back to A): Screen B: blur → beforeRemove → unmount Screen A: (native view re-attach) → focus Key difference from JS stack: JS stack: all screens always have native views Native stack: inactive screens may have native views detached This saves significant native memory in deep stacks 
// Screen lifecycle hooks
const ScreenWithLifecycle = ({ navigation }) => {
  // Mount/unmount (standard React)
  useEffect(() => {
    console.log('Screen mounted');
    return () => console.log('Screen unmounted');
  }, []);

  // Focus/blur (navigation-specific)
  useEffect(() => {
    const unsub = navigation.addListener('focus', () => {
      console.log('Screen focused — refresh data');
    });
    return unsub;
  }, [navigation]);

  // transitionStart/transitionEnd (native stack specific)
  useEffect(() => {
    const unsub = navigation.addListener('transitionEnd', (e) => {
      console.log('Transition complete, closing:', e.data.closing);
    });
    return unsub;
  }, [navigation]);
};

9. Memory Management

// Native stack memory considerations:

// 1. Each screen creates a native container
//    iOS: UIViewController (~50-100KB overhead)
//    Android: Fragment (~30-80KB overhead)

// 2. Screen content (React views) adds to memory
//    A complex screen: ~2-5MB (views + images + state)

// 3. Deep stacks accumulate memory
//    10 screens deep: ~20-50MB just for navigation

// Mitigation strategies:

// Strategy 1: Detach inactive screens
// react-native-screens detaches native views of inactive screens
// React components stay mounted (state preserved)
// Native memory freed, JS memory retained

// Strategy 2: Freeze inactive screens
import { enableFreeze } from 'react-native-screens';
enableFreeze(true);
// Prevents re-renders of inactive screens
// Uses React.Suspense under the hood
// Inactive screens don't respond to context changes

// Strategy 3: Limit stack depth
// Use navigation.replace() instead of navigate() when appropriate
// Use navigation.popToTop() to clear the stack
// Reset navigation state for flows that shouldn't accumulate

navigation.reset({
  index: 0,
  routes: [{ name: 'Home' }],
});

// Strategy 4: Monitor memory
// Use Xcode Instruments (iOS) / Android Profiler
// Watch for memory growth during navigation
// Set alerts for memory thresholds
Memory Leak Pattern

A common memory leak in native stack: screens that start timers, subscriptions, or network requests in useEffect but don't clean up on unmount. Because screens stay mounted when blurred (not unmounted), these resources accumulate. Use useFocusEffect for resources that should only be active when the screen is visible, and always return cleanup functions from useEffect.

10. Architect Considerations

Staff-Level Mental Model

Native stack is the right default for production apps. The performance difference is not marginal — it's the difference between "feels like a web app" and "feels like a native app." The key trade-off is customization vs performance: native stack constrains your transitions and headers to platform conventions, which is usually what you want. For the 10% of screens that need custom behavior, use JS stack selectively or hide the native header and build your own. The architecture should be: native stack for the main navigation skeleton, with escape hatches for custom flows.
