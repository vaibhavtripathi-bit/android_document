1. Screen Lazy Loading

By default, tab navigators render all tab screens on first mount. Lazy loading defers rendering until the user first visits a tab.

// Tab navigator lazy loading
<Tab.Navigator
  screenOptions={{
    lazy: true,  // Don't render tab content until first visit
  }}
>
  <Tab.Screen name="Home" component={HomeScreen} />
  <Tab.Screen name="Search" component={SearchScreen} />
  <Tab.Screen name="Profile" component={ProfileScreen} />
</Tab.Navigator>

// With lazy=true:
// Initial render: only Home tab content is rendered
// User taps Search: Search tab content renders for first time
// User taps Profile: Profile tab content renders for first time
// After first visit, tab content stays mounted

// Without lazy (default in some versions):
// Initial render: ALL tab contents render simultaneously
// This can cause a slow initial load if tabs have heavy content

// Custom lazy placeholder:
<Tab.Navigator
  screenOptions={{
    lazy: true,
    lazyPlaceholder: () => (
      <View style={styles.placeholder}>
        <ActivityIndicator size="large" />
      </View>
    ),
  }}
/>
Stack Navigator Lazy Behavior

Stack navigators are inherently lazy — screens are only rendered when they're pushed onto the stack. There's no lazy prop for stack navigators because screens are created on-demand by definition. The performance concern with stacks is the opposite: screens stay mounted after being pushed, accumulating memory.

2. Detaching Inactive Screens

react-native-screens can detach the native views of inactive screens while keeping their React components mounted. This saves native memory without losing React state.

Screen Detaching: Without detaching: With detaching: ═══════════════════ ═══════════════════ Native hierarchy: Native hierarchy: ScreenStack ScreenStack ├─ Screen A (native views) ├─ Screen A (DETACHED — no native views) ├─ Screen B (native views) ├─ Screen B (DETACHED — no native views) └─ Screen C (native views, visible) └─ Screen C (native views, visible) React tree (same in both): ├─ Screen A component (mounted, state preserved) ├─ Screen B component (mounted, state preserved) └─ Screen C component (mounted, state preserved) Memory savings: ~2-5MB per detached screen (native views freed) 
// Enable screen detaching
import { enableScreens } from 'react-native-screens';
enableScreens(true);

// Per-screen detaching control:
<Stack.Screen
  name="HeavyScreen"
  component={HeavyScreen}
  options={{
    detachPreviousScreen: true,  // Detach screen below this one
  }}
/>

// Tab navigator detaching:
<Tab.Navigator
  screenOptions={{
    detachInactiveScreens: true,  // Detach inactive tab screens
  }}
>
  {/* tabs */}
</Tab.Navigator>

// What happens when a detached screen is re-activated:
// 1. Native views are re-created from React component state
// 2. Layout is recomputed
// 3. There may be a brief flash as views are re-attached
// 4. React state (useState, useRef) is preserved
// 5. Effects don't re-run (component was never unmounted)

3. Freezing Inactive Screens

Screen freezing prevents React re-renders on inactive screens. Even if a context value changes or a parent re-renders, frozen screens don't update until they become active again.

import { enableFreeze } from 'react-native-screens';
enableFreeze(true);

// How freezing works:
// 1. Uses React.Suspense under the hood
// 2. When a screen becomes inactive, it's wrapped in a Suspense boundary
// 3. A special "freeze" promise is thrown (like lazy loading)
// 4. React suspends the subtree — no re-renders
// 5. When screen becomes active, the promise resolves
// 6. React resumes rendering the subtree

// Benefits:
// - Inactive screens don't respond to context changes
// - Inactive screens don't re-render when parent state changes
// - Reduces JS thread work significantly in apps with many screens

// Example impact:
// App with 5 tabs, each with 3 screens deep
// Without freezing: theme change re-renders all 15 screens
// With freezing: theme change re-renders only 3 active screens
// 80% reduction in re-render work
Freezing Caveats

Frozen screens don't receive context updates until they're unfrozen. If your app relies on screens updating in the background (e.g., a chat screen receiving new messages while on another tab), freezing may cause stale UI when the user returns. Use useFocusEffect to refresh data when a frozen screen becomes active again.

4. Transition Performance

// Transition performance: JS stack vs Native stack

// JS Stack transition performance issues:
// 1. Transition animation runs on JS thread
// 2. If JS thread is busy (data fetching, rendering), animation stutters
// 3. New screen renders during transition, competing for JS thread

// Fix 1: Use InteractionManager to defer heavy work
import { InteractionManager } from 'react-native';

const HeavyScreen = () => {
  const [isReady, setIsReady] = useState(false);

  useEffect(() => {
    const task = InteractionManager.runAfterInteractions(() => {
      // Heavy work: data processing, complex rendering
      setIsReady(true);
    });

    return () => task.cancel();
  }, []);

  if (!isReady) return <ScreenPlaceholder />;
  return <HeavyContent />;
};

// Fix 2: Use native stack (transitions don't depend on JS)
// Native stack transitions run on the UI thread
// Even if JS is completely blocked, transitions are smooth

// Fix 3: Minimize initial render of new screen
// Render a lightweight skeleton first
// Load data and heavy content after transition completes
const OptimizedScreen = () => {
  const [data, setData] = useState(null);

  useEffect(() => {
    const task = InteractionManager.runAfterInteractions(async () => {
      const result = await fetchData();
      setData(result);
    });
    return () => task.cancel();
  }, []);

  return data ? <FullContent data={data} /> : <Skeleton />;
};

5. Reducing JS Thread Work During Transitions

// Problem: JS thread is shared between transition animation and screen rendering
// During a push transition:
// - Transition animation needs JS thread (JS stack) or doesn't (native stack)
// - New screen component mounts and renders
// - Data fetching may start
// - All competing for the same JS thread

// Strategy 1: Lightweight initial render
const Screen = () => {
  const [phase, setPhase] = useState('skeleton');

  useEffect(() => {
    // Phase 1: Show skeleton immediately (fast render)
    // Phase 2: After transition, render content
    InteractionManager.runAfterInteractions(() => {
      setPhase('content');
    });
  }, []);

  if (phase === 'skeleton') return <ScreenSkeleton />;
  return <ScreenContent />;
};

// Strategy 2: Avoid heavy computations in render
// Bad: computing derived data during render
const BadScreen = ({ items }) => {
  const sorted = items.sort((a, b) => b.date - a.date); // Expensive!
  return <FlatList data={sorted} />;
};

// Good: memoize expensive computations
const GoodScreen = ({ items }) => {
  const sorted = useMemo(
    () => [...items].sort((a, b) => b.date - a.date),
    [items]
  );
  return <FlatList data={sorted} />;
};

// Strategy 3: Defer non-critical effects
useEffect(() => {
  // Critical: fetch screen data
  fetchScreenData();

  // Non-critical: defer analytics, prefetch
  requestAnimationFrame(() => {
    analytics.trackScreenView('Details');
    prefetchRelatedContent();
  });
}, []);

6. Native Driver for Transitions

// Native driver offloads animation to the UI thread
// JS thread is free for other work during animation

// React Navigation JS stack uses Animated API for transitions
// By default, it uses the native driver for opacity and transform

// Custom transition with native driver:
const forFadeNative = ({ current }) => ({
  cardStyle: {
    opacity: current.progress,  // opacity supports native driver
  },
});

// Properties that support native driver:
// ✓ opacity
// ✓ transform (translateX, translateY, scale, rotate)
// ✗ backgroundColor (layout change)
// ✗ width, height (layout change)
// ✗ borderRadius (layout change)

// Native stack doesn't use Animated API at all
// Transitions are entirely native platform animations
// This is why native stack is always smoother

// Reanimated for custom native-driven transitions:
import Animated, { FadeIn, SlideInRight } from 'react-native-reanimated';

const AnimatedScreen = () => (
  <Animated.View entering={FadeIn.duration(300)} style={{ flex: 1 }}>
    <ScreenContent />
  </Animated.View>
);

7. Preloading Screens

// Preload screen data before navigation

// Pattern 1: Prefetch data on hover/press-in
const ListItem = ({ item, navigation }) => {
  const prefetch = useCallback(() => {
    queryClient.prefetchQuery(['details', item.id], () =>
      fetchDetails(item.id)
    );
  }, [item.id]);

  return (
    <Pressable
      onPressIn={prefetch}  // Start prefetch on press-in
      onPress={() => navigation.navigate('Details', { id: item.id })}
    >
      <Text>{item.title}</Text>
    </Pressable>
  );
};

// Pattern 2: Preload images for next screen
import { Image } from 'react-native';

const preloadDetailsScreen = (item) => {
  Image.prefetch(item.heroImageUrl);
  Image.prefetch(item.authorAvatarUrl);
};

// Pattern 3: Prerender screen component
// Keep a hidden instance of the next screen ready
// This is an advanced pattern — use sparingly
const PrerenderedScreen = ({ shouldPrerender, children }) => (
  <View style={shouldPrerender ? styles.hidden : styles.visible}>
    {children}
  </View>
);

10. Architect Considerations

Staff-Level Mental Model

Navigation performance is about managing the tension between responsiveness and resource usage. Key principles: (1) Native stack by default — transitions that don't depend on JS are always smoother. (2) Defer heavy work — use InteractionManager to delay data fetching and complex rendering until after transitions complete. (3) Freeze and detach — inactive screens should consume minimal resources. (4) Memory budgeting — calculate the memory cost of your navigation depth and set limits. (5) Preload intelligently — prefetch data on press-in, not on mount, to hide latency. (6) Profile regularly — navigation performance degrades gradually as features are added; continuous profiling catches regressions early.
