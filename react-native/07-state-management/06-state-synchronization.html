1. JS ↔ Native State Sync

React Native apps have state in two runtimes: the JavaScript VM and the native platform. Keeping these in sync is a fundamental challenge, especially with the new architecture where JSI enables synchronous access.

1.1 TurboModule State Sync

// TurboModule exposes native state to JS synchronously via JSI
export interface Spec extends TurboModule {
  getDeviceState(): {
    batteryLevel: number;
    isCharging: boolean;
    thermalState: 'nominal' | 'fair' | 'serious' | 'critical';
    availableMemory: number;
  };

  addStateListener(callback: (state: DeviceState) => void): void;
  removeStateListener(): void;
}

// JS hook that syncs native device state into Zustand
function useDeviceStateSync() {
  const setDeviceState = useDeviceStore((s) => s.setDeviceState);

  useEffect(() => {
    const initial = DeviceModule.getDeviceState();
    setDeviceState(initial);

    DeviceModule.addStateListener((state) => {
      setDeviceState(state);
    });

    return () => DeviceModule.removeStateListener();
  }, []);
}

2. Shared State in Brownfield Apps

Brownfield apps embed React Native within an existing native app. State must flow between the native host and RN surfaces, often through multiple RN root views.

3. Multi-Surface State Sync

Large apps may run multiple React Native root views simultaneously (e.g., a main app and a notification extension). Each has its own JS runtime, so in-memory state is not shared.

// Cross-surface state sync via MMKV + listeners
class CrossSurfaceStateSync {
  private storage: MMKV;
  private listeners = new Map<string, Set<(value: any) => void>>();

  constructor() {
    this.storage = new MMKV({ id: 'cross-surface' });

    this.storage.addOnValueChangedListener((key) => {
      const callbacks = this.listeners.get(key);
      if (callbacks) {
        const value = this.get(key);
        callbacks.forEach(cb => cb(value));
      }
    });
  }

  set(key: string, value: any) {
    this.storage.set(key, JSON.stringify(value));
  }

  get(key: string): any {
    const raw = this.storage.getString(key);
    return raw ? JSON.parse(raw) : undefined;
  }

  subscribe(key: string, callback: (value: any) => void) {
    if (!this.listeners.has(key)) {
      this.listeners.set(key, new Set());
    }
    this.listeners.get(key)!.add(callback);
    return () => this.listeners.get(key)!.delete(callback);
  }
}

4. Background State Updates

Background tasks (fetch, processing, location updates) can modify state while the UI is not visible. When the app returns to foreground, the UI must reflect these changes.

// Background fetch handler updates persistent state
import BackgroundFetch from 'react-native-background-fetch';

BackgroundFetch.configure(
  {
    minimumFetchInterval: 15, // minutes
    stopOnTerminate: false,
    startOnBoot: true,
  },
  async (taskId) => {
    // JS context may be fresh — no in-memory state
    const storage = new MMKV({ id: 'app' });
    const token = storage.getString('auth.token');

    if (token) {
      const updates = await api.checkForUpdates(token);
      storage.set('pending.updates', JSON.stringify(updates));
      storage.set('last.background.fetch', Date.now());
    }

    BackgroundFetch.finish(taskId);
  },
  (taskId) => {
    BackgroundFetch.finish(taskId);
  }
);

// Foreground hook reads background-updated state
function useBackgroundUpdates() {
  const [updates, setUpdates] = useState<Update[]>([]);

  useAppForeground(() => {
    const storage = new MMKV({ id: 'app' });
    const raw = storage.getString('pending.updates');
    if (raw) {
      setUpdates(JSON.parse(raw));
      storage.delete('pending.updates');
    }
  });

  return updates;
}

5. Push Notification State

import messaging from '@react-native-firebase/messaging';

// Foreground notification → update in-memory state
messaging().onMessage(async (remoteMessage) => {
  const { type, data } = remoteMessage.data;

  switch (type) {
    case 'new_message':
      useChatStore.getState().addMessage(data);
      break;
    case 'badge_update':
      useNotificationStore.getState().setBadgeCount(data.count);
      break;
  }
});

// Background notification → persist to storage (JS may not be running)
messaging().setBackgroundMessageHandler(async (remoteMessage) => {
  const storage = new MMKV({ id: 'notifications' });
  const queue = JSON.parse(storage.getString('pending') ?? '[]');
  queue.push(remoteMessage.data);
  storage.set('pending', JSON.stringify(queue));
});

// Notification tap → navigate to relevant screen
messaging().onNotificationOpenedApp((remoteMessage) => {
  const { type, id } = remoteMessage.data;
  handleDeepNavigation(type, id);
});

// Cold start from notification tap
messaging().getInitialNotification().then((remoteMessage) => {
  if (remoteMessage) {
    const { type, id } = remoteMessage.data;
    useAppStore.getState().setPendingNavigation({ type, id });
  }
});
Key Insight

Push notifications create three distinct state update paths: (1) foreground — update in-memory store directly; (2) background — persist to MMKV since JS runtime may be suspended; (3) cold start from tap — set pending navigation before the app mounts. Each path requires different handling, and missing any one creates a broken user experience.

6. Deep Link State Restoration

// Deep link state restoration with auth guard
function useDeepLinkHandler() {
  const isAuthenticated = useAuthStore((s) => !!s.token);
  const pendingLink = useAppStore((s) => s.pendingDeepLink);
  const clearPending = useAppStore((s) => s.clearPendingDeepLink);

  useEffect(() => {
    if (pendingLink && isAuthenticated) {
      const state = buildNavigationState(pendingLink);
      navigationRef.resetRoot(state);
      clearPending();
    }
  }, [pendingLink, isAuthenticated]);

  useEffect(() => {
    const subscription = Linking.addEventListener('url', ({ url }) => {
      const parsed = parseDeepLink(url);
      if (!parsed) return;

      if (isAuthenticated) {
        navigateToDeepLink(parsed);
      } else {
        useAppStore.getState().setPendingDeepLink(parsed);
      }
    });

    return () => subscription.remove();
  }, [isAuthenticated]);
}

function buildNavigationState(link: ParsedDeepLink): NavigationState {
  return {
    index: 1,
    routes: [
      { name: 'HomeTabs' },
      { name: link.screen, params: link.params },
    ],
  };
}

7. State Machine Patterns

Complex mobile flows (onboarding, checkout, media upload) benefit from explicit state machines that make impossible states unrepresentable.

import { createMachine, assign } from 'xstate';
import { useMachine } from '@xstate/react';

const uploadMachine = createMachine({
  id: 'upload',
  initial: 'idle',
  context: {
    file: null as File | null,
    progress: 0,
    error: null as string | null,
    url: null as string | null,
  },
  states: {
    idle: {
      on: { SELECT_FILE: { target: 'selected', actions: 'setFile' } },
    },
    selected: {
      on: {
        UPLOAD: 'uploading',
        CANCEL: { target: 'idle', actions: 'clearFile' },
      },
    },
    uploading: {
      invoke: {
        src: 'uploadFile',
        onDone: { target: 'success', actions: 'setUrl' },
        onError: { target: 'error', actions: 'setError' },
      },
      on: {
        PROGRESS: { actions: 'updateProgress' },
        CANCEL: { target: 'idle', actions: 'abortUpload' },
      },
    },
    success: {
      on: { RESET: 'idle' },
    },
    error: {
      on: {
        RETRY: 'uploading',
        CANCEL: 'idle',
      },
    },
  },
});

function UploadScreen() {
  const [state, send] = useMachine(uploadMachine);

  return (
    <View>
      {state.matches('idle') && <FilePicker onSelect={(f) => send({ type: 'SELECT_FILE', file: f })} />}
      {state.matches('uploading') && <ProgressBar progress={state.context.progress} />}
      {state.matches('error') && <RetryButton onPress={() => send('RETRY')} />}
      {state.matches('success') && <SuccessView url={state.context.url} />}
    </View>
  );
}
Architect Insight

State machines eliminate entire categories of bugs in mobile apps. A file upload can't be "uploading" and "idle" simultaneously. A checkout flow can't skip the payment step. An onboarding wizard can't go backward past the terms acceptance. By making impossible states unrepresentable, you eliminate defensive coding and the subtle bugs that come from ad-hoc boolean flags.

8. Undo/Redo in Mobile Apps

// Command pattern for undo/redo
interface Command<T> {
  execute: () => T;
  undo: () => void;
  description: string;
}

class UndoManager {
  private undoStack: Command<any>[] = [];
  private redoStack: Command<any>[] = [];
  private maxHistory = 50;

  execute<T>(command: Command<T>): T {
    const result = command.execute();
    this.undoStack.push(command);
    this.redoStack = []; // Clear redo on new action

    if (this.undoStack.length > this.maxHistory) {
      this.undoStack.shift();
    }

    return result;
  }

  undo(): boolean {
    const command = this.undoStack.pop();
    if (!command) return false;

    command.undo();
    this.redoStack.push(command);
    return true;
  }

  redo(): boolean {
    const command = this.redoStack.pop();
    if (!command) return false;

    command.execute();
    this.undoStack.push(command);
    return true;
  }

  get canUndo(): boolean { return this.undoStack.length > 0; }
  get canRedo(): boolean { return this.redoStack.length > 0; }
}

9. Architect Considerations

Staff-Level Mental Model

State synchronization in React Native is a distributed systems problem. You have multiple runtimes (JS, native, background tasks), multiple persistence layers (memory, MMKV, Keychain), and multiple entry points (app launch, deep link, push notification, background fetch). A Staff engineer designs synchronization with clear sources of truth for each state domain, explicit conflict resolution strategies, and eventual consistency guarantees. The key insight: treat persistent storage (MMKV) as the shared bus between all these contexts, and in-memory stores as derived caches that rehydrate from it.
