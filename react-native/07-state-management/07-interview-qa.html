State in React Native

Q1: How does state management in React Native differ from web React?

A: RN introduces mobile-specific state concerns absent on web: (1) App lifecycle — apps can be killed by the OS while backgrounded with no warning event, requiring proactive persistence; (2) Navigation state — stack-based navigation maintains a tree of mounted screens, unlike URL-based web routing; (3) Memory pressure — the OS can send memory warnings requiring cache eviction; (4) Platform state — permissions, device sensors, and connectivity have no web equivalent; (5) Persistence — no localStorage/sessionStorage; you need MMKV, SQLite, or Keychain; (6) Background updates — push notifications and background fetch can modify state while the UI is suspended.

Q2: What are the categories of state in a React Native app and how should each be managed?

A: Four categories: (1) Ephemeral state (form inputs, modal visibility, animation values) — useState/useReducer, lost on unmount, that's fine; (2) Session state (auth, navigation stack, feature flags) — global store (Zustand/Redux), lives in memory for the app session; (3) Server state (user profiles, feed data, notifications) — TanStack Query or Apollo, cached with stale-while-revalidate; (4) Persistent state (preferences, offline queue, drafts) — MMKV/SQLite, survives app kills. A Staff engineer assigns each piece of state to exactly one category and uses the appropriate tool for that category.

Q3: How do you handle state when the app transitions between foreground and background?

A: Use AppState.addEventListener('change') to detect transitions. On background: persist critical state to MMKV, pause expensive operations (WebSocket, location tracking), cancel non-essential network requests. On foreground: check elapsed time since backgrounding — if beyond a staleness threshold (e.g., 5 minutes), invalidate server state caches (TanStack Query's focusManager), refresh auth tokens, and re-establish WebSocket connections. On iOS, handle the intermediate inactive state during system dialogs.

Q4: How do you persist and restore navigation state across app restarts?

A: React Navigation's NavigationContainer accepts initialState and onStateChange props. On state change, serialize the navigation state to MMKV. On app launch, read the persisted state and pass it as initialState. Critical: validate the persisted state against the current navigator structure — if screens were added/removed between app versions, the persisted state may reference non-existent screens. Fall back to the default initial route on validation failure.

Q5: How does memory pressure affect state management in React Native?

A: The OS sends memory warnings at increasing severity levels. Your app should respond by: (1) evicting non-essential caches (image cache, inactive query results); (2) unmounting off-screen navigation stacks via detachInactiveScreens; (3) reducing TanStack Query's gcTime dynamically; (4) at critical levels, persisting emergency state snapshots to disk. React Navigation keeps all stack screens mounted by default — a deep stack of 10+ screens with images can consume hundreds of MB. Use react-native-screens to detach inactive screens.

Redux in React Native

Q6: Why use Redux Toolkit over vanilla Redux in React Native?

A: RTK eliminates boilerplate while enforcing best practices: createSlice generates action creators and reducers from a single definition; Immer enables direct mutation syntax for immutable updates; createAsyncThunk handles async lifecycle (pending/fulfilled/rejected); createEntityAdapter provides normalized CRUD operations with memoized selectors. RTK also configures the store with sensible defaults (thunk middleware, serializable check, dev tools). The reduction in boilerplate is significant for mobile where bundle size and developer velocity matter.

Q7: How does Redux Persist work with MMKV in React Native?

A: Redux Persist serializes the Redux store to a storage engine and rehydrates on app launch. The MMKV storage adapter wraps MMKV's synchronous API in Promises (required by Redux Persist's interface). Since MMKV reads are ~0.01ms via JSI (vs ~5-10ms for AsyncStorage via the bridge), rehydration completes in a single microtask tick. Configure with whitelist/blacklist to control which slices persist, nested persist configs for different encryption levels, and createMigrate for schema evolution across app versions.

Q8: How do you optimize Redux selector performance in React Native?

A: Use createSelector from Reselect (included in RTK) to memoize derived data. Key patterns: (1) Never create new objects/arrays in selectors — use memoization; (2) Use factory selectors (makeSelectPostsByAuthor) for parameterized selectors so each component instance gets its own memoized selector; (3) Normalize state with createEntityAdapter for O(1) lookups; (4) Select primitives when possible (no equality issues). In RN, unnecessary re-renders are costlier than web because they trigger shadow tree diffs and native view updates.

Q9: How would you implement offline-first with Redux in React Native?

A: Create an offline middleware that intercepts actions with meta.requiresNetwork. When offline: queue the action in a persisted array, apply an optimistic update locally. When connectivity returns (NetInfo listener), replay the queue in order, handling conflicts (server state may have changed). Use Redux Persist to ensure the queue survives app kills. For conflict resolution, implement last-write-wins, server-wins, or merge strategies depending on the data type. Track retry counts and implement exponential backoff for failed replays.

Q10: When would you choose Redux over Context in React Native?

A: Choose Redux when: (1) state changes frequently (Context re-renders all consumers on any change); (2) you need middleware (analytics, offline queue, navigation side effects); (3) you need DevTools with time-travel debugging; (4) you need persistence with migrations; (5) multiple developers need predictable patterns. Use Context for: infrequently-changing values (theme, locale), dependency injection, and simple auth state. Context is especially dangerous in RN because a provider wrapping NavigationContainer re-renders all mounted screens on any change.

Zustand & MobX

Q11: How does Zustand achieve fine-grained re-renders without Context?

A: Zustand uses useSyncExternalStore (React 18) with a selector function. Each component subscribes to the store with a selector that extracts only the needed slice. On state change, Zustand runs each subscriber's selector and compares the result with the previous value using Object.is. Only components whose selected value changed re-render. No Context provider means no provider-level re-renders. The store lives outside React's component tree as a vanilla JS object.

Q12: What's the common pitfall with Zustand selectors that causes infinite re-renders?

A: Selecting an object creates a new reference every time: useStore(s => ({ a: s.a, b: s.b })) always returns a new object, failing Object.is comparison. Solutions: (1) use useShallow from zustand/react/shallow for shallow equality; (2) select primitives individually; (3) use a custom equality function. The useShallow approach is preferred for selecting multiple values.

Q13: How does Zustand's persist middleware integrate with MMKV?

A: Use createJSONStorage with an MMKV adapter that implements getItem/setItem/removeItem. The persist middleware serializes the store to JSON and writes to MMKV on every state change. It supports partialize to persist only specific fields, version + migrate for schema evolution, and onRehydrateStorage for error handling. Hydration state is accessible via useStore.persist.hasHydrated().

Q14: How does MobX's reactivity model differ from Redux's?

A: Redux uses a pull model: components explicitly declare what state they need via selectors, and re-render when selected values change. MobX uses a push model: observables are Proxy-wrapped objects that track which properties are read during render (via observer HOC). When an observed property changes, MobX automatically notifies only the components that read it. No selectors needed — the framework handles granularity. Trade-off: MobX is more magical (harder to debug), Redux is more explicit (more boilerplate).

Q15: What advantages does MobX-State-Tree (MST) offer for React Native?

A: MST adds: (1) Runtime type checking — catches invalid state at the boundary; (2) Snapshots — plain JSON serialization of the entire tree, trivial to persist to MMKV; (3) Patches — granular change tracking for sync/undo; (4) Actions — enforced mutation boundaries; (5) References — type-safe relationships between models; (6) Middleware — action logging, undo/redo. For RN, snapshots are the killer feature: onSnapshot(store, snapshot => mmkv.set('store', JSON.stringify(snapshot))) gives you persistence in one line.

Q16: Compare Redux, Zustand, and MobX for a large React Native app.

A: Redux (RTK): Best for large teams needing predictable patterns, rich middleware (offline, analytics), mature DevTools, and Redux Persist. ~13KB. Zustand: Best for most RN apps — minimal API (~1.2KB), built-in persist, no boilerplate, works outside React (getState in native callbacks). MobX/MST: Best for complex domain models with computed values and real-time data — automatic re-render optimization, MST snapshots for persistence. Choose based on team familiarity, complexity profile, and debugging needs.

Server State

Q17: Why should server state be managed separately from client state?

A: Server state has fundamentally different characteristics: it's owned by the backend (not the client), can become stale, needs background refreshing, requires cache invalidation strategies, and benefits from deduplication. Mixing server state into Redux leads to manual loading/error/stale tracking for every API call. Libraries like TanStack Query handle this automatically with stale-while-revalidate, background refetching, cache garbage collection, and optimistic updates — concerns that would require hundreds of lines of custom Redux middleware.

Q18: How do you configure TanStack Query for React Native's app lifecycle?

A: Two integrations: (1) focusManager.setEventListener connected to AppState.addEventListener — triggers refetch of stale queries when app returns to foreground; (2) onlineManager.setEventListener connected to NetInfo — pauses queries when offline and resumes when online. Set networkMode: 'offlineFirst' to serve cached data while offline. For persistence, use PersistQueryClientProvider with an MMKV-backed sync storage persister.

Q19: How does Apollo Client's normalized cache work?

A: Apollo stores GraphQL entities by __typename:id as cache keys. A query returning User:1 with posts creates separate cache entries for the User and each Post. Any subsequent query returning User:1 shares the same cache entry. When a mutation returns an updated User:1, all queries referencing that user automatically reflect the update. This normalization eliminates data duplication and enables automatic cache consistency — but requires that all entities have stable IDs and that mutations return updated entities.

Q20: What makes Relay different from Apollo and TanStack Query?

A: Relay uses a compiler that analyzes GraphQL fragments at build time. Key differences: (1) Fragment colocation — each component declares its data needs as a fragment, the compiler composes them into queries; (2) Data masking — components can only access fields in their own fragment, preventing implicit dependencies; (3) Automatic garbage collection — unreferenced data is collected; (4) Optimized runtime — the compiler generates efficient normalization code. Trade-off: requires a build step and strict GraphQL conventions, but scales to Meta-sized apps.

Q21: How do you implement infinite scroll pagination with TanStack Query?

A: Use useInfiniteQuery with getNextPageParam for cursor-based pagination. Connect fetchNextPage to FlatList's onEndReached with onEndReachedThreshold. Critical for RN: set maxPages to limit memory (infinite queries accumulate all pages). Flatten pages with data.pages.flatMap(p => p.items) in a useMemo. For bidirectional pagination (chat), also implement getPreviousPageParam and fetchPreviousPage.

Persistent Storage

Q22: Why is MMKV preferred over AsyncStorage for React Native?

A: MMKV uses JSI for synchronous, bridge-free access (~0.01ms reads vs ~5-10ms for AsyncStorage). It supports typed values (string, number, boolean, buffer) without JSON serialization, built-in AES encryption, practically unlimited storage (vs AsyncStorage's 6MB Android default), and value change listeners. The synchronous API enables instant state rehydration on cold start — no loading screen needed. MMKV is backed by memory-mapped files (mmap) with protobuf encoding and CRC32 integrity checks.

Q23: When would you choose SQLite over MMKV?

A: Choose SQLite when you need: (1) Relational queries — joins, aggregations, complex WHERE clauses; (2) Indexed search — full-text search, range queries; (3) Large datasets — thousands of records with pagination; (4) Transaction guarantees — atomic multi-row updates; (5) Schema enforcement — typed columns, foreign keys. MMKV is key-value only — you can't query "all messages in thread X sorted by date." Use modern JSI-based SQLite (op-sqlite) for synchronous access without bridge overhead.

Q24: How does Realm differ from SQLite architecturally?

A: Realm is an object database with zero-copy architecture — objects are memory-mapped directly from disk as live proxies. Reads don't copy data into JS objects; they access the underlying storage directly. Realm queries return live result sets that update automatically when data changes. SQLite returns static snapshots that must be re-queried. Realm supports object relationships natively (no joins), automatic schema migration, and built-in encryption. Trade-off: Realm objects can't be used across threads without explicit handling, and the Realm file format is proprietary.

Q25: How should sensitive data be stored in React Native?

A: Use iOS Keychain / Android Keystore via react-native-keychain for auth tokens, encryption keys, and biometric data. These provide hardware-backed security. For bulk sensitive data, store the encryption key in Keychain/Keystore and use it to encrypt MMKV or SQLite (SQLCipher). Never store secrets in AsyncStorage, app bundle, or environment variables. Configure Keychain access control for biometric protection and WHEN_UNLOCKED_THIS_DEVICE_ONLY accessibility.

Q26: How do you migrate from AsyncStorage to MMKV in a production app?

A: One-time migration on app launch: (1) Check a migration flag in MMKV; (2) If not migrated, read all keys from AsyncStorage via getAllKeys + multiGet; (3) Write each key-value pair to MMKV; (4) Set the migration flag; (5) Optionally clear AsyncStorage. This is a forward-only migration — once migrated, the app only uses MMKV. Ship the migration in a version that still includes AsyncStorage as a dependency, then remove it in the next version after confirming migration success via analytics.

State Synchronization

Q27: How do you sync state between JS and native in a brownfield app?

A: Four patterns: (1) Initial props — native passes state to RN via initialProperties on RCTRootView; (2) Event bridge — native emits events via RCTEventEmitter, JS listens and updates store; (3) Shared MMKV — both runtimes read/write the same MMKV instance (same process, JSI); (4) TurboModule — JS calls native synchronously for current state. For bidirectional sync, combine event-driven updates (native → JS) with TurboModule calls (JS → native).

Q28: How do push notifications affect state in React Native?

A: Three paths: (1) Foreground — onMessage handler updates in-memory store directly; (2) Background — setBackgroundMessageHandler persists to MMKV since JS runtime may be suspended (no in-memory store available); (3) Cold start from tap — getInitialNotification sets pending navigation before the app mounts, which is consumed after auth check. Each path requires different handling. Missing the background path means lost notifications; missing the cold-start path means tapping a notification does nothing.

Q29: How do you restore state from a deep link?

A: Parse the deep link URL into a screen name and params. If the user is authenticated, construct a navigation state with the home screen at the bottom and the deep-linked screen on top (so back navigation works). If not authenticated, store the parsed link as pending state, complete the auth flow, then navigate. Use React Navigation's linking config for declarative URL-to-screen mapping, with getStateFromPath for custom state construction. Always validate deep link params before navigation.

Q30: When would you use state machines (XState) in a React Native app?

A: Use state machines for complex flows where impossible states must be unrepresentable: file upload (idle → selected → uploading → success/error), checkout (cart → shipping → payment → confirmation), onboarding wizards, media recording, Bluetooth pairing. State machines eliminate bugs from ad-hoc boolean flags (isLoading && isError — impossible but representable). XState provides visualization tools, guards for conditional transitions, and hierarchical/parallel states for complex flows.

Advanced Patterns

Q31: How do you implement undo/redo in a React Native app?

A: Command pattern: each mutation is wrapped in a Command object with execute() and undo() methods. Maintain an undo stack and redo stack. On execute, push to undo stack and clear redo stack. On undo, pop from undo stack, call undo(), push to redo stack. Limit stack size for memory. For Redux, use redux-undo which wraps reducers with past/present/future state. For Zustand, implement temporal middleware. MST patches provide built-in undo capability.

Q32: How do you handle state rehydration without showing a loading screen?

A: Use MMKV's synchronous API for tiered rehydration: (1) Read critical state (auth token, theme) synchronously during the initial render — no loading screen needed; (2) Render the app immediately with this minimal state; (3) Rehydrate session state asynchronously after first render; (4) Rehydrate screen-specific state on-demand when screens mount. This eliminates the splash-to-loading-to-app transition that AsyncStorage-based rehydration requires.

Q33: How do you version and migrate persisted state schemas?

A: Store a version number alongside persisted state. On rehydration, compare persisted version with current version. If different, run sequential migration functions (v1→v2, v2→v3, etc.). Redux Persist provides createMigrate for this. Zustand's persist middleware supports version + migrate. MST validates snapshots against the current model schema. Always test migrations with production data snapshots. Never delete migration functions — a user updating from v1 to v5 needs all intermediate migrations.

Q34: How do you handle state in a React Native app with multiple navigation stacks?

A: Use a global store (Zustand/Redux) for state shared across stacks. Each stack can have its own screen-local state via useState. For tab navigators, remember that all tabs remain mounted — their state persists when switching tabs. For stack navigators with detachInactiveScreens, unmounted screens lose their state — persist important screen state to MMKV and rehydrate on remount. Navigation params should be minimal (IDs, not full objects) to keep the navigation state serializable.

Q35: How do you handle optimistic updates that fail?

A: Pattern: (1) Save the previous state before the optimistic update; (2) Apply the optimistic update immediately (user sees instant feedback); (3) Send the mutation to the server; (4) On success, optionally refetch to get the server's canonical response; (5) On failure, roll back to the saved previous state and show an error. TanStack Query's useMutation provides onMutate (save + optimistic), onError (rollback), and onSettled (refetch). Apollo Client automatically rolls back optimistic responses on error.

Q36: How do you prevent state bloat in long-running React Native apps?

A: (1) Set gcTime on TanStack Query to garbage-collect stale cache entries; (2) Use maxPages on infinite queries; (3) Implement LRU eviction for custom caches; (4) Clear navigation-related state when screens are unmounted; (5) Respond to memory pressure events by evicting non-essential state; (6) Use detachInactiveScreens to unmount off-screen components; (7) Profile memory with Flipper/Xcode Instruments to identify leaks.

Q37: How do you test state management in React Native?

A: (1) Unit test reducers/stores — pure functions, easy to test with Jest; (2) Test selectors — verify memoization and derived data; (3) Test middleware — mock store.dispatch and verify side effects; (4) Integration test hooks — use renderHook from @testing-library/react-native with a provider wrapper; (5) Test persistence — mock MMKV, verify serialization/deserialization; (6) Test migrations — apply migration functions to old state snapshots and verify output.

Q38: How do you handle concurrent state updates from multiple sources?

A: Define clear sources of truth: server-authoritative data uses server state (TanStack Query) with refetch-on-focus; client-authoritative data uses local stores with conflict resolution on sync. For real-time data (WebSocket + REST), use a single store entry point that merges updates. For offline mutations, implement conflict resolution: last-write-wins (simple), server-wins (safe), or three-way merge (complex). Use timestamps or vector clocks for ordering.

Q39: What's the performance impact of different state management solutions on React Native frame rates?

A: Re-renders in RN trigger shadow tree diffs and native view updates — more expensive than web DOM updates. Redux with poor selectors can cause O(n) re-renders in FlatList. MobX's automatic tracking typically produces fewer re-renders but each observer() wrapper adds overhead. Zustand's selector model is similar to Redux but with less infrastructure overhead. Key metrics: measure re-render count with React DevTools Profiler, measure JS frame time with Flipper, and measure native frame time with Xcode Instruments or Android Profiler.

Q40: How do you architect state for a feature that works both online and offline?

A: Layer the architecture: (1) Persistence layer (MMKV/SQLite) as the source of truth; (2) In-memory cache (Zustand/Redux) derived from persistence; (3) Server sync layer (TanStack Query) that updates persistence on fetch; (4) Mutation queue that persists pending mutations and replays on connectivity. Reads always go through the in-memory cache (fast). Writes update local state immediately (optimistic) and queue a server sync. The offline queue is persisted so it survives app kills.

Architecture & Design

Q41: How would you design the state architecture for a messaging app in React Native?

A: (1) Messages: SQLite for storage (queryable, indexed by thread), TanStack Query for server sync, cursor-based pagination for FlatList; (2) Threads: Zustand store with MMKV persistence for thread list and metadata; (3) Presence/typing: Ephemeral state via WebSocket, no persistence; (4) Auth: Keychain for tokens, Zustand for session; (5) Media: File system for attachments, SQLite for metadata; (6) Offline: Mutation queue in MMKV for unsent messages, optimistic UI. Use SQLite's FTS5 for message search.

Q42: How do you handle feature flags in React Native state management?

A: Fetch flags on app launch and store in Zustand with MMKV persistence (so flags are available on cold start before network). Refresh flags on foreground. Use a typed interface for flags to catch usage of removed flags at compile time. For A/B tests, include the variant assignment in the flag value. Never use flags in selectors that run frequently — cache the flag value at the component level. Provide a fallback default for every flag in case persistence is empty on first launch.

Q43: How do you handle state in React Native when the JS bundle updates (CodePush)?

A: CodePush updates the JS bundle but not native code. Persisted state schemas may change between bundle versions. Always version your persisted state and include migration logic. On CodePush update: (1) the app restarts with the new bundle; (2) state rehydration runs migrations from old version to new; (3) if migration fails, fall back to defaults and log the error. Never assume persisted state matches the current schema — CodePush can update the bundle without going through the app store review process.

Q44: How do you handle global error state in React Native?

A: Implement a global error boundary at the navigation root that catches render errors. For async errors: use TanStack Query's onError callbacks and Redux middleware to funnel errors to a centralized error store. Display errors via a toast/snackbar system that reads from the error store. For fatal errors, persist the error details to MMKV and show a recovery screen on next launch. Integrate with Crashlytics/Sentry for remote error reporting. Never swallow errors silently — always log and optionally display.

Q45: How do you share state between a React Native app and its widget/extension?

A: iOS: Use App Groups to share a UserDefaults suite or MMKV instance between the main app and extensions (Today Widget, Share Extension, Watch app). Android: Use SharedPreferences with a content provider, or share an MMKV instance via the same process. The extension runs in a separate process — in-memory state is not shared. Persist shared state to the shared storage container, and have the extension read from it. Keep shared data minimal and serializable.

Q46: What's the role of Flipper in debugging state management?

A: Flipper provides: (1) Redux Debugger plugin — action log, state tree, time-travel; (2) Network inspector — correlate API responses with state changes; (3) React DevTools — component re-render profiling; (4) Database inspector — view SQLite/Realm contents; (5) Shared Preferences viewer — inspect AsyncStorage/MMKV. For production debugging, use Flipper's custom plugins to send sanitized state snapshots to a remote dashboard.

Q47: How do you handle authentication state expiry in React Native?

A: Store access token and refresh token in Keychain. Implement an API interceptor that checks token expiry before each request. On 401: attempt token refresh using the refresh token. If refresh succeeds, retry the original request. If refresh fails, clear auth state, cancel all pending queries, and navigate to login. During refresh, queue concurrent requests and replay them after refresh completes (avoid multiple simultaneous refresh calls). Persist the pending deep link so the user returns to their previous screen after re-authentication.

Q48: How do you handle state for forms with complex validation in React Native?

A: Use React Hook Form or Formik for form state — keep it local to the form component, not in global state. For multi-step forms that span screens, persist form state to MMKV between steps so it survives navigation and app kills. Validate with Zod or Yup schemas. For real-time server validation (username availability), use TanStack Query with debounced queries. Never store form state in Redux — it changes too frequently and doesn't need global visibility.

Q49: How do you implement state-based feature gating for gradual rollouts?

A: Fetch rollout configuration from the server (percentage, user segments, device criteria). Store in Zustand with MMKV persistence. Evaluate gate conditions locally: isFeatureEnabled('new_checkout', { userId, deviceType, appVersion }). Cache the evaluation result for consistency within a session. Use a deterministic hash of userId + featureKey for percentage-based rollouts (ensures the same user always gets the same variant). Log exposure events for analytics.

Q50: How would you design state management for a React Native super-app with multiple mini-apps?

A: Each mini-app gets its own isolated store (Zustand instance) with its own MMKV namespace. A shared core provides: (1) Auth state (read-only for mini-apps); (2) User preferences; (3) Navigation coordination; (4) Analytics context. Mini-apps communicate through a message bus (event emitter) rather than shared mutable state. The host app manages lifecycle — mounting/unmounting mini-app stores, clearing their caches on memory pressure. This isolation prevents one mini-app's state bugs from affecting others.

Q51: How do you handle state consistency when using both REST and WebSocket in React Native?

A: Designate the server as the source of truth. REST responses update TanStack Query cache. WebSocket messages update the same cache entries via queryClient.setQueryData. Use sequence numbers or timestamps to resolve ordering conflicts (a WebSocket update may arrive before the REST response for the same mutation). Implement a reconciliation layer that merges updates from both sources, preferring the most recent version. On WebSocket reconnect, fetch the latest state via REST to catch any missed messages.
