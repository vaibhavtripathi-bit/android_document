Zustand & MobX in React Native

Zustand offers minimal-API stores with built-in persistence middleware and MMKV integration. MobX provides observable-based reactivity with MobX-State-Tree (MST) for structured, snapshot-capable state. Understanding when each excels over Redux — and their internal mechanisms — is critical Staff-level knowledge.

1. Zustand Core Concepts

Zustand ("state" in German) is a ~1KB state management library that uses a publish-subscribe model with React's useSyncExternalStore under the hood. It requires no providers, no boilerplate, and supports middleware composition.

Zustand Architecture ════════════════════════════════════════════════════ create(stateCreator) │ ▼ ┌─────────────────────────────────────────┐ │ Zustand Store │ │ ┌─────────────────────────────────┐ │ │ │ State Object (plain JS) │ │ │ │ { count: 0, user: null, ... } │ │ │ └─────────────────────────────────┘ │ │ │ │ ┌─────────────────────────────────┐ │ │ │ Subscribers Set │ │ │ │ Set<(state, prevState) => void>│ │ │ └─────────────────────────────────┘ │ │ │ │ API: getState, setState, subscribe, │ │ destroy │ └─────────────────────────────────────────┘ │ ▼ useStore(selector) → useSyncExternalStore │ ▼ Component re-renders only when selected slice changes (Object.is comparison)

1.1 Basic Store

import { create } from 'zustand';

interface AuthStore {
  token: string | null;
  user: User | null;
  isLoading: boolean;
  login: (credentials: Credentials) => Promise<void>;
  logout: () => void;
}

const useAuthStore = create<AuthStore>((set, get) => ({
  token: null,
  user: null,
  isLoading: false,

  login: async (credentials) => {
    set({ isLoading: true });
    try {
      const { token, user } = await api.login(credentials);
      set({ token, user, isLoading: false });
    } catch (error) {
      set({ isLoading: false });
      throw error;
    }
  },

  logout: () => {
    set({ token: null, user: null });
    navigationRef.resetRoot({ index: 0, routes: [{ name: 'Login' }] });
  },
}));

// Usage — selector extracts only what this component needs
function ProfileHeader() {
  const user = useAuthStore((s) => s.user);
  if (!user) return null;
  return <Text>{user.displayName}</Text>;
}

// Access outside React (middleware, utils, native modules)
const token = useAuthStore.getState().token;
Key Insight

Zustand's getState() works outside React components — in middleware, native module callbacks, background tasks, and push notification handlers. This is a significant advantage over Context-based solutions in React Native where state access requires being inside the component tree.

1.2 Selector Optimization

// Zustand uses Object.is for equality by default
// Selecting an object creates a new reference each time → infinite re-renders

// BAD: New object reference every render
const { user, token } = useAuthStore((s) => ({
  user: s.user,
  token: s.token,
}));

// GOOD: Use shallow equality for object selectors
import { useShallow } from 'zustand/react/shallow';

const { user, token } = useAuthStore(
  useShallow((s) => ({ user: s.user, token: s.token }))
);

// BEST: Select primitives individually (no equality issues)
const token = useAuthStore((s) => s.token);
const userName = useAuthStore((s) => s.user?.displayName);

2. Zustand Persistence & MMKV

2.1 Persist Middleware with MMKV

import { create } from 'zustand';
import { persist, createJSONStorage } from 'zustand/middleware';
import { MMKV } from 'react-native-mmkv';

const mmkv = new MMKV({ id: 'zustand-storage' });

const zustandMMKVStorage = createJSONStorage(() => ({
  getItem: (name: string) => mmkv.getString(name) ?? null,
  setItem: (name: string, value: string) => mmkv.set(name, value),
  removeItem: (name: string) => mmkv.delete(name),
}));

interface SettingsStore {
  theme: 'light' | 'dark' | 'system';
  language: string;
  notificationsEnabled: boolean;
  setTheme: (theme: SettingsStore['theme']) => void;
  setLanguage: (lang: string) => void;
}

const useSettingsStore = create<SettingsStore>()(
  persist(
    (set) => ({
      theme: 'system',
      language: 'en',
      notificationsEnabled: true,
      setTheme: (theme) => set({ theme }),
      setLanguage: (language) => set({ language }),
    }),
    {
      name: 'settings',
      storage: zustandMMKVStorage,
      partialize: (state) => ({
        theme: state.theme,
        language: state.language,
        notificationsEnabled: state.notificationsEnabled,
      }),
      version: 1,
      migrate: (persisted, version) => {
        if (version === 0) {
          return { ...persisted, notificationsEnabled: true };
        }
        return persisted;
      },
    }
  )
);

2.2 Hydration Awareness

// Zustand persist exposes hydration state
function App() {
  const hasHydrated = useSettingsStore.persist.hasHydrated;

  if (!hasHydrated()) {
    return <SplashScreen />;
  }

  return <MainApp />;
}

// Or use the onRehydrateStorage callback
const useStore = create()(
  persist(
    (set) => ({ /* ... */ }),
    {
      name: 'store',
      storage: zustandMMKVStorage,
      onRehydrateStorage: () => (state, error) => {
        if (error) {
          console.error('Rehydration failed:', error);
          crashlytics().recordError(error);
        }
      },
    }
  )
);

3. Zustand Advanced Patterns

3.1 Middleware Composition

import { devtools, subscribeWithSelector } from 'zustand/middleware';
import { immer } from 'zustand/middleware/immer';

const useStore = create<AppStore>()(
  devtools(
    subscribeWithSelector(
      persist(
        immer((set, get) => ({
          todos: [] as Todo[],
          addTodo: (text: string) =>
            set((state) => {
              state.todos.push({ id: uuid(), text, done: false });
            }),
          toggleTodo: (id: string) =>
            set((state) => {
              const todo = state.todos.find(t => t.id === id);
              if (todo) todo.done = !todo.done;
            }),
        })),
        { name: 'app-store', storage: zustandMMKVStorage }
      )
    ),
    { name: 'AppStore' }
  )
);

// subscribeWithSelector enables fine-grained subscriptions outside React
useStore.subscribe(
  (state) => state.todos.filter(t => !t.done).length,
  (unfinished) => {
    updateBadgeCount(unfinished);
  }
);

3.2 Store Slicing Pattern

// Split large stores into slices (similar to Redux slices)
interface AuthSlice {
  token: string | null;
  login: (creds: Credentials) => Promise<void>;
}

interface UISlice {
  theme: 'light' | 'dark';
  sidebarOpen: boolean;
}

const createAuthSlice: StateCreator<AuthSlice & UISlice, [], [], AuthSlice> =
  (set) => ({
    token: null,
    login: async (creds) => {
      const { token } = await api.login(creds);
      set({ token });
    },
  });

const createUISlice: StateCreator<AuthSlice & UISlice, [], [], UISlice> =
  (set) => ({
    theme: 'light',
    sidebarOpen: false,
  });

const useStore = create<AuthSlice & UISlice>()((...args) => ({
  ...createAuthSlice(...args),
  ...createUISlice(...args),
}));

4. MobX Core Concepts

MobX uses transparent reactive programming — it automatically tracks which observables a component reads and re-renders only when those specific observables change. No selectors needed.

MobX Reactivity Model ════════════════════════════════════════════════════ @observable state @computed derived @action mutations ───────────────── ────────────────── ───────────────── Proxy-wrapped objects Cached derived values Batched state changes Track read access Auto-invalidate on Transaction boundaries Notify on write dependency change ┌──────────────┐ reads ┌──────────────┐ │ Observable │ ◄───────── │ Computed │ │ user.name │ │ fullName │ └──────┬───────┘ └──────┬───────┘ │ notifies │ notifies ▼ ▼ ┌──────────────┐ ┌──────────────┐ │ Reaction │ │ observer() │ │ autorun() │ │ Component │ └──────────────┘ └──────────────┘ Key: MobX tracks dependencies at RUNTIME via Proxy getters, not at compile time. This is fundamentally different from Redux selectors which are defined statically.

4.1 MobX Store Class

import { makeAutoObservable, runInAction, reaction } from 'mobx';
import { observer } from 'mobx-react-lite';

class TodoStore {
  todos: Todo[] = [];
  filter: 'all' | 'active' | 'done' = 'all';

  constructor() {
    makeAutoObservable(this, {}, { autoBind: true });
  }

  // Computed: automatically cached, recomputes only when todos or filter change
  get filteredTodos(): Todo[] {
    switch (this.filter) {
      case 'active': return this.todos.filter(t => !t.done);
      case 'done': return this.todos.filter(t => t.done);
      default: return this.todos;
    }
  }

  get activeCount(): number {
    return this.todos.filter(t => !t.done).length;
  }

  addTodo(text: string) {
    this.todos.push({ id: uuid(), text, done: false });
  }

  async fetchTodos() {
    const data = await api.getTodos();
    runInAction(() => {
      this.todos = data;
    });
  }
}

// observer() wraps component to track observable access
const TodoList = observer(({ store }: { store: TodoStore }) => (
  <FlatList
    data={store.filteredTodos}
    keyExtractor={(item) => item.id}
    renderItem={({ item }) => <TodoItem todo={item} />}
    ListHeaderComponent={<Text>{store.activeCount} remaining</Text>}
  />
));
Warning

MobX's observer HOC uses a custom forceUpdate mechanism. In React Native, every observer component creates a reaction that tracks observable access during render. If you forget observer, the component won't react to state changes. If you destructure observables before passing to a non-observer child, tracking is lost. Always wrap components that read observables with observer.

5. MobX-State-Tree (MST)

MST adds structure to MobX: typed models, runtime type checking, snapshots, patches, and middleware. It's particularly powerful for React Native because snapshots enable trivial persistence and time-travel debugging.

import { types, flow, getSnapshot, applySnapshot } from 'mobx-state-tree';

const Todo = types
  .model('Todo', {
    id: types.identifier,
    text: types.string,
    done: false,
    createdAt: types.optional(types.Date, () => new Date()),
  })
  .actions(self => ({
    toggle() { self.done = !self.done; },
    setText(text: string) { self.text = text; },
  }));

const RootStore = types
  .model('RootStore', {
    todos: types.array(Todo),
    filter: types.optional(
      types.enumeration(['all', 'active', 'done']),
      'all'
    ),
  })
  .views(self => ({
    get filteredTodos() {
      switch (self.filter) {
        case 'active': return self.todos.filter(t => !t.done);
        case 'done': return self.todos.filter(t => t.done);
        default: return self.todos;
      }
    },
  }))
  .actions(self => ({
    addTodo(text: string) {
      self.todos.push({ id: uuid(), text, done: false });
    },
    fetchTodos: flow(function* () {
      const data = yield api.getTodos();
      self.todos.replace(data);
    }),
  }));

6. MST Snapshots & Persistence

import { onSnapshot, getSnapshot, applySnapshot } from 'mobx-state-tree';
import { MMKV } from 'react-native-mmkv';

const storage = new MMKV({ id: 'mst-snapshots' });

function createPersistedStore() {
  const savedSnapshot = storage.getString('root-store');
  let initialState = {};

  if (savedSnapshot) {
    try {
      initialState = JSON.parse(savedSnapshot);
    } catch (e) {
      console.warn('Failed to parse MST snapshot, using defaults');
    }
  }

  const store = RootStore.create(initialState);

  // Auto-persist on every state change (debounced)
  let timer: NodeJS.Timeout;
  onSnapshot(store, (snapshot) => {
    clearTimeout(timer);
    timer = setTimeout(() => {
      storage.set('root-store', JSON.stringify(snapshot));
    }, 500);
  });

  return store;
}
Architect Insight

MST snapshots are plain JSON objects — no special serialization needed. This makes persistence trivial compared to Redux Persist's transform pipeline. Combined with MST's runtime type checking, you get automatic validation on rehydration: if the persisted snapshot doesn't match the current model schema, MST throws a clear error rather than silently corrupting state.

7. Redux vs Zustand vs MobX

Aspect Redux (RTK) Zustand MobX
Paradigm Flux: actions → reducers → state Flux-lite: setState with selectors Reactive: observables + reactions
Bundle size ~13KB (RTK + react-redux) ~1.2KB ~16KB (mobx + mobx-react-lite)
Boilerplate Moderate (slices, thunks) Minimal (single create call) Low (classes + makeAutoObservable)
Re-render control Manual selectors + memoization Manual selectors (simpler API) Automatic (observer tracks reads)
DevTools Excellent (Redux DevTools, Flipper) Good (Redux DevTools via middleware) Good (mobx-devtools, MST middleware)
Persistence Redux Persist (mature ecosystem) Built-in persist middleware MST snapshots (trivial)
TypeScript Good (RTK improves inference) Excellent (minimal generics) Good (MST types are runtime)
Learning curve Steep (concepts, middleware, patterns) Gentle (familiar API) Moderate (reactive paradigm shift)
Best for RN Large teams, complex offline, enterprise Most RN apps, rapid development Complex domains, real-time data

8. Architect Considerations

Staff-Level Mental Model

The choice between Redux, Zustand, and MobX is not about which is "better" — it's about matching the reactivity model to your team's mental model and your app's complexity profile. Redux excels when you need deterministic state transitions with rich middleware (offline-first, analytics, undo/redo). Zustand excels when you want Redux-like patterns without the ceremony. MobX/MST excels when your domain model is complex and you want the framework to handle re-render optimization automatically. A Staff engineer chooses based on the team's strengths, the app's persistence requirements, and the debugging needs — not personal preference.
