Server State in React Native

Server state — data owned by the backend and cached on the client — requires fundamentally different management than client state. TanStack Query, Apollo Client, and Relay each solve this problem with different trade-offs around caching, offline support, and data fetching patterns critical for mobile.

1. Server State vs Client State

The fundamental insight behind server state libraries is that data from APIs is not owned by the client. It's a cache of remote data that can become stale, needs background refreshing, and requires different lifecycle management than client-generated state.

Client State vs Server State ════════════════════════════════════════════════════════ Client State Server State ──────────── ───────────── Source: User actions Source: Remote API Owner: This app instance Owner: Backend/database Staleness: N/A Staleness: Time-based Sync: Not needed Sync: Background refetch Examples: Examples: • Theme preference • User profile • Form input • Feed items • Modal open/closed • Notifications • Navigation state • Search results Tool: useState/Zustand/Redux Tool: TanStack Query/Apollo/Relay ───────────────────────────────────────────────────── Mobile-specific concerns for server state: • Refetch on app foreground (stale after background) • Offline cache persistence (MMKV/SQLite) • Network-aware fetching (wifi vs cellular) • Memory pressure → cache eviction • Pagination for infinite scroll lists

2. TanStack Query in React Native

TanStack Query (formerly React Query) manages server state with automatic caching, background refetching, stale-while-revalidate, and built-in support for pagination and infinite queries.

2.1 Setup with Mobile Optimizations

import { QueryClient, QueryClientProvider, focusManager, onlineManager } from '@tanstack/react-query';
import { AppState, Platform } from 'react-native';
import NetInfo from '@react-native-community/netinfo';

// Connect TanStack Query to RN app lifecycle
focusManager.setEventListener((handleFocus) => {
  const sub = AppState.addEventListener('change', (state) => {
    handleFocus(state === 'active');
  });
  return () => sub.remove();
});

// Connect to network state
onlineManager.setEventListener((setOnline) => {
  return NetInfo.addEventListener((state) => {
    setOnline(!!state.isConnected);
  });
});

const queryClient = new QueryClient({
  defaultOptions: {
    queries: {
      staleTime: 5 * 60 * 1000,       // 5 min before considered stale
      gcTime: 30 * 60 * 1000,          // 30 min before garbage collected
      retry: 3,
      retryDelay: (attempt) => Math.min(1000 * 2 ** attempt, 30000),
      refetchOnWindowFocus: true,    // Refetch on app foreground
      networkMode: 'offlineFirst',    // Serve cache while offline
    },
    mutations: {
      networkMode: 'offlineFirst',
    },
  },
});

2.2 Query Patterns

// Basic query with mobile-specific options
function useUserProfile(userId: string) {
  return useQuery({
    queryKey: ['user', userId],
    queryFn: () => api.getUser(userId),
    staleTime: 10 * 60 * 1000,
    placeholderData: (previousData) => previousData,
  });
}

// Optimistic mutation
function useLikePost() {
  const queryClient = useQueryClient();

  return useMutation({
    mutationFn: (postId: string) => api.likePost(postId),
    onMutate: async (postId) => {
      await queryClient.cancelQueries({ queryKey: ['post', postId] });
      const previous = queryClient.getQueryData(['post', postId]);

      queryClient.setQueryData(['post', postId], (old: Post) => ({
        ...old,
        likes: old.likes + 1,
        isLiked: true,
      }));

      return { previous };
    },
    onError: (err, postId, context) => {
      queryClient.setQueryData(['post', postId], context?.previous);
    },
    onSettled: (data, err, postId) => {
      queryClient.invalidateQueries({ queryKey: ['post', postId] });
    },
  });
}

2.3 Offline Persistence

import { createSyncStoragePersister } from '@tanstack/query-sync-storage-persister';
import { PersistQueryClientProvider } from '@tanstack/react-query-persist-client';
import { MMKV } from 'react-native-mmkv';

const mmkv = new MMKV({ id: 'query-cache' });

const persister = createSyncStoragePersister({
  storage: {
    getItem: (key) => mmkv.getString(key) ?? null,
    setItem: (key, value) => mmkv.set(key, value),
    removeItem: (key) => mmkv.delete(key),
  },
  throttleTime: 1000,
});

function App() {
  return (
    <PersistQueryClientProvider
      client={queryClient}
      persistOptions={{
        persister,
        maxAge: 24 * 60 * 60 * 1000,  // 24 hours
        dehydrateOptions: {
          shouldDehydrateQuery: (query) =>
            query.state.status === 'success' &&
            !query.queryKey.includes('ephemeral'),
        },
      }}
    >
      <MainApp />
    </PersistQueryClientProvider>
  );
}
Architect Insight

TanStack Query's networkMode: 'offlineFirst' combined with MMKV persistence creates a powerful offline experience: the app renders instantly from cache, then background-refetches when online. This is the stale-while-revalidate pattern adapted for mobile — users see content immediately on cold start, even without network.

3. Apollo Client for GraphQL

Apollo Client provides a normalized cache that stores GraphQL entities by their __typename and id, enabling automatic cache updates when mutations return updated entities.

3.1 Cache Normalization

Apollo Cache Normalization ════════════════════════════════════════════════════ GraphQL Response: Normalized Cache: { ROOT_QUERY user(id: "1") { user({"id":"1"}) → User:1 id: "1" name: "Alice" User:1 posts { __typename: "User" id: "10" id: "1" title: "Hello" name: "Alice" } posts → [Post:10] } } Post:10 __typename: "Post" id: "10" title: "Hello" Key: Entities are stored ONCE by __typename:id. Any query returning User:1 shares the same cache entry. Updating User:1 in one query updates it everywhere.

3.2 Optimistic Updates

const [likePost] = useMutation(LIKE_POST, {
  optimisticResponse: {
    likePost: {
      __typename: 'Post',
      id: postId,
      likes: currentLikes + 1,
      isLiked: true,
    },
  },
  update: (cache, { data }) => {
    cache.modify({
      id: cache.identify(data.likePost),
      fields: {
        likes: () => data.likePost.likes,
        isLiked: () => data.likePost.isLiked,
      },
    });
  },
  onError: (error) => {
    // Apollo automatically rolls back optimistic response
    showToast('Failed to like post');
  },
});

3.3 Offline Mutations Queue

import { ApolloClient, InMemoryCache } from '@apollo/client';
import { persistCache } from 'apollo3-cache-persist';
import { MMKV } from 'react-native-mmkv';

const mmkv = new MMKV({ id: 'apollo-cache' });

const cache = new InMemoryCache({
  typePolicies: {
    Query: {
      fields: {
        feed: {
          keyArgs: ['type'],
          merge(existing = [], incoming, { args }) {
            if (!args?.after) return incoming;
            return [...existing, ...incoming];
          },
        },
      },
    },
  },
});

await persistCache({
  cache,
  storage: {
    getItem: (key) => mmkv.getString(key) ?? null,
    setItem: (key, value) => mmkv.set(key, value),
    removeItem: (key) => mmkv.delete(key),
  },
});

4. Relay: Compiler-Driven Fetching

Relay takes a fundamentally different approach: a compiler analyzes your GraphQL fragments at build time, generating optimized runtime artifacts. This enables automatic query composition, data masking, and garbage collection.

4.1 Fragment Colocation

// Each component declares exactly what data it needs
function PostCard({ postRef }: { postRef: PostCard_post$key }) {
  const post = useFragment(
    graphql`
      fragment PostCard_post on Post {
        id
        title
        author {
          name
          avatarUrl
        }
        likeCount
        createdAt
      }
    `,
    postRef
  );

  return (
    <View>
      <Text>{post.title}</Text>
      <Text>by {post.author.name}</Text>
    </View>
  );
}

// Parent query composes child fragments automatically
function FeedScreen() {
  const data = useLazyLoadQuery(
    graphql`
      query FeedScreenQuery {
        feed {
          edges {
            node {
              ...PostCard_post
            }
          }
        }
      }
    `,
    {}
  );

  return (
    <FlatList
      data={data.feed.edges}
      renderItem={({ item }) => <PostCard postRef={item.node} />}
    />
  );
}
Key Insight

Relay's data masking means PostCard can only access fields declared in its own fragment — not fields from parent or sibling fragments. This prevents implicit data dependencies and makes components truly self-contained. When you remove a field from a fragment, the compiler guarantees no other component was depending on it.

5. Network State Integration

// Adaptive fetching based on network quality
function useAdaptiveQuery<T>(
  queryKey: string[],
  queryFn: () => Promise<T>,
  options?: UseQueryOptions<T>
) {
  const netInfo = useNetInfo();

  const adaptedOptions = useMemo(() => {
    const isSlow = netInfo.type === 'cellular' &&
      netInfo.details?.cellularGeneration === '2g';

    return {
      ...options,
      staleTime: isSlow ? 30 * 60 * 1000 : 5 * 60 * 1000,
      refetchOnWindowFocus: !isSlow,
      refetchInterval: isSlow ? false : options?.refetchInterval,
    };
  }, [netInfo.type, netInfo.details?.cellularGeneration]);

  return useQuery({ queryKey, queryFn, ...adaptedOptions });
}

6. Pagination Patterns in Mobile

6.1 Infinite Scroll with TanStack Query

function useFeed() {
  return useInfiniteQuery({
    queryKey: ['feed'],
    queryFn: ({ pageParam }) => api.getFeed({ cursor: pageParam, limit: 20 }),
    initialPageParam: undefined as string | undefined,
    getNextPageParam: (lastPage) => lastPage.nextCursor ?? undefined,
    maxPages: 10,  // Limit memory: keep only 10 pages in cache
  });
}

function FeedScreen() {
  const { data, fetchNextPage, hasNextPage, isFetchingNextPage } = useFeed();

  const items = useMemo(
    () => data?.pages.flatMap(p => p.items) ?? [],
    [data?.pages]
  );

  return (
    <FlatList
      data={items}
      keyExtractor={(item) => item.id}
      renderItem={({ item }) => <FeedItem item={item} />}
      onEndReached={() => {
        if (hasNextPage && !isFetchingNextPage) fetchNextPage();
      }}
      onEndReachedThreshold={0.5}
      ListFooterComponent={isFetchingNextPage ? <Spinner /> : null}
    />
  );
}
Warning

Infinite queries without maxPages will accumulate all fetched pages in memory. On a feed with hundreds of pages, this can consume significant RAM. Always set maxPages for infinite scroll in React Native and implement bidirectional pagination if users need to scroll back to earlier content.

7. TanStack Query vs Apollo vs Relay

Aspect TanStack Query Apollo Client Relay
API Type REST / any async GraphQL GraphQL (strict)
Cache Query-key based Normalized (entity) Normalized + GC
Build step None Optional codegen Required compiler
Bundle size ~13KB ~33KB ~25KB
Offline Persist + offlineFirst apollo3-cache-persist Custom (relay-offline)
Data masking No No Yes (enforced)
Best for RN REST APIs, flexibility GraphQL with offline Large GraphQL apps (Meta-scale)

8. Architect Considerations

Staff-Level Mental Model

Server state management in React Native is fundamentally about cache coherence in a hostile environment. The network is unreliable, the app can be killed at any time, and users expect instant content on launch. A Staff engineer designs the server state layer with three guarantees: (1) instant render from persisted cache on cold start, (2) background refresh when online to keep data fresh, and (3) graceful degradation when offline with optimistic mutations queued for replay. The choice between TanStack Query, Apollo, and Relay depends on your API layer (REST vs GraphQL), your cache coherence requirements (query-based vs normalized), and your team's willingness to adopt build-time tooling (Relay compiler).
