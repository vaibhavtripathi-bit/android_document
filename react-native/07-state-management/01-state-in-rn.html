State in React Native

React Native state management inherits React's paradigm but introduces mobile-specific concerns: navigation state, app lifecycle transitions, memory pressure, platform permissions, and persistent storage across restarts. Understanding these dimensions separates Staff-level engineers from those who treat RN as "React for mobile."

1. State Categories in Mobile Apps

Unlike web apps where a page refresh resets everything, mobile apps maintain state across complex lifecycle events. A Staff-level engineer must categorize state by its lifetime, scope, and persistence requirements.

State Taxonomy in React Native ═══════════════════════════════════════════════════════════════ Ephemeral State Session State Persistent State ───────────── ────────────── ───────────────── • UI animations • Auth tokens • User preferences • Form input • Navigation stack • Cached API data • Modal visibility • In-memory cache • Offline queue • Scroll position • WebSocket conn • Draft content │ │ │ Lifetime: Component Lifetime: App session Lifetime: Across restarts Storage: useState Storage: Context/Store Storage: MMKV/SQLite/AS Loss: Re-render OK Loss: Re-auth needed Loss: Data loss ───────────────────────────────────────────────────────────── Platform State (cross-cutting) • Permissions (camera, location, notifications) • Device info (orientation, network, battery) • App state (active, background, inactive) • Deep link / push notification payloads
Architect Insight

The critical difference from web: mobile apps can be killed by the OS at any time while backgrounded. There is no "beforeunload" event. State that hasn't been persisted is gone. This means your state architecture must define clear persistence boundaries — what survives a kill, what doesn't, and how fast rehydration happens on cold start.

1.1 State Ownership Matrix

// Defining clear state ownership prevents duplication and sync bugs
interface StateArchitecture {
  // Component-local: useState/useReducer
  ephemeral: {
    formInputs: string;
    modalOpen: boolean;
    animationProgress: Animated.Value;
  };

  // Global store: Redux/Zustand/MobX
  session: {
    auth: { token: string; user: User };
    navigation: NavigationState;
    featureFlags: Record<string, boolean>;
  };

  // Server cache: TanStack Query / Apollo
  server: {
    userProfile: QueryResult<User>;
    feedItems: InfiniteQueryResult<Post[]>;
    notifications: QueryResult<Notification[]>;
  };

  // Persistent: MMKV / SQLite / Realm
  persistent: {
    preferences: UserPreferences;
    offlineQueue: PendingMutation[];
    cachedMedia: MediaCache;
  };
}

2. Navigation State

Navigation state in React Native is fundamentally different from web routing. React Navigation maintains a stack-based state tree that represents the entire navigation hierarchy — nested stacks, tabs, drawers — as a serializable object.

2.1 Navigation State Structure

// React Navigation state is a recursive tree
const navigationState = {
  type: 'stack',
  key: 'stack-1',
  index: 1,
  routes: [
    {
      key: 'home-1',
      name: 'HomeTabs',
      state: {
        type: 'tab',
        index: 0,
        routes: [
          { key: 'feed-1', name: 'Feed', params: {} },
          { key: 'search-1', name: 'Search', params: {} },
          { key: 'profile-1', name: 'Profile', params: {} },
        ],
      },
    },
    {
      key: 'detail-1',
      name: 'PostDetail',
      params: { postId: 'abc123' },
    },
  ],
};

// Persisting navigation state for app restart restoration
function App() {
  const [isReady, setIsReady] = useState(false);
  const [initialState, setInitialState] = useState();

  useEffect(() => {
    const restoreState = async () => {
      const savedState = await storage.getString('NAV_STATE');
      if (savedState) {
        const state = JSON.parse(savedState);
        // Validate state matches current navigator structure
        if (isValidNavigationState(state)) {
          setInitialState(state);
        }
      }
      setIsReady(true);
    };
    restoreState();
  }, []);

  if (!isReady) return null;

  return (
    <NavigationContainer
      initialState={initialState}
      onStateChange={(state) =>
        storage.set('NAV_STATE', JSON.stringify(state))
      }
    >
      {/* navigators */}
    </NavigationContainer>
  );
}
Warning

Navigation state persistence must handle schema migrations. If you add/remove/rename screens between app versions, the persisted state may reference screens that no longer exist. Always validate persisted navigation state against the current navigator structure before restoring, and fall back to the initial route on mismatch.

2.2 Deep Link State Injection

// Deep links must construct valid navigation state
const linking = {
  prefixes: ['myapp://', 'https://myapp.com'],
  config: {
    screens: {
      HomeTabs: {
        screens: {
          Feed: 'feed',
          Profile: 'profile/:userId',
        },
      },
      PostDetail: 'post/:postId',
      Settings: 'settings',
    },
  },
  // Custom state construction for complex deep links
  getStateFromPath: (path, config) => {
    const defaultState = getStateFromPath(path, config);
    // Ensure Home is always in the stack below deep-linked screen
    if (defaultState?.routes[0]?.name !== 'HomeTabs') {
      return {
        routes: [
          { name: 'HomeTabs' },
          ...defaultState.routes,
        ],
      };
    }
    return defaultState;
  },
};

3. Background/Foreground State Transitions

Mobile apps cycle through distinct lifecycle states. React Native exposes these through the AppState API, but the implications for state management are profound.

App Lifecycle States ════════════════════════════════════════════════════════ ┌──────────┐ user switches ┌────────────┐ │ Active │ ──────────────────→ │ Background │ │ (fg, JS │ ←────────────────── │ (JS may │ │ running)│ user returns │ suspend) │ └──────────┘ └────────────┘ ↑ │ │ app launch │ OS kills (no event!) │ ↓ ┌──────────┐ ┌────────────┐ │ Cold │ │ Dead │ │ Start │ │ (process │ └──────────┘ │ gone) │ ↑ └────────────┘ │ │ └────────────────────────────────┘ user taps icon again iOS-specific: "inactive" state during transitions/notifications Android-specific: "background" can last indefinitely with services

3.1 AppState Listener Pattern

function useAppStateHandler() {
  const appState = useRef(AppState.currentState);
  const backgroundTimestamp = useRef<number | null>(null);

  useEffect(() => {
    const subscription = AppState.addEventListener(
      'change',
      (nextAppState) => {
        const prev = appState.current;

        if (prev === 'active' && nextAppState.match(/inactive|background/)) {
          // App going to background
          backgroundTimestamp.current = Date.now();
          persistCriticalState();
          pauseExpensiveOperations();
        }

        if (prev.match(/inactive|background/) && nextAppState === 'active') {
          // App returning to foreground
          const elapsed = Date.now() - (backgroundTimestamp.current ?? 0);
          const STALE_THRESHOLD = 5 * 60 * 1000; // 5 minutes

          if (elapsed > STALE_THRESHOLD) {
            invalidateStaleQueries();
            refreshAuthToken();
          }
          resumeOperations();
        }

        appState.current = nextAppState;
      }
    );

    return () => subscription.remove();
  }, []);
}
Key Insight

On iOS, the app transitions through inactive before reaching background. On Android, it goes directly to background. The inactive state also fires during system dialogs (permission prompts, incoming calls). Your state management logic must handle these platform differences.

3.2 State Invalidation on Foreground

// TanStack Query integration with AppState
import { focusManager } from '@tanstack/react-query';
import { AppState } from 'react-native';

// Refetch stale queries when app returns to foreground
focusManager.setEventListener((handleFocus) => {
  const subscription = AppState.addEventListener(
    'change',
    (state) => { handleFocus(state === 'active'); }
  );
  return () => subscription.remove();
});

4. Platform-Specific State

Mobile platforms expose state that has no web equivalent. Permissions, device sensors, connectivity, and hardware capabilities all require dedicated state management patterns.

4.1 Permission State Machine

// Permissions follow a state machine: undetermined → granted|denied|blocked
type PermissionStatus =
  | 'undetermined'   // Never asked
  | 'granted'        // User approved
  | 'denied'         // User denied (can ask again on Android)
  | 'blocked'        // Permanently denied (must go to Settings)
  | 'limited';       // iOS 14+ limited photo access

function usePermission(permission: Permission) {
  const [status, setStatus] = useState<PermissionStatus>('undetermined');

  const check = useCallback(async () => {
    const result = await checkPermission(permission);
    setStatus(result);
    return result;
  }, [permission]);

  const request = useCallback(async () => {
    const result = await requestPermission(permission);
    setStatus(result);

    if (result === 'blocked') {
      // Show dialog directing user to Settings
      showSettingsDialog(permission);
    }
    return result;
  }, [permission]);

  // Re-check on foreground (user may have changed in Settings)
  useAppForeground(() => { check(); });

  return { status, check, request };
}

4.2 Network State

import NetInfo from '@react-native-community/netinfo';

function useNetworkState() {
  const [network, setNetwork] = useState({
    isConnected: true,
    type: 'unknown',      // wifi | cellular | none | unknown
    isInternetReachable: true,
    details: null,        // cellular generation, wifi SSID, etc.
  });

  useEffect(() => {
    const unsubscribe = NetInfo.addEventListener((state) => {
      setNetwork({
        isConnected: state.isConnected ?? false,
        type: state.type,
        isInternetReachable: state.isInternetReachable ?? false,
        details: state.details,
      });
    });
    return unsubscribe;
  }, []);

  return network;
}

5. State and Memory Pressure

Mobile devices have limited RAM. The OS will kill backgrounded apps under memory pressure, and even foreground apps can receive memory warnings. State management must account for this.

Memory Pressure Response Strategy ═══════════════════════════════════════════════════════ Memory Warning Level 1 (iOS) / onTrimMemory MODERATE (Android) ├── Drop image caches ├── Clear non-visible screen state from navigation └── Reduce query cache size Memory Warning Level 2 / onTrimMemory RUNNING_LOW ├── Persist critical state to disk immediately ├── Release all non-essential caches └── Cancel background network requests Memory Warning Level 3 / onTrimMemory RUNNING_CRITICAL ├── Emergency state snapshot ├── Release everything possible └── Prepare for potential process death 
// Responding to memory pressure in React Native
import { NativeEventEmitter, NativeModules } from 'react-native';

class MemoryPressureManager {
  private cacheManager: CacheManager;
  private queryClient: QueryClient;

  constructor(cacheManager: CacheManager, queryClient: QueryClient) {
    this.cacheManager = cacheManager;
    this.queryClient = queryClient;
  }

  handleMemoryWarning(level: 'moderate' | 'critical') {
    if (level === 'moderate') {
      this.queryClient.getQueryCache().getAll()
        .filter(q => !q.isActive())
        .forEach(q => this.queryClient.removeQueries({ queryKey: q.queryKey }));

      this.cacheManager.evictLRU(0.5); // Drop 50% of LRU cache
    }

    if (level === 'critical') {
      this.queryClient.clear();
      this.cacheManager.clear();
      persistEmergencySnapshot();
    }
  }
}
Warning

React Navigation keeps all screens in the stack mounted by default. A deep navigation stack (e.g., 10+ screens) can consume significant memory. Use detachInactiveScreens (react-native-screens) to unmount off-screen components, and be aware that this means their state is lost unless persisted.

6. AsyncStorage vs MMKV vs SQLite

Choosing the right persistence layer is a critical architectural decision. Each option has fundamentally different performance characteristics and use cases.

Feature AsyncStorage MMKV SQLite
API Type Async (Promise) Synchronous (JSI) Async / Sync (JSI)
Bridge Uses Bridge/NativeModules JSI (no bridge) JSI (react-native-quick-sqlite)
Read Speed ~5-10ms per read ~0.01-0.05ms per read ~0.1-1ms per query
Data Model Key-value (string only) Key-value (typed) Relational tables
Max Size ~6MB (Android default) Practically unlimited Practically unlimited
Encryption No built-in Built-in AES SQLCipher extension
Query Capability Get by key only Get by key only Full SQL queries
Best For Simple prefs (legacy) Fast KV, state persist Structured data, offline DB

6.1 MMKV — The Modern Default

import { MMKV } from 'react-native-mmkv';

// MMKV uses JSI for synchronous, bridge-free access
const storage = new MMKV({
  id: 'app-storage',
  encryptionKey: 'optional-encryption-key',
});

// Synchronous reads — no await needed
const token = storage.getString('auth.token');
const darkMode = storage.getBoolean('prefs.darkMode');
const lastSync = storage.getNumber('sync.lastTimestamp');

// Synchronous writes
storage.set('auth.token', newToken);
storage.set('prefs.darkMode', true);

// Store complex objects as JSON
storage.set('user.profile', JSON.stringify(profile));
const profile = JSON.parse(storage.getString('user.profile') ?? '{}');
Architect Insight

MMKV's synchronous API via JSI is transformative for state rehydration. With AsyncStorage, your app shows a loading screen while awaiting persisted state. With MMKV, you can read state synchronously during the initial render — enabling instant UI with persisted data. This is why MMKV has become the de facto standard for Redux Persist and Zustand persistence in production RN apps.

7. State Rehydration Patterns

State rehydration is the process of restoring persisted state into memory on app startup. The pattern you choose directly impacts cold start time and perceived performance.

7.1 Eager vs Lazy Rehydration

Eager Rehydration Lazy Rehydration ═══════════════════ ═══════════════════ App Start App Start │ │ ├─ Read ALL persisted state ├─ Read critical state only │ (auth, prefs, cache, etc.) │ (auth token, theme) │ │ ├─ Hydrate entire store ├─ Hydrate minimal store │ │ ├─ Show app (delayed) ├─ Show app (fast!) │ │ └─ Ready ├─ User navigates to screen │ ├─ Read screen-specific state │ on demand │ └─ Hydrate incrementally Trade-off: startup time vs Trade-off: fast start vs consistent initial state potential loading states

7.2 Selective Rehydration with MMKV

// Tiered rehydration: critical state first, rest lazily
class StateRehydrator {
  private storage: MMKV;

  // Tier 1: Synchronous, blocks render (must be fast)
  rehydrateCritical(): CriticalState {
    return {
      authToken: this.storage.getString('auth.token') ?? null,
      theme: this.storage.getString('prefs.theme') ?? 'system',
      onboardingComplete: this.storage.getBoolean('onboarding.done') ?? false,
    };
  }

  // Tier 2: Async, after first render
  async rehydrateSession(): Promise<SessionState> {
    const raw = this.storage.getString('session.state');
    if (!raw) return defaultSessionState();

    const parsed = JSON.parse(raw);
    return migrateSessionState(parsed);
  }

  // Tier 3: On-demand, when screen mounts
  rehydrateScreenState(screenKey: string): ScreenState | null {
    const raw = this.storage.getString(`screen.${screenKey}`);
    return raw ? JSON.parse(raw) : null;
  }
}

7.3 Version-Safe Rehydration

// Always version your persisted state schemas
interface PersistedState {
  _version: number;
  data: unknown;
}

const CURRENT_VERSION = 3;

const migrations: Record<number, (data: any) => any> = {
  1: (data) => ({ ...data, newField: 'default' }),
  2: (data) => {
    const { oldField, ...rest } = data;
    return { ...rest, renamedField: oldField };
  },
  3: (data) => ({ ...data, addedInV3: [] }),
};

function rehydrateWithMigration(raw: string): AppState {
  const persisted: PersistedState = JSON.parse(raw);
  let { _version, data } = persisted;

  while (_version < CURRENT_VERSION) {
    _version++;
    data = migrations[_version](data);
  }

  return data as AppState;
}

8. Architect Considerations

Staff-Level Mental Model

State architecture in React Native requires thinking in four dimensions: (1) scope — component, screen, global, or cross-app; (2) lifetime — ephemeral, session, or persistent; (3) source of truth — client-generated vs server-authoritative; (4) platform constraints — memory limits, background kills, permission state machines. A Staff engineer designs state boundaries that align with these dimensions, choosing the right tool (useState, Zustand, TanStack Query, MMKV) for each cell in this matrix rather than forcing everything through a single global store.
