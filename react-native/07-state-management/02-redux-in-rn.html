1. Redux Toolkit Setup for RN

Redux Toolkit (RTK) is the official, opinionated way to write Redux logic. It wraps the Redux core with utilities that eliminate boilerplate while enforcing best practices — Immer for immutable updates, createAsyncThunk for side effects, and createEntityAdapter for normalized state.

1.1 Store Configuration

import { configureStore, combineReducers } from '@reduxjs/toolkit';
import {
  persistStore, persistReducer,
  FLUSH, REHYDRATE, PAUSE, PERSIST, PURGE, REGISTER,
} from 'redux-persist';
import { reduxStorage } from './mmkv-storage';

const rootReducer = combineReducers({
  auth: authReducer,
  feed: feedReducer,
  notifications: notificationsReducer,
  settings: settingsReducer,
});

const persistConfig = {
  key: 'root',
  version: 1,
  storage: reduxStorage,
  whitelist: ['auth', 'settings'],  // Only persist these slices
  blacklist: ['feed'],                // Never persist feed (server state)
};

const persistedReducer = persistReducer(persistConfig, rootReducer);

export const store = configureStore({
  reducer: persistedReducer,
  middleware: (getDefaultMiddleware) =>
    getDefaultMiddleware({
      serializableCheck: {
        ignoredActions: [FLUSH, REHYDRATE, PAUSE, PERSIST, PURGE, REGISTER],
      },
    }).concat(analyticsMiddleware, navigationMiddleware),
});

export const persistor = persistStore(store);

export type RootState = ReturnType<typeof rootReducer>;
export type AppDispatch = typeof store.dispatch;

1.2 Typed Hooks

import { useDispatch, useSelector, TypedUseSelectorHook } from 'react-redux';

export const useAppDispatch = () => useDispatch<AppDispatch>();

1.3 Slice with createEntityAdapter

import { createSlice, createEntityAdapter, createAsyncThunk } from '@reduxjs/toolkit';

interface Post {
  id: string;
  title: string;
  authorId: string;
  createdAt: number;
}

const postsAdapter = createEntityAdapter<Post>({
  sortComparer: (a, b) => b.createdAt - a.createdAt,
});

export const fetchPosts = createAsyncThunk(
  'posts/fetchAll',
  async (_, { rejectWithValue }) => {
    try {
      const response = await api.getPosts();
      return response.data;
    } catch (err) {
      return rejectWithValue(err.message);
    }
  }
);

const postsSlice = createSlice({
  name: 'posts',
  initialState: postsAdapter.getInitialState({
    loading: false,
    error: null as string | null,
  }),
  reducers: {
    postAdded: postsAdapter.addOne,
    postUpdated: postsAdapter.updateOne,
    postRemoved: postsAdapter.removeOne,
  },
  extraReducers: (builder) => {
    builder
      .addCase(fetchPosts.pending, (state) => {
        state.loading = true;
        state.error = null;
      })
      .addCase(fetchPosts.fulfilled, (state, action) => {
        state.loading = false;
        postsAdapter.setAll(state, action.payload);
      })
      .addCase(fetchPosts.rejected, (state, action) => {
        state.loading = false;
        state.error = action.payload as string;
      });
  },
});

export const {
  selectAll: selectAllPosts,
  selectById: selectPostById,
  selectIds: selectPostIds,
} = postsAdapter.getSelectors((state: RootState) => state.posts);

2. Redux Persist for Mobile

Redux Persist serializes the Redux store to persistent storage and rehydrates it on app launch. For React Native, the storage engine choice is critical — MMKV vastly outperforms AsyncStorage.

2.1 MMKV Storage Engine

import { MMKV } from 'react-native-mmkv';
import { Storage } from 'redux-persist';

const mmkv = new MMKV({ id: 'redux-persist' });

export const reduxStorage: Storage = {
  setItem(key: string, value: string): Promise<void> {
    mmkv.set(key, value);
    return Promise.resolve();
  },
  getItem(key: string): Promise<string | null> {
    const value = mmkv.getString(key);
    return Promise.resolve(value ?? null);
  },
  removeItem(key: string): Promise<void> {
    mmkv.delete(key);
    return Promise.resolve();
  },
};
Key Insight

Redux Persist's API requires Promises, but MMKV is synchronous. The adapter wraps synchronous calls in Promise.resolve(). This means rehydration completes in a single microtask tick rather than waiting for bridge round-trips — reducing cold start rehydration from ~200ms (AsyncStorage) to <5ms (MMKV).

2.2 Nested Persist Configs

// Different slices can have different persistence strategies
const authPersistConfig = {
  key: 'auth',
  storage: encryptedStorage,  // Sensitive data in encrypted MMKV
  whitelist: ['token', 'refreshToken'],
};

const settingsPersistConfig = {
  key: 'settings',
  storage: reduxStorage,      // Regular MMKV for preferences
};

const rootReducer = combineReducers({
  auth: persistReducer(authPersistConfig, authReducer),
  settings: persistReducer(settingsPersistConfig, settingsReducer),
  feed: feedReducer,          // Not persisted
});

3. Middleware Patterns

Redux middleware in React Native handles cross-cutting concerns: analytics tracking, navigation side effects, offline queue management, and crash reporting.

3.1 Analytics Middleware

const analyticsMiddleware: Middleware<{}, RootState> =
  (store) => (next) => (action) => {
    const result = next(action);

    const trackableActions: Record<string, string> = {
      'auth/login/fulfilled': 'user_login_success',
      'auth/logout': 'user_logout',
      'posts/postAdded': 'content_created',
      'purchase/completed': 'purchase_complete',
    };

    const eventName = trackableActions[action.type];
    if (eventName) {
      analytics.track(eventName, {
        payload: action.payload,
        stateSnapshot: selectAnalyticsContext(store.getState()),
      });
    }

    return result;
  };

3.2 Navigation Integration Middleware

const navigationMiddleware: Middleware<{}, RootState> =
  (store) => (next) => (action) => {
    const result = next(action);

    switch (action.type) {
      case 'auth/logout':
        navigationRef.resetRoot({
          index: 0,
          routes: [{ name: 'Login' }],
        });
        break;

      case 'deepLink/received':
        const { path, params } = action.payload;
        const state = store.getState();
        if (state.auth.isAuthenticated) {
          navigationRef.navigate(path, params);
        } else {
          store.dispatch(setPendingDeepLink({ path, params }));
        }
        break;
    }

    return result;
  };

4. Performance Optimization

Redux performance in React Native is dominated by two factors: selector efficiency (how many components re-render on state changes) and state shape (normalized vs nested).

Redux Re-render Flow ════════════════════════════════════════════════════ dispatch(action) │ ▼ Reducer produces new state object │ ▼ useSelector in each connected component runs │ ├── Selector returns same reference → NO re-render │ └── Selector returns new reference → RE-RENDER │ ▼ Component tree reconciliation │ ▼ Shadow tree diff → Native view updates Key insight: Creating new objects/arrays in selectors causes unnecessary re-renders even if data hasn't changed

4.1 Memoized Selectors with Reselect

import { createSelector } from '@reduxjs/toolkit';

// BAD: Creates new array every call → re-renders every subscriber
const selectActivePosts = (state: RootState) =>
  selectAllPosts(state).filter(p => p.status === 'active');

// GOOD: Memoized — only recomputes when posts change
const selectActivePosts = createSelector(
  [selectAllPosts],
  (posts) => posts.filter(p => p.status === 'active')
);

// Parameterized selector with factory pattern
const makeSelectPostsByAuthor = () =>
  createSelector(
    [selectAllPosts, (_: RootState, authorId: string) => authorId],
    (posts, authorId) => posts.filter(p => p.authorId === authorId)
  );

// Each component instance gets its own memoized selector
function AuthorPosts({ authorId }: { authorId: string }) {
  const selectPostsByAuthor = useMemo(makeSelectPostsByAuthor, []);
  const posts = useAppSelector(
    (state) => selectPostsByAuthor(state, authorId)
  );
  // ...
}

4.2 Normalized State Shape

// Normalized state prevents deep nesting and enables O(1) lookups
interface NormalizedState {
  entities: {
    users: Record<string, User>;
    posts: Record<string, Post>;
    comments: Record<string, Comment>;
  };
  ids: {
    users: string[];
    posts: string[];
    comments: string[];
  };
  relationships: {
    postComments: Record<string, string[]>;  // postId → commentIds
    userPosts: Record<string, string[]>;     // userId → postIds
  };
}
Architect Insight

In React Native, unnecessary re-renders are more expensive than on web because each re-render can trigger shadow tree diffs and native view updates across the bridge (or through Fabric). A single poorly-written selector in a FlatList renderItem can cause O(n) re-renders on every state change, each triggering native layout recalculation. Normalized state + memoized selectors is not optional at scale — it's a performance requirement.

6. Offline-First with Redux

Offline-first architecture queues mutations when offline and replays them when connectivity returns. Redux's deterministic state updates make it well-suited for this pattern.

6.1 Offline Queue Middleware

interface QueuedAction {
  id: string;
  action: AnyAction;
  timestamp: number;
  retryCount: number;
}

const offlineMiddleware: Middleware<{}, RootState> =
  (store) => (next) => (action) => {
    if (!action.meta?.requiresNetwork) {
      return next(action);
    }

    const { isConnected } = store.getState().network;

    if (isConnected) {
      return next(action);
    }

    // Queue the action for later replay
    store.dispatch(enqueueAction({
      id: uuid(),
      action,
      timestamp: Date.now(),
      retryCount: 0,
    }));

    // Apply optimistic update locally
    if (action.meta?.optimistic) {
      return next({
        ...action,
        meta: { ...action.meta, isOptimistic: true },
      });
    }
  };

// Replay queued actions when connectivity returns
function useOfflineSync() {
  const dispatch = useAppDispatch();
  const queue = useAppSelector(selectOfflineQueue);
  const isConnected = useAppSelector(selectIsConnected);

  useEffect(() => {
    if (isConnected && queue.length > 0) {
      replayQueue(queue, dispatch);
    }
  }, [isConnected, queue.length]);
}

7. Redux vs Context in RN

Aspect Redux (RTK) React Context
Re-render granularity Selector-based — only subscribers whose selected value changed re-render All consumers re-render on any context value change
Middleware Rich middleware ecosystem (thunks, sagas, offline, analytics) No middleware concept
DevTools Time-travel debugging, action replay, state diff React DevTools only
Persistence Redux Persist with migrations Manual implementation required
Bundle size ~11KB (RTK) + ~2KB (react-redux) 0KB (built-in)
Boilerplate Moderate (RTK reduces significantly) Minimal
Best for Complex apps, shared state, offline, debugging needs Theme, locale, simple auth, dependency injection
Warning

Context's re-render behavior is particularly dangerous in React Native. A context provider wrapping your navigation container will cause every screen to re-render on any context change — including screens not currently visible but still mounted in the navigation stack. This is why Context should be reserved for infrequently-changing values (theme, locale) and not used for frequently-updating state (user input, real-time data).

9. Architect Considerations

Staff-Level Mental Model

Redux in React Native is not just "Redux on mobile." The persistence layer (Redux Persist + MMKV) becomes a critical path for cold start performance. Middleware handles mobile-specific concerns (navigation, offline queues, deep links). Selector optimization directly impacts frame rates because re-renders trigger native view updates. A Staff engineer treats Redux as the synchronization backbone between JS state, persisted state, navigation state, and native state — not just a global variable store.
