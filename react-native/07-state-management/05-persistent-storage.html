Choosing the right storage engine — AsyncStorage, MMKV, SQLite, or Realm — is an architectural decision that impacts cold start time, query capability, encryption, and scalability. This document dissects each option at the internals level expected for Staff-level interviews.

1. AsyncStorage: Legacy & Limitations

AsyncStorage was React Native's original persistence API — a simple async key-value store backed by SQLite on Android and serialized dictionaries on iOS. It's now community-maintained and has significant limitations.

AsyncStorage Architecture ════════════════════════════════════════════════════ JS Thread Bridge Native Thread ───────── ────── ───────────── AsyncStorage.getItem() │ ├── Serialize key ──→ JSON message ──→ Native module │ │ │ (async wait) ├── Android: SQLite query │ │ SELECT value FROM │ │ catalystLocalStorage │ │ WHERE key = ? │ │ │ ├── iOS: Read from │ │ RCTAsyncLocalStorage │ │ (NSUserDefaults or files) │ │ ◄── JSON response ◄── Bridge message ◄───┘ │ └── Promise resolves with string value Problems: • Every read/write crosses the bridge (async, ~5-10ms) • Android: 6MB default limit (RCT_ASYNC_STORAGE_MAX_SIZE) • String-only values (JSON.stringify/parse overhead) • No encryption • No batch transaction guarantees 
// AsyncStorage API — simple but slow
import AsyncStorage from '@react-native-async-storage/async-storage';

// Every operation is async and crosses the bridge
await AsyncStorage.setItem('user', JSON.stringify(user));
const raw = await AsyncStorage.getItem('user');
const user = raw ? JSON.parse(raw) : null;

// Multi-get for batch reads (still async)
const pairs = await AsyncStorage.multiGet(['key1', 'key2', 'key3']);
// pairs = [['key1', 'val1'], ['key2', 'val2'], ['key3', 'val3']]
Warning

AsyncStorage is not deprecated in the official sense, but it is widely considered legacy for new projects. Its bridge-based architecture means every read blocks on an async round-trip — making it unsuitable for synchronous state rehydration, high-frequency writes, or any performance-sensitive path. New projects should default to MMKV.

2. MMKV: JSI-Based Synchronous Storage

MMKV (Memory-Mapped Key-Value) is Tencent's high-performance storage engine, adapted for React Native via JSI. It uses memory-mapped files for near-instant reads and writes without crossing the bridge.

MMKV Architecture ════════════════════════════════════════════════════ JS Thread (Hermes/JSC) │ ├── JSI binding (C++ → JS, no bridge) │ ▼ ┌─────────────────────────────────────────┐ │ MMKV C++ Core │ │ │ │ ┌───────────────────────────────────┐ │ │ │ Memory-Mapped File (mmap) │ │ │ │ ┌─────────────────────────────┐ │ │ │ │ │ Protobuf-encoded KV pairs │ │ │ │ │ │ key1=val1|key2=val2|... │ │ │ │ │ └─────────────────────────────┘ │ │ │ └───────────────────────────────────┘ │ │ │ │ • Reads: Direct memory access (~0.01ms)│ │ • Writes: Append to mmap, OS flushes │ │ • Encryption: AES-CFB (optional) │ │ • CRC32 checksum for integrity │ │ • Auto-trim on file size threshold │ └─────────────────────────────────────────┘ │ ▼ File System (persisted by OS page cache)

2.1 MMKV Internals

import { MMKV } from 'react-native-mmkv';

// Multiple isolated instances for different data domains
const appStorage = new MMKV({ id: 'app' });
const cacheStorage = new MMKV({ id: 'cache' });
const secureStorage = new MMKV({
  id: 'secure',
  encryptionKey: 'my-encryption-key',
});

// Typed getters — no JSON.parse for primitives
appStorage.set('count', 42);
appStorage.set('enabled', true);
appStorage.set('name', 'Alice');
appStorage.set('data', new Uint8Array([1, 2, 3]));

const count = appStorage.getNumber('count');     // 42
const enabled = appStorage.getBoolean('enabled'); // true
const name = appStorage.getString('name');       // 'Alice'
const data = appStorage.getBuffer('data');       // Uint8Array

// Listener for value changes
appStorage.addOnValueChangedListener((key) => {
  console.log(`Key changed: ${key}`);
});

// Enumerate all keys
const allKeys = appStorage.getAllKeys(); // string[]

// Check existence without reading value
const exists = appStorage.contains('name'); // boolean

2.2 MMKV React Hook

import { useMMKVString, useMMKVBoolean, useMMKVNumber } from 'react-native-mmkv';

function SettingsScreen() {
  const [theme, setTheme] = useMMKVString('settings.theme', appStorage);
  const [darkMode, setDarkMode] = useMMKVBoolean('settings.dark', appStorage);
  const [fontSize, setFontSize] = useMMKVNumber('settings.fontSize', appStorage);

  return (
    <View>
      <Switch value={darkMode} onValueChange={setDarkMode} />
      <Slider value={fontSize ?? 16} onValueChange={setFontSize} />
    </View>
  );
}

3. SQLite: Relational Power

SQLite provides full relational database capabilities. Modern RN SQLite libraries use JSI for synchronous access, eliminating the bridge overhead that plagued older implementations.

3.1 react-native-quick-sqlite

import { open } from '@op-engineering/op-sqlite';

const db = open({ name: 'app.db' });

// Synchronous via JSI — no await needed
db.execute(`
  CREATE TABLE IF NOT EXISTS messages (
    id TEXT PRIMARY KEY,
    thread_id TEXT NOT NULL,
    sender_id TEXT NOT NULL,
    content TEXT NOT NULL,
    created_at INTEGER NOT NULL,
    is_read INTEGER DEFAULT 0
  )
`);

db.execute(`CREATE INDEX IF NOT EXISTS idx_messages_thread
  ON messages(thread_id, created_at DESC)`);

// Parameterized queries prevent SQL injection
db.execute(
  'INSERT INTO messages (id, thread_id, sender_id, content, created_at) VALUES (?, ?, ?, ?, ?)',
  [msg.id, msg.threadId, msg.senderId, msg.content, Date.now()]
);

// Query with results
const result = db.execute(
  'SELECT * FROM messages WHERE thread_id = ? ORDER BY created_at DESC LIMIT ?',
  [threadId, 50]
);

// result.rows is an array of objects
const messages: Message[] = result.rows;

3.2 Transaction Support

// Batch operations in a transaction for atomicity and performance
db.execute('BEGIN TRANSACTION');
try {
  for (const msg of messages) {
    db.execute(
      'INSERT OR REPLACE INTO messages VALUES (?, ?, ?, ?, ?)',
      [msg.id, msg.threadId, msg.senderId, msg.content, msg.createdAt]
    );
  }
  db.execute('COMMIT');
} catch (error) {
  db.execute('ROLLBACK');
  throw error;
}

4. Realm: Object Database

Realm is an object database that stores data as native objects rather than rows in tables. It uses a zero-copy architecture — objects are memory-mapped directly from disk, eliminating serialization overhead.

import Realm from 'realm';

class Message extends Realm.Object {
  static schema = {
    name: 'Message',
    primaryKey: 'id',
    properties: {
      id: 'string',
      threadId: { type: 'string', indexed: true },
      senderId: 'string',
      content: 'string',
      createdAt: 'date',
      isRead: { type: 'bool', default: false },
      attachments: 'Attachment[]',
    },
  };
}

const realm = await Realm.open({
  schema: [Message, Attachment, Thread],
  schemaVersion: 3,
  migration: (oldRealm, newRealm) => {
    if (oldRealm.schemaVersion < 2) {
      const messages = newRealm.objects('Message');
      for (const msg of messages) {
        msg.isRead = false;
      }
    }
  },
});

// Live queries — results update automatically
const unreadMessages = realm
  .objects('Message')
  .filtered('threadId == $0 AND isRead == false', threadId)
  .sorted('createdAt', true);

// Write transactions
realm.write(() => {
  realm.create('Message', {
    id: uuid(),
    threadId,
    senderId: currentUser.id,
    content: 'Hello!',
    createdAt: new Date(),
  });
});
Key Insight

Realm's live objects are fundamentally different from SQLite rows. When you hold a reference to a Realm object, it's a live proxy to the underlying data — reads always return the latest value without re-querying. This makes Realm ideal for real-time UIs like chat, but it also means objects can't be used after the Realm is closed or from a different thread without explicit handling.

6. Keychain & Keystore Integration

Sensitive credentials (auth tokens, encryption keys, biometric data) should never be stored in general-purpose storage. iOS Keychain and Android Keystore provide hardware-backed secure storage.

import * as Keychain from 'react-native-keychain';

// Store credentials with biometric protection
await Keychain.setGenericPassword('auth', authToken, {
  accessControl: Keychain.ACCESS_CONTROL.BIOMETRY_CURRENT_SET,
  accessible: Keychain.ACCESSIBLE.WHEN_UNLOCKED_THIS_DEVICE_ONLY,
  securityLevel: Keychain.SECURITY_LEVEL.SECURE_HARDWARE,
  storage: Keychain.STORAGE_TYPE.AES,
});

// Retrieve with biometric prompt
const credentials = await Keychain.getGenericPassword({
  authenticationPrompt: {
    title: 'Authenticate to access your account',
  },
});

if (credentials) {
  const token = credentials.password;
}

// Pattern: Store encryption key in Keychain, use it for MMKV
async function getOrCreateEncryptionKey(): Promise<string> {
  const existing = await Keychain.getGenericPassword({ service: 'mmkv-key' });
  if (existing) return existing.password;

  const key = generateSecureRandomKey();
  await Keychain.setGenericPassword('mmkv', key, { service: 'mmkv-key' });
  return key;
}
Architect Insight

The secure storage pattern for production apps: store the encryption key in Keychain/Keystore (hardware-backed), then use that key to encrypt MMKV or SQLite. This gives you both the performance of MMKV and the security of hardware-backed key storage. Never hardcode encryption keys or store them in the app bundle.

7. Storage Migration Strategies

// Migrating from AsyncStorage to MMKV
async function migrateFromAsyncStorage(mmkv: MMKV) {
  const migrated = mmkv.getBoolean('__migrated_from_async_storage');
  if (migrated) return;

  const keys = await AsyncStorage.getAllKeys();
  const pairs = await AsyncStorage.multiGet(keys);

  for (const [key, value] of pairs) {
    if (value !== null) {
      mmkv.set(key, value);
    }
  }

  mmkv.set('__migrated_from_async_storage', true);

  // Optionally clear AsyncStorage after successful migration
  await AsyncStorage.clear();
}

8. Large Dataset Handling

Storage Selection by Data Size ════════════════════════════════════════════════════ Data Size Recommended Storage Reason ───────── ─────────────────── ────── < 1KB MMKV Fastest for small KV 1KB - 1MB MMKV Still efficient 1MB - 10MB SQLite Queryable, indexed 10MB - 100MB SQLite + pagination Avoid loading all in memory 100MB+ Realm or SQLite Need careful memory management with cursor-based pagination Rule of thumb: • Key-value data → MMKV • Relational/queryable data → SQLite • Object graph with live queries → Realm • Never store large blobs in any DB — use file system 
// Cursor-based pagination for large SQLite datasets
function fetchMessagePage(
  db: Database,
  threadId: string,
  cursor?: number,
  limit = 50
): Message[] {
  const query = cursor
    ? 'SELECT * FROM messages WHERE thread_id = ? AND created_at < ? ORDER BY created_at DESC LIMIT ?'
    : 'SELECT * FROM messages WHERE thread_id = ? ORDER BY created_at DESC LIMIT ?';

  const params = cursor
    ? [threadId, cursor, limit]
    : [threadId, limit];

  return db.execute(query, params).rows;
}

9. Architect Considerations

Staff-Level Mental Model

Storage architecture in React Native follows a tiered model: (1) MMKV for hot data — preferences, tokens, small state snapshots that need synchronous access; (2) SQLite for warm data — queryable collections, offline caches, message history; (3) Keychain/Keystore for secrets — auth tokens, encryption keys, biometric data; (4) File system for cold data — images, documents, large blobs. A Staff engineer never stores everything in one tier. The storage tier determines read latency, query capability, encryption model, and memory footprint — each must be chosen deliberately based on the data's access pattern and sensitivity.
