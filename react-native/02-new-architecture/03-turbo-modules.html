TurboModules

TurboModules are the JSI-based replacement for React Native's legacy NativeModules system. They provide synchronous method calls, lazy initialization, type-safe interfaces generated by Codegen, and direct C++ communication — eliminating the bridge overhead that made native module calls the primary performance bottleneck in complex apps.

1. What TurboModules Are

TurboModules are native modules that communicate with JavaScript through JSI instead of the JSON bridge. Each TurboModule is defined by a spec file (Flow or TypeScript) that describes its interface. Codegen generates C++ glue code from this spec, creating a type-safe, synchronous binding between JS and native code.

Architect Insight

The key architectural decision in TurboModules is the inversion of the initialization model. Legacy NativeModules were eagerly loaded at startup — every registered module was instantiated and its methods were registered on the bridge, even if the app never used them. TurboModules are lazily initialized: the module is not instantiated until JS first accesses it. For apps with 50+ native modules, this can save 200-500ms of startup time.

2. TurboModule Architecture

The TurboModule system has four layers: the JS spec (interface definition), the Codegen output (C++ glue), the TurboModuleManager (registry and lifecycle), and the platform implementation (ObjC++/Java).

3. Spec Files and Type-Safe Interfaces

Every TurboModule starts with a spec file — a TypeScript or Flow interface definition that describes the module's methods, their parameter types, and return types. Codegen reads this spec and generates C++ abstract classes, ObjC++ protocols, and Java interfaces.

3.1 TypeScript Spec

// NativeCamera.ts — TurboModule spec file
import type { TurboModule } from 'react-native';
import { TurboModuleRegistry } from 'react-native';

export interface CameraOptions {
  quality: number;        // 0.0 - 1.0
  flashMode: string;      // 'on' | 'off' | 'auto'
  cameraType: string;     // 'front' | 'back'
}

export interface PhotoResult {
  uri: string;
  width: number;
  height: number;
  fileSize: number;
}

export interface Spec extends TurboModule {
  // Synchronous method — returns immediately
  isAvailable(): boolean;

  // Asynchronous method — returns a Promise
  takePicture(options: CameraOptions): Promise<PhotoResult>;

  // Method with nullable parameter
  setFlashMode(mode: string | null): void;

  // Method returning nullable
  getLastPhoto(): PhotoResult | null;

  // Constants — evaluated once at module init
  getConstants(): {
    maxResolution: number;
    supportedFlashModes: string[];
  };
}

export default TurboModuleRegistry.getEnforcing<Spec>('Camera');

3.2 Codegen Output (C++ Abstract Class)

// Auto-generated by Codegen from NativeCamera.ts
class JSI_EXPORT NativeCameraSpecJSI : public TurboModule {
protected:
  NativeCameraSpecJSI(std::shared_ptr<CallInvoker> jsInvoker);

public:
  virtual bool isAvailable(jsi::Runtime& rt) = 0;

  virtual jsi::Value takePicture(
    jsi::Runtime& rt,
    jsi::Object options) = 0;

  virtual void setFlashMode(
    jsi::Runtime& rt,
    const jsi::Value& mode) = 0; // nullable → jsi::Value

  virtual jsi::Value getLastPhoto(
    jsi::Runtime& rt) = 0; // nullable return → jsi::Value

  virtual jsi::Object getConstants(
    jsi::Runtime& rt) = 0;
};
Warning

The spec file is the single source of truth for the module's interface. If the spec declares isAvailable(): boolean but the native implementation returns a string, the Codegen-generated glue code will catch the type mismatch at compile time (C++) or protocol conformance time (ObjC++). This is a massive improvement over legacy NativeModules, where type mismatches were silent runtime bugs.

4. Lazy Initialization

TurboModules are not instantiated until JavaScript first accesses them. The TurboModuleRegistry installs a global proxy (__turboModuleProxy) via JSI. When JS calls TurboModuleRegistry.get('Camera'), the proxy triggers the TurboModuleManager to instantiate the module.

// How __turboModuleProxy works internally
// Installed during React Native initialization via JSI

runtime.global().setProperty(
    runtime,
    "__turboModuleProxy",
    Function::createFromHostFunction(
        runtime, name, 1,
        [manager](Runtime& rt, const Value&,
                  const Value* args, size_t) -> Value {
            std::string moduleName = args[0].asString(rt).utf8(rt);

            // Check cache first
            if (auto cached = manager->getModule(moduleName)) {
                return std::move(*cached);
            }

            // Instantiate on first access (lazy)
            auto module = manager->createModule(rt, moduleName);
            if (!module) {
                return Value::undefined();
            }

            // Cache for subsequent accesses
            manager->cacheModule(moduleName, module);
            return std::move(*module);
        }));
Key Insight

Lazy initialization has a compounding effect. An app with 60 native modules but only 15 used on the initial screen saves the instantiation cost of 45 modules at startup. Each module's init may involve allocating native resources (database connections, sensor listeners, file handles), so the savings extend beyond just object construction. Measure with performance.now() around TurboModuleRegistry.get() to profile individual module init times.

5. Synchronous Method Calls

TurboModules support both synchronous and asynchronous method calls. Synchronous calls execute on the JS thread and return immediately — no Promise, no callback, no next-tick delay. This enables patterns that were impossible with the bridge.

5.1 Sync vs Async Decision

Use Synchronous When Use Asynchronous When
Reading cached/in-memory data Network requests
Checking feature availability File I/O (large files)
Getting device constants Camera/sensor operations
Layout measurement Database queries (complex)
Cryptographic operations (small data) Image processing
Accessing shared preferences (single key) Bluetooth/NFC operations 
// Synchronous: returns immediately, blocks JS thread
const isAvailable = Camera.isAvailable(); // true — no await needed

// Asynchronous: returns Promise, does not block
const photo = await Camera.takePicture({ quality: 0.8 });

// The spec determines which is which:
//   isAvailable(): boolean          → sync (no Promise)
//   takePicture(): Promise<Result>  → async (returns Promise)
Warning

Synchronous TurboModule calls block the JS thread. A synchronous method that takes 50ms will cause a 3-frame drop at 60fps. Only use synchronous calls for operations that complete in under 1ms. If you're unsure, default to async. Profile with Systrace to verify — what seems fast on a Pixel 7 may be slow on a budget device.

          resolve:(RCTPromiseResolveBlock)resolve
           reject:(RCTPromiseRejectBlock)reject {
  double quality = options.quality();
  NSString *flashMode = options.flashMode();

  // Dispatch to camera queue (off JS thread)
  dispatch_async(self.cameraQueue, ^{
    // ... capture photo ...
    NSDictionary *result = @{

      @"width": @(photo.size.width),
      @"height": @(photo.size.height),
      @"fileSize": @(fileSize)

  });
}

    .maxResolution = 4032 * 3024,

  });
}

6.2 Android Implementation (Kotlin)

// CameraModule.kt
class CameraModule(reactContext: ReactApplicationContext) :
    NativeCameraSpec(reactContext) {

  override fun getName() = "Camera"

  override fun isAvailable(): Boolean {
    val manager = reactApplicationContext
      .getSystemService(Context.CAMERA_SERVICE) as CameraManager
    return manager.cameraIdList.isNotEmpty()
  }

  override fun takePicture(options: ReadableMap, promise: Promise) {
    val quality = options.getDouble("quality")
    val flashMode = options.getString("flashMode")

    // Launch coroutine on IO dispatcher
    CoroutineScope(Dispatchers.IO).launch {
      try {
        val photo = capturePhoto(quality, flashMode)
        val result = Arguments.createMap().apply {
          putString("uri", photo.uri)
          putInt("width", photo.width)
          putInt("height", photo.height)
          putDouble("fileSize", photo.fileSize.toDouble())
        }
        promise.resolve(result)
      } catch (e: Exception) {
        promise.reject("CAPTURE_FAILED", e.message, e)
      }
    }
  }

  override fun getTypedExportedConstants(): Map<String, Any> = mapOf(
    "maxResolution" to 4032 * 3024,
    "supportedFlashModes" to listOf("on", "off", "auto")
  )
}

7. TurboModule vs NativeModule

Aspect Legacy NativeModule TurboModule
Communication Async JSON bridge Synchronous JSI
Initialization Eager (all at startup) Lazy (on first access)
Type safety None (runtime JSON parsing) Codegen from spec (compile-time)
Sync methods Not supported Supported (non-Promise return)
Startup cost (50 modules) ~200-500ms ~0ms (deferred to first use)
Call latency 5-15ms (serialize + bridge + deserialize) ~0.01ms (JSI virtual call)
Interface definition Runtime registration (error-prone) Spec file → Codegen (validated)
Constants Eagerly evaluated at init Lazily evaluated via getConstants()
Engine dependency Coupled to JSC bridge Engine-agnostic via JSI
Error propagation Serialized error objects (lossy) jsi::JSError with stack traces

8. Migration Path

Migrating from NativeModules to TurboModules is a multi-step process. React Native provides an interop layer that allows legacy modules to work alongside TurboModules during the transition.

8.1 Migration Steps

// Step 1: Create a spec file for your existing module
// NativeMyModule.ts
import type { TurboModule } from 'react-native';
import { TurboModuleRegistry } from 'react-native';

export interface Spec extends TurboModule {
  doSomething(value: string): Promise<string>;
}

export default TurboModuleRegistry.getEnforcing<Spec>('MyModule');

// Step 2: Run Codegen to generate C++ glue
// npx react-native codegen

// Step 3: Update native implementation to conform to generated spec
// iOS: Adopt NativeMyModuleSpec protocol
// Android: Extend NativeMyModuleSpec class

// Step 4: Update JS imports
// Before:
import { NativeModules } from 'react-native';
const { MyModule } = NativeModules;

// After:
import MyModule from './NativeMyModule';

9. Backward Compatibility

React Native provides an interop layer that wraps legacy NativeModules in a TurboModule-compatible interface. This allows apps to incrementally migrate without a big-bang rewrite.

Interop Layer — Legacy Module Wrapped as TurboModule JS: TurboModuleRegistry.get('LegacyCamera') │ ▼ TurboModuleManager: "LegacyCamera" not found in TurboModule registry │ ▼ Falls back to interop layer InteropModule wraps NativeModules.LegacyCamera │ ▼ Translates JSI calls → bridge calls Legacy NativeModule executes via bridge Performance: Same as bridge (no JSI benefit) Benefit: Unified API surface during migration
Key Insight

The interop layer means you can enable TurboModules globally without migrating every module first. Modules with specs get full JSI performance; modules without specs fall back to the bridge transparently. This is critical for large apps with dozens of third-party native modules — you don't need to wait for every library to migrate before adopting the new architecture.

10. Architect Considerations

Staff-Level Mental Model

TurboModules represent a shift from runtime-defined interfaces to build-time-defined interfaces. In the old architecture, the module's API was whatever methods happened to be registered at runtime — there was no contract, no validation, and no documentation. TurboModules invert this: the spec file is the contract, Codegen enforces it at build time, and the native implementation must conform. This is the same pattern that gRPC uses (protobuf spec → generated stubs), and it provides the same benefits: type safety, documentation, and cross-platform code generation. The trade-off is rigidity — changing a module's API now requires updating the spec, re-running Codegen, and updating all platform implementations.

Key Interview Points

  • Lazy initialization — modules are instantiated on first access via __turboModuleProxy, saving 200-500ms startup time for apps with many native modules.
  • Synchronous calls — non-Promise methods execute synchronously on the JS thread via JSI. Use for sub-millisecond operations only.
  • Spec files — TypeScript/Flow interfaces that serve as the single source of truth. Codegen produces C++ abstract classes, ObjC++ protocols, and Java/Kotlin base classes.
  • Interop layer — legacy NativeModules work alongside TurboModules via a bridge-based fallback, enabling incremental migration.
  • Platform implementation — iOS uses ObjC++ with protocol conformance; Android uses Java/Kotlin extending generated base classes. Both receive type-safe parameters from Codegen.
  • Constants — getConstants() is lazily evaluated (unlike legacy modules where constants were eagerly computed at startup for all modules).
