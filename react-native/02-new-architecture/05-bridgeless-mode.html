Bridgeless Mode

Bridgeless mode is the culmination of the new architecture — the complete removal of the legacy bridge. Instead of JSON message passing, all communication flows through JSI, RuntimeExecutor, and CallInvoker. This document covers the internal mechanics, migration path, and what changes for app developers.

1. What Is Bridgeless Mode?

Bridgeless mode is React Native running without the CxxBridge (also called RCTCxxBridge on iOS or CatalystInstance on Android). In this mode, the legacy bridge infrastructure — the JSON message queue, the batched native call mechanism, and the NativeModules registry — is entirely removed from the runtime. All JS↔native communication happens through JSI-based primitives: RuntimeExecutor for scheduling work on the JS thread, CallInvoker for dispatching calls across threads, and direct JSI bindings for synchronous access.

Architect Insight

Bridgeless mode is not just "the bridge but faster." It's a fundamentally different execution model. The bridge was a message queue with batching semantics — calls were enqueued, batched, serialized, and processed asynchronously. Bridgeless mode replaces this with direct function dispatch. This changes the concurrency model, error propagation, initialization order, and the entire mental model for how JS and native interact. Staff engineers must understand that this isn't a drop-in replacement — it's a new runtime contract.

3. RuntimeExecutor

RuntimeExecutor is the primary mechanism for scheduling work on the JavaScript thread in bridgeless mode. It replaces the bridge's internal message dispatch. Any native code that needs to call into JavaScript must go through RuntimeExecutor to ensure thread safety — JS engines are single-threaded and all JS execution must happen on the JS thread.

// RuntimeExecutor type definition
using RuntimeExecutor = std::function<
  void(std::function<void(jsi::Runtime& runtime)>&& callback)
>;

// Usage: schedule work on the JS thread from any native thread
void NativeTimerModule::setTimeout(
    jsi::Runtime& rt,
    jsi::Function callback,
    double delayMs) {

  // Capture the callback as a shared JSI value
  auto sharedCallback = std::make_shared<jsi::Function>(
    std::move(callback));

  // Schedule on a background timer thread
  timerThread_.postDelayed(delayMs, [
    this,
    sharedCallback = std::move(sharedCallback)
  ]() {
    // Use RuntimeExecutor to get back to JS thread
    runtimeExecutor_([sharedCallback](jsi::Runtime& rt) {
      sharedCallback->call(rt);
    });
  });
}
Threading Contract

RuntimeExecutor guarantees that the callback runs on the JS thread with a valid jsi::Runtime reference. Never capture jsi::Runtime& across thread boundaries — it is only valid on the JS thread. Always use RuntimeExecutor to re-enter the JS context. Violating this contract causes undefined behavior: crashes, memory corruption, or subtle data races that are extremely difficult to debug.

4. CallInvoker

CallInvoker is the abstraction for dispatching function calls to a specific thread. In bridgeless mode, there are two primary CallInvokers: jsCallInvoker (dispatches to the JS thread) and nativeCallInvoker (dispatches to the native modules thread). TurboModules use CallInvoker to manage async method dispatch.

// CallInvoker interface
class CallInvoker {
public:
  virtual ~CallInvoker() = default;

  // Schedule a function to run on this invoker's thread
  virtual void invokeAsync(std::function<void()>&& func) = 0;

  // Schedule with priority (for concurrent rendering)
  virtual void invokeAsync(
    SchedulerPriority priority,
    std::function<void()>&& func) = 0;

  // Synchronous invocation (blocks until complete)
  virtual void invokeSync(std::function<void()>&& func) = 0;
};

// How TurboModules use CallInvoker for async methods
class TurboModule {
protected:
  std::shared_ptr<CallInvoker> jsInvoker_;

  // Resolve a JS Promise from a native background thread
  void resolvePromise(
      std::shared_ptr<jsi::Function> resolve,
      jsi::Value result) {
    jsInvoker_->invokeAsync([resolve, result = std::move(result)]() {
      resolve->call(*runtime_, result);
    });
  }
};

Key Detail

RuntimeExecutor and jsCallInvoker both dispatch to the JS thread, but they serve different purposes. RuntimeExecutor provides a jsi::Runtime& reference in the callback — use it when you need to create or manipulate JS values. CallInvoker provides a plain void() callback — use it for scheduling work that doesn't need direct JSI access, or when you've already captured the necessary JS references. In practice, TurboModules use CallInvoker for Promise resolution (the resolve/reject functions are already captured), while RuntimeExecutor is used for event emission and global state updates.

5. Event Loop Integration

In bridgeless mode, React Native integrates with the JS engine's event loop rather than implementing its own batching mechanism. With Hermes, this means native microtasks, queueMicrotask(), and proper Promise resolution ordering that matches web semantics.

// Bridgeless event loop integration with Hermes
class HermesInstance : public ReactInstance {
  void initializeRuntime() {
    // Enable microtask queue (Hermes-specific)
    hermesRuntime_->enableMicrotasks();

    // Install native timer functions via JSI (not bridge)
    installTimerFunctions(*hermesRuntime_, runtimeExecutor_);

    // Install global error handler
    installGlobalErrorHandler(*hermesRuntime_);

    // Event loop: drain microtasks after each macrotask
    runtimeExecutor_([](jsi::Runtime& rt) {
      auto& hermesRt = static_cast<HermesRuntime&>(rt);
      hermesRt.drainMicrotasks();
    });
  }
};

6. Module Initialization Without Bridge

In bridge mode, all native modules were registered at startup and initialized eagerly — the bridge needed a complete module registry to dispatch calls. In bridgeless mode, modules are registered with TurboModuleRegistry and initialized lazily on first access via JSI. This fundamentally changes the startup cost model.

// Bridge mode: eager initialization of ALL modules at startup
// (simplified from old RCTCxxBridge)
for (auto& moduleData : moduleDataByName_) {
  // Every module is instantiated, even if never used
  moduleData.second->setUp();   // ~2-5ms per module
}
// 80 modules × 3ms avg = ~240ms startup cost

// Bridgeless mode: lazy initialization via TurboModuleRegistry
class TurboModuleBinding {
  static jsi::Value getModule(
      jsi::Runtime& rt,
      const jsi::Value& name) {
    auto moduleName = name.asString(rt).utf8(rt);

    // Check cache first
    auto it = moduleCache_.find(moduleName);
    if (it != moduleCache_.end()) {
      return jsi::Value(rt, it->second);
    }

    // Create module on first access only
    auto module = provider_->getModule(moduleName, jsInvoker_);
    if (module) {
      moduleCache_[moduleName] = module;
      return module->getJSIValue(rt);
    }
    return jsi::Value::undefined();
  }
};

7. Error Handling

Error handling changes significantly in bridgeless mode. The bridge swallowed exceptions across the boundary — native errors became generic "bridge error" messages. In bridgeless mode, JSI enables synchronous exception propagation, and errors carry full stack traces across the JS↔native boundary.

// Bridgeless error handling: synchronous exceptions propagate
try {
  // Synchronous TurboModule call via JSI
  auto result = turboModule->get(rt, propName);
} catch (const jsi::JSError& jsError) {
  // JS exceptions propagate as C++ exceptions
  LOG(ERROR) << "JS Error: " << jsError.getMessage();
  LOG(ERROR) << "Stack: " << jsError.getStack();
} catch (const std::exception& nativeError) {
  // Native exceptions can be converted to JS errors
  throw jsi::JSError(rt, nativeError.what());
}

// Global error handler in bridgeless mode
void ReactInstance::handleJSError(
    jsi::Runtime& rt,
    const jsi::JSError& error) {
  // Report to ErrorUtils (JS-side error boundary)
  auto errorUtils = rt.global()
    .getPropertyAsObject(rt, "ErrorUtils");
  auto reportFatalError = errorUtils
    .getPropertyAsFunction(rt, "reportFatalError");
  reportFatalError.call(rt, error.value());
}

8. Interop Mode (Backward Compatibility)

Interop mode is the backward compatibility layer that allows legacy bridge-based native modules and Fabric components to work in bridgeless mode. It provides a shim that translates old-style bridge calls into JSI calls, enabling gradual migration without requiring all libraries to update simultaneously.

Architect Insight

Interop mode is a critical architectural decision. Rather than requiring a big-bang migration (which would break the ecosystem), React Native provides a compatibility layer that wraps legacy modules in a TurboModule-compatible interface. This is a classic "strangler fig" pattern — the old system continues to work while being gradually replaced. The trade-off: interop modules don't get the full performance benefits (they still serialize arguments), but they don't block adoption of the new architecture for the rest of the app.

// Interop layer: wrapping a legacy NativeModule as a TurboModule
// (simplified from React Native's ObjCInteropTurboModule)
class LegacyInteropModule : public TurboModule {
public:
  LegacyInteropModule(
    std::shared_ptr<CallInvoker> jsInvoker,
    id<RCTBridgeModule> legacyModule)
    : TurboModule("LegacyInterop", jsInvoker),
      legacyModule_(legacyModule) {}

  jsi::Value get(jsi::Runtime& rt, const jsi::PropNameID& name) {
    auto methodName = name.utf8(rt);

    // Look up method in legacy module's method table
    auto method = [legacyModule_ methodForSelector:
      NSSelectorFromString(methodName)];

    // Return a JSI function that bridges to the legacy method
    return jsi::Function::createFromHostFunction(rt, name, 0,
      [this, methodName](
        jsi::Runtime& rt,
        const jsi::Value& thisVal,
        const jsi::Value* args,
        size_t count) -> jsi::Value {
        // Convert JSI args → NSDictionary (serialization still happens)
        auto nsArgs = convertJSIToObjC(rt, args, count);
        // Invoke legacy method
        [legacyModule_ invokeMethod:methodName withArgs:nsArgs];
        return jsi::Value::undefined();
      });
  }
};

9. Migration Strategy

Enabling bridgeless mode is a per-app configuration change, but the real work is ensuring all native modules and components are compatible. The recommended strategy is incremental: enable the new architecture with interop mode first, then migrate individual modules to TurboModules over time.

// Android: enable bridgeless mode in ReactNativeHost
// gradle.properties
newArchEnabled=true

// MainApplication.kt
class MainApplication : Application(), ReactApplication {
  override val reactNativeHost = object : DefaultReactNativeHost(this) {
    override fun getJSMainModuleName() = "index"
    override fun getUseDeveloperSupport() = BuildConfig.DEBUG
    override val isNewArchEnabled = BuildConfig.IS_NEW_ARCHITECTURE_ENABLED
    override val isHermesEnabled = BuildConfig.IS_HERMES_ENABLED
  }
}

// iOS: enable bridgeless in AppDelegate
// RCTAppDelegate automatically uses bridgeless when new arch is enabled
- (BOOL)bridgelessEnabled {
  return YES;
}
Migration Risk

Enabling bridgeless mode changes the module initialization order. Some legacy modules depend on being initialized before others (implicit ordering via bridge registration). In bridgeless mode, modules initialize lazily on first access, which can expose hidden dependencies. Test thoroughly: if module A depends on module B being initialized first, you need explicit initialization ordering or dependency injection.

10. Performance Improvements

Metric Bridge Mode Bridgeless Mode Improvement
App startup (TTI) ~1200ms ~800ms ~33% faster
Native call latency 5-10ms (async + serialize) 0.005-0.05ms (sync JSI) 100-1000× faster
Memory (module overhead) All modules in memory Only used modules ~40% reduction
JS→Native event latency ~16ms (next frame batch) <1ms (immediate dispatch) 16× faster
Binary data transfer Base64 (33% overhead + encode/decode) Direct ArrayBuffer (zero-copy) No overhead
Key Detail

The startup improvement comes primarily from lazy module initialization, not from the removal of bridge code itself. In a typical app with 60-100 native modules, only 10-20 are used during the initial render. Bridgeless mode defers the other 40-80 modules until they're actually needed, saving 200-400ms of startup time. The per-call latency improvement (100-1000×) matters most for high-frequency operations like gesture handling, animations, and real-time data processing.

11. What Changes for App Developers

For most app developers writing JavaScript/TypeScript, bridgeless mode is transparent — the React Native framework handles the transition. However, there are important behavioral differences that affect debugging, error handling, and module usage patterns.
