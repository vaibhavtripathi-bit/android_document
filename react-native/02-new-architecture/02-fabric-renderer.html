Fabric Renderer

Fabric is React Native's new rendering system — a C++ core that manages a ShadowNode tree, performs synchronous layout via Yoga, supports concurrent rendering, and delegates platform-specific mounting to iOS ComponentViews and Android Fabric Mounting. It replaces the old UIManager and eliminates the asynchronous bridge bottleneck in the rendering pipeline.

1. What Fabric Is

Fabric is the complete rewrite of React Native's rendering system. The old renderer (UIManager) communicated with native views through the asynchronous JSON bridge, creating unavoidable latency between React's virtual tree and the actual platform UI. Fabric moves the entire rendering pipeline into C++, communicates via JSI, and performs layout synchronously on the JS thread.

Architect Insight

Fabric's core design decision was to make the ShadowNode tree a C++ data structure shared between JS and native via JSI. This eliminates the serialization boundary that existed in the old architecture. The JS thread can read layout results synchronously (no round-trip), and the native thread can read the tree without waiting for a bridge message. This shared-memory model is what enables synchronous onLayout and concurrent rendering.

The name "Fabric" comes from the concept of weaving together the JS and native rendering pipelines into a single, unified system — rather than two separate systems communicating through a message queue.

Old Renderer (UIManager) New Renderer (Fabric) ──────────────────────── ───────────────────── React Reconciler React Reconciler │ │ ▼ ▼ JS Shadow Tree C++ ShadowNode Tree (JS objects) (shared via JSI) │ │ │ JSON bridge (async) │ Direct C++ (sync) ▼ ▼ Native UIManager Yoga Layout (C++) (dispatches commands) │ │ ▼ ▼ Mounting Layer Platform Views (diff + commit) (created one-by-one) │ ▼ Platform Views (batched mutations)

2. The ShadowNode Tree

The ShadowNode tree is Fabric's central data structure — a C++ tree that mirrors the React element tree. Each ShadowNode holds the component's props, state, layout information (computed by Yoga), and a reference to its children. The tree is immutable: updates create new tree revisions rather than mutating in place.

2.1 ShadowNode Structure

// Simplified ShadowNode (from ReactCommon/react/renderer/core/ShadowNode.h)
class ShadowNode {
public:
  using Shared = std::shared_ptr<const ShadowNode>;
  using ListOfShared = std::vector<Shared>;

  // Every node has a unique tag (like React's key but for native views)
  Tag getTag() const;

  // Component descriptor (View, Text, Image, etc.)
  const ComponentDescriptor& getComponentDescriptor() const;

  // Props — immutable, shared_ptr for structural sharing
  Props::Shared getProps() const;

  // State — component-local mutable state (e.g., image load state)
  State::Shared getState() const;

  // Children — ordered list of child ShadowNodes
  const ListOfShared& getChildren() const;

  // Layout metrics computed by Yoga
  LayoutMetrics getLayoutMetrics() const;

private:
  Tag tag_;
  Props::Shared props_;
  State::Shared state_;
  ListOfShared children_;
  LayoutMetrics layoutMetrics_;
};

2.2 Immutability and Structural Sharing

ShadowNodes are immutable — once created, they never change. When React updates a component, Fabric creates a new revision of the affected nodes, reusing (sharing) unchanged subtrees via shared_ptr. This is the same structural sharing pattern used by persistent data structures in Clojure or Immutable.js.

Tree Revision N Tree Revision N+1 (before update) (after updating Node B) A ─────────────┐ A' ────────────┐ / \ │ / \ │ B C │ B' C ◄──shared │ / \ │ / \ │ D E │ D' E ◄──shared │ │ │ │ │ Nodes A,B,D cloned (new props) C and E reused (same shared_ptr)
Key Insight

Immutability is what enables concurrent rendering. Multiple tree revisions can exist simultaneously — one being rendered on screen, another being prepared in the background. There are no race conditions because no tree is ever mutated. The mounting layer atomically swaps from one revision to the next.

3. Rendering Pipeline

Fabric's rendering pipeline has three distinct phases: Render (create/update ShadowNodes), Layout (compute positions via Yoga), and Mount (apply changes to platform views). Understanding which thread each phase runs on is critical.

3.1 Three-Phase Pipeline

Phase 1: RENDER (JS Thread) ────────────────────────────────────────────────── React reconciler diffs virtual tree → creates new ShadowNodes via JSI calls to C++ ShadowNodeFactory setState({count: 1}) │ ▼ React Reconciler (JS) │ ▼ JSI call (synchronous) ShadowNodeFactory.createNode(tag, props) │ ▼ New ShadowNode tree revision (C++) Phase 2: LAYOUT (JS Thread or Background Thread) ────────────────────────────────────────────────── Yoga computes layout for the new tree revision. Can run on JS thread (sync) or background (concurrent). ShadowNode tree │ ▼ Yoga::calculateLayout(root, width, height) │ ▼ LayoutMetrics populated on each node (x, y, width, height, padding, border) Phase 3: MOUNT (UI Thread) ────────────────────────────────────────────────── Diff old tree vs new tree → generate MountingInstructions → execute on UI thread as batched view mutations. Tree(N) vs Tree(N+1) │ ▼ Diff algorithm → MountingInstructions [Create(tag=5, View), Update(tag=3, {color:"red"}), Delete(tag=7)] │ ▼ Platform Mounting Manager executes on UI thread

3.2 Synchronous Layout

In the old architecture, layout was computed asynchronously: JS would send layout props over the bridge, native Yoga would compute layout on a separate thread, and results would be sent back over the bridge. This created a minimum one-frame delay for layout-dependent operations.

// Old architecture: onLayout fires asynchronously (next frame)
// JS cannot read layout results synchronously

// Fabric: layout is computed synchronously in C++
// JS can read layout results immediately via JSI
const layout = fabricNode.getLayoutMetrics();
// { x: 0, y: 44, width: 375, height: 600 }
// Available IMMEDIATELY — no async, no next-frame delay
Warning

Synchronous layout means Yoga runs on the JS thread. Complex layout trees (1000+ nodes with flexbox) can block the JS thread for several milliseconds. Fabric mitigates this with incremental layout — only re-computing layout for the subtree that changed — but deeply nested flex layouts remain a performance concern. Profile with Systrace if you see JS thread jank.

4. Concurrent Rendering Support

Fabric was designed from the ground up to support React 18's concurrent rendering. The immutable ShadowNode tree enables multiple in-progress renders without race conditions, and priority-based scheduling allows urgent updates (touch responses) to interrupt lower-priority renders (data fetches).

4.1 Priority-Based Rendering

Priority Source Behavior Example
Discrete (highest) Touch, press events Synchronous, cannot be interrupted Button press handler
Continuous Scroll, drag events High priority, batched per frame Scroll position update
Default setState, network response Normal priority, interruptible API data rendering
Transition (lowest) useTransition, startTransition Background, fully interruptible Screen navigation prep

4.2 Multiple Tree Revisions

Architect Insight

Concurrent rendering in Fabric is possible only because the ShadowNode tree is immutable. If trees were mutable, an interrupted render would leave the tree in an inconsistent state. Immutability guarantees that any tree revision is always complete and consistent — you can safely discard a partially-computed revision and start over from the latest committed tree.

5. View Flattening

View flattening is an optimization where Fabric removes unnecessary wrapper views from the native hierarchy. React components often create deeply nested <View> wrappers for styling purposes that have no visual effect — no background color, no border, no opacity, no event handlers. Fabric detects these "layout-only" views and flattens them out of the native tree.

5.1 Before and After Flattening

React Tree (what you write) Native View Tree (old) Native View Tree (Fabric) ─────────────────────────── ────────────────────── ───────────────────────── <View style={styles.outer}> View (outer) View (outer) <View style={styles.row}> View (row) ◄─ layout only Text "Hello" <Text>Hello</Text> Text "Hello" Text "World" <Text>World</Text> Text "World" </View> View (spacer) ◄─ layout only <View style={{height: 20}} /> </View> 3 native views eliminated 2 views flattened away (5 total) (3 total)

5.2 Flattening Criteria

A view is eligible for flattening if it has none of the following:

// A ShadowNode is "layout-only" if ALL of these are true:
bool isLayoutOnly(const ViewProps& props) {
  return
    props.backgroundColor == ClearColor &&
    props.borderWidth == 0 &&
    props.borderRadius == 0 &&
    props.opacity == 1.0 &&
    props.shadowRadius == 0 &&
    props.transform == Identity &&
    !props.hasEventHandlers &&     // no onPress, onLayout, etc.
    !props.nativeId.empty() &&     // no testID / nativeID
    !props.accessible &&           // no accessibility role
    props.pointerEvents == "box-none";
}
Key Insight

View flattening reduces the native view count by 30-50% in typical apps. Fewer native views means less memory, faster layout, and faster mounting. However, adding onLayout, testID, or a background color to a previously layout-only view will un-flatten it, potentially causing a layout shift. Be deliberate about which views carry visual or interactive properties.

6. The Mounting Layer

The mounting layer is responsible for diffing two tree revisions and producing a minimal set of MountingInstructions that transform the on-screen view hierarchy from revision N to revision N+1. It runs on the UI thread.

6.1 Mounting Instructions

// MountingInstructions generated by tree diff
enum class MountingInstruction {
  Create,       // Create a new native view
  Delete,       // Remove a native view
  Insert,       // Insert a view into a parent at index
  Remove,       // Remove a view from its parent
  UpdateProps,  // Update props on an existing view
  UpdateState,  // Update component state
  UpdateLayout // Update layout metrics (position, size)
};

// Example: updating a Text component's color
// Diff produces: [UpdateProps(tag=42, {color: "red"})]
// Mounting manager calls: textView.setTextColor(Color.RED)

6.2 Batched Mounting

Mounting Transaction (atomic, on UI thread) ┌─────────────────────────────────────────────────┐ │ 1. Pre-mount: allocate native views for Create │ │ 2. Execute mutations in dependency order: │ │ a. Remove children from deleted parents │ │ b. Delete removed views │ │ c. Create new views │ │ d. Insert views into parents │ │ e. Update props on existing views │ │ f. Update layout metrics │ │ 3. Post-mount: trigger onLayout callbacks │ └─────────────────────────────────────────────────┘ All mutations are batched into a single UI thread dispatch — no intermediate states are visible to the user. This eliminates the "flickering" that could occur with the old bridge's unbatched updates.

7. Platform-Specific Renderers

Fabric's C++ core is platform-agnostic. The platform-specific rendering is handled by ComponentViews (iOS) and Fabric Mounting Manager (Android), which translate MountingInstructions into UIKit/Android View operations.

7.1 iOS: ComponentView

// iOS ComponentView — maps a ShadowNode to a UIView
@interface RCTViewComponentView : UIView <RCTComponentViewProtocol>

// Called when props change (from UpdateProps instruction)
- (void)updateProps:(Props::Shared const &)props
          oldProps:(Props::Shared const &)oldProps;

// Called when layout changes (from UpdateLayout instruction)
- (void)updateLayoutMetrics:(LayoutMetrics const &)metrics
              oldLayoutMetrics:(LayoutMetrics const &)oldMetrics;

// Called when state changes (e.g., Image loaded state)
- (void)updateState:(State::Shared const &)state
          oldState:(State::Shared const &)oldState;

// Mounting lifecycle
- (void)mountChildComponentView:(UIView<RCTComponentViewProtocol> *)child
                          index:(NSInteger)index;
- (void)unmountChildComponentView:(UIView<RCTComponentViewProtocol> *)child
                            index:(NSInteger)index;

@end

8. Fabric vs Old UIManager

Aspect Old UIManager Fabric
Shadow tree location JS objects (JS thread) C++ ShadowNodes (shared via JSI)
JS ↔ Native communication Async JSON bridge Synchronous JSI calls
Layout engine Yoga (called async via bridge) Yoga (called sync in C++)
Layout results available Next frame (async callback) Immediately (synchronous)
Concurrent rendering Not possible (mutable tree) Supported (immutable tree)
View flattening Limited (JS-side only) Automatic (C++ analysis)
Mounting Unbatched, one-by-one commands Batched atomic transactions
Cross-platform code Minimal (mostly JS) Extensive (C++ core shared)
Priority scheduling FIFO queue (no priorities) Priority-based (discrete, continuous, default, transition)
Memory overhead Dual trees (JS + native copies) Single C++ tree with structural sharing
Startup cost Lower (no C++ initialization) Slightly higher (C++ infrastructure setup)
Warning

Fabric is not backward-compatible with old native view managers. Components using the legacy requireNativeComponent API must be migrated to Fabric ComponentViews. React Native provides an interop layer that wraps legacy components in a Fabric-compatible shell, but this layer adds overhead and does not support all Fabric features (e.g., concurrent rendering for wrapped components is degraded).

9. Architect Considerations

Staff-Level Mental Model

Fabric is not just a performance optimization — it is an architectural paradigm shift. By moving the shadow tree to C++ and making it immutable, Fabric transforms React Native from a "JS app with native views" into a "C++ rendering engine driven by JS." This has profound implications: the rendering pipeline is now deterministic (same input → same output), testable in isolation (C++ unit tests without a device), and extensible across platforms (the same C++ core renders on iOS, Android, and potentially desktop/web). The key trade-off is complexity: debugging now spans three languages (JS, C++, ObjC/Java), and the learning curve for custom native components is steeper.

Key Interview Points

  • Immutable ShadowNode tree — enables concurrent rendering, structural sharing, and deterministic diffing. Multiple tree revisions can coexist safely.
  • Three-phase pipeline — Render (JS thread, creates ShadowNodes), Layout (Yoga, synchronous), Mount (UI thread, batched mutations). Know which thread each phase runs on.
  • Synchronous layout — JS can read layout results immediately via JSI, eliminating the one-frame delay of the old architecture. Critical for gesture-driven animations.
  • View flattening — automatically removes layout-only wrapper views, reducing native view count by 30-50%. Understand what disqualifies a view from flattening.
  • Priority-based rendering — discrete events (touch) preempt transitions (navigation), ensuring UI responsiveness during heavy renders.
  • Platform mounting — iOS uses ComponentViews (UIView subclasses), Android uses MountingManager. Both receive batched MountingInstructions from the C++ diff algorithm.
