Migration Guide

Migrating from the old architecture to the new architecture (Fabric + TurboModules + Bridgeless) is a multi-phase process. This guide covers the step-by-step configuration for both platforms, library compatibility strategies, interop layers, performance benchmarking, and rollback plans — the practical knowledge expected of a Staff engineer leading a migration.

1. Migration Overview

The migration from old to new architecture involves three independent but related changes: enabling Fabric (new renderer), enabling TurboModules (new module system), and enabling bridgeless mode (removing the bridge). These can be adopted incrementally — you don't need all three at once. The interop layer allows old-style components and modules to work alongside new ones during the transition.

Migration Phases ════════════════ Phase 0: Old Architecture (current state) ┌──────────────────────────────────────────┐ │ Bridge │ Old Renderer │ NativeModules │ └──────────────────────────────────────────┘ │ ▼ Phase 1: Enable New Architecture (with interop) ┌──────────────────────────────────────────┐ │ Bridge │ Fabric │ TurboModules │ │ (kept) │ + interop │ + interop │ │ │ for old │ for old │ │ │ components │ modules │ └──────────────────────────────────────────┘ │ ▼ Phase 2: Migrate libraries (remove interop dependencies) ┌──────────────────────────────────────────┐ │ Bridge │ Fabric │ TurboModules │ │ (kept) │ (all native) │ (all native) │ └──────────────────────────────────────────┘ │ ▼ Phase 3: Enable Bridgeless Mode ┌──────────────────────────────────────────┐ │ No │ Fabric │ TurboModules │ │ Bridge │ (native) │ (native) │ └──────────────────────────────────────────┘
Architect Insight

The phased approach is critical for large apps. A big-bang migration (switching everything at once) is high-risk: if something breaks, you don't know which change caused it. By enabling Fabric with interop first, you validate the rendering pipeline while legacy modules continue working. Then migrate modules one at a time, testing each. Finally, enable bridgeless mode when all modules are native. This is the "strangler fig" pattern applied to mobile architecture — the old system is gradually replaced while remaining functional throughout.

3. Android Configuration

Enabling the new architecture on Android involves Gradle property changes, CMake configuration for C++ compilation, and updating the Application class. The build time will increase significantly on the first build due to C++ compilation of Fabric and TurboModule infrastructure.

# Step 1: gradle.properties
newArchEnabled=true
hermesEnabled=true

# Step 2: android/app/build.gradle — add CMake config
android {
    defaultConfig {
        externalNativeBuild {
            cmake { abiFilters "armeabi-v7a", "arm64-v8a", "x86", "x86_64" }
        }
    }
    externalNativeBuild {
        cmake { path "src/main/jni/CMakeLists.txt" }
    }
}

// Step 3: src/main/jni/CMakeLists.txt
cmake_minimum_required(VERSION 3.13)
project(appmodules)
include(${REACT_ANDROID_DIR}/cmake-utils/ReactNative-application.cmake)

// Step 4: src/main/jni/OnLoad.cpp
#include <DefaultComponentsRegistry.h>
#include <DefaultTurboModuleManagerDelegate.h>
#include <fbjni/fbjni.h>
#include <rncli.h>

JNIEXPORT jint JNICALL JNI_OnLoad(JavaVM* vm, void*) {
  return facebook::jni::initialize(vm, [] {
    facebook::react::DefaultTurboModuleManagerDelegate::
      moduleLookupProviders(rncli_moduleLookupProviders);
    facebook::react::DefaultComponentsRegistry::
      registerComponentDescriptorsFromEntryPoint(rncli_registerProviders);
  });
}
// Step 5: MainApplication.kt — key overrides
class MainApplication : Application(), ReactApplication {
  override val reactNativeHost = object : DefaultReactNativeHost(this) {
    override fun getPackages() = PackageList(this).packages
    override fun getJSMainModuleName() = "index"
    override fun getUseDeveloperSupport() = BuildConfig.DEBUG
    override val isNewArchEnabled = BuildConfig.IS_NEW_ARCHITECTURE_ENABLED
    override val isHermesEnabled = BuildConfig.IS_HERMES_ENABLED
  }
}

4. iOS Configuration

On iOS, the new architecture is enabled through CocoaPods environment variables and AppDelegate changes. The key change is using RCTAppDelegate as the base class, which handles Fabric and TurboModule setup automatically.

# Step 1: ios/Podfile
ENV['RCT_NEW_ARCH_ENABLED'] = '1'

require_relative '../node_modules/react-native/scripts/react_native_pods'
platform :ios, min_ios_version_supported
prepare_react_native_project!

target 'MyApp' do
  config = use_native_modules!
  use_react_native!(
    :path => config[:reactNativePath],
    :hermes_enabled => true,
    :fabric_enabled => true,
    :app_path => "#{Pod::Config.instance.installation_root}/.."
  )
end

# Step 2: cd ios && bundle exec pod install

// Step 3: AppDelegate.mm — use RCTAppDelegate base class
#import <React/RCTAppDelegate.h>

@interface AppDelegate : RCTAppDelegate
@end

@implementation AppDelegate
- (BOOL)application:(UIApplication *)application
    didFinishLaunchingWithOptions:(NSDictionary *)launchOptions {
  self.moduleName = @"MyApp";
  self.initialProps = @{};
  return [super application:application
    didFinishLaunchingWithOptions:launchOptions];
}

- (NSURL *)sourceURLForBridge:(RCTBridge *)bridge {
#if DEBUG
  return [[RCTBundleURLProvider sharedSettings]
    jsBundleURLForBundleRoot:@"index"];
#else
  return [[NSBundle mainBundle]
    URLForResource:@"main" withExtension:@"jsbundle"];
#endif
}

// Optional Phase 3: enable bridgeless
- (BOOL)bridgelessEnabled { return YES; }
@end

5. Library Compatibility

Library compatibility is the biggest practical challenge in migration. Libraries must support Fabric (for UI components) and TurboModules (for native modules) to work natively with the new architecture. The interop layer handles most legacy libraries, but with reduced performance benefits.

Library New Arch Support Migration Notes
react-native-reanimated Full (v3+) Uses JSI directly — benefits most from new arch
react-native-gesture-handler Full (v2.9+) Fabric-native gesture system
react-native-screens Full (v3.20+) Fabric-native screen management
react-native-svg Full (v13.4+) Fabric component specs
react-native-maps Interop Works via interop layer, no native Fabric support
react-native-webview Full (v13+) Fabric-native
@react-native-async-storage Full (v1.19+) TurboModule spec
react-native-mmkv Full JSI-native from the start
Compatibility Check

Before migrating, audit every native dependency. Run npx react-native-new-architecture-check to scan your project for compatibility issues. Libraries that use requireNativeComponent without Fabric specs, or NativeModules without TurboModule specs, will fall back to the interop layer. They'll work, but won't get full performance benefits. Prioritize migrating libraries that are in your app's hot path (navigation, animation, storage).

6. Interop Layers

React Native provides two interop layers that allow legacy code to work with the new architecture without modification. These are enabled by default when you turn on the new architecture.

Interop Layer What It Does Performance Impact
Fabric Interop Wraps requireNativeComponent views as Fabric components Extra wrapper layer, no view flattening for interop views
TurboModule Interop Wraps legacy NativeModules as TurboModules Still serializes arguments (no JSI direct access) 
// Legacy module (works via interop shim):
import { NativeModules } from 'react-native';
NativeModules.LegacyStorage.getItem('key');
// Internally: JSI call → serialize to NSDictionary → legacy method → serialize back

// Migrated module (direct JSI — no serialization):
import NativeStorage from './NativeStorage';
NativeStorage.getItem('key');

7. Common Migration Issues

Issue Symptom Solution
C++ build errors CMake/NDK compilation failures Verify NDK 25+, clean build, check CMakeLists.txt paths
Pod install failures CocoaPods can't resolve new arch specs Update CocoaPods, run pod install --repo-update
findNodeHandle returns null Fabric components don't support findNodeHandle Use ref.current.measure() or UIManager.measure
setNativeProps not working Fabric doesn't support setNativeProps Use state or Reanimated shared values instead
Module not found TurboModuleRegistry returns null Check module registration, verify Codegen ran
Layout measurement timing onLayout fires at different times Fabric layout is synchronous — adjust timing assumptions
Shadow node crashes Crashes in C++ ShadowNode code Check for null props, verify Codegen types match
Build time increase First build takes 2-5× longer Normal — C++ compilation. Subsequent builds use cache
Key Detail

The most common migration blocker is setNativeProps. Many animation and performance optimization patterns in the old architecture relied on setNativeProps to bypass React's reconciler and update native views directly. Fabric doesn't support this — all updates must go through React state or Reanimated's shared values. This is architecturally correct (it prevents the view tree from diverging from React's virtual tree) but requires refactoring animation code. Plan for this as the largest code change in most migrations.

8. Performance Benchmarking

Measure before and after migration to quantify improvements and catch regressions. Focus on metrics that matter to users: startup time, interaction latency, and frame rate during animations.

// Key benchmarks to measure before and after migration

// 1. Startup time (TTI)
performance.mark('app_start');
// ... app initialization ...
performance.mark('app_interactive');
performance.measure('TTI', 'app_start', 'app_interactive');

// 2. Native module call latency
const start = performance.now();
await NativeStorage.getItem('key');
console.log(`Storage read: ${performance.now() - start}ms`);

// 3. Frame rate — use Flipper or systrace (target 60fps)
// 4. Memory — compare heap snapshots (expect lower baseline)
// 5. JS bundle execution — unchanged, but module init deferred

9. Gradual Migration Strategy

Architect Insight

For large apps (50+ screens, 20+ native modules), a gradual migration over 2-4 sprints is safer than a single-sprint switch. The key insight: the interop layer means you don't need 100% library compatibility before enabling the new architecture. Enable it with interop, then migrate libraries one at a time based on impact. Start with libraries in the critical path (navigation, storage, networking), then move to less critical ones. Track each library's migration status in a shared document and assign owners.

// Phase 1: Enable new architecture (Week 1)
// □ Update RN to 0.73+, enable Hermes
// □ Set newArchEnabled=true / RCT_NEW_ARCH_ENABLED=1
// □ Fix build errors, run full test suite
// □ Benchmark: TTI, memory, frame rate

// Phase 2: Migrate critical libraries (Week 2-3)
// □ Update reanimated v3+, gesture-handler v2.9+, screens v3.20+
// □ Migrate custom native modules to TurboModule specs
// □ Replace setNativeProps with state/Reanimated
// □ Re-benchmark after each library migration

// Phase 3: Enable bridgeless mode (Week 4)
// □ Verify all modules work without bridge
// □ Test module initialization ordering
// □ Final benchmark, gradual rollout

10. Testing During Migration

Testing during migration requires running the same test suite against both architectures to catch regressions. The most effective approach is a CI pipeline that builds and tests both configurations in parallel.

# CI: test both architectures in parallel
# .github/workflows/test-new-arch.yml
jobs:
  test-old-arch:
    runs-on: macos-latest
    env: { RCT_NEW_ARCH_ENABLED: "0" }
    steps:
      - run: npx react-native run-ios
      - run: npx detox test --configuration ios.sim.release

  test-new-arch:
    runs-on: macos-latest
    env: { RCT_NEW_ARCH_ENABLED: "1" }
    steps:
      - run: npx react-native run-ios
      - run: npx detox test --configuration ios.sim.release
# Test passing on old but failing on new = migration regression
Test Type What to Verify Tools
Unit tests Component rendering, hooks, state logic Jest + React Native Testing Library
E2E tests Full user flows, navigation, native interactions Detox, Maestro, Appium
Visual regression Layout differences between old/new renderer Screenshot comparison tools
Performance tests TTI, frame rate, memory under load Flashlight, custom benchmarks
Crash monitoring New crash patterns after migration Sentry, Crashlytics, Bugsnag

11. Rollback Strategies

Every migration needs a rollback plan. The new architecture can be disabled with a single build flag, making rollback straightforward — but only if you haven't removed legacy code paths. Keep the old architecture code in place until the migration is fully validated in production.

// Rollback: single flag change per platform
// Android: newArchEnabled=false
// iOS: ENV['RCT_NEW_ARCH_ENABLED'] = '0'  // then pod install

// Rollback only works if you haven't removed:
// 1. Legacy NativeModule registrations
// 2. requireNativeComponent calls
// 3. Bridge-specific initialization code
Rollback Risk

The biggest rollback risk is removing legacy code too early. If you delete requireNativeComponent calls and replace them with Fabric component specs, you can't roll back to the old architecture without reverting those changes. The safe approach: keep both implementations during the migration period. Use conditional compilation or runtime checks to select the right implementation based on the architecture flag. Only remove legacy code after the new architecture has been stable in production for at least one release cycle.

Rollback Decision Tree ══════════════════════ Deployed to production │ ▼ Crash rate up? ──No──► Monitor 2 cycles → Remove legacy code │ Yes ▼ Arch-related? ──No──► Fix normally │ Yes ▼ Set newArchEnabled=false Rebuild + ship hotfix │ ▼ Investigate, fix, re-enable next release
