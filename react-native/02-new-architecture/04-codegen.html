Codegen

React Native Codegen is the build-time tool that reads Flow or TypeScript spec files and generates type-safe native interfaces — C++ abstract classes, ObjC++ protocols, and Java/Kotlin base classes. It is the mechanism that transforms JavaScript type annotations into compile-time guarantees across the JS-native boundary, eliminating an entire class of runtime type errors.

1. What Codegen Is

Codegen is a build-time code generation tool that bridges the gap between JavaScript's type system (Flow/TypeScript) and native type systems (C++, ObjC++, Java). It reads spec files — specially structured JS/TS files that define TurboModule and Fabric component interfaces — and produces native code that enforces those interfaces at compile time.

Architect Insight

Codegen solves the contract enforcement problem in cross-language systems. Without Codegen, the JS side might call module.setVolume("loud") while the native side expects a number — and the error would only surface at runtime, possibly in production. With Codegen, the generated C++ class declares virtual void setVolume(jsi::Runtime& rt, double volume) = 0, and passing a string triggers a compile-time error. This is the same philosophy behind Protocol Buffers, Thrift, and GraphQL code generation — define the contract once, enforce it everywhere.

2. The Codegen Pipeline

The pipeline has three stages: Parse (read spec files), Schema (produce intermediate representation), and Generate (emit native code). Each stage is independent and testable.

2.1 Stage 1: Parsing

Codegen supports both Flow and TypeScript specs. The parser extracts the module interface, method signatures, parameter types, and return types from the spec file's AST.

// TypeScript spec — Codegen parses this file
import type { TurboModule } from 'react-native';
import { TurboModuleRegistry } from 'react-native';

export interface Spec extends TurboModule {
  getVolume(): number;
  setVolume(level: number): void;
  playTrack(uri: string, options?: PlayOptions): Promise<PlayResult>;
}

export default TurboModuleRegistry.getEnforcing<Spec>('AudioPlayer');

2.2 Stage 2: Schema Generation

The parser output is transformed into a platform-agnostic JSON schema. This schema is the intermediate representation that all native generators consume.

// Intermediate schema (simplified) produced by Codegen
{
  "modules": {
    "AudioPlayer": {
      "type": "NativeModule",
      "spec": {
        "methods": [
          {
            "name": "getVolume",
            "typeAnnotation": {
              "type": "FunctionTypeAnnotation",
              "params": [],
              "returnTypeAnnotation": {
                "type": "NumberTypeAnnotation"
              }
            }
          },
          {
            "name": "setVolume",
            "typeAnnotation": {
              "type": "FunctionTypeAnnotation",
              "params": [
                {
                  "name": "level",
                  "typeAnnotation": { "type": "NumberTypeAnnotation" }
                }
              ],
              "returnTypeAnnotation": {
                "type": "VoidTypeAnnotation"
              }
            }
          },
          {
            "name": "playTrack",
            "typeAnnotation": {
              "type": "FunctionTypeAnnotation",
              "params": [
                {
                  "name": "uri",
                  "typeAnnotation": { "type": "StringTypeAnnotation" }
                },
                {
                  "name": "options",
                  "optional": true,
                  "typeAnnotation": { "type": "ObjectTypeAnnotation", "...": "..." }
                }
              ],
              "returnTypeAnnotation": {
                "type": "PromiseTypeAnnotation",
                "elementType": { "type": "ObjectTypeAnnotation", "...": "..." }
              }
            }
          }
        ]
      }
    }
  }
}

2.3 Stage 3: Native Code Generation

The schema is consumed by platform-specific generators that produce compilable native code. Each generator understands its target language's type system and idioms.

4. Type Mapping (JS → Native)

Codegen maps JavaScript/TypeScript types to their native equivalents across three target languages. Understanding this mapping is essential for writing correct spec files.

4.1 Primitive Type Mapping

TypeScript Type C++ (JSI) ObjC++ Java/Kotlin
boolean bool BOOL boolean / Boolean
number double double double / Double
string jsi::String NSString * String
void void void void
Promise<T> jsi::Value (async) resolve/reject blocks Promise
Array<T> jsi::Array NSArray * ReadableArray
Object (typed) jsi::Object Generated struct ReadableMap
T | null std::optional<T> nullable T @Nullable T

4.2 Float vs Double

Warning

JavaScript's number is always IEEE 754 double-precision (64-bit). Codegen maps it to double in all native languages. If your native API expects float (32-bit), you must cast explicitly in your implementation. Codegen does not support a Float32 annotation — this is a deliberate design choice to match JavaScript's semantics. Precision loss from double→float narrowing is your responsibility to handle.

5. Native Code Generation

5.1 C++ Output (Shared Across Platforms)

// Auto-generated: NativeAudioPlayerSpecJSI.h
#pragma once

#include <ReactCommon/TurboModule.h>
#include <jsi/jsi.h>

namespace facebook::react {

class JSI_EXPORT NativeAudioPlayerSpecJSI : public TurboModule {
protected:
  NativeAudioPlayerSpecJSI(
    std::shared_ptr<CallInvoker> jsInvoker);

public:
  // getVolume(): number → synchronous, returns double
  virtual double getVolume(jsi::Runtime& rt) = 0;

  // setVolume(level: number): void → synchronous
  virtual void setVolume(jsi::Runtime& rt, double level) = 0;

  // playTrack(uri, options?): Promise → async with resolve/reject
  virtual jsi::Value playTrack(
    jsi::Runtime& rt,
    jsi::String uri,
    std::optional<jsi::Object> options) = 0;
};

} // namespace facebook::react

5.2 ObjC++ Output (iOS)

// Auto-generated: NativeAudioPlayerSpec.h
@protocol NativeAudioPlayerSpec <RCTBridgeModule, RCTTurboModule>

- (NSNumber *)getVolume;
- (void)setVolume:(double)level;
- (void)playTrack:(NSString *)uri
        options:(NSDictionary *)options
        resolve:(RCTPromiseResolveBlock)resolve
         reject:(RCTPromiseRejectBlock)reject;

@end

// Conformance check — compile error if methods are missing
namespace facebook::react {
  static_assert(
    std::is_base_of_v<NativeAudioPlayerSpec, YourImplementation>,
    "YourImplementation must conform to NativeAudioPlayerSpec");
}

6. Enums and Nullable Types

6.1 Enum Handling

Codegen supports string literal unions as enums. They are mapped to platform-appropriate enum types.

// TypeScript spec with enum-like union type
export interface Spec extends TurboModule {
  setRepeatMode(mode: 'none' | 'one' | 'all'): void;
  getRepeatMode(): 'none' | 'one' | 'all';
}
// Generated C++ enum
enum class NativeAudioPlayerRepeatMode : int32_t {
  None = 0,
  One = 1,
  All = 2
};

// Generated conversion functions
inline NativeAudioPlayerRepeatMode
parseRepeatMode(jsi::Runtime& rt, const jsi::Value& value) {
  auto str = value.asString(rt).utf8(rt);
  if (str == "none") return NativeAudioPlayerRepeatMode::None;
  if (str == "one")  return NativeAudioPlayerRepeatMode::One;
  if (str == "all")  return NativeAudioPlayerRepeatMode::All;
  throw jsi::JSError(rt, "Invalid RepeatMode: " + str);
}

6.2 Nullable Types

// TypeScript spec with nullable types
export interface Spec extends TurboModule {
  // Nullable parameter
  setTitle(title: string | null): void;

  // Nullable return
  getCurrentTrack(): TrackInfo | null;

  // Optional parameter (implicitly nullable)
  seek(position: number, tolerance?: number): void;
}
// Generated C++ — nullable maps to std::optional or jsi::Value
virtual void setTitle(
  jsi::Runtime& rt,
  const jsi::Value& title) = 0;  // Value can be null or string

virtual jsi::Value getCurrentTrack(
  jsi::Runtime& rt) = 0;  // Returns null or Object

virtual void seek(
  jsi::Runtime& rt,
  double position,
  std::optional<double> tolerance) = 0;  // optional → std::optional
Key Insight

Codegen distinguishes between T | null (explicitly nullable) and T? / optional parameter (may be omitted). In C++, both map to std::optional or jsi::Value, but the generated validation code differs: a nullable parameter that receives undefined will throw, while an optional parameter that receives undefined will be treated as absent. This distinction matters for API correctness.

7. Custom Type Support

Codegen supports structured object types defined inline or as separate interfaces. These are mapped to generated structs (C++), typed dictionaries (ObjC++), or ReadableMap accessors (Java).

// Custom types in spec file
export interface TrackInfo {
  id: string;
  title: string;
  artist: string;
  duration: number;
  artwork: string | null;
  metadata: Readonly<{
    genre: string;
    year: number;
    bitrate: number;
  }>;
}
// Generated C++ struct builder (iOS)
namespace JS::NativeAudioPlayer {
  struct TrackInfo {
    NSString *id() const;
    NSString *title() const;
    NSString *artist() const;
    double duration() const;
    NSString *_Nullable artwork() const;

    struct Metadata {
      NSString *genre() const;
      double year() const;
      double bitrate() const;
    };
    Metadata metadata() const;
  };
}

8. Codegen CLI

Codegen can be invoked manually via the CLI or automatically as part of the build process. Manual invocation is useful for debugging generated code or verifying spec correctness.

# Generate native code from all spec files in the project
npx react-native codegen

# Generate for a specific library
npx react-native codegen --path ./node_modules/react-native-camera

# Specify output directory
npx react-native codegen --outputDir ./codegen-output

# Generate only for a specific platform
npx react-native codegen --platform ios
npx react-native codegen --platform android

# Verbose output for debugging
npx react-native codegen --verbose

8.1 package.json Configuration

// package.json — Codegen configuration for a library
{
  "name": "react-native-audio-player",
  "version": "2.0.0",
  "codegenConfig": {
    "name": "RNAudioPlayerSpec",
    "type": "modules",
    "jsSrcsDir": "./src",
    "android": {
      "javaPackageName": "com.audioplayer"
    }
  }
}
Warning

The codegenConfig.name must be unique across all libraries in your project. If two libraries use the same name, the generated files will collide. Convention is to use the library name with "Spec" suffix: RNAudioPlayerSpec, RNCameraSpec, etc. Also note that type must be either "modules" (for TurboModules) or "components" (for Fabric components) or "all" (for both).

9. Build System Integration

Codegen integrates with both Gradle (Android) and CocoaPods (iOS) to run automatically during the build process. Understanding this integration is essential for debugging build failures.

9.1 iOS: CocoaPods Integration

# Podfile — Codegen runs during pod install
# The react_native_post_install hook triggers Codegen

target 'MyApp' do
  config = use_native_modules!

  use_react_native!(
    :path => config[:reactNativePath],
    :hermes_enabled => true,
    :fabric_enabled => true,
    :app_path => "#{Pod::Config.instance.installation_root}/.."
  )

  post_install do |installer|
    react_native_post_install(
      installer,
      config[:reactNativePath],
      :mac_catalyst_enabled => false
    )
    # ↑ This triggers Codegen for all libraries with codegenConfig
  end
end

9.2 Android: Gradle Integration

// android/app/build.gradle
// Codegen is triggered by the react-native Gradle plugin

apply plugin: "com.facebook.react"

react {
    // Codegen configuration
    jsRootDir = file("../src")
    libraryName = "AppSpec"
    codegenJavaPackageName = "com.myapp"
}

// Generated code output location:
// android/app/build/generated/source/codegen/
// ├── java/com/myapp/NativeAudioPlayerSpec.java
// ├── jni/
// │   ├── NativeAudioPlayerSpecJSI.h
// │   └── NativeAudioPlayerSpecJSI.cpp
// └── schema.json

10. Architect Considerations

Staff-Level Mental Model

Codegen is the contract enforcement layer of the new architecture. It transforms the informal "JS calls native with these arguments" convention into a formal, machine-verified contract. This has three architectural consequences: (1) Type errors shift left — from runtime crashes in production to compile-time errors in CI. (2) Documentation is generated — the spec file serves as both the interface definition and the documentation. (3) Cross-platform consistency is enforced — the same spec generates iOS and Android code, guaranteeing identical interfaces. The trade-off is build complexity: Codegen adds a build step, increases build times, and introduces a new failure mode (spec parsing errors). For large teams, the type safety benefits far outweigh the build cost.

Key Interview Points

  • Three-stage pipeline — Parse (Flow/TS → AST), Schema (AST → JSON), Generate (JSON → C++/ObjC++/Java). Each stage is independent and testable.
  • Type mapping — JS number → C++ double, string → jsi::String, Promise<T> → resolve/reject callbacks. Nullable types use std::optional or jsi::Value.
  • Enum support — string literal unions map to C++ enum class with generated parse/serialize functions that throw on invalid values.
  • Build integration — CocoaPods runs Codegen during pod install via react_native_post_install; Gradle runs it via the com.facebook.react plugin. Generated code lives in build output directories.
  • codegenConfig — declared in package.json with name (unique identifier), type (modules/components/all), and jsSrcsDir (where to find spec files).
  • Contract enforcement — if native implementation doesn't match the spec, the build fails. This eliminates an entire class of runtime type errors that plagued the old NativeModules system.
