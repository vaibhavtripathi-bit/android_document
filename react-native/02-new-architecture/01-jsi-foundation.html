JSI Foundation

The JavaScript Interface (JSI) is the C++ abstraction layer that replaces the asynchronous JSON bridge with direct, synchronous function calls between JavaScript and native code. It is the single most important architectural change in React Native's history.

1. What JSI Is and Why It Exists

JSI (JavaScript Interface) is a lightweight, general-purpose C++ API that provides an abstraction over any JavaScript engine. It defines a contract for embedding a JS runtime into a C++ host application, enabling direct function calls, shared memory, and synchronous communication — none of which were possible with the old bridge.

Architect Insight

JSI is engine-agnostic by design. The same C++ code works with Hermes, JavaScriptCore, or V8. This decoupling was a deliberate architectural choice: React Native's core no longer depends on any specific JS engine. This is what enabled the Hermes migration — without JSI, swapping engines would have required rewriting every native module.

The old bridge serialized every cross-boundary call into JSON, batched messages asynchronously, and deserialized on the other side. This created three fundamental problems: serialization overhead, asynchronous-only communication, and no shared memory. JSI eliminates all three.

Old Architecture (Bridge) New Architecture (JSI) ───────────────────────── ────────────────────── JS Thread JS Thread ┌──────────┐ ┌──────────┐ │ JS Code │ │ JS Code │ └────┬─────┘ └────┬─────┘ │ JSON.stringify() │ Direct C++ call ▼ ▼ ┌──────────┐ ┌──────────────┐ │ Bridge │ async, batched │ jsi::Runtime │ │ Queue │ JSON messages │ (C++ API) │ └────┬─────┘ └──────┬───────┘ │ JSON.parse() │ Synchronous ▼ ▼ ┌──────────┐ ┌──────────────┐ │ Native │ │ Native Code │ │ Module │ │ (C++) │ └──────────┘ └──────────────┘ Latency: ~5-15ms per call Latency: ~0.01ms per call Memory: Copy semantics Memory: Shared references Calls: Async only Calls: Sync + Async

2. jsi::Runtime Abstraction

jsi::Runtime is the central abstraction — an abstract C++ class that every JS engine must implement. It provides methods to create values, evaluate scripts, access the global object, and manage object lifecycles. All JSI operations flow through a Runtime reference.

2.1 Core Runtime Interface

// Simplified jsi::Runtime interface (from ReactCommon/jsi/jsi.h)
namespace facebook {
namespace jsi {

class Runtime {
public:
  virtual ~Runtime();

  // Evaluate a JS string and return the result
  virtual Value evaluateJavaScript(
    const std::shared_ptr<const Buffer>& buffer,
    const std::string& sourceURL) = 0;

  // Access the global object (equivalent to globalThis in JS)
  virtual Object global() = 0;

  // Create JS values from C++ types
  virtual String createStringFromUtf8(
    const uint8_t* utf8, size_t length) = 0;

  virtual Object createObject() = 0;

  // PropNameID for efficient property access (interned strings)
  virtual PropNameID createPropNameIDFromUtf8(
    const uint8_t* utf8, size_t length) = 0;
};

} // namespace jsi
} // namespace facebook

2.2 JSI Value Types

JSI maps JavaScript's type system into C++ with a discriminated union called jsi::Value. Every JS value crossing the boundary is represented as one of these types:

JS Type JSI C++ Type Ownership Notes
undefined Value::undefined() Value type Default return for void functions
null Value::null() Value type Distinct from undefined in JSI
boolean Value(bool) Value type No heap allocation
number Value(double) Value type Always IEEE 754 double
string jsi::String GC-managed pointer Immutable, engine-interned
object jsi::Object GC-managed pointer Includes arrays, functions
symbol jsi::Symbol GC-managed pointer ES6 Symbol support
bigint jsi::BigInt GC-managed pointer Added in newer JSI versions
Warning

jsi::String, jsi::Object, and jsi::Symbol are move-only types — they cannot be copied. This is intentional: they hold pointers into the JS engine's GC heap, and copying would create dangling references if the GC relocates the object. Always use std::move() or pass by reference.

3. HostObject and HostFunction

HostObject and HostFunction are the two primary mechanisms for exposing C++ functionality to JavaScript. They are the building blocks on which TurboModules, Fabric, and every JSI-based library are built.

3.1 HostFunction — Exposing C++ Functions to JS

A HostFunction is a C++ lambda or function that can be called directly from JavaScript. No serialization, no bridge, no async queue — the JS engine calls the C++ function pointer synchronously on the JS thread.

// Register a synchronous C++ function callable from JS
auto multiply = Function::createFromHostFunction(
    runtime,
    PropNameID::forAscii(runtime, "multiply"),
    2, // parameter count
    [](Runtime& rt,
       const Value& thisVal,
       const Value* args,
       size_t count) -> Value {

        double a = args[0].asNumber();
        double b = args[1].asNumber();
        return Value(a * b);
    });

// Install on the global object
runtime.global().setProperty(
    runtime,
    PropNameID::forAscii(runtime, "multiply"),
    std::move(multiply));
// Now callable from JavaScript — synchronous, zero overhead
const result = multiply(6, 7); // 42 — no bridge, no JSON, no async

3.2 HostObject — Exposing C++ Objects to JS

A HostObject is a C++ object that appears as a regular JavaScript object with get and set traps — similar to a Proxy. Property access on the JS side triggers C++ method calls.

class DatabaseHostObject : public jsi::HostObject {
public:
  DatabaseHostObject(std::shared_ptr<Database> db) : db_(db) {}

  Value get(Runtime& rt, const PropNameID& name) override {
    auto propName = name.utf8(rt);

    if (propName == "execute") {
      return Function::createFromHostFunction(
        rt, name, 1,
        [this](Runtime& rt, const Value&,
               const Value* args, size_t) -> Value {
          std::string sql = args[0].asString(rt).utf8(rt);
          auto result = db_->execute(sql);
          return convertToJSI(rt, result);
        });
    }

    if (propName == "isOpen") {
      return Value(db_->isOpen());
    }

    return Value::undefined();
  }

  void set(Runtime&, const PropNameID&, const Value&) override {
    throw std::runtime_error("DatabaseHostObject is read-only");
  }

private:
  std::shared_ptr<Database> db_;
};
// Install the HostObject on the JS global
auto db = std::make_shared<DatabaseHostObject>(database);
runtime.global().setProperty(
    runtime,
    PropNameID::forAscii(runtime, "nativeDB"),
    Object::createFromHostObject(runtime, db));
// JavaScript usage — looks like a normal object
const rows = nativeDB.execute("SELECT * FROM users");
console.log(nativeDB.isOpen); // true — synchronous property access
Key Insight

HostObject's get trap is called on every property access. This means you can implement lazy computation, caching, or delegation patterns. Libraries like react-native-mmkv use this to expose key-value storage where storage.username triggers a synchronous C++ read from memory-mapped files — no async, no bridge, no Promise.

4. How JSI Replaces the Bridge

The bridge was a message-passing system: JS would enqueue a JSON message, the bridge would batch and transfer it, and native would deserialize and dispatch. JSI replaces this entire pipeline with direct function pointers.

4.1 Bridge Call Flow (Old)

JS: NativeModules.Camera.takePicture({quality: 0.8}) 1. JS serializes: JSON.stringify({module:"Camera", method:"takePicture", args:[{quality:0.8}]}) 2. Message queued in MessageQueue.js (batched, min 5ms delay) 3. Bridge transfers JSON string to native thread 4. Native deserializes: JSON.parse(...) 5. Native dispatches to CameraModule.takePicture() 6. Result serialized back: JSON.stringify({uri: "file://..."}) 7. Bridge transfers JSON back to JS thread 8. JS deserializes result Total overhead: ~10ms+ per round trip (serialization + queuing + deserialization) Memory: Full copy of all data at each boundary crossing

4.2 JSI Call Flow (New)

JS: global.CameraModule.takePicture({quality: 0.8}) 1. JS engine calls C++ HostFunction pointer directly 2. C++ receives jsi::Object with quality property (zero-copy reference) 3. C++ calls platform camera API 4. C++ returns jsi::String with URI Total overhead: ~0.01ms (one virtual function call) Memory: Shared references, no copying
Aspect Old Bridge JSI
Serialization JSON.stringify / JSON.parse on every call None — direct C++ value passing
Call semantics Asynchronous only (batched queue) Synchronous by default, async optional
Latency per call 5-15ms (serialization + queue + deserialization) ~0.01ms (virtual function dispatch)
Memory model Full copy at each boundary Shared references via GC pointers
Engine coupling Tightly coupled to JSC Engine-agnostic (Hermes, JSC, V8)
Type safety None — runtime JSON parsing C++ type checking + Codegen
Threading Bridge thread required Runs on JS thread (no extra thread)
Batching Required for performance Unnecessary — calls are already fast

5. Memory Management Across Boundaries

Memory management is the hardest problem in JSI. JavaScript uses garbage collection; C++ uses manual or RAII-based memory management. JSI must bridge these two worlds without leaks or use-after-free bugs.

5.1 Pointer Types and Ownership

// jsi::Object, jsi::String, jsi::Function hold GC-tracked pointers.
// They are MOVE-ONLY — the GC must know about every reference.

// WRONG: Capturing a jsi::Object by value in a long-lived lambda
auto obj = args[0].asObject(rt);
someQueue.push([obj]() {  // ❌ obj may be GC'd before lambda runs
    // use-after-free crash
});

// CORRECT: Use jsi::Value::asObject + shared_ptr or copy data out
auto data = args[0].asObject(rt)
    .getProperty(rt, "name")
    .asString(rt)
    .utf8(rt);  // ✅ Extracts std::string — safe to move across threads

someQueue.push([data = std::move(data)]() {
    // data is a plain std::string — no GC dependency
});

5.2 The shared_ptr Bridge Pattern

HostObjects use std::shared_ptr for preventing premature destruction. When JS holds a reference to a HostObject, the shared_ptr prevents the C++ side from being destroyed. When JS drops the reference, the GC invokes the destructor.

JavaScript GC Heap C++ Heap ┌──────────────────┐ ┌──────────────────────┐ │ JS Object │ │ HostObject │ │ (proxy) │─ weak ref ──────▶│ ref_count: 2 │ │ │ │ ┌────────────────┐ │ └──────────────────┘ │ │ shared_ptr to │ │ │ │ native resource│ │ C++ code also holds ──── shared_ptr ─▶│ └────────────────┘ │ └──────────────────────┘ When JS GC collects the proxy: ref_count → 1 (C++ still holds) When C++ releases shared_ptr: ref_count → 0 (destructor runs)
Warning

Never access JSI objects from non-JS threads. The jsi::Runtime is not thread-safe. If you need to pass data to a background thread, extract primitive C++ types (std::string, double, std::vector) first. To call back into JS from a background thread, use RuntimeExecutor or CallInvoker to schedule work on the JS thread.

6. Error Handling Across the Boundary

C++ exceptions and JavaScript exceptions are different systems. JSI provides a translation layer via jsi::JSError, which wraps a JavaScript Error object as a C++ exception.

6.1 C++ to JS Error Propagation

// Throwing from C++ — appears as a JS exception
auto riskyFunction = Function::createFromHostFunction(
    runtime, name, 1,
    [](Runtime& rt, const Value&,
       const Value* args, size_t count) -> Value {

        if (count == 0) {
            // This becomes a JS Error with message and stack
            throw jsi::JSError(rt, "Expected at least one argument");
        }

        if (!args[0].isString()) {
            // You can also throw with a JS value
            auto error = Object(rt);
            error.setProperty(rt, "code", "INVALID_TYPE");
            error.setProperty(rt, "message", "Argument must be string");
            throw jsi::JSError(rt, std::move(error));
        }

        return Value::undefined();
    });

6.2 JS to C++ Error Catching

// Calling JS from C++ — catching JS exceptions
try {
    auto jsFunc = runtime.global()
        .getPropertyAsFunction(runtime, "riskyJSFunction");
    auto result = jsFunc.call(runtime);
} catch (const jsi::JSError& e) {
    // e.getMessage() — the Error message string
    // e.getStack()   — the JS stack trace
    // e.value()      — the raw jsi::Value that was thrown
    LOG("JS Error: %s\nStack: %s", e.getMessage().c_str(),
        e.getStack().c_str());
} catch (const std::exception& e) {
    // Non-JS C++ exceptions (e.g., std::bad_alloc)
    LOG("C++ Error: %s", e.what());
}
Key Insight

JSI error propagation is synchronous and stack-preserving. When a HostFunction throws jsi::JSError, the JS engine receives it as a native exception with a proper stack trace that includes both the C++ throw site and the JS call site. This is a massive debugging improvement over the old bridge, where errors were serialized as JSON and lost their stack context.

7. JSI vs Bridge Performance

The performance difference between JSI and the bridge is not incremental — it is orders of magnitude. This section quantifies the difference with real-world benchmarks.

7.1 Micro-benchmarks

Architect Insight

JSI's performance characteristics change the design space for React Native apps. Operations that were impractical over the bridge — synchronous layout measurement, real-time gesture processing, frame-by-frame animation control — become viable. This is why Reanimated 2+ and Fabric both depend on JSI: they need synchronous, sub-millisecond cross-boundary calls to hit 60fps.

8. Advanced JSI Patterns

8.1 Shared ArrayBuffer for Zero-Copy Data

// Expose a C++ buffer directly to JS without copying
class SharedBufferHostObject : public jsi::HostObject {
  std::vector<uint8_t> buffer_;

public:
  Value get(Runtime& rt, const PropNameID& name) override {
    if (name.utf8(rt) == "getBuffer") {
      return Function::createFromHostFunction(
        rt, name, 0,
        [this](Runtime& rt, const Value&,
               const Value*, size_t) -> Value {
          // Create an ArrayBuffer backed by the C++ vector
          auto ab = rt.global()
            .getPropertyAsFunction(rt, "ArrayBuffer")
            .callAsConstructor(rt, (int)buffer_.size())
            .asObject(rt);
          // Copy data into the ArrayBuffer
          auto rawBuf = ab.getArrayBuffer(rt);
          memcpy(rawBuf.data(rt), buffer_.data(), buffer_.size());
          return std::move(ab);
        });
    }
    return Value::undefined();
  }
};

9. Architect Considerations

Staff-Level Mental Model

JSI is not just a faster bridge — it is a fundamentally different communication model. The bridge was message-passing (like IPC between processes); JSI is shared-memory with direct function calls (like linking a library). This distinction matters because it changes what is architecturally possible: synchronous layout, shared state without serialization, and engine-agnostic abstractions. Every component of the new architecture — Fabric, TurboModules, Codegen, Reanimated, MMKV — is built on JSI. Understanding JSI is understanding the foundation of modern React Native.

Key Interview Points

  • JSI is engine-agnostic — it abstracts over Hermes, JSC, and V8 via the jsi::Runtime interface, enabling engine swaps without rewriting native modules.
  • HostObject and HostFunction are the two primitives for exposing C++ to JS. HostFunction for callable functions, HostObject for stateful objects with property access traps.
  • Synchronous calls eliminate the bridge's async-only limitation, enabling patterns like synchronous layout measurement and real-time gesture processing.
  • Memory management is the primary complexity — JSI types are move-only, GC-tracked, and must never be accessed from non-JS threads.
  • Error handling is stack-preserving via jsi::JSError, unlike the bridge where errors lost their context during JSON serialization.
  • Performance gains are 100-1000× for typical operations, fundamentally changing what is architecturally feasible in React Native apps.
