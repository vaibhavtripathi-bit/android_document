Reconciliation to Native

How React's reconciliation algorithm maps to native platform views — from virtual DOM diffing through shadow tree mutations to actual UIView/android.view.View operations. Understanding this pipeline is the foundation for all React Native performance work.

1. Pipeline Overview

React Native's rendering pipeline transforms React element trees into native platform views through a multi-stage process. Unlike React DOM which targets a single rendering surface (the browser DOM), React Native must bridge two fundamentally different execution environments: JavaScript and native platform code.

React Component Tree Bridge/JSI Native View Hierarchy ═══════════════════ ═══════════════════ ═══════════════════════ <View> Shadow Tree UIView (iOS) <Text>Hello</Text> ├─ ViewShadowNode ├─ RCTView <Image src={..}/> │ ├─ TextShadowNode │ ├─ RCTText </View> │ └─ ImageShadowNode │ └─ RCTImageView └─ (Yoga layout) └─ (Auto Layout) Reconciler Layout Engine Platform Renderer (JS Thread) (Background) (Main/UI Thread)
Architect Insight

The three-thread model (JS, Shadow/Background, Main/UI) is the fundamental architectural constraint of React Native. Every performance issue traces back to work being done on the wrong thread, or synchronization overhead between threads. Fabric (New Architecture) collapses some of these boundaries with synchronous access via JSI.

1.1 The Full Pipeline

Step 1: React Reconciliation (JS Thread) ├─ Component render() produces React Elements ├─ Fiber tree diffing (same algorithm as React DOM) └─ Produces list of mutations (create, update, delete, reorder) Step 2: Shadow Tree Update (Background Thread — Old Arch / JS Thread — Fabric) ├─ Shadow nodes created/updated to mirror fiber tree ├─ Yoga layout engine computes positions and sizes └─ Layout results stored on shadow nodes Step 3: Native View Operations (Main/UI Thread) ├─ createView(tag, className, props) ├─ updateView(tag, className, changedProps) ├─ manageChildren(tag, moveFrom, moveTo, addTags, addAtIndices, removeTags) └─ setChildren(tag, childTags)

2. Component Types in React Native

React Native distinguishes between two fundamental component categories that determine how reconciliation output maps to native operations.

2.1 Host Components

Host components are the leaf nodes that map directly to native platform views. They are the equivalent of DOM elements (div, span) in React DOM.

// Host components map to native views
import { View, Text, Image, ScrollView, TextInput } from 'react-native';

// View → UIView (iOS) / android.view.View (Android)
// Text → RCTText → UILabel (iOS) / TextView (Android)
// Image → RCTImageView → UIImageView (iOS) / ImageView (Android)
// ScrollView → RCTScrollView → UIScrollView (iOS) / ScrollView (Android)
// TextInput → RCTTextInput → UITextField (iOS) / EditText (Android)

// Each host component has a corresponding ViewManager on native side
// that knows how to create and update the native view

2.2 Composite Components

Composite components are JavaScript-only constructs that produce a tree of host components. They never appear in the native view hierarchy.

// Composite component — exists only in JS
const UserCard = ({ name, avatar }) => (
  <View style={styles.card}>
    <Image source={{ uri: avatar }} style={styles.avatar} />
    <Text style={styles.name}>{name}</Text>
  </View>
);

// After reconciliation, only host components remain:
// View (native)
//   ├─ Image (native)
//   └─ Text (native)
// UserCard itself has no native representation

Native viewYes — creates platform viewNo — JS only

3. Reconciliation Phase

React Native uses the exact same reconciliation algorithm (React Fiber) as React DOM. The reconciler is framework-agnostic — it operates on fibers and delegates actual view operations to the host config (renderer).

3.1 Fiber Tree Diffing

// The reconciler compares previous and next fiber trees
// using the same heuristics as React DOM:

// 1. Different element types → destroy old tree, create new
// 2. Same element type → update props, recurse into children
// 3. Keys used to match children across reorders

// Previous render:
<View>
  <Text key="a">Hello</Text>
  <Text key="b">World</Text>
</View>

// Next render (reordered):
<View>
  <Text key="b">World</Text>
  <Text key="a">Hello</Text>
</View>

// Reconciler output: MOVE operations (not destroy + create)
// This maps to manageChildren() on native side

3.2 Host Config: The Renderer Contract

The React reconciler communicates with React Native through a host config — a set of functions that the renderer must implement. This is the contract between React's core and the platform.

// Simplified React Native host config (renderer)
const hostConfig = {
  createInstance(type, props, rootContainer, hostContext, fiber) {
    // Called when reconciler needs a new native view
    // Sends createView command to native via UIManager
    const tag = ReactNativePrivateInterface.allocateTag();
    UIManager.createView(tag, type, rootContainer, props);
    return tag;
  },

  commitUpdate(instance, updatePayload, type, oldProps, newProps) {
    // Called when props change on an existing native view
    UIManager.updateView(instance, type, updatePayload);
  },

  appendChild(parentInstance, child) {
    UIManager.setChildren(parentInstance, [child]);
  },

  removeChild(parentInstance, child) {
    UIManager.manageChildren(
      parentInstance, [], [], [], [], [0]
    );
  },

  prepareUpdate(instance, type, oldProps, newProps) {
    // Diff props and return update payload
    // Returns null if no changes needed
    return diffProperties(oldProps, newProps);
  },
};
Key Insight

The reconciler never directly touches native views. It only calls host config functions. This abstraction is what allows the same React reconciler to power React DOM, React Native, React Three Fiber, Ink (terminal), and any custom renderer.

4. Shadow Tree Construction

The shadow tree is an intermediate representation between the React fiber tree and the native view hierarchy. Each shadow node mirrors a host component fiber and stores layout information computed by Yoga.

4.1 Shadow Node Lifecycle

Fiber Tree (JS) Shadow Tree (C++) Native Views (Platform) ═══════════════ ═══════════════════ ═══════════════════════ ViewFiber ──────────→ ViewShadowNode ──────────→ UIView / android.view.View │ │ │ ├─ TextFiber ──────→ TextShadowNode ──────────→ UILabel / TextView │ │ │ └─ ImageFiber ─────→ ImageShadowNode ─────────→ UIImageView / ImageView Yoga Layout Engine computes: x, y, width, height for each shadow node

4.2 Shadow Node Properties

// Each shadow node stores:
// 1. React tag (unique identifier)
// 2. View config (component type metadata)
// 3. Props (serialized from JS)
// 4. Yoga node (layout configuration)
// 5. Layout results (x, y, width, height)
// 6. Children (ordered list of child shadow nodes)

// In Fabric (New Architecture), shadow nodes are C++ objects
// accessible from both JS (via JSI) and native threads

// Shadow node creation in Fabric:
class ViewShadowNode : public ConcreteViewShadowNode<
  ViewShadowNode,
  ViewProps,
  ViewEventEmitter
> {
  // Inherits Yoga layout node
  // Stores ViewProps (backgroundColor, borderRadius, etc.)
  // Manages child shadow nodes
};

5. View Descriptors

View descriptors are the metadata objects that define how a React component type maps to its native counterpart. They contain the configuration needed to create and update native views.

// View descriptor contains:
const viewDescriptor = {
  uiViewClassName: 'RCTView',        // Native class name
  bubblingEventTypes: {              // Events that bubble
    topChange: {
      phasedRegistrationNames: {
        bubbled: 'onChange',
        captured: 'onChangeCapture',
      },
    },
  },
  directEventTypes: {                // Events that don't bubble
    topLayout: {
      registrationName: 'onLayout',
    },
  },
  validAttributes: {                 // Props that can be sent to native
    style: { process: processStyles },
    accessible: true,
    accessibilityLabel: true,
    hitSlop: true,
    nativeID: true,
    testID: true,
    // ... many more
  },
};
Common Interview Pitfall

Candidates often confuse view descriptors with ViewManagers. View descriptors are JS-side metadata used during reconciliation to validate props and route events. ViewManagers are native-side classes that actually create and manage platform views. They work together but live on different sides of the bridge.

6. Props Diffing and Native Updates

When a component re-renders with new props, React Native must determine which props actually changed and serialize only the diff to native. This is a critical performance path.

6.1 The diffProperties Algorithm

// Simplified props diffing logic
function diffProperties(prevProps, nextProps, validAttributes) {
  let updatePayload = null;

  // Check for removed props
  for (const propKey in prevProps) {
    if (nextProps[propKey] !== undefined) continue;
    if (!validAttributes[propKey]) continue;
    if (!updatePayload) updatePayload = {};
    updatePayload[propKey] = null; // Signal removal
  }

  // Check for added/changed props
  for (const propKey in nextProps) {
    const nextValue = nextProps[propKey];
    const prevValue = prevProps[propKey];

    if (nextValue === prevValue) continue;
    if (!validAttributes[propKey]) continue;

    // Style props get special nested diffing
    if (propKey === 'style') {
      const styleDiff = diffNestedStyles(prevValue, nextValue);
      if (styleDiff) {
        if (!updatePayload) updatePayload = {};
        updatePayload.style = styleDiff;
      }
    } else {
      if (!updatePayload) updatePayload = {};
      updatePayload[propKey] = nextValue;
    }
  }

  return updatePayload; // null means no changes
}

6.2 Style Prop Optimization

// Style objects are deeply diffed, not replaced wholesale
// Previous: { backgroundColor: 'red', padding: 10, margin: 5 }
// Next:     { backgroundColor: 'blue', padding: 10, margin: 5 }
// Diff:     { backgroundColor: 'blue' }  ← only changed prop sent

// StyleSheet.create() returns numeric IDs, not objects
const styles = StyleSheet.create({
  container: { flex: 1, backgroundColor: 'white' },
});
// styles.container is a number (e.g., 42)
// The actual style object is stored in a registry
// and resolved during props diffing via StyleSheet.flatten()

7. Batch Updates to Native

React Native batches multiple view operations into a single native transaction to minimize bridge crossings and ensure atomic UI updates.

7.1 Operation Batching (Old Architecture)

JS Thread Bridge UI Thread ═════════ ═══════════════ ═══════════ setState() ──→ Reconcile setState() ──→ Reconcile Batch queue: setState() ──→ Reconcile ┌──────────────────────┐ │ createView(1, View) │ End of JS │ createView(2, Text) │──→ Execute all execution frame │ updateView(3, props) │ atomically │ setChildren(1, [2,3]) │ on UI thread └──────────────────────┘

7.2 Fabric: Synchronous Mounting

// In Fabric, the mounting phase can be synchronous
// for specific use cases (e.g., text input)

// Old Architecture: async batch
// 1. JS reconciles → queues operations
// 2. Operations serialized to JSON
// 3. JSON sent over bridge
// 4. Native deserializes and executes
// Latency: ~3-5 frames for complex updates

// Fabric: synchronous when needed
// 1. JS reconciles → creates C++ shadow tree diff
// 2. Diff applied directly to native via JSI
// 3. No serialization overhead
// Latency: ~1 frame for synchronous path

// Fabric mutation instructions:
enum class ShadowViewMutation::Type {
  Create,   // Create new native view
  Delete,   // Remove native view
  Insert,   // Insert child at index
  Remove,   // Remove child at index
  Update,   // Update props on existing view
};
Fabric Batching

Fabric still batches operations by default for performance. The synchronous path is opt-in for specific scenarios like TextInput where immediate feedback is critical. The key improvement is eliminating JSON serialization — mutations are C++ structs passed via shared memory.

8. Event Handling: Native → JS

Events flow in the opposite direction from rendering: from native platform views back to JavaScript. This reverse path has its own performance characteristics and batching behavior.

8.1 Event Flow

Native Touch Event JS Event Handler ══════════════════ ═══════════════════ UIView touchesBegan: ──→ RCTEventDispatcher ├─ coalescable? │ (scroll, touch move) ├─ serialize to JSON └─ dispatch to JS │ ┌───────────┘ ▼ EventEmitter ├─ find fiber by reactTag ├─ traverse fiber tree (bubble/capture) └─ call onPress / onScroll / etc.

8.2 Event Types and Coalescing

// React Native has three event categories:

// 1. Discrete events — delivered individually, never coalesced
//    Examples: onPress, onLongPress, onChange
//    These must be processed in order

// 2. Coalescing events — merged when queued
//    Examples: onScroll, onTouchMove
//    Only the latest event is kept if JS is behind

// 3. Continuous events — high-frequency, special handling
//    Examples: onGestureEvent (Reanimated)
//    May bypass JS thread entirely (worklets)

// Event coalescing prevents JS thread flooding:
// If user scrolls rapidly and JS is busy,
// intermediate scroll positions are dropped.
// Only the latest contentOffset is delivered.

class ScrollEvent {
  canCoalesce() { return true; }
  coalesce(newer) {
    return newer; // Latest scroll position wins
  }
}

8.3 Fabric Event System

// Fabric uses C++ EventEmitter instead of bridge-based dispatch
// Events are C++ objects that can be processed synchronously

// Priority-based event dispatch in Fabric:
enum class EventPriority {
  SynchronousUnbatched, // Immediate — text input
  SynchronousBatched,   // Sync but batched — discrete touches
  AsynchronousUnbatched,// Async — scroll events
  AsynchronousBatched,  // Async batched — layout events
};

// This priority system allows Fabric to handle
// text input synchronously (no frame delay)
// while still batching scroll events for efficiency

10. Architect Considerations

Staff-Level Mental Model

The reconciliation-to-native pipeline is a producer-consumer system with the JS thread as producer and the UI thread as consumer. Performance optimization is fundamentally about: (1) reducing what the producer generates (fewer re-renders, memoization), (2) reducing serialization cost (props diffing, view flattening), and (3) ensuring the consumer can keep up (60fps budget = 16.6ms per frame on UI thread). Fabric's key innovation is making this pipeline partially synchronous, eliminating the async gap that caused visual glitches in the old architecture.

10.1 Performance Implications

// Anti-pattern: inline style objects cause props diff on every render
const BadComponent = ({ color }) => (
  // New object every render → always triggers native update
  <View style={{ flex: 1, backgroundColor: color }} />
);

// Better: stable references for static styles
const GoodComponent = ({ color }) => (
  <View style={[styles.container, { backgroundColor: color }]} />
);

const styles = StyleSheet.create({
  container: { flex: 1 }, // Stable reference — numeric ID
});

// Best: useMemo for computed styles
const BestComponent = ({ color }) => {
  const dynamicStyle = useMemo(
    () => ({ backgroundColor: color }),
    [color]
  );
  return <View style={[styles.container, dynamicStyle]} />;
};
