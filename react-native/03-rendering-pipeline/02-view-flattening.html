View Flattening

View flattening is a critical optimization that removes unnecessary native views from the hierarchy, reducing memory usage, layout computation, and draw calls. Understanding when and why flattening applies — and when it breaks — is essential for Staff-level performance work.

1. The Problem: Deep View Hierarchies

React component composition naturally produces deeply nested view trees. Each <View> in JSX creates a native platform view, and native view hierarchies have real performance costs.

React Component Tree Without Flattening (Native) ════════════════════ ═══════════════════════════ <View> ← outer container UIView (1) <View> ← padding wrapper UIView (2) ← unnecessary! <View> ← flex row UIView (3) ← unnecessary! <Text>Hi</Text> UILabel (4) </View> </View> </View> With Flattening (Native): UIView (1) UILabel (4) ← Views 2 and 3 removed, layout absorbed by parent

Architect Insight

A typical React Native screen can have 200-500 React components but should ideally produce only 50-150 native views after flattening. The ratio of React components to native views is a key health metric. Tools like Flipper's Layout Inspector show the actual native hierarchy post-flattening.

2. Layout-Only Views

A "layout-only" view is one that contributes only layout (flexbox positioning) to its children but has no visual properties of its own. These views are candidates for removal because their layout contribution can be absorbed by their parent.

2.1 Identifying Layout-Only Views

// Layout-only: only has layout/flex properties
<View style={{ flex: 1, flexDirection: 'row', padding: 10 }}>
  <Text>Hello</Text>
</View>
// ↑ This View only does layout — no backgroundColor, border, shadow, etc.
// It CAN be flattened: its layout is merged into the parent

// NOT layout-only: has visual properties
<View style={{ flex: 1, backgroundColor: 'red' }}>
  <Text>Hello</Text>
</View>
// ↑ This View draws a red background — CANNOT be flattened

// NOT layout-only: has event handlers
<View style={{ flex: 1 }} onLayout={handleLayout}>
  <Text>Hello</Text>
</View>
// ↑ Needs native view to receive onLayout event

2.2 Properties That Prevent Flattening

// Any of these properties on a View prevent flattening:

// Visual properties
backgroundColor      // Draws color
borderWidth / borderColor / borderRadius  // Draws borders
opacity              // Requires compositing layer
shadowColor / shadowOffset / shadowRadius // iOS shadows
elevation            // Android shadows

// Overflow
overflow: 'hidden'   // Requires clipping bounds

// Transforms
transform            // Requires native layer

// Event handlers
onLayout             // Needs native view for measurement
onPress / onLongPress // Needs native touch target
accessible / accessibilityLabel // Accessibility tree node

// Other
nativeID             // Explicit native reference
testID               // Testing reference
collapsable={false}  // Explicit opt-out
pointerEvents        // Touch handling configuration

3. The Flattening Algorithm

The flattening algorithm runs during the shadow tree construction phase, before native views are created. It identifies layout-only nodes and merges their layout contributions into their parents.

3.1 Algorithm Steps

Input Shadow Tree: After Flattening: ═══════════════════ ═══════════════════ ViewShadowNode (A) ViewShadowNode (A) │ flex: 1 │ flex: 1 │ │ ├─ ViewShadowNode (B) ├─ TextShadowNode (D) │ │ flexDirection: 'row' │ [layout adjusted for B+C] │ │ padding: 10 │ │ │ ← LAYOUT ONLY └─ ImageShadowNode (E) │ │ [layout adjusted for B+C] │ ├─ TextShadowNode (D) │ │ │ └─ ViewShadowNode (C) │ │ margin: 5 │ │ ← LAYOUT ONLY │ │ │ └─ ImageShadowNode (E) Nodes B and C are removed. Their layout (padding, margin, flex) is absorbed into the positioning of D and E relative to A.

3.2 Implementation Detail

// Simplified flattening check (Old Architecture)
function isLayoutOnly(viewConfig, props) {
  // Check if component type supports flattening
  if (viewConfig.uiViewClassName !== 'RCTView') {
    return false; // Only plain Views can be flattened
  }

  // Check for visual properties
  if (props.backgroundColor) return false;
  if (props.borderWidth || props.borderColor) return false;
  if (props.opacity != null && props.opacity !== 1) return false;
  if (props.shadowColor) return false;
  if (props.elevation) return false;
  if (props.overflow === 'hidden') return false;
  if (props.transform) return false;

  // Check for event handlers
  if (props.onLayout) return false;
  if (props.accessible) return false;
  if (props.collapsable === false) return false;
  if (props.nativeID) return false;

  return true; // Safe to flatten
}

4. When Flattening Applies

4.1 Common Flattening Scenarios

// Scenario 1: Wrapper views for layout
const Row = ({ children }) => (
  <View style={{ flexDirection: 'row' }}>
    {children}
  </View>
);
// ↑ This wrapper View is layout-only → will be flattened

// Scenario 2: Spacing/padding wrappers
const Padded = ({ children }) => (
  <View style={{ padding: 16 }}>
    {children}
  </View>
);
// ↑ Layout-only → will be flattened

// Scenario 3: Conditional rendering wrappers
const ConditionalWrapper = ({ wrap, children }) => (
  wrap ? <View style={{ flex: 1 }}>{children}</View> : children
);
// ↑ When wrap=true, the View is layout-only → flattened

4.2 Flattening with StyleSheet

// StyleSheet.create styles are analyzed at creation time
const styles = StyleSheet.create({
  // This style only has layout props → View using it can be flattened
  layoutOnly: {
    flex: 1,
    flexDirection: 'row',
    justifyContent: 'center',
    alignItems: 'center',
    padding: 16,
    margin: 8,
  },

  // This style has visual props → View using it CANNOT be flattened
  visual: {
    flex: 1,
    backgroundColor: '#f0f0f0',
    borderRadius: 8,
  },
});

5. When Flattening Breaks

Understanding when flattening cannot apply is crucial for debugging layout issues and unexpected native view counts.

5.1 Border and Shadow Edge Cases

// Even borderWidth: 0 with borderColor prevents flattening!
<View style={{ borderWidth: 0, borderColor: 'transparent' }}>
  <Text>Not flattened</Text>
</View>
// ↑ borderColor is set → native view created

// borderRadius alone prevents flattening
<View style={{ borderRadius: 8 }}>
  <Text>Not flattened</Text>
</View>
// ↑ borderRadius implies visual rendering

// overflow: 'hidden' prevents flattening
<View style={{ overflow: 'hidden' }}>
  <Text>Not flattened</Text>
</View>
// ↑ Clipping requires native view bounds

5.2 Accessibility Prevents Flattening

// Any accessibility prop prevents flattening
<View accessible={true} accessibilityLabel="container">
  <Text>Not flattened</Text>
</View>
// ↑ Must exist in native accessibility tree

// accessibilityRole also prevents flattening
<View accessibilityRole="button">
  <Text>Not flattened</Text>
</View>
Accessibility vs Performance Trade-off

Adding accessibility props prevents view flattening, which increases the native view count. This is an intentional trade-off — accessibility is more important than the marginal performance gain from flattening. However, be strategic: apply accessibility props at the right level of the hierarchy, not on every intermediate wrapper.

5.3 Dynamic Styles That Break Flattening

// Inline styles are checked at render time
const DynamicView = ({ highlighted }) => (
  <View style={{
    flex: 1,
    // When highlighted=true, backgroundColor prevents flattening
    // When highlighted=false, View can be flattened
    backgroundColor: highlighted ? 'yellow' : undefined,
  }}>
    <Text>Content</Text>
  </View>
);
// ↑ Flattening status changes between renders!
// This causes native view creation/deletion on state change
// which is more expensive than a simple prop update

6. The collapsable Prop

The collapsable prop provides explicit control over view flattening. It's primarily used when you need to prevent flattening for measurement or native reference purposes.

// Prevent flattening: collapsable={false}
<View collapsable={false} style={{ flex: 1 }}>
  <Text>This View will always exist in native hierarchy</Text>
</View>

// Common use case: findNodeHandle / measure
const viewRef = useRef(null);

// Without collapsable={false}, this View might be flattened
// and findNodeHandle would return null or the wrong node
<View
  ref={viewRef}
  collapsable={false}
  style={{ flex: 1 }}
  onLayout={(e) => {
    // onLayout already prevents flattening,
    // but collapsable={false} makes intent explicit
    console.log(e.nativeEvent.layout);
  }}
/>

// Note: collapsable is Android-only in the old architecture
// iOS never flattened views in Paper (old arch)
// Fabric applies flattening on both platforms
Platform Difference

In the old architecture (Paper), view flattening was Android-only. iOS did not flatten views. In Fabric (New Architecture), view flattening applies to both platforms, making collapsable={false} relevant on iOS as well.

7. Fabric's Improved Flattening

Fabric introduces a more sophisticated flattening mechanism that operates on the shadow tree diff rather than the initial tree construction.

7.1 Diff-Based Flattening

Old Architecture (Paper): Fabric: ═════════════════════════ ═══════ Shadow tree built Shadow tree built ↓ ↓ Flattening applied once Flattening applied to DIFF during tree construction between old and new trees ↓ ↓ Native views created Mutation instructions generated from flattened tree with flattening considered ↓ Only changed views created/removed

7.2 Fabric Flattening Implementation

// Fabric uses a "flattening" flag on ShadowNode
// that determines if the node should be mounted as a native view

// In C++ (simplified):
bool ViewShadowNode::isLayoutOnly() const {
  const auto &props = getConcreteProps();

  return
    props.backgroundColor == SharedColor() &&
    props.borderWidths == CascadedBorderWidths() &&
    props.borderColors == CascadedBorderColors() &&
    props.borderRadii == CascadedBorderRadii() &&
    props.shadowColor == SharedColor() &&
    props.transform == Transform::Identity() &&
    props.opacity == 1.0 &&
    props.overflow == Overflow::Visible &&
    !props.onLayout &&
    !props.accessible &&
    props.collapsable != false &&
    props.nativeId.empty();
}

// Fabric tree diff algorithm considers flattening:
// If a node transitions from layout-only to non-layout-only,
// a Create mutation is generated.
// If it transitions from non-layout-only to layout-only,
// a Delete mutation is generated.

8. Debugging Flattened Hierarchies

8.1 Tools and Techniques

// 1. Flipper Layout Inspector
// Shows actual native view hierarchy (post-flattening)
// Compare React DevTools tree (pre-flattening) with Flipper (post)

// 2. Xcode View Debugger (iOS)
// Debug → View Debugging → Capture View Hierarchy
// Shows 3D exploded view of native hierarchy

// 3. Android Layout Inspector
// Android Studio → Tools → Layout Inspector
// Shows real-time native view tree

// 4. Programmatic check
import { findNodeHandle, UIManager } from 'react-native';

const DebugView = () => {
  const ref = useRef(null);

  useEffect(() => {
    const tag = findNodeHandle(ref.current);
    if (tag === null) {
      console.log('View was flattened — no native node');
    } else {
      console.log('Native view exists, tag:', tag);
    }
  }, []);

  return <View ref={ref} style={{ flex: 1 }} />;
};

8.2 Common Debugging Pitfalls

// Pitfall 1: ref on flattened view returns null
const ref = useRef(null);
<View ref={ref} style={{ flex: 1 }}>
  <Text>Content</Text>
</View>
// ref.current.measure() may fail if View is flattened
// Fix: add collapsable={false} or onLayout

// Pitfall 2: nativeID on flattened view
<View nativeID="my-view" style={{ flex: 1 }}>
  <Text>Content</Text>
</View>
// nativeID prevents flattening, so this always works
// But adding nativeID "just in case" wastes a native view

9.2 Optimization Strategies

// Strategy 1: Prefer layout-only wrapper components
const Row = ({ children }) => (
  <View style={{ flexDirection: 'row' }}>{children}</View>
);
// ↑ Will be flattened — zero native cost

// Strategy 2: Avoid unnecessary visual props
// Bad: backgroundColor: 'transparent' prevents flattening!
<View style={{ flex: 1, backgroundColor: 'transparent' }}>
  <Text>Content</Text>
</View>

// Good: omit backgroundColor entirely
<View style={{ flex: 1 }}>
  <Text>Content</Text>
</View>

// Strategy 3: Consolidate visual props at fewer levels
// Bad: visual props at every level
<View style={{ backgroundColor: 'white' }}>
  <View style={{ borderWidth: 1 }}>
    <View style={{ borderRadius: 8 }}>
      <Text>Content</Text>
    </View>
  </View>
</View>
// ↑ 3 native views — none can be flattened

// Good: combine visual props on one view
<View style={{
  backgroundColor: 'white',
  borderWidth: 1,
  borderRadius: 8,
}}>
  <Text>Content</Text>
</View>
// ↑ 1 native view

10. Architect Considerations

Staff-Level Mental Model

View flattening is an automatic optimization that rewards good component design. The key insight: React component boundaries should be driven by logic, not by visual boundaries. Create as many wrapper components as you need for code organization — the flattening algorithm will remove the unnecessary native views. The exception is when you need native-level features (events, accessibility, clipping) — then each View must exist natively. A Staff engineer's job is to ensure the team understands this distinction and designs component hierarchies accordingly.

10.1 Design System Implications

// Design system components should be flattening-aware

// Good: Spacer component that flattens away
const Spacer = ({ size }) => (
  <View style={{ height: size, width: size }} />
);
// ↑ Layout-only → flattened → zero native cost

// Good: Stack component that flattens away
const VStack = ({ spacing, children }) => (
  <View style={{ gap: spacing }}>
    {children}
  </View>
);
// ↑ Layout-only → flattened

// Aware: Card component that cannot flatten
const Card = ({ children }) => (
  <View style={styles.card}>
    {children}
  </View>
);
const styles = StyleSheet.create({
  card: {
    backgroundColor: 'white',
    borderRadius: 12,
    shadowColor: '#000',
    shadowOffset: { width: 0, height: 2 },
    shadowOpacity: 0.1,
    elevation: 3,
    padding: 16,
  },
});
// ↑ Visual props → CANNOT flatten → 1 native view per Card
// This is expected and correct
