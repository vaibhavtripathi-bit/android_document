Native View Management

How React Native creates, updates, and manages native platform views through UIManager, ViewManagers, and the shadow view system. Understanding this layer is essential for building custom native components and debugging rendering issues.

1. UIManager

UIManager is the central coordinator for all native view operations in React Native. It receives commands from the reconciler and dispatches them to the appropriate ViewManagers on the native side.

UIManager — Central View Coordinator JS Thread UIManager UI Thread ═════════ ═════════ ═════════ Reconciler Command Queue ViewManagers │ │ │ ├─ createView(tag, type, props)│ │ ├─ updateView(tag, type, props)│──→ Batch & dispatch ──→ │ ├─ manageChildren(...) │ on UI thread │ ├─ setChildren(tag, children) │ │ ├─ measure(tag, callback) │ │ └─ dispatchCommand(tag, cmd) │ │ Old Architecture: UIManager is a native module (bridge-based) Fabric: UIManager replaced by direct C++ mounting via JSI

1.1 UIManager API

import { UIManager } from 'react-native';

// Core operations (called by reconciler, not directly by app code):

// Create a new native view
UIManager.createView(
  tag,          // Unique numeric identifier
  viewName,     // "RCTView", "RCTText", etc.
  rootTag,      // Root container tag
  props         // Initial properties
);

// Update properties on existing view
UIManager.updateView(
  tag,          // View to update
  viewName,     // View type
  props         // Changed properties only (diff)
);

// Manage children (add, remove, move)
UIManager.manageChildren(
  parentTag,
  moveFromIndices,   // Indices to move from
  moveToIndices,     // Indices to move to
  addChildTags,      // New child tags to add
  addAtIndices,      // Indices to add at
  removeAtIndices    // Indices to remove
);

// Set children (replace all children)
UIManager.setChildren(
  parentTag,
  childTags     // Ordered array of child tags
);

// Dispatch a command to a native view
UIManager.dispatchViewManagerCommand(
  findNodeHandle(ref),
  UIManager.getViewManagerConfig('ScrollView').Commands.scrollTo,
  [0, 100, true]  // arguments: x, y, animated
);

2. ViewManagers

ViewManagers are native classes that know how to create and configure specific native view types. Each React Native host component has a corresponding ViewManager.

2.1 iOS: RCTViewManager

// iOS ViewManager (Objective-C)
// Each ViewManager is registered with the bridge

@interface RCTViewManager : NSObject <RCTBridgeModule>
// Creates the native view instance
- (UIView *)view;

// Exports properties that JS can set
// RCT_EXPORT_VIEW_PROPERTY(propName, propType)

// Exports custom commands
// RCT_EXPORT_METHOD(commandName:(nonnull NSNumber *)reactTag ...)
@end

// Example: RCTView's ViewManager
@implementation RCTViewManager
RCT_EXPORT_MODULE()

- (UIView *)view {
  return [[RCTView alloc] init];
}

RCT_EXPORT_VIEW_PROPERTY(backgroundColor, UIColor)
RCT_EXPORT_VIEW_PROPERTY(borderRadius, CGFloat)
RCT_EXPORT_VIEW_PROPERTY(opacity, CGFloat)
RCT_EXPORT_VIEW_PROPERTY(overflow, RCTOverflow)
RCT_CUSTOM_VIEW_PROPERTY(borderWidth, CGFloat, RCTView) {
  view.layer.borderWidth = json ? [RCTConvert CGFloat:json] : 0;
}
@end

2.2 Android: ViewManager

// Android ViewManager (Java/Kotlin)
public class ReactViewManager
    extends ViewGroupManager<ReactViewGroup> {

  @Override
  public String getName() {
    return "RCTView";
  }

  @Override
  protected ReactViewGroup createViewInstance(ThemedReactContext context) {
    return new ReactViewGroup(context);
  }

  @ReactProp(name = "backgroundColor", customType = "Color")
  public void setBackgroundColor(ReactViewGroup view, int color) {
    view.setBackgroundColor(color);
  }

  @ReactProp(name = "borderRadius", defaultFloat = 0f)
  public void setBorderRadius(ReactViewGroup view, float radius) {
    view.setBorderRadius(radius);
  }

  @Override
  public void receiveCommand(ReactViewGroup view, String cmd, ReadableArray args) {
    // Handle commands dispatched from JS
  }
}
ViewManager Registration

ViewManagers are registered during app startup. On iOS, they're discovered via the RCT_EXPORT_MODULE() macro. On Android, they're listed in the createViewManagers() method of the ReactPackage. Fabric uses a different registration mechanism via ComponentDescriptorProvider in C++.

3. Custom Native Views

// Creating a custom native view (iOS - Swift)

// 1. Create the native view
class GradientView: UIView {
  var colors: [UIColor] = [] {
    didSet { updateGradient() }
  }
  var startPoint: CGPoint = .zero
  var endPoint: CGPoint = CGPoint(x: 1, y: 1)

  private lazy var gradientLayer = CAGradientLayer()

  private func updateGradient() {
    gradientLayer.colors = colors.map { $0.cgColor }
    gradientLayer.startPoint = startPoint
    gradientLayer.endPoint = endPoint
  }
}

// 2. Create the ViewManager
@objc(GradientViewManager)
class GradientViewManager: RCTViewManager {
  override func view() -> UIView! {
    return GradientView()
  }

  override static func requiresMainQueueSetup() -> Bool {
    return true
  }
}

// 3. JS wrapper component
import { requireNativeComponent } from 'react-native';

const NativeGradient = requireNativeComponent('GradientView');

const GradientView = ({ colors, start, end, style }) => (
  <NativeGradient
    colors={colors}
    startPoint={start}
    endPoint={end}
    style={style}
  />
);

4. View Property Updates

4.1 Update Flow

Prop Update Flow: JS: setState({ color: 'blue' }) │ ↓ Reconciler: diffProperties(oldProps, newProps) │ Result: { backgroundColor: 'blue' } ↓ UIManager.updateView(tag, 'RCTView', { backgroundColor: 'blue' }) │ ↓ (dispatched to UI thread) │ ViewManager.setBackgroundColor(view, Color.BLUE) │ ↓ Native view updated on screen

4.2 Prop Types and Serialization

// Props are serialized differently based on type:

// Primitive types — sent directly
// string, number, boolean → JSON primitive

// Colors — processed through processColor()
// 'red' → 0xFFFF0000 (packed ARGB integer)
// '#FF0000' → 0xFFFF0000
// 'rgba(255,0,0,0.5)' → 0x80FF0000

// Styles — flattened and processed
// StyleSheet IDs resolved to objects
// Array styles merged (later overrides earlier)
// Platform-specific transforms applied

// Images — resolved to source descriptor
// require('./img.png') → { uri: 'asset://img.png', width: 100, height: 100 }
// { uri: 'https://...' } → passed as-is

// Events — registered as callbacks
// onPress → native touch handler → bridge callback
// onLayout → native layout event → bridge callback

5. Direct Manipulation (setNativeProps)

setNativeProps bypasses the React reconciliation cycle to update native view properties directly. This is useful for performance-critical animations but breaks React's declarative model.

// setNativeProps updates native view without re-render
const viewRef = useRef(null);

// Instead of setState (triggers reconciliation):
// setBackgroundColor('red') → re-render → diff → updateView

// Direct manipulation (bypasses React):
viewRef.current.setNativeProps({
  style: { backgroundColor: 'red' },
});
// Goes directly to UIManager.updateView
// No reconciliation, no virtual DOM diff

// Common use case: gesture-driven animations
const onPanGesture = (event) => {
  viewRef.current.setNativeProps({
    style: {
      transform: [
        { translateX: event.translationX },
        { translateY: event.translationY },
      ],
    },
  });
};

// WARNING: setNativeProps creates a divergence between
// React's virtual tree and the actual native state.
// The next React re-render may overwrite setNativeProps changes.
// Prefer Animated API or Reanimated for animations.
setNativeProps is Deprecated in Fabric

In the New Architecture (Fabric), setNativeProps is not supported. Use Animated with useNativeDriver: true or Reanimated's shared values for direct native manipulation. Fabric's synchronous rendering makes the performance gap between setState and setNativeProps much smaller.

6. Native View Lifecycle

Native View Lifecycle: 1. CREATE ├─ ViewManager.createViewInstance() called ├─ Native view allocated (UIView/android.view.View) ├─ Initial props applied via setter methods └─ View NOT yet in hierarchy 2. MOUNT ├─ View added to parent via addSubview/addView ├─ Layout computed by Yoga ├─ Frame/bounds set on native view └─ onLayout event fires (if handler registered) 3. UPDATE ├─ Changed props dispatched to ViewManager setters ├─ Layout recomputed if layout-affecting props changed └─ View redrawn if visual props changed 4. UNMOUNT ├─ View removed from parent (removeFromSuperview/removeView) ├─ Cleanup: cancel pending operations, remove observers └─ View deallocated (ARC on iOS, GC on Android) 
// iOS: View lifecycle hooks
class CustomView: RCTView {
  override func didMoveToWindow() {
    super.didMoveToWindow()
    if window != nil {
      // View is now visible — start animations, observers
    } else {
      // View removed from window — cleanup
    }
  }

  override func layoutSubviews() {
    super.layoutSubviews()
    // Called after Yoga layout applied
    // bounds.size contains the computed size
  }
}

// Android: View lifecycle hooks
class CustomView(context: Context) : ReactViewGroup(context) {
  override fun onAttachedToWindow() {
    super.onAttachedToWindow()
    // View attached — start operations
  }

  override fun onDetachedFromWindow() {
    super.onDetachedFromWindow()
    // View detached — cleanup
  }

  override fun onLayout(changed: Boolean, l: Int, t: Int, r: Int, b: Int) {
    // Called after layout pass
  }
}

8. Measuring Native Views from JS

// measure() — get position relative to root view
const viewRef = useRef(null);

viewRef.current.measure((x, y, width, height, pageX, pageY) => {
  // x, y: position relative to parent
  // width, height: view dimensions
  // pageX, pageY: position relative to root view (screen)
  console.log({ x, y, width, height, pageX, pageY });
});

// measureInWindow() — get position relative to device screen
viewRef.current.measureInWindow((x, y, width, height) => {
  // x, y: position relative to device window
  // Accounts for status bar, navigation bar offset
  console.log({ x, y, width, height });
});

// measureLayout() — get position relative to another view
viewRef.current.measureLayout(
  findNodeHandle(ancestorRef.current),
  (x, y, width, height) => {
    // x, y: position relative to ancestor view
    console.log({ x, y, width, height });
  },
  () => { console.log('Measurement failed'); }
);

// Important: All measure calls are ASYNC in old architecture
// They cross the bridge to read native view frames
// In Fabric: measureInWindow can be SYNCHRONOUS via JSI

9. Layout Events

// onLayout fires after native layout is computed
<View
  onLayout={(event) => {
    const { x, y, width, height } = event.nativeEvent.layout;
    console.log(`Layout: ${width}x${height} at (${x},${y})`);
  }}
>
  <Text>Content</Text>
</View>

// onLayout fires:
// 1. On initial mount
// 2. When view dimensions change
// 3. When view position changes (relative to parent)
// Does NOT fire when view scrolls (position in scroll view changes)

// Common pattern: measure container for responsive layout
const ResponsiveGrid = () => {
  const [containerWidth, setContainerWidth] = useState(0);
  const numColumns = containerWidth > 600 ? 3 : 2;

  return (
    <View onLayout={(e) => setContainerWidth(e.nativeEvent.layout.width)}>
      <FlatList
        data={items}
        numColumns={numColumns}
        key={numColumns} // Force remount when columns change
        renderItem={renderItem}
      />
    </View>
  );
};

// Performance note: onLayout prevents view flattening
// The view must exist natively to report layout events
// Use sparingly — only on views that need measurement

10. Architect Considerations

Staff-Level Mental Model

Native view management is the bridge between React's declarative world and the imperative native platform. A Staff engineer must understand: (1) The UIManager command queue and its batching behavior — this determines update latency. (2) ViewManager registration and prop routing — this is the extension point for custom native components. (3) The shadow view / native view duality — shadow views compute layout off the main thread, native views render on the main thread. (4) Measurement APIs and their async nature — this affects any layout-dependent logic. In Fabric, many of these constraints are relaxed (synchronous measurement, C++ shadow nodes), but the mental model of "compute layout → apply to native" remains.

10.1 When to Build Custom Native Views

// Build a custom native view when:
// 1. No existing RN component covers the functionality
//    (e.g., camera preview, map, video player)
// 2. Performance requires native-level optimization
//    (e.g., custom drawing, GPU shaders)
// 3. Platform-specific UI that can't be composed from primitives
//    (e.g., native date picker, native share sheet)

// Do NOT build custom native views when:
// 1. The UI can be composed from existing RN components
// 2. The performance issue is in JS, not native rendering
// 3. A community library already exists and is maintained

// Fabric component (New Architecture):
// Uses codegen from JS spec → generates C++ and native boilerplate
// Type-safe props, commands, and events
// Significantly less manual bridging code
