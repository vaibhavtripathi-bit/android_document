Reconciliation & Native Mapping

Q1: How does React reconciliation map to native view operations in React Native?

React's Fiber reconciler diffs the previous and next component trees to produce a list of mutations (create, update, delete, reorder). These mutations are translated to UIManager commands: createView, updateView, manageChildren, and setChildren. In the old architecture, these commands are batched and sent over the async bridge as JSON. In Fabric, they become C++ mutation instructions (Create, Delete, Insert, Remove, Update) applied via JSI without serialization. The reconciler itself is platform-agnostic — it delegates to the host config (renderer), which is the React Native-specific layer.

Q2: What is the difference between a host component and a composite component in React Native?

Host components (View, Text, Image, ScrollView) map directly to native platform views — each has a corresponding ViewManager that creates and manages the native view. Composite components are JavaScript-only constructs that render trees of host components. They never appear in the native view hierarchy. After reconciliation, only host components remain in the output that gets sent to native. This distinction matters because composite components have zero native cost — you can create as many wrapper components as needed for code organization without impacting native performance.

Q3: Explain the shadow tree and its role in the rendering pipeline.

The shadow tree is an intermediate representation between the React fiber tree and the native view hierarchy. Each shadow node mirrors a host component fiber and stores a Yoga layout node. The shadow tree's purpose is to compute layout (positions and sizes) off the main thread using Yoga's flexbox engine. In the old architecture, shadow nodes are Java/ObjC objects on a background thread. In Fabric, they're C++ objects shared across threads via JSI. After Yoga computes layout, the results (x, y, width, height) are used to set frames on the actual native views on the UI thread.

Q4: What are view descriptors and how do they differ from ViewManagers?

View descriptors are JS-side metadata objects that define how a React component type maps to its native counterpart. They contain the native class name (e.g., 'RCTView'), valid attributes (props that can be sent to native), and event type registrations (bubbling vs direct events). ViewManagers are native-side classes that actually create and configure platform views. View descriptors are used during reconciliation for prop validation and event routing, while ViewManagers are invoked on the UI thread to create/update actual native views.

Q5: How does props diffing work in React Native? Why is it important?

Props diffing compares previous and next props to produce a minimal update payload containing only changed properties. The diffProperties function iterates both prop objects, checking against validAttributes to filter out JS-only props. Style objects get special nested diffing — only changed style properties are included. This is critical for performance because every prop in the update payload must be serialized (old arch) or copied (Fabric) and processed on the native side. Sending unchanged props wastes bridge bandwidth and native processing time.

Q6: How does React Native batch updates to native views?

In the old architecture, all UIManager commands generated during a JS execution frame are queued and sent as a single batch over the bridge. This ensures atomic UI updates — the user never sees a partially updated screen. The batch is dispatched to the UI thread at the end of the JS frame. In Fabric, mutations are still batched by default, but specific operations (like TextInput updates) can use the synchronous path for immediate feedback. Fabric batches are C++ mutation instruction lists, avoiding JSON serialization overhead.

Q7: Describe the event flow from native touch to JS handler in React Native.

A native touch event (e.g., UIView touchesBegan on iOS) is captured by the RCTEventDispatcher. The event is classified as discrete (onPress), coalescing (onScroll), or continuous (gesture). Coalescing events may be merged if the JS thread is behind. The event is serialized and dispatched to JS, where the EventEmitter finds the corresponding fiber by reactTag, traverses the fiber tree for bubble/capture phase, and invokes the registered handler (onPress, onScroll, etc.). In Fabric, events are C++ objects with priority levels, and high-priority events (text input) can be processed synchronously.

Q8: What is event coalescing and why does React Native need it?

Event coalescing merges multiple events of the same type when the JS thread is busy. For example, during a fast scroll, if the JS thread is processing a previous frame, intermediate scroll position events are dropped and only the latest contentOffset is delivered. This prevents JS thread flooding — without coalescing, a 60fps scroll would queue 60 events per second, and if JS can't keep up, the queue grows unboundedly. Discrete events (onPress, onChange) are never coalesced because each one represents a distinct user action.

Q9: How does Fabric improve upon the old architecture's rendering pipeline?

Fabric introduces: (1) C++ shadow nodes shared across threads via JSI, eliminating JSON serialization. (2) Synchronous rendering path for latency-sensitive operations (TextInput). (3) Priority-based event dispatch (SynchronousUnbatched, SynchronousBatched, AsynchronousUnbatched, AsynchronousBatched). (4) Improved view flattening that operates on the shadow tree diff. (5) Support for React 18 concurrent rendering features. (6) Synchronous measurement APIs (measureInWindow). The key architectural shift is from async message passing to shared memory with synchronous access.

Q10: Why does inline style creation cause performance issues in React Native?

Inline style objects (style={{ flex: 1, backgroundColor: color }}) create a new object reference on every render. Since React Native uses reference equality for initial comparison, a new object always triggers the props diffing path, which then sends an updateView command to native even if the actual values haven't changed. StyleSheet.create() returns stable numeric IDs that pass reference equality checks. For dynamic styles, useMemo with proper dependencies ensures new objects are only created when values actually change.

View Flattening

Q11: What is view flattening and why is it important?

View flattening removes unnecessary native views from the hierarchy by identifying "layout-only" views — views that contribute only flexbox layout (padding, margin, flex) without any visual properties (backgroundColor, border, shadow) or event handlers. Their layout contribution is absorbed into their parent's positioning of children. This reduces native view count by 40-60%, saving memory (~2-4KB per view), reducing layout pass time, draw calls, hit testing depth, and GPU overdraw.

Q12: What properties prevent a View from being flattened?

Any visual property: backgroundColor, borderWidth/borderColor/borderRadius, opacity (not 1), shadowColor/shadowOffset, elevation, transform, overflow:'hidden'. Any event handler: onLayout, onPress, onLongPress. Any accessibility prop: accessible, accessibilityLabel, accessibilityRole. Other: nativeID, testID, collapsable={false}, pointerEvents. Even backgroundColor: 'transparent' prevents flattening because the prop is set, even though it's visually invisible.

Q13: How does the collapsable prop work and when would you use it?

collapsable={false} prevents view flattening, ensuring the View always exists in the native hierarchy. Use it when you need to: (1) Get a ref to the native view for measurement (measure/measureInWindow). (2) Use findNodeHandle to get the native tag. (3) Ensure a specific view exists for native module interaction. In the old architecture, collapsable was Android-only (iOS never flattened). In Fabric, it applies to both platforms. Note that onLayout and nativeID already prevent flattening, making collapsable redundant in those cases.

Q14: How does Fabric improve view flattening compared to the old architecture?

In the old architecture (Paper), flattening was applied once during shadow tree construction and was Android-only. In Fabric, flattening operates on the shadow tree diff between old and new trees, and applies to both iOS and Android. This means Fabric can handle dynamic flattening transitions — if a View gains a backgroundColor (becoming non-layout-only), Fabric generates a Create mutation; if it loses it, a Delete mutation. This diff-based approach is more efficient and handles edge cases that Paper couldn't.

Q15: A developer reports that their ref.current.measure() returns null. What's likely happening?

The View was likely flattened — it doesn't exist in the native hierarchy, so there's no native node to measure. This happens when the View has only layout props (flex, padding, margin) and no visual props or event handlers. Fix: add collapsable={false} to force the View to exist natively, or add onLayout which both prevents flattening and provides the measurement directly. In Fabric, this is more common because flattening now applies to iOS as well.

Q16: How does accessibility interact with view flattening?

Any accessibility prop (accessible, accessibilityLabel, accessibilityRole, accessibilityHint) prevents view flattening because the view must exist in the native accessibility tree for screen readers to discover it. This is an intentional trade-off — accessibility is more important than the marginal performance gain from flattening. The architectural guidance is to apply accessibility props at the right level of the hierarchy (on semantic containers, not every wrapper) to minimize the impact on view count.

Q17: Can a View's flattening status change between renders? What are the implications?

Yes. If a View conditionally has visual props (e.g., backgroundColor: highlighted ? 'yellow' : undefined), its flattening status changes between renders. When it transitions from flattened to non-flattened, a native view must be created (more expensive than a prop update). When it transitions back, the native view is destroyed. This create/destroy cycle is more expensive than simply updating a prop on an existing view. Avoid conditional visual props on wrapper views — instead, use a consistent non-flattened view with opacity or color changes.

Q18: How would you audit a screen for unnecessary native views?

Use Flipper's Layout Inspector to see the actual native view hierarchy (post-flattening). Compare it with React DevTools component tree (pre-flattening). On iOS, use Xcode's View Debugger (Debug → View Debugging → Capture View Hierarchy) for a 3D exploded view. On Android, use Android Studio's Layout Inspector. Look for: (1) Views with only layout props that aren't being flattened (check for hidden visual props). (2) Deeply nested native hierarchies that could be simplified. (3) backgroundColor: 'transparent' on wrapper views. Target a ratio of ~3:1 React components to native views.

List Rendering & Performance

Q19: Explain the relationship between FlatList, VirtualizedList, and ScrollView.

FlatList is a thin wrapper around VirtualizedList that assumes array data (provides getItem and getItemCount). VirtualizedList implements the core windowed rendering algorithm and renders into a ScrollView. VirtualizedList manages which items are mounted based on scroll position, windowSize, and batch settings. ScrollView is the native scrolling container (UIScrollView on iOS, ReactScrollView on Android). SectionList also wraps VirtualizedList, flattening sections into a single virtual list with headers as items.

Q20: How does windowed rendering work in FlatList?

VirtualizedList maintains a "window" around the visible area defined by windowSize (default: 21, meaning 10 screens above + visible + 10 screens below). Only items within this window are mounted as React components. Items outside the window are replaced by blank spacer views whose heights are calculated from cached measurements or getItemLayout. As the user scrolls, the window shifts — new items are mounted in batches (maxToRenderPerBatch, default: 10) with delays between batches (updateCellsBatchingPeriod, default: 50ms).

Q21: Does FlatList recycle cells? How does this differ from native list components?

No. FlatList does NOT recycle cells. When an item scrolls out of the window, its React component is unmounted (destroyed). When it scrolls back in, a new component instance is created. This means full component lifecycle (useState initialization, useEffect setup/cleanup, native view creation/destruction) for each mount/unmount. Native UITableView (iOS) and RecyclerView (Android) reuse view instances — they update props on existing views instead of creating new ones. This is why FlatList is slower than native lists for large datasets.

Q22: Why is getItemLayout the most impactful FlatList optimization?

Without getItemLayout, FlatList must render items to measure them, scrollToIndex requires rendering all items up to the target, and spacer heights are estimates that cause scroll position jumps. With getItemLayout, FlatList knows exact item positions without rendering, enabling instant scrollToIndex, accurate spacer heights (no scroll jumps), and faster initial render. It provides { length, offset, index } for any item index, calculated from known item heights and separator heights.

Q23: What happens when you use array index as keyExtractor in FlatList?

Using index as key breaks reconciliation for dynamic lists. If items are reordered, inserted, or deleted, React matches items by index position rather than identity. This causes: (1) Wrong data displayed in wrong cells. (2) Lost component state (e.g., text input content, animation state). (3) Unnecessary re-renders — React updates props on all shifted items instead of moving them. (4) Animation glitches — items appear to morph rather than move. Always use stable unique IDs (item.id) as keys for lists that can change.

Q24: How does FlashList achieve better performance than FlatList?

FlashList (Shopify) introduces true cell recycling via RecyclerListView. Instead of unmounting/remounting components, FlashList reuses component instances by updating their props. This eliminates: component lifecycle overhead, native view creation/destruction, memory allocation/GC pressure. FlashList also uses AutoLayoutView (native) for layout tracking, estimatedItemSize for initial layout, and getItemType for typed recycling pools. The result is ~10x less blank area during scroll, ~30-50% less memory, and ~5x less JS thread work.

Q25: What is the FlashList gotcha with useState inside list items?

Because FlashList recycles cells, useState inside list items persists across different data items. If Item A has local state (e.g., expanded: true) and its cell is recycled for Item B, Item B inherits Item A's expanded state. Fix: (1) Derive all visual state from props, not local state. (2) Use a key on the inner component that changes with item identity to force remount. (3) Use useEffect to reset state when the item prop changes. This is the fundamental trade-off of recycling: performance gain vs state management complexity.

Q26: How would you optimize a FlatList that shows complex cells with images?

Optimization checklist: (1) React.memo on renderItem component. (2) Stable renderItem reference via useCallback. (3) getItemLayout if heights are known. (4) Thumbnails instead of full-resolution images. (5) FastImage for better caching. (6) Reduce windowSize (5-11) to limit mounted items. (7) Smaller maxToRenderPerBatch (5) for smoother batching. (8) removeClippedSubviews={true} to detach off-screen views. (9) Avoid inline functions in renderItem. (10) Consider FlashList for cell recycling. (11) Prefetch next page of images.

Q27: Explain the onEndReached double-firing problem and how to fix it.

onEndReached can fire multiple times because: (1) It fires on every scroll event where the threshold is met, not just the first time. (2) Content layout changes (new items added) can re-trigger the threshold check. Fix: Use a loading flag guard — set loading=true before fetching, check it at the start of the handler, set loading=false after fetch completes. Also ensure onEndReachedThreshold is appropriate (0.5 = half screen from end). Some implementations also use a "has more" flag to stop fetching when all data is loaded.

Q28: When would you choose SectionList over FlatList?

Use SectionList when data is naturally grouped with headers: contact lists (A, B, C sections), settings screens (grouped preferences), timelines (date-grouped events). SectionList provides renderSectionHeader, renderSectionFooter, and stickySectionHeadersEnabled. Internally, it flattens sections into a single VirtualizedList where headers and footers are items. Note: getItemLayout with SectionList must account for header/footer heights in offset calculations, making it more complex than FlatList's getItemLayout.

Q29: What is the windowSize prop and how do you tune it?

windowSize is a multiplier for the visible area that determines how many screens of content are rendered. Default is 21 (10 above + visible + 10 below). Higher values: more items rendered → more memory, less blank area during fast scroll. Lower values: fewer items → less memory, more blank area. Tuning strategy: start with default, reduce to 5-11 if memory is constrained (Android low-end devices), increase if users report blank areas during scroll. Always test on the lowest-spec target device.

Q30: How does removeClippedSubviews work and what are its trade-offs?

removeClippedSubviews detaches native views that are outside the ScrollView's visible bounds from the native view hierarchy (removeFromSuperview/removeView). The React components stay mounted — only the native views are detached. Benefits: reduces GPU overdraw and compositing work. Trade-offs: can cause rendering artifacts (views briefly missing when scrolling back), doesn't reduce JS memory (components still mounted), and can interfere with animations on off-screen items. Use cautiously and test thoroughly on both platforms.

Text Rendering

Q31: How does React Native render nested Text components?

Nested Text components are rendered as a single native text view with attributed/spannable string regions. On iOS, the entire text tree becomes one RCTText with an NSAttributedString where each nested Text applies style attributes to its range. On Android, it becomes one ReactTextView with a SpannableStringBuilder with StyleSpans. This is a key performance optimization — 10 nested Text components create 1 native view, not 10. Style inheritance flows from parent to child Text, and each nested Text can have its own onPress handler (hit testing uses text layout to determine which span was tapped).

Q32: Why is text measurement expensive and how can you minimize its cost?

Text measurement requires the native text layout engine (Core Text on iOS, android.text.Layout on Android) to compute exact dimensions based on font metrics, text content, available width, and style attributes. This must happen on the main thread (or with main-thread font access). Minimize cost by: (1) Avoiding dynamic fontSize changes (use transform for size animations). (2) Caching text measurements (React Native does this automatically — cache key includes all text attributes). (3) Using fixed heights where possible (numberOfLines + known lineHeight). (4) Minimizing the number of unique text style combinations.

Q33: Explain the font loading race condition and how to prevent it.

If Text renders before custom fonts are loaded, the platform uses a fallback font (San Francisco on iOS, Roboto on Android). When the custom font loads, all text re-measures and re-renders, causing a visible layout shift (FOUT — Flash of Unstyled Text). Prevention: gate app rendering on font loading completion. With expo-font: use useFonts() hook and show a splash/loading screen until fontsLoaded is true. With react-native-asset: fonts are bundled at build time and available immediately (no race condition). Always test with slow network to catch font loading issues.

Q34: What are the key platform differences in text rendering?

Major differences: (1) lineHeight — iOS uses exact pixel value, Android adds extra padding (fix: includeFontPadding: false). (2) letterSpacing — iOS uses absolute points, Android uses EM units relative to fontSize. (3) fontWeight — iOS supports precise 100-900 values, Android has limited weight support. (4) fontFamily — iOS uses PostScript name, Android uses filename. (5) ellipsizeMode — head/middle only work single-line on Android. (6) Default font — San Francisco (iOS) vs Roboto (Android). (7) textAlignVertical — Android only. These require per-platform normalization in a design system.

Q35: How does selectable text work differently on iOS and Android?

On iOS, selectable={true} switches the underlying native component from UILabel to UITextView, which supports text selection natively. UITextView is heavier than UILabel, so this has a performance cost. On Android, selectable={true} calls textIsSelectable(true) on the existing TextView. Android also supports selectionColor to customize the highlight color. Only use selectable={true} when needed — it adds weight on iOS and changes touch handling behavior on both platforms.

Q36: How do you handle dynamic type / font scaling in React Native?

React Native respects system font size preferences by default (allowFontScaling={true}). To cap scaling: use maxFontSizeMultiplier (e.g., 1.5 limits to 150% of specified size). For responsive text that shrinks to fit: use adjustsFontSizeToFit with numberOfLines={1} and minimumFontScale (iOS only). Design consideration: always test with maximum system font size to ensure layouts don't break. Some elements (icons, badges) should use allowFontScaling={false} to maintain fixed sizes.

Q37: Why can't you put a View inside a Text component?

Text uses inline text layout (similar to CSS inline elements), while View uses flexbox layout. These are incompatible layout systems. The entire Text subtree is rendered as a single native text view with attributed/spannable string — there's no concept of embedding a flexbox-laid-out view within a text run. On Android, there's partial support for inline views via ReactTextInlineImageSpan, but it's limited. For mixed text+view layouts, use a View container with Text and other components as siblings, using flexDirection: 'row' and flexWrap: 'wrap' for inline-like behavior.

Q38: How would you build a typography system for a cross-platform React Native app?

Create typography tokens (h1, h2, body, caption) as StyleSheet objects with platform-specific values: fontFamily (PostScript name for iOS, filename for Android), lineHeight (adjusted for Android's includeFontPadding), letterSpacing (converted between iOS points and Android EM). Wrap in a Typography component that enforces maxFontSizeMultiplier and applies platform normalization. Gate app on font loading. Test every variant on both platforms with system font scaling at minimum and maximum. Use a single source of truth (design tokens) that generates platform-specific values.

Image Handling

Q39: Describe the image loading pipeline in React Native.

Five stages: (1) Source resolution — parse URI, headers, determine loader (network/local/data URI). (2) Cache check — iOS: NSURLCache (HTTP cache), Android: Fresco multi-tier cache (bitmap → encoded → disk). (3) Download — iOS: NSURLSession, Android: OkHttp via Fresco. Progress callbacks fire. (4) Decode — iOS: ImageIO (CGImageSource), Android: BitmapFactory. Happens on background thread. May downsample if image exceeds display size. (5) Display — decoded bitmap assigned to native image view, resize mode applied, onLoad fires.

Q40: How does image caching differ between iOS and Android?

iOS uses NSURLCache — a standard HTTP cache that respects Cache-Control, ETag, and Last-Modified headers. Default: 50MB memory, 200MB disk. The cache prop (default/reload/force-cache/only-if-cached) is iOS-only. Android uses Fresco's multi-tier cache: L1 bitmap memory cache (decoded, fastest), L2 encoded memory cache (needs decode, less memory), L3 disk cache (persistent). Default: 24MB bitmap, 2MB encoded, 40MB disk. There's no cross-platform cache control API — use FastImage for consistent behavior.

Q41: A photo gallery app is crashing with memory warnings. How do you diagnose and fix it?

Diagnosis: A 4000×3000 RGBA bitmap consumes 48MB. 10 full-res photos = 480MB, approaching iOS kill threshold. Fixes: (1) Server-side resizing — request images at 2× display size, not original resolution. (2) Use thumbnails in grid view, full-res only in detail view. (3) Implement progressive loading (blur-up thumbnail → full image). (4) On Android, enable Fresco downsampling. (5) Reduce FlatList windowSize to limit concurrent images. (6) Use FastImage for better cache management. (7) Clear image cache when navigating away from gallery. (8) Monitor memory with Xcode Instruments / Android Profiler.

Q42: Why does the built-in Image component require width and height for network images?

React Native's layout engine (Yoga) needs dimensions to compute layout synchronously. For local images (require('./img.png')), Metro bundler extracts dimensions at build time and embeds them in the asset metadata. For network images, dimensions are unknown until the image is downloaded. Without explicit width/height, the view would have zero size until the image loads, then jump to the image's natural size — causing a layout shift. This is a deliberate API design to prevent layout instability.

Q43: Compare FastImage to the built-in Image component.

FastImage wraps SDWebImage (iOS) and Glide (Android) instead of NSURLSession/Fresco. Advantages: (1) Cross-platform cache control (immutable, web, cacheOnly). (2) Priority loading (low/normal/high). (3) Full auth header support. (4) Batch preloading API. (5) Cache clearing API (clearMemoryCache/clearDiskCache). (6) Better GIF support. (7) More consistent cross-platform behavior. The built-in Image is adequate for simple use cases, but FastImage (or Expo Image) should be the default for production apps with significant image content.

Q44: How do you implement progressive image loading?

Two approaches: (1) Two-image overlay — render a blurred thumbnail immediately, overlay with full-resolution image, fade out thumbnail on full image load. Use blurRadius on the thumbnail and opacity animation on load. (2) iOS defaultSource — shows a local placeholder while the network image loads (iOS only). (3) Progressive JPEG — server sends JPEG in progressive format, iOS and Fresco (Android) render increasingly detailed versions as data arrives. For best UX, combine approaches: show blurred thumbnail immediately, progressive JPEG for the full image.

Q45: When would you use SVG vs rasterized images in React Native?

SVG (via react-native-svg): icons that need to scale across screen sizes, illustrations with few paths, animated graphics (path morphing), icons that change color dynamically. Rasterized (PNG/JPEG): photos, complex illustrations with many paths, images that don't need scaling, performance-critical scenarios (complex SVGs are expensive to render). Icon fonts: large icon sets with single-color icons (most efficient for icon libraries). Decision: for simple icons, SVG or icon fonts. For photos, always rasterized. For illustrations, SVG if < ~50 paths, rasterized if more complex.

Q46: How do you handle authenticated images in React Native?

Built-in Image supports headers: source={{ uri, headers: { Authorization: 'Bearer token' } }}. However, this has limitations — headers aren't included in cache keys by default, so different users may see cached images from other users. FastImage handles this better with proper header-aware caching. For signed URLs (AWS S3, CloudFront), generate time-limited URLs server-side and use URL-based caching (FastImage.cacheControl.immutable with versioned URLs). Always ensure auth tokens in image URLs have appropriate expiration.

Native View Management

Q47: What is UIManager and what role does it play?

UIManager is the central coordinator for all native view operations. It receives commands from the React reconciler (createView, updateView, manageChildren, setChildren) and dispatches them to the appropriate ViewManagers on the UI thread. In the old architecture, UIManager is a native module accessed via the bridge. It batches commands and executes them atomically on the UI thread. In Fabric, UIManager is replaced by direct C++ mounting via JSI — mutations are C++ structs applied without serialization.

Q48: Explain setNativeProps and why it's deprecated in Fabric.

setNativeProps bypasses React reconciliation to update native view properties directly via UIManager.updateView. It's faster for high-frequency updates (gestures, animations) because it skips the JS → reconcile → diff → bridge path. However, it creates a divergence between React's virtual tree and native state — the next React re-render may overwrite setNativeProps changes. Fabric deprecates it because: (1) Fabric's synchronous rendering makes setState nearly as fast. (2) Reanimated's shared values provide a better model for native-driven animations. (3) The divergence problem is architecturally unsound.

Q49: How do measure(), measureInWindow(), and measureLayout() differ?

measure() returns position relative to parent (x, y) and absolute position (pageX, pageY) plus dimensions. measureInWindow() returns position relative to the device screen, accounting for status bar and navigation bar. measureLayout() returns position relative to a specified ancestor view. All are async in the old architecture (cross the bridge). In Fabric, measureInWindow can be synchronous via JSI. Common pitfall: calling measure on a flattened view returns null — use collapsable={false} to ensure the native view exists.

Q50: When should you build a custom native view vs composing existing components?

Build custom native views when: (1) No existing RN component covers the functionality (camera, maps, video). (2) Performance requires native-level optimization (custom drawing, GPU shaders). (3) Platform-specific UI can't be composed (native date picker, share sheet). (4) Third-party SDK provides a native view (ad SDKs, payment forms). Don't build custom views when: the UI can be composed from existing components, the performance issue is in JS not rendering, or a maintained community library exists. In Fabric, use codegen from JS specs for type-safe native components.

Q51: How does onLayout interact with view flattening and performance?

onLayout prevents view flattening because the view must exist natively to report layout events. It fires on initial mount and whenever the view's dimensions or position (relative to parent) change. Performance implications: (1) Prevents flattening → extra native view. (2) Fires asynchronously → layout-dependent logic has one frame delay. (3) Can cause cascading re-renders if onLayout triggers setState that changes layout. Use sparingly — only on views that genuinely need measurement. For responsive layouts, consider useWindowDimensions() instead of onLayout on the root view.

Platform Differences

Q52: What is the most common cross-platform rendering bug and how do you prevent it?

The default overflow behavior difference: iOS defaults to overflow: 'visible' (children render outside parent bounds), Android defaults to overflow: 'hidden' (children clipped). This causes content to appear on iOS but be invisibly clipped on Android. Prevention: always explicitly set overflow when content may extend beyond parent bounds. In design system components, document overflow behavior. Test every screen on both platforms — this bug is invisible if you only test on one platform.

Q53: How do shadows work differently on iOS and Android?

iOS: Fine-grained control via shadowColor, shadowOffset (x,y), shadowOpacity (0-1), shadowRadius. Shadows follow the view's shape (including borderRadius). Don't affect z-ordering. Expensive (offscreen render pass). Android: Only elevation prop — controls shadow size, blur, and z-ordering simultaneously. Fixed dark gray color, fixed light source direction (top). Follows outline on API 21+. Hardware-accelerated. Both require backgroundColor. Cross-platform: create a Shadow component that uses platform-appropriate props and maps elevation levels to iOS shadow parameters.

Q54: How do you handle the keyboard differently on iOS and Android?

iOS: Keyboard slides up with animation, KeyboardAvoidingView works well with behavior="padding". Android: Keyboard behavior depends on windowSoftInputMode in AndroidManifest.xml — adjustResize (resizes activity, recommended), adjustPan (pans activity), adjustNothing. KeyboardAvoidingView behavior should be "height" on Android. react-native-keyboard-aware-scroll-view provides better cross-platform behavior. For chat-like UIs, use KeyboardAvoidingView + FlatList with inverted={true}. Always test keyboard interaction on both platforms with different keyboard types.

Q55: How does renderToHardwareTextureAndroid differ from shouldRasterizeIOS?

renderToHardwareTextureAndroid creates a GPU texture for the view subtree. Useful for views that animate as a unit (e.g., a card that fades or translates). The texture is re-rendered only when the view content changes. shouldRasterizeIOS rasterizes the CALayer to a bitmap cache. Useful for complex static content that scrolls. Both trade memory (texture/bitmap storage) for rendering speed (skip re-drawing). Use during animations, remove after animation completes to free GPU memory. Neither is needed for simple views — only for complex subtrees.

Q56: What's the recommended approach for safe area handling?

Use react-native-safe-area-context (not the built-in SafeAreaView which is iOS-only and limited). Wrap app root in SafeAreaProvider. Use SafeAreaView for full-screen containers, useSafeAreaInsets() hook for fine-grained control (e.g., only top padding on headers, only bottom padding on tab bars). Typical insets: iPhone 15 Pro (top: 59, bottom: 34), iPhone SE (top: 20, bottom: 0), Android (top: 24-48, bottom: 0-48 depending on navigation mode). Always test on devices with notches, rounded corners, and gesture navigation.

Q57: How would you architect a design system that handles platform differences?

Create abstraction components for every platform-divergent feature: Shadow (maps elevation levels to platform shadows), Typography (normalizes lineHeight, letterSpacing, fontFamily), SafeArea (wraps react-native-safe-area-context), Input (normalizes padding, keyboard behavior), Card (combines shadow + border + overflow). Use Platform.select for style differences, platform-specific file extensions (.ios.js/.android.js) for behavioral differences. Define design tokens (colors, spacing, typography) as platform-agnostic, with platform-specific implementations. Test every component on both platforms in a Storybook-like environment. Document known platform differences for the team.
