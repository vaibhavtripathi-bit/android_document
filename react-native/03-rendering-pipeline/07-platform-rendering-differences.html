Platform Rendering Differences

iOS and Android have fundamentally different rendering engines, view systems, and platform conventions. Understanding these differences is essential for building truly cross-platform UIs and debugging platform-specific rendering issues.

1.1 Rendering Performance Implications

// iOS: Views are inherently GPU-accelerated
// Animations on CALayer properties (opacity, transform) are free
// because they happen on the GPU without CPU involvement

// Android: Need to explicitly enable hardware layers
<View
  renderToHardwareTextureAndroid={true}
  // Creates a GPU texture for this view subtree
  // Useful for: complex views that animate as a unit
  // Cost: GPU memory for the texture
  // Benefit: animations don't re-draw the view
>
  <ComplexContent />
</View>

// iOS equivalent (always on):
<View
  shouldRasterizeIOS={true}
  // Rasterizes the layer to a bitmap
  // Useful for: complex static content that scrolls
  // Cost: memory for rasterized bitmap
  // Benefit: faster compositing during scroll
/>

2. Elevation and Shadow Differences

Shadows are one of the most visually different aspects between iOS and Android in React Native.

// iOS shadows: fine-grained control via CALayer shadow properties
<View style={{
  shadowColor: '#000',
  shadowOffset: { width: 0, height: 2 },
  shadowOpacity: 0.25,
  shadowRadius: 3.84,
  backgroundColor: 'white', // REQUIRED — no shadow without background
}}>
  <Text>iOS shadow</Text>
</View>

// Android shadows: elevation only (Material Design)
<View style={{
  elevation: 5,
  backgroundColor: 'white', // REQUIRED — no shadow without background
}}>
  <Text>Android shadow</Text>
</View>

// Cross-platform shadow:
const shadow = Platform.select({
  ios: {
    shadowColor: '#000',
    shadowOffset: { width: 0, height: 2 },
    shadowOpacity: 0.25,
    shadowRadius: 3.84,
  },
  android: {
    elevation: 5,
  },
});
Shadow Gotchas

Both platforms require backgroundColor for shadows to render. On iOS, shadows are rendered by the CALayer and follow the view's shape (including borderRadius). On Android, elevation shadows are rectangular by default — they only follow borderRadius on API 21+ and may not match iOS exactly. Android elevation also affects z-ordering (higher elevation = on top), while iOS shadows don't affect z-order.

3. Border Rendering

// Border rendering differs significantly between platforms

// Uniform borders work the same on both:
<View style={{
  borderWidth: 1,
  borderColor: '#ccc',
  borderRadius: 8,
}}>
  <Text>Consistent on both platforms</Text>
</View>

// Per-side borders: platform differences
<View style={{
  borderTopWidth: 2,
  borderTopColor: 'red',
  borderBottomWidth: 1,
  borderBottomColor: 'blue',
  borderRadius: 8, // ← Problem!
}}>
  <Text>May render differently</Text>
</View>
// iOS: Renders correctly with per-side colors and radius
// Android: May show rendering artifacts with mixed borders + radius

// Per-corner radius:
<View style={{
  borderTopLeftRadius: 16,
  borderTopRightRadius: 16,
  borderBottomLeftRadius: 0,
  borderBottomRightRadius: 0,
}}>
  <Text>Rounded top only</Text>
</View>
// Works on both platforms, but anti-aliasing may differ

// borderStyle differences:
// 'solid' — works on both
// 'dotted' — iOS only (Android falls back to solid)
// 'dashed' — iOS only (Android falls back to solid)

4. Overflow Behavior

// DEFAULT overflow behavior differs!

// iOS default: overflow: 'visible'
// Children can render outside parent bounds

// Android default: overflow: 'hidden'
// Children are clipped to parent bounds

// This causes one of the most common cross-platform bugs:
<View style={{ width: 100, height: 100 }}>
  <View style={{
    width: 150, height: 150,
    backgroundColor: 'red',
  }} />
</View>
// iOS: Red view extends 50px beyond parent (visible)
// Android: Red view clipped at parent bounds (hidden)

// Fix: explicitly set overflow
<View style={{ overflow: 'visible' }}>
  // Consistent on both platforms
</View>

// Shadow + overflow interaction:
// iOS: shadowColor with overflow: 'hidden' clips the shadow
// Solution: use a wrapper view for shadow, inner view for clipping
<View style={{ shadowColor: '#000', shadowOffset: { width: 0, height: 2 } }}>
  <View style={{ overflow: 'hidden', borderRadius: 8 }}>
    <Image source={src} style={{ width: 200, height: 200 }} />
  </View>
</View>
Overflow and Touch Handling

On iOS, even with overflow: 'visible', touch events are only received within the parent's bounds by default. Content that overflows is visible but not touchable unless you override hitTest:withEvent:. On Android, overflow: 'visible' makes content both visible and touchable outside bounds.

5. ScrollView Differences

// ScrollView maps to different native components:
// iOS: UIScrollView (highly optimized, rubber-band physics)
// Android: ReactScrollView (custom ScrollView subclass)

// iOS-only ScrollView features:
<ScrollView
  bounces={true}              // Rubber-band effect (iOS only)
  bouncesZoom={true}          // Bounce on zoom (iOS only)
  alwaysBounceVertical={true} // Bounce even when content fits
  scrollIndicatorInsets={{ right: 1 }} // Scroll indicator position
  automaticallyAdjustContentInsets={false}
  contentInsetAdjustmentBehavior="automatic"
  indicatorStyle="black"     // Scroll indicator color
  decelerationRate="fast"    // Scroll deceleration
/>

// Android-only ScrollView features:
<ScrollView
  overScrollMode="always"    // Overscroll glow effect
  fadingEdgeLength={100}     // Fading edge indicator
  nestedScrollEnabled={true} // Allow nested scrolling
  persistentScrollbar={true} // Always show scrollbar
/>

// Scroll momentum difference:
// iOS: Smooth deceleration with rubber-band at edges
// Android: Abrupt stop with overscroll glow
// This is the most noticeable UX difference between platforms

6. Text Input Differences

// TextInput maps to:
// iOS: UITextField (single-line) or UITextView (multi-line)
// Android: EditText

// Padding behavior differs:
<TextInput
  style={{
    padding: 10,
    // iOS: padding works as expected
    // Android: has built-in padding that adds to your padding
  }}
/>

// Fix Android extra padding:
<TextInput
  style={{
    padding: 10,
    ...Platform.select({
      android: {
        paddingVertical: 0, // Remove Android default vertical padding
      },
    }),
  }}
/>

// Keyboard behavior:
// iOS: Keyboard slides up, content adjusts automatically
// Android: Keyboard behavior depends on windowSoftInputMode
//   - adjustResize: resizes the activity
//   - adjustPan: pans the activity
//   - adjustNothing: no adjustment

// returnKeyType mapping:
// 'done'    → iOS: "Done"    | Android: IME_ACTION_DONE
// 'go'      → iOS: "Go"      | Android: IME_ACTION_GO
// 'next'    → iOS: "Next"    | Android: IME_ACTION_NEXT
// 'search'  → iOS: "Search"  | Android: IME_ACTION_SEARCH
// 'send'    → iOS: "Send"    | Android: IME_ACTION_SEND

7. Safe Area Handling

// Safe areas: regions not obscured by system UI
// iOS: notch, home indicator, status bar, rounded corners
// Android: status bar, navigation bar, display cutouts

import { SafeAreaView } from 'react-native';
// Built-in SafeAreaView — iOS only, limited

// Recommended: react-native-safe-area-context
import {
  SafeAreaProvider,
  SafeAreaView,
  useSafeAreaInsets,
} from 'react-native-safe-area-context';

// Provider at app root:
<SafeAreaProvider>
  <App />
</SafeAreaProvider>

// Use insets for fine-grained control:
const Header = () => {
  const insets = useSafeAreaInsets();
  return (
    <View style={{ paddingTop: insets.top }}>
      <Text>Header content</Text>
    </View>
  );
};

// Typical inset values:
// iPhone 15 Pro: { top: 59, bottom: 34, left: 0, right: 0 }
// iPhone SE:     { top: 20, bottom: 0, left: 0, right: 0 }
// Android:       { top: 24, bottom: 0, left: 0, right: 0 } (status bar)
// Android + nav: { top: 24, bottom: 48, left: 0, right: 0 }

8. Status Bar

import { StatusBar } from 'react-native';

// iOS: Status bar overlays content (transparent by default)
// Android: Status bar has its own space (opaque by default)

// Make Android status bar transparent (like iOS):
<StatusBar
  translucent={true}         // Android: content renders behind status bar
  backgroundColor="transparent" // Android: remove status bar background
  barStyle="dark-content"   // Both: dark icons on light background
/>

// barStyle options:
// 'default'       — platform default
// 'light-content' — white icons (for dark backgrounds)
// 'dark-content'  — dark icons (for light backgrounds)

// Per-screen status bar (with React Navigation):
const DarkScreen = () => (
  <View style={{ flex: 1, backgroundColor: '#000' }}>
    <StatusBar barStyle="light-content" />
    <Text style={{ color: 'white' }}>Dark screen</Text>
  </View>
);

// StatusBar.currentHeight (Android only):
// Returns the status bar height in pixels
// Useful for manual padding calculations
const statusBarHeight = StatusBar.currentHeight || 0;

  }
}

10. Architect Considerations

Staff-Level Mental Model

Platform differences are not bugs — they're fundamental to each platform's identity. A Staff engineer's approach should be: (1) Embrace platform conventions — don't force iOS patterns on Android or vice versa. (2) Abstract differences in a design system — create components like Shadow, Card, SafeArea that handle platform differences internally. (3) Test on both platforms continuously — never assume cross-platform consistency. (4) Know when to diverge — some features (navigation transitions, haptics, status bar) should feel native to each platform. The goal is consistent brand experience, not pixel-identical rendering.

10.1 Cross-Platform Abstraction Strategy

// Design system shadow component
const Shadow = ({ level = 1, children, style }) => {
  const shadowStyles = Platform.select({
    ios: {
      shadowColor: '#000',
      shadowOffset: { width: 0, height: level },
      shadowOpacity: 0.08 * level,
      shadowRadius: level * 2,
    },
    android: {
      elevation: level * 2,
    },
  });

  return (
    <View style={[shadowStyles, { backgroundColor: 'white' }, style]}>
      {children}
    </View>
  );
};

// Usage: platform differences hidden from consumers
<Shadow level={3}>
  <Text>Consistent shadow on both platforms</Text>
</Shadow>
