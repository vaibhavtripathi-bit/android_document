1. VirtualizedList: The Foundation

All list components in React Native are built on VirtualizedList, which implements the core windowed rendering algorithm. Understanding VirtualizedList is essential because FlatList and SectionList are thin wrappers around it.

Component Hierarchy: FlatList ──────────→ VirtualizedList ──────────→ ScrollView │ │ │ │ data[] │ getItem(data, index) │ Native ScrollView │ renderItem │ getItemCount(data) │ (UIScrollView / │ keyExtractor │ renderItem │ android ScrollView) │ │ windowSize │ │ │ maxToRenderPerBatch │ └──────────────────────┘ │ │ SectionList ───────→ VirtualizedList ─────────────────┘ │ │ sections[] │ renderItem │ renderSectionHeader

1.1 VirtualizedList Architecture

// VirtualizedList core state
interface VirtualizedListState {
  first: number;      // First rendered item index
  last: number;       // Last rendered item index
}

// The rendering window is defined by:
// - Visible area (what the user sees)
// - Overscan area (buffer above/below visible area)
// - windowSize prop controls total window = visible * windowSize

// VirtualizedList key props:
interface VirtualizedListProps {
  data: any;                    // Data source
  getItem: (data, index) => any; // Extract item from data
  getItemCount: (data) => number;// Total item count
  renderItem: (info) => Element; // Render function
  windowSize: number;            // Render window multiplier (default: 21)
  maxToRenderPerBatch: number;   // Items per render batch (default: 10)
  updateCellsBatchingPeriod: number; // ms between batches (default: 50)
  initialNumToRender: number;   // Items in first render (default: 10)
  onEndReachedThreshold: number; // Trigger distance (default: 2)
}
Architect Insight

VirtualizedList's windowSize default of 21 means it renders 10 screens worth of content above and below the visible area. This is deliberately generous to prevent blank areas during fast scrolling. For memory-constrained scenarios, reducing windowSize to 5-11 trades scroll smoothness for lower memory usage.

2. FlatList Internals

FlatList is the primary list component. It wraps VirtualizedList with a simpler API that assumes array data.

// FlatList is essentially:
class FlatList extends React.PureComponent {
  render() {
    return (
      <VirtualizedList
        {...this.props}
        getItem={(data, index) => data[index]}
        getItemCount={(data) => data?.length ?? 0}
        keyExtractor={this.props.keyExtractor ?? this._defaultKeyExtractor}
      />
    );
  }

  _defaultKeyExtractor = (item, index) => {
    return item.key ?? item.id ?? String(index);
  };
}

// FlatList adds convenience features:
// - numColumns support (wraps items in row containers)
// - columnWrapperStyle for multi-column layouts
// - Automatic key extraction from item.key or item.id

2.1 Render Cycle

FlatList Render Cycle: 1. Initial Mount ├─ Render initialNumToRender items (default: 10) ├─ Measure visible area └─ Schedule fill batches 2. Fill Phase (async, batched) ├─ Render maxToRenderPerBatch items (default: 10) ├─ Wait updateCellsBatchingPeriod ms (default: 50) ├─ Repeat until window is filled └─ Window = visible area × windowSize 3. Scroll Update ├─ onScroll fires (from native ScrollView) ├─ Recalculate visible range ├─ Determine items to add/remove └─ Batch render new items, unmount distant items 4. Steady State ├─ Only items within window are mounted ├─ Items outside window are unmounted (not recycled!) └─ Blank spacers fill unmounted regions

3. Windowed Rendering

Windowed rendering is the core optimization: only items within a "window" around the visible area are actually rendered as React components. Items outside the window are replaced with blank spacer views.

3.1 Window Calculation

// Window size calculation
// windowSize = 21 (default)
// visibleLength = 800px (screen height)

// Total window = visibleLength × windowSize = 800 × 21 = 16,800px
// Above visible: (21-1)/2 × 800 = 8,000px
// Below visible: (21-1)/2 × 800 = 8,000px

// With 50px items: ~336 items rendered at any time
// Out of potentially thousands in the data array

// Reducing windowSize:
<FlatList
  data={items}
  renderItem={renderItem}
  windowSize={5}  // Only 2 screens above + visible + 2 screens below
  // Total window = 800 × 5 = 4,000px
  // ~80 items rendered (vs 336 with default)
  // Trade-off: may see blank areas during fast scroll
/>

3.2 Spacer Views

// VirtualizedList renders the list as:

// [Header spacer — height of all unmounted items above window]
// [Rendered item 50]
// [Rendered item 51]
// [Rendered item 52]
// ... (items within window)
// [Rendered item 150]
// [Footer spacer — height of all unmounted items below window]

// Spacer heights are calculated from:
// 1. getItemLayout (if provided) — exact heights
// 2. Cached measurements from previously rendered items
// 3. Estimated heights (can cause scroll jumps)

// This is why getItemLayout is so important:
// Without it, spacer heights are estimates that may be wrong,
// causing the scroll position to jump when items are measured

4. Cell Recycling

Critical Distinction

React Native's built-in FlatList does NOT recycle cells. When an item scrolls out of the window, its React component is unmounted (destroyed). When it scrolls back in, a new component is created. This is fundamentally different from native UITableView/RecyclerView which reuse view instances. FlashList (covered in section 9) adds true cell recycling.

4.1 FlatList: Mount/Unmount Model

FlatList (VirtualizedList) — No Recycling: Scroll Down: Item 1 [MOUNTED] → Item 1 [UNMOUNTED — destroyed] Item 2 [MOUNTED] → Item 2 [MOUNTED] Item 3 [MOUNTED] → Item 3 [MOUNTED] Item 4 [not mounted] → Item 4 [MOUNTED — new instance] Cost: Full React component lifecycle for each mount/unmount - constructor / useState initialization - useEffect setup/cleanup - Native view creation/destruction - Memory allocation/GC pressure Native Equivalent (UITableView/RecyclerView) — Recycling: Cell A [showing Item 1] → Cell A [showing Item 4] ← reused! Cell B [showing Item 2] → Cell B [showing Item 2] Cell C [showing Item 3] → Cell C [showing Item 3] Cost: Only prop updates on existing views — much cheaper

5. getItemLayout Optimization

getItemLayout is the single most impactful optimization for FlatList. It provides exact layout information without requiring measurement, enabling instant scroll-to-index and accurate spacer heights.

// Without getItemLayout:
// - Items must be rendered to be measured
// - scrollToIndex requires rendering all items up to target
// - Spacer heights are estimates → scroll jumps

// With getItemLayout (fixed-size items):
const ITEM_HEIGHT = 72;
const SEPARATOR_HEIGHT = 1;

<FlatList
  data={items}
  renderItem={renderItem}
  getItemLayout={(data, index) => ({
    length: ITEM_HEIGHT,
    offset: (ITEM_HEIGHT + SEPARATOR_HEIGHT) * index,
    index,
  })}
  ItemSeparatorComponent={Separator}
/>

// With getItemLayout (variable-size items with known heights):
const itemHeights = items.map(item =>
  item.type === 'header' ? 120 : 72
);

const offsets = itemHeights.reduce((acc, h, i) => {
  acc[i] = i === 0 ? 0 : acc[i - 1] + itemHeights[i - 1];
  return acc;
}, []);

const getItemLayout = (data, index) => ({
  length: itemHeights[index],
  offset: offsets[index],
  index,
});

6. keyExtractor and Reconciliation

Keys in FlatList serve the same purpose as keys in React DOM lists — they enable the reconciler to match items across re-renders and minimize unnecessary work.

// Bad: using index as key
<FlatList
  data={items}
  renderItem={renderItem}
  keyExtractor={(item, index) => String(index)}
/>
// Problem: if items are reordered, inserted, or deleted,
// React will update props on wrong components instead of moving them
// This causes: wrong data displayed, lost component state,
// unnecessary re-renders, animation glitches

// Good: stable unique ID
<FlatList
  data={items}
  renderItem={renderItem}
  keyExtractor={(item) => item.id}
/>
// React can correctly match items across re-renders
// Reorders become move operations, not destroy+create

// Key collision detection (development only):
// React Native warns when duplicate keys are found
// "Encountered two children with the same key"
// This causes unpredictable behavior — items may disappear or duplicate

7. Infinite Scroll with onEndReached

const InfiniteList = () => {
  const [data, setData] = useState(initialData);
  const [loading, setLoading] = useState(false);
  const [hasMore, setHasMore] = useState(true);

  const loadMore = useCallback(async () => {
    if (loading || !hasMore) return;
    setLoading(true);
    const newItems = await fetchPage(data.length);
    if (newItems.length === 0) {
      setHasMore(false);
    } else {
      setData(prev => [...prev, ...newItems]);
    }
    setLoading(false);
  }, [loading, hasMore, data.length]);

  return (
    <FlatList
      data={data}
      renderItem={renderItem}
      keyExtractor={item => item.id}
      onEndReached={loadMore}
      onEndReachedThreshold={0.5}
      ListFooterComponent={loading ? <ActivityIndicator /> : null}
    />
  );
};

// onEndReachedThreshold = 0.5 means:
// Trigger when user is within 0.5 × visibleLength from the end
// With 800px visible area: triggers at 400px from bottom

// Common bug: onEndReached fires multiple times
// Fix: guard with loading flag (shown above)
// Or use onMomentumScrollEnd for more controlled triggering

9. FlashList (Shopify)

FlashList is Shopify's high-performance replacement for FlatList that introduces true cell recycling to React Native, similar to native UITableView/RecyclerView behavior.

9.1 Architecture

FlashList Architecture: FlashList (JS) └─ RecyclerListView (JS) — layout management └─ AutoLayoutView (Native) — native layout tracking └─ Cell Pool ├─ Cell A [type: "message"] ← reused across items ├─ Cell B [type: "message"] ← reused across items ├─ Cell C [type: "image"] ← reused for image items └─ Cell D [type: "header"] ← reused for headers Key Difference from FlatList: FlatList: scroll out → UNMOUNT → scroll in → MOUNT (new instance) FlashList: scroll out → RECYCLE → scroll in → UPDATE PROPS (same instance)

9.2 Cell Recycling Mechanism

// FlashList reuses component instances by changing their props
// This avoids the cost of mount/unmount cycles

import { FlashList } from '@shopify/flash-list';

<FlashList
  data={messages}
  renderItem={({ item }) => <MessageRow message={item} />}
  estimatedItemSize={72}
  // estimatedItemSize is REQUIRED — FlashList uses it for
  // initial layout before items are measured
/>

// overrideItemLayout for variable-size items:
<FlashList
  data={items}
  renderItem={renderItem}
  estimatedItemSize={72}
  overrideItemLayout={(layout, item, index, maxColumns, extraData) => {
    layout.size = item.type === 'header' ? 120 : 72;
    layout.span = item.type === 'header' ? maxColumns : 1;
  }}
/>

// getItemType for recycling pools:
<FlashList
  data={items}
  renderItem={renderItem}
  estimatedItemSize={72}
  getItemType={(item) => item.type}
  // Items with same type share a recycling pool
  // "message" cells are only reused for other "message" items
  // This prevents expensive re-renders when cell structure differs
/>

FlashList Gotcha: Component Identity

Because FlashList recycles cells, your renderItem component receives new props on the same instance. This means useState inside list items persists across different data items. Always derive state from props, or use key on the inner component to force remount when the item identity changes.

10. Large List Optimization Strategies

10.1 Comprehensive Optimization Checklist

// 1. Use React.memo on renderItem component
const ItemRow = React.memo(({ item }) => (
  <View style={styles.row}>
    <Text>{item.title}</Text>
  </View>
));

// 2. Stable renderItem reference
const renderItem = useCallback(
  ({ item }) => <ItemRow item={item} />,
  []
);

// 3. Avoid inline functions in renderItem
// Bad: creates new function every render
renderItem={({ item }) => (
  <Pressable onPress={() => handlePress(item.id)}>
    <Text>{item.title}</Text>
  </Pressable>
)}

// Good: pass id as prop, handle in memoized component
const ItemRow = React.memo(({ id, title, onPress }) => (
  <Pressable onPress={() => onPress(id)}>
    <Text>{title}</Text>
  </Pressable>
));

// 4. Reduce re-renders with extraData
<FlatList
  data={items}
  renderItem={renderItem}
  extraData={selectedId} // Only re-render when selection changes
/>

// 5. removeClippedSubviews (use with caution)
<FlatList
  removeClippedSubviews={true}
  // Detaches off-screen views from native hierarchy
  // Reduces memory but can cause rendering artifacts
/>

10.2 Memory Management

// Memory optimization for large lists:

// 1. Image memory — use thumbnails in lists
const ListImage = ({ uri }) => (
  <Image
    source={{ uri: getThumbnailUrl(uri, 100, 100) }}
    style={{ width: 50, height: 50 }}
  />
);

// 2. Reduce windowSize for memory-constrained devices
const windowSize = Platform.OS === 'android'
  ? 5   // Android: more memory-constrained
  : 11; // iOS: more generous

// 3. maxToRenderPerBatch — smaller batches = less memory spikes
<FlatList
  maxToRenderPerBatch={5}
  updateCellsBatchingPeriod={100}
  // Renders 5 items every 100ms instead of 10 every 50ms
  // Smoother memory allocation, slightly slower fill
/>
Staff-Level Mental Model

List performance is the most common React Native performance bottleneck. The key architectural decision is FlatList vs FlashList: FlatList is simpler and works well for lists under ~100 items with simple cells. FlashList is essential for lists with 1000+ items, complex cells, or when blank area during scroll is unacceptable. At Staff level, you should also know when to drop down to native list components (UICollectionView/RecyclerView) via native modules for the most demanding scenarios.
