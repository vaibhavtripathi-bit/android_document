Text Rendering

Text rendering in React Native maps to fundamentally different native components on each platform. Understanding the native text stack, measurement pipeline, font loading, and platform differences is critical for building consistent cross-platform typography.

1. Native Text Components

React Native's <Text> component maps to completely different native views on iOS and Android, each with distinct rendering characteristics.

React Native <Text> │ ├──── iOS ──────────────────────────────────────── │ RCTText (custom UIView subclass) │ └─ Uses NSAttributedString for rendering │ └─ NSTextStorage + NSLayoutManager + NSTextContainer │ └─ Core Text framework for glyph layout │ └─ Supports: UILabel (single-line), UITextView (multi-line) │ └──── Android ──────────────────────────────────── ReactTextView (extends android.widget.TextView) └─ Uses SpannableStringBuilder └─ android.text.Layout for measurement └─ StaticLayout (immutable) or DynamicLayout (editable) └─ Skia for glyph rendering

1.1 Text Component Hierarchy

// Simple text — maps to a single native text view
<Text style={{ fontSize: 16, color: '#333' }}>
  Hello World
</Text>

// iOS: Creates RCTText → renders NSAttributedString
// Android: Creates ReactTextView → renders SpannableString

// Text is NOT a View — it has special behavior:
// 1. Text uses inline layout (not flexbox)
// 2. Text children are concatenated, not stacked
// 3. Nested Text inherits parent styles
// 4. Text cannot contain non-Text children (except on Android)

// This is INVALID:
<Text>
  Hello <View /> World  // Error: View inside Text
</Text>

// This is VALID:
<Text>
  Hello <Text style={{ fontWeight: 'bold' }}>World</Text>
</Text>
Key Insight

Text in React Native is fundamentally different from View. While Views use flexbox layout, Text uses inline text layout (similar to CSS inline elements). This is why you cannot mix View and Text children — they use incompatible layout systems. The entire text subtree is rendered as a single native text view with attributed/spannable string.

2. Nested Text Components

Nested <Text> components create a single native text view with styled spans, not multiple native views.

// Nested Text creates a single attributed string
<Text style={{ fontSize: 16 }}>
  Hello{' '}
  <Text style={{ fontWeight: 'bold' }}>bold{' '}</Text>
  <Text style={{ color: 'red' }}>red{' '}</Text>
  <Text style={{ fontStyle: 'italic' }}>italic</Text>
</Text>

// iOS: Single RCTText with NSAttributedString:
// "Hello " (fontSize: 16)
// "bold " (fontSize: 16, fontWeight: bold)
// "red " (fontSize: 16, color: red)
// "italic" (fontSize: 16, fontStyle: italic)

// Android: Single ReactTextView with SpannableStringBuilder:
// SpannableStringBuilder("Hello bold red italic")
//   .setSpan(StyleSpan(BOLD), 6, 11)
//   .setSpan(ForegroundColorSpan(RED), 11, 15)
//   .setSpan(StyleSpan(ITALIC), 15, 21)

// Style inheritance: child Text inherits parent styles
<Text style={{ fontSize: 20, color: 'blue' }}>
  Parent text
  <Text style={{ fontWeight: 'bold' }}>
    Bold child (inherits fontSize: 20, color: blue)
  </Text>
</Text>

// Nested Text can have individual press handlers
<Text>

3. Text Measurement

Text measurement is one of the most expensive operations in React Native because it requires native text layout engines to compute the exact dimensions.

3.1 Measurement Pipeline

Text Measurement Pipeline: 1. JS: Text component renders with style props 2. Shadow Tree: TextShadowNode created with text attributes 3. Yoga: Requests text measurement (measure function callback) 4. Native: Text layout engine computes dimensions ├─ iOS: NSTextStorage.boundingRect(with:options:context:) └─ Android: StaticLayout.getHeight() / .getLineWidth() 5. Yoga: Receives measured size, completes layout 6. Native: View created with computed frame Key constraint: Text measurement MUST happen on the main thread (or at least with access to font metrics and text layout engine) This is why text-heavy screens can be slow to lay out.

3.2 Measurement Caching

// React Native caches text measurements to avoid redundant work
// Cache key includes: text content, font, fontSize, fontWeight,
// lineHeight, maxWidth, numberOfLines, etc.

// Cache invalidation happens when any text attribute changes
// This is why dynamic text styles are expensive:

// Bad: fontSize changes on every render → cache miss
const AnimatedText = ({ progress }) => (
  <Text style={{ fontSize: 16 + progress * 8 }}>
    Growing text
  </Text>
);
// Every frame: new fontSize → new measurement → expensive!

// Better: use transform for size animations
const AnimatedText = ({ progress }) => (
  <Animated.Text style={{
    fontSize: 16,
    transform: [{ scale: 1 + progress * 0.5 }],
  }}>
    Growing text
  </Animated.Text>
);
// Transform doesn't trigger re-measurement

4. Font Loading (Custom Fonts)

4.1 Platform Font Registration

// iOS: Fonts registered in Info.plist
// <key>UIAppFonts</key>
// <array>
//   <string>Inter-Regular.ttf</string>
//   <string>Inter-Bold.ttf</string>
// </array>

// Android: Fonts placed in android/app/src/main/assets/fonts/
// Referenced by filename (without extension)

// Usage in React Native:
<Text style={{ fontFamily: 'Inter-Regular' }}>
  Custom font text
</Text>

// Platform difference in fontFamily:
// iOS: Uses PostScript name (e.g., "Inter-Regular")
// Android: Uses filename (e.g., "Inter-Regular")
// These usually match, but not always!

// Cross-platform font family with fontWeight:
<Text style={{
  fontFamily: Platform.select({
    ios: 'Inter',         // iOS uses family name + weight
    android: 'Inter-Bold', // Android needs specific file
  }),
  fontWeight: 'bold',
}}>
  Bold text
</Text>

4.2 Dynamic Font Loading

// expo-font for dynamic font loading
import * as Font from 'expo-font';

const App = () => {
  const [fontsLoaded] = Font.useFonts({
    'Inter-Regular': require('./assets/fonts/Inter-Regular.ttf'),
    'Inter-Bold': require('./assets/fonts/Inter-Bold.ttf'),
  });

  if (!fontsLoaded) return <AppLoading />;

  return <MainApp />;
};

// Font loading is async because:
// 1. Font file must be read from disk/network
// 2. Font must be registered with platform font system
// 3. Font metrics must be computed
// Rendering text before font loads → fallback font → layout shift
Font Loading Race Condition

If you render Text before custom fonts are loaded, the platform uses a fallback font. When the custom font loads, all text re-measures and re-renders, causing a visible layout shift. Always gate your app on font loading completion to avoid this flash of unstyled text (FOUT).

5. Rich Text with Nested Styles

// Building a rich text renderer
const RichText = ({ segments }) => (
  <Text>
    {segments.map((segment, i) => {
      const style = {
        fontWeight: segment.bold ? 'bold' : 'normal',
        fontStyle: segment.italic ? 'italic' : 'normal',
        textDecorationLine: segment.underline ? 'underline' : 'none',
        color: segment.color || '#000',
        fontSize: segment.fontSize || 16,
      };

      if (segment.link) {
        return (
          <Text
            key={i}
            style={[style, { color: 'blue' }]}
            onPress={() => openURL(segment.link)}
          >
            {segment.text}
          </Text>
        );
      }

      return <Text key={i} style={style}>{segment.text}</Text>;
    })}
  </Text>
);

// All segments render as a SINGLE native text view
// with attributed/spannable string regions

6. Text Truncation

// numberOfLines controls truncation
<Text numberOfLines={2} ellipsizeMode="tail">
  This is a very long text that will be truncated after two lines
  with an ellipsis at the end of the second line...
</Text>

// ellipsizeMode options:
// "head"   — ...long text here
// "middle" — long...text here
// "tail"   — long text here... (default)
// "clip"   — long text he (no ellipsis, just clips)

// Platform behavior differences:
// iOS: All modes work correctly
// Android: "head" and "middle" only work with numberOfLines={1}
// Multi-line "head"/"middle" truncation on Android falls back to "tail"

// Detecting truncation (onTextLayout):
<Text
  numberOfLines={3}
  onTextLayout={(e) => {
    const { lines } = e.nativeEvent;
    const isTruncated = lines.length >= 3;
    // Can show "Read more" button if truncated
  }}
>
  {longText}
</Text>

7. Selectable Text

// Text is NOT selectable by default in React Native
<Text selectable={true}>
  This text can be selected and copied
</Text>

// iOS: selectable={true} uses UITextView instead of UILabel
// UITextView supports text selection natively
// UILabel does NOT support selection

// Android: selectable={true} sets textIsSelectable(true) on TextView

// Performance implication:
// UITextView is heavier than UILabel
// Only use selectable={true} when needed

// selectionColor (Android only):
<Text selectable={true} selectionColor="rgba(0, 100, 255, 0.3)">
  Custom selection highlight color
</Text>

// Copy-to-clipboard without selection UI:
import Clipboard from '@react-native-clipboard/clipboard';

<Text onLongPress={() => Clipboard.setString(text)}>
  Long press to copy
</Text>

8. Accessibility for Text

// Text is automatically accessible (read by screen readers)
<Text>Hello World</Text>
// VoiceOver/TalkBack reads: "Hello World"

// Nested text is read as a single string
<Text>
  Hello <Text style={{ fontWeight: 'bold' }}>World</Text>
</Text>
// VoiceOver reads: "Hello World" (single announcement)

// accessibilityRole for semantic meaning
<Text accessibilityRole="header">
  Section Title
</Text>
// VoiceOver: "Section Title, heading"

// Dynamic type / font scaling
<Text
  style={{ fontSize: 16 }}
  allowFontScaling={true}   // default: true
  maxFontSizeMultiplier={1.5} // cap scaling at 1.5x
>
  Scales with system font size preference
</Text>

// adjustsFontSizeToFit (iOS) — shrinks text to fit container
<Text
  adjustsFontSizeToFit
  numberOfLines={1}
  minimumFontScale={0.5}
>
  This text shrinks to fit on one line
</Text>

10. Architect Considerations

Staff-Level Mental Model

Text rendering is the most platform-divergent aspect of React Native. A Staff engineer must understand that: (1) Text measurement is synchronous and expensive — minimize dynamic text style changes. (2) Font loading must be gated — never render text before fonts are ready. (3) Platform differences in lineHeight, letterSpacing, and fontWeight require per-platform normalization in your design system. (4) Nested Text is a single native view — this is both a performance win (fewer views) and a constraint (no non-Text children). Build a typography system that abstracts these differences.

10.1 Typography System Design

// Design system typography tokens
const typography = {
  h1: {
    fontSize: 32,
    lineHeight: 40,
    fontWeight: '700' as const,
    fontFamily: Platform.select({
      ios: 'Inter-Bold',
      android: 'Inter-Bold',
    }),
    ...Platform.select({
      android: { includeFontPadding: false },
    }),
  },
  body: {
    fontSize: 16,
    lineHeight: 24,
    fontWeight: '400' as const,
    fontFamily: Platform.select({
      ios: 'Inter-Regular',
      android: 'Inter-Regular',
    }),
    ...Platform.select({
      android: { includeFontPadding: false },
    }),
  },
};

// Typography component that enforces consistency
const Typography = ({ variant = 'body', children, style, ...props }) => (
  <Text
    style={[typography[variant], style]}
    maxFontSizeMultiplier={1.5}
    {...props}
  >
    {children}
  </Text>
);
