Interview Q&A: Platform-Specific Development

55+ L6/Staff-level questions covering the Platform module, iOS/Android specifics, responsive design, accessibility, internationalization, and React Native for Web.

← React Native for Web

Platform Module

Q1: What is the difference between Platform.OS and Platform.select, and when would you use each?

Platform.OS returns a string ('ios', 'android', 'web') for conditional branching. Platform.select takes an object keyed by platform and returns the matching value. Use Platform.OS for imperative logic (if/else blocks, function calls). Use Platform.select for declarative value selection (styles, config objects). Platform.select also supports native key (matches iOS + Android) and default fallback, making it more expressive for multi-platform code.

Q2: How does Metro resolve platform-specific file extensions, and what is the resolution priority?

Metro resolves imports by appending platform extensions in priority order: .ios.tsx → .native.tsx → .tsx (for iOS builds). The full matrix is: exact platform extension × each source extension, then .native × each source extension, then bare source extensions. This is a build-time mechanism — only the matching file is included in the bundle. The .native extension matches any non-web platform, making it useful for shared mobile code in universal apps.

Q3: When should you use platform-specific files (.ios.js/.android.js) vs Platform.select?

Use platform files when >30% of the component logic differs, when wrapping different native modules, or when the UX pattern is fundamentally different (e.g., date picker). Use Platform.select for styling differences, small behavioral tweaks, and configuration values. Platform files produce smaller bundles (unused platform code is excluded) but are harder to maintain (changes must be synchronized across files). Platform.select keeps code in one file but includes all branches in the bundle.

Q4: What is PlatformColor and how does it enable automatic dark mode support?

PlatformColor references native platform color resources (UIColor semantic colors on iOS, @android:color resources on Android). These colors are resolved at render time by the native view manager, meaning they automatically update when the system theme changes (dark mode toggle) without re-rendering the React tree. On iOS, PlatformColor('label') maps to UIColor.label which is black in light mode and white in dark mode. DynamicColorIOS provides explicit light/dark values for custom colors.

Q5: How would you architect a platform abstraction layer for a large React Native codebase?

Create a services layer with interfaces defining platform capabilities (haptics, biometrics, storage, notifications). Implement each interface in platform-specific files (.ios.ts, .android.ts, .web.ts). Feature code imports from the service interface, never calling Platform.OS directly. This makes the codebase extensible — adding a new platform (Windows, macOS) only requires implementing the service interfaces. Enforce this pattern via ESLint rules that disallow direct Platform.OS usage in feature modules.

Q6: What is Platform.Version and why does it return different types on iOS vs Android?

Platform.Version returns a string on iOS (e.g., "16.4") matching the OS version format, and a number on Android representing the API level (e.g., 33 for Android 13). This reflects the native APIs: iOS uses semantic versioning strings, Android uses integer API levels. Always use explicit type coercion when comparing: parseInt(Platform.Version, 10) on iOS, direct numeric comparison on Android.

iOS-Specific

Q7: Compare the built-in SafeAreaView with react-native-safe-area-context.

Built-in SafeAreaView only works on iOS, only applies padding (not margin), and doesn't expose inset values. react-native-safe-area-context works cross-platform, provides useSafeAreaInsets() hook for custom layouts, supports selective edge application via edges prop, and measures actual device insets at runtime via native UIView.safeAreaInsets. Always prefer the library for production apps.

Q8: How do iOS keyboard events differ from Android, and how does this affect KeyboardAvoidingView?

iOS provides keyboardWillShow/keyboardWillHide (fires before animation with duration and easing curve) and keyboardDidShow/keyboardDidHide. Android only reliably fires Did events. This means iOS can synchronize layout animations with the keyboard, while Android adjustments appear after the keyboard is fully shown. KeyboardAvoidingView uses behavior="padding" on iOS (works with will events) and behavior="height" on Android.

Q9: How does the iOS permission model differ from Android's, and what are the implications for UX?

iOS shows the system permission dialog only once per permission — after denial, the app can never show it again (user must go to Settings). Android allows re-requesting until the user checks "Don't ask again." This means iOS apps must pre-educate users before triggering the system dialog (explain why the permission is needed). iOS also requires Info.plist usage description strings that appear in the dialog. iOS 14+ added App Tracking Transparency as a separate permission layer.

Q10: How would you integrate Dynamic Island / Live Activities from a React Native app?

Dynamic Island and Live Activities require native Swift code with WidgetKit — they cannot be implemented in JavaScript. The integration pattern: (1) Define ActivityAttributes in Swift. (2) Create a Widget extension target in Xcode. (3) Bridge start/update/end calls via a TurboModule or Expo Module. (4) Use APNs push tokens for server-driven updates. The React Native layer calls the native module to start/update activities, but the UI is defined in SwiftUI within the widget extension.

Q11: What App Store guidelines are most relevant for React Native apps?

Key guidelines: (1) 2.5.2 — OTA updates (CodePush) must not change app behavior significantly. (2) 4.7 — WebView-heavy apps may be rejected. (3) 5.1.1 — Privacy nutrition labels required for all data collection. (4) 2.5.6 — Background modes must be justified. (5) iOS 17+ requires a privacy manifest declaring API usage reasons. (6) Apps must follow HIG patterns for navigation and controls.

Android-Specific

Q12: Explain BackHandler's event propagation model and how it integrates with React Navigation.

BackHandler uses a LIFO (last-in-first-out) subscription model. The most recently registered handler gets the event first. If it returns true, the event is consumed. If false, it propagates to the previous handler. React Navigation registers a handler that calls navigation.goBack(). Custom handlers (modals, drawers) should register after navigation and return true to prevent navigation when they handle the back press. Android 13+ predictive back gestures require migrating to OnBackPressedCallback.

Q13: How do Android 13+ granular media permissions affect React Native apps?

Android 13 (API 33) split READ_EXTERNAL_STORAGE into READ_MEDIA_IMAGES, READ_MEDIA_VIDEO, and READ_MEDIA_AUDIO. Apps must request only the specific media types they need. Code must check Platform.Version >= 33 and request the appropriate granular permissions, falling back to READ_EXTERNAL_STORAGE for older versions. This also affects the react-native-permissions library configuration.

Q14: What are Android notification channels and why are they required?

Android 8.0+ (API 26) requires notification channels — user-controllable categories with importance, sound, vibration, and LED settings. Each notification must specify a channel ID. Users can independently control each channel in system settings. Apps should create channels at startup with meaningful names and descriptions. Common pattern: separate channels for messages (high importance), updates (low), and marketing (default). Libraries like @notifee/react-native provide the channel management API.

Q15: How does ProGuard/R8 affect React Native Android builds?

R8 performs code shrinking, obfuscation, and optimization. React Native requires keep rules because native modules are registered via reflection — R8 would strip them as "unused." Critical keep rules: all classes extending ReactContextBaseJavaModule, methods annotated with @ReactMethod, TurboModule classes, and Hermes-related classes. R8 full mode is more aggressive and can break RN if rules are incomplete. Always test release builds thoroughly.

Q16: How do you handle the Android 12+ splash screen API in React Native?

Android 12+ automatically shows a splash screen using the SplashScreen API. Configure it via values-v31/styles.xml with windowSplashScreenBackground, windowSplashScreenAnimatedIcon, and postSplashScreenTheme. Keep the splash visible until the app is ready using setKeepOnScreenCondition. Libraries like react-native-bootsplash bridge this to JS, allowing you to hide the splash after data loading completes.

Responsive Design

Q17: What is the difference between Dimensions.get('window') and Dimensions.get('screen')?

window returns the usable area (excludes status bar and navigation bar on Android). screen returns full physical screen dimensions. On iOS, they're nearly identical (both include the status bar area). On Android, screen.height = window.height + statusBar + navBar. Always use window for layout calculations. Use useWindowDimensions() hook instead of Dimensions.get() for reactive updates on orientation change or split-screen resize.

Q18: How does React Native's Flexbox differ from CSS Flexbox?

Key differences: (1) Default flexDirection is 'column' (CSS: 'row'). (2) Default flexShrink is 0 (CSS: 1). (3) Default alignContent is 'flex-start' (CSS: 'stretch'). (4) No position: static — only relative and absolute. (5) gap support added in RN 0.71+ via Yoga. (6) All dimensions are unitless (density-independent pixels). These differences reflect mobile-first defaults where vertical scrolling is primary.

Q19: How would you implement a responsive master-detail layout for tablets?

Use useWindowDimensions() to detect tablet width (>768dp) or Platform.isPad. On tablets, render a side-by-side layout with a fixed-width master list and flex:1 detail view. On phones, use a stack navigator. For React Navigation, conditionally render a drawer navigator (tablet) or stack navigator (phone). Handle orientation changes — landscape phone may qualify for master-detail. Use a custom useDeviceType() hook that combines width and platform detection.

Q20: How do you handle dynamic font scaling while preventing layout breakage?

Never disable font scaling globally (allowFontScaling={false}) — it breaks accessibility. Instead: (1) Use maxFontSizeMultiplier on containers where layout would break. (2) Design layouts to accommodate 2x text size. (3) Use numberOfLines and ellipsizeMode for constrained spaces. (4) Test at maximum font scale (200% on iOS). (5) Use PixelRatio.getFontScale() to adjust spacing proportionally. (6) For fixed-size UI (tab bars, toolbars), cap scaling at 1.3x.

Q21: How does percentage-based sizing work in React Native, and what are its limitations?

Percentage values (e.g., width: '50%') are relative to the parent's dimension. They work for width, height, margin, padding, and position offsets. Key limitation: the parent must have a defined dimension — a parent with no explicit height/flex makes percentage height resolve to 0. flex: 1 on the parent provides a defined dimension. Percentage-based sizing doesn't work for border radius, font size, or gap. For responsive font sizes, use a utility function that scales based on screen width.

Accessibility

Q22: How does React Native map accessibility props to native platform APIs?

accessible={true} maps to isAccessibilityElement (iOS) and importantForAccessibility="yes" (Android). accessibilityLabel maps to accessibilityLabel (iOS) and contentDescription (Android). accessibilityRole maps to accessibilityTraits bitmask (iOS) and className (Android, e.g., "android.widget.Button"). accessibilityState maps to traits (iOS) and stateDescription/checked/selected (Android). The bridge layer handles these mappings in the native view manager.

Q23: How do you implement focus management for screen readers in React Native?

Use AccessibilityInfo.setAccessibilityFocus(findNodeHandle(ref)) to programmatically move focus. Critical scenarios: (1) After form validation errors, focus the error message. (2) When a modal opens, focus the first element. (3) After content loads, focus the new content. Use accessibilityViewIsModal={true} (iOS) to trap focus within modals. On Android, use importantForAccessibility="no-hide-descendants" to hide background content from TalkBack.

Q24: What are accessibility actions and when should you use them?

Accessibility actions provide alternative interactions for gestures that screen reader users can't perform (swipe-to-delete, drag-to-reorder). Define them via accessibilityActions array and handle in onAccessibilityAction. Built-in actions: activate (default tap), increment/decrement (adjustable), magicTap (iOS two-finger double-tap for primary action), escape (iOS two-finger Z-gesture for dismiss). Custom actions appear in VoiceOver's Actions rotor and TalkBack's local context menu.

Q25: How do accessibilityLiveRegion and announceForAccessibility differ?

accessibilityLiveRegion is declarative — it automatically announces when the content of the marked element changes. Values: 'polite' (waits for idle), 'assertive' (interrupts). announceForAccessibility is imperative — you call it explicitly to make an announcement. Use live regions for content that updates frequently (counters, status). Use announcements for one-time events (toast messages, action confirmations). Live regions are more reliable on Android; announcements work better on iOS.

Q26: What WCAG criteria are most relevant for mobile apps, and how do you test compliance?

Key criteria: 1.1.1 (non-text content needs labels), 1.4.3 (4.5:1 contrast ratio), 1.4.4 (text resizable to 200%), 2.5.5 (44×44pt touch targets), 2.4.3 (logical focus order), 4.1.2 (name, role, value for all interactive elements). Test with: Xcode Accessibility Inspector, Android Accessibility Scanner, react-native-ama for runtime auditing, @testing-library/react-native for unit tests with accessibility queries, and manual testing with VoiceOver/TalkBack.

Q27: How do you make a swipeable list item accessible?

Swipe gestures are invisible to screen readers. Provide accessibility actions as alternatives: define accessibilityActions with actions like 'delete', 'archive', 'pin'. Handle them in onAccessibilityAction. Group the list item content with accessible={true} and provide a descriptive accessibilityLabel. On iOS, actions appear in VoiceOver's Actions rotor (swipe up/down). On Android, they appear in TalkBack's local context menu (swipe up then right).

Internationalization

Q28: How does I18nManager.forceRTL work and why does it require an app restart?

I18nManager.forceRTL(true) sets a native flag that controls the layout direction for the entire app. It requires a restart because: (1) Yoga layout engine reads the RTL flag at initialization. (2) All existing native views have cached layout directions. (3) The JavaScript context needs to reinitialize with the new direction. Use react-native-restart to programmatically restart. Persist the RTL preference so it's applied before the first render on next launch.

Q29: What are logical style properties and why are they important for RTL?

Logical properties (paddingStart, paddingEnd, marginStart, marginEnd, start, end) automatically flip in RTL layouts. paddingStart is paddingLeft in LTR and paddingRight in RTL. Using logical properties means your layouts automatically mirror without conditional code. Physical properties (paddingLeft, paddingRight) don't flip and should be used for elements that shouldn't mirror (media controls, progress bars).

Q30: How does Hermes handle Intl API support?

Hermes has built-in Intl support since RN 0.70+ (enabled via hermes.intl flag). It uses ICU data bundled with the engine, supporting Intl.DateTimeFormat, Intl.NumberFormat, Intl.Collator, and Intl.RelativeTimeFormat. The ICU data adds 2-6MB to the binary depending on included locales. For older versions or JSC, use @formatjs/intl-* polyfills. Hermes's Intl is more performant than polyfills since it's implemented in C++.

Q31: How do you handle pluralization across languages with different plural rules?

Use CLDR plural rules via i18next. English has 2 forms (one, other). Arabic has 6 (zero, one, two, few, many, other). Russian has 3 (one, few, many). i18next maps these using suffixed keys: item_one, item_other, etc. The count interpolation variable triggers plural selection. Never hardcode plural logic — always use the i18n library's built-in CLDR support. Test with Arabic and Russian to catch missing plural forms.

Q32: What considerations are needed for CJK (Chinese/Japanese/Korean) font handling?

CJK fonts are 5-20MB per weight (vs ~100KB for Latin). Strategies: (1) Use system fonts for body text — both iOS and Android have excellent CJK system fonts. (2) For custom fonts, use variable fonts to reduce weight count. (3) Consider on-demand font loading for CJK locales. (4) CJK text doesn't use spaces between words, affecting text wrapping. (5) Line height and letter spacing may need locale-specific adjustments. (6) Some CJK characters are wider than Latin, affecting fixed-width layouts.

React Native for Web

Q33: How does react-native-web convert StyleSheet to CSS?

react-native-web converts RN styles to atomic CSS classes. Each style property-value pair becomes a unique class (e.g., .r-flex-1 { flex: 1 }). Benefits: (1) CSS size grows logarithmically — duplicate styles across components are deduplicated. (2) No specificity conflicts — all classes have equal specificity. (3) Deterministic output — same style always produces same class. The atomic approach means a large app might only have a few hundred unique CSS rules despite thousands of styled components.

Q34: What are the main challenges of SSR with react-native-web and Next.js?

Challenges: (1) useWindowDimensions returns 0 during SSR — need fallback values and hydration-safe rendering. (2) Animations (Reanimated) don't work server-side — use CSS animations as web fallback. (3) Some RN libraries import native modules that crash on web — need .web.ts stubs. (4) StyleSheet generates CSS that must be extracted during SSR for proper hydration. (5) Platform.OS is 'web' on both server and client, but window APIs are only available client-side.

Q35: How does Solito bridge React Navigation and Next.js routing?

Solito provides universal hooks (useRouter, useLink, useParam) that delegate to React Navigation on native and Next.js Router on web. useLink returns props that render as <a href> on web (SEO-friendly) and trigger navigation.navigate() on native. useParam reads from React Navigation's route params on native and useRouter().query on web. This enables shared screen components with platform-appropriate navigation.

Q36: What is Tamagui's optimizing compiler and how does it improve performance?

Tamagui's compiler runs at build time and: (1) Extracts styles to static CSS on web (avoiding runtime style computation). (2) Flattens component trees by removing unnecessary wrapper Views. (3) Converts dynamic styles to static where possible (constant propagation). (4) On native, it pre-computes StyleSheet objects. The result: fewer React elements, less runtime work, and smaller CSS bundles. It analyzes the AST to determine which styles are static vs dynamic.

Q37: What is a realistic code sharing percentage for a universal RN app, and what typically can't be shared?

70-80% code sharing is realistic. Shared: business logic, state management, API layer, hooks, types, utilities, most screen layouts. Not shared: navigation configuration (stack vs URL routing), native module wrappers, platform-specific gestures, push notification handling, deep link configuration, some animations (Reanimated vs CSS), and storage implementations (Keychain vs cookies). The key is to share the "what" (business logic) and platform-specific the "how" (UI interactions).

Advanced / Cross-Cutting

Q38: How would you design a responsive design system for a React Native app targeting phones, tablets, and web?

Architecture: (1) Define breakpoints: sm (<480), md (<768), lg (<1024), xl (≥1024). (2) Create responsive hooks: useBreakpoint(), useDeviceType(), useOrientation(). (3) Build responsive primitives: Row (becomes column on small screens), Grid (adjusts columns), Container (max-width on web). (4) Design token scale that adjusts per breakpoint (spacing, typography). (5) Test on smallest device at max font scale — that's the hardest constraint. (6) For Tamagui: use responsive props ($sm, $md) for declarative breakpoints.

Q39: How do you handle platform-specific navigation patterns (iOS swipe-back, Android back button, web URL routing)?

Use React Navigation with platform-specific configuration: (1) iOS: enable gestureEnabled for swipe-back on stack navigators. (2) Android: BackHandler integration (React Navigation handles this automatically). (3) Web: use linking config to map screens to URLs. For universal apps, Solito or Expo Router provides a unified API. Key insight: navigation structure may differ per platform — tabs might be bottom on mobile and sidebar on web. Use conditional navigator selection based on platform/breakpoint.

Q40: How do you test accessibility in a CI/CD pipeline for React Native?

Layers: (1) Unit tests with @testing-library/react-native using accessibility queries (getByRole, getByLabelText). (2) ESLint plugin for accessibility (eslint-plugin-react-native-a11y) — catches missing labels, roles. (3) Snapshot tests that verify accessibility props are present. (4) Detox E2E tests that interact via accessibility IDs. (5) Automated tools: axe-core for web builds, Accessibility Scanner for Android (via Detox artifacts). (6) Manual testing checklist for VoiceOver/TalkBack before each release.

Q41: How do you handle deep linking across iOS (Universal Links), Android (App Links), and web?

iOS: Configure apple-app-site-association file on your domain, add Associated Domains entitlement. Android: Add intent-filter with autoVerify="true" in AndroidManifest, host assetlinks.json. Web: Standard URL routing. In React Native, use React Navigation's linking config to map URLs to screens. Use Linking.getInitialURL() for cold starts and Linking.addEventListener('url') for warm starts. Expo Router handles this declaratively via file-based routing.

Q42: What is the impact of I18nManager.forceRTL on animations and gesture handlers?

RTL affects: (1) Flexbox row direction reverses. (2) Absolute positioning with start/end flips. (3) Gesture handler directions may need adjustment — a "swipe right to go back" becomes "swipe left" in RTL. (4) Animated transforms using translateX may need sign inversion. (5) React Navigation's swipe-back gesture automatically adjusts. Best practice: use logical directions in animations and test all animations in RTL mode.

Q43: How do you handle platform-specific secure storage (Keychain, Keystore, cookies)?

Create a SecureStorage interface with platform implementations: iOS uses react-native-keychain (wraps Keychain Services), Android uses react-native-keychain (wraps Android Keystore with EncryptedSharedPreferences fallback), Web uses httpOnly secure cookies or Web Crypto API. Use platform-specific files: secureStorage.ios.ts, secureStorage.android.ts, secureStorage.web.ts. Feature code imports from the interface, never platform-specific implementations.

Q44: How would you implement a translation key validation system in CI?

Steps: (1) Parse all source files for t('key') calls using AST analysis. (2) Compare extracted keys against each locale's JSON file. (3) Report: missing translations per locale, unused keys (in JSON but not in code), keys with inconsistent interpolation variables. (4) Fail CI on missing translations for production locales. (5) Warn on unused keys (may indicate dead code). (6) Validate JSON structure (no duplicate keys, valid nesting). Tools: i18next-parser for extraction, custom scripts for validation.

Q45: How do you handle different date/time formats across locales without bloating the bundle?

Use Intl.DateTimeFormat (built into Hermes) which uses ICU data for locale-aware formatting. Avoid bundling moment.js or date-fns locale data. For Hermes, ICU data is already in the binary. For specific formatting needs, use Intl.DateTimeFormat options (dateStyle, timeStyle) rather than custom format strings. For relative time ("2 hours ago"), use Intl.RelativeTimeFormat. This approach adds zero JS bundle size since it's native to the engine.

Q46: What are the performance implications of supporting multiple platforms in a single codebase?

Potential issues: (1) Platform.select branches that aren't tree-shaken increase bundle size. (2) Platform-specific files are better for bundle size (only matching file included). (3) Universal libraries (Tamagui, Solito) add abstraction overhead. (4) Web builds may include unused native-specific code if not properly split. Mitigations: use platform file extensions for large divergences, configure Metro/webpack to exclude platform-irrelevant code, use Tamagui's compiler for static extraction, and monitor per-platform bundle sizes independently.

Q47: How do you handle accessibility for custom gesture-based interactions?

Every gesture must have an accessible alternative: (1) Swipe actions → accessibilityActions. (2) Pinch-to-zoom → zoom buttons or adjustable role with increment/decrement. (3) Long press → context menu with accessibilityActions. (4) Drag-to-reorder → "Move up"/"Move down" actions. (5) Pull-to-refresh → automatic (ScrollView handles it). (6) Double-tap → already the activation gesture for screen readers. Detect screen reader with AccessibilityInfo.isScreenReaderEnabled() and provide alternative UI when active.

Q48: How would you handle a scenario where the app needs to support both LTR and RTL content simultaneously?

Use writingDirection: 'auto' on Text components to let the Unicode Bidi algorithm determine direction per paragraph. For mixed-direction layouts: (1) Set the app's primary direction via I18nManager. (2) Use writingDirection on individual text elements. (3) For chat apps, detect message language and set direction per bubble. (4) Use Unicode control characters (LRM, RLM) for inline direction overrides. (5) Test with mixed Arabic/English content to verify alignment.

Q49: Compare NativeWind, Tamagui, and plain StyleSheet for a universal React Native app.

StyleSheet: Zero dependencies, maximum control, no responsive props, no design tokens, manual platform branching. NativeWind: Tailwind CSS syntax, compiles to StyleSheet on native and CSS on web, great for Tailwind-familiar teams, limited component library. Tamagui: Full UI kit, optimizing compiler, design tokens, responsive props ($sm, $md), theme support, best for universal apps but steeper learning curve. For Staff-level: Tamagui for universal apps needing a design system, NativeWind for rapid development, StyleSheet for maximum performance control.

Q50: How do you ensure consistent behavior across iOS and Android for form inputs?

Key differences to normalize: (1) KeyboardAvoidingView behavior differs (padding vs height). (2) returnKeyType labels differ. (3) Android TextInput has default padding/underline. (4) iOS auto-correct/auto-capitalize behavior differs. (5) Keyboard dismiss behavior differs. Create a FormInput wrapper that normalizes: consistent padding, platform-appropriate keyboard avoidance, unified error display, consistent focus/blur behavior, and platform-appropriate keyboard types.

Q51: How do you handle platform-specific animations (iOS spring physics vs Android material motion)?

iOS users expect spring-based animations (UIKit springs). Android users expect Material motion (standard easing curves). Use Reanimated's withSpring for iOS and withTiming with Material easing for Android. Create a platform animation utility: platformTransition() that returns spring config on iOS and timing config on Android. For shared animations, use spring physics everywhere — it feels natural on both platforms and is more forgiving of timing differences.

Q52: What is the strategy for handling platform-specific error messages and user-facing text?

Separate technical errors from user-facing messages. Create an error mapping layer that: (1) Maps native error codes to i18n keys. (2) Handles platform-specific error formats (NSError vs Java Exception). (3) Provides platform-appropriate recovery actions (e.g., "Open Settings" on iOS for permission denial vs "Grant Permission" on Android). (4) Uses accessibilityLiveRegion="assertive" to announce errors to screen readers. Store error message templates in translation files, not in code.

Q53: How do you optimize a React Native app for foldable devices?

Foldable considerations: (1) Use useWindowDimensions() — dimensions change when folding/unfolding. (2) Detect fold state via react-native-device-info or Jetpack WindowManager bridge. (3) Adapt layout: show master-detail when unfolded, stack when folded. (4) Handle the hinge/fold area — don't place interactive elements across it. (5) Test with Samsung Fold emulator. (6) Consider table-top mode (half-folded) for video playback. The key is treating foldables as dynamic screen size changes, not a separate device category.

Q54: How would you implement a platform-specific biometric authentication flow?

Use react-native-biometrics or expo-local-authentication. iOS: Face ID or Touch ID — requires NSFaceIDUsageDescription in Info.plist. Android: Fingerprint, face, or iris via BiometricPrompt API. Create a BiometricService interface with: isAvailable(), authenticate(prompt), getSupportedTypes(). Handle fallbacks: device passcode if biometrics unavailable, app PIN if device security not set. Store authentication tokens in platform-specific secure storage (Keychain/Keystore) gated by biometric access control.

Q55: How do you handle app-level theming (dark mode) across iOS, Android, and web?

Architecture: (1) Detect system theme via useColorScheme() hook (works on all platforms). (2) Allow user override stored in preferences. (3) Use PlatformColor for native semantic colors that auto-adapt. (4) For custom colors, create a theme context with light/dark palettes. (5) On web, respect prefers-color-scheme media query. (6) Use Appearance.addChangeListener for real-time theme changes. (7) For Tamagui: use built-in theme tokens. Key: never hardcode colors — always reference theme tokens.
