Android-Specific Considerations

Android's open ecosystem introduces unique challenges — hardware back button handling, runtime permission flows, notification channels, ProGuard minification, and multi-window support. Staff engineers must understand the Android lifecycle primitives that React Native abstracts and where those abstractions break down.

1. Back Button Handling

Android's hardware/gesture back button is a fundamental navigation primitive. React Native exposes it via BackHandler, which uses a subscription-based event system with priority ordering.

import { BackHandler } from 'react-native';

// Basic usage — prevent default back behavior
useEffect(() => {
  const handler = BackHandler.addEventListener(
    'hardwareBackPress',
    () => {
      // Return true = event consumed (prevent default)
      // Return false = pass to next handler
      if (isModalOpen) {
        closeModal();
        return true;
      }
      return false;
    }
  );
  return () => handler.remove();
}, [isModalOpen]);

// Exit app programmatically
BackHandler.exitApp();

1.1 Handler Priority Chain

BackHandler Event Propagation ═════════════════════════════ Hardware Back Press │ ↓ ┌─────────────────────┐ │ Last registered │ ← Highest priority (LIFO) │ handler │ Return true → consumed │ (e.g., Modal) │ Return false → propagate └──────────┬──────────┘ ↓ ┌─────────────────────┐ │ Previous handler │ │ (e.g., Screen) │ └──────────┬──────────┘ ↓ ┌─────────────────────┐ │ Navigation handler │ ← React Navigation registers here │ (goBack) │ └──────────┬──────────┘ ↓ ┌─────────────────────┐ │ Default behavior │ ← Exits app / finishes Activity └─────────────────────┘

1.2 Custom Hook for Back Handler

const useBackHandler = (handler: () => boolean) => {
  useEffect(() => {
    const sub = BackHandler.addEventListener(
      'hardwareBackPress', handler
    );
    return () => sub.remove();
  }, [handler]);
};

// Double-tap to exit pattern
const useDoubleBackExit = () => {
  const lastBackRef = useRef(0);

  useBackHandler(useCallback(() => {
    const now = Date.now();
    if (now - lastBackRef.current < 2000) {
      BackHandler.exitApp();
      return true;
    }
    lastBackRef.current = now;
    ToastAndroid.show('Press back again to exit', ToastAndroid.SHORT);
    return true;
  }, []));
};
Android 13+ Predictive Back

Android 13 introduced predictive back gestures that show a preview of the destination before the back action completes. React Native 0.73+ supports this via android:enableOnBackInvokedCallback="true" in AndroidManifest.xml. This requires migrating from BackHandler to the new OnBackPressedCallback pattern in native modules.

2. Android Permissions Model

import { PermissionsAndroid, Platform } from 'react-native';

const requestCameraPermission = async () => {
  if (Platform.OS !== 'android') return true;

  const granted = await PermissionsAndroid.request(
    PermissionsAndroid.PERMISSIONS.CAMERA,
    {
      title: 'Camera Permission',
      message: 'App needs camera access to scan documents',
      buttonNeutral: 'Ask Me Later',
      buttonNegative: 'Cancel',
      buttonPositive: 'OK',
    }
  );

  return granted === PermissionsAndroid.RESULTS.GRANTED;
};

// Multiple permissions at once
const requestMultiplePermissions = async () => {
  const results = await PermissionsAndroid.requestMultiple([
    PermissionsAndroid.PERMISSIONS.CAMERA,
    PermissionsAndroid.PERMISSIONS.READ_EXTERNAL_STORAGE,
    PermissionsAndroid.PERMISSIONS.ACCESS_FINE_LOCATION,
  ]);
  // results is { [permission]: 'granted' | 'denied' | 'never_ask_again' }
};
Android Permission Lifecycle ════════════════════════════ ┌───────────────┐ │ Not Requested │ └───────┬───────┘ │ request() ↓ ┌───────────────┐ Allow ┌─────────┐ │ System Dialog │──────────────→│ GRANTED │ └───────┬───────┘ └─────────┘ │ Deny ↓ ┌───────────────┐ │ DENIED │ ← Can request again └───────┬───────┘ │ request() + Deny again ↓ ┌───────────────────┐ │ NEVER_ASK_AGAIN │ ← "Don't ask again" checked └───────────────────┘ Must open app Settings │ ↓ Linking.openSettings() Android 11+: Auto-reset permissions for unused apps Android 12+: Approximate vs precise location split Android 13+: Granular media permissions (images/video/audio)

2.1 Android 13+ Photo/Media Permissions

const requestMediaPermission = async () => {
  if (Platform.Version >= 33) {
    // Android 13+: granular media permissions
    return PermissionsAndroid.requestMultiple([
      PermissionsAndroid.PERMISSIONS.READ_MEDIA_IMAGES,
      PermissionsAndroid.PERMISSIONS.READ_MEDIA_VIDEO,
    ]);
  } else {
    // Android 12 and below
    return PermissionsAndroid.request(
      PermissionsAndroid.PERMISSIONS.READ_EXTERNAL_STORAGE
    );
  }
};

5. Notification Channels

Android 8.0+ (API 26) requires notification channels. Each channel has user-controllable importance, sound, vibration, and LED settings.

// Using @notifee/react-native for notification channels
import notifee, { AndroidImportance } from '@notifee/react-native';

const setupChannels = async () => {
  await notifee.createChannel({
    id: 'messages',
    name: 'Messages',
    description: 'New message notifications',
    importance: AndroidImportance.HIGH,
    sound: 'message_sound',
    vibration: true,
    vibrationPattern: [300, 500],
    lights: true,
    lightColor: '#FF0000',
  });

  await notifee.createChannel({
    id: 'updates',
    name: 'App Updates',
    importance: AndroidImportance.LOW,
  });

  // Channel groups
  await notifee.createChannelGroup({
    id: 'social',
    name: 'Social',
  });
};

7. Split-Screen & Multi-Window

// AndroidManifest.xml configuration
// <activity
//   android:resizeableActivity="true"
//   android:configChanges="keyboard|keyboardHidden|orientation|
//     screenSize|screenLayout|smallestScreenSize|uiMode">

// Detecting multi-window mode in React Native
import { useWindowDimensions } from 'react-native';

const AdaptiveLayout = () => {
  const { width, height } = useWindowDimensions();
  const isCompact = width < 400;

  return isCompact
    ? <CompactLayout />
    : <FullLayout />;
};
Warning

Multi-window mode triggers onConfigurationChanged which can cause React Native to re-measure layouts. Ensure your app handles dimension changes gracefully via useWindowDimensions rather than caching Dimensions.get() at startup. The latter returns stale values after a split-screen resize.

10. ProGuard/R8 Configuration

R8 (ProGuard's successor) performs code shrinking, obfuscation, and optimization. React Native requires specific keep rules to prevent stripping of reflection-based native module registration.

# proguard-rules.pro for React Native

# React Native core
-keep class com.facebook.react.** { *; }
-keep class com.facebook.hermes.** { *; }
-keep class com.facebook.jni.** { *; }

# Keep native module classes (registered via reflection)
-keep class * extends com.facebook.react.bridge.ReactContextBaseJavaModule { *; }
-keep class * extends com.facebook.react.bridge.NativeModule { *; }
-keep class * extends com.facebook.react.uimanager.ViewManager { *; }

# TurboModules (New Architecture)
-keep class * extends com.facebook.react.turbomodule.core.TurboModule { *; }

# Keep @ReactMethod annotated methods
-keepclassmembers class * {
  @com.facebook.react.bridge.ReactMethod <methods>;
}

# Hermes bytecode
-keep class com.facebook.hermes.unicode.** { *; }

# OkHttp (used by RN networking)
-dontwarn okhttp3.**
-dontwarn okio.**
-keep class okhttp3.** { *; }
Architect Insight

R8 full mode (android.enableR8.fullMode=true) is more aggressive and can break React Native if keep rules are incomplete. Always test release builds thoroughly. Common symptoms of missing keep rules: ClassNotFoundException for native modules, missing methods in TurboModules, and crash on app startup. Use -printusage to audit what R8 removes.

R8 FeatureImpact on RNConfiguration
Code shrinkingRemoves unused Java/Kotlin codeKeep rules for native modules
ObfuscationRenames classes/methodsKeep rules for reflection-based access
OptimizationInlines methods, removes dead codeGenerally safe for RN
