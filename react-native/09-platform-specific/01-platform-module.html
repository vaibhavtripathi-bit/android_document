Platform Module

The Platform module is React Native's primary mechanism for runtime platform detection and conditional code execution. Understanding its internals — from module resolution algorithms to PlatformColor bridging — separates Staff engineers from those who merely use Platform.OS.

1. Platform.OS and Platform.Version

Platform.OS returns a string identifying the current platform at runtime. This value is determined during the native module initialization phase and is immutable for the lifetime of the JS context.

import { Platform } from 'react-native';

// Platform.OS values:
// 'ios'     — iPhone, iPad, iPod touch
// 'android' — Android phones, tablets, TV
// 'web'     — react-native-web
// 'windows' — react-native-windows
// 'macos'   — react-native-macos

if (Platform.OS === 'ios') {
  // iOS-specific code
} else if (Platform.OS === 'android') {
  // Android-specific code
}

Platform.Version provides the OS version. On iOS it returns a string (e.g., "16.4"), on Android it returns the API level as a number (e.g., 33).

// iOS: string comparison
if (Platform.OS === 'ios' && parseInt(Platform.Version, 10) >= 16) {
  // iOS 16+ features
}

// Android: numeric comparison (API level)
if (Platform.OS === 'android' && Platform.Version >= 33) {
  // Android 13+ (API 33) features
}
Warning

Platform.Version returns different types per platform — a string on iOS and a number on Android. TypeScript definitions reflect this via conditional types, but runtime code must handle both. Always use explicit type coercion when comparing.

2. Platform.select

Platform.select is a utility that picks a value from a map keyed by platform name. It supports a default fallback and resolves at runtime.

const Component = Platform.select({
  ios: () => require('./IOSComponent'),
  android: () => require('./AndroidComponent'),
  default: () => require('./DefaultComponent'),
})();

// For simple values
const topPadding = Platform.select({
  ios: 44,
  android: 0,
  default: 0,
});

// For style objects
const styles = StyleSheet.create({
  container: {
    ...Platform.select({
      ios: { shadowColor: '#000', shadowOffset: { width: 0, height: 2 } },
      android: { elevation: 4 },
    }),
  },
});

2.1 Implementation Detail

// Simplified Platform.select source
static select<T>(specifics: {
  ios?: T, android?: T, native?: T, default?: T, web?: T
}): T {
  if (specifics[Platform.OS] !== undefined) {
    return specifics[Platform.OS];
  }
  if (specifics.native !== undefined && Platform.OS !== 'web') {
    return specifics.native;
  }
  return specifics.default;
}
Key Detail

The native key matches both iOS and Android (any non-web platform). Resolution order: exact platform → native (if non-web) → default. This is critical for react-native-web projects where you need mobile-vs-web branching.

4. Platform-Specific File Extensions

Metro bundler resolves platform-specific files via a file extension priority system. This is a build-time mechanism — the bundler includes only the correct platform file in the final bundle.

// File structure:
// components/
//   Button.ios.js      ← bundled for iOS
//   Button.android.js  ← bundled for Android
//   Button.native.js   ← fallback for any native platform
//   Button.web.js      ← bundled for web (react-native-web)
//   Button.js          ← universal fallback

// Import is platform-agnostic:
import Button from './components/Button';
// Metro resolves to the correct file at bundle time

4.1 Resolution Priority

Metro File Resolution (for iOS build) ══════════════════════════════════════ import './Button' 1. Button.ios.js ← highest priority (exact platform) 2. Button.native.js ← native fallback (iOS + Android) 3. Button.js ← universal fallback 4. Button/index.ios.js ← directory with platform index 5. Button/index.js ← directory with universal index For Android build: .android.js → .native.js → .js For Web build: .web.js → .js (no .native.js!)

4.2 TypeScript with Platform Extensions

// To get proper types, create a shared type file:
// Button.d.ts
import { ComponentType } from 'react';

5. Module Resolution Algorithm

Metro's resolver uses a configurable list of platform extensions. The resolution algorithm is deterministic and follows a strict priority order.

// metro.config.js — customizing resolution
module.exports = {
  resolver: {
    // Default platform extensions
    platforms: ['ios', 'android', 'native', 'web'],

    // Source extensions (order matters for resolution)
    sourceExts: ['tsx', 'ts', 'jsx', 'js', 'json'],

    // Custom platform for out-of-tree platforms
    // e.g., react-native-windows adds 'windows'
    platforms: ['ios', 'android', 'native', 'windows'],
  },
};
Full Resolution Matrix ══════════════════════ import './Foo' on iOS with sourceExts: ['tsx', 'ts', 'js'] Attempt 1: Foo.ios.tsx Attempt 2: Foo.ios.ts Attempt 3: Foo.ios.js Attempt 4: Foo.native.tsx Attempt 5: Foo.native.ts Attempt 6: Foo.native.js Attempt 7: Foo.tsx Attempt 8: Foo.ts Attempt 9: Foo.js Attempt 10: Foo/index.ios.tsx ...continues with directory index variants... Total attempts = platforms.length × sourceExts.length × 2 (file + dir)
Architect Insight

Platform file extensions create implicit code splitting at the bundler level. Each platform bundle only includes its own files, reducing bundle size. However, this means platform files are never tree-shaken — they're entirely included or excluded. For small differences, Platform.select inside a single file may produce smaller bundles since the unused branch can be dead-code eliminated by Hermes.

6. PlatformColor

PlatformColor references native platform color resources, enabling your app to respect system themes (dark mode, high contrast) without manual color management.

import { PlatformColor, Platform } from 'react-native';

// iOS: references UIColor semantic colors
const iosStyles = {
  color: PlatformColor('label'),                  // UIColor.label
  backgroundColor: PlatformColor('systemBackground'), // UIColor.systemBackground
  tintColor: PlatformColor('systemBlue'),           // UIColor.systemBlue
};

// Android: references @android:color or @color resources
const androidStyles = {
  color: PlatformColor('@android:color/primary_text_dark'),
  backgroundColor: PlatformColor('@android:color/background_light'),
  tintColor: PlatformColor('?attr/colorPrimary'),
};

// Cross-platform usage with Platform.select
const styles = StyleSheet.create({
  text: {
    color: Platform.select({
      ios: PlatformColor('label'),
      android: PlatformColor('@android:color/primary_text_dark'),
      default: '#000000',
    }),
  },
});
How PlatformColor Works

PlatformColor creates a special object with { semantic: [...colorNames] } on iOS or { resource_paths: [...paths] } on Android. The native view manager resolves these references to actual color values at render time, meaning colors update automatically when the system theme changes (dark mode toggle) without re-rendering the React tree.

7. Conditional Platform Code Patterns

7.1 Inline Conditional

const MyComponent = () => (
  <View style={{ paddingTop: Platform.OS === 'ios' ? 44 : 0 }}>
    {Platform.OS === 'ios' && <StatusBar barStyle="dark-content" />}
    <Text>Content</Text>
  </View>
);

7.2 Platform.select for Objects

const shadow = Platform.select({
  ios: {
    shadowColor: '#000',
    shadowOffset: { width: 0, height: 2 },
    shadowOpacity: 0.1,
    shadowRadius: 8,
  },
  android: {
    elevation: 4,
  },
  default: {},
});

7.3 Factory Pattern

// Platform-specific module factory
const createStorageAdapter = () => {
  if (Platform.OS === 'ios') {
    return require('./storage/KeychainAdapter');
  }
  if (Platform.OS === 'android') {
    return require('./storage/EncryptedSharedPrefsAdapter');
  }
  return require('./storage/AsyncStorageAdapter');
};

7.4 Higher-Order Component Pattern

const withPlatformBehavior = (
  IOSComponent: ComponentType,
  AndroidComponent: ComponentType,
) => {
  return Platform.select({
    ios: IOSComponent,
    android: AndroidComponent,
  });
};

const NativeButton = withPlatformBehavior(IOSButton, AndroidButton);

9. Platform Files vs Platform.select

CriterionPlatform Files (.ios.js)Platform.select
Bundle sizeOnly platform file includedAll branches included (dead code may remain)
Code divergenceEntire file can differSmall inline differences
Type safetyRequires shared .d.tsInline, fully typed
RefactoringMust update multiple filesSingle file to update
TestingTest each file separatelyMock Platform.OS
Code reviewHarder to see differencesDifferences visible inline
Native modulesIdeal for wrapping different native APIsNot suitable
Architect Insight

Decision framework: Use platform files when >30% of the component logic differs between platforms, when wrapping platform-specific native modules, or when the component has fundamentally different UX patterns (e.g., date picker). Use Platform.select for styling differences, small behavioral tweaks, and configuration values. In a large codebase, establish a team convention and enforce it via ESLint rules.

Decision Tree: Platform Files vs Platform.select ═════════════════════════════════════════════════ Is the difference > 30% of the file? ├── YES → Use platform files (.ios.js / .android.js) │ ├── Different native module wrappers │ ├── Different component hierarchies │ └── Different third-party dependencies └── NO → Is it a style-only difference? ├── YES → Platform.select in StyleSheet └── NO → Is it a single value/config? ├── YES → Platform.select inline └── NO → Platform.select with object map

10. Architect Considerations

Architect Insight

Platform abstraction layer: In large codebases, create a platform abstraction layer that encapsulates all platform-specific decisions behind a unified API. Feature code should never directly call Platform.OS — instead, it should use platform services that internally handle branching. This makes platform support extensible (adding web, Windows) without modifying feature code.

// Platform abstraction layer pattern
// services/platform/index.ts
export interface PlatformServices {
  haptics: HapticsService;
  biometrics: BiometricsService;
  storage: SecureStorageService;
  notifications: NotificationService;
}

// services/platform/platform.ios.ts
export const platformServices: PlatformServices = {
  haptics: new IOSHapticsService(),
  biometrics: new FaceIDService(),
  storage: new KeychainService(),
  notifications: new APNSService(),
};

// services/platform/platform.android.ts
export const platformServices: PlatformServices = {
  haptics: new AndroidVibrationService(),
  biometrics: new FingerprintService(),
  storage: new EncryptedSharedPrefsService(),
  notifications: new FCMService(),
};

// Feature code — no Platform.OS checks needed
import { platformServices } from '../services/platform';
await platformServices.biometrics.authenticate();
