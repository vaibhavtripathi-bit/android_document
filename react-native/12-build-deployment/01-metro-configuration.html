1. Metro Architecture Overview

Metro is a JavaScript bundler specifically designed for React Native. Unlike general-purpose bundlers (webpack, Rollup), Metro is optimized for fast incremental builds and hot module replacement in a mobile development context. It processes modules through three distinct phases: Resolution, Transformation, and Serialization.

Metro Bundler Pipeline ══════════════════════ Entry Point (index.js) │ ▼ ┌─────────────────────────────────────────────────┐ │ 1. RESOLUTION PHASE │ │ ┌───────────────────────────────────────────┐ │ │ │ Resolver │ │ │ │ • Resolve import/require statements │ │ │ │ • Apply platform extensions (.ios/.android)│ │ │ │ • Handle node_modules resolution │ │ │ │ • Process asset references │ │ │ │ • Build complete dependency graph │ │ │ └───────────────────────────────────────────┘ │ └──────────────────────┬──────────────────────────┘ │ Dependency Graph ▼ ┌─────────────────────────────────────────────────┐ │ 2. TRANSFORMATION PHASE │ │ ┌───────────────────────────────────────────┐ │ │ │ Transformer (per module, parallelized) │ │ │ │ • Parse AST (Babel / Hermes parser) │ │ │ │ • Apply Babel transforms │ │ │ │ • Minify (Terser / Hermes bytecode) │ │ │ │ • Extract dependencies from AST │ │ │ │ • Generate per-module source maps │ │ │ └───────────────────────────────────────────┘ │ └──────────────────────┬──────────────────────────┘ │ Transformed Modules ▼ ┌─────────────────────────────────────────────────┐ │ 3. SERIALIZATION PHASE │ │ ┌───────────────────────────────────────────┐ │ │ │ Serializer │ │ │ │ • Concatenate modules into bundle │ │ │ │ • Generate module registry (__d calls) │ │ │ │ • Produce final source map │ │ │ │ • Apply bundle splitting (if configured) │ │ │ │ • Output: bundle.js + bundle.js.map │ │ │ └───────────────────────────────────────────┘ │ └─────────────────────────────────────────────────┘
Architect Insight

Metro's three-phase architecture is deliberately simple compared to webpack's plugin system. This simplicity is a feature — it makes Metro fast and predictable for the mobile use case. However, it also means customization is more constrained. Understanding which phase to customize for a given requirement is a key Staff-level skill.

2. metro.config.js Structure

The Metro configuration file exports an object (or async function returning one) that is merged with Metro's defaults via mergeConfig. Since React Native 0.72+, the recommended pattern uses @react-native/metro-config as the base.

const { getDefaultConfig, mergeConfig } = require('@react-native/metro-config');
const path = require('path');

const defaultConfig = getDefaultConfig(__dirname);

const config = {
  // Project root — where Metro starts looking for files
  projectRoot: __dirname,

  // Additional roots to watch (monorepo packages, shared libs)
  watchFolders: [
    path.resolve(__dirname, '../../packages/shared'),
    path.resolve(__dirname, '../../node_modules'),
  ],

  resolver: { /* ... */ },
  transformer: { /* ... */ },
  serializer: { /* ... */ },
  watcher: { /* ... */ },
  server: { /* ... */ },
};

module.exports = mergeConfig(defaultConfig, config);
Common Pitfall

Never spread or directly modify defaultConfig. Always use mergeConfig — it performs a deep merge that correctly handles arrays and nested objects. Spreading will overwrite default arrays (like sourceExts) instead of extending them, breaking resolution of standard file types.

3. Resolver Configuration

The resolver determines how import and require statements map to files on disk. It handles platform extensions, custom file extensions, module aliasing, and blocklisting.

const config = {
  resolver: {
    // File extensions to resolve (order matters — first match wins)
    sourceExts: [...defaultConfig.resolver.sourceExts, 'cjs', 'mjs', 'svg'],

    // Asset extensions (copied, not transformed)
    assetExts: defaultConfig.resolver.assetExts.filter(
      ext => ext !== 'svg'  // Handle SVG as source, not asset
    ),

    // Platform-specific extensions (resolved in order)
    platforms: ['ios', 'android', 'native', 'web'],

    // Block specific modules from being resolved
    blockList: [
      new RegExp(`.*/__tests__/.*`),
      new RegExp(`.*\\.test\\.(js|ts|tsx)$`),
      new RegExp(`.*node_modules/.*/node_modules/react-native/.*`),
    ],

    // Node.js core module polyfills (for libraries expecting Node APIs)
    extraNodeModules: {
      crypto: require.resolve('react-native-quick-crypto'),
      stream: require.resolve('readable-stream'),
      buffer: require.resolve('@craftzdog/react-native-buffer'),
    },

    // Resolve symlinks to their real paths (important for monorepos)
    unstable_enableSymlinks: true,

    // Enable package.json "exports" field resolution
    unstable_enablePackageExports: true,

    // Custom conditions for "exports" field
    unstable_conditionNames: ['react-native', 'import', 'require'],

    // Custom resolver function — ultimate escape hatch
    resolveRequest(context, moduleName, platform) {
      if (moduleName === 'my-legacy-module') {
        return {
          filePath: path.resolve(__dirname, 'src/compat/legacy.js'),
          type: 'sourceFile',
        };
      }
      return context.resolveRequest(context, moduleName, platform);
    },

    // Node modules search paths (for monorepos with hoisted deps)
    nodeModulesPaths: [
      path.resolve(__dirname, 'node_modules'),
      path.resolve(__dirname, '../../node_modules'),
    ],
  },
};

3.1 Platform Resolution Order

When resolving import './Button' on iOS, Metro checks files in this order:

Platform Resolution Order (iOS) ═══════════════════════════════ import './Button' │ ├──→ Button.ios.tsx ← Platform-specific (highest priority) ├──→ Button.ios.ts ├──→ Button.ios.jsx ├──→ Button.ios.js ├──→ Button.native.tsx ← Shared native (iOS + Android) ├──→ Button.native.ts ├──→ Button.native.jsx ├──→ Button.native.js ├──→ Button.tsx ← Universal fallback ├──→ Button.ts ├──→ Button.jsx ├──→ Button.js ├──→ Button/index.tsx ← Directory index └──→ Button/index.js
Package Exports Support

Metro 0.76+ supports the exports field in package.json via unstable_enablePackageExports. This enables libraries to define platform-specific entry points using conditions like "react-native", "import", and "default". Enable this for modern library compatibility, but test thoroughly — some older libraries have incorrect exports configurations.

4. Transformer Options

The transformer converts individual modules from source code to a format Metro can serialize. It handles Babel transformation, minification, and can be customized to process non-standard file types.

const config = {
  transformer: {
    // Babel transformer configuration
    babelTransformerPath: require.resolve(
      'react-native-svg-transformer'
    ),

    // Pass options to the Babel transformer
    getTransformOptions: async () => ({
      transform: {
        experimentalImportSupport: false,
        inlineRequires: true,  // Defer require() until first use
        nonInlinedRequires: [
          'React',
          'react',
          'react-native',
          'react/jsx-runtime',
        ],
      },
    }),

    // Minification (production builds)
    minifierPath: 'metro-minify-terser',
    minifierConfig: {
      compress: {
        drop_console: true,
        dead_code: true,
        global_defs: { __DEV__: false },
      },
      mangle: { toplevel: true },
    },

    // Enable Hermes bytecode compilation
    hermesParser: true,

    // Async require support (for lazy loading)
    asyncRequireModulePath: 'metro-runtime/src/modules/asyncRequire',

    // Number of worker threads for transformation
    maxWorkers: 4,

    // Enable CSS support (experimental in RN 0.73+)
    unstable_allowRequireContext: true,
  },
};

4.1 Inline Requires Deep Dive

inlineRequires is one of the most impactful performance optimizations in Metro. It transforms top-level require() calls into inline calls at the point of first use, deferring module initialization until the module is actually needed.

// BEFORE inlineRequires (standard behavior)
const HeavyComponent = require('./HeavyComponent');  // Initialized at module load
const analytics = require('./analytics');            // Initialized at module load

function render() {
  if (showHeavy) {
    return <HeavyComponent />;
  }
}

// AFTER inlineRequires (Metro transforms to this)
function render() {
  if (showHeavy) {
    return <(require('./HeavyComponent')) />;  // Initialized only when needed
  }
}
Architect Insight

Inline requires can reduce app startup time by 20-40% on large apps by deferring initialization of modules not needed on the initial screen. However, it can cause subtle bugs if modules have side effects that depend on initialization order. The nonInlinedRequires list exempts critical modules (React, react-native) that must initialize eagerly. Profile with require.Systrace to identify which modules benefit most from deferral.

5. Serializer Customization

The serializer takes the transformed module graph and produces the final output — typically a JavaScript bundle and source map. Customizing the serializer enables bundle splitting, tree-shaking analysis, and custom output formats.

const config = {
  serializer: {
    // Custom module ID generation (default: incrementing integers)
    createModuleIdFactory() {
      const ids = new Map();
      let nextId = 0;
      return (modulePath) => {
        if (!ids.has(modulePath)) {
          ids.set(modulePath, nextId++);
        }
        return ids.get(modulePath);
      };
    },

    // Process modules before serialization (tree-shaking, analysis)
    processModuleFilter(module) {
      // Exclude test files from production bundles
      if (module.path.includes('__tests__')) return false;
      if (module.path.endsWith('.test.js')) return false;
      return true;
    },

    // Polyfills to prepend to every bundle
    getPolyfills: () => [
      require.resolve('./src/polyfills/intl.js'),
      require.resolve('./src/polyfills/crypto.js'),
    ],

    // Custom serializer for bundle analysis
    async customSerializer(entryPoint, preModules, graph, options) {
      const defaultSerializer =
        require('metro/src/DeltaBundler/Serializers/baseJSBundle');
      const bundle = await defaultSerializer(
        entryPoint, preModules, graph, options
      );

      // Log bundle statistics
      console.log(`Bundle: ${graph.dependencies.size} modules`);
      console.log(`Size: ${bundle.code.length} bytes`);

      return bundle;
    },
  },
};

5.1 Module ID Stability for OTA Updates

Critical for CodePush / EAS Update

Metro assigns module IDs as incrementing integers by default. Adding or removing a module shifts all subsequent IDs, making the entire bundle different even if most modules are unchanged. For OTA updates that rely on delta diffing, use a stable module ID factory based on file paths. This dramatically reduces delta sizes — a key optimization for CodePush and EAS Update.

// Stable module IDs based on file path hashing
const crypto = require('crypto');

createModuleIdFactory() {
  return (modulePath) => {
    const relativePath = path.relative(__dirname, modulePath);
    const hash = crypto.createHash('md5')
      .update(relativePath)
      .digest('hex')
      .substring(0, 8);
    return parseInt(hash, 16);
  };
}

6. Asset Handling Pipeline

Metro handles assets (images, fonts, audio) differently from source files. Assets are resolved, hashed, and copied to the output directory. The resolution system supports @2x/@3x suffixes for images and platform-specific variants.

Asset Resolution Pipeline ═════════════════════════ require('./logo.png') │ ▼ ┌─────────────────────────────────────────┐ │ Asset Resolver │ │ 1. Find base file: logo.png │ │ 2. Scan for variants: │ │ logo@1x.png (72 dpi) │ │ logo@2x.png (144 dpi) │ │ logo@3x.png (216 dpi) │ │ logo.ios.png (platform-specific) │ │ 3. Compute content hash │ └──────────────────┬──────────────────────┘ │ ┌───────────┴───────────┐ ▼ (Dev) ▼ (Prod) ┌──────────────┐ ┌──────────────────┐ │ Serve via │ │ Copy to output │ │ Metro dev │ │ with hash name │ │ server │ │ logo.abc123.png │ └──────────────┘ └──────────────────┘ 
// Custom asset plugins for non-standard assets
const config = {
  transformer: {
    assetPlugins: [
      require.resolve('./metro-plugins/image-optimizer'),
    ],
    assetRegistryPath: require.resolve(
      'react-native/Libraries/Image/AssetRegistry'
    ),
  },
  resolver: {
    assetExts: [...defaultConfig.resolver.assetExts, 'lottie', 'glb'],
  },
};

7. Source Map Generation

Source maps are critical for debugging production crashes. Metro generates source maps that map minified/compiled bundle positions back to original source locations. For Hermes, an additional step maps bytecode offsets to source positions.

Source Map Type Generated By Maps From Maps To Use Case
JS Source Map Metro serializer Minified JS bundle Original source Non-Hermes debugging
Hermes Source Map hermes-compiler Hermes bytecode (.hbc) Original source Hermes crash symbolication
Composed Source Map compose-source-maps Hermes bytecode Pre-transform source Full-fidelity debugging 
// Generate source maps for production builds
// iOS: sourced from Xcode build phase
// npx react-native bundle \
//   --platform ios \
//   --dev false \
//   --entry-file index.js \
//   --bundle-output ios/main.jsbundle \
//   --sourcemap-output ios/main.jsbundle.map

// For Hermes: compose Metro source map with Hermes compiler map
const { composeSourceMaps } = require('metro-source-map');

const composed = composeSourceMaps([
  metroSourceMap,     // Metro: original → minified JS
  hermesSourceMap,    // Hermes: minified JS → bytecode
]);

8. Monorepo Support

Monorepo support is one of the most complex Metro configuration challenges. The core issues are: hoisted dependencies, symlinked packages, multiple React Native copies, and watch folder configuration.

// metro.config.js for a monorepo package
// Structure: /monorepo/packages/mobile-app/metro.config.js
const path = require('path');
const { getDefaultConfig, mergeConfig } = require('@react-native/metro-config');

const monorepoRoot = path.resolve(__dirname, '../..');
const defaultConfig = getDefaultConfig(__dirname);

const config = {
  projectRoot: __dirname,

  watchFolders: [
    path.resolve(monorepoRoot, 'packages/shared-ui'),
    path.resolve(monorepoRoot, 'packages/api-client'),
    path.resolve(monorepoRoot, 'node_modules'),
  ],

  resolver: {
    unstable_enableSymlinks: true,

    nodeModulesPaths: [
      path.resolve(__dirname, 'node_modules'),
      path.resolve(monorepoRoot, 'node_modules'),
    ],

    // Prevent duplicate React/RN instances in the bundle
    extraNodeModules: {
      'react': path.resolve(__dirname, 'node_modules/react'),
      'react-native': path.resolve(__dirname, 'node_modules/react-native'),
    },

    blockList: [
      // Block other packages' node_modules copies of react/react-native
      new RegExp(
        `${monorepoRoot}/packages/(?!mobile-app).*/node_modules/react-native/.*`
      ),
    ],
  },
};
Duplicate React Instance

The most common monorepo bug is bundling multiple copies of React or React Native. This causes "Invalid hook call" errors because hooks track state per React instance. Always use extraNodeModules to force resolution to a single copy, and blockList to prevent Metro from finding duplicates in other packages' node_modules.

9. Platform-Specific Resolution

Metro's platform resolution enables writing platform-specific code without runtime checks. Files with .ios.js or .android.js extensions are automatically selected based on the target platform. The .native.js extension serves as a shared mobile fallback when web is also a target.

// File structure for cross-platform module
// src/storage/
//   index.ts          ← web fallback (localStorage)
//   index.native.ts   ← shared mobile (MMKV)
//   index.ios.ts      ← iOS-specific (if needed)
//   index.android.ts  ← Android-specific (if needed)

// Consumer code — same import regardless of platform
import { storage } from './storage';
// Metro resolves to the correct file at bundle time

// Custom platform: adding 'tv' platform
const config = {
  resolver: {
    platforms: ['ios', 'android', 'native', 'tv', 'web'],
  },
};

10. Watch Mode & Caching

Metro's file watcher monitors the filesystem for changes and triggers incremental rebuilds. The watcher configuration affects development experience — especially in large codebases and monorepos where watching too many files causes high CPU usage.

const config = {
  watcher: {
    // Directories to ignore (reduces watcher overhead)
    additionalExts: ['cjs', 'mjs'],

    watchman: {
      // Use Watchman for file watching (recommended for large projects)
      deferStates: ['hg.update'],
    },

    healthCheck: {
      enabled: true,
      interval: 30000,  // Check watcher health every 30s
      timeout: 5000,
      filePrefix: '.metro-health-check',
    },
  },

  // Cache configuration
  cacheStores: [
    new (require('metro-cache').FileStore)({
      root: path.resolve(__dirname, '.metro-cache'),
    }),
  ],

  // Reset cache: npx react-native start --reset-cache
};
Watchman vs fs.watch

Metro uses Watchman (from Meta) by default when available, falling back to Node's fs.watch. Watchman is significantly more efficient for large codebases — it maintains a persistent index and uses OS-level file notification APIs. Install Watchman via brew install watchman on macOS. In CI, Watchman is unnecessary since you're doing single builds, not watching.

12. Architect Considerations

Metro Configuration Strategy

At Staff level, Metro configuration is not just about making things work — it's about designing a build system that scales with team size and codebase growth. Key decisions: (1) Monorepo resolver strategy that prevents duplicate dependencies. (2) Stable module IDs for efficient OTA updates. (3) Custom serializer for bundle size monitoring in CI. (4) Transformer configuration that balances startup performance with build speed. (5) Watcher configuration that keeps developer experience snappy as the codebase grows.

Common L6 Interview Pitfall

Candidates often describe Metro as "just a bundler" without understanding its three-phase architecture. When asked "How would you optimize React Native build times?", a Staff-level answer addresses each phase independently: resolver (reduce search paths, block unnecessary files), transformer (worker count, inline requires, Hermes), and serializer (stable IDs, custom output). Generic answers about "caching" without specifying which cache (Metro transform cache, Gradle build cache, CocoaPods cache) signal L5 thinking.

Key Architectural Decisions

  • Bundle splitting strategy: Metro doesn't natively support code splitting like webpack. For large apps, evaluate React.lazy() with asyncRequireModulePath or re-architecture into micro-frontends with separate bundles.
  • SVG handling: Decide between react-native-svg-transformer (inline SVG components, larger bundle) vs asset-based SVG loading (smaller bundle, no dynamic styling). This affects both resolver and transformer config.
  • Polyfill strategy: Node.js polyfills (crypto, stream, buffer) add significant bundle size. Evaluate whether the consuming library truly needs the polyfill or if a lighter alternative exists.
  • CI vs dev configuration: Consider environment-specific Metro configs — more workers and no watcher for CI, persistent cache and Watchman for development.
