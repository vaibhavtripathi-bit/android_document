1. OTA Update Fundamentals

Over-the-air updates allow React Native apps to update their JavaScript bundle and assets without going through the app store review process. This is possible because RN apps load JavaScript at runtime — the native shell remains unchanged while the JS bundle is swapped. OTA updates can reduce deployment cycles from days (app store review) to minutes.

OTA Update Architecture ═══════════════════════ ┌─────────────────────────────────────────────────────┐ │ App Binary (v1.0) │ │ │ │ ┌──────────────────┐ ┌─────────────────────────┐ │ │ │ Native Shell │ │ JS Bundle │ │ │ │ (unchanged) │ │ (updatable via OTA) │ │ │ │ │ │ │ │ │ │ • RN Runtime │ │ • App logic │ │ │ │ • Native Mods │ │ • UI components │ │ │ │ • Hermes VM │ │ • Business rules │ │ │ │ • Bridge/JSI │ │ • Assets (images, etc) │ │ │ └──────────────────┘ └────────────┬────────────┘ │ └─────────────────────────────────────┼───────────────┘ │ │ OTA Update ▼ ┌─────────────────────────────────────────────────────┐ │ Update Server │ │ (CodePush / EAS Update / Custom) │ │ │ │ 1. App checks for updates on launch │ │ 2. Server compares app version + runtime version │ │ 3. If compatible update exists → download bundle │ │ 4. App installs update (next restart or immediate) │ │ 5. Rollback if update causes crash │ └─────────────────────────────────────────────────────┘
Architect Insight

OTA updates can only change what the JavaScript runtime loads — the JS bundle and JS-accessible assets. They cannot change: native modules, native UI components, native build configuration, app permissions, or the Hermes/JSC engine version. This boundary is the fundamental constraint that drives all OTA architecture decisions. A Staff engineer must design systems that enforce this boundary and fail safely when it's violated.

2. CodePush Architecture

CodePush (originally from Microsoft, now part of App Center) was the first widely-adopted OTA solution for React Native. It uses a deployment-key model where each app binary is configured with a deployment key that maps to an update channel.

CodePush Update Flow ════════════════════ App Launch │ ▼ ┌──────────────────────────────────────┐ │ CodePush.sync() called │ │ Sends: deployment key, app version, │ │ current package hash │ └──────────────────┬───────────────────┘ │ ▼ ┌──────────────────────────────────────┐ │ CodePush Server │ │ 1. Find deployment by key │ │ 2. Check target binary version │ │ 3. Compare package hash │ │ 4. Return: update metadata + URL │ │ OR: no update available │ └──────────────────┬───────────────────┘ │ ┌──────────┴──────────┐ ▼ ▼ ┌──────────────┐ ┌──────────────────┐ │ No Update │ │ Update Found │ │ Load current│ │ 1. Download zip │ │ bundle │ │ 2. Verify hash │ │ │ │ 3. Unzip to │ │ │ │ CodePush dir │ │ │ │ 4. Install: │ │ │ │ immediate OR │ │ │ │ next restart │ └──────────────┘ └──────────────────┘ 
// CodePush integration — comprehensive configuration
import codePush from 'react-native-code-push';

const codePushOptions = {
  checkFrequency: codePush.CheckFrequency.ON_APP_RESUME,
  installMode: codePush.InstallMode.ON_NEXT_RESTART,
  mandatoryInstallMode: codePush.InstallMode.IMMEDIATE,
  minimumBackgroundDuration: 60 * 5, // 5 minutes in background
  deploymentKey: Platform.select({
    ios: Config.CODEPUSH_IOS_KEY,
    android: Config.CODEPUSH_ANDROID_KEY,
  }),
};

function App() {
  const [updateStatus, setUpdateStatus] = useState(null);

  useEffect(() => {
    codePush.sync(
      {
        installMode: codePush.InstallMode.ON_NEXT_RESTART,
        mandatoryInstallMode: codePush.InstallMode.IMMEDIATE,
      },
      (status) => {
        switch (status) {
          case codePush.SyncStatus.CHECKING_FOR_UPDATE:
            setUpdateStatus('Checking...');
            break;
          case codePush.SyncStatus.DOWNLOADING_PACKAGE:
            setUpdateStatus('Downloading...');
            break;
          case codePush.SyncStatus.INSTALLING_UPDATE:
            setUpdateStatus('Installing...');
            break;
          case codePush.SyncStatus.UP_TO_DATE:
            setUpdateStatus(null);
            break;
        }
      },
      ({ receivedBytes, totalBytes }) => {
        const progress = (receivedBytes / totalBytes) * 100;
        setUpdateStatus(`Downloading: ${progress.toFixed(0)}%`);
      }
    );
  }, []);

  return <MainApp updateStatus={updateStatus} />;
}

export default codePush(codePushOptions)(App);
CodePush Sunset Notice

Microsoft App Center (including CodePush) was retired on March 31, 2025. Teams using CodePush must migrate to alternatives: EAS Update (Expo), a self-hosted solution using react-native-code-push with a custom server, or community forks. This migration is a significant architectural decision — evaluate alternatives based on your team's Expo adoption level and infrastructure capabilities.

3. EAS Update

EAS Update is Expo's OTA update service that uses a branch-and-channel model. Updates are published to branches, and app builds are configured to listen on channels that map to branches. This model supports sophisticated deployment strategies.

// Publishing updates with EAS
// eas update --branch production --message "Fix payment crash"
// eas update --branch staging --message "Test new checkout flow"

// Programmatic update checking with expo-updates
import * as Updates from 'expo-updates';

async function checkForUpdates() {
  try {
    const update = await Updates.checkForUpdateAsync();

    if (update.isAvailable) {
      const result = await Updates.fetchUpdateAsync();

      if (result.isNew) {
        // Option 1: Reload immediately
        await Updates.reloadAsync();

        // Option 2: Show prompt to user
        Alert.alert(
          'Update Available',
          'A new version is ready. Restart to apply?',
          [
            { text: 'Later', style: 'cancel' },
            { text: 'Restart', onPress: () => Updates.reloadAsync() },
          ]
        );
      }
    }
  } catch (error) {
    // Update check failed — continue with current bundle
    console.warn('Update check failed:', error);
  }
}

// Listen for automatic update events
Updates.addListener((event) => {
  if (event.type === Updates.UpdateEventType.UPDATE_AVAILABLE) {
    // New update downloaded and ready
  } else if (event.type === Updates.UpdateEventType.NO_UPDATE_AVAILABLE) {
    // Already on latest
  } else if (event.type === Updates.UpdateEventType.ERROR) {
    // Update check/download failed
  }
});

4. Update Strategies

The update strategy determines how and when users receive updates. Each strategy balances immediacy against user experience disruption.

Strategy User Experience Deployment Speed Use Case
Silent (background) No disruption — applied on next restart Slow (depends on user restart) Minor fixes, non-critical updates
Optional prompt Notification — user chooses when Medium (depends on user action) Feature updates, improvements
Mandatory prompt Must restart — can defer briefly Fast (user must accept) Important bug fixes
Immediate App reloads mid-session Instant Critical security fixes only
On next resume Applied when app returns from background Fast (next app open) Good balance for most updates 
// Implementing a multi-strategy update system
async function handleUpdate(updateMetadata) {
  const { isMandatory, severity, releaseNotes } = updateMetadata;

  if (severity === 'critical') {
    // Immediate: download and reload without user input
    await Updates.fetchUpdateAsync();
    await Updates.reloadAsync();
    return;
  }

  if (isMandatory) {
    // Mandatory: show non-dismissable dialog
    await Updates.fetchUpdateAsync();
    showMandatoryUpdateDialog(releaseNotes, () => {
      Updates.reloadAsync();
    });
    return;
  }

  // Optional: download in background, apply on next restart
  const result = await Updates.fetchUpdateAsync();
  if (result.isNew) {
    showOptionalUpdateBanner(releaseNotes);
  }
}
Strategy Selection Framework

Default to silent background updates for most releases. Reserve mandatory/immediate for security fixes and critical bugs. The key metric is "time to 90% adoption" — silent updates typically reach 90% in 3-7 days (as users naturally restart), while mandatory updates reach 90% in 1-2 days. Immediate updates reach 100% instantly but risk disrupting users mid-task, which can cause data loss in forms or transactions.

5. Bundle Compatibility Checking

The most dangerous OTA failure mode is deploying a JS bundle that references native modules or APIs not present in the app binary. This causes runtime crashes that are worse than the bug the update was meant to fix.

Bundle Compatibility Matrix ═══════════════════════════ App Binary v1.0 App Binary v1.1 (native modules A,B,C) (native modules A,B,C,D) │ │ │ Compatible: │ Compatible: │ • JS using A,B,C │ • JS using A,B,C,D │ • JS using A,B │ • JS using A,B,C │ │ │ INCOMPATIBLE: │ INCOMPATIBLE: │ • JS using D ← CRASH │ • JS using E ← CRASH │ • JS using new RN API │ Runtime Version enforces this boundary: v1.0 binary → only receives updates with runtimeVersion "1.0" v1.1 binary → only receives updates with runtimeVersion "1.1" 
// Fingerprint-based compatibility (Expo SDK 50+)
// The "fingerprint" policy auto-generates a hash from:
//   - Native dependencies (Podfile.lock, build.gradle)
//   - Expo SDK version
//   - Config plugin outputs
//   - Native module registrations
//
// Any native change produces a new fingerprint,
// automatically preventing incompatible OTA updates.

{
  "expo": {
    "runtimeVersion": {
      "policy": "fingerprint"
    }
  }
}

// Manual compatibility check in update handler
async function safeUpdateCheck() {
  const update = await Updates.checkForUpdateAsync();

  if (update.isAvailable) {
    const manifest = update.manifest;
    const updateRuntime = manifest.runtimeVersion;
    const appRuntime = Updates.runtimeVersion;

    if (updateRuntime !== appRuntime) {
      console.warn(
        `Runtime mismatch: app=${appRuntime}, update=${updateRuntime}`
      );
      return; // Skip incompatible update
    }

    await Updates.fetchUpdateAsync();
  }
}
Critical: The Compatibility Boundary

The #1 cause of OTA-related production incidents is deploying a JS bundle that assumes a native module exists when it doesn't. This happens when: (1) A developer adds a new native dependency and publishes an OTA update before the new binary is in the store. (2) Runtime version is manually managed and not incremented after a native change. (3) The update targets the wrong deployment channel. Use fingerprint-based runtime versioning to make this class of bug impossible.

6. Rollback Mechanisms

Rollback is the safety net for OTA updates. When an update causes crashes, the app must automatically revert to the previous working bundle. Both CodePush and EAS Update implement automatic crash-based rollback.

Automatic Rollback Flow ═══════════════════════ Update Installed (v2) │ ▼ App Launches with v2 │ ├── App runs successfully for > 10 seconds │ └── Mark v2 as "stable" │ (clear rollback flag) │ └── App crashes within 10 seconds └── Native crash handler detects crash │ ▼ App Relaunches │ ▼ Check: was last launch a crash after update? │ ├── YES → Rollback to v1 (previous bundle) │ Log rollback event to analytics │ Report to update server │ └── NO → Normal launch 
// CodePush rollback configuration
codePush.sync({
  installMode: codePush.InstallMode.ON_NEXT_RESTART,
  rollbackRetryOptions: {
    delayInHours: 24,          // Wait 24h before retrying after rollback
    maxRetryAttempts: 1,       // Only retry once
  },
});

// Manual rollback trigger (server-side)
// appcenter codepush rollback MyApp-iOS Production
// This promotes the previous release to the latest position

// EAS Update rollback — publish previous bundle to branch
// eas update:republish --group UPDATE_GROUP_ID --branch production

// Custom rollback detection
import * as Updates from 'expo-updates';

function useUpdateHealth() {
  useEffect(() => {
    const launchTime = Date.now();

    const healthCheck = setTimeout(() => {
      // App survived 30 seconds — mark update as healthy
      reportUpdateHealth({
        updateId: Updates.updateId,
        status: 'healthy',
        launchDuration: Date.now() - launchTime,
      });
    }, 30000);

    return () => clearTimeout(healthCheck);
  }, []);
}

7. A/B Testing with OTA

OTA updates enable server-controlled A/B testing by deploying different bundles to different user segments. This is more powerful than feature flags alone because it can include entirely different UI implementations, not just toggled features.

// A/B testing via EAS Update channels
// Channel mapping:
//   production-control  → Branch: production (current version)
//   production-variant-a → Branch: experiment-new-checkout
//   production-variant-b → Branch: experiment-new-onboarding

// Server-side: assign users to channels based on user ID hash
function getUpdateChannel(userId) {
  const hash = hashCode(userId) % 100;

  if (hash < 80) return 'production-control';       // 80%
  if (hash < 90) return 'production-variant-a';    // 10%
  return 'production-variant-b';                     // 10%
}

// Client-side: configure update channel at startup
// This requires a custom native module or Expo config plugin
// to set the update URL/channel before the first update check
Architect Insight

OTA-based A/B testing is powerful but operationally complex. Each variant is a separate bundle that must be built, tested, and monitored independently. For most teams, feature flags (LaunchDarkly, Statsig, custom) within a single bundle are simpler and sufficient. Reserve OTA-based A/B testing for experiments that require fundamentally different code paths — like testing an entirely new navigation architecture or a rewritten core module.

8. Update Size Optimization & Delta Updates

Update size directly impacts adoption speed and user data costs. Smaller updates download faster, fail less often, and are more acceptable to users on cellular connections.

Optimization Technique Size Reduction Implementation
Stable module IDs Hash-based IDs in Metro serializer 60-80% smaller deltas metro.config.js
Tree shaking Remove unused exports 10-30% smaller bundle Metro + Babel config
Asset optimization Compress images, use vector assets Variable (often 50%+) Build pipeline
Hermes bytecode Compiled bytecode vs raw JS 30-50% smaller Hermes enabled
Delta updates Binary diff of old → new bundle 80-95% smaller CodePush / custom server
Gzip compression Compress bundle for transfer 60-70% smaller transfer Server configuration 
// Delta update concept (CodePush implements this automatically)
// Instead of sending the full 5MB bundle, send only the diff

// Full bundle: 5.0 MB
// Delta (bsdiff): 50-200 KB (typical for small code changes)

// Server-side delta generation (conceptual)
async function generateDelta(oldBundleHash, newBundle) {
  const oldBundle = await getBundleByHash(oldBundleHash);

  if (oldBundle) {
    const delta = bsdiff(oldBundle, newBundle);
    const ratio = delta.length / newBundle.length;

    if (ratio < 0.5) {
      return { type: 'delta', data: delta, baseHash: oldBundleHash };
    }
  }

  return { type: 'full', data: newBundle };
}

9. App Store Policies on OTA

Both Apple and Google have policies governing OTA updates. Understanding these policies is critical — violating them can result in app rejection or removal from the store.

Policy Aspect Apple (App Store) Google (Play Store)
JS bundle updates Allowed (Section 3.3.2) Allowed
Interpreted code Allowed if doesn't change primary purpose Allowed
Native code updates Prohibited (must go through review) Prohibited
Feature changes Must not change app's primary purpose Must not introduce deceptive behavior
Monetization changes Cannot bypass IAP via OTA Cannot bypass Play billing
Privacy changes Cannot add new data collection via OTA Must match data safety declaration
Apple Section 3.3.2

"Interpreted code may be downloaded to an Application but only so long as such code: (a) does not change the primary purpose of the Application by providing features or functionality that are inconsistent with the intended and advertised purpose of the Application as submitted to the App Store, (b) does not create a store or storefront for other code or applications, and (c) does not bypass signing, sandbox, or other security features of the OS." Using OTA to add entirely new features (not bug fixes or minor improvements) risks App Store rejection. Keep OTA updates focused on fixes and incremental improvements.

10. Monitoring Update Adoption

Monitoring OTA update adoption is essential for understanding deployment velocity and detecting problems. Key metrics include adoption rate, crash rate delta, rollback frequency, and download failure rate.

// Custom update telemetry
import * as Updates from 'expo-updates';
import analytics from './analytics';

function trackUpdateMetrics() {
  analytics.setUserProperty('js_bundle_version', Updates.updateId ?? 'embedded');
  analytics.setUserProperty('runtime_version', Updates.runtimeVersion);
  analytics.setUserProperty('update_channel', Updates.channel ?? 'default');
  analytics.setUserProperty('is_embedded_bundle',
    String(Updates.isEmbeddedLaunch));
}

11. Staged Rollouts

Staged rollouts gradually increase the percentage of users who receive an update, allowing you to detect problems before they affect the entire user base. This is the most important risk mitigation strategy for OTA updates.

Staged Rollout Timeline ═══════════════════════ Hour 0 Hour 4 Hour 12 Hour 24 Hour 48 │ │ │ │ │ ▼ ▼ ▼ ▼ ▼ ┌───┐ ┌─────┐ ┌───────┐ ┌─────────┐ ┌───────────┐ │ 1%│ │ 5% │ │ 25% │ │ 50% │ │ 100% │ └───┘ └─────┘ └───────┘ └─────────┘ └───────────┘ At each stage, monitor: ├── Crash rate (must be ≤ baseline) ├── Error rate (must be ≤ baseline + threshold) ├── Core metric (e.g., checkout completion rate) └── User reports (support tickets, reviews) If any metric degrades → HALT rollout → investigate → rollback 
// Server-side staged rollout logic
function shouldReceiveUpdate(userId, rolloutPercentage) {
  // Deterministic: same user always gets same result for same percentage
  const userHash = hashCode(userId) % 100;
  return userHash < rolloutPercentage;
}

// CodePush staged rollout
// appcenter codepush release-react MyApp-iOS ios -r 25
// (releases to 25% of users)
// appcenter codepush patch MyApp-iOS Production -r 50
// (increase to 50%)
// appcenter codepush patch MyApp-iOS Production -r 100
// (full rollout)

12. Architect Considerations

OTA Update Strategy Framework

A Staff-level OTA strategy addresses five dimensions: (1) Safety — runtime version enforcement, automatic rollback, staged rollouts, and crash monitoring ensure updates don't make things worse. (2) Speed — delta updates, CDN distribution, and background downloads minimize time-to-install. (3) Observability — adoption tracking, crash rate comparison, and rollback alerting provide confidence in each release. (4) Compliance — app store policies, privacy regulations, and enterprise requirements constrain what can be updated OTA. (5) Developer experience — the update publishing workflow must be simple, fast, and integrated with CI/CD.

Common L6 Interview Pitfall

Candidates often describe OTA updates as a simple "push new JS bundle to users" without addressing the critical failure modes. A Staff-level answer covers: What happens if the update crashes on launch? (Automatic rollback.) How do you prevent incompatible updates? (Runtime versioning.) How do you limit blast radius? (Staged rollouts.) How do you know the update is working? (Adoption monitoring + crash rate comparison.) How do you comply with app store policies? (JS-only changes, no primary purpose change.) The depth of your failure mode analysis signals Staff-level thinking.

Key Architectural Decisions

  • CodePush vs EAS Update vs custom: CodePush is sunset — migrate to EAS Update (if using Expo) or a self-hosted solution. EAS Update provides the best DX for Expo projects. Custom solutions offer maximum control but require significant infrastructure investment.
  • Runtime version strategy: Use fingerprint-based versioning (Expo) or automated native dependency hashing to eliminate manual version management errors. Manual runtime versioning is error-prone at scale.
  • Update checking frequency: Check on app launch and resume. Avoid polling on a timer — it wastes battery and bandwidth. Use push notifications or silent pushes to trigger immediate update checks for critical fixes.
  • Rollback threshold: Define automatic rollback criteria before deploying. Example: "If crash rate exceeds 2x baseline within 1 hour of rollout, automatically halt and rollback." This must be automated — human response time is too slow for production incidents.
  • Multi-platform coordination: iOS and Android updates must be coordinated but can be deployed independently. Use the same branch/channel model but allow platform-specific rollout pacing based on each platform's crash rate.
