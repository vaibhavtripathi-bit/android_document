Metro Bundler

Q1: What are the three phases of Metro bundling, and what happens in each?

Resolution: Starting from the entry point, Metro builds a dependency graph by resolving every import/require statement to an absolute file path. It uses platform-specific extension resolution (.ios.ts → .native.ts → .ts) and respects the resolver config for custom module paths. Transformation: Each resolved file is passed through a transformer (Babel by default, or a custom transformer) that compiles JSX, TypeScript, and Flow into plain JavaScript. Transforms run in parallel worker processes. Serialization: The transformed modules are concatenated into a single JavaScript bundle with a module registry. Each module gets a numeric ID, and require() calls are rewritten to use these IDs. The serializer also generates the source map.

Q2: How does Metro's file watcher work, and why is it important for development?

Metro uses fb-watchman (or fs.watch fallback) to monitor the filesystem for changes. When a file changes, Watchman notifies Metro, which invalidates only the changed module and its dependents in the dependency graph — it doesn't rebuild the entire graph. The transformer cache is keyed by file content hash, so unchanged files use cached transforms. This incremental approach is what makes hot reloads fast (typically <500ms). The watchFolders config controls which directories Metro monitors, which is critical in monorepos where you need to watch workspace packages outside the app directory.

Q3: Explain inline requires and how they improve startup performance.

By default, all require() calls execute at module load time — when the bundle loads, every module's factory function runs. Inline requires (inlineRequires: true in Metro config) transforms top-level require calls into lazy inline calls that execute only when the imported value is first accessed. This defers module initialization, reducing startup time because modules not needed during the initial render are loaded on-demand. The trade-off: first access of a lazily-required module has a small overhead, and circular dependencies can cause issues if a module is accessed before its factory has run. Metro can selectively apply inline requires via a blocklist for problematic modules.

Q4: What is tree shaking, and why doesn't Metro support it natively? How do you work around this?

Tree shaking eliminates unused exports from the final bundle by analyzing static import/export relationships. Metro doesn't support tree shaking because it uses CommonJS-style module wrapping internally — even if you write ESM, Metro converts to CJS during serialization, which makes static analysis of unused exports impossible. Workarounds: (1) use package.json sideEffects: false to enable module-level elimination, (2) prefer importing from specific subpaths (import { map } from 'lodash-es/map') instead of barrel exports, (3) use @babel/plugin-transform-remove-console to strip console logs, (4) consider re.pack (webpack for RN) which supports tree shaking, (5) Metro is exploring ESM support which would enable tree shaking in future versions.

Q5: How does Metro handle platform-specific file resolution, and what is the full resolution priority?

When resolving import './Button' on iOS, Metro tries extensions in this order: Button.ios.tsx → Button.ios.ts → Button.ios.jsx → Button.ios.js → Button.native.tsx → Button.native.ts → Button.native.jsx → Button.native.js → Button.tsx → Button.ts → Button.jsx → Button.js → Button/index.ios.tsx → ... (same pattern for index files). The .native extension matches any non-web platform. This is a build-time mechanism — only the resolved file is included in the bundle. Configure custom extensions via resolver.sourceExts and platform-specific extensions via resolver.platforms.

Q6: What is RAM bundle format, and when would you use it over a plain bundle?

RAM (Random Access Modules) bundles store each module as a separate entry in an indexed file, allowing the runtime to load individual modules on demand rather than parsing the entire bundle at startup. On Android, this uses an indexed format; on iOS, it uses unbundled files. RAM bundles are beneficial for very large apps (>10MB bundles) where startup time is critical, because the JS engine only parses modules as they're required. However, with Hermes bytecode precompilation, RAM bundles provide diminishing returns — Hermes already achieves fast startup by executing precompiled bytecode. RAM bundles are most useful with JSC (JavaScriptCore) on older architectures.

Q7: How would you configure Metro to support a monorepo with shared packages?

Three critical configurations: (1) watchFolders: add the monorepo root so Metro can see workspace packages outside the app directory. (2) resolver.nodeModulesPaths: add both the app's node_modules and the root node_modules so hoisted dependencies are found. (3) resolver.extraNodeModules: pin react and react-native to the app's copies to prevent duplicate instances (which breaks hooks). Additionally, use resolver.blockList to exclude other apps' directories from the dependency graph, and set resolver.disableHierarchicalLookup: true to prevent Metro from walking up the directory tree and finding wrong module copies.

Q8: Explain Metro's transformer cache and how to optimize it.

Metro caches the output of each file's transformation keyed by a hash of the file content, transformer configuration, and Babel/SWC config. The cache is stored on disk (default: /tmp/metro-*) and persists across Metro restarts. Optimization strategies: (1) use SWC instead of Babel for 20-70x faster transforms, (2) increase maxWorkers for parallel transformation on CI machines with many cores, (3) pre-warm the cache by running a build before development sessions, (4) use cacheStores config to point to a persistent cache location that survives CI container restarts, (5) avoid Babel plugins that depend on non-deterministic inputs (timestamps, random values) which bust the cache.

Q9: What is the difference between Metro's development and production bundle outputs?

Development bundles include: source maps (inline or separate), __DEV__ flag set to true, Hot Module Replacement (HMR) client code, React DevTools integration hooks, unminified code with original variable names, and console log preservation. Production bundles: __DEV__ set to false (enabling dead code elimination of dev-only paths), minification via Metro's minifier (Terser or Hermes bytecode), no HMR code, no source maps by default (configurable), and global name mangling. The __DEV__ flag is particularly important because React uses it internally to gate development warnings, PropTypes checking, and double-rendering in StrictMode.

Q10: How does Hermes bytecode precompilation integrate with Metro?

When Hermes is enabled, Metro's production build pipeline adds a post-bundling step: the serialized JS bundle is passed to the Hermes compiler (hermesc) which compiles it to Hermes bytecode (.hbc). This bytecode is what ships in the app binary. At runtime, Hermes loads and executes the bytecode directly — no parsing or JIT compilation needed. This reduces startup time by 30-50% compared to JSC parsing raw JavaScript. The bytecode format is version-specific — a bytecode file compiled with Hermes v0.12 won't run on Hermes v0.11. This has implications for OTA updates: the OTA bundle must be compiled with the same Hermes version embedded in the native shell.

Native Builds

Q11: Walk through the complete iOS build pipeline from source to .ipa file.

(1) yarn install + pod install resolve JS and native dependencies. (2) Metro bundles the JS code into main.jsbundle (production) or serves it via dev server (development). (3) Xcode/xcodebuild compiles Objective-C/Swift source files and linked native modules into object files. (4) The linker combines object files into the app binary. (5) Asset catalog compiler processes images and app icons. (6) The provisioning profile is embedded as embedded.mobileprovision. (7) Code signing signs each framework, dylib, and the main binary. (8) The .app bundle is created containing the binary, JS bundle, assets, and provisioning profile. (9) For distribution, the .app is packaged into an .ipa (a ZIP archive). (10) For App Store, the .ipa is uploaded via altool or Transporter.

Q12: What is the difference between Debug and Release build configurations in React Native?

Debug: JS bundle loaded from Metro dev server (not embedded), Flipper/debugging enabled, Hermes bytecode compilation skipped, __DEV__=true, no code signing required for simulator, ProGuard/R8 disabled on Android, compiler optimizations off (-O0). Release: JS bundle embedded in the app binary, Flipper stripped, Hermes bytecode precompiled, __DEV__=false, code signing required, ProGuard/R8 enabled (Android), compiler optimizations on (-Os), source maps generated separately. The Release configuration also strips debug symbols, which reduces binary size by 30-60% but makes crash symbolication require separate dSYM/mapping files.

Q13: How does React Native autolinking work, and what problems does it solve?

Autolinking (introduced in RN 0.60) automatically discovers and links native modules by scanning node_modules for packages with a react-native.config.js or native code in standard locations. For iOS, it generates a Podfile entry; for Android, it generates Gradle includes. Before autolinking, developers had to manually run react-native link which modified native project files and was error-prone (duplicate entries, merge conflicts). Autolinking runs at build time, is idempotent, and respects the dependencies field in react-native.config.js for customization. In monorepos, autolinking may need explicit root paths when native modules are hoisted to the monorepo root.

Q14: Explain Android App Bundles (AAB) vs APK. Why does Google require AAB?

An APK is a self-contained installable package. An AAB is a publishing format that Google Play uses to generate optimized APKs per device configuration (screen density, CPU architecture, language). Google requires AAB because it reduces download size by 15-30% — users only download resources for their specific device. AAB enables Dynamic Feature Modules for on-demand code delivery. For React Native, AAB means the JS bundle and native libraries are split by ABI (armeabi-v7a, arm64-v8a, x86_64), so a 64-bit device doesn't download 32-bit native code. The trade-off: AAB requires Google Play App Signing (Google holds the signing key), and you can't sideload AABs for testing — use bundletool to generate test APKs locally.

Q15: How do you reduce React Native app binary size?

JS layer: enable Hermes (smaller bytecode vs raw JS), enable inline requires, audit and remove unused dependencies, use dynamic imports for large features, avoid barrel exports. iOS: enable Bitcode (deprecated in Xcode 14), strip unused architectures, use asset catalogs with app thinning, review linked frameworks. Android: enable ProGuard/R8 (minifyEnabled true), enable resource shrinking (shrinkResources true), use AAB for per-device optimization, split APKs by ABI (enableSeparateBuildPerCPUArchitecture), compress native libraries (useLegacyPackaging false). Cross-platform: audit image assets (use WebP, compress PNGs), remove unused fonts, analyze bundle with react-native-bundle-visualizer, set up CI size budgets with alerts on regression.

Q16: What is ProGuard/R8, and how does it affect React Native Android builds?

R8 (successor to ProGuard) is Android's code shrinker, obfuscator, and optimizer. It removes unused Java/Kotlin classes, renames symbols to short names, and optimizes bytecode. For React Native, R8 must be configured carefully: (1) keep rules are needed for classes accessed via reflection (JNI calls from C++, JavaScript bridge calls), (2) React Native ships default ProGuard rules in proguard-rules.pro, (3) third-party native modules may need additional keep rules, (4) R8 can break native modules that use reflection-based registration. Always test release builds thoroughly — R8 issues only manifest in release mode and can cause runtime crashes that don't appear in debug builds.

Q17: How do you set up different build environments (dev, staging, production) in React Native?

iOS: use Xcode build configurations (Debug-Dev, Debug-Staging, Release-Production) with corresponding schemes. Each configuration sets different preprocessor macros, bundle identifiers, and provisioning profiles. Android: use Gradle product flavors with applicationIdSuffix for distinct package names, allowing all environments to be installed simultaneously. Cross-platform: use react-native-config or expo-constants to inject environment variables at build time from .env files. Each environment should have distinct: API base URL, bundle identifier (so all can be installed on one device), app icon (with visual indicator), display name (e.g., "MyApp DEV"), and signing configuration.

Q18: What are Gradle build variants, and how do they compose with React Native?

Build variants are the Cartesian product of build types (debug, release) and product flavors (dev, staging, production). So 2 build types × 3 flavors = 6 variants: devDebug, devRelease, stagingDebug, stagingRelease, productionDebug, productionRelease. Each variant can have its own source set (src/dev/, src/staging/), resources, manifest entries, and dependencies. React Native's Gradle plugin hooks into this system — the JS bundle is generated for release build types, and the Metro dev server URL is configured for debug build types. The ./gradlew assembleProductionRelease command builds a specific variant.

Q19: How does Flipper integrate with React Native builds, and what's the performance impact?

Flipper is a desktop debugging tool that connects to RN apps via a native SDK. It's included in debug builds by default (RN 0.62+) and adds: network inspector, React DevTools, Hermes debugger, layout inspector, and shared preferences viewer. Performance impact: Flipper adds ~2-4MB to the debug binary, increases build time by 10-20 seconds (due to additional CocoaPods/Gradle dependencies), and has a small runtime overhead from the WebSocket connection. For production, Flipper is stripped automatically in release builds. Many teams now remove Flipper entirely (RN 0.73+ made it opt-in) in favor of standalone tools like React DevTools and Chrome DevTools, which have zero build-time impact.

Expo & EAS

Q20: What is the difference between Expo Go, Development Builds, and standalone builds?

Expo Go: a pre-built app containing a fixed set of native modules. Your JS code runs inside it — no native build step needed. Limited to Expo SDK modules; can't use custom native code. Development Builds: a custom-built debug app containing your specific native modules + expo-dev-client. Like Expo Go but with your native dependencies. Requires an EAS Build or local build. Standalone/Production Builds: the final release binary with all native code, signed for distribution. Development builds are the recommended development workflow for any project using custom native modules — they provide the Expo Go developer experience with the flexibility of bare React Native.

Q21: How does EAS Build differ from building locally with Xcode/Gradle?

EAS Build runs on Expo's cloud infrastructure (macOS VMs for iOS, Linux for Android). Key differences: (1) no local Xcode/Android Studio required, (2) credential management is handled by EAS (certificates, provisioning profiles, keystores), (3) builds are reproducible across team members (same environment), (4) build logs and artifacts are stored in the cloud, (5) build queue with parallelization for multiple platforms. Trade-offs: slower feedback loop (queue time + upload), requires internet, costs money for high-volume usage, less control over the build environment. EAS Build uses Fastlane under the hood for iOS builds and Gradle for Android.

Q22: Explain the Expo prebuild system and how it generates native projects.

expo prebuild generates the ios/ and android/ directories from app.json/app.config.js and Expo config plugins. It's a "managed to bare" bridge — you describe your native configuration declaratively, and prebuild generates the native project files. Config plugins are JavaScript functions that modify the native project during prebuild (e.g., adding permissions to Info.plist, modifying AndroidManifest.xml). The generated native projects are typically gitignored and regenerated on each build. This is called "Continuous Native Generation" (CNG) — it treats native projects as build artifacts rather than source code, similar to how you don't commit compiled binaries.

Q23: When should you eject from Expo (or use bare workflow), and what are the trade-offs?

With modern Expo (SDK 50+), "ejecting" is largely obsolete — you run expo prebuild and commit the native directories if you need direct native access. Reasons to go bare: (1) custom native modules not available as Expo modules, (2) need to modify native build settings not exposed via config plugins, (3) integration with existing native codebases, (4) specific CI/CD requirements incompatible with EAS. Trade-offs of bare: you own the native project maintenance (Xcode/Gradle upgrades, dependency conflicts), lose CNG benefits, and must manage signing credentials yourself. The recommended approach: stay managed as long as possible, use config plugins for customization, and only commit native directories when config plugins are insufficient.

Q24: How do Expo config plugins work, and how would you write a custom one?

Config plugins are functions that receive an Expo config object and return a modified version. They run during expo prebuild to modify native project files. A plugin receives the config and can use "mods" to modify specific files: withInfoPlist (iOS Info.plist), withAndroidManifest, withAppDelegate, withMainActivity, etc. Custom plugin example: const withCustomScheme = (config, scheme) => withInfoPlist(config, (config) => { config.modResults.CFBundleURLTypes = [...]; return config; }). Plugins compose — multiple plugins can modify the same file, and they execute in order. This is how the Expo ecosystem avoids manual native project modifications.

Q25: What is EAS Update, and how does it compare to CodePush?

EAS Update is Expo's OTA update service that delivers JS bundle and asset updates without app store submission. Compared to CodePush: (1) EAS Update uses "update groups" and "branches" for targeting, while CodePush uses deployment keys, (2) EAS Update integrates natively with expo-updates library, CodePush uses its own runtime, (3) EAS Update supports Hermes bytecode bundles, (4) EAS Update has built-in rollback via "republishing" a previous update, (5) both support staged rollouts, but EAS Update's channel/branch model is more flexible. EAS Update is the recommended choice for Expo projects; CodePush (now App Center) is being deprecated by Microsoft.

Q26: How does Expo's fingerprint system determine native compatibility for OTA updates?

Expo fingerprint (@expo/fingerprint) generates a hash of all native-impacting configuration: native dependencies, config plugin outputs, app.json native fields, iOS/Android project files, and native source code. When an OTA update is published, its fingerprint is compared against the running app's fingerprint. If they match, the update is compatible (JS-only change). If they differ, the update requires a new native build. This prevents the dangerous scenario of pushing a JS update that references a native module not present in the installed binary. The fingerprint is essentially a content-addressable identifier for the native shell.

Q27: How would you architect a CI/CD pipeline using EAS for a team of 20 developers?

(1) Development: developers use development builds (eas build --profile development) — rebuild only when native deps change (fingerprint-based). (2) PR builds: run eas update on PR branches for preview — reviewers scan a QR code to test changes in their dev build. (3) Staging: merge to staging branch triggers eas update --branch staging for QA testing. (4) Production: merge to main triggers eas build --profile production + eas submit for store submission. Use EAS Build's autoSubmit to chain build → submit. Configure eas.json profiles for each environment with distinct channels, credentials, and environment variables. Set up GitHub Actions to orchestrate the pipeline with fingerprint checks to skip unnecessary native builds.

OTA Updates

Q28: What can and cannot be updated via OTA in React Native?

Can update: JavaScript code, TypeScript (compiled to JS), JSON data files, images/assets referenced from JS, style changes, navigation structure, business logic. Cannot update: native module code (Objective-C, Swift, Java, Kotlin, C++), native UI components, new native permissions, changes to Info.plist or AndroidManifest.xml, new native dependencies, Hermes engine version, React Native version (if it includes native changes). The boundary is the JS-native interface — anything that changes the native binary requires a store release.

Q29: Explain the security risks of OTA updates and how to mitigate them.

Risks: (1) Man-in-the-middle attacks could inject malicious JS bundles, (2) compromised update server could push malicious code to all users, (3) bundle tampering during transit. Mitigations: (1) code signing — sign bundles with a private key, verify signature on device before applying (EAS Update and CodePush both support this), (2) HTTPS with certificate pinning for update downloads, (3) integrity checks — compare bundle hash against server-provided hash, (4) rollback capability — if the updated bundle crashes on launch, automatically revert to the embedded bundle, (5) access control — restrict who can publish OTA updates via CI/CD with approval gates, (6) audit logging — track every OTA publish with author, timestamp, and content hash.

Q30: How do you handle OTA update compatibility with multiple native shell versions?

Use a runtime version or fingerprint system to map JS bundles to compatible native shells. With expo-updates, set runtimeVersion in app.json — the update server only delivers updates matching the device's runtime version. Strategy: (1) increment runtime version whenever native code changes, (2) maintain separate update branches/channels per runtime version, (3) use Expo's fingerprint for automatic compatibility detection, (4) for CodePush, use target binary version ranges. The critical failure mode: pushing a JS update that calls a native module added in a newer native version — users on older native shells crash. Always test OTA updates against the oldest supported native shell version.

Q31: What update strategies exist (on launch, on demand, background), and when would you use each?

On launch (blocking): check for update on app start, download and apply before showing UI. Guarantees latest version but adds startup latency. Use for: compliance-critical apps, apps requiring exact version parity with backend. Background (non-blocking): check and download in background, apply on next launch. No UX impact but users run old version for one session. Use for: most consumer apps. On demand: user-triggered update check (e.g., "Check for updates" button). Use for: enterprise apps where users control update timing. Hybrid: background download + prompt user to restart when update is ready. Best balance of freshness and UX. Configure via expo-updates checkAutomatically and fallbackToCacheTimeout settings.

Q32: How does CodePush handle rollback, and what triggers an automatic rollback?

CodePush tracks whether an update was successfully applied by monitoring the notifyAppReady() call. If the app crashes before calling notifyAppReady() on the first launch after an update, CodePush considers the update failed and automatically rolls back to the previous bundle on the next launch. This is the "install → crash → rollback" safety net. Manual rollback: use appcenter codepush rollback to revert the latest release on the server — new clients will download the previous version. You can also promote a known-good release to override a bad one. The rollback counter is tracked per-deployment, and you can configure the maximum number of rollback retries.

Q33: What are Apple's and Google's policies on OTA updates for React Native apps?

Apple Guideline 3.3.2: apps may download and execute interpreted code (JavaScript) as long as it doesn't change the app's primary purpose or advertised functionality, and doesn't create a store-within-a-store. React Native JS updates are explicitly permitted under this guideline. Google Play has no equivalent restriction — dynamically loaded code is allowed but must comply with the Malicious Behavior policy. Key constraints: (1) OTA updates must not add new features that bypass App Review, (2) must not change the app's category or primary function, (3) must not introduce new data collection without updated privacy disclosures, (4) must not bypass age-rating requirements. Use OTA for bug fixes, performance improvements, and minor UI tweaks — not for major feature launches.

Q34: How do you implement staged OTA rollouts with percentage-based targeting?

Both EAS Update and CodePush support staged rollouts. CodePush: appcenter codepush release -a MyApp --rollout 25 delivers to 25% of users, determined by a hash of the device ID (consistent — same device always gets the same decision). EAS Update: use channels and branches with percentage-based rollout policies. Implementation pattern: (1) release to 1% and monitor crash rate for 4-6 hours, (2) increase to 10% and monitor for 24 hours, (3) increase to 50%, then 100%. Automated rollout: write a script that checks crash-free rate from Sentry/Crashlytics and automatically advances the rollout percentage if metrics are healthy, or halts if degraded.

Q35: How does Hermes bytecode affect OTA updates?

When Hermes is enabled, OTA bundles must be compiled to Hermes bytecode (.hbc) using the same Hermes compiler version as the native shell. If you push a plain JS bundle to a Hermes-enabled app, it will fail to load. If you push bytecode compiled with Hermes v0.12 to an app running Hermes v0.11, it may crash. EAS Update handles this automatically — it compiles bundles with the correct Hermes version based on the target runtime. For CodePush, you must run hermesc manually or via a build script before publishing. The bytecode format is not backward-compatible, so each native shell version pins a specific Hermes bytecode version.

Q36: Design an OTA update system for an app with 10 million users across 5 native shell versions.

(1) Version mapping: maintain a compatibility matrix of JS bundle versions to native shell versions using runtime version identifiers. (2) Channel architecture: create channels per environment (dev, staging, production) with branches per runtime version. (3) Rollout strategy: 1% → 5% → 25% → 100% over 48 hours with automated health checks between stages. (4) Monitoring: real-time crash rate per bundle version per native shell version — alert if any combination exceeds threshold. (5) Rollback: automated rollback if crash-free rate drops below 99% within 1 hour of rollout increase. (6) CDN: serve bundles from a global CDN with edge caching for fast downloads. (7) Delta updates: only download changed chunks, not the full bundle. (8) Forced updates: for critical security fixes, show a blocking modal requiring update before app use.

App Signing

Q37: Explain the complete iOS code signing chain from Apple Root CA to app installation.

Apple Root CA (embedded in every iOS device) → signs Apple Worldwide Developer Relations CA → signs your Distribution Certificate (which contains your public key). When you build: (1) Xcode signs the app binary with your private key (stored in Keychain), (2) the signature includes the certificate chain, (3) the provisioning profile (signed by Apple) is embedded in the app. On device installation: (1) iOS verifies the provisioning profile's signature against Apple's CA, (2) verifies the app binary's signature against the certificate in the provisioning profile, (3) checks the certificate chains to Apple Root CA, (4) verifies the bundle ID matches the profile, (5) for dev/ad hoc, checks the device UDID is in the profile. Any break in this chain prevents installation.

Q38: What is the difference between a development certificate and a distribution certificate?

Development certificates sign apps for debugging on registered devices — they're tied to individual developer Apple IDs and each developer can have their own. Distribution certificates sign apps for TestFlight, App Store, and Ad Hoc distribution — they're shared across the team and limited to 3 per Apple Developer team. Key differences: (1) development certs allow debugging (attaching LLDB), distribution certs don't, (2) development profiles list specific device UDIDs, App Store distribution profiles don't, (3) development-signed apps can only run on registered devices, distribution-signed apps run on any device (via App Store) or registered devices (Ad Hoc). In CI/CD, you typically use a distribution certificate for all non-development builds.

Q39: How does Google Play App Signing work, and why should you use it?

Google Play App Signing (GPAS) separates the upload key (developer holds) from the app signing key (Google holds). You sign your AAB/APK with the upload key, upload to Play Console, Google verifies the upload signature, strips it, and re-signs with the app signing key before delivering to users. Benefits: (1) if your upload key is compromised, you can reset it without affecting the app's identity — users see no change, (2) Google can optimize delivery (APK splits from AAB) because they hold the signing key, (3) Google stores the app signing key in secure hardware (HSM). Risk: you're trusting Google with your app's identity. For new apps, GPAS is mandatory. For existing apps, you can opt in by uploading your existing key to Google.

Q40: How does Fastlane Match solve iOS certificate management for teams?

Match stores iOS certificates and provisioning profiles in an encrypted Git repository (or cloud storage). When a developer or CI machine runs match, it fetches the shared credentials from the repo, decrypts them, and installs them in the local Keychain. This ensures every team member and CI runner uses the same certificate, preventing the "too many certificates" problem and profile conflicts. Match enforces a single certificate per type per team — it will revoke existing certificates and create new ones if needed (match nuke). In CI, match runs in readonly mode (never creates new certs) with a temporary keychain that's destroyed after the build. The encryption password is the single secret that protects all credentials.

Q41: What happens when an iOS distribution certificate expires mid-release cycle?

Immediate impact: you cannot sign new builds for submission. Existing apps on the App Store continue to work — Apple re-signs App Store apps with Apple's own certificate during distribution. TestFlight builds signed with the expired certificate continue to work for their 90-day TestFlight window. Ad Hoc builds signed with the expired certificate stop working after the certificate expiry date. Recovery: (1) create a new distribution certificate in Apple Developer Portal, (2) regenerate all provisioning profiles that referenced the old certificate, (3) update CI secrets with the new certificate and profiles, (4) rebuild and resubmit. Prevention: monitor certificate expiry dates with automated alerts at 90, 60, and 30 days before expiry.

Q42: How do you handle app signing in a CI/CD pipeline securely?

iOS: (1) store the .p12 certificate and provisioning profiles in an encrypted store (Fastlane Match Git repo, or CI secrets as base64), (2) create a temporary keychain on the CI runner, (3) import the certificate into the temporary keychain, (4) copy provisioning profiles to ~/Library/MobileDevice/Provisioning Profiles/, (5) build and sign, (6) delete the temporary keychain. Android: (1) store the keystore as a base64-encoded CI secret, (2) decode it to a file at build time, (3) pass keystore path and passwords via environment variables to Gradle, (4) delete the keystore file after build. Critical: never log signing credentials, restrict secrets to protected branches, use short-lived CI runner instances that are destroyed after each build.

Q43: Explain Android's V1, V2, V3, and V4 signing schemes.

V1 (JAR signing): signs individual ZIP entries in the APK. Vulnerable because unsigned ZIP metadata (comments, ordering) can be modified. Required for Android <7.0. V2 (APK Signature Scheme): signs the entire APK as a binary blob. Any modification to the APK invalidates the signature. Required for Android 7.0+. V3: extends V2 with a proof-of-rotation structure that enables signing key rotation — you can change your signing key while maintaining update compatibility. Required for Android 9.0+. V4 (Incremental): adds a Merkle hash tree over the APK, enabling streaming installation where the OS can verify individual blocks as they're downloaded. Required for Android 11+ incremental installs. Modern APKs should include V2+V3+V4 signatures for maximum compatibility.

Q44: How do you sign React Native apps for multiple environments (dev, staging, prod)?

iOS: create separate App IDs (com.example.app.dev, .staging, .app) in Apple Developer Portal, each with its own provisioning profile. Use Xcode build configurations mapped to schemes — each scheme selects the appropriate bundle ID, entitlements file, and provisioning profile. Android: use Gradle product flavors with applicationIdSuffix — each flavor gets a distinct package name and can reference a different signing config (debug keystore for dev, release keystore for staging/prod). Both platforms: all environments can be installed simultaneously on the same device because they have different bundle IDs/package names. Use visual indicators (colored app icons, banner overlays) to distinguish environments.

Q45: What is your disaster recovery plan if the Android app signing key is compromised?

If using Google Play App Signing: the upload key can be reset by contacting Google — no user impact. If the app signing key (held by Google) is compromised, Google's HSM infrastructure makes this extremely unlikely, but if it happens, the app must be republished as a new listing with a new package name. If NOT using GPAS and the signing key is compromised: (1) the app cannot be updated on Google Play — a new listing with a new package name is required, (2) all existing users must manually install the new app, (3) you lose all Play Store ratings, reviews, and install count. Prevention: (1) always use GPAS for new apps, (2) store keystore backups in multiple secure locations (HSM, encrypted vault), (3) never store keystores in source control, (4) use Android V3 signing scheme which supports key rotation.

Release Management

Q46: How do you coordinate versioning between the JS bundle and native shell?

Use a single source of truth (package.json version) and sync it to native projects at build time. The marketing version (e.g., "3.2.1") comes from package.json and is synced to CFBundleShortVersionString (iOS) and versionName (Android). Build numbers are managed separately — iOS CFBundleVersion and Android versionCode are monotonically increasing integers managed by CI (e.g., CI build number or git commit count). For OTA updates, add a runtimeVersion that changes only when native code changes. This creates a three-level version system: marketing version (user-facing), build number (store-facing), runtime version (OTA compatibility).

Q47: Describe your ideal release branch strategy for a React Native app with biweekly releases.

Trunk-based development with release branches: (1) all feature work merges to main via short-lived PRs (1-3 days), (2) every other Monday, cut release/X.Y from main — this is the feature freeze, (3) Tuesday-Wednesday: QA tests the RC build from the release branch, (4) bug fixes go to the release branch and are cherry-picked to main, (5) Thursday: submit to stores, (6) following week: monitor rollout, increase staged rollout percentage. Hotfixes: branch from the release tag, apply minimal fix, submit as patch version. Feature flags ensure incomplete features on main are hidden in the release branch without cherry-pick gymnastics.

Q48: How do feature flags interact with mobile release management?

Feature flags decouple deployment (shipping code) from release (enabling features). In mobile, this is critical because you can't instantly deploy fixes like web. Pattern: (1) develop feature behind a flag, merge to main, (2) feature ships in the next binary release but is disabled, (3) after the binary is stable, enable the flag for a percentage of users, (4) monitor metrics, gradually increase to 100%, (5) remove the flag and dead code in a future release. Flags also enable: A/B testing (show different variants), kill switches (disable broken features without a new release), user-level targeting (beta testers get early access), and platform-specific rollouts (enable on Android first, then iOS). Use a remote flag service (LaunchDarkly, Statsig) with offline caching for mobile.

Q49: What metrics would you monitor in the first 24 hours after a release?

Stability: crash-free rate (>99.5%), ANR rate (<0.47% for Android), JS error rate, native exception rate. Performance: cold start time (P50, P95), time-to-interactive, frame drop rate, memory usage. Business: conversion rate, revenue per user, session duration, feature adoption — compare against the previous version's baseline. Operational: API error rates, network timeout rates, OTA update adoption rate. User sentiment: app store rating trend, support ticket volume, social media mentions. Set up automated alerts: (1) crash-free rate drops below 99%, (2) any single crash affects >0.1% of users, (3) ANR rate exceeds Google Play threshold, (4) P95 startup time regresses >20%.

Q50: How do you handle a critical production bug that requires an immediate fix?

Triage by layer: (1) If JS-only bug and OTA is available: push a CodePush/EAS Update fix — reaches users in minutes. (2) If feature-flag-gated: disable the flag remotely — instant mitigation. (3) If native bug or no OTA: halt staged rollout (Google Play) or pause phased release (App Store), create hotfix branch from release tag, apply minimal fix, fast-track QA (test only the fix + critical paths), submit to stores with expedited review request (Apple). (4) Communicate: post in #incidents channel, update status page, draft user-facing communication if impact is visible. Post-incident: write a post-mortem covering root cause, detection time, mitigation time, and preventive measures. Add regression tests for the specific failure mode.

Q51: Compare Google Play staged rollout vs Apple phased release.

Google Play: fully configurable percentage (any value 0-100%), can be paused and resumed, can be halted (stops at current percentage), percentage is based on device ID hash (consistent assignment), takes effect immediately. Apple: fixed 7-day schedule (1%, 2%, 5%, 10%, 20%, 50%, 100%), can be paused for up to 30 days, cannot customize percentages, only applies to automatic updates (manual "Update" button always gets latest), can be released to all users immediately at any point. Key difference: Google Play gives you granular control; Apple gives you a safety net but less flexibility. Both: cannot target specific users or segments — for that, combine with feature flags.

Q52: How do you implement a forced update mechanism in React Native?

Server-side: maintain a minimum supported version endpoint that returns { minVersion: "3.1.0", hardBlock: true, message: "..." }. Client-side: on app launch, compare the installed version against the server's minimum version. If below minimum: show a blocking modal with "Update Now" button that deep-links to the App Store / Play Store listing. Implementation: (1) use react-native-version-check or custom API call, (2) show the modal above all navigation (portal/overlay), (3) disable back button / gesture dismissal for hard blocks, (4) for soft blocks, show a dismissible banner. Caution: Apple may reject apps that force-update too aggressively — only use for genuine security or compatibility requirements, and always allow the user to reach the store listing (don't block indefinitely if the store update isn't available yet).

Monorepo Architecture

Q53: What is the "duplicate React" problem in monorepos, and how do you solve it?

When workspace packages have react as a direct dependency, the package manager may install multiple copies — one in the app's node_modules and another in the package's node_modules. React hooks use module-level state (a singleton), so two React instances means hooks in shared packages use a different React than the app, causing "Invalid hook call" errors. Solutions: (1) shared packages declare react as peerDependency only, never dependency, (2) in Metro config, use extraNodeModules to force all packages to resolve React from the app's node_modules, (3) in Yarn, use resolutions field to pin a single React version, (4) in pnpm, use .pnpmfile.cjs hooks to redirect React resolution. Always verify with yarn why react that only one copy exists.

Q54: How do you configure Metro to work with a monorepo?

Four essential configurations: (1) watchFolders: [monorepoRoot] — tells Metro to watch files outside the app directory (workspace packages). (2) resolver.nodeModulesPaths: [appNodeModules, rootNodeModules] — tells Metro where to find hoisted dependencies. (3) resolver.extraNodeModules: { react: appReactPath, 'react-native': appRNPath } — prevents duplicate singleton packages. (4) resolver.blockList: [/apps\/web\/.*/] — excludes irrelevant workspace packages from the dependency graph (prevents slow bundling and Haste module name collisions). Optional: resolver.disableHierarchicalLookup: true to prevent Metro from walking up directories and finding wrong module copies.

Q55: Compare Turborepo and Nx for React Native monorepos.

Turborepo: minimal configuration (single turbo.json), works with any package manager, focuses on task running and caching, no code generation, lightweight (~5MB). Best for: teams that want caching and parallel execution without framework lock-in. Nx: full build system with project graph, code generators, affected-command detection, framework-specific plugins (@nx/react-native), module boundary enforcement, and dependency visualization. Best for: large organizations needing governance, code generation, and deep tooling integration. Both support remote caching (Vercel/Nx Cloud). For most React Native monorepos, Turborepo is sufficient. Choose Nx if you need generators, module boundary rules, or have 20+ packages.

Q56: How do native dependencies (CocoaPods, Gradle) work in a monorepo?

Native dependencies are resolved relative to the app's native project, not the monorepo root. Challenges: (1) autolinking scans node_modules for native modules — in a monorepo, modules may be hoisted to the root node_modules, so autolinking must be told where to look. (2) CocoaPods Podfile must reference the correct node_modules path (often ../../node_modules instead of ../node_modules). (3) Android Gradle must resolve React Native's Gradle plugin from the correct location. Solutions: use react-native.config.js to specify explicit roots for native modules, update Podfile paths to reference the monorepo root's node_modules, and use settings.gradle to include the correct autolinking script path.

Q57: How do you share code between React Native and React web in a monorepo?

Three-layer approach: (1) Platform-agnostic packages: pure logic (API clients, state management, validation, utilities) — works everywhere without modification. (2) Shared UI with platform files: components with .native.tsx and .web.tsx variants sharing types and logic but with platform-specific rendering. Metro resolves .native.tsx for RN; webpack/Vite resolves .web.tsx for web. (3) Platform-specific packages: code that only makes sense on one platform (push notifications, biometrics). Use react-native-web as a compatibility layer for simple components that can share a single implementation. The shared UI package should declare react and react-native as peer dependencies, with the consuming app providing the concrete implementations.

Q58: How do you optimize CI/CD for a monorepo with React Native apps?

(1) Change detection: use dorny/paths-filter, Turborepo --filter, or Nx affected to only run jobs for changed packages. (2) Selective native builds: only trigger expensive macOS/iOS builds when native code or native dependencies change — use Expo fingerprint or custom hash of native-impacting files. (3) Caching: cache node_modules (keyed by lockfile hash), CocoaPods (keyed by Podfile.lock), Gradle (~/.gradle/caches), Turborepo remote cache. (4) Parallelization: run lint, typecheck, and unit tests in parallel on cheap Linux runners; only use expensive macOS runners for iOS builds. (5) Build matrix: test shared packages once, not once per consuming app. (6) Incremental builds: Turborepo/Nx only rebuild packages whose inputs changed.

Q59: What governance patterns do you need for a large-scale monorepo?

(1) CODEOWNERS: require team approval for changes to their packages — packages/ui/* @design-system-team. (2) Dependency constraints: enforce that shared packages never import from app packages (circular dependency prevention) — use Nx module boundaries or ESLint import rules. (3) API review process: changes to shared package public APIs require an RFC or design doc. (4) Versioning policy: internal packages use workspace:* protocol (no semver), published packages use semver with changesets. (5) CI gates: shared package changes must pass all consuming apps' tests before merge. (6) Documentation: each package has a README with API docs, usage examples, and ownership information.

Q60: How would you migrate from separate repos to a monorepo for an existing React Native project?

Phase 1 (Foundation): create the monorepo structure with workspace config, shared TypeScript/ESLint configs, and Turborepo/Nx setup. Add CI/CD for the monorepo. Phase 2 (First app): move the primary React Native app using git subtree add or git filter-repo to preserve history. Verify builds, tests, and CI work. Phase 3 (Extract packages): identify shared code, extract into workspace packages (packages/ui, packages/api-client). Update imports in the app. Phase 4 (Additional apps): move web app, admin tools, etc. into the monorepo one at a time. Phase 5 (Decommission): archive old repos, update all documentation and CI links. Key risk: CI/CD complexity — solve it completely in Phase 1 before migrating production apps. Run both structures in parallel during transition.
