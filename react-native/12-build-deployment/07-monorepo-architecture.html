1. Why Monorepo for React Native

A monorepo houses multiple packages (apps, libraries, tools) in a single repository. For React Native projects, this is particularly valuable because code is shared across mobile (iOS/Android), web, backend-for-frontend, and design system packages.

Monorepo vs Polyrepo for React Native ══════════════════════════════════════ Polyrepo (separate repos) Monorepo (single repo) ───────────────────────── ────────────────────── repo: mobile-app repo: product repo: web-app ├── apps/ repo: shared-ui │ ├── mobile/ (React Native) repo: api-client │ ├── web/ (Next.js) repo: design-tokens │ └── admin/ (React) ├── packages/ Problems: │ ├── ui/ (shared components) ✗ Version sync across repos │ ├── api-client/ (shared API layer) ✗ Diamond dependency conflicts │ ├── utils/ (shared utilities) ✗ Cross-repo refactoring is painful │ └── tokens/ (design tokens) ✗ CI/CD duplication └── tools/ ✗ Inconsistent tooling configs └── eslint-config/ Benefits: ✓ Atomic cross-package changes ✓ Single version of shared deps ✓ Unified CI/CD pipeline ✓ Consistent tooling and linting ✓ Easier code review across boundaries
Architect Insight

The decision to adopt a monorepo is an organizational architecture decision, not just a technical one. Monorepos work best when teams share code frequently and need atomic cross-package changes. If teams are fully autonomous with minimal code sharing, the coordination overhead of a monorepo may outweigh the benefits. The key question: "How often do changes span multiple packages?"

2. Workspace Setup

2.1 Yarn Workspaces

// package.json (root)
{
  "name": "my-product",
  "private": true,
  "workspaces": [
    "apps/*",
    "packages/*",
    "tools/*"
  ],
  "engines": {
    "node": ">=18.0.0"
  },
  "packageManager": "yarn@4.1.0"
}

# Yarn workspace commands
yarn install                          # Install all workspace deps
yarn workspace @product/mobile start  # Run command in specific workspace
yarn workspaces foreach run build     # Run build in all workspaces
yarn workspaces list                  # List all workspaces

2.2 pnpm Workspaces

# pnpm-workspace.yaml (root)
packages:
  - 'apps/*'
  - 'packages/*'
  - 'tools/*'

# .npmrc — critical for React Native compatibility
node-linker=hoisted    # Required: RN expects flat node_modules
shamefully-hoist=true  # Hoist all deps (RN native modules need this)
strict-peer-dependencies=false

# pnpm workspace commands
pnpm install                                 # Install all
pnpm --filter @product/mobile start          # Run in specific package
pnpm --filter @product/mobile... build       # Build package + all deps
pnpm --filter "...@product/mobile" test      # Test package + all dependents
pnpm + React Native: Hoisting Required

pnpm's default strict isolation (symlinked node_modules) breaks React Native because Metro bundler and native autolinking expect a flat node_modules structure. You must set node-linker=hoisted or shamefully-hoist=true. This negates some of pnpm's strictness benefits but is required for React Native compatibility. Yarn Berry with nodeLinker: node-modules has the same requirement.

4. Shared Packages

4.1 Shared UI Package

// packages/ui/package.json
{
  "name": "@product/ui",
  "version": "0.0.0",
  "main": "src/index.ts",
  "types": "src/index.ts",
  "react-native": "src/index.ts",
  "peerDependencies": {
    "react": ">=18.0.0",
    "react-native": ">=0.72.0"
  }
}

// packages/ui/src/Button/Button.tsx (shared logic)
import { ButtonProps } from './types';

export function useButtonLogic(props: ButtonProps) {
  const [pressed, setPressed] = useState(false);
  const handlePress = useCallback(() => {
    if (!props.disabled) props.onPress?.();
  }, [props.disabled, props.onPress]);
  return { pressed, setPressed, handlePress };
}

// packages/ui/src/Button/Button.native.tsx
import { Pressable, Text } from 'react-native';
import { useButtonLogic } from './Button';

export function Button(props: ButtonProps) {
  const { handlePress } = useButtonLogic(props);
  return (
    <Pressable onPress={handlePress}>
      <Text>{props.label}</Text>
    </Pressable>
  );
}

4.2 Shared API Client

// packages/api-client/src/index.ts
// Platform-agnostic API client using fetch (available in RN + web)
export class ApiClient {
  private baseUrl: string;
  private getToken: () => Promise<string | null>;

  constructor(config: ApiConfig) {
    this.baseUrl = config.baseUrl;
    this.getToken = config.getToken;
  }

  async request<T>(endpoint: string, options?: RequestOptions): Promise<T> {
    const token = await this.getToken();
    const response = await fetch(`${this.baseUrl}${endpoint}`, {
      headers: {
        'Content-Type': 'application/json',
        ...(token && { 'Authorization': `Bearer ${token}` }),
      },
      ...options,
    });
    if (!response.ok) throw new ApiError(response);
    return response.json();
  }
}

// Consumed identically in mobile and web:
// import { ApiClient } from '@product/api-client';

5. Metro Configuration for Monorepo

Metro bundler must be configured to resolve modules from the monorepo root, watch files across workspace packages, and handle symlinks created by the package manager.

5.1 Metro Config for Workspaces

// apps/mobile/metro.config.js
const path = require('path');
const { getDefaultConfig, mergeConfig } = require('@react-native/metro-config');

const monorepoRoot = path.resolve(__dirname, '../..');

const config = {
  watchFolders: [monorepoRoot],

  resolver: {
    nodeModulesPaths: [
      path.resolve(__dirname, 'node_modules'),
      path.resolve(monorepoRoot, 'node_modules'),
    ],
    disableHierarchicalLookup: true,

    extraNodeModules: {
      'react': path.resolve(__dirname, 'node_modules/react'),
      'react-native': path.resolve(__dirname, 'node_modules/react-native'),
    },

    blockList: [
      new RegExp(`${monorepoRoot}/apps/web/.*`),
    ],
  },

  transformer: {
    getTransformOptions: async () => ({
      transform: {
        experimentalImportSupport: false,
        inlineRequires: true,
      },
    }),
  },
};

module.exports = mergeConfig(getDefaultConfig(__dirname), config);

5.2 Common Metro Monorepo Issues

Issue Symptom Solution
Duplicate React instances "Invalid hook call" error extraNodeModules pointing to single React copy
Module not found Cannot resolve @product/ui Add monorepo root to watchFolders
Slow bundling Metro watches entire monorepo blockList to exclude irrelevant packages
Symlink resolution Metro can't follow workspace symlinks unstable_enableSymlinks: true (Metro 0.79+)
Haste module naming Duplicate module name collision blockList for conflicting packages
The Duplicate React Problem

The most common monorepo issue: workspace packages resolve their own copy of react instead of the app's copy, resulting in two React instances. Hooks break because they rely on a module-level singleton. Always pin React resolution to the app's node_modules via extraNodeModules and ensure workspace packages declare react as a peerDependency, never a direct dependency.

6. Native Dependencies in Monorepo

React Native libraries with native code (CocoaPods, Gradle) require special handling in monorepos because autolinking searches for native modules relative to the app's node_modules.

// react-native.config.js (in apps/mobile/)
module.exports = {
  project: {
    ios: { sourceDir: './ios' },
    android: { sourceDir: './android' },
  },
  dependencies: {
    // Explicitly tell autolinking where to find native modules
    // that are hoisted to monorepo root
    'react-native-reanimated': {
      root: path.resolve(__dirname, '../../node_modules/react-native-reanimated'),
    },
  },
};

# ios/Podfile — must reference monorepo root for pods
require_relative '../../node_modules/react-native/scripts/react_native_pods'
require_relative '../../node_modules/@react-native-community/cli-platform-ios/native_modules'

platform :ios, min_ios_version_supported

target 'MyApp' do
  config = use_native_modules!(packages_to_skip: [])
  use_react_native!(path: '../../node_modules/react-native')
end

7. Turborepo Integration

Turborepo is a build system for monorepos that provides task caching, parallel execution, and dependency-aware task ordering. It's lightweight and integrates with any package manager.

// turbo.json (root)
{
  "$schema": "https://turbo.build/schema.json",
  "globalDependencies": ["tsconfig.base.json"],
  "tasks": {
    "build": {
      "dependsOn": ["^build"],
      "outputs": ["dist/**", "build/**"],
      "cache": true
    },
    "test": {
      "dependsOn": ["build"],
      "cache": true,
      "inputs": ["src/**", "test/**", "jest.config.*"]
    },
    "lint": {
      "cache": true,
      "inputs": ["src/**", ".eslintrc.*"]
    },
    "typecheck": {
      "dependsOn": ["^build"],
      "cache": true
    },
    "@product/mobile#start": {
      "dependsOn": ["^build"],
      "cache": false,
      "persistent": true
    }
  }
}

# Turborepo commands
turbo run build                          # Build all packages in dependency order
turbo run build --filter=@product/ui     # Build specific package + deps
turbo run build --filter=@product/ui...  # Build package + all dependents
turbo run test lint --parallel           # Run independent tasks in parallel
turbo run build --dry-run                # Show execution plan without running
Turborepo Task Execution ═════════════════════════ turbo run build --filter=@product/mobile Execution Graph (respects dependsOn: ["^build"]): ───────────────────────────────────────────────── @product/tokens ──build──→ ┐ ├──→ @product/ui ──build──→ ┐ @product/utils ──build───→ ┘ │ ├──→ @product/mobile ──build @product/api-client ──build──→ ──────────────────────── ┘ Cache Behavior: ─────────────── @product/tokens → CACHE HIT (no changes since last build) @product/utils → CACHE HIT @product/ui → CACHE MISS (src/Button.tsx changed) @product/mobile → CACHE MISS (dependency changed) Result: Only @product/ui and @product/mobile actually rebuild
Remote Caching

Turborepo supports remote caching via Vercel or self-hosted endpoints. When developer A builds @product/ui, the cache artifact is uploaded. When developer B (or CI) runs the same build with the same inputs, it downloads the cached output instead of rebuilding. This can reduce CI build times by 40-80% for large monorepos. Enable with turbo login + turbo link.

8. Nx Integration

Nx is a more full-featured monorepo tool that provides project graph visualization, code generators, affected-command detection, and deep framework integration.

// nx.json (root)
{
  "targetDefaults": {
    "build": {
      "dependsOn": ["^build"],
      "cache": true
    },
    "test": {
      "cache": true,
      "inputs": ["default", "^default"]
    }
  },
  "namedInputs": {
    "default": ["{projectRoot}/**/*"],
    "production": [
      "default",
      "!{projectRoot}/**/*.test.ts",
      "!{projectRoot}/**/*.spec.ts"
    ]
  }
}

# Nx commands
npx nx build @product/mobile              # Build specific project
npx nx affected --target=test             # Test only affected projects
npx nx graph                              # Visualize dependency graph
npx nx run-many --target=lint --parallel  # Lint all projects in parallel

8.1 Turborepo vs Nx Comparison

Aspect Turborepo Nx
Philosophy Minimal, task runner only Full-featured build system
Configuration Single turbo.json nx.json + per-project project.json
Code generation None (use external tools) Built-in generators + plugins
Affected detection Via --filter + git Built-in nx affected
Dependency graph Implicit from package.json Explicit + implicit, visualizable
Remote cache Vercel or custom Nx Cloud or custom
Learning curve Low Medium-High
RN support Works with any setup Official @nx/react-native plugin

10. Testing in Monorepo

// jest.config.base.js (root) — shared Jest configuration
module.exports = {
  preset: 'react-native',
  moduleNameMapper: {
    '^@product/ui(.*)$': '<rootDir>/../../packages/ui/src$1',
    '^@product/utils(.*)$': '<rootDir>/../../packages/utils/src$1',
    '^@product/api-client(.*)$': '<rootDir>/../../packages/api-client/src$1',
  },
  transformIgnorePatterns: [
    'node_modules/(?!(react-native|@react-native|@react-navigation)/)',
  ],
  coverageThreshold: {
    global: { branches: 80, functions: 80, lines: 80 },
  },
};

// Run tests for affected packages only (CI optimization)
# With Turborepo:
turbo run test --filter=...[origin/main]

# With Nx:
npx nx affected --target=test --base=origin/main

11. CI/CD for Monorepo

# GitHub Actions — monorepo-aware CI
name: CI
on:
  pull_request:
    branches: [main]

jobs:
  detect-changes:
    runs-on: ubuntu-latest
    outputs:
      mobile: ${{ steps.filter.outputs.mobile }}
      web: ${{ steps.filter.outputs.web }}
      packages: ${{ steps.filter.outputs.packages }}
    steps:
      - uses: actions/checkout@v4
      - uses: dorny/paths-filter@v3
        id: filter
        with:
          filters: |
            mobile:
              - 'apps/mobile/**'
              - 'packages/**'
            web:
              - 'apps/web/**'
              - 'packages/**'
            packages:
              - 'packages/**'

  test-packages:
    needs: detect-changes
    if: needs.detect-changes.outputs.packages == 'true'
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: yarn install --frozen-lockfile
      - run: turbo run test lint typecheck --filter='./packages/*'

  build-mobile:
    needs: detect-changes
    if: needs.detect-changes.outputs.mobile == 'true'
    runs-on: macos-latest
    steps:
      - uses: actions/checkout@v4
      - run: yarn install --frozen-lockfile
      - run: turbo run build --filter=@product/mobile...
      - run: cd apps/mobile/ios && pod install
      - run: turbo run test --filter=@product/mobile
CI Cost Optimization

macOS CI runners cost 10x more than Linux runners. Only run iOS builds when mobile or shared package code changes. Use change detection (paths-filter, Turborepo affected, Nx affected) to skip unnecessary jobs. Cache node_modules, CocoaPods, and Gradle dependencies aggressively. For large monorepos, Turborepo remote caching can reduce CI time by 60-80% by sharing build artifacts across PR builds.

12. Architect Considerations

Monorepo Governance

At scale, monorepos need governance: (1) CODEOWNERS files to enforce review requirements per package, (2) package boundary enforcement via ESLint rules (no direct imports across app boundaries), (3) dependency constraints (shared packages cannot depend on app packages), (4) versioning policy (internal packages use 0.0.0 with workspace protocol, published packages use semver), (5) contribution guidelines for shared packages (RFC process for API changes).

Migration Strategy: Polyrepo to Monorepo

Migrate incrementally: (1) start with a new monorepo containing one app, (2) extract shared code into workspace packages, (3) move additional apps one at a time, (4) preserve git history using git subtree or git filter-repo, (5) set up CI/CD for the monorepo before migrating production apps, (6) run both structures in parallel during transition. The biggest risk is CI/CD complexity — solve it before migrating the second app.
