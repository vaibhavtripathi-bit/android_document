1. Expo Platform Overview

Expo has evolved from a "simplified React Native wrapper" into a comprehensive platform for building, deploying, and updating React Native applications. Modern Expo (SDK 49+) is no longer a walled garden — it's a set of composable tools that work with any React Native project, including bare projects with custom native code.

Architect Insight

The most important mental model shift for Staff engineers evaluating Expo: it's no longer "Expo OR bare React Native." Modern Expo is a layer on top of React Native. You can use EAS Build with a bare project, use Expo modules without Expo Go, and use config plugins to modify native code without ejecting. The question is not "should we use Expo?" but "which Expo tools add value to our specific project?"

2. Expo Go

Expo Go is a pre-built native app (available on App Store and Google Play) that can load any Expo project over the network. It contains a fixed set of native modules from the Expo SDK, enabling rapid prototyping without any native compilation.

Aspect Expo Go Development Build
Native modules Fixed set (Expo SDK only) Any native module
Custom native code Not supported Fully supported
Build required No (download from store) Yes (once per native change)
Setup time ~1 minute ~10-15 minutes (first build)
Push notifications Expo Push only Any provider (FCM, APNs)
App size ~70MB (includes all SDK modules) Custom (only included modules)
Best for Prototyping, learning, demos Production development
Expo Go Limitations

Expo Go cannot run libraries with custom native code (react-native-maps with Google Maps, react-native-ble-plx, custom TurboModules). Starting with SDK 49, Expo recommends Development Builds over Expo Go for any project that will ship to production. Expo Go remains valuable for teaching, prototyping, and Expo SDK-only projects.

3. Development Builds

Development Builds are custom debug builds of your app that include the expo-dev-client package. They provide the same developer experience as Expo Go (QR code scanning, fast refresh) but with your project's specific native modules included.

// Install expo-dev-client
// npx expo install expo-dev-client

// Build a development build locally
// npx expo run:ios
// npx expo run:android

// Or build in the cloud with EAS
// eas build --profile development --platform ios

// eas.json — development build profile
{
  "build": {
    "development": {
      "developmentClient": true,
      "distribution": "internal",
      "ios": {
        "simulator": true
      }
    },
    "development-device": {
      "developmentClient": true,
      "distribution": "internal",
      "ios": {
        "simulator": false
      }
    }
  }
}
Development Build Workflow ══════════════════════════ Native Code Change? │ ├── YES → Rebuild dev client (eas build / npx expo run:ios) │ ~10-15 min (cloud) or ~5 min (local) │ └── NO (JS only) → Hot reload via Metro ~instant

4. EAS Build

EAS Build is Expo's cloud build service. It compiles iOS and Android binaries on Expo's infrastructure, eliminating the need for local Xcode/Android Studio installations and macOS machines for iOS builds.

// eas.json — complete build configuration
{
  "cli": { "version": ">= 5.0.0" },
  "build": {
    "development": {
      "developmentClient": true,
      "distribution": "internal",
      "env": {
        "APP_ENV": "development"
      }
    },
    "preview": {
      "distribution": "internal",
      "env": {
        "APP_ENV": "staging"
      },
      "android": {
        "buildType": "apk"
      },
      "ios": {
        "enterpriseProvisioning": "adhoc"
      }
    },
    "production": {
      "env": {
        "APP_ENV": "production"
      },
      "android": {
        "buildType": "app-bundle"
      },
      "ios": {
        "autoIncrement": true
      },
      "autoSubmit": true,
      "submitProfile": "production"
    }
  },
  "submit": {
    "production": {
      "ios": {
        "appleId": "team@company.com",
        "ascAppId": "1234567890",
        "appleTeamId": "ABCDE12345"
      },
      "android": {
        "serviceAccountKeyPath": "./google-services.json",
        "track": "internal"
      }
    }
  }
}
EAS Build vs Local Builds vs CI

EAS Build abstracts away native toolchain management — no Xcode version mismatches, no Android SDK issues, no macOS requirement for iOS builds. The trade-off is build time (queue wait + build) and cost. For teams without dedicated mobile CI infrastructure, EAS Build is often the most cost-effective option. For large teams with existing CI (GitHub Actions + macOS runners), local/CI builds may be faster and cheaper at scale.

5. EAS Submit

EAS Submit automates uploading binaries to the App Store and Google Play. It handles the authentication, validation, and upload process that traditionally requires manual interaction with App Store Connect and Google Play Console.

// Submit to both stores after a production build
// eas submit --platform all --latest

// Submit a specific build
// eas submit --platform ios --id BUILD_ID

// Submit a local binary
// eas submit --platform android --path ./app-release.aab

// Automated pipeline: build → submit → update
// eas build --auto-submit --platform all

6. EAS Update

EAS Update is Expo's over-the-air update service, replacing the older expo-updates classic updates. It uses a branch-based model where updates are published to named branches, and builds are configured to check specific branches for updates.

// Publish an update to the "production" branch
// eas update --branch production --message "Fix checkout crash"

// app.json / app.config.js — update configuration
{
  "expo": {
    "updates": {
      "url": "https://u.expo.dev/PROJECT_ID",
      "enabled": true,
      "fallbackToCacheTimeout": 0,
      "checkAutomatically": "ON_LOAD",
      "requestHeaders": {
        "expo-channel-name": "production"
      }
    },
    "runtimeVersion": {
      "policy": "appVersion"
    }
  }
}
EAS Update Branch Model ═══════════════════════ Build v1.0 ──────→ Channel: production ──→ Branch: production │ ├── Update 1 (hotfix) ├── Update 2 (feature) └── Update 3 (rollback) Build v1.1 ──────→ Channel: production ──→ Branch: production-v1.1 │ └── Update 1 (hotfix) Build v1.0-rc ───→ Channel: staging ────→ Branch: staging │ └── Update 1 (test fix) Runtime Version determines compatibility: Build v1.0 (runtime: 1.0) only receives updates with runtime: 1.0 Build v1.1 (runtime: 1.1) only receives updates with runtime: 1.1

7. Config Plugins

Config plugins are the mechanism for modifying native project files (Info.plist, AndroidManifest.xml, build.gradle, Podfile) from JavaScript. They run during expo prebuild and enable native configuration without directly editing native files.

// app.config.js — using config plugins
module.exports = {
  expo: {
    plugins: [
      // Built-in plugins (from Expo SDK packages)
      ['expo-camera', { cameraPermission: 'Allow $(PRODUCT_NAME) to access camera' }],
      ['expo-location', { locationAlwaysAndWhenInUsePermission: 'Allow $(PRODUCT_NAME) to use your location' }],

      // Third-party plugins
      '@react-native-firebase/app',
      ['expo-build-properties', {
        ios: { deploymentTarget: '15.0', flipper: false },
        android: { compileSdkVersion: 34, minSdkVersion: 24 },
      }],

      // Custom inline plugin
      withCustomSplashScreen,
    ],
  },
};

// Custom config plugin — modifying Info.plist
const { withInfoPlist } = require('expo/config-plugins');

function withCustomSplashScreen(config) {
  return withInfoPlist(config, (config) => {
    config.modResults.UILaunchStoryboardName = 'CustomSplash';
    config.modResults.UIStatusBarStyle = 'UIStatusBarStyleLightContent';
    return config;
  });
}

// Custom config plugin — modifying AndroidManifest.xml
const { withAndroidManifest } = require('expo/config-plugins');

function withDeepLinks(config, { scheme }) {
  return withAndroidManifest(config, (config) => {
    const mainActivity =
      config.modResults.manifest.application[0].activity[0];
    mainActivity['intent-filter'].push({
      action: [{ $: { 'android:name': 'android.intent.action.VIEW' } }],
      category: [
        { $: { 'android:name': 'android.intent.category.DEFAULT' } },
        { $: { 'android:name': 'android.intent.category.BROWSABLE' } },
      ],
      data: [{ $: { 'android:scheme': scheme } }],
    });
    return config;
  });
}
Architect Insight

Config plugins are the key innovation that makes Expo viable for complex production apps. They provide a declarative, version-controlled, reproducible way to configure native projects. Instead of manually editing Xcode project files (which cause merge conflicts), you express native configuration as JavaScript functions. The native project becomes a build artifact generated from config, not a source of truth. This is a paradigm shift that Staff engineers must evaluate carefully.

8. Prebuild System (expo prebuild)

expo prebuild generates the ios/ and android/ directories from your app.json configuration and config plugins. This is the core of Expo's "continuous native generation" approach — native projects are generated, not manually maintained.

They are regenerated on every build.

9. Managed vs Bare Workflow

The distinction between "managed" and "bare" has blurred significantly. Modern Expo uses the term "Continuous Native Generation" (CNG) for what was previously "managed workflow." The key question is whether you commit native directories to git.

Aspect CNG (Managed / Prebuild) Bare (Manual Native)
Native dirs in git No (gitignored, generated) Yes (committed, manually maintained)
Native config via Config plugins + app.json Direct file editing
Upgrade path Change SDK version, re-prebuild Manual diff + patch (react-native upgrade helper)
Custom native code Via config plugins or local modules Direct modification
Merge conflicts Rare (no .pbxproj in git) Frequent (.pbxproj is notorious)
Reproducibility High (deterministic generation) Lower (manual drift over time)
Flexibility Constrained by plugin API Unlimited
Team onboarding Fast (no native toolchain needed) Slow (Xcode + Android Studio setup)
The Hybrid Approach

Many production teams use a hybrid: CNG for most native configuration, with a few custom config plugins for complex native integrations. If a config plugin doesn't exist for your need, you write one — this is preferable to abandoning CNG entirely. The threshold for switching to bare should be high: only when the config plugin API fundamentally cannot express your native requirement.

10. Expo Router

Expo Router brings file-system-based routing to React Native, inspired by Next.js. Routes are defined by file structure in the app/ directory, with automatic deep linking, type-safe navigation, and universal (web + native) support.

// File structure → Route mapping
// app/
//   _layout.tsx          → Root layout (Stack/Tabs)
//   index.tsx            → / (home screen)
//   settings.tsx         → /settings
//   (auth)/
//     _layout.tsx        → Auth group layout
//     login.tsx          → /login
//     register.tsx       → /register
//   (tabs)/
//     _layout.tsx        → Tab navigator
//     home.tsx           → /home (tab)
//     profile.tsx        → /profile (tab)
//   users/
//     [id].tsx           → /users/:id (dynamic route)
//     [id]/
//       posts.tsx        → /users/:id/posts
//   +not-found.tsx       → 404 handler
//   +html.tsx            → Custom HTML wrapper (web)

// app/_layout.tsx — Root layout
import { Stack } from 'expo-router';

export default function RootLayout() {
  return (
    <Stack>
      <Stack.Screen name="index" options={{ title: 'Home' }} />
      <Stack.Screen name="(tabs)" options={{ headerShown: false }} />
      <Stack.Screen name="users/[id]" options={{ title: 'User Profile' }} />
    </Stack>
  );
}

// Type-safe navigation
import { router } from 'expo-router';

router.push('/users/123');
router.push({ pathname: '/users/[id]', params: { id: '123' } });
router.replace('/login');
router.back();

11. Expo Modules

Expo Modules API is a modern framework for writing native modules that work with both the old and new React Native architecture. It uses Swift (iOS) and Kotlin (Android) with a declarative DSL, replacing the verbose Objective-C/Java bridge code.

// Swift — iOS native module using Expo Modules API
import ExpoModulesCore

public class MyAnalyticsModule: Module {
  public func definition() -> ModuleDefinition {
    Name("MyAnalytics")

    Function("trackEvent") { (name: String, properties: [String: Any]) -> Void in
      Analytics.shared.track(name, properties: properties)
    }

    AsyncFunction("getDeviceId") { () -> String in
      return UIDevice.current.identifierForVendor?.uuidString ?? ""
    }

    Events("onAnalyticsReady")

    OnCreate {
      Analytics.shared.initialize()
      self.sendEvent("onAnalyticsReady", ["initialized": true])
    }
  }
}

13. Architect Considerations

Expo Adoption Decision Framework

The decision to adopt Expo tools should be evaluated per-tool, not all-or-nothing. (1) EAS Build: adopt if you lack CI infrastructure or want to eliminate macOS CI runner costs. (2) Config plugins + CNG: adopt if .pbxproj merge conflicts are a team pain point and your native customizations can be expressed as plugins. (3) Expo Router: adopt if you need deep linking, universal (web + native) routing, or type-safe navigation. (4) EAS Update: adopt if you need OTA updates with a managed service (vs self-hosted CodePush). (5) Expo Modules: adopt for new native modules — the Swift/Kotlin DSL is dramatically simpler than the legacy bridge API.

Common L6 Interview Pitfall

Candidates often hold outdated views of Expo — either dismissing it as "not for production" (pre-2022 thinking) or treating it as a magic solution that eliminates all native complexity. A Staff-level answer demonstrates nuanced evaluation: Expo's CNG model trades flexibility for reproducibility, EAS Build trades control for convenience, and config plugins trade direct native access for declarative configuration. Articulate the specific trade-offs for your project's constraints.

Key Architectural Decisions

  • CNG vs bare: CNG eliminates .pbxproj merge conflicts and simplifies upgrades, but requires all native config to be expressible as plugins. Evaluate your native integration complexity.
  • EAS vs self-hosted CI: EAS Build costs $99+/month for teams. Compare against macOS CI runner costs ($40-200/month per runner) and maintenance overhead.
  • Expo Router vs React Navigation: Expo Router builds on React Navigation but adds file-based routing and deep linking. It's opinionated — evaluate whether the conventions fit your app's navigation patterns.
  • Vendor lock-in: Expo SDK libraries work without Expo services. EAS services have alternatives (CodePush, Fastlane, GitHub Actions). The lock-in risk is lower than it appears, but migration cost is non-zero.
