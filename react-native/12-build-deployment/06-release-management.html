1. Versioning Strategy

React Native apps have a dual versioning problem: the JavaScript bundle follows semver conventions, while native builds require platform-specific version numbers that the app stores enforce. These two version systems must be coordinated.

1.1 Version Number Anatomy

React Native Version Components в•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җ Marketing Version (user-facing) Build Number (store-facing) в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җ в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җ iOS: CFBundleShortVersionString CFBundleVersion "3.2.1" "142" Android: versionName versionCode "3.2.1" 142 Semver Mapping: в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җ MAJOR.MINOR.PATCH 3 . 2 . 1 MAJOR вҶ’ Breaking changes, major redesign, platform version bump MINOR вҶ’ New features, non-breaking API additions PATCH вҶ’ Bug fixes, performance improvements Build Number Rules: в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җ iOS: String, must increase for each TestFlight/App Store upload Can be "3.2.1.142" or just "142" вҖ” must be unique per version Android: Integer, must strictly increase across ALL versions versionCode 142 вҶ’ next must be вүҘ 143, even if versionName changes

1.2 Automated Version Bumping

// package.json вҖ” source of truth for marketing version
{
  "name": "my-rn-app",
  "version": "3.2.1"
}

# Fastlane version management
lane :bump_version do |options|
  bump_type = options[:type] || "patch"  # major | minor | patch

  # Bump package.json version
  version = npm_version_bump(type: bump_type)

  # Sync to iOS
  increment_version_number(
    version_number: version,
    xcodeproj: "ios/MyApp.xcodeproj"
  )
  increment_build_number(xcodeproj: "ios/MyApp.xcodeproj")

  # Sync to Android
  android_set_version_name(version_name: version, gradle_file: "android/app/build.gradle")
  android_set_version_code(gradle_file: "android/app/build.gradle")

  # Commit and tag
  git_commit(path: ["package.json", "ios/", "android/"], message: "Bump version to #{version}")
  add_git_tag(tag: "v#{version}")
end
Android versionCode Pitfall

Android's versionCode must be a strictly increasing integer across all builds ever uploaded to Google Play. If you use a scheme like MAJOR * 10000 + MINOR * 100 + PATCH (e.g., 30201 for 3.2.1), you can never release more than 99 patches per minor version. A safer scheme is to use a monotonically increasing counter stored in CI, or derive from the git commit count.

3. Feature Flags for Gradual Rollout

Feature flags decouple deployment from release. Code ships to production behind flags, then flags are toggled remotely to enable features for specific user segments вҖ” without a new app store submission.

3.1 Feature Flag Architecture

// Feature flag service abstraction
interface FeatureFlagService {
  isEnabled(flag: string, context?: UserContext): boolean;
  getVariant(flag: string, context?: UserContext): string;
  getAllFlags(): Record<string, boolean>;
}

// React hook for feature flags
function useFeatureFlag(flagKey: string): boolean {
  const client = useFeatureFlagClient();
  const [enabled, setEnabled] = useState(
    () => client.isEnabled(flagKey)
  );

  useEffect(() => {
    const unsubscribe = client.onFlagChange(flagKey, setEnabled);
    return unsubscribe;
  }, [flagKey]);

  return enabled;
}

// Usage in components
function PaymentScreen() {
  const showNewCheckout = useFeatureFlag('new-checkout-flow');

  if (showNewCheckout) {
    return <NewCheckoutFlow />;
  }
  return <LegacyCheckoutFlow />;
}

3.2 Feature Flag Providers Comparison

Provider RN SDK Offline Support Targeting A/B Testing
LaunchDarkly Official Yes (cached) User/segment/rule Yes (experiments)
Firebase Remote Config Official (via RNFB) Yes (cached) User property/audience Via A/B Testing
Statsig Official Yes User/custom Yes (gates + experiments)
Unleash (OSS) Community Yes Strategy-based Via variants

Offline-First Flag Evaluation

Mobile apps must evaluate flags offline. The flag SDK should cache the last-known flag state locally and use it when the network is unavailable. On app launch, fetch fresh flags in the background and apply them on the next screen transition вҖ” never block the UI on a flag fetch. This means users may see stale flag values for one session after a flag change.

4. Staged Rollouts

Both Google Play and Apple provide mechanisms to gradually roll out releases to a percentage of users, reducing blast radius if a release contains bugs.

4.1 Google Play Staged Rollout

# Fastlane staged rollout for Google Play
lane :staged_rollout do |options|
  rollout = options[:percentage] || 0.1  # Start at 10%

  upload_to_play_store(
    track: "production",
    rollout: rollout.to_s,
    aab: "android/app/build/outputs/bundle/productionRelease/app-production-release.aab",
    skip_upload_metadata: true,
    skip_upload_images: true
  )
end

# Typical rollout schedule:
# Day 0: 1% вҶ’ monitor crash rate for 24h
# Day 1: 5% вҶ’ monitor ANR rate, user feedback
# Day 2: 20% вҶ’ monitor performance metrics
# Day 3: 50% вҶ’ monitor revenue/conversion metrics
# Day 5: 100% вҶ’ full rollout

# Halt rollout if metrics degrade
lane :halt_rollout do
  upload_to_play_store(
    track: "production",
    rollout: "0",  # Halts the staged rollout
    version_code: ENV["HALT_VERSION_CODE"]
  )
end

4.2 TestFlight / App Store Phased Release

# iOS phased release via Fastlane
lane :submit_for_review do
  deliver(
    submit_for_review: true,
    automatic_release: false,
    phased_release: true,  # Enable 7-day phased release
    submission_information: {
      add_id_info_uses_idfa: false
    }
  )
end

# Apple's phased release schedule (fixed, not configurable):
# Day 1:  1%
# Day 2:  2%
# Day 3:  5%
# Day 4: 10%
# Day 5: 20%
# Day 6: 50%
# Day 7: 100%
# Note: Users who manually check for updates get the new version immediately
Phased Release Limitations

Apple's phased release only applies to automatic updates вҖ” users who manually tap "Update" in the App Store always get the latest version. There's no way to target specific user segments. Google Play's staged rollout is more flexible (any percentage, pausable, resumable) but also cannot target specific users. For true user-level targeting, combine store rollouts with feature flags.

5. Release Notes Automation

# Generate release notes from conventional commits
# Requires conventional commit format: type(scope): description

# Using standard-version / release-please
npx standard-version --release-as minor

# Custom script to generate store-friendly release notes
function generateReleaseNotes() {
  const commits = execSync(
    `git log v${lastVersion}..HEAD --pretty=format:"%s"`
  ).toString().split('\n');

  const features = commits.filter(c => c.startsWith('feat'));
  const fixes = commits.filter(c => c.startsWith('fix'));

  return [
    features.length && `What's New:\n${features.map(formatNote).join('\n')}`,
    fixes.length && `Bug Fixes:\n${fixes.map(formatNote).join('\n')}`,
  ].filter(Boolean).join('\n\n');
}

// Store release notes have character limits:
// App Store: 4000 characters
// Google Play: 500 characters per language

6. Post-Release Monitoring

The release isn't done when the binary ships вҖ” it's done when metrics confirm stability. A Staff engineer's release process includes automated monitoring gates.

6.1 Key Metrics to Monitor

Crash-free rate > 99.5% < 99.0% Crashlytics, Sentry, Bugsnag

6.2 Automated Release Health Dashboard

// Post-release monitoring automation
interface ReleaseHealthCheck {
  version: string;
  platform: 'ios' | 'android';
  metrics: {
    crashFreeRate: number;
    anrRate: number;
    jsErrorRate: number;
    coldStartP50: number;
    coldStartP95: number;
    adoptionRate: number;
  };
  status: 'healthy' | 'degraded' | 'critical';
}

// Automated Slack notification on release
async function postReleaseCheck(version: string) {
  const health = await fetchReleaseHealth(version);

  if (health.metrics.crashFreeRate < 99.0) {
    await alertOncall(`CRITICAL: ${version} crash-free rate at ${health.metrics.crashFreeRate}%`);
    await haltStagedRollout(version);
  }

  if (health.metrics.anrRate > 0.47) {
    await alertOncall(`WARNING: ${version} ANR rate at ${health.metrics.anrRate}% вҖ” exceeds Play threshold`);
  }
}
Google Play Bad Behavior Thresholds

Google Play enforces "bad behavior" thresholds: crash rate > 1.09% or ANR rate > 0.47% triggers reduced visibility in search results and a warning in Play Console. Exceeding these thresholds for an extended period can result in the app being flagged or removed. Monitor these metrics within the first 24 hours of any rollout increase.

7. Rollback Strategies

Mobile rollbacks are fundamentally different from web rollbacks вҖ” you cannot force users to downgrade. This requires a layered rollback strategy.

8. Hotfix Process

# Hotfix workflow for critical production bugs

# 1. Create hotfix branch from the release tag
git checkout -b hotfix/3.2.1 v3.2.0

# 2. Apply the minimal fix
# (cherry-pick from main if fix already exists, or commit directly)
git cherry-pick <fix-commit-sha>

# 3. Bump patch version
# package.json: 3.2.0 вҶ’ 3.2.1
# iOS: increment build number
# Android: increment versionCode

# 4. CI builds and signs automatically
# 5. QA validates the fix (minimal regression test)

# 6. Submit to stores
# iOS: Request expedited review if critical
# Android: Submit to production track

# 7. Merge hotfix back to main
git checkout main
git merge hotfix/3.2.1 --no-ff

# 8. Tag the hotfix release
git tag v3.2.1
git push origin v3.2.1
Apple Expedited Review

Apple offers expedited review for critical bug fixes. Submit through App Store Connect вҶ’ select the build вҶ’ request expedited review with a detailed explanation of the critical issue. Expedited reviews typically complete within 24 hours. Reserve this for genuine emergencies вҖ” overuse may result in Apple declining future requests.

9. Release Train Model

A release train is a fixed-cadence release schedule where a new version ships every N weeks regardless of which features are ready. Features that miss the train wait for the next one.

    tags: ['v*']

12. Architect Considerations

Release Engineering as a Platform

At Staff level, treat release management as an internal platform: (1) self-service release triggers (any engineer can initiate a release via CLI/Slack bot), (2) automated quality gates (crash rate, test coverage, bundle size checks), (3) release dashboards showing adoption curves, health metrics, and rollout status across both platforms, (4) runbooks for every failure mode (review rejection, signing failure, rollout halt), (5) post-mortem process for every release incident. The goal is to make releases boring and predictable.

OTA + Store Release Coordination

When using OTA updates alongside store releases, define clear policies: (1) OTA for JS-only bug fixes and copy changes, (2) store releases for native code changes, new permissions, and major features, (3) version compatibility matrix between JS bundles and native shells, (4) forced update mechanism for critical native security patches, (5) analytics to track which JS bundle version runs on which native shell version.
