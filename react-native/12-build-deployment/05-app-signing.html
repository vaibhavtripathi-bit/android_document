1. Why App Signing Exists

Code signing is a cryptographic mechanism that guarantees two things: identity (the app was produced by a known developer) and integrity (the app hasn't been tampered with since signing). Both iOS and Android enforce signing at the OS level — unsigned or incorrectly signed apps cannot be installed.

Code Signing Trust Chain ════════════════════════ Developer Platform User Device ───────── ──────── ─────────── Private Key ──sign──→ Signed Binary ──upload──→ Store ──install──→ OS Verifies │ │ Signature │ │ │ Certificate ◄──issued── Certificate Trust Anchor (public key) Authority (Root CA in OS) (Apple/Google) iOS: Apple is the sole CA — all certs chain to Apple Root CA Android: Developer self-signs, but Play App Signing adds Google as intermediary
Architect Insight

Signing is not just a release-time concern — it's a security architecture decision that affects your entire CI/CD pipeline, team onboarding, key rotation strategy, and disaster recovery plan. A compromised signing key means an attacker can push malicious updates to all your users. Treat signing keys with the same security posture as production database credentials.

2. iOS Code Signing Deep Dive

iOS code signing involves three interconnected components: signing certificates, provisioning profiles, and entitlements. Each serves a distinct purpose in Apple's security model.

2.1 Signing Certificates

A signing certificate binds a public/private key pair to a developer or team identity. The private key stays on the developer's machine (or CI keychain); the certificate (containing the public key) is issued by Apple.

# Generate a Certificate Signing Request (CSR)
# This creates a private key and a CSR file
openssl req -new -newkey rsa:2048 -nodes \
  -keyout ios_distribution.key \
  -out ios_distribution.csr \
  -subj "/CN=Your Name/O=Your Org/C=US"

# Upload CSR to Apple Developer Portal → Certificates
# Apple signs it with Apple Root CA → download .cer file

# Convert to .p12 for portability (includes private key)
openssl pkcs12 -export \
  -inkey ios_distribution.key \
  -in ios_distribution.cer \
  -out ios_distribution.p12 \
  -password pass:"$CERT_PASSWORD"

2.2 Certificate Types

Certificate Type Purpose Limit Validity
Apple Development Debug builds on physical devices Unlimited per team 1 year
Apple Distribution App Store + TestFlight + Ad Hoc 3 per team 1 year
Apple Push Services APNs push notifications 2 per app (dev + prod) 1 year
Pass Type ID Apple Wallet passes Per pass type 1 year
Critical: Distribution Certificate Limit

Apple allows only 3 distribution certificates per team. Revoking a certificate invalidates all provisioning profiles that reference it. In a large organization, never create distribution certificates tied to individual developer accounts — use a shared CI service account and manage the certificate centrally.

2.3 The Signing Process

iOS Code Signing Flow ═════════════════════ Xcode / xcodebuild ├── 1. Compile source → binary ├── 2. Locate signing identity (certificate + private key) │ └── Search order: Keychain → match/fastlane → Xcode managed ├── 3. Locate provisioning profile │ └── Must match: bundle ID + certificate + device UDIDs (if dev/adhoc) ├── 4. Embed provisioning profile → .app/embedded.mobileprovision ├── 5. Sign each framework/dylib individually ├── 6. Generate CodeResources (hash of every file in bundle) ├── 7. Sign main binary with CMS signature └── 8. Output: .app bundle (or .ipa archive) Verification (on device): ├── OS checks CMS signature against embedded certificate ├── Certificate chains to Apple Root CA ├── Provisioning profile matches bundle ID ├── Device UDID in profile (dev/adhoc) OR App Store receipt └── Entitlements match profile capabilities

3. Provisioning Profiles

A provisioning profile is a signed plist file from Apple that ties together: (1) an App ID (bundle identifier), (2) one or more signing certificates, (3) a list of device UDIDs (for development/ad hoc), and (4) entitlements. It's the authorization document that says "this app, signed by this certificate, can run on these devices with these capabilities."

3.1 Profile Types

Profile Type Device Limit App Store Use Case
Development 100 per device type No Local debugging on physical devices
Ad Hoc 100 per device type No QA testing, beta distribution outside TestFlight
App Store Unlimited (via Store) Yes TestFlight + App Store distribution
Enterprise (In-House) Unlimited (internal) No Internal apps, requires $299/yr Enterprise account

3.2 Profile Anatomy

# Inspect a provisioning profile
security cms -D -i embedded.mobileprovision

# Key fields in the decoded plist:
# - AppIDName: "My React Native App"
# - TeamIdentifier: ["ABCDE12345"]
# - ProvisionedDevices: ["udid1", "udid2", ...]  (dev/adhoc only)
# - DeveloperCertificates: [<cert data>]
# - Entitlements: { ... }
# - ExpirationDate: 2025-03-15T00:00:00Z
# - ProvisionsAllDevices: true  (enterprise only)
100-Device Limit Reset

The 100-device limit per device type (iPhone, iPad, Apple Watch, Apple TV, Mac) resets annually at the start of your membership year. Devices can be disabled but not removed until the annual reset. This limit applies to both Development and Ad Hoc profiles. Enterprise profiles have no device limit but require a separate $299/year Apple Developer Enterprise Program membership.

4. Entitlements

Entitlements are key-value pairs embedded in the app binary that declare which OS capabilities the app is allowed to use. They must match what's configured in the provisioning profile — any mismatch causes a signing failure.

<!-- ios/MyApp/MyApp.entitlements -->
<?xml version="1.0" encoding="UTF-8"?>
<!DOCTYPE plist PUBLIC "-//Apple//DTD PLIST 1.0//EN"
  "http://www.apple.com/DTDs/PropertyList-1.0.dtd">
<plist version="1.0">
<dict>
  <!-- Push Notifications -->
  <key>aps-environment</key>
  <string>production</string>

  <!-- Associated Domains (deep links) -->
  <key>com.apple.developer.associated-domains</key>
  <array>
    <string>applinks:example.com</string>
    <string>webcredentials:example.com</string>
  </array>

  <!-- App Groups (shared data between app + extensions) -->
  <key>com.apple.security.application-groups</key>
  <array>
    <string>group.com.example.myapp</string>
  </array>

  <!-- Keychain Sharing -->
  <key>keychain-access-groups</key>
  <array>
    <string>$(AppIdentifierPrefix)com.example.myapp</string>
  </array>
</dict>
</plist>

5. iOS Distribution Types

5.1 Distribution Comparison

Method Audience Review Required Device Registration Certificate Type
App Store Public Yes (Apple Review) No Distribution
TestFlight (Internal) Up to 100 team members No No Distribution
TestFlight (External) Up to 10,000 testers Yes (Beta Review) No Distribution
Ad Hoc Up to 100 devices/type No Yes (UDID in profile) Distribution
Enterprise (In-House) Internal employees No No Enterprise Distribution
Enterprise Distribution Risks

Apple actively monitors enterprise certificate usage. Distributing enterprise-signed apps to non-employees violates the Enterprise Developer Agreement and results in certificate revocation — instantly bricking all apps signed with that certificate across the entire organization. Never use enterprise distribution as a way to bypass App Store review.

6. Android App Signing

Android uses Java KeyStore (JKS) or PKCS#12 files to store signing keys. Unlike iOS, Android doesn't require a central authority — developers generate their own keys. However, the signing key is permanently bound to the app identity on Google Play.

6.1 Keystore Generation

# Generate a release keystore
keytool -genkeypair \
  -v \
  -storetype PKCS12 \
  -keystore my-release-key.keystore \
  -alias my-key-alias \
  -keyalg RSA \
  -keysize 2048 \
  -validity 10000 \
  -storepass "$KEYSTORE_PASSWORD" \
  -keypass "$KEY_PASSWORD" \
  -dname "CN=Your Name, OU=Mobile, O=Your Org, L=City, ST=State, C=US"

# Verify keystore contents
keytool -list -v -keystore my-release-key.keystore

# Extract SHA-256 fingerprint (needed for Google APIs, deep links)
keytool -list -v -keystore my-release-key.keystore \
  | grep "SHA256:"

6.2 Gradle Signing Configuration

// android/app/build.gradle
android {
    signingConfigs {
        debug {
            storeFile file('debug.keystore')
            storePassword 'android'
            keyAlias 'androiddebugkey'
            keyPassword 'android'
        }
        release {
            if (project.hasProperty('MYAPP_UPLOAD_STORE_FILE')) {
                storeFile file(MYAPP_UPLOAD_STORE_FILE)
                storePassword MYAPP_UPLOAD_STORE_PASSWORD
                keyAlias MYAPP_UPLOAD_KEY_ALIAS
                keyPassword MYAPP_UPLOAD_KEY_PASSWORD
            }
        }
    }
    buildTypes {
        release {
            signingConfig signingConfigs.release
            minifyEnabled true
            proguardFiles getDefaultProguardFile('proguard-android-optimize.txt'),
                          'proguard-rules.pro'
        }
    }
}

6.3 V1/V2/V3/V4 Signing Schemes

Scheme Android Version Mechanism Protection
V1 (JAR signing) All versions Signs individual ZIP entries Partial — unsigned ZIP metadata can be modified
V2 (APK Signature) Android 7.0+ (API 24) Signs entire APK as blob Full APK integrity
V3 (Key Rotation) Android 9.0+ (API 28) V2 + proof-of-rotation struct Enables signing key rotation
V4 (Incremental) Android 11+ (API 30) Merkle hash tree over APK Streaming install verification

7. Google Play App Signing

Google Play App Signing (GPAS) is a service where Google holds the actual app signing key and developers upload with an upload key. This adds a layer of security — if the upload key is compromised, it can be reset without affecting the app's identity.

Google Play App Signing Flow ════════════════════════════ Developer Google Play User Device ───────── ─────────── ─────────── ┌─────────────────────┐ Upload Key ──sign──→ AAB ──→ │ Verify upload sig │ (developer holds) │ Strip upload sig │ │ Generate APKs │ │ Sign with App Key │──→ Install APK │ (Google holds) │ signed by └─────────────────────┘ App Key Key Compromise Scenarios: ───────────────────────── Upload key lost/compromised → Contact Google → Reset upload key → No user impact App signing key compromised → Cannot be reset → App must be republished as new listing 
# Export upload key for Play App Signing enrollment
# Option 1: Let Google generate the app signing key
# (recommended for new apps)

# Option 2: Export existing key using PEPK tool
java -jar pepk.jar \
  --keystore=my-release-key.keystore \
  --alias=my-key-alias \
  --output=encrypted-key.zip \
  --include-cert \
  --encryptionkey="eb10fe8f7c7c9df715022017b00c6471f8ba8170b13049a11e766aa0..."
AAB vs APK Signing

When using Android App Bundles (AAB), Google Play generates optimized APKs for each device configuration and signs them with the app signing key. The developer never produces the final APKs. This means the SHA-256 fingerprint used for Google APIs (Maps, OAuth) must be the app signing key's fingerprint (from Play Console), not the upload key's fingerprint.

8. Signing for Multiple Environments

Production React Native apps typically maintain 3+ environments (development, staging, production), each requiring distinct signing configurations, bundle identifiers, and API endpoints.

8.1 iOS Multi-Environment Setup

// Using Xcode build configurations + schemes
// Each scheme maps to a configuration with distinct:
//   - Bundle ID (com.example.app.dev / .staging / .app)
//   - Provisioning profile
//   - Entitlements file
//   - App icon set
//   - Display name

// Fastfile example for multi-env iOS builds
lane :build_ios do |options|
  env = options[:environment] || "production"

  match(
    type: env == "production" ? "appstore" : "adhoc",
    app_identifier: "com.example.app.#{env}",
    readonly: true
  )

  gym(
    scheme: "MyApp-#{env.capitalize}",
    export_method: env == "production" ? "app-store" : "ad-hoc",
    output_directory: "./build"
  )
end

8.2 Android Multi-Environment with Flavors

// android/app/build.gradle
android {
    flavorDimensions "environment"
    productFlavors {
        dev {
            dimension "environment"
            applicationIdSuffix ".dev"
            versionNameSuffix "-dev"
            resValue "string", "app_name", "MyApp DEV"
            signingConfig signingConfigs.debug
        }
        staging {
            dimension "environment"
            applicationIdSuffix ".staging"
            versionNameSuffix "-staging"
            resValue "string", "app_name", "MyApp STG"
            signingConfig signingConfigs.release
        }
        production {
            dimension "environment"
            resValue "string", "app_name", "MyApp"
            signingConfig signingConfigs.release
        }
    }
}

9. Automated Signing in CI/CD

CI/CD signing is the most error-prone part of mobile DevOps. The challenge: signing credentials must be available to the build machine without being committed to source control.

9.1 Fastlane Match (iOS)

# Matchfile — stores certs/profiles in encrypted Git repo or cloud storage
git_url("https://github.com/org/certificates.git")
storage_mode("git")  # or "google_cloud", "s3"
type("appstore")
app_identifier(["com.example.app", "com.example.app.dev"])
username("ci@example.com")

# CI pipeline usage
# 1. Create temporary keychain
create_keychain(
  name: "ci_keychain",
  password: ENV["KEYCHAIN_PASSWORD"],
  default_keychain: true,
  unlock: true,
  timeout: 3600
)

# 2. Fetch and install certs/profiles
match(
  type: "appstore",
  keychain_name: "ci_keychain",
  keychain_password: ENV["KEYCHAIN_PASSWORD"],
  readonly: true
)

# 3. Build and sign
gym(scheme: "MyApp", export_method: "app-store")

# 4. Clean up keychain
delete_keychain(name: "ci_keychain")

9.2 Android CI Signing

# GitHub Actions example for Android signing
# Store keystore as base64-encoded secret

# Encode: base64 -i my-release-key.keystore | pbcopy
# Store as ANDROID_KEYSTORE_BASE64 secret

# In workflow:
- name: Decode keystore
  run: |
    echo "$ANDROID_KEYSTORE_BASE64" | base64 --decode > android/app/release.keystore
  env:
    ANDROID_KEYSTORE_BASE64: ${{ secrets.ANDROID_KEYSTORE_BASE64 }}

- name: Build release AAB
  run: |
    cd android && ./gradlew bundleProductionRelease
  env:
    MYAPP_UPLOAD_STORE_FILE: release.keystore
    MYAPP_UPLOAD_STORE_PASSWORD: ${{ secrets.KEYSTORE_PASSWORD }}
    MYAPP_UPLOAD_KEY_ALIAS: ${{ secrets.KEY_ALIAS }}
    MYAPP_UPLOAD_KEY_PASSWORD: ${{ secrets.KEY_PASSWORD }}

9.3 EAS Build Signing

// eas.json — Expo Application Services handles signing automatically
{
  "build": {
    "production": {
      "ios": {
        "credentialsSource": "remote",
        // EAS manages certs/profiles in Expo's secure storage
        // Alternatively: "credentialsSource": "local" for self-managed
      },
      "android": {
        "credentialsSource": "remote",
        // EAS generates and stores keystore
        "buildType": "app-bundle"
      }
    }
  }
}

# EAS credential management commands
eas credentials                     # Interactive credential manager
eas credentials --platform ios      # Manage iOS certs/profiles
eas credentials --platform android  # Manage Android keystores

10. Certificate Lifecycle Management

Certificate Lifecycle ═════════════════════ Generate Register Distribute Monitor Rotate Revoke ──────── ──────── ────────── ─────── ────── ────── CSR/Key → Apple/ → CI/CD → Expiry → New cert → Old cert creation Google secrets alerts + profiles invalidated portal vault (90 days + CI update before) Automation Checklist: ☐ Expiry monitoring (PagerDuty/Slack alerts at 90, 60, 30 days) ☐ Automated rotation via fastlane match / EAS credentials ☐ CI secret rotation runbook ☐ Provisioning profile regeneration after cert rotation ☐ Key escrow / backup in secure vault (1Password, HashiCorp Vault) ☐ Access audit log (who accessed signing keys and when)
Certificate Expiry Disaster

When a distribution certificate expires, you cannot submit new builds until a new certificate is created and all provisioning profiles are regenerated. If you discover this during a critical hotfix, it can add hours to your release timeline. Automate expiry monitoring and rotate certificates proactively — never wait for expiry.

11. Signing Security Threats

11.1 Threat Model

Threat Impact Mitigation
Keystore committed to Git Anyone with repo access can sign apps .gitignore, pre-commit hooks, git-secrets
CI secret exfiltration Malicious PR extracts signing keys Restrict secrets to protected branches only
Developer laptop theft Private key exposure FileVault/BitLocker, keychain password, remote wipe
Shared credentials in Slack Uncontrolled key distribution Secrets manager (Vault, 1Password), audit logs
Supply chain attack on CI Compromised build signs malicious code Pinned CI images, reproducible builds, SLSA compliance

12. Architect Considerations

Signing Architecture for Scale

At L6/Staff level, design signing infrastructure as a platform service: (1) centralized credential management with role-based access, (2) automated certificate rotation with zero-downtime transitions, (3) signing-as-a-service where CI pipelines request signatures from a hardened signing server rather than holding keys directly, (4) audit trails for every signing operation, (5) disaster recovery plan for key compromise scenarios. Consider hardware security modules (HSMs) for the highest-value signing keys.

Multi-App Organization Strategy

For organizations with multiple React Native apps: (1) use a single Apple Developer team with app-specific provisioning profiles, (2) standardize on Play App Signing for all Android apps, (3) create a shared fastlane match repository for iOS credentials, (4) implement a signing key hierarchy — organization root key signs per-app upload keys, (5) document the complete key inventory with ownership, expiry dates, and rotation schedule.
