Animated API

Q1: How does Animated.Value work internally?

A: Animated.Value is a mutable wrapper around a number that bypasses React's state system. It maintains an internal _value, a set of listeners, and references to dependent animated nodes (interpolations, additions). When the value changes, it directly updates native view properties without triggering React re-renders or reconciliation. This is achieved through the Animated node graph — a directed acyclic graph where value changes propagate through transform nodes to view property updates.

Q2: What's the difference between Animated.timing, spring, and decay?

A: timing: Animates from current value to target over a fixed duration with an easing curve (linear, ease-in-out, cubic-bezier). Predictable duration. spring: Physics-based animation using spring dynamics (stiffness, damping, mass). Duration is not fixed — it depends on physics parameters. Feels natural for UI interactions. decay: Starts with an initial velocity and decelerates to rest. No target value — the final position depends on velocity and deceleration rate. Used for momentum/fling gestures.

Q3: How does useNativeDriver work under the hood?

A: When useNativeDriver: true, the entire animation description (frames, easing, target value) is serialized to JSON and sent to the native side in a single message at animation start. The native animation driver (CADisplayLink on iOS, Choreographer on Android) then runs the animation independently, updating view properties directly on the UI thread each frame (~16ms). The JS thread is completely free during the animation. This is why native-driven animations maintain 60fps even when JS is busy. The trade-off: only non-layout properties (transform, opacity) are supported because layout changes require Yoga recalculation on the shadow thread.

Q4: What are the limitations of useNativeDriver?

A: (1) Only transform and opacity properties — no width, height, margin, padding, backgroundColor (layout properties require Yoga); (2) No conditional logic during animation — the animation is "fire and forget"; (3) No per-frame JS callbacks — can't read the value synchronously each frame; (4) Limited string interpolation (only deg, rad, colors); (5) Can't dynamically change animation parameters mid-flight. For any of these needs, use Reanimated instead.

Q5: How does Animated.event work and when would you use it?

A: Animated.event maps native event properties directly to Animated.Values without JS thread involvement (when useNativeDriver: true). Primary use case: scroll-driven animations — mapping nativeEvent.contentOffset.y to an Animated.Value that drives header collapse, parallax, or opacity changes. The event mapping is declared once and runs on the native thread for each scroll frame. Requires scrollEventThrottle={16} on ScrollView for 60fps updates.

Q6: How does interpolation work in the Animated API?

A: Interpolation creates a new Animated node that maps an input range to an output range. It uses linear interpolation between segments, with configurable extrapolation (extend, clamp, identity). A single Animated.Value can drive multiple interpolations — e.g., scroll position driving opacity (0→1), scale (1→0.8), and rotation (0deg→15deg) simultaneously. Interpolation nodes are part of the animated node graph and execute on the native thread with useNativeDriver. Supports numeric, degree, radian, and color string outputs.

Q7: How do composition operators (sequence, parallel, stagger) work?

A: sequence: Runs animations one after another — each starts when the previous completes. parallel: Runs all animations simultaneously — completes when the longest finishes (or first, with stopTogether: false). stagger: Like parallel but with incremental delay between each animation. loop: Repeats an animation N times or infinitely. delay: Waits before starting. These are composable — you can nest sequence inside parallel, etc. All composition happens on the JS side; the individual animations still use native driver.

Reanimated Internals

Q8: How does Reanimated v3 differ architecturally from the Animated API?

A: The Animated API sends a static animation description to native — no JS runs during the animation. Reanimated runs actual JavaScript code (worklets) on the UI thread via a second JS VM. This means worklets can have conditionals, loops, math, and any JS logic — executing at 60fps on the UI thread. The Babel plugin serializes worklet functions at build time, and the Reanimated runtime instantiates them in a lightweight Hermes/JSC VM on the UI thread. Shared values (C++ objects via JSI) enable zero-copy communication between threads.

Q9: What is a worklet and how does the Babel plugin process it?

A: A worklet is a JS function marked with the 'worklet' directive (or used in a worklet context like useAnimatedStyle). The Reanimated Babel plugin: (1) extracts the function body as a string; (2) captures closure variables; (3) generates a __workletHash for caching; (4) creates a serializable worklet object with __initData containing the code string. At runtime, this serialized function is sent to the UI thread VM via JSI and instantiated there. Worklets can only access shared values and other worklets — not React state, hooks, or most JS APIs.

Q10: How do shared values work across threads?

A: Shared values are C++ objects exposed to both the JS thread and UI thread via JSI. There's no serialization — both threads access the same memory through JSI host object proxies. On the JS thread, setting .value with withSpring/withTiming schedules an animation on the UI thread. On the UI thread (in worklets), reading .value returns the current value directly. Writing .value in a worklet updates immediately and triggers dependent useAnimatedStyle recalculations. This zero-copy sharing is what makes Reanimated performant.

Q11: What are the limitations of worklets?

A: Worklets run in a restricted environment: (1) No React hooks or state access; (2) No DOM/native module APIs; (3) No network requests or async operations; (4) Limited standard library (Math, basic string ops); (5) Closure variables must be serializable; (6) Debugging is harder (separate VM, limited source maps); (7) Errors in worklets can crash the app (UI thread crash = app crash). Use runOnJS to bounce back to the JS thread for anything worklets can't do.

Q12: How does useAnimatedStyle differ from regular style objects?

A: useAnimatedStyle returns a worklet-backed style object that's evaluated on the UI thread every frame when its dependent shared values change. Regular styles are evaluated once on the JS thread during render. useAnimatedStyle can contain conditionals, interpolation, and any JS logic — all running at 60fps on the UI thread. It only triggers native view updates when the computed style actually changes (shallow comparison). Must be used with Animated.View (from Reanimated), not regular View.

Q13: Explain useDerivedValue and useAnimatedReaction.

A: useDerivedValue: Creates a read-only shared value computed from other shared values. The computation worklet runs on the UI thread whenever dependencies change. Like useMemo but for shared values. useAnimatedReaction: Runs a side-effect worklet when a tracked value changes. Takes two worklets: a "prepare" function that returns the tracked value, and a "react" function that receives current and previous values. Used for cross-value synchronization, threshold detection, and triggering JS-side effects via runOnJS.

Gesture Handler

Q14: Why does React Native need Gesture Handler instead of PanResponder?

A: PanResponder processes touch events on the JS thread — if JS is busy, gestures are delayed or dropped. Gesture Handler uses platform-native gesture recognizers (UIGestureRecognizer on iOS, GestureDetector on Android) that process touches on the native thread. Benefits: (1) Reliable gesture detection regardless of JS thread load; (2) Native gesture composition (simultaneous, exclusive); (3) Platform-consistent behavior; (4) Direct integration with Reanimated (gesture callbacks as worklets on UI thread). The entire gesture → animation pipeline never touches the JS thread.

Q15: Explain the three gesture composition modes.

A: Simultaneous: Both gestures can be active at the same time — e.g., pinch + rotate on a photo. Exclusive: Only one gesture activates; the first in the list has priority — e.g., double-tap takes priority over single-tap. Race: The first gesture to meet its activation criteria wins; the other fails — e.g., horizontal swipe vs vertical scroll (determined by initial movement direction). These map to native gesture recognizer delegate methods on iOS and are emulated on Android by RNGH.

Q16: How do you handle the iOS back swipe gesture conflicting with a horizontal pan?

A: iOS has a system edge-swipe gesture for back navigation. To prevent conflicts: (1) Use activeOffsetX to require a minimum horizontal movement before the pan activates; (2) Use failOffsetX(0) to fail the gesture if the user swipes from the left edge; (3) Alternatively, use simultaneousHandlers to allow both gestures. For React Navigation, the native stack navigator handles this automatically. For custom gestures, test on device — the iOS back gesture has a ~20px activation zone from the left edge.

Q17: How does Gesture Handler integrate with Reanimated?

A: When Reanimated is installed, Gesture Handler's callbacks (onStart, onUpdate, onEnd) automatically run as worklets on the UI thread — no 'worklet' directive needed. This means gesture events directly update shared values without any thread crossing. The entire pipeline: native gesture recognizer → worklet callback → shared value update → useAnimatedStyle recalculation → native view update — all happens on the UI thread in a single frame.

Q18: What platform differences exist in gesture handling?

A: (1) iOS uses UIGestureRecognizer with native delegate methods for composition; Android emulates this. (2) iOS has system edge-swipe for back navigation; Android uses a back button. (3) Simultaneous gesture handling is native on iOS but emulated on Android. (4) Long-press-to-pan transitions are smoother on iOS. (5) Scroll gesture interaction differs — iOS uses UIScrollView's gesture delegate, Android uses NestedScrollingChild. (6) Haptic feedback APIs differ between platforms.

Layout Animations

Q19: How do Reanimated entering/exiting animations work?

A: Entering: When a component with an entering prop mounts, Reanimated sets initial style values (e.g., opacity: 0, translateY: 50) and animates to the target values (opacity: 1, translateY: 0). The animation runs on the UI thread via worklets. Exiting: When React unmounts a component with an exiting prop, Reanimated intercepts the unmount, keeps the native view alive, runs the exit animation, and only removes the native view after completion. React has already forgotten about the component — Reanimated manages the native lifecycle independently.

Q20: What is a layout transition and when would you use it?

A: A layout transition animates a component's position/size change when the surrounding layout changes (e.g., a sibling is added/removed). Without it, components jump instantly to new positions. With layout={LinearTransition}, Reanimated captures the before and after positions, then animates the transition on the UI thread. Use for: reorderable lists, expandable sections, dynamic grids, and any UI where items shift position. Available transitions: LinearTransition, SequencedTransition, FadingTransition, JumpingTransition.

Q21: How do Keyframe animations work in Reanimated?

A: Keyframes define animation states at specific percentage points (0-100), similar to CSS @keyframes. You specify style properties at each keypoint, and Reanimated interpolates between them. Keyframes can be used as entering/exiting animations. They're defined with new Keyframe({ 0: {...}, 50: {...}, 100: {...} }) and support duration, delay, and easing per segment. Useful for complex multi-step animations like bounce, shake, or attention-seeking effects that can't be expressed as simple A→B transitions.

Q22: Compare Reanimated layout animations with the legacy LayoutAnimation API.

A: Legacy LayoutAnimation: Global scope (affects all layout changes), limited configuration (presets + basic spring/easing), requires experimental flag on Android, no per-component control, no composition. Reanimated: Per-component scope (entering + exiting + layout on each element), full customization (worklets, keyframes), works on both platforms, composable, runs on UI thread. Use legacy LayoutAnimation only for simple cases where you don't want the Reanimated dependency. For anything non-trivial, use Reanimated.

Q23: How do you animate FlatList items with Reanimated?

A: Two approaches: (1) Wrap renderItem content in Animated.View with entering/exiting/layout props — gives per-item control but can be expensive for large lists with staggered delays. (2) Use itemLayoutAnimation prop on FlatList itself — coordinates layout transitions for all items automatically. For staggered entrance, calculate delay based on visible index (not data index) to avoid large delays for off-screen items. Always use keyExtractor to ensure Reanimated can track items across renders.

Shared Element Transitions

Q24: How do shared element transitions work in React Native?

A: The process: (1) Measure the source element's position/size on Screen A; (2) Measure the target element's position/size on Screen B; (3) Create a clone on an overlay layer; (4) Animate the clone from source to target position/size; (5) Hide source and target during animation; (6) Remove clone and show target on completion. Reanimated's sharedTransitionTag automates this by matching tags across screens. The animation runs on the UI thread. Works best with native stack navigator and simple geometries (images, shapes).

Q25: What are the challenges of shared element transitions with text?

A: Text is problematic because: (1) Font rendering differs between iOS and Android — same font/size produces different pixel dimensions; (2) Animating font size causes visible reflow and re-rendering; (3) Text wrapping changes as width animates; (4) Line height and letter spacing don't interpolate smoothly. Solutions: use fade transitions for text shared elements instead of move/resize; use fixed-size text containers; or animate a snapshot image of the text rather than live text.

Q26: How do you optimize shared element transition performance?

A: (1) Use native stack navigator (not JS-based stack); (2) Prefetch target images before navigation; (3) Use placeholder dimensions to avoid layout shift on the target screen; (4) Defer heavy content rendering on the target screen until the transition completes; (5) Keep shared elements simple (images, basic shapes); (6) Avoid shared transitions on elements with complex children; (7) Test on low-end devices — shared transitions are CPU-intensive during the measurement phase.

Advanced Patterns

Q27: How do you implement a bottom sheet with snap points?

A: Use Gesture Handler Pan + Reanimated: (1) Track translateY with a shared value; (2) On pan update, clamp translateY between min and max snap points; (3) On pan end, project the final position using velocity, find the nearest snap point, and animate to it with withSpring (passing the gesture velocity for natural feel); (4) Add a backdrop with opacity interpolated from the sheet position; (5) Handle scroll interaction inside the sheet (simultaneous gesture with internal ScrollView). Libraries like @gorhom/bottom-sheet implement this pattern with additional features.

Q28: How do you implement Tinder-style swipeable cards?

A: Card stack with Pan gesture: (1) Render top 2-3 cards with decreasing scale/offset for depth effect; (2) Only the top card has an active pan gesture; (3) On pan update, translate the card and rotate proportionally to horizontal displacement; (4) On pan end, if displacement exceeds threshold, animate off-screen with withTiming and callback to remove the card; otherwise, spring back to center; (5) Behind cards scale up as the top card is dragged, creating a "next card rising" effect. Use interpolate to map drag distance to rotation and background card scale.

Q29: How do you create a custom pull-to-refresh animation?

A: Pan gesture on the scroll container: (1) On downward pan, translate content with rubber-band effect (diminishing returns past threshold); (2) Show a custom spinner/animation above the content, with opacity/rotation driven by pull distance; (3) On release, if past trigger threshold, snap to the refresh position and call the refresh callback; otherwise, spring back to 0; (4) After refresh completes, animate back to 0. The rubber-band effect uses translationY * 0.5 (or a logarithmic function) to resist over-pulling.

Q30: How do you implement skeleton loading animations?

A: Two approaches: (1) Opacity pulse: Use withRepeat(withTiming(...)) to oscillate opacity between 0.3 and 0.7 on placeholder views shaped like the content. Simple and lightweight. (2) Shimmer effect: Use Skia with a LinearGradient that translates horizontally across the placeholder, creating a shine effect. More visually appealing but requires Skia dependency. Match placeholder shapes to actual content layout for a seamless transition when data loads.

Q31: When would you use Lottie vs Reanimated for animations?

A: Lottie: Complex illustrative animations designed in After Effects — loading spinners, success checkmarks, onboarding illustrations, micro-interactions. Fire-and-forget, non-interactive. Reanimated: Interactive, gesture-driven animations — bottom sheets, swipeable cards, collapsing headers, drag-and-drop. Responds to user input in real-time. Use Lottie when a motion designer creates the animation; use Reanimated when the animation is driven by user interaction or app state. Lottie can be controlled by Reanimated for hybrid cases (gesture-driven Lottie progress).

Skia Graphics

Q32: What is React Native Skia and when would you use it?

A: @shopify/react-native-skia brings the Skia 2D graphics engine (used by Chrome, Android, Flutter) to React Native. It renders to its own GPU surface using a declarative JSX API. Use for: custom charts with smooth curves, blur/frosted glass effects, gradient borders, image filters (color matrix, saturation), drawing apps, particle effects, and any visual that exceeds what the view system + transforms can achieve. Don't use for standard UI (buttons, text, lists) — those should use native views for accessibility and platform consistency.

Q33: How does Skia's rendering differ from React Native's view system?

A: RN's view system renders through the platform's native view hierarchy (UIView/Android View) with Yoga for layout. Skia renders directly to a GPU surface (Metal on iOS, OpenGL/Vulkan on Android) — it's a canvas, not views. Key differences: (1) Skia has no layout system — you position everything manually with x/y coordinates; (2) No accessibility on Skia elements; (3) No native touch handling (use Gesture Handler); (4) Can't mix native views inside a Skia canvas; (5) GPU-accelerated — shaders and filters run on the GPU; (6) Supports custom GLSL shaders for effects impossible with views.

Q34: How does Skia integrate with Reanimated?

A: Skia properties can be driven by Reanimated shared values via useDerivedValue. Shared values update on the UI thread, and Skia re-renders its canvas when dependent values change. This enables 60fps animations of Skia drawings — moving shapes, animating paths, transitioning colors, and driving shader uniforms. The Skia canvas re-renders only the changed portions. For gesture-driven Skia (drawing apps), combine Gesture Handler events with Skia path construction.

Q35: What are Skia shaders and how do they work?

A: Skia shaders are GPU programs written in SkSL (Skia Shading Language, similar to GLSL) that compute the color of each pixel. They run massively parallel on the GPU — each pixel is computed independently. Shaders can create effects impossible with the view system: animated gradients, wave distortions, noise patterns, blur transitions. Uniforms (parameters) can be driven by Reanimated shared values for animation. Shaders are compiled at runtime and cached. Performance is excellent for visual effects but the programming model (GPU, per-pixel computation) is fundamentally different from view-based UI.

Performance & Architecture

Q36: How do you achieve 60fps animations in React Native?

A: The key principle: keep the JS thread out of the animation loop. (1) Use useNativeDriver: true with the Animated API for simple transitions; (2) Use Reanimated worklets for complex animations — they run on the UI thread; (3) Use Gesture Handler for touch processing — native gesture recognizers, not JS PanResponder; (4) Avoid setState during animations — it triggers reconciliation on the JS thread; (5) Use interpolate to derive multiple properties from a single animated value; (6) Profile with Flipper/Xcode Instruments to identify frame drops.

Q37: What causes animation jank in React Native and how do you diagnose it?

A: Common causes: (1) Animations running on JS thread (no native driver, no Reanimated); (2) Heavy re-renders during animation (setState in scroll handler); (3) Complex view hierarchies causing slow layout; (4) Image decoding on the main thread; (5) Garbage collection pauses in Hermes/JSC. Diagnosis: Enable the RN Performance Monitor (JS/UI frame rates), use Flipper's performance plugin, use Xcode Instruments (Time Profiler, Core Animation), use Android Profiler (CPU, GPU rendering). Look for JS frame time > 16ms or UI frame time > 16ms.

Q38: How do you profile animation performance?

A: (1) RN Perf Monitor: Shows JS and UI frame rates in real-time; (2) Flipper: Performance plugin with flame charts; (3) React DevTools Profiler: Identifies unnecessary re-renders during animations; (4) Xcode Instruments: Core Animation instrument shows GPU rendering, Time Profiler shows CPU hotspots; (5) Android Profiler: CPU, memory, and GPU rendering; (6) Reanimated useFrameCallback: Measure per-frame timing in worklets; (7) systrace: Low-level Android tracing for bridge/JSI overhead.

Q39: How do you decide between Animated API, Reanimated, and Skia?

A: Decision framework: Animated API — simple transitions (fade, slide, scale) where useNativeDriver suffices and you want zero extra dependencies. Reanimated — gesture-driven animations, conditional logic during animation, layout animations, anything requiring JS on the UI thread. Skia — custom drawing, GPU shaders, blur effects, charts, image filters — anything beyond what the view system can render. Most apps use Reanimated + Gesture Handler as the primary animation stack, with Skia for specific visual effects. The Animated API is a fallback for simple cases in libraries that want minimal dependencies.

Q40: How do you handle animations during navigation transitions?

A: (1) Use native stack navigator for hardware-accelerated transitions; (2) Defer heavy component mounting until transition completes (InteractionManager or setTimeout); (3) Use react-native-screens for native screen management; (4) For shared elements, match sharedTransitionTag across screens; (5) Custom transitions via cardStyleInterpolator (JS stack) or animation option (native stack); (6) Avoid starting component-level animations during screen transitions — they compete for UI thread time.

Q41: How do you implement spring physics correctly for natural-feeling animations?

A: Use the stiffness/damping/mass model (not friction/tension which is less precise). Guidelines: (1) Stiffness (100-300): Higher = faster response; (2) Damping (10-30): Higher = less oscillation; (3) Mass (0.5-2): Higher = more inertia. For UI interactions: damping ~15, stiffness ~150 (snappy with slight bounce). For decorative animations: damping ~8, stiffness ~100 (more playful). Always pass the gesture's release velocity to the spring for natural momentum continuation. Test on device — spring feel is subjective and hard to judge in simulators.

Q42: How do you handle animation cancellation and interruption?

A: Animated API: animation.stop() or starting a new animation on the same value (implicitly stops the previous). The completion callback receives { finished: false }. Reanimated: Assigning a new value to a shared value cancels any running animation on it. Use cancelAnimation(sharedValue) for explicit cancellation. For gestures: the gesture's onStart should save the current animated value (not the target) so interrupting a running animation starts from the current position, not the start or end.

Q43: How do you animate color transitions in React Native?

A: Animated API: Use interpolate with color string outputs — outputRange: ['rgb(255,0,0)', 'rgb(0,0,255)']. Limited to rgb/rgba strings, not supported with useNativeDriver. Reanimated: Use interpolateColor which supports hex, rgb, rgba, and hsl — runs on UI thread. Skia: Use shader-based color transitions for complex gradients. For theme transitions (light/dark mode), animate a progress shared value 0→1 and interpolateColor all affected properties.

Q44: How do you implement parallax scrolling in React Native?

A: Map scroll position to different translation speeds for foreground/background layers. With Animated API: use Animated.event to track scroll position, then interpolate with different output ranges for each layer (background moves at 0.5x scroll speed, foreground at 1x). With Reanimated: use useAnimatedScrollHandler to track scroll in a shared value, then drive layer translations in useAnimatedStyle. The key is that all layers are driven by a single scroll value with different multipliers.

Q45: How do you implement a gesture-driven image viewer (pinch-to-zoom, pan, double-tap)?

A: Compose three gestures with Gesture.Simultaneous (pinch + pan) and Gesture.Exclusive (double-tap vs single-tap): (1) Pinch: Scale from saved scale * event.scale, clamp between 1x and 5x; (2) Pan: Translate within bounds (bounds depend on current scale); (3) Double-tap: Toggle between 1x and 2x with spring animation, centering on tap point; (4) On gesture end, if scale < 1, spring back to 1. Use Gesture.Simultaneous(pan, pinch) for multi-touch, and Gesture.Exclusive(doubleTap, singleTap) for tap handling.

Design & Architecture

Q46: How do you architect an animation system for a large React Native app?

A: (1) Define a motion design system: standard spring configs (gentle, bouncy, snappy), standard durations (fast: 200ms, normal: 300ms, slow: 500ms), standard easing curves; (2) Create reusable animated components (AnimatedPressable, AnimatedCard, AnimatedList); (3) Use a shared animation constants file; (4) Establish patterns: entering animations for screen content, spring for interactive elements, timing for transitions; (5) Profile regularly — add animation performance budgets to CI; (6) Document which animation layer to use for which pattern.

Q47: How do you handle animations for accessibility?

A: (1) Respect AccessibilityInfo.isReduceMotionEnabled() — disable or simplify animations for users who prefer reduced motion; (2) Replace spring/bounce animations with simple fades for reduced motion; (3) Ensure animated elements have proper accessibility labels; (4) Don't rely on animation to convey information — always have a non-animated fallback; (5) Skia-rendered elements have no accessibility — provide accessible alternatives; (6) Test with VoiceOver/TalkBack to ensure animations don't interfere with screen reader navigation.

Q48: How do you test animations in React Native?

A: (1) Unit tests: Mock Reanimated's withTiming/withSpring to return final values immediately; test that shared values reach expected states; (2) Snapshot tests: Verify animated component renders correctly in initial and final states; (3) Integration tests: Use jest.advanceTimersByTime with mocked animation drivers; (4) Visual regression: Screenshot comparison tools (Detox screenshots, Appium); (5) Manual testing: Animations are inherently visual — always verify on real devices, especially for spring physics and gesture interactions.

Q49: How do you handle animation state persistence across screen transitions?

A: Challenge: React Navigation unmounts screens (with detachInactiveScreens) or keeps them mounted but off-screen. Options: (1) Store animation progress in shared values (survive re-renders but not unmounts); (2) Persist animation state to MMKV for critical animations; (3) Use navigation state params to pass animation context between screens; (4) For shared element transitions, Reanimated handles this automatically via sharedTransitionTag; (5) For tab navigators (screens stay mounted), animation state naturally persists.

Q50: How do you implement micro-interactions that feel native?

A: (1) Press feedback: Scale down to 0.95 on press begin, spring back on release (use Gesture.Tap with onBegin/onFinalize); (2) Haptic feedback: Trigger haptics at key moments (selection, success, error); (3) Spring physics: Use springs instead of timing for interactive elements — they respond naturally to interruption; (4) Overshoot: Allow slight overshoot on springs (damping ~12-15) for playful feel; (5) Stagger: Delay list item animations by 50-100ms for cascading entrance; (6) Follow-through: Elements should slightly overshoot their target and settle back.

Q51: How would you implement a drag-to-reorder list?

A: (1) Long-press gesture to activate drag mode (scale up the item, add shadow); (2) Pan gesture (simultaneous with long-press) to track position; (3) As the dragged item moves, calculate which position it's hovering over; (4) Animate other items to make room using layout transitions; (5) On release, animate the item to its new position with spring; (6) Update the data array and notify parent. Use Reanimated shared values for all position tracking. Libraries like react-native-draggable-flatlist implement this with virtualization support.

Q52: How do you handle animation performance on low-end devices?

A: (1) Detect device tier (RAM, CPU cores) and adjust animation complexity; (2) Reduce spring iterations (higher damping) on low-end devices; (3) Disable layout animations and use simple fades; (4) Reduce Skia canvas resolution; (5) Avoid simultaneous animations — sequence them instead; (6) Use shouldRasterizeIOS and renderToHardwareTextureAndroid for complex static views that animate; (7) Profile on the lowest-tier device you support — if it can't maintain 60fps, simplify.

Q53: How do you implement a collapsing header with scroll-driven animations?

A: With Reanimated: (1) Use useAnimatedScrollHandler to track scroll offset in a shared value; (2) In useAnimatedStyle, interpolate header height, opacity, and title scale from the scroll value; (3) Clamp the header between expanded and collapsed sizes; (4) For sticky behavior, use transform: translateY to keep the collapsed header visible; (5) Animate the title from large (expanded) to small (collapsed) using scale transform. The entire animation runs on the UI thread — no JS involvement per scroll frame.

Q54: What is the role of useFrameCallback in Reanimated?

A: useFrameCallback runs a worklet on every frame (via requestAnimationFrame on the UI thread). It provides frameInfo with timestamp and time since previous frame. Use for: (1) Custom physics simulations (gravity, collision); (2) Frame-rate-independent animations; (3) Particle systems; (4) Performance monitoring (detecting frame drops); (5) Continuous animations that don't fit the withTiming/withSpring model. Be cautious — running expensive logic every frame can cause jank even on the UI thread.

Q55: How do you create a morphing shape animation?

A: Use Skia paths with Reanimated: (1) Define two SVG paths (source and target) with the same number of control points; (2) Use a shared value (0→1) to interpolate between the paths using Skia.Path.MakeFromSVGString and path interpolation; (3) Animate the progress value with withTiming or gesture-driven. Alternatively, use Skia's PathInterpolation utility. For paths with different point counts, use a library that can normalize paths to the same number of segments before interpolation.
