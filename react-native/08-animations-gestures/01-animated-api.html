1. Animated.Value & Animated.ValueXY

An Animated.Value is a mutable wrapper around a number that can drive style properties without triggering React re-renders. It's the fundamental building block of the Animated API.

Animated.Value Architecture ════════════════════════════════════════════════════ Animated.Value(0) │ ├── Internal: _value = 0 (mutable number) │ ├── Listeners: Set of callbacks notified on change │ ├── Attached nodes: Animated nodes that depend on this value │ (interpolations, additions, multiplications) │ └── Native node ID: Reference to native-side animated node (when useNativeDriver: true) Key insight: Animated.Value does NOT use setState. It mutates an internal number and directly updates native view properties, bypassing React reconciliation. ┌──────────────┐ setValue() ┌──────────────┐ │ Animated.Value│ ──────────────── → │ Native View │ │ _value: 0.5 │ (direct prop │ opacity: 0.5│ └──────────────┘ update, no └──────────────┘ re-render) 
import { Animated, useRef } from 'react-native';

function FadeInView({ children }: { children: React.ReactNode }) {
  const opacity = useRef(new Animated.Value(0)).current;

  useEffect(() => {
    Animated.timing(opacity, {
      toValue: 1,
      duration: 300,
      useNativeDriver: true,
    }).start();
  }, []);

  return (
    <Animated.View style={{ opacity }}>
      {children}
    </Animated.View>
  );
}

// Animated.ValueXY for 2D animations
const position = useRef(new Animated.ValueXY({ x: 0, y: 0 })).current;

// ValueXY provides convenience methods
position.getLayout();        // { left: x, top: y }
position.getTranslateTransform(); // [{ translateX: x }, { translateY: y }]

// Reading current value (synchronous)
const currentOpacity = opacity.__getValue(); // internal API

// Listening to value changes
const listenerId = opacity.addListener(({ value }) => {
  console.log('Current opacity:', value);
});
opacity.removeListener(listenerId);
Warning

Never create Animated.Value inside a render function without useRef. Each render would create a new value, breaking the animation. The value must persist across renders. Use useRef(new Animated.Value(0)).current or useMemo(() => new Animated.Value(0), []).

2. Animation Drivers: timing, spring, decay

2.1 Animated.timing

// Linear or eased animation over a fixed duration
Animated.timing(value, {
  toValue: 1,
  duration: 300,              // milliseconds
  easing: Easing.bezier(0.25, 0.1, 0.25, 1), // cubic-bezier
  delay: 100,                // start delay
  useNativeDriver: true,
}).start((result) => {
  // result.finished === true if animation completed
  // result.finished === false if animation was interrupted
});

// Common easing functions
Easing.linear          // constant velocity
Easing.ease            // default ease-in-out
Easing.in(Easing.quad)  // accelerating from zero
Easing.out(Easing.quad) // decelerating to zero
Easing.inOut(Easing.quad) // accelerate then decelerate
Easing.bezier(0.42, 0, 0.58, 1) // custom cubic bezier

2.2 Animated.spring

// Physics-based spring animation
Animated.spring(value, {
  toValue: 1,
  // Bounciness model (simpler)
  friction: 7,               // Controls damping (higher = less bouncy)
  tension: 40,               // Controls speed (higher = faster)

  // OR: Stiffness model (more precise)
  // stiffness: 100,
  // damping: 10,
  // mass: 1,

  // OR: iOS UIKit model
  // speed: 12,
  // bounciness: 8,

  useNativeDriver: true,
  restSpeedThreshold: 0.001,  // When to consider "at rest"
  restDisplacementThreshold: 0.001,
}).start();

2.3 Animated.decay

// Momentum-based animation that decelerates from initial velocity
Animated.decay(value, {
  velocity: 0.5,             // Initial velocity (from gesture)
  deceleration: 0.997,       // Rate of deceleration (0-1)
  useNativeDriver: true,
}).start();

// Common use: fling gesture → decay animation
// The gesture handler provides the release velocity,
// decay continues the motion with natural deceleration

3. Composition: sequence, parallel, stagger

Animation Composition ════════════════════════════════════════════════════ sequence([A, B, C]) parallel([A, B, C]) stagger(100, [A, B, C]) ───────────────── ───────────────────── ────────────────────── Time → Time → Time → ├── A ──┤ ├── A ──────────┤ ├── A ──────────┤ ├── B ──┤ ├── B ──────────┤ ├── B ──────────┤ ├── C ├── C ──────────┤ ├── C ──────────┤ ├100ms┤ delay between each delay(200, A) loop({ animation: A }) ───────────── ────────────────────── Time → Time → ├─ wait ─├── A ──┤ ├── A ──├── A ──├── A ──├── ... ├─ 200ms ┤ iterations: -1 = infinite 
// Sequence: animations run one after another
Animated.sequence([
  Animated.timing(opacity, { toValue: 1, duration: 200, useNativeDriver: true }),
  Animated.timing(scale, { toValue: 1.2, duration: 150, useNativeDriver: true }),
  Animated.timing(scale, { toValue: 1, duration: 150, useNativeDriver: true }),
]).start();

// Parallel: animations run simultaneously
Animated.parallel([
  Animated.timing(opacity, { toValue: 1, duration: 300, useNativeDriver: true }),
  Animated.spring(translateY, { toValue: 0, useNativeDriver: true }),
]).start();

// Stagger: parallel with incremental delay
const items = [anim1, anim2, anim3, anim4, anim5];
Animated.stagger(100, items.map((anim) =>
  Animated.timing(anim, { toValue: 1, duration: 300, useNativeDriver: true })
)).start();

// Loop: repeat animation
Animated.loop(
  Animated.sequence([
    Animated.timing(pulse, { toValue: 1.1, duration: 500, useNativeDriver: true }),
    Animated.timing(pulse, { toValue: 1, duration: 500, useNativeDriver: true }),
  ]),
  { iterations: -1 } // infinite
).start();

// Delay
Animated.sequence([
  Animated.delay(500),
  Animated.timing(opacity, { toValue: 1, duration: 300, useNativeDriver: true }),
]).start();

4. Interpolation

Interpolation maps an Animated.Value's range to a different output range, enabling a single value to drive multiple properties with different scales, units, and even non-linear mappings.

const scrollY = useRef(new Animated.Value(0)).current;

// Map scroll position to header opacity
const headerOpacity = scrollY.interpolate({
  inputRange: [0, 100],
  outputRange: [1, 0],
  extrapolate: 'clamp',  // Don't go beyond 0 or 1
});

// Map scroll to header height
const headerHeight = scrollY.interpolate({
  inputRange: [0, 200],
  outputRange: [200, 60],
  extrapolate: 'clamp',
});

// Map to rotation (string output for transforms)
const rotation = progress.interpolate({
  inputRange: [0, 1],
  outputRange: ['0deg', '360deg'],
});

// Map to color
const backgroundColor = progress.interpolate({
  inputRange: [0, 0.5, 1],
  outputRange: ['rgb(255,0,0)', 'rgb(0,255,0)', 'rgb(0,0,255)'],
});

// Non-linear mapping with multiple segments
const scale = scrollY.interpolate({
  inputRange: [-100, 0, 100, 200],
  outputRange: [1.5, 1, 1, 0.8],
  extrapolate: 'clamp',
});

// Extrapolation modes
// 'extend' (default): continue the curve beyond the range
// 'clamp': stop at the boundary value
// 'identity': use the input value directly beyond the range
Key Insight

Interpolation is the key to efficient animations. A single Animated.Value (e.g., scroll position) can drive dozens of properties (opacity, scale, translateY, color, rotation) through interpolation — all on the native thread with useNativeDriver. This means one value change propagates to all dependent properties without any JS thread involvement.

5. Animated.event for Gestures

// Animated.event maps native events directly to Animated.Values
const scrollY = useRef(new Animated.Value(0)).current;

<Animated.ScrollView
  onScroll={Animated.event(
    [{ nativeEvent: { contentOffset: { y: scrollY } } }],
    { useNativeDriver: true }
  )}
  scrollEventThrottle={16}  // 60fps — every frame
>
  {/* content */}
</Animated.ScrollView>

// The scrollY value now tracks scroll position on the native thread
// No JS thread involvement for each scroll frame

// Collapsing header driven by scroll
<Animated.View
  style={{
    height: scrollY.interpolate({
      inputRange: [0, 200],
      outputRange: [200, 60],
      extrapolate: 'clamp',
    }),
    opacity: scrollY.interpolate({
      inputRange: [0, 100],
      outputRange: [1, 0],
      extrapolate: 'clamp',
    }),
  }}
/>

6. useNativeDriver Internals

The native driver is the most important performance optimization in the Animated API. Understanding how it works internally is essential for Staff-level interviews.

useNativeDriver: true — How It Works ════════════════════════════════════════════════════ JS Thread Native/UI Thread ───────── ────────────────── 1. Animated.timing(value, { toValue: 1, duration: 300, useNativeDriver: true }).start() │ ▼ 2. Serialize animation config to JSON: { type: "frames", frames: [0, 0.03, 0.06, ...], toValue: 1, iterations: 1 } │ ▼ 3. Send config to native ──────→ 4. Native animation driver via bridge/JSI receives config (ONE message) │ ▼ JS thread is FREE. 5. Native CADisplayLink (iOS) No more JS involvement. or Choreographer (Android) ticks every frame (~16ms) │ ▼ 6. Update native view properties directly (no bridge round-trip) opacity: 0 → 0.03 → 0.06 → ... │ ▼ 7. Animation complete → notify JS via callback Result: 60fps animations even when JS thread is busy (e.g., during heavy computation, network calls, etc.) 
// Properties that support useNativeDriver: true
// ✅ Transform properties: translateX, translateY, scale, rotate, etc.
// ✅ opacity
// ❌ Layout properties: width, height, top, left, margin, padding
// ❌ backgroundColor (partially supported)
// ❌ borderRadius
// ❌ Any property that affects layout

// The native driver can only animate non-layout properties because
// layout changes require Yoga recalculation, which happens on the
// shadow thread — not the UI thread where native animations run.
Architect Insight

The native driver works by serializing the entire animation description (frames, easing curve, target value) into a single message sent to the native side at animation start. The native animation driver then runs independently using the platform's frame callback (CADisplayLink on iOS, Choreographer on Android). This is why the JS thread can be completely blocked and native-driven animations still run at 60fps. However, this also means the animation is "fire and forget" — you can't dynamically change the animation based on JS-side conditions mid-flight.

7. Limitations & Workarounds

Limitation Impact Workaround
No layout property animation with native driver Can't animate width/height natively Use transform: scaleX/scaleY instead
No conditional logic during animation Can't branch based on value mid-animation Use Reanimated worklets
No JS callback per frame with native driver Can't read value synchronously each frame Use Reanimated shared values
String interpolation limited Only deg, rad, and color strings Use Reanimated for complex string interpolation
No gesture-driven conditional animations Can't do "if dragged past threshold, snap" Use Reanimated + Gesture Handler 
// Workaround: Animate width using scaleX + native driver
const widthAnim = useRef(new Animated.Value(0)).current;

<Animated.View
  style={{
    width: 200, // Fixed base width
    transform: [{
      scaleX: widthAnim.interpolate({
        inputRange: [0, 1],
        outputRange: [0, 1],
      }),
    }],
    transformOrigin: 'left center', // Scale from left edge
  }}
/>

8. Architect Considerations

Staff-Level Mental Model

The Animated API is React Native's baseline animation system. It's sufficient for simple transitions (fade, slide, scale) and scroll-driven animations. Its key strength is useNativeDriver — offloading animation to the native thread for guaranteed 60fps. Its key weakness is the inability to run JS logic per frame on the native thread. For anything requiring conditional logic, gesture-driven interactions, or layout animations, you need Reanimated. A Staff engineer uses Animated for simple cases (lower dependency footprint) and Reanimated for complex interactions, understanding the architectural trade-off: Animated sends a static animation description to native, while Reanimated runs actual JS (worklets) on the UI thread.
