Debugging Tools for React Native

React Native debugging spans two worlds — JavaScript and native. Staff engineers must master Flipper for cross-cutting inspection, React DevTools for component debugging, Hermes debugger for JS profiling, and native debuggers (Xcode/Android Studio) for platform-specific issues. Understanding sourcemaps and symbolication is critical for production debugging.

1. Flipper

Flipper is Meta's extensible desktop debugging platform for mobile apps. It connects to React Native apps via a native SDK and provides plugins for network inspection, layout debugging, database browsing, and Hermes profiling.

Flipper Deprecation Note

React Native 0.73+ is moving away from Flipper as the default debugger in favor of the new Hermes-based debugging experience integrated directly into Chrome DevTools. Flipper remains useful for its plugins (network, databases, layout) but is no longer the primary JS debugger. The new approach uses Chrome DevTools Protocol (CDP) directly with Hermes.

3. Chrome DevTools

// New debugging experience (RN 0.73+)
// 1. Start Metro: npx react-native start
// 2. Press 'j' in Metro terminal to open debugger
// 3. Chrome DevTools opens with Hermes connection

// Features available:
// - Sources: breakpoints, step-through, watch expressions
// - Console: evaluate JS in app context
// - Network: (limited, use Flipper for full inspection)
// - Performance: CPU profiling via Hermes
// - Memory: heap snapshots

// Legacy remote debugging (pre-0.73)
// Runs JS in Chrome's V8 instead of device's Hermes/JSC
// WARNING: behavior differences between V8 and Hermes
// - Date parsing differs
// - Regex behavior differs
// - TypedArray support differs
// - Performance characteristics completely different
Warning

Legacy Chrome remote debugging runs your JS code in Chrome's V8 engine, not the device's Hermes/JSC. This means bugs that only appear on-device (Hermes-specific behavior, JSC quirks) won't reproduce in the debugger. Always prefer the new Hermes-native debugging (RN 0.73+) which debugs directly on the device's engine via CDP.

4. Hermes Debugger

// Hermes supports Chrome DevTools Protocol (CDP) natively
// Connection: chrome://inspect → Hermes React Native

// Hermes-specific debugging features:
// 1. Bytecode-level debugging (see compiled output)
// 2. Accurate CPU profiling (no V8 proxy overhead)
// 3. Heap snapshots for memory analysis
// 4. Sampling profiler for production builds

// Programmatic Hermes profiling
if (global.HermesInternal) {
  global.HermesInternal.enableSamplingProfiler();
  // ... run code to profile ...
  const fileName = '/tmp/hermes-profile.cpuprofile';
  global.HermesInternal.dumpSampledTraceToFile(fileName);
  global.HermesInternal.disableSamplingProfiler();
}

// Open profile in Chrome DevTools → Performance tab
// hermes-profile-transformer converts to Chrome format

6. console.log Performance Impact

// console.log in React Native is EXPENSIVE
// Each log call:
// 1. Serializes the message to JSON
// 2. Sends it over the bridge to native
// 3. Native forwards to Flipper/Metro/debugger
// 4. This blocks the JS thread

// Production: strip console.* calls
// babel.config.js
module.exports = {
  presets: ['module:@react-native/babel-preset'],
  env: {
    production: {
      plugins: ['transform-remove-console'],
    },
  },
};

// Alternative: conditional logging
const logger = {
  debug: (...args) => __DEV__ && console.log(...args),
  warn: (...args) => __DEV__ && console.warn(...args),
  error: (...args) => console.error(...args), // keep errors
};
Architect Insight

In production React Native apps, console.log calls can cause measurable frame drops — each call serializes data and crosses the bridge. Always strip console statements from production builds using babel-plugin-transform-remove-console. For production logging, use a structured logging library that batches and sends logs asynchronously without blocking the JS thread.

10. Architect Considerations

Architect Insight

Debugging infrastructure for Staff engineers: (1) Set up sourcemap upload in CI for every release — without it, production crashes are unreadable. (2) Use the new Hermes CDP debugging (RN 0.73+) over legacy Chrome remote debugging. (3) Create a structured logging system that works in dev (console) and production (remote logging service). (4) Strip console.* from production builds. (5) Train the team on Xcode/Android Studio native debugging — many RN bugs manifest at the native layer. (6) Set up Flipper plugins for the team's specific needs (custom plugins for app-specific data).
