Detox E2E Testing

Detox is a gray-box E2E testing framework purpose-built for React Native. Unlike black-box tools (Appium), Detox synchronizes with the app's internal state — waiting for animations, network requests, and timers to complete before asserting. Staff engineers must understand this synchronization model, element matching strategies, and how to build reliable E2E suites in CI.

2. Device Synchronization

// Detox automatically synchronizes with:
// 1. UI thread idle (animations complete)
// 2. JS thread idle (no pending timers/promises)
// 3. Network requests (RN bridge idle)
// 4. React Native bridge queue empty

// Disable synchronization for specific cases
await device.disableSynchronization();
// ... interact with long-running animation
await device.enableSynchronization();

// Wait for specific conditions
await waitFor(element(by.id('loading')))
  .not.toBeVisible()
  .withTimeout(10000);

// Synchronization debugging
await device.setURLBlacklist([
  '.*analytics.*',    // ignore analytics requests
  '.*tracking.*',     // ignore tracking
]);
// Blacklisted URLs won't block synchronization
Synchronization Internals

Detox injects synchronization resources into the app at build time. On iOS, it hooks into CADisplayLink for animations, NSURLSession for network, and dispatch_async for main thread. On Android, it uses IdlingResource (Espresso pattern) for UI thread, OkHttp interceptor for network, and Looper for message queue. When all resources report idle, Detox proceeds with the next action.

3. Element Matching

// Matcher types
element(by.id('submit-button'));           // testID prop
element(by.text('Submit'));                  // visible text
element(by.label('Submit form'));             // accessibilityLabel
element(by.type('RCTTextInput'));             // native view type
element(by.traits(['button']));              // accessibility traits

// Compound matchers
element(by.id('item').withAncestor(by.id('list')));
element(by.id('title').withDescendant(by.text('Hello')));
element(by.id('item').and(by.text('Active')));

// Index-based matching (for duplicate elements)
element(by.id('item')).atIndex(2);  // third matching element

// Setting testID in React Native
<TouchableOpacity testID="submit-button">
  <Text>Submit</Text>
</TouchableOpacity>

4. Actions & Assertions

// Actions
await element(by.id('btn')).tap();
await element(by.id('btn')).longPress(1500);
await element(by.id('btn')).multiTap(2);
await element(by.id('input')).typeText('Hello');
await element(by.id('input')).replaceText('World');
await element(by.id('input')).clearText();
await element(by.id('scroll')).scroll(200, 'down');
await element(by.id('scroll')).scrollTo('bottom');
await element(by.id('card')).swipe('left', 'fast', 0.75);

// Assertions
await expect(element(by.id('title'))).toBeVisible();
await expect(element(by.id('title'))).toExist();
await expect(element(by.id('title'))).toHaveText('Hello');
await expect(element(by.id('input'))).toHaveValue('World');
await expect(element(by.id('toggle'))).toHaveToggleValue(true);
await expect(element(by.id('modal'))).not.toBeVisible();

// Device actions

await device.launchApp({ newInstance: true, url: 'myapp://profile/123' });
await device.sendToHome();
await device.launchApp({ newInstance: false }); // resume
await device.shake();
await device.setOrientation('landscape');

6. CI Setup

# GitHub Actions — Detox iOS
name: E2E Tests
on: [push]
jobs:
  e2e-ios:
    runs-on: macos-14
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with: { node-version: 20 }
      - run: npm ci
      - run: brew tap wix/brew && brew install applesimutils
      - run: npx detox build --configuration ios.sim.release
      - run: npx detox test --configuration ios.sim.release
            --cleanup --headless
      - uses: actions/upload-artifact@v4
        if: failure()
        with:
          name: detox-artifacts
          path: e2e/artifacts/

8. Handling Flaky Tests

// Retry flaky tests
// npx detox test --retries 2

// Increase timeout for slow operations
await waitFor(element(by.id('loaded')))
  .toBeVisible()
  .withTimeout(30000);  // 30s for slow network

// Blacklist flaky network endpoints
await device.setURLBlacklist(['.*cdn.example.com.*']);

// Reset app state between tests
beforeEach(async () => {
  await device.launchApp({
    newInstance: true,
    delete: true,       // clear app data
    permissions: { notifications: 'YES', camera: 'YES' },
  });
});
Warning

Common flakiness sources: (1) Animations that don't complete (disable or mock in E2E builds). (2) Network requests to real servers (use mock server). (3) Date/time-dependent logic (mock system time). (4) Race conditions between JS and native threads (use waitFor). (5) Simulator state leaking between tests (use delete: true in launchApp).

10. Architect Considerations

Architect Insight

E2E testing strategy for Staff engineers: (1) Only test critical user journeys — signup, login, checkout, core feature flows. (2) Run E2E on release builds, not debug — this catches ProGuard/R8 issues and production-only bugs. (3) Use a mock API server (not real backend) for deterministic tests. (4) Run on both iOS and Android in CI — platform-specific bugs are common. (5) Keep E2E tests independent — each test should set up its own state. (6) Budget 10-30 E2E tests total — more than that becomes a maintenance burden. (7) Use test artifacts (screenshots, videos) for debugging failures in CI.
