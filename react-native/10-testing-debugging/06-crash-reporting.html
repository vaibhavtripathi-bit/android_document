Crash Reporting in React Native

React Native apps can crash in two distinct layers — JavaScript exceptions and native crashes — each requiring different capture, symbolication, and reporting mechanisms. Staff engineers must architect a crash reporting pipeline that covers both layers, provides actionable stack traces, and enables crash-free rate monitoring across releases.

// Global JS error handler
import { ErrorUtils } from 'react-native';

const originalHandler = ErrorUtils.getGlobalHandler();

ErrorUtils.setGlobalHandler((error, isFatal) => {
  if (isFatal) {
    crashReporter.captureException(error, {
      level: 'fatal',
      tags: { fatal: true },
    });
  } else {
    crashReporter.captureException(error, {
      level: 'error',
    });
  }
  originalHandler(error, isFatal);
});

// Unhandled Promise rejection handler
const tracking = require('promise/setimmediate/rejection-tracking');
tracking.enable({
  allRejections: true,
  onUnhandled: (id, error) => {
    crashReporter.captureException(error, {
      tags: { unhandledRejection: true },
    });
  },
});

2. Error Boundaries

class ErrorBoundary extends React.Component {
  state = { hasError: false, error: null };

  static getDerivedStateFromError(error) {
    return { hasError: true, error };
  }

  componentDidCatch(error, errorInfo) {
    Sentry.captureException(error, {
      extra: { componentStack: errorInfo.componentStack },
    });
  }

  render() {
    if (this.state.hasError) {
      return (
        <View style={styles.errorContainer}>
          <Text style={styles.errorTitle}>Something went wrong</Text>
          <TouchableOpacity
            onPress={() => this.setState({ hasError: false })}
          >
            <Text>Try Again</Text>
          </TouchableOpacity>
        </View>
      );
    }
    return this.props.children;
  }
}

// Granular error boundaries per feature
const App = () => (
  <ErrorBoundary fallback={<AppCrashScreen />}>
    <NavigationContainer>
      <ErrorBoundary fallback={<ScreenError />}>
        <HomeScreen />
      </ErrorBoundary>
    </NavigationContainer>
  </ErrorBoundary>
);
Warning

Error boundaries only catch errors during rendering, lifecycle methods, and constructors. They do NOT catch: event handler errors, async code (setTimeout, Promises), or errors in the error boundary itself. For event handlers, use try/catch. For async code, use the global error handler or Promise rejection tracking.

3. Sentry Integration

import * as Sentry from '@sentry/react-native';

Sentry.init({
  dsn: 'https://key@sentry.io/project',
  environment: __DEV__ ? 'development' : 'production',
  release: `com.myapp@${version}+${buildNumber}`,
  dist: buildNumber,
  tracesSampleRate: 0.2,  // 20% of transactions
  enableAutoSessionTracking: true,
  sessionTrackingIntervalMillis: 30000,
  attachStacktrace: true,
  enableNative: true,     // capture native crashes
  enableNativeCrashHandling: true,

  beforeSend(event) {
    // Scrub sensitive data
    if (event.request?.headers) {
      delete event.request.headers['Authorization'];
    }
    return event;
  },

  integrations: [
    new Sentry.ReactNativeTracing({
      routingInstrumentation:
        new Sentry.ReactNavigationInstrumentation(),
    }),
  ],
});

// Manual error capture with context
try {
  await processPayment(order);
} catch (error) {
  Sentry.captureException(error, {
    tags: { feature: 'checkout', paymentMethod: 'card' },
    extra: { orderId: order.id, amount: order.total },
    level: 'error',
  });
}

6. Breadcrumbs & Context

// Automatic breadcrumbs (Sentry captures these by default):
// - Navigation events (screen changes)
// - Network requests (URL, status, duration)
// - Console logs (console.log, warn, error)
// - Touch events (component, coordinates)
// - App state changes (foreground/background)

// Manual breadcrumbs for business context
Sentry.addBreadcrumb({
  category: 'checkout',
  message: 'Added item to cart',
  data: { productId: 'abc', quantity: 2 },
  level: 'info',
});

// User context
Sentry.setUser({
  id: 'user-123',
  email: 'user@example.com',
  subscription: 'premium',
});

// Tags for filtering
Sentry.setTag('feature', 'checkout');
Sentry.setTag('experiment', 'new-flow-v2');

9. ANR Detection

// ANR = Application Not Responding
// Android: main thread blocked > 5 seconds
// iOS: watchdog kills app if main thread blocked > ~10 seconds

// Common ANR causes in React Native:
// 1. Synchronous bridge calls blocking main thread
// 2. Large JSON serialization on main thread
// 3. Synchronous storage operations
// 4. Heavy Yoga layout calculations
// 5. Native module doing I/O on main thread

// Sentry ANR detection
Sentry.init({
  enableAppHangTracking: true,  // iOS
  appHangTimeoutInterval: 2,    // seconds
  enableAnrTracking: true,      // Android
  anrTimeout: 5000,             // milliseconds
});

10. Architect Considerations

Architect Insight

Crash reporting architecture for Staff engineers: (1) Use Sentry for comprehensive JS + native crash reporting with sourcemap support. (2) Implement granular error boundaries — per-screen, per-feature, with recovery options. (3) Automate sourcemap/dSYM upload in CI for every release. (4) Set up crash-free rate alerts with escalation policies. (5) Add business context via breadcrumbs and tags — a crash report without context is useless. (6) Track ANR rates alongside crash rates — ANRs are worse UX than crashes. (7) Never ship without symbolication artifacts — unsymbolicated crashes waste engineering time.
