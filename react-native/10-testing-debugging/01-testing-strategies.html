Testing Strategies for React Native

React Native's dual-layer architecture (JavaScript + native) demands a testing strategy that spans both worlds. Staff engineers must design test pyramids that balance confidence, speed, and maintenance cost — knowing exactly what to test at each level and where platform-specific tests are critical.

1. Testing Pyramid for Mobile

Mobile Testing Pyramid ══════════════════════ ┌───────────┐ │ Manual │ ← Exploratory, UX validation │ Testing │ ~5% of effort ├───────────┤ │ E2E Tests │ ← Detox, Appium │ (Critical │ ~15% of effort │ Flows) │ Slow, expensive, high confidence ├─────────────┤ │ Integration │ ← Component + hook tests │ Tests │ ~30% of effort │ (Screens, │ Medium speed, good confidence │ Features) │ ├───────────────┤ │ Unit Tests │ ← Pure functions, hooks, reducers │ (Logic, │ ~50% of effort │ Utilities) │ Fast, cheap, focused └─────────────────┘ Key metrics: - Unit: <10ms per test, 100s of tests - Integration: 50-200ms per test, 10s of tests - E2E: 30-120s per test, critical paths only
Architect Insight

The mobile testing pyramid differs from web: E2E tests are more expensive (device/simulator startup, app installation), integration tests are more valuable (they catch platform-specific rendering issues), and unit tests should focus on business logic rather than component rendering. Aim for 70% unit, 20% integration, 10% E2E by test count.

2. Unit Testing

Unit tests in React Native run in a Node.js environment with Jest. They test isolated JavaScript logic without native module involvement.

// Pure function test
import { formatPrice, calculateDiscount } from '../utils/pricing';

describe('formatPrice', () => {
  it('formats USD correctly', () => {
    expect(formatPrice(1234.5, 'USD')).toBe('$1,234.50');
  });

  it('handles zero', () => {
    expect(formatPrice(0, 'USD')).toBe('$0.00');
  });
});

// Reducer test
import { cartReducer, addItem, removeItem } from '../store/cartSlice';

describe('cartReducer', () => {
  it('adds item to empty cart', () => {
    const state = cartReducer(undefined, addItem({ id: '1', qty: 1 }));
    expect(state.items).toHaveLength(1);
    expect(state.items[0].id).toBe('1');
  });

  it('increments quantity for existing item', () => {
    const initial = { items: [{ id: '1', qty: 1 }] };
    const state = cartReducer(initial, addItem({ id: '1', qty: 1 }));
    expect(state.items[0].qty).toBe(2);
  });
});

2.1 Testing Custom Hooks

import { renderHook, act } from '@testing-library/react-native';
import { useCounter } from '../hooks/useCounter';

describe('useCounter', () => {
  it('increments counter', () => {
    const { result } = renderHook(() => useCounter(0));

    act(() => result.current.increment());

    expect(result.current.count).toBe(1);
  });

  it('respects max value', () => {
    const { result } = renderHook(() =>
      useCounter(9, { max: 10 })
    );

    act(() => result.current.increment());
    act(() => result.current.increment());

    expect(result.current.count).toBe(10);
  });
});

3. Integration Testing

import { render, screen, fireEvent, waitFor }
  from '@testing-library/react-native';
import { LoginScreen } from '../screens/LoginScreen';

describe('LoginScreen', () => {
  it('shows error on invalid credentials', async () => {
    const mockLogin = jest.fn().mockRejectedValue(
      new Error('Invalid credentials')
    );

    render(<LoginScreen onLogin={mockLogin} />);

    fireEvent.changeText(
      screen.getByPlaceholderText('Email'), 'bad@email.com'
    );
    fireEvent.changeText(
      screen.getByPlaceholderText('Password'), 'wrong'
    );
    fireEvent.press(screen.getByRole('button', { name: 'Sign In' }));

    await waitFor(() => {
      expect(screen.getByText('Invalid credentials')).toBeTruthy();
    });
  });

  it('disables button while loading', async () => {
    const mockLogin = jest.fn(
      () => new Promise(() => {})  // never resolves
    );

    render(<LoginScreen onLogin={mockLogin} />);

    fireEvent.changeText(screen.getByPlaceholderText('Email'), 'a@b.com');
    fireEvent.changeText(screen.getByPlaceholderText('Password'), 'pass');
    fireEvent.press(screen.getByRole('button', { name: 'Sign In' }));

    await waitFor(() => {
      expect(screen.getByRole('button')).toBeDisabled();
    });
  });
});

4. End-to-End Testing

// Detox E2E test example
describe('Checkout Flow', () => {
  beforeAll(async () => {
    await device.launchApp({ newInstance: true });
  });

  it('completes purchase end-to-end', async () => {
    await element(by.id('product-list')).scrollTo('bottom');
    await element(by.id('product-item-1')).tap();
    await element(by.id('add-to-cart')).tap();
    await element(by.id('cart-tab')).tap();
    await element(by.id('checkout-button')).tap();

    await expect(element(by.text('Order Confirmed'))).toBeVisible();
  });
});

6. Platform-Specific Testing

// Testing platform-specific behavior
import { Platform } from 'react-native';

describe('PlatformButton', () => {
  it('renders iOS style on iOS', () => {
    Platform.OS = 'ios';
    const { getByTestId } = render(<PlatformButton />);
    expect(getByTestId('button').props.style).toMatchObject({
      borderRadius: 8,
    });
  });

  it('renders Android style on Android', () => {
    Platform.OS = 'android';
    const { getByTestId } = render(<PlatformButton />);
    expect(getByTestId('button').props.style).toMatchObject({
      elevation: 2,
    });
  });
});

7. Snapshot Testing

import { render } from '@testing-library/react-native';

it('matches snapshot', () => {
  const tree = render(<ProfileCard user={mockUser} />);
  expect(tree.toJSON()).toMatchSnapshot();
});
ProsCons
Catches unintended UI changesLarge snapshots are hard to review
Easy to writeDevelopers blindly update snapshots
Documents component outputBrittle — breaks on any change
Good for design system componentsDoesn't test behavior
Architect Insight

Snapshot tests are most valuable for stable, shared components (design system). They're counterproductive for rapidly evolving screens. Best practice: (1) Use inline snapshots for small components. (2) Keep snapshots focused (render only the component, not the full tree). (3) Review snapshot diffs in PRs as carefully as code changes. (4) Consider toMatchInlineSnapshot() for better code review experience.

9. CI/CD Integration

Test Execution in CI Pipeline ═════════════════════════════ Push / PR │ ├─→ Lint + Type Check (30s) │ ├── ESLint │ └── TypeScript │ ├─→ Unit Tests (1-2min) │ └── Jest --ci --coverage │ ├─→ Integration Tests (2-5min) │ └── Jest --ci --testPathPattern=integration │ ├─→ E2E Tests (10-30min) ← Run on merge to main │ ├── iOS: Detox on simulator │ └── Android: Detox on emulator │ └─→ Coverage Report └── Fail if below threshold

10. Architect Considerations

Architect Insight

Design your test strategy around confidence, not coverage: (1) Test business logic exhaustively at the unit level. (2) Test user-facing behavior at the integration level using @testing-library/react-native — query by role and label, not testID. (3) Reserve E2E for critical revenue paths (signup, checkout, payment). (4) Run E2E on both platforms in CI — platform-specific bugs are common. (5) Set coverage thresholds per module (higher for shared logic, lower for UI). (6) Invest in test infrastructure (factories, builders, MSW handlers) to make writing tests fast.
