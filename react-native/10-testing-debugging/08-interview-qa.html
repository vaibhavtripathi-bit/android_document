Interview Q&A: Testing & Debugging

55+ L6/Staff-level questions covering testing strategies, Jest configuration, Detox E2E testing, debugging tools, performance profiling, crash reporting, and CI/CD pipelines for React Native.

Testing Strategies

Q1: How does the testing pyramid differ for React Native compared to web applications?

Mobile testing pyramids have a thinner E2E layer because E2E tests are more expensive (device/simulator startup, app installation, slower execution). Integration tests are more valuable in RN because they catch platform-specific rendering issues. The recommended ratio is 70% unit (business logic, hooks, reducers), 20% integration (screen behavior with RNTL), 10% E2E (critical user journeys with Detox). Web apps can afford more E2E tests because browser automation is faster and cheaper than mobile device automation.

Q2: What should you test at each level of the testing pyramid for a React Native app?

Unit: Business logic, reducers, selectors, utility functions, custom hooks, API response transformers, validation logic. Integration: Screen behavior (render + interact + assert), form flows, state changes, API integration with mocked server (MSW), accessibility properties. E2E: Critical user journeys only — signup, login, checkout, core feature flows. Visual: Design system components via Storybook. Don't test: animations at unit level, native module internals in JS tests, or edge cases at E2E level.

Q3: What are the pros and cons of snapshot testing in React Native?

Pros: Catches unintended UI changes, easy to write, documents component output, good for stable design system components. Cons: Large snapshots are hard to review, developers blindly update snapshots (jest -u), brittle (breaks on any change), doesn't test behavior, gives false confidence. Best practice: use inline snapshots for small components, keep snapshots focused (render only the component), review snapshot diffs as carefully as code, and prefer behavioral tests over snapshots for feature code.

Q4: How do you handle platform-specific testing in React Native?

Three approaches: (1) Mock Platform.OS in Jest to test both platform branches in unit tests. (2) Use Detox with separate configurations (ios.sim.release, android.emu.release) to run E2E tests on both platforms. (3) For platform-specific files (.ios.tsx, .android.tsx), test each file independently. In CI, run unit tests once (they mock Platform.OS) but run E2E tests on both platforms — platform-specific bugs are common and only caught by running on real devices/simulators.

Jest Testing

Q5: Why is transformIgnorePatterns critical in Jest configuration for React Native?

React Native packages ship untranspiled ES modules (using import/export syntax). Jest runs in Node.js which doesn't natively support ES modules. transformIgnorePatterns tells Jest which node_modules to transform via Babel. Without it, you get SyntaxError: Unexpected token 'export'. You must exclude RN packages from the ignore pattern: node_modules/(?!(react-native|@react-native|@react-navigation|...)). This list must be maintained as you add new RN-ecosystem dependencies.

Q6: How do you mock native modules in Jest for React Native?

Three approaches: (1) Manual mocks: Create __mocks__/react-native-camera.js in the project root — Jest auto-discovers these. (2) Factory mocks: jest.mock('module', () => ({...})) in test files or jest.setup.js. (3) Library-provided mocks: Many RN libraries ship mock files (e.g., @react-native-async-storage/async-storage/jest/async-storage-mock). For complex native modules, create partial mocks using jest.requireActual() to keep real logic while mocking native calls.

Q7: What is the recommended query priority in @testing-library/react-native?

Priority (most to least preferred): (1) getByRole — tests accessibility for free. (2) getByLabelText — tests accessibility labels. (3) getByPlaceholderText — for inputs. (4) getByText — for visible text. (5) getByDisplayValue — for input values. (6) getByTestId — last resort, doesn't test accessibility. Using accessible queries means your tests verify that the app is accessible to screen readers. If you can't find an element by role or label, it's a sign the component has accessibility issues.

Q8: How do you test async operations in React Native with Jest?

Use waitFor to poll until an assertion passes: await waitFor(() => expect(screen.getByText('Loaded')).toBeTruthy()). Use findBy* queries which combine getBy + waitFor: await screen.findByText('Loaded'). For API mocking, use MSW (Mock Service Worker) which intercepts actual fetch/axios calls, testing the real code path. For timer-based async, use jest.useFakeTimers() + jest.advanceTimersByTime(). Always await act() when state updates happen outside React's batching.

Q9: How do you test navigation in React Native?

Two approaches: (1) Mock navigation: Spy on useNavigation to verify navigate() calls: jest.spyOn(require('@react-navigation/native'), 'useNavigation').mockReturnValue({ navigate: mockNavigate }). (2) Render with NavigationContainer: Create a renderWithNavigation() helper that wraps components in a real navigator for integration tests. The first approach is simpler for unit tests; the second tests the full navigation flow including screen transitions and params passing.

Q10: How do you handle timer mocking in React Native tests?

Use jest.useFakeTimers() in beforeEach and jest.useRealTimers() in afterEach. Advance timers with jest.advanceTimersByTime(ms) for specific durations, or jest.runAllTimers() to flush all pending timers. For debounced inputs, advance past the debounce delay and assert the callback was called once. Caution: fake timers can interfere with waitFor — if using both, advance timers inside the waitFor callback or use jest.advanceTimersByTimeAsync().

Detox E2E Testing

Q11: What makes Detox a "gray-box" testing framework and why does it matter?

Detox injects synchronization resources into the app at build time, giving it visibility into the app's internal state. It knows when animations complete (CADisplayLink/Choreographer), when network requests finish (NSURLSession/OkHttp interceptor), when JS timers fire, and when the bridge queue is empty. This means Detox automatically waits for the app to be idle before each action/assertion — no arbitrary sleep() calls needed. This makes tests deterministic and fast. Black-box tools (Appium) must poll the UI, leading to flaky tests and slower execution.

Q12: How do you handle flaky E2E tests in Detox?

Strategies: (1) Use device.setURLBlacklist() to exclude analytics/tracking URLs from synchronization. (2) Increase withTimeout() for slow operations. (3) Reset app state between tests with device.launchApp({ newInstance: true, delete: true }). (4) Use --retries 2 for CI runs. (5) Disable animations in E2E builds. (6) Use a mock API server instead of real backend. (7) Avoid date/time-dependent assertions. (8) Use waitFor().toBeVisible().withTimeout() instead of expect().toBeVisible() for elements that appear asynchronously.

Q13: Compare Detox and Appium for React Native E2E testing.

Detox: Gray-box (app-aware synchronization), fast (no arbitrary waits), low flakiness, JavaScript only, purpose-built for RN, moderate setup. Appium: Black-box (polling-based), slower, higher flakiness, any language (WebDriver protocol), generic mobile testing, high setup complexity. Choose Detox for RN-specific projects where speed and reliability matter. Choose Appium when you need cross-platform testing (including non-RN apps), multi-language test suites, or when testing native apps alongside RN.

Q14: How do you set up Detox for parallel test execution in CI?

Use --workers N flag to run tests across N simulators/emulators simultaneously. Configure multiple device entries in .detoxrc.js. For cross-machine sharding: npx detox test --workers 2 -- --shard=1/3 on machine 1, --shard=2/3 on machine 2, etc. Each shard runs a subset of test files. Ensure tests are independent (no shared state) for parallel execution. Collect and merge test artifacts (screenshots, videos) from all shards in CI.

Debugging

Q15: How does the new Hermes debugging experience (RN 0.73+) differ from legacy Chrome remote debugging?

Legacy Chrome debugging runs JS code in Chrome's V8 engine, not the device's Hermes. This means behavior differences (Date parsing, regex, TypedArrays) and inaccurate performance profiling. The new approach uses Chrome DevTools Protocol (CDP) directly with Hermes on the device — JS runs on the actual engine, breakpoints work on real bytecode, and CPU profiling reflects actual device performance. Connect via Metro terminal (press 'j') or chrome://inspect. Flipper is no longer the primary debugger.

Q16: Why is console.log expensive in React Native and how do you mitigate it?

Each console.log call: (1) Serializes the message to JSON. (2) Sends it over the bridge to native. (3) Native forwards to Flipper/Metro/debugger. This blocks the JS thread and can cause frame drops with frequent logging. Mitigation: (1) Use babel-plugin-transform-remove-console to strip all console calls from production builds. (2) Create a conditional logger that only logs in __DEV__. (3) For production logging, use a structured logging library that batches and sends asynchronously.

Q17: How do you debug native crashes in React Native?

iOS: Open .xcworkspace in Xcode, set breakpoints in ObjC/Swift code, use LLDB console (po [view recursiveDescription]), Memory Graph Debugger for retain cycles, View Debugger for 3D hierarchy. Android: Open android/ in Android Studio, attach debugger to process, use Logcat with filters (tag:ReactNative, tag:Hermes), Layout Inspector for view hierarchy. For crash symbolication: upload dSYM files (iOS) and mapping.txt (Android) to your crash reporting service.

Q18: What is the sourcemap chain for Hermes and why does it matter?

Hermes compiles JS to bytecode, creating a two-step transformation: TypeScript → JavaScript (Metro sourcemap) → Hermes Bytecode (Hermes sourcemap). These must be composed into a single sourcemap (TypeScript → Bytecode) for crash symbolication. Without the composed map, crash reports show bytecode offsets instead of source file locations. Use compose-source-maps tool or Sentry's automatic composition. Upload the composed sourcemap to your crash reporting service for every release build.

Performance Profiling

Q19: How do you use the React DevTools Profiler to identify unnecessary re-renders?

Open Profiler tab → Record → Interact → Stop. The flame graph shows each render commit. Gray bars = component did not re-render (memoized). Colored bars = rendered (color intensity indicates duration). Click a component to see "Why did this render?" — shows which props/state/hooks changed. Use this to identify: (1) Components re-rendering due to new object/array references in props. (2) Context consumers re-rendering when unrelated context values change. (3) Missing React.memo() on expensive components.

Q20: How do you profile startup performance in React Native?

Startup phases: native init (50-200ms) → JS engine init (50-150ms) → bundle execution (100-500ms) → first render (50-200ms) → TTI. Measure with: (1) Set global.__APP_START_TIME__ in native code at app launch. (2) Record Date.now() in root component's useEffect for TTI. (3) Use Hermes sampling profiler during startup to find slow module initializations. (4) Use Xcode Instruments Time Profiler for native init. (5) Check Systrace for bridge congestion during startup. Target: TTI < 2 seconds on mid-range devices.

Q21: What tools do you use for memory profiling in React Native?

RN has two memory spaces: (1) JS heap: Use Hermes heap snapshots (global.HermesInternal.createHeapSnapshot()), open in Chrome DevTools Memory tab. Compare snapshots before/after navigation to find leaks. (2) Native heap: Use Xcode Instruments Allocations (iOS) and Android Studio Memory Profiler (Android). Common leak sources: event listeners not removed, timers not cleared, closures capturing large objects, native module references not released, unbounded image cache.

Q22: How does Systrace help debug React Native performance on Android?

Systrace captures system-wide trace data showing CPU scheduling, disk I/O, and thread activity. For RN, it reveals: (1) JS thread activity (React renders, bridge calls). (2) UI thread activity (native view operations). (3) Shadow thread (Yoga layout calculations). (4) Bridge message queue depth. Start via Dev Menu → Start Systrace, or command line. Custom trace sections: Systrace.beginEvent('loadProducts'). Analyze the trace in Chrome to identify which thread is the bottleneck and where bridge congestion occurs.

Q23: How do you use Xcode Instruments for React Native iOS profiling?

Key templates: (1) Time Profiler: CPU usage per thread/function — filter by "com.facebook.react.JavaScript" thread. (2) Allocations: Memory allocation tracking — find objects that grow without bounds. (3) Leaks: Automatic leak detection. (4) Core Animation: Frame rate and GPU usage — enable "Color Blended Layers" (red = expensive blending), "Color Offscreen-Rendered" (yellow = offscreen render pass). (5) Energy Log: Battery impact — important for App Store review.

Crash Reporting

Q24: How do JavaScript crashes differ from native crashes in React Native?

JS crashes: Unhandled exceptions, Promise rejections, type errors. Caught by ErrorUtils.setGlobalHandler and Error Boundaries. Symbolicated via JS sourcemaps. Usually non-fatal (Error Boundary catches them). Native crashes: SIGSEGV, SIGABRT, NSException, Java exceptions. Caught by signal handlers / uncaught exception handlers. Symbolicated via dSYM (iOS) / mapping.txt + native symbols (Android). Always fatal. Both types need separate symbolication pipelines. A complete crash reporting setup must handle both.

Q25: What are the limitations of Error Boundaries in React Native?

Error Boundaries only catch errors during: rendering, lifecycle methods, and constructors of the tree below them. They do NOT catch: (1) Event handler errors (use try/catch). (2) Async code (setTimeout, Promises — use global handler). (3) Server-side rendering. (4) Errors in the Error Boundary itself. Strategy: use granular Error Boundaries per feature/screen with recovery UI, plus a root-level boundary as last resort. Combine with ErrorUtils.setGlobalHandler for non-rendering errors.

Q26: Compare Sentry and Firebase Crashlytics for React Native crash reporting.

Sentry: Excellent JS + native crash capture, full sourcemap support (CLI upload), automatic + manual breadcrumbs, performance monitoring (transactions), release health tracking, self-hosted option, paid (free tier). Crashlytics: Excellent native crash capture, good JS support (via recordError), manual breadcrumbs only, limited sourcemap support, free, no self-hosted. Choose Sentry for comprehensive JS crash reporting with sourcemaps. Choose Crashlytics if already in Firebase ecosystem and native crashes are the primary concern.

Q27: How do you set up a symbolication pipeline for production React Native builds?

Pipeline: (1) Metro generates JS bundle + sourcemap. (2) Hermes compiles to bytecode + generates bytecode sourcemap. (3) Compose both sourcemaps into one (TypeScript → Bytecode). (4) Upload composed sourcemap to Sentry/Crashlytics via CLI. (5) Upload dSYM files (iOS) from Xcode release build. (6) Upload mapping.txt (Android) from Gradle release build. (7) Tag upload with release version + build number. Automate all uploads in CI — never ship without symbolication artifacts.

Q28: What is ANR detection and why is it important for React Native apps?

ANR (Application Not Responding) occurs when the main thread is blocked — Android shows a dialog after 5 seconds, iOS watchdog kills the app after ~10 seconds. Common RN causes: synchronous bridge calls, large JSON serialization, synchronous storage operations, heavy Yoga layout calculations, native modules doing I/O on main thread. ANRs are worse than crashes for UX — the app appears frozen. Sentry detects ANRs via enableAnrTracking (Android) and enableAppHangTracking (iOS). Track ANR rate alongside crash-free rate.

CI/CD

Q29: Compare EAS Build and Fastlane for React Native CI/CD.

EAS Build: Cloud-based, handles code signing automatically, manages native dependencies, build caching, integrated with Expo ecosystem, supports bare RN projects, paid (free tier). Fastlane: Self-hosted (runs on your CI), requires manual code signing setup (Match), highly customizable via Ruby lanes, free, works with any CI provider. Choose EAS for Expo projects or teams that want managed infrastructure. Choose Fastlane for maximum control, existing CI infrastructure, or when you need custom build steps not supported by EAS.

Q30: How does code signing work for iOS and Android, and how do you manage it in CI?

iOS: Requires Apple Developer Certificate (.p12) + Provisioning Profile (.mobileprovision). Types: Development, Ad Hoc, App Store. Use Fastlane Match to store encrypted certificates in a Git repo — CI runs match appstore --readonly to install them. Android: Requires Java Keystore (.jks) with private key. Types: Debug (auto-generated), Release (manual). Store keystore file and passwords as CI secrets. Never commit keystores or certificates to the repository.

Q31: What are the limitations and risks of CodePush OTA updates?

Limitations: (1) Can only update JS bundle and assets — not native code, native modules, or native dependencies. (2) Apple guideline 2.5.2 restricts OTA updates that change app behavior significantly. (3) Google Play has similar policies. (4) Version mismatch between JS and native can cause crashes if native API changed. (5) No rollback mechanism for native code. Risks: shipping broken JS that crashes on launch (use rollback), bypassing QA process, creating version fragmentation. Use for bug fixes and minor UI changes only.

Q32: How do you structure a CI pipeline for React Native with both unit and E2E tests?

Two-phase pipeline: PR phase (fast, every PR): lint + typecheck (1-2min) → unit tests with coverage (2-5min) → integration tests (3-8min) → build check (compile only, 5-15min). Merge phase (thorough, on merge to main): all PR checks + E2E tests on iOS (15-30min) + E2E tests on Android (15-30min) → build + sign → upload sourcemaps → distribute to TestFlight/Play Internal. E2E tests run only on merge because they're slow and expensive.

Q33: What caching strategies reduce React Native CI build times?

Cache layers: (1) node_modules — key on package-lock.json hash. (2) CocoaPods — key on Podfile.lock hash. (3) Gradle caches + wrapper — key on *.gradle* + gradle-wrapper.properties hashes. (4) Hermes pre-built binary — key on RN version. (5) Xcode DerivedData — key on project file + Podfile.lock hash. (6) Metro bundler cache. Impact: uncached iOS build ~20-30min, cached ~8-12min. Uncached Android build ~15-20min, cached ~5-10min.

Q34: How do you manage environment variables across development, staging, and production?

Use react-native-config with .env.development, .env.staging, .env.production files. Never commit .env.production to git — use CI secrets. Build with specific env: ENVFILE=.env.staging npx react-native run-android. For EAS: use eas.json build profiles with environment variables. Create .env.example with placeholder values for documentation. Validate required env vars at app startup to fail fast if misconfigured.

Advanced / Cross-Cutting

Q35: How would you design a testing strategy for a React Native app with native modules?

Layered approach: (1) JS interface tests: Mock native module, test JS wrapper logic (argument validation, response transformation). (2) Native unit tests: XCTest (iOS) / JUnit (Android) for native module logic. (3) Integration tests: Test JS-to-native communication with RNTL + mocked native module. (4) E2E tests: Detox tests that exercise the full native module path on real devices. (5) Contract tests: Verify JS and native sides agree on method signatures and data shapes. The key is testing the JS wrapper and native implementation independently, then verifying integration.

Q36: How do you test React Native apps that use Reanimated animations?

Animations are generally not unit-testable. Strategy: (1) Mock Reanimated in Jest setup: jest.mock('react-native-reanimated', () => require('react-native-reanimated/mock')). (2) Test the state changes that trigger animations, not the animation values. (3) Test that components appear/disappear based on animation completion. (4) For visual animation testing, use Detox E2E tests with screenshots. (5) Use jest.useFakeTimers() + jest.advanceTimersByTime() for Animated API (not Reanimated) timing tests.

Q37: How do you implement automated performance regression testing?

Pipeline: (1) Define performance budgets (TTI < 2s, scroll FPS > 55, JS heap < 150MB). (2) In Detox E2E tests, add custom performance markers: performance.mark('screen_load_start') before navigation, performance.mark('screen_load_end') after content visible. (3) Extract metrics from Hermes sampling profiler output. (4) Compare against baseline in CI — fail build if regression exceeds threshold. (5) Track metrics over time in a dashboard. (6) Profile on a consistent low-end device (not developer's flagship).

Q38: How do you debug a crash that only occurs in production release builds?

Steps: (1) Check crash reporting service (Sentry/Crashlytics) for symbolicated stack trace. (2) If unsymbolicated, verify sourcemaps/dSYMs were uploaded for that release. (3) Build a local release build: npx react-native run-ios --configuration Release. (4) Check if ProGuard/R8 stripped a needed class (Android) — review keep rules. (5) Check if Hermes bytecode compilation changed behavior vs JSC. (6) Enable Hermes sampling profiler in production to capture CPU profile around the crash. (7) Add breadcrumbs to narrow down the code path.

Q39: How do you test deep linking in React Native?

Three levels: (1) Unit: Test URL parsing logic — given a URL, verify correct screen name and params are extracted. (2) Integration: Mock Linking.getInitialURL() and verify the correct screen renders with RNTL. (3) E2E: Detox supports launching with URL: device.launchApp({ url: 'myapp://product/123' }) — verify correct screen is displayed. Test both cold start (app not running) and warm start (app in background) scenarios. Test malformed URLs to verify error handling.

Q40: How do you set up a release train for a React Native app?

Weekly release train: (1) Monday: cut release branch from main, bump version. (2) Monday-Wednesday: QA on release branch (TestFlight/Play Internal). (3) Wednesday: fix critical bugs on release branch (cherry-pick from main). (4) Thursday: submit to App Store/Play Store. (5) Friday: monitor crash-free rate after staged rollout. CodePush for hotfixes between releases. Automate: version bumping, changelog generation, store submission, sourcemap upload. Use staged rollout (10% → 25% → 50% → 100%) with crash-free rate gates.

Q41: How do you handle test data management for E2E tests?

Strategies: (1) Mock API server: Run a local server (json-server, MSW) with deterministic test data — most reliable. (2) Seed database: Use API endpoints to create test data before each test — requires backend support. (3) Test accounts: Pre-created accounts in staging environment — fragile, state leaks between tests. (4) App state injection: Detox supports device.launchApp({ launchArgs: { mockData: true } }) — app reads launch args and uses mock data. Best practice: mock API server + clean app state (delete: true) between tests.

Q42: How do you monitor and improve crash-free rate?

Monitoring: (1) Track crash-free rate per release in Sentry/Crashlytics. (2) Set alerts: < 99% = P1, < 99.5% = P2. (3) Monitor adoption rate — if users aren't updating, old crashes persist. (4) Track crash-free rate by device, OS version, and app version. Improvement: (1) Triage top crashes weekly. (2) Add Error Boundaries around crash-prone features. (3) Use staged rollout — halt at 10% if crash-free drops. (4) Add breadcrumbs to top crashes for better debugging. (5) Use CodePush for JS-only hotfixes. Target: > 99.5% crash-free.

Q43: How do you test push notifications in React Native?

Layered approach: (1) Unit: Test notification handler logic — given a notification payload, verify correct action (navigate, update badge, show in-app alert). (2) Integration: Mock the notification library, fire events, verify UI response. (3) E2E: Detox supports sending notifications: device.sendUserNotification({ trigger: { type: 'push' }, ... }). (4) Manual: Use Firebase Console or APNs tool to send test notifications to specific devices. Test: cold start (app not running), foreground, and background scenarios.

Q44: How do you handle flaky network conditions in testing?

Unit/Integration: Use MSW to simulate network conditions — delayed responses (delay(3000)), errors (HttpResponse.error()), timeouts. E2E: Use Detox's network mocking or a proxy server (Charles Proxy in CI). Test scenarios: (1) Slow network (3G simulation). (2) Network timeout. (3) Server error (500). (4) Offline mode. (5) Network recovery (offline → online). For offline testing in Detox: device.setURLBlacklist(['.*']) to block all requests, then clear to restore.

Q45: How do you implement visual regression testing for React Native?

Approaches: (1) Storybook + Chromatic: Render components in Storybook, Chromatic captures screenshots and diffs against baseline. Works for web builds via react-native-web. (2) Detox screenshots: Take screenshots at key points in E2E tests, compare with baseline images using pixel-diff tools. (3) Snapshot testing: Jest snapshots capture component tree structure (not visual appearance). Best for design system components. Limitation: RN doesn't render to pixels in Jest — visual testing requires a running app (Detox) or web rendering (Storybook).

Q46: How do you test accessibility in a CI pipeline?

Layers: (1) ESLint: eslint-plugin-react-native-a11y catches missing labels, roles in code review. (2) Unit tests: RNTL queries by role/label — if a test can't find an element by accessible query, it's an accessibility bug. (3) Snapshot tests: Verify accessibility props are present in component output. (4) Detox E2E: Interact via accessibility IDs, verifying screen reader compatibility. (5) Automated audit: axe-core for web builds, Accessibility Scanner for Android (via Detox artifacts). (6) Manual checklist: VoiceOver/TalkBack testing before each release.

Q47: How do you debug a memory leak in a React Native app?

Process: (1) Reproduce: navigate to suspect screen, go back, repeat 10 times. (2) Take Hermes heap snapshots before and after — compare retained objects. (3) Look for: growing arrays, event listener accumulation, component instances not garbage collected. (4) Check native memory with Xcode Instruments (Allocations) or Android Studio Profiler. (5) Common causes: missing cleanup in useEffect return, setInterval not cleared, addEventListener without removeEventListener, closures capturing component state. (6) Use WeakRef for caches that should allow GC.

Q48: How do you handle code coverage reporting in a monorepo with React Native?

Setup: (1) Configure Jest with collectCoverageFrom patterns per package. (2) Set per-module thresholds: higher for shared logic (90%+), lower for UI (70%+). (3) Use --coverage --coverageReporters=json in CI. (4) Merge coverage reports from multiple packages using istanbul-merge. (5) Upload to Codecov/Coveralls for PR comments and trend tracking. (6) Enforce coverage gates: fail CI if coverage drops below threshold. (7) Exclude generated files, mocks, and type definitions from coverage.

Q49: How do you test React Native apps on real devices in CI?

Options: (1) Device farms: AWS Device Farm, Firebase Test Lab, BrowserStack — run Detox/Appium tests on real devices in the cloud. (2) Self-hosted devices: Mac Minis with connected iPhones/Androids — cheaper for high volume. (3) Simulators/emulators: Most common for CI — faster, cheaper, but miss device-specific issues. Strategy: run E2E on simulators for every merge, run on real devices weekly or before releases. Real devices catch: performance issues on low-end hardware, camera/sensor integration, and device-specific rendering bugs.

Q50: How do you implement feature flags in React Native and test them?

Implementation: Use a feature flag service (LaunchDarkly, Unleash, Firebase Remote Config). Wrap features in flag checks: if (flags.newCheckout) { ... }. Testing: (1) Unit tests: mock flag values to test both paths. (2) Integration tests: render component with flag on/off, verify correct behavior. (3) E2E tests: set flags via API before test, verify feature presence/absence. (4) In CI: run test suite twice — once with all flags on, once with all flags off — to catch regressions in both paths.

Q51: How do you handle app store review rejection and iterate quickly?

Prevention: (1) Test with production build before submission. (2) Follow platform guidelines (HIG, Material Design). (3) Include privacy manifest (iOS 17+). (4) Provide demo account for reviewers. (5) Write clear app description and screenshots. When rejected: (1) Read rejection reason carefully. (2) Fix the specific issue (don't change other things). (3) Reply to the review with explanation if it's a misunderstanding. (4) Use CodePush for JS-only fixes that don't need re-review. (5) Resubmit with expedited review request if critical.

Q52: How do you implement automated app store submission in CI?

iOS: Fastlane deliver uploads to App Store Connect, sets metadata, screenshots, and submits for review. Use App Store Connect API key (not password) for CI authentication. Android: Fastlane upload_to_play_store with service account JSON key. For EAS: eas submit --profile production --platform all. Automate: version bumping (auto-increment build number), changelog generation from git commits, screenshot generation via Detox + device.takeScreenshot(), metadata localization from translation files.

Q53: How do you handle backward compatibility when updating React Native versions?

Process: (1) Read the changelog and breaking changes for each version between current and target. (2) Use the React Native Upgrade Helper (react-native-community/upgrade-helper) for diff. (3) Update incrementally (one minor version at a time) rather than jumping multiple versions. (4) Run full test suite after each increment. (5) Test on both platforms. (6) Check third-party library compatibility — some libraries lag behind RN versions. (7) For major upgrades (New Architecture), create a feature branch and test extensively before merging.

Q54: How do you debug performance issues in React Native list rendering?

Diagnosis: (1) Enable React DevTools Profiler — check if list items re-render unnecessarily. (2) Check getItemLayout — without it, FlatList must measure each item. (3) Check keyExtractor — unstable keys cause full re-renders. (4) Use Flipper Performance plugin — check JS FPS during scroll. (5) Use Systrace (Android) to check bridge congestion during scroll. Fixes: (1) React.memo() on list item components. (2) Provide getItemLayout for fixed-height items. (3) Increase windowSize and maxToRenderPerBatch. (4) Use removeClippedSubviews on Android. (5) Move heavy computation to useMemo.

Q55: How would you architect a comprehensive quality assurance system for a large React Native app?

Architecture: (1) Static analysis: TypeScript strict mode, ESLint with accessibility rules, Prettier. (2) Unit tests: Jest with 80%+ coverage on business logic, custom hooks, reducers. (3) Integration tests: RNTL for screen behavior, MSW for API mocking. (4) E2E tests: Detox for 10-20 critical user journeys on both platforms. (5) Visual tests: Storybook + Chromatic for design system. (6) Performance tests: Automated TTI and FPS regression detection. (7) Crash monitoring: Sentry with 99.5% crash-free gate. (8) Release process: Staged rollout with automatic halt on crash-free regression. (9) Manual QA: Accessibility testing with VoiceOver/TalkBack before each release.
