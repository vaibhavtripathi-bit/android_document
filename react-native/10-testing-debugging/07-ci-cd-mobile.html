CI/CD for React Native

Mobile CI/CD is fundamentally more complex than web — code signing, platform-specific builds, app store submissions, and OTA updates add layers that Staff engineers must architect carefully. From EAS Build and Fastlane to CodePush and environment management, this document covers the complete pipeline.

2. Fastlane Integration

# Fastfile — iOS lane
platform :ios do
  lane :beta do
    increment_build_number
    match(type: "appstore")
    build_app(
      workspace: "ios/MyApp.xcworkspace",
      scheme: "MyApp",
      configuration: "Release",
      export_method: "app-store"
    )
    upload_to_testflight(
      skip_waiting_for_build_processing: true
    )
    sentry_upload_dsym
  end

  lane :release do
    beta
    deliver(
      submit_for_review: true,
      automatic_release: false,
      force: true
    )
  end

3. GitHub Actions Setup

# .github/workflows/build.yml
name: Build & Test
on:
  push: { branches: [main] }
  pull_request: { branches: [main] }

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with: { node-version: 20, cache: 'npm' }
      - run: npm ci
      - run: npx tsc --noEmit
      - run: npx eslint src/
      - run: npx jest --ci --coverage --forceExit

  build-ios:
    needs: test
    runs-on: macos-14
    if: github.ref == 'refs/heads/main'
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with: { node-version: 20, cache: 'npm' }
      - uses: ruby/setup-ruby@v1
        with: { ruby-version: 3.2, bundler-cache: true }
      - run: npm ci
      - run: cd ios && pod install --repo-update
      - run: bundle exec fastlane ios beta
        env:
          MATCH_PASSWORD: ${{ secrets.MATCH_PASSWORD }}
          APP_STORE_CONNECT_API_KEY: ${{ secrets.ASC_KEY }}

  build-android:
    needs: test
    runs-on: ubuntu-latest
    if: github.ref == 'refs/heads/main'
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with: { node-version: 20, cache: 'npm' }
      - uses: actions/setup-java@v4
        with: { distribution: 'zulu', java-version: 17 }
      - run: npm ci
      - run: bundle exec fastlane android beta
        env:
          ANDROID_KEYSTORE_PASSWORD: ${{ secrets.KS_PASS }}

6. CodePush OTA Updates

// CodePush enables over-the-air JS bundle updates
// Bypasses app store review for JS-only changes

import codePush from 'react-native-code-push';

const codePushOptions = {
  checkFrequency: codePush.CheckFrequency.ON_APP_RESUME,
  installMode: codePush.InstallMode.ON_NEXT_RESTART,
  mandatoryInstallMode: codePush.InstallMode.IMMEDIATE,
  minimumBackgroundDuration: 60 * 5,  // 5 min in background
};

const App = codePush(codePushOptions)(RootApp);

// Manual update check
const checkForUpdate = async () => {
  const update = await codePush.checkForUpdate();
  if (update) {
    const pkg = await update.download((progress) => {
      setDownloadProgress(progress.receivedBytes / progress.totalBytes);
    });
    await pkg.install(codePush.InstallMode.IMMEDIATE);
    codePush.restartApp();
  }
};

// Release update via CLI:
// appcenter codepush release-react -a Owner/MyApp-iOS \
//   -d Production --mandatory
Warning

CodePush limitations: (1) Can only update JS bundle and assets — not native code. (2) Apple's App Store guideline 2.5.2 restricts OTA updates that change app behavior significantly. (3) Google Play has similar policies. (4) Version mismatches between JS and native can cause crashes. (5) Always test OTA updates on both platforms before releasing. Use CodePush for bug fixes and minor UI changes, not new features.

8. Build Caching

# GitHub Actions caching strategy

# Node modules
- uses: actions/setup-node@v4
  with: { node-version: 20, cache: 'npm' }

# CocoaPods
- uses: actions/cache@v4
  with:
    path: ios/Pods
    key: pods-${{ hashFiles('ios/Podfile.lock') }}

# Gradle
- uses: actions/cache@v4
  with:
    path: |
      ~/.gradle/caches
      ~/.gradle/wrapper
    key: gradle-${{ hashFiles('**/*.gradle*', '**/gradle-wrapper.properties') }}

# Hermes pre-built (saves 5-10 min)
- uses: actions/cache@v4
  with:
    path: node_modules/react-native/sdks/hermesc
    key: hermes-${{ hashFiles('node_modules/react-native/package.json') }}

# Xcode derived data
- uses: actions/cache@v4
  with:
    path: ios/build/DerivedData
    key: xcode-${{ hashFiles('ios/Podfile.lock', 'ios/MyApp.xcodeproj/project.pbxproj') }}

const apiUrl = Config.API_URL;

// Build with specific env:
// ENVFILE=.env.staging npx react-native run-android
// ENVFILE=.env.production fastlane ios release
Architect Insight

Never commit .env.production to git. Use CI secrets management (GitHub Secrets, EAS Secrets) to inject production values at build time. Create a .env.example with placeholder values for documentation. For sensitive keys (API keys, signing credentials), use the CI platform's encrypted secrets — never store them in the repository, even encrypted.

10. Architect Considerations

Architect Insight

CI/CD architecture for Staff engineers: (1) Use EAS Build for Expo projects — it handles code signing, native dependencies, and build caching. (2) For bare RN, use Fastlane + GitHub Actions/CircleCI. (3) Implement a release train: weekly beta → staged production rollout. (4) Automate sourcemap upload for every build — never ship without symbolication. (5) Use CodePush/EAS Update for JS-only hotfixes, but never for feature releases. (6) Cache aggressively — CocoaPods, Gradle, Hermes, node_modules. (7) Run E2E tests on merge to main, not on every PR (too slow/expensive). (8) Implement build number auto-increment to avoid store rejection.
