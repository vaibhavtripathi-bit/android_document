Threading Model

React Native operates across three primary threads with distinct responsibilities. Understanding thread communication, blocking behavior, and scheduling is essential for building responsive apps and diagnosing performance issues at Staff level.

2. JS Thread Deep Dive

The JS thread runs the JavaScript VM (Hermes/JSC) and is single-threaded. All React reconciliation, state updates, event handler callbacks, and business logic execute on this thread.

// Everything on the JS thread is sequential:

// Frame timeline on JS thread:
// 1. Process native events (touch, scroll, timer callbacks)
// 2. Execute event handlers (onPress, onScroll, etc.)
// 3. Process state updates (setState batching)
// 4. React reconciliation (fiber tree walk)
// 5. Generate UIManager commands
// 6. Flush bridge queue
// 7. Execute microtasks (Promise.then, queueMicrotask)
// 8. Execute requestAnimationFrame callbacks

// Why single-threaded?
// 1. JavaScript is inherently single-threaded (no shared memory)
// 2. React's reconciliation assumes single-threaded execution
// 3. Simplifies state management (no locks, no races)
// 4. Matches web mental model (event loop)

// Consequence: long-running JS blocks EVERYTHING
function expensiveCalculation() {
  // This blocks the JS thread for 100ms
  let sum = 0;
  for (let i = 0; i < 10_000_000; i++) sum += Math.sqrt(i);
  return sum;
}
// During those 100ms:
// - No touch events processed
// - No state updates
// - No animations (JS-driven ones)
// - UI thread continues independently (native animations OK)
Critical Understanding

Blocking the JS thread does NOT freeze the UI thread. Native animations, touch recognition, and scroll physics continue running on the UI thread. But any JS-driven response (event handlers, state updates, JS-animated values) will be delayed. This is why native-driven animations (useNativeDriver: true) remain smooth even when JS is busy.

3. Main/UI Thread Deep Dive

// The UI thread (main thread) handles all platform UI operations

// iOS: Main thread = main dispatch queue
// - UIKit operations MUST be on main thread
// - CADisplayLink fires on main thread (60/120fps)
// - Touch events originate on main thread
// - Core Animation compositing

// Android: Main thread = UI thread = Looper.getMainLooper()
// - View operations MUST be on UI thread
// - Choreographer.doFrame() on UI thread
// - Touch events dispatched on UI thread
// - Hardware-accelerated rendering

// What runs on UI thread in React Native:
// 1. Native view creation/mutation (from shadow thread results)
// 2. Touch event recognition and dispatch
// 3. Native-driven animations (Animated with useNativeDriver)
// 4. Scroll physics and momentum
// 5. Keyboard show/hide animations
// 6. Navigation transitions (native stack)
// 7. Accessibility announcements
// 8. StatusBar updates

// DANGER: Blocking UI thread = frozen app
// Unlike JS thread (where only JS responses are delayed),
// blocking UI thread freezes EVERYTHING:
// - No touch response
// - No scroll
// - No animations
// - ANR dialog on Android (>5s block)
// - Watchdog kill on iOS (>10s block)

4. Shadow/Background Thread

// The shadow thread handles layout computation

// Responsibilities:
// 1. Maintain shadow node tree (C++ objects)
// 2. Run Yoga layout calculations
// 3. Diff shadow trees to produce minimal mutations
// 4. Text measurement coordination

// iOS: Dedicated serial dispatch queue
//   dispatch_queue_create("com.facebook.react.ShadowQueue", SERIAL)

// Android: Dedicated thread
//   Thread("native_modules_thread") — shared with NativeModules

// Shadow thread workflow per frame:
// 1. Receive UIManager commands from bridge
// 2. Create/update/delete shadow nodes
// 3. Mark dirty nodes in Yoga tree
// 4. Calculate layout (YGNodeCalculateLayout)
// 5. Collect layout changes
// 6. Dispatch view mutations to UI thread

// The shadow thread is the "translator" between
// React's declarative descriptions and native imperative APIs

5. Thread Communication via Bridge

Thread Communication Patterns ═════════════════════════════ 1. JS → Native (most common): JS Thread ──bridge──→ Shadow Thread ──dispatch──→ UI Thread (setState) (layout calc) (view update) 2. Native → JS (events): UI Thread ──bridge──→ JS Thread (touch) (onPress handler) 3. Timer flow: Timer Thread ──signal──→ JS Thread (setTimeout expired) (callback execution) 4. Native animation (bypasses JS): UI Thread ──direct──→ UI Thread (CADisplayLink) (update animated values) No bridge crossing! No JS involvement! 5. Gesture handler (react-native-gesture-handler): UI Thread ──direct──→ UI Thread (gesture recognition) ──bridge──→ JS Thread (only for state callbacks)

6. Native Animations on UI Thread

// Native-driven animations run entirely on UI thread

import { Animated } from 'react-native';

const opacity = useRef(new Animated.Value(0)).current;

// JS-driven animation (BAD for performance):
Animated.timing(opacity, {
  toValue: 1,
  duration: 300,
  useNativeDriver: false,  // ← runs on JS thread
}).start();
// Each frame: JS calculates value → bridge → native updates view
// If JS thread is busy, animation drops frames

// Native-driven animation (GOOD):
Animated.timing(opacity, {
  toValue: 1,
  duration: 300,
  useNativeDriver: true,  // ← runs on UI thread
}).start();
// Animation config sent to native ONCE
// Native CADisplayLink/Choreographer drives each frame
// JS thread completely uninvolved after start()

// Limitation: useNativeDriver only supports non-layout props:
// ✅ opacity, transform (translateX/Y, scale, rotate)
// ❌ width, height, padding, margin, top, left, flex
// Reason: layout props require Yoga recalculation (shadow thread)

// react-native-reanimated v2+ solves this:
// Runs JS worklets on UI thread via JSI
// Can animate ANY prop including layout props
import Reanimated, { useSharedValue, withTiming } from 'react-native-reanimated';

const width = useSharedValue(100);
width.value = withTiming(200, { duration: 300 });
// Runs entirely on UI thread, including layout recalculation

7. InteractionManager

InteractionManager allows deferring expensive work until after animations and interactions complete, preventing jank during transitions.

import { InteractionManager } from 'react-native';

// Defer expensive work until after navigation animation
useEffect(() => {
  const task = InteractionManager.runAfterInteractions(() => {
    // This runs after all animations/interactions complete
    fetchLargeDataset();
    initializeExpensiveModule();
    precomputeSearchIndex();
  });

  return () => task.cancel();
}, []);

// How it works internally:
// 1. Animations register as "interactions" when started
// 2. InteractionManager maintains a count of active interactions
// 3. When count reaches 0, queued callbacks execute
// 4. Callbacks run on JS thread (still blocks JS if expensive)

// Creating custom interactions:
const handle = InteractionManager.createInteractionHandle();
// ... do animation or transition ...
InteractionManager.clearInteractionHandle(handle);
// Queued runAfterInteractions callbacks now execute

// Common pattern: screen load optimization
function ProfileScreen() {
  const [dataReady, setDataReady] = useState(false);

  useEffect(() => {
    InteractionManager.runAfterInteractions(() => {
      // Load data AFTER navigation animation completes
      loadProfileData().then(() => setDataReady(true));
    });
  }, []);

  if (!dataReady) return <SkeletonLoader />;
  return <ProfileContent />;
}

8. requestAnimationFrame Behavior

// requestAnimationFrame in React Native vs Browser

// Browser: callback tied to display refresh (vsync)
// - Runs before paint
// - Guaranteed to be in sync with display
// - Callback receives DOMHighResTimeStamp

// React Native: callback tied to Choreographer/CADisplayLink
// - Fires on JS thread (NOT UI thread)
// - Triggered by native vsync signal crossing bridge to JS
// - Inherent latency: native vsync → bridge → JS callback
// - If JS thread is busy, callback is delayed

requestAnimationFrame((timestamp) => {
  // timestamp: performance.now() value
  // This runs on JS thread
  // Any view updates still need to cross bridge to native
  // Total latency: vsync → JS → bridge → shadow → UI = 1-2 frames
});

// For smooth 60fps animations, DON'T use rAF + setState:
// BAD:
function animate() {
  requestAnimationFrame(() => {
    setValue(prev => prev + 1);  // setState → reconcile → bridge → native
    animate();                    // Always 1-2 frames behind
  });
}

// GOOD: Use Animated API with native driver
// or react-native-reanimated for UI thread animations

console.log('1: synchronous');

setTimeout(() => console.log('5: setTimeout'), 0);

setImmediate(() => console.log('4: setImmediate'));

Promise.resolve().then(() => console.log('2: microtask'));

// Microtasks (Promise, queueMicrotask) run before macrotasks
// setImmediate runs before setTimeout(fn, 0) in RN
// setTimeout(fn, 0) has bridge latency (~5ms)

10. Thread Safety Considerations

// Thread safety issues in React Native

// 1. NativeModule methods can be called from any thread
// iOS: Each module has its own methodQueue (serial)
// Android: All modules share NativeModulesThread by default

// 2. Shared state between JS and native is dangerous
// The bridge is the synchronization mechanism
// No shared memory = no data races (in old arch)
// But: stale data is possible (async updates)

// 3. UI operations MUST be on main thread
// iOS: dispatch_async(dispatch_get_main_queue(), ^{ ... });
// Android: runOnUiThread { ... } or Handler(Looper.getMainLooper())

// 4. Yoga is NOT thread-safe
// All Yoga operations must be on shadow thread
// Never access YGNode from UI thread

// 5. New architecture (JSI) introduces shared memory
// JSI HostObjects are accessible from multiple threads
// Must use mutexes or ensure single-thread access

// Common pattern: thread-safe native module (iOS)
@implementation ThreadSafeModule {
  NSMutableDictionary *_cache;
  dispatch_queue_t _cacheQueue;
}

- (instancetype)init {
  if (self = [super init]) {
    _cache = [NSMutableDictionary new];
    _cacheQueue = dispatch_queue_create(
      "com.app.cache",
      DISPATCH_QUEUE_CONCURRENT
    );
  }
  return self;
}

// Read: concurrent access OK
- (id)getFromCache:(NSString *)key {
  __block id result;
  dispatch_sync(_cacheQueue, ^{
    result = _cache[key];
  });
  return result;
}

// Write: barrier for exclusive access
- (void)setInCache:(NSString *)key value:(id)value {
  dispatch_barrier_async(_cacheQueue, ^{
    _cache[key] = value;
  });
}
@end

11. Architect Considerations

11.1 Thread-Aware Architecture Decisions

Scenario Thread Impact Solution
Heavy JS computation Blocks JS thread → no event processing Move to native module or use Web Workers (limited)
Scroll-driven animation Each frame crosses bridge twice Animated.event with useNativeDriver or Reanimated
Large list rendering Many createView calls congest bridge FlatList virtualization, view recycling
Navigation transitions JS busy during transition = jank Native stack navigator, InteractionManager
Image decoding Can block UI thread if synchronous Async decoding (Fresco/SDWebImage default)
Database queries Large results congest bridge Paginate, use JSI-based DB (WatermelonDB)
Staff-Level Mental Model

The threading model is the key to understanding React Native performance. At L6, your mental model should be: the JS thread is your bottleneck, the UI thread is your lifeline, and the bridge is your tax. Every architectural decision should minimize JS thread work during interactions (use native drivers, defer with InteractionManager), keep the UI thread free (never block with synchronous native calls), and reduce bridge traffic (batch operations, use JSI where possible). The new architecture fundamentally changes this model by allowing JS and native to share memory via JSI, enabling synchronous communication without serialization — but the threading model itself (JS thread, UI thread, background threads) remains.
