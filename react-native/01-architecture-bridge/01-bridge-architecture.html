Bridge Architecture

React Native's original architecture centers on a JSON-serialized asynchronous bridge connecting JavaScript and native realms. Understanding its design, bottlenecks, and communication patterns is essential for Staff-level mobile engineers.

1. High-Level Architecture Overview

React Native's original architecture (pre-0.68) is built around a bridge that connects two separate runtimes: a JavaScript VM (JavaScriptCore or Hermes) and the native platform (iOS UIKit / Android View system). These runtimes share no memory — all communication is serialized as JSON strings passed asynchronously across the bridge.

┌─────────────────────────────────────────────────────────────────┐ │ React Native Application │ ├──────────────────┬──────────────────┬───────────────────────────┤ │ JS Thread │ Shadow Thread │ Main (UI) Thread │ │ │ │ │ │ React Component │ Yoga Layout │ Native View Hierarchy │ │ Tree + Logic │ Engine (C++) │ (UIView / android.View) │ │ │ │ │ │ Hermes / JSC │ Shadow Nodes │ Platform Renderers │ │ JS Runtime │ Layout Calc │ Event Handlers │ ├──────────────────┴──────────────────┴───────────────────────────┤ │ │ │ ┌──────────────────────────┐ │ │ │ THE BRIDGE │ │ │ │ JSON Serialization │ │ │ │ Async Message Queue │ │ │ │ BatchedBridge │ │ │ └──────────────────────────┘ │ │ │ └─────────────────────────────────────────────────────────────────┘
Architect Insight

The bridge architecture was a deliberate design choice: by keeping JS and native completely isolated, React Native achieved platform independence and crash isolation. A JS exception cannot crash the native app. However, this isolation comes at the cost of serialization overhead and the inability to make synchronous calls — the fundamental trade-off that drove the new architecture (JSI/Fabric/TurboModules).

3. Bridge Internals: MessageQueue

At its core, the bridge is implemented as a MessageQueue — a bidirectional message-passing system. The JavaScript side maintains a queue of calls destined for native, and native maintains a queue of calls destined for JavaScript.

// Simplified MessageQueue structure (from React Native source)
class MessageQueue {
  _lazyCallableModules: {[key: string]: () => Object};
  _queue: [number[], number[], any[], number];  // [moduleIDs, methodIDs, params, callID]
  _successCallbacks: Map<number, Function>;
  _failureCallbacks: Map<number, Function>;
  _callID: number;

  // Called by native to invoke JS methods
  callFunctionReturnFlushedQueue(module: string, method: string, args: any[]) {
    this.__guard(() => {
      this.__callFunction(module, method, args);
    });
    return this.flushedQueue();
  }

  // Flush pending calls to native — returns batched queue
  flushedQueue(): ?[number[], number[], any[]] {
    const queue = this._queue;
    this._queue = [[], [], [], this._callID];
    return queue[0].length ? queue : null;
  }

  // Enqueue a call to a native module
  enqueueNativeCall(moduleID: number, methodID: number, params: any[],
                    onFail: ?Function, onSucc: ?Function) {
    this._callID++;
    this._queue[0].push(moduleID);
    this._queue[1].push(methodID);
    this._queue[2].push(params);
    // Store callbacks indexed by callID
    if (onFail || onSucc) {
      this._successCallbacks.set(this._callID, onSucc);
      this._failureCallbacks.set(this._callID, onFail);
    }
  }
}
Key Detail

The queue stores calls as parallel arrays of [moduleIDs, methodIDs, params] rather than an array of objects. This is a deliberate optimization — parallel arrays serialize to smaller JSON and are faster to iterate on the native side.

4. BatchedBridge and Call Batching

BatchedBridge is the singleton MessageQueue instance that all JS-to-native communication flows through. Rather than sending each native call individually, calls are batched and flushed together.

// BatchedBridge is the global MessageQueue instance
const BatchedBridge = new MessageQueue();

// Batching strategy:
// 1. JS code enqueues multiple native calls
// 2. At end of JS execution frame, queue is flushed
// 3. Native receives batch: [[moduleIDs], [methodIDs], [params]]

// Example: A single render cycle might batch these calls:
BatchedBridge.enqueueNativeCall(UIManager, createView, [tag, 'RCTView', rootTag, props]);
BatchedBridge.enqueueNativeCall(UIManager, createView, [tag2, 'RCTText', rootTag, props2]);
BatchedBridge.enqueueNativeCall(UIManager, setChildren, [rootTag, [tag, tag2]]);
// All three sent as a single JSON payload to native
Batching Timeline ═════════════════ JS Thread: ┌─────────────────────────────────────────────┐ │ setState() → reconcile → enqueue calls │ │ call1, call2, call3, call4, call5 │ └──────────────────────┬──────────────────────┘ │ flush (end of frame) ▼ Bridge: ┌───────────────────┐ │ JSON.stringify() │ │ [[1,1,1,1,1], │ │ [2,3,4,5,6], │ │ [p1,p2,p3,p4]] │ └───────┬───────────┘ │ Native Thread: ▼ ┌─────────────────────────────────────────────┐ │ JSON.parse() → dispatch to modules │ │ UIManager.createView(...) │ │ UIManager.createView(...) │ │ UIManager.setChildren(...) │ └─────────────────────────────────────────────┘

4.1 Flush Triggers

The queue is flushed in several scenarios:

  • End of JS execution — after any native-to-JS call completes, the return value includes the flushed queue
  • Timer expiration — native polls the JS queue at a minimum interval (typically 5ms)
  • Explicit flush — certain high-priority operations trigger immediate flush
  • Native callback — when native invokes a JS callback, the response includes queued calls

5. JSON Serialization Protocol

Every value crossing the bridge must be serializable to JSON. This imposes fundamental constraints:

Supported Not Supported Workaround
Strings, numbers, booleans Functions Callback IDs (integers)
Arrays, plain objects Dates ISO string or timestamp
null undefined Converted to null
Nested structures Circular references Must be avoided
— Binary data (ArrayBuffer) Base64 encoding
— BigInt, Symbol, Map, Set Manual conversion 
// What actually crosses the bridge for a simple view update:
// JS side enqueues:
UIManager.updateView(reactTag, 'RCTView', { backgroundColor: '#ff0000' });

// Serialized as JSON:
// [[7], [3], [[42, "RCTView", {"backgroundColor": "#ff0000"}]]]
//  ↑moduleID ↑methodID  ↑params array

// For a scroll event going JS-ward:
// {"type": "topScroll", "tag": 42, "data": {
//   "contentOffset": {"x": 0, "y": 347.5},
//   "contentSize": {"width": 375, "height": 2400},
//   "layoutMeasurement": {"width": 375, "height": 667}
// }}
Performance Warning

A single scroll event serializes to ~200 bytes of JSON. At 60fps, that's 12KB/s of serialization overhead just for scroll events. When combined with other bridge traffic (layout, touch, animation), the bridge can become saturated — this is the primary cause of "bridge congestion" that manifests as dropped frames.

6. Module Registration

Native modules must register themselves so the bridge can route calls. Registration happens at app startup and creates a mapping from module names to module IDs.

// iOS: Module registration via macro
// ObjC implementation
@implementation RCTMyModule

RCT_EXPORT_MODULE()  // Registers as "MyModule" (strips RCT prefix)

RCT_EXPORT_METHOD(doSomething:(NSString *)name
                  resolver:(RCTPromiseResolveBlock)resolve
                  rejecter:(RCTPromiseRejectBlock)reject)
{
  // Runs on a dedicated GCD queue by default
  resolve(@{@"result": name});
}

@end

// Android: Module registration via ReactPackage
class MyModule(reactContext: ReactApplicationContext)
  : ReactContextBaseJavaModule(reactContext) {

  override fun getName() = "MyModule"

  @ReactMethod
  fun doSomething(name: String, promise: Promise) {
    promise.resolve(WritableNativeMap().apply {
      putString("result", name)
    })
  }
}

// Package registration
class MyPackage : ReactPackage {
  override fun createNativeModules(ctx: ReactApplicationContext) =
    listOf(MyModule(ctx))
  override fun createViewManagers(ctx: ReactApplicationContext) =
    emptyList()
}

7. NativeModules Proxy

On the JavaScript side, NativeModules is a Proxy object that lazily creates module accessors. When you access NativeModules.MyModule, it returns an object whose methods enqueue bridge calls.

// Simplified NativeModules proxy implementation
const NativeModules = {};

// Module config received from native at startup:
// [moduleName, constants, methods, promiseMethods, syncMethods]
function genModule(config, moduleID) {
  const [moduleName, constants, methods, promiseMethods, syncMethods] = config;
  const module = { ...constants };

  methods.forEach((methodName, methodID) => {
    const isPromise = promiseMethods?.includes(methodName);

    module[methodName] = (...args) => {
      if (isPromise) {
        return new Promise((resolve, reject) => {
          BatchedBridge.enqueueNativeCall(
            moduleID, methodID, args, reject, resolve
          );
        });
      }
      BatchedBridge.enqueueNativeCall(moduleID, methodID, args);
    };
  });

  NativeModules[moduleName] = module;
}

// Usage in app code:
import { NativeModules } from 'react-native';
const result = await NativeModules.MyModule.doSomething('hello');
// Internally: enqueueNativeCall(moduleID=0, methodID=0, ['hello'], reject, resolve)
Implementation Detail

Callbacks are stored in the MessageQueue indexed by callID. When native completes the call, it invokes invokeCallbackAndReturnFlushedQueue(callID, args), which looks up and executes the stored callback. This is why callbacks can only be invoked once — the entry is deleted after invocation.

8. Why the Bridge Is a Bottleneck

The bridge architecture has several fundamental limitations that become critical at scale:

8.1 Serialization Cost

Every value crossing the bridge must be JSON-serialized on one side and parsed on the other. For large data structures (e.g., long lists, complex state), this cost is non-trivial.

// Benchmark: Bridge serialization overhead
// Sending 1000 items across the bridge:

const data = Array.from({ length: 1000 }, (_, i) => ({
  id: i,
  title: `Item ${i}`,
  description: `Description for item ${i} with some additional text`,
  timestamp: Date.now(),
  metadata: { category: 'test', priority: i % 5 }
}));

// JSON.stringify(data) → ~120KB of JSON text
// Serialization time: ~2-5ms on modern devices
// Deserialization time: ~3-8ms on native side
// Total bridge overhead: ~5-13ms per transfer
// At 16ms frame budget (60fps), this alone can cause dropped frames

8.2 Async-Only Communication

The bridge is strictly asynchronous. There is no way for JS to synchronously read a native value. This creates fundamental UX problems:

// Problem: Cannot synchronously read layout
// In web: element.getBoundingClientRect() is synchronous
// In React Native (old arch): must use async callback

viewRef.current.measure((x, y, width, height, pageX, pageY) => {
  // By the time this callback fires, the layout may have changed
  // This makes tooltip positioning, drag-and-drop, and
  // gesture-based animations fundamentally harder
});

// Problem: Scroll event handling
// Each scroll event: Native → serialize → bridge → JS → process
//                    → bridge → serialize → Native → update
// Minimum 2 bridge crossings per frame = guaranteed frame drops

8.3 No Direct Memory Access

JS and native cannot share memory. Every piece of data must be copied through serialization. This is especially costly for:

  • Images — pixel data cannot be shared; must use native image references
  • Large datasets — database query results must be fully serialized
  • Binary data — must be base64 encoded (33% size increase)
  • Frequent updates — animation values, sensor data, scroll positions
Staff-Level Analysis

The bridge bottleneck is not just about raw throughput — it's about latency. Even if serialization is fast, the async nature means a minimum of one frame delay for any JS↔native round-trip. For 60fps rendering, that's 16.67ms of inherent latency. For gesture-driven animations that need JS logic, this creates visible jank because the animation is always one frame behind the user's finger. This is the core motivation behind JSI (synchronous calls) and Fabric (C++ rendering).

9. Communication Patterns

9.1 JS → Native (Commands)

// Pattern 1: Fire-and-forget
NativeModules.Analytics.track('screen_view', { screen: 'Home' });

// Pattern 2: Callback
NativeModules.Camera.takePicture(
  { quality: 0.8 },
  (error) => console.error(error),
  (uri) => setPhoto(uri)
);

// Pattern 3: Promise
const token = await NativeModules.Auth.getToken();

// Pattern 4: UIManager commands (view operations)
UIManager.dispatchViewManagerCommand(
  findNodeHandle(scrollViewRef),
  UIManager.getViewManagerConfig('RCTScrollView').Commands.scrollTo,
  [0, 100, true]  // x, y, animated
);

9.2 Native → JS (Events)

// Native sends events via RCTDeviceEventEmitter
// iOS (ObjC):
[self.bridge.eventDispatcher sendAppEventWithName:@"onLocationUpdate"
                                             body:@{@"lat": @37.7749, @"lng": @-122.4194}];

// JS side subscribes:
import { NativeEventEmitter, NativeModules } from 'react-native';

const emitter = new NativeEventEmitter(NativeModules.LocationManager);
const subscription = emitter.addListener('onLocationUpdate', (event) => {
  console.log(event.lat, event.lng);
});

// Cleanup
subscription.remove();

9.3 Communication Pattern Summary

Promise: JS ──call──→ Native ──resolve/reject──→ JS 4. UIManager command: JS ──command──→ UIManager ──→ Native View Native → JS Patterns: ═══════════════════════ 1. Events: Native ──event──→ EventEmitter ──→ JS listener 2. Direct call: Native ──callFunction──→ JS module method 3. Touch events: Native ──touch──→ JS responder system 4. Timer callbacks: Native ──timer──→ JS setTimeout/setInterval

10. Architect Considerations

10.1 Bridge Traffic Optimization

Strategy Technique Impact
Reduce crossing frequency Debounce/throttle events, batch state updates Fewer serialization cycles
Reduce payload size Send only changed fields, use compact keys Less JSON to serialize/parse
Move logic to native Native animations, native gesture handlers Bypass bridge entirely
Use native driver useNativeDriver: true for Animated Animation runs on UI thread
Avoid bridge for layout Use onLayout instead of measure() Layout info comes with render cycle

10.2 When Bridge Congestion Manifests

// Scenario: Scroll-driven header animation (BAD — bridge-heavy)
<ScrollView
  onScroll={(e) => {
    // Every scroll event crosses bridge: Native → JS
    const y = e.nativeEvent.contentOffset.y;
    // Animation update crosses bridge: JS → Native
    headerHeight.setValue(Math.max(60, 200 - y));
  }}
  scrollEventThrottle={16}
/>

// Fix: Use Animated.event with native driver
const scrollY = useRef(new Animated.Value(0)).current;
const headerHeight = scrollY.interpolate({
  inputRange: [0, 140],
  outputRange: [200, 60],
  extrapolate: 'clamp',
});

<Animated.ScrollView
  onScroll={Animated.event(
    [{ nativeEvent: { contentOffset: { y: scrollY } } }],
    { useNativeDriver: false }  // layout props can't use native driver
  )}
  scrollEventThrottle={16}
/>
Staff-Level Mental Model

The bridge is React Native's original sin and greatest strength simultaneously. It enabled the "learn once, write anywhere" promise by providing a clean abstraction boundary. But it also created a performance ceiling that no amount of optimization can break through — you cannot make JSON serialization free, and you cannot make async communication synchronous. The new architecture (JSI + Fabric + TurboModules) doesn't just optimize the bridge — it eliminates it by allowing direct C++ function calls between JS and native.
