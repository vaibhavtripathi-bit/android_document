1. NativeModules Overview

NativeModules provide the bridge between JavaScript business logic and platform-specific native APIs. Every time your React Native app accesses the camera, file system, Bluetooth, biometrics, or any other platform capability, it goes through a NativeModule.

NativeModule Call Flow (Old Architecture) ═════════════════════════════════════════ JS Thread Bridge Native Thread ───────── ────── ───────────── NativeModules.Camera JSON.stringify() CameraModule .takePicture({ ──────────────→ .takePicture() quality: 0.8 [moduleID, │ }) methodID, │ capture photo .then(uri => ...) [params]] │ ↑ ▼ │ JSON.parse() resolve(uri) └─────────────────── ←────────────── callback(uri) Promise resolved
Architect Insight

The NativeModule system was designed for extensibility over performance. Any native developer can expose platform APIs to JS with minimal boilerplate. This made React Native's ecosystem grow rapidly — but the cost is that every module call pays the bridge tax (serialization + async). TurboModules fix this by using JSI for direct C++ calls, but the registration and design patterns remain similar.

2. iOS Module Registration

On iOS, native modules are Objective-C or Swift classes that conform to the RCTBridgeModule protocol and register via macros.

// iOS NativeModule — Objective-C implementation

// Header: RCTDeviceInfo.h
#import <React/RCTBridgeModule.h>
#import <React/RCTEventEmitter.h>

@interface RCTDeviceInfo : RCTEventEmitter <RCTBridgeModule>
@end

// Implementation: RCTDeviceInfo.m
@implementation RCTDeviceInfo

// RCT_EXPORT_MODULE() registers the module with the bridge
// Internally: creates a +load method that adds the class to a registry
// The module name defaults to the ObjC class name minus "RCT" prefix
RCT_EXPORT_MODULE()  // Registers as "DeviceInfo"

// Or with explicit name:
// RCT_EXPORT_MODULE(CustomName)

// Export a method — macro generates the method descriptor
RCT_EXPORT_METHOD(getDeviceId:(RCTPromiseResolveBlock)resolve
                  rejecter:(RCTPromiseRejectBlock)reject)
{
  NSString *deviceId = [[[UIDevice currentDevice] identifierForVendor] UUIDString];
  resolve(deviceId);
}

// Export constants — evaluated once at module init
- (NSDictionary *)constantsToExport {
  return @{
    @"platform": @"ios",
    @"osVersion": [[UIDevice currentDevice] systemVersion],
    @"model": [[UIDevice currentDevice] model]
  };
}

// Specify which thread methods run on
- (dispatch_queue_t)methodQueue {
  // Default: dedicated background queue per module
  // Return dispatch_get_main_queue() for UI operations
  return dispatch_get_main_queue();
}

// For event emitters — declare supported events
- (NSArray<NSString *> *)supportedEvents {
  return @[@"onDeviceOrientationChange", @"onBatteryLevelChange"];
}

@end

2.1 RCT_EXPORT_MODULE Macro Internals

// What RCT_EXPORT_MODULE() expands to (simplified):

// 1. Creates a +moduleName class method
+ (NSString *)moduleName { return @"DeviceInfo"; }

// 2. Creates a +load method that registers the class
+ (void)load {
  RCTRegisterModule(self);
  // Adds to global array: RCTModuleClasses
}

// At bridge startup, RCTModuleClasses is iterated to:
// 1. Instantiate each module class
// 2. Extract method descriptors (from RCT_EXPORT_METHOD)
// 3. Extract constants (from constantsToExport)
// 4. Build module config: [name, constants, methods, promiseMethods]
// 5. Send config to JS as __fbBatchedBridgeConfig

3. Android Module Registration

// Android NativeModule — Kotlin implementation

class DeviceInfoModule(
  private val reactContext: ReactApplicationContext
) : ReactContextBaseJavaModule(reactContext) {

  // Module name exposed to JS
  override fun getName(): String = "DeviceInfo"

  // Constants sent to JS at module init
  override fun getConstants(): Map<String, Any> = mapOf(
    "platform" to "android",
    "osVersion" to Build.VERSION.RELEASE,
    "model" to Build.MODEL
  )

  // @ReactMethod marks methods callable from JS
  // isBlockingSynchronousMethod = false (default, async)
  @ReactMethod
  fun getDeviceId(promise: Promise) {
    try {
      val deviceId = Settings.Secure.getString(
        reactContext.contentResolver,
        Settings.Secure.ANDROID_ID
      )
      promise.resolve(deviceId)
    } catch (e: Exception) {
      promise.reject("DEVICE_ID_ERROR", e.message, e)
    }
  }

  // Synchronous method (blocks JS thread — use sparingly)
  @ReactMethod(isBlockingSynchronousMethod = true)
  fun getDeviceIdSync(): String {
    return Settings.Secure.getString(
      reactContext.contentResolver,
      Settings.Secure.ANDROID_ID
    )
  }
}

// Package registration — groups modules for the app
class DeviceInfoPackage : ReactPackage {
  override fun createNativeModules(
    reactContext: ReactApplicationContext
  ): List<NativeModule> = listOf(DeviceInfoModule(reactContext))

  override fun createViewManagers(
    reactContext: ReactApplicationContext
  ): List<ViewManager<*, *>> = emptyList()
}

// In MainApplication.kt:
override fun getPackages(): List<ReactPackage> = listOf(
  MainReactPackage(),
  DeviceInfoPackage()  // Register custom package
)

5. Callbacks and Promises

// Callback pattern — can only be invoked ONCE
// Stored in MessageQueue._successCallbacks / _failureCallbacks

// JS side:
NativeModules.FileSystem.readFile('/path/to/file', (error, content) => {
  if (error) {
    console.error('Read failed:', error);
    return;
  }
  console.log('Content:', content);
});

// Promise pattern — preferred for modern code
async function readFile() {
  try {
    const content = await NativeModules.FileSystem.readFileAsync('/path/to/file');
    console.log('Content:', content);
  } catch (error) {
    console.error('Read failed:', error.code, error.message);
  }
}

// Event pattern — for continuous data (location, sensors, etc.)
import { NativeEventEmitter, NativeModules } from 'react-native';

const locationEmitter = new NativeEventEmitter(NativeModules.LocationManager);

useEffect(() => {
  const sub = locationEmitter.addListener('onLocationUpdate', (event) => {
    setLocation({ lat: event.latitude, lng: event.longitude });
  });
  NativeModules.LocationManager.startUpdating();
  return () => {
    sub.remove();
    NativeModules.LocationManager.stopUpdating();
  };
}, []);
Common Pitfall

Callbacks in the old bridge architecture can only be invoked once. After invocation, the callback reference is deleted from the MessageQueue. If native code tries to invoke the callback a second time, it silently fails. For repeated notifications, use the event emitter pattern instead.

6. Threading Model

NativeModule Threading ══════════════════════ iOS Default: Each module gets its own serial GCD queue ┌─────────────────────────────────────────────┐ │ Module A Queue ──→ methodA1 → methodA2 → ...│ │ Module B Queue ──→ methodB1 → methodB2 → ...│ │ Main Queue ──→ UI operations │ └─────────────────────────────────────────────┘ Override with: - (dispatch_queue_t)methodQueue Android Default: All module methods run on a shared NativeModules thread ┌─────────────────────────────────────────────┐ │ NativeModulesThread ──→ all module methods │ │ (single background thread) │ └─────────────────────────────────────────────┘ Override with: @ReactMethod on specific thread 
// iOS: Controlling method execution thread

// Run all methods on main thread (for UI operations)
- (dispatch_queue_t)methodQueue {
  return dispatch_get_main_queue();
}

// Run on a custom serial queue
- (dispatch_queue_t)methodQueue {
  static dispatch_queue_t queue;
  static dispatch_once_t onceToken;
  dispatch_once(&onceToken, ^{
    queue = dispatch_queue_create("com.app.MyModule", DISPATCH_QUEUE_SERIAL);
  });
  return queue;
}

// Per-method thread control
RCT_EXPORT_METHOD(heavyComputation:(NSDictionary *)params
                  resolver:(RCTPromiseResolveBlock)resolve
                  rejecter:(RCTPromiseRejectBlock)reject)
{
  // Dispatch to background even if module queue is main
  dispatch_async(dispatch_get_global_queue(QOS_CLASS_USER_INITIATED, 0), ^{
    id result = [self performHeavyWork:params];
    resolve(result);
  });
}

7. Batch Calls and Optimization

// Multiple native calls in a single JS frame are batched

// This JS code:
NativeModules.Analytics.track('event1', { key: 'val1' });
NativeModules.Analytics.track('event2', { key: 'val2' });
NativeModules.Analytics.track('event3', { key: 'val3' });

// Produces a single bridge message:
// [[moduleID, moduleID, moduleID],
//  [methodID, methodID, methodID],
//  [['event1',{key:'val1'}], ['event2',{key:'val2'}], ['event3',{key:'val3'}]]]

// Optimization: batch on the native side too
// Instead of 3 separate native method invocations,
// design modules to accept arrays:

// BAD: N bridge calls
items.forEach(item => NativeModules.DB.insert(item));

// GOOD: 1 bridge call
NativeModules.DB.insertBatch(items);

8. Module Lazy Loading

In the old architecture, all native modules are instantiated at app startup, even if never used. This contributes to slow startup times.

// Problem: Eager initialization
// At startup, React Native:
// 1. Iterates all registered ReactPackages
// 2. Calls createNativeModules() on each
// 3. Instantiates EVERY module class
// 4. Calls getConstants() on each (synchronous!)
// 5. Serializes all module configs to JS

// For an app with 50 native modules, this can add 100-300ms to startup

// Partial mitigation: LazyReactPackage (Android)
class MyLazyPackage : LazyReactPackage() {
  override fun getNativeModules(ctx: ReactApplicationContext) = listOf(
    ModuleSpec(DeviceInfoModule::class.java, DeviceInfoModuleProvider(ctx))
  )
  // Module instantiated only when first accessed from JS
}

// TurboModules solve this completely:
// - Modules initialized on first access (true lazy loading)
// - No upfront getConstants() call
// - No module config serialization at startup

9. TurboModules: The Replacement

TurboModules are the new architecture's replacement for NativeModules. They use JSI instead of the bridge, enabling synchronous calls, type-safe interfaces, and true lazy initialization.

NativeModule vs TurboModule ════════════════════════════ NativeModule (Old): JS ──JSON.stringify──→ Bridge ──JSON.parse──→ Native JS ←──JSON.parse──── Bridge ←──JSON.stringify── Native • Async only • All modules initialized at startup • No type safety across boundary • ~5-10ms round-trip overhead TurboModule (New): JS ──JSI (C++ call)──→ Native (direct function pointer) JS ←──JSI (C++ return)── Native • Sync or async • Lazy initialization • Type-safe (codegen from specs) • ~0.01ms call overhead 
// TurboModule spec file (TypeScript)
// NativeDeviceInfo.ts
import type { TurboModule } from 'react-native';
import { TurboModuleRegistry } from 'react-native';

export interface Spec extends TurboModule {
  getConstants(): {
    platform: string;
    osVersion: string;
    model: string;
  };
  getDeviceId(): Promise<string>;
  getDeviceIdSync(): string;  // Synchronous! Not possible with old bridge
}

export default TurboModuleRegistry.getEnforcing<Spec>('DeviceInfo');

// Codegen generates:
// - C++ JSI bindings (NativeDeviceInfoSpecJSI.h/cpp)
// - ObjC++ protocol (NativeDeviceInfoSpec.h)
// - Java interface (NativeDeviceInfoSpec.java)

// Usage in app code (identical to NativeModules):
import NativeDeviceInfo from './NativeDeviceInfo';
const id = NativeDeviceInfo.getDeviceIdSync();  // Synchronous!
const asyncId = await NativeDeviceInfo.getDeviceId();

10. Architect Considerations

10.1 Module Design Principles

Principle Old Architecture New Architecture
Minimize bridge crossings Batch operations, reduce call frequency Less critical (JSI is fast), but still good practice
Thread safety Module queue isolation, dispatch_async JSI calls can be synchronous — must be thread-aware
Error handling Promise reject with code + message C++ exceptions mapped to JS errors via JSI
Type safety Runtime type checking only Compile-time via codegen from specs
Initialization Eager (all at startup) Lazy (on first access)
Staff-Level Mental Model

NativeModules are the escape hatch from JavaScript's sandbox into platform capabilities. At L6, you should understand not just how to write them, but how to architect them: batch operations to minimize bridge crossings, choose the right threading model (don't block the main thread for I/O, don't use background threads for UI mutations), design for both old and new architecture compatibility, and plan the migration path to TurboModules. The key insight is that TurboModules don't change what you can do — they change the cost of doing it, making previously impractical patterns (synchronous native calls, fine-grained native access) viable.
