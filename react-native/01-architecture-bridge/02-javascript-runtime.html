1. Engine Landscape

Unlike web browsers where the JS engine is fixed (V8 in Chrome, SpiderMonkey in Firefox), React Native can use different engines. The engine choice significantly impacts startup time, runtime performance, memory usage, and debugging capabilities.

Engine Evolution in React Native ═════════════════════════════════ 2015-2019: JavaScriptCore (JSC) — default on both platforms iOS: System JSC framework (no JIT on iOS) Android: Bundled JSC (with JIT) 2019: Hermes introduced (RN 0.60) Android-first, optional 2021: Hermes becomes Android default (RN 0.70) 2022: Hermes available on iOS (RN 0.70) 2023+: Hermes default on both platforms JSI-based, optimized for React Native Alternative: V8 via react-native-v8 (community)
Architect Insight

The engine choice is not just a performance decision — it affects your entire toolchain. Hermes uses a custom bytecode format that requires ahead-of-time compilation, changing your build pipeline. JSC on iOS uses the system framework (no JIT due to iOS restrictions), while Android JSC bundles the full engine. V8 brings Chrome DevTools protocol but adds ~7MB to APK size. Each choice has cascading implications for debugging, profiling, and CI/CD.

2. JavaScriptCore (JSC)

JavaScriptCore is the WebKit project's JavaScript engine. React Native originally used JSC on both platforms.

2.1 iOS: System JSC

On iOS, React Native uses the system-provided JavaScriptCore.framework. This has a critical implication: no JIT compilation. Apple's security model prevents third-party apps from generating executable code at runtime, so JSC runs in interpreter-only mode on iOS.

// iOS JSC execution pipeline (no JIT)
// Source Code → Parser → AST → Bytecode → LLInt (interpreter)
//                                          ↑ stops here on iOS
//
// Full JSC pipeline (macOS/Android with JIT):
// Source → Parser → AST → Bytecode → LLInt → Baseline JIT → DFG → FTL
//                                     ↑         ↑           ↑     ↑
//                                  interpret  tier 1      tier 2  tier 3

// Performance impact:
// Without JIT, hot loops are 5-10x slower than JIT-compiled code
// This is why Hermes (with AOT bytecode) is faster on iOS

2.2 Android: Bundled JSC

On Android, React Native bundles a custom build of JSC. This includes JIT compilation tiers but adds significant APK size (~3-4MB per ABI).

// Android JSC characteristics:
// - Full JIT pipeline available
// - APK size: ~3.5MB per ABI (arm64-v8a, armeabi-v7a, x86, x86_64)
// - Startup: must parse + compile JS source at runtime
// - Memory: JIT code cache adds to memory footprint
// - Debugging: WebKit Inspector protocol (Safari DevTools)

// Startup timeline with JSC:
// 1. Load JSC library (~50ms)
// 2. Create JSC context (~10ms)
// 3. Load JS bundle from disk (~30-100ms)
// 4. Parse JS source (~100-500ms for large bundles)
// 5. Execute top-level code (~50-200ms)
// Total: ~240-860ms before first render

3. Hermes Engine

Hermes is a JavaScript engine built by Meta specifically for React Native. Its key innovation is ahead-of-time (AOT) bytecode compilation — JavaScript is compiled to Hermes bytecode during the build step, not at runtime.

3.1 Architecture

Hermes Compilation Pipeline ════════════════════════════ Build Time (on developer machine / CI): ┌──────────┐ ┌──────────┐ ┌───────────┐ ┌──────────────┐ │ JS Source │───→│ Parser │───→│ IR (Herm │───→│ Hermes │ │ (.js) │ │ │ │ bytecode)│ │ Bytecode │ │ │ │ │ │ │ │ (.hbc file) │ └──────────┘ └──────────┘ └───────────┘ └──────────────┘ │ │ shipped in APK/IPA ▼ Runtime (on device): ┌──────────────┐ ┌───────────────────────────┐ │ Hermes │───→│ Hermes VM │ │ Bytecode │ │ - Memory-mapped execution │ │ (.hbc file) │ │ - No parse step needed │ │ │ │ - Interpreter only (no JIT) │ └──────────────┘ └───────────────────────────┘

3.2 Key Features

// Hermes-specific features and optimizations:

// 1. Bytecode precompilation — no runtime parsing
//    Build: hermes -emit-binary -out bundle.hbc bundle.js
//    Result: binary bytecode ready for direct execution

// 2. Memory-mapped bytecode loading
//    Instead of reading entire file into memory:
//    mmap() maps the .hbc file into virtual memory
//    Pages loaded on-demand by OS virtual memory system
//    Reduces RSS (Resident Set Size) significantly

// 3. Optimized garbage collector
//    GenGC: generational, moving garbage collector
//    Young generation: bump-pointer allocation (fast)
//    Old generation: mark-compact (reduces fragmentation)
//    Incremental collection to avoid long pauses

// 4. Lazy compilation
//    Functions compiled to bytecode but not fully optimized
//    until first call — reduces startup work

// 5. ES6+ support (subset)
//    Proxy: supported (required for NativeModules)
//    WeakRef: supported
//    Intl: partial (via hermes-engine with Intl)
//    eval(): disabled by default (security + no runtime compiler)
Critical Limitation

Hermes does not support eval() or new Function() by default because it has no runtime compiler. This breaks libraries that rely on dynamic code generation (some template engines, certain ORMs). The -allow-eval flag enables a basic interpreter for eval'd code, but at significant performance cost.

4. V8 Integration

V8 (Chrome's engine) can be used via the community react-native-v8 package. It provides the most complete ES specification support and the Chrome DevTools Protocol for debugging.

// V8 integration characteristics:
// - Full JIT compilation (TurboFan optimizing compiler)
// - Chrome DevTools Protocol (native Chrome debugging)
// - Complete ES2023+ support
// - APK size: ~7-8MB per ABI
// - Higher memory usage due to JIT code cache
// - Faster peak performance for compute-heavy workloads

// V8 execution tiers:
// 1. Ignition (interpreter) — fast startup
// 2. Sparkplug (baseline compiler) — quick compilation
// 3. Maglev (mid-tier compiler) — balanced optimization
// 4. TurboFan (optimizing compiler) — peak performance

// Trade-off: V8's JIT gives better peak perf but
// worse startup time and higher memory than Hermes

6. Bytecode Precompilation

Hermes's bytecode precompilation is its most important optimization. The compilation happens during the build step, eliminating runtime parsing entirely.

// Hermes bytecode compilation pipeline

// Step 1: Metro bundles JS into a single file
// metro bundle → index.android.bundle (plain JS)

// Step 2: Hermes compiler converts to bytecode
// hermesc -emit-binary -out index.android.bundle.hbc index.android.bundle

// Step 3: Bytecode file structure (.hbc format)
// ┌─────────────────────────────────────┐
// │ Magic Number (0x1F 0x1F)            │
// │ Bytecode Version                     │
// │ Source Hash (SHA1)                   │
// │ Global Code Flag                     │
// ├─────────────────────────────────────┤
// │ Function Header Table               │
// │   - offset, paramCount, frameSize   │
// │   - environmentSize, nameID         │
// ├─────────────────────────────────────┤
// │ String Table (deduplicated)         │
// ├─────────────────────────────────────┤
// │ RegExp Table                        │
// ├─────────────────────────────────────┤
// │ Bytecode Instructions               │
// │   (compact binary opcodes)          │
// └─────────────────────────────────────┘

// Runtime loading:
// 1. mmap() the .hbc file (no copy into heap)
// 2. Validate header (fast checksum)
// 3. Begin executing global bytecode immediately
// 4. Function bytecodes loaded on-demand via page faults

7. Engine Initialization

// Engine initialization sequence (simplified C++ pseudocode)

// 1. Create runtime
auto runtime = facebook::hermes::makeHermesRuntime(
  hermes::vm::RuntimeConfig::Builder()
    .withGCConfig(GCConfig::Builder()
      .withInitHeapSize(32 << 20)   // 32MB initial heap
      .withMaxHeapSize(512 << 20)    // 512MB max heap
      .build())
    .withEnableSampleProfiling(true)
    .build()
);

// 2. Install native bindings (global functions)
runtime->global().setProperty(
  *runtime,
  "nativeFlushQueueImmediate",
  Function::createFromHostFunction(*runtime, ...)
);

// 3. Install console, performance, timers
installGlobals(*runtime);

// 4. Load and execute bundle
auto buffer = readBundleFile("index.bundle");
runtime->evaluateJavaScript(std::move(buffer), "index.bundle");

8. Global Object Setup

React Native sets up a custom global object that differs significantly from both browser window and Node.js global.

// React Native global object (accessible as `global` or `globalThis`)
// Key properties injected by the runtime:

global.__DEV__                    // boolean: development mode flag
global.__BUNDLE_START_TIME__      // timestamp: when bundle started loading
global.nativeFlushQueueImmediate  // function: flush bridge queue
global.nativeCallSyncHook         // function: synchronous native calls (debug)
global.nativeRequire              // function: load RAM bundle segments
global.performance                // object: Performance API subset
global.ErrorUtils                 // object: global error handler

// NOT available (unlike browser):
// window, document, navigator (partially polyfilled)
// localStorage, sessionStorage
// XMLHttpRequest (polyfilled by RN)
// fetch (polyfilled by RN)
// WebSocket (polyfilled by RN)

// Hermes-specific globals:
global.HermesInternal             // Hermes runtime info
HermesInternal.hasPromise()       // true — native Promise
HermesInternal.enablePromiseRejectionTracker(...)

// Detecting the engine at runtime:
const isHermes = () => !!global.HermesInternal;
const isV8 = () => !!global._v8runtime;
const isJSC = () => !isHermes() && !isV8();

9. JSI: The Foundation for New Architecture

The JavaScript Interface (JSI) is a C++ API that provides a thin, engine-agnostic abstraction over JavaScript runtimes. It's the foundation that enables the new architecture (Fabric, TurboModules) by allowing direct, synchronous communication between JS and native code.

// JSI provides a common interface across engines:
// jsi::Runtime is the abstract base class

class Runtime {
  // Evaluate JavaScript code
  virtual Value evaluateJavaScript(
    const shared_ptr<Buffer>& buffer,
    const string& sourceURL) = 0;

  // Access global object
  virtual Object global() = 0;

  // Create JS values from C++
  // String, Object, Array, Function — all created via Runtime
};

// HostObject: expose C++ objects directly to JS
class MyHostObject : public jsi::HostObject {
  Value get(Runtime& rt, const PropNameID& name) override {
    if (name.utf8(rt) == "getValue") {
      return Function::createFromHostFunction(rt,
        PropNameID::forAscii(rt, "getValue"), 0,
        [](Runtime& rt, const Value& thisVal,
           const Value* args, size_t count) -> Value {
          return Value(42);  // Direct return, no serialization!
        });
    }
    return Value::undefined();
  }
};

// Install on JS global:
runtime.global().setProperty(runtime, "myObject",
  Object::createFromHostObject(runtime,
    make_shared<MyHostObject>()));
Architect Insight

JSI is the single most important architectural change in React Native's history. By providing a C++ abstraction over the JS runtime, it enables: (1) synchronous JS↔native calls without serialization, (2) shared memory between JS and C++ via HostObjects, (3) engine-agnostic native modules (TurboModules), and (4) C++ rendering (Fabric). The bridge becomes optional — JSI replaces it with direct function calls.

10. Polyfills and Environment Differences

React Native polyfills several Web APIs to provide a familiar development experience, but the implementations differ from browser versions in important ways.

XMLHttpRequest Native browser API Polyfill over native HTTP Bridge-based; progress events cross bridge
WebSocket Native browser API Polyfill over native WS Native socket; messages cross bridge

// Common gotchas with RN polyfills:

// 1. console.log is expensive — serializes across bridge
console.log(largeObject);  // Serializes entire object to JSON for native logger
// In production: strip console.* calls via babel plugin

// 2. setTimeout minimum resolution differs
setTimeout(fn, 0);  // Browser: ~4ms, RN: depends on bridge flush cycle (~5ms)

// 3. fetch has no CORS (no browser security sandbox)
fetch('https://api.example.com/data');  // No preflight needed

// 4. No DOM APIs at all
document.getElementById('foo');  // ❌ ReferenceError
window.addEventListener('resize', fn);  // ❌ ReferenceError

// 5. TextEncoder/TextDecoder may not be available
new TextEncoder().encode('hello');  // ❌ on older Hermes versions

// 6. Intl support varies by engine
new Intl.NumberFormat('en-US').format(1234);  // May need hermes + intl build

11. Architect Considerations

11.1 Engine Selection Decision Matrix

Priority Best Engine Rationale
Fastest startup Hermes AOT bytecode eliminates parse time
Lowest memory Hermes mmap'd bytecode, no JIT cache
Peak compute perf V8 TurboFan JIT produces fastest code
Smallest binary JSC (iOS) / Hermes System JSC is free; Hermes is ~1.5MB
Best debugging V8 / Hermes Chrome DevTools protocol
Full ES spec V8 Most complete and up-to-date
New architecture Hermes Best JSI integration, co-developed with RN
Staff-Level Mental Model

The JS engine is the foundation of your React Native app's performance profile. Hermes is the right default for most apps because it optimizes for what matters most on mobile: startup time and memory. But understanding why is what separates L6 from L5. Hermes trades peak throughput (no JIT) for predictable, fast startup (AOT bytecode + mmap). This is the correct trade-off for mobile because: (1) users perceive startup latency directly, (2) mobile memory is constrained, and (3) most RN apps are I/O-bound (network, storage), not compute-bound. If your app does heavy computation (ML inference, complex calculations), V8's JIT may be worth the startup and memory cost.
