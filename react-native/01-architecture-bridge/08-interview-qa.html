Bridge Architecture

Q1: Explain React Native's bridge architecture. What are its fundamental design trade-offs?

The bridge is an asynchronous, JSON-serialized message-passing system connecting the JavaScript VM and native platform. JS and native share no memory — all data is serialized to JSON strings, batched in a MessageQueue, and sent across the bridge. The fundamental trade-offs are: (1) Isolation vs performance — complete separation means JS crashes can't crash the native app, but serialization adds overhead; (2) Async vs latency — async communication prevents thread blocking but means minimum 1-frame latency for any JS↔native round-trip; (3) Platform independence vs direct access — the bridge abstraction enables cross-platform code but prevents direct memory sharing (e.g., no shared ArrayBuffers between JS and native).

Q2: What is BatchedBridge and how does call batching work?

BatchedBridge is the singleton MessageQueue instance through which all JS-to-native communication flows. Instead of sending each native call individually, calls are accumulated in parallel arrays [moduleIDs[], methodIDs[], params[]] and flushed together. Flushing occurs at: (1) end of JS execution frame, (2) when native invokes a JS callback (the return includes queued calls), (3) native polling at ~5ms intervals, or (4) explicit flush for high-priority operations. The parallel array format is a deliberate optimization — it serializes to smaller JSON than an array of objects and is faster to iterate on the native side.

Q3: Why is the bridge considered a bottleneck? Give specific scenarios where it causes performance issues.

Serialization cost: A scroll event serializes to ~200 bytes; at 60fps that's 12KB/s just for scroll events. Async-only: Gesture-driven animations need JS logic but the bridge adds 1-frame latency, making animations always behind the user's finger. No shared memory: Binary data (images, audio) must be base64-encoded (33% size increase). Bridge congestion: During complex renders, hundreds of UIManager calls (createView, updateView, manageChildren) compete with event traffic, causing frame drops. Measurement latency: Unlike web's synchronous getBoundingClientRect(), React Native's measure() requires an async bridge round-trip, making tooltip positioning and drag-and-drop fundamentally harder.

Q4: How does the NativeModules proxy work on the JavaScript side?

At startup, native sends a module configuration to JS: [moduleName, constants, methods, promiseMethods, syncMethods]. JS creates proxy objects where each method is a wrapper that calls BatchedBridge.enqueueNativeCall(moduleID, methodID, params, onFail, onSucc). For Promise methods, the wrapper returns a new Promise whose resolve/reject callbacks are stored in the MessageQueue indexed by callID. When native completes the call, it invokes invokeCallbackAndReturnFlushedQueue(callID, args), which looks up and executes the stored callback. Callbacks can only be invoked once — the entry is deleted after invocation.

Q5: What data types can cross the bridge? What are the limitations?

Only JSON-serializable types: strings, numbers, booleans, null, arrays, and plain objects. Not supported: functions (replaced by callback IDs), undefined (converted to null), Dates (must use ISO strings or timestamps), circular references, binary data/ArrayBuffer (must base64 encode), BigInt, Symbol, Map, Set. These limitations mean you cannot pass closures to native, cannot share mutable references, and binary data transfer has 33% overhead from base64 encoding.

Q6: Describe the three realms in React Native and their responsibilities.

JavaScript realm (JS thread): runs React reconciliation, business logic, state management, event handler callbacks in Hermes/JSC. Shadow realm (background thread): maintains the shadow tree (C++ objects), runs Yoga layout calculations, diffs shadow trees to produce minimal mutations. Native realm (UI/main thread): creates and mutates platform views (UIView/android.View), handles touch events, runs native animations, manages system events. All communication between realms crosses the bridge with JSON serialization.

Q7: How would you optimize an app experiencing bridge congestion during list scrolling?

(1) Use useNativeDriver: true for scroll-driven animations to keep them on the UI thread. (2) Reduce scrollEventThrottle or use Animated.event with native driver instead of onScroll callbacks. (3) Virtualize with FlatList to limit the number of views in the hierarchy. (4) Use getItemLayout to avoid async measurement. (5) Move complex scroll-driven logic to react-native-reanimated worklets (UI thread). (6) Batch state updates to reduce reconciliation frequency. (7) Consider migrating to the new architecture (Fabric) which eliminates bridge serialization for rendering.

JavaScript Runtime

Q8: Compare Hermes, JavaScriptCore, and V8 for React Native. When would you choose each?

Hermes: Best default — AOT bytecode eliminates parse time (~350ms startup vs ~800ms JSC), lowest memory (mmap'd bytecode, ~35MB RSS), smallest binary (~1.5MB/ABI). No JIT, so lower peak compute performance. JSC: iOS system framework (free binary size), but no JIT on iOS (Apple restriction). Android JSC has JIT but ~3.5MB/ABI and slow startup (must parse at runtime). V8: Highest peak performance (TurboFan JIT), most complete ES spec, Chrome DevTools native, but ~7MB/ABI and highest memory (~65MB RSS). Choose Hermes for most apps (startup + memory), V8 for compute-heavy apps (ML, complex calculations), JSC only for legacy compatibility.

Q9: How does Hermes bytecode precompilation work? Why does it improve startup time?

During the build step, the Hermes compiler (hermesc) converts JavaScript source to Hermes bytecode (.hbc format). The .hbc file contains a binary header, function header table, deduplicated string table, and compact bytecode instructions. At runtime, the .hbc file is memory-mapped (mmap()) — the OS maps it into virtual memory without copying to heap. Pages are loaded on-demand via page faults. This eliminates the two most expensive startup steps: (1) parsing JS source into AST (~100-500ms for large bundles), and (2) compiling AST to bytecode. The result is ~50-60% faster startup compared to JSC.

Q10: Why doesn't Hermes support eval() by default? What are the implications?

Hermes has no runtime compiler — all code must be compiled to bytecode at build time. eval() and new Function() require runtime compilation, which Hermes cannot do. The -allow-eval flag enables a basic interpreter for eval'd code, but at significant performance cost. Implications: libraries relying on dynamic code generation break (some template engines, certain ORMs, code-push solutions that eval JS). Workarounds include pre-compiling dynamic code or using alternative libraries that don't require eval.

Q11: What is JSI and why is it the foundation of the new architecture?

JSI (JavaScript Interface) is a C++ API providing an engine-agnostic abstraction over JavaScript runtimes (jsi::Runtime). It enables: (1) synchronous JS↔native calls without serialization, (2) shared memory via HostObjects (C++ objects directly accessible from JS), (3) engine independence (same native code works with Hermes, JSC, or V8). JSI is the foundation because TurboModules use it for direct native calls (replacing the bridge), Fabric uses it for C++ rendering (replacing UIManager), and it enables the bridgeless mode where the legacy bridge is completely eliminated.

Q12: How does the React Native global object differ from browser's window?

RN's global lacks: window, document, navigator (partially polyfilled), localStorage, DOM APIs. RN polyfills: fetch (over native HTTP, no CORS), XMLHttpRequest (bridge-based), WebSocket (native socket), setTimeout (native timer module), console (bridges to native logger — expensive for objects). RN-specific globals: __DEV__, __BUNDLE_START_TIME__, nativeFlushQueueImmediate, ErrorUtils. Hermes adds HermesInternal. Key gotcha: console.log serializes across the bridge, making it expensive in production — strip with babel plugin.

Native Modules

Q13: Walk through the complete lifecycle of a NativeModule call from JS to native and back.

(1) JS calls NativeModules.Camera.takePicture({quality: 0.8}). (2) The proxy wrapper creates a Promise and stores resolve/reject callbacks in MessageQueue indexed by callID. (3) enqueueNativeCall(moduleID, methodID, [{quality: 0.8}], reject, resolve) adds to the batch queue. (4) At frame end, queue is flushed — JSON.stringify produces [[moduleID], [methodID], [[{quality: 0.8}]]]. (5) Native receives, JSON.parse, dispatches to Camera module's method queue. (6) Camera module executes on its designated thread (iOS: custom GCD queue, Android: NativeModules thread). (7) Native calls resolve(uri) which triggers invokeCallbackAndReturnFlushedQueue(callID, [uri]). (8) JS MessageQueue looks up callID, executes the stored resolve callback. (9) Promise resolves with the URI.

Q14: How does module registration differ between iOS and Android?

iOS: Uses ObjC macros — RCT_EXPORT_MODULE() creates a +load method that registers the class in a global array at app launch. RCT_EXPORT_METHOD() generates method descriptors. Module name defaults to class name minus "RCT" prefix. Android: Uses ReactPackage interface — createNativeModules() returns module instances, getName() provides the JS-visible name, @ReactMethod annotation marks callable methods. Packages are registered in MainApplication.getPackages(). Key difference: iOS registration is automatic via +load, Android requires explicit package registration.

Q15: Why can bridge callbacks only be invoked once? How do you handle repeated notifications?

Callbacks are stored in MessageQueue maps indexed by callID. After invocation, the entry is deleted to prevent memory leaks. Invoking a second time silently fails because the callID no longer exists. For repeated notifications (location updates, sensor data, push notifications), use the NativeEventEmitter pattern: native sends events via RCTDeviceEventEmitter, JS subscribes with emitter.addListener(eventName, callback). Events don't use the callback storage mechanism — they're dispatched as regular bridge messages to the JS event system.

Q16: What threading model do NativeModules use? How do you control which thread a method runs on?

iOS: Each module gets its own serial GCD queue by default. Override -methodQueue to return dispatch_get_main_queue() for UI operations or a custom queue. Individual methods can dispatch to other queues internally. Android: All module methods run on a shared NativeModules thread (single background thread). Use @ReactMethod(isBlockingSynchronousMethod = true) for synchronous calls (blocks JS thread). For UI operations, use runOnUiThread. Key principle: never block the main thread with I/O, never do UI mutations off the main thread.

Q17: How do TurboModules improve upon NativeModules?

TurboModules use JSI instead of the bridge: (1) Synchronous calls — direct C++ function invocation, ~0.01ms vs ~5-10ms bridge overhead. (2) No serialization — values passed as C++ types, not JSON. (3) Lazy initialization — modules instantiated on first access, not all at startup. (4) Type safety — codegen from TypeScript/Flow specs generates native interfaces. (5) No upfront constants — getConstants() not called at startup. Migration: write a spec file (TypeScript interface extending TurboModule), run codegen, implement the generated native interface.

Shadow Tree & Yoga

Q18: Explain the three-tree model: React tree → Shadow tree → Native tree.

The React element tree (JS) is an immutable description created every render — plain objects with type, props, children. The shadow tree (C++) is a mutable representation on the shadow thread — each shadow node owns a Yoga node for layout calculation. The native view tree (platform) is the actual UIView/android.View hierarchy on the UI thread. Flow: React reconciliation diffs element trees → generates UIManager commands → shadow thread creates/updates shadow nodes → Yoga calculates layout → layout diffs dispatched to UI thread → native views created/mutated. The shadow tree exists because: layout must be off the main thread, Yoga is C++, and diffing produces minimal native mutations.

Q19: How does Yoga's layout algorithm work? Describe the two-pass process.

Pass 1 — Measure (bottom-up): Starting from leaf nodes, Yoga determines intrinsic sizes. Text nodes call platform text measurement functions (NSAttributedString on iOS, StaticLayout on Android). Nodes with explicit dimensions skip measurement. Pass 2 — Layout (top-down): Starting from root, Yoga applies the Flexbox algorithm — distributes available space among children based on flex properties, computes final positions (x, y, width, height) for each node. Complexity is O(n) for most layouts, O(n²) worst case with measure functions and flexWrap. Text measurement is the bottleneck because it requires platform text engines.

Q20: What are the key differences between Yoga (React Native) and CSS Flexbox?

(1) flexDirection default: CSS=row, Yoga=column. (2) flexShrink default: CSS=1, Yoga=0 (items overflow instead of shrink). (3) flex shorthand: CSS=grow/shrink/basis, Yoga=single number (flexGrow only). (4) Units: CSS supports px/em/rem/vw/vh/calc(), Yoga only dp (unitless) and %. (5) position: CSS has static/relative/absolute/fixed/sticky, Yoga has relative/absolute/static. (6) display: CSS has block/inline/flex/grid/none, Yoga has flex/none only. (7) No CSS Grid support. (8) overflow affects clipping only, not layout.

Q21: What is view flattening and why is it important for performance?

View flattening removes "layout-only" views from the native hierarchy — views that exist solely for Flexbox grouping with no visual properties (no background, border, shadow, opacity, transform, event handlers, or accessibility props). The shadow tree retains these nodes for layout calculation, but the native tree skips them, re-parenting children with adjusted offsets. Impact: fewer native views means less memory, faster native layout passes, and reduced overdraw. On Android, collapsable={true} (default) enables this. A typical screen can see 30-50% native view reduction. Disable with collapsable={false} when you need to measure or reference the view.

Rendering Pipeline

Q22: Trace the complete rendering pipeline from setState to pixels on screen.

(1) setState enqueued on JS thread. (2) React reconciliation walks fiber tree, diffs old vs new elements. (3) Generates UIManager commands: createView, updateView, manageChildren. (4) Commands batched on BatchedBridge, JSON-serialized, sent to native. (5) Shadow thread receives, creates/updates shadow nodes and Yoga nodes. (6) Yoga calculates layout for dirty subtrees. (7) Shadow tree diff produces minimal native mutations. (8) Mutations dispatched to UI thread. (9) UI thread creates/updates native views, sets frames from Yoga results. (10) Platform compositing renders to screen. Total latency: 1-3 frames (16-50ms) from setState to pixels.

Q23: What is UIManager and what are its key methods?

UIManager is the central native module managing the shadow tree and view operations. Key methods: createView(tag, viewName, rootTag, props) — creates shadow node + native view; updateView(tag, viewName, changedProps) — updates properties; manageChildren(parentTag, moveFrom, moveTo, addTags, addAt, removeAt) — manages child hierarchy; setChildren(parentTag, childTags) — simpler initial mount; measure(tag, callback) — async layout measurement; dispatchViewManagerCommand(tag, commandID, args) — sends commands to view managers. UIManager is the most heavily used native module in any RN app.

Q24: Explain setNativeProps. When should you use it and what are the risks?

setNativeProps bypasses React reconciliation to update native view properties directly via the bridge. Use for high-frequency updates where React re-renders are too expensive (gesture handlers, manual animations). Risks: (1) Creates divergence between React's virtual tree and actual native state — next React re-render may overwrite changes. (2) No batching with React updates. (3) Deprecated in Fabric (new architecture) — use react-native-reanimated instead for UI-thread animations. It's an escape hatch, not a pattern — prefer Animated API with native driver or Reanimated for production code.

Threading Model

Q25: Describe the three main threads in React Native and what happens when each is blocked.

JS thread: Runs React, business logic, event handlers. Blocking delays all JS responses (touch handlers, state updates, JS-driven animations) but native animations and scroll physics continue. UI/main thread: Runs native view operations, touch recognition, native animations. Blocking freezes everything — no touch response, no scroll, no animations. Android shows ANR after 5s, iOS watchdog kills after ~10s. Shadow thread: Runs Yoga layout calculations. Blocking delays layout updates but doesn't freeze UI or JS. Additional threads: NativeModules thread (Android), timer thread, network threads, image decoding threads.

Q26: How do native-driven animations work? Why are they smoother than JS-driven animations?

With useNativeDriver: true, the animation configuration (start value, end value, duration, easing) is sent to native once. Native creates a CADisplayLink (iOS) or Choreographer callback (Android) that updates the animated property every frame on the UI thread. JS thread is completely uninvolved after start(). JS-driven animations calculate each frame's value on the JS thread, serialize it across the bridge, then apply on the UI thread — minimum 1-frame latency and vulnerable to JS thread blocking. Limitation: native driver only supports non-layout props (opacity, transform) because layout props require Yoga recalculation on the shadow thread.

Q27: What is InteractionManager and how does it improve perceived performance?

InteractionManager defers expensive work until after animations/interactions complete. Animations register as "interactions" when started; InteractionManager maintains a count. When count reaches zero, queued runAfterInteractions callbacks execute. Common pattern: on screen mount, show a skeleton loader, then InteractionManager.runAfterInteractions(() => loadData()) — this ensures the navigation animation completes smoothly before data fetching begins. Callbacks still run on JS thread, so they can still block if expensive. For truly non-blocking work, combine with requestIdleCallback or chunk work across frames.

Q28: How does requestAnimationFrame differ between browser and React Native?

Browser: callback tied to display vsync, runs before paint, on the main thread, guaranteed in sync with display. React Native: triggered by native vsync signal (Choreographer/CADisplayLink) that crosses the bridge to JS thread. Inherent latency: native vsync → bridge → JS callback. If JS thread is busy, callback is delayed. Any view updates from the callback still need to cross the bridge back to native — total latency of 1-2 frames from vsync to pixels. For smooth animations, don't use rAF + setState; use Animated API with native driver or Reanimated worklets.

Q29: Explain the timer execution order in React Native: setTimeout vs setImmediate vs Promise.then vs queueMicrotask.

Execution order: (1) Synchronous code, (2) Microtasks (Promise.then, queueMicrotask) — run before any macrotask, (3) setImmediate — next JS event loop tick, (4) setTimeout(fn, 0) — has bridge latency (~5ms minimum because the timer fires on native and callback crosses bridge to JS). Key difference from browser: setImmediate runs before setTimeout(fn, 0) in RN. setTimeout precision is limited by bridge latency (~5ms minimum vs browser's ~4ms).

Metro Bundler

Q30: Describe Metro's three-phase architecture: resolver, transformer, serializer.

Resolver: Finds actual files for import statements using Node.js resolution algorithm with platform-specific extensions (.ios.js, .android.js, .native.js). Resolution priority: platform-specific > .native > generic. Transformer: Compiles each module independently via Babel — JSX→JS, TypeScript→JS, Flow→JS — then wraps in __d(factory, moduleID, deps, path) module definition. Serializer: Traverses dependency graph from entry point, orders modules, concatenates factories, prepends polyfills/runtime, appends entry point __r(entryID), generates source maps. Each phase caches independently, enabling fast incremental builds.

Q31: How does delta bundling enable Fast Refresh?

Client connects to Metro dev server via WebSocket. Initial request gets the full bundle (base revision). When a file changes, Metro's file watcher detects it, re-transforms only that file, and sends a delta message: {added: [], modified: [{id, code}], deleted: []}. Client replaces the module factory in its registry. Fast Refresh then: (1) re-executes the updated module, (2) React re-renders affected components while preserving hooks state, (3) only the edited component and its subtree re-render. Limitations: class components lose state (full remount), non-component modules trigger full reload, module-level side effects may cause issues.

Q32: What are RAM bundles? How do they improve startup time?

RAM (Random Access Modules) bundles allow loading individual modules on demand instead of parsing the entire bundle at startup. Indexed RAM bundle (iOS): binary header with module offsets, modules loaded via offset lookup + partial file read. File-based RAM bundle (Android): each module as a separate file in assets, loaded via nativeRequire(moduleID). Startup improvement: instead of parsing ~2MB of JS (300-800ms), only parse the startup-critical modules (~50-150ms). Remaining modules loaded lazily on first require(). Combined with inline requires, can achieve 50-70% faster TTI.

Q33: Why doesn't Metro support tree shaking? What are the implications?

Metro uses CommonJS-style module wrapping (__d() factories with require()) which is dynamic and cannot be statically analyzed for dead code. Tree shaking requires ES module static analysis (import/export declarations). Implications: import { View } from 'react-native' includes the entire react-native module. Workarounds: use deep imports where available, babel plugins for import rewriting (babel-plugin-module-resolver), keep dependencies lean, audit bundle with react-native-bundle-visualizer. The Hermes compiler does some dead code elimination at the bytecode level, partially compensating.

Q34: How does Hermes bytecode compilation integrate with Metro?

Hermes compilation is the last step — after Metro serialization. Metro produces a plain JS bundle, then the build system (Gradle on Android, CocoaPods on iOS) runs hermesc to compile it to .hbc bytecode. Metro doesn't need to know about Hermes. Source maps are chained: original source → Metro source map → Hermes source map, composed into a single map for debugging. The .hbc file replaces the .bundle file in the app package (APK assets/ or iOS app bundle).

Advanced & Cross-cutting

Q35: How would you architect a React Native app for optimal startup time?

(1) Use Hermes (AOT bytecode, ~50% faster startup). (2) Enable RAM bundles + inline requires (lazy module loading). (3) Minimize native module count (each adds startup cost). (4) Use InteractionManager to defer non-critical work. (5) Lazy-load screens with React.lazy + Suspense. (6) Audit and reduce bundle size (remove unused deps, use lighter alternatives). (7) Optimize images (use appropriate formats, lazy load). (8) Profile with Systrace/Perfetto to identify bottlenecks. (9) Consider native splash screen to mask JS initialization. (10) Migrate to new architecture (TurboModules with lazy init).

Q36: Compare the old architecture (Bridge) with the new architecture (JSI/Fabric/TurboModules) across all dimensions.

Communication: Bridge uses JSON serialization (async, ~5-10ms overhead) vs JSI uses direct C++ calls (sync/async, ~0.01ms). Rendering: Old UIManager sends commands over bridge vs Fabric uses C++ shadow tree shared across threads. Modules: NativeModules are eager-loaded with bridge calls vs TurboModules are lazy-loaded with JSI calls. Layout: Yoga results cross bridge to UI thread vs Fabric computes layout in shared C++ and mounts synchronously. Threading: Strict thread separation via bridge vs shared memory via JSI enables flexible threading. Type safety: Runtime only vs codegen from specs. Concurrent rendering: Not possible with async bridge vs enabled by Fabric's synchronous rendering.

Q37: You're seeing 200ms frame drops during a complex list scroll. Walk through your debugging process.

(1) Profile with Systrace/Perfetto: identify which thread is bottlenecked (JS, UI, shadow). (2) If JS thread: check for expensive reconciliation (React Profiler), unnecessary re-renders (why-did-you-render), heavy event handlers. Fix: React.memo, useMemo, move logic to native. (3) If bridge: check for high-frequency bridge traffic (scroll events, animation values). Fix: useNativeDriver, Reanimated, reduce scrollEventThrottle. (4) If UI thread: check for expensive native views (shadows, complex borders), too many views. Fix: view flattening, simplify styles, use FlatList with getItemLayout. (5) If shadow thread: check for expensive layout (deep nesting, many text nodes). Fix: flatten hierarchy, virtualize, cache text measurements.

Q38: How does React Native handle images differently from the web?

Web: <img> loads via HTTP, browser handles caching/decoding. RN: Image component sends URI to native, which uses platform image libraries (SDWebImage/Fresco) for downloading, caching, and decoding on background threads. Key differences: (1) No CSS background-image — use ImageBackground component. (2) Image dimensions must be specified (no intrinsic sizing from network images). (3) Binary pixel data never crosses the bridge — native handles everything via image references. (4) Prefetching: Image.prefetch(uri) downloads to native cache. (5) Local images: require('./image.png') resolved by Metro, bundled as assets with @1x/@2x/@3x variants.

Q39: Explain the concept of "bridge congestion" and how it manifests in user experience.

Bridge congestion occurs when the volume of serialized messages exceeds the bridge's throughput capacity. It manifests as: (1) Dropped frames during scroll (scroll events + render updates compete for bridge bandwidth). (2) Delayed touch response (touch events queued behind render traffic). (3) Animation jank (JS-driven animation values arrive late). (4) Stale UI (native state diverges from JS state due to queued updates). Root causes: too many bridge crossings per frame (>50), large payloads (>10KB per crossing), or JS thread blocking (delays queue flushing). Mitigation: native driver, batch operations, reduce bridge traffic, move to new architecture.

Q40: How would you design a real-time collaborative editing feature in React Native?

Architecture: (1) WebSocket connection for real-time sync (RN's polyfilled WebSocket goes through native). (2) CRDT or OT algorithm running on JS thread for conflict resolution. (3) Native text input with controlled component pattern. Challenge: text input events cross bridge (native→JS), edits from server also need to update native text (JS→native) — two bridge crossings per keystroke. Optimization: (1) Use JSI-based WebSocket (TurboModule) for lower latency. (2) Debounce outgoing changes. (3) Use setNativeProps or Reanimated for cursor position updates. (4) Consider native module for the CRDT engine to avoid JS thread blocking. (5) In new architecture, Fabric's synchronous rendering eliminates the bridge latency for text updates.

Q41: What happens when a JavaScript exception occurs in React Native? How does error propagation work?

JS exceptions are caught by ErrorUtils.setGlobalHandler. In development: the red error screen (LogBox) displays the error with stack trace. In production: the global handler is invoked — default behavior logs to console and may crash the JS context. Key point: JS exceptions do NOT crash the native app (bridge isolation). The native app continues running but the JS context may be in an inconsistent state. Error boundaries (React) catch render errors and display fallback UI. For native module errors: Promise rejections propagate to JS as rejected promises; uncaught rejections trigger the global handler. Native crashes (ObjC exceptions, Java exceptions) crash the entire app regardless of JS state.

Q42: How does React Native handle accessibility? What are the threading implications?

Accessibility props (accessible, accessibilityLabel, accessibilityRole) are set on native views via the bridge, same as any other prop. Platform accessibility services (VoiceOver on iOS, TalkBack on Android) interact directly with native views on the UI thread — no bridge involvement. Threading implication: accessibility announcements (AccessibilityInfo.announceForAccessibility) cross the bridge (JS→native), so there's latency. Focus management (setAccessibilityFocus) also crosses the bridge. In Fabric, accessibility props are set synchronously via C++, reducing latency for dynamic accessibility updates.

Q43: Explain how React Native's Animated API works under the hood for both JS-driven and native-driven animations.

JS-driven: Animated.Value stores current value in JS. On each frame (requestAnimationFrame), the animation driver calculates the next value, calls setValue(), which triggers listeners that call setNativeProps or setState to update views via the bridge. Cost: 1 bridge crossing per animated prop per frame. Native-driven: Animation config (start, end, duration, easing) serialized and sent to native once via NativeAnimatedModule.startAnimatingNode. Native creates a parallel animation graph with NativeAnimatedNodes. CADisplayLink/Choreographer drives updates on UI thread. Animated.Value on JS side is NOT updated during animation — only on completion. Interpolation, addition, multiplication all happen in native animation graph.

Q44: What is the performance impact of console.log in React Native production builds?

console.log serializes all arguments to JSON, sends them across the bridge to the native logger (NSLog on iOS, Log.d on Android). For complex objects, this can take several milliseconds per call. In a hot loop or frequent event handler, console.log can cause significant bridge congestion and JS thread blocking. Best practice: use babel-plugin-transform-remove-console to strip all console.* calls in production builds. Alternative: use a logging library that batches and throttles log output, or conditionally log only in __DEV__ mode.

Q45: How does StyleSheet.create optimize performance compared to inline style objects?

StyleSheet.create registers style objects and returns numeric IDs. In the old architecture, these IDs reference pre-serialized style data that doesn't need to be re-serialized on each render. Inline style objects (style={{flex: 1}}) create a new object every render, requiring serialization every time the component renders. Additional benefits: (1) validation in development mode, (2) potential future optimization (style deduplication, native-side caching). However, the performance difference is modest in practice — the bigger win is avoiding unnecessary re-renders caused by new object references breaking React.memo/PureComponent shallow comparison.

Q46: How would you implement a high-performance drag-and-drop interface in React Native?

(1) Use react-native-gesture-handler for gesture recognition on the UI thread (no bridge for gesture detection). (2) Use react-native-reanimated for drag animations — worklets run on UI thread via JSI, updating transform props at 60fps without bridge crossings. (3) Use shared values for position tracking. (4) For drop zone detection, pre-calculate zone boundaries and check in worklets (avoid bridge round-trip for measurement). (5) Only cross to JS thread for state updates on drop (not during drag). (6) Use LayoutAnimation or Reanimated for reorder animations. Architecture: gesture recognition + visual feedback on UI thread, state persistence on JS thread — minimize bridge crossings to drop events only.

Q47: Explain the difference between Animated.event with useNativeDriver and react-native-reanimated worklets.

Animated.event with native driver maps native events (scroll, gesture) directly to animated values on the UI thread — no JS involvement. But it's limited to predefined mappings (e.g., scroll offset → animated value) with predefined interpolations. Reanimated worklets are JavaScript functions that run on the UI thread via JSI. They can contain arbitrary logic (conditionals, loops, math) and animate any prop including layout props. Worklets are compiled to a restricted JS subset that Reanimated's UI-thread runtime can execute. Key advantage: worklets enable complex gesture-driven animations with custom logic that would be impossible with Animated.event's declarative mapping.

Q48: What is the impact of deep view nesting on React Native performance?

Deep nesting impacts: (1) Yoga layout: more nodes to traverse, though still O(n). (2) Shadow tree: more shadow nodes to create, diff, and manage. (3) Native views: more views in hierarchy (unless flattened), increasing memory and overdraw. (4) Bridge traffic: more createView/manageChildren calls per render. (5) Touch handling: deeper hit-testing traversal. (6) Accessibility: deeper tree for screen readers to navigate. Mitigation: flatten hierarchy with style composition, use view flattening (collapsable), extract deeply nested structures into memoized components, use FlatList for long lists.

Q49: How does React Native handle keyboard interactions? What are the threading implications?

Keyboard events originate on the UI thread (system keyboard). Keyboard.addListener('keyboardDidShow', handler) sends events across bridge to JS. KeyboardAvoidingView adjusts layout in response — this requires: native event → bridge → JS handler → setState → reconcile → bridge → shadow → layout → bridge → native view update. Multiple bridge crossings cause visible delay between keyboard appearing and layout adjusting. Optimization: use android:windowSoftInputMode="adjustResize" on Android (native handling, no bridge), or react-native-keyboard-controller which uses Reanimated to animate layout changes on the UI thread.

Q50: Describe how you would profile and optimize a React Native app's memory usage.

(1) JS heap: Use Hermes sampling profiler or Chrome DevTools Memory tab. Look for: retained closures, large arrays, event listener leaks, unmounted component state. (2) Native memory: Use Xcode Instruments (Allocations) or Android Studio Profiler (Memory). Look for: image cache size (Fresco/SDWebImage), native view count, leaked native modules. (3) Bridge memory: Large serialized payloads create temporary allocations on both sides. (4) Optimization: virtualize lists (FlatList), resize images before display, clear caches on memory warning (AppState listener), use removeClippedSubviews for off-screen views, audit native module lifecycle. (5) Hermes-specific: mmap'd bytecode reduces RSS, GenGC's incremental collection reduces pause times.

Q51: How does React Native's FlatList achieve virtualization? What are its performance trade-offs?

FlatList renders only visible items plus a configurable buffer (windowSize). As the user scrolls, items entering the window are mounted and items leaving are unmounted (or rendered as blank spacers). Internally uses VirtualizedList which tracks scroll position, calculates visible range, and renders items via CellRendererComponent. Trade-offs: (1) Mounting/unmounting has cost — mitigate with maxToRenderPerBatch and updateCellsBatchingPeriod. (2) Blank areas during fast scroll — increase windowSize or use initialNumToRender. (3) getItemLayout avoids async measurement but requires known heights. (4) keyExtractor is critical for correct recycling.

Q52: What security considerations are unique to React Native's architecture?

(1) JS bundle: shipped in the app package, can be extracted and reverse-engineered. Hermes bytecode is harder to read than plain JS but not encrypted. (2) Bridge traffic: JSON messages could be intercepted on rooted/jailbroken devices. (3) NativeModules: expose platform APIs to JS — a compromised JS context could call any registered module. (4) Code Push: OTA updates bypass app store review — must validate bundle integrity. (5) eval(): disabled in Hermes by default (good for security). (6) Deep links: URL handling crosses bridge to JS — validate/sanitize inputs. Mitigations: obfuscate JS (metro-minify-terser), use certificate pinning, validate all bridge inputs, encrypt sensitive data before bridge crossing.

Q53: How would you implement offline-first architecture in React Native?

(1) Local database: Use JSI-based DB (WatermelonDB, op-sqlite) for fast queries without bridge overhead. (2) Sync engine: background sync via native module (survives app backgrounding). (3) Optimistic updates: update local DB immediately, queue server sync. (4) Conflict resolution: last-write-wins or CRDT depending on data model. (5) Network detection: NetInfo module (native event → bridge → JS). (6) Queue management: persist pending operations in local DB, replay on reconnect. (7) Image caching: use native image cache (Fresco/SDWebImage) with disk persistence. Architecture consideration: heavy DB operations should use native modules or JSI to avoid bridge congestion during sync.

Q54: Explain the role of Choreographer (Android) and CADisplayLink (iOS) in React Native's rendering.

Choreographer (Android): coordinates animation, input, and drawing with the display vsync signal. React Native registers a frame callback that: (1) processes pending native-to-JS calls, (2) flushes the bridge queue, (3) dispatches pending UI operations. All view mutations are batched within a single Choreographer frame for atomic updates. CADisplayLink (iOS): timer synchronized with display refresh rate. React Native uses it for: (1) native-driven animations (update animated values each frame), (2) triggering JS requestAnimationFrame callbacks via bridge. Both ensure rendering work aligns with display refresh, preventing tearing and optimizing battery usage.

Q55: How does the new architecture change the mental model for React Native performance optimization?

Old model: "minimize bridge crossings" — every optimization reduces serialization and async latency. New model: "leverage synchronous rendering and shared memory." Specific changes: (1) JSI eliminates serialization cost — fine-grained native calls become cheap. (2) Fabric's C++ shadow tree enables synchronous layout — no bridge for rendering. (3) TurboModules' lazy init reduces startup cost. (4) Concurrent rendering enables priority-based updates — navigation can interrupt heavy renders. (5) Shared memory via HostObjects enables zero-copy data sharing. The optimization focus shifts from "reduce bridge traffic" to "optimize React rendering" (memoization, concurrent features) and "leverage native capabilities" (Reanimated worklets, native modules for compute).
