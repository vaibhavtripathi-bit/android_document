Metro Bundler

Metro is React Native's JavaScript bundler, purpose-built for mobile development. Understanding its architecture — resolver, transformer, serializer — and optimizations like delta bundling, RAM bundles, and Hermes bytecode integration is essential for optimizing app startup and developer experience.

1. Metro Architecture Overview

Metro Bundler Pipeline ══════════════════════ Source Files (.js/.ts/.tsx) │ ▼ ┌─────────────────┐ │ RESOLVER │ Module resolution: find files for import statements │ │ - Node.js resolution algorithm │ │ - Platform-specific extensions (.ios.js, .android.js) │ │ - Custom resolvers (assets, native modules) └────────┬────────┘ │ ▼ ┌─────────────────┐ │ TRANSFORMER │ Code transformation: compile each module │ │ - Babel/SWC transformation │ │ - JSX → JS, TypeScript → JS, Flow → JS │ │ - Module wrapping (__d() factory) │ │ - Asset extraction (images, fonts) └────────┬────────┘ │ ▼ ┌─────────────────┐ │ SERIALIZER │ Bundle assembly: combine modules into output │ │ - Dependency graph traversal │ │ - Module ordering │ │ - Source map generation │ │ - Bundle format (plain, RAM, HBC) └────────┬────────┘ │ ▼ Bundle Output (.bundle / .hbc)
Architect Insight

Metro's three-phase architecture (resolve → transform → serialize) is deliberately simpler than webpack's plugin-heavy approach. This simplicity enables Metro's key advantage: incremental builds. Each phase caches its output independently, so changing one file only re-transforms that file and re-serializes the affected bundle segment. This is what makes Fast Refresh sub-second even for large codebases.

2. Resolver

// Metro's resolver finds the actual file for each import

// Resolution order for: import Button from './Button'
// 1. Exact match: ./Button (no extension)
// 2. Platform-specific: ./Button.ios.js or ./Button.android.js
// 3. Extension resolution: ./Button.js, ./Button.ts, ./Button.tsx
// 4. Directory index: ./Button/index.js, ./Button/index.ts
// 5. Package.json main: ./Button/package.json → "main" field

// Platform-specific resolution (key RN feature):
// Given files:
//   Button.js          ← shared implementation
//   Button.ios.js      ← iOS-specific
//   Button.android.js  ← Android-specific
//   Button.native.js   ← native (iOS + Android)
//   Button.web.js      ← web (react-native-web)

// When bundling for iOS:
//   import Button from './Button'
//   → resolves to ./Button.ios.js (platform-specific wins)

// Resolution priority: .ios.js > .native.js > .js

// metro.config.js resolver configuration:
module.exports = {
  resolver: {
    sourceExts: ['js', 'jsx', 'ts', 'tsx', 'json'],
    assetExts: ['png', 'jpg', 'gif', 'svg', 'ttf', 'otf'],
    platforms: ['ios', 'android'],
    nodeModulesPaths: ['./node_modules'],
    resolverMainFields: ['react-native', 'browser', 'main'],
  },
};

3. Transformer

// Metro's transformer compiles each module independently

// Input: raw source file
// Output: transformed JS + dependency list + source map

// Default transformation pipeline:
// 1. Parse source (Babel parser / hermes-parser)
// 2. Apply Babel transforms:
//    - @babel/plugin-transform-flow-strip-types
//    - @babel/plugin-transform-typescript (if .ts/.tsx)
//    - @babel/plugin-transform-react-jsx
//    - metro-react-native-babel-preset (RN-specific transforms)
// 3. Wrap in module factory:

// Original source:
import React from 'react';
import { View } from 'react-native';
export default () => <View />;

// After transformation:
__d(function(global, _$$_REQUIRE, _$$_IMPORT_DEFAULT, _$$_IMPORT_ALL,
             module, exports, _dependencyMap) {
  var React = _$$_REQUIRE(_dependencyMap[0], "react");
  var _reactNative = _$$_REQUIRE(_dependencyMap[1], "react-native");
  exports.default = function() {
    return React.createElement(_reactNative.View);
  };
}, 42, [0, 1], "src/App.js");
//  ↑moduleID  ↑depIDs  ↑module path (debug)

// __d() is Metro's module definition function
// It registers the factory but does NOT execute it
// Execution happens when require(moduleID) is first called

4. Serializer

// The serializer assembles transformed modules into a bundle

// Serialization steps:
// 1. Build dependency graph (traverse from entry point)
// 2. Assign module IDs (deterministic for caching)
// 3. Order modules (entry point dependencies first)
// 4. Concatenate module factories
// 5. Prepend polyfills and runtime
// 6. Append entry point require()
// 7. Generate source map

// Bundle structure:
// ┌──────────────────────────────────────┐
// │ Polyfills (Promise, fetch, console)  │
// │ Metro runtime (__d, __r functions)   │
// ├──────────────────────────────────────┤
// │ __d(factory, 0, [...], "react")      │ ← module 0
// │ __d(factory, 1, [...], "react-native")│ ← module 1
// │ __d(factory, 2, [...], "src/App.js") │ ← module 2
// │ ... (hundreds of modules)            │
// ├──────────────────────────────────────┤
// │ __r(2);  // require entry point      │
// └──────────────────────────────────────┘

6. Delta Bundling and Fast Refresh

// Delta bundling: only send changed modules to the device

// Traditional bundling (without delta):
// Edit 1 file → rebuild entire bundle → send ~2MB → reload app
// Time: 5-30 seconds

// Delta bundling:
// Edit 1 file → transform only that file → send ~2KB delta → hot update
// Time: <1 second

// Delta bundle protocol:
// 1. Client connects to Metro dev server via WebSocket
// 2. Initial request: full bundle (base)
// 3. File change detected by Metro file watcher
// 4. Metro re-transforms only changed file
// 5. Delta message sent: { added: [], modified: [{id, code}], deleted: [] }
// 6. Client applies delta: replace module factory in registry

// Fast Refresh (built on delta bundling):
// 1. Delta bundle updates module code
// 2. React re-renders affected components
// 3. State is preserved (hooks state survives)
// 4. Only the edited component and its subtree re-render

// Fast Refresh limitations:
// - Class components: full remount (state lost)
// - Non-component modules: full reload
// - Syntax errors: error overlay, retry on fix
// - Module-level side effects: may cause issues
Key Detail

Delta bundling works by maintaining a revision graph on the server. Each client connection tracks which revision it has. When a file changes, Metro computes the minimal diff between the client's revision and the new revision. This is why Metro keeps running — it maintains the full dependency graph and transformation cache in memory.

7. RAM Bundles

RAM (Random Access Modules) bundles allow loading individual modules on demand instead of parsing the entire bundle at startup. This significantly reduces Time to Interactive (TTI).

Plain Bundle vs RAM Bundle ══════════════════════════ Plain Bundle (startup): ┌──────────────────────────────────┐ │ Parse ALL modules at startup │ │ Module 0 │ Module 1 │ ... │ 500 │ ← parse 2MB of JS │ ████████████████████████████████ │ │ Time: 300-800ms │ └──────────────────────────────────┘ Indexed RAM Bundle (startup): ┌──────────────────────────────────┐ │ Binary Index (module offsets) │ ← read index: <1ms ├──────────────────────────────────┤ │ Module 0 │ Module 1 │ ... │ 500 │ │ ████ │ │ │ │ ← parse only needed modules │ Time: 50-150ms (initial) │ │ + lazy load as needed │ └──────────────────────────────────┘ 
// Indexed RAM bundle format (binary header):
// Bytes 0-3:   Magic number (0xFB0BD1E5)
// Bytes 4-7:   Number of modules
// Bytes 8-11:  Startup section length
// Then: Module table (8 bytes per module):
//   Bytes 0-3: offset from start of file
//   Bytes 4-7: length of module code
// Then: Module code sections

// Loading a module from RAM bundle:
// 1. Look up module ID in binary index → get offset + length
// 2. Read bytes at [offset, offset+length] from file
// 3. Parse and execute only that module's code
// 4. Cache the module exports

// global.nativeRequire(moduleID) triggers native-side loading:
// Native reads module bytes from RAM bundle → injects into JS VM

// Enable RAM bundles in metro.config.js:
module.exports = {
  serializer: {
    createModuleIdFactory: () => {
      let id = 0;
      return (path) => id++;
    },
  },
  transformer: {
    getTransformOptions: async () => ({
      transform: {
        experimentalImportSupport: false,
        inlineRequires: true,  // Required for RAM bundles
      },
    }),
  },
};

8. Inline Requires

// Inline requires: defer module loading until first use

// Without inline requires (eager):
import HeavyChart from './HeavyChart';  // loaded at module init
import RareFeature from './RareFeature';  // loaded even if never used

export default function Screen() {
  if (showChart) return <HeavyChart />;
  return <View />;
}

// With inline requires (lazy):
// Babel transform moves require() to point of use

export default function Screen() {
  if (showChart) {
    const HeavyChart = require('./HeavyChart').default;  // loaded only when needed
    return <HeavyChart />;
  }
  return <View />;
}

// Enable in metro.config.js:
// transform: { inlineRequires: true }

// Or per-file with directive:
// @inline-requires

// Impact: reduces startup time by deferring unused module parsing
// Trade-off: first access to a lazy module has a small delay

9. Hermes Bytecode Integration

// Metro + Hermes bytecode compilation pipeline

// Standard Metro pipeline:
// Source → Resolve → Transform (Babel) → Serialize → .bundle (JS text)

// With Hermes:
// Source → Resolve → Transform (Babel) → Serialize → .bundle (JS text)
//                                                          │
//                                                          ▼
//                                                    hermesc compiler
//                                                          │
//                                                          ▼
//                                                    .hbc (bytecode)

// Hermes compilation is the LAST step — after Metro serialization
// This means Metro doesn't need to know about Hermes
// The build system (Gradle/CocoaPods) runs hermesc on Metro's output

// Android (build.gradle):
// hermesEnabled = true
// → Metro produces index.android.bundle (JS text)
// → hermesc compiles to index.android.bundle (HBC bytecode)
// → HBC file packaged in APK assets/

// iOS (Podfile):
// :hermes_enabled => true
// → Metro produces main.jsbundle (JS text)
// → hermesc compiles to main.jsbundle (HBC bytecode)
// → HBC file included in app bundle

Tree Shaking Limitation

Metro does not perform tree shaking (dead code elimination). If you import { View } from 'react-native', the entire react-native module is included in the bundle. This is a known limitation. Workarounds include: using deep imports where available (react-native/Libraries/...), babel plugins for import rewriting, and keeping dependencies lean.

Staff-Level Mental Model

Metro is optimized for developer experience (fast incremental builds, Fast Refresh) at the cost of production optimization (no tree shaking, limited code splitting). At L6, you should understand this trade-off and know how to compensate: use Hermes bytecode for runtime performance, RAM bundles for startup time, inline requires for lazy loading, and manual dependency management for bundle size. The key architectural insight is that Metro's simplicity is intentional — it does one thing well (bundle React Native apps fast) rather than trying to be a general-purpose bundler. For production optimization, the Hermes compiler is your primary tool, not Metro itself.
