Rendering Optimization

Rendering is where React's virtual tree becomes native views. Every unnecessary re-render, every extra wrapper view, every unstable key costs real milliseconds. This doc covers FlatList tuning, memoization strategies, view flattening, FlashList, and the techniques that separate 60fps apps from janky ones.

1. React Native Render Pipeline

Render Pipeline: JS → Native Views ═══════════════════════════════════ setState() │ ▼ React Reconciliation (JS Thread) ├── Diff virtual tree (old vs new) ├── Determine minimal changes └── Produce update instructions │ ▼ Shadow Tree Update (Shadow Thread) ├── Apply changes to shadow tree ├── Run Yoga layout (flexbox) └── Compute absolute positions │ ▼ Native View Update (UI Thread) ├── Create new views (if needed) ├── Update view properties ├── Apply layout frames └── Trigger platform rendering Cost at each stage: ┌──────────────────────────────────────────────┐ │ Reconciliation: O(n) where n = changed nodes │ │ Layout: O(n) Yoga computation │ │ View updates: O(n) native view mutations │ │ │ │ Total cost = sum of all three stages │ │ Must complete within 16.67ms for 60fps │ └──────────────────────────────────────────────┘

2. shouldComponentUpdate / React.memo

// React.memo — shallow comparison of props
const UserCard = React.memo(({ user, onPress }) => {
  return (
    <TouchableOpacity onPress={() => onPress(user.id)}>
      <Image source={{ uri: user.avatar }} style={styles.avatar} />
      <Text>{user.name}</Text>
      <Text>{user.email}</Text>
    </TouchableOpacity>
  );
});

// Custom comparison function for deep equality
const UserCard = React.memo(({ user, onPress }) => {
  // ... render
}, (prevProps, nextProps) => {
  // Return true if props are equal (skip re-render)
  return (
    prevProps.user.id === nextProps.user.id &&
    prevProps.user.name === nextProps.user.name &&
    prevProps.user.avatar === nextProps.user.avatar
    // Intentionally skip onPress — it's stable via useCallback
  );
});

// Class component equivalent
class UserCard extends React.PureComponent {
  // PureComponent does shallow prop comparison automatically
  // Override shouldComponentUpdate for custom logic:
  shouldComponentUpdate(nextProps) {
    return this.props.user.id !== nextProps.user.id;
  }
}
Memo Pitfall: Inline Objects and Functions

React.memo uses shallow comparison. If you pass inline objects (style={{ flex: 1 }}) or inline functions (onPress={() => {}}), they create new references every render, defeating memoization entirely. Always extract styles to StyleSheet.create() and functions to useCallback() when passing to memoized children.

3. FlatList Optimization

// Fully optimized FlatList configuration
const ITEM_HEIGHT = 80;

function OptimizedFeedList({ items }) {
  const getItemLayout = useCallback((data, index) => ({
    length: ITEM_HEIGHT,
    offset: ITEM_HEIGHT * index,
    index,
  }), []);

  const keyExtractor = useCallback(
    (item) => item.id, []
  );

  const renderItem = useCallback(
    ({ item }) => <MemoizedFeedItem item={item} />, []
  );

  return (
    <FlatList
      data={items}
      renderItem={renderItem}
      keyExtractor={keyExtractor}
      getItemLayout={getItemLayout}

      // Windowing configuration
      windowSize={5}              // render 5 screens worth
      maxToRenderPerBatch={10}    // items per batch render
      updateCellsBatchingPeriod={50} // ms between batch renders
      initialNumToRender={10}     // items on first render

      // Memory optimization
      removeClippedSubviews={Platform.OS === 'android'}

      // Scroll performance
      scrollEventThrottle={16}    // 60fps scroll events
      onEndReachedThreshold={0.5} // trigger load at 50%

      // Avoid re-renders on scroll
      maintainVisibleContentPosition={{
        minIndexForVisible: 0,
      }}
    />
  );
}

4. FlashList Adoption

// FlashList — drop-in FlatList replacement by Shopify
// Uses recycling (like RecyclerView) instead of unmount/remount
import { FlashList } from '@shopify/flash-list';

function RecyclingList({ items }) {
  return (
    <FlashList
      data={items}
      renderItem={({ item }) => <FeedItem item={item} />}
      estimatedItemSize={80}     // required — approximate height
      keyExtractor={(item) => item.id}

      // FlashList-specific optimizations
      drawDistance={250}          // pixels ahead to pre-render
      overrideItemLayout={(layout, item) => {
        layout.size = item.type === 'header' ? 120 : 80;
      }}
    />
  );
}

// FlashList vs FlatList performance comparison
// 10,000 items, iPhone 12:
//                    FlatList    FlashList
// Initial render:    450ms       120ms
// Scroll FPS:        45-55fps    58-60fps
// Memory (scroll):   180MB       95MB
// Blank areas:       Frequent    Rare
FlashList Architecture

FlashList uses cell recycling — the same pattern as iOS UICollectionView and Android RecyclerView. Instead of unmounting off-screen items and mounting new ones (FlatList's approach), FlashList reuses the same native view hierarchy and updates props. This eliminates the cost of native view creation/destruction during scrolling. The trade-off: your renderItem must handle being called with different data for the same view instance.

// Views that CAN be flattened (layout-only):
// - Views with only: flex, padding, margin, alignItems, etc.
// - No: backgroundColor, borderRadius, opacity, shadow, transform

// BEFORE: 5 native views
<View style={{ flex: 1 }}>
  <View style={{ padding: 16 }}>
    <View style={{ flexDirection: 'row' }}>
      <Text>Hello</Text>
      <Text>World</Text>
    </View>
  </View>
</View>

// AFTER optimization: 3 native views (2 Views flattened)
// React Native's view flattening automatically merges
// layout-only views into their parent

// Force disable flattening (when you need the view for measurement)
<View collapsable={false} style={{ flex: 1 }}>
  <Text>This view won't be flattened</Text>
</View>

// Manually reduce view count
// BEFORE: unnecessary wrapper
<View style={styles.card}>
  <View style={styles.cardContent}>
    <Text>{title}</Text>
  </View>
</View>

// AFTER: merged styles
<View style={[styles.card, styles.cardContent]}>
  <Text>{title}</Text>
</View>

6. Key Stability

// WRONG: Unstable keys cause full remount on every render
items.map((item, index) => (
  <ItemRow key={index} item={item} />  // index changes on reorder!
));

items.map((item) => (
  <ItemRow key={Math.random()} item={item} />  // always remounts!
));

// CORRECT: Stable, unique keys
items.map((item) => (
  <ItemRow key={item.id} item={item} />  // stable across renders
));

// Impact of unstable keys:
// 1. Component unmounts and remounts (loses state)
// 2. Native view destroyed and recreated (expensive)
// 3. All child components re-render
// 4. Animations reset
// 5. Focus lost on inputs

// For lists without natural IDs, generate stable keys
const itemsWithKeys = useMemo(() =>
  rawItems.map((item, i) => ({
    ...item,
    _stableKey: item.id || `${item.type}-${item.title}-${i}`,
  })),
  [rawItems]
);

7. List Item Memoization

// Memoized list item — prevents re-render when list scrolls
const FeedItem = React.memo(({ item, onPress, onLike }) => {
  return (
    <View style={styles.feedItem}>
      <Image source={{ uri: item.image }} style={styles.image} />
      <View style={styles.content}>
        <Text style={styles.title}>{item.title}</Text>
        <Text style={styles.body}>{item.body}</Text>
      </View>
      <View style={styles.actions}>
        <TouchableOpacity onPress={() => onLike(item.id)}>
          <Text>{item.likes} likes</Text>
        </TouchableOpacity>
      </View>
    </View>
  );
});

// Parent must provide stable callbacks
function FeedScreen() {
  const [items, setItems] = useState([]);

  const handlePress = useCallback((id) => {
    navigation.navigate('Detail', { id });
  }, [navigation]);

  const handleLike = useCallback((id) => {
    setItems(prev => prev.map(item =>
      item.id === id
        ? { ...item, likes: item.likes + 1 }
        : item
    ));
  }, []);

  const renderItem = useCallback(({ item }) => (
    <FeedItem item={item} onPress={handlePress} onLike={handleLike} />
  ), [handlePress, handleLike]);

  return <FlatList data={items} renderItem={renderItem} />;
}

8. Avoiding Unnecessary Wrappers

// BEFORE: Excessive nesting (common in component libraries)
function Card({ children }) {
  return (
    <View style={styles.cardOuter}>
      <View style={styles.cardInner}>
        <View style={styles.cardContent}>
          {children}
        </View>
      </View>
    </View>
  );
}

// AFTER: Minimal views
function Card({ children, style }) {
  return (
    <View style={[styles.card, style]}>
      {children}
    </View>
  );
}

// Use Fragment to avoid wrapper views
function UserInfo({ user }) {
  return (
    <>
      <Text>{user.name}</Text>
      <Text>{user.email}</Text>
    </>
  );
}

Staff-Level Mental Model

Rendering optimization in React Native is a three-layer problem: (1) React layer — minimize reconciliation work via memoization, stable keys, and state architecture, (2) Layout layer — reduce Yoga computation via fewer views and simpler hierarchies, (3) Native layer — minimize view mutations via view flattening and recycling (FlashList). The most impactful optimization is usually at the React layer (preventing unnecessary re-renders), because it eliminates work at all downstream layers. FlashList is the single highest-ROI change for list-heavy apps.
