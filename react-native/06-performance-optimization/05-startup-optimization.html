Startup Optimization

App startup time is the first impression users have of your app. React Native adds overhead compared to pure native — JS bundle loading, bridge initialization, and module registration all contribute. This doc covers every technique from Hermes bytecode to lazy loading to reduce cold start time to under 2 seconds.

2. Hermes Bytecode Precompilation

// Hermes compiles JS to bytecode at BUILD time
// This eliminates parse + compile at runtime

// Without Hermes (JSC):
// 1. Load .jsbundle from disk (~50ms)
// 2. Parse JavaScript source (~100-200ms)
// 3. Compile to machine code (~100-300ms)
// Total: 250-550ms

// With Hermes:
// 1. mmap .hbc file from disk (~5ms)
// 2. Verify bytecode header (~1ms)
// 3. Execute bytecode directly
// Total: ~6ms

// Enable Hermes in react-native.config.js or app.json
// Android: android/app/build.gradle
// project.ext.react = [
//   enableHermes: true
// ]

// iOS: ios/Podfile
// :hermes_enabled => true

// Verify Hermes bytecode is being used
if (global.HermesInternal) {
  console.log('Hermes bytecode engine active');
  console.log(`Version: ${global.HermesInternal.getRuntimeProperties()['OSS Release Version']}`);
}
Hermes mmap Optimization

Hermes bytecode files (.hbc) are memory-mapped (mmap), not loaded into heap memory. The OS loads pages on demand as code is executed. This means: (1) only executed code pages are loaded into RAM, (2) the OS can evict pages under memory pressure and reload them transparently, (3) startup doesn't wait for the entire bundle to load. This is the single biggest startup optimization in React Native's history — it reduced bundle load time by 90%+.

3. RAM Bundles and Inline Requires

// RAM Bundles split the bundle into modules loaded on demand
// Each module is a separate entry that can be loaded independently

// metro.config.js — enable RAM bundles
module.exports = {
  serializer: {
    getModulesRunBeforeMainModule() {
      return [
        require.resolve('./src/polyfills'),
      ];
    },
    createModuleIdFactory() {
      return (path) => {
        // Stable module IDs for consistent RAM bundle layout
      };
    },
  },
};

// Inline requires — defer module loading until first use
// BEFORE: All modules loaded at startup
import HeavyChart from './HeavyChart';
import PDFViewer from './PDFViewer';
import VideoPlayer from './VideoPlayer';

// AFTER: Modules loaded when first accessed
function ChartScreen() {
  const HeavyChart = require('./HeavyChart').default;
  return <HeavyChart />;
}

// Enable inline requires globally in metro.config.js
module.exports = {
  transformer: {
    getTransformOptions: async () => ({
      transform: {
        experimentalImportSupport: false,
        inlineRequires: true,
      },
    }),
  },
};

4. Lazy Module Initialization

// React.lazy + Suspense for screen-level code splitting
const SettingsScreen = React.lazy(() => import('./screens/Settings'));
const ProfileScreen = React.lazy(() => import('./screens/Profile'));
const AnalyticsScreen = React.lazy(() => import('./screens/Analytics'));

function App() {
  return (
    <NavigationContainer>
      <Stack.Navigator>
        <Stack.Screen name="Home" component={HomeScreen} />
        <Stack.Screen name="Settings">
          {() => (
            <Suspense fallback={<LoadingScreen />}>
              <SettingsScreen />
            </Suspense>
          )}
        </Stack.Screen>
      </Stack.Navigator>
    </NavigationContainer>
  );
}

// Lazy initialization pattern for heavy singletons
let _analyticsInstance = null;
function getAnalytics() {
  if (!_analyticsInstance) {
    const Analytics = require('./analytics/heavy-sdk').default;
    _analyticsInstance = new Analytics({ apiKey: CONFIG.ANALYTICS_KEY });
  }
  return _analyticsInstance;
}

5. Native Module Lazy Loading

Native Module Loading: Legacy vs TurboModules ══════════════════════════════════════════════ Legacy (all loaded at startup): App Start → Load Module A → Load Module B → ... → Load Module N Each module: ~1-5ms init time 50 modules × 3ms = 150ms added to startup TurboModules (lazy loaded): App Start → Install proxy → Ready! Module A accessed → Load Module A (3ms) Module B never accessed → Never loaded (0ms) Startup savings = (total modules - used modules) × avg init time If 50 modules exist but only 10 used at startup: Savings = 40 × 3ms = 120ms

6. Splash Screen Strategies

// react-native-splash-screen or expo-splash-screen
import * as SplashScreen from 'expo-splash-screen';

// Keep splash visible while loading
SplashScreen.preventAutoHideAsync();

function App() {
  const [isReady, setIsReady] = useState(false);

  useEffect(() => {
    async function prepare() {
      try {
        // Pre-load critical data
        await Promise.all([
          loadFonts(),
          loadUserSession(),
          prefetchCriticalData(),
        ]);
      } finally {
        setIsReady(true);
      }
    }
    prepare();
  }, []);

  const onLayoutRootView = useCallback(async () => {
    if (isReady) {
      // Hide splash after first render is committed
      await SplashScreen.hideAsync();
    }
  }, [isReady]);

  if (!isReady) return null;

  return (
    <View onLayout={onLayoutRootView} style={{ flex: 1 }}>
      <AppNavigator />
    </View>
  );
}

7. Measuring Startup Time

// Measuring startup phases
const startupMetrics = {
  nativeStart: null,      // Set in native code (AppDelegate/Application)
  jsStart: Date.now(),    // Set at top of index.js
  appMounted: null,       // Set in App component useEffect
  firstRender: null,      // Set in onLayout of root view
  interactive: null,      // Set when data loaded + UI ready
};

// In index.js (earliest JS execution point)
global.__jsStartTime = Date.now();

// In App.tsx
function App() {
  useEffect(() => {
    const mountTime = Date.now();
    const jsToMount = mountTime - global.__jsStartTime;
    console.log(`JS → Mount: ${jsToMount}ms`);

    // Report to analytics
    analytics.track('startup_complete', {
      jsToMount,
      totalColdStart: mountTime - global.__nativeStartTime,
    });
  }, []);
}

8. Reducing Bundle Size

// Analyze bundle contents
// npx react-native-bundle-visualizer
// or: npx source-map-explorer main.jsbundle main.jsbundle.map

// Common bundle bloat sources:
// 1. moment.js (330KB) → use date-fns (tree-shakeable) or dayjs (2KB)
// 2. lodash (72KB) → use lodash-es or individual imports
// 3. Unused exports from large libraries
// 4. Polyfills for features Hermes already supports
// 5. Development-only code not stripped in production

// Import optimization
// BEFORE: imports entire library
import _ from 'lodash';
_.debounce(fn, 300);

// AFTER: imports only what's needed
import debounce from 'lodash/debounce';
debounce(fn, 300);

9. Tree Shaking and Dead Code

Metro's Tree Shaking Limitations

Metro bundler has limited tree shaking compared to webpack or Rollup. It cannot eliminate unused exports from CommonJS modules (only ES modules). Many React Native libraries use CommonJS, so unused code is included in the bundle. Workarounds: (1) use libraries that ship ES modules, (2) import specific files instead of barrel exports, (3) use babel-plugin-transform-remove-console to strip console.log in production.

// babel.config.js — production optimizations
module.exports = {
  presets: ['module:@react-native/babel-preset'],
  plugins: [
    // Remove console.log in production
    ...(process.env.NODE_ENV === 'production'
      ? ['transform-remove-console']
      : []),
  ],
  env: {
    production: {
      plugins: [
        // Dead code elimination for __DEV__ checks
        ['transform-inline-environment-variables'],
      ],
    },
  },
};
Staff-Level Mental Model

Startup optimization is a pipeline problem — you must optimize every stage because the total time is the sum of all stages. The highest-impact optimizations in order: (1) Hermes bytecode — eliminates parse/compile time (saves 200-500ms), (2) Inline requires — defers module loading (saves 50-200ms), (3) TurboModule lazy loading — defers native module init (saves 50-150ms), (4) Bundle size reduction — less code to load and execute (saves 50-200ms), (5) Splash screen management — perceived performance while loading. Measure each stage independently and optimize the largest contributor first.
