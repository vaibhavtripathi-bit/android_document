Memory Optimization

Memory management in React Native spans two worlds — the JavaScript heap (managed by Hermes/JSC GC) and native memory (images, views, caches). Memory leaks, excessive allocation, and poor image management are the top causes of OOM crashes and degraded performance on mobile.

4. Memory Leak Patterns

// Leak 1: Event listeners not cleaned up
function LeakyComponent() {
  useEffect(() => {
    const handler = (data) => processData(data);
    DeviceEventEmitter.addListener('dataUpdate', handler);
    // MISSING: return () => subscription.remove();
  }, []);
}

// Leak 2: Timers not cleared
function LeakyTimer() {
  useEffect(() => {
    const interval = setInterval(() => {
      fetchNewData(); // keeps running after unmount
    }, 5000);
    // MISSING: return () => clearInterval(interval);
  }, []);
}

// Leak 3: Closures capturing stale references
function LeakyClosure() {
  const [data, setData] = useState(largeDataset);

  useEffect(() => {
    const ws = new WebSocket('wss://...');
    ws.onmessage = (msg) => {
      // This closure captures 'data' — prevents GC of largeDataset
      // even after component unmounts if ws isn't closed
      setData([...data, msg.data]);
    };
    return () => ws.close();
  }, [data]); // Re-creates ws on every data change!
}

// Leak 4: Navigation stack retaining screens
// React Navigation keeps unmounted screens in memory
// for back navigation. Deep stacks accumulate memory.
// Fix: Use unmountOnBlur: true for heavy screens
<Stack.Screen
  name="HeavyScreen"
  component={HeavyScreen}
  options={{ unmountOnBlur: true }}
/>

// Leak 5: Animated values not cleaned up
function LeakyAnimation() {
  const anim = useRef(new Animated.Value(0)).current;

  useEffect(() => {
    Animated.loop(
      Animated.timing(anim, { toValue: 1, duration: 1000, useNativeDriver: true })
    ).start();
    // MISSING: return () => anim.stopAnimation();
  }, []);
}
The Navigation Stack Leak

React Navigation's stack navigator keeps all screens in memory by default (for instant back navigation). A user navigating through 20 screens accumulates 20 screen components, their state, images, and native views. For memory-heavy screens, use unmountOnBlur: true or switch to a tab/drawer navigator that naturally limits the screen count.

// Captures: JS objects, closures, strings, arrays
// Open in Chrome DevTools → Memory tab
// Compare snapshots to find growing objects

// Tool 2: Xcode Memory Graph Debugger
// Debug → Debug Memory Graph
// Shows: retain cycles, leaked objects, ownership graph

// Tool 3: Android Studio Memory Profiler
// View → Tool Windows → Profiler → Memory
// Shows: heap allocations, GC events, native memory

// Tool 4: Flipper Leak Canary plugin (Android)
// Automatically detects Activity/Fragment leaks

// Programmatic memory monitoring
function useMemoryMonitor(intervalMs = 10000) {
  useEffect(() => {
    const timer = setInterval(() => {
      if (global.HermesInternal) {
        const stats = global.HermesInternal.getInstrumentedStats();
        const heapMB = stats.js_heapSize / 1024 / 1024;

        if (heapMB > 200) {
          console.warn(`High memory: ${heapMB.toFixed(0)}MB`);
          analytics.track('high_memory_warning', { heapMB });
        }
      }
    }, intervalMs);

    return () => clearInterval(timer);
  }, [intervalMs]);
}

7. Large List Memory

// FlatList memory optimization
<FlatList
  data={items}
  renderItem={renderItem}

  // Key memory props:
  windowSize={5}             // Only render 5 screens of content
  removeClippedSubviews={true} // Detach off-screen views (Android)
  maxToRenderPerBatch={5}    // Fewer items per batch

  // For very long lists (1000+ items):
  initialNumToRender={10}    // Don't render all at once
  getItemLayout={getItemLayout} // Skip measurement
/>

// Pagination — don't load everything into memory
function usePaginatedList(fetchPage) {
  const [items, setItems] = useState([]);
  const [page, setPage] = useState(0);

  const loadMore = useCallback(async () => {
    const newItems = await fetchPage(page);
    setItems(prev => [...prev, ...newItems]);
    setPage(p => p + 1);
  }, [page, fetchPage]);

  // Memory-conscious: trim old items if list gets too long
  useEffect(() => {
    if (items.length > 500) {
      setItems(prev => prev.slice(-300));
    }
  }, [items.length]);

  return { items, loadMore };
}

Staff-Level Mental Model

Memory optimization in React Native requires thinking in two domains simultaneously: JS heap (managed by GC, leaks via closures/listeners/timers) and native memory (images, views, caches — managed manually). The most impactful optimization is almost always image memory — a single unoptimized image can consume more memory than your entire React component tree. At Staff level, implement: (1) a memory monitoring system that tracks both JS and native memory in production, (2) automatic image resizing based on display size, (3) aggressive cache eviction under memory pressure, and (4) navigation stack management to prevent unbounded screen accumulation.
