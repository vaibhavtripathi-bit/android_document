Architect Insight

The key mental model: React Native has three frame-critical threads. If ANY of them takes >16.67ms per frame, you drop frames. The JS thread runs your React code. The shadow thread computes layout. The UI thread renders pixels. In the legacy architecture, these communicate through the bridge (async, serialized). In the New Architecture, they communicate through JSI/Fabric (synchronous, zero-copy). Understanding which thread is the bottleneck determines your optimization strategy.

2. 60fps Target and Frame Budget

Frame Budget at 60fps ═════════════════════ Total budget per frame: 16.67ms (1000ms / 60fps) ┌──────────────────────────────────────────────────────┐ │ 16.67ms budget │ │ │ │ JS Thread: ████████░░░░░░░░ ~8ms available │ │ Bridge/JSI: ██░░░░░░░░░░░░░░ ~2ms overhead │ │ Layout: ███░░░░░░░░░░░░░ ~3ms for Yoga │ │ UI Thread: ███░░░░░░░░░░░░░ ~3ms for rendering │ │ Buffer: ░░░░░░░░░░░░░░░░ ~0.67ms │ └──────────────────────────────────────────────────────┘ Frame drop scenarios: ───────────────────── JS takes 20ms → 1 frame dropped (visible jank) JS takes 35ms → 2 frames dropped (noticeable stutter) JS takes 100ms → 6 frames dropped (interaction feels broken) JS takes 500ms → 30 frames dropped (app appears frozen) ProMotion displays (120fps): budget = 8.33ms per frame 
// Measuring frame drops programmatically
import { PerformanceObserver } from 'react-native';

// Using the JS frame callback to detect drops
let lastFrameTime = performance.now();
let droppedFrames = 0;

function frameCallback() {
  const now = performance.now();
  const delta = now - lastFrameTime;

  if (delta > 18) { // 16.67ms + 1ms tolerance
    const dropped = Math.floor(delta / 16.67) - 1;
    droppedFrames += dropped;
    console.warn(`Dropped ${dropped} frames (${delta.toFixed(1)}ms)`);
  }

  lastFrameTime = now;
  requestAnimationFrame(frameCallback);
}

requestAnimationFrame(frameCallback);

3. Bridge Overhead

// Measuring bridge overhead
const start = performance.now();
await NativeModules.MyModule.doWork('input');
const end = performance.now();
console.log(`Bridge round-trip: ${(end - start).toFixed(2)}ms`);

// Typical bridge overhead breakdown:
// JS serialization:     0.5-2ms (depends on payload size)
// Queue dispatch:       0-5ms (batching delay)
// Native deserialization: 0.5-2ms
// Method execution:     varies
// Result serialization: 0.5-2ms
// Return to JS:         0-5ms (batching delay)
// Total overhead:       2-16ms (excluding method execution)

// Bridge traffic patterns that cause jank:
// 1. Scroll events → JS handler → bridge → native update (per frame)
// 2. Animation values → bridge → native view update (per frame)
// 3. Large data transfer (images, JSON) → serialization spike
Bridge Flood

The most common performance anti-pattern is "bridge flooding" — sending too many messages per frame. Each scroll event, each animation tick, each state update that triggers a native view change crosses the bridge. At 60fps with a complex scroll handler, you can easily generate 100+ bridge messages per second. The bridge batches every 5ms, but serialization of large batches still blocks the JS thread.

5. Performance Monitoring Tools

5.1 Flipper

// Flipper — React Native debugger with performance plugins
// Built-in plugins:
// • React DevTools — component tree, props, state
// • Network Inspector — request/response timing
// • Layout Inspector — view hierarchy
// • Databases — AsyncStorage, SQLite inspection
// • Hermes Debugger — JS profiling

// Custom performance logging via Flipper
import { addPlugin } from 'react-native-flipper';

addPlugin({
  getId: () => 'performance-monitor',
  onConnect(connection) {
    const observer = new PerformanceObserver((list) => {
      for (const entry of list.getEntries()) {
        connection.send('performanceEntry', {
          name: entry.name,
          duration: entry.duration,
          startTime: entry.startTime,
        });
      }
    });
    observer.observe({ entryTypes: ['measure'] });
  },
  onDisconnect() {},
  runInBackground: () => false,
});

7. Identifying Bottlenecks

Bottleneck Identification Decision Tree ════════════════════════════════════════ Symptom: Jank / dropped frames │ ├── Is JS thread busy? (Hermes profiler / Systrace) │ ├── YES → JS bottleneck │ │ ├── Heavy computation? → Offload to native/worker │ │ ├── Too many re-renders? → Memoization, state splitting │ │ ├── Large JSON parsing? → Streaming parse, native parse │ │ └── Slow event handlers? → Debounce, InteractionManager │ │ │ └── NO → Check other threads │ ├── Is UI thread busy? (Instruments / Android Profiler) │ ├── YES → Native bottleneck │ │ ├── Too many view ops? → View flattening, fewer wrappers │ │ ├── Image decoding? → Resize before display, caching │ │ ├── Complex shadows? → Simplify, use elevation (Android) │ │ └── Sync native calls? → Make async, use TurboModules │ │ │ └── NO → Check bridge │ └── Is bridge saturated? (Systrace bridge markers) ├── YES → Bridge bottleneck │ ├── Too many calls/frame? → Batch, reduce frequency │ ├── Large payloads? → Reduce data size, use refs │ └── Scroll/animation traffic? → useNativeDriver, Reanimated │ └── NO → Check layout (Yoga) or memory pressure 
// Quick bottleneck identification technique
// Add this to your app's dev mode

function usePerformanceMonitor() {
  useEffect(() => {
    if (!__DEV__) return;

    let frameCount = 0;
    let lastCheck = performance.now();

    const check = () => {
      frameCount++;
      const now = performance.now();
      if (now - lastCheck >= 1000) {
        const fps = Math.round(frameCount * 1000 / (now - lastCheck));
        if (fps < 50) {
          console.warn(`Low FPS: ${fps}`);
        }
        frameCount = 0;
        lastCheck = now;
      }
      requestAnimationFrame(check);
    };

    requestAnimationFrame(check);
  }, []);
}

8. Key Performance Metrics

// Production performance monitoring
const metrics = {
  // Startup metrics
  coldStartTime: null,     // native launch → first meaningful paint
  ttInteractive: null,     // launch → first user interaction possible
  bundleLoadTime: null,    // bundle fetch + parse + execute

  // Runtime metrics
  jsFrameTime: [],         // ms per JS frame (target: <16ms)
  droppedFrames: 0,        // total dropped frames in session
  bridgeCallsPerSecond: 0, // bridge traffic volume

  // Memory metrics
  jsHeapUsed: null,        // Hermes heap usage
  nativeMemory: null,      // total app memory

  // Navigation metrics
  screenTransitionTime: [],// ms per screen transition
};

// Report to analytics
function reportPerformance() {
  analytics.track('performance_snapshot', {
    avgFps: calculateAvgFps(metrics.jsFrameTime),
    p95FrameTime: percentile(metrics.jsFrameTime, 95),
    droppedFrameRate: metrics.droppedFrames / sessionDuration,
    memoryUsageMB: metrics.jsHeapUsed / (1024 * 1024),
  });
}
Staff-Level Mental Model

Performance optimization in React Native is a multi-dimensional problem. You cannot optimize by looking at one thread in isolation — a fix on the JS thread (e.g., moving work to native) may create a bottleneck on the native modules thread. The Staff-level approach is: (1) establish performance budgets before writing code, (2) instrument continuously (not just when things are slow), (3) profile before optimizing (measure, don't guess), (4) optimize the bottleneck thread (not the one that's easiest to optimize), and (5) validate that optimizations don't regress other metrics.
