JS Thread Optimization

The JavaScript thread is the single most common bottleneck in React Native apps. It runs your React reconciliation, event handlers, state management, and business logic — all competing for a 16ms frame budget. This doc covers Hermes optimizations, computation offloading, and re-render reduction strategies.

1. Hermes Engine Optimizations

Hermes vs JavaScriptCore (JSC) ══════════════════════════════ Hermes JSC ────── ─── Compilation: AOT bytecode JIT compilation Startup: Fast (precompiled) Slow (parse + compile) Peak perf: Good Better (JIT optimized) Memory: Lower (~30% less) Higher GC: GenGC (generational) Riptide (concurrent) Profiling: Built-in sampling Limited Bundle format: .hbc (bytecode) .jsbundle (source) Binary size: Smaller Larger Hermes Bytecode Pipeline: ───────────────────────── .js source → Hermes compiler → .hbc bytecode → Hermes VM (build time) (runtime) Key insight: Hermes moves compilation cost from runtime to build time. This is why Hermes apps start faster but may have slightly lower peak throughput than JIT engines. 
// Verify Hermes is enabled
const isHermes = () => !!global.HermesInternal;
console.log(`Engine: ${isHermes() ? 'Hermes' : 'JSC'}`);

// Hermes-specific optimizations:
// 1. Use typed arrays instead of regular arrays for numeric data
const buffer = new Float64Array(1000); // faster than Array(1000)

// 2. Avoid eval() and new Function() — Hermes doesn't JIT
// 3. Prefer for loops over forEach for hot paths
// 4. Avoid arguments object — use rest params
// 5. Use Map/Set instead of plain objects for dynamic keys

// Hermes GC tuning (react-native.config.js)
// hermesFlags: '--max-heap-size=512' // MB
Hermes GenGC

Hermes uses a generational garbage collector (GenGC) with young and old generations. Young gen collections are fast (~1-3ms) but old gen collections can take 10-50ms and cause frame drops. To minimize GC pressure: (1) reduce object allocation rate, (2) avoid creating short-lived objects in hot loops, (3) reuse objects where possible, (4) use typed arrays for numeric data.

2. Avoiding Expensive JS Operations

// EXPENSIVE: Creating new objects every render
function BadComponent({ items }) {
  return (
    <FlatList
      data={items}
      style={{ flex: 1 }}           // new object every render!
      contentContainerStyle={{       // new object every render!
        padding: 16
      }}
      renderItem={({ item }) => (   // new function every render!
        <ItemRow item={item} />
      )}
      keyExtractor={(item) =>       // new function every render!
        item.id.toString()
      }
    />
  );
}

// OPTIMIZED: Stable references
const styles = StyleSheet.create({
  container: { flex: 1 },
  content: { padding: 16 },
});

const keyExtractor = (item) => item.id.toString();
const renderItem = ({ item }) => <ItemRow item={item} />;

function GoodComponent({ items }) {
  return (
    <FlatList
      data={items}
      style={styles.container}
      contentContainerStyle={styles.content}
      renderItem={renderItem}
      keyExtractor={keyExtractor}
    />
  );
}

3. InteractionManager

InteractionManager lets you defer work until after animations and interactions complete, preventing jank during transitions.

import { InteractionManager } from 'react-native';

function ScreenWithHeavyLoad() {
  const [data, setData] = useState(null);
  const [isReady, setIsReady] = useState(false);

  useEffect(() => {
    // Wait for navigation animation to complete
    const handle = InteractionManager.runAfterInteractions(() => {
      // Now safe to do heavy work
      const result = processLargeDataset();
      setData(result);
      setIsReady(true);
    });

    return () => handle.cancel();
  }, []);

  if (!isReady) {
    return <SkeletonLoader />;
  }

  return <DataView data={data} />;
}

// Creating custom interaction handles
const handle = InteractionManager.createInteractionHandle();
// ... animation running ...
InteractionManager.clearInteractionHandle(handle);
// Deferred work now runs

4. useMemo/useCallback in RN Context

// useMemo — cache expensive computations
function FilteredList({ items, filter, sortBy }) {
  // Without memo: runs every render (~50ms for 10K items)
  // With memo: runs only when items, filter, or sortBy change
  const processedItems = useMemo(() => {
    return items
      .filter(item => matchesFilter(item, filter))
      .sort((a, b) => compareBy(a, b, sortBy));
  }, [items, filter, sortBy]);

  return <FlatList data={processedItems} renderItem={renderItem} />;
}

// useCallback — stabilize function references for child memoization
function ParentScreen() {
  const [count, setCount] = useState(0);

  // Without useCallback: new function ref every render
  // → MemoizedChild re-renders even though its props didn't change
  const handlePress = useCallback(() => {
    setCount(c => c + 1);
  }, []);

  return (
    <>
      <Text>{count}</Text>
      <MemoizedChild onPress={handlePress} />
    </>
  );
}

const MemoizedChild = React.memo(({ onPress }) => {
  console.log('Child rendered'); // only when onPress changes
  return <Button onPress={onPress} title="Press" />;
});
Memoization Overhead

Memoization is not free — useMemo and useCallback have overhead for storing the cached value and comparing dependencies. Only memoize when: (1) the computation is expensive (>1ms), (2) the component re-renders frequently with the same inputs, or (3) the result is passed to a memoized child. Memoizing cheap operations (string concatenation, simple math) can actually be slower than recomputing.

5. Heavy Computation Offloading

Computation Offloading Strategies ═════════════════════════════════ Strategy 1: Native Module (recommended for >10ms work) ─────────────────────────────────────────────────────── JS Thread → TurboModule → Native Thread → Result back to JS Pro: Full platform API access, best performance Con: Platform-specific code, build complexity Strategy 2: Worklets (Reanimated) ───────────────────────────────── JS Thread → worklet → UI Thread (synchronous) Pro: Access to UI thread, no bridge Con: Limited API, only for UI-related computation Strategy 3: Web Workers (experimental) ────────────────────────────────────── JS Thread → Worker Thread → Message back to JS Pro: Pure JS, familiar API Con: Experimental in RN, data serialization overhead Strategy 4: Chunked Processing ────────────────────────────── JS Thread → process chunk → yield → process chunk → yield Pro: No native code needed, works everywhere Con: Total time increases, more complex logic 
// Strategy 4: Chunked processing with requestAnimationFrame
async function processInChunks<T>(
  items: T[],
  processor: (item: T) => void,
  chunkSize: number = 50
): Promise<void> {
  for (let i = 0; i < items.length; i += chunkSize) {
    const chunk = items.slice(i, i + chunkSize);
    chunk.forEach(processor);

    if (i + chunkSize < items.length) {
      // Yield to allow frame rendering
      await new Promise(resolve =>
        requestAnimationFrame(resolve)
      );
    }
  }
}

// Usage
await processInChunks(largeDataset, (item) => {
  transformedItems.push(expensiveTransform(item));
}, 100);

6. JSON Parsing Optimization

// Problem: JSON.parse blocks JS thread
// 1MB JSON ≈ 20-50ms parse time on Hermes

// Solution 1: Parse in native module
const data = await NativeModules.FastJSON.parse(jsonString);

// Solution 2: Streaming parse for large responses
async function fetchLargeData(url: string) {
  const response = await fetch(url);
  const reader = response.body?.getReader();
  const decoder = new TextDecoder();
  let buffer = '';
  const items = [];

  while (true) {
    const { done, value } = await reader.read();
    if (done) break;

    buffer += decoder.decode(value, { stream: true });
    // Parse complete JSON objects from buffer
    const parsed = extractCompleteObjects(buffer);
    items.push(...parsed.objects);
    buffer = parsed.remaining;
  }

  return items;
}

7. Reducing Re-renders

// Anti-pattern: Context causing tree-wide re-renders
const AppContext = createContext({ user: null, theme: 'light', cart: [] });

// Every consumer re-renders when ANY value changes
function Header() {
  const { user } = useContext(AppContext); // re-renders on cart change!
  return <Text>{user.name}</Text>;
}

// Fix: Split contexts by update frequency
const UserContext = createContext(null);
const ThemeContext = createContext('light');
const CartContext = createContext([]);

// Fix: Use selectors with external stores (Zustand, Jotai)
import { useStore } from 'zustand';

const useUserName = () => useStore(store => store.user.name);
// Only re-renders when user.name actually changes

// Fix: Composition pattern — move state down
function ParentWithCounter() {
  return (
    <>
      <ExpensiveTree />        {/* doesn't re-render */}
      <CounterSection />       {/* state is local here */}
    </>
  );
}

function CounterSection() {
  const [count, setCount] = useState(0);
  return <Button onPress={() => setCount(c => c + 1)} />;
}

Staff-Level Mental Model

JS thread optimization is about time budgeting. You have 16ms per frame. React reconciliation takes 2-5ms for a typical screen. That leaves 11-14ms for your code. Every re-render, every event handler, every computation competes for this budget. The Staff-level approach: (1) profile first to find the actual bottleneck, (2) reduce work (fewer re-renders, simpler computations), (3) defer work (InteractionManager, requestIdleCallback), (4) move work off the JS thread (native modules, worklets). Never optimize without measuring — intuition about performance is almost always wrong.
