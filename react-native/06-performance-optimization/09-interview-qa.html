Performance Model

Q1: Describe the threading model of React Native and how each thread contributes to frame rendering.

React Native has three frame-critical threads: (1) JS Thread — runs Hermes/JSC, executes React reconciliation, event handlers, and business logic. (2) Shadow Thread — runs Yoga layout engine, computes flexbox positions. (3) UI/Main Thread — creates/updates native views, handles touch events, runs native animations. Plus a NativeModules Thread for async native method execution. All must complete their work within 16.67ms for 60fps. The bridge (legacy) or JSI (new arch) connects them.

Q2: What is the frame budget at 60fps and how is it divided across threads?

16.67ms total per frame. Approximate division: JS thread ~8ms (reconciliation + event handlers), bridge/JSI ~2ms (communication overhead), shadow thread ~3ms (Yoga layout), UI thread ~3ms (native view updates), ~0.67ms buffer. If any single thread exceeds its budget, frames drop. On 120Hz ProMotion displays, the budget halves to 8.33ms — making optimization even more critical.

Q3: How does the bridge batch native module calls and what is the performance impact?

The bridge batches JS-to-native calls on a 5ms timer. Calls are queued, then serialized as a single JSON message every 5ms (or when the queue reaches a size threshold). This adds up to 5ms latency per call and means multiple calls within 5ms are batched together. The batching amortizes serialization overhead but creates unpredictable latency. TurboModules eliminate this by using direct JSI calls with no batching.

Q4: How do you determine whether a performance issue is on the JS thread, UI thread, or bridge?

Use Systrace (Android) or Instruments (iOS) to see per-thread activity. If mqt_js thread shows long execution blocks → JS bottleneck. If the UI thread shows long view operations → native bottleneck. If bridge markers show large serialization times → bridge bottleneck. Quick heuristic: if animations jank but native-driven animations don't → JS thread. If everything janks → UI thread. If only bridge-crossing operations are slow → bridge.

Q5: What performance improvements does the New Architecture (Fabric + TurboModules) provide?

TurboModules: ~100x faster native calls (10μs vs 1-5ms), lazy loading (faster startup), synchronous methods. Fabric: concurrent rendering (non-blocking UI updates), C++ layout computation (off JS thread), synchronous view updates (lower latency), view flattening at the C++ level. Combined: interaction latency drops from 50-100ms to 16-33ms, scroll performance becomes native-bound instead of bridge-bound.

Q6: What performance budgets would you set for a production React Native app?

Cold start TTI: <2s. Screen transition: <300ms. Scroll FPS: 60fps (never below 45fps). JS frame time: <8ms (p95). Memory (JS heap): <150MB. Bundle size: <2MB JS. Bridge calls per frame: <5. Image decode time: <100ms. Network request (API): <500ms (p95). OTA update download: <500KB. These should be enforced in CI with automated performance testing.

Q7: How would you set up continuous performance monitoring in production?

Instrument startup time (native start → JS start → first render → interactive), frame rate (sampling JS frame callback), memory (Hermes heap stats), screen transition time (navigation events), and network latency. Report to analytics (Datadog, New Relic, custom). Set up alerts for p95 regressions. Use A/B testing to validate performance changes. Track per-device-model and per-OS-version to identify device-specific issues.

JS Thread Optimization

Q8: How does Hermes differ from JSC in terms of performance characteristics?

Hermes uses AOT bytecode compilation (fast startup, lower peak throughput). JSC uses JIT compilation (slower startup, higher peak throughput after warmup). Hermes has lower memory usage (~30% less) due to no JIT code cache. Hermes GC (GenGC) is stop-the-world for old gen; JSC GC (Riptide) is mostly concurrent. For React Native, Hermes wins on startup and memory; JSC wins on sustained computation. Most apps benefit more from Hermes.

Q9: What is InteractionManager and when should you use it?

InteractionManager.runAfterInteractions() defers work until after all animations and interactions complete. Use it when navigating to a new screen — defer heavy data loading until the transition animation finishes. Without it, heavy JS work during the transition causes jank. It works by tracking "interaction handles" — animations create handles, and deferred work runs when all handles are cleared.

Q10: When does useMemo actually help performance in React Native vs when is it overhead?

Helps when: computation is expensive (>1ms), component re-renders frequently with same inputs, or result is passed to a memoized child component. Overhead when: computation is cheap (string concat, simple math), component rarely re-renders, or dependencies change every render anyway. useMemo has ~0.1ms overhead per call for storing and comparing. Rule of thumb: profile first, memoize only measured bottlenecks.

Q11: How do you offload heavy computation from the JS thread?

Four strategies: (1) Native module — move computation to native thread via TurboModule (best for >10ms work). (2) Reanimated worklets — run on UI thread (for UI-related computation). (3) Chunked processing — split work into chunks with requestAnimationFrame yields between chunks (no native code needed). (4) Web Workers — experimental in RN, run JS on separate thread. Choose based on: computation type, latency requirements, and team's native code expertise.

Q12: How do you optimize JSON parsing for large API responses?

JSON.parse blocks the JS thread — 1MB JSON takes 20-50ms on Hermes. Solutions: (1) Parse in a native module (runs on native thread). (2) Use streaming parsing for large responses. (3) Use binary formats (Protocol Buffers, FlatBuffers) via C++ TurboModule — 5-10x faster. (4) Paginate API responses to keep individual payloads small (<100KB). (5) Use GraphQL to request only needed fields.

Q13: How do you reduce unnecessary re-renders caused by Context?

Context causes all consumers to re-render when any value changes. Solutions: (1) Split contexts by update frequency (UserContext, ThemeContext, CartContext). (2) Use external state management with selectors (Zustand, Jotai) — only re-render when selected value changes. (3) Memoize context values with useMemo. (4) Use the composition pattern — move state down to the component that needs it. (5) Use React.memo on consumer components with custom comparison.

Q14: What are the performance implications of Hermes's garbage collector?

Hermes GenGC: young gen collections are fast (~1-3ms, barely noticeable). Old gen collections are stop-the-world (~10-50ms, can cause frame drops). Triggered when old gen fills up. Mitigation: reduce allocation rate (fewer short-lived objects), reuse objects, use typed arrays for numeric data, avoid creating objects in hot loops. Monitor with HermesInternal.getInstrumentedStats() — track js_numGCs and correlate with frame drops.

Rendering Optimization

Q15: What are the key FlatList props for performance and what does each do?

windowSize (default 21): screens of content to render — lower = less memory, more blank space during fast scroll. maxToRenderPerBatch (default 10): items per render batch — lower = less JS blocking per batch. getItemLayout: skips measurement, enables scrollToIndex. removeClippedSubviews: detaches off-screen native views (Android). initialNumToRender: items on first render. keyExtractor: stable keys prevent unnecessary remounts.

Q16: How does FlashList achieve better performance than FlatList?

FlashList uses cell recycling (like RecyclerView/UICollectionView). Instead of unmounting off-screen items and creating new ones, it reuses the same native view hierarchy and updates props. This eliminates native view creation/destruction during scrolling — the most expensive operation. FlatList creates new views for each item entering the viewport. FlashList requires estimatedItemSize and works best when items have consistent structure.

Q17: What is view flattening and how does React Native implement it?

View flattening merges "layout-only" views (views with only flex/padding/margin props) into their parent, reducing the native view count. React Native does this automatically for views without visual properties (no backgroundColor, borderRadius, opacity, shadow, transform). Fewer native views = faster layout computation, less memory, faster rendering. Use collapsable={false} to prevent flattening when you need the view for measurement or ref access.

Q18: Why do unstable keys cause performance problems in lists?

Unstable keys (index-based or random) cause React to treat items as new components on every render. This triggers: (1) full component unmount + remount (expensive), (2) native view destruction + creation, (3) loss of component state, (4) animation reset, (5) re-execution of all effects. Stable keys (item.id) let React match items across renders, enabling minimal updates (just prop changes). For a 100-item list, unstable keys can cause 100x more work per update.

Q19: How do you effectively memoize list items to prevent re-renders during scroll?

Wrap the item component in React.memo. Ensure all props passed to it have stable references: use useCallback for event handlers, StyleSheet.create for styles, and avoid inline objects. The renderItem function itself should be wrapped in useCallback. For items with complex data, use a custom comparison function in React.memo that only checks the fields that affect rendering.

Memory Optimization

Q20: What are the main memory consumers in a React Native app?

JS heap (30-150MB): React fiber tree, state objects, closures, module cache, string pool. Native memory (50-300MB): decoded image bitmaps (largest — a 4K image = 33MB), native view hierarchy, WebView instances (~30-50MB each), video buffers. Shared memory: mmap'd Hermes bytecode, MMKV storage, shared libraries. Images are almost always the #1 memory consumer.

Q21: List the most common memory leak patterns in React Native and how to fix each.

(1) Event listeners not removed — fix: return cleanup function from useEffect. (2) Timers not cleared — fix: clearInterval/clearTimeout in cleanup. (3) Closures capturing large objects — fix: use functional state updates, minimize closure scope. (4) Navigation stack accumulation — fix: unmountOnBlur: true for heavy screens. (5) Animated values not stopped — fix: anim.stopAnimation() in cleanup. (6) WebSocket connections not closed — fix: close in cleanup.

Q22: How do you profile memory in a React Native app?

JS heap: Hermes heap snapshot (Chrome DevTools Memory tab), HermesInternal.getInstrumentedStats(). iOS native: Xcode Memory Graph Debugger (retain cycles), Instruments Allocations (heap growth). Android native: Android Studio Memory Profiler (heap allocations, GC events), LeakCanary (Activity leaks). Compare heap snapshots before and after a suspected leak to identify growing objects.

Q23: How do you optimize image memory in React Native?

Memory = width × height × 4 bytes. Solutions: (1) Request exact display size from CDN (?w=200&h=200). (2) Use FastImage with caching (cacheControl: immutable). (3) Implement progressive loading (thumbnail → full image). (4) Clear image cache under memory pressure. (5) Use WebP format (30% smaller decoded). (6) Limit concurrent image loads. (7) For lists, only load images for visible items (FlatList windowing handles this).

Q24: How does React Navigation's stack navigator affect memory?

Stack navigator keeps all screens in memory by default for instant back navigation. A deep stack (20+ screens) accumulates components, state, images, and native views. Solutions: (1) unmountOnBlur: true for memory-heavy screens. (2) Use tab/drawer navigators to limit screen count. (3) Reset stack when navigating to a new flow (navigation.reset()). (4) Implement screen-level cleanup in useFocusEffect to release resources when screen loses focus.

Q25: How do you handle low-memory warnings in React Native?

Listen to AppState.addEventListener('memoryWarning') on JS side. On iOS, implement applicationDidReceiveMemoryWarning in AppDelegate to clear URL cache and image cache. On Android, override onTrimMemory in Application class. When triggered: clear image caches, reduce navigation stack, release non-critical data, switch to reduced-quality mode. Monitor memory warnings in production analytics to identify memory-hungry screens.

Startup Optimization

Q26: Break down the cold start timeline of a React Native app.

(1) OS process creation ~50ms. (2) Native app init (AppDelegate/Application, SDK init) ~100ms. (3) RN bridge init (create bridge, init native modules) ~50ms. (4) JS bundle loading (mmap for Hermes, parse for JSC) ~100-300ms. (5) JS execution (module init, React tree creation) ~100-300ms. (6) Native view creation (create views, Yoga layout) ~50-100ms. (7) First paint ~50ms. Total: 500-1000ms optimized, 1500-3000ms unoptimized.

Q27: How does Hermes bytecode precompilation improve startup?

Hermes compiles JS to bytecode at build time, eliminating parse + compile at runtime. JSC: load source (~50ms) + parse (~100-200ms) + compile (~100-300ms) = 250-550ms. Hermes: mmap .hbc file (~5ms) + verify header (~1ms) = ~6ms. The bytecode file is memory-mapped, so only accessed pages load into RAM. This is the single biggest startup optimization — it reduced bundle load time by 90%+.

Q28: What are inline requires and how do they improve startup?

Inline requires transform top-level import statements into lazy require() calls at the point of first use. Without inline requires, all imported modules execute at startup. With inline requires, modules only execute when first accessed. This defers initialization of modules not needed for the initial screen. Enable via inlineRequires: true in Metro config. Can save 50-200ms depending on module count.

Q29: How do TurboModules improve startup time compared to legacy NativeModules?

Legacy: ALL native modules are instantiated at bridge creation — each takes ~1-5ms, 50 modules = 50-250ms. TurboModules: only a proxy is installed at startup. Modules are instantiated on first access via __turboModuleProxy. Modules never accessed are never created. Savings = (total modules - startup modules) × avg init time. For an app with 50 modules where only 10 are used at startup: saves ~120ms.

Q30: What splash screen strategy provides the best perceived startup performance?

Use a native splash screen (not React-rendered) that displays instantly at app launch. Keep it visible while JS loads and the first screen renders. Hide it only after the first meaningful content is painted (use onLayout callback). Pre-load critical data (auth token, user profile, fonts) behind the splash. Use SplashScreen.preventAutoHideAsync() + SplashScreen.hideAsync() for precise control. The splash should match the first screen's layout to create a seamless transition.

Animation Performance

Q31: Explain the difference between JS-driven and native-driven animations.

JS-driven: every frame, JS calculates the animation value → sends it across the bridge → native updates the view. If JS is busy, frames drop. Native-driven (useNativeDriver: true): JS sends the animation configuration once → native calculates and updates every frame independently. JS thread is free during the animation. Native-driven is limited to opacity and transform. Reanimated extends this to any property via worklets.

Q32: What properties can useNativeDriver animate and why are others excluded?

Supported: opacity, transform (translate, scale, rotate, skew). Excluded: width, height, padding, margin, backgroundColor, borderRadius, fontSize. Reason: excluded properties trigger Yoga layout recalculation, which must run on the shadow thread (not the UI thread where native driver runs). Layout changes require the full render pipeline: JS → shadow → UI. Only visual properties that don't affect layout can be animated natively.

Q33: How does Reanimated's worklet system work architecturally?

Reanimated installs a lightweight JS runtime on the UI thread (separate from Hermes). Functions marked with 'worklet' are compiled and transferred to this runtime. useSharedValue creates values accessible from both runtimes without serialization (shared memory). useAnimatedStyle creates a worklet that runs every frame on the UI thread, reading shared values and producing style updates. runOnJS bridges back to the main JS thread when needed.

Q34: How do you implement gesture-driven animations that run at 60fps?

Use react-native-gesture-handler + Reanimated. Gesture handlers run on the UI thread (not JS thread). In the gesture's onUpdate callback (a worklet), update shared values directly. useAnimatedStyle reads these shared values and produces style updates — all on the UI thread, zero bridge crossing. The entire gesture → animation → view update pipeline runs at 60fps regardless of JS thread load.

Q35: What is the most common cause of animation jank and how do you fix it?

The most common cause is JS thread contention — heavy computation or re-renders during animation. Fixes: (1) Use useNativeDriver: true or Reanimated for all user-facing animations. (2) Defer heavy work with InteractionManager.runAfterInteractions(). (3) Avoid state updates during animations. (4) Use requestAnimationFrame instead of setTimeout for frame-synced work. (5) Profile with Systrace to confirm the JS thread is the bottleneck.

Network Optimization

Q36: How does React Native's fetch() differ from the browser Fetch API?

RN's fetch() is a JS wrapper around native HTTP clients (NSURLSession on iOS, OkHttp on Android). Differences: (1) native-level performance with HTTP/2 multiplexing and connection pooling, (2) no CORS restrictions (native networking), (3) cookie handling differs from browser, (4) streaming/ReadableStream support is limited, (5) native caching (NSURLCache/OkHttp cache) is automatic. The native layer handles GZIP decompression, certificate validation, and TLS.

Q37: Describe an effective caching strategy for a React Native app.

Four layers: (1) In-memory cache (React Query/SWR) — TTL seconds to minutes, fastest access. (2) Persistent cache (MMKV) — TTL hours to days, survives app restart. (3) HTTP cache (NSURLCache/OkHttp) — based on Cache-Control headers, automatic. (4) Network request — actual server call. Stale-while-revalidate pattern: serve cached data immediately, fetch fresh data in background, update UI when fresh data arrives. MMKV is preferred over AsyncStorage for speed (synchronous, memory-mapped).

Q38: How do you implement offline support in a React Native app?

Architecture: (1) Cache all API responses in persistent storage (MMKV/SQLite). (2) Serve cached data when offline. (3) Queue mutations (writes) in an offline mutation queue. (4) Flush the queue when connectivity returns (with conflict resolution). (5) Use @react-native-community/netinfo to detect connectivity changes. (6) Show stale indicators on cached data. (7) Use optimistic updates for instant UI feedback. Libraries: React Query with persistence, WatermelonDB for complex offline-first apps.

Q39: How do you optimize image loading in a list that scrolls quickly?

(1) Use FastImage with aggressive caching (cacheControl: immutable). (2) Request thumbnails from CDN, not full-size images. (3) Preload images for items about to enter the viewport. (4) Use placeholder/skeleton while loading. (5) Implement progressive loading (blur-up pattern). (6) Cancel image loads for items that scroll out of view. (7) Limit concurrent image downloads (FastImage handles this). (8) Use WebP format for 30% smaller files.

Q40: How do you optimize GraphQL queries for mobile performance?

(1) Request only needed fields (avoid over-fetching). (2) Use persisted queries (send hash instead of full query — smaller payload). (3) Implement normalized caching (Apollo InMemoryCache with type policies). (4) Use optimistic updates for mutations. (5) Persist cache to MMKV for instant app restart. (6) Implement pagination with cursor-based approach. (7) Use subscriptions sparingly (WebSocket connections drain battery). (8) Batch multiple queries into a single request.

Bundle Size

Q41: Why is Hermes bytecode larger than minified JS but still better for performance?

Hermes bytecode (.hbc) is ~2.5x larger than minified JS because it includes instruction encoding, unminified string table, function metadata, and constant pool. However, it's memory-mapped (mmap), so only accessed pages load into RAM. The OS can evict and reload pages transparently. Net effect: larger file on disk, but faster startup (no parse/compile), less runtime memory (no source + AST + compiled code in heap), and better GC behavior.

Q42: Why does Metro have limited tree shaking and how do you work around it?

Metro can't tree-shake CommonJS modules (dynamic require() prevents static analysis). Many RN libraries use CommonJS. Workarounds: (1) Use path imports (lodash/debounce not lodash). (2) Avoid barrel exports (import from specific files). (3) Choose libraries that ship ES modules. (4) Use babel-plugin-transform-imports to auto-convert barrel imports. (5) Enable experimentalImportSupport in Metro. (6) Audit with source-map-explorer to find bloated modules.

Q43: How do you implement code splitting in React Native?

Screen-level: React.lazy(() => import('./Screen')) with Suspense fallback. Feature-level: dynamic import() for heavy libraries loaded on demand. Role-based: only import admin screens for admin users. The key limitation: Metro doesn't support multiple output chunks like webpack. React.lazy works because it defers the require() call, not because it creates separate chunks. For true chunk splitting, use RAM bundles or re-exports.

Q44: How do Proguard/R8 reduce Android app size?

R8 (successor to Proguard): (1) Code shrinking — removes unused Java/Kotlin classes and methods. (2) Obfuscation — shortens class/method names (smaller bytecode). (3) Optimization — inlines methods, removes dead branches. (4) Resource shrinking — removes unused resources (images, layouts). For RN: must keep React Native classes with ProGuard rules. Combined with ABI splits and AAB format, can reduce APK size by 30-50%.

Q45: How do you minimize OTA update size for CodePush/EAS Update?

OTA size = diff between old and new bundle. Minimize by: (1) Use stable module IDs (deterministic builds). (2) Avoid changing core dependencies (react, react-native) in OTA updates. (3) Keep assets separate from JS bundle. (4) Use Hermes bytecode (consistent output for same input). (5) Avoid reformatting unchanged code. (6) Use runtime version pinning to ensure compatibility. Typical sizes: bug fix 10-50KB, feature 100-500KB, major refactor 1-5MB.

Cross-Cutting Performance

Q46: Design a performance monitoring system for a React Native app at scale.

Layers: (1) Startup metrics — native start, JS start, first render, TTI (measured via native timestamps + JS performance.now). (2) Runtime metrics — JS frame time (requestAnimationFrame sampling), dropped frames, bridge call count. (3) Screen metrics — per-screen render time, transition time. (4) Memory — Hermes heap stats, native memory (periodic sampling). (5) Network — request latency, cache hit rate. Report to analytics with device model, OS version, app version segmentation. Alert on p95 regressions.

Q47: How do you prevent performance regressions in a large team?

(1) Bundle size budget in CI — fail builds exceeding limit. (2) Automated performance tests (Detox/Maestro) measuring startup time and FPS. (3) Dependency size audit on every PR (bundlephobia check). (4) Performance review checklist for code reviews (memoization, stable refs, native driver). (5) Per-commit performance tracking in CI. (6) Production monitoring with automated regression alerts. (7) Performance champions per team who review perf-sensitive changes.

Q48: Compare the performance trade-offs of different state management solutions in RN.

React Context: simple, but causes tree-wide re-renders on any change. Redux: predictable, but selector overhead and middleware chain add ~0.5-1ms per dispatch. Zustand: lightweight, selector-based re-renders (only affected components). Jotai: atomic state, minimal re-renders, ~0.1ms per atom read. MobX: observable-based, automatic fine-grained re-renders, but proxy overhead. For RN performance: Zustand or Jotai for most apps, Redux for complex state machines, avoid Context for frequently-changing values.

Q49: How do you optimize a screen that displays a complex form with 50+ fields?

(1) Use react-hook-form (uncontrolled inputs — no re-render per keystroke). (2) Split the form into sections with lazy rendering (only render visible section). (3) Memoize each field component with React.memo. (4) Use FlatList or SectionList for the form layout (virtualization). (5) Debounce validation. (6) Avoid controlled inputs for text fields (use refs). (7) Use keyboardShouldPersistTaps="handled" to prevent keyboard-related re-renders.

Q50: How do you optimize a chat screen with thousands of messages?

(1) Use FlashList with inverted={true} for bottom-to-top rendering. (2) Implement message pagination (load 50 at a time, load more on scroll up). (3) Memoize message components aggressively. (4) Use stable keys (message IDs). (5) Optimize image messages (thumbnails, lazy loading). (6) Implement message grouping to reduce item count. (7) Use Reanimated for typing indicators and read receipts. (8) Trim old messages from state when count exceeds threshold.

Q51: What is the performance impact of enabling the New Architecture and how do you measure it?

Expected improvements: native module calls 100x faster, startup 10-20% faster (lazy TurboModules), interaction latency 50-70% lower, scroll performance improved (synchronous Fabric updates). Measure by: A/B testing old vs new arch with same app, comparing: cold start time, FPS during scroll, interaction latency (tap → visual response), memory usage. Use Systrace/Instruments to compare per-thread activity. Note: initial migration may temporarily regress due to interop layer overhead.

Q52: How do you optimize a React Native app for low-end Android devices?

(1) Reduce view count (<30 per screen). (2) Use FlashList instead of FlatList. (3) Disable complex shadows and rounded corners. (4) Reduce image quality and size. (5) Lower animation complexity (fewer concurrent animations). (6) Increase FlatList updateCellsBatchingPeriod. (7) Use removeClippedSubviews. (8) Reduce JS bundle size (fewer dependencies). (9) Implement device-tier detection and adapt UI complexity. (10) Use Hermes (lower memory, faster startup).

Q53: How do you profile and optimize a slow screen transition?

Profile: measure time from navigation trigger to screen interactive. Use React DevTools Profiler to see render time of the new screen. Use Systrace to see bridge/JSI traffic during transition. Common causes: (1) heavy component mounting — fix with lazy loading and InteractionManager. (2) Large data fetch blocking render — fix with skeleton/placeholder. (3) Animation jank — fix with native-driven transitions. (4) Too many views created at once — fix with incremental rendering.

Q54: What is the performance difference between Animated and Reanimated?

Animated (with useNativeDriver): limited to opacity/transform, configuration sent once to native, native calculates per frame. Good for simple animations. Animated (without native driver): JS calculates every frame, crosses bridge — worst performance. Reanimated: worklets run on UI thread, supports any property, any logic (conditionals, math), gesture integration. Performance: Reanimated ≈ native driver for simple animations, significantly better for complex/gesture-driven animations.

Q55: How do you handle performance in a React Native app that uses WebView heavily?

Each WebView instance consumes ~30-50MB. Solutions: (1) Reuse a single WebView instance (navigate within it). (2) Unmount WebView when not visible. (3) Use react-native-webview's incognito mode to prevent cache accumulation. (4) Inject minimal CSS/JS into the WebView. (5) Use native components instead of WebView where possible. (6) Implement a WebView pool (pre-create 1-2 instances, reuse). (7) Monitor WebView memory and force reload if it grows too large.

Q56: How do you optimize touch responsiveness in React Native?

(1) Keep JS thread frame time <8ms (responsive to touch events). (2) Use Pressable over TouchableOpacity (more configurable). (3) Add visual feedback immediately (opacity change via native driver). (4) Debounce expensive handlers. (5) Use react-native-gesture-handler for gesture recognition on the native thread. (6) Avoid setTimeout in touch handlers. (7) Use hitSlop to increase touch targets without visual changes.

Q57: What are the performance implications of using TypeScript in React Native?

Zero runtime performance impact — TypeScript is stripped at build time by Babel/Metro. The compiled JS is identical whether you write TS or JS. Build-time impact: type checking adds ~5-20s to build (can be parallelized with tsc --noEmit in CI). Development-time: IDE type checking can slow large projects. Indirect performance benefit: types catch bugs that could cause runtime errors or unnecessary re-renders (wrong prop types).

Q58: Design a performance optimization strategy for an app with 100+ screens.

(1) Categorize screens by criticality (core flow vs settings vs admin). (2) Lazy load non-core screens with React.lazy. (3) Performance budget per screen category (core: <200ms render, settings: <500ms). (4) Shared component library with pre-optimized, memoized components. (5) Automated testing — measure render time of each screen in CI. (6) Bundle analysis — per-screen bundle contribution. (7) Navigation optimization — limit stack depth, use unmountOnBlur. (8) Image CDN with automatic resizing.

Q59: How do you balance developer experience with performance optimization?

Encapsulate optimizations in reusable abstractions: optimized list components, memoized wrappers, performance-aware hooks. Don't require every developer to understand Reanimated worklets — provide pre-built animation components. Use lint rules to catch common anti-patterns (inline styles, missing keys). Automate performance testing in CI so developers get feedback without manual profiling. Document performance patterns in team guidelines. Reserve manual optimization for measured bottlenecks.

Q60: If you could only make three performance optimizations to a React Native app, what would they be?

(1) Enable Hermes — single biggest startup improvement (200-500ms saved), lower memory, built-in profiling. (2) Replace FlatList with FlashList — highest-ROI change for list-heavy apps, 2-5x better scroll performance. (3) Use Reanimated for all animations — eliminates the most common source of jank (JS-driven animations crossing the bridge). These three changes address the three most common performance complaints: slow startup, janky scrolling, and choppy animations.
