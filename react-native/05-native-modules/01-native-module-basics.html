Native Module Basics

Native modules are the escape hatch that gives React Native its power — they let JavaScript invoke arbitrary platform code on iOS and Android. Understanding the registration, method export, threading, and communication patterns is essential for any Staff-level mobile engineer.

1. Bridge Architecture for Native Modules

In the legacy (pre-New Architecture) React Native, native modules communicate with JavaScript through the bridge — an asynchronous, serialized message queue. Every method invocation crosses this boundary as a JSON-serialized message.

┌─────────────────────────────────────────────────────────────────┐ │ React Native Application │ │ │ │ ┌──────────────┐ Bridge (JSON) ┌──────────────────────┐ │ │ │ JS Thread │◄──────────────────►│ Native Thread(s) │ │ │ │ │ │ │ │ │ │ NativeModules │ ┌──────────────┐ │ iOS: RCTBridgeModule │ │ │ │ .MyModule │──│ MessageQueue │──│ Android: ReactModule │ │ │ │ .doWork() │ │ (async JSON) │ │ .doWork() │ │ │ └──────────────┘ └──────────────┘ └──────────────────────┘ │ │ │ │ Serialization cost: ~1ms per call for small payloads │ │ Batching: bridge batches calls every 5ms by default │ └─────────────────────────────────────────────────────────────────┘
Architect Insight

The bridge is the single biggest performance bottleneck in legacy RN. Every native module call requires: (1) JSON serialization on JS side, (2) async dispatch to native queue, (3) JSON deserialization on native side, (4) method invocation, (5) reverse path for results. TurboModules (covered in doc 02) eliminate this by using JSI for direct C++ function calls.

1.1 Module Discovery and Registration

At app startup, React Native discovers all registered native modules and creates a module registry. On the JS side, NativeModules is a proxy object that lazily resolves module references.

// JS side — accessing a native module
import { NativeModules } from 'react-native';

const { MyCustomModule } = NativeModules;

// Under the hood, NativeModules is populated from the native registry
// via __fbBatchedBridgeConfig at startup time
// Each module entry contains: name, methods[], constants{}

2. iOS Native Module (Objective-C)

The canonical way to create an iOS native module uses Objective-C with the RCTBridgeModule protocol.

// MyCustomModule.h
#import <React/RCTBridgeModule.h>
#import <React/RCTEventEmitter.h>

@interface MyCustomModule : RCTEventEmitter <RCTBridgeModule>
@end

// MyCustomModule.m
#import "MyCustomModule.h"

@implementation MyCustomModule

// Registers module with React Native — name defaults to ObjC class name
RCT_EXPORT_MODULE();

// Or with explicit name:
// RCT_EXPORT_MODULE(CustomName);

// Export a method — runs on a background serial queue by default
RCT_EXPORT_METHOD(doWork:(NSString *)input
                  resolver:(RCTPromiseResolveBlock)resolve
                  rejecter:(RCTPromiseRejectBlock)reject)
{
  @try {
    NSString *result = [self processInput:input];
    resolve(result);
  } @catch (NSException *exception) {
    reject(@"E_PROCESSING", exception.reason, nil);
  }
}

// Specify which queue this module's methods run on
- (dispatch_queue_t)methodQueue {
  return dispatch_get_main_queue(); // or custom queue
}

@end
Critical Detail

RCT_EXPORT_MODULE() uses the __attribute__((constructor)) GCC extension to register the module at load time — before main() is called. This means module registration has zero runtime cost but adds to binary startup. Having hundreds of native modules can measurably impact cold start time.

3. iOS Native Module (Swift)

Swift modules require a bridging header because RCT macros are Objective-C preprocessor macros that Swift cannot use directly.

// MySwiftModule.swift
import Foundation

@objc(MySwiftModule)
class MySwiftModule: NSObject {

  @objc
  static func requiresMainQueueSetup() -> Bool {
    return false  // true if init accesses UIKit
  }

  @objc
  func doWork(_ input: String,
              resolver resolve: @escaping RCTPromiseResolveBlock,
              rejecter reject: @escaping RCTPromiseRejectBlock) {
    DispatchQueue.global(qos: .userInitiated).async {
      do {
        let result = try self.processInput(input)
        resolve(result)
      } catch {
        reject("E_PROCESSING", error.localizedDescription, error)
      }
    }
  }

  @objc
  func constantsToExport() -> [String: Any] {
    return ["API_VERSION": "2.0", "PLATFORM": "ios"]
  }
}

// MySwiftModule.m — ObjC bridging file (required)
#import <React/RCTBridgeModule.h>

@interface RCT_EXTERN_MODULE(MySwiftModule, NSObject)
RCT_EXTERN_METHOD(doWork:(NSString *)input
                  resolver:(RCTPromiseResolveBlock)resolve
                  rejecter:(RCTPromiseRejectBlock)reject)
@end

4. Android Native Module (Java)

Android native modules implement ReactContextBaseJavaModule and are registered through a ReactPackage.

// MyCustomModule.java
package com.myapp.modules;

import com.facebook.react.bridge.ReactApplicationContext;
import com.facebook.react.bridge.ReactContextBaseJavaModule;
import com.facebook.react.bridge.ReactMethod;
import com.facebook.react.bridge.Promise;
import com.facebook.react.bridge.WritableMap;
import com.facebook.react.bridge.Arguments;

public class MyCustomModule
    extends ReactContextBaseJavaModule {

  MyCustomModule(ReactApplicationContext context) {
    super(context);
  }

  @Override
  public String getName() {
    return "MyCustomModule"; // must match JS side
  }

  @ReactMethod
  public void doWork(String input, Promise promise) {
    try {
      String result = processInput(input);
      promise.resolve(result);
    } catch (Exception e) {
      promise.reject("E_PROCESSING", e.getMessage(), e);
    }
  }

  @ReactMethod(isBlockingSynchronousMethod = true)
  public String getConfigSync() {
    return "sync_value";
  }

  @Override
  public Map<String, Object> getConstants() {
    Map<String, Object> constants = new HashMap<>();
    constants.put("API_VERSION", "2.0");
    constants.put("PLATFORM", "android");
    return constants;
  }
}

// MyCustomPackage.java — registers the module
public class MyCustomPackage implements ReactPackage {
  @Override
  public List<NativeModule> createNativeModules(
      ReactApplicationContext reactContext) {
    List<NativeModule> modules = new ArrayList<>();
    modules.add(new MyCustomModule(reactContext));
    return modules;
  }

  @Override
  public List<ViewManager> createViewManagers(
      ReactApplicationContext reactContext) {
    return Collections.emptyList();
  }
}

5. Android Native Module (Kotlin)

// MyCustomModule.kt
package com.myapp.modules

import com.facebook.react.bridge.*

class MyCustomModule(
  reactContext: ReactApplicationContext
) : ReactContextBaseJavaModule(reactContext) {

  override fun getName(): String = "MyCustomModule"

  @ReactMethod
  fun doWork(input: String, promise: Promise) {
    try {
      val result = processInput(input)
      promise.resolve(result)
    } catch (e: Exception) {
      promise.reject("E_PROCESSING", e.message, e)
    }
  }

  @ReactMethod(isBlockingSynchronousMethod = true)
  fun getConfigSync(): String = "sync_value"

  override fun getConstants(): Map<String, Any> = mapOf(
    "API_VERSION" to "2.0",
    "PLATFORM" to "android"
  )
}

6. Callback Patterns

The original native module communication pattern uses callbacks. Each callback can only be invoked once — invoking it a second time crashes the app.

// iOS — Objective-C callback method
RCT_EXPORT_METHOD(fetchData:(NSString *)url
                  successCallback:(RCTResponseSenderBlock)onSuccess
                  errorCallback:(RCTResponseSenderBlock)onError)
{
  [self.networkManager fetch:url completion:^(NSData *data, NSError *error) {
    if (error) {
      onError(@[error.localizedDescription]);
    } else {
      onSuccess(@[[data base64EncodedStringWithOptions:0]]);
    }
  }];
}

// Android — Java callback method
@ReactMethod
public void fetchData(String url, Callback onSuccess, Callback onError) {
  try {
    String data = networkManager.fetch(url);
    onSuccess.invoke(data);
  } catch (Exception e) {
    onError.invoke(e.getMessage());
  }
}

// JS side — using callbacks
NativeModules.MyModule.fetchData(
  'https://api.example.com/data',
  (data) => console.log('Success:', data),
  (error) => console.error('Error:', error)
);
Callback Pitfalls

Callbacks are fire-once. If your native code has multiple code paths that might invoke the callback, you must guard against double invocation. Also, callbacks hold strong references to the JS closure — if the native operation is long-lived, this prevents garbage collection of the closure's captured scope.

7. Promise-Based Methods

Promises are the preferred pattern for async native methods. The last two parameters must be the resolver and rejecter.

// JS side — promise-based (preferred)
async function loadUserData() {
  try {
    const data = await NativeModules.UserModule.getUserProfile('user_123');
    return JSON.parse(data);
  } catch (error) {
    // error.code  → 'E_NOT_FOUND'
    // error.message → 'User not found'
    // error.nativeStackIOS / error.nativeStackAndroid
    handleError(error);
  }
}

// iOS — promise method signature
RCT_EXPORT_METHOD(getUserProfile:(NSString *)userId
                  resolver:(RCTPromiseResolveBlock)resolve
                  rejecter:(RCTPromiseRejectBlock)reject)
{
  [self.db fetchUser:userId completion:^(User *user, NSError *error) {
    if (error) {
      reject(@"E_NOT_FOUND", error.localizedDescription, error);
    } else {
      resolve([user toJSON]);
    }
  }];
}

// Android — promise method signature
@ReactMethod
public void getUserProfile(String userId, Promise promise) {
  db.fetchUser(userId, new Callback<User>() {
    @Override
    public void onSuccess(User user) {
      WritableMap map = Arguments.createMap();
      map.putString("id", user.getId());
      map.putString("name", user.getName());
      promise.resolve(map);
    }
    @Override
    public void onError(Exception e) {
      promise.reject("E_NOT_FOUND", e.getMessage(), e);
    }
  });
}

8. Constants Export

Constants are synchronously available on the JS side at module load time. They are evaluated once during module initialization and frozen.

// JS side — accessing constants
import { NativeModules, Platform } from 'react-native';

const { DeviceInfo } = NativeModules;

console.log(DeviceInfo.MODEL);        // 'iPhone 15 Pro'
console.log(DeviceInfo.OS_VERSION);    // '17.2'
console.log(DeviceInfo.APP_VERSION);   // '2.1.0'
console.log(DeviceInfo.IS_TABLET);     // false

// Constants are frozen — cannot be modified at runtime
// DeviceInfo.MODEL = 'hacked'; // silently fails
Performance Note

Constants are serialized and sent across the bridge during module initialization. Large constant objects (e.g., feature flag maps with hundreds of entries) add to startup time. For large datasets, prefer a promise-based method that loads on demand.

9. Event Emitters

Event emitters allow native code to push data to JavaScript without a JS-initiated call. This is essential for hardware events, background tasks, and real-time updates.

// iOS — Event Emitter
@implementation LocationModule

- (NSArray<NSString *> *)supportedEvents {
  return @[@"onLocationUpdate", @"onLocationError"];
}

- (void)locationManager:(CLLocationManager *)manager
     didUpdateLocations:(NSArray<CLLocation *> *)locations {
  CLLocation *loc = locations.lastObject;
  [self sendEventWithName:@"onLocationUpdate"
                     body:@{
    @"latitude": @(loc.coordinate.latitude),
    @"longitude": @(loc.coordinate.longitude),
    @"accuracy": @(loc.horizontalAccuracy),
    @"timestamp": @(loc.timestamp.timeIntervalSince1970)
  }];
}

@end

// Android — Event Emitter
private void sendLocationUpdate(Location location) {
  WritableMap params = Arguments.createMap();
  params.putDouble("latitude", location.getLatitude());
  params.putDouble("longitude", location.getLongitude());
  params.putDouble("accuracy", location.getAccuracy());

  getReactApplicationContext()
    .getJSModule(DeviceEventManagerModule.RCTDeviceEventEmitter.class)
    .emit("onLocationUpdate", params);
}

// JS side — subscribing to events
import { NativeEventEmitter, NativeModules } from 'react-native';

const locationEmitter = new NativeEventEmitter(NativeModules.LocationModule);

const subscription = locationEmitter.addListener(
  'onLocationUpdate',
  (event) => {
    console.log(`Lat: ${event.latitude}, Lng: ${event.longitude}`);
  }
);

// CRITICAL: Remove listener to prevent memory leaks
subscription.remove();

10. Threading Annotations

Understanding which thread native module methods execute on is critical for correctness and performance.

Threading Model for Native Modules ═══════════════════════════════════ iOS Default Behavior: ───────────────────── • Methods run on a dedicated background serial GCD queue • Override methodQueue to change: - dispatch_get_main_queue() → UI thread - custom serial queue → dedicated queue - custom concurrent queue → parallel execution Android Default Behavior: ───────────────────────── • @ReactMethod runs on NativeModules thread (AsyncTask thread pool) • @ReactMethod(isBlockingSynchronousMethod = true) → runs on JS thread (BLOCKS JavaScript execution) • @UiThread annotation → runs on Android UI thread Thread Safety Requirements: ─────────────────────────── • Multiple JS calls can arrive concurrently on Android • iOS serial queue provides implicit synchronization • Shared mutable state MUST be synchronized 
// Android — synchronous method (blocks JS thread!)
@ReactMethod(isBlockingSynchronousMethod = true)
public String getInstallationId() {
  // This runs ON the JS thread — must be fast (<1ms)
  // Use only for reading cached/in-memory values
  return cachedInstallId;
}

// iOS — specifying method queue
- (dispatch_queue_t)methodQueue {
  // All methods in this module run on this queue
  return dispatch_queue_create(
    "com.myapp.crypto",
    DISPATCH_QUEUE_SERIAL
  );
}

// iOS — requiring main queue setup
+ (BOOL)requiresMainQueueSetup {
  return YES; // init touches UIKit — must run on main thread
}
Staff-Level Threading Knowledge

isBlockingSynchronousMethod = true is a powerful but dangerous annotation. It bypasses the bridge's async queue and executes the method synchronously on the JS thread via JNI. If the method takes >16ms, you drop frames. If it blocks on I/O, you freeze the entire JS runtime. Use it only for reading pre-computed in-memory values (e.g., cached device IDs, feature flags). In the New Architecture, TurboModules provide synchronous methods safely via JSI.

11. Module Lifecycle

// iOS — lifecycle methods
@implementation MyModule

- (instancetype)init {
  self = [super init];
  // Called during module registration
  // Bridge is NOT yet available here
  return self;
}

- (void)setBridge:(RCTBridge *)bridge {
  // Bridge is now available — safe to access other modules
  [super setBridge:bridge];
}

- (void)invalidate {
  // Called when bridge is torn down (app background, reload)
  // Clean up timers, observers, file handles
  [self.locationManager stopUpdatingLocation];
  [self.timer invalidate];
}

@end

// Android — lifecycle methods
public class MyModule
    extends ReactContextBaseJavaModule
    implements LifecycleEventListener {

  @Override
  public void initialize() {
    // Called after module creation, context is available
    getReactApplicationContext().addLifecycleEventListener(this);
  }

  @Override
  public void onHostResume() {
    // Activity resumed — start sensors, reconnect
  }

  @Override
  public void onHostPause() {
    // Activity paused — pause sensors, save state
  }

  @Override
  public void onHostDestroy() {
    // Activity destroyed — full cleanup
  }

  @Override
  public void invalidate() {
    // Bridge torn down — release all resources
    getReactApplicationContext()
      .removeLifecycleEventListener(this);
  }
}

Staff-Level Mental Model

Legacy native modules are the foundation upon which all of React Native's platform integration is built. Even with TurboModules and Expo Modules, the core concepts — registration, method export, threading, callbacks/promises, events, and lifecycle — remain identical. The difference is in the transport layer: bridge (JSON serialization) vs JSI (direct C++ calls). Master the concepts here, and TurboModules become a straightforward optimization rather than a paradigm shift.
