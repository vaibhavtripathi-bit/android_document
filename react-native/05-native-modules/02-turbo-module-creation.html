TurboModule Creation

TurboModules replace the legacy bridge-based NativeModules with direct JSI (JavaScript Interface) calls. They eliminate JSON serialization overhead, enable synchronous method invocation, support lazy initialization, and provide type-safe codegen from TypeScript/Flow specs.

1. TurboModule Architecture

Architect Insight

TurboModules are backed by JSI HostObject instances. When JS accesses global.__turboModuleProxy, it triggers a C++ get() call that lazily instantiates the native module. Method calls go directly through C++ virtual dispatch — no serialization, no queuing, no batching. This is why TurboModules can support synchronous return values, which was impossible with the bridge.

2. Spec Files (TypeScript/Flow)

Every TurboModule starts with a spec file — a TypeScript or Flow interface that defines the module's API contract. The codegen reads this spec to generate platform-specific boilerplate.

// specs/NativeCalculator.ts
import type { TurboModule } from 'react-native';
import { TurboModuleRegistry } from 'react-native';

export interface Spec extends TurboModule {
  // Synchronous method — returns value directly
  add(a: number, b: number): number;

  // Asynchronous method — returns Promise
  computeAsync(expression: string): Promise<number>;

  // Method with complex types
  processData(input: {
    values: number[];
    operation: string;
    precision?: number;
  }): Promise<{
    result: number;
    metadata: string;
  }>;

  // Constants
  getConstants(): {
    PI: number;
    E: number;
    MAX_PRECISION: number;
  };

  // Callback-based method
  subscribe(eventName: string, callback: (data: string) => void): void;
}

export default TurboModuleRegistry.getEnforcing<Spec>(
  'NativeCalculator'
);
Spec File Naming Convention

The file must be named Native<ModuleName>.ts (or .js for Flow). The Native prefix is required by codegen. The module name passed to TurboModuleRegistry.getEnforcing() must match exactly. Using getEnforcing throws if the module isn't found; use get() for optional modules that return null.

// NativeCalculator.h
#import <NativeCalculatorSpecJSI.h>

@interface NativeCalculator : NSObject <NativeCalculatorSpec>
@end

// NativeCalculator.mm (ObjC++ — note the .mm extension)
#import "NativeCalculator.h"
#import <React/RCTLog.h>

@implementation NativeCalculator

RCT_EXPORT_MODULE()

// Synchronous — called directly from JS thread via JSI
- (NSNumber *)add:(double)a b:(double)b {
  return @(a + b);
}

// Asynchronous — returns via promise
- (void)computeAsync:(NSString *)expression
               resolve:(RCTPromiseResolveBlock)resolve
                reject:(RCTPromiseRejectBlock)reject {
  dispatch_async(dispatch_get_global_queue(QOS_CLASS_USER_INITIATED, 0), ^{
    @try {
      NSExpression *expr = [NSExpression expressionWithFormat:expression];
      NSNumber *result = [expr expressionValueWithObject:nil context:nil];
      resolve(result);
    } @catch (NSException *e) {
      reject(@"E_EVAL", e.reason, nil);
    }
  });
}

// Complex types — codegen generates proper ObjC++ types
- (void)processData:(JS::NativeCalculator::ProcessDataInput &)input
             resolve:(RCTPromiseResolveBlock)resolve
              reject:(RCTPromiseRejectBlock)reject {
  NSArray *values = input.values();
  NSString *operation = input.operation();
  std::optional<double> precision = input.precision();
  // ... process and resolve
}

- (facebook::react::ModuleConstants<
    JS::NativeCalculator::Constants::Builder>)getConstants {
  return facebook::react::typedConstants<
    JS::NativeCalculator::Constants::Builder>({
    .PI = M_PI,
    .E = M_E,
    .MAX_PRECISION = 15
  });
}

- (std::shared_ptr<facebook::react::TurboModule>)
    getTurboModule:(const facebook::react::ObjCTurboModule::InitParams &)params {

5. Android Implementation (Java/Kotlin)

// NativeCalculatorModule.kt
package com.myapp.calculator

import com.facebook.react.bridge.*
import com.myapp.calculator.NativeCalculatorSpec

class NativeCalculatorModule(
  reactContext: ReactApplicationContext
) : NativeCalculatorSpec(reactContext) {

  override fun getName(): String = "NativeCalculator"

  // Synchronous — runs on JS thread via JNI
  override fun add(a: Double, b: Double): Double = a + b

  // Asynchronous — runs on native modules thread
  override fun computeAsync(expression: String, promise: Promise) {
    try {
      val result = evaluateExpression(expression)
      promise.resolve(result)
    } catch (e: Exception) {
      promise.reject("E_EVAL", e.message, e)
    }
  }

  // Complex types — codegen generates ReadableMap/WritableMap handling
  override fun processData(input: ReadableMap, promise: Promise) {
    val values = input.getArray("values")
    val operation = input.getString("operation")
    val precision = if (input.hasKey("precision"))
      input.getDouble("precision").toInt() else 10

    val result = Arguments.createMap().apply {
      putDouble("result", computedResult)
      putString("metadata", "computed")
    }
    promise.resolve(result)
  }

  override fun getTypedExportedConstants(): Map<String, Any> = mapOf(
    "PI" to Math.PI,
    "E" to Math.E,
    "MAX_PRECISION" to 15
  )
}

// NativeCalculatorPackage.kt — registration
class NativeCalculatorPackage : TurboReactPackage() {
  override fun getModule(
    name: String,
    reactContext: ReactApplicationContext
  ): NativeModule? {
    return when (name) {
      NativeCalculatorModule.NAME -> NativeCalculatorModule(reactContext)
      else -> null
    }
  }

  override fun getReactModuleInfoProvider() =
    ReactModuleInfoProvider {
      mapOf(
        NativeCalculatorModule.NAME to ReactModuleInfo(
          NativeCalculatorModule.NAME,
          NativeCalculatorModule::class.java.name,
          false, // canOverrideExistingModule
          false, // needsEagerInit
          true,  // hasConstants
          false, // isCxxModule
          true   // isTurboModule
        )
      )
    }
}

7. Synchronous Methods

TurboModules can return values synchronously because JSI calls are direct C++ function invocations — no bridge, no serialization, no async dispatch.

// Spec — synchronous method (no Promise return type)
export interface Spec extends TurboModule {
  getDeviceId(): string;           // sync — returns immediately
  isFeatureEnabled(key: string): boolean;  // sync
  fetchData(): Promise<string>;    // async — returns Promise
}

// JS usage — sync methods return directly, no await needed
const deviceId = NativeCalculator.getDeviceId();  // string, not Promise
const enabled = NativeCalculator.isFeatureEnabled('dark_mode');

// Async methods still need await
const data = await NativeCalculator.fetchData();
Sync Method Performance

Synchronous TurboModule methods execute on the JS thread. The call overhead is ~10μs (vs ~1-5ms for bridge calls). However, the method body still runs on the JS thread, so expensive computation will block JavaScript. Use sync methods for: reading cached values, simple computations, and state checks. Use async methods for: I/O, heavy computation, and anything that might take >1ms.

9. Lazy Initialization

Unlike legacy NativeModules (all loaded at startup), TurboModules are instantiated on first access via __turboModuleProxy.

Lazy Initialization Flow ═════════════════════════ App Start │ ├── Legacy: ALL NativeModules instantiated ──► Slow startup │ └── TurboModules: Only proxy installed │ ├── JS accesses NativeCalculator │ │ │ ▼ │ __turboModuleProxy("NativeCalculator") │ │ │ ▼ │ C++ TurboModuleManager.getModule() │ │ │ ▼ │ Platform module factory creates instance │ │ │ ▼ │ Module cached for subsequent calls │ └── Unused modules: NEVER instantiated ──► Fast startup

10. Testing TurboModules

// __mocks__/NativeCalculator.ts — Jest mock
const NativeCalculator = {
  add: jest.fn((a: number, b: number) => a + b),
  computeAsync: jest.fn((expr: string) =>
    Promise.resolve(eval(expr))
  ),
  processData: jest.fn((input) =>
    Promise.resolve({ result: 0, metadata: 'mock' })
  ),
  getConstants: jest.fn(() => ({
    PI: 3.14159,
    E: 2.71828,
    MAX_PRECISION: 15,
  })),
};

export default NativeCalculator;

// __tests__/Calculator.test.ts
import NativeCalculator from '../specs/NativeCalculator';

jest.mock('../specs/NativeCalculator');

describe('Calculator integration', () => {
  it('calls native add method', () => {
    const result = NativeCalculator.add(2, 3);
    expect(result).toBe(5);
    expect(NativeCalculator.add).toHaveBeenCalledWith(2, 3);
  });

  it('handles async computation', async () => {
    const result = await NativeCalculator.computeAsync('2 + 3');
    expect(result).toBe(5);
  });
});

11. Migration from NativeModules

1. API Definition Implicit (native code defines API) Explicit spec file (Native*.ts)

Migration Strategy

React Native supports interop mode where legacy NativeModules work alongside TurboModules. You don't need to migrate everything at once. Start with high-traffic modules (called frequently) to get the biggest performance wins. Modules called once at startup benefit less from the migration.

Staff-Level Mental Model

TurboModules are fundamentally a transport optimization: replacing JSON-over-bridge with JSI-over-C++. The programming model (methods, promises, events, constants) stays the same. The key architectural wins are: (1) type safety via codegen, (2) lazy loading for startup performance, (3) synchronous methods for latency-sensitive paths, and (4) a foundation for future optimizations like shared memory and zero-copy data transfer.
