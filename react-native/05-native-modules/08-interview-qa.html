Native Modules — Interview Q&A

55+ Staff-level questions covering native module creation, TurboModules, native UI components, platform APIs, brownfield integration, C++ modules, and Expo Modules API.

Native Module Basics

Q1: How does RCT_EXPORT_MODULE() work under the hood on iOS?

It uses the GCC __attribute__((constructor)) extension to register a function that runs before main(). This function adds the module class to a global registry (RCTModuleClasses). At bridge initialization, RN iterates this registry to instantiate all modules. The macro expands to a +load-time registration — zero runtime discovery cost but contributes to binary startup time.

Q2: What is the difference between RCTBubblingEventBlock and RCTDirectEventBlock?

RCTBubblingEventBlock dispatches events that propagate up the React component tree (like DOM bubbling). RCTDirectEventBlock sends events only to the target component. Use bubbling for interaction events (touch, scroll) where parents might intercept; use direct for component-specific notifications (onLoad, onError). On Android, the distinction maps to getExportedCustomBubblingEventTypeConstants vs getExportedCustomDirectEventTypeConstants.

Q3: Why can callbacks only be invoked once in legacy native modules?

The bridge tracks callback IDs in a one-time-use map. When a callback is invoked, its ID is removed from the map and the corresponding JS function reference is released. Invoking the same ID again causes a crash because the reference no longer exists. This is a design choice to prevent memory leaks — without this constraint, native code could hold JS function references indefinitely.

Q4: What thread do @ReactMethod methods run on by default on Android?

They run on the NativeModules thread, which is a background thread from the AsyncTask thread pool. This is different from the JS thread and the UI thread. Multiple @ReactMethod calls can execute concurrently, so shared mutable state must be synchronized. Use @UiThread for methods that need the main thread, or isBlockingSynchronousMethod = true to run on the JS thread (dangerous — blocks JS execution).

Q5: How do you handle the case where a native module method is called after the bridge is invalidated?

Check self.bridge (iOS) or getReactApplicationContext().hasActiveReactInstance() (Android) before performing operations. Implement the invalidate method to set a flag and cancel pending operations. For event emitters, guard sendEvent calls with a validity check. In production, wrap all bridge interactions in try-catch to prevent crashes during teardown.

Q6: What is requiresMainQueueSetup and when should it return YES?

It tells RN whether the module's init method must run on the main thread. Return YES if initialization accesses UIKit (creating views, accessing UIScreen, etc.) or any API that requires the main thread. Return NO for modules that only do computation or I/O in their init. Returning NO when it should be YES causes crashes; returning YES unnecessarily slows startup.

Q7: How does the bridge batch native module calls?

The bridge batches JS-to-native calls on a 5ms timer. When JS calls a native method, the call is queued. Every 5ms (or when the queue reaches a size threshold), the entire batch is serialized as a single JSON message and dispatched to native. This amortizes serialization overhead but adds up to 5ms latency to every call. TurboModules eliminate this by using direct JSI calls with no batching.

Q8: How do you export constants that change at runtime?

You can't — getConstants() / constantsToExport is called once during module initialization and the result is frozen. For values that change, use a promise-based method (getCurrentValue()) or an event emitter that pushes updates. A common pattern is to export initial values as constants and use events for subsequent changes.

TurboModules

Q9: How do TurboModules eliminate bridge overhead?

TurboModules use JSI (JavaScript Interface) — a C++ API that allows JS to call C++ functions directly. Instead of serializing method calls to JSON and dispatching through an async message queue, JSI provides direct virtual function dispatch. The call path is: JS → JSI (C++ vtable lookup) → native method. This eliminates serialization (~1ms), queuing (~5ms), and deserialization (~1ms), reducing per-call overhead from ~2-7ms to ~10μs.

Q10: What is __turboModuleProxy and how does lazy loading work?

__turboModuleProxy is a global JSI function installed at runtime startup. When JS accesses a TurboModule for the first time, it calls this proxy with the module name. The proxy delegates to TurboModuleManager in C++, which calls the platform-specific factory to create the module instance. The instance is cached for subsequent accesses. Modules never accessed are never instantiated — this is why TurboModules improve startup time.

Q11: Why must TurboModule spec files be named with a Native prefix?

The codegen tool scans for files matching Native*.ts (or .js for Flow). This naming convention is hardcoded in the codegen configuration. The prefix serves as a signal that the file defines a native module interface (not a regular TypeScript file). The module name in TurboModuleRegistry.get('ModuleName') must match the name used in native registration.

Q12: How do synchronous TurboModule methods work without blocking?

They do block — but only the JS thread, and for microseconds instead of milliseconds. A sync TurboModule method executes on the JS thread via direct C++ function call. The JS thread is blocked for the duration of the method body. This is acceptable for fast operations (reading cached values, simple math) but dangerous for I/O or heavy computation. The key insight is that the blocking time is the method execution time itself, not the transport overhead.

Q13: What is the codegen pipeline and what does it generate?

Codegen reads TypeScript/Flow spec files and generates: (1) C++ JSI interface headers with type-safe method signatures, (2) iOS ObjC++ protocol conformance that the implementation must satisfy, (3) Android Java/Kotlin abstract base class that the implementation extends. This ensures type safety across the JS-native boundary at compile time rather than runtime.

Q14: How do you migrate a legacy NativeModule to a TurboModule incrementally?

React Native supports interop mode where both systems coexist. Steps: (1) Create a spec file matching the existing API, (2) Run codegen to generate platform boilerplate, (3) Implement the generated interface, delegating to existing logic, (4) Change JS imports from NativeModules.X to TurboModuleRegistry.get('X'), (5) Update Android package from ReactPackage to TurboReactPackage. The native implementation can be migrated method by method.

Native UI Components

Q15: What is the difference between SimpleViewManager and ViewGroupManager on Android?

SimpleViewManager is for leaf views (no children) — it extends BaseViewManager with a simpler API. ViewGroupManager is for container views that can host React children — it handles child view addition, removal, and reordering. Use SimpleViewManager for components like progress bars, images, and charts. Use ViewGroupManager for components like custom scroll views or layout containers.

Q16: How do direct commands differ from prop updates for native views?

Props are declarative — React sets them during reconciliation, and the view updates accordingly. Commands are imperative — JS explicitly invokes an action (scrollTo, play, reset) that doesn't map to a prop change. Props trigger on every render that changes them; commands trigger exactly when called. Use props for state that should be reflected in the view; use commands for one-shot actions.

Q17: How does Fabric change native component creation compared to the legacy renderer?

Legacy: ViewManager creates platform views, UIManager dispatches prop updates via bridge. Fabric: ComponentDescriptor + ShadowNode compute layout in C++ (Yoga), then platform ComponentView receives pre-computed layout. Key differences: (1) layout happens in C++ not on the platform thread, (2) props are C++ structs not JSON, (3) view creation is driven by the C++ renderer not the JS bridge, (4) concurrent rendering is possible because ShadowNodes are immutable.

Q18: How do you implement custom measurement for a native component?

Override YogaMeasureFunction (Android) or provide a measureFunction in the ShadowNode (Fabric). The measure function receives constraints (width/height with modes: EXACTLY, AT_MOST, UNDEFINED) and returns the component's desired size. This is called during Yoga layout when the component has no explicit dimensions. Common for text-like components that need to measure content to determine size.

Q19: What is onAfterUpdateTransaction and why is it important?

On Android, props are set individually via separate @ReactProp setter calls. onAfterUpdateTransaction is called after ALL props in a single update batch have been applied. This is where you should trigger expensive operations that depend on multiple props (e.g., re-rendering a chart that depends on both data and color). Without this, you'd re-render once per prop change instead of once per batch.

Platform APIs

Q20: Why can iOS permission requests only show the system dialog once?

Apple's privacy design: the system dialog is shown only on the first request() call. If the user denies, subsequent requests return BLOCKED without any UI. This forces apps to: (1) show a pre-permission rationale screen explaining why the permission is needed, (2) only request when the user is likely to accept, (3) provide a path to Settings for users who denied. On Android 11+, denying twice also permanently blocks the permission.

Q21: How do you handle background location tracking across iOS and Android?

iOS: Enable UIBackgroundModes: location in Info.plist, use CLLocationManager with allowsBackgroundLocationUpdates = true, request "Always" permission. Android: Use a foreground service with FOREGROUND_SERVICE_LOCATION type, show a persistent notification, request ACCESS_BACKGROUND_LOCATION (separate from foreground). Both platforms throttle background location updates to save battery.

Q22: What is the difference between cryptographic and simple biometric authentication?

Simple: simplePrompt() — shows biometric dialog, returns success/failure boolean. No cryptographic proof. Vulnerable to rooted/jailbroken device spoofing. Cryptographic: createSignature() — generates an RSA key pair stored in the Secure Enclave (iOS) / Keystore (Android), biometric auth unlocks the private key to sign a payload. The server verifies the signature, providing proof that the specific device's biometric was used.

Q23: How do you design a graceful degradation strategy for platform capabilities?

Build a capability detection layer at startup that maps: (device capability) × (OS version) × (permission state) × (feature flag) → available features. Gate UI elements declaratively based on this map. Provide fallback UX for every gated feature (e.g., password auth when biometrics unavailable, manual entry when camera unavailable). Never crash on a missing capability — always have a code path for the "not available" case.

Q24: How does push notification handling differ between foreground and background?

Foreground: onMessage callback fires in JS, you must manually display a local notification (the OS doesn't show remote notifications when the app is foregrounded). Background: setBackgroundMessageHandler runs in a headless JS context — no UI access, limited execution time (~30s iOS, ~10min Android). The OS displays the notification automatically. Killed state: the notification is displayed by the OS; tapping it launches the app with the notification data in launch options.

Brownfield Integration

Q25: Should you use a single shared bridge or multiple isolated bridges in a brownfield app?

Single bridge (recommended for most cases): lower memory (~30-50MB saved per additional bridge), shared JS state (Redux, context), shared native modules. Multiple bridges: crash isolation (one surface crashing doesn't kill others), independent loading, useful for completely independent features. The trade-off is memory vs isolation. Most brownfield apps use a single bridge with multiple AppRegistry.registerComponent() entries.

Q26: How do you handle navigation between native screens and React Native screens?

Create a NavigationBridge native module that exposes: pushNativeScreen(name, params), popToNative(), presentRNModal(moduleName, props). On the native side, map screen names to native view controllers/activities. On the RN side, use the bridge to trigger native navigation. The key challenge is back-button handling — you need to intercept the hardware back button (Android) and swipe-back gesture (iOS) to pop the correct stack.

Q27: What is the pre-warming strategy for React Native in brownfield apps?

Initialize the RN bridge on a background thread shortly after native app launch, before the user navigates to an RN screen. This hides the ~200-500ms bridge initialization time. The bridge loads the JS bundle and initializes modules in the background. When the user opens an RN screen, you create an RCTRootView using the pre-warmed bridge — the view appears nearly instantly. Trade-off: ~30-50MB memory used even if the user never opens an RN screen.

Q28: How do you share authentication state between native and React Native?

Use a shared storage layer (MMKV or Keychain/Keystore) that both native and RN code can read/write. Create a SharedAuth native module that wraps the existing native auth manager. On login (either side), write the token to shared storage and emit an event. On the RN side, subscribe to auth state changes. On the native side, observe the shared storage. MMKV is ideal because it's memory-mapped and changes are visible immediately across both runtimes.

Q29: What are the biggest risks of brownfield integration?

(1) Binary size increase (~7-15MB for RN runtime), (2) build time increase (Metro bundler + native compilation), (3) crash surface area (bridge crashes can take down the whole app), (4) memory overhead (~30-50MB for the JS runtime), (5) team coordination (native and RN developers must agree on bridge APIs), (6) version coupling (RN upgrades may require native code changes), (7) debugging complexity (stack traces span JS and native).

C++ TurboModules

Q30: Why would you choose a C++ TurboModule over a platform-specific one?

When you need: (1) cross-platform code sharing — write once, run on both iOS and Android, (2) maximum performance — C++ TurboModules have ~10x less overhead than platform-specific ones because they skip ObjC message dispatch / JNI calls, (3) integration with C++ libraries (SQLite, OpenSSL, protobuf), (4) compute-intensive operations (crypto, image processing, ML inference). The trade-off is C++ complexity and harder debugging.

Q31: How does JSI HostObject enable exposing C++ classes to JavaScript?

jsi::HostObject is a C++ class with virtual get() and set() methods. When JS accesses a property on the HostObject proxy, JSI calls get() with the property name. You return jsi::Function for methods, jsi::Value for properties. The HostObject lives on the C++ heap, and its destructor is called when the JS garbage collector collects the proxy object. This enables natural JS API design backed by C++ objects.

Q32: What are the memory management pitfalls when mixing JS GC and C++ ownership?

(1) Never store jsi::Value beyond the current function call — it's invalidated when the call returns. (2) Use shared_ptr for C++ objects referenced by HostObjects — the HostObject destructor (called by JS GC) releases the shared_ptr. (3) Use weak_ptr to break cycles between C++ objects and HostObjects. (4) Never capture raw pointers to JS runtime objects. (5) Be aware that JS GC timing is non-deterministic — C++ destructors may run at any time.

Q33: How do you handle errors across the C++/JS boundary?

Throw jsi::JSError(runtime, message) from C++ — this becomes a JavaScript Error object that can be caught with try/catch. Always wrap C++ code in try/catch and convert C++ exceptions to jsi::JSError. Uncaught C++ exceptions (like std::bad_alloc) will crash the app. Use a catch-all (catch (...)) as a safety net to convert unknown exceptions to JS errors.

Q34: How do you integrate a third-party C++ library (e.g., OpenSSL) into a React Native project?

Android: Add the library to CMakeLists.txt with find_library or include pre-built binaries. iOS: Add the library via CocoaPods (pod 'OpenSSL-Universal') or as a static library in the Xcode project. Both platforms: create a C++ wrapper that abstracts the library API, then expose it via a C++ TurboModule. The wrapper isolates your code from library version changes and provides a clean JSI-compatible interface.

Expo Modules

Q35: How does the Expo Modules Swift/Kotlin DSL work under the hood?

The DSL uses builder pattern closures. Module { } creates a ModuleDefinition object. Function("name") { } registers a closure with the module's method map. At runtime, expo-modules-core installs JSI HostObjects that delegate to these closures. Type conversion (Swift/Kotlin types ↔ JSI types) is handled automatically by the core library using reflection and generic type mapping.

Q36: What is the difference between Function and AsyncFunction in Expo Modules?

Function executes synchronously on the JS thread and returns a value directly. AsyncFunction dispatches to a background thread (iOS: GCD queue, Android: coroutine dispatcher) and returns a Promise. Use Function for fast operations (<1ms) like reading cached values. Use AsyncFunction for I/O, network calls, or any operation that might take more than a few milliseconds.

Q37: What are config plugins and why are they necessary?

Config plugins are JavaScript functions that modify native project files (Info.plist, AndroidManifest.xml, build.gradle, Podfile, entitlements) at prebuild time. They're necessary because the managed workflow generates native directories from app.json — there's no persistent native project to edit manually. Plugins enable: adding permissions, modifying build settings, adding native dependencies, and configuring third-party SDKs — all from JavaScript.

Q38: Why can't custom native modules run in Expo Go?

Expo Go is a pre-compiled app distributed through the App Store / Play Store. It contains a fixed set of native modules compiled at Expo's build time. Your custom native code isn't compiled into this binary. Development builds solve this by creating a custom Expo Go-like app that includes your native modules. This is why npx expo run:ios or EAS Build with developmentClient: true is required for custom native code.

Q39: How does the Expo prebuild system work?

npx expo prebuild reads app.json, applies all config plugins, and generates the ios/ and android/ directories. These directories are complete native projects (Xcode project, Gradle project) with all dependencies, permissions, and configurations applied. They can be .gitignored because they're fully reproducible from the config. Running prebuild again regenerates them, incorporating any config changes.

Cross-Cutting Concerns

Q40: Compare the performance characteristics of bridge calls, TurboModule calls, and C++ TurboModule calls.

Bridge: ~1-5ms per call (JSON serialization + async queue + deserialization), batched every 5ms. Platform TurboModule: ~50-150μs per call (JSI → ObjC message dispatch or JNI call). C++ TurboModule: ~10μs per call (JSI → C++ virtual dispatch, no platform layer). For a module called 100 times per second: bridge = 100-500ms overhead, platform TurboModule = 5-15ms, C++ TurboModule = 1ms.

Q41: How do you decide between creating a native module vs using an existing library?

Use an existing library when: it's well-maintained (recent commits, active issues), covers your requirements, has acceptable performance, and the team doesn't need deep customization. Create a custom module when: no library exists, existing libraries have unacceptable performance, you need tight integration with existing native code (brownfield), or you need to control the entire stack for security/compliance reasons. At Staff level, also consider: maintenance burden, team expertise, and upgrade path.

Q42: What is the role of CallInvoker in TurboModules?

CallInvoker provides a way to schedule work on the JS thread from native code. When a native async operation completes, you can't directly call JS functions (wrong thread). CallInvoker::invokeAsync() queues a function to run on the JS thread. This replaces the bridge's event dispatch mechanism with a direct, type-safe callback mechanism. It's essential for promise resolution and event emission in TurboModules.

Q43: How do you handle native module versioning in a monorepo with multiple apps?

Treat native modules as internal packages with semantic versioning. Define the JS API in a spec file (TurboModule) or TypeScript interface (Expo Module). Version the spec separately from the implementation. Use a compatibility matrix that maps spec versions to native implementation versions. In CI, run integration tests that verify the JS API works with the current native implementation on both platforms.

Q44: What happens when a TurboModule spec defines a method that the native implementation doesn't provide?

With codegen, this is a compile-time error — the generated protocol (iOS) or abstract class (Android) requires all methods to be implemented. Without codegen (manual TurboModule), it's a runtime crash when JS calls the missing method. This is one of the key advantages of the codegen approach: API mismatches are caught at build time, not in production.

Q45: How do you test native modules in a CI environment without physical devices?

Unit tests: mock the native module in Jest (jest.mock()) for JS-side logic. Integration tests: use iOS Simulator / Android Emulator in CI (GitHub Actions, CircleCI). C++ modules: use Google Test for pure C++ logic without any RN dependency. Platform tests: XCTest (iOS) and JUnit (Android) for native implementation logic. E2E: Detox or Maestro running on emulators. The key is layering: test C++ logic independently, native logic independently, and JS integration with mocks.

Q46: What is the interop layer and how does it enable gradual migration to the New Architecture?

The interop layer allows legacy NativeModules to work alongside TurboModules in the same app. It wraps legacy modules with a JSI-compatible interface so they can be accessed through __turboModuleProxy. This means you don't need to migrate all modules at once — the bridge and JSI coexist. The interop layer adds a small overhead compared to native TurboModules but is significantly better than the pure bridge path.

Q47: How do you handle platform-specific native module behavior in a cross-platform codebase?

Define a common TypeScript interface that both platforms implement. Use Platform.select() or .ios.ts / .android.ts file extensions for platform-specific JS wrappers. In the native implementation, each platform implements the same spec but with platform-appropriate APIs. For features only available on one platform, use optional methods in the spec (getFeature?(): string) and check availability at runtime.

Q48: What are the security implications of exposing native APIs to JavaScript?

Every native module method is callable from JS — including from any JS code injected via XSS or supply chain attacks. Mitigations: (1) validate all inputs in native code (don't trust JS-provided data), (2) don't expose sensitive operations without authentication (e.g., require biometric auth before signing), (3) use code signing to verify bundle integrity, (4) minimize the API surface — only expose what's needed, (5) never expose raw file system access or shell execution.

Q49: How does the Fabric renderer change the way native UI components handle layout?

In legacy, layout is computed on the JS thread (Yoga in JS), then applied to native views via the bridge. In Fabric, layout is computed in C++ (Yoga in C++), producing a ShadowTree with pre-computed frames. The C++ renderer then creates/updates platform views with these frames. This means: (1) layout doesn't block the JS thread, (2) layout can happen concurrently with JS execution, (3) custom measurement functions are C++ instead of Java/ObjC.

Q50: What is the recommended approach for a new React Native project that needs custom native modules?

Use Expo with development builds and the Expo Modules API. Create modules using the Swift/Kotlin DSL for minimal boilerplate. Use config plugins for native project modifications. Use EAS Build for CI/CD. This gives you: (1) the simplest native module API, (2) reproducible native builds via prebuild, (3) OTA updates via EAS Update, (4) the full power of native code without managing Xcode/Gradle manually. Fall back to bare workflow only if Expo's abstractions are insufficient.

Q51: How do you profile the performance of a native module call?

iOS: Use Xcode Instruments (Time Profiler) to measure native method execution time. Android: Use Android Studio Profiler or Systrace. For bridge overhead: add timestamps before/after the JS call and in the native method to measure round-trip time. For TurboModules: use the Hermes profiler to see JSI call overhead. For C++ modules: use std::chrono for microsecond-precision timing. Compare bridge vs TurboModule overhead by implementing the same method both ways.

Q52: What is the maximum data size you should send through the bridge vs TurboModules?

Bridge: JSON serialization becomes expensive above ~100KB. At 1MB, expect ~10-50ms serialization time. Above 10MB, the bridge can cause ANR/freeze. TurboModules (JSI): no serialization, but large jsi::String creation still copies data. For large data: use shared memory (C++ TurboModule with ArrayBuffer), file-based transfer (write to file in native, read path in JS), or streaming (event emitter with chunks).

Q53: How do you handle native module crashes gracefully in production?

Wrap all native module methods in try-catch (ObjC: @try/@catch, Java: try/catch, C++: try/catch). Convert native exceptions to promise rejections or error events — never let them propagate to crash the app. Use crash reporting (Sentry, Crashlytics) with native symbolication. For critical modules, implement a fallback path in JS that activates when the native module fails. Consider using error boundaries in React to catch and recover from native view crashes.

Q54: What is the future direction of native modules in React Native?

The trajectory is: (1) TurboModules become the default (bridge deprecated), (2) Codegen becomes mandatory for type safety, (3) C++ TurboModules for cross-platform performance-critical code, (4) Expo Modules API as the recommended high-level API, (5) Static Hermes enabling ahead-of-time compilation for even faster JSI calls, (6) Shared C++ core (Yoga, Fabric, TurboModules) with thin platform bindings. The long-term vision is a C++ runtime with JSI as the universal interface.

Q55: Design a native module architecture for a large-scale app with 50+ native modules.

Layer the modules: (1) Core layer — C++ TurboModules for cross-platform logic (crypto, storage, networking), (2) Platform layer — platform-specific TurboModules for OS APIs (camera, biometrics, notifications), (3) Feature layer — Expo Modules for feature-specific native code. Use a module registry with dependency injection. Implement lazy loading for all modules. Group modules into packages by feature domain. Define TypeScript specs for all modules (type safety). Use a monorepo with shared codegen configuration. Monitor module initialization time in production telemetry.
