Building a Multi-App Platform (White-Label)

A white-label platform delivers multiple branded apps from a shared codebase. This requires a theming system, per-app feature flags, isolated build configurations, shared native modules, independent release cycles, monorepo structure, and a CI/CD pipeline that can build, test, and deploy N apps from one repository.

1. Shared Core with App-Specific Customization

The fundamental tension in white-label architecture is maximizing code reuse while allowing deep per-app customization. The solution is an inversion-of-control pattern: the shared core defines extension points, and each app fills them.

White-Label Architecture Layers: ┌─────────────────────────────────────────────────────┐ │ App Shell (per app) │ │ AppA/ AppB/ AppC/ │ │ - branding - branding - branding │ │ - feature config - feature config - feature config │ │ - custom screens - overrides - overrides │ └──────────────────────┬──────────────────────────────┘ │ consumes ┌──────────────────────▼──────────────────────────────┐ │ Shared Platform Core │ │ @platform/ui @platform/auth @platform/api │ │ @platform/nav @platform/analytics │ │ @platform/native @platform/config │ └──────────────────────┬──────────────────────────────┘ │ depends on ┌──────────────────────▼──────────────────────────────┐ │ React Native + Native Modules │ └─────────────────────────────────────────────────────┘

1.1 App Registry Pattern

// packages/platform-core/src/AppRegistry.ts
interface AppConfig {
  appId: string;
  displayName: string;
  theme: ThemeConfig;
  features: FeatureFlags;
  screens: ScreenOverrides;
  apiBaseUrl: string;
  deepLinkPrefix: string;
  analytics: { writeKey: string; endpoint: string };
}

class PlatformApp {
  private config: AppConfig;

  constructor(config: AppConfig) {
    this.config = config;
  }

  getScreen(name: string): React.ComponentType {
    return this.config.screens[name]
      ?? defaultScreens[name];
  }

  isFeatureEnabled(feature: string): boolean {
    return this.config.features[feature] ?? false;
  }
}

// apps/app-a/src/index.ts
const appA = new PlatformApp({
  appId: 'app_a',
  displayName: 'Brand A',
  theme: brandATheme,
  features: { checkout: true, loyalty: false, chat: true },
  screens: { Home: BrandAHomeScreen },
  apiBaseUrl: 'https://api.brand-a.com',
  deepLinkPrefix: 'branda://',
  analytics: { writeKey: 'key_a', endpoint: 'https://analytics.brand-a.com' },
});
Architect Insight

Avoid the temptation to use if (appId === 'brand_a') conditionals scattered through shared code. This creates an unmaintainable web of app-specific logic. Instead, use dependency injection: each app provides its configuration and overrides at the boundary, and the shared core consumes them through well-defined interfaces.

2. Theming System

Each white-label app needs distinct visual identity — colors, typography, spacing, icons, and component variants — without duplicating UI code. The theming system must be type-safe, performant, and support runtime switching for dark mode.

2.2 Theme Provider with Dark Mode

import { useColorScheme } from 'react-native';

const ThemeContext = createContext<ThemeTokens>(defaultTheme);

function ThemeProvider({ lightTheme, darkTheme, children }: Props) {
  const colorScheme = useColorScheme();
  const theme = colorScheme === 'dark' ? darkTheme : lightTheme;

  const memoizedTheme = useMemo(() => theme, [colorScheme]);

  return (
    <ThemeContext.Provider value={memoizedTheme}>
      {children}
    </ThemeContext.Provider>
  );
}

function useTheme(): ThemeTokens {
  return useContext(ThemeContext);
}

function useThemedStyles<T>(factory: (theme: ThemeTokens) => T): T {
  const theme = useTheme();
  return useMemo(() => factory(theme), [theme]);
}
Performance: StyleSheet.create vs Dynamic Styles

With theming, you can't use StyleSheet.create at module scope because tokens aren't available yet. Use useMemo with the theme as a dependency to create styles per-render-cycle. For hot paths (FlatList items), consider caching styles in a WeakMap keyed by theme object reference to avoid re-creation.

3. Feature Flags per App

Feature flags control which capabilities are available in each white-label app. They operate at two levels: build-time flags (remove code from bundle) and runtime flags (toggle features without a new build).

3.1 Two-Tier Flag System

Feature Flag Tiers: Build-Time Flags Runtime Flags ───────────────── ───────────────── Defined in app config Fetched from server Dead-code eliminated Evaluated at runtime Smaller bundles Instant rollout/rollback Requires new build No build needed ┌──────────────┐ ┌──────────────┐ │ metro.config │ │ Remote Config │ │ __APP_CHAT__ │ │ chat_v2: true │ │ = false │ │ new_nav: 50% │ └──────┬───────┘ └──────┬───────┘ │ compile time │ runtime ▼ ▼ Code for chat if (flags.chat_v2) not in bundle show new chat UI

3.3 Runtime Flag Provider

interface FeatureFlagProvider {
  isEnabled(flag: string): boolean;
  getValue<T>(flag: string, defaultValue: T): T;
  refresh(): Promise<void>;
}

class RemoteFeatureFlags implements FeatureFlagProvider {
  private flags: Map<string, unknown> = new Map();
  private appId: string;

  async refresh(): Promise<void> {
    const response = await fetch(
      `/api/flags?app=${this.appId}`
    );
    const data = await response.json();
    this.flags = new Map(Object.entries(data));
    await this.persistToCache(data);
  }

  isEnabled(flag: string): boolean {
    return this.flags.get(flag) === true;
  }
}
Flag Hygiene

Feature flags accumulate technical debt. Establish a lifecycle: every flag has an owner, a creation date, and an expiry date. After a feature is fully rolled out, remove the flag and its conditional code. Stale flags create combinatorial complexity — with 20 boolean flags, there are over 1 million possible configurations to test.

4. Build Configuration per App

Each white-label app needs its own bundle identifier, app icon, splash screen, signing certificates, and native configuration. This must be automated — manual configuration for N apps doesn't scale.

4.1 iOS Configuration (Xcode Targets)

  pod 'React', path: '../node_modules/react-native'
  pod 'RNReanimated', path: '../node_modules/react-native-reanimated'
  # ... shared dependencies
end

target 'BrandA' do
  shared_pods
  # BrandA-specific: bundle id, signing, entitlements
end

target 'BrandB' do
  shared_pods
end

5. Shared Native Modules

Native modules (camera, biometrics, push notifications) are shared across all apps but may need per-app configuration (different push notification keys, different biometric prompts).

5.1 Configurable Native Module Pattern

// packages/platform-native/src/PushNotifications.ts
interface PushConfig {
  fcmSenderId: string;
  apnsEnvironment: 'development' | 'production';
  channelId: string;
  defaultIcon: string;
}

class PushNotificationService {
  private config: PushConfig;

  constructor(config: PushConfig) {
    this.config = config;
  }

  async initialize(): Promise<void> {
    await NativeModules.PushModule.configure({
      senderId: this.config.fcmSenderId,
      channelId: this.config.channelId,
    });
  }

  async getToken(): Promise<string> {
    return NativeModules.PushModule.getToken();
  }
}

6. Independent Release Cycles

Different brands may have different release cadences, approval timelines, and rollout strategies. The platform must support releasing Brand A on Monday and Brand B on Thursday from the same monorepo.

6.1 Branching Strategy

Release Branching for Multi-App: main ─────●─────●─────●─────●─────●─────●──→ \ \ \ \ \ release/brand-b/2.1 \ \ \ release/brand-a/3.5 \ release/brand-a/3.4 Each app has its own release branches. Hotfixes cherry-pick from main to the relevant release branch. Shared code changes land on main and flow to all future releases.

6.2 Versioning Strategy

// Each app maintains independent versioning
// apps/brand-a/package.json
{
  "name": "@apps/brand-a",
  "version": "3.5.0",
  "dependencies": {
    "@platform/core": "workspace:*",
    "@platform/ui": "workspace:*"
  }
}

// Shared packages use semver independently
// packages/platform-core/package.json
{
  "name": "@platform/core",
  "version": "1.12.0"
}

// Changesets manage versioning across the monorepo
// .changeset/cool-feature.md
// ---
// "@platform/core": minor
// "@apps/brand-a": patch
// ---
// Added new payment flow to core, integrated in Brand A
CodePush for Decoupled JS Updates

Use CodePush (or Expo Updates) to decouple JavaScript updates from native binary releases. This allows pushing bug fixes to Brand A's JS bundle without going through the App Store review cycle. Each app has its own CodePush deployment key, enabling independent OTA updates.

8. CI/CD for Multiple Apps

The CI/CD pipeline must build, test, and deploy N apps efficiently. Naive approaches (build all apps on every commit) waste resources. Smart pipelines detect which apps are affected by a change and build only those.

8.1 Affected App Detection

#!/bin/bash — detect-affected.sh
# Uses Turborepo's built-in change detection
AFFECTED=$(npx turbo run build --filter=...[origin/main] --dry-run=json \
  | jq -r '.packages[] | select(startswith("@apps/"))')

if [ -z "$AFFECTED" ]; then
  echo "No apps affected — skipping build"
  exit 0
fi

for app in $AFFECTED; do
  echo "Building $app..."
  npx turbo run build:ios build:android --filter=$app
done

Build Cache is Critical

Without caching, building 5 apps × 2 platforms = 10 native builds per commit. Use Turborepo remote caching for JS builds and Gradle build cache / Xcode derived data caching for native builds. A well-configured cache can reduce CI time from 2 hours to 15 minutes for unchanged apps.

10. Architect Considerations

10.1 Customization Spectrum

Customization Depth vs. Maintenance Cost: Maintenance Cost ▲ │ ● Full fork │ (N codebases) │ │ ● Deep overrides │ (custom screens per app) │ │ ● Theme + feature flags │ (config-driven) │ │ ● Branding only │ (colors, logo, name) │ └──────────────────────────────────────→ Customization Depth
Staff-Level Mental Model

A white-label platform is a product-line architecture problem. The key insight is that customization has a cost curve: branding-only changes are nearly free, feature flag toggles are cheap, screen-level overrides are moderate, and deep behavioral changes are expensive. The architect's job is to draw the customization boundary — what can apps customize, and what must remain consistent. Push this boundary too far toward customization and you have N independent apps with shared infrastructure overhead. Pull it too far toward consistency and clients can't differentiate their products. The sweet spot is a configuration-driven platform with well-defined extension points.
