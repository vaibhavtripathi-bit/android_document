1. High-Level Architecture

A chat app is a distributed real-time system with strict ordering, delivery, and consistency requirements. The mobile client must handle unreliable networks, background/foreground transitions, and multi-device synchronization — all while maintaining a responsive 60fps UI.

┌─────────────────────────────────────────────────────────────────────┐ │ CHAT SYSTEM ARCHITECTURE │ ├─────────────────────────────────────────────────────────────────────┤ │ │ │ ┌──────────────┐ WebSocket ┌──────────────────────────┐ │ │ │ RN Client │◄───────────────►│ WebSocket Gateway │ │ │ │ │ (wss://) │ (Connection Manager) │ │ │ │ ┌────────┐ │ └──────────┬───────────────┘ │ │ │ │ SQLite │ │ │ │ │ │ │ Cache │ │ ┌──────────▼───────────────┐ │ │ │ └────────┘ │ │ Message Router │ │ │ │ │ REST/HTTP │ (Pub/Sub + Routing) │ │ │ │ ┌────────┐ │◄───────────────►└──────────┬───────────────┘ │ │ │ │Offline │ │ │ │ │ │ │ Queue │ │ ┌──────────▼───────────────┐ │ │ │ └────────┘ │ │ Message Store │ │ │ └──────────────┘ │ (Cassandra / DynamoDB) │ │ │ └──────────────────────────┘ │ │ │ │ ┌──────────────┐ ┌──────────────────────────┐ │ │ │ APNs / FCM │◄────────────────│ Push Service │ │ │ │ (Platform) │ │ (Offline Delivery) │ │ │ └──────────────┘ └──────────────────────────┘ │ │ │ └─────────────────────────────────────────────────────────────────────┘
Architect Insight

The most critical design decision is the source of truth boundary. The server is the authoritative source for message ordering (via server-assigned timestamps and sequence numbers), but the client must maintain a local cache that allows instant UI updates. This dual-write model — optimistic local writes with server reconciliation — is the foundation of every production chat system.

2. WebSocket Management

WebSockets provide full-duplex, persistent connections essential for real-time messaging. In React Native, managing the WebSocket lifecycle across app states, network changes, and reconnection scenarios is the first major engineering challenge.

2.1 Connection Lifecycle

class ChatSocketManager {
  private ws: WebSocket | null = null;
  private reconnectAttempts = 0;
  private maxReconnectDelay = 30000;
  private heartbeatInterval: NodeJS.Timer | null = null;
  private pendingAcks = new Map<string, PendingMessage>();

  connect(token: string): void {
    this.ws = new WebSocket(`wss://chat.api.com/ws?token=${token}`);

    this.ws.onopen = () => {
      this.reconnectAttempts = 0;
      this.startHeartbeat();
      this.flushOfflineQueue();
      this.requestMissedMessages();
    };

    this.ws.onclose = (event) => {
      this.stopHeartbeat();
      if (!event.wasClean) {
        this.scheduleReconnect();
      }
    };

    this.ws.onerror = () => {
      this.ws?.close();
    };

    this.ws.onmessage = (event) => {
      this.handleMessage(JSON.parse(event.data));
    };
  }

  private scheduleReconnect(): void {
    const delay = Math.min(
      1000 * Math.pow(2, this.reconnectAttempts) + Math.random() * 1000,
      this.maxReconnectDelay
    );
    this.reconnectAttempts++;
    setTimeout(() => this.connect(this.token), delay);
  }
}

2.2 Reconnection Strategy

Strategy Formula Pros Cons
Fixed interval delay = 5000ms Simple, predictable Thundering herd on server recovery
Exponential backoff delay = min(base * 2^n, max) Reduces server load Slow recovery for individual clients
Exponential + jitter delay = random(0, min(base * 2^n, max)) Spreads reconnection load Less predictable for users
Network-aware Backoff + NetInfo listener Only reconnects when network available Depends on OS network detection

2.3 Heartbeat / Keep-Alive

private startHeartbeat(): void {
  this.heartbeatInterval = setInterval(() => {
    if (this.ws?.readyState === WebSocket.OPEN) {
      this.ws.send(JSON.stringify({ type: 'ping', ts: Date.now() }));
      this.pongTimeout = setTimeout(() => {
        this.ws?.close(4000, 'Pong timeout');
      }, 5000);
    }
  }, 25000);
}

private handlePong(): void {
  if (this.pongTimeout) {
    clearTimeout(this.pongTimeout);
    this.pongTimeout = null;
  }
}
Critical: App State Transitions

On iOS, WebSocket connections are suspended when the app enters background. On Android, they may persist briefly but are killed by Doze mode. You must listen to AppState changes: disconnect gracefully on background, reconnect on foreground, and rely on push notifications for background delivery. Never assume the WebSocket is alive after a background period.

3. Message Delivery Guarantees

Chat systems require at-least-once delivery with client-side deduplication to achieve effectively-exactly-once semantics. This is implemented through a three-phase acknowledgment protocol.

3.1 Three-Phase Delivery Protocol

Sender Server Receiver │ │ │ │── MSG (clientId: abc) ─►│ │ │ │── STORE + ASSIGN seqId ─│ │◄── ACK (seqId: 42) ────│ │ │ │── DELIVER (seqId: 42) ─►│ │ │◄── DELIVERED ACK ───────│ │◄── DELIVERED (seqId:42)─│ │ │ │ │ Phase 1: Sent (client → server ACK) Phase 2: Delivered (server → recipient ACK) Phase 3: Read (recipient opens conversation)

3.2 Client-Side Message States

enum MessageStatus {
  QUEUED    = 'queued',     // in offline queue, not yet sent
  SENDING   = 'sending',    // WebSocket send() called
  SENT      = 'sent',       // server ACK received (seqId assigned)
  DELIVERED = 'delivered',  // recipient device confirmed receipt
  READ      = 'read',       // recipient opened the conversation
  FAILED    = 'failed',     // max retries exceeded
}

interface ChatMessage {
  clientId: string;        // UUID generated on client (dedup key)
  serverId?: number;       // server-assigned sequence number
  conversationId: string;
  senderId: string;
  body: string;
  status: MessageStatus;
  createdAt: number;       // client timestamp
  serverTs?: number;       // server-assigned timestamp (ordering)
}

3.3 Idempotency and Deduplication

class MessageDeduplicator {
  private seen = new Set<string>();

  isDuplicate(msg: ChatMessage): boolean {
    const key = msg.serverId
      ? `server:${msg.serverId}`
      : `client:${msg.clientId}`;
    if (this.seen.has(key)) return true;
    this.seen.add(key);
    if (this.seen.size > 10000) {
      const first = this.seen.values().next().value;
      this.seen.delete(first);
    }
    return false;
  }
}
Why Client-Generated UUIDs

The clientId is generated before any network call, allowing the message to appear in the UI instantly (optimistic update). The server uses this clientId for idempotency — if the same message is sent twice (e.g., after a reconnection), the server returns the existing serverId rather than creating a duplicate. This is the same pattern used by WhatsApp, Signal, and Slack.

4. Offline Message Queue

When the device is offline, messages must be queued locally and flushed in order when connectivity returns. The queue must survive app restarts, handle partial flushes, and maintain conversation ordering.

4.1 Persistent Queue Implementation

class OfflineMessageQueue {
  private db: SQLiteDatabase;

  async enqueue(msg: ChatMessage): Promise<void> {
    await this.db.executeSql(
      `INSERT INTO outbox (client_id, conversation_id, body, created_at, retry_count)
       VALUES (?, ?, ?, ?, 0)`,
      [msg.clientId, msg.conversationId, msg.body, msg.createdAt]
    );
  }

  async flush(socket: ChatSocketManager): Promise<void> {
    const pending = await this.db.executeSql(
      `SELECT * FROM outbox ORDER BY created_at ASC LIMIT 50`
    );

    for (const row of pending.rows) {
      try {
        await socket.sendWithAck(row);
        await this.db.executeSql(
          `DELETE FROM outbox WHERE client_id = ?`,
          [row.client_id]
        );
      } catch (err) {
        await this.db.executeSql(
          `UPDATE outbox SET retry_count = retry_count + 1 WHERE client_id = ?`,
          [row.client_id]
        );
        if (row.retry_count >= 5) {
          await this.markFailed(row.client_id);
        }
        break; // preserve ordering — don't skip failed messages
      }
    }
  }
}
Architect Insight — Queue Ordering

The break on failure is deliberate: messages within a conversation must be delivered in order. If message N fails, messages N+1, N+2... must wait. An alternative is per-conversation queues that allow independent progress, but this adds complexity. WhatsApp uses per-conversation queues; simpler apps use a single ordered queue.

5. Message Persistence

Local message storage enables instant conversation loading, offline access, and search. The schema must handle millions of messages across thousands of conversations efficiently.

5.1 SQLite Schema Design

-- Core messages table with covering indexes
CREATE TABLE messages (
  client_id    TEXT PRIMARY KEY,
  server_id    INTEGER UNIQUE,
  conv_id      TEXT NOT NULL,
  sender_id    TEXT NOT NULL,
  body         TEXT,
  media_url    TEXT,
  media_type   TEXT,
  status       TEXT NOT NULL DEFAULT 'queued',
  created_at   INTEGER NOT NULL,
  server_ts    INTEGER,
  is_deleted   INTEGER DEFAULT 0
);

CREATE INDEX idx_messages_conv_ts
  ON messages(conv_id, server_ts DESC);

CREATE INDEX idx_messages_status
  ON messages(status) WHERE status IN ('queued', 'sending');

-- FTS5 for full-text search
CREATE VIRTUAL TABLE messages_fts USING fts5(
  body, content=messages, content_rowid=rowid
);

5.2 Conversation Summary Table

CREATE TABLE conversations (
  conv_id        TEXT PRIMARY KEY,
  title          TEXT,
  last_msg_body  TEXT,
  last_msg_ts    INTEGER,
  unread_count   INTEGER DEFAULT 0,
  is_muted       INTEGER DEFAULT 0,
  is_pinned      INTEGER DEFAULT 0,
  updated_at     INTEGER
);

CREATE INDEX idx_conv_sort
  ON conversations(is_pinned DESC, last_msg_ts DESC);
Storage Budget

On mobile, SQLite databases over 50MB cause noticeable performance degradation. Implement a retention policy: keep the last N messages per conversation locally, archive older messages to server-only storage. Use PRAGMA auto_vacuum = INCREMENTAL and periodically run PRAGMA incremental_vacuum to reclaim space without blocking the UI thread.

6. Read Receipts

Read receipts tell senders when their messages have been seen. The challenge is batching these signals efficiently — sending a read receipt for every individual message would overwhelm the network.

6.1 Batched Read Receipt Protocol

class ReadReceiptManager {
  private pendingReceipts = new Map<string, number>();
  private flushTimer: NodeJS.Timer | null = null;

  markRead(conversationId: string, lastSeqId: number): void {
    const current = this.pendingReceipts.get(conversationId) ?? 0;
    if (lastSeqId > current) {
      this.pendingReceipts.set(conversationId, lastSeqId);
    }
    this.scheduleFlush();
  }

  private scheduleFlush(): void {
    if (this.flushTimer) return;
    this.flushTimer = setTimeout(() => {
      this.flush();
      this.flushTimer = null;
    }, 2000); // batch window: 2 seconds
  }

  private flush(): void {
    const batch = Array.from(this.pendingReceipts.entries())
      .map(([convId, seqId]) => ({ convId, lastReadSeqId: seqId }));
    this.socket.send({ type: 'read_receipts', receipts: batch });
    this.pendingReceipts.clear();
  }
}
High-Water Mark Pattern

Read receipts use a high-water mark: "I've read up to sequence ID 42" implicitly marks all messages 1-42 as read. This is far more efficient than tracking individual message read states. The server stores one integer per user per conversation, not one boolean per message.

7. Typing Indicators

Typing indicators are ephemeral signals — they don't need persistence, delivery guarantees, or ordering. They're fire-and-forget with client-side timeout.

7.1 Throttled Typing Events

class TypingIndicator {
  private lastSent = 0;
  private stopTimer: NodeJS.Timer | null = null;
  private static THROTTLE_MS = 3000;
  private static STOP_DELAY_MS = 5000;

  onTextChange(conversationId: string): void {
    const now = Date.now();
    if (now - this.lastSent > TypingIndicator.THROTTLE_MS) {
      this.socket.send({
        type: 'typing', conversationId, isTyping: true
      });
      this.lastSent = now;
    }
    if (this.stopTimer) clearTimeout(this.stopTimer);
    this.stopTimer = setTimeout(() => {
      this.socket.send({
        type: 'typing', conversationId, isTyping: false
      });
    }, TypingIndicator.STOP_DELAY_MS);
  }
}

// Receiver side: auto-expire typing indicators
const useTypingIndicators = (conversationId: string) => {
  const [typingUsers, setTypingUsers] = useState<Map<string, number>>(new Map());

  useEffect(() => {
    const interval = setInterval(() => {
      const now = Date.now();
      setTypingUsers(prev => {
        const next = new Map(prev);
        for (const [uid, ts] of next) {
          if (now - ts > 6000) next.delete(uid);
        }
        return next;
      });
    }, 1000);
    return () => clearInterval(interval);
  }, []);
};

8. Push Notification Integration

Push notifications are the only reliable delivery channel when the app is backgrounded or killed. The server must detect when a recipient's WebSocket is disconnected and fall back to push.

8.1 Delivery Decision Flow

Server receives message for User B: Is User B's WebSocket connected? ├── YES → Deliver via WebSocket │ Wait 5s for DELIVERED ACK │ ├── ACK received → Done │ └── No ACK → Fall back to push │ └── NO → Send push notification immediately Store message for later WebSocket delivery (User B fetches on next connect)

8.2 Push Payload Design

// Minimal push payload (APNs limit: 4KB, FCM: 4KB)
{
  "notification": {
    "title": "Alice",
    "body": "Hey, are you free for lunch?",
    "badge": 3,
    "thread-id": "conv_abc123"
  },
  "data": {
    "type": "new_message",
    "conv_id": "conv_abc123",
    "msg_server_id": 42,
    "sender_id": "user_alice"
  }
}
Push Notification Privacy

Never include the full message body in push payloads for sensitive apps. Use notification service extensions (iOS) or FirebaseMessagingService (Android) to fetch and decrypt the message content on-device after the push arrives. Signal uses this pattern — the push only contains a "you have a new message" signal, and the app fetches the encrypted content.

9. Image and File Sharing

Media messages require a separate upload pipeline — you never send binary data over the WebSocket. The pattern is: upload to object storage, get a URL, send the URL as a message.

9.1 Upload Pipeline

class MediaUploader {
  async uploadAndSend(
    file: Asset,
    conversationId: string
  ): Promise<void> {
    const clientId = uuid();
    const thumbnail = await generateThumbnail(file, { width: 200 });

    // 1. Show optimistic message with local URI
    dispatch(addOptimisticMessage({
      clientId, conversationId,
      mediaLocalUri: file.uri,
      thumbnailUri: thumbnail.uri,
      uploadProgress: 0,
      status: 'uploading',
    }));

    // 2. Upload with progress tracking
    const { url } = await uploadToS3(file, {
      onProgress: (pct) => dispatch(updateProgress(clientId, pct)),
      contentType: file.type,
    });

    // 3. Send message with media URL
    await socket.sendWithAck({
      clientId, conversationId,
      body: null,
      mediaUrl: url,
      mediaType: file.type,
      thumbnailUrl: thumbnail.remoteUrl,
    });
  }
}

10. Search and Pagination

10.1 Cursor-Based Pagination

async function loadMessages(
  conversationId: string,
  cursor?: number,
  limit = 30
): Promise<MessagePage> {
  // Try local first
  const local = await db.executeSql(
    `SELECT * FROM messages
     WHERE conv_id = ? AND (? IS NULL OR server_ts < ?)
     ORDER BY server_ts DESC LIMIT ?`,
    [conversationId, cursor, cursor, limit + 1]
  );

  const hasMore = local.rows.length > limit;
  const messages = local.rows.slice(0, limit);

  if (messages.length < limit) {
    // Local cache exhausted — fetch from server
    const remote = await api.getMessages(conversationId, cursor, limit);
    await insertMessages(remote.messages);
    return remote;
  }

  return {
    messages,
    nextCursor: hasMore ? messages[messages.length - 1].server_ts : null,
  };
}

10.2 Full-Text Search

async function searchMessages(query: string): Promise<SearchResult[]> {
  const results = await db.executeSql(
    `SELECT m.*, snippet(messages_fts, 0, '<b>', '</b>', '...', 32) as snippet
     FROM messages_fts
     JOIN messages m ON m.rowid = messages_fts.rowid
     WHERE messages_fts MATCH ?
     ORDER BY rank
     LIMIT 50`,
    [`"${query}"`]
  );
  return results.rows;
}

11. Architect Considerations

11.1 End-to-End Encryption

For privacy-sensitive chat, implement the Signal Protocol (Double Ratchet): each message is encrypted with a unique key derived from a ratcheting key exchange. The server never sees plaintext. This affects every layer: push payloads must be opaque, search must be client-side only, and key management adds significant complexity.

Staff-Level Summary

A production chat system is a distributed state machine with the client as an eventually-consistent replica. The key architectural decisions are: (1) optimistic local writes with server reconciliation, (2) sequence-number-based ordering rather than timestamps, (3) at-least-once delivery with client dedup, (4) push notification fallback for offline delivery, and (5) cursor-based pagination with local-first reads. Every decision trades latency against consistency — and in chat, users always prefer latency.
