1. Biometric Auth Architecture

Biometric authentication in mobile apps does not replace server-side authentication — it protects locally stored credentials that are used to authenticate with the server. The biometric check unlocks access to a token or credential stored in the device's secure enclave, which is then sent to the server. The server never sees biometric data.

Biometric Authentication Flow ═════════════════════════════ User Device Server ──── ────── ────── Tap "Login" │ ▼ Biometric prompt (Face ID / Fingerprint) │ ▼ success ┌──────────────────┐ │ Secure Enclave │ │ / Keystore │ │ │ │ Decrypt stored │ │ refresh token │ └────────┬─────────┘ │ ▼ Send refresh token ──────────────────────▶ Validate token │ ▼ Issue new access token │ Receive access token ◀────────────────────┘ │ ▼ App is authenticated KEY INSIGHT: The server never receives biometric data. Biometrics only gate access to locally stored credentials.

1.1 Security Model

interface BiometricAuthConfig {
  // What biometric protects
  protectedCredential: 'refresh_token' | 'encryption_key' | 'pin';

  // When to require biometric
  requireBiometricFor: {
    appUnlock: boolean;
    sensitiveActions: boolean;   // payments, transfers
    viewSensitiveData: boolean;  // SSN, account numbers
  };

  // Fallback chain
  fallbackOrder: ('biometric' | 'device_passcode' | 'app_pin' | 'password')[];

  // Re-auth timeout
  sessionTimeout: number;  // ms before requiring re-auth

  // Invalidation policy
  invalidateOnBiometryChange: boolean;
}
Architect Insight — What Biometrics Actually Protect

A common misconception is that biometric auth "logs the user in." In reality, biometrics are a local unlock mechanism for credentials stored in hardware-backed secure storage. The security strength depends entirely on what you store and how you store it. Storing a refresh token in Keychain with kSecAccessControlBiometryCurrentSet means the token is cryptographically bound to the current set of enrolled biometrics — if the user adds a new fingerprint, the token becomes inaccessible, forcing re-authentication with the server.

2. Face ID & Touch ID (iOS)

iOS biometric authentication uses the Local Authentication framework (LAContext) for simple presence checks, and the Keychain Services API with access control for cryptographically-backed biometric protection. Staff engineers must understand both layers.

2.1 LAContext vs Keychain-Backed

Approach Security Level How It Works Use Case
LAContext (presence check) Medium OS verifies biometric, returns boolean Quick unlock, non-sensitive actions
Keychain + Access Control High Credential encrypted by Secure Enclave, decrypted only after biometric Token storage, payment auth

import ReactNativeBiometrics from 'react-native-biometrics';

const biometrics = new ReactNativeBiometrics({

async function checkBiometricAvailability(): Promise<BiometricStatus> {

  return {

async function authenticateWithBiometric(): Promise<boolean> {
  const { success } = await biometrics.simplePrompt({
    promptMessage: 'Confirm your identity',
    cancelButtonText: 'Use Password',
    fallbackPromptMessage: 'Use device passcode',
  });

  return success;
}

2.3 Keychain-Backed Biometric Storage (iOS)

import * as Keychain from 'react-native-keychain';

async function storeCredentialWithBiometric(
  key: string,
  credential: string
) {
  await Keychain.setGenericPassword(key, credential, {
    accessControl: Keychain.ACCESS_CONTROL.BIOMETRY_CURRENT_SET,
    accessible: Keychain.ACCESSIBLE.WHEN_PASSCODE_SET_THIS_DEVICE_ONLY,
    authenticationType: Keychain.AUTHENTICATION_TYPE.BIOMETRICS,
    securityLevel: Keychain.SECURITY_LEVEL.SECURE_HARDWARE,
    storage: Keychain.STORAGE_TYPE.RSA,
  });
}

async function retrieveCredentialWithBiometric(
  key: string
): Promise<string | null> {
  try {
    const result = await Keychain.getGenericPassword({
      authenticationPrompt: {
        title: 'Authenticate to access your account',
        cancel: 'Cancel',
      },
    });

    if (result === false) return null;
    return result.password;
  } catch (error) {
    if (error.message.includes('User canceled')) return null;
    throw error;
  }
}
Face ID Info.plist Requirement

iOS requires the NSFaceIDUsageDescription key in Info.plist to use Face ID. Without it, the app crashes when attempting Face ID authentication. Touch ID does not require a usage description. This is the most common cause of biometric auth crashes in production — it works in development (Touch ID simulator) but crashes on Face ID devices.

3. Fingerprint & Face Unlock (Android)

Android's biometric landscape is more fragmented than iOS. The BiometricPrompt API (introduced in Android 9) provides a unified interface, but the underlying hardware varies wildly — from secure fingerprint sensors to weak face recognition that can be fooled by a photo.

3.1 Android Biometric Classification

Class Security Level Hardware Requirement Examples
Class 3 (Strong) Cryptographic operations TEE / Secure Element Pixel fingerprint, Samsung ultrasonic
Class 2 (Weak) Unlock only, no crypto Software-based acceptable Most face unlock implementations
Class 1 (Convenience) Not recommended None Deprecated iris scanners

3.2 Android BiometricPrompt Integration

import ReactNativeBiometrics from 'react-native-biometrics';

async function androidBiometricAuth(): Promise<AuthResult> {
  const biometrics = new ReactNativeBiometrics({
    allowDeviceCredentials: true,
  });

  const { available, biometryType } = await biometrics.isSensorAvailable();

  if (!available) {
    return { success: false, fallbackRequired: true };
  }

  const { success, error } = await biometrics.simplePrompt({
    promptMessage: 'Verify your identity',
    cancelButtonText: 'Use PIN instead',
  });

  return { success, error };
}

async function signChallenge(challenge: string): Promise<string> {
  const biometrics = new ReactNativeBiometrics();

  const { success, signature } = await biometrics.createSignature({
    promptMessage: 'Sign in',
    payload: challenge,
  });

  if (!success || !signature) {
    throw new Error('Biometric signature failed');
  }

  return signature;
}
Cryptographic vs Simple Prompt

simplePrompt is a presence check — the OS confirms the user is who they claim, but your app receives only a boolean. createSignature is cryptographic — the biometric unlocks a private key in the Android Keystore (TEE-backed), which signs a server challenge. The server verifies the signature with the stored public key. Cryptographic auth is significantly more secure because it proves the biometric check happened on this specific device — it cannot be replayed from another device.

4. Fallback Mechanisms

Biometric auth fails more often than developers expect — wet fingers, masks, poor lighting, hardware damage, and OS-level lockouts all prevent biometric recognition. A robust fallback chain is essential for user retention.

4.1 Fallback Chain Design

Biometric Fallback Chain ════════════════════════ Attempt biometric │ ├── Success ──────────────────────────▶ Authenticated │ ├── User cancelled ──▶ Show fallback options │ │ │ ├── Device passcode │ │ (OS-managed, secure) │ │ │ ├── App PIN │ │ (app-managed, 6-digit) │ │ │ └── Full password │ (server-validated) │ ├── Biometric locked ──▶ Force device passcode │ (too many failures) (OS requirement after 5 failures) │ ├── Biometric not ─────▶ Skip biometric, show PIN/password │ enrolled │ └── Biometric not ─────▶ Device does not support biometric available Show PIN/password only

4.2 Fallback Implementation

class AuthenticationManager {
  private config: BiometricAuthConfig;

  async authenticate(): Promise<AuthResult> {
    const biometricStatus = await this.getBiometricStatus();

    if (biometricStatus.available && biometricStatus.enrolled) {
      const result = await this.attemptBiometric();
      if (result.success) return result;
      if (result.error === 'LOCKED_OUT') {
        return this.requireDevicePasscode();
      }
    }

    for (const method of this.config.fallbackOrder) {
      switch (method) {
        case 'device_passcode':
          return this.requireDevicePasscode();
        case 'app_pin':
          return this.requireAppPin();
        case 'password':
          return this.requirePassword();
      }
    }

    throw new Error('No authentication method available');
  }

  private async getBiometricStatus(): Promise<BiometricStatus> {
    const biometrics = new ReactNativeBiometrics();
    const { available, biometryType, error } =
      await biometrics.isSensorAvailable();

    return {
      available,
      type: biometryType,
      enrolled: available && !error,
      locked: error === 'BIOMETRIC_ERROR_LOCKOUT',
    };
  }
}

5. Secure Credential Storage

The security of biometric auth depends entirely on how credentials are stored. Both platforms provide hardware-backed secure storage that ties credential access to biometric verification — but the APIs and guarantees differ significantly.

5.1 Platform Secure Storage Comparison

Feature iOS Keychain + Secure Enclave Android Keystore + TEE/StrongBox
Hardware isolation Secure Enclave (all devices since iPhone 5s) TEE (most devices), StrongBox (Pixel 3+, Samsung S10+)
Key never leaves hardware Yes (SE-backed keys) Yes (TEE/StrongBox-backed)
Biometric binding kSecAccessControlBiometryCurrentSet setUserAuthenticationRequired(true)
Invalidation on new enrollment Yes (BiometryCurrentSet) Yes (setInvalidatedByBiometricEnrollment(true))
Backup inclusion Configurable per item Keys are never backed up
Wipe on device reset Yes Yes

5.2 Secure Token Storage Pattern

class SecureBiometricStorage {
  private readonly SERVICE_NAME = 'com.myapp.biometric-auth';

  async storeRefreshToken(token: string): Promise<void> {
    await Keychain.setGenericPassword(
      'refresh_token',
      token,
      {
        service: this.SERVICE_NAME,
        accessControl: Keychain.ACCESS_CONTROL.BIOMETRY_CURRENT_SET,
        accessible: Keychain.ACCESSIBLE.WHEN_PASSCODE_SET_THIS_DEVICE_ONLY,
        securityLevel: Keychain.SECURITY_LEVEL.SECURE_HARDWARE,
      }
    );
  }

  async getRefreshToken(): Promise<string | null> {
    try {
      const credentials = await Keychain.getGenericPassword({
        service: this.SERVICE_NAME,
        authenticationPrompt: {
          title: 'Unlock your account',
        },
      });
      return credentials ? credentials.password : null;
    } catch (error) {
      if (this.isBiometryInvalidated(error)) {
        await this.clearStoredCredentials();
        return null;
      }
      throw error;
    }
  }

  private isBiometryInvalidated(error: any): boolean {
    return (
      error.message?.includes('Key permanently invalidated') ||
      error.message?.includes('Biometry has been changed')
    );
  }

  async clearStoredCredentials(): Promise<void> {
    await Keychain.resetGenericPassword({
      service: this.SERVICE_NAME,
    });
  }
}
BIOMETRY_CURRENT_SET Invalidation

When using BIOMETRY_CURRENT_SET (iOS) or setInvalidatedByBiometricEnrollment(true) (Android), the stored credential becomes permanently inaccessible if the user adds or removes a biometric (new fingerprint, re-enrolled Face ID). Your app must detect this failure, clear the invalidated credential, and prompt the user to re-authenticate with their password to re-enroll biometric access. Failing to handle this results in users being permanently locked out.

6. Re-Authentication Flows

Re-authentication protects sensitive operations (payments, password changes, data export) by requiring a fresh biometric check even if the user is already logged in. The challenge is balancing security with friction.

6.1 Re-Auth Trigger Points

const REAUTH_TRIGGERS: Record<string, ReAuthPolicy> = {
  viewBankAccount: {
    maxAge: 5 * 60 * 1000,   // 5 minutes
    method: 'biometric',
    fallback: 'pin',
  },
  initiateTransfer: {
    maxAge: 0,                // Always re-auth
    method: 'biometric',
    fallback: 'password',
  },
  changePassword: {
    maxAge: 0,
    method: 'password',       // Biometric not sufficient
    fallback: null,
  },
  exportData: {
    maxAge: 2 * 60 * 1000,   // 2 minutes
    method: 'biometric',
    fallback: 'password',
  },
};

class ReAuthGuard {
  private lastAuthTimestamp: number = 0;

  async guard(action: string): Promise<boolean> {
    const policy = REAUTH_TRIGGERS[action];
    if (!policy) return true;

    const elapsed = Date.now() - this.lastAuthTimestamp;
    if (elapsed < policy.maxAge) return true;

    const authManager = new AuthenticationManager();
    const result = await authManager.authenticate();

    if (result.success) {
      this.lastAuthTimestamp = Date.now();
    }

    return result.success;
  }
}

7. Biometric Enrollment Detection

Your app must detect whether biometrics are available, enrolled, and unchanged since the user last authenticated. Each state requires different UX handling.

7.1 Enrollment State Machine

Biometric Enrollment States ═══════════════════════════ App Launch │ ▼ Check hardware ──── No hardware ──▶ Hide biometric option │ Show PIN/password only │ Has hardware │ ▼ Check enrollment ── Not enrolled ──▶ Show "Enable in Settings" │ prompt with deep link │ Is enrolled │ ▼ Check if changed ── Changed ───────▶ Invalidate stored credential │ (new fingerprint) Force password re-auth │ Re-enroll biometric │ Not changed │ ▼ Biometric ready ──────────────────▶ Show biometric login option

7.2 Comprehensive Status Check

type BiometricReadiness =
  | { status: 'ready'; type: string }
  | { status: 'no_hardware' }
  | { status: 'not_enrolled'; canPromptSettings: boolean }
  | { status: 'invalidated'; reason: string }
  | { status: 'locked_out'; permanent: boolean }
  | { status: 'no_passcode' };

async function getBiometricReadiness(): Promise<BiometricReadiness> {
  const biometrics = new ReactNativeBiometrics();
  const { available, biometryType, error } =
    await biometrics.isSensorAvailable();

  if (!available) {
    if (error === 'BIOMETRIC_ERROR_NO_HARDWARE') {
      return { status: 'no_hardware' };
    }
    if (error === 'BIOMETRIC_ERROR_NONE_ENROLLED') {
      return {
        status: 'not_enrolled',
        canPromptSettings: Platform.OS === 'ios',
      };
    }
    if (error === 'BIOMETRIC_ERROR_LOCKOUT') {
      return { status: 'locked_out', permanent: false };
    }
    if (error === 'BIOMETRIC_ERROR_LOCKOUT_PERMANENT') {
      return { status: 'locked_out', permanent: true };
    }
  }

  const storedCredential = await checkStoredCredentialValidity();
  if (storedCredential.invalidated) {
    return {
      status: 'invalidated',
      reason: 'Biometric enrollment changed',
    };
  }

  return { status: 'ready', type: biometryType! };
}

function BiometricLoginButton() {

  useEffect(() => {
    getBiometricReadiness().then(setBiometricStatus);

      accessibilityLabel={label}
      accessibilityRole="button"
    >
      <Icon name={icon} />
      <Text>{label}</Text>
    </Pressable>
  );
}

9. Accessibility Considerations

Biometric auth introduces unique accessibility challenges. Users with physical disabilities, prosthetics, skin conditions, or facial differences may be unable to use biometrics. Your auth system must be fully functional without biometric input.

9.1 Accessibility Requirements

  • Never make biometric the only option: Always provide an alternative authentication method (PIN, password). Some users physically cannot use biometrics.
  • Clear error messaging: When biometric fails, explain what happened and what to do next. "Authentication failed" is not helpful — "Face not recognized. Tap below to use your PIN instead" is.
  • VoiceOver / TalkBack support: The system biometric prompt is automatically accessible, but your fallback UI must be fully labeled. Announce state changes ("Biometric authentication required") to screen readers.
  • Motor accessibility: Users with tremors may have difficulty holding the device steady for face recognition. Ensure generous timeout and clear retry affordance.
  • Respect user preferences: If a user disables biometric login in your app settings, remember the preference and don't prompt again. Repeated biometric prompts for users who can't use them are a significant accessibility barrier.

WCAG and Biometrics

WCAG 2.1 Success Criterion 3.3.7 (Accessible Authentication) states that cognitive function tests should not be required for authentication. Biometrics satisfy this because they don't require memory or cognitive effort — but only if an alternative is available for users who can't use biometrics. The alternative must not be harder than the biometric path.

10. Architect Considerations

10.1 Production Checklist

  • Test biometric enrollment changes: Add a new fingerprint on a test device and verify your app handles the invalidation gracefully — not with a crash or permanent lockout.
  • Handle jailbroken/rooted devices: On compromised devices, biometric APIs can be hooked to always return success. For high-security apps (banking), detect jailbreak/root and disable biometric auth, falling back to server-validated password.
  • Biometric opt-in, not opt-out: Don't enable biometric auth by default. Prompt users to enable it after successful password login. This avoids confusing users who don't understand biometric auth and ensures they have a working password as fallback.
  • Session timeout design: Balance security and convenience. Banking apps: 5-minute timeout with biometric re-auth. Social apps: 30-day session with biometric for sensitive actions only. The timeout should be configurable via remote config.
  • Audit logging: Log every biometric auth attempt (success, failure, fallback used) server-side. This creates an audit trail for security incidents and helps detect brute-force attempts.
  • Biometric spoofing awareness: Android Class 2 face unlock can be fooled by a photo on some devices. For financial apps, require Class 3 biometrics only, or combine biometric with a secondary factor (PIN confirmation for transactions above a threshold).
Staff-Level Mental Model

Biometric authentication is a local trust delegation mechanism — you're delegating the "is this the right person?" question to the device's hardware. The security of this delegation depends on three factors: the quality of the biometric sensor (Class 3 vs Class 2), the protection of the stored credential (hardware-backed keystore vs plain storage), and the robustness of the fallback chain (what happens when biometric fails). A Staff engineer designs for the weakest link: the cheapest Android device your app supports, the user who just added a new fingerprint, and the accessibility user who can't use biometrics at all. The system must be secure, usable, and inclusive — simultaneously.
