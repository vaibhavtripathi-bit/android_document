1. Audit Methodology

A performance audit is not ad-hoc profiling. It's a structured process: define metrics, measure baselines, identify bottlenecks, fix them, verify improvements, and establish guardrails to prevent regression.

Architect Insight

Always measure on real devices in release mode. Development builds include the dev menu, JS debugger overhead, and unminified code — performance numbers from dev builds are meaningless. Use the lowest-spec device your users actually have (check analytics) as your benchmark device. A Pixel 3a or iPhone SE 2 is more representative than a Pixel 8 Pro.

2. Startup Time Analysis

Startup time is the single most impactful performance metric — every 100ms of delay reduces conversion by 1-2%. React Native startup involves native initialization, JS engine boot, bundle loading, JS execution, and first render.

2.1 Startup Phases

COLD START TIMELINE (React Native) 0ms 200ms 500ms 800ms 1200ms 1800ms │───────────│───────────│───────────│───────────│───────────│ │ │ │ │ │ │ ├─ Native ──┤ │ │ │ │ │ init │ │ │ │ │ │ ├─ JS Engine┤ │ │ │ │ │ boot │ │ │ │ │ │ ├─ Bundle ──┤ │ │ │ │ │ load + │ │ │ │ │ │ parse │ │ │ │ │ │ ├─ JS exec ─┤ │ │ │ │ │ (require) │ │ │ │ │ │ ├─ First ───┤ │ │ │ │ │ render │ │ │ │ │ │ + layout │ │ │ │ │ │ │ SPLASH ─────────────────────────────────────────► TTI Target: TTI < 2s on low-end device

2.2 Measuring Startup

// Instrument startup phases with performance marks
import { PerformanceObserver, performance } from 'react-native-performance';

// In index.js (earliest possible point)
performance.mark('js_bundle_start');

// In App.tsx
function App() {
  useEffect(() => {
    performance.mark('first_render_complete');
    performance.measure('startup', 'js_bundle_start', 'first_render_complete');

    const entry = performance.getEntriesByName('startup')[0];
    analytics.track('startup_time', { duration: entry.duration });
  }, []);

  return <AppNavigator />;
}

2.3 Startup Optimization Checklist

Optimization Impact Effort How
Enable Hermes 30-50% faster TTI Low Bytecode precompilation eliminates parse time
Inline requires 10-30% faster JS exec Low Defer module loading until first use
Reduce bundle size Proportional to reduction Medium Tree shaking, code splitting, smaller deps
Defer non-critical init 20-40% faster TTI Medium InteractionManager.runAfterInteractions
Preload critical data Eliminates loading states Medium Cache last-known state, show immediately
RAM bundles (legacy) 20-30% faster on JSC High Unbundled mode with inline requires
Hermes Bytecode Advantage

Hermes precompiles JavaScript to bytecode at build time. This eliminates the parse + compile phase that JavaScriptCore performs at runtime. On a Pixel 3a, this alone reduces TTI by 300-500ms. The bytecode is also more compact than source JS, reducing bundle load time. Always verify Hermes is enabled: check global.HermesInternal at runtime.

3. Frame Rate Profiling

React Native targets 60fps (16.67ms per frame). Frame drops ("jank") occur when either the JS thread or the UI thread takes longer than 16.67ms to process a frame. Identifying which thread is the bottleneck is the first step.

3.1 Thread-Level Frame Analysis

FRAME BUDGET: 16.67ms (60fps) JS Thread UI Thread ┌────────────────────┐ ┌────────────────────┐ │ React reconcile │ │ Layout (Yoga) │ │ State updates │ │ View creation │ │ Business logic │ │ Drawing │ │ Bridge serialization│ │ Compositing │ └────────────────────┘ └────────────────────┘ If JS > 16.67ms → JS thread jank (dropped frames on interaction) If UI > 16.67ms → UI thread jank (dropped frames on animation) If both > 16.67ms → double jank (worst case) Common causes: JS jank: large re-renders, expensive computations, JSON.parse UI jank: complex layouts, shadow/elevation, image decoding

3.3 Common Jank Patterns and Fixes

Navigation transition jank Choppy screen transitions Heavy component mount during animation InteractionManager, lazy screens

Image loading jank Stutter when images appear Main thread image decoding FastImage, prefetch, thumbnails

4. Memory Leak Detection

Memory leaks in React Native are insidious — the app works fine for minutes but degrades over hours, eventually crashing with an OOM kill. There are two memory spaces to monitor: the JS heap (managed by Hermes/JSC) and native memory (managed by the platform).

4.1 Common Leak Patterns

// LEAK 1: Uncleared timers/subscriptions
function LeakyComponent() {
  useEffect(() => {
    const interval = setInterval(fetchData, 5000);
    // Missing: return () => clearInterval(interval);
  }, []);
}

// LEAK 2: Event listener not removed
function LeakyListener() {
  useEffect(() => {
    const sub = AppState.addEventListener('change', handler);
    // Missing: return () => sub.remove();
  }, []);
}

// LEAK 3: Closure over stale large data
function LeakyClosure() {
  const [data] = useState(loadLargeDataset()); // 50MB
  const callback = useCallback(() => {
    // This closure retains a reference to `data`
    // even if the component unmounts, if callback
    // is passed to a long-lived subscriber
    process(data);
  }, [data]);
}

// LEAK 4: Animated value not cleaned up
function LeakyAnimation() {
  const anim = useRef(new Animated.Value(0)).current;
  useEffect(() => {
    Animated.loop(
      Animated.timing(anim, { toValue: 1, duration: 1000, useNativeDriver: true })
    ).start();
    // Missing: return () => anim.stopAnimation();
  }, []);
}

4.2 Memory Profiling Workflow

MEMORY LEAK DETECTION WORKFLOW 1. Establish baseline └── Open app, navigate to home, record heap size: H₀ 2. Stress test └── Navigate to Screen A → Back → Screen A → Back (×20) Record heap after each cycle: H₁, H₂, ... H₂₀ 3. Analyze trend └── If H₂₀ - H₀ > threshold → leak detected Plot H₀..H₂₀: linear growth = leak, plateau = normal GC 4. Isolate └── Take heap snapshot at H₀ and H₂₀ Diff snapshots: objects present in H₂₀ but not H₀ Group by constructor/type → find retained objects 5. Fix and verify └── Fix the leak, repeat steps 1-3 Verify H₂₀ ≈ H₀ (within GC variance)

4.3 Automated Leak Detection

// Automated memory regression test
async function memoryLeakTest(scenario: () => Promise<void>, iterations = 20) {
  const measurements: number[] = [];

  for (let i = 0; i < iterations; i++) {
    await scenario();
    await forceGC();
    await sleep(500);
    measurements.push(await getJSHeapSize());
  }

  const slope = linearRegression(measurements).slope;
  const totalGrowth = measurements[measurements.length - 1] - measurements[0];

  return {
    leaked: slope > 0.1, // MB per iteration threshold
    growthMB: totalGrowth / 1024 / 1024,
    slopePerIteration: slope,
    measurements,
  };
}
Native Memory Leaks

JS heap profiling only catches JavaScript-side leaks. Native memory leaks (image caches not released, native modules holding references, WebView instances) require platform-specific tools: Xcode Memory Graph Debugger for iOS and Android Studio Profiler for Android. A common React Native native leak is large images cached by the native image loader that are never evicted — always set cache limits.

5. Bundle Size Analysis

Bundle size directly impacts startup time (more bytes to load and parse), update size (OTA and app store), and install conversion (users abandon large installs). Every KB matters on mobile.

5.1 Analysis Tools

// Generate Metro bundle stats
npx react-native bundle \
  --platform android \
  --dev false \
  --entry-file index.js \
  --bundle-output /tmp/bundle.js \
  --sourcemap-output /tmp/bundle.js.map

// Analyze with source-map-explorer
npx source-map-explorer /tmp/bundle.js /tmp/bundle.js.map

// Or use react-native-bundle-visualizer
npx react-native-bundle-visualizer --platform android

5.2 Size Budget Breakdown

Category Budget Typical Offenders
React + React Native core ~600KB (Hermes bytecode) Unavoidable baseline
Navigation < 150KB @react-navigation + screens
State management < 50KB Redux + middleware can bloat
UI library < 200KB Full imports of NativeBase/Paper
Utilities < 100KB lodash (full), moment.js
Application code < 500KB Unoptimized, no code splitting
Total JS bundle < 1.5MB

5.3 Size Reduction Strategies

// Replace lodash with individual imports
// Before: import _ from 'lodash';  (71KB)
// After:
import debounce from 'lodash/debounce';  // 1.2KB

// Replace moment.js with date-fns or dayjs
// moment.js: 67KB min + locale data
// dayjs:     2KB min + tree-shakeable plugins
import dayjs from 'dayjs';
import relativeTime from 'dayjs/plugin/relativeTime';
dayjs.extend(relativeTime);

// Metro config: enable tree shaking (experimental)
// metro.config.js
module.exports = {
  transformer: {
    getTransformOptions: async () => ({
      transform: {
        experimentalImportSupport: true,
        inlineRequires: true,
      },
    }),
  },
};

6. Network Waterfall Analysis

Network performance on mobile is fundamentally different from desktop: higher latency, variable bandwidth, and frequent connection changes. A waterfall analysis reveals serialized requests, redundant fetches, and missing caches.

6.1 Waterfall Optimization

BEFORE OPTIMIZATION (serialized requests): 0ms 200ms 400ms 600ms 800ms 1000ms 1200ms 1400ms │───────│───────│───────│───────│───────│───────│───────│ ├─ Auth token refresh ──┤ │ ├─ User profile ──┤ │ │ ├─ Feed data ──────┤ │ │ │ ├─ Images... Total: ~1400ms + image loading AFTER OPTIMIZATION (parallel + prefetch): 0ms 200ms 400ms 600ms 800ms │───────│───────│───────│───────│ ├─ Auth refresh ────────┤ ├─ Feed (cached) ─┤ │ ├─ Profile (cached)┤ │ │ ├─ Images (prefetched)──┤ Total: ~600ms (57% reduction)

6.2 Network Optimization Strategies

// 1. Request deduplication
class RequestDeduplicator {
  private inflight = new Map<string, Promise<any>>();

  async fetch(url: string, options?: RequestInit): Promise<Response> {
    const key = `${options?.method ?? 'GET'}:${url}`;
    if (this.inflight.has(key)) {
      return this.inflight.get(key)!;
    }
    const promise = fetch(url, options).finally(() => {
      this.inflight.delete(key);
    });
    this.inflight.set(key, promise);
    return promise;
  }
}

// 2. Stale-while-revalidate pattern
async function fetchWithSWR<T>(key: string, fetcher: () => Promise<T>): Promise<T> {
  const cached = await cache.get(key);
  if (cached && !isExpired(cached)) {
    // Return stale data immediately, revalidate in background
    fetcher().then(fresh => cache.set(key, fresh));
    return cached.data;
  }
  const fresh = await fetcher();
  await cache.set(key, fresh);
  return fresh;
}

7. Performance Budgets

Performance budgets are quantitative thresholds that, when exceeded, block deployment. They transform performance from a subjective concern into an engineering constraint.

7.1 Budget Definition

    "frozen_frames_pct": 0.5

    "leak_rate_mb_per_hour": 2

  }
}
Architect Insight — Budget Enforcement

Budgets without enforcement are aspirations. Integrate budget checks into CI: fail the build if the JS bundle exceeds the size budget, fail the PR if Detox performance tests show regression beyond the threshold. Use Lighthouse CI patterns adapted for mobile: store historical measurements, compare against baselines, and generate trend reports. The budget should be owned by the team, reviewed quarterly, and tightened as the team improves.

8.2 Detox Performance Test

describe('Performance Regression', () => {
  it('startup TTI within budget', async () => {
    const start = Date.now();
    await device.launchApp({ newInstance: true });
    await waitFor(element(by.id('home-feed')))
      .toBeVisible()
      .withTimeout(10000);
    const tti = Date.now() - start;

    expect(tti).toBeLessThan(3000);
    reportMetric('startup_tti', tti);
  });

  it('feed scroll maintains 55+ fps', async () => {
    await element(by.id('home-feed')).swipe('up', 'slow', 0.8);
    await element(by.id('home-feed')).swipe('up', 'fast', 0.8);

    const metrics = await device.getPerformanceMetrics();
    expect(metrics.averageFPS).toBeGreaterThan(55);
    expect(metrics.slowFramesPct).toBeLessThan(5);
  });
});

10. Architect Considerations

Staff-Level Summary

A performance audit is a system-level discipline, not a one-time activity. The Staff engineer's role is to: (1) establish the measurement infrastructure (instrumentation, dashboards, alerts), (2) define budgets tied to business metrics (retention, conversion), (3) build automated regression gates into CI/CD, (4) create a culture where performance is a feature, not an afterthought, and (5) prioritize fixes by user impact (p95 matters more than p50 — fix the worst experiences first). The goal is not to make the app fast once — it's to make it impossible for the app to become slow.
