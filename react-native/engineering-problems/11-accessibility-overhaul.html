Overhauling Accessibility in a React Native App

Accessibility is not a feature — it's a quality dimension. An overhaul involves auditing the entire app against WCAG 2.1 AA, fixing VoiceOver/TalkBack issues, implementing focus management, supporting dynamic font scaling, ensuring color contrast, respecting reduced motion preferences, and building automated testing into CI.

1. Audit Process

An accessibility audit systematically evaluates every screen, component, and interaction against WCAG 2.1 AA criteria. The audit produces a prioritized backlog of issues with severity ratings.

1.1 Audit Methodology

Accessibility Audit Pipeline: Phase 1: Automated Scan Phase 2: Manual Testing ────────────────────── ────────────────────── Run axe/jest-axe on VoiceOver walkthrough component tree TalkBack walkthrough Check color contrast Keyboard navigation Validate ARIA roles Switch control testing │ │ ▼ ▼ Phase 3: User Testing Phase 4: Remediation ────────────────────── ────────────────────── Assistive tech users Prioritize by severity test key flows Fix P0 (blockers) first Record pain points Add regression tests Validate fixes Track WCAG criteria

1.2 Issue Severity Classification

Severity Definition Example SLA
P0 — Blocker User cannot complete core flow Login button not focusable Fix in current sprint
P1 — Critical Major flow degraded Missing labels on form fields Fix within 2 sprints
P2 — Major Usable but confusing Incorrect reading order Fix within quarter
P3 — Minor Cosmetic or enhancement Verbose accessibility labels Backlog

1.3 Screen Inventory Checklist

// Audit checklist per screen
interface ScreenAudit {
  screenName: string;
  checks: {
    allInteractiveElementsFocusable: boolean;
    logicalFocusOrder: boolean;
    accessibilityLabelsPresent: boolean;
    accessibilityRolesCorrect: boolean;
    colorContrastPasses: boolean;
    dynamicFontScalingWorks: boolean;
    reducedMotionRespected: boolean;
    screenReaderAnnouncementsWork: boolean;
    noInfoConveyedByColorAlone: boolean;
    touchTargetMinimum44pt: boolean;
  };
  issues: AuditIssue[];
  wcagCriteria: string[];  // e.g., ["1.1.1", "2.4.3", "4.1.2"]
}
Architect Insight

Don't audit every screen at once. Start with the critical user journey: onboarding → login → primary feature → settings. These 5-8 screens cover 80% of user interactions. Fix those first, then expand to secondary flows. This delivers measurable improvement quickly and builds team expertise for the remaining screens.

2. VoiceOver / TalkBack Testing

Screen readers are the primary assistive technology for blind and low-vision users. VoiceOver (iOS) and TalkBack (Android) have different behaviors, gesture models, and API mappings that must be tested independently.

2.1 React Native Accessibility Props

// Core accessibility props in React Native
<TouchableOpacity
  accessible={true}
  accessibilityLabel="Add item to cart"
  accessibilityHint="Double tap to add this product to your shopping cart"
  accessibilityRole="button"
  accessibilityState={{ disabled: !inStock, busy: isLoading }}
  accessibilityValue={{ text: `$${price}` }}
  onPress={addToCart}
>
  <Icon name="cart-plus" />
  <Text>Add to Cart</Text>
</TouchableOpacity>

2.2 Platform Behavior Differences

Behavior VoiceOver (iOS) TalkBack (Android)
Navigation gesture Swipe left/right Swipe left/right (or explore by touch)
Activation Double tap Double tap
accessibilityRole mapping Maps to UIAccessibilityTraits Maps to AccessibilityNodeInfo className
accessibilityHint Read after label with delay Often ignored or read differently
Live regions accessibilityLiveRegion (limited) accessibilityLiveRegion: "polite" | "assertive"
Custom actions Rotor actions (swipe up/down) Action menu (local context menu)
Grouping accessible={true} on parent importantForAccessibility="yes"

2.3 Common Pitfalls

// BAD: Image without label — screen reader says "image"
<Image source={productImage} />

// GOOD: Descriptive label
<Image
  source={productImage}
  accessible={true}
  accessibilityLabel="Red running shoes, Nike Air Max"
  accessibilityRole="image"
/>

// BAD: Decorative image announced by screen reader
<Image source={decorativeDivider} accessible={true} />

// GOOD: Hide decorative elements
<Image
  source={decorativeDivider}
  accessible={false}
  importantForAccessibility="no-hide-descendants"
/>

// BAD: Redundant label repeats visible text
<TouchableOpacity accessibilityLabel="Submit button">
  <Text>Submit</Text>
</TouchableOpacity>

// GOOD: Label matches visible text (WCAG 2.5.3)
<TouchableOpacity
  accessibilityRole="button"
  accessibilityLabel="Submit"
>
  <Text>Submit</Text>
</TouchableOpacity>
The accessible={true} Grouping Trap

Setting accessible={true} on a parent View makes it a single focusable unit. All children are read as one concatenated label. This is useful for cards (read "Product name, $29.99, 4 stars") but harmful when children need independent interaction (buttons inside a card). Test every grouped element with a screen reader to verify the reading makes sense.

3. Focus Management

Focus management controls where the screen reader cursor moves after state changes — modal opens, navigation transitions, error messages, dynamic content updates. Poor focus management is the #1 cause of screen reader confusion.

3.1 Focus After Navigation

import { useFocusEffect } from '@react-navigation/native';
import { AccessibilityInfo, findNodeHandle } from 'react-native';

function ProductDetailScreen() {
  const headingRef = useRef<Text>(null);

  useFocusEffect(
    useCallback(() => {
      const timer = setTimeout(() => {
        const node = findNodeHandle(headingRef.current);
        if (node) {
          AccessibilityInfo.setAccessibilityFocus(node);
        }
      }, 500); // delay for transition animation

      return () => clearTimeout(timer);
    }, [])
  );

  return (
    <View>
      <Text
        ref={headingRef}
        accessibilityRole="header"
      >
        Product Details
      </Text>
      {/* ... */}
    </View>
  );
}

3.2 Focus Trapping in Modals

function AccessibleModal({ visible, onClose, children }: Props) {
  const modalRef = useRef<View>(null);

  useEffect(() => {
    if (visible) {
      const timer = setTimeout(() => {
        const node = findNodeHandle(modalRef.current);
        if (node) AccessibilityInfo.setAccessibilityFocus(node);
      }, 300);
      return () => clearTimeout(timer);
    }
  }, [visible]);

  if (!visible) return null;

  return (
    <Modal
      transparent
      visible={visible}
      onRequestClose={onClose}
    >
      <View
        ref={modalRef}
        accessible={true}
        accessibilityViewIsModal={true}
        accessibilityLabel="Dialog"
      >
        {children}
        <Pressable
          accessibilityRole="button"
          accessibilityLabel="Close dialog"
          onPress={onClose}
        >
          <Text>Close</Text>
        </Pressable>
      </View>
    </Modal>
  );
}
accessibilityViewIsModal

accessibilityViewIsModal={true} tells VoiceOver to ignore sibling views, effectively trapping focus within the modal. On Android, the Modal component handles this natively. Without this prop, VoiceOver users can swipe past the modal and interact with content behind it — a critical accessibility violation.

4. Dynamic Font Scaling

iOS Dynamic Type and Android font scaling allow users to increase text size system-wide. React Native respects this by default for <Text>, but layouts often break at larger sizes because they were designed for a single font size.

4.1 Handling Font Scale

import { PixelRatio, useWindowDimensions } from 'react-native';

function useFontScale() {
  const fontScale = PixelRatio.getFontScale();

  return {
    fontScale,
    isLargeText: fontScale > 1.3,
    isExtraLargeText: fontScale > 1.8,
  };
}

function AdaptiveCard({ title, subtitle, action }: Props) {
  const { isLargeText } = useFontScale();

  return (
    <View style={isLargeText ? styles.cardVertical : styles.cardHorizontal}>
      <View style={styles.textContainer}>
        <Text
          style={styles.title}
          numberOfLines={isLargeText ? 3 : 1}
          adjustsFontSizeToFit={false}
        >
          {title}
        </Text>
        <Text style={styles.subtitle}>{subtitle}</Text>
      </View>
      <Pressable style={styles.actionButton}>
        <Text>{action}</Text>
      </Pressable>
    </View>
  );
}

4.2 Font Scaling Limits

// Cap maximum font scale to prevent layout breakage
// while still supporting larger text
<Text
  style={{ fontSize: 16 }}
  maxFontSizeMultiplier={1.5}  // caps at 24pt
>
  This text scales up to 1.5x but no further
</Text>

// For icons that use text-based icon fonts
<Text
  style={{ fontFamily: 'MaterialIcons', fontSize: 24 }}
  allowFontScaling={false}  // icons should not scale
>
  {iconCharacter}
</Text>
Never Disable Font Scaling Globally

Setting Text.defaultProps.allowFontScaling = false is a common anti-pattern that breaks accessibility for all low-vision users. Instead, design layouts that accommodate 200% text scaling (WCAG 1.4.4). Use maxFontSizeMultiplier on specific elements where layout constraints are unavoidable, and test at every scale factor from 1.0 to 2.0.

5. Color Contrast

WCAG 2.1 AA requires a minimum contrast ratio of 4.5:1 for normal text and 3:1 for large text (18pt+ or 14pt+ bold). This applies to both light and dark themes.

5.1 Contrast Ratio Calculation

function getLuminance(hex: string): number {
  const rgb = hexToRgb(hex);
  const [r, g, b] = rgb.map(c => {
    c = c / 255;
    return c <= 0.03928
      ? c / 12.92
      : Math.pow((c + 0.055) / 1.055, 2.4);
  });
  return 0.2126 * r + 0.7152 * g + 0.0722 * b;
}

function getContrastRatio(fg: string, bg: string): number {
  const l1 = getLuminance(fg);
  const l2 = getLuminance(bg);
  const lighter = Math.max(l1, l2);
  const darker = Math.min(l1, l2);
  return (lighter + 0.05) / (darker + 0.05);
}

// Usage in theme validation
function validateThemeContrast(theme: ThemeTokens): ContrastIssue[] {
  const issues: ContrastIssue[] = [];
  const ratio = getContrastRatio(theme.colors.onBackground, theme.colors.background);

  if (ratio < 4.5) {
    issues.push({
      pair: 'onBackground / background',
      ratio,
      required: 4.5,
      wcag: '1.4.3',
    });
  }
  return issues;
}

5.2 Contrast Requirements Summary

Element WCAG Criterion Minimum Ratio Level
Normal text (< 18pt) 1.4.3 4.5:1 AA
Large text (≥ 18pt or 14pt bold) 1.4.3 3:1 AA
UI components (borders, icons) 1.4.11 3:1 AA
Focus indicators 2.4.7 3:1 (against adjacent) AA
Enhanced text contrast 1.4.6 7:1 AAA

6. Reduced Motion

Users with vestibular disorders, motion sensitivity, or seizure conditions enable "Reduce Motion" in system settings. The app must detect this preference and replace animations with instant transitions or subtle fades.

6.1 Detecting Reduced Motion

import { AccessibilityInfo } from 'react-native';

function useReducedMotion(): boolean {
  const [reduceMotion, setReduceMotion] = useState(false);

  useEffect(() => {
    AccessibilityInfo.isReduceMotionEnabled().then(setReduceMotion);

    const subscription = AccessibilityInfo.addEventListener(
      'reduceMotionChanged',
      setReduceMotion
    );
    return () => subscription.remove();
  }, []);

  return reduceMotion;
}

// Usage with Reanimated
function AnimatedCard({ children }: Props) {
  const reduceMotion = useReducedMotion();
  const translateY = useSharedValue(100);

  useEffect(() => {
    translateY.value = withTiming(0, {
      duration: reduceMotion ? 0 : 300,
      easing: Easing.out(Easing.cubic),
    });
  }, []);

  const animatedStyle = useAnimatedStyle(() => ({
    transform: [{ translateY: translateY.value }],
    opacity: reduceMotion ? 1 : interpolate(translateY.value, [100, 0], [0, 1]),
  }));

  return <Animated.View style={animatedStyle}>{children}</Animated.View>;
}

6.2 Navigation Transitions

// Disable slide animations when reduced motion is enabled
const screenOptions = {
  animation: reduceMotion ? 'none' : 'default',
  animationDuration: reduceMotion ? 0 : 350,
};

// For react-navigation
<Stack.Navigator
  screenOptions={{
    animation: reduceMotion ? 'fade' : 'slide_from_right',
    animationDuration: reduceMotion ? 150 : 350,
  }}
>

7. Screen Reader Announcements

When content changes dynamically (loading states, form validation, toast messages), screen readers must be notified. Without announcements, users have no way to know something changed.

7.1 Announcement API

import { AccessibilityInfo, Platform } from 'react-native';

function announce(message: string): void {
  AccessibilityInfo.announceForAccessibility(message);
}

// Announce form validation errors
function FormField({ label, error, ...props }: Props) {
  useEffect(() => {
    if (error) {
      announce(`Error: ${label} — ${error}`);
    }
  }, [error]);

  return (
    <View>
      <Text accessibilityRole="text">{label}</Text>
      <TextInput
        accessibilityLabel={label}
        accessibilityState={{ invalid: !!error }}
        {...props}
      />
      {error && (
        <Text
          accessibilityRole="alert"
          accessibilityLiveRegion="assertive"
          style={styles.error}
        >
          {error}
        </Text>
      )}
    </View>
  );
}

7.2 Live Region Patterns

// Loading state announcement
function LoadingIndicator({ loading }: { loading: boolean }) {
  useEffect(() => {
    if (loading) announce('Loading content');
    else announce('Content loaded');
  }, [loading]);

  return loading ? <ActivityIndicator accessibilityLabel="Loading" /> : null;
}

// Toast notification with live region
function Toast({ message, visible }: Props) {
  return visible ? (
    <View
      accessibilityRole="alert"
      accessibilityLiveRegion="polite"
    >
      <Text>{message}</Text>
    </View>
  ) : null;
}

8. Custom Accessibility Actions

Custom actions allow screen reader users to perform operations that sighted users do with gestures (swipe to delete, long press for options). Without custom actions, these features are invisible to assistive technology users.

8.1 Swipe-to-Delete Alternative

function SwipeableListItem({ item, onDelete, onArchive }: Props) {
  const accessibilityActions = [
    { name: 'delete', label: 'Delete item' },
    { name: 'archive', label: 'Archive item' },
  ];

  const onAccessibilityAction = (event: AccessibilityActionEvent) => {
    switch (event.nativeEvent.actionName) {
      case 'delete':
        onDelete(item.id);
        announce(`${item.title} deleted`);
        break;
      case 'archive':
        onArchive(item.id);
        announce(`${item.title} archived`);
        break;
    }
  };

  return (
    <View
      accessible={true}
      accessibilityLabel={`${item.title}, ${item.subtitle}`}
      accessibilityActions={accessibilityActions}
      onAccessibilityAction={onAccessibilityAction}
    >
      <Text>{item.title}</Text>
      <Text>{item.subtitle}</Text>
    </View>
  );
}

8.2 Standard vs Custom Actions

Action Type VoiceOver Gesture TalkBack Gesture Use Case
activate Double tap Double tap Primary action (built-in)
increment Swipe up Volume up gesture Sliders, steppers
decrement Swipe down Volume down gesture Sliders, steppers
Custom (delete) Rotor → Actions → Swipe Local context menu Swipe-to-delete alternative
Custom (share) Rotor → Actions → Swipe Local context menu Long-press menu alternative

9. Automated Accessibility Testing

Automated testing catches a subset of accessibility issues (missing labels, incorrect roles, contrast violations) early in the development cycle. It cannot replace manual screen reader testing but prevents regressions.

9.1 Component-Level Testing

import { render, screen } from '@testing-library/react-native';

describe('ProductCard accessibility', () => {
  it('has accessible label with product info', () => {
    render(<ProductCard name="Shoes" price={29.99} />);

    const card = screen.getByRole('button');
    expect(card.props.accessibilityLabel)
      .toBe('Shoes, $29.99');
  });

  it('announces loading state', () => {
    const { rerender } = render(
      <ProductCard name="Shoes" loading={true} />
    );

    expect(screen.getByLabelText('Loading')).toBeTruthy();
  });

  it('supports delete custom action', () => {
    const onDelete = jest.fn();
    render(<ProductCard name="Shoes" onDelete={onDelete} />);

    const card = screen.getByRole('button');
    expect(card.props.accessibilityActions)
      .toContainEqual({ name: 'delete', label: 'Delete item' });
  });

  it('has minimum touch target size', () => {
    const { getByRole } = render(<ProductCard name="Shoes" />);
    const button = getByRole('button');

    const { height, width } = button.props.style;
    expect(height).toBeGreaterThanOrEqual(44);
    expect(width).toBeGreaterThanOrEqual(44);
  });
});

Staff-Level Mental Model

Accessibility is a cross-cutting quality attribute like performance or security — it cannot be bolted on after the fact without significant rework. The architect's role is to embed accessibility into the component library, design system, code review checklist, and CI pipeline so that accessible behavior is the default path, not an opt-in effort. Every shared component should ship with correct roles, labels, and focus behavior. Every PR template should include an accessibility checklist. Every sprint should allocate time for screen reader testing. This systemic approach is the only way to maintain WCAG compliance at scale.
