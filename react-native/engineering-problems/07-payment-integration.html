1. Payment Architecture Overview

Mobile payments split into two fundamentally different models: direct payment processing (Stripe, Braintree) where you control the payment flow and pay ~2.9% + $0.30 per transaction, and in-app purchases (Apple/Google) where the store controls the flow and takes 15-30% commission. The choice is dictated by what you're selling — digital goods must use IAP per store policies.

Payment Architecture Decision Tree ═══════════════════════════════════ What are you selling? │ ├── Physical goods / services ──▶ Stripe / Braintree │ (food delivery, ride-share, (2.9% + $0.30) │ e-commerce, SaaS consumed │ outside the app) │ ├── Digital goods consumed ─────▶ In-App Purchase (REQUIRED) │ inside the app (15-30% commission) │ (premium features, coins, │ subscriptions, content) │ └── Reader apps ────────────────▶ External link (no in-app payment) (Netflix, Spotify, Kindle) (0% commission, but no in-app purchase flow allowed) ┌─────────────────────────────────────────────────────────────┐ │ CRITICAL: Apple rejects apps that use Stripe/Braintree │ │ for digital goods. Google is slightly more lenient but │ │ still enforces for most categories. Violating this policy │ │ results in app rejection or removal. │ └─────────────────────────────────────────────────────────────┘

2. Stripe SDK Integration

Stripe's React Native SDK provides a PCI-compliant payment sheet that handles card input, 3D Secure authentication, Apple Pay, Google Pay, and saved payment methods — all without sensitive card data touching your servers.

2.1 Setup and Payment Sheet

import { StripeProvider, useStripe } from '@stripe/stripe-react-native';

function App() {
  return (
    <StripeProvider
      publishableKey="pk_live_..."
      merchantIdentifier="merchant.com.myapp"
      urlScheme="myapp"
    >
      <PaymentScreen />
    </StripeProvider>
  );
}

function PaymentScreen() {
  const { initPaymentSheet, presentPaymentSheet } = useStripe();
  const [loading, setLoading] = useState(false);

  const initializePayment = async (amount: number) => {
    const { paymentIntent, ephemeralKey, customer } =
      await fetchPaymentSheetParams(amount);

    const { error } = await initPaymentSheet({
      merchantDisplayName: 'My App',
      customerId: customer,
      customerEphemeralKeySecret: ephemeralKey,
      paymentIntentClientSecret: paymentIntent,
      allowsDelayedPaymentMethods: true,
      defaultBillingDetails: { name: 'User Name' },
      applePay: { merchantCountryCode: 'US' },
      googlePay: { merchantCountryCode: 'US', testEnv: __DEV__ },
      returnURL: 'myapp://stripe-redirect',
    });

    if (error) throw error;
  };

  const handlePayment = async () => {
    setLoading(true);
    try {
      await initializePayment(2999);
      const { error } = await presentPaymentSheet();

      if (error) {
        if (error.code === 'Canceled') return;
        showError(error.message);
      } else {
        showSuccess('Payment confirmed!');
      }
    } finally {
      setLoading(false);
    }
  };
}

2.2 Server-Side Payment Intent

// Node.js backend — create PaymentIntent
const stripe = require('stripe')('sk_live_...');

async function createPaymentSheet(req, res) {
  const { amount, currency, userId } = req.body;

  let customer = await getStripeCustomerId(userId);
  if (!customer) {
    const created = await stripe.customers.create({
      metadata: { userId },
    });
    customer = created.id;
    await saveStripeCustomerId(userId, customer);
  }

  const ephemeralKey = await stripe.ephemeralKeys.create(
    { customer },
    { apiVersion: '2023-10-16' }
  );

  const paymentIntent = await stripe.paymentIntents.create({
    amount,
    currency: currency || 'usd',
    customer,
    automatic_payment_methods: { enabled: true },
    metadata: { userId, orderId: req.body.orderId },
  });

  res.json({
    paymentIntent: paymentIntent.client_secret,
    ephemeralKey: ephemeralKey.secret,
    customer,
  });
}
Never Trust the Client

The payment amount must be set server-side when creating the PaymentIntent. Never accept the amount from the client — a malicious user can modify the request to pay $0.01 for a $100 item. The client only receives the client_secret to confirm the payment; it cannot modify the amount.

3. In-App Purchases (StoreKit 2 & Google Play Billing)

In-app purchases are mandatory for digital goods. The react-native-iap library provides a unified API across both platforms, but the underlying systems (StoreKit 2 and Google Play Billing Library v5+) have significant differences.

3.1 Product Types

Type Apple Term Google Term Example
Consumable Consumable INAPP (consumed) Coins, gems, credits
Non-consumable Non-consumable INAPP (not consumed) Remove ads, unlock level
Auto-renewable sub Auto-renewable SUBS Monthly premium
Non-renewing sub Non-renewing INAPP (with expiry) Season pass

3.2 Purchase Flow Implementation

import {
  initConnection, getProducts, requestPurchase,
  finishTransaction, purchaseUpdatedListener,
  purchaseErrorListener, type ProductPurchase,
} from 'react-native-iap';

const PRODUCT_IDS = ['premium_monthly', 'premium_yearly', 'coins_100'];

class IAPManager {
  private purchaseListener: any;
  private errorListener: any;

  async initialize() {
    await initConnection();

    this.purchaseListener = purchaseUpdatedListener(
      async (purchase: ProductPurchase) => {
        const receipt = purchase.transactionReceipt;
        if (!receipt) return;

        const validated = await this.validateReceipt(purchase);

        if (validated.success) {
          await finishTransaction({ purchase, isConsumable: validated.isConsumable });
          this.onPurchaseSuccess(validated);
        } else {
          this.onPurchaseFailure(validated.error);
        }
      }
    );

    this.errorListener = purchaseErrorListener((error) => {
      if (error.code === 'E_USER_CANCELLED') return;
      this.onPurchaseFailure(error.message);
    });
  }

  async getAvailableProducts() {
    return getProducts({ skus: PRODUCT_IDS });
  }

  async purchase(productId: string) {
    await requestPurchase({ sku: productId });
  }

  cleanup() {
    this.purchaseListener?.remove();
    this.errorListener?.remove();
  }
}
Pending Purchases

Both platforms support "pending" purchases — transactions that require additional action (parental approval, slow payment method). Your app must handle the PENDING state: don't grant the entitlement yet, but don't show an error either. Listen for the purchase to transition to PURCHASED state, which may happen minutes or days later.

4. Subscription Management

Subscriptions are the most complex payment type. The lifecycle includes trial periods, grace periods, billing retries, upgrades/downgrades, cancellation, and resubscription — each with platform-specific behavior.

4.1 Subscription Lifecycle

Subscription State Machine ══════════════════════════ ┌──────────┐ purchase ┌──────────┐ │ None │───────────────▶│ Trial │ └──────────┘ └────┬─────┘ │ trial ends ▼ ┌──────────┐ billing ┌──────────┐ │ Active │───failure───▶│ Grace │ └────┬─────┘ │ Period │ │ └────┬─────┘ cancel│ retry │ │ success │ ▼ │ │ ┌──────────┐ │ │ retry │ Will Not │◀──────────────┘ │ exhausted │ Renew │ │ └────┬─────┘ ▼ │ period ends ┌──────────┐ ▼ │ Billing │ ┌──────────┐ │ Retry │ │ Expired │◀─────────────│ Period │ └────┬─────┘ all retries └──────────┘ │ exhausted resubscribe │ ▼ ┌──────────┐ │ Active │ └──────────┘

4.2 Server-Side Subscription Status

interface SubscriptionStatus {
  productId: string;
  state: 'trial' | 'active' | 'grace_period' | 'billing_retry'
       | 'expired' | 'revoked';
  expiresAt: Date;
  willRenew: boolean;
  platform: 'ios' | 'android';
  originalTransactionId: string;
}

class SubscriptionService {
  async checkEntitlement(userId: string): Promise<boolean> {
    const sub = await this.getSubscription(userId);
    if (!sub) return false;

    const entitled = ['trial', 'active', 'grace_period'];
    return entitled.includes(sub.state) && sub.expiresAt > new Date();
  }

  async handleServerNotification(notification: StoreNotification) {
    switch (notification.type) {
      case 'DID_RENEW':
        await this.extendSubscription(notification);
        break;
      case 'DID_FAIL_TO_RENEW':
        await this.enterGracePeriod(notification);
        break;
      case 'EXPIRED':
        await this.revokeEntitlement(notification);
        break;
      case 'REFUND':
        await this.handleRefund(notification);
        break;
    }
  }
}

5. Receipt Validation

Receipt validation is the security backbone of in-app purchases. Without server-side validation, users can use jailbreak tweaks or modified APKs to fake purchase receipts and get premium features for free.

5.1 Validation Flow

Receipt Validation Pipeline ═══════════════════════════ Client Your Server Store Server ────── ─────────── ──────────── Purchase completes │ ▼ Send receipt ────────▶ Receive receipt to your server │ ▼ Validate with ──────────▶ Apple: /verifyReceipt store server Google: purchases.products.get │ ◀────────────────── Validation response │ ▼ Check response: • Valid signature? • Correct product ID? • Not already used? • Transaction ID unique? │ ▼ Grant entitlement ──────▶ Client receives in your database entitlement status NEVER validate on the client — receipts can be forged.

5.2 Apple App Store Server API v2

import { SignedDataVerifier } from '@apple/app-store-server-library';

class AppleReceiptValidator {
  private verifier: SignedDataVerifier;

  constructor() {
    this.verifier = new SignedDataVerifier(
      appleCertificates,
      true,
      Environment.PRODUCTION,
      bundleId
    );
  }

  async validateTransaction(signedTransaction: string) {
    const transaction = await this.verifier
      .verifyAndDecodeTransaction(signedTransaction);

    return {
      transactionId: transaction.transactionId,
      productId: transaction.productId,
      purchaseDate: new Date(transaction.purchaseDate),
      expiresDate: transaction.expiresDate
        ? new Date(transaction.expiresDate)
        : null,
      isUpgraded: transaction.isUpgraded,
      environment: transaction.environment,
    };
  }
}
Architect Insight — RevenueCat

For most teams, building and maintaining a receipt validation and subscription management backend is not worth the engineering investment. RevenueCat handles receipt validation, entitlement management, server notifications, analytics, and cross-platform subscription status for both Apple and Google — with a React Native SDK. It abstracts the most error-prone parts of IAP. The trade-off is a per-transaction fee (free up to $2.5K MTR, then 1%) and vendor dependency.

6. PCI Compliance

PCI DSS (Payment Card Industry Data Security Standard) governs how card data is handled. Mobile apps that use Stripe or Braintree SDKs achieve PCI compliance by never touching raw card numbers — the SDK tokenizes card data directly to the payment processor.

6.1 Compliance Levels

Approach PCI Scope Your Responsibility Audit Required
Stripe Payment Sheet SAQ A (minimal) Use SDK correctly, HTTPS only Self-assessment questionnaire
Stripe CardField SAQ A-EP Secure hosting, no card data logging Self-assessment + vulnerability scan
Custom card input SAQ D (full scope) Full PCI audit, network segmentation On-site audit by QSA
Never Log Card Data

Even with Stripe SDK, ensure your logging, analytics, and crash reporting tools do not capture card numbers, CVVs, or full expiry dates. Audit your network interceptors, Redux DevTools serialization, and error boundary payloads. A single log line containing a card number puts you in full PCI scope.

7. Secure Payment Flow

A secure payment flow must prevent amount tampering, replay attacks, race conditions, and man-in-the-middle attacks. The principle is simple: the client initiates, but the server decides.

7.1 End-to-End Secure Flow

class SecurePaymentFlow {
  async initiatePayment(orderId: string): Promise<PaymentResult> {
    const idempotencyKey = uuid();

    const { clientSecret, orderId: confirmedOrderId } =
      await api.post('/payments/create', {
        orderId,
        idempotencyKey,
      });

    const { error, paymentIntent } = await confirmPayment(clientSecret);

    if (error) {
      await api.post('/payments/failed', {
        orderId: confirmedOrderId,
        errorCode: error.code,
        idempotencyKey,
      });
      throw new PaymentError(error.code, error.message);
    }

    const confirmation = await api.post('/payments/confirm', {
      paymentIntentId: paymentIntent.id,
      orderId: confirmedOrderId,
      idempotencyKey,
    });

    return confirmation;
  }
}
Idempotency Keys

Generate a unique idempotency key per payment attempt and send it with every API call in the flow. If the network drops and the client retries, the server recognizes the duplicate key and returns the original result instead of creating a double charge. Stripe supports idempotency keys natively on all API endpoints.

8. Error Handling & Retry

Payment errors fall into categories that determine whether retry is appropriate. Retrying a declined card wastes the user's time; not retrying a network timeout loses a sale.

8.1 Error Classification

Error Type Retryable User Action Example
Network error Yes (auto) None — retry silently Timeout, connection reset
Server error (5xx) Yes (with backoff) "Please try again" Stripe outage, server crash
Card declined No "Try a different card" Insufficient funds, expired
3DS authentication failed Yes (user retry) "Please authenticate again" User cancelled 3DS
Invalid request No Bug — log and alert Invalid amount, missing field
Rate limited Yes (after delay) "Please wait and try again" Too many attempts

8.2 Retry with Exponential Backoff

async function retryablePaymentCall<T>(
  fn: () => Promise<T>,
  maxRetries = 3
): Promise<T> {
  let lastError: Error;

  for (let attempt = 0; attempt < maxRetries; attempt++) {
    try {
      return await fn();
    } catch (error) {
      lastError = error;

      if (!isRetryable(error)) throw error;

      const delay = Math.min(
        1000 * Math.pow(2, attempt) + Math.random() * 1000,
        10000
      );
      await new Promise(r => setTimeout(r, delay));
    }
  }

  throw lastError!;
}

function isRetryable(error: any): boolean {
  if (error.type === 'StripeConnectionError') return true;
  if (error.type === 'StripeAPIError' && error.statusCode >= 500) return true;
  if (error.type === 'StripeRateLimitError') return true;
  return false;
}

9. Testing Payment Flows

Payment testing requires sandbox environments, test cards, and simulated edge cases. You cannot test payments in production — both Stripe and the app stores provide dedicated test infrastructure.

9.1 Testing Matrix

Scenario Stripe Test Apple Sandbox Google Test
Successful payment 4242424242424242 Sandbox Apple ID License test account
Declined card 4000000000000002 N/A (always succeeds) N/A
3DS required 4000002500003155 N/A N/A
Subscription renewal Test clock API Accelerated (5 min = 1 month) Test subscription
Refund Dashboard / API Sandbox refund voided_purchases API
Network failure Mock / intercept Airplane mode during purchase Airplane mode
Apple Sandbox Accelerated Time

Apple Sandbox accelerates subscription durations: a 1-month subscription renews every 5 minutes, a 1-year subscription renews every 1 hour. Subscriptions auto-renew up to 6 times in sandbox before expiring. This lets you test the full renewal lifecycle in minutes instead of months — but it also means your server notification handler must be robust against rapid-fire renewal events.

10. Architect Considerations

10.1 Choosing a Payment Strategy

  • Stripe vs IAP is not a choice — it's a policy: Digital goods consumed in-app must use IAP. Physical goods and services must use direct payment. Hybrid apps (e.g., a marketplace with both) need both systems.
  • Webhook reliability: Stripe webhooks and App Store Server Notifications can be delayed, duplicated, or arrive out of order. Your handler must be idempotent and handle events arriving in any sequence. Store the raw event payload for debugging.
  • Cross-platform subscriptions: A user who subscribes on iOS and switches to Android expects their subscription to work. This requires server-side entitlement management that checks both Apple and Google subscription status for each user.
  • Grace periods: Both Apple (6-16 days) and Google (3-7 days) offer grace periods where the subscription remains active during billing retry. Respect these — revoking access during a grace period causes churn.
  • Price localization: App Store and Play Store handle currency conversion for IAP. For Stripe, you must manage multi-currency pricing yourself. Always display prices from the store API, never hardcode them.
  • Refund handling: Apple can refund without notifying you (consumption API). Google sends RTDN (Real-Time Developer Notifications). Your server must handle refunds by revoking entitlements and optionally flagging abuse.
Staff-Level Mental Model

Payment integration is a distributed transaction problem with real money at stake. The client, your server, and the payment provider form a three-party system where any link can fail at any time. The cardinal rules: never trust the client for amounts, always validate server-side, make every operation idempotent, and design for the failure case first. A payment system that works 99% of the time but double-charges 1% of users will destroy trust faster than any bug. Test the unhappy paths more than the happy paths.
