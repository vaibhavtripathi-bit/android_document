1. Offline-First Philosophy

Offline-first is not "add caching." It's an architectural inversion: the local database is the primary data source, and the network is a sync channel that eventually reconciles local state with the server. Every read comes from local storage; every write goes to local storage first, then syncs.

Traditional (Online-First) Offline-First ────────────────────────── ───────────── UI ──► API ──► Server DB UI ──► Local DB ──► Sync Engine ──► Server ◄── ◄── ◄── ◄── Failure mode: spinner/error Failure mode: works, syncs later Latency: network round-trip Latency: disk read (~1-5ms) Offline: broken Offline: fully functional
Architect Insight

The fundamental trade-off of offline-first is availability over consistency (CAP theorem applied to the client). You're choosing to always show data and accept writes, even when you can't verify they're consistent with the server. This means you must have a conflict resolution strategy — it's not optional, it's a first-class architectural concern.

2.2 Schema Design for Sync

-- Every synced table needs these metadata columns
CREATE TABLE tasks (
  id            TEXT PRIMARY KEY,   -- UUID (client-generated)
  title         TEXT NOT NULL,
  description   TEXT,
  status        TEXT DEFAULT 'todo',
  assigned_to   TEXT,
  project_id    TEXT NOT NULL,

  -- Sync metadata
  created_at    INTEGER NOT NULL,   -- client timestamp
  updated_at    INTEGER NOT NULL,   -- last local modification
  server_ts     INTEGER,            -- last known server timestamp
  sync_status   TEXT DEFAULT 'created', -- created|updated|synced|deleted
  version       INTEGER DEFAULT 1,  -- optimistic locking version
  is_deleted    INTEGER DEFAULT 0   -- soft delete for sync
);

CREATE INDEX idx_tasks_sync
  ON tasks(sync_status) WHERE sync_status != 'synced';

CREATE INDEX idx_tasks_project
  ON tasks(project_id, status, updated_at DESC);
Never Hard Delete

In an offline-first system, you must soft delete records (set is_deleted = 1). Hard deletes create "tombstone" problems: if Device A deletes a record while Device B is offline, Device B has no way to know the record was deleted when it syncs — it would re-create it. Soft deletes with a sync timestamp allow the sync engine to propagate deletions correctly.

3. Sync Engine Architecture

The sync engine is the most complex component. It must push local changes to the server, pull remote changes, detect and resolve conflicts, and do all of this incrementally without re-downloading the entire dataset.

3.1 Pull-Push Sync Protocol

CLIENT SERVER │ │ │── PULL (lastSyncTs: 1700000) ────►│ │ │── Query: WHERE updated_at > 1700000 │◄── {changes: [...], ts: 1700050} ─│ │ │ │── Apply remote changes locally │ │ (detect conflicts) │ │ │ │── PUSH (localChanges: [...]) ────►│ │ │── Apply changes (with conflict check) │◄── {accepted: [...], │ │ rejected: [...], │ │ ts: 1700051} │ │ │ │── Mark accepted as synced │ │ Resolve rejected (conflicts) │ │ │ │── Update lastSyncTs = 1700051 │

3.2 Sync Engine Implementation

class SyncEngine {
  private db: Database;
  private api: SyncAPI;
  private conflictResolver: ConflictResolver;

  async sync(): Promise<SyncResult> {
    const lastSync = await this.getLastSyncTimestamp();

    // Phase 1: Pull remote changes
    const remote = await this.api.pull(lastSync);
    const pullConflicts = await this.applyRemoteChanges(remote.changes);

    // Phase 2: Push local changes
    const localChanges = await this.getUnsynced();
    const pushResult = await this.api.push(localChanges);

    // Phase 3: Handle results
    await this.markSynced(pushResult.accepted);
    const pushConflicts = await this.resolveConflicts(pushResult.rejected);

    // Phase 4: Update sync checkpoint
    await this.setLastSyncTimestamp(remote.serverTimestamp);

    return {
      pulled: remote.changes.length,
      pushed: pushResult.accepted.length,
      conflicts: [...pullConflicts, ...pushConflicts],
    };
  }

  private async applyRemoteChanges(changes: Change[]): Promise<Conflict[]> {
    const conflicts: Conflict[] = [];
    await this.db.transaction(async (tx) => {
      for (const change of changes) {
        const local = await tx.find(change.table, change.id);
        if (local?.sync_status !== 'synced') {
          conflicts.push({ local, remote: change });
        } else {
          await tx.upsert(change.table, {
            ...change.data, sync_status: 'synced'
          });
        }
      }
    });
    return conflicts;
  }
}

4. Conflict Resolution Strategies

Conflicts occur when the same record is modified on multiple devices (or client + server) before syncing. The resolution strategy is the most important architectural decision in an offline-first system.

4.1 Strategy Comparison

Strategy How It Works Data Loss Risk Complexity Best For
Last-Write-Wins (LWW) Highest timestamp wins High — silent overwrites Low Simple settings, preferences
Server-Wins Server version always wins Medium — local changes lost Low Authoritative server data
Client-Wins Client version always wins Medium — server changes lost Low User-owned personal data
Field-Level Merge Merge non-conflicting fields Low Medium Structured records (tasks, contacts)
CRDT Mathematically convergent types None High Collaborative editing
Manual Resolution Show both versions to user None High (UX) Critical business data

4.2 Field-Level Merge Implementation

class FieldLevelMerger implements ConflictResolver {
  resolve(
    base: Record,    // last synced version (common ancestor)
    local: Record,   // current local version
    remote: Record   // incoming server version
  ): MergeResult {
    const merged: Record = { ...base };
    const conflicts: FieldConflict[] = [];

    for (const field of Object.keys(base)) {
      const localChanged = local[field] !== base[field];
      const remoteChanged = remote[field] !== base[field];

      if (localChanged && !remoteChanged) {
        merged[field] = local[field];       // only local changed
      } else if (!localChanged && remoteChanged) {
        merged[field] = remote[field];      // only remote changed
      } else if (localChanged && remoteChanged) {
        if (local[field] === remote[field]) {
          merged[field] = local[field];     // same change — no conflict
        } else {
          conflicts.push({ field, local: local[field], remote: remote[field] });
          merged[field] = remote[field];    // default: server wins per-field
        }
      }
    }
    return { merged, conflicts };
  }
}
Architect Insight — Three-Way Merge

Field-level merge requires a base version (the common ancestor). Without it, you can't distinguish "both changed the same field" from "one changed, one didn't." This means you must store the last-synced version of every record locally. WatermelonDB and Realm both support this pattern natively. With raw SQLite, you need a shadow table or a JSON column storing the base snapshot.

5. Queue-Based Mutation Sync

Instead of syncing entire records, an alternative approach syncs operations (mutations). This preserves user intent and enables more sophisticated conflict resolution.

5.1 Operation Log Pattern

interface Mutation {
  id: string;
  table: string;
  recordId: string;
  operation: 'create' | 'update' | 'delete';
  fields?: Record<string, any>;  // only changed fields for updates
  timestamp: number;
  synced: boolean;
}

class MutationQueue {
  async record(mutation: Omit<Mutation, 'id' | 'synced'>): Promise<void> {
    await this.db.executeSql(
      `INSERT INTO mutation_log (id, tbl, record_id, op, fields, ts, synced)
       VALUES (?, ?, ?, ?, ?, ?, 0)`,
      [uuid(), mutation.table, mutation.recordId,
       mutation.operation, JSON.stringify(mutation.fields), mutation.timestamp]
    );
  }

  async flush(): Promise<void> {
    const pending = await this.db.executeSql(
      `SELECT * FROM mutation_log WHERE synced = 0 ORDER BY ts ASC`
    );
    const compressed = this.compressMutations(pending.rows);
    const result = await this.api.pushMutations(compressed);
    await this.markSynced(result.accepted);
  }

  private compressMutations(mutations: Mutation[]): Mutation[] {
    const byRecord = new Map<string, Mutation[]>();
    for (const m of mutations) {
      const key = `${m.table}:${m.recordId}`;
      if (!byRecord.has(key)) byRecord.set(key, []);
      byRecord.get(key)!.push(m);
    }
    // Compress: create+update → create, update+delete → delete
    return Array.from(byRecord.values()).map(this.compressGroup);
  }
}
Mutation Compression

Before syncing, compress the mutation log: if a record was created then updated 5 times, send a single "create" with the final state. If created then deleted, send nothing. This dramatically reduces sync payload size and server processing. The compression must be order-preserving across different records but can collapse operations on the same record.

6. Optimistic UI Updates

In an offline-first app, every write is optimistic by definition — the UI updates immediately from local state, and sync happens in the background. The challenge is handling rollbacks when the server rejects a change.

6.1 Optimistic Update with Rollback

function useOptimisticMutation<T>(table: string) {
  const db = useDatabase();
  const syncEngine = useSyncEngine();

  return useCallback(async (id: string, updates: Partial<T>) => {
    // 1. Snapshot for rollback
    const snapshot = await db.find(table, id);

    // 2. Apply locally (UI updates instantly via reactive query)
    await db.update(table, id, {
      ...updates,
      sync_status: 'updated',
      updated_at: Date.now(),
    });

    // 3. Record mutation for sync
    await syncEngine.mutationQueue.record({
      table, recordId: id,
      operation: 'update',
      fields: updates,
      timestamp: Date.now(),
    });

    // 4. Attempt immediate sync (non-blocking)
    syncEngine.requestSync().catch(async (err) => {
      if (err.code === 'CONFLICT') {
        await db.update(table, id, snapshot);
        showConflictUI(snapshot, updates, err.serverVersion);
      }
    });
  }, [db, syncEngine, table]);
}

7. Network State Detection

Accurate network detection determines when to attempt sync. The OS-reported network state is necessary but not sufficient — you need application-level reachability checks.

7.1 Multi-Layer Network Detection

class NetworkMonitor {
  private isOSOnline = true;
  private isServerReachable = false;
  private listeners = new Set<(online: boolean) => void>();

  start(): void {
    // Layer 1: OS network state (fast, but unreliable)
    NetInfo.addEventListener(state => {
      this.isOSOnline = state.isConnected ?? false;
      if (this.isOSOnline) this.checkServerReachability();
      else this.setReachable(false);
    });

    // Layer 2: Periodic server ping (accurate, but slow)
    setInterval(() => {
      if (this.isOSOnline) this.checkServerReachability();
    }, 30000);
  }

  private async checkServerReachability(): Promise<void> {
    try {
      const controller = new AbortController();
      setTimeout(() => controller.abort(), 5000);
      await fetch('https://api.example.com/health', {
        signal: controller.signal
      });
      this.setReachable(true);
    } catch {
      this.setReachable(false);
    }
  }
}

8. Background Sync

Background sync ensures data is pushed to the server even when the app isn't in the foreground. This is heavily constrained by OS power management policies.

8.2 React Native Background Sync

// Android: HeadlessJsTask for background sync
import { AppRegistry } from 'react-native';

const backgroundSyncTask = async () => {
  const syncEngine = getSyncEngineInstance();
  const networkMonitor = getNetworkMonitor();

  if (!await networkMonitor.isReachable()) return;

  try {
    const result = await syncEngine.sync();
    console.log(`Background sync: ${result.pushed} pushed, ${result.pulled} pulled`);
  } catch (err) {
    console.error('Background sync failed:', err);
  }
};

AppRegistry.registerHeadlessTask('BackgroundSync', () => backgroundSyncTask);
iOS Background Limitations

iOS is extremely restrictive with background execution. BGAppRefreshTask gives you ~30 seconds and the system decides when (and if) to run it based on user behavior patterns. For critical sync, use silent push notifications to wake the app: the server sends a content-available push when there's data to sync, giving the app ~30 seconds of background time. This is the pattern used by most production offline-first iOS apps.

9. Data Versioning

Data versioning enables conflict detection, audit trails, and the ability to roll back changes. There are two approaches: record-level versioning and operation-level versioning.

9.1 Optimistic Locking with Version Numbers

// Server-side: reject updates with stale versions
async function updateRecord(table: string, id: string, data: any, clientVersion: number) {
  const result = await db.query(
    `UPDATE ${table}
     SET data = $1, version = version + 1, updated_at = NOW()
     WHERE id = $2 AND version = $3
     RETURNING *`,
    [data, id, clientVersion]
  );

  if (result.rowCount === 0) {
    const current = await db.query(`SELECT * FROM ${table} WHERE id = $1`, [id]);
    throw new ConflictError({
      clientVersion,
      serverVersion: current.rows[0].version,
      serverData: current.rows[0],
    });
  }
  return result.rows[0];
}

10. Migration Strategies

Schema changes in an offline-first app are dangerous: the local database may have unsynced data in the old schema when the app updates. Migrations must be non-destructive and forward-compatible.

10.1 Migration Pipeline

const migrations: Migration[] = [
  {
    version: 2,
    up: async (db) => {
      await db.executeSql(`ALTER TABLE tasks ADD COLUMN priority TEXT DEFAULT 'medium'`);
    },
  },
  {
    version: 3,
    up: async (db) => {
      await db.executeSql(`CREATE TABLE labels (id TEXT PRIMARY KEY, name TEXT, color TEXT)`);
      await db.executeSql(`CREATE TABLE task_labels (task_id TEXT, label_id TEXT, PRIMARY KEY(task_id, label_id))`);
    },
  },
  {
    version: 4,
    up: async (db) => {
      // Rename column: SQLite doesn't support ALTER COLUMN, so recreate
      await db.executeSql(`ALTER TABLE tasks RENAME TO tasks_old`);
      await db.executeSql(`CREATE TABLE tasks (/* new schema */)`);
      await db.executeSql(`INSERT INTO tasks SELECT /* mapped columns */ FROM tasks_old`);
      await db.executeSql(`DROP TABLE tasks_old`);
    },
  },
];

async function runMigrations(db: SQLiteDatabase): Promise<void> {
  const currentVersion = await getDbVersion(db);
  const pending = migrations.filter(m => m.version > currentVersion);

  for (const migration of pending) {
    await db.transaction(async (tx) => {
      await migration.up(tx);
      await tx.executeSql(`PRAGMA user_version = ${migration.version}`);
    });
  }
}
Sync-Safe Migrations

When adding columns, always provide defaults so existing unsynced records remain valid. When removing columns, keep them in the local schema until all devices have synced their pending changes (use a "deprecated" flag and remove in a later version). Never rename columns in a single step — add the new column, migrate data, deprecate the old column, remove it in the next version.

Staff-Level Summary

Offline-first is an architectural commitment, not a feature toggle. It requires: (1) local-first reads and writes, (2) a sync engine with pull-push protocol, (3) conflict resolution as a first-class concern, (4) mutation queuing with compression, (5) background sync within OS constraints, and (6) non-destructive schema migrations. The hardest part isn't the implementation — it's choosing the right conflict resolution strategy for each data type and communicating conflicts to users without breaking their mental model.
