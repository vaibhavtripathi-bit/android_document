1. Why Migrate (and When Not To)

Migration from native to React Native is a multi-quarter investment with significant risk. The decision must be driven by concrete business and engineering metrics, not technology hype.

1.1 Migration Decision Matrix

Factor Favors Migration Favors Staying Native
Team composition More web/JS engineers than native Strong native team, hard to hire JS
Feature velocity Shipping same feature twice is bottleneck Platforms diverge intentionally
Code sharing 70%+ business logic is identical Platforms have unique features
Performance needs Standard CRUD/content app Heavy GPU, AR/VR, real-time audio
Native API depth Standard platform APIs Deep OS integration (widgets, extensions)
App size Medium (50-200 screens) Very large (500+ screens) or very small
Release cadence Want OTA updates, faster iteration Infrequent releases, heavy QA process
When NOT to Migrate

Do not migrate if: (1) the app's core value proposition depends on platform-specific capabilities (AR, real-time audio processing, custom rendering engines), (2) the native codebase is well-maintained with good test coverage and the team is productive, (3) the app has fewer than 20 screens (migration overhead exceeds benefit), or (4) leadership expects migration to be "done in a quarter" — realistic timelines are 6-18 months for medium apps.

2. Brownfield Integration Strategy

Brownfield integration means embedding React Native into an existing native app, rather than rewriting from scratch. This is the only viable approach for production apps — it allows incremental migration with continuous deployment.

2.2 iOS Integration Setup

// AppDelegate.swift — Initialize React Native
import React

class AppDelegate: UIResponder, UIApplicationDelegate {
  var reactBridge: RCTBridge?

  func application(_ application: UIApplication,
    didFinishLaunchingWithOptions opts: [UIApplication.LaunchOptionsKey: Any]?
  ) -> Bool {
    self.reactBridge = RCTBridge(
      delegate: self,
      launchOptions: opts
    )
    return true
  }
}

extension AppDelegate: RCTBridgeDelegate {
  func sourceURL(for bridge: RCTBridge) -> URL? {
    #if DEBUG
    return RCTBundleURLProvider.sharedSettings()
      .jsBundleURL(forBundleRoot: "index")
    #else
    return Bundle.main.url(forResource: "main", withExtension: "jsbundle")
    #endif
  }
}

// Presenting an RN screen from native
class NativeViewController: UIViewController {
  func openRNScreen(moduleName: String, props: [String: Any]) {
    guard let bridge = (UIApplication.shared.delegate as? AppDelegate)?.reactBridge
    else { return }

    let rootView = RCTRootView(
      bridge: bridge,
      moduleName: moduleName,
      initialProperties: props
    )
    let vc = UIViewController()
    vc.view = rootView
    navigationController?.pushViewController(vc, animated: true)
  }
}

// MainApplication.kt — Initialize React Native
class MainApplication : Application(), ReactApplication {
  override val reactNativeHost = object : DefaultReactNativeHost(this) {
    override fun getPackages() = PackageList(this).packages.apply {
      add(SharedDataModule())

    override fun getJSMainModuleName() = "index"
    override fun getUseDeveloperSupport() = BuildConfig.DEBUG
    override val isNewArchEnabled = true
    override val isHermesEnabled = true
  }

// Launching an RN screen from a native Activity
class ReactScreenActivity : ReactActivity() {
  override fun getMainComponentName() =
    intent.getStringExtra("moduleName") ?: "MainApp"

  override fun createReactActivityDelegate() =
    DefaultReactActivityDelegate(this, mainComponentName, fabricEnabled)
}

// From any native Activity:
fun openRNScreen(moduleName: String, props: Bundle) {
  val intent = Intent(this, ReactScreenActivity::class.java)
  intent.putExtra("moduleName", moduleName)
  intent.putExtras(props)
  startActivity(intent)
}
Single Bridge Instance

Always share a single RCTBridge / ReactInstanceManager across the entire app. Creating multiple bridges wastes memory (each bridge spins up its own JS runtime) and prevents state sharing between RN screens. Initialize the bridge at app launch and reuse it for every RN screen. Pre-warm it during the splash screen to hide initialization latency.

3. Shared Navigation

Navigation is the hardest integration problem. Native screens and RN screens must share a single navigation stack with seamless transitions, deep linking, and back button behavior.

3.1 Navigation Architecture Options

Approach How It Works Pros Cons
Native-owned navigation Native nav controller, RN screens as child views Native transitions, deep link compat RN can't control nav stack
RN-owned navigation React Navigation manages everything, native screens wrapped Unified nav logic in JS Hard to wrap complex native screens
Hybrid (recommended) Native owns top-level tabs/stack, RN owns sub-flows Pragmatic, incremental Two nav systems to coordinate

3.2 Native-to-RN Navigation Bridge

// NavigationBridge.ts — Bidirectional navigation
import { NativeModules, NativeEventEmitter } from 'react-native';

const { NavigationBridge } = NativeModules;
const emitter = new NativeEventEmitter(NavigationBridge);

export const navigation = {
  // RN → Native: navigate to a native screen
  pushNative(screenName: string, params: Record<string, any>) {
    NavigationBridge.pushNativeScreen(screenName, params);
  },

  // RN → Native: go back (pop current RN container)
  goBack() {
    NavigationBridge.popCurrentScreen();
  },

  // Native → RN: listen for navigation events
  onDeepLink(callback: (url: string) => void) {
    return emitter.addListener('onDeepLink', callback);
  },

  // Shared deep link resolution
  resolveRoute(url: string): Route {
    const parsed = parseDeepLink(url);
    if (MIGRATED_SCREENS.has(parsed.screen)) {
      return { type: 'rn', module: parsed.screen, params: parsed.params };
    }
    return { type: 'native', screen: parsed.screen, params: parsed.params };
  },
};

4. Incremental Migration Plan

Migration must be incremental — never attempt a big-bang rewrite. The plan should deliver value at every phase and allow stopping at any point without leaving the app in a broken state.

4.1 Four-Phase Migration

PHASE 1: Foundation (4-6 weeks) ├── Integrate RN into native project (brownfield setup) ├── Establish shared data layer (native modules for DB, auth, analytics) ├── Set up CI/CD for hybrid builds ├── Migrate 1 simple, low-risk screen as proof of concept └── Benchmark: RN screen perf vs native equivalent PHASE 2: Leaf Screens (8-12 weeks) ├── Migrate leaf screens (no complex navigation dependencies) ├── Settings, About, Help, Profile editing ├── Build shared component library ├── Establish testing patterns (Detox + native tests) └── Team ramp-up: pair programming, code reviews PHASE 3: Core Flows (12-20 weeks) ├── Migrate core user flows (feed, search, detail screens) ├── Transition navigation ownership to RN for migrated flows ├── Performance parity validation per screen ├── Feature flag each migrated screen for rollback └── A/B test: native vs RN for key metrics PHASE 4: Completion (8-12 weeks) ├── Migrate remaining screens ├── Remove native screen implementations (behind feature flags) ├── Transition to RN-owned navigation ├── Remove brownfield scaffolding └── Final performance audit and optimization pass

Architect Insight — Feature Flags

Every migrated screen must be behind a feature flag that can instantly revert to the native implementation. This is non-negotiable. The flag should be server-controlled (not a build-time flag) so you can roll back without an app store release. Use percentage-based rollouts: 1% → 10% → 50% → 100%, monitoring crash rates, performance metrics, and user engagement at each stage.

5. Feature Parity Assessment

Before migrating a screen, you must verify that React Native can replicate every behavior of the native implementation. Gaps must be identified early and addressed with native modules or design compromises.

5.1 Parity Checklist per Screen

Category Check Common Gaps
Visual fidelity Pixel-perfect match with native Custom fonts, shadows, blur effects
Animations Same transitions and micro-interactions Shared element transitions, spring physics
Accessibility VoiceOver/TalkBack parity Custom actions, focus management
Gestures Swipe, long press, pinch behaviors Simultaneous gesture recognition
Platform APIs Camera, location, notifications, etc. Background processing, HealthKit
Performance Frame rate, memory, startup impact Complex lists, heavy animations
Edge cases Low memory, interrupted flows, rotation State restoration, multitasking

6. Performance Benchmarking

Every migrated screen must meet or exceed the native version's performance. Measure before migration (native baseline) and after (RN implementation) on the same device.

6.1 Benchmark Protocol

// Automated benchmark comparison
interface ScreenBenchmark {
  screenName: string;
  implementation: 'native' | 'react-native';
  device: string;
  metrics: {
    ttRender: number;       // time to first meaningful render (ms)

    memoryPeak: number;     // peak memory during interaction (MB)
    memoryDelta: number;    // memory increase from opening screen (MB)

  };
}

// Acceptance criteria
const PERF_THRESHOLDS = {
  ttRenderMaxRegressionPct: 15,    // RN can be up to 15% slower

  memoryDeltaMaxRegressionPct: 25, // RN can use up to 25% more memory

};

7. Team Training and Transition

The hardest part of migration isn't the code — it's the people. Native iOS/Android engineers must learn React, JavaScript/TypeScript, and the React Native ecosystem. This takes months, not weeks.

7.1 Training Roadmap

TEAM TRANSITION TIMELINE Week 1-2: JavaScript/TypeScript fundamentals └── For native devs: async/await, closures, modules, types Week 3-4: React fundamentals └── Components, hooks, state, effects, reconciliation Week 5-6: React Native specifics └── Core components, styling, navigation, native modules Week 7-8: Project architecture └── State management, testing, CI/CD, brownfield patterns Week 9+: Paired migration └── Native dev + experienced RN dev migrate screens together └── Code reviews focused on RN patterns and anti-patterns

7.2 Team Structure During Migration

Role Responsibility Staffing
Migration Lead (Staff) Architecture, prioritization, risk management 1 dedicated
RN Platform Team Brownfield infra, native modules, shared components 2-3 engineers
Migration Squads Screen-by-screen migration, feature parity Pairs (native + RN)
Native Maintenance Bug fixes and critical features in native code 1-2 engineers (rotating)
QA/Test Parity testing, regression testing, performance 1-2 dedicated
Retaining Native Expertise

Even after full migration, you need native expertise for: native modules, platform-specific debugging, app store submissions, performance optimization at the native layer, and integrating new OS features. Plan for at least 1-2 engineers per platform to maintain native skills. These engineers become your "platform team" — they build the native infrastructure that RN screens consume.

8. Risk Mitigation

8.1 Risk Register

Risk Likelihood Impact Mitigation
Performance regression in migrated screens High High Per-screen benchmarks, feature flags, A/B testing
RN version upgrade breaks native integration Medium High Pin RN version, upgrade in dedicated sprints, test matrix
Team productivity drops during learning curve High Medium Pair programming, dedicated training time, gradual ramp
Critical native feature can't be replicated in RN Medium High Parity assessment upfront, keep native for complex screens
App size increases significantly High Medium Monitor per-release, Hermes bytecode, Proguard, asset optimization
Stakeholder loses patience, cancels migration Medium Critical Deliver value every phase, clear metrics, regular demos

8.2 Shared Data Layer

// Native module exposing shared database to RN
// This ensures both native and RN screens read/write the same data

// SharedDataModule.swift (iOS)
@objc(SharedDataModule)
class SharedDataModule: NSObject {
  @objc
  func getUserProfile(
    _ resolve: @escaping RCTPromiseResolveBlock,
    rejecter reject: @escaping RCTPromiseRejectBlock
  ) {
    let profile = CoreDataManager.shared.currentUserProfile()
    resolve(profile.toDictionary())
  }

  @objc
  func updateUserProfile(
    _ data: NSDictionary,
    resolve: @escaping RCTPromiseResolveBlock,
    rejecter reject: @escaping RCTPromiseRejectBlock
  ) {
    CoreDataManager.shared.updateProfile(data)
    resolve(nil)
  }
}

9. Rollback Plan

Every migration step must be reversible. The rollback plan is not a sign of failure — it's a sign of engineering maturity.

9.1 Rollback Mechanisms

ROLLBACK LAYERS Layer 1: Feature Flag (seconds) └── Server-side flag switches screen from RN → native No app update required. Instant rollback. Layer 2: OTA Update (minutes) └── Push a CodePush update that reverts the RN bundle Fixes JS-side bugs without app store release. Layer 3: App Store Release (hours/days) └── Ship a new binary with the RN screen removed Last resort for native integration issues. ROLLBACK DECISION TREE: Crash rate > 2x baseline? → Layer 1 (immediate flag off) Performance regression? → Layer 1 (flag off) + investigate JS bug in RN screen? → Layer 2 (OTA fix) or Layer 1 Native integration crash? → Layer 3 (binary release)

9.2 Feature Flag Implementation

// Native side: check flag before screen presentation
func navigateToProfile(userId: String) {
  if FeatureFlags.shared.isEnabled("rn_profile_screen") {
    openRNScreen(
      moduleName: "ProfileScreen",
      props: ["userId": userId]
    )
  } else {
    let vc = NativeProfileViewController(userId: userId)
    navigationController?.pushViewController(vc, animated: true)
  }
}

// Feature flag service with percentage rollout
class FeatureFlags {
  func isEnabled(_ flag: String) -> Bool {
    guard let config = remoteConfig[flag] else { return false }
    if config.killSwitch { return false }
    let userBucket = hash(userId) % 100
    return userBucket < config.rolloutPercentage
  }
}
Keep Native Code Alive

Do not delete native screen implementations until the RN replacement has been at 100% rollout for at least 2 release cycles (typically 4-6 weeks). The native code is your safety net. Mark it as deprecated, exclude it from active development, but keep it compilable. Only remove it when you're confident the RN version is stable and you've validated with crash-free metrics.

Staff-Level Summary

A native-to-RN migration is a organizational transformation disguised as a technical project. The Staff engineer's role is to: (1) make the business case with data (velocity metrics, hiring costs, time-to-market), (2) design the brownfield architecture that allows incremental migration, (3) establish the safety net (feature flags, rollback plans, performance benchmarks), (4) manage the human side (training, pairing, maintaining morale during the learning curve), and (5) communicate progress and risks to leadership continuously. The migration succeeds not when all screens are in RN, but when the team is more productive and the app is as good or better than before.
