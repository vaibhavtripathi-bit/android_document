Background Processing in React Native

Mobile operating systems aggressively restrict background execution to preserve battery. React Native apps must navigate Headless JS (Android), Background Fetch (iOS), WorkManager, BGTaskScheduler, and third-party libraries to perform work when the app isn't in the foreground — while respecting platform constraints and battery optimization policies.

1. Platform Background Constraints

iOS and Android have fundamentally different approaches to background execution. iOS is restrictive by default — apps get seconds, not minutes. Android is more permissive but has tightened restrictions with each API level since Android 8 (Oreo).

1.1 Platform Comparison

Background execution time ~30 seconds after backgrounding ~10 minutes (foreground service), varies
Background fetch frequency System-determined (adaptive) Configurable via WorkManager
Long-running tasks Requires specific capability (audio, location, VoIP) Foreground service with notification
Periodic work BGAppRefreshTask (minimum 15 min) WorkManager (minimum 15 min)

The JS Engine Problem

When a React Native app is backgrounded, the JS engine may be suspended or terminated by the OS. On iOS, there is no way to keep the JS engine running indefinitely. On Android, Headless JS can start the JS engine without a UI, but this requires a native service. Any background processing strategy must account for the possibility that JavaScript is not running.

2. Headless JS (Android)

Headless JS is React Native's built-in mechanism for running JavaScript tasks on Android when the app is not in the foreground. It starts the JS engine, executes a registered task, and shuts down when complete.

2.1 Registering a Headless Task

// index.js — register at app entry point
import { AppRegistry } from 'react-native';

AppRegistry.registerHeadlessTask(
  'SyncDataTask',
  () => async (taskData) => {
    const { lastSyncTimestamp } = taskData;

    try {
      const changes = await fetchChangesSince(lastSyncTimestamp);
      await applyChangesToLocalDB(changes);
      await updateLastSyncTimestamp(Date.now());
    } catch (error) {
      console.error('Background sync failed:', error);
    }
  }
);

2.2 Starting from Native (Android Service)

// android/app/src/main/java/.../SyncService.java
public class SyncService extends HeadlessJsTaskService {

  @Override
  protected @Nullable HeadlessJsTaskConfig getTaskConfig(
      Intent intent) {

    Bundle extras = intent.getExtras();
    WritableMap data = Arguments.fromBundle(extras);

    return new HeadlessJsTaskConfig(
      "SyncDataTask",      // must match JS registration
      data,                  // taskData passed to JS
      30000,                // timeout in ms
      true                  // allow in foreground
    );
  }
}

// Triggered from a BroadcastReceiver, AlarmManager, or WorkManager
public class SyncReceiver extends BroadcastReceiver {
  @Override
  public void onReceive(Context context, Intent intent) {
    if (!HeadlessJsTaskService.acquireWakeLockNow(context)) {
      return;
    }
    Intent serviceIntent = new Intent(context, SyncService.class);
    context.startService(serviceIntent);
  }
}
Headless JS Limitations

Headless JS is Android-only — there is no iOS equivalent in React Native core. The task runs with a timeout (default 5 minutes). If the task doesn't complete in time, the service is killed. The JS engine starts fresh each time — there is no shared state with the foreground app unless you use persistent storage (MMKV, SQLite).

2.3 Headless JS Architecture

Headless JS Execution Flow: Trigger (WorkManager / Alarm / Push) │ ▼ ┌──────────────────┐ │ BroadcastReceiver │ acquireWakeLock └──────┬───────────┘ ▼ ┌──────────────────┐ │ HeadlessJsTask │ start JS engine (if not running) │ Service │ └──────┬───────────┘ ▼ ┌──────────────────┐ │ JS Runtime │ execute registered task │ (Hermes) │ access storage, network └──────┬───────────┘ ▼ Task completes → service stops → engine may shut down

3. Background Fetch (iOS)

iOS Background Fetch allows the system to wake your app periodically to fetch new content. The system decides when and how often based on user behavior patterns — apps the user opens frequently get more fetch opportunities.

3.1 Enabling Background Fetch

// ios/AppDelegate.m — configure background fetch
- (BOOL)application:(UIApplication *)application
    didFinishLaunchingWithOptions:(NSDictionary *)launchOptions {

  [application setMinimumBackgroundFetchInterval:
    UIApplicationBackgroundFetchIntervalMinimum];

  // ... React Native setup
  return YES;
}

- (void)application:(UIApplication *)application
    performFetchWithCompletionHandler:
    (void (^)(UIBackgroundFetchResult))completionHandler {

  // Bridge to JS via event emitter or native module
  [self.bridge.eventDispatcher
    sendAppEventWithName:@"backgroundFetch"
    body:@{}];

  // Must call completion handler within ~30 seconds
  dispatch_after(
    dispatch_time(DISPATCH_TIME_NOW, 25 * NSEC_PER_SEC),
    dispatch_get_main_queue(),
    ^{ completionHandler(UIBackgroundFetchResultNewData); }
  );
}

3.2 react-native-background-fetch

import BackgroundFetch from 'react-native-background-fetch';

async function configureBackgroundFetch() {
  await BackgroundFetch.configure(
    {
      minimumFetchInterval: 15,    // minutes (iOS minimum)
      stopOnTerminate: false,       // Android: continue after app killed
      startOnBoot: true,            // Android: start after device reboot
      enableHeadless: true,         // Android: use Headless JS
      requiredNetworkType: BackgroundFetch.NETWORK_TYPE_ANY,
    },
    async (taskId) => {
      // This runs when the OS wakes the app
      const data = await fetchNewContent();
      await cacheLocally(data);

      BackgroundFetch.finish(taskId);
    },
    (taskId) => {
      // Timeout callback — task took too long
      BackgroundFetch.finish(taskId);
    }
  );
}

4. react-native-background-actions

For long-running tasks that need to survive backgrounding (data sync, file processing), react-native-background-actions uses a foreground service on Android and background task on iOS to keep JavaScript alive.

4.1 Long-Running Background Task

import BackgroundService from 'react-native-background-actions';

const syncTask = async (taskData: any) => {
  const { totalItems } = taskData;

  for (let i = 0; i < totalItems; i++) {
    if (!BackgroundService.isRunning()) break;

    await syncItem(i);
    await BackgroundService.updateNotification({
      progressBar: {
        value: i + 1,
        max: totalItems,
      },
      taskDesc: `Syncing item ${i + 1} of ${totalItems}`,
    });

    await sleep(100);
  }
};

async function startBackgroundSync(totalItems: number) {
  await BackgroundService.start(syncTask, {
    taskName: 'DataSync',
    taskTitle: 'Syncing Data',
    taskDesc: 'Starting sync...',
    taskIcon: { name: 'ic_sync', type: 'drawable' },
    color: '#1A73E8',
    linkingURI: 'myapp://sync',
    parameters: { totalItems },
  });
}

async function stopBackgroundSync() {
  await BackgroundService.stop();
}

4.2 Platform Implementation Details

Aspect Android iOS
Mechanism Foreground Service beginBackgroundTask + expiration handler
User visibility Persistent notification (required) No visible indicator
Max duration Unlimited (with notification) ~3 minutes (iOS 13+)
JS engine Kept alive by service Kept alive by background task
Termination risk Low (foreground service priority) High (OS may kill after timeout)
iOS Background Time is Severely Limited

On iOS 13+, beginBackgroundTask gives approximately 30 seconds (down from 3 minutes in earlier versions). For longer tasks, you must use specific background modes (audio, location, VoIP) or BGTaskScheduler. There is no general-purpose "run JS in background for 10 minutes" capability on iOS. Design your sync to be resumable — save progress so it can continue in the next background window.

5. Background Geolocation

Background location tracking is one of the few capabilities both platforms allow for extended background execution. It requires the "location" background mode and careful configuration to balance accuracy with battery consumption.

5.1 Configuration

import BackgroundGeolocation from 'react-native-background-geolocation';

async function configureGeolocation() {
  await BackgroundGeolocation.ready({
    desiredAccuracy: BackgroundGeolocation.DESIRED_ACCURACY_HIGH,
    distanceFilter: 50,              // meters between updates
    stopOnTerminate: false,          // continue after app killed
    startOnBoot: true,               // restart after device reboot
    stationaryRadius: 25,            // meters — switch to stationary mode
    activityRecognitionInterval: 10000,

    // HTTP auto-sync to server
    url: 'https://api.example.com/locations',
    batchSync: true,
    autoSync: true,
    autoSyncThreshold: 5,           // batch 5 locations before sending
    maxBatchSize: 50,

    // Battery optimization
    preventSuspend: true,            // iOS: prevent app suspension
    heartbeatInterval: 60,           // seconds — keep-alive ping

    // Android foreground service
    notification: {
      title: 'Tracking Location',
      text: 'Your location is being tracked for delivery',
      channelName: 'Location Tracking',
    },
  });

  BackgroundGeolocation.onLocation((location) => {
    console.log('[location]', location.coords);
  });

  BackgroundGeolocation.onMotionChange((event) => {
    console.log('[motion]', event.isMoving);
  });

  await BackgroundGeolocation.start();
}

6. Background Audio

The "audio" background mode allows apps to play or record audio while backgrounded. This is the most reliable background execution mode on iOS because the system keeps the process alive as long as audio is active.

6.1 Audio Session Configuration

import TrackPlayer from 'react-native-track-player';

async function setupAudioPlayer() {
  await TrackPlayer.setupPlayer({
    waitForBuffer: true,
    autoHandleInterruptions: true,
  });

  await TrackPlayer.updateOptions({
    capabilities: [
      TrackPlayer.CAPABILITY_PLAY,
      TrackPlayer.CAPABILITY_PAUSE,
      TrackPlayer.CAPABILITY_SKIP_TO_NEXT,
      TrackPlayer.CAPABILITY_SKIP_TO_PREVIOUS,
      TrackPlayer.CAPABILITY_SEEK_TO,
    ],
    compactCapabilities: [
      TrackPlayer.CAPABILITY_PLAY,
      TrackPlayer.CAPABILITY_PAUSE,
    ],
    notificationCapabilities: [
      TrackPlayer.CAPABILITY_PLAY,
      TrackPlayer.CAPABILITY_PAUSE,
      TrackPlayer.CAPABILITY_SKIP_TO_NEXT,
    ],
  });
}

// Register playback service for background control
TrackPlayer.registerPlaybackService(() =>
  async function playbackService() {
    TrackPlayer.addEventListener(
      'remote-play',
      () => TrackPlayer.play()
    );
    TrackPlayer.addEventListener(
      'remote-pause',
      () => TrackPlayer.pause()
    );
    TrackPlayer.addEventListener(
      'remote-next',
      () => TrackPlayer.skipToNext()
    );
  }
);
Audio Background Mode as a Side Channel

Some apps abuse the audio background mode by playing silent audio to keep the process alive for non-audio tasks. Apple explicitly rejects apps that do this during App Store review. The audio session must produce audible output or actively record audio. Apple's review team tests for this by monitoring the audio buffer.

7. Background Upload / Download

Both platforms support background transfers that continue even after the app is terminated. These use native URL session (iOS) and DownloadManager (Android) — the OS handles the transfer, not your app process.

7.1 iOS NSURLSession Background Transfer

// Native module wrapping NSURLSession background config
// ios/BackgroundTransfer.m
- (NSURLSession *)backgroundSession {
  static NSURLSession *session = nil;
  static dispatch_once_t onceToken;
  dispatch_once(&onceToken, ^{
    NSURLSessionConfiguration *config =
      [NSURLSessionConfiguration
        backgroundSessionConfigurationWithIdentifier:
          @"com.app.background-upload"];
    config.sessionSendsLaunchEvents = YES;
    config.discretionary = NO;
    config.allowsCellularAccess = YES;
    session = [NSURLSession
      sessionWithConfiguration:config
      delegate:self
      delegateQueue:nil];
  });
  return session;
}

7.2 React Native Upload/Download

import RNFS from 'react-native-fs';
import RNBackgroundDownloader from 'react-native-background-downloader';

// Background download — survives app termination
const task = RNBackgroundDownloader.download({
  id: 'file_123',
  url: 'https://cdn.example.com/large-file.zip',
  destination: `${RNFS.DocumentDirectoryPath}/large-file.zip`,
  headers: { Authorization: `Bearer ${token}` },
});

task.begin((expectedBytes) => {
  console.log(`Download size: ${expectedBytes} bytes`);
});

task.progress((percent) => {
  updateProgressBar(percent);
});

task.done(() => {
  console.log('Download complete');
});

task.error((error) => {
  console.error('Download failed:', error);
});

8. WorkManager Integration (Android)

WorkManager is Android's recommended API for deferrable, guaranteed background work. It handles constraints (network, battery, charging), retry policies, chaining, and survives process death and device reboots.

8.1 Native WorkManager Setup

// android/app/src/main/java/.../SyncWorker.kt
class SyncWorker(
  context: Context,
  params: WorkerParameters
) : CoroutineWorker(context, params) {

  override suspend fun doWork(): Result {
    return try {
      // Start Headless JS task
      val intent = Intent(applicationContext, SyncService::class.java)
      intent.putExtra("taskType", "periodic_sync")

      if (Build.VERSION.SDK_INT >= Build.VERSION_CODES.O) {
        applicationContext.startForegroundService(intent)
      } else {
        applicationContext.startService(intent)
      }

      Result.success()
    } catch (e: Exception) {
      if (runAttemptCount < 3) Result.retry()
      else Result.failure()
    }
  }
}

// Schedule periodic work
fun scheduleSyncWork(context: Context) {
  val constraints = Constraints.Builder()
    .setRequiredNetworkType(NetworkType.CONNECTED)
    .setRequiresBatteryNotLow(true)
    .build()

  val syncRequest = PeriodicWorkRequestBuilder<SyncWorker>(
    15, TimeUnit.MINUTES   // minimum interval
  )
    .setConstraints(constraints)
    .setBackoffCriteria(
      BackoffPolicy.EXPONENTIAL,
      30, TimeUnit.SECONDS
    )
    .addTag("periodic_sync")
    .build()

  WorkManager.getInstance(context)
    .enqueueUniquePeriodicWork(
      "sync",
      ExistingPeriodicWorkPolicy.KEEP,
      syncRequest
    )
}

9. BGTaskScheduler Integration (iOS)

BGTaskScheduler (iOS 13+) replaces the deprecated background fetch API. It supports two task types: BGAppRefreshTask (short, ~30 seconds) and BGProcessingTask (longer, minutes, runs during charging/idle).

9.1 Registering Background Tasks

// ios/AppDelegate.m
@import BackgroundTasks;

- (BOOL)application:(UIApplication *)application
    didFinishLaunchingWithOptions:(NSDictionary *)launchOptions {

  [BGTaskScheduler.sharedScheduler
    registerForTaskWithIdentifier:@"com.app.refresh"
    usingQueue:nil
    launchHandler:^(BGTask *task) {
      [self handleAppRefresh:(BGAppRefreshTask *)task];
    }];

  [BGTaskScheduler.sharedScheduler
    registerForTaskWithIdentifier:@"com.app.db-cleanup"
    usingQueue:nil
    launchHandler:^(BGTask *task) {
      [self handleDatabaseCleanup:(BGProcessingTask *)task];
    }];

  return YES;
}

- (void)handleAppRefresh:(BGAppRefreshTask *)task {
  [self scheduleAppRefresh];

  task.expirationHandler = ^{
    // Clean up — task is about to be killed
  };

  // Bridge to React Native JS
  [self.bridge.eventDispatcher
    sendAppEventWithName:@"backgroundRefresh"
    body:@{}];

  // Mark complete when JS signals done
  [task setTaskCompletedWithSuccess:YES];
}

- (void)scheduleAppRefresh {
  BGAppRefreshTaskRequest *request =
    [[BGAppRefreshTaskRequest alloc]
      initWithIdentifier:@"com.app.refresh"];
  request.earliestBeginDate =
    [NSDate dateWithTimeIntervalSinceNow:15 * 60];

  NSError *error = nil;
  [BGTaskScheduler.sharedScheduler
    submitTaskRequest:request error:&error];
}

Testing BGTaskScheduler

BGTasks are difficult to test because the system controls scheduling. Use the Xcode debugger command e -l objc -- (void)[[BGTaskScheduler sharedScheduler] _simulateLaunchForTaskWithIdentifier:@"com.app.refresh"] to trigger a task immediately during development. In production, the system may delay tasks by hours or even days if the user rarely opens the app.

10. Battery Optimization

Background processing is the primary cause of battery drain complaints. Every background strategy must be evaluated against its battery cost, and the app must adapt to low-battery conditions.

10.1 Battery-Aware Processing

import { NativeModules, Platform } from 'react-native';
import DeviceInfo from 'react-native-device-info';

async function getBackgroundPolicy(): Promise<BackgroundPolicy> {
  const batteryLevel = await DeviceInfo.getBatteryLevel();
  const isCharging = await DeviceInfo.isBatteryCharging();
  const isPowerSaver = await DeviceInfo.getPowerState()
    .then(s => s.lowPowerMode);

  if (isPowerSaver || batteryLevel < 0.15) {
    return {
      syncInterval: 60 * 60 * 1000,  // 1 hour
      locationAccuracy: 'low',
      batchSize: 100,
      enableAnalytics: false,
    };
  }

  if (isCharging) {
    return {
      syncInterval: 5 * 60 * 1000,    // 5 minutes
      locationAccuracy: 'high',
      batchSize: 20,
      enableAnalytics: true,
    };
  }

  return {
    syncInterval: 15 * 60 * 1000,   // 15 minutes
    locationAccuracy: 'balanced',
    batchSize: 50,
    enableAnalytics: true,
  };
}

Staff-Level Mental Model

Background processing in mobile is a negotiation with the operating system, not a capability you control. Both iOS and Android are moving toward stricter background restrictions with each release. The architect's job is to minimize background work by using push notifications to trigger sync (server-initiated), OS-managed transfers for large files, and periodic tasks only for truly time-insensitive work. Design every background task to be idempotent (safe to run twice), resumable (can continue from where it stopped), and bounded (completes within the platform's time limit). The most battery-efficient background processing is the work you eliminate entirely by moving it to the server.
