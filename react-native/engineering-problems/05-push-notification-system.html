1. Notification Architecture Overview

Push notifications traverse a multi-hop pipeline from your backend through platform-specific relay services to the user's device. Each hop introduces latency, failure modes, and configuration requirements that a Staff engineer must understand end-to-end.

Push Notification Pipeline ══════════════════════════ Your Backend Platform Services Device ──────────── ───────────────── ────── ┌──────────┐ ┌─────────┐ ┌──────────────────────────┐ │ App │ │ FCM │ │ Android Device │ │ Server │───▶│ (Google)│───▶│ ┌────────────────────┐ │ │ │ └─────────┘ │ │ System Tray / Heads │ │ │ or │ │ │ Up / Lock Screen │ │ │ │ ┌─────────┐ │ └────────────────────┘ │ │ Cloud │───▶│ APNs │ └──────────────────────────┘ │ Function │ │ (Apple) │ ┌──────────────────────────┐ │ │ └─────────┘───▶│ iOS Device │ └──────────┘ │ ┌────────────────────┐ │ │ │ │ Banner / Alert / │ │ ▼ │ │ Lock Screen │ │ Token Registry │ └────────────────────┘ │ (device tokens └──────────────────────────┘ mapped to users) Flow: Register token → Store mapping → Send payload → Relay → Display

1.1 Token Registration Flow

import messaging from '@react-native-firebase/messaging';

async function registerForPushNotifications(): Promise<string | null> {
  const authStatus = await messaging().requestPermission();
  const enabled =
    authStatus === messaging.AuthorizationStatus.AUTHORIZED ||
    authStatus === messaging.AuthorizationStatus.PROVISIONAL;

  if (!enabled) {
    return null;
  }

  const token = await messaging().getToken();

  await registerTokenWithBackend(token);

  messaging().onTokenRefresh(async (newToken) => {
    await registerTokenWithBackend(newToken);
  });

  return token;
}

async function registerTokenWithBackend(token: string) {
  await fetch('/api/devices/register', {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify({
      token,
      platform: Platform.OS,
      appVersion: DeviceInfo.getVersion(),
      userId: getCurrentUserId(),
    }),
  });
}
Token Lifecycle Pitfalls

FCM tokens can change at any time — after app reinstall, data clear, or when FCM rotates them internally. You must handle onTokenRefresh and update your backend mapping. Stale tokens cause silent delivery failures with no error feedback from FCM. Implement periodic token validation by sending a dry-run push and pruning tokens that return InvalidRegistration.

2. FCM & APNs Integration

Firebase Cloud Messaging acts as a unified layer for both platforms, but understanding the underlying APNs mechanics is critical for iOS-specific features like critical alerts, time-sensitive notifications, and provisional authorization.

2.1 Platform Payload Differences

Aspect FCM (Android) APNs (iOS via FCM)
Max payload size 4 KB 4 KB (APNs limit)
TTL (time-to-live) Up to 28 days Up to 30 days
Priority levels HIGH / NORMAL 10 (immediate) / 5 (power-considerate) / 1 (low)
Collapse key collapse_key apns-collapse-id
Background delivery Data-only message content-available: 1
Critical alerts Not supported Requires Apple entitlement
Grouping Notification channels thread-id

2.2 Server-Side Payload Construction

// Node.js backend — FCM Admin SDK
const admin = require('firebase-admin');

async function sendNotification(userId: string, payload: NotificationPayload) {
  const tokens = await getDeviceTokens(userId);

  const message = {
    tokens,
    notification: {
      title: payload.title,
      body: payload.body,
      imageUrl: payload.imageUrl,
    },
    data: {
      deepLink: payload.deepLink,
      notificationId: payload.id,
      type: payload.type,
    },
    android: {
      priority: 'high',
      notification: {
        channelId: payload.channelId || 'default',
        sound: 'default',
        clickAction: 'OPEN_FROM_NOTIFICATION',
      },
    },
    apns: {
      headers: {
        'apns-priority': '10',
        'apns-collapse-id': payload.collapseKey,
      },
      payload: {
        aps: {
          sound: 'default',
          badge: payload.badgeCount,
          'mutable-content': 1,
          'thread-id': payload.threadId,
        },
      },
    },
  };

  const response = await admin.messaging().sendEachForMulticast(message);

  // Prune failed tokens
  response.responses.forEach((resp, idx) => {
    if (resp.error?.code === 'messaging/registration-token-not-registered') {
      removeDeviceToken(tokens[idx]);
    }
  });
}
Architect Insight — FCM vs Direct APNs

FCM proxies to APNs for iOS, adding ~50-200ms latency. For time-critical apps (trading, emergency), consider sending directly to APNs using the HTTP/2 provider API. This requires managing APNs certificates/keys yourself but gives you lower latency, direct error feedback, and access to APNs-specific features like Live Activities push updates.

3. Notification Channels (Android)

Android 8.0+ (API 26) requires notification channels. Each channel defines the visual and auditory behavior for a category of notifications. Users can independently control each channel in system settings — your app cannot override their choices.

3.1 Channel Architecture

import notifee, { AndroidImportance, AndroidVisibility }
  from '@notifee/react-native';

async function createNotificationChannels() {
  await notifee.createChannelGroup({
    id: 'social',
    name: 'Social',
  });

  await notifee.createChannel({
    id: 'messages',
    name: 'Messages',
    groupId: 'social',
    importance: AndroidImportance.HIGH,
    visibility: AndroidVisibility.PRIVATE,
    sound: 'message_tone',
    vibration: true,
    vibrationPattern: [300, 500],
    lights: true,
    lightColor: '#0066FF',
    badge: true,
  });

  await notifee.createChannel({
    id: 'promotions',
    name: 'Promotions',
    importance: AndroidImportance.LOW,
    visibility: AndroidVisibility.PUBLIC,
  });

  await notifee.createChannel({
    id: 'silent-sync',
    name: 'Background Sync',
    importance: AndroidImportance.NONE,
    visibility: AndroidVisibility.SECRET,
  });
}
Importance Level Behavior Use Case
HIGH Sound + heads-up popup Messages, calls, urgent alerts
DEFAULT Sound, appears in tray Social updates, order status
LOW No sound, appears in tray Promotions, recommendations
MIN No sound, no visual interruption Weather, tips
NONE Blocked (only for background work) Silent data sync
Channel Immutability

Once a channel is created, you cannot programmatically change its importance, sound, or vibration pattern — only the user can modify these in system settings. If you need to change channel behavior, you must create a new channel with a different ID and delete the old one. Plan your channel IDs with versioning (e.g., messages_v2) to allow future migrations.

5. Deep Link Handling from Notifications

Notifications must navigate users to the correct screen when tapped — whether the app is in the foreground, background, or killed. This requires coordinating notification data, navigation state, and authentication state.

5.1 Three App States

Notification Tap → Navigation Routing ══════════════════════════════════════ App State Handler Timing ───────── ─────── ────── Foreground onMessage / onForeground Immediate — app is visible │ Event Navigate directly │ Background onNotificationOpened App resumes from memory │ OpenedApp Navigate after rehydration │ Killed getInitialNotification Cold start │ Navigate after full init ▼ + auth check All three must route to the same screen with the same data. Race condition: navigation may not be ready when handler fires.

5.2 Unified Deep Link Router

import { NavigationContainerRef } from '@react-navigation/native';

const navigationRef = createRef<NavigationContainerRef>();
let pendingDeepLink: string | null = null;

function handleNotificationDeepLink(deepLink: string) {
  if (!navigationRef.current?.isReady()) {
    pendingDeepLink = deepLink;
    return;
  }

  const route = parseDeepLink(deepLink);

  switch (route.screen) {
    case 'chat':
      navigationRef.current.navigate('ChatStack', {
        screen: 'ChatDetail',
        params: { threadId: route.params.threadId },
      });
      break;
    case 'order':
      navigationRef.current.navigate('OrderStack', {
        screen: 'OrderDetail',
        params: { orderId: route.params.orderId },
      });
      break;
  }
}

// Flush pending deep link once navigation is ready
function onNavigationReady() {
  if (pendingDeepLink) {
    handleNotificationDeepLink(pendingDeepLink);
    pendingDeepLink = null;
  }
}

// Wire up all three app-state handlers
messaging().onNotificationOpenedApp((msg) => {
  if (msg.data?.deepLink) handleNotificationDeepLink(msg.data.deepLink);
});

messaging().getInitialNotification().then((msg) => {
  if (msg?.data?.deepLink) handleNotificationDeepLink(msg.data.deepLink);
});
Navigation Readiness Race

When the app is cold-started from a notification, getInitialNotification resolves before React Navigation mounts. The pending deep link pattern above solves this. An alternative is to use a linking configuration with React Navigation's linking prop, which internally queues URLs until the navigator is ready.

6. Notification Preferences

Users expect granular control over notification types. A well-designed preference system syncs between the device and your backend, respects OS-level permissions, and degrades gracefully.

6.1 Preference Architecture

interface NotificationPreferences {
  globalEnabled: boolean;
  channels: {
    messages: { push: boolean; inApp: boolean; email: boolean };
    orderUpdates: { push: boolean; inApp: boolean; email: boolean };
    promotions: { push: boolean; inApp: boolean; email: boolean };
    socialActivity: { push: boolean; inApp: boolean; email: boolean };
  };
  quietHours: {
    enabled: boolean;
    start: string;  // "22:00"
    end: string;    // "08:00"
    timezone: string;
  };
}

class NotificationPreferenceManager {
  async syncPreferences(): Promise<void> {
    const osPermission = await messaging().hasPermission();
    const serverPrefs = await fetchPreferences();

    if (osPermission === messaging.AuthorizationStatus.DENIED) {
      serverPrefs.globalEnabled = false;
      await updatePreferences(serverPrefs);
    }

    await AsyncStorage.setItem(
      'notification_prefs',
      JSON.stringify(serverPrefs)
    );
  }

  async shouldDeliver(type: string, channel: string): Promise<boolean> {
    const prefs = await this.getPreferences();
    if (!prefs.globalEnabled) return false;
    if (this.isQuietHours(prefs)) return false;
    return prefs.channels[type]?.[channel] ?? true;
  }
}

7. Silent Push for Background Updates

Silent push notifications wake the app in the background without displaying anything to the user. They're essential for pre-fetching content, syncing data, and invalidating caches — but both platforms impose strict limits.

7.1 Platform Constraints

Constraint iOS Android
Execution time ~30 seconds ~20 seconds (FCM)
Throttling System throttles based on app usage patterns Doze mode delays non-high-priority messages
Reliability Not guaranteed — iOS may drop if battery low HIGH priority bypasses Doze for data messages
Trigger content-available: 1 with no alert/sound Data-only message (no notification key)
Rate limit 2-3 per hour (system-managed) No hard limit, but Doze batches

7.2 Background Handler Implementation

// Must be registered at the top level (index.js), not inside a component
messaging().setBackgroundMessageHandler(async (remoteMessage) => {
  const { type, payload } = remoteMessage.data;

  switch (type) {
    case 'CACHE_INVALIDATE':
      await invalidateQueryCache(payload.queryKeys);
      break;

    case 'CONTENT_PREFETCH':
      await prefetchContent(payload.contentIds);
      break;

    case 'CONFIG_UPDATE':
      await fetchRemoteConfig();
      break;

    case 'BADGE_UPDATE':
      await notifee.setBadgeCount(parseInt(payload.count, 10));
      break;
  }
});
Architect Insight — Silent Push Reliability

Never rely on silent push as your sole background sync mechanism. iOS aggressively throttles silent push for apps the user rarely opens. Combine with Background Fetch (iOS) and WorkManager (Android) for reliable sync. Use silent push as a hint that new data is available, with the actual sync handled by a more reliable background task API.

8. Notification Analytics

Measuring notification effectiveness requires tracking the full funnel: sent → delivered → displayed → opened → converted. Each stage has different measurement challenges.

8.1 Analytics Funnel

Notification Analytics Funnel ═════════════════════════════ Sent (server) 100% ← Backend logs │ Delivered (device) 92% ← FCM delivery receipts (Android only) │ APNs feedback service Displayed 88% ← Notification Service Extension (iOS) │ onMessageReceived (Android) Opened (tapped) 12% ← onNotificationOpenedApp │ Converted 4% ← In-app event tracking │ ─────┴───── Drop-off reasons: • Token stale (8%) • OS suppressed — DND, Focus, battery saver (4%) • User dismissed without opening (76%) • App uninstalled between send and delivery

8.2 Tracking Implementation

class NotificationAnalytics {
  async trackDelivered(notificationId: string) {
    await analytics().logEvent('notification_delivered', {
      notification_id: notificationId,
      timestamp: Date.now(),
      app_state: AppState.currentState,
    });
  }

  async trackOpened(notificationId: string, source: string) {
    const deliveredAt = await this.getDeliveryTimestamp(notificationId);
    const timeToOpen = Date.now() - deliveredAt;

    await analytics().logEvent('notification_opened', {
      notification_id: notificationId,
      time_to_open_ms: timeToOpen,
      source,
    });
  }

  async trackConversion(notificationId: string, action: string) {
    await analytics().logEvent('notification_converted', {
      notification_id: notificationId,
      conversion_action: action,
    });
  }
}

9. A/B Testing Notifications

Notification A/B testing optimizes open rates, conversion, and user retention. The testing framework must handle variant assignment, holdout groups, and statistical significance across both platforms.

9.1 Experiment Architecture

interface NotificationExperiment {
  experimentId: string;
  variants: {
    id: string;
    weight: number;  // 0.0 - 1.0, must sum to 1.0
    title: string;
    body: string;
    imageUrl?: string;
    sendTime?: string;  // "immediate" | "optimal" | ISO time
  }[];
  holdoutPercentage: number;
  targetAudience: AudienceFilter;
  successMetric: 'open_rate' | 'conversion' | 'retention_d7';
  minSampleSize: number;
}

// Server-side variant assignment (deterministic hashing)
function assignVariant(userId: string, experiment: NotificationExperiment): string {
  const hash = murmurhash3(`${experiment.experimentId}:${userId}`);
  const bucket = (hash % 10000) / 10000;

  if (bucket < experiment.holdoutPercentage) {
    return 'holdout';
  }

  let cumulative = experiment.holdoutPercentage;
  for (const variant of experiment.variants) {
    cumulative += variant.weight * (1 - experiment.holdoutPercentage);
    if (bucket < cumulative) return variant.id;
  }

  return experiment.variants[0].id;
}
Optimal Send Time

Send-time optimization uses per-user engagement history to determine when each user is most likely to open a notification. This requires tracking open timestamps over weeks, clustering users into time-preference cohorts, and scheduling sends per-cohort. Firebase A/B Testing and OneSignal's Intelligent Delivery offer this out of the box — building it in-house requires a dedicated ML pipeline.

10.2 Production Checklist

  • Token hygiene: Prune stale tokens on every failed delivery. Implement periodic dry-run validation. Track token-to-user cardinality (users with 10+ tokens indicate a registration bug).
  • Idempotency: Assign a unique notification ID server-side. Deduplicate on the client using a local seen-set to prevent duplicate display after retry.
  • Graceful degradation: If push permission is denied, fall back to in-app notification center. Track the permission-denied rate and surface a re-prompt at contextually appropriate moments.
  • Rate limiting: Cap notifications per user per hour/day. Users who receive too many notifications disable them entirely — or worse, uninstall.
  • iOS provisional authorization: Request provisional permission first (delivered quietly to Notification Center). This avoids the permission dialog on first launch and lets users experience notifications before deciding.
  • Android 13+ runtime permission: Starting with Android 13 (API 33), POST_NOTIFICATIONS is a runtime permission. You must request it explicitly, just like camera or location.
Staff-Level Mental Model

A push notification system is a distributed messaging pipeline with unreliable delivery guarantees. Treat it like any distributed system: design for at-least-once delivery with client-side deduplication, implement observability at every hop, and never make push the sole communication channel for critical information. The best notification systems are the ones users don't want to turn off — which means respecting preferences, optimizing send times, and measuring the unsubscribe rate as a first-class metric alongside open rate.
