1. Event Tracking Architecture

The SDK's core responsibility is capturing structured events with minimal overhead. Every event flows through a pipeline: capture → enrich → validate → queue → persist → batch → send.

Event Lifecycle Pipeline: User Action SDK Core Queue Network ────────── ────────────────── ────────────── ──────────── │ │ tap/screen → │ enrich(event) │ → in-memory → batch & send navigate → │ validate(event) │ ring buffer to endpoint error → │ timestamp │ │ │ sessionId │ ▼ │ userId │ SQLite / MMKV └──────────────────┘ (persistence) │ ▼ retry on failure exponential backoff

1.1 Event Schema Design

interface AnalyticsEvent {
  id: string;                // UUID v4 — deduplication key
  name: string;              // e.g., "product_viewed"
  category: string;          // e.g., "commerce", "navigation"
  properties: Record<string, unknown>;
  timestamp: number;         // Unix ms — captured at event time
  sessionId: string;
  userId: string | null;
  anonymousId: string;
  context: EventContext;
}

interface EventContext {
  app: { version: string; build: string };
  device: { model: string; os: string; osVersion: string };
  screen: { width: number; height: number; density: number };
  network: { type: string; effectiveType: string };
  locale: string;
  timezone: string;
}

Architect Insight

The middleware pattern is critical for extensibility. Each middleware receives the event, can transform it, and passes it downstream. This allows teams to inject enrichment (device info, experiment assignments), filtering (PII scrubbing), and routing (send to multiple backends) without modifying the core SDK.

2. Batching and Persistence

Sending every event individually is wasteful. Batching amortizes HTTP overhead, reduces battery drain, and enables server-side deduplication. Persistence ensures events survive app crashes and cold restarts.

2.1 In-Memory Ring Buffer

class EventQueue {
  private buffer: AnalyticsEvent[] = [];
  private maxSize: number;
  private flushAt: number;
  private flushTimer: ReturnType<typeof setTimeout> | null = null;

  enqueue(event: AnalyticsEvent): void {
    if (this.buffer.length >= this.maxSize) {
      this.buffer.shift();  // drop oldest — bounded memory
    }
    this.buffer.push(event);
    this.persistToDisk(event);

    if (this.buffer.length >= this.flushAt) {
      this.flush();
    } else {
      this.scheduleFlush();
    }
  }

  private async persistToDisk(event: AnalyticsEvent): Promise<void> {
    await db.execute(
      `INSERT INTO events (id, payload, created_at, status)
       VALUES (?, ?, ?, 'pending')`,
      [event.id, JSON.stringify(event), event.timestamp]
    );
  }
}

2.2 Persistence Layer Comparison

Storage Write Speed Crash Safety Query Support Best For
AsyncStorage Slow (JSON serialize) Moderate None (key-value) Simple apps
MMKV Very fast (mmap) High (CRC checksum) None (key-value) High-throughput events
SQLite (op-sqlite) Fast (WAL mode) Very high (ACID) Full SQL Complex querying, retry
WatermelonDB Fast (lazy loading) Very high Query builder Reactive sync scenarios
Critical: Persistence Must Be Synchronous-Feeling

If track() awaits disk I/O, it blocks the calling code and risks dropping events during rapid-fire scenarios (scroll tracking, gesture events). Use a write-ahead approach: enqueue in memory immediately, persist asynchronously, and reconcile on startup by replaying un-sent events from disk.

2.3 Startup Reconciliation

async function reconcileOnStartup(): Promise<void> {
  const pending = await db.query(
    `SELECT * FROM events WHERE status = 'pending'
     ORDER BY created_at ASC LIMIT 500`
  );

  if (pending.length > 0) {
    await sendBatch(pending);
    await db.execute(
      `DELETE FROM events WHERE id IN (?)`,
      [pending.map(e => e.id)]
    );
  }

  // Prune events older than 7 days to prevent unbounded growth
  await db.execute(
    `DELETE FROM events WHERE created_at < ?`,
    [Date.now() - 7 * 24 * 60 * 60 * 1000]
  );
}

3. Network-Aware Sending

A production SDK must adapt its behavior to network conditions. Sending batches over a metered cellular connection wastes user data; sending over no connection wastes battery on retries.

3.1 Network State Machine

Network-Aware Flush Strategy: ┌──────────┐ wifi ┌───────────────┐ │ OFFLINE │ ─────────→ │ FLUSH_EAGER │ flush every 15s, batch 10 └──────────┘ └───────────────┘ │ │ │ cellular │ wifi lost ▼ ▼ ┌──────────────┐ ┌──────────┐ │ FLUSH_LAZY │ ←────── │ OFFLINE │ │ flush 60s, │ └──────────┘ │ batch 50 │ └──────────────┘ OFFLINE: queue only, no sends, persist everything FLUSH_LAZY: larger batches, longer intervals (save data) FLUSH_EAGER: smaller batches, shorter intervals (low latency)

3.2 NetInfo Integration

import NetInfo from '@react-native-community/netinfo';

class NetworkAwareTransport {
  private state: 'offline' | 'cellular' | 'wifi' = 'wifi';

  constructor() {
    NetInfo.addEventListener(netState => {
      if (!netState.isConnected) {
        this.state = 'offline';
        this.pauseFlushing();
      } else if (netState.type === 'wifi') {
        this.state = 'wifi';
        this.setFlushConfig({ interval: 15000, batchSize: 10 });
        this.drainQueue();
      } else {
        this.state = 'cellular';
        this.setFlushConfig({ interval: 60000, batchSize: 50 });
      }
    });
  }

  async send(batch: AnalyticsEvent[]): Promise<void> {
    try {
      const response = await fetch(this.endpoint, {
        method: 'POST',
        headers: { 'Content-Encoding': 'gzip' },
        body: gzip(JSON.stringify({ batch })),
      });
      if (!response.ok) throw new Error(response.status);
    } catch (err) {
      this.scheduleRetry(batch);
    }
  }
}

3.3 Retry with Exponential Backoff

async function retryWithBackoff(
  fn: () => Promise<void>,
  maxRetries: number = 3,
  baseDelay: number = 1000
): Promise<void> {
  for (let attempt = 0; attempt < maxRetries; attempt++) {
    try {
      await fn();
      return;
    } catch (err) {
      const jitter = Math.random() * 1000;
      const delay = baseDelay * Math.pow(2, attempt) + jitter;
      await new Promise(r => setTimeout(r, delay));
    }
  }
  // After max retries, events remain in SQLite for next startup
}

4. Session Management

Sessions group user activity into logical visits. A session starts when the app foregrounds and expires after a configurable inactivity timeout (typically 30 minutes).

4.1 Session State Machine

Session Lifecycle: App Launch / Foreground │ ▼ ┌─────────────┐ │ NEW_SESSION │ ── generate sessionId, emit session_start └──────┬──────┘ ▼ ┌─────────────┐ user activity ┌─────────────┐ │ ACTIVE │ ←─────────────────── │ ACTIVE │ └──────┬──────┘ reset timer └─────────────┘ │ │ 30 min inactivity OR app background ▼ ┌─────────────┐ │ EXPIRED │ ── emit session_end, persist duration └─────────────┘ │ │ foreground again ▼ ┌─────────────┐ │ NEW_SESSION │ ── new sessionId └─────────────┘

4.2 AppState-Driven Session Tracking

import { AppState } from 'react-native';
import { v4 as uuid } from 'uuid';

class SessionManager {
  private sessionId: string = uuid();
  private sessionStart: number = Date.now();
  private lastActivity: number = Date.now();
  private timeout: number = 30 * 60 * 1000; // 30 min

  constructor(private analytics: Analytics) {
    AppState.addEventListener('change', this.handleAppState);
  }

  handleAppState = (state: string) => {
    if (state === 'active') {
      const elapsed = Date.now() - this.lastActivity;
      if (elapsed > this.timeout) {
        this.endSession();
        this.startSession();
      }
    } else if (state === 'background') {
      this.lastActivity = Date.now();
      this.analytics.track('app_backgrounded');
    }
  };

  touch(): void {
    this.lastActivity = Date.now();
  }

  private startSession(): void {
    this.sessionId = uuid();
    this.sessionStart = Date.now();
    this.analytics.track('session_start', {
      sessionId: this.sessionId,
    });
  }

  private endSession(): void {
    this.analytics.track('session_end', {
      sessionId: this.sessionId,
      duration: Date.now() - this.sessionStart,
    });
  }
}

5. User Identification

Identity resolution maps anonymous device activity to known user profiles. The SDK must handle the transition from anonymous to identified state and merge event histories.

5.1 Identity Resolution Flow

Identity Resolution: First Launch Login Logout ──────────── ────────────── ────────────── anonymousId: "abc" identify("user_42") reset() userId: null userId: "user_42" anonymousId: "xyz" (new) anonymousId: "abc" userId: null │ ▼ Server merges "abc" → "user_42" All prior anonymous events attributed to user_42

5.2 Identify and Reset API

class IdentityManager {
  private anonymousId: string;
  private userId: string | null = null;
  private traits: Record<string, unknown> = {};

  constructor() {
    this.anonymousId = this.loadOrCreateAnonymousId();
  }

  identify(userId: string, traits?: Record<string, unknown>): void {
    const previousId = this.userId;
    this.userId = userId;
    this.traits = { ...this.traits, ...traits };

    this.persistIdentity();
    this.analytics.enqueue({
      type: 'identify',
      userId,
      anonymousId: this.anonymousId,
      traits: this.traits,
      previousId,
    });
  }

  reset(): void {
    this.userId = null;
    this.traits = {};
    this.anonymousId = uuid();
    this.persistIdentity();
  }

  private loadOrCreateAnonymousId(): string {
    const stored = MMKV.getString('analytics_anonymous_id');
    if (stored) return stored;
    const id = uuid();
    MMKV.set('analytics_anonymous_id', id);
    return id;
  }
}
Anonymous ID Persistence

The anonymous ID must persist across app restarts but reset on reset() (logout). Store it in MMKV or Keychain — not AsyncStorage, which is slower and can be cleared by OS storage pressure. On iOS, consider using the Keychain so the ID survives app reinstalls (useful for attribution).

6. Screen Tracking with Navigation

Automatic screen tracking requires deep integration with React Navigation. The SDK must capture screen views without requiring manual instrumentation on every screen.

6.1 Navigation Listener Integration

import { useNavigationContainerRef } from '@react-navigation/native';

function useAnalyticsScreenTracking() {
  const navigationRef = useNavigationContainerRef();
  const routeNameRef = useRef<string>();

  const onReady = useCallback(() => {
    routeNameRef.current = navigationRef.getCurrentRoute()?.name;
  }, []);

  const onStateChange = useCallback(() => {
    const previousRoute = routeNameRef.current;
    const currentRoute = navigationRef.getCurrentRoute();

    if (currentRoute && previousRoute !== currentRoute.name) {
      analytics.screen(currentRoute.name, {
        params: currentRoute.params,
        previousScreen: previousRoute,
      });
    }

    routeNameRef.current = currentRoute?.name;
  }, []);

  return { navigationRef, onReady, onStateChange };
}

// Usage in App.tsx
function App() {
  const { navigationRef, onReady, onStateChange } =
    useAnalyticsScreenTracking();

  return (
    <NavigationContainer
      ref={navigationRef}
      onReady={onReady}
      onStateChange={onStateChange}
    >
      {/* navigators */}
    </NavigationContainer>
  );
}

6.2 Screen Time Tracking

class ScreenTimeTracker {
  private screenEntryTime: Map<string, number> = new Map();

  onScreenEnter(screenName: string): void {
    this.screenEntryTime.set(screenName, Date.now());
  }

  onScreenLeave(screenName: string): void {
    const entry = this.screenEntryTime.get(screenName);
    if (entry) {
      analytics.track('screen_time', {
        screen: screenName,
        duration: Date.now() - entry,
      });
      this.screenEntryTime.delete(screenName);
    }
  }
}

7. Performance Metrics Collection

The analytics SDK is uniquely positioned to collect performance telemetry because it already has the batching and transport infrastructure. Key metrics include app startup time, screen render time, JS thread frame drops, and API latency.

7.1 Startup Time Measurement

// Native module exposes process start timestamp
import { NativeModules } from 'react-native';

const nativeStartTime = NativeModules.StartupTiming.startTime;

function measureStartup() {
  const jsReady = Date.now();
  const coldStartMs = jsReady - nativeStartTime;

  analytics.track('app_startup', {
    coldStartMs,
    jsLoadMs: jsReady - global.__jsLoadStart,
    bundleSize: global.__bundleSize,
  });
}

Don't Measure What You Can't Act On

Every metric you collect has a cost: storage, bandwidth, and engineering time to analyze. Define actionable thresholds and alerts before adding a metric. If a metric degrades and nobody investigates, it's wasted infrastructure.

8. Privacy Compliance

GDPR, CCPA, and App Tracking Transparency (ATT) impose strict requirements on what data you collect, how you store it, and whether you can send it. The SDK must support consent management as a first-class concern.

8.1 Consent State Machine

Consent Flow: App Launch │ ▼ ┌──────────────┐ user grants ┌───────────────┐ │ UNKNOWN │ ──────────────────→ │ GRANTED │ │ (queue only) │ │ (send events) │ └──────┬───────┘ └───────────────┘ │ │ │ user denies │ user revokes ▼ ▼ ┌──────────────┐ ┌───────────────┐ │ DENIED │ │ REVOKED │ │ (purge queue)│ │ (stop + purge)│ └──────────────┘ └───────────────┘

8.2 Consent-Aware Event Pipeline

type ConsentStatus = 'unknown' | 'granted' | 'denied' | 'revoked';

class ConsentManager {
  private status: ConsentStatus = 'unknown';
  private pendingQueue: AnalyticsEvent[] = [];

  setConsent(status: ConsentStatus): void {
    this.status = status;

    switch (status) {
      case 'granted':
        this.flushPending();
        break;
      case 'denied':
      case 'revoked':
        this.purgeAllData();
        break;
    }
  }

  shouldTrack(event: AnalyticsEvent): boolean {
    if (this.status === 'denied' || this.status === 'revoked') {
      return false;
    }
    if (this.status === 'unknown') {
      this.pendingQueue.push(event);
      return false;
    }
    return true;
  }

  private async purgeAllData(): Promise<void> {
    this.pendingQueue = [];
    await db.execute('DELETE FROM events');
    await MMKV.delete('analytics_anonymous_id');
  }
}
Architect Insight — ATT on iOS

iOS 14.5+ requires App Tracking Transparency permission before accessing the IDFA. If your analytics SDK uses IDFA for cross-app attribution, you must request ATT permission and handle denial gracefully. Many teams now avoid IDFA entirely and rely on first-party anonymous IDs, which don't require ATT but limit cross-app attribution capabilities.

9. A/B Testing Integration

The analytics SDK and experimentation platform share infrastructure: both need user identification, event tracking, and property enrichment. Integrating them avoids duplicate SDKs and ensures experiment assignments are automatically attached to all analytics events.

9.1 Experiment Assignment Flow

interface Experiment {
  key: string;           // "checkout_redesign"
  variant: string;       // "control" | "treatment_a" | "treatment_b"
  payload: unknown;      // feature flags, config values
}

class ExperimentManager {
  private assignments: Map<string, Experiment> = new Map();

  async fetchAssignments(userId: string): Promise<void> {
    const response = await fetch(
      `/api/experiments?user=${userId}`
    );
    const experiments: Experiment[] = await response.json();

    experiments.forEach(exp => {
      this.assignments.set(exp.key, exp);
    });
  }

  getExperimentContext(): Record<string, string> {
    const context: Record<string, string> = {};
    this.assignments.forEach((exp, key) => {
      context[`experiment_${key}`] = exp.variant;
    });
    return context;
  }
}

// Middleware that enriches every event with experiment context
function addExperimentContext(event: AnalyticsEvent): AnalyticsEvent {
  return {
    ...event,
    properties: {
      ...event.properties,
      ...experimentManager.getExperimentContext(),
    },
  };
}

9.2 Exposure Tracking

// Track when a user actually sees the experiment variant
function useExperiment(experimentKey: string) {
  const experiment = experimentManager.get(experimentKey);
  const exposureTracked = useRef(false);

  useEffect(() => {
    if (experiment && !exposureTracked.current) {
      analytics.track('experiment_exposure', {
        experimentKey,
        variant: experiment.variant,
      });
      exposureTracked.current = true;
    }
  }, [experiment]);

  return experiment;
}

10.2 Testing Strategy

// Mock analytics for unit tests
class MockAnalytics implements AnalyticsInterface {
  events: AnalyticsEvent[] = [];

  track(name: string, props?: Record<string, unknown>): void {
    this.events.push({ name, properties: props ?? {} } as AnalyticsEvent);
  }

  assertTracked(name: string, props?: Partial<Record<string, unknown>>): void {
    const match = this.events.find(e =>
      e.name === name &&
      (!props || objectContains(e.properties, props))
    );
    if (!match) throw new Error(`Event "${name}" not tracked`);
  }
}
Staff-Level Mental Model

An analytics SDK is a distributed systems problem disguised as a client library. It must handle unreliable networks, app crashes, storage limits, identity resolution across devices, privacy regulations across jurisdictions, and real-time experiment assignment — all while adding zero perceptible latency to the host app. The key architectural principle is eventual delivery with bounded resource usage: events will arrive at the server eventually, but the SDK must never consume unbounded memory, storage, or battery to achieve this.
