Device Fragmentation, Foldables & Accessibility

Why this matters at L6: Google's Android ecosystem spans thousands of device configurations. L6 engineers are expected to design UIs that are adaptive — not just "it works on Pixel". They also treat accessibility as a first-class requirement, not an afterthought. These topics appear in both system design and behavioral rounds.

2. Adaptive Layouts — The Modern Approach

Window Size Classes (Jetpack)

// Window Size Classes: replace "is this a tablet?" checks
// Based on window width, NOT device type (foldable can be compact or expanded)

// Width breakpoints:
// Compact:  < 600dp  (phone portrait)
// Medium:   600–840dp (phone landscape, small tablet, foldable open)
// Expanded: > 840dp  (large tablet, desktop)

@Composable
fun AdaptiveFeedScreen(
    windowSizeClass: WindowSizeClass,
    viewModel: FeedViewModel = hiltViewModel()
) {
    val uiState by viewModel.uiState.collectAsStateWithLifecycle()

    when (windowSizeClass.widthSizeClass) {
        WindowWidthSizeClass.Compact -> {
            // Single column, bottom navigation
            FeedOneColumnLayout(uiState)
        }
        WindowWidthSizeClass.Medium -> {
            // Two column, side rail navigation
            FeedTwoColumnLayout(uiState)
        }
        WindowWidthSizeClass.Expanded -> {
            // Three column, permanent navigation drawer
            FeedThreeColumnLayout(uiState)
        }
    }
}

// In Activity:
class MainActivity : ComponentActivity() {
    override fun onCreate(savedInstanceState: Bundle?) {
        super.onCreate(savedInstanceState)
        val windowSizeClass = calculateWindowSizeClass(this)
        setContent {
            AdaptiveFeedScreen(windowSizeClass = windowSizeClass)
        }
    }
}

Navigation Adaptation Pattern

@Composable
fun AppNavigation(windowSizeClass: WindowSizeClass) {
    val navController = rememberNavController()
    val isExpanded = windowSizeClass.widthSizeClass == WindowWidthSizeClass.Expanded

    if (isExpanded) {
        // Permanent NavigationRail or NavigationDrawer (always visible)
        PermanentNavigationDrawer(
            drawerContent = { AppDrawerContent(navController) }
        ) {
            AppNavHost(navController)
        }
    } else {
        // BottomNavigationBar (compact) or NavigationRail (medium)
        Scaffold(
            bottomBar = {
                if (windowSizeClass.widthSizeClass == WindowWidthSizeClass.Compact) {
                    AppBottomBar(navController)
                }
            }
        ) { padding ->
            AppNavHost(navController, Modifier.padding(padding))
        }
    }
}

3. Foldable Devices

Foldable States and Postures

Foldable device states: FLAT (fully open): HALF_OPEN (tabletop/book): CLOSED (phone mode): ┌─────────────────────┐ ┌─────────────────────┐ ┌──────────┐ │ │ │ Top half — media │ │ │ │ Full tablet UI │ │─────────────────────│ │ Phone │ │ │ │ Bottom — controls │ │ UI │ └─────────────────────┘ └─────────────────────┘ └──────────┘ WindowSizeClass.Expanded FoldingFeature.State.HALF_OPEN Compact
// Detect fold state with Jetpack WindowManager
class FeedActivity : AppCompatActivity() {
    override fun onCreate(savedInstanceState: Bundle?) {
        super.onCreate(savedInstanceState)
        lifecycleScope.launch {
            repeatOnLifecycle(Lifecycle.State.STARTED) {
                WindowInfoTracker.getOrCreate(this@FeedActivity)
                    .windowLayoutInfo(this@FeedActivity)
                    .collect { layoutInfo ->
                        updateLayoutForWindowInfo(layoutInfo)
                    }
            }
        }
    }

    private fun updateLayoutForWindowInfo(info: WindowLayoutInfo) {
        val foldingFeature = info.displayFeatures
            .filterIsInstance<FoldingFeature>()
            .firstOrNull()

        when {
            foldingFeature == null -> {
                // Regular phone or tablet — use WindowSizeClass
            }
            foldingFeature.state == FoldingFeature.State.HALF_OPEN
                && foldingFeature.orientation == FoldingFeature.Orientation.HORIZONTAL -> {
                // Tabletop mode: top half = content, bottom half = controls
                applyTabletopLayout(foldingFeature.bounds)
            }
            foldingFeature.state == FoldingFeature.State.HALF_OPEN
                && foldingFeature.orientation == FoldingFeature.Orientation.VERTICAL -> {
                // Book mode: left = primary, right = detail
                applyBookLayout(foldingFeature.bounds)
            }
            foldingFeature.state == FoldingFeature.State.FLAT -> {
                // Fully open: use full screen layout
                applyExpandedLayout()
            }
        }
    }
}

// Hinge avoidance: don't put content on the hinge itself
// The FoldingFeature.bounds gives the hinge rectangle in window coordinates
// Ensure no interactive elements overlap this rectangle

4. OEM Quirks — Production Reality

The Background Process Killing Problem

The biggest OEM-specific issue: Samsung, Xiaomi, Huawei, OnePlus, and others aggressively kill background processes beyond what AOSP does. This breaks WorkManager, background sync, FCM delivery, and foreground services.
OEMMechanismUser FixEngineering Mitigation
Samsung (One UI)"Sleeping apps" list, Battery optimizationAdd to "Never sleeping apps"Foreground Service + ongoing notification
Xiaomi (MIUI)MIUI Autostart, battery optimizerEnable Autostart in settingsFCM high-priority, user-facing prompt to enable autostart
Huawei (EMUI)Phone Manager, App Launch manager"Manage manually"Same as Xiaomi
OnePlusApp auto-launch, battery optimizationLock app in recent appsForeground Service
// Detect if battery optimization is restricting the app and prompt user
fun checkBatteryOptimization(context: Context) {
    val pm = context.getSystemService(PowerManager::class.java)
    if (!pm.isIgnoringBatteryOptimizations(context.packageName)) {
        // Show rationale then:
        val intent = Intent(Settings.ACTION_REQUEST_IGNORE_BATTERY_OPTIMIZATIONS).apply {
            data = Uri.parse("package:${context.packageName}")
        }
        startActivity(intent)
    }
}

Runtime Permission Handling Across OEMs

// Some OEMs show permission dialogs differently or add extra steps
// Always handle shouldShowRequestPermissionRationale correctly

class PermissionManager(private val activity: AppCompatActivity) {

    private val permissionLauncher = activity.registerForActivityResult(
        ActivityResultContracts.RequestMultiplePermissions()
    ) { permissions ->
        val granted = permissions.all { it.value }
        if (granted) {
            onPermissionsGranted()
        } else {
            val permanentlyDenied = permissions.any { (permission, isGranted) ->
                !isGranted && !activity.shouldShowRequestPermissionRationale(permission)
            }
            if (permanentlyDenied) {
                // User selected "Don't ask again" — direct to settings
                showGoToSettingsDialog()
            } else {
                showRationaleDialog()
            }
        }
    }

    fun requestNotificationPermission() {
        if (Build.VERSION.SDK_INT >= Build.VERSION_CODES.TIRAMISU) {
            permissionLauncher.launch(arrayOf(Manifest.permission.POST_NOTIFICATIONS))
        }
        // Pre-API 33: permission granted automatically
    }

    private fun showGoToSettingsDialog() {
        // User must go to App Settings to re-enable permanently denied permissions
        val intent = Intent(Settings.ACTION_APPLICATION_DETAILS_SETTINGS).apply {
            data = Uri.fromParts("package", activity.packageName, null)
        }
        activity.startActivity(intent)
    }
}

TalkBack — Screen Reader Support

// Compose accessibility semantics

@Composable
fun PostCard(
    post: Post,
    isLiked: Boolean,
    likeCount: Int,
    onLikeClick: () -> Unit,
    onPostClick: () -> Unit
) {
    Card(
        modifier = Modifier
            .clickable(
                onClickLabel = "Open post: ${post.title}",  // TalkBack reads this
                onClick = onPostClick
            )
            .semantics {
                // Full semantic description for screen readers
                contentDescription = "Post: ${post.title}, by ${post.author}, ${post.timeAgo}"
            }
    ) {
        // Like button: needs a meaningful label, not just "button"
        IconButton(
            onClick = onLikeClick,
            modifier = Modifier.semantics {
                contentDescription = if (isLiked) {
                    "Unlike post. Currently liked by $likeCount people"
                } else {
                    "Like post. Currently liked by $likeCount people"
                }
                // Expose toggle state for switch-access users
                stateDescription = if (isLiked) "Liked" else "Not liked"
            }
        ) {
            Icon(
                imageVector = if (isLiked) Icons.Filled.Favorite else Icons.Outlined.FavoriteBorder,
                contentDescription = null  // Described by parent semantics
            )
        }

        // Decorative images: explicitly mark as not meaningful
        AsyncImage(
            model = post.thumbnailUrl,
            contentDescription = null,  // null = decorative, TalkBack skips it
            modifier = Modifier.semantics { invisibleToUser() }
        )
    }
}

// Merge semantics: treat a group of elements as one TalkBack item
@Composable
fun UserRow(user: User) {
    Row(modifier = Modifier.semantics(mergeDescendants = true) {}) {
        // TalkBack reads these as one: "Alice, @alice_dev, 1.2K followers"
        Image(painter = painterResource(user.avatarRes), contentDescription = null)
        Text(user.displayName)
        Text("@${user.handle}")
        Text("${user.followerCount} followers")
    }
}

Touch Target Size and Focus Management

// WCAG 2.5.5 requires 44x44pt minimum — Android recommends 48x48dp
@Composable
fun SmallIconButton(onClick: () -> Unit, icon: ImageVector) {
    Box(
        modifier = Modifier
            .size(48.dp)  // Minimum touch target
            .clickable(onClick = onClick),
        contentAlignment = Alignment.Center
    ) {
        Icon(
            imageVector = icon,
            contentDescription = null,
            modifier = Modifier.size(24.dp)  // Visual size can be smaller
        )
    }
}

// Focus order management for keyboard/D-pad navigation
@Composable
fun LoginForm() {
    val focusRequester1 = remember { FocusRequester() }
    val focusRequester2 = remember { FocusRequester() }
    val focusManager = LocalFocusManager.current

    TextField(
        value = email,
        onValueChange = { email = it },
        keyboardOptions = KeyboardOptions(imeAction = ImeAction.Next),
        keyboardActions = KeyboardActions(
            onNext = { focusRequester2.requestFocus() }  // Move to password field
        ),
        modifier = Modifier.focusRequester(focusRequester1)
    )
    TextField(
        value = password,
        onValueChange = { password = it },
        keyboardOptions = KeyboardOptions(imeAction = ImeAction.Done),
        keyboardActions = KeyboardActions(
            onDone = { focusManager.clearFocus(); submitLogin() }
        ),
        modifier = Modifier.focusRequester(focusRequester2)
    )
}

Live Regions — Dynamic Content Announcements

// Live regions announce changes to screen reader users without moving focus

// Compose: use liveRegion semantic
@Composable
fun SyncStatusBanner(syncState: SyncState) {
    val message = when (syncState) {
        SyncState.Syncing -> "Syncing..."
        SyncState.Success -> "All caught up"
        SyncState.Error -> "Sync failed. Check connection."
        SyncState.Offline -> "Offline mode"
    }

    Text(
        text = message,
        modifier = Modifier.semantics {
            liveRegion = LiveRegionMode.Polite  // Announces when message changes
            // Use Assertive for critical changes (errors, warnings)
        }
    )
}

Contrast and Text Scaling

// Minimum contrast ratios (WCAG AA):
// Normal text: 4.5:1
// Large text (18sp+ or 14sp+ bold): 3:1
// UI components and graphics: 3:1

// Use Material3 color system — it provides on-colors that meet contrast requirements
MaterialTheme(
    colorScheme = lightColorScheme(
        primary = Color(0xFF1565C0),           // WCAG AA compliant
        onPrimary = Color.White,               // on-colors always meet contrast
        background = Color(0xFFFFFBFE),
        onBackground = Color(0xFF1C1B1F)       // 15.5:1 contrast ratio
    )
) { /* app content */ }

// Text scaling: never hardcode sp to dp conversion
// BAD: 16.dp.toSp()  → ignores user's font scale preference
// GOOD: 16.sp        → respects font size accessibility setting

// Handle large text gracefully: use weight + max lines + ellipsis
Text(
    text = userName,
    style = MaterialTheme.typography.bodyLarge,
    maxLines = 2,
    overflow = TextOverflow.Ellipsis,
    modifier = Modifier.weight(1f)  // Allows text to expand without breaking layout
)

// Opt into non-linear font scaling (Android 14+)
// Allows system to scale small text more, large text less (avoids huge headlines)
resources.configuration.fontScale  // Read current scale (1.0 = normal, 2.0 = largest)

6. RTL (Right-to-Left) Support

// Never use Left/Right — always use Start/End
// BAD (breaks RTL):
Row {
    Spacer(Modifier.width(16.dp))
    Text(text, Modifier.padding(left = 8.dp))
}

// GOOD (works in LTR and RTL):
Row {
    Spacer(Modifier.width(16.dp))
    Text(text, Modifier.padding(start = 8.dp))
}

// In View XML:
// BAD:  android:paddingLeft, android:layout_marginLeft
// GOOD: android:paddingStart, android:layout_marginStart

// Force RTL in layout preview to catch issues early
// In Android Studio: Layout editor → Locale dropdown → RTL pseudo-locale

// Bi-directional text (mixed LTR + RTL content):
Text(
    text = "\u202B$arabicContent\u202C $englishSuffix",  // Unicode bidi markers
    textDirection = TextDirection.ContentOrRtl  // Let content determine direction
)

// Icons that need mirroring in RTL (arrows, back buttons):
// In XML: android:autoMirrored="true" on the drawable
// In Compose:
Icon(
    imageVector = Icons.AutoMirrored.Filled.ArrowBack,  // Auto-mirrors in RTL
    contentDescription = "Back"
)

// supportsRtl in Manifest:
// <application android:supportsRtl="true">
// Without this: system won't mirror layout for RTL locales

9. Interview Narration Scripts

When asked "How do you handle device fragmentation?"

"I think about fragmentation in three dimensions: screen, API level, and OEM behavior.

For screen fragmentation, I use Window Size Classes from Jetpack instead of checking 'is this a tablet?' — because a foldable phone in open position is a medium-width window, not a tablet. My layouts adapt to Compact, Medium, and Expanded widths, and I specifically test foldable postures like tabletop mode where the hinge is horizontal. For foldables, I use WindowInfoTracker to detect fold state and avoid placing interactive elements on the hinge bounds.

For API level, I lean on Jetpack libraries which handle compat — but I use feature flags to gate features that require newer APIs, and I have CI run tests against API 26 as our lowest supported version.

For OEM behavior — especially background process killing on Samsung and Xiaomi — I architect critical background work as Foreground Services with user-visible notifications, use FCM high-priority messages for time-sensitive delivery, and for messaging apps I show an onboarding prompt guiding users to disable battery optimization."

When asked "How do you approach accessibility in your designs?"

"I treat accessibility as a correctness requirement, not a post-ship audit. It's built into the definition of done for every component.

Concretely: every interactive Compose element has a content description that describes the action, not just the element — 'Like post, currently 47 likes' instead of 'heart icon'. I merge semantics for grouped elements so TalkBack reads them as one coherent unit. Touch targets are always at least 48dp even when the visual element is smaller. I use live regions for dynamic content like sync status or error banners so they're announced without stealing focus.

I integrate the Accessibility Test Framework with Espresso so every CI run catches missing labels and contrast violations automatically. And I do manual TalkBack passes with eyes closed before each release — if I can't complete the critical user journey by touch alone, it ships with a bug."

10. Rejection Patterns

What Signals L4/L5 on Fragmentation & A11y

  • "I test on a few devices" — No systematic strategy, no CI device matrix
  • "I check if screenWidthDp > 600 for tablet layouts" — Hardcoded breakpoints don't handle foldables; Window Size Classes is the correct approach
  • Not knowing Window Size Classes — This was introduced in 2022 and is the Google-recommended API; missing it is a gap
  • "I add contentDescription to images" — Bare minimum. L6 talks about semantic merging, live regions, focus management, stateDescription
  • Treating accessibility as optional or post-launch — L6 engineers know WCAG requirements and build accessibility into CI
  • Using Left/Right instead of Start/End — Shows RTL was never considered
  • No mention of OEM background killing — A major production reliability issue ignored
