Build System, App Bundles & Dynamic Feature Modules

Why this matters at L6: At Google scale, build times and binary size directly impact team velocity and user acquisition. L6 engineers don't just use Gradle — they diagnose build slowdowns, own the module graph, make strategic decisions about dynamic delivery, and understand why Play delivers different APKs to different devices.

1. Android App Bundle vs APK

The Publishing Format Shift

APK (legacy)App Bundle (.aab)
What it isComplete installable archive, one size for allUpload format — Play assembles optimized APKs per device
Typical size differenceBaseline (e.g. 80MB)~40% smaller delivered APK (e.g. 48MB)
ABI splitsManual multi-APK or fat binaryAutomatic — arm64, x86_64 delivered separately
Language splitsAll strings bundledOnly device language + fallback delivered
Screen densityAll drawables bundledOnly matching density delivered
Dynamic featuresNoYes — on-demand download post-install
Required sinceN/AAugust 2021 for new apps on Play Store

How Play Builds the Optimized APK

App Bundle (.aab) ├── base/ ← Always installed │ ├── dex/ ← All compiled Kotlin/Java code │ ├── res/ ← All resources (base module) │ ├── assets/ │ └── manifest/ ├── feature_camera/ ← Dynamic feature module │ ├── dex/ │ └── res/ └── feature_ar/ ← On-demand dynamic feature Play Store per-device APK assembly: Device: arm64, xxhdpi, English → base.apk (arm64 libs + xxhdpi res + en strings) + feature_camera.apk (if installed)

2. Dynamic Feature Modules (DFM)

When to Use DFMs

Delivery TypeWhen downloadedUse case
Install-timeDuring app installAlways-needed features, just splits binary
On-demandUser triggers downloadRare features: AR mode, advanced editor, guest checkout
Fast-followShortly after install, backgroundFrequently used but not critical at first launch
ConditionalIf device meets conditions at install timeFeature only for devices with specific hardware
// On-demand module download flow
class FeatureDownloadManager(private val splitInstallManager: SplitInstallManager) {

    fun installCameraFeature(
        onSuccess: () -> Unit,
        onFailure: (Exception) -> Unit,
        onProgress: (Int) -> Unit
    ) {
        val request = SplitInstallRequest.newBuilder()
            .addModule("feature_camera")
            .build()

        splitInstallManager.startInstall(request)
            .addOnSuccessListener { sessionId ->
                // Monitor install progress
                splitInstallManager.registerListener(
                    SplitInstallStateUpdatedListener { state ->
                        when (state.status()) {
                            SplitInstallSessionStatus.DOWNLOADING -> {
                                val progress = (state.bytesDownloaded() * 100 / state.totalBytesToDownload()).toInt()
                                onProgress(progress)
                            }
                            SplitInstallSessionStatus.INSTALLED -> onSuccess()
                            SplitInstallSessionStatus.FAILED -> onFailure(
                                Exception("Install failed: ${state.errorCode()}")
                            )
                            SplitInstallSessionStatus.REQUIRES_USER_CONFIRMATION -> {
                                // Show system dialog for large installs (>10MB)
                                splitInstallManager.startConfirmationDialogForResult(state, activity, REQUEST_CODE)
                            }
                        }
                    }
                )
            }
            .addOnFailureListener { onFailure(it) }
    }

    // Check if module is already installed
    fun isCameraInstalled(): Boolean =
        splitInstallManager.installedModules.contains("feature_camera")

    // Uninstall unused modules to reclaim storage
    fun deferredUninstall(modules: List<String>) {
        splitInstallManager.deferredUninstall(modules)
    }
}

4. Module Graph Design

The L6 Module Architecture

:app (thin, just wires everything) ├── :feature:feed:impl │ └── :feature:feed:api (interface only, no impl) │ └── :domain:feed (use cases, domain models) │ └── :core:data (Repository interfaces) ├── :feature:profile:impl │ └── :feature:profile:api │ └── :domain:user ├── :core:network (OkHttp, Retrofit setup) ├── :core:database (Room, DAOs) ├── :core:ui (shared Compose components, themes) ├── :core:testing (FakeRepositories, TestDispatchers, shared test utils) └── :core:analytics (AnalyticsClient interface)

The api/impl Split — Why It Matters for Build Speed

// :feature:feed:api module — interface ONLY, no implementation
// Other modules depend on this to call feed features
interface FeedRepository {
    fun feedFlow(): Flow<List<Post>>
    suspend fun refresh(): Result<Unit>
}

// :feature:feed:impl module — depends on :feature:feed:api
// Only :app knows about :impl modules
class FeedRepositoryImpl @Inject constructor(
    private val api: FeedApi,
    private val dao: FeedDao
) : FeedRepository { ... }

// BUILD SPEED IMPACT:
// Without api/impl split: changing FeedRepositoryImpl recompiles every module that uses it
// With api/impl split: FeedRepository interface unchanged → dependent modules NOT recompiled
// Result: changes to implementation = only :feature:feed:impl recompiles

Preventing Circular Dependencies

// WRONG: Circular dependency
// :feature:feed depends on :feature:profile
// :feature:profile depends on :feature:feed  ← CIRCULAR!

// FIX 1: Extract shared model to :core:models
// :core:models (User, Post models — no dependencies)
// :feature:feed depends on :core:models
// :feature:profile depends on :core:models

// FIX 2: Use events / callbacks instead of direct dependency
// :feature:feed emits PostClickedEvent
// :app handles PostClickedEvent → navigates to profile
// Neither feature knows about the other

// FIX 3: Enforce at build time with ArchUnit or module-graph-assert
// build.gradle.kts

//     maxHeight = 4  // Fail if module graph is deeper than 4 levels
//     restricted = [":feature:.* -> :feature:.*"]  // Features can't depend on features

5. R8 / ProGuard — Production Hardening

R8 Full Mode — What Changes

R8 featureWhat it doesSize impact
ShrinkingRemove unused classes, methods, fieldsOften 50% reduction
ObfuscationRename to single letters (a, b, c...)Additional 5–10%
OptimizationInline methods, remove dead branches, constant foldingAdditional 5%
Dex optimizationRewrite bytecode for ART efficiencyStartup speed improvement
Full modeAggressive: assumes non-null by default, more inliningAdditional 5–10%
# proguard-rules.pro — what L6 engineers know how to write

# Keep data classes used in JSON serialization (Gson, Moshi)
-keepclassmembers class com.example.data.model.** {
    <fields>;
}

# Keep Parcelable implementations (auto-generated CREATOR)
-keep class * implements android.os.Parcelable {
    public static final android.os.Parcelable$Creator CREATOR;
}

# Keep enum values (R8 removes unused enum entries)
-keepclassmembers enum * {
    public static **[] values();
    public static ** valueOf(java.lang.String);
}

# Keep classes accessed by reflection (Room, WorkManager)
-keep class * extends androidx.room.RoomDatabase
-keep @androidx.room.Entity class *
-keep @androidx.room.Dao class *

# Keep Kotlin Serialization
-keepattributes *Annotation*, InnerClasses
-dontnote kotlinx.serialization.AnnotationsKt

# SDK: always add consumerProguardFiles
# library/build.gradle.kts
android {
    defaultConfig {
        consumerProguardFiles("consumer-rules.pro")  // Applied to consuming app
    }
}

6. App Startup Time — Build Perspective

What Inflates Startup Time at Build Level

CauseImpactFix
Too many ContentProvidersEach runs on startup, synchronouslyReplace with App Startup library (single provider)
Multidex (legacy)Class loading overhead pre-API 21minSdk 21+ eliminates multidex overhead
Large DEX filesART verification time increasesEnable ART profile-guided compilation (baseline profiles)
Heavy static initializersRun at class load time, blocks main threadLazy initialization; R8 can sometimes inline these

Baseline Profiles — The Most Impactful Startup Optimization

// Baseline profiles tell ART which code paths to AOT-compile
// Result: 30-40% cold start improvement, 15-20% frame time improvement

// 1. Generate with Macrobenchmark
@ExperimentalBaselineProfilesApi
class BaselineProfileGenerator {
    @get:Rule
    val rule = BaselineProfileRule()

    @Test
    fun generate() = rule.collect(
        packageName = "com.example.app"
    ) {
        // Define critical user journeys
        pressHome()
        startActivityAndWait()  // Cold start

        // Navigate the feed
        device.findObject(By.res("feed_list")).wait(Until.hasObject(By.res("post_item")), 3000)

        // Open first post
        device.findObject(By.res("post_item")).click()
        device.waitForIdle()
    }
}

// 2. Output: baseline-prof.txt (human-readable) → compiled to baseline.profm
// 3. Include in app bundle → Play delivers profile with APK
// 4. ART pre-compiles these methods on device → faster startup

// build.gradle.kts
plugins {
    id("androidx.baselineprofile")
}
dependencies {
    baselineProfile(project(":benchmark"))
}

9. Custom Gradle Plugins for Team Conventions

Convention Plugins — Enforce Consistency at Scale

// build-logic/convention/src/main/kotlin/AndroidFeatureConventionPlugin.kt
// This plugin encodes team conventions — every feature module applies it
class AndroidFeatureConventionPlugin : Plugin<Project> {
    override fun apply(target: Project) {
        with(target) {
            with(pluginManager) {
                apply("com.android.library")
                apply("org.jetbrains.kotlin.android")
                apply("com.google.devtools.ksp")
                apply("dagger.hilt.android.plugin")
            }

            extensions.configure<LibraryExtension> {
                compileSdk = 35
                defaultConfig { minSdk = 26 }
                buildFeatures { compose = true }
                compileOptions {
                    sourceCompatibility = JavaVersion.VERSION_17
                    targetCompatibility = JavaVersion.VERSION_17
                }
            }

            dependencies {
                // Every feature module gets these automatically
                add("implementation", libs.findLibrary("hilt.android").get())
                add("ksp", libs.findLibrary("hilt.compiler").get())
                add("implementation", libs.findLibrary("compose.ui").get())
                add("testImplementation", libs.findLibrary("junit").get())
                add("testImplementation", libs.findLibrary("coroutines.test").get())
                add("testImplementation", project(":core:testing"))
            }
        }
    }
}

10. L6 Interview Narration Scripts

When asked "How do you manage build performance at scale?"

"Build performance is a team productivity problem at scale. When a full clean build takes 15 minutes, engineers avoid rebuilding, which means they run stale code. I approach this at three layers.

First, the module graph. The api/impl split is the single highest-leverage change: when you change implementation code, only one module recompiles instead of the whole dependency chain. Second, I enable configuration cache and build cache in Gradle properties — configuration cache alone saves 30–60 seconds on every incremental build. Third, for annotation processing, I migrate from KAPT to KSP wherever possible — Hilt and Room both support KSP and it's 2–3x faster because it doesn't generate stubs.

I also use convention plugins so every team member gets optimized build settings automatically — no tribal knowledge required. And I monitor build times in CI with build scans to catch regressions before they compound."

When asked about App Bundles and download size

"App Bundles shifted responsibility for APK assembly to the Play Store, which can now deliver device-optimized APKs. A user on an arm64 phone in Germany only gets the arm64 native libraries, xxhdpi resources, and German+English strings — not the 3 other ABIs and 50 other language packs. In practice this is a 30–50% size reduction with zero engineering effort beyond switching from APK to AAB publishing.

For features that are genuinely rarely used — say, an AR mode or an advanced editor — I'd consider Dynamic Feature Modules. But I'd evaluate this carefully: DFMs add complexity (delayed navigation, download error handling, module deinstall logic), so the size savings need to justify that cost. My threshold is roughly: if a feature adds more than 5MB and fewer than 20% of users ever touch it, DFM is worth the investment."

11. Rejection Patterns

What Signals L4/L5 on Build System

  • "I just use the default Gradle setup" — No awareness of build performance as a team problem
  • Not knowing KSP vs KAPT difference — Shows unfamiliarity with modern Android build toolchain
  • All modules depending on :app — Circular dependency or feature-on-feature deps show weak architecture judgment
  • "App Bundle is just a new APK format" — Missing the device-optimized delivery insight
  • No mention of baseline profiles — Missing the most impactful modern startup optimization
  • Flat module structure (everything in :app) — Can't scale, no build parallelism, no API isolation
