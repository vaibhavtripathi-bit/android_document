01 — Architecture Patterns

Android L6 System Design · Clean Architecture · MVI · MVVM · Modularization · DI

Why candidates with 15+ years get rejected here
Architecture is the topic where experienced engineers over-explain what they know and under-demonstrate why their choices matter. Getting rejected despite knowing the right answers almost always means: you described the architecture but didn't articulate the trade-offs, didn't connect to team/product impact, and waited for the interviewer to ask the hard questions instead of raising them yourself. This guide teaches you to lead, not follow.

1. How Architecture Questions Actually Work at L6

The interviewer gives you a product prompt — "Design a news feed app," "Design the architecture for Google Photos." Your job in the first 10 minutes is to set up a system they can push on — not to build something perfect. They will probe every layer. The real test is whether your answers to their probes sound like a tech lead who has run this system in production, not a candidate who read about it.

The Three Levels of Answer Quality

LevelWhat it sounds likeSignal
L4 — Descriptive "I'd use MVVM with a ViewModel and LiveData for the UI layer, a Repository for data, and Retrofit for network calls." Knows the pattern names. No differentiation.
L5 — Justified "I'd use MVVM because it has good lifecycle integration with Jetpack. I'd use StateFlow instead of LiveData because it has a structured concurrency integration and doesn't re-emit on resubscription." Can justify choices. Still reactive to prompts.
L6 — Proactive + Consequence-Aware "I'd use MVI with a unidirectional data flow because this feed has complex state (loading, error, pagination, selection state) that can easily get inconsistent in a bidirectional MVVM model. I've seen that cause subtle UI bugs in multi-action scenarios. The trade-off is more boilerplate — that's acceptable here because the team has 8 engineers and the explicit action sealed class acts as a contract. I'd also align the architecture with our module boundaries — each feature module owns its ViewModel and the :data layer is shared..." Raises the right problems before being asked. Talks about team, failure modes, and trade-offs in one breath.
The L6 Principle
Every architectural decision you make should come with: (1) the specific problem it solves in this context, (2) the trade-off you're accepting, and (3) the team or operational consequence. If you're just naming patterns, you're presenting at L5.

2. Clean Architecture in Android — The Right Depth

Clean Architecture is the foundation for any L6 Android design answer. You need to know it well enough to deviate from it intelligently, not just apply it blindly.

The Layers (and What Each One Actually Owns)

UI Layer
Composables / Views, ViewModel, UI State, UI Events. Owns: rendering logic, user interaction handling, navigation triggers. Must not: contain business logic, hold raw domain entities, call repositories directly.
↕ (one-way: UI observes ViewModel state)
Domain Layer
Use Cases (Interactors), Domain Models, Domain Repository Interfaces. Owns: business rules, orchestration of multiple data sources. Must not: import Android framework classes — pure Kotlin/Java only. This is what makes it testable without a device.
↕ (domain defines interfaces; data implements them)
Data Layer
Repository Implementations, Remote DataSource (Retrofit), Local DataSource (Room), Mappers. Owns: fetching, caching, source-of-truth decisions. Must not: leak network DTOs to domain or UI.
↕
Core / Common
Shared utilities, DI setup, network config, logging, analytics interfaces. Depended upon by all layers but depends on nothing project-specific.

The Critical Rule: Dependency Direction

Dependency Rule
Dependencies always point inward: UI → Domain ← Data. The Domain layer never imports Data or UI. This is enforced via module boundaries in a multi-module build — the :domain module has no dependency on :data or :ui in Gradle.
Interview Trap — Don't Skip This
Interviewers at L6 will ask: "How do you enforce these layer boundaries?" A hand-wavy "we use good practices" is an L4 answer.

L6 answer: "In a multi-module build, the Gradle module dependency graph is the enforcement mechanism. The :domain module has no Gradle dependency on :data — so it's impossible to import a Room entity from a Use Case at compile time. We also run ArchUnit tests (or custom lint rules) in CI that verify no class in the domain package imports from data or android.*."

When to Skip the Domain Layer

This is an L6 signal most candidates miss — knowing when NOT to apply Clean Architecture in full.

ScenarioDomain Layer Justified?Reasoning
Simple CRUD screen, 1 data sourceNoA Use Case that just delegates to a repository is pure overhead. ViewModel calls the repository directly.
Complex business rules, multiple data sourcesYesUse Cases isolate the orchestration logic and make it independently testable.
SDK / LibraryPartiallyThe domain boundary exists but may be internal only; public API is the only contract that matters.
Large team, multiple featuresYesThe domain layer provides a stable contract that the UI team and the data team can work against independently.
Say This in the Interview
"I'll use Clean Architecture with three layers, but I want to call out: I'd only add a Domain layer with explicit Use Cases where there's real business logic to isolate. For the feed-loading flow, a simple GetFeedUseCase is just delegation — in that case I'd have the ViewModel call the repository directly to avoid unnecessary indirection. I'll add the domain layer for the sync conflict resolution logic, where we need to combine data from local DB and network response with actual business rules."

3. MVI vs MVVM — The Trade-Off You Must Own

This is the most commonly probed architecture question at Android L6. Don't just recite differences — know when each one fails in production and be able to articulate it with a concrete scenario.

MVVM — When It Works and When It Doesn't

// MVVM ViewModel — simple, Jetpack-native
class FeedViewModel(
    private val repo: FeedRepository
) : ViewModel() {

    private val _items = MutableStateFlow<List<FeedItem>>(emptyList())
    val items: StateFlow<List<FeedItem>> = _items

    private val _isLoading = MutableStateFlow(false)
    val isLoading: StateFlow<Boolean> = _isLoading

    private val _error = MutableStateFlow<String?>(null)
    val error: StateFlow<String?> = _error

    // Problem: items, isLoading, error can become inconsistent
    // if updated separately. E.g., isLoading=false but error=null
    // and items=empty — is that initial state or failed-empty-result?
    fun loadFeed() { /* ... */ }
}
MVVM Failure Mode
When you have 3+ state variables in a ViewModel, they can represent inconsistent combinations. isLoading=false, error=null, items=empty — is that initial state or a successful empty result? The UI can't tell without extra flags. This leads to subtle bugs in screens with complex interaction: pull-to-refresh + pagination + selection state + error + empty state all interacting.

MVI — Single State Object, Unidirectional

// MVI — all state in one sealed class, actions as sealed class
data class FeedUiState(
    val items: List<FeedItem> = emptyList(),
    val isLoading: Boolean = false,
    val error: String? = null,
    val isRefreshing: Boolean = false,
    val selectedItem: FeedItem? = null
    // State is always a valid snapshot — no partial updates
)

sealed class FeedIntent {
    object LoadFeed : FeedIntent()
    object Refresh : FeedIntent()
    data class SelectItem(val item: FeedItem) : FeedIntent()
    data class LoadNextPage(val cursor: String) : FeedIntent()
}

class FeedViewModel(...) : ViewModel() {
    private val _state = MutableStateFlow(FeedUiState())
    val state: StateFlow<FeedUiState> = _state

    fun handleIntent(intent: FeedIntent) {
        when (intent) {
            is FeedIntent.LoadFeed -> loadFeed()
            is FeedIntent.Refresh -> refresh()
            is FeedIntent.SelectItem -> select(intent.item)
            is FeedIntent.LoadNextPage -> loadPage(intent.cursor)
        }
    }

    private fun loadFeed() {
        _state.update { it.copy(isLoading = true, error = null) }
        // ... fetch and update atomically
    }
}

The Trade-Off Table — Memorize This

DimensionMVVMMVI
State consistencyRisk of inconsistent multi-stream stateSingle source of truth — state is always valid
DebuggingHard to replay — which stream changed first?Intent log = complete audit trail of what happened
TestabilityTest each stream separatelyTest: given state + intent → expected new state
BoilerplateLower — less ceremonyHigher — sealed classes, reducers, effects
Compose integrationGood — collectAsState() per streamExcellent — single collectAsState(), one recomposition trigger
Team scaleFine for small teamsBetter for large teams — Intent contract is explicit
Best forSimple screens, few interactionsComplex screens: feed, chat, multi-step form, player
L6 Trade-off Script — Say This
"For this feed screen I'd choose MVI over MVVM. The reason is that we have at least five concurrent state dimensions: loading, refreshing, pagination, selection, and error. With separate StateFlows there's a risk of intermediate inconsistent states being observed by the UI — for example, items updated but isLoading still true for one frame. That leads to UI flicker that's hard to reproduce and debug.

With MVI, every state transition is an atomic copy() on the UiState data class. The trade-off I'm accepting is more boilerplate — the Intent sealed class and explicit reducer. That's worth it for this screen. For simpler screens in the same app — settings, profile — I'd stick with MVVM to avoid ceremony. The choice should match the complexity of the screen, not be a blanket rule."

4. UI State Design — The Part Most Candidates Rush

Your UiState data class design is often the first thing an interviewer evaluates. It tells them how well you think about all the states a screen can be in.

State Modelling Mistakes to Avoid

L4/L5 — Incomplete State Model
data class FeedState(
  val items: List<Item>,
  val isLoading: Boolean
)
// Missing: error, empty, refreshing,
// paginating, offline, selected
L6 — All Valid States Modelled
data class FeedUiState(
  val feed: FeedContent = FeedContent.Loading,
  val isRefreshing: Boolean = false,
  val pagingState: PagingState = Idle,
  val userMessage: UiMessage? = null
)
sealed class FeedContent {
  object Loading : FeedContent()
  data class Success(val items: List<Item>) : FeedContent()
  data class Error(val msg: String) : FeedContent()
  object Empty : FeedContent()
}

One-Time Events — The Snackbar Problem

This is a classic L6 probe: "How do you handle a Snackbar that should show once — not re-show on configuration change?"

// WRONG: String? in state — will re-show on recomposition
data class FeedState(val errorMessage: String? = null)

// CORRECT: Wrap in consumable event
data class UiMessage(
    val id: UUID = UUID.randomUUID(), // unique per event
    val message: String
)
// UI calls: viewModel.messageShown(message.id)
// ViewModel: _state.update { it.copy(userMessage = null) }

// Alternative: Channel<UiEffect> for true fire-and-forget side effects
sealed class FeedEffect {
    data class ShowError(val msg: String) : FeedEffect()
    data class NavigateTo(val route: String) : FeedEffect()
}
// Channel<> not StateFlow — no replay, consumed once
private val _effects = Channel<FeedEffect>(Channel.BUFFERED)
val effects = _effects.receiveAsFlow()
L6 Answer to "How do you handle one-time events?"
"There are two schools of thought and I've worked with both. The first is wrapping events in the UiState as a consumable holder — a data class with a UUID so the UI can mark it as consumed, and the ViewModel removes it. That's safe with Compose because state flows are conflated — you won't miss an event if you're subscribed.

The second is a separate Channel<UiEffect> exposed as receiveAsFlow(). The advantage is fire-and-forget semantics — navigation, analytics triggers, one-time sounds. The risk is that Channel can drop events if the consumer isn't active. For navigation I use the Channel approach with a BUFFERED channel. For error messages I use the consumable state pattern because I want the ability to show it again after the user dismisses and re-enters. The choice is deliberate, not arbitrary."

5. Modularization — The Real L6 Topic

At L6, modularization is not about code organization — it's about team boundaries, build performance, and deployment contracts. This is where you demonstrate staff-level thinking.

Module Types — Know Each One's Job

Module TypeContainsCan depend onExample
:appApplication class, DI root, nav graph stitchingAll feature modules:app
:feature:XUI, ViewModel, local Use Cases for one feature:core:*, :data:*, :domain:feature:feed
:data:XRepository impl, DataSource, Room DAO, Retrofit API:core:network, :core:database:data:feed
:domainUse Cases, Domain Models, Repository interfacesNothing except :core:common:domain
:core:XShared infra: network client, DB, DI, utils, analyticsExternal libraries only:core:network, :core:ui
:test:fixturesShared test fakes, factories, test rules:domain, :core:*:test:fixtures

Build Performance — The L6 Module Design Constraint

Staff-Level Concern
At scale (50+ modules, 100+ engineers), module graph design directly impacts developer productivity. A poorly structured graph where every module depends on a single fat :core module means every change to :core invalidates the entire build cache and triggers a full rebuild. This is a real cost: if it adds 3 minutes to every CI run and you have 500 PRs/day, that's 25+ engineer-hours wasted daily. 
// BAD: Fat :core — everything depends on it, every change = full rebuild
:feature:feed → :core // :core has network, DB, analytics, DI, utils, models
:feature:profile → :core
:feature:search → :core

// GOOD: Thin :core modules — each feature takes only what it needs
:feature:feed → :core:network, :core:ui, :data:feed
:feature:profile → :core:ui, :data:user
// Changing :core:network only rebuilds modules that depend on it

Avoiding Circular Dependencies

A circular dependency between modules causes a Gradle build failure. The fix is always introducing an abstraction layer — but knowing which abstraction is the L6 skill.

// PROBLEM: :feature:feed needs user info from :feature:profile
// If :feature:profile also uses :feature:feed → circular dep

// SOLUTION 1: Extract shared model to :domain or :core:model
:feature:feed → :core:model // UserSummary lives here
:feature:profile → :core:model
// Neither knows about the other

// SOLUTION 2: API module pattern
// :feature:profile:api — contains only interfaces/models
// :feature:profile:impl — contains implementation
// :feature:feed depends on :feature:profile:api, not :impl
// :app wires together :impl modules at startup
L6 Script — Modularization Answer
"My module structure would follow the api/impl split for feature modules that expose contracts to other features. :feature:profile:api contains only the UserProfileProvider interface and the UserSummary model. :feature:profile:impl contains the actual implementation. When :feature:feed needs to show user avatars, it depends only on :feature:profile:api — so it can't accidentally use Profile's internal ViewModels or DAOs.

This also has a build benefit: changing Profile's implementation doesn't trigger a recompile of Feed, because Feed only has a compile-time dependency on the API module, which changes rarely. We measure this impact with Gradle build scans and we've seen 40% improvement in incremental build times after this refactor at my previous company."

6. Dependency Injection — What the Interviewer Is Really Asking

When an interviewer asks about DI, they're not testing whether you know Hilt annotations. They want to know if you understand the object graph, scoping, and the consequences of getting it wrong.

Scoping — The Most Common Bug Source

// Hilt scopes — wrong scope = memory leaks or incorrect behavior

@Singleton           // Lives for app lifetime. Use for: network client, DB, analytics
class FeedRepository

@ActivityRetainedScoped // Survives config change, destroyed with activity
class FeedViewModel  // Hilt ViewModels are this scope by default

@ActivityScoped      // Destroyed on config change — rarely what you want for VMs
class Navigator      // Use for things that need the Activity context

@ViewModelScoped     // Tied to ViewModel lifetime — for UseCase instances that
class GetFeedUseCase // hold state (e.g., a pager, a flow)
Common Scoping Bug
Injecting an @ActivityScoped dependency into an @Singleton — this compiles but crashes at runtime with a Hilt scope violation. The rule: a narrower scope can depend on a broader scope, never the reverse. Singleton → ActivityRetainedScoped → ActivityScoped → FragmentScoped → ViewScoped.

Testing with DI — The L6 Design Requirement

// Your DI design must make it easy to swap implementations in tests
// Hilt testing: @TestInstallIn replaces a module entirely

@Module
@TestInstallIn(
    components = [SingletonComponent::class],
    replaces = [NetworkModule::class]
)
object FakeNetworkModule {
    @Provides @Singleton
    fun provideFakeApi(): FeedApi = FakeFeedApi()
}

// L6 insight: Your real NetworkModule should be set up so that
// FeedApi is injectable (not constructed inline), otherwise
// you can't replace it in tests without refactoring production code.
// Design for testability at the DI layer — not as an afterthought.

7. Navigation Architecture — Underrated at L6

Navigation is rarely a whole interview question but it comes up as a follow-up in almost every Android design. Having a crisp answer signals platform depth.

Key Decisions and What to Say

DecisionOptionsL6 Call
Nav framework Jetpack Navigation, manual back stack, custom router Jetpack Navigation for most apps; custom router only when you need imperative control not supported by the XML/Compose DSL (e.g., complex multi-back-stack with conditional auth flows).
Deep link handling Intent-based, nav graph deep links, manual routing Nav graph deep links for standard URIs. Add a DeepLinkHandler singleton in :core that normalizes incoming intents and dispatches to the nav graph — this decouples Activity from routing logic.
Auth gate Activity-per-flow, nav graph global action, conditional start destination Conditional start destination based on auth state in the nav graph. ViewModel observes auth state and emits a navigation effect. Never put auth logic in Activity.onCreate() — it's untestable.
Back stack between features Single nav graph, nested graphs, multiple back stacks Multiple back stacks (Jetpack Nav 2.4+) for bottom-nav tabs. Each tab is a nested nav graph with its own back stack. This matches user expectations and is the Google-recommended pattern since 2021.

8. Multi-Module Testing Strategy

Testing Across Module Boundaries

When you modularize, you introduce a new testing challenge: each module must be testable in isolation AND modules must be verified to integrate correctly. L6 engineers design for both layers.

Layer 1 — Module-Isolated Unit Tests

// :feature:feed:impl — tests FeedRepositoryImpl in full isolation
// Real interface from :feature:feed:api; everything else is a fake from :core:testing
class FeedRepositoryImplTest {
    private val fakeApi = FakeFeedApi()
    private val fakeDao = FakeFeedDao()
    private val repo = FeedRepositoryImpl(fakeApi, fakeDao, UnconfinedTestDispatcher())

    @Test
    fun `returns cached feed then refreshes from network`() = runTest {
        fakeDao.seed(cachedPosts)
        repo.feedFlow().test {
            assertEquals(cachedPosts, awaitItem())   // cache emitted immediately
            fakeApi.enqueueSuccess(freshPosts)
            assertEquals(freshPosts, awaitItem())    // fresh data after network
            cancelAndIgnoreRemainingEvents()
        }
    }
}

Layer 2 — Integration: Verify Hilt Wiring in :app

// :app module — integration test that the DI graph resolves correctly
@HiltAndroidTest
class AppModuleIntegrationTest {
    @get:Rule val hiltRule = HiltAndroidRule(this)
    @Inject lateinit var feedRepository: FeedRepository

    @Before fun setUp() = hiltRule.inject()

    @Test
    fun `FeedRepository binds to FeedRepositoryImpl`() {
        assertThat(feedRepository).isInstanceOf(FeedRepositoryImpl::class.java)
    }
}
// Missing @Provides → compile error (safe)
// Wrong scope (@ActivityRetainedScoped vs @Singleton) → caught here, not in prod

The :core:testing Module — Shared Fakes

// :core:testing is testImplementation-only — shared fakes live here
// build.gradle.kts for any module:
//   testImplementation(project(":core:testing"))
//   androidTestImplementation(project(":core:testing"))

// Benefits:
// 1. Fake fails to compile when interface changes → always in sync
// 2. One fake serves all test modules → no drift between feature test fakes
// 3. Test data factories and MainDispatcherRule shared across all modules
Say this unprompted: "I keep all shared fakes in a :core:testing module. The key benefit: when an interface changes, the fake fails to compile immediately — you can't have a fake that silently diverges from the real contract. The module is never in production dependencies, only testImplementation."

9. What Gets You Rejected — The Honest List

Based on patterns from real L6 debriefs, these are the architecture-domain reasons candidates with strong technical knowledge still get rejected:

Rejection Pattern 1 — Correct but Generic
You described Clean Architecture correctly but never connected it to the specific requirements of the question. "I'd use MVVM with a Repository" sounds the same for a video player as for a settings screen. The interviewer wants to hear why this pattern for this product.
Rejection Pattern 2 — Waited to Be Asked
You only talked about failure modes, testing strategy, and team impact when the interviewer prompted. At L6 you're supposed to bring those up yourself. If you need to be asked "what about error handling?" or "how would you test this?", you're demonstrating L5 scope, not L6.
Rejection Pattern 3 — Single Option
You chose MVVM and never mentioned MVI, or chose Clean Architecture and never mentioned the cost. At L6, presenting only one option signals either you don't know the alternatives or you haven't thought deeply about trade-offs.
Rejection Pattern 4 — No Team/Scale Thinking
You designed a great architecture for one developer. No mention of how module boundaries enforce team ownership, how a new engineer would onboard, or how the CI/CD pipeline would exercise each module independently.
Rejection Pattern 5 — Perfect Theory, No Production War Stories
"I would use X because the documentation says..." reads as a junior. "I've seen Y fail in production when Z happens, so I prefer X in this scenario" reads as a staff engineer. Frame your choices with the failure you're avoiding, even if hypothetical.

9. Full Sample Answer — "Design the Architecture for a News Feed App"

This is a complete 12-minute architecture narration at L6 level. Use it as a template.

Complete L6 Architecture Narration
Requirements Phase (2 min):
"Before I start, a few questions. Is this a social feed (user-generated content) or editorial? What's the expected DAU — are we at millions or hundreds of millions? I'm asking because it changes caching strategy. Do we need offline reading? What's the minimum supported API level? And should I focus on the Android architecture or do you want me to cover the back-end API surface too?"

[Interviewer: Social feed, ~50M DAU, offline reading required, API 24+, Android-only for now]

Architecture Overview (3 min):
"I'll use Clean Architecture with three layers enforced by Gradle module boundaries. The UI layer is a :feature:feed module with Compose and an MVI ViewModel. I'm choosing MVI over MVVM because this screen has at least six state dimensions — initial load, refreshing, paginating, error, empty, and selected state. Separate StateFlows can become inconsistent. MVI gives me atomic state snapshots via a single data class. The domain layer has a GetFeedUseCase that orchestrates between local DB and network. The data layer has a FeedRepository backed by a Room database as the single source of truth and a Retrofit-based remote data source.

Critical Path: User opens app → FeedViewModel dispatches LoadFeed intent → GetFeedUseCase asks FeedRepository → Repository first emits cached data from Room (immediate display, no blank screen), then fetches from network, merges, and updates Room → Room flow propagates changes to ViewModel → UI updates. This is the offline-first pattern."

Data Model (2 min):
"I'll separate three types: the network DTO from the API, the domain model, and the Room entity. They're different for a reason. The DTO mirrors what the backend sends — we don't control that. The Room entity is optimized for our query patterns — it may denormalize some data for faster queries. The domain model is what the UI knows about — it has no Room annotations, no Gson annotations, it's a pure Kotlin data class. Mappers convert between these layers at the data layer boundary. This way, a backend API change doesn't force a rewrite of UI code."

Trade-offs (3 min):
"Two explicit trade-offs I want to call out. First: Room as single source of truth vs serving network response directly to the UI. Serving network directly is simpler and faster to implement. But it breaks offline reading and creates inconsistency when the network response overlaps with locally-drafted content. Room as source of truth adds latency for first fetch but gives us offline for free and a single consistent stream. Given the requirement for offline reading, that's the right call here.

Second trade-off: pagination. I'll use cursor-based pagination rather than offset. With 50M DAU and a feed that's constantly being updated, offset pagination gives you duplicates and gaps when new items are inserted. Cursor-based pagination is stable under insertion. The downside is that jump-to-page is impossible — that's fine for an infinite feed where users scroll linearly."

Staff Layer — Things I'd raise without being asked (2 min):
"A few things I'd flag for the team. One: we need a feed cache eviction policy. Room will grow unbounded if we don't prune stale feed items. I'd keep the last 200 items and the last 7 days — delete on background WorkManager job. Two: observability. I'd instrument time-to-first-render and time-to-first-data separately. If first-data is fast but first-render is slow, that's a Compose performance issue. If first-data is slow, that's a cache miss or network issue. You can't tell them apart without separate metrics. Three: rollout. The first version of this feed will have issues. I'd ship it with a feature flag backed by remote config, with a kill switch that falls back to the existing list. That way we can disable it instantly without a Play Store release if we see a spike in ANRs or crashes."
