Testing Strategy for Android L6

Why this matters at L6: Testability is a first-class L6 signal. Google interviewers don't just ask if you write tests — they probe how you architect for testability, how you handle slow instrumented tests, how you structure fakes vs mocks, and whether your system can be tested without a device. L6 engineers own the testing strategy for their team, not just their own code.

1. The Android Testing Pyramid

▲ /E2E\ ← UI tests (Espresso/Compose UI Test) /─────\ ~5% of suite, slow, flaky, expensive / Integ \ ← Integration tests (Room, WorkManager) /──────────\ ~20% of suite, medium speed / Unit Tests \ ← Unit tests (JVM, no Android) /────────────────\ ~75% of suite, fast, deterministic
LayerSpeedFlakinessWhat it TestsTools
Unit~10msNoneLogic, algorithms, use cases, ViewModelsJUnit 5, kotlinx-coroutines-test, Turbine
Integration~100ms–1sLowRoom queries, WorkManager, DataStoreRobolectric, instrumented tests
E2E / UI~5–60sHighFull user flows, animations, real devicesEspresso, Compose UI Test, UI Automator
L6 framing: "I design systems so the vast majority of business logic can be tested at the unit level with zero Android dependencies. Every Android dependency is behind an interface. Slow instrumented tests are reserved for verifying framework integration, not business rules."

2. Fake-Based Architecture — The L6 Standard

Fakes vs Mocks — Know the Difference

Mock (Mockito, MockK)Fake (hand-written)
What it isProxy that records calls, returns configured valuesSimplified real implementation for testing
BehaviourConfigured per test (every.mock.call stubbed)Has internal state, behaves like real thing
Refactoring safetyBreaks when interface changes (good) but also when internals change (bad)Fails to compile when interface changes → safe
Integration testingPoor — doesn't exercise real logicExcellent — can simulate state machines
L6 preferenceOnly for leaf-node external deps (analytics, network)For all internal interfaces you control
// Interface (in :domain module)
interface UserRepository {
    suspend fun getUser(id: String): Result<User>
    fun userFlow(id: String): Flow<User>
    suspend fun updateUser(user: User): Result<Unit>
}

// Fake (in :test-utils module, shared across test suites)
class FakeUserRepository : UserRepository {
    private val _users = MutableStateFlow<Map<String, User>>(emptyMap())
    var shouldFail = false
    var failureException: Exception = IOException("Network error")

    override suspend fun getUser(id: String): Result<User> {
        if (shouldFail) return Result.failure(failureException)
        return _users.value[id]
            ?.let { Result.success(it) }
            ?: Result.failure(NoSuchElementException("User $id not found"))
    }

    override fun userFlow(id: String): Flow<User> {
        return _users.mapNotNull { it[id] }
    }

    override suspend fun updateUser(user: User): Result<Unit> {
        if (shouldFail) return Result.failure(failureException)
        _users.update { it + (user.id to user) }
        return Result.success(Unit)
    }

    // Test helpers
    fun seedUser(user: User) = _users.update { it + (user.id to user) }
    fun simulateNetworkError() { shouldFail = true }
    fun simulateSuccess() { shouldFail = false }
}

// MainDispatcherRule.kt (shared test utility)
class MainDispatcherRule(
    val testDispatcher: TestCoroutineDispatcher = TestCoroutineDispatcher()
) : TestWatcher() {
    override fun starting(description: Description) =
        Dispatchers.setMain(testDispatcher)
    override fun finished(description: Description) =
        Dispatchers.resetMain()
}

// FeedViewModelTest.kt
class FeedViewModelTest {

    @get:Rule
    val mainDispatcherRule = MainDispatcherRule()

    private val fakeFeedRepository = FakeFeedRepository()
    private lateinit var viewModel: FeedViewModel

    @Before
    fun setup() {
        viewModel = FeedViewModel(fakeFeedRepository)
    }

    @Test
    fun `initial state is Loading`() = runTest {
        // Turbine: collect and assert Flow emissions cleanly
        viewModel.uiState.test {
            assertEquals(FeedUiState.Loading, awaitItem())
            cancelAndIgnoreRemainingEvents()
        }
    }

    @Test
    fun `load success emits Success state`() = runTest {
        val fakeFeed = listOf(Post(id = "1", title = "Test"))
        fakeFeedRepository.seedFeed(fakeFeed)

        viewModel.uiState.test {
            assertEquals(FeedUiState.Loading, awaitItem())  // initial
            viewModel.load()
            assertEquals(FeedUiState.Success(fakeFeed), awaitItem())  // after load
            cancelAndIgnoreRemainingEvents()
        }
    }

    @Test
    fun `load failure emits Error state`() = runTest {
        fakeFeedRepository.simulateNetworkError()

        viewModel.uiState.test {
            skipItems(1)  // skip Loading
            viewModel.load()
            val state = awaitItem()
            assertTrue(state is FeedUiState.Error)
            cancelAndIgnoreRemainingEvents()
        }
    }

    @Test
    fun `refresh clears previous data and reloads`() = runTest {
        val initialFeed = listOf(Post(id = "1"))
        val updatedFeed  = listOf(Post(id = "1"), Post(id = "2"))
        fakeFeedRepository.seedFeed(initialFeed)

        viewModel.uiState.test {
            skipItems(1) // Loading
            viewModel.load()
            assertEquals(FeedUiState.Success(initialFeed), awaitItem())

            fakeFeedRepository.seedFeed(updatedFeed)
            viewModel.refresh()
            assertEquals(FeedUiState.Loading, awaitItem())
            assertEquals(FeedUiState.Success(updatedFeed), awaitItem())
            cancelAndIgnoreRemainingEvents()
        }
    }
}
Turbine is the correct tool for testing Flows — it handles timing, backpressure, and provides clean assertion APIs. Never use toList() or manual launch { collect {} } in tests.

4. Testing Room Database

In-Memory Room Testing

// Room tests run as instrumented tests OR with Robolectric
// Prefer Robolectric for speed (JVM, no device needed)

@RunWith(RobolectricTestRunner::class)
class UserDaoTest {

    private lateinit var db: AppDatabase
    private lateinit var dao: UserDao

    @Before
    fun setup() {
        db = Room.inMemoryDatabaseBuilder(
            ApplicationProvider.getApplicationContext(),
            AppDatabase::class.java
        )
        .allowMainThreadQueries()  // OK in tests, never in prod
        .build()
        dao = db.userDao()
    }

    @After
    fun teardown() { db.close() }

    @Test
    fun insertAndRetrieveUser() = runTest {
        val user = UserEntity(id = "1", name = "Alice", email = "alice@example.com")
        dao.insert(user)
        val retrieved = dao.getById("1")
        assertEquals(user, retrieved)
    }

    @Test
    fun `Flow emits new values when database updates`() = runTest {
        dao.getAllUsersFlow().test {
            assertEquals(emptyList<UserEntity>(), awaitItem())

            dao.insert(UserEntity(id = "1", name = "Alice"))
            assertEquals(1, awaitItem().size)

            dao.insert(UserEntity(id = "2", name = "Bob"))
            assertEquals(2, awaitItem().size)

            cancelAndIgnoreRemainingEvents()
        }
    }

    @Test
    fun upsert_updatesExistingRecord() = runTest {
        dao.insert(UserEntity(id = "1", name = "Alice", version = 1))
        dao.upsert(UserEntity(id = "1", name = "Alice Updated", version = 2))
        val result = dao.getById("1")
        assertEquals("Alice Updated", result?.name)
    }
}

// Migration testing — ALWAYS write these before shipping migrations

    fun migrate_2_to_3() {

        helper.createDatabase(TEST_DB, 2).apply {
            execSQL("INSERT INTO users VALUES ('1', 'Alice', 'alice@test.com')")
            close()
        }

        // Apply migration and validate schema
        val db = helper.runMigrationsAndValidate(TEST_DB, 3, true, MIGRATION_2_3)

        // Verify data integrity after migration
        db.query("SELECT * FROM users WHERE id = '1'").use { cursor ->
            assertTrue(cursor.moveToFirst())
            assertEquals("Alice", cursor.getString(cursor.getColumnIndex("name")))
            // Verify new column has default value
            assertNotNull(cursor.getString(cursor.getColumnIndex("display_name")))
        }
    }
}

5. Testing WorkManager

// WorkManager testing with TestListenableWorkerBuilder
@RunWith(AndroidJUnit4::class)
class SyncWorkerTest {

    private lateinit var context: Context
    private lateinit var fakeSyncRepository: FakeSyncRepository

    @Before
    fun setup() {
        context = ApplicationProvider.getApplicationContext()
        // Configure WorkManager for testing
        WorkManagerTestInitHelper.initializeTestWorkManager(context)
        fakeSyncRepository = FakeSyncRepository()
    }

    @Test
    fun `worker returns success when sync completes`() = runTest {
        val worker = TestListenableWorkerBuilder<SyncWorker>(context)
            .setWorkerFactory(SyncWorkerFactory(fakeSyncRepository))
            .build()

        val result = worker.doWork()
        assertTrue(result is ListenableWorker.Result.Success)
    }

    @Test
    fun `worker returns retry on network error`() = runTest {
        fakeSyncRepository.simulateNetworkError()

        val worker = TestListenableWorkerBuilder<SyncWorker>(context)
            .setWorkerFactory(SyncWorkerFactory(fakeSyncRepository))
            .setRunAttemptCount(1)
            .build()

        val result = worker.doWork()
        assertTrue(result is ListenableWorker.Result.Retry)
    }

    @Test
    fun `worker returns failure after max retries exceeded`() = runTest {
        fakeSyncRepository.simulateNetworkError()

        val worker = TestListenableWorkerBuilder<SyncWorker>(context)
            .setWorkerFactory(SyncWorkerFactory(fakeSyncRepository))
            .setRunAttemptCount(3)  // Simulate 3 previous attempts
            .build()

        val result = worker.doWork()
        assertTrue(result is ListenableWorker.Result.Failure)
    }

    @Test
    fun `worker is enqueued with correct constraints`() {
        val testDriver = WorkManagerTestInitHelper.getTestDriver(context)!!
        val workManager = WorkManager.getInstance(context)

        SyncWorker.enqueue(workManager)

        val workInfos = workManager.getWorkInfosByTag(SyncWorker.TAG).get()
        assertEquals(1, workInfos.size)
        assertEquals(WorkInfo.State.ENQUEUED, workInfos[0].state)

        // Simulate constraints met
        testDriver.setAllConstraintsMet(workInfos[0].id)
        assertEquals(WorkInfo.State.RUNNING, workManager.getWorkInfoById(workInfos[0].id).get()?.state)
    }
}

8. Integration Testing Architecture

Testing the Full Stack with Fake Server

// MockWebServer for network integration tests
class FeedRepositoryIntegrationTest {

    private val mockWebServer = MockWebServer()
    private lateinit var api: FeedApi
    private lateinit var db: AppDatabase
    private lateinit var repository: FeedRepository

    @Before
    fun setup() {
        mockWebServer.start()
        api = Retrofit.Builder()
            .baseUrl(mockWebServer.url("/"))
            .addConverterFactory(GsonConverterFactory.create())
            .build()
            .create(FeedApi::class.java)
        db = Room.inMemoryDatabaseBuilder(context, AppDatabase::class.java).build()
        repository = FeedRepositoryImpl(api, db.feedDao())
    }

    @After
    fun teardown() {
        mockWebServer.shutdown()
        db.close()
    }

    @Test
    fun `getFeed returns cached data while refreshing`() = runTest {
        // Seed cache
        db.feedDao().insertAll(listOf(FeedEntity(id = "cached", title = "Cached Post")))

        // Queue network response
        mockWebServer.enqueue(MockResponse()
            .setBody("""{"posts": [{"id": "new", "title": "New Post"}]}""")
            .setResponseCode(200))

        repository.getFeed().test {
            // First emission: cached data
            val cached = awaitItem()
            assertEquals("Cached Post", cached[0].title)

            // Second emission: fresh network data
            val fresh = awaitItem()
            assertEquals("New Post", fresh[0].title)

            cancelAndIgnoreRemainingEvents()
        }
    }

    @Test
    fun `getFeed returns cached data on network error`() = runTest {
        db.feedDao().insertAll(listOf(FeedEntity(id = "1", title = "Cached")))
        mockWebServer.enqueue(MockResponse().setResponseCode(500))

        repository.getFeed().test {
            assertEquals("Cached", awaitItem()[0].title)
            cancelAndIgnoreRemainingEvents()
        }
        // No error thrown — offline-first returns cache gracefully
    }
}

9. Test Doubles Decision Framework

TypeWhen to UseExample
FakeInternal interfaces you own: repositories, use cases, data sourcesFakeUserRepository with in-memory map
StubReturn values, no behaviour verificationevery { prefs.getString("key") } returns "value"
MockVerify interaction: was this called with these args?verify { analytics.track("screen_view") }
SpyReal object + verify calls. Use sparingly.Wrapping real Retrofit client to verify headers
DummyRequired parameter, never used in testpassing null context to a presenter
Anti-pattern: Mocking everything in every test. If your test verifies 5 mock interactions ("verify this was called, then this, then this") you're testing implementation not behaviour. Prefer fakes that allow behaviour-level assertions.

11. What L6 Says About Testing Strategy

When asked "How do you approach testing in Android?"

"I design for testability from the start — not as an afterthought. Every class that contains business logic has an interface extracted so it can be replaced with a fake in tests. I never mock what I own; I write fakes that have real behaviour — a FakeRepository backed by an in-memory map that emits new Flow values when updated.

My testing pyramid has 75% unit tests running on the JVM in milliseconds. For the ViewModel layer, I inject dispatchers so I can use TestCoroutineDispatcher and Turbine for Flow assertions — no waiting for real time. Room tests use in-memory databases with Robolectric — no emulator needed, runs in CI in seconds. I reserve instrumented tests for framework integration: database migrations, WorkManager constraints, and Compose UI tests that require a real rendering engine.

At the platform level, I ensure CI blocks merges on unit and Robolectric test failures (under 5 minutes), and runs the full device matrix on Firebase Test Lab post-merge. For screenshot testing I use Paparazzi — no device, just golden images in the repo. When a PR changes a composable, CI automatically generates new screenshots and surfaces them in the PR for visual review."

12. L6 Rejection Patterns

What Signals L4/L5 on Testing

  • "I write unit tests for ViewModels and instrumented tests for everything else" — No pyramid strategy, missing the middle layer
  • "I mock the repository in every test" — Mocking what you own = brittle tests that break on refactoring
  • No mention of Flow testing tools — "I just collect to a list" shows unawareness of timing issues
  • Not injecting dispatchers — "I use Dispatchers.IO directly" → tests require real threads, flaky and slow
  • Instrumented tests for everything — Shows inability to design testable architecture; slow CI is a team productivity tax
  • No CI strategy — L6 owns testing at the team/org level, not just their own code
  • No mention of screenshot/regression testing — Missing a whole class of automated quality gates
