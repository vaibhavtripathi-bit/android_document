11 — Case Study: Analytics & Crash Reporting SDK

Full 45-minute L6 answer · SDK-first thinking · API design · Battery impact · Privacy · Reliability

Why this case study matters most for L6
Designing a production SDK is the highest-fidelity test of Staff Engineer thinking. You must demonstrate: API surface design and versioning, consumer impact awareness (binary size, battery, memory), reliability without harming the host app, privacy compliance by default, and operational ownership of a system used by teams you don't control. This question exposes every gap in L5→L6 thinking.

Requirements Phase 3–4 min

Requirements Narration
"Let me understand the scope before I design.

Audience: Is this an internal SDK used by our own app teams, or a public SDK distributed to third-party developers? The API design requirements are very different.

[Interviewer: Both — initially internal, with a roadmap to open-source it for third parties.]

Features: Are we combining analytics and crash reporting in one SDK, or are these separate? What kinds of analytics — custom events, screen views, user identification, A/B experiment exposure? For crash reporting — uncaught exceptions, ANRs, non-fatal errors, breadcrumbs?

[Interviewer: One SDK, both features. Analytics: custom events, screen tracking, user identity. Crash: uncaught exceptions, ANRs, non-fatal errors, breadcrumbs, device context.]

Scale: How many host apps will integrate this? How many events per day at peak?

[Interviewer: 50 internal apps initially, ~1B events/day at peak.]

Constraints: Any battery or binary size budget? Privacy requirements (GDPR, CCPA)?

[Interviewer: Must not impact battery life measurably. Must support GDPR consent. Keep APK impact under 500KB.]

My non-functional requirements: SDK must never crash the host app, must work offline with deferred upload, must support consent-gated initialization, and must have a public API surface that can be versioned without breaking consumers."

Architecture Overview 4–5 min

Public API Surface (analytics-api module — thin, stable) ├── AnalyticsSdk.track(event: AnalyticsEvent) ├── AnalyticsSdk.identify(userId: String, traits: Map) ├── AnalyticsSdk.screen(screenName: String, properties: Map) ├── CrashReporter.captureException(throwable: Throwable) ├── CrashReporter.addBreadcrumb(crumb: Breadcrumb) └── CrashReporter.setUser(userId: String) Internal Implementation (analytics-impl module — private) ├── EventPipeline │ ├── EventQueue (in-memory ring buffer, max 1000 events) │ ├── EventEnricher (session ID, device info, timestamp) │ ├── EventValidator (schema validation, PII scrubbing) │ └── BatchUploader (network delivery, retry) │ ├── EventStore (Room DB — persistence across kills) │ └── pending_events table │ ├── CrashCapture │ ├── UncaughtExceptionHandler │ ├── AnrWatchdog (watchdog thread) │ ├── BreadcrumbBuffer (ring buffer, last 100) │ └── CrashReportSerializer (file-based, sync write) │ ├── UploadScheduler (WorkManager — guaranteed delivery) └── ConsentManager (gate all collection behind user consent)
Module Split — The L6 API Design Choice
The analytics-api module contains only the public interfaces and data classes. The analytics-impl module contains the implementation. Consumer apps depend only on analytics-api. This means: changing the implementation doesn't trigger a recompile of consumer apps. Adding a new method to the impl doesn't break the consumer's API file. The impl can be replaced entirely without the consumer knowing.

Public API Design 5–6 min

// analytics-api module — everything here is a public contract

// Initialization — supports both auto-init and manual init
object AnalyticsSdk {

    @Volatile private var instance: AnalyticsClient? = null

    fun initialize(context: Context, config: AnalyticsConfig): AnalyticsClient {
        return instance ?: synchronized(this) {
            instance ?: AnalyticsClientImpl(context.applicationContext, config).also { instance = it }
        } // double-checked locking — safe, idempotent
    }

    fun getInstance(): AnalyticsClient =
        instance ?: throw IllegalStateException("AnalyticsSdk not initialized. Call initialize() first.")

    // Auto-init support: reads API key from manifest metadata
    internal fun autoInit(context: Context) {
        val apiKey = context.packageManager
            .getApplicationInfo(context.packageName, PackageManager.GET_META_DATA)
            .metaData?.getString("analytics_api_key") ?: return
        initialize(context, AnalyticsConfig.Builder(apiKey).build())
    }
}

// The client interface — what consumers code against
interface AnalyticsClient {
    fun track(event: AnalyticsEvent)
    fun identify(userId: String, traits: Map<String, Any> = emptyMap())
    fun screen(screenName: String, properties: Map<String, Any> = emptyMap())
    fun flush()        // force upload, e.g., before logout
    fun reset()        // clear all local data, e.g., on logout
    fun optOut()       // disable all collection (GDPR)
    fun optIn()        // re-enable collection after consent
}
// Interface enables: mocking in tests, no-op implementation for CI, future replacement

// Event model — typed, extensible
sealed class AnalyticsEvent {
    abstract val name: String
    abstract val properties: Map<String, Any>

    data class Custom(
        override val name: String,
        override val properties: Map<String, Any> = emptyMap()
    ) : AnalyticsEvent()

    data class ButtonTap(val buttonId: String, val screenName: String) : AnalyticsEvent() {
        override val name = "button_tap"
        override val properties = mapOf("button_id" to buttonId, "screen" to screenName)
    }
    // Standard events → typed, no stringly-typed names → compile-time safety
    // Custom events → flexible for one-off tracking
}

// Config — Builder pattern for forward compatibility
class AnalyticsConfig private constructor(
    val apiKey: String,
    val endpoint: String,
    val batchSize: Int,
    val flushIntervalMs: Long,
    val maxStoredEvents: Int,
    val isDebugLogging: Boolean
) {
    class Builder(private val apiKey: String) {
        private var endpoint = "https://api.analytics.example.com/v1"
        private var batchSize = 50
        private var flushIntervalMs = 30_000L
        private var maxStoredEvents = 5_000
        private var debugLogging = false

        fun endpoint(url: String) = apply { endpoint = url }
        fun batchSize(n: Int) = apply { batchSize = n }
        fun flushEvery(ms: Long) = apply { flushIntervalMs = ms }
        fun maxStoredEvents(n: Int) = apply { maxStoredEvents = n }
        fun debugLogging() = apply { debugLogging = true }
        fun build() = AnalyticsConfig(apiKey, endpoint, batchSize, flushIntervalMs, maxStoredEvents, debugLogging)
    }
}

Event Pipeline — Performance & Reliability 6–8 min

// track() must be non-blocking — called from UI thread in high-frequency scenarios
class AnalyticsClientImpl(...) : AnalyticsClient {
    private val queue = ArrayBlockingQueue<RawEvent>(1000) // bounded ring buffer
    private val scope = CoroutineScope(Dispatchers.IO + SupervisorJob())
    private val consentGranted = AtomicBoolean(true)

    override fun track(event: AnalyticsEvent) {
        try {
            if (!consentGranted.get()) return  // GDPR opt-out
            val raw = RawEvent(
                event = event,
                timestamp = System.currentTimeMillis(),
                sessionId = sessionManager.currentSessionId(),
                deviceInfo = deviceInfoCache  // pre-computed at init, not per-event
            )
            val offered = queue.offer(raw) // non-blocking; drops if full
            if (!offered) {
                metrics.increment("analytics.event_dropped") // track drops
            }
            if (queue.size >= config.batchSize) {
                scope.launch { flushAsync() } // trigger flush when batch ready
            }
        } catch (e: Exception) {
            // NEVER propagate from track() — SDK must not crash host app
            internalLog("track() failed", e)
        }
    }

    private suspend fun flushAsync() {
        val batch = mutableListOf<RawEvent>()
        queue.drainTo(batch, config.batchSize)
        if (batch.isEmpty()) return

        // Validate + scrub PII before persisting
        val validated = batch.mapNotNull { validator.validate(it) }

        // Persist to DB first
        eventStore.insertAll(validated.map { it.toEntity() })

        // Upload
        try {
            uploader.upload(validated)
            eventStore.deleteAll(validated.map { it.id })
        } catch (e: IOException) {
            // Network failed — events stay in DB for next flush
            metrics.increment("analytics.upload_failed")
        }
    }

    // Periodic flush via timer
    private fun startFlushTimer() {
        scope.launch {
            while (isActive) {
                delay(config.flushIntervalMs)
                flushAsync()
            }
        }
    }

    // ProcessLifecycleOwner — flush when app backgrounds
    init {
        ProcessLifecycleOwner.get().lifecycle.addObserver(
            LifecycleEventObserver { _, event ->
                if (event == Lifecycle.Event.ON_STOP) scope.launch { flushAsync() }
            }
        )
    }
}

PII Scrubbing — Privacy by Design

// Validate and scrub before any persistence or upload
class EventValidator(
    private val piiPatterns: List<Regex> = listOf(
        Regex("""\b[A-Za-z0-9._%+-]+@[A-Za-z0-9.-]+\.[A-Z|a-z]{2,}\b"""), // email
        Regex("""\b\d{10,11}\b"""),         // phone numbers
        Regex("""\b\d{4}[-\s]\d{4}[-\s]\d{4}[-\s]\d{4}\b""") // credit cards
    )
) {
    fun validate(event: RawEvent): RawEvent? {
        if (event.event.name.isBlank()) return null
        if (event.event.name.length > 100) return null  // reject suspiciously long names
        return event.copy(
            event = event.event.scrubPii(piiPatterns)   // replace PII with "[REDACTED]"
        )
    }
}
// PII scrubbing happens before DB write — never persisted even locally

Crash Reporting — Full Implementation 6–8 min

class CrashReporterImpl(
    private val context: Context,
    private val crashStore: CrashStore
) {
    private val breadcrumbs = BreadcrumbBuffer(maxSize = 100)
    private val anrWatchdog = AnrWatchdog(timeout = 5_000) { stack -> onAnrDetected(stack) }
    private val previousHandler = Thread.getDefaultUncaughtExceptionHandler()

    fun install() {
        // Register our handler BEFORE anrWatchdog starts
        Thread.setDefaultUncaughtExceptionHandler { thread, throwable ->
            try {
                captureAndWriteCrash(thread, throwable, isFatal = true)
            } finally {
                previousHandler?.uncaughtException(thread, throwable) // must call previous
            }
        }
        anrWatchdog.start()
        uploadPendingCrashReports() // upload from previous crashed session
    }

    fun captureException(throwable: Throwable, context: Map<String, Any> = emptyMap()) {
        try {
            captureAndWriteCrash(Thread.currentThread(), throwable, isFatal = false, context)
            scope.launch { uploadPendingCrashReports() } // upload non-fatals immediately
        } catch (e: Exception) {
            internalLog("captureException failed", e) // never propagate
        }
    }

    private fun captureAndWriteCrash(
        thread: Thread, throwable: Throwable,
        isFatal: Boolean, extraContext: Map<String, Any> = emptyMap()
    ) {
        val report = CrashReport(
            id = UUID.randomUUID().toString(),
            timestamp = System.currentTimeMillis(),
            isFatal = isFatal,
            threadName = thread.name,
            stackTrace = throwable.stackTraceToString(),
            breadcrumbs = breadcrumbs.snapshot(),
            deviceInfo = collectDeviceInfo(),
            memoryInfo = collectMemoryInfo(),
            extraContext = extraContext
        )
        // For fatal crashes: write synchronously to file — process is dying
        // Cannot use Room (DB might be locked) or coroutines (thread is dying)
        // Must use blocking I/O: Files.write() to a private directory
        if (isFatal) {
            crashStore.writeToFileSynchronously(report) // ~1-2ms, blocking but OK — app is crashing
        } else {
            scope.launch { crashStore.writeToDb(report) } // non-fatal: async Room write
        }
    }

    private fun uploadPendingCrashReports() {
        // Upload crash reports from previous sessions
        // Called at SDK init on next app launch after a crash
        // Use WorkManager with CONNECTED constraint — guaranteed delivery
        workManager.enqueueUniqueWork(
            "crash_upload",
            ExistingWorkPolicy.REPLACE,
            OneTimeWorkRequestBuilder<CrashUploadWorker>()
                .setExpedited(OutOfQuotaPolicy.RUN_AS_NON_EXPEDITED_WORK_REQUEST)
                .setConstraints(Constraints(requiredNetworkType = NetworkType.CONNECTED))
                .build()
        )
    }

    private fun collectDeviceInfo() = DeviceInfo(
        manufacturer = Build.MANUFACTURER,
        model = Build.MODEL,
        osVersion = Build.VERSION.SDK_INT,
        availableMemoryMb = getAvailableMemoryMb(),
        totalMemoryMb = getTotalMemoryMb(),
        isLowMemory = isLowMemoryDevice()
    )
}

L6 Script — Battery Budget
"Let me quantify the battery impact. The biggest cost for any analytics SDK is network — each radio activation on cellular costs roughly 15mAh due to the radio tail power. Without batching, 100 events tracked per session = 100 radio activations. With batching at 50 events per flush, that's 2 radio activations — a 50x improvement. Combined with time-based flushing (every 30 seconds), and background flush on app stop, we minimize radio activations to a predictable ceiling: approximately 1 activation per 30 seconds of active use.

I'd measure this in production using the Android Battery Historian tool and target staying below the 'bad behavior' threshold in Android Vitals — which flags apps that are in the top 5% for battery drain. I'd add an internal metric: 'network requests per session' — if this rises above 10 for a 10-minute session, our batching is broken."

Staff+ Layer 5–6 min

Complete Staff+ Narration
On the public API contract:
"Every public method I add today is a promise to every team that integrates this SDK. I'd use the Kotlin Binary Compatibility Validator as a required CI check — the API file is committed to version control, and any public API change requires an explicit version bump decision. I'd also have a deprecation policy: nothing is removed in the same major version it was deprecated in. Teams need 3-6 months notice and a migration guide before anything is removed."

On testing the SDK:
"The SDK needs two levels of testing. Unit tests for the pipeline logic — given N events, assert the batch is formed correctly, assert PII is scrubbed, assert the upload is retried on network failure. Integration tests run against a real Room database and a fake HTTP server. End-to-end tests: an instrumented test that initializes the real SDK in a test app and verifies events appear in the mock backend. This full test suite runs in CI before every release."

On rollout of the SDK itself:
"Because this SDK is used by 50 internal apps, I can't just push a breaking change. My versioning strategy: all internal apps pin to a specific version (not a range). When I release a new major version, I provide a migration guide and a 6-week migration window. I'd hold a migration office hour where app teams can ask questions. I'd also provide a codemod script for mechanical migrations (e.g., renaming a method). The SDK team owns the migration experience — it's not the consuming team's problem."

On privacy compliance by default:
"GDPR compliance is not an SDK option — it's the default behavior. The SDK starts in 'collection enabled' mode, but the documentation explicitly states: if your app operates in the EU, you must call optOut() before SDK init and only call optIn() after user consent. I'd provide a ConsentManager utility class that wraps the common pattern. All internal apps would go through a privacy review that checks this pattern is implemented correctly — and we'd add a lint rule that flags AnalyticsSdk.initialize() calls not accompanied by a consent check."

Follow-up Questions to Prepare

  • "How do you handle SDK initialization on a background thread?" — The initialize() method is thread-safe via double-checked locking with @Volatile. Room initialization is deferred to a background coroutine — if a track() call arrives before Room is ready, events go into the in-memory queue and are persisted when Room is ready.
  • "What if an app calls track() 10,000 times per second?" — The bounded queue (1000 capacity) drops events when full, increments a 'dropped' metric, and never allocates unboundedly. The SDK self-limits naturally. If a consumer is calling track() at 10K/s, they have a bug — we'd surface it via the dropped events metric.
  • "How do you deobfuscate stack traces from host apps?" — Each host app uploads their ProGuard/R8 mapping.txt to our crash dashboard. We store mappings keyed by app version code. When a crash comes in, we look up the mapping for that version and deobfuscate the stack trace server-side before displaying it in the dashboard.
  • "How do you test the ANR watchdog?" — In a test app, we deliberately block the main thread for 6 seconds (in a debug build only — gated by a test flag). The watchdog should fire and capture the main thread stack trace. This test runs as an instrumented test in the SDK's CI pipeline.
