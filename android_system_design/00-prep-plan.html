1. What to Expect at L6

Android system design at L6/Staff level is fundamentally different from L4/L5. The interviewer isn't checking if you know how RecyclerView works — they're evaluating whether you can design the architecture of a product feature end-to-end, make principled trade-offs, anticipate failure modes, and think about multi-team impact.

L6 Signal
At L6 the question is not "can you build this?" — it's "how would you lead a team to build this reliably at scale, and what would you have done differently the second time?"
AspectL4/L5L6 Staff
ScopeSingle feature or componentCross-feature, cross-team, cross-platform
Trade-offsCorrect answer expectedNo single correct answer — justification matters
Failure modesNice-to-haveMust proactively address reliability, degradation, rollback
API designInternal onlyPublic SDK surface, versioning, backwards compat
OwnershipMy modulePlatform impact, team contracts, deprecation paths
Android-specific depthFramework usageFramework internals + where to diverge from defaults

Round Formats You'll Encounter

  • Pure Android system design — "Design YouTube's video player for Android." Entirely client-side, deep platform knowledge required.
  • Full-stack mobile design — "Design a ride-sharing app." You'll cover both Android architecture and back-end API surface, caching, offline.
  • Platform / SDK design — "Design a crash reporting SDK for Android." Focuses on API design, reliability, minimal footprint, distribution.
  • Cross-cutting infrastructure — "Design a feature-flag system for a large Android codebase." Touches build, testing, rollout, observability.

2. The L6 Bar — How You're Judged

Google's evaluation maps to the following signals. Hitting all of these in 45 minutes is the bar.

SignalWhat it looks like at L6Common failure mode
Problem Decomposition Identifies non-obvious sub-problems; asks the right clarifying questions about scale, constraints, platform versions Jumps straight into components without understanding the problem
Architecture & Design Clean layering, clear module boundaries, correct use of Android primitives; names the pattern and explains why Over-engineers with patterns not justified by the requirements
Trade-off Reasoning Explicitly compares 2–3 options with concrete pros/cons tied to the specific product constraints States only one approach; can't defend against pushback
Android Platform Depth Cites specific APIs, explains their behavior, knows when not to use them; talks about Binder, Looper, Vsync, ART, etc. Generic answers that could apply to any mobile platform
Reliability & Observability Covers crash handling, ANR avoidance, graceful degradation, metrics/logging, OOM strategies Designs the happy path only
Scalability & Performance Quantifies constraints (memory budget, frame budget), discusses batching, pagination, prefetch, diffing Vague "we can optimize it later"
API / Contract Design Defines clean interfaces, considers versioning and consumer ergonomics, avoids leaking internals Skips API design entirely; jumps to implementation
Staff-Level Thinking Discusses rollout strategy, A/B testing, feature flags, migration path from existing system, team impact Treats the problem as a solo implementation task

3. Answer Framework — RADIO+

Use a structured approach to avoid going silent or losing the thread. The RADIO framework (from Mobile System Design by Alex Lementuev) is well-suited to mobile — here's the L6-extended version:

R
Requirements (3–4 min) Ask about functional requirements (core use cases), non-functional requirements (scale, offline, battery, API levels, accessibility), and what's explicitly out-of-scope. Confirm DAU/device targets to calibrate your design.
A
Architecture Overview (4–5 min) Draw the high-level component diagram. Name the architecture pattern (MVI, MVVM, layered, hexagonal) and justify it for this use case. Identify the critical path — the sequence of components that must work for the core use case.
D
Data Model & Flow (5–6 min) Define the key data entities and their relationships. Describe data flow: local DB schema (Room), in-memory representation, network DTO vs domain model vs UI model. Discuss caching strategy and source of truth.
I
Interface / API Design (5–6 min) Design the public API surface: Repository interfaces, ViewModel contracts, or SDK APIs. Define threading guarantees. At L6, also address versioning, deprecation, and backwards compatibility.
O
Optimizations & Trade-offs (8–10 min) This is where L6 interviews are won or lost. Cover: memory management, battery impact, rendering performance, network efficiency, offline support, error handling, and at least 2 explicit trade-off comparisons with alternatives you rejected.
+
Staff+ Layer: Reliability & Rollout (5–6 min) How do you ship this safely? Feature flags, kill switches, gradual rollout, observability (metrics, traces, alerts), incident playbook. How do you migrate from the old system? What's the rollback plan?
Time Management
Most candidates spend too long on architecture and run out of time before trade-offs and reliability. Treat the first 15 minutes as setup — the last 20 minutes are where you demonstrate L6 thinking.

4. The 8 Core Domains

  • 🏗️
    App Architecture & Modularity Clean architecture, multi-module builds, dependency graphs, build performance, dynamic feature modules. You must be able to design the architecture of a large app and explain module boundaries.
  • 🌐
    Networking & Offline-First Caching strategies, pagination (cursor vs offset), request deduplication, optimistic updates, conflict resolution, background sync with WorkManager, handling flaky connectivity.
  • 🗄️
    Local Storage & Data Layer Room schema design, migrations, DataStore vs SharedPreferences, SQLite performance (indexes, WAL mode, query plans), encrypted storage, data eviction policies.
  • 🎬
    Media & Rendering Performance ExoPlayer architecture, adaptive bitrate, surface/texture pipeline, Vsync/Choreographer, frame pacing, image loading pipelines (Glide/Coil internals), RecyclerView prefetch, Compose recomposition.
  • ⚙️
    Background Work & Process Model WorkManager constraints, chaining, expedited work, foreground services, JobScheduler, battery optimization (Doze, App Standby Buckets), push vs poll, process lifecycle.
  • 🔒
    Security & Privacy KeyStore, EncryptedSharedPreferences, certificate pinning, ProGuard/R8, permission model, WebView security, data minimization, privacy sandbox.
  • 📦
    SDK & Library Design Initialization patterns (lazy vs eager, ContentProvider trick), API surface design, version compatibility, Jetpack-style API conventions, transitive dependency management, binary size impact.
  • 📊
    Observability & Release Engineering Crash reporting, ANR detection, custom metrics/traces, A/B testing infrastructure, feature flags, staged rollout, canary releases, build variants, CI/CD pipeline design.

5. Question Bank by Domain

These are real-world question types seen at FAANG Android L6 interviews. Each question has a Domain tag and difficulty.

App Architecture

  • L6 Design the architecture for Google Photos — offline gallery, sync engine, album management, search. Hard
  • L6 Design a large-scale modular Android app (e.g., super-app with 50+ teams). How do you manage module graph, avoid circular deps, keep build times sane? Hard
  • L6 Design the navigation architecture for an app with 200+ destinations, deep linking, and back-stack requirements. Medium
  • Design a theming system that supports runtime theme switching without an Activity restart.

Networking & Offline

  • L6 Design an offline-first messaging app (like WhatsApp). Focus on message ordering, delivery receipts, conflict resolution. Hard
  • L6 Design a news feed with pagination, stale-while-revalidate caching, and offline reading. Medium
  • Design the network layer for a social app: caching, retry policies, authentication refresh, request queuing.
  • Design a file download manager for Android with pause/resume, priority queues, and progress tracking.

Media & Rendering

  • L6 Design the Android video player for YouTube — adaptive streaming, PiP, background playback, casting. Hard
  • L6 Design an image loading library from scratch for Android. Compare to Glide/Coil. Medium
  • Design a TikTok-style infinite scroll feed with video preloading, memory management, and smooth playback transitions. Hard
  • Design a camera capture pipeline that supports real-time filters, multiple codecs, and low-latency preview.

Background Work

  • L6 Design a background sync engine for a productivity app (calendar, contacts, emails) that works correctly under Doze and App Standby. Hard
  • Design a robust upload manager: retry on failure, chunk large files, handle app kill mid-upload.
  • Design a notification system that supports scheduling, grouping, priority, and Do Not Disturb awareness.

SDK Design

  • L6 Design an analytics SDK for Android (similar to Firebase Analytics). Focus on: batching, persistence, battery impact, initialization, privacy controls. Hard
  • L6 Design a crash reporting SDK (similar to Crashlytics). Signal handling, ANR detection, breadcrumbs, non-fatal errors, deobfuscation. Hard
  • Design a feature-flag SDK: local evaluation, remote config sync, experiment assignment, offline support.
  • Design a logging SDK that is safe, low-overhead, and compliant with privacy regulations.

Security

  • Design secure credential storage for an Android banking app.
  • Design certificate pinning with bypass detection and graceful update mechanism.
  • L6 Design an app attestation system to detect rooted/compromised devices.

Observability & Release

  • L6 Design the A/B testing and feature flag infrastructure for a large Android app with 100M+ users.
  • Design a custom metrics and distributed tracing system for Android client performance.
  • Design a staged rollout system: canary, progressive rollout, automatic rollback triggers.

6. 12-Week Study Plan

How to Use This Plan
Each week has a depth topic (go deep, build a design doc), a question set (practice 1–2 full 45-min answers), and a mock goal. Adjust pace based on your existing strengths.
Phase 1 — Foundation (Weeks 1–3)

Week 1: Framework + Architecture

  • Master the RADIO+ framework
  • Study Clean Architecture, MVI, MVVM deep
  • Practice: Design a news feed app architecture
  • Record a 45-min answer, watch it back

Week 2: Networking & Offline

  • Offline-first patterns, caching strategies
  • Room + WorkManager + Retrofit interaction
  • Practice: Design offline-first messaging

Week 3: Background Work & Process Model

  • WorkManager internals, constraints, chaining
  • Doze, App Standby Buckets, FCM push
  • Practice: Design background sync engine
Phase 2 — Deep Dives (Weeks 4–7)

Week 4: Media & Rendering

  • ExoPlayer, MediaCodec, Surface pipeline
  • Image loading: Glide/Coil internals
  • Practice: Design YouTube video player

Week 5: SDK Design

  • API design principles, initialization patterns
  • Analytics SDK battery/privacy trade-offs
  • Practice: Design crash reporting SDK

Week 6: Performance & Memory

  • ART GC, memory budgets, Bitmap pools
  • Systrace, Perfetto, frame analysis
  • Practice: Design a performant infinite list

Week 7: Security

  • KeyStore, EncryptedSharedPreferences
  • Certificate pinning, attestation
  • Practice: Design a banking app data layer
Phase 3 — Staff-Level (Weeks 8–10)

Week 8: Observability & Release

  • Custom metrics, traces, alerting
  • Staged rollout, canary, kill switches
  • Practice: Design A/B testing infrastructure

Week 9: Modularization at Scale

  • Module graph design, build performance
  • API modules, implementation modules, Anvil/Hilt
  • Practice: Design super-app module structure

Week 10: Cross-cutting Design

  • Full-stack questions: client + API surface
  • Team impact, migration paths, deprecation
  • Practice: Design Google Drive Android app
Phase 4 — Simulation (Weeks 11–12)

Week 11: Mock Interviews

  • 2–3 full mock interviews with peers
  • Focus on trade-off articulation
  • Fix the top 3 weaknesses from feedback

Week 12: Polish & Review

  • Review all 8 domains with 1-page design docs
  • Practice whiteboard diagrams under time pressure
  • Final 2 full mocks targeting L6 signals

8. Mock Interview Checklist

Use this before and after every mock session to self-evaluate against the L6 bar.

Before the Interview

  • Have a whiteboard or drawing tool ready
  • Know your RADIO+ steps cold — no hesitation at the start
  • Set a timer for 45 minutes

During — Self-Check at Each Phase

PhaseDid you...?
Requirements Ask about scale, target API levels, offline requirements, battery sensitivity? Confirm what's out of scope?
Architecture Name the pattern and justify it? Identify the critical path? Draw a component diagram with clear boundaries?
Data Model Define entities and relationships? Separate network DTO from domain model from UI state? State the source of truth?
API Design Define interface contracts with threading guarantees? Address versioning for public APIs?
Optimizations Give at least 2 explicit trade-off comparisons? Address memory, battery, and network?
Staff+ Layer Cover rollout strategy, observability, and rollback? Discuss migration from existing system?

After — Debrief Questions

  1. What was the hardest trade-off? Could you defend it against pushback?
  2. Did you spend more than 15 minutes on architecture before getting to trade-offs?
  3. Did you proactively bring up failure modes, or only when the interviewer prompted?
  4. Did you mention any Android-specific APIs or internals that a generic mobile engineer wouldn't know?
  5. Did you demonstrate cross-team / platform-wide impact thinking at any point?
The L6 Differentiator
The candidates who clear L6 system design are the ones who proactively bring up the hard parts: failure modes, migration complexity, observability gaps, and team-level impact — without being asked. Own the full lifecycle of the system, not just the happy path.
