03 — Local Storage & Data Layer

Android L6 System Design · Room Schema · Migrations · DataStore · Encrypted Storage · Query Performance

Why candidates fail here
Most engineers know how to use Room. Very few can talk about schema design decisions, migration strategies for a live production app, WAL mode and query performance, and what breaks when you get any of these wrong. Those are the L6 questions.

1. Storage Solution Decision Tree

The first thing you should say in any design that involves local state is: "Let me map each data type to the right storage primitive." Using Room for everything or SharedPreferences for everything is an L4 signal.

Data TypeBest StorageWhy
Structured relational data (feed items, messages, users)Room (SQLite)Queryable, relational, observable with Flow, survives app kill
Simple key-value preferences (theme, notifications toggle, user ID)DataStore (Preferences)Coroutine-native, crash-safe, no ANR risk unlike SharedPreferences
Typed structured preferences (user config object)DataStore (Proto)Schema-enforced, no stringly-typed keys, backwards compatible with proto evolution
Sensitive data (tokens, PIN, biometric keys)EncryptedSharedPreferences or KeyStoreHardware-backed encryption; never store tokens in plaintext Room
Large binary blobs (downloaded PDFs, cached videos)File system (scoped storage)Room is not for blobs; use file path in Room, content in files
Ephemeral in-memory cache (decoded bitmaps, computed values)LruCache / in-process memoryNo I/O, fastest; lost on process kill — that's acceptable for computed values
Interview Probe — "Why not SharedPreferences?"
SharedPreferences has three serious problems you must know: (1) apply() is async but getSharedPreferences() blocks the main thread on first load — this is a real ANR source, measurable with Systrace. (2) It uses XML, not a crash-safe write protocol — a crash mid-write corrupts the file. (3) No Flow support — you have to register listeners manually. DataStore solves all three: uses atomic file writes via DataStore.updateData(), coroutine-native, and exposes data: Flow<T> directly.

2. Room Schema Design — L6 Depth

Separate Network DTOs from Room Entities — Always

// WRONG: using the network DTO as the Room entity
@Entity
data class FeedItem(  // This is both your network DTO and DB entity
    @PrimaryKey val id: String,
    @SerializedName("post_body") val postBody: String,
    @SerializedName("author_id") val authorId: String
    // Problem: any backend API change forces a DB migration
    // Problem: Gson annotations bleed into the persistence layer
)

// CORRECT: Three separate types with mappers between them
// 1. Network DTO — matches API shape
data class FeedItemDto(
    @SerializedName("id") val id: String,
    @SerializedName("post_body") val postBody: String,
    @SerializedName("author_id") val authorId: String
)

// 2. Room Entity — optimized for our queries, not API shape
@Entity(
    tableName = "feed_items",
    indices = [Index("authorId"), Index("createdAt")]
)
data class FeedItemEntity(
    @PrimaryKey val id: String,
    val body: String,
    val authorId: String,
    val createdAt: Long,   // stored as epoch ms — fast range queries
    val cachedAt: Long     // for freshness tracking
)

// 3. Domain Model — what the UI knows about, pure Kotlin
data class FeedItem(
    val id: String,
    val body: String,
    val author: Author,  // nested — joined query result in repository
    val postedAt: Instant
)

Indexing — The Query Performance Question

L6 — Interviewers will ask "How do you make this query fast?"
Without indexes, every Room query is a full table scan — O(N). With a feed of 10,000 items, that's noticeable on a mid-range device. You need to know what to index and why. 
// Rule: Add an index on every column used in WHERE, ORDER BY, or JOIN
@Entity(
    tableName = "feed_items",
    indices = [
        Index(value = ["createdAt"]),          // ORDER BY createdAt DESC
        Index(value = ["authorId"]),            // WHERE authorId = ?
        Index(value = ["authorId", "createdAt"]) // composite: filter + sort
    ]
)

// Composite index matters: (authorId, createdAt) is not the same as
// two separate indexes. Composite index supports:
// WHERE authorId = ? ORDER BY createdAt DESC  ✓ (uses composite)
// WHERE authorId = ?                          ✓ (uses composite, prefix match)
// ORDER BY createdAt DESC                     ✗ (can't use composite — no prefix)
// → still need a separate index on createdAt alone for this query

// Cost of indexes: write overhead, storage overhead
// Don't index every column — only query-hot columns
// Use EXPLAIN QUERY PLAN in SQLite to verify index usage

Relationships — The Right Way to Model Them

// One-to-many: Post → Comments
data class PostWithComments(
    @Embedded val post: PostEntity,
    @Relation(
        parentColumn = "id",
        entityColumn = "postId"
    )
    val comments: List<CommentEntity>
)

// Many-to-many: Post → Tags
@Entity(primaryKeys = ["postId", "tagId"])
data class PostTagCrossRef(val postId: String, val tagId: String)

data class PostWithTags(
    @Embedded val post: PostEntity,
    @Relation(
        parentColumn = "id",
        entityColumn = "id",
        associateBy = Junction(PostTagCrossRef::class)
    )
    val tags: List<TagEntity>
)

// Performance note: @Relation runs a second query per parent item
// For large datasets, consider a manual JOIN + custom data class instead
// @Relation(N+1 problem) vs JOIN (single query) — L6 trade-off to mention
L6 Signal — N+1 vs JOIN
"I'd use @Relation for small result sets — it's more ergonomic. But I'd flag a performance risk: @Relation uses N+1 queries internally — one query for posts, then one per post for comments. For a list of 200 posts, that's 201 DB queries. At Room's speed that's still fast, but if I'm seeing jank during scroll, I'd replace it with a manual JOIN query returning a flat result type, then map to the nested structure in the repository. I'd validate the choice with EXPLAIN QUERY PLAN in a debug build."

3. Database Migrations — The Production-Safety Question

This is one of the most underrated L6 topics. Getting migrations wrong in production causes crashes for all users on update — a P0 incident. Interviewers know this and will probe it.

What happens if you get this wrong
If a Room migration is incorrect or missing, Room throws an IllegalStateException on database open — which crashes the app immediately for every user who updates. With 10M+ users, this is a P0 incident that requires a hotfix release. You need to design migrations defensively.

Migration Anatomy

@Database(
    entities = [FeedItemEntity::class, UserEntity::class],
    version = 3,  // NEVER decrement this. Ever.
    exportSchema = true  // always true — exports schema JSON for version control
)
abstract class AppDatabase : RoomDatabase() {

    companion object {
        // Migration: v1 → v2: added cachedAt column to feed_items
        val MIGRATION_1_2 = object : Migration(1, 2) {
            override fun migrate(db: SupportSQLiteDatabase) {
                // SQLite: can ADD column but cannot change type or drop NOT NULL
                db.execSQL("ALTER TABLE feed_items ADD COLUMN cachedAt INTEGER NOT NULL DEFAULT 0")
                // Default 0 = treat all existing rows as "needs refresh"
            }
        }

        // Migration: v2 → v3: renamed table
        val MIGRATION_2_3 = object : Migration(2, 3) {
            override fun migrate(db: SupportSQLiteDatabase) {
                // SQLite doesn't support RENAME COLUMN until 3.25
                // Safe approach: create new table, copy data, drop old
                db.execSQL("CREATE TABLE feed_items_new (...)")
                db.execSQL("INSERT INTO feed_items_new SELECT id, body AS content, authorId, createdAt, cachedAt FROM feed_items")
                db.execSQL("DROP TABLE feed_items")
                db.execSQL("ALTER TABLE feed_items_new RENAME TO feed_items")
            }
        }

        fun build(context: Context): AppDatabase =
            Room.databaseBuilder(context, AppDatabase::class.java, "app.db")
                .addMigrations(MIGRATION_1_2, MIGRATION_2_3)
                // .fallbackToDestructiveMigration() — NEVER use in production
                .build()
    }
}

Migration Safety Rules — Memorize These

RuleWhy
Always exportSchema = true and commit the JSON filesRoom's MigrationTestHelper uses these to verify migrations against historical schema. Without them you can't test migrations properly.
Never fallbackToDestructiveMigration() in productionThis silently deletes all user data on schema mismatch. Acceptable only in debug builds.
Always provide a NOT NULL DEFAULT when adding columnsSQLite requires a default for NOT NULL columns added via ALTER TABLE. Without it the migration crashes.
Test every migration path with MigrationTestHelperTest v1→v3 directly (not just v1→v2→v3) — users who skipped an intermediate release will jump versions.
Never use room-generated schema as the migration — write it explicitlyAuto-generated migrations can miss edge cases in data transformation logic.
 
// Migration testing — required at L6
@RunWith(AndroidJUnit4::class)
class MigrationTest {
    @get:Rule
    val helper = MigrationTestHelper(
        InstrumentationRegistry.getInstrumentation(),
        AppDatabase::class.java
    )

    @Test
    fun migrate1To3() {
        // Create DB at version 1
        helper.createDatabase("test", 1).apply {
            execSQL("INSERT INTO feed_items VALUES ('id1', 'body', 'author1', 1000)")
            close()
        }
        // Run migrations 1→2→3
        val db = helper.runMigrationsAndValidate("test", 3, true,
            MIGRATION_1_2, MIGRATION_2_3)
        // Verify data integrity post-migration
        val cursor = db.query("SELECT content FROM feed_items WHERE id = 'id1'")
        cursor.moveToFirst()
        assertEquals("body", cursor.getString(0))
    }
}
L6 Script — Migration Safety
"Migrations are a P0 risk — a bad migration crashes the app on launch for every user who updates. My standard practice: export schema JSON into version control so I have a historical record of every schema version. I write migrations explicitly, never use fallbackToDestructiveMigration in production. And I test every migration combination with MigrationTestHelper — including non-sequential jumps like v1 to v4 directly, because some users skip intermediate versions if they haven't opened the app in months. I'd also add a canary metric: 'database open failure rate' — if it spikes on a release, that's a migration bug and we roll back immediately."

4. WAL Mode and Performance

Write-Ahead Logging is a SQLite feature that dramatically improves concurrent read/write performance. At L6 you need to know what it is and when to use it.

// Room enables WAL mode by default since Room 2.2
// But you should know WHY and WHEN it matters

// WAL mode: writes go to a separate -wal file first
// Readers never block writers; writers never block readers
// vs default journal mode: readers block writers and vice versa

// This matters in your architecture when you have:
// - Background sync writing to DB at the same time as UI reading
// - WorkManager workers inserting data while RecyclerView reads it
// Without WAL: the worker blocks the UI thread's DB reads → jank

Room.databaseBuilder(...)
    .setJournalMode(JournalMode.WRITE_AHEAD_LOGGING) // explicit, for clarity
    .build()

// WAL checkpoint: periodically the -wal file is merged back into main DB
// Default: automatic checkpoint at 1000 pages
// Large checkpoint can cause a brief stall — measure with Perfetto if needed
// Can manually trigger: db.query("PRAGMA wal_checkpoint(TRUNCATE)", null)
L6 Signal — Mentioning WAL Unprompted
"Room uses WAL mode by default and I'd leave that on. It matters here because we have background WorkManager sync jobs writing feed items to Room while the UI layer is simultaneously reading from the same table for the scroll view. Without WAL, SQLite's default journal mode causes readers to block writers — the sync job would stall the UI's DB reads and cause jank. WAL decouples reads and writes entirely."

5. DataStore — Full Picture

Preferences DataStore vs Proto DataStore

DimensionPreferences DataStoreProto DataStore
SchemaNo schema — stringly-typed keysProtobuf schema — typed, versioned
Type safetyKeys are typed but key names are strings — typos compile fineFully type-safe — proto compiler generates types
Schema evolutionManual — no toolingProto field numbers — safe additions, deprecation via reserved
Use whenFew simple preferences, small teamComplex user config, shared across teams, long-lived
 
// Preferences DataStore — correct usage pattern
val Context.userPreferences: DataStore<Preferences> by preferencesDataStore("user_prefs")

object PreferenceKeys {
    val DARK_MODE = booleanPreferencesKey("dark_mode")
    val NOTIFICATION_ENABLED = booleanPreferencesKey("notifications_enabled")
    val LAST_SYNC = longPreferencesKey("last_sync_timestamp")
}

class UserPreferencesRepository(private val dataStore: DataStore<Preferences>) {
    val isDarkMode: Flow<Boolean> = dataStore.data
        .catch { e -> if (e is IOException) emit(emptyPreferences()) else throw e }
        .map { it[PreferenceKeys.DARK_MODE] ?: false }
    // catch IOException: if DataStore file is corrupted, emit defaults
    // rather than crashing — DataStore will reset to clean state

    suspend fun setDarkMode(enabled: Boolean) {
        dataStore.edit { it[PreferenceKeys.DARK_MODE] = enabled }
        // edit() is a suspend function — safe from any coroutine, never blocks main thread
    }
}
Critical: Don't Create Multiple Instances
DataStore uses a file lock. Creating two DataStore instances pointing to the same file causes data corruption. Always use the by preferencesDataStore() property delegate — it's a singleton at the Context level. Inject it via Hilt as a @Singleton.

6. Encrypted Storage — The Security Layer

// EncryptedSharedPreferences — for tokens, session IDs
// Backed by Android KeyStore (hardware-backed on modern devices)
val masterKey = MasterKey.Builder(context)
    .setKeyScheme(MasterKey.KeyScheme.AES256_GCM)
    .build()

val encryptedPrefs = EncryptedSharedPreferences.create(
    context, "secure_prefs", masterKey,
    EncryptedSharedPreferences.PrefKeyEncryptionScheme.AES256_SIV,
    EncryptedSharedPreferences.PrefValueEncryptionScheme.AES256_GCM
)

// What lives here: access tokens, refresh tokens, user ID
// What does NOT live here: passwords (use KeyStore directly + biometric)
// What does NOT live here: large data (encrypted Room for that)

// Encrypted Room database — for sensitive structured data
// Use SQLCipher via Room's SupportOpenHelperFactory
val passphrase = SQLiteDatabase.getBytes(keyFromKeyStore) // key from KeyStore
val factory = SupportFactory(passphrase)
Room.databaseBuilder(...)
    .openHelperFactory(factory)
    .build()
// Perf note: SQLCipher adds ~15-20% overhead on queries vs unencrypted
// Only use for genuinely sensitive data (health, financial, messages)
L6 Script — Storage Security
"For auth tokens I'd use EncryptedSharedPreferences backed by the Android KeyStore with AES256-GCM. The key is hardware-backed on devices with a secure enclave — even if someone extracts the app's data partition, they can't decrypt the tokens without the KeyStore key, which never leaves the hardware.

For the feed content itself, I wouldn't encrypt the Room database — feed items are not sensitive, and SQLCipher adds ~15-20% query overhead. If this were a health app or banking app, I'd use SQLCipher for the whole database with a key derived from the user's biometric authentication. The encryption decision should match the data sensitivity — encrypting everything 'just to be safe' has real performance and complexity costs."

8. Database Corruption Recovery

Room databases can corrupt due to: process killed mid-write, disk full during a transaction, or hardware failure. L6 engineers design a recovery path rather than assuming corruption never happens.

Detecting and Recovering

// Corruption surfaces as SQLiteException with "malformed" or "corrupt" in message
suspend fun safeDatabaseRead(block: suspend () -> T): T? {
    return try {
        block()
    } catch (e: SQLiteException) {
        if (e.message?.contains("malformed") == true ||
            e.message?.contains("corrupt") == true) {
            analytics.track("db_corruption_detected", mapOf("msg" to e.message))
            recoverFromCorruption()
        }
        null
    }
}

fun recoverFromCorruption() {
    // Strategy A: data is a server cache → safe to delete and repopulate
    context.deleteDatabase("app.db")
    // Room recreates on next access; next sync repopulates from server

    // Strategy B: data includes unsynced user writes → try integrity check first
    val isOk = runIntegrityCheck()
    if (!isOk) {
        backupRawDbFile()       // copy corrupt file for debugging
        context.deleteDatabase("app.db")
    }
}

fun runIntegrityCheck(db: SupportSQLiteDatabase): Boolean {
    return try {
        db.query("PRAGMA integrity_check").use { cursor ->
            cursor.moveToFirst() && cursor.getString(0) == "ok"
        }
    } catch (e: SQLiteException) { false }
}

Prevention: WAL Mode + Disk Space Guard

// WAL (Write-Ahead Log) is crash-safe — partial writes don't corrupt the DB
Room.databaseBuilder(context, AppDatabase::class.java, "app.db")
    .setJournalMode(RoomDatabase.JournalMode.WRITE_AHEAD_LOGGING) // default on API 16+
    .build()

// Guard before large write operations
fun hasSufficientDisk(minBytes: Long = 10 * 1024 * 1024 /* 10MB */): Boolean {
    val stat = StatFs(context.filesDir.path)
    return stat.availableBlocksLong * stat.blockSizeLong > minBytes
}
// If disk is critically low, defer the bulk insert rather than risk a torn write
Say this unprompted: "My recovery strategy depends on data ownership. If the database is purely a cache of server data, I delete and recreate — the user loses nothing. If it contains unsynced user writes (drafts, offline posts), I run PRAGMA integrity_check first and only delete as a last resort, after copying the raw file for debugging. Either way, I track corruption events in production analytics — a spike tells me there's an underlying disk or hardware issue on certain devices."

9. What Gets You Rejected Here

Rejection Pattern — One Storage Solution for Everything
"I'd store everything in Room" without distinguishing between structured data, preferences, tokens, and large files. This misses the nuance expected at L6.
Rejection Pattern — No Migration Strategy
You mentioned Room versioning exists but didn't talk about how to safely evolve the schema in production with existing user data. Migrations are where production incidents happen.
Rejection Pattern — "Just use SharedPreferences"
SharedPreferences for anything user-facing shows you're not up to date with Android best practices. DataStore has been the recommended replacement since 2021.
Rejection Pattern — No Performance Awareness
No mention of indexes, query limits, WAL mode, or the N+1 risk of @Relation. At L6 you're expected to think about the performance implications of your schema design.
