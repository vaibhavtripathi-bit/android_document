08 — Observability & Release Engineering

Android L6 System Design · Metrics · Traces · Feature Flags · Staged Rollout · A/B Testing · Incident Response

This is the most pure L6 topic in the set
Observability and release engineering are almost never discussed by L4/L5 candidates unless prompted. At L6 you raise them unprompted — because you've shipped code to millions of users and lived through incidents where you couldn't tell what was broken, or couldn't roll back fast enough. This section is your most powerful differentiator.

1. The Observability Mindset

Observability means: given any unexpected behavior in production, you can understand what's happening without deploying new code. For an Android app, this is harder than for a server — you can't SSH into your users' devices.

Three Pillars Applied to Android
  • Metrics: Numeric measurements over time — crash rate, ANR rate, p95 latency, session length, feature usage rate
  • Traces: End-to-end operation timings — time from button tap to content visible, network request lifecycle
  • Logs: Structured event records — what the user did, what the system did, in what order
All three must be collected and queryable. Metrics tell you something is wrong. Traces tell you where. Logs tell you why.

Instrumentation Code Pattern

// Abstract your analytics behind an interface — swap providers without code changes
interface MetricsReporter {
    fun increment(metric: String, vararg tags: Pair<String, String>)
    fun recordHistogram(metric: String, value: Long, vararg tags: Pair<String, String>)
    fun recordGauge(metric: String, value: Double, vararg tags: Pair<String, String>)
    fun startTrace(name: String): Trace
}

// Usage — measure every user-facing operation
val trace = metrics.startTrace("feed.load")
val result = feedRepository.getFeed()
trace.putAttribute("source", if (result.fromCache) "cache" else "network")
trace.putMetric("item_count", result.items.size.toLong())
trace.stop()

// Instrument at the right level of granularity:
// Too coarse: "feed loaded in 2s" — doesn't tell you if it was DB or network
// Too fine: every function call — noise, overhead, unactionable
// Right: "feed.db_read_ms", "feed.network_ms", "feed.total_ms" — each actionable

3. Distributed Tracing for Mobile

// Trace a user action end-to-end: button tap → server response → UI updated
// Use a trace ID that spans both client and server

class TracedFeedRepository(
    private val feedApi: FeedApi,
    private val tracer: Tracer
) {
    suspend fun getFeed(): List<FeedItem> {
        val traceId = UUID.randomUUID().toString() // one ID for this user action
        val span = tracer.startSpan("feed.repository.getFeed", traceId)
        try {
            // Pass trace ID to server — enables correlation in server logs
            val response = feedApi.getFeed(traceId = traceId)
            span.setStatus("ok")
            return response.items
        } catch (e: Exception) {
            span.setStatus("error", e.message)
            throw e
        } finally {
            span.end()
        }
    }
}

// On the server: log the X-Trace-ID header on every request
// When investigating a user complaint: query server logs by traceId
// to see the full request → DB query → response lifecycle for that exact action

// OkHttp interceptor to inject trace ID into every request
class TraceInterceptor(private val tracer: Tracer) : Interceptor {
    override fun intercept(chain: Interceptor.Chain): Response {
        val traceId = tracer.currentTraceId() ?: UUID.randomUUID().toString()
        return chain.proceed(
            chain.request().newBuilder()
                .header("X-Trace-ID", traceId)
                .header("X-Device-Model", Build.MODEL)
                .header("X-App-Version", BuildConfig.VERSION_NAME)
                .build()
        )
    }
}
L6 Script — Observability
"I instrument every user-facing operation with three measurements: time-to-first-data, time-to-first-render, and a source tag (cache vs network). These are separate because if first-data is fast but first-render is slow, that's a Compose performance issue — a completely different fix than a slow network call.

For production debugging, I use distributed tracing with a trace ID that spans both the Android client and the server. Every API call gets an X-Trace-ID header. When a user reports an issue, they can provide a trace ID from their session and we can query server logs to see exactly what happened on both sides. Without this, debugging user-specific issues means looking at aggregate logs and hoping — that's not acceptable at staff level."

4. Feature Flags — The Safe Deployment Contract

L6 Principle
Every significant new feature ships behind a feature flag. Not because you don't trust your code — but because you can't anticipate every device, locale, carrier, OS version combination. Feature flags let you disable a feature in production without a Play Store release, which takes 1-3 days to roll out. 
// Feature flag types you need to know

// 1. Kill switch — disable a broken feature immediately
// Backed by remote config, evaluated on every app start
// Critical: must have a safe local default if remote config unreachable
val isNewFeedEnabled: Boolean
    get() = remoteConfig.getBoolean("feature_new_feed", defaultValue = false)
    // defaultValue = false = disabled if config fetch fails
    // NEVER use a default that could cause a crash if the flag is not fetched

// 2. Gradual rollout flag — percentage of users
// Backed by server-side evaluation tied to user ID
// Consistent: same user always gets same variant (hash(userId + flagName) % 100)
fun isUserInRollout(userId: String, flag: String, percentage: Int): Boolean {
    val hash = ("${userId}_${flag}").hashCode().and(0x7FFFFFFF) % 100
    return hash < percentage
}

// 3. A/B test flag — variant assignment
// Returns variant name — "control", "variant_a", "variant_b"
// Same user always gets same variant for statistical validity
fun getVariant(userId: String, experimentName: String): String {
    val hash = ("${userId}_${experimentName}").hashCode().and(0x7FFFFFFF) % 100
    return when (hash) {
        in 0..49 -> "control"
        else -> "variant_a"
    }
}

// Feature flag client — caches flags locally with TTL
class FeatureFlagClient(
    private val remoteConfig: RemoteConfigApi,
    private val localCache: FeatureFlagCache
) {
    private var lastFetch: Long = 0

    suspend fun refresh() {
        if (System.currentTimeMillis() - lastFetch < TTL_MS) return
        try {
            val flags = remoteConfig.fetchFlags()
            localCache.save(flags)
            lastFetch = System.currentTimeMillis()
        } catch (e: Exception) {
            // Network failure — use cached flags; never crash due to flag fetch failure
        }
    }

    fun isEnabled(flag: String): Boolean =
        localCache.get(flag) ?: defaultFlags[flag] ?: false
}

Flag Evaluation — Evaluation Order Matters

PrioritySourcePurpose
1 (highest)Local override (debug builds only)QA and developer testing — force any flag value
2Remote config (real-time)Kill switch, emergency disable
3Cached remote configNormal operation, TTL-based freshness
4 (lowest)Compiled defaultOffline fallback — must be safe

5. Staged Rollout — The Release Strategy

// Play Store staged rollout + feature flags = two independent dimensions
//
// Play Store rollout: controls which users get the APK at all
// Feature flag: controls which users see the new feature WITHIN the APK
//
// Standard release pipeline:
// 1% rollout → watch crash/ANR metrics for 24h
// 5% rollout → watch for 24h
// 10% → 25% → 50% → 100% → full release
// Auto-rollback trigger: if crash rate > 1.5x baseline → halt rollout

// Within the rolled-out APK, feature flag controls visibility:
// New feed: 0% on day 1 → 5% canary → 20% → 50% → 100%
// This gives two-level control:
// Bug in binary (crash): roll back at Play Store level
// Bug in feature: disable feature flag, keep APK rolling out

// Rollback plan — must be documented before rollout starts
// Q: How do we undo this release?
// A1: Feature flag kill switch — feature off within 60s, no new release needed
// A2: Halt Play Store rollout — stops new installs but doesn't affect existing
// A3: Emergency hotfix release — for critical security/crash bugs
// Have all three documented in the feature design doc before shipping
L6 Script — Staged Rollout
"Every significant feature I ship has a two-layer release strategy. The first layer is the Play Store staged rollout: 1% → 5% → 25% → 100% over several days, with automated metric checks at each gate. I watch crash-free session rate, ANR rate, and the specific metrics for the new feature. If the crash rate goes above 1.5x the pre-release baseline, the rollout halts automatically and pages the on-call engineer.

The second layer is a feature flag. The feature flag lets me disable the feature instantly — without a Play Store release — if I see a bug in the feature logic itself (not a crash, but wrong behavior). Play Store releases take 1-3 days to propagate. A feature flag response time is under 60 seconds for 100% of users.

Before any rollout, I write three sentences in the design doc: what's the kill switch, who owns the rollback decision, and what's the recovery plan if we need to roll back a DB migration that's already been applied to 50M devices. That last one is the hardest and most often not discussed."

6. A/B Testing Infrastructure

// A/B test design requirements:
// 1. Consistent assignment: same user gets same variant across sessions
// 2. Mutually exclusive experiments: user in experiment A can't be in experiment B
// 3. Statistical validity: minimum sample size before declaring a winner
// 4. Holdout groups: users permanently excluded from experiments (control baseline)

// Experiment assignment on device
data class Experiment(
    val name: String,
    val variants: List<Variant>,
    val eligibilityCriteria: EligibilityCriteria
)

class ExperimentAssigner(
    private val userId: String,
    private val remoteExperiments: List<Experiment>
) {
    fun getVariant(experimentName: String): String? {
        val experiment = remoteExperiments.find { it.name == experimentName } ?: return null
        if (!experiment.eligibilityCriteria.isEligible(userId)) return null

        // Deterministic hash — same userId + experimentName always maps to same bucket
        val bucket = murmurhash("${userId}_${experimentName}") % 100
        var cursor = 0
        for (variant in experiment.variants) {
            cursor += variant.weight
            if (bucket < cursor) return variant.name
        }
        return experiment.variants.last().name
    }
}

// Exposure logging — critical for valid A/B test analysis
// Log an exposure event the FIRST TIME a user encounters the variant
// Not when assigned — when they actually see it
// Exposure = user was exposed to the variant that could affect their behavior
fun getNewFeedVariant(): String {
    val variant = assigner.getVariant("new_feed_layout")
    if (variant != null) {
        analytics.track(ExperimentExposure("new_feed_layout", variant))
        // Log once per session, not per call — deduplicate with a seen-set
    }
    return variant ?: "control"
}
A/B Test Common Mistakes
  • Assignment ≠ Exposure: A user can be assigned a variant but never see it. Only count exposures for statistical analysis.
  • Novelty effect: New UI gets higher engagement for 2 weeks because it's new, not because it's better. Run experiments for at least 2 weeks.
  • Multiple testing problem: If you run 20 experiments simultaneously with p=0.05 threshold, 1 will show "significance" by chance. Use Bonferroni correction or Bayesian methods.
  • Interaction effects: Experiment A changes the feed algorithm. Experiment B changes the feed layout. Users in both get a confounded result.

7. Incident Response for Mobile

// Mobile incident response differs from server: you can't push a fix instantly
// Three response tiers:

// Tier 1 (Minutes): Feature flag kill switch
// Response time: < 60 seconds for 100% of users
// Use for: broken feature, wrong behavior, non-crash bug
// Limitation: requires feature to be behind a flag

// Tier 2 (Hours): Server-side fix
// Response time: deploy server fix, all app versions benefit immediately
// Use for: API issues, backend logic bugs, data corruption
// Limitation: only works if the bug is server-side

// Tier 3 (Days): Play Store hotfix release
// Response time: 1-3 days for 100% rollout, expedited review possible
// Use for: critical crash, security vulnerability, payment bug
// Play Store expedited review: request via Play Console for critical security issues

// Incident runbook — write this before shipping, not during the incident
// 1. What are the alerting thresholds? (crash rate, error rate, p99 latency)
// 2. Who gets paged? (on-call rotation)
// 3. How do we kill the feature? (flag name, who has access)
// 4. What's the rollback plan for DB migrations?
// 5. How do we communicate to affected users?
L6 Script — Incident Response
"Before I ship any feature with data model changes, I write a rollback plan for the worst case: what if 50M devices apply a DB migration that turns out to be wrong? The honest answer is: you can't undo a migration on devices that have already run it. So the migration must be additive-only — add columns, never drop or rename. If you need to rename a column, keep the old column, write new data to both, then drop the old column in a future release after verifying everything is healthy. I've seen teams ship non-backwards-compatible migrations and spend two weeks on a data recovery strategy. The migration design is the release strategy."

9. Statistical Significance for Canary Decisions

A common L6 failure: deciding to roll forward (or rollback) a release based on raw crash counts rather than statistically valid comparisons. L6 engineers know when they have enough data to make a confident decision — and when they don't.

The Core Problem with Raw Counts

At 1% rollout, your canary has 1/100th the traffic of production. If production normally has 50 crashes/hour, the canary will have ~0.5 crashes/hour. Seeing "2 crashes in the canary" might look bad — but is it statistically different from 0.5? You need a framework to answer that, not a gut check.

The Right Metrics for Canary Comparison

MetricUse rate, not countWhy
Crash ratecrashes / sessions (not crashes / hour)Session volume differs between canary and control
ANR rateANRs / sessionsSame — normalize to sessions
Error ratefailed requests / total requestsTraffic volume proportional, rate is comparable
LatencyP50/P95/P99 (not mean)Mean is skewed by outliers; percentiles show tail behavior

Sample Size Requirement Before Deciding

// Minimum sessions needed for a statistically valid canary decision
// Rule of thumb: enough to detect a 10% regression with 95% confidence

// If baseline crash rate = 0.5% (0.005):
// Minimum sample size ≈ 16 / (baseline_rate) for 80% power
// = 16 / 0.005 = 3,200 sessions minimum in the canary group

// At 1% rollout with 1M DAU: ~10,000 sessions/day in canary
// → meaningful data after ~8 hours

// Practical thresholds Google uses (approximate):
// - Hold at 1% for minimum 24 hours regardless of crash count
// - Do not proceed to 5% if canary crash rate > 2x baseline (with >1000 sessions)
// - Use error budget: if canary would consume >10% of monthly error budget at full rollout, hold

// Statistical test for two proportions (crash rate canary vs. baseline):
fun isCrashRateSignificantlyHigher(
    canarySessionCount: Int,
    canaryCrashCount: Int,
    baselineCrashRate: Double,  // e.g. 0.005 for 0.5%
    confidenceLevel: Double = 0.95
): Boolean {
    val canaryRate = canaryCrashCount.toDouble() / canarySessionCount
    // Z-test for proportion
    val se = sqrt(baselineCrashRate * (1 - baselineCrashRate) / canarySessionCount)
    val z = (canaryRate - baselineCrashRate) / se
    // z > 1.645 → reject null hypothesis at 95% confidence (one-tailed)
    return z > 1.645
}

The Decision Framework

ConditionDecisionWhy
Canary crash rate > 2× baseline AND >1000 sessionsRollback immediatelyStatistically significant regression with sufficient data
Canary crash rate > 2× baseline BUT <1000 sessionsHold — do not proceed or roll back yetInsufficient data; could be noise
Canary crash rate within 1.2× baseline AND >5000 sessionsProceed to 5%Sufficient evidence of no regression
Ambiguous (between 1.2× and 2×)Hold 24h, re-evaluateGather more data; watch trend direction
Say this unprompted: "I never make a rollout decision based on raw crash counts — I compare crash rates normalized to sessions, and only after the canary has enough traffic for the comparison to be statistically meaningful. At 1% rollout that's roughly 1000+ sessions before I trust the numbers. The question I'm always asking is: is this canary's crash rate statistically distinguishable from baseline, or is it just noise from small sample size?"

10. What Gets You Rejected Here

Rejection Pattern — "We'd monitor for crashes"
That's a starting point. Crashes are lagging indicators — your users have already had a bad experience. What are your leading indicators? Time-to-first-meaningful-paint regression, retry rate spike, cache hit rate drop — these tell you something is wrong before users start reporting crashes.
Rejection Pattern — No Rollback Plan
You described the feature and the rollout, but when asked "how do you roll this back?", you don't have a crisp answer. At L6 the rollback plan is part of the design. It's documented in the design doc, reviewed by the team, and tested before release.
Rejection Pattern — Only Happy-Path Metrics
You track feature adoption (positive) but not: feature error rate, fallback rate, performance regression, or retry rate. Positive metrics tell you what's working; negative metrics tell you what's broken silently.
