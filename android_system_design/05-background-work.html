05 — Background Work & Process Model

Android L6 System Design · WorkManager · Doze · App Standby · FCM · Foreground Services · Process Lifecycle

The rejection risk here
Background work is Android-specific and nuanced. Generic answers ("use a background thread") or oversimplified answers ("use WorkManager for everything") signal you haven't dealt with the battery/process management reality at scale. At L6 you need to demonstrate you understand WHY the Android background execution model is the way it is, and make choices accordingly.

1. The Android Process Model — Foundation

Android can and will kill your process. The OS has a priority ladder, and your app's process position on that ladder determines how aggressively it gets killed.

Process PriorityExampleKill likelihood
ForegroundActive Activity or Foreground ServiceNever killed unless extreme memory pressure
VisibleActivity partially visible (behind dialog)Very low
ServiceRunning background Service (no notification)Low — but can be killed on API 26+ after ~1 minute
CachedApp in background, not running anythingHigh — killed freely by LMK when memory is needed
L6 Implication
Any background work that must complete — even if the user kills the app — cannot rely on a coroutine in a ViewModel or a regular Service. It must be persisted (WorkManager enqueue to DB) before the work starts. If the process is killed mid-work, WorkManager will re-run the worker when the process comes back. This is the guaranteed execution contract that distinguishes WorkManager from every other approach.

Use this to pick the right tool. Using the wrong one is a signal that you don't know the Android execution model.

3. WorkManager — Deep Dive

Architecture Under the Hood

// WorkManager internals — what it actually does
// 1. Enqueued work is written to a Room database immediately
// 2. WorkManager monitors constraints (network, charging, battery)
// 3. When constraints are met, it dispatches to JobScheduler (API 23+)
// 4. JobScheduler schedules execution with the OS — survives reboots
// 5. On API < 23: falls back to AlarmManager + BroadcastReceiver
// 6. Worker runs in a process — if killed, JobScheduler re-runs it
// This is why guaranteed execution works: the OS itself tracks the job

// One-time work with constraints
val syncRequest = OneTimeWorkRequestBuilder<FeedSyncWorker>()
    .setConstraints(
        Constraints.Builder()
            .setRequiredNetworkType(NetworkType.CONNECTED)
            .setRequiresBatteryNotLow(true)  // don't run below ~15% battery
            .build()
    )
    .setBackoffCriteria(BackoffPolicy.EXPONENTIAL, 30, TimeUnit.SECONDS)
    .setInputData(workDataOf("userId" to userId))
    .addTag("feed_sync")
    .build()

// Unique work — prevent duplicate jobs
workManager.enqueueUniqueWork(
    "feed_sync_$userId",
    ExistingWorkPolicy.KEEP,  // if already queued, don't re-enqueue
    syncRequest                  // REPLACE would cancel + re-enqueue
)

// Periodic work — minimum 15 minutes
val periodicSync = PeriodicWorkRequestBuilder<FeedSyncWorker>(1, TimeUnit.HOURS)
    .setConstraints(constraints)
    .setInitialDelay(10, TimeUnit.MINUTES) // don't run immediately at install
    .build()
workManager.enqueueUniquePeriodicWork(
    "periodic_feed_sync",
    ExistingPeriodicWorkPolicy.UPDATE, // update constraints if already exists
    periodicSync
)

Worker Implementation — Production Patterns

class FeedSyncWorker(
    private val context: Context,
    params: WorkerParameters,
    // Hilt injection via HiltWorker + WorkerFactory
    private val feedRepository: FeedRepository,
    private val analytics: Analytics
) : CoroutineWorker(context, params) {

    override suspend fun doWork(): Result {
        val userId = inputData.getString("userId")
            ?: return Result.failure() // can't retry if input is missing

        analytics.track("feed_sync.started", mapOf("attempt" to runAttemptCount))

        return try {
            feedRepository.sync(userId)
            analytics.track("feed_sync.success")
            Result.success()
        } catch (e: IOException) {
            analytics.track("feed_sync.failure", mapOf("reason" to e.message))
            if (runAttemptCount >= 3) {
                Result.failure( // stop retrying after 3 attempts
                    workDataOf("error" to e.message)
                )
            } else {
                Result.retry() // WorkManager applies backoff and retries
            }
        } catch (e: AuthException) {
            // Auth failure — retrying won't help, need user action
            notifyUserReauthRequired()
            Result.failure()
        }
    }

    override suspend fun getForegroundInfo(): ForegroundInfo {
        // Required for long-running workers — shows notification
        // WorkManager calls this automatically when needed on API 31+
        return createForegroundInfo(createSyncNotification())
    }
}

4. Doze Mode and App Standby — The Battery Reality

This is the Android background execution constraint that trips up most engineers at interview. If your design doesn't account for Doze, it won't work reliably on any modern Android device.

What Doze Does
When the device is stationary, screen off, and unplugged for a period, Doze kicks in. It defers: network access, wakelocks, JobScheduler, AlarmManager. Your background sync will NOT run during Doze windows. It waits for the next maintenance window (every few hours in deep Doze).
System StateEffect on Background WorkWorkaround
Doze ModeDefers network, jobs, alarms. Runs only during maintenance windows.FCM high-priority message — wakes device for one network window. Use for urgent notifications only.
App StandbyApp hasn't been used in hours/days → placed in a Standby Bucket. Lower buckets get fewer job execution windows per day.FCM keeps the app "active". Design sync to work with fewer executions per day.
Battery SaverAll background work deferred. Only foreground services run.Design gracefully — sync when battery saver is off. Show stale indicator to user.
Data SaverBackground network restricted. Foreground-only network access.Check ConnectivityManager.getRestrictBackgroundStatus(). Compress payloads for foreground requests.

App Standby Buckets — Know the Ladder

// Five buckets — system assigns your app based on usage patterns
// ACTIVE: user used the app recently → full background access
// WORKING_SET: used in past few days → some limits
// FREQUENT: used occasionally → more limits, deferred jobs
// RARE: rarely used → heavily restricted, jobs deferred by hours
// RESTRICTED: manually restricted by user → almost no background execution

// Design implication: your app CANNOT assume background sync runs reliably
// Strategy: use FCM to wake the device AND trigger sync immediately
// That way you're not depending on job scheduling — you're responding to a push

// Check current bucket (API 28+)
val usageStatsManager = context.getSystemService(UsageStatsManager::class.java)
val bucket = usageStatsManager.getAppStandbyBucket()
// Log this metric — know what bucket your users are in production
L6 Script — Background Sync Under Doze/Standby
"For a reliable background sync system I'd use FCM as the primary trigger rather than periodic jobs. Here's why: periodic WorkManager jobs are subject to Doze deferral and App Standby bucket restrictions. A user who hasn't opened the app in 3 days is in the RARE bucket — their periodic sync might run once every 4 hours at best. That's not acceptable for a messaging app.

Instead: the server sends a high-priority FCM message when there's new data. High-priority FCM is exempt from Doze — the device wakes up for a network window. The FCM receiver enqueues a WorkManager expedited worker that does the actual sync. This way, the delivery is guaranteed by the FCM infrastructure and the sync happens within seconds of the server knowing about new data, regardless of battery state.

The WorkManager periodic job still exists as a backup — but it's a safety net for the 'server didn't send FCM' case, not the primary path."

5. Foreground Services — When and How

// Foreground Service requirements (API 26+):
// 1. Must show a visible notification within 5 seconds of startForeground()
// 2. Notification cannot be dismissed by user until service stops
// 3. Required for: media playback, navigation, file downloads, ongoing calls
// 4. API 31+: must declare foreground service type in manifest

// Manifest declaration (API 34+ requires type permission)
// <service android:name=".DownloadService"
//          android:foregroundServiceType="dataSync" />
// <uses-permission android:name="android.permission.FOREGROUND_SERVICE_DATA_SYNC"/>

class FileDownloadService : Service() {
    private val scope = CoroutineScope(Dispatchers.IO + SupervisorJob())

    override fun onStartCommand(intent: Intent?, flags: Int, startId: Int): Int {
        val url = intent?.getStringExtra("url") ?: run { stopSelf(); return START_NOT_STICKY }

        // Must call startForeground() immediately
        startForeground(NOTIFICATION_ID, createDownloadNotification())

        scope.launch {
            downloadFile(url) { progress ->
                updateNotification(progress) // update progress in notification
            }
            stopSelf() // stop when done — don't linger
        }
        return START_REDELIVER_INTENT // if killed, restart with same intent
    }

    override fun onDestroy() {
        scope.cancel() // cancel all coroutines when service is destroyed
        super.onDestroy()
    }
}

// When NOT to use a Foreground Service:
// - Short sync operations → WorkManager expedited
// - Polling → replace with FCM push
// - "Background location" for tracking without user awareness → rejected on Play Store
API 34 Foreground Service Restrictions
Starting with API 34 (Android 14), foreground services require explicit type declarations and corresponding permissions. There are now 13 foreground service types: camera, connectedDevice, dataSync, health, location, mediaPlayback, mediaProjection, microphone, phoneCall, remoteMessaging, shortService, specialUse, systemExempted. Choosing the wrong type causes a crash at runtime. This is a breaking change for apps that used generic foreground services.

6. FCM Integration — Push Architecture

// FCM message types — choose correctly
// Notification message: system handles display, app gets onMessageReceived() in background
// Data message: app always gets onMessageReceived(), must handle display manually
// Combined: both — system shows notification, app can process data payload

// For sync triggers: use data-only message
// Server sends: { "data": { "type": "sync_feed", "userId": "123" } }
// No "notification" key — this is a silent push

class AppFirebaseMessagingService : FirebaseMessagingService() {

    override fun onMessageReceived(message: RemoteMessage) {
        when (message.data["type"]) {
            "sync_feed" -> enqueueFeedSync(message.data["userId"])
            "invalidate_cache" -> enqueueEviction(message.data["cacheKey"])
            "new_message" -> {
                showMessageNotification(message.data) // show notification
                enqueueSyncMessages()                 // and sync
            }
        }
    }

    override fun onNewToken(token: String) {
        // FCM token rotated — MUST send to server
        // Use WorkManager to ensure delivery even if this call fails
        workManager.enqueue(updateFcmTokenWork(token))
        // Don't update server inline — this may be called at startup
    }

    private fun enqueueFeedSync(userId: String?) {
        val work = OneTimeWorkRequestBuilder<FeedSyncWorker>()
            .setExpedited(OutOfQuotaPolicy.RUN_AS_NON_EXPEDITED_WORK_REQUEST)
            .setInputData(workDataOf("userId" to userId))
            .build()
        workManager.enqueueUniqueWork("fcm_sync", ExistingWorkPolicy.REPLACE, work)
        // REPLACE: if a sync is already running, cancel it and re-run with fresh trigger
        // This prevents stale syncs when multiple FCM messages arrive in quick succession
    }
}
L6 Script — FCM Token Refresh Handling
"FCM token rotation is a reliability concern that's easy to miss. If the token changes and we don't update the server, the user stops receiving push notifications silently. The onNewToken callback can be called at any time — including when there's no network. So I don't make the server call inline; I enqueue a WorkManager job with a CONNECTED constraint. This guarantees the token update reaches the server even if the device is offline at that moment. I'd also add a 'last FCM token update' timestamp to our analytics — if it's more than 30 days old for any user, we proactively re-register."

8. Battery Impact — The L6 Concern

Staff-Level Background Work Principle
Battery impact is a product concern, not just a technical one. At 100M users, saving 1% battery per user per day = 1M user-hours of device time. At Google scale, Play Store's Vitals battery section will flag your app if it's in the top 5% of battery drainers — that can affect discoverability. You need to design background work as if the OS is actively adversarial toward you.
PatternBattery ImpactBetter Alternative
Polling every 5 minutesHigh — wakes radio every 5 minFCM push — radio only wakes when there's new data
Wakelock for background workVery high — prevents CPU sleepWorkManager handles wakelock internally, releases when done
Background location pollingExtremely high — GPS + radioGeofencing API — OS triggers only on boundary cross
Frequent small network callsHigh — radio activation costBatch requests; radio stays active ~10s after each activation
WorkManager without constraintsMedium — can run on batteryAdd setRequiresBatteryNotLow(true) for non-urgent work

9. Worker Cancellation & Cleanup

WorkManager cancellation is often treated as "fire and forget" but L6 engineers design explicit cleanup — especially for workers that hold resources, write partial data, or trigger side effects that must be rolled back.

How WorkManager Cancels a Running Worker

// When WorkManager.cancelWorkById() or scope cancellation fires:
// 1. worker.onStopped() is called  → your signal to clean up
// 2. worker.isStopped returns true → polling check
// 3. If CoroutineWorker: coroutineContext is cancelled → CancellationException

class SyncWorker @AssistedInject constructor(
    @Assisted context: Context,
    @Assisted params: WorkerParameters,
    private val syncRepo: SyncRepository
) : CoroutineWorker(context, params) {

    override suspend fun doWork(): Result {
        return try {
            // Check cancellation before each expensive step
            ensureActive()
            val batch = syncRepo.fetchPendingBatch()

            ensureActive()
            syncRepo.uploadBatch(batch)

            // Mark completed only if not cancelled
            syncRepo.markBatchSynced(batch.id)
            Result.success()

        } catch (e: CancellationException) {
            // Worker was cancelled — clean up any partial state
            syncRepo.markBatchPending()   // reset to retry-able state
            throw e  // MUST rethrow — never swallow CancellationException
        } catch (e: IOException) {
            Result.retry()
        }
    }

    override fun onStopped() {
        // Called on main thread when WorkManager decides to stop this worker
        // Use for non-suspend cleanup: close files, release wakelocks, cancel downloads
        super.onStopped()
        // Note: for CoroutineWorker, coroutine is already cancelled by the time
        // onStopped() is called — so suspend functions won't run here.
        // Use withContext(NonCancellable) in doWork() for cleanup that MUST complete.
    }
}

Cleanup That Must Complete (NonCancellable)

override suspend fun doWork(): Result {
    val tempFile = createTempFile()
    return try {
        processAndUpload(tempFile)
        Result.success()
    } finally {
        // This block runs even if the coroutine is cancelled
        withContext(NonCancellable) {
            tempFile.delete()                    // always clean up temp files
            analytics.track("worker_finished")   // always log completion
        }
    }
}

Cancellation Scenarios — Each Behaves Differently

TriggerWhat happensYour responsibility
cancelWorkById()Worker coroutine cancelled, onStopped() calledClean up resources, reset partial state
App process killed by OSWorker stops mid-execution, WorkManager reschedulesMake doWork() idempotent — safe to re-run from start
Constraint no longer met (network lost)Worker stopped, rescheduled when constraint returnsSame as process kill — idempotency is the fix
cancelAllWork()All enqueued and running work cancelledGood for logout/account switch — ensure clean slate
Say this unprompted: "I design every worker to be idempotent — safe to run twice with the same input, because WorkManager can and does restart workers after process death. For partial state (half-uploaded batch), I use a pending/in-progress/done state machine in Room. On restart, the worker picks up from 'pending' — it won't re-upload what's already done."

10. What Gets You Rejected Here

Rejection Pattern — Ignoring Doze
You designed a background sync that "runs every hour" without mentioning that Doze will defer it for hours on an idle device, or that App Standby can limit it to once a day. The interviewer will probe this.
Rejection Pattern — "Use a Service"
Using a raw Service for background work on API 26+ is wrong — the OS will kill it after ~1 minute. Saying "background Service" without specifying Foreground Service (with notification) or WorkManager shows you're behind on modern Android.
Rejection Pattern — No Battery Budget
No mention of polling frequency, FCM vs polling trade-offs, or the cost of radio activations. At L6 this is table stakes.
Rejection Pattern — WorkManager for Everything
WorkManager for short in-process async operations, UI-bound work, or anything that needs to run immediately and interactively. WorkManager has dispatch latency — it's not a coroutine replacement.
