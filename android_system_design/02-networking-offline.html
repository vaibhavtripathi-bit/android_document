02 — Networking & Offline-First

Android L6 System Design · Caching · Pagination · Conflict Resolution · Sync · Retry

Why this domain is a rejection risk
Networking questions are where candidates go shallow fastest. You say "use Retrofit with OkHttp, cache with Room" and stop. At L6 the interviewer immediately probes: What's your eviction policy? How do you handle concurrent writes from the UI and a background sync? What happens when the cache is stale but the network is down? How do you paginate a feed that updates in real time? This guide teaches you to raise all of those before being asked.

1. The Offline-First Mental Model

Offline-first is not a nice-to-have — it's the correct default architecture for any consumer Android app. Even with perfect network, the Room-as-source-of-truth pattern gives you faster perceived performance because the UI renders from cache instantly while the network fetch happens in the background.

The Core Principle
The local database is always the single source of truth. The network is a data supplier, not the source of truth. The Repository's job is to keep the local DB fresh — the ViewModel only ever observes the DB, never the network directly.

The Read Flow — Stale-While-Revalidate

1
ViewModel subscribes to Room Flow — Room emits immediately with whatever is cached. UI renders. No blank screen.
2
Repository triggers network fetch — Retrofit call is made. This happens concurrently with the Room emission, not blocking it.
3
Network success → write to Room — Fresh data is written to Room. Room Flow automatically emits the updated data. ViewModel and UI update reactively. Zero manual wiring.
3
Network failure → stale data stays — Room already emitted stale data. User can see and interact with it. A UiMessage is emitted indicating the data may be stale. No crash, no blank screen.
class FeedRepository(
    private val feedDao: FeedDao,
    private val feedApi: FeedApi,
    private val freshnessPolicy: FreshnessPolicy
) {
    fun getFeed(): Flow<Result<List<FeedItem>>> = flow {
        // Step 1: emit cached data immediately
        val cached = feedDao.getAll().first()
        if (cached.isNotEmpty()) emit(Result.Success(cached.toDomain()))

        // Step 2: decide if network refresh is needed
        if (freshnessPolicy.isStale(cached)) {
            try {
                val fresh = feedApi.getFeed()
                feedDao.replaceAll(fresh.toEntity()) // write to DB
                // Step 3: DB change triggers Room Flow → UI updates automatically
            } catch (e: IOException) {
                // Network failed — stale data already shown, emit warning
                emit(Result.Error("Showing cached data", e))
            }
        }
    }.distinctUntilChanged()
        .conflate() // drop intermediate emissions if UI is busy
}
Probe: "What if the cache is empty AND network fails?"
This is the truly offline state. Your UiState must model this: FeedContent.Offline is different from FeedContent.Error — the UI shows a different message ("You're offline, connect to load your feed" vs "Something went wrong"). Always have an explicit offline state in your sealed class. This shows you've thought about the full state space.

2. Caching Strategy — The Full Decision Tree

Caching is not binary. At L6 you need to choose the right strategy for each type of data in your app and explain the reasoning.

StrategyHow it worksBest forPitfall
Cache-first (Stale-While-Revalidate) Return cache immediately, refresh in background Feed, articles, search results — tolerate slight staleness Cache may show outdated critical info (e.g., price)
Network-first Try network, fall back to cache on failure Financial data, inventory, real-time availability Blank screen on slow network; worse perceived performance
Cache-only Never hits network for this data User preferences, locally-computed data Stale forever if not invalidated explicitly
Network-only Always fetches fresh, never caches OTP, session tokens, one-time codes No offline support; burns battery/data
TTL-based expiry Cache is valid for N minutes, expired cache triggers refresh Configuration, feature flags, user profile Clock skew on device; user may get stale config if offline

Freshness Policy — Make It Explicit

// Don't bury cache logic inside the repository — make it a first-class type
class FreshnessPolicy(
    private val ttlMs: Long = TimeUnit.MINUTES.toMillis(5),
    private val clock: Clock = Clock.systemUTC() // injectable for tests!
) {
    fun isStale(items: List<CachedFeedItem>): Boolean {
        if (items.isEmpty()) return true
        val oldestRefresh = items.minOf { it.cachedAt }
        return clock.millis() - oldestRefresh > ttlMs
    }
}
// L6 signal: clock is injected → you can test freshness logic without
// Thread.sleep() or mocking System.currentTimeMillis()
L6 Script — Cache Strategy Justification
"For the feed, I'll use stale-while-revalidate because feed content is tolerant of 5-minute staleness — a post that was published 3 minutes ago showing 1 second later is acceptable. The immediate benefit is that the user sees content instantly even on slow networks.

For the user's unread notification count in the top bar, I'd flip to network-first with a short TTL, because that number is used for user decision-making — seeing a stale badge count erodes trust. For auth tokens, no caching — always network-only with refresh token handling.

The point is: you shouldn't have one caching strategy for the whole app. Each data type has a different staleness tolerance and a different consequence of being wrong."

3. Cache Eviction — The Part Candidates Always Forget

If you design a cache and don't design its eviction policy, you've designed a memory leak. This is a near-certain L6 probe.

Classic Rejection Moment
Interviewer: "You said you'd cache feed items in Room. How do you prevent the database from growing unbounded?" Candidate: "Good point, we could add some cleanup..."

If you wait to be asked this, you've already signaled L5. Raise cache eviction yourself, immediately after you say you'll cache data. 
// Eviction strategies — mention all three, choose based on requirements

// 1. Count-based: keep last N items
@Query("DELETE FROM feed_items WHERE id NOT IN (SELECT id FROM feed_items ORDER BY createdAt DESC LIMIT :maxItems)")
suspend fun evictOldItems(maxItems: Int = 200)

// 2. TTL-based: delete items older than N days
@Query("DELETE FROM feed_items WHERE cachedAt < :cutoff")
suspend fun evictExpiredItems(cutoff: Long)

// 3. LRU-based: track lastViewedAt, evict least-recently-viewed
// Useful for image caches — keep what the user has actively viewed
@Query("DELETE FROM feed_items WHERE lastViewedAt < :cutoff ORDER BY lastViewedAt ASC LIMIT :count")
suspend fun evictLruItems(cutoff: Long, count: Int)

// When to evict: run as a WorkManager periodic task, not on the main path
class CacheEvictionWorker(ctx: Context, params: WorkerParameters) : CoroutineWorker(ctx, params) {
    override suspend fun doWork(): Result {
        feedDao.evictOldItems(maxItems = 500)
        feedDao.evictExpiredItems(cutoff = System.currentTimeMillis() - TimeUnit.DAYS.toMillis(7))
        return Result.success()
    }
}
Raise This Proactively
"One thing I want to proactively address — cache eviction. I'm going to store feed items in Room, which means without a cleanup policy, the DB will grow unbounded. For a feed I'd use a hybrid approach: keep at most 500 items, and also delete anything older than 7 days. The eviction runs as a daily WorkManager periodic task, not on the hot path. I'd also set a DB size alert in our monitoring — if average Room file size exceeds 20MB per user, that tells me eviction isn't keeping up and we need to tighten the policy."

4. Pagination — Cursor vs Offset (The L6 Depth Question)

Every FAANG feed uses cursor-based pagination. You need to know exactly why offset fails at scale and be able to articulate it clearly.

Why Offset Fails at Scale

Offset Pagination — Breaks at Scale
// GET /feed?offset=20&limit=10
// Page 1: items 0-9 at T=0
// New item posted at T=1
// Page 2: items 10-19 at T=1
// → item 9 appears twice!
// (it shifted to position 10)

// Also: OFFSET N requires the DB
// to scan and skip N rows —
// O(N) at the DB level
Cursor Pagination — Stable
// GET /feed?after=cursor_xyz&limit=10
// cursor = opaque token encoding
// last-seen item's ID + timestamp
// New items don't affect cursor
// → no duplicates, no gaps

// DB uses indexed seek — O(log N)
// Works even with deletions

Cursor Design — What the Cursor Actually Contains

// A cursor is NOT just an ID. It's an opaque token on the wire
// but internally encodes enough to resume stable pagination
data class PaginationCursor(
    val lastItemId: String,
    val lastItemCreatedAt: Long, // for feeds sorted by time
    val rankingScore: Double?   // for ranked feeds — ML score at snapshot time
)
// Serialized as base64(json) → sent to server → server decodes and uses
// "WHERE (createdAt, id) < (cursor.createdAt, cursor.lastItemId)"
// This is a keyset pagination query — uses composite index on (createdAt, id)

Android Paging 3 — Integration Pattern

// PagingSource bridges Paging 3 and your data source
class FeedPagingSource(
    private val api: FeedApi,
    private val dao: FeedDao
) : PagingSource<String, FeedItem>() { // Key=cursor String

    override suspend fun load(params: LoadParams<String>): LoadResult<String, FeedItem> {
        return try {
            val cursor = params.key // null = first page
            val response = api.getFeed(cursor = cursor, limit = params.loadSize)
            dao.insertAll(response.items.toEntity()) // cache as we paginate
            LoadResult.Page(
                data = response.items.toDomain(),
                prevKey = null, // feeds don't paginate backwards
                nextKey = response.nextCursor // null = end of feed
            )
        } catch (e: IOException) {
            LoadResult.Error(e) // Paging 3 handles retry automatically
        }
    }

    override fun getRefreshKey(state: PagingState<String, FeedItem>): String? {
        // On config change: resume from the anchor page, not page 1
        return state.anchorPosition?.let { state.closestItemToPosition(it)?.cursor }
    }
}

// RemoteMediator: use when you want DB-backed paging with network fallback
// Paging 3 queries Room first, fetches network when Room runs out of pages
// This is the offline-first paging pattern
class FeedRemoteMediator(
    private val db: AppDatabase,
    private val api: FeedApi
) : RemoteMediator<Int, FeedEntity>() {
    override suspend fun load(loadType: LoadType, state: PagingState<Int, FeedEntity>): MediatorResult {
        val cursor = when (loadType) {
            LoadType.REFRESH -> null // fresh start
            LoadType.PREPEND -> return MediatorResult.Success(endOfPaginationReached = true)
            LoadType.APPEND -> db.remoteKeyDao().getNextCursor()
        }
        return try {
            val response = api.getFeed(cursor)
            db.withTransaction { // atomic: insert items + update cursor together
                if (loadType == LoadType.REFRESH) db.feedDao().clearAll()
                db.feedDao().insertAll(response.items.toEntity())
                db.remoteKeyDao().saveNextCursor(response.nextCursor)
            }
            MediatorResult.Success(endOfPaginationReached = response.nextCursor == null)
        } catch (e: IOException) { MediatorResult.Error(e) }
    }
}
L6 Pagination Script
"I'll use cursor-based pagination for three reasons. First, the feed is a live-updating list — offset-based pagination produces duplicates when new items are inserted above, because all positions shift. Second, keyset queries on the server side are O(log N) via a composite index, whereas OFFSET N forces a full scan of N rows. At 50M items in the table, that matters. Third, cursors are stable under deletions — if a post is removed, the cursor still points to the right position.

On Android I'll use Paging 3 with RemoteMediator. RemoteMediator gives me the offline-first paging pattern: Room is the primary source, the network loads more data into Room when the user scrolls to the end of what's cached. The cursor is stored in a separate RemoteKeys table and updated atomically with the feed items in a Room transaction — so I can never have a state where items are saved but the next-page cursor is stale."

5. Network Layer Design — Retry, Auth, and Request Queuing

Retry Policy — Don't Just Say "Exponential Backoff"

// OkHttp Interceptor for retry with exponential backoff + jitter
class RetryInterceptor(
    private val maxRetries: Int = 3
) : Interceptor {
    override fun intercept(chain: Interceptor.Chain): Response {
        var attempt = 0
        var lastException: IOException? = null
        while (attempt < maxRetries) {
            try {
                val response = chain.proceed(chain.request())
                if (response.isSuccessful || response.code == 400 || response.code == 404) {
                    return response // don't retry client errors
                }
                if (response.code == 429) {
                    // Rate limited — respect Retry-After header
                    val retryAfter = response.header("Retry-After")?.toLongOrNull() ?: 60
                    Thread.sleep(retryAfter * 1000)
                }
            } catch (e: IOException) { lastException = e }
            val backoff = (2.0.pow(attempt) * 1000 + Random.nextLong(1000)).toLong()
            Thread.sleep(backoff) // exponential backoff + jitter
            attempt++
        }
        throw lastException ?: IOException("Max retries exceeded")
    }
}
Retry Rules — Know These Cold
  • Retry on: 5xx (server errors), 429 (rate limited — with backoff), IOException (network failure)
  • Never retry on: 400 (bad request — retrying won't fix it), 401 (unauthorized — need token refresh first), 404 (not found), 403 (forbidden)
  • Jitter is required: Without jitter, all clients retry at the same exponential interval → thundering herd that amplifies the outage. Add random(0, 1000ms) to each backoff.

Auth Token Refresh — The Concurrent Refresh Problem

// Problem: 5 concurrent requests all get 401, all try to refresh token
// → 5 simultaneous refresh calls → race condition, token invalidation

// Solution: use a Mutex to ensure only one refresh at a time
class AuthInterceptor(
    private val tokenStore: TokenStore,
    private val authApi: AuthApi
) : Authenticator { // use Authenticator, not Interceptor, for 401 handling

    private val refreshMutex = Mutex()

    override fun authenticate(route: Route?, response: Response): Request? {
        val newToken = runBlocking {
            refreshMutex.withLock {
                // Re-check: maybe another coroutine already refreshed
                val currentToken = tokenStore.getAccessToken()
                if (currentToken != response.request.header("Authorization")) {
                    return@withLock currentToken // someone else already refreshed
                }
                try {
                    val refreshed = authApi.refresh(tokenStore.getRefreshToken())
                    tokenStore.save(refreshed)
                    refreshed.accessToken
                } catch (e: Exception) {
                    tokenStore.clear() // refresh failed → force logout
                    null
                }
            }
        }
        return newToken?.let { response.request.newBuilder().header("Authorization", "Bearer $it").build() }
    }
}
L6 Signal — Raise the Concurrent Refresh Problem
"For authentication, I'll implement the token refresh using OkHttp's Authenticator interface rather than an Interceptor — that's important because Authenticator is designed for 401 handling with built-in single-retry semantics. But I also need to handle the concurrent refresh problem: if we have 5 in-flight requests and all of them get 401 simultaneously, they'll all try to call the refresh endpoint at once. That creates a race where multiple valid refresh tokens are burned simultaneously and the backend invalidates all of them.

The fix is a Mutex in the Authenticator. Only one coroutine can perform the refresh at a time. When the mutex is acquired, the first thing you do is re-read the current token and compare it to the one in the failed request — if they're different, another coroutine already refreshed, so just use the new token without another network call. This is the double-checked locking pattern adapted for coroutines."

6. Optimistic Updates — When and How

Optimistic updates give instant UI feedback before the server confirms an action. They're expected in a modern app and an L6 interview topic.

// Like a post — show heart immediately, revert on failure
fun likePost(postId: String) {
    viewModelScope.launch {
        // 1. Optimistic update — immediate UI change
        feedDao.setLiked(postId, liked = true)
        _state.update { it.copy(likeCount = it.likeCount + 1) }

        // 2. Network call
        val result = runCatching { postApi.likePost(postId) }

        // 3. On failure: revert
        result.onFailure {
            feedDao.setLiked(postId, liked = false)
            _state.update { it.copy(likeCount = it.likeCount - 1) }
            emitEffect(FeedEffect.ShowError("Couldn't like post. Try again."))
        }
    }
}

// Harder case: optimistic create (new message in chat)
// Give the message a local UUID immediately
// Show it with a "sending" status indicator
// On success: update status to "sent", replace local ID with server ID
// On failure: show "failed" with retry button
L6 Consideration — Idempotency
For optimistic creates (especially messages), you must pass a client-generated idempotency key to the server. If the network request succeeds but the response is lost (device went offline), a retry should not create a duplicate message. The server deduplicates by idempotency key. This is a cross-team concern — your Android design should specify this requirement on the API contract.

7. Write Queueing — Offline Writes

Users expect to be able to compose and send while offline. The write must be queued and delivered when connectivity returns.

// WorkManager is the right tool for guaranteed delivery
// It survives app kills, device reboots, and network outages
fun sendMessageOfflineSafe(message: DraftMessage) {
    // 1. Write to local DB immediately — user sees it instantly
    messageDao.insert(message.toEntity(status = MessageStatus.PENDING))

    // 2. Enqueue a WorkManager job to deliver it
    val work = OneTimeWorkRequestBuilder<SendMessageWorker>()
        .setConstraints(Constraints(requiredNetworkType = NetworkType.CONNECTED))
        .setBackoffCriteria(BackoffPolicy.EXPONENTIAL, 15, TimeUnit.SECONDS)
        .setInputData(workDataOf("messageId" to message.localId))
        .addTag("send_message_${message.localId}") // for deduplication
        .build()

    workManager.enqueueUniqueWork(
        "send_${message.localId}",
        ExistingWorkPolicy.KEEP, // don't re-enqueue if already queued
        work
    )
}

class SendMessageWorker(...) : CoroutineWorker(...) {
    override suspend fun doWork(): Result {
        val messageId = inputData.getString("messageId") ?: return Result.failure()
        val message = messageDao.getById(messageId) ?: return Result.failure()
        return try {
            val serverMessage = messageApi.send(message.toRequest())
            messageDao.updateStatus(messageId, MessageStatus.SENT, serverMessage.id)
            Result.success()
        } catch (e: IOException) {
            if (runAttemptCount > 3) {
                messageDao.updateStatus(messageId, MessageStatus.FAILED)
                Result.failure() // show retry button to user
            } else Result.retry()
        }
    }
}

8. Conflict Resolution — The Hard Problem

When offline writes collide with server state, you need a strategy. This is an L6-only topic — most candidates never reach it.

StrategyHow it worksUse whenDownside
Last-write-wins Server timestamp determines winner; later write overwrites earlier Profile updates, preferences, non-collaborative data Can silently discard valid user edits
Server-wins Server state always prevails; client resyncs on conflict Financial transactions, inventory, authoritative data User loses optimistic edits silently
Client-wins Local change is applied even after server state is fetched Almost never correct for shared data Can corrupt shared state
Merge (CRDT) Both changes are merged deterministically (e.g., union of sets) Collaborative editing, tags, bookmarks, likes Complex to implement; not all data types support it
Notify + User Resolution Show user both versions, ask them to pick Document editing, complex form data Disruptive UX; only appropriate for high-stakes data
L6 Script — Conflict Resolution
"Conflict resolution is a requirement I'd raise with the product team, not decide unilaterally. The right answer depends on the data type. For user profile edits — bio, avatar — I'd use last-write-wins with a server timestamp, and I'd make sure we include the version number in every write so the server can detect a conflict and return a 409. On 409, the client re-fetches and re-applies the local diff if possible. For messages in a chat, messages are append-only — there's no conflict, just ordering. For collaborative document editing, you'd need CRDTs or operational transforms — that's a separate design. The key is that I'd design the API contract with conflict semantics in mind from day one, not retrofit it later."

9. Observability for Networking

At L6, you need to instrument your network layer. Not just crash reporting — business-meaningful metrics.

// OkHttp EventListener — attach to every request lifecycle
class NetworkMetricsListener(
    private val metrics: MetricsReporter
) : EventListener() {
    private var callStart: Long = 0
    private var dnsStart: Long = 0

    override fun callStart(call: Call) { callStart = SystemClock.elapsedRealtime() }
    override fun dnsStart(call: Call, domainName: String) { dnsStart = SystemClock.elapsedRealtime() }
    override fun dnsEnd(call: Call, domainName: String, inetAddressList: List<InetAddress>) {
        metrics.recordHistogram("network.dns_ms", SystemClock.elapsedRealtime() - dnsStart)
    }
    override fun callEnd(call: Call) {
        metrics.recordHistogram("network.total_ms", SystemClock.elapsedRealtime() - callStart,
            tag("endpoint", call.request().url.encodedPath))
    }
    override fun callFailed(call: Call, ioe: IOException) {
        metrics.increment("network.failure", tag("reason", ioe::class.simpleName))
    }
}
L6 — Metrics That Matter
Track these per endpoint, segmented by network type (WiFi vs Cellular vs Offline):
  • p50/p95/p99 latency — mean is meaningless; p99 is what your worst-off users experience
  • Cache hit rate — if it drops suddenly, your eviction policy is too aggressive
  • Retry rate — if it spikes, your server is degraded before alerts fire
  • Offline fallback rate — % of sessions where user saw stale data; business visibility into offline usage

10. HTTP/2 vs HTTP/3 (QUIC) — What L6 Knows

HTTP/1.1HTTP/2HTTP/3 (QUIC)
TransportTCPTCPUDP + QUIC
MultiplexingNo — one request per connectionYes — multiple streams per TCP connectionYes — truly independent streams
Head-of-line blockingAt request levelAt TCP level — one lost packet stalls ALL streamsEliminated — QUIC recovers per-stream independently
Connection setupTCP(1RTT) + TLS(1-2RTT)Same as HTTP/1.10-RTT on reconnect (QUIC + TLS 1.3 combined)
Network switching (WiFi→LTE)Connection drops, full reconnectConnection dropsConnection migration — same session survives IP change
OkHttp supportDefaultAutomatic when server supports itVia Cronet or OkHttp Cronet interceptor

Why This Matters on Mobile Specifically

Mobile networks have higher packet loss and frequent switches between WiFi and cellular. HTTP/2's TCP head-of-line blocking means a single lost packet stalls all multiplexed streams until the OS retransmits it. On a feed loading 6 thumbnails in parallel over a congested network, this is measurable latency. HTTP/3 recovers each stream independently.

// Enable HTTP/3 via Cronet
val cronetEngine = CronetEngine.Builder(context)
    .enableQuic(true)
    .enableHttp2(true)
    .addQuicHint("api.example.com", 443, 443)  // tell Cronet this host supports QUIC
    .build()

val client = OkHttpClient.Builder()
    .callFactory(CronetInterceptor.newBuilder(cronetEngine).build())
    .build()

// Verify in production: track which protocol is actually negotiated
val loggingEventListener = object : EventListener() {
    override fun connectionAcquired(call: Call, connection: Connection) {
        analytics.track("http_protocol", mapOf("protocol" to connection.protocol().toString()))
        // Expect: h3, h2, or http/1.1
    }
}
Say this unprompted: "HTTP/2 is free with OkHttp — no code change needed if the server supports it. For HTTP/3 I'd add Cronet specifically for features where mobile network quality matters most: video streaming, large image grid loads. The connection migration property is particularly valuable for ride-sharing or navigation apps where users move between networks frequently."

11. What Gets You Rejected Here

Rejection Pattern — "Room + Retrofit" and Stop
You described the components without addressing: eviction, freshness policy, concurrent refresh, pagination correctness, conflict resolution, or observability. That's a component list, not a system design.
Rejection Pattern — Only Happy Path
You never mentioned what happens when the network is down, when a write fails mid-flight, when a 429 is returned, or when the cache is empty. At L6 the unhappy paths are as important as the happy path.
Rejection Pattern — No Metrics
You designed a sophisticated caching system but no way to know if it's working in production. Observability is not optional at L6 — it's how you demonstrate you've shipped and operated real systems.
