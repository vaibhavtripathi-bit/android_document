07 — Security & Privacy

Android L6 System Design · KeyStore · Certificate Pinning · Permissions · Attestation · GDPR/CCPA · ProGuard

Why security questions trip up L6 candidates
Security topics feel like a checklist — "use HTTPS, use KeyStore, pin certificates." But at L6 the question is: when does each mechanism fail, what's your recovery plan, and how do you balance security hardening with maintainability? Over-secured designs (that break on certificate rotation) and under-secured designs (plaintext tokens) are both L5 signals.

KeyStore — How It Actually Works

// Android KeyStore — the key never leaves the hardware (on devices with TEE/StrongBox)
// Use case: generate a key, use it to encrypt data — key itself is not accessible

// Generate an AES-256-GCM key in KeyStore
val keyGenerator = KeyGenerator.getInstance(KeyProperties.KEY_ALGORITHM_AES, "AndroidKeyStore")
keyGenerator.init(
    KeyGenParameterSpec.Builder("my_key_alias", KeyProperties.PURPOSE_ENCRYPT or KeyProperties.PURPOSE_DECRYPT)
        .setBlockModes(KeyProperties.BLOCK_MODE_GCM)
        .setEncryptionPaddings(KeyProperties.ENCRYPTION_PADDING_NONE)
        .setKeySize(256)
        .setUserAuthenticationRequired(false) // true = biometric required before use
        .setStrongBoxBacked(true)             // use dedicated security chip if available
        .build()
)
val key = keyGenerator.generateKey()

// Encrypt data using the KeyStore key
val cipher = Cipher.getInstance("AES/GCM/NoPadding")
cipher.init(Cipher.ENCRYPT_MODE, key)
val iv = cipher.iv // save this alongside ciphertext — needed for decryption
val ciphertext = cipher.doFinal(plaintext.toByteArray())

// Decrypt
val decryptCipher = Cipher.getInstance("AES/GCM/NoPadding")
decryptCipher.init(Cipher.DECRYPT_MODE, key, GCMParameterSpec(128, savedIv))
val plaintext = String(decryptCipher.doFinal(ciphertext))

// StrongBox: dedicated security chip (Pixel 3+, Samsung S10+)
// Falls back to TEE (Trusted Execution Environment) if StrongBox unavailable
// Always wrap setStrongBoxBacked(true) in try/catch StrongBoxUnavailableException
try {
    /* setStrongBoxBacked(true) */
} catch (e: StrongBoxUnavailableException) {
    // Fall back to TEE-backed key — still secure, just not in dedicated chip
}
L6 Script — KeyStore Depth
"For credential storage I use EncryptedSharedPreferences for tokens — it wraps KeyStore transparently and is the right level of abstraction for most apps. If you need biometric-gated access — for example, in a banking app where a transaction signing key should only be usable after the user authenticates — then I'd work directly with the KeyStore API and pass a KeyGenParameterSpec with setUserAuthenticationRequired(true). When that flag is set, the key is invalidated if biometric enrollment changes — for example if the user adds a new fingerprint — so the app detects the invalidated key and prompts for re-authentication. That prevents an attacker from adding their fingerprint and reusing the key."

2. Certificate Pinning — The Right Way

Certificate Pinning is a Double-Edged Sword
Done wrong, certificate pinning is worse than not having it at all. If you pin the leaf certificate and it expires, or if you pin to the wrong intermediary and it rotates, you've just bricked every installation of your app until users update. The design must include rotation. 
// Preferred: Network Security Config (declarative, no code)
// res/xml/network_security_config.xml
<network-security-config>
    <domain-config cleartextTrafficPermitted="false">
        <domain includeSubdomains="true">api.yourcompany.com</domain>
        <pin-set expiration="2026-01-01">
            <!-- Primary pin: SHA-256 of public key (not cert) -->
            <pin digest="SHA-256">BASE64_OF_PUBLIC_KEY_HASH</pin>
            <!-- Backup pin: intermediate CA — survives leaf rotation -->
            <pin digest="SHA-256">BACKUP_INTERMEDIATE_CA_HASH</pin>
        </pin-set>
    </domain-config>
</network-security-config>

// Key decisions:
// Pin the PUBLIC KEY hash, not the certificate — public key survives cert renewal
// Always have a backup pin (intermediate CA) — if primary rotates, backup still works
// Set expiration date — forces users to update before pins expire

// OkHttp programmatic pinning (for more control)
val certificatePinner = CertificatePinner.Builder()
    .add("api.yourcompany.com", "sha256/PRIMARY_KEY_HASH")
    .add("api.yourcompany.com", "sha256/BACKUP_KEY_HASH") // backup
    .build()

val client = OkHttpClient.Builder()
    .certificatePinner(certificatePinner)
    .build()

Safe Certificate Rotation — The L6 Requirement

// Problem: you need to rotate the server certificate but millions of apps
// are pinned to the old key. Rotation = all those apps get SSL errors.

// Safe rotation strategy — must be planned months in advance:
// Phase 1: Add new key as a backup pin in current app version (still ship old key too)
// Phase 2: Wait for old app versions to naturally diminish (e.g., 95% on new version)
// Phase 3: Rotate server certificate to new key — old app's backup pin now matches
// Phase 4: New app release removes old key, keeps only new key

// Remote pin update — for emergency rotation
// Fetch pinned certs from a /config endpoint at startup
// Cache locally with short TTL
// If local pins fail, fall back to fetched pins
// Risk: the /config endpoint itself must be pinned (circular problem)
// Solution: pin /config to a long-lived CA cert, pin /api to the leaf cert
L6 Script — Certificate Pinning
"I'd implement certificate pinning via the Network Security Config XML — it's declarative, auditable, and handles the trust chain correctly. I'd pin the public key hash rather than the certificate, so that certificate renewal doesn't break existing apps. I'd always include at least two pins: the primary leaf key and the intermediate CA as a backup.

The rotation story is more important than the pinning setup itself. Certificate rotation without a rotation plan has caused major outages — including for some large FAANG apps. My plan: add the new key as a backup pin in a release that ships at least 3 months before the planned rotation. Once analytics show 95%+ of users are on the new version, rotate the server cert. Then clean up the old pin in the next release. With the expiration date set in the config, users who haven't updated get a forced-upgrade prompt before pins expire — not a silent SSL failure."

3. Network Security — Transport Layer

// Enforce HTTPS everywhere — no cleartext in production
// res/xml/network_security_config.xml
<network-security-config>
    <base-config cleartextTrafficPermitted="false" />
    <debug-overrides>
        <!-- Allow cleartext only in debug for proxy tools like Charles -->
        <trust-anchors>
            <certificates src="user" /> <!-- trust user-installed CAs for Charles/Proxyman -->
        </trust-anchors>
    </debug-overrides>
</network-security-config>

// TLS 1.3 — enforce minimum version
// OkHttp uses TLS 1.3 by default on API 29+
// For older devices, TLS 1.2 is the minimum acceptable
val connectionSpec = ConnectionSpec.Builder(ConnectionSpec.MODERN_TLS)
    .tlsVersions(TlsVersion.TLS_1_3, TlsVersion.TLS_1_2) // 1.3 preferred, 1.2 fallback
    .cipherSuites(
        CipherSuite.TLS_AES_256_GCM_SHA384,
        CipherSuite.TLS_CHACHA20_POLY1305_SHA256,
        CipherSuite.TLS_AES_128_GCM_SHA256
    )
    .build()

4. Permission Model — Principle of Least Privilege

// Dangerous permissions: request only when needed, explain why
// Never request at launch — only when user initiates the feature

// Permission request pattern with rationale
val requestPermission = registerForActivityResult(
    ActivityResultContracts.RequestPermission()
) { isGranted ->
    if (isGranted) startCamera()
    else showPermissionDeniedExplanation()
}

fun onCaptureButtonClick() {
    when {
        ContextCompat.checkSelfPermission(this, CAMERA) == PERMISSION_GRANTED -> {
            startCamera()
        }
        shouldShowRequestPermissionRationale(CAMERA) -> {
            // User denied before — show WHY we need camera before asking again
            showRationaleDialog { requestPermission.launch(CAMERA) }
        }
        else -> {
            requestPermission.launch(CAMERA)
        }
    }
}

// Use granular permissions where available (API 33+)
// READ_MEDIA_IMAGES instead of READ_EXTERNAL_STORAGE
// ACCESS_FINE_LOCATION vs ACCESS_COARSE_LOCATION — use coarse unless you need exact
// Background location (ACCESS_BACKGROUND_LOCATION) requires separate prompt since API 30
// and Play Store review — only request if absolutely necessary
L6 Permission Principle
At L6 you'd raise the permission design as a product conversation: "What's the minimum permission surface for this feature? Can we achieve 80% of the value with ACCESS_COARSE_LOCATION instead of ACCESS_FINE_LOCATION? Fewer dangerous permissions → higher install conversion and lower user churn."

// Does: shrinking (remove unused code) + obfuscation (rename) + optimization

// -keepnames class * implements java.io.Serializable  // Serializable

6. Root Detection and App Attestation

// Why: banking, healthcare, DRM-protected content apps
// Goal: verify the app is running on a genuine unmodified device

// Google Play Integrity API (replacement for SafetyNet Attestation, deprecated 2024)
val integrityManager = IntegrityManagerFactory.create(context)
val nonce = generateCryptographicNonce() // server-generated, prevents replay

val request = StandardIntegrityManager.StandardIntegrityTokenRequest.builder()
    .setRequestHash(nonce)
    .build()

// Token is a JWS signed by Google — send to your server for verification
// Server decodes the verdict:
// MEETS_DEVICE_INTEGRITY: hardware-backed key, certified by Google
// MEETS_BASIC_INTEGRITY: device passes CTS but isn't Google-certified
// fails: rooted/modified device, emulator, unofficial firmware

// Decision at server: based on verdict, allow/deny/step-up auth
// Don't block entirely on integrity failure — may affect legitimate devices
// Better: require additional auth factor (OTP, biometric) for sensitive operations
Root Detection Trade-off
Root detection has false positives. Security researchers, developers, and accessibility tool users often run on modified systems. A strict "block on root" policy would also block some legitimate users. The L6 design is: use integrity verdict to determine risk level, require step-up authentication (not block) for high-risk operations. Never use root detection as the sole security control — treat it as one signal in a defense-in-depth strategy.

7. Privacy — Data Minimization and GDPR/CCPA

L6 Privacy Principle
At Staff level, privacy is an architectural concern, not a compliance checkbox. You design the data model to collect only what you need, retain it only as long as required, and make deletion (right to be forgotten) technically tractable. If deletion is not in your architecture from day one, retrofitting it costs 10x more.

// Deletion architecture — must be designed in from the start
class UserDataDeletionService @Inject constructor(
    private val userDao: UserDao,
    private val feedDao: FeedDao,
    private val encryptedPrefs: EncryptedSharedPreferences,
    private val fileStorage: FileStorage,
    private val analyticsApi: AnalyticsApi
) {
    suspend fun deleteUserData(userId: String): DeletionResult {
        return try {
            // 1. Delete from server (trigger server-side deletion pipeline)
            analyticsApi.requestDeletion(userId)

            // 2. Delete local DB in transaction
            db.withTransaction {
                userDao.deleteUser(userId)
                feedDao.deleteFeedForUser(userId)
                // ... all tables with user data
            }

            // 3. Clear encrypted storage
            encryptedPrefs.edit().clear().apply()

            // 4. Delete cached files
            fileStorage.deleteUserFiles(userId)

            DeletionResult.Success
        } catch (e: Exception) {
            // Log failure — partial deletion is a compliance risk
            // Retry via WorkManager — deletion must complete eventually
            DeletionResult.Failed(e.message)
        }
    }
}

8. WebView Security

// WebView is a common security surface — configure defensively
webView.settings.apply {
    javaScriptEnabled = true     // only enable if you need it; default should be false
    allowFileAccess = false      // prevent file:// access from web content
    allowContentAccess = false   // prevent content:// access
    allowFileAccessFromFileURLs = false
    allowUniversalAccessFromFileURLs = false
    // These two are false by default — never enable them
}

// JavaScript bridge — huge attack surface
// Only add @JavascriptInterface methods you specifically need
// Validate every parameter from JavaScript — treat it as untrusted user input
class AppBridge {
    @JavascriptInterface
    fun showToast(message: String) {
        // Validate input — message could be malicious
        if (message.length > 200) return // reject suspiciously long strings
        Toast.makeText(context, message.take(100), Toast.LENGTH_SHORT).show()
    }
}

// Only load trusted URLs in WebViews that have JS bridges
// Validate URL before loading: check scheme (https only) and host (allowlist)
fun loadUrl(url: String) {
    val uri = Uri.parse(url)
    require(uri.scheme == "https" && uri.host in allowedHosts) { "Untrusted URL" }
    webView.loadUrl(url)
}

9. APK Signing Key Rotation

Signing key rotation is one of the highest-stakes operations in Android — done wrong, it permanently blocks existing users from updating the app. L6 engineers know the rotation strategy before they're asked, not after an incident.

Why Rotation Is Dangerous

Android verifies APK signatures on update — the new APK must be signed with the same key as the installed version. If you lose the original key or rotate without the proper lineage, existing users on the old signature cannot install updates. They must uninstall and reinstall, losing all local data.

Google Play Signing — The Safe Path

// Google Play App Signing (recommended since 2017, mandatory for new apps since 2021):
// - Google holds the upload key AND the app signing key
// - You sign APKs with your upload key; Play re-signs with the app signing key for distribution
// - If your upload key is compromised: request an upload key reset from Play Console
//   → Google re-signs new APKs with the same app signing key
//   → Zero user impact; existing installs continue to update normally

// KEY INSIGHT: with Play Signing, losing your local signing key is survivable.
// Without Play Signing, losing your key means you can never update the app again.

// Enrollment in Play Console:
// Setup → App Integrity → App signing → Opt in to Google Play signing

Key Rotation with APK Signature Scheme v3

# APK Signature Scheme v3 (API 28+) supports key rotation with proof of lineage
# The lineage proves: "old key trusted new key"

# Step 1: Generate new signing key
keytool -genkey -v -keystore new-release-key.jks \
    -alias new-key -keyalg RSA -keysize 4096 -validity 10000

# Step 2: Create rotation lineage (one-time operation linking old key → new key)
apksigner rotate \
    --out rotation-lineage.bin \
    --old-signer --ks old-release-key.jks --ks-key-alias old-key \
    --new-signer --ks new-release-key.jks --ks-key-alias new-key

# Step 3: Sign APK with both old lineage and new key
apksigner sign \
    --ks new-release-key.jks \
    --ks-key-alias new-key \
    --lineage rotation-lineage.bin \
    --min-sdk-version 28 \
    --out app-release-signed.apk \
    app-release-unsigned.apk

# Result:
# API 28+ devices: accept new key (verifies lineage)
# API 27-  devices: need v1/v2 signature still signed with old key (use --v1-signing-enabled)

The Four-Phase Safe Rotation Strategy

PhaseActionDurationRollback possible?
1. Dual-signShip APKs signed with BOTH old and new key2–4 weeksYes — revert to old key only
2. MonitorTrack update success rate by API level; verify no signature failures1–2 weeksYes
3. New-key onlyDrop old key from signing, ship with new key onlyAfter 99%+ users on dual-signed versionNo — point of no return
4. Revoke oldRevoke old key in Play Console; destroy key materialAfter monitoring confirms stabilityNo
Say this unprompted: "Key rotation is a point-of-no-return operation. I'd never rotate without first enrolling in Google Play App Signing — that way if something goes wrong, Play can re-sign with the app signing key and existing users are unaffected. For rotation itself, I'd use APK Signature Scheme v3 lineage on API 28+ and dual-sign for at least two full release cycles before dropping the old key."

10. What Gets You Rejected Here

Rejection Pattern — "Just use HTTPS"
HTTPS without certificate pinning is vulnerable to MITM via trusted CA compromise. At L6 for an app handling sensitive data, HTTPS is the baseline, not the complete solution.
Rejection Pattern — Pinning Leaf Certificate Only
Pinning the leaf certificate means every certificate renewal breaks existing apps. Always pin the public key hash, and always have a backup pin at the intermediate CA level.
Rejection Pattern — No Deletion Architecture
You designed a data model but never mentioned how you'd handle deletion requests. In 2025+, GDPR and CCPA compliance is not optional — and retrofitting deletion onto an existing schema is expensive.
Rejection Pattern — Binary Root Blocking
"If rooted, deny access" — this blocks legitimate users, researchers, and accessibility tool users, and any skilled attacker will bypass it anyway. The L6 answer is risk-graduated response, not a binary block.
