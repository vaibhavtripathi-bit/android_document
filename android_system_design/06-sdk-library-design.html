06 — SDK & Library Design

Android L6 System Design · API Surface · Initialization · Versioning · Analytics SDK · Crash SDK · Binary Size

Why this is the highest-signal L6 topic
SDK design is almost exclusively an L6 question because it requires thinking beyond your own app: API ergonomics, backwards compatibility, consumer impact, binary size footprint, initialization side effects — all of which affect engineers you'll never meet. If the interviewer asks you to design an analytics or crash SDK, they're directly testing staff-level thinking.

1. The SDK Design Mindset Shift

Designing an SDK is fundamentally different from designing an app. The key shift:

DimensionApp DesignSDK Design
AudienceEnd usersOther engineers — their DX is your UX
API changesYour team controls the migrationBreaking changes affect every app that integrates you
Error handlingShow error UI, crash is recoverableSDK crashes propagate to host app — must never crash the host
DependenciesUse whatever you needEvery dependency you add bloats every consumer's APK
InitializationYou control App.onCreate()Must not assume timing; must be safe to call multiple times
ThreadingAssumed coroutine contextMust work with any threading model; document every threading guarantee
PrivacyHandle your own dataYou process other apps' user data — GDPR/CCPA compliance is your problem
The Golden Rule of SDK Design
Your SDK must never be the reason the host app crashes. Wrap every internal operation in try/catch. Degrade gracefully — if analytics fail to initialize, the host app still runs perfectly. If your SDK throws an exception that propagates to the host, you've broken every app that uses you. Log it, swallow it, and alert your team.

2. Initialization Patterns

The ContentProvider Trick — Auto-Initialization

// Firebase, WorkManager, LeakCanary — all use this pattern
// A ContentProvider is initialized by Android before Application.onCreate()
// This lets your SDK initialize itself without requiring the app developer to call init()

class AnalyticsSdkInitializer : ContentProvider() {

    override fun onCreate(): Boolean {
        val context = context ?: return false
        // Initialize with defaults — no API key needed at this point
        AnalyticsSdk.autoInit(context)
        return true
    }

    // Manifest: authorities must be unique per app — use applicationId
    // <provider
    //   android:name=".AnalyticsSdkInitializer"
    //   android:authorities="${applicationId}.analytics_sdk_init"
    //   android:exported="false" />

    // All other methods: return null/0/-1 — this is not a real ContentProvider
    override fun query(...) = null
    override fun getType(...) = null
    override fun insert(...) = null
    override fun delete(...) = 0
    override fun update(...) = 0
}

// Modern replacement: Jetpack App Startup library
// Consolidates multiple ContentProvider initializers into one
// Reduces startup time when multiple SDKs each use their own ContentProvider
class AnalyticsInitializer : Initializer<AnalyticsSdk> {
    override fun create(context: Context): AnalyticsSdk {
        return AnalyticsSdk.init(context)
    }
    override fun dependencies(): List<Class<out Initializer<*>>> = emptyList()
}
ContentProvider Startup Cost
Each ContentProvider adds ~2-4ms to app cold start on a mid-range device. At Google Play scale, if 50 SDKs each use a ContentProvider initializer, that's 100-200ms of startup time stolen from the host app. This is why App Startup exists — it consolidates all SDK ContentProviders into one. At L6, you'd design your SDK to support both patterns and document the startup overhead.

Manual Init vs Auto-Init — The Design Decision

Auto-Init (ContentProvider)
  • Zero-config for the consumer
  • Less chance of "forgot to initialize" bugs
  • Used by Firebase, LeakCanary, WorkManager

Best for: SDKs with sensible defaults that don't need an API key at init time

Manual Init
  • Consumer must call Sdk.init(apiKey)
  • Can defer initialization to after user consent (GDPR)
  • Consumer controls timing precisely

Best for: SDKs that need API keys, or where consent-gated init matters

L6 Script — Init Strategy
"For initialization, I'd offer both patterns. Default behavior: ContentProvider auto-init using App Startup library — the SDK is ready before Application.onCreate() with sensible defaults. For apps that need consent-gated initialization (GDPR — don't start tracking until the user accepts), they'd disable auto-init via a manifest metadata flag and call AnalyticsSdk.init(context, config) explicitly after consent is granted.

Critical: my init method must be idempotent — calling it twice produces no side effects. And it must not crash if called on a background thread. I'd protect the init block with a @GuardedBy lock and a boolean flag, and I'd wrap the entire init in try/catch so a bad API key or missing resource doesn't crash the host app."

3. API Surface Design — The Contract You Can Never Break

Every public method you add to your SDK is a promise you're making to every app that integrates it. Breaking that promise in a future version requires every consumer to update their code.

API Design Principles

// Principle 1: Minimal surface area — only expose what's necessary
// If it doesn't need to be public, make it internal
internal class EventQueue     // consumers don't need this
internal class BatchUploader  // implementation detail
class AnalyticsSdk {          // only this is public
    fun track(event: AnalyticsEvent) { ... }
    fun identify(userId: String) { ... }
    fun flush() { ... }
}

// Principle 2: Builder pattern for configuration — extensible without breaking changes
data class AnalyticsConfig private constructor(
    val apiKey: String,
    val batchSize: Int,
    val flushIntervalMs: Long,
    val endpoint: String,
    val isDebug: Boolean
) {
    class Builder(val apiKey: String) {
        private var batchSize: Int = 50
        private var flushIntervalMs: Long = 30_000
        private var endpoint: String = "https://api.analytics.com/v1/events"
        private var isDebug: Boolean = false

        fun batchSize(size: Int) = apply { batchSize = size }
        fun flushIntervalMs(ms: Long) = apply { flushIntervalMs = ms }
        fun endpoint(url: String) = apply { endpoint = url }
        fun debug() = apply { isDebug = true }
        fun build() = AnalyticsConfig(apiKey, batchSize, flushIntervalMs, endpoint, isDebug)
    }
}
// Adding a new config option = add to Builder with a default = no breaking change
// If you used a constructor: adding a param = breaking change for every consumer

// Principle 3: Type-safe event model
sealed class AnalyticsEvent {
    abstract val name: String
    abstract val properties: Map<String, Any>

    data class Custom(override val name: String, override val properties: Map<String, Any> = emptyMap()) : AnalyticsEvent()
    data class ScreenView(val screenName: String) : AnalyticsEvent() {
        override val name = "screen_view"
        override val properties = mapOf("screen_name" to screenName)
    }
}
// Sealed class: adding a new subclass is non-breaking for the SDK producer
// But consumer's when() expressions will get a warning — document this

Versioning and Backwards Compatibility

// Semantic versioning: MAJOR.MINOR.PATCH
// MAJOR: breaking changes — must increment for any API removal or signature change
// MINOR: new features, backwards compatible — consumers can opt in
// PATCH: bug fixes, no API changes

// Deprecation lifecycle — never remove without a deprecation window
// Step 1: Mark @Deprecated with replacement
@Deprecated(
    message = "Use track(AnalyticsEvent) instead",
    replaceWith = ReplaceWith("track(AnalyticsEvent.Custom(eventName, properties))"),
    level = DeprecationLevel.WARNING  // WARNING → ERROR → remove over 3 major versions
)
fun trackEvent(eventName: String, properties: Map<String, Any>) {
    track(AnalyticsEvent.Custom(eventName, properties))
}

// Binary compatibility: use the Kotlin Binary Compatibility Validator plugin
// It generates an .api file that records the public API surface
// CI fails if the public API changes without a corresponding version bump
// gradle task: ./gradlew apiCheck

// Java interop — if your SDK may be used from Java:
// @JvmStatic for companion object functions
// @JvmOverloads for default parameters
// Don't use extension functions in public API — not callable from Java
L6 Script — API Versioning
"For versioning I'd use semantic versioning strictly and enforce it in CI. The Kotlin Binary Compatibility Validator generates an .api file on every build — if the public API changes without a corresponding MAJOR version bump, the CI job fails. This makes breaking changes impossible to ship accidentally.

For deprecations: I'd never remove a method in the same major version it was deprecated in. The lifecycle is: deprecate with WARNING in v2.3, escalate to ERROR in v3.0, remove in v4.0. Each step gives consumers time to migrate. I'd include the migration guide in the deprecation annotation's replaceWith field so IDEs show the fix automatically."

4. Analytics SDK Design — Full Architecture

The Core Design Decisions

// Event pipeline: Track → Queue → Batch → Upload
class AnalyticsSdk private constructor(
    private val context: Context,
    private val config: AnalyticsConfig
) {
    private val eventQueue = EventQueue(maxSize = 1000) // in-memory queue
    private val eventStore = EventStore(context)           // Room persistence
    private val uploader = EventUploader(config)
    private val scope = CoroutineScope(Dispatchers.IO + SupervisorJob())

    fun track(event: AnalyticsEvent) {
        try {
            val enriched = event.enrich(buildContext()) // add sessionId, timestamp, deviceInfo
            eventQueue.offer(enriched)        // non-blocking, immediate
            if (eventQueue.size >= config.batchSize) {
                scope.launch { flush() }       // flush asynchronously when batch full
            }
        } catch (e: Exception) {
            // NEVER propagate — SDK crash must not affect host app
            internalLog("Analytics track failed", e)
        }
    }

    private suspend fun flush() {
        val batch = eventQueue.drainTo(config.batchSize)
        if (batch.isEmpty()) return

        // Persist to DB first — survive app kill before upload
        eventStore.save(batch)

        try {
            uploader.upload(batch)
            eventStore.delete(batch) // delete from DB only after confirmed upload
        } catch (e: IOException) {
            // Network failed — events remain in DB, retry next flush
            // Don't re-add to in-memory queue — DB is now source of truth
        }
    }

    private fun setupPeriodicFlush() {
        scope.launch {
            while (isActive) {
                delay(config.flushIntervalMs)
                flush()
            }
        }
        // Also flush on app background (ProcessLifecycleOwner)
        ProcessLifecycleOwner.get().lifecycle.addObserver(
            LifecycleEventObserver { _, event ->
                if (event == Lifecycle.Event.ON_STOP) scope.launch { flush() }
            }
        )
    }
}

Batching Strategy — The Core Trade-off

TriggerWhenTrade-off
Count-basedWhen N events queued (e.g., 50)Timely delivery; frequent if user is active; risk of large batch on slow network
Time-basedEvery N seconds (e.g., 30s)Predictable; slow if events are important; events can be lost between flushes
App backgroundWhen app moves to backgroundAlways — captures the last session events before process kill
Network availableWhen network connects after offline periodDelivers queued events from DB when connectivity returns
L6 Script — Batching and Persistence
"The analytics pipeline has a two-level durability guarantee. In-memory queue for performance — track() is non-blocking and completes in microseconds. When the queue hits the batch size threshold (50 events by default), I flush asynchronously. The flush persists the batch to Room first, then uploads. The critical ordering here: persist to DB before attempting upload, delete from DB only after confirmed server response. If the process is killed mid-upload, the events are still in the DB and will be uploaded on the next flush.

I always flush on app background via ProcessLifecycleOwner.ON_STOP. If I don't, the last minute of user activity can be lost if the process is killed while in background. On reconnection from offline, a WorkManager worker triggers a flush to clear the DB backlog. The backlog has a max size — if it exceeds 5,000 events, I start evicting the oldest. I'd rather lose old events than blow up the user's storage."

5. Crash Reporting SDK — Design

// Crash reporting has two capture modes
// 1. Uncaught exceptions — Thread.UncaughtExceptionHandler
// 2. ANRs — separate mechanism via signal or watchdog

class CrashReporter private constructor(val context: Context) {

    private val previousHandler = Thread.getDefaultUncaughtExceptionHandler()

    fun install() {
        Thread.setDefaultUncaughtExceptionHandler { thread, throwable ->
            try {
                captureCrash(thread, throwable)  // write to disk synchronously
            } finally {
                // MUST call the previous handler — system needs to handle the crash too
                previousHandler?.uncaughtException(thread, throwable)
            }
        }
    }

    private fun captureCrash(thread: Thread, throwable: Throwable) {
        // This runs AFTER the crash — must be synchronous and fast
        // Cannot use coroutines — the app is dying
        // Must write to file, not Room (DB may be in inconsistent state)
        val report = CrashReport(
            timestamp = System.currentTimeMillis(),
            thread = thread.name,
            stackTrace = throwable.stackTraceToString(),
            breadcrumbs = breadcrumbBuffer.snapshot(),
            deviceInfo = collectDeviceInfo(),
            memoryInfo = collectMemoryInfo()
        )
        // Write to private file — upload on NEXT app launch
        writeCrashReportToFile(report)
    }
}

// ANR Detection — not an exception, harder to detect
// Method 1: Watchdog thread — sends message to main Looper,
// if not processed in N seconds → ANR suspected
class AnrWatchdog(
    private val timeout: Long = 5_000L
) : Thread("anr-watchdog") {
    private val handler = Handler(Looper.getMainLooper())
    private val tick = AtomicLong(0)

    override fun run() {
        while (!isInterrupted()) {
            val expected = tick.incrementAndGet()
            handler.post { tick.set(expected) } // post to main thread
            Thread.sleep(timeout)
            if (tick.get() != expected) {
                // Main thread didn't process our message → ANR
                val mainStackTrace = Looper.getMainLooper().thread.stackTrace
                reportAnr(mainStackTrace)
            }
        }
    }
}

// Breadcrumbs — contextual trail before the crash
class BreadcrumbBuffer(maxSize: Int = 100) {
    private val buffer = ArrayDeque<Breadcrumb>(maxSize)

    fun add(crumb: Breadcrumb) {
        synchronized(this) {
            if (buffer.size >= maxSize) buffer.removeFirst() // ring buffer
            buffer.addLast(crumb)
        }
    }
    fun snapshot(): List<Breadcrumb> = synchronized(this) { buffer.toList() }
}
// Auto-breadcrumbs: activity lifecycle, network requests, user actions
// Use ActivityLifecycleCallbacks registered in Application — zero consumer setup

6. Binary Size and Dependency Management

L6 Constraint — Binary Size
For a consumer-facing SDK, every KB you add shows up in every app's APK. A 2MB SDK added to 100 apps = 200MB added to the Play Store. Play Store research shows every 10MB added reduces install conversion by ~1%. This is a product concern, not just engineering. 
// Dependency hygiene — the L6 rules

// 1. Use 'compileOnly' for optional integrations
// If your SDK can optionally log to Timber, don't force Timber on every consumer
dependencies {
    compileOnly "com.jakewharton.timber:timber:5.0.1"
    // Consumer can choose to add Timber — your SDK works without it
}

// 2. Avoid transitive dependency conflicts
// Your SDK pulls in OkHttp 4.9. Consumer uses OkHttp 4.12.
// Gradle resolves to 4.12, but your SDK was tested with 4.9.
// Solution: minimize your dependency surface, or use dependency exclusions

// 3. Check with dependency-guard plugin
// Records your exact dependency tree — CI fails if it changes unexpectedly
// Prevents accidental dependency additions by team members

// 4. Use ProGuard consumer rules
// proguard-rules.pro — consumed by apps that use R8/ProGuard
// Keep your public API classes from being obfuscated
// -keep class com.yourcompany.sdk.AnalyticsSdk { *; }
// -keep class com.yourcompany.sdk.AnalyticsEvent { *; }

7. Custom Lint Rules — Enforcing SDK Contracts

Custom Lint rules let you encode SDK usage contracts as compile-time errors rather than documentation that nobody reads. For SDK authors at L6, this is a first-class tool for preventing misuse at scale across many consuming teams.

When to Write a Custom Lint Rule

  • SDK must be initialized before use → detect calls before initialize()
  • Deprecated API with a safe replacement → auto-suggest migration
  • Thread-safety constraint → detect calls from wrong thread
  • Resource leak pattern → detect opened resources never closed
  • PII logging prohibition → detect strings containing emails/tokens in log calls
// build.gradle.kts for your :lint module
plugins { id("java-library") }
dependencies {
    compileOnly("com.android.tools.lint:lint-api:31.5.0")
    compileOnly("com.android.tools.lint:lint-checks:31.5.0")
}

// Example: Detect Analytics.track() called before Analytics.initialize()
class UninitializedSdkDetector : Detector(), Detector.UastScanner {

    override fun getApplicableMethodNames() = listOf("track", "flush", "setUserId")

    override fun visitMethodCall(context: JavaContext, node: UCallExpression, method: PsiMethod) {
        val containingClass = method.containingClass?.qualifiedName ?: return
        if (containingClass != "com.example.sdk.Analytics") return

        // Check if initialize() was called in this file/class
        val enclosingClass = node.getContainingUClass() ?: return
        val hasInit = enclosingClass.methods.any { m ->
            m.uastBody?.sourcePsi?.text?.contains("Analytics.initialize") == true
        }

        if (!hasInit) {
            context.report(
                issue = ISSUE,
                scope = node,
                location = context.getLocation(node),
                message = "Analytics.${method.name}() called before Analytics.initialize(). " +
                          "Call Analytics.initialize(context, config) in Application.onCreate() first."
            )
        }
    }

    companion object {
        val ISSUE = Issue.create(
            id = "AnalyticsNotInitialized",
            briefDescription = "Analytics SDK used before initialization",
            explanation = "The Analytics SDK must be initialized via Analytics.initialize() before any tracking calls.",
            category = Category.CORRECTNESS,
            priority = 9,
            severity = Severity.ERROR,       // ERROR = blocks build by default
            implementation = Implementation(
                UninitializedSdkDetector::class.java,
                Scope.JAVA_FILE_SCOPE
            )
        )
    }
}

// Register in IssueRegistry
class SdkIssueRegistry : IssueRegistry() {
    override val issues = listOf(UninitializedSdkDetector.ISSUE)
    override val api = CURRENT_API
}

// In :sdk/build.gradle.kts — ship lint rules with the AAR:
android {
    lint { abortOnError = true }
}
dependencies {
    lintPublish(project(":sdk-lint"))  // Lint rules bundled into AAR
}

Testing Lint Rules

class UninitializedSdkDetectorTest {
    @Test
    fun `flags track() call without initialize()`() {
        lint().files(

8. What Gets You Rejected Here

Rejection Pattern — SDK That Can Crash the Host
No try/catch around the public API. An uncaught exception in track() propagates to the consumer's call stack and crashes their app. That's a P0 for every consumer — your SDK would be removed immediately.
Rejection Pattern — Breaking Changes Without Versioning
You designed an API surface but didn't mention semantic versioning, deprecation lifecycle, or the Binary Compatibility Validator. At L6, you own the contract your API represents.
Rejection Pattern — No Persistence for Critical Events
Analytics events are discarded if the app is killed mid-session. Crash reports are lost if written to memory only. These are not edge cases — they're common scenarios with real business impact.
Rejection Pattern — No Size/Dependency Awareness
You added Retrofit, OkHttp, Gson, Room, and Coroutines as SDK dependencies without noting the size and dependency conflict implications for consumers.
