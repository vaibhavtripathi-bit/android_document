Android Domain Tips

These are the proactive signals that separate L6 Android answers from L5 Android answers. For each topic, the tip tells you what to say unprompted — before the interviewer asks.

Architecture

Tip AND-01 · Always say WHY you chose MVI or MVVM for this specific screen

Don't just name the pattern. Say: "I'm choosing MVI here because this screen has multiple independent data streams (user, settings, network status) that could create inconsistent UI state if managed as separate LiveData/StateFlow objects. A single sealed UiState eliminates that class of bug." Naming the bug the pattern prevents is what L6 sounds like.

"I'll use MVI here — single sealed UiState class — because this screen merges data from three sources. With MVVM and separate streams, I risk showing a loading spinner from stream A while stream B has already emitted an error. MVI makes that state combination impossible at the type level."
Tip AND-02 · Name the dependency rule enforcement mechanism

Don't just say "Clean Architecture." Say how you enforce the dependency rule at build time. "The domain module has no Android dependencies — if a developer accidentally imports an Android class into domain, the Gradle build fails. That's enforced by the module having no Android dependency in its build.gradle."

"I enforce the dependency rule at build time, not through code review. The :domain module doesn't depend on the Android Gradle plugin — it's a pure Kotlin module. Any accidental import of an Android class causes a compile error immediately."
Tip AND-03 · Raise the one-time event problem before the interviewer asks

Whenever you discuss navigation events, toasts, or snackbars, proactively say: "One-time events are tricky with StateFlow because rotation causes re-collection and re-triggers the event. I handle these with a SharedFlow with replay=0, or a Channel — the key property I need is consume-once semantics." This is a known L6 question trap.

"For navigation events, I'll use a SharedFlow with replay=0 rather than StateFlow. The reason: StateFlow replays the last value on re-subscription — so rotating the device would re-trigger the navigation and send the user to a duplicate screen. SharedFlow with replay=0 fires once and is gone."

Coroutines & Concurrency

Tip AND-04 · Always inject dispatchers — say why

Every time you write coroutine code, say "I inject the dispatcher rather than hardcoding Dispatchers.IO." If they ask why: "Because in tests I replace it with TestCoroutineDispatcher — no real threads, instant execution, deterministic ordering."

"I inject the dispatcher as a constructor parameter rather than hardcoding Dispatchers.IO. This means in tests I pass UnconfinedTestDispatcher — no thread spawning, instant execution, no flakiness."
Tip AND-05 · Raise CancellationException handling when discussing coroutines

Proactively say: "One subtle bug is catching CancellationException accidentally — if you write catch(e: Exception) in a coroutine, you swallow cancellation and the coroutine never terminates cleanly. I always catch specific exceptions and rethrow CancellationException explicitly."

"A common production bug: catch(e: Exception) in a coroutine catches CancellationException, which breaks structured concurrency — the parent scope never gets the cancellation signal and can hang. I always catch specific exception types and rethrow CancellationException."
Tip AND-06 · Explain WhileSubscribed(5000) for StateFlow

When you mention stateIn(), always specify the started policy and explain the number. "WhileSubscribed(5000) keeps the upstream flow alive for 5 seconds after the last subscriber disappears. That 5-second window absorbs Activity recreation during rotation — so the upstream network call isn't cancelled and restarted just because the user rotated their phone."

"I use WhileSubscribed(5000) — not Eagerly. The 5-second window handles configuration changes: when the user rotates, the Activity is destroyed and recreated within ~500ms. With WhileSubscribed, the upstream Flow stays alive through that gap, so there's no extra network call."

Networking & Offline

Tip AND-07 · State the thundering herd problem every time you mention retry

The moment you mention retry logic, add: "I add jitter to the backoff delay — a random offset to desynchronize retries. Without jitter, all clients that got a 503 retry simultaneously after exactly 2 seconds, creating a thundering herd that can take the server down again." This is a precise production insight that interviewers love.

"I always add jitter — a random offset of ±25% of the backoff delay. Without it, if 100,000 users all hit a 503 at the same moment, they all retry simultaneously after exactly 2 seconds. That spike can take the server down again even after it recovered. Jitter desynchronizes the wave."
Tip AND-08 · Raise cursor-based pagination over offset before being asked

Whenever pagination comes up: "I'd use cursor-based rather than offset pagination. With offset, if a new post is inserted while the user is mid-scroll, their page 3 repeats an item that shifted into page 2. Cursor-based anchors to a specific item ID so the position is stable regardless of insertions."

"I'd use cursor-based pagination here. Offset pagination has a silent bug in live feeds: an insertion at position 5 while the user is reading page 3 means they see a duplicate item on page 4. Cursor-based eliminates this by anchoring to a real item ID."
Tip AND-09 · Name idempotency keys for the offline write queue

When you describe offline write queueing: "Each queued write gets a client-generated idempotency key. If WorkManager retries the upload after a network error, the server recognizes the key and doesn't create a duplicate record. Without idempotency keys, every retry risks a double post or double purchase."

"Each queued write gets a UUID idempotency key generated at creation time. On retry, the key is re-sent in the request header. The server responds with success if it's already processed that key — no duplicate record. This is critical for financial or irreversible operations."

Local Storage

Tip AND-10 · Name the 3-class separation for Room

Don't just say "I'd use Room." Say: "I separate the data model into three classes: the network DTO which mirrors the API response, the Room Entity which mirrors the database schema, and the Domain model which the UI and use cases actually consume. The mapper transforms between them at layer boundaries, so a backend schema change never leaks into the UI."

"I keep three model classes: NetworkDto mirrors the API contract, RoomEntity mirrors the DB schema, and DomainModel is what the rest of the app uses. This means a backend field rename only requires a one-line mapper change — nothing else in the codebase changes."
Tip AND-11 · Raise migration safety unprompted

Whenever you mention Room schema, proactively add the migration safety rules: "I never use fallbackToDestructiveMigration in production — it wipes user data silently on schema version mismatch. I always write an explicit migration, export the schema to a JSON file in source control, and test it with MigrationTestHelper in CI."

"Three migration rules I never break: one, export the schema to JSON and commit it — so any schema change is visible in the diff. Two, never fallbackToDestructiveMigration — it silently deletes user data. Three, test every migration with MigrationTestHelper before shipping — create a V1 database, apply the migration, verify data integrity."

Performance

Tip AND-12 · Lead with Baseline Profiles for startup optimization

Whenever startup performance comes up, the first thing you say should be Baseline Profiles: "The highest-leverage startup optimization today is Baseline Profiles — they tell ART which code paths to AOT-compile before first launch. Play delivers the compiled profile alongside the APK. We've seen 30–40% cold start improvement from this before touching any other optimization."

"The first thing I'd reach for on startup is Baseline Profiles. They're generated from a Macrobenchmark that runs the critical user journeys, and the compiled profile is delivered by Play alongside the APK. ART pre-compiles those paths. 30–40% cold start improvement, zero code changes to the app itself."
Tip AND-13 · Always name the measurement tool before the optimization

Never propose a performance optimization without first naming how you'd measure it. "Before I optimize, I'd establish a baseline with Macrobenchmark running the cold start journey 10 times on a real device in release mode. That gives me a reproducible number to compare against after the change."

"Before making any change, I'd run Macrobenchmark to establish a baseline — 10 iterations of the cold start journey on a Pixel 7 in release mode, no ART profile. That's my reproducible baseline. Any optimization gets re-benchmarked against it in CI to prevent regressions slipping in later."

Background Work & Lifecycle

Tip AND-14 · Explain WHY WorkManager over plain coroutines

"I'd use WorkManager rather than a plain coroutine because: one, WorkManager persists the work request to its own Room database before executing, so the job survives process death. Two, it delegates to JobScheduler which can reschedule after a reboot. Three, it handles constraints like network availability — the coroutine approach requires you to re-implement all of that." Name all three reasons, not just "WorkManager is more reliable."

"WorkManager here, not a coroutine. Three specific reasons: it writes the work request to its own Room DB before executing — so process death doesn't lose the job. JobScheduler can reschedule it after a reboot. And it supports constraints — I can say 'only run when unmetered network is available' with one line."
Tip AND-15 · Raise OEM background killing proactively for reliability-critical features

For any feature that depends on background execution: "I need to flag that Samsung and Xiaomi aggressively kill background processes beyond AOSP behavior. For reliability-critical features like message delivery or health monitoring, I'd use a Foreground Service with a user-visible notification — OEMs can't kill those under normal conditions."

"One production reality I'd flag: Samsung One UI and Xiaomi MIUI kill background processes more aggressively than stock Android. For a feature that must reliably execute in the background, I'd use a Foreground Service with an ongoing notification — OEMs treat those with the same priority as phone calls."

Security & Privacy

Tip AND-16 · Name the backup pin when discussing certificate pinning

"I always configure a backup pin alongside the primary pin. Pinning to only the leaf certificate means a certificate rotation requires a forced app update — which you may not control if users are slow to update. I pin to the intermediate CA and keep a backup pin to handle rotation without an emergency release."

"When I set up cert pinning, I pin to the intermediate CA's public key hash — not the leaf cert. And I always add a backup pin. If I only pin the leaf cert and it rotates, every user on the old app gets blocked until they update. The backup pin gives me a transition window."
Tip AND-17 · Raise GDPR deletion architecture for any feature touching user data

Whenever user data storage comes up: "Since this stores user data, I need to think about the deletion story. GDPR requires deletion within 30 days of a user request. I'd implement this as an atomic WorkManager chain: delete from Room, delete from remote, delete from encrypted file storage, clear SharedPreferences keys — all in a single worker with a single retry policy."

"Since this persists user data, I need to account for deletion. I'd implement account deletion as a WorkManager chain: delete Room rows, call the deletion API, clear any encrypted SharedPreferences keys, delete any cached files. The whole chain is atomic — if any step fails, WorkManager retries from the failed step."

Testing

Tip AND-18 · Always mention fakes over mocks for owned interfaces

"For my own interfaces — repositories, use cases, data sources — I write hand-crafted fakes rather than Mockito mocks. Fakes have real in-memory state and emit real Flow values when that state changes. This lets me test state machine transitions across multiple emissions, which mock-based tests can't do cleanly."

"I use fakes for interfaces I own, not mocks. A FakeUserRepository backed by a MutableStateFlow lets me test that the ViewModel correctly transitions from Loading → Success when data arrives, and Loading → Error when the fake is configured to fail. Mock-based tests would require verifying call sequences, which is fragile."
Tip AND-19 · Name Turbine when discussing Flow testing

When Flow testing comes up: "I use Turbine for testing Flows — it provides a clean API for asserting individual emissions in order without the boilerplate of manually launching collection coroutines. It handles timing, cancellation, and buffering correctly. awaitItem(), awaitComplete(), awaitError()."

"For Flow testing I use Turbine. Without it, you end up with a launched coroutine collecting to a list, which has timing dependencies. Turbine gives me awaitItem() which suspends until the next emission and fails with a clear message if it doesn't arrive — deterministic and readable."

The Quick-Fire Responses — Practice Until Automatic

If the interviewer asks...Lead with this
"How would you handle offline?""Room as single source of truth. UI reads only from Room, never from network response directly. Write queue with idempotency keys for offline writes."
"How would you handle pagination?""Cursor-based — anchored to item ID, not offset position. Offset has a silent duplicate bug on live feeds under insertion."
"How would you improve startup time?""Baseline Profiles first — 30–40% cold start improvement before any code change. Then audit ContentProviders with App Startup library."
"How would you handle token refresh?""Mutex with double-checked locking. One coroutine does the refresh; others suspend on the Mutex. After acquiring, check again if token is still expired — another coroutine may have already refreshed it."
"How would you test a ViewModel?""Inject TestCoroutineDispatcher, use fake repository, assert Flow emissions with Turbine. No Android framework, runs in milliseconds on JVM."
"What's your approach to background sync?""WorkManager — persists to Room before executing, survives process death, delegates to JobScheduler, handles network constraints natively."
"How would you ship this safely?""Kill switch for instant disable (<60s, no release). Gradual rollout: 1%→5%→25%→100% with automated metric gates. Rollback = toggle the kill switch."
