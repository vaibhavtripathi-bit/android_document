Algorithm & Coding Interview Tips

Google's coding rounds for L6 go beyond correctness — they score how you solve the problem: thinking process, edge case awareness, complexity analysis, and clean code. Practice saying these things out loud, not just writing the code.

The Mandatory 5-Step Process

Follow this every time, without exception. Skipping any step is a signal the interviewer will note.

1Understand the problem — ask clarifying questions. Input types, constraints (size, sign, null?), expected output, edge cases. Never write code before you've confirmed you understand the input/output contract.
"Before I write anything — can the input array be empty? Can values be negative? Should I modify in-place or return a new result? And is there a space complexity constraint I should target?"
2Work through an example by hand. Write a small example on the board/editor, trace through it manually. This confirms your understanding AND reveals edge cases before you commit to an approach.
"Let me trace through a simple example first to make sure I'm understanding correctly. Input: [3, 1, 4, 1, 5]. I expect the output to be... [trace]. Okay, that matches my understanding."
3State a brute force solution — then optimize. Always name the O(n²) or naive approach first. Then say: "That works but is O(n²) time. Can I do better?" This shows you know there's an optimization path and sets up the conversation correctly.
"The brute force is two nested loops — O(n²) time, O(1) space. That's correct but slow for large inputs. I can improve this to O(n) with a hash map — let me think through that approach."
4State your approach before coding. Describe the algorithm in plain language before writing a single line. 2–3 sentences is enough. This lets the interviewer redirect if you're headed somewhere wrong — saving you 10 minutes of coding the wrong solution.
"My approach: I'll do a single pass through the array using a hash map to store each value and its index. For each element, I check if target minus the current value is already in the map. If yes, I've found the pair. If no, I store the current value. This is O(n) time and O(n) space."
5After coding — test with edge cases out loud. Run through: empty input, single element, all same values, negative values, overflow conditions. State each one you're testing. This is L6-level thoroughness.
"Let me test a few edge cases: empty array — my function returns null, that's correct. Single element — loop runs once, nothing in the map, returns null. Duplicate values — hash map stores the most recent index, which is what I want. Looks good."

How to Handle Being Stuck

Tip AC-01 · Narrate while you're thinking

The worst thing you can do is go quiet for 3 minutes. Even if you're stuck, keep talking: describe the constraints, what approaches you've considered and rejected, what property of the data you'd need to exploit. Interviewers can give hints to candidates who are reasoning out loud — they can't help silent candidates.

"I know I need better than O(n²). The key property of this problem is that I'm looking for a pair, which suggests I could trade space for time using a hash map. Let me think about whether sorting first would help... if I sort, I can use two pointers. Let me compare the trade-offs."
Tip AC-02 · Ask for a hint without giving up

If you've been stuck for 3+ minutes and genuinely can't move forward, it's better to ask for a hint than to stay frozen. But frame it with what you've already tried.

"I've tried the hash map approach and a sorting approach — both have issues with [specific problem]. I'm not seeing the path forward. Can you give me a hint about which data structure property I should be thinking about?"
Tip AC-03 · When you realize mid-coding your approach is wrong

Stop, explain what you realized, and course-correct. Don't try to patch a broken approach — it's obvious and it wastes time. Pivoting cleanly is a positive signal.

"Wait — I realize this approach has a problem: it doesn't handle [edge case] correctly. Rather than patching it, let me step back and use [alternative approach] which handles this cleanly. Here's why..."

Complexity Analysis — Always Do This Proactively

Tip AC-04 · State complexity before being asked

After writing the solution, immediately say the time and space complexity. Don't wait to be asked. L6 candidates analyze their own code; L4 candidates wait for the question.

"Time complexity is O(n log n) — the sort dominates. Space complexity is O(1) since I'm sorting in place and only using a constant number of extra variables. If the interviewer needs O(n) time, I'd trade the O(n) space for a hash map instead."

Code Quality Signals

Tip AC-06 · Name your variables meaningfully

Single-letter variables (except i, j for loop indices) signal L4. Use names that describe intent: left/right for two pointers, charFreq for a frequency map, windowSum for a sliding window sum. Interviewers reading your code can follow it without asking you to explain.

Tip AC-07 · Write helper functions instead of long inline blocks

If a block of logic is getting long, extract it into a named helper function even if you're in a coding interview. This shows clean code instincts. "I'm going to extract this into an isValid() helper to keep the main function readable."

Tip AC-08 · Handle null/empty at the top, not deep in the logic

Early returns for edge cases (empty input, null checks) at the start of the function make the core logic cleaner. This is a strong signal of production code instincts.

fun longestSubstring(s: String, k: Int): Int {
    if (s.isEmpty() || k <= 0) return 0   // ← Guard clauses first
    if (k == 1) return s.length            // ← Simple case shortcut
    // ... main logic
}
Tip AC-09 · Kotlin-specific: use idiomatic patterns

In a Kotlin interview, use idiomatic Kotlin rather than Java-style code. Interviewers writing Kotlin daily will notice. Use getOrDefault, groupBy, takeWhile, partition, sortedBy, destructuring, let/also/run where they clarify intent. But don't over-chain — readability wins.

// Java-style (signals unfamiliarity with Kotlin)
val map = HashMap<Char, Int>()
for (c in s) {
    if (map.containsKey(c)) map[c] = map[c]!! + 1
    else map[c] = 1
}

// Idiomatic Kotlin
val freq = s.groupingBy { it }.eachCount()
// or
val freq = buildMap { s.forEach { merge(it, 1, Int::plus) } }

Common Patterns — Know These Cold

Tip AC-10 · Recognise the pattern before choosing the algorithm

Train yourself to see the pattern type first. The pattern determines the algorithm.

Pattern signal in the problemLikely approach
"Find pair / triplet with target sum"Two pointers (sorted) or HashMap
"Longest / shortest subarray with property"Sliding window
"Top K / K most frequent"Min-heap of size K
"Find median from stream"Two heaps (max-heap + min-heap)
"All permutations / combinations / subsets"Backtracking
"Optimal substructure, overlapping subproblems"Dynamic programming
"Shortest path in unweighted graph"BFS
"Shortest path in weighted graph"Dijkstra (non-negative weights)
"Interval overlap / merge"Sort by start, then merge pass
"Tree / balanced?"DFS with return type carrying height + validity
"Number of islands / connected components"BFS/DFS or Union-Find
"Binary search on answer (not index)"Binary search on the value range

Closing Your Solution

Tip AC-11 · After testing, discuss improvements proactively

After verifying correctness, mention what you'd improve in production code without being asked. This takes 30 seconds and signals engineering maturity beyond just solving the puzzle.

"The solution is correct. In production code I'd also add: input validation at the API boundary, a meaningful exception message if the contract is violated, and I'd consider whether this is called in a hot path — if so, I'd profile the allocation of the intermediate list and potentially use an in-place approach."
Tip AC-12 · State what the follow-up variations would require

Interviewers often follow up with "what if the input is a stream?" or "what if we need this to be thread-safe?" Anticipate this and mention it after your solution. It shows you're thinking beyond the stated problem.

"For the stream variant, the sliding window approach doesn't directly work because we don't know the end of the window in advance. I'd shift to a deque-based approach. For thread safety, I'd either wrap in a lock or redesign as an immutable transformation on each new input."
