System Design Interview Tips

These are the proactive signals interviewers are scoring you on. Say them before you're asked. Every tip here has a "say this" phrase — practice each one until it comes naturally in under 10 seconds.

Opening — The First 5 Minutes

Tip SD-01 · Open with the component diagram, not with code

The moment you start talking about classes or functions before drawing the system, you've signalled L4. Always start with boxes and arrows. "Let me draw the component boundaries first before I go into implementation." This resets the conversation to the right level of abstraction.

"Before I get into any details, let me draw the high-level component structure so we're aligned on the system boundary first."
Tip SD-02 · Ask the four clarifying questions

Every system design needs answers to these before you draw anything: (1) Scale — DAU, request rate, data volume. (2) Primary user journey — what's the core flow? (3) Offline/reliability requirements — what's the SLA? (4) Existing infrastructure — are there systems we build on or integrate with? Asking these before answering signals maturity.

"A few questions before I start: What scale are we targeting — roughly how many daily active users? What's the primary user journey we're optimizing for? Do we have any offline or SLA requirements? And are there existing backend services I should build on top of?"
Tip SD-03 · Name what's out of scope

Explicitly stating what you're NOT designing is as important as what you are. It shows you've thought about the full problem space and made deliberate choices about where to invest the 45 minutes. Interviewers notice when candidates try to solve everything and end up solving nothing deeply.

"I'm going to focus on the client-side architecture and offline behavior. I'll treat the backend API as a black box and won't go into server-side design unless you want me to."

During the Design — Proactive Signals

Tip SD-04 · Say the failure mode before the interviewer asks

After describing any significant component, immediately name what can fail. "One failure mode here is X, so I'd add Y." This is the single most consistently scored L6 signal in system design rounds. Practice saying it reflexively after every major design decision.

"One failure mode here is that the network request can fail or time out. So I'd add a retry policy with exponential backoff and jitter — and critically, I'd never retry on a 400 or 404, only on 5xx and timeouts."
Tip SD-05 · Say the observability story before being asked

After describing a component, say: "Here's how I'd know if this was broken in production." Name one success metric, one error metric, and one alert threshold. This takes 30 seconds and scores heavily — most candidates never say this unless explicitly asked.

"To know if the sync is working in production, I'd track: sync success rate (alert below 95%), sync latency P95 (alert above 30s), and the queue depth of pending writes (alert above 500 — means we're falling behind). Each sync job would carry a trace ID so I can correlate client errors to server logs."
Tip SD-06 · Say the rollout plan before being asked

After completing the design, always add a rollout section before the interviewer wraps up. Kill switch + staged rollout + metric gates. This takes 60 seconds. Skipping it is the most common reason technically correct designs fail to get an L6 hire decision.

"Before I close out — I'd ship this behind a server-side kill switch so I can disable it in under 60 seconds without a release if I see anomalies. Then ramp: 1% for 24 hours watching crash rate and ANR rate, then 5%, 25%, 100% with automated metric gates. If any gate fails, the rollout pauses automatically."
Tip SD-07 · State the scale limit of your design

Proactively name where your design breaks. "This design works up to roughly X users. Beyond that, the bottleneck is Y, and I'd address it by Z." This shows you're reasoning about the future, not just the immediate requirements. You don't need to solve the problem for 100x — just name it.

"This design holds comfortably to ~10M DAU. Beyond that, the bottleneck is the Room database write throughput under concurrent workers. I'd address it by sharding the worker queue by user ID, or migrating heavy write operations to an append-only event log."
Tip SD-08 · Raise the security surface unprompted

Before finishing any design that involves user data, raise the security concern. You don't need to fully solve it — naming it is the signal. "Since this touches user data, I want to flag a few things: encryption at rest, certificate pinning, and making sure PII is excluded from logs."

"Since this feature touches user location data, I want to flag it: the data should be encrypted at rest with AES-256-GCM via the Android KeyStore. It should be excluded from crash logs and analytics by default. And it should be deleted within 30 days of the user requesting account deletion — I'd implement that as an atomic WorkManager chain."
Tip SD-09 · Explicitly connect your design to the requirements

After laying out a component, loop back to the original requirements. "This satisfies requirement X because..." Interviewers are checking if you're solving the stated problem or just showing off architecture knowledge. Connect the dots explicitly.

"This stale-while-revalidate approach satisfies the offline requirement you mentioned earlier — the user always sees cached data immediately, even on a flaky connection. The sync happens in the background and they see an update when it arrives."

The Five Things to Say at Every Design Phase

Requirements phase

  • Ask about the primary metric — what does "success" look like? (engagement, latency, reliability?)
  • Ask about the failure tolerance — is stale data acceptable? Can we lose a write?
  • Confirm the device/API level floor — "Are we targeting API 26+ or lower?"

Architecture phase

  • Name the dependency rule — UI depends on Domain, Domain does NOT depend on Data
  • Say why you chose this specific pattern (MVI vs MVVM) for this specific problem
  • Name the module boundary and why it's drawn there

Data layer phase

  • Name the single source of truth — "The UI always reads from Room, never directly from the network response"
  • State your cache invalidation strategy — TTL, event-driven, or stale-while-revalidate
  • Raise the migration story — "What happens when the schema changes in a future release?"

Networking phase

  • State the retry policy — never retry 4xx, always use jitter, reason: thundering herd
  • Name the auth token refresh race condition and how you handle it with Mutex
  • Raise the offline write queue — "What happens when the user writes while offline?"

Closing phase

  • Kill switch + staged rollout (say this even if unprompted)
  • Production observability — 3 metrics + alert threshold
  • Open questions — "If we had more time, I'd explore X, Y, Z"

Traps Interviewers Set — And How to Avoid Them

Trap 1 · "Just make it work, don't over-engineer"

This is testing if you can distinguish necessary complexity from gold-plating. The right answer: acknowledge the trade-off and explain specifically which parts are necessary for the stated requirements vs. which parts you'd defer. "You're right — the CRDT conflict resolution is over-engineering for this scale. I'd start with last-write-wins and only revisit if we see a specific conflict rate problem."

Trap 2 · "What if we need 10x the scale?"

Don't redesign the whole system. Name the specific bottleneck at 10x and describe the targeted change. "At 10x, the Room write throughput becomes the bottleneck. I'd address that specifically by batching writes and adding a write-ahead buffer — the rest of the design holds."

Trap 3 · Presenting one option as obviously correct

If you say "obviously you'd use X here" without naming why the alternative doesn't work, the interviewer will push back. Nothing is obvious. Name the alternative, explain why it doesn't fit, then commit to your choice.

Trap 4 · Spending all 45 minutes on architecture

The architecture section is easy to expand indefinitely. Check the clock at the 15-minute mark. If you haven't started the data layer yet, compress architecture now. The observability and rollout sections at the end are where L6 decisions are made — don't run out of time before you get there.
