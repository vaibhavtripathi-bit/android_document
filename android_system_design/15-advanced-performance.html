Advanced Performance: Systrace, ART GC & Memory Profiling

Why this matters at L6: L6 engineers don't guess at performance problems — they instrument, measure, and diagnose from data. Interviewers expect you to name specific tools, describe what you look for in a trace, and explain the mechanism behind each optimization. "I used the profiler" is L4. Describing what you saw in the Systrace and why it caused jank is L6.

What to Look for in a Jank Trace

Timeline (zoomed to a janky frame): Main Thread ───────────────────────────────────────────────────────── [Choreographer#doFrame] ← should complete in <16.67ms (60fps) │ ├── [measure] ← <2ms expected ├── [layout] ← <2ms expected ├── [draw] ← <4ms expected — if >16ms here, it's a draw bottleneck │ └── [binder call to SurfaceFlinger] ← should be instant ↓ if this is long: CPU contention, system overloaded RenderThread ──────────────────────────────────────────────────────── [DrawFrame] ← GPU execution — if long: overdraw or shader compilation SurfaceFlinger ────────────────────────────────────────────────────── [onMessageReceived] ← compositor — frame dropped if main thread missed deadline Key jank patterns to identify: 1. Main thread blocked by binder → IPC to another process (ContentProvider, system service) 2. Main thread blocked by lock → Mutex or synchronized blocking 3. Long GC pause on main thread → allocation pressure 4. RecyclerView.onBindViewHolder taking >2ms → heavy bind logic 5. Shader compilation jank → first render of new GPU shader program (fix: AGP shader prewarming)

How to Reduce GC Pressure

// 1. AVOID ALLOCATIONS IN HOT PATHS (onDraw, onBindViewHolder, scroll listeners)
// BAD: Allocating in onDraw
override fun onDraw(canvas: Canvas) {
    val paint = Paint()  // ❌ New allocation every frame!
    val rect = RectF()   // ❌ New allocation every frame!
    canvas.drawRect(rect, paint)
}

// GOOD: Pre-allocate, reuse
private val paint = Paint(Paint.ANTI_ALIAS_FLAG)
private val rect = RectF()

override fun onDraw(canvas: Canvas) {
    rect.set(0f, 0f, width.toFloat(), height.toFloat())  // Reuse
    canvas.drawRect(rect, paint)
}

// 2. OBJECT POOLS for frequent allocations
class PointPool {
    private val pool = ArrayDeque<PointF>(capacity = 32)

    fun obtain(): PointF = pool.removeLastOrNull() ?: PointF()
    fun recycle(point: PointF) {
        point.set(0f, 0f)
        if (pool.size < 32) pool.addLast(point)
    }
}

// 3. BITMAP REUSE (see 04-media-rendering.html)
// BitmapFactory.Options.inBitmap — reuse pixel buffer

// 4. AVOID BOXING in performance-critical code
// BAD: Boxing int to Integer in HashMap
val counters = HashMap<String, Int>()
val count = counters["key"] ?: 0  // Boxes/unboxes

// GOOD: SparseIntArray or SparseArray for int/long keys
val counters = SparseIntArray()

// 5. KOTLIN DATA CLASSES — be careful with copy()
// copy() allocates a new object even if one field changes
// In hot paths, consider manual mutation or mutable holders

4. Memory Leak Detection

LeakCanary Integration and What It Tells You

// LeakCanary: add to debugImplementation only
debugImplementation("com.squareup.leakcanary:leakcanary-android:2.14")
// No code changes needed — it hooks into Activity/Fragment lifecycle automatically

// Common leak patterns LeakCanary catches:

// 1. STATIC REFERENCE TO CONTEXT
object AppState {
    var context: Context? = null  // ❌ Static reference leaks Activity!
}

// 2. INNER CLASS HOLDING OUTER CLASS REFERENCE
class MyActivity : AppCompatActivity() {
    private val handler = object : Handler(Looper.getMainLooper()) {
        override fun handleMessage(msg: Message) {
            // ❌ Anonymous inner class holds implicit reference to MyActivity
            updateUI()
        }
    }
    // If handler has pending messages when Activity destroys → leak
}

// 3. COROUTINE IN WRONG SCOPE
class MyFragment : Fragment() {
    fun loadData() {
        // ❌ GlobalScope keeps reference to fragment's lambda
        GlobalScope.launch {
            val data = repo.fetch()
            view?.let { updateUI(it, data) }  // view might be null, but fragment isn't released
        }
    }
    // Fix: use viewLifecycleOwner.lifecycleScope.launch
}

// 4. REGISTERED BUT NEVER UNREGISTERED LISTENER
class MyFragment : Fragment() {
    override fun onStart() {
        super.onStart()
        locationManager.addListener(this)  // ❌ Fragment registered
    }
    // Missing: onStop { locationManager.removeListener(this) }
}

// LeakCanary output tells you:
// - Which object is leaking (e.g., MyActivity)
// - What reference chain is preventing collection
// - Line number of the leak

Memory Monitoring in Production

// Track memory metrics in production (not just development)
class MemoryMonitor(private val analytics: AnalyticsClient) {

    fun reportMemoryStats() {
        val runtime = Runtime.getRuntime()
        val usedMb = (runtime.totalMemory() - runtime.freeMemory()) / 1_048_576L
        val maxMb = runtime.maxMemory() / 1_048_576L
        val utilization = usedMb.toFloat() / maxMb

        analytics.track("memory_utilization", mapOf(
            "used_mb" to usedMb,
            "max_mb" to maxMb,
            "utilization_pct" to (utilization * 100).toInt()
        ))

        // Alert if approaching OOM
        if (utilization > 0.85f) {
            analytics.track("memory_pressure_high", mapOf("used_mb" to usedMb))
            // Proactively evict caches
            Glide.get(context).clearMemory()
            imageMemoryCache.evictAll()
        }
    }

    // ComponentCallbacks2 — OS signals memory pressure
    override fun onTrimMemory(level: Int) {
        when (level) {
            ComponentCallbacks2.TRIM_MEMORY_UI_HIDDEN -> {
                // App backgrounded: release UI caches
                imageMemoryCache.evictAll()
            }
            ComponentCallbacks2.TRIM_MEMORY_RUNNING_CRITICAL,
            ComponentCallbacks2.TRIM_MEMORY_COMPLETE -> {
                // System is critically low: release everything possible
                imageMemoryCache.evictAll()
                networkCache.evictAll()
                analytics.track("trim_memory_critical")
            }
        }
    }
}

5. Startup Performance Deep Dive

Cold Start Anatomy

t=0ms Process fork (zygote) t=50ms Application.onCreate() starts t=100ms Hilt component graph built (all @Singleton providers) t=150ms ContentProviders initialized (each one adds ~10-30ms!) t=200ms Application.onCreate() complete t=250ms Activity.onCreate() starts t=400ms setContentView / Compose content set t=600ms First measure/layout/draw pass t=800ms First frame rendered (TTID — Time To Initial Display) t=1500ms Data loaded, UI fully populated (TTFD — Time To Full Display) Budget targets (Google's definition of "good"): Cold start TTID: < 1000ms Cold start TTFD: < 2000ms Warm start: < 500ms Hot start: < 200ms
// Measure startup with Macrobenchmark (runs on real device in release mode)
@RunWith(AndroidJUnit4::class)
class StartupBenchmark {
    @get:Rule
    val benchmarkRule = MacrobenchmarkRule()

    @Test
    fun coldStart() = benchmarkRule.measureRepeated(
        packageName = "com.example.app",
        metrics = listOf(
            StartupTimingMetric(),   // TTID + TTFD
            FrameTimingMetric()      // Jank during startup
        ),
        iterations = 10,
        startupMode = StartupMode.COLD  // Kill process before each iteration
    ) {
        pressHome()
        startActivityAndWait()  // Wait until app reports fully drawn
    }
}

// Report fully drawn from Activity (critical for TTFD accuracy!)
class MainActivity : AppCompatActivity() {
    override fun onCreate(savedInstanceState: Bundle?) {
        super.onCreate(savedInstanceState)
        // ...
        viewModel.uiState.observe(this) { state ->

                render(state)
                reportFullyDrawn()  // ← Signal that content is visible
            }
        }
    }
}

// 4. DiffUtil with payload: only rebind changed fields, not whole item
class PostDiffCallback : DiffUtil.ItemCallback<Post>() {
    override fun areItemsTheSame(old: Post, new: Post) = old.id == new.id
    override fun areContentsTheSame(old: Post, new: Post) = old == new
    override fun getChangePayload(old: Post, new: Post): Any? {
        // Return only what changed — adapter receives this in onBindViewHolder(holder, pos, payloads)
        val diff = mutableMapOf<String, Any>()
        if (old.likeCount != new.likeCount) diff["likes"] = new.likeCount
        if (old.isLiked != new.isLiked) diff["isLiked"] = new.isLiked
        return if (diff.isNotEmpty()) diff else null
    }
}

// In adapter: handle payloads to avoid full rebind
override fun onBindViewHolder(holder: PostViewHolder, position: Int, payloads: List<Any>) {
    if (payloads.isEmpty()) {
        super.onBindViewHolder(holder, position, payloads)
        return
    }
    @Suppress("UNCHECKED_CAST")
    val diff = payloads[0] as Map<String, Any>
    diff["likes"]?.let { holder.updateLikeCount(it as Int) }
    diff["isLiked"]?.let { holder.updateLikeButton(it as Boolean) }
}

8. FrameMetrics in Production

// Collect per-frame timing in production (low overhead)
class JankTracker(private val analytics: AnalyticsClient) {

    private val metricsAggregator = FrameMetricsAggregator(
        FrameMetricsAggregator.TOTAL_DURATION or
        FrameMetricsAggregator.DRAW_DURATION or
        FrameMetricsAggregator.LAYOUT_MEASURE_DURATION
    )

    fun startTracking(activity: Activity) {
        metricsAggregator.add(activity)
    }

    fun stopAndReport(screenName: String) {
        val metrics = metricsAggregator.reset() ?: return

        val totalDurations = metrics[FrameMetricsAggregator.TOTAL_INDEX] ?: return

        var totalFrames = 0
        var jankFrames = 0   // > 16.67ms
        var frozenFrames = 0 // > 700ms (ANR risk)

        for (i in 0 until totalDurations.size()) {
            val durationMs = totalDurations.valueAt(i)
            val count = totalDurations.get(durationMs)
            totalFrames += count
            if (durationMs > 16) jankFrames += count
            if (durationMs > 700) frozenFrames += count
        }

        val jankRate = jankFrames.toFloat() / totalFrames
        analytics.track("frame_metrics", mapOf(
            "screen" to screenName,
            "total_frames" to totalFrames,
            "jank_rate_pct" to (jankRate * 100).toInt(),
            "frozen_frames" to frozenFrames
        ))

        // Alert if jank rate exceeds threshold
        if (jankRate > 0.05f) { // > 5% jank frames
            analytics.track("jank_threshold_exceeded", mapOf("screen" to screenName, "rate" to jankRate))
        }
    }
}

9. L6 Interview Narration Scripts

When asked "How do you diagnose a jank issue in production?"

"My process has three phases: detect, diagnose, fix.

For detection, I instrument production with FrameMetricsAggregator to collect per-frame timing and report jank rate (frames over 16.67ms) to our analytics pipeline by screen. This gives me a continuous signal — if a deploy causes a spike in jank rate on the Feed screen, I know within hours.

For diagnosis, I capture a Perfetto trace during the janky scenario. I look at the main thread timeline for frames that exceed the 16.67ms budget and identify what's taking time: is it the layout pass, the draw call, a binder IPC to SurfaceFlinger, or a GC pause? I also check the RenderThread — if the GPU is the bottleneck, I look at overdraw with the GPU overdraw visualizer in Developer Options.

For fixes: if it's layout inflation, I move to Compose which has no inflation overhead. If it's RecyclerView bind time, I add payload diffing so only changed fields rebind. If it's GC, I profile allocations and find objects being allocated in the hot path — onDraw, onBindViewHolder, scroll listeners — and either pre-allocate or use object pools."

When asked about startup optimization

"The highest-leverage startup optimization today is Baseline Profiles. ART compiles the critical paths — Activity creation, first RecyclerView population, first network request setup — ahead of time based on real usage patterns from a Macrobenchmark. Play delivers the compiled profile alongside the APK. We've seen 30–40% cold start improvements from this alone.

Beyond that, I audit ContentProviders — each one adds 10–30ms synchronously on the main thread. I consolidate them with the App Startup library into a single provider with an Initializer chain, and move heavy initializers to async or lazy. I use StrictMode in debug builds to catch any main thread disk or network I/O before it ships — those are often the 100–500ms startup killers that are easy to miss in development."

10. Rejection Patterns

What Signals L4/L5 on Performance

  • "I used the profiler to find the slow method" — Too vague. L6 names the tool, the trace pattern, the specific metric.
  • Not knowing FrameMetricsAggregator — Shows no production performance monitoring beyond crash rates
  • "I just avoid doing too much on the main thread" — Generic advice with no specifics on detection or measurement
  • Not knowing about Baseline Profiles — Missing the most impactful modern startup optimization
  • Confusing TTID and TTFD — L6 knows the difference: TTID = first frame, TTFD = content fully loaded
  • No mention of Macrobenchmark — Performance optimization without reproducible benchmarks isn't engineering, it's guessing
  • "I call System.gc() to free memory" — This causes a stop-the-world GC pause. Immediate red flag.
