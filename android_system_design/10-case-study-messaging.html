10 — Case Study: Offline-First Messaging App

Full 45-minute L6 answer · Message ordering · Delivery receipts · Offline writes · Conflict resolution

Why this is the hardest case study
Messaging combines every domain: offline-first architecture, local storage schema, background delivery, real-time updates, conflict resolution, and security. It's the question where candidates with 15 years of experience can fail by going too shallow on any one area. The secret is knowing which parts to go deep on (offline writes, message ordering) and which to summarize quickly (auth, basic UI).

Requirements Phase 3–4 min

Functional: One-to-one messaging or group chat too? Media attachments (images, video)? Read receipts? Typing indicators? Voice/video calls?

[Interviewer: 1:1 and group chat up to 256 members, text and image attachments, read receipts. No calls for now.]

Scale: DAU? Message volume? Are we WhatsApp-scale (2B users) or a startup?

[Interviewer: 100M DAU, ~500M messages per day. Android client focus.]

Offline requirements: Should users be able to compose and queue messages while offline? Should they be able to read previous messages offline?

[Interviewer: Yes to both — this is explicitly offline-first.]

Consistency model: Is eventual consistency acceptable for message delivery, or do we need strong ordering guarantees?

[Interviewer: Messages within a conversation must appear in the order they were sent. Delivery can be eventual.]

Non-functional requirements I'll design for: messages display instantly when sent (optimistic UI), received messages appear in real-time when online, app works fully offline for read + draft, delivery and read receipts are reliable."

Data Model — The Foundation 5–6 min

// Message entity — designed for messaging query patterns
@Entity(
    tableName = "messages",
    indices = [
        Index(value = ["conversationId", "serverTimestamp"]), // primary query: messages by conversation ordered by time
        Index(value = ["localId"], unique = true),               // deduplication by localId
        Index(value = ["status"])                              // query: all PENDING messages for retry
    ]
)
data class MessageEntity(
    @PrimaryKey val localId: String,    // UUID generated on device — never null
    val serverId: String?,              // null until server confirms delivery
    val conversationId: String,
    val senderId: String,
    val content: String,               // encrypted at rest for E2EE apps
    val attachmentLocalPath: String?,  // local file path for images
    val attachmentUrl: String?,        // CDN URL after upload
    val clientTimestamp: Long,         // time user composed the message
    val serverTimestamp: Long?,        // null until server assigns canonical order
    val status: MessageStatus,         // COMPOSING, PENDING, SENT, DELIVERED, READ, FAILED
    val retryCount: Int = 0
)

enum class MessageStatus {
    COMPOSING,   // draft, not yet submitted
    PENDING,     // submitted but not yet delivered to server
    SENT,        // server received it (has serverId)
    DELIVERED,   // recipient device received it (delivery receipt)
    READ,        // recipient opened conversation (read receipt)
    FAILED       // permanently failed after max retries
}

// Conversation entity — summary for list screen
@Entity(tableName = "conversations")
data class ConversationEntity(
    @PrimaryKey val id: String,
    val participantIds: String,  // JSON array — simple denormalization for list view
    val lastMessageId: String?,
    val lastMessagePreview: String?,
    val unreadCount: Int,
    val updatedAt: Long
)
L6 Signal — localId vs serverId
The localId is the primary key, not the server ID. This is the critical decision for offline-first messaging. The user can compose a message while offline — at that point, there is no server ID. If the primary key is the server ID, we can't insert the row into Room until the server responds. With a client-generated UUID as the primary key, we can write to Room immediately and fill in the server ID later. This pattern is non-negotiable for offline-first design.

Message Ordering — The Hard Problem 6–8 min

Message ordering is the most subtle and most probed area of messaging system design. There are multiple clocks in play: the device clock (unreliable), the server clock (reliable but introduces latency), and the relative ordering of events.

The Problem With Client Timestamps

Raise This Proactively
"I want to address message ordering proactively because it's subtle. The naive approach is to sort messages by their clientTimestamp — the time the user sent the message on their device. This breaks in three ways: device clocks can be wrong (user manually set their clock back), two users can send messages at the same millisecond, and offline messages have a client timestamp from when they were composed, not when they were delivered — so a message composed at 10am while offline and delivered at 11am appears before messages sent at 10:30am by others, even though the recipient received them first.

The correct ordering is: server-assigned logical order. Each message gets a server-assigned sequence number within a conversation. On the device, we display messages sorted by server sequence. For pending messages (not yet delivered to server), we optimistically show them at the end with their client timestamp — they shift to their correct server-assigned position when the ACK comes back. This is the pattern used by iMessage, WhatsApp, and Slack."

The Send Flow — Offline-First 5–6 min

// Complete send flow — works offline and online
class SendMessageUseCase(
    private val messageDao: MessageDao,
    private val workManager: WorkManager,
    private val attachmentRepo: AttachmentRepository
) {
    suspend fun invoke(
        conversationId: String,
        content: String,
        attachmentUri: Uri? = null
    ): String { // returns localId

        val localId = UUID.randomUUID().toString()

        // Step 1: Write to Room immediately — UI shows message instantly
        messageDao.insert(
            MessageEntity(
                localId = localId,
                serverId = null,
                conversationId = conversationId,
                senderId = currentUserId,
                content = content,
                attachmentLocalPath = attachmentUri?.path,
                clientTimestamp = System.currentTimeMillis(),
                status = MessageStatus.PENDING
            )
        )
        // UI already updated via Room Flow — user sees their message instantly

        // Step 2: Enqueue delivery — survives app kill, offline, reboots
        workManager.enqueueUniqueWork(
            "send_$localId",
            ExistingWorkPolicy.KEEP,
            OneTimeWorkRequestBuilder<MessageDeliveryWorker>()
                .setConstraints(Constraints(requiredNetworkType = NetworkType.CONNECTED))
                .setInputData(workDataOf("localId" to localId))
                .setBackoffCriteria(BackoffPolicy.EXPONENTIAL, 10, TimeUnit.SECONDS)
                .build()
        )
        return localId
    }
}

class MessageDeliveryWorker(...) : CoroutineWorker(...) {
    override suspend fun doWork(): Result {
        val localId = inputData.getString("localId") ?: return Result.failure()
        val message = messageDao.getByLocalId(localId) ?: return Result.failure()

        // Upload attachment first if present
        val attachmentUrl = message.attachmentLocalPath?.let {
            attachmentRepo.upload(it).getOrElse { return Result.retry() }
        }

        return try {
            val response = messageApi.sendMessage(
                SendMessageRequest(
                    idempotencyKey = localId, // server deduplicates by this key
                    conversationId = message.conversationId,
                    content = message.content,
                    attachmentUrl = attachmentUrl
                )
            )
            // Server assigned serverId and serverTimestamp
            messageDao.onServerDelivered(localId, response.serverId, response.serverTimestamp)
            // status → SENT, message moves to correct position in conversation
            Result.success()
        } catch (e: IOException) {
            if (runAttemptCount >= 5) {
                messageDao.updateStatus(localId, MessageStatus.FAILED)
                Result.failure() // show retry button to user
            } else Result.retry()
        }
    }
}
The Idempotency Key — Don't Skip This
The idempotency key (localId) is sent with every message to the server. If the network request succeeds but the response is lost (device went offline), the WorkManager will retry. Without the idempotency key, the server creates a duplicate message. With it, the server detects "I already processed this localId" and returns the same response — no duplicate. This is a required API contract, not an optional optimization.

class MessagingFcmService : FirebaseMessagingService() {
    override fun onMessageReceived(message: RemoteMessage) {
        when (message.data["type"]) {
            "new_message" -> {
                // Show notification immediately from FCM payload
                showNotification(message.data)
                // Enqueue expedited sync to fetch full message + history
                workManager.enqueueUniqueWork(
                    "sync_messages",
                    ExistingWorkPolicy.REPLACE,
                    buildSyncWork().setExpedited(OutOfQuotaPolicy.RUN_AS_NON_EXPEDITED_WORK_REQUEST).build()
                )
            }
            "delivery_receipt" -> updateMessageStatus(message.data["messageId"], MessageStatus.DELIVERED)
            "read_receipt" -> updateMessageStatus(message.data["messageId"], MessageStatus.READ)
        }
    }
}

// Path 2: WebSocket — when app is foreground
// Lower latency than FCM (~50ms vs ~500ms)
// Connected only when app is in foreground
class MessageWebSocketManager(...) {
    fun connect() {
        val wsListener = object : WebSocketListener() {
            override fun onMessage(webSocket: WebSocket, text: String) {
                val event = parseEvent(text)
                scope.launch { processEvent(event) }
            }
            override fun onFailure(webSocket: WebSocket, t: Throwable, response: Response?) {
                reconnectWithBackoff() // exponential backoff reconnection
            }
        }
        okHttpClient.newWebSocket(request, wsListener)
    }

    private suspend fun processEvent(event: WsEvent) {
        when (event) {
            is WsEvent.NewMessage -> {
                messageDao.insertOrIgnore(event.message.toEntity())
                // Room Flow propagates to UI automatically — zero extra code
                // insertOrIgnore: if FCM delivered it first, ignore duplicate
            }
            is WsEvent.TypingIndicator -> updateTypingState(event)
        }
    }
}

Delivery & Read Receipts 3–4 min

// Delivery receipt: recipient device received the message
// Read receipt: recipient opened and viewed the message
// Both must be reliable — queue in WorkManager like messages

fun onConversationOpened(conversationId: String) {
    viewModelScope.launch {
        // Mark all unread messages as read locally
        messageDao.markAllRead(conversationId, currentUserId)

        // Queue read receipt delivery — fire and forget with WorkManager
        workManager.enqueue(
            OneTimeWorkRequestBuilder<SendReadReceiptWorker>()
                .setConstraints(Constraints(requiredNetworkType = NetworkType.CONNECTED))
                .setInputData(workDataOf("conversationId" to conversationId))
                .build()
        )
    }
}

Staff+ Layer 6–8 min

Complete Staff+ Narration — Raise These Unprompted
Message ordering edge case I'd flag:
"What happens if the user is offline for 3 days and then reconnects? They have queued messages that were composed 3 days ago — their clientTimestamp is old. When delivered, the server will insert them into the conversation with their server-assigned timestamp (now), not the composition timestamp. So the messages will appear at the bottom of the conversation, not at the time they were written. This is actually correct behavior — the recipient should see them in the order they arrived, not in the order they were composed. But we should show the composition timestamp in the message bubble so the recipient understands context."

Pagination in the message list:
"I'll paginate the conversation — load the most recent 50 messages initially, load more on scroll to top. I'll use Paging 3 with RemoteMediator: Room is the source for what we have, network loads older history when user scrolls to the beginning. The pagination key is the oldest serverTimestamp in the current page — pass it to the API as 'give me messages before this timestamp.'"

Attachment handling:
"For image attachments: the upload happens before or during message delivery. I'd upload the image to a CDN first, get the URL, then send the message with the URL. While uploading, the message shows with the local file path as a thumbnail. If upload fails, the message goes to FAILED state with a retry option. One important detail: I store the local file path AND the CDN URL separately in the entity. The local file path lets me show the thumbnail instantly even if CDN is unreachable."

Observability:
"The metrics I'd track: message delivery success rate (target >99.9%), p50/p95 delivery latency (from send to server ACK), notification delivery rate (FCM delivery vs messages sent — tells me if users are receiving push). I'd also track offline queue depth at sync — how many messages were queued while offline. If this is consistently high for a segment of users, they may be on very unreliable networks and we should optimize for more aggressive batching."

Rollout:
"The messaging DB schema is the hardest rollout constraint. Once I ship a schema with this structure, every migration from here forward must be additive. I'd document the schema decisions in the design review and get sign-off from the team before shipping — because changing the primary key strategy from localId to serverId later would require a full data migration."

Follow-up Questions to Prepare

  • "How would you add end-to-end encryption?" — Keys are generated on-device (Signal Protocol or similar), never leave the device. Server stores only encrypted ciphertext. Key exchange happens via a DH handshake at first message. Key storage: Android KeyStore. Key backup: user-managed recovery phrase, not server-stored.
  • "How do you handle group messages of 256 members?" — Message is delivered to the server once. Server fans out to all recipients. Client doesn't need to know about fan-out. Read receipts in groups are aggregated — "Seen by 47 people" rather than per-user status, to avoid privacy concerns and reduce storage.
  • "What happens if two users send a message at exactly the same time?" — Both messages get different server-assigned sequence numbers. The one that hits the server first gets a lower sequence number — that's the ordering. No special handling needed — server serializes concurrent writes.
  • "How do you handle message editing or deletion?" — These are soft operations: edited messages have an editedAt field and updated content. Deleted messages: content replaced with a tombstone marker ("This message was deleted") — the row stays in Room so the conversation timeline doesn't have gaps. Recipients get an FCM event that updates the local Room row.
