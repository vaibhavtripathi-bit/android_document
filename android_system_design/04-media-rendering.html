04 — Media & Rendering Performance

Android L6 System Design · ExoPlayer · Image Loading · Vsync · Compose Recomposition · Frame Budget

Why this is an L6 trap
Media questions are high surface area. Candidates either go too shallow ("I'd use ExoPlayer") or too narrow ("let me explain ExoPlayer's buffering algorithm for 20 minutes"). The L6 answer is architectural: how does the media component fit into the overall system, what are the lifecycle risks, memory risks, and platform-specific constraints?

1. The Frame Budget — The Number You Must Know

Everything in rendering performance flows from one constraint: 16ms per frame at 60fps (8ms at 120fps on high-refresh-rate displays). If any work on the main thread takes longer, you drop a frame. The user sees jank.

BudgetRefresh RateImplication
16.67ms60 Hz (most devices)Standard target — must not be exceeded on the main thread
8.33ms120 Hz (Pixel 6+, Galaxy S22+)High-refresh displays — any optimization matters twice as much
33.3ms30 Hz (background/throttled)System throttles animations when battery-saver is on
L6 Signal
When you say "this is expensive", quantify it. "Decoding a 4MP image on the main thread takes ~40ms on a mid-range device — that's 2.5 dropped frames at 60fps." Numbers convert a generic concern into an engineering argument.

2. Image Loading Pipeline — Design It from Scratch

At L6 you should be able to design an image loading library, not just use Coil/Glide. This question comes up frequently.

The 5-Layer Pipeline

1
Request Resolution — Normalize URL, apply transformations (resize, crop, placeholder). Build a cache key from URL + target size + transformations.
2
Memory Cache (LruCache<Bitmap>) — Check in-process bitmap cache first. Cache key = URL + decoded size. Hit = immediate return, no I/O.
3
Disk Cache — Check local file cache (DiskLruCache). If hit, decode from file on background thread → add to memory cache → display.
4
Network Fetch — OkHttp download. Respect HTTP cache-control headers. Write to disk cache. Decode on background thread. Downscale to target view size before decoding — never decode at full resolution.
5
Target Delivery — Set bitmap on ImageView / AsyncImage. Cancel inflight request when View is recycled or Composable leaves composition.

The Critical Details That Signal L6

// 1. Downscale BEFORE decoding — the most important optimization
// Full-resolution decode wastes memory and CPU
val options = BitmapFactory.Options()
options.inJustDecodeBounds = true
BitmapFactory.decodeStream(stream, null, options) // reads dimensions without pixels
options.inSampleSize = calculateInSampleSize(options, targetWidth, targetHeight)
// inSampleSize=4 means decode at 1/4 the resolution — 16x less memory
options.inJustDecodeBounds = false
val bitmap = BitmapFactory.decodeStream(stream, null, options)

// 2. Bitmap pool — reuse pixel buffers to avoid GC pressure
// Coil and Glide both use BitmapPool internally
// BitmapFactory.Options.inBitmap = existing bitmap of the right size
// Decoder reuses the pixel buffer — no new allocation, no GC event
options.inBitmap = bitmapPool.get(targetWidth, targetHeight, Bitmap.Config.ARGB_8888)
options.inMutable = true

// 3. Memory cache sizing
// Rule of thumb: 15-25% of available memory for image cache
val maxMemory = Runtime.getRuntime().maxMemory().toInt()
val cacheSize = maxMemory / 6 // ~16% of app's memory budget
val memoryCache = LruCache<String, Bitmap>(cacheSize) {
    override fun sizeOf(key: String, value: Bitmap) = value.byteCount
}

// 4. Request deduplication — same URL, two ImageViews
// Only one network request should be in flight
// Map<String, Deferred<Bitmap>> in the ImageLoader
private val inflight = ConcurrentHashMap<String, Deferred<Bitmap>>()
suspend fun load(url: String): Bitmap {
    return inflight.getOrPut(url) {
        viewModelScope.async { fetchAndDecode(url) }
    }.await().also { inflight.remove(url) }
}
L6 Script — Image Loading Design
"I'd design the image loader with a two-level cache: memory cache (LruCache, ~16% of app heap) and disk cache (DiskLruCache, 50-100MB on disk). The cache key is a hash of URL plus decoded dimensions — the same image at different sizes is a different cache entry.

The most critical optimization happens before decoding: I use inJustDecodeBounds to read the image dimensions without loading pixels, then compute inSampleSize so I decode at the exact size the view needs. Loading a 4MP photo full-resolution for a 48dp thumbnail would be 16MB of allocation — with downscaling it's under 100KB.

I'd also use bitmap pooling with inBitmap. In a RecyclerView, the same memory location can be reused when cells scroll off-screen. This avoids GC events that can cause frame drops.

For request cancellation: every load is tied to the Composable's lifecycle via rememberAsyncImagePainter in Coil or Glide's RequestManager lifecycle binding. When the composable leaves the composition, the inflight request is cancelled. Without this, you leak requests and risk setting images on recycled views."

Adaptive Bitrate Streaming (ABR) — The L6 Video Topic

// ExoPlayer's ABR algorithm monitors:
// 1. Available bandwidth (via chunk download speed measurement)
// 2. Buffer fill level (how much video is buffered ahead)

// ExoPlayer switches between representations automatically

val trackSelector = DefaultTrackSelector(context)
trackSelector.setParameters(
    trackSelector.buildUponParameters()

        .setForceHighestSupportedBitrate(true) // force highest on WiFi

// Check network type to adapt quality strategy
val isMetered = connectivityManager.isActiveNetworkMetered()
trackSelector.setParameters(
    trackSelector.buildUponParameters()

Player Lifecycle — The Most Common Bug Source

// ExoPlayer holds a MediaCodec, AudioTrack, Surface — expensive OS resources
// Must be released when not needed, restored when coming back

// In a ViewModel — player outlives configuration changes
class VideoViewModel(
    private val context: Application
) : AndroidViewModel(context) {

    val player: ExoPlayer = ExoPlayer.Builder(context).build()

    fun prepareMedia(url: String) {
        player.setMediaItem(MediaItem.fromUri(url))
        player.playWhenReady = true
        player.prepare()
    }

    override fun onCleared() {
        player.release() // releases MediaCodec, AudioTrack, Surface binding
        super.onCleared()
    }
}

// In the composable / fragment
// Detach Surface when going to background — free GPU memory
val lifecycleOwner = LocalLifecycleOwner.current()
DisposableEffect(lifecycleOwner) {
    val observer = LifecycleEventObserver { _, event ->
        when (event) {
            Lifecycle.Event.ON_PAUSE -> player.pause()
            Lifecycle.Event.ON_RESUME -> player.play()
            else -> {}
        }
    }
    lifecycleOwner.lifecycle.addObserver(observer)
    onDispose { lifecycleOwner.lifecycle.removeObserver(observer) }
}
Interview Probe — "What happens if the user gets a phone call mid-video?"
Audio focus management. ExoPlayer doesn't handle this automatically. You must request AudioFocus and pause/reduce volume when it's lost. This is an L6 detail because it affects the user experience in a real-world scenario that's easy to forget in design but obvious once reported as a bug. 
// Audio focus — required for production video apps
val audioAttributes = AudioAttributes.Builder()
    .setUsage(C.USAGE_MEDIA)
    .setContentType(C.AUDIO_CONTENT_TYPE_MOVIE)
    .build()
player.setAudioAttributes(audioAttributes, /* handleAudioFocus= */ true)
// handleAudioFocus=true: ExoPlayer pauses on audio focus loss,
// resumes when focus returns. This handles: phone calls, navigation,
// other audio apps stealing focus. Don't implement this yourself.

4. Video Preloading in a Feed — The TikTok Pattern

This is the most common advanced video question at FAANG: "Design a short-video feed where the next video starts instantly when you swipe."

// Strategy: maintain a pool of ExoPlayer instances
// Current item: playing
// Next item: buffered and paused
// Previous item: kept alive for back-swipe
// Items beyond that: released

class VideoPlayerPool(
    private val context: Context,
    private val poolSize: Int = 3
) {
    private val players = LinkedHashMap<String, ExoPlayer>() // URL → player

    fun getOrCreate(url: String): ExoPlayer {
        return players.getOrPut(url) {
            if (players.size >= poolSize) {
                val oldest = players.keys.first()
                players.remove(oldest)?.release() // evict LRU
            }
            ExoPlayer.Builder(context).build().also { player ->
                player.setMediaItem(MediaItem.fromUri(url))
                player.prepare() // start buffering, don't play yet
                player.playWhenReady = false
            }
        }
    }

    fun play(url: String) { players[url]?.playWhenReady = true }
    fun pause(url: String) { players[url]?.playWhenReady = false }
    fun releaseAll() { players.values.forEach { it.release() }; players.clear() }
}

// In the pager composable: preload adjacent items
LaunchedEffect(currentPage) {
    // Play current
    videoPlayerPool.play(items[currentPage].videoUrl)
    // Preload next
    items.getOrNull(currentPage + 1)?.let { videoPlayerPool.getOrCreate(it.videoUrl) }
    // Pause previous
    items.getOrNull(currentPage - 1)?.let { videoPlayerPool.pause(it.videoUrl) }
}
L6 Script — Video Feed Preloading
"For the short video feed, instant playback requires that the next video is already buffered when the user swipes. I'd use a pool of 3 ExoPlayer instances — current, next, and previous. The current one plays. The next one has prepare() called but playWhenReady = false — it buffers in the background. The previous one stays alive for the back-swipe case. When the user swipes to a fourth video, I release the oldest player from the pool and create a new one.

The tricky part is Surface binding. Each ExoPlayer must be bound to its own SurfaceView or TextureView. When the player is preloading off-screen, it can use a detached TextureView — no GPU cost. When it comes on screen, I bind the Surface, and because the video data is already buffered, playback starts in under 100ms.

Memory budget: 3 ExoPlayer instances + their codec allocations is roughly 60-100MB on a mid-range device. I'd instrument this with a memory profiler and dial down pool size if I see pressure."

5. Compose Recomposition — The Performance Trap

Compose recomposition bugs don't crash the app — they cause subtle performance issues that are hard to attribute. At L6 you need to understand the recomposition model well enough to design Composables that don't recompose unnecessarily.

How Recomposition Works

// Compose recomposes a Composable when its inputs change
// "Inputs" = the arguments to the Composable function
// If inputs haven't changed (by structural equality), Compose SKIPS recomposition
// This is called "skippability" — a critical performance property

// BAD: Unstable class — Compose can't skip recomposition
data class FeedItem(val tags: MutableList<String>) // MutableList is unstable!

// GOOD: Stable class — Compose can skip if reference didn't change
data class FeedItem(val tags: List<String>) // immutable List is stable
// Or annotate explicitly:
@Stable
data class FeedItem(val tags: ImmutableList<String>) // kotlinx.collections.immutable

// Rule: Every class that flows into a Composable must be either:
// 1. A primitive or String
// 2. A data class with only stable fields
// 3. Annotated with @Stable or @Immutable
// Otherwise Compose assumes it might have changed → recomposes

Lambda Stability — The Hidden Cost

// BAD: lambda captures state from the outer scope on every recomposition
@Composable
fun FeedScreen(viewModel: FeedViewModel) {
    val items = viewModel.state.collectAsState()
    LazyColumn {
        items(items.value) { item ->
            FeedItemCard(
                item = item,
                onClick = { viewModel.handleIntent(FeedIntent.SelectItem(item)) }
                // New lambda instance every recomposition → FeedItemCard always recomposes
            )
        }
    }
}

// GOOD: stable lambda via remember or passing stable function reference
@Composable
fun FeedScreen(viewModel: FeedViewModel) {
    val items = viewModel.state.collectAsState()
    val onItemClick: (FeedItem) -> Unit = remember(viewModel) {
        { item -> viewModel.handleIntent(FeedIntent.SelectItem(item)) }
    }
    // Same lambda instance unless viewModel changes → FeedItemCard can skip
}

Key Recomposition Rules — Know All Five

RuleImpact
Use @Stable / @Immutable on all UiState classesEnables skippability — Compose can skip unchanged composables
Use remember(key) for expensive objectsPrevents re-creation on every recomposition
Use key() in LazyList for stable identityPrevents full recomposition of the list on single-item update
Use derivedStateOf for computed valuesRecomputes only when inputs change, not on every recomposition
Read state as late as possible in the treeLimits the scope of recomposition — only the composable that reads the state recomposes
L6 Tool — Recomposition Counting
Use the Compose compiler metrics: ./gradlew assembleRelease -PenableComposeCompilerReports=true. It outputs a report of every Composable's stability and whether it's skippable. At L6, you'd mandate that all leaf composables in performance-critical screens (feed, player) are marked as skippable in this report as a CI gate.

6. RecyclerView Performance — Still Relevant

In mixed View/Compose codebases and for performance-critical lists, RecyclerView knowledge is still tested.

// DiffUtil — never call notifyDataSetChanged()
// notifyDataSetChanged() redraws the ENTIRE list — no animation, no perf
class FeedDiffCallback(private val old: List<FeedItem>, private val new: List<FeedItem>) : DiffUtil.Callback() {
    override fun areItemsTheSame(old: Int, new: Int) = this.old[old].id == this.new[new].id
    override fun areContentsTheSame(old: Int, new: Int) = this.old[old] == this.new[new]
    override fun getChangePayload(old: Int, new: Int): Any? {
        // Return a Payload to do partial bind updates (e.g., only like count changed)
        // Without payload: full onBindViewHolder() call for every changed field
        // With payload: only the changed fields are updated — faster, no flicker
        return FeedItemPayload.compute(this.old[old], this.new[new])
    }
}

// RecyclerView prefetch — pre-creates ViewHolders before they scroll on screen
recyclerView.recycledViewPool.setMaxRecycledViews(VIEW_TYPE_FEED, 10)
// Default pool size is 5 — if you scroll fast, pool exhausts and new ViewHolders
// are created on the main thread. Set pool size to expected visible count + buffer.

8. Hardware Codec Fallback Strategy

Hardware codecs (MediaCodec with hardware acceleration) vary wildly across Android OEMs. Some devices have buggy hardware decoders for specific formats or resolutions. L6 engineers design graceful fallback rather than assuming hardware always works.

LevelDecoderPerformanceRisk
1 (preferred)Hardware (OMX/C2 HW)Best — dedicated chip, low batteryOEM-specific bugs at certain resolutions
2 (fallback)Software (OMX.google / C2 SW)Higher CPU, more batteryConsistent across all devices
3 (last resort)Lower resolution / bitrateDegraded qualityAlways works
// ExoPlayer: enable automatic software fallback
val renderersFactory = DefaultRenderersFactory(context).apply {
    setExtensionRendererMode(
        DefaultRenderersFactory.EXTENSION_RENDERER_MODE_PREFER
        // PREFER: try hardware first, fall back to software extension if HW fails
    )
}

// Track which decoder is actually selected in production
player.addAnalyticsListener(object : AnalyticsListener {
    override fun onVideoDecoderChanged(
        eventTime: AnalyticsListener.EventTime,
        decoderName: String?,
        initializedTimestampMs: Long,
        initializationDurationMs: Long
    ) {
        val isHardware = decoderName?.let {
            !it.contains("OMX.google") && !it.contains("c2.android")
        } ?: false
        analytics.track("decoder_selected", mapOf(
            "name" to (decoderName ?: "unknown"),
            "is_hardware" to isHardware,
            "device" to Build.MODEL
        ))
    }
})

// Known-bad OEM decoders: maintain a blocklist
object CodecBlocklist {
    // Samsung Exynos HW H.264 decoder has bugs at 4K on some S21 variants
    // MediaTek HW VP9 decoder is broken on certain firmware versions
    private val blocked = listOf(
        "OMX.Exynos.avc.dec" to "SM-G99",   // model prefix
        "OMX.MTK.VIDEO.DECODER.VP9" to null  // all MTK devices
    )
    fun shouldBlock(decoderName: String, model: String) =
        blocked.any { (name, prefix) ->
            decoderName.contains(name, ignoreCase = true) &&
            (prefix == null || model.startsWith(prefix))
        }
}

// Apply blocklist via custom MediaCodecSelector
val codecSelector = MediaCodecSelector { mimeType, requiresSecure, requiresTunneling ->
    MediaCodecSelector.DEFAULT
        .getDecoderInfos(mimeType, requiresSecure, requiresTunneling)
        .filter { !CodecBlocklist.shouldBlock(it.name, Build.MODEL) }
}
val renderersFactory = DefaultRenderersFactory(context)
    .setMediaCodecSelector(codecSelector)
Say this unprompted: "I track the decoder name and whether it's hardware or software in production analytics. If I see a spike in software decoder selections on a specific device model, it tells me the hardware decoder is failing on that OEM. I maintain a codec blocklist for known-bad OEM decoders rather than disabling hardware decoding globally — targeted fix, not a blanket downgrade."

9. What Gets You Rejected Here

Rejection Pattern — "I'd use ExoPlayer"
That's the starting point, not a design. Which MediaSource? What LoadControl settings? How do you handle lifecycle? Audio focus? Background playback? Preloading for the feed? Any one of these is an L6 follow-up.
Rejection Pattern — No Memory Budget
You never quantified the memory footprint of your design. How many MB does a 3-player pool consume? What's the image cache size budget? On Android, memory is a first-class constraint — ignoring it is an L4 signal.
Rejection Pattern — Happy-Path Rendering
No mention of what happens when the user scrolls fast (image loading lag), when the device is under memory pressure (system kills cached process), or when the network is slow (video stalls, buffering spinner). These are the real user experiences.
