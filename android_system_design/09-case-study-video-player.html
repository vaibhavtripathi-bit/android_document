Architecture Overview 4–5 min

UI Layer (Compose) ├── PlayerScreen (Composable) │ ├── PlayerSurface (SurfaceView — hardware accelerated) │ ├── PlayerControls (play/pause/seek/quality) │ └── PlayerUiState (MVI — single state object) │ ├── PlayerViewModel │ ├── ExoPlayer instance (lives here — survives config change) │ └── PlayerIntent → PlayerState reducer │ Domain Layer ├── GetVideoUseCase → VideoMetadata + StreamManifestUrl ├── DownloadVideoUseCase → triggers WorkManager offline download └── TrackPlaybackUseCase → analytics events │ Data Layer ├── VideoRepository │ ├── VideoApi (Retrofit — fetches DASH/HLS manifest URL) │ └── DownloadRepository (WorkManager + local file storage) │ └── ExoPlayer (Media Engine) ├── DashMediaSource / HlsMediaSource (reads adaptive manifest) ├── CacheDataSource.Factory (disk cache for video segments) ├── DefaultLoadControl (buffer config) └── DefaultTrackSelector (ABR — quality selection)
Architecture Narration
"I'll use Clean Architecture with MVI for the player screen. I choose MVI because the player has complex concurrent state: playback state (playing/paused/buffering/ended), quality level, PiP mode, error state, and castability — and these can change simultaneously. A single UiState data class prevents inconsistent combinations.

The ExoPlayer instance lives in the ViewModel, not the Fragment or Composable. That's critical — it survives configuration changes. If ExoPlayer lives in a Fragment, rotating the device creates a new ExoPlayer, interrupts playback, and loses buffer — a terrible user experience.

The critical path: user taps a video → ViewModel fetches video metadata and manifest URL → ExoPlayer receives the DASH manifest and begins adaptive streaming → first segment buffers → playback begins. I target <2s to first frame on WiFi."

Deep Dives & Trade-offs 12–15 min

1. Adaptive Bitrate — How and Why

Trade-off Narration
"For streaming, I'll use DASH (Dynamic Adaptive Streaming over HTTP). DASH gives us: multiple quality representations in one manifest, segment-based delivery (each ~2-10 seconds, independently cacheable), and standardized ABR support in ExoPlayer.

ExoPlayer's ABR algorithm monitors three things: estimated bandwidth from chunk download speed, current buffer fill level, and whether the device can decode a higher quality in real-time. It switches quality at segment boundaries — so quality changes are imperceptible.

The trade-off I'll make: on a metered connection, cap quality at 720p even if bandwidth allows 1080p. The user can override this in settings. Why: data cost is a real concern for many users, and 720p is indistinguishable from 1080p on most phone screens. I'd surface this as a 'Save data' toggle."

2. Startup Time — The Most Visible Metric

// Target: <2s from tapping the video to first frame
// Breakdown of the 2-second budget:
// 0-50ms: ViewModel starts, ExoPlayer already created
// 50-200ms: API call for video metadata (CDN-cached, fast)
// 200-400ms: ExoPlayer parses DASH manifest
// 400-800ms: First segment downloaded (2.5s buffer for 2.5s buffer start)
// 800-1800ms: Segment decoded and first frame displayed

// Optimizations to hit <2s:
// 1. Parallel fetch: request manifest AND video metadata simultaneously
viewModelScope.launch {
    val metadataDeferred = async { videoApi.getMetadata(videoId) }
    val manifestDeferred = async { videoApi.getManifestUrl(videoId) }
    // Both fetches run in parallel — saves 200-400ms vs sequential
    val metadata = metadataDeferred.await()
    val manifestUrl = manifestDeferred.await()
    player.setMediaItem(MediaItem.fromUri(manifestUrl))
    player.prepare()
}

// 2. Prefetch manifest when hovering/long-pressing a thumbnail
// Start ExoPlayer.prepare() before user taps — buffer already filling
// If user doesn't tap: cancel (ExoPlayer.stop())

// 3. ExoPlayer init at app start (singleton) — not per-video
// Creating ExoPlayer is ~50ms — do it once at startup

// 4. Shorter initial buffer — start playback after 2.5s of buffer
// bufferForPlayback = 2_500ms (vs default 2500ms — already good)
// After starting: let buffer fill to 50s (maxBuffer) in background

3. Background Playback — Audio Continues When Screen Off

// Required for YouTube Premium-style background play
// When app goes to background: two strategies

// Strategy A: Foreground Service with MediaSession
// Use when: audio-only background play is a feature
// ExoPlayer 2.x + media3: use MediaSessionService
class VideoPlaybackService : MediaSessionService() {
    private lateinit var player: ExoPlayer
    private lateinit var mediaSession: MediaSession

    override fun onCreate() {
        super.onCreate()
        player = ExoPlayer.Builder(this).build()
        mediaSession = MediaSession.Builder(this, player).build()
        // MediaSessionService handles foreground service + notification automatically
        // Notification shows: play/pause, skip, album art — lockscreen controls
    }

    override fun onGetSession(controllerInfo: MediaSession.ControllerInfo) = mediaSession
    override fun onDestroy() { player.release(); mediaSession.release(); super.onDestroy() }
}
// Manifest: <service android:foregroundServiceType="mediaPlayback"/>

// Strategy B: Pause when backgrounded (simpler — if background play is not a feature)
// ProcessLifecycleOwner.get().lifecycle detects app-level background
val observer = LifecycleEventObserver { _, event ->
    when (event) {
        Lifecycle.Event.ON_STOP -> player.playWhenReady = false
        Lifecycle.Event.ON_START -> player.playWhenReady = true
        else -> {}
    }
}

4. Picture-in-Picture (PiP)

// PiP: user can minimize the player to a floating window
// API 26+ for basic PiP; API 31+ for seamless PiP transitions

// Enter PiP on home button press (automatic on API 31+)
override fun onUserLeaveHint() {
    if (viewModel.state.value.isPlaying) {
        enterPipMode()
    }
}

private fun enterPipMode() {
    val params = PictureInPictureParams.Builder()
        .setAspectRatio(Rational(16, 9))
        .setActions(listOf(
            buildPipAction(R.drawable.ic_pause, "Pause", PIP_ACTION_PAUSE),
            buildPipAction(R.drawable.ic_forward, "Forward 10s", PIP_ACTION_FORWARD)
        ))
        .build()
    enterPictureInPictureMode(params)
}

// In Compose: detect PiP mode and hide controls
@Composable
fun PlayerScreen(state: PlayerUiState) {
    if (!state.isPipMode) {
        PlayerControls(state) // hide controls in PiP
    }
    PlayerSurface(player = player) // SurfaceView stays in both modes
}

5. Offline Download

// WorkManager + ExoPlayer Download Manager
// ExoPlayer has a built-in DownloadManager for DASH/HLS segment downloads
val downloadManager = DownloadManager(
    context,
    DatabaseProvider(context),
    SimpleCache(downloadDirectory, NoOpCacheEvictor(), secretKey),
    buildDataSourceFactory()
)

// Download a specific quality — user chooses (360p for save space, 1080p for quality)
val downloadRequest = DownloadRequest.Builder(videoId, manifestUri)
    .setStreamKeys(listOf(StreamKey(0, 1))) // select 720p quality stream
    .setCustomCacheKey(videoId)
    .build()
downloadManager.addDownload(downloadRequest)

// Monitor download progress via Flow
downloadManager.downloadIndex.getDownloads().forEach { download ->
    when (download.state) {
        Download.STATE_DOWNLOADING -> updateProgress(download.percentDownloaded)
        Download.STATE_COMPLETED -> markAvailableOffline(download.request.id)
        Download.STATE_FAILED -> notifyDownloadFailed(download.request.id)
    }
}

// Play downloaded content
val offlineMediaSource = ProgressiveMediaSource.Factory(
    CacheDataSource.Factory()
        .setCache(downloadCache)
        .setUpstreamDataSourceFactory(null) // no network fallback in offline mode
        .setCacheWriteDataSinkFactory(null)  // read-only
).createMediaSource(MediaItem.fromUri(localUri))

Staff+ Layer: Reliability & Rollout 6–8 min

Complete Staff+ Narration — Raise All of These Unprompted
Failure modes I'd proactively design for:

"First, network degradation mid-playback. ExoPlayer's ABR handles this gracefully — it drops to lower quality. But if the network drops completely, ExoPlayer will stall and show a buffering state. I'd set a maximum stall timeout — if the player has been buffering for more than 8 seconds, I'd show a 'Retry' button and an error message, rather than spinning indefinitely.

Second, codec unavailability. Not all devices support all codecs for all quality levels. I'd use ExoPlayer's DefaultRenderersFactory with EXTENSION_RENDERER_MODE_PREFER to prefer software decoders as fallback. For VP9/AV1, I'd detect codec support at startup and pass the result to the ABR algorithm — don't select a quality level the device can't decode in real-time.

Third, audio focus loss. Phone call comes in — ExoPlayer must pause. I've configured setAudioAttributes(audioAttributes, handleAudioFocus = true) so ExoPlayer handles this automatically. It also ducks volume for navigation audio and resumes when focus is restored.

Observability:
I'd instrument three key metrics: time-to-first-frame (target <2s on WiFi), rebuffer rate (% of sessions with >1 stall), and quality distribution (% of viewing time at each quality level). These are the core 'quality of experience' metrics. I'd alert on rebuffer rate >5% — that means either our CDN is degraded or our ABR algorithm is too aggressive for current network conditions.

Rollout:
The video player is the most user-visible feature in the app. I'd ship it behind a feature flag, rolling out at 1% → 5% → 20% → 100% with 24-hour holds at each gate. The specific metric I'd watch during rollout is rebuffer rate — if it's significantly higher on the new player than the old one, I'd halt and investigate before proceeding. The kill switch is the feature flag — falls back to the previous player implementation instantly, no release needed."

Questions to Expect as Follow-Ups

  • "How would you handle a device that doesn't support the video codec?" — Detect codec support at startup via MediaCodecUtil, exclude unsupported representations from the track selector, fall back to a universally-supported codec (H.264 baseline is safe for all Android 4.1+ devices).
  • "How do you handle the player when the device is low on memory?" — ComponentCallbacks2.onTrimMemory() signals memory pressure. On TRIM_MEMORY_RUNNING_CRITICAL, pause download, reduce buffer targets, release the video cache. ExoPlayer itself handles this via DefaultLoadControl min/max buffer settings.
  • "What happens when the user scrubs to an un-buffered position?" — ExoPlayer seeks to the nearest keyframe within the cached range (fast, no rebuffer), then fetches segments around the new position in background. If the target position has no cached segments, it shows buffering state while fetching. This is called a "discontinuity" — ExoPlayer fires onPlaybackStateChanged(STATE_BUFFERING) and we update the UI accordingly.
  • "How do you handle DRM-protected content?" — ExoPlayer supports Widevine DRM (L1 on most modern devices, L3 as fallback). The license request goes to a DRM license server URL embedded in the DASH manifest. Key: Widevine L1 content (1080p+ protected) requires a hardware-backed DRM implementation — not all devices support it. Detect and gracefully degrade to L3 (720p limit).
