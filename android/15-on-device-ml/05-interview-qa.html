Interview Q&A — On-Device ML

20 questions from Google, Samsung, and Flipkart engineering rounds on on-device ML

Section A — TFLite Core

1
Why use TFLite instead of running a model with the full TensorFlow runtime on Android? GoogleL5

The full TensorFlow runtime is ~300MB and requires significant CPU and RAM to initialize. TFLite is a purpose-built inference-only runtime: ~1MB binary, loads in milliseconds, uses a flat FlatBuffer format that can be memory-mapped directly from disk without parsing.

TFLite also has delegate support (GPU, NNAPI, Hexagon) that the full TF runtime lacks. It supports INT8 quantized models natively, which are 4× smaller and 2–4× faster on mobile hardware. The full TF runtime is designed for training — TFLite is designed purely for inference.

2
Explain why the TFLite Interpreter is not thread-safe, and show the correct pattern for concurrent inference. GoogleL6

The Interpreter holds internal state — tensor buffers, graph execution state, delegate context. If two threads call run() simultaneously, they corrupt each other's state — wrong results or crashes.

Pattern A — Single thread (simplest): One Interpreter, one dedicated Executors.newSingleThreadExecutor(). All inference calls are serialized on that thread. Works well when inference rate is manageable.

Pattern B — Interpreter pool (for high throughput):

class InterpreterPool(model: MappedByteBuffer, poolSize: Int = 3) {
    private val pool = ArrayBlockingQueue<Interpreter>(poolSize)

    init {
        repeat(poolSize) {
            pool.offer(Interpreter(model, Interpreter.Options().apply { numThreads = 2 }))
        }
    }

    suspend fun <T> runInference(block: (Interpreter) -> T): T {
        val interpreter = withContext(Dispatchers.IO) { pool.take() }
        return try { block(interpreter) } finally { pool.offer(interpreter) }
    }
}

Each thread gets its own Interpreter from the pool. The pool limits concurrency to avoid memory pressure. GPU delegate cannot use this pattern — it needs a single fixed thread.

3
What is the difference between dynamic range quantization, full INT8 quantization, and QAT? When would you use each? GoogleL6

Dynamic range quantization: Weights are converted to INT8 at conversion time. Activations remain FP32 at runtime (dynamically quantized on each inference). No calibration data needed. 4× smaller file, ~2× faster on CPU. Use when: you need quick results, no representative dataset available, or model accuracy is critical.

Full INT8 quantization: Both weights AND activations are INT8. Requires a representative dataset (100+ samples) to calibrate the scale factors. Enables Hexagon delegate and most NPUs. Use when: you need maximum speed on DSP/NPU, or deploying to Snapdragon devices with Hexagon.

QAT (Quantization-Aware Training): Fake quantization nodes inserted during training. Model learns to compensate for quantization noise. Best accuracy at INT8 — accuracy loss typically <0.5% vs FP32. Use when: accuracy is paramount and you have training infrastructure. Most effort, best result.

4
What is a representative dataset and why does full INT8 quantization require one? GoogleL6

Full INT8 quantization must map each layer's FP32 activation range to the INT8 range [-128, 127]. To find the correct scale factor (e.g., 1 INT8 unit = 0.023 FP32), the converter needs to observe the actual min/max values that each activation takes during inference — this is called calibration.

The representative dataset provides ~100–500 real input samples that the converter runs through the model to collect these statistics. Using random noise gives wrong scale factors (noise has different statistics than real images) — the model loses 5–20% accuracy. Using real production data gives accurate scale factors and minimal accuracy loss (<1%).

Rule: the representative dataset should be diverse and representative of your actual use case distribution — same lighting, subjects, and conditions as production data.

Section B — Delegates & Performance

5
You deploy a TFLite model with the GPU delegate. On some devices it's fast; on others it's slower than CPU. Why? SamsungL6

Several reasons:

  • Unsupported ops: If the model has ops the GPU delegate doesn't support, those ops fall back to CPU. Each CPU↔GPU data transfer (tensor copy) adds latency. A model with 70% of ops on GPU and 30% on CPU with frequent transfers can be slower than pure CPU.
  • GPU warm-up: The GPU delegate compiles OpenGL/Vulkan shaders on the first run (~100–500ms). On devices with slow GPUs, this initialization cost dominates short inference sessions.
  • Small models on powerful CPUs: MobileNet-tiny on a Snapdragon 8 Gen 3 CPU with 4 fast cores may be faster than GPU dispatch overhead for that model size.
  • GPU is busy: If the GPU is saturated with rendering (e.g., heavy game), inference latency spikes.

Fix: always benchmark on target device diversity before choosing a delegate. Use the delegate fallback chain — measure each and pick the fastest for your specific model + device combination.

6
What is the Hexagon delegate and when should you use it? SamsungL6

The Hexagon delegate offloads inference to Qualcomm's Hexagon DSP (Digital Signal Processor) — a fixed-function processor optimized for INT8 matrix math. Available on Snapdragon 835 and later.

When to use: Your model is fully INT8 quantized (mandatory requirement), you're targeting Snapdragon devices (Snapdragon 8 series, mid-range 7xx series), and you need maximum power efficiency — the DSP is 3–5× more power-efficient than the CPU for INT8 inference.

Advantages: 2–4× faster than CPU INT8, 3–5× better power efficiency, doesn't compete with GPU for rendering bandwidth.

Limitations: Snapdragon only (not available on Exynos, MediaTek, or Tensor). All ops must be INT8 — any FP32 op causes the entire model to fall back to CPU. Requires libhexagon_interface.so in the APK.

7
Your real-time camera ML feature runs at 30fps initially but degrades to 15fps after 2 minutes. Diagnose and fix. GoogleL6

Root cause: Thermal throttling. Sustained CPU/GPU inference heats the SoC. The thermal governor reduces clock frequencies to prevent hardware damage — inference latency doubles.

Diagnosis: Log SystemClock.elapsedRealtimeNanos() per inference frame. If latency grows monotonically over 2 minutes, it's thermal. Confirm with adb shell cat /sys/class/thermal/thermal_zone*/temp.

Fixes:

  • Switch from CPU to GPU or NNAPI delegate — more power-efficient, generates less heat.
  • Reduce inference rate: skip frames. For a 30fps camera stream, run inference every 3rd frame (10fps) if 30fps isn't necessary for the feature.
  • Use EXECUTION_PREFERENCE_LOW_POWER in NNAPI for sustained operation.
  • Set CameraX STRATEGY_KEEP_ONLY_LATEST — drop frames when inference is slow, don't queue them (queue causes latency growth and additional CPU work).
  • For camera features: use a smaller model (MobileNet vs EfficientNet). A model 2× smaller is often 3× faster after thermal throttling.

Section C — ML Kit

8
When would you choose ML Kit over raw TFLite for a text recognition feature? GoogleL5

Choose ML Kit when: you need text recognition in a standard supported script (Latin, Devanagari, Chinese, etc.), you don't need to customize the model, and you want to ship fast without managing model files or pre-processing code. ML Kit handles camera rotation correction, input scaling, and post-processing automatically. Supports 50+ scripts out of the box.

Choose raw TFLite when: you have a custom fine-tuned model (e.g., specialized for handwritten receipts in a specific font), you need to run a model ML Kit doesn't support, you need more control over pre/post-processing, or you need to minimize APK size by using a single TFLite runtime rather than both TFLite and the ML Kit abstraction layer.

9
You're implementing a barcode scanner. The camera freezes after scanning a few barcodes. What's wrong? FlipkartL5

The ImageProxy is not being closed. In ImageAnalysis.Analyzer, the camera pipeline delivers a fixed pool of image buffers. When you forget to call imageProxy.close() — especially in error paths — the buffer is never returned to the pool. After a few frames, all buffers are exhausted and the camera stops delivering new frames.

Fix: always close in addOnCompleteListener, not in addOnSuccessListener:

scanner.process(inputImage)
    .addOnSuccessListener { barcodes -> handleBarcodes(barcodes) }
    .addOnFailureListener { e -> Log.e("Scan", "Error", e) }
    .addOnCompleteListener { imageProxy.close() }  // ALWAYS called
10
What is STREAM_MODE vs SINGLE_IMAGE_MODE in ML Kit object detection, and when do you use each? GoogleL5

STREAM_MODE: Optimized for continuous video frames. Trades some accuracy for lower latency. Provides a stable trackingId — the same physical object gets the same ID across frames. Use for: real-time camera features, AR overlays, continuous object tracking.

SINGLE_IMAGE_MODE: Higher accuracy, processes each image independently (no inter-frame context). No tracking ID. Slower — more compute per frame. Use for: single photos (tap to analyze), gallery image processing, when accuracy matters more than real-time performance.

The key differentiator is trackingId. If you need to track "this specific object" as it moves across the frame (e.g., user taps an object and you highlight it), use STREAM_MODE. If you just want to know what objects are in a photo, use SINGLE_IMAGE_MODE.

Section D — System Design

11
Design a visual search feature for Flipkart: user points the camera at a product, app shows similar products. FlipkartDesignL6

On-device pipeline:

  1. CameraX ImageAnalysis at 10fps (not 30fps — thermal management).
  2. ML Kit Object Detection (STREAM_MODE) — detect and crop the main product in the frame.
  3. TFLite embedding model (MobileNetV3 fine-tuned on product images) — convert cropped product to a 128-dim embedding vector. Run with GPU delegate.
  4. When user taps "Search" — take the embedding, send to backend.

Backend:

  1. Receive embedding vector (128 floats = 512 bytes).
  2. ANN (Approximate Nearest Neighbor) search in product embedding index (FAISS / ScaNN).
  3. Return top-10 similar product IDs with similarity scores.

Why this split: Embedding extraction on-device avoids sending raw images (privacy, bandwidth). The embedding is tiny (512 bytes vs ~200KB JPEG). The ANN index (millions of products) lives on the server — too large for device.

Model update strategy: Embed via Firebase ML — push updated embedding model without app update when catalog changes significantly.

12
How would you implement smart auto-categorization of expenses in a fintech app — OCR the receipt, extract amount and merchant, categorize — all on-device? DesignL6

Pipeline:

  1. OCR: ML Kit Text Recognition (bundled, Latin script). Process receipt photo → extract full text string.
  2. Entity extraction: ML Kit Entity Extraction API — identifies money amounts, dates, addresses from text. Or a custom regex + NLP model for merchant name extraction.
  3. Categorization: Small TFLite text classification model (fine-tuned MobileBERT or DistilBERT quantized to INT8, ~15MB). Input: merchant name + description text. Output: category probabilities (food, travel, utilities, etc.).

Why fully on-device: Financial data is highly sensitive — sending receipt photos or text to a server creates privacy and regulatory risk. On-device keeps data local, works offline (important for travel receipts), and is faster (no network RTT).

Model deployment: Bundle categorization model in APK. Update via Firebase ML when category list expands. Always keep a bundled fallback for offline first-launch.

Section E — Quick-Fire

13
What is knowledge distillation's "temperature" parameter?

Temperature T softens the teacher model's output probability distribution before computing KL divergence loss. Without T, the teacher outputs near-one-hot predictions — the student learns nothing from the soft labels. Higher T (e.g., 4–10) makes the distribution softer, encoding more relational information between classes. T is only used during training; the student is deployed at T=1.

14
Why must you rewind a ByteBuffer before each inference?

ByteBuffer.rewind() resets the position to 0. After you write pixel data into it, the position is at the end. If you don't rewind before passing to interpreter.run(), TFLite reads from position=end, sees no data, and produces wrong/zero output. Always call rewind() after filling and before running.

15
What is the CameraX backpressure strategy STRATEGY_KEEP_ONLY_LATEST and why is it critical for ML?

It tells CameraX to drop incoming frames if the analyzer is still processing the previous one — keeping only the most recent frame in the queue. Without it (STRATEGY_BLOCK_PRODUCER), the camera blocks and frames pile up in memory. This causes: growing latency (analyzing 3-second-old frames), OOM from buffered frames, and eventually ANR if the camera thread stalls. For ML inference where latency matters, always use KEEP_ONLY_LATEST.

16
What is the practical size limit for a bundled TFLite model in an APK?

No hard technical limit, but practical limits apply: Google Play recommends keeping download size under 150MB total. A bundled model should realistically be under 20–30MB to avoid user complaints about app size. Use INT8 quantization to achieve this — a FP32 MobileNetV2 is 14MB, INT8 version is 3.5MB. For larger models (>10MB), consider Firebase ML downloadable delivery so the model is fetched post-install on WiFi.

17
Name two differences between ML Kit bundled and downloadable models.
  • APK size: Bundled adds model size to APK. Downloadable keeps APK lean; model is fetched from Play Services on first use.
  • Availability: Bundled works immediately on install, offline. Downloadable may not be ready on first launch — must handle "model not yet downloaded" state gracefully.
18
What does setPrecisionLossAllowed(true) do on the GPU delegate?

Allows the GPU delegate to use FP16 (16-bit floating point) arithmetic instead of FP32 where supported. Most mobile GPUs have 2× throughput for FP16 vs FP32. The accuracy impact is negligible for image classification and detection models (typically <0.5% top-1 accuracy loss). Recommended to enable unless your model has unusual numerical sensitivity. Without it, the GPU forces FP32, roughly halving inference speed.

19
How do you update a TFLite model shipped in the APK without a Play Store release?

Use Firebase ML: host the new .tflite file in Firebase, call FirebaseModelDownloader.getModel() with DownloadType.LOCAL_MODEL_UPDATE_IN_BACKGROUND. The SDK downloads the new model in the background (on WiFi if configured). Next launch, the app uses the updated model. Always keep the original bundled model as a fallback for devices that haven't downloaded the update yet.

20
What is NHWC format and why does TFLite use it for image tensors?

NHWC = Batch × Height × Width × Channels. A single 224×224 RGB image tensor has shape [1, 224, 224, 3]. The "Channels last" layout means all 3 color values for a pixel are contiguous in memory — efficient for convolution on ARM CPUs (NEON SIMD processes 4 channels at once). TFLite defaults to NHWC. Some GPU delegates prefer NCHW (Channels first) internally and transpose automatically. When filling your input ByteBuffer, always fill in NHWC order: for each pixel, write R then G then B.
