Android Integration

viewModelScope, lifecycleScope, repeatOnLifecycle, lifecycle-aware Flow collection, Compose coroutine APIs, and the patterns that tie coroutines to the Android lifecycle

1. viewModelScope

Provided by androidx.lifecycle:lifecycle-viewmodel-ktx. Automatically cancelled when the ViewModel is cleared (onCleared()).

class UserViewModel(private val repo: UserRepository) : ViewModel() {

    private val _state = MutableStateFlow<UiState>(UiState.Loading)
    val state: StateFlow<UiState> = _state.asStateFlow()

    fun loadUser(id: String) {
        viewModelScope.launch {
            _state.value = UiState.Loading
            try {
                val user = repo.getUser(id) // suspend call
                _state.value = UiState.Success(user)
            } catch (e: Exception) {
                _state.value = UiState.Error(e.message ?: "Unknown error")
            }
        }
    }
}

Under the Hood

// From AOSP — simplified
public val ViewModel.viewModelScope: CoroutineScope
    get() {
        val scope = getTag<CloseableCoroutineScope>(JOB_KEY)
        if (scope != null) return scope
        return setTagIfAbsent(JOB_KEY,
            CloseableCoroutineScope(
                SupervisorJob() + Dispatchers.Main.immediate
            )
        )
    }
// Key facts:
// 1. SupervisorJob — one child failing doesn't cancel siblings
// 2. Dispatchers.Main.immediate — no re-dispatch if already on main thread
// 3. Stored as a ViewModel tag — survives configuration changes
// 4. Cancelled in ViewModel.clear() → all children cancelled
Staff-Level Insight — Why SupervisorJob + Main.immediate

SupervisorJob because a ViewModel typically launches multiple independent operations (load user, load posts, load settings). If loading posts fails, you don't want to cancel the user load. Main.immediate because most ViewModel work ends with a UI state update — you want that update to happen inline without an extra message queue hop. If you need IO or computation, use withContext(Dispatchers.IO) inside the launch block.

2. lifecycleScope

Provided by androidx.lifecycle:lifecycle-runtime-ktx. Tied to a LifecycleOwner (Activity, Fragment). Cancelled when the lifecycle reaches DESTROYED.

class UserActivity : AppCompatActivity() {

    override fun onCreate(savedInstanceState: Bundle?) {
        super.onCreate(savedInstanceState)

        // Launched in lifecycleScope — cancelled when Activity is destroyed
        lifecycleScope.launch {
            val data = fetchData() // This work dies with the Activity
            updateUI(data)
        }
    }
}
lifecycleScope Survives Config Changes — But Shouldn't

lifecycleScope is tied to the Activity/Fragment instance. On configuration change (rotation), the old instance is destroyed → scope cancelled → work lost. The new instance gets a new scope. This is why data-fetching should happen in viewModelScope (survives config changes), not lifecycleScope. Use lifecycleScope only for UI-specific work like animations, one-time navigation, or collecting flows.

3. repeatOnLifecycle — The Correct Way to Collect Flows

This is the single most important Android coroutine API to understand for interviews:

The Problem

// BAD — keeps collecting even when app is in background
lifecycleScope.launch {
    viewModel.locationFlow.collect { location ->
        updateMap(location) // Wasting battery + GPS when app is backgrounded
    }
}

// BAD — launchWhenStarted suspends but doesn't cancel upstream
lifecycleScope.launchWhenStarted { // DEPRECATED
    viewModel.locationFlow.collect { location ->
        updateMap(location)
        // When lifecycle goes below STARTED, collection is SUSPENDED but the
        // upstream Flow keeps emitting into a buffer → memory leak
    }
}

The Solution

// GOOD — repeatOnLifecycle restarts collection when lifecycle enters state,
// CANCELS it when lifecycle drops below state
lifecycleScope.launch {
    repeatOnLifecycle(Lifecycle.State.STARTED) {
        // This block is executed when STARTED, cancelled when STOPPED
        // Re-executed when STARTED again (e.g., returning from background)
        viewModel.locationFlow.collect { location ->
            updateMap(location)
        }
    }
}

// Multiple parallel collections
lifecycleScope.launch {
    repeatOnLifecycle(Lifecycle.State.STARTED) {
        launch { viewModel.uiState.collect { renderState(it) } }
        launch { viewModel.events.collect { handleEvent(it) } }
    }
}
Staff-Level Insight — Why launchWhenStarted Was Deprecated

launchWhenStarted / launchWhenResumed suspend the coroutine when the lifecycle drops below the target state, but they don't cancel the upstream Flow. For cold flows, the producer keeps running and buffering values. For hot flows like location updates, the upstream keeps emitting into a buffer, wasting resources. repeatOnLifecycle cancels the entire block, which cancels the Flow collection, which signals the upstream to stop. This is the correct behavior for lifecycle-aware collection.

4. flowWithLifecycle — Single Flow Convenience

// When you only need to collect a single flow, this is more concise:
lifecycleScope.launch {
    viewModel.uiState
        .flowWithLifecycle(lifecycle, Lifecycle.State.STARTED)
        .collect { state -> renderState(state) }
}

// Equivalent to:
lifecycleScope.launch {
    repeatOnLifecycle(Lifecycle.State.STARTED) {
        viewModel.uiState.collect { state -> renderState(state) }
    }
}

// For multiple flows, repeatOnLifecycle is cleaner (avoids nested launches)

5. Compose + Coroutines

collectAsState / collectAsStateWithLifecycle

@Composable
fun UserScreen(viewModel: UserViewModel) {
    // collectAsStateWithLifecycle — lifecycle-aware (recommended)
    val state by viewModel.uiState.collectAsStateWithLifecycle()

    // collectAsState — NOT lifecycle-aware (keeps collecting in background)
    // val state by viewModel.uiState.collectAsState()

    when (val s = state) {
        is UiState.Loading -> CircularProgressIndicator()
        is UiState.Success -> UserContent(s.user)
        is UiState.Error -> ErrorMessage(s.message)
    }
}
Always Use collectAsStateWithLifecycle in Android

collectAsState() collects the flow for the entire lifetime of the composition — even when the app is in the background. collectAsStateWithLifecycle() (from androidx.lifecycle:lifecycle-runtime-compose) stops collection when the lifecycle drops below STARTED, matching the behavior of repeatOnLifecycle. This is the recommended API for all Android Compose code.

LaunchedEffect — Side Effects in Compose

@Composable
fun SearchScreen(viewModel: SearchViewModel) {
    val query by viewModel.query.collectAsStateWithLifecycle()

    // LaunchedEffect launches a coroutine scoped to the composition
    // Re-launches when the key changes (cancels previous)
    LaunchedEffect(query) {
        delay(300) // Debounce
        viewModel.search(query)
    }

    // LaunchedEffect(Unit) — runs once when composable enters composition
    LaunchedEffect(Unit) {
        viewModel.events.collect { event ->
            when (event) {
                is UiEvent.ShowSnackbar -> snackbarHostState.showSnackbar(event.message)
            }
        }
    }
}

rememberCoroutineScope — Event Handlers

@Composable
fun SettingsScreen() {
    // For launching coroutines from event handlers (onClick, etc.)
    val scope = rememberCoroutineScope()
    val snackbarHostState = remember { SnackbarHostState() }

    Button(onClick = {
        scope.launch {
            // This scope is tied to the composition
            // Cancelled when SettingsScreen leaves composition
            snackbarHostState.showSnackbar("Settings saved!")
        }
    }) {
        Text("Save")
    }
}

produceState — Convert Suspend to State

@Composable
fun UserAvatar(userId: String) {
    // produceState launches a coroutine and produces a State
    val avatar by produceState<Bitmap?>(initialValue = null, userId) {
        value = withContext(Dispatchers.IO) {
            imageLoader.loadAvatar(userId)
        }
    }

    avatar?.let { Image(bitmap = it.asImageBitmap(), contentDescription = null) }
        ?: CircularProgressIndicator()
}

Compose Coroutine API Summary

APIWhen to UseScope Lifetime
LaunchedEffect(key) One-time or key-dependent side effects (API calls, event collection) Composition. Relaunches when key changes.
rememberCoroutineScope() Launching coroutines from callbacks (onClick, onScroll) Composition. Cancelled when composable leaves.
produceState(initial, key) Converting suspend functions to State<T> Composition. Relaunches when key changes.
collectAsStateWithLifecycle() Collecting Flow as Compose State (lifecycle-aware) Lifecycle-aware (STARTED by default).
snapshotFlow { } Converting Compose State reads into a Flow Depends on where collected.
Staff-Level Insight — snapshotFlow for Compose → Flow Bridge

snapshotFlow reads Compose snapshot state and emits whenever the read values change. This is the bridge from Compose's state system to the Flow world:

LaunchedEffect(listState) {
    snapshotFlow { listState.firstVisibleItemIndex }
        .distinctUntilChanged()
        .filter { it > 10 }
        .collect { viewModel.onScrolledPastThreshold() }
}

This is how you implement "load more when scrolled to bottom" or "show/hide FAB based on scroll position" reactively in Compose.

6. Coroutines in Services and WorkManager

LifecycleService

class SyncService : LifecycleService() {
    // LifecycleService provides lifecycleScope
    override fun onStartCommand(intent: Intent?, flags: Int, startId: Int): Int {
        super.onStartCommand(intent, flags, startId)
        lifecycleScope.launch {
            performSync()
            stopSelf(startId)
        }
        return START_NOT_STICKY
    }
}

CoroutineWorker

class SyncWorker(
    context: Context,
    params: WorkerParameters
) : CoroutineWorker(context, params) {

    // doWork() runs on Dispatchers.Default by default
    override suspend fun doWork(): Result {
        return try {
            val data = api.fetchData()
            database.save(data)
            Result.success()
        } catch (e: Exception) {
            if (runAttemptCount < 3) Result.retry()
            else Result.failure()
        }
    }
}

7. Testing Coroutines

// Use kotlinx-coroutines-test
@Test
fun `loadUser updates state`() = runTest {
    // runTest uses TestCoroutineScheduler — virtual time
    // delay() advances instantly, no real waiting

    val repo = FakeUserRepository()
    val viewModel = UserViewModel(repo)

    viewModel.loadUser("123")

    // advanceUntilIdle() processes all pending coroutines
    advanceUntilIdle()

    assertEquals(UiState.Success(fakeUser), viewModel.state.value)
}

// Testing Flows
@Test
fun `state flow emits loading then success`() = runTest {
    val viewModel = UserViewModel(FakeRepo())

    val states = mutableListOf<UiState>()
    val job = launch(UnconfinedTestDispatcher(testScheduler)) {
        viewModel.state.toList(states)
    }

    viewModel.loadUser("123")
    advanceUntilIdle()

    assertEquals(listOf(UiState.Loading, UiState.Success(fakeUser)), states)
    job.cancel()
}

// Turbine — preferred Flow testing library
@Test
fun `state flow with Turbine`() = runTest {
    val viewModel = UserViewModel(FakeRepo())

    viewModel.state.test {
        assertEquals(UiState.Loading, awaitItem())
        viewModel.loadUser("123")
        assertEquals(UiState.Success(fakeUser), awaitItem())
        cancelAndIgnoreRemainingEvents()
    }
}

TestDispatcher Types

DispatcherBehaviorUse For
StandardTestDispatcher Queues coroutines. Must call advanceUntilIdle() to execute. Precise control over execution order. Default in runTest.
UnconfinedTestDispatcher Executes coroutines eagerly (immediately on launch). Collecting flows in tests, when you want immediate execution.
Staff-Level Insight — Injecting Dispatchers for Testability

Hardcoding Dispatchers.IO or Dispatchers.Main in your code makes it untestable. The L6 pattern is to inject dispatchers:

class UserRepository(
    private val api: UserApi,
    private val ioDispatcher: CoroutineDispatcher = Dispatchers.IO
) {
    suspend fun getUser(id: String) = withContext(ioDispatcher) {
        api.fetchUser(id)
    }
}

// In tests:
val repo = UserRepository(fakeApi, testDispatcher)

This also applies to viewModelScope. In Lifecycle 2.8+, you can override the default dispatcher via Dispatchers.setMain(testDispatcher) in tests, or use the viewModelScope constructor parameter.

8. Performance Profiling for Coroutines

Systrace / Perfetto

// Add custom trace sections around coroutine work
import android.os.Trace

suspend fun fetchAndProcess() {
    Trace.beginSection("fetchAndProcess")
    try {
        val data = withContext(Dispatchers.IO) {
            Trace.beginSection("network-fetch")
            try { api.fetch() } finally { Trace.endSection() }
        }
        withContext(Dispatchers.Default) {
            Trace.beginSection("heavy-parse")
            try { parse(data) } finally { Trace.endSection() }
        }
    } finally {
        Trace.endSection()
    }
}

// Capture: adb shell perfetto -o /data/misc/perfetto-traces/trace -t 10s \
//   -c - --txt <

Android Studio Profiler

  • CPU Profiler: Shows thread activity. Coroutines appear as work on DefaultDispatcher-worker-N threads. Look for main-thread work that should be on IO/Default.
  • Thread states: RUNNING (doing work), SLEEPING (parked/waiting), WAITING (blocked on lock). Too many threads in WAITING = lock contention. Too many in RUNNING on main = jank source.
  • Method tracing: Sample or instrument to find which suspend functions take the longest. Suspension points appear as gaps in the thread timeline.
Staff-Level Insight — What to Look For

The three coroutine performance anti-patterns visible in profiling:

  1. Main thread stalls: Dispatchers.Main coroutines doing computation or blocking I/O. Fix: withContext(Dispatchers.Default/IO).
  2. Thread pool exhaustion: All DefaultDispatcher-worker-* threads in WAITING state. Cause: synchronized or blocking calls inside coroutines on Default. Fix: use Mutex or move blocking calls to Dispatchers.IO.
  3. Excessive context switching: Too many withContext calls causing thread hops visible as tiny gaps in the timeline. Fix: batch work in a single withContext block instead of switching per operation.

9. RxJava → Coroutines Migration

Many production Android codebases are migrating from RxJava to coroutines. Here's the mapping:

Bridge APIs for Incremental Migration

// kotlinx-coroutines-rx3 (or rx2) dependency

// RxJava → Coroutines
val flow: Flow<Data> = observable.asFlow()
val result: Data = single.await()
val maybeResult: Data? = maybe.awaitOrNull()
completable.await()

// Coroutines → RxJava (for legacy consumers)
val observable: Observable<Data> = flow.asObservable(scope)
val single: Single<Data> = scope.rxSingle { suspendFunction() }
val completable: Completable = scope.rxCompletable { suspendFunction() }
Staff-Level Insight — Migration Strategy

The recommended migration path at Google scale:

  1. Bottom-up: Start at the data layer (repositories, data sources). Convert Single to suspend fun, Observable to Flow.
  2. Bridge at boundaries: Use .asFlow() / .asObservable() at the boundary between migrated and unmigrated code.
  3. ViewModel last: Convert ViewModel subscriptions from CompositeDisposable to viewModelScope.launch + collect.
  4. Remove RxJava dependency only when all usages are migrated.

Why bottom-up: data layer changes are isolated and testable. ViewModel changes affect UI and are harder to test incrementally. Never mix RxJava and coroutines in the same class if you can avoid it — the mental overhead of two concurrency models is a maintenance nightmare.
