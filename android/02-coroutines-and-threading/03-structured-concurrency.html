Structured Concurrency

The single most important concept in Kotlin coroutines — scopes, the Job tree, cancellation propagation, and exception handling that actually makes sense

1. What Is Structured Concurrency?

Structured concurrency is the principle that a coroutine's lifetime is bounded by its scope. Every coroutine has a parent, and:

  • A parent waits for all children to complete before completing itself
  • Cancelling a parent cancels all children
  • An uncaught exception in a child cancels the parent (and therefore all siblings)

This is the opposite of unstructured concurrency (raw threads, Executors) where tasks are fire-and-forget with no lifecycle relationship.

2. CoroutineScope

A CoroutineScope is just a wrapper around a CoroutineContext. Its purpose is to be the owner of coroutines:

public interface CoroutineScope {
    public val coroutineContext: CoroutineContext
}

// Creating a scope
val myScope = CoroutineScope(SupervisorJob() + Dispatchers.Main)

// Launching in a scope
myScope.launch { /* child coroutine */ }

// Cancelling the scope cancels all children
myScope.cancel()

coroutineScope {} — The Scoping Function

suspend fun fetchUserWithPosts(userId: String): UserWithPosts {
    // coroutineScope creates a NEW scope that inherits the parent's context
    // It WAITS for all children to complete before returning
    return coroutineScope {
        val user = async { userApi.getUser(userId) }
        val posts = async { postApi.getPosts(userId) }
        UserWithPosts(user.await(), posts.await())
    }
    // If either async throws, the other is cancelled automatically
}
Staff-Level Insight — coroutineScope vs CoroutineScope()

These are completely different things despite the similar names:

  • coroutineScope { } — a suspend function that creates a child scope, waits for all children, and propagates exceptions. Used inside suspend functions for parallel decomposition.
  • CoroutineScope(context) — a constructor that creates a standalone scope. Used to create top-level scopes (like in a ViewModel or Repository). You're responsible for calling cancel().

Using CoroutineScope() inside a suspend function is a code smell — it breaks structured concurrency because the new scope isn't a child of the caller.

4. Cancellation

Cancellation in coroutines is cooperative. Calling job.cancel() doesn't forcefully stop the coroutine — it sets a flag that the coroutine must check.

Cancellation Propagation — The Complete Picture

How Cancellation Works

  1. job.cancel() sets the Job's state to "cancelling" and installs a CancellationException as the cause
  2. All child Jobs are notified recursively — their state also becomes "cancelling"
  3. At the next suspension point, each coroutine checks isActive and throws CancellationException
  4. All kotlinx.coroutines suspend functions (delay, yield, withContext, await) are cancellation-aware — they check automatically
  5. CPU-bound loops must check manually
  6. Once all children complete (finish cancelling), the parent Job transitions from CANCELLING → CANCELLED
// BAD — not cancellable (no suspension points)
suspend fun processLargeList(items: List<Item>) {
    for (item in items) {
        heavyComputation(item) // Never suspends → never checks cancellation
    }
}

// GOOD — check isActive or use yield()
suspend fun processLargeList(items: List<Item>) {
    for (item in items) {
        ensureActive() // Throws CancellationException if cancelled
        heavyComputation(item)
    }
}

// Alternative: yield() — also checks cancellation + gives other coroutines a chance to run
suspend fun processLargeList(items: List<Item>) {
    for (item in items) {
        yield()
        heavyComputation(item)
    }
}

CancellationException Is Special

// CancellationException is NOT treated as a failure
// It does NOT propagate to the parent as an error
// It does NOT trigger CoroutineExceptionHandler

scope.launch {
    try {
        delay(Long.MAX_VALUE)
    } catch (e: CancellationException) {
        // Cleanup here
        throw e // MUST re-throw! Swallowing it breaks cancellation
    }
}

// DANGER: generic catch swallows cancellation
scope.launch {
    try {
        someWork()
    } catch (e: Exception) { // CancellationException IS an Exception
        // This silently swallows cancellation — BUG
        log("Error: $e")
    }
}

// FIX: always re-throw CancellationException
scope.launch {
    try {
        someWork()
    } catch (e: Exception) {
        if (e is CancellationException) throw e
        log("Error: $e")
    }
}
The #1 Coroutine Bug in Production

Catching Exception (or Throwable) without re-throwing CancellationException is the most common coroutine bug. It silently breaks cancellation, causing coroutines to continue running after their scope is cancelled — leading to memory leaks, crashes from accessing destroyed UI, and wasted resources. Always use runCatching carefully or catch specific exception types.

NonCancellable — Running Cleanup After Cancellation

scope.launch {
    try {
        doWork()
    } finally {
        // After cancellation, the coroutine is in "cancelling" state
        // Any suspend call here will immediately throw CancellationException
        withContext(NonCancellable) {
            // This block runs even during cancellation
            saveProgressToDb() // suspend call works here
            closeResources()
        }
    }
}

5. Exception Handling

Exception handling in coroutines is the most confusing topic. Here's the complete mental model:

Rule 1: launch propagates, async defers

// launch: exception propagates to parent IMMEDIATELY
scope.launch {
    throw RuntimeException("boom") // → propagates to scope → crashes (if no handler)
}

// async: exception is stored in the Deferred, thrown on await()
val deferred = scope.async {
    throw RuntimeException("boom") // → stored, not propagated yet
}
deferred.await() // → throws RuntimeException HERE

Rule 2: Exception propagation path

CoroutineExceptionHandler (CEH)

val handler = CoroutineExceptionHandler { _, exception ->
    Log.e("Coroutine", "Uncaught: ${exception.message}")
    // Report to Crashlytics, etc.
}

// CEH only works on ROOT coroutines (launched directly on a scope)
val scope = CoroutineScope(SupervisorJob() + Dispatchers.Main + handler)

scope.launch {
    throw RuntimeException("handled by CEH")
}

scope.launch {
    launch {
        throw RuntimeException("also handled by CEH — propagates up to root")
    }
}
Staff-Level Insight — Where CEH Works and Doesn't

CoroutineExceptionHandler is a last resort, not a try-catch replacement. It only works on:

  • Root coroutines created with launch (not async)
  • Must be in the context of the root scope or the launch call

It does NOT work on:

  • async — exceptions are deferred to await()
  • Child coroutines — the exception propagates to the parent first
  • coroutineScope { } — exceptions are re-thrown from the scope function

In production Android code, the pattern is: SupervisorJob at the scope level + try-catch inside individual coroutines + CEH as a safety net for logging/crash reporting.

The Complete Exception Handling Toolkit

// 1. try-catch inside the coroutine (most common)
scope.launch {
    try {
        riskyOperation()
    } catch (e: IOException) {
        showError(e.message)
    }
}

// 2. supervisorScope for independent parallel work
suspend fun loadDashboard() = supervisorScope {
    val user = async { fetchUser() }      // If this fails...
    val feed = async { fetchFeed() }      // ...this continues
    val ads = async { fetchAds() }        // ...this continues

    DashboardData(
        user = runCatching { user.await() }.getOrNull(),
        feed = runCatching { feed.await() }.getOrNull(),
        ads = runCatching { ads.await() }.getOrNull()
    )
}

// 3. coroutineScope for all-or-nothing parallel work
suspend fun transferMoney(from: Account, to: Account, amount: Double) = coroutineScope {
    val debit = async { bankApi.debit(from, amount) }
    val credit = async { bankApi.credit(to, amount) }
    // If either fails, both are cancelled — atomicity
    debit.await()
    credit.await()
}

6. Common Patterns and Anti-Patterns

Anti-Pattern: GlobalScope

// NEVER do this in Android
GlobalScope.launch {
    fetchData() // Lives forever, no lifecycle awareness, no cancellation
}
// GlobalScope uses Dispatchers.Default, has no parent Job
// Coroutines launched here outlive Activities, ViewModels, everything

Anti-Pattern: Breaking Structured Concurrency

// BAD — creating a new scope inside a suspend function
suspend fun fetchData() {
    CoroutineScope(Dispatchers.IO).launch {
        // This coroutine is NOT a child of the caller
        // Cancelling the caller does NOT cancel this
        // This is a fire-and-forget leak
    }
}

// GOOD — use coroutineScope or withContext
suspend fun fetchData() = withContext(Dispatchers.IO) {
    // This is part of the caller's structured hierarchy
}

Pattern: Scope Per Feature

class SearchFeature {
    private val searchScope = CoroutineScope(SupervisorJob() + Dispatchers.Main.immediate)

    fun onQueryChanged(query: String) {
        searchScope.coroutineContext.cancelChildren() // Cancel previous search
        searchScope.launch {
            delay(300) // Debounce
            val results = withContext(Dispatchers.IO) { searchApi.search(query) }
            showResults(results)
        }
    }

    fun onDestroy() {
        searchScope.cancel() // Clean up everything
    }
}
Staff-Level Insight — The Scope Design Decision

At L6, you should be able to articulate why you chose a particular scope structure. The decision framework:

  • viewModelScope — for work tied to a screen's data (survives config changes, cancelled when user leaves)
  • lifecycleScope — for work tied to a specific Activity/Fragment instance (cancelled on destroy)
  • Custom scope in Repository/UseCase — for work that must outlive the UI (e.g., "upload must finish even if user navigates away"). This is the most debated pattern — some teams use WorkManager instead.
  • Application-scoped — for truly global work (analytics, crash reporting). Use sparingly.
