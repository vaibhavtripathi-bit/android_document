Flows & Channels

Cold vs hot streams, the Flow API, SharedFlow, StateFlow, Channels, backpressure strategies, and when to use which

1. Cold Flow — The Foundation

A Flow is a cold, asynchronous stream of values. "Cold" means the producer doesn't start until someone collects:

// Creating a flow
fun fetchUsers(): Flow<User> = flow {
    val users = api.getUsers()       // suspend call
    for (user in users) {
        emit(user)                   // Send value downstream
        delay(100)                   // Can suspend between emissions
    }
}

// Collecting a flow (terminal operator)
viewModelScope.launch {
    fetchUsers().collect { user ->
        // Called for each emitted value
        // This lambda is a suspend function — it can call suspend functions
        processUser(user)
    }
}

Key Properties of Cold Flow

  • Cold: Nothing happens until collect is called. Each collector gets its own independent execution of the flow body.
  • Sequential: emit() suspends until the collector processes the value. Producer and consumer are synchronized.
  • Cancellable: When the collector's coroutine is cancelled, the flow is cancelled at the next suspension point.
  • Context preservation: The flow body runs in the collector's context by default. Use flowOn to change upstream context.

2. Flow Operators

Intermediate Operators (Lazy — Return a New Flow)

flowOf(1, 2, 3, 4, 5)
    .filter { it % 2 == 0 }           // 2, 4
    .map { it * 10 }                   // 20, 40
    .onEach { log("Processing: $it") } // Side effect
    .take(1)                           // 20 (then cancels upstream)
    .collect { println(it) }           // Terminal operator

Context Switching with flowOn

fun fetchData(): Flow<Data> = flow {
    // This runs on Dispatchers.IO (set by flowOn below)
    val data = blockingApi.fetch()
    emit(data)
}
.flowOn(Dispatchers.IO) // Changes UPSTREAM context only
// Everything BELOW flowOn runs in the collector's context
flowOn Changes Upstream Only

flowOn affects everything above it in the chain. It introduces a buffer (channel) between upstream and downstream. You cannot use withContext inside a flow { } builder to change the emission context — that violates context preservation and throws IllegalStateException.

Deep Dive: channelFlow If you have a truly complex case where you need to emit from multiple contexts or concurrent coroutines (e.g., listening to multiple sockets at once), you should use channelFlow instead of flow. 

    channelFlow is thread-safe and allows for concurrent emissions from different contexts because it uses a Channel under the hood to buffer values back to the collector.
    val myChannelFlow = channelFlow {
        launch(Dispatchers.IO) {
            send(api1.fetch()) // 'send' is used instead of 'emit'
        }
        launch(Dispatchers.Default) {
            send(api2.fetch())
        }
    }

Error Handling in Flows

// catch — handles upstream exceptions
flow { emit(riskyOperation()) }
    .catch { e ->
        // Handle error, optionally emit fallback values
        emit(fallbackValue)
        // Or re-throw: throw e
    }
    .collect { value -> updateUI(value) }

// onCompletion — runs when flow completes (success or failure)
flow { emit(fetchData()) }
    .onCompletion { cause ->
        if (cause != null) log("Flow failed: $cause")
        else log("Flow completed successfully")
        hideLoadingSpinner()
    }
    .collect { updateUI(it) }

// retry / retryWhen
flow { emit(api.fetch()) }
    .retryWhen { cause, attempt ->
        if (cause is IOException && attempt < 3) {
            delay(1000L * (attempt + 1)) // Exponential backoff
            true // retry
        } else {
            false // give up
        }
    }
    .collect { updateUI(it) }
Staff-Level Insight — catch Position Matters

catch only catches exceptions from upstream operators. If the collect lambda throws, catch won't see it. This is by design — the collector is downstream. To handle collector exceptions, wrap the collect in a try-catch, or use onEach + launchIn:

flow.onEach { updateUI(it) } // moves "collect" logic to an operator
    .catch { handleError(it) } // now catches everything including UI errors
    .launchIn(scope)           // terminal operator that launches collection

3. SharedFlow — Hot, Multi-Subscriber Stream

A SharedFlow is a hot flow that can have multiple collectors. Emissions happen regardless of whether anyone is collecting.

// Creating a SharedFlow
private val _events = MutableSharedFlow<UiEvent>(
    replay = 0,              // How many past values new subscribers get
    extraBufferCapacity = 64, // Buffer for when collectors are slow
    onBufferOverflow = BufferOverflow.DROP_OLDEST // What to do when buffer is full
)
val events: SharedFlow<UiEvent> = _events.asSharedFlow()

// Emitting (from a coroutine)
viewModelScope.launch {
    _events.emit(UiEvent.ShowSnackbar("Saved!"))
}

// Collecting (multiple collectors get the same events)
lifecycleScope.launch {
    viewModel.events.collect { event ->
        when (event) {
            is UiEvent.ShowSnackbar -> showSnackbar(event.message)
            is UiEvent.Navigate -> navigateTo(event.route)
        }
    }
}

SharedFlow Configuration

ParameterDefaultEffect
replay0Number of values replayed to new subscribers. 0 = new subscribers only see future emissions.
extraBufferCapacity0Additional buffer beyond replay. Total buffer = replay + extraBufferCapacity.
onBufferOverflowSUSPENDSUSPEND: emit() suspends. DROP_OLDEST: drops oldest buffered. DROP_LATEST: drops the new emission.

4. StateFlow — Observable State Holder

StateFlow is a specialized SharedFlow designed for holding state. It's the coroutines replacement for LiveData:

private val _uiState = MutableStateFlow(UiState.Loading)
val uiState: StateFlow<UiState> = _uiState.asStateFlow()

// Updating state
fun loadData() {
    viewModelScope.launch {
        _uiState.value = UiState.Loading
        try {
            val data = repository.fetchData()
            _uiState.value = UiState.Success(data)
        } catch (e: Exception) {
            _uiState.value = UiState.Error(e.message)
        }
    }
}

// Thread-safe atomic update
_uiState.update { currentState ->
    when (currentState) {
        is UiState.Success -> currentState.copy(isRefreshing = true)
        else -> currentState
    }
}

Staff-Level Insight — StateFlow Equality Gotcha

StateFlow uses Any.equals() for conflation. If you emit the same value twice, the second emission is dropped. This catches people when using data classes: _state.value = UiState.Success(sameData) won't trigger re-collection if the data hasn't changed. This is usually desirable (avoids unnecessary recomposition in Compose), but can be surprising when you want to force a refresh. Workaround: include a refreshId counter in your state, or use SharedFlow(replay=1) instead.

The One-Time Event Problem

Using StateFlow for one-time events (navigation, snackbar) is a common mistake. Because StateFlow replays the last value to new subscribers, a navigation event can fire again after configuration change. Use SharedFlow(replay=0) for events, or wrap events in a consumed/unconsumed wrapper. This is one of the most debated patterns in Android architecture.

6. Channels — Communication Between Coroutines

A Channel is a hot, concurrent queue for sending values between coroutines. Unlike Flow, a Channel value is consumed by exactly one receiver (fan-out).

val channel = Channel<Int>(capacity = Channel.BUFFERED)

// Producer
launch {
    for (i in 1..5) {
        channel.send(i) // Suspends if buffer is full
    }
    channel.close() // Signal no more values
}

// Consumer
launch {
    for (value in channel) { // Iterates until channel is closed
        println("Received: $value")
    }
}

Channel Capacity Strategies

CapacityBehaviorUse Case
Channel.RENDEZVOUS (0)No buffer. send() suspends until receive() is called.Strict synchronization between producer and consumer
Channel.BUFFERED (64)Default buffer. send() suspends when buffer is full.General purpose
Channel.UNLIMITEDUnlimited buffer. send() never suspends.When you can't afford to suspend the producer (danger: OOM)
Channel.CONFLATEDBuffer of 1. New value overwrites old.Only latest value matters (like StateFlow)

callbackFlow — Bridging Callbacks to Flow via Channel

fun locationUpdates(): Flow<Location> = callbackFlow {
    val client = LocationServices.getFusedLocationProviderClient(context)
    val callback = object : LocationCallback() {
        override fun onLocationResult(result: LocationResult) {
            for (location in result.locations) {
                trySend(location) // Non-suspending send (drops if buffer full)
            }
        }
    }

    client.requestLocationUpdates(locationRequest, callback, Looper.getMainLooper())

    awaitClose {
        // Called when the flow collector cancels
        client.removeLocationUpdates(callback)
    }
}
Staff-Level Insight — Flow vs Channel

The mental model: Flow is for data streams (one producer, many consumers get all values). Channel is for communication (values are consumed by exactly one receiver). In practice:

  • Use Flow for reactive data (database changes, API responses, sensor data)
  • Use Channel for work distribution (fan-out to multiple workers) or when you need send() from non-suspend contexts
  • Use callbackFlow to convert callback-based APIs to Flow (it uses a Channel internally)

In modern Android, you almost never use Channel directly. SharedFlow and callbackFlow cover 99% of use cases.

7. Backpressure

Backpressure occurs when the producer emits faster than the consumer can process. Flow handles this with operators:

// buffer — decouple producer and consumer with a channel
flow { /* fast producer */ }
    .buffer(capacity = 64) // Producer runs ahead, buffering values
    .collect { /* slow consumer */ }

// conflate — drop intermediate values, only process latest
flow { /* fast producer */ }
    .conflate() // If consumer is busy, skip to latest value
    .collect { /* slow consumer */ }

// collectLatest — cancel previous collection when new value arrives
flow { /* fast producer */ }
    .collectLatest { value ->
        // If a new value arrives while processing, THIS block is cancelled
        // and restarted with the new value
        heavyProcessing(value)
    }

// debounce — wait for silence before emitting
searchQueryFlow
    .debounce(300) // Only emit after 300ms of no new values
    .collect { query -> performSearch(query) }

// sample — emit the latest value at fixed intervals
sensorDataFlow
    .sample(100) // Emit latest value every 100ms, regardless of emission rate
    .collect { reading -> updateUI(reading) }
Staff-Level Insight — collectLatest Is the Android Default

In Android UI, collectLatest is almost always what you want. When the user types fast in a search box, you want to cancel the previous search and start a new one — not queue up all intermediate searches. When state changes rapidly, you want to render the latest state, not process every intermediate state. This is why repeatOnLifecycle + collectLatest is the recommended pattern for collecting StateFlow in Activities/Fragments.

8. Flow Composition

Flow composition operators fall into two categories: synchronization operators that combine multiple streams into one (combine, zip, merge), and flattening operators that handle flows-of-flows (flatMapLatest, flatMapConcat, flatMapMerge). Choosing the wrong one is a common source of subtle bugs.

combine vs zip — Synchronization Logic

These two operators look similar but solve fundamentally different problems.

combine — The UI State Builder

Behaviour: Waits for the first emission from every source flow. After that, whenever any source emits, it re-calculates immediately using the latest values from all others.

// Classic use-case: merging independent data sources into a single ViewState
val uiState: StateFlow<DashboardState> = combine(
    userRepository.user,        // Flow<User>
    settingsRepository.settings, // Flow<Settings>
    networkStatus               // Flow<Boolean>
) { user, settings, isOnline ->
    DashboardState(user, settings, isOnline)
}.stateIn(viewModelScope, SharingStarted.WhileSubscribed(5_000), DashboardState.Loading)
Staff-Level Insight
combine is the gold standard for MVI/MVVM architectures. Each repository exposes its own StateFlow, and the ViewModel merges them into a single ViewState. Each source is independently updated, independently cached, and independently testable. The UI just collects one stream.

zip — The Strict Pairer

Behaviour: Pairs emissions 1-to-1: 1st from A with 1st from B, 2nd from A with 2nd from B. The faster flow waits for the slower one to catch up.

// Good use-case: pairing items that have a strict 1:1 relationship
val usersWithAvatars = userFlow.zip(avatarFlow) { user, avatar ->
    UserWithAvatar(user, avatar)
}
The zip Risk
If userFlow emits 100 times and avatarFlow emits only 2 times, the zipped flow emits exactly twice. The other 98 values from userFlow sit in an internal buffer waiting for avatarFlow to catch up. If avatarFlow never completes, those 98 values are held in memory indefinitely — a memory leak. Only use zip when you are certain both flows emit at the same rate.
combinezip

merge — The Simple Interleaver

Behaviour: Funnels multiple streams into one output without any transformation. Values appear in the output in the order they arrive, regardless of which source emitted them.

// Perfect for Action Streams — multiple buttons that trigger the same command
val refreshTrigger: Flow<Unit> = merge(
    pullToRefreshFlow,    // Flow<Unit> from swipe gesture
    retryButtonFlow,      // Flow<Unit> from error screen button
    appForegroundFlow     // Flow<Unit> when app returns from background
)

viewModelScope.launch {
    refreshTrigger
        .flatMapLatest { fetchData() }
        .collect { updateUi(it) }
}
Staff-Level Insight
merge is the correct tool for Action Streams. Instead of three separate listeners each calling viewModel.refresh(), merge their flows and process them in one place. This keeps your ViewModel's logic centralised and eliminates duplicate event handling code.

The flatMap Trio — Flattening Strategies

When each upstream emission triggers a new inner Flow (e.g., a search query triggering an API call that returns a Flow), you need to "flatten" the stream-of-streams. How you flatten changes the behaviour entirely.

flatMapLatest — "Forget the Past"

Mechanism: When a new upstream value arrives, it immediately cancels the previous inner flow and starts a new one. This prevents race conditions where an old, slow API call delivers its result after a newer, faster one.

// Classic: search-as-you-type
searchQueryFlow
    .debounce(300)               // Wait for user to pause typing
    .distinctUntilChanged()       // Don't re-search the same query
    .flatMapLatest { query ->
        if (query.isBlank()) flowOf(emptyList())
        else searchRepository.search(query)   // Previous call cancelled here
    }
    .collect { results -> _uiState.update { it.copy(results = results) } }
Best For
Search-as-you-type, tab switching, any scenario where only the latest request matters and previous in-flight work should be discarded.

flatMapConcat — "Wait Your Turn"

Mechanism: Waits for the current inner flow to complete before starting the next one. Strictly preserves the order of upstream emissions. The slowest and most predictable option.

// Sequential upload: each file must finish before the next begins
fileList.asFlow()
    .flatMapConcat { file ->
        uploadRepository.upload(file)   // Flow<UploadProgress>
    }
    .collect { progress -> updateProgressBar(progress) }
Best For
Sequential ordered processing: uploading files one-by-one, running migrations in order, or any pipeline where task N must complete before task N+1 starts.

flatMapMerge — "All at Once"

Mechanism: Starts all inner flows immediately and concurrently as upstream values arrive. Fastest option but results may arrive out of order.

// Fetch details for multiple IDs in parallel
itemIds.asFlow()
    .flatMapMerge(concurrency = 3) { id ->   // Max 3 concurrent requests
        detailRepository.fetchDetail(id)
    }
    .collect { detail -> updateItem(detail) }
Use the concurrency Parameter
Without a limit, flatMapMerge starts every inner flow at once. With 1000 IDs, that is 1000 simultaneous network requests. Always set concurrency to a safe ceiling (typically 3–10 for network, higher for CPU-bound tasks).

9. Flow Internals — How Operators Chain

Understanding how Flow works under the hood is what separates L6 from L5. Every Flow is an implementation of this interface:

public interface Flow<out T> {
    public suspend fun collect(collector: FlowCollector<T>)
}

public fun interface FlowCollector<in T> {
    public suspend fun emit(value: T)
}

SafeFlow and AbstractFlow

When you write flow { emit(value) }, the compiler creates a SafeFlow:

// What you write:
val myFlow = flow {
    emit(1)
    emit(2)
}

// What the runtime creates (simplified):
internal class SafeFlow<T>(
    private val block: suspend FlowCollector<T>.() -> Unit
) : AbstractFlow<T>() {
    override suspend fun collectSafely(collector: FlowCollector<T>) {
        collector.block() // Runs YOUR flow { } body with the collector as receiver
    }
}

// AbstractFlow enforces two invariants:
// 1. Context preservation — emit() must happen in the same context as collect()
// 2. No concurrent emissions — emit() is not thread-safe by design
public abstract class AbstractFlow<T> : Flow<T> {
    override suspend fun collect(collector: FlowCollector<T>) {
        val safeCollector = SafeCollector(collector, coroutineContext)
        collectSafely(safeCollector)
        // SafeCollector checks on every emit():
        //   - Is the emission context the same as collection context?
        //   - If not → IllegalStateException
    }
}

How Operators Chain — The Wrapping Pattern

Each operator creates a new Flow that wraps the upstream Flow. When you call collect() on the outermost Flow, it triggers a chain of nested collect() calls:

// How .filter() works internally (simplified from kotlinx.coroutines source):
public inline fun <T> Flow<T>.filter(
    crossinline predicate: suspend (T) -> Boolean
): Flow<T> = flow { // Creates a NEW flow
    collect { value ->   // Collects from upstream (this@filter)
        if (predicate(value)) {
            emit(value)  // Emits to downstream (whoever collects this new flow)
        }
    }
}

// How .map() works internally:
public inline fun <T, R> Flow<T>.map(
    crossinline transform: suspend (T) -> R
): Flow<R> = flow {
    collect { value ->
        emit(transform(value))
    }
}

// Key insight: operators are just flow { collect { emit() } } wrappers
// No magic, no special runtime — pure Kotlin
Staff-Level Insight — Why Flow Is Zero-Cost

Unlike RxJava (which allocates Observable, Observer, Disposable objects per operator), Flow operators are inline functions that create lambda wrappers. The Kotlin compiler inlines the lambdas, so the entire operator chain compiles down to nested function calls with zero object allocation per element. The only allocation is the Flow objects themselves (one per operator), created once when the chain is built. This is why Flow outperforms RxJava in benchmarks — it's essentially a direct function call chain.

10. Context Preservation — The Deep Why

// This throws IllegalStateException:
flow {
    withContext(Dispatchers.IO) {
        emit(fetchData()) // ILLEGAL — different context than collector
    }
}.collect { updateUI(it) }

// Why? Because emit() is a SUSPEND call on the collector.
// The collector (updateUI) expects to run on Dispatchers.Main.
// If emit() is called from Dispatchers.IO, the collector would
// suddenly execute on IO — violating the caller's expectations.

// The CORRECT way:
flow {
    val data = withContext(Dispatchers.IO) { fetchData() } // Switch for work
    emit(data) // Emit back on the original context
}.collect { updateUI(it) }

// Or use flowOn (which handles this with a channel buffer):
flow { emit(fetchData()) }
    .flowOn(Dispatchers.IO) // Upstream runs on IO
    .collect { updateUI(it) } // Downstream runs on collector's context

How flowOn Works Internally

flowOn introduces a Channel between upstream and downstream. The upstream runs on the specified dispatcher and sends values into the channel. The downstream reads from the channel on the collector's dispatcher:

Staff-Level Insight — flowOn Performance Cost

Each flowOn creates a Channel and a new coroutine for upstream collection. This means: (1) allocation of the channel buffer, (2) context switch overhead for every emission, (3) potential backpressure if downstream is slow. Use one flowOn at the source, not on every operator. Multiple flowOn calls create multiple channels — each adding latency and memory. The pattern flow { }.map { }.filter { }.flowOn(IO) is correct — one flowOn moves the entire upstream chain to IO.

11. SharedFlow & StateFlow Internals

SharedFlow Under the Hood

MutableSharedFlow maintains a replay buffer (array) and a list of subscriber slots. On emit():

  1. Value is added to the replay buffer (circular array)
  2. All active subscribers are resumed with the new value
  3. If a subscriber is slow and the buffer is full, behavior depends on onBufferOverflow: SUSPEND (emit suspends), DROP_OLDEST (oldest buffer entry removed), DROP_LATEST (new value discarded)

StateFlow Under the Hood

MutableStateFlow is a SharedFlow with replay=1, onBufferOverflow=DROP_OLDEST, and equality-based conflation. On .value = newValue:

  1. Compare newValue with current value using Any.equals()
  2. If equal → no-op, no subscribers are notified
  3. If different → update internal value, resume all suspended subscribers
// StateFlow is essentially:
MutableSharedFlow<T>(
    replay = 1,
    extraBufferCapacity = 0, // Total buffer = 1
    onBufferOverflow = BufferOverflow.DROP_OLDEST
).apply {
    // Plus: equality conflation on every emission
    // Plus: .value property for synchronous access
    // Plus: always has a value (never empty)
}
Staff-Level Insight — The update() Atomicity

StateFlow.update { } uses a CAS (compare-and-swap) loop internally. It reads the current value, applies your transform, and attempts to set it. If another thread changed the value between read and write, it retries. This is lock-free and thread-safe, but your transform function may be called multiple times. Keep it pure (no side effects).

// Simplified internal implementation:
fun update(function: (T) -> T) {
    while (true) {
        val current = value
        val new = function(current)
        if (compareAndSet(current, new)) return // CAS success
        // CAS failed — another thread changed value, retry
    }
}
