Part 1
The Suspension Engine — How the Compiler Output Runs

The generated classes, the class hierarchy they inherit from, and how local variables survive suspension.

1. The Continuation Class Hierarchy

Your suspend function compiles to a class. That class doesn't exist in isolation — it inherits from a precise 4-layer tower, each layer in a different module with a single added responsibility.

Why 3 Abstract Classes Instead of 1?

They live in different modules. BaseContinuationImpl is in kotlin-stdlib — it cannot depend on kotlinx.coroutines. ContinuationImpl adds interceptor awareness, which requires CoroutineContext from the coroutines library. SuspendLambda adds create()/invoke() for the compiler's codegen. Collapsing them would create a dependency from the Kotlin standard library to the coroutines library — an architectural mistake. Each layer adds exactly one responsibility.

The resumeWith() Iterative Loop

When a continuation completes, it needs to resume its caller's continuation. A naive approach: completion.resumeWith(result). But 100 non-suspending completions would stack 100 frames → StackOverflowError. The actual code uses a loop:

// BaseContinuationImpl.resumeWith() — simplified from kotlin-stdlib source
override fun resumeWith(result: Result<Any?>) {
    var current = this
    var param = result

    while (true) {
        val outcome = current.invokeSuspend(param)   // Run YOUR code

        if (outcome === COROUTINE_SUSPENDED) return   // Actually suspended — thread freed

        val completion = current.completion             // The caller's continuation
        if (completion is BaseContinuationImpl) {
            // LOOP instead of recurse — avoids stack growth
            current = completion
            param = Result.success(outcome)
        } else {
            // Reached a non-BaseContinuationImpl (e.g., DispatchedContinuation)
            completion.resumeWith(Result.success(outcome))
            return
        }
    }
}
Why Iterative, Not Recursive?

Deep continuation chains (100 nested withContext calls, or a deeply recursive suspend function that never actually suspends) would cause StackOverflowError with recursive resumeWith(). The loop unwinds the chain on the heap, not the stack. This is a trampoline pattern — the same technique used by tail-call-optimized functional languages.

2. Variable Spilling — The Stack Frame on the Heap

Before each suspension point, the compiler generates code to save all live local variables into the continuation object's fields. After resumption, they're restored. The continuation IS the stack frame, but it lives on the heap.

Why L$0/I$0 Instead of Named Fields?

The compiler reuses fields across labels. At label 0, L$0 holds id. If the function had a different variable url live only at label 2, L$0 could hold url there. This is register allocation for heap fields — fewer fields means smaller objects. Primitives live across suspension points get boxed to Any? (Object) when the same slot must hold different types. This boxing is the one hidden cost of suspension.

Part 2
The Dispatch Pipeline — From resumeWith() to Thread

How a resumed continuation actually reaches a thread. The interception, dispatching, and undispatched fast paths.

3. DispatchedContinuation — The Interception Chain

When a coroutine resumes, the value doesn't magically appear on the right thread. A precise chain of objects routes it there.

Why Does DispatchedContinuation Implement Runnable?

Dispatchers are thread pools. Thread pools execute Runnables. By making the continuation itself a Runnable, the dispatch path avoids allocating a separate wrapper object for every resume. Without this, every coroutine resumption would create garbage. The continuation is both the callback (Continuation) and the task (Runnable) in one object.

Why Is intercepted() Cached?

ContinuationImpl stores the result of intercepted() in a private var intercepted field. The dispatcher's interceptContinuation() is called only once, on the first suspension. Subsequent resumes reuse the cached DispatchedContinuation. Without caching, every resume would allocate a new wrapper and call the dispatcher's interception method again.

4. CancellableContinuationImpl — The Atomic State Machine

suspendCancellableCoroutine is how you bridge callback-based APIs into coroutines. It creates a CancellableContinuationImpl with a 3-state atomic machine that resolves a fundamental race condition.

The Race: resume() vs cancel()

Consider: a network callback fires continuation.resume(data) on Thread A at the exact instant the parent Job calls cancel() on Thread B. Without atomic coordination, both could succeed — causing a double-resume crash or a lost cancellation (resource leak).

Why Atomic CAS Instead of synchronized?

resume() fires from any thread (OkHttp callbacks, timer threads, other coroutines). cancel() propagates from the Job tree. A synchronized block would work for correctness, but would be slower — lock acquisition involves the JVM's monitor mechanism (potentially OS futex). A single CAS instruction resolves the race in ~1 nanosecond. For a mechanism that runs on every suspendCancellableCoroutine call in the app, that matters.

Why Does the Fast Path Exist?

If the callback fires before suspendCancellableCoroutine finishes setting up, the value is already available. The UNDECIDED → RESUMED transition means the coroutine never actually suspends — COROUTINE_SUSPENDED is never returned. This is the same zero-cost principle as the basic state machine: suspension is free when it doesn't happen. Cache hits, already-connected sockets, and in-memory data all take this path.

The Job Engine — State Machines All the Way Down

The internal state machine that drives Job lifecycle, cancellation, and completion. Plus withContext, which secretly creates a Job.

5. Job Internal State Machine

Previously in 03 you saw the Job tree and cancellation propagation. This section shows the actual _state field, its atomic transitions, and the one method that makes SupervisorJob different.

The _state: Any? Tagged Union

The JobSupport base class has a single volatile field: _state: Any?. Its runtime type encodes both the Job's current phase and its associated data. No separate enum needed.

Job vs SupervisorJob — The ONE Method

// The ENTIRE implementation difference:
// In JobSupport (base class):
open fun childCancelled(cause: Throwable): Boolean {
    if (cause is CancellationException) return true  // Always handle CancellationException
    return cancelImpl(cause)                          // Propagate failure → cancel parent
}

// In SupervisorJobImpl:
override fun childCancelled(cause: Throwable): Boolean = false  // That's it. One line.
Why Any? Instead of an Enum for _state?

Because the state field also carries the result. A completed Job stores the return value directly in _state. A cancelled Job stores CompletedExceptionally(cause) in _state. Using a separate enum + result field would add 8+ bytes to every Job object. With millions of coroutines, that adds up. The Any? tagged-union is the same pattern used throughout the coroutines library for zero-overhead state encoding.

6. withContext Internals

withContext is not launch + join. It creates a DispatchedCoroutine that is smarter about thread switching and exception handling.

withContext vs launch+join — Exception Behavior

withContext(IO) { throw Error() }launch(IO) { throw Error() }.join()
Exception goes to Thrown directly at the call site. You can try/catch it. Propagates UP to parent Job. Parent cancels siblings. join() doesn't rethrow.
Sibling coroutines Unaffected. No propagation. Cancelled via parent Job.
Mental model A synchronous function call that runs on a different thread. A separate concurrent coroutine with its own failure domain.
Why Does withContext Create a DispatchedCoroutine Even on Fast Path?

Because it still needs a child Job for structured concurrency. If the parent is cancelled while the withContext block runs, the block must be cancelled too. The DispatchedCoroutine links to the parent Job, so cancellation propagates correctly. The fast path skips dispatching, not Job management.

Part 4
Event Loops, Trampolines, and the Context Map

The execution environments that don't use thread pools, and the data structure behind CoroutineContext.

7. Event Loop and Trampoline

runBlocking's Internal Event Loop

runBlocking blocks the calling thread, but the thread isn't wasted — it becomes an event loop that processes coroutine tasks.

Unconfined Dispatcher's Trampoline

Dispatchers.Unconfined resumes on whatever thread called resume(). But what if a resumed coroutine immediately completes another coroutine's suspension, which resumes another, and so on? Without protection: unbounded recursion.

The trampoline breaks this: the first resume runs inline. Any nested resumes within that execution are queued. After the first resume finishes, the queued resumes are processed iteratively. Result: flat stack, no overflow.

Why Unconfined Is Dangerous in Production

The thread your coroutine runs on becomes data-dependent. If OkHttp's response arrives on its callback thread, your coroutine runs there. If the value was cached and returned synchronously, your coroutine runs on the caller's thread. You can't reason about thread safety when the thread is non-deterministic. The trampoline prevents crashes, but not thread-safety bugs.

8. CoroutineContext Algebra

You use context + Dispatchers.IO + CoroutineName("x") daily. Here's the data structure beneath it.

CombinedContext: A Linked List, Not a HashMap

The + Operator: Right Overrides Left

// What happens when contexts are combined:
val parent = Job_A + Dispatchers.Default
val child  = Job_B + CoroutineName("x")

val result = parent + child
// Step 1: Remove from 'parent' any keys that exist in 'child'
//         Job_A removed (child has Job_B with same Key)
//         Dispatchers.Default kept (child has no Dispatcher)
// Step 2: Combine: Dispatchers.Default + Job_B + CoroutineName("x")
//
// This is why child's Job REPLACES parent's Job, but parent's Dispatcher is INHERITED.
Why Is Every Element Also a CoroutineContext?

So you can write launch(Dispatchers.IO) { } instead of launch(EmptyContext + Dispatchers.IO) { }. A single element is a context with one entry. This algebraic identity eliminates the wrapping ceremony. The + operator handles the combination: element + element creates a CombinedContext, element + EmptyContext returns the element.

Staff-Level Q&A

Staff / Principal Q&A
Walk through what happens at the JVM level when a coroutine hits delay(1000) and then resumes. Name every class involved.
delay(1000) calls suspendCancellableCoroutine, which creates a CancellableContinuationImpl wrapping the current continuation. The delay implementation schedules a task on the dispatcher's DelayedTaskQueue (or the event loop for runBlocking). CancellableContinuationImpl transitions from UNDECIDED to SUSPENDED. invokeSuspend() returns COROUTINE_SUSPENDED to the BaseContinuationImpl.resumeWith() loop, which returns, freeing the thread. After 1000ms, the scheduler fires the delayed task, calling continuation.resumeWith(Unit). The CancellableContinuationImpl CAS-transitions from SUSPENDED to RESUMED, then delegates to the wrapped continuation. ContinuationImpl.intercepted() returns the cached DispatchedContinuation, which calls dispatcher.dispatch(this). The continuation (as a Runnable) is enqueued on the CoroutineScheduler's work queue. A worker thread picks it up, calls DispatchedTask.run(), which calls BaseContinuationImpl.resumeWith(), which calls invokeSuspend() at the next label. Your code continues.
Staff / Principal Q&A
Why is withContext(Dispatchers.IO) from a Default worker almost free, while launch(Dispatchers.IO) { }.join() is not?
withContext checks whether the target dispatcher requires a dispatch from the current thread. Default and IO share the same CoroutineScheduler — withContext(IO) from a Default worker only changes the worker's state from CPU_ACQUIRED to BLOCKING (an atomic flag flip). The coroutine may stay on the same physical thread. There's one DispatchedCoroutine allocation and potentially zero thread switches. launch(IO) { }.join() creates a full separate coroutine with its own Job, always dispatches the block (guaranteed thread switch to the IO pool), and then join() suspends the caller (second thread switch back). Beyond the performance difference, the failure semantics differ: withContext throws the exception at the call site (like a function call), while launch propagates the exception up to the parent Job, which cancels all siblings. withContext is a "synchronous call on a different thread." launch + join is "a concurrent operation you happen to wait for."
Staff / Principal Q&A
Explain the race condition that CancellableContinuationImpl solves, and why a simple synchronized block wouldn't work as well.
The race: resume(value) on Thread A (e.g., OkHttp callback) and cancel() on Thread B (Job cancellation propagating down the tree) can happen simultaneously. Without coordination, both succeed: the coroutine resumes with the value AND throws CancellationException, causing a double-resume crash. CancellableContinuationImpl uses a CAS on an atomic _decision field: whoever successfully CAS-transitions from SUSPENDED to their target state wins. The loser's operation is silently discarded. A synchronized block would give correctness but at higher cost: lock acquisition involves the JVM's monitor mechanism (potential futex syscall under contention), whereas CAS is a single CPU instruction (~1 ns). Since suspendCancellableCoroutine is the primitive for ALL callback bridges in the coroutines ecosystem (Retrofit, Room, OkHttp, Firebase, etc.), this cost compounds. Additionally, synchronized blocks can't be used from inline functions and have restrictions on Kotlin/Native, making CAS the more portable choice.
Staff / Principal Q&A
How does SupervisorJob differ from Job at the implementation level?
Exactly one method override. JobSupport (the base class for all Jobs) has an open method childCancelled(cause: Throwable): Boolean. In the default Job implementation, childCancelled() calls cancelImpl(cause) — propagating the child's failure to the parent, which enters CANCELLING and cancels all other children. SupervisorJobImpl overrides this single method to return false unconditionally, meaning "I don't care about this child's failure." The parent stays ACTIVE, siblings are unaffected. Everything else — the state machine, _state encoding, NodeList, completion handlers, the CAS transitions — is identical. The name "supervisor" suggests a complex separate implementation, but it's literally one line: override fun childCancelled(cause) = false.
