Interview Q&A — Coroutines & Threading

30 questions covering threading fundamentals, coroutine internals, structured concurrency, Flows, Android integration, and concurrency patterns — click to expand answers

Threading Fundamentals (Q1–Q5)

Q1 What is the main thread in Android and why is it special? L4

The main thread (UI thread) is created when the app process starts. It's the only thread that can touch the View hierarchy, process input events, and run lifecycle callbacks. It has a Looper that continuously pulls messages from a MessageQueue — this is the event loop. Blocking it for >5 seconds on input events causes an ANR.

Staff-level addition: The main thread's Looper is set up in ActivityThread.main(). The MessageQueue uses Linux epoll via nativePollOnce() for efficient sleeping when idle. The Choreographer posts VSYNC callbacks to this same queue, meaning any main-thread work competes with frame rendering for time.
Q2 Explain the Looper/Handler/MessageQueue relationship. L5

Looper.prepare() creates a MessageQueue and binds it to the current thread via ThreadLocal. Looper.loop() enters an infinite loop calling queue.next() (blocks if empty) and dispatching each message to its target Handler. A Handler is bound to a Looper — handler.post() enqueues a Message into that Looper's queue. The message's target field points back to the Handler, so the Looper knows which Handler should process it.

What juniors miss: MessageQueue.next() doesn't busy-wait. It uses nativePollOnce() which calls epoll_wait() — the thread sleeps with zero CPU usage until a message arrives or a timeout expires. This is why the main thread doesn't burn battery when idle.
Q3 What are the ANR timeout thresholds and how do they differ? L5

Input events: 5 seconds. BroadcastReceiver: 10 seconds (foreground) / 60 seconds (background). Service: 20 seconds (foreground) / 200 seconds (background). ContentProvider: 10 seconds.

Staff-level addition: ANR is worse than a crash for app quality metrics. Google Play's "bad behavior" threshold is 0.47% ANR rate. ANRs are tracked in Android Vitals and affect search ranking. The system detects ANRs via InputDispatcher (for input ANRs) and BroadcastQueue/ActiveServices (for component ANRs). Use StrictMode in debug builds and systrace in profiling to catch potential ANRs early.
Q4 Why are raw threads dangerous on Android? What replaced them? L4

Raw threads have no lifecycle awareness (outlive Activities, causing crashes/leaks), no structured cancellation, no exception propagation, consume ~1MB each, and require manual synchronization. They were replaced by coroutines, which provide structured concurrency, cooperative cancellation, lightweight suspension, and lifecycle integration via scopes.

Staff-level addition: The evolution was: raw Thread → AsyncTask (deprecated API 30) → Executors → RxJava → Coroutines. Each step added more structure. Coroutines won because they're a language feature (compiler support), not just a library, giving them zero-cost abstractions that RxJava couldn't match.
Q5 What is the Choreographer and how does it relate to frame rendering? L6

The Choreographer receives VSYNC signals from SurfaceFlinger and posts a doFrame() callback to the main thread's MessageQueue. Each frame has four phases: INPUT → ANIMATION → TRAVERSAL (measure/layout/draw) → COMMIT. All four must complete within ~16ms (60fps) or ~8.3ms (120fps). The Choreographer is how Android synchronizes UI updates with the display refresh rate.

Staff-level addition: Dispatchers.Main posts coroutine resumptions as messages to the same MessageQueue. This means coroutine work on Main competes with Choreographer frame callbacks. Heavy Main-thread coroutine work directly causes frame drops. This is why Dispatchers.Main.immediate is preferred — it executes inline without going through the queue, reducing latency.

Coroutine Internals (Q6–Q12)

Q6 What does the suspend keyword actually do at the compiler level? L6

The compiler transforms each suspend function into a state machine using Continuation Passing Style (CPS). It adds a Continuation parameter, creates a state machine class with a label field, and converts each suspension point into a state. The function returns COROUTINE_SUSPENDED when it actually suspends, or the result directly if it completes synchronously. Each resume re-enters the state machine at the appropriate label.

Staff-level addition: The key insight: calling a suspend function does NOT always suspend. If the underlying operation completes immediately (cached value, delay(0)), the function returns the result directly without ever yielding. The COROUTINE_SUSPENDED sentinel is how the runtime distinguishes "actually suspended" from "completed synchronously." This is why suspend functions have zero overhead when they don't suspend.
Q7 What is a Continuation and how does suspendCancellableCoroutine work? L6

Continuation<T> is the callback interface with resumeWith(Result<T>) and a context. suspendCancellableCoroutine captures the current continuation and gives you access to it in a lambda. You store the continuation, and when the async callback fires, you call cont.resume(value) or cont.resumeWithException(e). The "cancellable" variant integrates with structured concurrency — cont.invokeOnCancellation { } lets you clean up when the coroutine is cancelled.

What juniors miss: Always use suspendCancellableCoroutine over suspendCoroutine. The non-cancellable variant ignores scope cancellation, meaning the callback can fire after the coroutine is dead, causing subtle bugs. Also: you must call resume exactly once — calling it twice throws IllegalStateException.
Q8 Explain the difference between Dispatchers.Default and Dispatchers.IO. Do they share threads? L5

Yes, they share the same CoroutineScheduler thread pool. Default is limited to CPU core count (for CPU-bound work). IO allows up to 64 threads (for blocking I/O). When you withContext(Dispatchers.IO) from Default, the coroutine may stay on the same physical thread — only the parallelism limit changes. This avoids unnecessary context switches.

Staff-level addition: The scheduler uses a work-stealing algorithm — each worker thread has a local deque, and idle threads steal from others. limitedParallelism(n) on IO creates threads beyond the 64 limit (additive), while on Default it creates a view within the existing pool (restrictive). This distinction matters for capacity planning in high-throughput apps.
Q9 What is Dispatchers.Main.immediate and why does it matter? L5

Dispatchers.Main always posts to the main thread's Handler, even if you're already on the main thread. Main.immediate checks if you're already on the main thread — if so, it executes inline without going through the MessageQueue. This avoids an extra frame of latency and is why viewModelScope and lifecycleScope use Main.immediate by default.

Q10 What is the difference between launch and async? When does each propagate exceptions? L5

launch returns a Job and propagates exceptions to the parent immediately. async returns a Deferred<T> and stores exceptions until await() is called. However, even with async, if the exception isn't handled (no one calls await()), it still propagates to the parent when the Deferred is garbage collected or the scope completes.

Staff-level addition: A subtle trap: async inside a coroutineScope still cancels siblings on failure, even though the exception is "deferred." The cancellation propagates through the Job hierarchy, not through the exception. The exception is only deferred for the caller — the structured concurrency machinery still reacts immediately.
Q11 What is withContext and how does it differ from launch? L4

withContext switches the coroutine's context (typically the dispatcher) and suspends until the block completes. It does NOT create a new coroutine — it's a context switch within the same coroutine. launch creates a new child coroutine that runs concurrently. Use withContext for sequential "do this on a different thread" and launch for concurrent work.

Q12 Why should you never use runBlocking on the main thread? L4

runBlocking blocks the calling thread until the coroutine completes. On the main thread, this causes ANR. On a dispatcher thread, it steals a thread from the pool, potentially causing thread starvation. Valid uses: fun main(), JUnit tests, and bridging to legacy synchronous Java code.

Staff-level addition: A particularly nasty bug: runBlocking(Dispatchers.Main) called from the main thread deadlocks. The coroutine needs the main thread to run, but runBlocking is blocking the main thread waiting for the coroutine. This applies to any dispatcher where the blocking thread is the same thread the coroutine needs.

Structured Concurrency (Q13–Q18)

Q13 What is structured concurrency and why does it matter? L5

Structured concurrency means every coroutine's lifetime is bounded by its scope. Parents wait for children, cancelling a parent cancels all children, and child exceptions propagate to the parent. This prevents leaked coroutines, ensures cleanup, and makes concurrent code reasoning as simple as sequential code. Without it (GlobalScope, raw threads), tasks outlive their callers, causing leaks, crashes, and resource waste.

Q14 What is the difference between Job and SupervisorJob? Give a concrete example of when to use each. L6

Job: child failure cancels parent + all siblings. Use for all-or-nothing operations — e.g., coroutineScope { async { debit() }; async { credit() } } — if either fails, both are cancelled.

SupervisorJob: child failure only cancels that child. Use for independent operations — e.g., viewModelScope (loading user profile failing shouldn't cancel loading the feed).

Staff-level addition: supervisorScope { } creates a temporary supervisor scope inside a suspend function. This is the correct way to do independent parallel work within a function. Don't pass SupervisorJob() to launch — that breaks the parent-child relationship entirely.
Q15 What happens if you catch CancellationException and don't re-throw it? L6

You silently break cancellation. The coroutine continues running even though its scope was cancelled. This causes: memory leaks (coroutine holds references to destroyed UI), wasted resources (network calls that no one needs), and potential crashes (updating views that no longer exist). This is the #1 coroutine bug in production Android apps.

Fix: Never catch Exception broadly. Catch specific types, or always re-throw CancellationException.

Q16 Where does CoroutineExceptionHandler work and where doesn't it? L6

Works: On root coroutines created with launch. Must be in the context of the root scope or the launch call itself.

Doesn't work: On async (exceptions deferred to await()), on child coroutines (exception propagates to parent first), inside coroutineScope { } (exceptions re-thrown from the scope function).

Staff-level addition: CEH is a last-resort safety net, not a try-catch replacement. The production pattern: SupervisorJob at scope level + try-catch inside individual coroutines for expected errors + CEH for logging/crash reporting of unexpected errors. Think of CEH like Thread.UncaughtExceptionHandler.
Q17 What is the difference between coroutineScope { } and CoroutineScope()? L5

coroutineScope { } is a suspend function that creates a child scope, waits for all children, and propagates exceptions. It's used inside suspend functions for parallel decomposition. CoroutineScope(context) is a constructor that creates a standalone scope — you must manage its lifecycle and call cancel(). Using CoroutineScope() inside a suspend function breaks structured concurrency.

Q18 Why is GlobalScope bad? What should you use instead? L4

GlobalScope has no parent Job, no lifecycle awareness, and no cancellation. Coroutines launched in it live forever (until they complete or the process dies). Use viewModelScope for ViewModel work, lifecycleScope for Activity/Fragment work, or create a custom CoroutineScope with a SupervisorJob that you cancel when appropriate.

Flows & Reactive Streams (Q19–Q24)

Q19 What is the difference between a cold Flow and a hot Flow? Give examples. L5

Cold Flow (flow { }): Producer starts when collect is called. Each collector gets its own independent execution. Example: flow { emit(api.fetch()) }.

Hot Flow (SharedFlow, StateFlow): Producer runs regardless of collectors. Multiple collectors share the same emissions. Example: MutableStateFlow(initialValue).

Staff-level addition: The cold/hot distinction maps to push vs pull. Cold flows are pull-based — the collector drives the producer. Hot flows are push-based — the producer drives, collectors react. In Android, data sources (Room, DataStore) are cold flows. UI state holders (ViewModel state) are hot flows (StateFlow). Events (navigation, snackbar) are hot flows (SharedFlow with replay=0).
Q20 StateFlow vs SharedFlow — when do you use each? L5

StateFlow: For state that always has a current value. Conflates (skips duplicate values via equals()). Replays latest value to new subscribers. Use for UI state (loading/success/error).

SharedFlow: For events that may or may not have subscribers. Configurable replay and buffer. Use for one-time events (navigation, snackbar, toasts) with replay=0.

What juniors miss: Using StateFlow for one-time events causes the event to replay on configuration change (new subscriber gets the last value). Using SharedFlow(replay=0) for state means new subscribers miss the current state. Getting this wrong is a very common source of bugs.
Q21 What is flowOn and why can't you use withContext inside flow { }? L5

flowOn changes the context of upstream operators by introducing a buffer (channel) between upstream and downstream. Using withContext inside flow { } would violate the context preservation invariant — the flow body must run in a single, predictable context. The runtime throws IllegalStateException if you try.

Staff-level addition: Context preservation exists because Flow's sequential nature (emit suspends until collector processes) depends on both sides running in a predictable context. flowOn breaks this sequential contract by introducing a channel buffer, which is why it's the only sanctioned way to change upstream context. The buffer also means flowOn has a performance cost — use it once, not on every operator.
Q22 Explain the difference between buffer, conflate, and collectLatest for backpressure. L6

buffer(n): Producer runs ahead, buffering up to n values. Consumer processes at its own pace. No values dropped.

conflate(): If consumer is busy, intermediate values are dropped. Consumer always gets the latest available value when it's ready.

collectLatest { }: When a new value arrives while the previous collection lambda is still running, the previous lambda is cancelled and restarted with the new value.

Staff-level addition: collectLatest is the default choice for Android UI. When state changes rapidly (typing, scrolling), you want to render the latest state, not queue up intermediate renders. conflate is for sensor data where you want the latest reading. buffer is for when every value matters (analytics events, database writes).
Q23 How does callbackFlow work and when do you use it? L5

callbackFlow creates a Flow backed by a Channel. You register a callback in the lambda, use trySend() to emit values from the callback, and awaitClose { } to clean up when the flow is cancelled. It's the standard way to convert callback-based APIs (location, sensors, Firebase listeners) into Flows.

awaitClose is mandatory — without it, the flow completes immediately after the lambda returns, before any callbacks fire.

Q24 Where does the catch operator catch exceptions and where doesn't it? L6

catch catches exceptions from upstream operators only. It does NOT catch exceptions thrown in the collect lambda (which is downstream). To catch collector exceptions, use onEach { } .catch { } .launchIn(scope) — this moves the "collection" logic into an upstream operator where catch can see it.

Android Integration (Q25–Q28)

Q25 Why was launchWhenStarted deprecated? What replaced it? L6

launchWhenStarted suspends the coroutine when lifecycle drops below STARTED, but doesn't cancel upstream Flow collection. Hot flows keep emitting into a buffer, wasting resources. Cold flows keep their producer running. repeatOnLifecycle(STARTED) replaced it — it cancels the entire block when lifecycle drops below the target state, then re-launches when it re-enters. This properly stops upstream producers.

Q26 What is the difference between collectAsState and collectAsStateWithLifecycle in Compose? L5

collectAsState() collects for the entire composition lifetime — even when the app is in the background. collectAsStateWithLifecycle() stops collection when the lifecycle drops below STARTED (default), matching repeatOnLifecycle behavior. Always use collectAsStateWithLifecycle in Android to avoid wasting resources when backgrounded.

Q27 Explain viewModelScope internals. Why SupervisorJob + Main.immediate? L6

viewModelScope uses SupervisorJob() so one failing operation doesn't cancel unrelated operations in the same ViewModel. It uses Dispatchers.Main.immediate because most ViewModel work ends with a UI state update — immediate avoids an extra message queue hop when already on the main thread. The scope is stored as a ViewModel tag (survives config changes) and cancelled in ViewModel.clear().

Q28 How do you make coroutines testable? What is the dispatcher injection pattern? L6

Inject dispatchers as constructor parameters instead of hardcoding Dispatchers.IO. In tests, pass a TestDispatcher which uses virtual time (no real delays). Use runTest for test functions, advanceUntilIdle() to process pending coroutines, and Turbine library for Flow testing. For viewModelScope, use Dispatchers.setMain(testDispatcher) in a test rule.

Staff-level addition: StandardTestDispatcher queues coroutines (explicit control via advanceUntilIdle). UnconfinedTestDispatcher executes eagerly (useful for Flow collection in tests). The choice affects test determinism — Standard gives you full control, Unconfined is more convenient but can mask ordering bugs.

Concurrency Patterns & Design (Q29–Q30)

Q29 Why should you never use synchronized inside a coroutine? What should you use instead? L6

synchronized blocks the thread while waiting for the lock. On a bounded thread pool like Dispatchers.Default (= CPU cores), blocking one thread means one fewer core for all coroutines. With enough contention, you get thread starvation — all pool threads are blocked waiting for locks, and no coroutine can make progress.

Use Mutex instead — it suspends the coroutine (freeing the thread) while waiting for the lock. Or use thread confinement (limitedParallelism(1)) to avoid locks entirely.

Staff-level addition: There's one exception: if the critical section is extremely short (no suspension points, just a few assignments), synchronized can be faster than Mutex because it avoids the suspend/resume overhead. But this is a micro-optimization — default to Mutex unless profiling shows it's a bottleneck. Also note: Mutex is NOT reentrant, unlike synchronized.
Q30 Design a search-as-you-type feature using coroutines and Flow. Walk through the complete architecture. L6+

Architecture:

// ViewModel
class SearchViewModel(private val repo: SearchRepository) : ViewModel() {
    private val _query = MutableStateFlow("")
    val query: StateFlow<String> = _query

    val results: StateFlow<SearchState> = _query
        .debounce(300)
        .filter { it.length >= 2 }
        .distinctUntilChanged()
        .flatMapLatest { q ->
            flow {
                emit(SearchState.Loading)
                val results = repo.search(q)
                emit(SearchState.Success(results))
            }.catch { emit(SearchState.Error(it.message)) }
        }
        .stateIn(viewModelScope, SharingStarted.WhileSubscribed(5000), SearchState.Idle)

    fun onQueryChanged(text: String) { _query.value = text }
}

// Activity/Fragment
lifecycleScope.launch {
    repeatOnLifecycle(Lifecycle.State.STARTED) {
        viewModel.results.collect { state -> renderState(state) }
    }
}

// Or in Compose
@Composable fun SearchScreen(vm: SearchViewModel) {
    val state by vm.results.collectAsStateWithLifecycle()
    // render state
}

Why this design works:

  • debounce(300) — avoids API call on every keystroke
  • distinctUntilChanged() — avoids re-searching same query
  • flatMapLatest — cancels previous search when new query arrives
  • stateIn(WhileSubscribed(5000)) — keeps upstream alive for 5s after last subscriber (survives config change)
  • repeatOnLifecycle — stops collection when backgrounded
Staff-level addition: The WhileSubscribed(5000) timeout is a production optimization. Without it, rotating the device restarts the entire flow chain. With 5s timeout, the upstream survives a quick config change. The 5000ms value is a heuristic — long enough for rotation, short enough to not waste resources. Google's Now in Android sample uses this exact pattern.

Deep Internals & Staff-Level (Q31–Q42)

Q31 Explain the CoroutineScheduler's work-stealing algorithm. Why LIFO local and FIFO steal? L6+

The CoroutineScheduler (backing Default and IO) gives each worker thread a local deque. Tasks submitted by a coroutine running on that worker go to the local deque (no contention). External submissions go to a global lock-free MPMC queue.

When a worker's local deque is empty, it checks the global queue, then steals from other workers. Local tasks are processed LIFO (most recent first) for cache locality — the newest task likely shares cache lines with the current worker. Stolen tasks are taken FIFO (oldest first) to minimize cache pollution on the victim.

Idle workers call LockSupport.park() — zero CPU. New work triggers LockSupport.unpark() on one parked worker.

Staff-level addition: This is the same algorithm used by Java's ForkJoinPool, Go's goroutine scheduler, and Tokio (Rust). The key insight: LIFO-local gives temporal locality (recently submitted = recently accessed data), FIFO-steal gives spatial fairness (oldest tasks waited longest). CPU workers are limited to core count; BLOCKING workers expand beyond 64 for IO. limitedParallelism(n) on IO creates additional threads beyond 64 (additive), while on Default it creates a view within the existing pool (restrictive).
Q32 What happens-before guarantees do coroutines provide? When do you still need synchronization? L6

Coroutines guarantee happens-before at three points:

  1. Suspension point: All writes before a suspension point are visible after resumption, even on a different thread.
  2. Coroutine start: All writes before launch/async are visible inside the new coroutine.
  3. Coroutine completion: All writes inside a coroutine are visible after join()/await().

You still need synchronization when two concurrent coroutines (e.g., two launch blocks) access the same mutable state. There's no happens-before between parallel coroutines. Use Mutex, Atomic*, or thread confinement.

Staff-level addition: The happens-before at suspension points is implemented by the coroutine runtime flushing memory (similar to a volatile write/read pair) when dispatching a continuation to a different thread. This means sequential coroutine code is safe without volatile — but the moment you share mutable state between concurrent coroutines, you're in classic multi-threaded territory.
Q33 How do you debug coroutines in production? What tools are available? L6

Debug builds:

  • -Dkotlinx.coroutines.debug — appends coroutine name + ID to thread name (e.g., "worker-3 @user-fetch#42"). Low overhead, safe for debug.
  • DebugProbes.install() — captures creation stack traces. High overhead, test/debug only. DebugProbes.dumpCoroutines() shows all active coroutines with state and stack.

Production:

  • CoroutineName("meaningful-name") on every scope — shows up in crash reports' thread name field.
  • CoroutineExceptionHandler on root scopes — catches and reports uncaught exceptions to Crashlytics/Bugsnag.
  • Structured logging at coroutine entry/exit with coroutine context info.
  • Custom ContinuationInterceptor wrappers for distributed trace ID propagation.
What juniors miss: They don't add CoroutineName to their scopes, so crash reports show generic thread names like "DefaultDispatcher-worker-3" with no context about which coroutine crashed. Adding CoroutineName is zero-cost and dramatically improves debuggability.
Q34 How does Flow operator chaining work internally? Why is it zero-cost? L6+

Each operator (map, filter, etc.) creates a new Flow that wraps the upstream. When collect() is called on the outermost Flow, it triggers a chain of nested collect() calls inward to the source. When the source emit()s, the value flows outward through each operator's lambda.

It's zero-cost because operators are inline functions — the Kotlin compiler inlines the lambdas, so the entire chain compiles to nested function calls with zero object allocation per element. The only allocation is the Flow wrapper objects, created once when the chain is built.

AbstractFlow enforces two invariants via SafeCollector: (1) context preservation — emit() must happen in the same context as collect(), and (2) no concurrent emissions.

Staff-level addition: This is fundamentally different from RxJava, which allocates Observable + Observer + Disposable per operator. Flow's inline approach means a 10-operator chain has the same per-element overhead as a direct function call. This is why Flow benchmarks 2-5x faster than RxJava for equivalent pipelines. The tradeoff: flowOn breaks the zero-cost model by introducing a Channel buffer — use it sparingly.
Q35 Why does flowOn introduce a buffer? What's the performance implication? L6

flowOn changes the upstream context, but downstream must stay on the collector's context. Since two different dispatchers can't share the same sequential emit→collect call chain, flowOn inserts a Channel between upstream and downstream. Upstream runs on the specified dispatcher and sends values into the channel. Downstream reads from the channel on the collector's dispatcher.

Performance cost: Channel allocation, context switch on every emission, potential backpressure if downstream is slow. Multiple flowOn calls create multiple channels — each adding latency. Best practice: use one flowOn at the source, not per operator.

Q36 How does StateFlow.update() achieve thread safety without locks? L6

StateFlow.update { } uses a CAS (compare-and-swap) loop. It reads the current value, applies the transform function, then attempts an atomic compare-and-set. If another thread changed the value between read and write, the CAS fails and the loop retries with the new current value.

// Simplified internal:
fun update(function: (T) -> T) {
    while (true) {
        val cur = value
        val new = function(cur)
        if (compareAndSet(cur, new)) return
    }
}

This is lock-free and wait-free for uncontended cases. The transform function may be called multiple times under contention — keep it pure (no side effects).

Staff-level addition: The equality-based conflation in StateFlow also uses this CAS mechanism. If new.equals(cur) is true, the update is a no-op — no subscribers are notified. This is why data class state objects with the same fields don't trigger recomposition in Compose, which is usually desirable but can surprise you when you want to force a refresh.
Q37 You're migrating a large app from RxJava to coroutines. What's your strategy? L6+

Bottom-up migration:

  1. Data layer first: Convert Single<T> → suspend fun(): T, Observable<T> → Flow<T> in repositories and data sources.
  2. Bridge at boundaries: Use kotlinx-coroutines-rx3 bridge: .asFlow(), .await(), .asObservable() at the boundary between migrated and unmigrated layers.
  3. Domain layer: Convert use cases. Replace subscribeOn/observeOn with flowOn and dispatcher injection.
  4. ViewModel last: Replace CompositeDisposable with viewModelScope.launch + collect. Replace BehaviorSubject with StateFlow, PublishSubject with SharedFlow(replay=0).
  5. Remove RxJava dependency only when all usages are gone.
Staff-level addition: Key mapping: switchMap → flatMapLatest, flatMap → flatMapMerge, concatMap → flatMapConcat. The biggest gotcha: RxJava's onError terminates the stream. Flow's catch only catches upstream. Plan error handling carefully during migration. Also: never mix RxJava schedulers and coroutine dispatchers in the same class — the mental overhead is a maintenance nightmare.
Q38 What are the three coroutine performance anti-patterns visible in profiling? L6
  1. Main thread stalls: Coroutines on Dispatchers.Main doing computation or blocking I/O. Visible as long blocks on the main thread in systrace. Fix: withContext(Dispatchers.Default/IO).
  2. Thread pool exhaustion: All DefaultDispatcher-worker-* threads in WAITING/BLOCKED state. Cause: synchronized blocks or blocking calls inside coroutines on Default. Fix: use Mutex or move to Dispatchers.IO.
  3. Excessive context switching: Many withContext calls causing thread hops visible as tiny gaps in the CPU profiler timeline. Fix: batch work in a single withContext block.
Staff-level addition: At Google, teams use custom Perfetto trace points (Trace.beginSection/endSection) around key coroutine operations. This makes coroutine work visible in systrace alongside framework events (VSYNC, measure/layout/draw). The combination lets you see exactly which coroutine work is stealing frame budget.
Q39 Explain the difference between SharingStarted.WhileSubscribed(5000) and SharingStarted.Eagerly. When do you use each? L6

Eagerly: Starts upstream immediately when stateIn/shareIn is called. Never stops. Use for data that should always be fresh (e.g., user auth state, connectivity status).

Lazily: Starts upstream on first subscriber. Never stops after that. Use for expensive data that should only be computed if needed.

WhileSubscribed(stopTimeout, replayExpiration): Starts on first subscriber, stops stopTimeout ms after last subscriber leaves. Use for screen-specific data — the 5000ms timeout survives configuration changes (rotation takes ~1-2s) but stops wasting resources when the user navigates away.

Staff-level addition: WhileSubscribed(5000, 0) is the Google-recommended default for screen-level StateFlows. The second parameter (replayExpiration) controls how long the cached value is kept after upstream stops. Setting it to 0 means the cached value is kept forever (until the ViewModel is cleared). Setting it to a duration means stale data is cleared — useful for security-sensitive data that shouldn't linger.
Q40 What is the context preservation invariant in Flow and why does withContext inside flow { } throw? L6

emit() is a suspend call on the FlowCollector. The collector expects to run in its own context (e.g., Dispatchers.Main). If emit() is called from a different context (e.g., inside withContext(Dispatchers.IO)), the collector's code would suddenly execute on IO — violating the caller's expectations.

AbstractFlow enforces this via SafeCollector, which checks the coroutine context on every emit(). If the context changed, it throws IllegalStateException.

Correct patterns: Use withContext for work but emit outside it, or use flowOn which handles the context boundary with a Channel buffer.

Q41 How does repeatOnLifecycle work internally? Why is it better than launchWhenStarted? L6

repeatOnLifecycle(STARTED) registers a LifecycleObserver. When lifecycle enters STARTED, it launches a new coroutine to execute the block. When lifecycle drops below STARTED, it cancels that coroutine. When lifecycle re-enters STARTED, it launches a fresh coroutine.

launchWhenStarted only suspends the coroutine when lifecycle drops below STARTED — it doesn't cancel upstream. For hot flows, the upstream keeps emitting into a buffer (memory leak). For cold flows, the producer keeps running (resource waste). repeatOnLifecycle cancels the entire collection, which signals upstream to stop.

Staff-level addition: The internal implementation uses Lifecycle.repeatOnLifecycle() which is a suspend function that never returns (it loops forever, launching and cancelling blocks as the lifecycle transitions). This is why it must be called inside lifecycleScope.launch { } — it needs a coroutine to suspend in. The cancellation on STOPPED is what makes cold flows restart from scratch on STARTED, which is correct behavior for things like database queries (re-query to get fresh data).
Q42 You notice your app has 80+ threads at steady state. Diagnose and fix. L6+

Diagnosis:

  1. Thread dump: adb shell kill -3 <pid> or Android Studio Profiler → CPU → Thread view. Identify thread names.
  2. Common culprits:
    • OkHttp ConnectionPool — default creates threads per connection. Fix: share a single OkHttpClient instance.
    • DefaultDispatcher-worker-* — should be ≤ CPU cores. If more, something is creating custom dispatchers.
    • Dispatchers.IO threads — up to 64. If more, limitedParallelism() is creating additional threads.
    • Legacy Executors.newCachedThreadPool() — unbounded, creates threads on demand.
    • Third-party SDKs (Firebase, analytics) — often create their own thread pools.

Fixes:

  • Replace custom thread pools with coroutine dispatchers
  • Use limitedParallelism() to cap concurrency instead of creating new pools
  • Share a single OkHttpClient and Retrofit instance app-wide
  • Audit third-party SDKs for thread creation
  • Target: <30 threads at steady state for a well-optimized app
Staff-level addition: At Google, thread count is a monitored performance metric. Apps with 50+ threads get flagged. The cost isn't just memory (~1MB per thread stack) — it's kernel scheduler contention, lock contention, and CPU cache thrashing. The L6 answer: centralize all async work on coroutine dispatchers, audit third-party SDKs, and monitor thread count in production via Android Vitals or custom metrics.

async + Scope Failure Semantics — The Real Model (Q43–Q45)

Q43 Inside coroutineScope { }, when does an async { } failure actually propagate — at throw time or at await()? L6+

At throw time — immediately. This is the most common L6 misconception.

The popular mental model is: "async defers the exception until you call await()." That is only true inside a supervisorScope. Inside a coroutineScope, the exception propagates structurally the moment the child fails:

  1. Child (async block) throws.
  2. Exception is immediately propagated up to the parent (coroutineScope).
  3. Parent cancels all sibling coroutines.
  4. Scope completes exceptionally.
  5. await() — if it would have been reached — rethrows the stored exception to the caller.

The critical insight: await() is irrelevant to cancellation. It only controls when you observe the exception, not when it happens. Siblings are cancelled before you ever get a chance to call await().

// coroutineScope = fail-fast system: one async failure kills the whole scope
coroutineScope {
    val a = async { fetchUser() }   // throws NetworkException
    val b = async { fetchFeed() }   // gets cancelled immediately
    a.await()  // rethrows NetworkException — but b was already dead
    b.await()  // never reached
}
What juniors miss: They think wrapping calls in async inside a coroutineScope gives independent error handling per-call. It doesn't. You don't control failure timing — the scope does. Use supervisorScope if you need fault isolation.
Staff-level addition: This is implemented via JobSupport.childCancelled(). When a child completes exceptionally, it calls childCancelled() on its parent Job. A regular Job returns true — "yes, cancel me too." A SupervisorJob overrides this and returns false. The exception is not merely "stored in the Deferred waiting for await()" — it is a structural cancellation signal that propagates through the Job hierarchy immediately, independently of any caller calling await().
Q44 What fundamentally changes in supervisorScope when an async { } child fails? L6+

Failure becomes data, not a structural event.

A supervisorScope changes exactly one rule: a child's failure does not propagate upward as cancellation. The exception is captured inside the Deferred and stays there until observed.

    val feed = b.await()             // works fine
    val user = try {

    } catch (e: NetworkException) {

Critical detail often missed: If you never call await() on a failed Deferred inside a supervisorScope, the exception is silently lost — it will not surface anywhere. Always either await() with a try-catch, or install a CoroutineExceptionHandler on the scope as a safety net.
Staff-level addition: viewModelScope uses SupervisorJob, making it behave like supervisorScope for all launched coroutines. This is intentional: one ViewModel operation failing should never cancel unrelated operations. The trade-off: you must explicitly handle each coroutine's failure — the scope won't surface it for you.
Q45 What happens to an async coroutine's own children when it fails? Explain the "every coroutine is a mini-scope" model. L6+

Every coroutine is a mini scope for its own children — regardless of what the outer scope is. When an async block's logic throws:

  1. The async coroutine cancels all its own children immediately (always fail-fast within itself).
  2. The async coroutine completes exceptionally.
  3. It reports failure to its parent.
  4. The parent reacts based on its own scope type (coroutineScope = cancel all siblings, supervisorScope = isolate).
supervisorScope {
    val result = async {
        val x = launch { step1() }  // child of this async
        val y = launch { step2() }  // child of this async
        throw RuntimeException("inner failure")
        // x and y are cancelled immediately — mini-scope fail-fast
    }
    // outer supervisorScope: other siblings of `result` are NOT cancelled
    val other = async { loadSettings() }  // survives
}

Two independent levels, two independent rules:

  • Inner children → always cancelled immediately (the async coroutine is a regular Job for its own children)
  • Outer siblings → determined by parent scope (coroutineScope = die too, supervisorScope = survive)
Staff-level addition — the unified model:

The scope decides cancellation behavior, not the coroutine builder.

async and launch are just ways to start a coroutine. Whether failure propagates sideways depends entirely on whether the parent Job is a regular Job (fail-fast) or a SupervisorJob (isolate). This is why passing SupervisorJob() into a launch(SupervisorJob()) { } call is an anti-pattern — it detaches the new coroutine from the parent hierarchy entirely rather than providing scoped isolation. Use supervisorScope { } instead.
