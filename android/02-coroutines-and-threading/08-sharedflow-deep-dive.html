SharedFlow Deep Dive

Anatomy of a production-grade SharedFlow config, use-case selection, emit vs tryEmit, the Event Gap problem, sticky events, delivery guarantees, and SharedFlow vs Channel

1. The Anatomy of Your Config

A MutableSharedFlow has three constructor parameters that together define how it behaves under pressure. Each one solves a specific real-world problem. Here is the production-grade configuration and why every knob is set the way it is:

MutableSharedFlow<UiEvent>(
    replay              = 0,
    extraBufferCapacity = 64,
    onBufferOverflow    = BufferOverflow.DROP_OLDEST
)

replay = 0 — The "One-Time" Event

The Problem: If you set replay = 1, and the user rotates their phone, the new Activity will subscribe and immediately receive the last event again — e.g., the "Delete Successful" Toast pops up twice.

The Fix: replay = 0 ensures that events are hot and ephemeral. Only active listeners get the message. New subscribers see nothing from the past.

Common Mistake
Setting replay = 1 on a navigation or snackbar event flow is a classic rotation bug. The new Activity subscribes, the replay buffer fires the old "Navigate to Detail" event, and the user is launched somewhere unexpected.

extraBufferCapacity = 64 — The "Safety Net"

The Problem: In a standard Flow, if the collector is busy (e.g., processing a heavy UI animation), the producer suspends (pauses). If your producer is a UI interaction handler, you never want to suspend the UI thread waiting for a downstream collector to catch up.

The Fix: This creates a "waiting room" for 64 events. The producer can keep emitting without waiting for the collector to finish processing each one.

onBufferOverflow = BufferOverflow.DROP_OLDEST — The "Pressure Valve"

The Problem: What happens when the 65th event arrives and the collector is still busy? Without an overflow strategy, emit() would suspend — blocking the producer.

The Fix: DROP_OLDEST ensures that emit() never suspends. It evicts the oldest buffered message to make room for the newest. This is the right call for UI events, where the user's latest intent is always more important than stale queued events.

Staff-Level Insight
DROP_OLDEST is not the only option. DROP_LATEST silently discards the new event if the buffer is full — useful when you want to preserve earlier events (e.g., ordered command queues). SUSPEND (the default) creates back-pressure by blocking the producer, which is appropriate for pipeline-style data processing where no event can be skipped.

2. Practical Use Cases: Which Config for Which Problem?

Choosing a SharedFlow configuration is a design decision. The wrong choice leads to silent bugs that only surface in production under load or after device rotation. Here are the three canonical patterns:

Case A: The "Toast / Navigation" Problem

You want to show a message or trigger a navigation event exactly once, regardless of when the subscriber attaches.

SettingValueReason
replay0Don't replay the event to new subscribers (e.g. after rotation)
extraBufferCapacity64Absorb bursts without blocking the ViewModel
onBufferOverflowDROP_OLDESTLatest intent wins; never suspend the producer
Key Insight
If the user isn't on the screen, they missed the transient feedback — and that is usually acceptable. A toast that fires while the app is backgrounded should simply be dropped, not queued indefinitely.

Case B: The "Latest Score" Problem (Real-Time State)

You have a sports app. When a user opens the screen they need the current score immediately, then receive live updates.

SettingValueReason
replay1New subscribers get the most recent value immediately upon tuning in
extraBufferCapacity0 or smallState is replaced, not queued — old values are irrelevant
onBufferOverflowDROP_OLDESTAlways prefer the freshest state
Staff-Level Insight
If replay = 1 with a single state value is all you need, you should just use StateFlow. It is semantically clearer, its initial value is non-nullable, and it enforces the "always has a value" contract explicitly.

Case C: The "High-Frequency Sensor" Problem

You are tracking GPS coordinates or accelerometer data. Events arrive faster than the UI can render frames.

SettingValueReason
replay0A GPS coordinate from 2 seconds ago is stale and useless
extraBufferCapacity128Large buffer absorbs sensor bursts between UI frames
onBufferOverflowDROP_OLDESTUI should always draw the freshest position, not catch up on history

3. Clean Architecture Implementation

In a clean architecture, we always hide the mutable MutableSharedFlow behind an immutable SharedFlow interface. This prevents the UI layer from "cheating" by injecting its own events directly — all mutations must go through the ViewModel's public API.

sealed class UiEvent {
    data class ShowSnackbar(val message: String) : UiEvent()
    data class Navigate(val route: String) : UiEvent()
    data object PlaySuccessSound : UiEvent()
}

class MyViewModel : ViewModel() {

    // 1. Private mutable version — the Producer
    private val _events = MutableSharedFlow<UiEvent>(
        replay              = 0,
        extraBufferCapacity = 64,
        onBufferOverflow    = BufferOverflow.DROP_OLDEST
    )

    // 2. Public immutable version — the Consumer's view
    val events: SharedFlow<UiEvent> = _events.asSharedFlow()

    fun onUserClickedDelete(itemId: String) {
        viewModelScope.launch {
            repository.delete(itemId)
            // tryEmit is safe here because of the buffer + DROP_OLDEST config
            _events.tryEmit(UiEvent.ShowSnackbar("Item Deleted"))
        }
    }

    fun onUserClickedOpenDetail(itemId: String) {
        // tryEmit is a regular function — safe to call outside a coroutine
        _events.tryEmit(UiEvent.Navigate("detail/$itemId"))
    }
}
// Collecting safely in the UI layer (Fragment / Composable)
viewLifecycleOwner.lifecycleScope.launch {
    repeatOnLifecycle(Lifecycle.State.STARTED) {
        viewModel.events.collect { event ->
            when (event) {
                is UiEvent.ShowSnackbar -> showSnackbar(event.message)
                is UiEvent.Navigate     -> findNavController().navigate(event.route)
                UiEvent.PlaySuccessSound -> soundManager.playSuccess()
            }
        }
    }
}
Android-Specific Warning
Always collect SharedFlow inside repeatOnLifecycle(STARTED). Using lifecycleScope.launch { flow.collect {} } without repeatOnLifecycle will keep collecting events even when the Fragment is in the back stack, causing leaked subscriptions and incorrect UI updates.

4. emit() vs tryEmit()

With a buffer and a non-SUSPEND overflow strategy configured, you gain access to a more powerful tool than emit().

FunctionTypeBehaviourWhen to use
emit(value) suspend Suspends if the buffer is full and overflow is SUSPEND. With DROP_OLDEST, it will technically never need to suspend. When you are already in a coroutine and want explicit back-pressure control
tryEmit(value) regular Returns true if delivered or buffered. Returns false if the buffer is full and overflow would have caused a drop. Never suspends. Non-coroutine contexts (click listeners, callbacks), or any time you need a non-blocking fire-and-forget
Rule of Thumb
With a DROP_OLDEST SharedFlow, prefer tryEmit() for UI events. It communicates intent clearly: "send this if possible, and never block." Use emit() when you are inside a coroutine pipeline where you want structured back-pressure. 
// Good: tryEmit from a non-coroutine context
button.setOnClickListener {
    // This is called on the main thread, not in a coroutine
    // tryEmit is the only correct choice here
    _events.tryEmit(UiEvent.Navigate("profile"))
}

// Also fine: emit inside a coroutine
viewModelScope.launch {
    val result = repository.doWork()
    _events.emit(UiEvent.ShowSnackbar(result.message))
}

5. SharedFlow vs StateFlow — The Decision

This is one of the most common sources of confusion in Android architecture. The distinction is conceptual before it is technical:

PropertyStateFlowSharedFlow (replay=0)
Has an initial value?Yes (required)No
Replays to new subscribers?Always (1 value)Never (with replay=0)
Deduplication?Yes — skips equal valuesNo — emits every call
Conceptual modelState (observable value)Event (one-time command)
LiveData equivalentYesNo direct equivalent
Primary use caseUI state (content, loading)One-shot effects (navigate, toast)
Staff-Level Insight — The "Undelivered Event" Problem
SharedFlow(replay=0) is not a perfect solution for one-time events. If the UI is in the STOPPED state (user pressed Home) when an event is emitted, and you are using repeatOnLifecycle(STARTED), the event will be silently dropped. For navigation events this is usually acceptable. For events with side-effects like writing to a database, it is not. In those cases, move the side-effect into the ViewModel and only emit a "confirmation" event to the UI.

Combining Both: The Two-Stream Pattern

A production ViewModel typically exposes both a StateFlow for UI state and a SharedFlow for one-time effects — they serve different purposes and should never be conflated into one stream.

class ProductDetailViewModel : ViewModel() {

    // StateFlow — "What does the screen currently look like?"
    private val _uiState = MutableStateFlow<ProductDetailState>(ProductDetailState.Loading)
    val uiState: StateFlow<ProductDetailState> = _uiState.asStateFlow()

    // SharedFlow — "What should happen exactly once?"
    private val _events = MutableSharedFlow<ProductDetailEvent>(
        replay = 0,
        extraBufferCapacity = 64,
        onBufferOverflow = BufferOverflow.DROP_OLDEST
    )
    val events: SharedFlow<ProductDetailEvent> = _events.asSharedFlow()

    fun onAddToCart(product: Product) {
        viewModelScope.launch {
            _uiState.update { it.copy(isAddingToCart = true) }      // State change
            val success = cartRepository.add(product)
            _uiState.update { it.copy(isAddingToCart = false) }     // State change
            if (success) {
                _events.tryEmit(ProductDetailEvent.ShowAddedToCartSnackbar) // One-shot event
            }
        }
    }
}

6. The "Event Gap" Problem — Intent vs. Side-Effect

This is the most sophisticated challenge in Android event architecture — the gap between a junior developer who thinks "It works on my emulator" and a Staff engineer who knows "It will fail in the background."

When you use repeatOnLifecycle(STARTED), the coroutine is cancelled the moment the app goes to the background. Since SharedFlow(replay=0) stores nothing, any event emitted during that blackout window is silently lost forever.

The "Junior" Mistake — UI as the Executor

The critical error is delegating a side-effect (like saving to a database) to the UI layer via an event. The UI may not be listening:

// ViewModel
fun onSaveClicked() {
    _events.emit(UiEvent.SaveToDatabase) // ❌ DANGEROUS
}

// UI (Activity/Fragment)
lifecycleScope.launch {
    repeatOnLifecycle(Lifecycle.State.STARTED) {
        viewModel.events.collect { event ->
            if (event is UiEvent.SaveToDatabase) {
                repository.save() // ❌ Will NOT run if app is in the background
            }
        }
    }
}
Critical Bug
The user taps "Save," presses Home before the collector processes the event, and the data is silently never saved. There is no crash, no log, no indication anything went wrong.

The "Staff" Approach — ViewModel Ownership

The rule is: the ViewModel performs any action that has a side-effect. The Flow only communicates the result to the UI for visual feedback. viewModelScope is tied to the ViewModel's lifecycle, not the UI's — it persists even when the user backgrounds the app.

// ViewModel
fun onSaveClicked() {
    viewModelScope.launch {
        // 1. Perform the side-effect in viewModelScope — survives backgrounding
        val result = repository.save()

        // 2. Only then notify the UI (this is fine to drop if UI is gone)
        _events.tryEmit(UiEvent.ShowSaveSuccess(result.message))
    }
}

// UI — safe because the event is just a visual notification, not the actual work
lifecycleScope.launch {
    repeatOnLifecycle(Lifecycle.State.STARTED) {
        viewModel.events.collect { event ->
            if (event is UiEvent.ShowSaveSuccess) {
                showSnackbar(event.message) // UI feedback only
            }
        }
    }
}
Staff-Level Insight
The key mental model: never let the UI be the primary executor of business logic. The UI is a display surface. Any work that must complete regardless of UI visibility belongs in viewModelScope (or WorkManager for truly durable work). The Flow is only the result notification channel.

7. When the Event Must Be Seen — The "Sticky" Problem

What if the event truly must reach the UI — for example, a critical "Account Suspended" dialog that cannot be silently dropped? SharedFlow(replay=0) is the wrong tool. You have two better options.

Option A: The Buffer Hack (Fragile — Avoid)

Increasing extraBufferCapacity buys more time, but it is a guessing game. If 10 events fire while the app is backgrounded and your buffer holds 5, you still lose half. This provides false confidence.

Warning
Do not rely on a large buffer to solve the sticky event problem. The buffer exists to smooth out rate mismatches between producer and consumer, not as persistent storage for missed events.

Option B: The State-Based Event (Recommended)

Model the pending event as part of your UiState. State persists in StateFlow and is always replayed to new subscribers. The UI clears it after displaying, preventing re-display on the next recomposition.

// The state carries the pending notification
data class UiState(
    val content: List<Item> = emptyList(),
    val isLoading: Boolean = false,
    val pendingAlert: AlertMessage? = null  // null = nothing to show
)

data class AlertMessage(val id: UUID = UUID.randomUUID(), val text: String)

// ViewModel
fun onAccountSuspended(reason: String) {
    _uiState.update { it.copy(
        pendingAlert = AlertMessage(text = reason)
    )}
}

// Called by UI after showing the dialog
fun onAlertDismissed() {
    _uiState.update { it.copy(pendingAlert = null) }
}

// UI
lifecycleScope.launch {
    repeatOnLifecycle(Lifecycle.State.STARTED) {
        viewModel.uiState.collect { state ->
            state.pendingAlert?.let { alert ->
                showDialog(alert.text) {
                    viewModel.onAlertDismissed() // Clear after shown
                }
            }
        }
    }
}
Staff-Level Insight — Idempotency via UUID
The UUID id on AlertMessage is critical. StateFlow deduplicates by equality — if you emit the same AlertMessage object twice, the second emission is swallowed. The UUID ensures every new alert is a distinct object, so the same text can appear again for a different occurrence. Track shown IDs in the UI to prevent re-showing the same dialog on recomposition.

8. Delivery Guarantee Comparison

No single tool covers every scenario. Choosing the right one depends entirely on the delivery contract you need:

Tool Delivery during Background Multiple Collectors Best For
SharedFlow(replay=0) Lost All active collectors receive it (broadcast) Navigation, non-critical Snackbars, sounds
SharedFlow(replay=1) Replayed once All collectors, including new ones, get latest Latest value that new subscribers should see immediately
StateFlow Always present All collectors see current value UI state, toggles, visibility, loading flags
State in UiState Persists until cleared All collectors see it Critical dialogs, "must-be-seen" alerts
Channel Buffered (waits) Only one collector receives each item (handoff) Ordered command queues, 1-to-1 work dispatch

9. The Channel Alternative

If you truly need a one-time event that waits for the collector to come back online instead of being dropped, a Channel is semantically safer than a SharedFlow(replay=0).

A Channel will hold onto the event (suspending the sender or buffering it) until a collector is ready to receive it. This creates a handoff guarantee rather than a broadcast:

// ViewModel
private val _navigationChannel = Channel<String>(Channel.BUFFERED)
val navigationEvents = _navigationChannel.receiveAsFlow()

fun onUserClickedProfile(userId: String) {
    viewModelScope.launch {
        _navigationChannel.send("profile/$userId")
        // send() suspends if the buffer is full, ensuring no event is lost
    }
}

// UI
lifecycleScope.launch {
    repeatOnLifecycle(Lifecycle.State.STARTED) {
        viewModel.navigationEvents.collect { route ->
            findNavController().navigate(route)
        }
    }
}
Critical Caveat — Single Consumer Only
A Channel is a 1-to-1 queue. If you have multiple collectors (e.g., two Fragments both observing the same ViewModel), only one of them will receive each event — whichever gets scheduled first. Use SharedFlow for broadcasting; use Channel only when there is exactly one consumer and delivery must be guaranteed.

SharedFlow vs Channel — The Decision Rule

QuestionIf Yes, Use
Can the event be dropped if the UI isn't visible?SharedFlow(replay=0)
Does the event represent "current state" that new subscribers need?StateFlow
Must the event be processed exactly once, even after backgrounding?Channel or State-based event
Is there a single consumer and guaranteed delivery matters more than freshness?Channel
Is there more than one consumer (broadcast semantics)?SharedFlow
