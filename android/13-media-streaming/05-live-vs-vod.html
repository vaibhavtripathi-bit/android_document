Live Streaming vs VOD

Live window, timeline model, playlist refresh, DVR, latency modes, seek behaviour — what changes and why

1. The Fundamental Difference

VOD (Video on Demand)

  • Content is fully encoded before playback
  • Manifest is static — downloaded once
  • Segment count is fixed and known upfront
  • Full seek range available immediately
  • Timeline: fixed start and end
  • No playlist refresh needed
  • HLS: has #EXT-X-ENDLIST
  • DASH: @type="static", has @mediaPresentationDuration

Live Streaming

  • Content is encoded and published in real time
  • Manifest grows as new segments appear
  • Segment count grows; old segments may be deleted
  • Seek range = DVR window (last N minutes)
  • Timeline: fixed start, advancing "live edge"
  • Playlist must be refreshed periodically
  • HLS: no #EXT-X-ENDLIST
  • DASH: @type="dynamic", has @minimumUpdatePeriod

2. ExoPlayer's Timeline Model

ExoPlayer represents all playback state in a Timeline object. A Timeline contains Windows and Periods.

VOD Timeline: Window[0] ├── isSeekable = true ├── isDynamic = false ├── defaultPositionUs = 0 ├── durationUs = 7200_000_000 (2 hours in microseconds) └── Period[0] → Period[1] → ... (DASH periods, or single for HLS) Live Timeline: Window[0] ├── isSeekable = true (within DVR window) ├── isDynamic = true ← KEY FLAG ├── defaultPositionUs = liveEdge - targetLiveOffset (3s behind live) ├── durationUs = C.TIME_UNSET ← unknown/infinite └── Period grows as segments added, old periods removed

In your UI code, check player.isCurrentMediaItemLive (which reads Timeline.Window.isDynamic) to decide whether to show a "LIVE" badge vs a seek bar.

3. Playlist Refresh — How Live Manifests Stay Fresh

HLS Live Refresh

HlsPlaylistTracker in ExoPlayer polls the media playlist URL periodically. The refresh interval is:

  • On successful load with no new segments: wait targetDuration / 2 then retry.
  • On error: exponential backoff (500ms → 1s → 2s → … up to 5s).
  • When a new segment appears: notify downstream immediately.

DASH Live Refresh

The MPD specifies @minimumUpdatePeriod (e.g., PT4S = every 4 seconds). DashMediaSource re-fetches the MPD on this schedule and diffs the new vs old MPD to discover new segments.

L6 Insight — Playlist Refresh at IPL Scale
During IPL peak, Hotstar serves ~25 million concurrent viewers. Each viewer's player fetches the playlist every 3–6 seconds. That's 4–8 million playlist requests per second on the CDN — just for manifests, not segments. This is why CDN edge caching of manifests is critical. Live manifests must have short Cache-Control TTLs (matching targetDuration/2) so edge nodes serve stale-but-acceptable playlists.

4. DVR Window — Seeking in Live Streams

A pure live stream has no seekable range — you can only play at the live edge. A DVR (Digital Video Recorder) window lets users seek back in time within a rolling window.

Live stream with 2-hour DVR window: Timeline (growing →) │◄──── DVR window: 2 hours ────►│ Live Edge │ │ │ T-2h T-0 NOW Player's seekable range = [T-2h, T-0] → User can pause and rewind up to 2 hours into the past → Segments older than 2h are deleted from CDN storage HLS: controlled by server — old #EXTINF lines removed from media playlist DASH: @timeShiftBufferDepth="PT2H" in MPD 
// Detecting live + DVR in ExoPlayer
val window = Timeline.Window()
player.currentTimeline.getWindow(player.currentMediaItemIndex, window)

val isLive = window.isDynamic
val hasDvr = window.isSeekable && window.durationUs != C.TIME_UNSET

// Seek to live edge
fun seekToLiveEdge() {
    player.seekToDefaultPosition()  // ExoPlayer sets this = targetLiveOffset behind edge
}

// Seek back 30 minutes from live edge
fun seekBack30Minutes() {
    val liveEdgeMs = player.duration  // in live streams, duration = distance from start
    player.seekTo((liveEdgeMs - 30 * 60 * 1000).coerceAtLeast(0))
}

// Show live indicator in UI
fun isAtLiveEdge(): Boolean {
    val window = Timeline.Window()
    player.currentTimeline.getWindow(player.currentMediaItemIndex, window)
    val distanceFromLiveEdgeMs = window.defaultPositionUs / 1000 - player.currentPosition
    return distanceFromLiveEdgeMs < 5_000  // within 5 seconds of live edge
}

5. Target Live Offset — The Latency Tradeoff

ExoPlayer doesn't play at the absolute live edge — it targets a position slightly behind, called the target live offset. This buffer absorbs network jitter.

Live edge (latest encoded segment) │ │ ← target live offset (e.g. 15 seconds) │ Playhead position │ │ ← safety buffer (2-3 segments) │ Loaded buffer start
ScenarioTarget offsetTradeoff
Cricket commentary3–8 seconds (LL-HLS)Low latency, risk of rebuffering on mobile
Sports event (IPL)10–15 secondsBalance of latency and stability
Standard live TV20–30 secondsVery stable, acceptable delay
Satellite-equivalent45+ secondsHighly reliable, traditional broadcast feel
// Configure target live offset in Media3
val mediaItem = MediaItem.Builder()
    .setUri(liveStreamUrl)
    .setLiveConfiguration(
        MediaItem.LiveConfiguration.Builder()
            .setTargetOffsetMs(8_000)         // target 8s behind live edge
            .setMinOffsetMs(3_000)            // never closer than 3s (risk of stall)
            .setMaxOffsetMs(20_000)           // never farther than 20s (too stale)
            .setMinPlaybackSpeed(0.97f)       // slow down slightly to catch up buffer
            .setMaxPlaybackSpeed(1.03f)       // speed up slightly if drifting behind
            .build()
    )
    .build()
L6 Insight — Playback Speed for Latency Control
ExoPlayer uses playback speed adjustment (0.97×–1.03×) to maintain the target live offset without visible quality impact. If the player drifts too far from live edge (e.g., due to a network hiccup), it speeds up to 1.03× to close the gap. This is imperceptible to users but keeps the stream synchronized with the live event.

6. Live Edge vs Buffering — The Live Stall Problem

Live streams are uniquely vulnerable to stalls at the live edge. With VOD, the entire stream is pre-encoded — segments always exist. With live, the next segment may not be encoded yet:

Time: T+0 Encoder outputs: seg_042.ts ← just finished encoding Player needs: seg_043.ts ← NOT YET ENCODED → Player polls CDN for seg_043.ts → CDN returns 404 (segment not published yet) → Player waits 1s, retries → seg_043.ts appears: CDN returns 200 → Playback continues This "future segment wait" is normal for live. ExoPlayer handles it via playlist refresh — it does not mark 404 as an error for live streams.
Critical Bug Pattern
Do not treat 404 responses as fatal errors in live stream pipelines. ExoPlayer's DefaultLoadErrorHandlingPolicy already does the right thing — retrying 404s for live content. If you override the error handling policy, ensure 404s during live playback trigger a retry, not a fatal error.

7. Ad Insertion in Live Streams (SSAI)

Server-Side Ad Insertion (SSAI) — also called "ad stitching" — splices ads directly into the live manifest at the server. The player sees a single continuous stream; ads look like regular content segments.

Without SSAI (CSAI - Client-Side): Player plays content → detects VAST ad cue → pauses → loads ad → resumes Problem: user can skip ads with adblocker; player needs VAST SDK With SSAI: CDN/ad server stitches ads into HLS/DASH manifest ├── seg_041.ts (content) ├── seg_042.ts (content) ├── [#EXT-X-DISCONTINUITY] ← marks content→ad transition ├── ad_seg_001.ts (ad - looks like regular content to player) ├── ad_seg_002.ts (ad) ├── [#EXT-X-DISCONTINUITY] ← marks ad→content transition └── seg_043.ts (content continues) 
// Detecting ad boundaries in ExoPlayer (SSAI with HLS discontinuities)
player.addListener(object : Player.Listener {
    override fun onPositionDiscontinuity(
        oldPosition: Player.PositionInfo,
        newPosition: Player.PositionInfo,
        reason: Int
    ) {
        if (reason == Player.DISCONTINUITY_REASON_AUTO_TRANSITION) {
            // Could be ad→content or content→ad boundary
            // Use IMA SDK or custom metadata to determine which
            checkAdBoundary(newPosition.positionMs)
        }
    }

    override fun onMetadata(metadata: Metadata) {
        // HLS ID3 tags or SCTE-35 cues embedded in the stream
        // indicate upcoming ad break
        for (i in 0 until metadata.length()) {
            val entry = metadata[i]
            if (entry is EventMessage) {
                handleScte35Cue(entry)
            }
        }
    }
})

8. Live Stream Architecture — End to End

Camera/Encoder │ RTMP / SRT / WebRTC ingest ▼ Origin Ingest Server (e.g. AWS MediaLive, Wowza) │ Encodes to multiple bitrates in real-time │ Packages into HLS/DASH segments ▼ Origin Storage (S3 / GCS) │ Segments stored with short TTL ▼ CDN Edge Network (Akamai / Cloudflare / Hotstar's own CDN) │ Caches segments geographically close to viewers ▼ Android Client (ExoPlayer) ├── Fetches manifest (M3U8 / MPD) every ~3-6s ├── Downloads segments from nearest CDN edge └── ABR selects quality based on edge-to-device bandwidth

Interview Cheat Sheet — Live vs VOD

  • How does ExoPlayer know it's live? — HLS: no #EXT-X-ENDLIST. DASH: @type="dynamic". ExoPlayer sets Timeline.Window.isDynamic = true.
  • What is a DVR window? — A rolling time window (e.g. 2 hours) where users can seek back in a live stream. Old segments beyond the window are deleted from CDN. Controlled by @timeShiftBufferDepth in DASH.
  • Target live offset? — ExoPlayer plays N seconds behind the live edge for jitter absorption. Configurable via MediaItem.LiveConfiguration. Uses subtle playback speed adjustment (0.97–1.03×) to maintain it.
  • Why 404 is not fatal in live? — The next segment may not be encoded yet. ExoPlayer retries 404s for live streams via DefaultLoadErrorHandlingPolicy.
  • How to seek to live edge? — player.seekToDefaultPosition() seeks to the target live offset position.
  • SSAI vs CSAI? — SSAI: ads stitched server-side into the manifest (seamless, ad-blocker-proof). CSAI: client fetches and plays ads separately via VAST (can be blocked). Boundary marked by #EXT-X-DISCONTINUITY in HLS.
  • How to detect being at live edge in UI? — Compare player.currentPosition to Timeline.Window.defaultPositionUs. Within 5s = show "LIVE" badge in red.
  • Playlist refresh rate? — HLS: targetDuration/2. DASH: @minimumUpdatePeriod. Both must be CDN-cached with matching TTL or you create massive origin load at scale.
