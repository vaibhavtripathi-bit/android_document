MediaSession & MediaController

Background playback, lock-screen controls, notification, Bluetooth commands, Android Auto / Wear OS handoff — the complete picture

1. Why MediaSession Exists

When your app plays audio or video in the background, the system and external clients (lock screen, Bluetooth headset, Android Auto, Google Assistant, Wear OS) need a standardised way to:

  • Know what is playing (title, artist, artwork, duration).
  • Send transport controls (play, pause, skip, seek).
  • Query playback state (is it playing? at what position?).

MediaSession (Media3) is the contract that enables all of this. Your player publishes state to the session. Clients — regardless of whether they live in your app, the system UI, or another app entirely — use MediaController to read and control it.

┌─────────────────────────────────────────────────────────┐ │ External Clients │ │ Lock Screen Bluetooth Android Auto Google Assistant │ │ Wear OS Now Playing Another App │ └──────────────────────┬──────────────────────────────────┘ │ MediaController (binder IPC) ┌────────▼────────┐ │ MediaSession │ ← your app publishes here │ (Media3) │ └────────┬────────┘ │ direct call (same process) ┌────────▼────────┐ │ ExoPlayer │ └─────────────────┘

In Interviews
If asked "how did you integrate MediaSession with ExoPlayer?" and you say "I used MediaSessionConnector" — that's the old way (ExoPlayer 2.x / media-compat). The correct Media3 answer is: MediaSession.Builder(context, player).build() — no connector needed, the session is backed directly by the Player interface.

3. Setting Up MediaSession in Media3

// In your playback Service
class PlaybackService : MediaSessionService() {

    private lateinit var player: ExoPlayer
    private lateinit var mediaSession: MediaSession

    override fun onCreate() {
        super.onCreate()
        player = ExoPlayer.Builder(this).build()

        mediaSession = MediaSession.Builder(this, player)
            // Custom command handling (e.g., "shuffle", "repeat", app-specific actions)
            .setCallback(MediaSessionCallback())
            // Session ID — useful if you run multiple sessions (e.g., music + podcast)
            .setId("main_playback")
            .build()
    }

    // MediaSessionService requires this — returns session to incoming controllers
    override fun onGetSession(controllerInfo: MediaSession.ControllerInfo): MediaSession =
        mediaSession

    override fun onDestroy() {
        mediaSession.release()
        player.release()
        super.onDestroy()
    }

    // Handle custom/extended commands
    inner class MediaSessionCallback : MediaSession.Callback {
        override fun onConnect(
            session: MediaSession,
            controller: MediaSession.ControllerInfo
        ): MediaSession.ConnectionResult {
            // Accept all controllers (lock screen, BT, Auto...)
            // In production: validate packageName + signature for security
            return MediaSession.ConnectionResult.accept(
                SessionCommands.EMPTY,
                Player.Commands.ALL_COMMANDS
            )
        }

        override fun onCustomCommand(
            session: MediaSession,
            controller: MediaSession.ControllerInfo,
            customCommand: SessionCommand,
            args: Bundle
        ): ListenableFuture<SessionResult> {
            return when (customCommand.customAction) {
                "TOGGLE_SHUFFLE" -> {
                    player.shuffleModeEnabled = !player.shuffleModeEnabled
                    Futures.immediateFuture(SessionResult(SessionResult.RESULT_SUCCESS))
                }
                else -> super.onCustomCommand(session, controller, customCommand, args)
            }
        }
    }
}
L6 Insight — MediaSessionService vs Service
Extend MediaSessionService (not plain Service) for audio/video playback. The system uses this subclass to:
  • Display a media notification automatically (with play/pause/skip buttons).
  • Allow the system to rebind to your service when a Bluetooth device connects, even if the app is not in the foreground.
  • Participate in the "Now Playing" API for Google Assistant and Android Auto.
Using a plain Service means none of this works without substantial custom plumbing.

4. Connecting a MediaController from the UI

// In your Activity or Fragment — connect to the playback service
class PlayerActivity : AppCompatActivity() {

    private var mediaController: MediaController? = null

    override fun onCreate(savedInstanceState: Bundle?) {
        super.onCreate(savedInstanceState)

        val sessionToken = SessionToken(
            this,
            ComponentName(this, PlaybackService::class.java)
        )

        // Async — connects over binder IPC
        val controllerFuture = MediaController.Builder(this, sessionToken)
            .buildAsync()

        controllerFuture.addListener({
            mediaController = controllerFuture.get()
            // Attach to PlayerView — same Player interface!
            binding.playerView.player = mediaController
        }, MoreExecutors.directExecutor())
    }

    override fun onDestroy() {
        MediaController.releaseFuture(controllerFuture)
        super.onDestroy()
    }
}
Key Insight
MediaController implements the Player interface — the same interface as ExoPlayer. So you can pass a MediaController directly to PlayerView.player. Your UI code doesn't care whether it's talking to a local ExoPlayer or a remote session — same API.

5. Media Notification — Automatic in Media3

When you use MediaSessionService, Media3 automatically creates and manages a media notification. The notification includes:

  • Album art (from MediaItem.mediaMetadata.artworkUri)
  • Track title and artist
  • Play/Pause button
  • Previous/Next skip buttons
  • Seek bar (Android 13+)
// Set rich metadata on MediaItem — feeds the notification
val mediaItem = MediaItem.Builder()
    .setUri(streamUrl)
    .setMediaMetadata(
        MediaMetadata.Builder()
            .setTitle("Scam 1992 — Episode 1")
            .setArtist("Sony LIV Original")
            .setAlbumTitle("Scam 1992")
            .setArtworkUri(Uri.parse("https://cdn.sonyliv.com/artwork/scam1992_ep1.jpg"))
            .setGenre("Drama")
            .setTotalTrackCount(10)
            .setTrackNumber(1)
            .build()
    )
    .build()

Customising the Notification

// Override DefaultMediaNotificationProvider for custom actions
class CustomNotificationProvider(context: Context) : DefaultMediaNotificationProvider(context) {

    override fun getMediaButtons(
        session: MediaSession,
        playerCommands: Player.Commands,
        customLayout: ImmutableList<CommandButton>,
        showPauseButton: Boolean
    ): ImmutableList<CommandButton> {
        // Add a "10-second rewind" button before the default controls
        val rewindButton = CommandButton.Builder()
            .setDisplayName("Rewind 10s")
            .setIconResId(R.drawable.ic_rewind_10)
            .setSessionCommand(SessionCommand("REWIND_10", Bundle.EMPTY))
            .build()

        return ImmutableList.builder<CommandButton>()
            .add(rewindButton)
            .addAll(super.getMediaButtons(session, playerCommands, customLayout, showPauseButton))
            .build()
    }
}

// Register in your service
override fun onCreate() {
    super.onCreate()
    setMediaNotificationProvider(CustomNotificationProvider(this))
    // ... rest of setup
}

6. Bluetooth & Headset Controls

When a user presses Play/Pause on Bluetooth headphones or the hardware media key on older phones, the system fires a KeyEvent to the active MediaSession. Media3 handles this automatically when you use MediaSessionService.

The routing priority for media button events:

  1. The session that most recently had active playback.
  2. If multiple sessions: the one registered with setMediaButtonReceiver().
  3. Fallback: system broadcasts ACTION_MEDIA_BUTTON intent.
// Declare MediaButtonReceiver in AndroidManifest.xml for pre-Lollipop + lock screen
<receiver android:name="androidx.media.session.MediaButtonReceiver"
          android:exported="true">
    <intent-filter>
        <action android:name="android.intent.action.MEDIA_BUTTON"/>
    </intent-filter>
</receiver>

// Declare your service with the media session intent filter
<service android:name=".PlaybackService"
         android:foregroundServiceType="mediaPlayback"
         android:exported="true">
    <intent-filter>
        <action android:name="androidx.media3.session.MediaSessionService"/>
    </intent-filter>
</service>

7. Audio Focus — The Most Missed Topic

Audio focus is the system-level mechanism that prevents two apps from playing audio simultaneously. ExoPlayer does NOT manage audio focus by itself — you must configure it.

// Configure audio focus in ExoPlayer
val audioAttributes = AudioAttributes.Builder()
    .setUsage(C.USAGE_MEDIA)                  // stream type: media (not alarm, not notification)
    .setContentType(C.AUDIO_CONTENT_TYPE_MOVIE) // movie (vs music, speech)
    .build()

val player = ExoPlayer.Builder(context)
    .setAudioAttributes(audioAttributes, /* handleAudioFocus= */ true)
    //                                    ↑ ExoPlayer manages focus automatically
    .build()

When handleAudioFocus = true, ExoPlayer:

  • Requests AUDIOFOCUS_GAIN on play().
  • Pauses on AUDIOFOCUS_LOSS (another app takes focus — e.g., phone call).
  • Ducks (lowers volume) on AUDIOFOCUS_LOSS_TRANSIENT_CAN_DUCK (e.g., navigation prompt).
  • Resumes on AUDIOFOCUS_GAIN regained.
Production Bug
Setting handleAudioFocus = false and forgetting to manage focus manually means your app keeps playing audio during a phone call or over another user's music. This is a guaranteed user complaint. Always set handleAudioFocus = true for media playback.

8. Android Auto Integration

Android Auto uses MediaBrowser to browse your content library and MediaController to control playback. Your MediaSessionService must also implement MediaLibraryService to support browsing:

// For Android Auto / CarPlay: extend MediaLibraryService instead of MediaSessionService
class PlaybackService : MediaLibraryService() {

    private lateinit var mediaLibrarySession: MediaLibrarySession

    override fun onCreate() {
        super.onCreate()
        val player = ExoPlayer.Builder(this).build()

        mediaLibrarySession = MediaLibrarySession.Builder(this, player, LibraryCallback())
            .build()
    }

    override fun onGetSession(controllerInfo: MediaSession.ControllerInfo) =
        mediaLibrarySession

    inner class LibraryCallback : MediaLibrarySession.Callback {

        // Android Auto calls this to get root browse node
        override fun onGetLibraryRoot(
            session: MediaLibrarySession,
            browser: MediaSession.ControllerInfo,
            params: LibraryParams?
        ): ListenableFuture<LibraryResult<MediaItem>> {
            return Futures.immediateFuture(
                LibraryResult.ofItem(
                    MediaItem.Builder()
                        .setMediaId("root")
                        .setMediaMetadata(
                            MediaMetadata.Builder()
                                .setIsBrowsable(true)
                                .setIsPlayable(false)
                                .build()
                        )
                        .build(),
                    params
                )
            )
        }

        // Android Auto calls this to populate a menu node
        override fun onGetChildren(
            session: MediaLibrarySession,
            browser: MediaSession.ControllerInfo,
            parentId: String,
            page: Int,
            pageSize: Int,
            params: LibraryParams?
        ): ListenableFuture<LibraryResult<ImmutableList<MediaItem>>> {
            val items = when (parentId) {
                "root" -> listOf(
                    buildBrowsableItem("continue_watching", "Continue Watching"),
                    buildBrowsableItem("my_list", "My List"),
                    buildBrowsableItem("trending", "Trending Now")
                )
                "continue_watching" -> fetchContinueWatchingItems()
                else -> emptyList()
            }
            return Futures.immediateFuture(LibraryResult.ofItemList(items, params))
        }
    }
}

9. Now Playing — Google Assistant & System

Android 10+ has a "Now Playing" system feature. When your MediaSession reports active playback, the system can show it in Quick Settings and allow Google Assistant commands like "pause", "skip", "what's playing?".

For this to work:

  • Your session must be active (call mediaSession.isActive = true when playing — Media3 does this automatically).
  • MediaMetadata must have title, artist, and duration set.
  • Your package must be whitelisted or the user must have interacted with your app recently (Android's "media resume" feature).

10. Wear OS Handoff

// On Wear OS side — connect to phone's MediaSession
// (uses ChannelClient / MessageClient under the hood in Wearable Data Layer)
val sessionToken = SessionToken(
    context,
    ComponentName("com.yourapp", "com.yourapp.PlaybackService")
)

// Same MediaController API works across devices via Wearable Data Layer
val controllerFuture = MediaController.Builder(context, sessionToken).buildAsync()
controllerFuture.addListener({
    val controller = controllerFuture.get()
    controller.play()   // controls playback on the phone from the watch
}, MoreExecutors.directExecutor())

11. The Full Lifecycle — Background Audio Pattern

User presses "Play" in UI │ Activity → MediaController.play() │ IPC Service (MediaSessionService) │ ExoPlayer.play() │ Audio Focus requested (AUDIOFOCUS_GAIN) │ Foreground notification shown (Media3 auto) │ User swipes app away (Activity destroyed) │ Service continues in foreground │ (notification keeps it alive) Lock screen shows playback controls │ (via MediaSession state) │ User presses pause on Bluetooth headset │ KeyEvent → MediaSession → ExoPlayer.pause() │ Audio focus released (AUDIOFOCUS_ABANDON) │ Foreground notification updated to "paused" state │ User returns to app │ Activity rebinds via MediaController → resumes control

Interview Cheat Sheet — MediaSession & MediaController

  • What does MediaSession do? — Publishes playback state + metadata so external clients (lock screen, BT, Auto, Assistant) can read and control playback via a standard IPC contract.
  • Media3 vs old way? — Old: MediaSessionCompat + MediaSessionConnector to bridge ExoPlayer. Media3: MediaSession.Builder(context, player).build() — session backed directly by ExoPlayer, no connector.
  • MediaSessionService vs Service? — Use MediaSessionService. It automatically manages the media notification, allows system rebind for BT/Auto, and participates in Now Playing.
  • MediaController implements Player? — Yes. Pass it directly to PlayerView.player. UI code is identical whether controlling local ExoPlayer or remote session.
  • Audio focus? — Set handleAudioFocus = true in ExoPlayer.Builder. Pauses on phone call, ducks on navigation prompt, resumes when focus is regained. Never skip this.
  • Android Auto? — Extend MediaLibraryService. Implement onGetLibraryRoot() and onGetChildren() to provide a browsable content tree. Auto uses MediaBrowser to navigate it.
  • Bluetooth media buttons? — Handled automatically by MediaSessionService. Declare MediaButtonReceiver in manifest for pre-L / lock screen fallback.
  • Background survival? — The foreground notification keeps MediaSessionService alive. When user swipes the notification away, service stops. When user dismisses via notification "✕", call player.stop() + stopSelf().
