Adaptive Bitrate Streaming (ABR)

How quality selection works, EWMA bandwidth estimation, buffer-based algorithms, and how to customize ExoPlayer's ABR logic

1. What ABR Actually Does

ABR selects the bitrate (quality level) for each upcoming segment. The goal is a single sentence:

Maximize video quality while preventing rebuffering.

These two goals are in tension. Higher quality = more data per second = more risk of emptying the buffer if bandwidth drops. ABR is the algorithm that navigates this tradeoff in real time.

2. The Two ABR Strategies

Strategy A — Bandwidth-Based ABR

Estimate current network throughput, pick the highest bitrate that can be downloaded faster than it is played.

quality = highest bitrate where: bitrate < (bandwidth × safety_factor)

Safety factor is typically 0.75–0.85 to leave headroom for measurement noise.

Pros: Reacts quickly to bandwidth changes. Simple to understand.
Cons: Bandwidth estimates are noisy. Can cause oscillation (flickering between quality levels).

Strategy B — Buffer-Based ABR (BBA)

Ignore bandwidth. Choose quality purely based on current buffer level. More buffer → higher quality. Developed by researchers at Stanford.

Pros: Very stable. No measurement noise. Works even when bandwidth estimation is wrong.
Cons: Slow to react — must wait for buffer changes to propagate.

Strategy C — Hybrid (what ExoPlayer uses)

ExoPlayer's AdaptiveTrackSelection combines both: use bandwidth estimate as the primary signal, but apply buffer-level-based damping to prevent excessive quality switches.

3. Bandwidth Estimation — EWMA

ExoPlayer uses Exponentially Weighted Moving Average (EWMA) to estimate bandwidth. Two EWMA values are tracked simultaneously:

  • Fast EWMA (half-life ≈ 3s) — reacts quickly to drops.
  • Slow EWMA (half-life ≈ 8s) — stable, less noise.

The reported bandwidth is the minimum of the two — conservative by design:

estimated_bandwidth = min(ewma_fast, ewma_slow)

This prevents over-estimating bandwidth during a brief good-network burst.

// ExoPlayer internals — BandwidthMeter
// Each time a chunk finishes downloading:
val downloadedBytes: Long = chunk.bytesLoaded()
val elapsedMs: Long = chunk.loadDurationMs
val throughputBps = downloadedBytes * 8 * 1000 / elapsedMs  // bits per second

// Update both EWMAs
fastEwma = fastEwma * (1 - fastAlpha) + throughputBps * fastAlpha
slowEwma = slowEwma * (1 - slowAlpha) + throughputBps * slowAlpha

// Conservative estimate
val estimatedBandwidth = min(fastEwma, slowEwma)
L6 Insight — Why Min of Two EWMAs?
If bandwidth suddenly drops from 10Mbps to 1Mbps, the fast EWMA drops immediately. The slow EWMA still reads ~5Mbps. Taking the minimum gives 1Mbps — the player immediately picks a lower quality. This prevents the player from downloading a 10Mbps segment on a 1Mbps connection, which would stall.

The reverse is also protected: if bandwidth briefly spikes (burst from CDN), min() ignores the spike and stays conservative until both EWMAs converge up.

4. AdaptiveTrackSelection — ExoPlayer's ABR Logic

AdaptiveTrackSelection is the default implementation of ExoTrackSelection for adaptive streams. It runs the ABR decision on every chunk download completion:

// Simplified logic from AdaptiveTrackSelection
fun updateSelectedTrack(
    playbackPositionUs: Long,
    bufferedDurationUs: Long,
    availableDurationUs: Long
) {
    val effectiveBitrate = bandwidthMeter.bitrateEstimate * bandwidthFraction  // e.g. 0.75

    // Find highest format whose bitrate is below effective bandwidth
    val idealFormat = formats
        .filter { it.bitrate <= effectiveBitrate }
        .maxByOrNull { it.bitrate }
        ?: formats.last()  // fallback to lowest if all exceed bandwidth

    // Buffer-based damping: if buffer is low, don't upgrade quality
    val bufferedSeconds = bufferedDurationUs / 1_000_000f
    if (bufferedSeconds < minDurationForQualityIncreaseMs / 1000f) {
        // Not enough buffer to risk an upgrade — stay current or go lower
        selectedFormat = minOf(selectedFormat, idealFormat, compareBy { it.bitrate })
    } else {
        selectedFormat = idealFormat
    }
}

5. ABR Quality Switch — What Happens Mechanically

Segment N: playing 720p │ ▼ ABR decides: upgrade to 1080p (buffer healthy, bandwidth high) │ ├─ Next chunk URL = 1080p/seg_N+1.ts │ ├─ Are codecs the same? (check init.mp4 headers) │ ├─ YES → seamless switch. No decoder flush needed. │ └─ NO → must flush MediaCodec, re-configure → ~200ms visual stutter │ ├─ New segment downloaded faster than old? → buffer grows │ └─ Renderer submits 1080p frames → PlayerView shows higher resolution
Seamless vs Discontinuous Switches
A codec-compatible switch (same codec, same profile) is seamless — MediaCodec is not re-initialized. A switch between H.264 and HEVC, or between different profiles, requires a decoder flush. Always encode all renditions with the same codec family to enable seamless ABR switching.

6. Custom ABR — Implementing Your Own TrackSelection

For a Staff-level question: "How would you implement data-saver mode — capping quality at 480p when the user is on mobile data?"

class DataSaverTrackSelectionFactory(
    private val isOnMobileData: () -> Boolean
) : ExoTrackSelection.Factory {

    override fun createTrackSelections(
        definitions: Array,
        bandwidthMeter: BandwidthMeter,
        mediaPeriodId: MediaSource.MediaPeriodId,
        timeline: Timeline
    ): Array {
        return definitions.map { definition ->
            if (definition == null) return@map null

            val formats = (0 until definition.group.length)
                .map { definition.group.getFormat(it) }

            val filtered = if (isOnMobileData()) {
                // Data-saver: cap at 480p (height <= 480)
                formats.filter { it.height <= 480 }.ifEmpty { listOf(formats.last()) }
            } else {
                formats  // Full quality on WiFi
            }

            // Use adaptive selection within the filtered set
            AdaptiveTrackSelection(
                definition.group,
                filtered.map { formats.indexOf(it) }.toIntArray(),
                bandwidthMeter
            )
        }.toTypedArray()
    }
}

// Wire it up
val player = ExoPlayer.Builder(context)
    .setTrackSelector(DefaultTrackSelector(context, DataSaverTrackSelectionFactory {
        connectivityManager.activeNetwork?.let {
            connectivityManager.getNetworkCapabilities(it)
                ?.hasTransport(NetworkCapabilities.TRANSPORT_CELLULAR)
        } == true
    }))
    .build()

L6 Insight — Rebuffer vs Quality Tradeoff
Netflix research showed users tolerate moderate quality (720p) far better than rebuffering. A 1-second rebuffer stall reduces session watch time by ~4%. Prioritize avoiding stalls over maximizing bitrate. This is why ExoPlayer's default bandwidthFraction is 0.75 — it deliberately under-estimates bandwidth to keep a safety margin.

8. ABR and Android Network APIs

// React to network type changes and adjust ABR aggressiveness
val connectivityManager = getSystemService(ConnectivityManager::class.java)

connectivityManager.registerDefaultNetworkCallback(object : ConnectivityManager.NetworkCallback() {
    override fun onCapabilitiesChanged(network: Network, caps: NetworkCapabilities) {
        val isWifi = caps.hasTransport(NetworkCapabilities.TRANSPORT_WIFI)
        val isMetered = !caps.hasCapability(NetworkCapabilities.NET_CAPABILITY_NOT_METERED)

        // On metered connection: be conservative
        trackSelector.parameters = trackSelector.buildUponParameters()
            .setMaxVideoBitrate(if (isMetered) 2_000_000 else Int.MAX_VALUE)
            .build()
    }

    override fun onLost(network: Network) {
        // Network dropped — pause or show "connecting" UI
        player.pause()
    }
})

9. ABR Challenges at Scale — What Hotstar Deals With

  • CDN Throughput Collapse — During IPL final, millions of concurrent viewers. CDN edge nodes saturate. Bandwidth estimates are useless because the bottleneck is CDN, not last-mile. Solution: cap max bitrate dynamically via server-sent bitrate hints in the manifest.
  • Oscillation — Rapid ping-pong between quality levels (looks like flickering). Solution: add hysteresis — require bandwidth to be 20% above a level before upgrading to it.
  • Startup Optimization — Don't start at lowest quality just to be safe. Use a "startup phase" with aggressive bandwidth probing. ExoPlayer does this via the initial bitrate estimate in DefaultBandwidthMeter.
  • Device Capability — Low-end devices cannot decode 1080p at 30fps in real time even if bandwidth is available. ExoPlayer checks MediaCodecInfo.isHardwareAccelerated() and limits quality accordingly.

Interview Cheat Sheet — ABR

  • What is ABR? — Dynamically selecting video quality for each segment based on network conditions, to maximize quality while avoiding stalls.
  • Two main strategies? — Bandwidth-based (pick quality below estimated throughput) and buffer-based (pick quality based on buffer health). ExoPlayer uses a hybrid.
  • EWMA? — Exponentially Weighted Moving Average. ExoPlayer tracks fast (3s) and slow (8s) EWMAs. Takes the minimum — conservative estimate to avoid over-shooting.
  • Why bandwidthFraction = 0.75? — Uses only 75% of estimated bandwidth to leave headroom for measurement error and CDN jitter.
  • Seamless quality switch? — Only possible if codecs are compatible (same codec, profile). Otherwise MediaCodec must be flushed → ~200ms stall. Keep all renditions in the same codec family.
  • How to implement data-saver? — Custom ExoTrackSelection.Factory that filters formats to height ≤ 480 when on mobile data. Wire to NetworkCapabilities.TRANSPORT_CELLULAR.
  • Key metric? — Rebuffer rate. A 1s stall reduces watch time by ~4% (Netflix research). Prioritize stall avoidance over bitrate maximization.
  • ABR at IPL scale? — CDN saturation makes bandwidth estimates unreliable. Use server-side bitrate caps sent in manifest. Add per-device capability checks.
