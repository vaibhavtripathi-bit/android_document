HLS & DASH Protocols

Manifest structure, segment formats, codec negotiation, and how ExoPlayer walks the protocol stack

1. The Big Picture — Why Adaptive Protocols?

A 2-hour 1080p movie is ~8 GB. You cannot download it all before playback, and you cannot predict network speed ahead of time. Adaptive streaming solves both problems:

  1. Segmentation — content is pre-encoded into short chunks (2–10 seconds each).
  2. Multiple bitrates — each segment exists at multiple quality levels (e.g., 400kbps, 1.5Mbps, 4Mbps, 8Mbps).
  3. Manifest — a small text file (downloaded first) lists all available quality levels and the URL of every segment.
  4. ABR algorithm — the player picks the quality for each segment based on current bandwidth.
CDN Server ├── master.m3u8 ← Master playlist (HLS) or manifest.mpd (DASH) ├── 360p/ │ ├── seg000.ts │ ├── seg001.ts ... ├── 720p/ │ ├── seg000.ts │ ├── seg001.ts ... └── 1080p/ ├── seg000.ts ├── seg001.ts ...

2. HLS — HTTP Live Streaming

Developed by Apple (2009). The dominant format on iOS and widely used on Android. Uses .m3u8 playlist files and .ts (MPEG-2 Transport Stream) or .fmp4 (fragmented MP4) segments.

Master Playlist (M3U8)

#EXTM3U
#EXT-X-VERSION:6

# Video renditions (different bitrates)
#EXT-X-STREAM-INF:BANDWIDTH=800000,RESOLUTION=640x360,CODECS="avc1.42c01e,mp4a.40.2"
360p/index.m3u8

#EXT-X-STREAM-INF:BANDWIDTH=2800000,RESOLUTION=1280x720,CODECS="avc1.64001f,mp4a.40.2"
720p/index.m3u8

#EXT-X-STREAM-INF:BANDWIDTH=5000000,RESOLUTION=1920x1080,CODECS="avc1.640028,mp4a.40.2"
1080p/index.m3u8

# Alternate audio track (Hindi)
#EXT-X-MEDIA:TYPE=AUDIO,GROUP-ID="audio",LANGUAGE="hi",NAME="Hindi",URI="audio_hi/index.m3u8"

# Subtitles
#EXT-X-MEDIA:TYPE=SUBTITLES,GROUP-ID="subs",LANGUAGE="en",NAME="English",URI="subs_en/index.m3u8"

Media Playlist (per rendition)

#EXTM3U
#EXT-X-VERSION:6
#EXT-X-TARGETDURATION:6       ← max segment duration
#EXT-X-MEDIA-SEQUENCE:0

#EXTINF:6.000,
seg000.ts
#EXTINF:6.000,
seg001.ts
#EXTINF:5.987,
seg002.ts
#EXT-X-ENDLIST              ← only present for VOD; absent for live
L6 Insight — #EXT-X-ENDLIST
The presence/absence of #EXT-X-ENDLIST is how ExoPlayer's HlsPlaylistTracker determines VOD vs Live. For live, it refreshes the media playlist periodically (every ~targetDuration/2) to discover new segments. This is a key difference from DASH, which has the equivalent in @type="dynamic".

Key HLS Tags You Must Know

TagMeaning
#EXT-X-KEYEncryption key URI — HLS-native AES-128 or signalling for Widevine
#EXT-X-DISCONTINUITYTimeline break (ad insertion, encoding change) — renderer must flush
#EXT-X-BYTERANGESegment is a byte range within a larger file — reduces CDN file count
#EXT-X-MAPInitialization segment (required for fMP4 segments)
#EXT-X-PROGRAM-DATE-TIMEWall-clock timestamp for each segment — critical for live DVR
#EXT-X-SERVER-CONTROLLow-latency HLS: push delivery, part target duration
#EXT-X-PARTSub-segments (low-latency HLS) — 200ms parts within a 6s segment

3. DASH — Dynamic Adaptive Streaming over HTTP

Developed by MPEG (2011), standardized as ISO/IEC 23009. The preferred format for Android/Chrome and streaming giants like Netflix and YouTube. Uses XML manifest called MPD (Media Presentation Description).

MPD Structure

<MPD type="static" mediaPresentationDuration="PT2H3M14S"
     minBufferTime="PT2S" profiles="urn:mpeg:dash:profile:isoff-on-demand:2011">

  <Period>
    <!-- Video AdaptationSet -->
    <AdaptationSet mimeType="video/mp4" codecs="avc1.64001f" frameRate="24">
      <Representation id="v360" bandwidth="800000" width="640" height="360">
        <SegmentTemplate media="360p/seg$Number$.m4s" initialization="360p/init.mp4"
                         startNumber="1" duration="240000" timescale="24000"/>
      </Representation>
      <Representation id="v720" bandwidth="2800000" width="1280" height="720">
        <SegmentTemplate media="720p/seg$Number$.m4s" initialization="720p/init.mp4"
                         startNumber="1" duration="240000" timescale="24000"/>
      </Representation>
      <Representation id="v1080" bandwidth="5000000" width="1920" height="1080">
        <SegmentTemplate media="1080p/seg$Number$.m4s" initialization="1080p/init.mp4"
                         startNumber="1" duration="240000" timescale="24000"/>
      </Representation>
    </AdaptationSet>

    <!-- Audio AdaptationSet -->
    <AdaptationSet mimeType="audio/mp4" codecs="mp4a.40.2" lang="hi">
      <Representation id="a128" bandwidth="128000">
        <SegmentTemplate media="audio_hi/seg$Number$.m4s" initialization="audio_hi/init.mp4"
                         startNumber="1" duration="240000" timescale="24000"/>
      </Representation>
    </AdaptationSet>

    <!-- DRM -->
    <ContentProtection schemeIdUri="urn:uuid:EDEF8BA9-79D6-4ACE-A3C8-27DCD51D21ED">
      <cenc:pssh>AAAA...base64...</cenc:pssh>
    </ContentProtection>
  </Period>
</MPD>
L6 Insight — DASH Terminology
Period = a contiguous time range (e.g., content + mid-roll ad is 2 Periods).
AdaptationSet = a group of Representations that are interchangeable (same content, different bitrate).
Representation = one specific bitrate/resolution encoding.
This maps directly to ExoPlayer's TrackGroup (= AdaptationSet) and Format (= Representation).

4. HLS vs DASH — Side-by-Side

HLS

  • Manifest: M3U8 (plain text)
  • Segments: .ts or .fMP4
  • Origin: Apple (2009)
  • Native iOS support
  • Simpler, widely deployed
  • Low-latency variant: LL-HLS
  • DRM: Widevine via SAMPLE-AES

DASH

  • Manifest: MPD (XML)
  • Segments: .m4s (fMP4 only)
  • Origin: MPEG standard (2011)
  • Preferred on Android/Chrome
  • More expressive (Periods, Roles)
  • Low-latency variant: LL-DASH
  • DRM: CENC (Common Encryption)

5. Segment Formats — TS vs fMP4

MPEG-2 Transport Stream (.ts)

The legacy HLS segment format. Fixed 188-byte packets, multiplexes video and audio into a single stream. Issues:

  • Cannot carry HEVC (H.265) in older encoders.
  • No random access — must parse from the start of each segment.
  • Larger overhead than fMP4.

Fragmented MP4 (.m4s + init.mp4)

The modern format for both HLS (v6+) and DASH. Structure:

init.mp4 (initialization segment) ├── ftyp box ← file type ├── moov box ← track metadata, codec parameters, sample tables │ └── trak, tkhd, mdia, minf, stbl... seg001.m4s (media segment) ├── styp box ├── sidx box ← segment index (byte offsets for random access) └── moof + mdat boxes (one per fragment) ├── moof ← Movie Fragment header (decode timestamps, sample flags) └── mdat ← raw encoded media data (H.264/HEVC/AAC bytes)
Why fMP4 Matters
ExoPlayer must download init.mp4 before it can decode any segment. If you switch quality levels (ABR switch), it checks if the new rendition's init segment is compatible. If codecs differ, it must flush the decoder and re-initialize MediaCodec — an expensive operation (~200ms stall).

6. Codec Negotiation

The manifest declares codecs using RFC 6381 strings. ExoPlayer checks device capability via MediaCodecList before selecting a track:

L6 Insight — AV1 on Android
Netflix and YouTube are migrating to AV1 because it delivers ~30% better compression than HEVC at the same quality. On Android, AV1 hardware decode is available on Pixel 3+ and many Snapdragon 888+ chips. ExoPlayer falls back to software LibGav1 decoder if no hardware AV1 decoder exists — check via MediaCodecUtil.getDecoderInfos("video/av01", false, false).

8. Low-Latency Protocols

Standard HLS/DASH has 15–30 seconds of latency (buffer = N segments × segment_duration). Low-latency variants reduce this to 1–3 seconds for live events.

Low-Latency HLS (LL-HLS)

  • Segments split into parts (e.g., 200ms each, published as #EXT-X-PART tags).
  • Client uses Blocking Playlist Reload — server holds the playlist response until new part is available (HTTP long-poll via _HLS_msn query param).
  • Requires CDN support for CAN-BLOCK-RELOAD=YES.

Low-Latency DASH (LL-DASH)

  • Segments published as they are encoded using chunked transfer encoding.
  • Client reads each chunk as it arrives without waiting for segment to complete.
  • ExoPlayer supports LL-DASH via DashMediaSource with setFallbackTargetLiveOffsetMs().
Interview Gotcha
LL-HLS and LL-DASH require specific CDN and origin server support. Simply using a small segment duration does NOT give you low latency — it increases origin load. Real low latency needs server push or chunked transfer.

Interview Cheat Sheet — HLS & DASH

  • HLS manifest file? — .m3u8 (M3U8). Master playlist + per-rendition media playlists.
  • DASH manifest file? — .mpd (XML). Single file with Period → AdaptationSet → Representation hierarchy.
  • Segment formats? — HLS: .ts (legacy) or .m4s (modern fMP4). DASH: .m4s only. fMP4 needs init.mp4 downloaded first.
  • How does ExoPlayer know it's live? — HLS: no #EXT-X-ENDLIST. DASH: @type="dynamic" in MPD.
  • What is a discontinuity in HLS? — Timeline break (e.g., ad insertion). ExoPlayer must flush decoders at a discontinuity tag.
  • DASH Period? — A contiguous time span. Multiple Periods = ad break workflow. ExoPlayer maps each Period to a MediaPeriod.
  • Low-latency mechanism? — LL-HLS uses sub-second parts + blocking playlist reload. LL-DASH uses chunked transfer encoding. Both target <3s latency.
  • Codec in manifest? — RFC 6381 strings: avc1.640028 = H.264, hvc1 = HEVC, av01 = AV1, mp4a.40.2 = AAC. ExoPlayer checks against MediaCodecList before selecting.
  • Why does ABR switch sometimes stall? — New rendition needs a different init.mp4 (different codec config). MediaCodec flush + reconfigure = ~200ms stall. Avoid by keeping codecs consistent across renditions.
