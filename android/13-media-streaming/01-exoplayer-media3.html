ExoPlayer & Media3

Architecture internals, component model, Media3 migration, and what L6 engineers must know cold

1. Why ExoPlayer Exists

Android's built-in MediaPlayer has two fundamental limitations that make it unsuitable for streaming apps:

  • No DASH/HLS support — MediaPlayer can play a direct URL but cannot parse adaptive streaming manifests.
  • Black-box implementation — you cannot customize buffering, track selection, or error recovery.

Google (YouTube team) built ExoPlayer in 2014 as an open-source, fully customizable player. In 2022 it was reorganized under the Media3 umbrella library to unify ExoPlayer, MediaSession, and Transformer under one namespace and one release cycle.

Migration Status
ExoPlayer 2.x is in maintenance mode. All new features ship in Media3 only. If an interviewer asks "why did you choose ExoPlayer over MediaPlayer?", they want to hear about customizability — not just format support.

3. MediaItem — The Modern Entry Point

Since ExoPlayer 2.12 (and Media3), MediaItem replaced the raw URI pattern. It carries everything needed to play an item: URI, MIME type, DRM config, subtitles, and metadata.

// Media3 / ExoPlayer 2.12+
val player = ExoPlayer.Builder(context).build()

val mediaItem = MediaItem.Builder()
    .setUri("https://cdn.hotstar.com/content/episode.m3u8")
    .setMimeType(MimeTypes.APPLICATION_M3U8)         // HLS
    .setDrmConfiguration(
        MediaItem.DrmConfiguration.Builder(C.WIDEVINE_UUID)
            .setLicenseUri("https://license.hotstar.com/widevine")
            .setLicenseRequestHeaders(mapOf("Authorization" to "Bearer $token"))
            .build()
    )
    .build()

player.setMediaItem(mediaItem)
player.prepare()
player.playWhenReady = true
L6 Insight
MediaItem is not just a convenience wrapper. It decouples intent (what to play) from mechanism (how to play it). This allows the same MediaItem to drive ExoPlayer, a Cast session, or a MediaController — the Player interface abstraction handles the routing.

4. Threading Model

ExoPlayer is single-threaded by design. All calls must happen on the application main thread (or any single thread you designate at construction time). Internally:

  • A dedicated playback thread handles loading, decoding, rendering.
  • Listeners (Player.Listener) are called back on the main thread.
  • You never need to synchronize — ExoPlayer marshals everything internally.
Common Mistake
Calling player.play() from a background thread will throw IllegalStateException in debug builds and produce undefined behavior in release. Always use mainHandler.post { player.play() } if you're on a background thread.

5. MediaSource Pipeline in Detail

HlsMediaSource │ ├── HlsPlaylistTracker ← fetches & refreshes master playlist (M3U8) │ └── HlsPlaylistParser │ ├── HlsChunkSource ← selects next segment to load based on ABR │ └── DefaultHlsTrackNameProvider │ └── HlsSampleStreamWrapper ← wraps each track (video/audio/text) └── DefaultHlsExtractorFactory → TsExtractor / FragmentedMp4Extractor

Each MediaSource exposes a MediaPeriod which ExoPlayer uses to load samples into a SampleQueue. The renderer reads from the SampleQueue and feeds MediaCodec.

6. TrackSelector — How Quality is Picked

DefaultTrackSelector uses a constraint-based system. You set parameters; it finds the best matching track:

val trackSelector = DefaultTrackSelector(context).apply {
    parameters = buildUponParameters()
        .setMaxVideoSizeSd()                        // cap at 480p on SD screens
        .setPreferredAudioLanguage("hi")             // prefer Hindi audio
        .setPreferredTextLanguage("en")              // prefer English subtitles
        .setForceHighestSupportedBitrate(false)      // let ABR decide
        .build()
}

val player = ExoPlayer.Builder(context)
    .setTrackSelector(trackSelector)
    .build()
L6 Insight — Custom TrackSelector
For a staff-level question: "How would you implement a quality lock feature (e.g., always play 1080p)?" Answer: Subclass MappingTrackSelector, override selectTracks(), and return a fixed rendererTrackSelection ignoring bandwidth estimates. Do not subclass DefaultTrackSelector — its internal state is fragile.

7. LoadControl — Buffer Policy

DefaultLoadControl controls two things: when to start buffering (resume loading) and when to stop (enough buffered). Key parameters:

val loadControl = DefaultLoadControl.Builder()
    .setBufferDurationsMs(
        /* minBufferMs  = */ 15_000,   // start playback after 15s buffered
        /* maxBufferMs  = */ 50_000,   // don't buffer beyond 50s
        /* bufferForPlaybackMs = */ 2_500,  // rebuffer threshold after stall
        /* bufferForPlaybackAfterRebufferMs = */ 5_000
    )
    .setPrioritizeTimeOverSizeThresholds(true)
    .build()
ParameterEffectTune for
minBufferMsWon't start playback until this much is bufferedLower = faster start, more rebuffering
maxBufferMsStops loading ahead of this windowHigher = smoother but more memory/data usage
bufferForPlaybackMsResumes after a stall once this much is re-bufferedLower = less stall time but risk of immediate re-stall

8. Renderers and MediaCodec

MediaCodecVideoRenderer is the default video renderer. Internally it:

  1. Calls MediaCodec.configure() with the decoder name, Surface, and MediaCrypto (for DRM).
  2. Queues encoded input buffers via queueInputBuffer().
  3. Dequeues decoded output buffers and renders to SurfaceView or TextureView.
  4. Synchronizes playback position using AudioTrack timestamps (audio is the master clock).
SurfaceView vs TextureView
Prefer SurfaceView for video — it renders in a dedicated hardware layer, saving GPU compositing. Use TextureView only when you need transformations (rotation, arbitrary clip). PlayerView uses SurfaceView by default.

9. Error Handling and Retry

player.addListener(object : Player.Listener {
    override fun onPlayerError(error: PlaybackException) {
        when (error.errorCode) {
            PlaybackException.ERROR_CODE_IO_NETWORK_CONNECTION_FAILED -> {
                // network drop — retry after delay
                retryWithBackoff()
            }
            PlaybackException.ERROR_CODE_DECODING_FAILED -> {
                // codec issue — try software fallback
                switchToSoftwareDecoder()
            }
            PlaybackException.ERROR_CODE_DRM_LICENSE_EXPIRED -> {
                // DRM license gone — re-fetch and re-prepare
                refreshDrmLicense()
            }
        }
    }

    override fun onPlaybackStateChanged(playbackState: Int) {
        when (playbackState) {
            Player.STATE_BUFFERING -> showSpinner()
            Player.STATE_READY     -> hideSpinner()
            Player.STATE_ENDED     -> showNextEpisode()
            Player.STATE_IDLE      -> { /* not started or after error */ }
        }
    }
})

10. Player Lifecycle — Correct Resource Management

// In a Fragment / Activity
private var player: ExoPlayer? = null

override fun onStart() {
    super.onStart()
    // API 24+: multi-window, init here
    initializePlayer()
}

override fun onStop() {
    super.onStop()
    // API 24+: release here (app goes to background)
    releasePlayer()
}

private fun initializePlayer() {
    player = ExoPlayer.Builder(requireContext()).build().also { exoPlayer ->
        binding.playerView.player = exoPlayer
        exoPlayer.setMediaItem(mediaItem)
        exoPlayer.prepare()
        exoPlayer.playWhenReady = true
    }
}

private fun releasePlayer() {
    player?.run {
        playbackPosition = currentPosition
        currentMediaItemIndex = currentMediaItemIndex
        playerPlayWhenReady = playWhenReady
        release()  // releases MediaCodec, AudioTrack, all resources
    }
    player = null
}
Critical Bug
Not calling player.release() leaks a MediaCodec decoder. Android has a limited pool of hardware decoders (often 2–3). Leaked decoders cause MediaCodec.createDecoderByType() to throw on the next video open.

Interview Cheat Sheet — ExoPlayer / Media3

  • Why ExoPlayer over MediaPlayer? — Full customizability: TrackSelector, LoadControl, DataSource, Renderer are all pluggable. MediaPlayer is a black box. Also supports DASH, HLS, SmoothStreaming natively.
  • What is Media3? — Umbrella Jetpack library that unifies ExoPlayer, MediaSession, and Transformer. Same player API, one release train, seamless session integration.
  • Threading rule? — ExoPlayer must be accessed from one thread (usually main). Internally uses a separate playback thread. Never call from background thread.
  • MediaItem vs MediaSource? — MediaItem is a declarative descriptor (URI, DRM config, metadata). MediaSource is the operational object that actually fetches and parses. ExoPlayer converts MediaItem → MediaSource internally via MediaSourceFactory.
  • SurfaceView vs TextureView? — SurfaceView renders in a separate hardware layer (no GPU compositing). TextureView is a regular View, supports transformations but costs GPU compositing. Always prefer SurfaceView for video.
  • How to implement quality lock? — Override selectTracks() in a custom TrackSelector subclassing MappingTrackSelector. Return a FixedTrackSelection for the desired track.
  • Decoder leak? — Always call player.release() in onStop(). Android has a finite pool of hardware decoders.
  • How does ExoPlayer sync audio/video? — Audio is the master clock. Video renderer reads AudioTrack position and drops/renders frames to match.
