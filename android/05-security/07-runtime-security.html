Runtime Security Vulnerabilities & Defenses

IPC security, exported component attacks, PendingIntent mutability, SQL injection, WebView JS injection, tapjacking, clipboard security, logging, root detection

1. IPC Security — Intent Validation

Exported vs Non-Exported Components

Android 12 (API 31) introduced a breaking change: all components with intent-filters must explicitly declare android:exported. Before that, components with intent-filters defaulted to exported="true", exposing them to any app on the device.

// AndroidManifest.xml — VULNERABLE (pre-API 31 default)
<activity
    android:name=".DeepLinkActivity"
    android:exported="true">  <!-- REQUIRED on API 31+ -->
  <intent-filter>
    <action android:name="android.intent.action.VIEW" />
    <category android:name="android.intent.category.DEFAULT" />
    <data android:scheme="myapp" android:host="open" />
  </intent-filter>
</activity>

// Internal-only components — MUST be non-exported
<activity
    android:name=".InternalSettingsActivity"
    android:exported="false" />

Intent Spoofing Attacks

An attacker can craft an explicit intent targeting your exported component with malicious extras. If you don't validate the caller, you may trust attacker-controlled data.

// VULNERABLE: Attacker sends intent with extras
// attacker app sends:
// Intent().setComponent(ComponentName("com.yourapp", "com.yourapp.DeepLinkActivity"))
//     .putExtra("user_id", "123")
//     .putExtra("admin", true)

override fun onCreate(savedInstanceState: Bundle?) {
    super.onCreate(savedInstanceState)
    val userId = intent.getStringExtra("user_id") ?: return  // Attacker-controlled!
    val isAdmin = intent.getBooleanExtra("admin", false)      // Attacker can set true!
    loadUserData(userId, isAdmin)  // Privilege escalation
}

// SECURE: Validate caller and sanitize extras
override fun onCreate(savedInstanceState: Bundle?) {
    super.onCreate(savedInstanceState)
    if (!isCallerTrusted()) return  // Only allow same app or trusted packages
    val userId = intent.getStringExtra("user_id")?.takeIf { it.matches(Regex("^[a-zA-Z0-9_-]+$")) }
        ?: return
    val isAdmin = false  // Never trust extras for privilege decisions
    loadUserData(userId, isAdmin)
}

private fun isCallerTrusted(): Boolean {
    val callingPackage = callingActivity?.packageName ?: return false
    return callingPackage == packageName || trustedPackages.contains(callingPackage)
}

Intent Redirection Vulnerability

If your app receives an intent and forwards it (e.g., via startActivity()) without validating the target, an attacker can use your app as a launch pad to reach internal components of other apps.

// VULNERABLE: Intent redirection
override fun onNewIntent(intent: Intent?) {
    super.onNewIntent(intent)
    val redirectUrl = intent?.getStringExtra("redirect_url")
    if (redirectUrl != null) {
        startActivity(Intent(Intent.ACTION_VIEW, Uri.parse(redirectUrl)))
        // Attacker can pass: intent://com.victim.app/.InternalActivity#Intent;scheme=intent;end
    }
}

// SECURE: Whitelist allowed targets
override fun onNewIntent(intent: Intent?) {
    super.onNewIntent(intent)
    val redirectUrl = intent?.getStringExtra("redirect_url") ?: return
    val uri = Uri.parse(redirectUrl)
    if (uri.scheme != "https" || !allowedHosts.contains(uri.host)) return
    startActivity(Intent(Intent.ACTION_VIEW, uri))
}
Staff-Level Insight

Implicit intents are inherently safer for inter-app communication because the system resolves the target — the user can choose. Explicit intents bypass the resolver and go directly to a component. For exported components, always prefer explicit intents from trusted callers only, or use Intent.setPackage() to limit resolution to your app. At Google scale, intent validation is enforced in security reviews — missing caller checks are a common cause of P0 bugs.

2. PendingIntent Security

Mutable vs Immutable PendingIntents

Android 12 (API 31) requires explicit intent mutability flags. A PendingIntent is a token that can be sent to another process — when the recipient uses it, the system fills in the intent with the creator's identity. The mutability flag determines whether the recipient can modify the wrapped intent.

FlagBehaviorUse Case
FLAG_IMMUTABLERecipient cannot modify the intent extras or actionDefault: notifications, alarms, most use cases
FLAG_MUTABLERecipient can modify extras (e.g., add inline reply text)Notification actions with inline reply, bubbles

PendingIntent Hijacking Attack

If you use FLAG_MUTABLE and a malicious app receives the PendingIntent (e.g., via a notification action), it can override the extras before firing. The system will execute the intent with the creator's identity but with attacker-controlled data.

// VULNERABLE: Malicious app receives PendingIntent, modifies extras, fires it
// Your app created: PendingIntent for "delete_item" with extra item_id=42
// Attacker: pendingIntent.send() with extra item_id=1 (admin account)
// System executes as YOUR app, but with item_id=1

// SECURE: Use FLAG_IMMUTABLE when possible
val pendingIntent = PendingIntent.getActivity(
    context, requestCode, intent,
    PendingIntent.FLAG_IMMUTABLE or PendingIntent.FLAG_UPDATE_CURRENT
)

// When FLAG_MUTABLE is required (e.g., RemoteInput for inline reply):
val replyIntent = Intent(context, ReplyHandler::class.java).apply {
    putExtra("action", "reply")
}
val pendingIntent = PendingIntent.getBroadcast(
    context, requestCode, replyIntent,
    PendingIntent.FLAG_MUTABLE or PendingIntent.FLAG_UPDATE_CURRENT
)
// CRITICAL: Validate ALL extras in ReplyHandler — never trust them for privilege decisions
Staff-Level Insight

FLAG_IMMUTABLE is the default recommendation because it prevents the recipient from changing the intent. When you need FLAG_MUTABLE (e.g., RemoteInput for notification inline replies, or bubbles), the recipient can add extras. Your handler must validate that extras are within expected bounds — never use mutable PendingIntent extras for authorization (e.g., "delete this item") without server-side verification.

3. Content Provider Security

SQL Injection via ContentProvider

ContentProvider queries accept selection and selectionArgs. If you concatenate user input into the selection string, you enable SQL injection.

// VULNERABLE: Selection parameter injection
override fun query(
    uri: Uri,
    projection: Array<out String>?,
    selection: String?,
    selectionArgs: Array<out String>?,
    sortOrder: String?
): Cursor {
    // Attacker passes selection: "1=1) OR (1=1"
    // Selection: "_id = " + selection  → SQL injection
    val raw = "_id = $selection"
    return db.rawQuery("SELECT * FROM data WHERE $raw", null)
}

// SECURE: Use parameterized queries
override fun query(
    uri: Uri,
    projection: Array<out String>?,
    selection: String?,
    selectionArgs: Array<out String>?,
    sortOrder: String?
): Cursor {
    // selectionArgs are always bound as strings — no SQL injection
    return db.query("data", projection, selection, selectionArgs, null, null, sortOrder)
}

// Or with rawQuery — use ? placeholders
db.rawQuery("SELECT * FROM data WHERE _id = ?", arrayOf(userInput))

Path Traversal Attacks

If your ContentProvider serves files based on URI path segments, validate against .. or absolute paths.

// VULNERABLE: content://com.app/files/../../../data/data/com.app/databases/secret.db
override fun openFile(uri: Uri, mode: String): ParcelFileDescriptor {
    val path = uri.pathSegments.joinToString("/")  // Contains ../
    val file = File(baseDir, path)
    return ParcelFileDescriptor.open(file, ParcelFileDescriptor.MODE_READ_ONLY)
}

// SECURE: Resolve to canonical path, validate it's under baseDir
override fun openFile(uri: Uri, mode: String): ParcelFileDescriptor {
    val path = uri.pathSegments.joinToString("/").replace("..", "")
    val file = File(baseDir, path).canonicalFile
    val basePath = baseDir.canonicalPath
    if (!file.canonicalPath.startsWith(basePath)) throw SecurityException("Path traversal denied")
    return ParcelFileDescriptor.open(file, ParcelFileDescriptor.MODE_READ_ONLY)
}

android:grantUriPermissions and Temporary URI Permissions

When you pass a content:// URI to another app via Intent.setData() and FLAG_GRANT_READ_URI_PERMISSION, the system grants temporary read access. Use FLAG_GRANT_PERSISTABLE_URI_PERMISSION only when you need the grant to survive reboots — it's rare and increases risk.

4. WebView Security

JavaScript Injection (XSS via addJavascriptInterface)

Before API 17, any public method could be called from JavaScript. Attackers could inject JS into loaded pages (e.g., via XSS) and invoke arbitrary Java methods. API 17+ requires the @JavascriptInterface annotation — only annotated methods are exposed.

// VULNERABLE (pre-API 17): All public methods exposed
class WebAppInterface(val context: Context) {
    fun showToast(message: String) { Toast.makeText(context, message, LENGTH_SHORT).show() }
    fun getSecret() = "sensitive"  // Exposed to JS!
}
webView.addJavascriptInterface(WebAppInterface(context), "Android")

// SECURE: Only @JavascriptInterface methods are exposed (API 17+)
class WebAppInterface(val context: Context) {
    @JavascriptInterface
    fun showToast(message: String) {
        Toast.makeText(context, message.take(100), LENGTH_SHORT).show()
    }
    // getSecret() is NOT annotated — not exposed
}
webView.settings.javaScriptEnabled = true  // Only if truly needed
webView.addJavascriptInterface(WebAppInterface(context), "Android")

WebView File Access Restrictions

Disable file access when loading remote content:

webView.settings.apply {
    allowFileAccess = false
    allowContentAccess = false
    allowFileAccessFromFileURLs = false
    allowUniversalAccessFromFileURLs = false  // Critical: prevents file:// from accessing any origin
}

Safe Browsing API and Mixed Content

Use WebViewClient.onSafeBrowsingHit() to handle Safe Browsing warnings. For HTTPS pages, ensure mixed content (HTTP resources) is blocked — default on API 21+.

Staff-Level Insight

WebView process isolation (Android 8+) runs WebView in a separate process. If the WebView is compromised, the main app process is not directly exposed. However, the @JavascriptInterface bridge still allows controlled calls into the app — keep the interface minimal and never pass secrets. For highly sensitive flows, consider using a custom URL scheme or postMessage instead of addJavascriptInterface.

5. Tapjacking / Overlay Attacks

How Overlay Attacks Work

A malicious app draws an invisible overlay over your UI. When the user taps "Login," they're actually tapping the attacker's overlay, which may capture credentials or trigger a different action.

// DEFENSE: filterTouchesWhenObscured — ignore touches when overlay on top
override fun onCreate(savedInstanceState: Bundle?) {
    super.onCreate(savedInstanceState)
    // Option 1: Window-level — all touches ignored when obscured
    window.setFlags(
        WindowManager.LayoutParams.FLAG_FILTER_TOUCHES_WHEN_OBSCURED,
        WindowManager.LayoutParams.FLAG_FILTER_TOUCHES_WHEN_OBSCURED
    )
    // Option 2: Per-view — for sensitive buttons (login, pay)
    sensitiveButton.setFilterTouchesWhenObscured(true)
}

FLAG_SECURE

Prevents screenshots and screen recording for the window. Use for sensitive data (passwords, payment info).

window.setFlags(WindowManager.LayoutParams.FLAG_SECURE, WindowManager.LayoutParams.FLAG_SECURE)

Android 12+ restricts overlay touch passthrough — overlays cannot receive touches unless they explicitly opt in, and the system shows a warning when an app draws over others.

6. Clipboard Security

Android 12+ shows a toast when an app reads the clipboard. Android 13+ can auto-clear the clipboard after a short period. Use ClipDescription.EXTRA_IS_SENSITIVE to mark clipboard content as sensitive — the system may suppress the read notification.

// Marking clipboard content as sensitive (API 33+)
val clip = ClipData.newPlainText("label", "sensitive data")
clip.description.extras = PersistableBundle().apply {
    putBoolean(ClipDescription.EXTRA_IS_SENSITIVE, true)
}
clipboard.setPrimaryClip(clip)

// NEVER put passwords in clipboard
// Any app can read it. Use Autofill or in-memory only.
Critical

Never put passwords, tokens, or other secrets in the clipboard. Any app with READ_CLIPBOARD_IN_BACKGROUND (or foreground) can read it. The clipboard is a shared IPC channel — treat it as public. Use AutofillManager for password fields, or keep credentials in memory only.

7. Logging & Data Leakage

Logcat Security

Pre-Android 4.1, any app could read logcat via logcat shell command. On 4.1+, only the app's own logs are readable by default. However, on rooted devices or debug builds, logcat is still accessible.

// VULNERABLE: Logging sensitive data
Log.d("Auth", "Token: $token")
Log.d("User", "Email: ${user.email}")

// SECURE: Strip logs in release or use Timber with release tree
// build.gradle (R8/ProGuard):
// -assumenosideeffects class android.util.Log { *; }

// Or with Timber:
if (BuildConfig.DEBUG) {
    Timber.plant(Timber.DebugTree())
} else {
    Timber.plant(ReleaseTree())  // No-op or crash reporting only
}
Timber.d("Auth: token received")  // Never log token value

StrictMode for Detecting Leaks

Enable StrictMode in debug to detect accidental disk/network on main thread, and to detect leaks.

if (BuildConfig.DEBUG) {
    StrictMode.setThreadPolicy(
        StrictMode.ThreadPolicy.Builder()
            .detectAll()
            .penaltyLog()
            .build()
    )
    StrictMode.setVmPolicy(
        StrictMode.VmPolicy.Builder()
            .detectLeakedSqlLiteObjects()
            .detectLeakedClosableObjects()
            .penaltyLog()
            .build()
    )
}

8. Root Detection & Tamper Detection

Why Root Detection Is Unreliable

Magisk Hide, Magisk DenyList, and similar tools can hide root from apps. Checking for /su, su binary, or ro.debuggable is trivially bypassed. Root detection creates a cat-and-mouse game — attackers always win.

// Naive root detection — easily bypassed
fun isRooted(): Boolean {
    val paths = listOf("/system/bin/su", "/system/xbin/su", "/sbin/su")
    return paths.any { File(it).exists() }
}
// Magisk: these paths don't exist or are hidden via mount namespace

Play Integrity API

The recommended approach. Google attests device integrity, app integrity, and account details. You get a signed token to verify server-side.

// Play Integrity API (replaces SafetyNet)
val integrityManager = IntegrityManagerFactory.create(context)
val integrityTokenRequest = IntegrityTokenRequest.builder()
    .setNonce(generateNonce())
    .setCloudProjectNumber(projectNumber)
    .build()
integrityManager.requestIntegrityToken(integrityTokenRequest)
    .addOnSuccessListener { response ->
        val token = response.token()
        // Send to your server; verify with Google's API
    }
    .addOnFailureListener { e ->
        // Device may be compromised or emulator
    }

Debugger and Frida/Xposed Detection

Detecting debugger attachment: Debug.isDebuggerConnected(). Frida/Xposed inject into the process — detection is possible but fragile (check for loaded libraries, port checks). These are often used for fraud detection in high-value apps, but expect bypasses.

Staff-Level Insight

Defense in depth: combine Play Integrity API (primary), device attestation, and behavioral signals. Don't rely on root detection alone. For banking/finance apps, server-side checks (e.g., device fingerprint, velocity, anomaly detection) matter more than client-side checks. Client-side tamper detection is a deterrent, not a guarantee.

9. Binder Security Model

// Every cross-process call in Android goes through Binder.
// Binder provides caller identity (UID/PID) that cannot be forged.

// Checking caller identity in a Service or ContentProvider:
class SecureService : Service() {
    override fun onBind(intent: Intent): IBinder = binder

    private val binder = object : IMyService.Stub() {
        override fun sensitiveOperation(data: String) {
            // 1. Check caller's UID
            val callerUid = Binder.getCallingUid()
            val callerPackage = packageManager.getNameForUid(callerUid)

            // 2. Verify caller is allowed
            if (callerPackage != "com.example.trusted") {
                throw SecurityException("Unauthorized caller: $callerPackage")
            }

            // 3. Or use permission-based check:
            enforceCallingPermission(
                "com.example.SECURE_ACCESS",
                "Caller lacks SECURE_ACCESS permission"
            )

            // 4. If this service needs to call another service
            //    on behalf of the caller with elevated privileges:
            val token = Binder.clearCallingIdentity()
            try {
                // getCallingUid() now returns THIS service's UID
                performPrivilegedWork()
            } finally {
                Binder.restoreCallingIdentity(token) // ALWAYS restore
            }
        }
    }
}
Staff-Level Insight — Binder Identity and Confused Deputy

The confused deputy attack: a malicious app asks your app (the "deputy") to perform an action that your app has permission for but the caller doesn't. Example: App A asks your ContentProvider to read contacts — your provider has READ_CONTACTS but App A doesn't. If your provider doesn't check the caller's permissions, it acts as a proxy for unauthorized access. Defense: always use enforceCallingPermission() (not enforceCallingOrSelfPermission()) when the operation should require the caller's permission. Use enforceCallingOrSelfPermission() only when the service itself legitimately needs to perform the operation.

10. Broadcast Security

// Broadcasts are a major attack surface — any app can send broadcasts
// to exported receivers.

// VULNERABLE — exported receiver without permission:
// <receiver android:name=".PaymentReceiver"
//     android:exported="true">
//     <intent-filter>
//         <action android:name="com.example.PAYMENT_COMPLETE" />
//     </intent-filter>
// </receiver>
// Any app can send: Intent("com.example.PAYMENT_COMPLETE")

// SECURE — restrict with permission:
// <receiver android:name=".PaymentReceiver"
//     android:exported="true"
//     android:permission="com.example.PAYMENT_PERMISSION">
// Only apps with PAYMENT_PERMISSION can trigger this receiver

// SECURE — use LocalBroadcastManager or explicit intents:
// For app-internal broadcasts, don't use system broadcasts at all:
val intent = Intent(context, PaymentReceiver::class.java)
intent.putExtra("amount", 100.0)
context.sendBroadcast(intent) // Explicit — only your receiver gets it

// Ordered broadcast interception:
// sendOrderedBroadcast() delivers to receivers by priority.
// A higher-priority receiver can abortBroadcast() — preventing
// lower-priority receivers from getting it.
// Attack: malicious app registers a high-priority receiver for
// SMS_RECEIVED, reads the SMS, then aborts it.
// Fix: Android 4.2+ restricts SMS_RECEIVED to system apps.

// Sticky broadcasts (deprecated):
// sendStickyBroadcast() persists the intent — any app can read it later.
// NEVER use for sensitive data. Deprecated since API 21.

11. Service Binding Security

// Bound services are vulnerable to connection hijacking:

// VULNERABLE — exported service without permission:
// <service android:name=".AuthService"
//     android:exported="true" />
// Any app can bind to this service and call its methods

// SECURE — restrict with signature permission:
// <service android:name=".AuthService"
//     android:exported="true"
//     android:permission="com.example.BIND_AUTH_SERVICE" />
// Only apps signed with the same certificate can bind

// Validating the caller in onBind():
override fun onBind(intent: Intent): IBinder? {
    val callerUid = Binder.getCallingUid()
    val callerPackages = packageManager.getPackagesForUid(callerUid)

    // Verify the caller's signing certificate
    val callerInfo = packageManager.getPackageInfo(
        callerPackages?.first() ?: return null,
        PackageManager.GET_SIGNING_CERTIFICATES
    )
    val callerCert = callerInfo.signingInfo?.apkContentsSigners?.first()

    if (callerCert != expectedCertificate) {
        Log.w(TAG, "Unauthorized bind attempt from UID $callerUid")
        return null // Refuse to bind
    }

    return binder
}

12. Deep Link & App Links — Validation and Hijacking

// Deep links vs App Links:
// Deep link: any URI scheme (myapp://...) — NO verification, any app can claim it
// App Link: https:// with Digital Asset Links verification — ONLY verified app handles it

// App Link setup (AndroidManifest.xml):
// <intent-filter android:autoVerify="true">
//     <action android:name="android.intent.action.VIEW" />
//     <category android:name="android.intent.category.DEFAULT" />
//     <category android:name="android.intent.category.BROWSABLE" />
//     <data android:scheme="https" android:host="example.com" />
// </intent-filter>

// Server-side: /.well-known/assetlinks.json
// [{ "relation": ["delegate_permission/common.handle_all_urls"],
//    "target": { "namespace": "android_app",
//                "package_name": "com.example.app",
//                "sha256_cert_fingerprints": ["AB:CD:..."] } }]

// SECURITY RISKS:

// 1. Intent scheme URL injection (WebView):
// A malicious page loads: intent://evil#Intent;scheme=myapp;end
// If your WebView handles this, it can launch arbitrary intents.
// Defense:
class SafeWebViewClient : WebViewClient() {
    override fun shouldOverrideUrlLoading(view: WebView, request: WebResourceRequest): Boolean {
        val url = request.url
        return when (url.scheme) {
            "https", "http" -> false // Let WebView handle
            else -> {
                // NEVER blindly parse intent:// URIs
                // Validate against allowlist of known schemes
                if (url.scheme == "myapp" && url.host == "verified-action") {
                    handleDeepLink(url)
                }
                true
            }
        }
    }
}

// 2. Intent redirection attack:
// Attacker sends an Intent with an embedded Intent in extras.
// Your Activity reads the extra and starts it — launching attacker's target.
// WRONG:
// val redirect = intent.getParcelableExtra<Intent>("next_intent")
// startActivity(redirect) // Attacker controls the target!

// CORRECT — validate before forwarding:
fun handleRedirect(intent: Intent) {
    val nextIntent = intent.getParcelableExtra<Intent>("next_intent") ?: return
    // Validate: only allow intents targeting YOUR package
    val targetPackage = nextIntent.component?.packageName
    if (targetPackage != packageName) {
        Log.w(TAG, "Blocked intent redirection to $targetPackage")
        return
    }
    // Also verify the target component is not exported
    startActivity(nextIntent)
}

// 3. Unverified deep link hijacking:
// If you use custom schemes (myapp://), ANY app can register the same scheme.
// The OS shows a chooser — or worse, the malicious app is the default handler.
// Defense: always use App Links (https:// with autoVerify) for sensitive flows.
// For custom schemes: validate all parameters, never trust data from deep links.

13. ContentProvider Path Traversal

// Path traversal: attacker uses "../" to escape the intended directory.
// If your ContentProvider serves files, this is a critical vulnerability.

// VULNERABLE:
class UnsafeProvider : ContentProvider() {
    override fun openFile(uri: Uri, mode: String): ParcelFileDescriptor? {
        val path = uri.lastPathSegment // Could be "../../secret.db"
        val file = File(context!!.filesDir, path!!)
        return ParcelFileDescriptor.open(file, ParcelFileDescriptor.MODE_READ_ONLY)
    }
}
// Attack: content://com.example.provider/files/../../databases/secret.db
// Result: reads your private database

// SECURE — canonical path validation:
class SafeProvider : ContentProvider() {
    private val baseDir by lazy { context!!.filesDir }

    override fun openFile(uri: Uri, mode: String): ParcelFileDescriptor? {
        val path = uri.lastPathSegment ?: throw SecurityException("No path")
        val file = File(baseDir, path).canonicalFile

        // Verify the resolved path is still under our base directory
        if (!file.path.startsWith(baseDir.canonicalPath)) {
            throw SecurityException("Path traversal blocked: ${uri.path}")
        }

        if (!file.exists()) throw FileNotFoundException(uri.path)
        return ParcelFileDescriptor.open(file, ParcelFileDescriptor.MODE_READ_ONLY)
    }
}

// Additional defenses:
// 1. Use FileProvider instead of custom ContentProvider for file sharing
// 2. Reject paths containing ".." at the URI level
// 3. Use allowlist of permitted file names/patterns
// 4. Set android:grantUriPermissions="true" with specific path patterns
//    rather than granting access to the entire provider
Staff-Level Insight — Binder Driver UID Enforcement

A common L6 question: "Why can't an app forge Binder.getCallingUid()?" The answer is that the Binder kernel driver (binder_transaction() in drivers/android/binder.c) copies the caller's UID/PID from current->cred — the kernel's process credentials. This happens in kernel space, so userspace code cannot tamper with it. The UID is set by the kernel when the process is forked from Zygote via setuid(). This is the foundation of all Android IPC security.
