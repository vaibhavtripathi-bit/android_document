01 — Dependency Injection Fundamentals

The core principle, SOLID alignment, Android's DI evolution from God Activities to Hilt, manual DI, scopes, and the Service Locator distinction

1. What Is Dependency Injection and Why It Matters

Dependency Injection is a single, deceptively simple rule: a class should never create its own collaborators. Instead, collaborators are provided from the outside — "injected" — typically through the constructor. This inverts the flow of control: the caller decides what concrete implementation to supply, not the class itself.

The Three Injection Styles

StyleMechanismPreferred?Android Caveat
Constructor injection Dependencies passed as constructor parameters Always preferred Not possible for Activity/Fragment/Service (OS creates them)
Field (property) injection Framework sets lateinit var fields after construction When constructor isn't available Used by Hilt's @Inject lateinit var in Android entry points
Method injection Dependencies passed as method parameters Rare Useful for one-off operations where the dep varies per call

Without DI vs With DI

Without DI — tightly coupled

class OrderRepository {
    // Creates its own dependency — impossible to test
    // without a real database
    private val db = RoomDatabase.create()
    private val api = RetrofitClient.create()

    fun placeOrder(order: Order) {
        api.submit(order)
        db.orders().insert(order)
    }
}

With DI — testable, swappable

class OrderRepository(
    private val api: OrderApi,
    private val dao: OrderDao,
) {
    fun placeOrder(order: Order) {
        api.submit(order)
        dao.insert(order)
    }
}

// In tests: pass fakes
val repo = OrderRepository(FakeApi(), FakeDao())

The left version is a dead end for unit testing — you'd need Robolectric or an instrumented test just to verify business logic. The right version can be tested with plain JVM unit tests in milliseconds.

DI vs Service Locator — Push vs Pull

Both patterns solve the same problem (decoupling creation from use), but they differ in who initiates the resolution: 

  Dependency Injection (Push)              Service Locator (Pull)
  ─────────────────────────                ──────────────────────────
  Caller ──creates──▶ Dependencies         Class ──asks──▶ Global Registry
         ──passes──▶ Target class                 ◀──gets── Dependency

  class Foo(val bar: Bar)                  class Foo {
                                               val bar = ServiceLocator.get<Bar>()
  ✓ Dependencies visible in signature      ✗ Hidden dependency on the locator
  ✓ Compile-time verification              ✗ Runtime failure if not registered
  ✓ Easy to test (just pass fakes)         ✗ Must configure global state in tests
Gotcha

Koin is often called a "DI framework" but is technically a Service Locator — classes call get() or inject() to pull dependencies from a global registry. This distinction matters in interviews: if asked "what pattern does Koin use?", the precise answer is Service Locator with a DSL for module declaration.

2. SOLID Principles and DI

DI isn't just a convenience — it's the mechanical enforcement of several SOLID principles:

Dependency Inversion Principle (the "D" in SOLID)

"High-level modules should not depend on low-level modules. Both should depend on abstractions." DI makes this concrete: instead of OrderRepository depending on RetrofitOrderApi (a concretion), it depends on OrderApi (an interface). The DI framework wires the concrete implementation. 

// Abstraction — owned by the domain layer
interface OrderApi {
    suspend fun submit(order: Order): Result<OrderConfirmation>
}

// Concretion — lives in the data layer
class RetrofitOrderApi(
    private val service: OrderService,
) : OrderApi {
    override suspend fun submit(order: Order) =
        runCatching { service.submitOrder(order.toDto()).toDomain() }
}

// Consumer — depends only on the interface
class PlaceOrderUseCase(
    private val api: OrderApi,       // abstraction
    private val repo: OrderRepository, // abstraction
)

Interface Segregation

When you inject interfaces, you naturally keep them small. A class that only needs to read orders depends on OrderReader, not a fat OrderRepository that also writes and deletes. DI encourages this because you feel the pain of a bloated interface every time you write a test fake.

Open/Closed Principle

Swapping an implementation (e.g., replacing RetrofitOrderApi with GrpcOrderApi) requires changing only the DI wiring — zero changes to consumers. The system is open for extension (new implementations) and closed for modification (existing consumers untouched).

3. Evolution of DI in Android

Evolution — The Full Timeline

2010–2013: The Dark Ages. No DI. Activities were God objects — 2000-line classes that created their own Retrofit instances, database helpers, and shared preferences inline. Testing meant "run it on a device and tap around." Refactoring was terrifying.

2013–2015: RoboGuice. First serious DI attempt on Android. Used runtime reflection to resolve @Inject annotations. Worked, but added 300–500ms to cold start on low-end devices. Reflection also broke ProGuard aggressively. Abandoned by 2016.

2014–2016: Dagger 1 (Square). Pioneered compile-time code generation for DI on Android. Used @Module + @Provides with an ObjectGraph. Faster than RoboGuice but still used some reflection for graph validation. API was awkward — ObjectGraph.inject(this) in every Activity.

2016–2019: Dagger 2 (Google). Complete rewrite. Zero reflection — pure annotation processing with javac/kapt. Generated code was readable and debuggable. Became the de facto standard at Google and across the industry. Downside: steep learning curve (Components, Subcomponents, Scopes, Qualifiers, multibindings, assisted inject…).

2017–present: Koin. Kotlin-first, DSL-based. No code generation, no annotation processing — pure Kotlin. Resolution happens at runtime, so no compile-time safety. Extremely easy to learn. Popular in smaller teams and KMP projects.

2020–present: Hilt. Google's opinionated layer on Dagger 2. Eliminates boilerplate: no manual Component creation, automatic injection into Android entry points, predefined scopes (@Singleton, @ActivityScoped, @ViewModelScoped). The officially recommended DI solution for Android.

2023–present: kotlin-inject. KSP-based (no kapt overhead). Multiplatform-compatible. Uses Kotlin's type system more idiomatically than Dagger. Still maturing — limited Android lifecycle integration compared to Hilt. Watch this space for KMP projects.

Evolution — Why Android DI Is Uniquely Hard

Unlike server-side frameworks (Spring, Guice) where the framework owns main() and can construct every object, Android's entry points — Activity, Fragment, Service, BroadcastReceiver, ContentProvider — are instantiated by the OS via reflection using a no-arg constructor. You cannot do constructor injection on these classes. This single constraint shaped the entire Android DI ecosystem:

  • RoboGuice used Activity.onCreate() as the injection point
  • Dagger 2 required manual AndroidInjector boilerplate
  • Hilt finally automated this with @AndroidEntryPoint + bytecode transformation
  • Koin uses by inject() delegated properties — Service Locator at the entry point

The pattern that emerged: field injection at the boundary, constructor injection everywhere else. This is the canonical Android DI architecture.

4. Manual DI — The Foundation

Before reaching for Hilt or Koin, every Android developer should understand manual DI. It's what all frameworks automate, and it's what you'll fall back on in KMP modules or library code where Hilt isn't available.

The AppContainer Pattern

// Application-scoped container — lives as long as the process
class AppContainer(context: Context) {
    // Singletons
    val database: AppDatabase = Room.databaseBuilder(
        context, AppDatabase::class.java, "app.db"
    ).build()

    val httpClient: OkHttpClient = OkHttpClient.Builder()
        .addInterceptor(AuthInterceptor(tokenStore))
        .build()

    val orderApi: OrderApi = RetrofitOrderApi(
        Retrofit.Builder()
            .client(httpClient)
            .baseUrl("https://api.example.com/")
            .build()
            .create(OrderService::class.java)
    )

    val orderRepository: OrderRepository = OrderRepositoryImpl(
        api = orderApi,
        dao = database.orderDao(),
    )
}

// Application class holds the container
class MyApp : Application() {
    lateinit var container: AppContainer
        private set

    override fun onCreate() {
        super.onCreate()
        container = AppContainer(this)
    }
}

Activity-Scoped Container

// Scoped to a single Activity's lifetime
class OrderActivityContainer(private val app: AppContainer) {
    val placeOrderUseCase = PlaceOrderUseCase(
        api = app.orderApi,
        repo = app.orderRepository,
    )
    val viewModelFactory = OrderViewModelFactory(placeOrderUseCase)
}

class OrderActivity : AppCompatActivity() {
    private lateinit var container: OrderActivityContainer

    override fun onCreate(savedInstanceState: Bundle?) {
        container = OrderActivityContainer(
            (application as MyApp).container
        )
        super.onCreate(savedInstanceState)

        val viewModel: OrderViewModel by viewModels { container.viewModelFactory }
    }
}
Staff Insight — Why Manual DI Still Matters

In a Google L6 interview, you might be asked to design a DI system from scratch. Understanding the manual approach reveals what Hilt does under the hood: component hierarchy = nested containers, scopes = container lifetimes, @Provides = factory methods in the container. If you can build manual DI, you truly understand the generated Dagger code.

Manual DI is also the recommended approach for small apps (per official Android docs) and for shared KMP modules where neither Hilt nor Dagger are available.

5. Service Locator vs Dependency Injection — Deep Comparison

Aspect Dependency Injection Service Locator
Dependency direction Pushed into the class (constructor/field) Pulled by the class from a registry
Visibility All deps visible in the constructor signature Deps hidden inside method bodies
Compile-time safety Missing dep = compilation error (Dagger/Hilt) Missing dep = runtime crash
Test setup Pass fakes directly: Foo(fakeBar) Must configure global registry before each test
Parallel test execution Safe — no shared mutable state Risky — global registry can leak between tests
Refactoring IDE tracks constructor usages String-based or reified-type lookups are invisible to refactoring tools
Android entry points Requires framework support (Hilt's @AndroidEntryPoint) Works naturally — just call get() in onCreate
Examples Dagger 2, Hilt, kotlin-inject, manual constructor DI Koin, Kodein, custom registries
Key Takeaway

In practice, most Android apps use a hybrid: true DI (constructor injection) for all business logic, with a Service Locator–like pattern at the Android boundary (Activities pulling from the DI graph). Hilt makes this hybrid seamless — @AndroidEntryPoint is field injection (locator-ish), but everything it injects was wired via constructor injection in the graph.

6. DI Scopes and Android Lifecycles

  Scope Hierarchy (longest → shortest lived)
  ═══════════════════════════════════════════

  @Singleton / Application scope
  │   Lives: entire process lifetime
  │   Examples: Retrofit, OkHttpClient, Room DB, Repositories
  │
  ├── @ActivityRetainedScoped
  │   │   Lives: survives config changes (tied to ViewModel store)
  │   │   Examples: use cases shared across fragments in one activity
  │   │
  │   ├── @ActivityScoped
  │   │   │   Lives: single Activity instance (destroyed on config change)
  │   │   │   Examples: Navigator, Activity-specific formatters
  │   │   │
  │   │   ├── @FragmentScoped
  │   │   │       Lives: single Fragment instance
  │   │   │       Examples: Fragment-specific presenters
  │   │   │
  │   │   └── @ViewScoped
  │   │           Lives: single View instance
  │   │
  │   └── @ViewModelScoped
  │           Lives: ViewModel lifetime (survives config change)
  │           Examples: SavedStateHandle wrappers, paging sources
  │
  └── @ServiceScoped (custom)
          Lives: single Service instance

The Scoping Trap

Mismatched scopes are the #1 source of DI-related bugs:

MistakeWhat HappensSymptom
Short-lived scope holds long-lived object Fine — the long-lived object outlives the holder No issue (this is normal)
Long-lived scope holds short-lived object The short-lived object is kept alive by the long-lived scope Memory leak — e.g., Singleton holding an Activity reference
Unscoped dependency used across scopes New instance created every time it's injected Unexpected state — two fragments get different repository instances 
// DANGEROUS: Singleton holding Activity-scoped dependency
@Singleton
class AnalyticsTracker @Inject constructor(
    private val activity: Activity  // Activity will be GC-blocked forever
)

// CORRECT: Inject what you actually need, at the right scope
@Singleton
class AnalyticsTracker @Inject constructor(
    @ApplicationContext private val context: Context  // Application context — safe
)
Bad Design

A common anti-pattern in legacy codebases: storing Activity or Fragment references in singleton-scoped objects "for convenience." This causes memory leaks that are invisible until the app runs long enough for the GC to not reclaim the leaked Activities. Hilt's scope validation catches some of these at compile time, but not all — especially when you pass this from an Activity into a singleton-scoped class manually.

Practical Guidance

New Android-only project? Start with Hilt. It's the path of least resistance with the best tooling and documentation.
KMP project? Koin or kotlin-inject, depending on whether you value simplicity (Koin) or compile-time safety (kotlin-inject).
Library module? Manual DI or kotlin-inject — don't force Hilt on your consumers.
Existing Dagger 2 project? Migrate to Hilt incrementally using @EntryPoint for interop.

8. Staff-Level Considerations

DI at Scale: 100+ Module Builds

In large monorepos (Google, Uber, Cash App), Dagger/Hilt's kapt annotation processing becomes a build-time bottleneck. Each module that uses @Inject triggers a kapt task. With 200+ modules, this adds minutes to clean builds.

  • Anvil (Square) — Kotlin compiler plugin that replaces kapt for Dagger component merging. Cuts build time 30–50% in large projects.
  • KSP migration — Hilt is actively migrating from kapt to KSP. KSP is 2–3× faster and supports incremental processing.
  • Module structure matters — keep Dagger/Hilt annotations in leaf modules. Shared "core" modules should expose interfaces, not Dagger bindings.
Staff Insight — Component Hierarchy Design

At scale, the DI component hierarchy mirrors the app's architecture. A common pattern in large apps:

  • SingletonComponent — network, database, analytics, auth
  • ActivityRetainedComponent — feature-scoped use cases, shared state across fragments
  • ViewModelComponent — screen-specific logic, SavedStateHandle wrappers

The anti-pattern is putting everything in @Singleton. This causes two problems: (1) the app's object graph becomes enormous at startup, increasing cold start time, and (2) objects that should be garbage-collected after a feature is closed are kept alive for the process lifetime. A well-designed hierarchy means objects are created lazily and scoped to the shortest appropriate lifetime.

The "Too Much Abstraction" Trap

A common over-engineering pattern: creating an interface for every single class, even when there's only one implementation and no testing benefit. This adds cognitive overhead and makes navigation harder. 

// Over-engineered — interface adds nothing
interface StringFormatter {
    fun formatDate(date: LocalDate): String
}
class StringFormatterImpl : StringFormatter { ... }

// Better — just use the class directly
class DateFormatter {
    fun format(date: LocalDate): String = ...
}

// When interfaces ARE justified:
// 1. Multiple implementations (real + fake for testing)
// 2. Cross-module boundary (interface in :domain, impl in :data)
// 3. The implementation has side effects (network, disk, system services)

When NOT to Use DI

Not every class needs to go through the DI graph:

  • Data classes / value objects — data class User(val id: String, val name: String) is created inline, not injected
  • Extension functions — stateless utilities don't need injection
  • Kotlin object singletons — if it's truly stateless (e.g., a Comparator), a Kotlin object is fine
  • Mappers / pure functions — fun Order.toDto(): OrderDto has no dependencies to inject
  • Compose UI — composable functions receive state as parameters (a form of DI already); don't inject ViewModels into every composable
Gotcha — Injecting Everything

Teams new to DI often put every class into the Dagger graph, including mappers, validators, and formatters with zero external dependencies. This bloats the generated code, slows compilation, and makes the dependency graph harder to reason about. Rule of thumb: if a class has no dependencies and no side effects, don't inject it.
