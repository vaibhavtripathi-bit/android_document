05 — Testing Coroutines & Flows

Dispatcher injection, runTest, virtual time, StateFlow/SharedFlow testing, Turbine, and every pitfall that catches senior engineers

1. The Dispatcher Injection Pattern

The single most important rule for testable coroutine code: never hardcode dispatchers. Every Dispatchers.IO or Dispatchers.Default in production code is a test seam you've welded shut.

Why Hardcoded Dispatchers Break Tests

When a ViewModel calls withContext(Dispatchers.IO), the coroutine escapes the test dispatcher's control. Virtual time doesn't advance it, advanceUntilIdle() doesn't reach it, and your test becomes a flaky race against real threads.

The DispatcherProvider Pattern

interface DispatcherProvider {
    val main: CoroutineDispatcher
    val io: CoroutineDispatcher
    val default: CoroutineDispatcher
    val unconfined: CoroutineDispatcher
}

class DefaultDispatcherProvider : DispatcherProvider {
    override val main: CoroutineDispatcher = Dispatchers.Main
    override val io: CoroutineDispatcher = Dispatchers.IO
    override val default: CoroutineDispatcher = Dispatchers.Default
    override val unconfined: CoroutineDispatcher = Dispatchers.Unconfined
}

class TestDispatcherProvider(
    testDispatcher: TestDispatcher = StandardTestDispatcher()
) : DispatcherProvider {
    override val main: CoroutineDispatcher = testDispatcher
    override val io: CoroutineDispatcher = testDispatcher
    override val default: CoroutineDispatcher = testDispatcher
    override val unconfined: CoroutineDispatcher = testDispatcher
}

Hilt Module for Dispatcher Injection

@Module
@InstallIn(SingletonComponent::class)
object DispatcherModule {
    @Provides
    @Dispatcher(AppDispatchers.IO)
    fun provideIoDispatcher(): CoroutineDispatcher = Dispatchers.IO

    @Provides
    @Dispatcher(AppDispatchers.Default)
    fun provideDefaultDispatcher(): CoroutineDispatcher = Dispatchers.Default
}

@Qualifier
@Retention(AnnotationRetention.BINARY)
annotation class Dispatcher(val dispatcher: AppDispatchers)

enum class AppDispatchers { IO, Default }

Testability Difference — Injected vs Hardcoded

// ❌ Hardcoded — untestable with virtual time
class BadRepository(private val api: Api) {
    suspend fun fetchUser(id: String): User = withContext(Dispatchers.IO) {
        api.getUser(id)
    }
}

// ✅ Injected — fully controllable in tests
class GoodRepository(
    private val api: Api,
    @Dispatcher(AppDispatchers.IO) private val ioDispatcher: CoroutineDispatcher
) {
    suspend fun fetchUser(id: String): User = withContext(ioDispatcher) {
        api.getUser(id)
    }
}
Staff-Level Insight — Dispatcher Injection Is Architecture

At L6, dispatcher injection isn't a "testing trick" — it's an architectural boundary. The dispatcher determines threading, scheduling, and cancellation behavior. Injecting it means you can swap a StandardTestDispatcher for deterministic tests, a limitedParallelism(1) dispatcher for serialized access, or an Unconfined dispatcher for simpler integration tests. Google's Now in Android sample and all modern Jetpack libraries follow this pattern. If you're reviewing code that hardcodes dispatchers, that's a design review comment, not a nit.

2. kotlinx-coroutines-test — The Testing Library

Evolution — The Coroutine Test API Rewrites

Pre-1.6: TestCoroutineDispatcher and TestCoroutineScope were separate classes requiring manual lifecycle management. You had to call cleanupTestCoroutines() yourself, and the interaction between the dispatcher and scope was confusing.

1.6.0: The entire API was rewritten. runTest replaced runBlockingTest. StandardTestDispatcher and UnconfinedTestDispatcher replaced TestCoroutineDispatcher. Automatic cleanup, unified scope, and virtual time control became first-class.

Current (1.8+): runTest is the standard. All old APIs are deprecated. TestScope provides backgroundScope for long-running collectors. The API is stable and mature.

Core APIs at a Glance

APIPurposeKey Behavior
runTest { }Creates a TestScope, runs the blockAuto-advances virtual time, fails on uncaught exceptions, 60s default timeout
StandardTestDispatcherQueues coroutines for explicit advancementNothing runs until you call advanceUntilIdle(), advanceTimeBy(), or runCurrent()
UnconfinedTestDispatcherRuns coroutines eagerlyLaunched coroutines start immediately (like Dispatchers.Unconfined), less control over ordering
advanceUntilIdle()Run all pending coroutinesAdvances virtual time until the queue is empty
advanceTimeBy(ms)Advance virtual clock by msRuns coroutines whose delay() has expired at each step
runCurrent()Run only currently queued workDoes not advance time — runs what's ready right now
currentTimeRead virtual clockReturns the current virtual time in milliseconds

StandardTestDispatcher vs UnconfinedTestDispatcher

@Test
fun standardDispatcher_queuesWork() = runTest {
    var value = 0
    launch { value = 1 }
    assertEquals(0, value)   // not yet executed
    advanceUntilIdle()
    assertEquals(1, value)   // now executed
}

@Test
fun unconfinedDispatcher_runsEagerly() = runTest(UnconfinedTestDispatcher()) {
    var value = 0
    launch { value = 1 }
    assertEquals(1, value)   // already executed
}

Rule of thumb: Use StandardTestDispatcher (the default) for anything involving time, ordering, or state transitions. Use UnconfinedTestDispatcher only when you want "fire and check" simplicity and don't care about intermediate states.

Suspend Function with Delay

// Production
suspend fun retryWithBackoff(block: suspend () -> Unit) {
    var delayMs = 1000L
    repeat(3) {
        try { block(); return } catch (e: Exception) {
            delay(delayMs)
            delayMs *= 2
        }
    }
}

// Test — virtual time makes this instant
@Test
fun retryWithBackoff_retriesThreeTimes() = runTest {
    var attempts = 0
    retryWithBackoff {
        attempts++
        if (attempts < 3) throw IOException("fail")
    }
    assertEquals(3, attempts)
    // Virtual time advanced: 1000 + 2000 = 3000ms
    assertEquals(3000, currentTime)
}

4. Testing ViewModels with viewModelScope

The Problem

viewModelScope uses Dispatchers.Main.immediate internally. On the JVM (no Android Looper), this throws:

java.lang.IllegalStateException:
  Module with the Main dispatcher had failed to initialize.
  For tests, Dispatchers.setMain from kotlinx-coroutines-test module can be used.

Solution 1: Manual setMain / resetMain

class MyViewModelTest {
    private val testDispatcher = StandardTestDispatcher()

    @Before
    fun setup() {
        Dispatchers.setMain(testDispatcher)
    }

    @After
    fun tearDown() {
        Dispatchers.resetMain()
    }

    @Test
    fun loadData_updatesState() = runTest(testDispatcher) {
        val vm = MyViewModel(FakeRepository())
        vm.loadData()
        advanceUntilIdle()
        assertEquals(UiState.Success(data), vm.uiState.value)
    }
}

Solution 2: MainDispatcherRule (JUnit 4)

class MainDispatcherRule(
    private val testDispatcher: TestDispatcher = UnconfinedTestDispatcher()
) : TestWatcher() {
    override fun starting(description: Description) {
        Dispatchers.setMain(testDispatcher)
    }
    override fun finished(description: Description) {
        Dispatchers.resetMain()
    }
}

// Usage
class MyViewModelTest {
    @get:Rule
    val mainDispatcherRule = MainDispatcherRule()

    @Test
    fun loadData_updatesState() = runTest {
        val vm = MyViewModel(FakeRepository())
        vm.loadData()
        advanceUntilIdle()
        assertEquals(UiState.Success(data), vm.uiState.value)
    }
}

Solution 3: Inject Dispatcher into ViewModel (Best)

@HiltViewModel
class SearchViewModel @Inject constructor(
    private val repo: SearchRepository,
    @Dispatcher(AppDispatchers.IO) private val ioDispatcher: CoroutineDispatcher
) : ViewModel() {

    private val _results = MutableStateFlow<UiState>(UiState.Idle)
    val results: StateFlow<UiState> = _results.asStateFlow()

    fun search(query: String) {
        viewModelScope.launch {
            _results.value = UiState.Loading
            val data = withContext(ioDispatcher) { repo.search(query) }
            _results.value = UiState.Success(data)
        }
    }
}

// Test — no Dispatchers.setMain needed if ViewModel doesn't use Main directly
@Test
fun search_emitsLoadingThenSuccess() = runTest {
    Dispatchers.setMain(StandardTestDispatcher(testScheduler))
    val vm = SearchViewModel(FakeSearchRepo(), StandardTestDispatcher(testScheduler))

    vm.search("kotlin")
    assertEquals(UiState.Loading, vm.results.value)

    advanceUntilIdle()
    assertIs<UiState.Success>(vm.results.value)

    Dispatchers.resetMain()
}
Gotcha — testScheduler Must Be Shared

When using StandardTestDispatcher, the dispatcher you pass to runTest and the one injected into the ViewModel must share the same TestCoroutineScheduler. Otherwise, advanceUntilIdle() only advances the test scope's scheduler, and the ViewModel's coroutines remain queued on a separate scheduler. Use StandardTestDispatcher(testScheduler) inside runTest to share the scheduler, or use the testScheduler property of TestScope.

5. Testing StateFlow and SharedFlow

StateFlow: Direct Value Access

StateFlow always has a current value. The simplest test approach is to check .value after triggering actions.

@Test
fun counter_incrementsState() = runTest {
    val vm = CounterViewModel()
    assertEquals(0, vm.count.value)

    vm.increment()
    advanceUntilIdle()
    assertEquals(1, vm.count.value)

    vm.increment()
    advanceUntilIdle()
    assertEquals(2, vm.count.value)
}

StateFlow: Collecting Emissions Over Time

@Test
fun stateFlow_collectsAllTransitions() = runTest {
    val vm = CounterViewModel()
    val emissions = mutableListOf<Int>()

    // backgroundScope keeps the collector alive without blocking runTest
    backgroundScope.launch(UnconfinedTestDispatcher(testScheduler)) {
        vm.count.collect { emissions.add(it) }
    }

    vm.increment()
    advanceUntilIdle()
    vm.increment()
    advanceUntilIdle()

    assertEquals(listOf(0, 1, 2), emissions)
}

SharedFlow: Must Collect Before Emission

class EventViewModel : ViewModel() {
    private val _events = MutableSharedFlow<UiEvent>()
    val events: SharedFlow<UiEvent> = _events.asSharedFlow()

    fun triggerNavigation() {
        viewModelScope.launch { _events.emit(UiEvent.NavigateBack) }
    }
}

@Test
fun sharedFlow_collectBeforeEmit() = runTest {
    Dispatchers.setMain(UnconfinedTestDispatcher(testScheduler))
    val vm = EventViewModel()
    val events = mutableListOf<UiEvent>()

    // ⚠️ Must start collecting BEFORE triggering emission
    backgroundScope.launch(UnconfinedTestDispatcher(testScheduler)) {
        vm.events.collect { events.add(it) }
    }

    vm.triggerNavigation()
    advanceUntilIdle()

    assertEquals(listOf(UiEvent.NavigateBack), events)
    Dispatchers.resetMain()
}
Trap — SharedFlow Emissions Are Lost Without Active Collectors

SharedFlow (with default replay = 0) drops emissions that have no active subscribers. If you call emit() before collect(), the event vanishes silently. Your test passes with an empty list — giving you a false green. Always launch the collector in backgroundScope before triggering the action. This is the #1 SharedFlow testing mistake.

6. Turbine — The Flow Testing Library

Turbine (app.cash.turbine) replaces manual collection boilerplate with a structured, timeout-based API. It's the de-facto standard for Flow testing in production Android codebases.

Evolution — Turbine API Progression

v0.x: Standalone flow.test { } extension. Each test block had an implicit 1-second timeout. No support for testing multiple flows simultaneously.

v1.0+: Added turbineScope { } for testing multiple flows in a single block. Added testIn(scope) for manual lifecycle control. Improved error messages with emission history. Configurable timeouts. Integration with runTest virtual time.

Core Turbine API

MethodWhat It DoesFails When
awaitItem()Suspends until next emission, returns itTimeout (no emission within default 3s)
awaitComplete()Suspends until Flow completesFlow emits an item instead of completing
awaitError()Suspends until Flow throwsFlow emits an item or completes normally
expectNoEvents()Asserts nothing happened (no emit, complete, or error)Any event occurred
cancelAndIgnoreRemainingEvents()Cancels collection, ignores pending eventsNever fails — cleanup utility
cancelAndConsumeRemainingEvents()Cancels and returns remaining eventsNever fails — returns list of events

Basic Turbine Usage

@Test
fun stateFlow_withTurbine() = runTest {
    val vm = CounterViewModel()

    vm.count.test {
        assertEquals(0, awaitItem())       // initial value

        vm.increment()
        assertEquals(1, awaitItem())

        vm.increment()
        assertEquals(2, awaitItem())

        cancelAndIgnoreRemainingEvents()
    }
}

Testing a Flow Operator Chain

fun userNamesFlow(repo: UserRepository): Flow<String> =
    repo.observeUsers()
        .map { it.name }
        .filter { it.isNotBlank() }
        .distinctUntilChanged()

@Test
fun userNamesFlow_filtersAndDeduplicates() = runTest {
    val fakeRepo = FakeUserRepository()

    userNamesFlow(fakeRepo).test {
        fakeRepo.emit(User("Alice"))
        assertEquals("Alice", awaitItem())

        fakeRepo.emit(User("Alice"))  // duplicate
        expectNoEvents()               // filtered by distinctUntilChanged

        fakeRepo.emit(User(""))       // blank
        expectNoEvents()               // filtered by filter

        fakeRepo.emit(User("Bob"))
        assertEquals("Bob", awaitItem())

        cancelAndIgnoreRemainingEvents()
    }
}

Testing Hot Flows (SharedFlow) with Turbine

@Test
fun sharedFlow_withTurbine() = runTest {
    val events = MutableSharedFlow<UiEvent>()

    events.test {
        // Turbine's .test {} starts collecting immediately — no race condition
        events.emit(UiEvent.ShowSnackbar("Saved"))
        assertEquals(UiEvent.ShowSnackbar("Saved"), awaitItem())

        events.emit(UiEvent.NavigateBack)
        assertEquals(UiEvent.NavigateBack, awaitItem())

        cancelAndIgnoreRemainingEvents()
    }
}

Testing Multiple Flows with turbineScope

@Test
fun multipleFlows_withTurbineScope() = runTest {
    val vm = DashboardViewModel(FakeRepo())

    turbineScope {
        val stateTurbine = vm.uiState.testIn(backgroundScope)
        val eventTurbine = vm.events.testIn(backgroundScope)

        assertEquals(UiState.Success(data), stateTurbine.awaitItem())

        eventTurbine.cancelAndIgnoreRemainingEvents()
    }
}

7. Testing Flow Operators

Testing debounce

class SearchViewModel(private val repo: SearchRepository) : ViewModel() {
    private val _query = MutableStateFlow("")
    val results: StateFlow<List<String>> = _query
        .debounce(300)
        .filter { it.length >= 2 }
        .flatMapLatest { repo.search(it) }
        .stateIn(viewModelScope, SharingStarted.WhileSubscribed(5000), emptyList())

    fun onQueryChanged(q: String) { _query.value = q }
}

@Test
fun search_debouncesInput() = runTest {
    Dispatchers.setMain(StandardTestDispatcher(testScheduler))
    val vm = SearchViewModel(FakeSearchRepo())

    vm.results.test {
        assertEquals(emptyList(), awaitItem())  // initial

        vm.onQueryChanged("ko")
        advanceTimeBy(100)
        expectNoEvents()  // debounce hasn't fired

        vm.onQueryChanged("kot")
        advanceTimeBy(100)
        expectNoEvents()  // debounce reset

        advanceTimeBy(300)  // debounce fires for "kot"
        assertEquals(listOf("Kotlin"), awaitItem())

        cancelAndIgnoreRemainingEvents()
    }
    Dispatchers.resetMain()
}

fun safeUserFlow(repo: UserRepository): Flow<UiState> =
    repo.observeUsers()
        .map<List<User>, UiState> { UiState.Success(it) }
        .catch { emit(UiState.Error(it.message ?: "Unknown")) }

@Test
fun safeUserFlow_catchesErrors() = runTest {

        assertIs<UiState.Error>(state)
        assertEquals("Network error", state.message)

Testing stateIn / shareIn Timing

@Test
fun stateIn_whileSubscribed_stopsAfterTimeout() = runTest {
    val upstream = MutableSharedFlow<Int>()
    val stateFlow = upstream.stateIn(
        scope = backgroundScope,
        started = SharingStarted.WhileSubscribed(stopTimeoutMillis = 5000),
        initialValue = 0
    )

    val turbine = stateFlow.testIn(backgroundScope)
    assertEquals(0, turbine.awaitItem())

    upstream.emit(42)
    assertEquals(42, turbine.awaitItem())

    turbine.cancel()
    // After 5000ms with no subscribers, upstream collection stops
    advanceTimeBy(5001)
    // Upstream is now inactive — new subscriber gets stale value
}

8. Testing Channels and callbackFlow

Testing a Channel

@Test
fun channel_sendsAndReceives() = runTest {
    val channel = Channel<String>(Channel.BUFFERED)

    launch { channel.send("hello") }
    launch { channel.send("world") }

    advanceUntilIdle()
    assertEquals("hello", channel.receive())
    assertEquals("world", channel.receive())
    channel.close()
}

Testing callbackFlow

// Production — wrapping a callback-based API
fun observeLocation(client: LocationClient): Flow<Location> = callbackFlow {
    val callback = object : LocationCallback() {
        override fun onLocationResult(result: LocationResult) {
            result.lastLocation?.let { trySend(it) }
        }
    }
    client.requestLocationUpdates(callback)
    awaitClose { client.removeLocationUpdates(callback) }
}

// Test — fake the callback source
@Test
fun observeLocation_emitsLocations() = runTest {
    val fakeClient = FakeLocationClient()

    observeLocation(fakeClient).test {
        fakeClient.pushLocation(Location(37.7749, -122.4194))
        val loc = awaitItem()
        assertEquals(37.7749, loc.latitude, 0.001)

        fakeClient.pushLocation(Location(34.0522, -118.2437))
        assertEquals(34.0522, awaitItem().latitude, 0.001)

        cancelAndIgnoreRemainingEvents()
    }

    // Verify cleanup was called
    assertTrue(fakeClient.isRemoved)
}

// The fake
class FakeLocationClient : LocationClient {
    private var callback: LocationCallback? = null
    var isRemoved = false

    override fun requestLocationUpdates(cb: LocationCallback) { callback = cb }
    override fun removeLocationUpdates(cb: LocationCallback) { isRemoved = true }

    fun pushLocation(loc: Location) {
        callback?.onLocationResult(LocationResult.create(listOf(loc)))
    }
}
Staff-Level Insight — callbackFlow Testing Reveals Design Quality

The testability of a callbackFlow is a direct measure of how well you've designed the abstraction boundary. If testing requires mocking 5 layers of Android framework classes, the callbackFlow is too tightly coupled to the platform. The pattern above — a thin LocationClient interface with a fake — is the right level of abstraction. In interviews, demonstrating that you'd extract an interface for the callback source (rather than mocking FusedLocationProviderClient directly) signals L6 thinking: you're designing for testability, not fighting the framework.

9. Common Pitfalls and Patterns

Pitfall: Forgetting advanceUntilIdle()

// ❌ This test PASSES but tests NOTHING — the coroutine never ran
@Test
fun broken_forgotToAdvance() = runTest {
    val vm = MyViewModel(FakeRepo())
    vm.loadData()
    // Missing: advanceUntilIdle()
    assertEquals(UiState.Idle, vm.uiState.value)  // still initial state!
}

// ✅ Fixed
@Test
fun correct_advancesTime() = runTest {
    val vm = MyViewModel(FakeRepo())
    vm.loadData()
    advanceUntilIdle()
    assertIs<UiState.Success>(vm.uiState.value)
}

Pitfall: UnconfinedTestDispatcher Hides Ordering Bugs

// UnconfinedTestDispatcher runs everything eagerly — you can't observe
// intermediate states like Loading → Success transitions
@Test
fun misleading_unconfinedHidesTransitions() = runTest(UnconfinedTestDispatcher()) {
    val vm = MyViewModel(FakeRepo())
    vm.loadData()
    // You'll never see UiState.Loading — it was already replaced by Success
    // Use StandardTestDispatcher to observe the Loading → Success transition
}

Pitfall: Missing Dispatchers.Main Setup

// This crashes immediately:
// IllegalStateException: Module with the Main dispatcher had failed to initialize
@Test
fun crashes_noMainDispatcher() = runTest {
    val vm = MyViewModel(FakeRepo())  // viewModelScope uses Dispatchers.Main
    vm.loadData()  // 💥
}

// Fix: set Main before test
@get:Rule val mainRule = MainDispatcherRule()

Pitfall: runTest Timeout with Real Delays

// ❌ If production code uses a real (non-test) dispatcher with delay(),
// runTest's virtual time can't advance it. The test hangs for 60s then fails.
@Test
fun hangs_realDispatcherDelay() = runTest {
    withContext(Dispatchers.Default) {  // escapes test scheduler!
        delay(10_000)  // real 10-second wait
    }
}
// Fix: inject the dispatcher so delay() uses virtual time

Pattern: backgroundScope for Long-Running Collectors

@Test
fun backgroundScope_keepsCollectorAlive() = runTest {
    val vm = MyViewModel(FakeRepo())
    val states = mutableListOf<UiState>()

    // backgroundScope is cancelled automatically when runTest finishes
    // Without it, the infinite collect() would prevent runTest from completing
    backgroundScope.launch(UnconfinedTestDispatcher(testScheduler)) {
        vm.uiState.collect { states.add(it) }
    }

    vm.loadData()
    advanceUntilIdle()

    assertEquals(
        listOf(UiState.Idle, UiState.Loading, UiState.Success(data)),
        states
    )
}

Pattern: turbineScope for Multiple Simultaneous Flows

@Test
fun turbineScope_testMultipleFlows() = runTest {
    val vm = ChatViewModel(FakeRepo())

    turbineScope {
        val messages = vm.messages.testIn(backgroundScope)
        val typing = vm.typingIndicator.testIn(backgroundScope)
        val errors = vm.errors.testIn(backgroundScope)

        assertEquals(emptyList(), messages.awaitItem())
        assertEquals(false, typing.awaitItem())

        vm.sendMessage("Hello")
        assertEquals(true, typing.awaitItem())

        advanceUntilIdle()
        assertEquals(listOf(Message("Hello")), messages.awaitItem())
        assertEquals(false, typing.awaitItem())

        messages.cancelAndIgnoreRemainingEvents()
        typing.cancelAndIgnoreRemainingEvents()
        errors.cancelAndIgnoreRemainingEvents()
    }
}
