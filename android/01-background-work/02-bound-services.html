Bound Services — Overview

Client-server pattern within Android, the three binding approaches, and when to use each Full IPC deep-dive → separate doc

1. What Is a Bound Service

A bound service acts as a server in a client-server interface. Components (Activities, Fragments, other Services) bind to it, get an IBinder reference, and use it to call methods directly on the service.

Key difference from started services: the bound service's lifecycle is tied to its clients. When the last client unbinds, the system destroys it (unless it's also a started service).

2. The Three Binding Approaches

Approach Same Process? Cross-Process? Complexity When to Use
Binder (local) Yes No Low Service is private to your app, same process. Most common case.
Messenger Yes Yes Medium Simple cross-process messaging. Single-threaded (uses Handler). No concurrent calls.
AIDL Yes Yes High Full cross-process IPC. Multi-threaded. Needed when multiple apps call your service concurrently.

2a. Local Binder Pattern (Most Common)

class LocalMusicService : Service() {

    inner class LocalBinder : Binder() {
        fun getService(): LocalMusicService = this@LocalMusicService
    }

    private val binder = LocalBinder()

    override fun onBind(intent: Intent): IBinder = binder

    fun getCurrentTrack(): Track { /* ... */ }
    fun play() { /* ... */ }
    fun pause() { /* ... */ }
}

// In Activity:
class PlayerActivity : AppCompatActivity() {
    private var musicService: LocalMusicService? = null
    private var bound = false

    private val connection = object : ServiceConnection {
        override fun onServiceConnected(name: ComponentName, service: IBinder) {
            val binder = service as LocalMusicService.LocalBinder
            musicService = binder.getService()
            bound = true
        }

        override fun onServiceDisconnected(name: ComponentName) {
            // Called only on unexpected disconnection (crash, process kill)
            // NOT called on normal unbind
            bound = false
        }
    }

    override fun onStart() {
        super.onStart()
        Intent(this, LocalMusicService::class.java).also { intent ->
            bindService(intent, connection, Context.BIND_AUTO_CREATE)
        }
    }

    override fun onStop() {
        super.onStop()
        if (bound) {
            unbindService(connection)
            bound = false
        }
    }
}
Staff-Level Insight — Binding Flags

BIND_AUTO_CREATE creates the service if it doesn't exist. Without this flag, bindService() only binds if the service is already running (started via startService()). Other flags:

  • BIND_IMPORTANT — raises service process priority to match the client (foreground if client is foreground)
  • BIND_ABOVE_CLIENT — service process is more important than the client. System kills client before service under memory pressure.
  • BIND_NOT_FOREGROUND — prevents bound service from getting foreground priority even if client is foreground. Useful for non-critical bindings.

2b. Messenger (Brief)

Wraps a Handler into an IBinder. Messages are serialized across process boundaries. Single-threaded — all messages queue into one Handler.

// Service side
class MessengerService : Service() {
    private val handler = object : Handler(Looper.getMainLooper()) {
        override fun handleMessage(msg: Message) {
            when (msg.what) {
                MSG_SAY_HELLO -> { /* handle */ }
            }
        }
    }
    private val messenger = Messenger(handler)

    override fun onBind(intent: Intent): IBinder = messenger.binder
}

// Client side — sends Message objects, not direct method calls

2c. AIDL (Brief)

Android Interface Definition Language. You define a .aidl file, the build system generates Java/Kotlin stubs. Calls are dispatched to a Binder thread pool (not the main thread), so the service must be thread-safe.

// IRemoteService.aidl
interface IRemoteService {
    int getPid();
    void registerCallback(IRemoteCallback cb);
}
Scope Note

AIDL and the Binder driver internals (transaction buffer, Parcel serialization, thread pool management) will be covered in a dedicated IPC deep-dive document. For this series, know the three approaches and when to pick each.

3. Lifecycle Nuances

onUnbind() and onRebind()

When the last client unbinds, onUnbind() is called. If onUnbind() returns true, the next client that binds triggers onRebind() instead of onBind(). This is useful for services that are also started — they persist after unbind and can rebind efficiently.

Staff-Level Insight — Context Leaks

The most dangerous pattern: binding a Service from an Activity with BIND_AUTO_CREATE and forgetting to unbind. If the Activity is destroyed (config change, back press) without calling unbindService(), you get a ServiceConnectionLeaked exception. The system logs it and force-unbinds, but the service may be destroyed unexpectedly.

Rule: Always unbind in the symmetric lifecycle callback — bind in onStart() → unbind in onStop(). Never bind in onCreate() and unbind in onDestroy() for Activities (config changes would leak).

Interview Gotcha

Q: "What's the difference between onServiceDisconnected and unbinding?"
onServiceDisconnected() is called only when the service crashes or is killed — NOT during normal unbind. After unbindService(), neither onServiceDisconnected nor onServiceConnected is called. Many candidates confuse this.

4. Started + Bound (Hybrid)

A service can be both started and bound. This is common for music players:

  • startService() keeps it alive independently
  • bindService() lets the UI control it
  • Destroyed only when both conditions are met: stopped AND all clients unbound
class MusicService : Service() {
    private val binder = LocalBinder()

    inner class LocalBinder : Binder() {
        fun getService() = this@MusicService
    }

    override fun onStartCommand(intent: Intent?, flags: Int, startId: Int): Int {
        // Handle play/pause/skip commands via Intent
        return START_STICKY
    }

    override fun onBind(intent: Intent): IBinder = binder
}
