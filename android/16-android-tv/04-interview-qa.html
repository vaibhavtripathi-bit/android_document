Android TV — Interview Q&A

20 L6-depth questions from Hotstar, Sony LIV, Amazon Prime TV, and YouTube TV interviews

Leanback UI & Fragment Architecture

Q1. Explain the structure of a typical Android TV home screen built with Leanback. What fragments are involved and how do they relate to each other? Hotstar

The root is BrowseSupportFragment, which has two panels: HeadersSupportFragment (left rail — category labels) and RowsSupportFragment (right area — horizontal card rows). Each row is a ListRow = HeaderItem + ArrayObjectAdapter + Presenter. A Presenter is the factory that creates and binds card views — equivalent to RecyclerView's ViewHolder + Adapter combined. For sections that need a fully custom layout (e.g. a live sports scoreboard), use a PageRow which swaps the entire right panel with a custom Fragment returned by onCreateMainFragment().

Q2. What is a Presenter in Leanback and how does it differ from a RecyclerView.Adapter? Sony LIV

A Presenter handles both view creation and binding for a single item type. It has three methods: onCreateViewHolder(parent) → creates the card view, onBindViewHolder(holder, item) → loads data, onUnbindViewHolder(holder) → clears resources (images, listeners). Unlike RecyclerView.Adapter, a Presenter is not coupled to a specific list — the same Presenter can be reused in multiple rows across the app. The ObjectAdapter (usually ArrayObjectAdapter) holds the data and is given a Presenter to use for all items. The key difference: Presenter is item-type-aware, not position-aware.

Q3. How would you implement a "Continue Watching" row that updates in real time as the user watches content in another session? Hotstar Hard

Use a Flow from Room that observes the watch_progress table. The ViewModel exposes StateFlow<List<VideoItem>>. The BrowseFragment collects it and calls rowAdapter.setItems(newList, DiffCallback()) on update — this performs diffing internally and only rebinds changed cards. The key is DiffCallback: without it, setItems() clears and reloads the entire row, causing a visible flash. Server-side sync: use a background WorkManager task that polls or WebSocket-pushes watch progress from the server and writes to Room, which triggers the Flow automatically.

Q4. When would you use PlaybackSupportFragment with LeanbackPlayerAdapter vs building a fully custom playback UI? Amazon

LeanbackPlayerAdapter: use for prototyping, internal tools, or apps where playback controls are simple (play/pause, seek). It gives free D-pad seek bar, transport controls overlay, and system integration with no extra code.

Custom playback UI: production streaming apps always go custom. Reasons: custom skip animations (+10s circle), episode carousel overlay, subtitle track picker, audio quality selector, LIVE badge with DVR scrubbing indicator, multi-audio track selection. Extend VideoSupportFragment directly, inflate your own control overlay, and handle every KEYCODE_DPAD_* and KEYCODE_MEDIA_* yourself. The extra code cost is ~1000 lines but results in the polished UX users expect from a streaming app.

D-pad Navigation & Focus

Q5. A user on your TV app reports that after a dialog dismisses, they can no longer navigate with the D-pad. What's the bug and how do you fix it? Sony LIV Hard

This is a focus trap. When the dialog appeared, focus moved into it. When it dismissed, focus was not explicitly returned to the underlying content. With no focused view, D-pad events have nowhere to go. Fix: in the dialog's dismiss callback, call previouslyFocusedView.requestFocus() — save a reference to the focused view before showing the dialog. If the original view is unknown, call activity.window.decorView.requestFocus() and let Android find the first focusable view. Prevention: always test dialogs by pressing Back without touching any button inside — this triggers the dismiss path that many developers forget to handle.

Q6. How does Android determine which view receives focus when the user presses D-pad right? How do you override this? YouTube

Android uses a geometric search algorithm: it finds all focusable views in the requested direction, scores them by their overlap with a beam projected from the current focused view's right edge, and picks the highest-scored candidate. This works well for regular grid layouts but fails when views have gaps, are partially off-screen, or when the layout is irregular (e.g. a header row with different height).

Override with android:nextFocusRight="@id/target_view" in XML or view.nextFocusRightId = R.id.target_view in code. For programmatic focus control in RecyclerView: override LinearLayoutManager.onRequestChildFocus() to scroll the target into view and then call child.requestFocus().

Q7. How do you implement a "hold D-pad right to fast seek" interaction in a video player? Hotstar

Use KeyEvent.getRepeatCount() in Activity.onKeyDown(). On the first press (repeatCount == 0), seek +10s. After 5 repeats, seek +30s per event. After 15 repeats, seek +60s. This gives the Netflix-style accelerating scrub feel:

KEYCODE_DPAD_RIGHT → repeatCount 0–4: seek +10s | 5–14: seek +30s | 15+: seek +60s

On ACTION_UP, reset the acceleration state. Show a toast/overlay with the total seek offset while the key is held. Always consume the event (return true) so focus doesn't traverse while seeking.

Q8. A card row in your TV app loses scroll position every time the user returns from the detail screen. How do you fix this? Amazon

In BrowseSupportFragment, save selectedPosition (row index) and within each row, save the card index from the row's HorizontalGridView.selectedPosition. Do this in onPause() or via the ViewModel as StateFlow. On return (onResume()), call setSelectedPosition(savedRow, false) to restore the row without animation, then find the row's ListRowPresenter.ViewHolder and call gridView.setSelectedPosition(savedCardIndex). Store these indices in the ViewModel so they survive configuration changes.

Content Discovery & TV Platform

Q9. How do you publish content to the Android TV home screen so your app's content is visible without the user opening the app? Hotstar

Use TV Home Screen Channels via TvContractCompat:
1. Insert a Channel with TYPE_PREVIEW into the content resolver — this registers your channel.
2. Insert PreviewProgram rows for each content item — title, description, poster art URI, duration, and an intentUri that deep-links directly to playback.
3. On first launch, call ChannelHelper.requestChannelBrowsable(context, channelId) to show the "Add Channel" system dialog — do this only once.
4. Update programs periodically (every few hours) via WorkManager to reflect fresh content.
When a user taps a card on the home screen, your app receives the deep link intent and jumps directly to playback — zero extra navigation taps.

Q10. How does voice search integration work on Android TV? Walk through the full flow from "Hey Google, play Mirzapur on Hotstar" to the video playing. Amazon Hard

1. Google Assistant receives the voice query and identifies the app via Android TV app indexing (you must index your content with Google so it knows "Mirzapur" maps to your content ID).
2. Assistant sends a VIEW intent with your deep link URI: hotstar://play/mirzapur-s1-e1.
3. Your MainActivity receives the intent in onNewIntent() (or onCreate() if cold start), extracts the content ID, and calls navigateToPlayback(contentId).
4. Alternatively, Assistant sends a SEARCH intent with the query string to your registered SearchActivity.
Registration: declare android.intent.action.SEARCH intent filter + searchable.xml meta-data + publish your content catalog to Google via the TV app content API (requires Google partnership for direct playback routing).

Q11. Your TV app needs to support both the old Leanback launcher and the new Google TV launcher. What are the differences and how do you handle both? YouTube

Leanback launcher (Android TV 5–9, many budget TVs): shows a grid of app icons + channels from TvContractCompat. App banner is the entry point.

Google TV launcher (Android TV 10+, Chromecast with Google TV): curated content rows driven by Google's discovery algorithm. Apps must integrate with the Watch Next API (WatchNextProgram) and publish content to Google's indexing service for their content to appear in Google TV's "For You" tab.

Handle both by: publishing home screen channels (works on Leanback launcher) AND calling TvContractCompat.WatchNextPrograms.CONTENT_URI to add to Watch Next (works on Google TV). Both coexist — the right API surfaces on the right launcher automatically.

Performance & Memory

Q12. Your TV app runs smoothly on a Nvidia Shield but crashes on budget TVs like Vu or Thomson. What are the likely causes? Sony LIV

Budget India TVs typically have: 1 GB RAM (vs 3 GB on Shield), slower Cortex-A53 CPU, Mali-400 GPU, and 8 GB storage with ~4 GB available.
Common causes:
OOM crash: image cache too large. Fix: set MemoryCategory.LOW in Glide, cap memory cache to 50 MB, load only focused card backdrop images.
ANR: heavy work on main thread. Fix: all network/DB on coroutines + IO dispatcher.
Codec not found: budget TVs may lack AV1 or HEVC hardware decoders. ExoPlayer falls back to software decode but this can be too slow. Fix: check codec availability and restrict to H.264 on low-end TVs.
Storage OOM: cache directory fills 4 GB storage. Fix: set Glide disk cache to 100 MB max; clear download cache when storage below threshold.

Q13. How do you prevent ExoPlayer initialization latency when navigating from the detail screen to playback? Hotstar Hard

Use a singleton PlayerManager scoped to the Application or Activity. Create ExoPlayer once when the app launches (or at least when the user enters the detail screen) and reuse it. When entering the detail screen, pre-load the media item (player.setMediaItem(item); player.prepare()) but don't call player.play() yet. When the user clicks "Play", the media is already buffered and playback starts in under 100ms. The pattern:
DetailFragment.onResume() → playerManager.preload(videoUrl)
PlaybackFragment.onStart() → playerManager.play()
This eliminates the black screen + spinner that users see when ExoPlayer is created fresh on every playback start.

Q14. How do you handle memory pressure when your TV app has a BrowseFragment with 8 rows of 20 cards each, each card loading a thumbnail image? Amazon

160 images in memory at once is dangerous on 1 GB RAM TVs. Mitigations:
1. Glide handles off-screen recycling: Leanback's HorizontalGridView recycles card views — Glide's onUnbindViewHolder() calls Glide.with(ctx).clear(imageView), releasing the bitmap from memory.
2. Limit image resolution: card thumbnails don't need full resolution. Override with .override(320, 180) — at that size, all 160 cards use ~160 × 320 × 180 × 4 bytes ≈ 36 MB — manageable.
3. Load backdrop only for focused card: full-resolution backdrop images (~2 MB each) are only loaded when the card is focused (using setOnItemViewSelectedListener). Cancel the previous load on focus change.
4. Trim cache on onTrimMemory(TRIM_MEMORY_RUNNING_LOW): call Glide.get(context).trimMemory(level).

DRM, 4K & Security

Q15. A premium 4K content item plays fine on a Pixel TV but shows only 480p on a budget TV. Explain why and how your app should handle this gracefully. Sony LIV Hard

The root cause is Widevine security level. The Pixel TV has Widevine L1 (hardware TEE); the budget TV has L3 (software DRM). Content owners (studios) enforce that L3 devices can only receive streams up to 480p/720p — enforced in the DRM license returned by the license server, not in ExoPlayer. ExoPlayer's TrackSelector automatically caps resolution to what the license permits.

Handle gracefully:
1. Check security level at startup: MediaDrm.getPropertyString("securityLevel") → "L1" or "L3".
2. If L3, show a banner: "Your TV supports up to HD quality. Upgrade your TV for 4K."
3. Never show a generic error — the user sees a confusing playback failure instead of an explanation.
4. Tag the device's DRM level in your analytics so you can measure how many users are affected.

Q16. How does HDMI-CEC affect your Android TV app and what should you handle? YouTube

HDMI-CEC (Consumer Electronics Control) lets the TV remote control your Android TV app even if the user is using the TV manufacturer's remote, not a dedicated Android TV remote. Key behaviors to handle:
Play/Pause from TV remote: routed as KEYCODE_MEDIA_PLAY_PAUSE to your MediaSession. If you've set up MediaSession correctly, ExoPlayer handles this automatically.
TV power off: triggers Activity.onStop(). Pause playback, save position.
Input switch: when user switches HDMI input, your app goes to background. Pause + save position. Resume on return.
Volume control: HDMI-CEC volume is handled by the TV OS — don't intercept KEYCODE_VOLUME_UP/DOWN in a TV app. Let the system handle it.

System Design

Q17. Design the Android TV client architecture for a live cricket match stream on Hotstar. Include: latency targets, bitrate selection, DRM, scoreboard overlay, and multi-audio tracks. System Design Hard

Streaming: LL-HLS (Low-Latency HLS) for <3s latency. ExoPlayer with DefaultHlsExtractorFactory. Target live offset: 3–5s. Backoff on 404 segment (segment not ready yet on live edge).
Bitrate/ABR: EWMA bandwidth estimation (fast 3s + slow 8s, take min). TV on WiFi typically has 50–100 Mbps — default to highest quality tier. Allow manual quality override in settings overlay (D-pad accessible during playback).
DRM: Widevine. Check L1 vs L3 at startup. L1 → offer 4K/1080p. L3 → cap at 720p with explanation banner.
Scoreboard overlay: WebSocket connection to score API, updates every ball. Overlay is a custom FrameLayout above SurfaceView — score updates via withContext(Dispatchers.Main). Overlay visibility toggled by D-pad UP from transport controls.
Multi-audio: ExoPlayer TrackSelectionParameters to switch between Hindi/English commentary. Present as D-pad accessible list in settings overlay.
Resilience: reconnect with exponential backoff on network drop. Resume from last live edge position, not from stream start. Show "Reconnecting..." overlay without destroying the SurfaceView.

Q18. How would you design the "Continue Watching" feature across TV, phone, and web for a streaming service? What are the sync challenges? System Design

Data model: { userId, contentId, positionMs, totalDurationMs, deviceType, updatedAt } stored server-side.
Sync strategy:
— On TV: save position to Room DB every 10s during playback + on pause/stop. Sync to server via WorkManager (WiFi-only, batched).
— On resume: fetch latest position from server (network-first, Room fallback). If local position is newer (user was offline), use local.
Conflict resolution: last-write-wins using updatedAt timestamp. If two devices have the same updatedAt, prefer the larger positionMs.
TV-specific: TV users often stop watching mid-episode (put the TV on sleep). Must save position on Activity.onStop() synchronously (not just via WorkManager) — use a blocking coroutine with runBlocking with a 2s timeout so the position is not lost.
Row ordering: sort by updatedAt DESC, show max 20 items, exclude fully-watched (position > 95% of total).

Q19. How would you implement a parental controls PIN screen on Android TV that blocks access to 18+ content? Sony LIV

PIN entry: use GuidedStepSupportFragment with a numeric PIN action (4 digits). Store the PIN hash (SHA-256 + salt) in Android EncryptedSharedPreferences using Jetpack Security — never plain text.
Lock flow: when user navigates to 18+ content, check if PIN is set. If yes, launch PIN entry fragment as a modal over the detail screen. On correct PIN, store an in-memory "unlocked until" timestamp (e.g. 30 minutes). Subsequent 18+ accesses within the window skip the PIN prompt.
D-pad PIN entry: each digit is a separate GuidedAction with a number picker, or use a custom numpad layout with 10 digit buttons (0–9) navigated by D-pad. Show dots (●) not digits for entered characters.
Brute-force protection: after 3 wrong attempts, lock out for 60 seconds. Store attempt count in EncryptedSharedPreferences.
Session reset: clear in-memory unlock on Application.onTrimMemory(TRIM_MEMORY_UI_HIDDEN) — when app goes to background.

Q20. Your TV app's startup time is 4 seconds on a budget TV. Walk through how you'd diagnose and fix it. Hotstar Hard

Measure first: use adb shell am start -S -W com.hotstar/.MainActivity to get ThisTime (Activity creation) and TotalTime (first frame). Use Android Studio's CPU profiler with "Start recording on app launch" to trace startup.
Common bottlenecks:
1. Application.onCreate() doing too much: SDK initializations (Analytics, Crashlytics, Firebase) on main thread. Fix: initialize lazily or move to background thread with App Startup library.
2. Synchronous network call on startup: fetching home screen data before showing UI. Fix: show a skeleton/placeholder immediately, load data asynchronously.
3. Heavy Leanback row inflation: inflating all 8 rows at once. Fix: load 2 rows immediately, load remaining rows in postDelayed(500ms) after first frame is visible.
4. Glide not pre-warmed: first image loads slowly as Glide initializes its thread pool. Fix: initialize Glide in Application.onCreate().
5. Cold ExoPlayer init: codec enumeration takes 200–400ms on budget hardware. Fix: initialize ExoPlayer in a background coroutine during splash or skeleton phase.
Target: 1.5s cold start. Show first meaningful content (at least 2 rows) within 2s.

Interview Strategy — TV Questions
TV interviews at Hotstar/Sony LIV always anchor on three themes: (1) Focus management — they will describe a broken navigation scenario and ask you to debug it. Know focus traps, focus restoration, D-pad key events cold. (2) DRM on budget hardware — India-specific. Know Widevine L1 vs L3 and why budget TVs cap at 480p. (3) Performance on low-end devices — 1 GB RAM TVs are common in India. Know how to limit memory, initialize ExoPlayer lazily, and diagnose ANRs. Mention device-specific testing (Vu, Thomson, iFFALCON) — it signals real-world TV experience.
