The RenderThread

Android's dedicated GPU thread — DisplayList replay, RenderNode tree, GPU command submission, UI thread sync, frame pipelining, and RenderThread animations

1. Why a Separate Thread?

Before Android 5.0, all rendering happened on the UI thread. If the UI thread was busy (GC, layout, network on main thread), frames were dropped. The RenderThread (introduced in Android 5.0) decouples GPU work from the UI thread:

  • UI Thread: Records drawing commands into DisplayLists (fast — just appending to a list)
  • RenderThread: Replays DisplayLists on the GPU, submits the frame to SurfaceFlinger
  • They run in parallel — while RenderThread renders Frame N, UI Thread prepares Frame N+1

2. The Sync Phase

Between UI Thread recording and RenderThread rendering, there's a sync phase where the RenderThread copies the latest DisplayList data from the UI Thread:

// Sync phase (simplified from AOSP hwui/):
// 1. UI Thread finishes recording DisplayLists
// 2. UI Thread signals RenderThread: "frame ready"
// 3. RenderThread acquires a lock
// 4. RenderThread copies/swaps the DisplayList data from UI Thread's RenderNodes
//    - This is a shallow copy — display list commands reference shared GPU resources
//    - Render properties (translation, alpha, etc.) are copied by value
// 5. RenderThread releases the lock
// 6. UI Thread is now FREE to start the next frame
// 7. RenderThread processes the DisplayList → GPU commands

// The sync is typically < 1ms.
// If RenderThread is still busy with the previous frame when UI Thread
// finishes recording, the UI Thread BLOCKS until sync can happen.
// This is visible in Systrace as "DrawFrame" on the UI Thread.
Staff-Level Insight — Sync Stalls

The most common RenderThread-related jank: the UI thread blocks during sync because the RenderThread is still processing the previous frame. In Systrace, this appears as a long "DrawFrame" section on the UI thread. Causes: (1) GPU-heavy content (complex shaders, large bitmaps), (2) too many RenderNodes (deep view hierarchy), (3) overdraw (drawing the same pixel multiple times). The fix is usually to reduce GPU workload: flatten the hierarchy, reduce overdraw, use hardware layers for complex static content, or simplify custom drawing.

3. DisplayList and RenderNode Tree

// Each View has a RenderNode. The RenderNode tree mirrors the View tree:
// DecorView.renderNode
//   ├── ActionBar.renderNode
//   ├── ContentFrameLayout.renderNode
//   │   ├── LinearLayout.renderNode
//   │   │   ├── TextView.renderNode (DisplayList: drawText command)
//   │   │   └── ImageView.renderNode (DisplayList: drawBitmap command)
//   │   └── Button.renderNode (DisplayList: drawRoundRect + drawText)
//   └── NavigationBar.renderNode

// A DisplayList contains:
// - Drawing commands (drawRect, drawText, drawBitmap, drawRenderNode, etc.)
// - Clip operations (clipRect, clipPath)
// - Save/restore operations
// - References to child RenderNodes (drawRenderNode)

// When a child View is invalidated:
// 1. Only that View's DisplayList is re-recorded
// 2. Parent's DisplayList still has "drawRenderNode(child)" — no change needed
// 3. RenderThread replays from root, but child's updated DisplayList is used
// This is why invalidate() is cheap — only the dirty view re-records

4. RenderThread Animations

// Some animations run ENTIRELY on the RenderThread, bypassing the UI Thread:

// 1. RippleDrawable — the touch ripple effect
//    Runs on RenderThread using a hardware-accelerated shader
//    UI Thread is not involved after the initial touch event

// 2. CircularReveal animation
//    Uses a clip path that animates on RenderThread

// 3. ViewPropertyAnimator with hardware layer
view.animate()
    .translationX(100f)
    .alpha(0.5f)
    .setDuration(300)
    .withLayer() // Creates a hardware layer for the duration
    .start()
// The animation updates RenderNode properties directly on RenderThread
// UI Thread is only notified on completion

// 4. Window animations (activity transitions)
//    Managed by WindowManager, animated on RenderThread/SurfaceFlinger
Staff-Level Insight — Hardware Layers

A hardware layer (View.LAYER_TYPE_HARDWARE) renders the view's DisplayList into an off-screen GPU texture. Subsequent frames reuse the texture instead of re-rendering the DisplayList. This is beneficial when: (1) the view's content doesn't change but its render properties do (e.g., alpha animation), (2) the view is complex and expensive to re-render. The cost: GPU memory for the texture, and the texture must be re-created when the view's content changes. Always remove the layer after the animation: .withEndAction { view.setLayerType(LAYER_TYPE_NONE, null) }. Leaving hardware layers on static views wastes GPU memory.

5. Frame Completion — Fence Signaling

// After RenderThread finishes GPU commands:
// 1. GPU commands are submitted to the GPU driver
// 2. A GPU fence is created — signals when the GPU actually finishes
// 3. The buffer (with the rendered frame) is queued to the BufferQueue
// 4. SurfaceFlinger is notified that a new buffer is available
// 5. SurfaceFlinger waits for the GPU fence to signal
// 6. SurfaceFlinger composites the buffer with other layers
// 7. The composited frame is sent to the display

// The GPU fence is critical:
// - It allows the buffer to be queued BEFORE the GPU finishes
// - SurfaceFlinger can start planning composition while GPU is still rendering
// - The fence ensures SurfaceFlinger doesn't read incomplete data
// - This is how the pipeline stays full without synchronization stalls
