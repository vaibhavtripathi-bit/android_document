01 — Gradle Build System Internals

The three-phase lifecycle, task DAG, AGP internals, dependency resolution, daemon mechanics, incremental builds, build cache, configuration cache — everything an Android Architect must know about how Gradle actually works under the hood

1. Gradle Build Lifecycle — The Three Phases

Every Gradle invocation — whether ./gradlew assembleDebug or ./gradlew help — passes through exactly three phases, in strict order. Understanding these phases is the foundation of every build optimization and the root cause of most build performance problems.

Phase 1: Initialization

Gradle evaluates settings.gradle.kts to determine which projects participate in the build. For each included project, Gradle creates a Project instance (but does not evaluate its build script yet).

// settings.gradle.kts
rootProject.name = "MyApp"

include(":app")
include(":core:data")
include(":core:domain")
include(":core:ui")
include(":feature:home")
include(":feature:search")
include(":feature:profile")

Key internals during initialization:

  • Settings object is created — this is the receiver for settings.gradle.kts
  • includeBuild() calls register composite builds (evaluated in their own Gradle invocation)
  • The gradle.properties file is loaded (JVM args, feature flags, project properties)
  • The Gradle Daemon is selected or started (more in Section 6)

Phase 2: Configuration

Gradle evaluates every build.gradle.kts file for every included project. This is where plugins are applied, dependencies are declared, tasks are registered, and the task graph (a directed acyclic graph) is constructed.

Critical Performance Fact
ALL modules are configured on EVERY build, even if you're only building :app. Running ./gradlew :app:assembleDebug still configures :feature:search, :core:ui, and every other module. This is why expensive operations in build scripts (network calls, file I/O, heavy computation) slow down every single build, including ./gradlew help.

Phase 3: Execution

Gradle walks the task DAG and executes only the tasks required for the requested target, in topological order. Tasks whose inputs haven't changed are marked UP-TO-DATE and skipped. Tasks with cached outputs are marked FROM-CACHE and restored.

Staff Insight — Configuration Phase is the Hidden Tax
In a 200-module project, the configuration phase alone can take 15–30 seconds. This runs on every build — even ./gradlew tasks. The two most impactful optimizations are configuration avoidance (lazy task registration) and configuration cache (serializing the task graph). An L6 engineer should be able to explain why both exist and when each helps.

2. Gradle's Core Model — Project, Task, Configuration

Project

A Project represents a single module in the build. Each build.gradle.kts file is evaluated in the scope of its Project instance. Projects form a hierarchy: the root project contains subprojects.

// In any build.gradle.kts, `this` is the Project
println(project.name)       // e.g., "app"
println(project.path)       // e.g., ":app" or ":core:data"
println(project.buildDir)   // e.g., /project/app/build

Task

A Task is the fundamental unit of work. Every build operation — compiling Kotlin, merging resources, running dex, packaging the APK — is a task. Tasks declare:

  • Inputs — files, properties, or other values the task reads
  • Outputs — files or directories the task produces
  • Dependencies — other tasks that must run first
  • Actions — the actual work (lambdas or @TaskAction methods)

Task Graph (DAG)

Tasks declare dependencies on other tasks, forming a directed acyclic graph. Gradle executes tasks in topological order — a task only runs after all its dependencies have completed.

  assembleDebug
  ├── packageDebug
  │   ├── dexBuilderDebug
  │   │   └── compileDebugKotlin
  │   │       └── compileDebugAidl
  │   ├── mergeDebugResources
  │   │   └── processDebugResources
  │   ├── mergeDebugAssets
  │   └── mergeDebugJniLibFolders
  └── validateSigningDebug

  Execution order (topological sort):
  compileDebugAidl → compileDebugKotlin → dexBuilderDebug ─┐
  processDebugResources → mergeDebugResources ─────────────┤
  mergeDebugAssets ─────────────────────────────────────────┼→ packageDebug → assembleDebug
  mergeDebugJniLibFolders ─────────────────────────────────┤
  validateSigningDebug ────────────────────────────────────┘

Lazy Task Registration — Configuration Avoidance

One of the most important performance concepts in modern Gradle:

// EAGER — task is created immediately during configuration
// The action lambda is configured NOW, even if the task never runs
tasks.create("myTask") {
    doLast { println("Running") }
}

// LAZY — task is only created when needed (another task depends on it,
// or it matches the requested target)
tasks.register("myTask") {
    doLast { println("Running") }
}

// Lazy reference to an existing task — does NOT trigger creation
val compileTask = tasks.named("compileDebugKotlin")

The Provider / Property API extends this laziness to values:

abstract class MyTask : DefaultTask() {
    @get:Input
    abstract val message: Property<String>

    @get:OutputFile
    abstract val outputFile: RegularFileProperty

    @TaskAction
    fun execute() {
        // Values resolved HERE (execution time), not during configuration
        outputFile.get().asFile.writeText(message.get())
    }
}

// Registration — no values resolved yet
tasks.register<MyTask>("generateFile") {
    message.set(provider { computeExpensiveValue() }) // deferred
    outputFile.set(layout.buildDirectory.file("output.txt"))
}

Configuration — Named Dependency Buckets

A Configuration in Gradle is a named set of dependencies with specific visibility and scope semantics:

Configuration Compile Classpath Runtime Classpath Transitive to Consumers Use Case
implementation Yes Yes No Default — internal dependency
api Yes Yes Yes Dependency is part of your public API
compileOnly Yes No No Annotation processors, compile-time checks
runtimeOnly No Yes No SLF4J bindings, database drivers
testImplementation Test only Test only No JUnit, MockK, Turbine
ksp / kapt Processor classpath No No Annotation/symbol processing

3. build.gradle.kts vs build.gradle — Kotlin DSL vs Groovy DSL

Gradle supports two scripting languages for build files. Google recommends Kotlin DSL for new Android projects (default in Android Studio since 2023).

Key Differences

Aspect Groovy DSL (.gradle) Kotlin DSL (.gradle.kts)
Type safety None — dynamic dispatch Full — compile-time checked
IDE support Limited auto-completion Full auto-completion, navigation, refactoring
String quoting Single or double quotes Double quotes only
Method calls implementation 'lib' (no parens) implementation("lib") (parens required)
Property assignment minSdk 24 minSdk = 24
First compilation Faster (interpreted) Slower (compiled to bytecode, then cached)
Subsequent builds Similar Similar (script classes cached)
Extension access android { ... } (magic) android { ... } (type-safe accessor generated)

Plugin Application — Legacy vs Modern

// LEGACY: buildscript block — classpath declared in root, applied in modules
// root build.gradle.kts
buildscript {
    repositories { google(); mavenCentral() }
    dependencies {
        classpath("com.android.tools.build:gradle:8.5.0")
        classpath("org.jetbrains.kotlin:kotlin-gradle-plugin:2.0.0")
    }
}

// module build.gradle.kts
apply(plugin = "com.android.application")

// ─────────────────────────────────────────────────────────

// MODERN: plugins block — resolved from Gradle Plugin Portal / repositories
// root build.gradle.kts
plugins {
    id("com.android.application") version "8.5.0" apply false
    id("org.jetbrains.kotlin.android") version "2.0.0" apply false
}

// module build.gradle.kts
plugins {
    id("com.android.application")
    id("org.jetbrains.kotlin.android")
}

The plugins { } block is preferred because it enables Gradle to resolve plugins from the pluginManagement { } block in settings.gradle.kts, supports version catalogs, and allows Gradle to optimize plugin loading.

Migration Cheat Sheet — Groovy → Kotlin DSL

// Groovy                          →  Kotlin DSL
// ──────────────────────────────────────────────────────
apply plugin: 'com.android.app'   →  plugins { id("com.android.application") }
implementation 'com.lib:v1.0'     →  implementation("com.lib:v1.0")
minSdkVersion 24                  →  minSdk = 24
buildConfigField "String", ...    →  buildConfigField("String", ...)
testInstrumentationRunner "..."   →  testInstrumentationRunner = "..."
signingConfigs { debug { ... } }  →  signingConfigs { getByName("debug") { ... } }
                                      // or: named("debug") { ... }
def myVar = "value"               →  val myVar = "value"
ext.myProp = "value"              →  extra["myProp"] = "value"

4. Android Gradle Plugin (AGP) Internals

AGP is the plugin that makes Gradle understand Android. It is applied via com.android.application (for app modules) or com.android.library (for library modules), and it registers the vast majority of tasks in an Android build.

What AGP Does During Configuration

  1. Creates the android { } extension (ApplicationExtension or LibraryExtension)
  2. Reads build types, product flavors, and dimensions → computes the variant matrix
  3. For each variant, registers dozens of tasks: compile, merge resources, AIDL, dex, package, lint, test
  4. Configures dependency resolution for each variant's compile and runtime classpaths
  5. Wires up the Kotlin compiler plugin (if Kotlin is applied)

Build Variants — The Cartesian Product

android {
    buildTypes {
        getByName("debug") { ... }
        getByName("release") { ... }
    }
    flavorDimensions += "tier"
    productFlavors {
        create("free") { dimension = "tier" }
        create("paid") { dimension = "tier" }
    }
}

// Resulting variants (buildType × flavor):
// freeDebug, freeRelease, paidDebug, paidRelease
// Each variant gets its OWN set of tasks:
//   compileFreeDebugKotlin, compileFreeReleaseKotlin, ...
//   mergePaidDebugResources, mergePaidReleaseResources, ...
Staff Insight — Variant Explosion
AGP registers ~80 tasks per variant. With 2 build types × 3 flavors × 2 dimensions, you get 12 variants → ~960 tasks per module. In a 50-module project, that's ~48,000 tasks registered during configuration. This is why minimizing unused flavors and dimensions is one of the highest-impact build optimizations. Use variantFilter (legacy) or the Variant API to disable variants you don't need.

The Variant API (AGP 7+)

The modern, type-safe way to manipulate variants and their artifacts:

androidComponents {
    // Disable variants you don't need
    beforeVariants(selector().withBuildType("release").withFlavor("tier" to "free")) {
        it.enable = false
    }

    // Modify variant properties
    onVariants(selector().withBuildType("debug")) { variant ->
        variant.applicationId.set("com.example.debug")
        variant.manifestPlaceholders.put("appLabel", "MyApp Debug")
    }

    // Transform artifacts
    onVariants { variant ->
        val mergedManifest = variant.artifacts.get(SingleArtifact.MERGED_MANIFEST)
        // Use mergedManifest as input to a custom task
    }
}

5. Dependency Resolution — How Gradle Picks Versions

Dependency resolution happens during the Configuration phase. When Gradle resolves a configuration (e.g., debugCompileClasspath), it builds a dependency graph, fetches metadata from repositories, and selects a single version for each module.

The Resolution Algorithm

  1. Gradle collects all declared dependencies (direct + transitive) into a graph
  2. For each module that appears at multiple versions, Gradle applies conflict resolution
  3. Default strategy: highest version wins — if :app requests OkHttp 4.9 and :core:data requests OkHttp 4.12, Gradle selects 4.12 for the entire build
  4. The resolved graph is used to construct compile and runtime classpaths

Inspecting the Dependency Graph

// Full dependency tree for a specific configuration
./gradlew :app:dependencies --configuration debugRuntimeClasspath

// Dependency insight — why was a specific version selected?
./gradlew :app:dependencyInsight --dependency okhttp --configuration debugRuntimeClasspath

// Output shows:
// com.squareup.okhttp3:okhttp:4.12.0
//   Variant: ...
//   Selection reasons:
//     - By conflict resolution: between versions 4.9.3 and 4.12.0

api vs implementation — The Classpath Boundary

  :feature:home                :core:data                  :core:network
  ┌─────────────┐             ┌─────────────┐             ┌─────────────┐
  │             │ ──impl──→   │             │ ──api──→    │             │
  │             │             │             │             │  OkHttp     │
  │             │             │  Retrofit   │ ──impl──→   │  (exposed)  │
  │             │             │  (hidden)   │             │             │
  └─────────────┘             └─────────────┘             └─────────────┘

  :feature:home sees:
    ✓ :core:data (direct dependency)
    ✓ :core:network (transitive via api)
    ✓ OkHttp (transitive via api from :core:network)
    ✗ Retrofit (hidden — :core:data used implementation)

  Why this matters for builds:
    If OkHttp's API changes → :feature:home MUST recompile (it's on the compile classpath)
    If Retrofit's API changes → :feature:home does NOT recompile (not on its compile classpath)

Resolution Strategies

configurations.all {
    resolutionStrategy {
        // Force a specific version globally
        force("com.squareup.okhttp3:okhttp:4.12.0")

        // Fail the build if there's a version conflict (instead of silently upgrading)
        failOnVersionConflict()

        // Prefer project modules over external artifacts
        preferProjectModules()

        // Cache dynamic versions for 10 minutes (default: 24 hours)
        cacheDynamicVersionsFor(10, "minutes")
    }
}

// Per-dependency version constraints
dependencies {
    implementation("com.squareup.okhttp3:okhttp") {
        version {
            strictly("4.12.0")  // MUST be exactly this version — fails if conflict
            prefer("4.12.0")    // prefer this, but allow upgrade
            require("4.10.+")   // any 4.10.x
        }
    }

    // Exclude a transitive dependency
    implementation("com.example:library:1.0") {
        exclude(group = "org.unwanted", module = "transitive-dep")
    }
}

// Module substitution — replace published artifact with local project
configurations.all {
    resolutionStrategy.dependencySubstitution {
        substitute(module("com.mycompany:shared-lib"))
            .using(project(":shared-lib"))
            .because("Developing locally")
    }
}
Watch Out — Version Catalogs Don't Enforce Versions
A version catalog (libs.versions.toml) is a declaration convenience, not an enforcement mechanism. If module A uses libs.okhttp (4.12.0) and module B hard-codes "com.squareup.okhttp3:okhttp:4.9.0", Gradle still resolves to 4.12.0 via conflict resolution — but silently. Use failOnVersionConflict() or a dependency locking file to catch these mismatches.

6. The Gradle Daemon

The Gradle Daemon is a long-running JVM process that persists between builds. It is the single biggest reason why the second build is dramatically faster than the first.

Cold vs Warm Builds

Aspect Cold (No Daemon) Warm (Daemon Running)
JVM startup ~2–5 seconds 0 (already running)
Class loading Load all Gradle + plugin classes Already loaded in memory
JIT compilation None — interpreted bytecode Hot paths already JIT-compiled
File system cache Cold OS page cache Warm — recently accessed files in memory
Typical overhead +5–15 seconds ~0

Daemon Lifecycle

// Daemon starts automatically on first build (default behavior)
./gradlew assembleDebug          // starts daemon if not running

// Explicitly manage the daemon
./gradlew --stop                 // kill all daemons for this Gradle version
./gradlew --no-daemon            // run this build without a daemon
./gradlew --status               // list running daemons

// Daemon memory configuration — gradle.properties
org.gradle.jvmargs=-Xmx4g -XX:+UseParallelGC -XX:MaxMetaspaceSize=512m

// Daemon idle timeout (default: 3 hours)
org.gradle.daemon.idletimeout=10800000

Daemon Compatibility Rules

A daemon can only be reused if its JVM arguments match the requested build. If gradle.properties changes org.gradle.jvmargs, a new daemon is started (the old one idles and eventually expires). This means:

  • Different branches with different JVM args → multiple daemons → higher memory usage
  • Changing -Xmx frequently → daemon churn → slower builds
  • Standardize JVM args across the team to maximize daemon reuse
Staff Insight — Daemon Strategy on CI
On CI with ephemeral containers (each build gets a fresh VM), the daemon provides zero benefit — it starts, runs one build, and is discarded. In this case, --no-daemon with a higher heap (-Xmx6g) can actually be faster because it avoids the daemon startup overhead and inter-process communication. However, if your CI uses persistent agents (reused across builds), keep the daemon — the warm JVM and JIT compilation provide significant speedups on subsequent builds.

7. Incremental Builds

Incremental builds are the single most impactful build performance feature. Instead of re-executing every task, Gradle tracks task inputs and outputs and skips tasks whose inputs haven't changed.

How Input/Output Tracking Works

// Gradle uses annotations to declare what a task reads and writes
abstract class ProcessDataTask : DefaultTask() {

    @get:InputFile
    abstract val inputData: RegularFileProperty      // tracked input

    @get:Input
    abstract val configValue: Property<String>       // tracked input (scalar)

    @get:OutputDirectory
    abstract val outputDir: DirectoryProperty         // tracked output

    @TaskAction
    fun process() {
        val data = inputData.get().asFile.readText()
        outputDir.get().asFile.resolve("result.json").writeText(
            transform(data, configValue.get())
        )
    }
}

// Build 1: Task executes, Gradle snapshots inputs → stores hash
// Build 2: Gradle re-hashes inputs → same hash → UP-TO-DATE (skipped)
// Build 3: inputData file changed → hash differs → task re-executes

Task Outcome States

Outcome Meaning Performance Impact
EXECUTED Task ran its action Full cost
UP-TO-DATE Inputs unchanged since last execution Near zero (hash check only)
FROM-CACHE Outputs restored from build cache I/O cost of restoring files
SKIPPED Task was disabled or had an onlyIf that returned false Zero
NO-SOURCE Task's input files collection was empty Zero

Kotlin Incremental Compilation

The Kotlin compiler has its own incremental compilation layer on top of Gradle's task-level incrementality:

  • ABI change detection: Kotlin tracks the Application Binary Interface (public signatures) of each file. If a file's ABI changes (public function signature, class hierarchy), all files that depend on it are recompiled. If only the implementation changes (function body), only the changed file is recompiled.
  • Classpath snapshot: Kotlin maintains a snapshot of the classpath. If a dependency's ABI changes, affected files are recompiled.
  • kotlin.incremental=true (default since Kotlin 1.4) — enables incremental compilation
  • kotlin.incremental.usePreciseJavaTracking=true — finer-grained tracking of Java→Kotlin dependencies
Staff Insight — The Incremental Build Killer
Incremental builds are fragile. A single task that reads all source files as input (common with annotation processors using kapt) forces a full recompile whenever any file changes. This is why migrating from kapt to ksp is a high-priority build optimization — KSP supports true incremental processing, while most kapt processors do not. In a 200-module project, a non-incremental annotation processor can add 30+ seconds to every build.

8. Build Cache

While incremental builds avoid re-executing tasks when inputs haven't changed locally, the build cache goes further — it stores task outputs keyed by input hash, allowing outputs to be reused across clean builds, branch switches, and even different machines.

Local Build Cache

// Enable in gradle.properties
org.gradle.caching=true

// Local cache location: ~/.gradle/caches/build-cache-1/
// Cache entries are keyed by:
//   hash(task class + task inputs + Gradle version + JVM version)

// Example: after a clean build, switch branches, switch back
// → all tasks are FROM-CACHE instead of EXECUTED

Remote Build Cache

// settings.gradle.kts
buildCache {
    local {
        isEnabled = true
        directory = file("${rootDir}/.gradle/build-cache")
    }
    remote<HttpBuildCache> {
        url = uri("https://cache.mycompany.com/cache/")
        isPush = System.getenv("CI") == "true"  // only CI pushes
        credentials {
            username = System.getenv("CACHE_USER") ?: ""
            password = System.getenv("CACHE_PASS") ?: ""
        }
    }
}

Cache Key Composition

  Cache Key = hash of:
  ┌─────────────────────────────────────────────────────┐
  │  Task implementation                                │
  │  ├── Task class name + classpath hash               │
  │  ├── Task action lambdas (bytecode hash)            │
  │  │                                                   │
  │  Task inputs                                        │
  │  ├── Input files (content hash, not path)           │
  │  ├── Input properties (serialized values)           │
  │  ├── Classpath inputs (normalized jar hash)         │
  │  │                                                   │
  │  Build environment                                  │
  │  ├── Gradle version                                 │
  │  ├── JVM version + vendor                           │
  │  └── Operating system (for native tasks)            │
  └─────────────────────────────────────────────────────┘

  Cache miss reasons (check with --build-cache --info):
  • Input file changed → different content hash
  • Task class changed → plugin upgrade
  • Non-cacheable task → @CacheableTask not declared
  • Absolute paths in inputs → breaks relocatability
  • Cache evicted → LRU eviction when cache is full

Making Tasks Cacheable

// A task must opt in to caching with @CacheableTask
@CacheableTask
abstract class GenerateApiClient : DefaultTask() {

    // Inputs MUST use @PathSensitive to enable cache relocatability
    @get:InputFile
    @get:PathSensitive(PathSensitivity.RELATIVE)  // only relative path matters
    abstract val specFile: RegularFileProperty

    @get:Input
    abstract val apiVersion: Property<String>

    @get:OutputDirectory
    abstract val outputDir: DirectoryProperty

    @TaskAction
    fun generate() { /* ... */ }
}

// Without @PathSensitive, the ABSOLUTE path is part of the cache key
// → /Users/alice/project/spec.yaml ≠ /Users/bob/project/spec.yaml
// → cache miss on every machine, defeating the remote cache
Staff Insight — Remote Cache ROI
A remote build cache is transformative for large teams. When CI builds push cached outputs, a developer doing a clean build (after branch switch, git clean, or fresh checkout) can reuse 70–90% of task outputs from CI. A 10-minute clean build becomes 2 minutes. The ROI is enormous: for a team of 50 engineers each doing 10 builds/day, saving 8 minutes per clean build (assume 30% are clean) translates to ~200 engineer-hours saved per month. This is a staff-level initiative worth driving.

9. Configuration Cache

The configuration cache (stable since Gradle 8.1) is the next frontier of build performance. It caches the entire task graph from the Configuration phase, allowing subsequent builds to skip configuration entirely.

How It Works

  First build (cache miss):
  ┌──────────────┐     ┌──────────────┐     ┌──────────────┐
  │ Initialization│ ──→ │Configuration │ ──→ │  Execution   │
  │   (~1s)      │     │   (~15s)     │     │   (~60s)     │
  └──────────────┘     └──────┬───────┘     └──────────────┘
                              │
                              ▼ serialize task graph
                     ┌─────────────────┐
                     │ Configuration   │
                     │ Cache (.gradle/ │
                     │ configuration-  │
                     │ cache/)         │
                     └─────────────────┘

  Subsequent build (cache hit):
  ┌──────────────┐     ┌──────────────┐     ┌──────────────┐
  │ Initialization│ ──→ │  Deserialize │ ──→ │  Execution   │
  │   (~1s)      │     │   (~0.5s)    │     │   (~60s)     │
  └──────────────┘     └──────────────┘     └──────────────┘
                        (skips ALL build
                         script evaluation)

Enabling Configuration Cache

// gradle.properties
org.gradle.configuration-cache=true

// Optional: max entries (default: varies by Gradle version)
org.gradle.configuration-cache.max-problems=512

// Run with reporting to see problems
./gradlew assembleDebug --configuration-cache

// First run: "Calculating task graph as no cached configuration is available"
// Second run: "Reusing configuration cache"

Configuration Cache Restrictions

The configuration cache imposes strict rules on what build scripts can do. These restrictions exist because the task graph must be serializable and reproducible:

Forbidden at Execution Time Why Fix
task.project Project is not serializable Pass needed values as task Property inputs
System.getenv() without declaring Env vars are build inputs — must be tracked Use providers.environmentVariable("KEY")
File("...").readText() in task action File reads are undeclared inputs Declare as @InputFile
gradle.taskGraph.whenReady Callback captures Project state Use Provider API for deferred values
Shared mutable state between tasks Tasks may run in parallel; state is not serialized Use BuildService for shared state

Common Migration Issues

// BROKEN: accessing Project at execution time
tasks.register("printVersion") {
    doLast {
        println(project.version)  // ERROR: cannot access Project
    }
}

// FIXED: capture the value during configuration as a Property
tasks.register("printVersion") {
    val ver = project.version.toString()  // captured during configuration
    doLast {
        println(ver)  // uses the captured value
    }
}

// BETTER: use Provider API
tasks.register("printVersion") {
    val ver = provider { project.version.toString() }
    doLast {
        println(ver.get())
    }
}

// BROKEN: reading environment variable without declaring it
tasks.register("deploy") {
    doLast {
        val token = System.getenv("DEPLOY_TOKEN")  // undeclared input
    }
}

// FIXED: use Gradle's provider
tasks.register("deploy") {
    val token = providers.environmentVariable("DEPLOY_TOKEN")
    doLast {
        println(token.get())
    }
}
Migration Reality Check
Enabling configuration cache in an existing large project is a multi-sprint effort. Third-party plugins (especially older Gradle plugins, some Hilt/Dagger processors, Firebase plugins) frequently violate configuration cache rules. Use --configuration-cache-problems=warn to identify all violations before enforcing. Expect 50–200 problems in a typical 100-module project. Fix your own build scripts first, then file issues against plugin authors.

10. Staff-Level Corner Cases & Advanced Patterns

allprojects / subprojects — The Configuration Coupling Anti-Pattern

// root build.gradle.kts — ANTI-PATTERN
allprojects {
    repositories {
        google()
        mavenCentral()
    }
}

subprojects {
    apply(plugin = "org.jetbrains.kotlin.android")

    tasks.withType<KotlinCompile>().configureEach {
        kotlinOptions.jvmTarget = "17"
    }
}

// Problems:
// 1. Couples ALL modules to the root build script — any change triggers full reconfiguration
// 2. Applies Kotlin Android plugin to ALL subprojects (even pure JVM modules)
// 3. Breaks configuration avoidance — tasks.withType<> eagerly realizes all matching tasks
// 4. Breaks configuration cache — cross-project access
//
// BETTER: Use convention plugins (covered in doc 06)

afterEvaluate — The Escape Hatch That Breaks Everything

// afterEvaluate runs AFTER the project's build script is fully evaluated
// Often used to access values set by plugins or other scripts
afterEvaluate {
    tasks.named("assembleDebug") {
        doLast { println("Build complete: ${android.defaultConfig.versionName}") }
    }
}

// Why it's problematic:
// 1. Ordering is fragile — multiple afterEvaluate blocks run in registration order
// 2. Breaks configuration cache (captures Project state)
// 3. Defeats lazy task configuration (forces task realization)
// 4. Makes build logic harder to reason about (deferred execution)
//
// BETTER: Use the Provider API or Variant API to defer value resolution

Task Ordering — dependsOn vs mustRunAfter vs shouldRunAfter

tasks.register("taskA") { doLast { println("A") } }
tasks.register("taskB") { doLast { println("B") } }
tasks.register("taskC") { doLast { println("C") } }

// dependsOn: HARD dependency — B will always run A first, even if you only request B
tasks.named("taskB") { dependsOn("taskA") }

// mustRunAfter: SOFT ordering — if BOTH A and C are in the execution graph,
// C runs after A. But requesting only C does NOT pull in A.
tasks.named("taskC") { mustRunAfter("taskA") }

// shouldRunAfter: OPTIONAL ordering — same as mustRunAfter but Gradle may
// ignore it to break circular dependencies
tasks.named("taskC") { shouldRunAfter("taskB") }

// finalizedBy: reverse dependency — if A runs, B ALWAYS runs after it
tasks.named("taskA") { finalizedBy("taskB") }

Composite Builds — Developing Plugins Alongside the Project

// settings.gradle.kts
pluginManagement {
    // Include a local plugin project — changes are picked up immediately
    includeBuild("build-logic")
}

// build-logic/build.gradle.kts
plugins {
    `kotlin-dsl`
}

dependencies {
    implementation("com.android.tools.build:gradle:8.5.0")
    implementation("org.jetbrains.kotlin:kotlin-gradle-plugin:2.0.0")
}

// build-logic/src/main/kotlin/my-convention.gradle.kts
// Convention plugin — applied to modules that need it
plugins {
    id("com.android.library")
    id("org.jetbrains.kotlin.android")
}

android {
    compileSdk = 35
    defaultConfig { minSdk = 24 }
    compileOptions {
        sourceCompatibility = JavaVersion.VERSION_17
        targetCompatibility = JavaVersion.VERSION_17
    }
}

// module build.gradle.kts — clean, minimal
plugins {
    id("my-convention")  // all shared config comes from the convention plugin
}

dependencies {
    implementation(project(":core:domain"))
}
Staff Insight — Composite Builds as the Foundation
Composite builds (includeBuild("build-logic")) are the recommended way to share build logic in multi-module projects. Unlike allprojects/subprojects, convention plugins in a composite build are type-safe, independently testable, and don't break configuration avoidance or configuration cache. At Google-scale, this pattern is non-negotiable — it's the only way to maintain build scripts across hundreds of modules without them diverging into inconsistent configurations.
