Build Performance

Profiling, configuration avoidance, task caching, incremental compilation, kapt vs KSP, daemon tuning, and CI optimization strategies for production-scale Android builds

1. Build Profiling — Measure Before You Optimize

The cardinal rule of build performance is never optimize without profiling first. The bottleneck is almost never where you expect it. Teams routinely blame compilation when the real culprit is configuration phase overhead, dependency resolution, or a single non-incremental annotation processor.

Profiling Tools

ToolCommand / SetupWhat It Shows
HTML Profile--profilePer-task timing in build/reports/profile/
Build Scan--scanConfiguration time, task execution, cache hits, dependency resolution, timeline
Gradle EnterpriseServer-hostedHistorical trends, build comparison, cache analytics, flaky test detection
Build AnalyzerAndroid StudioCritical path, annotation processor impact, always-run tasks
# Quick profile — generates local HTML report
./gradlew assembleDebug --profile

# Build Scan — detailed online report (requires accepting TOS)
./gradlew assembleDebug --scan

# Verbose logging — shows cache hit/miss reasons per task
./gradlew assembleDebug --info

# Dry run — verify task graph without executing anything
./gradlew assembleDebug --dry-run

Key Metrics to Track

  • Total build time — wall-clock from invocation to completion
  • Configuration phase time — time spent evaluating all build.gradle.kts files and building the task graph
  • Task execution time — time spent actually running tasks
  • Task outcomes — how many tasks were EXECUTED vs UP-TO-DATE vs FROM-CACHE
  • Dependency resolution time — time spent resolving and downloading artifacts
  • Cache hit rate — percentage of cacheable tasks served from build cache
Staff Insight
In a 60-module project, a common profile reveals: configuration phase takes 8 seconds, compilation takes 12 seconds, and kapt takes 25 seconds. The junior instinct is to optimize compilation. The staff engineer migrates kapt to KSP and cuts 20 seconds. Always let the profiler drive your optimization priorities.

2. Configuration Avoidance

The configuration phase evaluates every build.gradle.kts in the project and builds the complete task graph. This happens on every single build invocation — even ./gradlew help. In a 100-module project, this can take 10+ seconds before any real work begins.

Lazy Task Registration

The most impactful configuration avoidance technique is using tasks.register instead of tasks.create:

// BAD: eager creation — task object is instantiated during configuration
// even if this task is never executed
tasks.create("generateDocs") {
    doLast { /* ... */ }
}

// GOOD: lazy registration — task object is created only when needed
tasks.register("generateDocs") {
    doLast { /* ... */ }
}

With register, the task is only instantiated if it appears in the execution graph. For a project with hundreds of custom tasks, this difference is measurable.

Lazy Properties: Provider<T> and Property<T>

abstract class ProcessDataTask : DefaultTask() {
    // Lazy — value resolved only during execution phase
    @get:Input
    abstract val inputCount: Property<Int>

    @get:InputFile
    abstract val sourceFile: RegularFileProperty

    @get:OutputDirectory
    abstract val outputDir: DirectoryProperty

    @TaskAction
    fun process() {
        val count = inputCount.get()  // resolved NOW, not during configuration
        val src = sourceFile.get().asFile
        // ...
    }
}

tasks.register<ProcessDataTask>("processData") {
    inputCount.set(provider { computeExpensiveValue() })  // deferred
    sourceFile.set(layout.projectDirectory.file("data/input.json"))
    outputDir.set(layout.buildDirectory.dir("processed"))
}

Configuration-Phase Anti-Patterns

Avoid These
Every one of these forces unnecessary work during configuration, slowing down every build invocation.
Anti-PatternWhy It's BadFix
allprojects { } Forces configuration of every project, even those not in the task graph Use convention plugins applied per-module
subprojects { } Same problem as allprojects Convention plugins or plugins { } block
Resolving configurations at config time Triggers full dependency resolution before any task runs Resolve inside task actions using Provider
File I/O in build.gradle.kts Disk reads on every configuration Wrap in provider { } or move to task

// BAD: resolving dependencies during configuration

println("Found ${jarFiles.size} jars")  // runs on EVERY build

// GOOD: defer resolution to execution
tasks.register("listDeps") {
    val implConfig = configurations.named("runtimeClasspath")
    doLast {
        val jarFiles = implConfig.get().resolve()
        println("Found ${jarFiles.size} jars")
    }
}

3. Task Avoidance — UP-TO-DATE and FROM-CACHE

Gradle's incremental build model tracks every task's inputs and outputs. If inputs haven't changed since the last execution, the task is skipped entirely. Understanding task outcomes is essential for diagnosing slow builds.

Task Outcome Reference

OutcomeMeaningPerformance Impact
EXECUTEDTask ran normallyFull cost
UP-TO-DATEInputs unchanged since last local runNear-zero (input hash check only)
FROM-CACHEOutput restored from build cacheCache lookup + output copy
SKIPPEDTask disabled via onlyIf or conditionZero
NO-SOURCETask has @SkipWhenEmpty inputs and none existZero

Declaring Inputs and Outputs

@CacheableTask
abstract class GenerateApiClient : DefaultTask() {

    @get:Input
    abstract val apiVersion: Property<String>

    @get:InputFile
    @get:PathSensitive(PathSensitivity.RELATIVE)
    abstract val specFile: RegularFileProperty

    @get:OutputDirectory
    abstract val outputDir: DirectoryProperty

    @TaskAction
    fun generate() {
        val spec = specFile.get().asFile.readText()
        val outDir = outputDir.get().asFile
        // Generate client code from OpenAPI spec...
    }
}

Key annotations:

  • @Input / @InputFile / @InputFiles — what the task reads
  • @OutputFile / @OutputDirectory — what the task produces
  • @CacheableTask — opt this task into the build cache
  • @PathSensitive(PathSensitivity.RELATIVE) — tells Gradle that absolute paths don't matter for cache key computation, dramatically improving cross-machine cache hits

Incremental Task Inputs

@CacheableTask
abstract class CompileTemplates : DefaultTask() {

    @get:Incremental
    @get:InputFiles
    @get:PathSensitive(PathSensitivity.RELATIVE)
    abstract val templateFiles: ConfigurableFileCollection

    @get:OutputDirectory
    abstract val outputDir: DirectoryProperty

    @TaskAction
    fun compile(inputChanges: InputChanges) {
        // Only process files that actually changed
        inputChanges.getFileChanges(templateFiles).forEach { change ->
            when (change.changeType) {
                ChangeType.ADDED, ChangeType.MODIFIED -> compileTemplate(change.file)
                ChangeType.REMOVED -> removeOutput(change.file)
            }
        }
    }
}

With @Incremental + InputChanges, the task processes only the delta — not the full input set. For tasks processing hundreds of files, this turns a 30-second task into a sub-second one on typical changes.

4. Build Cache — Local and Remote

The build cache stores task outputs keyed by a hash of the task's class, inputs, and configuration. When a task with matching inputs is encountered, Gradle restores the cached outputs instead of re-executing.

Cache Architecture

Developer Machine A                    CI Server
┌──────────────────┐                  ┌──────────────────┐
│ ./gradlew build  │                  │ ./gradlew build  │
│                  │                  │                  │
│ 1. Check local   │                  │ 1. Execute tasks │
│    cache (~/.     │                  │ 2. Push outputs  │
│    gradle/caches/ │                  │    to remote     │
│    build-cache-1/)│                  │    cache         │
│ 2. Miss → check  │                  └────────┬─────────┘
│    remote cache  │                           │
│ 3. Hit → restore │    ┌──────────────┐       │
│    outputs       │◄───│ Remote Cache │◄──────┘
│ 4. Skip task     │    │ (HTTP/GE)    │
└──────────────────┘    └──────────────┘

Developer Machine B
┌──────────────────┐
│ Same task, same  │
│ inputs → remote  │───► Cache HIT → outputs restored
│ cache hit        │
└──────────────────┘

Cache Key = hash(task class + task inputs + Gradle version + JVM args)

Enabling the Build Cache

# gradle.properties
org.gradle.caching=true

Local cache is automatic once enabled. For remote cache, configure in settings.gradle.kts:

// settings.gradle.kts
buildCache {
    local {
        directory = File(rootDir, ".gradle/build-cache")
        removeUnusedEntriesAfterDays = 14
    }
    remote<HttpBuildCache> {
        url = uri("https://cache.mycompany.com/cache/")
        isPush = System.getenv("CI") != null  // only CI pushes
        credentials {
            username = System.getenv("CACHE_USER") ?: ""
            password = System.getenv("CACHE_PASS") ?: ""
        }
    }
}

Staff Insight
The biggest remote cache ROI comes from a simple workflow: CI builds populate the cache, developer builds consume it. A clean checkout on a developer machine that would normally take 8 minutes completes in 90 seconds because every compilation task hits the remote cache. This single optimization has more impact than all other build performance techniques combined for most teams.

5. Configuration Cache

While the build cache stores task outputs, the configuration cache stores the entire task graph from the configuration phase. On subsequent builds, Gradle deserializes the cached graph and skips configuration entirely.

How It Works

First Build (cold)                    Subsequent Builds (warm)
┌─────────────────────┐              ┌─────────────────────┐
│ 1. Evaluate all     │              │ 1. Deserialize      │
│    build.gradle.kts │  6-15s       │    cached task      │  <1s
│ 2. Build task graph │              │    graph            │
│ 3. Serialize graph  │              │ 2. Execute tasks    │
│    to cache         │              │                     │
│ 4. Execute tasks    │              │                     │
└─────────────────────┘              └─────────────────────┘

Savings: 5-30 seconds per build on large projects
# gradle.properties
org.gradle.configuration-cache=true

# During migration — warn instead of failing on problems
org.gradle.configuration-cache.problems=warn

Configuration Cache Restrictions

The configuration cache imposes strict rules on what build scripts and plugins can do:

  • Cannot access Project at execution time — no task.project.name inside @TaskAction
  • Cannot use Task.project in task actions — pass needed values as @Input properties instead
  • System properties and environment variables must be declared as build inputs
  • External processes started during configuration must be deterministic
// BAD: accessing project at execution time
abstract class BadTask : DefaultTask() {
    @TaskAction
    fun run() {
        val name = project.name  // FAILS with configuration cache
        println("Building $name")
    }
}

// GOOD: pass project data as an input property
abstract class GoodTask : DefaultTask() {
    @get:Input
    abstract val moduleName: Property<String>

    @TaskAction
    fun run() {
        println("Building ${moduleName.get()}")
    }
}

tasks.register<GoodTask>("myTask") {
    moduleName.set(project.name)  // resolved during configuration, serialized
}
Migration Tip
Many third-party plugins don't support configuration cache yet. Use --configuration-cache-problems=warn during migration to identify incompatible plugins without breaking your build. Track plugin compatibility at gradle-configuration-cache-status.

6. Parallel Execution

By default, Gradle executes tasks sequentially. Enabling parallel execution allows independent tasks across different modules to run simultaneously.

# gradle.properties
org.gradle.parallel=true
org.gradle.workers.max=8  # default: number of CPU cores

How Parallel Execution Works

Sequential (default)
────────────────────────────────────────────────────────► time
:core:model:compile ──► :core:data:compile ──► :feature:home:compile ──► :app:compile

Parallel (org.gradle.parallel=true)
────────────────────────────────────────────────────────► time
:core:model:compile ──────────────────────────────┐
:core:network:compile ────────────────────────┐    ├──► :app:compile
:core:ui:compile ─────────────────────────────┤    │
                                              ▼    │
                              :feature:home:compile┘
                              :feature:search:compile

Parallel execution works at the MODULE level.
Tasks within the same module still run sequentially.

Parallel execution is most effective when your module graph is wide (many independent modules) rather than deep (long dependency chains). A monolithic single-module app gets zero benefit.

Caveat
Shared mutable state between tasks can cause race conditions under parallel execution. Custom tasks that write to shared directories or modify global state must use proper synchronization or the Worker API with classloader/process isolation.

7. Incremental Compilation — Kotlin and Java

Incremental compilation is the difference between a 30-second rebuild and a 2-second one. Instead of recompiling every source file, the compiler analyzes what changed and recompiles only the affected files.

Kotlin Incremental Compilation

Kotlin uses ABI (Application Binary Interface) change detection to minimize recompilation:

Change in UserRepository.kt
          │
          ▼
┌─────────────────────────────┐
│ ABI Change Detection        │
│                             │
│ Did the PUBLIC API change?  │
│ (public/protected classes,  │
│  functions, properties,     │
│  return types, parameters)  │
└──────────┬──────────────────┘
           │
     ┌─────┴─────┐
     │            │
     ▼            ▼
   YES           NO
     │            │
     ▼            ▼
Recompile:     Recompile:
UserRepository  UserRepository ONLY
+ ALL files     (implementation-only
  that import    change)
  UserRepository
# gradle.properties — both enabled by default
kotlin.incremental=true
kotlin.incremental.useClasspathSnapshot=true

Classpath snapshots (useClasspathSnapshot) enable more precise change detection. Instead of treating the entire classpath as a single input, Kotlin analyzes the ABI of each dependency. If a dependency's implementation changes but its ABI doesn't, downstream modules skip recompilation.

What Breaks Incremental Compilation

ScenarioImpactWhy
Public API changeRecompiles all dependentsABI changed — dependents may reference the modified API
Adding a new public functionRecompiles all dependentsABI changed (new symbol added)
Changing a private function bodyRecompiles only that fileABI unchanged
Changing a constant (const val)Recompiles all dependentsConstants are inlined at compile time
Non-incremental annotation processorFull recompilationProcessor can't declare which files it affects
Staff Insight
Module boundaries are your best friend for incremental compilation. If :feature:home depends on :core:data via api configuration, any ABI change in :core:data forces recompilation of :feature:home. But if the dependency is implementation, the ABI boundary is at the module level — internal changes in :core:data that don't affect its public API won't trigger recompilation in :feature:home. This is why implementation should be your default dependency configuration and api used only when genuinely needed.

8. kapt vs KSP — Annotation Processing Performance

Annotation processing is often the single largest contributor to build time in Android projects. Understanding the difference between kapt and KSP is critical.

kapt: The Legacy Approach

kapt Pipeline
┌──────────────┐     ┌──────────────────┐     ┌──────────────────┐
│ Kotlin Source │────►│ Generate Java    │────►│ Run Java AP on   │
│ Files (.kt)  │     │ Stubs (.java)    │     │ Stubs            │
└──────────────┘     │ (full compilation│     │ (Dagger, Room,   │
                     │  pass just for   │     │  etc.)           │
                     │  stub generation)│     └────────┬─────────┘
                     └──────────────────┘              │
                                                       ▼
                                              ┌──────────────────┐
                                              │ Generated Code   │
                                              │ (.java/.kt)      │
                                              └──────────────────┘

Problems:
1. Stub generation = extra compilation pass (expensive)
2. Non-incremental — ANY source change reruns ALL processors
3. Processors see Java stubs, not actual Kotlin (lossy)

KSP: The Modern Approach

KSP Pipeline
┌──────────────┐     ┌──────────────────┐     ┌──────────────────┐
│ Kotlin Source │────►│ Kotlin Compiler  │────►│ KSP Processor    │
│ Files (.kt)  │     │ Front-end        │     │ (native Kotlin   │
└──────────────┘     │ (PSI/FIR)        │     │  symbol access)  │
                     └──────────────────┘     └────────┬─────────┘
                                                       │
                                                       ▼
                                              ┌──────────────────┐
                                              │ Generated Code   │
                                              │ (.kt)            │
                                              └──────────────────┘

Advantages:
1. No stub generation — processes Kotlin directly
2. Incremental — only processes changed files + dependents
3. 2-5x faster than kapt for most processors

Migration: kapt to KSP

// build.gradle.kts — BEFORE (kapt)
plugins {
    id("com.google.devtools.ksp") version "2.1.0-1.0.29"  // add KSP
    // kotlin("kapt")  ← remove after migration
}

dependencies {
    // Dagger/Hilt — kapt → ksp
    // kapt("com.google.dagger:hilt-compiler:2.53.1")
    ksp("com.google.dagger:hilt-compiler:2.53.1")

    // Room — kapt → ksp
    // kapt("androidx.room:room-compiler:2.6.1")
    ksp("androidx.room:room-compiler:2.6.1")

    // Moshi — kapt → ksp
    // kapt("com.squareup.moshi:moshi-kotlin-codegen:1.15.1")
    ksp("com.squareup.moshi:moshi-kotlin-codegen:1.15.1")
}

KSP Migration Status (Major Libraries)

LibraryKSP SupportSince Version

Staff Insight
kapt is the single biggest contributor to slow incremental builds. In a project with Hilt + Room + Moshi on kapt, a one-line change in a @HiltViewModel triggers full annotation processing across the entire module — often 15-25 seconds. Migrating to KSP cuts this to 2-4 seconds because KSP is incremental. This is the highest-ROI build performance optimization for most Android projects. Prioritize it above all other build optimizations.

9. Gradle Daemon Tuning

The Gradle Daemon is a long-lived JVM process that avoids the startup cost of a fresh JVM on every build. Tuning its JVM arguments can significantly impact build performance.

Recommended gradle.properties for Large Projects

# gradle.properties

# --- Daemon JVM ---
org.gradle.jvmargs=-Xmx4g \
  -XX:MaxMetaspaceSize=512m \
  -XX:+HeapDumpOnOutOfMemoryError \
  -XX:+UseG1GC

# --- Parallelism ---
org.gradle.parallel=true
org.gradle.workers.max=8

# --- Caching ---
org.gradle.caching=true
org.gradle.configuration-cache=true

# --- File System Watching ---
org.gradle.vfs.watch=true

# --- Kotlin ---
kotlin.incremental=true
kotlin.incremental.useClasspathSnapshot=true

JVM Tuning Breakdown

FlagDefaultRecommendedWhy
-Xmx512m4g (large projects)Prevents OOM during compilation of large modules; too high wastes memory
-XX:MaxMetaspaceSizeUnlimited512mCaps classloader metadata; prevents runaway metaspace from many plugins
-XX:+UseG1GCG1 (JDK 11+)G1Low-pause collector, good for large heaps; ZGC is experimental alternative
-XX:+HeapDumpOnOutOfMemoryErrorOffOnProduces heap dump for post-mortem analysis on OOM

File System Watching

With org.gradle.vfs.watch=true, Gradle maintains a Virtual File System (VFS) that tracks file changes via OS-level file watchers (inotify on Linux, FSEvents on macOS). Without it, Gradle scans the entire project directory tree on every build to determine what changed.

# Impact on UP-TO-DATE checks:
# Without VFS watching: scan all input files → hash → compare (slow)
# With VFS watching:    OS notifies of changes → only re-hash changed files (fast)

Daemon Lifecycle

# Keep daemon alive longer for frequent builds (default: 3 hours)
org.gradle.daemon.idletimeout=10800000

# Check daemon status
./gradlew --status

# Stop all daemons (useful when daemon is misbehaving)
./gradlew --stop

10. CI Optimization

CI environments have different characteristics than developer machines: they may be ephemeral containers, have different CPU/memory profiles, and run full builds rather than incremental ones. Optimizing CI builds requires a different strategy.

Test Parallelization

// build.gradle.kts — module-level
tasks.withType<Test>().configureEach {
    maxParallelForks = (Runtime.getRuntime().availableProcessors() / 2)
        .coerceAtLeast(1)

    // Fork a new JVM every N tests to prevent memory leaks
    forkEvery = 100
}

CI Pipeline Example (GitHub Actions)

# .github/workflows/build.yml
name: Build
on: [push, pull_request]

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - uses: actions/setup-java@v4
        with:
          distribution: 'zulu'
          java-version: '17'

      - uses: gradle/actions/setup-gradle@v4
        with:
          cache-read-only: ${{ github.ref != 'refs/heads/main' }}

      - name: Build
        run: ./gradlew assembleDebug --build-cache

      - name: Unit Tests
        run: ./gradlew testDebugUnitTest --build-cache

      - name: Lint
        run: ./gradlew lintDebug --build-cache
Key Detail
cache-read-only on PR builds prevents cache pollution from feature branches. Only the main branch pushes to the cache, ensuring cache entries reflect the canonical codebase state.
Staff Insight
The biggest CI mistake is running ./gradlew clean assembleDebug on every build. clean destroys all incremental build state, local cache entries, and UP-TO-DATE information. Every task re-executes from scratch. Remove clean from your CI scripts and rely on Gradle's correctness guarantees instead. If you don't trust Gradle's UP-TO-DATE checks, that's a bug to fix — not a reason to throw away all caching.

11. Staff-Level Corner Cases

buildSrc: The Silent Build Killer

buildSrc is a special Gradle directory that compiles before the main build. Any change to buildSrc — even a comment — invalidates the entire build. Every task across every module re-executes.

# buildSrc change impact:
#
# Before: 162 tasks UP-TO-DATE, 0 EXECUTED
# After changing ONE file in buildSrc: 0 UP-TO-DATE, 162 EXECUTED
#
# Fix: migrate buildSrc to convention plugins in an included build
# (see 06-convention-plugins.html)

The fix is to move shared build logic from buildSrc to convention plugins in a composite build (e.g., build-logic/). Changes to a convention plugin only invalidate modules that apply that specific plugin.

Gradle Version Upgrades and Cache Invalidation

Gradle version is part of the build cache key. Upgrading Gradle invalidates the entire remote cache — every task produces a cache miss on the first build after the upgrade. Plan Gradle upgrades to happen early in the week so CI has time to repopulate the cache before developers need it.

Non-Deterministic Tasks

Tasks that produce different outputs for the same inputs cause perpetual cache misses:

// BAD: timestamp in generated file → different output every build
abstract class GenerateBuildInfo : DefaultTask() {
    @get:OutputFile
    abstract val outputFile: RegularFileProperty

    @TaskAction
    fun generate() {
        outputFile.get().asFile.writeText("""
            buildTime=${System.currentTimeMillis()}
            version=1.0.0
        """.trimIndent())
    }
}

// GOOD: separate volatile data from cacheable data
abstract class GenerateBuildInfo : DefaultTask() {
    @get:Input
    abstract val version: Property<String>

    @get:OutputFile
    abstract val outputFile: RegularFileProperty

    @TaskAction
    fun generate() {
        outputFile.get().asFile.writeText("""
            version=${version.get()}
        """.trimIndent())
    }
}

// Volatile data (timestamp, git hash) goes in a separate non-cached task

Build Scan Comparison

When a build is unexpectedly slow, compare it against a known-good build using Gradle Enterprise's scan comparison. This highlights differences in task outcomes, cache hit rates, and configuration time — pinpointing exactly what regressed.

# Force all tasks to re-execute (useful for debugging cache correctness)
./gradlew assembleDebug --rerun-tasks

# Scan A (fast): https://ge.mycompany.com/s/scan-id-a
# Scan B (slow): https://ge.mycompany.com/s/scan-id-b/compare/scan-id-a

Compilation Avoidance vs Incremental Compilation

These are distinct mechanisms that work together:

Staff Insight
Build performance optimization is a system design problem, not a collection of flags. The highest-leverage approach is: (1) profile to find the actual bottleneck, (2) fix the top bottleneck, (3) re-profile. Applying all optimizations blindly without measurement is a junior mistake. A staff engineer establishes a build performance dashboard (Gradle Enterprise or custom metrics), sets regression alerts, and treats build time as a team-wide SLO — not an afterthought.
