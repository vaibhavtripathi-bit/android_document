Problem 05

Crash Resilience and ANR Prevention in a Background-Heavy App

A data-collection SDK runs mostly in the background, reacting to system broadcasts, processing database operations, and managing long-running coroutines. Any unhandled exception kills the host app. Any work on the main thread risks an ANR. This document describes the defensive patterns used to keep the SDK stable.

The Problem

Background SDKs face unique stability challenges:

  • BroadcastReceiver 10-second limit: Android kills the process if onReceive() doesn't return within ~10 seconds. Heavy processing in a receiver causes ANRs.
  • Coroutine failures propagate: An unhandled exception in a coroutine can cancel its parent scope, taking down unrelated work.
  • Bad data causes crash loops: A corrupted database row that causes OOM on read will crash the app every time the reporting worker runs — creating an infinite crash loop.
  • OEM API instability: Querying device-specific ContentProviders or telephony APIs can throw unexpected exceptions on certain devices/OS versions.
  • Module failures shouldn't cascade: If the voice module crashes during startup, the coverage module should still work.

Approaches Used

1. Background BroadcastReceiver Pattern

Instead of doing work directly in onReceive(), an abstract base class immediately moves all processing off the main thread:

  • The base BroadcastReceiver overrides onReceive() as final.
  • Inside, it launches a coroutine on the IO dispatcher using a safe-launch wrapper.
  • Subclasses implement onBackgroundReceive() instead, which runs on a background thread.
  • The main thread returns immediately, avoiding the 10-second ANR window.

Key Insight: By making onReceive() final, the base class guarantees that no subclass accidentally does heavy work on the main thread. The pattern is enforced by the type system, not by code review.

2. Safe Coroutine Launch with Error Absorption

A utility function wraps coroutine launches with error handling:

  • The coroutine runs in its own CoroutineScope (not a shared scope), so failures don't propagate.
  • invokeOnCompletion catches any Throwable (except CancellationException) and logs it instead of crashing.
  • An optional error callback allows callers to add custom error handling.
  • CancellationException is explicitly excluded — cancellation is a normal control flow, not an error.

For structured concurrency where parent-child relationships matter, a separate utility creates scopes with SupervisorJob — child failures don't cancel siblings.

3. safeTry Wrappers for Module Orchestration

The lifecycle manager wraps every module start/stop call in a try-catch:

  • If VoiceProvider.moduleStart() throws, the exception is logged and the loop continues to CoverageProvider.moduleStart().
  • The same pattern is used in the module state observer — if one listener's callback throws, others still receive the event.
  • This creates a "best effort" model: as many modules as possible will work, even if some are broken.

4. Defensive Database Record Handling

The reporting pipeline includes multiple safeguards against bad data:

  • Oversized row deletion: On startup, any event row whose payload exceeds the configured size limit is deleted. This prevents OOM when the reporting worker tries to load and serialize all pending events.
  • Stuck status reset: Report entries left in "SENDING" status (from a crash during upload) are reset to "NOT_SENT" so they can be retried.
  • Debug vs. Release behavior: In debug builds, JSON serialization failures throw exceptions to surface bugs. In release builds, the bad record is silently deleted to prevent crash loops.

Key Insight: The debug/release split is crucial. During development, you want to know about bad data immediately (crash = fix the bug). In production, you want the app to survive (delete the bad record, keep running).

5. Process Exit Reason Tracking

On Android 11+, the SDK uses ActivityManager.getHistoricalProcessExitReasons() to understand why the app previously died:

  • It categorizes exits by reason: crash, ANR, excessive resource usage, low memory, user request, etc.
  • This data is collected as analytics events, giving visibility into stability issues across the device fleet.
  • The counts help prioritize which crash/ANR categories to fix first.

6. SecurityException Blanket Handling

Telephony and OEM ContentProvider queries are wrapped in try-catch blocks that specifically handle SecurityException:

  • Some devices revoke permissions unexpectedly or have broken permission models.
  • OEM-specific ContentProviders may not exist on all devices, throwing when queried.
  • Rather than crashing, the SDK logs the failure and returns default/empty data.

7. Crash Recovery via Persisted State

Modules that track ongoing sessions (e.g., battery monitoring, satellite sessions) persist their state to SharedPreferences:

  • If the process dies mid-session, the module can recover its state on restart.
  • Session duration calculators store start timestamps so elapsed time can be computed even after a restart.
  • State machines (FSMs) persist their current state, allowing them to resume from the correct point.

Key Takeaways

  • Never do work on the main thread in receivers. Use a base class that enforces background execution via the type system.
  • Isolate coroutine failures. Use separate scopes or SupervisorJob to prevent one failure from cascading.
  • Module failures must be independent. The orchestrator must survive any individual module's crash.
  • Bad data is inevitable. The database will eventually contain corrupt or oversized records. Handle them proactively.
  • Debug and release need different error policies. Crash early in debug; survive gracefully in release.
  • Persist state for crash recovery. Any long-running operation should be resumable after process death.
