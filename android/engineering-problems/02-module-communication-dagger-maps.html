Problem 02

Dynamic Module Lifecycle Management via Dependency Injection

In a large SDK with 20+ feature modules, each module needs to be independently started, stopped, and controlled at runtime based on server configuration, user consent, and device state. This document describes how Dagger/Hilt multibinding maps create a plugin-style architecture where the orchestrator doesn't know about specific modules.

The Problem

Imagine an SDK with modules for voice analytics, coverage tracking, battery monitoring, heartbeat, latency measurement, and many more. Each module:

  • Should start only if its server-side config says isEnabled: true.
  • Should stop if user consent is revoked or a "panic mode" is activated.
  • Should react to lifecycle changes (satellite mode, screen state, battery level).
  • Must be independently deployable — adding a new module shouldn't require changing the orchestrator.

The naive approach — a giant when/switch statement in the orchestrator that knows about every module — violates the Open/Closed Principle and becomes unmaintainable at scale.

The Architecture

1. Three Interfaces, One Provider

Each feature module implements up to three interfaces:

  • ModuleStarter: Has shouldStart(lifecycleStatus) and moduleStart(). The shouldStart method checks config, consent, panic mode, etc.
  • ModuleStopper: Has moduleStop(). Called when the module should shut down.
  • ModuleController: Has onChange(lifecycleStatus). Called on every lifecycle update so the module can react to changes without full restart.

A single provider class (e.g., VoiceProvider) typically implements all three, but this isn't required. Some modules only need start/stop.

2. Dagger Multibinding Maps for Registration

Each module registers itself into the appropriate maps using Dagger's @Binds @IntoMap annotations with a custom key annotation:

  • A central DI module declares @Multibinds for each map type (so the map exists even if empty).
  • Each feature's DI module contributes its provider with @Binds @IntoMap @ModuleStarterKey(VoiceProvider::class).
  • The key is the provider's KClass, making each entry uniquely identifiable.

Key Insight: Adding a new module requires only: (1) implement the interfaces, (2) add a @Binds @IntoMap entry. The orchestrator code never changes. This is the Open/Closed Principle enforced by the DI framework.

3. The Orchestrator: Lifecycle Manager

A single AppLifecycleManager class injects all three maps and orchestrates module lifecycle:

  • It subscribes to a LifecycleStatusUpdateEvent on the app's EventBus.
  • On each event (config change, consent change, panic mode toggle), it iterates the maps.
  • Start logic: For each entry in the starter map, if shouldStart() returns true, call moduleStart().
  • Stop logic: For each entry in the stopper map, if the same class also exists in the starter map AND shouldStart() is true, skip the stop (module should stay running). Otherwise, call moduleStop().
  • Controller logic: For every controller, call onChange() unconditionally so modules can react to state changes.

The orchestrator wraps each start/stop call in a safeTry block — if one module throws, others still get processed.

4. The shouldStart Decision Tree

Each module's shouldStart() method checks a combination of:

  • Global config: Is the module's section present and isEnabled: true?
  • User consent: Has the user completed the out-of-box experience (OOBE) and granted consent?
  • Panic mode: Is the SDK in a self-panic state (emergency shutdown)?
  • Satellite mode: Is the device in satellite mode (some modules should stop)?
  • Module-specific conditions: E.g., voice module checks if voice config exists; reporting checks if reporting URL is configured.

All these conditions are aggregated in a LifecycleStatusProvider that reads from config, SharedPreferences, and a module state repository.

5. Why Not Runtime DI Graph Changes?

Hilt/Dagger graphs are fixed at compile time. You cannot add or remove modules from the DI graph at runtime. The solution elegantly sidesteps this:

  • All modules are always in the DI graph (compiled in).
  • The behavioral enable/disable happens in application logic (shouldStart / moduleStart / moduleStop).
  • This means the DI graph is stable and predictable, while runtime behavior is flexible.

6. Manifest Component Toggling (Complementary)

For Android components that must be completely disabled (e.g., BroadcastReceivers that shouldn't even receive intents), the module uses PackageManager.setComponentEnabledSetting() to enable/disable manifest-declared components at runtime.

This is complementary to the DI-based approach — DI controls the module's business logic, while manifest toggling controls the OS-level component registration.

Key Takeaways

  • Multibinding maps are a plugin system. They let you register implementations without the consumer knowing about them at compile time.
  • Separate "can run" from "is wired." All modules exist in the DI graph; runtime logic decides which ones are active.
  • Three interfaces give fine-grained control. Start, stop, and onChange cover the full lifecycle spectrum.
  • Fault isolation is critical. One module's failure during start/stop must not prevent others from being processed.
  • The orchestrator is dumb by design. It only knows about interfaces, not implementations. This is the power of the pattern.
