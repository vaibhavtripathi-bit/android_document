Problem 07

Remote Configuration with Fallback, Caching, and Safe Propagation

Every module in the SDK is controlled by server-side configuration — rate limits, enabled/disabled flags, reporting intervals, trigger thresholds. But the server might be down, the config might be malformed, or the device might be offline. This document describes a robust configuration system with multiple fallback layers.

The Problem

The SDK's behavior is almost entirely config-driven. Consider what config controls:

  • Which modules are enabled or disabled
  • How many events per hour each trigger type can fire
  • How often reports are uploaded
  • What data fields to collect
  • Which apps to monitor for latency
  • Retention sizes and daily limits

If config download fails, the SDK must still function with reasonable defaults. If config changes, modules must react without a restart. If the config is partially malformed, valid sections should still apply.

The Configuration Pipeline

1. Three-Layer Fallback Chain

Configuration is resolved through a priority chain:

  1. Server config (highest priority): Downloaded periodically via a WorkManager job. Parsed with kotlinx.serialization. If valid, it becomes the active config.
  2. Cached config (middle priority): The last successfully downloaded config is persisted in SharedPreferences. If the server is unreachable, the cached version is used.
  3. Asset fallback (lowest priority): A configuration.json file bundled in the app's assets provides sensible defaults. Used on first launch before any server config has been downloaded.

Key Insight: The asset fallback ensures the SDK works immediately after install, even if the device is offline. The cached config ensures it works after a server outage. The server config ensures it can be tuned without app updates.

2. Diff-Based Config Propagation

When new config is downloaded, it's compared against the current active config:

  • A compareConfig() function identifies which sections changed.
  • Only changed sections trigger update events on the EventBus.
  • Modules subscribe to config events for their specific section and only react when their config actually changed.

This prevents unnecessary module restarts when an unrelated config section is updated.

3. Config-Blocked States

Certain device states block config downloads entirely:

  • Satellite mode: When the device is in satellite mode, network requests should be minimized. Config downloads are suppressed.
  • Self-panic mode: An emergency shutdown state where the SDK should do as little as possible. Config downloads are suppressed.
  • Cache short-circuit: An optional flag allows the SDK to skip server downloads entirely and use only cached/asset config. Useful for testing or when the server is known to be down.

4. WorkManager-Based Periodic Download

Config is downloaded on a schedule, not on-demand:

  • A periodic WorkManager job runs at a configurable interval (itself part of the config — the interval for downloading config is in the config).
  • The job has a NetworkType.CONNECTED constraint — it won't run without network.
  • An optional "traffic even out" initial delay prevents all devices from hitting the server simultaneously.
  • When the config changes the download interval, the WorkManager job is rescheduled with the new interval.

5. Module-Specific Config Models

The monolithic server config JSON is deserialized into a hierarchy of typed Kotlin data classes:

  • A top-level Config class contains optional fields for each module: voice, coverage, analytics, heartbeat, etc.
  • Each module's config is its own data class with typed fields (not raw JSON).
  • Modules access their config through the IConfigProvider interface, which exposes a Flow of config updates.
  • Default values in the data classes ensure that missing fields don't cause crashes.

6. Config Drives Everything

The config system is the single source of truth for SDK behavior. It controls:

  • Module enable/disable: Each module section has an isEnabled flag.
  • Rate limits: eventsPerHour per trigger type.
  • Reporting: Upload intervals, retention sizes, daily limits.
  • Data collection: Which fields to collect, which apps to monitor.
  • Scheduling: WorkManager intervals for various periodic jobs.

This means the backend team can effectively reprogram the SDK's behavior without an app update — a critical capability for a deployed SDK.

Key Takeaways

  • Always have a fallback chain. Server → Cache → Asset ensures the app works in any connectivity scenario.
  • Diff before propagating. Only notify modules about config sections that actually changed.
  • Config should control its own refresh. The download interval being part of the config creates a self-tuning system.
  • Block downloads in emergency states. Satellite mode and panic mode should suppress non-essential network activity.
  • Use typed models, not raw JSON. Kotlin data classes with defaults are safer and more ergonomic than parsing JSON at point of use.
