Problem 10

Multi-Process Safety and Reliable Background Work

Android apps can run in multiple processes (e.g., for services, content providers, or FCM). Each process gets its own Application.onCreate() call, its own memory space, and its own DI graph. This document describes how to handle initialization safely across processes and keep background work alive.

The Problem

When an Android app declares components in separate processes (e.g., android:process=":fcm"), the system creates separate OS processes. Each process:

  • Runs Application.onCreate() independently: Your full initialization code runs in every process, not just the main one.
  • Has its own memory: Singletons in one process don't exist in another.
  • Has its own DI graph: Hilt creates a separate component in each process.
  • Can start unexpectedly: A push notification can start the FCM process without the main process running.

If the SDK's full initialization (database setup, module startup, WorkManager scheduling) runs in every process, you get duplicated work, database contention, and wasted resources.

Approaches Used

1. Process Name Gating

The Application class checks the current process name before running initialization:

  • On onCreate(), the app calls getProcessName() and compares it to the default package name.
  • If the process name matches the package name, it's the main process — full initialization runs.
  • If the process name is different (e.g., :fcm, :service), only minimal setup runs.
  • This prevents modules from initializing, databases from being set up, and WorkManager jobs from being scheduled in non-main processes.

Key Insight: Without this check, every push notification that wakes the FCM process would trigger full SDK initialization — registering receivers, scheduling workers, and starting modules. This wastes resources and can cause race conditions with the main process.

2. Keep-Alive Service for Process Persistence

Android aggressively kills background processes to reclaim memory. For a telemetry SDK that needs to run continuously, this is a problem:

  • A sticky foreground service keeps the process alive with higher priority.
  • The service is managed by a KeepAliveManager that can be started/stopped based on config.
  • The "sticky" flag (START_STICKY) tells Android to restart the service if it's killed.
  • This is a deliberate trade-off: the service consumes a notification slot and some resources, but ensures data collection isn't interrupted.

3. WorkManager as the Primary Background Scheduler

Instead of relying on services or alarms for periodic work, the SDK uses WorkManager extensively:

  • Survives process death: WorkManager persists work requests in a database. If the process dies, the work runs when the process restarts.
  • Handles constraints: Network connectivity, battery level, and other constraints are checked before running.
  • Deduplication: enqueueUniquePeriodicWork with policies like KEEP (don't replace existing) or REPLACE (cancel and reschedule) prevents duplicate workers.
  • Backoff: Failed workers retry with exponential backoff (typically 5-10 minutes initial delay).

4. Initialization Race Prevention

Even within the main process, initialization order matters. Some specific races are handled:

  • Certain initialization steps (Firebase, Play Services) run on background threads via GlobalScope.launch(Dispatchers.IO).
  • Comments in the codebase document specific race conditions discovered in production — e.g., a periodic report worker starting before SharedPreferences are initialized.
  • The initialization framework (with dependency resolution) ensures that modules don't start before their dependencies are ready.
  • The report upload worker skips the first ~15 minutes after scheduling to allow initial data accumulation and module startup to complete.

5. WorkManager Unique Work Policies

Different features use different policies for handling existing work:

  • KEEP: If a worker with the same name already exists, keep it. Used for heartbeat network workers — don't restart a running heartbeat.
  • REPLACE: Cancel the existing worker and schedule a new one. Used when config changes the interval — the old schedule is outdated.
  • CANCEL_AND_REENQUEUE: Similar to REPLACE but with different semantics for in-progress work.
  • UPDATE: Update the existing worker's parameters without cancelling it.

Choosing the right policy prevents both duplicate workers (wasting resources) and lost workers (missing scheduled work).

Key Takeaways

  • Always check the process name. In a multi-process app, Application.onCreate() runs in every process. Gate your initialization.
  • Use WorkManager for periodic background work. It survives process death, handles constraints, and prevents duplicates.
  • Keep-alive services are a trade-off. They keep the process alive but consume resources. Use them only when continuous operation is truly required.
  • Document race conditions. When you discover and fix a race, leave a comment explaining it. The next developer will thank you.
  • Choose WorkManager policies carefully. KEEP vs. REPLACE vs. UPDATE have very different behaviors for in-progress and pending work.
