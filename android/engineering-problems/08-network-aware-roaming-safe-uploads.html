Problem 08

Network-Aware and Roaming-Safe Data Uploads

A telemetry SDK must upload data to servers, but doing so at the wrong time can cost users money (roaming charges), waste battery (retrying on flaky connections), or fail silently (no network). This document describes a multi-layered approach to network-aware data transmission.

The Problem

The SDK generates report files that need to reach the server. But network conditions vary wildly:

  • No network: The device is in airplane mode or a dead zone. Uploads must be deferred.
  • Cellular roaming: The user is abroad. Uploading telemetry data on roaming cellular could incur significant charges.
  • Flaky connection: The network is technically available but unreliable. Uploads fail mid-transfer.
  • WiFi available while roaming: Even when cellular is roaming, WiFi might be available and free to use.
  • Process death mid-upload: The app is killed while an upload is in progress. The report status is stuck in "SENDING."

Approaches Used

1. WorkManager Constraints as First Gate

The report upload worker uses WorkManager's built-in constraint system:

  • NetworkType.CONNECTED constraint ensures the worker only runs when the device has a validated network connection.
  • WorkManager handles the scheduling — if the constraint isn't met, the work is deferred until it is.
  • This is the coarse-grained gate: no network = no attempt.

Note: The constraint is CONNECTED, not UNMETERED. This means uploads can happen on cellular, which is intentional — the SDK needs to report data even when WiFi isn't available. The roaming check (below) handles the cost concern.

2. Roaming Check at Upload Time

Even after WorkManager's constraint is satisfied, the upload code performs a fine-grained roaming check:

  • Before each HTTP POST, the code checks: Is the device roaming on cellular?
  • If roaming on cellular: Is WiFi also connected?
  • Uploads proceed only if: (a) not roaming, OR (b) roaming but WiFi is active.
  • If the check fails, the upload is skipped for this cycle. The report file stays on disk for the next attempt.

Key Insight: The two-layer check (WorkManager constraint + runtime roaming check) is necessary because WorkManager's CONNECTED constraint doesn't distinguish between roaming and non-roaming cellular. The runtime check adds the business logic that WorkManager can't express.

3. Sequential File Upload with Status Tracking

Report files are uploaded one at a time, not in parallel:

  • The sender iterates all files with status NOT_SENT.
  • Before sending, the status is updated to SENDING.
  • On success, the status becomes SENT and the file is eligible for cleanup.
  • On failure, the status reverts to NOT_SENT for retry.

Sequential processing prevents flooding the server with parallel requests and makes it easy to stop mid-batch if network conditions change.

4. Stuck Status Recovery

If the process dies while a report is in SENDING status, it would be stuck forever without intervention:

  • On startup, the reporting engine scans for reports stuck in SENDING status.
  • These are reset to NOT_SENT, making them eligible for retry.
  • This prevents data loss from process death during upload.

5. Pending Request Queue in Database

The network layer maintains its own queue of pending requests:

  • Each network request is persisted as a database row with status PENDING.
  • A startPendingRequests() method loads all pending requests and processes them sequentially.
  • This ensures requests survive process death and are retried on next startup.

6. Retry Configuration (Modeled, Not Fully Implemented)

The network layer includes a retry configuration model with:

  • Max retry count: How many times to retry a failed request.
  • Retry type: Linear or exponential backoff.
  • Initial backoff: The delay before the first retry.

WorkManager jobs that wrap network operations use their own backoff policies (typically exponential with 5-10 minute initial delays).

The Decision Flow

CONDITION RESULT
No networkWorkManager defers the job
Connected, not roamingUpload proceeds
Connected, roaming on cellular, no WiFiUpload skipped this cycle
Connected, roaming on cellular, WiFi activeUpload proceeds (via WiFi)
Upload fails mid-transferStatus reverts to NOT_SENT, retried next cycle

Key Takeaways

  • Layer your network checks. WorkManager constraints handle coarse checks; runtime code handles business logic like roaming.
  • Never block on roaming cellular. Users will uninstall an app that costs them money abroad.
  • Track upload status in the database. This enables retry after process death and prevents duplicate uploads.
  • Upload sequentially, not in parallel. It's simpler, easier to stop mid-batch, and doesn't flood the server.
  • Recover stuck states on startup. Any status that can be interrupted by process death needs a recovery path.
