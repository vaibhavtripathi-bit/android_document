Problem 04

Dependency-Aware Ordered Initialization of Modules

When an app has dozens of modules that must initialize at startup — and some depend on others being ready first — you need a system that resolves dependencies, detects cycles, and ensures each module initializes exactly once. This document describes a topological-sort-based initialization framework.

The Problem

At app startup, the following must happen in order:

  1. Configuration must download/load before any module can read its config.
  2. Reporting infrastructure must be ready before modules try to submit reports.
  3. Consent/OOBE must be checked before data collection begins.
  4. Secret code handlers and communication managers need config but not reporting.
  5. Miscellaneous initializers (logging, debug tools) have no dependencies.

Hardcoding this order in a sequential list is fragile — adding a new module means manually finding the right insertion point. Removing a module might break the chain.

The Approach

Dependency Graph (example): MiscInitializer ──────────────────────────────┐ (no deps) │ ▼ ConfigProvider ──────────▶ ReportProvider ──▶ [Ready] (no deps) (needs Config) │ ▼ SecretCodeManager OobeProvider (needs Config) (needs Config) │ ▼ OrcaCommManager (needs Config, SecretCode)

1. ModuleInitializer Base Class

Every module that needs startup initialization extends a ModuleInitializer base class with two key methods:

  • dependencies(): Returns a list of other ModuleInitializer classes that must complete before this one runs. Returns empty list if no dependencies.
  • runInitialize(): Contains the actual initialization logic. Called exactly once.

The base class includes a @Synchronized guard ensuring runInitialize() is called only once, even if the resolution algorithm visits it multiple times.

2. Multibinding Map for Discovery

All initializers are registered in a Dagger multibinding map:

  • Key: Class<out ModuleInitializer>
  • Value: The initializer instance

The AppInitializerManager injects this map and has access to every initializer without knowing about specific implementations.

3. Dependency Resolution with Cycle Detection

At startup, the manager resolves the initialization order:

  • For each initializer, recursively resolve its dependencies() first.
  • Dependencies are looked up in the injected map by class.
  • In debug builds, a cycle detection algorithm runs to catch circular dependencies early (e.g., A depends on B, B depends on A). This surfaces configuration bugs during development.
  • In release builds, cycle detection is skipped for performance — the assumption is that debug testing caught any cycles.

Key Insight: The "run once" guard in the base class means even if the resolution visits an initializer multiple times (because multiple modules depend on it), the actual initialization code runs exactly once. This makes the resolution algorithm simpler — it doesn't need to track "already initialized" state externally.

4. Separation from Lifecycle Management

Initialization and lifecycle are deliberately separate concerns:

  • Initialization (via ModuleInitializer): Runs once at app startup. Sets up databases, registers receivers, loads cached state. Order matters.
  • Lifecycle (via ModuleStarter/Stopper): Runs repeatedly as config/consent changes. Starts and stops data collection. Order doesn't matter (each module decides independently).

This separation means a module can be initialized (database ready, receivers registered) but not started (not collecting data) — which is the correct state when consent hasn't been granted yet.

Key Takeaways

  • Declare dependencies, don't hardcode order. Each module knows what it needs, not where it sits in a list. The framework resolves the order.
  • Run-once semantics simplify everything. The base class guard means the resolution algorithm can be simple and recursive.
  • Cycle detection in debug only. It's a development-time safety net, not a runtime cost.
  • Separate initialization from lifecycle. "Set up" and "start collecting" are different concerns with different ordering requirements.
  • Multibinding maps make it extensible. New modules register themselves; the manager discovers them automatically.
