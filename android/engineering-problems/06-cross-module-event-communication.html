Problem 06

Cross-Module Event Communication Without Tight Coupling

In a multi-module SDK, the reporting module needs to know when config changes. The analytics module needs to know when the screen turns on. The coverage module needs to know about satellite mode. But none of these modules should directly depend on each other. This document describes three complementary communication mechanisms that maintain loose coupling.

The Problem

Consider these real communication needs:

  • When server config is downloaded, every module needs to re-evaluate whether it should be running.
  • When the screen turns on/off, coverage and analytics modules need to record events.
  • When the device enters satellite mode, most modules should pause.
  • When a voice call ends, the voice module needs to trigger report generation.
  • When consent is granted, all modules should start collecting data.

If each module directly called the others, you'd have an N*N dependency graph — every module depending on every other module. This makes the codebase unmaintainable and untestable.

1. EventBus: SharedFlow-Based Pub/Sub

A lightweight, in-process event bus built on Kotlin's MutableSharedFlow:

  • Publishing: Any module can post(event) on the IO dispatcher. Events are typed using a ticket/event class hierarchy.
  • Subscribing: Modules call register<EventType>() to get a Flow of events of that type, or subscribeAsJob() for a fire-and-forget listener.
  • Scope: Application-wide singleton. All modules share the same bus instance.

Used for: Config update events, lifecycle status changes, analytics tickets, consent changes, OOBE events, communication triggers.

Key Insight: Using SharedFlow instead of a custom callback registry gives you backpressure handling, coroutine integration, and automatic cleanup when the collecting scope is cancelled. No manual unregister needed.

2. Module State Observer: Filtered System Event Distribution

For system-level events (screen on/off, battery changes, satellite mode), a dedicated observer pattern provides filtered delivery:

  • A ModuleEventEmitter emits ModuleStateUpdateEvent instances (a sealed class hierarchy: screen events, battery events, satellite events, OOBE events, etc.) into a SharedFlow with a buffer of 64.
  • A ModuleStateObserver injects a Dagger multibinding map of IModuleStateUpdateListener implementations.
  • Each listener declares which event types it cares about via supportedSystemEvents().
  • The observer filters the event stream per listener, only delivering events that match.

Why separate from EventBus? This layer is specifically for system state changes that affect module behavior. The filtering mechanism (each listener declares its interests) prevents modules from receiving irrelevant events. The buffered SharedFlow with suspend-on-overflow ensures no events are dropped even if a listener is slow.

3. Interface-Based Direct Communication (via DI)

For cases where one module needs to call another's API (not just react to events), narrow interfaces are used:

  • Each feature module exposes a provider interface (e.g., ICoverageProvider, IReportProvider).
  • The central DI module binds concrete implementations to these interfaces.
  • Consuming modules depend on the interface, not the implementation.
  • This allows direct method calls (e.g., reportProvider.submitEvent(event)) without coupling to the implementation.

Used for: Submitting reports to the reporting engine, requesting config values, querying module status.

4. Debounced Lifecycle Events

The lifecycle manager applies a debounce (~30 seconds) to lifecycle status update events before processing them:

  • Multiple rapid changes (e.g., config update + consent change + satellite mode toggle in quick succession) are collapsed into a single processing pass.
  • This prevents modules from being rapidly started, stopped, and restarted.
  • The debounce window is long enough to absorb cascading state changes but short enough to feel responsive.

Key Takeaways

  • One communication mechanism is not enough. Different needs (broadcast, filtered delivery, direct calls, lifecycle) require different patterns.
  • SharedFlow is the modern EventBus. It gives you backpressure, coroutine integration, and scope-based cleanup for free.
  • Listeners should declare their interests. The filtered observer pattern prevents modules from processing irrelevant events.
  • Debounce lifecycle changes. Rapid state changes should be collapsed to avoid thrashing modules on and off.
  • Interfaces decouple direct calls. When you need to call another module's method, depend on its interface, not its class.
