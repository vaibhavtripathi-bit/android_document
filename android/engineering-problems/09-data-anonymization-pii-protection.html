Problem 09

Config-Driven Data Anonymization and PII Protection

A telemetry SDK collects device identifiers, phone numbers, and subscriber IDs for diagnostic purposes. But transmitting raw PII violates privacy regulations and user trust. This document describes a pseudo-anonymization system that protects sensitive data while preserving analytical utility.

The Problem

The SDK needs device identifiers to correlate telemetry data across sessions and modules. But these identifiers are personally identifiable information (PII):

  • Phone numbers (MSISDN): Directly identifies a person.
  • Device IDs (IMEI): Uniquely identifies a physical device.
  • Subscriber IDs (IMSI): Identifies a SIM card / subscription.
  • Advertising IDs (GAID): Used for ad tracking, considered PII in many jurisdictions.

Sending these in plaintext is unacceptable. But simply removing them makes the data useless for correlation. The solution: pseudo-anonymize them so the same input always produces the same output (enabling correlation) but the output can't be reversed to the original value.

Approaches Used

1. Salted Hash-Based Pseudo-Anonymization

Each PII field is transformed using a one-way hash with a salt:

  • The raw identifier is combined with a salt (a secret string).
  • The combination is hashed using a cryptographic hash function.
  • The hash output replaces the raw identifier in all reports.
  • The same input + salt always produces the same hash, enabling cross-session correlation.
  • The hash cannot be reversed to recover the original identifier.

Key Insight: The salt is critical. Without it, an attacker could precompute hashes of all possible phone numbers (a rainbow table attack). The salt makes this infeasible. Different product flavors use different salts, preventing cross-flavor correlation.

2. Config-Driven Anonymization Rules

Which fields get anonymized is controlled by server configuration, not hardcoded:

  • The config specifies which identifier types should be anonymized.
  • This allows the backend team to enable/disable anonymization per field without an app update.
  • New identifier types can be added to the anonymization list via config changes.

This flexibility is important because privacy regulations vary by region and evolve over time.

3. Debug Bypass for Development

During development and debugging, anonymized data is hard to work with — you can't look up a specific device by its hash. The system includes a bypass:

  • When a debug mode flag is active, anonymization is skipped and raw identifiers are sent.
  • This flag is only available in debug builds or when a specific debug mode is enabled.
  • In production builds, the bypass is not available, ensuring PII is always protected.

4. Flavor-Specific Salt

Different product variants (e.g., internal vs. OEM builds) use different salts:

  • Each flavor provides its own salt through a flavor-specific logic provider interface.
  • This means the same device ID hashed in the internal build produces a different hash than in the OEM build.
  • Cross-flavor data correlation is impossible, providing an additional privacy boundary.

5. Encryption for In-Memory Sensitive Data

Beyond anonymization for transmission, sensitive data held in memory is also protected:

  • Authentication tokens are stored in an encrypted memory cache using AES encryption.
  • The encryption key is managed internally, and tokens are decrypted only at point of use.
  • This protects against memory dump attacks where an attacker reads process memory.

6. Hilt EntryPoint for Non-DI Contexts

The anonymization provider needs to be accessible from places that don't have Hilt injection (e.g., static utility methods, legacy code):

  • A Hilt @EntryPoint is defined on the SingletonComponent.
  • Non-injected code can access the anonymization provider via EntryPointAccessors.fromApplication().
  • This bridges the gap between DI-managed and non-DI code without breaking the architecture.

Key Takeaways

  • Pseudo-anonymize, don't remove. Hashing preserves correlation ability while protecting identity.
  • Salt your hashes. Without a salt, rainbow table attacks can reverse the anonymization.
  • Make anonymization rules configurable. Privacy requirements change; the app shouldn't need an update to comply.
  • Separate debug and production behavior. Developers need raw data; users need protection. Build variants enforce this.
  • Protect data in memory too. Anonymization at transmission time doesn't help if tokens are readable in memory.
  • Use different salts per product variant. This creates privacy boundaries between different deployments of the same codebase.
