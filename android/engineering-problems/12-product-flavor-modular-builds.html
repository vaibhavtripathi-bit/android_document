Problem 12

Supporting Multiple Product Variants from a Single Codebase

The same SDK needs to ship as an internal diagnostics tool, an OEM-integrated component, and a development playground — each with different features, UI, permissions, and behaviors. This document describes how Gradle product flavors, build-type-specific source sets, and DI-based flavor abstraction achieve this without code duplication.

The Problem

A single codebase needs to produce multiple app variants:

  • Internal diagnostics variant: Full feature set, FCM push support, binary SMS triggers, internal API keys.
  • OEM variant: Subset of features, different UI, OEM-specific integrations, different API endpoints.
  • Debug builds: Extensive playground UI for testing individual modules, database browsers, crash test screens, WorkManager status viewers.
  • Release builds: No debug UI, no playground, optimized and stripped.

The challenge: 95% of the code is shared. Only the thin outer shell (UI, triggers, keys, DI bindings) differs. Forking the codebase would be unmaintainable. Copy-pasting would lead to drift.

The Architecture

1. Gradle Product Flavors

The app module defines product flavors along a dimension:

  • Each flavor gets its own source set directory (e.g., src/InternalFlavor/, src/OEMFlavor/).
  • Flavor-specific code includes: FCM service implementations, SMS receivers, module enumerations, and constants.
  • The build system merges the flavor source set with the main source set at compile time.
  • Code in the flavor source set overrides or supplements the main source set — same package, different implementation.

2. Build-Type Source Sets (Debug vs Release)

The debug source set contains extensive development tooling:

  • Playground activities: Individual test screens for each module (analytics, coverage, voice, heartbeat, etc.).
  • Database browser: A screen to inspect Room databases at runtime.
  • WorkManager status viewer: Shows all scheduled, running, and completed workers.
  • Crash test screen: Intentionally triggers crashes to test crash handling.
  • Debug ViewModels and utilities: Tools that only make sense during development.

None of this code exists in the release APK — it's compiled only for debug builds.

3. Flavor + Build Type Combination Source Sets

For behavior that varies by both flavor and build type, combined source sets are used:

  • src/InternalFlavorDebug/ — Internal flavor, debug build specific code.
  • src/InternalFlavorRelease/ — Internal flavor, release build specific code.
  • src/OEMFlavorDebug/ — OEM flavor, debug build specific code.
  • src/OEMFlavorRelease/ — OEM flavor, release build specific code.

This 2D matrix (flavor x build type) allows precise control over what code ships in each variant.

4. DI-Based Flavor Abstraction

Rather than scattering if (flavor == "internal") checks throughout the code, flavor-specific behavior is abstracted behind an interface:

  • An IFlavorLogicProvider interface defines flavor-specific operations: SMS handling, file saving preferences, version strings, API keys, etc.
  • Each flavor + build type combination provides its own Hilt module that binds a concrete implementation to this interface.
  • Core code injects IFlavorLogicProvider and calls its methods — it never knows which flavor it's running in.

Key Insight: This is the Strategy pattern enforced by DI. The flavor-specific Hilt module is the only place that knows about the concrete implementation. All other code works with the abstraction. Adding a new flavor means adding a new Hilt module and implementation — no changes to core code.

5. Debug-Only Gradle Dependencies

Playground and development modules are included only in debug builds:

  • debugImplementation dependencies pull in playground modules for each feature.
  • These modules contain test UIs, mock data generators, and development tools.
  • In release builds, these dependencies don't exist — the code isn't even compiled, let alone shipped.
  • This keeps the release APK lean while giving developers rich tooling.

6. Standalone Feature Apps for Isolated Testing

Each feature module has an optional companion *_app module:

  • These are standalone Android apps (@HiltAndroidApp) that include only one feature module and its dependencies.
  • Developers can run and test a single feature in isolation, without the full SDK.
  • This dramatically speeds up development iteration — build times are seconds, not minutes.
  • The standalone app uses the same DI setup as the full app, so behavior is consistent.

7. Shared Build Logic

Common Gradle configuration is extracted to avoid duplication across 50+ modules:

  • A shared common.gradle script defines common Android settings, dependency versions, and build features.
  • A buildSrc directory contains shared build logic and dependency helpers.
  • Helper functions like allCoreModuleDependency() add standard dependencies to feature modules.
  • This ensures consistency across all modules and makes version bumps a single-line change.

Key Takeaways

  • Use product flavors for variant-specific behavior. Don't scatter if checks — use source sets and DI.
  • Abstract flavor differences behind interfaces. Core code should never know which flavor it's running in.
  • Debug tooling belongs in debug source sets. It should never ship to users, and debugImplementation ensures it doesn't.
  • Standalone feature apps accelerate development. Testing one module shouldn't require building the entire SDK.
  • Centralize build configuration. With 50+ modules, duplicated Gradle config becomes a maintenance nightmare.
  • The flavor x build type matrix is powerful. Four source sets (2 flavors x 2 build types) cover most variant needs without code duplication.
