Problem 11

Concurrency Control and Thread Safety in Event-Driven Systems

An event-driven SDK processes system broadcasts, coroutine callbacks, WorkManager jobs, and Flow emissions — all potentially on different threads simultaneously. Without careful concurrency control, counters overflow, state corrupts, and race conditions cause subtle bugs. This document describes the concurrency patterns used to maintain correctness.

The Problem

Consider these real concurrency scenarios:

  • Two screen-on broadcasts arrive simultaneously on different threads. Both check the rate limit counter, both see it's under the limit, both increment it — the limit is exceeded by one.
  • A periodic reset coroutine resets the hourly counter to zero at the exact moment another coroutine is reading and incrementing it — the increment is lost.
  • Multiple SIM subscription change callbacks fire in rapid succession, each spawning a coroutine that reads SIM info — the results interleave and corrupt state.
  • A module's start() and stop() are called from different EventBus subscribers — the module ends up in an inconsistent half-started state.

Approaches Used

1. Kotlin Mutex for Coroutine-Safe Critical Sections

For rate-limit counters and other shared state accessed from coroutines, Kotlin's Mutex provides non-blocking mutual exclusion:

  • A Mutex instance is shared between the event handler and the periodic reset coroutine.
  • The event handler acquires the mutex, reads the counter, checks the limit, increments, and releases.
  • The reset coroutine acquires the same mutex before zeroing the counter.
  • A utility function launchWithMutexLock wraps the acquire-execute-release pattern to prevent forgetting to release.

Key Insight: Kotlin Mutex is a suspending lock, not a blocking one. When a coroutine can't acquire the mutex, it suspends (freeing the thread) rather than blocking. This is critical in coroutine-heavy code — a blocking lock on the IO dispatcher could exhaust the thread pool.

2. Dedicated Thread Pool with Queued Execution

For operations that must execute strictly sequentially (not just mutually exclusively), a custom thread pool with a queue is used:

  • A DedicatedThreadPool wraps a fixed-size executor with a mutex-protected FIFO queue of lambdas.
  • Callers submit work via runQueued(), which adds the lambda to the queue.
  • A processing loop dequeues and executes one lambda at a time.
  • This ensures strict ordering — work items execute in the order they were submitted, even if submitted from different threads.

This is used for operations where not just mutual exclusion but ordering matters — e.g., processing events in the order they occurred.

3. ConcurrentLinkedQueue for Event Ordering

For event triggers that arrive from system broadcasts (which can come on any thread), a ConcurrentLinkedQueue provides thread-safe ordering:

  • Incoming events are added to the queue from any thread (broadcast receivers, callbacks).
  • A single processing coroutine dequeues and handles events one at a time.
  • This converts a multi-threaded event stream into a single-threaded processing pipeline.

4. Debounce for Rapid Event Coalescing

When events come in bursts and only the final state matters, debouncing eliminates redundant processing:

  • Flow debounce: Kotlin Flow's debounce() operator waits for a quiet period before emitting. Used for consent changes (60s window) and lifecycle events (30s window).
  • Job cancellation pattern: For non-Flow contexts, the pattern is: cancel the previous job, launch a new one with a delay. If another event arrives before the delay, the job is cancelled and a new one starts. Only the last event's job completes. Used for SIM subscription changes (5s delay).

5. @Synchronized for Simple Mutual Exclusion

For simpler cases where coroutine suspension isn't needed, Kotlin's @Synchronized annotation provides JVM-level locking:

  • Used on scheduler methods that should not be called concurrently (e.g., scheduling a WorkManager job while another call is already scheduling).
  • Used on the ModuleInitializer.runInitialize() base method to ensure each initializer runs exactly once.
  • Simpler than Mutex but blocks the thread — appropriate only for short critical sections.

6. SharedFlow Buffer and Overflow Strategy

The module event emitter uses a SharedFlow with specific buffer and overflow settings:

  • Buffer size: 64 — events are buffered so slow subscribers don't miss events.
  • Overflow: SUSPEND — if the buffer is full, the emitter suspends rather than dropping events.
  • This ensures no system events are lost, even if a subscriber is temporarily slow (e.g., doing database work).
  • The trade-off: a very slow subscriber can cause backpressure on the emitter. The buffer size (64) provides enough headroom for typical bursts.

7. SupervisorJob for Failure Isolation

When multiple coroutines run under a shared scope, a failure in one can cancel all others. SupervisorJob prevents this:

  • Scopes created with SupervisorJob() allow child coroutines to fail independently.
  • If the voice module's coroutine throws, the coverage module's coroutine keeps running.
  • This is used in the scoped IO scope utility that creates isolated scopes for module work.

Key Takeaways

  • Prefer Mutex over synchronized in coroutine code. Blocking a coroutine thread starves the dispatcher; suspending frees it.
  • Ordering and exclusion are different problems. Mutex gives exclusion; a queue gives ordering. Know which you need.
  • Debounce is a concurrency tool. It's not just for UI — it prevents redundant processing of rapid system events.
  • Buffer your SharedFlows. Without a buffer, slow subscribers cause the emitter to suspend, creating unexpected backpressure.
  • Use SupervisorJob for independent work. One module's failure shouldn't take down the entire coroutine tree.
