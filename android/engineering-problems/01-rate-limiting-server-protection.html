Problem 01

Preventing Server Overload from Client-Side Requests

When an SDK runs on thousands (or millions) of devices, each generating events from screen changes, location updates, network sessions, and system broadcasts — the backend can easily be overwhelmed. This document describes how to throttle, space, and cap client-side event generation to protect server infrastructure.

The Problem

Consider a telemetry SDK installed on millions of devices. Each device reacts to system events — screen on/off, data sessions starting/stopping, location changes, satellite mode toggles. If every event immediately triggers a server request, the backend faces:

  • Thundering herd: A popular OS update causes millions of devices to reboot and fire events simultaneously.
  • Unbounded event rates: A user rapidly toggling screen on/off generates dozens of events per minute.
  • Resource exhaustion: The server's ingestion pipeline saturates, causing cascading failures.
  • Cost explosion: Cloud infrastructure costs scale linearly with request volume.

Approaches Used

1. Config-Driven Minimum Inter-Event Spacing

Instead of allowing every event to fire, the app enforces a minimum time gap between consecutive events of the same type. The gap is derived from a server-provided eventsPerHour configuration value.

How it works:

  • The server sends a config like eventsPerHour: 12 for screen-triggered events.
  • The client computes: minInterval = 3600000ms / 12 = 300000ms (5 minutes).
  • A safety floor (e.g., 1 second) prevents division-by-zero or absurdly small intervals.
  • On each event, the client checks: (now - lastEventTime) > minInterval. If not, the event is silently dropped.
  • A Mutex protects the check-and-update to prevent race conditions from concurrent triggers.

Key Insight: The server controls the rate without requiring an app update. Changing eventsPerHour from 12 to 6 instantly halves the event rate across all devices on next config sync.

Pros: Simple, predictable, server-tunable. No complex state management. Works per-event-type.
Cons: Events are dropped, not deferred. Bursty patterns lose data. Not a true "N per hour" guarantee — it's spacing-based.

2. Hourly Counter with Calendar-Hour Reset

For some event types, a strict "N events per clock hour" cap is more appropriate than spacing. The app maintains a counter that resets at the top of each hour.

How it works:

  • A persistent counter tracks events fired in the current clock hour.
  • When the hour-of-day changes, the counter resets to zero.
  • Each event increments the counter. If counter >= maxPerHour, the event is blocked.
  • The counter is persisted in SharedPreferences to survive process death.
Pros: Guarantees a hard cap per hour. Simple mental model. Survives process restarts.
Cons: Allows bursts at the start of each hour. A device could fire all N events in the first second, then go silent for 59 minutes.

3. Periodic Reset with Mutex-Protected Counters

For features where both the counter and the reset timing need to be precise (e.g., throughput session tracking), a background coroutine handles hourly resets while a Mutex ensures atomic counter operations.

How it works:

  • A coroutine calculates the delay until the next clock-hour boundary and sleeps.
  • On wake, it acquires a Mutex and resets the counter in SharedPreferences.
  • Event producers also acquire the same Mutex before incrementing the counter.
  • This prevents races where a reset and an increment happen simultaneously, which could either lose a count or allow an extra event.

Key Insight: The Mutex solves a subtle bug — without it, a counter read-check-increment sequence is not atomic, and concurrent coroutines can exceed the limit.

4. Cumulative Counters with External Reset Triggers

Some event types use a simpler model: a monotonically increasing counter that only resets when an external signal arrives (e.g., a new config push or a module restart).

How it works:

  • Each trigger type (e.g., data session start, data session stop) has its own persistent counter.
  • On each event, the counter increments. If it exceeds the configured limit, the event is dropped.
  • The counter resets only when the module is explicitly restarted or config changes.
Pros: Extremely simple. No timer management. Good for "fire and forget" limits.
Cons: If the module runs for days without restart, the counter never resets. Requires careful coordination with module lifecycle.

5. Debouncing Rapid Event Streams

For events that naturally come in bursts (e.g., SIM changes, consent toggles), a debounce window collapses multiple rapid events into one.

How it works:

  • Kotlin Flow's debounce() operator is applied with a window (e.g., 60 seconds for consent changes, 5 seconds for SIM subscription changes).
  • Only the last event in the window is processed; earlier ones are discarded.
  • For non-Flow patterns, a "cancel previous job + delay" approach achieves the same effect.

6. Staggered WorkManager Scheduling (Traffic Evening-Out)

When periodic background jobs (config downloads, report uploads) are scheduled, all devices would fire at the same offset from install time. This creates predictable server load spikes.

How it works:

  • An optional "traffic even out" flag adds a random initial delay to the first execution of periodic WorkManager jobs.
  • This spreads the first-run across the scheduling interval, so not all devices hit the server at the same time.
  • Subsequent runs naturally drift apart due to device-specific timing.

Key Insight: Without this, if 100K devices install an update at noon, they'd all try to download config at exactly noon + interval. The random initial delay turns a spike into a smooth curve.

Key Takeaways

  • Server-controlled knobs are essential. Every rate limit is driven by remote configuration, allowing the backend team to tune without app releases.
  • Different event types need different strategies. Screen events use spacing; satellite events use hourly caps; SIM changes use debounce. One size does not fit all.
  • Persistence matters. Counters survive process death via SharedPreferences, so a process restart doesn't reset limits mid-hour.
  • Thread safety is non-negotiable. When multiple coroutines or threads can trigger events, Mutex or synchronized blocks prevent limit overruns.
  • Stagger periodic jobs. Random initial delays prevent thundering herd from synchronized WorkManager schedules.
