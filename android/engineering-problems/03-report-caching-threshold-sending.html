Problem 03

Intelligent Report Caching and Threshold-Based Data Transmission

A telemetry SDK generates thousands of small events daily. Sending each one individually wastes bandwidth and battery. Accumulating too much risks data loss on process death. This document describes a multi-layered caching and batching pipeline that balances efficiency, reliability, and resource constraints.

The Problem

The SDK collects events from 20+ modules — voice call metrics, coverage data, battery stats, network measurements. Each event is a small JSON payload (a few KB). The challenges:

  • Individual sends are wasteful: Each HTTP request has overhead (TLS handshake, headers, connection setup). Sending 1000 tiny payloads is far more expensive than sending 10 batched payloads.
  • Unbounded accumulation is risky: If the app accumulates events in memory and crashes, all unsent data is lost.
  • Storage is finite: The device has limited disk space. Old, unsent reports must be cleaned up.
  • Daily limits exist: Some event types have daily caps (e.g., 1000 analytics events per day) to prevent runaway collection.
  • Oversized payloads crash the app: A single event with a massive payload (e.g., huge cell info dump) can cause OOM when loaded from the database.

1. Two-Table Database Design

The reporting engine uses two Room database tables:

  • Events table: Stores individual ClientSideEvent payloads as JSON strings, along with their byte size. Events without a reportId are "pending" — not yet batched into a file.
  • Reports table: Stores metadata for generated report files — file path, status (NOT_SENT / SENDING / SENT), timestamps.

A separate size-tracking entity maintains running totals of totalEventsSize and totalNumberOfEvents for pending events, avoiding expensive SUM() queries on every insert.

2. Dual-Threshold Batching (Size + Count)

When a new event arrives, the engine checks two thresholds before inserting:

  • Size threshold (1MB): If currentEventSize + totalPendingSize > 1MB, flush all pending events to a JSON file first, then insert the new event.
  • Count threshold (500 events): If totalPendingCount >= 500, flush regardless of size.

This ensures report files are neither too large (risking OOM on read) nor too numerous (creating file management overhead).

Key Insight: The 1MB threshold isn't a "per day" limit — it's a "per batch" limit. It controls how much data accumulates in the database before being flushed to a file. This keeps the DB lean and the files manageable.

3. Oversized Event Handling

If a single event exceeds 1MB (e.g., a massive cell info dump), it bypasses the normal batching pipeline entirely:

  • The event is written directly to its own JSON file without entering the events table.
  • A report entry is created immediately for this single-event file.
  • This prevents the database from storing payloads that would cause OOM when queried.

4. Daily Event Count Limits

Some modules enforce daily caps on event generation (not transmission):

  • A SharedPreferences counter tracks events generated today.
  • A "day indicator" (date string) detects calendar-day rollover and resets the counter.
  • When the counter reaches the configured limit (e.g., 1000 analytics events/day), new events are silently dropped at the source — they never enter the reporting pipeline.

This is a collection-side limit, not a transmission-side limit. It prevents the database from growing unboundedly on high-activity days.

5. Feature-Specific Batching (Bundle Threshold)

Some modules (e.g., analytics) have their own batching layer before events reach the central reporting engine:

  • Raw events are stored in a feature-specific database.
  • When the row count reaches eventsPerBundle (configurable), the module reads up to numEventsBundled rows, wraps them in a single report JSON, and submits one ClientSideEvent to the reporting engine.
  • This two-level batching (feature-level + reporting-level) keeps both the feature DB and the reporting DB manageable.

6. Disk Retention with Size-Based Cleanup

Report files accumulate on disk until successfully sent. To prevent unbounded disk usage:

  • A configurable retentionSize (in MB) sets the maximum total size of report files on disk.
  • Before generating new reports, the engine checks the total folder size.
  • If it exceeds the limit, the oldest files are deleted until the folder is under the cap.
  • This is a FIFO eviction policy — oldest data is sacrificed first.

7. Defensive Database Cleanup

The reporting engine includes several safety mechanisms for database health:

  • Oversized row deletion: On startup, any event rows whose payload exceeds the size limit are deleted to prevent OOM on future queries.
  • Stuck status reset: Report entries stuck in "SENDING" status (from a previous crash mid-upload) are reset to "NOT_SENT" for retry.
  • Bad JSON handling: In release builds, events that fail JSON serialization are silently deleted rather than crashing the app. In debug builds, they throw to surface the bug.

8. Periodic Upload via WorkManager

Report files are not sent immediately. A periodic WorkManager job handles transmission:

  • The worker runs at a configurable interval (e.g., every few hours).
  • It skips the first ~15 minutes after scheduling to allow initial data accumulation.
  • It requests reports from all modules, flushes any remaining pending events to files, then iterates all NOT_SENT report files and POSTs them sequentially.
  • Network connectivity is a WorkManager constraint — the job won't run without a network connection.

Key Takeaways

  • Batch at multiple levels. Feature-level batching reduces DB pressure; reporting-level batching reduces network calls. Two layers are better than one.
  • Track running totals, don't query. Maintaining a separate size-tracking entity avoids expensive SUM() queries on every insert.
  • Handle the extremes. Single events > 1MB get special treatment. Empty payloads get filtered. The pipeline must handle both ends.
  • Separate collection limits from transmission limits. Daily event caps prevent runaway collection; size-based batching controls transmission efficiency.
  • Be defensive about database health. Oversized rows, stuck statuses, and bad JSON are all real failure modes that need explicit handling.
  • FIFO eviction is the safest default. When disk space runs out, the oldest data is least valuable. Delete it first.
