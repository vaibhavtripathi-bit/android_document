Interview Q&A — Large Screen & Foldables

20 questions from Google, Flipkart, and Amazon — with model L6 answers

Section A — WindowSizeClass & Breakpoints

1
What are the three WindowSizeClass breakpoints and what navigation pattern should you use for each? GoogleL5

Compact (<600dp): Phone portrait. Use BottomNavigationBar — thumb-reachable, familiar to phone users.

Medium (600–840dp): Phone landscape, 7" tablet, foldable outer screen. Use NavigationRail — vertical rail on the left, provides more horizontal space for content.

Expanded (>840dp): 10"+ tablet, unfolded foldable, desktop. Use a persistent NavigationDrawer — always visible, no need to open/close. Content sits to the right of it.

In Compose: NavigationSuiteScaffold handles this automatically from one composable. In Views: switch between Bottom Nav, Rail, and Drawer fragments based on the computed size class.

2
Why is Configuration.screenWidthDp wrong for layout decisions on large screens? GoogleL5

Configuration.screenWidthDp reports the screen width — the full physical display. In split-screen mode on a 1200dp tablet, your app's window might be 600dp wide, but screenWidthDp still returns 1200dp. Using it would make your app show a two-pane tablet layout in a 600dp window, causing content to overflow and clip.

The correct API is WindowMetricsCalculator.getOrCreate().computeCurrentWindowMetrics(activity) which returns the actual window bounds. In Compose, calculateWindowSizeClass(activity) does this automatically.

3
A Pixel Fold is unfolded. The user enters split-screen with another app. How does your WindowSizeClass change and how does your UI adapt? GoogleL6

Unfolded Pixel Fold inner screen ≈ 1840 × 2160px → ~840dp wide → Expanded. Two-pane layout, persistent nav drawer.

When the user enters split-screen 50/50, your window is now ~420dp wide → Compact. Your layout should automatically collapse to single-pane with bottom navigation.

This works correctly only if you observe WindowMetricsCalculator (window width, not screen width). In Compose, calculateWindowSizeClass recomposes automatically when the window is resized. In Views, onConfigurationChanged fires and you re-query window metrics.

The user dragging the split-screen divider triggers continuous onConfigurationChanged calls — your layout should fluidly transition between size classes as the divider moves.

Section B — Foldable Postures

4
What is FoldingFeature.isSeparating and why should you use it instead of checking state == FLAT? GoogleL6

isSeparating = true means the OEM considers the hinge a logical boundary that divides the screen into two distinct areas. It's the signal to show a two-pane layout split at the hinge.

state == FLAT means the device is fully unfolded — but some FLAT foldables still have a visible crease that should be respected as a separator. Others may not. The OEM sets isSeparating based on the actual physical and UX characteristics of their device.

Rule: use isSeparating to decide two-pane split. Use state to know the physical posture (for tabletop layout decisions). Use orientation to know which axis the hinge runs along.

5
A user is using a Galaxy Z Flip (clamshell) in tabletop mode — half-open horizontally. How should your video player app behave? GoogleL6

Detect: FoldingFeature.state == HALF_OPENED and orientation == HORIZONTAL.

The device sits on a table with the hinge as a natural fulcrum. The user looks at the top half and touches the bottom half. Ideal layout:

  • Top half (above hinge at foldingFeature.bounds.top): Video player — full-width PlayerView with no overlapping UI.
  • Bottom half (below hinge): Playback controls, title, episode list, progress bar.

This matches the physical ergonomics — no need to hold the device, controls are at finger level. Also disable auto-rotate when in tabletop mode (requestedOrientation = SCREEN_ORIENTATION_LOCKED) since orientation doesn't make sense here.

6
What is occlusionType == FULL and what must you do when you detect it? GoogleL6

occlusionType == FULL means the hinge physically covers part of the display — pixels in foldingFeature.bounds are not visible to the user. This occurred on some early foldable devices (e.g., original Galaxy Fold) where the hinge mechanism covered a strip of the screen.

You must ensure no interactive or important content is placed within foldingFeature.bounds. Use the bounds rect to set padding/margin or to position the divider in a two-pane layout so that content sits on either side of the occluded area.

Modern foldables (Pixel Fold, Z Fold 6) have occlusionType == NONE — the hinge is visible but all pixels are usable. The crease is cosmetic. Still avoid placing primary content directly over it.

Section C — Adaptive Layouts

7
Walk me through implementing a Gmail-style list-detail layout that works on both phone and tablet. GoogleDesignL6

Compose approach with ListDetailPaneScaffold:

  1. Use rememberListDetailPaneScaffoldNavigator<EmailId>() to manage pane navigation state.
  2. In ListDetailPaneScaffold, pass listPane and detailPane composables.
  3. On Compact (phone): scaffold shows only one pane at a time. Tapping an email calls navigator.navigateTo(Detail, emailId) — detail fills the screen. Back press calls navigator.navigateBack() via BackHandler.
  4. On Expanded (tablet): scaffold shows both panes simultaneously, split at roughly 40/60. Selecting an email updates the detail pane without navigation.

View approach: SlidingPaneLayout with list fragment (320dp min) and detail fragment (480dp min). System stacks or slides them based on available width. Add SlidingPaneLayout.SimpleSlidingPaneLayout.BackPressedCallback for back navigation.

8
How do you make a product grid (like Flipkart's home) responsive across phone, tablet, and foldable? FlipkartL5

Use LazyVerticalGrid with GridCells.Adaptive(minSize = 180.dp). This automatically calculates the number of columns as floor(windowWidth / 180dp):

  • Phone 360dp → 2 columns
  • Tablet 720dp → 4 columns
  • Foldable 1080dp → 6 columns

Never hardcode column count. Also adjust card content padding and image aspect ratio to look good at all sizes. For the card itself, use Modifier.fillMaxWidth() — never a fixed width.

For headers and banners, use LazyVerticalGrid with span { GridItemSpan(maxLineSpan) } to make them full-width regardless of column count.

9
What is NavigationSuiteScaffold and what does it do internally? GoogleL6

NavigationSuiteScaffold is a Material3 Adaptive composable that automatically switches navigation component based on WindowAdaptiveInfo (window size + fold state):

  • Compact width → NavigationBar (bottom)
  • Medium width OR compact height → NavigationRail (left side)
  • Expanded width → NavigationDrawer (persistent left panel)

Internally it calls NavigationSuiteDefaults.calculateFromAdaptiveInfo(windowAdaptiveInfo) to determine the NavigationSuiteType. You can override this via the layoutType parameter if your design differs from the default. The items are declared once via a DSL and rendered into whichever component is active — no code duplication.

Section D — Multi-Window & Resizability

10
What changed in Android 12L regarding resizeableActivity="false"? GoogleL5

Before 12L, resizeableActivity="false" caused the app to run in letterbox/pillarbox mode — the system showed the app at a fixed aspect ratio with black bars. Users could not resize the window.

On Android 12L+, Google changed this: the system may still resize the app on large-screen devices, ignoring the flag. The system may also apply compatibility mode (showing the app in a fixed-size compatibility window), but apps are expected to handle size changes gracefully. Setting false is no longer a reliable way to avoid resize.

The practical implication: every app targeting Android 12L+ on tablets must handle configuration changes and window size changes correctly, regardless of the manifest flag.

11
What is multi-resume and how does it affect camera and audio focus in split-screen? GoogleL6

Multi-resume (Android 10+): all visible windows in split-screen can be in the RESUMED state simultaneously. Previously, only the focused window was RESUMED; the other was PAUSED.

Camera: Only one app can use the camera at a time. When both windows are RESUMED, the system uses ExclusiveAccessPreference — the app that most recently received focus gets camera access. When you lose camera access, CameraDevice.StateCallback.onDisconnected() is called. Re-acquire when focus returns.

Audio focus: Both apps can hold audio focus. The system uses AudioFocus policies. ExoPlayer with handleAudioFocus = true correctly requests/releases focus on play()/pause(). Two video apps in split-screen will compete for focus — the most recently played one wins.

12
How do you implement drag-and-drop from your app to another app in split-screen? What permission is required? AmazonL6

Use the Compose Modifier.dragAndDropSource API (or View's View.startDragAndDrop()). Build a DragAndDropTransferData with a ClipData containing the content URI.

Required flags:

  • DRAG_FLAG_GLOBAL — allows drag to cross app boundaries.
  • DRAG_FLAG_GLOBAL_URI_READ — grants the receiving app temporary read permission on the content URI. Without this, the drop target gets a SecurityException when it tries to open the URI.
  • Add DRAG_FLAG_GLOBAL_URI_WRITE if the receiver needs write access.

The system grants this URI permission automatically for the duration of the drag — you don't need to call grantUriPermission() manually.

Section E — System Design & Architecture

13
Design the Flipkart product detail screen to work well on phones, tablets, and foldables. FlipkartDesignL6

Compact (phone): Vertical scroll — image gallery at top, product title, price, ratings, then CTA buttons, then description, then reviews. Bottom persistent bar with "Add to Cart" + "Buy Now".

Medium (phone landscape / small tablet): Two-column — left column: image gallery. Right column: title, price, CTA buttons. Below: description and reviews span full width. NavigationRail for category navigation.

Expanded (tablet / foldable unfolded): Three-column — left: navigation rail + category tree. Middle: image gallery + descriptions. Right: sticky purchase panel (price, size selection, CTA). No need to scroll to see the buy button.

Foldable tabletop: If HALF_OPENED + HORIZONTAL, show product images on top, purchase panel on bottom.

Implementation: single composable tree that reads WindowSizeClass. No separate layouts or Activities. All panes share the same ViewModel — state is device-agnostic.

14
How would you test that your app correctly shows a two-pane layout on a tablet and single-pane on a phone, without physical devices? GoogleL5

Unit / Compose tests: Use WindowSizeClass.calculateFromSize(DpSize(...)) to inject fake window sizes. Pass the computed class directly to your composable (hoist it as a parameter, not computed inside).

// Inject compact size → assert single pane
val compact = WindowSizeClass.calculateFromSize(DpSize(360.dp, 800.dp))
composeTestRule.setContent { MyScreen(windowSizeClass = compact) }
composeTestRule.onNodeWithTag("detail_pane").assertDoesNotExist()

// Inject expanded size → assert two panes
val expanded = WindowSizeClass.calculateFromSize(DpSize(1000.dp, 800.dp))
composeTestRule.setContent { MyScreen(windowSizeClass = expanded) }
composeTestRule.onNodeWithTag("detail_pane").assertIsDisplayed()

Emulator: Use the Resizable emulator in Android Studio — supports phone, unfolded, tablet, and desktop modes from one AVD. Also use the Pixel Fold emulator for fold/unfold testing.

Foldable state tests: Inject fake FoldingFeature via TestWindowInfoTracker.overrideDecorator().

Section F — Quick-Fire

15
What is the Medium WindowSizeClass typically used for?

Phone in landscape orientation, 7-inch tablets, and the outer cover screen of book-fold devices. The recommended navigation pattern is NavigationRail — a vertical icon bar on the left side. Provides more horizontal content space than BottomNavigationBar while not using as much width as a full NavigationDrawer.

16
What does GridCells.Adaptive(minSize = 180.dp) do?

Calculates the maximum number of columns that fit in the current window width with each column at least 180dp wide. Columns expand to fill any remaining space. On a 360dp phone: 2 columns. On a 720dp tablet: 4 columns. Automatically responsive with no code changes needed when the window size changes.

17
What is WindowCompat.setDecorFitsSystemWindows(window, false) and why do you need it?

Tells the system not to fit window content within system bar insets automatically — the app draws edge-to-edge behind status bar and navigation bar. Required for tablets and large screens to avoid awkward white bars. You then consume insets manually via Modifier.windowInsetsPadding() or Scaffold contentWindowInsets to ensure content isn't obscured.

18
What configChanges must you declare to avoid Activity recreation on window resize?

orientation|screenSize|screenLayout|smallestScreenSize|density. Declaring these tells the system to call onConfigurationChanged() instead of recreating the Activity when these parameters change — which happens continuously as the user drags the split-screen divider.

19
When does FoldingFeature.state == HALF_OPENED combined with orientation == HORIZONTAL occur?

On a clamshell foldable (like Galaxy Z Flip) when the user half-opens it and sets it on a surface — "tabletop mode". The hinge runs horizontally (left-to-right). The top half faces the user (content area), the bottom half lies flat (control area). Ideal for video playback, video calls, or camera with controls below the hinge.

20
Name the three canonical large-screen layouts recommended by Google.
  • List-Detail — master list on left, detail on right. E.g., Gmail, Settings, Files.
  • Supporting Pane — main content in centre/left, supplementary panel on right. E.g., Google Maps + search results panel.
  • Feed — multi-column grid of cards expanding to fill available width. E.g., Google Photos, News, Flipkart home.
