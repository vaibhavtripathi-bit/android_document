Multi-Window, Drag & Drop, Resizability

Split-screen, freeform mode, configChanges, drag-and-drop API, resizeableActivity — what changes in a multi-window world

2. Declaring Resizability

<!-- AndroidManifest.xml -->
<activity
    android:name=".MainActivity"
    android:resizeableActivity="true"   <!-- opt-in to multi-window and freeform -->
    android:configChanges="orientation|screenSize|screenLayout|smallestScreenSize|density">

    <!-- Minimum size constraint for freeform mode -->
    <layout
        android:minWidth="320dp"
        android:minHeight="480dp"
        android:defaultWidth="600dp"
        android:defaultHeight="800dp"
        android:gravity="center" />
</activity>
Android 12L change
On Android 12L and later, all apps on large-screen devices are treated as resizable regardless of the resizeableActivity flag. Setting it to false used to letterbox the app; now the system may still resize it. Your app must handle size changes gracefully — you cannot opt out on large-screen Android 12L+.

3. Handling Configuration Changes Without Recreation

In multi-window mode, the user can drag the divider to resize your window at any time. Each resize triggers a configuration change. If you don't handle it, the Activity is recreated — losing unsaved state and causing a visible flash.

// Handle these config changes in-process (no Activity recreation)
android:configChanges="orientation|screenSize|screenLayout|smallestScreenSize|density"

// Then in your Activity:
override fun onConfigurationChanged(newConfig: Configuration) {
    super.onConfigurationChanged(newConfig)
    // If using Compose: WindowSizeClass State updates automatically
    // If using Views: re-query WindowMetricsCalculator and update layout
    val windowMetrics = WindowMetricsCalculator.getOrCreate()
        .computeCurrentWindowMetrics(this)
    val widthDp = windowMetrics.bounds.width() / resources.displayMetrics.density
    updateLayoutForWidth(widthDp)
}
L6 Insight — Compose wins here
In Compose, calculateWindowSizeClass is backed by a State that recomposes automatically when the window changes — even without Activity recreation. In a View-based app you must manually call computeCurrentWindowMetrics in onConfigurationChanged and re-lay out. This is a concrete advantage of Compose for large-screen work.

4. Lifecycle in Multi-Window

Multi-window changes the resumed/paused contract. Prior to Android 10, only the focused window was RESUMED. Since Android 10 (multi-resume), all visible windows can be in the RESUMED state simultaneously.

Android versionResumed windowsImpact
< Android 10Only the focused windowCamera, microphone only accessible in focused window
Android 10+All visible windows (multi-resume)Camera/mic access via ExclusiveAccessPreference; media playback continues in non-focused window
// Respond to focus changes in multi-window
override fun onWindowFocusChanged(hasFocus: Boolean) {
    super.onWindowFocusChanged(hasFocus)
    if (hasFocus) {
        // Regained focus — resume camera preview, re-request audio focus
        resumeCamera()
    } else {
        // Lost focus (other window active) — pause camera in API < 29
        if (Build.VERSION.SDK_INT < Build.VERSION_CODES.Q) {
            pauseCamera()
        }
        // On API 29+: camera can remain active in non-focused window
    }
}

5. Drag and Drop

Drag-and-drop across apps in split-screen is supported since Android 7.0. Your app can be both a drag source and a drop target.

Composable as Drop Target

// Compose drop target
@Composable
fun DropTargetBox(onFilesDropped: (List<Uri>) -> Unit) {
    var isDragOver by remember { mutableStateOf(false) }

    Box(
        modifier = Modifier
            .fillMaxSize()
            .background(if (isDragOver) Color.Blue.copy(alpha = 0.1f) else Color.Transparent)
            .dragAndDropTarget(
                shouldStartDragAndDrop = { event ->
                    // Accept only image URIs dragged from other apps
                    event.mimeTypes().any { it.startsWith("image/") }
                },
                target = object : DragAndDropTarget {
                    override fun onStarted(event: DragAndDropEvent) {
                        isDragOver = true
                    }
                    override fun onEnded(event: DragAndDropEvent) {
                        isDragOver = false
                    }
                    override fun onDrop(event: DragAndDropEvent): Boolean {
                        val clipData = event.toAndroidDragEvent().clipData
                        val uris = (0 until clipData.itemCount).map {
                            clipData.getItemAt(it).uri
                        }
                        onFilesDropped(uris)
                        return true  // consumed
                    }
                }
            )
    ) {
        Text("Drop images here")
    }
}

Starting a Drag from a Composable

@Composable
fun DraggableImage(imageUri: Uri) {
    val view = LocalView.current
    Image(
        painter = rememberAsyncImagePainter(imageUri),
        contentDescription = null,
        modifier = Modifier
            .size(120.dp)
            .dragAndDropSource {
                detectTapGestures(onLongPress = {
                    startTransfer(
                        DragAndDropTransferData(
                            clipData = ClipData.newUri(
                                view.context.contentResolver,
                                "Image",
                                imageUri
                            ),
                            flags = View.DRAG_FLAG_GLOBAL or  // allow cross-app drag
                                    View.DRAG_FLAG_GLOBAL_URI_READ  // grant read permission
                        )
                    )
                })
            }
    )
}
Cross-app drag permissions
When dragging a content URI to another app, you must include DRAG_FLAG_GLOBAL_URI_READ (and/or DRAG_FLAG_GLOBAL_URI_WRITE). Without these flags the receiving app cannot read the URI — it will get a SecurityException. The system temporarily grants URI permission to the drop target for the duration of the drop.

6. Picture-in-Picture (PiP)

// Enter PiP — typically on home button press during video playback
override fun onUserLeaveHint() {
    if (player.isPlaying) {
        enterPipMode()
    }
}

private fun enterPipMode() {
    val aspectRatio = Rational(16, 9)
    val params = PictureInPictureParams.Builder()
        .setAspectRatio(aspectRatio)
        .setSeamlessResizeEnabled(true)  // API 31+: smooth resize animation
        .setActions(listOf(
            buildPipAction(R.drawable.ic_pause, "Pause", ACTION_PAUSE),
            buildPipAction(R.drawable.ic_skip_next, "Next", ACTION_NEXT)
        ))
        .build()
    enterPictureInPictureMode(params)
}

// Respond to PiP state changes
override fun onPictureInPictureModeChanged(
    isInPictureInPictureMode: Boolean,
    newConfig: Configuration
) {
    super.onPictureInPictureModeChanged(isInPictureInPictureMode, newConfig)
    if (isInPictureInPictureMode) {
        // Hide all UI except PlayerView
        binding.controls.visibility = View.GONE
    } else {
        // Restore full UI
        binding.controls.visibility = View.VISIBLE
    }
}

Interview Cheat Sheet — Multi-Window

  • Three multi-window modes? — Split-screen (API 24), Freeform (API 24 opt-in / always-on Android 12L+), Picture-in-Picture (API 26).
  • resizeableActivity="false" on 12L? — Ignored for large-screen devices. System may resize anyway. Must handle size changes gracefully.
  • Minimum configChanges to declare? — orientation|screenSize|screenLayout|smallestScreenSize|density. Prevents Activity recreation on window resize.
  • Multi-resume (Android 10+)? — All visible windows can be RESUMED simultaneously. Camera/mic use ExclusiveAccessPreference. Media continues playing in non-focused window.
  • Cross-app drag permission? — Must set DRAG_FLAG_GLOBAL | DRAG_FLAG_GLOBAL_URI_READ. System grants temporary URI permission to the drop target.
  • PiP entry point? — Override onUserLeaveHint() to enter PiP when home is pressed during playback. Update UI in onPictureInPictureModeChanged().
  • Freeform minimum size? — Declare in <layout android:minWidth android:minHeight> inside the activity manifest element.
