WindowSizeClass & Responsive Design

The canonical breakpoint system for Android — how to observe, classify, and react to window size changes

1. The Problem WindowSizeClass Solves

Before WindowSizeClass, developers wrote code like:

// Bad — hardcoded pixel checks, breaks on all non-standard screens
if (resources.displayMetrics.widthPixels > 1200) { showTwoPane() }

This fails on foldables (the window changes size without a config change), on desktop mode (arbitrary window width), and on tablets with unusual DPI. WindowSizeClass provides stable, meaningful breakpoints based on dp width/height of the current window — not the screen.

2. The Three Size Classes

ClassWidth thresholdTypical scenarioRecommended layout
Compact< 600dpPhone portrait, outer screen of foldableSingle column, BottomNavigationView
Medium600–840dpPhone landscape, 7" tablet, foldable half-openNavigationRail, optional side panel
Expanded> 840dpUnfolded foldable, 10"+ tablet, desktop windowPersistent NavigationDrawer, list-detail

Both width and height have their own WindowSizeClass. Most layout decisions are driven by width (horizontal space). Height class is used for decisions like whether to show a bottom bar vs a side rail.

3. Observing WindowSizeClass in Compose

// Add dependency: androidx.compose.material3.adaptive:adaptive
@Composable
fun MyApp() {
    val windowSizeClass = calculateWindowSizeClass(activity = LocalActivity.current)

    when (windowSizeClass.widthSizeClass) {
        WindowWidthSizeClass.Compact -> {
            // Phone layout
            CompactLayout()
        }
        WindowWidthSizeClass.Medium -> {
            // Tablet / landscape layout
            MediumLayout()
        }
        WindowWidthSizeClass.Expanded -> {
            // Large tablet / foldable unfolded
            ExpandedLayout()
        }
    }
}
L6 Insight — calculateWindowSizeClass is Activity-scoped
calculateWindowSizeClass measures the current window, not the screen. In multi-window / split-screen mode, two apps share the screen — each gets a smaller window. Your app correctly adapts because it reads its own window size, not the full display size. This is why you must never use Display.getRealSize() for layout decisions.

4. Observing in a View-based Activity

class MainActivity : AppCompatActivity() {

    override fun onCreate(savedInstanceState: Bundle?) {
        super.onCreate(savedInstanceState)

        // Collect window size changes as a Flow
        lifecycleScope.launch {
            lifecycle.repeatOnLifecycle(Lifecycle.State.STARTED) {
                WindowInfoTracker.getOrCreate(this@MainActivity)
                    .windowLayoutInfo(this@MainActivity)
                    .collect { layoutInfo ->
                        val windowMetrics = WindowMetricsCalculator
                            .getOrCreate()
                            .computeCurrentWindowMetrics(this@MainActivity)

                        val widthDp = windowMetrics.bounds.width() /
                            resources.displayMetrics.density

                        val sizeClass = WindowWidthSizeClass.compute(widthDp)
                        updateLayoutForSizeClass(sizeClass)
                    }
            }
        }
    }

    private fun updateLayoutForSizeClass(sizeClass: WindowWidthSizeClass) {
        when (sizeClass) {
            WindowWidthSizeClass.COMPACT  -> showBottomNav()
            WindowWidthSizeClass.MEDIUM   -> showNavigationRail()
            WindowWidthSizeClass.EXPANDED -> showNavigationDrawer()
        }
    }
}

5. Why NOT Use Configuration.screenWidthDp

Common Mistake
Configuration.screenWidthDp reports the screen width, not the window width. In split-screen mode, a tablet might report 1200dp screen width but your window is only 600dp. Using screen width causes your app to show a two-pane layout in a 600dp window — UI overflow and clipping guaranteed. Always use WindowMetricsCalculator or calculateWindowSizeClass.

6. Reacting to Size Changes Without Recreation

On foldables, unfolding the device changes the window size but may not trigger an Activity recreation (depending on android:configChanges). The correct pattern:

// AndroidManifest.xml — handle these config changes without recreation
<activity
    android:name=".MainActivity"
    android:configChanges="orientation|screenSize|screenLayout|smallestScreenSize">
</activity>
// In Activity — override to handle the change
override fun onConfigurationChanged(newConfig: Configuration) {
    super.onConfigurationChanged(newConfig)
    // WindowInfoTracker flow will emit automatically — no manual work needed
    // if you're using the flow pattern above
}
L6 Insight — Compose handles this automatically
In Compose, calculateWindowSizeClass returns a State-backed value. When the window size changes (fold/unfold, split-screen resize), it automatically recomposes. You don't need to handle onConfigurationChanged manually for layout decisions. This is one of the key advantages of Compose for large-screen development.

7. Testing WindowSizeClass Without a Physical Device

// In Compose tests — provide a fake window size
@Test
fun compactLayout_showsBottomNav() {
    composeTestRule.setContent {
        // Simulate Compact window
        val fakeWindowSizeClass = WindowSizeClass.calculateFromSize(DpSize(400.dp, 800.dp))
        MyApp(windowSizeClass = fakeWindowSizeClass)
    }
    composeTestRule.onNodeWithTag("bottom_nav").assertIsDisplayed()
}

@Test
fun expandedLayout_showsNavDrawer() {
    composeTestRule.setContent {
        val fakeWindowSizeClass = WindowSizeClass.calculateFromSize(DpSize(1000.dp, 800.dp))
        MyApp(windowSizeClass = fakeWindowSizeClass)
    }
    composeTestRule.onNodeWithTag("nav_drawer").assertIsDisplayed()
}

Also use the Android Studio Resizable emulator — it supports phone, unfolded foldable, tablet, and desktop modes from one AVD.

Interview Cheat Sheet — WindowSizeClass

  • Three classes? — Compact (<600dp), Medium (600–840dp), Expanded (>840dp). Based on window width in dp, not screen or pixels.
  • Why not Configuration.screenWidthDp? — Reports screen width, not window width. Incorrect in split-screen and multi-window. Use WindowMetricsCalculator instead.
  • How to observe in Compose? — calculateWindowSizeClass(activity) returns a reactive State. Recomposes automatically on fold/unfold.
  • How to observe in Views? — WindowInfoTracker.windowLayoutInfo(activity).collect { } in a lifecycle-aware coroutine.
  • Navigation pattern per class? — Compact → BottomNavigationView. Medium → NavigationRail. Expanded → NavigationDrawer (persistent).
  • Foldable size change without recreation? — Declare screenLayout|smallestScreenSize in configChanges. Compose reacts automatically via State. View-based apps use the WindowInfoTracker flow.
  • Testing? — WindowSizeClass.calculateFromSize(DpSize(...)) in Compose tests. Resizable emulator in Android Studio.
