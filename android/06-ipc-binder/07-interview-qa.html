07 — IPC & Binder Interview Q&A

53 L6 Staff Engineer questions covering Binder architecture, call flow, AIDL, Zygote, system services, IPC performance, failure scenarios, and system design

Binder Architecture & Kernel Internals

Q1. Why did Android choose Binder over Unix sockets or shared memory for IPC?

Binder provides five properties no single Linux IPC mechanism offers: (1) One-copy performance — sender's data is copied directly into the receiver's mmap'd buffer (vs two copies for sockets). (2) Kernel-enforced security — caller UID/PID are set by the kernel from current->cred, unforgeable from userspace. (3) Object references — IBinder handles can be passed between processes; the kernel translates them transparently. (4) Synchronous RPC — built-in call-and-wait semantics with reply. (5) Death notifications — the kernel notifies clients when a remote process dies. Shared memory is zero-copy but has no synchronization or security. Unix sockets have two copies and no object references.

Q2. Explain the one-copy optimization in Binder. How does mmap make it possible?

When a process opens /dev/binder, it calls mmap() to map ~1MB of kernel-managed physical pages into its virtual address space (read-only). These same physical pages are also mapped into the kernel's address space (read-write). When a sender calls ioctl(BINDER_WRITE_READ), the kernel performs copy_from_user() to copy the sender's data directly into the receiver's mmap'd pages. The receiver reads from its mapped region with no second copy. Traditional IPC requires copy_from_user() into a kernel buffer, then copy_to_user() into the receiver — two copies. Binder eliminates the second copy because the kernel buffer IS the receiver's buffer.

Q3. What are the key kernel data structures in the Binder driver?

binder_proc: one per process, holds red-black trees of threads, nodes (hosted services), and refs (remote references), plus the mmap allocator. binder_thread: one per thread that enters the driver, has its own todo list and transaction stack. binder_node: represents a BBinder (service object) hosted in a process, with a pointer to the userspace object and a list of references. binder_ref: a handle to a remote binder_node, with a descriptor number (handle 0 = ServiceManager). binder_transaction: represents an in-flight transaction with source/target threads, buffer pointer, and data size. All stored in red-black trees for O(log n) lookup.

Q4. Why can't an app forge Binder.getCallingUid()?

In binder_transaction() (kernel space), the driver copies the caller's UID/PID from current->cred — the kernel's process credential structure set during fork() via setuid(). Since credentials live in kernel memory, no userspace code can modify them. The chain is: app → ioctl(BINDER_WRITE_READ) → kernel reads current->cred->uid → writes it into the transaction → receiver reads it via Binder.getCallingUid(). This kernel-enforced identity is the foundation of all Android permission checks and IPC security.

Q5. What is the 1MB Binder buffer limit? What happens when it's exceeded?

Each process mmaps ~1MB (1,048,576 - 8,192 bytes) for Binder transactions, set in ProcessState::init(). This buffer is shared across ALL concurrent incoming transactions. Exceeding it causes TransactionTooLargeException. It can be triggered by: (1) a single large Parcel (>1MB), or (2) many small pending transactions (e.g., rapid oneway callbacks) whose cumulative size exceeds the buffer. The limit cannot be changed by apps. For large data, use SharedMemory, content URIs, or CursorWindow (which uses ashmem internally).

Q6. Explain the three Binder domains: /dev/binder, /dev/hwbinder, /dev/vndbinder.

Introduced with Project Treble (Android 8) to separate framework from vendor: /dev/binder — framework Binder for apps and system_server, ServiceManager at handle 0. /dev/hwbinder — hardware Binder for HIDL HAL interfaces, HwServiceManager at handle 0. /dev/vndbinder — vendor Binder for vendor-to-vendor IPC, VndServiceManager at handle 0. Each has its own mmap space, thread pool, and SELinux policies. This separation allows framework updates without breaking vendor HALs. Android 11+ migrates HALs from HIDL (hwbinder) to AIDL (regular binder), phasing out hwbinder.

Call Flow, Proxy/Stub & Thread Pools

Q7. Trace a synchronous Binder call from client to server and back.

Client: (1) Proxy marshals args into Parcel, (2) calls BinderProxy.transact() → JNI → BpBinder::transact() → IPCThreadState::transact() → ioctl(BINDER_WRITE_READ) with BC_TRANSACTION. Kernel: (3) binder_transaction() finds target node, copies caller UID/PID, allocates buffer in target's mmap, copy_from_user(), translates IBinder objects, enqueues BR_TRANSACTION on target, wakes target thread, (4) caller sleeps. Server: (5) Binder thread wakes, IPCThreadState::executeCommand() → BBinder::transact() → JNI → Stub.onTransact(), (6) dispatches by transaction code, executes method, writes result to reply Parcel, (7) ioctl() with BC_REPLY. Kernel: (8) copies reply to caller's mmap, wakes caller. Client: (9) reads reply, returns result.

Q8. What is the Binder thread pool? How many threads does an app get?

Each process has a pool of threads dedicated to handling incoming Binder transactions. Default: 15 spawnable threads + 1 main Binder thread = 16 total. Set via BINDER_SET_MAX_THREADS ioctl. Threads are spawned on demand: when all threads are busy and a new transaction arrives, the kernel sends BR_SPAWN_LOOPER. Each thread loops in IPCThreadState::joinThreadPool(), blocking on ioctl(BINDER_WRITE_READ). These are separate from the app's UI thread, coroutine dispatchers, and worker threads. system_server increases its pool to 31 via BinderInternal.setMaxThreads(31).

Q9. Compare synchronous and oneway Binder transactions.

Synchronous: client blocks until server replies, has a reply Parcel, exceptions propagate via RemoteException, sequential per-thread ordering. Oneway: client returns immediately, no reply Parcel, exceptions are silently swallowed, FIFO ordering per binder_node but not across nodes. Oneway creates buffer pressure because transactions queue up if the receiver is slow. Use oneway for callbacks, notifications, fire-and-forget. Use synchronous for queries and operations where failure must be reported. Oneway is declared with the oneway keyword in AIDL (per-interface or per-method).

Q10. How does Stub.asInterface() optimize same-process calls?

It calls queryLocalInterface(DESCRIPTOR) on the IBinder. If the IBinder is a local Binder (same process), it returns the Stub directly — no Proxy, no Parcel, no kernel, no IPC. The method call becomes a direct Java method invocation. This means: (1) same-process bound services have near-zero overhead, (2) your AIDL implementation may run on the caller's thread (not a Binder thread) in the same-process case, (3) you can't assume Binder.getCallingUid() returns a remote UID — it returns your own UID for local calls.

Q11. How do DeathRecipient notifications work internally?

Client calls linkToDeath() → sends BC_REQUEST_DEATH_NOTIFICATION to the kernel → kernel adds a binder_ref_death to the binder_ref. When the target process exits, binder_release() frees its binder_proc, iterates all binder_nodes, finds all refs with death notifications, and sends BR_DEAD_BINDER to each registered process. The framework calls DeathRecipient.binderDied() on a Binder thread (not the main thread). Race condition: the remote process can die between receiving the IBinder and calling linkToDeath() — always check pingBinder() after linking.

Q12. What is Binder.clearCallingIdentity() and why is forgetting to restore it dangerous?

It saves the caller's UID/PID (packed into a long) and replaces them with the current process's own UID/PID. Used when a system service needs to perform privileged operations on behalf of a caller after verifying permissions. restoreCallingIdentity() reverses it. Forgetting to restore is a privilege escalation vulnerability: subsequent permission checks would use the server's UID (e.g., system_server's UID 1000) instead of the actual caller's UID, bypassing all permission checks. Must always use try/finally. This has been the root cause of multiple Android CVEs.

AIDL Internals

Q13. What does the AIDL compiler generate? Walk through the key classes.

For IMyService.aidl, the compiler generates: (1) IMyService interface extending IInterface with the declared methods. (2) IMyService.Stub abstract class extending Binder, implementing IMyService, with onTransact() that dispatches by transaction code, and asInterface() for local/remote detection. (3) IMyService.Stub.Proxy implementing IMyService, wrapping a BinderProxy, marshalling each method's args into a Parcel, calling transact(), and unmarshalling the reply. Transaction codes are sequential integers starting from IBinder.FIRST_CALL_TRANSACTION.

Q14. Explain AIDL direction tags: in, out, inout. What are the performance implications?

in (default): data sent client→server only, 1 serialization. out: data sent server→client only (in reply), 1 serialization. inout: data sent both ways, 2 serializations (client writes + server writes reply). Always use in unless the server must modify the parameter. inout doubles Parcel size and serialization cost. For primitives, direction is implicit (always in). For Parcelable parameters, direction must be explicitly declared.

Q15. What is versioned/stable AIDL? When would you use it?

Versioned AIDL freezes interface snapshots so old clients work with new servers. New methods can only be added (never removed/changed), new Parcelable fields can only be appended (with defaults). Auto-generated getVersion() and getHash() let clients check the server's version before calling new methods. Used by: system services (framework↔vendor stability), AIDL HALs (hardware interface stability), and Play Services (client↔server version skew). Without versioning, adding a method changes transaction codes, causing old clients to call the wrong method or crash.

Q16. Compare AIDL, Messenger, ContentProvider, and Broadcast for IPC.

AIDL: bidirectional typed RPC on Binder thread pool (concurrent), Parcel serialization, supports oneway, best for complex multi-method APIs with return values. Messenger: wraps IMessenger.aidl with a Handler — single-threaded (all messages serialized), no concurrency concerns, no typed returns (reply via separate Message), best for simple request queues. Internally Messenger IS Binder — just a convenience wrapper. ContentProvider: CRUD+query semantics, results via CursorWindow (ashmem/SharedMemory — zero-copy for large datasets), Binder thread pool, cross-app data sharing with URI permissions, but each query() is a synchronous Binder call. Broadcast: one-to-many, Intent+extras (limited size), delivered on main thread by default (goAsync for background), no return value, high latency (AMS dispatch). Key trade-offs: AIDL gives you concurrency but requires thread safety; Messenger gives thread safety but serializes everything; ContentProvider gives shared-memory efficiency for large data but has cold-start cost (~500ms for first query to a new process); Broadcast is loosely coupled but unreliable for guaranteed delivery.

Q17. Why is Parcelable preferred over Serializable for Binder IPC?

Parcelable uses direct read/write to a flat buffer — no reflection, no temporary objects, no class metadata per element. Serializable uses Java reflection to discover fields, creates temporary ObjectOutputStream/ObjectInputStream, writes class descriptors, and generates significant garbage. Parcelable is ~10x faster and produces ~3x smaller output. @Parcelize (Kotlin compiler plugin) generates Parcelable boilerplate automatically, eliminating the verbosity argument for Serializable. Parcelable is not suitable for persistent storage (format is unversioned and platform-dependent) — only for IPC.

Q18. What is the difference between writeTypedList and writeList in Parcel?

writeTypedList: the type is known at compile time, so only field data is written per element — no class name overhead. writeList: writes the fully-qualified class name for EACH element (e.g., "com.example.User" repeated 1000 times), plus type-checking overhead. For a list of 1000 objects, writeList can produce 5x more data and be 3x slower. Always use writeTypedList (or writeTypedArray) when the type is known. The AIDL compiler generates writeTypedList by default for typed list parameters.

Zygote & Process Model

Q19. Why does Android use Zygote instead of starting each app from scratch?

Starting an app requires loading the ART runtime (~30MB), ~10,000 framework classes, shared resources, and native libraries — 2-5 seconds from scratch. Zygote preloads all of this once at boot, then fork()s for each new app. fork() uses copy-on-write (COW): the child shares all of Zygote's memory pages until it modifies them. Result: app startup in ~100-300ms. Memory savings: 50-80MB of preloaded data is shared across all processes instead of duplicated. Without Zygote, 50 apps would consume 2.5-4GB just for framework overhead; with Zygote, it's ~200-400MB.

Q20. Walk through the fork-and-specialize process when AMS starts a new app.

(1) AMS calls ZygoteProcess.start(), connects to /dev/socket/zygote (Unix domain socket). (2) Sends arguments: UID, GID, supplemental GIDs, nice-name, target SDK, SELinux info, entry point class. (3) ZygoteServer.runSelectLoop() receives the request, ZygoteConnection.processCommand() parses it. (4) Zygote.forkAndSpecialize() → native fork(). (5) In the child: setuid()/setgid()/setgroups() to drop privileges, selinux_android_setcontext() to transition from "zygote" to "untrusted_app" SELinux domain, apply seccomp filter, close Zygote's server socket. (6) Call ActivityThread.main() → create main Looper → attach to AMS via Binder → AMS sends the pending component to launch.

Q21. What happens if Zygote dies?

Zygote is declared as a critical service in init.zygote64_32.rc with critical and onrestart directives. If Zygote dies: (1) init detects the death via SIGCHLD. (2) The onrestart triggers restart zygote which also restarts all services that depend on it — including system_server. (3) All Java processes (every app) are killed because they're children of Zygote. (4) init re-launches Zygote, which re-preloads classes, re-forks system_server, and system_server re-registers all services. (5) Apps are re-launched as needed. This is a "soft reboot" — the Linux kernel stays running, but the entire Android userspace restarts. Total recovery time: ~15-30 seconds (similar to a normal boot minus kernel init). Zygote is designed to be extremely stable — after boot, it only listens on a socket and forks. Crashes are rare and usually indicate system-level bugs (bad ROM, kernel memory corruption, or a native library crash during preload).

Q22. How does Android assign UIDs to apps? What are supplemental GIDs?

PackageManagerService assigns a unique UID (range 10000-19999 for user 0) at install time, stored in /data/system/packages.xml as userId="10042" and in /data/system/packages.list (kernel-readable). Multi-user: User N's UID = N*100000 + appId (e.g., user 1's app 10042 → UID 110042). UserHandle.getAppId(uid) extracts the appId, UserHandle.getUserId(uid) extracts the user. Supplemental GIDs are the bridge between Android permissions and Linux capabilities: when PMS grants INTERNET permission, it maps it to GID 3003 (inet) via platform.xml's <permission><group gid="inet"/>. During Zygote.forkAndSpecialize(), these GIDs are passed as the gids parameter and applied via setgroups(). Key mappings: GID 3003 (inet) → INTERNET (socket creation), GID 1028 (sdcard_r) → READ_EXTERNAL_STORAGE, GID 1015 (sdcard_rw) → WRITE_EXTERNAL_STORAGE, GID 1023 (media_rw) → media access, GID 9997 (everybody) → shared cache. Without GID 3003, an app literally cannot create a socket — the kernel denies it. The deprecated sharedUserId let apps share a UID (same sandbox, can read each other's files), deprecated in Android 13.

Q23. What is AppZygote? How does it differ from the main Zygote?

AppZygote (Android 10+) is a per-app Zygote for isolated processes. An app with android:useAppZygote="true" gets its own Zygote forked from the main Zygote. The app can preload its own classes/libraries into its AppZygote. Isolated services (android:isolatedProcess="true") then fork from AppZygote instead of the main Zygote, starting faster because app classes are already loaded. WebView uses a similar mechanism (WebView Zygote) to preload Chromium libraries. AppZygote runs in a separate SELinux domain and has a separate UID.

Q24. How does the system choose between 32-bit and 64-bit Zygote for an app?

Two Zygotes start at boot: zygote64 (primary) and zygote32 (secondary). PMS determines the app's primary ABI at install time by inspecting native libraries: arm64-v8a libs → 64-bit, armeabi-v7a only → 32-bit, both → 64-bit (primary ABI), no native libs → 64-bit (default). Stored in packages.xml as primaryCpuAbi. Some Android 12+ devices are 64-bit only (no 32-bit Zygote), saving ~200MB RAM. 32-bit and 64-bit processes communicate via Binder normally — Parcel handles pointer-size differences.

System Services

Q25. How does system_server boot? What is the service start order?

Zygote forks system_server (UID 1000) → SystemServer.run() → creates system context → starts services in three phases: Bootstrap (critical dependencies first): Installer, ActivityTaskManagerService, ActivityManagerService, PowerManagerService, PackageManagerService, UserManagerService. Core: BatteryService, UsageStatsService, WebViewUpdateService. Other (~80+ services): WindowManagerService, InputManagerService, AlarmManagerService, ConnectivityService, NotificationManagerService, etc. Order matters because of dependencies — AMS must exist before WMS. After all services start, system_server enters Looper.loop().

Q26. What is ServiceManager and why is it special?

ServiceManager is a native daemon (frameworks/native/cmds/servicemanager/) that becomes handle 0 by being the first process to call ioctl(fd, BINDER_SET_CONTEXT_MGR, 0) on /dev/binder. This ioctl sets the calling process as the "context manager" — the kernel assigns it the special context_mgr_node that all processes can reach as handle 0. It's the "DNS of Android": addService(name, binder) registers, getService(name) looks up, checkService(name) does a non-blocking check. SELinux enforcement: ServiceManager checks the caller's SELinux context against service_contexts — a mapping of service names to allowed domains (e.g., activity u:object_r:activity_service:s0 → only system_server domain can register it). A malicious app trying to register "activity" gets denied by SELinux before ServiceManager even processes the request. It's a minimal native daemon (not Java) for reliability — if it crashed, no service lookups would work, but init restarts it and system_server re-registers all services.

Q27. How does getSystemService() work under the hood?

Three caching layers: (1) SystemServiceRegistry (static initializer) registers factory lambdas for each service name. (2) First call: factory creates a wrapper (e.g., ActivityManager) that internally calls ServiceManager.getService("activity") to get the IBinder, then IActivityManager.Stub.asInterface() to get the proxy. (3) The wrapper and IBinder are cached per-Context. Subsequent calls return the cached wrapper. ServiceManager itself caches IBinder lookups in a HashMap. BinderProxy instances are also cached. Result: first call triggers one Binder transaction to ServiceManager; subsequent calls are pure Java.

Q28. What is the Watchdog in system_server?

A dedicated thread that monitors critical threads (main, display, animation, I/O) and locks (AMS, WMS, PMS locks). Every 30 seconds, it posts a message to each monitored handler. If not processed within 60s → half-watchdog (warning). If not processed within 120s → full watchdog: dumps all thread stacks, dumps kernel info, then kills system_server (Process.killProcess()). init then restarts everything. Common causes: lock contention (AMS lock held too long), Binder thread pool exhaustion (all 31 threads busy), disk I/O on the main thread, deadlocks between services.

Q29. How do AIDL HALs differ from HIDL HALs?

HIDL (Android 8-11): separate IDL language, uses /dev/hwbinder, generates C++ stubs, HIDL transport. AIDL HALs (Android 11+): standard AIDL, uses /dev/binder, same compiler as framework AIDL, supports versioning natively, Java and C++ backends. Advantages of AIDL HALs: unified tooling, versioned interfaces, no separate IDL to learn, better testing infrastructure, and they run on the same Binder domain as framework services. VINTF manifest declares available HALs. Google is migrating all HALs from HIDL to AIDL.

IPC Performance & Bottlenecks

Q30. What causes TransactionTooLargeException? How do you fix it?

Two causes: (1) A single Parcel exceeds the remaining buffer space (e.g., Bitmap in Intent extras). (2) Cumulative pending transactions exceed the 1MB per-process buffer (e.g., rapid oneway callbacks to a slow receiver). Common culprits: Bitmaps in Intents, large savedInstanceState (many Fragments), ContentProvider returning huge cursors. Fixes: use content URIs for large data, SharedMemory for bulk bytes, ViewModel for transient state (not savedInstanceState), bulkInsert()/applyBatch() for batching, rate-limit oneway calls. Measure with Parcel.dataSize() and keep individual transactions under 100KB.

Q31. How can Binder thread pool exhaustion cause ANRs in other apps?

If your app hosts a ContentProvider or bound Service, other apps' Binder calls to you are handled by YOUR thread pool. If all 16 threads are busy (e.g., doing disk I/O), incoming transactions queue in the kernel. The calling app's main thread is blocked waiting for the reply. If the wait exceeds 5 seconds, the calling app gets an ANR — even though the problem is in YOUR process. Fix: dispatch heavy work off Binder threads immediately (use a coroutine/executor), keep Binder thread handlers lightweight, and return quickly.

Q32. Describe the distributed deadlock scenario with Binder.

Process A calls Process B synchronously (A's thread blocks). B's handler calls back to A synchronously. If all of A's Binder threads are waiting for B's reply, B's callback to A has no free thread. Both processes are stuck permanently. This is invisible to single-process deadlock detectors because no single process has a cycle — the cycle spans two processes via the kernel. Prevention: (1) use oneway for all callbacks, (2) design unidirectional call graphs (A→B, never B→A synchronously), (3) dispatch callbacks to a separate handler thread instead of calling back on the Binder thread.

Q33. How does ContentProvider use shared memory for query results?

ContentProvider.query() returns a Cursor backed by a CursorWindow. CursorWindow is a shared memory region (ashmem/SharedMemory, default 2MB) that the provider fills with result data. The file descriptor for this shared memory is passed via Binder (kernel dups it into the client process). The client maps the shared memory read-only and reads directly — zero-copy after the initial fill. This is why ContentProvider can return large result sets efficiently. The Binder transaction itself only carries metadata (column names, row count) and the fd — not the actual data.

Q34. What system APIs are synchronous Binder calls that can ANR on the main thread?

Common culprits and WHY they block: PackageManager.getInstalledPackages() → PMS holds a lock while scanning; on low-end devices with 200+ apps, this can take 500ms+. ContentResolver.query() → if the provider is in another process, AMS may need to start that process first (Zygote fork, ~200ms), then the query runs on the provider's Binder thread. ActivityManager.getRunningAppProcesses() → AMS holds its global lock while iterating ProcessRecords; contention from other apps' lifecycle events causes delays. ConnectivityManager.getActiveNetworkInfo() → ConnectivityService may be processing a network change event, holding its lock. SharedPreferences.commit() → synchronous disk write + can trigger a ContentProvider backup. Root cause pattern: each is a synchronous Binder call to system_server, which has only 31 Binder threads shared across ALL apps. If those threads are busy (boot storm, app install, GC pause), your call queues. Worst case: system_server's main thread is blocked → Watchdog fires → soft reboot. Fix: (1) move to Dispatchers.IO, (2) use async alternatives (apply() not commit()), (3) cache results, (4) use StrictMode.detectCustomSlowCalls() to find hidden Binder calls.

Q35. How would you optimize an app that makes 1000 Binder calls per second?

(1) Batch: replace per-item calls with bulk operations (bulkInsert(), applyBatch(), or a single AIDL method taking a list). (2) Cache: cache results client-side, invalidate via oneway callbacks. (3) SharedMemory: for bulk data, write to SharedMemory and pass the fd. (4) Reduce frequency: debounce/throttle calls, aggregate updates. (5) Oneway for fire-and-forget: saves a kernel round-trip. (6) Same-process optimization: if possible, move the service into the same process (no IPC). Each Binder call costs ~50-100μs overhead; 1000 calls = 50-100ms of pure IPC cost.

Advanced & Design Questions

Q36. How does Binder handle passing IBinder objects inside a transaction?

When a Parcel contains a flat binder object (BINDER_TYPE_BINDER or BINDER_TYPE_HANDLE), the kernel translates it during binder_transaction(). If the sender passes a local BBinder, the kernel creates (or finds) a binder_node for it and gives the receiver a binder_ref (handle). If the sender passes a handle (remote reference), the kernel translates it to the receiver's handle namespace. This enables transparent passing of Binder objects between any processes — the kernel maintains a global graph of nodes and references. Passing to a third process creates a new reference to the same node.

Q37. Explain Binder reference counting. What prevents use-after-free?

Two-level scheme: strong references (BC_ACQUIRE/BC_RELEASE) keep the binder_node alive; weak references (BC_INCREFS/BC_DECREFS) allow checking liveness without preventing destruction. When strong refs drop to 0, the kernel sends BR_RELEASE to the owning process, which destroys the BBinder. The driver tracks counts per binder_ref. For death notifications, weak refs are used. The kernel atomically manages these counts, preventing races. Use-after-free is prevented because the kernel won't deliver transactions to a dead node — it returns BR_DEAD_REPLY to the caller.

Q38. What is the oneway buffer exhaustion problem? How do you detect and prevent it?

Oneway transactions queue in the target's 1MB mmap buffer. If the target is slow (e.g., GC pause, disk I/O), rapid oneway calls fill the buffer. New transactions silently fail with "FAILED BINDER TRANSACTION" in logcat — the sender gets no error. Detection: monitor logcat for "FAILED BINDER TRANSACTION", use dumpsys meminfo to check Binder buffer usage, Systrace to see queued transactions. Prevention: (1) rate-limit oneway calls, (2) batch multiple updates into a single callback, (3) use flow control (server sends "ready" before client sends next batch), (4) monitor and alert on buffer usage.

Q39. How does the Parcel handle file descriptor passing?

Parcel.writeFileDescriptor(fd) writes a flat binder object of type BINDER_TYPE_FD. During binder_transaction(), the kernel calls dup() to duplicate the fd into the target process's fd table. The receiver gets a new fd number pointing to the same kernel file description. This is how CursorWindow (ashmem fd), SharedMemory, and ParcelFileDescriptor work across processes. The kernel handles the duplication atomically. The sender can close its fd after the transaction — the receiver's fd is independent.

Q40. Design a high-throughput IPC system for streaming sensor data between two Android apps.

Architecture: (1) Use SharedMemory as a ring buffer for sensor data — producer writes, consumer reads, zero-copy. (2) Use a Binder AIDL interface for control: start(), stop(), getSharedMemory(). (3) Use oneway Binder callbacks for signaling: producer sends onDataAvailable(offset, length) when new data is written. (4) Consumer maps the SharedMemory read-only and reads from the signaled offset. Why not pure Binder? At 100Hz sensor data × 1KB per sample = 100KB/s of Parcel data + 100 transactions/s. SharedMemory reduces this to ~100 oneway signals/s with zero-copy data access. Synchronization: use atomic counters in the shared memory header (write index, read index) for lock-free producer-consumer. Failure handling: DeathRecipient to detect producer death, timeout on consumer side if no signal arrives.

Q41. What is the role of Parcel.enforceInterface() and what attack does it prevent?

Every AIDL-generated onTransact() starts with data.enforceInterface(DESCRIPTOR), which reads the interface token from the Parcel and verifies it matches the expected descriptor string. This prevents a malicious client from sending a transaction intended for one interface to a different interface on the same Binder object. Without it, a client could craft a Parcel with the wrong transaction code and data layout, potentially causing the server to misinterpret the data. It's a defense-in-depth measure against confused-interface attacks.

Q42. How does system_server handle Binder thread pool pressure during boot?

During boot, dozens of apps start simultaneously, all making Binder calls to system_server (AMS for process registration, PMS for package info, WMS for window setup). system_server's 31 Binder threads can become saturated. Mitigation: (1) Boot-critical services use oneway where possible. (2) AMS batches process start requests. (3) PackageManagerService pre-scans packages at boot (not on-demand). (4) SystemServiceRegistry defers non-critical service initialization. (5) Android's boot animation runs until system_server signals readiness. A Staff Engineer should know that adding synchronous Binder calls during app startup contributes to boot time regression.

Q43. How would you debug a "Binder thread pool exhausted" issue in production?

(1) adb shell dumpsys activity processes — shows per-process Binder thread states. (2) adb shell cat /proc/binder/proc/<pid> — kernel-level view of threads, pending transactions, buffer usage. (3) Systrace/Perfetto — "binder" track shows transaction durations and thread utilization. (4) Thread dump (kill -3 <pid>) — shows what each Binder thread is doing (likely blocked on I/O or a lock). (5) Look for the root cause: long-running operations on Binder threads (database queries, network calls, lock contention). Fix: dispatch heavy work to a dedicated thread pool immediately upon receiving the Binder call.

Q44. What is the savedInstanceState TransactionTooLargeException and how do you prevent it?

When an Activity is stopped, the system saves its state via onSaveInstanceState(). This Bundle is sent to AMS via Binder. If the Activity has many Fragments each saving state, the cumulative Bundle can exceed the 1MB buffer. Common with ViewPager + many Fragments. Prevention: (1) Use ViewModel for transient UI state (not saved in Bundle). (2) Use SavedStateHandle for small essentials only (IDs, scroll position). (3) Keep total savedInstanceState under 50KB. (4) Use Navigation component's SavedStateHandle instead of Fragment arguments. (5) Measure with Parcel.dataSize() in debug builds.

Q45. Design a secure, high-performance IPC architecture for a suite of 5 apps that share user data.

Architecture: One "data hub" app hosts a ContentProvider with signature-level permission (protectionLevel="signature") — only apps signed with the same key can access it. Data layer: ContentProvider backed by Room database. Use bulkInsert()/applyBatch() for writes. Queries return Cursors with CursorWindow (shared memory). Real-time sync: Data hub exposes an AIDL service with oneway callbacks. Client apps register via bindService(). Hub sends onDataChanged(uri) oneway callbacks. Clients re-query only changed URIs. Security: Signature permission ensures only your apps can access. enforceCallingPermission() in the provider. URI-level permissions for fine-grained access. Performance: Cache frequently-accessed data client-side. Invalidate via oneway callbacks. Use ContentResolver.registerContentObserver() for reactive updates. Versioning: Use versioned AIDL for the service interface so apps can update independently.

Failure Scenarios, Security & Advanced Design

Q46. What happens when a Binder transaction fails the SELinux check? Trace the failure path.

In binder_transaction(), after finding the target node, the kernel calls security_binder_transaction() which invokes the SELinux hook selinux_binder_transaction(). This checks the SELinux policy: "can source domain X perform binder_call to target domain Y?" If denied: (1) kernel logs avc: denied { call } scontext=u:r:untrusted_app:s0 tcontext=u:r:system_server:s0, (2) binder_transaction() returns -EPERM, (3) the caller receives BR_FAILED_REPLY, (4) IPCThreadState::waitForResponse() translates this to a SecurityException or RemoteException. Similarly, security_binder_transfer_binder() checks if an IBinder can be passed to the target, and security_binder_transfer_file() checks fd passing. These SELinux checks are why a malicious app can't directly call HAL services or register fake system services.

Q47. A Binder call never returns. What are the possible causes and how do you debug it?

Binder has no built-in timeout — a synchronous call blocks indefinitely until the server replies or dies. Causes: (1) Server deadlock — the Binder thread handling your call is waiting on a lock held by another thread that's also blocked. (2) Server Binder pool exhaustion — all 16 threads are busy; your transaction sits in the kernel queue. (3) Distributed deadlock — A calls B, B calls back to A, A has no free threads. (4) Server in GC — long GC pause blocks all Binder threads. (5) Server process frozen — cached process frozen by the kernel (Android 11+ freezer). Debug: (a) adb shell kill -3 <server_pid> for thread dump — look for blocked Binder threads, (b) cat /proc/binder/proc/<pid> to see pending transactions and thread states, (c) Systrace/Perfetto binder track shows transaction start without end, (d) adb shell dumpsys activity processes shows process state (frozen/cached). The caller's ANR trace will show android.os.BinderProxy.transactNative in the stack.

Q48. What is USAP (Unspecialized App Process)? How does it improve app launch time?

USAP (Android 10+) is a pool of pre-forked processes that are already forked from Zygote but not yet specialized (no UID, no SELinux context, no app code). When AMS needs a new process, instead of asking Zygote to fork (which takes fork + specialize = ~50-100ms), it picks a USAP and sends specialization args via an eventfd. The USAP specializes itself (setuid, SELinux, seccomp) in ~5-20ms — 3-5x faster. Zygote refills the pool via UsapPoolRefillAction. Source: Zygote.java (nativeForkUsap()). Not used for system_server (always direct fork). The pool size is configurable via system properties. USAP processes block on getUsapPoolEventFD() waiting for specialization commands.

Q49. How does Binder handle transaction nesting? What prevents deadlock?

Binder supports nested transactions via the transaction_stack on each binder_thread. When A calls B synchronously and B calls back to A, the kernel detects that A has a sleeping thread waiting for B's reply. It delivers B's callback to A's same sleeping thread (not a random pool thread). This prevents deadlock because the thread that's waiting for B can still handle B's callback. The transaction_stack tracks the chain: A→B→A. After A handles the callback and replies, B continues and replies to the original call. Key nuance: this only works for synchronous nesting. If A calls B with oneway and B calls A synchronously, there's no nesting — B's call goes to a random free thread in A's pool. If A's pool is exhausted, B's call blocks — this IS a deadlock risk that nesting doesn't solve.

Q50. What are the boot phases in SystemServiceManager? Why do they matter?

SystemServiceManager defines ordered phases that system services receive via onBootPhase(): PHASE_WAIT_FOR_DEFAULT_DISPLAY (100) — display ready; PHASE_LOCK_SETTINGS_READY (480) — lock screen settings available; PHASE_SYSTEM_SERVICES_READY (500) — core services can interact; PHASE_ACTIVITY_MANAGER_READY (550) — AMS ready, can launch apps; PHASE_THIRD_PARTY_APPS_CAN_START (600) — safe to start third-party apps; PHASE_BOOT_COMPLETED (1000) — boot animation done, user can interact. They matter because: (1) services must not call other services before PHASE_SYSTEM_SERVICES_READY (they may not be initialized), (2) third-party apps must not start before phase 600 (system not ready), (3) a service doing heavy work in an early phase delays all subsequent phases and boot time. A Staff Engineer adding a new system service must choose the correct phase for initialization.

Q51. What is LocalServices and why does it exist alongside ServiceManager?

LocalServices is an in-process registry within system_server. System services register "Internal" interfaces (e.g., DisplayManagerInternal) via LocalServices.addService(). Other services in the same process look them up via LocalServices.getService(). This avoids Binder overhead for intra-system_server calls: no Parcel serialization, no ioctl, no thread switch — just a direct Java method call. The "Internal" interfaces expose more than the public Binder API (methods that shouldn't be accessible to apps). It's a simple ArrayMap<Class, Object>. Without it, system services would make ~1000s of unnecessary Binder transactions per second to each other, wasting CPU and Binder thread pool capacity.

Q52. How would you design a Binder service that handles 10,000 concurrent clients without bottlenecking?

Thread pool: Default 16 Binder threads are insufficient. Dispatch incoming calls immediately to a dedicated coroutine/thread pool (e.g., Dispatchers.IO with limitedParallelism(64)). Binder threads only accept and dispatch — never do heavy work. Batching: Instead of per-client polling, use oneway callbacks. Clients register once; server batches updates and sends a single oneway callback with a list of changes. Shared memory: For bulk data, use SharedMemory as a ring buffer. Server writes, clients map read-only. Binder only carries signaling (offset + length). Connection management: Use DeathRecipient to auto-cleanup dead clients. Maintain a RemoteCallbackList (handles death, threading, and iteration). Rate limiting: Enforce per-client rate limits to prevent buffer exhaustion from a single misbehaving client. Monitoring: Track Binder buffer usage, thread pool utilization, and transaction latency as SLIs. Alert when buffer exceeds 500KB or thread utilization exceeds 80%.

Q53. Design an IPC system that survives process death and recovers state automatically.

Architecture: (1) Persistent state: Server stores all client registrations and pending operations in a Room database. On restart, it reloads state and resumes. (2) Client reconnection: Clients use DeathRecipient to detect server death. On binderDied(), client schedules reconnection with exponential backoff. Use bindService() with BIND_AUTO_CREATE — the system restarts the server automatically. (3) Idempotent operations: All write operations include a client-generated UUID. Server deduplicates on restart — replayed operations are no-ops. (4) State recovery: After reconnection, client sends its last-known state version. Server sends only the delta. (5) Ordering guarantee: Use a monotonic sequence number per client. Server rejects out-of-order operations. (6) Timeout: Client-side timeout on synchronous calls (wrap in withTimeout() coroutine). On timeout, assume server is dead and trigger reconnection. This pattern is used by Google Play Services for its Task-based APIs.
