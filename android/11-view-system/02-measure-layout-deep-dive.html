02 — Measure & Layout Deep Dive

MeasureSpec encoding, the onMeasure() contract, ViewGroup child measurement, multi-pass exponential blowup, ConstraintLayout's solver, onLayout(), LayoutParams, and staff-level corner cases

1. MeasureSpec — The Measurement Contract

Every measurement in the Android View system is communicated through a MeasureSpec — a single Int that packs two pieces of information into 32 bits: a mode (2 high bits) and a size (30 low bits). This encoding is a deliberate performance optimization — passing one primitive instead of an object avoids allocation during the tight measure loop that runs every frame.

Bit Layout

  Bits 31-30: MODE          Bits 29-0: SIZE (max ~1 billion px)
  ┌──────────┬──────────────────────────────────────────┐
  │  2 bits  │              30 bits                     │
  └──────────┴──────────────────────────────────────────┘
  00 = UNSPECIFIED    01 = EXACTLY    10 = AT_MOST

The Three Modes

ModeConstantMeaningTypical Source
EXACTLY 1 << 30 Parent dictates an exact pixel size. Child must be exactly this size. match_parent when parent knows its own size, or a fixed dp value
AT_MOST 2 << 30 Parent sets a maximum. Child can be smaller but not larger. wrap_content — child decides its own size up to the parent's limit
UNSPECIFIED 0 << 30 No constraints. Child can be any size it wants. ScrollView measuring its child (needs full content height)

Creating and Extracting MeasureSpecs

val spec = MeasureSpec.makeMeasureSpec(500, MeasureSpec.EXACTLY)
val mode = MeasureSpec.getMode(spec)   // MeasureSpec.EXACTLY
val size = MeasureSpec.getSize(spec)   // 500

// Under the hood — pure bitwise ops, zero allocation
fun makeMeasureSpec(size: Int, mode: Int): Int =
    (size and (1 shl 30).inv() - 1) or (mode and (0x3 shl 30))

Parent LayoutParams → Child MeasureSpec Mapping

The critical question: given a parent's own MeasureSpec and a child's LayoutParams, what MeasureSpec does the child receive? This is resolved by ViewGroup.getChildMeasureSpec().

Child LayoutParams Parent EXACTLY (e.g. 500px) Parent AT_MOST (e.g. max 500px) Parent UNSPECIFIED
fixed dp (e.g. 200dp) EXACTLY 200 EXACTLY 200 EXACTLY 200
match_parent EXACTLY 500 AT_MOST 500 UNSPECIFIED 0
wrap_content AT_MOST 500 AT_MOST 500 UNSPECIFIED 0
Staff Insight — Why This Table Matters

This 3×3 matrix is the single most important thing to internalize about Android measurement. Every custom View bug, every "why is my View 0px tall?" question, every ScrollView mystery traces back to this table. In an L6 interview, being able to derive any cell from first principles (not memorization) signals deep understanding.

2. The Measure Pass — onMeasure() Contract

Measurement flows top-down through the View tree. The public entry point is View.measure(widthMeasureSpec, heightMeasureSpec), which is a final method — you cannot override it. It performs caching checks, then delegates to onMeasure(), which is the method custom Views override.

The measure() → onMeasure() Flow

  child.measure(widthSpec, heightSpec)     [final — cannot override]
    │
    ├─ 1. Cache lookup: mMeasureCache[(wSpec,hSpec)] → hit + no FORCE? skip
    ├─ 2. Check PFLAG_FORCE_LAYOUT (set by requestLayout/forceLayout)
    ├─ 3. Specs unchanged + no force? skip
    ├─ 4. Call onMeasure(widthSpec, heightSpec)  ← YOUR OVERRIDE
    ├─ 5. Verify PFLAG_MEASURED_DIMENSION_SET → else IllegalStateException
    └─ 6. Cache result in mMeasureCache

  onMeasure() contract: MUST call setMeasuredDimension(w, h) before returning

A Minimal Custom View onMeasure()

class CircleView(context: Context, attrs: AttributeSet?) : View(context, attrs) {

    override fun onMeasure(widthMeasureSpec: Int, heightMeasureSpec: Int) {
        val desiredSize = 200 // intrinsic size in px

        val width = resolveSize(desiredSize, widthMeasureSpec)
        val height = resolveSize(desiredSize, heightMeasureSpec)

        // Contract: MUST call this before returning
        setMeasuredDimension(width, height)
    }
}

// View.resolveSize() — utility that respects the MeasureSpec mode:
//   EXACTLY → return spec size (parent's dictated size)
//   AT_MOST → return min(desired, spec size)
//   UNSPECIFIED → return desired size

measuredWidth vs width

PropertySet DuringSourceCan Differ?
measuredWidth / measuredHeight Measure pass setMeasuredDimension() Yes — the parent's onLayout() can assign different bounds than what the child measured
width / height Layout pass layout(l, t, r, b) → right - left, bottom - top

In practice, measured and actual dimensions are usually identical. They diverge when a parent intentionally overrides the child's preference — for example, LinearLayout distributing leftover space via layout_weight may assign a child more pixels than it measured.

3. ViewGroup Measurement — Measuring Children

A ViewGroup is responsible for measuring all of its children during its own onMeasure(). The framework provides helper methods, but the logic of how to combine children's sizes is entirely up to the ViewGroup subclass.

Helper Methods

MethodWhat It DoesAccounts For
measureChildren(wSpec, hSpec) Iterates all children, calls measureChild() for each (skips GONE views) Parent padding
measureChild(child, wSpec, hSpec) Calls getChildMeasureSpec() then child.measure() Parent padding
measureChildWithMargins(child, wSpec, wUsed, hSpec, hUsed) Same as above but also subtracts child margins and already-used space Parent padding + child margins + used space

getChildMeasureSpec() — The Core Algorithm

This static method is the engine behind the 3×3 mapping table from Section 1. Simplified AOSP logic: 

fun getChildMeasureSpec(parentSpec: Int, padding: Int, childDimension: Int): Int {
    val specMode = MeasureSpec.getMode(parentSpec)
    val specSize = MeasureSpec.getSize(parentSpec)
    val available = max(0, specSize - padding)

    val (resultSize, resultMode) = when {
        childDimension >= 0 ->          // fixed dp — always EXACTLY
            childDimension to MeasureSpec.EXACTLY
        childDimension == MATCH_PARENT -> when (specMode) {
            MeasureSpec.EXACTLY    -> available to MeasureSpec.EXACTLY
            MeasureSpec.AT_MOST    -> available to MeasureSpec.AT_MOST
            else /* UNSPECIFIED */  -> 0 to MeasureSpec.UNSPECIFIED
        }
        else /* WRAP_CONTENT */ -> when (specMode) {
            MeasureSpec.EXACTLY    -> available to MeasureSpec.AT_MOST
            MeasureSpec.AT_MOST    -> available to MeasureSpec.AT_MOST
            else /* UNSPECIFIED */  -> 0 to MeasureSpec.UNSPECIFIED
        }
    }
    return MeasureSpec.makeMeasureSpec(resultSize, resultMode)
}

4. Multi-Pass Measurement — The Performance Trap

Not all ViewGroups measure their children once. Some require multiple measurement passes per frame, and nesting these creates an exponential blowup that is the single biggest performance pitfall in the View system.

Which ViewGroups Measure Twice?

ViewGroupPassesWhy
LinearLayout (with layout_weight) 2 Pass 1: measure natural sizes. Pass 2: distribute remaining space proportionally.
RelativeLayout 2 Pass 1: resolve horizontal rules (left/right/centerHorizontal). Pass 2: resolve vertical rules (top/bottom/centerVertical).
LinearLayout with measureWithLargestChild 2 Pass 1: measure all children. Pass 2: re-measure smaller children to match the largest.
FrameLayout 1–2 Usually 1. But if it has match_parent children and was measured with AT_MOST, it re-measures those children with EXACTLY after determining its own size.

The Exponential Blowup

  Each level measures children twice → 2^N calls at leaf depth N:

  Level 0:  1 call
  Level 1:  2 calls
  Level 2:  4 calls
  Level 3:  8 calls   ← RelativeLayout → LinearLayout(weight) → RelativeLayout → TextView
  Level 5:  32 calls
  Level 10: 1,024 calls
Performance Danger — Exponential Measurement

A layout tree just 5 levels deep with double-measuring parents triggers 32 onMeasure() calls per leaf View per frame. At 10 levels, it's 1,024 calls. This is not theoretical — it was the #1 cause of jank in pre-ConstraintLayout apps. The fix: flatten your hierarchy, avoid nesting RelativeLayout inside LinearLayout with weights, or switch to ConstraintLayout.

Detecting Multi-Pass Issues

// Debug wrapper — count onMeasure calls per frame:
class InstrumentedLayout(ctx: Context, attrs: AttributeSet?) : LinearLayout(ctx, attrs) {
    override fun onMeasure(wSpec: Int, hSpec: Int) {
        Log.d("MeasureCount", "${javaClass.simpleName} onMeasure #${++count}")
        super.onMeasure(wSpec, hSpec)
    }
    companion object { var count = 0 }
}
// Also: Layout Inspector + Systrace to see measure/layout duration per frame

5. ConstraintLayout Solver

ConstraintLayout was created specifically to eliminate the multi-pass measurement problem. It uses a Cassowary-based linear arithmetic constraint solver to resolve all child positions and sizes in a single pass, regardless of constraint complexity.

How the Solver Works

  1. Constraint collection — child constraints translated into linear equations/inequalities.
  2. System construction — simplex tableau built; variables = each child's left/right/top/bottom/width/height.
  3. Optimization — simplex algorithm finds values satisfying all constraints; conflicts resolved by priority.
  4. Result extraction — solved values position children directly, skipping recursive measure-then-layout.

Performance Characteristics

MetricNested Multi-PassConstraintLayout
Measure passes per child O(2N) where N = nesting depth O(1) — single pass
Total work for n children O(n × 2depth) O(n) — linear in child count
Solver overhead None (no solver) Simplex pivot operations — fast for typical UI constraints (~100 variables)
View hierarchy depth Deep (many nested ViewGroups) Flat (single parent for all children)

Layout Helpers — Constraints Without Nesting

HelperPurposeReplaces
Guideline Invisible anchor line at a fixed position or percentage Nested LinearLayout with weights for proportional splits
Barrier Dynamic edge that tracks the furthest extent of a set of views Wrapping views in a container just to align against them
Chain Distributes space among a group (spread, spread_inside, packed, weighted) LinearLayout with layout_weight
Flow Virtual layout that wraps children into rows/columns FlexboxLayout or nested LinearLayouts
Group Controls visibility of multiple views simultaneously Wrapping views in a container for visibility toggling

ConstraintSet — Programmatic Manipulation

val set = ConstraintSet().apply { clone(constraintLayout) }
set.connect(R.id.title, ConstraintSet.START, ConstraintSet.PARENT_ID, ConstraintSet.START)
set.connect(R.id.title, ConstraintSet.END, ConstraintSet.PARENT_ID, ConstraintSet.END)

TransitionManager.beginDelayedTransition(constraintLayout)
set.applyTo(constraintLayout)
Staff Insight — Cross-Platform Solver Lineage

ConstraintLayout's solver is based on the Cassowary algorithm (Badros et al., 1998) — the same algorithm that powers Apple's Auto Layout. Knowing this demonstrates cross-platform systems awareness. The key difference: Auto Layout runs the solver on every frame (which caused performance issues in early iOS), while ConstraintLayout caches resolved positions and only re-solves when constraints change. This caching strategy is why ConstraintLayout can handle hundreds of children without jank.

6. The Layout Pass — onLayout() Contract

After measurement completes, the layout pass assigns each View its actual position within its parent. Like measurement, layout flows top-down: each parent calls layout() on its children with the computed coordinates.

layout() → onLayout() Flow

  Parent calls child.layout(l, t, r, b)
    │
    ▼  View.layout()
    1. setFrame() → store mLeft/mTop/mRight/mBottom, compute changed
    2. If changed OR PFLAG_LAYOUT_REQUIRED → call onLayout(changed, l, t, r, b)
    3. Clear PFLAG_LAYOUT_REQUIRED
    4. Notify OnLayoutChangeListeners
    │
    ▼  onLayout() [ViewGroup override]
    For each child: child.layout(childL, childT, childR, childB)

Position Properties After Layout

PropertyMeaningCoordinate Space
left / topPosition of the View's top-left cornerRelative to parent
right / bottomPosition of the View's bottom-right cornerRelative to parent
widthright - left—
heightbottom - top—
x / yleft + translationX / top + translationYRelative to parent (includes translation)

Direct Position Manipulation

view.translationX = 100f   // shifts visual position; layout position unchanged
view.left = 200            // changes mLeft, triggers invalidation — NOT a layout pass

A Simple Custom ViewGroup onLayout()

class VerticalStackLayout(ctx: Context, attrs: AttributeSet?) : ViewGroup(ctx, attrs) {

    override fun onLayout(changed: Boolean, l: Int, t: Int, r: Int, b: Int) {
        var currentTop = paddingTop
        for (i in 0 until childCount) {
            val child = getChildAt(i).takeIf { it.visibility != GONE } ?: continue
            val lp = child.layoutParams as MarginLayoutParams
            val childLeft = paddingLeft + lp.leftMargin
            val childTop = currentTop + lp.topMargin
            child.layout(childLeft, childTop,
                childLeft + child.measuredWidth, childTop + child.measuredHeight)
            currentTop = childTop + child.measuredHeight + lp.bottomMargin
        }
    }

    override fun generateDefaultLayoutParams() =
        MarginLayoutParams(LayoutParams.WRAP_CONTENT, LayoutParams.WRAP_CONTENT)
    override fun generateLayoutParams(attrs: AttributeSet?) = MarginLayoutParams(context, attrs)
    override fun checkLayoutParams(p: LayoutParams?) = p is MarginLayoutParams
}

7. Layout Parameters

LayoutParams are the child's way of communicating sizing and positioning preferences to its parent. Each ViewGroup subclass defines its own LayoutParams subclass with attributes specific to that layout strategy.

LayoutParams Hierarchy

  ViewGroup.LayoutParams  (width, height)
    └─ MarginLayoutParams  (margins)
         ├─ LinearLayout.LayoutParams   (+weight, gravity)
         ├─ FrameLayout.LayoutParams    (+gravity)
         ├─ RelativeLayout.LayoutParams (+rules[])
         └─ ConstraintLayout.LayoutParams (+constraints, dimensionRatio, chains)

How LayoutParams Are Created

// From XML: LayoutInflater calls parent.generateLayoutParams(attrs)

// Programmatically:
val params = LinearLayout.LayoutParams(MATCH_PARENT, WRAP_CONTENT).apply {
    weight = 1f
    gravity = Gravity.CENTER_HORIZONTAL
    setMargins(16.dp, 8.dp, 16.dp, 8.dp)
}
view.layoutParams = params

// Updating existing params (triggers requestLayout):
view.updateLayoutParams<LinearLayout.LayoutParams> { weight = 2f }
Common Casting Trap

view.layoutParams returns the LayoutParams type of the parent, not the View itself. Casting to the wrong type is a runtime crash: 

// View is inside a FrameLayout, but you cast to LinearLayout.LayoutParams:
val lp = view.layoutParams as LinearLayout.LayoutParams  // ClassCastException!

// Safe pattern:
val lp = view.layoutParams as? LinearLayout.LayoutParams ?: return

8. Common Measurement Pitfalls

match_parent Inside ScrollView

ScrollView measures its child with UNSPECIFIED height (it needs to know the full content height to enable scrolling). When the child has match_parent height, the UNSPECIFIED spec means "no constraint" — so match_parent degrades to wrap_content behavior. 

// This does NOT fill the ScrollView's visible area:
<ScrollView android:layout_height="match_parent">
    <LinearLayout
        android:layout_height="match_parent"  <!-- behaves like wrap_content! -->
        ...>
    </LinearLayout>
</ScrollView>

// Fix: use android:fillViewport="true" on the ScrollView
// This makes ScrollView re-measure the child with EXACTLY if the child
// is shorter than the viewport, giving match_parent its expected behavior.

RecyclerView Inside NestedScrollView

// NEVER do this with wrap_content height:
<NestedScrollView>
    <RecyclerView
        android:layout_height="wrap_content"  <!-- measures ALL items -->
        ...>
</RecyclerView>
</NestedScrollView>

// The RecyclerView receives UNSPECIFIED height → measures to full content
// → creates ALL ViewHolders at once → defeats recycling entirely
// → OOM on large datasets

// If you must combine scrolling, use:
// 1. ConcatAdapter to merge headers/footers into the RecyclerView
// 2. RecyclerView with different view types
// 3. A fixed height for the RecyclerView (but this breaks nested scroll)

Pre-Measuring a View

// Natural size — pass UNSPECIFIED for both axes
view.measure(
    MeasureSpec.makeMeasureSpec(0, MeasureSpec.UNSPECIFIED),
    MeasureSpec.makeMeasureSpec(0, MeasureSpec.UNSPECIFIED)
)
val naturalWidth = view.measuredWidth

// Constrained width — AT_MOST with known max
view.measure(
    MeasureSpec.makeMeasureSpec(maxWidth, MeasureSpec.AT_MOST),
    MeasureSpec.makeMeasureSpec(0, MeasureSpec.UNSPECIFIED)
)

TextView Wrap-Content Width Ordering

When a TextView has wrap_content width, its measured width depends on how much horizontal space is available (which determines line breaks). If a sibling's size changes the available space, the TextView may wrap differently — making measurement order-dependent. This is why RelativeLayout needs two passes: horizontal constraints first, then vertical (because text height depends on the width assigned in the horizontal pass).

9. Staff-Level Corner Cases

Multiple onMeasure() Calls Per Frame

It is perfectly normal for onMeasure() to be called multiple times in a single frame. A parent may speculatively measure a child with one spec, then re-measure with a different spec after computing other children's sizes. Your onMeasure() must be idempotent — it should produce the same result for the same inputs and have no side effects beyond calling setMeasuredDimension().

The Measure Cache — mMeasureCache

// mMeasureCache: LongSparseLongArray
// Key:   (widthSpec.toLong() shl 32) or heightSpec  — two specs packed
// Value: (measuredWidth.toLong() shl 32) or measuredHeight

// measure() packs specs → cache lookup → hit + no FORCE_LAYOUT? skip onMeasure()
// forceLayout() clears this cache and sets PFLAG_FORCE_LAYOUT

PFLAG3_MEASURE_NEEDED_BEFORE_LAYOUT

A subtle race condition: what if a View is invalidated (its content changes) between the measure pass and the layout pass of the same traversal? The framework handles this with PFLAG3_MEASURE_NEEDED_BEFORE_LAYOUT. When layout() is called and this flag is set, the View re-measures itself before proceeding with layout — ensuring the layout pass uses up-to-date measured dimensions.

requestLayout() vs forceLayout() vs invalidate()

MethodTriggersPropagationUse Case
requestLayout() Measure + Layout Walks up to ViewRootImpl, schedules full traversal Size or position may have changed
forceLayout() Clears measure cache, sets FORCE_LAYOUT flag Does NOT propagate up — only affects this View Used internally by ViewGroups to force child re-measurement
invalidate() Draw only Marks dirty region, schedules draw traversal Visual appearance changed but size/position unchanged

isLayoutRequested()

// True between requestLayout() and next layout pass completion — avoid redundant requests
if (!view.isLayoutRequested) view.requestLayout()

// Batch layout changes in custom ViewGroups:
child1.layoutParams.width = newWidth1
child2.layoutParams.width = newWidth2
if (!isLayoutRequested) requestLayout()  // single request for both changes
Staff Insight — Why forceLayout() Exists Separately

forceLayout() is a subtle but critical API. Unlike requestLayout(), it does not walk up the tree to schedule a traversal — it only clears the local measure cache. This is used by ViewGroups like LinearLayout during multi-pass measurement: after the first pass, it calls forceLayout() on children that need re-measurement, then calls child.measure() again with new specs. If it used requestLayout() instead, it would trigger a new traversal from the root — an infinite loop. Understanding this distinction shows you grasp the internal traversal machinery, not just the public API.

Measure → Layout Dimension Mismatch

override fun onLayout(changed: Boolean, l: Int, t: Int, r: Int, b: Int) {
    val child = getChildAt(0)
    child.layout(0, 0, 300, 150)  // child measured 200×100 → parent overrides

    // child.measuredWidth = 200  (measure pass)
    // child.width = 300          (layout pass) — DIFFERENT, and legal
}
// LinearLayout with weight: measure → child wants 100px → weighted → gets 250px

Key Takeaways

  • MeasureSpec is a 32-bit int packing mode + size — learn the 3×3 parent-child mapping table cold.
  • onMeasure() must call setMeasuredDimension() — this is a hard contract, not a suggestion.
  • Multi-pass measurement creates exponential blowup — the #1 reason ConstraintLayout exists.
  • measuredWidth and width can differ — measured is the child's preference, width is the parent's decision.
  • forceLayout() clears the local cache without scheduling a traversal — critical for multi-pass ViewGroups.
  • The measure cache (mMeasureCache) prevents redundant onMeasure() calls — but forceLayout() bypasses it.
