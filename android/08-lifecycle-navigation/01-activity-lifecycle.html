01 — Activity Lifecycle Deep Dive

AOSP internals of how AMS drives lifecycle transitions, the ClientTransaction mechanism, ActivityRecord states, launch sequences, and staff-level corner cases

1. The Lifecycle State Machine

An Activity has 7 lifecycle callbacks, but the system thinks in terms of states, not callbacks. Callbacks are the edges between states. Understanding this distinction is critical at L6.

What Each Callback Means — From the System's Perspective

CallbackSystem MeaningWhat Juniors Miss
onCreate()The Activity object has been instantiated via Instrumentation.newActivity(). The PhoneWindow is created but no view hierarchy is attached to WMS yet.The window does not exist in SurfaceFlinger at this point. No pixels are drawn.
onStart()The Activity is becoming visible but is not yet in the foreground. The window may be partially visible (e.g., behind a transparent Activity).Visibility ≠ interactability. The Activity can be visible but not receive input.
onResume()The Activity is in the foreground and eligible to receive input. mResumedActivity in the Task now points to this Activity.onResume() does NOT mean the first frame has been drawn. The window becomes visible only after ViewRootImpl.performTraversals() completes.
onPause()Another Activity is being launched or the device is going to sleep. The system needs this Activity to yield the foreground. The mResumedActivity pointer is cleared.AMS waits up to 500ms for onPause() to complete before force-proceeding. Blocking here delays the next Activity's launch.
onStop()The Activity is no longer visible. Its window surface may be destroyed. The system considers this Activity a candidate for reclamation under memory pressure.onStop() is posted as an IdleHandler message — it runs only when the main thread's MessageQueue is idle. This is why there's a visible delay between onPause() and onStop().
onDestroy()The Activity is being permanently torn down — either finish() was called or the system is reclaiming resources. The window is removed from WMS.If the process is killed by LMK, onDestroy() is never called. You cannot rely on it for cleanup.
onRestart()The Activity is transitioning from STOPPED back to STARTED. This only happens when navigating back to a stopped Activity (e.g., pressing Back from Activity B to Activity A).This is the only callback that distinguishes a fresh start from a re-start. Useful for refreshing stale data.

2. AOSP Internals — How AMS Drives the Lifecycle

ActivityRecord — The System's View of Your Activity

Every Activity instance in your app has a corresponding ActivityRecord object in system_server. This is the authoritative source of truth for the Activity's state.

// frameworks/base/services/core/java/com/android/server/wm/ActivityRecord.java

// Key fields in ActivityRecord:
// - mState: State        → current lifecycle state (INITIALIZING, RESUMED, etc.)
// - app: WindowProcessController → the process hosting this Activity
// - task: Task           → the Task (back stack) this Activity belongs to
// - resultTo: ActivityRecord → who to deliver onActivityResult() to
// - info: ActivityInfo   → parsed manifest metadata
// - mVisibleRequested    → whether the system wants this Activity visible
// - finishing: boolean   → true after finish() is called
// - mLaunchMode          → standard, singleTop, singleTask, singleInstance

ClientTransaction — The Batched Lifecycle Mechanism

Since Android P (API 28), lifecycle transitions are delivered as batched transactions via the ClientTransaction framework. This replaced the older individual Binder calls like schedulePauseActivity().

// The transaction model:
//
// system_server (AMS)                          app process
// ─────────────────                          ───────────
//
// 1. AMS builds a ClientTransaction:
//    ClientTransaction t = ClientTransaction.obtain(appThread, activityToken)
//    t.addCallback(LaunchActivityItem)        ← "execute this action"
//    t.setLifecycleStateRequest(ResumeActivityItem)  ← "end in this state"
//
// 2. AMS sends it via Binder:
//    t.schedule()
//      → IApplicationThread.scheduleTransaction(t)
//      → Binder call to app process
//
// 3. App process receives it:
//    ActivityThread.ApplicationThread.scheduleTransaction(t)
//      → sendMessage(EXECUTE_TRANSACTION, t)
//      → posted to main thread's Handler
//
// 4. Main thread processes it:
//    TransactionExecutor.execute(t)
//      → executeCallbacks(t)    ← run LaunchActivityItem
//      → executeLifecycleState(t) ← transition to RESUMED state

// ClientTransactionItem subtypes:
// ┌──────────────────────────┬──────────────────────────────────┐
// │ Callback Items           │ Lifecycle State Items            │
// ├──────────────────────────┼──────────────────────────────────┤
// │ LaunchActivityItem       │ ResumeActivityItem (ON_RESUME)   │
// │ NewIntentItem            │ PauseActivityItem  (ON_PAUSE)    │
// │ ActivityResultItem       │ StopActivityItem   (ON_STOP)     │
// │ ActivityConfigurationChangeItem │ DestroyActivityItem (ON_DESTROY)│
// │ TopResumedActivityChangeItem    │                          │
// └──────────────────────────┴──────────────────────────────────┘

TransactionExecutor — Lifecycle Path Resolution

The most subtle part of the system: TransactionExecutor determines which intermediate callbacks to invoke when transitioning between non-adjacent states.

// frameworks/base/core/java/android/app/servertransaction/TransactionExecutor.java

// executeLifecycleState() does this:
// 1. Get current state of the Activity (e.g., ON_START)
// 2. Get target state from the transaction (e.g., ON_RESUME)
// 3. Compute the path: ON_START → ON_RESUME
// 4. Execute each intermediate callback

// The lifecycle path array (cycleToPath):
//   ON_CREATE → ON_START → ON_RESUME → ON_PAUSE → ON_STOP → ON_DESTROY
//
// Example paths:
//   Current: ON_CREATE, Target: ON_RESUME
//     → calls: onStart(), onResume()
//
//   Current: ON_RESUME, Target: ON_STOP
//     → calls: onPause(), onStop()
//
//   Current: ON_STOP, Target: ON_RESUME
//     → calls: onRestart(), onStart(), onResume()
//
// This is WHY you never see onPause() without onStop() following
// (unless another Activity becomes visible before onStop is dispatched)
Staff-Level Insight — The IdleHandler Trick for onStop

onStop() is not dispatched immediately after onPause(). AMS schedules it as an IdleHandler on the app's main thread via StopActivityItem. The IdleHandler fires only when the MessageQueue has no more pending messages — this allows the new Activity's onResume() and first frame to complete before the old Activity is stopped. If the main thread never goes idle (e.g., heavy animation), AMS has a 10-second timeout (STOP_TIMEOUT) after which it force-stops the Activity. This is why you can observe onPause() → [new Activity onResume + first frame] → onStop() with a noticeable gap. In a Systrace, you'll see the stop message sitting in the queue behind rendering messages.

3. ActivityRecord States in AOSP

The system tracks Activity states with finer granularity than the 7 callbacks expose. ActivityRecord.mState uses these values:

StateMeaning
INITIALIZINGActivityRecord created in system_server, but no lifecycle transaction sent yet
STARTEDonStart() completed — visible but not in foreground
RESUMEDonResume() completed — foreground, receiving input
PAUSINGPauseActivityItem sent, waiting for app to acknowledge
PAUSEDApp confirmed onPause() completed
STOPPINGStopActivityItem sent, waiting for app to acknowledge
STOPPEDApp confirmed onStop() completed
FINISHINGfinish() was called — Activity is winding down
DESTROYINGDestroyActivityItem sent
DESTROYEDActivity is fully torn down
// The mResumedActivity pointer:
//
// Each Task maintains a single mResumedActivity reference.
// Only ONE Activity per Task can be in RESUMED state at a time.
//
// Task (back stack)
// ├── ActivityRecord A  [STOPPED]
// ├── ActivityRecord B  [STOPPED]
// └── ActivityRecord C  [RESUMED]  ← mResumedActivity points here
//
// Exception: Multi-resume mode (Android 10+, freeform/split-screen)
// In multi-resume, Activities in DIFFERENT Tasks can be simultaneously
// RESUMED. But within a single Task, the invariant still holds.
//
// The system uses this pointer to:
// 1. Know which Activity to pause when launching a new one
// 2. Deliver input events to the correct Activity
// 3. Determine the "top" Activity for recents/screenshots

4. Activity Launch Sequence — Step by Step

From startActivity() to the first visible frame — every step that matters.

// Step 1: App calls startActivity(intent)
//   → Activity.startActivity()
//   → Instrumentation.execStartActivity()
//   → ActivityTaskManager.getService().startActivity()
//   → Binder IPC to system_server

// Step 2: Intent Resolution (system_server)
//   → ActivityTaskManagerService.startActivity()
//   → ActivityStarter.execute()
//     → resolveIntent() — PackageManagerService resolves the target
//     → resolveActivity() — finds the ActivityInfo
//     → startActivityUnchecked()
//       → Determines Task: create new or reuse existing
//       → Applies launch flags (FLAG_ACTIVITY_NEW_TASK, etc.)
//       → Creates ActivityRecord for the new Activity

// Step 3: Pause the Current Activity
//   → Task.startPausingLocked()
//   → Sends PauseActivityItem to the current foreground app
//   → Waits up to 500ms (PAUSE_TIMEOUT) for acknowledgment
//   → Current Activity: onPause() executes

// Step 4: Process Resolution
//   → ActivityTaskSupervisor.startSpecificActivity()
//   → Is the target app's process running?
//     YES → proceed to Step 5
//     NO  → AMS.startProcessLocked()
//           → Zygote.forkAndSpecialize()
//           → New process calls ActivityThread.main()
//           → ActivityThread.attach() → binds to AMS
//           → AMS.attachApplicationLocked() → proceed to Step 5

// Step 5: Launch the Activity
//   → ClientLifecycleManager.scheduleTransaction(
//       LaunchActivityItem + ResumeActivityItem)
//   → Binder IPC to app process

// Step 6: App Process Handles Launch
//   → TransactionExecutor.execute()
//   → ActivityThread.handleLaunchActivity()
//     → Instrumentation.newActivity() — reflective instantiation
//     → Activity.attach() — creates PhoneWindow, sets base context
//     → Instrumentation.callActivityOnCreate()
//       → Activity.onCreate() — YOUR CODE RUNS HERE
//       → setContentView() → LayoutInflater inflates XML → View tree created
//     → Activity.onStart()
//   → ActivityThread.handleResumeActivity()
//     → Activity.onResume()
//     → WindowManager.addView(decorView)
//       → ViewRootImpl created
//       → ViewRootImpl.setView(decorView)
//         → Session.addToDisplayAsUser() → WMS adds the window
//         → requestLayout() → schedules first traversal

// Step 7: First Frame
//   → Choreographer.postCallback(CALLBACK_TRAVERSAL)
//   → On next VSYNC:
//     → ViewRootImpl.performTraversals()
//       → performMeasure() → View tree measured
//       → performLayout()  → View tree laid out
//       → performDraw()    → View tree drawn to Surface
//     → RenderThread submits frame to SurfaceFlinger
//   → Window becomes visible to the user
Warning — The Visibility Gap

There is a gap between onResume() and the window actually becoming visible. During this gap, the Activity is "resumed" but the user still sees the previous Activity (or a blank window). This is why reportFullyDrawn() exists — it lets you signal when your content is actually ready, not just when the lifecycle says you're "resumed".

5. Activity Finish & Destruction

The finish() Path

// Activity.finish()
//   → ActivityClient.getInstance().finishActivity(mToken, resultCode, resultData)
//   → Binder IPC to system_server
//
// In system_server:
//   → ActivityRecord.finishIfPossible()
//     → Sets finishing = true
//     → If Activity is RESUMED:
//       → startPausingLocked() — pause first, then proceed
//     → If Activity is PAUSED/STOPPED:
//       → finishCurrentActivityLocked()
//         → Sends DestroyActivityItem to app process
//
// In app process:
//   → TransactionExecutor resolves path to ON_DESTROY
//   → ActivityThread.handleDestroyActivity()
//     → Activity.onDestroy()
//     → WindowManager.removeView() — removes from WMS
//     → ActivityClient.activityDestroyed(token) — notifies AMS

// The critical distinction in onDestroy():
override fun onDestroy() {
    super.onDestroy()

    if (isFinishing) {
        // Permanent teardown — release everything
        // Cancel all coroutines, close DB connections, unregister receivers
        releaseAllResources()
    } else if (isChangingConfigurations) {
        // Config change — Activity will be recreated immediately
        // DON'T release ViewModel-managed resources
        // DON'T cancel ongoing network requests held by ViewModel
    } else {
        // System reclamation — save state in onSaveInstanceState (already called)
        // This path is rare when the process is still alive
    }
}
Danger — Never Rely on onDestroy for Critical Cleanup

When LMK kills the process, no lifecycle callbacks fire. If you need to persist data, do it in onStop() or onSaveInstanceState(). The only reliable signal for process death after the fact is ApplicationExitInfo (API 30+), which you can query on next launch.

6. onNewIntent() and onActivityResult() Timing

onNewIntent()

// onNewIntent() is called when an existing Activity instance is reused
// instead of creating a new one. This happens with:
//
// 1. launchMode="singleTop" + Activity is at top of stack
// 2. launchMode="singleTask" — always reuses existing instance
// 3. launchMode="singleInstance" — always reuses existing instance
// 4. FLAG_ACTIVITY_SINGLE_TOP on the Intent
//
// The callback ordering depends on the Activity's current state:
//
// Case A: Activity is RESUMED (already in foreground)
//   onPause() → onNewIntent() → onResume()
//
// Case B: Activity is STOPPED (in background)
//   onNewIntent() → onRestart() → onStart() → onResume()
//
// IMPORTANT: The Intent returned by getIntent() is NOT updated automatically.
// You must call setIntent(intent) in onNewIntent() if you want getIntent()
// to return the new Intent in subsequent lifecycle callbacks.

override fun onNewIntent(intent: Intent) {
    super.onNewIntent(intent)
    setIntent(intent) // Critical — without this, getIntent() returns the OLD intent
    handleDeepLink(intent)
}

// In AOSP, this is delivered via NewIntentItem in the ClientTransaction:
// ClientTransaction {
//   callbacks: [NewIntentItem(intent)]
//   lifecycleState: ResumeActivityItem
// }
// TransactionExecutor runs NewIntentItem.execute() first, then resolves
// the lifecycle path to ON_RESUME.

onActivityResult() and the Modern Activity Result API

// Legacy: onActivityResult() timing
//   Activity B finishes → Activity A receives:
//   onActivityResult() → onRestart() → onStart() → onResume()
//
// The result is delivered BEFORE onResume(), which means you can
// use the result data to update UI in onStart().

// Modern: registerForActivityResult() — the lifecycle-safe replacement
// Why it's better:
// 1. Decouples result handling from the Activity class
// 2. Survives configuration changes (registered in onCreate, not overridden)
// 3. Type-safe contracts (ActivityResultContracts)

class MyActivity : AppCompatActivity() {

    // MUST be registered before STARTED state (onCreate or class init)
    private val pickImage = registerForActivityResult(
        ActivityResultContracts.GetContent()
    ) { uri: Uri? ->
        uri?.let { displayImage(it) }
    }

    // Under the hood:
    // - ActivityResultRegistry stores the callback keyed by a unique string
    // - On config change, the Activity is recreated, registerForActivityResult()
    //   re-registers with the SAME key, so the pending result is matched
    // - The callback is lifecycle-aware: it's only invoked when the Activity
    //   is at least STARTED (uses LifecycleEventObserver internally)

    fun onPickImageClick() {
        pickImage.launch("image/*")
    }
}

7. Staff-Level Corner Cases

The "Pause-Then-Launch" Race

// When Activity B is launched while Activity A is in foreground:
//
// 1. AMS sends PauseActivityItem to Activity A
// 2. AMS waits for A to acknowledge pause (up to 500ms)
// 3. ONLY THEN does AMS send LaunchActivityItem for Activity B
//
// But what if A's onPause() takes too long?
//   → After PAUSE_TIMEOUT (500ms), AMS proceeds anyway
//   → Activity B is launched even though A hasn't finished pausing
//   → A's onPause() eventually completes asynchronously
//
// This means: if you do heavy work in onPause(), the user sees a
// delayed transition. AMS logs: "Activity pause timeout for ActivityRecord{...}"
//
// In multi-window mode, the race is more complex because both
// Activities can be RESUMED simultaneously (multi-resume).

moveTaskToBack() vs Home Button

// Home button press:
//   → Launches the Home Activity (launcher) with FLAG_ACTIVITY_NEW_TASK
//   → Current Activity: onPause() → onStop()
//   → The Task is moved to background by the launcher taking focus
//
// moveTaskToBack(true):
//   → Directly tells AMS to move the entire Task behind all others
//   → Current Activity: onPause() → onStop()
//   → No new Activity is launched — the previous Task's top Activity resumes
//
// Key difference: moveTaskToBack doesn't create a new Task entry.
// If the user presses Recents, the Task is still there.
// But the lifecycle callbacks are identical: onPause() → onStop().

Visible but Not RESUMED

// An Activity can be STARTED (visible) but NOT RESUMED in these cases:
//
// 1. Transparent Activity on top:
//    Activity A [STARTED, visible] ← partially visible behind B
//    Activity B [RESUMED, transparent theme]
//    → A receives onPause() but NOT onStop() because it's still visible
//
// 2. Dialog-themed Activity:
//    Activity A [STARTED, visible] ← visible around the dialog
//    Activity B [RESUMED, Theme.Dialog]
//    → Same as above: A is paused but not stopped
//
// 3. Multi-window (split-screen):
//    Activity A [STARTED or RESUMED] ← in one split
//    Activity B [RESUMED]            ← in the other split
//    → Pre-Android 10: only the focused Activity is RESUMED
//    → Android 10+: both can be RESUMED (multi-resume)
//
// 4. Picture-in-Picture:
//    Activity A [PAUSED] ← in PiP window (visible but not interactive)
//    Activity B [RESUMED] ← full-screen
//    → PiP Activity is paused because it doesn't receive input
//    → Android 12+ changed this: PiP Activities can be STARTED

android:noHistory and android:excludeFromRecents

// android:noHistory="true"
//   → Activity is finished automatically when the user navigates away
//   → Specifically: when another Activity comes to the foreground
//   → onDestroy() IS called (unlike LMK kill)
//   → Use case: splash screens, payment confirmation screens
//   → Gotcha: the Activity is also destroyed when the user presses Home,
//     so returning from Recents creates a NEW instance

// android:excludeFromRecents="true"
//   → Activity's Task is not shown in the Recents screen
//   → The Activity still exists and follows normal lifecycle
//   → Use case: authentication screens, notification trampolines
//   → Can also be set dynamically via FLAG_ACTIVITY_EXCLUDE_FROM_RECENTS

Trampoline Activities and Android 12 Restrictions

// The "trampoline" pattern:
//   Activity A (invisible) → startActivity(B) → finish()
//   Used for: notification click routing, deep link resolution, auth gates
//
// Android 12 (API 31) restriction:
//   Apps targeting API 31+ cannot start Activities from Services or
//   BroadcastReceivers that are in the background, UNLESS they have
//   a visible window or fall under specific exemptions.
//
//   This broke the common pattern of:
//   NotificationReceiver.onReceive() → startActivity(TrampolineActivity)
//
//   Fix: use PendingIntent that directly targets the destination Activity
//   instead of routing through a trampoline.
//
// The trampoline Activity itself still works when launched from the
// foreground — the restriction is about background Activity starts.

// Modern replacement for notification trampolines:
val pendingIntent = PendingIntent.getActivity(
    context,
    requestCode,
    Intent(context, DestinationActivity::class.java).apply {
        putExtra("deepLinkData", data)
    },
    PendingIntent.FLAG_IMMUTABLE or PendingIntent.FLAG_UPDATE_CURRENT
)

val notification = NotificationCompat.Builder(context, channelId)
    .setContentIntent(pendingIntent) // Direct to destination, no trampoline
    .build()
Staff-Level Insight — ClientTransaction Batching and Lifecycle Coalescing

AMS can batch multiple lifecycle transitions into a single ClientTransaction. For example, when an Activity is launched and the current one needs to be stopped, AMS may send a single transaction with PauseActivityItem as a callback and StopActivityItem as the lifecycle state request. TransactionExecutor then resolves the full path: ON_RESUME → ON_PAUSE (from callback) → ON_STOP (from lifecycle state). This batching is why you sometimes see onPause() and onStop() fire in rapid succession with no interleaving messages — they're part of the same transaction execution. Understanding this is critical for debugging lifecycle-related race conditions: if you post a Runnable to the main Handler in onPause(), it will NOT execute before onStop() if they're in the same transaction, because the TransactionExecutor runs synchronously through the entire path.

Warning — ActivityThread Handler Methods

Each lifecycle stage has a corresponding ActivityThread.handle* method: handleLaunchActivity(), handleResumeActivity(), handlePauseActivity(), handleStopActivity(), handleDestroyActivity(). These are where the actual work happens — creating the Activity object, attaching the window, calling your callbacks. If you're reading AOSP traces or debugging with breakpoints, these are the methods to watch. They're invoked by the corresponding ClientTransactionItem.execute() methods, not directly by the Handler.

8. AMS vs ATMS Split (Android 10)

Before Android 10, ActivityManagerService (AMS) handled everything: Activity lifecycle, process management, broadcast dispatch, and content providers. In Android 10, Google split Activity-related logic into ActivityTaskManagerService (ATMS) while AMS retained process/broadcast/provider management. Key implications:

// Pre-Android 10:
//   AMS.startActivity() → AMS.startActivityAsUser() → ActivityStarter
//
// Android 10+:
//   ATMS.startActivity() → ATMS.startActivityAsUser() → ActivityStarter
//   AMS still handles: getRunningAppProcesses(), forceStopPackage(), etc.
//
// The split was motivated by:
// 1. Separation of concerns (Activity management vs process management)
// 2. Multi-display support (ATMS manages per-display task stacks)
// 3. Code maintainability (AMS was 20,000+ lines)
//
// Both services run in system_server. ATMS holds the ActivityTaskManagerInternal
// interface that AMS uses when it needs Activity information.

9. Activity Lifecycle and Bound Services

When an Activity binds to a Service, the binding lifecycle must be carefully managed relative to Activity lifecycle:

// WRONG — binding in onCreate, unbinding in onDestroy
// Problem: during config change, onDestroy is called → unbind → Service may stop
// if this was the last binding. Then onCreate → bind again → Service restarts.
// This causes unnecessary Service restarts on every rotation.

// CORRECT — bind/unbind aligned with visibility:
override fun onStart() {
    super.onStart()
    bindService(serviceIntent, connection, Context.BIND_AUTO_CREATE)
}

override fun onStop() {
    super.onStop()
    unbindService(connection)
}

// Or for config-change-safe binding, use ViewModel + ServiceConnection:
class MyViewModel : ViewModel() {
    private val binder = MutableLiveData<IMyService?>()
    private val connection = object : ServiceConnection {
        override fun onServiceConnected(name: ComponentName, service: IBinder) {
            binder.value = IMyService.Stub.asInterface(service)
        }
        override fun onServiceDisconnected(name: ComponentName) {
            binder.value = null
        }
    }
    // ViewModel survives config change → ServiceConnection survives
    // But: ViewModel does NOT survive process death → must rebind
}

@Test
fun activityRecreation_preservesState() {
    val scenario = ActivityScenario.launch(MyActivity::class.java)

    // Move to specific state
    scenario.moveToState(Lifecycle.State.CREATED)

    // Simulate config change (destroy + recreate)

    }

@Test

    val scenario = ActivityScenario.launch(MyActivity::class.java)

Staff-Level Insight — Activity Leak Patterns

The most insidious Activity leaks at L6 scale: (1) Anonymous inner class in static field: an OnClickListener lambda capturing this stored in a companion object. (2) Handler/Runnable leak: a delayed Runnable posted to a Handler holds a reference to the Activity via the enclosing class; if the Activity is destroyed before the Runnable fires, it's leaked until the message is processed. Fix: use WeakReference or lifecycleScope.launch { delay(...) }. (3) InputMethodManager leak: a known framework bug where IMM holds a reference to the focused View's root, which chains back to the Activity. Workaround: clear focus in onDestroy(). (4) Registered but unregistered callbacks: registerReceiver() without unregisterReceiver(), SensorManager.registerListener() without unregister. Always pair registration with lifecycle-aware unregistration.
