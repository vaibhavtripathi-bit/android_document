Zustand: Minimal External Store

Zustand creates a store outside the React tree and uses useSyncExternalStore for subscription-based re-renders. No providers needed.

import { create } from 'zustand';

// Store lives outside React — no Provider needed
const useBearStore = create((set, get) => ({
  bears: 0,
  fish: 0,
  increasePopulation: () => set((state) => ({ bears: state.bears + 1 })),
  removeAllBears: () => set({ bears: 0 }),
  fetchFish: async () => {
    const response = await fetch('/api/fish');
    const fish = await response.json();
    set({ fish });
  },
  // get() reads current state (useful in async actions)
  eatFish: () => {
    const { fish } = get();
    if (fish > 0) set({ fish: fish - 1 });
  },
}));

// Usage — selector-based subscription:
function BearCounter() {
  const bears = useBearStore((state) => state.bears);
  // Only re-renders when bears changes, NOT when fish changes
  return <h1>{bears} bears</h1>;
}

function Controls() {
  const increase = useBearStore((state) => state.increasePopulation);
  // Stable function reference — this component rarely re-renders
  return <button onClick={increase}>Add bear</button>;
}

Zustand Internals

// Simplified Zustand create implementation:
function create(createState) {
  let state;
  const listeners = new Set();

  const setState = (partial) => {
    const nextState = typeof partial === 'function'
      ? partial(state)
      : partial;
    if (!Object.is(nextState, state)) {
      const previousState = state;
      state = Object.assign({}, state, nextState);
      listeners.forEach((listener) => listener(state, previousState));
    }
  };

  const getState = () => state;

  const subscribe = (listener) => {
    listeners.add(listener);
    return () => listeners.delete(listener);
  };

  // Initialize state
  state = createState(setState, getState, { setState, getState, subscribe });

  // The hook uses useSyncExternalStore internally:
  function useStore(selector = (s) => s) {
    return useSyncExternalStore(
      subscribe,
      () => selector(getState()),
    );
  }

  useStore.getState = getState;
  useStore.setState = setState;
  useStore.subscribe = subscribe;
  return useStore;
}

Zustand Middleware

import { create } from 'zustand';
import { persist, devtools, immer } from 'zustand/middleware';

const useStore = create(
  devtools(
    persist(
      immer((set) => ({
        todos: [],
        addTodo: (text) => set((state) => {
          state.todos.push({ id: Date.now(), text, done: false });
        }),
        toggleTodo: (id) => set((state) => {
          const todo = state.todos.find(t => t.id === id);
          if (todo) todo.done = !todo.done;
        }),
      })),
      { name: 'todo-storage' } // localStorage key
    ),
    { name: 'TodoStore' } // DevTools name
  )
);

Jotai: Atomic State Model

Jotai takes a bottom-up approach: state is composed from individual atoms. Each atom is an independent unit of state, and derived atoms compute values from other atoms.

import { atom, useAtom, useAtomValue, useSetAtom } from 'jotai';

// Primitive atoms (writable):
const countAtom = atom(0);
const nameAtom = atom('');
const todosAtom = atom([]);

// Derived atom (read-only, computed):
const doubleCountAtom = atom((get) => get(countAtom) * 2);

const completedTodosAtom = atom((get) => {
  const todos = get(todosAtom);
  return todos.filter(t => t.completed);
});

const todoStatsAtom = atom((get) => ({
  total: get(todosAtom).length,
  completed: get(completedTodosAtom).length,
}));

// Derived atom (read-write):
const uppercaseNameAtom = atom(
  (get) => get(nameAtom).toUpperCase(),     // read
  (get, set, newName) => set(nameAtom, newName) // write
);

// Async atom:
const userAtom = atom(async (get) => {
  const id = get(userIdAtom);
  const response = await fetch(`/api/users/${id}`);
  return response.json();
});

// Usage:
function Counter() {
  const [count, setCount] = useAtom(countAtom);
  const doubled = useAtomValue(doubleCountAtom); // read-only
  const setName = useSetAtom(nameAtom);          // write-only

  return (
    <div>
      <p>{count} (doubled: {doubled})</p>
      <button onClick={() => setCount(c => c + 1)}>+</button>
    </div>
  );
}

Jotai Internals

// Jotai uses a WeakMap-based store that tracks atom dependencies:

// Simplified atom store:
class Store {
  atomStateMap = new WeakMap(); // atom → { value, dependents, listeners }

  readAtom(atom) {
    const atomState = this.getAtomState(atom);
    if (atom.read) {
      // Derived atom: track dependencies
      const get = (depAtom) => {
        atomState.dependencies.add(depAtom);
        return this.readAtom(depAtom);
      };
      atomState.value = atom.read(get);
    }
    return atomState.value;
  }

  writeAtom(atom, value) {
    const atomState = this.getAtomState(atom);
    atomState.value = typeof value === 'function'
      ? value(atomState.value)
      : value;
    // Notify dependents (derived atoms that read this atom)
    this.notifyDependents(atom);
    // Notify subscribers (React components)
    atomState.listeners.forEach(l => l());
  }
}

// Key insight: Jotai tracks which atoms depend on which,
// so when atom A changes, only derived atoms that read A
// recompute, and only components subscribed to those atoms re-render.
Jotai Dependency Graph: countAtom (primitive) │ ├──▶ doubleCountAtom (derived: count * 2) │ │ │ └──▶ Component A (subscribes to doubleCount) │ └──▶ Component B (subscribes to count) nameAtom (primitive) │ └──▶ Component C (subscribes to name) When countAtom changes: ✓ doubleCountAtom recomputes ✓ Component A re-renders ✓ Component B re-renders ✗ Component C does NOT re-render (no dependency)

Signals: Fine-Grained Reactivity

Signals bypass React's re-rendering entirely. Instead of re-rendering components, signals update DOM nodes directly — similar to how Solid.js and Svelte work.

// Preact Signals with React adapter:
import { signal, computed, effect } from '@preact/signals-react';

// Create signals (reactive values):
const count = signal(0);
const name = signal('Alice');

// Computed (derived, auto-tracked):
const doubled = computed(() => count.value * 2);

// Effect (side effect, auto-tracked):
effect(() => {
  console.log(`Count is ${count.value}`);
  // Automatically re-runs when count changes
});

// In React components — signals bypass React re-rendering:
function Counter() {
  // This component does NOT re-render when count changes!
  // The signal updates the DOM text node directly.
  return (
    <div>
      <p>Count: {count}</p>  {/* signal auto-subscribes */}
      <p>Doubled: {doubled}</p>
      <button onClick={() => count.value++}>+</button>
    </div>
  );
}

// How it works: @preact/signals-react patches React's rendering
// to detect signal usage in JSX and set up direct DOM subscriptions.
// When count.value changes, it updates only the text node "Count: X"
// without re-rendering the Counter component.
Signals and React: Compatibility Concerns

Signals bypass React's rendering model, which can conflict with React's assumptions about when and how components update. The React team has expressed concerns about this approach. Signals work well in frameworks designed for them (Solid, Preact, Svelte) but are an escape hatch in React. Use with caution in production React apps.

Performance Characteristics

// Zustand: O(1) selector comparison per subscriber
// When state changes, each subscriber's selector runs.
// If selector result is unchanged (Object.is), component skips re-render.
const bears = useBearStore(s => s.bears); // selector runs on every state change

// Jotai: O(1) per atom dependency
// When an atom changes, only atoms that depend on it recompute.
// Only components subscribed to changed atoms re-render.
const [count] = useAtom(countAtom); // only re-renders when countAtom changes

// Signals: O(1) per signal, no React re-render
// Signal changes update DOM directly via subscriptions.
// Zero React re-renders for signal-driven UI updates.
return <p>{count}</p>; // text node updates without component re-render
Re-render Comparison (100 components, 1 state change): Zustand (with selectors): State changes → 100 selectors run → 1 selector returns new value → 1 component re-renders Cost: 100 selector comparisons + 1 re-render Jotai (atoms): Atom changes → dependency graph traversed → 1 atom affected → 1 component re-renders Cost: 1 dependency lookup + 1 re-render Signals: Signal changes → 1 DOM text node updated → 0 component re-renders Cost: 1 DOM mutation Context (no selectors): Context changes → ALL 100 consumers re-render Cost: 100 re-renders

Choosing the Right Tool

Architect Insight — Selection Guide

Zustand: Best for teams coming from Redux who want less boilerplate. Single store, familiar patterns, excellent middleware. Choose when you want a simple, proven solution. Jotai: Best for apps with many independent pieces of state that compose. Bottom-up model avoids the "mega-store" problem. Choose when state is naturally atomic. Signals: Best for performance-critical UIs where React re-renders are the bottleneck. Choose only if you've profiled and confirmed re-renders are the issue — the React compatibility trade-offs are significant.

// Decision tree:
// 1. Do you need a single global store? → Zustand
// 2. Is state naturally composed of independent atoms? → Jotai
// 3. Are React re-renders your measured bottleneck? → Signals (with caution)
// 4. Do you need middleware, DevTools, time-travel? → Zustand or Redux
// 5. Are you building a new app with simple state? → Start with useState/useReducer
// 6. Is it server data? → TanStack Query (not a state manager)
