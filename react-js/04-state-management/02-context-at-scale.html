Context: Dependency Injection vs Global State

Context was designed for dependency injection — passing values deep into the tree without prop drilling. It was not designed as a general-purpose state management solution, though it can serve that role for infrequently-changing values.

// GOOD use of context: dependency injection (rarely changes)
const ThemeContext = createContext('light');
const LocaleContext = createContext('en');
const AuthContext = createContext(null);
const RouterContext = createContext(null);

// These values change infrequently (theme toggle, login/logout, navigation).
// Context is ideal here.

// PROBLEMATIC use of context: frequently-changing state
const SearchContext = createContext({ query: '', results: [] });
// If query changes on every keystroke, EVERY consumer re-renders.
// Even consumers that only read 'results' re-render when 'query' changes.
Architect Insight

Context has no built-in selector mechanism. When the context value changes, every component that calls useContext(MyContext) re-renders, regardless of which part of the value they use. This is the fundamental limitation that makes context unsuitable for high-frequency state updates at scale.

How Context Re-Rendering Works

// When a Provider's value changes:
// 1. React compares old and new value with Object.is
// 2. If different, it marks ALL consumers for re-render
// 3. Consumers re-render even if wrapped in React.memo
//    (context bypasses memo's shallow prop comparison)

function App() {
  const [theme, setTheme] = useState('light');
  const [count, setCount] = useState(0);

  // BUG: new object every render → all consumers re-render
  // even when only count changes (theme didn't change)
  return (
    <ThemeContext.Provider value={{ theme, setTheme }}>
      <button onClick={() => setCount(c => c + 1)}>{count}</button>
      <ThemedComponent /> {/* re-renders on count change! */}
    </ThemeContext.Provider>
  );
}

// FIX: memoize the context value
function App() {
  const [theme, setTheme] = useState('light');
  const [count, setCount] = useState(0);

  const themeValue = useMemo(
    () => ({ theme, setTheme }),
    [theme] // only recreate when theme changes
  );

  return (
    <ThemeContext.Provider value={themeValue}>
      <button onClick={() => setCount(c => c + 1)}>{count}</button>
      <ThemedComponent /> {/* no longer re-renders on count change */}
    </ThemeContext.Provider>
  );
}

Context Splitting Strategies

The primary technique for scaling context is splitting: separate contexts for separate concerns.

Strategy 1: Split by Concern

// BEFORE: monolithic context
const AppContext = createContext({
  user: null,
  theme: 'light',
  locale: 'en',
  notifications: [],
  setUser: () => {},
  setTheme: () => {},
  setLocale: () => {},
  addNotification: () => {},
});
// Problem: changing notifications re-renders theme consumers

// AFTER: separate contexts per concern
const UserContext = createContext(null);
const ThemeContext = createContext('light');
const LocaleContext = createContext('en');
const NotificationContext = createContext({ items: [], add: () => {} });

// Now changing notifications only re-renders notification consumers

Strategy 2: Split State from Dispatch

// Separate read-only state from action dispatchers
const TodoStateContext = createContext([]);
const TodoDispatchContext = createContext(null);

function TodoProvider({ children }) {
  const [todos, dispatch] = useReducer(todoReducer, []);

  return (
    <TodoStateContext.Provider value={todos}>
      <TodoDispatchContext.Provider value={dispatch}>
        {children}
      </TodoDispatchContext.Provider>
    </TodoStateContext.Provider>
  );
}

// Components that only dispatch actions don't re-render on state changes:
function AddTodoButton() {
  const dispatch = useContext(TodoDispatchContext);
  // dispatch is stable (from useReducer) — this component never re-renders
  // due to todo list changes
  return (
    <button onClick={() => dispatch({ type: 'ADD', text: 'New' })}>
      Add
    </button>
  );
}

// Components that read state re-render when todos change:
function TodoList() {
  const todos = useContext(TodoStateContext);
  return todos.map(t => <TodoItem key={t.id} todo={t} />);
}

Strategy 3: Split by Update Frequency

// Separate rarely-changing from frequently-changing values
const UserProfileContext = createContext(null);  // changes on login/logout
const UserPresenceContext = createContext(null);  // changes every few seconds

function UserProvider({ children }) {
  const [profile, setProfile] = useState(null);
  const [presence, setPresence] = useState('offline');

  useEffect(() => {
    const interval = setInterval(() => {
      fetchPresence().then(setPresence);
    }, 5000);
    return () => clearInterval(interval);
  }, []);

  return (
    <UserProfileContext.Provider value={profile}>
      <UserPresenceContext.Provider value={presence}>
        {children}
      </UserPresenceContext.Provider>
    </UserProfileContext.Provider>
  );
}

// Profile consumers don't re-render when presence changes

Provider Composition Pattern

// Problem: provider nesting hell
function App() {
  return (
    <ThemeProvider>
      <AuthProvider>
        <LocaleProvider>
          <NotificationProvider>
            <RouterProvider>
              <Layout />
            </RouterProvider>
          </NotificationProvider>
        </LocaleProvider>
      </AuthProvider>
    </ThemeProvider>
  );
}

// Solution: compose providers
function composeProviders(...providers) {
  return function ComposedProvider({ children }) {
    return providers.reduceRight(
      (child, Provider) => <Provider>{child}</Provider>,
      children
    );
  };
}

const AppProviders = composeProviders(
  ThemeProvider,
  AuthProvider,
  LocaleProvider,
  NotificationProvider,
  RouterProvider
);

function App() {
  return (
    <AppProviders>
      <Layout />
    </AppProviders>
  );
}

Preventing Unnecessary Re-Renders with Context

Technique 1: Memoize Context Value

function AuthProvider({ children }) {
  const [user, setUser] = useState(null);

  // Memoize to prevent re-renders when parent re-renders
  const value = useMemo(() => ({
    user,
    login: (credentials) => { /* ... */ setUser(userData); },
    logout: () => setUser(null),
  }), [user]);

  return <AuthContext.Provider value={value}>{children}</AuthContext.Provider>;
}

Technique 2: Memoize Consumer Components

// Even with context, React.memo can help for components that
// receive BOTH context and props:
const UserAvatar = React.memo(function UserAvatar({ size }) {
  const { user } = useContext(AuthContext);
  return <img src={user?.avatar} width={size} />;
});
// Re-renders when AuthContext changes (unavoidable)
// Does NOT re-render when parent re-renders with same size prop

Technique 3: Extract Context Consumer

// Push context consumption into a small wrapper component
// to isolate re-renders:

// BEFORE: entire ExpensiveComponent re-renders on theme change
function ExpensiveComponent() {
  const theme = useContext(ThemeContext);
  return (
    <div style={{ background: theme.bg }}>
      {/* 1000 lines of expensive rendering */}
    </div>
  );
}

// AFTER: only the wrapper re-renders
function ExpensiveComponent({ style }) {
  return (
    <div style={style}>
      {/* 1000 lines of expensive rendering — doesn't re-render */}
    </div>
  );
}
const ThemedExpensiveComponent = React.memo(ExpensiveComponent);

function ExpensiveComponentWrapper() {
  const theme = useContext(ThemeContext);
  return <ThemedExpensiveComponent style={{ background: theme.bg }} />;
}

Why React Doesn't Have Context Selectors

// Desired API (doesn't exist natively):
const theme = useContext(AppContext, state => state.theme);
// Only re-render when theme changes, not when other parts change

// Why it doesn't exist:
// 1. Context propagation is tied to the fiber tree traversal
// 2. React marks ALL consumers when context changes — no per-consumer filtering
// 3. Adding selectors would require tracking which selector each consumer uses
//    and comparing selector results, adding significant complexity

// Workarounds:

// 1. use-context-selector library (by dai-shi)
import { createContext, useContextSelector } from 'use-context-selector';
const AppContext = createContext(null);
const theme = useContextSelector(AppContext, v => v.theme);

// 2. Split into multiple contexts (recommended by React team)
const ThemeContext = createContext('light');
const UserContext = createContext(null);

// 3. useSyncExternalStore (for store-like patterns)
function useTheme() {
  return useSyncExternalStore(
    themeStore.subscribe,
    () => themeStore.getTheme()
  );
}

useSyncExternalStore as Context Alternative

// For frequently-changing state, useSyncExternalStore with
// a simple store can outperform context:

function createStore(initialState) {
  let state = initialState;
  const listeners = new Set();

  return {
    getState: () => state,
    setState: (updater) => {
      state = typeof updater === 'function' ? updater(state) : updater;
      listeners.forEach(l => l());
    },
    subscribe: (listener) => {
      listeners.add(listener);
      return () => listeners.delete(listener);
    },
  };
}

const appStore = createStore({ theme: 'light', count: 0 });

// Selector-based subscription — only re-renders when selected value changes
function useStore(selector) {
  return useSyncExternalStore(
    appStore.subscribe,
    () => selector(appStore.getState())
  );
}

function ThemeDisplay() {
  const theme = useStore(s => s.theme);
  // Only re-renders when theme changes, not when count changes
  return <div>Theme: {theme}</div>;
}

function Counter() {
  const count = useStore(s => s.count);
  // Only re-renders when count changes, not when theme changes
  return <div>Count: {count}</div>;
}

When Context Is the Right Tool

Locale / i18nLarge lists or tables
Feature flagsAnimation state
Dependency injection (services)Anything needing selectors
Router stateHigh-frequency counters/timers
Architect Insight — Context at Scale Decision

For L6 interviews: Context is a dependency injection mechanism, not a state manager. It excels for low-frequency, widely-consumed values (theme, auth, locale). For high-frequency updates or selective subscriptions, use useSyncExternalStore with a simple store, or adopt Zustand/Jotai. The React team explicitly recommends external stores for complex state management — context is the escape hatch for prop drilling, not the solution for global state.
