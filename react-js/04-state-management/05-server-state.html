Server State vs Client State

CharacteristicClient StateServer State
OwnershipFully controlled by the appOwned by the server, cached locally
Source of truthIn-memory (React state)Remote database/API
FreshnessAlways currentCan be stale (out of sync)
PersistenceLost on refreshPersists on server
SharedSingle userMultiple users can modify
ExamplesUI state, form input, modalsUser profiles, posts, products
Architect Insight

The key realization: server state is fundamentally different from client state. It's a cache of remote data, not state you own. It needs cache invalidation, background refetching, deduplication, and staleness management — problems that useState/Redux don't solve. This is why TanStack Query and SWR exist.

TanStack Query Core Concepts

import {
  QueryClient,
  QueryClientProvider,
  useQuery,
  useMutation,
  useQueryClient,
} from '@tanstack/react-query';

const queryClient = new QueryClient({
  defaultOptions: {
    queries: {
      staleTime: 5 * 60 * 1000,    // 5 minutes
      gcTime: 10 * 60 * 1000,       // 10 minutes (garbage collection)
      retry: 3,
      refetchOnWindowFocus: true,
      refetchOnReconnect: true,
    },
  },
});

function App() {
  return (
    <QueryClientProvider client={queryClient}>
      <MyApp />
    </QueryClientProvider>
  );
}

Query Keys

// Query keys uniquely identify cached data.
// They're arrays that support hierarchical invalidation.

// Simple key:
useQuery({ queryKey: ['users'], queryFn: fetchUsers });

// Parameterized key:
useQuery({ queryKey: ['users', userId], queryFn: () => fetchUser(userId) });

// Complex key with filters:
useQuery({
  queryKey: ['users', { role: 'admin', page: 2 }],
  queryFn: () => fetchUsers({ role: 'admin', page: 2 }),
});

// Key hierarchy enables targeted invalidation:
queryClient.invalidateQueries({ queryKey: ['users'] });
// Invalidates ALL queries starting with ['users']:
// ['users'], ['users', 1], ['users', { role: 'admin' }], etc.

Stale-While-Revalidate

Stale-While-Revalidate Timeline: Time ──────────────────────────────────────────────────────▶ ┌─ Fresh ─┐┌──── Stale ────────────────────┐┌─ Garbage ─┐ │ ││ ││ Collected │ │ staleTime││ ││ gcTime │ │ (5min) ││ ││ (10min) │ └─────────┘└────────────────────────────────┘└───────────┘ Component mounts: ├── Cache HIT (fresh) → return cached data, no fetch ├── Cache HIT (stale) → return cached data, fetch in background ├── Cache MISS → show loading, fetch data └── Cache EXPIRED (gc) → removed from cache entirely Background refetch triggers: ├── Window focus (refetchOnWindowFocus) ├── Network reconnect (refetchOnReconnect) ├── Interval (refetchInterval) └── Manual invalidation (invalidateQueries) 
function UserProfile({ userId }) {
  const { data, isLoading, isError, error, isFetching, isStale } = useQuery({
    queryKey: ['users', userId],
    queryFn: () => fetchUser(userId),
    staleTime: 5 * 60 * 1000, // fresh for 5 minutes
  });

  // isLoading: true on first fetch (no cached data)
  // isFetching: true during any fetch (including background refetch)
  // isStale: true when data is older than staleTime

  if (isLoading) return <Spinner />;
  if (isError) return <Error message={error.message} />;

  return (
    <div>
      {isFetching && <RefreshIndicator />}
      <h1>{data.name}</h1>
    </div>
  );
}

Cache Invalidation Strategies

const queryClient = useQueryClient();

// Strategy 1: Invalidate after mutation
const mutation = useMutation({
  mutationFn: updateUser,
  onSuccess: () => {
    queryClient.invalidateQueries({ queryKey: ['users'] });
  },
});

// Strategy 2: Optimistic update (update cache before server confirms)
const mutation = useMutation({
  mutationFn: updateTodo,
  onMutate: async (newTodo) => {
    await queryClient.cancelQueries({ queryKey: ['todos'] });
    const previousTodos = queryClient.getQueryData(['todos']);

    queryClient.setQueryData(['todos'], (old) =>
      old.map(t => t.id === newTodo.id ? { ...t, ...newTodo } : t)
    );

    return { previousTodos }; // context for rollback
  },
  onError: (err, newTodo, context) => {
    queryClient.setQueryData(['todos'], context.previousTodos); // rollback
  },
  onSettled: () => {
    queryClient.invalidateQueries({ queryKey: ['todos'] }); // refetch
  },
});

// Strategy 3: Direct cache update (skip refetch)
const mutation = useMutation({
  mutationFn: updateUser,
  onSuccess: (updatedUser) => {
    queryClient.setQueryData(['users', updatedUser.id], updatedUser);
  },
});

// Strategy 4: Prefetching
function UserList({ users }) {
  const queryClient = useQueryClient();

  return users.map(user => (
    <Link
      key={user.id}
      to={`/users/${user.id}`}
      onMouseEnter={() => {
        queryClient.prefetchQuery({
          queryKey: ['users', user.id],
          queryFn: () => fetchUser(user.id),
          staleTime: 60 * 1000,
        });
      }}
    >
      {user.name}
    </Link>
  ));
}

Request Deduplication

// TanStack Query automatically deduplicates concurrent requests
// with the same query key:

function UserAvatar({ userId }) {
  const { data } = useQuery({
    queryKey: ['users', userId],
    queryFn: () => fetchUser(userId),
  });
  return <img src={data?.avatar} />;
}

function UserName({ userId }) {
  const { data } = useQuery({
    queryKey: ['users', userId],
    queryFn: () => fetchUser(userId),
  });
  return <span>{data?.name}</span>;
}

// Both components use the same query key ['users', 1].
// TanStack Query makes ONE network request, not two.
// Both components share the same cached data.

Infinite Queries & Pagination

import { useInfiniteQuery } from '@tanstack/react-query';

function InfinitePostList() {
  const {
    data,
    fetchNextPage,
    hasNextPage,
    isFetchingNextPage,
  } = useInfiniteQuery({
    queryKey: ['posts'],
    queryFn: ({ pageParam }) => fetchPosts({ cursor: pageParam }),
    initialPageParam: 0,
    getNextPageParam: (lastPage) => lastPage.nextCursor ?? undefined,
  });

  return (
    <div>
      {data?.pages.map((page, i) => (
        <React.Fragment key={i}>
          {page.posts.map(post => <PostCard key={post.id} post={post} />)}
        </React.Fragment>
      ))}
      <button
        onClick={() => fetchNextPage()}
        disabled={!hasNextPage || isFetchingNextPage}
      >
        {isFetchingNextPage ? 'Loading...' : hasNextPage ? 'Load More' : 'No more'}
      </button>
    </div>
  );
}

// Cursor-based pagination:
// Page 1: { posts: [...], nextCursor: 'abc123' }
// Page 2: { posts: [...], nextCursor: 'def456' }
// Page 3: { posts: [...], nextCursor: null } → no more pages

Mutation Lifecycle

Mutation Lifecycle: mutate(variables) │ ├── onMutate(variables) ← optimistic update │ └── return context │ ├── mutationFn(variables) ← actual API call │ │ │ ├── SUCCESS: │ │ ├── onSuccess(data, variables, context) │ │ └── onSettled(data, null, variables, context) │ │ │ └── ERROR: │ ├── onError(error, variables, context) ← rollback │ └── onSettled(null, error, variables, context) │ └── State updates: isPending → true during mutationFn isSuccess → true after success isError → true after error

Architect Insight — Server State Strategy

For L6 interviews: (1) Never store server data in Redux/Zustand — use TanStack Query or SWR. (2) Server state is a cache problem, not a state management problem. (3) The stale-while-revalidate pattern provides the best UX: show cached data immediately, update in background. (4) Optimistic updates are essential for perceived performance in mutation-heavy UIs. (5) Query key design is the most important architectural decision — it determines invalidation granularity.
