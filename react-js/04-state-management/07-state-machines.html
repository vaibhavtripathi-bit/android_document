Finite State Machines vs Statecharts

A Finite State Machine (FSM) has a finite number of states, transitions between them triggered by events, and can only be in one state at a time. Statecharts extend FSMs with hierarchy (nested states), parallelism (concurrent states), history, and actions.

Simple FSM (Traffic Light): ┌─────────┐ TIMER ┌──────────┐ TIMER ┌─────────┐ │ GREEN │──────────▶│ YELLOW │──────────▶│ RED │ └─────────┘ └──────────┘ └─────────┘ ▲ │ └────────────────── TIMER ───────────────────┘ States: { green, yellow, red } Events: { TIMER } Transitions: green→yellow, yellow→red, red→green 
import { createMachine, assign } from 'xstate';

// Simple FSM:
const trafficLightMachine = createMachine({
  id: 'trafficLight',
  initial: 'green',
  states: {
    green:  { on: { TIMER: 'yellow' } },
    yellow: { on: { TIMER: 'red' } },
    red:    { on: { TIMER: 'green' } },
  },
});

// Statechart with hierarchy, guards, and actions:
const authMachine = createMachine({
  id: 'auth',
  initial: 'idle',
  context: {
    user: null,
    error: null,
    retries: 0,
  },
  states: {
    idle: {
      on: { LOGIN: 'authenticating' },
    },
    authenticating: {
      invoke: {
        src: 'loginService',
        onDone: {
          target: 'authenticated',
          actions: assign({ user: (_, event) => event.data }),
        },
        onError: [
          {
            target: 'authenticating',
            guard: 'canRetry',
            actions: assign({ retries: (ctx) => ctx.retries + 1 }),
          },
          {
            target: 'error',
            actions: assign({ error: (_, event) => event.data.message }),
          },
        ],
      },
    },
    authenticated: {
      on: { LOGOUT: 'idle' },
      entry: assign({ error: null, retries: 0 }),
    },
    error: {
      on: {
        RETRY: 'authenticating',
        RESET: 'idle',
      },
    },
  },
}, {
  guards: {
    canRetry: (ctx) => ctx.retries < 3,
  },
  services: {
    loginService: (ctx, event) =>
      fetch('/api/login', {
        method: 'POST',
        body: JSON.stringify(event.credentials),
      }).then(r => r.json()),
  },
});

XState in React: useMachine

import { useMachine } from '@xstate/react';

function AuthForm() {
  const [state, send] = useMachine(authMachine);

  switch (true) {
    case state.matches('idle'):
      return (
        <button onClick={() => send({
          type: 'LOGIN',
          credentials: { username, password }
        })}>
          Log In
        </button>
      );

    case state.matches('authenticating'):
      return <Spinner />;

    case state.matches('authenticated'):
      return (
        <div>
          <p>Welcome, {state.context.user.name}</p>
          <button onClick={() => send({ type: 'LOGOUT' })}>Log Out</button>
        </div>
      );

    case state.matches('error'):
      return (
        <div>
          <p>Error: {state.context.error}</p>
          <button onClick={() => send({ type: 'RETRY' })}>Retry</button>
          <button onClick={() => send({ type: 'RESET' })}>Reset</button>
        </div>
      );
  }
}

// useMachine returns:
// state.value — current state name
// state.context — extended state (data)
// state.matches('stateName') — check current state
// state.can({ type: 'EVENT' }) — check if transition is valid
// send — dispatch events to the machine

Guards, Actions, and Services

// Guards: conditions that must be true for a transition
const machine = createMachine({
  states: {
    cart: {
      on: {
        CHECKOUT: [
          { target: 'payment', guard: 'hasItems' },
          { target: 'cart' }, // fallback: stay in cart
        ],
      },
    },
  },
}, {
  guards: {
    hasItems: (ctx) => ctx.items.length > 0,
    isAuthenticated: (ctx) => ctx.user !== null,
    hasValidPayment: (ctx) => ctx.paymentMethod !== null,
  },
});

// Actions: side effects that run on transitions
const machine = createMachine({
  states: {
    idle: {
      on: {
        SUBMIT: {
          target: 'submitting',
          actions: ['clearErrors', 'logSubmission'],
        },
      },
    },
    submitting: {
      entry: 'showLoadingIndicator',   // runs when entering state
      exit: 'hideLoadingIndicator',    // runs when leaving state
    },
    success: {
      entry: assign({ result: (_, event) => event.data }),
    },
  },
}, {
  actions: {
    clearErrors: assign({ errors: null }),
    logSubmission: (ctx, event) => analytics.track('form_submit'),
    showLoadingIndicator: () => NProgress.start(),
    hideLoadingIndicator: () => NProgress.done(),
  },
});

// Services (invoked actors): async operations tied to a state
const machine = createMachine({
  states: {
    fetching: {
      invoke: {
        id: 'fetchData',
        src: 'dataService',
        onDone: { target: 'success', actions: assign({ data: (_, e) => e.data }) },
        onError: { target: 'failure', actions: assign({ error: (_, e) => e.data }) },
      },
    },
  },
}, {
  services: {
    dataService: (ctx) => fetch(`/api/data/${ctx.id}`).then(r => r.json()),
  },
});

Modeling Complex UI Flows

Multi-Step Wizard

const wizardMachine = createMachine({
  id: 'wizard',
  initial: 'personalInfo',
  context: {
    personalInfo: {},
    address: {},
    payment: {},
  },
  states: {
    personalInfo: {
      on: {
        NEXT: {
          target: 'address',
          guard: 'personalInfoValid',
          actions: assign({
            personalInfo: (_, event) => event.data,
          }),
        },
      },
    },
    address: {
      on: {
        NEXT: {
          target: 'payment',
          guard: 'addressValid',
          actions: assign({ address: (_, event) => event.data }),
        },
        BACK: 'personalInfo',
      },
    },
    payment: {
      on: {
        SUBMIT: {
          target: 'submitting',
          guard: 'paymentValid',
          actions: assign({ payment: (_, event) => event.data }),
        },
        BACK: 'address',
      },
    },
    submitting: {
      invoke: {
        src: 'submitOrder',
        onDone: 'success',
        onError: 'payment', // go back to payment on error
      },
    },
    success: {
      type: 'final',
    },
  },
});

function Wizard() {
  const [state, send] = useMachine(wizardMachine);

  return (
    <div>
      <StepIndicator current={state.value} />
      {state.matches('personalInfo') && (
        <PersonalInfoStep
          data={state.context.personalInfo}
          onNext={(data) => send({ type: 'NEXT', data })}
        />
      )}
      {state.matches('address') && (
        <AddressStep
          data={state.context.address}
          onNext={(data) => send({ type: 'NEXT', data })}
          onBack={() => send({ type: 'BACK' })}
        />
      )}
      {state.matches('payment') && (
        <PaymentStep
          data={state.context.payment}
          onSubmit={(data) => send({ type: 'SUBMIT', data })}
          onBack={() => send({ type: 'BACK' })}
        />
      )}
      {state.matches('submitting') && <Spinner />}
      {state.matches('success') && <SuccessMessage />}
    </div>
  );
}

Async Operation with Retry

const fetchMachine = createMachine({
  id: 'fetch',
  initial: 'idle',
  context: { data: null, error: null, retries: 0 },
  states: {
    idle: {
      on: { FETCH: 'loading' },
    },
    loading: {
      invoke: {
        src: 'fetchData',
        onDone: {
          target: 'success',
          actions: assign({ data: (_, e) => e.data, error: null }),
        },
        onError: [
          {
            target: 'retrying',
            guard: (ctx) => ctx.retries < 3,
          },
          {
            target: 'failure',
            actions: assign({ error: (_, e) => e.data }),
          },
        ],
      },
    },
    retrying: {
      after: {
        // Exponential backoff: 1s, 2s, 4s
        RETRY_DELAY: {
          target: 'loading',
          actions: assign({ retries: (ctx) => ctx.retries + 1 }),
        },
      },
    },
    success: {
      on: { REFRESH: 'loading' },
    },
    failure: {
      on: {
        RETRY: {
          target: 'loading',
          actions: assign({ retries: 0 }),
        },
      },
    },
  },
}, {
  delays: {
    RETRY_DELAY: (ctx) => Math.pow(2, ctx.retries) * 1000,
  },
});

Testing State Machines

import { createMachine, interpret } from 'xstate';

// State machines are highly testable because they're pure:
// Given a state and an event, the next state is deterministic.

describe('authMachine', () => {
  it('transitions from idle to authenticating on LOGIN', () => {
    const nextState = authMachine.transition('idle', { type: 'LOGIN' });
    expect(nextState.value).toBe('authenticating');
  });

  it('transitions to error after max retries', () => {
    const stateWithRetries = authMachine.transition(
      { value: 'authenticating', context: { retries: 3 } },
      { type: 'error', data: { message: 'Failed' } }
    );
    expect(stateWithRetries.value).toBe('error');
  });

  it('retries when under retry limit', () => {
    const stateWithRetries = authMachine.transition(
      { value: 'authenticating', context: { retries: 1 } },
      { type: 'error', data: { message: 'Failed' } }
    );
    expect(stateWithRetries.value).toBe('authenticating');
    expect(stateWithRetries.context.retries).toBe(2);
  });

  it('resets context on entering authenticated', () => {
    const service = interpret(authMachine).start();
    service.send({ type: 'LOGIN' });
    // Simulate successful login
    // ...
    expect(service.state.context.error).toBeNull();
    expect(service.state.context.retries).toBe(0);
  });
});

// Model-based testing: XState can generate test paths
import { createModel } from '@xstate/test';

const testModel = createModel(authMachine).withEvents({
  LOGIN: () => { /* simulate login click */ },
  LOGOUT: () => { /* simulate logout click */ },
});

const testPlans = testModel.getShortestPathPlans();
testPlans.forEach(plan => {
  describe(plan.description, () => {
    plan.paths.forEach(path => {
      it(path.description, async () => {
        await path.test(/* test context */);
      });
    });
  });
});

When State Machines Are Essential vs Overkill

EssentialOverkill
Multi-step wizards with complex navigationSimple toggle (open/closed)
Authentication flows with retriesBasic form with 3 fields
Complex async operations (upload with progress, retry, cancel)Simple data fetching (use TanStack Query)
UI with many states and impossible state preventionCounter or simple CRUD
Flows that need visualization for stakeholdersState that maps 1:1 to a boolean
Embedded systems / protocol implementationsPrototype or MVP

Comparison with useReducer

// useReducer — ad-hoc state transitions
function reducer(state, action) {
  switch (action.type) {
    case 'FETCH':
      return { ...state, status: 'loading' };
    case 'SUCCESS':
      return { ...state, status: 'success', data: action.data };
    case 'ERROR':
      return { ...state, status: 'error', error: action.error };
    default:
      return state;
  }
}
// Problems:
// 1. No guard against invalid transitions (can dispatch SUCCESS from idle)
// 2. No visualization
// 3. Impossible states possible ({ status: 'idle', data: staleData })
// 4. Side effects mixed with state logic

// XState — explicit state machine
const machine = createMachine({
  initial: 'idle',
  states: {
    idle: { on: { FETCH: 'loading' } },     // can only FETCH from idle
    loading: {
      invoke: { src: 'fetchData', onDone: 'success', onError: 'error' },
    },
    success: { on: { REFRESH: 'loading' } }, // explicit transitions only
    error: { on: { RETRY: 'loading' } },
  },
});
// Benefits:
// 1. Invalid transitions are impossible (can't SUCCESS from idle)
// 2. Visualizable (XState Viz)
// 3. Impossible states are unrepresentable
// 4. Side effects are declared, not imperative
Architect Insight — State Machine Adoption

For L6 interviews: State machines formalize what useReducer does informally. The value is in making impossible states impossible and providing a visual model of the system. Introduce them for complex flows where bugs stem from invalid state combinations. Don't use them for simple state — the overhead isn't justified. XState's model-based testing is a powerful argument: the machine definition generates test cases automatically.
