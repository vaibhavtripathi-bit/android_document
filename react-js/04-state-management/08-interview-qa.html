Interview Q&A: State Management

Local State Patterns

Q1: When should you use useReducer instead of useState? L6
Use useReducer when: (1) next state depends on previous state in complex ways, (2) multiple sub-values update together and impossible states should be unrepresentable (discriminated unions), (3) you want to pass dispatch down instead of multiple callbacks (dispatch is stable), (4) state transitions need to be testable independently of components. useState suffices for simple, independent values.
Q2: What is the state colocation principle and why does it matter? L6
State colocation means keeping state as close as possible to where it's consumed. Benefits: (1) minimizes re-render scope (only the consuming component re-renders), (2) improves code locality (state and its consumers are in the same file), (3) makes components more self-contained and reusable, (4) reduces the need for context/global stores. Start local, lift only when siblings need the same state.
Q3: What's the difference between lifting state up and pushing state down? L6
Lifting up: moving state to a common ancestor when siblings need to share it (e.g., accordion where only one panel can be open). Pushing down: extracting state into a child component when only part of the parent needs it, preventing unnecessary re-renders of the parent's other children. Both are colocation strategies — lifting up broadens scope when needed, pushing down narrows scope when possible.
Q4: Why is storing derived state in useState an anti-pattern? L6
Storing derived state creates synchronization bugs: the derived value lags by one render cycle (computed in useEffect, which runs after render). It also causes unnecessary re-renders (effect → setState → re-render). Instead, compute derived values during render (inline or with useMemo). The source of truth should be the original state; derived values should be computed, not stored.
Q5: How does component composition solve prop drilling without Context? L6
Instead of passing data through intermediate components, pass pre-built JSX elements as props (children or named slots). The parent creates <UserInfo user={user} /> and passes it to <Layout sidebar={...} />. Layout doesn't know about user — it just renders the slot. This inverts the dependency: the top-level component composes the tree, intermediates are agnostic to the data.
Q6: How does React 18's automatic batching change state management patterns? L6
Pre-React 18, only event handler updates were batched; updates in promises/timeouts caused multiple re-renders. React 18 batches all updates automatically, so multiple setState calls always produce one re-render. This reduces the advantage of useReducer for atomic multi-value updates (useState batching achieves the same). However, useReducer still wins for type-safe impossible state prevention and testable transition logic.

Context at Scale

Q7: Why is Context not a good general-purpose state management solution? L6
Context has no selector mechanism. When the context value changes, every component calling useContext(MyContext) re-renders, regardless of which part of the value they use. This makes it unsuitable for frequently-changing state with many consumers. Context is designed for dependency injection (theme, auth, locale) — values that change infrequently and are consumed widely.
Q8: Describe three strategies for splitting context to improve performance. L6
(1) Split by concern: separate ThemeContext, AuthContext, LocaleContext instead of one AppContext. (2) Split state from dispatch: separate TodoStateContext (changes often) from TodoDispatchContext (stable dispatch function). Components that only dispatch don't re-render on state changes. (3) Split by update frequency: separate UserProfileContext (changes on login) from UserPresenceContext (changes every few seconds).
Q9: Why doesn't React have built-in context selectors? L6
Context propagation is tied to the fiber tree traversal. When a context value changes, React marks ALL consumer fibers for re-render during the tree walk. Adding selectors would require: (1) tracking which selector each consumer uses, (2) running selectors and comparing results before marking for re-render, (3) significant complexity in the reconciler. The React team recommends splitting contexts or using useSyncExternalStore instead.
Q10: How can useSyncExternalStore serve as a context alternative? L6
Create a simple store outside React with getState, setState, and subscribe. Use useSyncExternalStore(store.subscribe, () => selector(store.getState())) with a selector. Only components whose selector returns a different value re-render. This provides selector-based subscriptions that context lacks, with concurrent-mode safety (tearing prevention).
Q11: How do you prevent a context provider from causing unnecessary re-renders? L6
(1) Memoize the context value with useMemo so it only changes when its dependencies change. (2) Separate the provider's own state from unrelated parent state. (3) Use the children pattern — <Provider>{children}</Provider> where children is a stable reference (not inline JSX). (4) Split state from dispatch into separate contexts. (5) Extract context consumers into small wrapper components to isolate re-renders.

Redux Internals

Q12: Explain Redux's middleware pipeline. How does a middleware intercept actions? L6
Middleware has the signature store => next => action => result. applyMiddleware composes middlewares so each wraps the next's dispatch. When an action is dispatched, it flows through each middleware in order. Each middleware can: (1) pass the action to next (continue the chain), (2) modify the action, (3) dispatch additional actions via store.dispatch, (4) swallow the action (don't call next), or (5) perform async work before/after calling next.
Q13: How does Immer enable "mutable" syntax in Redux Toolkit reducers? L6
Immer wraps the state in a JavaScript Proxy. When you write state.user.name = 'Alice', the Proxy's set trap intercepts the mutation, marks the path as modified, and performs copy-on-write: it clones objects along the modified path up to the root. The original state is never mutated. The result is a new immutable state tree where only the changed paths have new references.
Q14: How does Redux DevTools time-travel debugging work? L6
Redux is pure: given the same initial state and action sequence, you always get the same result. DevTools stores the initial state and every dispatched action. To "time-travel" to state N, it replays actions 0 through N-1 from the initial state. It can also skip/reorder actions and recompute state. This works because reducers are pure functions with no side effects.
Q15: How does Reselect's createSelector achieve memoization? What's its cache size? L6
createSelector takes input selectors and a result function. It memoizes with a default cache size of 1 (last result). On each call, it runs input selectors and compares results with the previous call using reference equality. If all inputs are the same, it returns the cached result without running the result function. Cache size of 1 means it thrashes when called with alternating arguments (e.g., in a list). Solutions: increase maxSize, use per-component instances, or use createEntityAdapter.
Q16: When should you choose Redux over useReducer + Context? L6
Choose Redux when: (1) you need selector-based subscriptions (useSelector only re-renders when selected value changes), (2) you need middleware for async logic (thunk, saga), (3) you need DevTools for debugging complex state flows, (4) you need RTK Query for server state, (5) the app has many state slices with many consumers. Choose useReducer+Context for small/medium apps with simple state and few consumers.
Q17: How does RTK Query compare to TanStack Query? L6
RTK Query is tightly integrated with Redux (lives in the Redux store, uses Redux DevTools). It uses tag-based cache invalidation. TanStack Query is framework-agnostic with a dedicated cache, uses key-based invalidation, and has richer features (infinite queries, prefetching, optimistic updates with rollback). Choose RTK Query if already using Redux; choose TanStack Query for standalone server state management or non-Redux apps.

Zustand, Jotai & Signals

Q18: How does Zustand achieve selector-based re-renders without a Provider? L6
Zustand creates a store outside the React tree with a subscription mechanism. The hook uses useSyncExternalStore internally, passing a selector. When state changes, each subscriber's selector runs. If the selector result is unchanged (Object.is comparison), the component skips re-rendering. No Provider is needed because the store is a module-level singleton, not a React context.
Q19: Explain Jotai's atomic state model. How does it differ from Zustand's top-down approach? L6
Jotai uses a bottom-up model: state is composed from individual atoms (like useState but shareable). Each atom is independent, and derived atoms compute from other atoms. Jotai tracks dependencies automatically — when atom A changes, only derived atoms that read A recompute, and only components subscribed to those atoms re-render. Zustand is top-down: one store, selectors pick slices. Jotai avoids the "mega-store" problem by keeping state naturally decomposed.
Q20: How do Signals bypass React's re-rendering model? What are the trade-offs? L6
Signals (e.g., @preact/signals-react) patch React's rendering to detect signal usage in JSX. When a signal changes, it updates the DOM text node directly without re-rendering the component. Trade-offs: (1) bypasses React's reconciliation, potentially conflicting with React's assumptions, (2) the React team has expressed concerns about compatibility, (3) limited ecosystem support, (4) debugging is harder (React DevTools don't see signal updates). Best suited for frameworks designed for signals (Solid, Preact).
Q21: Compare the re-render characteristics of Zustand, Jotai, Signals, and Context for 100 components with 1 state change. L6
Context: all 100 consumers re-render (no selectors). Zustand: 100 selectors run, only components whose selector returns a new value re-render (typically 1-5). Jotai: dependency graph traversed, only atoms affected by the change recompute, only subscribed components re-render (typically 1-5). Signals: 0 component re-renders — DOM nodes update directly. Context is worst for performance; Signals is best but with compatibility trade-offs.
Q22: How would you choose between Zustand, Jotai, and Redux for a new project? L6
Decision factors: (1) Team familiarity: Redux if team knows it, Zustand for Redux-like simplicity. (2) State shape: single store → Zustand/Redux; many independent atoms → Jotai. (3) Bundle size: Zustand (~1.5KB) and Jotai (~3KB) are much smaller than Redux (~11KB). (4) DevTools need: Redux has the best DevTools; Zustand/Jotai have middleware for Redux DevTools. (5) Middleware: Redux has the richest ecosystem; Zustand has persist/devtools/immer. (6) Simplicity: start with useState, adopt Zustand/Jotai when needed.

Server State

Q23: Why should server data not be stored in Redux or Zustand? L6
Server data is a cache of remote state, not state you own. It needs: stale-while-revalidate semantics, background refetching, cache invalidation, request deduplication, optimistic updates with rollback, and garbage collection. Redux/Zustand don't provide these — you'd have to build them manually. TanStack Query/SWR are purpose-built for these problems. Mixing server and client state in one store also makes cache invalidation harder.
Q24: Explain the stale-while-revalidate pattern in TanStack Query. L6
Data has a staleTime (fresh period) and gcTime (cache lifetime). When a component mounts: (1) if data is fresh, return it without fetching; (2) if data is stale, return cached data immediately AND fetch in background — update when the fetch completes; (3) if no cache, show loading and fetch. This provides instant UI with eventual consistency. Background refetches also trigger on window focus, network reconnect, and manual invalidation.
Q25: How do you implement optimistic updates with rollback in TanStack Query? L6
In useMutation: (1) onMutate: cancel in-flight queries, snapshot current cache, optimistically update cache, return snapshot as context. (2) onError: use the context to rollback cache to the snapshot. (3) onSettled: invalidate queries to refetch fresh data regardless of success/failure. This provides instant UI feedback while ensuring eventual consistency with the server.
Q26: How does TanStack Query deduplicate concurrent requests? L6
TanStack Query uses query keys as cache identifiers. When multiple components mount simultaneously with the same query key, only one network request is made. The query is tracked in a central QueryCache; subsequent calls with the same key find the existing query and subscribe to its result. When the request completes, all subscribers receive the data. This is automatic and requires no configuration.
Q27: Describe query key design best practices for a large application. L6
Use hierarchical arrays: ['entity', id, 'sub-resource', filters]. Examples: ['users'], ['users', userId], ['users', userId, 'posts'], ['users', userId, 'posts', { status: 'published' }]. This enables granular invalidation: invalidateQueries(['users']) invalidates all user-related queries. Create a query key factory: const userKeys = { all: ['users'], detail: (id) => ['users', id], posts: (id) => ['users', id, 'posts'] }.
Q28: Compare TanStack Query and SWR. When would you choose each? L6
TanStack Query: richer features (mutations, optimistic updates, infinite queries, DevTools, offline support), larger bundle (~13KB). Choose for complex apps with many mutations and cache invalidation needs. SWR: simpler API, smaller bundle (~4KB), less opinionated. Choose for simpler apps focused on data fetching with minimal mutations. Both implement stale-while-revalidate; TanStack Query is more feature-complete.

Form State

Q29: Why does React Hook Form outperform controlled forms? L6
RHF uses uncontrolled inputs with refs internally. Field values are stored in a ref (not useState), so typing doesn't trigger re-renders. Only specific subscriptions (errors, isDirty, isValid) cause re-renders, and only in the subscribing component. A controlled form with 20 fields re-renders all 20 inputs on every keystroke; RHF re-renders 0-1 components per keystroke.
Q30: How does the resolver pattern work in React Hook Form with Zod? L6
The resolver option accepts a function that validates the entire form data against a schema. zodResolver(schema) returns a function that: (1) receives form values, (2) calls schema.safeParse(values), (3) returns { values, errors } in RHF's expected format. This decouples validation from the form — the schema is the single source of truth for both TypeScript types (z.infer) and runtime validation.
Q31: Explain React 19's server actions for form handling. L6
Server actions are async functions marked with 'use server' that run on the server. They can be passed to a form's action prop: <form action={serverAction}>. When submitted, React serializes the FormData, sends it to the server, executes the action, and handles the response. This works with progressive enhancement (no JS needed for basic submission) and integrates with useFormStatus (pending state) and useFormState (action return value).
Q32: When should form state be local vs global? L6
Form state should almost always be local. Forms are inherently component-scoped — the input values, validation errors, dirty/touched state, and submission state are only relevant to the form component. Putting form state in Redux/Zustand adds unnecessary complexity, couples the form to the global store, and makes the form harder to reuse. The only exception: multi-page forms where state must persist across route changes (use URL params or session storage instead).

State Machines

Q33: What's the difference between a finite state machine and a statechart? L6
An FSM has flat states with transitions between them. A statechart extends FSMs with: (1) hierarchy — nested states (a "loading" state can have sub-states "fetching" and "parsing"), (2) parallelism — concurrent regions (a form can be in "editing" and "validating" simultaneously), (3) history — remembering which sub-state was active, (4) guards — conditional transitions, (5) actions — side effects on entry/exit/transition. XState implements statecharts.
Q34: How do state machines prevent impossible states? L6
State machines explicitly define which states exist and which transitions are valid. You can't be in a "success" state without going through "loading" first. You can't dispatch a "SUBMIT" event from the "success" state unless explicitly allowed. With useReducer, any action can be dispatched from any state — the reducer must manually guard against invalid combinations. State machines make invalid transitions structurally impossible, not just logically prevented.
Q35: How does XState's model-based testing work? L6
XState can generate all possible paths through a state machine. createModel(machine).getShortestPathPlans() produces test plans that cover every state and transition. Each plan contains a sequence of events to send and assertions to check. The machine definition IS the test specification — you define the behavior once and get comprehensive test coverage automatically. This catches edge cases that manual test writing often misses.
Q36: When are state machines overkill? When are they essential? L6
Overkill: simple toggles, basic CRUD, counters, simple data fetching (use TanStack Query), prototypes. Essential: multi-step wizards with complex navigation, authentication flows with retries and timeouts, file upload with progress/pause/resume/cancel, complex async workflows, any flow where bugs stem from invalid state combinations, systems that need visual documentation of behavior.
Q37: Compare XState with useReducer for modeling a multi-step form wizard. L6
useReducer: you must manually guard transitions (can dispatch NEXT from any step), handle async operations in effects, and test each transition individually. XState: transitions are explicitly defined per state (NEXT only works in specific steps), async operations are declared as invoked services, guards validate before transitioning, and the machine can be visualized and tested automatically. XState provides stronger guarantees but more upfront design cost.

Architecture & Cross-Cutting

Q38: Design the state management architecture for a large e-commerce application. L6
Server state (products, orders, user profile): TanStack Query with hierarchical query keys and optimistic mutations. Client state (cart, UI preferences): Zustand with persist middleware (cart survives refresh). Auth state: React Context (changes infrequently, consumed everywhere). Form state: React Hook Form + Zod (local to each form). Complex flows (checkout wizard): XState for the multi-step flow. Theme/locale: React Context. No Redux — each concern uses the right tool.
Q39: How do you decide what goes in global state vs local state vs URL state? L6
URL state: anything that should be shareable/bookmarkable (search query, filters, pagination, selected tab). Global state: truly app-wide concerns (auth, theme, feature flags, shopping cart). Local state: everything else (form inputs, dropdown open/closed, hover state, component-specific UI). Default to local; promote to URL for shareability; promote to global only when multiple unrelated components need it.
Q40: How would you handle real-time data (WebSocket) in a React app's state? L6
Use useSyncExternalStore to subscribe to a WebSocket-backed store, ensuring concurrent-mode safety. Alternatively, TanStack Query with queryClient.setQueryData to update the cache when WebSocket messages arrive, combined with refetchOnWindowFocus: false (server pushes updates). For complex real-time state (collaborative editing), consider a CRDT library (Yjs, Automerge) with a Zustand store as the local view.
Q41: What state management patterns change with React Server Components? L6
Server Components can't use hooks or state — they fetch data with async/await and pass it as props. Client state is pushed to Client Components (marked with 'use client'). Server actions replace API routes for mutations. The boundary between server and client state becomes explicit. Data fetching moves from useEffect/TanStack Query to server-side async functions. Client Components still use hooks for interactivity.
Q42: How do you handle state that needs to persist across page navigations? L6
Options by persistence scope: (1) URL search params: for shareable state (filters, pagination) — use useSearchParams. (2) Session storage: for tab-scoped state (form drafts) — Zustand with persist middleware. (3) Local storage: for cross-session state (theme, cart) — Zustand persist or dedicated hook. (4) Server: for user-specific state (preferences) — API + TanStack Query. (5) In-memory: for module-scoped singletons (Zustand store) — survives navigation but not refresh.
Q43: How do you prevent state management from becoming the bottleneck in a large team? L6
(1) Colocate by default: teams own their components and their state. (2) Separate concerns: server state (TanStack Query), client state (Zustand/local), form state (RHF) — no monolithic store. (3) Feature-based slices: each feature team owns their Zustand slice or query keys. (4) Shared contracts: define query key factories and Zod schemas as shared packages. (5) Avoid global state: most state is local; global state should be minimal and well-documented.
Q44: What are the testing strategies for each state management approach? L6
useState/useReducer: test via component tests (render, interact, assert). Context: wrap test components in providers. Redux: unit test reducers/selectors (pure functions), integration test with renderWithProviders. Zustand: reset store between tests, test selectors as pure functions. TanStack Query: use QueryClientProvider in tests, mock fetch, test loading/error/success states. XState: unit test transitions (machine.transition(state, event)), model-based testing for coverage.
Q45: How do you migrate from Redux to a modern alternative (Zustand/Jotai) incrementally? L6
(1) Identify state categories: server state → migrate to TanStack Query first (biggest win). (2) Extract server state from Redux slices into queries/mutations. (3) For remaining client state, create Zustand stores per feature domain. (4) Replace useSelector calls with Zustand selectors one component at a time. (5) Keep Redux running alongside during migration — both can coexist. (6) Remove Redux slices as they're fully migrated. (7) Remove Redux dependency when all slices are gone.
Q46: Explain the concept of "state colocation spectrum" and how it applies to architecture decisions. L6
The spectrum ranges from most local to most global: (1) Component state (useState) → (2) Lifted state (parent props) → (3) Composed state (children pattern) → (4) Context (subtree-wide) → (5) External store (app-wide) → (6) URL state (shareable) → (7) Server state (persistent). Each level adds scope and complexity. The architectural principle: start at the most local level that satisfies the requirement. Premature globalization is the most common state management mistake.
Q47: How would you implement undo/redo functionality in a React application? L6
Use a state history pattern: maintain a stack of past states, current state, and future states. On action: push current to past, apply action. On undo: push current to future, pop past. On redo: push current to past, pop future. Implement as Zustand middleware: temporal middleware from zundo library. Or with useReducer: wrap the reducer to track history. For large state, store diffs (patches) instead of full snapshots using Immer's produceWithPatches.
Q48: What are the performance implications of different state management approaches at scale (1000+ components)? L6
Context: O(n) re-renders per change — unacceptable at scale. Redux: O(n) selector evaluations per change, but only O(k) re-renders (k = components with changed selectors). Zustand: same as Redux (useSyncExternalStore + selectors). Jotai: O(d) dependency graph traversal (d = depth), O(k) re-renders. Signals: O(k) DOM mutations, 0 re-renders. For 1000+ components, avoid Context for frequent updates; use selector-based stores or atomic state.
