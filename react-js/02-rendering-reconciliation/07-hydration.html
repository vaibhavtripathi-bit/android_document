1. What Is Hydration?

Hydration is the process of attaching React's event handlers and state management to server-rendered HTML. The server sends static HTML for fast initial display; the client "hydrates" it by making it interactive.

Server-Side Rendering + Hydration: Server: React component tree → renderToString() → HTML string "<div><h1>Hello</h1><button>Click</button></div>" │ ▼ Send HTML to client (fast first paint) Client: 1. Browser renders HTML (visible but NOT interactive) 2. JavaScript bundle loads 3. hydrateRoot() runs: a. React walks the existing DOM b. Matches DOM nodes to fiber tree c. Attaches event listeners d. Sets up state and effects 4. Page is now fully interactive Timeline: ├── HTML arrives ──┤── JS loads ──┤── Hydration ──┤── Interactive ──► FCP (visible) TTI (clickable) 
// Server
import { renderToString } from 'react-dom/server';
const html = renderToString(<App />);
// Send html in HTTP response

// Client
import { hydrateRoot } from 'react-dom/client';
const root = hydrateRoot(
  document.getElementById('root'),
  <App />
);
Hydration vs Rendering

Normal createRoot().render() creates DOM nodes from scratch. hydrateRoot() reuses existing DOM nodes — it walks the server-rendered HTML and attaches React's internal structures (fibers, event listeners, state) to the existing nodes. This avoids the expensive step of destroying and recreating the entire DOM tree.

2. hydrateRoot API

import { hydrateRoot } from 'react-dom/client';

const root = hydrateRoot(
  domNode,          // Container with server-rendered HTML
  reactNode,        // <App /> — must match server output
  {
    // Error callbacks (React 19)
    onRecoverableError(error, errorInfo) {
      // Called for hydration mismatches and other recoverable errors
      console.warn('Hydration error:', error);
    },
    onCaughtError(error, errorInfo) {
      // Error caught by error boundary during hydration
    },
    onUncaughtError(error, errorInfo) {
      // Uncaught error during hydration
    },
    identifierPrefix: 'app-',  // For useId() uniqueness
  }
);

// After hydration, root works like createRoot:
root.render(<UpdatedApp />); // Re-render
root.unmount();               // Unmount

3. Mismatch Detection and Recovery

During hydration, React compares the server-rendered DOM with what the client would render. Mismatches indicate bugs that can cause visual glitches and state issues.

// Common mismatch causes:

// 1. Date/time rendering
function Clock() {
  return <span>{new Date().toLocaleTimeString()}</span>;
  // Server: "10:30:00 AM" (server time)
  // Client: "10:30:05 AM" (client time, 5 seconds later)
  // → MISMATCH
}

// 2. Browser-only APIs
function WindowSize() {
  return <span>{window.innerWidth}</span>;
  // Server: window is undefined → error or fallback
  // Client: "1920"
  // → MISMATCH
}

// 3. Random values
function RandomId() {
  return <div id={Math.random().toString(36)}>...</div>;
  // Different random value on server vs client
  // → MISMATCH (use useId() instead)
}

// 4. Invalid HTML nesting
<p><div>nested</div></p>
// Browser auto-corrects invalid nesting during HTML parsing
// React sees different DOM structure than expected
// → MISMATCH

3.1 Recovery Behavior

// When React detects a mismatch:

// React 18 behavior:
// 1. Log a warning in development
// 2. Attempt to patch the mismatch client-side
// 3. For text content: update the text node
// 4. For element mismatches: delete and recreate the subtree
// 5. Call onRecoverableError callback

// suppressHydrationWarning: silence expected mismatches
<span suppressHydrationWarning>
  {new Date().toLocaleTimeString()}
</span>
// Only suppresses the warning — React still patches the mismatch
// Use sparingly for known-unavoidable mismatches (timestamps, etc.)
Why Mismatches Matter

Hydration mismatches cause: (1) Content flicker — user sees server content, then it changes to client content. (2) State loss — React may destroy and recreate subtrees, losing component state. (3) Performance degradation — recreating DOM is expensive, defeating the purpose of SSR. (4) SEO issues — if server and client render different content, search engines see different content than users. At scale, mismatches should be treated as bugs and tracked in error monitoring.

4. Selective Hydration

// Selective hydration: hydrate sections independently
// Requires: renderToPipeableStream + Suspense + hydrateRoot

// Server
const { pipe } = renderToPipeableStream(
  <App>
    <Header />
    <Suspense fallback={<Spinner />}>
      <HeavyContent />
    </Suspense>
    <Suspense fallback={<Spinner />}>
      <Comments />
    </Suspense>
  </App>
);

// Hydration order (without user interaction):
// 1. Header hydrates first (outside Suspense)
// 2. HeavyContent hydrates when its HTML arrives
// 3. Comments hydrates when its HTML arrives

// Hydration order (with user interaction):
// 1. Header hydrates first
// 2. User clicks on Comments section
// 3. React PRIORITIZES Comments hydration
// 4. Comments hydrates immediately
// 5. Click event is replayed after hydration
// 6. HeavyContent hydrates later
Selective hydration with user interaction: Time → ├── Header HTML ─┤── Content HTML ──┤── Comments HTML ──┤ │ │ │ │ │ Hydrate Header │ │ User clicks │ │ │ │ Comments! │ │ │ │ ↓ │ │ │ │ PRIORITIZE │ │ │ │ Comments │ │ │ │ hydration │ │ │ │ ↓ │ │ │ │ Hydrate Comments │ │ │ │ Replay click │ │ │ Hydrate Content │ │ │ │ (lower priority) │ │

5. Progressive Hydration Strategies

// Strategy 1: Suspense-based (React 18 native)
// Wrap sections in Suspense boundaries
// Each boundary hydrates independently

// Strategy 2: Lazy hydration (third-party)
// Libraries like react-lazy-hydration delay hydration
// until the component is visible (IntersectionObserver)

import { LazyHydrate } from 'react-lazy-hydration';

<LazyHydrate whenVisible>
  <Footer />  {/* Only hydrates when scrolled into view */}
</LazyHydrate>

<LazyHydrate whenIdle>
  <Analytics />  {/* Hydrates during idle time */}
</LazyHydrate>

<LazyHydrate on="click">
  <Dropdown />  {/* Hydrates on first interaction */}
</LazyHydrate>

// Strategy 3: Islands architecture (Astro, Fresh)
// Only hydrate interactive "islands" in a sea of static HTML
// Most of the page is never hydrated (zero JS)

6. Streaming SSR + Selective Hydration Flow

// Complete flow: streaming SSR → progressive HTML → selective hydration

// Server (Node.js)
const { pipe } = renderToPipeableStream(<App />, {
  bootstrapScripts: ['/main.js'],
  onShellReady() {
    res.statusCode = 200;
    res.setHeader('Content-Type', 'text/html');
    pipe(res);
  }
});

// What the server streams:
// 1. Shell HTML (header, layout, navigation)
// 2. Suspense fallback placeholders
// 3. <script> tag to load main.js
// 4. As data resolves: replacement HTML chunks + inline scripts
//    that swap fallbacks with real content

// Client (main.js)
hydrateRoot(document, <App />);
// React starts hydrating as HTML streams in
// Each Suspense boundary hydrates independently

7. Hydration Performance Optimization

Optimization Technique Impact
Reduce JS bundle Code splitting, tree shaking, lazy loading Faster JS load → faster hydration start
Selective hydration Suspense boundaries around sections Interactive sooner (critical sections first)
Lazy hydration Hydrate on visibility/interaction Less JS executed on initial load
Server Components RSC — zero client JS for static parts Dramatically less hydration work
Streaming SSR renderToPipeableStream Faster TTFB, progressive rendering
Avoid mismatches useId(), suppressHydrationWarning No DOM recreation during hydration 
// useId() generates stable IDs across server and client
function FormField({ label }) {
  const id = useId();
  return (
    <>
      <label htmlFor={id}>{label}</label>
      <input id={id} />
    </>
  );
}
// Server and client generate the same ID
// No hydration mismatch!

8. Architect Considerations

Staff-Level Hydration Architecture

At L6, design your SSR/hydration strategy around these principles: (1) Streaming over string rendering — renderToPipeableStream gives faster TTFB and enables selective hydration. (2) Suspense boundaries as hydration units — each boundary can hydrate independently, prioritized by user interaction. (3) Minimize hydration surface — use React Server Components for static content (zero client JS), lazy hydration for below-fold content, and interaction-triggered hydration for dormant widgets. (4) Monitor mismatches — use onRecoverableError to track hydration issues in production. (5) Measure TTI, not just FCP — the gap between visible and interactive is the hydration cost. Target: TTI within 100ms of FCP for critical interactions.
