1. react-dom as Host Renderer

React's architecture separates the reconciler (react-reconciler) from the renderer. react-dom is the host renderer for web browsers — it implements the "host config" interface that tells the reconciler how to create, update, and delete DOM nodes.

React Architecture: ┌─────────────┐ ┌──────────────────┐ ┌───────────────┐ │ react │ │ react-reconciler │ │ react-dom │ │ │ │ │ │ │ │ createElement│────►│ Fiber tree │────►│ createElement │ │ hooks │ │ Diffing │ │ appendChild │ │ Component │ │ Scheduling │ │ removeChild │ │ │ │ Effect mgmt │ │ setProperty │ └─────────────┘ └──────────────────┘ │ Event system │ └───────────────┘ Other renderers use the same reconciler: react-native → native views react-three-fiber → Three.js objects ink → terminal output react-pdf → PDF documents 
// The host config interface (simplified)
// react-dom implements these for the browser DOM:

const hostConfig = {
  createInstance(type, props) {
    return document.createElement(type);
  },
  createTextInstance(text) {
    return document.createTextNode(text);
  },
  appendChild(parent, child) {
    parent.appendChild(child);
  },
  removeChild(parent, child) {
    parent.removeChild(child);
  },
  commitUpdate(domElement, updatePayload) {
    // Apply property changes to DOM element
    applyDOMProperties(domElement, updatePayload);
  },
  // ... ~40 more methods
};

2. createRoot vs Legacy ReactDOM.render

// Legacy API (React 17 and earlier)
ReactDOM.render(<App />, document.getElementById('root'));

// React 18 API
const root = ReactDOM.createRoot(document.getElementById('root'));
root.render(<App />);

3. Property vs Attribute Handling

React uses DOM properties (not HTML attributes) for most element interactions. This distinction is subtle but important.

// React JSX → DOM property (not attribute)
<div className="box" />
// → element.className = "box" (property)
// NOT element.setAttribute("class", "box")

<label htmlFor="input" />
// → element.htmlFor = "input" (property)
// HTML attribute is "for", but JS property is "htmlFor"

<input value="hello" />
// → element.value = "hello" (property)
// This is how controlled inputs work

<div tabIndex={0} />
// → element.tabIndex = 0 (property)
// HTML attribute is "tabindex" (lowercase)

3.1 JSX to DOM Name Mapping

JSX Prop HTML Attribute DOM Property Why Different
className class className class is a JS reserved word
htmlFor for htmlFor for is a JS reserved word
onChange onchange N/A (React event) React's synthetic event system
dangerouslySetInnerHTML innerHTML innerHTML Name signals danger
style (object) style (string) style (CSSStyleDeclaration) React uses object, sets properties individually

4. Controlled vs Uncontrolled Components

// CONTROLLED: React owns the value
function ControlledInput() {
  const [value, setValue] = useState('');
  return (
    <input
      value={value}
      onChange={e => setValue(e.target.value)}
    />
  );
}
// React sets element.value on every render
// User input triggers onChange → setState → re-render → DOM update
// React is the "source of truth"

// UNCONTROLLED: DOM owns the value
function UncontrolledInput() {
  const ref = useRef(null);
  return (
    <input
      ref={ref}
      defaultValue="initial"
      onChange={() => console.log(ref.current.value)}
    />
  );
}
// React sets defaultValue only on mount
// DOM manages the value after that
// Read value via ref when needed
Controlled Input Internals

When React sets element.value on a controlled input, the browser doesn't fire a change event (programmatic value changes don't trigger events). React must explicitly set the value after every render to keep the DOM in sync with state. If you set value without onChange, the input becomes read-only — user typing is immediately overwritten by React's render.

5. Event Delegation System

React doesn't attach event listeners to individual DOM elements. Instead, it uses event delegation — a single listener on the root container captures all events.

// React 17+ attaches listeners to the root container
// (React 16 attached to document)

const root = createRoot(container);
// React attaches listeners to `container`:
// container.addEventListener('click', dispatchEvent, false)
// container.addEventListener('click', dispatchEvent, true) // capture
// container.addEventListener('change', dispatchEvent, false)
// ... for ALL supported event types

// When a click happens on <button> inside the tree:
// 1. Native event bubbles up to container
// 2. React's dispatchEvent fires
// 3. React finds the fiber for the event target
// 4. Walks UP the fiber tree collecting onClick handlers
// 5. Creates a SyntheticEvent
// 6. Calls handlers in order (capture phase, then bubble phase)
Event flow in React 17+: Native DOM event: button → div → section → container (React root) │ ▼ React's dispatchEvent() │ ▼ Find fiber for event.target │ ▼ Walk fiber tree (return pointers) Collect all onClick handlers │ ▼ Create SyntheticEvent │ ▼ Execute handlers: 1. Capture phase (top → bottom) 2. Bubble phase (bottom → top)

5.1 Why Event Delegation?

Benefit Explanation
Memory efficiency One listener per event type vs one per element
Dynamic elements New elements automatically handled (no add/remove listeners)
Consistent behavior React controls event timing and batching
Priority integration React assigns lanes based on event type
Portals Events from portals bubble through React tree, not DOM tree

6. SyntheticEvent

// SyntheticEvent wraps the native browser event
// with a cross-browser compatible interface

function handleClick(event) {
  // event is a SyntheticEvent, not a native Event
  event.type;            // "click"
  event.target;          // DOM element that was clicked
  event.currentTarget;   // DOM element with the handler
  event.nativeEvent;     // The underlying native browser event
  event.preventDefault();  // Works as expected
  event.stopPropagation(); // Stops React's synthetic propagation
}

// React 17+ changes:
// - Event pooling REMOVED (events are no longer reused)
// - Events attached to root, not document
// - onScroll no longer bubbles (matches native behavior)
// - onFocus/onBlur use native focusin/focusout under the hood

// React 16 had event pooling:
// SyntheticEvents were reused across different events
// event.persist() was needed to keep a reference
// This was removed in React 17 — events are normal objects now

7. Event Priorities

// React assigns different priorities to different event types
// This determines the lane for setState calls within handlers

// Discrete events → SyncLane (highest priority)
// click, keydown, keyup, mousedown, mouseup, input, change,
// focus, blur, submit, reset, pointerdown, pointerup
// → Processed synchronously, user expects immediate response

// Continuous events → InputContinuousLane
// mousemove, mouseenter, mouseleave, mouseover, mouseout,
// pointermove, pointerenter, pointerleave, scroll, drag,
// dragenter, dragleave, dragover, touchmove, wheel
// → May be batched more aggressively, slight delay acceptable

// Default events → DefaultLane
// Everything else (load, error, animation events, etc.)
// → Standard priority

8. Architect Considerations

Staff-Level Event System Knowledge

Understanding React's event delegation explains several behaviors: (1) event.stopPropagation() in React doesn't stop native propagation to the root — use event.nativeEvent.stopImmediatePropagation() if needed. (2) Events from portals bubble through the React tree (fiber return pointers), not the DOM tree — a click inside a portal bubbles to the portal's React parent, not its DOM parent. (3) Mixing React and non-React event handlers on the same element can cause ordering issues — React's handler runs when the event reaches the root, which may be after native handlers on intermediate elements.
