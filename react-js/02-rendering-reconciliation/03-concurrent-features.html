1. What Is Concurrent Rendering?

Concurrent rendering means React can work on multiple versions of the UI simultaneously, interrupt low-priority work to handle high-priority updates, and keep the UI responsive even during expensive renders.

Synchronous rendering: ───────────────────── User types "abc" while list is rendering: ├──── Render 10,000 items (200ms) ────┤ Paint ┤ Process "a" ┤ Process "b" ┤ UI frozen for 200ms! User sees delayed input Concurrent rendering: ──────────────────── User types "abc" while list is rendering: ├─ Render ─┤ "a" ┤─ Render ─┤ "b" ┤─ Render ─┤ "c" ┤─ Render ─┤ Paint ┤ 5ms input 5ms input 5ms input 5ms (interrupted by user input — restart with new query) UI stays responsive!
Key Insight

Concurrent rendering doesn't make rendering faster — it makes the UI feel faster by ensuring user interactions are never blocked. The total CPU work is the same (or slightly more due to interrupted/restarted renders), but the perceived responsiveness is dramatically better. This is the same principle as preemptive multitasking in operating systems.

2. startTransition and useTransition

2.1 startTransition

import { startTransition } from 'react';

function SearchPage() {
  const [query, setQuery] = useState('');
  const [results, setResults] = useState([]);

  function handleChange(e) {
    // Urgent: update input immediately
    setQuery(e.target.value);

    // Non-urgent: update results list (can be interrupted)
    startTransition(() => {
      setResults(filterLargeList(e.target.value));
    });
  }

  return (
    <>
      <input value={query} onChange={handleChange} />
      <ResultsList results={results} />
    </>
  );
}

2.2 useTransition

import { useTransition } from 'react';

function TabContainer() {
  const [isPending, startTransition] = useTransition();
  const [tab, setTab] = useState('home');

  function selectTab(nextTab) {
    startTransition(() => {
      setTab(nextTab);
    });
  }

  return (
    <>
      <TabBar onSelect={selectTab} />
      {isPending && <Spinner />}
      <TabContent tab={tab} />
    </>
  );
}

// isPending is true while the transition is rendering
// Use it to show loading indicators without blocking the UI

2.3 How Transitions Work Internally

// When startTransition wraps a setState:
// 1. React assigns the update a TransitionLane (not SyncLane)
// 2. TransitionLane work is interruptible
// 3. If a higher-priority update arrives (SyncLane):
//    a. React pauses the transition render
//    b. Processes the sync update immediately
//    c. Commits the sync update to DOM
//    d. Restarts the transition render with latest state
// 4. The old UI stays visible until the transition completes

// This is fundamentally different from debouncing:
// - Debounce: delays the UPDATE (user types, waits, then filters)
// - Transition: delays the RENDER (update happens immediately,
//   rendering is interruptible and shows old UI while computing)

3. useDeferredValue

import { useDeferredValue } from 'react';

function SearchResults({ query }) {
  // deferredQuery lags behind query during heavy renders
  const deferredQuery = useDeferredValue(query);

  // Show stale results while new ones are computing
  const isStale = query !== deferredQuery;

  return (
    <div style={{ opacity: isStale ? 0.5 : 1 }}>
      <ExpensiveList query={deferredQuery} />
    </div>
  );
}

// How it works:
// 1. First render: deferredQuery = query (same value)
// 2. When query changes: React renders with OLD deferredQuery first
//    (fast — ExpensiveList doesn't re-render if memoized)
// 3. Then starts a background render with NEW deferredQuery
//    (interruptible — uses TransitionLane)
// 4. When background render completes: swap in new results
Aspect useTransition useDeferredValue
Wraps The state update (setState) The value (prop/state)
Control You control which updates are transitions React defers the value automatically
Use when You own the setState call You receive a value from props
isPending Yes (from useTransition) Compare value !== deferredValue
Internal mechanism Assigns TransitionLane to update Re-renders with TransitionLane

4. Interruptible Render Trees

// In concurrent mode, React can maintain multiple "in-progress" renders:

// Scenario: User types "a", "b", "c" rapidly

// Render 1: query="a" (TransitionLane)
//   Processing... 3ms into 200ms render
//   → User types "b" (SyncLane — higher priority)
//   → INTERRUPT render 1

// Sync render: update input to show "ab"
//   → Committed immediately

// Render 2: query="ab" (TransitionLane — restarts from scratch)
//   Processing... 5ms into 200ms render
//   → User types "c" (SyncLane)
//   → INTERRUPT render 2

// Sync render: update input to show "abc"

// Render 3: query="abc" (TransitionLane — restarts again)
//   Processing... completes after 200ms (no more interruptions)
//   → Committed: results for "abc" appear

// User saw: instant input updates + results appear once
// Without concurrent: input frozen for 200ms × 3 = 600ms

5. Selective Hydration with Suspense

// Selective hydration: hydrate components on interaction

// Server sends HTML with Suspense boundaries:
<Suspense fallback={<Spinner />}>
  <Comments />  {/* Heavy component */}
</Suspense>

// Hydration behavior:
// 1. React starts hydrating the page top-down
// 2. When it hits a Suspense boundary, it can SKIP it
//    and continue hydrating other parts
// 3. If user clicks on the un-hydrated Comments section:
//    → React PRIORITIZES hydrating Comments
//    → Interrupts other hydration work
//    → Hydrates Comments synchronously
//    → Replays the click event
// 4. User never notices the component wasn't hydrated
Selective Hydration Requirements

Selective hydration requires: (1) Streaming SSR with renderToPipeableStream, (2) Suspense boundaries around independently-hydratable sections, (3) hydrateRoot (not legacy hydrate). The Suspense boundaries act as hydration units — React can hydrate them independently and in any order based on user interaction priority.

6. Transition Lanes

// React has 16 transition lanes (TransitionLane1 through TransitionLane16)
// This allows multiple independent transitions to coexist

// Example: Two independent transitions
startTransition(() => setFilter(newFilter));   // Gets TransitionLane1
startTransition(() => setSort(newSort));       // Gets TransitionLane2

// React can process these independently:
// - If filter completes first, commit it
// - Sort continues rendering in background
// - Each transition shows its own isPending state

// Lane assignment is round-robin:
// First transition → Lane1, second → Lane2, ..., 17th → Lane1 (wraps)
// This prevents lane exhaustion in apps with many transitions

7. Concurrent vs Synchronous Rendering

Aspect Synchronous (Legacy) Concurrent (React 18+)
Rendering Uninterruptible, blocks main thread Interruptible, yields every 5ms
Priority All updates equal Lane-based priority system
Multiple versions One version of UI at a time Multiple concurrent renders
User input during render Blocked until render completes Processed between time slices
Transitions Not available startTransition, useDeferredValue
Suspense for data Limited (code splitting only) Full support (data fetching, streaming)
Hydration All-or-nothing Selective, priority-based

8. Real-World Use Cases

8.1 Search with Live Filtering

function ProductSearch({ products }) {
  const [query, setQuery] = useState('');
  const deferredQuery = useDeferredValue(query);
  const isStale = query !== deferredQuery;

  const filtered = useMemo(
    () => products.filter(p =>
      p.name.toLowerCase().includes(deferredQuery.toLowerCase())
    ),
    [products, deferredQuery]
  );

  return (
    <>
      <input value={query} onChange={e => setQuery(e.target.value)} />
      <div style={{ opacity: isStale ? 0.7 : 1 }}>
        <ProductGrid products={filtered} />
      </div>
    </>
  );
}

8.2 Tab Switching with Heavy Content

function Dashboard() {
  const [tab, setTab] = useState('overview');
  const [isPending, startTransition] = useTransition();

  return (
    <>
      <nav>
        {tabs.map(t => (
          <button
            key={t.id}
            onClick={() => startTransition(() => setTab(t.id))}
            className={tab === t.id ? 'active' : ''}
          >
            {t.label}
          </button>
        ))}
      </nav>
      {isPending && <LoadingBar />}
      <TabContent tab={tab} />
    </>
  );
}

9. Architect Considerations

Staff-Level Design Principles

Concurrent features are not "opt-in performance" — they're a UX architecture decision. At L6, you should be able to: (1) Identify which updates are urgent (input, navigation) vs non-urgent (filtering, sorting, data display). (2) Design component hierarchies where expensive renders are wrapped in transitions. (3) Use Suspense boundaries as both loading and hydration units. (4) Understand that concurrent rendering can cause "tearing" if components read from external mutable stores — this is why useSyncExternalStore exists. (5) Measure the impact with React Profiler and browser performance tools.
