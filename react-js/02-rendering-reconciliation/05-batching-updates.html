1. What Is Batching?

Batching is React's optimization of grouping multiple state updates into a single re-render. Without batching, each setState would trigger a separate render — with batching, React collects all updates and processes them together.

Without batching: ───────────────── setCount(1) → render → commit → paint setName("a") → render → commit → paint setFlag(true)→ render → commit → paint Total: 3 renders, 3 paints With batching: ────────────── setCount(1) ─┐ setName("a") ─┤→ single render → commit → paint setFlag(true) ─┘ Total: 1 render, 1 paint 
// Batching in action
function handleClick() {
  setCount(c => c + 1);  // Does NOT trigger render yet
  setName('Alice');       // Does NOT trigger render yet
  setFlag(true);           // Does NOT trigger render yet
  // → React batches all three into ONE render
  // → Component renders once with all three updates applied
}

2. Legacy Batching (React ≤17)

In React 17 and earlier, batching only worked inside React event handlers. Updates in setTimeout, Promises, and native event handlers were NOT batched.

// React 17 batching behavior:

// ✅ BATCHED: React event handler
function handleClick() {
  setCount(c => c + 1);
  setFlag(true);
  // → 1 render (batched)
}

// ❌ NOT BATCHED: setTimeout
setTimeout(() => {
  setCount(c => c + 1);  // → render 1
  setFlag(true);           // → render 2
}, 1000);

// ❌ NOT BATCHED: Promise
fetch('/api').then(() => {
  setCount(c => c + 1);  // → render 1
  setFlag(true);           // → render 2
});

// ❌ NOT BATCHED: native event listener
element.addEventListener('click', () => {
  setCount(c => c + 1);  // → render 1
  setFlag(true);           // → render 2
});

2.1 Why Legacy Batching Was Limited

// React 17 used a synchronous batching flag:
let isBatchingUpdates = false;

// React event handlers wrapped in:
function batchedUpdates(fn) {
  isBatchingUpdates = true;
  fn();  // Your event handler runs here
  isBatchingUpdates = false;
  flushUpdates();  // Process all queued updates
}

// Inside setState:
if (isBatchingUpdates) {
  queue.push(update);  // Queue for later
} else {
  processUpdate(update);  // Process immediately
}

// setTimeout/Promise callbacks run AFTER batchedUpdates returns,
// so isBatchingUpdates is false → each setState processes immediately
unstable_batchedUpdates

React 17 provided ReactDOM.unstable_batchedUpdates(fn) as an escape hatch to force batching in non-React contexts. Libraries like Redux used this internally to batch store updates. In React 18, this is no longer needed but still works for backward compatibility.

3. Automatic Batching (React 18+)

React 18 introduced automatic batching: all state updates are batched regardless of where they originate — event handlers, setTimeout, Promises, native events, or any other context.

// React 18: ALL of these are batched

// ✅ React event handler (same as before)
function handleClick() {
  setCount(c => c + 1);
  setFlag(true);
  // → 1 render
}

// ✅ setTimeout (NEW: now batched)
setTimeout(() => {
  setCount(c => c + 1);
  setFlag(true);
  // → 1 render (was 2 in React 17)
}, 1000);

// ✅ Promise (NEW: now batched)
fetch('/api').then(() => {
  setCount(c => c + 1);
  setFlag(true);
  // → 1 render (was 2 in React 17)
});

// ✅ Native event listener (NEW: now batched)
element.addEventListener('click', () => {
  setCount(c => c + 1);
  setFlag(true);
  // → 1 render (was 2 in React 17)
});

3.1 How Automatic Batching Works

// React 18 uses a different mechanism:
// Instead of a synchronous flag, it uses microtask scheduling.

// When setState is called:
// 1. Enqueue the update on the fiber's update queue
// 2. Mark the fiber and its ancestors with lanes
// 3. Schedule a render via microtask (if not already scheduled)

// The microtask runs AFTER the current synchronous code completes:
setCount(1);  // enqueue, schedule microtask
setFlag(true); // enqueue (microtask already scheduled)
// ... synchronous code finishes ...
// microtask fires → process ALL queued updates → 1 render

// This works regardless of context because microtasks
// always run after the current task completes.
Migration Note

Automatic batching requires createRoot (React 18 API). If you use the legacy ReactDOM.render, you get the old batching behavior. This is the primary reason to migrate to createRoot — it enables all React 18 features including automatic batching, concurrent rendering, and transitions.

4. flushSync — Opting Out of Batching

import { flushSync } from 'react-dom';

function handleClick() {
  flushSync(() => {
    setCount(c => c + 1);
  });
  // DOM is updated HERE — count is visible in DOM

  flushSync(() => {
    setFlag(true);
  });
  // DOM is updated HERE — flag is visible in DOM

  // Total: 2 renders (batching bypassed)
}

// Use cases for flushSync:
// 1. Reading DOM after state update (measurements)
flushSync(() => setItems([...items, newItem]));
const height = listRef.current.scrollHeight; // Accurate!

// 2. Third-party library integration that needs sync DOM
flushSync(() => setOpen(true));
thirdPartyLib.positionTooltip(ref.current); // DOM exists

// 3. Scroll to newly added item
flushSync(() => setMessages([...messages, newMsg]));
lastMessageRef.current.scrollIntoView();
flushSync Performance Warning

flushSync forces synchronous rendering — it bypasses batching AND concurrent features. The render and commit happen immediately, blocking the main thread. Use it sparingly and only when you need to read the DOM after a state update. In most cases, useLayoutEffect or useEffect are better alternatives.

5. Microtask Scheduling

// React 18's batching relies on microtask timing:

// Browser event loop order:
// 1. Execute macrotask (event handler, setTimeout callback)
// 2. Execute ALL microtasks (Promise.then, queueMicrotask)
// 3. Render (layout, paint)
// 4. Next macrotask

// React schedules renders as microtasks for SyncLane updates:
setState(x);  // Enqueue update, schedule microtask
setState(y);  // Enqueue update (microtask already scheduled)
// ... current task finishes ...
// [microtask] React processes all updates → render → commit
// [render] Browser paints

// For transition updates, React uses MessageChannel (macrotask)
// to allow yielding between time slices.
Event handler execution: ├── Macrotask: onClick handler ──────────────────────┤ │ setState(a) setState(b) setState(c) │ │ (all enqueued, microtask scheduled) │ ├────────────────────────────────────────────────────┤ │ Microtask: React render + commit │ │ (processes a, b, c together → 1 render) │ ├────────────────────────────────────────────────────┤ │ Browser: layout, paint │ ├────────────────────────────────────────────────────┤ setTimeout callback: ├── Macrotask: setTimeout callback ──────────────────┤ │ setState(a) setState(b) │ │ (all enqueued, microtask scheduled) │ ├────────────────────────────────────────────────────┤ │ Microtask: React render + commit │ │ (processes a, b together → 1 render) │ │ ← THIS is why React 18 batches setTimeout! │ ├────────────────────────────────────────────────────┤

6. Multiple setState Calls

// Multiple setState with same state variable:

// Pattern 1: Value updates (last one wins)
setCount(1);
setCount(2);
setCount(3);
// Result: count = 3 (only last value matters)

// Pattern 2: Updater functions (all applied sequentially)
setCount(c => c + 1);
setCount(c => c + 1);
setCount(c => c + 1);
// Result: count = prev + 3 (all three applied)

// Pattern 3: Mixed (updaters applied to latest value)
setCount(5);           // queue: [5]
setCount(c => c + 1); // queue: [5, c => c + 1]
// Processing: start with 5, apply c => c + 1 → 6
// Result: count = 6

// React processes the update queue during render:
// For each update in queue:
//   If function: newState = update(prevState)
//   If value: newState = update
// Final state = result of processing all updates

7. Batching and State Derivation

// Common question: can I read the "new" state after setState?

function handleClick() {
  setCount(count + 1);
  console.log(count); // Still the OLD value!
  // setState doesn't update the variable immediately.
  // `count` is a closure over the render's value.
  // The new value is only available in the NEXT render.
}

// If you need the new value immediately:
function handleClick() {
  const nextCount = count + 1;
  setCount(nextCount);
  logAnalytics(nextCount); // Use the computed value directly
}

// Or use useEffect to react to state changes:
useEffect(() => {
  logAnalytics(count); // Runs after render with new count
}, [count]);

8. Architect Considerations

8.1 Batching Behavior Summary

Context React 17 React 18 (createRoot) React 18 (legacy render)
React event handler Batched Batched Batched
setTimeout / setInterval Not batched Batched Not batched
Promise.then / async-await Not batched Batched Not batched
Native event listener Not batched Batched Not batched
Inside flushSync N/A Not batched (forced sync) Not batched
Staff-Level Insight

Automatic batching is one of the most impactful React 18 changes for existing apps. Code that previously caused multiple renders per event handler (especially in async contexts) now causes one render. This is usually a performance win, but can cause subtle behavior changes if code relied on intermediate renders (e.g., reading DOM between setState calls). The migration path: upgrade to createRoot, test thoroughly, and use flushSync only where intermediate DOM reads are required. At scale, automatic batching can reduce render counts by 30-50% in apps with heavy async state updates.
