1. Scheduler Package Overview

React's scheduler is a separate package (scheduler) that manages task prioritization and time-slicing. It's React's own cooperative scheduling system, independent of the browser's built-in scheduling APIs.

2. Priority Lanes System

Lanes are a 31-bit bitmask system that replaced the older "expiration time" priority model. Each lane represents a priority level, and lanes can be combined using bitwise operations.

// Lane definitions (from ReactFiberLane.js)
const TotalLanes = 31;

// Individual lanes (each is a single bit)
const NoLane           = 0b0000000000000000000000000000000;
const SyncLane         = 0b0000000000000000000000000000010;
const InputContinuousLane = 0b0000000000000000000000000001000;
const DefaultLane      = 0b0000000000000000000000000100000;

// Transition lanes (16 lanes for concurrent transitions)
const TransitionLane1  = 0b0000000000000000000000001000000;
const TransitionLane2  = 0b0000000000000000000000010000000;
// ... through TransitionLane16

const IdleLane         = 0b0100000000000000000000000000000;
const OffscreenLane    = 0b1000000000000000000000000000000;

2.1 Why Lanes Replaced Expiration Times

Aspect Expiration Times (old) Lanes (current)
Representation Single number (ms timestamp) 31-bit bitmask
Multiple priorities Can't batch different priorities Combine with bitwise OR
Priority check Numeric comparison Bitwise AND (faster)
Concurrent transitions All transitions share one priority 16 separate transition lanes
Selective rendering All-or-nothing Process specific lanes independently

2.2 Lane Operations

// Combine lanes
const pendingLanes = SyncLane | DefaultLane;

// Check if a lane is included
const hasSyncWork = (pendingLanes & SyncLane) !== NoLane; // true

// Remove a lane
const remaining = pendingLanes & ~SyncLane; // only DefaultLane

// Get highest priority lane
const highest = pendingLanes & -pendingLanes; // lowest set bit

// Check if lanes overlap
const overlap = (lanesA & lanesB) !== NoLane;

3. MessageChannel Scheduling

React uses MessageChannel to schedule work in the browser's macrotask queue. This was chosen over setTimeout, requestAnimationFrame, and requestIdleCallback for specific reasons.

// React's scheduling mechanism (simplified from SchedulerHostConfig.js)
const channel = new MessageChannel();
const port = channel.port2;

let scheduledHostCallback = null;
let isMessageLoopRunning = false;

channel.port1.onmessage = function performWorkUntilDeadline() {
  if (scheduledHostCallback !== null) {
    const currentTime = getCurrentTime();
    const deadline = currentTime + yieldInterval; // 5ms

    let hasMoreWork = true;
    try {
      hasMoreWork = scheduledHostCallback(currentTime);
    } finally {
      if (hasMoreWork) {
        port.postMessage(null); // Schedule next time slice
      } else {
        isMessageLoopRunning = false;
        scheduledHostCallback = null;
      }
    }
  }
};

function requestHostCallback(callback) {
  scheduledHostCallback = callback;
  if (!isMessageLoopRunning) {
    isMessageLoopRunning = true;
    port.postMessage(null); // Trigger first time slice
  }
}

3.1 Why Not Other APIs?

API Problem for React
setTimeout(fn, 0) HTML spec mandates 4ms minimum delay after 5 nested calls. Too slow for 5ms time slicing.
requestAnimationFrame Fires once per frame (~16ms). Can't do multiple work slices per frame. Paused when tab is hidden.
requestIdleCallback ~50ms intervals, inconsistent timing, not in Safari, no priority support, throttled when backgrounded.
queueMicrotask Microtasks run before browser can paint — would block rendering just like synchronous code.
MessageChannel Fires as macrotask with near-zero delay. Browser can paint between messages. Works in all browsers.

4. Time Slicing: 5ms Yield Intervals

// React's default yield interval
let yieldInterval = 5; // milliseconds

// Why 5ms?
// At 60fps, each frame has ~16.6ms budget:
//   - Browser input handling: ~2ms
//   - React work: ~5ms
//   - Layout + Paint: ~5ms
//   - Compositing: ~2ms
//   - Buffer: ~2ms
//
// 5ms gives React enough time to make progress
// while leaving room for the browser to stay responsive.

// The yield interval is adjustable:
function forceFrameRate(fps) {
  if (fps > 0) {
    yieldInterval = Math.floor(1000 / fps);
  } else {
    yieldInterval = 5; // Reset to default
  }
}

5. Task Expiration and Starvation Prevention

Each priority level has an expiration timeout. If a task isn't completed before its expiration, it's promoted to synchronous priority to prevent starvation.

// Expiration timeouts by priority
const IMMEDIATE_PRIORITY_TIMEOUT = -1;     // Already expired (sync)
const USER_BLOCKING_PRIORITY_TIMEOUT = 250; // 250ms
const NORMAL_PRIORITY_TIMEOUT = 5000;      // 5 seconds
const LOW_PRIORITY_TIMEOUT = 10000;        // 10 seconds
const IDLE_PRIORITY_TIMEOUT = 1073741823;  // ~12.4 days (never)

// When scheduling a task:
const expirationTime = startTime + timeout;

// In the work loop, expired tasks are processed synchronously:
function workLoop(currentTime) {
  let currentTask = peek(taskQueue);

  while (currentTask !== null) {
    if (currentTask.expirationTime > currentTime && shouldYield()) {
      // Not expired AND time slice exhausted → yield
      break;
    }

    // Either expired OR still within time slice → process
    const callback = currentTask.callback;
    const continuationCallback = callback(currentTime);

    if (typeof continuationCallback === 'function') {
      currentTask.callback = continuationCallback; // More work
    } else {
      pop(taskQueue); // Task complete
    }
    currentTask = peek(taskQueue);
  }

  return currentTask !== null; // hasMoreWork
}
Starvation Prevention

Without expiration, a continuous stream of high-priority updates (e.g., rapid typing) could starve lower-priority work (e.g., a transition) indefinitely. The expiration mechanism ensures that even low-priority work eventually gets processed. After 5 seconds, a normal-priority task becomes synchronous — React processes it immediately regardless of other pending work. This is a critical fairness guarantee in the scheduling system.

6. shouldYield() Mechanism

// shouldYield checks if the current time slice is exhausted
function shouldYieldToHost() {
  const currentTime = getCurrentTime();

  // Check if we've exceeded the 5ms deadline
  if (currentTime >= deadline) {
    // Check if there's pending browser work
    if (needsPaint() || isInputPending()) {
      return true; // Yield to browser
    }
    // No pending browser work — extend the deadline
    // (up to 300ms max to prevent starvation)
    const elapsed = currentTime - startTime;
    if (elapsed < 300) {
      return false; // Continue working
    }
  }
  return false;
}

// isInputPending() is a browser API (navigator.scheduling.isInputPending)
// that checks if there's pending user input (click, key, etc.)
// React uses this to prioritize user responsiveness
shouldYield() decision flow: Time slice started (5ms budget) │ ┌────▼────┐ │ 5ms │ No → continue working │ elapsed?│ └────┬────┘ │ Yes ┌────▼──────────┐ │ Input pending? │ Yes → YIELD (user wants to interact) │ Need paint? │ └────┬──────────┘ │ No ┌────▼────────┐ │ 300ms total │ Yes → YIELD (starvation prevention) │ elapsed? │ └────┬────────┘ │ No Continue working (browser is idle anyway)

8.1 Scheduler Priority Mapping

// Discrete events (click, keydown) → SyncLane → ImmediatePriority
// These must be processed synchronously for responsiveness

// Continuous events (mousemove, scroll) → InputContinuousLane

// Normal updates (setState in useEffect) → DefaultLane

Staff-Level Insight — Scheduler Independence

React's scheduler is a separate package that could theoretically be used by any JavaScript application needing cooperative scheduling. It implements a min-heap priority queue with expiration-based starvation prevention — a well-studied scheduling algorithm. Understanding this separation helps explain why React can target multiple platforms (DOM, Native, Canvas) — the scheduler is platform-agnostic, and only the host config (how to create/update/delete platform nodes) changes per renderer.
