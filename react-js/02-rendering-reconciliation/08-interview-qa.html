Render Phases

Q1. Describe the two phases of React rendering and what happens in each. Senior

Render phase: Pure computation — React calls component functions, diffs old vs new elements, and collects effect flags on fibers. This phase is interruptible (can pause/resume) and may run multiple times (StrictMode). No DOM mutations, no side effects. Functions: beginWork (descend into children) and completeWork (bubble up, create DOM nodes in memory).

Commit phase: Synchronous — applies all collected effects to the real DOM. Three sub-phases: Before Mutation (getSnapshotBeforeUpdate), Mutation (DOM insertions/updates/deletions), Layout (useLayoutEffect, componentDidMount/Update, ref assignments). After commit, passive effects (useEffect) are scheduled asynchronously.

Q2. What is beginWork and what does it do for different fiber types? L6

beginWork processes a fiber during the downward traversal. For FunctionComponent: calls the function, processes hooks, reconciles children. For ClassComponent: calls render(), manages lifecycle. For HostComponent (div): reconciles children elements. For ContextProvider: pushes context value. For SuspenseComponent: handles suspension state. It first checks for bailout (same props, no pending lanes) — if nothing changed, it skips the entire subtree. Returns the first child fiber or null (leaf node).

Q3. What is completeWork and why does it run bottom-up? L6

completeWork runs during the upward traversal after all children are processed. For HostComponent: creates the DOM element (on mount) or diffs properties (on update), appends children to the DOM element. It runs bottom-up because children must exist before the parent can append them. It also bubbles up effect flags via bubbleProperties — aggregating subtreeFlags from children so the commit phase knows which subtrees need work. This aggregation is a critical optimization: if subtreeFlags is NoFlags, the commit phase skips the entire subtree.

Q4. Explain the three commit sub-phases and what runs in each. L6

Before Mutation: DOM is still old. Runs getSnapshotBeforeUpdate (class) — last chance to read old DOM. Mutation: DOM changes happen. Placements (appendChild/insertBefore), updates (property changes), deletions (removeChild + cleanup effects). Deletions run first to avoid conflicts. After mutation, root.current = finishedWork swaps trees. Layout: DOM is new but not painted. Runs useLayoutEffect, componentDidMount/Update, ref callbacks. Can read updated layout (getBoundingClientRect). Blocks painting — keep it fast. After layout, passive effects (useEffect) are scheduled as a separate async task.

Q5. What is the difference between useEffect and useLayoutEffect timing? Senior

useLayoutEffect runs synchronously after DOM mutations but before the browser paints. The user never sees the intermediate state. Use for DOM measurements, scroll position, tooltip positioning. useEffect runs asynchronously after the browser paints. The user sees the updated DOM before effects run. Use for data fetching, subscriptions, analytics. useLayoutEffect blocks painting — long-running code causes visible jank. useEffect doesn't block — it's the default choice for most side effects.

Q6. Why does StrictMode double-invoke render functions? Senior

StrictMode double-invokes render functions, state initializers, and state updaters to detect impure renders. If your render has side effects (mutating external variables, making API calls, modifying DOM), the double invocation exposes them — the side effect happens twice. This is critical for concurrent rendering: React may start rendering, pause, and restart from scratch. If render has side effects, the restart would cause them to run again, leading to bugs. StrictMode catches these issues in development. The double invocation only happens in development mode.

Q7. What is useInsertionEffect and when would you use it? L6

useInsertionEffect runs before any DOM mutations — even before useLayoutEffect. It's designed exclusively for CSS-in-JS libraries (Emotion, styled-components) to inject <style> tags before layout effects read computed styles. Without it, a layout effect might read stale styles because the CSS-in-JS library hasn't injected the new styles yet. Regular application code should never use useInsertionEffect — it can't access refs or schedule updates. The execution order is: useInsertionEffect → DOM mutations → useLayoutEffect → paint → useEffect.

Q8. How do effect flags and subtreeFlags optimize the commit phase? L6

Each fiber has flags (its own effects: Placement, Update, Deletion) and subtreeFlags (aggregated from all descendants). During completeWork, child flags bubble up: parent.subtreeFlags |= child.flags | child.subtreeFlags. In the commit phase, if a fiber's subtreeFlags === NoFlags, React skips the entire subtree — no descendants need work. This turns the commit phase from O(n) (visit every fiber) to O(changed) (visit only fibers with effects). In a large app where most of the tree is unchanged, this is a massive optimization.

Q9. What happens if you call setState inside useLayoutEffect? Senior

React processes the state update synchronously before the browser paints. This triggers another render → commit cycle within the same commit phase. The user sees only the final result — no intermediate state is visible. This is useful for measuring DOM and adjusting layout (e.g., tooltip positioning). However, it's expensive: two full render cycles before a single paint. Use sparingly and only when you need to read DOM layout and immediately update based on it. The React docs call this a "synchronous re-render."

Q10. Trace a setState call through the complete render → commit pipeline. L6

(1) setState(newValue) enqueues an update on the fiber's update queue. (2) React marks the fiber and ancestors with lanes. (3) Schedules a render via microtask (SyncLane) or MessageChannel (TransitionLane). (4) Render phase: creates workInProgress tree, calls beginWork on each fiber (top-down), processes the update queue to compute new state, calls component function with new state, reconciles children, calls completeWork (bottom-up) to create DOM nodes and collect flags. (5) Commit phase: Before Mutation (snapshots), Mutation (DOM changes), tree swap, Layout (useLayoutEffect, refs). (6) Schedules passive effects (useEffect) as async task. (7) Browser paints. (8) Passive effects run.

Scheduler & Priorities

Q11. Why does React use MessageChannel instead of setTimeout for scheduling? L6

setTimeout(fn, 0) has a minimum 4ms delay after 5 nested calls (HTML spec). For 5ms time slicing, this means React would waste 4ms waiting between slices — nearly doubling the time per slice. MessageChannel fires as a macrotask with near-zero delay. The browser can paint and handle input between MessageChannel callbacks, maintaining responsiveness. React also rejected requestAnimationFrame (once per frame, paused when hidden) and requestIdleCallback (50ms intervals, no Safari support, no priority control).

Q12. Explain the 5ms time slice and how shouldYield() works. L6

React's work loop processes one fiber at a time. After each fiber, it calls shouldYield() which checks: (1) Has 5ms elapsed since the slice started? If no, continue. (2) If yes, is there pending user input (navigator.scheduling.isInputPending()) or does the browser need to paint? If yes, yield. (3) If no pending browser work, continue up to 300ms total to prevent starvation. The 5ms budget leaves ~11ms per frame for browser layout, paint, and input handling at 60fps. This interleaving is what makes React feel responsive during heavy renders.

Q13. How does task expiration prevent starvation? L6

Each scheduled task has an expiration time based on priority: Immediate (-1ms, already expired), UserBlocking (250ms), Normal (5000ms), Low (10000ms), Idle (~12 days). In the work loop, expired tasks are processed synchronously regardless of shouldYield() — they can't be interrupted. This prevents a continuous stream of high-priority updates (e.g., rapid typing) from starving lower-priority work (e.g., a data fetch transition) indefinitely. After 5 seconds, a normal-priority task becomes synchronous and must be processed.

Q14. What data structure does the scheduler use for its task queue? L6

The scheduler uses a min-heap (binary heap) sorted by expiration time. peek() returns the task with the earliest expiration in O(1). push() and pop() are O(log n). There are actually two heaps: taskQueue (tasks ready to run, sorted by expiration) and timerQueue (delayed tasks, sorted by start time). When a timer's start time arrives, it's moved from timerQueue to taskQueue. This is more efficient than a sorted array (O(n) insert) for the typical workload of many tasks with varying priorities.

Q15. How do lanes differ from the old expiration time system? L6

Expiration times used a single number — you couldn't express "this fiber has both a sync update AND a transition update pending." Lanes use a 31-bit bitmask where each bit represents a priority level. Multiple lanes combine with bitwise OR: SyncLane | TransitionLane1. This enables: (1) Multiple concurrent transitions (16 transition lanes), (2) Selective rendering (process only specific lanes), (3) Fast priority checks (bitwise AND), (4) Independent tracking of different update sources on the same fiber. Lanes are the foundation of concurrent rendering's ability to process different priority updates independently.

Q16. How does React assign lanes to different types of updates? Senior

Discrete events (click, keydown) → SyncLane (synchronous, immediate). Continuous events (mousemove, scroll) → InputContinuousLane (high priority, may batch). Normal setState (in useEffect, setTimeout) → DefaultLane. startTransition(() => setState()) → TransitionLane (interruptible, low priority). flushSync(() => setState()) → SyncLane (forced synchronous). useDeferredValue → TransitionLane (deferred rendering). The lane assignment happens at the point of the setState call, based on the current execution context (event type, transition wrapper, etc.).

Q17. Why are there 16 transition lanes instead of just one? L6

Multiple transition lanes allow React to track and process different transitions independently. If a filter transition and a sort transition are both pending, they get different lanes (TransitionLane1, TransitionLane2). React can: (1) Complete the filter transition and commit it while sort is still rendering. (2) Interrupt one transition for another without losing progress on the first. (3) Show independent isPending states for each transition. Lane assignment is round-robin: first transition gets Lane1, second gets Lane2, etc., wrapping after Lane16. This prevents lane exhaustion in apps with many concurrent transitions.

Q18. What is isInputPending and how does React use it? L6

navigator.scheduling.isInputPending() is a browser API that checks if there's pending user input (click, key press, etc.) without yielding. React uses it in shouldYield(): even if the 5ms time slice isn't exhausted, if the user has pending input, React yields immediately to ensure the input is processed promptly. This is a more intelligent yielding strategy than pure time-based — React continues working when the browser is idle (no pending input) but yields immediately when the user wants to interact. Not all browsers support it; React falls back to time-based yielding.

Q19. How does the scheduler interact with the browser's event loop? L6

React's scheduler posts macrotasks via MessageChannel. Each macrotask is one time slice (~5ms) of React work. Between macrotasks, the browser's event loop can: process input events (click, keypress), run microtasks (Promise callbacks), perform layout and paint, process other macrotasks. This interleaving ensures the browser stays responsive. For SyncLane updates, React uses microtask scheduling (queueMicrotask) — these run before the browser can paint, ensuring synchronous updates are committed in the same frame as the triggering event.

Q20. Can React's scheduler be used independently of React? L6

Yes — the scheduler package is published separately on npm and has no React dependencies. It implements a generic cooperative scheduling system with priority queues, time slicing, and starvation prevention. Any JavaScript application needing to schedule work with priorities could use it. However, it's not widely used outside React because: (1) The API is marked as unstable, (2) Most apps don't need custom scheduling, (3) The browser's own scheduling APIs (requestIdleCallback, scheduler.postTask) are evolving. The React team has been involved in the scheduler.postTask browser API proposal, which would provide native priority scheduling.

Concurrent Features

Q21. What is the difference between startTransition and debouncing? Senior

Debouncing delays the update — the user types, waits for a delay, then the state updates and renders. During the delay, nothing happens. startTransition delays the render — the state updates immediately, but React renders the update at low priority (TransitionLane). The old UI stays visible while the new UI renders in the background. If a higher-priority update arrives (user types again), React interrupts the transition and restarts with the latest state. Transitions are more responsive because the input always updates immediately, and the expensive render is interruptible.

Q22. When should you use useTransition vs useDeferredValue? Senior

useTransition: when you own the setState call and want to mark it as non-urgent. Wraps the setState in startTransition. Provides isPending for loading indicators. useDeferredValue: when you receive a value from props and want to defer its rendering. Creates a "lagging" copy of the value. Compare value !== deferredValue for staleness. Rule of thumb: if you control the state update, use useTransition. If you receive a prop that changes frequently, use useDeferredValue. Both use TransitionLane internally.

Q23. What is "tearing" in concurrent rendering and how is it prevented? L6

Tearing occurs when different parts of the UI show data from different points in time. In concurrent rendering, React may pause mid-render. If a component reads from an external mutable store (Redux, Zustand), the store might change between the first and second halves of the render — some components see the old value, others see the new value. useSyncExternalStore prevents this by: (1) Reading the store synchronously during render, (2) Checking if the value changed after render completes, (3) If changed, re-rendering synchronously to ensure consistency. Libraries must use this hook (or its equivalent) to be concurrent-safe.

Q24. How does selective hydration work with Suspense boundaries? L6

Suspense boundaries serve as hydration units. During hydration: (1) React starts hydrating top-down. (2) When it hits a Suspense boundary whose HTML hasn't arrived yet, it skips it and continues with other parts. (3) When the HTML streams in, React hydrates that boundary. (4) If the user interacts with an un-hydrated section, React prioritizes that section's hydration — it interrupts other hydration work, hydrates the clicked section synchronously, and replays the user's event. This requires streaming SSR (renderToPipeableStream) and hydrateRoot. The result: the most important sections (the ones the user interacts with) hydrate first.

Q25. What happens when a transition is interrupted by a higher-priority update? L6

(1) React is rendering a transition (TransitionLane). (2) User clicks a button (SyncLane — higher priority). (3) React checks shouldYield() → sees pending input → yields. (4) React processes the SyncLane update: full render → commit → paint. (5) React restarts the transition render from scratch with the latest state (including the sync update). The interrupted transition's partial work is discarded (safe because render is pure). The user sees the sync update immediately, and the transition result appears later. This is why render purity is essential — discarded renders must have no side effects.

Q26. Explain useSyncExternalStore and why it exists. L6

useSyncExternalStore(subscribe, getSnapshot, getServerSnapshot) safely reads from external mutable stores in concurrent rendering. It prevents tearing by: (1) Reading the store value synchronously during render. (2) Subscribing to store changes. (3) After render, checking if the value changed during the render. (4) If changed, forcing a synchronous re-render to ensure all components see the same value. The getServerSnapshot parameter provides the value during SSR (where subscriptions don't exist). Redux, Zustand, and other state libraries use this internally. It's the only safe way to read mutable external state in concurrent React.

Q27. Does concurrent rendering make React faster? Senior

No — concurrent rendering doesn't reduce the total CPU work. It may even increase it slightly (interrupted renders are restarted, overhead of scheduling). What it does is make the UI feel faster by ensuring user interactions are never blocked. The total time to complete a render is the same or slightly longer, but the time between user action and visible response is much shorter. It's the same principle as preemptive multitasking in operating systems — the CPU does the same total work, but no single task monopolizes it. The real performance gains come from avoiding unnecessary work (memoization, bailouts) and reducing the amount of work per render (React Compiler).

Q28. How do you measure the impact of concurrent features? L6

Use React Profiler (DevTools) to measure render times and identify which components benefit from transitions. Use browser Performance tab to measure: (1) Input delay (time from user action to first response), (2) Total blocking time (TBT — sum of long tasks), (3) Interaction to Next Paint (INP — measures responsiveness). Compare with and without transitions: INP should improve significantly for expensive updates. Also measure: number of interrupted/restarted renders (React Profiler shows this), and whether transitions complete before the user's next action. At scale, track INP as a core web vital — concurrent features directly improve it.

Q29. What is the Activity API (formerly Offscreen)? L6

The Activity API (experimental, formerly called Offscreen) allows React to pre-render content that's not currently visible. Use cases: (1) Pre-render the next tab while the current tab is visible. (2) Keep unmounted route state alive (like Android's back stack). (3) Pre-render content below the fold. It uses OffscreenLane (lowest priority) so it doesn't interfere with visible UI updates. The pre-rendered content is hidden via CSS (display: none) but its fiber tree and state are preserved. When revealed, it's already rendered — instant display. This is React's answer to the "keep-alive" pattern.

Q30. Design a responsive search experience using concurrent features. L6

Architecture: (1) Input field updates with SyncLane (always responsive). (2) Results list uses useDeferredValue(query) — shows stale results while computing new ones. (3) Results component wrapped in React.memo — only re-renders when deferredQuery changes. (4) Visual feedback: opacity: query !== deferredQuery ? 0.7 : 1 to indicate staleness. (5) For API-backed search: use use() with Suspense for data fetching, start fetch in transition. (6) Prefetch on hover/focus for common queries. (7) Virtualize the results list (react-window) for large result sets. This gives: instant input response, progressive result updates, no loading spinners for fast queries, graceful degradation for slow queries.

Suspense & Batching

Q31. How does Suspense catch thrown promises internally? L6

When a component throws during render, React's work loop catches it with try/catch. It checks if the thrown value is a "thenable" (has a .then method). If yes: it's a Suspense suspension, not an error. React walks up the fiber tree via return pointers to find the nearest SuspenseComponent fiber. It sets the DidCapture flag on that fiber and attaches .then() callbacks to the promise that will trigger a re-render of the Suspense boundary when resolved. The Suspense fiber then renders its fallback prop instead of its children. If no Suspense boundary is found, it's treated as an error.

Q32. What is the difference between Suspense for code splitting and data fetching? Senior

Both use the same mechanism (thrown promises caught by Suspense boundaries), but differ in caching: Code splitting (React.lazy): the module is cached after first load — subsequent renders resolve immediately. The promise represents a JavaScript chunk download. Data fetching (use() hook): requires explicit cache management — data may need refetching on navigation, invalidation, etc. The promise represents an API call. Code splitting has been stable since React 16.6. Data fetching with Suspense is stable in React 19 (use() hook). Frameworks like Next.js and Remix have supported Suspense for data fetching earlier via their own integrations.

Q33. How does streaming SSR with renderToPipeableStream work? L6

(1) Server starts rendering the component tree. (2) When it hits a Suspense boundary with pending data, it sends the fallback HTML immediately and continues rendering other parts. (3) The "shell" (everything outside Suspense boundaries) is sent first — this is the onShellReady callback. (4) As each Suspense boundary's data resolves, the server streams a replacement HTML chunk wrapped in an inline <script> that swaps the fallback with the real content. (5) The client receives content progressively — each section appears as its data arrives. (6) hydrateRoot on the client hydrates each section independently as it arrives.

Q34. How does React prevent the waterfall problem with Suspense? Senior

Waterfalls happen when data fetches are sequential (fetch A, wait, fetch B, wait). Prevention strategies: (1) Start all fetches at the parent level (parallel promises), pass them down as props. (2) Use separate Suspense boundaries for independent data — each shows its own fallback and resolves independently. (3) Prefetch data at the route level (loaders) before components render. (4) Use React's cache() to deduplicate requests — multiple components can call the same cached fetch without triggering multiple network requests. The key principle: start fetches as early as possible, don't wait for parent data before fetching child data.

Q35. What changed with batching in React 18? Senior

React 18 introduced automatic batching: all state updates are batched regardless of context. Previously (React 17), only updates inside React event handlers were batched. Updates in setTimeout, Promise callbacks, native event listeners, and other async contexts triggered separate renders for each setState. React 18 uses microtask scheduling — setState enqueues the update and schedules a microtask. All setState calls in the same synchronous block are processed together when the microtask fires. This requires createRoot (not legacy ReactDOM.render). Use flushSync to opt out when needed.

Q36. When and why would you use flushSync? Senior

flushSync forces synchronous rendering — the DOM is updated before the next line executes. Use cases: (1) Reading DOM measurements after a state update (flushSync(() => setItems(...)) then scrollHeight). (2) Third-party library integration that needs the DOM to be updated synchronously. (3) Scroll-to-bottom after adding a message. (4) Accessibility: screen reader announcements that need immediate DOM presence. Avoid for performance — it bypasses batching and concurrent features, blocking the main thread until render + commit completes. In most cases, useLayoutEffect is a better alternative.

Q37. How does React process multiple setState calls on the same state? Senior

React queues all updates and processes them sequentially during render. For value updates (setState(5)), the last value wins. For updater functions (setState(prev => prev + 1)), each function is applied to the result of the previous one. Mixed: value updates set the base, updaters apply on top. Example: setState(5), setState(c => c + 1) → result is 6. The queue is processed in processUpdateQueue during beginWork. This is why updater functions are preferred when the new state depends on the previous state — they always see the latest accumulated value.

Q38. Why can't you read the new state immediately after setState? Senior

State variables in function components are closures captured at render time. setState enqueues an update but doesn't change the current closure's value — that value is fixed for this render. The new value is only available in the next render. This is by design: it ensures consistency within a single render (all code sees the same state snapshot) and enables batching (multiple setState calls are processed together). If you need the new value immediately, compute it in a variable: const next = count + 1; setCount(next); doSomething(next);.

Q39. What is React's cache() function and how does it relate to Suspense? L6

cache(fn) memoizes an async function so that multiple calls with the same arguments return the same Promise. This prevents duplicate network requests when multiple components need the same data. The cache is scoped to the current server request (in RSC) or the current render tree (on client). Combined with use() and Suspense: const data = use(cachedFetch(id)) — the first component to call this suspends, subsequent components with the same id get the same Promise (already pending or resolved). This enables a "render-as-you-fetch" pattern where data dependencies are declared at the component level without waterfall or duplication.

Q40. How do transitions interact with Suspense boundaries? L6

When a transition triggers a Suspense boundary (new data fetch), React keeps showing the old UI instead of the fallback. Without transition: navigating to a new page immediately shows the Suspense fallback (spinner). With transition: the old page stays visible (with isPending indicator) until the new page's data resolves. This prevents the "flash of loading state" during navigation. Implementation: transition renders use TransitionLane, and Suspense boundaries in transition renders don't immediately show fallbacks — they wait for the transition to complete. If the transition takes too long, React eventually shows the fallback (configurable timeout).

React DOM & Events

Q41. How does React's event delegation work in React 17+? Senior

React attaches a single event listener per event type to the root container (not document, as in React 16). When a native event fires, it bubbles to the root. React's handler: (1) Finds the fiber for event.target, (2) Walks up the fiber tree collecting handlers (onClick, etc.), (3) Creates a SyntheticEvent wrapping the native event, (4) Executes handlers in order — capture phase (top→bottom), then bubble phase (bottom→top). Benefits: memory efficiency (one listener vs thousands), automatic handling of dynamic elements, consistent behavior across browsers, integration with React's priority system.

Q42. Why did React 17 move event listeners from document to root? Senior

To support gradual React upgrades and multiple React roots on the same page. With document-level listeners (React 16), two React roots would interfere — events from one root would be processed by both. With root-level listeners, each React root handles only its own events. This also fixes issues with event.stopPropagation() in non-React code — stopping propagation before the root container prevents React from seeing the event, which is the expected behavior. It also enables better integration with non-React parts of the page (jQuery, vanilla JS).

Q43. How do events from React portals bubble? L6

Events from portals bubble through the React tree (fiber return pointers), not the DOM tree. A portal renders its children in a different DOM container, but the fiber tree maintains the original parent-child relationship. So a click inside a portal bubbles to the portal's React parent, even though the DOM parent is different. This means an onClick on a parent component catches clicks from its portal children. Native DOM events still bubble through the DOM tree normally — it's React's synthetic event system that follows the fiber tree.

Q44. What is the difference between controlled and uncontrolled components internally? Senior

Controlled: React sets element.value (DOM property) on every render. User input triggers onChange → setState → re-render → DOM update. React is the source of truth. If you set value without onChange, the input is read-only — user typing is immediately overwritten. Uncontrolled: React sets defaultValue only on mount (via the defaultValue HTML attribute). The DOM manages the value after that. Read the value via ref when needed. Internally, React tracks whether an input is controlled (has a value prop) and warns if you switch between controlled and uncontrolled.

Q45. How does React handle the style prop differently from HTML? Senior

In HTML, style is a string: style="color: red; font-size: 14px". In React, style is an object: style={{ color: 'red', fontSize: 14 }}. React sets individual style properties via element.style.color = 'red', element.style.fontSize = '14px'. Numeric values are automatically appended with 'px' for properties that accept pixel values (except unitless properties like opacity, zIndex, flex). camelCase property names map to CSS kebab-case. React diffs style objects property by property — only changed properties are updated in the DOM.

Q46. How does React assign event priorities to different event types? L6

React categorizes events into three priority tiers: Discrete (click, keydown, input, change, focus, blur, submit) → SyncLane. These are "one-shot" events where the user expects immediate response. Continuous (mousemove, scroll, drag, touchmove, wheel, pointermove) → InputContinuousLane. These fire rapidly and can be batched more aggressively. Default (load, error, animation events) → DefaultLane. The priority is assigned when React's event handler dispatches the event — it sets the current update priority context, and any setState calls within the handler inherit that priority.

Q47. What changed with SyntheticEvent pooling in React 17? Senior

React 16 reused SyntheticEvent objects across different events (pooling). After the event handler returned, all properties were nullified. Accessing event.target in a setTimeout would return null. You had to call event.persist() to keep the event. React 17 removed pooling entirely — SyntheticEvents are normal objects that aren't reused. event.persist() is a no-op. This was possible because modern browsers create event objects efficiently, and the pooling optimization caused more developer confusion than performance benefit.

Q48. What is the host config interface and how does it make React renderer-agnostic? L6

The host config is an interface (~40 methods) that a renderer must implement: createInstance, appendChild, removeChild, commitUpdate, prepareUpdate, etc. The reconciler (react-reconciler) calls these methods without knowing what they do — it just knows "create a node," "append a child," "update properties." react-dom implements these for browser DOM, react-native for native views, react-three-fiber for Three.js objects. This separation means the reconciler's complex scheduling, diffing, and effect management code is shared across all platforms. Creating a custom renderer means implementing the host config interface.

Hydration

Q49. What is hydration and why is it necessary for SSR? Senior

Hydration attaches React's event handlers, state, and interactivity to server-rendered HTML. The server sends static HTML (fast first paint, good SEO), but it's not interactive — no event handlers, no state, no effects. Hydration walks the existing DOM, matches it to the fiber tree, attaches event listeners (via delegation on root), sets up state and hooks, and runs effects. Without hydration, you'd have to destroy the server HTML and re-render from scratch (flash of empty content). Hydration reuses existing DOM nodes, avoiding the expensive recreation.

Q50. What causes hydration mismatches and how does React handle them? Senior

Common causes: (1) Date/time rendering (different on server vs client), (2) Browser-only APIs (window.innerWidth), (3) Random values (Math.random()), (4) Invalid HTML nesting (browser auto-corrects), (5) Browser extensions modifying DOM. React handles mismatches by: logging a warning in development, attempting to patch the content client-side (update text, recreate elements), and calling onRecoverableError. For text mismatches, React updates the text node. For structural mismatches, React may destroy and recreate the subtree. Use suppressHydrationWarning for known-unavoidable mismatches, and useId() for stable IDs.

Q51. Explain selective hydration and its requirements. L6

Selective hydration allows React to hydrate different parts of the page independently and in priority order. Requirements: (1) renderToPipeableStream for streaming SSR, (2) Suspense boundaries around independently-hydratable sections, (3) hydrateRoot (not legacy hydrate). How it works: React hydrates the shell first, then each Suspense boundary as its HTML arrives. If the user interacts with an un-hydrated section, React prioritizes that section — interrupts other hydration, hydrates it synchronously, replays the event. This means the most important sections (user-interacted) hydrate first, regardless of HTML arrival order.

Q52. How does streaming SSR differ from renderToString? Senior

renderToString: synchronous, blocks until the entire tree is rendered, returns a complete HTML string. Can't use Suspense for data fetching (only code splitting). TTFB equals total render time. renderToPipeableStream: asynchronous, streams HTML as it's ready. Sends the shell immediately (fast TTFB), then streams Suspense boundary content as data resolves. Supports Suspense for data fetching, selective hydration, and progressive rendering. The client sees content progressively instead of waiting for everything. For large pages with multiple data dependencies, streaming can reduce TTFB from seconds to milliseconds.

Q53. What is the islands architecture and how does it relate to hydration? L6

Islands architecture (Astro, Fresh) renders most of the page as static HTML with zero JavaScript. Only interactive "islands" (components that need state/events) are hydrated. The rest of the page is never hydrated — no JS loaded, no event listeners, no React overhead. This dramatically reduces the hydration cost: instead of hydrating the entire page (header, footer, static content), you only hydrate a few interactive widgets. React Server Components (RSC) achieve a similar effect within React: server components send zero JS to the client, only client components are hydrated. Both approaches minimize the gap between FCP and TTI.

Q54. How does useId() prevent hydration mismatches? Senior

useId() generates deterministic IDs based on the component's position in the fiber tree. The algorithm uses the fiber's tree path (parent indices) to create a unique, stable ID that's the same on server and client. Unlike Math.random() or incrementing counters (which differ between server and client execution), the fiber tree structure is identical on both sides (assuming no mismatches). The identifierPrefix option in createRoot/hydrateRoot adds a prefix to avoid collisions between multiple React roots on the same page.

Q55. Design a hydration strategy for a large e-commerce product page. L6

Architecture: (1) Streaming SSR with renderToPipeableStream — send header/navigation shell immediately. (2) Critical section (product image, title, price, Add to Cart): outside Suspense, hydrates first, interactive within 100ms of FCP. (3) Product details (description, specs): Suspense boundary, streams when data ready, hydrates on scroll/visibility. (4) Reviews section: Suspense boundary, lazy hydration (hydrate on scroll into view or click). (5) Recommendations: Suspense boundary, lowest priority, hydrate during idle. (6) Footer: static HTML, never hydrated (React Server Component or static). (7) Monitor: track hydration mismatches via onRecoverableError, measure FCP→TTI gap per section, target: Add to Cart interactive within 200ms of FCP on 3G.
