Render Phases

1. The Two-Phase Model

Every React update goes through exactly two phases. Understanding the boundary between them is critical for writing correct, performant React code.

setState() called │ ▼ ┌──────────────────────────────────────────────────┐ │ RENDER PHASE │ │ │ │ • Pure computation (no side effects) │ │ • Interruptible (can pause and resume) │ │ • May be called multiple times (StrictMode) │ │ • Builds workInProgress fiber tree │ │ • Calls component functions / render methods │ │ • Diffs old vs new elements │ │ • Collects effect flags on fibers │ │ │ │ Functions: beginWork() → completeWork() │ └──────────────────────┬───────────────────────────┘ │ ▼ ┌──────────────────────────────────────────────────┐ │ COMMIT PHASE │ │ │ │ • Synchronous (cannot be interrupted) │ │ • Runs exactly once per update │ │ • Touches the real DOM │ │ • Runs layout effects (useLayoutEffect) │ │ • Runs passive effects (useEffect) │ │ • Swaps current ↔ workInProgress trees │ │ │ │ Functions: commitWork() (3 sub-phases) │ └──────────────────────────────────────────────────┘
Architect Insight

The render/commit split is React's fundamental architectural decision. By making the render phase pure and interruptible, React can: (1) pause rendering to handle user input, (2) render multiple versions of the UI simultaneously (concurrent features), (3) throw away incomplete renders without side effects, (4) call render functions multiple times safely (StrictMode). The commit phase is the "point of no return" — once it starts, it must finish synchronously to avoid showing partial DOM updates.

2. Render Phase Deep Dive

The render phase walks the fiber tree, calling component functions and diffing their output against the previous render. It produces a list of effects (DOM mutations needed) but does not apply them.

// The render phase work loop (simplified)
function renderRootConcurrent(root, lanes) {
  // Prepare the workInProgress tree
  prepareFreshStack(root, lanes);

  // Process fibers one at a time
  while (workInProgress !== null && !shouldYield()) {
    performUnitOfWork(workInProgress);
  }

  if (workInProgress !== null) {
    // Incomplete — schedule continuation
    return RootInProgress;
  }

  // Complete — ready for commit
  return RootCompleted;
}

function performUnitOfWork(unitOfWork) {
  const current = unitOfWork.alternate;

  // Phase 1: Process this fiber, get first child
  const next = beginWork(current, unitOfWork, renderLanes);
  unitOfWork.memoizedProps = unitOfWork.pendingProps;

  if (next === null) {
    // No children — complete this fiber and siblings
    completeUnitOfWork(unitOfWork);
  } else {
    workInProgress = next; // Descend into child
  }
}

3. beginWork — Descending Into Children

beginWork is called for each fiber during the downward traversal. Its job: process the fiber based on its tag and return the first child fiber (or null if leaf).

// Simplified beginWork (from ReactFiberBeginWork.js)
function beginWork(current, workInProgress, renderLanes) {
  // Bailout check: can we skip this fiber?
  if (current !== null) {
    const oldProps = current.memoizedProps;
    const newProps = workInProgress.pendingProps;

    if (oldProps === newProps && !hasScheduledUpdate(current, renderLanes)) {
      // No changes — bail out of entire subtree
      return bailoutOnAlreadyFinishedWork(current, workInProgress);
    }
  }

  // Process based on fiber tag
  switch (workInProgress.tag) {
    case FunctionComponent:
      return updateFunctionComponent(current, workInProgress, renderLanes);
    case ClassComponent:
      return updateClassComponent(current, workInProgress, renderLanes);
    case HostComponent:
      return updateHostComponent(current, workInProgress, renderLanes);
    case HostText:
      return null; // Text nodes have no children
    case SuspenseComponent:
      return updateSuspenseComponent(current, workInProgress, renderLanes);
    case ContextProvider:
      return updateContextProvider(current, workInProgress, renderLanes);
    // ... more cases for each fiber tag
  }
}

3.1 beginWork for Function Components

// What happens when beginWork processes a FunctionComponent:
function updateFunctionComponent(current, workInProgress, renderLanes) {
  const Component = workInProgress.type;
  const props = workInProgress.pendingProps;

  // 1. Set up hooks dispatcher (mount vs update)
  prepareHooks(current, workInProgress);

  // 2. Call the component function
  let children = Component(props);
  // → This is where YOUR component code runs
  // → useState, useEffect, useMemo all execute here
  // → Returns ReactElement tree

  // 3. Finalize hooks
  finishHooks();

  // 4. Reconcile children (diff old vs new)
  reconcileChildren(current, workInProgress, children, renderLanes);

  // 5. Return first child fiber
  return workInProgress.child;
}

4. completeWork — Bubbling Up

completeWork is called for each fiber during the upward traversal (after all children are processed). Its job: create DOM nodes for host components and bubble up effect flags.

// Simplified completeWork
function completeWork(current, workInProgress) {
  switch (workInProgress.tag) {
    case HostComponent: {
      const type = workInProgress.type; // "div", "span", etc.

      if (current !== null && workInProgress.stateNode !== null) {
        // UPDATE: DOM node exists, diff props
        const updatePayload = diffProperties(
          current.memoizedProps,
          workInProgress.pendingProps
        );
        if (updatePayload) {
          workInProgress.flags |= Update;
          workInProgress.updateQueue = updatePayload;
        }
      } else {
        // MOUNT: Create new DOM node
        const domElement = createElement(type);
        appendAllChildren(domElement, workInProgress);
        workInProgress.stateNode = domElement;
        setInitialProperties(domElement, type, workInProgress.pendingProps);
      }
      // Bubble up flags from children
      bubbleProperties(workInProgress);
      break;
    }
    case FunctionComponent:
    case ClassComponent:
      bubbleProperties(workInProgress);
      break;
  }
}
bubbleProperties — Effect Flag Aggregation

bubbleProperties collects flags and subtreeFlags from all children into the parent's subtreeFlags. This creates a bitmask that tells the commit phase whether any descendant needs work. If subtreeFlags === NoFlags, the commit phase can skip the entire subtree — a critical optimization for large trees where most nodes don't change.

5. Commit Phase Deep Dive

The commit phase takes the completed workInProgress tree and applies all collected effects to the real DOM. It runs synchronously — once started, it cannot be interrupted.

// Simplified commitRoot
function commitRoot(root) {
  const finishedWork = root.finishedWork;

  // Sub-phase 1: Before Mutation
  commitBeforeMutationEffects(finishedWork);

  // Sub-phase 2: Mutation (DOM changes happen here)
  commitMutationEffects(finishedWork);

  // SWAP: workInProgress becomes current
  root.current = finishedWork;

  // Sub-phase 3: Layout (synchronous effects)
  commitLayoutEffects(finishedWork);

  // Schedule passive effects (useEffect) as microtask
  schedulePassiveEffects();
}

7. Why Render Must Be Pure (StrictMode)

// StrictMode double-invokes render functions to detect impurity
<React.StrictMode>
  <App />
</React.StrictMode>

// In development, React calls each component function TWICE:
// 1. First call: normal render
// 2. Second call: detect side effects (result should be identical)

// If your render has side effects, the double call exposes them:

// BAD: Side effect in render
let count = 0;
function Counter() {
  count++; // Side effect! Double-invoked = count incremented twice
  return <div>{count}</div>;
}

// GOOD: Pure render
function Counter({ initialCount }) {
  const [count, setCount] = useState(initialCount);
  return <div>{count}</div>; // Same output for same input
}
Why Purity Matters for Concurrent Rendering

In concurrent mode, React may start rendering a component, pause to handle user input, and then either resume or discard the partial render. If the render had side effects (network requests, DOM mutations, global state changes), discarding it would leave the app in an inconsistent state. Purity guarantees that discarding a render is always safe — no observable effects occurred.

9. Architect Considerations

9.1 Effect Timing Cheat Sheet

// When does each effect type run?

// useEffect: AFTER paint (non-blocking)
useEffect(() => {
  // DOM is updated AND painted
  // Safe for: data fetching, subscriptions, analytics
  // NOT for: DOM measurements that affect layout
}, [deps]);

// useLayoutEffect: BEFORE paint (blocking)
useLayoutEffect(() => {
  // DOM is updated but NOT painted
  // Safe for: DOM measurements, scroll position, tooltips
  // WARNING: blocks painting — keep it fast
}, [deps]);

// useInsertionEffect: BEFORE DOM mutations (React 18+)
useInsertionEffect(() => {
  // Runs before any DOM changes
  // For CSS-in-JS libraries to inject styles
  // before layout effects read them
}, [deps]);

9.2 Common Render Phase Violations

// These are side effects that break render purity:

// ❌ Mutating external variables
let renderCount = 0;
function App() { renderCount++; /* ... */ }

// ❌ Calling APIs during render
function App() { fetch('/api'); /* ... */ }

// ❌ Subscribing during render
function App() { window.addEventListener('resize', fn); /* ... */ }

// ❌ Writing to DOM during render
function App() { document.title = 'New'; /* ... */ }

// ✅ All of these belong in useEffect or event handlers
Staff-Level Insight

The render/commit boundary is the single most important concept for understanding React's concurrent features. Everything in the render phase can be thrown away (transitions, Suspense retries, interrupted renders). Everything in the commit phase is permanent (DOM mutations, effect callbacks). When designing components, ask: "If React called my render function twice and threw away the first result, would anything break?" If yes, you have a purity violation that will cause bugs in concurrent mode.
