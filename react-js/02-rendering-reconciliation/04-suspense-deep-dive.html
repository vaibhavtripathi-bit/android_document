1. Suspense Boundary Mechanism

A Suspense boundary is a component that catches "suspensions" (thrown promises) from its descendants and shows a fallback UI until the promise resolves.

<Suspense fallback={<Spinner />}> <UserProfile /> ← throws Promise (data loading) <UserPosts /> ← also throws Promise </Suspense> Timeline: ───────── t=0: Render attempt UserProfile throws Promise_A UserPosts throws Promise_B Suspense catches → shows <Spinner /> t=1: Promise_A resolves React re-renders Suspense boundary UserProfile renders successfully UserPosts still throws Promise_B Suspense still shows <Spinner /> t=2: Promise_B resolves React re-renders Suspense boundary Both render successfully Suspense shows <UserProfile /> + <UserPosts /> 
// Suspense fiber has special handling in beginWork:
function updateSuspenseComponent(current, workInProgress, renderLanes) {
  const nextProps = workInProgress.pendingProps;

  let showFallback = false;
  const didSuspend = (workInProgress.flags & DidCapture) !== NoFlags;

  if (didSuspend) {
    showFallback = true;
    workInProgress.flags &= ~DidCapture; // Clear flag
  }

  if (showFallback) {
    // Render fallback as primary children
    const fallbackChildren = nextProps.fallback;
    return mountSuspenseFallbackChildren(
      workInProgress, fallbackChildren, renderLanes
    );
  } else {
    // Render primary children
    const primaryChildren = nextProps.children;
    return mountSuspensePrimaryChildren(
      workInProgress, primaryChildren, renderLanes
    );
  }
}

2. Thrown Promises: How React Catches Them

// When a component throws a promise during render:

// 1. React's work loop catches the thrown value
try {
  beginWork(current, workInProgress, renderLanes);
} catch (thrownValue) {
  if (isThenable(thrownValue)) {
    // It's a promise — this is a Suspense suspension
    handleSuspension(workInProgress, thrownValue);
  } else {
    // It's a real error — propagate to error boundary
    handleError(workInProgress, thrownValue);
  }
}

// 2. React walks up the fiber tree to find nearest Suspense
function handleSuspension(fiber, thenable) {
  let node = fiber.return;
  while (node !== null) {
    if (node.tag === SuspenseComponent) {
      // Found Suspense boundary
      node.flags |= DidCapture;
      // Attach listener to re-render when promise resolves
      thenable.then(
        () => retryTimedOutBoundary(node),
        () => retryTimedOutBoundary(node)
      );
      return;
    }
    node = node.return;
  }
  // No Suspense boundary found — this is an error
}
Thrown Promises Are Not Errors

React uses JavaScript's throw mechanism for Suspense, but thrown promises are not errors. They're a control flow mechanism — a way for a component to say "I'm not ready yet, here's a promise that resolves when I will be." This is why error boundaries don't catch Suspense suspensions, and Suspense boundaries don't catch errors. They're separate propagation paths through the fiber tree.

3. use() Hook for Data Fetching

import { use, Suspense } from 'react';

// use() reads the value of a Promise (suspending if pending)
function UserProfile({ userPromise }) {
  const user = use(userPromise);
  return <h1>{user.name}</h1>;
}

// Parent creates the promise OUTSIDE of render
function App() {
  const [userId, setUserId] = useState(1);
  const userPromise = useMemo(
    () => fetchUser(userId),
    [userId]
  );

  return (
    <Suspense fallback={<Spinner />}>
      <UserProfile userPromise={userPromise} />
    </Suspense>
  );
}

// use() behavior:
// - Promise pending → throws promise (Suspense shows fallback)
// - Promise resolved → returns the value
// - Promise rejected → throws the error (error boundary catches)

// Unlike useEffect-based fetching:
// - No loading state management (Suspense handles it)
// - No race conditions (React manages promise lifecycle)
// - Works with streaming SSR

4. Streaming SSR with renderToPipeableStream

// Server-side: stream HTML with Suspense boundaries
import { renderToPipeableStream } from 'react-dom/server';

const { pipe } = renderToPipeableStream(
  <App />,
  {
    onShellReady() {
      // Shell (everything outside Suspense) is ready
      response.statusCode = 200;
      response.setHeader('Content-Type', 'text/html');
      pipe(response); // Start streaming
    },
    onShellError(error) {
      response.statusCode = 500;
      response.send('<h1>Error</h1>');
    },
    onAllReady() {
      // All Suspense boundaries resolved (for crawlers)
    }
  }
);

6. Waterfall Problem and Solutions

// PROBLEM: Sequential data fetching (waterfall)
function UserPage({ userId }) {
  const user = use(fetchUser(userId));     // Fetch 1: 200ms
  const posts = use(fetchPosts(user.id));  // Fetch 2: 300ms (waits for 1)
  // Total: 500ms (sequential)
}

// SOLUTION 1: Parallel fetching at parent level
function UserPage({ userId }) {
  const userPromise = useMemo(() => fetchUser(userId), [userId]);
  const postsPromise = useMemo(() => fetchPosts(userId), [userId]);

  return (
    <>
      <Suspense fallback={<UserSkeleton />}>
        <UserInfo userPromise={userPromise} />
      </Suspense>
      <Suspense fallback={<PostsSkeleton />}>
        <UserPosts postsPromise={postsPromise} />
      </Suspense>
    </>
  );
  // Both fetch in parallel! Total: max(200, 300) = 300ms
}

// SOLUTION 2: Preload / prefetch at route level
function loader({ params }) {
  // Start fetching before component renders
  return {
    user: fetchUser(params.userId),
    posts: fetchPosts(params.userId)
  };
}
Waterfall Prevention Architecture

At L6, design data fetching to avoid waterfalls: (1) Start all fetches as early as possible (route loaders, not component effects). (2) Use parallel Suspense boundaries so independent data shows independently. (3) Prefetch data on hover/focus before navigation. (4) Use a data layer (React Query, Relay) that manages caching and deduplication. The goal: every network request starts at the earliest possible moment, and UI sections appear as their data arrives.

8. Cache Integration

// React's cache() function (React 19) memoizes async functions
import { cache } from 'react';

const fetchUser = cache(async (id) => {
  const res = await fetch(`/api/users/${id}`);
  return res.json();
});

// Multiple components can call fetchUser(1)
// Only ONE network request is made per render pass
// The cache is scoped to the current server request (RSC)
// or the current render tree (client)

function UserName({ id }) {
  const user = use(fetchUser(id)); // Deduped!
  return <span>{user.name}</span>;
}

function UserAvatar({ id }) {
  const user = use(fetchUser(id)); // Same cache entry!
  return <img src={user.avatar} />;
}

9. Architect Considerations

// Suspense boundary placement strategy:

// 1. Route-level: each page has its own Suspense
//    → Shows page skeleton during navigation

// 2. Section-level: independent content sections
//    → Header loads instantly, content streams in

// 3. Component-level: individual data-dependent components
//    → Most granular, but many loading states

// Balance: too few boundaries = large loading states
//          too many boundaries = flickering skeleton soup

// Rule of thumb: one boundary per independent data dependency
Staff-Level Suspense Architecture

Suspense is not just a loading state mechanism — it's a rendering coordination primitive. It unifies code splitting, data fetching, streaming SSR, and selective hydration under one model. At L6, design your Suspense boundaries to match your data dependencies: each boundary should correspond to an independently-fetchable unit of data. Use nested boundaries for progressive disclosure (show what you have, load the rest). Combine with transitions to prevent existing content from being replaced by fallbacks during navigation.
