Internal Implementation

Both useMemo and useCallback store their result and dependencies on the hook's memoizedState as a tuple. The only difference: useMemo stores the computed value; useCallback stores the function itself.

// Mount implementations:
function mountMemo(nextCreate, deps) {
  const hook = mountWorkInProgressHook();
  const nextDeps = deps === undefined ? null : deps;
  const nextValue = nextCreate(); // compute immediately
  hook.memoizedState = [nextValue, nextDeps];
  return nextValue;
}

function mountCallback(callback, deps) {
  const hook = mountWorkInProgressHook();
  const nextDeps = deps === undefined ? null : deps;
  hook.memoizedState = [callback, nextDeps]; // store function, not result
  return callback;
}

// Update implementations:
function updateMemo(nextCreate, deps) {
  const hook = updateWorkInProgressHook();
  const nextDeps = deps === undefined ? null : deps;
  const prevState = hook.memoizedState;

  if (nextDeps !== null) {
    const prevDeps = prevState[1];
    if (areHookInputsEqual(nextDeps, prevDeps)) {
      return prevState[0]; // deps unchanged — return cached value
    }
  }

  const nextValue = nextCreate(); // deps changed — recompute
  hook.memoizedState = [nextValue, nextDeps];
  return nextValue;
}

function updateCallback(callback, deps) {
  const hook = updateWorkInProgressHook();
  const nextDeps = deps === undefined ? null : deps;
  const prevState = hook.memoizedState;

  if (nextDeps !== null) {
    const prevDeps = prevState[1];
    if (areHookInputsEqual(nextDeps, prevDeps)) {
      return prevState[0]; // deps unchanged — return cached function
    }
  }

  hook.memoizedState = [callback, nextDeps];
  return callback;
}
Architect Insight

useCallback(fn, deps) is exactly equivalent to useMemo(() => fn, deps). The separate API exists purely for readability and intent signaling. Internally, both use the same [value, deps] tuple on hook.memoizedState.

Hook State Shape

useMemo(() => expensiveCompute(a, b), [a, b]): hook.memoizedState = [ computedResult, // the return value of the factory [a, b] // frozen dependency snapshot ] useCallback((x) => doSomething(x, a), [a]): hook.memoizedState = [ (x) => doSomething(x, a), // the callback itself [a] // frozen dependency snapshot ]

Dependency Comparison Mechanism

function areHookInputsEqual(nextDeps, prevDeps) {
  // Same function used by useEffect, useMemo, useCallback
  if (prevDeps === null) return false;

  for (let i = 0; i < prevDeps.length && i < nextDeps.length; i++) {
    if (Object.is(nextDeps[i], prevDeps[i])) continue;
    return false;
  }
  return true;
}

// Implications:
// 1. Primitive deps (number, string, boolean) work well
// 2. Object/array deps compare by reference — new object = cache miss
// 3. No length check — if deps array length changes, it still
//    compares up to the shorter length (ESLint plugin catches this)

When Memoization Helps

Case 1: Expensive Computation

function SearchResults({ query, items }) {
  // WITHOUT useMemo: filters 10,000 items on every render
  // even if only an unrelated prop changed
  const filtered = items.filter(item =>
    item.name.toLowerCase().includes(query.toLowerCase())
  );

  // WITH useMemo: only recomputes when query or items change
  const filtered = useMemo(() =>
    items.filter(item =>
      item.name.toLowerCase().includes(query.toLowerCase())
    ),
    [query, items]
  );

  return filtered.map(item => <Item key={item.id} data={item} />);
}

Case 2: Referential Equality for Child Props

// React.memo + useCallback pattern:
const ExpensiveChild = React.memo(({ onClick, config }) => {
  // Only re-renders if onClick or config reference changes
  return <div onClick={onClick}>{/* expensive render */}</div>;
});

function Parent({ userId }) {
  const [count, setCount] = useState(0);

  // WITHOUT useCallback: new function every render → child re-renders
  const handleClick = () => fetchUser(userId);

  // WITH useCallback: stable reference when userId unchanged
  const handleClick = useCallback(() => fetchUser(userId), [userId]);

  // WITHOUT useMemo: new object every render → child re-renders
  const config = { theme: 'dark', userId };

  // WITH useMemo: stable reference when userId unchanged
  const config = useMemo(() => ({ theme: 'dark', userId }), [userId]);

  return (
    <>
      <button onClick={() => setCount(c => c + 1)}>{count}</button>
      <ExpensiveChild onClick={handleClick} config={config} />
    </>
  );
}

When Memoization Hurts

ScenarioWhy It Hurts
Trivial computationOverhead of comparing deps + storing tuple exceeds saved work
Deps change every renderCache always misses — you pay comparison cost AND computation cost
No downstream consumersStable reference doesn't matter if nothing uses React.memo
Primitive return valuesPrimitives are compared by value anyway — memoization adds nothing
Memory pressureEach memoized value is retained until deps change, increasing memory
 
// UNNECESSARY memoization examples:

// 1. Trivial computation
const doubled = useMemo(() => count * 2, [count]);
// Just write: const doubled = count * 2;

// 2. Deps change every render
const result = useMemo(() => transform(data), [data]);
// If data is a new object every render, this never caches

// 3. No React.memo consumer
function Parent() {
  const handler = useCallback(() => doThing(), []);
  // Pointless if Child is NOT wrapped in React.memo
  return <Child onClick={handler} />;
}

// 4. Inline objects in JSX (common mistake)
<Child style={useMemo(() => ({ color: 'red' }), [])} />
// Only useful if Child is React.memo'd
The Memoization Tax

Every useMemo/useCallback call: (1) allocates a hook node on mount, (2) stores a [value, deps] tuple, (3) runs Object.is comparison per dependency on every render. For cheap computations, this overhead exceeds the cost of just recomputing. Profile before memoizing.

React.memo: The Other Half

// React.memo wraps a component to skip re-renders when props are equal
const MemoizedComponent = React.memo(function MyComponent(props) {
  return <div>{props.name}</div>;
});

// Default comparison: shallow equality (Object.is per prop)
// Custom comparison:
const MemoizedComponent = React.memo(MyComponent, (prevProps, nextProps) => {
  // Return true to SKIP re-render (props are "equal")
  // Return false to RE-RENDER
  return prevProps.id === nextProps.id;
});

// Internal implementation (simplified):
function memo(type, compare) {
  const elementType = {
    $$typeof: REACT_MEMO_TYPE,
    type,                    // the wrapped component
    compare: compare || null, // custom comparator
  };
  return elementType;
}

// During reconciliation, React checks:
// 1. If compare exists: compare(prevProps, nextProps)
// 2. Otherwise: shallowEqual(prevProps, nextProps)
// If equal AND no state/context changes: bail out of rendering

The Full Pattern: React.memo + useCallback + useMemo

Parent re-renders (state change) │ ├─ useCallback(fn, [dep]) → stable function ref (if dep unchanged) ├─ useMemo(() => obj, [dep]) → stable object ref (if dep unchanged) │ ├─ <MemoizedChild onClick={stableFn} data={stableObj} /> │ │ │ └─ React.memo comparison: │ Object.is(prevProps.onClick, nextProps.onClick) → true ✓ │ Object.is(prevProps.data, nextProps.data) → true ✓ │ → SKIP re-render of MemoizedChild │ └─ Without useCallback/useMemo: <MemoizedChild onClick={newFn} data={newObj} /> │ └─ React.memo comparison: Object.is(prevFn, newFn) → false ✗ → RE-RENDER MemoizedChild (memo defeated)

useMemo for Derived State

// Anti-pattern: storing derived state
function TodoList({ todos, filter }) {
  const [filteredTodos, setFilteredTodos] = useState([]);

  useEffect(() => {
    setFilteredTodos(todos.filter(t => t.status === filter));
  }, [todos, filter]);
  // Problem: extra render cycle, effect runs after paint

  // Correct: derive during render with useMemo
  const filteredTodos = useMemo(
    () => todos.filter(t => t.status === filter),
    [todos, filter]
  );
  // Computed synchronously during render, no extra cycle
}

// Even simpler — if the computation is cheap, skip useMemo:
function TodoList({ todos, filter }) {
  const filteredTodos = todos.filter(t => t.status === filter);
  // Fine for small lists — no memoization overhead
}

React Compiler (React Forget)

The React Compiler is a build-time tool that automatically inserts memoization, eliminating the need for manual useMemo, useCallback, and React.memo.

How It Works

// What you write:
function ProductPage({ product }) {
  const price = formatPrice(product.price);
  const handleBuy = () => addToCart(product.id);

  return (
    <div>
      <span>{price}</span>
      <BuyButton onClick={handleBuy} />
    </div>
  );
}

// What the compiler outputs (conceptually):
function ProductPage({ product }) {
  const $ = useMemoCache(4); // compiler-managed cache

  let price;
  if ($[0] !== product.price) {
    price = formatPrice(product.price);
    $[0] = product.price;
    $[1] = price;
  } else {
    price = $[1];
  }

  let handleBuy;
  if ($[2] !== product.id) {
    handleBuy = () => addToCart(product.id);
    $[2] = product.id;
    $[3] = handleBuy;
  } else {
    handleBuy = $[3];
  }

  return (
    <div>
      <span>{price}</span>
      <BuyButton onClick={handleBuy} />
    </div>
  );
}

Compiler Limitations

The compiler assumes code follows the Rules of React: components are pure, props/state are immutable, hooks follow the rules. Code that mutates props, uses non-idempotent renders, or relies on render-time side effects will produce incorrect results. The compiler includes a validation pass and will skip functions it can't safely optimize.

Cache Invalidation Behavior

// useMemo/useCallback cache is invalidated when:
// 1. Any dependency changes (Object.is comparison)
// 2. The component unmounts (hook state is destroyed)
// 3. React drops the fiber (e.g., key change forces remount)

// Cache is NOT invalidated by:
// - Sibling re-renders
// - Parent re-renders (if deps unchanged)
// - Context changes (unless context value is a dependency)

// React reserves the right to drop memoized values:
// "You may rely on useMemo as a performance optimization,
//  not as a semantic guarantee." — React docs
//
// In practice, React currently never drops cached values
// unless the component unmounts, but future versions might
// (e.g., offscreen components could have their cache evicted).
Architect Insight — Memoization Strategy

For L6 interviews: (1) Default to no memoization — React is fast enough for most cases. (2) Profile first with React DevTools Profiler. (3) Apply React.memo to expensive subtrees, then useCallback/useMemo for their props. (4) Consider the React Compiler for greenfield projects. (5) Never memoize as a premature optimization — it adds complexity and can mask real performance issues.
