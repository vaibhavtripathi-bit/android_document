Hooks as the Primitive for Logic Reuse

Before hooks, React offered three patterns for logic reuse: Higher-Order Components (HOCs), Render Props, and Mixins (class-only). Each had significant drawbacks. Custom hooks replaced all three with a simpler composition model.

PatternDrawbackHooks Advantage
HOCsWrapper hell, prop collision, static compositionNo wrapper components, no prop conflicts
Render PropsCallback nesting, false hierarchy in JSXFlat composition, no nesting
MixinsName clashes, implicit dependencies, class-onlyExplicit dependencies, works with functions
 
// HOC pattern (before hooks):
function withWindowSize(WrappedComponent) {
  return function WithWindowSize(props) {
    const [size, setSize] = useState(getWindowSize());
    useEffect(() => {
      const handler = () => setSize(getWindowSize());
      window.addEventListener('resize', handler);
      return () => window.removeEventListener('resize', handler);
    }, []);
    return <WrappedComponent {...props} windowSize={size} />;
  };
}
const EnhancedComponent = withWindowSize(MyComponent);

// Custom hook equivalent (cleaner):
function useWindowSize() {
  const [size, setSize] = useState(getWindowSize());
  useEffect(() => {
    const handler = () => setSize(getWindowSize());
    window.addEventListener('resize', handler);
    return () => window.removeEventListener('resize', handler);
  }, []);
  return size;
}

function MyComponent() {
  const size = useWindowSize(); // direct, no wrapper
  return <div>{size.width}x{size.height}</div>;
}

Extracting Custom Hooks

A custom hook is a function that starts with use and calls other hooks. It's not a special React API — it's a naming convention that enables the ESLint plugin to enforce rules of hooks.

// Extraction pattern: identify reusable stateful logic

// BEFORE: logic duplicated across components
function UserProfile({ userId }) {
  const [user, setUser] = useState(null);
  const [loading, setLoading] = useState(true);
  const [error, setError] = useState(null);

  useEffect(() => {
    setLoading(true);
    fetchUser(userId)
      .then(data => { setUser(data); setError(null); })
      .catch(err => setError(err))
      .finally(() => setLoading(false));
  }, [userId]);

  if (loading) return <Spinner />;
  if (error) return <Error message={error.message} />;
  return <div>{user.name}</div>;
}

// AFTER: extracted custom hook
function useFetch(url) {
  const [data, setData] = useState(null);
  const [loading, setLoading] = useState(true);
  const [error, setError] = useState(null);

  useEffect(() => {
    let cancelled = false;
    setLoading(true);

    fetch(url)
      .then(res => {
        if (!res.ok) throw new Error(`HTTP ${res.status}`);
        return res.json();
      })
      .then(data => {
        if (!cancelled) { setData(data); setError(null); }
      })
      .catch(err => {
        if (!cancelled) setError(err);
      })
      .finally(() => {
        if (!cancelled) setLoading(false);
      });

    return () => { cancelled = true; };
  }, [url]);

  return { data, loading, error };
}

function UserProfile({ userId }) {
  const { data: user, loading, error } = useFetch(`/api/users/${userId}`);
  if (loading) return <Spinner />;
  if (error) return <Error message={error.message} />;
  return <div>{user.name}</div>;
}

Shared State Semantics

Critical Misconception

Multiple components calling the same custom hook do NOT share state. Each component gets its own independent copy of the hook's state. Custom hooks share logic, not state.

function useCounter(initial = 0) {
  const [count, setCount] = useState(initial);
  const increment = useCallback(() => setCount(c => c + 1), []);
  return { count, increment };
}

function ComponentA() {
  const { count, increment } = useCounter(0);
  // count is independent — only for ComponentA
  return <button onClick={increment}>A: {count}</button>;
}

function ComponentB() {
  const { count, increment } = useCounter(0);
  // count is independent — only for ComponentB
  return <button onClick={increment}>B: {count}</button>;
}

// Clicking A's button does NOT affect B's count.
// Each component has its own fiber with its own hook linked list.
ComponentA fiber ComponentB fiber ├── memoizedState ├── memoizedState │ └── Hook #0 (useState) │ └── Hook #0 (useState) │ memoizedState: 3 │ memoizedState: 7 │ └── Hook #1 (useCallback) │ └── Hook #1 (useCallback) │ │ │ Independent state! │ Independent state! 
// To SHARE state, use external stores or context:

// Option 1: Lift state up
function Parent() {
  const { count, increment } = useCounter(0);
  return (
    <>
      <DisplayA count={count} />
      <DisplayB count={count} increment={increment} />
    </>
  );
}

// Option 2: External store (Zustand, etc.)
const useSharedCounter = create((set) => ({
  count: 0,
  increment: () => set((s) => ({ count: s.count + 1 })),
}));

// Option 3: Context
const CounterContext = createContext();
function CounterProvider({ children }) {
  const counter = useCounter(0);
  return (
    <CounterContext.Provider value={counter}>
      {children}
    </CounterContext.Provider>
  );
}

Naming Convention: use* Prefix

// The 'use' prefix is enforced by eslint-plugin-react-hooks:
// - Functions starting with 'use' are treated as hooks
// - They must follow rules of hooks (top-level, React functions)
// - Functions NOT starting with 'use' cannot call hooks

// Valid:
function useAuth() { /* can call hooks */ }
function useFormValidation() { /* can call hooks */ }

// Invalid — ESLint error:
function getAuth() {
  const [user, setUser] = useState(null); // ERROR: hooks in non-hook function
}

// The prefix is purely a naming convention — there's no runtime check.
// But the ESLint plugin uses it to determine which functions are hooks
// and should follow the rules.

Hook Composition Patterns

Pattern 1: Encapsulating Subscriptions

function useOnlineStatus() {
  const [isOnline, setIsOnline] = useState(navigator.onLine);

  useEffect(() => {
    const handleOnline = () => setIsOnline(true);
    const handleOffline = () => setIsOnline(false);

    window.addEventListener('online', handleOnline);
    window.addEventListener('offline', handleOffline);

    return () => {
      window.removeEventListener('online', handleOnline);
      window.removeEventListener('offline', handleOffline);
    };
  }, []);

  return isOnline;
}

Pattern 2: Encapsulating Timers

function useInterval(callback, delay) {
  const savedCallback = useRef(callback);

  useEffect(() => {
    savedCallback.current = callback;
  });

  useEffect(() => {
    if (delay === null) return; // pause when delay is null

    const id = setInterval(() => savedCallback.current(), delay);
    return () => clearInterval(id);
  }, [delay]);
}

function useDebounce(value, delay) {
  const [debouncedValue, setDebouncedValue] = useState(value);

  useEffect(() => {
    const timer = setTimeout(() => setDebouncedValue(value), delay);
    return () => clearTimeout(timer);
  }, [value, delay]);

  return debouncedValue;
}

function useThrottle(value, interval) {
  const [throttled, setThrottled] = useState(value);
  const lastUpdated = useRef(Date.now());

  useEffect(() => {
    const now = Date.now();
    if (now - lastUpdated.current >= interval) {
      lastUpdated.current = now;
      setThrottled(value);
    } else {
      const id = setTimeout(() => {
        lastUpdated.current = Date.now();
        setThrottled(value);
      }, interval - (now - lastUpdated.current));
      return () => clearTimeout(id);
    }
  }, [value, interval]);

  return throttled;
}

Pattern 3: Encapsulating Event Listeners

function useEventListener(eventName, handler, element = window) {
  const savedHandler = useRef(handler);

  useEffect(() => {
    savedHandler.current = handler;
  }, [handler]);

  useEffect(() => {
    const isSupported = element && element.addEventListener;
    if (!isSupported) return;

    const listener = (event) => savedHandler.current(event);
    element.addEventListener(eventName, listener);
    return () => element.removeEventListener(eventName, listener);
  }, [eventName, element]);
}

function useKeyPress(targetKey) {
  const [pressed, setPressed] = useState(false);

  useEventListener('keydown', (e) => {
    if (e.key === targetKey) setPressed(true);
  });

  useEventListener('keyup', (e) => {
    if (e.key === targetKey) setPressed(false);
  });

  return pressed;
}

Pattern 4: Composing Hooks

function useLocalStorage(key, initialValue) {
  const [storedValue, setStoredValue] = useState(() => {
    try {
      const item = window.localStorage.getItem(key);
      return item ? JSON.parse(item) : initialValue;
    } catch {
      return initialValue;
    }
  });

  const setValue = useCallback((value) => {
    setStoredValue(prev => {
      const valueToStore = typeof value === 'function' ? value(prev) : value;
      window.localStorage.setItem(key, JSON.stringify(valueToStore));
      return valueToStore;
    });
  }, [key]);

  return [storedValue, setValue];
}

// Composing multiple hooks:
function usePersistedTheme() {
  const [theme, setTheme] = useLocalStorage('theme', 'light');
  const prefersDark = useMediaQuery('(prefers-color-scheme: dark)');

  useEffect(() => {
    document.documentElement.setAttribute('data-theme', theme);
  }, [theme]);

  const toggleTheme = useCallback(() => {
    setTheme(t => t === 'light' ? 'dark' : 'light');
  }, [setTheme]);

  return { theme, toggleTheme, prefersDark };
}

Testing Custom Hooks

// Using @testing-library/react's renderHook:
import { renderHook, act } from '@testing-library/react';

// Test useCounter:
test('should increment counter', () => {
  const { result } = renderHook(() => useCounter(0));

  expect(result.current.count).toBe(0);

  act(() => {
    result.current.increment();
  });

  expect(result.current.count).toBe(1);
});

// Test with changing props:
test('should reset on key change', () => {
  const { result, rerender } = renderHook(
    ({ userId }) => useFetch(`/api/users/${userId}`),
    { initialProps: { userId: '1' } }
  );

  expect(result.current.loading).toBe(true);

  // Simulate prop change
  rerender({ userId: '2' });
  expect(result.current.loading).toBe(true); // re-fetching
});

// Test with wrapper (for context-dependent hooks):
test('should read theme from context', () => {
  const wrapper = ({ children }) => (
    <ThemeProvider theme="dark">{children}</ThemeProvider>
  );

  const { result } = renderHook(() => useTheme(), { wrapper });
  expect(result.current.theme).toBe('dark');
});

// Test async hooks:
test('should fetch data', async () => {
  const { result } = renderHook(() => useFetch('/api/data'));

  expect(result.current.loading).toBe(true);

  await waitFor(() => {
    expect(result.current.loading).toBe(false);
  });

  expect(result.current.data).toEqual({ name: 'test' });
});

Rules for Custom Hooks

Custom Hook Rules Summary
  1. Name must start with use — enables ESLint enforcement
  2. Must call other hooks — a function that doesn't call hooks isn't a hook (just a utility)
  3. Follow rules of hooks — top-level calls only, no conditionals/loops
  4. Return what consumers need — state, handlers, computed values
  5. Accept configuration as arguments — initial values, options, callbacks
  6. Don't over-abstract — extract when logic is reused or complex, not preemptively
// Anti-pattern: hook that doesn't call hooks (should be a utility)
function useFormatDate(date) {
  return new Intl.DateTimeFormat('en-US').format(date);
}
// This is just a function — no hooks inside. Use a plain function instead.

// Anti-pattern: over-abstraction
function useBoolean(initial = false) {
  const [value, setValue] = useState(initial);
  const setTrue = useCallback(() => setValue(true), []);
  const setFalse = useCallback(() => setValue(false), []);
  const toggle = useCallback(() => setValue(v => !v), []);
  return { value, setTrue, setFalse, toggle };
}
// Arguably too thin — useState(false) is already simple enough.
// Extract when the logic is genuinely complex or reused.
Architect Insight — Hook Extraction Heuristics

Extract a custom hook when: (1) the same stateful logic appears in 2+ components, (2) a component's hook logic exceeds ~15 lines and has a clear responsibility boundary, (3) you want to test stateful logic independently of rendering, (4) the logic involves coordinating multiple hooks (state + effect + ref). Don't extract for single useState calls or trivial derived values.
