Hooks Architecture

Q1: How does React store hook state internally? L6
Hook state is stored as a singly-linked list on the fiber node's memoizedState field. Each hook call during render creates (on mount) or walks to (on update) a node in this list. Each node has memoizedState (the hook's value), queue (for state updates), and next (pointer to the next hook). This positional design is why hooks must be called in the same order every render.
Q2: Explain the difference between HooksDispatcherOnMount and HooksDispatcherOnUpdate. L6
They are two separate objects mapping hook names to implementations. HooksDispatcherOnMount creates new hook nodes and appends them to the linked list (building it from scratch). HooksDispatcherOnUpdate walks the existing linked list from the current fiber, clones each hook node to the work-in-progress fiber, and processes any pending update queues. React selects the dispatcher based on whether fiber.alternate (the current fiber) exists.
Q3: Why must hooks be called in the same order every render? What happens if they aren't? L6
Hooks are identified by position in the linked list, not by name. On update, React walks the list sequentially — the Nth hook call reads the Nth node. If a hook is conditionally skipped, subsequent hooks read the wrong node's state. This causes silent data corruption: a useEffect might read a useState's data, or vice versa. The ESLint plugin rules-of-hooks prevents this by building a control flow graph and verifying every hook is on every execution path.
Q4: What is the ContextOnlyDispatcher and when is it active? L6
After a component finishes rendering, React swaps in the ContextOnlyDispatcher where every hook function throws an error: "Invalid hook call." This prevents hooks from being called outside the render phase — in event handlers, timeouts, or regular functions. It's the runtime enforcement mechanism complementing the ESLint static analysis.
Q5: How does React know which component is currently rendering when you call useState()? L6
React maintains a module-scoped variable currentlyRenderingFiber in ReactFiberHooks.js. Before calling the component function, renderWithHooks() sets this variable to the work-in-progress fiber. When useState() calls resolveDispatcher(), it reads ReactCurrentDispatcher.current, which was set based on this fiber. The hook then operates on currentlyRenderingFiber's hook list.
Q6: What happens if a state update is triggered during rendering? L6
React detects this via didScheduleRenderPhaseUpdateDuringThisPass. It re-renders the component (up to 25 times) using HooksDispatcherOnRerender, which processes render-phase updates without cloning from the current fiber. If the limit is exceeded, React throws "Too many re-renders" to prevent infinite loops.
Q7: Explain the double-buffering of hook state between current and work-in-progress fibers. L6
React maintains two fiber trees: current (on screen) and work-in-progress (being built). Each has its own hook linked list. During update, WIP hooks are cloned from current hooks, then updated. After commit, the WIP tree becomes current. The dispatch function is stable because it's bound to the queue object, which is shared between both trees' hook nodes.

useState & useReducer

Q8: How is useState implemented in terms of useReducer? L6
useState calls useReducer with basicStateReducer: function basicStateReducer(state, action) { return typeof action === 'function' ? action(state) : action; }. This handles both direct values (setState(5)) and functional updates (setState(prev => prev + 1)). The update queue, dispatch mechanism, and processing are identical.
Q9: Describe the update queue data structure for state hooks. L6
The update queue is a circular singly-linked list. queue.pending points to the last update, and pending.next points to the first. This allows O(1) appending: set newUpdate.next = pending.next (new points to first), then pending.next = newUpdate (old last points to new), then queue.pending = newUpdate (pending points to new last). Processing starts at pending.next (first) and walks until it loops back.
Q10: What is the eager state bail-out optimization? When does it fail? L6
Before scheduling a re-render, dispatchSetState eagerly computes the new state using the last rendered reducer and compares it with Object.is. If identical, it skips the re-render. This only works when the fiber has no pending updates (fiber.lanes === NoLanes). If there are already queued updates, React can't determine the final state without processing the entire queue, so it must schedule a render even if the value appears unchanged.
Q11: Why does React use Object.is instead of deep equality for state comparison? L6
Deep equality is O(n) and potentially unbounded for nested structures, making it unsuitable for a hot path that runs on every state update. Object.is is O(1) — a single reference comparison for objects. It also handles edge cases better than ===: NaN === NaN is false but Object.is(NaN, NaN) is true, and 0 === -0 is true but Object.is(0, -0) is false. The trade-off is that developers must ensure referential stability for objects.
Q12: Explain automatic batching in React 18. How does flushSync break it? L6
React 18 batches all state updates into a single re-render, regardless of origin (event handlers, promises, timeouts, native events). Updates are queued and processed together in the next microtask. flushSync() forces synchronous processing: it immediately renders and commits the updates inside its callback before returning, then subsequent updates start a new batch.
Q13: What is lazy initialization in useState and why is it important? L6
When useState receives a function, it calls it only during mount to compute the initial state. On subsequent renders, the initializer is ignored. This avoids running expensive computations (parsing JSON, reading localStorage) on every render. Without lazy init, useState(expensiveCompute()) calls expensiveCompute() every render even though the result is discarded after mount.
Q14: Why do functional updates (prev => next) solve the stale closure problem with batching? L6
With direct values, all setState(count + 1) calls in a batch read the same count from the closure, producing the same result. With functional updates, each update receives the accumulated state from processing prior updates in the queue: basicStateReducer(0, prev => prev + 1) → 1, then basicStateReducer(1, prev => prev + 1) → 2. The reducer chains updates sequentially.
Q15: What's the key difference between useState's and useReducer's dispatch functions regarding bail-out? L6
useState uses dispatchSetState which attempts eager bail-out (computes new state with Object.is before scheduling). useReducer uses dispatchReducerAction which always schedules a re-render without eager computation. The rationale: user-provided reducers might have side effects or depend on external state, making eager computation unsafe.

useEffect Lifecycle

Q16: How are effects stored internally? How does this differ from state hooks? L6
Effects are stored in two places: (1) on the hook's memoizedState as an Effect object, and (2) on the fiber's updateQueue as a separate circular linked list of all effects. The updateQueue list is what React iterates during commit. State hooks only use the hook linked list; effects need the additional list for efficient commit-phase processing.
Q17: Explain HookFlags and how React distinguishes between effect types. L6
Each effect has a tag field with bitwise flags: Passive (0b1000) for useEffect, Layout (0b0100) for useLayoutEffect, Insertion (0b0010) for useInsertionEffect, and HasEffect (0b0001) indicating the effect needs to fire. On mount, effects get HasEffect | type. On update, if deps haven't changed, the effect gets only the type flag (without HasEffect), so it's skipped during commit.
Q18: What is the exact execution order of useInsertionEffect, useLayoutEffect, and useEffect? L6
(1) useInsertionEffect cleanup then create — before DOM mutations. (2) DOM mutations (appendChild, etc.). (3) useLayoutEffect cleanup then create — after DOM mutations, before paint. (4) Browser paints. (5) useEffect cleanup then create — asynchronously after paint. Insertion and layout effects block painting; passive effects do not.
Q19: When exactly does useEffect cleanup run? Is it only on unmount? L6
No. Cleanup runs: (1) before every re-execution of the effect (when deps change), and (2) when the component unmounts. Specifically, all passive effect cleanups from the previous render run in a batch before any passive effect creates of the new render. This ensures the previous effect is fully torn down before the new one sets up.
Q20: Explain the stale closure problem and three different solutions. L6
Stale closures occur when an effect captures a variable from an earlier render and never updates. Solutions: (1) Add the variable to the dependency array so the effect re-runs with fresh values. (2) Use functional updates (setState(prev => prev + 1)) to avoid depending on the variable. (3) Store the value in a ref (ref.current = value) and read the ref inside the effect — refs always hold the latest value.
Q21: Why does React double-invoke effects in StrictMode? L6
StrictMode simulates an unmount/remount cycle (setup → cleanup → setup) to verify that effects are properly paired. This catches bugs like missing cleanup (leaked subscriptions, unclosed connections) and non-idempotent effects. It only runs in development and helps prepare for features like Offscreen where components may be mounted/unmounted without user interaction.
Q22: How does the dependency array comparison work internally? L6
areHookInputsEqual iterates both arrays and compares each element with Object.is. If any element differs, it returns false (effect re-runs). null deps (no array) always returns false (run every render). Empty array [] has no elements to compare, so it always returns true after mount (run only once). The function doesn't check array length — the ESLint plugin handles that statically.
Q23: Why are passive effects processed in two passes (all cleanups, then all creates)? L6
Running all cleanups first ensures that teardown in one component doesn't interfere with setup in another. For example, if component A's cleanup removes a DOM node that component B's create needs to measure, running them interleaved could cause B's measurement to fail. The two-pass approach provides a clean separation between the old and new effect states.

useMemo & useCallback

Q24: How are useMemo and useCallback implemented internally? What's stored on the hook? L6
Both store a [value, deps] tuple on hook.memoizedState. useMemo stores the computed result; useCallback stores the function itself. On update, both compare deps with areHookInputsEqual. If deps match, they return the cached value. If not, useMemo re-invokes the factory and useCallback stores the new function. useCallback(fn, deps) is equivalent to useMemo(() => fn, deps).
Q25: When does memoization hurt performance? Give specific examples. L6
Memoization hurts when: (1) The computation is trivial (e.g., useMemo(() => a * 2, [a])) — comparison overhead exceeds saved work. (2) Dependencies change every render — cache always misses, paying both comparison and computation costs. (3) No downstream React.memo consumer — stable references don't prevent any re-renders. (4) Primitive return values — compared by value anyway. (5) Memory pressure — cached values are retained until deps change.
Q26: Explain the React.memo + useCallback pattern. Why are both needed? L6
React.memo wraps a child component to skip re-renders when props are shallowly equal. But if the parent creates a new function/object on each render, React.memo's comparison fails (different references). useCallback stabilizes the function reference, and useMemo stabilizes object references, so React.memo's shallow comparison succeeds. Without React.memo on the child, useCallback alone does nothing.
Q27: How does the React Compiler (Forget) eliminate manual memoization? L6
The React Compiler analyzes the AST at build time, tracks which values each expression depends on (reactive scopes), and inserts a useMemoCache hook with guard conditions. Each expression gets a cache slot; the guard checks if dependencies changed. It auto-memoizes values, callbacks, and JSX elements, eliminating the need for useMemo, useCallback, and React.memo. It assumes code follows Rules of React (pure components, immutable props/state).
Q28: Can React drop memoized values? What does the documentation say? L6
The docs state: "You may rely on useMemo as a performance optimization, not as a semantic guarantee." React reserves the right to discard cached values (e.g., for offscreen components or memory pressure). In practice, React currently never drops them unless the component unmounts, but future versions might. Code should work correctly even if memoized values are recomputed.

useRef & Imperative Handles

Q29: How is useRef implemented internally? Why is it the simplest hook? L6
mountRef creates { current: initialValue } and stores it on hook.memoizedState. updateRef simply returns hook.memoizedState — the same object. No queue, no reducer, no effect list. The object identity is stable because it's the same reference stored on the fiber. Mutations to .current are immediate and don't trigger re-renders.
Q30: What are callback refs and when should you prefer them over object refs? L6
Callback refs are functions called with the DOM node (on attach) or null (on detach). Prefer them when: (1) you need notification when a ref attaches/detaches, (2) you need to perform setup/teardown (observers, measurements), (3) you need to merge multiple refs, (4) the target element is conditionally rendered. React 19 adds cleanup return values to callback refs, making them even more powerful.
Q31: How does useImperativeHandle work internally? L6
It's implemented as a layout effect that sets ref.current to the return value of the create function. This runs synchronously after DOM mutations, ensuring the imperative handle is available before useEffect runs. On cleanup, it sets ref.current = null. The deps array controls when the handle is recreated.
Q32: Why shouldn't you read/write ref.current during render in concurrent mode? L6
In concurrent mode, React may render a component multiple times before committing, or abandon a render entirely. If you write to a ref during render, the mutation persists even if the render is discarded. If you read a ref that was written during a previous abandoned render, you get stale/incorrect data. Refs should be read/written in effects or event handlers, which only run after commit.
Q33: What changed with refs in React 19? L6
Two changes: (1) ref is now a regular prop — function components can accept it directly without forwardRef. (2) Callback refs support cleanup functions — returning a function from a callback ref runs it on detach, similar to useEffect cleanup. This simplifies patterns like ResizeObserver attachment where cleanup was previously manual.

Custom Hooks

Q34: Do multiple components using the same custom hook share state? L6
No. Each component has its own fiber with its own hook linked list. Calling the same custom hook in two components creates independent hook nodes on each fiber. Custom hooks share logic (the code), not state (the data). To share state, use context, external stores, or lift state to a common ancestor.
Q35: Why must custom hooks start with "use"? Is it enforced at runtime? L6
The use prefix is a naming convention enforced by the ESLint plugin, not at runtime. The plugin uses the prefix to identify which functions are hooks and should follow the rules of hooks. A function named getUser that calls useState would not be flagged by the plugin, potentially hiding rule violations. The prefix is the contract between developers and the static analysis tool.
Q36: How do you test custom hooks? What tools are available? L6
@testing-library/react provides renderHook which renders a hook inside a test component. It returns result.current (the hook's return value) and rerender (to simulate prop changes). State updates must be wrapped in act(). For async hooks, use waitFor. For context-dependent hooks, pass a wrapper option with the provider.
Q37: When should you extract a custom hook vs keep logic inline? L6
Extract when: (1) the same stateful logic appears in 2+ components, (2) the hook logic exceeds ~15 lines with a clear responsibility, (3) you want to test stateful logic independently, (4) the logic coordinates multiple hooks. Don't extract: single useState calls, trivial derived values, or logic used in only one component with no testing need. Over-abstraction adds indirection without benefit.

use() Hook (React 19)

Q38: What is use() and how does it differ from other hooks? L6
use() reads a Promise or Context value. Unlike hooks, it doesn't create a node in the fiber's hook linked list — it's a special API that reads from external resources. Because it doesn't rely on positional identity, it can be called conditionally, inside loops, and after early returns. It integrates with Suspense (for promises) and ErrorBoundary (for rejections).
Q39: How does use() suspend a component? What's the Suspense protocol? L6
When use() encounters a pending promise, it throws the promise. The nearest <Suspense> boundary catches it, shows the fallback, and attaches a listener. When the promise resolves, React re-renders the suspended component. On re-render, use() finds the promise fulfilled and returns the value synchronously. If rejected, it throws the error for an ErrorBoundary to catch.
Q40: Why does use() require promise caching? What happens without it? L6
use() tracks promises by reference identity. If a new promise is created every render, each render sees a pending promise, throws it (suspending), then re-renders and creates another new pending promise — an infinite loop. The promise must be created outside the component, cached, or managed by a data fetching library that deduplicates requests.
Q41: How does use() replace useContext? What's the advantage? L6
use(ThemeContext) reads context just like useContext(ThemeContext). The advantage is that use() can be called conditionally: if (condition) { const theme = use(ThemeContext); }. With useContext, this violates rules of hooks. This enables patterns where context is only needed in certain code paths.
Q42: Compare useEffect-based data fetching with use()-based fetching. L6
useEffect fetches after render + paint (waterfall), requires manual loading/error state, manual race condition handling, and doesn't work on the server. use() fetches during render (parallel with Suspense), has automatic loading (Suspense fallback) and error handling (ErrorBoundary), built-in race condition prevention via promise identity, and works with SSR + streaming.

Concurrent Hooks

Q43: How does useTransition work internally? What lane does it use? L6
startTransition sets ReactCurrentBatchConfig.transition, which causes requestUpdateLane() to return TransitionLane instead of SyncLane. Updates inside the callback get this lower-priority lane. The work loop can interrupt transition renders when higher-priority (SyncLane) work arrives. isPending is managed by two setState calls: one urgent (true) and one inside the transition (false).
Q44: When should you use useTransition vs useDeferredValue? L6
Use useTransition when you control the state update and want to mark it as non-urgent (e.g., tab switching, search filtering). Use useDeferredValue when you receive a value as a prop and want to defer expensive re-renders based on it. useTransition wraps the setState call; useDeferredValue wraps the value consumption.
Q45: What is tearing and how does useSyncExternalStore prevent it? L6
Tearing occurs in concurrent rendering when React yields between rendering components, an external store updates, and subsequent components read the new value while earlier components show the old value — inconsistent UI. useSyncExternalStore takes a snapshot before rendering and uses a synchronous layout effect to detect if the store changed during render. If it did, it forces a synchronous re-render.
Q46: How does useId generate stable IDs across server and client? L6
useId generates IDs based on the component's position in the fiber tree, not random values. The algorithm encodes the path through the tree (sibling indices at each level) into a deterministic string. Since server and client traverse the tree in the same order, they produce identical IDs, preventing hydration mismatches.
Q47: Why is useInsertionEffect restricted to CSS-in-JS libraries? L6
useInsertionEffect fires before DOM mutations, so refs are not yet attached and DOM measurements are invalid. It cannot schedule state updates. Its sole purpose is injecting styles before the browser calculates layout, preventing the flash of unstyled content that occurs with useLayoutEffect (which runs after DOM mutations). Application code should never need it.
Q48: Explain how React's lane system enables concurrent rendering with hooks. L6
Lanes are bit-flags assigned to updates: SyncLane for urgent (user input), TransitionLane for deferred work. The work loop checks shouldYield() during concurrent renders — if higher-priority lanes have pending work, it pauses the current render. Hooks like useTransition assign TransitionLane to their updates, while useState defaults to SyncLane. useSyncExternalStore forces SyncLane to prevent tearing.

Advanced & Cross-Cutting

Q49: How would you implement a useEventCallback hook that avoids stale closures without re-running effects? L6
Store the latest callback in a ref, update it every render, and return a stable wrapper: const ref = useRef(fn); ref.current = fn; return useCallback((...args) => ref.current(...args), []). The returned function never changes (stable reference) but always calls the latest closure via the ref. React's experimental useEffectEvent formalizes this pattern.
Q50: What is the relationship between fiber flags and hook effects? L6
When a hook effect needs to fire, it sets flags on both the hook (HookHasEffect) and the fiber (PassiveEffect, UpdateEffect, etc.). During commit, React walks the fiber tree checking fiber flags to find fibers with effects, then iterates the fiber's effect list checking hook flags to find which specific effects need to fire. This two-level flagging enables efficient tree traversal.
Q51: Design a custom hook that synchronizes state with URL search params. What hooks would you compose? L6
Compose useState (for the value), useEffect (to sync state → URL on change), useSyncExternalStore (to subscribe to popstate events for URL → state), and useCallback (for stable setter). Use useSyncExternalStore with subscribe: (cb) => { window.addEventListener('popstate', cb); return () => window.removeEventListener('popstate', cb); } and getSnapshot: () => new URLSearchParams(location.search).get(key).
Q52: How does the React Compiler know which values to memoize? L6
The compiler parses the AST, identifies reactive values (props, state, context), and builds a dependency graph. Each expression is assigned a "reactive scope" based on which reactive values it depends on. The compiler inserts cache slots with guard conditions that check if dependencies changed. It groups expressions into scopes to minimize cache entries — if two expressions share the same dependencies, they share a scope.
Q53: What happens when you call setState inside useLayoutEffect? How does this differ from useEffect? L6
setState inside useLayoutEffect triggers a synchronous re-render before the browser paints. The user sees only the final state — no visual flicker. In useEffect, setState schedules a re-render after paint, causing a visible flash of the intermediate state. Use useLayoutEffect for DOM measurements that immediately update state (e.g., tooltip positioning).
Q54: How would you debug a component that re-renders too often due to hooks? L6
(1) React DevTools Profiler to identify which components re-render and why. (2) React.memo with a custom comparator to log prop changes. (3) Check for new object/array/function references in props (common with inline objects and missing useCallback/useMemo). (4) Check for context value changes (new object every render). (5) Check for unnecessary state updates (setState with same value but new reference). (6) Use why-did-you-render library for automated detection.
Q55: Explain the complete lifecycle of a useState call from dispatch to DOM update. L6
(1) setState(value) calls dispatchSetState. (2) Creates an Update object, appends to circular queue. (3) Attempts eager bail-out with Object.is. (4) If not bailed out, calls scheduleUpdateOnFiber. (5) Scheduler picks up the work, calls renderWithHooks. (6) Sets dispatcher to HooksDispatcherOnUpdate. (7) Component re-renders, updateReducer processes the queue. (8) Returns new state. (9) Reconciler diffs old and new children. (10) Commit phase applies DOM mutations. (11) Layout effects fire. (12) Browser paints. (13) Passive effects fire.
Q56: How do hooks interact with React's error boundaries? L6
If a hook throws during render (e.g., use() with a rejected promise), the error propagates up the fiber tree to the nearest error boundary. Effect cleanup functions that throw are caught and logged but don't crash the app. useState/useReducer reducers that throw during render cause the error boundary to activate. Error boundaries must be class components — there's no hook equivalent for getDerivedStateFromError.
Q57: What is the "rules of hooks" ESLint plugin actually checking under the hood? L6
The rules-of-hooks rule builds a control flow graph (CFG) from the AST. It identifies all hook calls and verifies that every hook is on every possible execution path from function entry to exit. If a hook is inside a branch (if/else, loop, early return), it's not on all paths — violation. The exhaustive-deps rule analyzes closures to find variables used inside effects/memos that aren't in the dependency array.
Q58: How would you migrate a class component with complex lifecycle methods to hooks? L6
Map lifecycles: constructor → useState with lazy init. componentDidMount → useEffect(..., []). componentDidUpdate → useEffect with deps. componentWillUnmount → useEffect cleanup. shouldComponentUpdate → React.memo. getDerivedStateFromProps → compute during render or useMemo. Instance variables → useRef. Split one large lifecycle into multiple focused effects by concern, not by lifecycle timing.
Q59: What are the performance implications of having many hooks in a single component? L6
Each hook adds a node to the linked list (memory), requires traversal during update (O(n) per render), and each useMemo/useCallback runs dependency comparison. For most components this is negligible, but components with 50+ hooks may see measurable overhead. Mitigation: extract groups of related hooks into custom hooks (doesn't reduce work but improves readability), split the component, or use useReducer to consolidate multiple useState calls.
Q60: How do Server Components change the hooks landscape? L6
Server Components cannot use hooks at all — they're async functions that run on the server and produce serializable output. Only Client Components (marked with 'use client') use hooks. This creates a clear boundary: Server Components handle data fetching with async/await, Client Components handle interactivity with hooks. The use() API bridges them by allowing Client Components to read promises passed from Server Components.
Q61: Design a hook system for a component that needs to coordinate animation, data fetching, and user input simultaneously. L6
Use useTransition for data fetching (non-urgent), useRef + useLayoutEffect for animation frame coordination (synchronous DOM access), useState for input state (urgent), and useDeferredValue for rendering results based on input. The animation hook uses requestAnimationFrame in a layout effect, the fetch hook uses startTransition to avoid blocking input, and useDeferredValue ensures the results list doesn't block typing.
Q62: What would break if React switched from a linked list to an array for hook storage? L6
Functionally, very little — arrays would also provide positional access. The linked list was chosen for: (1) O(1) append during mount (no array resizing), (2) efficient cloning during update (clone nodes as you walk, don't pre-allocate), (3) natural fit for the work loop's incremental processing (can pause mid-list). An array would need pre-allocation or dynamic resizing, and the memory layout differences are negligible for typical hook counts.
