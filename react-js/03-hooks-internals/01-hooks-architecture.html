Hooks Architecture

How React stores, resolves, and dispatches hooks via fiber-level linked lists and phase-specific dispatchers.

Hooks as a Linked List on Fiber

Every function component in React corresponds to a fiber node in the fiber tree. When a component uses hooks, React stores all hook state on the fiber's memoizedState field as a singly-linked list. Each node in this list represents one hook call.

// Simplified hook node structure (ReactFiberHooks.js)
type Hook = {
  memoizedState: any,    // current state value (varies by hook type)
  baseState: any,        // base state before pending updates
  baseQueue: Update | null,  // first update in the base queue
  queue: UpdateQueue | null, // pending update queue
  next: Hook | null,     // pointer to next hook in the list
};

// On the fiber:
fiber.memoizedState → Hook1 → Hook2 → Hook3 → null
                      (useState) (useEffect) (useMemo)
Architect Insight

The linked-list design means hooks are positionally identified. React doesn't use names or keys — it relies on call order. This is the fundamental reason hooks must be called in the same order every render: the Nth hook call maps to the Nth node in the linked list.

Mount vs Update Dispatchers

React uses two completely different sets of hook implementations depending on whether a component is mounting (first render) or updating (re-render). These are called dispatchers.

// From ReactFiberHooks.js — simplified
const HooksDispatcherOnMount = {
  useState:    mountState,
  useReducer:  mountReducer,
  useEffect:   mountEffect,
  useMemo:     mountMemo,
  useCallback: mountCallback,
  useRef:      mountRef,
  // ... other hooks
};

const HooksDispatcherOnUpdate = {
  useState:    updateState,
  useReducer:  updateReducer,
  useEffect:   updateEffect,
  useMemo:     updateMemo,
  useCallback: updateCallback,
  useRef:      updateRef,
  // ... other hooks
};

// During render, React sets:
ReactCurrentDispatcher.current = isMount
  ? HooksDispatcherOnMount
  : HooksDispatcherOnUpdate;

Mount Phase: Building the Linked List

function mountWorkInProgressHook(): Hook {
  const hook: Hook = {
    memoizedState: null,
    baseState: null,
    baseQueue: null,
    queue: null,
    next: null,
  };

  if (workInProgressHook === null) {
    // First hook in the component — set as head of list
    currentlyRenderingFiber.memoizedState = hook;
  } else {
    // Append to end of list
    workInProgressHook.next = hook;
  }
  workInProgressHook = hook;
  return hook;
}

// mountState creates the hook node and initializes state:
function mountState(initialState) {
  const hook = mountWorkInProgressHook();
  if (typeof initialState === 'function') {
    initialState = initialState(); // lazy initialization
  }
  hook.memoizedState = hook.baseState = initialState;
  const queue = { pending: null, dispatch: null, /* ... */ };
  hook.queue = queue;
  const dispatch = queue.dispatch = dispatchSetState.bind(
    null, currentlyRenderingFiber, queue
  );
  return [hook.memoizedState, dispatch];
}

Update Phase: Walking the Existing List

function updateWorkInProgressHook(): Hook {
  // Walk the existing linked list from the current fiber
  let nextCurrentHook;
  if (currentHook === null) {
    const current = currentlyRenderingFiber.alternate;
    nextCurrentHook = current?.memoizedState ?? null;
  } else {
    nextCurrentHook = currentHook.next;
  }

  // Clone the hook for the work-in-progress fiber
  const newHook: Hook = {
    memoizedState: nextCurrentHook.memoizedState,
    baseState:     nextCurrentHook.baseState,
    baseQueue:     nextCurrentHook.baseQueue,
    queue:         nextCurrentHook.queue,
    next:          null,
  };

  // Append to work-in-progress list
  if (workInProgressHook === null) {
    currentlyRenderingFiber.memoizedState = newHook;
  } else {
    workInProgressHook.next = newHook;
  }
  workInProgressHook = newHook;
  currentHook = nextCurrentHook;
  return newHook;
}
Critical

If the number of hooks changes between renders (e.g., a hook inside an if block), updateWorkInProgressHook will walk to the wrong node in the linked list. Hook #3 on update might read state from what was Hook #2 on mount. This causes silent data corruption, not just a crash.

Dispatcher Switching Mechanism

React maintains a global variable ReactCurrentDispatcher that points to the active dispatcher. This is how useState() — a plain function call — knows whether to mount or update.

// ReactCurrentDispatcher.js
const ReactCurrentDispatcher = {
  current: null  // set before rendering each component
};

// When you call useState():
function useState(initialState) {
  const dispatcher = resolveDispatcher();
  return dispatcher.useState(initialState);
}

function resolveDispatcher() {
  const dispatcher = ReactCurrentDispatcher.current;
  // In DEV, throws if null (hooks called outside component)
  return dispatcher;
}

The ContextOnlyDispatcher

After rendering completes, React swaps in a third dispatcher — ContextOnlyDispatcher — where every hook throws an error. This is how React enforces that hooks can only be called during rendering.

const ContextOnlyDispatcher = {
  useState: throwInvalidHookError,
  useEffect: throwInvalidHookError,
  useReducer: throwInvalidHookError,
  // ... all hooks throw
};

function throwInvalidHookError() {
  throw new Error(
    'Invalid hook call. Hooks can only be called inside ' +
    'the body of a function component.'
  );
}
DEV-Only Dispatchers

In development mode, React uses additional dispatchers (HooksDispatcherOnMountInDEV, HooksDispatcherOnUpdateInDEV) that wrap each hook with extra validation — checking for hook order changes, invalid nesting, and providing better error messages.

Rules of Hooks Enforcement

The "Rules of Hooks" are not arbitrary conventions — they are structural requirements of the linked-list architecture:

Rule 1: Only Call Hooks at the Top Level

// VIOLATION: conditional hook
function MyComponent({ showExtra }) {
  const [name, setName] = useState('');  // Hook #0

  if (showExtra) {
    const [extra, setExtra] = useState(''); // Hook #1 (sometimes)
  }

  useEffect(() => { /* ... */ });  // Hook #1 or #2 depending on branch

  // On mount with showExtra=true:
  //   Hook #0: useState('') → name
  //   Hook #1: useState('') → extra
  //   Hook #2: useEffect
  //
  // On update with showExtra=false:
  //   Hook #0: useState → reads node #0 ✓ (name)
  //   Hook #1: useEffect → reads node #1 ✗ (gets extra's state!)
}

Rule 2: Only Call Hooks from React Functions

Hooks rely on ReactCurrentDispatcher.current being set, which only happens inside renderWithHooks(). Calling hooks from a regular function means the dispatcher is ContextOnlyDispatcher, which throws.

ESLint Plugin: eslint-plugin-react-hooks

// .eslintrc.js
module.exports = {
  plugins: ['react-hooks'],
  rules: {
    'react-hooks/rules-of-hooks': 'error',     // checks call order
    'react-hooks/exhaustive-deps': 'warn',      // checks dependency arrays
  },
};

// The rules-of-hooks rule uses static analysis to verify:
// 1. Hooks are not called inside conditions, loops, or nested functions
// 2. Hooks are called from React function components or custom hooks
// 3. Custom hooks (use* prefix) follow the same rules
//
// It builds a control flow graph (CFG) and checks that every
// hook call is on every possible execution path.

How React Knows Which Component Is Rendering

React tracks the currently rendering fiber via module-scoped variables in ReactFiberHooks.js:

// Module-scoped state in ReactFiberHooks.js
let currentlyRenderingFiber: Fiber | null = null;
let currentHook: Hook | null = null;            // current tree's hook
let workInProgressHook: Hook | null = null;     // WIP tree's hook

function renderWithHooks(
  current: Fiber | null,      // current fiber (null on mount)
  workInProgress: Fiber,      // work-in-progress fiber
  Component: Function,
  props: any,
  secondArg: any,             // ref for forwardRef
  nextRenderLanes: Lanes,
): any {
  currentlyRenderingFiber = workInProgress;
  workInProgress.memoizedState = null;  // reset on mount
  workInProgress.updateQueue = null;

  // Choose dispatcher based on mount vs update
  ReactCurrentDispatcher.current =
    current === null || current.memoizedState === null
      ? HooksDispatcherOnMount
      : HooksDispatcherOnUpdate;

  // Render the component
  let children = Component(props, secondArg);

  // Handle re-renders triggered during render
  if (didScheduleRenderPhaseUpdateDuringThisPass) {
    let numberOfReRenders = 0;
    do {
      didScheduleRenderPhaseUpdateDuringThisPass = false;
      numberOfReRenders++;
      if (numberOfReRenders > RE_RENDER_LIMIT) { // 25
        throw new Error('Too many re-renders.');
      }
      // Reset and re-render with ReRender dispatcher
      workInProgressHook = null;
      currentHook = null;
      ReactCurrentDispatcher.current = HooksDispatcherOnRerender;
      children = Component(props, secondArg);
    } while (didScheduleRenderPhaseUpdateDuringThisPass);
  }

  // Prevent hooks outside render
  ReactCurrentDispatcher.current = ContextOnlyDispatcher;

  // Cleanup
  currentlyRenderingFiber = null;
  currentHook = null;
  workInProgressHook = null;

  return children;
}

Hook Resolution During Render

Component calls useState(0) │ ├─ resolveDispatcher() │ └─ returns ReactCurrentDispatcher.current │ (HooksDispatcherOnMount or HooksDispatcherOnUpdate) │ ├─ dispatcher.useState(0) │ │ │ ├── MOUNT PATH (mountState): │ │ ├─ mountWorkInProgressHook() │ │ │ └─ Create new Hook node, append to linked list │ │ ├─ Initialize: hook.memoizedState = initialState │ │ ├─ Create update queue │ │ ├─ Bind dispatch to fiber + queue │ │ └─ Return [state, dispatch] │ │ │ └── UPDATE PATH (updateState → updateReducer): │ ├─ updateWorkInProgressHook() │ │ └─ Clone next hook from current fiber's list │ ├─ Process pending updates from queue │ ├─ Compute new state via reducer │ └─ Return [newState, dispatch] │ └─ Component receives [state, setState]

Double Buffering: Current vs Work-In-Progress

React maintains two fiber trees: current (what's on screen) and work-in-progress (being built). Hooks exist on both, and during update, the WIP hooks are cloned from current hooks:

// The alternate relationship:
currentFiber.alternate = wipFiber;
wipFiber.alternate = currentFiber;

// During update, hooks are cloned:
// current fiber:  Hook_A → Hook_B → Hook_C
// WIP fiber:      Hook_A' → Hook_B' → Hook_C'  (cloned, then updated)

// After commit, the WIP tree becomes the new current tree.
// On next render, the old current tree becomes the new WIP tree.
// This is why dispatch functions are stable — they're bound to
// the queue object, which is shared between current and WIP hooks.
Architect Insight — Dispatch Stability

The dispatch function returned by useState is created once during mount and bound to the queue object. Since both current and WIP hook nodes share the same queue reference, the dispatch function remains stable across re-renders without needing useCallback.

The Re-render Dispatcher

When a state update is triggered during rendering (e.g., calling setState inside the render body), React uses a special HooksDispatcherOnRerender. This handles the case where updates are queued on hooks that have already been processed in the current render pass.

// Re-render scenario:
function Counter() {
  const [count, setCount] = useState(0);

  // This triggers a re-render during render
  if (count === 0) {
    setCount(1); // schedules update on already-processed hook
  }

  return <div>{count}</div>;
}

// React detects this via didScheduleRenderPhaseUpdateDuringThisPass
// and re-renders with HooksDispatcherOnRerender, which processes
// the render-phase updates from the queue without cloning from current.

Interview Tip

When asked "why can't hooks be called conditionally?", don't just say "it's a rule." Explain the linked-list architecture: hooks are positionally mapped to nodes, so changing the call order means reading the wrong node's state. The ESLint plugin enforces this statically by building a control flow graph.
