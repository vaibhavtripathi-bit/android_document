1. App Router Architecture

The Next.js App Router (introduced in Next.js 13.4) represents a fundamental architectural shift. It's built on React Server Components, uses file-system conventions for routing, and co-locates data fetching with route segments.

App Router Architecture: Request → Next.js Server │ ├── Middleware (edge) ─── redirect/rewrite/headers │ ├── Route Matching (file-system based) │ └── /app/dashboard/[id]/page.tsx │ ├── Component Tree Construction │ ├── layout.tsx (Server Component — cached) │ ├── loading.tsx → Suspense boundary │ ├── error.tsx → Error boundary │ └── page.tsx (Server Component — data fetching) │ ├── Server Rendering │ ├── RSC Payload (serialized component tree) │ └── HTML Stream (progressive) │ └── Client Hydration ├── Selective hydration (only 'use client' components) └── Router state initialization
Architect Insight

The App Router unifies three previously separate concerns: routing, rendering, and data fetching. In the Pages Router, you chose between getServerSideProps, getStaticProps, and client-side fetching. In the App Router, every component is a Server Component by default — data fetching is just await in the component body. This eliminates the mental model of "where does this data come from?" — it comes from the server, always, unless you explicitly opt into client-side behavior.

2.1 Special File Conventions

File Purpose Renders As
page.tsx Route UI — makes segment publicly accessible Route component
layout.tsx Shared UI wrapping children — preserved across navigations Wraps {children}
loading.tsx Loading UI — instant Suspense fallback <Suspense fallback={...}>
error.tsx Error UI — catches errors in segment and children Error boundary
not-found.tsx 404 UI — shown when notFound() is called Not-found boundary
template.tsx Like layout but re-mounts on navigation Wraps {children}
default.tsx Fallback for parallel route slots Slot fallback
route.ts API endpoint (no UI) Request handler

3. Server Components in Routes

// app/users/[userId]/page.tsx — Server Component by default
import { db } from '@/lib/db';
import { notFound } from 'next/navigation';

interface Props {
  params: { userId: string };
}

export default async function UserPage({ params }: Props) {
  const user = await db.user.findUnique({
    where: { id: params.userId },
  });

  if (!user) notFound();

  return (
    <div>
      <h1>{user.name}</h1>
      <p>{user.email}</p>
      <Suspense fallback={<PostsSkeleton />}>
        <UserPosts userId={user.id} />
      </Suspense>
    </div>
  );
}

// UserPosts — also a Server Component, streams in via Suspense
async function UserPosts({ userId }: { userId: string }) {
  const posts = await db.post.findMany({
    where: { authorId: userId },
    orderBy: { createdAt: 'desc' },
  });

  return (
    <ul>
      {posts.map(post => <li key={post.id}>{post.title}</li>)}
    </ul>
  );
}
Server Component Benefits for Routing

Server Components in routes mean: (1) zero client JS for data fetching — no useEffect, no loading states, (2) direct database/filesystem access — no API layer needed, (3) secrets stay on the server — API keys never reach the client, (4) automatic streaming — Suspense boundaries stream HTML progressively, (5) reduced bundle size — server-only code is never sent to the client.

4. Layout Nesting

// app/layout.tsx — Root layout (required)
export default function RootLayout({ children }) {
  return (
    <html lang="en">
      <body>
        <Header />
        {children}
        <Footer />
      </body>
    </html>
  );
}

// app/dashboard/layout.tsx — Nested layout
export default function DashboardLayout({ children }) {
  return (
    <div className="dashboard">
      <DashboardSidebar />
      <main>{children}</main>
    </div>
  );
}
URL: /dashboard/settings Component Nesting: <RootLayout> ← app/layout.tsx (persists across ALL routes) <Header /> <DashboardLayout> ← app/dashboard/layout.tsx (persists within /dashboard/*) <DashboardSidebar /> <main> <SettingsPage /> ← app/dashboard/settings/page.tsx </main> </DashboardLayout> <Footer /> </RootLayout> Navigation: /dashboard/settings → /dashboard/analytics - RootLayout: NOT re-rendered (preserved) - DashboardLayout: NOT re-rendered (preserved, including sidebar state) - SettingsPage → AnalyticsPage: swapped
Layout vs Template

Layouts persist across navigations — their state is preserved and they don't re-mount. Templates (template.tsx) re-mount on every navigation, creating a new instance. Use templates when you need: (1) enter/exit animations per route, (2) per-page analytics tracking via useEffect, (3) resetting state on navigation (e.g., form reset).

5. Loading & Error Conventions

// app/dashboard/loading.tsx — instant loading UI
export default function DashboardLoading() {
  return <DashboardSkeleton />;
}
// Next.js wraps page.tsx in <Suspense fallback={<DashboardLoading />}>

// app/dashboard/error.tsx — must be a Client Component
'use client';

export default function DashboardError({
  error,
  reset,
}: {
  error: Error & { digest?: string };
  reset: () => void;
}) {
  useEffect(() => {
    reportError(error);
  }, [error]);

  return (
    <div>
      <h2>Something went wrong</h2>
      <p>{error.message}</p>
      <button onClick={reset}>Try again</button>
    </div>
  );
}

// app/dashboard/not-found.tsx
export default function NotFound() {
  return (
    <div>
      <h2>Dashboard Not Found</h2>
      <p>The requested dashboard page does not exist.</p>
    </div>
  );
}
Component Hierarchy with Conventions: <Layout> <Template> <ErrorBoundary fallback={<error.tsx />}> <Suspense fallback={<loading.tsx />}> <NotFoundBoundary fallback={<not-found.tsx />}> <page.tsx /> </NotFoundBoundary> </Suspense> </ErrorBoundary> </Template> </Layout> Error boundary wraps Suspense → errors during loading are caught Suspense wraps page → loading.tsx shows while page.tsx data loads

6. Parallel Routes

Parallel routes allow rendering multiple pages simultaneously in the same layout, each with independent loading and error states.

// File structure for parallel routes
app/dashboard/
├── layout.tsx
├── page.tsx
├── @analytics/           // slot name: analytics
│   ├── page.tsx
│   ├── loading.tsx
│   └── default.tsx
├── @team/                // slot name: team
│   ├── page.tsx
│   ├── loading.tsx
│   └── default.tsx
└── @notifications/       // slot name: notifications
    ├── page.tsx
    └── default.tsx

// app/dashboard/layout.tsx — receives slots as props
export default function DashboardLayout({
  children,       // default slot (page.tsx)
  analytics,      // @analytics slot
  team,           // @team slot
  notifications,  // @notifications slot
}) {
  return (
    <div className="dashboard-grid">
      <main>{children}</main>
      <aside>{analytics}</aside>
      <section>{team}</section>
      <section>{notifications}</section>
    </div>
  );
}

// Each slot loads independently — if analytics is slow,
// team and notifications still render immediately

7. Intercepting Routes

Intercepting routes allow you to load a route from another part of the application within the current layout — the primary use case is modals.

// File structure for intercepting routes (Instagram-style photo modal)
app/
├── layout.tsx
├── feed/
│   └── page.tsx           // photo grid
├── photo/
│   └── [id]/
│       └── page.tsx       // full photo page (direct URL access)
└── @modal/
    └── (.)photo/          // (.) intercepts same-level route
        └── [id]/
            └── page.tsx   // photo modal (when navigating from feed)

// Interception conventions:
// (.)  — intercept same level
// (..) — intercept one level up
// (..)(..) — intercept two levels up
// (...) — intercept from root

// app/layout.tsx
export default function RootLayout({ children, modal }) {
  return (
    <html>
      <body>
        {children}
        {modal}  {/* modal renders on top when intercepted */}
      </body>
    </html>
  );
}

// Behavior:
// Click photo in feed → modal opens (intercepted route)
// Direct URL /photo/42 → full photo page (not intercepted)
// Refresh while modal open → full photo page
// Share URL → recipient sees full photo page

8. Dynamic Routes & generateStaticParams

// app/blog/[slug]/page.tsx — Dynamic route
interface Props {
  params: { slug: string };
}

export default async function BlogPost({ params }: Props) {
  const post = await getPost(params.slug);
  return <article>{post.content}</article>;
}

// generateStaticParams — pre-render at build time (SSG)
export async function generateStaticParams() {
  const posts = await getAllPosts();
  return posts.map(post => ({
    slug: post.slug,
  }));
}
// Generates: /blog/intro-to-react, /blog/hooks-deep-dive, etc.

// Catch-all segments
// app/docs/[...slug]/page.tsx
// Matches: /docs/a, /docs/a/b, /docs/a/b/c
// params.slug: ['a'], ['a', 'b'], ['a', 'b', 'c']

// Optional catch-all
// app/docs/[[...slug]]/page.tsx
// Also matches: /docs (params.slug: undefined)

// generateMetadata — dynamic metadata per route
export async function generateMetadata({ params }: Props) {
  const post = await getPost(params.slug);
  return {
    title: post.title,
    description: post.excerpt,
    openGraph: { images: [post.coverImage] },
  };
}

9. Middleware for Routing

// middleware.ts (at project root)
import { NextResponse } from 'next/server';
import type { NextRequest } from 'next/server';

export function middleware(request: NextRequest) {
  // Auth check
  const token = request.cookies.get('auth-token');
  if (!token && request.nextUrl.pathname.startsWith('/dashboard')) {
    return NextResponse.redirect(new URL('/login', request.url));
  }

  // Geo-based routing
  const country = request.geo?.country;
  if (country === 'DE' && request.nextUrl.pathname === '/') {
    return NextResponse.rewrite(new URL('/de', request.url));
  }

  // A/B testing via cookie
  const response = NextResponse.next();
  if (!request.cookies.has('experiment-group')) {
    const group = Math.random() > 0.5 ? 'A' : 'B';
    response.cookies.set('experiment-group', group);
  }

  return response;
}

// Matcher: only run middleware on specific paths
export const config = {
  matcher: ['/dashboard/:path*', '/', '/api/:path*'],
};

10. Pages Router vs App Router

Feature Pages Router App Router
Component model All Client Components Server Components by default
Data fetching getServerSideProps, getStaticProps async components, fetch with caching
Layouts Manual (per-page getLayout) Built-in nested layouts (preserved)
Loading states Manual (per-page) loading.tsx convention
Error handling _error.tsx (global) error.tsx per segment
Streaming Not supported Built-in via Suspense
Parallel routes Not supported @slot convention
Route groups Not supported (group) convention
Bundle size All component code sent to client Server Components = zero client JS
Mutations API routes + client fetch Server Actions (RPC-style)

11. Server Actions Integration

// app/users/[id]/page.tsx
import { revalidatePath } from 'next/cache';

export default async function UserPage({ params }) {
  const user = await db.user.findUnique({ where: { id: params.id } });

  async function updateName(formData: FormData) {
    'use server';
    await db.user.update({
      where: { id: params.id },
      data: { name: formData.get('name') },
    });
    revalidatePath(`/users/${params.id}`);
  }

  return (
    <form action={updateName}>
      <input name="name" defaultValue={user.name} />
      <button type="submit">Save</button>
    </form>
  );
}

// Server Actions work without JavaScript enabled (progressive enhancement)
// With JS: form submission is intercepted, action runs via fetch
// Without JS: form submits as traditional POST, server handles it
Architect Insight

Server Actions represent a return to the simplicity of traditional web frameworks (Rails, Django) where form submissions go directly to server-side handlers. The key difference is that Next.js provides this with full React interactivity — optimistic updates via useOptimistic, pending states via useFormStatus, and progressive enhancement. This is the convergence of SPA interactivity with MPA simplicity.
