Client-Side Routing (Questions 1–10)

1. Explain the difference between pushState and replaceState. When would you use each? Senior

pushState adds a new entry to the browser's history stack, while replaceState modifies the current entry in-place without creating a new one.

Use pushState for standard navigation where the user should be able to go back (page transitions, opening new views). Use replaceState for URL updates that shouldn't create a new history entry: updating search params during filtering, correcting a URL after a redirect, updating URL after form submission to prevent re-submission on back, or replacing a loading URL with the final URL.

Both methods accept a state object (serializable, ~640KB limit), an unused title parameter, and a URL (must be same-origin). Neither fires the popstate event — only back/forward navigation does.

2. Why doesn't pushState fire the popstate event? What are the implications for router libraries? L6

popstate only fires for user-initiated navigation (back/forward buttons) or programmatic history.back()/history.forward()/history.go(). pushState and replaceState are considered programmatic URL updates, not navigations.

Implications: Router libraries cannot rely solely on popstate for all navigation events. They must: (1) wrap pushState/replaceState in their own navigate() function that triggers route matching, (2) maintain an internal event system (e.g., history.listen() in React Router), (3) handle both programmatic navigation (their own events) and browser navigation (popstate). This is why the Navigation API was created — it fires for all navigations.

3. Compare hash routing and browser (History API) routing. When is hash routing still appropriate? Senior

Hash routing uses the URL fragment (#/path) which is never sent to the server. Browser routing uses the History API (pushState) with clean URLs (/path).

Hash routing is still appropriate when: (1) you can't configure the server to return index.html for all routes (static file hosting like GitHub Pages without SPA configuration), (2) the app is embedded in an iframe or environment where you can't control the server, (3) you need to support very old browsers (IE9). For all other cases, browser routing is preferred because it provides clean URLs, SEO compatibility, SSR support, and standard analytics tracking.

4. How does scroll restoration work in SPAs? What are the challenges? L6

Browsers automatically restore scroll position for MPA navigations, but SPAs must handle it manually because the page doesn't actually reload. The approach: (1) set history.scrollRestoration = 'manual' to disable browser's automatic behavior, (2) save scroll position before each navigation (keyed by history entry), (3) restore scroll position on back/forward navigation, (4) scroll to top on forward navigation.

Challenges: (1) Virtualized lists — DOM nodes don't exist yet at the saved scroll position, requiring the virtualizer to seek to the correct offset. (2) Lazy-loaded content — images and async content change page height after scroll restoration. (3) Multiple scroll containers — not just window but also overflow containers. (4) Suspense boundaries — content may not be rendered when scroll restoration runs. (5) Timing — must wait for DOM update before scrolling, typically using requestAnimationFrame.

5. Explain the Navigation API and how it improves on the History API. L6

The Navigation API (Chrome 102+) addresses fundamental limitations of the History API: (1) navigate event fires for ALL navigations (push, replace, back, forward, link clicks, form submissions) — unlike popstate which only fires for back/forward. (2) event.intercept() provides clean navigation interception with an async handler. (3) navigation.entries() exposes the full history stack (History API only gives history.length). (4) navigation.transition tracks in-progress navigations. (5) Built-in abort signal for canceling navigations. (6) navigation.canGoBack/canGoForward properties.

This eliminates the need for router libraries to monkey-patch pushState, maintain internal event systems, or use delegated click handlers for link interception.

6. How would you implement a client-side router from scratch? Walk through the key components. Architect

A minimal client-side router needs: (1) Route registry — a data structure mapping URL patterns to handlers, with pattern-to-regex compilation for dynamic segments. (2) History wrapper — wraps pushState/replaceState with an event system that notifies listeners on all navigations, not just popstate. (3) Route matcher — given a URL, finds the best matching route using ranked scoring (static segments > dynamic > splat). (4) Link interceptor — delegated click handler on document that intercepts same-origin <a> clicks, respecting modifier keys and target attributes. (5) Scroll manager — saves/restores scroll positions keyed by history entry. (6) Navigation blocker — prevents navigation when conditions aren't met (unsaved changes).

For React integration, add: context providers for location/navigation state, hooks (useLocation, useParams, useNavigate), and an Outlet component for nested route rendering.

7. What is the same-origin restriction on pushState and why does it exist? Senior

pushState can only set URLs with the same origin (protocol + hostname + port) as the current page. Attempting to set a different origin throws a SecurityError. This exists to prevent phishing attacks — a malicious page could otherwise change the URL to look like a trusted site (e.g., bank.com) while still running attacker-controlled JavaScript. The same-origin restriction ensures the URL bar always truthfully reflects which origin's code is executing.

8. How does the URLPattern API work and what problems does it solve? L6

URLPattern is a new web standard that provides Express-style route matching natively in the browser. It accepts patterns like /users/:id and returns matched groups. It solves: (1) every router library reimplementing pattern matching with slightly different syntax, (2) no standard way to match URLs in service workers, (3) inconsistent pattern syntax across frameworks. It supports named groups (:id), wildcards (*), regex constraints (:id(\\d+)), and optional segments. Currently available in Chrome, Deno, and Cloudflare Workers.

9. What server configuration is needed for client-side routing and why? Senior

The server must be configured to return index.html for all routes that don't match a static file. This is called a "catch-all" or "fallback" configuration. Without it, navigating directly to /dashboard returns a 404 because no dashboard.html file exists on the server — the route only exists in client-side JavaScript.

Examples: Nginx uses try_files $uri $uri/ /index.html. Apache uses FallbackResource /index.html. Express uses app.get('*', (req, res) => res.sendFile('index.html')). Netlify uses a _redirects file with /* /index.html 200. This is one reason hash routing exists — the hash is never sent to the server, so no server configuration is needed.

10. Explain the history.state size limit and its implications for routing. L6

The state object passed to pushState/replaceState is serialized using the structured clone algorithm and has a browser-specific size limit (~640KB in most browsers, 16MB in Firefox). Exceeding it throws a DataCloneError. The state must be serializable — no functions, DOM nodes, or class instances.

Implications: Don't store large data in history state. Use it for metadata (scroll position, modal background location, navigation direction) not application data. For large state, store a key in history state and look up the actual data from a separate store (sessionStorage, IndexedDB, or in-memory cache).

React Router Internals (Questions 11–20)

11. How does React Router v6's ranked route matching differ from v5's first-match approach? L6

React Router v5 used first-match: routes were tried in order, and the first matching route won. This required careful manual ordering (specific routes before general ones) and the exact prop. v6 uses ranked matching: all routes are scored by specificity (static segments = 10 points, dynamic = 3, splat = penalty), and the highest-scoring match wins regardless of definition order.

This eliminates: (1) the need for exact prop, (2) bugs from incorrect route ordering, (3) the Switch component (replaced by automatic best-match). The algorithm is similar to CSS specificity — more specific patterns always win.

12. Explain the relationship between @remix-run/router, react-router, and react-router-dom. L6

@remix-run/router is the framework-agnostic core — it handles route matching, data loading (loaders/actions), navigation state management, and history abstraction with zero React dependency. react-router adds React bindings: context providers, hooks (useNavigate, useLocation, useParams), and components (Outlet, Navigate). react-router-dom adds DOM-specific bindings: BrowserRouter, Link, Form, ScrollRestoration, and the createBrowserRouter factory.

This layered architecture means the core routing logic is testable without React, reusable in other frameworks (the same core powers Remix), and the React layer is a thin binding. This is a strong architectural pattern — separating the domain logic from the framework binding.

13. What is the Outlet component and how does it enable nested routing? Senior

Outlet is a placeholder component that renders the matched child route's element. When a URL matches a nested route tree (e.g., /dashboard/settings matches [Root, Dashboard, Settings]), each parent route renders its own UI plus an <Outlet /> where the child route's element appears. This creates a component hierarchy that mirrors the URL hierarchy.

Internally, Outlet reads from a context that contains the array of matched routes and renders the next match in the sequence. It also supports an optional context prop for passing data from parent to child routes without prop drilling, accessible via useOutletContext().

14. How do loaders and actions work in React Router's data router? What problems do they solve? L6

Loaders are functions that fetch data before a route component renders. Actions handle mutations (form submissions). When a URL matches, all matched routes' loaders run in parallel before any component renders, eliminating fetch waterfalls. Actions run on form submission, then automatically revalidate all active loaders.

They solve: (1) fetch waterfalls — all data loads in parallel, (2) race conditions — in-flight loaders are automatically aborted on new navigation via request.signal, (3) loading states — useNavigation() provides global loading state, (4) error handling — thrown Responses become route-level errors, (5) revalidation — after mutations, data is automatically refreshed. The loader/action pattern mirrors traditional web frameworks (GET = loader, POST = action).

15. Explain relative path resolution in React Router v6. Why does '..' navigate to the parent route, not the parent URL segment? L6

In v6, relative paths resolve against the route hierarchy, not URL segments. .. means "parent route" — if the current route is /dashboard/users/:id and its parent route is /dashboard, then .. navigates to /dashboard, not /dashboard/users.

This matters for pathless layout routes. If you have a layout route with no path wrapping /users/:id, .. skips the pathless layout and goes to its parent. This is more intuitive for the route tree structure but can be confusing when you think in URL segments. Use relative="path" on <Link> to get URL-segment-based resolution instead.

16. What is createMemoryRouter and when would you use it? Senior

createMemoryRouter creates a router that stores history in memory rather than using the browser's History API. Use cases: (1) Testing — test route components without a browser environment, with full control over initial URL and navigation. (2) SSR — server has no browser history; memory router simulates it. (3) React Native — no browser history available. (4) Embedded widgets — routing within a widget that shouldn't affect the main page's URL. It accepts initialEntries and initialIndex to set up the starting state.

17. How does shouldRevalidate work and when would you use it? L6

shouldRevalidate is a function on route definitions that controls when a route's loader re-runs. By default, all active loaders revalidate after any action (mutation) or navigation. shouldRevalidate receives { currentUrl, nextUrl, formAction, formMethod, defaultShouldRevalidate } and returns a boolean.

Use it to: (1) prevent revalidation when only search params change (e.g., client-side filtering), (2) skip revalidation for routes whose data couldn't have been affected by the action, (3) optimize performance by reducing unnecessary network requests. Always call defaultShouldRevalidate as a fallback to maintain correct behavior for cases you haven't explicitly handled.

18. What is the location.key property and why is it important? L6

location.key is a unique string generated for each history entry. It's critical because the same URL can appear multiple times in the history stack (user visits /home, navigates away, then returns to /home). The key distinguishes these entries, enabling: (1) correct scroll restoration — each visit to the same URL may have different scroll positions, (2) cache keying — different data may be loaded for the same URL at different times, (3) transition tracking — knowing which specific history entry we're navigating to/from.

19. How would you implement type-safe routes in React Router? Architect

Type-safe routes ensure that link targets, params, and search params are validated at compile time. Approaches: (1) Route path constants — define paths as typed constants and use template literal types to extract param names. (2) Code generation — tools like @tanstack/router generate route types from the route tree. (3) Wrapper functions — create typed link() and navigate() helpers that enforce correct paths and params. (4) Zod schemas — validate search params with Zod and infer TypeScript types.

TanStack Router takes this furthest with fully type-safe route trees, params, search params, and loader data — all inferred from the route configuration. This is a significant advantage over React Router for large codebases where route typos cause runtime errors.

20. What is the difference between useNavigate and the redirect function? Senior

useNavigate is a React hook used inside components for programmatic navigation. redirect() is a utility function used inside loaders and actions (outside React components) that returns a Response object with a 302 status and Location header.

Use useNavigate for: event handlers, effects, conditional navigation in components. Use redirect() for: auth checks in loaders, post-mutation redirects in actions, any navigation logic that runs before the component renders. redirect() is the data router equivalent of <Navigate> — it works at the routing level, not the rendering level.

Data Loading & Code Splitting (Questions 21–35)

21. Explain the fetch waterfall problem in React applications. How do route loaders solve it? L6

Fetch waterfalls occur when components fetch data in useEffect — child components don't mount until their parent renders with data, creating sequential fetches. In a 3-level nested route, this means 3 sequential network requests.

Route loaders solve this by decoupling data fetching from rendering. When a URL matches, the router identifies all matched routes and runs their loaders in parallel before any component renders. The total loading time becomes max(loader1, loader2, loader3) instead of loader1 + loader2 + loader3. This is possible because the router knows the full route tree at match time, unlike components which only know about their own data needs.

22. How does defer() work in React Router? When should you use it vs awaiting all data? L6

defer() allows a loader to return a mix of resolved values and unresolved Promises. Resolved values are available immediately; Promises are streamed to the client and rendered via <Await> components wrapped in <Suspense> boundaries.

Await all data when: the page is meaningless without it (user identity, primary content, page title). Defer when: the data is supplementary (recommendations, analytics, comments), the fetch is slow and shouldn't block the critical path, or you want progressive loading (show the page structure immediately, fill in data as it arrives). The decision maps to "above the fold" content prioritization.

23. How do you handle race conditions during rapid navigation? L6

Race conditions occur when a user navigates rapidly — a slow fetch from navigation A may resolve after a fast fetch from navigation B, overwriting correct data with stale data. Solutions: (1) AbortController — cancel in-flight requests when a new navigation starts. React Router's loaders receive request.signal which is automatically aborted. (2) Request ID tracking — assign an ID to each request and only apply results if the ID matches the current request. (3) Ignore stale results — use a boolean flag in useEffect cleanup to mark results as stale.

React Router's data router handles this automatically — when a new navigation starts, all in-flight loaders from the previous navigation are aborted via their AbortSignal. This is one of the strongest arguments for using the data router pattern over manual useEffect fetching.

24. What is React.lazy and how does it work internally? L6

React.lazy takes a function that returns a dynamic import() Promise and returns a special lazy component type. Internally, it creates an object with $$typeof: REACT_LAZY_TYPE and an _init function. When React tries to render this component: (1) if the module is already loaded, it returns the default export, (2) if loading, it throws the Promise — Suspense catches this and shows the fallback, (3) if failed, it throws the error — an error boundary catches it.

Limitations: only works with default exports (named exports need a wrapper), doesn't work with SSR out of the box (need @loadable/component or React 18 streaming), and the import is only triggered when the component first renders (not when the route matches).

25. How would you implement retry logic for failed chunk loads in production? L6

Chunk load failures commonly occur during deployments when old HTML references chunk filenames that no longer exist. Implementation: (1) Wrap import() in a retry function with exponential backoff (e.g., 3 retries with 1s, 2s, 4s delays). (2) Detect ChunkLoadError specifically (check error.name or message). (3) On final failure, show a user-friendly message with a "Refresh" button that does window.location.reload() to get fresh HTML with correct chunk URLs. (4) Use content-hash filenames and immutable caching headers so old chunks remain available. (5) Keep previous deployment's chunks alive for a TTL period (e.g., 24 hours).

26. Compare route-based code splitting vs component-based code splitting. Senior

Route-based: split at route boundaries — each page is a separate chunk. Highest impact, lowest complexity. Users only download code for the current page. Natural split points that align with user navigation. Component-based: split individual components within a page (e.g., a heavy chart library, a rich text editor). More granular but adds complexity — each lazy component needs a Suspense boundary.

Start with route-based splitting (80/20 rule — biggest impact, least effort). Add component-based splitting only for heavy components that aren't needed on initial render (modals, below-fold content, rarely-used features). Over-splitting creates too many small chunks with HTTP overhead.

27. How do webpack magic comments work for chunk optimization? Senior

Webpack magic comments are special comments inside import() that control chunk behavior: webpackChunkName sets the output filename (for debugging and caching). webpackPrefetch: true adds a <link rel="prefetch"> tag — browser downloads the chunk during idle time (low priority). webpackPreload: true adds <link rel="preload"> — downloads immediately alongside the current chunk (high priority). Multiple imports with the same webpackChunkName are merged into a single chunk.

Prefetch is for chunks likely needed on the next navigation. Preload is for chunks needed during the current page render but loaded asynchronously. Misusing preload (loading chunks not needed immediately) wastes bandwidth and can delay critical resources.

28. What prefetching strategies exist for route chunks and how do you choose between them? L6

Strategies ranked by aggressiveness: (1) On hover/focus — prefetch when user hovers over or tabs to a link. ~100-300ms head start, minimal wasted bandwidth. Best default. (2) On viewport — prefetch when a link scrolls into view via IntersectionObserver. Good for below-fold navigation. (3) On idle — prefetch during requestIdleCallback. Speculative, may waste bandwidth. (4) On render — prefetch immediately when the page renders. Most aggressive, highest bandwidth cost.

Choose based on: navigation likelihood (hover = high intent), network conditions (save data mode = less aggressive), chunk size (small chunks = more aggressive OK), and user patterns (analytics data on common navigation paths).

29. How does Module Federation work and when is it appropriate? Architect

Module Federation (Webpack 5) allows separately built and deployed applications to share modules at runtime. A "host" application declares "remotes" (other apps), and remotes "expose" modules. Shared dependencies (React, etc.) are negotiated at runtime — only one copy loads, using the highest compatible version.

Appropriate when: multiple teams need independent deployment cycles, the application is large enough that a monolith slows development, and the organizational benefit outweighs the technical complexity. Not appropriate for: single-team projects, small-to-medium apps, or when you can use a monorepo with route-based splitting. The runtime complexity (version negotiation, remote failures, shared singleton management) is significant.

30. How do you analyze and optimize bundle size for a React application? L6

Analysis tools: webpack-bundle-analyzer (visual treemap), source-map-explorer (per-file breakdown), bundlephobia.com (package size lookup), import-cost VS Code extension (inline size display). Optimization checklist: (1) Route-based code splitting with React.lazy. (2) Tree shaking — ensure ES module imports, check sideEffects flag in package.json. (3) Replace heavy libraries (moment → date-fns, lodash → lodash-es or individual imports). (4) Remove unused code and dependencies. (5) Vendor chunk splitting for long-term caching. (6) Dynamic imports for heavy features (charts, editors). (7) Analyze and eliminate duplicate dependencies. (8) Use module/nomodule pattern for modern browsers.

31. What is stale-while-revalidate in the context of route data loading? L6

Stale-while-revalidate (SWR) is a caching strategy where stale (cached) data is returned immediately while fresh data is fetched in the background. For route data: (1) on first visit, fetch and cache the data, (2) on subsequent visits, return cached data instantly (no loading state) while refetching in the background, (3) when fresh data arrives, update the UI.

React Router's built-in revalidation is a form of SWR — after actions, it shows current data while revalidating all active loaders. For custom SWR, implement a cache layer in loaders that returns cached data immediately and triggers a background refresh. Libraries like TanStack Query provide this pattern out of the box with configurable stale times.

32. How does Promise.allSettled differ from Promise.all for data loading? Senior

Promise.all rejects immediately if any promise rejects — one failed fetch fails the entire batch. Promise.allSettled waits for all promises to settle (resolve or reject) and returns an array of results with status: 'fulfilled' or status: 'rejected'.

Use Promise.all when all data is required — if any fetch fails, the page can't render. Use Promise.allSettled when data is independent and partial failure is acceptable — e.g., user profile loads but recommendations fail; show the profile with a "recommendations unavailable" message rather than failing the entire page.

33. How would you implement optimistic navigation in React Router? L6

Optimistic navigation shows the new page's skeleton immediately while data loads, rather than showing a spinner on the current page. In React Router: (1) Use useNavigation() to detect state === 'loading'. (2) Show a progress bar or dim the current content. (3) For mutations, use useFetcher() and read fetcher.formData to show the optimistic value before the server confirms.

For more advanced optimistic UI: use fetcher.formData to derive the expected new state, render it immediately, and revert if the action fails. This gives instant feedback for operations like toggling a checkbox, liking a post, or reordering items.

34. How does Vite's chunk splitting differ from Webpack's? Senior

Vite uses Rollup for production builds, which has a different splitting philosophy. Rollup's default splitting is based on dynamic imports — each import() creates a chunk, and shared modules are automatically extracted. Vite's manualChunks option (in rollupOptions.output) allows explicit control, either as an object mapping chunk names to module arrays, or as a function that receives module IDs.

Key differences: Rollup tends to create fewer, larger chunks by default (better for HTTP/2). Webpack's SplitChunksPlugin has more granular control (minSize, maxSize, minChunks, cacheGroups). Vite doesn't support magic comments (webpackPrefetch, etc.) — use <link rel="modulepreload"> instead. Vite's dev server uses native ES modules (no bundling), so splitting behavior only applies to production builds.

35. Design a data loading architecture for a large dashboard application with 20+ routes. Architect

Architecture: (1) Use React Router data router with loaders for all routes — eliminates waterfalls. (2) Implement a caching layer (TanStack Query or custom) that loaders delegate to — provides SWR, deduplication, and background revalidation. (3) Critical data is awaited in loaders; supplementary data is deferred with defer(). (4) Shared data (user profile, permissions) loaded in root loader and accessed via useRouteLoaderData('root'). (5) shouldRevalidate on each route to minimize unnecessary refetches. (6) Prefetch on hover for likely navigation targets. (7) Error boundaries per route segment for graceful degradation.

Performance targets: navigation should feel instant (<200ms for cached data), initial load under 3s on 3G, and no loading spinners for back/forward navigation (cached data).

Navigation Patterns (Questions 36–42)

36. Compare component-based auth guards vs loader-based auth guards. Which is better and why? L6

Component-based (ProtectedRoute wrapper): checks auth during render, may flash protected content before redirect, doesn't work with SSR, and the redirect happens at the React level (Navigate component). Loader-based: checks auth before any component renders, no flash of protected content, works with SSR (loaders run on server), redirect is a Response object (works with progressive enhancement), and auth check can be combined with data fetching in a single round trip.

Loader-based is superior for data router apps. The auth check happens at the routing level, not the rendering level, which is architecturally cleaner and avoids the flash-of-content problem. Use a layout route with a loader for protecting groups of routes.

37. How do you implement modal routes that support deep linking? L6

Modal routes give modals their own URL while keeping the background page visible. Implementation: (1) When opening a modal from a list, pass the current location as state.backgroundLocation. (2) In the root component, check for backgroundLocation. If present, render routes for the background location AND render a second <Routes> for the modal. (3) Direct URL access (no background location) renders the full page version.

This provides: deep linkable modals, proper back button behavior (closes modal), SEO (URL is crawlable), and progressive enhancement (direct URL shows full page). Next.js App Router provides this natively via intercepting routes ((.)photo/[id]).

38. How do you generate breadcrumbs from a route hierarchy? Senior

Use React Router's useMatches() hook combined with route handle metadata. Each route defines a handle.breadcrumb property (string or function). useMatches() returns all matched routes from root to current, each with their pathname and data. Map over matches, filter those with breadcrumb handles, and render links. For dynamic breadcrumbs (e.g., showing a product name), the handle function receives the match object including loader data.

This approach is declarative, type-safe, and automatically updates when the route tree changes. It also works with nested routes — each segment contributes its own breadcrumb.

39. What state should live in the URL vs component state vs global state? Architect

URL state (search params, path params): anything that should survive refresh, be shareable via link, or be bookmarkable. Examples: filters, pagination, sort order, selected tab, view mode, search query. Component state: ephemeral UI state that doesn't need persistence. Examples: dropdown open/closed, hover state, form input before submission, animation state. Global state (context/store): cross-cutting concerns shared across routes. Examples: auth status, theme, locale, toast notifications.

Decision framework: "If a user shares this URL, should the recipient see the same view?" If yes → URL state. "Does this state matter after the component unmounts?" If no → component state. "Do multiple unrelated components need this?" If yes → global state.

40. How do you implement useBlocker for preventing navigation with unsaved changes? Senior

React Router's useBlocker hook accepts a function that receives { currentLocation, nextLocation } and returns a boolean (true = block). When blocked, blocker.state becomes 'blocked', and you can call blocker.proceed() to continue or blocker.reset() to cancel. Show a confirmation dialog when blocked.

Important: useBlocker only blocks client-side navigation. For full page unloads (closing tab, external navigation), also add a beforeunload event listener. Combine both for complete coverage. Also consider auto-saving as an alternative — it eliminates the need for blocking entirely.

41. How do you encode complex filter state in URLs without making them unreadable? L6

Strategies: (1) Flat key-value pairs — ?category=electronics&sort=price&page=2. Simple, readable, works for most cases. (2) Comma-separated arrays — ?tags=react,hooks,typescript. Compact for multi-value filters. (3) Range notation — ?price=10-100. Good for numeric ranges. (4) Omit defaults — don't include params that match default values. Keeps URLs clean. (5) Compressed encoding — for very complex state (dashboard layouts), use base64-encoded compressed JSON. Last resort — URLs become opaque.

Best practice: use human-readable key-value pairs for common filters, omit defaults, and only compress when the state is too complex for flat params. Always validate and sanitize URL params on parse — they're user input.

42. How do you implement tab-based navigation that preserves tab state? Senior

Two approaches: (1) Search params — ?tab=posts. Tab state is in the URL, shareable and bookmarkable. Tab content re-renders on switch. Good for tabs with independent data. (2) Nested routes — /users/42/posts. Each tab is a child route with its own loader. Better for tabs with distinct data needs and deep linking.

To preserve tab state across switches (e.g., scroll position within a tab), either: (a) keep all tab panels mounted but hidden (CSS display: none), (b) cache tab state in a ref or context, or (c) use keepPreviousData in a data fetching library. Nested routes with React Router's layout preservation naturally keep parent state across tab switches.

Next.js App Router (Questions 43–52)

43. Explain the Next.js App Router's file-system routing conventions. What are the special files? Senior

The App Router maps the filesystem to routes: directories create URL segments, and special files define behavior. page.tsx makes a segment publicly accessible. layout.tsx wraps children and persists across navigations. loading.tsx creates an automatic Suspense boundary. error.tsx creates an error boundary (must be a Client Component). not-found.tsx handles 404s. template.tsx is like layout but re-mounts on navigation. default.tsx is the fallback for parallel route slots. route.ts creates an API endpoint.

Naming conventions: [param] for dynamic segments, [...slug] for catch-all, [[...slug]] for optional catch-all, (group) for route groups (no URL segment), @slot for parallel routes, (.)path for intercepting routes.

44. What is the difference between a layout and a template in the App Router? L6

Layouts persist across navigations — they don't re-render or re-mount when a child route changes. State, effects, and DOM are preserved. Templates re-mount on every navigation — they create a new component instance, resetting all state and re-running effects.

Use layouts for: persistent navigation, sidebars, headers, and any UI that should maintain state across route changes. Use templates for: per-page enter/exit animations (useEffect runs on each navigation), analytics page view tracking, resetting form state on navigation, or any case where you need fresh state per route.

45. How do parallel routes work in Next.js? What problem do they solve? L6

Parallel routes (defined with @slot directories) allow rendering multiple pages simultaneously in the same layout, each with independent loading and error states. The layout receives slots as props alongside children.

They solve: (1) dashboard layouts where multiple panels load independently — if analytics is slow, the team panel still renders. (2) Conditional rendering based on auth — show different content for logged-in vs anonymous users using the same URL. (3) Modal patterns — a @modal slot renders on top of the main content. Each slot has its own loading.tsx and error.tsx, so failures are isolated.

46. Explain intercepting routes in Next.js. How do they enable the Instagram-style photo modal pattern? L6

Intercepting routes allow loading a route from another part of the app within the current layout. Convention: (.) intercepts same level, (..) one level up, (...) from root. For the photo modal: clicking a photo in the feed intercepts /photo/[id] and renders it as a modal (via a @modal parallel slot). Direct URL access to /photo/[id] renders the full page (not intercepted).

This provides: (1) modal opens from feed with background preserved, (2) direct URL shows full page, (3) refresh while modal open shows full page, (4) back button closes modal, (5) shared URL shows full page to recipient. The interception only happens for soft navigations (client-side); hard navigations (direct URL, refresh) bypass interception.

47. How does Next.js middleware work and what are its limitations? L6

Middleware runs on the Edge Runtime before every matched request. It can: redirect, rewrite, set headers/cookies, and return responses. Defined in middleware.ts at the project root with a config.matcher to specify which paths it applies to.

Limitations: (1) Runs on Edge Runtime — limited Node.js API support (no fs, no native modules, limited crypto). (2) 1MB size limit for the middleware bundle. (3) Can't access the database directly (no Node.js drivers). (4) Can't modify the response body (only headers/cookies). (5) Runs before static file serving — must exclude static assets in the matcher. (6) Cold start latency on some platforms. Best used for: auth redirects, A/B testing, geo-routing, header manipulation.

48. Compare Next.js Pages Router vs App Router. When would you choose each? Architect

Pages Router: mature, stable, well-documented, simpler mental model (all Client Components), getServerSideProps/getStaticProps for data. Choose for: existing projects, teams unfamiliar with Server Components, simpler apps, or when stability is paramount.

App Router: Server Components by default (smaller bundles), nested layouts (preserved state), streaming, parallel routes, intercepting routes, Server Actions, and loading.tsx/error.tsx conventions. Choose for: new projects, apps needing nested layouts, performance-critical apps (Server Components reduce client JS), and teams ready for the Server Component paradigm.

The App Router is the future of Next.js, but the Pages Router remains supported. Migration can be incremental — both routers can coexist in the same project.

49. How do Server Actions work with routing in Next.js? L6

Server Actions are async functions marked with 'use server' that run on the server. They can be passed as the action prop to <form> elements. When the form submits: (1) with JS enabled, the form data is sent via fetch to the server, the action runs, and the page revalidates without a full reload. (2) Without JS, the form submits as a traditional POST request — progressive enhancement.

Integration with routing: revalidatePath('/path') invalidates the cache for a specific route. revalidateTag('tag') invalidates all fetches with that cache tag. redirect('/path') navigates after the action. Server Actions replace the need for separate API routes for mutations in most cases.

50. How does generateStaticParams work and how does it relate to ISR? Senior

generateStaticParams is the App Router equivalent of getStaticPaths. It returns an array of param objects that should be pre-rendered at build time. For /blog/[slug], it returns [{ slug: 'post-1' }, { slug: 'post-2' }]. Pages not in the list are generated on-demand (ISR behavior by default).

Relation to ISR: pages generated by generateStaticParams are static at build time. On-demand pages are generated on first request and cached. Revalidation happens via: (1) time-based — export const revalidate = 3600 (revalidate every hour), (2) on-demand — revalidatePath() or revalidateTag() in Server Actions or API routes. This provides the best of static (fast) and dynamic (fresh) rendering.

51. How does the App Router handle caching and when does it cause issues? L6

The App Router has four caching layers: (1) Request Memoization — deduplicates identical fetch calls within a single render. (2) Data Cache — persists fetch results across requests (opt out with cache: 'no-store'). (3) Full Route Cache — caches rendered HTML and RSC payload for static routes. (4) Router Cache — client-side cache of visited routes (30s for dynamic, 5min for static).

Common issues: stale data after mutations (need explicit revalidatePath/revalidateTag), router cache showing old data on back navigation, data cache persisting across deployments. The aggressive caching defaults prioritize performance but require understanding the invalidation mechanisms.

52. Design the routing architecture for a large SaaS application using Next.js App Router. Architect

Route structure: (1) Route groups: (marketing) for public pages (SSG), (auth) for login/signup, (app) for authenticated dashboard. (2) Root layout with providers (theme, auth). (3) (app)/layout.tsx with auth check, sidebar, and header. (4) Parallel routes: @modal for dialogs, @notifications for real-time panel.

Data strategy: Server Components for all data fetching. Shared data (user, org) in root layout loader. Per-page data in page-level Server Components. Client Components only for interactivity (forms, real-time features). Server Actions for mutations with revalidatePath.

Performance: Static generation for marketing pages. Streaming for dashboard (loading.tsx per section). Middleware for auth redirects and tenant routing. Edge rendering for latency-sensitive pages.
