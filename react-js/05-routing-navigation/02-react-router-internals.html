1. React Router v6 Architecture

React Router v6 is a complete rewrite from v5. The core insight is that routing is a tree-matching problem: given a URL and a tree of route definitions, find the branch of routes that matches and render them as nested components.

Architect Insight

The shift from JSX route configuration (<Route path="/">) to plain objects is significant. Object-based routes enable: (1) static analysis and type-safe route generation, (2) server-side route matching without React, (3) build-time route manifest generation, and (4) framework-level optimizations like Remix's parallel data loading. This is why createBrowserRouter is the recommended API over <BrowserRouter>.

4. The Matching Algorithm

React Router v6 uses a ranked route matching algorithm. Unlike v5 (which used first-match ordering), v6 scores every route and picks the most specific match.

// Internal: how matchRoutes works (simplified)
function matchRoutes(routes, pathname) {
  // 1. Flatten route tree into all possible branch paths
  const branches = flattenRoutes(routes);

  // 2. Score and rank each branch
  rankRouteBranches(branches);

  // 3. Try matching from highest-ranked to lowest
  for (const branch of branches) {
    const matches = matchRouteBranch(branch, pathname);
    if (matches) return matches;
  }
  return null;
}

// Scoring constants (from React Router source)
const paramRe = /^:\w+$/;
const dynamicSegmentValue = 3;
const indexRouteValue = 2;
const emptySegmentValue = 1;
const staticSegmentValue = 10;
const splatPenalty = -2;

function computeScore(path, index) {
  const segments = path.split('/').filter(Boolean);
  let initialScore = segments.length;

  if (segments.some(s => s === '*')) {
    initialScore += splatPenalty;
  }

  if (index) {
    initialScore += indexRouteValue;
  }

  return segments
    .filter(s => s !== '*')
    .reduce((score, segment) =>
      score + (paramRe.test(segment) ? dynamicSegmentValue : staticSegmentValue),
      initialScore
    );
}
Matching '/users/new' against route tree: Route Tree: Flattened Branches (scored): / /, /users, /users/new → score 33 ✓ WINS ├── users /, /users, /users/:userId → score 26 │ ├── new /, /users → score 23 │ ├── :userId /, /files/* → score 9 │ └── index └── files/* The branch [/, /users, /users/new] wins because 'new' is a static segment (10 points) vs ':userId' dynamic (3 points)

5. Nested Routes & Outlet

Nested routes are React Router's most powerful feature. Each matched route renders inside its parent's <Outlet />, creating a component hierarchy that mirrors the URL hierarchy.

// Route configuration
const routes = [
  {
    path: '/',
    element: <RootLayout />,     // always renders
    children: [
      {
        path: 'dashboard',
        element: <DashboardLayout />, // renders inside RootLayout's Outlet
        children: [
          { index: true, element: <DashboardHome /> },
          { path: 'analytics', element: <Analytics /> },
        ],
      },
    ],
  },
];

// RootLayout — renders matched child via Outlet
function RootLayout() {
  return (
    <div>
      <Header />
      <main>
        <Outlet />  {/* DashboardLayout renders here */}
      </main>
      <Footer />
    </div>
  );
}

// DashboardLayout — also has an Outlet for its children
function DashboardLayout() {
  return (
    <div>
      <DashboardNav />
      <Outlet />  {/* Analytics or DashboardHome renders here */}
    </div>
  );
}

  // location.key       — 'ae4cz2j' (unique per history entry)

// useParams — dynamic route parameters
function UserProfile() {
  const { userId } = useParams();  // from path ':userId'
  // Always strings — parse as needed: Number(userId)

// useSearchParams — URLSearchParams wrapper with React state
function UsersList() {
  const [searchParams, setSearchParams] = useSearchParams();
  const page = Number(searchParams.get('page') ?? '1');

  function nextPage() {
    setSearchParams(prev => {
      prev.set('page', String(page + 1));
      return prev;
    });
  }
}

// useNavigation — track pending navigation state
function GlobalSpinner() {
  const navigation = useNavigation();
  // navigation.state: 'idle' | 'loading' | 'submitting'
  // navigation.location: the location being navigated to
  // navigation.formData: FormData if submitting
  return navigation.state !== 'idle' ? <Spinner /> : null;
}

7. Data Router: Loaders & Actions

The data router pattern (introduced in v6.4) co-locates data fetching with route definitions. Loaders fetch data before rendering; actions handle mutations (form submissions).

// Loader — runs before the route component renders
async function userProfileLoader({ params, request }) {
  // params.userId from ':userId' in route path
  const url = new URL(request.url);
  const tab = url.searchParams.get('tab') ?? 'overview';

  const user = await fetchUser(params.userId);
  if (!user) {
    throw new Response('Not Found', { status: 404 });
  }
  return { user, tab };
}

// Action — handles form submissions (mutations)
async function userProfileAction({ params, request }) {
  const formData = await request.formData();
  const intent = formData.get('intent');

  if (intent === 'updateName') {
    await updateUser(params.userId, {
      name: formData.get('name'),
    });
    return { success: true };
  }

  if (intent === 'delete') {
    await deleteUser(params.userId);
    return redirect('/users');
  }
}

// Component — uses loader data
function UserProfile() {
  const { user, tab } = useLoaderData();
  const actionData = useActionData();  // result from action
  const navigation = useNavigation();

  return (
    <div>
      <h1>{user.name}</h1>
      <Form method="post">
        <input name="name" defaultValue={user.name} />
        <button name="intent" value="updateName"
          disabled={navigation.state === 'submitting'}>
          {navigation.state === 'submitting' ? 'Saving...' : 'Save'}
        </button>
      </Form>
    </div>
  );
}
Architect Insight

The loader/action pattern mirrors how traditional web frameworks (Rails, Django) handle requests: loaders are GET handlers, actions are POST handlers. This is intentional — React Router's data APIs are designed to work with the web platform's request/response model, making progressive enhancement and eventual SSR migration straightforward. The request parameter is a real Request object, and loaders can return Response objects.

8. Route-Level Error Boundaries

// Each route can have its own errorElement
const routes = [
  {
    path: '/',
    element: <RootLayout />,
    errorElement: <RootError />,  // catches errors from any child route
    children: [
      {
        path: 'users/:id',
        element: <UserProfile />,
        errorElement: <UserError />,  // catches errors only in this route
        loader: async ({ params }) => {
          const res = await fetch(`/api/users/${params.id}`);
          if (!res.ok) throw new Response('Not found', { status: 404 });
          return res.json();
        },
      },
    ],
  },
];

// Error component — uses useRouteError
function UserError() {
  const error = useRouteError();

  if (isRouteErrorResponse(error)) {
    // Thrown Response objects
    return (
      <div>
        <h1>{error.status} {error.statusText}</h1>
        <p>{error.data}</p>
      </div>
    );
  }

  // Unexpected errors
  return <div>Something went wrong: {error.message}</div>;
}
Error Bubbling

Errors bubble up the route tree to the nearest errorElement. If a child route throws and has no errorElement, the error propagates to its parent route's errorElement. If no route has an errorElement, the entire app crashes. Always define at least a root-level errorElement.

9. Index Routes & Relative Paths

9.1 Index Routes

An index route is a child route with no path that renders at the parent's URL. It fills the parent's <Outlet /> when no other child matches.

const routes = [
  {
    path: 'dashboard',
    element: <DashboardLayout />,
    children: [
      { index: true, element: <DashboardHome /> },  // renders at /dashboard
      { path: 'analytics', element: <Analytics /> },  // renders at /dashboard/analytics
      { path: 'settings', element: <Settings /> },   // renders at /dashboard/settings
    ],
  },
];

// Without index route, /dashboard would render DashboardLayout
// with an empty Outlet — usually not what you want

9.2 Relative vs Absolute Paths

// Relative paths — resolved against the current route's path
// In a component rendered at /dashboard/users/42:

<Link to="edit">        // → /dashboard/users/42/edit
<Link to="../settings"> // → /dashboard/users/settings (up one level)
<Link to=".">           // → /dashboard/users/42 (current route)
<Link to="..">          // → /dashboard/users (parent route)

// Absolute paths — always from root
<Link to="/dashboard">  // → /dashboard (always)

// IMPORTANT: '..' navigates to the parent ROUTE, not the parent URL segment
// If current route is /dashboard/users/:id and parent route is /dashboard,
// then '..' goes to /dashboard (not /dashboard/users)
Relative Path Resolution

React Router v6 resolves relative paths against the route hierarchy, not the URL segments. This is a common source of confusion. .. means "parent route" not "remove one URL segment." This distinction matters when routes don't have a 1:1 mapping with URL segments (e.g., pathless layout routes).
