1. The Waterfall Problem

The most common performance issue in React applications is the fetch waterfall: components fetch data in their useEffect, but child components don't mount (and thus don't start fetching) until their parent has finished rendering with data.

Fetch Waterfall (fetch-on-render): Time ──────────────────────────────────────────────────▶ Layout: [──render──][──fetch──────][──render with data──] │ Dashboard: [──render──][──fetch──────][──render──] │ UserList: [──render──][──fetch──] Total: t_layout + t_dashboard + t_userlist (sequential!) Parallel Loading (fetch-then-render): Time ──────────────────────────────────────────────────▶ Layout: [──fetch──────]──┐ Dashboard: [──fetch──────]──┼── all start simultaneously UserList: [──fetch──]──────┘ │ All: [──render everything──] Total: max(t_layout, t_dashboard, t_userlist) (parallel!) 
// ❌ Fetch waterfall pattern (fetch-on-render)
function Dashboard() {
  const [user, setUser] = useState(null);

  useEffect(() => {
    fetchUser().then(setUser);  // fetch starts after mount
  }, []);

  if (!user) return <Spinner />;

  // UserPosts won't mount until user is loaded
  // → UserPosts' fetch starts AFTER Dashboard's fetch completes
  return <UserPosts userId={user.id} />;
}

function UserPosts({ userId }) {
  const [posts, setPosts] = useState(null);

  useEffect(() => {
    fetchPosts(userId).then(setPosts);  // waterfall: waits for parent
  }, [userId]);

  if (!posts) return <Spinner />;
  return posts.map(p => <Post key={p.id} post={p} />);
}
The Real Cost

In a 3-level nested route with each fetch taking 200ms, a waterfall costs 600ms total. With parallel loading, it's 200ms. At scale with slow networks, waterfalls can add seconds to page transitions. This is the single most impactful performance issue in React applications and a guaranteed L6 interview topic.

2. Route-Level Data Loading

React Router's data router pattern eliminates waterfalls by decoupling data fetching from component rendering. Loaders run in parallel before any component renders.

// Route configuration with loaders
const routes = [
  {
    path: '/',
    element: <RootLayout />,
    loader: async () => {
      return { user: await fetchCurrentUser() };
    },
    children: [
      {
        path: 'dashboard',
        element: <Dashboard />,
        loader: async () => {
          return { stats: await fetchDashboardStats() };
        },
        children: [
          {
            path: 'projects/:projectId',
            element: <ProjectDetail />,
            loader: async ({ params }) => {
              return { project: await fetchProject(params.projectId) };
            },
          },
        ],
      },
    ],
  },
];

// When navigating to /dashboard/projects/42:
// 1. Router matches: [RootLayout, Dashboard, ProjectDetail]
// 2. ALL three loaders run in parallel
// 3. Once all resolve, all three components render with data

function ProjectDetail() {
  const { project } = useLoaderData();  // data is already loaded
  return <h1>{project.name}</h1>;
}

3. Parallel Data Fetching

3.1 Within a Single Loader

// ❌ Sequential fetches within a loader
async function dashboardLoader() {
  const user = await fetchUser();           // 200ms
  const stats = await fetchStats();          // 150ms
  const notifications = await fetchNotifications(); // 100ms
  return { user, stats, notifications };    // Total: 450ms
}

// ✅ Parallel fetches with Promise.all
async function dashboardLoader() {
  const [user, stats, notifications] = await Promise.all([
    fetchUser(),           // 200ms ─┐
    fetchStats(),          // 150ms ─┼── all parallel
    fetchNotifications(),  // 100ms ─┘
  ]);
  return { user, stats, notifications };    // Total: 200ms (max)
}

// ✅ Promise.allSettled for independent data (partial failure OK)
async function dashboardLoader() {
  const results = await Promise.allSettled([
    fetchUser(),
    fetchStats(),
    fetchNotifications(),
  ]);

  return {
    user: results[0].status === 'fulfilled' ? results[0].value : null,
    stats: results[1].status === 'fulfilled' ? results[1].value : null,
    notifications: results[2].status === 'fulfilled' ? results[2].value : [],
  };
}

4. defer() and Streaming

defer() allows loaders to return a mix of resolved data and promises. Critical data blocks rendering; deferred data streams in later with Suspense fallbacks.

import { defer, useLoaderData, Await } from 'react-router-dom';

// Loader: critical data awaited, non-critical deferred
async function dashboardLoader() {
  const userPromise = fetchUser();  // critical — await it

  return defer({
    user: await userPromise,          // resolved before render
    stats: fetchStats(),              // deferred — streams in later
    recommendations: fetchRecommendations(), // deferred
  });
}

// Component: uses Await + Suspense for deferred data
function Dashboard() {
  const { user, stats, recommendations } = useLoaderData();

  return (
    <div>
      <h1>Welcome, {user.name}</h1>  {/* renders immediately */}

      <Suspense fallback={<StatsPlaceholder />}>
        <Await resolve={stats} errorElement={<StatsError />}>
          {(resolvedStats) => <StatsPanel data={resolvedStats} />}
        </Await>
      </Suspense>

      <Suspense fallback={<RecsPlaceholder />}>
        <Await resolve={recommendations}>
          {(recs) => <Recommendations items={recs} />}
        </Await>
      </Suspense>
    </div>
  );
}
defer() Timeline: Time ──────────────────────────────────────────────────▶ fetchUser(): [████████]──resolved──▶ render user immediately fetchStats(): [████████████████]──▶ stream into Suspense boundary fetchRecommendations(): [████████████████████████]──▶ stream in last Page: [loading] → [user + placeholders] → [user + stats + placeholder] → [complete] User sees content progressively instead of waiting for everything
Architect Insight

The decision of what to await vs defer is an architectural one. Critical data (user identity, page title, primary content) should be awaited — the page is meaningless without it. Secondary data (analytics, recommendations, comments) should be deferred — the page is usable without it. This maps to the concept of "above the fold" content prioritization.

6. Optimistic Navigation

Optimistic navigation shows the new page immediately while data loads in the background, rather than showing a loading spinner on the current page.

// React Router provides optimistic UI via useNavigation
function Layout() {
  const navigation = useNavigation();

  return (
    <div>
      {/* Show progress bar during navigation */}
      {navigation.state === 'loading' && <ProgressBar />}

      {/* Dim current content while loading next page */}
      <div style={{
        opacity: navigation.state === 'loading' ? 0.7 : 1,
        transition: 'opacity 200ms',
      }}>
        <Outlet />
      </div>
    </div>
  );
}

// Optimistic updates for mutations
function TodoItem({ todo }) {
  const fetcher = useFetcher();

  // Use optimistic value while submission is in flight
  const isComplete = fetcher.formData
    ? fetcher.formData.get('complete') === 'true'
    : todo.complete;

  return (
    <fetcher.Form method="post">
      <input type="hidden" name="complete" value={!todo.complete} />
      <button>
        {isComplete ? '✓' : '○'} {todo.title}
      </button>
    </fetcher.Form>
  );
}

7. Prefetching Strategies

// Strategy 1: Prefetch on hover (intent-based)
function PrefetchLink({ to, children }) {
  const prefetchedRef = useRef(false);

  function handleMouseEnter() {
    if (!prefetchedRef.current) {
      prefetchedRef.current = true;
      // Prefetch the route's loader data
      router.prefetch(to);
      // Also prefetch the route's code chunk
      import(/* webpackPrefetch: true */ `./pages/${to}`);
    }
  }

  return (
    <Link to={to} onMouseEnter={handleMouseEnter}>
      {children}
    </Link>
  );
}

// Strategy 2: Prefetch on viewport visibility
function ViewportPrefetchLink({ to, children }) {
  const ref = useRef(null);

  useEffect(() => {
    const observer = new IntersectionObserver(
      ([entry]) => {
        if (entry.isIntersecting) {
          router.prefetch(to);
          observer.disconnect();
        }
      },
      { rootMargin: '200px' }
    );
    if (ref.current) observer.observe(ref.current);
    return () => observer.disconnect();
  }, [to]);

  return <Link ref={ref} to={to}>{children}</Link>;
}

// Strategy 3: Prefetch on idle (speculative)
function usePrefetchOnIdle(routes) {
  useEffect(() => {
    if ('requestIdleCallback' in window) {
      const id = requestIdleCallback(() => {
        routes.forEach(route => router.prefetch(route));
      });
      return () => cancelIdleCallback(id);
    }
  }, [routes]);
}
Strategy Trigger Tradeoff
On hover Mouse enters link ~100-300ms head start; wasted if user doesn't click
On focus Tab navigation to link Accessibility-friendly; similar to hover
On viewport Link scrolls into view Good for below-fold links; more bandwidth
On idle Browser idle time Speculative; may waste bandwidth
On render Component mounts Most aggressive; highest bandwidth cost

8. Stale-While-Revalidate for Routes

// React Router's built-in revalidation
// After an action (form submission), all active loaders revalidate
async function updateTodoAction({ request }) {
  const formData = await request.formData();
  await updateTodo(formData);
  // React Router automatically revalidates all active loaders
  // No manual cache invalidation needed
  return { ok: true };
}

// shouldRevalidate — fine-grained revalidation control
{
  path: 'dashboard',
  loader: dashboardLoader,
  shouldRevalidate: ({ currentUrl, nextUrl, formAction, defaultShouldRevalidate }) => {
    // Don't revalidate if only search params changed
    if (currentUrl.pathname === nextUrl.pathname) {
      return false;
    }
    return defaultShouldRevalidate;
  },
}

// Custom stale-while-revalidate with cache
const cache = new Map();

async function cachedLoader({ request }) {
  const key = request.url;
  const cached = cache.get(key);

  if (cached && Date.now() - cached.timestamp < 30000) {
    // Return stale data immediately, revalidate in background
    fetch(request.url)
      .then(r => r.json())
      .then(data => cache.set(key, { data, timestamp: Date.now() }));
    return cached.data;
  }

  const data = await fetch(request.url).then(r => r.json());
  cache.set(key, { data, timestamp: Date.now() });
  return data;
}

9. Race Condition Handling

Navigation race conditions occur when a user navigates rapidly — starting a new navigation before the previous one completes. Without proper handling, stale data from an earlier navigation can overwrite fresh data.

// ❌ Race condition with useEffect (no cancellation)
function UserProfile({ userId }) {
  const [user, setUser] = useState(null);

  useEffect(() => {
    fetchUser(userId).then(setUser);
    // If userId changes before fetch completes,
    // old fetch may resolve AFTER new fetch → stale data!
  }, [userId]);
}

// ✅ Race condition handling with AbortController
function UserProfile({ userId }) {
  const [user, setUser] = useState(null);

  useEffect(() => {
    const controller = new AbortController();

    fetchUser(userId, { signal: controller.signal })
      .then(setUser)
      .catch(err => {
        if (err.name !== 'AbortError') throw err;
      });

    return () => controller.abort();  // cancel on unmount/re-render
  }, [userId]);
}

// ✅ React Router handles this automatically
// Loaders receive request.signal (AbortSignal)
async function userLoader({ params, request }) {
  const res = await fetch(`/api/users/${params.id}`, {
    signal: request.signal,  // automatically aborted on new navigation
  });
  return res.json();
}
Race Condition Scenario: User clicks: User A → User B → User C (rapidly) Without cancellation: fetch(A) ─────────────────────────▶ resolve(A) → setUser(A) ← STALE! fetch(B) ──────────▶ resolve(B) → setUser(B) ← STALE! fetch(C) ───▶ resolve(C) → setUser(C) ← correct but overwritten Final state: User A (wrong!) With AbortController: fetch(A) ──── ABORTED fetch(B) ──── ABORTED fetch(C) ───▶ resolve(C) → setUser(C) ← correct Final state: User C (correct!)
Critical Pattern

React Router's data router handles race conditions automatically by aborting in-flight loaders when a new navigation starts. If you're using useEffect for data fetching (without a data router), you MUST handle cancellation yourself via AbortController or a boolean flag. This is one of the strongest arguments for using the data router pattern.

10. Server-Side Data Loading

// Next.js App Router — Server Component data loading
// Data fetches happen on the server, zero client JS
async function DashboardPage() {
  const [user, stats] = await Promise.all([
    fetchUser(),
    fetchStats(),
  ]);

  return (
    <div>
      <h1>Welcome, {user.name}</h1>
      <StatsPanel data={stats} />
      <Suspense fallback={<CommentsPlaceholder />}>
        <Comments />  {/* streams in */}
      </Suspense>
    </div>
  );
}

// Remix — loader runs on server, hydrates on client
export async function loader({ request }) {
  const session = await getSession(request.headers.get('Cookie'));
  const user = await db.user.findUnique({
    where: { id: session.userId },
  });
  return json({ user });
}
