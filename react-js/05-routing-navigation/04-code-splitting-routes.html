1. Why Route-Based Code Splitting

Without code splitting, the entire application is bundled into a single JavaScript file. Users download code for every page on initial load, even pages they may never visit. Route-based splitting is the highest-impact optimization because routes represent natural boundaries — users only need the code for the page they're viewing.

Without Code Splitting: main.js (800KB) ├── HomePage code (50KB) ├── Dashboard code (200KB) ├── Settings code (80KB) ├── Admin Panel code (150KB) ├── Analytics code (120KB) ├── Shared libraries (200KB) └── User visits only HomePage → downloaded 750KB of unused code With Route-Based Splitting: main.js (200KB) ← shared code + current route ├── Shared libraries (200KB) └── HomePage chunk (50KB) ← loaded immediately dashboard.chunk.js (200KB) ← loaded when user navigates settings.chunk.js (80KB) ← loaded on demand admin.chunk.js (150KB) ← loaded on demand analytics.chunk.js (120KB) ← loaded on demand Initial load: 250KB vs 800KB (69% reduction)
Architect Insight

Route-based splitting is the "80/20 rule" of bundle optimization: it provides the largest reduction in initial load with the least complexity. Component-level splitting (lazy-loading individual components within a page) has diminishing returns and adds complexity. Start with route-level splitting, measure, then optimize further only if needed.

2. Dynamic import() Internals

The import() expression is an ECMAScript proposal (now stage 4) that returns a Promise resolving to the module's namespace object. Bundlers use it as a split point to create separate chunks.

// Static import — bundled into main chunk
import { Dashboard } from './Dashboard';

// Dynamic import — creates a separate chunk
const DashboardModule = await import('./Dashboard');
const Dashboard = DashboardModule.default;

// What the bundler generates (conceptual Webpack output):
// 1. A separate file: dashboard.chunk.js containing Dashboard code
// 2. In main.js, import('./Dashboard') becomes:
__webpack_require__.e("dashboard")  // load chunk via script tag
  .then(__webpack_require__.bind(null, "./src/Dashboard.tsx"))
  .then(module => module.default);

// The runtime creates a <script> tag:
// <script src="/static/js/dashboard.chunk.js"></script>
// When loaded, the chunk registers its modules with the webpack runtime

2.1 Webpack Magic Comments

// Named chunks — control output filename
const Dashboard = lazy(() => import(
  /* webpackChunkName: "dashboard" */
  './Dashboard'
));
// Output: dashboard.chunk.js (instead of 123.chunk.js)

// Prefetch — download during idle time (low priority)
const Settings = lazy(() => import(
  /* webpackPrefetch: true */
  './Settings'
));
// Adds: <link rel="prefetch" href="settings.chunk.js"> to <head>

// Preload — download immediately (high priority)
const CriticalFeature = lazy(() => import(
  /* webpackPreload: true */
  './CriticalFeature'
));
// Adds: <link rel="preload" href="critical.chunk.js"> to <head>

// Chunk grouping — merge multiple imports into one chunk
const AdminUsers = lazy(() => import(
  /* webpackChunkName: "admin" */ './admin/Users'
));
const AdminSettings = lazy(() => import(
  /* webpackChunkName: "admin" */ './admin/Settings'
));
// Both bundled into single admin.chunk.js

3. React.lazy Deep Dive

React.lazy is React's built-in mechanism for rendering a dynamically imported component. It works with Suspense to show a fallback while the chunk loads.

// Basic usage
const Dashboard = lazy(() => import('./Dashboard'));

// What React.lazy does internally (simplified):
function lazy(factory) {
  let status = 'pending';
  let result;
  let thenable = null;

  return {
    $$typeof: REACT_LAZY_TYPE,
    _init(payload) {
      if (status === 'resolved') return result;
      if (status === 'rejected') throw result;

      if (!thenable) {
        thenable = factory();
        thenable.then(
          module => { status = 'resolved'; result = module.default; },
          error => { status = 'rejected'; result = error; }
        );
      }

      // Throw the promise — Suspense catches it
      throw thenable;
    },
    _payload: null,
  };
}

// React.lazy ONLY works with default exports
// For named exports, wrap the import:
const UserProfile = lazy(() =>
  import('./components').then(mod => ({ default: mod.UserProfile }))
);
React.lazy Limitations

React.lazy only works for client-side rendering. It does NOT work with SSR out of the box — the server can't wait for dynamic imports. For SSR, use @loadable/component or React 18's built-in SSR streaming with lazy. Also, React.lazy only supports default exports; named exports require a wrapper.

4. Webpack/Vite Chunk Strategies

4.1 Webpack SplitChunksPlugin

// webpack.config.js
module.exports = {
  optimization: {
    splitChunks: {
      chunks: 'all',
      cacheGroups: {
        // Vendor chunk: node_modules shared across routes
        vendor: {
          test: /[\\/]node_modules[\\/]/,
          name: 'vendor',
          chunks: 'all',
          priority: 20,
        },
        // React ecosystem in its own chunk (rarely changes → long cache)
        react: {
          test: /[\\/]node_modules[\\/](react|react-dom|react-router)[\\/]/,
          name: 'react-vendor',
          chunks: 'all',
          priority: 30,
        },
        // Common code shared by 2+ routes
        common: {
          minChunks: 2,
          name: 'common',
          chunks: 'all',
          priority: 10,
          reuseExistingChunk: true,
        },
      },
    },
  },
};

4.2 Vite Chunk Strategy

// vite.config.ts
import { defineConfig } from 'vite';

export default defineConfig({
  build: {
    rollupOptions: {
      output: {
        manualChunks: {
          // Group React into its own chunk
          'react-vendor': ['react', 'react-dom', 'react-router-dom'],
          // Group charting library
          'charts': ['recharts', 'd3'],
        },
        // Or use a function for dynamic grouping
        manualChunks(id) {
          if (id.includes('node_modules')) {
            if (id.includes('react')) return 'react-vendor';
            if (id.includes('lodash')) return 'lodash';
            return 'vendor';
          }
        },
      },
    },
  },
});

5. Prefetching Route Chunks

// Approach 1: Eager prefetch with link tags
function prefetchRoute(chunkUrl) {
  const link = document.createElement('link');
  link.rel = 'prefetch';
  link.href = chunkUrl;
  link.as = 'script';
  document.head.appendChild(link);
}

// Approach 2: Trigger the import (loads and caches the module)
const routeModules = {
  dashboard: () => import('./pages/Dashboard'),
  settings: () => import('./pages/Settings'),
  admin: () => import('./pages/AdminPanel'),
};

function prefetchRoute(routeName) {
  routeModules[routeName]?.();  // import() caches — second call is instant
}

// Approach 3: Prefetch on link hover
function PrefetchLink({ to, importFn, children }) {
  return (
    <Link
      to={to}
      onMouseEnter={() => importFn()}
      onFocus={() => importFn()}
    >
      {children}
    </Link>
  );
}

// Usage
<PrefetchLink to="/dashboard" importFn={() => import('./pages/Dashboard')}>
  Dashboard
</PrefetchLink>

Analysis Checklist

When analyzing bundles, look for: (1) large libraries that could be replaced (moment.js → date-fns), (2) duplicate dependencies across chunks, (3) polyfills included for browsers you don't support, (4) dev-only code leaking into production, (5) CSS-in-JS runtime size, and (6) chunks that are too large (>250KB gzipped) or too small (<5KB — merge overhead).

7. Suspense Fallbacks During Loading

// Granular fallbacks per route
function PageSkeleton({ type }) {
  switch (type) {
    case 'dashboard':
      return (
        <div className="skeleton">
          <div className="skeleton-header" />
          <div className="skeleton-grid">
            {Array.from({ length: 4 }, (_, i) => (
              <div key={i} className="skeleton-card" />
            ))}
          </div>
        </div>
      );
    default:
      return <div className="skeleton-page" />;
  }
}

// Avoiding layout shift with skeleton dimensions
// Match the skeleton's dimensions to the actual content
// to prevent CLS (Cumulative Layout Shift)

// Minimum display time to avoid flash of loading state
function DelayedSuspense({ fallback, minDelay = 200, children }) {
  const [showFallback, setShowFallback] = useState(false);

  useEffect(() => {
    const timer = setTimeout(() => setShowFallback(true), minDelay);
    return () => clearTimeout(timer);
  }, [minDelay]);

  return (
    <Suspense fallback={showFallback ? fallback : null}>
      {children}
    </Suspense>
  );
}

8. Retry Logic for Failed Chunks

Chunk loading can fail due to network issues, CDN problems, or deployment cache invalidation. Production apps need retry logic.

// Retry wrapper for dynamic imports
function retryImport(importFn, retries = 3, delay = 1000) {
  return new Promise((resolve, reject) => {
    importFn()
      .then(resolve)
      .catch((error) => {
        if (retries === 0) {
          reject(error);
          return;
        }

        setTimeout(() => {
          retryImport(importFn, retries - 1, delay * 2)
            .then(resolve)
            .catch(reject);
        }, delay);
      });
  });
}

// Usage with React.lazy
const Dashboard = lazy(() => retryImport(
  () => import('./pages/Dashboard')
));

// Handle chunk load errors with error boundary
class ChunkErrorBoundary extends Component {
  state = { hasError: false, error: null };

  static getDerivedStateFromError(error) {
    const isChunkError = error.name === 'ChunkLoadError' ||
      error.message?.includes('Loading chunk');
    return { hasError: true, error, isChunkError };
  }

  handleRetry = () => {
    this.setState({ hasError: false });
    window.location.reload();  // full reload to get fresh chunk URLs
  };

  render() {
    if (this.state.hasError && this.state.isChunkError) {
      return (
        <div>
          <p>A new version is available.</p>
          <button onClick={this.handleRetry}>Refresh</button>
        </div>
      );
    }
    return this.props.children;
  }
}
Deployment Cache Invalidation

The most common cause of chunk load failures in production is deploying a new version while users have the old version's HTML. The old HTML references chunk filenames that no longer exist on the CDN. Solutions: (1) keep old chunks alive for a TTL period, (2) use content-hash filenames with immutable caching, (3) detect ChunkLoadError and prompt a page refresh, (4) use service workers to cache chunks.

9. Module Federation for Micro-Frontends

Module Federation (Webpack 5) allows separately deployed applications to share code at runtime, enabling micro-frontend architectures where different teams own different routes.

// Host application (shell) — webpack.config.js
const { ModuleFederationPlugin } = require('webpack').container;

module.exports = {
  plugins: [
    new ModuleFederationPlugin({
      name: 'shell',
      remotes: {
        dashboard: 'dashboard@https://dashboard.example.com/remoteEntry.js',
        settings: 'settings@https://settings.example.com/remoteEntry.js',
      },
      shared: {
        react: { singleton: true, requiredVersion: '^18.0.0' },
        'react-dom': { singleton: true, requiredVersion: '^18.0.0' },
        'react-router-dom': { singleton: true },
      },
    }),
  ],
};

// Remote application (dashboard) — webpack.config.js
module.exports = {
  plugins: [
    new ModuleFederationPlugin({
      name: 'dashboard',
      filename: 'remoteEntry.js',
      exposes: {
        './DashboardApp': './src/DashboardApp',
      },
      shared: {
        react: { singleton: true },
        'react-dom': { singleton: true },
      },
    }),
  ],
};

// Shell routes — loading remote micro-frontends
const DashboardApp = lazy(() => import('dashboard/DashboardApp'));
const SettingsApp = lazy(() => import('settings/SettingsApp'));

const routes = [
  {
    path: '/',
    element: <ShellLayout />,
    children: [
      { path: 'dashboard/*', element: <Suspense><DashboardApp /></Suspense> },
      { path: 'settings/*', element: <Suspense><SettingsApp /></Suspense> },
    ],
  },
];
Aspect Route-Based Splitting Module Federation
Deployment Single deploy Independent per team
Shared deps Bundler handles Runtime negotiation
Version skew Not possible Must handle carefully
Complexity Low High (runtime loading, version management)
Use case Single team, any size Multiple teams, large org
Architect Insight

Module Federation is an organizational scaling solution, not a performance optimization. It adds runtime complexity (version negotiation, shared singleton management, error handling for remote failures) in exchange for deployment independence. Only adopt it when the organizational benefit (team autonomy, independent release cycles) outweighs the technical cost. Most applications are better served by a monorepo with route-based code splitting.
