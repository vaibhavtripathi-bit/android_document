1. Why SPAs Need Routing

Traditional multi-page applications (MPAs) rely on the browser to handle navigation: each link click triggers a full HTTP request, the server returns a new HTML document, and the browser replaces the entire page. Single-page applications (SPAs) break this model by intercepting navigation and rendering new views in JavaScript without a full page reload.

This creates a fundamental problem: the URL in the address bar no longer reflects the application state. Without routing, SPAs would have a single URL for all views, breaking bookmarks, the back button, link sharing, and accessibility. Client-side routing bridges this gap by synchronizing the URL with the application's view state.

Traditional MPA Navigation: Browser ──GET /about──▶ Server ──HTML──▶ Browser (full page reload) Browser ──GET /contact──▶ Server ──HTML──▶ Browser (full page reload) SPA with Client-Side Routing: Browser ──click "About"──▶ JS intercepts ├── pushState('/about') ← URL updates without request ├── match('/about') → AboutComponent └── render(AboutComponent) ← DOM update, no reload Browser ──click "Back"──▶ popstate event fires ├── match(previous URL) → PreviousComponent └── render(PreviousComponent)
Architect Insight

At the Staff level, you should be able to explain why client-side routing exists as a consequence of the SPA architecture, not as a standalone feature. The tradeoffs — initial bundle size, SEO challenges, accessibility concerns, and the need for server-side fallback configuration — are all direct consequences of moving navigation from the server to the client.

2. The History API

The History API is the browser primitive that makes client-side routing possible. Introduced in HTML5, it allows JavaScript to manipulate the browser's session history without triggering page reloads.

2.1 Core Methods

// pushState: Add a new entry to the history stack
// Signature: history.pushState(state, unused, url)
history.pushState(
  { page: 'about', scrollY: 0 },  // state object (serializable, 640KB limit in most browsers)
  '',                                    // title (ignored by all browsers)
  '/about'                               // new URL (must be same-origin)
);

// replaceState: Replace the current entry (no new entry created)
history.replaceState(
  { page: 'about', scrollY: 150 },
  '',
  '/about?section=team'
);

// Navigation methods
history.back();        // equivalent to browser back button
history.forward();     // equivalent to browser forward button
history.go(-2);       // go back 2 entries
history.go(0);        // reload current page

// Reading current state
console.log(history.state);   // the state object from pushState/replaceState
console.log(history.length);  // number of entries in the session history

2.2 The popstate Event

The popstate event fires when the user navigates via back/forward buttons (or history.back()/history.forward()). Critically, it does not fire on pushState or replaceState — this is the most common source of confusion.

window.addEventListener('popstate', (event) => {
  console.log('Navigation to:', window.location.pathname);
  console.log('State:', event.state);

  // event.state is the state object passed to pushState/replaceState
  // for the entry being navigated TO
  // It's null for the initial page load entry
});

// This does NOT fire popstate:
history.pushState(null, '', '/new-page');

// This DOES fire popstate:
history.back();
Critical Detail

pushState and replaceState do NOT fire popstate. This means routers must manually trigger their own route matching after calling these methods. This is why every router library has its own internal event system (e.g., React Router's history.listen()) rather than relying solely on popstate.

2.3 History Stack Mental Model

History Stack After Navigation Sequence: 1. User visits /home Stack: [/home*] 2. pushState('/about') Stack: [/home, /about*] 3. pushState('/contact') Stack: [/home, /about, /contact*] 4. history.back() Stack: [/home, /about*, /contact] popstate fires with /about state 5. pushState('/blog') Stack: [/home, /about, /blog*] /contact is DISCARDED (forward entries pruned) 6. replaceState('/blog?p=1') Stack: [/home, /about, /blog?p=1*] /blog replaced in-place * = current entry

3. Hash Routing vs Browser Routing

Two fundamentally different approaches to client-side routing exist, each with distinct tradeoffs.

Aspect Hash Routing Browser (History) Routing
URL format example.com/#/about example.com/about
Browser API hashchange event History API (pushState)
Server config None required — hash is never sent to server Requires catch-all fallback to index.html
SEO Poor — crawlers may ignore hash fragments Good — real URLs are crawlable
SSR compatibility Not possible — hash is client-only Fully compatible
Deep linking Works but looks unprofessional Clean, semantic URLs
Analytics Requires custom tracking Standard page view tracking works
Use case Legacy browsers, static file hosting Modern production apps 
// Hash routing: listen for hashchange
window.addEventListener('hashchange', () => {
  const path = window.location.hash.slice(1); // remove '#'
  renderRoute(path);
});

// Navigate by setting hash
window.location.hash = '#/about';

// Browser routing: use History API
function navigate(path) {
  history.pushState(null, '', path);
  renderRoute(path);  // must call manually — pushState doesn't fire popstate
}

window.addEventListener('popstate', () => {
  renderRoute(window.location.pathname);
});

4. Route Matching Algorithms

Route matching is the process of mapping a URL pathname to a route configuration. Modern routers use ranked matching with specificity scoring rather than simple first-match.

4.1 Simple Pattern Matching

// Basic route matching: convert route patterns to RegExp
function pathToRegex(pattern) {
  const paramNames = [];
  const regexStr = pattern
    .replace(/:(\w+)/g, (_, name) => {
      paramNames.push(name);
      return '([^/]+)';  // capture group for params
    })
    .replace(/\*/g, '(.*)');  // wildcard/splat

  return {
    regex: new RegExp(`^${regexStr}$`),
    paramNames
  };
}

// Match URL against route
function matchRoute(pattern, pathname) {
  const { regex, paramNames } = pathToRegex(pattern);
  const match = pathname.match(regex);
  if (!match) return null;

  const params = {};
  paramNames.forEach((name, i) => {
    params[name] = decodeURIComponent(match[i + 1]);
  });
  return { params };
}

// Examples:
matchRoute('/users/:id', '/users/42');
// → { params: { id: '42' } }

matchRoute('/files/*', '/files/docs/readme.md');
// → { params: { '*': 'docs/readme.md' } }

4.2 Ranked Route Matching

React Router v6 uses a ranked matching algorithm that scores routes by specificity, similar to CSS specificity. This eliminates the need for manual route ordering.

Route Ranking Scores (higher = more specific): Pattern Score Reason ────────────────────────────────────────────────── /users/new 6 static segment × 2 /users/:id 4 static + dynamic /users/:id/edit 7 static + dynamic + static /users/* 1 static + splat (lowest) / 3 index route bonus URL: /users/new /users/new → score 6 ✓ (wins — most specific) /users/:id → score 4 (matches but lower rank) /users/* → score 1 (matches but lowest rank) 
// Simplified ranking algorithm (React Router v6 approach)
function computeScore(pattern) {
  const segments = pattern.split('/').filter(Boolean);
  let score = 0;

  for (const segment of segments) {
    if (segment === '*') {
      score += 1;           // splat: lowest priority
    } else if (segment.startsWith(':')) {
      score += 2;           // dynamic param
    } else {
      score += 3;           // static segment: highest priority
    }
  }

  // Bonus for index routes (exact match)
  if (!pattern.includes('*') && !pattern.includes(':')) {
    score += 1;
  }

  return score;
}

5. URL Parsing & the URL API

The URL and URLSearchParams APIs provide robust URL parsing without regex hacks. Staff engineers should use these instead of manual string splitting.

const url = new URL('https://example.com/users/42?tab=posts&sort=recent#comments');

url.protocol;   // 'https:'
url.hostname;   // 'example.com'
url.pathname;   // '/users/42'
url.search;     // '?tab=posts&sort=recent'
url.hash;       // '#comments'
url.origin;     // 'https://example.com'

// Relative URL resolution
const resolved = new URL('../settings', 'https://example.com/users/42');
resolved.pathname; // '/settings'

// URL pattern matching (new URLPattern API — Chrome 95+)
const pattern = new URLPattern({ pathname: '/users/:id' });
const result = pattern.exec('https://example.com/users/42');
result.pathname.groups; // { id: '42' }
URLPattern API

The URLPattern API is a new web standard that brings Express-style route matching to the browser. It supports named groups, wildcards, and regex constraints. While not yet available in all browsers, it's the future of URL matching and is already used in Cloudflare Workers and Deno.

6. Search Params Management

Search parameters (query strings) are a critical part of URL state. They're used for filters, pagination, sorting, and other ephemeral UI state that should survive page refreshes and be shareable.

// URLSearchParams API
const params = new URLSearchParams('?tab=posts&sort=recent&tag=react&tag=hooks');

params.get('tab');        // 'posts' (first value)
params.getAll('tag');     // ['react', 'hooks'] (all values)
params.has('sort');       // true
params.toString();        // 'tab=posts&sort=recent&tag=react&tag=hooks'

// Mutation (creates new string, doesn't modify URL)
params.set('page', '2');
params.delete('sort');
params.append('tag', 'typescript');

// Integrating with History API
function updateSearchParams(updates) {
  const params = new URLSearchParams(window.location.search);
  Object.entries(updates).forEach(([key, value]) => {
    if (value === null) {
      params.delete(key);
    } else {
      params.set(key, value);
    }
  });
  const newUrl = `${window.location.pathname}?${params}`;
  history.pushState(null, '', newUrl);
}

// Type-safe search params pattern
function parseSearchParams<T>(
  search: string,
  schema: Record<string, (v: string) => unknown>
): T {
  const params = new URLSearchParams(search);
  const result: Record<string, unknown> = {};
  for (const [key, parser] of Object.entries(schema)) {
    const value = params.get(key);
    if (value !== null) result[key] = parser(value);
  }
  return result as T;
}

7. Navigation Interception

Client-side routers must intercept navigation events to prevent full page reloads. This involves handling link clicks, form submissions, and the new Navigation API.

7.1 Link Click Interception

// How routers intercept <a> clicks (delegated event handler)
document.addEventListener('click', (event) => {
  const anchor = event.target.closest('a[href]');
  if (!anchor) return;

  // Skip if modifier keys held (open in new tab)
  if (event.metaKey || event.ctrlKey || event.shiftKey || event.altKey) return;

  // Skip if target="_blank" or download attribute
  if (anchor.target === '_blank' || anchor.hasAttribute('download')) return;

  // Skip if external link (different origin)
  if (anchor.origin !== window.location.origin) return;

  // Skip non-left clicks
  if (event.button !== 0) return;

  event.preventDefault();
  const path = anchor.pathname + anchor.search + anchor.hash;
  navigate(path);
});

7.2 The Navigation API (Modern Replacement)

// Navigation API (Chrome 102+) — the modern approach
navigation.addEventListener('navigate', (event) => {
  const url = new URL(event.destination.url);

  // Only intercept same-origin navigations
  if (url.origin !== location.origin) return;

  // Skip if not a push or replace navigation
  if (event.navigationType !== 'push' && event.navigationType !== 'replace') return;

  event.intercept({
    async handler() {
      const content = await fetchRouteContent(url.pathname);
      renderContent(content);
    }
  });
});
Navigation API Benefits

The Navigation API solves many pain points of the History API: it fires for all navigations (not just back/forward), provides event.intercept() for clean interception, supports transition tracking via navigation.transition, and offers navigation.entries() to inspect the full history stack. It's the future of client-side routing.

8. Browser Back/Forward Handling

Correct back/forward button handling is one of the trickiest aspects of client-side routing. The router must distinguish between programmatic navigation and user-initiated back/forward.

// Tracking navigation direction
let currentIndex = 0;
const KEY = '__router_index';

function pushRoute(path) {
  currentIndex++;
  history.pushState({ index: currentIndex }, '', path);
  handleRouteChange('push');
}

window.addEventListener('popstate', (event) => {
  const prevIndex = currentIndex;
  currentIndex = event.state?.index ?? 0;

  if (currentIndex < prevIndex) {
    handleRouteChange('back');
  } else {
    handleRouteChange('forward');
  }
});

// Preventing back navigation (e.g., unsaved form data)
window.addEventListener('beforeunload', (event) => {
  if (hasUnsavedChanges()) {
    event.preventDefault();
    // Modern browsers ignore custom messages
    event.returnValue = '';
  }
});
beforeunload Limitation

beforeunload only fires for full page unloads (closing tab, navigating to external URL). It does NOT fire for client-side navigation via pushState. Routers must implement their own "navigation blocker" (e.g., React Router's useBlocker) to prevent in-app navigation when forms have unsaved changes.

9. Scroll Restoration

When users navigate back, they expect to return to their previous scroll position. Browsers handle this automatically for MPA navigation, but SPAs must manage it manually.

// Disable browser's automatic scroll restoration
if ('scrollRestoration' in history) {
  history.scrollRestoration = 'manual';
}

// Custom scroll restoration
const scrollPositions = new Map();

function saveScrollPosition(key) {
  scrollPositions.set(key, {
    x: window.scrollX,
    y: window.scrollY
  });
}

function restoreScrollPosition(key) {
  const position = scrollPositions.get(key);
  if (position) {
    // requestAnimationFrame ensures DOM has updated
    requestAnimationFrame(() => {
      window.scrollTo(position.x, position.y);
    });
  } else {
    window.scrollTo(0, 0);
  }
}

// Save on every navigation away
function navigate(path) {
  saveScrollPosition(window.location.pathname);
  history.pushState(null, '', path);
  window.scrollTo(0, 0);  // new page starts at top
  renderRoute(path);
}

// Restore on back/forward
window.addEventListener('popstate', () => {
  renderRoute(window.location.pathname);
  restoreScrollPosition(window.location.pathname);
});
Architect Insight

Scroll restoration is deceptively complex in practice. Challenges include: (1) restoring scroll in virtualized lists where the DOM nodes don't exist yet, (2) waiting for images/lazy content to load before restoring, (3) handling multiple scrollable containers (not just window), and (4) coordinating with route transitions and Suspense boundaries. React Router v6.4+ provides ScrollRestoration component that handles many of these edge cases.

10. Building a Minimal Router

Putting it all together — a minimal but complete client-side router that demonstrates all the concepts.

class Router {
  #routes = [];
  #listeners = new Set();
  #scrollPositions = new Map();

  constructor() {
    history.scrollRestoration = 'manual';

    window.addEventListener('popstate', () => {
      this.#handleNavigation('pop');
    });

    // Intercept link clicks
    document.addEventListener('click', (e) => {
      const a = e.target.closest('a[href]');
      if (!a || a.origin !== location.origin) return;
      if (e.metaKey || e.ctrlKey || e.shiftKey) return;
      if (a.target === '_blank') return;
      e.preventDefault();
      this.push(a.pathname + a.search);
    });
  }

  addRoute(pattern, handler) {
    const paramNames = [];
    const regexStr = pattern
      .replace(/:(\w+)/g, (_, n) => { paramNames.push(n); return '([^/]+)'; })
      .replace(/\*/g, '(.*)');

    this.#routes.push({
      pattern,
      regex: new RegExp(`^${regexStr}$`),
      paramNames,
      handler,
      score: this.#computeScore(pattern)
    });

    // Sort by score descending for ranked matching
    this.#routes.sort((a, b) => b.score - a.score);
  }

  push(path) {
    this.#scrollPositions.set(location.pathname, window.scrollY);
    history.pushState(null, '', path);
    window.scrollTo(0, 0);
    this.#handleNavigation('push');
  }

  replace(path) {
    history.replaceState(null, '', path);
    this.#handleNavigation('replace');
  }

  #handleNavigation(type) {
    const match = this.#matchRoute(location.pathname);
    if (match) {
      match.route.handler(match.params);
    }
    if (type === 'pop') {
      const savedY = this.#scrollPositions.get(location.pathname);
      if (savedY != null) requestAnimationFrame(() => window.scrollTo(0, savedY));
    }
    this.#listeners.forEach(fn => fn(location.pathname, type));
  }

  #matchRoute(pathname) {
    for (const route of this.#routes) {
      const m = pathname.match(route.regex);
      if (m) {
        const params = {};
        route.paramNames.forEach((n, i) => { params[n] = decodeURIComponent(m[i+1]); });
        return { route, params };
      }
    }
    return null;
  }

  #computeScore(pattern) {
    return pattern.split('/').filter(Boolean).reduce((s, seg) =>
      s + (seg === '*' ? 1 : seg.startsWith(':') ? 2 : 3), 0);
  }

  listen(fn) {
    this.#listeners.add(fn);
    return () => this.#listeners.delete(fn);
  }
}
Architect Insight

This minimal router demonstrates the core concepts but lacks features required for production: nested routes, lazy loading, data loading, transition management, error handling, and SSR support. Understanding what's missing — and why each feature is needed — is exactly the kind of reasoning expected at L6. The gap between this and React Router is the gap between a toy and a production routing solution.
