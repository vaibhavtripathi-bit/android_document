1. Modern Image Formats

Format Compression Browser Support Best For
JPEG Lossy, good Universal Photos (fallback)
PNG Lossless Universal Screenshots, transparency
WebP Lossy/lossless, 25-35% smaller than JPEG 97%+ browsers Photos, illustrations
AVIF Lossy/lossless, 50% smaller than JPEG ~92% browsers Photos (best compression)
SVG Vector (scales infinitely) Universal Icons, logos, illustrations 
// Serve modern formats with fallback using <picture>
<picture>
  <source srcset="hero.avif" type="image/avif" />
  <source srcset="hero.webp" type="image/webp" />
  <img src="hero.jpg" alt="Hero image" width="1200" height="600" />
</picture>

// Browser picks the first supported format:
// Chrome/Firefox → AVIF (smallest)
// Safari 16+ → WebP
// Older Safari → JPEG (fallback)
AVIF vs WebP

AVIF offers ~50% better compression than JPEG (vs WebP's ~30%), but encoding is significantly slower (10-100x slower than WebP). For user-uploaded images, WebP is more practical for real-time conversion. For static assets, pre-generate AVIF at build time. AVIF also supports HDR, wide color gamut, and film grain synthesis — features relevant for photography-heavy sites.

2. Responsive Images

// srcset: provide multiple resolutions
<img
  src="photo-800.jpg"
  srcset="
    photo-400.jpg 400w,
    photo-800.jpg 800w,
    photo-1200.jpg 1200w,
    photo-1600.jpg 1600w
  "
  sizes="
    (max-width: 640px) 100vw,
    (max-width: 1024px) 50vw,
    33vw
  "
  alt="Responsive photo"
  width="800"
  height="600"
/>

// How the browser chooses:
// 1. Evaluates 'sizes' media queries to determine display width
// 2. Multiplies by device pixel ratio (DPR)
// 3. Picks the smallest srcset image that covers the needed width
//
// Example: 640px viewport, DPR 2 → needs 640 * 2 = 1280px
// sizes says 100vw at this width → display width = 640px
// 640px * 2 DPR = 1280px needed → picks photo-1600.jpg

// Art direction: different crops for different viewports
<picture>
  <source media="(max-width: 640px)" srcset="hero-mobile.jpg" />
  <source media="(max-width: 1024px)" srcset="hero-tablet.jpg" />
  <img src="hero-desktop.jpg" alt="Hero" />
</picture>
Always Set width and height

Always include width and height attributes on <img> elements. The browser uses these to calculate the aspect ratio before the image loads, reserving the correct space and preventing Cumulative Layout Shift (CLS). Without them, the image area starts at 0 height and jumps when the image loads — a major CLS contributor.

3. Lazy Loading

// Native lazy loading (simplest)
<img src="photo.jpg" loading="lazy" alt="Lazy loaded" />
// Browser defers loading until image is near the viewport
// "Near" threshold varies by browser (~1250px in Chrome)

// IMPORTANT: Don't lazy-load above-the-fold images!
// LCP image should load eagerly:
<img src="hero.jpg" loading="eager" fetchpriority="high" alt="Hero" />

// Intersection Observer for custom lazy loading
function LazyImage({ src, alt, ...props }) {
  const [isLoaded, setIsLoaded] = useState(false);
  const ref = useRef(null);

  useEffect(() => {
    const observer = new IntersectionObserver(
      ([entry]) => {
        if (entry.isIntersecting) {
          setIsLoaded(true);
          observer.disconnect();
        }
      },
      { rootMargin: '200px' }
    );
    if (ref.current) observer.observe(ref.current);
    return () => observer.disconnect();
  }, []);

  return (
    <div ref={ref}>
      {isLoaded ? (
        <img src={src} alt={alt} {...props} />
      ) : (
        <div className="placeholder" style={{ aspectRatio: props.width / props.height }} />
      )}
    </div>
  );
}

4. Next.js Image Component

import Image from 'next/image';

// Next.js Image handles: format conversion, resizing, lazy loading,
// placeholder blur, and responsive srcset generation automatically

<Image
  src="/photos/hero.jpg"
  alt="Hero image"
  width={1200}
  height={600}
  priority              // disable lazy loading for LCP image
  placeholder="blur"    // show blurred preview while loading
  blurDataURL={blurHash} // base64 blur placeholder
  sizes="(max-width: 768px) 100vw, 50vw"
  quality={80}
/>

// What Next.js Image generates:
// 1. Converts to WebP/AVIF on-demand via image optimization API
// 2. Generates srcset with multiple widths (640, 750, 828, 1080, 1200...)
// 3. Adds loading="lazy" (unless priority is set)
// 4. Adds width/height for aspect ratio (prevents CLS)
// 5. Caches optimized images in .next/cache/images

// Remote images require configuration:
// next.config.js
module.exports = {
  images: {
    remotePatterns: [
      { protocol: 'https', hostname: 'cdn.example.com' },
    ],
    formats: ['image/avif', 'image/webp'],
    deviceSizes: [640, 750, 828, 1080, 1200, 1920],
  },
};

5. Video Optimization

// Lazy load video (don't autoplay offscreen)
<video
  preload="none"        // don't download until play
  poster="thumbnail.jpg" // show image before play
  controls
>
  <source src="video.webm" type="video/webm" />
  <source src="video.mp4" type="video/mp4" />
</video>

// For background/hero videos: use preload="metadata"
// Downloads only metadata (duration, dimensions) — not the video

// Replace GIFs with video (much smaller file size)
<video autoplay muted loop playsinline>
  <source src="animation.webm" type="video/webm" />
  <source src="animation.mp4" type="video/mp4" />
</video>
// A 5MB GIF → 500KB WebM video (90% reduction)

6. Font Loading Strategies

// font-display controls text visibility during font loading
@font-face {
  font-family: 'CustomFont';
  src: url('font.woff2') format('woff2');
  font-display: swap;  // show fallback immediately, swap when loaded
}

// font-display values:
// auto     — browser default (usually block)
// block    — invisible text for up to 3s, then fallback (FOIT)
// swap     — fallback immediately, swap when loaded (FOUT)
// fallback — invisible 100ms, fallback 3s, then swap or keep fallback
// optional — invisible 100ms, then fallback forever (font may not load)

// Preload critical fonts
<link rel="preload" href="/fonts/inter.woff2" as="font"
      type="font/woff2" crossorigin />

// Next.js font optimization (automatic)
import { Inter } from 'next/font/google';

const inter = Inter({
  subsets: ['latin'],
  display: 'swap',
  preload: true,
});
// Next.js: downloads font at build time, self-hosts it,
// generates @font-face with font-display: swap,
// and adds preload link — no external requests at runtime

// Subset fonts to reduce size
// Full Inter: ~300KB, Latin subset: ~20KB
// Use unicode-range for multi-language sites:
@font-face {
  font-family: 'Inter';
  src: url('inter-latin.woff2') format('woff2');
  unicode-range: U+0000-00FF;  // Basic Latin only
}

7. Critical Resource Prioritization

// fetchpriority attribute (Chrome 101+)
<img src="hero.jpg" fetchpriority="high" />   // LCP image
<img src="avatar.jpg" fetchpriority="low" />  // below-fold image

// Preload: fetch critical resources early
<link rel="preload" href="/hero.avif" as="image" type="image/avif" />
<link rel="preload" href="/critical.css" as="style" />
<link rel="preload" href="/font.woff2" as="font" crossorigin />

// Preconnect: establish early connections to required origins
<link rel="preconnect" href="https://cdn.example.com" />
<link rel="preconnect" href="https://fonts.googleapis.com" />

// DNS prefetch: resolve DNS for likely-needed origins
<link rel="dns-prefetch" href="https://analytics.example.com" />
Resource Loading Priority: Priority Resource Type Strategy ────────────────────────────────────────────────────── Highest HTML document — High CSS (render-blocking) Inline critical CSS High LCP image preload + fetchpriority="high" High Critical fonts preload + font-display: swap Medium JS bundles async/defer Medium Above-fold images eager loading Low Below-fold images loading="lazy" Low Prefetch (next page) <link rel="prefetch"> Lowest Analytics, tracking defer or requestIdleCallback

Architect Insight

Image optimization is often the single highest-impact performance improvement for content-heavy sites. A typical e-commerce page has 2-5MB of images. Converting to WebP/AVIF, serving responsive sizes, and lazy-loading below-fold images can reduce this to 200-500KB — a 5-10x improvement. Automate this: use an image CDN (Cloudinary, imgix) or Next.js Image component so developers don't have to think about optimization per-image.
