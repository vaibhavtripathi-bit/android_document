1. React's Rendering Model

"Rendering" in React means calling your component function to produce a new React element tree. This is distinct from the browser's rendering (layout, paint, composite). React rendering is a pure computation — it produces a virtual description of the UI. Only if this description differs from the current DOM does React commit changes (the "commit phase").

React Rendering Pipeline: Trigger Render Phase Commit Phase Browser Paint (state change) → (call components, → (apply DOM changes, → (layout, paint, build fiber tree, run effects) composite) diff with current) ┌─────────────────┐ ┌──────────────┐ ┌─────────────┐ State update ──▶ │ Call component │ ──▶ │ Update DOM │ ──▶ │ Browser │ │ functions │ │ nodes │ │ repaints │ │ (pure, no side │ │ Run layout │ │ screen │ │ effects) │ │ effects │ │ │ │ Diff old vs new │ │ Run effects │ │ │ └─────────────────┘ └──────────────┘ └─────────────┘ Can be interrupted Synchronous Browser-controlled (concurrent mode) (cannot interrupt)
Key Distinction

A React "render" does NOT mean the DOM was updated. React may render a component (call its function), diff the result, and determine that nothing changed — no DOM mutations occur. This is why rendering is relatively cheap: it's just JavaScript function calls. The expensive part is DOM mutations and browser repaints, which only happen when the diff finds changes.

2. What Triggers Re-Renders

There are exactly three things that cause a React component to re-render:

2.1 State Change

function Counter() {
  const [count, setCount] = useState(0);

  // Calling setCount triggers a re-render of Counter
  // AND all of its children (the entire subtree)
  return (
    <div>
      <p>{count}</p>
      <button onClick={() => setCount(c => c + 1)}>+</button>
      <ExpensiveChild />  {/* also re-renders! */}
    </div>
  );
}

// Bail-out optimization: if new state === old state (Object.is),
// React may bail out early and skip re-rendering children.
// But the component itself still runs once to check.
setCount(0);  // if count is already 0, React bails out

2.2 Parent Re-Render

function Parent() {
  const [count, setCount] = useState(0);

  return (
    <div>
      <p>{count}</p>
      {/* Child re-renders even though it receives NO props from Parent */}
      {/* This is the #1 source of unnecessary re-renders */}
      <Child />
    </div>
  );
}

function Child() {
  console.log('Child rendered');  // logs on every Parent state change
  return <div>I'm a child</div>;
}

2.3 Context Change

const ThemeContext = createContext('light');

function App() {
  const [theme, setTheme] = useState('light');
  return (
    <ThemeContext.Provider value={theme}>
      <DeepChild />  {/* re-renders when theme changes */}
    </ThemeContext.Provider>
  );
}

function DeepChild() {
  const theme = useContext(ThemeContext);
  // Re-renders whenever the context value changes
  // Even if this component is wrapped in React.memo!
  // Context bypasses memoization
  return <div className={theme}>Content</div>;
}
Common Misconception

Props changing does NOT directly cause re-renders. A component re-renders because its parent re-rendered, and it receives new props as a consequence. Even if props are identical, the child still re-renders (unless wrapped in React.memo). The re-render trigger is the parent re-rendering, not the props changing.

3. Why React Re-Renders Entire Subtrees

When a component's state changes, React re-renders that component AND every component in its subtree. This is a deliberate design choice, not a bug.

State change in B: A A (not re-rendered) / \ / \ B C B* C (not re-rendered) / \ / \ D E D* E* (both re-rendered) / / F F* (re-rendered) * = re-rendered Why? React cannot know if D, E, or F depend on B's state without actually calling them. The safest default is to re-render the entire subtree and let the diff determine what DOM changes are actually needed. 
// Why React can't skip children automatically:
function Parent() {
  const [user, setUser] = useState({ name: 'Alice' });

  // React doesn't know if Child uses user.name or not
  // without actually calling Child's render function
  return <Child user={user} />;
}

// Even without props, children might depend on parent state via:
// - Context (useContext)
// - Refs (useRef from parent passed down)
// - External stores (useSyncExternalStore)
// - Module-level variables
// React's only safe option is to re-render and diff

4. The Profiler API

import { Profiler } from 'react';

function onRenderCallback(
  id,                // the "id" prop of the Profiler tree
  phase,             // "mount" | "update" | "nested-update"
  actualDuration,    // time spent rendering the committed update (ms)
  baseDuration,      // estimated time to render entire subtree without memoization
  startTime,         // when React began rendering this update
  commitTime,        // when React committed this update
) {
  // Send to analytics
  performance.mark(`react-render-${id}`, {
    detail: { phase, actualDuration, baseDuration },
  });

  if (actualDuration > 16) {
    console.warn(`Slow render: ${id} took ${actualDuration.toFixed(1)}ms`);
  }
}

function App() {
  return (
    <Profiler id="Dashboard" onRender={onRenderCallback}>
      <Dashboard />
    </Profiler>
  );
}

// Profilers can be nested for granular measurement
<Profiler id="App" onRender={onRender}>
  <Header />
  <Profiler id="Sidebar" onRender={onRender}>
    <Sidebar />
  </Profiler>
  <Profiler id="MainContent" onRender={onRender}>
    <MainContent />
  </Profiler>
</Profiler>
actualDuration vs baseDuration

actualDuration is the time React spent rendering components that actually re-rendered (respecting memoization bail-outs). baseDuration is the estimated time if every component in the subtree re-rendered (no memoization). The ratio actualDuration / baseDuration tells you how effective your memoization is. If they're similar, memoization isn't helping much.

// Tool 2: Manual render counting

  const count = useRef(0);
  count.current++;
  useEffect(() => {

6.1 Common Causes of Unnecessary Re-Renders

Cause Example Fix
Inline object/array creation style={{ color: 'red' }} Extract to constant or useMemo
Inline function creation onClick={() => handleClick(id)} useCallback (only if child is memoized)
Context value changes New object on every render as context value useMemo the context value
State too high in tree Global state causing subtree re-renders Push state down, split context
Unrelated state in same component Timer state causing list re-render Extract into separate component

7. Render Cost Measurement

// Measuring render cost with Performance API
function ExpensiveComponent({ data }) {
  performance.mark('render-start');

  // ... component logic ...
  const processed = data.map(item => expensiveTransform(item));

  const result = (
    <div>
      {processed.map(item => <Row key={item.id} item={item} />)}
    </div>
  );

  performance.mark('render-end');
  performance.measure('ExpensiveComponent render', 'render-start', 'render-end');

  return result;
}

8. Component-Level Optimization Strategies

8.1 Children as Props (Composition Pattern)

// ❌ State change in Parent re-renders ExpensiveChild
function Parent() {
  const [count, setCount] = useState(0);
  return (
    <div>
      <button onClick={() => setCount(c => c + 1)}>{count}</button>
      <ExpensiveChild />  {/* re-renders every time! */}
    </div>
  );
}

// ✅ Move state into a wrapper, pass ExpensiveChild as children
function CounterWrapper({ children }) {
  const [count, setCount] = useState(0);
  return (
    <div>
      <button onClick={() => setCount(c => c + 1)}>{count}</button>
      {children}  {/* children is a prop — same reference, no re-render */}
    </div>
  );
}

function App() {
  return (
    <CounterWrapper>
      <ExpensiveChild />  {/* created in App, not CounterWrapper */}
    </CounterWrapper>
  );
}
// ExpensiveChild's React element is created in App's render.
// When CounterWrapper re-renders, children is the same reference
// (App didn't re-render), so React skips re-rendering ExpensiveChild.

8.2 State Colocation

// ❌ State too high — typing in search re-renders entire list
function Page() {
  const [search, setSearch] = useState('');
  const [items] = useState(generateLargeList());

  return (
    <div>
      <input value={search} onChange={e => setSearch(e.target.value)} />
      <ItemList items={items} />  {/* re-renders on every keystroke! */}
    </div>
  );
}

// ✅ Colocate search state with the input
function SearchInput({ onSearch }) {
  const [search, setSearch] = useState('');
  return (
    <input
      value={search}
      onChange={e => {
        setSearch(e.target.value);
        onSearch(e.target.value);
      }}
    />
  );
}

function Page() {
  const [items] = useState(generateLargeList());
  const [filteredItems, setFilteredItems] = useState(items);

  return (
    <div>
      <SearchInput onSearch={q => setFilteredItems(items.filter(...))} />
      <ItemList items={filteredItems} />
    </div>
  );
}

8.3 Splitting Components to Isolate Re-Renders

// ❌ Clock ticking re-renders the entire dashboard
function Dashboard() {
  const [time, setTime] = useState(new Date());

  useEffect(() => {
    const id = setInterval(() => setTime(new Date()), 1000);
    return () => clearInterval(id);
  }, []);

  return (
    <div>
      <p>{time.toLocaleTimeString()}</p>
      <ExpensiveChart />   {/* re-renders every second! */}
      <ExpensiveTable />   {/* re-renders every second! */}
    </div>
  );
}

// ✅ Extract the clock into its own component
function Clock() {
  const [time, setTime] = useState(new Date());
  useEffect(() => {
    const id = setInterval(() => setTime(new Date()), 1000);
    return () => clearInterval(id);
  }, []);
  return <p>{time.toLocaleTimeString()}</p>;
}

function Dashboard() {
  return (
    <div>
      <Clock />            {/* only Clock re-renders every second */}
      <ExpensiveChart />   {/* unaffected */}
      <ExpensiveTable />   {/* unaffected */}
    </div>
  );
}
Architect Insight

The three most effective rendering optimizations — in order of impact and simplicity — are: (1) State colocation — move state as close to where it's used as possible. (2) Composition — pass expensive components as children to avoid re-rendering them. (3) Component splitting — extract frequently-updating state into isolated components. These are all structural changes that don't require memoization. Reach for React.memo/useMemo/useCallback only after these patterns are insufficient.
