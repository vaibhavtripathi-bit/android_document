1. React.memo Deep Dive

React.memo is a higher-order component that memoizes a component's output. If the component's props haven't changed (shallow comparison by default), React skips re-rendering and reuses the last rendered result.

// Basic usage
const ExpensiveList = React.memo(function ExpensiveList({ items, onSelect }) {
  console.log('ExpensiveList rendered');
  return (
    <ul>
      {items.map(item => (
        <li key={item.id} onClick={() => onSelect(item.id)}>
          {item.name}
        </li>
      ))}
    </ul>
  );
});

// How React.memo works internally (simplified):
function memo(Component, arePropsEqual) {
  return function MemoizedComponent(props) {
    const prevProps = useRef(props);

    if (arePropsEqual
      ? arePropsEqual(prevProps.current, props)
      : shallowEqual(prevProps.current, props)
    ) {
      return previousResult;  // skip render, reuse last output
    }

    prevProps.current = props;
    return <Component {...props} />;  // render with new props
  };
}

// Custom comparison function
const UserCard = React.memo(
  function UserCard({ user, theme }) {
    return <div className={theme}>{user.name}</div>;
  },
  (prevProps, nextProps) => {
    // Return true if props are equal (skip re-render)
    return prevProps.user.id === nextProps.user.id
      && prevProps.theme === nextProps.theme;
  }
);
React.memo Does NOT Prevent Context Re-Renders

If a memoized component consumes context via useContext, it will re-render when the context value changes regardless of its props. Context bypasses React.memo. This is a common source of confusion — wrapping a component in React.memo doesn't help if the re-render is triggered by context.

2. useMemo for Value Memoization

// useMemo caches a computed value between re-renders
function FilteredList({ items, filter }) {
  // Only recomputes when items or filter change
  const filtered = useMemo(() => {
    return items.filter(item =>
      item.name.toLowerCase().includes(filter.toLowerCase())
    );
  }, [items, filter]);

  // Preserves referential equality for downstream React.memo
  const sortedAndFiltered = useMemo(() => {
    return [...filtered].sort((a, b) => a.name.localeCompare(b.name));
  }, [filtered]);

  return <MemoizedTable data={sortedAndFiltered} />;
}

// useMemo for referential equality (object/array props)
function Chart({ data }) {
  // Without useMemo: new object every render → child always re-renders
  const chartConfig = useMemo(() => ({
    xAxis: data.map(d => d.date),
    yAxis: data.map(d => d.value),
    color: '#61dafb',
  }), [data]);

  return <MemoizedChartRenderer config={chartConfig} />;
}

// How useMemo works internally (simplified):
function useMemo(factory, deps) {
  const hook = getCurrentHook();

  if (hook.memoizedState && depsAreSame(hook.deps, deps)) {
    return hook.memoizedState;  // return cached value
  }

  const value = factory();       // recompute
  hook.memoizedState = value;
  hook.deps = deps;
  return value;
}

3. useCallback for Function Memoization

// useCallback is useMemo for functions
// useCallback(fn, deps) === useMemo(() => fn, deps)

function ParentList({ items }) {
  const [selectedId, setSelectedId] = useState(null);

  // Without useCallback: new function every render
  // → MemoizedItem receives new onSelect prop → re-renders
  const handleSelect = useCallback((id) => {
    setSelectedId(id);
  }, []);  // stable reference — never changes

  return (
    <ul>
      {items.map(item => (
        <MemoizedItem
          key={item.id}
          item={item}
          isSelected={item.id === selectedId}
          onSelect={handleSelect}  // stable reference
        />
      ))}
    </ul>
  );
}

const MemoizedItem = React.memo(function Item({ item, isSelected, onSelect }) {
  return (
    <li
      className={isSelected ? 'selected' : ''}
      onClick={() => onSelect(item.id)}
    >
      {item.name}
    </li>
  );
});
useCallback Only Helps with Memoized Children

useCallback is pointless if the child component is NOT wrapped in React.memo. Without React.memo, the child re-renders when its parent re-renders regardless of prop stability. useCallback preserves referential equality of the function, but that only matters if something is checking for equality (i.e., React.memo's shallow comparison).

4. When to Memoize

Scenario Memoize? Reason
Expensive computation (sorting 10K items) Yes — useMemo Avoids redundant O(n log n) work
Object/array prop to React.memo child Yes — useMemo Preserves referential equality
Callback prop to React.memo child Yes — useCallback Preserves referential equality
Dependency of useEffect/useMemo Yes — useMemo/useCallback Prevents effect re-runs
Simple string/number computation No Memoization overhead > computation cost
Callback to non-memoized child No Child re-renders anyway (parent re-render)
JSX elements Rarely Use composition pattern instead
Context value Yes — useMemo Prevents all consumers from re-rendering 
// Decision flowchart:
//
// Is the computation expensive? (>1ms, O(n²), etc.)
//   → Yes: useMemo
//
// Is this value/function passed to a React.memo child?
//   → Yes: useMemo/useCallback
//   → No: don't memoize
//
// Is this value used in a dependency array?
//   → Yes: useMemo/useCallback to stabilize
//   → No: don't memoize
//
// Can you restructure to avoid the need? (composition, state colocation)
//   → Yes: restructure instead of memoizing

5. The Over-Memoization Anti-Pattern

// ❌ Over-memoization: every value and function memoized
function UserCard({ user }) {
  const fullName = useMemo(
    () => `${user.firstName} ${user.lastName}`,
    [user.firstName, user.lastName]
  );  // ❌ String concatenation is trivial — memoization costs more

  const handleClick = useCallback(() => {
    console.log(user.id);
  }, [user.id]);  // ❌ No memoized child consumes this

  const style = useMemo(
    () => ({ color: user.isActive ? 'green' : 'gray' }),
    [user.isActive]
  );  // ❌ No memoized child receives this

  return (
    <div style={style} onClick={handleClick}>
      {fullName}
    </div>
  );
}

// ✅ Clean version — no unnecessary memoization
function UserCard({ user }) {
  const fullName = `${user.firstName} ${user.lastName}`;

  return (
    <div
      style={{ color: user.isActive ? 'green' : 'gray' }}
      onClick={() => console.log(user.id)}
    >
      {fullName}
    </div>
  );
}
Architect Insight: The Cost of Memoization

Every useMemo/useCallback has a cost: (1) memory to store the cached value and dependency array, (2) CPU to compare dependencies on every render, (3) code complexity and cognitive load, (4) prevents garbage collection of cached values. For trivial computations, this overhead exceeds the savings. The rule of thumb: profile first, memoize only when you can measure a difference. Premature memoization is a form of premature optimization.

6. React Compiler (Forget)

The React Compiler (formerly "React Forget") is an ahead-of-time compiler that automatically adds memoization to React components. It eliminates the need for manual useMemo, useCallback, and React.memo.

// What you write (no manual memoization):
function ProductList({ products, onSelect }) {
  const sorted = products.sort((a, b) => a.price - b.price);
  const handleClick = (id) => onSelect(id);

  return (
    <ul>
      {sorted.map(p => (
        <li key={p.id} onClick={() => handleClick(p.id)}>
          {p.name} — ${p.price}
        </li>
      ))}
    </ul>
  );
}

// What the compiler outputs (conceptually):
function ProductList({ products, onSelect }) {
  const $ = useMemoCache(4);  // compiler-managed cache slots

  let sorted;
  if ($[0] !== products) {
    sorted = products.sort((a, b) => a.price - b.price);
    $[0] = products;
    $[1] = sorted;
  } else {
    sorted = $[1];
  }

  let jsx;
  if ($[2] !== sorted || $[3] !== onSelect) {
    jsx = (
      <ul>
        {sorted.map(p => (
          <li key={p.id} onClick={() => onSelect(p.id)}>
            {p.name} — ${p.price}
          </li>
        ))}
      </ul>
    );
    $[2] = sorted;
    $[3] = onSelect;
  }

  return jsx;
}
React Compiler Requirements

The compiler requires components to follow the Rules of React: (1) components must be pure (same inputs → same output), (2) hooks must follow the rules of hooks, (3) no mutation of props or state. The compiler uses static analysis to determine what can be safely memoized. Components that violate these rules may produce incorrect behavior with the compiler. This is why React has always emphasized purity — it enables compiler optimizations.

7. Memoization & Garbage Collection

// Memoization holds references, preventing garbage collection

// Problem: useMemo keeps old data alive
function DataViewer({ dataset }) {
  const processed = useMemo(() => {
    return heavyProcessing(dataset);  // 100MB result
  }, [dataset]);

  // Even when DataViewer unmounts, the 100MB result stays in memory
  // until React's fiber cleanup runs (which may be deferred)

  return <Chart data={processed} />;
}

// React.memo keeps the previous props alive for comparison
const HeavyComponent = React.memo(({ largeData }) => {
  // React.memo stores prevProps internally
  // largeData from the previous render is kept alive
  // until the next render provides new props
  return <div>{largeData.summary}</div>;
});

// Mitigation: use WeakRef for large cached values (advanced)
function useWeakMemo(factory, deps) {
  const ref = useRef({ weakRef: null, deps: null });

  if (!depsAreSame(ref.current.deps, deps) || !ref.current.weakRef?.deref()) {
    const value = factory();
    ref.current = { weakRef: new WeakRef(value), deps };
    return value;
  }

  return ref.current.weakRef.deref();
}

8. Profiling Memoization Effectiveness

// Use React Profiler to measure actual vs base duration
function onRender(id, phase, actualDuration, baseDuration) {
  const savings = baseDuration - actualDuration;
  const savingsPercent = ((savings / baseDuration) * 100).toFixed(1);

  console.log(
    `${id}: actual=${actualDuration.toFixed(1)}ms, ` +
    `base=${baseDuration.toFixed(1)}ms, ` +
    `saved=${savingsPercent}%`
  );

  // If savings < 10%, memoization isn't helping much
  // If savings > 50%, memoization is very effective
}

Architect Insight: Memoization Strategy at Scale

For large codebases, establish a memoization policy: (1) Don't memoize by default — write clean code first. (2) Profile before optimizing — measure actual render times. (3) Prefer structural solutions (composition, state colocation) over memoization. (4) When memoizing, document WHY (what performance issue it solves). (5) Adopt the React Compiler when stable — it eliminates manual memoization decisions entirely. (6) Review memoization in code reviews — challenge unnecessary useMemo/useCallback.
