2. Streaming SSR

// React 18: renderToPipeableStream (Node.js)
import { renderToPipeableStream } from 'react-dom/server';

app.get('*', (req, res) => {
  const { pipe, abort } = renderToPipeableStream(
    <App url={req.url} />,
    {
      bootstrapScripts: ['/static/js/main.js'],

      onShellReady() {
        // Shell (everything outside Suspense) is ready
        // Send it immediately — don't wait for all data
        res.statusCode = 200;
        res.setHeader('Content-Type', 'text/html');
        pipe(res);
      },

      onShellError(error) {
        // Shell failed — fall back to CSR
        res.statusCode = 500;
        res.send('<!doctype html><html>...fallback...</html>');
      },

      onAllReady() {
        // Everything (including Suspense boundaries) is ready
        // Useful for crawlers that need complete HTML
      },

      onError(error) {
        console.error(error);
      },
    }
  );

  setTimeout(abort, 10000);  // abort if rendering takes too long
});

// The App uses Suspense to define streaming boundaries:
function App() {
  return (
    <html>
      <body>
        <Header />           {/* in shell — sent immediately */}
        <HeroSection />      {/* in shell — sent immediately */}
        <Suspense fallback={<CommentsSkeleton />}>
          <Comments />       {/* streams in when data is ready */}
        </Suspense>
        <Suspense fallback={<RecommendationsSkeleton />}>
          <Recommendations /> {/* streams in independently */}
        </Suspense>
      </body>
    </html>
  );
}
Streaming SSR Timeline: Time ──────────────────────────────────────────────────▶ Server: [render shell]──send──▶[render comments]──send──▶[render recs]──send──▶ │ │ │ Browser: [display │ │ │ shell] │ │ │ [display header, hero] │ │ [replace skeleton │ │ with comments] │ │ [replace skeleton │ with recommendations] │ [hydrate interactive parts] User sees content progressively — no blank screen waiting for all data

3. Selective Hydration

// React 18 selective hydration:
// Instead of hydrating the entire page at once,
// React hydrates Suspense boundaries independently
// and prioritizes boundaries the user interacts with

function Page() {
  return (
    <div>
      <Header />  {/* hydrated first (in shell) */}

      <Suspense fallback={<Skeleton />}>
        <Sidebar />  {/* hydrated independently */}
      </Suspense>

      <Suspense fallback={<Skeleton />}>
        <MainContent />  {/* hydrated independently */}
      </Suspense>

      <Suspense fallback={<Skeleton />}>
        <Comments />  {/* hydrated independently */}
      </Suspense>
    </div>
  );
}

// If user clicks on Comments before it's hydrated:
// 1. React records the click event
// 2. Prioritizes hydrating Comments boundary
// 3. After hydration, replays the click event
// User doesn't need to click again!
Architect Insight

Selective hydration fundamentally changes the performance model of SSR. Previously, hydration was an all-or-nothing operation — the entire page had to hydrate before anything was interactive. With selective hydration, each Suspense boundary hydrates independently, and user interactions automatically prioritize the relevant boundary. This means the page becomes interactive progressively, with the most important parts (those the user is trying to interact with) hydrating first.

6. React Server Components

// Server Components: render on server, zero client JS
// The ultimate SSR performance optimization

// Server Component (default in App Router)
async function ProductPage({ params }) {
  const product = await db.product.findUnique({ where: { id: params.id } });

  return (
    <div>
      <h1>{product.name}</h1>
      <p>{product.description}</p>
      <ProductImage src={product.image} />  {/* Server Component */}
      <AddToCartButton productId={product.id} />  {/* Client Component */}
    </div>
  );
}

// Only AddToCartButton's JS is sent to the client
// ProductPage, h1, p, ProductImage = zero client JS
Server Components Bundle Impact: Traditional SSR: Server renders HTML → Client downloads ALL component JS → Hydrates everything Bundle: [ProductPage + ProductImage + AddToCartButton + db client + ...] React Server Components: Server renders RSC payload → Client only downloads Client Component JS Bundle: [AddToCartButton only] ProductPage code: NOT in client bundle (server only) ProductImage code: NOT in client bundle (server only) Database client: NOT in client bundle (server only) AddToCartButton code: IN client bundle (needs interactivity) Result: 70-90% reduction in client JS for content-heavy pages

7. SSG vs SSR vs ISR

Strategy When HTML is Generated TTFB Freshness Best For
SSG Build time Fastest (CDN cached) Stale until rebuild Blog, docs, marketing
SSR Every request Slowest (server render) Always fresh Personalized, real-time
ISR Build + background revalidation Fast (cached, revalidated) Stale for revalidate period E-commerce, news
Streaming SSR Every request (progressive) Fast (shell immediate) Always fresh Complex dashboards 
// Next.js: choosing rendering strategy per route

// SSG (default for pages without dynamic data)
export default function AboutPage() {
  return <div>About us</div>;
}

// SSR (dynamic data on every request)
export const dynamic = 'force-dynamic';
export default async function DashboardPage() {
  const data = await fetchDashboardData();
  return <Dashboard data={data} />;
}

// ISR (revalidate every 60 seconds)
export const revalidate = 60;
export default async function ProductPage({ params }) {
  const product = await fetchProduct(params.id);
  return <Product data={product} />;
}

8. Caching Strategies

// HTTP caching headers for SSR responses

// Static pages: cache aggressively
res.setHeader('Cache-Control', 'public, max-age=31536000, immutable');

// SSR pages: stale-while-revalidate
res.setHeader('Cache-Control', 'public, s-maxage=60, stale-while-revalidate=300');
// CDN serves cached version for 60s
// After 60s: serves stale while revalidating in background (up to 300s)

// Personalized pages: no CDN cache, browser cache only
res.setHeader('Cache-Control', 'private, no-cache, no-store');

// Vary header: cache different versions based on request headers
res.setHeader('Vary', 'Accept-Encoding, Cookie');

// Next.js fetch caching
// Cached by default (equivalent to getStaticProps)
const data = await fetch('https://api.example.com/data');

// Revalidate every 60 seconds
const data = await fetch('https://api.example.com/data', {
  next: { revalidate: 60 },
});

// No cache (fresh on every request)
const data = await fetch('https://api.example.com/data', {
  cache: 'no-store',
});

// Tag-based cache invalidation
const data = await fetch('https://api.example.com/products', {
  next: { tags: ['products'] },
});
// Invalidate: revalidateTag('products')

9. TTFB Optimization

// TTFB (Time to First Byte) optimization strategies:

// 1. Streaming: send shell HTML immediately
// Don't wait for all data — stream as it becomes available

// 2. CDN caching: cache SSR output at the edge
// s-maxage + stale-while-revalidate for dynamic pages

// 3. Database optimization
// - Connection pooling (don't create new connections per request)
// - Query optimization (indexes, N+1 prevention)
// - Read replicas (distribute read load)

// 4. Edge rendering for latency-sensitive routes
// Auth checks, personalization, geo-routing at the edge

// 5. Parallel data fetching on the server
async function Page() {
  // ❌ Sequential: TTFB = t1 + t2 + t3
  const user = await getUser();
  const posts = await getPosts(user.id);
  const comments = await getComments(posts[0].id);

  // ✅ Parallel: TTFB = max(t1, t2)
  const [user, posts] = await Promise.all([getUser(), getPosts()]);
}

// 6. Precompute at build time (SSG/ISR)
// If data doesn't change per-request, generate at build time

// 7. React cache() for request deduplication
import { cache } from 'react';

const getUser = cache(async (id) => {
  return db.user.findUnique({ where: { id } });
});
// Multiple components calling getUser(42) in the same request
// → only one database query (deduplicated)
Architect Insight: Rendering Strategy Decision Tree

Choose your rendering strategy based on data characteristics: (1) Static data (docs, blog) → SSG. (2) Slowly changing data (product catalog, news) → ISR with revalidation. (3) User-specific data (dashboard, profile) → SSR with streaming. (4) Real-time data (chat, live scores) → SSR shell + client-side updates. (5) Mixed (most apps) → per-route strategy. The goal is to push as much as possible to static/cached while keeping dynamic content fresh.
