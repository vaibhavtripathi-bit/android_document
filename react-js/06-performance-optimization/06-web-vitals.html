1. Core Web Vitals Overview

Core Web Vitals are Google's user-centric performance metrics that measure loading, interactivity, and visual stability. They directly impact search ranking and represent the three most important aspects of user experience.

Core Web Vitals (2024): Metric What It Measures Good Needs Work Poor ──────────────────────────────────────────────────────────────────── LCP Loading performance ≤ 2.5s ≤ 4.0s > 4.0s INP Interactivity ≤ 200ms ≤ 500ms > 500ms CLS Visual stability ≤ 0.1 ≤ 0.25 > 0.25 Timeline: Page Load ──────────────────────────────────────────────▶ [─── LCP ───] Largest element painted (hero image, heading) │ ▼ [─ INP ─] Worst interaction delay (click, tap, keypress) │ ▼ [──── CLS ────] Cumulative unexpected layout shifts

2. Largest Contentful Paint (LCP)

LCP measures when the largest content element in the viewport becomes visible. It's the primary loading metric — users perceive a page as loaded when the main content appears.

2.1 What Counts as LCP

Element Type LCP Candidate Notes
<img> Yes Most common LCP element
<video> poster Yes Poster image, not video content
CSS background-image Yes Via url() only
Text blocks (<h1>, <p>) Yes If largest in viewport
<svg> No SVGs are excluded

2.2 LCP Optimization

// 1. Preload the LCP image
<link rel="preload" href="/hero.avif" as="image"
      fetchpriority="high" type="image/avif" />

// 2. Don't lazy-load the LCP image
<img src="hero.jpg" loading="eager" fetchpriority="high" />

// 3. Inline critical CSS (avoid render-blocking stylesheets)
<style>
  .hero { ... }  /* critical above-fold styles */
</style>

      onload="this.media='all'" />

// 5. Avoid client-side rendering for LCP content

3. Interaction to Next Paint (INP)

INP measures the latency of all interactions (clicks, taps, key presses) throughout the page lifecycle and reports the worst one (at the 98th percentile). It replaced FID (First Input Delay) in March 2024.

INP Breakdown: User Click ──▶ [Input Delay] ──▶ [Processing Time] ──▶ [Presentation Delay] ──▶ Next Paint Input Delay: Time from click to event handler start Caused by: long tasks blocking main thread Processing Time: Time for event handlers to run Caused by: expensive state updates, heavy computation Presentation Delay: Time from handler completion to next paint Caused by: expensive re-renders, layout thrashing Total INP = Input Delay + Processing Time + Presentation Delay Target: ≤ 200ms total 
// Optimizing INP in React:

// 1. Use startTransition for non-urgent updates
function SearchInput() {
  const [query, setQuery] = useState('');
  const [results, setResults] = useState([]);

  function handleChange(e) {
    setQuery(e.target.value);  // urgent: update input immediately
    startTransition(() => {
      setResults(filterResults(e.target.value));  // non-urgent: can be deferred
    });
  }
}

// 2. Debounce expensive operations
const debouncedSearch = useMemo(
  () => debounce((query) => setResults(search(query)), 300),
  []
);

// 3. Use Web Workers for heavy computation
const worker = new Worker('./sort-worker.js');
worker.postMessage({ data: largeDataset, sortBy: 'name' });
worker.onmessage = (e) => setSortedData(e.data);

// 4. Break up long tasks with scheduler.yield()
async function processItems(items) {
  for (const item of items) {
    processItem(item);
    if (navigator.scheduling?.isInputPending()) {
      await scheduler.yield();  // yield to pending input
    }
  }
}

4. Cumulative Layout Shift (CLS)

CLS measures the total of all unexpected layout shifts during the page's lifetime. A layout shift occurs when a visible element changes position without user interaction.

// Common CLS causes and fixes:

// 1. Images without dimensions → reserve space
// ❌
<img src="photo.jpg" />
// ✅
<img src="photo.jpg" width="800" height="600" />
// Or use CSS aspect-ratio:
img { aspect-ratio: 4 / 3; width: 100%; height: auto; }

// 2. Dynamically injected content → reserve space
// ❌ Ad loads and pushes content down
// ✅ Reserve space with min-height
.ad-slot { min-height: 250px; }

// 3. Font swap causing text reflow
// Use font-display: optional (no swap) or
// size-adjust to match fallback metrics:
@font-face {
  font-family: 'CustomFont';
  src: url('font.woff2');
  font-display: swap;
  size-adjust: 105%;       // match fallback font metrics
  ascent-override: 90%;
  descent-override: 20%;
}

// 4. Dynamic content insertion
// ❌ Insert banner at top, pushing everything down
// ✅ Use CSS transform (doesn't cause layout shift)
.notification {
  transform: translateY(-100%);
  transition: transform 0.3s;
}
.notification.visible {
  transform: translateY(0);
}

5. Measurement APIs

// web-vitals library (Google's official library)
import { onLCP, onINP, onCLS } from 'web-vitals';

onLCP((metric) => {
  console.log('LCP:', metric.value, 'ms');
  console.log('Element:', metric.entries[0]?.element);
  sendToAnalytics({ name: 'LCP', value: metric.value, id: metric.id });
});

onINP((metric) => {
  console.log('INP:', metric.value, 'ms');
  console.log('Event:', metric.entries[0]?.name);
});

onCLS((metric) => {
  console.log('CLS:', metric.value);
  console.log('Shifts:', metric.entries.length);
});

// PerformanceObserver API (low-level)
const observer = new PerformanceObserver((list) => {
  for (const entry of list.getEntries()) {
    if (entry.entryType === 'largest-contentful-paint') {
      console.log('LCP candidate:', entry.startTime, entry.element);
    }
    if (entry.entryType === 'layout-shift' && !entry.hadRecentInput) {
      console.log('Layout shift:', entry.value, entry.sources);
    }
  }
});

observer.observe({ type: 'largest-contentful-paint', buffered: true });
observer.observe({ type: 'layout-shift', buffered: true });

6. RUM vs Lab Testing

Aspect Real User Monitoring (RUM) Lab Testing
Data source Real users in production Simulated environment
Network Actual user networks (3G, WiFi, etc.) Throttled simulation
Devices Real devices (phones, tablets, desktops) Emulated device
INP Yes (real interactions) No (no real user interactions)
Debugging Harder (aggregate data) Easier (reproducible)
Tools web-vitals, CrUX, SpeedCurve Lighthouse, WebPageTest, DevTools
Google ranking Yes (CrUX data used) No (informational only)
Architect Insight

Lab testing (Lighthouse) is for debugging and development. RUM (Chrome User Experience Report / CrUX) is what Google actually uses for search ranking. A perfect Lighthouse score doesn't guarantee good real-world performance — users on slow networks with old devices will have different experiences. Implement RUM with the web-vitals library, track p75 values, and set up alerts for regressions.

7. Lighthouse Scoring

// Lighthouse performance score breakdown (v11):
// FCP (First Contentful Paint):     10% weight
// SI  (Speed Index):                10% weight
// LCP (Largest Contentful Paint):   25% weight
// TBT (Total Blocking Time):        30% weight
// CLS (Cumulative Layout Shift):    25% weight

// TBT correlates with INP in lab testing
// (INP can't be measured in lab — no real user interactions)

// Running Lighthouse programmatically:
const lighthouse = require('lighthouse');
const chromeLauncher = require('chrome-launcher');

async function runLighthouse(url) {
  const chrome = await chromeLauncher.launch({ chromeFlags: ['--headless'] });
  const result = await lighthouse(url, {
    port: chrome.port,
    onlyCategories: ['performance'],
  });
  await chrome.kill();

  console.log('Score:', result.lhr.categories.performance.score * 100);
  console.log('LCP:', result.lhr.audits['largest-contentful-paint'].numericValue);
  console.log('TBT:', result.lhr.audits['total-blocking-time'].numericValue);
  console.log('CLS:', result.lhr.audits['cumulative-layout-shift'].numericValue);
}

8. React-Specific Impact

// React's impact on Core Web Vitals:

// LCP: Client-side rendering delays LCP
// CSR: HTML loads → JS loads → JS executes → data fetches → LCP renders
// SSR: HTML loads → LCP is in initial HTML (much faster)
// Fix: Use SSR/SSG for LCP content

// INP: Heavy re-renders block the main thread
// Large component trees re-rendering on state change → long tasks
// Fix: startTransition, useDeferredValue, virtualization, memoization

// CLS: Hydration mismatches cause layout shifts
// Server HTML → hydration changes layout → CLS
// Fix: Ensure server and client render identical initial HTML

// React 18 improvements for web vitals:
// - Automatic batching: fewer re-renders → better INP
// - startTransition: non-urgent updates don't block input → better INP
// - Streaming SSR: faster TTFB → better LCP
// - Selective hydration: interactive faster → better INP

// Measuring React-specific performance:
import { Profiler } from 'react';

<Profiler id="App" onRender={(id, phase, actualDuration) => {
  if (actualDuration > 50) {
    // Long task — will impact INP
    reportSlowRender({ id, phase, duration: actualDuration });
  }
}}>
  <App />
</Profiler>
Architect Insight: Performance Budget

Set concrete performance budgets: LCP < 2.5s, INP < 200ms, CLS < 0.1. Monitor in production with RUM. Set up CI checks with Lighthouse CI to catch regressions before deploy. Track p75 (75th percentile) — this is what Google uses for ranking. A site with great median performance but terrible p75 will still rank poorly.
