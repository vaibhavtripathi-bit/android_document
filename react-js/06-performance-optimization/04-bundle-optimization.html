Analysis Checklist

When analyzing bundles, look for: (1) large libraries that could be replaced (moment.js → date-fns), (2) duplicate dependencies across chunks, (3) polyfills for browsers you don't support, (4) dev-only code leaking into production (PropTypes, debug logs), (5) CSS-in-JS runtime size, (6) chunks that are too large (>250KB gzipped) or too small (<5KB — merge overhead), (7) unused exports from barrel files.

2. Tree Shaking

Tree shaking is dead code elimination for ES modules. Bundlers analyze import/export statements to determine which exports are actually used and remove the rest.

// ✅ ES modules: tree-shakeable
import { debounce } from 'lodash-es';
// Bundler includes only debounce, not the entire lodash library

// ❌ CommonJS: NOT tree-shakeable
const { debounce } = require('lodash');
// Bundler must include the entire lodash library
// because require() is dynamic — can't statically analyze

// ❌ Namespace import: may prevent tree shaking
import * as _ from 'lodash-es';
_.debounce(fn, 300);
// Some bundlers can tree-shake this, but named imports are safer

2.1 The sideEffects Flag

// package.json — tells bundler which files have side effects
{
  "name": "my-library",
  "sideEffects": false  // no files have side effects → safe to tree-shake everything
}

// Or specify files with side effects:
{
  "sideEffects": [
    "*.css",              // CSS imports are side effects
    "./src/polyfills.js"  // polyfill imports are side effects
  ]
}

// What is a side effect?
// Code that runs just by being imported:
import './styles.css';           // side effect: adds styles to DOM
import './polyfills';             // side effect: modifies globals
window.myGlobal = 'value';        // side effect: modifies global

// NOT a side effect (safe to tree-shake):
export function add(a, b) { return a + b; }
export const PI = 3.14159;
Tree Shaking in Action: Source (utils.js): After Tree Shaking: ┌──────────────────────────┐ ┌──────────────────────────┐ │ export function add() {} │ ──used──▶│ export function add() {} │ │ export function sub() {} │ ──dead──▶│ │ │ export function mul() {} │ ──dead──▶│ │ │ export function div() {} │ ──used──▶│ export function div() {} │ │ export const PI = 3.14 │ ──dead──▶│ │ └──────────────────────────┘ └──────────────────────────┘ app.js: import { add, div } from './utils'; Only add and div are included in the final bundle.

3. Code Splitting Strategies

Strategy Split Point Impact Complexity
Route-based Each page/route Highest (reduces initial load) Low
Component-based Heavy components (modals, editors) Medium Medium
Library-based Large dependencies (charts, maps) Medium Low
Vendor splitting node_modules vs app code Caching benefit Low
Feature-based Feature flags / A/B tests Variable High 
// Optimal chunk strategy for a typical React app:

// 1. Framework chunk (rarely changes → long cache)
//    react, react-dom, react-router → react-vendor.js

// 2. Vendor chunk (changes occasionally)
//    other node_modules → vendor.js

// 3. Common chunk (shared between routes)
//    components/utils used by 2+ routes → common.js

// 4. Route chunks (change frequently)
//    each route → dashboard.js, settings.js, etc.

// Webpack config:
optimization: {
  splitChunks: {
    chunks: 'all',
    cacheGroups: {
      react: {
        test: /[\\/]node_modules[\\/](react|react-dom|react-router)[\\/]/,
        name: 'react-vendor',
        priority: 30,
      },
      vendor: {
        test: /[\\/]node_modules[\\/]/,
        name: 'vendor',
        priority: 20,
      },
      common: {
        minChunks: 2,
        name: 'common',
        priority: 10,
        reuseExistingChunk: true,
      },
    },
  },
}

4. Dynamic Imports & Lazy Loading

// Route-level lazy loading
const Dashboard = lazy(() => import('./pages/Dashboard'));

// Component-level lazy loading (heavy features)
const RichTextEditor = lazy(() => import('./components/RichTextEditor'));
const ChartLibrary = lazy(() => import('./components/Chart'));

// Conditional lazy loading (feature flags)
const NewFeature = lazy(() => {
  if (featureFlags.newDashboard) {
    return import('./features/NewDashboard');
  }
  return import('./features/OldDashboard');
});

// Library-level lazy loading
async function generatePDF(data) {
  const { jsPDF } = await import('jspdf');  // 200KB loaded only when needed
  const doc = new jsPDF();
  doc.text(data.content, 10, 10);
  doc.save('document.pdf');
}

5. Dead Code Elimination

// Terser/UglifyJS removes dead code during minification

// 1. Unreachable code
function example() {
  return 42;
  console.log('never reached');  // removed by minifier
}

// 2. process.env.NODE_ENV checks
if (process.env.NODE_ENV === 'development') {
  // This entire block is removed in production builds
  // because webpack replaces process.env.NODE_ENV with 'production'
  // and the minifier removes the dead branch
  enableDevTools();
}

// 3. __DEV__ flag (used by React internally)
if (__DEV__) {
  validateProps(props);  // removed in production
}

// 4. Barrel file problem
// ❌ Barrel files can prevent tree shaking:
// components/index.ts
export { Button } from './Button';
export { Modal } from './Modal';     // 50KB
export { DataGrid } from './DataGrid'; // 100KB

// Importing just Button may pull in Modal and DataGrid
// if the bundler can't prove they're side-effect-free
import { Button } from './components';

// ✅ Direct imports bypass barrel files:
import { Button } from './components/Button';
Barrel File Anti-Pattern

Barrel files (index.ts that re-exports from multiple modules) are convenient but can defeat tree shaking. If any re-exported module has side effects, the bundler must include it even if the specific export isn't used. For large component libraries, prefer direct imports (import { Button } from './components/Button') over barrel imports (import { Button } from './components').

6. Import Cost Awareness

// Use 'import-cost' VS Code extension to see sizes inline
// Or check bundlephobia.com before adding dependencies

// Common offenders:
import moment from 'moment';           // 72KB gzipped (with locales: 232KB)
import _ from 'lodash';                // 71KB gzipped
import { Chart } from 'chart.js';      // 63KB gzipped
import * as Yup from 'yup';           // 14KB gzipped

// Lighter alternatives:
import { format } from 'date-fns';     // 3KB (just format function)
import { debounce } from 'lodash-es';  // 1KB (tree-shakeable)
import { z } from 'zod';               // 13KB (but type-safe)

8. Module/Nomodule Pattern

// Serve modern JS to modern browsers, legacy JS to old browsers
<!-- Modern browsers load this (smaller, no polyfills) -->
<script type="module" src="/app.modern.js"></script>

<!-- Legacy browsers load this (transpiled, with polyfills) -->
<script nomodule src="/app.legacy.js"></script>

// Modern browsers understand type="module" and ignore nomodule
// Legacy browsers ignore type="module" and load nomodule

// Size difference can be significant:
// Modern: 150KB (ES2020, no polyfills, native async/await)
// Legacy: 250KB (ES5, core-js polyfills, regenerator-runtime)

// Vite does this automatically with build.target
export default defineConfig({
  build: {
    target: 'es2020',  // modern output
  },
});
Architect Insight: Bundle Budget

Establish a bundle budget for your application: (1) Initial JS: <200KB gzipped for fast 3G load. (2) Per-route chunk: <50KB gzipped. (3) Total JS (all routes): <500KB gzipped. (4) CSS: <50KB gzipped. Enforce budgets in CI with tools like size-limit or webpack's performance.maxAssetSize. A bundle budget is the most effective way to prevent bundle bloat over time — it catches regressions before they ship.
