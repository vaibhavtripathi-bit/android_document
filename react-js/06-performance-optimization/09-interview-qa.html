Rendering Performance (Questions 1–12)

1. What are the three things that cause a React component to re-render? Senior

(1) State change — calling a state setter (useState, useReducer) triggers a re-render of the component and its entire subtree. (2) Parent re-render — when a parent component re-renders, all children re-render regardless of whether their props changed (unless wrapped in React.memo). (3) Context change — when a context value changes, all components consuming that context re-render, even if wrapped in React.memo.

Common misconception: props changing does NOT directly cause re-renders. The re-render happens because the parent re-rendered; the child receives new props as a consequence.

2. Why does React re-render the entire subtree when a component's state changes? L6

React cannot statically determine which descendants depend on a component's state without actually calling their render functions. Children might depend on the parent's state through: props, context, refs, external stores, or module-level variables. The safest default is to re-render the entire subtree and let the reconciliation algorithm determine what DOM changes are actually needed.

This is a deliberate design choice: rendering (calling component functions) is relatively cheap — it's just JavaScript computation. The expensive part is DOM mutations, which only happen when the diff finds actual changes. React optimizes for correctness over minimal rendering.

3. Explain the difference between React "rendering" and browser "rendering." L6

React rendering means calling component functions to produce a new virtual element tree, then diffing it against the current tree. This is pure JavaScript computation — no DOM interaction. Browser rendering is the pipeline of DOM → CSSOM → Layout → Paint → Composite that produces pixels on screen.

A React render does NOT necessarily cause a browser render. If the diff finds no changes, React commits nothing to the DOM, and the browser has nothing to repaint. Conversely, a single React render can cause multiple browser repaints if it triggers layout changes.

4. How does the React Profiler API work? What's the difference between actualDuration and baseDuration? L6

The <Profiler> component wraps a subtree and calls an onRender callback after each commit with timing data. actualDuration is the time React spent rendering components that actually re-rendered (respecting memoization bail-outs). baseDuration is the estimated time if every component in the subtree re-rendered without any memoization.

The ratio actualDuration / baseDuration measures memoization effectiveness. If they're similar, memoization isn't helping. If actualDuration is much smaller, memoization is effectively preventing unnecessary work. Other callback parameters include phase ("mount" or "update"), startTime, and commitTime.

5. What are the three most effective rendering optimizations that don't require memoization? L6

(1) State colocation — move state as close to where it's used as possible. If only a search input needs the search query state, put it in the search input component, not a parent. (2) Composition (children as props) — pass expensive components as children to a wrapper that owns the frequently-changing state. Since children is created in the parent's render, it maintains the same reference when the wrapper re-renders. (3) Component splitting — extract frequently-updating state into its own component so it doesn't cause siblings to re-render.

These structural patterns are more effective than memoization because they prevent the re-render from happening in the first place, rather than bailing out during the render.

6. How does the composition pattern (children as props) prevent unnecessary re-renders? L6

When you pass a component as children to a wrapper, the React element for that child is created in the parent's render scope, not the wrapper's. When the wrapper re-renders (due to its own state change), the children prop is the same React element reference (because the parent didn't re-render). React sees the same reference and skips re-rendering the children subtree.

This works because React elements are immutable objects — if the reference is the same, the content is guaranteed to be the same. This is essentially free memoization without any React.memo or useMemo.

7. How do you use React DevTools Profiler to identify performance bottlenecks? Senior

(1) Open React DevTools Profiler tab. (2) Click Record, perform the slow interaction, click Stop. (3) Flamegraph view: each bar is a component render; width = duration; color indicates speed (gray = didn't render, green = fast, yellow/red = slow). (4) Ranked view: sorts components by render duration, showing the most expensive first. (5) Look for: components rendering that shouldn't be (gray = good), wide yellow/red bars (expensive renders), and components rendering on every interaction.

Enable "Highlight updates when components render" in settings to see real-time re-render visualization (blue flashes). Also enable "Record why each component rendered" to see the specific trigger for each re-render.

8. What is the 16ms frame budget and how does it relate to React rendering? Senior

At 60fps, each frame has ~16.67ms for all work: JavaScript execution, layout, paint, and composite. If React rendering + DOM updates + browser paint exceeds 16ms, the frame is dropped, causing visible jank. React's concurrent mode helps by splitting work across frames, but individual component renders should still be fast.

For interactions (INP), the total budget is 200ms: input delay + processing time + presentation delay. React's startTransition helps by marking non-urgent updates as low priority, allowing the browser to process input events between render chunks.

9. How does startTransition improve rendering performance? L6

startTransition marks a state update as non-urgent (a "transition"). React can interrupt transition renders to handle urgent updates (user input). This means: (1) typing in a search input remains responsive even while a large list re-renders in the background, (2) the old UI stays visible during the transition (no Suspense fallback), (3) React can abandon stale transition renders if a new one starts.

Internally, transitions use a lower priority lane in React's scheduler. Urgent updates (setState outside transition) get the highest priority lane and are processed immediately. This is the key mechanism for maintaining responsiveness during expensive renders.

10. When should you use useDeferredValue vs startTransition? L6

startTransition wraps a state update — you control when the transition starts. Use it when you own the state setter (e.g., in an event handler). useDeferredValue wraps a value — React defers updating the value during urgent renders. Use it when you receive a value as a prop and can't control the state update.

Example: for a search input, use startTransition to wrap setSearchResults. For a child component receiving query as a prop, use useDeferredValue(query) to defer expensive re-renders based on that prop. Both achieve similar results but from different perspectives (producer vs consumer of the value).

11. How do you identify and fix unnecessary re-renders caused by context? L6

Identify: Use React DevTools "Record why each component rendered" — it shows "context changed" as the reason. Or use why-did-you-render library which logs context-triggered re-renders. Fix: (1) Split context — separate frequently-changing values (e.g., theme toggle) from rarely-changing ones (e.g., user data). (2) useMemo the context value to prevent new object references on every provider render. (3) Use context selectors (via use(context) in React 19 or useContextSelector library). (4) Move frequently-changing state to a dedicated state management library (Zustand, Jotai) that supports selectors.

12. Design a performance monitoring strategy for a large React application. Architect

Client-side: (1) web-vitals library for CWV (LCP, INP, CLS) sent to analytics. (2) React Profiler in production (onRender callback) for component-level timing. (3) Long task detection via PerformanceObserver for main thread blocking. (4) Memory monitoring via performance.memory API. CI/CD: (5) Lighthouse CI on every PR with performance budgets. (6) Bundle size checks with size-limit. Production: (7) RUM dashboard with p50/p75/p95 for all CWV metrics. (8) Alerts for regressions (e.g., p75 LCP > 3s). (9) A/B testing framework for measuring performance impact of changes. Track metrics by route, device type, and network speed for actionable insights.

Memoization (Questions 13–22)

13. When should you use React.memo? When should you NOT? L6

Use React.memo when: (1) the component renders often with the same props (e.g., list items where parent re-renders frequently), (2) the component is expensive to render (complex calculations, large DOM trees), (3) the component is a leaf node that rarely needs to update. Don't use when: (1) the component's props change on every render anyway (memo check is wasted), (2) the component is cheap to render (memo overhead > render cost), (3) the component always receives new object/function props (need useMemo/useCallback first).

Profile first: if a component renders in <1ms, wrapping it in React.memo adds complexity without measurable benefit.

14. Why is useCallback pointless without React.memo on the child? L6

useCallback preserves the referential identity of a function between renders. But this only matters if something is checking for referential equality. Without React.memo, the child re-renders whenever its parent re-renders regardless of prop changes — the stable function reference is never compared. You're paying the cost of useCallback (dependency comparison, memory for cached function) with zero benefit.

The exception: useCallback is useful when the function is a dependency of useEffect or useMemo — stabilizing it prevents unnecessary effect re-runs.

15. What is the React Compiler (Forget) and how does it change memoization? L6

The React Compiler is an ahead-of-time compiler that automatically adds memoization to React components. It analyzes component code at build time, identifies values that can be cached, and inserts memoization code. This eliminates the need for manual useMemo, useCallback, and React.memo.

It works by: (1) analyzing data flow within components, (2) identifying which values depend on which inputs, (3) generating cache slots for values that can be reused, (4) inserting conditional checks to return cached values when inputs haven't changed. The compiler requires components to follow the Rules of React (pure components, rules of hooks). This is why React has always emphasized purity — it enables compiler optimizations.

16. How does memoization affect garbage collection? L6

Memoization holds references to cached values, preventing garbage collection. useMemo keeps the computed value alive as long as the component is mounted. React.memo stores the previous props for comparison. useCallback stores the cached function and its closure.

For large values (processed datasets, complex objects), this means memory is held longer than necessary. Mitigation: (1) avoid memoizing large objects unless the computation is truly expensive, (2) use WeakRef for cache entries that should be GC-eligible, (3) unmount components that hold large memoized values when they're not needed.

17. What is the over-memoization anti-pattern? L6

Over-memoization is wrapping every value and function in useMemo/useCallback "just in case." This adds: (1) memory overhead for cached values and dependency arrays, (2) CPU overhead for dependency comparison on every render, (3) code complexity and cognitive load, (4) false sense of optimization (may not actually help). For trivial computations (string concatenation, simple math), the memoization overhead exceeds the computation cost.

The fix: profile before memoizing. Only memoize when you can measure a difference. Prefer structural optimizations (composition, state colocation) over memoization. The React Compiler will eventually make manual memoization unnecessary.

18. How do you profile whether memoization is actually helping? Senior

(1) Use React Profiler's actualDuration vs baseDuration — if they're similar, memoization isn't helping. (2) A/B test: profile with memoization, remove it, profile again. If the difference is <1ms, it's not worth the complexity. (3) Chrome DevTools Performance panel: record during interaction, compare total scripting time with and without memoization. (4) Check render counts: if a memoized component still renders on every interaction, the memoization is being defeated (likely by unstable props).

19. Why does React.memo not prevent re-renders from context changes? L6

React.memo only compares props. Context is not a prop — it's read internally via useContext. When a context value changes, React marks all consumers for re-render regardless of their memo status. This is by design: context represents a dependency that React.memo can't see from the outside.

Workarounds: (1) split context so consumers only subscribe to the values they need, (2) use a state management library with selectors (Zustand, Jotai), (3) memoize the context value with useMemo in the provider to prevent unnecessary context changes, (4) use the upcoming use(context) with selectors in React 19.

20. How does useMemo's dependency comparison work internally? L6

useMemo stores the previous dependency array and compares each element with the current dependencies using Object.is (same-value equality). If all dependencies are the same (Object.is returns true for each pair), the cached value is returned. If any dependency changed, the factory function re-runs.

Object.is is strict: Object.is(NaN, NaN) is true (unlike ===), Object.is(+0, -0) is false. For objects/arrays, it checks referential equality — {a: 1} !== {a: 1}. This is why you need useMemo for objects passed as dependencies — each render creates a new object reference even if the content is identical.

21. Can React drop memoized values? What are the implications? L6

Yes. React's documentation states that useMemo is a performance optimization, not a semantic guarantee. React may choose to discard cached values (e.g., for offscreen components) and recompute them later. This means: (1) don't use useMemo for side effects — the factory may run more times than expected, (2) don't rely on useMemo to run the factory exactly once, (3) the factory must be a pure function.

In practice, React currently doesn't drop memoized values for mounted components, but future features (like offscreen rendering) may change this. Write code that works correctly even if memoized values are recomputed.

22. Design a memoization strategy for a large React codebase. Architect

(1) Default: no memoization — write clean, well-structured components first. (2) Structural optimizations first — state colocation, composition, component splitting. (3) Profile before memoizing — use React Profiler to identify actual bottlenecks. (4) Memoize strategically — expensive computations, list items, context values. (5) Document why — every useMemo/React.memo should have a comment explaining what performance issue it solves. (6) Review in PRs — challenge unnecessary memoization. (7) Adopt React Compiler when stable — eliminates manual memoization entirely. (8) Lint rules — use eslint-plugin-react-hooks to catch missing dependencies.

Virtualization & Bundle (Questions 23–35)

23. How does virtual scrolling work? What are the key components? L6

Virtual scrolling renders only items visible in the viewport plus a buffer (overscan). Key components: (1) Scroll container — a fixed-height element with overflow: auto. (2) Inner container — sized to the total height of all items (creates correct scrollbar). (3) Visible items — absolutely positioned within the inner container at their calculated offsets. (4) Scroll handler — on scroll, recalculates which items are visible based on scrollTop. (5) Size calculator — for fixed-size lists, offset = index * itemHeight; for variable-size, sum of all previous heights.

24. Compare react-window, react-virtuoso, and TanStack Virtual. Senior

react-window: smallest bundle (~6KB), fixed/variable size lists and grids, requires manual size specification, no auto-measurement. Best for simple use cases with known item sizes. react-virtuoso: largest bundle (~15KB), auto-measures items with ResizeObserver, built-in infinite scroll, SSR support. Best for complex variable-height lists. TanStack Virtual: smallest (~3KB), framework-agnostic (headless), supports auto-measurement, most flexible but requires more setup. Best for custom implementations and non-React projects.

25. What are the challenges of variable-height virtualization? L6

(1) Position calculation requires summing all previous item heights — O(n) without optimization (use prefix sum arrays). (2) Unmeasured items need estimated heights for scroll position calculation; estimates may be wrong, causing scroll jumps. (3) Content changes after measurement (images loading, text expanding) change item heights, requiring re-measurement and position recalculation. (4) Scroll anchoring — when items above the viewport change size, the scroll position must be adjusted to prevent content jumping. (5) Performance — ResizeObserver callbacks for many items can cause layout thrashing.

26. What is tree shaking and what prevents it from working? L6

Tree shaking is dead code elimination based on ES module import/export analysis. Bundlers trace which exports are actually imported and remove unused ones. What prevents it: (1) CommonJS modules (require()) — dynamic, can't be statically analyzed. (2) Side effects — code that runs on import (CSS imports, polyfills, global mutations). (3) Missing sideEffects: false in package.json — bundler assumes all files have side effects. (4) Barrel files — re-exporting from index.ts may pull in unused modules. (5) Dynamic property access — lib[methodName]() prevents the bundler from knowing which exports are used.

27. How do you set up and enforce a bundle size budget? L6

(1) Define budgets: initial JS <200KB gzipped, per-route chunk <50KB, total CSS <50KB. (2) Use size-limit in package.json with budget definitions. (3) Add size-limit check to CI pipeline — fail the build if budget is exceeded. (4) Use webpack's performance.maxAssetSize and maxEntrypointSize for build-time warnings. (5) Review bundle analysis (webpack-bundle-analyzer) on every significant dependency addition. (6) Track bundle size over time in a dashboard to catch gradual growth.

28. Explain the barrel file problem and how to avoid it. L6

Barrel files (index.ts that re-export from multiple modules) can defeat tree shaking. If any re-exported module has side effects, the bundler must include it even if the specific export isn't used. Even without side effects, some bundlers struggle to tree-shake through barrel files, especially with complex re-export patterns.

Avoid: (1) Use direct imports (import { Button } from './components/Button') instead of barrel imports. (2) Ensure all packages have sideEffects: false in package.json. (3) Use bundler plugins that resolve barrel imports to direct imports. (4) For your own libraries, provide a flat export structure or use exports field in package.json for subpath exports.

29. What is the module/nomodule pattern? Senior

The module/nomodule pattern serves different JavaScript bundles to modern vs legacy browsers. <script type="module"> is loaded by modern browsers (ES2015+) and ignored by legacy ones. <script nomodule> is loaded by legacy browsers and ignored by modern ones. The modern bundle is smaller (no polyfills, native async/await, optional chaining) — typically 20-40% smaller. This gives modern browser users a faster experience without breaking legacy support.

30. Design a chunk splitting strategy for a large e-commerce application. Architect

Chunks: (1) react-vendor — React, ReactDOM, React Router (rarely changes, long cache). (2) vendor — other node_modules (changes occasionally). (3) common — shared components/utils used by 2+ routes. (4) Per-route chunks — each page is a separate chunk. (5) Heavy library chunks — chart library, rich text editor loaded on-demand. Caching: content-hash filenames, immutable cache headers for hashed files, short cache for HTML. Prefetching: prefetch likely next routes on hover. Budget: initial load <200KB, per-route <50KB. Monitoring: bundle analysis in CI, size-limit checks, track real-world loading times via RUM.

31. What accessibility challenges does virtualization create? L6

(1) Screen readers can't access off-screen items — they only see the ~20 rendered DOM nodes, not the full list. Mitigation: use ARIA attributes (aria-rowcount, aria-rowindex) to communicate the full list size. (2) Find-in-page (Ctrl+F) doesn't work for off-screen items. No good solution — consider server-side search. (3) Keyboard navigation must be implemented manually — arrow keys should scroll and focus correctly. (4) Print only shows visible items. Mitigation: render full list for print media. (5) Focus management — focused items may be unmounted during scroll, losing focus.

32. When should you NOT use virtualization? Senior

(1) Lists under ~100 items — the overhead of virtualization exceeds the benefit. (2) When accessibility is critical — screen readers and find-in-page work poorly with virtualization. (3) When items have complex internal state — virtualization unmounts items, losing their state. (4) When SEO matters — search engines may not see off-screen content. (5) When print support is needed. Alternatives: pagination (simplest, best for SEO), "show more" buttons, server-side filtering to reduce the dataset, or infinite scroll with pagination (load pages, render all loaded items).

33. How do you implement infinite scroll with virtualization? L6

Combine a virtualizer with a data fetching library that supports pagination. (1) Use TanStack Query's useInfiniteQuery to fetch pages of data. (2) Flatten all pages into a single array. (3) Pass the array to the virtualizer. (4) Monitor the last visible item's index — when it approaches the end of loaded data, trigger fetchNextPage(). (5) Add a loading row at the end while fetching. (6) For memory management, consider evicting old pages when the total exceeds a threshold (windowed data pattern).

34. How do you handle images in virtualized lists? Senior

(1) Use loading="lazy" on images — browser handles lazy loading natively. (2) Set explicit width and height on images to prevent layout shifts when they load. (3) Use placeholder images or blur-up technique while loading. (4) For variable-height lists with images, estimate item height based on image aspect ratio, then re-measure after the image loads. (5) Consider using an image CDN that serves appropriately sized images based on the container width. (6) Preload images for items just outside the viewport (overscan area) to prevent blank flashes during fast scrolling.

35. How does overscan work and how do you choose the right value? Senior

Overscan renders extra items above and below the visible area as a buffer. When the user scrolls, these pre-rendered items become visible before new items need to be rendered, preventing blank flashes. Higher overscan = smoother scrolling but more DOM nodes. Typical values: 3-10 for lists, 1-3 for grids. Choose based on: (1) scroll speed — fast scrolling needs more overscan, (2) item complexity — simple items render fast, need less overscan, (3) device performance — low-end devices benefit from less overscan (fewer DOM nodes), (4) item height — taller items need less overscan (fewer items visible at once).

Web Vitals, Memory & SSR (Questions 36–60)

36. What are the three Core Web Vitals and their thresholds? Senior

(1) LCP (Largest Contentful Paint) — measures loading. Good: ≤2.5s. The time until the largest content element (image, text block) in the viewport is painted. (2) INP (Interaction to Next Paint) — measures interactivity. Good: ≤200ms. The worst interaction latency (click, tap, keypress) throughout the page lifecycle. (3) CLS (Cumulative Layout Shift) — measures visual stability. Good: ≤0.1. The sum of all unexpected layout shifts during the page's lifetime.

37. How does INP differ from FID? Why was FID replaced? L6

FID (First Input Delay) only measured the delay of the first interaction — it missed slow interactions later in the page lifecycle. INP measures all interactions and reports the worst one (98th percentile). FID also only measured input delay (time before handler runs), not processing time or presentation delay. INP measures the full interaction latency: input delay + processing time + presentation delay.

FID was replaced because: (1) a page could have great FID but terrible subsequent interactions, (2) FID didn't capture the full interaction cost, (3) INP better represents the user's actual experience throughout their session.

38. How does React's client-side rendering affect LCP? How do you fix it? L6

CSR delays LCP because the browser must: (1) download the HTML shell, (2) download the JS bundle, (3) execute JavaScript, (4) fetch data, (5) render the component — only then does LCP content appear. This chain can take 3-5+ seconds on slow networks.

Fix: (1) Use SSR/SSG so LCP content is in the initial HTML. (2) Preload the LCP image with <link rel="preload">. (3) Inline critical CSS to avoid render-blocking stylesheets. (4) Use streaming SSR to send the shell immediately. (5) Don't lazy-load the LCP image. (6) Use fetchpriority="high" on the LCP image.

39. What causes CLS in React applications? How do you prevent it? L6

Causes: (1) Images without width/height — reserve space with dimensions or aspect-ratio. (2) Dynamically injected content (ads, banners) — reserve space with min-height. (3) Font swap — use font-display: optional or size-adjust to match fallback metrics. (4) Hydration mismatches — server and client render different content, causing layout changes. (5) Lazy-loaded components without size placeholders. (6) Client-side data fetching that changes layout (loading → content transition).

Prevention: always set image dimensions, use skeleton screens that match final layout dimensions, use content-visibility: auto for off-screen content, and test CLS with the web-vitals library in production.

40. What is the difference between RUM and lab testing for web vitals? Senior

RUM (Real User Monitoring) collects metrics from real users in production — actual devices, networks, and interaction patterns. Google uses RUM data (CrUX) for search ranking. Lab testing (Lighthouse, WebPageTest) runs in a controlled, simulated environment — consistent but not representative of real-world conditions.

Key difference: INP can only be measured via RUM (requires real user interactions). Lab testing uses TBT (Total Blocking Time) as a proxy. A perfect Lighthouse score doesn't guarantee good real-world performance. Use lab testing for development/debugging and RUM for monitoring production performance.

41. What are the five most common memory leak patterns in React? L6

(1) Event listeners — adding listeners to window/document without removing in cleanup. (2) Timers — setInterval/setTimeout not cleared on unmount. (3) Closures — callbacks capturing large objects that outlive the component. (4) Detached DOM — refs holding references to unmounted DOM nodes. (5) Growing state — unbounded arrays/maps that grow without eviction (e.g., chat message history, cache without size limit).

The common fix: always return a cleanup function from useEffect that releases all resources (remove listeners, clear timers, abort fetches, close connections).

42. How do you use Chrome DevTools to find memory leaks? L6

(1) Take Heap Snapshot A (baseline). (2) Perform the suspected leaking action (navigate away and back). (3) Force GC (trash can icon). (4) Take Heap Snapshot B. (5) Compare B vs A — look for objects allocated between snapshots that shouldn't exist. (6) Check the Retainers tab to find the reference chain keeping the object alive. (7) Repeat the action multiple times — if heap grows linearly with each repetition, it's a leak.

Also use the Allocation Timeline: record during interaction, look for persistent blue bars (allocated but never freed). The Allocation Sampling mode has lower overhead and is suitable for production profiling.

43. What is WeakRef and when would you use it in a React application? L6

WeakRef holds a reference to an object without preventing garbage collection. deref() returns the object if it's still alive, or undefined if it's been collected. Use cases in React: (1) caches that should release memory under pressure — cache large computed values with WeakRef, re-compute if GC'd. (2) Observer patterns — track components without preventing their cleanup. (3) FinalizationRegistry — get notified when an object is GC'd to clean up associated resources.

Caution: WeakRef behavior is non-deterministic (GC timing varies). Don't use it for correctness-critical logic — only for optimization (caching, cleanup).

44. How does streaming SSR improve performance over traditional SSR? L6

Traditional SSR waits for ALL data and renders the complete HTML before sending anything — TTFB is slow because it includes all data fetching time. Streaming SSR sends the HTML shell immediately (fast TTFB), then streams additional content as data becomes available. Each Suspense boundary defines a streaming chunk.

Benefits: (1) faster TTFB (shell sent immediately), (2) faster FCP (shell has content), (3) progressive loading (users see content as it arrives), (4) selective hydration (each chunk hydrates independently), (5) graceful degradation (if one data source is slow, the rest of the page is already visible).

45. What is selective hydration and how does it improve TTI? L6

Selective hydration (React 18) hydrates Suspense boundaries independently rather than the entire page at once. Benefits: (1) each boundary becomes interactive as soon as its JS loads and hydrates, (2) user interactions automatically prioritize hydrating the boundary the user is interacting with, (3) slow-loading boundaries don't block other boundaries from hydrating.

If a user clicks a button in an unhydrated boundary, React: (1) records the event, (2) prioritizes hydrating that boundary, (3) replays the event after hydration. The user doesn't need to click again — the interaction is preserved.

46. How do React Server Components reduce bundle size? L6

Server Components render on the server and send their output (RSC payload) to the client — their source code is never included in the client bundle. This means: (1) component code (render logic, data transformation) stays on the server, (2) server-only dependencies (database clients, file system, heavy libraries) are never bundled, (3) only Client Components (marked with 'use client') are included in the client bundle.

For content-heavy pages (blog, documentation, product pages), this can reduce client JS by 70-90%. The client only downloads JavaScript for interactive elements (buttons, forms, animations).

47. Compare SSG, SSR, ISR, and streaming SSR. When do you use each? Architect

SSG: build-time HTML, fastest TTFB (CDN), stale until rebuild. Use for: docs, blog, marketing pages. SSR: per-request HTML, slower TTFB, always fresh. Use for: personalized content, real-time data. ISR: build-time + background revalidation, fast TTFB (cached), configurable freshness. Use for: e-commerce products, news articles. Streaming SSR: per-request but progressive, fast TTFB (shell immediate), always fresh. Use for: complex dashboards, pages with mixed-speed data sources.

Decision: static data → SSG. Slowly changing → ISR. User-specific → SSR. Complex with mixed data speeds → streaming SSR. Most apps use a mix per-route.

48. What are the tradeoffs of edge rendering vs origin server rendering? L6

Edge: lower latency (closer to users), faster cold starts (~50ms vs ~500ms), limited runtime (V8 isolates, no full Node.js), remote database access adds latency. Origin: full Node.js runtime, direct database access (low latency), single region (higher latency for distant users), slower cold starts.

Use edge for: auth checks, A/B testing, geo-routing, personalization, static content with dynamic headers. Use origin for: data-heavy pages requiring database queries, complex server-side logic, pages using Node.js-specific APIs. Hybrid approach: edge middleware for routing/auth + origin for data-heavy rendering.

49. How do you optimize TTFB for server-rendered React applications? L6

(1) Streaming SSR — send shell HTML immediately, don't wait for all data. (2) CDN caching — cache SSR output with s-maxage and stale-while-revalidate. (3) Parallel data fetching — use Promise.all for independent data sources. (4) Database optimization — connection pooling, query optimization, read replicas. (5) Edge rendering for latency-sensitive routes. (6) Request deduplication — use React's cache() to deduplicate identical data fetches within a request. (7) Precompute — use SSG/ISR for pages that don't need per-request freshness.

50. How does Next.js caching work and what are the common pitfalls? L6

Next.js has four caching layers: (1) Request Memoization — deduplicates identical fetch calls within a single render pass. (2) Data Cache — persists fetch results across requests (opt out with cache: 'no-store'). (3) Full Route Cache — caches rendered HTML and RSC payload for static routes. (4) Router Cache — client-side cache of visited routes (30s dynamic, 5min static).

Pitfalls: (1) Data cache persisting stale data — need explicit revalidatePath/revalidateTag after mutations. (2) Router cache showing old data on back navigation. (3) Aggressive defaults caching dynamic data unexpectedly. (4) Cache invalidation not propagating across all layers. Understanding and controlling these layers is essential for correct behavior.

51. How do you optimize font loading for web performance? Senior

(1) Use font-display: swap — show fallback font immediately, swap when custom font loads (prevents invisible text). (2) Preload critical fonts — <link rel="preload" href="font.woff2" as="font" crossorigin>. (3) Subset fonts — only include characters you need (Latin subset: ~20KB vs full: ~300KB). (4) Use WOFF2 format (best compression). (5) Self-host fonts (avoid third-party requests). (6) Use size-adjust, ascent-override, descent-override to match fallback font metrics (reduces CLS from font swap). (7) Next.js next/font handles all of this automatically.

52. How do you use fetchpriority to improve LCP? Senior

fetchpriority is an HTML attribute that hints to the browser about the relative priority of a resource fetch. Set fetchpriority="high" on the LCP image to tell the browser to download it before other images. Set fetchpriority="low" on below-fold images to deprioritize them. This is especially impactful when the LCP image competes with other resources (scripts, stylesheets, other images) for bandwidth.

Combine with <link rel="preload"> for maximum impact — preload tells the browser to discover the resource early, fetchpriority tells it to prioritize the download.

53. How does V8's generational garbage collector work? L6

V8 divides the heap into two generations based on the observation that most objects die young. Young Generation (Scavenger): small space (~1-8MB), uses semi-space copying — two equal-sized spaces, live objects are copied from one to the other, dead objects are simply left behind. Fast and frequent. Objects surviving 2 scavenges are promoted to Old Generation. Old Generation (Mark-Sweep-Compact): larger space, uses mark-and-sweep — traverse from GC roots, mark reachable objects, sweep (free) unmarked objects. Optionally compacts to reduce fragmentation. Slower but less frequent.

GC roots include: global object, stack variables, active closures, event listeners, timers, and DOM references. Any object reachable from a root through any chain of references is considered alive.

54. How do you prevent memory leaks from closures in useEffect? L6

(1) Always return a cleanup function that releases resources (remove listeners, clear timers, abort fetches). (2) Use AbortController for fetch requests — abort in cleanup. (3) Avoid capturing large objects in closures — extract only the values the closure needs. (4) Use refs for values that change but shouldn't trigger re-renders — the closure captures the ref (stable), not the value. (5) Be careful with async operations in effects — the component may unmount before the async operation completes. Use a boolean flag or AbortController to prevent state updates on unmounted components.

55. What is progressive enhancement and how does it relate to SSR? Senior

Progressive enhancement means building a baseline experience that works without JavaScript, then enhancing it with JavaScript. SSR enables this: the server renders complete HTML that is functional without JS (links navigate, forms submit). When JS loads, it hydrates the HTML and adds interactivity (client-side navigation, optimistic updates, animations).

Next.js Server Actions exemplify this: forms work as traditional POST submissions without JS, and are enhanced with fetch-based submissions when JS is available. This provides: better accessibility, faster initial interaction, resilience to JS failures, and better SEO.

56. How do you set up HTTP caching headers for different types of SSR responses? L6

Static assets (JS, CSS, images with content hash): Cache-Control: public, max-age=31536000, immutable — cache forever, content hash changes on update. SSG pages: Cache-Control: public, s-maxage=31536000, stale-while-revalidate=31536000 — CDN caches indefinitely, revalidates on deploy. ISR pages: Cache-Control: public, s-maxage=60, stale-while-revalidate=300 — CDN caches 60s, serves stale up to 300s while revalidating. SSR personalized pages: Cache-Control: private, no-cache — no CDN caching, browser may cache with revalidation. API responses: varies by endpoint — use Vary header for content negotiation.

57. How does React's cache() function work for request deduplication? L6

React's cache() memoizes an async function for the duration of a single server render request. If multiple Server Components call cache(getUser)(userId) with the same arguments, only one database query executes — subsequent calls return the cached result. The cache is automatically cleared after the request completes.

This is different from useMemo (component-level, client-side) and Next.js fetch cache (cross-request, persistent). cache() is specifically for deduplicating work within a single render pass on the server, preventing N+1 query patterns when multiple components need the same data.

58. How do you handle hydration mismatches and why do they cause CLS? L6

Hydration mismatches occur when server-rendered HTML differs from what the client renders during hydration. React must patch the DOM to match the client output, which can cause visible layout changes (CLS). Common causes: (1) using Date.now() or Math.random() in render (different on server vs client), (2) checking window/localStorage during render, (3) browser extensions modifying the DOM before hydration.

Fix: (1) Use useEffect for client-only values (runs after hydration). (2) Use suppressHydrationWarning for intentional mismatches (timestamps). (3) Use useId() for stable IDs across server/client. (4) Avoid conditional rendering based on client-only APIs during initial render.

59. How do Web Workers help with INP optimization in React? L6

Web Workers run JavaScript in a separate thread, freeing the main thread for user interactions. For React: (1) move expensive computations (sorting, filtering, data transformation) to a worker — the main thread stays responsive during the computation. (2) Use Comlink library for ergonomic worker communication. (3) Post data to the worker, receive results via message, update state with the result.

Limitations: workers can't access the DOM, so they can't help with expensive renders (that's React's job). They help with the "processing time" portion of INP — the time spent in event handlers doing computation before React renders.
