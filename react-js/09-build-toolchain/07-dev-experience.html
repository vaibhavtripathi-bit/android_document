Architect Insight: exhaustive-deps

The react-hooks/exhaustive-deps rule is the most debated ESLint rule in React. It catches stale closure bugs by ensuring all values used inside effects are in the dependency array. At Staff level, you should understand why disabling it is almost always wrong: the "fix" of adding // eslint-disable-next-line hides real bugs. Instead, restructure the code — move functions inside the effect, use refs for values that shouldn't trigger re-runs, or use useReducer for complex state updates.

lint-staged Performance

lint-staged only runs linters on staged files (files in git add), not the entire codebase. This keeps pre-commit hooks fast even in large monorepos. For TypeScript type checking (which needs the full project), run tsc --noEmit in CI instead of pre-commit — it's too slow for a commit hook.

React DevTools Architecture:

  React App (page)              DevTools Extension
  ┌─────────────────┐          ┌─────────────────────┐
  │ React Runtime   │          │ DevTools Frontend    │
  │                 │          │ (React app itself)   │
  │ ┌─────────────┐│          │                      │
  │ │ __REACT_    ││  bridge  │ ┌──────────────────┐ │
  │ │ DEVTOOLS_   ││◄────────▶│ │ Component Tree   │ │
  │ │ GLOBAL_HOOK ││          │ │ Profiler          │ │
  │ └─────────────┘│          │ │ Settings          │ │
  │                 │          │ └──────────────────┘ │
  │ Fiber Tree     │          │                      │
  │ ┌─────────────┐│          │ Renders fiber tree   │
  │ │ Renderers   ││          │ as inspectable UI    │
  │ │ (ReactDOM)  ││          │                      │
  │ └─────────────┘│          └─────────────────────┘
  └─────────────────┘

  Communication: window.__REACT_DEVTOOLS_GLOBAL_HOOK__
  Bridge: postMessage (content script ↔ background ↔ devtools panel)

    // Called by ReactDOM when it initializes

  },
  onCommitFiberRoot(rendererID, root, priorityLevel) {
    // Called after every React commit
    // DevTools reads the fiber tree here
  },
  onCommitFiberUnmount(rendererID, fiber) {
    // Called when a fiber is unmounted
  },

// 2. ReactDOM detects the hook and registers itself
// 3. On every commit, React calls onCommitFiberRoot
// 4. DevTools walks the fiber tree and builds its own representation
// 5. Communication happens via a "bridge" (message passing)

5. React StrictMode

// StrictMode enables additional development checks
import { StrictMode } from 'react';

createRoot(document.getElementById('root')!).render(
  <StrictMode>
    <App />
  </StrictMode>
);

// StrictMode behaviors (development only, no production impact):

// 1. Double-invokes render functions
//    - Components render twice to detect impure renders
//    - Second render result is used, first is discarded

// 2. Double-invokes effects
//    - useEffect runs: setup → cleanup → setup
//    - Catches effects that don't clean up properly
//    - Simulates component unmount/remount (important for React 19)

// 3. Detects legacy APIs
//    - Warns about deprecated lifecycle methods
//    - Warns about legacy string refs
//    - Warns about findDOMNode usage
//    - Warns about legacy context API

// 4. Detects unexpected side effects
//    - Double-invokes constructors, render, shouldComponentUpdate
//    - Double-invokes state updater functions
//    - Double-invokes useMemo/useReducer initializers

StrictMode Double-Invoke Confusion

The most common confusion: "my effect runs twice!" This is intentional in StrictMode. It simulates a component being unmounted and remounted, which happens in React 19's Offscreen API and during Fast Refresh. If your effect breaks when run twice, it has a cleanup bug. The fix is always to add proper cleanup, not to remove StrictMode.

6. Error Overlay (react-error-overlay)

// Error overlay shows runtime errors in development
// Built into Create React App, Next.js, Vite plugin

// How it works:
// 1. Catches unhandled errors via window.onerror and
//    unhandledrejection event
// 2. Parses error stack traces
// 3. Maps to source code via source maps
// 4. Displays overlay with:
//    - Error message and type
//    - Source code with highlighted error line
//    - Component stack trace (React-specific)
//    - Click-to-open in editor

// React component stack traces (development only)
// React adds component names to error messages:
// "Error in Button (at App.tsx:15)
//   in Card (at App.tsx:10)
//   in App (at index.tsx:5)"

// Error Boundary integration
class ErrorBoundary extends React.Component {
  state = { hasError: false, error: null };

  static getDerivedStateFromError(error) {
    return { hasError: true, error };
  }

  componentDidCatch(error, errorInfo) {
    // errorInfo.componentStack contains the React component stack
    console.error('Component stack:', errorInfo.componentStack);
    // Send to error tracking service
    reportError(error, errorInfo);
  }

  render() {
    if (this.state.hasError) {
      return <ErrorFallback error={this.state.error} />;
    }
    return this.props.children;
  }
}

7. Fast Refresh Internals

Fast Refresh Pipeline:

  File Save → Bundler HMR → React Fast Refresh Runtime

  1. File changed (App.tsx)
  2. Bundler recompiles single module
  3. HMR sends update to client
  4. Fast Refresh runtime:
     a. Detect which exports are React components
     b. For each component:
        ┌─────────────────────────────┐
        │ Compute "signature" from:   │
        │ - Hook call order           │
        │ - Hook types (useState, etc)│
        │ - Custom hook names         │
        └─────────────────────────────┘
     c. If signature unchanged:
        → Re-render component, PRESERVE state
     d. If signature changed (hooks added/removed):
        → Remount component, RESET state
     e. If non-component export changed:
        → Full module reload (bubble up to parent)

// Fast Refresh registration (injected by bundler plugin)
// After each component definition, the plugin adds:

var _s = $RefreshSig$();

function App() {
  _s();  // Register hook signature
  const [count, setCount] = useState(0);
  const theme = useContext(ThemeContext);
  // ...
}

_s(App, "useState{[count, setCount]}\\nuseContext{theme}");
// Signature encodes: hook types and variable names

$RefreshReg$(App, "App");
// Register component with Fast Refresh runtime

// When file is updated:
// 1. New module executes, re-registering components
// 2. Runtime compares old and new signatures
// 3. If match: schedule re-render with existing fiber
// 4. If mismatch: schedule remount (new fiber)

// What triggers a FULL RELOAD (no state preservation):
// - Editing a file that exports non-component values
// - Syntax errors
// - Class components (limited support)
// - Files that run side effects at module level

Fast Refresh Limitations

Fast Refresh works best with: (1) files that export only React components, (2) function components (class components get remounted), (3) named exports (default exports work but are harder to track). If a file exports both components and utilities, editing the utility forces a full reload. This is why separating components and utilities into different files improves the development experience.

8. Environment Variables

// .env files (loaded by Vite, Next.js, CRA)
// .env                — always loaded
// .env.local          — always loaded, gitignored
// .env.development    — loaded in dev mode
// .env.production     — loaded in production build
// .env.test           — loaded in test mode

// Priority (highest to lowest):
// 1. Shell environment variables
// 2. .env.{mode}.local
// 3. .env.{mode}
// 4. .env.local
// 5. .env

// Vite: only VITE_ prefixed vars are exposed to client
VITE_API_URL=https://api.example.com
VITE_APP_TITLE=My App
SECRET_KEY=abc123  // NOT exposed to client (no VITE_ prefix)

// Access in code:
const apiUrl = import.meta.env.VITE_API_URL;
const mode = import.meta.env.MODE;       // 'development' | 'production'
const isDev = import.meta.env.DEV;       // boolean
const isProd = import.meta.env.PROD;     // boolean

// Next.js: NEXT_PUBLIC_ prefix for client exposure
NEXT_PUBLIC_API_URL=https://api.example.com
DATABASE_URL=postgres://...  // Server-only (not in client bundle)

// TypeScript env type declarations
// env.d.ts
/// <reference types="vite/client" />
interface ImportMetaEnv {
  readonly VITE_API_URL: string;
  readonly VITE_APP_TITLE: string;
}

interface ImportMeta {
  readonly env: ImportMetaEnv;
}

Client-Side Environment Variable Security

Any environment variable exposed to the client (VITE_, NEXT_PUBLIC_, REACT_APP_) is embedded in the JavaScript bundle at build time. It is visible to anyone who inspects the bundle. NEVER put secrets (API keys with write access, database URLs, auth secrets) in client-exposed env vars. Use server-side API routes or a BFF (Backend for Frontend) to proxy requests that need secrets.
