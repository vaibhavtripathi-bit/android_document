Transpilation

Babel's three-phase architecture, SWC as Rust-based alternative, TypeScript compilation paths, JSX transformation, polyfills, and the build-time vs runtime transpilation spectrum.

1. What Is Transpilation?

Transpilation converts source code from one language (or language version) to another at the same abstraction level. Unlike compilation (high-level → low-level), transpilation transforms modern JavaScript/TypeScript/JSX into browser-compatible JavaScript.

Transpilation Pipeline:

  Source Code          AST              Transformed AST       Output Code
  ┌──────────┐      ┌──────────┐      ┌──────────────┐      ┌──────────┐
  │ JSX +    │─parse▶│ Abstract │─xform▶│ Transformed  │─gen─▶│ ES5/ES6  │
  │ TS +     │      │ Syntax   │      │ AST          │      │ Browser  │
  │ ES2024   │      │ Tree     │      │              │      │ Compat   │
  └──────────┘      └──────────┘      └──────────────┘      └──────────┘

  Tools:
  ├── Babel    (JS, plugin-based, mature)
  ├── SWC      (Rust, fast, Babel-compatible)
  ├── esbuild  (Go, very fast, limited transforms)
  └── tsc      (TS official, type-checking + emit)

2. Babel Architecture

2.1 Three Phases

Babel operates in three distinct phases: parsing, transformation, and code generation.

// Phase 1: PARSING (@babel/parser, formerly Babylon)
// Converts source string → AST
import { parse } from '@babel/parser';

const ast = parse('const x = <div>Hello</div>;', {
  sourceType: 'module',
  plugins: ['jsx', 'typescript', 'decorators-legacy'],
});

// Phase 2: TRANSFORMATION (@babel/traverse + plugins)
// Walks the AST and applies visitor-pattern transforms
import traverse from '@babel/traverse';

traverse(ast, {
  JSXElement(path) {
    // Transform JSX to React.createElement calls
  },
  ArrowFunctionExpression(path) {
    // Convert arrow functions to regular functions
  },
});

// Phase 3: CODE GENERATION (@babel/generator)
// Converts transformed AST → output string + source map
import generate from '@babel/generator';

const output = generate(ast, {
  sourceMaps: true,
  retainLines: false,
});
// output.code — transformed source
// output.map  — source map

2.3 Babel Plugins

// A Babel plugin is a function returning a visitor object
module.exports = function myPlugin({ types: t }) {
  return {
    visitor: {
      // Visit every Identifier node
      Identifier(path) {
        if (path.node.name === '__DEV__') {
          path.replaceWith(
            t.booleanLiteral(process.env.NODE_ENV !== 'production')
          );
        }
      },

      // Visit CallExpression nodes
      CallExpression(path) {
        // Remove console.log in production
        if (
          t.isMemberExpression(path.node.callee) &&
          t.isIdentifier(path.node.callee.object, { name: 'console' })
        ) {
          path.remove();
        }
      },

      // Enter/exit hooks for scoping
      FunctionDeclaration: {
        enter(path) { /* entering function scope */ },
        exit(path)  { /* leaving function scope */ },
      },
    },
  };
};

// Plugin ordering: plugins run BEFORE presets
// Plugins run first-to-last, presets run last-to-first
// babel.config.js
module.exports = {
  plugins: [
    'plugin-A',  // runs 1st
    'plugin-B',  // runs 2nd
  ],
  presets: [
    'preset-X',  // runs 2nd (presets are reversed)
    'preset-Y',  // runs 1st
  ],
};

2.4 Babel Presets

// @babel/preset-env — the most important preset
// Compiles modern JS to target environments using browserslist
module.exports = {
  presets: [
    ['@babel/preset-env', {
      targets: {
        browsers: ['> 0.5%', 'not dead', 'not ie 11'],
        // or use browserslist config in package.json
      },
      modules: false,           // Preserve ESM for tree shaking
      useBuiltIns: 'usage',    // Auto-import polyfills per usage
      corejs: 3,               // core-js version for polyfills
      debug: true,             // Log which transforms are applied
      bugfixes: true,          // Use smaller transforms
    }],
    ['@babel/preset-react', {
      runtime: 'automatic',    // New JSX transform (React 17+)
      development: process.env.NODE_ENV === 'development',
    }],
    ['@babel/preset-typescript', {
      isTSX: true,
      allExtensions: true,
      allowDeclareFields: true,
    }],
  ],
};

Architect Insight: preset-env Internals

@babel/preset-env uses the compat-table data (from kangax) to determine which transforms are needed for your target browsers. With useBuiltIns: 'usage', it analyzes your code's AST to find which built-in features you use (e.g., Array.prototype.includes, Promise.allSettled) and injects only those polyfills. This can reduce polyfill bundle size by 50-80% compared to importing all of core-js.

3. JSX Transformation

3.1 Classic Transform (Pre-React 17)

// Input JSX
const element = <div className="app">
  <h1>Hello</h1>
  <Button onClick={handler} />
</div>;

// Classic output (requires React in scope)
const element = React.createElement('div', { className: 'app' },
  React.createElement('h1', null, 'Hello'),
  React.createElement(Button, { onClick: handler })
);

3.2 Automatic Transform (React 17+)

// Input JSX (same as above)
const element = <div className="app">
  <h1>Hello</h1>
  <Button onClick={handler} />
</div>;

// Automatic output (no React import needed)
import { jsx as _jsx, jsxs as _jsxs } from 'react/jsx-runtime';

const element = _jsxs('div', {
  className: 'app',
  children: [
    _jsx('h1', { children: 'Hello' }),
    _jsx(Button, { onClick: handler }),
  ],
});

// Development mode uses jsxDEV with extra debugging info
import { jsxDEV as _jsxDEV } from 'react/jsx-dev-runtime';

const element = _jsxDEV('div', {
  className: 'app',
  children: [...]
}, undefined, true, {  // Extra dev info:
  fileName: '/src/App.jsx',
  lineNumber: 5,
  columnNumber: 8,
}, this);

Why the New Transform?

The automatic runtime: (1) eliminates the need to import React in every JSX file, (2) produces slightly smaller output (jsx vs React.createElement), (3) enables future optimizations in the JSX runtime without changing user code, (4) provides better debugging info in development mode.

4. SWC (Speedy Web Compiler)

SWC is a Rust-based transpiler that's 20-70x faster than Babel while maintaining compatibility with most Babel transforms.

// .swcrc configuration
{
  "$schema": "https://json.schemastore.org/swcrc",
  "jsc": {
    "parser": {
      "syntax": "typescript",
      "tsx": true,
      "decorators": true,
      "dynamicImport": true
    },
    "transform": {
      "react": {
        "runtime": "automatic",
        "development": false,
        "refresh": true          // React Fast Refresh support
      },
      "optimizer": {
        "globals": {
          "vars": {
            "__DEV__": "false"
          }
        }
      }
    },
    "target": "es2020",
    "minify": {
      "compress": { "dead_code": true, "unused": true },
      "mangle": true
    }
  },
  "module": {
    "type": "es6"
  },
  "minify": true
}

5. TypeScript Compilation Paths

Three TypeScript Compilation Strategies:

Strategy 1: tsc only
  .ts/.tsx ──tsc──▶ .js + .d.ts + type errors
  (Slow, full type checking, official)

Strategy 2: Babel/SWC + tsc type-check
  .ts/.tsx ──SWC──▶ .js  (fast, strips types only)
  .ts/.tsx ──tsc──▶ type errors  (parallel, --noEmit)

Strategy 3: Bundler integration
  .ts/.tsx ──Vite(esbuild)──▶ .js  (dev, fastest)
  .ts/.tsx ──Vite(Rollup+SWC)──▶ .js  (prod)
  .ts/.tsx ──tsc──▶ type errors  (CI, --noEmit)

// tsconfig.json for Strategy 2 (most common in React projects)
{
  "compilerOptions": {
    "target": "ES2020",
    "lib": ["DOM", "DOM.Iterable", "ES2020"],
    "module": "ESNext",
    "moduleResolution": "bundler",   // New in TS 5.0
    "jsx": "react-jsx",              // Automatic JSX transform
    "strict": true,
    "noEmit": true,                  // Don't emit — SWC/Babel does that
    "isolatedModules": true,         // Required for SWC/Babel compat
    "esModuleInterop": true,
    "skipLibCheck": true,
    "resolveJsonModule": true,
    "declaration": false,
    "paths": {
      "@/*": ["./src/*"]
    }
  },
  "include": ["src"],
  "exclude": ["node_modules", "dist"]
}

isolatedModules Is Critical

When using SWC/Babel/esbuild for TypeScript, you MUST enable isolatedModules: true. These tools process files individually (no cross-file type information), so features like const enum and export = that require cross-file analysis will silently produce incorrect output without this flag. With it enabled, TypeScript errors on unsupported patterns.

6. Polyfills

6.1 Syntax vs API Polyfills

// SYNTAX transforms — handled by transpiler (Babel/SWC)
// These change code structure:
const fn = () => {};           // → function fn() {}
const { a, ...rest } = obj;   // → var a = obj.a; var rest = _objectWithoutProperties(...)
class Foo extends Bar {}       // → function Foo() { ... } Foo.prototype = ...
const x = a ?? b;             // → var x = a !== null && a !== void 0 ? a : b;

// API polyfills — need runtime library (core-js)
// These add missing built-in methods/objects:
Array.prototype.includes       // Not in IE11
Promise.allSettled             // Not in older browsers
Object.fromEntries             // ES2019
globalThis                     // ES2020
structuredClone                // Recent addition

6.2 core-js Polyfill Strategies

// Strategy 1: useBuiltIns: 'entry' — import all polyfills for targets
// In your entry file:
import 'core-js/stable';
import 'regenerator-runtime/runtime';
// Babel replaces with only polyfills needed for your targets
// Result: ~80KB for modern targets, ~200KB for IE11

// Strategy 2: useBuiltIns: 'usage' — auto-inject per file
// No manual imports needed — Babel analyzes usage
// file.js
const result = [1, 2, 3].includes(2);
// Babel adds: import "core-js/modules/es.array.includes.js";
// Result: smallest bundle, only what you use

// Strategy 3: @babel/plugin-transform-runtime
// Avoids global pollution — uses helper imports instead
// Good for libraries (don't pollute consumer's globals)
['@babel/plugin-transform-runtime', {
  corejs: 3,        // Polyfill version
  helpers: true,     // Reuse helper functions
  regenerator: true, // Async/await support
}]
// import _includesInstanceProperty from "@babel/runtime-corejs3/core-js-stable/instance/includes";
// _includesInstanceProperty(arr).call(arr, 2);

require('./src/server.ts');

// Jest uses babel-jest or ts-jest for test-time transpilation
// jest.config.js
module.exports = {
  transform: {
    '^.+\\.tsx?$': ['@swc/jest', {
      jsc: { parser: { syntax: 'typescript', tsx: true } },
    }],
  },
};

Architect Insight: Transpilation Strategy for Large Teams

For a large React codebase, the optimal strategy is: (1) SWC for fast transpilation in dev and production builds, (2) tsc with --noEmit for type checking in CI (parallelized), (3) @babel/preset-env with useBuiltIns: 'usage' only if you need fine-grained polyfill control that SWC doesn't provide, (4) browserslist targeting modern browsers with differential serving for legacy support. This gives you sub-second rebuilds in development while maintaining type safety and broad browser support.

8. Differential Serving

<!-- Serve modern code to modern browsers, legacy to old -->
<!-- Modern browsers load type="module", ignore nomodule -->
<script type="module" src="/app.modern.js"></script>

<!-- Legacy browsers ignore type="module", load nomodule -->
<script nomodule src="/app.legacy.js"></script>

<!-- Modern bundle: ~40% smaller (no polyfills, modern syntax) -->
<!-- Legacy bundle: full polyfills, ES5 syntax -->
