npm/yarn (flat hoisting):              pnpm (strict, symlinked):

├── my-dep/                              │  │  └── node_modules/
│   └── (uses lodash internally)         │  │     └── react/

                                         │     └── node_modules/
You can import 'lodash' even if          │        └── lodash/
it's not in YOUR package.json!           ├── react → .pnpm/react@18.3.1/...
(phantom dependency)                     └── (only YOUR deps are linked here)

                                         Cannot import lodash unless it's
                                         in YOUR package.json (strict)

// Phantom dependency example:
// package.json has: { "dependencies": { "my-library": "^1.0.0" } }
// my-library depends on lodash internally

// With npm/yarn (hoisted):
import _ from 'lodash'; // Works! (but shouldn't — phantom dependency)
// If my-library drops lodash, YOUR code breaks

// With pnpm (strict):
import _ from 'lodash'; // ERROR: Module not found
// You must explicitly add lodash to your package.json

2.3 pnpm Workspaces

# pnpm-workspace.yaml
packages:
  - 'apps/*'
  - 'packages/*'
  - 'tools/*'

# Install dependencies
pnpm install

# Run command in specific package
pnpm --filter @myorg/web dev

# Run command in all packages
pnpm -r build

# Run only in packages that changed since main
pnpm --filter "...[origin/main]" build

# Add dependency to specific package
pnpm --filter @myorg/web add react-query

# Add shared dev dependency to root
pnpm add -Dw typescript eslint prettier
// Internal package reference (packages/ui/package.json)
{
  "name": "@myorg/ui",
  "version": "0.0.0",
  "main": "./src/index.ts",       // Point to source for dev
  "types": "./src/index.ts",
  "exports": {
    ".": "./src/index.ts",
    "./button": "./src/Button.tsx"
  }
}

// Consumer (apps/web/package.json)
{
  "name": "@myorg/web",
  "dependencies": {
    "@myorg/ui": "workspace:*",    // pnpm workspace protocol
    "@myorg/utils": "workspace:^"  // Resolves to workspace version
  }
}

3. Turborepo

Turborepo is a high-performance build system for JavaScript/TypeScript monorepos, focused on task orchestration and caching.

// turbo.json
{
  "$schema": "https://turbo.build/schema.json",
  "globalDependencies": ["**/.env.*local"],
  "tasks": {
    "build": {
      "dependsOn": ["^build"],     // Run deps' build first (topological)
      "outputs": ["dist/**", ".next/**", "!.next/cache/**"],
      "env": ["NODE_ENV", "API_URL"]
    },
    "test": {
      "dependsOn": ["build"],
      "inputs": ["src/**", "test/**"],
      "outputs": [],                // No cacheable outputs
      "env": ["CI"]
    },
    "lint": {
      "dependsOn": [],              // No dependencies — can run in parallel
      "outputs": []
    },
    "dev": {
      "cache": false,               // Don't cache dev server
      "persistent": true            // Long-running process
    },
    "typecheck": {
      "dependsOn": ["^build"],
      "outputs": []
    }
  }
}
Turborepo Task Graph:

  turbo run build

  ┌─────────────┐
  │ @myorg/utils │ ── build ──┐
  └─────────────┘             │
  ┌─────────────┐             ▼
  │ @myorg/ui    │ ── build ──┬──▶ ┌─────────────┐
  └─────────────┘             │    │ @myorg/web  │ ── build
  ┌─────────────┐             │    └─────────────┘
  │ @myorg/config│ ── build ──┘
  └─────────────┘                  ┌─────────────┐
                                   │ @myorg/api  │ ── build
                                   └─────────────┘

  ^build = run build in dependencies first (topological order)
  Parallel execution where no dependencies exist

// Local caching: Turborepo hashes inputs and caches outputs

// $ turbo run build

// @myorg/ui:build: cache hit, replaying output
// @myorg/web:build: cache miss, executing

4. Nx

Nx is a more opinionated build system with computation caching, affected commands, code generators, and deep framework integration.

// nx.json
{
  "targetDefaults": {
    "build": {
      "dependsOn": ["^build"],
      "inputs": ["production", "^production"],
      "outputs": ["{projectRoot}/dist"],
      "cache": true
    },
    "test": {
      "inputs": ["default", "^production", "{workspaceRoot}/jest.preset.js"],
      "cache": true
    },
    "lint": {
      "inputs": ["default", "{workspaceRoot}/.eslintrc.json"],
      "cache": true
    }
  },
  "namedInputs": {
    "default": ["{projectRoot}/**/*", "sharedGlobals"],
    "production": [
      "default",
      "!{projectRoot}/**/?(*.)+(spec|test).[jt]s?(x)",
      "!{projectRoot}/tsconfig.spec.json",
      "!{projectRoot}/jest.config.[jt]s"
    ],
    "sharedGlobals": ["{workspaceRoot}/tsconfig.base.json"]
  }
}

// Nx affected commands — only run on changed packages
// $ nx affected:build --base=main
// $ nx affected:test --base=main
// $ nx affected:lint --base=main

// Nx graph visualization
// $ nx graph  (opens interactive dependency graph in browser)

// Nx generators (scaffolding)
// $ nx generate @nx/react:component Button --project=ui
// $ nx generate @nx/react:library shared-utils

5. Lerna

// lerna.json (now powered by Nx under the hood)
{
  "$schema": "node_modules/lerna/schemas/lerna-schema.json",
  "version": "independent",  // or "fixed" for unified versioning
  "npmClient": "pnpm",
  "useWorkspaces": true,
  "command": {
    "publish": {
      "conventionalCommits": true,
      "message": "chore(release): publish %s"
    },
    "version": {
      "allowBranch": "main",
      "conventionalCommits": true
    }
  }
}

// Lerna commands
// $ lerna version          — bump versions based on conventional commits
// $ lerna publish          — publish changed packages to npm
// $ lerna run build        — run build in all packages (uses Nx caching)
// $ lerna changed          — list packages changed since last release

7. Versioning Strategies

// Fixed versioning: all packages share the same version
// Good for: tightly coupled packages (React, Babel)
// lerna.json: { "version": "2.5.0" }
// Every publish bumps ALL packages to the same version

// Independent versioning: each package has its own version
// Good for: loosely coupled packages, internal tools
// lerna.json: { "version": "independent" }
// Each package version bumps independently based on its changes

// Changesets — popular alternative to Lerna versioning
// $ npx changeset        — create a changeset
// $ npx changeset version — apply changesets, bump versions
// $ npx changeset publish — publish to npm

// .changeset/cool-feature.md (created by `npx changeset`)
---
"@myorg/ui": minor
"@myorg/utils": patch
---

Added new Button variant and fixed utility helper.

Architect Insight: Monorepo at Scale

For a large React monorepo (50+ packages, 20+ engineers): (1) Use pnpm for strict dependency management and disk efficiency, (2) Use Turborepo for simple task orchestration with remote caching, or Nx if you need code generators and deep framework integration, (3) Use changesets for versioning if you publish packages, (4) Set up CODEOWNERS for package-level ownership, (5) Use internal packages (not published) for shared code that doesn't need versioning. The key architectural decision is whether packages are "publishable" (versioned, published to registry) or "internal" (source-level imports, no versioning).

8. Build Caching and Remote Caching

// How build caching works:
// 1. Hash inputs: source files + deps + env vars + config
// 2. Check cache for matching hash
// 3. Cache hit → restore outputs from cache (skip build)
// 4. Cache miss → run build, store outputs in cache

// Turborepo cache structure:
// node_modules/.cache/turbo/
//   ├── {hash1}.tar.zst   — compressed build outputs
//   ├── {hash2}.tar.zst
//   └── ...

// Remote cache enables sharing across CI and developers:
// Developer A builds @myorg/ui → uploads to remote cache
// Developer B pulls @myorg/ui → cache hit, no build needed
// CI builds @myorg/ui → cache hit from Developer A's build

// - Source file changes
