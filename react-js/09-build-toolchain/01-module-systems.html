Module Systems

JavaScript module evolution from global scripts to ES Modules — resolution algorithms, dual packages, import maps, Module Federation, and tree shaking requirements.

1. The Problem: Global Scope Pollution

Before modules, every JavaScript file shared a single global scope. Libraries like jQuery, Backbone, and Angular 1.x attached themselves to window, creating collision risks and implicit dependency ordering.

<!-- Order matters — backbone depends on underscore -->
<script src="underscore.js"></script>
<script src="backbone.js"></script>
<script src="app.js"></script>

// app.js — everything is global
var model = new Backbone.Model({ name: "item" });
// If underscore.js fails to load, Backbone silently breaks

2. IIFE — The First Module Pattern

Immediately Invoked Function Expressions created private scopes via closures, exposing only a public API. This was the foundation of the "revealing module pattern."

var MyModule = (function() {
  // Private — not accessible outside
  var _count = 0;

  function _validate(n) {
    return typeof n === 'number' && n >= 0;
  }

  // Public API
  return {
    increment: function() { _count++; },
    getCount: function() { return _count; },
    setCount: function(n) {
      if (_validate(n)) _count = n;
    }
  };
})();

MyModule.increment();
MyModule.getCount(); // 1
MyModule._count;     // undefined — truly private

Architect Insight

IIFEs solved encapsulation but not dependency declaration. You still needed manual script ordering. The closure-based privacy pattern influenced the design of CommonJS and later ES Module private fields.

3. CommonJS (CJS)

Node.js adopted CommonJS in 2009, introducing require() and module.exports. Key characteristics:

// math.js — CommonJS module
const PI = 3.14159265;

function circleArea(r) {
  return PI * r * r;
}

function circumference(r) {
  return 2 * PI * r;
}

module.exports = { circleArea, circumference };
// or: exports.circleArea = circleArea;

// app.js — synchronous require
const { circleArea } = require('./math');
console.log(circleArea(5)); // 78.539...

CJS Internals

// Node.js wraps every module in a function:
(function(exports, require, module, __filename, __dirname) {
  // Your module code here
  const x = 42;
  module.exports = { x };
});

// require() algorithm (simplified):
function require(id) {
  // 1. Resolve: convert id to absolute path
  const filename = Module._resolveFilename(id);

  // 2. Cache check — modules are singletons
  if (Module._cache[filename]) {
    return Module._cache[filename].exports;
  }

  // 3. Create new module
  const module = new Module(filename);
  Module._cache[filename] = module;

  // 4. Load and execute
  module.load(filename);

  // 5. Return exports
  return module.exports;
}

Key CJS Properties

Synchronous loading — require() blocks until the file is read and executed. Fine for server-side (disk I/O), problematic for browsers (network I/O). Dynamic — require() can appear anywhere, including inside conditionals. Value copying — exports are copied by value (primitives) or reference (objects). Singleton cache — each module is loaded once and cached.

4. AMD (Asynchronous Module Definition)

RequireJS popularized AMD for browser environments where synchronous loading was unacceptable.

// AMD module definition
define('math', ['dependency1', 'dependency2'], function(dep1, dep2) {
  // Dependencies are resolved asynchronously before this runs
  return {
    add: function(a, b) { return a + b; },
    multiply: function(a, b) { return a * b; }
  };
});

// AMD require (async)
require(['math', 'ui'], function(math, ui) {
  ui.display(math.add(2, 3));
});

5. UMD (Universal Module Definition)

UMD wraps modules to work in CJS, AMD, and global contexts — the "write once, run anywhere" pattern for libraries.

(function(root, factory) {
  if (typeof define === 'function' && define.amd) {
    // AMD
    define(['dependency'], factory);
  } else if (typeof module === 'object' && module.exports) {
    // CommonJS
    module.exports = factory(require('dependency'));
  } else {
    // Browser global
    root.MyLibrary = factory(root.Dependency);
  }
}(typeof self !== 'undefined' ? self : this, function(Dependency) {
  // Module implementation
  return { /* public API */ };
}));

UMD Is Legacy

UMD was necessary when the ecosystem was fragmented. Modern libraries should publish ESM as primary format with CJS as fallback. UMD adds ~500 bytes of boilerplate per module and prevents tree shaking.

6. ES Modules (ESM)

ES2015 introduced native modules with import/export syntax — the first module system built into the language specification.

// math.mjs — named exports
export const PI = 3.14159265;

export function circleArea(r) {
  return PI * r * r;
}

// default export
export default class Calculator {
  add(a, b) { return a + b; }
}

// app.mjs — static imports (hoisted, resolved at parse time)
import Calculator, { circleArea, PI } from './math.mjs';
import * as MathUtils from './math.mjs'; // namespace import

// Re-exports
export { circleArea } from './math.mjs';
export { default as Calc } from './math.mjs';
export * from './math.mjs'; // re-export all named

Live Bindings Demonstration

// counter.mjs
export let count = 0;
export function increment() { count++; }

// app.mjs
import { count, increment } from './counter.mjs';
console.log(count); // 0
increment();
console.log(count); // 1 — live binding updates!

// In CJS, this would still be 0 (value was copied)

7. Dynamic Imports

// Dynamic import() returns a Promise — works in both ESM and CJS
const module = await import('./heavy-module.mjs');

// Conditional loading
if (user.isAdmin) {
  const { AdminPanel } = await import('./admin.mjs');
}

// React lazy loading
const LazyComponent = React.lazy(() => import('./HeavyComponent'));

// Webpack magic comments for chunk naming
const Chart = React.lazy(() =>
  import(/* webpackChunkName: "chart" */ './Chart')
);

// Vite glob imports
const modules = import.meta.glob('./modules/*.ts');
// Returns: { './modules/a.ts': () => import('./modules/a.ts'), ... }

8. Module Resolution Algorithms

Node.js CJS Resolution (require('X') from /a/b/c.js):

1. If X is a core module (fs, path, etc.) → return core module
2. If X starts with './' or '../' or '/'
   a. LOAD_AS_FILE(resolve(dirname, X))
   b. LOAD_AS_DIRECTORY(resolve(dirname, X))
3. LOAD_NODE_MODULES(X, dirname)

LOAD_AS_FILE(X):
  1. X                    → exact match
  2. X.js                 → append .js
  3. X.json               → append .json
  4. X.node               → native addon

LOAD_AS_DIRECTORY(X):
  1. X/package.json#main  → read "main" field
  2. X/index.js           → default index
  3. X/index.json

LOAD_NODE_MODULES(X, START):
  Walk up directory tree:
    /a/b/node_modules/X
    /a/node_modules/X
    /node_modules/X

ESM Resolution Differences

// ESM requires explicit file extensions (in Node.js)
import { foo } from './utils.js';     // ✓ explicit extension
import { foo } from './utils';        // ✗ ERR_MODULE_NOT_FOUND

// Bundlers (Webpack, Vite) resolve without extensions
// via resolve.extensions configuration
// webpack.config.js
module.exports = {
  resolve: {
    extensions: ['.tsx', '.ts', '.jsx', '.js'],
    alias: {
      '@': path.resolve(__dirname, 'src'),
      '@components': path.resolve(__dirname, 'src/components'),
    }
  }
};

9. package.json Exports Field

The "exports" field (Node.js 12.11+) provides fine-grained control over module entry points, replacing the older "main" field.

{
  "name": "my-library",
  "type": "module",
  "main": "./dist/cjs/index.js",       // Legacy CJS fallback
  "module": "./dist/esm/index.js",      // Bundler ESM (non-standard)
  "types": "./dist/types/index.d.ts",   // TypeScript declarations

  "exports": {
    ".": {
      "types": "./dist/types/index.d.ts",
      "import": "./dist/esm/index.js",
      "require": "./dist/cjs/index.cjs",
      "default": "./dist/esm/index.js"
    },
    "./utils": {
      "types": "./dist/types/utils.d.ts",
      "import": "./dist/esm/utils.js",
      "require": "./dist/cjs/utils.cjs"
    },
    "./package.json": "./package.json"
  },

  // Condition order matters — first match wins
  // "types" must come first for TypeScript
}

Architect Insight: Exports Encapsulation

The "exports" field acts as a public API boundary. Paths not listed in exports are inaccessible to consumers: import { internal } from 'my-library/dist/internal' will fail. This prevents consumers from depending on internal implementation details — a critical concern for library maintainers.

10. Dual CJS/ESM Packages

// Package structure for dual format
my-library/
├── dist/
│   ├── esm/
│   │   ├── index.js        // ESM output
│   │   └── package.json    // { "type": "module" }
│   └── cjs/
│       ├── index.cjs       // CJS output
│       └── package.json    // { "type": "commonjs" }
├── src/
│   └── index.ts            // Source (TypeScript)
├── package.json
└── tsconfig.json

The Dual Package Hazard

When a package is loaded as both CJS and ESM in the same application, two separate instances exist. Singletons break, instanceof checks fail, and state isn't shared. Mitigation: use a CJS wrapper that re-exports the ESM version, or use the "exports" field to ensure only one format loads per environment.

// CJS wrapper approach (avoids dual instance)
// dist/cjs/index.cjs
module.exports = require('../esm/index.js');
// This only works if the ESM doesn't use top-level await

// Alternative: isolate state in a separate module
// shared-state.cjs — always loaded as CJS
let instance = null;
module.exports = {
  getInstance() {
    if (!instance) instance = createInstance();
    return instance;
  }
};

11. Import Maps

Import maps allow bare specifier resolution in browsers without a bundler — a native alternative to bundler alias resolution.

<!-- index.html -->
<script type="importmap">
{
  "imports": {
    "react": "https://esm.sh/react@18.3.1",
    "react-dom/client": "https://esm.sh/react-dom@18.3.1/client",
    "lodash/": "https://esm.sh/lodash-es@4.17.21/",
    "@/": "./src/",
    "#utils": "./src/utils/index.js"
  },
  "scopes": {
    "/legacy/": {
      "react": "https://esm.sh/react@17.0.2"
    }
  }
}
</script>

<script type="module">
// Now bare specifiers work natively
import React from 'react';
import { createRoot } from 'react-dom/client';
import { debounce } from 'lodash/debounce';
import { api } from '@/services/api.js';
</script>

12. Module Federation (Webpack 5)

Module Federation enables micro-frontend architectures by allowing separately built applications to share modules at runtime.

Module Federation Architecture:

┌─────────────────────┐     ┌─────────────────────┐
│   Host Application  │     │  Remote Application  │
│                     │     │                      │
│  ┌───────────────┐  │     │  ┌───────────────┐   │
│  │ webpack config│  │     │  │ webpack config│   │
│  │ remotes: {    │  │     │  │ exposes: {    │   │
│  │   app2: url   │──┼────▶│  │   './Button'  │   │
│  │ }             │  │     │  │ }             │   │
│  └───────────────┘  │     │  └───────────────┘   │
│                     │     │                      │
│  import('app2/     │     │  Serves:             │
│    Button')         │     │  remoteEntry.js      │
│                     │     │  + shared chunks     │
└─────────────────────┘     └──────────────────────┘
         │                            │
         └──────── Shared ────────────┘
              react, react-dom
           (singleton, version check)

// Host webpack.config.js
const { ModuleFederationPlugin } = require('webpack').container;

module.exports = {
  plugins: [
    new ModuleFederationPlugin({
      name: 'host',
      remotes: {
        app2: 'app2@http://localhost:3002/remoteEntry.js',
      },
      shared: {
        react: { singleton: true, requiredVersion: '^18.0.0' },
        'react-dom': { singleton: true, requiredVersion: '^18.0.0' },
      },
    }),
  ],
};

// Remote webpack.config.js
module.exports = {
  plugins: [
    new ModuleFederationPlugin({
      name: 'app2',
      filename: 'remoteEntry.js',
      exposes: {
        './Button': './src/components/Button',
        './utils': './src/utils',
      },
      shared: {
        react: { singleton: true, requiredVersion: '^18.0.0' },
        'react-dom': { singleton: true, requiredVersion: '^18.0.0' },
      },
    }),
  ],
};

// Host app — consuming remote module
const RemoteButton = React.lazy(() => import('app2/Button'));

function App() {
  return (
    <Suspense fallback="Loading...">
      <RemoteButton onClick={() => alert('From remote!')} />
    </Suspense>
  );
}

13. Tree Shaking

Tree shaking eliminates unused exports from the final bundle. It requires specific conditions to work effectively.

Requirements for Tree Shaking

RequirementWhyHow
ES Module syntaxStatic structure enables analysisUse import/export, not require()
Side-effect-free modulesBundler must know removal is safe"sideEffects": false in package.json
Pure annotationsMark function calls as removable/*#__PURE__*/ comments
Production modeMinifier removes dead codemode: 'production' in webpack
No barrel re-export issuesBarrel files can defeat tree shakingDirect imports or optimized barrels
 
// package.json — sideEffects field
{
  "name": "my-library",
  "sideEffects": false,
  // or specify files WITH side effects:
  "sideEffects": [
    "*.css",
    "./src/polyfills.js",
    "./src/register-globals.js"
  ]
}

// Pure annotation — tells bundler this call can be removed
const Button = /*#__PURE__*/ React.memo(function Button(props) {
  return <button>{props.label}</button>;
});

// Barrel file anti-pattern
// index.ts (barrel)
export { Button } from './Button';
export { Modal } from './Modal';
export { Table } from './Table';
// import { Button } from './components';
// ↑ May pull in Modal and Table depending on bundler

// Better: direct imports
// import { Button } from './components/Button';

Architect Insight: Tree Shaking in Practice

At Staff level, you should know that tree shaking is a two-phase process: (1) the bundler marks unused exports during the "module concatenation" phase, (2) the minifier (Terser/esbuild) actually removes the dead code. Webpack's usedExports optimization marks exports, but without minification the code remains. Also, /*#__PURE__*/ annotations are critical for library authors — React itself uses them extensively.
