Interview Q&A — Build Toolchain

55+ L6/Staff-level questions on module systems, bundlers, transpilation, TypeScript with React, CSS solutions, monorepo tooling, and developer experience.

Module Systems

Q1: Explain the difference between CommonJS and ES Modules. Why does this matter for bundlers?

A: CommonJS uses require()/module.exports with synchronous loading and dynamic structure — require can appear inside conditionals. ES Modules use import/export with asynchronous loading and static structure — imports must be at the top level. This matters because ESM's static structure enables bundlers to perform tree shaking (dead code elimination) at build time. CJS's dynamic nature makes it impossible to statically determine which exports are used. Additionally, ESM uses live bindings (references) while CJS copies values, which affects how circular dependencies resolve.

Q2: What are "live bindings" in ES Modules and how do they differ from CJS exports?

A: In ESM, when you import a binding, you get a live reference to the original variable in the exporting module. If the exporting module changes the value, the importing module sees the update. In CJS, require() copies primitive values at the time of import — subsequent changes in the exporting module aren't reflected. For objects, CJS copies the reference, so mutations are shared, but reassignment isn't. This distinction matters for circular dependencies: ESM handles them gracefully via live bindings, while CJS can produce undefined values if modules haven't finished executing.

Q3: What is the package.json "exports" field and why is it important?

A: The "exports" field (Node.js 12.11+) provides fine-grained control over a package's public API entry points. It supports conditional exports (different files for import vs require), subpath exports (e.g., "./utils"), and encapsulation — paths not listed in exports are inaccessible. This is important because it: (1) enables dual CJS/ESM packages, (2) prevents consumers from importing internal implementation files, (3) allows different entry points for different environments (browser vs Node), and (4) the "types" condition must come first for TypeScript resolution.

Q4: Explain the "dual package hazard" and how to mitigate it.

A: When a package is loaded as both CJS and ESM in the same application (e.g., your code imports ESM, a dependency requires CJS), two separate instances exist in memory. Singletons break, instanceof checks fail across instances, and state isn't shared. Mitigation strategies: (1) use a CJS wrapper that re-exports the ESM version so only one instance exists, (2) use the "exports" field to ensure consistent resolution, (3) isolate shared state in a separate module, or (4) publish only ESM and let consumers handle CJS compatibility.

Q5: What are import maps and when would you use them?

A: Import maps are a browser-native mechanism (<script type="importmap">) that allows bare specifier resolution without a bundler. They map package names to URLs, similar to how bundlers resolve import React from 'react'. Use cases: (1) bundler-free development with CDN-hosted dependencies, (2) micro-frontend architectures where shared dependencies are loaded from a central location, (3) prototyping without build tooling. They support scoped mappings for version isolation per route.

Q6: How does Module Federation work in Webpack 5?

A: Module Federation allows separately built and deployed applications to share modules at runtime. A "remote" application exposes modules via a remoteEntry.js manifest file. A "host" application declares remotes and can dynamically import their exposed modules. Shared dependencies (like React) are configured as singletons with version constraints — the runtime negotiates which version to use. This enables micro-frontend architectures where teams deploy independently while sharing code. The key mechanism is async script loading of the remote entry, which registers available modules in a shared scope.

Q7: What conditions must be met for tree shaking to work effectively?

A: Five conditions: (1) ES Module syntax (import/export, not require) for static analysis, (2) "sideEffects": false in package.json to tell the bundler that unused imports can be safely removed, (3) /*#__PURE__*/ annotations on function calls that the bundler can't prove are side-effect-free, (4) production mode with minification (the bundler marks dead code, the minifier removes it), (5) avoiding barrel file anti-patterns where re-exporting everything from an index file can defeat tree shaking in some bundlers.

Bundler Internals

Q8: Explain Webpack's compilation lifecycle at a high level.

A: Webpack uses a Tapable plugin system with these phases: (1) Initialization — read config, create compiler, (2) Make — starting from entry points, parse each file's AST to find imports, resolve paths, and recursively build the dependency graph, (3) Seal — optimize the graph, create chunks based on code splitting configuration, (4) Emit — generate output files (bundles, source maps, manifests), write to disk. Loaders transform individual files during the Make phase, while plugins hook into any phase via Tapable hooks.

Q9: How does Vite achieve fast development server startup compared to Webpack?

A: Vite takes a fundamentally different approach: instead of bundling everything before serving, it serves native ES Modules directly to the browser. The browser's module loader handles resolution and loading. Vite only transforms files on demand when the browser requests them (using esbuild for speed). Dependencies from node_modules are pre-bundled once using esbuild (converting CJS to ESM and merging many small modules). This means startup time is nearly constant regardless of app size, while Webpack must build the entire dependency graph before serving anything.

Q10: What is the difference between Webpack loaders and plugins?

A: Loaders operate on individual files — they're functions that receive source content and return transformed content. They run during the module building phase, applied per-file based on test patterns (e.g., babel-loader for .tsx files). Loaders execute right-to-left (bottom-to-top in config). Plugins operate on the entire compilation — they hook into Webpack's lifecycle events via the Tapable system and can modify the dependency graph, chunks, assets, or output. Plugins have access to the full compilation object, while loaders only see individual file content.

Q11: Describe your code splitting strategy for a large React application.

A: A layered approach: (1) Route-level splitting — each route is a lazy-loaded chunk via React.lazy(), (2) Vendor splitting — separate chunks for React, router, and major libraries (long cache TTL), (3) Feature splitting — heavy features (charts, editors, admin panels) loaded on demand, (4) Shared chunk — common utilities used across routes extracted automatically via splitChunks.cacheGroups. The goal is maximizing cache hit rates: vendor chunks change rarely, route chunks change with features. I use webpack-bundle-analyzer to monitor chunk sizes and identify optimization opportunities.

Q12: How does Hot Module Replacement (HMR) work internally?

A: HMR is a five-step process: (1) file watcher detects a change, (2) bundler recompiles only the affected module(s), (3) bundler sends an update notification to the client via WebSocket (containing the module hash), (4) client-side HMR runtime fetches the updated module code, (5) the runtime applies the update by replacing the old module and calling accept handlers. If no accept handler exists for the changed module, the update "bubbles up" to parent modules. If it reaches the root without being accepted, a full page reload occurs.

Q13: Compare esbuild and Turbopack. When would you choose each?

A: esbuild (Go) is a standalone bundler/minifier focused on raw speed — 50-100x faster than Webpack. It has a limited plugin API and basic code splitting (ESM only). Best for: dependency pre-bundling (used by Vite), standalone builds of libraries, and as a minifier. Turbopack (Rust) is an incremental bundler built on function-level memoization — it caches individual transform results and only recomputes what changed. It's integrated with Next.js. Best for: large Next.js applications where incremental rebuilds matter. esbuild is more mature and standalone; Turbopack is newer and framework-specific.

Q14: What are source maps and what are the security considerations?

A: Source maps are JSON files that map minified/bundled code back to original source code, enabling debugging in browser DevTools. They use VLQ-encoded mappings to connect generated line/column positions to original file/line/column/name positions. Security concern: public source maps expose your entire source code. Mitigations: (1) hidden-source-map — generates maps without a reference comment in the bundle, (2) upload maps to error tracking services (Sentry) privately, (3) nosources-source-map — provides stack traces without source content, (4) restrict access to .map files via server configuration.

Transpilation

Q15: Explain Babel's three-phase architecture.

A: Phase 1: Parsing — @babel/parser (formerly Babylon) converts source code into an Abstract Syntax Tree (AST), a tree representation of the code's structure. Phase 2: Transformation — @babel/traverse walks the AST using the visitor pattern, and plugins modify nodes (e.g., converting JSX to function calls, arrow functions to regular functions). Plugins run first-to-last, presets run last-to-first. Phase 3: Code Generation — @babel/generator converts the transformed AST back into a source string plus an optional source map.

Q16: What is the difference between the classic and automatic JSX transforms?

A: The classic transform (pre-React 17) converts JSX to React.createElement() calls, requiring React to be in scope in every JSX file. The automatic transform (React 17+) converts JSX to jsx()/jsxs() calls from react/jsx-runtime, automatically imported by the compiler — no manual React import needed. The automatic transform also produces slightly smaller output and adds source location info in development mode (jsxDEV from react/jsx-dev-runtime). It's enabled via runtime: 'automatic' in Babel preset-react or jsx: 'react-jsx' in tsconfig.

Q17: Explain the difference between syntax transforms and API polyfills.

A: Syntax transforms change code structure at build time — the transpiler rewrites modern syntax to equivalent older syntax (arrow functions → regular functions, optional chaining → ternary checks, class fields → constructor assignments). API polyfills add missing built-in methods/objects at runtime — they provide implementations of Array.prototype.includes, Promise.allSettled, Object.fromEntries, etc. Babel handles syntax transforms via plugins; API polyfills come from core-js. The useBuiltIns: 'usage' option in preset-env automatically imports only the polyfills your code actually uses.

Q18: Why is isolatedModules important when using SWC or esbuild for TypeScript?

A: SWC, esbuild, and Babel process TypeScript files individually — they strip types without cross-file type information. Certain TypeScript features require cross-file analysis: const enum (needs to inline values from other files), export = / import = (CJS-style), and namespace merging. Without isolatedModules: true, these features silently produce incorrect output. With it enabled, TypeScript reports errors on unsupported patterns, ensuring your code is compatible with single-file transpilation tools.

Q19: Compare the three TypeScript compilation strategies for React projects.

A: Strategy 1: tsc only — official compiler handles both type checking and emit. Slow but correct. Strategy 2: SWC/Babel + tsc --noEmit — SWC/Babel strips types for fast builds, tsc runs separately for type checking (often in CI). Most common in production React projects. Strategy 3: Bundler integration — Vite uses esbuild for dev (fastest transforms) and Rollup+SWC for production, with tsc --noEmit in CI. The key insight is separating transpilation (fast, per-file) from type checking (slow, whole-project) and running them in parallel.

TypeScript with React

Q20: Why is React.FC discouraged and what should you use instead?

A: React.FC was discouraged because it: (1) implicitly included children in props (fixed in React 18 types), (2) made generic components awkward to type, (3) broke defaultProps type inference, (4) added unnecessary type complexity. Instead, use plain functions with typed props: function Button({ label }: ButtonProps) { ... }. This is simpler, supports generics naturally, and the return type is inferred. If you need an explicit return type, use JSX.Element or React.ReactNode.

Q21: How do you create a type-safe generic component in React?

A: Define the component as a generic function: function List<T>({ items, renderItem }: ListProps<T>). TypeScript infers the generic type from usage — when you pass items={users}, T is inferred as User, and renderItem receives (item: User). For arrow functions, use the syntax const List = <T,>(props: ListProps<T>) => ... (the trailing comma disambiguates from JSX). Generic components are essential for reusable data-driven components like tables, lists, selects, and form fields.

Q22: Explain discriminated unions for React component props.

A: Discriminated unions use a shared literal type field (the "discriminant") to create mutually exclusive prop combinations. Example: type Props = { variant: 'link'; href: string } | { variant: 'button'; onClick: () => void }. When you check props.variant, TypeScript narrows the type and knows which other props are available. This is superior to optional props because it prevents invalid combinations (a button with an href) and enables exhaustive checking with never in switch defaults.

Q23: How do you type a polymorphic "as" prop in React?

A: Use React.ElementType as the generic constraint and React.ComponentPropsWithoutRef to get the correct props for the rendered element: type Props<E extends React.ElementType> = { as?: E } & Omit<React.ComponentPropsWithoutRef<E>, 'as'>. When used as <Box as="a" href="/about">, TypeScript knows href is valid because the element is an anchor. This pattern is used by UI libraries like Chakra UI and Radix for maximum flexibility.

Q24: What is noUncheckedIndexedAccess and how does it affect React code?

A: noUncheckedIndexedAccess makes array/object index access return T | undefined instead of T. In React, this affects: items[0] returns Item | undefined (must null-check before use), Object.keys(obj).forEach(key => obj[key]) needs narrowing, and destructured array elements may be undefined. It catches real bugs (accessing beyond array bounds) but requires more null checks. Worth enabling for new projects; migration effort for existing ones.

CSS Solutions

Q25: Compare CSS Modules, styled-components, and Tailwind CSS for a large React project.

A: CSS Modules: zero runtime cost, standard CSS with local scoping via hashed class names. Good for teams comfortable with CSS, but no dynamic styling based on props. styled-components: runtime CSS generation from tagged template literals, excellent DX with prop-based styling, but adds ~16KB to bundle and has render-time performance cost. Tailwind CSS: utility-first classes, zero runtime, ~5-15KB CSS output via JIT purging. Excellent for consistent design systems but long class strings. For new large projects, the industry trend favors Tailwind or zero-runtime solutions (Vanilla Extract) over runtime CSS-in-JS, especially with React Server Components.

Q26: Why is runtime CSS-in-JS problematic with React Server Components?

A: React Server Components render on the server and stream HTML to the client without hydrating. Runtime CSS-in-JS libraries (styled-components, Emotion) generate styles during React rendering via JavaScript — they need the component to render in a JavaScript environment to produce CSS. Server Components don't execute on the client, so there's no JavaScript runtime to generate styles. This is why zero-runtime solutions (CSS Modules, Tailwind, Vanilla Extract, Linaria) are preferred for RSC-compatible architectures — they produce static CSS at build time.

Q27: Explain CSS cascade layers and how they help in large applications.

A: CSS @layer provides explicit control over the cascade order, independent of selector specificity. You declare layers in order: @layer reset, base, components, utilities; — later layers always win regardless of specificity. This solves the specificity wars problem in large applications where component styles accidentally override each other. Unlayered styles always beat layered styles. For React, this means you can ensure utility classes always override component styles without !important, and third-party CSS can be isolated in its own layer.

Q28: What is Vanilla Extract and why is it gaining adoption?

A: Vanilla Extract is a zero-runtime, type-safe CSS-in-TypeScript solution. You write styles in .css.ts files using TypeScript — the style() function accepts a typed CSS object where properties and values are checked at compile time. At build time, these are extracted to static CSS files with no JavaScript runtime. Benefits: full TypeScript autocomplete for CSS, theme tokens as typed variables, styleVariants for variant patterns, and zero bundle size impact. It combines the DX of CSS-in-JS with the performance of static CSS.

Monorepo Tooling

Q29: What are phantom dependencies and how does pnpm prevent them?

A: Phantom dependencies occur when your code imports a package that isn't in your package.json but is accessible because npm/yarn hoisted it from a transitive dependency. If that transitive dependency drops the package, your code breaks. pnpm prevents this with a strict, symlinked node_modules structure: only packages listed in your package.json are symlinked into your node_modules. Transitive dependencies are stored in a content-addressable store (.pnpm/) and only accessible to the packages that declared them. This enforces explicit dependency declarations.

Q30: Compare Turborepo and Nx for monorepo management.

A: Turborepo: minimal, focused on task orchestration and caching. Add a turbo.json and you're running. Uses content hashing for caching, supports Vercel remote cache. No code generators or framework plugins. Nx: full-featured with computation caching, affected commands (only run on changed packages), code generators, dependency graph visualization, and deep framework plugins. More setup but more features. Choose Turborepo for simplicity and incremental adoption; choose Nx for large organizations that benefit from generators, affected commands, and the visual dependency graph.

Q31: How does remote caching work in a monorepo build system?

A: Remote caching shares build artifacts across machines (developers and CI). The process: (1) hash all inputs for a task (source files, dependencies, env vars, config), (2) check remote cache for matching hash, (3) on hit — download cached outputs and replay logs (skip the build entirely), (4) on miss — run the build, upload outputs to remote cache. This means if Developer A builds a package, Developer B and CI get instant cache hits. Turborepo uses Vercel's remote cache; Nx uses Nx Cloud. Both support self-hosted alternatives. Cache artifacts are signed for integrity.

Q32: Explain the difference between fixed and independent versioning in a monorepo.

A: Fixed versioning: all packages share the same version number. When any package changes, all are bumped to the new version. Used by React, Babel, and Angular — good for tightly coupled packages where consumers expect version alignment. Independent versioning: each package has its own version, bumped based on its own changes. Good for loosely coupled packages like utility libraries. Changesets is a popular tool for independent versioning — developers create "changeset" files describing their changes, and the tool automatically determines version bumps and generates changelogs.

Developer Experience

Q33: Why is the exhaustive-deps ESLint rule important and how should violations be handled?

A: react-hooks/exhaustive-deps ensures all values referenced inside useEffect/useMemo/useCallback are in the dependency array, preventing stale closure bugs. Violations should NOT be handled with eslint-disable — that hides real bugs. Instead: (1) move functions inside the effect, (2) use refs for values that shouldn't trigger re-runs, (3) use useReducer for complex state updates that reference current state, (4) extract the logic into a custom hook. The rule exists because stale closures are one of the most common and hardest-to-debug React bugs.

Q34: How does React Fast Refresh work internally?

A: Fast Refresh is built on HMR with React-specific intelligence. The bundler plugin injects registration code after each component: $RefreshReg$(App, "App") and a signature based on hook call order: $RefreshSig$(App, "useState{[count]}\\nuseContext{theme}"). When a file is updated: (1) the new module executes, re-registering components, (2) the runtime compares old and new signatures, (3) if signatures match (same hooks), it re-renders preserving state, (4) if signatures differ (hooks added/removed), it remounts resetting state. Non-component exports trigger a full module reload that bubbles up.

Q35: What does React StrictMode do and why does it double-invoke effects?

A: StrictMode enables development-only checks: double-invoking render functions (to detect impure renders), double-invoking effects (setup → cleanup → setup, to catch missing cleanup), detecting legacy APIs, and detecting unexpected side effects. The double-invoke of effects simulates a component being unmounted and remounted, which happens during Fast Refresh and will happen with React's Offscreen API. If an effect breaks when run twice, it has a cleanup bug — the fix is always to add proper cleanup, not to remove StrictMode.

Q36: Explain the security implications of client-side environment variables.

A: Client-side env vars (VITE_, NEXT_PUBLIC_, REACT_APP_) are embedded in the JavaScript bundle at build time as string literals. Anyone can view them by inspecting the bundle. Never put secrets in client-exposed env vars — API keys with write access, database URLs, auth secrets, etc. Use server-side API routes or a BFF to proxy requests needing secrets. Even "read-only" API keys should be protected if they provide access to sensitive data. The prefix convention (VITE_, NEXT_PUBLIC_) is a safety mechanism — variables without the prefix are only available server-side.

Q37: How do you set up a pre-commit workflow for a React monorepo?

A: Use Husky for git hooks and lint-staged for running linters on staged files only. Setup: (1) husky init creates .husky/pre-commit, (2) lint-staged config runs ESLint --fix and Prettier --write on staged .ts/.tsx files, (3) commitlint enforces conventional commit messages via a commit-msg hook. Important: don't run tsc in pre-commit — it's too slow (needs the full project). Run type checking in CI instead. For monorepos, lint-staged runs per-package, and Turborepo can cache lint results.

Q38: How does the React DevTools extension communicate with the React runtime?

A: The extension injects a content script that sets window.__REACT_DEVTOOLS_GLOBAL_HOOK__ before React loads. When ReactDOM initializes, it detects this hook and registers itself as a renderer. On every commit (after React updates the DOM), React calls onCommitFiberRoot on the hook, giving DevTools access to the fiber tree. DevTools walks the fiber tree to build its component tree representation. Communication between the page and the DevTools panel uses a "bridge" — message passing through content script → background script → DevTools panel via postMessage.

Advanced / Cross-Cutting

Q39: Design the build toolchain for a new large-scale React application.

A: I'd choose: Vite as the dev server and build tool (fast ESM dev, Rollup production builds), SWC for transpilation (20-70x faster than Babel), TypeScript with strict mode and tsc --noEmit in CI for type checking, pnpm for strict dependency management, Turborepo for monorepo task orchestration with remote caching, Tailwind CSS for styling (zero runtime, consistent design system), ESLint flat config with react-hooks and accessibility plugins, Prettier for formatting, Husky + lint-staged for pre-commit, and changesets for versioning if publishing packages. This stack optimizes for developer velocity, build performance, and type safety.

Q40: How would you migrate a large CRA (Create React App) project to Vite?

A: Step-by-step: (1) Replace react-scripts with vite and @vitejs/plugin-react, (2) move index.html to project root and add <script type="module" src="/src/main.tsx">, (3) rename REACT_APP_ env vars to VITE_ and update process.env references to import.meta.env, (4) update import paths (Vite requires explicit extensions for some cases), (5) replace CRA-specific features (proxy in package.json → vite.config server.proxy), (6) update test setup (Jest → Vitest for consistency), (7) handle SVG imports (CRA's SVGR → vite-plugin-svgr), (8) verify CSS Modules work (usually seamless). Run in parallel with CRA during migration for safety.

Q41: How do you debug a production bundle size issue?

A: Systematic approach: (1) Run webpack-bundle-analyzer or rollup-plugin-visualizer to see a treemap of chunk contents, (2) check for duplicate dependencies (same library at different versions), (3) verify tree shaking is working — look for unused library code in the bundle, (4) check for barrel file issues pulling in entire libraries, (5) verify sideEffects: false is set in library package.json files, (6) check for large dependencies that could be lazy-loaded or replaced (moment.js → date-fns, lodash → lodash-es), (7) verify code splitting is working — large chunks should be split by route, (8) check source maps aren't included in production.

Q42: Explain differential serving and its trade-offs.

A: Differential serving produces two bundles: a modern bundle (ES2020+, no polyfills, ~40% smaller) served via <script type="module">, and a legacy bundle (ES5, full polyfills) served via <script nomodule>. Modern browsers load the module script and ignore nomodule; legacy browsers do the opposite. Trade-offs: doubles build time, complicates deployment and caching, requires careful testing of both bundles, and the legacy audience is shrinking. Most teams now target modern browsers only (ES2020+) and drop IE11 support, making differential serving unnecessary.

Q43: How would you set up a shared component library in a monorepo?

A: Create an internal package (e.g., packages/ui) with: (1) package.json pointing "main" to source files (not built output) for development, (2) TypeScript config extending the shared base, (3) components exported from an index.ts barrel file, (4) Storybook for component documentation and visual testing, (5) unit tests with React Testing Library. Consumer apps reference it via "@myorg/ui": "workspace:*". For publishing externally, add a build step that produces ESM + CJS + type declarations. Use changesets for versioning. The key decision: source-level imports (faster dev) vs built imports (closer to production).

Q44: What is the browserslist configuration and which tools consume it?

A: Browserslist defines target browsers using queries like > 0.5%, not dead, not ie 11. It's consumed by: Babel (preset-env determines which transforms and polyfills are needed), Autoprefixer (which CSS vendor prefixes to add), PostCSS (CSS transforms), ESLint (browser compatibility checks), and Stylelint. It can be configured in package.json's "browserslist" field or a .browserslistrc file. It supports environment-specific configs (production vs development). Use npx browserslist to see resolved browser versions.

Q45: How do you handle CSS custom properties (variables) for theming in React?

A: Set CSS custom properties on a wrapper element via React's style prop: style={{ '--color-primary': theme.primary } as React.CSSProperties}. CSS files consume them via var(--color-primary). Benefits over JS-based theming: (1) zero re-renders on theme change (CSS handles it), (2) works with any CSS solution (Modules, Tailwind, vanilla), (3) inherits through the DOM tree naturally, (4) can be animated with CSS transitions. For dark mode, toggle a class on the root element that changes the custom property values. This approach has zero runtime cost compared to context-based theming.

Q46: What is the tailwind-merge library and why is it needed?

A: tailwind-merge intelligently merges Tailwind CSS classes by resolving conflicts. Without it, className="px-4 px-6" applies both classes (last one wins in CSS, but order isn't guaranteed). twMerge("px-4 px-6") returns "px-6" — it knows px-4 and px-6 conflict and keeps the last one. This is essential for component libraries where a base component has default classes and consumers pass overriding classes. Combined with clsx for conditional classes, the cn() utility (twMerge(clsx(...))) is the standard pattern for Tailwind component APIs.

Q47: Explain the content-addressable store in pnpm.

A: pnpm stores all downloaded packages in a global content-addressable store (typically ~/.pnpm-store). Each file is stored once, identified by its content hash. When a project needs a package, pnpm creates hard links from the store to the project's node_modules/.pnpm/ directory. Benefits: (1) disk space savings — a file shared by 100 projects is stored once, (2) faster installs — no copying, just linking, (3) integrity — content-addressed means corruption is detectable. This is fundamentally different from npm/yarn which copy files into each project's node_modules.

Q48: How do you configure TypeScript path aliases to work across bundler, editor, and test runner?

A: Three places need configuration: (1) tsconfig.json — "paths": { "@/*": ["./src/*"] } for editor IntelliSense and type checking, (2) Bundler — Vite: resolve.alias: { '@': path.resolve('src') }, Webpack: resolve.alias, (3) Test runner — Jest: moduleNameMapper: { "^@/(.*)$": "<rootDir>/src/$1" }, Vitest: inherits from vite.config. All three must be kept in sync. Some tools like vite-tsconfig-paths read tsconfig paths automatically, reducing duplication.

Q49: What is the React Compiler and how does it relate to build tooling?

A: The React Compiler (formerly React Forget) is a build-time compiler that automatically adds memoization to React components. It analyzes component code and inserts useMemo, useCallback, and React.memo equivalents where beneficial, eliminating the need for manual memoization. It integrates as a Babel plugin or bundler plugin. This shifts optimization from runtime (manual hooks) to build time (automatic analysis). It requires components to follow React's rules (pure rendering, no side effects in render) — which StrictMode helps enforce.

Q50: How would you implement a custom Babel plugin for your React project?

A: A Babel plugin is a function returning a visitor object that matches AST node types. Example use cases: (1) auto-inject analytics tracking into event handlers, (2) transform custom DSL syntax, (3) strip development-only code. Implementation: use @babel/types for node creation, path.replaceWith() for substitution, path.remove() for deletion. Test with @babel/core's transformSync and snapshot tests comparing input/output. Key consideration: plugins run on every file in every build — keep them fast and avoid complex analysis.

Q51: How do you handle environment-specific builds in a React application?

A: Layer environment configuration: (1) .env files for environment-specific values (.env.development, .env.production, .env.staging), (2) build-time substitution via DefinePlugin (Webpack) or define (Vite) for compile-time constants, (3) runtime configuration for values that change without rebuilding (fetched from a config endpoint or injected via server-rendered HTML). The key principle: build-time config is baked into the bundle (requires rebuild to change), runtime config is loaded at startup (can change without rebuild). Use build-time for feature flags and API URLs; use runtime for A/B test assignments and dynamic settings.

Q52: What are the trade-offs of using TypeScript's strict mode in a large React codebase?

A: Benefits: catches null/undefined bugs (strictNullChecks), prevents implicit any types, enforces proper function typing, catches uninitialized class properties. Trade-offs: (1) migration effort — enabling strict on an existing codebase produces hundreds/thousands of errors, (2) more verbose code — null checks, type assertions, type guards add lines, (3) slower onboarding for developers new to TypeScript, (4) some patterns become awkward (dynamic property access, JSON parsing). Strategy: enable strict for new projects, migrate existing projects incrementally using // @ts-strict-ignore per-file or skipLibCheck for third-party types.

Q53: How do container queries differ from media queries and when should you use them?

A: Media queries respond to the viewport size; container queries respond to a parent container's size. This matters for reusable components: a Card component should adapt to its container width, not the viewport. With media queries, a Card in a sidebar behaves differently than the same Card in a main content area only if you add wrapper-specific classes. With container queries, the Card adapts automatically based on its container's dimensions. Use container queries for component-level responsiveness; use media queries for page-level layout changes.

Q54: How do you optimize Tailwind CSS for production?

A: Tailwind's JIT engine handles most optimization automatically: (1) only generates classes actually used in your source files (configured via content array), (2) produces a single CSS file typically 5-15KB gzipped. Additional optimizations: (3) use @apply sparingly — it duplicates CSS, (4) extract common patterns into components rather than repeated class strings, (5) use tailwind-merge to avoid conflicting classes, (6) configure theme.extend instead of overriding to keep the default design tokens, (7) use CSS layers to control specificity with third-party CSS.

Q55: Describe the ideal CI/CD pipeline for a React monorepo.

A: Pipeline stages: (1) Install — pnpm install with frozen lockfile, cache node_modules and pnpm store, (2) Lint — ESLint + Prettier check (parallelized across packages via Turborepo), (3) Type check — tsc --noEmit for each package, (4) Test — unit tests with coverage thresholds, affected-only for PRs, (5) Build — production builds with Turborepo remote caching, (6) E2E — Playwright tests against preview deployments, (7) Deploy — preview deploys for PRs, production deploy on main merge. Key optimizations: Turborepo remote cache across CI runs, affected-only testing for PRs, parallel jobs for independent packages, and caching of node_modules between runs.
