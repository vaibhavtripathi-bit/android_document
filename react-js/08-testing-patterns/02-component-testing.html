React Testing Library Philosophy

React Testing Library (RTL) is built on a single guiding principle: "The more your tests resemble the way your software is used, the more confidence they can give you." This means testing from the user's perspective — interacting with rendered output, not internal component state.

Enzyme (Implementation Testing): RTL (Behavior Testing): Component Instance Rendered DOM │ │ ├─ .state() ├─ screen.getByRole() ├─ .props() ├─ screen.getByText() ├─ .find('.className') ├─ userEvent.click() ├─ .simulate('click') ├─ userEvent.type() └─ .instance().method() └─ screen.findByText() (async) │ │ Tests HOW it works Tests WHAT user sees/does (fragile to refactoring) (resilient to refactoring)
Key Principle

RTL intentionally does NOT expose component instances, state, or props. You cannot access component.state or call internal methods. This constraint forces you to test through the public interface — the rendered DOM — which is exactly what users interact with. If you find yourself needing to test internal state, it's a signal that either the test is testing implementation details or the component needs to expose that state through its rendered output.

Core API: render, screen, userEvent

render()

import { render, screen } from '@testing-library/react';
import userEvent from '@testing-library/user-event';

// Basic render
test('renders greeting', () => {
  render(<Greeting name="Alice" />);
  expect(screen.getByText('Hello, Alice!')).toBeInTheDocument();
});

// render returns utilities (prefer screen for queries)
const { container, unmount, rerender } = render(<Counter initial={0} />);

// Rerender with new props (same component instance)
rerender(<Counter initial={5} />);

// Unmount the component
unmount();

// container gives direct DOM access (use sparingly)
const div = container.querySelector('.custom-class');

screen

// screen is the preferred way to query rendered output
// It queries the document.body, so it works with portals too

import { screen } from '@testing-library/react';

// screen provides all query variants
screen.getByRole('button', { name: /submit/i });
screen.getByText('Hello World');
screen.getByLabelText('Email');
screen.getByPlaceholderText('Search...');
screen.getByDisplayValue('current value');
screen.getByAltText('Profile photo');
screen.getByTitle('Close');
screen.getByTestId('custom-element'); // Last resort

// Debug: print current DOM state
screen.debug(); // Prints entire document
screen.debug(screen.getByRole('form')); // Prints specific element

userEvent

import userEvent from '@testing-library/user-event';

test('complete form interaction', async () => {
  const user = userEvent.setup();
  render(<RegistrationForm />);

  // Type into inputs (fires keydown, keypress, input, keyup)
  await user.type(screen.getByLabelText('Name'), 'Alice');
  await user.type(screen.getByLabelText('Email'), 'alice@example.com');

  // Clear and retype
  await user.clear(screen.getByLabelText('Name'));
  await user.type(screen.getByLabelText('Name'), 'Bob');

  // Click buttons
  await user.click(screen.getByRole('button', { name: /register/i }));

  // Keyboard interactions
  await user.keyboard('{Enter}');
  await user.tab();

  // Select from dropdown
  await user.selectOptions(
    screen.getByRole('combobox', { name: /country/i }),
    'US'
  );

  // Upload file
  const file = new File(['content'], 'photo.png', { type: 'image/png' });
  await user.upload(screen.getByLabelText('Avatar'), file);

  // Hover
  await user.hover(screen.getByText('Tooltip trigger'));
  expect(screen.getByRole('tooltip')).toBeInTheDocument();
});
userEvent vs fireEvent

Always prefer userEvent over fireEvent. userEvent simulates real user interactions — typing fires keydown, keypress, input, change, and keyup events in sequence. fireEvent dispatches a single synthetic event. userEvent catches bugs that fireEvent misses (e.g., a component that only listens for keydown, not change). The only exception: fireEvent for events userEvent doesn't support (e.g., custom events, scroll).

// getBy — element MUST exist (synchronous)
const button = screen.getByRole('button', { name: /submit/i });

// queryBy — element MIGHT NOT exist (for asserting absence)
expect(screen.queryByText('Error message')).not.toBeInTheDocument();

// findBy — element appears ASYNCHRONOUSLY (returns Promise)
const successMessage = await screen.findByText('Saved successfully');

// Pattern: assert absence, trigger action, assert presence
test('shows error after invalid submission', async () => {
  render(<Form />);

  // Initially no error
  expect(screen.queryByRole('alert')).not.toBeInTheDocument();

  // Submit empty form
  await userEvent.click(screen.getByRole('button', { name: /submit/i }));

  // Error appears (might be async due to validation)
  expect(await screen.findByRole('alert')).toHaveTextContent(
    'Name is required'
  );
});

Accessibility-First Queries

RTL provides a priority order for queries that mirrors how users (including assistive technology users) find elements. Following this order improves both test quality and application accessibility.

Query Priority (Most to Least Preferred): 1. getByRole ← Accessible to everyone (screen readers, visual users) 2. getByLabelText ← Form elements (what the label says) 3. getByPlaceholderText ← When no label exists 4. getByText ← Non-interactive elements (paragraphs, spans) 5. getByDisplayValue ← Filled form elements (current value) 6. getByAltText ← Images 7. getByTitle ← Title attribute (less accessible) 8. getByTestId ← LAST RESORT (not visible to users) 
// BEST: getByRole — tests accessibility AND finds element
screen.getByRole('button', { name: /submit order/i });
screen.getByRole('heading', { name: /product details/i, level: 2 });
screen.getByRole('textbox', { name: /email/i });
screen.getByRole('checkbox', { name: /agree to terms/i });
screen.getByRole('combobox', { name: /country/i });
screen.getByRole('navigation', { name: /main/i });
screen.getByRole('dialog', { name: /confirm deletion/i });
screen.getByRole('alert'); // ARIA live region
screen.getByRole('tab', { selected: true });
screen.getByRole('link', { name: /learn more/i });

// GOOD: getByLabelText — form fields
screen.getByLabelText('Password');
screen.getByLabelText(/phone number/i);

// OK: getByText — non-interactive content
screen.getByText('No results found');
screen.getByText(/total: \$\d+/i);

// LAST RESORT: getByTestId — when no semantic query works
// Add data-testid only when necessary
screen.getByTestId('complex-chart-container');

// If you need getByTestId, consider: can you add an ARIA role
// or label to make the element accessible AND queryable?
Architect Insight

Using accessibility-first queries creates a virtuous cycle: if your test can't find an element by role or label, it means the element isn't accessible. Fixing the test means fixing the accessibility. This is why RTL's query priority is so powerful — it makes accessibility testing a side effect of writing good tests.

Testing Hooks with renderHook

import { renderHook, act } from '@testing-library/react';

// Testing a simple custom hook
function useCounter(initial = 0) {
  const [count, setCount] = useState(initial);
  const increment = () => setCount(c => c + 1);
  const decrement = () => setCount(c => c - 1);
  const reset = () => setCount(initial);
  return { count, increment, decrement, reset };
}

test('useCounter increments and decrements', () => {
  const { result } = renderHook(() => useCounter(10));

  expect(result.current.count).toBe(10);

  act(() => result.current.increment());
  expect(result.current.count).toBe(11);

  act(() => result.current.decrement());
  expect(result.current.count).toBe(10);

  act(() => result.current.reset());
  expect(result.current.count).toBe(10);
});

// Testing hook with dependencies
test('useCounter respects initial value changes', () => {
  const { result, rerender } = renderHook(
    ({ initial }) => useCounter(initial),
    { initialProps: { initial: 0 } }
  );

  expect(result.current.count).toBe(0);

  // Rerender with new props
  rerender({ initial: 100 });
  // Note: count doesn't change (useState initial is only used on mount)
  expect(result.current.count).toBe(0);

  // But reset uses the new initial
  act(() => result.current.reset());
  expect(result.current.count).toBe(100);
});

// Testing async hooks
function useUser(userId) {
  const [user, setUser] = useState(null);
  const [loading, setLoading] = useState(true);

  useEffect(() => {
    fetch(`/api/users/${userId}`)
      .then(r => r.json())
      .then(data => {
        setUser(data);
        setLoading(false);
      });
  }, [userId]);

  return { user, loading };
}

test('useUser fetches user data', async () => {
  server.use(
    http.get('/api/users/1', () => {
      return HttpResponse.json({ name: 'Alice' });
    })
  );

  const { result } = renderHook(() => useUser('1'));

  expect(result.current.loading).toBe(true);

  await waitFor(() => {
    expect(result.current.loading).toBe(false);
  });

  expect(result.current.user).toEqual({ name: 'Alice' });
});

Testing Context Providers

// Custom render with providers
import { render } from '@testing-library/react';
import { ThemeProvider } from './ThemeContext';
import { AuthProvider } from './AuthContext';
import { QueryClient, QueryClientProvider } from '@tanstack/react-query';

function createTestQueryClient() {
  return new QueryClient({
    defaultOptions: {
      queries: { retry: false }, // Don't retry in tests
    },
  });
}

function AllProviders({ children }) {
  const queryClient = createTestQueryClient();
  return (
    <QueryClientProvider client={queryClient}>
      <AuthProvider initialUser={{ id: '1', name: 'Test User' }}>
        <ThemeProvider initialTheme="light">
          {children}
        </ThemeProvider>
      </AuthProvider>
    </QueryClientProvider>
  );
}

function renderWithProviders(ui, options = {}) {
  return render(ui, { wrapper: AllProviders, ...options });
}

// Usage in tests
test('displays user name from context', () => {
  renderWithProviders(<UserGreeting />);
  expect(screen.getByText('Hello, Test User')).toBeInTheDocument();
});

// Override specific provider for a test
test('shows login prompt when not authenticated', () => {
  function NoAuthProviders({ children }) {
    const queryClient = createTestQueryClient();
    return (
      <QueryClientProvider client={queryClient}>
        <AuthProvider initialUser={null}>
          <ThemeProvider initialTheme="light">
            {children}
          </ThemeProvider>
        </AuthProvider>
      </QueryClientProvider>
    );
  }

  render(<UserGreeting />, { wrapper: NoAuthProviders });
  expect(screen.getByText('Please log in')).toBeInTheDocument();
});

Snapshot Testing

When Snapshots Are Useful

// GOOD: Small, focused snapshots for serializable output
test('formatCurrency formats correctly', () => {
  expect(formatCurrency(1234.5, 'USD')).toMatchInlineSnapshot(
    `"$1,234.50"`
  );
});

// GOOD: Inline snapshots for small component output
test('renders badge with correct aria attributes', () => {
  const { container } = render(<Badge status="warning" />);
  expect(container.firstChild).toMatchInlineSnapshot(`
    <span
      class="badge badge-warning"
      role="status"
      aria-label="Warning"
    >
      Warning
    </span>
  `);
});

When Snapshots Are Harmful

// BAD: Large component tree snapshot
test('renders page', () => {
  const { container } = render(<ProductPage />);
  expect(container).toMatchSnapshot();
  // Creates a 500-line snapshot file that:
  // - Nobody reviews during code review
  // - Breaks on ANY change (even irrelevant ones)
  // - Gets blindly updated with `--updateSnapshot`
  // - Provides false confidence
});

// BAD: Snapshot of dynamic content
test('renders user list', () => {
  render(<UserList users={mockUsers} />);
  expect(screen.getByRole('list')).toMatchSnapshot();
  // Breaks whenever mock data changes
  // Doesn't test actual behavior
});
Snapshot Anti-Pattern

If your team routinely runs jest --updateSnapshot without reviewing changes, snapshots are providing zero value and adding maintenance cost. Delete them. Replace with explicit assertions that test specific behavior. The only good snapshot is one small enough to review inline during code review.

Testing Portals and Modals

// Portals render outside the parent DOM hierarchy
// but RTL's screen queries the entire document, so they work naturally

function Modal({ isOpen, onClose, children }) {
  if (!isOpen) return null;
  return createPortal(
    <div role="dialog" aria-modal="true">
      <button onClick={onClose} aria-label="Close">×</button>
      {children}
    </div>,
    document.body
  );
}

test('modal opens and closes', async () => {
  render(
    <div>
      <ModalTrigger />
    </div>
  );

  // Modal is not visible initially
  expect(screen.queryByRole('dialog')).not.toBeInTheDocument();

  // Open modal
  await userEvent.click(screen.getByRole('button', { name: /open/i }));

  // Modal is visible (even though it's a portal)
  expect(screen.getByRole('dialog')).toBeInTheDocument();
  expect(screen.getByText('Modal content')).toBeInTheDocument();

  // Close modal
  await userEvent.click(screen.getByRole('button', { name: /close/i }));
  expect(screen.queryByRole('dialog')).not.toBeInTheDocument();
});

// Testing focus trap in modal
test('traps focus within modal', async () => {
  render(<ModalWithFocusTrap />);
  await userEvent.click(screen.getByRole('button', { name: /open/i }));

  // Focus should be inside the modal
  expect(screen.getByRole('dialog')).toContainElement(
    document.activeElement
  );

  // Tab should cycle within modal
  await userEvent.tab();
  expect(screen.getByRole('dialog')).toContainElement(
    document.activeElement
  );
});
