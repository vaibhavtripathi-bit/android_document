Testing Trophy vs Testing Pyramid

The traditional testing pyramid (many unit tests, fewer integration tests, fewest E2E tests) was designed for backend systems with clear module boundaries. For React applications, Kent C. Dodds proposed the "testing trophy" — emphasizing integration tests as the highest-ROI testing layer.

Testing Pyramid (Traditional): Testing Trophy (React): /\ ___ /E2E\ /E2E\ /------\ /------\ /Integra-\ / Integra-\ / tion \ / tion \ /------------\ / (LARGEST) \ / Unit Tests \ /----------------\ / (LARGEST) \ / Unit Tests \ /------------------\ /--------------------\ / Static Analysis \ /------------------------\

Why the Trophy Shape for React

React components are inherently integrative — they compose other components, manage state, handle events, and interact with APIs. Testing a component in complete isolation (pure unit test) often requires so much mocking that the test provides little confidence. Integration tests that render a component with its real children, real state management, and mocked network layer provide the best confidence-to-effort ratio.

LayerPyramid PriorityTrophy PriorityReact Rationale
Static Analysis Not included Foundation TypeScript + ESLint catch entire categories of bugs with zero runtime cost
Unit Tests Largest layer Small layer Pure utility functions and hooks — but most React code is integrative
Integration Tests Medium layer Largest layer Render components with real children, mock only the network boundary
E2E Tests Smallest layer Small layer Critical user flows — login, checkout, data creation
Architect Insight

At L6, you define the testing strategy for your organization. The testing trophy is a guideline, not a rule. The right distribution depends on your application's architecture. A component library benefits from more unit tests (isolated components). A form-heavy app benefits from more integration tests. A multi-step workflow benefits from more E2E tests. The key is maximizing confidence per test dollar spent.

The Refactoring Litmus Test

A good test should not break when you refactor implementation without changing behavior. This is the single most important quality of a test — it determines long-term maintenance cost.

// BAD: Test breaks on refactor (tests implementation)
test('increments counter state', () => {
  const { result } = renderHook(() => useCounter());
  act(() => result.current.increment());
  expect(result.current.count).toBe(1);
});

// If you refactor useCounter to use useReducer instead of useState,
// the test might break even though behavior is identical.

// GOOD: Test survives refactor (tests behavior)
test('clicking increment shows updated count', async () => {
  render(<Counter />);
  expect(screen.getByText('Count: 0')).toBeInTheDocument();

  await userEvent.click(screen.getByRole('button', { name: /increment/i }));
  expect(screen.getByText('Count: 1')).toBeInTheDocument();
});

// This test passes regardless of whether Counter uses
// useState, useReducer, Zustand, or Redux internally.

What to Test in React Apps

High-Value Test Targets

// 1. User interactions and their effects
test('submitting form creates a new todo', async () => {
  render(<TodoApp />);
  await userEvent.type(
    screen.getByPlaceholderText('Add todo'),
    'Buy groceries'
  );
  await userEvent.click(screen.getByRole('button', { name: /add/i }));
  expect(screen.getByText('Buy groceries')).toBeInTheDocument();
});

// 2. Conditional rendering based on state/props
test('shows error message when form validation fails', async () => {
  render(<LoginForm />);
  await userEvent.click(screen.getByRole('button', { name: /login/i }));
  expect(screen.getByText(/email is required/i)).toBeInTheDocument();
});

// 3. Data fetching and display
test('displays user profile after loading', async () => {
  server.use(
    http.get('/api/user/1', () => {
      return HttpResponse.json({ name: 'Alice', role: 'Engineer' });
    })
  );
  render(<UserProfile userId="1" />);
  expect(await screen.findByText('Alice')).toBeInTheDocument();
  expect(screen.getByText('Engineer')).toBeInTheDocument();
});

// 4. Error states and edge cases
test('shows error boundary when API fails', async () => {
  server.use(
    http.get('/api/data', () => {
      return new HttpResponse(null, { status: 500 });
    })
  );
  render(<DataView />);
  expect(await screen.findByText(/something went wrong/i)).toBeInTheDocument();
});

Low-Value Test Targets (Usually Skip)

// DON'T TEST: Implementation details
test('sets loading state to true', () => { /* ... */ }); // Internal state

// DON'T TEST: Third-party library behavior
test('react-router navigates correctly', () => { /* ... */ }); // Not your code

// DON'T TEST: Styling/CSS (use visual regression instead)
test('button has blue background', () => { /* ... */ }); // Brittle

// DON'T TEST: Exact snapshot of large component trees
test('matches snapshot', () => {
  const { container } = render(<ComplexPage />);
  expect(container).toMatchSnapshot(); // 500-line snapshot nobody reviews
});

// DON'T TEST: Constants and configuration
test('API_URL is correct', () => { /* ... */ }); // Static analysis catches this

Testing Behavior vs Implementation

The most impactful testing principle: test what the component does (behavior), not how it does it (implementation). This single principle determines whether your test suite is an asset or a liability.

Implementation Testing (Fragile): ┌──────────────────────────────────────────┐ │ Test knows about: │ │ ├─ Internal state shape │ │ ├─ Hook names and call order │ │ ├─ CSS class names │ │ ├─ Component hierarchy │ │ └─ Private methods │ │ │ │ Breaks when: Refactoring internals │ │ even if behavior is unchanged │ └──────────────────────────────────────────┘ Behavior Testing (Resilient): ┌──────────────────────────────────────────┐ │ Test knows about: │ │ ├─ What the user sees (text, roles) │ │ ├─ What the user does (click, type) │ │ ├─ What happens after (new text, nav) │ │ └─ What the component renders │ │ │ │ Breaks when: Actual behavior changes │ │ (which is when you WANT tests to break) │ └──────────────────────────────────────────┘ 
// IMPLEMENTATION TEST (fragile)
test('toggles isOpen state on click', () => {
  const wrapper = shallow(<Dropdown />);
  expect(wrapper.state('isOpen')).toBe(false);
  wrapper.find('.dropdown-trigger').simulate('click');
  expect(wrapper.state('isOpen')).toBe(true);
});
// Problems:
// - Tests internal state name
// - Uses shallow rendering (misses child behavior)
// - Uses CSS class selector (breaks on styling changes)
// - Uses simulate (doesn't fire real events)

// BEHAVIOR TEST (resilient)
test('shows dropdown options when clicked', async () => {
  render(<Dropdown options={['Red', 'Blue', 'Green']} />);

  // Options are not visible initially
  expect(screen.queryByText('Red')).not.toBeInTheDocument();

  // Click the dropdown trigger
  await userEvent.click(screen.getByRole('combobox'));

  // Options are now visible
  expect(screen.getByText('Red')).toBeInTheDocument();
  expect(screen.getByText('Blue')).toBeInTheDocument();
  expect(screen.getByText('Green')).toBeInTheDocument();
});
// Survives: renaming state, changing CSS, refactoring to useReducer,
// switching from div to details/summary, etc.

Architect Insight

The optimal testing strategy maximizes confidence within your feedback budget. If your CI pipeline must complete in 10 minutes, you can run ~1000 integration tests or ~50 E2E tests. The 1000 integration tests likely cover more scenarios and provide more confidence per minute of CI time. Reserve E2E tests for critical paths where integration tests can't provide sufficient confidence (multi-page flows, real browser behavior, visual regression).

Coverage Metrics and Limitations

Coverage Types

// Statement coverage: Was this line executed?
function greet(name) {
  if (name) {           // Branch 1
    return `Hello ${name}`;  // Statement A
  }
  return 'Hello stranger';   // Statement B
}

// Test: greet('Alice') → 66% statement coverage (misses Statement B)
// Test: greet('Alice') + greet() → 100% statement coverage

// Branch coverage: Was every branch taken?
// Test: greet('Alice') → 50% branch coverage (only truthy branch)

// Function coverage: Was every function called?
// Test: greet('Alice') → 100% function coverage (function was called)

// Line coverage: Was every line executed?
// Similar to statement but counts lines, not statements

Why 100% Coverage Is Misleading

// 100% coverage but ZERO confidence:
function calculateTotal(items) {
  return items.reduce((sum, item) => sum + item.price * item.quantity, 0);
}

test('calculateTotal is a function', () => {
  // 100% coverage — the function runs
  // But tests NOTHING about correctness
  expect(calculateTotal([{ price: 10, quantity: 2 }])).toBeDefined();
});

// 80% coverage but HIGH confidence:
test('calculates total for multiple items', () => {
  const items = [
    { price: 10, quantity: 2 },
    { price: 5, quantity: 3 },
  ];
  expect(calculateTotal(items)).toBe(35);
});

test('returns 0 for empty cart', () => {
  expect(calculateTotal([])).toBe(0);
});
Coverage Anti-Patterns

Mandating specific coverage percentages (e.g., "80% coverage required") incentivizes writing low-value tests to hit the number. Teams write tests that execute code without asserting behavior. A better metric: track coverage trends (is coverage increasing?) and focus on critical path coverage (are the most important user flows tested?). At L6, advocate for meaningful coverage over numerical targets.

Testing Strategy for Large Codebases

// Testing strategy by component type:

// 1. Shared UI components (Button, Modal, Input)
//    → Unit tests + visual regression (Storybook + Chromatic)
//    → Test accessibility, variants, edge cases
//    → High coverage — used everywhere

// 2. Feature components (ProductCard, CheckoutForm)
//    → Integration tests with React Testing Library
//    → Mock network (MSW), render with real children
//    → Test user flows, error states, loading states

// 3. Page components (ProductPage, DashboardPage)
//    → Integration tests for data flow
//    → E2E tests for critical paths
//    → Test routing, data fetching, composition

// 4. Utility functions (formatCurrency, validateEmail)
//    → Unit tests with edge cases
//    → High coverage — pure functions are easy to test

// 5. Custom hooks (useAuth, useCart)
//    → renderHook for simple hooks
//    → Integration test via component for complex hooks

// 6. State management (Redux slices, Zustand stores)
//    → Unit tests for reducers/actions
//    → Integration tests for component interaction with store

The Cost of Testing

Tests are code that must be maintained. Every test has a cost — writing time, CI time, maintenance time, and cognitive load. Staff engineers must evaluate the ROI of each test.

CostDescriptionMitigation
Writing time Time to write the initial test Test generators, patterns, shared utilities
Maintenance time Time to update tests when code changes Test behavior not implementation, avoid snapshots
CI time Time tests add to the CI pipeline Parallel execution, test splitting, caching
False positives Tests that fail without real bugs (flaky) Deterministic tests, retry logic, quarantine flaky tests
False negatives Tests that pass despite real bugs Better assertions, behavior testing, mutation testing
Cognitive load Mental overhead of understanding test suite Clear naming, consistent patterns, documentation
Architect Insight

The most expensive test is one that exists but provides no confidence. Regularly audit your test suite: delete tests that test implementation details, consolidate redundant tests, and identify gaps in critical path coverage. A lean test suite with high-confidence tests is better than a bloated suite with low-confidence tests. At Google's scale, test maintenance is a significant engineering cost — the testing strategy must be sustainable.
