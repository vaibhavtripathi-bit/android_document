Interview Q&A — Testing Patterns

55+ Staff-level interview questions covering testing philosophy, component testing, integration testing, E2E, mocking, performance testing, and infrastructure.

Testing Philosophy

Q1: Explain the testing trophy and how it differs from the testing pyramid. L6

The testing pyramid (many unit tests, fewer integration, fewest E2E) was designed for backend systems with clear module boundaries. The testing trophy, proposed by Kent C. Dodds for React apps, emphasizes integration tests as the largest layer. The trophy shape: static analysis (foundation), unit tests (small), integration tests (largest), E2E tests (small). The rationale: React components are inherently integrative — they compose other components, manage state, and interact with APIs. Testing them in isolation requires so much mocking that tests provide little confidence. Integration tests with React Testing Library and MSW provide the best confidence-to-effort ratio for React applications.

Q2: What does "test behavior, not implementation" mean in practice? L6

Testing behavior means testing what the user sees and does — rendered output, interactions, and their effects. Testing implementation means testing internal state, hook calls, component hierarchy, or CSS classes. In practice: instead of checking component.state.isOpen === true, check that the dropdown options are visible after clicking. Instead of verifying useState was called, verify the counter displays the updated number. The key benefit: behavior tests survive refactoring. If you switch from useState to useReducer, or from a class component to a function component, behavior tests still pass because the user-facing behavior hasn't changed. Implementation tests break on every refactor, creating maintenance burden without catching real bugs.

Q3: How would you define a testing strategy for a large React codebase? L6

Layer the strategy by component type: (1) Shared UI components (Button, Modal): unit tests + visual regression via Storybook + Chromatic. High coverage since they're used everywhere. (2) Feature components (ProductCard, CheckoutForm): integration tests with RTL + MSW. Test user flows, error states, loading states. (3) Page components: integration tests for data flow + E2E for critical paths. (4) Utility functions: unit tests with edge cases. (5) Custom hooks: test via components (preferred) or renderHook. (6) State management: unit tests for reducers, integration tests for component interaction. Set coverage thresholds per directory (stricter for utils, looser for pages). Invest in testing infrastructure: fast CI, parallel execution, pre-commit hooks for related tests. Review test quality quarterly — delete low-value tests, add tests for recent bugs.

Q4: What are the limitations of code coverage as a metric? L6

Coverage measures execution, not correctness. A test that calls a function without meaningful assertions achieves 100% coverage with zero confidence. Specific limitations: (1) Coverage doesn't measure assertion quality — expect(result).toBeDefined() covers the code but tests nothing. (2) It misses edge cases — 100% branch coverage doesn't mean all input combinations are tested. (3) It incentivizes low-value tests when mandated as a gate. (4) It doesn't account for test isolation — a test might cover code as a side effect, not intentionally. Better metrics: track coverage trends (is it increasing?), focus on critical path coverage, use mutation testing (does changing code break tests?), and measure test quality through bug escape rate (bugs found in production that tests should have caught).

Q5: What is the cost of testing and how do you manage it? L6

Test costs include: writing time, maintenance time (updating tests when code changes), CI time (pipeline duration), false positives (flaky tests), false negatives (tests that pass despite bugs), and cognitive load (understanding the test suite). Management strategies: (1) Test behavior not implementation to reduce maintenance. (2) Use integration tests over unit tests for better ROI. (3) Delete low-value tests (large snapshots, implementation-coupled tests). (4) Invest in infrastructure (parallel CI, test splitting) to reduce CI time. (5) Quarantine and fix flaky tests immediately. (6) Use consistent patterns and shared utilities to reduce writing time. The goal is maximizing confidence per test dollar — a lean suite with high-confidence tests beats a bloated suite with low-confidence tests.

Component Testing

Q6: Explain the React Testing Library query priority and why it matters. L6

RTL's query priority: (1) getByRole — accessible to everyone, (2) getByLabelText — form elements, (3) getByPlaceholderText — when no label, (4) getByText — non-interactive content, (5) getByDisplayValue — filled inputs, (6) getByAltText — images, (7) getByTitle — title attribute, (8) getByTestId — last resort. This priority matters because it mirrors how users (including screen reader users) find elements. Using getByRole forces you to add proper ARIA roles and labels, improving accessibility as a side effect of writing tests. If you can't find an element by role, it's a signal that the element isn't accessible. getByTestId should be reserved for complex elements with no semantic query (e.g., a canvas-based chart).

Q7: What's the difference between getBy, queryBy, and findBy? L6

getBy: Synchronous, throws if element not found. Use when the element should exist. queryBy: Synchronous, returns null if not found. Use for asserting absence (expect(queryByText('Error')).not.toBeInTheDocument()). findBy: Asynchronous (returns Promise), throws on timeout. Use for elements that appear after async operations (data fetching, animations). Pattern: use queryBy to assert absence, trigger an action, then use findBy to wait for the element to appear. All three have All variants (getAllBy, queryAllBy, findAllBy) for multiple elements.

Q8: Why should you prefer userEvent over fireEvent? L6

userEvent simulates real user interactions by firing the complete sequence of events. Typing fires keydown, keypress, input, change, and keyup for each character. Clicking fires pointerdown, mousedown, pointerup, mouseup, click. fireEvent dispatches a single synthetic event. userEvent catches bugs that fireEvent misses — for example, a component that listens for keydown but not change, or one that checks event.isTrusted. Always use userEvent.setup() for the latest API. The only exception: use fireEvent for events userEvent doesn't support (custom events, scroll, resize).

Q9: When are snapshot tests useful vs harmful? L6

Useful: small, focused inline snapshots for serializable output (formatted strings, small component markup, API response shapes). Harmful: large component tree snapshots that create 500-line snapshot files nobody reviews, break on any change (even irrelevant ones), and get blindly updated with --updateSnapshot. The litmus test: if your team routinely runs updateSnapshot without reviewing changes, the snapshots provide zero value. Replace large snapshots with explicit assertions that test specific behavior. The only good snapshot is one small enough to review inline during code review.

Q10: How do you test components that use portals? L6

RTL's screen queries the entire document.body, so portal content is automatically queryable — no special setup needed. Portals render outside the parent DOM hierarchy but are still part of the document. Test modals, tooltips, and dropdowns the same way as any other component: trigger the open action, query for the portal content, interact with it, and verify the close behavior. For focus trap testing, verify that document.activeElement is contained within the portal element after opening and that Tab cycles within the portal.

Integration Testing

Q11: Why is MSW preferred over jest.mock for API mocking? L6

MSW intercepts at the network level (service worker in browser, node interceptor in tests), so your application's actual fetch/axios/react-query code runs exactly as in production. jest.mock replaces modules, meaning you're not testing the data fetching code itself — you're bypassing it. MSW advantages: (1) Tests the full request pipeline (headers, serialization, error handling). (2) Same handlers work in tests, Storybook, and development. (3) Request inspection (verify headers, body, query params). (4) Realistic error simulation (network errors, timeouts, status codes). (5) No coupling to implementation (works regardless of fetch library used).

Q12: How do you test routing in React applications? L6

Use MemoryRouter from react-router-dom with initialEntries to set the starting URL. Render the full route configuration (Routes + Route components) to test navigation between pages. Test: (1) correct component renders for each URL, (2) navigation via links and programmatic navigation, (3) route parameters are passed correctly, (4) protected routes redirect unauthenticated users, (5) 404 handling for unknown routes. For Next.js, mock next/navigation hooks (useRouter, usePathname, useSearchParams) or use the actual App Router test utilities.

Q13: How do you test error boundaries? L6

Render a component that throws inside an error boundary. Suppress console.error with a spy (React logs errors during error boundary handling). Verify the fallback UI is displayed. For recovery testing: trigger the error, verify fallback, fix the error condition, click retry, and verify the component recovers. Important: use jest.spyOn(console, 'error').mockImplementation() to prevent test output noise, and restore it after the test. Also test that errors in different parts of the tree are caught by the correct boundary (nested error boundaries).

Q14: How do you organize integration tests for a large feature? L6

Organize by user flow rather than by component. A test file named checkout-flow.test.tsx that tests the entire checkout process is more valuable than separate tests for CartButton, CartDrawer, CheckoutForm, and PaymentInput. Structure: (1) One describe block per major flow (happy path, error handling, edge cases). (2) Use beforeEach for common setup (seed data, render app). (3) Use MSW handler overrides for specific scenarios within tests. (4) Keep tests independent — each test should work in isolation. (5) Use a custom render function that wraps with all necessary providers (router, auth, query client, theme).

E2E Testing

Q15: When would you choose Playwright over Cypress? L6

Choose Playwright when: (1) you need multi-browser testing (Chromium, Firefox, WebKit), (2) you need multi-tab or multi-window testing, (3) you need built-in parallel execution without paid services, (4) you need iframe support, (5) you want language flexibility (Python, Java, C#), (6) you need mobile device emulation beyond viewport. Choose Cypress when: (1) your team values the time-travel debugger for debugging, (2) you want built-in component testing, (3) you prefer the in-browser test runner for development, (4) your team is already invested in the Cypress ecosystem. For new projects, Playwright is generally recommended due to its broader capabilities and free parallelization.

Q16: How do you prevent flaky E2E tests? L6

Root causes and fixes: (1) Race conditions: use auto-waiting (Playwright's built-in actionability checks) instead of hardcoded waits. (2) Test isolation: each test seeds its own data via API, never depend on execution order. (3) Animation interference: disable animations in test config (reducedMotion: 'reduce'). (4) Network timing: mock external APIs with page.route() for deterministic responses. (5) Shared state: reset database/state before each test. (6) Dynamic content: use stable selectors (roles, labels) not CSS classes or dynamic IDs. Infrastructure: quarantine flaky tests immediately, track flakiness rate, set up automatic retry with reporting, and investigate root causes rather than adding retries.

Q17: How do you test SSR and hydration in E2E tests? L6

Three approaches: (1) Disable JavaScript and verify content is visible — proves SSR works. Create a browser context with javaScriptEnabled: false, navigate to the page, and assert content is present. (2) Test hydration by verifying interactive elements work after page load — click buttons, submit forms, verify state changes. (3) Monitor console for hydration errors — listen for console.error messages containing "hydration" and assert none occur. Additionally, test that the page is functional during the hydration gap — content should be visible and links should work even before JS loads (progressive enhancement).

Q18: How do you implement visual regression testing? L6

Playwright's built-in screenshot comparison: expect(page).toHaveScreenshot() compares against a baseline image. Configuration: set maxDiffPixelRatio for tolerance, disable animations, use consistent viewport sizes. For component-level testing: use Storybook + Chromatic (cloud service) or Storybook + Playwright. Best practices: (1) Test at multiple viewport sizes (mobile, tablet, desktop). (2) Test interactive states (hover, focus, active, disabled). (3) Test with different data (empty state, long text, many items). (4) Use consistent fonts (install fonts in CI). (5) Review visual diffs in PRs before approving. Limitations: visual regression tests are slow and generate large baseline files — use selectively for critical UI components.

Mocking Patterns

Q19: When should you mock and when should you use real implementations? L6

Mock at system boundaries: network requests (MSW), browser APIs not in jsdom (IntersectionObserver, ResizeObserver), timers/dates (jest.useFakeTimers), third-party services (analytics, auth providers), and external services (Stripe, payment processors). Use real implementations for: your own components, utility functions, hooks (test via components), state management (Redux, Zustand), React Router (use MemoryRouter), and React Context (provide real context). The principle: every mock is a point where your test diverges from reality. Mock only what you must to make the test deterministic and fast. If you're mocking 5+ things for one test, the component likely has too many dependencies.

Q20: How do you mock timers in React tests? L6

Use jest.useFakeTimers() to replace setTimeout, setInterval, and Date.now. For userEvent compatibility, pass advanceTimers: jest.advanceTimersByTime to userEvent.setup(). Advance time with jest.advanceTimersByTime(ms) wrapped in act() to flush React updates. Test debounced inputs by typing, asserting the callback hasn't fired, advancing time past the debounce delay, then asserting it fired. Test intervals by advancing time in steps and counting calls. Always call jest.useRealTimers() in cleanup. Test timer cleanup by unmounting the component, advancing time, and verifying callbacks don't fire.

Q21: How do you mock IntersectionObserver for lazy loading tests? L6

Create a mock IntersectionObserver class that captures the callback and provides a way to trigger intersection. Assign it to window.IntersectionObserver. In the test, render the lazy component, get the callback from the mock's constructor calls, then call it with [{ isIntersecting: true, target: element }] to simulate the element becoming visible. Verify the lazy-loaded content appears. This pattern works for any observer-based API (ResizeObserver, MutationObserver). The key is capturing the callback and providing a programmatic way to trigger it.

Performance Testing

Q22: How do you use the React Profiler API for automated performance testing? L6

Wrap components in <Profiler id="name" onRender={callback}>. The callback receives: id, phase (mount/update), actualDuration (time spent rendering), baseDuration (estimated full re-render time), startTime, and commitTime. In tests: capture metrics in an array, render the component, assert actualDuration is within budget (e.g., <50ms for initial render). For memoization testing: trigger a parent state change and verify the child's onRender callback doesn't fire (no re-render). For update performance: trigger a state change and assert the update's actualDuration is within budget (e.g., <16ms for one frame).

Q23: How do you implement performance budgets in CI? L6

Multi-layer approach: (1) Build-time: use size-limit to enforce bundle size limits per chunk (e.g., main JS <150KB gzipped). Fails CI if exceeded. (2) Synthetic testing: Lighthouse CI with assertions on performance score (>90), LCP (<2.5s), CLS (<0.1), TBT (<300ms). Run against built app in CI. (3) Runtime: React Profiler tests asserting render times. (4) Regression: compare current metrics against baseline from previous release, allowing 10% tolerance. Store baselines in git, update intentionally after approved changes. The key is failing CI early (build-time checks are fastest) and providing actionable feedback (which chunk grew, which metric regressed).

Q24: How do you detect memory leaks in React tests? L6

Three approaches: (1) Event listener leaks: spy on addEventListener/removeEventListener, render and unmount the component, verify every added listener was removed. (2) Subscription leaks: verify that store.subscribe returns an unsubscribe function that's called on unmount. (3) State update after unmount: render a component with async operations, unmount it before the operation completes, wait for the operation to finish, and verify no "state update on unmounted component" warnings appear. For production detection: use Chrome DevTools heap snapshots in E2E tests — take a snapshot, perform actions, take another snapshot, and compare retained objects.

Testing Infrastructure

Q25: Compare Jest and Vitest for React testing. L6

Jest: mature, battle-tested, extensive ecosystem, requires separate transform config (Babel/SWC), moderate speed, experimental ESM support. Vitest: Vite-native (shares Vite config), significantly faster (especially watch mode with HMR-like re-runs), native ESM support, Jest-compatible API (drop-in replacement), built-in TypeScript and CSS handling. Choose Vitest for: new projects using Vite, projects wanting faster test execution, projects that value shared build/test config. Choose Jest for: existing projects with complex Jest config, projects not using Vite, teams familiar with Jest ecosystem. Migration path: Vitest's Jest-compatible API makes migration straightforward — often just config changes.

Q26: How do you optimize test execution in CI for a large codebase? L6

Strategies: (1) Parallel sharding: split tests across multiple CI jobs (jest --shard=1/4). (2) Timing-based splitting: use previous run times to distribute tests evenly across shards (avoids one shard being much slower). (3) Affected tests only: run only tests related to changed files (jest --changedSince=main) for PR checks. (4) Cache transforms: cache SWC/Babel output between runs. (5) Use faster transforms: SWC over Babel (4x faster). (6) Separate unit/integration from E2E: run fast tests first, E2E in parallel. (7) Pre-commit hooks: run related tests locally before push. (8) Monorepo optimization: only test affected packages (Turborepo/Nx affected). Target: full test suite completes in <10 minutes in CI.

Q27: How do you set up testing in a monorepo? L6

Configuration: root jest.config.ts with projects array pointing to each package's config. Each package has its own jest.config.ts with displayName for identification. Execution: use jest --selectProjects ui to run specific packages. For affected-only testing: Turborepo (turbo run test --filter=...[HEAD^1]) or Nx (nx affected:test) detect which packages changed and only test those plus their dependents. Coverage: merge coverage from all packages using istanbul-merge. CI: run each package's tests as a separate CI job for parallelism, merge results in a final job. Shared test utilities: create a packages/test-utils package with custom render functions, MSW handlers, and test factories.

Q28: What's the difference between jsdom and happy-dom? L6

jsdom: the most complete DOM implementation for Node.js. Battle-tested over 10+ years, extensive Web API support (fetch, URL, FormData), but moderate speed and higher memory usage. happy-dom: a newer, faster alternative (2-3x speed improvement) with lower memory usage, but less complete API coverage. Some edge cases may behave differently from real browsers. Recommendation: use jsdom as the default for maximum compatibility. Switch to happy-dom if test speed is a bottleneck and your tests don't rely on jsdom-specific APIs. You can set the environment per-file with doc comments for tests that need specific environments.

Advanced Topics

Q29: How would you test a complex form with multi-step validation? L6

Integration test approach: (1) Render the full form component with all providers. (2) Test the happy path: fill each step, advance, verify final submission. (3) Test validation per step: submit with empty fields, verify error messages, fix errors, verify they clear. (4) Test step navigation: advance and go back, verify state is preserved. (5) Test async validation: mock server-side validation (email uniqueness) with MSW, test both valid and invalid responses. (6) Test form submission: capture the request body with MSW, verify correct data is sent. (7) Test error handling: mock API failure, verify error UI, test retry. Use userEvent for all interactions to simulate real user behavior.

Q30: How do you test React Server Components? L6

RSC testing is still evolving. Current approaches: (1) Test server components as async functions — call the component function directly and assert the returned JSX structure. (2) Use Next.js testing utilities that render the full RSC pipeline. (3) Test the data fetching logic separately from the rendering. (4) E2E tests for the full RSC flow (server render + client hydration). (5) Test client components independently with RTL (they're standard React components). (6) Test Server Actions as async functions — call them with mock FormData and assert side effects (database changes, revalidation). The RSC testing story is improving with each framework release — check the latest Next.js and React documentation for current best practices.

Q31: How do you handle test data management for E2E tests? L6

Strategies: (1) API-based seeding: each test calls a seed endpoint before running (/api/test/seed) with the specific scenario data needed. (2) Factory pattern: use faker.js-based factories to generate realistic test data with overrides for specific scenarios. (3) Database snapshots: restore a known database state before each test suite. (4) Isolated test databases: each parallel worker gets its own database instance. (5) Cleanup: use afterEach to clean up created data, or use transactional tests that rollback after each test. The key principle: each test must be independent — never depend on data created by another test or on execution order.

Q32: What is mutation testing and how does it improve test quality? L6

Mutation testing modifies your source code (introduces "mutants" — small changes like flipping conditions, changing operators, removing statements) and runs your tests. If tests still pass after a mutation, the mutant "survived" — meaning your tests don't catch that type of bug. A high mutation score (% of mutants killed) indicates strong test assertions. Tools: Stryker Mutator for JavaScript/TypeScript. Mutation testing is expensive (runs tests many times) but reveals weak assertions that coverage metrics miss. Use it selectively on critical code paths rather than the entire codebase. It's particularly valuable for utility functions and business logic where correctness is critical.
