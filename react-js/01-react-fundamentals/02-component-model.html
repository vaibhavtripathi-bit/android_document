1. Internal Representation: Function vs Class

From React's perspective, both function and class components produce ReactElements. But internally, React treats them very differently during reconciliation and rendering.

1.1 How React Distinguishes Them

// Function component
function Greeting({ name }) {
  return <h1>Hello {name}</h1>;
}

// Class component
class Greeting extends React.Component {
  render() {
    return <h1>Hello {this.props.name}</h1>;
  }
}

// React checks for class components using this flag:
React.Component.prototype.isReactComponent = {};

// During rendering, React does:
if (Component.prototype && Component.prototype.isReactComponent) {
  // Class: new Component(props), then call .render()
  const instance = new Component(props);
  const element = instance.render();
} else {
  // Function: just call it
  const element = Component(props);
}
Why Not typeof or instanceof?

React doesn't use typeof Component === 'function' (both are functions) or instanceof React.Component (breaks across multiple React copies, iframes). The isReactComponent prototype flag is a simple, reliable duck-typing check that works across module boundaries.

1.2 Fiber Tag Assignment

Component Type Fiber Tag Value Rendering Strategy
Function component FunctionComponent 0 Call function, process hooks
Class component ClassComponent 1 Instantiate, call render(), manage lifecycle
Host element (div, span) HostComponent 5 Create/update DOM node
Host text ("hello") HostText 6 Create/update text node
Fragment Fragment 7 No DOM node, just children
Context Provider ContextProvider 10 Push context value
Suspense SuspenseComponent 13 Catch thrown promises
Lazy LazyComponent 16 Resolve promise, then render
Memo MemoComponent 14 Shallow compare props, skip if equal

2. ReactElement Type Structure

Every JSX expression produces a ReactElement — a plain, immutable JavaScript object that describes a piece of UI.

// The complete ReactElement shape (React 18+)
const element = {
  $$typeof: Symbol.for('react.element'),  // Security tag
  type: 'div',          // string | function | class | Symbol
  key: null,            // string | null — for reconciliation
  ref: null,            // ref object | callback | null
  props: {              // Frozen object of attributes + children
    className: 'box',
    children: [...]
  },
  _owner: fiber,        // Fiber that created this element (dev)
  _store: {},           // Dev-only validation metadata
};

2.1 The type Field

// type determines what React does during reconciliation:

// String → HostComponent (DOM node)
{ type: "div" }        // → creates/updates <div>
{ type: "span" }       // → creates/updates <span>

// Function → FunctionComponent
{ type: App }           // → calls App(props)

// Class → ClassComponent
{ type: Counter }       // → new Counter(props), counter.render()

// Symbol → special React types
{ type: Symbol.for('react.fragment') }   // Fragment
{ type: Symbol.for('react.profiler') }   // Profiler
{ type: Symbol.for('react.strict_mode') } // StrictMode
{ type: Symbol.for('react.suspense') }    // Suspense

// Object → wrapped components
{ type: { $$typeof: Symbol.for('react.memo'), compare, type } }
{ type: { $$typeof: Symbol.for('react.forward_ref'), render } }
{ type: { $$typeof: Symbol.for('react.lazy'), _init, _payload } }
Reconciliation Key Insight

React uses referential identity of the type field to decide whether to update or remount. If prevElement.type === nextElement.type, React reuses the fiber and updates props. If they differ, React destroys the old subtree and mounts a new one. This is why defining components inside render functions is catastrophic — it creates a new function reference every render, causing full remounts.

3. $$typeof — XSS Protection

The $$typeof field is React's defense against a specific class of XSS attacks involving JSON injection.

3.1 The Attack Vector

// Imagine a server returns user-controlled JSON:
const serverData = JSON.parse(response);
// serverData = {
//   type: "div",
//   props: {
//     dangerouslySetInnerHTML: {
//       __html: "<script>alert('xss')</script>"
//     }
//   }
// }

// If React rendered arbitrary objects as elements,
// an attacker could inject malicious elements via JSON.

// But $$typeof: Symbol.for('react.element') CANNOT
// appear in JSON — JSON.parse cannot produce Symbols.
// So React rejects any object without the Symbol tag.

3.2 The Protection Mechanism

JSON from server React.createElement() ───────────────── ───────────────────── { { type: "div", $$typeof: Symbol(react.element), ← CANNOT be in JSON props: {...} type: "div", } props: {...} } ↓ ↓ React: no $$typeof React: valid Symbol → REJECTED (not → ACCEPTED (rendered) a valid element) 
// React's validation check (simplified)
function isValidElement(object) {
  return (
    typeof object === 'object' &&
    object !== null &&
    object.$$typeof === REACT_ELEMENT_TYPE // Symbol check
  );
}

// For environments without Symbol support (IE11),
// React falls back to the magic number 0xeac7
const REACT_ELEMENT_TYPE = typeof Symbol === 'function'
  ? Symbol.for('react.element')
  : 0xeac7;
Security Limitation

The $$typeof check only protects against JSON injection. If an attacker can execute arbitrary JavaScript (e.g., through eval or a compromised dependency), they can create valid ReactElements with the correct Symbol. This is defense-in-depth, not a complete security boundary.

4. Children Normalization

React normalizes the children prop into a consistent internal representation regardless of how children are passed.

4.1 Children Forms

// Single child → stored directly (not wrapped in array)
<div><span /></div>
// props.children = ReactElement { type: "span", ... }

// Multiple children → stored as array
<div><span /><p /></div>
// props.children = [ReactElement(span), ReactElement(p)]

// Text child → stored as string
<div>Hello</div>
// props.children = "Hello"

// No children → children is undefined
<div />
// props.children = undefined

// Mixed children → array of mixed types
<div>Hello <span>World</span></div>
// props.children = ["Hello ", ReactElement(span)]

4.2 React.Children Utilities

// React.Children handles the inconsistent children shape
React.Children.map(children, fn)     // Flattens nested arrays, handles null
React.Children.forEach(children, fn)  // Like map but no return
React.Children.count(children)       // Counts including nested
React.Children.only(children)        // Asserts single child, returns it
React.Children.toArray(children)     // Normalizes to flat array with keys

// Why not just Array.isArray(children)?
// Because children might be a single element (not array),
// a string, a number, null, or a nested array.
// React.Children handles ALL these cases.

4.3 Key Assignment in Children.toArray

// React.Children.toArray assigns stable keys to children
// that don't have explicit keys:

const arr = React.Children.toArray(
  <>
    <A />
    <B key="b" />
    {[<C />, <D />]}
  </>
);
// Keys become: ".0", "b", ".2:0", ".2:1"
// The prefix encodes the nesting structure

5. Component Identity and Reconciliation

Component identity is determined by the type reference and key prop. This has profound implications for when React preserves or destroys state.

5.1 Same Position, Same Type = State Preserved

// State is preserved when type reference stays the same
function App({ isRed }) {
  return isRed
    ? <Counter color="red" />    // Same type: Counter
    : <Counter color="blue" />;  // Same position, same type
  // → Counter's state is PRESERVED across toggles
  // → Only the color prop changes
}

// State is DESTROYED when type changes
function App({ useAlt }) {
  return useAlt
    ? <CounterA />  // Different type reference
    : <CounterB />; // → full unmount/remount, state lost
}

5.2 The Key Prop as Identity Override

// Force state reset with key change
function App({ userId }) {
  // When userId changes, Profile unmounts and remounts
  // with fresh state — even though type is the same
  return <Profile key={userId} userId={userId} />;
}

// Without key: Profile updates with new userId prop
// but retains old state (stale data visible briefly)
// With key: clean remount, no stale state
Architect Insight — Component Identity Bugs

One of the most common Staff-level debugging scenarios: a component defined inside another component's render function gets a new type reference every render, causing full remounts and state loss. This is invisible in small apps but catastrophic at scale — forms lose input, animations restart, scroll positions reset.

6. React.lazy and Suspense Integration

6.1 React.lazy Internals

// React.lazy creates a special element type
const LazyComponent = React.lazy(() => import('./HeavyComponent'));

// Internally, this creates:
{
  $$typeof: Symbol.for('react.lazy'),
  _payload: {
    _status: -1,      // Uninitialized
    _result: () => import('./HeavyComponent')
  },
  _init: lazyInitializer
}

// Status transitions:
// -1 (Uninitialized) → 0 (Pending) → 1 (Resolved) | 2 (Rejected)

// When React encounters a lazy component during render:
// 1. Calls _init(_payload) which triggers the import()
// 2. If pending: THROWS the promise (caught by Suspense)
// 3. If resolved: returns the component, renders normally
// 4. If rejected: throws the error (caught by error boundary)

6.2 Suspense Boundary Interaction

<Suspense fallback={<Spinner />}> <LazyComponent /> </Suspense> Render attempt 1: LazyComponent._init() → import() starts → THROWS promise Suspense catches → renders <Spinner /> Promise resolves: React re-renders the Suspense boundary Render attempt 2: LazyComponent._init() → already resolved → returns Component Suspense renders <Component /> (replaces Spinner)

7. HOCs → Render Props → Hooks Evolution

7.1 Higher-Order Components (HOCs)

// HOC: function that takes a component and returns a new component
function withAuth(WrappedComponent) {
  return function AuthenticatedComponent(props) {
    const user = useCurrentUser();
    if (!user) return <Login />;
    return <WrappedComponent {...props} user={user} />;
  };
}

// Problems at scale:
// 1. Wrapper hell: withAuth(withTheme(withRouter(App)))
// 2. Prop collision: multiple HOCs inject same prop name
// 3. Static composition: can't use HOCs conditionally
// 4. Ref forwarding: refs don't pass through by default
// 5. DevTools: deeply nested component tree

7.2 Render Props

// Render prop: component that takes a function as children/prop
function MouseTracker({ render }) {
  const [pos, setPos] = useState({ x: 0, y: 0 });
  return (
    <div onMouseMove={e => setPos({ x: e.clientX, y: e.clientY })}>
      {render(pos)}
    </div>
  );
}

// Usage
<MouseTracker render={({ x, y }) => <Cursor x={x} y={y} />} />

// Problems:
// 1. Callback nesting: multiple render props = "callback hell"
// 2. Re-render: inline function creates new reference each render
// 3. Readability: logic scattered across component boundaries

7.3 Hooks — The Current Model

// Hooks: compose behavior without changing component hierarchy
function useMousePosition() {
  const [pos, setPos] = useState({ x: 0, y: 0 });
  useEffect(() => {
    const handler = e => setPos({ x: e.clientX, y: e.clientY });
    window.addEventListener('mousemove', handler);
    return () => window.removeEventListener('mousemove', handler);
  }, []);
  return pos;
}

// Usage — flat, composable, no wrapper components
function App() {
  const { x, y } = useMousePosition();
  const user = useAuth();
  const theme = useTheme();
  return <Cursor x={x} y={y} user={user} theme={theme} />;
}
Pattern Composition Tree Impact Type Safety Debugging
HOC Static (wrap at definition) Adds wrapper nodes Hard (prop injection) Deep nesting in DevTools
Render Props Dynamic (at render time) Adds wrapper nodes Good (explicit params) Callback nesting
Hooks Dynamic (in function body) No extra nodes Excellent (return types) Flat, inspectable

8. Architect Considerations

8.1 Component Definition Anti-Patterns

// ANTI-PATTERN: Component defined inside render
function Parent() {
  // New function reference every render → new type → full remount
  function Child() {
    return <input />;  // Input loses focus every Parent render!
  }
  return <Child />;
}

// CORRECT: Define outside or use useMemo (rarely needed)
function Child() {
  return <input />;
}
function Parent() {
  return <Child />;
}

8.2 React.memo Deep Dive

// React.memo wraps a component with a comparison layer
const MemoizedList = React.memo(
  function List({ items, onSelect }) {
    return items.map(item =>
      <Item key={item.id} item={item} onSelect={onSelect} />
    );
  },
  (prevProps, nextProps) => {
    // Custom comparison — return true to SKIP re-render
    return prevProps.items === nextProps.items
        && prevProps.onSelect === nextProps.onSelect;
  }
);

// Internally, React.memo creates a wrapper element type:
// {
//   $$typeof: Symbol.for('react.memo'),
//   type: List,           // the wrapped component
//   compare: customFn     // or null for shallow compare
// }

8.3 forwardRef Mechanics

// forwardRef creates a wrapper that passes ref as second argument
const FancyInput = React.forwardRef((props, ref) => {
  return <input ref={ref} className="fancy" {...props} />;
});

// Internally:
// {
//   $$typeof: Symbol.for('react.forward_ref'),
//   render: (props, ref) => ...,  // the render function
// }

// React 19 simplifies this — ref is just a prop:
function FancyInput({ ref, ...props }) {
  return <input ref={ref} className="fancy" {...props} />;
}
Staff-Level Design Principle

At L6, you should be able to explain why React chose functions as the primary component model: they're simpler to compose, easier to optimize (no instance allocation), naturally align with hooks' closure-based state, and enable the compiler (React Compiler/Forget) to automatically memoize without developer intervention. The class model's this-based state created a mutable reference that captured stale values in async callbacks — the fundamental problem hooks solve by capturing values in closures at render time.
