1. Why Fiber Exists

Before React 16, the reconciler (called the "Stack Reconciler") processed the entire component tree synchronously in a single, uninterruptible call stack. For large trees, this could block the main thread for hundreds of milliseconds, causing dropped frames and unresponsive UIs.

Stack Reconciler (React ≤15): ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━ │ Reconcile entire tree (synchronous, uninterruptible) │ ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━ 200ms+ blocked ↓ Dropped frames, janky UI Fiber Reconciler (React 16+): ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━ │ Work │ Yield │ Work │ Yield │ Work │ Yield │ Commit │ ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━ 5ms browser 5ms browser 5ms browser sync event paint input

Fiber was a complete rewrite of React's core algorithm, designed to make rendering interruptible. The key insight: by representing the component tree as a linked list of fiber nodes (rather than recursive function calls on the stack), React can pause work, do something more important, and resume later.

Architect Insight

Fiber is essentially React's own virtual call stack implemented in the heap. A real call stack (function calls) can't be paused and resumed — once you start recursing, you must finish. By moving the "stack" to heap-allocated fiber objects with explicit pointers (child, sibling, return), React gains full control over scheduling.

2. Fiber Node Structure

Each fiber is a JavaScript object with ~30 fields. Here are the critical ones:

// Simplified Fiber node structure (React 18)
type Fiber = {
  // === Identity ===
  tag: WorkTag,           // FunctionComponent(0), ClassComponent(1), HostComponent(5)...
  key: null | string,     // Reconciliation key from JSX
  elementType: any,       // The type from ReactElement (div, App, etc.)
  type: any,              // Resolved type (same as elementType for most)

  // === Tree Structure (linked list) ===
  return: Fiber | null,   // Parent fiber
  child: Fiber | null,    // First child fiber
  sibling: Fiber | null,  // Next sibling fiber
  index: number,          // Position among siblings

  // === State ===
  stateNode: any,         // DOM node | class instance | null (function)
  memoizedState: any,     // Hooks linked list (function) | state (class)
  memoizedProps: any,     // Props from last completed render
  pendingProps: any,      // Props for current render

  // === Effects ===
  flags: Flags,           // Placement | Update | Deletion | ...
  subtreeFlags: Flags,    // Aggregated flags from children
  updateQueue: any,       // Queue of state updates

  // === Double Buffering ===
  alternate: Fiber | null, // Points to the other version of this fiber

  // === Scheduling ===
  lanes: Lanes,           // Priority lanes for this fiber's pending work
  childLanes: Lanes,      // Aggregated lanes from subtree
};

2.1 Key Fields Explained

Field Purpose Example Value
tag Identifies the type of fiber for processing 0 (FunctionComponent)
stateNode The "real" thing this fiber represents DOM element, class instance, or null
memoizedState For function components: head of hooks linked list { memoizedState: 0, queue: {...}, next: hook2 }
flags Bitmask of side effects to perform in commit Placement | Update (binary OR)
alternate Points to the corresponding fiber in the other tree The workInProgress or current version
lanes Bitmask of priority levels with pending work SyncLane | DefaultLane

3. Current vs WorkInProgress (Double Buffering)

React maintains two fiber trees simultaneously, connected via the alternate pointer. This is analogous to double buffering in graphics programming.

current tree workInProgress tree (on screen) (being built) ─────────── ───────────────── ┌─────────┐ alternate ┌─────────┐ │ App │ ◄──────────────► │ App │ └────┬────┘ └────┬────┘ │ │ ┌────┴────┐ ┌────┴────┐ │ div │ ◄──────────────► │ div │ └──┬───┬──┘ └──┬───┬──┘ │ │ │ │ ┌─────┘ └─────┐ ┌─────┘ └─────┐ ┌────┴──┐ ┌────┴──┐ ┌────┴──┐ ┌────┴──┐ │ h1 │ │ p │ │ h1 │ │ p │ └───────┘ └───────┘ └───────┘ └───────┘ ▲ ▲ ▲ ▲ │ alternate │ │ alternate │ └───────────────┘ └───────────────┘ After commit: trees swap roles. workInProgress becomes current, old current becomes next workInProgress.

3.1 Why Double Buffering?

// Without double buffering:
// Modifying the current tree in-place means partially updated UI
// is visible if rendering is interrupted.

// With double buffering:
// 1. Build the entire new tree (workInProgress) off-screen
// 2. Only when complete, swap it in atomically (commit phase)
// 3. The user never sees a partially updated tree

// The swap is a single pointer reassignment:
root.current = finishedWork; // workInProgress becomes current

// The old current tree is reused as the next workInProgress
// (fiber nodes are recycled via alternate pointers)
Memory Efficiency

Fiber nodes are not garbage collected between renders. The alternate pointer keeps both versions alive. When building workInProgress, React clones the current fiber and updates only the changed fields. This avoids allocating entirely new objects each render — a critical optimization for 60fps rendering.

4. Why Linked List, Not Tree

Although we call it a "fiber tree," the data structure is actually a linked list with three pointer types: child, sibling, and return (parent).

4.1 Tree vs Linked List Representation

// Traditional tree (like the DOM):
// Each node has a children ARRAY
{
  type: "div",
  children: [
    { type: "h1", children: [...] },
    { type: "p", children: [...] },
    { type: "span", children: [...] }
  ]
}

// Fiber linked list:
// Each node has child (first child), sibling (next sibling), return (parent)
div.child    → h1
h1.sibling   → p
p.sibling    → span
span.sibling → null
h1.return    → div
p.return     → div
span.return  → div

4.2 Why This Matters for Interruptibility

Traversal order (depth-first): App ─────────────────────────────────────────────────► │ │ ▼ │ div ──────────────────────────────────► │ │ │ │ ▼ │ │ h1 ──► "Hello" ──► (complete h1) ──► p ──► (done) ──►│ │ │ ▼ │ "World" │ │ │ (complete p) │ │ │ (complete div) │ │ │ (complete App) ──────┘ At ANY point in this traversal, React can: 1. Save the current fiber pointer 2. Yield to the browser 3. Resume from the saved pointer

With a recursive tree traversal, the "current position" is implicit in the call stack — you can't save and restore it. With a linked list, the current position is an explicit pointer to a fiber node, stored in a variable that persists across yields.

5. Unit of Work Concept

Each fiber node represents one "unit of work." The work loop processes one fiber at a time, checking after each unit whether it should yield to the browser.

// Simplified React work loop (conceptual)
let workInProgress = null; // Current fiber being processed

function workLoop(deadline) {
  while (workInProgress !== null && !shouldYield()) {
    workInProgress = performUnitOfWork(workInProgress);
  }

  if (workInProgress !== null) {
    // More work to do — schedule continuation
    scheduleCallback(workLoop);
  } else {
    // Render phase complete — enter commit phase
    commitRoot();
  }
}

function performUnitOfWork(fiber) {
  // Phase 1: beginWork — process this fiber, return first child
  const next = beginWork(fiber);

  if (next !== null) {
    return next; // Descend into child
  }

  // Phase 2: completeWork — no children, bubble up
  let completedWork = fiber;
  while (completedWork !== null) {
    completeWork(completedWork);

    if (completedWork.sibling !== null) {
      return completedWork.sibling; // Process next sibling
    }
    completedWork = completedWork.return; // Bubble up to parent
  }
  return null; // Reached root — render complete
}

6. requestIdleCallback Origin Story

The original Fiber proposal (2016) planned to use the browser's requestIdleCallback API for scheduling. React ultimately abandoned this approach.

6.1 Why requestIdleCallback Seemed Perfect

// requestIdleCallback runs when the browser is idle
requestIdleCallback((deadline) => {
  while (deadline.timeRemaining() > 0 && workRemaining) {
    doWork();
  }
});

6.2 Why React Stopped Using It

Problem Detail
Low frequency Only fires ~20 times/sec (50ms intervals), too slow for smooth UI updates
Inconsistent timing Varies wildly across browsers and devices
Tab backgrounding Throttled or stopped when tab is not visible
No priority support All idle callbacks are equal — no way to express urgency
Safari support Not implemented in Safari (as of 2024)

6.3 What React Uses Instead: MessageChannel

// React's scheduler uses MessageChannel for ~5ms time slicing
const channel = new MessageChannel();
const port = channel.port2;

channel.port1.onmessage = () => {
  // This fires as a macrotask — after browser has had
  // a chance to paint and handle input
  const currentTime = getCurrentTime();
  const deadline = currentTime + 5; // 5ms time slice

  while (getCurrentTime() < deadline && hasWork()) {
    performWork();
  }

  if (hasWork()) {
    port.postMessage(null); // Schedule next slice
  }
};

// Why MessageChannel over setTimeout?
// setTimeout(fn, 0) has a minimum 4ms delay (HTML spec)
// MessageChannel fires immediately in the next macrotask

7. Fiber Tags

Every fiber has a tag field (a number) that tells React how to process it. Here are the most important tags:

// From ReactWorkTags.js in React source
const FunctionComponent = 0;
const ClassComponent = 1;
const IndeterminateComponent = 2; // Before first render (function or class?)
const HostRoot = 3;              // Root of the fiber tree (createRoot)
const HostPortal = 4;            // createPortal
const HostComponent = 5;         // DOM elements: div, span, input
const HostText = 6;              // Text nodes: "hello"
const Fragment = 7;              // React.Fragment
const Mode = 8;                  // StrictMode, ConcurrentMode
const ContextConsumer = 9;      // Context.Consumer
const ContextProvider = 10;     // Context.Provider
const ForwardRef = 11;          // React.forwardRef
const Profiler = 12;            // React.Profiler
const SuspenseComponent = 13;  // Suspense boundary
const MemoComponent = 14;       // React.memo
const SimpleMemoComponent = 15; // React.memo without custom compare
const LazyComponent = 16;      // React.lazy
const OffscreenComponent = 22; // Offscreen (Activity API)
IndeterminateComponent

When React first encounters a component, it doesn't know if it's a function or class. It assigns tag 2 (IndeterminateComponent). After calling it and checking the result, it reassigns to 0 (FunctionComponent) or 1 (ClassComponent). This only happens once per component type.

8. Priority Lanes System Overview

Lanes replaced the older "expiration time" priority system in React 18. Each lane is a bit in a 31-bit integer, allowing multiple priorities to be expressed and combined using bitwise operations.

// Lane definitions (from ReactFiberLane.js)
const NoLane       = 0b0000000000000000000000000000000;
const SyncLane     = 0b0000000000000000000000000000010; // Highest priority
const InputContinuousLane
                   = 0b0000000000000000000000000001000; // Drag, scroll
const DefaultLane  = 0b0000000000000000000000000100000; // Normal updates
const TransitionLane1
                   = 0b0000000000000000000000001000000; // startTransition
// ... TransitionLane2 through TransitionLane16 ...
const IdleLane     = 0b0100000000000000000000000000000; // Lowest priority
const OffscreenLane= 0b1000000000000000000000000000000; // Offscreen/hidden

// Lanes are combined with bitwise OR:
const pendingLanes = SyncLane | DefaultLane;

// Check if a lane is included:
const hasSyncWork = (pendingLanes & SyncLane) !== NoLane;

8.1 Lane Assignment

Update Source Lane Assigned Behavior
flushSync(() => setState()) SyncLane Synchronous, blocks until committed
Click handler setState() SyncLane Synchronous (discrete event)
Scroll/drag setState() InputContinuousLane High priority, may batch
Normal setState() DefaultLane Standard priority
startTransition(() => setState()) TransitionLane Low priority, interruptible
useDeferredValue TransitionLane Deferred, shows stale value first

9. Fiber Tree Traversal Algorithm

React traverses the fiber tree in a specific depth-first order, performing beginWork on the way down and completeWork on the way up.

// Given this component tree:
//   App
//   ├── Header
//   │   ├── Logo
//   │   └── Nav
//   └── Main
//       └── Content

// Traversal order:
// beginWork(App)       → child: Header
// beginWork(Header)    → child: Logo
// beginWork(Logo)      → child: null
// completeWork(Logo)   → sibling: Nav
// beginWork(Nav)       → child: null
// completeWork(Nav)    → sibling: null, return: Header
// completeWork(Header) → sibling: Main
// beginWork(Main)      → child: Content
// beginWork(Content)   → child: null
// completeWork(Content)→ sibling: null, return: Main
// completeWork(Main)   → sibling: null, return: App
// completeWork(App)    → DONE, enter commit phase
Staff-Level Insight — Bailout Optimization

During beginWork, React checks if a fiber's props and state have changed. If nothing changed and no pending lanes exist in the subtree (childLanes === NoLanes), React bails out — it skips the entire subtree without calling the component function. This is the fundamental mechanism behind React.memo, useMemo, and the upcoming React Compiler's automatic memoization. Understanding bailout conditions is essential for performance optimization at scale.

// Anti-pattern: excessive wrapper components

// Total fibers: 9 wrappers + actual content fibers

10.2 Fiber Recycling and Memory

// React reuses fiber nodes via the alternate pointer.
// When building workInProgress, React calls createWorkInProgress:

function createWorkInProgress(current, pendingProps) {
  let workInProgress = current.alternate;

  if (workInProgress === null) {
    // First render: create new fiber
    workInProgress = createFiber(current.tag, pendingProps, current.key);
    workInProgress.alternate = current;
    current.alternate = workInProgress;
  } else {
    // Subsequent renders: reuse existing fiber, update props
    workInProgress.pendingProps = pendingProps;
    workInProgress.flags = NoFlags;
    workInProgress.subtreeFlags = NoFlags;
  }

  // Copy fields from current
  workInProgress.child = current.child;
  workInProgress.memoizedState = current.memoizedState;
  // ...
  return workInProgress;
}
