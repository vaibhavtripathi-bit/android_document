Element vs Component vs Instance

ReactElement (immutable description object) vs Component (function/class template) vs Fiber (mutable work-in-progress instance). Understanding these three concepts and their relationships is essential for reasoning about React's behavior.

1. The Three Concepts

Key Mental Model

A Component is like a class definition. A ReactElement is like calling new to create a description (but not the actual object). A Fiber is the actual living instance that React manages. You write Components, JSX creates Elements, and React creates Fibers.

2. ReactElement — The Description

2.1 Immutability

// ReactElements are frozen in development mode
const el = <div className="box">Hello</div>;

// In development:
Object.freeze(el);
Object.freeze(el.props);

// This means you CANNOT mutate elements:
el.props.className = "new"; // TypeError in strict mode

// To "change" an element, create a new one:
const newEl = React.cloneElement(el, { className: "new" });

2.2 Elements Are Cheap

// Creating a ReactElement is just creating a plain object.
// No DOM operations, no side effects, no component execution.

// This is why React can create thousands of elements per render
// without performance issues — they're just descriptions.

// The cost comes later, when React PROCESSES these elements
// during reconciliation (comparing old vs new, updating DOM).

// Benchmark: creating 10,000 ReactElements ≈ 1-2ms
// Processing 10,000 elements through reconciliation ≈ 10-50ms
// Updating 10,000 DOM nodes ≈ 50-200ms

2.3 Element Trees

// A render function returns an element TREE (nested elements)
function App() {
  return (
    <div>
      <Header />
      <Main>
        <Content />
      </Main>
    </div>
  );
}

// This returns:
{
  type: "div",
  props: {
    children: [
      { type: Header, props: {} },
      { type: Main, props: {
        children: { type: Content, props: {} }
      }}
    ]
  }
}

// Note: Header, Main, Content are NOT called yet.
// They're just referenced by the type field.
// React will call them during reconciliation.

3. Component — The Template

3.1 Function Components

// A function component is just a function that:
// - Takes props as argument
// - Returns a ReactElement (or null, string, number, array, Fragment)
// - May call hooks

function Greeting({ name }: { name: string }): ReactElement {
  const [count, setCount] = useState(0);
  return <h1>Hello {name} ({count})</h1>;
}

// The function itself is the component.
// It has no instance — no `this`, no lifecycle methods.
// State lives in the FIBER, not in the component.
// Hooks are stored in fiber.memoizedState as a linked list.

3.2 Class Components

// A class component has an actual instance (created with `new`)
class Greeting extends React.Component {
  state = { count: 0 };

  render() {
    return <h1>Hello {this.props.name} ({this.state.count})</h1>;
  }
}

// The class instance is stored in fiber.stateNode
// State lives on the instance: this.state
// The instance persists across renders (same object, mutated)
// This is why `this.props` can be stale in async callbacks

3.3 Component Identity

// Component identity = the function/class REFERENCE
// React uses === to compare types during reconciliation

const A = () => <div />;
const B = () => <div />;

// A !== B (different function references)
// Even though they produce identical output,
// switching from <A /> to <B /> causes a full remount

// This is why component definitions must be STABLE:
// - Defined at module level (not inside other components)
// - Or memoized if dynamically created (rare)

4. Fiber — The Instance

4.1 What Fibers Store

Component Type stateNode memoizedState memoizedProps
Function component null Hooks linked list Last rendered props
Class component Class instance (this) this.state snapshot Last rendered props
Host component (div) DOM element null Last rendered attributes
Host text ("hello") Text node null Text content
HostRoot FiberRoot Root element null

4.2 Hooks in Fiber

// For function components, hooks are stored as a linked list
// in fiber.memoizedState

function Counter() {
  const [count, setCount] = useState(0);     // Hook 1
  const [name, setName] = useState("");     // Hook 2
  const ref = useRef(null);                  // Hook 3
  useEffect(() => { /* ... */ }, [count]);  // Hook 4
  // ...
}

// fiber.memoizedState points to:
// Hook1 { memoizedState: 0, queue: {...}, next: ──►
// Hook2 { memoizedState: "", queue: {...}, next: ──►
// Hook3 { memoizedState: { current: null }, next: ──►
// Hook4 { memoizedState: { effect, deps: [0] }, next: null }

// This is WHY hooks must be called in the same order:
// React walks the linked list by position, not by name.
// If you skip a hook conditionally, all subsequent hooks
// read the wrong node in the list.

5. Lifecycle: Creation to Destruction

JSX Expression Reconciliation DOM ────────────── ────────────── ─── <App /> beginWork(App fiber) │ │ ▼ ▼ ReactElement { Call App() → returns element tree type: App, │ props: {} ▼ } Create child fibers │ ▼ completeWork → create DOM nodes │ ▼ commitWork → insert into DOM → visible Re-render (state change): │ ▼ New ReactElement { beginWork(App fiber) type: App, │ props: {} ▼ } Call App() → returns new element tree │ ▼ Diff old vs new children │ ▼ Update changed fibers │ ▼ commitWork → update DOM → visible Unmount: Delete fiber, run cleanup │ ▼ commitDeletion → remove DOM → gone

6. Lazy Components and Dynamic Import

// React.lazy wraps a dynamic import in a special element type
const Dashboard = React.lazy(() => import('./Dashboard'));

// The lazy wrapper has its own $$typeof:
{
  $$typeof: Symbol.for('react.lazy'),
  _payload: {
    _status: -1,  // Uninitialized | Pending | Resolved | Rejected
    _result: () => import('./Dashboard')  // or resolved module
  },
  _init: lazyInitializer  // function to trigger loading
}

// Status values:
const Uninitialized = -1;
const Pending = 0;
const Resolved = 1;
const Rejected = 2;

// When React renders a lazy component:
// 1. _init checks _status
// 2. If Uninitialized: call _result() (the import), set Pending
// 3. If Pending: throw the promise (Suspense catches it)
// 4. If Resolved: return _result (the actual component)
// 5. If Rejected: throw _result (error boundary catches it)

7. Wrapper Layers: memo, forwardRef, lazy

React has several "wrapper" element types that add behavior around a component. Each creates a distinct $$typeof that React handles specially during reconciliation.

// React.memo — adds shallow comparison
const MemoComp = React.memo(MyComponent);
// Creates: { $$typeof: Symbol(react.memo), type: MyComponent, compare: null }
// Fiber tag: MemoComponent (14) or SimpleMemoComponent (15)

// React.forwardRef — passes ref as second argument
const FwdComp = React.forwardRef((props, ref) => ...);
// Creates: { $$typeof: Symbol(react.forward_ref), render: fn }
// Fiber tag: ForwardRef (11)

// React.lazy — deferred loading
const LazyComp = React.lazy(() => import('./Comp'));
// Creates: { $$typeof: Symbol(react.lazy), _init: fn, _payload: {...} }
// Fiber tag: LazyComponent (16)

// These can be composed:
const Composed = React.memo(React.forwardRef((props, ref) => ...));
// Outer: memo wrapper → Inner: forwardRef wrapper → render function
Wrapper $$typeof Fiber Tag Purpose
React.memo react.memo 14 / 15 Skip re-render if props unchanged
React.forwardRef react.forward_ref 11 Pass ref through to child
React.lazy react.lazy 16 Code splitting with Suspense
Context.Provider react.provider 10 Provide context value to subtree
Context.Consumer react.context 9 Read context via render prop

8. createElement vs cloneElement

8.1 createElement

// React.createElement creates a NEW element from scratch
React.createElement(type, props, ...children)

// Equivalent to JSX:
<Button color="blue">Click</Button>
// → React.createElement(Button, { color: "blue" }, "Click")

// Returns:
{
  $$typeof: Symbol.for('react.element'),
  type: Button,
  props: { color: "blue", children: "Click" },
  key: null,
  ref: null
}

8.2 cloneElement

// React.cloneElement creates a copy with merged/overridden props
React.cloneElement(element, newProps, ...newChildren)

const original = <Button color="blue" size="lg">Click</Button>;
const cloned = React.cloneElement(original, { color: "red" });

// cloned:
{
  type: Button,       // same type
  props: {
    color: "red",    // overridden
    size: "lg",      // preserved from original
    children: "Click" // preserved (no new children passed)
  },
  key: null,          // preserved from original
  ref: null           // preserved from original
}
cloneElement Is Discouraged

The React team discourages cloneElement in new code. It makes data flow implicit (parent secretly injects props into children), breaks with React.memo (cloned elements have new props objects), and is hard to type correctly in TypeScript. Prefer render props, context, or composition patterns instead. React 19's documentation explicitly marks it as a legacy API.

9. Architect Considerations

9.1 Element Creation Fast Path

// React optimizes element creation in the automatic JSX runtime.
// The jsx() function is simpler than createElement():

// createElement: handles key extraction, ref extraction,
// defaultProps, __source/__self, children normalization

// jsx/jsxs: key is a separate argument, children are in props,
// no defaultProps processing (deprecated in React 19),
// no __source/__self (separate jsxDEV for dev)

// This makes jsx() ~15% faster than createElement()
// for element-heavy renders (thousands of elements)

9.2 When Elements Become Expensive

// Elements themselves are cheap, but they trigger work:

// 1. New element with same type → reconciliation (prop diff)
// 2. New element with different type → full subtree remount
// 3. Array of elements → list reconciliation (key matching)

// The cost is proportional to:
// - Number of elements that CHANGED (not total elements)
// - Depth of changed subtrees
// - Number of DOM mutations required

// React.memo prevents element PROCESSING, not creation.
// The element is still created, but React skips calling
// the component function if props haven't changed.
Staff-Level Design Pattern

Understanding the element → fiber pipeline enables a powerful optimization: lifting elements out of render. If a JSX subtree doesn't depend on any props or state, you can define it as a module-level constant. React will see the same element reference every render and skip reconciliation entirely for that subtree. This is what the React Compiler automates — it identifies which JSX expressions are "stable" and hoists them.

// Manual element hoisting (what React Compiler does automatically)
const STATIC_FOOTER = (
  <footer>
    <p>Copyright 2024</p>
    <nav>
      <a href="/privacy">Privacy</a>
      <a href="/terms">Terms</a>
    </nav>
  </footer>
);

function App() {
  return (
    <div>
      <DynamicContent />
      {STATIC_FOOTER}  {/* Same reference every render → skip diff */}
    </div>
  );
}
