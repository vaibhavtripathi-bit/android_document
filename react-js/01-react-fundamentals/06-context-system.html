1. createContext Internals

// React.createContext returns an object with Provider and Consumer
const ThemeContext = React.createContext('light');

// Internally, this creates:
{
  $$typeof: Symbol.for('react.context'),
  _currentValue: 'light',     // Current value (primary renderer)
  _currentValue2: 'light',    // Current value (secondary renderer)
  _threadCount: 0,             // For concurrent rendering
  Provider: {
    $$typeof: Symbol.for('react.provider'),
    _context: /* circular ref to context object */
  },
  Consumer: {
    $$typeof: Symbol.for('react.context'),
    _context: /* circular ref to context object */
  },
  _defaultValue: 'light',     // The default passed to createContext
  _globalName: null            // For DevTools
}
Two Current Values

_currentValue and _currentValue2 exist because React supports multiple renderers simultaneously (e.g., react-dom and react-art in the same app). Each renderer reads from its own slot. In practice, most apps only use one renderer, so _currentValue2 is unused.

2. Provider/Consumer Model

// Provider: sets the context value for its subtree
<ThemeContext.Provider value="dark">
  <App />
</ThemeContext.Provider>

// Consumer (3 ways to read context):

// 1. useContext hook (recommended)
function Button() {
  const theme = useContext(ThemeContext);
  return <button className={theme}>Click</button>;
}

// 2. Consumer component (legacy, still works)
<ThemeContext.Consumer>
  {theme => <button className={theme}>Click</button>}
</ThemeContext.Consumer>

// 3. Class component contextType (legacy)
class Button extends React.Component {
  static contextType = ThemeContext;
  render() {
    return <button className={this.context}>Click</button>;
  }
}

3. Value Propagation Through Fiber Tree

Context value propagation: ThemeContext.Provider (value="dark") │ ├── Header ──── Logo │ └── uses ThemeContext ← reads "dark" │ ├── ThemeContext.Provider (value="light") ← nested override │ │ │ └── Sidebar │ └── uses ThemeContext ← reads "light" (nearest) │ └── Footer └── uses ThemeContext ← reads "dark" (outer provider) 
// During the render phase, when React encounters a Provider:
// 1. It pushes the new value onto an internal context stack
// 2. All descendants read from the top of the stack
// 3. When leaving the Provider subtree, it pops the value

// This is a stack-based approach:
// push(ThemeContext, "dark")    ← enter outer Provider
//   read → "dark"
//   push(ThemeContext, "light") ← enter inner Provider
//     read → "light"
//   pop(ThemeContext)           ← leave inner Provider
//   read → "dark"
// pop(ThemeContext)             ← leave outer Provider

4. How React Finds the Nearest Provider

// When useContext(ThemeContext) is called during render:
// React does NOT walk up the fiber tree looking for a Provider.
// Instead, it reads context._currentValue directly.

// The Provider's beginWork sets context._currentValue
// before processing children. Since rendering is depth-first,
// children always see the correct value.

// Simplified Provider beginWork:
function updateContextProvider(fiber) {
  const context = fiber.type._context;
  const newValue = fiber.pendingProps.value;

  // Push new value
  pushProvider(fiber, context, newValue);
  context._currentValue = newValue;

  // Process children (they'll read the new value)
  reconcileChildren(fiber, fiber.pendingProps.children);
}

// Simplified useContext:
function useContext(context) {
  // Mark this fiber as a consumer (for change propagation)
  readContext(context);
  return context._currentValue; // Direct read!
}

5. Bail-Out Optimization

When a Provider re-renders with the same value (by reference), React can bail out and skip re-rendering consumers.

// React compares old and new values with Object.is()
function updateContextProvider(fiber) {
  const oldValue = fiber.memoizedProps.value;
  const newValue = fiber.pendingProps.value;

  if (Object.is(oldValue, newValue)) {
    // Same value → bail out
    // Children won't re-render due to context change
    // (they might still re-render for other reasons)
    return;
  }

  // Different value → propagate change to all consumers
  propagateContextChange(fiber, context, newValue);
}
The Object Identity Trap

If the Provider's value is an object literal created during render, it will be a new reference every render, defeating the bail-out optimization. This is one of the most common context performance mistakes.

// BAD: New object every render → all consumers re-render
function App() {
  const [user, setUser] = useState(null);
  return (
    <UserContext.Provider value={{ user, setUser }}>
      {/* { user, setUser } is a NEW object every render */}
      <App />
    </UserContext.Provider>
  );
}

// GOOD: Memoize the value object
function App() {
  const [user, setUser] = useState(null);
  const value = useMemo(() => ({ user, setUser }), [user]);
  return (
    <UserContext.Provider value={value}>
      <App />
    </UserContext.Provider>
  );
}

6. Why Context Triggers Re-Renders in All Consumers

When a context value changes, every component that calls useContext for that context will re-render, regardless of whether it uses the specific part of the value that changed.

// Context with multiple values
const AppContext = React.createContext({
  theme: 'light',
  locale: 'en',
  user: null
});

// Component that only uses theme
function ThemeButton() {
  const { theme } = useContext(AppContext);
  return <button className={theme}>Click</button>;
}

// When user changes (but theme doesn't):
// ThemeButton STILL re-renders because the context VALUE
// (the entire object) changed reference.
// React cannot know that ThemeButton only uses .theme

6.1 The Propagation Mechanism

// When a Provider's value changes, React calls propagateContextChange.
// This walks the ENTIRE subtree of the Provider, looking for consumers.

// For each consumer found:
// 1. Mark the consumer fiber with a lane (schedule re-render)
// 2. Mark all ancestors up to the Provider with childLanes
// 3. This ensures the consumer is visited during the next render

// Key insight: this propagation CANNOT be stopped by React.memo
// or shouldComponentUpdate. Context changes bypass memoization.
// The consumer will re-render even if its parent is memoized.
Provider (value changed) │ ├── MemoizedParent (React.memo) ← would normally skip │ │ │ └── Consumer (useContext) ← STILL re-renders! │ Context change bypasses memo. │ React marks this fiber directly. │ └── AnotherConsumer ← also re-renders

7. Techniques to Avoid Unnecessary Re-Renders

7.1 Split Contexts

// Instead of one large context:
const AppContext = createContext({ theme, locale, user });

// Split into focused contexts:
const ThemeContext = createContext('light');
const LocaleContext = createContext('en');
const UserContext = createContext(null);

// Now ThemeButton only subscribes to ThemeContext
// Changes to user/locale don't trigger ThemeButton re-render

7.2 Separate State from Dispatch

// Common pattern: separate read-heavy and write-heavy contexts
const StateContext = createContext(null);
const DispatchContext = createContext(null);

function AppProvider({ children }) {
  const [state, dispatch] = useReducer(reducer, initialState);

  return (
    <DispatchContext.Provider value={dispatch}>
      <StateContext.Provider value={state}>
        {children}
      </StateContext.Provider>
    </DispatchContext.Provider>
  );
}

// dispatch is a stable reference (from useReducer)
// Components that only dispatch actions never re-render
// from state changes

7.3 Memoize Children

// The children pattern prevents re-rendering the subtree
// when only the Provider's own state changes

function ThemeProvider({ children }) {
  const [theme, setTheme] = useState('light');
  const value = useMemo(() => ({ theme, setTheme }), [theme]);

  return (
    <ThemeContext.Provider value={value}>
      {children}  {/* children is the SAME reference (from props) */}
    </ThemeContext.Provider>
  );
}

// Usage:
<ThemeProvider>
  <App />  {/* This element is created by ThemeProvider's parent */}
</ThemeProvider>

// When ThemeProvider re-renders (theme changes):
// - children prop is the same reference → subtree bails out
// - Only actual context consumers re-render

7.4 Selector Pattern (External Libraries)

// React doesn't natively support context selectors.
// Libraries like use-context-selector provide this:

import { createContext, useContextSelector } from 'use-context-selector';

const AppContext = createContext({ theme: 'light', count: 0 });

function ThemeButton() {
  // Only re-renders when theme changes, not count
  const theme = useContextSelector(AppContext, ctx => ctx.theme);
  return <button className={theme}>Click</button>;
}

// How it works internally:
// 1. Subscribes to context changes (like normal useContext)
// 2. When context changes, runs selector on new value
// 3. Compares selector result with previous
// 4. Only triggers re-render if selector result changed
// Uses useSyncExternalStore under the hood

8. use(Context) in React 19

// React 19 introduces the use() hook for reading context
import { use } from 'react';

function Button() {
  const theme = use(ThemeContext);
  return <button className={theme}>Click</button>;
}

// Key difference from useContext:
// use() can be called CONDITIONALLY and in loops!

function ConditionalTheme({ useTheme }) {
  if (useTheme) {
    const theme = use(ThemeContext); // Valid in React 19!
    return <div className={theme} />;
  }
  return <div />;
}

// use() also works with Promises:
function UserProfile({ userPromise }) {
  const user = use(userPromise); // Suspends until resolved
  return <h1>{user.name}</h1>;
}

9. Module-Level Context Patterns

// Pattern: Context module with custom hook and provider

// auth-context.tsx
const AuthContext = createContext<AuthState | null>(null);

export function AuthProvider({ children }: { children: ReactNode }) {
  const [user, setUser] = useState<User | null>(null);
  const [loading, setLoading] = useState(true);

  const value = useMemo(
    () => ({ user, loading, login, logout }),
    [user, loading]
  );

  return (
    <AuthContext.Provider value={value}>
      {children}
    </AuthContext.Provider>
  );
}

export function useAuth(): AuthState {
  const ctx = useContext(AuthContext);
  if (ctx === null) {
    throw new Error('useAuth must be used within AuthProvider');
  }
  return ctx;
}

// The null default + runtime check pattern ensures:
// 1. TypeScript knows the value is non-null after useAuth()
// 2. Missing Provider is caught immediately with clear error
// 3. No accidental use of stale default values

10. Architect Considerations

10.1 Context vs External State

Concern React Context External Store (Zustand, Redux)
Selector support No (all consumers re-render) Yes (fine-grained subscriptions)
Middleware Manual (wrap dispatch) Built-in (logging, persistence)
DevTools React DevTools only Dedicated DevTools
SSR Built-in (per-request isolation) Requires careful setup
Bundle size 0 KB (built-in) 1-10 KB
Best for Low-frequency, scoped state (theme, auth, locale) High-frequency, global state (UI state, cache)
Staff-Level Architecture Decision

At L6, the question isn't "context vs Redux" — it's "what's the right state management architecture for this app's scale and team?" Context is ideal for dependency injection (providing services, configuration, auth) where values change infrequently. For application state that changes frequently and is consumed by many components, an external store with selectors prevents the cascading re-render problem. Many large apps use both: context for DI, external store for UI state.
