1. The Tree Diffing Problem

Computing the minimum number of operations to transform one tree into another is a well-studied problem in computer science. The optimal algorithm has O(n³) time complexity — completely impractical for UI rendering where trees can have thousands of nodes and updates happen 60 times per second.

General tree edit distance: O(n³) ───────────────────────────────── For a tree with 1,000 nodes: 1,000³ = 1,000,000,000 operations At 1GHz ≈ 1 second per diff At 60fps budget of 16ms → impossible React's heuristic: O(n) ─────────────────────── For a tree with 1,000 nodes: 1,000 operations At 1GHz ≈ 1 microsecond per diff Leaves 15.999ms for everything else

React achieves O(n) by making two assumptions that hold true for virtually all real-world UI updates. When these assumptions are violated, React does more work than necessary (destroying and recreating subtrees) but never produces incorrect output.

2. React's Two Heuristic Assumptions

The Two Assumptions

Assumption 1: Two elements of different types will produce different trees. If a <div> changes to a <span>, React tears down the entire old subtree and builds a new one — it doesn't try to morph the div into a span.

Assumption 2: The developer can hint at which child elements are stable across renders with a key prop. Keys tell React "this element in the new list corresponds to that element in the old list."

2.1 Why These Assumptions Work

// Assumption 1 holds because:
// - Changing a <div> to <span> almost always means different content
// - Changing <Header> to <Footer> always means different subtree
// - In practice, type changes at the same position are rare

// Assumption 2 holds because:
// - List items usually have stable identifiers (database IDs)
// - Developers know which items are "the same" across updates
// - Without keys, React can only match by position (fragile)

3. Same-Type Element Diffing

When React compares two elements with the same type, it keeps the existing DOM node (or component instance) and only updates the changed attributes/props.

3.1 DOM Element Diffing

// Old
<div className="old" style={{ color: 'red', fontSize: 14 }} />

// New
<div className="new" style={{ color: 'red', fontSize: 16 }} />

// React's diff result:
// 1. className changed: "old" → "new"  → update DOM attribute
// 2. style.color unchanged: "red"      → skip
// 3. style.fontSize changed: 14 → 16   → update inline style
// The DOM node itself is REUSED (not destroyed/recreated)

3.2 Component Element Diffing

// Old
<UserProfile userId={1} showAvatar={true} />

// New
<UserProfile userId={2} showAvatar={true} />

// Same type (UserProfile) → component instance is REUSED
// React updates props: { userId: 1 } → { userId: 2 }
// Component re-renders with new props
// State is PRESERVED (same fiber, same instance)
// Lifecycle: componentDidUpdate / useEffect cleanup+run
Subtle Bug: State Preservation

When React reuses a component because the type is the same, it preserves state. This can cause bugs when you want state to reset — e.g., switching between two user profiles. The fix is to use a key prop: <UserProfile key={userId} userId={userId} />.

4. Cross-Type Destruction

When the type changes at the same position, React destroys the entire old subtree and builds a new one from scratch.

// Old tree
<div>
  <Counter />  // has state: count = 5
  <Timer />    // has running interval
</div>

// New tree (div → section)
<section>
  <Counter />
  <Timer />
</section>

// React's behavior:
// 1. Type changed: "div" → "section"
// 2. DESTROY entire old subtree:
//    - Counter: unmount, state LOST (count = 5 gone)
//    - Timer: unmount, interval cleared
//    - div: removed from DOM
// 3. BUILD entire new subtree:
//    - section: created in DOM
//    - Counter: fresh mount, state = 0
//    - Timer: fresh mount, new interval
Cross-type destruction cascade: Old: div ──┬── Counter (state: 5) └── Timer (interval running) Type change: div → section Step 1: Unmount old subtree (bottom-up) Timer.componentWillUnmount() → interval cleared Counter.componentWillUnmount() → state lost DOM: remove <div> and children Step 2: Mount new subtree (top-down) DOM: create <section> Counter: mount with initial state (0) Timer: mount with new interval DOM: append children to <section>

5. List Diffing with Keys

List reconciliation is the most complex part of the diffing algorithm. Without keys, React can only match children by position. With keys, React can identify which items moved, were added, or were removed.

5.1 Without Keys (Positional Matching)

// Old list
<ul>
  <li>Alice</li>    // position 0
  <li>Bob</li>      // position 1
</ul>

// New list (prepend Charlie)
<ul>
  <li>Charlie</li>  // position 0 → matches old "Alice"
  <li>Alice</li>    // position 1 → matches old "Bob"
  <li>Bob</li>      // position 2 → NEW (no old match)
</ul>

// Without keys, React matches by position:
// pos 0: "Alice" → "Charlie" (UPDATE text content)
// pos 1: "Bob" → "Alice" (UPDATE text content)
// pos 2: nothing → "Bob" (INSERT new <li>)
// Result: 3 DOM operations (2 updates + 1 insert)
// Optimal: 1 DOM operation (insert Charlie at position 0)

5.2 With Keys (Identity Matching)

// Old list
<ul>
  <li key="alice">Alice</li>
  <li key="bob">Bob</li>
</ul>

// New list (prepend Charlie)
<ul>
  <li key="charlie">Charlie</li>
  <li key="alice">Alice</li>
  <li key="bob">Bob</li>
</ul>

// With keys, React matches by key:
// key "charlie": no old match → INSERT new <li>
// key "alice": found at old pos 0 → REUSE (no update needed)
// key "bob": found at old pos 1 → REUSE (no update needed)
// Result: 1 DOM operation (insert Charlie)

5.3 Reordering with Keys

Old: [A] [B] [C] [D] Keys: a, b, c, d New: [D] [A] [B] [C] Keys: d, a, b, c React's algorithm (simplified): 1. Build map of old keys: { a:0, b:1, c:2, d:3 } 2. Walk new list: - key "d": found in old at pos 3 → MOVE to pos 0 - key "a": found in old at pos 0 → already in order relative to "d" - key "b": found in old at pos 1 → already in order - key "c": found in old at pos 2 → already in order 3. Result: 1 DOM move operation (move D before A)

6. Why Index as Key Is Problematic

Using array index as key (key={index}) is equivalent to having no key at all for any operation that changes the order or length of the list.

6.1 The Problem Demonstrated

// List with index keys
items.map((item, index) => <Item key={index} data={item} />)

// Old: items = ["Alice", "Bob", "Charlie"]
// Rendered: key=0:Alice, key=1:Bob, key=2:Charlie

// Delete "Alice" (first item)
// New: items = ["Bob", "Charlie"]
// Rendered: key=0:Bob, key=1:Charlie

// React's diff with index keys:
// key=0: "Alice" → "Bob" (UPDATE — wrong! Should reuse Bob's fiber)
// key=1: "Bob" → "Charlie" (UPDATE — wrong!)
// key=2: "Charlie" → nothing (DELETE)

// Problems:
// 1. Every remaining item gets updated (unnecessary DOM work)
// 2. Component state is associated with wrong data
// 3. Uncontrolled inputs show wrong values
// 4. Animations restart on wrong elements

6.2 When Index Keys Are Safe

Condition Index Key Safe? Reason
Static list (never reordered/filtered) Yes Position never changes
Items have no state or uncontrolled inputs Yes No state to misassociate
List is sorted/filtered/reordered No Position changes break identity
Items are added/removed from middle No Shifts all subsequent indices
Items have animations or transitions No Animations attach to wrong elements

7. The Reconciliation Algorithm in Detail

React's list reconciliation (in reconcileChildrenArray) uses a two-pass approach optimized for the common case where most items don't move.

7.1 First Pass: Linear Scan

// Pass 1: Walk both old and new lists from the start
// Compare old[i] with new[i] by key

// Old: [A, B, C, D, E]
// New: [A, B, X, D, E]  (C replaced with X)

// i=0: old=A, new=A → same key → UPDATE (reuse fiber)
// i=1: old=B, new=B → same key → UPDATE (reuse fiber)
// i=2: old=C, new=X → different key → BREAK (exit first pass)

7.2 Second Pass: Map-Based Lookup

// Pass 2: Build a Map of remaining old children by key
// Then walk remaining new children, looking up in the map

// Remaining old: { C: fiber_C, D: fiber_D, E: fiber_E }
// Remaining new: [X, D, E]

// new=X: not in map → CREATE new fiber
// new=D: found in map → REUSE fiber_D (move if needed)
// new=E: found in map → REUSE fiber_E (move if needed)

// Remaining in map after pass 2: { C: fiber_C }
// → DELETE fiber_C

7.3 Move Detection

// React tracks the last placed index to detect moves
let lastPlacedIndex = 0;

// For each reused fiber from the map:
// If oldIndex >= lastPlacedIndex → no move needed
// If oldIndex < lastPlacedIndex → MOVE (mark Placement flag)

// Example: Old [A,B,C,D] → New [D,A,B,C]
// After first pass breaks immediately (D≠A):
// Map: { A:0, B:1, C:2, D:3 }
//
// new=D: oldIndex=3, lastPlaced=0 → 3≥0, no move. lastPlaced=3
// new=A: oldIndex=0, lastPlaced=3 → 0<3, MOVE A
// new=B: oldIndex=1, lastPlaced=3 → 1<3, MOVE B
// new=C: oldIndex=2, lastPlaced=3 → 2<3, MOVE C
// Result: 3 moves (A, B, C moved after D)
Algorithm Limitation

React's algorithm is not optimal for all reordering patterns. Moving the last item to the front (as above) causes N-1 moves. Moving the first item to the end would only cause 1 move. The algorithm is optimized for the common case (appends, small changes) at the cost of worst-case reorderings. Libraries like inferno use the longest increasing subsequence (LIS) algorithm for optimal moves, but React chose simplicity and predictability.

8. Performance Characteristics

Operation Without Keys With Keys Optimal
Append to end O(1) — 1 insert O(1) — 1 insert Same
Prepend to start O(n) — update all O(1) — 1 insert Keys win
Delete from middle O(n) — update rest O(1) — 1 delete Keys win
Reverse entire list O(n) — update all O(n) — move all but 1 Similar
Shuffle randomly O(n) — update all O(n) — multiple moves Keys preserve state

9. Architect Considerations

9.1 Key Selection Strategy

// BEST: Stable, unique identifier from data
items.map(item => <Row key={item.id} />)

// ACCEPTABLE: Composite key when no single ID
items.map(item => <Row key={`${item.type}-${item.name}`} />)

// LAST RESORT: Generated ID (stable across renders)
const itemsWithIds = useMemo(
  () => items.map(item => ({ ...item, _id: crypto.randomUUID() })),
  [items]
);

// NEVER: Random ID generated during render
items.map(item => <Row key={Math.random()} />) // Remounts every render!

9.2 Reconciliation and Component Design

// Principle: Keep the reconciler's job easy

// 1. Avoid changing element types conditionally
// BAD: type changes cause subtree destruction
{isSpecial ? <SpecialList /> : <NormalList />}

// GOOD: same type, different props
<List variant={isSpecial ? "special" : "normal"} />

// 2. Stable component references
// BAD: new component type every render
const Wrapper = styled.div`...`; // inside render!

// GOOD: defined outside render
const Wrapper = styled.div`...`; // module level

// 3. Flat children when possible
// Nested arrays add reconciliation complexity
Staff-Level Insight — Virtual DOM Criticism

The virtual DOM is often criticized as "overhead" compared to fine-grained reactivity (Solid, Svelte). This is partially true — diffing an entire subtree is inherently more work than tracking exactly which DOM nodes depend on which state. However, the virtual DOM provides a consistent programming model (render everything, let React figure out changes) that scales better for large teams. The React Compiler (Forget) aims to close the performance gap by automatically memoizing at compile time, reducing the amount of diffing needed at runtime.
