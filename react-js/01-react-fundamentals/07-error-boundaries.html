1. Error Boundaries Overview

Error boundaries are React components that catch JavaScript errors in their child component tree during rendering, lifecycle methods, and constructors. They prevent a single component's error from crashing the entire application.

Without error boundaries: ───────────────────────── Error in <Widget /> │ ▼ Entire app unmounts → white screen User sees nothing. No recovery possible. With error boundaries: ────────────────────── Error in <Widget /> │ ▼ ErrorBoundary catches error │ ▼ Shows fallback UI for Widget section Rest of app continues working 
// Basic error boundary implementation
class ErrorBoundary extends React.Component {
  state = { hasError: false, error: null };

  static getDerivedStateFromError(error) {
    return { hasError: true, error };
  }

  componentDidCatch(error, errorInfo) {
    logErrorToService(error, errorInfo.componentStack);
  }

  render() {
    if (this.state.hasError) {
      return <ErrorFallback error={this.state.error} />;
    }
    return this.props.children;
  }
}

2. componentDidCatch vs getDerivedStateFromError

These two methods serve different purposes and run at different phases of the React lifecycle.

Aspect getDerivedStateFromError componentDidCatch
Phase Render phase Commit phase
Timing Synchronous, during render After DOM has been updated
Purpose Update state to show fallback UI Side effects (logging, reporting)
Static? Yes (no this access) No (instance method)
Arguments (error) (error, errorInfo)
Can call setState? Returns new state Yes, but deprecated for this use
Side effects allowed? No (pure function) Yes (logging, API calls) 
// The errorInfo object contains the component stack trace
componentDidCatch(error, errorInfo) {
  // errorInfo.componentStack:
  // "    at Widget (Widget.jsx:15)
  //     at div
  //     at Dashboard (Dashboard.jsx:8)
  //     at ErrorBoundary (ErrorBoundary.jsx:5)
  //     at App (App.jsx:3)"

  // This is a REACT component stack, not a JS call stack.
  // It shows the component hierarchy, not the function calls.
  // Invaluable for debugging which component caused the error.
}
Why Both Methods?

getDerivedStateFromError runs during the render phase so the fallback UI can be shown in the same render pass (no flash of broken UI). componentDidCatch runs during the commit phase so it can safely perform side effects like error reporting. Using componentDidCatch to set state (showing fallback) is deprecated because it causes an extra render — the broken UI briefly appears before the fallback.

3. Why Class-Only (No Hook Equivalent)

Error boundaries can only be implemented as class components. There is no useErrorBoundary hook. The React team has explained why:

// The fundamental issue: hooks run DURING render.
// Error boundaries need to CATCH errors from child renders.
// A hook in a component can't catch errors from its own render.

// Conceptually, error boundaries need try/catch around
// the child component's render:

// What error boundaries do (simplified):
try {
  renderChildren(); // Call child component functions
} catch (error) {
  showFallback();   // Show error UI instead
}

// A hook inside a component runs AS PART of that component's render.
// It can't wrap the children's render in a try/catch.
// The class lifecycle methods (getDerivedStateFromError,
// componentDidCatch) are called by the RECONCILER when it
// catches an error from a child — they're not part of render.

// Workaround: use a library like react-error-boundary
import { ErrorBoundary } from 'react-error-boundary';

<ErrorBoundary fallback={<Error />}>
  <Widget />
</ErrorBoundary>

4. Error Propagation Through Fiber Tree

Error propagation (bubbles up like DOM events): App ├── ErrorBoundary_A │ ├── Header │ └── ErrorBoundary_B │ ├── Widget ← ERROR THROWN HERE │ └── Sidebar └── Footer 1. Widget throws during render 2. React walks UP the fiber tree (return pointers) 3. First, checks ErrorBoundary_B → HAS getDerivedStateFromError → CAUGHT 4. ErrorBoundary_B shows fallback (Widget + Sidebar replaced) 5. Header, Footer, ErrorBoundary_A unaffected If ErrorBoundary_B didn't exist: 3. Checks ErrorBoundary_A → CAUGHT 4. ErrorBoundary_A shows fallback (entire Header + Widget + Sidebar replaced) If NO error boundary exists: 3. Error reaches root → entire app unmounts → white screen 
// What React does internally when an error is thrown:

// 1. During render phase, a component throws
// 2. React catches the error in the work loop
// 3. Walks up the fiber tree via return pointers
// 4. For each ancestor fiber:
//    - If ClassComponent with getDerivedStateFromError:
//      → Schedule state update with error
//      → Mark fiber for re-render
//      → STOP propagation
//    - If not an error boundary: continue up
// 5. If no boundary found: schedule root unmount

// The error boundary re-renders with hasError: true
// Its children are the fallback UI, not the broken subtree
// The broken subtree is unmounted (cleanup effects run)

5. Recovery Patterns

5.1 Retry Pattern

class RetryBoundary extends React.Component {
  state = { hasError: false, error: null, retryCount: 0 };

  static getDerivedStateFromError(error) {
    return { hasError: true, error };
  }

  handleRetry = () => {
    this.setState(prev => ({
      hasError: false,
      error: null,
      retryCount: prev.retryCount + 1
    }));
  };

  render() {
    if (this.state.hasError) {
      return (
        <div>
          <p>Something went wrong.</p>
          <button onClick={this.handleRetry}>Retry</button>
        </div>
      );
    }
    return this.props.children;
  }
}

5.2 Fallback with Context

// Provide error info to fallback via render prop
class ErrorBoundary extends React.Component {
  state = { hasError: false, error: null };

  static getDerivedStateFromError(error) {
    return { hasError: true, error };
  }

  resetError = () => {
    this.setState({ hasError: false, error: null });
  };

  render() {
    if (this.state.hasError) {
      return this.props.fallback({
        error: this.state.error,
        reset: this.resetError
      });
    }
    return this.props.children;
  }
}

// Usage
<ErrorBoundary
  fallback={({ error, reset }) => (
    <div>
      <p>Error: {error.message}</p>
      <button onClick={reset}>Try Again</button>
    </div>
  )}
>
  <Widget />
</ErrorBoundary>

6. Reset Keys

The "reset key" pattern forces an error boundary to remount its children when a key value changes, clearing the error state automatically.

// react-error-boundary library implements this pattern
import { ErrorBoundary } from 'react-error-boundary';

function App() {
  const [selectedId, setSelectedId] = useState(1);

  return (
    <ErrorBoundary
      resetKeys={[selectedId]}
      FallbackComponent={ErrorFallback}
    >
      <UserProfile userId={selectedId} />
    </ErrorBoundary>
  );
}

// When selectedId changes:
// 1. ErrorBoundary compares resetKeys with previous
// 2. If different AND currently in error state:
//    → Reset error state (hasError = false)
//    → Children re-render with new props
// 3. If the new render also throws → show error again

// Implementation (simplified):
componentDidUpdate(prevProps) {
  if (this.state.hasError) {
    const keysChanged = !arraysEqual(
      prevProps.resetKeys, this.props.resetKeys
    );
    if (keysChanged) {
      this.setState({ hasError: false, error: null });
    }
  }
}

7. React 19: onCaughtError / onUncaughtError

// React 19 adds error callbacks to createRoot
const root = createRoot(document.getElementById('root'), {
  // Called when an error is caught by an error boundary
  onCaughtError(error, errorInfo) {
    // error: the thrown error
    // errorInfo.componentStack: React component stack
    // errorInfo.errorBoundary: the boundary that caught it
    reportToSentry(error, {
      componentStack: errorInfo.componentStack,
      boundary: errorInfo.errorBoundary
    });
  },

  // Called when an error is NOT caught (no boundary)
  onUncaughtError(error, errorInfo) {
    // This replaces the default "white screen" behavior
    showGlobalErrorUI(error);
    reportCriticalError(error, errorInfo.componentStack);
  },

  // Called for recoverable errors (hydration mismatches, etc.)
  onRecoverableError(error, errorInfo) {
    reportWarning(error, errorInfo.componentStack);
  }
});
React 19 Error Architecture

The createRoot error callbacks provide a centralized error handling layer that complements per-boundary error handling. Use onCaughtError for telemetry (every caught error gets reported), onUncaughtError for critical error UI (replace white screen with helpful error page), and onRecoverableError for monitoring hydration and rendering issues that React auto-recovers from.

8. Granularity Strategies

Error Boundary Granularity Spectrum: Too coarse: Too fine: ───────────────── ───────────────── <ErrorBoundary> <ErrorBoundary> <EntireApp /> ← any error <SingleButton /> </ErrorBoundary> kills everything </ErrorBoundary> × 1000 boundaries = overhead + noise Ideal: boundary per independent feature ─────────────────────────────────────── <App> <ErrorBoundary><Header /></ErrorBoundary> <ErrorBoundary><Sidebar /></ErrorBoundary> <ErrorBoundary><MainContent /></ErrorBoundary> <ErrorBoundary><Chat /></ErrorBoundary> </App>

8.1 Boundary Placement Guidelines

Boundary Level Catches Fallback
Root (App level) Uncaught errors from anywhere Full-page error with reload button
Route level Errors within a page/route Page-level error with navigation
Feature level Errors in independent features Feature-specific error with retry
Widget level Errors in third-party or risky widgets Placeholder or hidden widget

9. Error Boundaries vs try/catch in Event Handlers

Critical Distinction

Error boundaries do NOT catch errors in event handlers. Event handlers run outside of React's render cycle — they're just regular JavaScript callbacks. Use standard try/catch for event handler errors.

// Error boundaries catch:
// ✅ Errors during rendering (component function / render method)
// ✅ Errors in lifecycle methods (componentDidMount, etc.)
// ✅ Errors in constructors
// ✅ Errors in useEffect/useLayoutEffect (commit phase)

// Error boundaries do NOT catch:
// ❌ Event handlers (onClick, onChange, etc.)
// ❌ Asynchronous code (setTimeout, fetch callbacks)
// ❌ Server-side rendering
// ❌ Errors in the error boundary itself

// For event handlers, use try/catch:
function Button() {
  const [error, setError] = useState(null);

  const handleClick = async () => {
    try {
      await riskyOperation();
    } catch (err) {
      setError(err);
      reportError(err);
    }
  };

  if (error) return <p>Error: {error.message}</p>;
  return <button onClick={handleClick}>Click</button>;
}

10. Production Error Reporting

// Production-grade error boundary with reporting
class MonitoredErrorBoundary extends React.Component {
  state = { hasError: false, error: null, eventId: null };

  static getDerivedStateFromError(error) {
    return { hasError: true, error };
  }

  componentDidCatch(error, errorInfo) {
    const eventId = Sentry.captureException(error, {
      contexts: {
        react: { componentStack: errorInfo.componentStack }
      },
      tags: {
        boundary: this.props.name,
        feature: this.props.feature
      }
    });
    this.setState({ eventId });
  }

  render() {
    if (this.state.hasError) {
      return (
        <ErrorFallback
          error={this.state.error}
          eventId={this.state.eventId}
          onReset={() => this.setState({
            hasError: false, error: null
          })}
        />
      );
    }
    return this.props.children;
  }
}
Staff-Level Error Architecture

At L6, design a layered error handling strategy: (1) Root boundary for catastrophic failures with full-page error UI. (2) Route-level boundaries for page isolation. (3) Feature-level boundaries for independent widgets. (4) createRoot callbacks for centralized telemetry. (5) try/catch in event handlers with user-facing error states. (6) Global window.onerror and unhandledrejection as the last safety net. Each layer should report to your error monitoring service with enough context (component stack, user action, app state) to reproduce the issue.
