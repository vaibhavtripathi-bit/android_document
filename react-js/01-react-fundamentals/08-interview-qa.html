JSX & Compilation

Q1. What does JSX compile to, and why does it not produce DOM nodes? Senior

JSX compiles to function calls — either React.createElement() (classic runtime) or jsx()/jsxs() (automatic runtime). These functions return plain JavaScript objects called ReactElements, not DOM nodes. A ReactElement is an immutable description: { $$typeof: Symbol(react.element), type, props, key, ref }. React reads these descriptions during reconciliation and manages the actual DOM. This separation is what makes React renderer-agnostic — the same JSX can target DOM, Native, Canvas, or any custom renderer.

Q2. Explain the difference between the classic and automatic JSX runtimes. Senior

The classic runtime compiles JSX to React.createElement(), requiring React to be in scope in every file. Children are passed as variadic arguments. The automatic runtime (React 17+) compiles to jsx()/jsxs() imported from react/jsx-runtime, eliminating the need for a manual React import. Children are passed as a props.children property, and keys are a separate argument. The automatic runtime also uses jsxs() for statically-known children arrays, enabling React to skip certain reconciliation checks. In development, jsxDEV() includes source location and self reference for better error messages.

Q3. What is jsxs() and why does it exist separately from jsx()? L6

jsxs() is used when the compiler can statically determine that an element has multiple children. It signals to React that the children array is fixed in shape — its length won't change between renders. This allows React to skip dynamic list reconciliation (key matching, insertion/deletion detection) for that specific children array. jsx() is used for single child or dynamic children where the shape might change. The distinction is a compile-time optimization that reduces runtime work.

Q4. How does the JSX pragma work, and when would you change it? Senior

The JSX pragma tells the compiler which function to call for JSX elements. File-level: /** @jsx h */ (classic) or /** @jsxImportSource preact */ (automatic). Project-level: tsconfig.json's jsx and jsxImportSource options. You'd change it to use Preact (h), Emotion (@emotion/react for the css prop), or a custom renderer. The pragma makes JSX framework-agnostic — the same syntax works with any library that provides the expected function signature.

Q5. Why can't you use statements (if/for/switch) inside JSX expression slots? Senior

JSX expression slots ({}) accept JavaScript expressions, not statements. This is because JSX compiles to function call arguments: jsx("div", { children: EXPRESSION }). Function arguments must be expressions that evaluate to a value. Statements (if, for, switch) don't produce values — they control flow. Use ternaries, logical operators (&&, ||), or .map() instead. IIFEs work but are discouraged for readability.

Q6. What's the difference between <>...</> and <React.Fragment>? Senior

They're functionally identical — both create Fragment elements. The short syntax <></> cannot accept props, including key. When you need a keyed fragment (e.g., mapping a list where each item returns multiple elements), you must use <React.Fragment key={id}>. The key prop is the only prop Fragment accepts. Both compile to the same element type: Symbol.for('react.fragment').

Q7. Explain the source map chain for a JSX file in a production build. L6

A JSX file goes through multiple transformations, each generating a source map: (1) Babel/SWC transforms JSX to JS (sourcemap A), (2) bundler (webpack/Vite) combines modules (sourcemap B), (3) minifier (Terser/esbuild) compresses code (sourcemap C). Browser DevTools chain these maps: minified → bundle → compiled → original JSX. Each step must correctly chain maps or debugging breaks. The jsxDEV transform adds __source metadata (file, line, column) that React DevTools uses for "jump to source" — this is stripped in production builds.

Q8. Compare Babel, SWC, esbuild, and TypeScript as JSX compilers. L6

Babel (JS): Rich plugin ecosystem, slowest (~1x baseline), used by CRA and custom setups. SWC (Rust): 20-70x faster, Wasm plugin system, used by Next.js and Parcel. esbuild (Go): 50-100x faster, limited plugin API, used by Vite for dev builds. TypeScript (tsc): ~1x speed, full type checking, often used with jsx: "preserve" to let another tool handle JSX. All support both classic and automatic runtimes. The choice depends on build speed requirements, plugin needs, and ecosystem integration.

Q9. Why does {0 && <Component />} render "0" instead of nothing? Senior

JavaScript's && operator returns the first falsy value or the last value. When the left side is 0 (falsy), it returns 0 (not false). React renders numbers as text nodes, so 0 appears in the DOM. false, null, and undefined render nothing, but 0 and "" (empty string) render as text. The fix: use {count > 0 && <Component />} or {!!count && <Component />} to ensure the left side evaluates to a boolean.

Q10. What is the bundle size impact of development vs production JSX transforms? L6

Development transforms add ~30-40% more code per JSX element. jsxDEV() includes source location (fileName, lineNumber, columnNumber) and self reference as additional arguments. In a large app with thousands of JSX elements, this significantly inflates the development bundle. Production transforms use jsx()/jsxs() with no debug metadata. Additionally, React's development build includes PropTypes validation, key warnings, double-rendering in StrictMode, and detailed error messages — all stripped in production. Always ensure production builds use NODE_ENV=production and the production JSX runtime.

Component Model

Q11. How does React distinguish function components from class components? Senior

React checks Component.prototype.isReactComponent, a property set on React.Component.prototype. It doesn't use typeof (both are functions) or instanceof (breaks across multiple React copies or iframes). This duck-typing approach is simple and works across module boundaries. If the flag exists, React instantiates with new and calls .render(). Otherwise, it calls the function directly and processes hooks.

Q12. What is $$typeof and what security problem does it solve? L6

$$typeof: Symbol.for('react.element') prevents XSS via JSON injection. If a server returns user-controlled JSON that looks like a ReactElement (with type, props, dangerouslySetInnerHTML), React would render it as a real element. But JSON.parse() cannot produce Symbols — they're not a JSON type. So React rejects any object without the Symbol tag. For environments without Symbol support (IE11), React falls back to the magic number 0xeac7, which provides weaker protection. This is defense-in-depth — it doesn't protect against attackers who can execute arbitrary JS.

Q13. Why is defining a component inside another component's render catastrophic? L6

Each render creates a new function reference, which React sees as a different type. Since prevType !== nextType, React destroys the entire old subtree and mounts a fresh one — every single render. This means: all state is lost, all DOM nodes are recreated, all effects run cleanup and re-run, all child components remount. In a form, inputs lose focus and values. In a list, scroll position resets. The fix: define components at module level or, in rare cases, memoize with useMemo.

Q14. Explain how React.Children.toArray works and why it assigns keys. Senior

React.Children.toArray flattens nested children into a flat array and assigns stable keys based on nesting structure. Keys use a prefix encoding: .0, .1 for top-level, .2:0, .2:1 for nested arrays. This ensures each child has a unique, stable key for reconciliation even without explicit keys. It handles all children forms: single element, array, nested arrays, strings, numbers, null. It's useful for manipulating children (sorting, filtering) while preserving reconciliation stability.

Q15. How does React.memo work internally? What fiber tag does it use? L6

React.memo creates a wrapper element type: { $$typeof: Symbol.for('react.memo'), type: WrappedComponent, compare: customFn }. It gets fiber tag 14 (MemoComponent) with custom compare, or 15 (SimpleMemoComponent) without. During beginWork, React compares previous and next props using the custom compare function or shallow equality. If props are equal, React bails out — it skips calling the component function and reuses the previous render output. Note: memo prevents component execution, not element creation. The parent still creates the ReactElement; memo just prevents processing it.

Q16. What changed with ref handling in React 19? Senior

React 19 makes ref a regular prop for function components, eliminating the need for forwardRef. Previously, ref was stripped from props and handled specially by React — function components couldn't access it without forwardRef. Now you can write function Input({ ref, ...props }) directly. forwardRef still works but is no longer necessary. This simplifies the component model and reduces the wrapper layer in the fiber tree (no ForwardRef fiber tag needed).

Q17. Compare HOCs, render props, and hooks for code reuse. Senior

HOCs: Static composition (wrap at definition), add wrapper nodes to tree, prop collision risk, hard to type in TS, poor DevTools experience. Render props: Dynamic composition (at render time), add wrapper nodes, callback nesting ("callback hell"), good type safety. Hooks: Dynamic composition (in function body), no extra tree nodes, flat and composable, excellent type safety, inspectable in DevTools. Hooks won because they compose without affecting the component tree, have no prop collision, and align with React's functional programming model. The React Compiler further optimizes hooks-based code.

Q18. Why did React choose closures (hooks) over this-based state (classes)? L6

Class components use this.props and this.state, which are mutable references. In async callbacks (setTimeout, fetch), this.props might point to newer props than when the callback was created — the "stale closure" problem in reverse. Function components with hooks capture props and state values in closures at render time. Each render gets its own snapshot of values. This makes behavior more predictable: the callback always sees the values from the render that created it. Additionally, closures enable the React Compiler to analyze data flow statically, and functions are simpler to optimize (no instance allocation, no prototype chain).

Q19. What is the IndeterminateComponent fiber tag? L6

When React first encounters a component, it doesn't know if it's a function or class (both are functions in JS). It assigns fiber tag 2 (IndeterminateComponent). On first render, React calls the component and checks: if the result has a render method, it's a class (tag → 1); otherwise it's a function (tag → 0). This determination happens once per component type and is cached on the fiber. This is why React can't determine the component type from the element alone — it needs to actually call/instantiate it.

Q20. Why is cloneElement discouraged in modern React? Senior

cloneElement makes data flow implicit — parents secretly inject props into children without the children knowing. This breaks with React.memo (cloned elements always have new prop objects), is hard to type in TypeScript (the cloned element's props type is opaque), and makes debugging difficult (where did this prop come from?). React 19 documentation marks it as legacy. Alternatives: render props for explicit data passing, context for deep injection, composition patterns where parents render children with explicit props.

Fiber Architecture

Q21. What problem did Fiber solve that the Stack Reconciler couldn't? Senior

The Stack Reconciler processed the entire component tree synchronously in one uninterruptible call stack. For large trees, this blocked the main thread for hundreds of milliseconds, causing dropped frames and unresponsive UIs. Fiber makes rendering interruptible by representing the tree as heap-allocated linked-list nodes instead of recursive stack frames. React can pause after processing any fiber node, yield to the browser for input handling or painting, and resume later. This enables concurrent rendering, priority-based scheduling, and time-slicing.

Q22. Explain the double-buffering technique in React's fiber trees. L6

React maintains two fiber trees: current (what's on screen) and workInProgress (being built). They're connected via alternate pointers. During render, React builds the workInProgress tree by cloning and updating fibers from current. The user never sees a partially-built tree. When rendering completes, React atomically swaps them: root.current = finishedWork. The old current becomes the next workInProgress (fiber nodes are recycled, not garbage collected). This is analogous to double buffering in graphics — you draw to a back buffer and flip it to the screen when ready.

Q23. Why is the fiber tree a linked list instead of a tree with children arrays? L6

A linked list (child/sibling/return pointers) enables interruptible traversal. With a children array, traversal requires recursion — the "current position" is implicit in the call stack and can't be saved/restored. With a linked list, the current position is an explicit pointer to a fiber node stored in a variable. React can save this pointer, yield to the browser, and resume from exactly where it stopped. The three pointers (child → first child, sibling → next sibling, return → parent) enable depth-first traversal without recursion.

Q24. What is a "unit of work" in React's work loop? Senior

Each fiber node is one unit of work. The work loop processes one fiber at a time: call beginWork (process the fiber, return first child), then if no child, call completeWork (bubble up, create DOM nodes). After each unit, React checks shouldYield() — if the 5ms time slice is exceeded, it saves the current fiber pointer and yields. The next time slice resumes from that exact fiber. This granularity allows React to interleave rendering with browser tasks (input handling, painting) for smooth 60fps interaction.

Q25. Why did React abandon requestIdleCallback for scheduling? L6

The original Fiber proposal planned to use requestIdleCallback, but it had critical limitations: (1) Low frequency — only ~20 calls/sec (50ms intervals), too slow for smooth UI. (2) Inconsistent timing across browsers/devices. (3) Throttled when tab is backgrounded. (4) No priority support — all callbacks are equal. (5) Not implemented in Safari. React built its own scheduler using MessageChannel, which fires as a macrotask with near-zero delay (unlike setTimeout's 4ms minimum). React's scheduler implements 5ms time slices, priority lanes, task expiration, and starvation prevention — none of which requestIdleCallback supports.

Q26. Explain the lanes priority system and how updates are assigned lanes. L6

Lanes are a 31-bit bitmask priority system replacing the older expiration times. Key lanes: SyncLane (highest — click handlers, flushSync), InputContinuousLane (drag/scroll), DefaultLane (normal setState), TransitionLanes (startTransition — 16 lanes for concurrent transitions), IdleLane (lowest). Updates are assigned lanes based on their source: discrete events get SyncLane, continuous events get InputContinuousLane, transitions get TransitionLanes. Lanes combine with bitwise OR and check with bitwise AND. Multiple transition lanes allow React to process different transitions independently, enabling selective rendering.

Q27. What does the stateNode field contain for different fiber types? Senior

For HostComponent (div, span): the actual DOM element. For ClassComponent: the class instance (this). For FunctionComponent: null (no instance). For HostRoot: the FiberRoot object (container info). For HostText: the DOM text node. This field connects the abstract fiber tree to the concrete platform objects. It's how React knows which DOM node to update when a fiber's props change, and how class components access this.

Q28. How are hooks stored in the fiber and why must they be called in order? L6

Hooks are stored as a singly-linked list in fiber.memoizedState. Each hook call creates a node: { memoizedState, queue, next }. On re-render, React walks this list sequentially — hook call N reads node N. There are no names or keys; identity is purely positional. If you skip a hook conditionally (e.g., if (x) useState()), all subsequent hooks read the wrong node — useState reads useEffect's data, etc. This is why the Rules of Hooks exist: always call hooks at the top level, in the same order, every render.

Q29. What is the bailout optimization in beginWork? L6

During beginWork, React checks if a fiber needs work: are props the same reference? Is there pending state? Are there lanes in this fiber or its subtree (childLanes)? If nothing changed and no pending work exists, React bails out — it skips the entire subtree without calling the component function. This is the mechanism behind React.memo, useMemo, and the React Compiler's automatic memoization. The subtreeFlags and childLanes fields bubble up information about whether any descendant needs work, enabling React to skip entire branches efficiently.

Q30. Explain the fiber traversal order for beginWork and completeWork. L6

beginWork runs top-down (parent before children): it processes the fiber and returns the first child. completeWork runs bottom-up (children before parent): it creates DOM nodes and bubbles up effect flags. The traversal is: beginWork(parent) → beginWork(firstChild) → ... → beginWork(leaf) → completeWork(leaf) → if sibling exists, beginWork(sibling), else completeWork(parent). This depth-first traversal ensures DOM nodes are created bottom-up (children exist before parent tries to append them) while component functions are called top-down (parent determines which children to render).

Reconciliation & Diffing

Q31. Why is React's diffing O(n) instead of the optimal O(n³)? Senior

The optimal tree edit distance algorithm is O(n³), impractical for UI (1000 nodes = 1 billion operations). React uses two heuristics: (1) Different types produce different trees — if type changes, destroy and rebuild (no cross-type morphing). (2) Keys identify stable elements in lists. These reduce the problem to O(n) by comparing nodes at the same tree position and using key-based matching for lists. The heuristics hold for virtually all real UI updates. When violated, React does more work (unnecessary remounts) but never produces incorrect output.

Q32. What happens when an element's type changes at the same position? Senior

React destroys the entire old subtree and builds a new one. All child components unmount (state lost, effects cleaned up, DOM removed). New components mount fresh (initial state, effects run, new DOM created). This applies to both host elements (div → section) and component elements (CounterA → CounterB). The destruction cascades through the entire subtree — even if children are identical, they're recreated. This is why conditional rendering with different types (isSpecial ? <A /> : <B />) should be avoided when state preservation matters.

Q33. Explain React's two-pass list reconciliation algorithm. L6

Pass 1 (linear scan): Walk old and new lists from the start, comparing keys at each position. If keys match, reuse the fiber. If keys differ, break out of the loop. This handles the common case where most items don't move. Pass 2 (map lookup): Build a Map of remaining old children by key. Walk remaining new children, looking up each in the map. Found → reuse fiber (mark for move if needed). Not found → create new fiber. After pass 2, any remaining entries in the map are deletions. Move detection uses a lastPlacedIndex — if an old item's index is less than lastPlacedIndex, it needs to move.

Q34. Why is using array index as key problematic? Senior

Index keys are equivalent to no keys for any operation that changes list order or length. When you delete the first item, all subsequent items shift indices — React sees key=0 change from "Alice" to "Bob" and updates the DOM content instead of removing Alice's node. This causes: (1) unnecessary DOM updates for every shifted item, (2) component state associated with wrong data (input values, scroll positions), (3) animations restarting on wrong elements. Index keys are only safe for static, never-reordered lists with stateless items.

Q35. What is the lastPlacedIndex optimization in list reconciliation? L6

lastPlacedIndex tracks the highest old index of a reused fiber that didn't need to move. For each reused fiber from the map: if its old index ≥ lastPlacedIndex, it's already in the right relative order (no move needed), and lastPlacedIndex updates. If old index < lastPlacedIndex, the item needs to move forward. This is a greedy algorithm — it minimizes moves for the common case (appends, small changes) but isn't optimal for all patterns. Moving the last item to the front causes N-1 moves; moving the first to the end causes only 1 move.

Q36. How does the key prop force state reset even with the same component type? Senior

When the key changes, React treats the element as a completely different instance — even if the type is the same. It unmounts the old fiber (destroying state, running cleanup) and mounts a new one (fresh state, fresh effects). This is because key is part of the element's identity during reconciliation. Common use: <Profile key={userId} /> ensures a clean remount when switching users, preventing stale state from the previous user. Without the key, React would reuse the fiber and only update props, preserving old state.

Q37. Compare React's reconciliation with Inferno's LIS-based approach. L6

React uses a greedy lastPlacedIndex algorithm — simple but not optimal for all reorderings. Inferno uses the Longest Increasing Subsequence (LIS) algorithm to find the minimum number of DOM moves. For the case [A,B,C,D] → [D,A,B,C], React does 3 moves (A,B,C after D), while LIS finds 1 move (D before A). However, LIS has higher constant overhead (O(n log n) for LIS computation). React chose simplicity and predictability over optimality — the common case (appends, small changes) is handled efficiently, and the worst case (full reversal) is rare in real UIs.

Q38. What is the virtual DOM and is it actually faster than direct DOM manipulation? L6

The virtual DOM is a lightweight JavaScript representation of the UI tree. It's not inherently faster than optimal direct DOM manipulation — it adds overhead (creating elements, diffing, applying patches). Its value is in providing a declarative programming model: describe what the UI should look like, and React figures out the minimal DOM changes. This is easier to reason about than imperative DOM manipulation, scales better for teams, and prevents common bugs (forgetting to clean up, inconsistent state). Fine-grained reactive systems (Solid, Svelte) skip the diffing overhead by tracking exactly which DOM nodes depend on which state — they're faster for updates but require a different programming model.

Q39. How does React handle mixed children types (text, elements, arrays)? Senior

React normalizes children during reconciliation. Strings and numbers become HostText fibers (DOM text nodes). ReactElements become fibers based on their type. Arrays are flattened. null, undefined, and boolean are ignored. Adjacent text nodes may be merged. During reconciliation, React matches children by position and key. Text nodes have no key, so they're always matched by position. This is why mixing text and elements in a list without keys can cause unexpected behavior — text nodes shift when elements are inserted/removed.

Q40. Design a key strategy for a complex list with filtering, sorting, and pagination. L6

Use stable, unique identifiers from the data (database IDs, UUIDs). Never use array index — filtering/sorting changes indices. For composite items without a single ID, create a composite key: `${type}-${name}-${date}`. For paginated lists, the same item might appear on different pages — the key should be the item's ID, not its page position. For virtualized lists (react-window), keys must be stable across scroll positions. If items can be duplicated (e.g., same product in different categories), include the category in the key. Generate stable IDs at data fetch time if the API doesn't provide them, using useMemo to prevent regeneration on re-render.

Context System

Q41. How does React propagate context values through the fiber tree? L6

React uses a stack-based approach. When entering a Provider during render, it pushes the new value onto an internal context stack and sets context._currentValue. All descendants read this value directly (no tree walking). When leaving the Provider subtree, it pops the value. Nested Providers of the same context push/pop correctly, so descendants see the nearest Provider's value. When the value changes, React calls propagateContextChange, which walks the Provider's subtree marking all consumer fibers with lanes, ensuring they re-render even if memoized ancestors would normally bail out.

Q42. Why can't React.memo prevent re-renders from context changes? L6

When a context value changes, React's propagateContextChange walks the fiber tree and directly marks consumer fibers with update lanes — bypassing the normal parent-to-child render flow. Even if a memoized ancestor would bail out (props unchanged), the consumer fiber is marked independently. During the next render, React sees the consumer has pending lanes and processes it regardless of ancestor bailouts. This is by design — context is a mechanism for "reaching through" the component tree, so it must bypass memoization boundaries.

Q43. What is the object identity trap with context values? Senior

React compares context values with Object.is(). If the Provider's value is an object literal created during render (value={{ user, theme }}), it's a new reference every render, defeating the bail-out optimization. Every consumer re-renders even if the actual data hasn't changed. Fix: memoize the value with useMemo, or split into separate contexts for independent values. For dispatch functions from useReducer, they're already stable references — put them in a separate context from state.

Q44. Describe the split context pattern and when to use it. Senior

Instead of one context with multiple values ({ theme, locale, user }), split into focused contexts: ThemeContext, LocaleContext, UserContext. Each consumer only subscribes to the context it needs. When user changes, only UserContext consumers re-render — ThemeContext consumers are unaffected. Further split: separate state from dispatch (StateContext + DispatchContext). Components that only dispatch actions (buttons, forms) subscribe to DispatchContext (stable reference) and never re-render from state changes.

Q45. How does use(Context) in React 19 differ from useContext? Senior

use(Context) can be called conditionally and inside loops — it doesn't follow the Rules of Hooks. This is possible because use() is not a hook in the traditional sense; it's a special API that React handles differently during rendering. It also works with Promises (use(promise) suspends until resolved). The context reading behavior is identical to useContext — same propagation, same re-render semantics. The flexibility of conditional calls enables patterns like reading context only in certain code paths without restructuring components.

Q46. When should you use context vs an external state manager? L6

Context: Best for low-frequency, scoped state — theme, auth, locale, dependency injection. Zero bundle cost, built-in SSR isolation, simple API. Limitation: no selectors, all consumers re-render on any value change. External store (Zustand, Redux, Jotai): Best for high-frequency, widely-consumed state — UI state, data cache, form state. Provides selectors (fine-grained subscriptions), middleware, DevTools, time-travel debugging. At scale, many apps use both: context for DI and configuration, external store for application state. The decision should be based on update frequency × number of consumers.

Q47. Explain the children-as-props pattern for optimizing context providers. L6

When a Provider component manages state, its re-renders would normally re-render all children (new JSX = new elements). The fix: accept children as props. function ThemeProvider({ children }) { const [theme] = useState('light'); return <ThemeContext.Provider value={theme}>{children}</ThemeContext.Provider>; }. The children prop is created by the parent of ThemeProvider — it's the same reference across ThemeProvider's re-renders. React sees the same children elements and bails out of re-rendering the subtree. Only actual context consumers re-render. This is a zero-cost optimization that should be the default pattern for all context providers.

Q48. How would you implement a context selector without external libraries? L6

React doesn't natively support context selectors, but you can approximate it: (1) Split contexts so each contains only related data. (2) Use useSyncExternalStore with a custom store that uses context for initialization but manages subscriptions externally. (3) Wrap consumers in React.memo with a parent that reads context and passes only selected values as props. (4) Use the use-context-selector library, which internally subscribes to context changes, runs the selector, and only triggers re-render if the selected value changed (using useSyncExternalStore). The React team has discussed adding native selector support but hasn't shipped it — the React Compiler may make it unnecessary by automatically memoizing.

Error Boundaries

Q49. Why can't error boundaries be implemented with hooks? L6

Error boundaries need to catch errors from child component renders. Hooks run as part of the component's own render — they can't wrap children's execution in a try/catch. The class lifecycle methods (getDerivedStateFromError, componentDidCatch) are called by the reconciler when it catches an error propagating up from a child fiber. This is a fundamentally different execution model than hooks. The React team could theoretically add a useErrorBoundary hook, but it would need special reconciler support — it's not a simple hook implementation. For now, use the react-error-boundary library which wraps a class component with a hooks-friendly API.

Q50. What's the timing difference between getDerivedStateFromError and componentDidCatch? Senior

getDerivedStateFromError runs during the render phase — synchronously, before the DOM is updated. It returns new state to show the fallback UI in the same render pass (no flash of broken UI). componentDidCatch runs during the commit phase — after the DOM has been updated with the fallback. It's safe for side effects (error reporting, analytics). Using componentDidCatch to setState (showing fallback) is deprecated because it causes an extra render — the broken UI briefly appears before the fallback replaces it.

Q51. What errors do error boundaries NOT catch? Senior

Error boundaries don't catch: (1) Event handler errors — they run outside React's render cycle, use try/catch. (2) Async code — setTimeout, fetch callbacks, async/await in event handlers. (3) Server-side rendering errors. (4) Errors thrown in the error boundary itself (propagates to next boundary up). (5) Errors in the root component with no boundary above it. For event handlers, manage error state with useState. For async operations, use try/catch with error state or Suspense error handling. React 19's onUncaughtError in createRoot catches errors that escape all boundaries.

Q52. Explain the reset keys pattern for error boundary recovery. Senior

Reset keys are values that, when changed, automatically clear the error boundary's error state. Implementation: in componentDidUpdate, compare current resetKeys prop with previous. If different and currently in error state, reset to { hasError: false }. This enables automatic recovery when the context changes — e.g., navigating to a different page, selecting a different item. The react-error-boundary library provides this out of the box with the resetKeys prop. It's more robust than manual retry buttons because it ties recovery to meaningful state changes.

Q53. How do React 19's createRoot error callbacks change error handling architecture? L6

React 19 adds three callbacks to createRoot: onCaughtError (error caught by a boundary — for telemetry), onUncaughtError (no boundary caught it — for global error UI), onRecoverableError (hydration mismatches, auto-recovered errors — for monitoring). This creates a centralized error handling layer complementing per-boundary handling. Architecture: boundaries handle local recovery (retry, fallback UI), createRoot callbacks handle global concerns (error reporting, analytics, global error page). This replaces the need for a root-level error boundary solely for reporting purposes.

Q54. Design an error boundary strategy for a large-scale application. L6

Layered approach: (1) Root boundary: catches catastrophic failures, shows full-page error with reload button and support link. (2) Route boundaries: per-route isolation, shows page-level error with navigation to other pages. (3) Feature boundaries: around independent features (chat widget, notification panel), shows feature-specific fallback with retry. (4) Widget boundaries: around third-party or risky components, silently degrades. (5) createRoot callbacks: centralized telemetry — every error (caught or not) reports to Sentry/DataDog with component stack, user context, and app state. (6) Event handler try/catch: per-interaction error handling with user-facing error states. Each layer should have increasing specificity in its error UI and recovery options.

Q55. How does error propagation work through the fiber tree? L6

When a component throws during render, React catches it in the work loop. It then walks up the fiber tree via return pointers, checking each ancestor. For each ClassComponent fiber, it checks if getDerivedStateFromError or componentDidCatch is defined. The first boundary found gets a state update scheduled with the error. React then unwinds the work — the broken subtree's fibers are marked for deletion (cleanup effects will run during commit). The boundary re-renders with error state, producing fallback UI. If no boundary is found, the error reaches the root and the entire tree unmounts. The component stack in errorInfo is built by walking the fiber tree, providing a React-specific stack trace.
