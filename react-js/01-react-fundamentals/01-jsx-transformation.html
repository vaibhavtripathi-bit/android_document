JSX Transformation

JSX is not HTML — it's a syntactic extension that compiles to plain JavaScript function calls. Understanding the transformation pipeline from Babel/SWC through React.createElement to the automatic JSX runtime is foundational knowledge for Staff-level engineers.

1. What JSX Actually Is

JSX (JavaScript XML) is a syntax extension defined in a draft specification that allows XML-like syntax within JavaScript. It is not part of the ECMAScript standard. JSX produces plain JavaScript objects — not DOM nodes, not HTML strings, not template literals.

Architect Insight

JSX is framework-agnostic by design. The same JSX syntax can target React DOM, React Native, Preact, Solid, or any custom renderer. The transformation layer is what binds JSX to a specific framework — this is why pragma configuration exists.

When you write JSX, a compiler (Babel, SWC, TypeScript, or esbuild) transforms it into function calls before the browser ever sees it. The browser never parses JSX — it only executes the compiled output.

Source Code (JSX) Compiler Runtime Output ───────────────── ────────────────── ───────────────────── │ Babel / SWC / │ <App name="x" /> │ TypeScript / │→ jsx(App, { name: "x" }) │ esbuild │ └─────────────────┘ ↓ ReactElement { $$typeof: Symbol(react.element), type: App, props: { name: "x" }, key: null, ref: null }

2. Classic Transform: React.createElement

Before React 17, all JSX compiled to React.createElement() calls. This is the "classic" runtime and is still supported.

2.1 Basic Transformation

// JSX input
const element = <div className="container">
  <h1>Hello</h1>
  <p>World</p>
</div>;

// Classic compiled output
const element = React.createElement(
  "div",
  { className: "container" },
  React.createElement("h1", null, "Hello"),
  React.createElement("p", null, "World")
);

2.2 createElement Signature

// React.createElement(type, props, ...children)
//
// type:     string ("div") or component (App) or Fragment
// props:    object of attributes/props, or null
// children: remaining arguments become props.children

React.createElement(
  type,       // "div" | MyComponent | React.Fragment
  props,      // { className: "x", onClick: fn } | null
  child1,     // These get collected into props.children
  child2,
  childN
);
Critical Implication

The classic transform requires React to be in scope in every file that uses JSX. This is why pre-React 17 code always had import React from 'react' at the top — even if React wasn't directly referenced. Linters would flag its removal as an error.

2.3 Component vs Host Element

// Host element: type is a string
<div>Hello</div>
→ React.createElement("div", null, "Hello")

// Component element: type is a function/class reference
<App name="x" />
→ React.createElement(App, { name: "x" })

// The type field is what React uses during reconciliation
// to decide whether to reuse or destroy a fiber

3. Automatic JSX Runtime (React 17+)

React 17 introduced the automatic JSX runtime — a new compilation target that eliminates the need for React to be in scope. The compiler auto-inserts imports from react/jsx-runtime.

3.1 Three Runtime Functions

Function Module Purpose
jsx(type, props, key) react/jsx-runtime Single child or no children (production)
jsxs(type, props, key) react/jsx-runtime Multiple static children (production)
jsxDEV(type, props, key, isStatic, source, self) react/jsx-dev-runtime Development mode with debug info

3.2 Automatic Transform Output

// JSX input
const element = <div className="container">
  <h1>Hello</h1>
  <p>World</p>
</div>;

// Automatic runtime compiled output (production)
import { jsxs as _jsxs, jsx as _jsx } from "react/jsx-runtime";

const element = _jsxs("div", {
  className: "container",
  children: [
    _jsx("h1", { children: "Hello" }),
    _jsx("p", { children: "World" })
  ]
});

3.3 Key Differences from Classic

Aspect Classic (createElement) Automatic (jsx/jsxs)
React import required Yes — must be in scope No — compiler auto-inserts
Children passing Variadic arguments props.children property
Key handling Part of props object Separate third argument
Static children optimization None jsxs for static, jsx for dynamic
Dev info __source, __self in props Separate jsxDEV parameters
Bundle impact Larger (full React import) Smaller (tree-shakeable imports)
Why jsxs Exists

jsxs signals to React that the children array is statically known at compile time and will never change shape between renders. This enables React to skip certain reconciliation checks — it knows the array length is fixed, so it doesn't need to handle dynamic insertions/deletions within that specific children array.

4. Babel vs SWC Pipeline

4.1 Babel Transform

Babel uses @babel/plugin-transform-react-jsx (or the preset @babel/preset-react). The plugin parses JSX AST nodes and replaces them with function call expressions.

// babel.config.json
{
  "presets": [
    ["@babel/preset-react", {
      "runtime": "automatic",  // "automatic" | "classic"
      "importSource": "react", // can be "preact" for Preact compat
      "development": true       // enables jsxDEV with source info
    }]
  ]
}
Babel Pipeline for JSX: Source (.jsx/.tsx) │ ▼ ┌──────────────────┐ │ @babel/parser │ Parse into AST (JSXElement, JSXFragment nodes) └──────┬───────────┘ ▼ ┌──────────────────────────────────┐ │ @babel/plugin-transform-react- │ Replace JSXElement → CallExpression │ jsx │ (jsx/jsxs/createElement) └──────┬───────────────────────────┘ ▼ ┌──────────────────┐ │ @babel/generator │ AST → JavaScript string + source map └──────────────────┘

4.2 SWC Transform

SWC (Speedy Web Compiler) is a Rust-based alternative that performs the same transformation 20-70x faster than Babel. Used by Next.js, Vite (via plugin), and Parcel.

// .swcrc
{
  "jsc": {
    "transform": {
      "react": {
        "runtime": "automatic",
        "importSource": "react",
        "development": false,
        "refresh": true  // React Fast Refresh support
      }
    }
  }
}

5. Fragment Desugaring

Fragments allow returning multiple elements without a wrapper DOM node. The short syntax <>...</> desugars to React.Fragment.

// Short syntax
<>
  <h1>Title</h1>
  <p>Body</p>
</>

// Classic transform
React.createElement(React.Fragment, null,
  React.createElement("h1", null, "Title"),
  React.createElement("p", null, "Body")
);

// Automatic runtime transform
import { Fragment as _Fragment, jsxs as _jsxs, jsx as _jsx }
  from "react/jsx-runtime";

_jsxs(_Fragment, {
  children: [
    _jsx("h1", { children: "Title" }),
    _jsx("p", { children: "Body" })
  ]
});
Keyed Fragments

The short syntax <></> cannot accept a key prop. When you need keys (e.g., mapping a list), use the explicit <React.Fragment key={id}> syntax. This is the only prop Fragment accepts.

6. Expression Slots and Limitations

JSX uses curly braces {} as expression slots. Understanding what can and cannot go inside them is critical.

6.1 What JSX Expressions Evaluate To

// Expressions (valid in JSX {})
{42}                      // → "42" (rendered as text)
{"hello"}                // → "hello"
{true}                    // → nothing (not rendered)
{false}                   // → nothing
{null}                    // → nothing
{undefined}               // → nothing
{["a", "b"]}             // → "ab" (array items rendered)
{condition && <X />}     // → X or nothing
{condition ? <A /> : <B />}  // → A or B

// GOTCHA: 0 is falsy but renders!
{count && <Items />}     // renders "0" when count is 0!
{count > 0 && <Items />} // correct guard

6.2 Statements vs Expressions

// INVALID: statements cannot go in JSX expression slots
<div>{if (x) { return "y" }}</div>  // SyntaxError
<div>{for (let i...) { }}</div>   // SyntaxError
<div>{switch(x) { }}</div>       // SyntaxError

// VALID: use expressions instead
<div>{x ? "y" : "z"}</div>       // ternary expression
<div>{items.map(i => ...)}</div>  // map returns array
<div>{(() => { ... })()}</div>   // IIFE (avoid)

7. Development vs Production Transforms

The JSX compiler produces different output in development and production modes. This has significant implications for bundle size, performance, and debugging.

7.1 Development Mode (jsxDEV)

// Development output (automatic runtime)
import { jsxDEV as _jsxDEV } from "react/jsx-dev-runtime";

const element = _jsxDEV(
  "div",                        // type
  { className: "container" },   // props
  undefined,                     // key
  false,                         // isStaticChildren
  {                              // __source (file/line/col)
    fileName: "/src/App.jsx",
    lineNumber: 5,
    columnNumber: 10
  },
  this                           // __self (component instance)
);

7.2 What Development Mode Adds

  • __source: File path, line number, column number — powers React DevTools "jump to source"
  • __self: Reference to the enclosing component — enables better error messages
  • Key warnings: React validates key props on list children and warns about missing keys
  • Prop type checking: PropTypes validation runs (if defined)
  • Double rendering: StrictMode invokes render functions twice to detect side effects

7.3 Production Mode Stripping

// Production output — no __source, no __self, no jsxDEV
import { jsx as _jsx } from "react/jsx-runtime";

const element = _jsx("div", { className: "container" });
Architect Insight — Bundle Size Impact

The development transform adds ~30-40% more code per JSX element (source metadata, self references). In a large app with thousands of JSX elements, this means the development bundle can be significantly larger. This is why production builds must always use the production transform — not just for React's internal checks, but for the JSX compilation output itself.

8. Source Maps and Debugging

Source maps bridge the gap between compiled output and original JSX source. They're essential for debugging because the code running in the browser looks nothing like what you wrote.

8.1 Source Map Chain

Original JSX (.jsx) │ ▼ [Babel/SWC transform — generates sourcemap A] Compiled JS (.js) │ ▼ [Bundler (webpack/Vite) — generates sourcemap B] Bundle (.js) │ ▼ [Minifier (Terser/esbuild) — generates sourcemap C] Minified Bundle (.js) Browser DevTools: minified.js → sourcemap C → bundle.js → sourcemap B → compiled.js → sourcemap A → original.jsx Each step must chain source maps correctly or debugging breaks.

8.2 Debugging JSX in DevTools

// With correct source maps, you can:
// 1. Set breakpoints in original .jsx files
// 2. See JSX in the call stack (not createElement calls)
// 3. Inspect original variable names (not minified)
// 4. React DevTools uses __source to link components to files

// Vite config for source maps
export default defineConfig({
  build: {
    sourcemap: true,  // generates .map files
  }
});

9. Pragma Configuration

The JSX pragma tells the compiler which function to call for JSX elements. This is how you can use JSX with non-React libraries.

9.1 File-Level Pragma

/** @jsx h */
/** @jsxFrag Fragment */
import { h, Fragment } from "preact";

// Now JSX in this file compiles to h() instead of React.createElement()
<div>Hello</div>  →  h("div", null, "Hello")

// For automatic runtime:
/** @jsxImportSource preact */
// Imports from "preact/jsx-runtime" instead of "react/jsx-runtime"

9.2 Project-Level Configuration

// tsconfig.json
{
  "compilerOptions": {
    "jsx": "react-jsx",        // automatic runtime
    // "jsx": "react-jsxdev",  // automatic runtime (dev)
    // "jsx": "react",         // classic: React.createElement
    // "jsx": "preserve",      // don't transform (let Babel handle)
    "jsxImportSource": "react" // or "preact", "@emotion/react", etc.
  }
}
Emotion CSS-in-JS Pragma

Libraries like Emotion use the pragma to intercept JSX and inject CSS processing: /** @jsxImportSource @emotion/react */. This makes the css prop work on any element. The Emotion jsx function wraps React's jsx, adding style processing before creating the element.

10. Architect Considerations

10.1 Migration Strategy: Classic → Automatic

// React provides a codemod for migration:
npx react-codemod update-react-imports

// This codemod:
// 1. Removes `import React from 'react'` where React isn't used directly
// 2. Adds named imports for hooks: `import { useState } from 'react'`
// 3. Keeps React import where React.* is used (React.memo, etc.)

10.2 Performance Implications

Concern Impact Mitigation
JSX creates objects every render GC pressure in hot paths React.memo, useMemo for expensive subtrees
Deep JSX nesting More createElement calls, deeper fiber tree Flatten component hierarchy where possible
Inline JSX in callbacks New elements every render, breaks memoization Extract to named components or useMemo
Spread props {...props} Object spread creates new object each render Explicit props when memoization matters

10.3 JSX Returns Objects, Not DOM

// This is the most important mental model shift:
const el = <div>Hello</div>;

// el is NOT a DOM node. It's a plain object:
// {
//   $$typeof: Symbol(react.element),
//   type: "div",
//   props: { children: "Hello" },
//   key: null,
//   ref: null,
//   _owner: null
// }

// You cannot do el.style.color = "red"
// You cannot do el.appendChild(...)
// It's a description of what the UI should look like,
// not the UI itself. React reads these descriptions
// and manages the actual DOM.
Staff-Level Mental Model

JSX is a declarative description language that compiles to object construction calls. These objects form a virtual tree that React diffs against the previous tree to compute minimal DOM mutations. Every layer of this pipeline is configurable: the compiler (Babel/SWC), the runtime (react/jsx-runtime), the reconciler (react-reconciler), and the renderer (react-dom/react-native). This separation of concerns is what makes React a platform, not just a library.
