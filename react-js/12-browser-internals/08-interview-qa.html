Interview Q&A — Browser Internals

Rendering Engine

Q1: Walk through the browser rendering pipeline from HTML bytes to pixels on screen.
The pipeline has six stages: (1) Parse — HTML bytes are tokenized and parsed into a DOM tree; CSS is parsed into a CSSOM tree. The parser is incremental (processes bytes as they arrive). (2) Style — DOM + CSSOM are combined to compute styles for every visible element, producing the render tree. Elements with display: none are excluded. (3) Layout — Calculate the exact position and size (box model) of every element in the render tree. (4) Paint — Convert the layout tree into paint instructions (draw commands) per compositor layer. (5) Composite — Combine painted layers in the correct order, applying transforms and opacity. This happens on the GPU. (6) Display — Send the composited frame to the screen.
Q2: What is layout thrashing and how do you prevent it?
Layout thrashing occurs when JavaScript interleaves DOM reads and writes in a loop. Each read (e.g., offsetWidth) forces the browser to synchronously calculate layout to return an accurate value. If followed by a write (e.g., style.width = ...), the layout is invalidated, and the next read forces another layout calculation. Prevention: batch all reads first, then batch all writes. Use requestAnimationFrame for write operations. Libraries like FastDOM automate this pattern. In React, this is less common because React batches DOM mutations, but it can still occur in useLayoutEffect or refs that directly manipulate the DOM.
Q3: Why should CSS animations use transform/opacity instead of top/left/width?
Properties like top, left, width, and height trigger layout (reflow), which recalculates geometry for the element and potentially its entire subtree, followed by paint and composite. This happens on the main thread and is expensive. transform and opacity only trigger composite — the cheapest operation. The element is already painted on its own compositor layer, and the GPU simply transforms or fades it. This runs on the compositor thread, independent of the main thread, so even if JavaScript is busy, the animation stays smooth at 60fps.
Q4: Explain will-change, CSS containment, and content-visibility.
will-change hints to the browser that a property will animate, prompting it to create a compositor layer in advance (avoiding first-frame jank). Remove it after animation to free GPU memory. contain tells the browser that an element's internals don't affect outside layout/paint, enabling optimization by skipping that subtree during parent recalculations. Values: layout, paint, style, size, strict (all). content-visibility: auto skips rendering for off-screen elements entirely — the browser doesn't layout or paint them until they're near the viewport. Combined with contain-intrinsic-size for placeholder dimensions, it can reduce initial render time by 50%+ on long pages.
Q5: What is the critical rendering path and how do you optimize it?
The critical rendering path is the minimum set of steps to render the first pixel: HTML → DOM, CSS → CSSOM, DOM + CSSOM → Render Tree → Layout → Paint. Optimization: (1) Inline critical CSS (above-the-fold styles) to eliminate a render-blocking request. (2) Defer non-critical CSS with media="print" trick or rel="preload". (3) Use defer/async on scripts to avoid parser blocking. (4) Preload critical resources (fonts, hero images) with <link rel="preload">. (5) Preconnect to critical origins. (6) Minimize the number of critical resources and their size. The goal: reduce the time between first byte and first paint.
Q6: How does the preload scanner work and why is it important?
When the HTML parser encounters a <script> tag (without async/defer), it blocks to download and execute the script. The preload scanner is a secondary, lightweight parser that continues scanning the HTML ahead of the blocked main parser, discovering resources (images, stylesheets, scripts) that can be fetched in parallel. Without it, resources after a blocking script would have to wait for the script to finish before being discovered. This is why the preload scanner is one of the most impactful browser optimizations — it enables parallel resource loading despite parser-blocking scripts.

V8 Engine

Q7: Describe V8's compilation pipeline from source code to optimized machine code.
V8's multi-tier pipeline: (1) Scanner/Parser — tokenizes and parses JavaScript into an AST. Uses lazy parsing for functions not immediately called. (2) Ignition — bytecode interpreter. Compiles AST to compact bytecode (~25-50% size of machine code). Collects type feedback in feedback vectors. (3) Sparkplug — non-optimizing baseline compiler. Quickly compiles bytecode to machine code without optimization, improving warm-up performance. (4) Maglev — mid-tier optimizing compiler (added 2023). Uses type feedback for moderate optimization, faster compilation than TurboFan. (5) TurboFan — full optimizing compiler. Speculative optimization based on type feedback, produces highly optimized machine code. If type assumptions are violated, deoptimization falls back to Ignition bytecode.
Q8: What are hidden classes (shapes/maps) and why do they matter for performance?
V8 assigns a "hidden class" (internally called a Map) to every object, describing its property names and their memory offsets. Objects with the same properties added in the same order share the same hidden class. This enables V8 to use fixed-offset property access (like C structs) instead of hash table lookups. Performance implications: (1) Create objects with consistent shapes — always add properties in the same order. (2) Don't delete properties (changes the hidden class). (3) Use classes (V8 optimizes class instances). (4) Initialize all properties in constructors (even to undefined). Inconsistent shapes lead to polymorphic/megamorphic inline caches, which are significantly slower.
Q9: Explain inline caches and the monomorphic → polymorphic → megamorphic progression.
Inline caches (ICs) speed up property access by caching the hidden class and property offset from previous calls. Monomorphic (1 shape): fastest — direct memory access using cached offset. Polymorphic (2-4 shapes): fast — checks a small list of cached shapes. Megamorphic (5+ shapes): slow — falls back to generic hash table lookup. In React, this matters for hot paths: if a function processes objects with many different shapes (e.g., polymorphic props), the IC becomes megamorphic and every property access is slow. Keep hot-path object shapes consistent.
Q10: How does V8's garbage collector work? Explain generational collection.
V8 uses generational GC based on the observation that most objects die young. Young generation (1-8 MB): uses Scavenge (Cheney's semi-space copying). Two semi-spaces — objects allocated in "from", live objects copied to "to", spaces swapped. Very fast (~1-5ms). Objects surviving 2 scavenges are promoted to old generation. Old generation (up to ~1.5 GB): uses Mark-Sweep-Compact. Mark reachable objects from roots, sweep unmarked, compact to defragment. Slower (~10-100ms). Orinoco optimizations: parallel (multiple GC threads), incremental (marking interleaved with JS), concurrent (marking on background threads while JS runs), lazy sweeping.
Q11: What causes deoptimization in V8 and how do you avoid it?
Deoptimization occurs when TurboFan's speculative assumptions are violated at runtime. Common causes: (1) Type instability — a function receives different types than what was profiled (e.g., numbers then strings). (2) Shape changes — objects with different hidden classes passed to the same function. (3) Deleting properties — changes the object's hidden class. (4) Arguments object — using arguments can prevent optimization. (5) eval/with — dynamic scope prevents static analysis. Prevention: keep types consistent, use TypeScript (helps humans, not V8 directly), initialize all properties, avoid polymorphic hot paths, use rest parameters instead of arguments.

Event Loop

Q12: Explain the difference between macrotasks and microtasks with execution order.
The event loop processes one macrotask, then drains the entire microtask queue, then optionally renders, then processes the next macrotask. Macrotasks: setTimeout, setInterval, MessageChannel, I/O, UI events, postMessage. Microtasks: Promise.then/catch/finally, queueMicrotask, MutationObserver. Key difference: microtasks queued during microtask processing are executed in the same cycle — the queue drains completely before any macrotask or rendering. This means infinite microtasks freeze the page, while infinite setTimeout(fn, 0) doesn't (browser renders between macrotasks). React uses this: state updates in the same synchronous block are batched because they're processed before the next macrotask.
Q13: When does requestAnimationFrame run relative to the event loop?
rAF runs after the current macrotask and microtask queue drain, but before the browser's rendering steps (style, layout, paint, composite). The sequence: macrotask → microtasks → rAF callbacks → style → layout → paint → composite. rAF is synchronized with the display refresh rate (typically 60fps = every 16.6ms). Important: rAF callbacks registered within a rAF callback run in the next frame, not the current one. This makes rAF ideal for visual updates — you're guaranteed the browser will render your changes in the next paint. It's also why React's useLayoutEffect runs before paint (similar timing to rAF).
Q14: How does React's scheduler use the event loop for concurrent rendering?
React's scheduler uses MessageChannel (not setTimeout, which has a minimum 4ms delay) to schedule work as macrotasks. Each work chunk runs for ~5ms, then yields back to the browser by returning from the MessageChannel callback. The browser can then handle user input, run rAF, and render. React resumes work in the next MessageChannel callback. This is how concurrent rendering keeps the page responsive — work is interruptible, and higher-priority updates (user input) can preempt lower-priority work (data fetching). The scheduler maintains priority queues and uses performance.now() for timing.
Q15: What is INP and how does it relate to the event loop?
INP (Interaction to Next Paint) measures the time from user interaction (click, tap, key press) to the next paint that reflects the change. It replaced FID as a Core Web Vital. INP includes three phases: (1) Input delay — time the event waits in the macrotask queue while the main thread is busy. (2) Processing time — event handler execution. (3) Presentation delay — time for the browser to render the result. Optimization: break up long tasks so input events aren't delayed, provide immediate visual feedback before heavy computation, use CSS transitions (compositor thread) instead of JS animations, and use scheduler.yield() to yield to the browser mid-task.

DOM APIs

Q16: Compare IntersectionObserver, ResizeObserver, and MutationObserver.
IntersectionObserver: detects when elements enter/exit the viewport (or a specified container). Use for: lazy loading images, infinite scroll, scroll-triggered animations. Replaces scroll event listeners. ResizeObserver: detects element size changes (not just window resize). Use for: responsive components, container queries (before CSS container queries), chart resizing. MutationObserver: detects DOM tree changes (child additions/removals, attribute changes, text changes). Use for: integrating with non-React code, detecting third-party DOM modifications. All three are more performant than polling or event listeners because the browser batches notifications and delivers them at optimal times.
Q17: Explain the DOM event model: capture, target, and bubble phases.
DOM events propagate in three phases: (1) Capture — event travels from window down to the target element. Listeners registered with { capture: true } fire in this phase. (2) Target — event reaches the target element. Both capture and bubble listeners fire. (3) Bubble — event travels from target back up to window. Default listener phase. event.stopPropagation() stops further propagation in any phase. event.stopImmediatePropagation() also prevents other listeners on the same element. Event delegation uses bubbling: attach one listener to a parent, use event.target to identify which child was clicked. React's event system uses delegation — all events are attached to the root container.
Q18: What are passive event listeners and why do they matter for scroll performance?
Passive listeners ({ passive: true }) promise the browser that the handler won't call preventDefault(). This matters for scroll/touch events because the browser normally has to wait for the handler to complete before scrolling (in case it calls preventDefault()). With passive listeners, the browser can start scrolling immediately in parallel with the handler. Chrome made touch/wheel listeners passive by default to improve scroll performance. If you need preventDefault() in a scroll handler (e.g., custom scroll behavior), you must explicitly set { passive: false }.
Q19: How does AbortController work and what are its use cases?
AbortController provides a universal cancellation mechanism. Create a controller, pass its signal to cancellable operations, call controller.abort() to cancel. Use cases: (1) Fetch cancellation — cancel in-flight requests when component unmounts or query changes. (2) Event listener cleanup — pass signal to addEventListener, abort removes all listeners at once. (3) Timeout — AbortSignal.timeout(5000) auto-aborts after 5 seconds. (4) Combining signals — AbortSignal.any([signal1, signal2]) aborts when any signal fires. In React, AbortController in useEffect cleanup prevents state updates on unmounted components.

Networking

Q20: Compare HTTP/1.1, HTTP/2, and HTTP/3. What problems does each solve?
HTTP/1.1: one request per connection (pipelining rarely used), 6 connections per origin, head-of-line blocking. HTTP/2: multiplexed streams over single TCP connection, header compression (HPACK), server push. Solves HTTP-level head-of-line blocking but still has TCP-level blocking (one lost packet blocks all streams). HTTP/3: uses QUIC (UDP-based), each stream is independent (no TCP head-of-line blocking), 0-RTT connection establishment, connection migration (survives IP changes). Frontend implications: HTTP/2 makes domain sharding harmful and file concatenation less important. HTTP/3 improves performance on lossy networks (mobile) and during network transitions.
Q21: Explain Service Worker caching strategies.
Common strategies: (1) Cache-first: check cache, fall back to network. Best for static assets that rarely change. (2) Network-first: try network, fall back to cache. Best for HTML and API data that should be fresh. (3) Stale-while-revalidate: return cached immediately, fetch fresh in background for next time. Best for data that can be slightly stale (user profiles, product listings). (4) Network-only: always fetch from network. For non-cacheable requests (POST, real-time data). (5) Cache-only: only serve from cache. For pre-cached assets during offline mode. The strategy should match the resource type and freshness requirements.
Q22: What are resource hints and when would you use each one?
preload: fetch a resource needed for the current page with high priority. Use for fonts, hero images, critical CSS that the parser hasn't discovered yet. prefetch: fetch a resource needed for the next navigation with low priority. Use for next-page JS/CSS bundles. preconnect: establish DNS + TCP + TLS connection to an origin. Use for API origins and CDNs. Saves 100-300ms. dns-prefetch: resolve DNS only (cheaper than preconnect). Use for third-party origins. modulepreload: preload and parse ES modules. Use for critical JS modules. fetchpriority: fine-tune priority (high/low/auto). Use to boost hero images or deprioritize below-fold images.

Storage

Q23: When would you choose IndexedDB over localStorage?
Choose IndexedDB when: (1) Data exceeds 5-10 MB (IndexedDB can use 50%+ of disk). (2) You need to store non-string data (objects, blobs, ArrayBuffers) without serialization. (3) You need indexes for efficient querying. (4) You need transactions for atomic operations. (5) You need async access to avoid blocking the main thread. Choose localStorage when: data is small (<1 MB), simple key-value pairs, synchronous access is acceptable, and you need cross-tab reactivity via the storage event. For most production apps, IndexedDB (with a wrapper like idb) is the better choice because localStorage's synchronous API blocks the main thread.
Q24: How do you implement cross-tab communication in a web app?
Three approaches: (1) BroadcastChannel (recommended): simple pub/sub API, supports structured clone data, works across tabs of the same origin. (2) localStorage events: the storage event fires in other tabs when localStorage changes. Wider browser support but limited to string data and has race conditions. (3) SharedWorker: a worker shared across tabs that can relay messages. Most powerful but most complex. Use cases: syncing auth state (logout all tabs), syncing cart state, deduplicating notifications, leader election (only one tab makes API calls). BroadcastChannel is the best balance of simplicity and capability.
Q25: What is the structured clone algorithm and what can't it clone?
Structured clone is the deep copy algorithm used by postMessage, IndexedDB, structuredClone(), and history.pushState(). It can clone: primitives, Date, RegExp, Blob, File, ArrayBuffer, TypedArrays, Map, Set, plain objects, arrays, and circular references. It cannot clone: functions, DOM nodes, Symbols, property descriptors (getters/setters), or prototype chains (only own properties). Error objects are partially cloned (message and name only). This is why you can't pass React components or callbacks to workers — they contain functions. Use Comlink to work around this by creating proxy objects.

Web Workers

Q26: Compare dedicated workers, shared workers, and service workers.
Dedicated workers: one-to-one relationship with a page. Use for CPU-intensive computation (image processing, data parsing, search indexing). Terminated when the page closes. Shared workers: shared across multiple tabs/windows of the same origin. Use for cross-tab state management, shared WebSocket connections, deduplicating work. Terminated when the last connected tab closes. Service workers: act as a network proxy between the page and the network. Use for caching, offline support, push notifications, background sync. Lifecycle managed by the browser (can be terminated when idle, restarted on events). Each type has its own event loop, independent from the main thread.
Q27: Explain transferable objects vs structured clone for worker communication.
Structured clone (default postMessage): deep copies the data. Safe but slow for large data — copying 100MB takes ~50ms. Both threads have independent copies. Transferable objects: moves ownership from one thread to another. Zero-copy, instant. The sending thread loses access (ArrayBuffer.byteLength becomes 0). Transferable types: ArrayBuffer, MessagePort, ReadableStream, WritableStream, OffscreenCanvas, ImageBitmap. Use transferable objects for large binary data (images, audio, video frames). Use structured clone for small messages where both threads need the data. Comlink handles this automatically for most cases.
Q28: How would you decide what computation to move to a Web Worker?
Move to a worker when: (1) The computation takes >50ms (long task threshold). (2) It's CPU-bound, not I/O-bound (fetch already runs off the main thread). (3) It doesn't need DOM access. (4) The serialization overhead (postMessage) is less than the computation time. Good candidates: image/video processing, data parsing (CSV, large JSON), search indexing (full-text search), cryptography, complex mathematical computations, WebAssembly modules. Bad candidates: simple state updates, small data transformations, anything needing DOM access, operations where serialization cost exceeds computation cost.
Q29: What is Comlink and how does it simplify worker communication?
Comlink (by Google Chrome team) wraps postMessage with Proxy objects to create an RPC-style API. Instead of manually serializing messages and handling responses, you expose an object in the worker and call its methods as async functions from the main thread. Under the hood, Comlink uses MessageChannel for request/response pairing and structured clone for data transfer. It also supports transferable objects via Comlink.transfer(). The result: worker code looks like regular async function calls, dramatically reducing boilerplate and error-prone message handling.
Q30: What are worklets and how do they differ from workers?
Worklets are lightweight, specialized workers that run on the rendering thread (not a separate thread like workers). They're designed for specific rendering tasks: Paint Worklet (CSS Houdini): custom CSS painting via registerPaint(). Animation Worklet: animations on the compositor thread. Audio Worklet: real-time audio processing (replaces ScriptProcessorNode). Layout Worklet: custom layout algorithms. Key differences from workers: worklets have a restricted API (no fetch, no IndexedDB), they're class-based (not event-based), they run on the rendering thread for low-latency access to rendering state, and they can be instantiated multiple times by the browser for parallelism.

Cross-Cutting

Q31: How does understanding browser internals help you write better React code?
Concrete examples: (1) Rendering pipeline → use useLayoutEffect for DOM measurements before paint, useEffect for side effects after paint. (2) Event loop → understand why React batches state updates (same microtask) and how concurrent rendering yields to the browser (MessageChannel macrotasks). (3) V8 hidden classes → keep object shapes consistent in hot paths (component props, list items). (4) Compositor layers → animate only transform/opacity for smooth 60fps. (5) IntersectionObserver → implement lazy loading without scroll listeners. (6) Service Workers → add offline support and caching. Platform knowledge lets you make informed decisions that framework abstractions can't guide.
Q32: A user reports that your React app feels "janky" during scrolling. Walk through your debugging process.
Systematic approach: (1) Performance tab in DevTools — record during scroll, look for long tasks (>50ms red bars). (2) Rendering tab — enable "Paint flashing" to see what's being repainted. Enable "Layout shift regions" for CLS. (3) Layers panel — check for excessive compositor layers (memory issue) or missing layers (elements that should be composited aren't). (4) Check scroll handlers — are they passive? Are they doing layout reads? (5) Check for layout thrashing — interleaved reads/writes in scroll handlers. (6) React Profiler — are components re-rendering during scroll? (7) Solutions: use passive scroll listeners, debounce/throttle handlers, use CSS will-change for animated elements, use content-visibility: auto for long lists, virtualize with react-window.
Q33: Explain how you'd optimize a React app's loading performance using browser APIs.
Multi-layered approach: (1) Resource hints: preconnect to API origin, preload critical fonts and hero image, prefetch next-page bundles on hover. (2) Code splitting: React.lazy for route-level splitting, dynamic import for heavy components. (3) Service Worker: cache-first for static assets, stale-while-revalidate for API data. (4) Critical CSS: inline above-the-fold CSS, defer the rest. (5) Image optimization: responsive images with srcset, lazy loading with loading="lazy", fetchpriority="high" for hero. (6) Script loading: defer all scripts, use modulepreload for critical modules. (7) Compression: Brotli for text assets. (8) Measure: PerformanceObserver for LCP, CLS, INP. Set performance budgets and alert on regressions.
