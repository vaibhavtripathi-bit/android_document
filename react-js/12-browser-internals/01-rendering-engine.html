HTML Parsing and DOM Construction

The HTML parser is incremental — it processes bytes as they arrive from the network, building the DOM tree progressively. This is why users see content before the entire page loads.

// The parser processes HTML tokens sequentially:
// <html> → create HTMLHtmlElement, push to stack
//   <head> → create HTMLHeadElement, push to stack
//     <title>Page</title> → create title element + text node
//   </head> → pop head from stack
//   <body> → create HTMLBodyElement, push to stack
//     <div class="app">
//       <p>Hello</p>
//     </div>
//   </body>
// </html>

// Parser-blocking resources:
// 1. <script> (without async/defer) — blocks parsing
// 2. <link rel="stylesheet"> — blocks rendering (not parsing)
// 3. <script> after <link rel="stylesheet"> — blocks on CSS

// Resource loading strategies:
<script src="app.js" defer></script>     // Parse, execute after DOMContentLoaded
<script src="app.js" async></script>     // Parse in parallel, execute when ready
<script type="module" src="app.js"></script> // Deferred by default

// Speculative parsing (preload scanner):
// While the main parser is blocked by a script, the preload
// scanner continues scanning ahead for resources to fetch:
// - <img src>, <link href>, <script src>
// This is why resources load in parallel despite parser blocking
Architect Insight

The preload scanner is one of the most important browser optimizations. It was introduced because <script> tags block the parser, but the browser can still scan ahead for resources to fetch. This is why moving <script> to the bottom of <body> was historically recommended — it let the preload scanner discover images and stylesheets earlier. With defer and async, this is less critical but still relevant.

CSSOM Construction

// CSS is render-blocking — the browser won't paint until CSSOM is built
// This is because rendering with partial CSS would cause FOUC

// CSSOM is built by:
// 1. Parsing CSS into tokens
// 2. Building a tree of CSS rules
// 3. Computing specificity and cascade
// 4. Resolving inheritance

// The cascade order (lowest to highest priority):
// 1. User agent stylesheet (browser defaults)
// 2. User stylesheet (user preferences)
// 3. Author stylesheet (your CSS)
// 4. Author !important
// 5. User !important
// 6. User agent !important

// Specificity calculation:
// (inline, IDs, classes/attributes/pseudo-classes, elements/pseudo-elements)
// #header .nav a:hover  → (0, 1, 2, 1) = 121
// div#main p.intro      → (0, 1, 1, 2) = 112
// style="color: red"    → (1, 0, 0, 0) = 1000

// CSS containment for performance
.widget {
  contain: layout style paint;
  /* Tells browser: this element's internals don't affect outside layout.
     Browser can skip this subtree during layout/paint of other elements. */
}

.offscreen-section {
  content-visibility: auto;
  contain-intrinsic-size: 0 500px;
  /* Browser skips rendering until element is near viewport.
     contain-intrinsic-size provides estimated size for layout. */
}

Render Tree Construction

DOM Tree + CSSOM → Render Tree ═══════════════════════════════════════════════════════════ DOM Tree: CSSOM: html body { font: 16px sans-serif } ├── head .hidden { display: none } │ ├── title p { color: #333 } │ └── style span { font-weight: bold } └── body ├── p.visible │ └── "Hello" ├── p.hidden │ └── "Secret" └── span └── "World" Render Tree (only visible elements): RenderBody { font: 16px sans-serif } ├── RenderBlock (p) { color: #333 } │ └── RenderText "Hello" └── RenderInline (span) { font-weight: bold } └── RenderText "World" NOTE: p.hidden is NOT in the render tree (display: none) NOTE: head is NOT in the render tree (not visible) NOTE: visibility: hidden IS in the render tree (takes space)

Layout (Reflow)

Layout calculates the exact position and size of every element in the render tree. It's one of the most expensive operations in the rendering pipeline.

// Layout calculates box model for every element:
// - Content box (width × height)
// - Padding
// - Border
// - Margin
// - Position (x, y relative to parent)

// LAYOUT THRASHING: reading then writing in a loop
// BAD: forces synchronous layout on every iteration
const elements = document.querySelectorAll('.item');
elements.forEach(el => {
  const height = el.offsetHeight;    // READ → forces layout
  el.style.height = height * 2 + 'px'; // WRITE → invalidates layout
  // Next READ will force layout again!
});

// GOOD: batch reads, then batch writes
const heights = [];
elements.forEach(el => {
  heights.push(el.offsetHeight);     // All READs together
});
elements.forEach((el, i) => {
  el.style.height = heights[i] * 2 + 'px'; // All WRITEs together
});

// Properties that trigger layout (reflow):
// - width, height, padding, margin, border
// - top, left, right, bottom
// - font-size, font-family, font-weight
// - display, position, float
// - offsetWidth, offsetHeight, offsetTop, offsetLeft
// - scrollWidth, scrollHeight, scrollTop, scrollLeft
// - clientWidth, clientHeight, clientTop, clientLeft
// - getComputedStyle(), getBoundingClientRect()

Paint (Rasterization)

// Paint converts the layout tree into actual pixels
// It happens per-layer (compositor layers)

// Properties that trigger paint (but NOT layout):
// - color, background-color, background-image
// - border-color, border-style, border-radius
// - box-shadow, text-shadow
// - outline, visibility
// - filter, mix-blend-mode

// Properties that trigger NEITHER layout NOR paint:
// - transform (translate, rotate, scale)
// - opacity
// These only trigger COMPOSITE — the cheapest operation

// This is why CSS animations should use transform/opacity:
// BAD: animating left triggers layout + paint + composite every frame
.slide-bad {
  transition: left 300ms;
  position: absolute;
  left: 0;
}
.slide-bad.active { left: 100px; }

// GOOD: animating transform triggers only composite
.slide-good {
  transition: transform 300ms;
  transform: translateX(0);
}
.slide-good.active { transform: translateX(100px); }

Compositing and GPU Acceleration

Compositing Pipeline ═══════════════════════════════════════════════════════════ Main Thread: Compositor Thread: ┌──────────────┐ ┌──────────────────┐ │ JavaScript │ │ │ │ Style calc │ │ Receive layers │ │ Layout │──── layers ──▶│ Transform/clip │ │ Paint │ │ Composite │ │ (rasterize) │ │ Send to GPU │ └──────────────┘ └──────────────────┘ Layers are created for: ├── Elements with will-change: transform/opacity ├── Elements with transform: translate3d() / translateZ() ├── Elements with position: fixed ├── <video>, <canvas>, <iframe> ├── Elements with opacity < 1 ├── Elements with filter └── Elements that overlap a composited layer WARNING: Too many layers = excessive memory usage Each layer consumes GPU memory (width × height × 4 bytes) A 1920×1080 layer = ~8MB of GPU memory 
// will-change: hint to browser to promote to compositor layer
.animated-element {
  will-change: transform;
  /* Browser creates a compositor layer BEFORE animation starts.
     Without this, the first frame may jank as the layer is created. */
}

// IMPORTANT: Remove will-change after animation completes
element.addEventListener('transitionend', () => {
  element.style.willChange = 'auto'; // Release GPU memory
});

// contain: CSS containment for rendering optimization
.card {
  contain: layout style paint;
  /* layout: element's internals don't affect outside layout
     style: counters/quotes don't escape this element
     paint: element's contents don't paint outside its bounds */
}

.card-strict {
  contain: strict;
  /* Equivalent to: contain: size layout style paint
     size: element's size doesn't depend on children
     (must set explicit width/height) */
}

// content-visibility: skip rendering of off-screen content
.section {
  content-visibility: auto;
  contain-intrinsic-size: auto 500px;
  /* Browser skips layout/paint for off-screen sections.
     contain-intrinsic-size provides placeholder size.
     'auto' keyword remembers the last rendered size. */
}
// Can reduce initial rendering time by 50%+ for long pages

Critical Rendering Path

// The critical rendering path is the sequence of steps
// the browser takes to render the first pixel:

// 1. HTML → DOM (incremental, can start rendering early)
// 2. CSS → CSSOM (render-blocking)
// 3. DOM + CSSOM → Render Tree
// 4. Layout → Paint → Composite → Display

// Optimization strategies:
// 1. Minimize critical resources (CSS, sync JS)
<link rel="stylesheet" href="critical.css">
<link rel="stylesheet" href="non-critical.css" media="print" onload="this.media='all'">

// 2. Inline critical CSS
<style>
  /* Above-the-fold CSS inlined here */
  .header { ... }
  .hero { ... }
</style>
<link rel="preload" href="full.css" as="style" onload="this.rel='stylesheet'">

// 3. Defer non-critical JS
<script src="app.js" defer></script>
<script src="analytics.js" async></script>

// 4. Preload critical resources
<link rel="preload" href="hero.webp" as="image">
<link rel="preload" href="font.woff2" as="font" type="font/woff2" crossorigin>

// 5. Preconnect to critical origins
<link rel="preconnect" href="https://api.example.com">
<link rel="dns-prefetch" href="https://cdn.example.com">

Architect Insight — React and the Rendering Pipeline

React's virtual DOM batches DOM mutations, but it can't prevent layout thrashing if your effects read layout properties. useLayoutEffect runs synchronously after DOM mutations but before paint — it's the right place for DOM measurements that inform visual updates. useEffect runs after paint, so measurements there may cause a visible flash. Understanding the rendering pipeline explains why these two hooks exist and when to use each.

Interview Tip

When asked about rendering performance, walk through the pipeline: "First I'd check if the issue is layout (reflow), paint, or composite. For animations, I'd ensure we're only animating transform/opacity to stay on the compositor thread. For layout thrashing, I'd batch DOM reads and writes. For long pages, I'd use content-visibility: auto. For complex widgets, I'd use CSS containment." This systematic approach shows deep platform understanding.
