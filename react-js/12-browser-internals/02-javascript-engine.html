V8 Compilation Pipeline

V8 Compilation Pipeline (2024) ═══════════════════════════════════════════════════════════ Source Code (JavaScript/TypeScript compiled to JS) │ ▼ ┌──────────────┐ │ Scanner │ Tokenization (lazy: only parse what's needed) │ (Lexer) │ └──────┬───────┘ ▼ ┌──────────────┐ │ Parser │ Builds AST (Abstract Syntax Tree) │ │ Pre-parser for functions not immediately called └──────┬───────┘ ▼ ┌──────────────┐ │ Ignition │ Bytecode interpreter │ (Bytecode) │ Fast startup, collects type feedback └──────┬───────┘ ▼ ┌──────────────┐ │ Sparkplug │ Non-optimizing compiler (fast baseline JIT) │ (Baseline) │ Compiles bytecode → machine code without optimization └──────┬───────┘ ▼ ┌──────────────┐ │ Maglev │ Mid-tier optimizing compiler (new in 2023) │ (Mid-tier) │ Uses type feedback, faster compilation than TurboFan └──────┬───────┘ ▼ ┌──────────────┐ │ TurboFan │ Full optimizing compiler │ (Optimized) │ Speculative optimization based on type feedback └──────────────┘ │ ▼ (if assumptions violated) ┌──────────────┐ │ Deoptimize │ Fall back to Ignition bytecode │ │ Re-collect type feedback └──────────────┘

Parsing: Lazy and Eager

// V8 uses LAZY PARSING by default
// Only the top-level code and immediately-invoked functions are fully parsed
// Inner functions are "pre-parsed" (syntax checked, but no AST built)

// Example: V8 parsing strategy
function outerFunction() {        // Fully parsed (top-level)
  const x = 1;

  function innerFunction() {      // Pre-parsed (not called yet)
    return x + 1;                 // AST not built until first call
  }

  return innerFunction;
}

// When innerFunction is first called:
// 1. V8 fully parses it (builds AST)
// 2. Ignition compiles to bytecode
// 3. Executes bytecode

// IIFE forces eager parsing:
const result = (function() {      // Fully parsed immediately
  return computeExpensiveThing();
})();

// Hint to V8: wrap in parentheses to suggest eager parsing
// (Used by bundlers like webpack for critical modules)
const module = (function() { /* ... */ });

Ignition: Bytecode Interpreter

// Ignition compiles JavaScript to bytecode
// Bytecode is ~25-50% the size of equivalent machine code
// Faster to compile than machine code, slower to execute

// Example: how a simple function becomes bytecode
function add(a, b) {
  return a + b;
}

// Ignition bytecode (simplified):
// Ldar a1          ; Load argument 'a' into accumulator
// Add a2, [0]      ; Add argument 'b' to accumulator, feedback slot [0]
// Return            ; Return accumulator value

// Type feedback collection:
// The [0] in Add is a feedback vector slot
// Ignition records what types it sees:
//   add(1, 2)     → slot[0] records: Smi + Smi (small integers)
//   add(1.5, 2.5) → slot[0] records: HeapNumber + HeapNumber
//   add("a", "b") → slot[0] records: String + String

// This type feedback drives TurboFan's speculative optimization

// Register-based bytecode (V8 uses register machine, not stack machine)
// Registers: r0, r1, r2, ... (virtual registers mapped to stack slots)
// Accumulator: implicit register for most operations

TurboFan: Optimizing Compiler

// TurboFan uses type feedback to generate optimized machine code
// It makes SPECULATIVE ASSUMPTIONS based on observed types

function sumArray(arr) {
  let sum = 0;
  for (let i = 0; i < arr.length; i++) {
    sum += arr[i];  // Type feedback: always saw numbers
  }
  return sum;
}

// After many calls with number arrays, TurboFan optimizes:
// 1. Assumes arr is always an Array of Smis (small integers)
// 2. Removes type checks inside the loop
// 3. Uses integer addition (not generic + operator)
// 4. Eliminates bounds checks (if it can prove safety)
// 5. Unrolls the loop (if small enough)

// The optimized code is ~10-100x faster than interpreted bytecode

// DEOPTIMIZATION: if assumptions are violated
sumArray([1, 2, 3]);       // Optimized for Smi arrays ✓
sumArray([1.5, 2.5, 3.5]); // Still numbers, might survive ✓
sumArray(["a", "b", "c"]); // STRINGS! Deoptimize! ✗

// Deoptimization:
// 1. TurboFan code is discarded
// 2. Execution falls back to Ignition bytecode
// 3. Type feedback is updated with new types
// 4. Eventually, TurboFan may re-optimize with broader type assumptions
Critical — Megamorphic Call Sites

If a function is called with too many different object shapes (typically >4), V8 marks the call site as megamorphic and gives up on inline caching. This means every call goes through a slow generic path. In React, this can happen with polymorphic component props — if a callback receives objects with many different shapes, V8 can't optimize the hot path. Keep hot-path object shapes consistent.

Hidden Classes (Shapes/Maps)

// V8 assigns a "hidden class" (called Map internally) to every object
// Objects with the same property names in the same order share a Map

// These share the same hidden class:
const obj1 = { x: 1, y: 2 };
const obj2 = { x: 3, y: 4 };
// Both have Map: { x: offset 0, y: offset 1 }

// These have DIFFERENT hidden classes:
const obj3 = { x: 1, y: 2 };
const obj4 = { y: 2, x: 1 };  // Different property ORDER
// obj3 Map: { x: offset 0, y: offset 1 }
// obj4 Map: { y: offset 0, x: offset 1 }

// Hidden class transitions:
const point = {};           // Map M0: {}
point.x = 1;               // Map M0 → M1: { x: offset 0 }
point.y = 2;               // Map M1 → M2: { x: offset 0, y: offset 1 }

// ANTI-PATTERN: adding properties in different orders
function createPoint(x, y, hasZ) {
  const p = {};
  p.x = x;
  p.y = y;
  if (hasZ) p.z = 0;  // Some objects get z, some don't
  return p;            // Two different hidden classes!
}

// BETTER: always add all properties (even if undefined)
function createPoint(x, y, hasZ) {
  return { x, y, z: hasZ ? 0 : undefined };
  // All objects have the same shape: { x, y, z }
}

// BEST: use a class (V8 optimizes class instances heavily)
class Point {
  constructor(x, y, z = undefined) {
    this.x = x;
    this.y = y;
    this.z = z;
  }
}

Hidden Class Transition Chain

Hidden Class Transitions ═══════════════════════════════════════════════════════════ const obj = {}; obj.x = 1; obj.y = 2; │ │ │ ▼ ▼ ▼ ┌─────────┐ ┌─────────────┐ ┌──────────────┐ │ Map C0 │──"x"──▶│ Map C1 │─"y"▶│ Map C2 │ │ {} │ │ { x: @0 } │ │ { x: @0, │ │ │ │ │ │ y: @1 } │ └─────────┘ └─────────────┘ └──────────────┘ If another object adds x then y in the same order, it follows the SAME transition chain → shares Map C2. This enables V8 to use fixed offsets for property access instead of hash table lookups.

Inline Caches (ICs)

// Inline caches speed up property access by remembering
// the hidden class and property offset from previous calls

function getX(obj) {
  return obj.x;  // IC: remembers Map + offset of 'x'
}

// First call: getX({ x: 1, y: 2 })
// IC state: UNINITIALIZED → MONOMORPHIC
// Caches: Map C2, offset 0

// Second call: getX({ x: 3, y: 4 })  (same shape)
// IC state: MONOMORPHIC (cache hit!)
// Direct memory access: obj + offset 0 → value

// Third call: getX({ x: 5, z: 6 })  (different shape!)
// IC state: MONOMORPHIC → POLYMORPHIC
// Caches: [Map C2 → offset 0, Map C3 → offset 0]

// After 4+ different shapes:
// IC state: POLYMORPHIC → MEGAMORPHIC
// Falls back to generic hash table lookup (slow)

// IC States:
// UNINITIALIZED → MONOMORPHIC → POLYMORPHIC → MEGAMORPHIC
//                  (1 shape)     (2-4 shapes)  (5+ shapes)
//                  Fastest       Fast           Slow

Garbage Collection (Orinoco)

V8 Heap Layout ═══════════════════════════════════════════════════════════ ┌─────────────────────────────────────────────────────┐ │ V8 HEAP │ ├──────────────────────┬──────────────────────────────┤ │ YOUNG GENERATION │ OLD GENERATION │ │ (Nursery) │ (Tenured) │ │ │ │ │ ┌────────┐ │ ┌────────────────────────┐ │ │ │ Semi- │ │ │ Old Space │ │ │ │ space │ │ │ (objects that survived │ │ │ │ (from) │ │ │ multiple young GCs) │ │ │ ├────────┤ │ ├────────────────────────┤ │ │ │ Semi- │ │ │ Large Object Space │ │ │ │ space │ │ │ (objects > 256KB) │ │ │ │ (to) │ │ ├────────────────────────┤ │ │ └────────┘ │ │ Code Space │ │ │ │ │ (compiled code) │ │ │ Size: 1-8 MB │ │ │ │ │ GC: Scavenge │ │ Size: up to ~1.5 GB │ │ │ (very fast, ~1ms) │ │ GC: Mark-Sweep-Compact│ │ │ │ │ (slower, ~10-100ms) │ │ └──────────────────────┴──────────────────────────────┘ 
// Young Generation: Scavenge (Cheney's algorithm)
// - Two semi-spaces: "from" and "to"
// - New objects allocated in "from" space
// - When full: copy live objects to "to" space
// - Swap "from" and "to"
// - Objects surviving 2 scavenges → promoted to old generation
// - Very fast: ~1-5ms, only copies live objects

// Old Generation: Mark-Sweep-Compact
// 1. MARK: traverse from roots, mark all reachable objects
// 2. SWEEP: reclaim memory of unmarked objects
// 3. COMPACT: defragment memory (move objects together)

// Orinoco: V8's concurrent/parallel/incremental GC
// - Parallel: multiple GC threads work simultaneously
// - Incremental: marking is split into small steps interleaved with JS
// - Concurrent: marking happens on background threads while JS runs
// - Lazy sweeping: sweeping happens incrementally as memory is needed

// Memory leak patterns in React:
// 1. Forgotten event listeners (no cleanup in useEffect)
useEffect(() => {
  const handler = () => { /* ... */ };
  window.addEventListener('resize', handler);
  return () => window.removeEventListener('resize', handler); // CLEANUP!
}, []);

// 2. Closures holding large objects
useEffect(() => {
  const largeData = fetchLargeDataset(); // Held in closure
  return () => { /* largeData is freed when effect cleans up */ };
}, []);

// 3. Detached DOM nodes (React usually handles this)
// But manual DOM manipulation can create leaks:
const div = document.createElement('div');
someMap.set(key, div); // div can't be GC'd even if removed from DOM

// 4. Unbounded caches
const cache = new Map();
function memoize(fn) {
  return (arg) => {
    if (cache.has(arg)) return cache.get(arg);
    const result = fn(arg);
    cache.set(arg, result); // Cache grows forever!
    return result;
  };
}
// Fix: use WeakMap or LRU cache with size limit

Optimization Tips for V8

// 1. Keep object shapes consistent
// BAD: different shapes
const users = data.map(d => {
  const user = { name: d.name };
  if (d.email) user.email = d.email;  // Some have email, some don't
  return user;
});

// GOOD: consistent shapes
const users = data.map(d => ({
  name: d.name,
  email: d.email || null,  // Always present
}));

// 2. Avoid deleting properties (changes hidden class)
// BAD:
delete obj.unusedProp;  // Transitions to a new Map

// GOOD:
obj.unusedProp = undefined;  // Same Map, just null value

// 3. Use TypedArrays for numeric data
// BAD: regular array with numbers
const data = [1.1, 2.2, 3.3, 4.4];  // Boxed HeapNumbers

// GOOD: Float64Array
const data = new Float64Array([1.1, 2.2, 3.3, 4.4]);  // Unboxed doubles

// 4. Avoid arguments object (prevents optimization)
// BAD:
function sum() {
  let total = 0;
  for (let i = 0; i < arguments.length; i++) total += arguments[i];
  return total;
}

// GOOD:
function sum(...args) {
  return args.reduce((a, b) => a + b, 0);
}

// 5. Avoid try-catch in hot loops (historically deoptimized)
// Modern V8 handles this better, but still worth noting
// BAD:
for (let i = 0; i < 1000000; i++) {
  try { doWork(i); } catch (e) { handleError(e); }
}

// GOOD:
try {
  for (let i = 0; i < 1000000; i++) { doWork(i); }
} catch (e) { handleError(e); }
Interview Tip

When discussing V8 internals, focus on the practical implications: (1) Consistent object shapes enable inline caches → faster property access. (2) Type stability enables TurboFan optimization → faster hot loops. (3) Understanding GC generations explains why short-lived objects are cheap (young gen scavenge) but long-lived leaks are expensive (old gen mark-sweep). (4) Knowing the compilation tiers explains startup vs steady-state performance tradeoffs.
