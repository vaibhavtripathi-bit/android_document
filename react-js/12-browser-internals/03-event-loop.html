Tasks vs Microtasks

// EXECUTION ORDER PUZZLE
console.log('1: sync');

setTimeout(() => console.log('2: setTimeout'), 0);

Promise.resolve().then(() => console.log('3: promise'));

queueMicrotask(() => console.log('4: queueMicrotask'));

console.log('5: sync');

// Output:
// 1: sync
// 5: sync
// 3: promise       (microtask)
// 4: queueMicrotask (microtask)
// 2: setTimeout    (macrotask — next iteration)

// WHY: Microtasks run after the current task completes,
// BEFORE the next macrotask. The synchronous code is one task.
// After it finishes, all microtasks drain, then the next
// macrotask (setTimeout) runs.

// ADVANCED PUZZLE: microtasks queued during microtask processing
Promise.resolve().then(() => {
  console.log('A: first microtask');
  Promise.resolve().then(() => {
    console.log('B: nested microtask');
  });
});

setTimeout(() => console.log('C: setTimeout'), 0);

// Output:
// A: first microtask
// B: nested microtask  (microtask queue drains COMPLETELY)
// C: setTimeout        (only after all microtasks done)

// WARNING: infinite microtask loop blocks rendering
// This will freeze the browser:
function infiniteMicrotask() {
  Promise.resolve().then(infiniteMicrotask);
}
// Microtask queue never empties → rendering never happens
Critical — Microtask Starvation

The microtask queue drains completely before the browser can render or process the next macrotask. If microtasks keep queueing more microtasks, the page freezes. This is different from macrotasks — even an infinite setTimeout(fn, 0) loop won't freeze the page because the browser renders between macrotasks. React's state batching leverages this: updates within the same microtask are batched together.

Timing APIs

// setTimeout / setInterval — macrotask queue
setTimeout(() => {
  console.log('Runs after at least 0ms');
  // Actual minimum delay: ~4ms (clamped by browser)
  // In background tabs: ~1000ms (throttled)
}, 0);

// requestAnimationFrame — runs before next paint
// Synchronized with display refresh rate (typically 60fps = 16.6ms)
function animate() {
  element.style.transform = `translateX(${x}px)`;
  x += 1;
  requestAnimationFrame(animate); // Schedule next frame
}
requestAnimationFrame(animate);

// rAF timing in the event loop:
// 1. Process macrotask
// 2. Drain microtask queue
// 3. If rendering opportunity:
//    a. Run rAF callbacks (in order of registration)
//    b. Style → Layout → Paint → Composite
// 4. rAF callbacks from WITHIN rAF run in the NEXT frame

// requestIdleCallback — runs when browser is idle
requestIdleCallback((deadline) => {
  while (deadline.timeRemaining() > 0 && tasks.length > 0) {
    processTask(tasks.shift());
  }
  if (tasks.length > 0) {
    requestIdleCallback(processRemainingTasks);
  }
}, { timeout: 2000 }); // Force run after 2s even if not idle

// MessageChannel — fires as a macrotask (faster than setTimeout)
const channel = new MessageChannel();
channel.port1.onmessage = () => {
  console.log('MessageChannel callback');
};
channel.port2.postMessage(undefined);
// React's scheduler uses MessageChannel for scheduling work
// because it fires sooner than setTimeout(fn, 0)

Event Loop and Rendering

// The browser aims for 60fps = 16.6ms per frame
// Each frame budget:

// ┌──────────────────────────────────────────────┐
// │                 16.6ms Frame                   │
// ├──────┬──────┬──────┬──────┬──────┬───────────┤
// │  JS  │ rAF  │Style │Layout│Paint │ Composite  │
// │ task │      │      │      │      │            │
// │ 10ms │ 1ms  │ 1ms  │ 1ms  │ 1ms  │   1ms     │
// └──────┴──────┴──────┴──────┴──────┴───────────┘
//                                        ~15ms used
//                                        ~1.6ms idle

// If JS takes too long:
// ┌──────────────────────────────────────────────┐
// │                 16.6ms Frame                   │
// ├──────────────────────────┬──────┬─────────────┤
// │      JS task (50ms!)     │ rAF  │ Render...   │
// │      LONG TASK           │      │             │
// └──────────────────────────┴──────┴─────────────┘
//                            ↑ Frame dropped! Jank!

// React's solution: break work into small chunks
// using the scheduler with MessageChannel
// Each chunk yields back to the browser after ~5ms

// Detecting long tasks:
const observer = new PerformanceObserver((list) => {
  for (const entry of list.getEntries()) {
    if (entry.duration > 50) {
      console.warn('Long task detected:', entry.duration, 'ms');
      // Report to monitoring
    }
  }
});
observer.observe({ type: 'longtask', buffered: true });

// Breaking up long tasks:
async function processLargeArray(items) {
  const CHUNK_SIZE = 100;
  for (let i = 0; i < items.length; i += CHUNK_SIZE) {
    const chunk = items.slice(i, i + CHUNK_SIZE);
    processChunk(chunk);

    // Yield to browser between chunks
    await new Promise(resolve => setTimeout(resolve, 0));
    // Or use scheduler.yield() (new API)
  }
}

Long Tasks and INP (Interaction to Next Paint)

// INP measures the latency between user interaction and visual response
// It replaced FID (First Input Delay) as a Core Web Vital in March 2024

// INP = time from input event → next paint that reflects the change
// Good: ≤200ms | Needs improvement: 200-500ms | Poor: >500ms

// INP includes:
// 1. Input delay (time waiting for main thread to be free)
// 2. Processing time (event handler execution)
// 3. Presentation delay (time for browser to paint the result)

// Measuring INP:
const observer = new PerformanceObserver((list) => {
  for (const entry of list.getEntries()) {
    if (entry.interactionId) {
      const inp = entry.duration;
      console.log(`Interaction: ${entry.name}, INP: ${inp}ms`);
    }
  }
});
observer.observe({ type: 'event', durationThreshold: 16, buffered: true });

// Optimizing INP:
// 1. Break up long event handlers
button.addEventListener('click', async () => {
  // Immediate visual feedback
  button.textContent = 'Processing...';
  button.disabled = true;

  // Yield to let browser paint the feedback
  await scheduler.yield?.() ?? new Promise(r => setTimeout(r, 0));

  // Then do the heavy work
  const result = await heavyComputation();
  updateUI(result);
});

// 2. Use CSS transitions instead of JS animations
// CSS transitions happen on the compositor thread
// and don't block the main thread

// 3. Debounce/throttle input handlers
// Especially for scroll, resize, mousemove

React's Scheduling and the Event Loop

// React 18+ uses its own scheduler built on MessageChannel
// This is how concurrent rendering works:

// 1. setState() queues an update
// 2. React schedules work via MessageChannel (macrotask)
// 3. Work starts in the next event loop iteration
// 4. React processes fiber work in ~5ms chunks
// 5. After each chunk, yields back to browser (via MessageChannel)
// 6. Browser can render, handle events between chunks
// 7. React resumes work in the next macrotask

// This is why React 18 state updates are batched:
function handleClick() {
  setCount(c => c + 1);    // Queued
  setFlag(f => !f);         // Queued
  setName('React');          // Queued
  // All three updates are batched into ONE re-render
  // because they're in the same synchronous event handler
}

// Even async updates are batched in React 18:
async function handleClick() {
  const data = await fetchData();
  setCount(data.count);     // Batched!
  setItems(data.items);     // Batched!
  // In React 17, these would be separate renders
  // In React 18, they're batched via automatic batching
}

// React's priority system maps to event loop timing:
// Immediate (Sync) → microtask-like (synchronous flush)
// UserBlocking     → MessageChannel (next macrotask)
// Normal           → MessageChannel (may be interrupted)
// Low              → requestIdleCallback-like
// Idle             → requestIdleCallback
React Concurrent Rendering and Event Loop ═══════════════════════════════════════════════════════════ Frame 1: Frame 2: Frame 3: ┌─────────┐ ┌─────────┐ ┌─────────┐ │ React │ │ React │ │ React │ │ work │ │ work │ │ work │ │ (5ms) │ │ (5ms) │ │ (3ms) │ ├─────────┤ ├─────────┤ ├─────────┤ │ Browser │ │ Browser │ │ Browser │ │ render │ │ render │ │ render │ │ (11ms) │ │ (11ms) │ │ (13ms) │ └─────────┘ └─────────┘ └─────────┘ React yields every ~5ms so the browser can: - Handle user input (clicks, typing) - Run requestAnimationFrame - Perform layout/paint/composite - Keep the page responsive If a higher-priority update arrives (e.g., user typing), React can INTERRUPT the current work and handle it first.

Web Workers and Their Event Loops

// Each Web Worker has its OWN event loop
// Independent from the main thread's event loop

// Main thread event loop:
// - Has DOM access
// - Has rendering pipeline
// - Handles user events

// Worker event loop:
// - NO DOM access
// - NO rendering
// - Has timers (setTimeout, setInterval)
// - Has microtasks (Promises)
// - Has message handling (onmessage)

// Communication between event loops:
// Main thread → Worker: postMessage (arrives as macrotask in worker)
// Worker → Main thread: postMessage (arrives as macrotask in main)

// SharedArrayBuffer: shared memory between event loops
// Requires cross-origin isolation headers:
// Cross-Origin-Opener-Policy: same-origin
// Cross-Origin-Embedder-Policy: require-corp

const sab = new SharedArrayBuffer(1024);
const view = new Int32Array(sab);

// Main thread and worker can both read/write view
// Use Atomics for synchronization:
Atomics.store(view, 0, 42);        // Write
const value = Atomics.load(view, 0); // Read
Atomics.wait(view, 0, 42);          // Block until value changes
Atomics.notify(view, 0, 1);         // Wake one waiting thread
Interview Tip

When asked about the event loop, draw the diagram: call stack → microtask queue (drains completely) → render opportunity (rAF, style, layout, paint) → macrotask queue (one task). Emphasize that microtasks can starve rendering, that React uses MessageChannel for scheduling, and that requestAnimationFrame runs before paint (not after). This level of precision distinguishes L6 candidates.
