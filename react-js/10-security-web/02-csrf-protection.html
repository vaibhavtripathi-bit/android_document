CSRF Protection

Cross-Site Request Forgery attacks, SameSite cookies, CSRF tokens, double-submit cookie pattern, Origin header checking, and CSRF protection in React and Next.js.

1. How CSRF Attacks Work

CSRF Attack Flow:

  1. User logs into bank.com (session cookie set)
  2. User visits evil.com (attacker's site)
  3. evil.com contains hidden form:

  evil.com                          bank.com
  ┌──────────────────┐              ┌──────────────────┐
  │ <form action=    │   POST       │                  │
  │  "bank.com/      │─────────────▶│ /transfer        │
  │   transfer"      │  + session   │                  │
  │  method="POST"> │  cookie      │ Transfers $10000 │
  │  <input name=   │  (auto-sent  │ to attacker!     │
  │   "to" value=   │   by browser)│                  │
  │   "attacker">   │              │ Server sees valid │
  │  <input name=   │              │ session cookie    │
  │   "amount"      │              │ and processes     │
  │   value="10000">│              │ the request       │
  │ </form>         │              │                  │
  │ <script>        │              │                  │
  │  form.submit()  │              │                  │
  │ </script>       │              │                  │
  └──────────────────┘              └──────────────────┘

  Key: Browser automatically sends cookies with cross-origin requests

// Attacker's page (evil.com)
<!-- Hidden form that auto-submits -->
<form id="csrf-form" action="https://bank.com/api/transfer" method="POST">
  <input type="hidden" name="to" value="attacker-account" />
  <input type="hidden" name="amount" value="10000" />
</form>
<script>document.getElementById('csrf-form').submit();</script>

<!-- Or using an image tag for GET requests -->
<img src="https://bank.com/api/transfer?to=attacker&amount=10000" />

<!-- Or using fetch (limited by CORS, but simple requests go through) -->
<script>
fetch('https://bank.com/api/transfer', {
  method: 'POST',
  credentials: 'include',  // Send cookies
  body: new URLSearchParams({ to: 'attacker', amount: '10000' }),
});
</script>

2. SameSite Cookies

The SameSite cookie attribute is the primary modern defense against CSRF. It controls when cookies are sent with cross-site requests.

// SameSite cookie values:

// Strict: Cookie NEVER sent with cross-site requests
Set-Cookie: session=abc123; SameSite=Strict; Secure; HttpOnly
// Not sent when user clicks a link from another site to yours
// Problem: user clicking a link from email/social media won't be logged in

// Lax (default in modern browsers): Cookie sent with top-level navigations
Set-Cookie: session=abc123; SameSite=Lax; Secure; HttpOnly
// Sent: clicking a link to your site (GET navigation)
// NOT sent: form POST from another site, fetch/XHR, iframe, img
// Best balance of security and usability

// None: Cookie always sent (old behavior, requires Secure)
Set-Cookie: session=abc123; SameSite=None; Secure; HttpOnly
// Required for legitimate cross-site scenarios (OAuth, embedded widgets)
// Must be combined with other CSRF protections
Request TypeSameSite=StrictSameSite=LaxSameSite=None
Top-level GET navigation (link click)Not sentSentSent
Form POST from another siteNot sentNot sentSent
fetch/XHR from another siteNot sentNot sentSent
iframe embeddingNot sentNot sentSent
Image/script loadingNot sentNot sentSent
Same-site requestsSentSentSent

Architect Insight: SameSite=Lax as Default

Since Chrome 80 (2020), cookies without an explicit SameSite attribute default to Lax. This single browser change eliminated most CSRF attacks for applications that rely on cookie-based authentication. However, you should still explicitly set SameSite=Lax (or Strict) because: (1) not all browsers default to Lax, (2) explicit is better than implicit for security, (3) SameSite=None is still needed for some legitimate use cases and requires additional CSRF protection.

3. CSRF Tokens

// CSRF token pattern: server generates a random token,
// embeds it in the page, and validates it on state-changing requests

// Server-side (Express example):
import crypto from 'crypto';

// Generate token and store in session
app.use((req, res, next) => {
  if (!req.session.csrfToken) {
    req.session.csrfToken = crypto.randomBytes(32).toString('hex');
  }
  next();
});

// Endpoint to get CSRF token
app.get('/api/csrf-token', (req, res) => {
  res.json({ token: req.session.csrfToken });
});

// Validate CSRF token on state-changing requests
app.post('/api/transfer', (req, res) => {
  const token = req.headers['x-csrf-token'] || req.body._csrf;
  if (token !== req.session.csrfToken) {
    return res.status(403).json({ error: 'Invalid CSRF token' });
  }
  // Process request...
});

// React client-side:
// Fetch CSRF token on app initialization
async function getCsrfToken() {
  const response = await fetch('/api/csrf-token', { credentials: 'include' });
  const { token } = await response.json();
  return token;
}

// Include token in all state-changing requests
async function transfer(data) {
  const csrfToken = await getCsrfToken();
  return fetch('/api/transfer', {
    method: 'POST',
    credentials: 'include',
    headers: {
      'Content-Type': 'application/json',
      'X-CSRF-Token': csrfToken,
    },
    body: JSON.stringify(data),
  });
}

4. Double-Submit Cookie Pattern

// Double-submit: CSRF token in both cookie AND request header/body
// Server doesn't need to store the token — just compare cookie vs header

// Server sets CSRF token as a cookie:
Set-Cookie: csrf-token=random123; SameSite=Strict; Secure; Path=/

// Client reads the cookie and sends it as a header:
const csrfToken = document.cookie
  .split('; ')
  .find(row => row.startsWith('csrf-token='))
  ?.split('=')[1];

fetch('/api/action', {
  method: 'POST',
  headers: {
    'X-CSRF-Token': csrfToken,  // From cookie value
  },
  credentials: 'include',       // Also sends the cookie
});

// Server validates: cookie value === header value
app.post('/api/action', (req, res) => {
  const cookieToken = req.cookies['csrf-token'];
  const headerToken = req.headers['x-csrf-token'];

  if (!cookieToken || cookieToken !== headerToken) {
    return res.status(403).json({ error: 'CSRF validation failed' });
  }
  // Process request...
});

// Why this works:
// - Attacker can cause the browser to SEND the cookie
// - But attacker CANNOT READ the cookie (Same-Origin Policy)
// - So attacker cannot set the matching header value

Double-Submit Limitations

The basic double-submit pattern is vulnerable if an attacker can set cookies on your domain (via a subdomain XSS or cookie injection). The signed double-submit variant mitigates this: the server signs the cookie value with a secret key, and the client sends the raw value in the header. The server verifies the cookie's signature and compares the raw values. This ensures the attacker cannot forge a valid cookie-header pair.

5. Origin Header Checking

// The Origin header is sent with POST/PUT/DELETE requests
// and cannot be spoofed by JavaScript

app.use((req, res, next) => {
  if (['POST', 'PUT', 'DELETE', 'PATCH'].includes(req.method)) {
    const origin = req.headers.origin;
    const referer = req.headers.referer;

    const allowedOrigins = [
      'https://myapp.com',
      'https://www.myapp.com',
    ];

    // Check Origin header (preferred)
    if (origin && !allowedOrigins.includes(origin)) {
      return res.status(403).json({ error: 'Invalid origin' });
    }

    // Fallback to Referer if Origin is missing
    if (!origin && referer) {
      const refererOrigin = new URL(referer).origin;
      if (!allowedOrigins.includes(refererOrigin)) {
        return res.status(403).json({ error: 'Invalid referer' });
      }
    }

    // If neither Origin nor Referer is present, block the request
    // (some privacy extensions strip these headers)
    if (!origin && !referer) {
      return res.status(403).json({ error: 'Missing origin information' });
    }
  }
  next();
});

6. CORS and CSRF Interaction

// CORS does NOT prevent CSRF — it controls response access, not request sending

// Simple requests (form-like) are sent WITHOUT preflight:
// - Methods: GET, HEAD, POST
// - Content-Type: application/x-www-form-urlencoded, multipart/form-data, text/plain
// - No custom headers

// The request IS SENT, the server processes it,
// CORS only blocks the RESPONSE from being read by JavaScript

// This means a CSRF attack using a simple POST still works!
// The attacker doesn't need to read the response — the damage is done

// Defense: require Content-Type: application/json
// This triggers a CORS preflight (OPTIONS request)
// The preflight is blocked if the origin isn't allowed

app.use(cors({
  origin: ['https://myapp.com'],
  credentials: true,
  methods: ['GET', 'POST', 'PUT', 'DELETE'],
  allowedHeaders: ['Content-Type', 'X-CSRF-Token'],
}));

// API routes should reject non-JSON content types:
app.use('/api', (req, res, next) => {
  if (['POST', 'PUT', 'PATCH'].includes(req.method)) {
    if (!req.is('application/json')) {
      return res.status(415).json({ error: 'Content-Type must be application/json' });
    }
  }
  next();
});

Common Misconception: CORS Prevents CSRF

CORS is NOT a CSRF defense. CORS controls whether JavaScript can read the response, not whether the request is sent. A form POST from an attacker's site will still be sent and processed by the server — CORS only prevents the attacker from reading the response. The only exception is when the request triggers a preflight (custom headers, JSON content-type), which IS blocked by CORS. This is why requiring Content-Type: application/json and a custom CSRF header together provide strong protection.

7. React Form CSRF Protection

// Custom hook for CSRF-protected forms
function useCsrfToken() {
  const [token, setToken] = useState('');

  useEffect(() => {
    fetch('/api/csrf-token', { credentials: 'include' })
      .then(res => res.json())
      .then(data => setToken(data.token));
  }, []);

  return token;
}

// CSRF-protected form component
function TransferForm() {
  const csrfToken = useCsrfToken();

  const handleSubmit = async (e) => {
    e.preventDefault();
    const formData = new FormData(e.target);

    await fetch('/api/transfer', {
      method: 'POST',
      credentials: 'include',
      headers: {
        'Content-Type': 'application/json',
        'X-CSRF-Token': csrfToken,
      },
      body: JSON.stringify(Object.fromEntries(formData)),
    });
  };

  return (
    <form onSubmit={handleSubmit}>
      <input name="to" placeholder="Recipient" />
      <input name="amount" type="number" placeholder="Amount" />
      <button type="submit">Transfer</button>
    </form>
  );
}

// Axios interceptor for automatic CSRF token inclusion
import axios from 'axios';

const api = axios.create({
  baseURL: '/api',
  withCredentials: true,
});

api.interceptors.request.use(async (config) => {
  if (['post', 'put', 'delete', 'patch'].includes(config.method)) {
    const { data } = await axios.get('/api/csrf-token');
    config.headers['X-CSRF-Token'] = data.token;
  }
  return config;
});

8. Server Actions and CSRF in Next.js

// Next.js Server Actions have built-in CSRF protection

// Server Action (app/actions.ts)
'use server';

export async function transferFunds(formData: FormData) {
  // Next.js automatically:
  // 1. Generates a CSRF token bound to the user's session
  // 2. Includes it in the form's hidden field
  // 3. Validates it on the server before executing the action
  // 4. Checks the Origin header

  const to = formData.get('to') as string;
  const amount = Number(formData.get('amount'));

  await db.transfer({ to, amount });
  revalidatePath('/account');
}

// Client component using Server Action
function TransferForm() {
  return (
    <form action={transferFunds}>
      <input name="to" />
      <input name="amount" type="number" />
      <button type="submit">Transfer</button>
    </form>
  );
  // Next.js adds a hidden CSRF token field automatically
}

// API Route protection in Next.js
// app/api/transfer/route.ts
import { NextRequest, NextResponse } from 'next/server';

export async function POST(request: NextRequest) {
  // Verify origin
  const origin = request.headers.get('origin');
  if (origin !== process.env.NEXT_PUBLIC_APP_URL) {
    return NextResponse.json({ error: 'Forbidden' }, { status: 403 });
  }

  // Process request...
}

9. State-Changing GET Requests Anti-Pattern

// NEVER use GET requests for state-changing operations

// WRONG: GET request that changes state
app.get('/api/delete-account/:id', (req, res) => {
  db.deleteAccount(req.params.id);
  res.json({ success: true });
});
// Attackable via: <img src="https://app.com/api/delete-account/123">

// WRONG: GET request that transfers money
app.get('/api/transfer', (req, res) => {
  db.transfer(req.query.to, req.query.amount);
  res.json({ success: true });
});
// Attackable via: <img src="https://app.com/api/transfer?to=attacker&amount=10000">

// CORRECT: Use POST/PUT/DELETE for state changes
app.post('/api/delete-account/:id', csrfProtection, (req, res) => {
  db.deleteAccount(req.params.id);
  res.json({ success: true });
});

// GET requests should be safe (no side effects) and idempotent
// This is a fundamental REST principle AND a security requirement

10. CSRF Protection Summary

DefenseStrengthLimitation
SameSite=Lax cookiesBlocks most CSRF automaticallyDoesn't protect GET navigations
SameSite=Strict cookiesBlocks all cross-site cookie sendingBreaks legitimate cross-site links
CSRF tokensStrong, proven defenseRequires server-side session state
Double-submit cookieStateless CSRF protectionVulnerable to subdomain attacks
Origin header checkSimple, effectiveSome browsers/proxies strip Origin
Custom request headersTriggers CORS preflightOnly works for AJAX, not forms
JSON content-typeTriggers CORS preflightOnly works for AJAX, not forms

Architect Insight: Layered CSRF Defense

For a modern React SPA with cookie-based auth: (1) Set SameSite=Lax on all authentication cookies — this alone prevents most CSRF, (2) Require Content-Type: application/json for all API mutations — this triggers CORS preflight, (3) Validate the Origin header server-side, (4) Use CSRF tokens for traditional form submissions. For Next.js Server Actions, CSRF protection is built-in. The key insight: SameSite cookies + JSON API + Origin checking provides strong CSRF protection without explicit tokens for SPAs.
