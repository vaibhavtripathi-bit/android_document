Browser Security Model

Same-Origin Policy, CORS, Content Security Policy, Permissions Policy, Referrer Policy, iframe sandboxing, postMessage security, Web Crypto API, and secure contexts.

1. Same-Origin Policy (SOP)

The Same-Origin Policy is the foundational security mechanism of the web. It restricts how documents and scripts from one origin can interact with resources from another origin.

Origin = scheme + host + port

  https://example.com:443/path
  └─┬──┘ └────┬─────┘└┬─┘└─┬─┘
  scheme     host    port  path (not part of origin)

Same origin:
  https://example.com/a  ↔  https://example.com/b     ✓ Same
  https://example.com    ↔  https://example.com:443    ✓ Same (443 is default)

Different origin:
  https://example.com    ↔  http://example.com         ✗ Different scheme
  https://example.com    ↔  https://api.example.com    ✗ Different host
  https://example.com    ↔  https://example.com:8080   ✗ Different port

// What SOP restricts:
// 1. JavaScript cannot read responses from cross-origin fetch/XHR
fetch('https://api.other-site.com/data')
  .then(res => res.json())  // Blocked by SOP (unless CORS allows it)

// 2. JavaScript cannot access cross-origin iframe content
const iframe = document.querySelector('iframe');
iframe.contentDocument;  // SecurityError if different origin

// 3. JavaScript cannot read cross-origin cookies
document.cookie;  // Only cookies for current origin

// What SOP does NOT restrict:
// - Embedding cross-origin resources (<img>, <script>, <link>, <iframe>)
// - Sending cross-origin requests (the request IS sent, response is blocked)
// - Form submissions to cross-origin URLs
// - Navigating to cross-origin URLs

2. CORS (Cross-Origin Resource Sharing)

2.1 Simple Requests vs Preflight

Simple Request (no preflight):            Preflighted Request:

  Browser          Server                  Browser          Server
  │                │                       │                │
  │──GET/POST─────▶│                       │──OPTIONS──────▶│ (preflight)
  │  Origin: A     │                       │  Origin: A     │
  │                │                       │  Access-Control-│
  │◀──Response─────│                       │  Request-Method │
  │  Access-Control│                       │  Request-Headers│
  │  -Allow-Origin │                       │                │
  │                │                       │◀──Preflight────│
                                           │  Response      │
  Conditions for simple:                   │  Allow-Origin  │
  • GET, HEAD, POST                        │  Allow-Methods │
  • Content-Type:                          │  Allow-Headers │
    application/x-www-form-urlencoded      │  Max-Age       │
    multipart/form-data                    │                │
    text/plain                             │──Actual────────▶│
  • No custom headers                     │  Request       │
                                           │◀──Response─────│

// Server CORS configuration (Express)
import cors from 'cors';

app.use(cors({
  origin: ['https://myapp.com', 'https://staging.myapp.com'],
  methods: ['GET', 'POST', 'PUT', 'DELETE', 'PATCH'],
  allowedHeaders: ['Content-Type', 'Authorization', 'X-CSRF-Token'],
  exposedHeaders: ['X-Total-Count', 'X-Request-Id'],
  credentials: true,       // Allow cookies
  maxAge: 86400,           // Cache preflight for 24 hours
  optionsSuccessStatus: 204,
}));

// Manual CORS headers
app.use((req, res, next) => {
  const allowedOrigins = ['https://myapp.com'];
  const origin = req.headers.origin;

  if (allowedOrigins.includes(origin)) {
    res.setHeader('Access-Control-Allow-Origin', origin);
    res.setHeader('Access-Control-Allow-Credentials', 'true');
    res.setHeader('Access-Control-Allow-Methods', 'GET,POST,PUT,DELETE');
    res.setHeader('Access-Control-Allow-Headers', 'Content-Type,Authorization');
    res.setHeader('Access-Control-Max-Age', '86400');
  }

  if (req.method === 'OPTIONS') {
    return res.sendStatus(204);
  }
  next();
});

CORS Misconfigurations

Common dangerous CORS configurations: (1) Access-Control-Allow-Origin: * with credentials: true — browsers reject this combination, but some implementations reflect the Origin header instead, which is equivalent to allowing all origins with credentials. (2) Reflecting the Origin header without validation — allows any site to make authenticated requests. (3) Using regex that's too permissive — /example\.com/ matches evil-example.com. Always use an explicit allowlist of origins.

3. Content Security Policy (CSP)

// Comprehensive CSP header for a React application
Content-Security-Policy:
  default-src 'self';
  script-src 'self' 'nonce-{random}' https://cdn.example.com;
  style-src 'self' 'nonce-{random}' https://fonts.googleapis.com;
  img-src 'self' data: https: blob:;
  font-src 'self' https://fonts.gstatic.com;
  connect-src 'self' https://api.example.com wss://ws.example.com;
  media-src 'self';
  object-src 'none';
  frame-src 'self' https://www.youtube.com;
  frame-ancestors 'none';
  base-uri 'self';
  form-action 'self';
  upgrade-insecure-requests;
  report-uri /api/csp-report;
  report-to csp-endpoint;

// CSP reporting
Content-Security-Policy-Report-Only: ...
// Use Report-Only first to test without blocking

// Report endpoint
app.post('/api/csp-report', (req, res) => {
  const report = req.body['csp-report'];
  console.warn('CSP Violation:', {
    blockedUri: report['blocked-uri'],
    violatedDirective: report['violated-directive'],
    documentUri: report['document-uri'],
    sourceFile: report['source-file'],
    lineNumber: report['line-number'],
  });
  res.sendStatus(204);
});

4. Permissions Policy

// Permissions Policy (formerly Feature Policy) controls browser features
Permissions-Policy:
  camera=(),
  microphone=(),
  geolocation=(self),
  payment=(self "https://payments.example.com"),
  fullscreen=(self),
  autoplay=(self),
  accelerometer=(),
  gyroscope=(),
  magnetometer=(),
  usb=()

// Disable features entirely: feature=()
// Allow for self only: feature=(self)
// Allow for specific origins: feature=(self "https://trusted.com")
// Allow for all: feature=*

// In HTML (for iframes):
<iframe
  src="https://embed.example.com"
  allow="camera; microphone; fullscreen"
></iframe>

// React: check if feature is available
function CameraComponent() {
  const [hasPermission, setHasPermission] = useState(false);

  useEffect(() => {
    navigator.permissions.query({ name: 'camera' })
      .then(result => {
        setHasPermission(result.state === 'granted');
        result.addEventListener('change', () => {
          setHasPermission(result.state === 'granted');
        });
      });
  }, []);

  if (!hasPermission) return <p>Camera access required</p>;
  return <VideoCapture />;
}

5. Referrer Policy

// Controls how much referrer information is sent with requests
Referrer-Policy: strict-origin-when-cross-origin

// Policy options (from most to least restrictive):
// no-referrer              — never send referrer
// no-referrer-when-downgrade — no referrer on HTTPS→HTTP
// origin                   — send origin only (no path)
// origin-when-cross-origin — full URL same-origin, origin cross-origin
// same-origin              — full URL same-origin, nothing cross-origin
// strict-origin            — origin on same security, nothing on downgrade
// strict-origin-when-cross-origin — RECOMMENDED default
// unsafe-url               — always send full URL (dangerous)

// In React, set on links:
<a href="https://external.com" rel="noreferrer">External Link</a>

// Meta tag
<meta name="referrer" content="strict-origin-when-cross-origin" />

// Per-request in fetch
fetch('https://api.example.com/data', {
  referrerPolicy: 'no-referrer',
});

6. X-Frame-Options / frame-ancestors

// Prevent clickjacking by controlling who can embed your page

// Legacy header (still widely supported)
X-Frame-Options: DENY              // No framing allowed
X-Frame-Options: SAMEORIGIN        // Only same-origin framing

// Modern CSP directive (preferred)
Content-Security-Policy: frame-ancestors 'none';           // Same as DENY
Content-Security-Policy: frame-ancestors 'self';           // Same as SAMEORIGIN
Content-Security-Policy: frame-ancestors 'self' https://trusted.com;

// Clickjacking attack:
// Attacker embeds your page in an invisible iframe
// overlaid with a fake UI. User thinks they're clicking
// the attacker's button but actually clicks your page's button.

// JavaScript frame-busting (defense in depth, not primary defense)
if (window.self !== window.top) {
  // Page is in an iframe
  window.top.location = window.self.location;
}

7. Sandboxed Iframes

// sandbox attribute restricts iframe capabilities
<iframe
  src="https://untrusted-content.com"
  sandbox="allow-scripts allow-same-origin"
  loading="lazy"
></iframe>

// sandbox with no value = maximum restrictions:
// - No JavaScript execution
// - No form submission
// - No popups
// - No top navigation
// - Treated as unique origin (can't access parent)

// Selective permissions:
// allow-scripts         — enable JavaScript
// allow-same-origin     — treat as same origin (careful!)
// allow-forms           — allow form submission
// allow-popups          — allow window.open
// allow-top-navigation  — allow navigating parent frame
// allow-modals          — allow alert(), confirm(), prompt()

// React component for sandboxed content
function SandboxedContent({ html }: { html: string }) {
  const srcDoc = `
    <!DOCTYPE html>
    <html>
    <head><style>body { font-family: sans-serif; }</style></head>
    <body>${html}</body>
    </html>
  `;

  return (
    <iframe
      srcDoc={srcDoc}
      sandbox=""
      style={{ width: '100%', border: 'none' }}
      title="Sandboxed content"
    />
  );
}

sandbox="allow-scripts allow-same-origin" Danger

Combining allow-scripts and allow-same-origin on a same-origin iframe is dangerous — the iframe can use JavaScript to remove its own sandbox attribute and escape all restrictions. Only use this combination for cross-origin iframes where the iframe content can't modify the parent's DOM. For same-origin sandboxing, use allow-scripts without allow-same-origin.

8. postMessage Security

// postMessage enables cross-origin communication between windows/iframes
// But it's a common source of vulnerabilities if not validated

// SENDING a message
const iframe = document.querySelector('iframe');
iframe.contentWindow.postMessage(
  { type: 'UPDATE_THEME', theme: 'dark' },
  'https://trusted-origin.com'  // ALWAYS specify target origin
);

// WRONG: using '*' as target origin
iframe.contentWindow.postMessage(data, '*');  // Any origin can receive!

// RECEIVING a message — ALWAYS validate origin
window.addEventListener('message', (event) => {
  // 1. Validate origin
  if (event.origin !== 'https://trusted-origin.com') {
    return; // Ignore messages from untrusted origins
  }

  // 2. Validate message structure
  if (typeof event.data !== 'object' || !event.data.type) {
    return;
  }

  // 3. Handle known message types only
  switch (event.data.type) {
    case 'UPDATE_THEME':
      setTheme(event.data.theme);
      break;
    case 'RESIZE':
      setHeight(event.data.height);
      break;
    default:
      console.warn('Unknown message type:', event.data.type);
  }
});

// React hook for postMessage
function usePostMessage(
  origin: string,
  handler: (data: unknown) => void
) {
  useEffect(() => {
    function onMessage(event: MessageEvent) {
      if (event.origin !== origin) return;
      handler(event.data);
    }

    window.addEventListener('message', onMessage);
    return () => window.removeEventListener('message', onMessage);
  }, [origin, handler]);
}

9. Web Crypto API

// Web Crypto API provides cryptographic primitives in the browser
// Only available in secure contexts (HTTPS)

// Hashing
async function sha256(message: string): Promise<string> {
  const encoder = new TextEncoder();
  const data = encoder.encode(message);
  const hashBuffer = await crypto.subtle.digest('SHA-256', data);
  const hashArray = Array.from(new Uint8Array(hashBuffer));
  return hashArray.map(b => b.toString(16).padStart(2, '0')).join('');
}

// Symmetric encryption (AES-GCM)
async function generateAESKey(): Promise<CryptoKey> {
  return crypto.subtle.generateKey(
    { name: 'AES-GCM', length: 256 },
    true,
    ['encrypt', 'decrypt']
  );
}

// Asymmetric key pair (RSA-OAEP)
async function generateRSAKeyPair() {
  return crypto.subtle.generateKey(
    {
      name: 'RSA-OAEP',
      modulusLength: 4096,
      publicExponent: new Uint8Array([1, 0, 1]),
      hash: 'SHA-256',
    },
    true,
    ['encrypt', 'decrypt']
  );
}

// Digital signatures (ECDSA)
async function signData(privateKey: CryptoKey, data: ArrayBuffer) {
  return crypto.subtle.sign(
    { name: 'ECDSA', hash: 'SHA-256' },
    privateKey,
    data
  );
}

// Random values (cryptographically secure)
const randomBytes = crypto.getRandomValues(new Uint8Array(32));
const randomUUID = crypto.randomUUID();

Architect Insight: Security Header Deployment

Deploy security headers at the infrastructure level (reverse proxy, CDN, middleware), not in application code. This ensures consistent coverage across all routes. For Next.js, use next.config.js headers or middleware. For Vercel/Netlify, use their headers configuration. Start with Content-Security-Policy-Report-Only to identify violations before enforcing. Use securityheaders.com to audit your headers. A complete set of security headers is a baseline expectation for any production application.
