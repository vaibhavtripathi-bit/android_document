Interview Q&A — Security & Web

55+ L6/Staff-level questions on XSS, CSRF, authentication, authorization, secure data handling, dependency security, and the browser security model.

XSS Prevention

Q1: What are the three types of XSS and how does each work?

A: Reflected XSS: attacker's payload is in the URL/request and reflected in the server's response (e.g., search query displayed without escaping). Stored XSS: payload is saved in the database and served to all users who view the content (e.g., malicious comment). DOM-based XSS: payload is processed entirely in client-side JavaScript without reaching the server (e.g., innerHTML = location.hash). React's JSX auto-escaping protects against reflected and stored XSS in most cases, but DOM-based XSS through dangerouslySetInnerHTML or direct DOM manipulation remains a risk.

Q2: How does React protect against XSS by default?

A: React provides two layers of XSS protection: (1) JSX auto-escaping — all values embedded in JSX are converted to strings and HTML-entity-encoded before insertion into the DOM. Characters like <, >, &, ", ' are escaped, preventing HTML injection. (2) $$typeof Symbol — React elements include a $$typeof: Symbol.for('react.element') property that cannot be produced by JSON.parse, preventing an attacker from crafting a JSON payload that React would render as a valid element with dangerouslySetInnerHTML.

Q3: What are the XSS risks with dangerouslySetInnerHTML and how do you mitigate them?

A: dangerouslySetInnerHTML bypasses React's auto-escaping and injects raw HTML into the DOM. Any user-controlled content passed through it can execute scripts. Mitigation: (1) use DOMPurify to sanitize HTML with a strict allowlist of tags and attributes, (2) prefer Markdown renderers (react-markdown) over raw HTML rendering, (3) use Content Security Policy as a defense-in-depth layer, (4) if possible, restructure the data to avoid needing raw HTML entirely. The name "dangerously" is intentional — treat every usage as a security-sensitive code path requiring review.

Q4: How does the $$typeof Symbol prevent XSS in React?

A: React elements contain $$typeof: Symbol.for('react.element'). Symbols cannot be represented in JSON — JSON.stringify drops them, and JSON.parse cannot produce them. This prevents an attack where a server returns user-controlled JSON that mimics a React element structure with dangerouslySetInnerHTML. Without $$typeof, React would render the malicious JSON as a real element. With it, React rejects any object that doesn't have the correct Symbol, ensuring only elements created by React.createElement or JSX are rendered.

Q5: Explain Content Security Policy (CSP) and its challenges with React.

A: CSP is a browser security mechanism that restricts which resources can be loaded via HTTP headers. Key directives: script-src (JavaScript sources), style-src (CSS sources), connect-src (fetch/XHR targets). Challenges with React: (1) runtime CSS-in-JS libraries (styled-components) inject inline styles, requiring style-src 'unsafe-inline' or nonces, (2) Next.js injects inline scripts for hydration data, requiring nonce-based CSP, (3) eval() in development tools requires script-src 'unsafe-eval' in dev mode. Use nonce-based CSP where possible and prefer zero-runtime CSS solutions.

Q6: What is the Trusted Types API and how does it prevent DOM XSS?

A: Trusted Types is a browser API that prevents DOM XSS by requiring typed values for dangerous DOM sinks like innerHTML, eval(), and document.write(). Instead of accepting raw strings, these sinks require TrustedHTML, TrustedScript, or TrustedScriptURL objects created through a policy. Policies are functions that sanitize input (e.g., using DOMPurify). Enabled via CSP: require-trusted-types-for 'script'. This creates a chokepoint where all dangerous DOM operations must go through sanitization, making XSS much harder.

Q7: How do you handle XSS risks with href attributes in React?

A: React does NOT fully protect against javascript: URLs in href attributes. An attacker can set href="javascript:alert(document.cookie)" and React will render it (with a warning in React 16.9+). Defense: validate all user-provided URLs before rendering — check that the protocol is http:, https:, mailto:, or tel:. Reject or sanitize any URL with a javascript:, data:, or vbscript: protocol. Create a SafeLink component that enforces this validation for all external URLs.

Q8: What XSS vectors exist in SVGs and how do you defend against them?

A: SVGs can contain: <script> tags, event handlers (onload, onerror), javascript: URLs in xlink:href, and CSS with url(javascript:...). Defense: (1) render user SVGs as <img src="..."> (no script execution in img context), (2) sanitize SVGs with DOMPurify using the SVG profile, (3) strip all event handlers and script elements, (4) use a CSP that blocks inline scripts even within SVGs. Never render user-uploaded SVGs via dangerouslySetInnerHTML without sanitization.

CSRF Protection

Q9: Explain how CSRF attacks work and why cookies are the vulnerability.

A: CSRF exploits the browser's automatic cookie-sending behavior. When a user is authenticated on bank.com (session cookie set), an attacker's page can make requests to bank.com that automatically include the session cookie. The server sees a valid session and processes the request. The key vulnerability: browsers send cookies with every request to a domain, regardless of which site initiated the request. The attacker can't read the response (Same-Origin Policy), but the damage (money transfer, password change) is done by the request itself.

Q10: How does SameSite cookie attribute prevent CSRF?

A: SameSite controls when cookies are sent with cross-site requests. Strict: cookie never sent cross-site (even for navigation links — breaks UX). Lax (default in modern browsers): cookie sent for top-level GET navigations but NOT for cross-site POST, fetch, or iframe requests — prevents most CSRF while maintaining usability. None: always sent (requires Secure flag). Since Chrome 80, cookies without SameSite default to Lax, which eliminated most CSRF attacks. Explicitly setting SameSite=Lax is still recommended for clarity and cross-browser consistency.

Q11: Does CORS prevent CSRF? Explain the common misconception.

A: No, CORS does NOT prevent CSRF. CORS controls whether JavaScript can read the response, not whether the request is sent. A form POST from an attacker's site (a "simple request") is sent and processed by the server — CORS only prevents the attacker from reading the response. The exception: requests that trigger a CORS preflight (custom headers, JSON content-type) ARE blocked if the origin isn't allowed. This is why requiring Content-Type: application/json and a custom header (like X-CSRF-Token) provides CSRF protection for AJAX requests — they trigger preflight.

Q12: Describe the double-submit cookie pattern and its limitations.

A: The server sets a CSRF token as a cookie. The client reads the cookie value and sends it as a request header. The server compares: if cookie value equals header value, the request is legitimate. It works because an attacker can cause the browser to send the cookie but cannot read it (Same-Origin Policy), so they can't set the matching header. Limitation: if an attacker can set cookies on your domain (via subdomain XSS or cookie injection), they can set both the cookie and header. The signed variant mitigates this: the server signs the cookie value with a secret, and the client sends the raw value.

Q13: How does Next.js protect Server Actions against CSRF?

A: Next.js Server Actions have built-in CSRF protection: (1) a CSRF token is automatically generated and bound to the user's session, (2) the token is included as a hidden field in forms using the action, (3) the server validates the token before executing the action, (4) the Origin header is checked to ensure the request comes from the same site. This is transparent to developers — you don't need to manually implement CSRF protection for Server Actions. For API routes, you should still implement your own CSRF protection (Origin checking, custom headers).

Authentication

Q14: Compare JWT and session-based authentication. When would you choose each?

A: JWT: stateless (token contains claims), scales easily (no shared session store), but hard to revoke (need a blocklist), larger request size, and token theft gives full access until expiry. Sessions: stateful (server stores data), easy to revoke (delete from store), small cookie size, but requires shared session store for horizontal scaling. Choose JWT for: microservices, cross-domain auth, mobile APIs. Choose sessions for: traditional web apps, when revocation is critical, when you already have a session store (Redis). Many modern apps use a hybrid: session-like refresh tokens with short-lived JWTs.

Q15: Explain the OAuth 2.0 Authorization Code flow with PKCE.

A: PKCE (Proof Key for Code Exchange) prevents authorization code interception in public clients (SPAs). Flow: (1) client generates a random code_verifier and its SHA-256 hash (code_challenge), (2) client redirects to auth server with code_challenge, (3) user authenticates, auth server redirects back with authorization code, (4) client exchanges code + code_verifier for tokens, (5) auth server verifies that SHA-256(code_verifier) matches the original code_challenge. Without PKCE, an attacker who intercepts the authorization code could exchange it for tokens. With PKCE, they'd also need the code_verifier, which was never transmitted.

Q16: What is the safest way to store authentication tokens in a React SPA?

A: The safest pattern is the BFF (Backend for Frontend) approach: (1) store refresh tokens in httpOnly, Secure, SameSite=Strict cookies — JavaScript can't access them, so XSS can't steal them, (2) store access tokens in memory (JavaScript variable) — lost on page refresh but silently renewed via the refresh token cookie, (3) the BFF server handles token exchange, keeping client secrets server-side. Never use localStorage (vulnerable to XSS). sessionStorage is slightly better (cleared on tab close) but still XSS-vulnerable. In-memory is safest for access tokens.

Q17: How do you implement protected routes in React?

A: Create a ProtectedRoute wrapper component that checks auth state from context: if loading, show a spinner; if not authenticated, redirect to /login with the current location in state (for redirect-back after login); if authenticated, render children. For role-based routes, add a RequireRole component that also checks the user's role. Important: client-side route protection is for UX only — it can be bypassed. The real security boundary is the API layer, which must independently verify authentication and authorization on every request.

Q18: How does SSR authentication differ from SPA authentication?

A: In SSR (Next.js App Router), authentication happens on the server before HTML is sent: (1) read the session cookie from the request, (2) verify the session/token server-side, (3) if invalid, redirect before rendering. This prevents the "flash of unauthenticated content" that SPAs have. Server Components can directly access the database to verify sessions without exposing auth logic to the client. Middleware can protect entire route groups. The trade-off: every navigation requires a server round-trip for auth checking, while SPAs can check cached auth state instantly.

Q19: Describe the architecture of Auth.js (NextAuth).

A: Auth.js has four layers: (1) Providers handle identity verification — OAuth providers (Google, GitHub), credentials (email/password), email magic links. (2) Adapters persist user/session data to databases (Prisma, Drizzle, MongoDB). (3) Callbacks customize the JWT/session data at various lifecycle points (jwt callback adds custom claims, session callback shapes the client-visible session). (4) Middleware provides route-level protection via the authorized callback. Sessions can be JWT-based (stateless, in cookie) or database-based (session ID in cookie, data in DB).

Authorization

Q20: Compare RBAC and ABAC. When would you use each?

A: RBAC: permissions assigned to roles (admin, editor, viewer), roles assigned to users. Simple, easy to audit, sufficient for most web apps. Limitation: "role explosion" when fine-grained permissions are needed. ABAC: decisions based on attributes of user, resource, action, and environment. More flexible (e.g., "editors can edit articles in their department during business hours"), but complex to implement and audit. Use RBAC for: most React apps, clear role hierarchies. Use ABAC for: multi-tenant apps, regulatory requirements, resource-level ownership checks. Many apps use hybrid: RBAC for coarse access + ownership checks for resource-level decisions.

Q21: Explain defense in depth for authorization in a React application.

A: Authorization must be enforced at every layer: (1) UI layer — hide/disable unauthorized elements (UX only, not security), (2) Route layer — redirect unauthorized users (UX only), (3) API layer — reject unauthorized requests with 403 (actual security boundary), (4) Database layer — row-level security policies. The UI and route layers can be bypassed with browser DevTools. The API layer is the real security boundary — it must independently verify permissions on every request. Never trust the client to enforce authorization. A Staff engineer ensures all four layers are consistent.

Q22: How do you implement authorization in React Server Components?

A: In RSC, authorization happens server-side before rendering: use cache()-wrapped functions to get the current user (avoiding redundant DB queries), throw errors or redirect if unauthorized. Server Actions should also verify permissions before executing mutations. The advantage over client-side authorization: the authorization logic never reaches the client, unauthorized data is never fetched, and there's no "flash of unauthorized content." Use React's cache() to deduplicate auth checks across components in the same request.

Q23: How do feature flags differ from authorization?

A: Authorization answers "Is the user ALLOWED to access this?" — it's a security control. Feature flags answer "Is this feature ENABLED for this user?" — it's a product/release control. Authorization denies access permanently based on permissions; feature flags toggle features based on rollout percentage, user segments, or environment. They can overlap (a feature flag might target specific roles), but they serve different purposes. Feature flags should not be the sole authorization mechanism — if a feature requires admin access, enforce that in the API layer regardless of the flag state.

Secure Data Handling

Q24: Why should you never store secrets in client-side environment variables?

A: Client-side env vars (VITE_, NEXT_PUBLIC_, REACT_APP_) are replaced with literal string values at build time by the bundler. They appear as plain text in the JavaScript bundle, visible to anyone who inspects the page source or uses browser DevTools. This means API keys with write access, database URLs, auth secrets, and any other sensitive values are exposed. Use a BFF (Backend for Frontend) to proxy requests that need secrets, keeping them server-side only. Even "read-only" API keys should be evaluated — if they provide access to sensitive data, they shouldn't be client-exposed.

Q25: How do you protect API keys in a React application?

A: Use a server-side proxy (BFF pattern): the client calls your API route, which adds the secret API key and forwards the request to the third-party service. The client never sees the key. For APIs that must be called directly from the client (Stripe, Google Maps), use restricted keys: domain-restricted (only works from your domain), scope-restricted (read-only, specific endpoints), and rate-limited. Additionally, monitor API key usage for anomalies and rotate keys regularly. Never commit keys to version control — use environment variables and secrets management services.

Q26: What GDPR requirements affect React frontend applications?

A: Key GDPR requirements for frontends: (1) Consent management — cookie consent banners that block tracking until consent is given, with granular options (essential, analytics, marketing), (2) Right to access/export — UI for users to download their data, (3) Right to deletion — account deletion that clears all local data (localStorage, sessionStorage, cookies), (4) Data minimization — only collect necessary data, (5) Privacy by design — PII masking in UI, no sensitive data in URLs/logs, (6) Breach notification — mechanisms to notify users within 72 hours.

Q27: How do you handle sensitive form data securely in React?

A: Best practices: (1) use correct autoComplete attributes (current-password, new-password, cc-number) to work with password managers, (2) use POST for form submission (never GET for sensitive data), (3) don't store sensitive form values in component state longer than necessary, (4) for credit cards, use Stripe Elements or similar — they render in an iframe on the payment provider's domain, so card data never touches your code, (5) implement client-side validation for UX but always validate server-side, (6) use HTTPS exclusively, (7) clear sensitive fields on unmount.

Dependency Security

Q28: What is a supply chain attack in the npm ecosystem?

A: A supply chain attack compromises the software delivery pipeline rather than the application directly. In npm: (1) Typosquatting — publishing packages with names similar to popular ones (e.g., lodas instead of lodash), (2) Account takeover — compromising a maintainer's npm account to publish malicious versions, (3) Dependency confusion — publishing a public package with the same name as an internal package, causing the build system to install the malicious public version, (4) Malicious install scripts — postinstall scripts that execute arbitrary code during npm install.

Q29: How do lock files protect against supply chain attacks?

A: Lock files (package-lock.json, pnpm-lock.yaml) pin exact dependency versions and include integrity hashes (SHA-512). When you run npm ci or pnpm install --frozen-lockfile, the package manager: (1) installs exact versions from the lock file (not resolving version ranges), (2) verifies each package's integrity hash matches — if tampered, installation fails. Without lock files, npm install resolves ^1.0.0 to the latest matching version, which could be a newly published malicious release. Always commit lock files and use --frozen-lockfile in CI.

Q30: How do you defend against dependency confusion attacks?

A: Dependency confusion exploits the fact that package managers check the public registry for packages that should come from a private registry. Defenses: (1) use scoped packages (@myorg/package) — scopes are unique to your organization, (2) register your internal package names on the public npm registry as empty placeholders, (3) configure your .npmrc to route specific scopes to your private registry, (4) use pnpm's strict resolution which doesn't fall back to public registry for private packages, (5) audit your CI/CD pipeline for registry configuration.

Q31: What is Subresource Integrity (SRI) and when should you use it?

A: SRI allows browsers to verify that externally loaded resources (scripts, stylesheets) haven't been tampered with. You add an integrity attribute with a cryptographic hash: <script src="cdn.js" integrity="sha384-abc..." crossorigin="anonymous">. The browser computes the hash of the downloaded file and compares it — if they don't match, the resource is blocked. Use SRI when loading resources from CDNs or third-party domains. It protects against CDN compromises and man-in-the-middle attacks. For self-hosted resources, SRI is less critical but still provides defense in depth.

Q32: Describe your dependency update strategy for a production React application.

A: Layered approach using Renovate: (1) Security patches — auto-merge immediately after CI passes, (2) Patch updates — auto-merge with branch strategy (no PR review needed), (3) Minor updates — auto-merge with PR and CI verification, (4) Major updates — manual review, labeled for team attention, (5) group related packages (react + react-dom) to update together, (6) weekly schedule for non-security updates, (7) SLA: security patches within 24 hours, regular updates within 1 week. Run npm audit in CI to catch known vulnerabilities.

Browser Security Model

Q33: Explain the Same-Origin Policy and what it does and doesn't restrict.

A: SOP restricts cross-origin interactions based on scheme + host + port. It restricts: JavaScript reading cross-origin fetch/XHR responses, accessing cross-origin iframe content, and reading cross-origin cookies. It does NOT restrict: embedding cross-origin resources (img, script, link, iframe), sending cross-origin requests (the request is sent, the response is blocked), form submissions to cross-origin URLs, and navigating to cross-origin URLs. This distinction is crucial: CSRF works because SOP allows sending requests — it only blocks reading responses.

Q34: What is the difference between a simple CORS request and a preflighted request?

A: Simple requests are sent directly without preflight: GET/HEAD/POST with standard content types (form-urlencoded, multipart, text/plain) and no custom headers. The server responds with CORS headers, and the browser decides whether JavaScript can read the response. Preflighted requests trigger an OPTIONS request first: any request with custom headers, non-standard content types (application/json), or methods like PUT/DELETE. The browser sends OPTIONS with the intended method/headers; the server responds with what's allowed; only if permitted does the browser send the actual request. This is why requiring JSON content-type provides CSRF protection.

Q35: What are common CORS misconfigurations and their security implications?

A: (1) Reflecting Origin header without validation — effectively Access-Control-Allow-Origin: * with credentials, allowing any site to make authenticated requests. (2) Overly permissive regex — /example\.com/ matches evil-example.com. (3) Wildcard with credentials — Access-Control-Allow-Origin: * with Access-Control-Allow-Credentials: true is rejected by browsers, but some implementations work around this incorrectly. (4) Exposing sensitive headers — Access-Control-Expose-Headers with auth tokens. Always use an explicit origin allowlist, never reflect the Origin header blindly.

Q36: Explain Content Security Policy and how to implement it for a React app.

A: CSP is an HTTP header that restricts resource loading. For React: set default-src 'self' as baseline, use nonce-based script-src (generate random nonce per request, add to script tags and CSP header), restrict connect-src to your API domains, set frame-ancestors 'none' to prevent clickjacking, and object-src 'none' to block plugins. Start with Content-Security-Policy-Report-Only to identify violations without breaking the app. Challenges: runtime CSS-in-JS needs style-src 'unsafe-inline' or nonces, and some third-party scripts may require specific source allowlisting.

Q37: How does postMessage work and what are its security risks?

A: postMessage enables cross-origin communication between windows/iframes. Security risks: (1) Missing origin validation — if the receiver doesn't check event.origin, any page can send messages and trigger actions. (2) Using '*' as target origin — when sending, targetWindow.postMessage(data, '*') allows any origin to receive the message. (3) Trusting message content — message data should be validated/sanitized like any user input. Defense: always specify the exact target origin when sending, always validate event.origin when receiving, validate message structure, and handle only known message types.

Q38: What is the sandbox attribute for iframes and when is it dangerous?

A: The sandbox attribute restricts iframe capabilities. With no value, it blocks: scripts, form submission, popups, top navigation, and treats the iframe as a unique origin. You selectively enable features: allow-scripts, allow-forms, allow-same-origin, etc. The dangerous combination is allow-scripts allow-same-origin on a same-origin iframe — the iframe can use JavaScript to remove its own sandbox attribute and escape all restrictions. For same-origin sandboxing, use allow-scripts without allow-same-origin. For cross-origin iframes, the combination is safe because the iframe can't modify the parent.

Q39: What are secure contexts and why do they matter?

A: A secure context is an environment served over HTTPS (or localhost for development). Many modern browser APIs require secure contexts: Web Crypto, Service Workers, Geolocation, Clipboard API, Payment Request, MediaDevices (camera/microphone), and Credential Management. This requirement ensures that sensitive operations only occur over encrypted connections, preventing man-in-the-middle attacks from intercepting data. HSTS (Strict-Transport-Security) ensures your site is always accessed over HTTPS, and HSTS preloading adds your domain to browsers' built-in HTTPS-only list.

Advanced / Cross-Cutting

Q40: Design a comprehensive security architecture for a React application.

A: Layered approach: (1) Transport: HTTPS everywhere, HSTS with preload, certificate pinning for mobile. (2) Headers: strict CSP with nonces, X-Frame-Options DENY, Referrer-Policy, Permissions-Policy. (3) Authentication: OAuth 2.0 + PKCE for SSO, httpOnly/Secure/SameSite cookies for sessions, short-lived JWTs in memory. (4) Authorization: RBAC at API layer, UI-level permission gates for UX. (5) Input: server-side validation, DOMPurify for HTML rendering, URL validation. (6) Dependencies: lock files, automated auditing, SRI for CDN resources. (7) Monitoring: CSP reporting, error tracking, audit logs.

Q41: How would you conduct a security review of a React application?

A: Systematic checklist: (1) Search for dangerouslySetInnerHTML — verify all instances use sanitization. (2) Check all href attributes for user-controlled URLs — verify protocol validation. (3) Review environment variables — ensure no secrets have client prefixes. (4) Check authentication flow — verify token storage, session management, CSRF protection. (5) Review API calls — ensure authorization headers, error handling, no sensitive data in URLs. (6) Run npm audit and license check. (7) Verify security headers (CSP, HSTS, X-Frame-Options). (8) Check for eval(), innerHTML, document.write(). (9) Review third-party script loading for SRI.

Q42: How do you handle authentication in a micro-frontend architecture?

A: Centralized auth with shared session: (1) a dedicated auth micro-frontend handles login/logout, (2) session cookie is shared across all micro-frontends (same domain), (3) each micro-frontend validates the session independently via a shared auth service, (4) use Module Federation's shared dependencies to share the auth context/hook. For cross-domain micro-frontends: use a central auth server with OAuth, each micro-frontend gets its own tokens, and shared state is communicated via postMessage or a shared BFF. Key concern: ensure consistent session invalidation across all micro-frontends.

Q43: What security considerations apply to React Server Components?

A: RSC introduces new security considerations: (1) Server-only code: use the server-only package to prevent server modules from being imported in client code (prevents secret leakage). (2) Serialization boundary: props passed from Server to Client Components are serialized — ensure no sensitive data crosses this boundary. (3) Server Actions: validate all inputs server-side (they're essentially public API endpoints), implement rate limiting, and verify authorization. (4) Direct database access: RSC can query databases directly — apply the same authorization checks as API routes.

Q44: How do you implement rate limiting for a React application's API?

A: Rate limiting should be implemented server-side: (1) use a sliding window or token bucket algorithm, (2) identify clients by IP, user ID, or API key, (3) store counters in Redis for distributed systems, (4) return 429 Too Many Requests with Retry-After header, (5) implement different limits for different endpoints (auth endpoints stricter than read endpoints). On the React side: handle 429 responses gracefully with retry logic (exponential backoff), show user-friendly messages, and implement client-side debouncing/throttling to reduce unnecessary requests.

Q45: Explain the Web Crypto API and its use cases in React applications.

A: Web Crypto provides cryptographic primitives in the browser (secure contexts only): hashing (SHA-256/384/512), symmetric encryption (AES-GCM, AES-CBC), asymmetric encryption (RSA-OAEP), digital signatures (ECDSA, RSA-PSS), key derivation (PBKDF2, HKDF), and secure random generation. Use cases in React: (1) client-side encryption of sensitive data before sending to server, (2) generating PKCE code verifiers for OAuth, (3) hashing data for integrity verification, (4) end-to-end encryption in messaging apps. Important: key management is the hard part — storing encryption keys securely in the browser is challenging.

Q46: How do you prevent clickjacking attacks on a React application?

A: Clickjacking embeds your page in an invisible iframe with a fake UI overlay. Defenses: (1) X-Frame-Options: DENY header (legacy but widely supported), (2) Content-Security-Policy: frame-ancestors 'none' (modern, more flexible), (3) JavaScript frame-busting as defense in depth: if (window.self !== window.top) window.top.location = window.self.location. The CSP frame-ancestors directive is preferred because it supports allowlisting specific origins that can embed your page, while X-Frame-Options only supports DENY or SAMEORIGIN.

Q47: What is the Referrer-Policy header and why does it matter for security?

A: Referrer-Policy controls how much URL information is sent in the Referer header when navigating or making requests. Security implications: (1) URLs may contain sensitive data (tokens, user IDs, search queries), (2) third-party services receiving full referrer URLs can track user behavior. Recommended: strict-origin-when-cross-origin — sends full URL for same-origin requests, only the origin for cross-origin, and nothing for HTTPS→HTTP downgrades. For sensitive pages (password reset, admin), use no-referrer. In React, add rel="noreferrer" to external links.

Q48: How do you secure WebSocket connections in a React application?

A: WebSocket security considerations: (1) use wss:// (WebSocket Secure) — never ws:// in production, (2) authenticate the WebSocket handshake — send auth token as a query parameter or in the first message (cookies are sent automatically with the handshake), (3) validate the Origin header on the server to prevent cross-site WebSocket hijacking, (4) implement message validation — treat all incoming messages as untrusted input, (5) rate limit messages to prevent abuse, (6) implement reconnection with exponential backoff, (7) don't send sensitive data that the user shouldn't see (WebSocket messages are visible in DevTools).

Q49: What are Cross-Origin-Opener-Policy and Cross-Origin-Embedder-Policy?

A: COOP (Cross-Origin-Opener-Policy: same-origin) isolates your browsing context — cross-origin windows opened via window.open can't access your window object and vice versa. This prevents Spectre-like side-channel attacks. COEP (Cross-Origin-Embedder-Policy: require-corp) ensures all cross-origin resources explicitly opt in to being loaded (via CORS or Cross-Origin-Resource-Policy headers). Together, COOP + COEP enable crossOriginIsolated mode, which unlocks high-resolution timers (performance.now()) and SharedArrayBuffer — required for WebAssembly threading.

Q50: How do you implement Content Security Policy reporting?

A: CSP reporting sends violation reports to a specified endpoint: (1) add report-uri /api/csp-report or the newer report-to directive to your CSP header, (2) create a server endpoint that receives and logs violation reports (JSON format with blocked-uri, violated-directive, source-file, line-number), (3) start with Content-Security-Policy-Report-Only to collect violations without blocking, (4) analyze reports to identify legitimate violations vs false positives, (5) gradually tighten the policy and switch to enforcing mode. Use services like Report URI or Sentry for aggregation and alerting.

Q51: How do you handle security in a progressive web app (PWA)?

A: PWA-specific security concerns: (1) Service Worker scope — service workers can intercept all requests within their scope; ensure they're served from your origin with proper CSP, (2) Cache poisoning — if a service worker caches a compromised response, it persists even after the server is fixed; implement cache versioning and update strategies, (3) Offline data — data cached in IndexedDB/Cache API is accessible to any script on the origin; encrypt sensitive cached data, (4) Push notifications — validate push message content server-side, (5) Install prompt — ensure HTTPS (required for service workers).

Q52: What security headers should every production React application have?

A: Essential headers: (1) Content-Security-Policy — restrict resource loading, (2) Strict-Transport-Security — force HTTPS, (3) X-Content-Type-Options: nosniff — prevent MIME sniffing, (4) X-Frame-Options: DENY — prevent clickjacking (or use CSP frame-ancestors), (5) Referrer-Policy: strict-origin-when-cross-origin — control referrer leakage, (6) Permissions-Policy — disable unused browser features. Additional: Cross-Origin-Opener-Policy: same-origin, Cross-Origin-Embedder-Policy: require-corp if you need cross-origin isolation. Deploy at the infrastructure level (reverse proxy, CDN) for consistent coverage.

Q53: How do you handle third-party script security in a React application?

A: Third-party scripts (analytics, ads, chat widgets) are a major attack surface: (1) load via SRI to detect tampering, (2) use CSP to restrict which domains can serve scripts, (3) load in sandboxed iframes when possible (isolates from your DOM), (4) audit third-party scripts regularly — they can change without your knowledge, (5) use a tag manager to centralize third-party script management, (6) implement a consent management platform for GDPR compliance, (7) monitor for unexpected network requests from third-party scripts, (8) consider self-hosting critical third-party scripts (with update monitoring).

Q54: How do you implement secure file upload in a React application?

A: Secure file upload checklist: (1) validate file type on both client (accept attribute, MIME check) and server (magic bytes, not just extension), (2) limit file size (client-side for UX, server-side for security), (3) scan for malware server-side, (4) store files outside the web root or in object storage (S3), (5) generate random filenames (don't use user-provided names), (6) serve uploaded files from a separate domain/CDN (prevents cookie access), (7) set Content-Disposition: attachment for downloads, (8) sanitize SVGs and HTML files, (9) use presigned URLs for direct-to-S3 uploads to avoid server load.

Q55: Describe how you would respond to a security incident in a React application.

A: Incident response plan: (1) Identify — determine the attack vector (XSS, compromised dependency, credential leak), scope, and affected users. (2) Contain — revoke compromised tokens/sessions, deploy a hotfix, block the attack vector. (3) Eradicate — remove the vulnerability (patch the code, update the dependency, rotate secrets). (4) Recover — restore normal operations, force password resets if credentials were exposed, clear compromised caches. (5) Notify — inform affected users (GDPR requires within 72 hours), report to relevant authorities. (6) Post-mortem — document the incident, identify root cause, implement preventive measures (new security tests, monitoring, policies).
