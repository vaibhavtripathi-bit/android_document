Infinite Scroll & Virtualization at Scale

How windowing libraries approximate O(visible) work for O(total) datasets: internal mechanics of fixed vs variable layouts, bidirectional chat loading, scroll restoration across navigations, memory contracts, IntersectionObserver triggers, and placeholder strategies. Written for Staff interviews where “use react-window” is never enough.

Core Invariant

Virtualization is a projection from logical index space to DOM nodes. Interviewers probe whether you understand that scroll height is often estimated, measurements are asynchronous, and state must survive remounts (router transitions, React StrictMode double effects).

1. How Windowing Libraries Think Internally

Libraries like react-window and react-virtuoso share a skeleton: maintain scrollTop, derive startIndex/endIndex from cumulative offsets, render only that slice, and pad with spacer elements (paddingTop/paddingBottom or absolute transforms). Differences show up in dynamic measurement, grouped lists, and reverse mode for chat.

Why Not Just Paginate?

Pagination changes the information architecture (explicit next page). Infinite scroll keeps a single growing surface—better for feeds, worse for SEO and some accessibility expectations. Staff answers name tab traps, scroll jank, and URL representability (deep link to “page 3” vs cursor token).

// Minimal fixed-row virtualizer core (illustrative, not production-complete)
type VirtualizerConfig = {
  itemCount: number;
  itemHeight: number;
  viewportHeight: number;
  scrollTop: number;
  overscan: number; // render extra rows above/below
};

export function computeWindow(c: VirtualizerConfig): {
  start: number;
  end: number;
  paddingTop: number;
  paddingBottom: number;
} {
  const { itemCount, itemHeight, viewportHeight, scrollTop, overscan } = c;
  const start = Math.max(
    0,
    Math.floor(scrollTop / itemHeight) - overscan,
  );
  const end = Math.min(
    itemCount - 1,
    Math.ceil((scrollTop + viewportHeight) / itemHeight) + overscan,
  );
  const paddingTop = start * itemHeight;
  const paddingBottom = (itemCount - end - 1) * itemHeight;
  return { start, end, paddingTop, paddingBottom };
}

2. Dynamic Row Heights

Variable content breaks the cheap arithmetic above. Strategies: (1) measure after mount with ResizeObserver, storing heights in a Map keyed by id or index; (2) binary search on a prefix-sum array of heights to map scrollTop → index in O(log n); (3) estimated height with correction pass (jank risk); (4) uniform in a band (e.g., tweet cards ~similar) with occasional remeasure.

Dynamic measurement lifecycle mount row i -> RO fires -> update height[i] -> recompute prefix sums | | +---- may shift scrollTop if anchor policy = "top"
Layout Thrash

Reading layout (offsetHeight) after every state change in the same frame forces synchronous reflow. Batch measurements with requestAnimationFrame or queueMicrotask carefully. Prefer ResizeObserver over polling innerHTML heights.

// Caching measured heights + prefix binary search (simplified)
export class HeightCache {
  private heights = new Map<number, number>();
  private prefix: number[] = [];

  setHeight(index: number, h: number): void {
    this.heights.set(index, h);
    this.rebuildPrefix(this.lastCount);
  }

  private lastCount = 0;

  rebuildPrefix(itemCount: number): void {
    this.lastCount = itemCount;
    this.prefix = new Array(itemCount + 1).fill(0);
    for (let i = 0; i < itemCount; i++) {
      const h = this.heights.get(i) ?? 32; // estimate fallback
      this.prefix[i + 1] = this.prefix[i]! + h;
    }
  }

  indexAtOffset(y: number): number {
    const p = this.prefix;
    let lo = 0,
      hi = p.length - 2;
    while (lo <= hi) {
      const mid = (lo + hi) >> 1;
      if (p[mid + 1]! <= y) lo = mid + 1;
      else hi = mid - 1;
    }
    return lo;
  }
}

3. Bidirectional Scrolling (Chat Timelines)

Chat UIs prepend older messages when the user scrolls near the top. The hard part is preserving scroll position when new items appear above the viewport: capture anchorId + offset from viewport top, splice data, restore so the user’s eyes do not jump.

4. Scroll Position Restoration

SPAs break native scroll restoration because the DOM is synthetic. Pattern: persist { key, scrollTop, anchorIndex, subpixel } in session storage or router state. On remount, virtualizer must seek before paint—often two-pass: render estimated window, measure, adjust.

// Hook sketch: restore after data + virtualizer ready
import { useEffect, useLayoutEffect, useRef } from 'react';

export function useVirtualScrollRestoration(args: {
  storageKey: string;
  scrollRef: React.RefObject<HTMLDivElement | null>;
  ready: boolean;
}) {
  const restored = useRef(false);

  useLayoutEffect(() => {
    if (!args.ready || restored.current) return;
    const raw = sessionStorage.getItem(args.storageKey);
    if (!raw) return;
    const { scrollTop } = JSON.parse(raw) as { scrollTop: number };
    args.scrollRef.current?.scrollTo({ top: scrollTop });
    restored.current = true;
  }, [args.ready, args.storageKey, args.scrollRef]);

  useEffect(() => {
    const el = args.scrollRef.current;
    if (!el) return;
    const onScroll = () => {
      sessionStorage.setItem(
        args.storageKey,
        JSON.stringify({ scrollTop: el.scrollTop }),
      );
    };
    el.addEventListener('scroll', onScroll, { passive: true });
    return () => el.removeEventListener('scroll', onScroll);
  }, [args.storageKey, args.scrollRef]);
}

5. Memory Management

Virtualization reduces DOM nodes, not necessarily JS heap for your record store. If you retain full message arrays of 100k objects, you still pay RAM. Mitigations: LRU eviction of far-offscreen data with skeleton reload, WeakRef patterns for auxiliary caches (limited), compact encoding for rows, and Web Workers for filtering so React state stays lean.

Memory layers [ User model cache (LRU) ] ---> [ Visible slice in React state ] | v [ Row components + DOM ]
List Stability

Random keys destroy reconciliation and cause unnecessary unmounts. Stable keys per logical id prevent state loss inside rows (expanded threads, in-flight mutations). When ids are server-assigned, never fall back to array index for keyed lists in virtualized feeds.

6. IntersectionObserver & Placeholders

IntersectionObserver is ideal for fetch triggers (load next page when sentinel visible) decoupled from scroll event frequency. Combine with a rootMargin to prefetch before the user hits the hard bottom. Placeholders should reserve approximate height to avoid CLS; use shimmer only if height estimate exists, otherwise fixed skeleton bands.

// Sentinel + IO for infinite fetch (React)
import { useEffect, useRef } from 'react';

export function InfiniteSentinel(props: {
  onLoadMore: () => Promise<void>;
  root: Element | null;
}) {
  const ref = useRef<HTMLDivElement | null>(null);
  useEffect(() => {
    const el = ref.current;
    if (!el || !props.root) return;
    const io = new IntersectionObserver(
      (entries) => {
        entries.forEach((e) => {
          if (e.isIntersecting) void props.onLoadMore();
        });
      },
      { root: props.root, rootMargin: '200px' },
    );
    io.observe(el);
    return () => io.disconnect();
  }, [props.root, props.onLoadMore]);
  return <div ref={ref} style={{ height: 1 }} />;
}

8. SSR, Hydration & Virtualization

Server-rendering 10 items then hydrating into a virtualizer that expects empty initial scroll is a classic mismatch. Patterns: render the first window only on the server with identical props, pass initialScrollTop=0, and defer full dataset attachment until after hydration. For Next.js App Router, watch double data-fetch: RSC payload + client fetch can duplicate work—prefer passing initial items through server components then adopting them into client state once.

SSR + virtualizer alignment Server HTML: first k rows + spacer metadata (total count optional) Client: hydrate k rows -> attach RO -> expand measurements Risk: CLS if server k differs from client estimate -> gate on media/layout

Horizontal virtualization

Data grids combine column and row windowing. The complexity is sticky columns and synced scrollbars. Often split into frozen and scrollable regions with two scroll containers and scroll-linked listeners—document performance implications (passive listeners, transform instead of left/top where possible).

Grid feature Virtualization impact
Column resize Invalidate measured widths; recompute horizontal offsets
Row reorder (DnD) Temporarily disable recycling or pin dragged row portal
Cell editors Focus retention when row leaves window—commit or cancel?
Mobile Safari

Momentum scrolling and rubber-banding interact badly with manual overflow containers. Test -webkit-overflow-scrolling: touch legacy behavior vs modern iOS. Position fixed toolbars further reduce visible viewport—your viewportHeight measurement must use ResizeObserver on the scroll root, not only window.innerHeight.

Performance budget sanity check

As a rule of thumb for interviews, quote 60fps ≈ 16.6ms per frame budget on the main thread. If each row commit averages 0.5ms and your overscan mounts 40 rows, you risk multi-millisecond commits before layout. Keep rows lean: memoize, avoid inline object literals, split heavy subtrees with React.memo + stable props, and prefer CSS over JS measurement where possible.

When asked “how large a list can we support?”, answer in terms of DOM nodes, heap retained by row models, and scroll event cadence—not raw item count alone.

Observability hooks

Expose virtualizer metrics to RUM: avgRowHeightEstimateError, windowRecyclesPerSecond, and scrollCorrectionCount after prepend. Spikes correlate with bad keys or image layout pops—useful when debugging “janky feed” tickets escalated from VIP users.

Interview Q&A

How do you implement sticky section headers inside a virtualized list? L6

Track which section header index is “active” based on scrollTop and prefix offsets; render the sticky header as a separate absolutely positioned layer synchronized with the scroll container. Alternatively, model sections as variable-size groups where the header is the first element in a group—libraries differ; the Staff answer is consistent offset accounting + avoiding double paint.

What breaks if the server changes sort order while the user scrolls? Architect

Indices shift; cursor-based pagination may skip or duplicate unless you use stable cursors tied to data identity, not numeric page. Client should reset or reconcile: show toast, snap to nearest stable anchor, or soft refresh with preserved id set.

Why might overscan trade off accessibility? Senior

Overscan increases the number of mounted nodes, which can expand the accessibility tree and affect screen reader virtual buffer behavior. Tune overscan per platform; test with VoiceOver/NVDA on large lists.

How do you avoid duplicate pages when the user scrolls past multiple sentinels quickly? L6

Maintain an inFlight map keyed by cursor token; ignore IO callbacks while a fetch is active; use AbortController to cancel stale responses. Optionally disable the sentinel until the previous promise settles. Server should be idempotent for the same cursor.
