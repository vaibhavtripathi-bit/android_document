Design System at Scale (50+ Teams)

Versioning with semver and changesets, breaking change playbooks (codemods, deprecation windows), adoption telemetry, per-component performance budgets, accessibility gates in CI, multi-brand theming via design tokens, and documentation generated from TypeScript types—the operational side of design systems Staff engineers own.

Platform Thesis

A design system is a distributed product. Success is measured by safe upgrades, not pixel-perfect Figma parity. Your job is to align API stability, bundle cost, and migration velocity across dozens of deploy pipelines you do not control.

1. Component Versioning Strategy

Use semantic versioning per package: MAJOR for breaking prop/API changes, MINOR for backward-compatible features, PATCH for fixes. Changesets (or equivalent) let contributors declare intent in PRs; release automation aggregates changelog entries and bumps versions consistently.

Monorepo release train PR A (changeset: minor @acme/button) PR B (changeset: major @acme/form) | v Release workflow -> build -> test matrix -> publish NPM -> notify consumers
Change type Semver Consumer expectation
Rename prop isDisabled → disabled MAJOR TypeScript errors until codemod/migration
New optional prop MINOR Safe upgrade without code changes
Internal refactor, same API PATCH Drop-in replacement
Implicit Breaking Changes

CSS specificity shifts, bundle side effects, and DOM structure changes can break consumers without TS noticing. Treat visual output as API: snapshot critical stories, document DOM contracts for integration-tested consumers.

2. Breaking Change Management

Ship @deprecated JSDoc + runtime console.warn behind a dev flag for one MAJOR cycle. Provide codemods (jscodeshift/ts-morph) with fixtures. Publish a migration guide with searchable before/after snippets.

// jscodeshift codemod sketch: rename prop
import type { API, FileInfo, Options } from 'jscodeshift';

export default function transform(file: FileInfo, api: API, _options: Options) {
  const j = api.jscodeshift;
  const root = j(file.source);

  root
    .find(j.JSXOpeningElement, { name: { name: 'Button' } })
    .forEach((path) => {
      path.node.attributes?.forEach((attr) => {
        if (attr.type === 'JSXAttribute' && attr.name.name === 'isDisabled') {
          attr.name.name = 'disabled';
        }
      });
    });

  return root.toSource({ quote: 'single' });
}

3. Adoption Metrics

Instrument package downloads (noisy) and in-repo imports (better): nightly job scans GitHub Enterprise via code search API or AST grep across an internal monoreor. Track version histogram per app; chase long tails before flipping defaults.

// Adoption tracker: scan lockfiles / package.json (Node script)
import fg from 'fast-glob';
import fs from 'node:fs/promises';

type Row = { repo: string; version: string };

export async function collectButtonVersions(roots: string[]): Promise<Row[]> {
  const files = await fg('**/package.json', { cwd: roots[0], ignore: ['**/node_modules/**'] });
  const rows: Row[] = [];
  for (const f of files) {
    const pkg = JSON.parse(await fs.readFile(f, 'utf8'));
    const v = pkg.dependencies?.['@acme/button'] ?? pkg.devDependencies?.['@acme/button'];
    if (v) rows.push({ repo: f, version: v });
  }
  return rows;
}
Privacy

Telemetry must respect enterprise policy: aggregate in CI artifacts, avoid sending source code to third-party SaaS without review. Prefer static analysis in your own runner.

4. Performance Budgets per Component

Set max gzip bytes per entry point and per top-level export path. Track tree-shaking effectiveness: accidental export * from heavy icons package nukes budgets. Use size-limit or custom Rollup stats JSON compared to baseline on main.

// CI: fail if @acme/button exceeds budget (pseudo-shell)
#!/usr/bin/env bash
set -euo pipefail
rollup packages/button/index.ts --file /tmp/button.js --format esm --silent
BYTES=$(gzip -c /tmp/button.js | wc -c | tr -d ' ')
if [ "$BYTES" -gt 12000 ]; then
  echo "Button bundle gzip $BYTES exceeds 12000"
  exit 1
fi

@acme/button ~12 KB Imported on every route shell
@acme/data-grid ~80 KB Lazy route-only import
@acme/icons per-icon ESM Barrel files forbidden

5. Accessibility Compliance in CI

Run axe-core in Storybook test-runner or Playwright against a story matrix. Gate on WCAG 2.2 AA rulesets for components claiming compliance. Pair automated checks with manual keyboard walkthroughs for composite widgets (combobox, date picker).

CI accessibility gates Storybook build -> test-runner + axe -> upload HTML report | +--> fail PR if new violations vs baseline (allowlist shrinks over time)
False Confidence

Automated a11y cannot prove usability. Staff narrative: “We enforce necessary conditions (roles, names, contrast) in CI; sufficiency is validated in quarterly audits with disabled users.”

6. Theming Architecture

Store tokens as CSS variables on :root or scoped data-theme containers. For multi-brand, load token sets at build time or runtime from JSON manifests. Avoid prop-drilling theme objects when possible—CSS variables compose with media queries and prefers-color-scheme.

/* tokens.css — compile-time brand swap */
:root[data-brand='acme'] {
  --color-primary: #0066cc;
  --radius-md: 8px;
  --font-sans: 'Inter', system-ui, sans-serif;
}

:root[data-brand='contoso'] {
  --color-primary: #7b1fa2;
  --radius-md: 4px;
  --font-sans: 'Segoe UI', system-ui, sans-serif;
}

8. Governance & RFC Process

New primitives need an RFC: problem statement, API sketch, a11y checklist, bundle impact, and migration plan for overlapping legacy components. Use design council office hours to avoid silent forks (“shadow design systems”).

Anti-pattern Outcome Prevention
One-off fork per product line Inconsistent UX + duplicated bugs Extension slots + token overrides
Unbounded props API Impossible to tree-shake, unclear contracts Composition over configuration
Skipping visual regression Silent pixel drift across brands Chromatic or self-hosted snapshot CI

9. React 19 & Forward Compatibility

Prepare for ref as prop, async transitions, and potential changes to event timing. Pin peer dependency ranges honestly: react: ^18.2 || ^19.0 once validated. Test Server Components boundaries if shipping wrappers that touch context—split client/server entry points clearly.

Peer Dependency Hell

Overly narrow peerDependencies block upgrades; overly wide ones hide incompatibility. Publish a compatibility matrix in docs and automate smoke tests across React versions weekly.

10. Supply Chain for UI Packages

Sign releases (npm provenance), enable 2FA for publishers, scan SBOM for vulnerabilities. Document allowed postinstall scripts policy—design system packages should be pure ESM without native addons when possible.

Consumer trust chain Maintainer -> signed tag -> CI provenance -> npm attestation -> consumer lockfile pin

11. Composition Patterns vs Configuration

Staff-level systems bias toward slots (leftIcon as ReactNode rather than enum of built-ins) and render props for escape hatches. Configuration-only APIs explode variant matrices; composition localizes complexity to call sites that actually need it. Document recommended recipes (copy-paste templates) to reduce improvisation drift.

// Prefer slot composition over prop explosion
export type CardProps = {
  media?: React.ReactNode;
  title: React.ReactNode;
  actions?: React.ReactNode;
  children?: React.ReactNode;
};

export function Card({ media, title, actions, children }: CardProps) {
  return (
    <article className="card">
      {media && <div className="card__media">{media}</div>}
      <header className="card__title">{title}</header>
      <div className="card__body">{children}</div>
      {actions && <footer className="card__actions">{actions}</footer>}
    </article>
  );
}

12. Lint Rules & API Boundaries

Custom ESLint rules encode policy: ban default imports from @acme/icons, forbid styled-components in app code when CSS modules are standard, require data-testid only in e2e harness packages. Ship the plugin as @acme/eslint-plugin versioned alongside components so rule changes ride the same train.

TypeScript API Extractor

API Extractor produces .api.md reports and enforces @public / @internal tags. Pair with --production entry trimming so accidental deep imports fail at build time, not runtime.

13. Style Dictionary Pipeline

Transform JSON tokens to CSS variables, iOS/Android resources, and Figma plugin JSON. Single pipeline prevents “token drift” between platforms. CI regenerates artifacts on token PRs; human reviewers focus on semantic naming, not hand-edited CSS.

tokens.json --> style-dictionary --> dist/css/variables.css --> dist/js/tokens.ts (for runtime theming in tests) --> dist/figma/tokens.json

14. Support Model & SLAs

Platform teams define SLAs: P0 accessibility regression fixed in 24h, P1 visual bug triaged in 72h. Run a public Slack/Discord channel with on-call rotation. Measure time-to-first-response for internal consumers—adoption drops when support feels like a black hole.

Burnout Risk

Without guardrails, “free custom work for every product” scales linearly with headcount. Enforce RFCs for new primitives and prioritize fixes that unblock multiple teams (N>3 rule of thumb).

Interview Q&A

How do you prevent icon packages from destroying bundle size? L6

Ship per-icon modules, forbid star exports, provide a build-time plugin or CLI to import only used glyphs, and enforce size-limit per icon in CI. Offer a sprite sheet path only for legacy consumers with documented trade-offs (cache granularity vs simplicity).

What is your strategy when two teams need incompatible visual tweaks? Architect

Prefer token overrides and slot composition over forking components. If they still diverge, expose a controlled variant prop backed by design-approved tokens—not arbitrary style props that become untestable.

How do you version Figma libraries relative to code packages? Senior

Map Figma library releases to npm semver ranges in release notes. Automate token export (Style Dictionary) so Figma and code share a JSON source of truth. When they desync, code wins for runtime; file design debt tickets for drift.

How would you roll out dark mode without doubling design QA cost? L6

Drive colors from semantic tokens (--color-surface-default) mapped per theme, not duplicated components. Snapshot both themes in CI for a reduced story grid (happy paths + high-risk widgets). Use forced-colors media queries for high-contrast validation.

What metrics prove a design system reduces engineering cost? Architect

Track time-to-ship for standard forms pre/post adoption, duplicate component count via AST search, defect rate tagged “UI inconsistency”, and upgrade downtime (hours spent per MAJOR). Combine qualitative NPS from partner teams quarterly.
