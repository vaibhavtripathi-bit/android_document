Migrating a Monolithic React App to Micro-Frontends

Strangler fig, shared auth, routing bridges, independent deploys, and the performance trade-offs staff engineers negotiate.

Migration thesis

Micro-frontends are an organizational scaling tool. The hard problems are not “import remote” but consistent identity, routing truth, dependency duplication, and blast-radius control during the years-long transition.

Strangler fig pattern

Incrementally replace slices of the monolith behind a stable shell. Each vertical owns UI, data fetching, and release cadence; the shell owns chrome, auth bootstrap, and top-level navigation.

Year 0 [████████████████ Monolith ████████████████] Year 1 [Shell │██ MFE-A ██│████████ Monolith ████████] Year 2 [Shell │ MFE-A │ MFE-B │██ Monolith (legacy) ██] Year 3 [Shell │ MFE-A │ MFE-B │ MFE-C │ (monolith retired) ]

The strangler wrapper mounts legacy routes until the MFE is ready, then swaps the implementation without changing the URL contract seen by users.

Strangler fig wrapper (React)

// StranglerHost.tsx — feature-toggled boundary
import { lazy, Suspense, type ComponentType } from 'react';

const LegacyCheckout = lazy(() => import('../legacy/CheckoutApp'));
const MfeCheckout = lazy(() => import('checkout_mfe/Checkout'));

type Props = { useMfe: boolean };

export function StranglerCheckout({ useMfe }: Props) {
  const Cmp: ComponentType = useMfe ? MfeCheckout : LegacyCheckout;
  return (
    <Suspense fallback={<ChromeSkeleton />}>
      <ErrorBoundary onError={reportToShell}>
        <Cmp />
      </ErrorBoundary>
    </Suspense>
  );
}
Boundary discipline

Each MFE should fail independently. The shell provides a shared error boundary contract (error IDs, retry, support links) so UX stays coherent when a remote fails to load.

Shared state during migration

Monoliths lean on a single Redux store; MFEs cannot safely share mutable singletons across webpack runtimes without version skew. Patterns:

  • Shell-owned global store — Zustand/Redux in the shell; MFEs receive props or subscribe via a narrow window.__SHELL_BUS__ adapter (versioned events).
  • URL as source of truth — encode cart ID, org ID, and feature flags in the route or query where possible.
  • Backend-for-frontend — duplicate client state is replaced by server reconciliation (GraphQL/REST) keyed by session.
Approach Coupling Risk When to use
Props from shell Low Prop drilling at depth Leaf MFEs, short-lived data
Custom events / bus Medium Implicit contracts Cross-MFE notifications
Shared npm package High Version hell Design tokens, auth client
Module Federation shared High Singleton mismatch react/react-dom only
React duplication

Two copies of React break hooks and context. Federation shared: { react: { singleton: true, strictVersion: true } } is mandatory—but align minor versions across CI or builds fail mysteriously at runtime.

Authentication across micro-frontends

Single auth context in the shell; MFEs never parse JWTs independently unless they are separately deployed SPAs with their own OIDC client (usually worse). The shell refreshes tokens and exposes a stable API.

Shared auth context

// shell/auth/AuthProvider.tsx
import { createContext, useContext, useEffect, useMemo, useState, type ReactNode } from 'react';

type User = { sub: string; orgId: string; roles: string[] };
type AuthState =
  | { status: 'loading' }
  | { status: 'authenticated'; user: User; accessToken: string }
  | { status: 'anonymous' };

const AuthCtx = createContext<AuthState | null>(null);

export function AuthProvider({ children }: { children: ReactNode }) {
  const [state, setState] = useState<AuthState>({ status: 'loading' });

  useEffect(() => {
    let cancelled = false;
    (async () => {
      const s = await bootstrapSession(); // silent refresh, cookie session, etc.
      if (!cancelled) setState(s);
    })();
    return () => {
      cancelled = true;
    };
  }, []);

  const value = useMemo(() => state, [state]);
  return <AuthCtx.Provider value={value}>{children}</AuthCtx.Provider>;
}

export function useShellAuth(): AuthState {
  const v = useContext(AuthCtx);
  if (!v) throw new Error('useShellAuth must run under shell');
  return v;
}

Expose a thin @org/shell-auth package that only re-exports types and hooks backed by the same context instance from the shell bundle—not a second provider.

Routing integration

The shell router owns prefixes; each MFE owns sub-routes. Example: shell matches /app/billing/* and renders a remote; inside, the MFE uses memory history or a basename-aware router.

Routing bridge

// shell/routing/MfeMount.tsx
import { useNavigate, useLocation, Routes, Route } from 'react-router-dom';
import { useEffect } from 'react';

type BridgeProps = {
  basename: string;
  children: JSX.Element;
};

/** Syncs shell URL ↔ MFE internal navigations via postMessage or shared history adapter */
export function MfeHistoryBridge({ basename, children }: BridgeProps) {
  const nav = useNavigate();
  const loc = useLocation();

  useEffect(() => {
    function onMessage(e: MessageEvent) {
      if (e.origin !== window.origin) return;
      if (e.data?.type === 'mfe:navigate' && e.data.basename === basename) {
        nav(e.data.to);
      }
    }
    window.addEventListener('message', onMessage);
    return () => window.removeEventListener('message', onMessage);
  }, [basename, nav]);

  useEffect(() => {
    window.dispatchEvent(
      new CustomEvent('shell:location', {
        detail: { pathname: loc.pathname, basename },
      }),
    );
  }, [loc.pathname, basename]);

  return children;
}
User URL: /app/billing/invoices/42 │ Shell router │ prefix=/app/billing ▼ ┌───────────────┐ │ MFE Billing │ internal: /invoices/42 (basename) └───────────────┘ │ deep link │ postMessage or shared navigate() ▼ Shell updates browser URL (bookmarkable)

Deployment pipeline

Independent CI/CD per MFE implies independent SemVer and contract tests. Minimum bar:

  1. Consumer-driven contracts (Pact) between shell and remotes for manifest URLs and exposed modules.
  2. Preview environments per PR with the shell loading the preview remote URL.
  3. Artifact registry for remoteEntry.js with immutable versions; shell pins ranges deliberately.

Manifest pinning

Never load “latest” remoteEntry in production. The shell should resolve billing@1.4.x to an exact build hash recorded in a manifest service so rollbacks are flipping a pointer, not redeploying the shell.

Performance implications

  • Waterfall loading — shell parses, then fetches remoteEntry, then chunks. Mitigate with rel=preload for critical remotes and route-based prefetching on intent (hover).
  • Duplicate dependencies — lodash, date libs, and charting stacks multiply. Enforce shared federation config and dependency budgets per MFE.
  • Runtime integration cost — postMessage bridges and extra error boundaries add overhead; measure INP on navigation boundaries.

When not to migrate

If team size and deploy pain do not justify integration complexity, modular monolith + package boundaries often delivers 80% of the benefit with 20% of the operational load.

Cross-cutting observability during migration

Treat each strangler boundary as a release candidate with its own SLO slice: shell TTFB, remote load time (time-to-remoteEntry), and post-hydration INP. Tag RUM with mfe_id and manifest_sha so regressions roll up to the owning squad instead of flooding a central frontend on-call.

Telemetry contract (sketch)

// shell/telemetry/mfeMetrics.ts
export function recordRemoteLoad(mfeId: string, durationMs: number, ok: boolean) {
  // Caller marks start when import() begins; we mark end after remote resolves.
  const name = `mfe:${mfeId}:load`;
  performance.mark(`${name}:end`);
  performance.measure(name, `${name}:start`, `${name}:end`);
  navigator.sendBeacon?.(
    '/api/rum',
    JSON.stringify({ type: 'mfe_load', mfeId, durationMs, ok }),
  );
}

Correlate remote load spikes with CDN cache hit ratio and geographic latency. A sudden climb in p95 often precedes a bad shared dependency upgrade rather than application logic changes.

Deep-dive Q&A

How do you test MFE integrations locally? Senior

Use a “shell dev proxy” that serves remotes from localhost ports with liveModule Federation, plus Cypress/Playwright suites that boot shell + stubbed remotes for speed, and nightly full integration against preview URLs.

What breaks SSR with Module Federation? L6

Classic Federation assumes browser script tags. SSR needs either a server-side remote resolver, RSC boundaries, or streaming fallbacks. Align with your meta-framework’s supported story (Next multi-zone, custom Node middleware) before committing.

How do you share design tokens without coupling? Arch

Publish a versioned tokens package consumed at build time; avoid runtime CSS-in-JS singletons across bundles. Prefer CSS variables emitted once from the shell theme layer.
