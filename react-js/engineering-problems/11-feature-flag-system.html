Feature Flag System

Client vs server evaluation, rollout strategies, experiment variants, dependency graphs, stale flag hygiene, render performance, React context design, and SSR-safe hydration patterns for production flag platforms.

Architecture lens

Flags are configuration snapshots versioned by etag or revision. The React layer should consume an immutable snapshot per request (SSR) or per tick (client poll/stream), never ad hoc fetches inside arbitrary components.

Client-side evaluation for latency

Ship a compact rules payload (JSON or WASM evaluator) so toggles resolve in microseconds. For security-sensitive flags, evaluate on the server only and pass boolean props—never embed secret gating rules in the bundle.

// flagEngine.ts — pure evaluation, easy to unit test
export type FlagRule =
  | { type: 'const'; value: boolean }
  | { type: 'pct'; key: string; rollout: number } // 0-100
  | { type: 'segment'; segment: string };

export type FlagSnapshot = {
  revision: string;
  rules: Record<string, FlagRule> | ReadonlyMap<string, FlagRule>;
  segments: Record<string, (ctx: EvalContext) => boolean>;
};

export type EvalContext = { userId: string; attrs: Record<string, unknown> };

export function evalFlag(
  name: string,
  snapshot: FlagSnapshot,
  ctx: EvalContext,
  salt: string
): boolean {
  const rule = snapshot.rules instanceof Map ? snapshot.rules.get(name) : snapshot.rules[name];
  if (!rule) return false;

  switch (rule.type) {
    case 'const':
      return rule.value;
    case 'segment': {
      const fn = snapshot.segments[rule.segment];
      return fn ? fn(ctx) : false;
    }
    case 'pct': {
      const bucket = stableBucket(`${salt}:${ctx.userId}:${rule.key}`, 100);
      return bucket < rule.rollout;
    }
    default:
      return false;
  }
}

function stableBucket(key: string, mod: number): number {
  let h = 2166136261;
  for (let i = 0; i < key.length; i++) {
    h ^= key.charCodeAt(i);
    h = Math.imul(h, 16777619);
  }
  return Math.abs(h) % mod;
}

Server-side flags and API fetch

On the client, poll or SSE with If-None-Match. When the revision changes, swap snapshot atomically and notify subscribers once. Avoid per-flag HTTP calls—batch always.

// flagClient.ts
export class FlagClient {
  private snapshot: FlagSnapshot | null = null;
  private listeners = new Set<() => void>();

  constructor(private baseUrl: string) {}

  subscribe(cb: () => void) {
    this.listeners.add(cb);
    return () => this.listeners.delete(cb);
  }

  private emit() {
    this.listeners.forEach((l) => l());
  }

  async refresh(etag?: string) {
    const res = await fetch(`${this.baseUrl}/flags`, {
      headers: etag ? { 'If-None-Match': etag } : {},
    });
    if (res.status === 304) return;
    const next = (await res.json()) as FlagSnapshot;
    this.snapshot = Object.freeze(next);
    this.emit();
  }

  getSnapshot() {
    return this.snapshot;
  }
}
Consistency

For A/B analysis, persist the assignment server-side the first time a user hits the experiment, then return the same variant on subsequent evaluations. Pure client hashing drifts if you change percentages mid-flight.

Gradual rollouts: percentage and segments

Percentage rollouts use a stable hash of (userId, flagKey, salt) so users do not flip on every page load. Segment rollouts compose predicates (country, plan, internal tester list). Order of operations: kill switch (const false) overrides everything.

export function rolloutPlan(
  flag: string,
  ctx: EvalContext,
  pct: number,
  salt: string
): boolean {
  if (pct <= 0) return false;
  if (pct >= 100) return true;
  const b = stableBucket(`${salt}:${ctx.userId}:${flag}`, 10_000);
  return b < pct * 100; // basis points
}

A/B testing: variants as flags

Represent experiments as enum variants resolved by the same engine. React renders variant === 'B' ? .... Track exposure with an impression event fired once per session per experiment (guard with useRef).

export function useExperiment(name: string): 'control' | 'variant' {
  const { evalVariant } = useFeatureFlags();
  const logged = useRef(false);

  const v = evalVariant(name);

  useEffect(() => {
    if (logged.current) return;
    logged.current = true;
    analytics.track('experiment_impression', { name, variant: v });
  }, [name, v]);

  return v;
}

Flag dependencies

Model dependencies as a DAG: newCheckoutUi requires checkoutV2Api. Evaluate in topological order; cycles are a build-time error in your flag registry CI.

// dependencyResolver.ts
export function resolveWithDeps(
  flags: string[],
  snapshot: FlagSnapshot,
  ctx: EvalContext,
  salt: string,
  deps: Record<string, string[]>
): Record<string, boolean> {
  const memo: Record<string, boolean> = {};
  const visit = (name: string): boolean => {
    if (name in memo) return memo[name];
    const parents = deps[name] ?? [];
    const ok = parents.every((p) => visit(p)) && evalFlag(name, snapshot, ctx, salt);
    memo[name] = ok;
    return ok;
  };
  return Object.fromEntries(flags.map((f) => [f, visit(f)]));
}
Kill switches

Emergency-off flags should bypass dependency checks or short-circuit first. Document the precedence rule in one place—otherwise on-call flips a parent flag and nothing happens because a child dependency still reads false.

// staleFlagDetector.ts — CI script sketch
import { execSync } from 'child_process';

export function findOrphanFlagKeys(declared: string[]) {
  const src = execSync('git ls-files "*.tsx" "*.ts"', { encoding: 'utf8' })
    .split('\n')
    .filter(Boolean)
    .map((f) => require('fs').readFileSync(f, 'utf8'))
    .join('\n');

  return declared.filter((key) => !src.includes(`'${key}'`) && !src.includes(`"${key}"`));
}

Performance: minimizing re-renders

Do not put the entire snapshot in a context value that changes reference on every poll. Split into:

  • FlagSnapshotContext (stable until revision changes)
  • useSyncExternalStore for subscribing components
  • Per-flag selectors with memoized equality
import {
  createContext,
  useContext,
  useMemo,
  useSyncExternalStore,
  type ReactNode,
} from 'react';

type Ctx = { client: FlagClient };

const FlagCtx = createContext<Ctx | null>(null);

export function FeatureFlagProvider({
  client,
  children,
}: {
  client: FlagClient;
  children: ReactNode;
}) {
  const value = useMemo(() => ({ client }), [client]);
  return <FlagCtx.Provider value={value}>{children}</FlagCtx.Provider>;
}

export function useFeatureFlag(name: string): boolean {
  const ctx = useContext(FlagCtx);
  if (!ctx) throw new Error('FeatureFlagProvider missing');

  return useSyncExternalStore(
    (cb) => ctx.client.subscribe(cb),
    () => {
      const snap = ctx.client.getSnapshot();
      const evalCtx = readEvalContextFromSession(); // your auth hook
      return snap ? evalFlag(name, snap, evalCtx, snap.revision) : false;
    },
    () => false // SSR fallback; see next section
  );
}

SSR: snapshots and hydration mismatch prevention

The server must pass the same flag snapshot to the client in the HTML (serialized JSON). On hydrate, the client uses that bootstrap before opening SSE. If remote flags differ seconds later, prefer soft update (post-paint) for non-critical UI; for layout-changing flags, accept flicker or gate rendering until client confirms.

// SSR-safe provider
export function createBootstrapSnapshot(): FlagSnapshot {
  if (typeof window === 'undefined') {
    return globalThis.__FLAGS_FROM_SSR__;
  }
  return window.__FLAG_BOOTSTRAP__;
}

export function HydrationSafeFlags({
  bootstrap,
  client,
  children,
}: {
  bootstrap: FlagSnapshot;
  client: FlagClient;
  children: ReactNode;
}) {
  // Initialize client with same revision as SSR
  useMemo(() => {
    (client as any).hydrate?.(bootstrap);
  }, [client, bootstrap]);

  return <FeatureFlagProvider client={client}>{children}</FeatureFlagProvider>;
}
SSR HTML contains: <script>window.__FLAG_BOOTSTRAP__ = {...rev:42...}</script> Server render uses rev 42 Client hydrate reads rev 42 → DOM matches After idle: refresh → rev 43 (optional re-render)
Staff takeaway

Your flag system is really a config delivery plane. Invest in schema migration, typed codegen (flags.gen.ts), and CI that fails when code references undeclared keys. The React hooks are the easy part.

Typed codegen and registry CI

Generate a union type of flag keys from the control plane export (flags.json). ESLint custom rule or TypeScript template literal types prevent typos. When a flag is archived, CI removes the key from the union and breaks builds until call sites are deleted—intentional pressure to delete dead branches.

// flags.gen.ts (generated — do not edit)
export const FLAG_KEYS = [
  'checkout_redesign',
  'billing_new_tax',
  'kill_switch_payments',
] as const;

export type FlagKey = (typeof FLAG_KEYS)[number];

export function assertFlagKey(k: string): asserts k is FlagKey {
  if (!(FLAG_KEYS as readonly string[]).includes(k)) {
    throw new Error(`Unknown flag key: ${k}`);
  }
}
// useTypedFlag.ts
import type { FlagKey } from './flags.gen';

export function useTypedFlag(key: FlagKey): boolean {
  assertFlagKey(key);
  return useFeatureFlag(key);
}

Streaming updates: SSE vs long poll

For near-real-time flag changes, Server-Sent Events keeps a single connection per tab. Reconnect with Last-Event-ID. On snapshot apply, diff revisions: if only unrelated flags changed, skip React updates by comparing derived booleans per subscribed key.

export function subscribeFlagsSSE(url: string, onRevision: (s: FlagSnapshot) => void) {
  const es = new EventSource(url, { withCredentials: true });
  es.addEventListener('flags', (e) => {
    const snap = JSON.parse((e as MessageEvent).data) as FlagSnapshot;
    onRevision(snap);
  });
  es.onerror = () => {
    es.close();
    setTimeout(() => subscribeFlagsSSE(url, onRevision), 2000 + Math.random() * 1000);
  };
  return () => es.close();
}
Multivariate flags

Boolean flags are a subset. Multivariate payloads (JSON config per variant) should be validated with the same schema on server and client. Store the resolved config in the snapshot, not just the variant name, to avoid double-eval divergence.

Edge cases: anonymous users and bot traffic

Before userId exists, derive a stable anonymous distinct id from a first-party cookie or localStorage UUID created on first visit. Bots may churn ids—exclude them from percentage experiments via a server-side bot score or lightweight heuristics (headless UA, zero interaction).

Anonymous ──▶ assign anonId (cookie) Logged in ──▶ merge assignments: map anonId → userId (server sticky table; client receives merged snapshot)
export type EvalContext = {
  userId?: string;
  anonId: string;
  attrs: Record<string, unknown>;
};

export function stableSubject(ctx: EvalContext): string {
  return ctx.userId ?? ctx.anonId;
}
Privacy

Do not send raw email or phone in flag evaluation context from the browser. Hash or tokenize identifiers; segments should resolve server-side when possible.

Interview Q&A

How do you prevent layout shift when a flag flips after load? L6

Reserve space with skeletons, defer mounting until flags resolve for above-the-fold content, or use CSS that accommodates both states. For marketing hero experiments, server-assign variant and inline critical CSS for the chosen branch only.

Why useSyncExternalStore for flags? Senior

It is the supported pattern for external stores in React 18 concurrent rendering: tearing is avoided when the store updates during render. Context alone is easy to misuse with frequent snapshot swaps.
