Building a Performance Monitoring System

Real User Monitoring, Core Web Vitals, budgets, regression detection, and A/B performance at staff-engineering depth.

Architecture lens

RUM is not “drop in Google Analytics.” At scale you own sampling, PII boundaries, batching, backpressure, and statistical rigor so dashboards reflect user pain—not developer laptops.

Real User Monitoring (RUM) architecture

RUM instruments the client, aggregates in an ingestion tier, and materializes rollups for dashboards and alerts. The critical design choice is what runs in the browser versus what runs server-side: browsers should do minimal work; the pipeline does enrichment, joins with deploy metadata, and anomaly detection.

Collector responsibilities

  • Schema versioning — every payload includes schema: 3 so you can evolve fields without breaking parsers.
  • Clock skew — send client_ts and let the server record server_ts; compute offset per session for ordering.
  • Sampling — deterministic hash on sessionId for 1–5% baseline + 100% for “synthetic” or internal users.
  • Privacy — strip query strings, hash user IDs, never send raw DOM; route PII through your DLP review.
Why batch?

Single-metric beacons amplify connection overhead and can starve low-end devices. Coalesce into ~2–8KB batches flushed on visibilitychange, pagehide, or idle callbacks—with exponential backoff on 429/5xx.

Custom metrics: PerformanceObserver and User Timing

The Performance Timeline API gives sub-millisecond resolution for long tasks, resource timing, and navigation. performance.mark / performance.measure bridge React lifecycle to observable durations (e.g., “hydration complete”).

RUM collector (browser)

// rum-collector.ts — minimal production-shaped client
type RumEvent =
  | { type: 'cwv'; name: 'LCP' | 'CLS' | 'INP'; value: number; id: string }
  | { type: 'custom'; name: string; durationMs: number; detail?: Record<string, unknown> };

const QUEUE: RumEvent[] = [];
let flushTimer: number | null = null;

function enqueue(e: RumEvent) {
  QUEUE.push(e);
  if (QUEUE.length >= 20) flush();
  else scheduleFlush();
}

function scheduleFlush() {
  if (flushTimer != null) return;
  flushTimer = window.setTimeout(() => {
    flushTimer = null;
    if (document.visibilityState === 'hidden') flush();
  }, 4000);
}

export async function flush() {
  if (!QUEUE.length) return;
  const batch = QUEUE.splice(0, QUEUE.length);
  const body = JSON.stringify({
    schema: 3,
    sessionId: getSessionId(),
    release: import.meta.env.VITE_RELEASE_SHA,
    events: batch,
  });
  try {
    await fetch('/api/rum', {
      method: 'POST',
      headers: { 'Content-Type': 'application/json' },
      body,
      keepalive: true,
    });
  } catch {
    QUEUE.unshift(...batch);
  }
}

function getSessionId(): string {
  const k = 'rum_sid';
  let id = sessionStorage.getItem(k);
  if (!id) {
    id = crypto.randomUUID();
    sessionStorage.setItem(k, id);
  }
  return id;
}

document.addEventListener('visibilitychange', () => {
  if (document.visibilityState === 'hidden') flush();
});

Custom metric hook (React + PerformanceObserver)

// usePerfObserver.ts
import { useEffect } from 'react';

export function useLongTaskReporter(onLongTask: (dur: number) => void) {
  useEffect(() => {
    if (!('PerformanceObserver' in window)) return;
    const po = new PerformanceObserver((list) => {
      for (const e of list.getEntries()) {
        if (e.entryType === 'longtask') {
          onLongTask((e as PerformanceEntry & { duration: number }).duration);
        }
      }
    });
    try {
      po.observe({ type: 'longtask', buffered: true } as PerformanceObserverInit);
    } catch {
      /* Safari may lack longtask */
    }
    return () => po.disconnect();
  }, [onLongTask]);
}

// Component boundary measure
export function measureHydration(componentName: string) {
  const markEnd = `${componentName}:hydrated`;
  performance.mark(markEnd);
  performance.measure(`${componentName}:hydration`, undefined, markEnd);
}

Performance budgets

Budgets turn “we should be fast” into failing CI. Split build-time budgets (bundle KB, chunk count) from runtime budgets (p75 LCP per route).

Budget type Typical gate Enforcement Rationale
Main bundle (gzip) < 180 KB initial JS webpack-bundle-analyzer + CI threshold Correlates with TTI on 4G mid-tier
LCP (lab) < 2.5s p75 Lighthouse CI on representative URLs Lab catches regressions before RUM noise
INP (field) < 200ms p75 RUM + SLO alert INP is interaction-heavy; lab is weaker
Long tasks < 3 per minute p95 RUM longtask observer Main-thread jank predictor
Budget anti-pattern

Using lab LCP alone for SPA navigations misses client-rendered routes. Pair Lighthouse URLs with web-vitals in production for each route template.

Performance budget CI check (Node)

// scripts/check-bundle-budget.mjs
import fs from 'node:fs';
import path from 'node:path';

const REPORT = path.join('dist', 'stats.json'); // from --json vite/webpack
const limits = { main: 190_000, vendor: 350_000 }; // bytes gzip approx via build plugin

const stats = JSON.parse(fs.readFileSync(REPORT, 'utf8'));
const assets = stats.assets ?? [];

function overBudget() {
  const main = assets.find((a) => a.name === 'main.js.gz');
  if (main && main.size > limits.main) {
    console.error(`main.js.gz ${main.size} > ${limits.main}`);
    return true;
  }
  return false;
}

process.exit(overBudget() ? 1 : 0);

Core Web Vitals: LCP, CLS, INP

Google’s CWV are percentile metrics over 28 days of Chrome field data—but your RUM should compute the same definitions locally for faster feedback.

  • LCP — largest contentful paint; use PerformanceObserver with type: 'largest-contentful-paint', report the last candidate before user input.
  • CLS — layout shift score; sum non-expected shifts in session windows; cap with the “session window” algorithm from the spec.
  • INP — interaction to next paint; track latency of all interactions until the next frame is painted; take worst (or high percentile) per page.
// cwv.ts — pattern aligned with web-vitals library internals
import { onLCP, onCLS, onINP } from 'web-vitals';

onLCP((m) => enqueue({ type: 'cwv', name: 'LCP', value: m.value, id: m.id }), { reportAllChanges: true });
onCLS((m) => enqueue({ type: 'cwv', name: 'CLS', value: m.value, id: m.id }));
onINP((m) => enqueue({ type: 'cwv', name: 'INP', value: m.value, id: m.id }));

Alerting and regression detection

Alerts on raw p75 spikes create pager fatigue. Prefer release-scoped comparison: for each deploy SHA, compare the subsequent 24h window to the prior baseline with a Mann–Whitney or bootstrap confidence interval on the metric delta.

Release N-1 ████████████████████░░ p75 LCP = 2.1s Release N ██████████████████████░ p75 LCP = 2.4s │ ▼ Δ = +0.3s (bootstrap 95% CI: +0.1s .. +0.5s) ───────────────────────────────────────────── IF CI lower > +0.15s AND sample > 5k → PAGE

Regression detector (pseudo-SQL + TypeScript glue)

// regression-detector.ts — compare windows per route
type Agg = { route: string; p75: number; n: number };

export function significantRegression(before: Agg, after: Agg): boolean {
  if (after.n < 5000 || before.n < 5000) return false;
  const rel = (after.p75 - before.p75) / before.p75;
  // Production: replace with proper effect size + sequential testing
  return rel > 0.12;
}

/*
-- BigQuery-style rollup
SELECT
  APP_INFO.release AS rel,
  route,
  APPROX_QUANTILES(lcp_ms, 100)[OFFSET(75)] AS p75_lcp,
  COUNT(*) AS n
FROM rum.events
WHERE event_date BETWEEN @start AND @end
GROUP BY 1, 2;
*/
Sequential testing

If you evaluate the same metric every hour after deploy, you inflate false positives. Use sequential methods (e.g., SPRT-style gates) or fixed windows only.

A/B testing performance impact

Variant assignment must tag RUM events so you can compare the same cohort under two experiences. Watch for Simpson’s paradox: overall INP may improve while mobile degrades if traffic mix shifts.

// ab-tagging.ts
declare global {
  interface Window {
    __EXP_FLAGS__?: Record<string, string>;
  }
}

export function rumContext() {
  return {
    experiments: window.__EXP_FLAGS__ ?? {},
    variant: window.__EXP_FLAGS__?.['checkout_v2'] ?? 'control',
  };
}

Analysis tip: stratify by deviceMemory, connection.effectiveType, and geography. Use CUPED or diff-in-diff when you have pre-period data per user.

React-specific

Concurrent features and transitions change INP distributions: a deferred update may improve perceived responsiveness while shifting work later—always pair RUM with long-task counts.

Deep-dive Q&A

How do you prevent RUM from becoming a DDoS vector? L6

Authenticate optional beacons for internal tiers; for public sites use strict rate limits per IP + per session, payload size caps, schema validation, and WAF rules. Prefer edge ingestion (CDN) with anomaly detection on request rate versus unique sessions.

Why not sample uniformly random on each event? Arch

Uniform per-event sampling fragments sessions and biases journey metrics. Session-level hashing keeps user paths coherent and stabilizes statistical tests across steps.

What breaks CLS measurement in SPAs? Senior

Soft navigations do not reset CLS in the same way as full page loads unless you implement session windows per route change. Align with the soft-nav proposal or reset counters on route transitions with explicit session IDs.
