Building a Real-Time Collaborative Editor

Staff-level treatment of replicated document editing: operational transformation versus conflict-free replicated data types, WebSocket fan-out topologies, presence overlays, and the subtle mechanics of undo under concurrent edits. The goal is not a toy demo but a production-shaped mental model you can defend in system design rounds.

Architect Invariant

Pick a replication contract before choosing libraries: “Users always converge to the same document state” is necessary but insufficient—you must also specify intention preservation (do concurrent inserts stay ordered sensibly?) and latency semantics (local edits are optimistic; remote edits may arrive out of order).

1. CRDT vs Operational Transformation

Operational Transformation (OT)

OT assumes operations are transformed against one another so that different orderings of the same logical ops yield identical document states. Classical centralized editors (e.g., early Google Docs) used a server as the sequencer: clients send ops; server assigns a monotonic revision; clients transform in-flight ops when they see replays.

OT with central server (total order) Client A: Ins(3,"X") ----\ +--> Server (seq) --> broadcast transformed ops Client B: Del(2,1) ----/ A transforms pending Ins against incoming Del before applying server order

Conflict-Free Replicated Data Types (CRDTs)

CRDTs bake commutativity/associativity/idempotency into the data structure. Text CRDTs (RGA, YATA, etc.) assign stable identifiers to characters so concurrent inserts merge deterministically without a central total order. Metadata (tombstones, anchors) trades space for correctness under partition.

CRDT merge (two replicas, no central order) Replica 1: [a][b] + insert "x" after a --> [a][x][b] Replica 2: [a][b] + insert "y" after a --> [a][y][b] Merge: same IDs for a,b ; tie-break x vs y by (replicaId, counter) rule Result: [a][x][y][b] OR [a][y][x][b] — deterministic globally
Dimension OT (classic) CRDT (text)
Ordering assumption Server or sequencer provides total order Partial order; merge function is total on states
Space O(ops) log with compaction milestones O(content + tombstones); compaction policies matter
Offline Harder without replay infrastructure Natural fit for op/state sync on reconnect
Complexity hotspot Transform functions for each op pair Identifier allocation, garbage collection
React integration Bind editor surface to op log + cursor map Bind Yjs/Automerge doc to editor; subscribe to updates
Interview Trap

Claiming “CRDTs need no server” is misleading at Staff level. You still want authorization, persistence, snapshotting, and backpressure. Peer-to-peer CRDTs shift the problem—they do not remove operational concerns.

/**
 * G-Counter CRDT (grow-only counter) — illustrates lattice merge.
 * Not for rich text, but shows merge commutativity in code.
 */
type ReplicaId = string;

export class GCounter {
  private readonly counts = new Map<ReplicaId, number>();

  increment(replica: ReplicaId, n = 1): void {
    this.counts.set(replica, (this.counts.get(replica) ?? 0) + n);
  }

  value(): number {
    let s = 0;
    for (const v of this.counts.values()) s += v;
    return s;
  }

  merge(other: GCounter): GCounter {
    const out = new GCounter();
    const keys = new Set([...this.counts.keys(), ...other.counts.keys()]);
    for (const k of keys) {
      out.counts.set(
        k,
        Math.max(this.counts.get(k) ?? 0, other.counts.get(k) ?? 0),
      );
    }
    return out;
  }
}

2. WebSocket Architecture for Real-Time Sync

At scale, a naive “one giant room per document” WebSocket on a single Node process fails on connection count and GC pressure. Production shapes include: sticky sessions with horizontal scale, Redis Pub/Sub or NATS for cross-node fan-out, snapshot + delta catch-up for reconnects, and backpressure via per-connection queues with drop policies for non-critical traffic (typing cursors vs document ops).

+------------------+ | Load Balancer | +--------+---------+ | +-----------------+------------------+ v v v +---------+ +---------+ +---------+ | WS Pod | | WS Pod | ... | WS Pod | +----+----+ +----+----+ +----+----+ | | | +----------------+-----------------+ v +--------+---------+ | Pub/Sub backbone | +--------+---------+ v +--------+---------+ | Persistence tier | | (op log / CRDT) | +------------------+
Wire Protocol Hygiene

Version your envelopes (v: 2), include docId, seq or hv (hybrid logical clock), and separate control (hello, auth, subscribe) from data frames. Consider MessagePack or protobuf on the wire if JSON becomes hot path—React does not care, but your gateway CPU will.

// Typed WebSocket envelope (client <-> sync service)
type WsEnvelope =
  | { t: 'hello'; v: number; capabilities: string[] }
  | { t: 'sub'; docId: string; fromSeq: number }
  | { t: 'op'; docId: string; seq: number; op: EncodedOp }
  | { t: 'snap'; docId: string; seq: number; state: Uint8Array }
  | { t: 'ack'; clientMutationId: string; seq: number }
  | { t: 'err'; code: string; message: string };

export function encode(e: WsEnvelope): ArrayBuffer {
  return new TextEncoder().encode(JSON.stringify(e)).buffer;
}

export function decode(buf: ArrayBuffer): WsEnvelope {
  const text = new TextDecoder().decode(buf);
  return JSON.parse(text) as WsEnvelope;
}

3. Conflict Resolution Strategies

Beyond text, collaborative apps embed structured fields (title, metadata). Strategies include: field-level last-write-wins with HLC timestamps, register CRDTs (multi-value, then resolve), user-visible merge (fork + choose), and locking for rare high-stakes sections. The Staff move is to separate human meaning from machine merge: CRDTs resolve machine state; product may still show a human merge UI.

Strategy When it works Sharp edge
LWW with wall clock Low contention keys Clock skew causes surprise winners
HLC / hybrid logical time Need causal ordering without tight clock sync Operational complexity in multi-region
Explicit branch + merge Legal / financial annotations UX cost; not invisible like CRDT

4. Cursor Presence

Presence (caret, selection, username color) should be ephemeral and loss-tolerant: use a high-frequency local throttle (e.g., 30–60Hz max), send relative positions or CRDT anchors—not raw pixel coordinates that break on layout. On the React side, isolate fast-moving presence in a subtree that does not rerender the full document.

// React: presence hook sketch (anchors, not pixels)
import { useCallback, useEffect, useRef, useState } from 'react';

export type Anchor = { blockId: string; offset: number };

export function useDocPresence(args: {
  docId: string;
  send: (p: { userId: string; anchor: Anchor; sel?: Anchor }) => void;
  userId: string;
}) {
  const [peers, setPeers] = useState<Record<string, Anchor>>({});
  const raf = useRef<number | null>(null);

  const publishCaret = useCallback(
    (anchor: Anchor, sel?: Anchor) => {
      if (raf.current != null) cancelAnimationFrame(raf.current);
      raf.current = requestAnimationFrame(() => {
        args.send({ userId: args.userId, anchor, sel });
      });
    },
    [args],
  );

  useEffect(() => () => {
    if (raf.current != null) cancelAnimationFrame(raf.current);
  }, []);

  return { peers, publishCaret, setPeers };
}

5. Undo / Redo Under Concurrency

Local undo is not globally invertible when others edit: classic approach is undo as local operation history scoped to the user’s causal window (Subtext, Google Docs–style), or inverse ops that are transformed against concurrent remote ops before application. CRDT libraries often expose transaction boundaries; Yjs uses undo stacks bound to a shared doc but scoped via captured snapshots.

Local undo stack vs global log User A: type "hello" -> stack: [Ins*h, Ins*e, ...] Remote: delete first char while A undoes Naive pop would violate intention -> must transform inverse against remote Staff answer: "My undo applies to MY operations in MY causal history"
Product + Formalism

Interviewers reward stating that global undo is ambiguous for collaborative text. Prefer language like “undo last user-visible intent” with clear UX when remote ops interleave: either block undo momentarily, scope undo, or show a non-modal explanation—engineering and design co-own the contract.

Testing Matrix

Fuzz concurrent schedules: fix a seed, generate random interleavings of insert/delete from N peers, assert convergence and invariants (no lost user-visible characters unless explicitly deleted). Property tests catch transform bugs faster than storybook clicks.

7. React Binding & Render Isolation

Binding a CRDT document to React is deceptively simple: subscribe to updates and call setState. The Staff trap is main-thread churn—every keystroke from remote peers can trigger a full React tree reconciliation if you map the entire doc into props. Patterns: (1) keep the editor imperative (ProseMirror/Slate) and treat React as chrome around it; (2) use a tiny useSyncExternalStore shim for read consistency during concurrent rendering; (3) batch remote updates with startTransition so local input stays urgent.

React layers around an imperative editor +------------------+ +---------------------------+ | Toolbar (React) | | Comment sidebar (React) | +------------------+ +---------------------------+ | Editor surface (imperative) <---> CRDT doc (Y.Doc) | | - applies local ops immediately | | - applies remote ops via plugin / observer | +------------------------------------------------------+ | Awareness (React overlay, memoized by peer id) | +------------------------------------------------------+ 
// useSyncExternalStore sketch: subscribe to Yjs without tearing in React 18
import { useCallback, useSyncExternalStore } from 'react';
import * as Y from 'yjs';

export function useYTextSnapshot(ytext: Y.Text): string {
  const subscribe = useCallback(
    (onStoreChange: () => void) => {
      const obs = () => onStoreChange();
      ytext.observe(obs);
      return () => ytext.unobserve(obs);
    },
    [ytext],
  );
  const getSnapshot = useCallback(() => ytext.toString(), [ytext]);
  const getServerSnapshot = useCallback(() => ytext.toString(), [ytext]);
  return useSyncExternalStore(subscribe, getSnapshot, getServerSnapshot);
}

Backpressure & fairness

When one peer pastes a megabyte, others should not starve. Implement per-document queues with chunked apply and requestIdleCallback or time-sliced application for non-local updates. Surface a subtle progress indicator if catch-up exceeds a threshold—users tolerate latency if the system is honest.

8. Persistence & Snapshot Cadence

Operational logs grow without bound. Production systems compact into periodic snapshots plus a tail of ops. The Staff detail: snapshotting must be coordinated with in-flight clients—use monotonic seq and teach clients to request snap + fromSeq on gap detection. Store authoritative snapshots in object storage; keep hot tail in Redis or embedded RocksDB depending on your scale story.

Storage timeline for one document snap@1000 --------+ tail ops 1001..1048 (hot) | | | v v v S3 / GCS metadata index pub/sub fan-out
Security: Authorization at Op Level

Subscribing to a room must not imply permission to mutate. Validate each op against ACLs (comment-only vs editor). For CRDTs, malicious clients can craft ops—treat the sync service as a policy enforcement point, not a dumb pipe.

Interview Q&A

Why can operational transformation be tricky with arbitrary rich text? L6

OT requires well-defined transforms for every operation pair on your model (split nodes, marks, block structure). Rich text increases the Cartesian product of op types; invariants like TP1/TP2 become hard to maintain. That is why many teams adopt CRDT text or reduce the problem domain (plain text, block-based JSON with limited ops).

How do you handle WebSocket authentication without leaking tokens to logs? Senior

Prefer short-lived tokens in the first frame or cookie-based session upgrade with SameSite restrictions. Rotate credentials on reconnect. Never log full URLs with query tokens. On the server, bind socket to principal after auth and reject late unauthorized frames.

What metrics prove your collaborative stack is healthy? Architect

Track p95/p99 apply-to-visible latency, reconnect catch-up time, op queue depth per document, percentage of ops requiring snapshot fallback, CRDT compaction cadence, and awareness update rates. Correlate with long-task timings in the editor thread.

Explain how you would roll out CRDT sync without downtime for existing OT-based clients. L6

Use a dual-write or adapter phase: server translates incoming OT ops into CRDT ops (or maintains parallel state machines) until all clients are upgraded. Alternatively, snapshot the document into CRDT form at a revision boundary, freeze edits briefly, migrate, resume. The interview win is naming the cutover risk and validation (round-trip tests, shadow reads comparing OT vs CRDT projections).

When is WebRTC data channel preferable to WebSocket for collaboration? Senior

WebRTC shines for mesh/SFU-assisted peer data when you want to reduce server relay costs for large payloads (e.g., binary assets) or ultra-low latency cursor sharing in LAN-like conditions. Most text collaboration still centralizes through a sync service for persistence and auth; WebRTC is often layered as an optimization, not the sole source of truth.
