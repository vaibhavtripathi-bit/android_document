Offline-First Architecture in React

Service Worker caching strategies, IndexedDB schema evolution, durable mutation queues, merge policies on reconnect, optimistic UI, Background Sync, and progressive enhancement—framed the way Staff interviewers expect: explicit failure modes, idempotency, and UX contracts when the network is a lie.

Truth You Own

Offline-first is not “cache static assets.” It is a consistency story between three stores: server, on-device durable store, and React UI state. Name what happens for duplicate retries, partial writes, and version skew after the app sat offline for a week.

1. Service Worker Caching Strategies

Cache-first

Check cache, fallback to network. Ideal for hashed static assets and app shells. Risk: stale HTML referencing deleted chunks after deploy—mitigate with versioned entry URLs or skip caching HTML entirely.

Network-first

Try network, fallback to cache. Useful for API JSON that should be fresh but may degrade offline. Pair with short TTL metadata in cache entries.

Stale-while-revalidate

Return cache immediately, update cache in background for next visit. Great for read-mostly documents; requires careful UI labeling when data might be stale.

// Workbox-style handler sketch (TypeScript)
declare const caches: CacheStorage;

export async function staleWhileRevalidate(req: Request, cacheName: string): Promise<Response> {
  const cache = await caches.open(cacheName);
  const cached = await cache.match(req);
  const networkPromise = fetch(req)
    .then((res) => {
      if (res.ok) void cache.put(req, res.clone());
      return res;
    })
    .catch(() => cached);

  return cached ?? (await networkPromise);
}

Security

Never cache personalized responses under shared cache keys without Vary or user-scoped namespacing. Sensitive GETs should often bypass SW entirely. Pair with strict CSP to reduce SW-tampering blast radius.

2. IndexedDB: Schema & Migrations

3. Sync Queue Structure & Retry Logic

Queue entries must carry idempotency keys the server understands. Retries use exponential backoff with jitter; cap attempts and surface a human-visible “needs attention” state. Order matters for dependent mutations—use a partial order or explicit dependency list.

Queue with dependencies [ createDraft ] --> [ attachFile ] --> [ publish ] | | +---------- on failure, block ------- 
// Retry policy (TypeScript)
export function nextBackoffMs(attempt: number): number {
  const base = Math.min(60_000, 1000 * 2 ** attempt);
  const jitter = Math.random() * 250;
  return base + jitter;
}

export async function drainQueue(db: AppDB, send: (m: QueuedMutation) => Promise<void>) {
  const pending = await db.mutations.where('status').equals('pending').sortBy('createdAt');
  for (const m of pending) {
    try {
      await send(m);
      await db.mutations.delete(m.id);
    } catch {
      await db.mutations.update(m.id, { attempts: m.attempts + 1, status: 'failed' });
      await new Promise((r) => setTimeout(r, nextBackoffMs(m.attempts)));
    }
  }
}

ETags & Preconditions

Use If-Match with version vectors or ETags for optimistic locking. On mismatch, fetch server state and run your merge function. React Query and TanStack Query expose patterns for this via meta and custom mutations.

// Conflict resolver (functional core)
export type Entity = { id: string; version: number; title: string };

export function resolveTitleConflict(local: Entity, remote: Entity): Entity {
  if (local.version === remote.version) return remote;
  // Example: field-level merge with user prompt hook
  if (local.title !== remote.title) {
    return {
      ...remote,
      title: mergeStringsWithMarker(local.title, remote.title),
      version: remote.version + 1,
    };
  }
  return remote;
}

function mergeStringsWithMarker(a: string, b: string): string {
  return `<<< local\n${a}\n===\n${b}\n>>> remote`;
}

6. Background Sync API

registration.sync.register('outbox') lets the browser flush when connectivity returns. Support is Chromium-heavy; iOS Safari limitations mean you still need foreground retry on online events and periodic visibilitychange wake-ups. Never rely solely on Background Sync for safety-critical workflows.

Double Submission

Users double-tap buttons; SW retries overlap with page retries. Idempotency keys must be generated once per user intent and stored before network attempt begins.

8. Offline-Aware Hook

// Combine navigator.onLine + SW ping for smarter status
import { useEffect, useState } from 'react';

export function useConnectivity() {
  const [online, setOnline] = useState(() => navigator.onLine);

  useEffect(() => {
    const onUp = () => setOnline(true);
    const onDown = () => setOnline(false);
    window.addEventListener('online', onUp);
    window.addEventListener('offline', onDown);
    return () => {
      window.removeEventListener('online', onUp);
      window.removeEventListener('offline', onDown);
    };
  }, []);

  useEffect(() => {
    let cancelled = false;
    async function probe() {
      try {
        await fetch('/api/health', { method: 'HEAD', cache: 'no-store' });
        if (!cancelled) setOnline(true);
      } catch {
        if (!cancelled) setOnline(false);
      }
    }
    const id = setInterval(probe, 30_000);
    return () => {
      cancelled = true;
      clearInterval(id);
    };
  }, []);

  return online;
}
Staff Follow-Up

Be ready to compare local-first (ElectricSQL, Replicache-shaped) vs optimistic SPA. Local-first pushes replication down-stack; optimistic SPAs keep canonical authority on server with shorter implementation path but weaker multi-tab guarantees unless you add BroadcastChannel coordination.

9. Multi-Tab Coordination

Two tabs writing the same IndexedDB queue without coordination can duplicate sends. Mitigations: navigator.locks.request around drain, leader election via BroadcastChannel heartbeat, or service worker as single writer. Mention test matrices across tabs and profiles.

10. Storage Quotas & Eviction

Browsers may evict your origin under pressure. navigator.storage.estimate() exposes usage and quota—surface warnings in settings when crossing thresholds. Separate hot data (queue + recent entities) from cold archives users can re-fetch. For Safari PWA, persistence may require user gesture prompts—design UX accordingly.

11. TanStack Query & Offline Cache

Server-state libraries cache normalized responses in memory by default—pair with persisted persisters (IndexedDB) for reload survival. Mutations should write through to the durable queue before firing the network layer so refresh does not lose intent. Configure networkMode: 'offlineFirst' where supported and reconcile refetchOnReconnect with your SW strategy to avoid thundering herds.

// Sketch: mutation writes queue then optimistic cache update
import { useMutation, useQueryClient } from '@tanstack/react-query';

export function useOfflineSave(db: AppDB) {
  const qc = useQueryClient();
  return useMutation({
    mutationFn: async (vars: { id: string; patch: Record<string, unknown> }) => {
      const idem = crypto.randomUUID();
      await db.mutations.add({
        id: crypto.randomUUID(),
        createdAt: Date.now(),
        entity: vars.id,
        payload: vars.patch,
        idempotencyKey: idem,
        attempts: 0,
        status: 'pending',
      });
      return fetch(`/api/items/${vars.id}`, {
        method: 'PATCH',
        headers: { 'Idempotency-Key': idem, 'Content-Type': 'application/json' },
        body: JSON.stringify(vars.patch),
      });
    },
    onMutate: async (vars) => {
      await qc.cancelQueries({ queryKey: ['item', vars.id] });
      const prev = qc.getQueryData(['item', vars.id]);
      qc.setQueryData(['item', vars.id], (old: any) => ({ ...old, ...vars.patch }));
      return { prev };
    },
    onError: (_e, vars, ctx) => {
      qc.setQueryData(['item', vars.id], ctx?.prev);
    },
  });
}

Interview Q&A

How do you test service worker updates without wedging users? L6

Use skipWaiting + clients.claim() only when safe; otherwise prompt users to refresh. Version your caches and delete old names in activate. Canary cohorts validate new SW scripts before full rollout.

What is the difference between eventual consistency and causal consistency in offline apps? Architect

Eventual consistency guarantees all replicas converge if they stop receiving updates—no promise about ordering interleavings users observe meanwhile. Causal consistency preserves happens-before relationships: if a user reads a write, later reads cannot observe a state before that write’s causes. Offline note apps sometimes aim for causal+ per document.

Why store idempotency keys client-side before the fetch fires? Senior

Crash or kill-tab immediately after network send but before persisting acknowledgment can produce an unknown outcome. If the key was durable first, retry is safe. If not, you may duplicate server-side effects.

How does a service worker interact with CORS and credentials? L6

The SW’s fetch inherits the page’s credentials mode when initiated from a controlled client, but opaque responses limit JS inspection. Cross-origin caching requires careful cache keying and may fail for no-cors opaque responses. Prefer same-origin BFF endpoints for sensitive JSON to keep caches inspectable and policy-bound.
