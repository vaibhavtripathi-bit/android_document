Component Architecture

Q1: What is the compound components pattern and when would you use it over render props?
Compound components share implicit state through React context, providing a declarative API similar to native HTML elements like <select>/<option>. The parent manages state, and children consume it via context. Use compound components when you want a clean, composable API where sub-components work together (e.g., Accordion, Tabs, Menu). Use render props when the consumer needs full control over what's rendered with the shared state, or when the component is used in class components where hooks aren't available. Compound components provide better DX for common cases; render props provide maximum flexibility.
Q2: Explain the evolution from HOCs to hooks. Why did the community move away from HOCs?
HOCs had three fundamental problems: (1) Wrapper hell — stacking 5+ HOCs created deeply nested component trees that were hard to debug in DevTools. (2) Prop collisions — multiple HOCs injecting props with the same name (e.g., loading) caused silent overwrites. (3) Static method loss — wrapping a component lost its static methods unless you used hoist-non-react-statics. (4) Ref forwarding — refs attached to the wrapper, not the inner component. Hooks solved all of these by extracting stateful logic into composable functions with explicit return values, no wrappers, and no prop injection.
Q3: What is the "headless UI" pattern and how does it relate to inversion of control?
Headless UI provides behavior and state management without rendering opinions. The consumer provides all UI through "prop getter" functions (e.g., getInputProps(), getMenuProps()). It's maximum inversion of control — the component library handles complex logic (keyboard navigation, ARIA, state machines) while the consumer controls every visual aspect. Libraries like Radix UI, Headless UI, and Downshift use this pattern. The tradeoff is more work for consumers in exchange for unlimited customization. It's ideal for design system primitives that must adapt to any visual design.
Q4: How would you design a component API that's both easy to use for simple cases and flexible for complex ones?
Use a layered API approach: (1) Simple API — sensible defaults so the common case requires minimal props (<Button>Click</Button>). (2) Slot props — accept ReactNode for customizable sections (leftIcon, rightIcon). (3) Render props — for cases needing access to internal state (renderItem). (4) Compound components — for full composition control (<Select.Trigger>). (5) Headless hook — for consumers who need to build from scratch (useSelect()). Spread remaining props to the root element, use forwardRef, and accept a className prop for styling overrides.
Q5: Explain the "asChild" pattern from Radix UI. Why is it preferred over the "as" polymorphic prop?
The as prop changes the rendered element type (<Button as="a" href="/">), but it creates TypeScript complexity — the component's prop types must change based on the as value. asChild instead renders the child element directly, merging the component's props and ref onto the child using @radix-ui/react-slot. This avoids polymorphic type gymnastics, works naturally with component composition, and lets consumers use any component (not just HTML elements) as the rendered element. It's simpler to implement, type-safe, and more composable.
Q6: When would you still use a render prop pattern in a modern React codebase?
Render props remain useful in three scenarios: (1) When the component needs to conditionally render based on shared state without the consumer handling it — e.g., React Query's <QueryErrorResetBoundary>. (2) In class component codebases where hooks aren't available. (3) When you need to share rendering logic (not just state) between components — the render prop receives both data and UI helpers. However, for most new code, custom hooks are preferred because they're simpler and more composable.
Q7: How do you handle component identity in dynamic UIs where components are conditionally rendered?
Component identity is determined by position in the tree and the key prop. When conditionally rendering, use explicit keys to control identity: same key = preserve state, different key = reset state. For example, a user profile form should use key={userId} so switching users resets form state. Without keys, React may reuse a fiber node for a different logical component, causing stale state bugs. The key prop is the primary tool for controlling component lifecycle in dynamic UIs.

Design Systems

Q8: How would you architect a design system for 50+ consuming applications?
Start with the token pipeline: Figma → Token Studio → Style Dictionary → CSS variables/JS constants. Build a monorepo (Turborepo/Nx) with packages for tokens, core UI components, icons, and hooks. Use compound components and headless patterns for flexibility. Document with Storybook, test visually with Chromatic, and enforce a11y with axe-core. Publish via Changesets with semantic versioning. Provide codemods for breaking changes. Track adoption metrics (which teams use which components, version distribution). The key insight: a design system is a product, not a project — it needs a dedicated team, roadmap, and support channel.
Q9: Compare CSS custom properties vs styled-components ThemeProvider for theming.
CSS custom properties: zero runtime cost, work across micro-frontends (any framework), excellent SSR support, enable theme switching without JS re-renders, but lack TypeScript type safety. styled-components ThemeProvider: full TypeScript support, co-located with component logic, but adds ~12KB runtime, requires SSR setup (ServerStyleSheet), and causes re-renders on theme change. The industry trend favors CSS custom properties as the theming primitive because they work universally and have zero performance cost. Even styled-components projects benefit from defining tokens as CSS variables.
Q10: How do you handle breaking changes in a design system used by 50+ teams?
Multi-phase approach: (1) Add @deprecated JSDoc tags with migration guides. (2) Support old and new APIs simultaneously for 2-3 minor versions. (3) Provide codemods (jscodeshift) for automated migration. (4) Track adoption metrics to know when old APIs can be removed. (5) Communicate via changelog, Slack, and migration guides. (6) Offer "office hours" for teams needing help. Never ship a major version without codemods. The goal is zero-effort migration for consuming teams.
Q11: What is the role of design tokens in a design system?
Design tokens are the atomic values (colors, spacing, typography, shadows, radii) that bridge design tools and code. They're defined in a platform-agnostic format (JSON following the Design Token Community Group spec), then transformed by Style Dictionary into platform-specific outputs: CSS variables for web, JS constants for React Native, Swift constants for iOS. Tokens ensure consistency across platforms and enable automated syncing from Figma via Token Studio. Semantic tokens (e.g., color.error → color.red.500) add a layer of abstraction that makes theme switching possible.

Micro-Frontends

Q12: When should you use micro-frontends vs a monolithic React app?
Micro-frontends solve organizational problems, not technical ones. Use them when: (1) Multiple teams (3+) need to deploy independently. (2) Teams own distinct product areas with clear boundaries. (3) Deployment coupling is causing velocity problems. Don't use them for: (1) Single team or small team. (2) Tightly coupled features that share lots of state. (3) When the added complexity (shared deps, routing, communication) outweighs the benefits. The key question: "Is deployment coupling our bottleneck?" If yes, consider micro-frontends. If no, a well-structured monolith with code ownership is simpler.
Q13: Explain Module Federation and why singleton: true is critical for React.
Module Federation (Webpack 5) allows applications to dynamically load code from other applications at runtime. Each app exposes modules via remoteEntry.js, and the host loads them on demand. singleton: true for React is critical because React hooks rely on a module-scoped ReactCurrentDispatcher variable. Two React instances mean two dispatchers, and hooks in one instance can't access state from the other — causing the "Invalid hook call" error. With singleton: true, Webpack ensures only one copy of React is loaded, shared across all micro-frontends.
Q14: How do you handle cross-micro-frontend communication?
Three patterns, from loosely to tightly coupled: (1) Custom Events — window.dispatchEvent(new CustomEvent('cart:updated', { detail })). Loosest coupling, works across frameworks, but untyped. (2) Event Bus — typed publish/subscribe system exposed on window. Better DX with TypeScript, still decoupled. (3) Shared Store — Zustand or Redux store loaded by the shell and shared via Module Federation. Tightest coupling but simplest for complex shared state. Recommendation: use custom events for cross-cutting concerns (auth, navigation), shared store only for tightly coupled state (cart across product and checkout MFEs).
Q15: How do you handle routing across micro-frontends?
The shell owns BrowserRouter and top-level routes. Each MFE uses MemoryRouter for internal navigation (never two BrowserRouters — they fight over history.pushState). Cross-MFE navigation happens via custom events: the MFE dispatches a navigate event, and the shell listens and calls navigate() on its router. URL synchronization: the shell updates the browser URL based on which MFE is active and its internal route. Deep linking works because the shell parses the URL and passes the relevant sub-path to the MFE.

Data Architecture

Q16: Explain the repository pattern in frontend architecture.
The repository pattern provides an abstraction layer between the data source (API) and the application. It handles: (1) Data transformation — mapping API response shapes (snake_case) to domain models (camelCase). (2) Business logic — computing derived fields, validating data. (3) API encapsulation — components never see API endpoints or response formats. Benefits: when the backend changes field names, you update one mapper, not 50 components. When migrating APIs, you swap the repository implementation. When adding caching or offline support, the repository is the integration point. It's the frontend equivalent of a backend service layer.
Q17: Why normalize data in the frontend? What problems does it solve?
Normalization eliminates data duplication by storing entities in flat lookup tables (like a relational database). Without normalization, the same user object might appear in 10 different API responses — updating the user's name requires finding and updating all 10 copies. With normalization, there's one canonical copy per entity, referenced by ID everywhere. This prevents inconsistencies, simplifies updates, and reduces memory usage. Tools: normalizr for manual normalization, Redux Toolkit's createEntityAdapter for Redux, or React Query's built-in cache which handles this automatically for most cases.
Q18: How would you implement optimistic updates that handle failures gracefully?
React Query's useMutation provides the pattern: (1) onMutate: cancel in-flight queries, snapshot current data, apply optimistic update to cache. (2) onError: roll back to snapshot, show error toast. (3) onSettled: invalidate queries to refetch fresh data regardless of success/failure. For offline scenarios, add a mutation queue (persisted to IndexedDB) that processes pending mutations when connectivity returns. The key insight: optimistic updates must be idempotent and reversible. Never optimistically update data that can't be rolled back (e.g., financial transactions).
Q19: Compare WebSocket, SSE, and polling for real-time data.
WebSocket: bidirectional, persistent connection, supports binary data. Best for chat, collaborative editing, gaming. Overhead: manual reconnection, doesn't multiplex over HTTP/2. SSE: server-to-client only, auto-reconnects, multiplexes over HTTP/2, text-only. Best for notifications, live feeds, stock tickers. Simpler than WebSocket for one-way data. Polling: simplest, works everywhere, no persistent connection. Best when data changes infrequently or exact real-time isn't needed. Use short polling (fixed interval) or long polling (server holds request until data changes). Choose based on: directionality (bidirectional → WS), simplicity (one-way → SSE), and fallback (unreliable network → polling).
Q20: How would you adapt CQRS for a React frontend?
CQRS separates read and write models. In React: Query side — React Query/SWR handles reads with stale-while-revalidate caching, background refresh, and deduplication. Command side — useMutation handles writes with optimistic updates and cache invalidation. The separation manifests as: query factory functions (read-only, cacheable) and mutation hooks (write-only, invalidate queries on success). Benefits: reads and writes can be optimized independently, queries can use different endpoints than mutations (e.g., read from CDN-cached API, write to primary), and the mental model is clearer.

Error Handling

Q21: Design an error handling strategy for a large React application.
Layered approach: (1) Categorize errors: network, HTTP 4xx/5xx, validation, business logic, runtime. (2) Error boundaries: root (full-page fallback), route-level (page fallback with navigation), widget-level (hide or placeholder). (3) Global handlers: window.onerror and unhandledrejection for uncaught errors. React 19's createRoot callbacks for caught/uncaught/recoverable errors. (4) Retry: exponential backoff with jitter for transient errors. (5) Circuit breaker: stop calling failing services, use cached fallback. (6) Reporting: Sentry with context (user, page, breadcrumbs). (7) Graceful degradation: critical features show errors, nice-to-have features hide silently.
Q22: What errors do React error boundaries NOT catch?
Error boundaries don't catch: (1) Event handler errors — use try/catch inside handlers. (2) Async errors — setTimeout, Promise rejections aren't caught during render. (3) SSR errors — server-side rendering uses try/catch, not boundaries. (4) Errors in the boundary itself — need a parent boundary. (5) Errors in useEffect — these are async and not caught by boundaries (though React does catch them in development). Design your error strategy knowing these gaps — combine boundaries with global handlers and try/catch for complete coverage.
Q23: Explain the circuit breaker pattern in frontend context.
The circuit breaker prevents cascading failures by stopping requests to a failing service. Three states: CLOSED (normal, counting failures), OPEN (rejecting requests, using fallback), HALF_OPEN (allowing limited requests to test recovery). In frontend: if a search API fails 5 times in a row, the circuit opens — subsequent searches use cached results instead of hitting the failing API. After a timeout (30s), it enters half-open and allows a few test requests. If they succeed, it closes; if they fail, it reopens. This prevents users from waiting for timeouts on a known-failing service and reduces load on struggling backends.

Scalability

Q24: Why is feature-based folder structure better than type-based for large codebases?
Type-based (components/, hooks/, utils/) scatters related code across directories — changing one feature touches 5+ folders. Feature-based (features/auth/, features/cart/) colocates all code for a feature — components, hooks, API calls, types, tests. Benefits: (1) Changes are localized — PRs touch fewer directories. (2) Code ownership is clear — team X owns features/search/. (3) Module boundaries are enforceable — ESLint rules prevent cross-feature imports except through public APIs (index.ts). (4) Deletion is easy — remove the feature directory. The shared/ directory holds truly cross-cutting code (design system, utilities).
Q25: What are the tree-shaking implications of barrel exports?
Barrel exports (index.ts re-exporting everything) can break tree-shaking because: (1) Wildcard re-exports (export *) create a single module that the bundler may not be able to partially eliminate. (2) If any re-exported module has side effects (CSS imports, top-level code), the entire barrel is included. (3) Webpack's tree-shaking is less effective with barrel files than with direct imports. Solutions: mark packages as "sideEffects": false, use named re-exports instead of wildcards, use direct imports for large modules, or use Next.js's optimizePackageImports. Measure impact with webpack-bundle-analyzer.
Q26: How would you implement a feature flag system for a React application?
Architecture: (1) Provider — fetches flags from API on app load, provides via context. (2) Hook — useFeatureFlag('flag-name') returns boolean. (3) Component — <FeatureGate flag="x"> for declarative gating. (4) SSR support — flags fetched server-side and serialized to client. (5) Refresh — periodic polling or SSE for real-time flag updates. (6) Stale flag cleanup — ESLint rule or script that finds flags not in the active flag list. (7) Analytics integration — automatically track which variant users see. Use LaunchDarkly, Unleash, or build in-house for simple boolean flags.
Q27: How do you prevent state management from becoming a bottleneck at scale?
Separate state by type: (1) Server state → React Query/SWR (not Redux). (2) URL state → URL params/search params (not Redux). (3) Client state → feature-scoped stores (Zustand/Jotai), not one global store. (4) Form state → react-hook-form (local, not global). The anti-pattern is putting everything in Redux — server cache, form state, UI state, URL state all in one global store. This creates coupling, unnecessary re-renders, and makes features impossible to develop independently. Each state type has an optimal home; use the right tool for each.

Accessibility

Q28: How do you handle focus management in a single-page application?
SPAs break natural focus management because route changes don't trigger page loads. Solutions: (1) On route change, move focus to the main content area (<main tabIndex={-1}>) and update document.title. (2) Implement skip links as the first focusable element. (3) For modals, implement focus traps — Tab cycles within the modal, Escape closes it, and focus returns to the trigger on close. (4) For dynamic content (search results, notifications), use aria-live regions to announce changes. (5) For composite widgets (tabs, menus, toolbars), use roving tabindex — one item has tabIndex={0}, others have tabIndex={-1}, arrow keys move focus.
Q29: What is the first rule of ARIA?
"Don't use ARIA if you can use a native HTML element." Native elements (<button>, <input>, <select>, <a>) have built-in keyboard behavior, focus management, and screen reader announcements. ARIA is for custom widgets that have no native equivalent (combobox, tree view, tab panel). Using ARIA on a <div> to make it a button requires reimplementing click, Enter, Space, focus, and screen reader behavior — all of which <button> provides for free. ARIA adds semantics but not behavior.
Q30: How would you make accessibility an architectural concern rather than an afterthought?
Embed a11y into every layer: (1) Design system — every component ships with keyboard navigation, ARIA, and focus styles. Contrast ratios are enforced by design tokens. (2) Linting — eslint-plugin-jsx-a11y catches issues in the editor. (3) Component tests — jest-axe runs on every component. (4) Storybook — a11y addon shows violations during development. (5) CI pipeline — axe-core integration tests run on every PR. (6) Code review — a11y is a merge requirement. (7) Quarterly audits — manual screen reader testing. This systemic approach prevents the "accessibility sprint" anti-pattern where a11y is bolted on before launch.
Q31: Explain aria-live regions and when you'd use polite vs assertive.
aria-live regions announce dynamic content changes to screen readers. polite: waits until the screen reader finishes its current announcement before reading the new content. Use for non-urgent updates: search result counts, form validation messages, status updates. assertive: interrupts the current announcement immediately. Use only for urgent, time-sensitive information: error alerts, session expiration warnings. Implementation: render a hidden live region at the app root, update its textContent programmatically. Clear the content first, then set it in a requestAnimationFrame to ensure the screen reader detects the change.

Cross-Cutting Architecture

Q32: You're joining a team with a 3-year-old React codebase. How do you assess its architecture?
Systematic assessment: (1) Bundle analysis — webpack-bundle-analyzer reveals dead code, duplicate dependencies, oversized chunks. (2) Dependency audit — outdated packages, security vulnerabilities, unnecessary dependencies. (3) State management — is server state in Redux? Is form state global? Are there unnecessary re-renders? (4) Component patterns — HOC wrapper hell? Prop drilling? Context overuse? (5) Testing — coverage, test quality, testing patterns. (6) Performance — Core Web Vitals, code splitting, lazy loading. (7) Developer experience — build times, hot reload speed, TypeScript adoption. (8) Module boundaries — are features coupled? Can teams work independently? Prioritize fixes by impact on user experience and developer velocity.
Q33: How do you balance consistency and autonomy across multiple frontend teams?
Define a "golden path" with escape hatches: (1) Mandatory: design system tokens, a11y standards, error reporting, security practices. (2) Recommended: shared tooling configs (ESLint, TypeScript, testing), state management patterns, API client. (3) Team choice: internal state management, CSS approach (within design system), testing tools, internal folder structure. Enforce mandatory items via CI checks and linting. Recommend via starter templates and documentation. Allow choice for team-internal decisions. The key: mandate outcomes (a11y compliance, performance budgets), not implementations.
Q34: How would you design a frontend platform that supports 10 product teams?
Platform layers: (1) Foundation — design system, shared utilities, API client, error reporting, analytics abstraction. (2) Tooling — shared ESLint/TypeScript configs, CI/CD templates, performance monitoring. (3) Infrastructure — CDN, feature flags, A/B testing, deployment pipeline. (4) Documentation — architecture decision records (ADRs), onboarding guides, API docs. (5) Support — office hours, Slack channel, migration guides. The platform team's job is to make the right thing the easy thing. Teams should be able to spin up a new feature with npx create-feature and get all the platform benefits automatically.
Q35: What metrics would you track to measure frontend architecture health?
Four categories: (1) User experience — Core Web Vitals (LCP, INP, CLS), error rates, crash-free sessions. (2) Developer experience — build time, hot reload time, PR merge time, time to first meaningful contribution for new developers. (3) Code health — bundle size trends, dependency freshness, TypeScript coverage, test coverage, lint violations. (4) Organizational — deployment frequency, change failure rate, mean time to recovery (DORA metrics). Track trends, not absolute numbers. Set budgets and alert on regressions. Dashboard these metrics and review in architecture reviews.
Q36: How do you make architectural decisions that affect multiple teams?
Use Architecture Decision Records (ADRs): (1) Write an RFC describing the problem, proposed solution, alternatives considered, and tradeoffs. (2) Share with affected teams for feedback (1-2 week review period). (3) Hold a decision meeting with stakeholders. (4) Document the decision, rationale, and dissenting opinions. (5) Create a migration plan with timeline. (6) Provide tooling (codemods, starter templates) to make adoption easy. The key: decisions should be reversible when possible, and the cost of reversal should be documented. Staff engineers make decisions that are "two-way doors" — easy to reverse if wrong.
Q37: How do you handle technical debt in a fast-moving product organization?
Make debt visible and quantifiable: (1) Track — maintain a tech debt backlog with estimated impact (developer hours wasted per week). (2) Categorize — deliberate (accepted tradeoff) vs accidental (didn't know better) vs bit rot (dependencies aged). (3) Budget — allocate 15-20% of sprint capacity to debt reduction. (4) Opportunistic — fix debt in the area you're already working on (Boy Scout Rule). (5) Strategic — schedule large debt items as projects with clear ROI. (6) Prevent — code review standards, linting, architecture reviews. The Staff engineer's job is to translate tech debt into business impact: "This debt costs us 2 developer-weeks per quarter in workarounds."
Q38: Design an analytics abstraction layer for a React application.
Create a vendor-agnostic AnalyticsService class that dispatches events to multiple providers (Google Analytics, Mixpanel, Amplitude). Expose via React context and useAnalytics() hook. Features: (1) Auto page tracking — hook into router for page views. (2) Event typing — TypeScript interface for all event names and properties. (3) Batching — queue events and flush periodically to reduce network calls. (4) Consent — respect user privacy preferences, only send to consented providers. (5) Debug mode — log events to console in development. (6) Error isolation — analytics failures never crash the app. The abstraction lets you swap vendors without touching application code.

Advanced Architecture

Q39: How would you implement offline-first architecture in a React app?
Layers: (1) Service Worker — cache-first for static assets, network-first for API calls with cache fallback. (2) IndexedDB — local database for entity storage, synced with server. (3) Mutation Queue — pending writes stored in IndexedDB, processed when online. (4) Conflict Resolution — last-write-wins for simple data, CRDTs for collaborative data. (5) UI — show online/offline indicator, mark pending items, handle sync errors. (6) Background Sync — Service Worker API to retry failed requests. The key challenge is conflict resolution — what happens when two users edit the same item offline? The strategy depends on the data model and business requirements.
Q40: Explain event sourcing adapted for frontend architecture.
Instead of storing current state, store a sequence of events (actions) that produced the state. Frontend adaptation: (1) Event log — every user action is recorded as an immutable event ({ type: 'ITEM_ADDED', payload: {...}, timestamp }). (2) State derivation — current state is computed by replaying events (like Redux reducers). (3) Benefits — complete audit trail, time-travel debugging, undo/redo for free, offline sync (send event log, not state). (4) Challenges — event log grows unbounded (snapshot periodically), event schema evolution, replay performance. Redux is essentially event sourcing — actions are events, reducers are projections. The pattern is most valuable for collaborative and audit-sensitive applications.
Q41: How do you evaluate whether to build vs buy a solution?
Decision framework: (1) Core competency — is this a differentiator for your product? Build if yes. (2) Maintenance cost — building means maintaining forever. Libraries get free bug fixes and features. (3) Customization needs — if you need 80%+ customization, building may be cheaper than fighting a library. (4) Team expertise — do you have the domain knowledge to build it well? (5) Time to market — buying ships faster. (6) Vendor risk — what if the library is abandoned? (7) License — is the license compatible? For most infrastructure (auth, analytics, error reporting), buy. For core product features, build. For UI components, use headless libraries and build the visual layer.
Q42: How would you migrate a large React app from class components to hooks?
Incremental migration: (1) Don't rewrite — migrate file-by-file as you touch them. (2) Extract logic first — move class component logic into custom hooks that can be used by both class (via wrapper) and function components. (3) Convert leaf components first — start with simple, stateless components, then work up to complex stateful ones. (4) Use codemods — react-codemod handles simple conversions. (5) Test coverage — ensure tests exist before converting. (6) No big bang — class and function components coexist fine. (7) Track progress — dashboard showing % of components converted. Timeline: 6-12 months for a large codebase, done alongside feature work.
