The Evolution of Component Patterns

React's component model has evolved through several paradigms. Understanding this evolution is essential because legacy codebases contain all of these patterns, and Staff engineers must decide when to migrate and when to leave working patterns alone.

Timeline of React Component Patterns ═══════════════════════════════════════════════════════════════ 2013-2015 Mixins (createClass) └─ Shared behavior via object merging └─ Problem: name collisions, implicit dependencies 2015-2017 Higher-Order Components (HOCs) └─ Shared behavior via component wrapping └─ Problem: wrapper hell, prop collision, static hoisting 2016-2018 Render Props └─ Shared behavior via function-as-child └─ Problem: callback nesting, "pyramid of doom" 2018-2019 Hooks └─ Shared behavior via composable functions └─ Current standard for logic reuse 2020+ Compound Components + Headless UI └─ Shared behavior + flexible rendering └─ State-of-the-art for library design

Presentational vs Container Components

Dan Abramov introduced this pattern in 2015, then deprecated it in 2019 when hooks made it unnecessary. Understanding both the pattern and why it was deprecated reveals how React thinking evolved.

// CONTAINER: owns state and data fetching
class UserListContainer extends React.Component {
  state = { users: [], loading: true };

  componentDidMount() {
    fetchUsers().then(users =>
      this.setState({ users, loading: false })
    );
  }

  render() {
    return (
      <UserList
        users={this.state.users}
        loading={this.state.loading}
      />
    );
  }
}

// PRESENTATIONAL: pure rendering, no side effects
function UserList({ users, loading }) {
  if (loading) return <Spinner />;
  return (
    <ul>
      {users.map(u => <li key={u.id}>{u.name}</li>)}
    </ul>
  );
}

// WITH HOOKS: the separation becomes unnecessary
function UserList() {
  const { users, loading } = useUsers(); // custom hook
  if (loading) return <Spinner />;
  return (
    <ul>
      {users.map(u => <li key={u.id}>{u.name}</li>)}
    </ul>
  );
}
Architect Insight

The presentational/container split was a workaround for class components lacking a composition mechanism for stateful logic. Hooks solved this by letting you extract stateful logic into reusable functions. However, the principle of separating data concerns from rendering concerns remains valid — it just manifests as custom hooks + components rather than container + presentational components.

Compound Components Pattern

Compound components provide an API where components work together implicitly through shared state, similar to how <select> and <option> work in HTML. The parent manages state, and children consume it via context.

// Compound component: Accordion
const AccordionContext = React.createContext(null);

function Accordion({ children, defaultOpen = null }) {
  const [openItem, setOpenItem] = useState(defaultOpen);

  const toggle = useCallback((id) => {
    setOpenItem(prev => prev === id ? null : id);
  }, []);

  const value = useMemo(
    () => ({ openItem, toggle }),
    [openItem, toggle]
  );

  return (
    <AccordionContext.Provider value={value}>
      <div role="region">{children}</div>
    </AccordionContext.Provider>
  );
}

function AccordionItem({ id, children }) {
  const { openItem, toggle } = useContext(AccordionContext);
  const isOpen = openItem === id;

  return (
    <div data-state={isOpen ? 'open' : 'closed'}>
      {React.Children.map(children, child =>
        React.isValidElement(child)
          ? React.cloneElement(child, { isOpen, onToggle: () => toggle(id) })
          : child
      )}
    </div>
  );
}

function AccordionTrigger({ isOpen, onToggle, children }) {
  return (
    <button
      aria-expanded={isOpen}
      onClick={onToggle}
    >
      {children}
    </button>
  );
}

function AccordionContent({ isOpen, children }) {
  if (!isOpen) return null;
  return <div role="region">{children}</div>;
}

// Attach sub-components for clean API
Accordion.Item = AccordionItem;
Accordion.Trigger = AccordionTrigger;
Accordion.Content = AccordionContent;

// Usage — clean, declarative API
<Accordion defaultOpen="faq-1">
  <Accordion.Item id="faq-1">
    <Accordion.Trigger>What is React?</Accordion.Trigger>
    <Accordion.Content>A UI library...</Accordion.Content>
  </Accordion.Item>
  <Accordion.Item id="faq-2">
    <Accordion.Trigger>What are hooks?</Accordion.Trigger>
    <Accordion.Content>Functions that...</Accordion.Content>
  </Accordion.Item>
</Accordion>

Render Props Pattern

Render props invert control by letting the consumer decide what to render with the shared state. This was the dominant pattern before hooks.

// Render prop: Mouse position tracker
function MouseTracker({ render }) {
  const [position, setPosition] = useState({ x: 0, y: 0 });

  useEffect(() => {
    const handler = (e) => setPosition({ x: e.clientX, y: e.clientY });
    window.addEventListener('mousemove', handler);
    return () => window.removeEventListener('mousemove', handler);
  }, []);

  return render(position);
}

// Usage
<MouseTracker
  render={({ x, y }) => (
    <div>Mouse: {x}, {y}</div>
  )}
/>

// Function-as-child variant
function MouseTracker({ children }) {
  const [position, setPosition] = useState({ x: 0, y: 0 });
  // ... same effect
  return children(position);
}

<MouseTracker>
  {({ x, y }) => <div>Mouse: {x}, {y}</div>}
</MouseTracker>

// Hook equivalent (preferred today)
function useMousePosition() {
  const [position, setPosition] = useState({ x: 0, y: 0 });
  useEffect(() => {
    const handler = (e) => setPosition({ x: e.clientX, y: e.clientY });
    window.addEventListener('mousemove', handler);
    return () => window.removeEventListener('mousemove', handler);
  }, []);
  return position;
}

function MyComponent() {
  const { x, y } = useMousePosition();
  return <div>Mouse: {x}, {y}</div>;
}
When Render Props Still Win

Render props remain useful when you need to conditionally render based on the shared state without the consumer needing to handle it. Libraries like React Query's <QueryErrorResetBoundary> still use render props for this reason. They also work in class components where hooks aren't available.

Higher-Order Components (HOCs)

HOCs are functions that take a component and return a new component with enhanced behavior. They were the primary code reuse mechanism before hooks.

// HOC: withAuth
function withAuth(WrappedComponent) {
  function AuthenticatedComponent(props) {
    const { user, loading } = useAuth();

    if (loading) return <Spinner />;
    if (!user) return <Redirect to="/login" />;

    return <WrappedComponent {...props} user={user} />;
  }

  // Preserve display name for DevTools
  AuthenticatedComponent.displayName =
    `withAuth(${WrappedComponent.displayName || WrappedComponent.name})`;

  // Hoist non-React static methods
  hoistNonReactStatics(AuthenticatedComponent, WrappedComponent);

  return AuthenticatedComponent;
}

// Usage
const ProtectedDashboard = withAuth(Dashboard);

// The "wrapper hell" problem with multiple HOCs:
const EnhancedComponent = withAuth(
  withTheme(
    withRouter(
      withIntl(
        connect(mapState, mapDispatch)(MyComponent)
      )
    )
  )
);
// DevTools shows: withAuth(withTheme(withRouter(withIntl(Connect(MyComponent)))))

The outermost HOC wins. HOC Problem: Ref Forwarding ═══════════════════════════════════════════ <EnhancedComponent ref={myRef} /> → ref attaches to the HOC wrapper, NOT the inner component → Must use React.forwardRef in every HOC HOC Problem: Static Method Loss ═══════════════════════════════════════════ MyComponent.getInitialProps = async () => { ... }; const Enhanced = withAuth(MyComponent); Enhanced.getInitialProps // undefined! Hooks as the Composition Primitive

Hooks solved every problem that HOCs and render props had: no wrapper hell, no prop collisions, no ref forwarding issues, and composable by nature.

// Composing multiple hooks — clean and explicit
function useAuthenticatedData(endpoint) {
  const { user, token } = useAuth();
  const { data, error, loading } = useFetch(endpoint, {
    headers: { Authorization: `Bearer ${token}` },
    enabled: !!user,
  });
  const analytics = useAnalytics();

  useEffect(() => {
    if (data) analytics.track('data_loaded', { endpoint });
  }, [data, endpoint, analytics]);

  return { user, data, error, loading };
}

// No wrapper hell, no prop collisions, explicit data flow
function Dashboard() {
  const { user, data, loading } = useAuthenticatedData('/api/dashboard');
  const theme = useTheme();
  const { t } = useTranslation();

  if (loading) return <Skeleton theme={theme} />;
  return <DashboardView data={data} user={user} t={t} />;
}
Architect Insight — Hook Composition Rules

When designing custom hooks for a large team, follow these principles: (1) Each hook should do one thing well. (2) Hooks should be composable — a hook can call other hooks. (3) Return values should be destructurable objects, not arrays (except for simple state pairs). (4) Accept configuration objects for extensibility. (5) Provide escape hatches for advanced use cases.

Headless UI Components

Headless components provide behavior and state management without any rendering opinions. The consumer provides all the UI. This is the pattern used by Radix UI, Headless UI, Downshift, and React Aria.

// Headless Combobox hook
function useCombobox({ items, onSelect, filterFn }) {
  const [query, setQuery] = useState('');
  const [isOpen, setIsOpen] = useState(false);
  const [activeIndex, setActiveIndex] = useState(-1);
  const inputRef = useRef(null);
  const listRef = useRef(null);

  const filtered = useMemo(
    () => filterFn ? items.filter(i => filterFn(i, query)) : items,
    [items, query, filterFn]
  );

  const getInputProps = useCallback(() => ({
    ref: inputRef,
    value: query,
    role: 'combobox',
    'aria-expanded': isOpen,
    'aria-autocomplete': 'list',
    'aria-activedescendant': activeIndex >= 0
      ? `option-${activeIndex}` : undefined,
    onChange: (e) => {
      setQuery(e.target.value);
      setIsOpen(true);
    },
    onFocus: () => setIsOpen(true),
    onKeyDown: (e) => {
      switch (e.key) {
        case 'ArrowDown':
          e.preventDefault();
          setActiveIndex(i => Math.min(i + 1, filtered.length - 1));
          break;
        case 'ArrowUp':
          e.preventDefault();
          setActiveIndex(i => Math.max(i - 1, 0));
          break;
        case 'Enter':
          if (activeIndex >= 0) {
            onSelect(filtered[activeIndex]);
            setIsOpen(false);
          }
          break;
        case 'Escape':
          setIsOpen(false);
          break;
      }
    },
  }), [query, isOpen, activeIndex, filtered, onSelect]);

  const getListProps = useCallback(() => ({
    ref: listRef,
    role: 'listbox',
  }), []);

  const getOptionProps = useCallback((index) => ({
    id: `option-${index}`,
    role: 'option',
    'aria-selected': index === activeIndex,
    onClick: () => {
      onSelect(filtered[index]);
      setIsOpen(false);
    },
  }), [activeIndex, filtered, onSelect]);

  return {
    query, isOpen, activeIndex, filtered,
    getInputProps, getListProps, getOptionProps,
  };
}

// Consumer has FULL control over rendering
function MyCombobox() {
  const {
    isOpen, filtered,
    getInputProps, getListProps, getOptionProps,
  } = useCombobox({
    items: countries,
    onSelect: (c) => console.log(c),
    filterFn: (item, q) => item.name.toLowerCase().includes(q.toLowerCase()),
  });

  return (
    <div className="my-custom-combobox">
      <input {...getInputProps()} className="my-input" />
      {isOpen && (
        <ul {...getListProps()} className="my-dropdown">
          {filtered.map((item, i) => (
            <li key={item.id} {...getOptionProps(i)} className="my-option">
              <Flag code={item.code} /> {item.name}
            </li>
          ))}
        </ul>
      )}
    </div>
  );
}

Slot Pattern

The slot pattern lets consumers inject content into predefined "slots" of a component layout, similar to Vue's named slots or Web Components' <slot> elements.

// Slot-based Card component
function Card({ header, footer, actions, children }) {
  return (
    <div className="card">
      {header && <div className="card-header">{header}</div>}
      <div className="card-body">{children}</div>
      {actions && <div className="card-actions">{actions}</div>}
      {footer && <div className="card-footer">{footer}</div>}
    </div>
  );
}

// Usage — consumer fills slots
<Card
  header={<h3>User Profile</h3>}
  actions={
    <>
      <Button variant="secondary">Cancel</Button>
      <Button variant="primary">Save</Button>
    </>
  }
  footer={<p className="text-muted">Last updated: today</p>}
>
  <UserForm />
</Card>

// Advanced: Typed slots with TypeScript
interface DialogProps {
  trigger: React.ReactElement;
  title: React.ReactNode;
  description?: React.ReactNode;
  actions: (close: () => void) => React.ReactNode; // render function slot
  children: React.ReactNode;
}

function Dialog({ trigger, title, description, actions, children }: DialogProps) {
  const [open, setOpen] = useState(false);
  const close = useCallback(() => setOpen(false), []);

  return (
    <>
      {React.cloneElement(trigger, { onClick: () => setOpen(true) })}
      {open && (
        <div className="dialog-overlay">
          <div className="dialog" role="dialog" aria-modal="true">
            <h2>{title}</h2>
            {description && <p>{description}</p>}
            <div className="dialog-body">{children}</div>
            <div className="dialog-actions">{actions(close)}</div>
          </div>
        </div>
      )}
    </>
  );
}

Inversion of Control

Inversion of control (IoC) in component design means the component doesn't decide what to render — the consumer does. This is the unifying principle behind render props, headless UI, and compound components.

// LOW IoC: Component controls everything
<Select options={options} />
// Consumer can't customize option rendering, grouping, or layout

// MEDIUM IoC: Slots for customization
<Select
  options={options}
  renderOption={(opt) => <CustomOption {...opt} />}
  renderGroup={(group) => <GroupHeader {...group} />}
/>

// HIGH IoC: Compound components
<Select value={value} onChange={onChange}>
  <Select.Trigger>
    <Select.Value placeholder="Choose..." />
  </Select.Trigger>
  <Select.Content>
    <Select.Group>
      <Select.Label>Fruits</Select.Label>
      <Select.Item value="apple">🍎 Apple</Select.Item>
      <Select.Item value="banana">🍌 Banana</Select.Item>
    </Select.Group>
  </Select.Content>
</Select>

// MAXIMUM IoC: Headless hook
const { getToggleProps, getMenuProps, getItemProps } = useSelect({
  items, selectedItem, onSelectedItemChange,
});
// Consumer builds entire UI from scratch

Component API Design for Reusability

// PRINCIPLE 1: Sensible defaults, full overrides
interface ButtonProps extends React.ButtonHTMLAttributes<HTMLButtonElement> {
  variant?: 'primary' | 'secondary' | 'ghost' | 'destructive';
  size?: 'sm' | 'md' | 'lg';
  loading?: boolean;
  leftIcon?: React.ReactNode;
  rightIcon?: React.ReactNode;
  asChild?: boolean; // Radix-style polymorphism
}

// PRINCIPLE 2: Polymorphic "as" prop
interface BoxProps<T extends React.ElementType = 'div'> {
  as?: T;
  children?: React.ReactNode;
}

type PolymorphicProps<T extends React.ElementType> =
  BoxProps<T> & Omit<React.ComponentPropsWithoutRef<T>, keyof BoxProps>;

function Box<T extends React.ElementType = 'div'>({
  as, children, ...props
}: PolymorphicProps<T>) {
  const Component = as || 'div';
  return <Component {...props}>{children}</Component>;
}

// PRINCIPLE 3: Composition over configuration
// BAD: Mega-component with dozens of props
<DataTable
  data={data}
  columns={columns}

  filterable
  paginated
  selectable
  expandable
  exportable
  virtualScroll
  stickyHeader

/>

// GOOD: Composable pieces

  <DataTable.Toolbar>

    <DataTable.Export />
  </DataTable.Toolbar>
  <DataTable.Header sticky>

Critical — The "asChild" Pattern (Radix)

Radix UI popularized asChild as an alternative to the as prop. Instead of polymorphism via prop, the component renders its child directly, merging props and refs. This avoids TypeScript complexity with polymorphic types and works better with component composition. Use @radix-ui/react-slot to implement this pattern.
