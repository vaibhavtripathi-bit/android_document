Component Library Architecture

A design system is more than a component library — it's a shared language between design and engineering. At scale, the architecture of this system determines whether teams adopt it willingly or fight against it.

Styled-Components ThemeProvider

// theme.ts
const lightTheme = {
  colors: {
    primary: { 50: '#eff6ff', 500: '#3b82f6', 900: '#1e3a8a' },
    bg: '#ffffff',
    text: '#0f172a',
  },
  space: { 1: '0.25rem', 2: '0.5rem', 4: '1rem', 8: '2rem' },
  radii: { sm: '0.25rem', md: '0.375rem', lg: '0.5rem' },
};

const darkTheme: typeof lightTheme = {
  colors: {
    primary: { 50: '#1e3a8a', 500: '#60a5fa', 900: '#eff6ff' },
    bg: '#0f172a',
    text: '#f1f5f9',
  },
  space: lightTheme.space,
  radii: lightTheme.radii,
};

// Provider
function App() {
  const [theme, setTheme] = useState(lightTheme);
  return (
    <ThemeProvider theme={theme}>
      <GlobalStyles />
      <Router />
    </ThemeProvider>
  );
}

// Component usage
const Button = styled.button`
  background: ${({ theme }) => theme.colors.primary[500]};
  padding: ${({ theme }) => `${theme.space[2]} ${theme.space[4]}`};
  border-radius: ${({ theme }) => theme.radii.md};
`;

Tailwind CSS Approach

// tailwind.config.ts
import type { Config } from 'tailwindcss';
import tokens from '@company/tokens';

const config: Config = {
  content: ['./src/**/*.{ts,tsx}'],
  theme: {
    extend: {
      colors: tokens.colors,
      spacing: tokens.spacing,
      borderRadius: tokens.radii,
      fontFamily: {
        sans: [tokens.fonts.sans, 'system-ui'],
        mono: [tokens.fonts.mono, 'monospace'],
      },
    },
  },
  plugins: [
    require('@tailwindcss/forms'),
    require('@tailwindcss/typography'),
  ],
};

// Component with Tailwind + cn utility
import { cn } from '@company/ui/utils';

interface ButtonProps extends React.ButtonHTMLAttributes<HTMLButtonElement> {
  variant?: 'primary' | 'secondary' | 'ghost';
  size?: 'sm' | 'md' | 'lg';
}

function Button({ variant = 'primary', size = 'md', className, ...props }: ButtonProps) {
  return (
    <button
      className={cn(
        'inline-flex items-center justify-center rounded-md font-medium transition-colors',
        'focus-visible:outline-none focus-visible:ring-2 focus-visible:ring-offset-2',
        {
          'bg-primary-500 text-white hover:bg-primary-600': variant === 'primary',
          'border border-gray-300 bg-white hover:bg-gray-50': variant === 'secondary',
          'hover:bg-gray-100': variant === 'ghost',
        },
        {
          'h-8 px-3 text-sm': size === 'sm',
          'h-10 px-4 text-base': size === 'md',
          'h-12 px-6 text-lg': size === 'lg',
        },
        className
      )}
      {...props}
    />
  );
}

Architect Insight

The industry trend is moving toward CSS custom properties as the theming primitive, regardless of the styling solution. Even styled-components and Tailwind projects benefit from defining tokens as CSS variables because they enable theme switching without JavaScript re-renders and work across micro-frontends that may use different styling solutions.

Design Tokens

Design tokens are the atomic values of a design system — colors, spacing, typography, shadows, etc. They bridge the gap between design tools (Figma) and code.

// tokens/colors.json (Design Token Community Group format)
{
  "color": {
    "primary": {
      "50":  { "$value": "#eff6ff", "$type": "color" },
      "100": { "$value": "#dbeafe", "$type": "color" },
      "500": { "$value": "#3b82f6", "$type": "color" },
      "900": { "$value": "#1e3a8a", "$type": "color" }
    },
    "semantic": {
      "success": { "$value": "{color.green.500}", "$type": "color" },
      "warning": { "$value": "{color.amber.500}", "$type": "color" },
      "error":   { "$value": "{color.red.500}",   "$type": "color" },
      "info":    { "$value": "{color.blue.500}",   "$type": "color" }
    }
  }
}

// Style Dictionary config for multi-platform output
// build.js
const StyleDictionary = require('style-dictionary');

StyleDictionary.extend({
  source: ['src/**/*.json'],
  platforms: {
    css: {
      transformGroup: 'css',
      buildPath: 'dist/css/',
      files: [{ destination: 'tokens.css', format: 'css/variables' }],
    },
    js: {
      transformGroup: 'js',
      buildPath: 'dist/js/',
      files: [{ destination: 'tokens.js', format: 'javascript/es6' }],
    },
    ts: {
      transformGroup: 'js',
      buildPath: 'dist/ts/',
      files: [{ destination: 'tokens.d.ts', format: 'typescript/es6-declarations' }],
    },
  },
}).buildAllPlatforms();

// Button.stories.tsx
import type { Meta, StoryObj } from '@storybook/react';
import { Button } from './Button';

const meta: Meta<typeof Button> = {
  title: 'Primitives/Button',
  component: Button,
  tags: ['autodocs'],
  argTypes: {
    variant: {
      control: 'select',
      options: ['primary', 'secondary', 'ghost', 'destructive'],
      description: 'Visual style variant',
      table: { defaultValue: { summary: 'primary' } },
    },
    size: {
      control: 'radio',
      options: ['sm', 'md', 'lg'],
    },
    disabled: { control: 'boolean' },
    loading: { control: 'boolean' },
  },
  parameters: {
    design: {
      type: 'figma',
      url: 'https://figma.com/file/xxx/Button',
    },
  },
};
export default meta;

type Story = StoryObj<typeof Button>;

export const Primary: Story = {
  args: { children: 'Button', variant: 'primary' },
};

export const AllVariants: Story = {
  render: () => (
    <div style={{ display: 'flex', gap: '1rem' }}>
      <Button variant="primary">Primary</Button>
      <Button variant="secondary">Secondary</Button>
      <Button variant="ghost">Ghost</Button>
      <Button variant="destructive">Destructive</Button>
    </div>
  ),
};

export const Loading: Story = {
  args: { children: 'Saving...', loading: true },
};

// Interaction test
export const ClickTracking: Story = {
  args: { children: 'Click me' },
  play: async ({ canvasElement, args }) => {
    const canvas = within(canvasElement);
    const button = canvas.getByRole('button');

  },
};

Accessibility in Design Systems

// Every component must meet these requirements:
// 1. Keyboard navigable
// 2. Screen reader announced
// 3. Sufficient color contrast (4.5:1 for text, 3:1 for UI)
// 4. Focus visible
// 5. Reduced motion respected

// Accessible Button with all requirements
const Button = forwardRef<HTMLButtonElement, ButtonProps>(
  ({ variant, size, loading, disabled, children, ...props }, ref) => {
    return (
      <button
        ref={ref}
        disabled={disabled || loading}
        aria-disabled={disabled || loading}
        aria-busy={loading}
        className={cn(buttonVariants({ variant, size }))}
        {...props}
      >
        {loading && (
          <Spinner
            className="mr-2"
            aria-hidden="true"  // decorative, button has aria-busy
          />
        )}
        {children}
      </button>
    );
  }
);

Critical — Breaking Changes

At scale (50+ consuming apps), breaking changes in a design system are extremely expensive. Strategies: (1) Use @deprecated JSDoc tags with migration guides. (2) Support old and new APIs simultaneously for 2-3 minor versions. (3) Provide codemods (jscodeshift) for automated migration. (4) Track adoption metrics to know when old APIs can be removed.

Interview Tip

When asked "how would you build a design system?", don't start with components. Start with: (1) Token pipeline from Figma. (2) Theming architecture (CSS vars). (3) Component API principles (compound components, headless). (4) Documentation (Storybook). (5) Testing (visual + a11y). (6) Publishing (changesets). (7) Adoption metrics. This shows Staff-level systems thinking.
