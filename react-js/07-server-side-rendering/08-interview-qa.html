Interview Q&A — Server-Side Rendering

55+ Staff-level interview questions covering SSR fundamentals, React Server Components, streaming, Next.js internals, meta-frameworks, edge rendering, and caching strategies.

SSR Fundamentals

Q1: What are the three React server rendering APIs and when would you use each? L6

renderToString (legacy): Synchronously renders to an HTML string. Blocks the event loop. No Suspense support. Use only for simple, non-streaming scenarios or legacy codebases.

renderToPipeableStream (Node.js): Modern streaming API. Supports Suspense boundaries, out-of-order streaming, and selective hydration. Use for Node.js server environments. Provides onShellReady, onAllReady, onShellError callbacks.

renderToReadableStream (Web Streams): Web Streams API equivalent for edge runtimes (Cloudflare Workers, Deno Deploy, Vercel Edge). Returns a ReadableStream. Use when deploying to edge environments that implement WHATWG Streams standard.

Q2: Explain the hydration process. What happens when React hydrates server-rendered HTML? L6

Hydration is the process of making server-rendered HTML interactive. React walks the existing DOM tree and the virtual React tree simultaneously. For each fiber node, it: (1) finds the corresponding DOM node via depth-first traversal, (2) validates that attributes and text content match (dev mode), (3) attaches event listeners via event delegation on the root container, (4) initializes component state with initial values, (5) schedules effects (useEffect callbacks). The key insight is that React reuses existing DOM nodes rather than creating new ones — it "attaches" to the server HTML. If a mismatch is detected, React logs a warning and may fall back to client-side rendering for that subtree.

Q3: What causes hydration mismatches and how do you prevent them? L6

Common causes: (1) Date.now() or Math.random() producing different values on server vs client, (2) browser-only APIs like window.innerWidth used during render, (3) locale/timezone differences between server and client, (4) browser extensions injecting DOM nodes, (5) CSS-in-JS libraries without SSR setup generating different class names. Prevention: use useEffect for client-only logic (effects don't run on server), use suppressHydrationWarning for intentional mismatches (e.g., timestamps), use useId() for stable IDs, and test with onRecoverableError callback in hydrateRoot.

Q4: Compare SSR, SSG, and ISR. When would you choose each? L6

SSG: Pages rendered at build time, served as static files. Best for content that rarely changes (docs, marketing pages). Fastest TTFB (CDN-served), zero server cost at runtime.

SSR: Pages rendered per-request. Best for personalized content, real-time data, authenticated pages. Guarantees fresh data but requires server compute for every request.

ISR: Hybrid — pages are statically generated but revalidated in the background after a configurable period. Best for content that changes periodically (product catalogs, blog posts). Combines SSG performance with bounded staleness.

The right choice depends on data freshness requirements, traffic patterns, and personalization needs. Most production apps use a mix — SSG for marketing, ISR for catalogs, SSR for dashboards.

Q5: How does streaming SSR improve performance compared to buffered SSR? L6

Buffered SSR (renderToString) waits for the entire page to render before sending any HTML. TTFB equals total render time. Streaming SSR (renderToPipeableStream) sends the shell immediately and streams Suspense boundary content as it resolves. TTFB equals shell render time (typically 10-30ms vs 100-500ms). The browser can start parsing, rendering, and even downloading CSS/JS while the server continues rendering. For pages with slow data dependencies, streaming can improve FCP by seconds. Additionally, streaming enables selective hydration — the client can start hydrating completed sections before the entire page arrives.

Q6: What is the "shell" in streaming SSR? L6

The shell is everything that renders synchronously without hitting a Suspense boundary — typically the page layout, navigation, footer, and Suspense fallbacks. It's the first HTML chunk sent to the browser and determines the initial visual state. The shell fires the onShellReady callback in renderToPipeableStream. If the shell fails to render (e.g., a component outside Suspense throws), onShellError fires and you should send a fallback error page. Shell design is an architectural decision — a well-designed shell shows meaningful layout immediately while slow content loads progressively.

Q7: How do you handle SSR for authenticated/personalized pages? L6

Personalized pages cannot be CDN-cached (each user sees different content). Strategies: (1) SSR with Cache-Control: private, no-cache — render per-request, no CDN caching. (2) Partial prerendering — static shell from CDN, dynamic personalized sections streamed from server. (3) Edge rendering — render at edge with user cookies, reducing latency. (4) Client-side personalization — serve a generic cached page and personalize via client-side JS after hydration. (5) Segmented caching — cache by user segment (e.g., free vs premium) rather than per-user. The best approach depends on how much of the page is personalized and latency requirements.

React Server Components

Q8: How do React Server Components differ from SSR? L6

SSR and RSC are orthogonal concepts. SSR is about when HTML is generated — on the server vs client. RSC is about where component code executes — server-only vs client. With SSR, all component code ships to the client for hydration. With RSC, server component code never reaches the client — only the rendered output (RSC payload) is sent. RSC reduces bundle size (server-only dependencies aren't bundled), enables direct backend access (database queries in components), and eliminates data fetching waterfalls (server components fetch data directly). You can have RSC without SSR and SSR without RSC; modern frameworks like Next.js combine both.

Q9: Explain the 'use client' directive. What exactly does it do? L6

'use client' marks a module boundary, not a component boundary. It tells the bundler: "this module and everything it imports is client code." The bundler creates a split point — the module is included in the client bundle. On the server, the component is replaced with a client reference ($$typeof: Symbol.for('react.client.reference')) containing the module ID. When the client receives the RSC payload, it resolves these references by loading the corresponding client component from the bundle. Key insight: 'use client' should be pushed as deep as possible in the component tree to minimize the client bundle.

Q10: What is the Flight protocol? L6

Flight is React's custom wire format for transmitting RSC output from server to client. It's a streaming, line-delimited format where each line has an ID, type, and JSON data. Types include: element trees (serialized React elements), client module references (I: lines telling the client which components to load), symbols, errors, and promise references. The format supports streaming — each line can be processed independently as it arrives. Client references are resolved lazily, allowing the shell to render before all client components are loaded. Flight is designed for efficiency — it's more compact than HTML for representing React trees and supports out-of-order streaming.

Q11: What types can be passed as props across the server/client boundary? L6

Serializable types that can cross the boundary: primitives (string, number, boolean, null, undefined, bigint), plain objects, arrays, Map, Set, Date, pre-rendered React elements (JSX from server components), promises (resolved on server), and Server Actions (serialized as references). Non-serializable types that CANNOT cross: functions (except Server Actions), class instances, Symbols (except well-known), closures capturing server state, and circular references. This constraint forces a clean separation — server components own data, client components own interactivity.

Q12: How do you pass server-rendered content as children to a client component? L6

Server components can pass JSX as children to client components. The server-rendered content is serialized as part of the RSC payload and passed as pre-rendered React elements. The client component receives these as children props without needing to know they were server-rendered. This pattern allows client components (e.g., a tab container with state) to wrap server content without pulling it into the client bundle. Example: a server component renders <ClientTabs><ServerContent /></ClientTabs> — the ServerContent is rendered on the server and passed as serialized elements to ClientTabs.

Q13: How do async server components work? L6

Server components can be async functions — they can use await directly in the component body. This enables direct data fetching without useEffect or external state management. When React encounters an async server component, it awaits the promise and renders the resolved output. Combined with Suspense, multiple async components can fetch data in parallel. Client components cannot be async (they would block rendering) — they use Suspense + use() hook instead. Async server components eliminate the need for data fetching libraries in many cases, as the component itself becomes the data fetching boundary.

Streaming Architecture

Q14: How does out-of-order streaming work in React? L6

Suspense boundaries can resolve and stream in any order, regardless of document position. When a boundary resolves, React sends: (1) a hidden <div> containing the resolved content at the end of the document, (2) an inline <script> that calls $RC(boundaryId, contentId) to swap the fallback with the real content. The browser executes these scripts as they arrive, moving content to the correct position. This works because HTML parsing is incremental. A boundary that resolves in 200ms streams before one that takes 500ms, even if the faster one appears later in the document.

Q15: Explain selective hydration and event replay. L6

Selective hydration allows React to hydrate Suspense boundaries independently, prioritizing user interaction. When a user clicks on a not-yet-hydrated boundary, React: (1) captures the event during the capture phase, (2) identifies the target Suspense boundary, (3) synchronously hydrates that boundary (interrupting any in-progress hydration), (4) replays the captured event on the now-hydrated component. The click is never lost — it's queued and replayed. This means the most important parts of the page (what the user is interacting with) hydrate first, regardless of document order.

Q16: What is backpressure in streaming SSR and how is it handled? L6

Backpressure occurs when the server produces HTML faster than the client can consume it (slow network, full TCP buffer). Without handling, the server buffers unbounded data in memory. React's pipe() handles this automatically — when res.write() returns false (buffer full), Fizz pauses rendering. When the 'drain' event fires (client consumed data), rendering resumes. This prevents memory exhaustion under high concurrency with slow clients. Always use pipe() rather than manually writing chunks.

Q17: How should you design Suspense boundaries for streaming? L6

Each Suspense boundary is a streaming unit. Design principles: (1) Align boundaries with independent data dependencies — each boundary should correspond to a distinct data source. (2) 2-4 boundaries per page is typical — too few means the page waits for the slowest source, too many causes excessive skeleton flashing. (3) Place critical content outside Suspense (in the shell) for immediate display. (4) Nest boundaries for progressive disclosure — outer boundary for page layout, inner boundaries for sections. (5) Consider CLS — each boundary swap can cause layout shift, so use fixed-size skeletons.

Next.js Internals

Q18: Explain the four caching layers in Next.js. L6

(1) Request Memoization: Deduplicates identical fetch() calls within a single server render. Automatic for same URL + options. Scope: single request. (2) Data Cache: Persists fetch() responses across requests and deployments. Supports time-based revalidation and tag-based invalidation. Stored in .next/cache. (3) Full Route Cache: Caches rendered RSC payload + HTML for static routes at build time. Invalidated when underlying Data Cache entries are revalidated. (4) Router Cache: Client-side in-memory cache of RSC payloads during navigation. Duration: 30s for dynamic pages, 5min for static. Invalidated by router.refresh() or server-side revalidation.

Q19: How do Server Actions work under the hood? L6

Server Actions are async functions marked with 'use server'. The framework generates an API endpoint for each action. When called from a client component, the framework serializes the arguments, sends a POST request to the generated endpoint, executes the function on the server, and returns the result. For forms, Server Actions work as the form's action attribute — without JavaScript, the form submits as a standard POST; with JavaScript, the submission is intercepted and handled via fetch. Server Actions can call revalidatePath/revalidateTag to invalidate caches and redirect for navigation.

Q20: What is Partial Prerendering (PPR) and how does it work? L6

PPR combines static and dynamic rendering in a single route. At build time, the static shell (everything outside Suspense boundaries wrapping dynamic content) is prerendered and served from the CDN. At request time, dynamic "holes" (Suspense boundaries containing components that use dynamic APIs like cookies(), headers(), or uncached fetch) are streamed from the server. The result: CDN-speed TTFB for the static shell with real-time dynamic content streamed in. PPR requires careful Suspense boundary design — each dynamic section needs its own boundary to create a "hole" in the static shell.

Q21: How does Next.js determine if a route is static or dynamic? L6

Next.js automatically detects dynamic rendering based on API usage: cookies(), headers(), searchParams prop, uncached fetch() (cache: 'no-store'), and unstable_noStore() all trigger dynamic rendering. You can also force it with export const dynamic = 'force-dynamic' or 'force-static'. If none of these dynamic APIs are used, the route is static by default. The detection happens during the build — Next.js renders each route and tracks which APIs were called.

Q22: Explain the difference between layouts and templates in Next.js. L6

Layouts persist across navigations — the same component instance is reused, preserving state, effects, and DOM. Templates re-mount on every navigation — React unmounts and remounts the component, resetting all state. Internally, layouts maintain the same React component identity across navigations, while templates use a different key prop for each route, forcing React to treat them as different components. Use layouts for persistent UI (sidebars, navigation). Use templates for per-page effects (entry animations, page-view tracking, resetting form state).

Meta-Frameworks

Q23: How does Remix's data loading model differ from Next.js? L6

Remix uses colocated loaders — each route defines a loader function that runs on the server before rendering. When navigating to a URL, Remix runs ALL matched route loaders in parallel (parent + child), eliminating waterfalls. Data is accessed via useLoaderData(). Mutations use action functions triggered by form submissions. Remix relies on HTTP caching (Cache-Control headers) rather than framework-level caching. Next.js App Router uses React Server Components for data fetching (async components with direct await), has a 4-layer caching system, and uses Server Actions for mutations. Remix is more explicit (you control caching via HTTP headers); Next.js is more automatic (caching is built-in with opt-out).

Q24: What is the islands architecture and how does Astro implement it? L6

Islands architecture renders pages as static HTML with isolated "islands" of interactivity. Each island is independently hydrated — the rest of the page ships zero JavaScript. Astro implements this with hydration directives: client:load (hydrate on page load), client:idle (hydrate when browser is idle), client:visible (hydrate when visible in viewport), client:media (hydrate when media query matches). Without a directive, components render as static HTML with no JS. Astro is framework-agnostic — islands can be React, Vue, Svelte, or any supported framework, even mixed on the same page.

Q25: When would you choose Remix over Next.js? L6

Choose Remix when: (1) progressive enhancement is critical (forms must work without JS), (2) you prefer explicit HTTP caching over framework magic, (3) the app is form-heavy with complex mutation flows, (4) you want web-standard patterns (Request/Response, FormData), (5) you need fine-grained control over data loading without framework caching layers. Choose Next.js when: (1) you need mixed rendering strategies (SSG + ISR + SSR in one app), (2) RSC and zero-bundle-size components are valuable, (3) you want built-in caching and optimization, (4) you need edge middleware, (5) the Vercel deployment ecosystem is attractive.

Edge Rendering

Q26: What are V8 isolates and how do they differ from containers? L6

V8 isolates are lightweight instances of the V8 JavaScript engine with their own heap and execution context. Multiple isolates share the same OS process, enabling near-instant cold starts (<5ms vs 100ms-5s for containers). Each isolate has ~128KB memory overhead vs 128MB+ for containers. The tradeoff: isolates run in a restricted environment — no filesystem access, no native modules, no long-running processes, limited memory (128MB), and execution time limits (typically 30s). They're ideal for stateless, compute-light operations like SSR, middleware, and API routing. Containers are better for heavy computation, native dependencies, and stateful operations.

Q27: What are the limitations of edge rendering for React? L6

Key limitations: (1) No Node.js APIs (fs, child_process, net), (2) no native modules (sharp, bcrypt, prisma), (3) 128MB memory limit, (4) 30s execution timeout, (5) no persistent connections (database connection pooling), (6) no WebSocket servers, (7) limited npm package compatibility. Solutions: use HTTP-based databases (PlanetScale, Neon), pure JS alternatives (bcryptjs), delegate heavy work to serverless functions, use edge-compatible ORMs (Drizzle with HTTP driver), and embed static data at build time. Edge rendering is best for simple, latency-sensitive rendering — not for complex server-side processing.

Q28: How would you architect a globally distributed React application? L6

Multi-layer architecture: (1) Edge middleware for auth, geo-routing, A/B testing, and feature flags — runs at 200+ PoPs. (2) Edge rendering for latency-sensitive, non-personalized pages — SSR at the nearest PoP. (3) Regional serverless for complex rendering with database access — deploy in 3-5 regions near your data. (4) CDN caching for static and ISR pages — global distribution without compute. (5) Edge databases (Turso, Neon) for read replicas near edge locations. The key is matching each route's requirements to the right compute tier — not everything needs edge rendering, and not everything can run at the edge.

Caching Strategies

Q29: Explain stale-while-revalidate and why it's important for SSR. L6

stale-while-revalidate (SWR) is a Cache-Control directive that allows serving stale cached content while revalidating in the background. When a cached response becomes stale (past max-age), the next request receives the stale response instantly while the cache fetches a fresh copy from the origin. This provides instant responses (no waiting for origin) with eventual freshness. For SSR, SWR is critical because server rendering is expensive (50-200ms). With s-maxage=60, stale-while-revalidate=86400, the CDN serves cached HTML for 1 minute, then serves stale for up to 24 hours while revalidating — users never wait for server rendering.

Q30: What is a cache stampede and how do you prevent it? L6

A cache stampede occurs when a cached entry expires and many concurrent requests simultaneously hit the origin, overwhelming the server. Prevention strategies: (1) Distributed lock: Only one request revalidates; others serve stale. Use Redis SETNX for the lock. (2) Stale-while-revalidate: Serve stale content while one request revalidates in the background. (3) Probabilistic early expiration: Add jitter so entries expire at slightly different times, spreading revalidation. (4) Cache warming: Proactively refresh high-traffic cache entries before they expire. (5) Request coalescing: CDNs like Cloudflare collapse concurrent cache misses into a single origin request.

Q31: How would you design a caching strategy for an e-commerce product page? L6

Multi-layer approach: (1) Static shell via PPR — product description, images, layout cached at CDN (s-maxage=3600). (2) Dynamic pricing — streamed from server, cached in Redis with 60s TTL, tag-based invalidation on price changes. (3) Inventory status — no cache or very short TTL (5s), fetched from edge to reduce latency. (4) Reviews — ISR with 5-minute revalidation, on-demand revalidation when new review is posted. (5) Personalized recommendations — client-side fetch after hydration, cached per user segment in Redis. (6) CDN: Vary on Accept-Language for i18n, no Vary on Cookie (use private for authenticated sections). CMS webhook triggers revalidateTag('product-{id}') on content updates.

Q32: Explain the difference between revalidatePath and revalidateTag in Next.js. L6

revalidatePath(path) invalidates the Full Route Cache for a specific URL path. It re-renders the page on the next request. You can pass 'page' or 'layout' as the second argument to control scope. revalidateTag(tag) invalidates all Data Cache entries tagged with that tag, which in turn invalidates any Full Route Cache entries that depend on those data entries. Tags are more granular and composable — a single tag can invalidate cache entries across multiple pages. Use revalidatePath for page-level invalidation and revalidateTag for data-level invalidation that may affect multiple pages.

Q33: How does React's cache() function work and how does it differ from useMemo? L6

cache() is a React function for request-level memoization of async functions. It deduplicates calls with the same arguments within a single server render — if two server components call getUser('123'), only one fetch happens. It's scoped to the current request and automatically cleared after rendering. useMemo is a hook for component-level memoization within a single render cycle — it caches computed values between re-renders of the same component instance. Key differences: cache() works across components (request-scoped), useMemo works within one component (instance-scoped). cache() is for async server data, useMemo is for synchronous client computations.

Q34: How would you implement cache warming for an SSR application? L6

Cache warming proactively populates caches before users request pages. Implementation: (1) After deployment, run a warming script that fetches high-traffic URLs from the new deployment. (2) Use a priority queue — warm the most-visited pages first (based on analytics data). (3) Limit concurrency to avoid overwhelming the origin (10-20 concurrent requests). (4) For ISR, trigger revalidation of critical pages immediately after deployment. (5) For CDN warming, make requests from multiple geographic locations to populate edge caches. (6) Schedule periodic warming for pages with time-based revalidation to prevent cache misses during traffic spikes. Monitor cache hit rates and adjust warming strategy based on miss patterns.

Advanced Architecture

Q35: How would you migrate a large CSR React app to SSR? L6

Incremental migration strategy: (1) Start with the most SEO-critical routes (landing pages, product pages). (2) Audit for SSR compatibility — identify window/document usage, CSS-in-JS setup, third-party scripts. (3) Add Next.js App Router alongside existing routes using rewrites/redirects. (4) Convert pages one at a time — extract data fetching to server components, mark interactive parts with 'use client'. (5) Handle hydration mismatches — replace Date.now() with server-passed timestamps, guard browser APIs with useEffect. (6) Set up monitoring for hydration errors, TTFB, and FCP. (7) Gradually move more routes to SSR/SSG/ISR based on performance data. Don't try to convert everything at once — incremental migration reduces risk.

Q36: Design the SSR architecture for a news website with 10M daily page views. L6

Architecture: (1) Homepage: ISR with 30s revalidation + CDN SWR. Breaking news triggers on-demand revalidation via CMS webhook. (2) Article pages: SSG at publish time + ISR with 5min revalidation. CDN-cached with s-maxage=300, stale-while-revalidate=86400. (3) Live updates: Client-side WebSocket for real-time content, SSR for initial load. (4) Personalized feed: Edge rendering with user preferences from edge KV store. (5) Infrastructure: Multi-region deployment (3 regions), CDN with 200+ PoPs, Redis cluster for application cache, edge middleware for A/B testing. (6) Cache stampede prevention: Request coalescing at CDN, distributed locks for ISR revalidation. Expected: 99%+ CDN cache hit rate, <50ms TTFB for cached pages, <500ms for cache misses.

Q37: What are the security considerations for SSR? L6

SSR-specific security concerns: (1) Data leakage: Server components can access secrets — ensure they never leak into client props or RSC payload. (2) Server Action validation: Always validate and sanitize Server Action inputs — they're public API endpoints. (3) XSS via dangerouslySetInnerHTML: Server-rendered HTML must be sanitized. (4) Cache poisoning: Ensure Vary headers are correct — a cached personalized page served to another user is a data breach. (5) SSRF: Server components making fetch calls can be exploited for server-side request forgery. (6) Timing attacks: SSR response times can leak information about data existence. (7) Prototype pollution: Deserialized props from RSC payload could be exploited. Mitigations: input validation, output encoding, proper cache headers, SSRF allowlists, and security headers (CSP, HSTS).
