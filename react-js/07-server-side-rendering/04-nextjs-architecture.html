App Router Internals

The Next.js App Router (introduced in v13.4) is built on React Server Components, Suspense, and streaming. It represents a fundamental architectural shift from the Pages Router — every component is a server component by default, layouts persist across navigations, and data fetching is colocated with components.

App Router Architecture: Request → Middleware → Route Matching → Rendering Pipeline │ ┌─────────────────────────┤ │ │ Static Route Dynamic Route │ │ Serve from cache Server Render (Full Route Cache) │ ┌────────┴────────┐ │ │ RSC Render HTML Stream (Flight payload) (renderToPipeableStream) │ │ └────────┬────────┘ │ Client receives RSC payload + HTML

File-System Routing Convention

The App Router uses a file-system convention where folder structure maps to URL segments. Special file names have specific roles in the rendering pipeline.

app/
├── layout.tsx          # Root layout (wraps entire app)
├── page.tsx            # Home page (/)
├── loading.tsx         # Suspense fallback for /
├── error.tsx           # Error boundary for /
├── not-found.tsx       # 404 page
├── global-error.tsx    # Root error boundary
│
├── dashboard/
│   ├── layout.tsx      # Dashboard layout (persists across sub-routes)
│   ├── page.tsx        # /dashboard
│   ├── loading.tsx     # Suspense fallback for /dashboard
│   │
│   ├── analytics/
│   │   └── page.tsx    # /dashboard/analytics
│   │
│   └── settings/
│       ├── page.tsx    # /dashboard/settings
│       └── error.tsx   # Error boundary for /dashboard/settings
│
├── blog/
│   ├── page.tsx        # /blog (list page)
│   └── [slug]/
│       ├── page.tsx    # /blog/:slug (dynamic segment)
│       └── opengraph-image.tsx  # OG image generation
│
├── api/
│   └── webhook/
│       └── route.ts    # API route: POST /api/webhook
│
└── (marketing)/        # Route group (no URL segment)

    ├── about/

Layouts and Templates

Layouts are the key architectural primitive in the App Router. They persist across navigations — their state, effects, and DOM are preserved when navigating between sibling routes.

// app/dashboard/layout.tsx — Server Component
// This layout persists when navigating between
// /dashboard/analytics and /dashboard/settings
export default function DashboardLayout({ children }) {
  return (
    <div className="dashboard">
      <Sidebar />
      <main>{children}</main>
    </div>
  );
}

// app/dashboard/template.tsx — Re-mounts on every navigation
// Use for: entry animations, per-page effects, resetting state
'use client';
export default function DashboardTemplate({ children }) {
  useEffect(() => {
    // Fires on EVERY navigation (unlike layout)
    trackPageView();
  });

  return <motion.div initial={{ opacity: 0 }} animate={{ opacity: 1 }}>
    {children}
  </motion.div>;
}
Layout vs Template Internals

Layouts use React's component identity to persist — the same layout component instance is reused across navigations, so useState and useRef values survive. Templates use a different key prop for each navigation, forcing React to unmount and remount the component tree.

Server Actions

Server Actions are async functions that execute on the server, callable from both server and client components. They replace traditional API routes for mutations and form handling.

// actions/posts.ts
'use server';

import { db } from '@/lib/db';
import { revalidatePath } from 'next/cache';
import { redirect } from 'next/navigation';
import { z } from 'zod';

const CreatePostSchema = z.object({
  title: z.string().min(1).max(200),
  content: z.string().min(1),
});

export async function createPost(formData: FormData) {
  // Validate input
  const parsed = CreatePostSchema.safeParse({
    title: formData.get('title'),
    content: formData.get('content'),
  });

  if (!parsed.success) {
    return { error: parsed.error.flatten().fieldErrors };
  }

  // Mutate data
  const post = await db.posts.create({
    data: parsed.data,
  });

  // Revalidate cache
  revalidatePath('/blog');

  // Redirect
  redirect(`/blog/${post.slug}`);
}
// Using Server Action in a form (progressive enhancement)
import { createPost } from '@/actions/posts';

export default function NewPostPage() {
  return (
    <form action={createPost}>
      <input name="title" required />
      <textarea name="content" required />
      <SubmitButton />
    </form>
  );
}

// Client component for pending state
'use client';
import { useFormStatus } from 'react-dom';

function SubmitButton() {
  const { pending } = useFormStatus();
  return (
    <button type="submit" disabled={pending}>
      {pending ? 'Creating...' : 'Create Post'}
    </button>
  );
}
Architect Insight

Server Actions work without JavaScript — the form submits as a standard POST request, and the server handles the redirect. When JavaScript is available, the form submission is intercepted, the action is called via fetch, and the UI updates without a full page reload. This progressive enhancement is a key architectural advantage over traditional SPA mutation patterns.

Route Handlers

// app/api/posts/route.ts
import { NextRequest, NextResponse } from 'next/server';

// GET /api/posts
export async function GET(request: NextRequest) {
  const searchParams = request.nextUrl.searchParams;
  const page = parseInt(searchParams.get('page') ?? '1');

  const posts = await db.posts.findMany({
    skip: (page - 1) * 20,
    take: 20,
  });

  return NextResponse.json(posts);
}

// POST /api/posts
export async function POST(request: NextRequest) {
  const body = await request.json();
  const post = await db.posts.create({ data: body });
  return NextResponse.json(post, { status: 201 });
}

// Dynamic route: app/api/posts/[id]/route.ts
export async function GET(
  request: NextRequest,
  { params }: { params: { id: string } }
) {
  const post = await db.posts.findUnique({
    where: { id: params.id }
  });

  if (!post) {
    return NextResponse.json(
      { error: 'Not found' },
      { status: 404 }
    );
  }

  return NextResponse.json(post);
}

Middleware

Next.js middleware runs before every request, at the edge. It can rewrite, redirect, modify headers, or return early responses. Middleware runs in the Edge Runtime (V8 isolates, no Node.js APIs).

// middleware.ts (root of project)
import { NextResponse } from 'next/server';
import type { NextRequest } from 'next/server';

export function middleware(request: NextRequest) {
  // Authentication check
  const token = request.cookies.get('session')?.value;

  if (request.nextUrl.pathname.startsWith('/dashboard')) {
    if (!token) {
      return NextResponse.redirect(
        new URL('/login', request.url)
      );
    }
  }

  // Geolocation-based routing
  const country = request.geo?.country ?? 'US';
  if (request.nextUrl.pathname === '/') {
    return NextResponse.rewrite(
      new URL(`/${country.toLowerCase()}`, request.url)
    );
  }

  // Add custom headers
  const response = NextResponse.next();
  response.headers.set('x-request-id', crypto.randomUUID());
  return response;
}

export const config = {
  matcher: ['/((?!api|_next/static|_next/image|favicon.ico).*)'],
};

The Four Caching Layers

Next.js has four distinct caching layers, each operating at a different level. Understanding all four is critical for debugging caching issues and designing performant applications.

Next.js Caching Layers: Layer 1: Request Memoization (per-request, server only) ├─ Deduplicates identical fetch() calls within a single render ├─ Scope: Single server request └─ Automatic for fetch() with same URL + options Layer 2: Data Cache (persistent, server only) ├─ Caches fetch() responses across requests ├─ Scope: Across requests and deployments ├─ Revalidation: time-based or on-demand └─ Stored in .next/cache or external store Layer 3: Full Route Cache (persistent, server only) ├─ Caches rendered RSC payload + HTML at build time ├─ Scope: Static routes only ├─ Invalidated when Data Cache is revalidated └─ Stored in .next/cache Layer 4: Router Cache (ephemeral, client only) ├─ Caches RSC payload in browser memory ├─ Scope: User session (client-side navigation) ├─ Duration: 30s (dynamic) / 5min (static) └─ Invalidated by router.refresh() or revalidation 
// Layer 1: Request Memoization
// These two calls are deduplicated within the same request
async function ProductPage({ params }) {
  const product = await fetch(`/api/products/${params.id}`);
  // ...
}
async function ProductReviews({ productId }) {
  // Same URL = same fetch, deduplicated automatically
  const product = await fetch(`/api/products/${productId}`);
  // ...
}

// Layer 2: Data Cache — controlling cache behavior
// Cached indefinitely (default)
fetch('https://api.example.com/data');

// Revalidate every 60 seconds
fetch('https://api.example.com/data', {
  next: { revalidate: 60 }
});

// No caching
fetch('https://api.example.com/data', {
  cache: 'no-store'
});

// Tag-based revalidation
fetch('https://api.example.com/posts', {
  next: { tags: ['posts'] }
});
// Later: revalidateTag('posts')
Common Caching Pitfall

The Router Cache (Layer 4) caches RSC payloads on the client for 30 seconds (dynamic pages) or 5 minutes (static pages). This means client-side navigations may show stale data even after server-side revalidation. Use router.refresh() to force a fresh fetch, or understand that users may see slightly stale data during client-side navigation.

Partial Prerendering

Partial Prerendering (PPR) is an experimental Next.js feature that combines static and dynamic rendering in a single route. The static shell is served from the CDN, and dynamic holes are streamed in.

Partial Prerendering: Build Time: ┌──────────────────────────────────────┐ │ Static Shell (prerendered) │ │ ├─ Header │ │ ├─ Navigation │ │ ├─ [Suspense hole: user greeting] │ ← dynamic │ ├─ Product description (static) │ │ ├─ [Suspense hole: price] │ ← dynamic │ ├─ [Suspense hole: inventory] │ ← dynamic │ └─ Footer │ └──────────────────────────────────────┘ Request Time: 1. CDN serves static shell instantly (0ms TTFB) 2. Server streams dynamic holes: ├─ User greeting (personalized) ├─ Price (real-time) └─ Inventory (real-time) 3. Client replaces Suspense fallbacks with streamed content 
// Partial Prerendering example
import { Suspense } from 'react';
import { unstable_noStore } from 'next/cache';

// Static: prerendered at build time
export default function ProductPage({ params }) {
  return (
    <div>
      <Header />

      <Suspense fallback={<GreetingSkeleton />}>
        <UserGreeting />  {/* Dynamic: personalized */}
      </Suspense>

      <ProductDescription slug={params.slug} />  {/* Static */}

      <Suspense fallback={<PriceSkeleton />}>
        <DynamicPrice productId={params.id} />  {/* Dynamic */}
      </Suspense>

      <Footer />
    </div>
  );
}

// This component opts into dynamic rendering
async function DynamicPrice({ productId }) {
  unstable_noStore(); // Marks as dynamic
  const price = await fetchCurrentPrice(productId);
  return <span>${price}</span>;
}
Architect Insight

PPR represents the convergence of SSG and SSR — you get CDN-speed TTFB for the static shell with real-time dynamic content streamed in. This eliminates the binary choice between static and dynamic rendering. At L6, understand that PPR requires careful Suspense boundary design: each dynamic section needs its own Suspense boundary to create a "hole" in the static shell.
