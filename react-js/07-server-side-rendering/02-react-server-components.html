RSC Mental Model

React Server Components (RSC) introduce a fundamental split in the React component model: some components execute only on the server, some only on the client, and the framework manages the boundary between them. This is not SSR — SSR renders client components on the server to produce HTML. RSC components never ship to the client bundle.

Traditional SSR: ┌─────────────────────────────────────────────┐ │ Server: Render ALL components → HTML string │ │ Client: Hydrate ALL components → Interactive│ │ │ │ Bundle: Contains ALL component code │ └─────────────────────────────────────────────┘ React Server Components: ┌─────────────────────────────────────────────┐ │ Server: Render Server Components → RSC │ │ payload (Flight format) │ │ │ │ Client: Render Client Components only │ │ Server Components = serialized │ │ output (no code shipped) │ │ │ │ Bundle: Contains ONLY Client component code │ └─────────────────────────────────────────────┘
Architect Insight

RSC is not a replacement for SSR — it's orthogonal. SSR is about when HTML is generated (server vs client). RSC is about where component code executes (server-only vs client). You can have RSC without SSR (client-side RSC rendering) and SSR without RSC (traditional SSR). Modern frameworks like Next.js App Router combine both.

Why RSC Exists

RSC solves three problems that SSR alone cannot:

  1. Bundle size — Server components and their dependencies (markdown parsers, syntax highlighters, date libraries) are never sent to the client
  2. Data fetching waterfalls — Server components can fetch data directly without client-server round trips or waterfall chains
  3. Direct backend access — Server components can query databases, read files, and call internal APIs without exposing endpoints

Server vs Client Component Boundary

The boundary between server and client components is the most important architectural concept in RSC. Understanding how data and rendering flow across this boundary is essential for Staff-level work.

Default: Server Components

In RSC-enabled frameworks (Next.js App Router), all components are server components by default. They execute on the server, have access to server resources, and their code is never included in the client bundle.

// app/page.tsx — Server Component (default)
// This code NEVER reaches the client
import { db } from '@/lib/database';
import { formatMarkdown } from '@/lib/markdown'; // 50KB library

export default async function BlogPost({ params }) {
  // Direct database access — no API layer needed
  const post = await db.posts.findUnique({
    where: { slug: params.slug }
  });

  // Heavy library used only on server
  const htmlContent = formatMarkdown(post.content);

  return (
    <article>
      <h1>{post.title}</h1>
      <div dangerouslySetInnerHTML={{ __html: htmlContent }} />
      <LikeButton postId={post.id} initialCount={post.likes} />
    </article>
  );
}

Client Components: 'use client'

Components that need interactivity (event handlers, state, effects, browser APIs) must be marked with the 'use client' directive. This creates a boundary — the component and everything it imports becomes part of the client bundle.

// components/LikeButton.tsx — Client Component
'use client';

import { useState, useTransition } from 'react';
import { likePost } from '@/actions/posts';

export function LikeButton({ postId, initialCount }) {
  const [count, setCount] = useState(initialCount);
  const [isPending, startTransition] = useTransition();

  function handleLike() {
    startTransition(async () => {
      const newCount = await likePost(postId);
      setCount(newCount);
    });
  }

  return (
    <button onClick={handleLike} disabled={isPending}>
      ♥ {count} {isPending ? '...' : ''}
    </button>
  );
}
Critical: The Boundary Is a Module Boundary

'use client' marks a module boundary, not a component boundary. When a server component imports a client component, that import creates the serialization boundary. Everything the client component imports (its entire dependency subtree) becomes client code. This is why you should push 'use client' as deep as possible in your component tree.

Component Composition Across the Boundary

Component Tree with RSC Boundary: ServerLayout (Server) │ ┌────┴────────────────┐ │ │ ServerSidebar MainContent (Server) (Server) │ │ ┌────┴─────┐ │ │ │ NavLinks Article InteractivePanel (Server) (Server) (Client) ← 'use client' │ ┌────┴────┐ │ │ TabBar Comments (Client) (Client) ↑ inherited from parent

A key pattern: server components can pass server-rendered content as children to client components. This allows client components to wrap server content without pulling it into the client bundle.

// Server Component — passes server content as children
export default function Page() {
  return (
    <ClientTabs>
      {/* These are server components passed as children */}
      <ExpensiveServerContent />
      <AnotherServerComponent />
    </ClientTabs>
  );
}

// Client Component — receives pre-rendered children
'use client';
export function ClientTabs({ children }) {
  const [activeTab, setActiveTab] = useState(0);
  return (
    <div>
      <div className="tabs">
        <button onClick={() => setActiveTab(0)}>Tab 1</button>
        <button onClick={() => setActiveTab(1)}>Tab 2</button>
      </div>
      {/* children are already rendered on the server */}
      {React.Children.toArray(children)[activeTab]}
    </div>
  );
}

Directives: 'use client' and 'use server'

'use client' Directive

Marks a module as a client entry point. The bundler creates a split point here — the module and its imports are included in the client bundle. The server sees a reference (placeholder) instead of the actual component code.

// How the bundler processes 'use client':

// Original: components/Counter.tsx
'use client';
export function Counter() { /* ... */ }

// Server sees this as a reference:
{
  $$typeof: Symbol.for('react.client.reference'),
  $$id: 'components/Counter.tsx#Counter',
  $$async: false
}

// Client bundle includes the actual component code

'use server' Directive

Marks a function as a Server Action — a function that executes on the server but can be called from client components. The framework generates an API endpoint automatically.

// actions/posts.ts
'use server';

import { db } from '@/lib/database';
import { revalidatePath } from 'next/cache';

export async function likePost(postId: string) {
  // This runs on the server — direct DB access
  const post = await db.posts.update({
    where: { id: postId },
    data: { likes: { increment: 1 } }
  });

  revalidatePath(`/posts/${postId}`);
  return post.likes;
}

// Under the hood, the framework generates:
// POST /actions/likePost
// Body: [postId]
// The client component calls this via fetch automatically

Zero-Bundle-Size Components

Server components achieve zero bundle size because their code is never sent to the client. Only their rendered output (as RSC payload) reaches the browser.

Bundle Impact Example

A blog page using marked (35KB), highlight.js (180KB), and sanitize-html (45KB) for markdown rendering. As a client component, this adds ~260KB to the bundle. As a server component, these libraries are never sent to the client — the rendered HTML is sent instead. The client bundle for this page could be near zero if the only interactive element is a small like button.

RSC Wire Format (Flight Protocol)

The Flight protocol is React's custom wire format for transmitting server component output to the client. It's a streaming, line-delimited format that encodes the React element tree, client component references, and serialized data.

// Example Flight payload (simplified)
// Each line is a chunk, format: ID:TYPE:DATA

0:["$","div",null,{"children":[
  ["$","h1",null,{"children":"Hello World"}],
  ["$","$L1",null,{"postId":"123","initialCount":42}]
]}]

1:I["components/LikeButton.tsx",["LikeButton"],""]

// Decoded:
// 0: Root element — a div with h1 and a client reference ($L1)
// 1: Client reference — tells the client to load LikeButton
//    from the client bundle
// $L1 = lazy client reference to chunk 1

Flight Payload Structure

Flight Protocol Wire Format: ┌──────────────────────────────────────────────────┐ │ Line Format: ID : TYPE : JSON_DATA │ │ │ │ Types: │ │ (none) — React element tree (JSON) │ │ I — Client module reference (import) │ │ S — Symbol (React.Suspense, Fragment) │ │ E — Error │ │ T — Text chunk │ │ P — Promise/async boundary │ │ │ │ Special tokens in element data: │ │ $ — React element │ │ $L{id} — Lazy client reference │ │ $F{id} — Server reference (Server Action) │ │ @{id} — Promise reference │ └──────────────────────────────────────────────────┘

The Flight format is designed for streaming — each line can be processed independently as it arrives. Client component references are resolved lazily, allowing the shell to render before all client components are loaded.

Async Server Components

Server components can be async functions — a capability that client components don't have. This enables direct data fetching without useEffect or external state management.

// Async Server Component — direct data fetching
async function ProductPage({ params }) {
  // These run in parallel on the server
  const [product, reviews, recommendations] = await Promise.all([
    fetchProduct(params.id),
    fetchReviews(params.id),
    fetchRecommendations(params.id),
  ]);

  return (
    <div>
      <ProductHeader product={product} />
      <Suspense fallback={<ReviewsSkeleton />}>
        <ReviewsList reviews={reviews} />
      </Suspense>
      <Suspense fallback={<RecommendationsSkeleton />}>
        <RecommendationsGrid items={recommendations} />
      </Suspense>
    </div>
  );
}

// Nested async components enable waterfall-free data fetching
async function ReviewsList({ reviews }) {
  const userProfiles = await fetchUserProfiles(
    reviews.map(r => r.userId)
  );

  return (
    <ul>
      {reviews.map(review => (
        <ReviewItem
          key={review.id}
          review={review}
          user={userProfiles[review.userId]}
        />
      ))}
    </ul>
  );
}

Serialization Constraints

Props passed from server components to client components must be serializable — they cross a network boundary. Understanding what can and cannot cross this boundary is critical for RSC architecture.

Serializable Types (Can Cross Boundary)

// These types can be passed as props to client components:
const validProps = {
  // Primitives
  string: "hello",
  number: 42,
  boolean: true,
  null: null,
  undefined: undefined,
  bigint: 42n,

  // Collections
  array: [1, 2, 3],
  object: { key: "value" },
  map: new Map([["key", "value"]]),
  set: new Set([1, 2, 3]),

  // Special React types
  element: <ServerComponent />,  // Pre-rendered RSC output
  promise: fetchData(),           // Resolved on server, serialized

  // Dates
  date: new Date(),

  // Server Actions (serialized as references)
  action: serverActionFunction,
};

Non-Serializable Types (Cannot Cross Boundary)

// These CANNOT be passed as props to client components:

// Functions (except Server Actions)
<ClientComponent onClick={() => {}} />  // ERROR

// Class instances
<ClientComponent db={new Database()} />  // ERROR

// Symbols (except well-known)
<ClientComponent id={Symbol('unique')} />  // ERROR

// Closures capturing server state
const serverSecret = process.env.SECRET;
<ClientComponent
  handler={() => serverSecret}  // ERROR: closure over server state
/>

// Circular references
const obj = {};
obj.self = obj;
<ClientComponent data={obj} />  // ERROR
Common Mistake: Event Handlers

You cannot pass event handler functions from server to client components. Instead, define event handlers inside client components, or use Server Actions for mutations. This is the most common RSC boundary error developers encounter.

// WRONG: Passing function from server to client
async function ServerPage() {
  function handleClick() { console.log('clicked'); }
  return <ClientButton onClick={handleClick} />; // ERROR
}

// RIGHT: Define handler in client component
// ClientButton.tsx
'use client';
export function ClientButton() {
  function handleClick() { console.log('clicked'); }
  return <button onClick={handleClick}>Click</button>;
}

// RIGHT: Use Server Action for mutations
import { submitForm } from '@/actions/form';
async function ServerPage() {
  return <ClientForm action={submitForm} />; // OK — Server Action
}
Architect Insight

The RSC serialization boundary forces a clean separation between data and behavior. Server components own data fetching and transformation; client components own interactivity. This constraint, while initially frustrating, leads to better architecture at scale — it prevents the common anti-pattern of mixing data fetching logic with UI interaction logic in the same component.
