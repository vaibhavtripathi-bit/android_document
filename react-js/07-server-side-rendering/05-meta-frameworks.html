Remix Architecture

Remix is built on web standards — it embraces the platform rather than abstracting it away. Its architecture centers on nested routes with colocated data loading (loaders) and mutations (actions), with progressive enhancement as a core principle.

Remix Request Lifecycle: Browser Request │ ▼ ┌─────────────────┐ │ Route Matching │ Match URL to nested route tree └────────┬────────┘ │ ▼ ┌─────────────────┐ │ Run Loaders │ Parallel data fetching for ALL │ (parallel) │ matched routes simultaneously └────────┬────────┘ │ ▼ ┌─────────────────┐ │ Render Routes │ Nested route components with │ (nested) │ loader data available via useLoaderData() └────────┬────────┘ │ ▼ ┌─────────────────┐ │ Stream HTML │ Progressive rendering with │ + Hydrate │ deferred data streaming └─────────────────┘

Nested Routes and Loaders

// Remix route structure — file-based routing
// app/routes/
// ├── _index.tsx           → /
// ├── dashboard.tsx        → /dashboard (layout)
// ├── dashboard._index.tsx → /dashboard (index)
// ├── dashboard.analytics.tsx → /dashboard/analytics
// └── dashboard.settings.tsx  → /dashboard/settings

// dashboard.tsx — Layout route with loader
import type { LoaderFunctionArgs } from '@remix-run/node';
import { json } from '@remix-run/node';
import { Outlet, useLoaderData } from '@remix-run/react';

export async function loader({ request }: LoaderFunctionArgs) {
  const user = await requireAuth(request);
  const notifications = await getNotifications(user.id);
  return json({ user, notifications });
}

export default function DashboardLayout() {
  const { user, notifications } = useLoaderData<typeof loader>();
  return (
    <div className="dashboard">
      <Sidebar user={user} notifications={notifications} />
      <main>
        <Outlet />  {/* Child route renders here */}
      </main>
    </div>
  );
}

// dashboard.analytics.tsx — Child route
export async function loader({ request }: LoaderFunctionArgs) {
  const user = await requireAuth(request);
  const analytics = await getAnalytics(user.id);
  return json({ analytics });
}

export default function Analytics() {
  const { analytics } = useLoaderData<typeof loader>();
  return <AnalyticsChart data={analytics} />;
}
Parallel Loaders

When navigating to /dashboard/analytics, Remix runs both the dashboard layout loader and the analytics page loader in parallel. This eliminates the waterfall problem where child components wait for parent data before fetching their own. Next.js achieves similar parallelism through React Server Components and parallel data fetching.

Actions and Mutations

// Remix actions — form-based mutations
import { ActionFunctionArgs, redirect } from '@remix-run/node';
import { Form, useActionData, useNavigation } from '@remix-run/react';

export async function action({ request }: ActionFunctionArgs) {
  const formData = await request.formData();
  const title = formData.get('title');
  const content = formData.get('content');

  const errors: Record<string, string> = {};
  if (!title) errors.title = 'Title is required';
  if (!content) errors.content = 'Content is required';

  if (Object.keys(errors).length) {
    return json({ errors }, { status: 400 });
  }

  const post = await createPost({ title, content });
  return redirect(`/posts/${post.slug}`);
}

export default function NewPost() {
  const actionData = useActionData<typeof action>();
  const navigation = useNavigation();
  const isSubmitting = navigation.state === 'submitting';

  return (
    <Form method="post">
      <input name="title" />
      {actionData?.errors?.title && (
        <span className="error">{actionData.errors.title}</span>
      )}
      <textarea name="content" />
      <button type="submit" disabled={isSubmitting}>
        {isSubmitting ? 'Creating...' : 'Create Post'}
      </button>
    </Form>
  );
}

Progressive Enhancement

Remix forms work without JavaScript — they submit as standard HTML forms. When JavaScript loads, Remix intercepts the submission and uses fetch, providing optimistic UI, pending states, and no full-page reload. This is progressive enhancement by default.

// This form works with AND without JavaScript
<Form method="post" action="/api/subscribe">
  <input type="email" name="email" required />
  <button type="submit">Subscribe</button>
</Form>

// Without JS: Standard form POST, server redirect
// With JS: fetch POST, client-side UI update, no reload

// Deferred data — stream slow data
import { defer } from '@remix-run/node';
import { Await, useLoaderData } from '@remix-run/react';

export async function loader() {
  const criticalData = await getCriticalData(); // Fast
  const slowData = getSlowData(); // Returns promise (not awaited)

  return defer({
    critical: criticalData,
    slow: slowData, // Streamed to client when ready
  });
}

export default function Page() {
  const { critical, slow } = useLoaderData<typeof loader>();
  return (
    <div>
      <CriticalSection data={critical} />
      <Suspense fallback={<Skeleton />}>
        <Await resolve={slow}>
          {(data) => <SlowSection data={data} />}
        </Await>
      </Suspense>
    </div>
  );
}

Gatsby Architecture

Gatsby is a static site generator with a GraphQL data layer. It pulls data from multiple sources at build time, transforms it through a plugin system, and generates static HTML pages.

Gatsby Build Pipeline: Data Sources GraphQL Layer Static Output ┌──────────┐ ┌──────────────┐ ┌──────────┐ │ CMS │──┐ │ │ │ HTML │ │ Markdown │──┤ Source │ Unified │ Build │ CSS │ │ APIs │──┼─ Plugins ─▶ GraphQL ├───────▶│ JS │ │ Database │──┤ │ Data Layer │ │ Images │ │ Files │──┘ │ │ │ JSON │ └──────────┘ └──────────────┘ └──────────┘ │ gatsby-node.js createPages() createResolvers() 
// gatsby-node.js — programmatic page creation
exports.createPages = async ({ graphql, actions }) => {
  const { createPage } = actions;

  const result = await graphql(`
    query {
      allMarkdownRemark {
        nodes {
          frontmatter { slug }
        }
      }
    }
  `);

  result.data.allMarkdownRemark.nodes.forEach(node => {
    createPage({
      path: `/blog/${node.frontmatter.slug}`,
      component: path.resolve('./src/templates/blog-post.tsx'),
      context: { slug: node.frontmatter.slug },
    });
  });
};

// src/templates/blog-post.tsx — page template
import { graphql, PageProps } from 'gatsby';

export const query = graphql`
  query BlogPost($slug: String!) {
    markdownRemark(frontmatter: { slug: { eq: $slug } }) {
      html
      frontmatter { title, date, author }
    }
  }
`;

export default function BlogPost({ data }: PageProps<Queries.BlogPostQuery>) {
  const post = data.markdownRemark;
  return (
    <article>
      <h1>{post.frontmatter.title}</h1>
      <div dangerouslySetInnerHTML={{ __html: post.html }} />
    </article>
  );
}

Astro Islands Architecture

Astro takes a radically different approach: pages are static by default with zero JavaScript, and interactive components ("islands") are hydrated independently. This is the islands architecture — each interactive component is an isolated island of interactivity in a sea of static HTML.

Astro Islands Architecture: ┌──────────────────────────────────────────────────┐ │ Static HTML Page (zero JS by default) │ │ │ │ ┌──────────┐ Static ┌──────────────────┐ │ │ │ Header │ content │ Article text │ │ │ │ (static) │ │ (static HTML) │ │ │ └──────────┘ └──────────────────┘ │ │ │ │ ┌─────────────────┐ ┌──────────────────┐ │ │ │ React Counter │ │ Vue Comments │ │ │ │ (island) │ │ (island) │ │ │ │ client:visible │ │ client:idle │ │ │ │ ← hydrates │ │ ← hydrates │ │ │ │ when visible │ │ when idle │ │ │ └─────────────────┘ └──────────────────┘ │ │ │ │ ┌──────────┐ │ │ │ Footer │ Static content │ │ │ (static) │ │ │ └──────────┘ │ └──────────────────────────────────────────────────┘ 
---
// src/pages/blog/[slug].astro — Astro page component
import Layout from '../../layouts/Layout.astro';
import ReactCounter from '../../components/Counter.tsx';
import VueComments from '../../components/Comments.vue';

const { slug } = Astro.params;
const post = await getPost(slug);
---

<Layout title={post.title}>
  <article>
    <h1>{post.title}</h1>
    <div set:html={post.html} />

    <!-- React island: hydrates when visible in viewport -->
    <ReactCounter client:visible initialCount={0} />

    <!-- Vue island: hydrates when browser is idle -->
    <VueComments client:idle postId={post.id} />

    <!-- Svelte island: hydrates on page load -->
    <SvelteNewsletter client:load />

    <!-- No directive = static HTML, zero JS -->
    <StaticFooter />
  </article>
</Layout>

When to Choose Which Framework

Architect Decision Matrix

Next.js: Full-stack React apps with mixed rendering needs. Best when you need SSR + SSG + ISR in one app, or when using React Server Components. Strongest ecosystem and deployment options (Vercel, self-hosted).

Remix: Apps that prioritize web standards and progressive enhancement. Best for form-heavy apps, apps that must work without JS, and teams that prefer explicit data loading over magic caching.

Gatsby: Content-heavy sites with complex data sourcing. Best when pulling from multiple CMS/API sources at build time. Less relevant since Next.js added SSG and ISR.

Astro: Content sites where minimal JS is critical. Best for blogs, docs, marketing sites. The islands architecture means you can use React for interactive parts while shipping zero JS for static content.

Micro-Frontend Integration

Meta-frameworks can serve as the shell for micro-frontend architectures, where different teams own different parts of the page.

// Module Federation with Next.js
// next.config.js
const { NextFederationPlugin } = require('@module-federation/nextjs-mf');

module.exports = {
  webpack(config) {
    config.plugins.push(
      new NextFederationPlugin({
        name: 'shell',
        remotes: {
          checkout: 'checkout@https://checkout.example.com/remoteEntry.js',
          catalog: 'catalog@https://catalog.example.com/remoteEntry.js',
        },
        shared: {
          react: { singleton: true, requiredVersion: '^18.0.0' },
          'react-dom': { singleton: true, requiredVersion: '^18.0.0' },
        },
      })
    );
    return config;
  },
};

// Using remote components
const RemoteCheckout = dynamic(
  () => import('checkout/CheckoutFlow'),
  { ssr: false, loading: () => <CheckoutSkeleton /> }
);
Micro-Frontend Tradeoffs

Micro-frontends add significant complexity: shared dependency management, cross-team CSS conflicts, runtime performance overhead, and complex deployment coordination. At L6, you should advocate for micro-frontends only when organizational scaling demands it (3+ teams working on the same app), not as a default architecture.
