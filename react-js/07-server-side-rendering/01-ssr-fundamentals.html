Why SSR Matters

Client-side rendering (CSR) ships an empty HTML shell and relies on JavaScript to render content. This creates three fundamental problems at scale: search engines may not index JavaScript-rendered content reliably, users see a blank screen until the JS bundle downloads/parses/executes, and perceived performance suffers on slower devices and networks.

Server-side rendering addresses all three by generating HTML on the server for each request. The browser receives fully-formed HTML, can display content immediately, and search engine crawlers see complete content without executing JavaScript.

CSR Timeline: ┌──────────────────────────────────────────────────────────────────┐ │ Request → Empty HTML → Download JS → Parse JS → Execute → Render│ │ ├─── TTFB ───┤ ├──── Blank Screen ────┤ ├── FCP ──┤ │ └──────────────────────────────────────────────────────────────────┘ SSR Timeline: ┌──────────────────────────────────────────────────────────────────┐ │ Request → Server Render → Full HTML → FCP → Download JS → TTI │ │ ├─── TTFB (longer) ───┤ ├─ FCP ─┤ ├── Hydration ──┤ │ └──────────────────────────────────────────────────────────────────┘
Architect Insight

SSR trades server compute cost for improved user experience. At L6, you must reason about this tradeoff quantitatively: SSR adds ~50-200ms of server rendering time to TTFB, but can reduce FCP by 1-3 seconds on typical connections. The ROI depends on your user demographics, caching strategy, and infrastructure costs.

React Server Rendering APIs

React provides three server rendering APIs, each designed for different runtime environments and streaming capabilities. Understanding when to use each is a core Staff-level competency.

renderToString (Legacy)

The original SSR API. Synchronously renders the entire React tree to an HTML string. Simple but fundamentally limited — it blocks the Node.js event loop during rendering and cannot stream partial content.

import { renderToString } from 'react-dom/server';
import App from './App';

function handleRequest(req, res) {
  const html = renderToString(<App url={req.url} />);

  res.send(`
    <!DOCTYPE html>
    <html>
      <head><title>My App</title></head>
      <body>
        <div id="root">${html}</div>
        <script src="/bundle.js"></script>
      </body>
    </html>
  `);
}
Critical Limitation

renderToString does not support Suspense boundaries on the server. Any <Suspense> fallback is rendered synchronously, and the component tree must be fully resolved before any HTML is sent. This makes it incompatible with streaming and modern React patterns. It is considered legacy — use renderToPipeableStream for new projects.

renderToPipeableStream (Node.js)

The modern SSR API for Node.js environments. Returns a pipeable Node.js stream that supports Suspense boundaries, streaming HTML, and out-of-order completion. This is the recommended API for production SSR.

import { renderToPipeableStream } from 'react-dom/server';
import App from './App';

function handleRequest(req, res) {
  let didError = false;

  const { pipe, abort } = renderToPipeableStream(
    <App url={req.url} />,
    {
      bootstrapScripts: ['/bundle.js'],

      onShellReady() {
        // The shell (content outside Suspense boundaries) is ready.
        // Start streaming to the client.
        res.statusCode = didError ? 500 : 200;
        res.setHeader('Content-Type', 'text/html');
        pipe(res);
      },

      onShellError(error) {
        // Shell failed to render — send fallback HTML
        res.statusCode = 500;
        res.send('<h1>Server Error</h1>');
      },

      onAllReady() {
        // All Suspense boundaries resolved.
        // Use this instead of onShellReady for crawlers/bots.
      },

      onError(error) {
        didError = true;
        console.error(error);
      }
    }
  );

  // Abort after timeout to prevent hanging requests
  setTimeout(() => abort(), 10000);
}

renderToReadableStream (Web Streams)

The Web Streams API equivalent, designed for edge runtimes (Cloudflare Workers, Deno Deploy, Vercel Edge) that implement the WHATWG Streams standard rather than Node.js streams.

import { renderToReadableStream } from 'react-dom/server';
import App from './App';

async function handleRequest(req) {
  const stream = await renderToReadableStream(
    <App url={req.url} />,
    {
      bootstrapScripts: ['/bundle.js'],
      signal: AbortSignal.timeout(10000),
    }
  );

  // For bots: wait for all content before responding
  const isBot = /bot|crawler|spider/i.test(
    req.headers.get('user-agent')
  );
  if (isBot) {
    await stream.allReady;
  }

  return new Response(stream, {
    headers: { 'Content-Type': 'text/html' },
  });
}

SSR vs SSG vs ISR

Modern React applications rarely use a single rendering strategy. Staff engineers must choose the right strategy per route based on data freshness requirements, traffic patterns, and infrastructure constraints.

Rendering Strategy Spectrum: Static ◄──────────────────────────────────────────────► Dynamic SSG ISR SSR CSR │ │ │ │ Build-time Build + Revalidate Per-request Client-only No server Periodic server Always server No server Stale OK Bounded staleness Always fresh Always fresh Fastest TTFB Fast TTFB Slower TTFB Slowest FCP CDN-cacheable CDN-cacheable Edge-cacheable CDN for assets

Static Site Generation (SSG)

Pages are rendered at build time and served as static files. Zero server compute at request time. Ideal for content that changes infrequently — marketing pages, documentation, blog posts.

// Next.js App Router — static by default
// This page is rendered at build time
export default async function BlogPost({ params }) {
  const post = await getPost(params.slug);
  return <Article content={post} />;
}

// Generate all possible paths at build time
export async function generateStaticParams() {
  const posts = await getAllPosts();
  return posts.map(post => ({ slug: post.slug }));
}

Incremental Static Regeneration (ISR)

A hybrid approach: pages are statically generated but revalidated in the background after a configurable time interval. Combines the performance of SSG with bounded data staleness.

// Next.js — ISR with time-based revalidation
export const revalidate = 60; // Revalidate every 60 seconds

export default async function ProductPage({ params }) {
  const product = await getProduct(params.id);
  return <ProductDetail product={product} />;
}

// On-demand revalidation via API route
// POST /api/revalidate?tag=products
import { revalidateTag } from 'next/cache';

export async function POST(request) {
  const tag = request.nextUrl.searchParams.get('tag');
  revalidateTag(tag);
  return Response.json({ revalidated: true });
}

Server-Side Rendering (SSR)

Pages are rendered on every request. Guarantees fresh data but requires server compute for each page view. Necessary for personalized content, real-time data, or authenticated pages.

// Next.js — force dynamic rendering
export const dynamic = 'force-dynamic';

export default async function DashboardPage() {
  const user = await getCurrentUser();
  const data = await getDashboardData(user.id);
  return <Dashboard user={user} data={data} />;
}

The Hydration Process

Hydration is the process of attaching event handlers and React's internal state to server-rendered HTML. It's the bridge between static server output and an interactive client application.

Hydration Process: Server: React Tree → renderToPipeableStream → HTML String │ ▼ Network: HTML transmitted to browser ────────────────── │ ▼ Client: Parse HTML → Display (non-interactive) → Download JS │ ▼ hydrateRoot() → Walk DOM + React Tree simultaneously │ ├─ Match DOM nodes to fiber nodes ├─ Attach event listeners ├─ Initialize component state ├─ Run effects (useEffect, useLayoutEffect) └─ App becomes interactive (TTI)

hydrateRoot API

import { hydrateRoot } from 'react-dom/client';
import App from './App';

// Hydrate server-rendered HTML
// React expects the DOM to match what the server rendered
const root = hydrateRoot(
  document.getElementById('root'),
  <App />,
  {
    onRecoverableError(error, errorInfo) {
      // Hydration mismatch — React recovered but log it
      console.error('Hydration error:', error);
      reportToMonitoring(error, errorInfo);
    }
  }
);
Hydration Mismatches

When server HTML doesn't match what React renders on the client, React must discard the server HTML and re-render from scratch — destroying the SSR performance benefit. Common causes: Date.now(), Math.random(), browser-only APIs (window.innerWidth), locale differences, and extension-injected DOM nodes. In React 18+, mismatches log warnings in development and attempt recovery in production.

Hydration Internals

During hydration, React performs a concurrent traversal of both the existing DOM and the virtual React tree. For each fiber node, React:

  1. Finds the corresponding DOM node — uses depth-first traversal matching the render order
  2. Validates attributes — checks that text content, attributes, and structure match (development only for most checks)
  3. Attaches event listeners — uses event delegation on the root container, not individual DOM nodes
  4. Initializes state — hooks are initialized with their initial values (which should match server state)
  5. Schedules effects — useEffect callbacks are queued (they don't run on the server)
// Simplified hydration pseudocode
function hydrateNode(fiber, domNode) {
  if (fiber.tag === HostComponent) {
    // Reuse existing DOM node instead of creating new one
    fiber.stateNode = domNode;

    // Validate props match (dev only for most checks)
    if (__DEV__) {
      validateHydration(fiber.pendingProps, domNode);
    }

    // Attach event listeners via delegation
    listenToAllSupportedEvents(domNode);

    // Recurse into children
    hydrateChildren(fiber, domNode.firstChild);
  } else if (fiber.tag === FunctionComponent) {
    // Execute the component function
    const children = renderWithHooks(fiber);
    // Continue hydrating children against existing DOM
    reconcileChildFibers(fiber, children);
  }
}

Node.js Rendering Pipeline

Understanding the full Node.js SSR pipeline is critical for debugging performance issues and designing scalable rendering infrastructure.

Node.js SSR Pipeline: Incoming Request │ ▼ ┌─────────────────┐ │ Parse URL/ │ │ Match Route │ └────────┬────────┘ │ ▼ ┌─────────────────┐ ┌──────────────┐ │ Data Fetching │────▶│ API / DB │ │ (parallel) │◀────│ Calls │ └────────┬────────┘ └──────────────┘ │ ▼ ┌─────────────────┐ │ Create React │ │ Element Tree │ └────────┬────────┘ │ ▼ ┌─────────────────┐ │ renderTo │ │ PipeableStream │──── Render Phase (CPU-bound) └────────┬────────┘ │ ▼ ┌─────────────────┐ │ Stream HTML │──── I/O Phase (non-blocking) │ to Response │ └────────┬────────┘ │ ▼ Client receives progressive HTML
Architect Insight

The render phase is CPU-bound and blocks the Node.js thread. For complex component trees, this can take 50-200ms. At scale, this means a single Node.js process can only handle 5-20 SSR requests per second. Mitigation strategies: worker threads for rendering, horizontal scaling, aggressive caching, and streaming to reduce time-to-first-byte.

Universal / Isomorphic JavaScript

Universal JavaScript runs on both server and client. This sounds simple but introduces subtle challenges around environment-specific APIs, module resolution, and state management.

Environment Detection Patterns

// Anti-pattern: checking typeof window everywhere
if (typeof window !== 'undefined') {
  window.addEventListener('resize', handler);
}

// Better: useEffect for client-only side effects
// Effects don't run on the server
function useWindowSize() {
  const [size, setSize] = useState({ width: 0, height: 0 });

  useEffect(() => {
    function handleResize() {
      setSize({
        width: window.innerWidth,
        height: window.innerHeight,
      });
    }
    handleResize();
    window.addEventListener('resize', handleResize);
    return () => window.removeEventListener('resize', handleResize);
  }, []);

  return size;
}

// Best: dynamic import for client-only modules
const MapComponent = dynamic(() => import('./Map'), {
  ssr: false,
  loading: () => <MapSkeleton />,
});

Request / Response Lifecycle

A complete SSR request involves multiple phases, each with distinct performance characteristics and failure modes.

// Complete SSR request lifecycle
import { renderToPipeableStream } from 'react-dom/server';
import { matchRoutes } from 'react-router-dom';
import { createStaticHandler } from '@remix-run/router';

async function handleSSRRequest(req, res) {
  const startTime = performance.now();

  // Phase 1: Route matching (~1ms)
  const routes = matchRoutes(routeConfig, req.url);
  if (!routes) {
    res.status(404).send('Not Found');
    return;
  }

  // Phase 2: Data fetching (~50-500ms, often the bottleneck)
  const dataPromises = routes.map(route =>
    route.route.loader?.({ params: route.params, request: req })
  );
  const loaderData = await Promise.all(dataPromises);

  // Phase 3: Create React tree with data (~1ms)
  const app = (
    <StaticRouter location={req.url}>
      <DataContext.Provider value={loaderData}>
        <App />
      </DataContext.Provider>
    </StaticRouter>
  );

  // Phase 4: Render to stream (~20-200ms for render, then streaming)
  const { pipe, abort } = renderToPipeableStream(app, {
    bootstrapScripts: ['/bundle.js'],
    onShellReady() {
      const renderTime = performance.now() - startTime;
      res.setHeader('Server-Timing', `render;dur=${renderTime}`);
      res.setHeader('Content-Type', 'text/html');
      pipe(res);
    },
    onError(error) {
      console.error('SSR Error:', error);
    }
  });

  setTimeout(() => abort(), 10000);
}

HTML Streaming

HTML streaming sends HTML to the browser in chunks rather than waiting for the entire page to render. The browser can parse and display partial HTML while the server continues rendering.

Traditional SSR (Buffered): Server: [====== Render entire page ======] Client: [Parse + Display] ├──────── TTFB (slow) ──────────┤ Streaming SSR: Server: [= Shell =][== Suspense 1 ==][== Suspense 2 ==] Client: [Parse] [Parse + Inject] [Parse + Inject] ├ TTFB ─┤ (fast)

Streaming works by sending the HTML shell first (everything outside Suspense boundaries), then injecting completed Suspense content using inline <script> tags that swap placeholder content with real content.

// What the browser receives during streaming:

// 1. Initial shell (sent immediately)
<!DOCTYPE html>
<html>
<body>
  <div id="root">
    <header>My App</header>
    <main>
      <!-- Suspense fallback placeholder -->
      <template id="B:0"></template>
      <div>Loading...</div>
      <!--/$-->
    </main>
  </div>

// 2. When Suspense boundary resolves (sent later)
  <div hidden id="S:0">
    <article>Actual content here...</article>
  </div>
  <script>
    // React's inline script swaps the placeholder
    $RC("B:0", "S:0");
  </script>

// 3. Final closing tags
</body>
</html>
Deep Dive

The $RC function (completeBoundary) is a tiny inline script React injects. It finds the template marker, removes the fallback content, and inserts the resolved content — all before the main JS bundle loads. This means users see real content progressively, even on slow connections where the bundle hasn't downloaded yet.

Streaming and SEO

Search engine crawlers handle streaming differently. Googlebot waits for the full response before indexing, so streaming doesn't affect SEO content. However, you can optimize for crawlers by using onAllReady (Node.js) or stream.allReady (Web Streams) to send the complete HTML at once for bot user agents, while streaming for real users.

const { pipe } = renderToPipeableStream(<App />, {
  bootstrapScripts: ['/bundle.js'],

  onShellReady() {
    if (!isBot(req)) {
      // Stream for real users — faster TTFB
      pipe(res);
    }
  },

  onAllReady() {
    if (isBot(req)) {
      // Send complete HTML for crawlers — better indexing
      pipe(res);
    }
  }
});
Architect Insight

At L6, you should be able to articulate the full SSR decision matrix: when to use SSG (content sites, marketing pages), ISR (e-commerce catalogs, CMS-driven pages), SSR (dashboards, personalized content), and CSR (highly interactive tools, authenticated-only apps). The answer is almost always "it depends on the route" — a single application typically uses multiple strategies.
