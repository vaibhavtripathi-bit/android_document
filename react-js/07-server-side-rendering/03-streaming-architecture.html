The Streaming Model

Traditional SSR buffers the entire HTML response before sending anything to the client. Streaming SSR sends HTML in chunks as components render, allowing the browser to start parsing and displaying content before the server finishes rendering the entire page.

Buffered SSR (renderToString): Server: [======= Render ALL components =======]──▶ Send Client: [== Parse + Display ==] Time: ├────── 200ms render ──────────────────┤ ├──── 50ms ────┤ TTFB = 200ms FCP = 250ms Streaming SSR (renderToPipeableStream): Server: [= Shell 30ms =][== Chunk 1 80ms ==][== Chunk 2 150ms ==] Client: [Parse Shell] [Parse + Inject] [Parse + Inject] Time: ├── 30ms ──┤ TTFB = 30ms FCP = 30ms (shell visible immediately)

The key insight: streaming decouples TTFB from total render time. Users see content in ~30ms (shell) instead of waiting ~200ms for the entire page. For pages with slow data dependencies, the difference can be seconds.

renderToPipeableStream Internals

Under the hood, renderToPipeableStream creates a Fizz renderer — React's streaming server renderer. Fizz traverses the React tree depth-first, writing HTML to an internal buffer that gets flushed to the Node.js writable stream.

// Simplified Fizz renderer internals
class FizzRenderer {
  constructor(rootElement, options) {
    this.destination = null;  // Node.js writable stream
    this.completedRootSegment = null;
    this.pendingBoundaries = new Set();
    this.completedBoundaries = [];

    // Start rendering the tree
    this.performWork(rootElement);
  }

  performWork(element) {
    // Depth-first traversal of React tree
    const segment = this.createSegment();

    try {
      this.renderNode(element, segment);
    } catch (thrownValue) {
      if (isPromise(thrownValue)) {
        // Hit a Suspense boundary — render fallback
        // and track the promise
        const boundary = this.createBoundary(segment);
        this.renderFallback(boundary);
        this.pendingBoundaries.add(boundary);

        thrownValue.then(() => {
          // Promise resolved — render the real content
          this.completeBoundary(boundary);
        });
      } else {
        throw thrownValue; // Real error
      }
    }
  }

  completeBoundary(boundary) {
    // Render the resolved content into a hidden div
    // Generate inline script to swap fallback with content
    this.completedBoundaries.push(boundary);
    this.pendingBoundaries.delete(boundary);
    this.flushCompletedBoundaries();
  }

  flushCompletedBoundaries() {
    if (!this.destination) return;

    for (const boundary of this.completedBoundaries) {
      // Write: 
...resolved content...

      // Write: 
      writeChunk(this.destination, boundary.completedHTML);
    }
    this.completedBoundaries = [];
  }
}

Fizz Rendering Phases

Fizz Rendering Phases: Phase 1: Shell Rendering (synchronous) ├─ Traverse tree depth-first ├─ Write HTML for non-suspended components ├─ Hit Suspense → write fallback, track promise ├─ Complete → fire onShellReady() └─ pipe(res) starts streaming shell to client Phase 2: Boundary Completion (async, incremental) ├─ Promises resolve (data fetching completes) ├─ Render resolved content into hidden segments ├─ Write inline <script> to swap fallback → content ├─ Flush completed boundaries to stream └─ Repeat until all boundaries complete Phase 3: Finalization ├─ All boundaries resolved → fire onAllReady() ├─ Write closing HTML tags └─ End the stream

Suspense Boundaries as Streaming Units

Each <Suspense> boundary in your component tree becomes a streaming unit. The fallback renders immediately as part of the shell, and the resolved content streams in later. This means your Suspense boundary placement directly controls the streaming granularity.

// Each Suspense boundary = one streaming unit
function ProductPage({ productId }) {
  return (
    <div>
      {/* Shell: renders immediately */}
      <Header />
      <Navigation />

      {/* Streaming unit 1: product details */}
      <Suspense fallback={<ProductSkeleton />}>
        <ProductDetails id={productId} />
      </Suspense>

      {/* Streaming unit 2: reviews (independent) */}
      <Suspense fallback={<ReviewsSkeleton />}>
        <Reviews productId={productId} />
      </Suspense>

      {/* Streaming unit 3: recommendations */}
      <Suspense fallback={<RecommendationsSkeleton />}>
        <Recommendations productId={productId} />
      </Suspense>

      {/* Shell: renders immediately */}
      <Footer />
    </div>
  );
}
Architect Insight

Suspense boundary placement is an architectural decision with significant UX impact. Too few boundaries: the entire page waits for the slowest data source. Too many boundaries: excessive skeleton flashing and layout shifts. The sweet spot is typically 2-4 boundaries per page, aligned with independent data dependencies and visual sections.

Nested Suspense Boundaries

Suspense boundaries can be nested. The outer boundary's fallback shows until its direct content resolves. Inner boundaries then show their own fallbacks independently.

// Nested Suspense — progressive disclosure
<Suspense fallback={<PageSkeleton />}>
  {/* Outer boundary: page layout + product data */}
  <ProductLayout product={product}>

    <Suspense fallback={<PriceSkeleton />}>
      {/* Inner boundary 1: pricing (separate API) */}
      <DynamicPricing productId={product.id} />
    </Suspense>

    <Suspense fallback={<InventorySkeleton />}>
      {/* Inner boundary 2: inventory (separate API) */}
      <InventoryStatus productId={product.id} />
    </Suspense>

  </ProductLayout>
</Suspense>

// Streaming order:
// 1. PageSkeleton (immediate)
// 2. ProductLayout + PriceSkeleton + InventorySkeleton (when product loads)
// 3. DynamicPricing (when pricing API responds)
// 4. InventoryStatus (when inventory API responds)
// Steps 3 and 4 are independent — either can complete first

The Shell Concept

The "shell" is everything that renders synchronously without hitting a Suspense boundary. It's the first HTML the browser receives and determines the initial visual state of the page.

Shell vs Streamed Content: ┌─────────────────────────────────────────────┐ │ <html> │ │ <head>...</head> │ │ <body> │ │ ┌─────────────────────────────────┐ │ │ │ SHELL (sent immediately) │ │ │ │ ├─ <Header /> │ │ │ │ ├─ <Nav /> │ │ │ │ ├─ Suspense fallback 1 │ │ │ │ ├─ Suspense fallback 2 │ │ │ │ └─ <Footer /> │ │ │ └─────────────────────────────────┘ │ │ │ │ ┌─────────────────────────────────┐ │ │ │ STREAMED (sent as resolved) │ │ │ │ ├─ <div hidden id="S:0">... │ │ │ │ │ <script>$RC("B:0","S:0") │ │ │ │ ├─ <div hidden id="S:1">... │ │ │ │ │ <script>$RC("B:1","S:1") │ │ │ │ └─ Bootstrap scripts │ │ │ └─────────────────────────────────┘ │ │ </body> │ │ </html> │ └─────────────────────────────────────────────┘

Shell Error vs Boundary Error

const { pipe, abort } = renderToPipeableStream(<App />, {
  onShellReady() {
    // Shell rendered successfully — start streaming
    // This is the happy path for real users
    res.statusCode = 200;
    pipe(res);
  },

  onShellError(error) {
    // Shell FAILED — nothing useful to stream
    // Send a fallback HTML page (static error page)
    // This fires when a component outside Suspense throws
    res.statusCode = 500;
    res.send(staticErrorPage);
  },

  onError(error) {
    // Any error during rendering (shell or boundary)
    // Use for logging — the error is already handled
    // by either onShellError or the Suspense error boundary
    logError(error);
  }
});

Out-of-Order Streaming

React's streaming SSR supports out-of-order completion — Suspense boundaries can resolve and stream in any order, regardless of their position in the document. This is possible because resolved content is injected via JavaScript, not by document order.

// Component tree with different data latencies
function Dashboard() {
  return (
    <div>
      <Suspense fallback={<ChartSkeleton />}>
        <AnalyticsChart />  {/* 500ms API call */}
      </Suspense>

      <Suspense fallback={<TableSkeleton />}>
        <DataTable />  {/* 200ms API call */}
      </Suspense>

      <Suspense fallback={<FeedSkeleton />}>
        <ActivityFeed />  {/* 800ms API call */}
      </Suspense>
    </div>
  );
}

// Streaming order (by completion time, NOT document order):
// t=0ms:   Shell + all three skeletons
// t=200ms: DataTable content (replaces TableSkeleton)
// t=500ms: AnalyticsChart content (replaces ChartSkeleton)
// t=800ms: ActivityFeed content (replaces FeedSkeleton)
How Out-of-Order Works

Each resolved boundary is sent as a hidden <div> at the end of the document, followed by an inline <script> that moves it to the correct position. The browser executes these scripts as they arrive, swapping fallbacks with real content. This works because HTML parsing is incremental — the browser processes each chunk as it arrives.

Selective Hydration

Selective hydration is the client-side counterpart to streaming SSR. Instead of hydrating the entire page at once, React hydrates Suspense boundaries independently, prioritizing boundaries the user is interacting with.

Selective Hydration Priority: 1. Shell hydration (always first) 2. Suspense boundaries hydrate in completion order 3. User interaction REPRIORITIZES hydration: Before click: [Shell ✓] [Boundary A: hydrating] [B: waiting] [C: waiting] User clicks C: [Shell ✓] [Boundary A: paused] [B: waiting] [C: HYDRATING] ↑ prioritized! After C done: [Shell ✓] [Boundary A: hydrating] [B: waiting] [C: ✓] 
// Selective hydration happens automatically with Suspense
// No special API needed — React detects user interaction

function App() {
  return (
    <div>
      <Header />  {/* Hydrates first (shell) */}

      <Suspense fallback={<Skeleton />}>
        <HeavyChart />  {/* Hydrates when ready */}
      </Suspense>

      <Suspense fallback={<Skeleton />}>
        <Comments />  {/* If user clicks here, hydrates FIRST */}
      </Suspense>

      <Suspense fallback={<Skeleton />}>
        <Sidebar />  {/* Hydrates last (no interaction) */}
      </Suspense>
    </div>
  );
}

// React captures the click event during capture phase,
// synchronously hydrates the target boundary,
// then replays the event on the now-hydrated component

Event Replay Mechanism

When a user clicks on a not-yet-hydrated Suspense boundary, React:

  1. Captures the event during the capture phase (before it reaches the target)
  2. Identifies which Suspense boundary contains the target
  3. Synchronously hydrates that boundary (interrupting any in-progress hydration)
  4. Replays the captured event on the now-hydrated component

This means the user's click is never lost — it's queued and replayed after hydration completes. From the user's perspective, the click just works (with a slight delay).

Backpressure Handling

Backpressure occurs when the server produces data faster than the client can consume it. In streaming SSR, this happens when the network is slow or the client's TCP receive buffer fills up.

// Node.js streams handle backpressure automatically
// pipe() respects the writable stream's drain events

const { pipe } = renderToPipeableStream(<App />, {
  onShellReady() {
    pipe(res); // pipe handles backpressure internally
  }
});

// Under the hood:
// 1. Fizz writes chunks to the response stream
// 2. If res.write() returns false (buffer full), Fizz pauses
// 3. When 'drain' event fires (client consumed data), Fizz resumes
// 4. This prevents unbounded memory growth on the server

// Manual backpressure handling (if not using pipe):
const { pipe } = renderToPipeableStream(<App />, {
  onShellReady() {
    // Don't do this in production — use pipe() instead
    // This is for illustration only
    const reader = createReadableFromPipeable(pipe);
    reader.on('data', (chunk) => {
      const canContinue = res.write(chunk);
      if (!canContinue) {
        reader.pause();
        res.once('drain', () => reader.resume());
      }
    });
  }
});
Memory Pressure

Without proper backpressure handling, a slow client can cause the server to buffer the entire rendered HTML in memory. For large pages (100KB+ HTML), this can cause memory pressure under high concurrency. Always use pipe() rather than manually writing chunks — it handles backpressure correctly.

Node.js Streams vs Web Streams

React provides two streaming APIs targeting different runtime environments. Understanding the differences is essential for cross-platform SSR.

// Web Streams: TransformStream for post-processing
import { renderToReadableStream } from 'react-dom/server';

async function handleRequest(req) {
  const stream = await renderToReadableStream(<App />, {
    bootstrapScripts: ['/bundle.js'],
  });

  // Transform the stream (e.g., inject analytics script)
  const transformedStream = stream.pipeThrough(
    new TransformStream({
      transform(chunk, controller) {
        // Modify chunks as they pass through
        const text = new TextDecoder().decode(chunk);
        const modified = text.replace(
          '</body>',
          '<script src="/analytics.js"></script></body>'
        );
        controller.enqueue(new TextEncoder().encode(modified));
      }
    })
  );

  return new Response(transformedStream, {
    headers: { 'Content-Type': 'text/html' },
  });
}
Architect Insight

Web Streams are the future — they're standardized across runtimes and compose better than Node.js streams. However, Node.js streams have a larger ecosystem and better tooling today. For new projects targeting edge deployment, prefer renderToReadableStream. For Node.js-only deployments, renderToPipeableStream has slightly better performance due to less abstraction overhead.

Error Handling in Streams

// Comprehensive error handling for streaming SSR
const { pipe, abort } = renderToPipeableStream(<App />, {
  onShellReady() {
    res.statusCode = shellError ? 500 : 200;
    res.setHeader('Content-Type', 'text/html');
    pipe(res);
  },

  onShellError(error) {
    // Shell failed — can't stream anything useful
    // Option 1: Send static fallback
    res.statusCode = 500;
    res.end(getStaticErrorPage());

    // Option 2: Redirect to CSR fallback
    // res.redirect('/client-only-fallback');
  },

  onError(error) {
    // Fires for ALL errors (shell + boundaries)
    // Boundary errors are caught by error boundaries
    // Shell errors trigger onShellError
    logToErrorService(error);
  }
});

// Timeout: abort rendering if it takes too long
const timeout = setTimeout(() => {
  abort(); // Sends error boundary fallback for pending boundaries
}, 10000);

res.on('close', () => clearTimeout(timeout));
