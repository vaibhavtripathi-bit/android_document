HTTP Caching

HTTP caching is the foundation of SSR performance. The Cache-Control header controls how browsers, CDNs, and proxies cache responses. Understanding every directive is essential for Staff-level SSR architecture.

Cache-Control Directives

// Cache-Control header anatomy
Cache-Control: public, max-age=60, s-maxage=3600, stale-while-revalidate=86400

// Directive breakdown:
// public          — Any cache (browser, CDN, proxy) can store this
// private         — Only the browser can cache (personalized content)
// max-age=60      — Browser cache: fresh for 60 seconds
// s-maxage=3600   — Shared cache (CDN): fresh for 1 hour
// no-cache        — Must revalidate with origin before using cached copy
// no-store        — Never cache (sensitive data)
// must-revalidate — Once stale, MUST revalidate (no serving stale)
// stale-while-revalidate=86400 — Serve stale for 24h while revalidating
// stale-if-error=86400 — Serve stale for 24h if origin errors

ETag and Conditional Requests

// Server generates ETag based on content hash
import crypto from 'crypto';

function handleSSR(req, res) {
  const html = renderPage(req.url);
  const etag = crypto
    .createHash('md5')
    .update(html)
    .digest('hex');

  // Check if client has current version
  if (req.headers['if-none-match'] === `"${etag}"`) {
    res.status(304).end(); // Not Modified — use cached version
    return;
  }

  res.setHeader('ETag', `"${etag}"`);
  res.setHeader('Cache-Control', 'public, max-age=0, must-revalidate');
  res.send(html);
}

// Flow:
// 1. First request: Server sends HTML + ETag: "abc123"
// 2. Second request: Browser sends If-None-Match: "abc123"
// 3. If content unchanged: Server responds 304 (no body)
// 4. If content changed: Server sends new HTML + new ETag

Stale-While-Revalidate (SWR)

stale-while-revalidate Timeline: Cache-Control: max-age=60, stale-while-revalidate=3600 t=0s: Request → Cache MISS → Origin renders → Cache stores → Response t=30s: Request → Cache HIT (fresh) → Instant response t=60s: Content becomes STALE t=90s: Request → Cache serves STALE instantly → Background revalidation to origin → Cache updates with fresh content t=3660s: Stale window expires → Cache MISS → Origin renders
Architect Insight

stale-while-revalidate is the most important caching directive for SSR. It provides instant responses (from stale cache) while ensuring freshness (background revalidation). For most SSR pages, Cache-Control: public, s-maxage=60, stale-while-revalidate=86400 provides an excellent balance: CDN serves cached content for 1 minute, then serves stale for up to 24 hours while revalidating in the background.

CDN Caching for SSR

CDN caching for SSR pages requires careful consideration of cache keys, personalization, and invalidation. Unlike static assets (which are immutable), SSR pages have dynamic content that must be cached intelligently.

// CDN caching strategy for SSR pages
function getSSRCacheHeaders(req) {
  const isAuthenticated = req.cookies.session;
  const isBot = /bot|crawler/i.test(req.headers['user-agent']);

  if (isAuthenticated) {
    // Personalized pages: no CDN caching
    return {
      'Cache-Control': 'private, no-cache',
      'CDN-Cache-Control': 'no-store',
    };
  }

  if (isBot) {
    // Bots: cache longer (they don't need real-time data)
    return {
      'Cache-Control': 'public, max-age=3600',
      'CDN-Cache-Control': 'public, max-age=86400',
    };
  }

  // Public pages: aggressive CDN caching with SWR
  return {
    'Cache-Control': 'public, max-age=0, must-revalidate',
    'CDN-Cache-Control': 'public, s-maxage=300, stale-while-revalidate=86400',
    'Surrogate-Control': 'max-age=300', // Fastly/Varnish
  };
}

// Vary header: different cache entries per variant
res.setHeader('Vary', 'Accept-Language, Accept-Encoding');
// CDN caches separate versions for:
// - Accept-Language: en → English version
// - Accept-Language: ja → Japanese version
// - Accept-Encoding: gzip vs br → compression variants

Application-Level Caching

React cache() Function

// React cache() — request-level memoization
import { cache } from 'react';

// Deduplicated within a single render/request
const getUser = cache(async (userId: string) => {
  const response = await fetch(`/api/users/${userId}`);
  return response.json();
});

// Both components call getUser('123') but only ONE fetch happens
async function UserProfile({ userId }) {
  const user = await getUser(userId);
  return <div>{user.name}</div>;
}

async function UserAvatar({ userId }) {
  const user = await getUser(userId); // Deduplicated!
  return <img src={user.avatar} />;
}

Next.js Caching Layers

// Layer 1: Request Memoization (automatic with fetch)
// Same fetch URL + options = deduplicated within one request
const data1 = await fetch('https://api.example.com/data');
const data2 = await fetch('https://api.example.com/data'); // Reuses data1

// Layer 2: Data Cache (persistent across requests)
// Default: cached indefinitely
const data = await fetch('https://api.example.com/data');

// Time-based revalidation
const data = await fetch('https://api.example.com/data', {
  next: { revalidate: 60 } // Revalidate every 60s
});

// Tag-based invalidation
const data = await fetch('https://api.example.com/posts', {
  next: { tags: ['posts'] }
});

// No caching
const data = await fetch('https://api.example.com/data', {
  cache: 'no-store'
});

// Layer 3: Full Route Cache (static routes only)
// Automatically caches RSC payload + HTML for static routes
// Invalidated when underlying Data Cache entries are revalidated

// Layer 4: Router Cache (client-side, in-memory)
// Caches RSC payloads during client-side navigation
// Duration: 30s (dynamic) / 5min (static) / session-based
import { useRouter } from 'next/navigation';
const router = useRouter();
router.refresh(); // Invalidates Router Cache for current route

Incremental Static Regeneration

ISR combines the performance of static generation with the freshness of server rendering. Pages are generated at build time and regenerated in the background when the revalidation period expires.

ISR Lifecycle: Build Time: Page generated → Cached as static HTML t=0: Request → Serve static HTML (instant, from CDN) t=30s: Request → Serve static HTML (still fresh) t=60s: Revalidation period expires (revalidate: 60) t=61s: Request → Serve STALE static HTML (instant) → Trigger background regeneration → New HTML generated and cached t=62s: Request → Serve NEW static HTML (fresh) On-Demand: revalidatePath('/blog/post-1') → Immediately regenerate revalidateTag('posts') → Regenerate all pages with 'posts' tag 
// ISR with time-based revalidation
// app/blog/[slug]/page.tsx
export const revalidate = 60; // Revalidate every 60 seconds

export async function generateStaticParams() {
  const posts = await getAllPosts();
  return posts.map(post => ({ slug: post.slug }));
}

export default async function BlogPost({ params }) {
  const post = await getPost(params.slug);
  return <Article post={post} />;
}

// ISR with on-demand revalidation
// app/api/revalidate/route.ts
import { revalidatePath, revalidateTag } from 'next/cache';

export async function POST(request: Request) {
  const { secret, path, tag } = await request.json();

  if (secret !== process.env.REVALIDATION_SECRET) {
    return Response.json({ error: 'Invalid secret' }, { status: 401 });
  }

  if (path) {
    revalidatePath(path);
  }
  if (tag) {
    revalidateTag(tag);
  }

  return Response.json({ revalidated: true, now: Date.now() });
}

On-Demand Revalidation

// Webhook-triggered revalidation (CMS publishes content)
// app/api/cms-webhook/route.ts
export async function POST(request: Request) {
  const payload = await request.json();

  // Verify webhook signature
  const signature = request.headers.get('x-cms-signature');
  if (!verifySignature(payload, signature)) {
    return Response.json({ error: 'Invalid signature' }, { status: 401 });
  }

  switch (payload.event) {
    case 'post.published':
    case 'post.updated':
      // Revalidate specific post page
      revalidatePath(`/blog/${payload.data.slug}`);
      // Revalidate blog listing
      revalidateTag('blog-posts');
      break;

    case 'post.deleted':
      revalidatePath(`/blog/${payload.data.slug}`);
      revalidateTag('blog-posts');
      break;

    case 'settings.updated':
      // Revalidate entire site layout
      revalidatePath('/', 'layout');
      break;
  }

  return Response.json({ revalidated: true });
}

// Server Action with revalidation
'use server';
export async function updateProduct(formData: FormData) {
  const id = formData.get('id');
  await db.products.update({
    where: { id },
    data: { name: formData.get('name') },
  });

  // Revalidate the product page and listing
  revalidatePath(`/products/${id}`);
  revalidateTag('products');
}

Tag-Based Cache Invalidation

// Tag-based caching — group related cache entries
// app/blog/page.tsx
async function BlogList() {
  const posts = await fetch('https://api.example.com/posts', {
    next: { tags: ['posts', 'blog-listing'] }
  }).then(r => r.json());

  return posts.map(post => <PostCard key={post.id} post={post} />);
}

// app/blog/[slug]/page.tsx
async function BlogPost({ params }) {
  const post = await fetch(`https://api.example.com/posts/${params.slug}`, {
    next: { tags: ['posts', `post-${params.slug}`] }
  }).then(r => r.json());

  const comments = await fetch(
    `https://api.example.com/posts/${params.slug}/comments`,
    { next: { tags: [`comments-${params.slug}`] } }
  ).then(r => r.json());

  return <Article post={post} comments={comments} />;
}

// Invalidation scenarios:
// revalidateTag('posts')              → All post pages + listing
// revalidateTag('post-hello-world')   → Just one post page
// revalidateTag('comments-hello-world') → Just comments for one post
// revalidateTag('blog-listing')       → Just the listing page

Distributed Caching

Redis Caching for SSR

// Redis-backed SSR cache
import Redis from 'ioredis';

const redis = new Redis(process.env.REDIS_URL);

async function cachedSSR(req, res, renderFn) {
  const cacheKey = `ssr:${req.url}`;

  // Try cache first
  const cached = await redis.get(cacheKey);
  if (cached) {
    const { html, headers, timestamp } = JSON.parse(cached);
    res.setHeader('X-Cache', 'HIT');
    res.setHeader('X-Cache-Age', Math.floor((Date.now() - timestamp) / 1000));
    Object.entries(headers).forEach(([k, v]) => res.setHeader(k, v));
    return res.send(html);
  }

  // Cache miss — render and store
  const html = await renderFn(req);
  const cacheEntry = {
    html,
    headers: { 'Content-Type': 'text/html' },
    timestamp: Date.now(),
  };

  // Store with TTL
  await redis.setex(cacheKey, 300, JSON.stringify(cacheEntry));

  res.setHeader('X-Cache', 'MISS');
  res.send(html);
}

// Cache warming — pre-populate cache for high-traffic pages
async function warmCache(urls: string[]) {
  const concurrency = 10;
  for (let i = 0; i < urls.length; i += concurrency) {
    const batch = urls.slice(i, i + concurrency);
    await Promise.all(
      batch.map(url =>
        fetch(`https://internal.example.com${url}`, {
          headers: { 'X-Cache-Warm': 'true' }
        })
      )
    );
  }
}

Cache Stampede Prevention

A cache stampede occurs when a cached entry expires and many concurrent requests hit the origin simultaneously, overwhelming the server. This is a critical concern for high-traffic SSR applications.

Cache Stampede: Normal: [Cache HIT] [Cache HIT] [Cache HIT] [Expire] [Cache MISS → Origin] Stampede: [Cache HIT] [Cache HIT] [Expire] ├─ Request 1 → Origin (render) ├─ Request 2 → Origin (render) ├─ Request 3 → Origin (render) ├─ ... └─ Request N → Origin (render) ALL hit origin simultaneously! Prevention (Lock): [Cache HIT] [Cache HIT] [Expire] ├─ Request 1 → Acquires lock → Origin ├─ Request 2 → Lock held → Serve stale ├─ Request 3 → Lock held → Serve stale └─ Request 1 completes → Updates cache → Release lock 
// Cache stampede prevention with distributed lock
async function getWithStampedeProtection(key, renderFn, ttl = 300) {
  const cached = await redis.get(`cache:${key}`);

  if (cached) {
    const entry = JSON.parse(cached);
    const age = (Date.now() - entry.timestamp) / 1000;

    if (age < ttl) {
      return entry.data; // Fresh — serve directly
    }

    // Stale — try to acquire lock for revalidation
    const lockKey = `lock:${key}`;
    const acquired = await redis.set(lockKey, '1', 'EX', 30, 'NX');

    if (acquired) {
      // We got the lock — revalidate in background
      renderFn().then(async (data) => {
        await redis.setex(
          `cache:${key}`,
          ttl * 2, // Store for 2x TTL (stale buffer)
          JSON.stringify({ data, timestamp: Date.now() })
        );
        await redis.del(lockKey);
      });
    }

    // Serve stale while revalidating (whether we got lock or not)
    return entry.data;
  }

  // No cache at all — must render synchronously
  const data = await renderFn();
  await redis.setex(
    `cache:${key}`,
    ttl * 2,
    JSON.stringify({ data, timestamp: Date.now() })
  );
  return data;
}

// Probabilistic early expiration (jitter)
function shouldRevalidateEarly(age, ttl) {
  // As age approaches TTL, probability of early revalidation increases
  // This spreads revalidation across time, preventing stampedes
  const remaining = ttl - age;
  const probability = Math.exp(-remaining / (ttl * 0.1));
  return Math.random() < probability;
}
Architect Insight

At L6, your caching strategy should be a multi-layer architecture: (1) Browser cache for repeat visits, (2) CDN/edge cache for geographic distribution, (3) Application cache (Redis) for computed data, (4) Framework cache (Next.js Data Cache) for fetch deduplication. Each layer has different TTLs and invalidation strategies. The key insight: cache invalidation is harder than caching — invest in robust invalidation (tags, webhooks, on-demand) rather than just TTLs.
